Sintaxis #

  • ESTABLECER NIVEL DE AISLAMIENTO DE TRANSACCIÓN { LEER SIN COMPROMISO | LEER COMPROMETIDO | LECTURA REPETIBLE | INSTANTÁNEA | SERIALIZABLE } [ ; ]

Referencia de MSDN: ESTABLECER NIVEL DE AISLAMIENTO DE TRANSACCIONES

Lectura no confirmada #

SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED

Este es el nivel de aislamiento más permisivo, ya que no provoca ningún bloqueo. Especifica que las declaraciones pueden leer todas las filas, incluidas las filas que se han escrito en transacciones pero que aún no se han confirmado (es decir, todavía están en transacción). Este nivel de aislamiento puede estar sujeto a “lecturas sucias”.

Lectura confirmada #

SET TRANSACTION ISOLATION LEVEL READ COMMITTED

Este nivel de aislamiento es el segundo más permisivo. Previene lecturas sucias. El comportamiento de READ COMMITTED depende de la configuración de READ_COMMITTED_SNAPSHOT:

  • Si se establece en APAGADO (la configuración predeterminada), la transacción utiliza bloqueos compartidos para evitar que otras transacciones modifiquen las filas utilizadas por la transacción actual, así como para impedir que la transacción actual lea las filas modificadas por otras transacciones.

  • Si se establece en ON, la sugerencia de la tabla READCOMMITTEDLOCK se puede usar para solicitar el bloqueo compartido en lugar de la versión de fila para las transacciones que se ejecutan en el modo READ COMMITTED.

Nota: READ COMMITTED es el comportamiento predeterminado de SQL Server.

¿Qué son las “lecturas sucias”? #

Las lecturas sucias (o lecturas no confirmadas) son lecturas de filas que están siendo modificadas por una transacción abierta.

Este comportamiento se puede replicar usando 2 consultas separadas: una para abrir una transacción y escribir algunos datos en una tabla sin confirmar, la otra para seleccionar los datos que se escribirán (pero aún no confirmados) con este nivel de aislamiento.

Consulta 1 - Prepara una transacción pero no la finalizas:

CREATE TABLE dbo.demo (
    col1 INT,
    col2 VARCHAR(255)
);
GO
--This row will get committed normally:
BEGIN TRANSACTION;
    INSERT INTO dbo.demo(col1, col2) 
    VALUES (99, 'Normal transaction');
COMMIT TRANSACTION;
--This row will be "stuck" in an open transaction, causing a dirty read
BEGIN TRANSACTION;
    INSERT INTO dbo.demo(col1, col2) 
    VALUES (42, 'Dirty read');
--Do not COMMIT TRANSACTION or ROLLBACK TRANSACTION here

Consulta 2: lea las filas, incluida la transacción abierta:

SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT * FROM dbo.demo;

Devoluciones:

columna1 columna2

99 Transacción normal 42 Lectura sucia

P.D.: No olvide limpiar estos datos de demostración:

COMMIT TRANSACTION;
DROP TABLE dbo.demo;
GO

Lectura repetible #

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ

Este nivel de aislamiento de transacciones es un poco menos permisivo que LEER CONFIRMADO, en el sentido de que se colocan bloqueos compartidos en todos los datos leídos por cada declaración en la transacción y se retienen hasta que se completa la transacción, en lugar de liberarse después de cada declaración. .

Nota: Use esta opción solo cuando sea necesario, ya que es más probable que provoque una degradación del rendimiento de la base de datos, así como interbloqueos, que LEER COMPROMETIDO.

Instantánea #

SET TRANSACTION ISOLATION LEVEL SNAPSHOT

Especifica que los datos leídos por cualquier declaración en una transacción serán la versión transaccionalmente coherente de los datos que existían al inicio de la transacción, es decir, solo leerá los datos que se hayan confirmado antes del inicio de la transacción.

Las transacciones SNAPSHOT no solicitan ni provocan ningún bloqueo en los datos que se están leyendo, ya que solo está leyendo la versión (o instantánea) de los datos que existían en el momento en que comenzó la transacción.

Una transacción que se ejecuta en el nivel de aislamiento SNAPSHOT solo lee sus propios cambios de datos mientras se está ejecutando. Por ejemplo, una transacción podría actualizar algunas filas y luego leer las filas actualizadas, pero ese cambio solo será visible para la transacción actual hasta que se confirme.

Nota: La opción de base de datos ALLOW_SNAPSHOT_ISOLATION debe estar activada antes de que se pueda usar el nivel de aislamiento SNAPSHOT.

Serializable #

SET TRANSACTION ISOLATION LEVEL SERIALIZEABLE

Este nivel de aislamiento es el más restrictivo. Solicita bloqueos de rango el rango de valores clave que lee cada declaración en la transacción. Esto también significa que las declaraciones INSERT de otras transacciones se bloquearán si las filas que se insertarán están en el rango bloqueado por la transacción actual.

Esta opción tiene el mismo efecto que establecer HOLDLOCK en todas las tablas en todas las declaraciones SELECT en una transacción.

Nota: Este aislamiento de transacciones tiene la simultaneidad más baja y solo debe usarse cuando sea necesario.
