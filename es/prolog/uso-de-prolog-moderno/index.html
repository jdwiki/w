CLP(FD) para aritmética entera #

Tradicionalmente, Prolog realizaba operaciones aritméticas usando los operadores is y =:=. Sin embargo, varios Prologs actuales ofrecen CLP(FD) (Programación Lógica de Restricciones sobre Dominios Finitos) como una alternativa más limpia para la aritmética de enteros. CLP(FD) se basa en almacenar las restricciones que se aplican a un valor entero y combinarlas en la memoria.

CLP(FD) es una extensión en la mayoría de los Prologs que la admiten, por lo que debe cargarse explícitamente. Una vez que se carga, la sintaxis #= puede tomar el lugar de is y =:=. Por ejemplo, en SWI-Prolog:

?- X is 2+2.
X = 4.

?- use_module(library(clpfd)).
?- X #= 2+2.
X = 4.

A diferencia de is, #= puede resolver ecuaciones simples y unificar en ambas direcciones:

?- 4 is 2+X.
ERROR: is/2: Arguments are not sufficiently instantiated

?- 4 #= 2+X.
X = 2.

CLP(FD) proporciona su propia sintaxis de generador.

?- between(1,100,X).
X = 1;
X = 2;
X = 3...

?- X in 1..100.
X in 1..100.

Tenga en cuenta que el generador en realidad no se ejecuta: solo se almacena la restricción de rango, lista para combinarla con restricciones posteriores. Se puede forzar la ejecución del generador (y restricciones de fuerza bruta) usando el predicado label:

?- X in 1..100, label([X]).
X = 1;
X = 2;
X = 3..

El uso de CLP puede permitir una reducción inteligente de los casos de fuerza bruta. Por ejemplo, usando la aritmética de enteros al viejo estilo:

?- trace.
?- between(1,10,X), Y is X+5, Y>10.
...
Exit: (8) 6 is 1+5 ? creep
Call: (8) 6 > 10 ? creep
...
X = 6, Y = 11; ...

Prolog aún recorre los valores 1-5 aunque es matemáticamente demostrable a partir de las condiciones dadas que estos valores no pueden ser útiles. Usando CLP (FD):

?- X in 1..10, Y #= X+5, Y #> 10.
X is 6..10,
X+5 #= Y,
Y is 11..15.

CLP(FD) inmediatamente hace los cálculos y calcula los rangos disponibles. Agregar label([Y]) hará que X recorra solo los valores útiles 6..10. En este ejemplo de juguete, esto no aumenta el rendimiento porque con un rango tan pequeño como 1-10, el procesamiento de álgebra lleva tanto tiempo como lo hubiera hecho el ciclo; pero cuando se procesa una gama más amplia de números, esto puede reducir considerablemente el tiempo de cálculo.

El soporte para CLP(FD) es variable entre Prologs. El mejor desarrollo reconocido de CLP(FD) está en SICStus Prolog, que es comercial y costoso. SWI-Prolog y otros Prolog abiertos a menudo tienen alguna implementación. Visual Prolog no incluye CLP(FD) en su biblioteca estándar, aunque hay bibliotecas de extensión disponibles.

Introducción #

Muchos sistemas Prolog modernos están en continuo desarrollo y han agregado nuevas funciones para abordar las deficiencias clásicas del lenguaje. Desafortunadamente, muchos libros de texto de Prolog e incluso cursos de enseñanza todavía presentan solo el prólogo desactualizado. Este tema pretende ilustrar cómo el Prolog moderno ha superado algunos de los problemas y la sintaxis bastante tosca que aparece en el Prolog anterior y que aún puede estar introduciéndose.

Forall en lugar de bucles impulsados ​​por fallas #

Algunos libros de texto de Prolog “clásicos” todavía usan la sintaxis de bucle impulsada por fallas, confusa y propensa a errores, donde se usa una construcción “falla” para forzar el retroceso para aplicar un objetivo a cada valor de un generador. Por ejemplo, para imprimir todos los números hasta un límite dado:

fdl(X) :- between(1,X,Y), print(Y), fail.
fdl(_).

La gran mayoría de Modern Prologs ya no requieren esta sintaxis, sino que proporcionan un predicado de orden superior para abordar esto.

nicer(X) :- forall(between(1,X,Y), print(Y)).

Esto no solo es mucho más fácil de leer, sino que si se usara un objetivo que podría fallar en lugar de imprimir, su falla se detectaría correctamente y se transmitiría, mientras que las fallas de las metas en un bucle impulsado por fallas se confunden con el falla forzada que impulsa el bucle.

Visual Prolog tiene un azúcar sintáctico personalizado para estos bucles, combinado con predicados de función (ver más abajo):

vploop(X) :- foreach Y = std::fromTo(1,X) do
                 console::write(X)
             end foreach.

Aunque esto parece un bucle imperativo for, aún sigue las reglas de Prolog: en particular, cada iteración de foreach es su propio alcance.

Predicados de estilo de función #

Tradicionalmente en Prolog, las “funciones” (con una salida y entradas enlazadas) se escribían como predicados regulares:

mangle(X,Y) :- Y is (X*5)+2.

Esto puede crear la dificultad de que si un predicado de estilo de función se llama varias veces, es necesario “conectar en cadena” las variables temporales.

multimangle(X,Y) :- mangle(X,A), mangle(A,B), mangle(B,Y).

En la mayoría de los prólogos, es posible evitar esto escribiendo un operador de infijo alternativo para usar en lugar de es, que expande las expresiones, incluida la función alternativa.

% Define the new infix operator
:- op(900, xfy, <-).

% Define our function in terms of the infix operator - note the cut to avoid
% the choice falling through
R <- mangle(X) :- R is (X*5)+2, !.

% To make the new operator compatible with is..
R <- X :-
    compound(X),            % If the input is a compound/function
    X =.. [OP, X2, X3],     % Deconstruct it
    R2 <- X2,               % Recurse to evaluate the arguments
    R3 <- X3,
    Expr =.. [OP, R2, R3],  % Rebuild a compound with the evaluated arguments
    R is Expr,              % And send it to is
    !.
R <- X :- R is X, !.        % If it's not a compound, just use is directly

Ahora podemos escribir:

multimangle(X,Y) :- X <- mangle(mangle(mangle(Y))).

Sin embargo, algunos Prologs modernos van más allá y ofrecen una sintaxis personalizada para este tipo de predicado. Por ejemplo, en Visual Prolog:

mangle(X) = Y :- Y = ((X*5)+2).
multimangle(X,Y) :- Y = mangle(mangle(mangle(X))).

Tenga en cuenta que el operador <- y el predicado de estilo funcional anterior todavía se comportan como relaciones: es legal que tengan puntos de elección y realicen unificación múltiple. En el primer ejemplo, evitamos esto usando cortes. En Visual Prolog, es normal usar la sintaxis funcional para las relaciones y los puntos de elección se crean de la manera normal; por ejemplo, el objetivo X = (std::fromTo(1,10))*10 tiene éxito con enlaces X =10, X=20, X=30, X=40, etc.

Declaraciones de flujo/modo #

Al programar en Prolog, no siempre es posible, o deseable, crear predicados que unifiquen todas las combinaciones posibles de parámetros. Por ejemplo, el predicado entre(X,Y,Z) que expresa que Z está numéricamente entre X e Y. Se implementa fácilmente en los casos en los que X, Y y Z están vinculados (ya sea que Z esté entre X y Y o no lo es), o donde X e Y están ligados y Z está libre (Z se unifica con todos los números entre X e Y, o el predicado falla si Y<X); pero en otros casos, como cuando X y Z están ligados e Y está libre, hay potencialmente un número infinito de unificaciones. Aunque esto se puede implementar, por lo general no lo sería.

Declaración de flujo o declaraciones de modo permiten una descripción explícita de cómo se comportan los predicados cuando se les llama con diferentes combinaciones de parámetros vinculados. En el caso de entre, la declaración sería:

%! between(+X,+Y,+Z) is semidet.
%! between(+X,+Y,-Z) is nondet.

Cada línea especifica un patrón de llamada potencial para el predicado. Cada argumento está decorado con + para indicar los casos en los que está enlazado, o - para indicar los casos en los que no lo está (también hay otras decoraciones disponibles para tipos más complejos, como tuplas o listas que pueden estar parcialmente enlazadas). La palabra clave después de is indica el comportamiento del predicado en ese caso, y puede ser uno de estos:

  • det si el predicado siempre tiene éxito sin punto de elección. Por ejemplo, add(+X,+Y,-Z) es det porque sumar dos números dados X e Y siempre tendrá exactamente una respuesta.
  • semidet si el predicado tiene éxito o falla, sin punto de elección. Como arriba, entre(+X,+Y,+Z) es semidet porque Z está entre X e Y o no lo está.
  • multi si el predicado siempre tiene éxito, pero puede tener puntos de elección (pero también puede que no). Por ejemplo, factor(+X,-Y) sería multi porque un número siempre tiene al menos un factor, él mismo, pero puede tener más.
  • nondet si el predicado puede tener éxito con los puntos de elección o fallar. Por ejemplo, entre(+X,+Y,-Z) es nondet porque puede haber varias unificaciones posibles de Z a números entre X e Y, o si Y<X entonces no hay números entre ellos y el el predicado falla.

Las declaraciones de flujo/modo también se pueden combinar con el etiquetado de argumentos para aclarar qué significan los términos, o con tipeo. Por ejemplo, entre(+Desde:Int, +Hasta:Int, +Mid:Int) es semidet.

En Prologs puros, las declaraciones de flujo y modo son opcionales y solo se usan para la generación de documentación, pero pueden ser extremadamente útiles para ayudar a los programadores a identificar la causa de los errores de creación de instancias.

En Mercury, las declaraciones (y tipos) de flujo y modo son obligatorias y son validadas por el compilador. La sintaxis utilizada es la anterior.

En Visual Prolog, las declaraciones y los tipos de flujo y modo también son obligatorios y la sintaxis es diferente. La declaración anterior se escribiría como:

between : (int From, int To, int Mid) determ (i,i,i) nondeterm (i,i,o).

El significado es el mismo que el anterior, pero con las diferencias que:

  • Las declaraciones de flujo/modo están separadas de las declaraciones de tipo (ya que se supone que el flujo/modo para un solo predicado no variará con la sobrecarga de tipos);
  • i y o se usan para + y - y se combinan con los parámetros según el orden;
  • Los términos utilizados son diferentes. det se convierte en procedimiento, semidet se convierte en determ y nondet se convierte en nondeterm (multi sigue siendo multi).
