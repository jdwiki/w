Eliminación #

Una vez creada la suscripción, es importante gestionar su correcta desasignación.

Los doctores nos dijeron que

Si una secuencia termina en un tiempo finito, no llamar a dispose o no usar addDisposableTo(disposeBag) no causará pérdidas de recursos permanentes. Sin embargo, esos recursos se utilizarán hasta que se complete la secuencia, ya sea finalizando la producción de elementos o devolviendo un error.

Hay dos formas de desasignar recursos.

  1. Usando el operador disposeBags y addDisposableTo.
  2. Usando el operador takeUntil.

En el primer caso, pasa manualmente la suscripción al objeto DisposeBag, que borra correctamente toda la memoria tomada.

let bag = DisposeBag()
Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(bag)

En realidad, no necesita crear DisposeBags en cada clase que cree, solo eche un vistazo al proyecto de RxSwift Community llamado NSObject+Rx. Usando el marco, el código anterior se puede reescribir de la siguiente manera:

Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(rx_disposeBag)

En el segundo caso, si el tiempo de suscripción coincide con el tiempo de vida del objeto self, es posible implementar la disposición usando takeUntil(rx_deallocated):

let _ = sequence
    .takeUntil(rx_deallocated)
    .subscribe {
        print($0)
    }

Conceptos básicos de RxSwift #

FRP, o Programación Reactiva Funcional, tiene algunos términos básicos que debe conocer.

Cada pieza de datos se puede representar como ‘Observable’, que es un flujo de datos asíncrono. El poder de FRP está en la representación de eventos síncronos y asíncronos como flujos, Observabless, y en proporcionar la misma interfaz para trabajar con ellos.

Por lo general, Observable contiene varios (o ninguno) eventos que contienen la fecha: eventos .Next, y luego se puede terminar con éxito (.Success) o con un error (.Error).

Echemos un vistazo al siguiente diagrama de mármol:

--(1)--(2)--(3)|-->

En este ejemplo hay un flujo de valores Int. A medida que avanza el tiempo, ocurrieron tres eventos .Next y luego la transmisión finalizó con éxito.

--X->

El diagrama anterior muestra un caso en el que no se emitieron datos y el evento ‘.Error’ finaliza el ‘Observable’.

Antes de continuar, hay algunos recursos útiles:

  1. RxSwift. Mire ejemplos, lea documentos y comience.
  2. RxSwift Slack room tiene algunos canales para la resolución de problemas educativos.
  3. Juega con RxCanicas para saber qué operador hace y cuál es el más útil en tu caso.
  4. Echa un vistazo en este ejemplo, explora el código por ti mismo.

Creando observables #

RxSwift ofrece muchas formas de crear un Observable, echemos un vistazo:

import RxSwift

let intObservale = Observable.just(123) // Observable<Int>
let stringObservale = Observable.just("RxSwift") // Observable<String>
let doubleObservale = Observable.just(3.14) // Observable<Double>

Entonces, se crean los observables. Tienen solo un valor y luego terminan con éxito. Sin embargo, nada sucedió después de su creación. ¿Por qué?

Hay dos pasos para trabajar con Observables: observas algo para crear una transmisión y luego suscribes a la transmisión o la vinculas a algo para interactuar con ella.

Observable.just(12).subscribe {
    print($0)
}

La consola imprimirá:

.Next(12)
.Completed()

Y si solo me interesara trabajar con datos, que tienen lugar en eventos .Next, usaría el operador subscribeNext:

Observable.just(12).subscribeNext {
    print($0) // prints "12" now
}

Si quiero crear un observable de muchos valores, uso diferentes operadores:

Observable.of(1,2,3,4,5).subscribeNext {
    print($0)
}
// 1
// 2
// 3
// 4
// 5

// I can represent existing data types as Observables also:
[1,2,3,4,5].asObservable().subscribeNext { 
    print($0) 
}
// result is the same as before.

Y finalmente, tal vez quiera un ‘Observable’ que funcione. Por ejemplo, es conveniente envolver una operación de red en Observable<SomeResultType>. Echemos un vistazo a cómo se puede lograr esto:

Observable.create { observer in    // create an Observable ...
    MyNetworkService.doSomeWorkWithCompletion { (result, error) in
        if let e = error {
            observer.onError(e)    // ..that either holds an error
        } else {
            observer.onNext(result) // ..or emits the data
            observer.onCompleted()  // ..and terminates successfully.
        }
    }
    return NopDisposable.instance // here you can manually free any resources
                                //in case if this observable is being disposed.
}

Enlaces #

Observable.combineLatest(firstName.rx_text, lastName.rx_text) { $0 + " " + $1 }
.map { "Greetings, \($0)" }
.bindTo(greetingLabel.rx_text)

Usando el operador combineLatest cada vez que cualquiera de los dos Observables emite un elemento, combine el último elemento emitido por cada Observable. Entonces, de esta manera, combinamos el resultado de los dos UITextField creando un nuevo mensaje con el texto "Saludos, \($0)" usando la interpolación de cadenas para luego enlazar con el texto de un UILabel.

Podemos vincular datos a cualquier UITableView y UICollectionView de una manera muy sencilla:

viewModel
.rows
.bindTo(resultsTableView.rx_itemsWithCellIdentifier("WikipediaSearchCell", cellType: WikipediaSearchCell.self)) { (_, viewModel, cell) in
    cell.title = viewModel.title
    cell.url = viewModel.url
}
.addDisposableTo(disposeBag)

Eso es un envoltorio Rx alrededor del método de fuente de datos cellForRowAtIndexPath. Y también Rx se ocupa de la implementación de numberOfRowsAtIndexPath, que es un método requerido en un sentido tradicional, pero no es necesario que lo implemente aquí, ya está resuelto.

##RxCocoa y ControlEvents RxSwift proporciona no solo las formas de controlar sus datos, sino también representar las acciones del usuario de manera reactiva.

RxCocoa contiene todo lo que necesitas. Envuelve la mayoría de las propiedades de los componentes de la interfaz de usuario en Observables, pero no realmente. Hay algunos ‘Observables’ actualizados llamados ‘ControlEvent’ (que representan eventos) y ‘ControlProperties’ (que representan propiedades, ¡sorpresa!). Estas cosas contienen flujos Observables debajo del capó, pero también tienen algunos matices:

  • Nunca falla, por lo que no hay errores.
  • Se Completará la secuencia cuando se desasigne el control.
  • Entrega eventos en el hilo principal (MainScheduler.instance).

Básicamente, puedes trabajar con ellos como de costumbre:

button.rx_tap.subscribeNext { _ in   // control event
    print("User tapped the button!")
}.addDisposableTo(bag)

textField.rx_text.subscribeNext { text in // control property
    print("The textfield contains: \(text)")
}.addDisposableTo(bag)
// notice that ControlProperty generates .Next event on subscription
// In this case, the log will display 
// "The textfield contains: "
// at the very start of the app.

Esto es muy importante de usar: siempre que use Rx, olvídese de las cosas @IBAction, todo lo que necesita puede enlazar y configurar a la vez. Por ejemplo, el método viewDidLoad de su controlador de vista es un buen candidato para describir cómo funcionan los componentes de la interfaz de usuario.

Ok, otro ejemplo: supongamos que tenemos un campo de texto, un botón y una etiqueta. Queremos validar el texto en el campo de texto cuando toquemos el botón y mostrar los resultados en la etiqueta. Sí, parece otra tarea de validación de correo electrónico, ¿eh?

En primer lugar, tomamos el button.rx_tap ControlEvent:

----()-----------------------()----->

Aquí, los paréntesis vacíos muestran los toques de los usuarios. A continuación, tomamos lo que está escrito en el campo de texto con el operador withLatestFrom (míralo aquí, imagina que el flujo superior representa los toques del usuario, el inferior representa el texto en el campo de texto).

button.rx_tap.withLatestFrom(textField.rx_text)

----("")--------------------("123")--->
//  ^ tap   ^ i wrote 123    ^ tap

Genial, tenemos un flujo de cadenas para validar, emitido solo cuando necesitamos validar.

Cualquier ‘Observable’ tiene operadores familiares como ‘mapa’ o ‘filtro’, tomaremos ‘mapa’ para validar el texto. Cree la función validateEmail usted mismo, use cualquier expresión regular que desee.

button.rx_tap                                // ControlEvent<Void>
        .withLatestFrom(textField.rx_text)   // Observable<String>
        .map(validateEmail)                  // Observable<Bool>
        .map { (isCorrect) in
            return isCorrect ? "Email is correct" : "Input the correct one, please"
        }                                    // Observable<String>
        .bindTo(label.rx_text)              
        .addDisposableTo(bag)

¡Hecho! Si necesita más lógica personalizada (como mostrar vistas de error en caso de error, hacer una transición a otra pantalla en caso de éxito…), simplemente suscríbase a la transmisión Bool final y escríbala allí.
