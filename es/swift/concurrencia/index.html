Sintaxis #

  • Rápido 3.0

  • DispatchQueue.main // Obtener la cola principal

  • DispatchQueue(label: “my-serial-queue”, atributos: [.serial, .qosBackground]) // Crea tu propia cola serial privada

  • DispatchQueue.global(attributes: [.qosDefault]) // Accede a una de las colas concurrentes globales

  • DispatchQueue.main.async { … } // Envía una tarea de forma asíncrona al subproceso principal

  • DispatchQueue.main.sync { … } // Enviar una tarea sincrónicamente al subproceso principal

  • DispatchQueue.main.asyncAfter(fecha límite: .now() + 3) { … } // Envía una tarea de forma asíncrona al subproceso principal para que se ejecute después de x segundos

  • Rápido <3.0

  • dispatch_get_main_queue() // Obtiene la cola principal ejecutándose en el hilo principal

  • dispatch_get_global_queue(dispatch_queue_priority_t, 0) // Obtener cola global con prioridad especificada dispatch_queue_priority_t

  • dispatch_async(dispatch_queue_t) { () -> Void in… } // Enviar una tarea de forma asíncrona en el dispatch_queue_t especificado

  • dispatch_sync(dispatch_queue_t) { () -> Void in… } // Distribuir una tarea sincrónicamente en el dispatch_queue_t especificado

  • dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(nanosegundos)), dispatch_queue_t, { … }); // Enviar una tarea en el dispatch_queue_t especificado después de nanosegundos

Obtener una cola de Grand Central Dispatch (GCD) #

Grand Central Dispatch trabaja sobre el concepto de “Colas de despacho”. Una cola de despacho ejecuta las tareas que usted designe en el orden en que se pasan. Hay tres tipos de colas de despacho:

  • Colas de envío en serie (también conocidas como colas de envío privadas) ejecutan una tarea a la vez, en orden. Se utilizan con frecuencia para sincronizar el acceso a un recurso.
  • Colas de envío simultáneas (también conocidas como colas de envío globales) ejecutan una o más tareas al mismo tiempo.
  • La Cola de envío principal ejecuta tareas en el subproceso principal.

Para acceder a la cola principal:

let mainQueue = DispatchQueue.main

let mainQueue = dispatch_get_main_queue()

El sistema proporciona colas de envío globales concurrentes (globales para su aplicación), con diferentes prioridades. Puede acceder a estas colas usando la clase DispatchQueue en Swift 3:

let globalConcurrentQueue = DispatchQueue.global(qos: .default)

equivalente a

let globalConcurrentQueue = DispatchQueue.global()

let globalConcurrentQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0)

En iOS 8 o posterior, los posibles valores de calidad de servicio que se pueden pasar son .userInteractive, .userInitiated, .default, .utility y .background. Estas reemplazan las constantes DISPATCH_QUEUE_PRIORITY_.

También puede crear sus propias colas con distintas prioridades:

let myConcurrentQueue = DispatchQueue(label: "my-concurrent-queue", qos: .userInitiated, attributes: [.concurrent], autoreleaseFrequency: .workItem, target: nil)
let mySerialQueue = DispatchQueue(label: "my-serial-queue", qos: .background, attributes: [], autoreleaseFrequency: .workItem, target: nil)

let myConcurrentQueue = dispatch_queue_create("my-concurrent-queue", DISPATCH_QUEUE_CONCURRENT)
let mySerialQueue = dispatch_queue_create("my-serial-queue", DISPATCH_QUEUE_SERIAL)

En Swift 3, las colas creadas con este inicializador son en serie de forma predeterminada, y pasar .workItem para la frecuencia de liberación automática garantiza que se cree y se vacíe un grupo de liberación automática para cada elemento de trabajo. También está .never, lo que significa que usted mismo administrará sus propios grupos de liberación automática, o .inherit, que hereda la configuración del entorno. En la mayoría de los casos probablemente no usará .never excepto en casos de personalización extrema.

Bucles simultáneos #

GCD proporciona un mecanismo para realizar un bucle, mediante el cual los bucles ocurren simultáneamente entre sí. Esto es muy útil cuando se realizan una serie de cálculos computacionalmente costosos.

Considere este bucle:

for index in 0 ..< iterations {
    // Do something computationally expensive here
}

Puede realizar esos cálculos simultáneamente usando concurrentPerform (en Swift 3) o dispatch_apply (en Swift 2):

DispatchQueue.concurrentPerform(iterations: iterations) { index in
    // Do something computationally expensive here
}

dispatch_apply(iterations, queue) { index in
    // Do something computationally expensive here
}

El cierre del ciclo se invocará para cada índice desde 0 hasta, pero sin incluir, iteraciones. Estas iteraciones se ejecutarán simultáneamente entre sí y, por lo tanto, no se garantiza el orden en que se ejecutan. La cantidad real de iteraciones que ocurren simultáneamente en un momento dado generalmente está dictada por las capacidades del dispositivo en cuestión (por ejemplo, cuántos núcleos tiene el dispositivo).

Un par de consideraciones especiales:

  • El concurrentPerform/dispatch_apply puede ejecutar los bucles concurrentemente entre sí, pero todo esto sucede sincrónicamente con respecto al subproceso desde el que lo llama. Por lo tanto, no llame a esto desde el subproceso principal, ya que esto bloqueará ese subproceso hasta que finalice el ciclo.

  • Debido a que estos bucles ocurren simultáneamente entre sí, usted es responsable de garantizar la seguridad de subprocesos de los resultados. Por ejemplo, si actualiza algún diccionario con los resultados de estos cálculos computacionalmente costosos, asegúrese de sincronizar esas actualizaciones usted mismo.

  • Tenga en cuenta que hay algunos gastos generales asociados con la ejecución de bucles simultáneos. Por lo tanto, si los cálculos que se realizan dentro del ciclo no son lo suficientemente intensivos desde el punto de vista computacional, es posible que el rendimiento obtenido mediante el uso de ciclos simultáneos se vea disminuido, si no completamente compensado, por la sobrecarga asociada con la sincronización de todos estos subprocesos simultáneos.

    So, you are responsible determining the correct amount of work to be performed in each iteration of the loop. If the calculations are too simple, you may employ “striding” to include more work per loop. For example, rather than doing a concurrent loop with 1 million trivial calculations, you may do 100 iterations in your loop, doing 10,000 calculations per loop. That way there is enough work being performed on each thread, so the overhead associated with managing these concurrent loops becomes less significant.

Ejecutar tareas en una cola de Grand Central Dispatch (GCD) #

Para ejecutar tareas en una cola de despacho, use los métodos sync, async y after.

Para enviar una tarea a una cola de forma asíncrona:

let queue = DispatchQueue(label: "myQueueName")

queue.async {
    //do something
    
    DispatchQueue.main.async {
        //this will be called in main thread
        //any UI updates should be placed here
    }
}
// ... code here will execute immediately, before the task finished

Para enviar una tarea a una cola sincrónicamente:

queue.sync {
    // Do some task
}
// ... code here will not execute until the task is finished

Para enviar una tarea a una cola después de una cierta cantidad de segundos:

queue.asyncAfter(deadline: .now() + 3) {
    //this will be executed in a background-thread after 3 seconds
}
// ... code here will execute immediately, before the task finished

NOTA: ¡Cualquier actualización de la interfaz de usuario debe llamarse en el hilo principal! Asegúrese de colocar el código para las actualizaciones de la interfaz de usuario dentro de DispatchQueue.main.async { ... }

Tipos de cola:

let mainQueue = dispatch_get_main_queue()
let highQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_HIGH, 0)
let backgroundQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0)

Para enviar una tarea a una cola de forma asíncrona:

dispatch_async(queue) {
    // Your code run run asynchronously. Code is queued and executed 
    // at some point in the future.
}
// Code after the async block will execute immediately

Para enviar una tarea a una cola sincrónicamente:

dispatch_sync(queue) {
    // Your sync code
}
// Code after the sync block will wait until the sync task finished

Para enviar una tarea después de un intervalo de tiempo (use NSEC_PER_SEC para convertir segundos a nanosegundos):

dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(2.5 * Double(NSEC_PER_SEC))), dispatch_get_main_queue()) {
    // Code to be performed in 2.5 seconds here
}

Para ejecutar una tarea de forma asíncrona y luego actualizar la interfaz de usuario:

dispatch_async(queue) {
    // Your time consuming code here
    dispatch_async(dispatch_get_main_queue()) {
        // Update the UI code 
    }
}

NOTA: ¡Cualquier actualización de la interfaz de usuario debe llamarse en el hilo principal! Asegúrese de colocar el código para las actualizaciones de la interfaz de usuario dentro de dispatch_async(dispatch_get_main_queue()) { ... }

Ejecutar tareas en una OperationQueue #

Puede pensar en una OperationQueue como una línea de tareas que esperan ser ejecutadas. A diferencia de las colas de despacho en GCD, las colas de operaciones no son FIFO (primero en entrar, primero en salir). En cambio, ejecutan tareas tan pronto como están listas para ser ejecutadas, siempre que haya suficientes recursos del sistema para permitirlo.

Obtén la OperationQueue principal:

let mainQueue = OperationQueue.main

Cree una OperationQueue personalizada:

let queue = OperationQueue()
queue.name = "My Queue"
queue.qualityOfService = .default

La calidad de servicio especifica la importancia del trabajo, o cuánto es probable que el usuario cuente con los resultados inmediatos de la tarea.

Agregue una Operación a una OperationQueue:

// An instance of some Operation subclass
let operation = BlockOperation {
    // perform task here
}

queue.addOperation(operation)

Agrega un bloque a una OperationQueue:

myQueue.addOperation {
    // some task
}

Agrega múltiples Operacioness a una OperationQueue:

let operations = [Operation]()
// Fill array with Operations

myQueue.addOperation(operations)

Ajuste cuántas ‘Operaciones’ se pueden ejecutar simultáneamente dentro de la cola:

myQueue.maxConcurrentOperationCount = 3 // 3 operations may execute at once

// Sets number of concurrent operations based on current system conditions
myQueue.maxConcurrentOperationCount = NSOperationQueueDefaultMaxConcurrentOperationCount

Suspender una cola evitará que comience la ejecución de cualquier operación existente no iniciada o de cualquier operación nueva agregada a la cola. La forma de reanudar esta cola es establecer isSuspended de nuevo en false:

myQueue.isSuspended = true

// Re-enable execution
myQueue.isSuspended = false

Suspender una OperationQueue no detiene ni cancela las operaciones que ya se están ejecutando. Solo debe intentar suspender una cola que haya creado, no las colas globales o la cola principal.

Creación de operaciones de alto nivel #

El marco Foundation proporciona el tipo Operación, que representa un objeto de alto nivel que encapsula una parte del trabajo que puede ejecutarse en una cola. La cola no solo coordina el rendimiento de esas operaciones, sino que también puede establecer dependencias entre operaciones, crear operaciones cancelables, restringir el grado de simultaneidad empleado por la cola de operaciones, etc.

Las ‘Operaciones’ están listas para ejecutarse cuando todas sus dependencias terminan de ejecutarse. La propiedad isReady luego cambia a true.

Cree una subclase Operación simple no concurrente:

class MyOperation: Operation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

class MyOperation: NSOperation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

Agrega una operación a una OperationQueue:

myQueue.addOperation(operation)

Esto ejecutará la operación simultáneamente en la cola.

Gestionar dependencias en una Operación.

Las dependencias definen otras Operaciones que deben ejecutarse en una cola antes de que esa Operación se considere lista para ejecutarse.

operation2.addDependency(operation1)

operation2.removeDependency(operation1)

Ejecute una Operación sin una cola:

   operation.start()

Las dependencias serán ignoradas. Si se trata de una operación concurrente, la tarea aún puede ejecutarse concurrentemente si su método start descarga el trabajo a las colas en segundo plano.

Operaciones Concurrentes.

Si la tarea que debe realizar una Operación es, en sí misma, asíncrona (por ejemplo, una tarea de datos URLSession), debe implementar la Operación como una operación concurrente. En este caso, su implementación isAsynchronous debería devolver true, generalmente tendría un método start que realiza alguna configuración, luego llama a su método main que realmente ejecuta la tarea.

Cuando comienza la implementación de una Operación asíncrona, debe implementar los métodos isExecuting, isFinished y KVO. Entonces, cuando comienza la ejecución, la propiedad isExecuting cambia a true. Cuando una Operación finaliza su tarea, isExecuting se establece en falso y isFinished se establece en verdadero. Si la operación se cancela, tanto isCancelled como isFinished cambian a true. Todas estas propiedades son clave-valor observables.

Cancelar una Operación.

Llamar a cancel simplemente cambia la propiedad isCancelled a true. Para responder a la cancelación desde su propia subclase Operación, debe verificar el valor de isCancelled al menos periódicamente dentro de main y responder adecuadamente.

operation.cancel()
