Conceptos básicos del manejo de errores #

Las funciones en Swift pueden devolver valores, lanzar errores, o ambos:

func reticulateSplines() // sin valor de retorno y sin error
func reticulateSplines() -> Int // siempre devuelve un valor
func reticulateSplines() lanza // sin valor de retorno, pero puede arrojar un error
func reticulateSplines() lanza -> Int // puede devolver un valor o arrojar un error

Cualquier valor que se ajuste al [protocolo de tipo de error] (https://developer.apple.com/library/ios/documentation/Swift/Reference/Swift_ErrorType_Protocol/index.html) (incluidos los objetos NSError) puede generarse como un error. Enumeraciones proporcionan una manera conveniente de definir errores personalizados:

enum NetworkError: ErrorType {
    case Offline
    case ServerError(String)
}

enum NetworkError: Error {
    // Swift 3 dictates that enum cases should be `lowerCamelCase`
    case offline
    case serverError(String)
}

Un error indica una falla no fatal durante la ejecución del programa y se maneja con las construcciones de flujo de control especializadas do/catch, throw y try.

func fetchResource(recurso: NSURL) lanza -> Cadena {
    if let (statusCode, responseString) = /* ...from elsewhere...*/ {
        if case 500..<600 = statusCode {
            throw NetworkError.serverError(responseString)
        } else {
            return responseString
        }
    } else {
        throw NetworkError.offline
    }
}

Los errores se pueden detectar con do/catch:

hacer {
    let response = try fetchResource(resURL)
    // If fetchResource() didn't throw an error, execution continues here:
    print("Got response: \(response)")
    ...
} atrapar {
    // If an error is thrown, we can handle it here.
    print("Whoops, couldn't fetch resource: \(error)")
}

Cualquier función que pueda arrojar un error debe ser llamada usando try, try?, o try!:

// error: la llamada puede lanzarse pero no está marcada con 'intentar'
let respuesta = fetchResource(resURL)

// "intentar" funciona dentro de do/catch, o dentro de otra función de lanzamiento:
hacer {
    let response = try fetchResource(resURL)
} atrapar {
    // Handle the error
}

func foo() lanza {
    // If an error is thrown, continue passing it up to the caller.
    let response = try fetchResource(resURL)
}

// "¿probar?" envuelve el valor de retorno de la función en un Opcional (nil si se produjo un error).
if dejar respuesta = intentar? fetchResource(resURL) {
    // no error was thrown
}

// "¡probar!" bloquea el programa en tiempo de ejecución si se produce un error.
let respuesta = ¡intentar! fetchResource(resURL)

Detectar diferentes tipos de errores #

Vamos a crear nuestro propio tipo de error para este ejemplo.

enum CustomError: ErrorType {
    case SomeError
    case AnotherError
}

func throwing() throws {
    throw CustomError.SomeError
}

enum CustomError: Error {
    case someError
    case anotherError
}

func throwing() throws {
    throw CustomError.someError
}

La sintaxis de Do-Catch permite capturar un error lanzado, y automáticamente crea una constante llamada error disponible en el bloque catch:

do {
    try throwing()
} catch {
    print(error)
}

También puede declarar una variable usted mismo:

do {
    try throwing()
} catch let oops {
    print(oops)
}

También es posible encadenar diferentes sentencias catch. Esto es conveniente si se pueden generar varios tipos de errores en el bloque Do.

Aquí Do-Catch primero intentará convertir el error como CustomError, luego como NSError si el tipo personalizado no coincide.

do {
    try somethingMayThrow()
} catch let custom as CustomError {
    print(custom)
} catch let error as NSError {
    print(error)
}

En Swift 3, no es necesario cambiar explícitamente a NSError.

do {
    try somethingMayThrow()
} catch let custom as CustomError {
    print(custom)
} catch {
    print(error)
}

Patrón de captura y cambio para el manejo explícito de errores #

class Plane {
    
    enum Emergency: ErrorType {
        case NoFuel
        case EngineFailure(reason: String)
        case DamagedWing
    }

    var fuelInKilograms: Int

    //... init and other methods not shown

    func fly() throws {
        // ...
        if fuelInKilograms <= 0 {
            // uh oh...
            throw Emergency.NoFuel
        }
    }

}

En la clase de cliente:

let airforceOne = Plane()
do {
    try airforceOne.fly()
} catch let emergency as Plane.Emergency {
    switch emergency {
    case .NoFuel:
        // call nearest airport for emergency landing
    case .EngineFailure(let reason):
        print(reason) // let the mechanic know the reason
    case .DamagedWing:
        // Assess the damage and determine if the president can make it
    }
}

Deshabilitar la propagación de errores #

Los creadores de Swift han puesto mucha atención en hacer que el lenguaje sea expresivo y el manejo de errores es exactamente eso, expresivo. Si intenta invocar una función que puede arrojar un error, la llamada a la función debe estar precedida por la palabra clave try. La palabra clave de prueba no es mágica. Todo lo que hace es hacer que el desarrollador sea consciente de la capacidad de lanzamiento de la función.

Por ejemplo, el siguiente código usa una función loadImage(atPath:), que carga el recurso de la imagen en una ruta determinada o genera un error si la imagen no se puede cargar. En este caso, debido a que la imagen se envía con la aplicación, no se generará ningún error en el tiempo de ejecución, por lo que es apropiado deshabilitar la propagación de errores.

let photo = try! loadImage(atPath: "./Resources/John Appleseed.jpg")

Crear error personalizado con descripción localizada #

Crear enum de errores personalizados

enum RegistrationError: Error {
    case invalidEmail
    case invalidPassword
    case invalidPhoneNumber
}

Cree una extensión de RegistrationError para manejar la descripción localizada.

extension RegistrationError: LocalizedError {
    public var errorDescription: String? {
        switch self {
        case .invalidEmail:
            return NSLocalizedString("Description of invalid email address", comment: "Invalid Email")
        case .invalidPassword:
            return NSLocalizedString("Description of invalid password", comment: "Invalid Password")
        case .invalidPhoneNumber:
            return NSLocalizedString("Description of invalid phoneNumber", comment: "Invalid Phone Number")
        }
    }
}

Manejar error:

let error: Error = RegistrationError.invalidEmail
print(error.localizedDescription)
