Los protocolos son una forma de especificar cómo usar un objeto. Describen un conjunto de propiedades y métodos que debe proporcionar una clase, estructura o enumeración, aunque los protocolos no imponen restricciones en la implementación.

Un protocolo Swift es una colección de requisitos que deben implementar los tipos conformes. Luego, el protocolo se puede usar en la mayoría de los lugares donde se espera un tipo, por ejemplo, matrices y requisitos genéricos.

Los miembros del protocolo siempre comparten el mismo calificador de acceso que el protocolo completo y no se pueden especificar por separado. Aunque un protocolo podría restringir el acceso con requisitos de getter o setter, como en los ejemplos anteriores.

Para obtener más información acerca de los protocolos, consulte El lenguaje de programación Swift.

[Los protocolos de Objective-C] (https://developer.apple.com/library/ios/documentation/Cocoa/Conceptual/ProgrammingWithObjectiveC/WorkingwithProtocols/WorkingwithProtocols.html) son similares a los protocolos de Swift.

Los protocolos también son comparables a las [interfaces de Java] (https://docs.oracle.com/javase/tutorial/java/IandI/createinterface.html).

Principios básicos del protocolo #

Acerca de los protocolos #

Un protocolo especifica inicializadores, propiedades, funciones, subíndices y tipos asociados requeridos de un tipo de objeto Swift (clase, estructura o enumeración) conforme al protocolo. En algunos idiomas, las ideas similares para las especificaciones de requisitos de objetos posteriores se conocen como “interfaces”.

Un protocolo declarado y definido es un tipo, en sí mismo, con una firma de sus requisitos establecidos, algo similar a la forma en que las funciones de Swift son un tipo en función de su firma de parámetros y retornos.

Las especificaciones del protocolo Swift pueden ser opcionales, explícitamente requeridas y/o dadas implementaciones predeterminadas a través de una instalación conocida como extensiones de protocolo. Un tipo de objeto Swift (clase, estructura o enumeración) que desee cumplir con un protocolo desarrollado con extensiones para todos sus requisitos específicos solo necesita indicar su deseo de cumplir para estar en total conformidad. La facilidad de implementación por defecto de Extensiones de Protocolo puede ser suficiente para cumplir con todas las obligaciones de conformidad con un Protocolo.

Los protocolos pueden ser heredados por otros protocolos. Esto, junto con las extensiones de protocolo, significa que los protocolos pueden y deben considerarse como una característica importante de Swift.

Los protocolos y las extensiones son importantes para lograr los objetivos y enfoques más amplios de Swift para la flexibilidad del diseño del programa y los procesos de desarrollo. El propósito principal declarado de la capacidad de extensión y protocolo de Swift es facilitar el diseño compositivo en la arquitectura y el desarrollo del programa. Esto se conoce como Programación Orientada a Protocolos. Los veteranos crujientes consideran que esto es superior a un enfoque en el diseño OOP.

Protocolos define interfaces que pueden ser implementadas por cualquier estructura, clase o enum:

protocol MyProtocol {
    init(value: Int)                      // required initializer
    func doSomething() -> Bool            // instance method
    var message: String { get }           // instance read-only property
    var value: Int { get set }            // read-write instance property
    subscript(index: Int) -> Int { get }  // instance subscript
    static func instructions() -> String  // static method
    static var max: Int { get }           // static read-only property
    static var total: Int { get set }     // read-write static property
}

Las propiedades definidas en los protocolos deben anotarse como { get } o { get set }. { get } significa que la propiedad debe poder obtenerse y, por lo tanto, puede implementarse como cualquier tipo de propiedad. { get set } significa que la propiedad debe ser tanto configurable como obtenible.

Una estructura, clase o enumeración puede conforme a un protocolo:

struct MiEstructura : MiProtocolo {
    // Implement the protocol's requirements here
}
class MiClase : MiProtocolo {
    // Implement the protocol's requirements here
}
enum MiEnum : MiProtocolo {
    case caseA, caseB, caseC
    // Implement the protocol's requirements here
}

Un protocolo también puede definir una implementación predeterminada para cualquiera de sus requisitos a través de una extensión:

extension MyProtocol {
    
    // default implementation of doSomething() -> Bool
    // conforming types will use this implementation if they don't define their own
    func doSomething() -> Bool {
        print("do something!")
        return true
    }
}

Un protocolo se puede usar como un tipo, siempre que no tenga [requisitos de tipo asociado][7]:

func hacerCosas(objeto: MiProtocolo) {
    // All of MyProtocol's requirements are available on the object
    print(object.message)
    print(object.doSomething())
}

let elementos: [MiProtocolo] = [MiEstructura(), MiClase(), MiEnum.caseA]

También puede definir un tipo abstracto que se ajuste a múltiples protocolos:

Con Swift 3 o superior, esto se hace separando la lista de protocolos con un ampersand (&):

func doStuff(objeto: MiProtocolo & OtroProtocolo) {
    // ...
}

let items : [MyProtocol & AnotherProtocol] = [MyStruct(), MyClass(), MyEnum.caseA]

Las versiones anteriores tienen la sintaxis protocolo<...> donde los protocolos son una lista separada por comas entre corchetes angulares <>.

protocolo Otro Protocolo {
    func doSomethingElse()
}

func hacerCosas(objeto: protocolo<MiProtocolo, OtroProtocolo>) {
    
    // All of MyProtocol & AnotherProtocol's requirements are available on the object
    print(object.message)
    object.doSomethingElse()
}

// MyStruct, MyClass y MyEnum ahora deben cumplir con MyProtocol y AnotherProtocol
let elementos: [protocolo<MiProtocolo, OtroProtocolo>] = [MiEstructura(), MiClase(), MiEnum.caseA]

Los tipos existentes se pueden ampliar para ajustarse a un protocolo:

extension String : MyProtocol {
    // Implement any requirements which String doesn't already satisfy
}

[7]: https://www.wikiod.com/es/swift/protocolos#Requisitos de tipo asociados

Patrón delegado #

Un delegado es un patrón de diseño común utilizado en los marcos Cocoa y CocoaTouch, donde una clase delega la responsabilidad de implementar alguna funcionalidad a otra. Esto sigue un principio de separación de preocupaciones, donde la clase del marco implementa la funcionalidad genérica mientras que una instancia delegada separada implementa el caso de uso específico.

Otra forma de ver el patrón de delegado es en términos de comunicación de objetos. Los ‘Objetos’ a menudo necesitan comunicarse entre sí y, para hacerlo, un objeto debe ajustarse a un ‘protocolo’ para convertirse en delegado de otro Objeto. Una vez que se ha realizado esta configuración, el otro objeto responde a sus delegados cuando suceden cosas interesantes.

Por ejemplo, una vista en la interfaz de usuario para mostrar una lista de datos debe ser responsable solo de la lógica de cómo se muestran los datos, no de decidir qué datos se deben mostrar.

Vamos a sumergirnos en un ejemplo más concreto. si tiene dos clases, un padre y un hijo:

class Parent { }
class Child { }

Y desea notificar al padre de un cambio del niño.

En Swift, los delegados se implementan mediante una declaración protocol, por lo que declararemos un protocolo que implementará el delegado. Aquí delegar es el objeto padre.

protocol ChildDelegate: class {
    func childDidSomething()
}

El niño necesita declarar una propiedad para almacenar la referencia al delegado:

class Child {
    weak var delegate: ChildDelegate?
}

Observe que la variable delegate es opcional y el protocolo ChildDelegate está marcado para ser implementado solo por tipo de clase (sin esto, la variable delegate no se puede declarar como una referencia débil evitando cualquier ciclo de retención. Esto significa que si ya no se hace referencia a la variable delegate en ningún otro lugar, se liberará). Esto es para que la clase principal solo registre el delegado cuando sea necesario y esté disponible.

Además, para marcar a nuestro delegado como “débil”, debemos restringir nuestro protocolo ChildDelegate a los tipos de referencia agregando la palabra clave “clase” en la declaración del protocolo.

En este ejemplo, cuando el niño hace algo y necesita notificar a su padre, el niño llamará:

delegate?.childDidSomething()

Si se ha definido el delegado, se notificará al delegado que el niño ha hecho algo.

La clase padre necesitará extender el protocolo ChildDelegate para poder responder a sus acciones. Esto se puede hacer directamente en la clase padre:

class Parent: ChildDelegate {
    ...

    func childDidSomething() {
        print("Yay!")
    }
}

O usando una extensión:

extension Parent: ChildDelegate {
    func childDidSomething() {
        print("Yay!")
    }
}

El padre también necesita decirle al niño que es el delegado del niño:

// In the parent
let child = Child()
child.delegate = self

Por defecto, un protocolo de Swift no permite implementar una función opcional. Estos solo se pueden especificar si su protocolo está marcado con el atributo @objc y el modificador opcional.

Por ejemplo, UITableView implementa el comportamiento genérico de una vista de tabla en iOS, pero el usuario debe implementar dos clases de delegado llamadas UITableViewDelegate y UITableViewDataSource que implementan cómo se ven y se comportan las celdas específicas.

@objc public protocol UITableViewDelegate : NSObjectProtocol, UIScrollViewDelegate {
        
        // Display customization
        optional public func tableView(tableView: UITableView, willDisplayCell cell: UITableViewCell, forRowAtIndexPath indexPath: NSIndexPath)
        optional public func tableView(tableView: UITableView, willDisplayHeaderView view: UIView, forSection section: Int)
        optional public func tableView(tableView: UITableView, willDisplayFooterView view: UIView, forSection section: Int)
        optional public func tableView(tableView: UITableView, didEndDisplayingCell cell: UITableViewCell, forRowAtIndexPath indexPath: NSIndexPath)
        ...
}

Puedes implementar este protocolo cambiando la definición de su clase, por ejemplo:

class MyViewController : UIViewController, UITableViewDelegate

Cualquier método que no esté marcado como opcional en la definición del protocolo (UITableViewDelegate en este caso) debe implementarse.

Requisitos de tipo asociado #

Los protocolos pueden definir requisitos de tipos asociados usando la palabra clave associatedtype:

contenedor de protocolo {
    associatedtype Element
    var count: Int { get }
    subscript(index: Int) -> Element { get set }
}

Los protocolos con requisitos de tipo asociados solo se pueden usar como restricciones genéricas:

// Estos NO están permitidos, porque el Contenedor tiene requisitos de tipo asociados:
func displayValues(contenedor: Contenedor) { ... }
clase MiClase { dejar contenedor: Contenedor }
// > error: el protocolo 'Contenedor' solo se puede usar como una restricción genérica
// > porque tiene requisitos de tipo propios o asociados

// Estos están permitidos:
func displayValues<T: Contenedor>(contenedor: T) { ... }
clase MiClase<T: Contenedor> { dejar contenedor: T }

Un tipo que se ajusta al protocolo puede satisfacer implícitamente un requisito de tipo asociado, al proporcionar un tipo dado donde el protocolo espera que aparezca el tipo asociado:

struct ContainerOfOne<T>: Contenedor {
    let count = 1          // satisfy the count requirement
    var value: T
    
    // satisfy the subscript associatedtype requirement implicitly,
    // by defining the subscript assignment/return type as T
    // therefore Swift will infer that T == Element
    subscript(index: Int) -> T {
        get {
            precondition(index == 0)
            return value
        }
        set {
            precondition(index == 0)
            value = newValue
        }
    }
}

let container = ContainerOfOne(valor: "Hola")

(Tenga en cuenta que para agregar claridad a este ejemplo, el tipo de marcador de posición genérico se denomina T; un nombre más adecuado sería Elemento, que sombrearía el Elemento de tipo asociado del protocolo. El compilador aún inferirá que el El marcador de posición genérico ‘Elemento’ se usa para satisfacer el requisito de ‘Elemento de tipo asociado’.)

Un tipo asociado también puede satisfacerse explícitamente mediante el uso de un typealias:

struct ContainerOfOne<T>: Contenedor {

    typealias Element = T
    subscript(index: Int) -> Element { ... }

    // ...
}

Lo mismo ocurre con las extensiones:

// Expose an 8-bit integer as a collection of boolean values (one for each bit).
extension UInt8: Container {

    // as noted above, this typealias can be inferred
    typealias Element = Bool

    var count: Int { return 8 }
    subscript(index: Int) -> Bool {
        get {
            precondition(0 <= index && index < 8)
            return self & 1 << UInt8(index) != 0
        }
        set {
            precondition(0 <= index && index < 8)
            if newValue {
                self |= 1 << UInt8(index)
            } else {
                self &= ~(1 << UInt8(index))
            }
        }
    }
}

Si el tipo conforme ya satisface el requisito, no se necesita implementación:

extension Array: Container {}  // Array satisfies all requirements, including Element

Protocolos solo de clase #

Un protocolo puede especificar que solo una clase puede implementarlo mediante el uso de la palabra clave clase en su lista de herencia. Esta palabra clave debe aparecer antes que cualquier otro protocolo heredado en esta lista.

protocolo ClassOnlyProtocol: clase, SomeOtherProtocol {
    // Protocol requirements 
}

Si un tipo que no es de clase intenta implementar ClassOnlyProtocol, se generará un error de compilación.

struct MyStruct: ClassOnlyProtocol { 
    // error: Non-class type 'MyStruct' cannot conform to class protocol 'ClassOnlyProtocol'  
}

Otros protocolos pueden heredar de ClassOnlyProtocol, pero tendrán el mismo requisito de solo clase.

protocol MyProtocol: ClassOnlyProtocol {
    // ClassOnlyProtocol Requirements
    // MyProtocol Requirements
}

class MySecondClass: MyProtocol {
    // ClassOnlyProtocol Requirements
    // MyProtocol Requirements
}

Semántica de referencia de protocolos solo de clase #

El uso de un protocolo de solo clase permite semántica de referencia cuando se desconoce el tipo conforme.

protocol Foo : class {
    var bar : String { get set }
}

func takesAFoo(foo:Foo) {

    // this assignment requires reference semantics,
    // as foo is a let constant in this scope.
    foo.bar = "new value"
}

En este ejemplo, como Foo es un protocolo solo de clase, la asignación a bar es válida ya que el compilador sabe que foo es un tipo de clase y, por lo tanto, tiene semántica de referencia.

Si Foo no fuera un protocolo solo de clase, se produciría un error de compilación, ya que el tipo conforme podría ser un tipo de valor, que requeriría una anotación var para ser mutable.

protocol Foo {
    var bar : String { get set }
}

func takesAFoo(foo:Foo) {
    foo.bar = "new value" // error: Cannot assign to property: 'foo' is a 'let' constant
}


func tomaAFoo(foo:Foo) {
    var foo = foo // mutable copy of foo
    foo.bar = "new value" // no error – satisfies both reference and value semantics
}

Variables débiles del tipo de protocolo #

Al aplicar el modificador débil a una variable de tipo de protocolo, ese tipo de protocolo debe ser solo de clase, ya que débil solo se puede aplicar a tipos de referencia.

weak var weakReference : ClassOnlyProtocol?

Extensión de protocolo para una clase conforme específica #

Puede escribir la implementación de protocolo predeterminada para una clase específica.

protocol MyProtocol {
    func doSomething()
}

extension MyProtocol where Self: UIViewController {
    func doSomething() {
        print("UIViewController default protocol implementation")
    }
}

class MyViewController: UIViewController, MyProtocol { }

let vc = MyViewController()
vc.doSomething() // Prints "UIViewController default protocol implementation"

Uso del protocolo RawRepresentable (Enumeración extensible) #

// RawRepresentable has an associatedType RawValue.
// For this struct, we will make the compiler infer the type
// by implementing the rawValue variable with a type of String
//
// Compiler infers RawValue = String without needing typealias
//
struct NotificationName: RawRepresentable {
    let rawValue: String

    static let dataFinished = NotificationNames(rawValue: "DataFinishedNotification")
}

Esta estructura se puede extender a otros lugares para agregar casos

extension NotificationName {
    static let documentationLaunched = NotificationNames(rawValue: "DocumentationLaunchedNotification")
}

Y una interfaz puede diseñarse en torno a cualquier tipo RawRepresentable o específicamente su estructura de enumeración

func post(notification notification: NotificationName) -> Void {
    // use notification.rawValue
}

En el sitio de la llamada, puede usar la abreviatura de sintaxis de puntos para el nombre de notificación seguro para tipos

post(notification: .dataFinished)

Usando la función genérica RawRepresentable

// RawRepresentable has an associate type, so the 
// method that wants to accept any type conforming to
// RawRepresentable needs to be generic
func observe<T: RawRepresentable>(object: T) -> Void {
    // object.rawValue
}

Implementando el protocolo Hashable #

Los tipos utilizados en Conjuntos y Diccionarios (clave) deben cumplir con el protocolo Hashable que hereda del protocolo Equatable.

Se debe implementar un tipo personalizado que se ajuste al protocolo Hashable

  • Una propiedad calculada hashValue
  • Defina uno de los operadores de igualdad, es decir, == o !=.

El siguiente ejemplo implementa el protocolo ‘Hashable’ para una ’estructura’ personalizada:

struct Cell {
    var row: Int
    var col: Int
    
    init(_ row: Int, _ col: Int) {
        self.row = row
        self.col = col
    }
}

extension Cell: Hashable {
    
    // Satisfy Hashable requirement
    var hashValue: Int {
        get {
            return row.hashValue^col.hashValue
        }
    }

    // Satisfy Equatable requirement
    static func ==(lhs: Cell, rhs: Cell) -> Bool {
        return lhs.col == rhs.col && lhs.row == rhs.row
    }
    
}

// Now we can make Cell as key of dictonary
var dict = [Cell : String]()

dict[Cell(0, 0)] = "0, 0"
dict[Cell(1, 0)] = "1, 0"
dict[Cell(0, 1)] = "0, 1"

// Also we can create Set of Cells
var set = Set<Cell>()

set.insert(Cell(0, 0))
set.insert(Cell(1, 0))

Nota: No es necesario que diferentes valores en el tipo personalizado tengan diferentes valores hash, las colisiones son aceptables. Si los valores hash son iguales, se utilizará el operador de igualdad para determinar la igualdad real.
