Swift tiene una guía de estilo oficial: [Pautas de diseño de API de Swift.org] (https://swift.org/documentation/api-design-guidelines/). Otra guía popular es The Official raywenderlich.com Swift Style Guide.

Uso fluido #

Uso del lenguaje natural

Las llamadas de funciones deben estar cerca del idioma inglés natural.

Ejemplo:

list.insert(element, at: index)

en vez de

list.insert(element, position: index)

Métodos de fábrica de nombres

Los métodos de fábrica deben comenzar con el prefijo `make`.

Ejemplo:

factory.makeObject()

Nombrar parámetros en inicializadores y métodos de fábrica

El nombre del primer argumento no debe estar involucrado en el nombre de un método de fábrica o inicializador.

Ejemplo:

factory.makeObject(key: value)

En vez de:

factory.makeObject(havingProperty: value)

Denominación según los efectos secundarios

  • Las funciones con efectos secundarios (funciones mutantes) deben nombrarse usando verbos o sustantivos con el prefijo form- .
  • Las funciones sin efectos secundarios (funciones que no mutan) deben nombrarse usando sustantivos o verbos con el sufijo -ing o -ed.
Ejemplo: Funciones mutantes: 
print(value)
array.sort()                 // in place sorting
list.add(value)              // mutates list
set.formUnion(anotherSet)    // set is now the union of set and anotherSet

Funciones no mutantes:

let sortedArray = array.sorted()     // out of place sorting
let union = set.union(anotherSet)    // union is now the union of set and another set

Funciones o variables booleanas

Las declaraciones que involucran booleanos deben leerse como aserciones.

Ejemplo:

set.isEmpty
line.intersects(anotherLine)

Protocolos de nombres

  • Los protocolos que describen qué es algo deben nombrarse usando sustantivos.
  • Los protocolos que describen capacidades deben tener -able, -ible o -ing como sufijo.
Ejemplo: 
Collection        // describes that something is a collection
ProgressReporting // describes that something has the capability of reporting progress
Equatable         // describes that something has the capability of being equal to something

Tipos y Propiedades

Los tipos, variables y propiedades deben leerse como sustantivos.

Ejemplo:

let factory = ...
let list = [1, 2, 3, 4]

Borrar uso #

Evite la ambigüedad

El nombre de clases, estructuras, funciones y variables debe evitar la ambigüedad.

Ejemplo:

extension List {
    public mutating func remove(at position: Index) -> Element {
        // implementation
    }
}

La llamada de función a esta función se verá así:

list.remove(at: 42)

De esta manera, se evita la ambigüedad. Si la llamada a la función fuera simplemente list.remove(42), no estaría claro si se eliminaría un elemento igual a 42 o si se eliminaría el elemento en el índice 42.

Evite la redundancia

El nombre de las funciones no debe contener información redundante.

Un mal ejemplo sería:

extension List {
    public mutating func removeElement(element: Element) -> Element? {
        // implementation
    }
}

Una llamada a la función puede verse como list.removeElement(someObject). La variable someObject ya indica que se elimina un elemento. Sería mejor que la firma de la función se viera así:

extension List {
    public mutating func remove(_ member: Element) -> Element? {
        // implementation
    }
}

La llamada a esta función se ve así: list.remove(someObject).

Nombrar variables según su rol

Las variables deben nombrarse por su función (p. ej., proveedor, saludo) en lugar de su tipo (p. ej., fábrica, cadena, etc.)

Alto acoplamiento entre el nombre del protocolo y los nombres de las variables

Si el nombre del tipo describe su función en la mayoría de los casos (por ejemplo, iterador), el tipo debe nombrarse con el sufijo `Tipo`. (por ejemplo, IteratorType)

Proporcione detalles adicionales cuando utilice parámetros mal escritos

Si el tipo de un objeto no indica claramente su uso en una llamada de función, la función debe nombrarse con un sustantivo precedente para cada parámetro débilmente tipado, describiendo su uso.
Ejemplo: 
func addObserver(_ observer: NSObject, forKeyPath path: String)

a la que una llamada se vería como `object.addObserver(self, forKeyPath: ruta)

en vez de

func add(_ observer: NSObject, for keyPath: String)

a la que una llamada se vería como object.add(self, for: path)

Capitalización #

Tipos y protocolos

Los nombres de tipo y protocolo deben comenzar con una letra mayúscula.

Ejemplo:

protocol Collection {}
struct String {}
class UIView {}
struct Int {}
enum Color {}

Todo lo demás...

Las variables, constantes, funciones y casos de enumeración deben comenzar con una letra minúscula.

Ejemplo:

let greeting = "Hello"
let height = 42.0

enum Color {
    case red
    case green
    case blue
}

func print(_ string: String) {
    ...
}

Estuche camello:

Todos los nombres deben usar el caso de camello apropiado. Mayúsculas camel para nombres de tipo/protocolo y minúsculas camel para todo lo demás.

Caso superior del camello:

protocol IteratorType { ... }

Caja de camello inferior:

let inputView = ...

Abreviaturas

Deben evitarse las abreviaturas a menos que se utilicen comúnmente (por ejemplo, URL, ID). Si se usa una abreviatura, todas las letras deben tener el mismo caso.

Ejemplo:

let userID: UserID = ...
let urlString: URLString = ...
