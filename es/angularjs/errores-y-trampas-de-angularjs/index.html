Cosas que hacer cuando se usa html5Mode #

Al usar html5Mode([mode]) es necesario que:

  1. Usted especifica la URL base para la aplicación con <base href=""> en el encabezado de su index.html.

  2. Es importante que la etiqueta base esté antes de cualquier etiqueta con solicitudes de URL. De lo contrario, podría generar este error: "Recurso interpretado como hoja de estilo pero transferido con tipo MIME text/html". Por ejemplo:

     <head>
     <meta charset="utf-8">
     <title>Job Seeker</title>

     <base href="/">

     <link rel="stylesheet" href="bower_components/bootstrap/dist/css/bootstrap.css" />
     <link rel="stylesheet" href="/styles/main.css">
 </head>

  3. Si no desea especificar una etiqueta base, configure $ LocationProvider para que no requiera una etiqueta base pasando un objeto de definición con requireBase:false a $ LocationProvider.html5Mode() como este :

     $locationProvider.html5Mode({
     enabled: true,
     requireBase: false
 });

  4. Para admitir la carga directa de URL HTML5, debe habilitar la reescritura de URL del lado del servidor. De AngularJS/Guía del desarrollador/Uso de $ubicación

El uso de este modo requiere la reescritura de URL en el lado del servidor, básicamente tienes que reescribir todos tus enlaces al punto de entrada de tu aplicación (por ejemplo, index.html). Requerir una etiqueta <base> también es importante para este caso, ya que le permite a Angular diferenciar entre la parte de la URL que es la base de la aplicación y la ruta que debe manejar la aplicación.

Se puede encontrar un recurso excelente para ejemplos de reescritura de solicitudes para varias implementaciones de servidores HTTP en las preguntas frecuentes de ui-router - Cómo: Configurar su servidor para trabajar con html5Mode. por ejemplo apache

    RewriteEngine on

    # Don't rewrite files or directories
    RewriteCond %{REQUEST_FILENAME} -f [OR]
    RewriteCond %{REQUEST_FILENAME} -d
    RewriteRule ^ - [L]

    # Rewrite everything else to index.html to allow html5 state links
    RewriteRule ^ index.html [L]

nginx

    server {
        server_name my-app;

        root /path/to/app;

        location / {
            try_files $uri $uri/ /index.html;
        }
    }

Expresar

    var express = require('express');
    var app = express();

    app.use('/js', express.static(__dirname + '/js'));
    app.use('/dist', express.static(__dirname + '/../dist'));
    app.use('/css', express.static(__dirname + '/css'));
    app.use('/partials', express.static(__dirname + '/partials'));

    app.all('/*', function(req, res, next) {
        // Just send the index.html for other files to support HTML5Mode
        res.sendFile('index.html', { root: __dirname });
    });

    app.listen(3006); //the port you want to use

El enlace de datos bidireccional deja de funcionar #

Se debe tener en cuenta que:

  1. El enlace de datos de Angular se basa en la herencia prototípica de JavaScript, por lo que está sujeto a [variable shadowing] (https://en.wikipedia.org/wiki/Variable_shadowing).
  2. Un ámbito secundario normalmente hereda prototípicamente de su ámbito principal. Una excepción a esta regla es una directiva que tiene un alcance aislado ya que no hereda prototípicamente.
  3. Hay algunas directivas que crean un nuevo alcance secundario: ng-repeat, ng-switch, ng-view, ng-if, ng-controller, ng-include, etc. .

Esto significa que cuando intenta vincular de forma bidireccional algunos datos a una primitiva que está dentro de un ámbito secundario (o viceversa), es posible que las cosas no funcionen como se esperaba. Aquí es un ejemplo de lo fácil que es “romper” AngularJS.

Este problema se puede evitar fácilmente siguiendo estos pasos:

  1. Tener un “.” dentro de su plantilla HTML cada vez que vincula algunos datos
  2. Use la sintaxis controllerAs ya que promueve el uso de enlaces a un objeto “punteado”
  3. $parent se puede usar para acceder a las variables del “alcance” principal en lugar del alcance secundario. como dentro de ng-if podemos usar ng-model="$parent.foo"..

Una alternativa para lo anterior es vincular ngModel a una función getter/setter que actualizará la versión en caché del modelo cuando se llame con argumentos, o la devolverá cuando se llame sin argumentos. Para usar una función getter/setter, debe agregar ng-model-options="{ getterSetter: true }" al elemento con el atributo ngModal y llamar a la función getter si desea mostrar su valor en expresión (Ejemplo de trabajo).

Ejemplo #

Vista:

<div ng-app="myApp" ng-controller="MainCtrl">
    <input type="text" ng-model="foo" ng-model-options="{ getterSetter: true }">
    <div ng-if="truthyValue">
        <!-- I'm a child scope (inside ng-if), but i'm synced with changes from the outside scope -->
        <input type="text" ng-model="foo">
    </div>
    <div>$scope.foo: {{ foo() }}</div>
</div>

Controlador:

angular.module('myApp', []).controller('MainCtrl', ['$scope', function($scope) {
    $scope.truthyValue = true;
      
    var _foo = 'hello'; // this will be used to cache/represent the value of the 'foo' model 
      
    $scope.foo = function(val) {
        // the function return the the internal '_foo' varibale when called with zero arguments,
        // and update the internal `_foo` when called with an argument
        return arguments.length ? (_foo = val) : _foo;
    };
}]);

Práctica recomendada: es mejor que los captadores sean rápidos porque es probable que Angular los llame con más frecuencia que otras partes de su código (referencia).

7 pecados capitales de AngularJS #

A continuación se muestra la lista de algunos errores que los desarrolladores suelen cometer durante el uso de las funcionalidades de AngularJS, algunas lecciones aprendidas y soluciones para ellos.

1. Manipulando DOM a través del controlador

Es legal, pero debe evitarse. Los controladores son los lugares donde define sus dependencias, vincula sus datos a la vista y crea una lógica comercial adicional. Técnicamente, puede manipular el DOM en un controlador, pero siempre que necesite una manipulación igual o similar en otra parte de su aplicación, se necesitará otro controlador. Entonces, la mejor práctica de este enfoque es crear una directiva que incluya todas las manipulaciones y usar la directiva en toda su aplicación. Por lo tanto, el controlador deja la vista intacta y hace su trabajo. En una directiva, la función de enlace es el mejor lugar para manipular el DOM. Tiene acceso completo al alcance y al elemento, por lo que al usar una directiva, también puede aprovechar la reutilización.

link: function($scope, element, attrs) {
    //The best place to manipulate DOM
}

Puede acceder a los elementos DOM en la función de enlace de varias maneras, como el parámetro element, el método angular.element() o Javascript puro.

2. Enlace de datos en transclusión

AngularJS es famoso por su enlace de datos bidireccional. Sin embargo, a veces puede encontrar que sus datos solo están enlazados en un solo sentido dentro de las directivas. Detente ahí, AngularJS no está mal, pero probablemente tú. Las directivas son lugares un poco peligrosos ya que están involucrados ámbitos secundarios y ámbitos aislados. Suponga que tiene la siguiente directiva con una transclusión

<my-dir>
    <my-transclusion>
    </my-transclusion>
</my-dir>

Y dentro de my-transclusion, tiene algunos elementos que están vinculados a los datos en el ámbito externo.

<my-dir>
    <my-transclusion>
        <input ng-model="name">
    </my-transclusion>
</my-dir>

El código anterior no funcionará correctamente. Aquí, la transclusión crea un alcance secundario y puede obtener la variable de nombre, correcto, pero cualquier cambio que realice en esta variable permanecerá allí. Entonces, realmente puede acceder a esta variable como $parent.name. Sin embargo, este uso podría no ser la mejor práctica. Un mejor enfoque sería envolver las variables dentro de un objeto. Por ejemplo, en el controlador puede crear:

$scope.data = {
    name: 'someName'
}

Luego, en la transclusión, puede acceder a esta variable a través del objeto ‘datos’ y ver que el enlace bidireccional funciona perfectamente.

<input ng-model="data.name">

No solo en las transclusiones, sino en toda la aplicación, es una buena idea usar la notación punteada.

3. Múltiples directivas juntas

En realidad, es legal usar dos directivas juntas dentro del mismo elemento, siempre y cuando cumpla con la regla: dos ámbitos aislados no pueden existir en el mismo elemento. En términos generales, al crear una nueva directiva personalizada, asigna un ámbito aislado para facilitar el paso de parámetros. Asumiendo que las directivas myDirA y myDirB tienen alcances aislados y myDirC no, el siguiente elemento será válido:

<input my-dir-a my-dirc>

mientras que el siguiente elemento causará un error de consola:

<input my-dir-a my-dir-b>

Por lo tanto, las directivas deben usarse con prudencia, teniendo en cuenta los alcances.

4. Mal uso de $emit

$emit, $broadcast y $on, estos funcionan en un principio de emisor-receptor. En otras palabras, son un medio de comunicación entre controladores. Por ejemplo, la siguiente línea emite ‘someEvent’ desde el controlador A, para ser capturado por el controlador B en cuestión.

$scope.$emit('someEvent', args);

Y la siguiente línea captura el ‘someEvent’

$scope.$on('someEvent', function(){});

Hasta aquí todo parece perfecto. Pero recuerde que, si el controlador B aún no se invoca, el evento no se detectará, lo que significa que se deben invocar los controladores emisor y receptor para que esto funcione. Entonces, nuevamente, si no está seguro de que definitivamente tiene que usar $emit, crear un servicio parece una mejor manera.

5. Mal uso de $scope.$watch

$scope.$watch se usa para observar un cambio de variable. Cada vez que una variable ha cambiado, se invoca este método. Sin embargo, un error común que se comete es cambiar la variable dentro de $scope.$watch. Esto causará inconsistencias y un bucle $digest infinito en algún momento.

$scope.$watch('myCtrl.myVariable', function(newVal) {
    this.myVariable++;
});

Entonces, en la función anterior, asegúrese de no tener operaciones en myVariable y newVal.

6. Enlazar métodos a vistas

Este es uno de los pecados más mortales. AngularJS tiene enlace bidireccional, y cada vez que algo cambia, las vistas se actualizan muchas veces. Por lo tanto, si vincula un método a un atributo de una vista, ese método podría llamarse cien veces, lo que también lo vuelve loco durante la depuración. Sin embargo, solo hay algunos atributos creados para el enlace de métodos, como ng-click, ng-blur, ng-on-change, etc., que esperan métodos como parámetro. Por ejemplo, suponga que tiene la siguiente vista en su marcado:

<input ng-disabled="myCtrl.isDisabled()" ng-model="myCtrl.name">

Aquí comprueba el estado deshabilitado de la vista a través del método isDisabled. En el controlador myCtrl, tienes:

vm.isDisabled = function(){
    if(someCondition)
        return true;
    else
        return false;
}

En teoría, puede parecer correcto, pero técnicamente esto provocará una sobrecarga, ya que el método se ejecutará innumerables veces. Para resolver esto, debe vincular una variable. En su controlador, debe existir la siguiente variable:

vm.isDisabled

Puede iniciar esta variable nuevamente en la activación del controlador

if(someCondition)
    vm.isDisabled = true
else
    vm.isDisabled = false

Si la condición no es estable, puede vincular esto a otro evento. Entonces debería vincular esta variable a la vista:

<input ng-disabled="myCtrl.isDisabled" ng-model="myCtrl.name">

Ahora, todos los atributos de la vista tienen lo que esperan y los métodos se ejecutarán solo cuando sea necesario.

7. No usar las funcionalidades de Angular

AngularJS brinda una gran comodidad con algunas de sus funcionalidades, no solo simplificando su código sino también haciéndolo más eficiente. Algunas de estas características se enumeran a continuación:

  1. angular.forEach for the loops (Caution, you can't "break;" it, you can only prevent getting into the body, so consider performance here.)
  2. angular.element for DOM selectors
  3. angular.copy: Use this when you should not modify the main object
  4. Form validations are already awesome. Use dirty, pristine, touched, valid, required and so on.
  5. Besides Chrome debugger, use remote debugging for mobile development too.
  6. And make sure you use Batarang. It's a free Chrome extension where you can easily inspect scopes
