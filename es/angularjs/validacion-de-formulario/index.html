Forma y estados de entrada #

Los formularios angulares y las entradas tienen varios estados que son útiles al validar el contenido

Estados de entrada

Estado Descripción
$tocado Campo ha sido tocado
$sin tocar El campo no ha sido tocado
$prístino El campo no ha sido modificado
$sucio El campo ha sido modificado
$válido El contenido del campo es válido
$inválido El contenido del campo no es válido

Todos los estados anteriores son propiedades booleanas y pueden ser verdaderos o falsos.

Con estos, es muy fácil mostrar mensajes a un usuario.

<form name="myForm" novalidate>
    <input name="myName" ng-model="myName" required>
    <span ng-show="myForm.myName.$touched && myForm.myName.$invalid">This name is invalid</span>
</form>

Aquí, estamos usando la directiva ng-show para mostrar un mensaje a un usuario si ha modificado un formulario pero no es válido.

Clases CSS #

Angular también proporciona algunas clases de CSS para formularios y entradas según su estado

Clase Descripción
ng-tocado Campo ha sido tocado
ng-sin tocar El campo no ha sido tocado
ng-prístino El campo no ha sido modificado
ng-sucio El campo ha sido modificado
de-válido El campo es válido
ng-no válido El campo no es válido

Puede usar estas clases para agregar estilos a sus formularios

input.ng-invalid {
  background-color: crimson;
}
input.ng-valid {
  background-color: green;
}

Validación de formulario básico #

Una de las fortalezas de Angular es la validación de formularios del lado del cliente.

Tratar con entradas de formularios tradicionales y tener que usar el procesamiento de estilo jQuery interrogativo puede llevar mucho tiempo y ser quisquilloso. Angular te permite producir formularios interactivos profesionales con relativa facilidad.

La directiva ng-model proporciona un enlace bidireccional con los campos de entrada y, por lo general, el atributo novalidate también se coloca en el elemento de formulario para evitar que el navegador realice una validación nativa.

Por lo tanto, un formulario simple se vería así:

<form name="form" novalidate>
  <label name="email"> Your email </label>
  <input type="email" name="email" ng-model="email" />
</form>

Para que Angular valide las entradas, use exactamente la misma sintaxis que un elemento input regular, excepto por la adición del atributo ng-model para especificar a qué variable enlazar en el alcance. El correo electrónico se muestra en el ejemplo anterior. Para validar un número, la sintaxis sería:

<input type="number" name="postalcode" ng-model="zipcode" />

Los pasos finales para la validación básica de formularios se conectan a una función de envío de formularios en el controlador mediante ng-submit, en lugar de permitir que se produzca el envío de formulario predeterminado. Esto no es obligatorio, pero generalmente se usa, ya que las variables de entrada ya están disponibles en el alcance y, por lo tanto, disponibles para su función de envío. También suele ser una buena práctica dar un nombre al formulario. Estos cambios darían como resultado la siguiente sintaxis:

<form name="signup_form" ng-submit="submitFunc()" novalidate>
  <label name="email"> Your email </label>
  <input type="email" name="email" ng-model="email" />
  <button type="submit">Signup</button>
</form>

Este código anterior es funcional, pero hay otra funcionalidad que proporciona Angular.

El siguiente paso es entender que Angular adjunta atributos de clase usando ng-pristine, ng-dirty, ng-valid y ng-invalid para el procesamiento de formularios. El uso de estas clases en su css le permitirá diseñar campos de entrada válido/no válido y prístino/sucio y así modificar la presentación a medida que el usuario ingresa datos en el formulario.

Validación de formulario personalizado #

En algunos casos, la validación básica no es suficiente. Validación personalizada de soporte angular agregando funciones de validación al objeto $validators en ngModelController:

angular.module('app', [])
  .directive('myValidator', function() {
    return {
      // element must have ng-model attribute
      // or $validators does not work
      require: 'ngModel',
      link: function(scope, elm, attrs, ctrl) {
        ctrl.$validators.myValidator = function(modelValue, viewValue) {
            // validate viewValue with your custom logic
            var valid = (viewValue && viewValue.length > 0) || false;
            return valid; 
        };
      } 
    };

El validador se define como una directiva que requiere ngModel, por lo que para aplicar el validador simplemente agregue la directiva personalizada al control del formulario de entrada.

<form name="form">
  <input type="text" 
         ng-model="model" 
         name="model" 
         my-validator>
  <pre ng-bind="'my-validator returned: ' + form.model.$valid"></pre>
</form>

Y my-validator no tiene que aplicarse en el control de formulario nativo. Puede ser cualquier elemento, siempre que sea ng-model en sus atributos. Esto es útil cuando tiene algún componente de interfaz de usuario de compilación personalizado.

ngMensajes #

ngMessages se usa para mejorar el estilo para mostrar mensajes de validación en la vista.

Enfoque tradicional #

Antes de ngMessages, normalmente mostrábamos los mensajes de validación usando las directivas predefinidas de Angular ng-class. Este enfoque era basura y una tarea repetitiva.

Ahora, usando ngMessages podemos crear nuestros propios mensajes personalizados.

Ejemplo #

HTML:

<form name="ngMessagesDemo">
  <input name="firstname" type="text" ng-model="firstname" required>
  <div ng-messages="ngMessagesDemo.firstname.$error">
    <div ng-message="required">Firstname is required.</div>
  </div>
</form>
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.3.16/angular.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.3.16/angular-messages.min.js"></script>

JS:

var app = angular.module('app', ['ngMessages']);

app.controller('mainCtrl', function ($scope) { 
    $scope.firstname = "Rohit";
});

Validadores asíncronos #

Los validadores asíncronos le permiten validar la información del formulario contra su backend (usando $http).

Este tipo de validadores son necesarios cuando necesita acceder a la información almacenada en el servidor que no puede tener en su cliente por varias razones, como la tabla de usuarios y otra información de la base de datos.

Para usar validadores asíncronos, accede al ng-model de su input y define las funciones de devolución de llamada para la propiedad $asyncValidators.

Ejemplo:

El siguiente ejemplo verifica si ya existe un nombre provisto, el backend devolverá un estado que rechazará la promesa si el nombre ya existe o si no se proporcionó. Si el nombre no existe, devolverá una promesa resuelta.

ngModel.$asyncValidators.usernameValidate = function (name) {               
     if (name) {
          return AuthenticationService.checkIfNameExists(name); // returns a promise
     } else {
          return $q.reject("This username is already taken!"); // rejected promise
     }
};

Ahora, cada vez que se cambie el ng-model de la entrada, esta función se ejecutará y devolverá una promesa con el resultado.

Formularios anidados #

A veces es deseable anidar formularios con el fin de agrupar controles y entradas de forma lógica en la página. Sin embargo, los formularios HTML5 no deben anidarse. Angular proporciona ng-form en su lugar.

<form name="myForm" noValidate>
  <!-- nested form can be referenced via 'myForm.myNestedForm' -->
  <ng-form name="myNestedForm" noValidate>
    <input name="myInput1" ng-minlength="1" ng-model="input1" required />
    <input name="myInput2" ng-minlength="1" ng-model="input2" required />
  </ng-form>

  <!-- show errors for the nested subform here -->
  <div ng-messages="myForm.myNestedForm.$error">
    <!-- note that this will show if either input does not meet the minimum -->
    <div ng-message="minlength">Length is not at least 1</div>
  </div>
</form>

<!-- status of the form -->
<p>Has any field on my form been edited? {{myForm.$dirty}}</p>
<p>Is my nested form valid? {{myForm.myNestedForm.$valid}}</p>
<p>Is myInput1 valid? {{myForm.myNestedForm.myInput1.$valid}}</p>

Cada parte del formulario contribuye al estado general del formulario. Por lo tanto, si una de las entradas myInput1 ha sido editada y es $dirty, su forma contenedora también será $dirty. Esto se aplica en cascada a cada formulario que lo contiene, por lo que tanto myNestedForm como myForm estarán $dirty.
