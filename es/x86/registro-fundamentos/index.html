Registros de 16 bits #

Cuando Intel definió el 8086 original, era un procesador de 16 bits con un bus de direcciones de 20 bits (ver más abajo). Definieron 8 registros de 16 bits de propósito general, pero les dieron roles específicos para ciertas instrucciones:

  • AX El registro Acumulador. Muchos códigos de operación asumieron este registro o fueron más rápidos si se especificaba.
  • DX El registro de Datos. Esto a veces se combinaba como los 16 bits altos de un valor de 32 bits con ‘AX’, por ejemplo, como resultado de una multiplicación.
  • CX El registro de conteo. Esto se usó en varias instrucciones orientadas a bucles como el contador implícito para esos bucles, por ejemplo, ‘LOOPNE’ (bucle si no es igual) y ‘REP’ (movimiento/comparación repetidos)
  • BX El registro Base. Esto podría usarse para indexar la base de una estructura en la memoria; ninguno de los registros anteriores podría usarse para indexar directamente en la memoria.
  • SI El registro de índice de origen. Este fue el índice fuente implícito en la memoria para ciertas operaciones de movimiento y comparación.
  • DI El registro de índice de destino. Este era el índice de destino implícito en la memoria para ciertas operaciones de movimiento y comparación.
  • SP El registro del puntero de pila. ¡Este es el registro menos general del conjunto! Apuntaba a la posición actual en la pila, que se usaba explícitamente para las operaciones PUSH y POP, implícitamente para CALL y RET con subrutinas, y MUY implícitamente durante las interrupciones. ¡Como tal, usarlo para cualquier otra cosa era peligroso para su programa!
  • BP El registro del puntero base. Cuando las subrutinas llaman a otras subrutinas, la pila contiene varios “marcos de pila”. BP podría usarse para mantener el marco de pila actual, y luego, cuando se llamó a una nueva subrutina, se podría guardar en la pila, crear y usar el nuevo marco de pila, y al regresar de la subrutina interna, el valor del marco de pila anterior podría ser restaurado

Notas: #

  1. Los primeros tres registros no se pueden usar para indexar en la memoria.

  2. BX, SI y DI por defecto se indexan en el segmento de datos actual (ver más abajo).

     MOV    AX, [BX+5]     ; Point into Data Segment
 MOV    AX, ES:[DI+5]  ; Override into Extra Segment

  3. ‘DI’, cuando se usa en operaciones de memoria a memoria como ‘MOVS’ y ‘CMPS’, usa únicamente el segmento adicional (ver más abajo). Esto no se puede anular.

  4. SP y BP usan el segmento de pila (ver más abajo) por defecto.

registros de 32 bits #

Cuando Intel produjo el 80386, actualizaron de un procesador de 16 bits a uno de 32 bits. El procesamiento de 32 bits significa dos cosas: tanto los datos que se manipulaban eran de 32 bits como las direcciones de memoria a las que se accedía eran de 32 bits. Para hacer esto, pero aún siendo compatibles con sus procesadores anteriores, introdujeron modos completamente nuevos para el procesador. Estaba en modo de 16 bits o de 32 bits, ¡pero podía anular este modo instrucción por instrucción para datos, direccionamiento o ambos!

En primer lugar, tenían que definir registros de 32 bits. Hicieron esto simplemente extendiendo los ocho existentes de 16 bits a 32 bits y dándoles nombres “extendidos” con un prefijo E: EAX, EBX, ECX, EDX, ESI, EDI, EBP y ESP. Los 16 bits inferiores de estos registros eran los mismos que antes, pero las mitades superiores de los registros estaban disponibles para operaciones de 32 bits como ADD y CMP. Las mitades superiores no eran accesibles por separado como lo habían hecho con los registros de 8 bits.

El procesador debía tener modos separados de 16 y 32 bits porque Intel usaba los mismos códigos de operación para muchas de las operaciones: CMP AX,DX en modo de 16 bits y CMP EAX,EDX en modo de 32 bits tenían exactamente los mismos códigos de operación! Esto significaba que el mismo código NO se podía ejecutar en ninguno de los dos modos:

El código de operación para “Mover inmediatamente a AX” es 0xB8, seguido de dos bytes del valor inmediato: 0xB8 0x12 0x34

El código de operación para “Mover inmediatamente a EAX” es 0xB8, seguido de cuatro bytes del valor inmediato: 0xB8 0x12 0x34 0x56 0x78

Entonces, el ensamblador debe saber en qué modo está el procesador al ejecutar el código, para que sepa emitir la cantidad correcta de bytes.

Registros de 8 bits #

Los primeros cuatro [registros de 16 bits] (https://www.wikiod.com/es/x86/registro-fundamentos#Registros de 16 bits) podrían tener acceso a sus mitades superior e inferior de bytes directamente como su registros propios:

  • AH y AL son las mitades alta y baja del registro AX.
  • BH y BL son las mitades alta y baja del registro BX.
  • CH y CL son las mitades alta y baja del registro CX.
  • DH y DL son las mitades alta y baja del registro DX.

¡Tenga en cuenta que esto significa que alterar AH o AL también alterará inmediatamente AX! También tenga en cuenta que cualquier operación en un registro de 8 bits no podría afectar a su “socio”: incrementar AL de modo que se desborde de 0xFF a 0x00 no alteraría AH.

https://www.wikiod.com/es/x86/registro-fundamentos#registros de 64 bits también tienen versiones de 8 bits que representan sus bytes inferiores:

  • SIL para RSI
  • DIL para RDI
  • BPL para RBP
  • SPL para RSP

Lo mismo se aplica a los registros R8 a R15: sus respectivas partes de bytes inferiores se denominan R8B – R15B.

Registros de segmento #

Segmentación #

Cuando Intel estaba diseñando el 8086 original, ya había varios procesadores de 8 bits que tenían capacidades de 16 bits, pero querían producir un verdadero procesador de 16 bits. También querían producir algo mejor y más capaz que lo que ya existía, por lo que querían poder acceder a más del máximo de 65 536 bytes de memoria que implican los registros de direccionamiento de 16 bits.

Registros de segmentos originales #

Así que implementaron la idea de “Segmentos”, un bloque de memoria de 64 kilobytes indexado por los registros de direcciones de 16 bits, que podría reorganizarse para abordar diferentes áreas de la memoria total. Para albergar estas bases de segmentos, incluían Registros de Segmentos:

  • CS El registro del segmento de código. Contiene el segmento del código que se está ejecutando actualmente, indexado por el registro implícito IP (Instruction Pointer).
  • DS El registro del segmento de datos. Esto contiene el segmento predeterminado para los datos manipulados por el programa.
  • ES El registro de Segmento Extra. Esto contiene un segundo segmento de datos, para operaciones de datos simultáneas en toda la memoria.
  • SS El registro del segmento de pila. Esto contiene el segmento de memoria que contiene la pila actual.

Tamaño del segmento? #

Los registros de segmento podían tener cualquier tamaño, pero hacerlos de 16 bits de ancho facilitaba la interoperabilidad con los otros registros. La siguiente pregunta era: ¿deberían superponerse los segmentos y, de ser así, cuánto? La respuesta a esa pregunta dictaría el tamaño total de la memoria a la que se podría acceder.

Si no hubiera ninguna superposición, entonces el espacio de direcciones sería de 32 bits, 4 gigabytes, ¡un tamaño totalmente inaudito en ese momento! Una superposición más “natural” de 8 bits produciría un espacio de direcciones de 24 bits o 16 megabytes. Al final, Intel decidió guardar cuatro pines de dirección más en el procesador haciendo que el espacio de direcciones fuera de 1 megabyte con una superposición de 12 bits; ¡lo consideraron lo suficientemente grande para la época!

¡Más registros de segmentos! #

Cuando Intel estaba diseñando el 80386, reconocieron que el conjunto existente de registros de 4 segmentos no era suficiente para la complejidad de los programas que querían que pudiera admitir. Así que agregaron dos más:

  • FS El registro del segmento lejano
  • GS El registro de Segmento Global

Estos nuevos registros de segmento no tenían ningún uso impuesto por el procesador: simplemente estaban disponibles para lo que el programador quisiera.

Algunos dicen que los nombres fueron elegidos simplemente para continuar con el tema C, D, E del conjunto existente…

registros de 64 bits #

AMD es un fabricante de procesadores que obtuvo la licencia del diseño del 80386 de Intel para producir versiones compatibles, pero competidoras. Hicieron cambios internos en el diseño para mejorar el rendimiento u otras mejoras en el diseño, sin dejar de poder ejecutar los mismos programas.

Para superar a Intel, crearon extensiones de 64 bits para el diseño de Intel de 32 bits y produjeron el primer chip de 64 bits que aún podía ejecutar código x86 de 32 bits. Intel acabó siguiendo el diseño de AMD en sus versiones de la arquitectura de 64 bits.

El diseño de 64 bits realizó una serie de cambios en el conjunto de registros, sin dejar de ser compatible con versiones anteriores:

  • Los registros de uso general existentes se ampliaron a 64 bits y se nombraron con un prefijo R: RAX, RBX, RCX, RDX, RSI, RDI, RBP, y ‘RSP’.

Una vez más, las mitades inferiores de estos registros eran los mismos registros con prefijo E que antes, y no se podía acceder a las mitades superiores de forma independiente.

  • Se agregaron 8 registros más de 64 bits, y no se nombraron sino simplemente se numeraron: R8, R9, R10, R11, R12, R13, R14 y R15.
  • La mitad inferior de 32 bits de estos registros son R8D a R15D (D para DWORD como de costumbre).
  • Se puede acceder a los 16 bits más bajos de estos registros agregando el sufijo ‘W’ al nombre del registro: ‘R8W’ a ‘R15W’.
  • Ahora se puede acceder a los 8 bits más bajos de todos los 16 registros:
  • Los tradicionales AL, BL, CL y DL;
  • Los bytes bajos de los registros de puntero (tradicionalmente): SIL, DIL, BPL y SPL;
  • Y los bytes bajos de los 8 nuevos registros: R8B a R15B.
  • Sin embargo, AH, BH, CH y DH son inaccesibles en instrucciones que usan un prefijo REX (para tamaño de operando de 64 bits, o para acceder a R8-R15, o para acceder a SIL, DIL , BPL o SPL). Con un prefijo REX, el patrón de bits de código de máquina que solía significar ‘AH’ en su lugar significa ‘SPL’, y así sucesivamente. Consulte la Tabla 3-1 del manual de referencia de instrucciones de Intel (volumen 2).

Escribir en un registro de 32 bits siempre pone a cero los 32 bits superiores del registro de ancho completo, a diferencia de escribir en un registro de 8 o 16 bits (que se fusiona con el valor anterior, que es una dependencia adicional para la ejecución desordenada). ).

Registro de banderas #

Cuando la Unidad Lógica Aritmética (ALU) x86 realiza operaciones como NOT y ADD, marca los resultados de estas operaciones (“se convirtió en cero”, “se desbordó”, “se volvió negativo”) en un FLAGS especial de 16 bits registro. Los procesadores de 32 bits actualizaron esto a 32 bits y lo llamaron EFLAGS, mientras que los procesadores de 64 bits actualizaron esto a 64 bits y lo llamaron RFLAGS.

Códigos de condición #

Pero no importa el nombre, no se puede acceder directamente al registro (a excepción de un par de instrucciones, consulte a continuación). En su lugar, se hace referencia a indicadores individuales en ciertas instrucciones, como Salto condicional o Conjunto condicional, conocidos como Jcc y SETcc, donde cc significa “código de condición” y hace referencia a la siguiente tabla:

Código de condición Nombre Definición
‘E’, ‘Z’ Igual, Cero ZF == 1
‘NE’, ‘NZ’ No igual, no cero ZF == 0
‘O’ Desbordamiento DE == 1
NO sin desbordamiento DE == 0
S Firmado SF == 1
NS No firmado SF == 0
P Paridad PF == 1
NP Sin paridad PF == 0
————– —- ———-
C, B, NAE Llevar, Abajo, No Arriba o Igual CF == 1
NC, NB, AE Sin acarreo, ni por debajo, ni por encima ni igual CF == 0
A, NBE Arriba, No Abajo o Igual CF==0 y ZF==0
NA, SER No por encima, por debajo o igual CF==1 o ZF==1
————— —- ———-
GE, NL Mayor o igual, no menor SF==DE
NGE, L Ni mayor ni igual, menor SF!=DE
G, NLE mayor, no menor o igual ZF==0 y SF==OF
NG, LE No mayor, menor o igual ZF==1 o SF!=OF

En 16 bits, restar ‘1’ de ‘0’ es ‘65,535’ o ‘-1’ dependiendo de si se usa aritmética sin signo o con signo, pero el destino contiene ‘0xFFFF’ de cualquier manera. Es solo interpretando los códigos de condición que el significado es claro. Es incluso más revelador si 1 se resta de 0x8000: en aritmética sin signo, eso simplemente cambia 32,768 en 32,767; mientras que en la aritmética con signos cambia -32,768 a 32,767 - ¡un desbordamiento mucho más notable!

Los códigos de condición se agrupan en tres bloques en la tabla: sin signo, sin signo y con signo. La denominación dentro de los dos últimos bloques utiliza “Arriba” y “Abajo” para los sin firmar, y “Mayor” o “Menos” para los que tienen signo. Entonces JB sería “Saltar si debajo” (sin firmar), mientras que JL sería “Saltar si menos” (firmado).

Accediendo a FLAGS directamente #

Los códigos de condición anteriores son útiles para interpretar conceptos predefinidos, pero los bits de bandera reales también están disponibles directamente con las siguientes dos instrucciones:

  • LAHF Carga el registro AH con Banderas
  • SAHF Guarda el registro AH en Flags

Solo ciertas banderas se copian con estas instrucciones. Todo el registro FLAGS / EFLAGS / RFLAGS se puede guardar o restaurar en la pila:

  • PUSHF / POPF Inserta/extrae FLAGS de 16 bits en/desde la pila
  • PUSHFD / POPFD Inserta/extrae EFLAGS de 32 bits en/desde la pila
  • PUSHFQ / POPFQ Inserta/extrae RFLAGS de 64 bits en/desde la pila

Tenga en cuenta que las interrupciones guardan y restauran el registro [R/E]FLAGS actual automáticamente.

Otras banderas #

Además de las banderas ALU descritas anteriormente, el registro FLAGS define otras banderas de estado del sistema:

  • IF La bandera de interrupción. Esto se configura con la instrucción ‘STI’ para habilitar las interrupciones globalmente y se borra con la instrucción ‘CLI’ para deshabilitarlas globalmente.
  • DF La bandera de dirección. Las operaciones de memoria a memoria como CMPS y MOVS (para comparar y moverse entre ubicaciones de memoria) incrementan o reducen automáticamente los registros de índice como parte de la instrucción. La bandera DF dicta cuál sucede: si se borra con la instrucción CLD, se incrementan; si se establece con la instrucción STD, se reducen.
  • TF La bandera trampa. Este es un indicador de depuración. Configurarlo pondrá el procesador en modo de “paso único”: después de que se ejecute cada instrucción, llamará al “Manejador de interrupción de paso único”, que se espera que sea manejado por un depurador. No hay instrucciones para establecer o borrar este indicador: debe manipular el bit mientras está en la memoria.

80286 Banderas #

Para admitir las nuevas funciones multitarea en el 80286, Intel agregó banderas adicionales al registro FLAGS:

  • IOPL El nivel de privilegio de E/S. Para proteger el código multitarea, algunas tareas necesitaban privilegios para acceder a los puertos de E/S, mientras que otras tenían que impedir el acceso a ellos. Intel introdujo una escala de privilegios de cuatro niveles, con 002 siendo el más privilegiado y 112 siendo el menos. Si ‘IOPL’ fuera menor que el nivel de privilegio actual, cualquier intento de acceder a los puertos de E/S, o habilitar o deshabilitar las interrupciones, provocaría una falla de protección general.
  • Indicador de tarea anidada NT. Este indicador se establecía si una tarea ‘LLAMABA’ a otra tarea, lo que provocaba un cambio de contexto. El indicador establecido le dijo al procesador que hiciera un cambio de contexto cuando se ejecutó RET.

80386 Banderas #

El ‘386 necesitaba banderas adicionales para admitir funciones adicionales diseñadas en el procesador.

  • RF La bandera de currículum. El `386 agregó registros de depuración, que podrían invocar el depurador en varios accesos de hardware como leer, escribir o ejecutar una determinada ubicación de memoria. Sin embargo, cuando el controlador de depuración volviera a ejecutar la instrucción, ¡el acceso volvería a invocar inmediatamente al controlador de depuración! se borra automáticamente después de cada instrucción. Si se establece el indicador de reanudación, no se invoca el controlador de depuración.
  • VM La Bandera Virtual 8086. Para admitir el código anterior de 16 bits, así como el código más nuevo de 32 bits, el 80386 podría ejecutar tareas de 16 bits en un modo “Virtual 8086”, con la ayuda de un ejecutivo Virtual 8086. El indicador VM indicaba que esta tarea era una tarea virtual 8086.

80486 Banderas #

A medida que la arquitectura de Intel mejoró, se volvió más rápida a través de tecnología como cachés y ejecución superescalar. Eso tenía que optimizar el acceso al sistema haciendo suposiciones. Para controlar esos supuestos, se necesitaban más banderas:

  • Indicador de comprobación de alineación AC La arquitectura x86 siempre podía acceder a valores de memoria de varios bytes en cualquier límite de bytes, a diferencia de algunas arquitecturas que requerían que estuvieran alineados en tamaño (los valores de 4 bytes debían estar en límites de 4 bytes). Sin embargo, era menos eficiente hacerlo, ya que se necesitaban múltiples accesos a la memoria para acceder a los datos no alineados. Si se estableció el indicador AC, entonces un acceso no alineado generaría una excepción en lugar de ejecutar el código. De esa manera, el código podría mejorarse durante el desarrollo con el conjunto AC, pero se desactivaría para el código de producción.

Banderas Pentium #

El Pentium agregó más soporte para la virtualización, además de soporte para la instrucción ‘CPUID’:

  • VIF El indicador de interrupción virtual. Esta es una copia virtual del ‘SI’ de esta Tarea, ya sea que esta Tarea quiera o no deshabilitar las interrupciones, sin afectar realmente las Interrupciones Globales.
  • VIP El indicador pendiente de interrupción virtual. Esto indica que una interrupción fue bloqueada virtualmente por ‘VIF’, por lo que cuando la Tarea hace un ‘STI’, se puede generar una interrupción virtual para ella.
  • ID La bandera permitida por CPUID. Si permitir o no que esta tarea ejecute la instrucción CPUID. Un monitor virtual podría rechazarlo y “mentir” a la tarea solicitante si ejecuta la instrucción.
