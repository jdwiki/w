Modo real #

Cuando Intel diseñó el x86 original, el 8086 (y el derivado del 8088), incluyeron Segmentación para permitir que el procesador de 16 bits acceda a más de 16 bits de dirección. Hicieron esto haciendo que las direcciones de 16 bits fueran relativas a un Registro de segmento de 16 bits dado, de los cuales definieron cuatro: Segmento de código (CS), Segmento de datos (DS), Segmento adicional (ES) y segmento de pila (SS).

La mayoría de las instrucciones implicaban qué registro de segmento utilizar: las instrucciones se basaban en el segmento de código, PUSH y POP implicaban el segmento de pila, y las referencias de datos simples implicaban el segmento de datos, aunque esto podría anularse para acceder a la memoria en cualquiera de los otros segmentos.

La implementación fue simple: para cada acceso a la memoria, la CPU tomaría el registro de segmento implícito (o explícito), lo desplazaría cuatro lugares a la izquierda y luego agregaría la dirección indicada:

        +-------------------+---------+
Segment | 16-bit value      | 0 0 0 0 |
        +-------------------+---------+
                   PLUS
        +---------+-------------------+
Address | 0 0 0 0 | 16-bit value      |
        +---------+-------------------+
                  EQUALS
        +-----------------------------+
Result  |  20-bit memory address      |
        +-----------------------------+

Esto permitió varias técnicas:

  • Permitir que el código, los datos y la pila sean mutuamente accesibles (CS, DS y SS tenían el mismo valor);
  • Mantener el código, los datos y la pila completamente separados entre sí (CS, DS y SS todos 4K (o más) separados entre sí; recuerde que se multiplica por 16, por lo que son 64K).

¡También permitió superposiciones extrañas y todo tipo de cosas raras!

Cuando se inventó el 80286, admitía este modo heredado (ahora llamado “Modo real”), pero agregó un nuevo modo llamado “Modo protegido” (q.v.).

Lo importante a tener en cuenta es que en Modo Real:

  • Se podía acceder a cualquier dirección de memoria, simplemente colocando el valor correcto dentro de un Registro de segmento y accediendo a la dirección de 16 bits;
  • El alcance de la “protección” era permitir que el programador separara diferentes áreas de la memoria para diferentes propósitos, y hacer más difícil escribir accidentalmente en datos incorrectos, al mismo tiempo que permitía hacerlo.

En otras palabras… ¡no muy protegido en absoluto!

Modo protegido #

Introducción #

Cuando se inventó el 80286, admitió la segmentación 8086 heredada (ahora llamada “Modo real”) y agregó un nuevo modo llamado “Modo protegido”. Este modo ha estado en todos los procesadores x86 desde entonces, aunque mejorado con varias mejoras, como el direccionamiento de 32 y 64 bits.

Diseño #

En el modo protegido, el simple “Agregar dirección al valor de registro de segmento desplazado” se eliminó por completo. Mantuvieron los Registros de Segmento, pero en lugar de usarlos para calcular una dirección, los usaron para indexar en una tabla (en realidad, una de dos…) que definía el Segmento al que se accedería. Esta definición no solo describía dónde estaba el segmento en la memoria (usando Base y Límite), sino también qué tipo de segmento era (Código, Datos, Pila o incluso Sistema) y qué tipos de programas podían acceder a él (Núcleo del sistema operativo, programa normal). , controlador de dispositivo, etc.).

Registro de segmento #

Cada registro de segmento de 16 bits tomó la siguiente forma:

+------------+-----+------+
| Desc Index | G/L | Priv |
+------------+-----+------+
 Desc Index = 13-bit index into a Descriptor Table (described below)
 G/L        = 1-bit flag for which Descriptor Table to Index: Global or Local
 Priv       = 2-bit field defining the Privilege level for access

Global / Local #

El bit global/local definía si el acceso era a una tabla global de descriptores (llamada, como era de esperar, tabla de descriptores globales o GDT), o una tabla de descriptores locales (LDT). La idea de la LDT era que cada programa pudiera tener su propia tabla de descriptores: el sistema operativo definiría un conjunto global de segmentos, y cada programa tendría su propio conjunto de códigos locales, datos y segmentos de pila. El sistema operativo administraría la memoria entre las diferentes tablas de descriptores.

Tabla de descriptores #

Cada tabla de descriptores (global o local) era una matriz de 64 000 de 8192 descriptores: cada uno de ellos era un registro de 8 bytes que definía varios aspectos del segmento que estaba describiendo. Los campos del índice de descriptores de registros de segmentos permitieron 8.192 descriptores: ¡no es coincidencia!

Descriptor #

Un descriptor contenía la siguiente información: tenga en cuenta que el formato del descriptor cambió a medida que se lanzaron nuevos procesadores, pero se mantuvo el mismo tipo de información en cada uno:

  • Base Esto definió la dirección de inicio del segmento de memoria.
  • Límite Esto definió el tamaño del segmento de memoria, más o menos. Tuvieron que tomar una decisión: ¿un tamaño de 0x0000 significaría un tamaño de 0, por lo que no es accesible? O tamaño máximo? En su lugar, eligieron una tercera opción: el campo Límite era la última ubicación direccionable dentro del segmento. Eso significaba que se podía definir un Segmento de despedida; o uno de tamaño máximo para el tamaño de la dirección.
  • Escribe Había varios tipos de segmentos: el código tradicional, los datos y la pila (ver más abajo), pero también se definieron otros segmentos del sistema:
  • Los segmentos de la tabla de descriptores locales definieron a cuántos descriptores locales se podía acceder;
  • Los segmentos de estado de tareas podrían usarse para cambios de contexto administrados por hardware;
  • “Call Gates” controlados que podrían permitir que los programas llamen al sistema operativo, pero solo a través de puntos de entrada cuidadosamente administrados.
  • Atributos También se mantuvieron ciertos atributos del Segmento, en lo relevante:
  • Solo lectura frente a lectura y escritura;
  • Si el segmento estaba actualmente presente o no, lo que permite la administración de memoria a pedido;
  • Qué nivel de código (SO, controlador, programa) podría acceder a este segmento.

¡Por fin, verdadera protección! #

Si el sistema operativo mantuviera las tablas de descriptores en segmentos a los que no se podía acceder mediante meros programas, entonces podría administrar de manera estricta qué segmentos se definieron y qué memoria se asignó y se pudo acceder a cada uno. Un programa podría fabricar cualquier valor de registro de segmento que quisiera, ¡pero si tuviera la audacia de realmente cargarlo en un registro de segmento!… el hardware de la CPU reconocería que el valor de descriptor propuesto rompía cualquiera de una gran cantidad de reglas, y en lugar de completar la solicitud, generaría una excepción en el sistema operativo para permitirle manejar el programa erróneo.

Esta Excepción solía ser la #13, la Excepción de Protección General - hecha mundialmente famosa por Microsoft Windows… (¿Alguien piensa que un ingeniero de Intel era supersticioso?)

errores #

Los tipos de errores que podrían ocurrir incluyen:

  • Si el índice de descriptores propuesto fuera más grande que el tamaño de la tabla;
  • Si el Descriptor propuesto era un Descriptor de sistema en lugar de Código, Datos o Pila;
  • Si el Descriptor propuesto fuera más privilegiado que el programa solicitante;
  • Si el Descriptor propuesto se marcó como No legible (como un segmento de código), pero se intentó leer en lugar de ejecutarlo;
  • Si el Descriptor propuesto fue marcado como No Presente.

Tenga en cuenta que el último puede no ser un problema fatal para el programa: el sistema operativo podría notar la bandera, restablecer el segmento, marcarlo como ahora Presente y luego permitir que la instrucción de falla continúe con éxito.

O, tal vez, el Descriptor se cargó con éxito en un Registro de segmento, pero luego un acceso futuro con él rompió una de varias reglas:

  • El registro de segmento se cargó con el índice de descriptor 0x0000 para la GDT. Esto fue reservado por el hardware como NULL;
  • Si el descriptor cargado se marcó como de solo lectura, pero se intentó escribirlo.
  • Si alguna parte del acceso (1, 2, 4 o más bytes) estuvo fuera del Límite del Segmento.

Cambiar al modo protegido #

Cambiar al modo protegido es fácil: solo necesita configurar un solo bit en un registro de control. Pero permanecer en modo protegido, sin que la CPU se dé por vencida y se reinicie por no saber qué hacer a continuación, requiere mucha preparación.

En resumen, los pasos necesarios son los siguientes:

  • Es necesario configurar un área de memoria para la tabla de descriptores globales para definir un mínimo de tres descriptores:
  1. El zeroeth, NULL Descriptor;
  2. Otro Descriptor para un Segmento de Código;
  3. Otro Descriptor para un Segmento de Datos.

    This can be used for both Data and Stack.

  • El registro de la tabla de descriptores globales (GDTR) debe inicializarse para apuntar a esta área definida de la memoria;

     GDT_Ptr    dw      SIZE GDT
            dd      OFFSET GDT

            ...

            lgdt    [GDT_Ptr]

  • El bit PM en CR0 debe configurarse:

         mov   eax, cr0      ; Get CR0 into register
     or    eax, 0x01     ; Set the Protected Mode bit
     mov   cr0, eax      ; We're now in Protected Mode!

  • Los registros de segmento deben cargarse desde la GDT para eliminar los valores actuales del modo real:

         jmp   0x0008:NowInPM  ; This is a FAR Jump. 0x0008 is the Code Descriptor

NowInPM:
     mov   ax, 0x0010      ; This is the Data Descriptor
     mov   ds, ax
     mov   es, ax
     mov   ss, ax
     mov   sp, 0x0000      ; Top of stack!

Tenga en cuenta que este es el mínimo desnudo, solo para poner la CPU en modo protegido. Para tener listo todo el sistema, es posible que se requieran muchos más pasos. Por ejemplo:

  • Es posible que se deban habilitar las áreas de memoria superior: apagar la puerta A20;
  • Las interrupciones definitivamente deberían estar deshabilitadas, pero tal vez los diversos controladores de fallas podrían configurarse antes de ingresar al modo protegido, para permitir errores al principio del procesamiento.

El autor original de esta sección escribió un tutorial completo sobre cómo ingresar al modo protegido y trabajar con él.

Modo irreal #

El modo irreal explota dos hechos sobre cómo los procesadores Intel y AMD cargan y guardan la información para describir un segmento.

  1. El procesador almacena en caché la información del descriptor obtenida durante un movimiento en un registro selector en modo protegido. Esta información se almacena en una parte arquitectónica invisible del registro del selector.

  2. En modo real los registros selectores se denominan registros de segmento pero, aparte de eso, designan el mismo conjunto de registros y como tal también tienen una parte invisible. Estas partes se llenan con valores fijos, pero para la base que se deriva del valor recién cargado.

Desde tal punto de vista, el modo real es solo un caso especial del modo protegido: donde la información de un segmento, como la base y el límite, se obtienen sin un GDT/LDT pero aún se leen de la parte oculta del registro del segmento.

Al cambiar en modo protegido y crear un GDT, es posible crear un segmento con los atributos deseados, por ejemplo, una base de 0 y un límite de 4GiB. A través de una carga sucesiva de un registro selector, dichos atributos se almacenan en caché, luego es posible volver al modo real y tener un registro de segmento a través del cual acceder a todo el espacio de direcciones de 32 bits.

BITS 16

jmp 7c0h:__START__

__START__:
 push cs
 pop ds
 push ds
 pop ss
 xor sp, sp

 
 lgdt [GDT]            ;Set the GDTR register
 

 cli                ;We don't have an IDT set, we can't handle interrupts


 ;Entering protected mode

 mov eax, cr0
 or ax, 01h            ;Set bit PE (bit 0) of CR0
 mov cr0, eax            ;Apply

 ;We are now in Protected mode

 mov bx, 08h           ;Selector to use, RPL = 0, Table = 0 (GDT), Index = 1

 mov fs, bx            ;Load FS with descriptor 1 info
 mov gs, bx            ;Load GS with descriptor 1 info

 ;Exit protected mode

 and ax, 0fffeh            ;Clear bit PE (bit0) of CR0
 mov cr0, eax                   ;Apply

 sti                

 ;Back to real mode

 ;Do nothing
 cli
 hlt 



 GDT:
    ;First entry, number 0
    ;Null descriptor
    ;Used to store a m16&32 object that tells the GDT start and size

    dw 0fh                 ;Size in byte -1 of the GDT (2 descriptors = 16 bytes)
    dd GDT + 7c00h         ;Linear address of GDT start (24 bits)
    dw 00h                 ;Pad 

    dd 0000ffffh           ;Base[15:00] = 0, Limit[15:00] = 0ffffh
    dd 00cf9200h           ;Base[31:24] = 0, G = 1, B = 1, Limit[19:16] = 0fh, 
               ;P = 1, DPL = 0, E = 0, W = 1, A = 0, Base[23:16] = 00h


 TIMES 510-($-$$) db 00h
 dw 0aa55h

Consideraciones

  • Tan pronto como se recarga un registro de segmento, incluso con el mismo valor, el procesador recarga los atributos ocultos de acuerdo con el modo actual. Esta es la razón por la que el código anterior usa fs y gs para contener los segmentos “extendidos”: es menos probable que dichos registros sean usados/guardados/restaurados por los diversos servicios de 16 bits.
  • La instrucción lgdt no carga un puntero lejano a la GDT, sino que carga una dirección lineal de 24 bits (se puede anular a 32 bits). Esta no es una dirección cercana, es la dirección física (ya que la paginación debe estar deshabilitada). Esa es la razón de GDT+7c00h.
  • El programa anterior es un gestor de arranque (para MBR, no tiene BPB) que establece cs/ds/ss tp 7c00h e inicia el contador de ubicación desde 0. Entonces, un byte en el desplazamiento X en el archivo está en el desplazamiento X en el segmento 7c00h y en la dirección lineal 7c00h + X.
  • Las interrupciones deben deshabilitarse ya que no se ha configurado un IDT para el viaje corto de ida y vuelta en modo protegido.
  • El código usa un truco para guardar 6 bytes de código. La estructura cargada por lgdt se guarda en la propia GDT, en el descriptor nulo (el primer descriptor).

Para obtener una descripción de los descriptores de GDT, consulte el Capítulo 3.4.3 del [Manual de Intel Volumen 3A] (https://www-ssl.intel.com/content/www/us/en/architecture-and-technology/64-ia- 32-architectures-software-developer-vol-3a-part-1-manual.html).
