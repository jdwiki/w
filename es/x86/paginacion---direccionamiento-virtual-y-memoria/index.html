Introducción #

Historia #

Las primeras computadoras #

Las primeras computadoras tenían un bloque de memoria en el que el programador ponía código y datos, y la CPU ejecutaba dentro de este entorno. Dado que las computadoras entonces eran muy costosas, era desafortunado que hiciera un trabajo, se detuviera y esperara a que se cargara el siguiente trabajo, y luego lo procesaría.

Multiusuario, multiprocesamiento #

Entonces, las computadoras rápidamente se volvieron más sofisticadas y admitieron múltiples usuarios y/o programas simultáneamente, pero fue entonces cuando comenzaron a surgir problemas con la idea simple de “un bloque de memoria”. Si una computadora estaba ejecutando dos programas simultáneamente, o ejecutando el mismo programa para varios usuarios, lo que por supuesto habría requerido datos separados para cada usuario, entonces la administración de esa memoria se volvió crítica.

Ejemplo #

Por ejemplo: si un programa fue escrito para trabajar en la dirección de memoria 1000, pero otro programa ya estaba cargado ahí, entonces el nuevo programa no podría cargarse. Una forma de resolver esto sería hacer que los programas funcionen con “direccionamiento relativo”: no importa dónde se cargó el programa, simplemente hizo todo lo relacionado con la dirección de memoria en la que se cargó. Pero eso requería soporte de hardware.

Sofisticación #

A medida que el hardware de la computadora se volvió más sofisticado, pudo admitir bloques de memoria más grandes, lo que permitió más programas simultáneos, y se volvió más complicado escribir programas que no interfirieran con lo que ya estaba cargado. Una referencia perdida a la memoria podría derribar no solo el programa actual, sino cualquier otro programa en la memoria, ¡incluido el propio sistema operativo!

Soluciones #

Lo que se necesitaba era un mecanismo que permitiera que los bloques de memoria tuvieran direcciones dinámicas. De esa manera, un programa podría escribirse para trabajar con sus bloques de memoria en direcciones que reconoció, y no poder acceder a otros bloques para otros programas (a menos que alguna cooperación lo permita).

Segmentación #

Un mecanismo que implementó esto fue la segmentación. Eso permitió definir bloques de memoria de todos los tamaños diferentes, y el programa necesitaría definir a qué segmento quería acceder todo el tiempo.

Problemas #

Esta técnica era poderosa, pero su misma flexibilidad era un problema. Dado que los segmentos esencialmente subdividían la memoria disponible en fragmentos de diferentes tamaños, la gestión de la memoria para esos segmentos era un problema: asignación, desasignación, crecimiento, reducción, fragmentación: todo requería rutinas sofisticadas y, a veces, copia masiva para implementar.

Paginación #

Una técnica diferente dividió toda la memoria en bloques de igual tamaño, llamados “Páginas”, lo que hizo que las rutinas de asignación y desasignación fueran muy simples y eliminó el crecimiento, la reducción y la fragmentación (excepto la fragmentación interna, que es simplemente un problema de pérdida).

Direccionamiento virtual #

Al dividir la memoria en estos bloques, podrían asignarse a diferentes programas según sea necesario con cualquier dirección en la que el programa lo necesite. Esta “asignación” entre la dirección física de la memoria y la dirección deseada del programa es muy poderosa y es la base para la administración de memoria de todos los principales procesadores (Intel, ARM, MIPS, Power et. al.) en la actualidad.

Soporte de hardware y sistema operativo #

El hardware realizaba la reasignación de forma automática y continua, pero requería memoria para definir las tablas de qué hacer. Por supuesto, la limpieza asociada con esta reasignación tenía que ser controlada por algo. El sistema operativo tendría que repartir la memoria según sea necesario y administrar las tablas de datos requeridas por el hardware para respaldar lo que requieren los programas.

Funciones de paginación #

Una vez que el hardware pudo hacer esta reasignación, ¿qué permitió? El principal impulsor fue el multiprocesamiento: la capacidad de ejecutar múltiples programas, cada uno con su “propia” memoria, protegidos entre sí. Pero otras dos opciones incluían “datos dispersos” y “memoria virtual”.

Multiprocesamiento #

A cada programa se le dio su propio “Espacio de direcciones” virtual, un rango de direcciones en las que podrían mapear la memoria física, en cualquier dirección que se desee. Siempre que hubiera suficiente memoria física para todos (aunque consulte “Memoria virtual” a continuación), se podrían admitir numerosos programas simultáneamente.

Además, esos programas no podían acceder a la memoria que no estaba asignada a su espacio de direcciones virtuales: la protección entre programas era automática. Si los programas necesitaban comunicarse, podían pedirle al sistema operativo que organizara un bloque de memoria compartido, un bloque de memoria física que se asignaba a los espacios de direcciones de dos programas diferentes simultáneamente.

Datos dispersos #

Permitir un gran espacio de direcciones virtuales (4 GB es típico, para corresponder con los registros de 32 bits que estos procesadores tenían normalmente) no desperdicia memoria en sí mismo, si grandes áreas de ese espacio de direcciones no se asignan. Esto permite la creación de enormes estructuras de datos en las que solo se mapean ciertas partes en un momento dado. Imagine una matriz tridimensional de 1000 bytes en cada dirección: ¡eso normalmente requeriría mil millones de bytes! Pero un programa podría reservar un bloque de su espacio de direcciones virtuales para “mantener” estos datos, pero solo mapear pequeñas secciones a medida que se completaron. Esto hace que la programación sea eficiente, sin desperdiciar memoria para datos que aún no se necesitan.

Memoria virtual #

Arriba utilicé el término “direccionamiento virtual” para describir el direccionamiento de virtual a físico realizado por el hardware. Esto a menudo se denomina “Memoria virtual”, pero ese término corresponde más correctamente a la técnica de usar el direccionamiento virtual para brindar la ilusión de más memoria de la que realmente está disponible.

Funciona así:

  • A medida que los programas se cargan y solicitan más memoria, el sistema operativo proporciona la memoria que tiene disponible. Además de realizar un seguimiento de qué memoria se ha asignado, el sistema operativo también realiza un seguimiento de cuándo se usa realmente la memoria: el hardware admite el marcado de páginas usadas.
  • Cuando el sistema operativo se queda sin memoria física, analiza toda la memoria que ya entregó para la página que se usó menos o no se usó por más tiempo. Guarda el contenido de esa página en particular en el disco duro, recuerda dónde estaba, lo marca como “No presente” en el hardware para el propietario original y luego pone a cero la página y se la entrega al nuevo propietario.
  • Si el propietario original intenta acceder a esa página nuevamente, el hardware notifica al sistema operativo. Luego, el sistema operativo asigna una nueva página (¡quizás tenga que hacer el paso anterior nuevamente!), Carga el contenido de la página anterior y luego entrega la nueva página al programa original.

El punto importante a tener en cuenta es que, dado que cualquier página se puede asignar a cualquier dirección, y cada página tiene el mismo tamaño, entonces una página es tan buena como cualquier otra, ¡siempre que el contenido sea el mismo!

  • Si un programa accede a una ubicación de memoria no asignada, el hardware notifica al sistema operativo como antes. Esta vez, el sistema operativo observa que no era una página que se había guardado, por lo que la reconoce como un error en el programa y la finaliza.

Esto es realmente lo que sucede cuando su aplicación desaparece misteriosamente de usted, tal vez con un cuadro de mensaje del sistema operativo. También es lo que (a menudo) causa una infame pantalla azul o una Mac triste: ¡el programa defectuoso era de hecho un controlador del sistema operativo que accedía a la memoria que no debería!

Decisiones de paginación #

Los arquitectos de hardware necesitaban tomar algunas decisiones importantes sobre la paginación, ya que el diseño afectaría directamente el diseño de la CPU. Un sistema muy flexible tendría una gran sobrecarga y requeriría grandes cantidades de memoria solo para administrar la infraestructura de paginación en sí.

¿Qué tamaño debe tener una página? #

En hardware, la implementación más sencilla de Paginación sería tomar una Dirección y dividirla en dos partes. La parte superior sería un indicador de a qué página acceder, mientras que la parte inferior sería el índice de la página para el byte requerido:

+-----------------+------------+
| Page index      | Byte index |
+-----------------+------------+

Sin embargo, rápidamente se hizo evidente que las páginas pequeñas requerirían grandes índices para cada programa: incluso la memoria que no estaba asignada necesitaría una entrada en la tabla que lo indicara.

Entonces, en su lugar, se usa un índice de varios niveles. La dirección se divide en varias partes (tres se indican en el siguiente ejemplo), y la parte superior (comúnmente llamada “Directorio”) se indexa en la siguiente parte y así sucesivamente hasta que se decodifica el índice de bytes final en la página final:

+-----------+------------+------------+
| Dir index | Page index | Byte index |
+-----------+------------+------------+

Eso significa que un índice de directorio puede indicar “no asignado” para una gran parte del espacio de direcciones, sin necesidad de numerosos índices de página.

¿Cómo optimizar el uso de las tablas de páginas? #

Cada acceso a la dirección que realizará la CPU deberá mapearse; por lo tanto, el proceso de virtual a físico debe ser lo más eficiente posible. Si se implementara el sistema de tres niveles descrito anteriormente, eso significaría que cada acceso a la memoria sería en realidad tres accesos: uno al Directorio; uno en la tabla de páginas; y finalmente los datos deseados en sí. Y si la CPU también necesitara realizar tareas de mantenimiento, como indicar que ahora se ha accedido o escrito en esta página, entonces eso requeriría aún más accesos para actualizar los campos.

La memoria puede ser rápida, ¡pero esto impondría una ralentización triple en todos los accesos a la memoria durante la paginación! Afortunadamente, la mayoría de los programas tienen una “localidad de alcance”, es decir, si acceden a una ubicación en la memoria, es probable que los accesos futuros estén cerca. Y dado que las páginas no son demasiado pequeñas, esa conversión de mapeo solo debería realizarse cuando se accedió a una nueva página: no para absolutamente todos los accesos.

Pero aún mejor sería implementar un caché de páginas a las que se accedió recientemente, no solo la más actual. El problema sería mantenerse al día con las páginas a las que se había accedido y a las que no: el hardware tendría que escanear el caché en cada acceso para encontrar el valor almacenado en caché. Por lo tanto, el caché se implementa como un caché direccionable por contenido: en lugar de acceder por dirección, se accede por contenido: si los datos solicitados están presentes, se ofrecen; de lo contrario, se marca una ubicación vacía para completar. El caché gestiona todo eso.

Esta memoria caché direccionable por contenido a menudo se denomina Búfer de búsqueda de traducción (TLB) y debe ser administrada por el sistema operativo como parte del subsistema de direccionamiento virtual. Cuando el sistema operativo modifica los directorios o las tablas de páginas, debe notificar a la TLB que actualice sus entradas, o simplemente que las invalide.

80386 Megafonía #

Diseño de alto nivel #

El 80386 es un procesador de 32 bits, con un espacio de memoria direccionable de 32 bits. Los diseñadores del subsistema de paginación notaron que un diseño de página 4K se mapeó en esos 32 bits de una manera bastante clara: 10 bits, 10 bits y 12 bits:

+-----------+------------+------------+
| Dir index | Page index | Byte index |
+-----------+------------+------------+
 3         2 2          1 1          0  Bit
 1         2 1          2 1          0  number

Eso significaba que el índice de bytes tenía 12 bits de ancho, lo que se indexaría en una página de 4K. Los índices de directorio y página eran de 10 bits, cada uno de los cuales se asignaría a una tabla de 1024 entradas, y si esas entradas de la tabla tuvieran 4 bytes cada una, serían 4 K por tabla: ¡también una página!

Así que eso fue lo que hicieron:

  • Cada programa tendría su propio directorio, una página con 1024 entradas de página, cada una de las cuales definía dónde estaba la tabla de página del siguiente nivel, si la hubiera.
  • Si la hubiera, esa tabla de páginas tendría 1024 entradas de página, cada una de las cuales definiría dónde estaba la página del último nivel, si es que había una.
  • Si lo hubiera, entonces esa Página podría tener su Byte directamente leído.

entrada de página #

Tanto el directorio de nivel superior como la tabla de páginas del siguiente nivel se componen de 1024 entradas de página. La parte más importante de estas entradas es la dirección de lo que está indexando: una tabla de páginas o una página real. Tenga en cuenta que esta dirección no necesita los 32 bits completos; dado que todo es una página, solo los 20 bits superiores son significativos. Por lo tanto, los otros 12 bits en la entrada de página se pueden usar para otras cosas: si el siguiente nivel está presente; limpieza en cuanto a si se ha accedido o escrito en la página; ¡e incluso si se deben permitir las escrituras!

+--------------+----+------+-----+---+---+
| Page Address | OS | Used | Sup | W | P |
+--------------+----+------+-----+---+---+
Page Address = Top 20 bits of Page Table or Page address
OS           = Available for OS use
Used         = Whether this page has been accessed or written to
Sup          = Whether this page is Supervisory - only accessible by the OS
W            = Whether this page is allowed to be Written
P            = Whether this page is even Present

Tenga en cuenta que si el bit P es 0, entonces el resto de la entrada puede tener cualquier cosa que el sistema operativo quiera poner allí, ¡como dónde debería estar el contenido de la página en el disco duro!

Registro base del directorio de páginas (PDBR) #

Si cada programa tiene su propio directorio, ¿cómo sabe el hardware dónde comenzar a mapear? Dado que la CPU solo ejecuta un programa a la vez, tiene un solo registro de control para contener la dirección del directorio del programa actual. Este es el registro base del directorio de páginas (CR3). A medida que el sistema operativo cambia entre diferentes programas, actualiza el PDBR con el directorio de páginas relevante para el programa.

fallas de página #

Cada vez que la CPU accede a la memoria, tiene que asignar la dirección virtual indicada a la dirección física adecuada. Este es un proceso de tres pasos:

  1. Indexe los 10 bits superiores de la dirección en la página indicada por el PDBR para obtener la dirección de la tabla de páginas adecuada;
  2. Indexe los siguientes 10 bits de la dirección en la Página indicada por el Directorio para obtener la dirección de la Página apropiada;
  3. Indexe los últimos 12 bits de la dirección para obtener los datos de esa página.

Debido a que los pasos 1 y 2 anteriores usan entradas de página, cada entrada podría indicar un problema:

  • El siguiente nivel puede estar marcado como “No presente”;
  • El siguiente nivel puede marcarse como “Solo lectura” y la operación es una Escritura;
  • El siguiente nivel puede estar marcado como “Supervisor”, y es el programa el que accede a la memoria, no el sistema operativo.

Cuando el hardware detecta un problema de este tipo, en lugar de completar el acceso, genera un error: Interrupción n.º 14, el error de página. También llena unos Registros de Control específicos con la información de por qué ocurrió la Falla: la dirección referenciada; si era un acceso de supervisor; y si fue un intento de escritura.

Se espera que el sistema operativo atrape esa falla, decodifique los registros de control y decida qué hacer. Si se trata de un acceso no válido, puede terminar el programa defectuoso. Sin embargo, si se trata de un acceso a la memoria virtual, el sistema operativo debe asignar una nueva página (que puede necesitar desocupar una página que ya está en uso), llenarla con los contenidos requeridos (ya sea todos ceros o el contenido anterior cargado desde el disco ), asigne la nueva página a la tabla de páginas adecuada, márquela como presente y luego reanude la instrucción de fallo. Esta vez, el acceso progresará con éxito y el programa procederá sin saber que sucedió algo especial (¡a menos que eche un vistazo al reloj!)

80486 Megafonía #

El subsistema de paginación 80486 era muy similar al 80386. Era compatible con versiones anteriores, y las únicas características nuevas eran permitir el control de la caché de memoria página por página: los diseñadores del sistema operativo podían marcar páginas específicas para que no se almacenaran en caché, o usar diferentes escritura simultánea o reescritura. técnicas de almacenamiento en caché.

En todos los demás aspectos, se aplica el ejemplo “80386 Paginación”.

Paginación Pentium #

Cuando se estaba desarrollando el Pentium, el tamaño de la memoria y los programas que se ejecutaban en ella aumentaban. El sistema operativo tuvo que hacer más y más trabajo para mantener el subsistema de paginación solo en la gran cantidad de índices de página que necesitaban actualizarse cuando se usaban grandes programas o conjuntos de datos.

Entonces, los diseñadores de Pentium agregaron un truco simple: pusieron un bit adicional en las entradas del directorio de páginas que indicaba si el siguiente nivel era una tabla de páginas (como antes), ¡o iba directamente a una página de 4 MB! Al tener el concepto de páginas de 4 MB, el sistema operativo no tendría que crear una tabla de páginas y llenarla con 1024 entradas que básicamente indexaban direcciones 4K más altas que la anterior.

Diseño de dirección #

+-----------+----------------------+
| Dir Index | 4MB Byte Index       |
+-----------+----------------------+
 3         2 2                    0   Bit
 1         2 1                    0   number

Diseño de entrada de directorio #

+-----------+----+---+------+-----+---+---+
| Page Addr | OS | S | Used | Sup | W | P |
+-----------+----+---+------+-----+---+---+
Page Addr = Top 20 bits of Page Table or Page address
OS        = Available for OS use
S         = Size of Next Level: 0 = Page Table, 1 = 4 MB Page
Used      = Whether this page has been accessed or written to
Sup       = Whether this page is Supervisory - onlly accessible by the OS
W         = Whether this page is allowed to be Written
P         = Whether this page is even Present

Por supuesto, eso tuvo algunas ramificaciones:

  • La página de 4 MB tenía que comenzar en un límite de dirección de 4 MB, al igual que las páginas de 4K tenían que comenzar en un límite de dirección de 4K.
  • Los 4 MB tenían que pertenecer a un solo programa, o ser compartidos por varios.

Esto era perfecto para usar con periféricos de gran memoria, como adaptadores de gráficos, que tenían grandes ventanas de espacio de direcciones que debían mapearse para que las usara el sistema operativo.

Extensión de dirección física (PAE) #

Introducción #

A medida que bajaban los precios de la memoria, las PC basadas en Intel podían tener más y más RAM de manera asequible, lo que aliviaba los problemas de muchos usuarios con la ejecución de muchas de las aplicaciones cada vez más grandes que se producían simultáneamente. Si bien la memoria virtual permitió que la memoria se “creara” virtualmente, intercambiando los contenidos de la página “antiguos” existentes en el disco duro para permitir que se almacenaran datos “nuevos”, esto ralentizó la ejecución de los programas ya que la página “golpeaba” continuamente intercambiando datos. encendido y apagado del disco duro.

Más RAM #

Lo que se necesitaba era la capacidad de acceder a más RAM física, pero ya era un bus de direcciones de 32 bits, por lo que cualquier aumento requeriría registros de direcciones más grandes. ¿O lo sería? Al desarrollar el Pentium Pro (e incluso el Pentium M), como medida provisional hasta que se pudieran producir procesadores de 64 bits, agregar más bits de dirección física (permitiendo más memoria física) sin cambiar el número de bits de registro. Esto podría lograrse ya que las direcciones virtuales se asignaron a direcciones físicas de todos modos; todo lo que se necesitaba cambiar era el sistema de asignación.

Diseño #

El sistema existente podría acceder a un máximo de 32 bits de direcciones físicas. Aumentar esto requirió un cambio completo de la estructura de entrada de página, de 32 a 64 bits. Se decidió mantener la granularidad mínima en 4K Páginas, por lo que la Entrada de 64 bits tendría 52 bits de Dirección y 12 bits de Control (al igual que la Entrada anterior tenía 20 bits de Dirección y 12 bits de Control).

Tener una entrada de 64 bits, pero un tamaño de página de (todavía) 4K, significaba que solo habría 512 entradas por tabla de página o directorio, en lugar de las 1024 anteriores. Eso significaba que la dirección virtual de 32 bits se dividiría de manera diferente que antes:

+-----+-----------+------------+------------+
| DPI | Dir Index | Page Index | Byte Index |
+-----+-----------+------------+------------+
 3   3 2         2 2          1 1          0   Bit
 1   0 9         1 0          2 1          0   number

 DPI        = 2-bit index into Directory Pointer Table
 Dir Index  = 9-bit index into Directory
 Page Index = 9-bit index into Page Table
 Byte Index = 12-bit index into Page (as before)

Cortar un bit tanto del índice de directorio como del índice de página dio dos bits para un tercer nivel de asignación: lo llamaron tabla de puntero de directorio de página (PDPT), una tabla de exactamente cuatro entradas de 64 bits que se dirigían a cuatro directorios en lugar del anterior. una. El PDBR (CR3) ahora apuntaba al PDPT en su lugar, que, dado que CR3 tenía solo 32 bits, necesitaba almacenarse en los primeros 4 GB de RAM para accesibilidad. Tenga en cuenta que dado que los bits bajos de CR3 se utilizan para el control, el PDPT debe comenzar en un límite de 32 bytes.

Extensión de tamaño de página (PSE) #

Y, dado que las páginas anteriores de 4 MB eran una buena idea, querían poder admitir páginas grandes nuevamente. Esta vez, sin embargo, la eliminación de la última capa del sistema de niveles no produjo páginas de 4 MB de 10+12 bits, sino páginas de 2 MB de 9+12 bits en su lugar.

PSE-32 (y PSE-40) #

Dado que el modo de extensión de dirección física (PAE) que se introdujo en Pentium Pro (y Pentum M) supuso un gran cambio en el subsistema de gestión de memoria del sistema operativo, cuando Intel diseñó el Pentium II, decidió mejorar el modo de página “normal” para admitir los nuevos bits de dirección física del procesador dentro de las entradas de 32 bits definidas previamente.

Se dieron cuenta de que cuando se usaba una página de 4 MB, la entrada de directorio se veía así:

+-----------+------------+---------+
| Dir Index |  Unused    | Control |
+-----------+------------+---------+

Las áreas de control e índice de directorios de la entrada eran las mismas, pero el bloque de bits no utilizados entre ellas, que sería utilizado por el índice de página si existiera, se desperdició. ¡Entonces decidieron usar esa área para definir los bits superiores de dirección física por encima de 31!

+-----------+------+-----+---------+
| Dir Index |Unused|Upper| Control |
+-----------+------+-----+---------+

Esto permitió que los sistemas operativos que no adoptaron el modo PAE tuvieran acceso a la RAM por encima de 4 GB; con un poco de lógica adicional, podrían proporcionar grandes cantidades de RAM adicional al sistema, aunque no más de los 4 GB normales para cada programa. Al principio, solo se agregaron 4 bits, lo que permitía el direccionamiento físico de 36 bits, por lo que este modo se denominó Extensión de tamaño de página 36 (PSE-36). Sin embargo, en realidad no cambió el tamaño de la página, solo el direccionamiento.

Sin embargo, la limitación de esto era que solo se podían definir páginas de 4 MB por encima de 4 GB; no se permitían páginas de 4K. La adopción de este modo [no fue amplia] 1: según los informes, era más lento que usar PAE, y Linux nunca terminó usándolo.

Sin embargo, en procesadores posteriores que tenían aún más bits de dirección física, tanto AMD como Intel ampliaron el área de PSE a 8 bits, que algunas personas denominaron “PSE-40”.
