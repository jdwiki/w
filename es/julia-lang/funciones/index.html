Sintaxis #

  • f(n) = …
  • función f(n) … fin
  • n::Tipo -x->…
  • f(n) hacer… fin

Además de las funciones genéricas (que son las más comunes), también existen funciones integradas. Tales funciones incluyen is, isa, typeof, throw y funciones similares. Las funciones integradas generalmente se implementan en C en lugar de Julia, por lo que no se pueden especializar en tipos de argumentos para el envío.

Introducción al Despacho #

Podemos usar la sintaxis :: para enviar el tipo de un argumento.

describe(n::Integer) = "integer $n"
describe(n::AbstractFloat) = "floating point $n"

Uso:

julia> describe(10)
"integer 10"

julia> describe(1.0)
"floating point 1.0"

A diferencia de muchos lenguajes, que normalmente proporcionan un envío múltiple estático o un envío único dinámico, Julia tiene un envío múltiple dinámico completo. Es decir, las funciones se pueden especializar para más de un argumento. Esto es útil cuando se definen métodos especializados para operaciones en ciertos tipos y métodos alternativos para otros tipos.

describe(n::Integer, m::Integer) = "integers n=$n and m=$m"
describe(n, m::Integer) = "only m=$m is an integer"
describe(n::Integer, m) = "only n=$n is an integer"

Uso:

julia> describe(10, 'x')
"only n=10 is an integer"

julia> describe('x', 10)
"only m=10 is an integer"

julia> describe(10, 10)
"integers n=10 and m=10"

Argumentos opcionales #

Julia permite que las funciones tomen argumentos opcionales. Detrás de escena, esto se implementa como otro caso especial de despacho múltiple. Por ejemplo, resolvamos el popular problema de Fizz Buzz. Por defecto, lo haremos para números en el rango 1:10, pero permitiremos un valor diferente si es necesario. También permitiremos que se usen diferentes frases para Fizz o Buzz.

function fizzbuzz(xs=1:10, fizz="Fizz", buzz="Buzz")
    for i in xs
        if i % 15 == 0
            println(fizz, buzz)
        elseif i % 3 == 0
            println(fizz)
        elseif i % 5 == 0
            println(buzz)
        else
            println(i)
        end
    end
end

Si inspeccionamos fizzbuzz en el REPL, dice que hay cuatro métodos. Se creó un método para cada combinación de argumentos permitida.

julia> fizzbuzz
fizzbuzz (generic function with 4 methods)

julia> methods(fizzbuzz)
# 4 methods for generic function "fizzbuzz":
fizzbuzz() at REPL[96]:2
fizzbuzz(xs) at REPL[96]:2
fizzbuzz(xs, fizz) at REPL[96]:2
fizzbuzz(xs, fizz, buzz) at REPL[96]:2

Podemos verificar que nuestros valores predeterminados se utilizan cuando no se proporcionan parámetros:

julia> fizzbuzz()
1
2
Fizz
4
Buzz
Fizz
7
8
Fizz
Buzz

pero que los parámetros opcionales son aceptados y respetados si los proporcionamos:

julia> fizzbuzz(5:8, "fuzz", "bizz")
bizz
fuzz
7
8

Envío paramétrico #

Suele suceder que una función deba enviar tipos paramétricos, como Vector{T} o Dict{K,V}, pero los parámetros de tipo no son fijos. Este caso se puede tratar mediante el envío paramétrico:

julia> foo{T<:Number}(xs::Vector{T}) = @show xs .+ 1
foo (generic function with 1 method)

julia> foo(xs::Vector) = @show xs
foo (generic function with 2 methods)

julia> foo([1, 2, 3])
xs .+ 1 = [2,3,4]
3-element Array{Int64,1}:
 2
 3
 4

julia> foo([1.0, 2.0, 3.0])
xs .+ 1 = [2.0,3.0,4.0]
3-element Array{Float64,1}:
 2.0
 3.0
 4.0

julia> foo(["x", "y", "z"])
xs = String["x","y","z"]
3-element Array{String,1}:
 "x"
 "y"
 "z"

Uno puede tener la tentación de simplemente escribir xs::Vector{Number}. Pero esto solo funciona para objetos cuyo tipo es explícitamente Vector{Número}:

julia> isa(Number[1, 2], Vector{Number})
true

julia> isa(Int[1, 2], Vector{Number})
false

Esto se debe a la [invariancia paramétrica][4]: el objeto Int[1, 2] no es un Vector{Número}, porque solo puede contener Ints, mientras que un Vector{Número} se esperaría que pudiera contener cualquier tipo de números.

Escribir código genérico #

Dispatch es una característica increíblemente poderosa, pero con frecuencia es mejor escribir código genérico que funcione para todos los tipos, en lugar de código especializado para cada tipo. Escribir código genérico evita la duplicación de código.

Por ejemplo, aquí hay un código para calcular la suma de los cuadrados de un vector de números enteros:

function sumsq(v::Vector{Int})
    s = 0
    for x in v
        s += x ^ 2
    end
    s
end

Pero este código solo funciona para un vector de Ints. No funcionará en un UnitRange:

julia> sumsq(1:10)
ERROR: MethodError: no method matching sumsq(::UnitRange{Int64})
Closest candidates are:
  sumsq(::Array{Int64,1}) at REPL[8]:2

No funcionará en un Vector{Float64}:

julia> sumsq([1.0, 2.0])
ERROR: MethodError: no method matching sumsq(::Array{Float64,1})
Closest candidates are:
  sumsq(::Array{Int64,1}) at REPL[8]:2

Una mejor manera de escribir esta función sumsq debería ser

function sumsq(v::AbstractVector)
    s = zero(eltype(v))
    for x in v
        s += x ^ 2
    end
    s
end

Esto funcionará en los dos casos enumerados anteriormente. Pero hay algunas colecciones de las que podríamos querer sumar los cuadrados que no son vectores en absoluto, en ningún sentido. Por ejemplo,

julia> sumsq(take(countfrom(1), 100))
ERROR: MethodError: no method matching sumsq(::Base.Take{Base.Count{Int64}})
Closest candidates are:
  sumsq(::Array{Int64,1}) at REPL[8]:2
  sumsq(::AbstractArray{T,1}) at REPL[11]:2

muestra que no podemos sumar los cuadrados de un [iterable perezoso][5].

Una implementación aún más genérica es simplemente

function sumsq(v)
    s = zero(eltype(v))
    for x in v
        s += x ^ 2
    end
    s
end

Que funciona en todos los casos:

julia> sumsq(take(countfrom(1), 100))
338350

Este es el código de Julia más idiomático y puede manejar todo tipo de situaciones. En algunos otros idiomas, la eliminación de anotaciones de tipo puede afectar el rendimiento, pero ese no es el caso en Julia; solo la [estabilidad del tipo][6] es importante para el rendimiento.

Elevar al cuadrado un número #

Esta es la sintaxis más fácil para definir una función:

square(n) = n * n

Para llamar a una función, use corchetes (sin espacios entre ellos):

julia> square(10)
100

Las funciones son objetos en Julia, y podemos mostrarlas en REPL como con cualquier otro objeto:

julia> square
square (generic function with 1 method)

Todas las funciones de Julia son genéricas (también conocidas como polimórficas) de forma predeterminada. Nuestra función square funciona igual de bien con valores de punto flotante:

julia> square(2.5)
6.25

…o incluso matrices:

julia> square([2 4
               2 1])
2×2 Array{Int64,2}:
 12  12
  6   9

Funciones recursivas #

Recurrencia simple #

Usando la recursividad y el operador condicional ternario, podemos crear una implementación alternativa de la función factorial incorporada:

myfactorial(n) = n == 0 ? 1 : n * myfactorial(n - 1)

Uso:

julia> myfactorial(10)
3628800

Trabajar con árboles #

Las funciones recursivas suelen ser más útiles en estructuras de datos, especialmente estructuras de datos de árbol. Dado que las expresiones en Julia son estructuras de árbol, la recursividad puede ser muy útil para la metaprogramación. Por ejemplo, la siguiente función reúne un conjunto de todos los encabezados utilizados en una expresión.

heads(ex::Expr) = reduce(∪, Set((ex.head,)), (heads(a) for a in ex.args))
heads(::Any) = Set{Symbol}()

Podemos comprobar que nuestra función funciona según lo previsto:

julia> heads(:(7 + 4x > 1 > A[0]))
Set(Symbol[:comparison,:ref,:call])

Esta función es compacta y utiliza una variedad de técnicas más avanzadas, como reduce [función de orden superior][4], el tipo de datos Set y expresiones generadoras.

Factorial imperativo #

Una sintaxis de formato largo está disponible para definir funciones de varias líneas. Esto puede ser útil cuando usamos estructuras imperativas como bucles. Se devuelve la expresión en la posición de cola. Por ejemplo, la siguiente función usa un bucle for para calcular el factorial de algún entero n:

function myfactorial(n)
    fact = one(n)
    for m in 1:n
        fact *= m
    end
    fact
end

Uso:

julia> myfactorial(10)
3628800

En funciones más largas, es común ver que se usa la instrucción return. La instrucción return no es necesaria en la posición de cola, pero todavía se usa a veces para mayor claridad. Por ejemplo, otra forma de escribir la función anterior sería

function myfactorial(n)
    fact = one(n)
    for m in 1:n
        fact *= m
    end
    return fact
end

que es idéntica en comportamiento a la función anterior.

Funciones anónimas #

Sintaxis de flecha #

Las funciones anónimas se pueden crear utilizando la sintaxis ->. Esto es útil para pasar funciones a funciones de orden superior, como la función map. La siguiente función calcula el cuadrado de cada número en una matriz A.

squareall(A) = map(x -> x ^ 2, A)

Un ejemplo del uso de esta función:

julia> squareall(1:10)
10-element Array{Int64,1}:
   1
   4
   9
  16
  25
  36
  49
  64
  81
 100

Sintaxis multilínea #

Las funciones anónimas de varias líneas se pueden crear utilizando la sintaxis función. Por ejemplo, el siguiente ejemplo calcula los [factoriales][4] de los primeros n números, pero usando una función anónima en lugar del factorial incorporado.

julia> map(function (n)
               product = one(n)
               for i in 1:n
                   product *= i
               end
               product
           end, 1:10)
10-element Array{Int64,1}:
       1
       2
       6
      24
     120
     720
    5040
   40320
  362880
 3628800

Hacer bloque de sintaxis #

Debido a que es tan común pasar una función anónima como el primer argumento de una función, existe una sintaxis de bloque do. la sintaxis

map(A) do x
    x ^ 2
end

es equivalente a

map(x -> x ^ 2, A)

pero lo primero puede ser más claro en muchas situaciones, especialmente si se realizan muchos cálculos en la función anónima. La sintaxis del bloque do es especialmente útil para [entrada y salida de archivos][5] por motivos de gestión de recursos.
