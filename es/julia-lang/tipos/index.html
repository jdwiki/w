Sintaxis #

  • MyType inmutable; campo; campo; final
  • escriba MiTipo; campo; campo; final

Los tipos son clave para el desempeño de Julia. Una idea importante para el rendimiento es la estabilidad de tipos, que ocurre cuando el tipo que devuelve una función solo depende de los tipos, no de los valores, de sus argumentos.

Despacho por tipos #

En Julia, puede definir más de un método para cada función. Supongamos que definimos tres métodos de la misma función:

foo(x) = 1
foo(x::Number) = 2
foo(x::Int) = 3

Al decidir qué método usar (llamado dispatch), Julia elige el método más específico que coincida con los tipos de argumentos:

julia> foo('one')
1

julia> foo(1.0)
2

julia> foo(1)
3

Esto facilita el [polimorfismo] (https://en.wikipedia.org/wiki/Polymorphism_(computer_science)). Por ejemplo, podemos crear fácilmente una [lista enlazada] (https://en.wikipedia.org/wiki/Linked_list) definiendo dos tipos inmutables, llamados Nil y Cons. Estos nombres se usan tradicionalmente para describir una lista vacía y una lista no vacía, respectivamente.

abstract LinkedList
immutable Nil <: LinkedList end
immutable Cons <: LinkedList
    first
    rest::LinkedList
end

Representaremos la lista vacía con Nil() y cualquier otra lista con Cons(first, rest), donde first es el primer elemento de la lista enlazada y rest es la lista enlazada que consta de todos los elementos restantes. elementos. Por ejemplo, la lista [1, 2, 3] se representará como

julia> Cons(1, Cons(2, Cons(3, Nil())))
Cons(1,Cons(2,Cons(3,Nil())))

¿La lista está vacía? #

Supongamos que queremos extender la función isempty de la biblioteca estándar, que funciona en una variedad de colecciones diferentes:

julia> methods(isempty)
# 29 methods for generic function "isempty":
isempty(v::SimpleVector) at essentials.jl:180
isempty(m::Base.MethodList) at reflection.jl:394
...

Simplemente podemos usar la sintaxis de envío de la función y definir dos métodos adicionales de isempty. Dado que esta función es del módulo Base, tenemos que calificarla como Base.isempty para extenderla.

Base.isempty(::Nil) = true
Base.isempty(::Cons) = false

Aquí, no necesitamos los valores de los argumentos para determinar si la lista está vacía. Simplemente el tipo por sí solo es suficiente para calcular esa información. Julia nos permite omitir los nombres de los argumentos, manteniendo solo su anotación de tipo, si no necesitamos usar sus valores.

Podemos probar que nuestros métodos isempty funcionan:

julia> using Base.Test

julia> @test isempty(Nil())
Test Passed
  Expression: isempty(Nil())

julia> @test !isempty(Cons(1, Cons(2, Cons(3, Nil()))))
Test Passed
  Expression: !(isempty(Cons(1,Cons(2,Cons(3,Nil())))))

y, de hecho, el número de métodos para isempty ha aumentado en 2:

julia> methods(isempty)
# 31 methods for generic function "isempty":
isempty(v::SimpleVector) at essentials.jl:180
isempty(m::Base.MethodList) at reflection.jl:394

Claramente, determinar si una lista enlazada está vacía o no es un ejemplo trivial. Pero lleva a algo más interesante:

¿Cuánto dura la lista? #

La función longitud de la biblioteca estándar nos da la longitud de una colección o ciertos iterables. Hay muchas formas de implementar longitud para una lista enlazada. En particular, usar un bucle while es probablemente el más rápido y el más eficiente en términos de memoria en Julia. Pero se debe evitar la [optimización prematura] (https://en.wikipedia.org/wiki/Program_optimization#When_to_optimize), así que supongamos por un segundo que nuestra lista vinculada no necesita ser eficiente. ¿Cuál es la forma más sencilla de escribir una función longitud?

Base.length(::Nil) = 0
Base.length(xs::Cons) = 1 + length(xs.rest)

La primera definición es sencilla: una lista vacía tiene una longitud 0. La segunda definición también es fácil de leer: para contar la longitud de una lista, contamos el primer elemento y luego contamos la longitud del resto de la lista. Podemos probar este método de manera similar a como probamos isempty:

julia> @test length(Nil()) == 0
Test Passed
  Expression: length(Nil()) == 0
   Evaluated: 0 == 0

julia> @test length(Cons(1, Cons(2, Cons(3, Nil())))) == 3
Test Passed
  Expression: length(Cons(1,Cons(2,Cons(3,Nil())))) == 3
   Evaluated: 3 == 3

Este ejemplo de juguete está bastante lejos de implementar toda la funcionalidad que se desearía en una lista enlazada. Falta, por ejemplo, la interfaz de iteración. Sin embargo, ilustra cómo se puede usar el envío para escribir código corto y claro.

Tipos inmutables #

El tipo compuesto más simple es un tipo inmutable. Las instancias de tipos inmutables, como tuplas, son valores. Sus campos no se pueden cambiar después de que se crean. En muchos sentidos, un tipo inmutable es como una ‘Tupla’ con nombres para el tipo en sí y para cada campo.

Tipos de singleton #

Los tipos compuestos, por definición, contienen varios tipos más simples. En Julia, este número puede ser cero; es decir, se permite que un tipo inmutable contenga no campos. Esto es comparable a la tupla vacía ().

¿Por qué podría ser útil? Estos tipos inmutables se conocen como “tipos singleton”, ya que solo podría existir una instancia de ellos. Los valores de tales tipos se conocen como “valores singleton”. La biblioteca estándar Base contiene muchos de estos tipos singleton. Aquí hay una breve lista:

  • Void, el tipo de nada. Podemos verificar que Void.instance (que es una sintaxis especial para recuperar el valor singleton de un tipo singleton) es de hecho nada.
  • Cualquier tipo de medio, como MIME"text/plain", es un tipo singleton con una única instancia, MIME("text/plain").
  • Los tipos Irrational{:π}, Irrational{:e}, Irrational{:φ} y similares son tipos singleton, y sus instancias singleton son los valores irracionales π = 3.1415926535897..., etc.
  • Los rasgos de tamaño del iterador Base.HasLength, Base.HasShape, Base.IsInfinite y Base.SizeUnknown son todos tipos singleton.
  • ¡En la versión 0.5 y posteriores, cada función es una instancia única de un tipo único! Como cualquier otro valor singleton, podemos recuperar la función sin, por ejemplo, desde typeof(sin).instance.

Debido a que no contienen nada, los tipos singleton son increíblemente livianos y, con frecuencia, el compilador puede optimizarlos para que no tengan una sobrecarga de tiempo de ejecución. Por lo tanto, son perfectos para rasgos, valores de etiquetas especiales y para cosas como funciones en las que a uno le gustaría especializarse.

Para definir un tipo singleton,

julia> immutable MySingleton end

Para definir la impresión personalizada para el tipo singleton,

julia> Base.show(io::IO, ::MySingleton) = print(io, "sing")

Para acceder a la instancia singleton,

julia> MySingleton.instance
MySingleton()

A menudo, uno asigna esto a una constante:

julia> const sing = MySingleton.instance
MySingleton()

Tipos de envoltorios #

Si los tipos inmutables de campo cero son interesantes y útiles, quizás los tipos inmutables de un campo sean aún más útiles. Dichos tipos se denominan comúnmente “tipos de contenedor” porque envuelven algunos datos subyacentes, proporcionando una interfaz alternativa para dichos datos. Un ejemplo de un tipo contenedor en Base es String. Definiremos un tipo similar a String, llamado MyString. Este tipo estará respaldado por un vector (unidimensional array) de bytes (UInt8).

Primero, la definición del tipo en sí y algunas demostraciones personalizadas:

immutable MyString <: AbstractString
    data::Vector{UInt8}
end

function Base.show(io::IO, s::MyString)
    print(io, "MyString: ")
    write(io, s.data)
    return
end

¡Ahora nuestro tipo MyString está listo para usar! Podemos alimentarlo con algunos datos UTF-8 sin procesar, y se muestra como nos gusta:

julia> MyString([0x48,0x65,0x6c,0x6c,0x6f,0x2c,0x20,0x57,0x6f,0x72,0x6c,0x64,0x21])
MyString: Hello, World!

Obviamente, este tipo de cadena necesita mucho trabajo antes de que sea tan utilizable como el tipo Base.String.

Tipos compuestos verdaderos #

Quizás lo más común es que muchos tipos inmutables contengan más de un campo. Un ejemplo es el tipo de biblioteca estándar Rational{T}, que contiene dos campos: un campo num para el numerador y un campo den para el denominador. Es bastante sencillo emular este diseño tipográfico:

immutable MyRational{T}
    num::T
    den::T
    MyRational(n, d) = (g = gcd(n, d); new(n÷g, d÷g))
end
MyRational{T}(n::T, d::T) = MyRational{T}(n, d)

Hemos implementado con éxito un constructor que simplifica nuestros números racionales:

julia> MyRational(10, 6)
MyRational{Int64}(5,3)
