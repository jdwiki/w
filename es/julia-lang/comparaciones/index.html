Sintaxis #

  • x < y # si x es estrictamente menor que y
  • x > y # si x es estrictamente mayor que y
  • x == y # si x es igual a y
  • x === y # alternativamente x ≡ y, si x es igual a y
  • x ≤ y # alternativamente x <= y, si x es menor o igual que y
  • x ≥ y # alternativamente x >= y, si x es mayor o igual que y
  • x ≠ y # alternativamente x != y, si x no es igual a y
  • x ≈ y # si x es aproximadamente igual a y

Tenga cuidado con cambiar los signos de comparación. Julia define muchas funciones de comparación por defecto sin definir la versión invertida correspondiente. Por ejemplo, se puede ejecutar

julia> Set(1:3) ⊆ Set(0:5)
true

pero no sirve para hacer

julia> Set(0:5) ⊇ Set(1:3)
ERROR: UndefVarError: ⊇ not defined

En el uso diario, la mayoría de estos operadores de comparación no son relevantes. Las más comunes utilizadas son las funciones matemáticas estándar para ordenar; consulte la sección Sintaxis para obtener una lista.

Como la mayoría de los otros operadores en Julia, los operadores de comparación son funciones y pueden llamarse como funciones. Por ejemplo, (<)(1, 2) tiene un significado idéntico a 1 < 2.

Comparaciones encadenadas #

Los múltiples operadores de comparación usados ​​juntos están encadenados, como si estuvieran conectados a través del operador &&. Esto puede ser útil para cadenas de comparación legibles y matemáticamente concisas, como

# same as 0 < i && i <= length(A)
isinbounds(A, i)       = 0 < i ≤ length(A)

# same as Set() != x && issubset(x, y)
isnonemptysubset(x, y) = Set() ≠ x ⊆ y

Sin embargo, existe una diferencia importante entre a > b > c y a > b && b > c; en el último, el término b se evalúa dos veces. Esto no importa mucho para los símbolos antiguos, pero podría importar si los términos mismos tienen efectos secundarios. Por ejemplo,

julia> f(x) = (println(x); 2)
f (generic function with 1 method)

julia> 3 > f("test") > 1
test
true

julia> 3 > f("test") && f("test") > 1
test
test
true

Echemos un vistazo más profundo a las comparaciones encadenadas y cómo funcionan, al ver cómo se analizan y reducen a expresiones. Primero, considere la comparación simple, que podemos ver es simplemente una vieja llamada de función:

julia> dump(:(a > b))
Expr
  head: Symbol call
  args: Array{Any}((3,))
    1: Symbol >
    2: Symbol a
    3: Symbol b
  typ: Any

Ahora, si encadenamos la comparación, notamos que el análisis ha cambiado:

julia> dump(:(a > b >= c))
Expr
  head: Symbol comparison
  args: Array{Any}((5,))
    1: Symbol a
    2: Symbol >
    3: Symbol b
    4: Symbol >=
    5: Symbol c
  typ: Any

Después del análisis, la expresión se reduce a su forma final:

julia> expand(:(a > b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

y observamos que esto es lo mismo que para a > b && b >= c:

julia> expand(:(a > b && b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

Números ordinales #

Veremos cómo implementar comparaciones personalizadas mediante la implementación de un tipo personalizado, [números ordinales] (https://en.wikipedia.org/wiki/Ordinal_number). Para simplificar la implementación, nos centraremos en un pequeño subconjunto de estos números: todos los números ordinales hasta ε₀, pero sin incluirlo. Nuestra implementación se centra en la simplicidad, no en la velocidad; sin embargo, la implementación tampoco es lenta.

Almacenamos números ordinales por su [forma normal de Cantor] (https://en.wikipedia.org/wiki/Ordinal_arithmetic#Cantor_normal_form). Debido a que la aritmética ordinal no es conmutativa, tomaremos la convención común de almacenar primero los términos más significativos.

immutable OrdinalNumber <: Number
    βs::Vector{OrdinalNumber}
    cs::Vector{Int}
end

Dado que la forma normal de Cantor es única, podemos probar la igualdad simplemente a través de la igualdad recursiva:

En la versión v0.5, hay una sintaxis muy buena para hacer esto de forma compacta:

import Base: ==
α::OrdinalNumber == β::OrdinalNumber = α.βs == β.βs && α.cs == β.cs

De lo contrario, defina la función como es más típico:

import Base: ==
==(α::OrdinalNumber, β::OrdinalNumber) = α.βs == β.βs && α.cs == β.cs

Para finalizar nuestro pedido, debido a que este tipo tiene un pedido total, debemos sobrecargar la función isless:

import Base: isless
function isless(α::OrdinalNumber, β::OrdinalNumber)
    for i in 1:min(length(α.cs), length(β.cs))
        if α.βs[i] < β.βs[i]
            return true
        elseif α.βs[i] == β.βs[i] && α.cs[i] < β.cs[i]
            return true
        end
    end
    return length(α.cs) < length(β.cs)
end

Para probar nuestro orden, podemos crear algunos métodos para hacer números ordinales. El cero, por supuesto, se obtiene al no tener términos en la forma normal de Cantor:

const ORDINAL_ZERO = OrdinalNumber([], [])
Base.zero(::Type{OrdinalNumber}) = ORDINAL_ZERO

Podemos definir un expω para calcular ω^α, y usarlo para calcular 1 y ω:

expω(α) = OrdinalNumber([α], [1])
const ORDINAL_ONE = expω(ORDINAL_ZERO)
Base.one(::Type{OrdinalNumber}) = ORDINAL_ONE
const ω = expω(ORDINAL_ONE)

Ahora tenemos una función de ordenamiento completamente funcional en números ordinales:

julia> ORDINAL_ZERO < ORDINAL_ONE < ω < expω(ω)
true

julia> ORDINAL_ONE > ORDINAL_ZERO
true

julia> sort([ORDINAL_ONE, ω, expω(ω), ORDINAL_ZERO])

4-element Array{OrdinalNumber,1}:
                                                                                                       OrdinalNumber(OrdinalNumber[],Int64[])
                                                                     OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])
                                   OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])
 OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])],[1])

En el último ejemplo, vemos que la impresión de números ordinales podría ser mejor, pero el resultado es el esperado.

Usando ==, === y es igual #

Hay tres operadores de igualdad: ==, === y desigual. (El último no es realmente un operador, pero es una función y todos los operadores son funciones).

Cuándo usar == #

== es igualdad de valor. Devuelve verdadero cuando dos objetos representan, en su estado actual, el mismo valor.

Por ejemplo, es obvio que

julia> 1 == 1
true

pero además

julia> 1 == 1.0
true

julia> 1 == 1.0 + 0.0im
true

julia> 1 == 1//1
true

Los lados derechos de cada igualdad anterior son de un tipo diferente, pero siguen representando el mismo valor.

Para objetos mutables, como matrices, == compara su valor actual.

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> C = [1, 3, 2]
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
true

julia> A == C
false

julia> A[2], A[3] = A[3], A[2]  # swap 2nd and 3rd elements of A
(3,2)

julia> A
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
false

julia> A == C
true

La mayoría de las veces, == es la elección correcta.

Cuándo usar === #

=== es una operación mucho más estricta que ==. En lugar de valorar la igualdad, mide la igualdad. Dos objetos son iguales si el propio programa no los puede distinguir entre sí. Así tenemos

julia> 1 === 1
true

ya que no hay manera de diferenciar un 1 de otro 1. Pero

julia> 1 === 1.0
false

porque aunque ‘1’ y ‘1.0’ tienen el mismo valor, son de tipos diferentes, por lo que el programa puede diferenciarlos.

Es más,

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> A === B
false

julia> A === A
true

que al principio puede parecer sorprendente! ¿Cómo podría el programa distinguir entre los dos vectores A y B? Debido a que los vectores son mutables, podría modificar ‘A’ y luego se comportaría de manera diferente a ‘B’. Pero no importa cómo modifique ‘A’, ‘A’ siempre se comportará igual que ‘A’ mismo. Entonces ‘A’ es igual a ‘A’, pero no igual a ‘B’.

Continuando por esta vena, observe

julia> C = A
3-element Array{Int64,1}:
 1
 2
 3

julia> A === C
true

Al asignar A a C, decimos que C tiene aliased A. Es decir, se ha convertido en otro nombre para A. Cualquier modificación hecha a A también será observada por C. Por lo tanto, no hay forma de saber la diferencia entre ‘A’ y ‘C’, por lo que son iguales.

Cuándo usar igual #

La diferencia entre == y isequal es muy sutil. La mayor diferencia está en cómo se manejan los números de punto flotante:

julia> NaN == NaN
false

Este resultado posiblemente sorprendente está definido por el estándar IEEE para tipos de punto flotante (IEEE-754). Pero esto no es útil en algunos casos, como la clasificación. Se proporciona “igual” para esos casos:

julia> isequal(NaN, NaN)
true

En el otro lado del espectro, == trata el cero negativo y el cero positivo de IEEE como el mismo valor (también como se especifica en IEEE-754). Sin embargo, estos valores tienen distintas representaciones en la memoria.

julia> 0.0
0.0

julia> -0.0
-0.0

julia> 0.0 == -0.0
true

Nuevamente, para fines de clasificación, “igual” distingue entre ellos.

julia> isequal(0.0, -0.0)
false
