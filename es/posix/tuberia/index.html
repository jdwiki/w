Las tuberías son un mecanismo para la comunicación unidireccional entre procesos o entre subprocesos dentro del alcance de una sola máquina. Lógicamente, una tubería consta de dos terminales conectados, uno en el que se pueden escribir datos y otro desde el que se pueden leer posteriormente, con un búfer de datos entre tales que no es necesario que las escrituras y las lecturas sean sincrónicas. Las tuberías deben distinguirse de las tuberías de caparazón, que son una aplicación de las tuberías.

Creación y uso básico #

Los conductos anónimos, o simplemente conductos, son objetos administrados por el núcleo expuestos a los procesos como un par de descriptores de archivo, uno para el terminal de lectura y otro para el terminal de escritura. Se crean mediante la función pipe(2):

int pipefds[2];
int result;

result = pipe(pipefds);

En caso de éxito, pipe() registra el descriptor del extremo de lectura de la tubería en el índice 0 de la matriz proporcionada, y el descriptor del extremo de escritura en el índice 1; estos índices son análogos a los números descriptores de archivo convencionales para los flujos estándar.

Habiendo creado una tubería, uno usa funciones de E/S POSIX para escribir en el extremo de escritura o leer desde el extremo de lectura:

Proceso 1:

ssize_t bytes_written = write(pipefds[1], "Hello, World!", 14);

Proceso 2:

char buffer[256];
ssize_t bytes_read = read(pipefds[0], buffer, sizeof(buffer));

Alternativamente, se puede usar fdopen() para envolver uno de los dos extremos de la tubería en una estructura FILE para usar con las funciones de C stdio:

FILE *write_end = fdopen(pipefds[1]);

if (write_end) {
    fputs("Hello, World!");
}

Los conductos tienen búferes de E/S finitos, y se bloquearán las escrituras ordinarias en conductos con búferes llenos. Por lo tanto, las tuberías no son un mecanismo seguro para que un subproceso se comunique consigo mismo, como si el subproceso que escribe en una tubería también fuera el único que lee de él, tan pronto como una escritura bloquea ese subproceso se interbloquea.

Una tubería persiste mientras cualquier proceso tenga una descripción de archivo abierta para cualquiera de los extremos de la tubería. La función close(2) se puede usar para cerrar un extremo de tubería representado por un descriptor de archivo, y fclose(3) se puede usar para cerrar un extremo de tubería a través de un FILE envuelto alrededor de él.

Estableciendo una canalización a un proceso hijo #

Los descriptores de archivo y los objetos FILE son recursos por proceso que no pueden intercambiarse entre procesos a través de E/S ordinarias. Por lo tanto, para que dos procesos distintos se comuniquen a través de una canalización anónima, uno o ambos procesos participantes deben heredar un extremo de canalización abierto del proceso que creó la canalización, que por lo tanto debe ser el proceso principal o un ancestro más lejano. El caso más simple es aquel en el que un proceso padre quiere comunicarse con un proceso hijo.

Debido a que el proceso hijo debe heredar la descripción necesaria del archivo abierto de su padre, primero se debe crear la canalización. El padre luego bifurca. Normalmente, cada proceso será estrictamente un lector o estrictamente un escritor; en ese caso, cada uno debe cerrar el extremo del tubo que no pretende utilizar.

void demo() {
    int pipefds[2];
    pid_t pid;

    // Create the pipe
    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (pid = fork()) {
        case -1:
            // error - abort ...
            break;
        case 0:   /* child */
            close(pipefds[0]);
            write(pipefds[1], "Goodbye, and thanks for all the fish!", 37);
            exit(0);
        default:  /* parent */
            close(pipefds[1]);

            char buffer[256];
            ssize_t nread = read(pipefds[0], sizeof(buffer) - 1);

            if (nread >= 0) {
                buffer[nread] = '\0';
                printf("My child said '%s'\n", buffer);
            }

            // collect the child
            wait(NULL);

            break;
    }
}

Conectar dos procesos secundarios a través de una tubería #

La conexión de dos procesos secundarios a través de una tubería se realiza conectando cada uno de los dos procesos secundarios al padre a través de diferentes extremos de la misma tubería. Por lo general, el padre no formará parte de la conversación entre los hijos, por lo que cierra sus copias de ambos extremos de la tubería.

int demo() {
    int pipefds[2];
    pid_t child1, child2;

    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            write(pipefds[1], "Hello, brother!", 15);
            exit(0);
        default:  /* parent */
            // nothing
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 2 */
            char buffer[256];
            ssize_t nread;

            close(pipefds[1]);
            nread = read(pipefds[0], buffer, sizeof(buffer) - 1);
            if (nread < 0) {
                // handle error
            } else {
                buffer[nread] = '\0';
                printf("My brother told me '%s'\n", buffer);
            }
            exit(0);
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    if (child1 >= 0) {
        wait(NULL);
        if (child2 >= 0) {
            wait(NULL);
        }
    }
}

Creando una canalización estilo shell #

Una canalización estilo shell consta de dos o más procesos, cada uno con su salida estándar conectada a la entrada estándar del siguiente. Las conexiones de salida a entrada están construidas sobre tuberías. Para establecer un conjunto de procesos similar a una tubería, se crean procesos secundarios conectados por tubería como se describe en otro ejemplo y, además, se usa la función dup2(2) para duplicar cada extremo de tubería en el descriptor de archivo estándar apropiado de su proceso. Por lo general, es una buena idea cerrar el descriptor de archivo de final de tubería original, especialmente si, como suele ser el caso, uno tiene la intención de que el niño ejecute después un comando diferente.

// Most error handling omitted for brevity, including ensuring that pipe ends are
// always closed and child processes are always collected as needed; see other
// examples for more detail.
int demo() {
    int pipefds[2];
    pid_t child1 = -1, child2 = -1;

    pipe(pipefds);

    switch (child1 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            dup2(pipefds[1], STDOUT_FILENO);
            close(pipefds[1]);
            execl("/bin/cat", "cat", "/etc/motd", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    switch (child2 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 2 */
            close(pipefds[1]);
            dup2(pipefds[0], STDIN_FILENO);
            close(pipefds[0]);
            execl("/bin/grep", "grep", "[Ww]ombat", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    wait(NULL);
    wait(NULL);
}

Uno de los errores más comunes al configurar una canalización es olvidar que uno o más de los procesos implicados cierran los extremos de la canalización que no se están utilizando. Como regla general, el resultado final debe ser que cada extremo de tubería sea propiedad de exactamente un proceso y esté abierto solo en ese proceso.

En un caso como el de la función de demostración anterior, si el segundo hijo o el padre no cierra pipefds[1], el segundo hijo se bloqueará, ya que grep continuará esperando la entrada hasta que vea EOF, y eso no se observará siempre que el extremo de escritura de la canalización permanezca abierto en cualquier proceso, como el proceso principal o el segundo hijo.
