En HTTP, los códigos de estado son un mecanismo legible por máquina que indica el resultado de una solicitud emitida anteriormente. De RFC 7231, sec. 6: “El elemento de código de estado es un código entero de tres dígitos que da el resultado del intento de comprender y satisfacer la solicitud”.

La gramática formal permite que los códigos estén entre 000 y 999. Sin embargo, solo el rango de 100 a 599 tiene significado asignado.

HTTP/1.1 define un número de [códigos de estado HTTP] (http://www.iana.org/assignments/http-status-codes/http-status-codes.xhtml) que aparecen en la línea de estado: la primera línea de una respuesta HTTP: para resumir lo que el cliente debe hacer con la respuesta.

El primer dígito de un código de estado define la clase de respuesta:

  • 1xx Informativo
  • 2xx Solicitud del cliente exitosa
  • 3xx Solicitud redirigida - acción adicional necesaria, como una nueva solicitud
  • 4xx Error del cliente - no repetir la misma solicitud
  • 5xx Error del servidor - tal vez inténtelo de nuevo

En la práctica, no siempre es fácil elegir el código de estado más adecuado.

404 No encontrado #

HTTP 404 No encontrado significa que el servidor no pudo encontrar la ruta usando el URI que solicitó el cliente.

HTTP/1.1 404 Not Found

La mayoría de las veces, el archivo solicitado se eliminó, pero a veces puede deberse a una configuración incorrecta de la raíz del documento o a la falta de permisos (aunque la falta de permisos desencadena con más frecuencia HTTP 403 Prohibido).

Por ejemplo, IIS de Microsoft escribe 404.0 (0 es el subestado) en sus archivos de registro cuando se elimina el archivo solicitado. Pero cuando las reglas de filtrado de solicitudes bloquean la solicitud entrante, escribe 404.5-404.19 en los archivos de registro según la regla que bloquea la solicitud. Puede encontrar una referencia de código de error más detallada en [Microsoft Support] (https://support.microsoft.com/en-us/kb/943891).

Responder a una solicitud condicional de contenido en caché #

Envíe un estado de respuesta 304 No modificado desde el envío del servidor en respuesta a una solicitud del cliente que contenga los encabezados If-Modified-Since y If-None-Match, si el recurso de la solicitud no ha cambiado.

Por ejemplo, si la solicitud de un cliente para una página web incluye el encabezado Si se modificó desde: viernes, 22 de julio de 2016 14:34:40 GMT y la página no se modificó desde entonces, responda con la línea de estado HTTP/ 1.1 304 No Modificado.

Error interno de servidor 500 #

Un Error interno del servidor HTTP 500 es un mensaje general que significa que el servidor encontró algo inesperado. Las aplicaciones (o el servidor web general) deben usar un 500 cuando hay un error al procesar la solicitud, es decir, se lanza una excepción o una condición del recurso impide que se complete el proceso.

Línea de estado de ejemplo:

HTTP/1.1 500 Internal Server Error

Denegar el acceso a los archivos protegidos #

Utilice 403 Prohibido cuando un cliente haya solicitado un recurso al que no se puede acceder debido a los controles de acceso existentes. Por ejemplo, si su aplicación tiene una ruta /admin a la que solo deben tener acceso los usuarios con derechos administrativos, puede usar 403 cuando un usuario normal solicite la página.

GET /admin HTTP/1.1
Host: example.com

HTTP/1.1 403 Forbidden

Solicitud exitosa #

Envíe una respuesta HTTP con el código de estado 200 para indicar una solicitud exitosa. La línea de estado de la respuesta HTTP es entonces:

HTTP/1.1 200 OK

El texto de estado OK es solo informativo. El cuerpo de la respuesta (carga del mensaje) debe contener una representación del recurso solicitado. Si no hay representación, se debe usar 201 Sin contenido.

Los 10 principales códigos de estado HTTP #

#2xx Éxito

  • 200 OK - Respuesta estándar para solicitudes HTTP exitosas.
  • 201 Creado: la solicitud se cumplió, lo que resultó en la creación de un nuevo recurso.
  • 204 Sin contenido: el servidor procesó correctamente la solicitud y no devuelve ningún contenido.

Redirección 3xx #

  • 304 No modificado: indica que el recurso no se ha modificado desde la versión especificada por los encabezados de solicitud If-Modified-Since o If-None-Match. #Error del cliente 4xx
  • 400 Solicitud incorrecta: el servidor no puede o no procesará la solicitud debido a un aparente error del cliente (por ejemplo, sintaxis de solicitud mal formada, tamaño demasiado grande, marco de mensaje de solicitud no válido o enrutamiento de solicitud engañoso).
  • 401 No autorizado - Similar a 403 Prohibido, pero específicamente para usar cuando se requiere autenticación y ha fallado o aún no se ha proporcionado. La respuesta debe incluir un campo de encabezado “WWW-Authenticate” que contenga un desafío aplicable al recurso solicitado.
  • 403 Prohibido: la solicitud era válida, pero el servidor se niega a responderla. Es posible que el usuario haya iniciado sesión pero no tenga los permisos necesarios para el recurso.
  • 404 No encontrado: no se pudo encontrar el recurso solicitado, pero es posible que esté disponible en el futuro. Se permiten solicitudes posteriores por parte del cliente.
  • 409 Conflicto: indica que la solicitud no se pudo procesar debido a un conflicto en la solicitud, como un conflicto de edición entre varias actualizaciones simultáneas. #Error del servidor 5xx
  • 500 Error interno del servidor: un mensaje de error genérico que aparece cuando se encuentra una condición inesperada y no es adecuado un mensaje más específico.
