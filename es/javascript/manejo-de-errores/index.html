Sintaxis #

  • prueba {… } atrapar (error) {… }
  • intente {… } finalmente {… }
  • intente {… } atrapar (error) {… } finalmente {… }
  • lanzar un nuevo error ([mensaje]);
  • lanzar Error([mensaje]);

try le permite definir un bloque de código para probar errores mientras se ejecuta.

catch le permite definir un bloque de código para ser ejecutado, si ocurre un error en el bloque try.

finally te permite ejecutar código independientemente del resultado. Sin embargo, tenga cuidado, las declaraciones de flujo de control de los bloques try y catch se suspenderán hasta que finalice la ejecución del bloque.

Objetos de error #

Los errores de tiempo de ejecución en JavaScript son instancias del objeto Error. El objeto Error también se puede utilizar tal cual o como base para las excepciones definidas por el usuario. Es posible arrojar cualquier tipo de valor, por ejemplo, cadenas, pero se recomienda enfáticamente que use Error o uno de sus derivados para asegurarse de que la información de depuración, como los seguimientos de la pila, se conserve correctamente.

El primer parámetro del constructor Error es el mensaje de error legible por humanos. Siempre debe intentar especificar un mensaje de error útil de lo que salió mal, incluso si se puede encontrar información adicional en otro lugar.

try {
   throw new Error('Useful message');
} catch (error) {
   console.log('Something went wrong! ' + error.message);
}

Interacción con Promesas #

Las excepciones son para el código síncrono lo que son los rechazos para el código asíncrono basado en promise. Si se lanza una excepción en un controlador de promesa, su error se detectará automáticamente y se usará para rechazar la promesa.

Promise.resolve(5)
    .then(result => {
        throw new Error("I don't like five");
    })
    .then(result => {
        console.info("Promise resolved: " + result);
    })
    .catch(error => {
        console.error("Promise rejected: " + error);
    });

 

Promise rejected: Error: I don't like five

La [propuesta de funciones asíncronas] (http://tc39.github.io/ecmascript-asyncawait/), que se espera que sea parte de ECMAScript 2017, amplía esto en la dirección opuesta. Si espera una promesa rechazada, su error se genera como una excepción:

async function main() {
  try {
    await Promise.reject(new Error("Invalid something"));
  } catch (error) {
    console.log("Caught error: " + error);
  }
}
main();

 

Caught error: Invalid something

Orden de operaciones más pensamientos avanzados #

Sin un bloque try catch, las funciones indefinidas generarán errores y detendrán la ejecución:

undefinedFunction("This will not get executed");
console.log("I will never run because of the uncaught error!");

Lanzará un error y no ejecutará la segunda línea:

// Uncaught ReferenceError: undefinedFunction is not defined

Necesita un bloque try catch, similar a otros lenguajes, para asegurarse de detectar ese error para que el código pueda continuar ejecutándose:

try {
    undefinedFunction("This will not get executed");
} catch(error) {
    console.log("An error occured!", error);
} finally {
    console.log("The code-block has finished");
}
console.log("I will run because we caught the error!");

Ahora, detectamos el error y podemos estar seguros de que nuestro código se ejecutará

// An error occured! ReferenceError: undefinedFunction is not defined(…)
// The code-block has finished
// I will run because we caught the error!

¿Qué pasa si ocurre un error en nuestro bloque catch?

try {
    undefinedFunction("This will not get executed");
} catch(error) {
    otherUndefinedFunction("Uh oh... ");
    console.log("An error occured!", error);
} finally {
    console.log("The code-block has finished");
}
console.log("I won't run because of the uncaught error in the catch block!");

No procesaremos el resto de nuestro bloque catch, y la ejecución se detendrá excepto por el último bloque.

// The code-block has finished
// Uncaught ReferenceError: otherUndefinedFunction is not defined(…)

Siempre puedes anidar tus bloques Try Catch… pero no deberías porque eso se volverá extremadamente complicado…

try {
    undefinedFunction("This will not get executed");
} catch(error) {
    try {
        otherUndefinedFunction("Uh oh... ");
    } catch(error2) {
        console.log("Too much nesting is bad for my heart and soul...");
    }
    console.log("An error occured!", error);
} finally {
    console.log("The code-block has finished");
}
console.log("I will run because we caught the error!");

Capturará todos los errores del ejemplo anterior y registrará lo siguiente:

//Too much nesting is bad for my heart and soul...
//An error occured! ReferenceError: undefinedFunction is not defined(…)
//The code-block has finished
//I will run because we caught the error!

Entonces, ¿cómo podemos detectar todos los errores? Para variables y funciones no definidas: no se puede.

Además, no debe envolver cada variable y función en un bloque de prueba/captura, porque estos son ejemplos simples que solo ocurrirán una vez hasta que los arregle. Sin embargo, para objetos, funciones y otras variables que sabe que existen, pero no sabe si existirán sus propiedades, subprocesos o efectos secundarios, o espera algunos estados de error en algunas circunstancias, debe abstraer su manejo de errores. de alguna manera. Aquí hay un ejemplo muy básico y una implementación.

Sin una forma protegida de llamar a métodos no confiables o de generación de excepciones:

function foo(a, b, c) {
    console.log(a, b, c);
    throw new Error("custom error!");
}
try {
    foo(1, 2, 3);
} catch(e) { 
    try {
        foo(4, 5, 6); 
    } catch(e2) {
        console.log("We had to nest because there's currently no other way...");
    }
    console.log(e);
}
// 1 2 3
// 4 5 6
// We had to nest because there's currently no other way...
// Error: custom error!(…)

Y con protección:

function foo(a, b, c) {
    console.log(a, b, c);
    throw new Error("custom error!");
}
function protectedFunction(fn, ...args) {
    try {
        fn.apply(this, args);
    } catch (e) {
        console.log("caught error: " + e.name + " -> " + e.message);
    }
}

protectedFunction(foo, 1, 2, 3);
protectedFunction(foo, 4, 5, 6);

// 1 2 3
// caught error: Error -> custom error!
// 4 5 6
// caught error: Error -> custom error!

Detectamos errores y seguimos procesando todo el código esperado, aunque con una sintaxis algo diferente. De cualquier manera funcionará, pero a medida que cree aplicaciones más avanzadas, querrá comenzar a pensar en formas de abstraer su manejo de errores.

Tipos de errores #

Hay seis constructores de errores centrales específicos en JavaScript:

  • EvalError - crea una instancia que representa un error que ocurre con respecto a la función global eval().

  • InternalError: crea una instancia que representa un error que ocurre cuando se genera un error interno en el motor de JavaScript. P.ej. “demasiada recursividad”. (Solo compatible con Mozilla Firefox)

  • RangeError: crea una instancia que representa un error que ocurre cuando una variable numérica o parámetro está fuera de su validez rango.

  • ReferenceError: crea una instancia que representa un error que ocurre al eliminar la referencia de una referencia no válida.

  • SyntaxError: crea una instancia que representa un error de sintaxis que ocurre al analizar el código en eval().

  • TypeError: crea una instancia que representa un error que ocurre cuando una variable o parámetro no es de un tipo válido.

  • URIError: crea una instancia que representa un error que ocurre cuando se pasan parámetros no válidos a encodeURI() o decodeURI().

Si está implementando un mecanismo de manejo de errores, puede verificar qué tipo de error está detectando en el código.

try {
    throw new TypeError();
}
catch (e){
    if(e instanceof Error){
        console.log('instance of general Error constructor');
    }

    if(e instanceof TypeError) {
        console.log('type error');
    }
}

En tal caso, e será una instancia de TypeError. Todos los tipos de error amplían el constructor base Error, por lo que también es una instancia de Error.

Tener eso en mente nos muestra que verificar que ’e’ sea una instancia de ‘Error’ es inútil en la mayoría de los casos.
