El bucle de eventos en un navegador web #

La gran mayoría de los entornos modernos de JavaScript funcionan según un bucle de eventos. Este es un concepto común en la programación de computadoras que esencialmente significa que su programa espera continuamente que sucedan cosas nuevas y, cuando suceden, reacciona ante ellas. El entorno host llama a su programa, generando un “giro” o “tick” o “tarea” en el bucle de eventos, que luego se ejecuta hasta su finalización. Cuando ese turno ha terminado, el entorno anfitrión espera a que suceda algo más, antes de que todo esto comience.

Un ejemplo simple de esto está en el navegador. Considere el siguiente ejemplo:

<!DOCTYPE html>
<title>Event loop example</title>

<script>
console.log("this a script entry point");

document.body.onclick = () => {
  console.log("onclick");
};

setTimeout(() => {
  console.log("setTimeout callback log 1");
  console.log("setTimeout callback log 2");
}, 100);
</script>

En este ejemplo, el entorno del host es el navegador web.

  1. El analizador HTML primero ejecutará <script>. Se ejecutará hasta su finalización.
  2. La llamada a setTimeout le dice al navegador que, después de 100 milisegundos, debe poner en cola una tarea para realizar la acción dada.
  3. Mientras tanto, el bucle de eventos es responsable de verificar continuamente si hay algo más que hacer: por ejemplo, renderizar la página web.
  4. Después de 100 milisegundos, si el bucle de eventos no está ocupado por alguna otra razón, verá la tarea que setTimeout se pone en cola y ejecutará la función, registrando esas dos declaraciones.
  5. En cualquier momento, si alguien hace clic en el cuerpo, el navegador publicará una tarea en el bucle de eventos para ejecutar la función de controlador de clics. El ciclo de eventos, a medida que revisa continuamente qué hacer, verá esto y ejecutará esa función.

Puede ver cómo en este ejemplo hay varios tipos diferentes de puntos de entrada en el código JavaScript, que invoca el bucle de eventos:

  • El elemento <script> se invoca inmediatamente
  • La tarea setTimeout se publica en el bucle de eventos y se ejecuta una vez
  • La tarea del controlador de clics se puede publicar muchas veces y ejecutar cada vez

Cada turno del bucle de eventos es responsable de muchas cosas; solo algunos de ellos invocarán estas tareas de JavaScript. Para obtener detalles completos, [consulte la especificación HTML] (https://html.spec.whatwg.org/multipage/webappapis.html#event-loop-processing-model)

Una última cosa: ¿qué queremos decir con que cada tarea de bucle de eventos “se ejecuta hasta su finalización”? Queremos decir que generalmente no es posible interrumpir un bloque de código que está en cola para ejecutarse como una tarea, y nunca es posible ejecutar código intercalado con otro bloque de código. Por ejemplo, incluso si hizo clic en el momento perfecto, nunca podría obtener el código anterior para registrar "onclick" entre los dos setTimeout callback log 1/2"s. Esto se debe a la forma en que la tarea: publicar obras; es cooperativo y basado en colas, en lugar de preventivo.

Operaciones asíncronas y el bucle de eventos #

Muchas operaciones interesantes en entornos de programación JavaScript comunes son asincrónicas. Por ejemplo, en el navegador vemos cosas como

window.setTimeout(() => {
  console.log("this happens later");
}, 100);

y en Node.js vemos cosas como

fs.readFile("file.txt", (err, data) => {
  console.log("data");
});

¿Cómo encaja esto con el bucle de eventos?

Cómo funciona esto es que cuando se ejecutan estas declaraciones, le dicen al entorno host (es decir, el navegador o el tiempo de ejecución de Node.js, respectivamente) que se active y haga algo, probablemente en otro subproceso. Cuando el entorno del host termine de hacer eso (respectivamente, esperar 100 milisegundos o leer el archivo file.txt), publicará una tarea en el ciclo de eventos, diciendo “llame a la devolución de llamada que me dieron anteriormente con estos argumentos”.

El bucle de eventos está entonces ocupado haciendo su trabajo: representar la página web, escuchar la entrada del usuario y buscar continuamente tareas publicadas. Cuando vea estas tareas publicadas para llamar a las devoluciones de llamada, volverá a llamar a JavaScript. ¡Así es como obtienes un comportamiento asíncrono!
