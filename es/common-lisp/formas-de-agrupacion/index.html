Prog1 y Prog2 #

A menudo, es útil evaluar múltiples expresiones y devolver el resultado de la primera o segunda forma en lugar de la última. Esto es fácil de lograr usando let y, por ejemplo:

(let ((form1-result form1))
  form2
  form3
  ;; ...
  form-n-1
  form-n
  form1-result)

Debido a que esta forma es común en algunas aplicaciones, Common Lisp incluye prog1 y prog2 que son como progn, pero devuelven el resultado de la primera y segunda forma, respectivamente. Por ejemplo:

(prog1
  42
  (print 'hello)
  (print 'goodbye))
;; => 42

(prog2
  (print 'hello)
  42
  (print 'goodbye))
;; => 42

Sin embargo, una distinción importante entre prog1/prog2 y progn es que progn devuelve todos los valores de la última forma, mientras que prog1 y * prog2* solo devuelve el valor primario de la primera y segunda forma. Por ejemplo:

(progn
  (print 'hello)
  (values 1 2 3))
;;=> 1, 2, 3

(prog1
  (values 1 2 3)
  (print 'hello))
;;=> 1              ; not 1, 2, 3

Para valores múltiples con evaluación de estilo prog1, use multiple-value-prog1 en su lugar. No existe un prog2 de valores múltiples similar, pero no es difícil de implementar si lo necesita.

Bloquear #

El operador especial block permite agrupar varias formas Lisp (como un prog implícito) y también requiere un nombre para nombrar el bloque. Cuando se evalúan los formularios dentro del bloque, se puede usar el operador especial return-from para salir del bloque. Por ejemplo:

(block foo
  (print 'hello)     ; evaluated
  (return-from foo)
  (print 'goodbye))  ; not evaluated
;;=> NIL

return-from también se puede proporcionar con un valor de retorno:

(block foo
  (print 'hello)     ; evaluated
  (return-from foo 42)
  (print 'goodbye))  ; not evaluated
;;=> 42

Los bloques con nombre son útiles cuando un fragmento de código tiene un nombre significativo o cuando los bloques están anidados. En algún contexto, solo es importante la capacidad de regresar temprano de un bloqueo. En ese caso, puede usar nil como nombre de bloque y return. Return es como return-from, excepto que el nombre del bloque siempre es nil.

Nota: los formularios adjuntos no son formularios de nivel superior. Eso es diferente de prog, donde los formularios adjuntos de un formulario prog de nivel superior todavía se consideran formularios de nivel superior.

¿Cuándo es necesario agrupar? #

En algunos lugares de Common Lisp, se evalúan una serie de formularios en orden. Por ejemplo, en el cuerpo de defun o lambda, o en el cuerpo de dotimes. En esos casos, escribir varios formularios en orden funciona como se esperaba. Sin embargo, en algunos lugares, como las partes then y else de las expresiones if, solo se permite una única forma. Por supuesto, uno puede querer evaluar múltiples expresiones en esos lugares. Para esas situaciones, se necesita algún tipo de forma de agrupación implícita o explícita.

Pronóstico #

El operador especial de propósito general prog se utiliza para evaluar cero o más formularios. Se devuelve el valor de la última forma. Por ejemplo, a continuación, se evalúa (print ‘hello) (y se ignora su resultado), y luego se evalúa 42 y se devuelve su resultado (42):

(progn 
  (print 'hello)
  42)
;=> 42

Si no hay formularios dentro del progn, se devuelve nil:

(progn)
;=> NIL

Además de agrupar una serie de formularios, progn también tiene la propiedad importante de que si el formulario progn es un formulario de nivel superior, entonces se procesan todos los formularios que contiene. como formularios de nivel superior. Esto puede ser importante al escribir macros que se expanden en múltiples formularios que deben procesarse como formularios de nivel superior.

Progn también es valioso porque devuelve todos los valores de la última forma. Por ejemplo,

(progn
  (print 'hello)
  (values 1 2 3))
;;=> 1, 2, 3

Por el contrario, algunas expresiones de agrupación solo devuelven el valor principal de la forma que produce el resultado.

Pronósticos implícitos #

Algunas formas usan progns implícitos para describir su comportamiento. Por ejemplo, las macros when y unless, que son esencialmente formas unilaterales de if, describen su comportamiento en términos de un progn implícito. Esto significa que una forma como

(when (foo-p foo)
  form1
  form2)

se evalúa y la condición (foo-p foo) es verdadera, entonces form1 y form2 se agrupan como si estuvieran contenidos dentro de un progn. La expansión de la macro cuando es esencialmente:

(if (foo-p foo)
  (progn
    form1
    form2)
  nil)

Etiqueta #

Para tener mucho control en los formularios de un grupo, el operador especial tagbody puede ser muy útil. Los formularios dentro de un formulario tagbody son go tags (que son solo símbolos o números enteros) o formularios para ejecutar. Dentro de un cuerpo de etiqueta, el operador especial go se usa para transferir la ejecución a una nueva ubicación. Este tipo de programación puede considerarse de nivel bastante bajo, ya que permite rutas de ejecución arbitrarias. El siguiente es un ejemplo detallado de cómo se vería un bucle for cuando se implementa como un cuerpo de etiqueta:

(let (x)               ; for (x = 0; x < 5; x++) { print(hello); }
  (tagbody
     (setq x 0)
   prologue
     (unless (< x 5)
       (go end))
   begin
     (print (list 'hello x))
   epilogue
     (incf x)
     (go prologue)
   end))

Si bien tagbody y go no se usan comúnmente, quizás debido a que “GOTO se considera dañino”, pueden ser útiles cuando se implementan estructuras de control complejas como máquinas de estado. Muchas construcciones de iteración también se expanden en un cuerpo de etiqueta implícito. Por ejemplo, el cuerpo de dotimes se especifica como una serie de etiquetas y formularios.

¿Qué formulario usar? #

Al escribir macros que se expanden en formas que pueden implicar la agrupación, vale la pena dedicar un tiempo a considerar en qué construcción de agrupación se expandirá.

Para formularios de estilo de definición, por ejemplo, una macro define-widget que generalmente aparecerá como un formulario de nivel superior, y que varias defuns, defstructs, etc., por lo general tiene sentido usar un progn, de modo que los formularios secundarios se procesen como formularios de nivel superior. Para formularios de iteración, un cuerpo de etiqueta implícito es más común.

Por ejemplo, el cuerpo de dotimes, dolist y do cada uno se expande en un tagbody implícito.

Para formularios que definen un “fragmento” de código con nombre, un bloque implícito suele ser útil. Por ejemplo, mientras que el cuerpo de defun está dentro de un prog implícito, ese prog implícito está dentro de un bloque que comparte el nombre de la función.
