El Propósito de las Macros #

Las macros están destinadas a generar código, transformar código y proporcionar nuevas notaciones. Estas nuevas notaciones pueden ser más adecuadas para expresar mejor el programa, por ejemplo, proporcionando construcciones a nivel de dominio o lenguajes incrustados completamente nuevos.

Las macros pueden hacer que el código fuente se explique por sí mismo, pero la depuración puede resultar más difícil. Como regla general, uno no debe usar macros cuando una función regular es suficiente. Cuando los use, evite las trampas habituales, intente apegarse a los patrones y convenciones de nomenclatura más utilizados.

Orden de macroexpansión #

En comparación con las funciones, las macros se expanden en orden inverso; lo más externo primero, lo más interno al final. Esto significa que, de forma predeterminada, no se puede usar una macro interna para generar la sintaxis requerida para una macro externa.

Orden de evaluación #

A veces, las macros necesitan mover formularios proporcionados por el usuario. Hay que asegurarse de no cambiar el orden en que se evalúan. El usuario puede confiar en que los efectos secundarios ocurran en orden.

Evaluar una sola vez #

La expansión de una macro a menudo necesita usar el valor del mismo formulario proporcionado por el usuario más de una vez. Es posible que el formulario tenga efectos secundarios, o podría estar llamando a una función costosa. Por lo tanto, la macro debe asegurarse de evaluar dichos formularios solo una vez. Por lo general, esto se hará asignando el valor a una variable local (cuyo nombre es GENSYMed).

Funciones usadas por Macros, usando EVAL-WHEN #

Las macros complejas a menudo tienen partes de su lógica implementadas en funciones separadas. Sin embargo, se debe recordar que las macros se expanden antes de compilar el código real. Al compilar un archivo, de forma predeterminada, las funciones y variables definidas en el mismo archivo no estarán disponibles durante la ejecución de la macro. Todas las definiciones de funciones y variables, en el mismo archivo, utilizadas por una macro deben incluirse dentro de un formulario [EVAL-WHEN] (http://www.lispworks.com/documentation/HyperSpec/Body/s_eval_w.htm) . El EVAL-CUÁNDO debe tener los tres tiempos especificados, cuando el código adjunto también debe evaluarse durante la carga y el tiempo de ejecución.

(eval-when (:compile-toplevel :load-toplevel :execute)
  (defun foobar () ...))

Esto no se aplica a las funciones llamadas desde la expansión de la macro, solo a las llamadas por la propia macro.

Macros anafóricas #

Una Macro Anafórica es una macro que introduce una variable (a menudo IT) que captura el resultado de un formulario proporcionado por el usuario. Un ejemplo común es el Anaphoric If, que es como un ‘IF’ regular, pero también define la variable ‘IT’ para referirse al resultado del formulario de prueba.

(defmacro aif (test-form then-form &optional else-form)
  `(let ((it ,test-form))
     (if it ,then-form ,else-form)))

(defun test (property plist)
  (aif (getf plist property)
       (format t "The value of ~s is ~a.~%" property it)
       (format t "~s wasn't in ~s!~%" property plist)))

(test :a '(:a 10 :b 20 :c 30))
; The value of :A is 10.
(test :d '(:a 10 :b 20 :c 30))
; :D wasn't in (:A 10 :B 20 :C 30)!

Backquote - escribir plantillas de código para macros #

Código de retorno de las macros. Dado que el código en Lisp consiste en listas, se pueden usar las funciones regulares de manipulación de listas para generarlo.

;; A pointless macro
(defmacro echo (form)
  (list 'progn
        (list 'format t "Form: ~a~%" (list 'quote form))
        form))

Esto suele ser muy difícil de leer, especialmente en macros más largas. La macro del lector Backquote permite escribir plantillas entrecomilladas que se completan mediante la evaluación selectiva de elementos.

(defmacro echo (form)
  `(progn
     (format t "Form: ~a~%" ',form)
     ,form))

(macroexpand '(echo (+ 3 4)))
;=> (PROGN (FORMAT T "Form: ~a~%" '(+ 3 4)) (+ 3 4))

Esta versión parece casi un código normal. Las comas se utilizan para evaluar FORM; todo lo demás se devuelve como está. Tenga en cuenta que en ',form la comilla simple está fuera de la coma, por lo que se devolverá.

También se puede usar ,@ para empalmar una lista en la posición.

(defmacro echo (&rest forms)
  `(progn
     ,@(loop for form in forms collect `(format t "Form: ~a~%" ,form))
     ,@forms))

(macroexpand '(echo (+ 3 4) 
                    (print "foo")
                    (random 10)))
;=> (PROGN
;    (FORMAT T "Form: ~a~%" (+ 3 4))
;    (FORMAT T "Form: ~a~%" (PRINT "foo"))
;    (FORMAT T "Form: ~a~%" (RANDOM 10))
;    (+ 3 4)
;    (PRINT "foo")
;    (RANDOM 10))

La comilla inversa también se puede usar fuera de las macros.

PIE #

En Common Lisp, hay un concepto de [Referencias Generalizadas] (http://www.lispworks.com/documentation/HyperSpec/Body/05_a.htm). Permiten que un programador establezca valores en varios “lugares” como si fueran variables. Las macros que hacen uso de esta capacidad a menudo tienen un sufijo F en el nombre. El lugar suele ser el primer argumento de la macro.

Ejemplos del estándar: INCF, DECF, [ROTATEF](http://www. lispworks.com/documentation/HyperSpec/Body/m_rotate.htm#rotatef), SHIFTF, REMF .

Un ejemplo tonto, una macro que cambia el signo de una tienda de números en un lugar:

(defmacro flipf (place)
  `(setf ,place (- ,place)))

CON-PIE #

Las macros que adquieren y liberan un recurso de manera segura generalmente se nombran con un prefijo CON-. La macro generalmente debe usar una sintaxis como:

(with-foo (variable details-of-the-foo...)
  body...)

Ejemplos del estándar: WITH-OPEN-FILE, [WITH-OPEN-STREAM](http:// www.lispworks.com/documentation/HyperSpec/Body/m_w_op_1.htm), CON-ENTRADA-DE-CADENA, CON-SALIDA-A-CADENA.

Un enfoque para implementar este tipo de macro que puede evitar algunos de los peligros de la contaminación de nombres y la evaluación múltiple no intencionada es implementar primero una versión funcional. Por ejemplo, el primer paso para implementar una macro with-widget que cree un widget de forma segura y luego lo limpie podría ser una función:

(defun call-with-widget (args function)
  (let ((widget (apply #'make-widget args))) ; obtain WIDGET
    (unwind-protect (funcall function widget)  ; call FUNCTION with WIDGET
      (cleanup widget)                         ; cleanup

Debido a que se trata de una función, no hay preocupaciones sobre el alcance de los nombres dentro de función o proveedor, y facilita la escritura de una macro correspondiente:

(defmacro with-widget ((var &rest args) &body body)
  `(call-with-widget (list ,@args) (lambda (,var) ,@body)))

#DO-FOO

Las macros que iteran sobre algo a menudo se nombran con un prefijo DO. La sintaxis de la macro generalmente debe estar en forma

(do-foo (variable the-foo-being-done return-value)
  body...)

Ejemplos del estándar: DOTIMES, [DOLIST](http://www.lispworks.com/documentation/ HyperSpec/Body/m_dolist.htm), DO-SYMBOLS.

FOOCASO, EFOOCASO, CFOOCASO #

Las macros que hacen coincidir una entrada con ciertos casos a menudo se nombran con un postfijo CASE. A menudo hay una variante E...CASE, que indica un error si la entrada no coincide con ninguno de los casos, y C...CASE, que indica un error continuable. Deben tener una sintaxis como

(foocase input
  (case-to-match-against (optionally-some-params-for-the-case)
   case-body-forms...)
  more-cases...
  [(otherwise otherwise-body)])

Ejemplos del estándar: CASE, [TYPECASE](http://www.lispworks.com/documentation/ HyperSpec/Body/m_tpcase.htm), HANDLER-CASE.

Por ejemplo, una macro que hace coincidir una cadena con expresiones regulares y vincula los grupos de registros a las variables. Utiliza CL-PPCRE para expresiones regulares.

(defmacro regexcase (input &body cases)
  (let ((block-sym (gensym "block"))
        (input-sym (gensym "input")))
    `(let ((,input-sym ,input))
       (block ,block-sym
         ,@(loop for (regex vars . body) in cases
                 if (eql regex 'otherwise)
                   collect `(return-from ,block-sym (progn ,vars ,@body))
                 else
                   collect `(cl-ppcre:register-groups-bind ,vars
                                (,regex ,input-sym)
                              (return-from ,block-sym
                                (progn ,@body))))))))

(defun test (input)
  (regexcase input
    ("(\\d+)-(\\d+)" (foo bar)
      (format t "Foo: ~a, Bar: ~a~%" foo bar))
    ("Foo: (\\w+)$" (foo)
      (format t "Foo: ~a.~%" foo))
    (otherwise (format t "Didn't match.~%"))))

(test "asd 23-234 qwe")
; Foo: 23, Bar: 234
(test "Foo: Foobar")
; Foo: Foobar.
(test "Foo: 43 - 23")
; Didn't match.

DEFINE-FOO, DEFFOO #

Las macros que definen cosas generalmente se nombran con el prefijo DEFINE- o DEF.

Ejemplos del estándar: DEFUN, [DEFMACRO](http://www.lispworks.com/documentation/ HyperSpec/Body/m_defmac.htm), DEFINE-CONDITION.

MACROEXPANSIÓN #

La expansión de macros es el proceso de convertir macros en código real. Esto suele ocurrir como parte del proceso de compilación. El compilador expandirá todas las formas de macro antes de compilar el código. La expansión de macros también ocurre durante la interpretación del código Lisp.

Uno puede llamar a MACROEXPAND manualmente para ver a qué se expande una macro.

CL-USER> (macroexpand '(with-open-file (file "foo")
                        (do-something-with file)))
(LET ((FILE (OPEN "foo")) (#:G725 T))
  (UNWIND-PROTECT
      (MULTIPLE-VALUE-PROG1 (PROGN (DO-SOMETHING-WITH FILE)) (SETQ #:G725 NIL))
    (WHEN FILE (CLOSE FILE :ABORT #:G725))))

MACROEXPAND-1 es lo mismo, pero solo se expande una vez. Esto es útil cuando se trata de dar sentido a un formulario de macro que se expande a otro formulario de macro.

CL-USER> (macroexpand-1 '(with-open-file (file "foo")
                          (do-something-with file)))
(WITH-OPEN-STREAM (FILE (OPEN "foo")) (DO-SOMETHING-WITH FILE))

Tenga en cuenta que ni ‘MACROEXPAND’ ni ‘MACROEXPAND-1’ expanden el código Lisp en todos los niveles. Solo expanden el formulario macro de nivel superior. Para macroexpandir un formulario completamente en todos los niveles, se necesita un caminante de código para hacerlo. Esta función no se proporciona en el estándar Common Lisp.

macros if-let, when-let, -let #

Estas macros combinan el flujo de control y el enlace. Son una mejora con respecto a las macros anafóricas porque permiten que el desarrollador comunique el significado a través de nombres. Como tal, se recomienda su uso sobre sus contrapartes anafóricas.

(if-let (user (get-user user-id))
  (show-dashboard user)
  (redirect 'login-page))

Las macros FOO-LET vinculan una o más variables, y luego usan esas variables como forma de prueba para el condicional correspondiente (SI, CUÁNDO). Varias variables se combinan con Y. La rama elegida se ejecuta con los enlaces vigentes. Una implementación simple de una variable de IF-LET podría verse así:

(defmacro if-let ((var test-form) then-form &optional else-form)
  `(let ((,var ,test-form))
     (if ,var ,then-form ,else-form)))

(macroexpand '(if-let (a (getf '(:a 10 :b 20 :c 30) :a))
               (format t "A: ~a~%" a)
               (format t "Not found.~%")))
; (LET ((A (GETF '(:A 10 :B 20 :C 30) :A)))
;   (IF A
;       (FORMAT T "A: ~a~%" A)
;       (FORMAT T "Not found.~%")))

Una versión que admite múltiples variables está disponible en la biblioteca Alexandria.

Símbolos únicos para evitar conflictos de nombres en macros #

La expansión de una macro a menudo necesita usar símbolos que el usuario no pasó como argumentos (como nombres para variables locales, por ejemplo). Uno debe asegurarse de que tales símbolos no puedan entrar en conflicto con un símbolo que el usuario esté usando en el código circundante.

Esto generalmente se logra usando GENSYM, una función que devuelve un nuevo símbolo no internado.

Malo

Considere la siguiente macro. Hace un bucle DOTIMES que también recopila el resultado del cuerpo en una lista, que se devuelve al final.

(defmacro dotimes+collect ((var count) &body body)
  `(let ((result (list)))
     (dotimes (,var ,count (nreverse result))
       (push (progn ,@body) result))))

(dotimes+collect (i 5)
  (format t "~a~%" i)
  (* i i))
; 0
; 1
; 2
; 3
; 4
;=> (0 1 4 9 16)

Esto parece funcionar en este caso, pero si el usuario tuviera un nombre de variable RESULTADO, que usa en el cuerpo, los resultados probablemente no serían los que el usuario espera. Considere este intento de escribir una función que recopile una lista de sumas de todos los enteros hasta N:

(defun sums-upto (n)
  (let ((result 0))
    (dotimes+collect (i n)
      (incf result i))))

(sums-upto 10) ;=> Error!

Bueno

Para solucionar el problema, necesitamos usar GENSYM para generar un nombre único para la variable RESULTADO en la expansión macro.

(defmacro dotimes+collect ((var count) &body body)
  (let ((result-symbol (gensym "RESULT")))
    `(let ((,result-symbol (list)))
       (dotimes (,var ,count (nreverse ,result-symbol))
         (push (progn ,@body) ,result-symbol)))))

(sums-upto 10) ;=> (0 1 3 6 10 15 21 28 36 45)

Uso de macros para definir estructuras de datos #

Un uso común de las macros es crear plantillas para estructuras de datos que obedecen reglas comunes pero que pueden contener diferentes campos. Al escribir una macro, puede permitir que se especifique la configuración detallada de la estructura de datos sin necesidad de repetir el código repetitivo, ni usar una estructura menos eficiente (como un hash) en la memoria simplemente para simplificar la programación.

Por ejemplo, supongamos que deseamos definir una serie de clases que tienen un rango de propiedades diferentes, cada una con un getter y un setter. Además, para algunas (pero no todas) de estas propiedades, deseamos que el setter llame a un método en el objeto para notificarle que la propiedad ha cambiado. Aunque Common LISP ya tiene una abreviatura para escribir getters y setters, escribir un setter personalizado estándar de esta manera normalmente requeriría duplicar el código que llama al método de notificación en cada setter, lo que podría ser una molestia si hay una gran cantidad de propiedades involucradas. . Sin embargo, al definir una macro se vuelve mucho más fácil:

(defmacro notifier (class slot) 
  "Defines a setf method in (class) for (slot) which calls the object's changed method."
   `(defmethod (setf ,slot) (val (item ,class))
     (setf (slot-value item ',slot) val)
     (changed item ',slot)))

(defmacro notifiers (class slots)
  "Defines setf methods in (class) for all of (slots) which call the object's changed method."
  `(progn 
     ,@(loop for s in slots collecting `(notifier ,class ,s))))

(defmacro defclass-notifier-slots (class nslots slots)  
  "Defines a class with (nslots) giving a list of slots created with notifiers, and (slots) giving a list of slots created with regular accessors."
  `(progn
     (defclass ,class () 
       ( ,@(loop for s in nslots collecting `(,s :reader ,s)) 
         ,@(loop for s in slots collecting `(,s :accessor ,s))))   
     (notifiers ,class ,nslots)))

Ahora podemos escribir (defclass-notifier-slots foo (bar baz qux) (waldo)) e inmediatamente definir una clase foo con un espacio normal waldo (creado por la segunda parte de la macro con la especificación (waldo :accessor waldo)), y las ranuras bar, baz y qux con setters que llaman al método changed (donde el getter está definido por la primera parte de la macro, (bar : barra del lector), y el establecedor por la macro notificador invocada).

Además de permitirnos definir rápidamente múltiples clases que se comportan de esta manera, con un gran número de propiedades, sin repetición, tenemos el beneficio habitual de la reutilización de código: si luego decidimos cambiar el funcionamiento de los métodos notificadores, simplemente podemos cambiar el macro, y la estructura de cada clase que la use cambiará.
