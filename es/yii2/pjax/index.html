Paso 1 Agregar estructura #

En vistas\sitio\form-submission.php

<?php Pjax::begin(['id'=>'id-pjax']); ?>
<?= Html::beginForm(['site/form-submission'], 'post', ['data-pjax' => '', 'class' => 'form-inline']); ?>
    <?= Html::input('text', 'string', Yii::$app->request->post('string'), ['class' => 'form-control']) ?>
    <?= Html::submitButton('Hash String', ['class' => 'btn btn-lg btn-primary', 'name' => 'hash-button']) ?>
<?= Html::endForm() ?>
<h3><?= $stringHash ?></h3>
<?php Pjax::end(); ?>

cómo usar pjax #

Agregue esta línea al principio de su vista.

<?php
use yii\widgets\Pjax;
?>

Agregue las siguientes dos líneas alrededor del contenido que necesita una actualización parcial.

<?php Pjax::begin(['id'=>'id-pjax']); ?>
Content that needs to be updated
<?php Pjax::end(); ?>

recargar pjax #

$.pjax.reload({container: '#id-pjax'});

usar el argumento de tiempo de espera en pjax #

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false]); ?>

puede especificar un valor entero para el argumento de tiempo de espera, que sería el número de milisegundos de espera (su valor predeterminado es 1000). Si el tiempo de ejecución en el servidor es mayor que este valor de tiempo de espera, se activará una carga de página completa.

Por defecto, pjax enviará el formulario utilizando el método GET. Puede cambiar el método de envío del formulario a POST como en el siguiente ejemplo

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false, 'clientOptions' => ['method' => 'POST']]); ?>

Paso 2 Código del lado del servidor #

public function actionFormSubmission()
{
    $security = new Security();
    $string = Yii::$app->request->post('string');
    $stringHash = '';
    if (!is_null($string)) {
        $stringHash = $security->generatePasswordHash($string);
    }
    return $this->render('form-submission', [
        'stringHash' => $stringHash,
    ]);
}

Uso avanzado de Pjax #

Yii Framework 2.0 viene con soporte incorporado para Pjax, una biblioteca de JavaScript que reduce los tiempos de carga de la página. Lo logra actualizando solo la parte de la página que ha cambiado a través de Ajax, lo que puede traducirse en ahorros sustanciales si tiene muchos otros activos en sus páginas. Algunos de nuestros proyectos usan esta funcionalidad y queríamos compartir algunas lecciones aprendidas.

Problema: La página 1 es una página estática simple que contiene pocos elementos. La página 2 incluye un ActiveForm así como otros widgets. Los recursos JavaScript de ActiveForm deben cargarse para que se ejecute el JavaScript en línea, pero dado que la página 1 no incluía esos recursos, la página 2 encontró un error de JavaScript al intentar ejecutar la línea de formulario activo: ‘Error de tipo no detectado: undefined is not a función’.

Solución: Incluya activos de ActiveForm en un paquete de activos compartidos que se cargará en todas las páginas, asegurándose de que cualquier página de entrada permita que estén disponibles los scripts correctos.

class AppAsset extends AssetBundle
{
    ...
    public $depends = [
        'yii\widgets\ActiveFormAsset',
        'yii\validators\ValidationAsset',
    ];
    ...
}

Problema: En el mismo ejemplo anterior, la página 1 incluye algunos widgets (barra de navegación, etc.). La página 2 incluye los mismos widgets y algunos más (ActiveForm, etc.). Al cargar la página a través de Pjax, se estaba ejecutando JavaScript en línea personalizado, pero el script en línea colocado por el widget de ActiveForm no parecía funcionar, ya que el código de validación no funcionaba. En la depuración, encontramos que la función de inicio de ActiveForm se estaba ejecutando, pero la variable ’esto’ no parecía corresponder a ActiveForm. En realidad, correspondía al div NavBar. Al investigar las ID de div, vimos que ActiveForm esperaba tener la ID de #w1, pero la barra de navegación ya tenía asignada esa ID en la página 1, ya que ese fue el primer widget que se encontró en esa página.

Solución: No confíes en Yii para generar automáticamente las ID de los widgets por ti. En su lugar, siempre pase una ID al crear el widget para mantener el control de esas ID.

Problema: la solicitud de Pjax se canceló exactamente 1000 ms después de que se inició la solicitud.

Solución: Aumente la configuración de tiempo de espera de Pjax. El valor predeterminado es 1 segundo, lo que debería ser aceptable para los sitios de producción. Sin embargo, en desarrollo, mientras usamos xdebug, los tiempos de carga de nuestra página superan regularmente este límite.

Problema: la aplicación web implementa el patrón Post-Redirect-Get (PRG). Pjax recarga la página completa en lugar de solo la redirección.

Solución: Este es el comportamiento previsto de Pjax. La redirección no cumple su propósito cuando se usa Pjax, por lo que puede determinar si una solicitud es Pjax y, de ser así, representar el contenido en lugar de redirigir. Un ejemplo puede verse como:

$endURL = "main/endpoint";
if (Yii::$app->request->isPjax) {
    return $this->run($endURL);
} else {
    return $this->redirect([$endURL]);
}
