  • Cargando recurso del mismo nombre desde múltiples JAR

Java permite la recuperación de recursos basados ​​en archivos almacenados dentro de un JAR junto con clases compiladas. Este tema se centra en cargar esos recursos y ponerlos a disposición de su código.

Un recurso son datos similares a archivos con un nombre similar a una ruta, que reside en la ruta de clase. El uso más común de los recursos es agrupar imágenes de aplicaciones, sonidos y datos de solo lectura (como la configuración predeterminada).

Se puede acceder a los recursos con ClassLoader.getResource y ClassLoader .getResourceAsStream métodos. El caso de uso más común es tener recursos ubicados en el mismo paquete que la clase que los lee; Class.getResource y [Class.getResourceAsStream](http Los métodos ://docs.oracle.com/javase/8/docs/api/java/lang/Class.html#getResourceAsStream-java.lang.String-) sirven para este caso de uso común.

La única diferencia entre un método getResource y un método getResourceAsStream es que el primero devuelve una URL, mientras que el segundo abre esa URL y devuelve un InputStream.

Los métodos de ClassLoader aceptan un nombre de recurso similar a una ruta como argumento y buscan en cada ubicación en la ruta de clases de ClassLoader una entrada que coincida con ese nombre.

  • Si una ubicación de classpath es un archivo .jar, una entrada jar con el nombre especificado se considera una coincidencia.
  • Si una ubicación de classpath es un directorio, un archivo relativo bajo ese directorio con el nombre especificado se considera una coincidencia.

El nombre del recurso es similar a la parte de la ruta de una URL relativa. En todas las plataformas, utiliza barras diagonales (/) como separadores de directorio. No debe comenzar con una barra oblicua.

Los métodos correspondientes de Class son similares, excepto:

  • El nombre del recurso puede comenzar con una barra inclinada, en cuyo caso se elimina la barra inclinada inicial y el resto del nombre se pasa al método correspondiente de ClassLoader.
  • Si el nombre del recurso no comienza con una barra inclinada, se trata como relativo a la clase cuyo método getResource o getResourceAsStream se llama. El nombre real del recurso se convierte en paquete/nombre, donde paquete es el nombre del paquete al que pertenece la clase, con cada punto reemplazado por una barra inclinada, y nombre es el argumento original dado al método.

Por ejemplo:

package com.example;

public class ExampleApplication {
    public void readImage()
    throws IOException {

        URL imageURL = ExampleApplication.class.getResource("icon.png");

        // The above statement is identical to:
        // ClassLoader loader = ExampleApplication.class.getClassLoader();
        // URL imageURL = loader.getResource("com/example/icon.png");

        Image image = ImageIO.read(imageURL);
    }
}

Los recursos deben colocarse en paquetes con nombre, en lugar de en la raíz de un archivo .jar, por la misma razón por la que las clases se colocan en paquetes: para evitar colisiones entre varios proveedores. Por ejemplo, si hay varios archivos .jar en el classpath y más de uno de ellos contiene una entrada config.properties en su raíz, las llamadas a los métodos getResource o getResourceAsStream devolverán las config.properties de cualquier .jar que esté en la lista. primero en el classpath. Este no es un comportamiento predecible en entornos donde el orden de classpath no está bajo el control directo de la aplicación, como Java EE.

Todos los métodos getResource y getResourceAsStream devuelven null si el recurso especificado no existe. Dado que los recursos deben agregarse a la aplicación en el momento de la compilación, sus ubicaciones deben conocerse cuando se escribe el código; una falla al encontrar un recurso en tiempo de ejecución suele ser el resultado de un error del programador.

Los recursos son de solo lectura. No hay forma de escribir en un recurso. Los desarrolladores novatos a menudo cometen el error de asumir que dado que el recurso es un archivo físico separado cuando se desarrolla en un IDE (como Eclipse), será seguro tratarlo como un archivo físico separado en el caso general. Sin embargo, esto no es correcto; las aplicaciones casi siempre se distribuyen como archivos como archivos .jar o .war, y en tales casos, un recurso no será un archivo separado y no se podrá escribir. (El método getFile de la clase URL no es una solución para esto; a pesar de su nombre, simplemente devuelve la parte de la ruta de una URL, que de ninguna manera garantiza que sea un nombre de archivo válido).

No existe una forma segura de enumerar los recursos en tiempo de ejecución. Una vez más, dado que los desarrolladores son responsables de agregar archivos de recursos a la aplicación en el momento de la compilación, los desarrolladores ya deberían conocer sus rutas. Si bien existen soluciones alternativas, no son confiables y eventualmente fallarán.

Cargando configuración por defecto #

Para leer las propiedades de configuración predeterminadas:

package com.example;

public class ExampleApplication {
    private Properties getDefaults() throws IOException {
        Properties defaults = new Properties();

        try (InputStream defaultsStream =
            ExampleApplication.class.getResourceAsStream("config.properties")) {

            defaults.load(defaultsStream);
        }

        return defaults;
    }
}

Cargando una imagen desde un recurso #

Para cargar una imagen empaquetada:

package com.example;

public class ExampleApplication {
    private Image getIcon() throws IOException {
        URL imageURL = ExampleApplication.class.getResource("icon.png");
        return ImageIO.read(imageURL);
    }
}

Encontrar y leer recursos usando un cargador de clases #

La carga de recursos en Java comprende los siguientes pasos:

  1. Encontrar la Clase o ClassLoader que encontrará el recurso.
  2. Encontrar el recurso.
  3. Obtención del flujo de bytes del recurso.
  4. Lectura y procesamiento del flujo de bytes.
  5. Cerrar el flujo de bytes.

Los últimos tres pasos generalmente se logran al pasar la URL a un método de biblioteca o constructor para cargar el recurso. Por lo general, usará un método getResource en este caso. También es posible leer los datos de recursos en el código de la aplicación. Por lo general, usará getResourceAsStream en este caso.

Rutas de recursos absolutas y relativas #

Los recursos que se pueden cargar desde el classpath se indican con una ruta. La sintaxis de la ruta es similar a la ruta de un archivo UNIX/Linux. Consiste en nombres simples separados por caracteres de barra inclinada (/). Una ruta relativa comienza con un nombre y una ruta absoluta comienza con un separador.

Como describen los ejemplos de Classpath, la classpath de una JVM define un espacio de nombres superponiendo los espacios de nombres de los directorios y los archivos JAR o ZIP en la classpath. Cuando se resuelve una ruta absoluta, los cargadores de clases interpretan el / inicial como la raíz del espacio de nombres. Por el contrario, una ruta relativa puede resolverse en relación con cualquier “carpeta” en el espacio de nombres. La carpeta utilizada dependerá del objeto que utilice para resolver la ruta.

Obtención de una clase o cargador de clases #

Un recurso se puede ubicar usando un objeto Class o un objeto ClassLoader. Un objeto Clase puede resolver rutas relativas, por lo que normalmente usará uno de estos si tiene un recurso relativo (clase). Hay una variedad de formas de obtener un objeto Clase. Por ejemplo:

  • Un literal de clase le dará el objeto Clase para cualquier clase que pueda nombrar en el código fuente de Java; p.ej. String.class te da el objeto Class para el tipo String.

  • Object.getClass() te dará el objeto Class para el tipo de cualquier objeto; p.ej. "hello".getClass() es otra forma de obtener Class del tipo String.

  • El método Class.forName(String) (si es necesario) cargará dinámicamente una clase y devolverá su objeto Class; p.ej. Clase.forName("java.lang.String").

Un objeto ClassLoader normalmente se obtiene llamando a getClassLoader() en un objeto Class. También es posible hacerse con el cargador de clases predeterminado de la JVM utilizando el método estático ClassLoader.getSystemClassLoader().

Los métodos get #

Una vez que tenga una instancia de Class o ClassLoader, puede encontrar un recurso usando uno de los siguientes métodos:

Métodos Descripción
ClassLoader.getResource(ruta)
ClassLoader.getResources(ruta)		 Devuelve una URL que representa la ubicación del recurso con la ruta dada.
ClassLoader.getResources(ruta)
Class.getResources(ruta)		 Devuelve una Enumeración<URL> que proporciona las URL que se pueden usar para ubicar el recurso foo.bar; ver abajo.
ClassLoader.getResourceAsStream(ruta)
Class.getResourceStream(ruta)		 Devuelve un InputStream desde el que puede leer el contenido del recurso foo.bar como una secuencia de bytes.

Notas:

  • La principal diferencia entre las versiones ClassLoader y Class de los métodos está en la forma en que se interpretan las rutas relativas.

    • The Class methods resolve a relative path in the “folder” that corresponds to the classes package.
    • The ClassLoader methods treat relative paths as if they were absolute; i.e. the resolve them in the “root folder” of the classpath namespace.

  • Si no se puede encontrar el recurso (o recursos) solicitado, los métodos getResource y getResourceAsStreamdevuelvennull, y los métodos getResourcesdevuelven unaEnumeración` vacía.

  • Las URL devueltas se podrán resolver mediante URL.toStream(). Podrían ser URL archivo: u otras URL convencionales, pero si el recurso reside en un archivo JAR, serán URL jar: que identifican el archivo JAR y un recurso específico dentro de él.

  • Si su código usa un método getResourceAsStream (o URL.toStream()) para obtener un InputStream, es responsable de cerrar el objeto de flujo. Si no se cierra la secuencia, se podría producir una fuga de recursos.

Cargando recurso del mismo nombre desde múltiples JAR #

El recurso con la misma ruta y nombre puede existir en más de un archivo JAR en el classpath. Los casos comunes son recursos que siguen una convención o que forman parte de una especificación de empaquetado. Ejemplos de tales recursos son

  • META-INF/MANIFEST.MF
  • META-INF/beans.xml (especificación CDI)
  • Propiedades de ServiceLoader que contienen proveedores de implementación

Para tener acceso a todos estos recursos en diferentes jars, uno tiene que usar un ClassLoader, que tiene un método para esto. La ‘Enumeración’ devuelta se puede convertir convenientemente en una ‘Lista’ usando una función de Colecciones.

Enumeration<URL> resEnum = MyClass.class.getClassLoader().getResources("META-INF/MANIFEST.MF");
ArrayList<URL> resources = Collections.list(resEnum);
