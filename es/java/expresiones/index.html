Las expresiones en Java son la construcción principal para hacer cálculos.

Para obtener una referencia sobre los operadores que se pueden usar en expresiones, consulte [Operadores] (https://www.wikiod.com/es/java/operadores).

Precedencia de operadores #

Cuando una expresión contiene múltiples operadores, potencialmente se puede leer de diferentes maneras. Por ejemplo, la expresión matemática 1 + 2 x 3 podría leerse de dos formas:

  1. Sume 1 y 2 y multiplique el resultado por 3. Esto da la respuesta 9. Si agregamos paréntesis, esto se vería como ( 1 + 2 ) x 3.
  2. Suma 1 al resultado de multiplicar 2 y 3. Esto da la respuesta 7. Si agregamos paréntesis, esto se vería como 1 + (2 x 3).

En matemáticas, la convención es leer la expresión de la segunda manera. La regla general es que la multiplicación y la división se hacen antes que la suma y la resta. Cuando se usa una notación matemática más avanzada, el significado es “evidente” (¡para un matemático capacitado!), o se agregan paréntesis para eliminar la ambigüedad. En cualquier caso, la efectividad de la notación para transmitir significado depende de la inteligencia y el conocimiento compartido de los matemáticos.

Java tiene las mismas reglas claras sobre cómo leer una expresión, según la precedencia de los operadores que se utilizan.

En general, a cada operador se le atribuye un valor de precedencia; consulte la tabla a continuación.

Por ejemplo:

  1 + 2 * 3

La precedencia de + es menor que la precedencia de *, por lo que el resultado de la expresión es 7, no 9.

Orden de evaluación de expresiones #

Las expresiones Java se evalúan siguiendo las siguientes reglas:

  • Los operandos se evalúan de izquierda a derecha.
  • Los operandos de un operador se evalúan antes que el operador.
  • Los operadores se evalúan según la precedencia del operador.
  • Las listas de argumentos se evalúan de izquierda a derecha.

Ejemplo sencillo #

En el siguiente ejemplo:

int i = method1() + method2();

el orden de evaluacion es:

  1. El operando izquierdo del operador = se evalúa como la dirección de i.
  2. Se evalúa el operando izquierdo del operador + (method1()).
  3. Se evalúa el operando derecho del operador + (method2()).
  4. Se evalúa la operación +.
  5. Se evalúa la operación =, asignando el resultado de la suma a i.

Tenga en cuenta que si los efectos de las llamadas son observables, podrá observar que la llamada a method1 ocurre antes que la llamada a method2.

Ejemplo con un operador que tiene un efecto secundario #

En el siguiente ejemplo:

int i = 1;
intArray[i] = ++i + 1;

el orden de evaluacion es:

  1. Se evalúa el operando izquierdo del operador =. Esto da la dirección de intArray[1].
  2. Se evalúa el preincremento. Esto suma 1 a i y se evalúa como 2.
  3. Se evalúa el operando de la derecha de +.
  4. La operación + se evalúa como: 2 + 1 -> 3.
  5. Se evalúa la operación =, asignando 3 a intArray[1].

Tenga en cuenta que dado que el operando de la izquierda de = se evalúa primero, no está influenciado por el efecto secundario de la subexpresión ++i.

Referencia:

  • JLS 15.7 - Orden de evaluación

Fundamentos de las expresiones #

Las expresiones en Java son la construcción principal para hacer cálculos. Aquí hay unos ejemplos:

1                 // A simple literal is an expression
1 + 2             // A simple expression that adds two numbers
(i + j) / k       // An expression with multiple operations
(flag) ? c : d    // An expression using the "conditional" operator
(String) s        // A type-cast is an expression
obj.test()        // A method call is an expression
new Object()      // Creation of an object is an expression
new int[]         // Creation of an object is an expression

En general, una expresión consta de las siguientes formas:

  • Nombres de expresiones que consisten en:
    • Simple identifiers; e.g. someIdentifier
    • Qualified identifiers; e.g. MyClass.someField
  • Primarias que consisten en:
    • Literals; e.g. 1, 1.0, 'X', "hello", false and null
    • Class literal expressions; e.g. MyClass.class
    • this and <TypeName> . this
    • Parenthesized expressions; e.g. ( a + b )
    • Class instance creation expressions; e.g. new MyClass(1, 2, 3)
    • Array instance creation expressions; e.g. new int[3]
    • Field access expressions; e.g. obj.someField or this.someField
    • Array access expressions; e.g. vector[21]
    • Method invocations; e.g. obj.doIt(1, 2, 3)
    • Method references (Java 8 and later); e.g. MyClass::doIt
  • Expresiones de operadores unarios; p.ej. !a o i++
  • Expresiones de operadores binarios; p.ej. a + b o obj == nulo
  • Expresiones de operadores ternarios; p.ej. (objetivo == nulo) ? 1 : obj.getCount()
  • Expresiones lambda (Java 8 y posterior); p.ej. obj -> obj.getCount()

Los detalles de las diferentes formas de expresión se pueden encontrar en otros Temas.

  • El tema Operadores cubre expresiones de operadores unarios, binarios y ternarios.
  • El tema Expresiones lambda cubre las expresiones lambda y las expresiones de referencia de métodos.
  • El tema Clases y objetos cubre las expresiones de creación de instancias de clase.
  • El tema Matrices cubre las expresiones de acceso a matrices y las expresiones de creación de instancias de matrices.
  • El tema Literales cubre los diferentes tipos de expresiones literales.

El tipo de una expresión #

En la mayoría de los casos, una expresión tiene un tipo estático que se puede determinar en tiempo de compilación examinando y sus subexpresiones. Estas se denominan expresiones independientes.

Sin embargo, (en Java 8 y versiones posteriores) los siguientes tipos de expresiones pueden ser expresiones polivinílicas:

  • Expresiones entre paréntesis
  • Expresiones de creación de instancias de clase
  • Expresiones de invocación de métodos
  • Expresiones de referencia de métodos
  • Expresiones condicionales
  • Expresiones lambda

Cuando una expresión es una poliexpresión, su tipo puede estar influenciado por el tipo objetivo de la expresión; es decir, para qué se está utilizando.

El valor de una expresión #

El valor de una expresión es asignación compatible con su tipo. La excepción a esto es cuando se ha producido contaminación en pilas; p.ej. porque las advertencias de “conversión no segura” se han suprimido o ignorado (inapropiadamente).

Declaraciones de expresión #

A diferencia de muchos otros lenguajes, Java generalmente no permite que las expresiones se usen como declaraciones. Por ejemplo:

public void compute(int i, int j) {
    i + j;   // ERROR
}

Dado que el resultado de evaluar una expresión como no se puede usar, y dado que no puede afectar la ejecución del programa de ninguna otra manera, los diseñadores de Java tomaron la posición de que tal uso es un error o está equivocado.

Sin embargo, esto no se aplica a todas las expresiones. Un subconjunto de expresiones son (de hecho) legales como declaraciones. El conjunto comprende:

  • Expresión de asignación, incluidas las asignaciones operación-y-se-convierte.
  • Pre y post expresiones de incremento y decremento.
  • Llamadas a métodos (void o non-void).
  • Expresiones de creación de instancias de clase.

Expresiones constantes #

Una expresión constante es una expresión que produce un tipo primitivo o una cadena, y cuyo valor se puede evaluar en tiempo de compilación a un literal. La expresión debe evaluarse sin lanzar una excepción y debe estar compuesta únicamente por lo siguiente:

  • Literales primitivos y de cadena.

  • Escriba conversiones a tipos primitivos o String.

  • Los siguientes operadores unarios: +, -, ~ y !.

  • Los siguientes operadores binarios: *, /, %, +, -, <<, >>, >>>, <, <= , >, >=, ==, !=, &, ^, |, && y ||.

  • El operador condicional ternario ? :.

  • Expresiones constantes entre paréntesis.

  • Nombres simples que se refieren a variables constantes. (Una variable constante es una variable declarada como final donde la expresión inicializadora es en sí misma una expresión constante).

  • Nombres calificados de la forma <TypeName> . <Identificador> que se refieren a variables constantes.

Tenga en cuenta que la lista anterior excluye ++ y --, los operadores de asignación, clase e ​​instancia de, llamadas a métodos y referencias a variables o campos generales.

Las expresiones constantes de tipo String dan como resultado una String “interna”, y las operaciones de coma flotante en expresiones constantes se evalúan con semántica estricta de FP.

Usos de expresiones constantes #

Las expresiones constantes se pueden usar (prácticamente) en cualquier lugar donde se pueda usar una expresión normal. Sin embargo, tienen un significado especial en los siguientes contextos.

Se requieren expresiones constantes para las etiquetas de casos en las declaraciones de cambio. Por ejemplo:

switch (someValue) {
case 1 + 1:            // OK
case Math.min(2, 3):   // Error - not a constant expression
    doSomething();
}

Cuando la expresión del lado derecho de una asignación es una expresión constante, la asignación puede realizar una conversión de restricción primitiva. Esto está permitido siempre que el valor de la expresión constante esté dentro del rango del tipo del lado izquierdo. (Ver JLS 5.1.3 y 5.2) Por ejemplo:

byte b1 = 1 + 1;             // OK - primitive narrowing conversion.
byte b2 = 127 + 1;           // Error - out of range
byte b3 = b1 + 1;            // Error - not a constant expession
byte b4 = (byte) (b1 + 1);   // OK

Cuando una expresión constante se usa como condición en do, while o for, entonces afecta el análisis de legibilidad. Por ejemplo:

while (false) {
    doSomething();           // Error - statenent not reachable
}
boolean flag = false;
while (flag) {
    doSomething();           // OK
}

(Tenga en cuenta que esto no se aplica a las declaraciones if. El compilador de Java permite que el bloque then o else de una declaración if sea inalcanzable. Este es el análogo de Java de la compilación condicional en C y C++).

Finalmente, los campos static final en una clase o interfaz con inicializadores de expresión constante se inicializan con entusiasmo. Por lo tanto, se garantiza que estas constantes se observarán en el estado inicializado, incluso cuando hay un ciclo en el gráfico de dependencia de inicialización de clase.

Para obtener más información, consulte JLS 15.28. Expresiones constantes.
