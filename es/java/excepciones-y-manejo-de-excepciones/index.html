Los objetos de tipo Throwable y sus subtipos pueden enviarse a la pila con la palabra clave throw y capturarse con declaraciones try...catch.

Sintaxis #

  • void someMethod() lanza SomeException { } //declaración de método, fuerza método personas que llaman para detectar si SomeException es un tipo de excepción verificado

  • probar {

     someMethod(); //code that might throw an exception

    }

  • captura (AlgunaExcepción e) {

      System.out.println("SomeException was thrown!"); //code that will run if certain exception (SomeException) is thrown

    }

  • finalmente {

      //code that will always run, whether try block finishes or not

Atrapar una excepción con try-catch #

Una excepción puede ser capturada y manejada usando la sentencia try...catch. (De hecho, las declaraciones try toman otras formas, como se describe en otros ejemplos sobre try... catch...finally y try-with-resources.)

Try-catch con un bloque catch #

La forma más simple se ve así:

try {
    doSomething();
} catch (SomeException e) {
    handle(e);
}
// next statement

El comportamiento de un simple try...catch es el siguiente:

  • Se ejecutan las sentencias del bloque try.
  • Si las sentencias del bloque try no arrojan ninguna excepción, entonces el control pasa a la siguiente sentencia después de try...catch.
  • Si se lanza una excepción dentro del bloque try.
    • The exception object is tested to see if it is an instance of SomeException or a subtype.
    • If it is, then the catch block will catch the exception:
      • The variable e is bound to the exception object.
      • The code within the catch block is executed.
      • If that code throws an exception, then the newly thrown exception is propagated in place of the original one.
      • Otherwise, control passes to the next statement after the try...catch.
    • If it is not, the original exception continues to propagate.

Try-catch con capturas múltiples #

Un try...catch también puede tener múltiples bloques catch. Por ejemplo:

try {
    doSomething();
} catch (SomeException e) {
    handleOneWay(e)
} catch (SomeOtherException e) {
    handleAnotherWay(e);
}
// next statement

Si hay varios bloques catch, se prueban de uno en uno, comenzando por el primero, hasta que se encuentra una coincidencia para la excepción. El controlador correspondiente se ejecuta (como se indicó anteriormente), y luego se pasa el control a la siguiente declaración después de la declaración try...catch. Los bloques catch después del que coincide siempre se omiten, incluso si el código del controlador arroja una excepción.

La estrategia de emparejamiento “de arriba hacia abajo” tiene consecuencias para los casos en los que las excepciones en los bloques catch no son disjuntas. Por ejemplo:

try {
    throw new RuntimeException("test");
} catch (Exception e) {
    System.out.println("Exception");
} catch (RuntimeException e) {
    System.out.println("RuntimeException");
}

Este fragmento de código generará “Excepción” en lugar de “RuntimeException”. Dado que RuntimeException es un subtipo de Exception, la primera (más general) catch coincidirá. El segundo catch (más específico) nunca se ejecutará.

La lección que se debe aprender de esto es que los bloques catch más específicos (en términos de los tipos de excepción) deberían aparecer primero y los más generales deberían ser los últimos. (Algunos compiladores de Java le advertirán si nunca se puede ejecutar un catch, pero esto no es un error de compilación).

Bloques catch de excepciones múltiples #

A partir de Java SE 7, un solo bloque catch puede manejar una lista de excepciones no relacionadas. El tipo de excepción se enumera, separado por un símbolo de barra vertical (|). Por ejemplo:

try {
    doSomething();
} catch (SomeException | SomeOtherException e) {
    handleSomeException(e);
}

El comportamiento de una captura de excepción múltiple es una extensión simple para el caso de excepción única. El catch coincide si la excepción lanzada coincide (al menos) con una de las excepciones enumeradas.

Hay alguna sutileza adicional en la especificación. El tipo de e es una unión sintética de los tipos de excepción en la lista. Cuando se usa el valor de e, su tipo estático es el supertipo menos común de la unión de tipos. Sin embargo, si e se vuelve a lanzar dentro del bloque catch, los tipos de excepción que se lanzan son los tipos en la unión. Por ejemplo:

public void method() throws IOException, SQLException
    try {
        doSomething();
    } catch (IOException | SQLException e) {
        report(e);
        throw e;
    }

En lo anterior, IOException y SQLException son excepciones comprobadas cuyo supertipo menos común es Exception. Esto significa que el método report debe coincidir con report(Exception). Sin embargo, el compilador sabe que throw puede lanzar solo una IOException o una SQLException. Por lo tanto, método se puede declarar como throws IOException, SQLException en lugar de throws Exception. (Lo cual es bueno: consulte Pitfall - Throwing Throwable, Exception, Error or RuntimeException.)

La declaración de prueba con recursos #

Como ilustra el ejemplo try-catch-finalstatement, la limpieza de recursos usando una cláusula finally requiere una cantidad significativa de código “repetitivo” para implementar correctamente los casos extremos. Java 7 proporciona una forma mucho más sencilla de abordar este problema en forma de declaración pruebe-con-recursos.

¿Qué es un recurso? #

Java 7 introdujo la interfaz java.lang.AutoCloseable para permitir que las clases se administren usando la instrucción try-with-resources. Las instancias de clases que implementan AutoCloseable se conocen como recursos. Por lo general, estos deben eliminarse de manera oportuna en lugar de depender del recolector de basura para eliminarlos.

La interfaz AutoCloseable define un solo método:

public void close() throws Exception

Un método close() debería disponer del recurso de manera apropiada. La especificación establece que debería ser seguro llamar al método en un recurso que ya se ha eliminado. Además, las clases que implementan Autocloseable son fuertemente recomendadas para declarar el método close() para lanzar una excepción más específica que Exception, o ninguna excepción en absoluto.

Una amplia gama de clases e interfaces estándar de Java implementan AutoCloseable. Éstos incluyen:

  • InputStream, OutputStream y sus subclases
  • Lector, Escritor y sus subclases
  • Socket y ServerSocket y sus subclases
  • Channel y sus subclases, y
  • las interfaces JDBC Connection, Statement y ResultSet y sus subclases.

Las clases de aplicación y de terceros también pueden hacer esto.

La declaración básica de prueba con recursos #

La sintaxis de try-with-resources se basa en las formas clásicas try-catch, try-finally y try-catch-finally. Aquí hay un ejemplo de un formulario “básico”; es decir, la forma sin catch o finally.

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

Los recursos a gestionar se declaran como variables en la sección (...) después de la cláusula try. En el ejemplo anterior, declaramos una variable de recurso stream y la inicializamos en un PrintStream recién creado.

Una vez inicializadas las variables de recurso, se ejecuta el bloque try. Cuando se complete, stream.close() se llamará automáticamente para garantizar que el recurso no se filtre. Tenga en cuenta que la llamada close() ocurre sin importar cómo se complete el bloque.

Las declaraciones mejoradas de prueba con recursos #

La instrucción try-with-resources se puede mejorar con bloques catch y finally, como con la sintaxis anterior a Java 7 try-catch-finally. El siguiente fragmento de código agrega un bloque catch al anterior para lidiar con la excepción FileNotFoundException que puede generar el constructor PrintStream:

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
} catch (FileNotFoundException ex) {
    System.err.println("Cannot open the file");
} finally {
    System.err.println("All done");
}

Si la inicialización del recurso o el bloque try arrojan la excepción, se ejecutará el bloque catch. El bloque finally siempre se ejecutará, como con una declaración convencional try-catch-finally.

Sin embargo, hay un par de cosas a tener en cuenta:

  • La variable de recurso está fuera de alcance en los bloques catch y finally.
  • La limpieza de recursos ocurrirá antes de que la declaración intente hacer coincidir el bloque catch.
  • Si la limpieza automática de recursos lanzó una excepción, eso podría quedar atrapado en uno de los bloques catch.

Gestión de varios recursos #

Los fragmentos de código anteriores muestran la gestión de un solo recurso. De hecho, pruebe-con-recursos puede administrar múltiples recursos en una declaración. Por ejemplo:

try (InputStream is = new FileInputStream(file1);
     OutputStream os = new FileOutputStream(file2)) {
    // Copy 'is' to 'os'
}

Esto se comporta como cabría esperar. Tanto is como os se cierran automáticamente al final del bloque try. Hay un par de puntos a tener en cuenta:

  • Las inicializaciones ocurren en el orden del código, y los inicializadores de variables de recursos posteriores pueden usar los valores de los anteriores.
  • Se limpiarán todas las variables de recursos que se inicializaron con éxito.
  • Las variables de recursos se limpian en orden inverso de sus declaraciones.

Por lo tanto, en el ejemplo anterior, is se inicializa antes que os y se limpia después, y is se limpiará si hay una excepción al inicializar os.

Equivalencia de try-with-resource y try-catch-finally clásico

La especificación del lenguaje Java especifica el comportamiento de los formularios try-with-resource en términos de la declaración clásica try-catch-finally. (Consulte el JLS para obtener todos los detalles).

Por ejemplo, este pruebe-con-recurso básico:

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

se define como equivalente a este try-catch-finally:

// Note that the constructor is not part of the try-catch statement
PrintStream stream = new PrintStream("hello.txt");

// This variable is used to keep track of the primary exception thrown
// in the try statement. If an exception is thrown in the try block,
// any exception thrown by AutoCloseable.close() will be suppressed.
Throwable primaryException = null;

// The actual try block
try {
    stream.println("Hello world!");
} catch (Throwable t) {
    // If an exception is thrown, remember it for the finally block
    primaryException = t;
    throw t;
} finally {
    if (primaryException == null) {
        // If no exception was thrown so far, exceptions thrown in close() will
        // not be caught and therefore be passed on to the enclosing code.
        stream.close();
    } else {
        // If an exception has already been thrown, any exception thrown in
        // close() will be suppressed as it is likely to be related to the
        // previous exception. The suppressed exception can be retrieved
        // using primaryException.getSuppressed().
        try {
            stream.close();
        } catch (Throwable suppressedException) {
            primaryException.addSuppressed(suppressedException);
        }
    }
}

(El JLS especifica que las variables reales t y primaryException serán invisibles para el código Java normal).

La forma mejorada de pruebe-con-recursos se especifica como una equivalencia con la forma básica. Por ejemplo:

try (PrintStream stream = new PrintStream(fileName)) {
    stream.println("Hello world!");
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

es equivalente a:

try {
    try (PrintStream stream = new PrintStream(fileName)) {
        stream.println("Hello world!");
    }
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

Excepciones personalizadas #

En la mayoría de las circunstancias, es más sencillo desde el punto de vista del diseño del código utilizar las clases genéricas Exception existentes al generar excepciones. Esto es especialmente cierto si solo necesita que la excepción transmita un mensaje de error simple. En ese caso, generalmente se prefiere RuntimeException, ya que no es una excepción marcada. Existen otras clases de excepción para clases comunes de errores:

  • UnsupportedOperationException: una determinada operación no es compatible
  • IllegalArgumentException: se pasó un valor de parámetro no válido a un método
  • IllegalStateException: su API ha alcanzado internamente una condición que nunca debería ocurrir o que ocurre como resultado de usar su API de una manera inválida

Los casos en los que sí desea utilizar una clase de excepción personalizada incluyen los siguientes:

  • Está escribiendo una API o biblioteca para que la usen otros, y desea permitir que los usuarios de su API puedan capturar y manejar excepciones de su API específicamente, y ser capaces de diferenciar esas excepciones de otras excepciones más genéricas .
  • Está lanzando excepciones para un tipo específico de error en una parte de su programa, que desea capturar y manejar en otra parte de su programa, y ​​desea poder diferenciar estos errores de otros, más errores genéricos.

Puede crear sus propias excepciones personalizadas extendiendo RuntimeException para una excepción no verificada, o una excepción verificada extendiendo cualquier Exception que no sea también una subclase de RuntimeException, porque:

Las subclases de excepción que no son también subclases de RuntimeException son excepciones comprobadas

public class StringTooLongException extends RuntimeException {
    // Exceptions can have methods and fields like other classes
    // those can be useful to communicate information to pieces of code catching
    // such an exception
    public final String value;
    public final int maximumLength;

    public StringTooLongException(String value, int maximumLength){
        super(String.format("String exceeds maximum Length of %s: %s", maximumLength, value));
        this.value = value;
        this.maximumLength = maximumLength;
    }
}

Esos se pueden usar solo como excepciones predefinidas:

void validateString(String value){
    if (value.length() > 30){
        throw new StringTooLongException(value, 30);
    }
}

Y los campos se pueden usar donde se captura y maneja la excepción:

void anotherMethod(String value){
    try {
        validateString(value);
    } catch(StringTooLongException e){
        System.out.println("The string '" + e.value + 
                "' was longer than the max of " + e.maximumLength );
    }
}

Manejo de excepciones interrumpidas #

InterruptedException es una bestia confusa: aparece en métodos aparentemente inocuos como Thread.sleep(), pero manejarlo incorrectamente genera un código difícil de administrar que se comporta mal en entornos concurrentes.

En su forma más básica, si se detecta una InterruptedException, significa que alguien, en algún lugar, llamó Thread.interrupt() en el subproceso en el que se está ejecutando actualmente su código. Es posible que se incline a decir “Es mi código! ¡Nunca lo interrumpiré!” y por lo tanto hacer algo como esto:

// Bad. Don't do this.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  // disregard
}

Pero esta es exactamente la forma incorrecta de manejar un evento “imposible”. Si sabe que su aplicación nunca encontrará una “Excepción interrumpida”, debe tratar dicho evento como una violación grave de las suposiciones de su programa y salir lo más rápido posible.

La forma correcta de manejar una interrupción “imposible” es así:

// When nothing will interrupt your code
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  throw new AssertionError(e);
}

Esto hace dos cosas; primero restaura el estado de interrupción del subproceso (como si la InterruptedException no se hubiera lanzado en primer lugar), y luego lanza un AssertionError que indica que se han violado las invariantes básicas de su aplicación. Si sabe con certeza que nunca interrumpirá el hilo en el que se ejecuta este código, es seguro ya que nunca se debe alcanzar el bloque catch.

Usar la clase Uninterruptibles de Guava ayuda a simplificar este patrón; llamar a Uninterruptibles.sleepUninterruptably() ignora el estado interrumpido de un subproceso hasta que la duración de la suspensión haya expirado (momento en el que se restaura para llamadas posteriores para inspeccionar y lanzar su propia InterruptedException). Si sabe que nunca interrumpirá dicho código, esto evita de manera segura la necesidad de envolver sus llamadas de sueño en un bloque de prueba y captura.

Sin embargo, con mayor frecuencia, no puede garantizar que su hilo nunca se interrumpa. En particular, si está escribiendo código que será ejecutado por un [Executor][6] o alguna otra gestión de subprocesos, es fundamental que su código responda rápidamente a las interrupciones, de lo contrario, su aplicación se detendrá o incluso se bloqueará.

En tales casos, lo mejor que se puede hacer generalmente es permitir que la InterruptedException se propague hacia arriba en la pila de llamadas, agregando throws InterruptedException a cada método por turno. Esto puede parecer torpe, pero en realidad es una propiedad deseable: las firmas de su método ahora indican a las personas que llaman que responderá rápidamente a las interrupciones.

// Let the caller determine how to handle the interrupt if you're unsure
public void myLongRunningMethod() throws InterruptedException {
  ...
}

En casos limitados (por ejemplo, al anular un método que no ’lanza’ ninguna excepción comprobada), puede restablecer el estado interrumpido sin generar una excepción, esperando que el código que se ejecute a continuación maneje la interrupción. Esto retrasa el manejo de la interrupción pero no la suprime por completo.

// Suppresses the exception but resets the interrupted state letting later code
// detect the interrupt and handle it properly.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  return ...; // your expectations are still broken at this point - try not to do more work.
}

Devuelve sentencias en el bloque try catch #

Aunque es una mala práctica, es posible agregar varias declaraciones de devolución en un bloque de manejo de excepciones:

 public static int returnTest(int number){
    try{
        if(number%2 == 0) throw new Exception("Exception thrown");
        else return x;
    }
    catch(Exception e){
        return 3;
    }
    finally{
        return 7;
    }
}

Este método siempre devolverá 7 ya que el bloque finalmente asociado con el bloque try/catch se ejecuta antes de que se devuelva nada. Ahora, como finalmente tiene return 7;, este valor reemplaza los valores de retorno de try/catch.

Si el bloque catch devuelve un valor primitivo y ese valor primitivo se cambia posteriormente en el bloque finalmente, se devolverá el valor devuelto en el bloque catch y se ignorarán los cambios del bloque finalmente.

El siguiente ejemplo imprimirá “0”, no “1”.

public class FinallyExample {

    public static void main(String[] args) {
        int n = returnTest(4);
        
        System.out.println(n);
    }

    public static int returnTest(int number) {
        
        int returnNumber = 0; 
        
        try {
            if (number % 2 == 0)
                throw new Exception("Exception thrown");
            else
                return returnNumber;
        } catch (Exception e) {
            return returnNumber;
        } finally {
            returnNumber = 1;
        }
    }
}

La jerarquía de excepciones de Java: excepciones no verificadas y verificadas #

Todas las excepciones de Java son instancias de clases en la jerarquía de clases de excepciones. Esto se puede representar de la siguiente manera:

  • java.lang.Throwable - Esta es la clase base para todas las clases de excepción. Sus métodos y constructores implementan una gama de funciones comunes a todas las excepciones.
    • java.lang.Exception - This is the superclass of all normal exceptions.
      • various standard and custom exception classes.
      • java.lang.RuntimeException - This the superclass of all normal exceptions that are unchecked exceptions.
        • various standard and custom runtime exception classes.
    • java.lang.Error - This is the superclass of all “fatal error” exceptions.

Notas:

  1. La distinción entre las excepciones marcadas y no marcadas se describe a continuación.
  2. Las clases Throwable, Exception y RuntimeException deben tratarse como abstractas; consulte https://www.wikiod.com/es/java/errores-de-java-uso-de-excepcionespitfall-throwing-throwable-exception-error-or-runtimeexception#t=201610240803173325134.
  3. Las excepciones Error son lanzadas por la JVM en situaciones en las que sería inseguro o imprudente que una aplicación intentara recuperarse.
  4. No sería prudente declarar subtipos personalizados de Throwable. Las herramientas y bibliotecas de Java pueden asumir que Error y Exception son los únicos subtipos directos de Throwable, y comportarse mal si esa suposición es incorrecta.

Excepciones marcadas versus no marcadas #

Una de las críticas al soporte de excepciones en algunos lenguajes de programación es que es difícil saber qué excepciones podría arrojar un método o procedimiento determinado. Dado que una excepción no controlada puede hacer que un programa se bloquee, esto puede hacer que las excepciones sean una fuente de fragilidad.

El lenguaje Java aborda este problema con el mecanismo de excepción comprobado. Primero, Java clasifica las excepciones en dos categorías:

  • Las excepciones marcadas generalmente representan eventos anticipados que una aplicación debería poder manejar. Por ejemplo, IOException y sus subtipos representan condiciones de error que pueden ocurrir en las operaciones de E/S. Los ejemplos incluyen fallas en la apertura de archivos porque un archivo o directorio no existe, fallas en las lecturas y escrituras en la red porque se ha interrumpido una conexión de red, etc.

  • Las excepciones no verificadas generalmente representan eventos imprevistos que una aplicación no puede manejar. Suelen ser el resultado de un error en la aplicación.

(En lo siguiente, “lanzado” se refiere a cualquier excepción lanzada explícitamente (mediante una declaración throw), o implícitamente (en una desreferencia fallida, conversión de tipos, etc.). De manera similar, “propagado” se refiere a una excepción que se lanzó en una llamada anidada y no se detectó dentro de esa llamada. El código de ejemplo a continuación ilustrará esto).

La segunda parte del mecanismo de excepción verificada es que existen restricciones en los métodos en los que puede ocurrir una excepción verificada:

  • Cuando una excepción verificada es lanzada o propagada en un método, debe ser capturada por el método o listada en la cláusula throws del método. (El significado de la cláusula throws se describe en este ejemplo.)
  • Cuando se lanza o propaga una excepción verificada en un bloque inicializador, debe capturarse en el bloque.
  • Una excepción comprobada no se puede propagar mediante una llamada de método en una expresión de inicialización de campo. (No hay forma de detectar tal excepción).

En resumen, una excepción verificada debe manejarse o declararse.

Estas restricciones no se aplican a las excepciones no verificadas. Esto incluye todos los casos en los que se lanza una excepción implícitamente, ya que todos esos casos lanzan excepciones no verificadas.

Ejemplos de excepciones comprobadas #

Estos fragmentos de código están destinados a ilustrar las restricciones de excepción comprobadas. En cada caso, mostramos una versión del código con un error de compilación y una segunda versión con el error corregido.

// This declares a custom checked exception.
public class MyException extends Exception {
    // constructors omitted.
}

// This declares a custom unchecked exception.
public class MyException2 extends RuntimeException {
    // constructors omitted.
}

El primer ejemplo muestra cómo las excepciones verificadas lanzadas explícitamente pueden declararse como “lanzadas” si no deben manejarse en el método.

// INCORRECT
public void methodThrowingCheckedException(boolean flag) {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

// CORRECTED
public void methodThrowingCheckedException(boolean flag) throws MyException {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

El segundo ejemplo muestra cómo se puede tratar una excepción comprobada propagada.

// INCORRECT 
public void methodWithPropagatedCheckedException() {
    InputStream is = new FileInputStream("someFile.txt");  // Compilation error
    // FileInputStream throws IOException or a subclass if the file cannot 
    // be opened.  IOException is a checked exception.
    ...
}

// CORRECTED (Version A) 
public void methodWithPropagatedCheckedException() throws IOException {
    InputStream is = new FileInputStream("someFile.txt"); 
    ...
}

// CORRECTED (Version B) 
public void methodWithPropagatedCheckedException() {
    try {
        InputStream is = new FileInputStream("someFile.txt"); 
        ...
    } catch (IOException ex) {
        System.out.println("Cannot open file: " + ex.getMessage());
    }
}

El ejemplo final muestra cómo lidiar con una excepción verificada en un inicializador de campo estático.

// INCORRECT
public class Test {
    private static final InputStream is = 
            new FileInputStream("someFile.txt");  // Compilation error
}

// CORRECTED
public class Test {
    private static final InputStream is;
    static {
        InputStream tmp = null;
        try {
            tmp = new FileInputStream("someFile.txt");
        } catch (IOException ex) {
            System.out.println("Cannot open file: " + ex.getMessage());
        }
        is = tmp;
    }
}

Tenga en cuenta que en este último caso, también tenemos que lidiar con los problemas de que “es” no se puede asignar más de una vez y, sin embargo, también debe asignarse, incluso en el caso de una excepción.

Exception in thread "main" java.lang.ArithmeticException: / by zero 
    at Division.main(Disivion.java:14)

La división por cero es una operación no válida que produciría un valor que no se puede representar como un número entero. Java soluciona esto lanzando una excepción. En este caso, la excepción es una instancia de la clase ArithmeticException.

Nota: El ejemplo de creación y lectura de seguimientos de pila explica qué significa el resultado después de los dos números.

La utilidad de una excepción es el control de flujo que permite. Sin usar excepciones, una solución típica a este problema puede ser verificar primero si b == 0:

class Division {
    public static void main(String[] args) {
 
        int a, b, result;

        Scanner input = new Scanner(System.in);
        System.out.println("Input two integers");
 
        a = input.nextInt();
        b = input.nextInt();
 
        if (b == 0) {
            System.out.println("You cannot divide by zero.");
            return;
        }

        result = a / b;
 
        System.out.println("Result = " + result);
    }
}

Esto imprime el mensaje No se puede dividir por cero. en la consola y sale del programa con gracia cuando el usuario intenta dividir por cero. Una forma equivalente de lidiar con este problema a través del manejo de excepciones sería reemplazar el control de flujo if con un bloque try-catch:

...

a = input.nextInt();
b = input.nextInt();
 
try {
    result = a / b;
}
catch (ArithmeticException e) {
    System.out.println("An ArithmeticException occurred. Perhaps you tried to divide by zero.");
    return;
}
 
...

Un bloque try catch se ejecuta de la siguiente manera:

  1. Comience a ejecutar el código en el bloque try.
  2. Si ocurre una excepción en el bloque try, cancele inmediatamente y verifique si esta excepción es atrapada por el bloque catch (en este caso, cuando la excepción es una instancia de ArithmeticException).
  3. Si la excepción es capturada, se asigna a la variable e y se ejecuta el bloque catch.
  4. Si se completa el bloque try o catch (es decir, no se producen excepciones no detectadas durante la ejecución del código), continúe ejecutando el código debajo del bloque try-catch.

Por lo general, se considera una buena práctica usar el manejo de excepciones como parte del control de flujo normal de una aplicación donde, de otro modo, el comportamiento sería indefinido o inesperado. Por ejemplo, en lugar de devolver null cuando falla un método, suele ser una mejor práctica lanzar una excepción para que la aplicación que utiliza el método pueda definir su propio control de flujo para la situación a través del manejo de excepciones del tipo ilustrado arriba. En cierto sentido, esto soluciona el problema de tener que devolver un tipo particular, ya que cualquiera de los múltiples tipos de excepciones puede lanzarse para indicar el problema específico que ocurrió.

Para obtener más consejos sobre cómo y cómo no usar excepciones, consulte https://www.wikiod.com/es/java/errores-de-java-uso-de-excepciones

Creando y leyendo stacktraces #

Cuando se crea un objeto de excepción (es decir, cuando lo ’nuevo’), el constructor ‘Throwable’ captura información sobre el contexto en el que se creó la excepción. Más tarde, esta información se puede generar en forma de seguimiento de pila, que se puede usar para ayudar a diagnosticar el problema que causó la excepción en primer lugar.

Imprimiendo un stacktrace #

Imprimir un stacktrace es simplemente una cuestión de llamar al método printStackTrace(). Por ejemplo:

try {
    int a = 0;
    int b = 0;
    int c = a / b;
} catch (ArithmeticException ex) {
    // This prints the stacktrace to standard output
    ex.printStackTrace();
}

El método printStackTrace() sin argumentos imprimirá en la salida estándar de la aplicación; es decir, el System.out actual. También hay sobrecargas printStackTrace(PrintStream) y printStackTrace(PrintWriter) que imprimen en un Stream o Writer especificado.

Notas:

  1. Stacktrace no incluye los detalles de la excepción en sí. Puedes usar el método toString() para obtener esos detalles; p.ej.

         // Print exception and stacktrace
     System.out.println(ex);
     ex.printStackTrace();

  2. La impresión Stacktrace debe usarse con moderación; consulte https://www.wikiod.com/es/java/errores-de-java-uso-de-excepciones#Pitfall - Stacktraces excesivos o inapropiados A menudo es mejor usar un marco de registro y pasar el objeto de excepción para que se registre.

Comprender un seguimiento de pila #

Considere el siguiente programa simple que consta de dos clases en dos archivos. (Hemos mostrado los nombres de archivo y los números de línea agregados con fines ilustrativos).

File: "Main.java"
1   public class Main {
2       public static void main(String[] args) {
3           new Test().foo();
4       }
5   }

File: "Test.java"
1   class Test {
2       public void foo() {
3           bar();
4       }
5   
6       public int bar() {
7           int a = 1;
8           int b = 0;
9           return a / b;
10      }

Cuando estos archivos se compilan y ejecutan, obtendremos el siguiente resultado.

Exception in thread "main" java.lang.ArithmeticException: / by zero
        at Test.bar(Test.java:9)
        at Test.foo(Test.java:3)
        at Main.main(Main.java:3)

Leamos esta línea a la vez para averiguar lo que nos está diciendo.

La línea #1 nos dice que el subproceso llamado “principal” ha terminado debido a una excepción no detectada. El nombre completo de la excepción es java.lang.ArithmeticException, y el mensaje de excepción es “/ por cero”.

Si buscamos los javadocs para esta excepción, dice:

Lanzado cuando se ha producido una condición aritmética excepcional. Por ejemplo, un entero “dividir por cero” arroja una instancia de esta clase.

De hecho, el mensaje “/ por cero” es un fuerte indicio de que la causa de la excepción es que algún código ha intentado dividir algo por cero. ¿Pero que?

Las 3 líneas restantes son el seguimiento de la pila. Cada línea representa una llamada de método (o constructor) en la pila de llamadas, y cada una nos dice tres cosas:

  • el nombre de la clase y el método que se estaba ejecutando,
  • el nombre del archivo del código fuente,
  • el número de línea del código fuente de la declaración que se estaba ejecutando

Estas líneas de un seguimiento de pila se enumeran con el marco de la llamada actual en la parte superior. El marco superior en nuestro ejemplo anterior está en el método Test.bar y en la línea 9 del archivo Test.java. Esa es la siguiente línea:

    return a / b;

Si miramos un par de líneas antes en el archivo donde se inicializa b, es evidente que b tendrá el valor cero. Podemos decir sin ninguna duda que esta es la causa de la excepción.

Si necesitáramos ir más allá, podemos ver en el stacktrace que bar() fue llamado desde foo() en la línea 3 de Test.java, y que foo() fue llamado a su vez desde Main. principal().

Nota: Los nombres de clase y método en los marcos de pila son los nombres internos de las clases y métodos. Deberá reconocer los siguientes casos inusuales:

  • Una clase anidada o interna se verá como “OuterClass$InnerClass”.
  • Una clase interna anónima se verá como “OuterClass$1”, “OuterClass$2”, etcétera.
  • Cuando se ejecuta el código en un constructor, un inicializador de campo de instancia o un bloque de inicializador de instancia, el nombre del método será “”.
  • Cuando se ejecuta código en un inicializador de campo estático o en un bloque de inicializador estático, el nombre del método será “”.

(En algunas versiones de Java, el código de formateo de stacktrace detectará y eliminará secuencias repetidas de marcos de pila, como puede ocurrir cuando una aplicación falla debido a una recursividad excesiva).

Encadenamiento de excepciones y stacktraces anidados #

El encadenamiento de excepciones ocurre cuando una pieza de código detecta una excepción y luego crea y lanza una nueva, pasando la primera excepción como la causa. Aquí hay un ejemplo:

File: Test,java
1   public class Test {
2      int foo() {
3           return 0 / 0;
4      }
5
6       public Test() {
7           try {
8               foo();
9           } catch (ArithmeticException ex) {
10              throw new RuntimeException("A bad thing happened", ex);
11          }
12      }
13
14      public static void main(String[] args) {
15          new Test();
16      }
17  }

Cuando la clase anterior se compila y ejecuta, obtenemos el siguiente seguimiento de pila:

Exception in thread "main" java.lang.RuntimeException: A bad thing happened
        at Test.<init>(Test.java:10)
        at Test.main(Test.java:15)
Caused by: java.lang.ArithmeticException: / by zero
        at Test.foo(Test.java:3)
        at Test.<init>(Test.java:8)
        ... 1 more

El seguimiento de la pila comienza con el nombre de la clase, el método y la pila de llamadas para la excepción que (en este caso) provocó el bloqueo de la aplicación. A esto le sigue una línea “Causado por:” que informa la excepción causa. Se informa el nombre de la clase y el mensaje, seguidos de los marcos de pila de la excepción de la causa. La traza termina con un “… N más” que indica que los últimos N fotogramas son los mismos que para la excepción anterior.

El “Causado por:” solo se incluye en la salida cuando la causa de la excepción principal no es nula). Las excepciones se pueden encadenar indefinidamente y, en ese caso, el seguimiento de la pila puede tener varios seguimientos “Causado por:”.

Nota: el mecanismo cause solo se expuso en la API Throwable en Java 1.4.0. Antes de eso, la aplicación debía implementar el encadenamiento de excepciones mediante un campo de excepción personalizado para representar la causa y un método printStackTrace personalizado.

Capturar un stacktrace como una cadena #

A veces, una aplicación necesita ser capaz de capturar un seguimiento de pila como una ‘Cadena’ de Java, para que pueda usarse para otros fines. El enfoque general para hacer esto es crear un ‘OutputStream’ o ‘Writer’ temporal que escribe en un búfer en memoria y lo pasa a ‘printStackTrace(…)’.

Las bibliotecas Apache Commons y Guava proporcionan métodos de utilidad para capturar un seguimiento de pila como una cadena:

org.apache.commons.lang.exception.ExceptionUtils.getStackTrace(Throwable)

com.google.common.base.Throwables.getStackTraceAsString(Throwable)

Si no puede usar bibliotecas de terceros en su base de código, entonces el siguiente método hará la tarea:

   /**
     * Returns the string representation of the stack trace.
     *
     * @param throwable the throwable
     * @return the string.
     */
    public static String stackTraceToString(Throwable throwable) {
        StringWriter stringWriter = new StringWriter();
        throwable.printStackTrace(new PrintWriter(stringWriter));
        return stringWriter.toString();
    }

Tenga en cuenta que si su intención es analizar el seguimiento de la pila, es más sencillo usar getStackTrace() y getCause() que intentar analizar un seguimiento de la pila.

Lanzar una excepción #

El siguiente ejemplo muestra los conceptos básicos para lanzar una excepción:

public void checkNumber(int number) throws IllegalArgumentException {
    if (number < 0) {
        throw new IllegalArgumentException("Number must be positive: " + number);
    }
}

La excepción se lanza en la tercera línea. Esta declaración se puede dividir en dos partes:

  • new IllegalArgumentException(...) está creando una instancia de la clase IllegalArgumentException, con un mensaje que describe el error que informa la excepción.

  • throw ... está lanzando el objeto de excepción.

Cuando se lanza la excepción, hace que las declaraciones adjuntas finalicen de manera anormal hasta que la excepción se maneje. Esto se describe en otros ejemplos.

Es una buena práctica crear y lanzar el objeto de excepción en una sola declaración, como se muestra arriba. También es una buena práctica incluir un mensaje de error significativo en la excepción para ayudar al programador a comprender la causa del problema. Sin embargo, este no es necesariamente el mensaje que debe mostrar al usuario final. (Para empezar, Java no tiene soporte directo para internacionalizar mensajes de excepción).

Hay un par de puntos más que hacer:

  • Hemos declarado checkNumber como throws IllegalArgumentException. Esto no era estrictamente necesario, ya que IllegalArgumentException es una excepción comprobada; consulte https://www.wikiod.com/es/java/excepciones-y-manejo-de-excepciones#La jerarquía de excepciones de Java: excepciones no verificadas y verificadas Sin embargo, es una buena práctica hacer esto y también incluir las excepciones lanzadas en los comentarios javadoc de un método.

  • El código inmediatamente después de una instrucción throw es inaccesible. Por lo tanto, si escribimos esto:

       throw new IllegalArgumentException("it is bad");
   return;

    the compiler would report a compilation error for the return statement.

Encadenamiento de excepciones #

Muchas excepciones estándar tienen un constructor con un segundo argumento causa además del argumento mensaje convencional. La causa le permite encadenar excepciones. Aquí hay un ejemplo.

Primero definimos una excepción no verificada que nuestra aplicación lanzará cuando encuentre un error no recuperable. Tenga en cuenta que hemos incluido un constructor que acepta un argumento causa.

    public class AppErrorException extends RuntimeException {
        public AppErrorException() {
            super();
        }

        public AppErrorException(String message) {
            super(message);
        }

        public AppErrorException(String message, Throwable cause) {
            super(message, cause);
        }
    }

A continuación, aquí hay un código que ilustra el encadenamiento de excepciones.

    public String readFirstLine(String file) throws AppErrorException {
        try (Reader r = new BufferedReader(new FileReader(file))) {
            String line = r.readLine();
            if (line != null) {
                return line;
            } else {
                throw new AppErrorException("File is empty: " + file);
            }
        } catch (IOException ex) {
            throw new AppErrorException("Cannot read file: " + file, ex);
        }
    }

El throw dentro del bloque try detecta un problema y lo informa a través de una excepción con un mensaje simple. Por el contrario, el throw dentro del bloque catch está manejando la IOException envolviéndola en una nueva excepción (marcada). Sin embargo, no descarta la excepción original. Al pasar la IOException como la causa, la registramos para que pueda imprimirse en el seguimiento de la pila, como se explica en https://www.wikiod.com/es/java/excepciones-y-manejo-de-excepciones 1815/creación-y-lectura-stacktraces#t=20161028160848944857.

Funciones avanzadas de Excepciones #

Este ejemplo cubre algunas características avanzadas y casos de uso para Excepciones.

Examinando la pila de llamadas programáticamente #

El uso principal de los seguimientos de pila de excepción es proporcionar información sobre un error de aplicación y su contexto para que el programador pueda diagnosticar y solucionar el problema. A veces se puede utilizar para otras cosas. Por ejemplo, una clase SecurityManager puede necesitar examinar la pila de llamadas para decidir si se debe confiar en el código que está realizando una llamada.

Puede usar excepciones para examinar la pila de llamadas programáticamente de la siguiente manera:

    Exception ex = new Exception();   // this captures the call stack
    StackTraceElement[] frames = ex.getStackTrace();
    System.out.println("This method is " + frames[0].getMethodName());
    System.out.println("Called from method " + frames[1].getMethodName());

Hay algunas advertencias importantes sobre esto:

  1. La información disponible en un StackTraceElement es limitada. No hay más información disponible que la que muestra printStackTrace. (Los valores de las variables locales en el marco no están disponibles).

  2. Los javadocs para getStackTrace() indican que una JVM puede omitir marcos:

    Some virtual machines may, under some circumstances, omit one or more stack frames from the stack trace. In the extreme case, a virtual machine that has no stack trace information concerning this throwable is permitted to return a zero-length array from this method.

Optimización de la construcción de excepciones #

Como se mencionó en otra parte, construir una excepción es bastante costoso porque implica capturar y registrar información sobre todos los marcos de pila en el subproceso actual. A veces, sabemos que esa información nunca se utilizará para una excepción determinada; p.ej. el stacktrace nunca se imprimirá. En ese caso, hay un truco de implementación que podemos usar en una excepción personalizada para hacer que la información no sea capturada.

La información del marco de pila necesaria para stacktraces se captura cuando los constructores Throwable llaman al método Throwable.fillInStackTrace(). Este método es público, lo que significa que una subclase puede anularlo. El truco es anular el método heredado de Throwable con uno que no hace nada; p.ej.

  public class MyException extends Exception {
      // constructors

      @Override 
      public void fillInStackTrace() {
          // do nothing
      }
  }

El problema con este enfoque es que una excepción que anula fillInStackTrace() nunca puede capturar el seguimiento de pila y es inútil en escenarios donde lo necesita.

Borrar o reemplazar el stacktrace #

En algunas situaciones, el seguimiento de pila de una excepción creada de forma normal contiene información incorrecta o información que el desarrollador no desea revelar al usuario. Para estos escenarios, Throwable.setStackTrace se puede usar para reemplazar la matriz de objetos StackTraceElement que contiene la información.

Por ejemplo, se puede usar lo siguiente para descartar la información de la pila de una excepción:

 exception.setStackTrace(new StackTraceElement[0]);

excepciones suprimidas #

Java 7 introdujo la construcción try-with-resources y el concepto asociado de supresión de excepciones. Considere el siguiente fragmento:

try (Writer w = new BufferedWriter(new FileWriter(someFilename))) {
    // do stuff
    int temp = 0 / 0;    // throws an ArithmeticException
}

Cuando se lanza la excepción, try llamará a close() en w, que vaciará cualquier salida almacenada en el búfer y luego cerrará FileWriter. Pero, ¿qué sucede si se lanza una IOException al vaciar la salida?

Lo que sucede es que cualquier excepción que se lanza al limpiar un recurso se suprime. La excepción se captura y se agrega a la lista de excepciones suprimidas de la excepción principal. A continuación, probar-con-recursos continuará con la limpieza de los otros recursos. Finalmente, se volverá a generar la excepción principal.

Un patrón similar ocurre si se lanza una excepción durante la inicialización del recurso, o si el bloque try se completa normalmente. La primera excepción lanzada se convierte en la excepción principal y las subsiguientes que surgen de la limpieza se suprimen.

Las excepciones suprimidas se pueden recuperar del objeto de excepción principal llamando a getSuppressedExceptions.

Las sentencias try-finally y try-catch-finally #

La sentencia try...catch...finally combina el manejo de excepciones con código de limpieza. El bloque finally contiene código que se ejecutará en todas las circunstancias. Esto los hace adecuados para la gestión de recursos y otros tipos de limpieza.

intentar-finalmente #

Aquí hay un ejemplo de la forma más simple (“intentar… finalmente”):

try {
    doSomething();  
} finally {
    cleanUp();
}

El comportamiento de try...finally es el siguiente:

  • Se ejecuta el código del bloque try.
  • Si no se lanzó ninguna excepción en el bloque try:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, control passes to the next statement after the try...finally.
  • Si se lanzó una excepción en el bloque de prueba:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, the original exception continues to propagate.

El código dentro del bloque finally siempre se ejecutará. (Las únicas excepciones son si se llama a System.exit(int), o si la JVM entra en pánico). Por lo tanto, un bloque finally es el código de lugar correcto que siempre debe ejecutarse; p.ej. cerrar archivos y otros recursos o liberar bloqueos.

intentar-atrapar-finalmente #

Nuestro segundo ejemplo muestra cómo catch y finally pueden usarse juntos. También ilustra que limpiar los recursos no es sencillo.

// This code snippet writes the first line of a file to a string
String result = null;
Reader reader = null;
try {
    reader = new BufferedReader(new FileReader(fileName));
    result = reader.readLine();
} catch (IOException ex) {
    Logger.getLogger.warn("Unexpected IO error", ex);  // logging the exception
} finally {
    if (reader != null) {
        try {
            reader.close();
        } catch (IOException ex) {
            // ignore / discard this exception
        }
    }
}

El conjunto completo de comportamientos (hipotéticos) de intentar... atrapar... finalmente' en este ejemplo es demasiado complicado para describirlo aquí. La versión simple es que el código en el bloque finally` siempre se ejecutará.

Mirando esto desde la perspectiva de la gestión de recursos:

  • Declaramos el “recurso” (es decir, la variable lector) antes del bloque intentar para que esté dentro del alcance del bloque finalmente.
  • Al colocar el nuevo FileReader(...), el catch puede manejar cualquier excepción IOError que se produzca al abrir el archivo.
  • Necesitamos un reader.close() en el bloque finally porque hay algunas rutas de excepción que no podemos interceptar ni en el bloque try ni en el bloque catch.
  • Sin embargo, dado que podría haberse lanzado una excepción antes de que se inicializara reader, también necesitamos una prueba null explícita.
  • Finalmente, la llamada reader.close() podría (hipotéticamente) generar una excepción. No nos importa eso, pero si no detectamos la excepción en el origen, tendríamos que tratarla más arriba en la pila de llamadas.

Java 7 y versiones posteriores proporcionan una sintaxis de prueba con recursos alternativa que simplifica significativamente la limpieza de recursos.

La cláusula ’throws’ en la declaración de un método #

El mecanismo de excepción verificada de Java requiere que el programador declare que ciertos métodos podrían lanzar excepciones verificadas especificadas. Esto se hace usando la cláusula throws. Por ejemplo:

public class OddNumberException extends Exception { // a checked exception
}

public void checkEven(int number) throws OddNumberException {
    if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

throws OddNumberException declara que una llamada a checkEven podría lanzar una excepción de tipo OddNumberException.

Una cláusula throws puede declarar una lista de tipos y puede incluir excepciones no verificadas así como excepciones verificadas.

public void checkEven(Double number) 
        throws OddNumberException, ArithmeticException {
    if (!Double.isFinite(number)) {
        throw new ArithmeticException("INF or NaN");
    } else if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

¿Cuál es el punto de declarar excepciones no verificadas como lanzadas?

La cláusula throws en la declaración de un método tiene dos propósitos:

  1. Le dice al compilador qué excepciones se lanzan para que el compilador pueda informar las excepciones no detectadas (verificadas) como errores.

  2. Le dice a un programador que está escribiendo código que llama al método qué excepciones esperar. Para este propósito, a menudo tiene sentido incluir excepciones no verificadas en una lista de “lanzamientos”.

Nota: la lista throws también es utilizada por la herramienta javadoc al generar la documentación de la API, y por las sugerencias típicas del método de “texto flotante” de un IDE.

Lanzamientos y anulación de métodos #

La cláusula throws forma parte de la firma de un método con el propósito de anular el método. Un método de anulación se puede declarar con el mismo conjunto de excepciones comprobadas que arroja el método anulado, o con un subconjunto. Sin embargo, el método de anulación no puede agregar excepciones verificadas adicionales. Por ejemplo:

@Override
public void checkEven(int number) throws NullPointerException // OK—NullPointerException is an unchecked exception
    ...

@Override
public void checkEven(Double number) throws OddNumberException // OK—identical to the superclass
    ...

class PrimeNumberException extends OddNumberException {}
class NonEvenNumberException extends OddNumberException {}

@Override
public void checkEven(int number) throws PrimeNumberException, NonEvenNumberException // OK—these are both subclasses

@Override
public void checkEven(Double number) throws IOExcepion         // ERROR

El motivo de esta regla es que si un método anulado puede generar una excepción comprobada que el método anulado no pudo generar, eso rompería la sustitución de tipos.
