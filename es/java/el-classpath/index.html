El classpath enumera los lugares donde el tiempo de ejecución de Java debe buscar clases y recursos. El classpath también es utilizado por el compilador de Java para encontrar dependencias externas y compiladas previamente.

Cargando clase Java

La JVM (Java Virtual Machine) cargará las clases cuando las clases sean requeridas (esto se denomina carga diferida). Las ubicaciones de las clases que se utilizarán se especifican en tres lugares: -

  1. Los requeridos por la plataforma Java se cargan primero, como los de la biblioteca de clases de Java y sus dependencias.
  2. Las clases de extensión se cargan a continuación (es decir, aquellas en jre/lib/ext/)
  3. A continuación, se cargan las clases definidas por el usuario a través de classpath.

Las clases se cargan usando clases que son subtipos de java.lang.ClassLoader. Esto se describe con más detalle en este tema: https://www.wikiod.com/es/java/cargadores-de-clases

Ruta de clases

El classpath es un parámetro utilizado por la JVM o el compilador que especifica las ubicaciones de las clases y los paquetes definidos por el usuario. Esto se puede configurar en la línea de comando como con la mayoría de estos ejemplos o a través de una variable ambiental (CLASSPATH)

Diferentes formas de especificar el classpath #

Hay tres formas de configurar el classpath.

  1. Se puede configurar usando la variable de entorno CLASSPATH:

     set CLASSPATH=...         # Windows and csh
 export CLASSPATH=...      # Unix ksh/bash

  2. Se puede configurar en la línea de comando de la siguiente manera

     java -classpath ...
 javac -classpath ...

Tenga en cuenta que la opción -classpath (o -cp) tiene prioridad sobre la variable de entorno CLASSPATH.

  1. El classpath para un archivo JAR ejecutable se especifica utilizando el elemento Class-Path en MANIFEST.MF:

     Class-Path: jar1-name jar2-name directory-name/jar3-name

Tenga en cuenta que esto solo se aplica cuando el archivo JAR se ejecuta de esta manera:

    java -jar some.jar ...

En este modo de ejecución, la opción -classpath y la variable de entorno CLASSPATH serán ignoradas, incluso si el archivo JAR no tiene el elemento Class-Path.

Si no se especifica una ruta de clase, entonces la ruta de clase predeterminada es el archivo JAR seleccionado cuando se usa java -jar, o el directorio actual de lo contrario.

Relacionado:

  • https://docs.oracle.com/javase/tutorial/deployment/jar/downman.html
  • http://docs.oracle.com/javase/7/docs/technotes/tools/windows/classpath.html

Agregando todos los JAR en un directorio al classpath #

Si desea agregar todos los JAR en el directorio a classpath, puede hacerlo de manera concisa usando la sintaxis de comodín de classpath; por ejemplo:

 someFolder/*

Esto le dice a la JVM que agregue todos los archivos JAR y ZIP en el directorio someFolder al classpath. Esta sintaxis se puede utilizar en un argumento -cp, una variable de entorno CLASSPATH o un atributo Class-Path en el archivo de manifiesto de un archivo JAR ejecutable. Consulte [Configuración de la ruta de clase: comodines de ruta de clase] 1 para ejemplos y advertencias.

Notas:

  1. Los comodines Classpath se introdujeron por primera vez en Java 6. Las versiones anteriores de Java no tratan “*” como un comodín.
  2. No puede poner otros caracteres antes o después del “”; p.ej. “someFolder/.jar” no es un comodín.
  3. Un comodín coincide solo con archivos con el sufijo “.jar” o “.JAR”. Los archivos ZIP se ignoran, al igual que los archivos JAR con sufijos diferentes.
  4. Un comodín coincide solo con archivos JAR en el propio directorio, no en sus subdirectorios.
  5. Cuando un grupo de archivos JAR coincide con una entrada comodín, no se especifica su orden relativo en el classpath.

Cargar un recurso desde el classpath #

Puede ser útil cargar un recurso (imagen, archivo de texto, propiedades, KeyStore, …) que está empaquetado dentro de un JAR. Para este propósito, podemos usar Class y ClassLoaders.

Supongamos que tenemos la siguiente estructura de proyecto:

program.jar
|
\-com
  \-project
    |
    |-file.txt
    \-Test.class

Y queremos acceder al contenido de file.txt desde la clase Test. Podemos hacerlo preguntando al cargador de clases:

InputStream is = Test.class.getClassLoader().getResourceAsStream("com/project/file.txt");

Al usar el cargador de clases, necesitamos especificar la ruta completamente calificada de nuestro recurso (cada paquete).

O alternativamente, podemos preguntarle directamente al objeto de la clase Test

InputStream is = Test.class.getResourceAsStream("file.txt");

Usando el objeto de clase, la ruta es relativa a la clase misma. Nuestra Test.class está en el paquete com.project, al igual que file.txt, no necesitamos especificar ninguna ruta.

Sin embargo, podemos usar rutas absolutas desde el objeto de clase, así:

 is = Test.class.getResourceAsStream("/com/project/file.txt");

Sintaxis de la ruta Classpath #

El classpath es una secuencia de entradas que son nombres de ruta de directorio, nombres de ruta de archivo JAR o ZIP, o especificaciones de comodín JAR/ZIP.

  • Para un classpath especificado en la línea de comando (por ejemplo, -classpath) o como una variable de entorno, las entradas deben separarse con caracteres ; (punto y coma) en Windows, o : (punto y coma) en otras plataformas ( Linux, UNIX, MacOSX, etc.).

  • Para el elemento Class-Path en MANIFEST.MF de un archivo JAR, use un solo espacio para separar las entradas.

A veces es necesario incrustar un espacio en una entrada de classpath

  • Cuando el classpath se especifica en la línea de comando, es simplemente una cuestión de usar las comillas de shell apropiadas. Por ejemplo:

     export CLASSPATH="/home/user/My JAR Files/foo.jar:second.jar"

(Los detalles pueden depender del shell de comandos que utilice).

  • Cuando el classpath se especifica en un archivo JAR como un archivo “MANIFEST.MF”, se debe usar la codificación URL.

      Class-Path: /home/user/My%20JAR%20Files/foo.jar second.jar

Classpath dinámico #

A veces, simplemente agregar todos los archivos JAR de una carpeta no es suficiente, por ejemplo, cuando tiene un código nativo y necesita seleccionar un subconjunto de archivos JAR. En este caso, necesitas dos métodos main(). El primero construye un cargador de clases y luego usa este cargador de clases para llamar al segundo main().

Aquí hay un ejemplo que selecciona el JAR nativo de SWT correcto para su plataforma, agrega todos los JAR de su aplicación y luego invoca el método main() real: http://stackoverflow.com/questions/2706222/create-cross-platform- aplicación-swt-java/3204032#3204032

Asignación de nombres de clases a rutas #

La cadena de herramientas estándar de Java (y las herramientas de terceros diseñadas para interactuar con ellas) tienen reglas específicas para asignar los nombres de las clases a las rutas de acceso de los archivos y otros recursos que los representan.

Los mapeos son los siguientes

  • Para las clases en el paquete predeterminado, los nombres de ruta son nombres de archivo simples.
  • Para las clases en un paquete con nombre, los componentes del nombre del paquete se asignan a directorios.
  • Para clases internas y anidadas con nombre, el componente de nombre de archivo se forma uniendo los nombres de clase con un carácter $.
  • Para clases internas anónimas, se utilizan números en lugar de nombres.

Esto se ilustra en la siguiente tabla:

Nombre de clase Nombre de ruta de origen Nombre de ruta del archivo de clase
AlgunaClase AlgunaClase.java AlgunaClase.clase
com.ejemplo.AlgunaClase com/ejemplo/AlgunaClase.java com/ejemplo/AlgunaClase.clase
AlgunaClase.Interior (en AlgunaClase.java ) AlgunaClase$Inner.class
AlgunaClase anon clases internas (en AlgunaClase.java ) AlgunaClase$1.clase, AlgunaClase$2.clase, etc

Qué significa el classpath: cómo funcionan las búsquedas #

El propósito del classpath es decirle a una JVM dónde encontrar clases y otros recursos. El significado del classpath y el proceso de búsqueda están entrelazados.

El classpath es una forma de ruta de búsqueda que especifica una secuencia de ubicaciones para buscar recursos. En un classpath estándar, estos lugares son un directorio en el sistema de archivos del host, un archivo JAR o un archivo ZIP. En cada caso, la ubicación es la raíz de un espacio de nombres que se buscará.

El procedimiento estándar para buscar una clase en el classpath es el siguiente:

  1. Asigne el nombre de clase a un nombre de ruta de archivo de clase relativo RP. La asignación de nombres de clases a nombres de archivos de clases se describe en otra parte.

  2. Para cada entrada E en el classpath:

  • Si la entrada es un directorio de sistema de archivos: - Resolve RP relative to E to give an absolute pathname AP. - Test if AP is a path for an existing file. - If yes, load the class from that file
  • Si la entrada es un archivo JAR o ZIP: - Lookup RP in the JAR / ZIP file index. - If the corresponding JAR / ZIP file entry exists, load the class from that entry.

El procedimiento para buscar un recurso en el classpath depende de si la ruta del recurso es absoluta o relativa. Para una ruta de recurso absoluta, el procedimiento es el anterior. Para una ruta de recurso relativa resuelta usando Class.getResource o Class.getResourceAsStream, la ruta del paquete de clases se antepone antes de la búsqueda.

(Tenga en cuenta que estos son los procedimientos implementados por los cargadores de clases estándar de Java. Un cargador de clases personalizado puede realizar la búsqueda de manera diferente).

El classpath de arranque #

Los cargadores de clases Java normales buscan clases primero en el classpath de arranque, antes de buscar extensiones y el classpath de la aplicación. De forma predeterminada, la ruta de clase de arranque consta del archivo “rt.jar” y algunos otros archivos JAR importantes que proporciona la instalación de JRE. Estos proporcionan todas las clases en la biblioteca de clases estándar de Java SE, junto con varias clases de implementación “internas”.

En circunstancias normales, no necesita preocuparse por esto. De forma predeterminada, los comandos como java, javac, etc., utilizarán las versiones apropiadas de las bibliotecas de tiempo de ejecución.

Muy ocasionalmente, es necesario anular el comportamiento normal del tiempo de ejecución de Java mediante el uso de una versión alternativa de una clase en las bibliotecas estándar. Por ejemplo, puede encontrarse con un error de “show stopper” en las bibliotecas de tiempo de ejecución que no puede solucionar por medios normales. En tal situación, es posible crear un archivo JAR que contenga la clase alterada y luego agregarlo al classpath de arranque que inicia la JVM.

El comando java proporciona las siguientes opciones -X para modificar el classpath de arranque:

  • -Xbootclasspath:<ruta> reemplaza la ruta de clases de arranque actual con la ruta provista.
  • -Xbootclasspath/a:<ruta> agrega la ruta proporcionada a la ruta de clase de arranque actual.
  • -Xbootclasspath/p:<ruta> antepone la ruta provista a la ruta de clases de arranque actual.

Tenga en cuenta que cuando usa las opciones de bootclasspath para reemplazar o anular una clase de Java (etcétera), técnicamente está modificando Java. * Puede haber * implicaciones de licencia si luego distribuye su código.
