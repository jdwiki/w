Colecciones concurrentes

Una colección concurrente es una colección que permite el acceso de más de un subproceso al mismo tiempo. Diferentes subprocesos normalmente pueden iterar a través del contenido de la colección y agregar o eliminar elementos. La colección es responsable de garantizar que la colección no se corrompa.

Colecciones seguras para subprocesos #

De forma predeterminada, los distintos tipos de colección no son seguros para subprocesos.

Sin embargo, es bastante fácil hacer que una colección sea segura para subprocesos.

List<String> threadSafeList = Collections.synchronizedList(new ArrayList<String>());
Set<String> threadSafeSet = Collections.synchronizedSet(new HashSet<String>());
Map<String, String> threadSafeMap = Collections.synchronizedMap(new HashMap<String, String>());

Cuando crea una colección segura para subprocesos, nunca debe acceder a ella a través de la colección original, solo a través del contenedor seguro para subprocesos.

A partir de Java 5, java.util.collections tiene varias colecciones nuevas seguras para subprocesos que no necesitan los diversos métodos Collections.synchronized.

List<String> threadSafeList = new CopyOnWriteArrayList<String>();
Set<String> threadSafeSet = new ConcurrentHashSet<String>();
Map<String, String> threadSafeMap = new ConcurrentHashMap<String, String>();

Colecciones concurrentes #

Las colecciones concurrentes son una generalización de las colecciones seguras para subprocesos, que permiten un uso más amplio en un entorno concurrente.

Si bien las colecciones seguras para subprocesos tienen la adición o eliminación segura de elementos de múltiples subprocesos, no necesariamente tienen una iteración segura en el mismo contexto (es posible que uno no pueda iterar de manera segura a través de la colección en un subproceso, mientras que otro lo modifica agregando / eliminación de elementos).

Aquí es donde se utilizan colecciones concurrentes.

Como la iteración es a menudo la implementación base de varios métodos masivos en colecciones, como addAll, removeAll, o también la copia de colección (a través de un constructor u otros medios), clasificación, … el caso de uso para colecciones concurrentes es en realidad bastante grande

Por ejemplo, Java SE 5 java.util.concurrent.CopyOnWriteArrayList es una implementación List segura y simultánea para subprocesos, su [javadoc](https://docs.oracle.com/javase/8/docs/api /java/util/concurrent/CopyOnWriteArrayList.html) estados:

El método de iterador de estilo “instantánea” utiliza una referencia al estado de la matriz en el punto en que se creó el iterador. Esta matriz nunca cambia durante la vida útil del iterador, por lo que la interferencia es imposible y se garantiza que el iterador no lanzará ConcurrentModificationException.

Por lo tanto, el siguiente código es seguro:

public class ThreadSafeAndConcurrent {

public static final List<Integer> LIST = new CopyOnWriteArrayList<>();

public static void main(String[] args) throws InterruptedException {
    Thread modifier = new Thread(new ModifierRunnable());
    Thread iterator = new Thread(new IteratorRunnable());
    modifier.start();
    iterator.start();
    modifier.join();
    iterator.join();
}

public static final class ModifierRunnable implements Runnable {
    @Override
    public void run() {
        try {
            for (int i = 0; i < 50000; i++) {
                LIST.add(i);
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

public static final class IteratorRunnable implements Runnable {
    @Override
    public void run() {
        try {
            for (int i = 0; i < 10000; i++) {
                long total = 0;
                for(Integer inList : LIST) {
                    total += inList;
                }
                System.out.println(total);
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
}

Otra colección concurrente relacionada con la iteración es ConcurrentLinkedQueue, que establece:

Los iteradores son débilmente consistentes y devuelven elementos que reflejan el estado de la cola en algún momento o desde la creación del iterador. No lanzan java.util.ConcurrentModificationException y pueden continuar simultáneamente con otras operaciones. Los elementos contenidos en la cola desde la creación del iterador se devolverán exactamente una vez.

Uno debe consultar los javadocs para ver si una colección es concurrente o no. Los atributos del iterador devueltos por el método iterator() (“fallo rápido”, “débilmente consistente”, …) es el atributo más importante a buscar.

Ejemplos seguros para subprocesos pero no simultáneos #

En el código anterior, cambiando la declaración LIST a

public static final List<Integer> LIST = Collections.synchronizedList(new ArrayList<>());

Podría (y estadísticamente lo hará en la mayoría de las arquitecturas modernas de múltiples CPU/núcleo) dar lugar a excepciones.

Las colecciones sincronizadas de los métodos de la utilidad Collections son seguras para subprocesos para la adición/eliminación de elementos, pero no para la iteración (a menos que la colección subyacente que se le pasa ya lo sea).

        if (previousValue != null)

       else

       }
