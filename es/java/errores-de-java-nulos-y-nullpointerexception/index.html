El valor null es el valor predeterminado para un valor no inicializado de un campo cuyo tipo es un tipo de referencia.

NullPointerException (o NPE) es la excepción que se lanza cuando intenta realizar una operación inapropiada en la referencia del objeto null. Tales operaciones incluyen:

  • llamar a un método de instancia en un objeto de destino null,
  • acceder a un campo de un objeto de destino nulo,
  • intentar indexar un objeto de matriz nulo o acceder a su longitud,
  • usar una referencia de objeto nulo como mutex en un bloque sincronizado,
  • emitir una referencia de objeto nulo,
  • desempaquetar una referencia de objeto null, y
  • lanzar una referencia de objeto nulo.

Las causas fundamentales más comunes de las NPE:

  • olvidarse de inicializar un campo con un tipo de referencia,
  • olvidarse de inicializar elementos de una matriz de un tipo de referencia, o
  • no probar los resultados de ciertos métodos API que se especifican como que devuelven null en ciertas circunstancias.

Los ejemplos de métodos comúnmente utilizados que devuelven null incluyen:

  • El método get(key) en la API Map devolverá un null si lo llama con una clave que no tiene un mapeo.
  • Los métodos getResource(path) y getResourceAsStream(path) en las API ClassLoader y Class devolverán null si no se puede encontrar el recurso.
  • El método get() en la API Reference devolverá null si el recolector de basura ha borrado la referencia.
  • Varios métodos getXxxx en las API de servlet de Java EE devolverán null si intenta obtener un parámetro de solicitud, una sesión o un atributo de sesión inexistentes, etc.

Existen estrategias para evitar NPE no deseados, como probar explícitamente null o usar “Notación Yoda”, pero estas estrategias a menudo tienen el resultado no deseado de ocultar problemas en su código que realmente deberían solucionarse.

Pitfall - Nulos inesperados “Haciendo buenos” #

En StackOverflow, a menudo vemos código como este en Respuestas:

public String joinStrings(String a, String b) {
    if (a == null) {
        a = "";
    }
    if (b == null) {
        b = "";
    }
    return a + ": " + b;
}

A menudo, esto va acompañado de una afirmación que es la “mejor práctica” para probar null como esta para evitar NullPointerException.

¿Es la mejor práctica? En resumen: no.

Hay algunas suposiciones subyacentes que deben cuestionarse antes de que podamos decir si es una buena idea hacer esto en nuestras joinStrings:

¿Qué significa que “a” o “b” sean nulos? #

Un valor de String puede tener cero o más caracteres, por lo que ya tenemos una forma de representar una cadena vacía. ¿null significa algo diferente a ""? Si no, entonces es problemático tener dos formas de representar una cadena vacía.

¿El valor nulo proviene de una variable no inicializada? #

Un nulo puede provenir de un campo no inicializado o de un elemento de matriz no inicializado. El valor podría no inicializarse por diseño o por accidente. Si fue por accidente, entonces esto es un error.

¿El valor nulo representa un “no sé” o un “valor faltante”?

A veces, un nulo puede tener un significado genuino; p.ej. que el valor real de una variable es desconocido o no disponible u “opcional”. En Java 8, la clase ‘Opcional’ proporciona una mejor manera de expresar eso.

Si esto es un error (o un error de diseño), ¿deberíamos “hacerlo bien”?

Una interpretación del código es que estamos “corrigiendo” un null inesperado usando una cadena vacía en su lugar. ¿Es la estrategia correcta? ¿Sería mejor dejar que ocurra la NullPointerException y luego capturar la excepción más arriba en la pila y registrarla como un error?

El problema de “hacer el bien” es que puede ocultar el problema o dificultar su diagnóstico.

¿Es esto eficiente/bueno para la calidad del código? #

Si el enfoque de “hacer el bien” se usa consistentemente, su código contendrá muchas pruebas nulas “defensivas”. Esto hará que sea más largo y más difícil de leer. Además, todas estas pruebas y “correcciones” pueden afectar el rendimiento de su aplicación.

En resumen #

Si null es un valor significativo, entonces probar el caso null es el enfoque correcto. El corolario es que si un valor nulo es significativo, entonces esto debería estar claramente documentado en los javadocs de cualquier método que acepte el valor nulo o lo devuelva.

De lo contrario, es una mejor idea tratar un null inesperado como un error de programación, y dejar que ocurra la NullPointerException para que el desarrollador sepa que hay un problema en el código.

Trampa: usar nulo para representar una matriz o colección vacía #

Algunos programadores piensan que es una buena idea ahorrar espacio utilizando un null para representar una matriz o colección vacía. Si bien es cierto que puede ahorrar una pequeña cantidad de espacio, la otra cara es que hace que su código sea más complicado y más frágil. Compare estas dos versiones de un método para sumar una matriz:

La primera versión es cómo codificaría normalmente el método:

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed
 * @return the sum
 **/
public int sum(int[] values) {
    int sum = 0;
    for (int value : values) {
        sum += value;
    }
    return sum;
}

La segunda versión es cómo necesita codificar el método si tiene la costumbre de usar null para representar una matriz vacía.

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed, or null.
 * @return the sum, or zero if the array is null.
 **/
public int sum(int[] values) {
    int sum = 0;
    if (values != null) {
        for (int value : values) {
            sum += value;
        }
    }
    return sum;
}

Como puede ver, el código es un poco más complicado. Esto es directamente atribuible a la decisión de usar null de esta manera.

Ahora considere si esta matriz que podría ser un nulo se usa en muchos lugares. En cada lugar donde lo use, debe considerar si necesita probar null. Si pierde una prueba null que debe estar allí, corre el riesgo de una NullPointerException. Por lo tanto, la estrategia de usar null de esta manera hace que su aplicación sea más frágil; es decir, más vulnerable a las consecuencias de los errores del programador.

La lección aquí es usar arreglos vacíos y listas vacías cuando eso es lo que quieres decir.

int[] values = new int[0];                     // always empty
List<Integer> list = new ArrayList();          // initially empty
List<Integer> list = Collections.emptyList();  // always empty

El espacio de sobrecarga es pequeño y hay otras formas de minimizarlo si vale la pena hacerlo.

Pitfall: devolver un valor nulo en lugar de lanzar una excepción #

Algunos programadores de Java tienen una aversión general a lanzar o propagar excepciones. Esto conduce a un código como el siguiente:

public Reader getReader(String pathname) {
    try {
        return new BufferedReader(FileReader(pathname));
    } catch (IOException ex) {
        System.out.println("Open failed: " + ex.getMessage());
        return null;
    }

}

Entonces, ¿cuál es el problema con eso?

El problema es que getReader devuelve un null como un valor especial para indicar que Reader no se pudo abrir. Ahora, el valor devuelto debe probarse para ver si es “nulo” antes de usarlo. Si se omite la prueba, el resultado será una NullPointerException.

En realidad, hay tres problemas aquí:

  1. La IOException se detectó demasiado pronto.
  2. La estructura de este código significa que existe el riesgo de filtrar un recurso.
  3. Se usó un null y luego se devolvió porque no había ningún Reader “real” disponible para devolver.

De hecho, asumiendo que la excepción necesitaba ser capturada temprano de esta manera, había un par de alternativas para devolver null:

  1. Sería posible implementar una clase NullReader; p.ej. uno donde las operaciones de la API se comportan como si el lector ya estuviera en la posición de “fin de archivo”.
  2. Con Java 8, sería posible declarar getReader como un Optional<Reader>.

Dificultad: no comprobar si un flujo de E/S ni siquiera se ha inicializado al cerrarlo #

Para evitar pérdidas de memoria, no se debe olvidar cerrar un flujo de entrada o un flujo de salida cuyo trabajo haya terminado. Esto generalmente se hace con una instrucción try-catch-finally sin la parte catch:

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        out.close();
    }
}

Si bien el código anterior puede parecer inocente, tiene una falla que puede hacer que la depuración sea imposible. Si la línea donde se inicializa out (out = new FileOutputStream(nombre de archivo)) arroja una excepción, entonces out será null cuando se ejecute out.close(), lo que resultará en una desagradable excepción NullPointerException !

Para evitar esto, simplemente asegúrese de que la secuencia no sea nula antes de intentar cerrarla.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        if (out != null)
            out.close();
    }
}

Un enfoque aún mejor es “probar” con recursos, ya que cerrará automáticamente la transmisión con una probabilidad de 0 para lanzar un NPE sin la necesidad de un bloque “finalmente”.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    try (FileOutputStream out = new FileOutputStream(filename)) {
        for(; count > 0; count--)
            out.write(0);
    }
}

Dificultad: el uso innecesario de contenedores primitivos puede dar lugar a NullPointerExceptions #

A veces, los programadores que son nuevos en Java usarán tipos primitivos y contenedores indistintamente. Esto puede conducir a problemas. Considere este ejemplo:

public class MyRecord {
    public int a, b;
    public Integer c, d;
}

...
MyRecord record = new MyRecord();
record.a = 1;               // OK
record.b = record.b + 1;    // OK
record.c = 1;               // OK
record.d = record.d + 1;    // throws a NullPointerException

Nuestra clase MyRecord1 se basa en la inicialización predeterminada para inicializar los valores en sus campos. Por lo tanto, cuando ’nuevamos’ un registro, los campos ‘a’ y ‘b’ se establecerán en cero, y los campos ‘c’ y ’d’ se establecerán en ’nulo’.

Cuando intentamos usar los campos inicializados predeterminados, vemos que los campos int funcionan todo el tiempo, pero los campos Integer funcionan en algunos casos y en otros no. Específicamente, en el caso de que falle (con d), lo que sucede es que la expresión en el lado derecho intenta desempaquetar una referencia null, y eso es lo que hace que se lance NullPointerException.

Hay un par de maneras de ver esto:

  • Si los campos c y d necesitan ser envoltorios primitivos, entonces no deberíamos confiar en la inicialización predeterminada, o deberíamos probar si hay null. Para el primero es el enfoque correcto a menos que haya un significado definido para los campos en el estado nulo.

  • Si los campos no necesitan ser envoltorios primitivos, entonces es un error convertirlos en envoltorios primitivos. Además de este problema, los contenedores primitivos tienen gastos generales adicionales en relación con los tipos primitivos.

La lección aquí es no usar tipos de envoltorios primitivos a menos que realmente lo necesite.

1: esta clase no es un ejemplo de buenas prácticas de codificación. Por ejemplo, una clase bien diseñada no tendría campos públicos. Sin embargo, ese no es el objetivo de este ejemplo.

Trampa: uso de la “notación Yoda” para evitar la excepción NullPointerException #

Una gran cantidad de código de ejemplo publicado en StackOverflow incluye fragmentos como este:

if ("A".equals(someString)) {
    // do something
}

Esto “previene” o “evita” una posible NullPointerException en el caso de que someString sea null. Además, es discutible que

    "A".equals(someString)

es mejor que:

    someString != null && someString.equals("A")

(Es más conciso y, en algunas circunstancias, podría ser más eficiente. Sin embargo, como argumentamos a continuación, la concisión podría ser negativa).

Sin embargo, el verdadero escollo es usar la prueba de Yoda para evitar NullPointerExceptions como una cuestión de hábito.

Cuando escribe "A".equals(someString), en realidad está “corrigiendo” el caso en el que someString resulta ser null. Pero como otro ejemplo (https://www.wikiod.com/es/java/errores-de-java-nulos-y-nullpointerexception#Trampa - “Hacer bien” nulos inesperados ) explica, “hacer bien” null valores pueden ser dañinos por una variedad de razones.

Esto significa que las condiciones de Yoda no son “mejores prácticas”1. A menos que se espere el null, es mejor dejar que ocurra la NullPointerException para que pueda obtener una falla en la prueba de la unidad (o un informe de error). Eso le permite encontrar y corregir el error que causó que apareciera el null inesperado / no deseado.

Las condiciones de Yoda solo se deben usar en casos en los que se espera el nulo porque el objeto que está probando proviene de una API que está documentada como que devuelve un nulo. Y podría decirse que podría ser mejor usar una de las formas menos bonitas de expresar la prueba porque eso ayuda a resaltar la prueba nula para alguien que está revisando su código.

1 - Según Wikipedia: “Las mejores prácticas de codificación son un conjunto de reglas informales que la comunidad de desarrollo de software ha aprendido con el tiempo y que pueden ayudar a mejorar la calidad del software”.. El uso de la notación de Yoda no logra esto. En muchas situaciones, empeora el código.
