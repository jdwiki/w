Varios usos indebidos del lenguaje de programación Java pueden hacer que un programa genere resultados incorrectos a pesar de estar compilado correctamente. El objetivo principal de este tema es enumerar los errores comunes relacionados con el manejo de excepciones y proponer la forma correcta de evitar tales errores.

Pitfall - Captura de Throwable, Exception, Error o RuntimeException #

Un patrón de pensamiento común para los programadores de Java sin experiencia es que las excepciones son “un problema” o “una carga” y la mejor manera de lidiar con esto es capturarlas todas1 tan pronto como sea posible. Esto lleva a un código como este:

....
try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (Exception ex) {
    System.out.println("Could not open file " + fileName);
}

El código anterior tiene un defecto significativo. El catch en realidad va a atrapar más excepciones de las que espera el programador. Supongamos que el valor de fileName es null, debido a un error en otra parte de la aplicación. Esto hará que el constructor FileInputStream lance una NullPointerException. El controlador detectará esto e informará al usuario:

    Could not open file null

lo cual es inútil y confuso. Peor aún, suponga que fue el código “procesar la entrada” el que arrojó la excepción inesperada (¡marcada o sin marcar!). Ahora el usuario recibirá el mensaje engañoso de un problema que no ocurrió al abrir el archivo y que puede no estar relacionado con E/S en absoluto.

La raíz del problema es que el programador ha codificado un controlador para Exception. Esto es casi siempre un error:

  • Capturar Exception capturará todas las excepciones marcadas, y también la mayoría de las excepciones no marcadas.
  • La captura de RuntimeException detectará la mayoría de las excepciones no verificadas.
  • La captura de Error detectará las excepciones no verificadas que señalan errores internos de JVM. Estos errores generalmente no son recuperables y no deben detectarse.
  • Capturar Throwable capturará todas las excepciones posibles.

El problema de capturar un conjunto demasiado amplio de excepciones es que el controlador normalmente no puede manejarlas todas de manera adecuada. En el caso de la Excepción y así sucesivamente, es difícil para el programador predecir qué podría ser capturado; es decir, qué esperar.

En general, la solución correcta es tratar con las excepciones que son lanzadas. Por ejemplo, puede atraparlos y manejarlos in situ:

try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (FileNotFoundException ex) {
    System.out.println("Could not open file " + fileName);
}

o puede declararlos como lanzados por el método adjunto.

Hay muy pocas situaciones en las que es apropiado capturar Exception. El único que surge comúnmente es algo como esto:

public static void main(String[] args) {
    try {
        // do stuff
    } catch (Exception ex) {
        System.err.println("Unfortunately an error has occurred. " +
                           "Please report this to X Y Z");
        // Write stacktrace to a log file.
        System.exit(1);
    }
}

Aquí realmente queremos tratar con todas las excepciones, por lo que capturar Exception (o incluso Throwable) es correcto.

1: también conocido como Gestión de excepciones de Pokémon.

Trampa - Ignorar o aplastar excepciones #

Este ejemplo se trata de ignorar deliberadamente o “aplastar” las excepciones. O, para ser más precisos, se trata de cómo capturar y manejar una excepción de manera que la ignore. Sin embargo, antes de describir cómo hacer esto, primero debemos señalar que aplastar las excepciones generalmente no es la forma correcta de tratarlas.

Las excepciones generalmente se lanzan (por algo) para notificar a otras partes del programa que ha ocurrido algún evento significativo (es decir, “excepcional”). Generalmente (aunque no siempre) una excepción significa que algo salió mal. Si codifica su programa para aplastar la excepción, existe una buena posibilidad de que el problema reaparezca en otra forma. Para empeorar las cosas, cuando aplasta la excepción, está desechando la información del objeto de excepción y su seguimiento de pila asociado. Es probable que eso haga que sea más difícil averiguar cuál fue la fuente original del problema.

En la práctica, la eliminación de excepciones ocurre con frecuencia cuando utiliza la función de corrección automática de un IDE para “arreglar” un error de compilación causado por una excepción no controlada. Por ejemplo, es posible que vea un código como este:

try {
    inputStream = new FileInputStream("someFile");
} catch (IOException e) {
    /* add exception handling code here */
}

Claramente, el programador ha aceptado la sugerencia del IDE de hacer desaparecer el error de compilación, pero la sugerencia fue inapropiada. (Si la apertura del archivo ha fallado, lo más probable es que el programa haga algo al respecto. Con la “corrección” anterior, es probable que el programa falle más adelante; por ejemplo, con una NullPointerException porque inputStream ahora es null).

Habiendo dicho eso, aquí hay un ejemplo de aplastamiento deliberado de una excepción. (Para los propósitos del argumento, suponga que hemos determinado que una interrupción mientras se muestra la selfie es inofensiva). El comentario le dice al lector que aplastamos la excepción deliberadamente y por qué lo hicimos.

try {
    selfie.show();
} catch (InterruptedException e) {
    // It doesn't matter if showing the selfie is interrupted.
}

Otra forma convencional de resaltar que estamos deliberadamente eliminando una excepción sin decir por qué es indicarlo con el nombre de la variable de excepción, así:

try { 
    selfie.show(); 
} catch (InterruptedException ignored) {  }

Algunos IDE (como IntelliJ IDEA) no mostrarán una advertencia sobre el bloque catch vacío si el nombre de la variable se establece en “ignorado”.

Pitfall - Lanzar Throwable, Exception, Error o RuntimeException #

Si bien capturar las excepciones Throwable, Exception, Error y RuntimeException es malo, lanzarlas es aún peor.

El problema básico es que cuando su aplicación necesita manejar excepciones, la presencia de las excepciones de nivel superior dificulta la discriminación entre diferentes condiciones de error. Por ejemplo

try {
    InputStream is = new FileInputStream(someFile);  // could throw IOException
    ...
    if (somethingBad) {
        throw new Exception();  // WRONG
    }
} catch (IOException ex) {
    System.err.println("cannot open ...");
} catch (Exception ex) {
    System.err.println("something bad happened");  // WRONG
}

El problema es que debido a que lanzamos una instancia de Exception, nos vemos obligados a atraparla. Sin embargo, como se describe en otro ejemplo, la captura de Exception es mala. En esta situación, se vuelve difícil discriminar entre el caso “esperado” de una “Excepción” que se lanza si “algo malo” es “verdadero” y el caso inesperado en el que detectamos una excepción no verificada como “NullPointerException”.

Si se permite que la excepción de nivel superior se propague, nos encontramos con otros problemas:

  • Ahora tenemos que recordar todas las diferentes razones por las que arrojamos el nivel superior y discriminarlas / manejarlas.
  • En el caso de Exception y Throwable también necesitamos agregar estas excepciones a la cláusula throws de los métodos si queremos que la excepción se propague. Esto es problemático, como se describe a continuación.

En resumen, no lance estas excepciones. Ejecute una excepción más específica que describa mejor el “evento excepcional” que ha ocurrido. Si lo necesita, defina y use una clase de excepción personalizada.

Declarar Throwable o Exception en los “lanzamientos” de un método es problemático.

Es tentador reemplazar una larga lista de excepciones lanzadas en la cláusula throws de un método con Exception o incluso `Throwable. Esta es una mala idea:

  1. Obliga a la persona que llama a manejar (o propagar) Exception.
  2. Ya no podemos confiar en que el compilador nos informe sobre las excepciones comprobadas específicas que deben manejarse.
  3. Es difícil manejar adecuadamente la Excepción. Es difícil saber qué excepciones reales pueden detectarse y, si no sabe qué podría detectarse, es difícil saber qué estrategia de recuperación es adecuada.
  4. Manejar ‘Throwable’ es aún más difícil, ya que ahora también tiene que hacer frente a posibles fallas de las que nunca debería recuperarse.

Este consejo significa que se deben evitar ciertos otros patrones. Por ejemplo:

try {
    doSomething();
} catch (Exception ex) {
    report(ex);
    throw ex;
}

Lo anterior intenta registrar todas las excepciones a medida que pasan, sin manejarlas definitivamente. Desafortunadamente, antes de Java 7, la declaración throw ex; hacía que el compilador pensara que se podía lanzar cualquier Exception. Eso podría forzarte a declarar el método adjunto como throws Exception. Desde Java 7 en adelante, el compilador sabe que el conjunto de excepciones que se pueden (volver a lanzar) allí es más pequeño.

Dificultad: uso de excepciones para el control de flujo normal #

Hay un mantra que algunos expertos en Java suelen recitar:

“Las excepciones solo deben usarse para casos excepcionales”.

(Por ejemplo: http://programmers.stackexchange.com/questions/184654)

La esencia de esto es que es una mala idea (en Java) usar excepciones y manejo de excepciones para implementar un control de flujo normal. Por ejemplo, compare estas dos formas de tratar con un parámetro que podría ser nulo.

public String truncateWordOrNull(String word, int maxLength) {
    if (word == null) {
        return "";
    } else {
        return word.substring(0, Math.min(word.length(), maxLength));
    }
}

public String truncateWordOrNull(String word, int maxLength) {
    try {
        return word.substring(0, Math.min(word.length(), maxLength));
    } catch (NullPointerException ex) {
        return "";
    }
}

En este ejemplo, estamos (por diseño) tratando el caso donde palabra es null como si fuera una palabra vacía. Las dos versiones tratan con null ya sea utilizando if … else convencional y o try … catch. ¿Cómo debemos decidir qué versión es mejor?

El primer criterio es la legibilidad. Si bien la legibilidad es difícil de cuantificar objetivamente, la mayoría de los programadores estarían de acuerdo en que el significado esencial de la primera versión es más fácil de discernir. De hecho, para comprender realmente la segunda forma, debe comprender que los métodos Math.min o String.substring no pueden generar una NullPointerException.

El segundo criterio es la eficiencia. En versiones de Java anteriores a Java 8, la segunda versión es significativamente (en órdenes de magnitud) más lenta que la primera versión. En particular, la construcción de un objeto de excepción implica capturar y registrar los marcos de pila, en caso de que se requiera el seguimiento de pila.

Por otro lado, hay muchas situaciones en las que usar excepciones es más legible, más eficiente y (a veces) más correcto que usar código condicional para lidiar con eventos “excepcionales”. De hecho, existen raras situaciones en las que es necesario utilizarlos para eventos “no excepcionales”; es decir, eventos que ocurren con relativa frecuencia. Para este último, vale la pena buscar formas de reducir los gastos generales de crear objetos de excepción.

Trampa - Atrapar excepción interrumpida #

Como ya se señaló en otras trampas, capturar todas las excepciones usando

try {
    // Some code
} catch (Exception) {
    // Some error handling
}

Viene con muchos problemas diferentes. Pero un problema particular es que puede provocar interbloqueos, ya que rompe el sistema de interrupción al escribir aplicaciones de subprocesos múltiples.

Si inicia un hilo, generalmente también necesita poder detenerlo abruptamente por varias razones.

Thread t = new Thread(new Runnable() {
    public void run() {
         while (true) {
             //Do something indefinetely
         }
    }
}

t.start();

//Do something else

// The thread should be canceld if it is still active. 
// A Better way to solve this is with a shared variable that is tested 
// regularily by the thread for a clean exit, but for this example we try to 
// forcibly interrupt this thread.
if (t.isAlive()) {
   t.interrupt();
   t.join();
}

//Continue with program

t.interrupt() generará una InterruptedException en ese hilo, que tiene la intención de cerrar el hilo. Pero, ¿qué pasa si el subproceso necesita limpiar algunos recursos antes de que se detenga por completo? Para esto, puede capturar la InterruptedException y hacer algo de limpieza.

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                //Do something indefinetely
            }
        } catch (InterruptedException ex) {
            //Do some quick cleanup

            // In this case a simple return would do. 
            // But if you are not 100% sure that the thread ends after 
            // catching the InterruptedException you will need to raise another 
            // one for the layers surrounding this code.                
            Thread.currentThread().interrupt(); 
        }
    }
}

Pero si tiene una expresión general en su código, la InterruptedException también será capturada y la interrupción no continuará. Lo que en este caso podría conducir a un interbloqueo ya que el subproceso principal espera indefinidamente a que este encabezado se detenga con t.join().

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                }
                catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Dead code as the interrupt exception was already caught in
            // the inner try-catch           
            Thread.currentThread().interrupt(); 
        }
    }
}

Por lo tanto, es mejor capturar las Excepciones individualmente, pero si insiste en usar un catch-all, al menos capture la InterruptedException individualmente de antemano.

Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                } catch (InterruptedException ex) {
                    throw ex; //Send it up in the chain
                } catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Some quick cleanup code 
    
            Thread.currentThread().interrupt(); 
        }
    }
}

Pitfall - Stacktraces excesivos o inapropiados #

Una de las cosas más molestas que pueden hacer los programadores es dispersar las llamadas a printStackTrace() a lo largo de su código.

El problema es que printStackTrace() va a escribir el seguimiento de pila en la salida estándar.

  • Para una aplicación que está destinada a usuarios finales que no son programadores de Java, un seguimiento de pila es poco informativo en el mejor de los casos y alarmante en el peor.

  • Para una aplicación del lado del servidor, lo más probable es que nadie mire la salida estándar.

Una mejor idea es no llamar a printStackTrace directamente, o si lo llama, hágalo de manera que el seguimiento de la pila se escriba en un archivo de registro o archivo de error en lugar de en la consola del usuario final.

Una forma de hacer esto es usar un marco de registro y pasar el objeto de excepción como un parámetro del evento de registro. Sin embargo, incluso registrar la excepción puede ser dañino si se hace de manera imprudente. Considera lo siguiente:

public void method1() throws SomeException {
    try {
        method2();
        // Do something
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method1", ex);
        throw ex;
    }
}

public void method2() throws SomeException {
    try {
        // Do something else
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method2", ex);
        throw ex;
    }
}

Si se lanza la excepción en method2, es probable que vea dos copias del mismo stacktrace en el archivo de registro, correspondientes a la misma falla.

En resumen, registre la excepción o vuelva a lanzarla (posiblemente envuelta con otra excepción). No hagas ambas cosas.

Pitfall - Subclasificación directa de Throwable #

Throwable tiene dos subclases directas, Exception y Error. Si bien es posible crear una nueva clase que amplíe Throwable directamente, esto no es recomendable ya que muchas aplicaciones asumen que solo existen Exception y Error.

Más concretamente, no hay ningún beneficio práctico en la subclasificación directa de Throwable, ya que la clase resultante es, en efecto, simplemente una excepción comprobada. La subclasificación de Exception en su lugar dará como resultado el mismo comportamiento, pero transmitirá más claramente su intención.
