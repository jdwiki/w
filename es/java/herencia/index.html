La herencia es una función básica orientada a objetos en la que una clase adquiere y amplía las propiedades de otra clase, utilizando la palabra clave “extiende”. Para interfaces y la palabra clave implementos, consulte interfaces.

Sintaxis #

  • clase ClassB extiende ClassA {…}
  • clase ClassB implementa InterfaceA {…}
  • la interfaz InterfazB extiende la InterfazA {…}
  • clase ClassB extiende ClassA implementa InterfaceC, InterfaceD {…}
  • clase abstracta AbstractClassB extiende ClassA {…}
  • clase abstracta AbstractClassB extiende AbstractClassA {…}
  • clase abstracta AbstractClassB extiende ClassA implementa InterfaceC, InterfaceD {…}

La herencia a menudo se combina con genéricos para que la clase base tenga uno o más parámetros de tipo. Consulte Crear una clase genérica.

1: https://www.wikiod.com/es/java/genericos#Crear una clase genérica

Clases abstractas #

Una clase abstracta es una clase marcada con la palabra clave abstract. Al contrario de la clase no abstracta, puede contener métodos abstractos, sin implementación. Sin embargo, es válido crear una clase abstracta sin métodos abstractos.

Una clase abstracta no puede ser instanciada. Se puede subclasificar (extender) siempre que la subclase también sea abstracta o implemente todos los métodos marcados como abstractos por las superclases.

Un ejemplo de una clase abstracta:

public abstract class Component {
    private int x, y;
    
    public setPosition(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public abstract void render();
}

La clase debe marcarse como abstracta, cuando tiene al menos un método abstracto. Un método abstracto es un método que no tiene implementación. Se pueden declarar otros métodos dentro de una clase abstracta que tenga implementación para proporcionar un código común para cualquier subclase.

Intentar instanciar esta clase proporcionará un error de compilación:

//error: Component is abstract; cannot be instantiated   
Component myComponent = new Component();

Sin embargo, una clase que extiende Componente y proporciona una implementación para todos sus métodos abstractos y puede ser instanciada.

public class Button extends Component {

    @Override
    public void render() {
        //render a button
    }
}

public class TextBox extends Component {

    @Override
    public void render() {
        //render a textbox
    }
}

Las instancias de clases heredadas también se pueden convertir como la clase principal (herencia normal) y proporcionan un efecto polimórfico cuando se llama al método abstracto.

Component myButton = new Button();
Component myTextBox = new TextBox();

myButton.render(); //renders a button
myTextBox.render(); //renders a text box

Clases abstractas vs Interfaces

Las clases abstractas y las interfaces proporcionan una forma de definir las firmas de los métodos y requieren que la clase de extensión/implementación proporcione la implementación.

Hay dos diferencias clave entre las clases abstractas y las interfaces:

  • Una clase solo puede extender una sola clase, pero puede implementar muchas interfaces.
  • Una clase abstracta puede contener campos de instancia (no “estáticos”), pero las interfaces solo pueden contener campos “estáticos”.

Los métodos declarados en las interfaces no podían contener implementaciones, por lo que se usaban clases abstractas cuando era útil proporcionar métodos adicionales cuyas implementaciones llamaban métodos abstractos.

Java 8 permite que las interfaces contengan [métodos predeterminados] (https://www.wikiod.com/es/java/metodos-predeterminados), generalmente [implementados usando los otros métodos de la interfaz] (https:// stackoverflow.com/documentation/java/113/default-methods/456/accessing-other-interface-methods-within-default-method#t=201607191734172336331), lo que hace que las interfaces y las clases abstractas sean igualmente poderosas en este sentido.

Subclases anónimas de clases abstractas

Como conveniencia, Java permite la instanciación de instancias anónimas de subclases de clases abstractas, que proporcionan implementaciones para los métodos abstractos al crear el nuevo objeto. Usando el ejemplo anterior, esto podría verse así:

Component myAnonymousComponent = new Component() {
    @Override
    public void render() {
        // render a quick 1-time use component
    }
}

Usar ‘final’ para restringir la herencia y anular #

clases finales #

Cuando se usa en una declaración de clase, el modificador final evita que se declaren otras clases que extiendan la clase. Una clase final es una clase “hoja” en la jerarquía de clases de herencia.

// This declares a final class
final class MyFinalClass {
    /* some code */
}

// Compilation error: cannot inherit from final MyFinalClass
class MySubClass extends MyFinalClass {
    /* more code */
}

Casos de uso para clases finales #

Las clases finales se pueden combinar con un constructor privado para controlar o evitar la instanciación de una clase. Esto se puede usar para crear una llamada “clase de utilidad” que solo define miembros estáticos; es decir, constantes y métodos estáticos.

public final class UtilityClass {

    // Private constructor to replace the default visible constructor
    private UtilityClass() {}

    // Static members can still be used as usual
    public static int doSomethingCool() {
        return 123;
    }

}

Las clases inmutables también deben declararse como final. (Una clase inmutable es aquella cuyas instancias no se pueden cambiar después de haber sido creadas; consulte el tema Iobjetos inmutables). Al hacer esto, hace que sea imposible crear una subclase mutable de una clase inmutable. Eso violaría el Principio de Sustitución de Liskov que requiere que un subtipo obedezca el “contrato de comportamiento” de sus supertipos.

Desde una perspectiva práctica, declarar una clase inmutable como “final” facilita el razonamiento sobre el comportamiento del programa. También aborda los problemas de seguridad en el escenario en el que se ejecuta código que no es de confianza en un entorno limitado de seguridad. (Por ejemplo, dado que String se declara como final, una clase confiable no necesita preocuparse de que pueda ser engañada para aceptar una subclase mutable, que la persona que llama no confiable podría cambiar subrepticiamente).

Una desventaja de las clases finales es que no funcionan con algunos marcos de simulación como Mockito. Actualización: la versión 2 de Mockito ahora admite la simulación de clases finales.

Métodos finales #

El modificador final también se puede aplicar a los métodos para evitar que se anulen en las subclases:

public class MyClassWithFinalMethod {

    public final void someMethod() {
    }
}

public class MySubClass extends MyClassWithFinalMethod {

    @Override
    public void someMethod() { // Compiler error (overridden method is final)
    }
}

Los métodos finales generalmente se usan cuando desea restringir lo que una subclase puede cambiar en una clase sin prohibir las subclases por completo.

El modificador final también se puede aplicar a las variables, pero el significado de final para las variables no está relacionado con la herencia.

Herencia #

Con el uso de la palabra clave extends entre las clases, todas las propiedades de la superclase (también conocida como Clase principal o Clase base) están presentes en la subclase (también conocida como Clase secundaria o Clase derivada) Clase)

public class BaseClass {

    public void baseMethod(){
        System.out.println("Doing base class stuff");
    }
}

public class SubClass extends BaseClass {

}

Las instancias de SubClass han heredado el método baseMethod():

SubClass s = new SubClass();
s.baseMethod();  //Valid, prints "Doing base class stuff"

Se puede agregar contenido adicional a una subclase. Hacerlo permite una funcionalidad adicional en la subclase sin ningún cambio en la clase base o cualquier otra subclase de esa misma clase base:

public class Subclass2 extends BaseClass {

    public void anotherMethod() {
        System.out.println("Doing subclass2 stuff");
    }
}

Subclass2 s2 = new Subclass2();
s2.baseMethod(); //Still valid , prints "Doing base class stuff"
s2.anotherMethod(); //Also valid, prints "Doing subclass2 stuff"

Los campos también se heredan:

public class BaseClassWithField {

    public int x;

}

public class SubClassWithField extends BaseClassWithField {

    public SubClassWithField(int x) {
        this.x = x; //Can access fields
    }
}

Los campos y métodos privados aún existen dentro de la subclase, pero no son accesibles:

public class BaseClassWithPrivateField {

    private int x = 5;

    public int getX() {
        return x;
    }
}

public class SubClassInheritsPrivateField extends BaseClassWithPrivateField {

    public void printX() {
        System.out.println(x); //Illegal, can't access private field x
        System.out.println(getX()); //Legal, prints 5
    }
}

SubClassInheritsPrivateField s = new SubClassInheritsPrivateField();
int x = s.getX(); //x will have a value of 5.

En Java, cada clase puede extenderse como máximo a otra clase.

public class A{}
public class B{}
public class ExtendsTwoClasses extends A, B {} //Illegal

Esto se conoce como herencia múltiple y, aunque es legal en algunos lenguajes, Java no lo permite con clases.

Como resultado de esto, cada clase tiene una cadena ancestral de clases que no se ramifica y conduce a ‘Objeto’, del cual descienden todas las clases.

El principio de sustitución de Liskov #

La sustituibilidad es un principio en la programación orientada a objetos introducido por Barbara Liskov en una conferencia de 1987 que establece que, si la clase B es una subclase de la clase A, entonces dondequiera que A` Se espera ``, se puede usar B``` en su lugar:

class A {...}
class B extends A {...}

public void method(A obj) {...}

A a = new B(); // Assignment OK
method(new B()); // Passing as parameter OK

Esto también se aplica cuando el tipo es una interfaz, donde no es necesaria ninguna relación jerárquica entre los objetos:

interface Foo {
    void bar();
}

class A implements Foo {
    void bar() {...}
}

class B implements Foo {
    void bar() {...}
}

List<Foo> foos = new ArrayList<>();
foos.add(new A()); // OK
foos.add(new B()); // OK

Ahora la lista contiene objetos que no pertenecen a la misma jerarquía de clases.

Herencia estática #

El método estático se puede heredar de manera similar a los métodos normales, sin embargo, a diferencia de los métodos normales, es imposible crear métodos “abstract” para forzar la anulación del método estático. Escribir un método con la misma firma que un método estático en una superclase parece ser una forma de invalidación, pero en realidad esto simplemente crea una nueva función que oculta la otra.

public class BaseClass {
    
    public static int num = 5;

    public static void sayHello() {
        System.out.println("Hello");
    }

    public static void main(String[] args) {
        BaseClass.sayHello();
        System.out.println("BaseClass's num: " + BaseClass.num);
            
        SubClass.sayHello();
        //This will be different than the above statement's output, since it runs
        //A different method
        SubClass.sayHello(true);
        
        StaticOverride.sayHello();
        System.out.println("StaticOverride's num: " + StaticOverride.num);
    }
}

public  class SubClass extends BaseClass {
    
    //Inherits the sayHello function, but does not override it   
    public static void sayHello(boolean test) {
        System.out.println("Hey");
    }
}

public static class StaticOverride extends BaseClass {

    //Hides the num field from BaseClass
    //You can even change the type, since this doesn't affect the signature
    public static String num = "test";
        
    //Cannot use @Override annotation, since this is static
    //This overrides the sayHello method from BaseClass
    public static void sayHello() {
        System.out.println("Static says Hi");
    }

}

Ejecutar cualquiera de estas clases produce el resultado:

Hello
BaseClass's num: 5
Hello
Hey
Static says Hi
StaticOverride's num: test

Tenga en cuenta que, a diferencia de la herencia normal, en la herencia estática los métodos no están ocultos. Siempre puedes llamar al método base sayHello usando BaseClass.sayHello(). Pero las clases heredan métodos estáticos si no se encuentran métodos con la misma firma en la subclase. Si las firmas de dos métodos varían, ambos métodos se pueden ejecutar desde la subclase, incluso si el nombre es el mismo.

Los campos estáticos se ocultan entre sí de forma similar.

Clase abstracta y uso de la interfaz: relación “Is-a” frente a capacidad “Has-a” #

Cuándo usar clases abstractas: para implementar el mismo comportamiento o uno diferente entre múltiples objetos relacionados

Cuándo usar interfaces: para implementar un contrato por múltiples objetos no relacionados

Las clases abstractas crean relaciones “es un”, mientras que las interfaces proporcionan la capacidad “tiene un”.

Esto se puede ver en el siguiente código:

  4. ‘Gato’ no es un ‘Hombre’ pero puede ‘Trepar’.

  5. ‘Perro’ no es un ‘Hombre’ pero puede ‘Aprender’

  6. ‘Hombre’ no es ni ‘Gato’ ni ‘Perro’, pero puede tener algunas de las capacidades de los dos últimos sin extender ‘Animal’, ‘Gato’ o ‘Perro’. Esto se hace con Interfaces.

  7. Aunque Animal es una clase abstracta, tiene un constructor, a diferencia de una interfaz.

TL;RD:

Las clases no relacionadas pueden tener capacidades a través de las interfaces, pero las clases relacionadas cambian el comportamiento a través de la extensión de las clases base.

Consulte la [página] 1 de la documentación de Java para comprender cuál usar en un caso de uso específico.

Considere usar clases abstractas si…

  1. Desea compartir código entre varias clases estrechamente relacionadas.
  2. Espera que las clases que amplían su clase abstracta tengan muchos métodos o campos comunes, o requieran modificadores de acceso que no sean públicos (como protegido y privado).
  3. Quiere declarar campos no estáticos o no finales.

Considere usar interfaces si…

  1. Espera que las clases no relacionadas implementen su interfaz. Por ejemplo, muchos objetos no relacionados pueden implementar la interfaz Serializable.
  2. Desea especificar el comportamiento de un tipo de datos en particular, pero no le preocupa quién implementa su comportamiento.
  3. Desea aprovechar la herencia múltiple de tipos.

Herencia y métodos estáticos #

En Java, las clases principal y secundaria pueden tener métodos estáticos con el mismo nombre. Pero en tales casos, la implementación del método estático en el niño es [ocultar] (https://docs.oracle.com/javase/specs/jls/se8/html/jls-8.html#jls-8.4.8.2) clase padre’ implementación, no es un método de anulación. Por ejemplo:

class StaticMethodTest {

  // static method and inheritance
  public static void main(String[] args) {
    Parent p = new Child();
    p.staticMethod(); // prints Inside Parent
    ((Child) p).staticMethod(); // prints Inside Child
  }

  static class Parent {
    public static void staticMethod() {
      System.out.println("Inside Parent");
    }
  }

  static class Child extends Parent {
    public static void staticMethod() {
      System.out.println("Inside Child");
    }
  }
}

Los métodos estáticos se vinculan a una clase, no a una instancia, y este método de vinculación ocurre en tiempo de compilación. Dado que en la primera llamada a staticMethod(), se usó la referencia de clase principal p, se invoca la versión Parent de staticMethod(). En el segundo caso, convertimos p en la clase Child, se ejecutó el staticMethod() de Child.

Estrechamiento y ampliación de referencias de objetos #

La conversión de una instancia de una clase base a una subclase como en: b = (B) a; se llama estrechamiento (ya que está tratando de reducir el objeto de la clase base a un objeto de clase más específico) y necesita un tipo explícito -emitir.

Convertir una instancia de una subclase en una clase base como en: A a = b; se denomina ampliación y no necesita conversión de tipo.

Para ilustrar, considere las siguientes declaraciones de clase y código de prueba:

class Vehicle {
}

class Car extends Vehicle {
}

class Truck extends Vehicle {
}

class MotorCycle extends Vehicle {
}

class Test {

    public static void main(String[] args) {
    
        Vehicle vehicle = new Car();
        Car car = new Car();        
    
        vehicle = car; // is valid, no cast needed

        Car c = vehicle // not valid
        Car c = (Car) vehicle; //valid
    }
}

La sentencia Vehicle vehicle = new Car(); es una sentencia Java válida. Cada instancia de ‘Coche’ es también un ‘Vehículo’. Por tanto, la cesión es legal sin necesidad de encasillamiento explícito.

Por otro lado, Auto c = vehículo; no es válido. El tipo estático de la variable vehicle es Vehicle, lo que significa que podría referirse a una instancia de Car, Truck, MotorCycleo cualquier otra subclase actual o futura deVehicle`. (O, de hecho, una instancia de ‘Vehicle’ en sí, ya que no la declaramos como una clase ‘abstracta’). La asignación no se puede permitir, ya que eso podría llevar a que ‘car’ se refiera a una instancia de ‘Camión’.

Para evitar esta situación, debemos agregar un tipo de conversión explícito:

Car c = (Car) vehicle;

El type-cast le dice al compilador que esperamos que el valor de vehicle sea un Car o una subclase de Car. Si es necesario, el compilador insertará código para realizar una verificación de tipos en tiempo de ejecución. Si la comprobación falla, se lanzará una ClassCastException cuando se ejecute el código.

Tenga en cuenta que no todas las conversiones de tipo son válidas. Por ejemplo:

String s = (String) vehicle;  // not valid

El compilador de Java sabe que una instancia cuyo tipo es compatible con Vehicle nunca puede ser compatible con String. La conversión de tipo nunca podría tener éxito, y el JLS exige que esto genere un error de compilación.

Programación a una interfaz #

La idea detrás de la programación de una interfaz es basar el código principalmente en las interfaces y solo usar clases concretas en el momento de la creación de instancias. En este contexto, un buen código que trate, p. Las colecciones de Java se verán así (no es que el método en sí sea de ninguna utilidad, solo es una ilustración):

public <T> Set<T> toSet(Collection<T> collection) {
  return Sets.newHashSet(collection);
}

mientras que el código incorrecto podría verse así:

public <T> HashSet<T> toSet(ArrayList<T> collection) {
  return Sets.newHashSet(collection);
}

No solo el primero se puede aplicar a una variedad más amplia de argumentos, sino que sus resultados serán más compatibles con el código proporcionado por otros desarrolladores que generalmente se adhieren al concepto de programación en una interfaz. Sin embargo, las razones más importantes para usar el primero son:

  • la mayoría de las veces el contexto, en el que se utiliza el resultado, no necesita ni debería necesitar tantos detalles como los que proporciona la implementación concreta;
  • adherirse a una interfaz fuerza un código más limpio y menos hacks, como otro método público que se agrega a una clase que sirve a un escenario específico;
  • el código es más comprobable ya que las interfaces son fácilmente burlables;
  • finalmente, el concepto ayuda incluso si solo se espera una implementación (al menos para la capacidad de prueba).

Entonces, ¿cómo se puede aplicar fácilmente el concepto de programación a una interfaz cuando se escribe código nuevo teniendo en mente una implementación en particular? Una opción que usamos comúnmente es una combinación de los siguientes patrones:

  • programación a una interfaz
  • fábrica
  • constructor

El siguiente ejemplo basado en estos principios es una versión simplificada y truncada de una implementación de RPC escrita para varios protocolos diferentes:

public interface RemoteInvoker {
  <RQ, RS> CompletableFuture<RS> invoke(RQ request, Class<RS> responseClass);
}

No se supone que la interfaz anterior se cree una instancia directamente a través de una fábrica, sino que derivamos otras interfaces más concretas, una para la invocación HTTP y otra para AMQP, cada una de las cuales tiene una fábrica y un constructor para construir instancias, que a su vez también son instancias de la interfaz anterior:

public interface AmqpInvoker extends RemoteInvoker {
  static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
    return new AmqpInvokerBuilder(instanceId, factory);
  }
}

Las instancias de RemoteInvoker para usar con AMQP ahora se pueden construir tan fácil como (o más complicado dependiendo del constructor):

RemoteInvoker invoker = AmqpInvoker.with(instanceId, factory)
  .requestRouter(router)
  .build();

Y la invocación de una solicitud es tan fácil como:

Response res = invoker.invoke(new Request(data), Response.class).get();

Debido a que Java 8 permite la colocación de métodos estáticos directamente en las interfaces, la fábrica intermedia se ha vuelto implícita en el código anterior reemplazado por AmqpInvoker.with(). En Java anterior a la versión 8, se puede lograr el mismo efecto con una clase Factory interna:

public interface AmqpInvoker extends RemoteInvoker {
  class Factory {
    public static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
      return new AmqpInvokerBuilder(instanceId, factory);
    }
  }
}

La instanciación correspondiente se convertiría entonces en:

RemoteInvoker invoker = AmqpInvoker.Factory.with(instanceId, factory)
  .requestRouter(router)
  .build();

El constructor utilizado anteriormente podría tener este aspecto (aunque se trata de una simplificación, ya que el actual permite definir hasta 15 parámetros que se desvían de los valores predeterminados). Tenga en cuenta que la construcción no es pública, por lo que solo se puede usar específicamente desde la interfaz AmqpInvoker anterior:

public class AmqpInvokerBuilder {
  ...
  AmqpInvokerBuilder(String instanceId, ConnectionFactory factory) {
    this.instanceId = instanceId;
    this.factory = factory;
  }

  public AmqpInvokerBuilder requestRouter(RequestRouter requestRouter) {
    this.requestRouter = requestRouter;
    return this;
  }

  public AmqpInvoker build() throws TimeoutException, IOException {
    return new AmqpInvokerImpl(instanceId, factory, requestRouter);
  }
}

En general, también se puede generar un constructor usando una herramienta como FreeBuilder.

Finalmente, la implementación estándar (y la única esperada) de esta interfaz se define como una clase local de paquete para hacer cumplir el uso de la interfaz, la fábrica y el constructor:

class AmqpInvokerImpl implements AmqpInvoker {
  AmqpInvokerImpl(String instanceId, ConnectionFactory factory, RequestRouter requestRouter) {
    ...
  }

  @Override
  public <RQ, RS> CompletableFuture<RS> invoke(final RQ request, final Class<RS> respClass) {
    ...
  }
}

Mientras tanto, este patrón demostró ser muy eficiente en el desarrollo de todo nuestro nuevo código sin importar cuán simple o compleja sea la funcionalidad.

Anulación en herencia #

La anulación en la herencia se usa cuando usa un método ya definido de una superclase en una subclase, pero de una manera diferente a cómo se diseñó originalmente el método en la superclase. La anulación permite al usuario reutilizar el código utilizando material existente y modificándolo para que se adapte mejor a las necesidades del usuario.

El siguiente ejemplo demuestra cómo ClassB anula la funcionalidad de ClassA cambiando lo que se envía a través del método de impresión:

Ejemplo:

public static void main(String[] args) {
    ClassA a = new ClassA();
    ClassA b = new ClassB();
    a.printing();
    b.printing();
}

class ClassA {
    public void printing() {        
        System.out.println("A");
    }
}

class ClassB extends ClassA {
    public void printing() {
         System.out.println("B");
    }
}
