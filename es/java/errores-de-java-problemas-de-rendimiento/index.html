Este tema describe una serie de “trampas” (es decir, errores que cometen los programadores de Java novatos) que se relacionan con el rendimiento de la aplicación Java.

Este tema describe algunas prácticas de codificación Java “micro” que son ineficaces. En la mayoría de los casos, las ineficiencias son relativamente pequeñas, pero aun así vale la pena evitarlas si es posible.

Pitfall: la concatenación de cadenas en un bucle no escala #

Considere el siguiente código como una ilustración:

public String joinWords(List<String> words) {
    String message = "";
    for (String word : words) {
        message = message + " " + word;
    }
    return message;
}

Desafortunadamente, este código es ineficiente si la lista de palabras es larga. La raíz del problema es esta afirmación:

message = message + " " + word;

Para cada iteración del ciclo, esta declaración crea una nueva cadena de mensaje que contiene una copia de todos los caracteres en la cadena de mensaje original con caracteres adicionales adjuntos. Esto genera muchas cadenas temporales y hace muchas copias.

Cuando analizamos joinWords, asumiendo que hay N palabras con una longitud promedio de M, encontramos que se crean O(N) cadenas temporales y O(M.N2) caracteres se copiarán en el proceso. El componente N2 es particularmente preocupante.

El enfoque recomendado para este tipo de problema1 es usar un StringBuilder en lugar de la concatenación de cadenas de la siguiente manera:

public String joinWords2(List<String> words) {
    StringBuilder message = new StringBuilder();
    for (String word : words) {
        message.append(" ").append(word);
    }
    return message.toString();
}

El análisis de joinWords2 debe tener en cuenta los gastos generales del “crecimiento” de la matriz de respaldo StringBuilder que contiene los caracteres del constructor. Sin embargo, resulta que el número de objetos nuevos creados es O(logN) y que el número de caracteres copiados es O(M.N) caracteres. Este último incluye caracteres copiados en la última llamada toString().

(Puede ser posible ajustar esto aún más, creando el StringBuilder con la capacidad correcta para comenzar. Sin embargo, la complejidad general sigue siendo la misma).

Volviendo al método joinWords original, resulta que la declaración crítica será optimizada por un compilador típico de Java a algo como esto:

  StringBuilder tmp = new StringBuilder();
  tmp.append(message).append(" ").append(word);
  message = tmp.toString();

Sin embargo, el compilador de Java no “elevará” el StringBuilder fuera del bucle, como hicimos a mano en el código para joinWords2.

Trampa: usar size() para probar si una colección está vacía es ineficiente. #

Java Collections Framework proporciona dos métodos relacionados para todos los objetos Collection:

  • size() devuelve el número de entradas en una Colección, y
  • El método isEmpty() devuelve verdadero si (y solo si) el ` La colección está vacía.

Ambos métodos se pueden utilizar para probar el vacío de la colección. Por ejemplo:

Collection<String> strings = new ArrayList<>();
boolean isEmpty_wrong = strings.size() == 0; // Avoid this
boolean isEmpty = strings.isEmpty();         // Best

Si bien estos enfoques tienen el mismo aspecto, algunas implementaciones de colecciones no almacenan el tamaño. Para tal colección, la implementación de size() necesita calcular el tamaño cada vez que se llama. Por ejemplo:

  • Una clase de lista enlazada simple (pero no java.util.LinkedList) podría necesitar recorrer la lista para contar los elementos.
  • La clase ConcurrentHashMap necesita sumar las entradas en todos los “segmentos” del mapa.
  • Una implementación perezosa de una colección puede necesitar realizar la colección completa en la memoria para poder contar los elementos.

Por el contrario, un método isEmpty() solo necesita probar si hay al menos un elemento en la colección. Esto no implica contar los elementos.

Si bien size() == 0 no siempre es menos eficiente que isEmpty(), es inconcebible que isEmpty() correctamente implementado sea menos eficiente que size() == 0. Por lo tanto, se prefiere isEmpty().

Dificultad: usar ’nuevo’ para crear instancias de contenedores primitivos es ineficiente #

El lenguaje Java le permite usar nuevo para crear instancias Integer, Boolean y así sucesivamente, pero generalmente es una mala idea. Es mejor usar el autoboxing (Java 5 y posterior) o el método valueOf.

 Integer i1 = new Integer(1);      // BAD
 Integer i2 = 2;                   // BEST (autoboxing)
 Integer i3 = Integer.valueOf(3);  // OK

La razón por la que usar new Integer(int) explícitamente es una mala idea es que crea un nuevo objeto (a menos que el compilador JIT lo optimice). Por el contrario, cuando se utiliza el encuadre automático o una llamada valueOf explícita, el tiempo de ejecución de Java intentará reutilizar un objeto Integer de una memoria caché de objetos preexistentes. Cada vez que el tiempo de ejecución tiene un “golpe” de caché, evita crear un objeto. Esto también ahorra memoria de almacenamiento dinámico y reduce los gastos generales del GC provocados por la rotación de objetos.

Notas:

  1. En implementaciones recientes de Java, el autoboxing se implementa llamando a valueOf, y hay cachés para Boolean, Byte, Short, Integer, Long y Character.
  2. El comportamiento de almacenamiento en caché para los tipos integrales es obligatorio según la especificación del lenguaje Java.

Pitfall - Interner strings para que puedas usar == es una mala idea #

Cuando algunos programadores ven este consejo:

“Probar cadenas usando == es incorrecto (a menos que las cadenas estén internadas)”

su reacción inicial es internar cadenas para que puedan usar ==. (Después de todo, == es más rápido que llamar a String.equals(...), ¿no es así?)

Este es el enfoque equivocado, desde varias perspectivas:

Fragilidad #

En primer lugar, solo puede usar == de forma segura si sabe que todos los objetos String que está probando han sido internados. El JLS garantiza que los literales de cadena en su código fuente se habrán internado. Sin embargo, ninguna de las API estándar de Java SE garantiza la devolución de cadenas internas, aparte de String.intern(String). Si pierde solo una fuente de objetos String que no han sido internados, su aplicación no será confiable. Esa falta de confiabilidad se manifestará como falsos negativos en lugar de excepciones, lo que puede dificultar su detección.

Costos de usar ‘intern()’ #

Bajo el capó, la internación funciona manteniendo una tabla hash que contiene objetos “String” previamente internados. Se utiliza algún tipo de mecanismo de referencia débil para que la tabla hash interna no se convierta en una fuga de almacenamiento. Si bien la tabla hash se implementa en código nativo (a diferencia de HashMap, HashTable, etc.), las llamadas internas siguen siendo relativamente costosas en términos de CPU y memoria utilizada.

Este coste hay que compararlo con el ahorro que vamos a conseguir usando == en lugar de equals. De hecho, no vamos a alcanzar el punto de equilibrio a menos que cada cadena interna se compare con otras cadenas “algunas” veces.

(Aparte: las pocas situaciones en las que vale la pena la pasantía tienden a reducir la huella de memoria de una aplicación donde las mismas cadenas se repiten muchas veces, y esas cadenas tienen una vida útil prolongada).

El impacto en la recolección de basura #

Además de los costos directos de CPU y memoria descritos anteriormente, las cadenas internas afectan el rendimiento del recolector de elementos no utilizados.

Para las versiones de Java anteriores a Java 7, las cadenas internas se guardan en el espacio “PermGen” que se recopila con poca frecuencia. Si es necesario recopilar PermGen, esto (generalmente) activa una recolección de basura completa. Si el espacio de PermGen se llena por completo, la JVM falla, incluso si hubiera espacio libre en los espacios de almacenamiento dinámico regulares.

En Java 7, el grupo de cadenas se movió de “PermGen” al montón normal. Sin embargo, la tabla hash seguirá siendo una estructura de datos de larga duración, lo que hará que las cadenas internas sean de larga duración. (Incluso si los objetos de cadena internos se asignaran en el espacio de Eden, lo más probable es que se promovieran antes de recopilarlos).

Por lo tanto, en todos los casos, internar una cadena prolongará su vida útil en relación con una cadena ordinaria. Eso aumentará los gastos generales de recolección de basura durante la vida útil de la JVM.

El segundo problema es que la tabla hash necesita usar un mecanismo de referencia débil de algún tipo para evitar que las cadenas internas pierdan memoria. Pero tal mecanismo es más trabajo para el recolector de basura.

Estos gastos generales de recolección de basura son difíciles de cuantificar, pero no hay duda de que existen. Si usa intern extensamente, podrían ser significativos.

El tamaño de la tabla hash del grupo de cadenas #

De acuerdo con esta fuente, desde Java 6 en adelante, el grupo de cadenas se implementa como una tabla hash de tamaño fijo con cadenas para manejar cadenas que se procesan en el mismo depósito. En las primeras versiones de Java 6, la tabla hash tenía un tamaño constante (cableado). Se agregó un parámetro de ajuste (-XX:StringTableSize) como una actualización de mediana edad a Java 6. Luego, en una actualización de mediana edad a Java 7, el tamaño predeterminado del grupo se cambió de 1009 a 60013 .

La conclusión es que si tiene la intención de usar intern de manera intensiva en su código, es aconsejable elegir una versión de Java donde el tamaño de la tabla hash se pueda ajustar y asegurarse de ajustar el tamaño de manera adecuada. De lo contrario, el rendimiento de intern puede degradarse a medida que el grupo crece.

La pasantía como potencial vector de denegación de servicio #

El algoritmo hashcode para cadenas es bien conocido. Si interna cadenas proporcionadas por usuarios o aplicaciones maliciosos, esto podría usarse como parte de un ataque de denegación de servicio (DoS). Si el agente malicioso hace arreglos para que todas las cadenas que proporciona tengan el mismo código hash, esto podría llevar a una tabla hash desequilibrada y al rendimiento O(N) para intern… donde N es el número de colisiones instrumentos de cuerda.

(Existen formas más sencillas/efectivas de lanzar un ataque DoS contra un servicio. Sin embargo, este vector podría usarse si el objetivo del ataque DoS es violar la seguridad o evadir las defensas DoS de primera línea).

Dificultad: problemas de eficiencia con las expresiones regulares #

La coincidencia de expresiones regulares es una herramienta poderosa (en Java y en otros contextos) pero tiene algunos inconvenientes. Uno de estos que las expresiones regulares tiende a ser bastante caro.

Las instancias de Pattern y Matcher deben reutilizarse #

Considere el siguiente ejemplo:

/**
 * Test if all strings in a list consist of English letters and numbers.
 * @param strings the list to be checked
 * @return 'true' if an only if all strings satisfy the criteria
 * @throws NullPointerException if 'strings' is 'null' or a 'null' element.
 */
public boolean allAlphanumeric(List<String> strings) {
    for (String s : strings) {
        if (!s.matches("[A-Za-z0-9]*")) {
            return false;
        }  
    }
    return true;
}

Este código es correcto, pero es ineficiente. El problema está en la llamada matches(...). Bajo el capó, s.matches("[A-Za-z0-9]*") es equivalente a esto:

Pattern.matches(s, "[A-Za-z0-9]*")

que a su vez es equivalente a

Pattern.compile("[A-Za-z0-9]*").matcher(s).matches()

La llamada Pattern.compile("[A-Za-z0-9]*") analiza la expresión regular, la analiza y construye un objeto Pattern que contiene la estructura de datos que utilizará el motor de expresiones regulares. Este es un cálculo no trivial. Luego se crea un objeto Matcher para envolver el argumento s. Finalmente, llamamos a match () para hacer la coincidencia de patrones real.

El problema es que todo este trabajo se repite para cada iteración del ciclo. La solución es reestructurar el código de la siguiente manera:

private static Pattern ALPHA_NUMERIC = Pattern.compile("[A-Za-z0-9]*");

public boolean allAlphanumeric(List<String> strings) {
    Matcher matcher = ALPHA_NUMERIC.matcher("");
    for (String s : strings) {
        matcher.reset(s);
        if (!matcher.matches()) {
            return false;
        }  
    }
    return true;
}

Tenga en cuenta que javadoc para Pattern dice:

Las instancias de esta clase son inmutables y son seguras para su uso por múltiples subprocesos simultáneos. Las instancias de la clase Matcher no son seguras para tal uso.

No uses match() cuando deberías usar find() #

Suponga que desea probar si una cadena s contiene tres o más dígitos seguidos. Puede expresar esto de varias maneras, incluyendo:

  if (s.matches(".*[0-9]{3}.*")) {
      System.out.println("matches");
  }

o

  if (Pattern.compile("[0-9]{3}").matcher(s).find()) {
      System.out.println("matches");
  }

El primero es más conciso, pero también es probable que sea menos eficiente. A primera vista, la primera versión intentará hacer coincidir toda la cadena con el patrón. Además, dado que “.*” es un patrón “codicioso”, es probable que el comparador de patrones avance “con entusiasmo” hasta el final de la cadena y retroceda hasta que encuentre una coincidencia.

Por el contrario, la segunda versión buscará de izquierda a derecha y dejará de buscar tan pronto como encuentre los 3 dígitos seguidos.

Use alternativas más eficientes a las expresiones regulares

Las expresiones regulares son una herramienta poderosa, pero no deberían ser su única herramienta. Muchas tareas se pueden hacer de manera más eficiente de otras maneras. Por ejemplo:

 Pattern.compile("ABC").matcher(s).find()

hace lo mismo que:

 s.contains("ABC")

excepto que este último es mucho más eficiente. (Incluso si puede amortizar el costo de compilar la expresión regular).

A menudo, la forma no regular es más complicada. Por ejemplo, la prueba realizada por la llamada matches() al método anterior allAlplanumeric se puede reescribir como:

 public boolean matches(String s) {
     for (char c : s) {
         if ((c >= 'A' && c <= 'Z') ||
             (c >= 'a' && c <= 'z') ||
             (c >= '0' && c <= '9')) {
              return false;
         }
     }
     return true;
 }

Eso es más código que usar un ‘Matcher’, pero también será significativamente más rápido.

Retroceso catastrófico #

(Esto es potencialmente un problema con todas las implementaciones de expresiones regulares, pero lo mencionaremos aquí porque es una trampa para el uso de Pattern).

Considere este ejemplo (artificioso):

Pattern pat = Pattern.compile("(A+)+B");
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB").matches());
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAC").matches());

La primera llamada println imprimirá rápidamente true. El segundo imprimirá falso. Finalmente. De hecho, si experimenta con el código anterior, verá que cada vez que agregue una ‘A’ antes de la ‘C’, el tiempo se duplicará.

Este comportamiento es un ejemplo de retroceso catastrófico. El motor de coincidencia de patrones que implementa la coincidencia de expresiones regulares está probando infructuosamente todas las posibles formas en que el patrón podría coincidir.

Veamos qué significa realmente (A+)+B. Superficialmente, parece decir “uno o más caracteres ‘A’ seguidos de un valor ‘B’”, pero en realidad dice uno o más grupos, cada uno de los cuales consta de uno o más caracteres ‘A’. Así por ejemplo:

  • ‘AB’ coincide de una sola manera: ‘(A)B’
  • ‘AAB’ coincide de dos maneras: ‘(AA)B’ o ‘(A)(A)B`
  • ‘AAAB’ coincide con cuatro formas: ‘(AAA)B’ o ‘(AA)(A)B o '(A)(AA)B o ‘(A)(A)(A)B`
  • y así

En otras palabras, el número de coincidencias posibles es 2N donde N es el número de caracteres A.

El ejemplo anterior es claramente artificial, pero los patrones que exhiben este tipo de características de rendimiento (es decir, O(2^N) o O(N^K) para una K grande) surgen con frecuencia cuando se usan expresiones regulares mal consideradas. son usados. Hay algunos remedios estándar:

  • Evite anidar patrones repetitivos dentro de otros patrones repetitivos.
  • Evite usar demasiados patrones repetitivos.
  • Utilice la repetición sin retroceso según corresponda.
  • No use expresiones regulares para tareas de análisis complicadas. (Escriba un analizador adecuado en su lugar).

Por último, tenga cuidado con las situaciones en las que un usuario o un cliente de API pueden proporcionar una cadena de expresiones regulares con características patológicas. Eso puede conducir a una “denegación de servicio” accidental o deliberada.

Dificultad: las lecturas/escrituras pequeñas en flujos sin búfer son ineficientes #

Considere el siguiente código para copiar un archivo a otro:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new FileInputStream(args[0]);
             OutputStream os = new FileOutputStream(args[1])) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

(Hemos omitido deliberadamente la verificación normal de argumentos, el informe de errores, etc. porque no son relevantes para el punto de este ejemplo).

Si compila el código anterior y lo usa para copiar un archivo enorme, notará que es muy lento. De hecho, será al menos un par de órdenes de magnitud más lento que las utilidades estándar de copia de archivos del sistema operativo.

(¡Agregue medidas de rendimiento reales aquí!)

La razón principal por la que el ejemplo anterior es lento (en el caso de archivos grandes) es que está realizando lecturas de un byte y escrituras de un byte en flujos de bytes sin búfer. La forma sencilla de mejorar el rendimiento es envolver los flujos con flujos almacenados en búfer. Por ejemplo:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new BufferedInputStream(
                     new FileInputStream(args[0]));
             OutputStream os = new BufferedOutputStream(
                     new FileOutputStream(args[1]))) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

Estos pequeños cambios mejorarán la tasa de copia de datos en al menos un par de órdenes de magnitud, dependiendo de varios factores relacionados con la plataforma. Los contenedores de flujo almacenados en búfer hacen que los datos se lean y escriban en fragmentos más grandes. Ambas instancias tienen buffers implementados como matrices de bytes.

  • Con is, los datos se leen del archivo en el búfer unos pocos kilobytes a la vez. Cuando se llama a read(), la implementación generalmente devolverá un byte del búfer. Solo leerá del flujo de entrada subyacente si se ha vaciado el búfer.

  • El comportamiento de os es análogo. Las llamadas a os.write(int) escriben bytes individuales en el búfer. Los datos solo se escriben en el flujo de salida cuando el búfer está lleno, o cuando os se vacía o se cierra.

¿Qué pasa con las transmisiones basadas en personajes? #

Como debe saber, Java I/O proporciona diferentes API para leer y escribir datos binarios y de texto.

  • InputStream y OutputStream son las API base para la E/S binaria basada en secuencias
  • Reader y Writer son las API base para la E/S de texto basada en secuencias.

Para E/S de texto, BufferedReader y BufferedWriter son los equivalentes de BufferedInputStream y BufferedOutputStream.

¿Por qué las secuencias almacenadas en búfer marcan tanta diferencia? #

La verdadera razón por la que los flujos almacenados en búfer ayudan al rendimiento tiene que ver con la forma en que una aplicación se comunica con el sistema operativo:

  • El método Java en una aplicación Java o las llamadas a procedimientos nativos en las bibliotecas de tiempo de ejecución nativas de JVM son rápidas. Por lo general, toman un par de instrucciones de la máquina y tienen un impacto mínimo en el rendimiento.

  • Por el contrario, las llamadas en tiempo de ejecución de JVM al sistema operativo no son rápidas. Implican algo conocido como “syscall”. El patrón típico para una llamada al sistema es el siguiente:

    1. Put the syscall arguments into registers.
    2. Execute a SYSENTER trap instruction.
    3. The trap handler switched to privileged state and changes the virtual memory mappings. Then it dispatches to the code to handle the specific syscall.
    4. The syscall handler checks the arguments, taking care that it isn’t being told to access memory that the user process should not see.
    5. The syscall specific work is performed. In the case of a read syscall, this may involve:
      1. checking that there is data to be read at the file descriptor’s current position
      2. calling the file system handler to fetch the required data from disk (or wherever it is stored) into the buffer cache,
      3. copying data from the buffer cache to the JVM-supplied address
      4. adjusting thstream pointerse file descriptor position
    6. Return from the syscall. This entails changing VM mappings again and switching out of privileged state.

Como puede imaginar, realizar una sola llamada al sistema puede generar miles de instrucciones de máquina. De manera conservadora, al menos dos órdenes de magnitud más largos que una llamada de método normal. (Probablemente tres o más.)

Dado esto, la razón por la que las secuencias almacenadas en búfer marcan una gran diferencia es que reducen drásticamente la cantidad de llamadas al sistema. En lugar de hacer una llamada al sistema para cada llamada read(), el flujo de entrada almacenado en el búfer lee una gran cantidad de datos en un búfer según sea necesario. La mayoría de las llamadas read() en el flujo almacenado en el búfer realizan algunas comprobaciones de límites simples y devuelven un byte que se leyó previamente. Se aplica un razonamiento similar en el caso del flujo de salida y también en los casos del flujo de caracteres.

(Algunas personas piensan que el rendimiento de E/S almacenado en búfer proviene de la falta de coincidencia entre el tamaño de la solicitud de lectura y el tamaño de un bloque de disco, la latencia de rotación del disco y cosas por el estilo. De hecho, un sistema operativo moderno utiliza una serie de estrategias para garantizar que el la aplicación típicamente no necesita esperar por el disco. Esta no es la verdadera explicación.)

¿Las secuencias almacenadas en búfer siempre son una victoria? #

No siempre. Las secuencias almacenadas en búfer son definitivamente una victoria si su aplicación va a hacer muchas lecturas o escrituras “pequeñas”. Sin embargo, si su aplicación solo necesita realizar lecturas o escrituras grandes en / desde un byte[] o char[] grande, entonces las secuencias almacenadas en búfer no le brindarán ningún beneficio real. De hecho, incluso podría haber una (pequeña) penalización de rendimiento.

¿Es esta la forma más rápida de copiar un archivo en Java? #

No, no lo es. Cuando utiliza las API basadas en flujo de Java para copiar un archivo, incurre en el costo de al menos una copia adicional de memoria a memoria de los datos. Es posible evitar esto si utiliza las API ByteBuffer y Channel de NIO. (Agregue un enlace a un ejemplo separado aquí.)

Trampa: los gastos generales de la creación de mensajes de registro #

Los niveles de registro TRACE y DEBUG están ahí para poder transmitir gran detalle sobre la operación del código dado en tiempo de ejecución. Por lo general, se recomienda establecer el nivel de registro por encima de estos, sin embargo, se debe tener cuidado para que estas declaraciones no afecten el rendimiento, incluso cuando aparentemente están “apagadas”.

Considere esta declaración de registro:

// Processing a request of some kind, logging the parameters
LOG.debug("Request coming from " + myInetAddress.toString() 
          + " parameters: " + Arrays.toString(veryLongParamArray));

Incluso cuando el nivel de registro se establece en INFO, los argumentos pasados ​​a debug() se evaluarán en cada ejecución de la línea. Esto hace que consuma innecesariamente en varios aspectos:

  • Concatenación String: se crearán varias instancias de String
  • InetAddress podría incluso hacer una búsqueda de DNS.
  • el veryLongParamArray puede ser muy largo: crear un String a partir de él consume memoria, lleva tiempo

Solución #

La mayoría de los marcos de registro proporcionan medios para crear mensajes de registro utilizando cadenas de corrección y referencias a objetos. El mensaje de registro se evaluará solo si el mensaje realmente se registra. Ejemplo:

// No toString() evaluation, no string concatenation if debug is disabled
LOG.debug("Request coming from {} parameters: {}", myInetAddress, parameters));

Esto funciona muy bien siempre que todos los parámetros se puedan convertir en cadenas usando String.valueOf(Object). Si el cálculo del mensaje de registro es más complejo, el nivel de registro se puede verificar antes de iniciar sesión:

if (LOG.isDebugEnabled()) {
    // Argument expression evaluated only when DEBUG is enabled
    LOG.debug("Request coming from {}, parameters: {}", myInetAddress,
              Arrays.toString(veryLongParamArray);
}

Aquí, LOG.debug() con el costoso cálculo Arrays.toString(Obect[]) se procesa solo cuando DEBUG está realmente habilitado.

Dificultad: llamar a ’nueva cadena (cadena)’ es ineficiente #

Usar new String(String) para duplicar una cadena es ineficiente y casi siempre innecesario.

  • Los objetos de cadena son inmutables, por lo que no es necesario copiarlos para protegerlos de los cambios.
  • En algunas versiones anteriores de Java, los objetos String pueden compartir matrices de respaldo con otros objetos String. En esas versiones, es posible perder memoria creando una subcadena (pequeña) de una cadena (grande) y reteniéndola. Sin embargo, desde Java 7 en adelante, las matrices de respaldo String no se comparten.

En ausencia de cualquier beneficio tangible, llamar a nueva Cadena(Cadena) es simplemente un desperdicio:

  • Hacer la copia requiere tiempo de CPU.
  • La copia usa más memoria, lo que aumenta la huella de memoria de la aplicación y/o aumenta los gastos generales del GC.
  • Las operaciones como equals(Object) y hashCode() pueden ser más lentas si se copian los objetos String.

Pitfall - Calling System.gc() es ineficiente #

Es (casi siempre) una mala idea llamar a System.gc().

El javadoc para el método gc() especifica lo siguiente:

“Llamar al método gc sugiere que la Máquina virtual de Java se esfuerce en reciclar los objetos no utilizados para que la memoria que ocupan actualmente esté disponible para su reutilización rápida. Cuando el control regresa de la llamada al método, la Máquina virtual de Java ha hecho un mejor esfuerzo esfuerzo por recuperar el espacio de todos los objetos desechados”.

Hay un par de puntos importantes que se pueden extraer de esto:

  1. El uso de la palabra “sugiere” en lugar de (decir) “dice” significa que la JVM es libre de ignorar la sugerencia. El comportamiento predeterminado de JVM (versiones recientes) es seguir la sugerencia, pero esto se puede anular configurando -XX:+DisableExplicitGC al iniciar JVM.

  2. La frase “un mejor esfuerzo para recuperar espacio de todos los objetos descartados” implica que llamar a gc activará una recolección de basura “completa”.

Entonces, ¿por qué llamar a System.gc() es una mala idea?

Primero, ejecutar una recolección de basura completa es costoso. Un GC completo implica visitar y “marcar” todos los objetos que aún se pueden alcanzar; es decir, todo objeto que no sea basura. Si activa esto cuando no hay mucha basura para recolectar, entonces el GC hace mucho trabajo por un beneficio relativamente pequeño.

En segundo lugar, una recolección de basura completa puede perturbar las propiedades de “localidad” de los objetos que no se recolectan. Los objetos que son asignados por el mismo subproceso aproximadamente al mismo tiempo tienden a estar ubicados juntos en la memoria. Esto es bueno. Es probable que los objetos que se asignan al mismo tiempo estén relacionados; es decir, hacer referencia entre sí. Si su aplicación usa esas referencias, entonces lo más probable es que el acceso a la memoria sea más rápido debido a varios efectos de almacenamiento en caché de memoria y página. Desafortunadamente, una recolección de basura completa tiende a mover los objetos, de modo que los objetos que alguna vez estuvieron cerca ahora están más separados.

En tercer lugar, ejecutar una recolección de basura completa puede hacer que su aplicación se detenga hasta que se complete la recolección. Mientras esto sucede, su aplicación no responderá.

De hecho, la mejor estrategia es dejar que la JVM decida cuándo ejecutar el GC y qué tipo de colección ejecutar. Si no interfiere, la JVM elegirá una hora y un tipo de recopilación que optimice el rendimiento o minimice los tiempos de pausa del GC.

Al principio decíamos “… (casi siempre) una mala idea…”. De hecho, hay un par de escenarios en los que podría ser una buena idea:

  1. Si está implementando una prueba unitaria para algún código que es sensible a la recolección de basura (por ejemplo, algo que involucre finalizadores o referencias débiles/suaves/fantasmas), entonces puede ser necesario llamar a System.gc().

  2. En algunas aplicaciones interactivas, puede haber momentos específicos en los que al usuario no le importe si hay una pausa en la recolección de elementos no utilizados. Un ejemplo es un juego donde hay pausas naturales en el “juego”; p.ej. al cargar un nuevo nivel.

Dificultad: el uso excesivo de tipos de envoltorios primitivos es ineficiente #

Considere estas dos piezas de código:

int a = 1000;
int b = a + 1;

y

Integer a = 1000;
Integer b = a + 1;

Pregunta: ¿Qué versión es más eficiente?

Respuesta: Las dos versiones parecen casi idénticas, pero la primera versión es mucho más eficiente que la segunda.

La segunda versión usa una representación para los números que usa más espacio y se basa en el auto-boxing y auto-unboxing detrás de escena. De hecho, la segunda versión es directamente equivalente al siguiente código:

Integer a = Integer.valueOf(1000);               // box 1000
Integer b = Integer.valueOf(a.intValue() + 1);   // unbox 1000, add 1, box 1001

Comparando esto con la otra versión que usa int, claramente hay tres llamadas de método adicionales cuando se usa Integer. En el caso de valueOf, cada una de las llamadas creará e inicializará un nuevo objeto Integer. Es probable que todo este trabajo adicional de empaquetado y desempaquetado haga que la segunda versión sea un orden de magnitud más lenta que la primera.

Además de eso, la segunda versión está asignando objetos en el montón en cada llamada valueOf. Si bien la utilización del espacio es específica de la plataforma, es probable que esté en la región de 16 bytes para cada objeto ‘Integer’. Por el contrario, la versión int no necesita espacio de almacenamiento adicional, suponiendo que a y b sean variables locales.

Otra gran razón por la que las primitivas son más rápidas que su equivalente en caja es cómo se distribuyen sus respectivos tipos de matriz en la memoria.

Si toma int[] e Integer[] como ejemplo, en el caso de un int[] los valores int se disponen de forma contigua en la memoria. Pero en el caso de un Integer[] no son los valores los que se presentan, sino las referencias (punteros) a los objetos Integer, que a su vez contienen los valores int reales.

Además de ser un nivel adicional de direccionamiento indirecto, esto puede ser un gran tanque cuando se trata de almacenar en caché la localidad al iterar sobre los valores. En el caso de un int[], la CPU podría obtener todos los valores en la matriz, en su caché a la vez, porque son contiguos en la memoria. Pero en el caso de un Integer[], la CPU potencialmente tiene que hacer una recuperación de memoria adicional para cada elemento, ya que la matriz solo contiene referencias a los valores reales.

En resumen, el uso de tipos de contenedores primitivos es relativamente costoso tanto en recursos de CPU como de memoria. Usarlos innecesariamente es ineficiente.

Dificultad: iterar las claves de un mapa puede ser ineficiente #

El siguiente código de ejemplo es más lento de lo necesario:

Map<String, String> map = new HashMap<>(); 
for (String key : map.keySet()) {
    String value = map.get(key);
    // Do something with key and value
}

Esto se debe a que requiere una búsqueda en el mapa (el método get()) para cada clave en el mapa. Esta búsqueda puede no ser eficiente (en un HashMap, implica llamar a hashCode en la clave, luego buscar el depósito correcto en las estructuras de datos internas y, a veces, incluso llamar a equals). En un mapa grande, esto puede no ser una sobrecarga trivial.

La forma correcta de evitar esto es iterar en las entradas del mapa, que se detalla en el [tema Colecciones] (https://www.wikiod.com/es/java/colecciones#Iterando sobre Colecciones 201608260922476000177)
