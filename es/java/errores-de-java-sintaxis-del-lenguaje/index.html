Varios usos indebidos del lenguaje de programación Java pueden hacer que un programa genere resultados incorrectos a pesar de estar compilado correctamente. El objetivo principal de este tema es enumerar las trampas comunes con sus causas y proponer la forma correcta de evitar caer en tales problemas.

Este tema trata sobre aspectos específicos de la sintaxis del lenguaje Java que son propensos a errores o que no deben usarse de ciertas maneras.

Dificultad: falta un ‘descanso’ en un caso de ‘cambio’ #

Estos problemas de Java pueden ser muy vergonzosos y, a veces, pasan desapercibidos hasta que se ejecutan en producción. El comportamiento fallthrough en las sentencias switch suele ser útil; sin embargo, perder una palabra clave de “romper” cuando no se desea tal comportamiento puede conducir a resultados desastrosos. Si se olvidó de poner un “pausa” en el “caso 0” en el ejemplo de código a continuación, el programa escribirá “Cero” seguido de “Uno”, ya que el flujo de control dentro de aquí pasará por toda la declaración de “cambio” hasta llega a una “ruptura”. Por ejemplo:

public static void switchCasePrimer() {
        int caseIndex = 0;
        switch (caseIndex) {
            case 0:
                System.out.println("Zero");
            case 1:
                System.out.println("One");
                break;
            case 2:
                System.out.println("Two");
                break;
            default:
                System.out.println("Default");
        }
}

En la mayoría de los casos, la solución más limpia sería usar interfaces y mover el código con un comportamiento específico a implementaciones separadas (composición sobre herencia)

Si una declaración de cambio es inevitable, se recomienda documentar fallas “esperadas” si ocurren. De esa manera, muestra a los compañeros desarrolladores que está al tanto de la pausa que falta y que este es el comportamiento esperado.

switch(caseIndex) {
    [...]
    case 2:
        System.out.println("Two");
        // fallthrough
    default:
        System.out.println("Default");

Dificultad: omitir las llaves: los problemas de “colgando si” y “colgando si no” #

La última versión de la guía de estilo Oracle Java exige que las declaraciones “then” y “else” en una declaración if siempre deben estar entre “llaves” o “corchetes”. Se aplican reglas similares a los cuerpos de varias sentencias de bucle.

if (a) {           // <- open brace
    doSomething();
    doSomeMore();
}                  // <- close brace

Esto no es realmente requerido por la sintaxis del lenguaje Java. De hecho, si la parte “entonces” de una declaración si es una sola declaración, es legal omitir las llaves

if (a)
    doSomething();

o incluso

if (a) doSomething();

Sin embargo, existen peligros al ignorar las reglas de estilo de Java y omitir las llaves. Específicamente, aumenta significativamente el riesgo de que el código con sangría defectuosa se lea mal.

El problema del “si pendiente”:

Considere el código de ejemplo de arriba, reescrito sin llaves.

if (a)
   doSomething();
   doSomeMore();

Este código parece decir que las llamadas a doSomething y doSomeMore ocurrirán si y solo si a es true. De hecho, el código está mal sangrado. La especificación del lenguaje Java de que la llamada doSomeMore() es una declaración separada que sigue a la declaración if. La sangría correcta es la siguiente:

if (a)
   doSomething();
doSomeMore();

El problema de “colgar más”

Aparece un segundo problema cuando agregamos else a la mezcla. Considere el siguiente ejemplo al que le faltan llaves.

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
else
   doZ();

El código anterior parece decir que doZ se llamará cuando a sea falso. De hecho, la sangría es incorrecta una vez más. La sangría correcta para el código es:

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
   else
      doZ();

Si el código se escribió de acuerdo con las reglas de estilo de Java, en realidad se vería así:

if (a) {
   if (b) {
      doX();
   } else if (c) {
      doY(); 
   } else {
      doZ();
   }
}

Para ilustrar por qué eso es mejor, suponga que accidentalmente sangró mal el código. Podrías terminar con algo como esto:

if (a) {                         if (a) {
   if (b) {                          if (b) {
      doX();                            doX();
   } else if (c) {                   } else if (c) {
      doY();                            doY();
} else {                         } else {
   doZ();                            doZ();
}                                    }
}                                }

Pero en ambos casos, el código mal sangrado “parece incorrecto” a los ojos de un programador Java experimentado.

Trampa - Declarar clases con los mismos nombres que las clases estándar #

A veces, los programadores que son nuevos en Java cometen el error de definir una clase con un nombre que es el mismo que el de una clase ampliamente utilizada. Por ejemplo:

package com.example;

/**
 * My string utilities
 */
public class String {
    ....
}

Luego se preguntan por qué obtienen errores inesperados. Por ejemplo:

package com.example;

public class Test {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

Si compila y luego intenta ejecutar las clases anteriores, obtendrá un error:

$ javac com/example/*.java
$ java com.example.Test
Error: Main method not found in class test.Test, please define the main method as:
   public static void main(String[] args)
or a JavaFX application class must extend javafx.application.Application

Alguien que mire el código de la clase Prueba vería la declaración de main y miraría su firma y se preguntaría de qué se queja el comando java. Pero, de hecho, el comando java está diciendo la verdad.

Cuando declaramos una versión de String en el mismo paquete que Test, esta versión tiene prioridad sobre la importación automática de java.lang.String. Por lo tanto, la firma del método Test.main es en realidad

void main(com.example.String[] args)

en vez de

void main(java.lang.String[] args)

y el comando java no reconocerá eso como un método de punto de entrada.

Lección: No defina clases que tengan el mismo nombre que las clases existentes en java.lang u otras clases de uso común en la biblioteca Java SE. Si hace eso, se expone a todo tipo de errores oscuros.

Trampa - Ignorar la visibilidad del método #

Incluso los desarrolladores de Java experimentados tienden a pensar que Java tiene solo tres modificadores de protección. ¡El idioma en realidad tiene cuatro! El nivel de visibilidad de paquete privado (también conocido como predeterminado) a menudo se olvida.

Debes prestar atención a los métodos que haces públicos. Los métodos públicos en una aplicación son la API visible de la aplicación. Esto debería ser lo más pequeño y compacto posible, especialmente si está escribiendo una biblioteca reutilizable (vea también el principio SOLID). Es importante considerar de manera similar la visibilidad de todos los métodos y usar solo el acceso protegido o privado del paquete cuando corresponda.

Cuando declara métodos que deberían ser privados como públicos, expone los detalles de implementación internos de la clase.

Un corolario de esto es que solo prueba unitaria los métodos públicos de su clase; de ​​hecho, solo puede probar métodos públicos. Es una mala práctica aumentar la visibilidad de los métodos privados solo para poder ejecutar pruebas unitarias contra esos métodos. La prueba de métodos públicos que llaman a los métodos con visibilidad más restrictiva debería ser suficiente para probar una API completa. Nunca debe expandir su API con más métodos públicos solo para permitir pruebas unitarias.

Escollo - Literales octales #

Considere el siguiente fragmento de código:

// Print the sum of the numbers 1 to 10
int count = 0;
for (int i = 1; i < 010; i++) {    // Mistake here ....
    count = count + i;
}
System.out.println("The sum of 1 to 10 is " + count);

Un principiante de Java podría sorprenderse al saber que el programa anterior imprime la respuesta incorrecta. En realidad, imprime la suma de los números del 1 al 8.

El motivo es que el compilador de Java interpreta un literal entero que comienza con el dígito cero (‘0’) como un literal octal, no como un literal decimal como cabría esperar. Por lo tanto, 010 es el número octal 10, que es 8 en decimal.

Trampa - Usar ‘==’ para probar un booleano #

A veces, un nuevo programador de Java escribirá un código como este:

public void check(boolean ok) {
    if (ok == true) {           // Note 'ok == true'
        System.out.println("It is OK");
    }
}

Un programador experimentado detectaría que es torpe y querría reescribirlo como:

public void check(boolean ok) {
    if (ok) {
       System.out.println("It is OK");
    }
}

Sin embargo, hay más mal con ok == true que simple torpeza. Considere esta variación:

public void check(boolean ok) {
    if (ok = true) {           // Oooops!
        System.out.println("It is OK");
    }
}

Aquí el programador ha escrito incorrectamente == como = … y ahora el código tiene un error sutil. La expresión x = true asigna incondicionalmente true a x y luego se evalúa como true. En otras palabras, el método verificar ahora imprimirá “Está bien” sin importar cuál fue el parámetro.

La lección aquí es dejar el hábito de usar == false y == true. Además de ser detallados, hacen que su codificación sea más propensa a errores.

Nota: una posible alternativa a ok == true que evita el escollo es usar condiciones de Yoda; es decir, coloque el literal en el lado izquierdo del operador relacional, como en true == ok. Esto funciona, pero la mayoría de los programadores probablemente estarían de acuerdo en que las condiciones de Yoda parecen extrañas. Ciertamente ok (o !ok) es más conciso y más natural.

Dificultad: punto y coma fuera de lugar y llaves faltantes #

Este es un error que causa verdadera confusión a los principiantes de Java, al menos la primera vez que lo hacen. En lugar de escribir esto:

if (feeling == HAPPY)
    System.out.println("Smile");
else
    System.out.println("Frown");

accidentalmente escriben esto:

if (feeling == HAPPY);
    System.out.println("Smile");
else
    System.out.println("Frown");

y se quedan perplejos cuando el compilador de Java les dice que else está fuera de lugar. El compilador de Java interpreta lo anterior de la siguiente manera:

if (feeling == HAPPY)
    /*empty statement*/ ;
System.out.println("Smile");   // This is unconditional
else                           // This is misplaced.  A statement cannot
                               // start with 'else'
System.out.println("Frown");

En otros casos, no habrá errores de compilación, pero el código no hará lo que el programador pretende. Por ejemplo:

for (int i = 0; i < 5; i++);
    System.out.println("Hello");

solo imprime “Hola” una vez. Una vez más, el punto y coma espurio significa que el cuerpo del bucle for es una declaración vacía. Eso significa que la llamada println que sigue es incondicional.

Otra variación:

for (int i = 0; i < 5; i++);
    System.out.println("The number is " + i);

Esto dará un error de “No se puede encontrar el símbolo” para i. La presencia del punto y coma espurio significa que la llamada println está intentando usar i fuera de su alcance.

En esos ejemplos, hay una solución directa: simplemente elimine el punto y coma falso. Sin embargo, hay algunas lecciones más profundas que se pueden extraer de estos ejemplos:

  1. El punto y coma en Java no es “ruido sintáctico”. La presencia o ausencia de un punto y coma puede cambiar el significado de su programa. No los agregue simplemente al final de cada línea.

  2. No confíes en la sangría de tu código. En el lenguaje Java, el compilador ignora los espacios en blanco adicionales al comienzo de una línea.

  3. Use un indentador automático. Todos los IDE y muchos editores de texto simples saben cómo sangrar correctamente el código Java.

  4. Esta es la lección más importante. Siga las últimas pautas de estilo de Java y ponga llaves alrededor de las declaraciones “entonces” y “si no” y la declaración del cuerpo de un ciclo. La llave abierta ({) no debe estar en una nueva línea.

Si el programador siguió las reglas de estilo, entonces el ejemplo si con un punto y coma fuera de lugar se vería así:

if (feeling == HAPPY); {
    System.out.println("Smile");
} else {
    System.out.println("Frown");
}

Eso parece extraño para un ojo experimentado. Si sangró automáticamente ese código, probablemente se vería así:

if (feeling == HAPPY); {
                           System.out.println("Smile");
                       } else {
                           System.out.println("Frown");
                       }

lo que debería destacarse como incorrecto incluso para un principiante.

Trampa: las importaciones de comodines pueden hacer que su código sea frágil #

Considere el siguiente ejemplo parcial:

import com.example.somelib.*;
import com.acme.otherlib.*;

public class Test {
    private Context x = new Context();   // from com.example.somelib
    ...
}

Suponga que cuando desarrolló el código por primera vez contra la versión 1.0 de somelib y la versión 1.0 de otherlib. Luego, en algún momento posterior, debe actualizar sus dependencias a versiones posteriores y decide usar otherlib versión 2.0. También supongamos que uno de los cambios que hicieron a otherlib entre 1.0 y 2.0 fue agregar una clase Context.

Ahora, cuando vuelva a compilar Test, obtendrá un error de compilación que le indicará que Context es una importación ambigua.

Si está familiarizado con el código base, esto probablemente sea solo un inconveniente menor. Si no es así, entonces tiene trabajo que hacer para abordar este problema, aquí y posiblemente en otros lugares.

El problema aquí son las importaciones de comodines. Por un lado, el uso de comodines puede hacer que sus clases sean unas pocas líneas más cortas. Por otra parte:

  • Los cambios compatibles con versiones posteriores en otras partes de su código base, en las bibliotecas estándar de Java o en bibliotecas de terceros pueden provocar errores de compilación.

  • La legibilidad sufre. A menos que esté utilizando un IDE, puede ser difícil averiguar cuál de las importaciones de comodín está extrayendo una clase nombrada.

La lección es que es una mala idea utilizar importaciones de comodines en un código que debe durar mucho tiempo. Las importaciones específicas (sin comodines) no requieren mucho esfuerzo de mantenimiento si usa un IDE, y el esfuerzo vale la pena.

Trampa: usar ‘afirmar’ para argumento o validación de entrada de usuario #

Una pregunta que surge ocasionalmente en StackOverflow es si es apropiado usar assert para validar los argumentos proporcionados a un método, o incluso las entradas proporcionadas por el usuario.

La respuesta simple es que no es apropiado.

Las mejores alternativas incluyen:

  • Lanzar una IllegalArgumentException usando código personalizado.
  • Usando los métodos Precondiciones disponibles en la biblioteca de Google Guava.
  • Usando los métodos Validar disponibles en la biblioteca Apache Commons Lang3.

Esto es lo que la Especificación del lenguaje Java (JLS 14.10, para Java 8) aconseja sobre este asunto:

Por lo general, la comprobación de aserciones se activa durante el desarrollo y las pruebas del programa y se desactiva para la implementación, a fin de mejorar el rendimiento.

Debido a que las aserciones pueden estar deshabilitadas, los programas no deben asumir que las expresiones contenidas en las aserciones serán evaluadas. Por lo tanto, estas expresiones booleanas generalmente deberían estar libres de efectos secundarios. La evaluación de una expresión booleana de este tipo no debería afectar ningún estado que sea visible después de que se complete la evaluación. No es ilegal que una expresión booleana contenida en una aserción tenga un efecto secundario, pero generalmente es inapropiado, ya que podría hacer que el comportamiento del programa varíe dependiendo de si las aserciones están habilitadas o deshabilitadas.

A la luz de esto, las afirmaciones no deben usarse para verificar argumentos en métodos públicos. La verificación de argumentos suele ser parte del contrato de un método, y este contrato debe cumplirse ya sea que las aserciones estén habilitadas o deshabilitadas.

Un problema secundario con el uso de aserciones para la verificación de argumentos es que los argumentos erróneos deberían dar como resultado una excepción de tiempo de ejecución adecuada (como IllegalArgumentException, ArrayIndexOutOfBoundsException o NullPointerException). Un error de aserción no generará una excepción adecuada. Nuevamente, no es ilegal usar aserciones para verificar argumentos en métodos públicos, pero generalmente es inapropiado. La intención es que AssertionError nunca sea detectado, pero es posible hacerlo, por lo tanto, las reglas para las sentencias de prueba deben tratar las afirmaciones que aparecen en un bloque de prueba de forma similar al tratamiento actual de las sentencias de lanzamiento.

Trampa: sobrecargar en lugar de anular #

Considere el siguiente ejemplo:

public final class Person {
    private final String firstName;
    private final String lastName;
   
    public Person(String firstName, String lastName) {
        this.firstName = (firstName == null) ? "" : firstName;
        this.lastName = (lastName == null) ? "" : lastName;
    }

    public boolean equals(String other) {
        if (!(other instanceof Person)) {
            return false;
        }
        Person p = (Person) other;
        return firstName.equals(p.firstName) &&
                lastName.equals(p.lastName);
    }

    public int hashcode() {
        return firstName.hashCode() + 31 * lastName.hashCode();
    }
}

Este código no se va a comportar como se esperaba. El problema es que los métodos equals y hashcode para Person no anulan los métodos estándar definidos por Object.

  • El método equals tiene la firma incorrecta. Debe declararse como equals(Object) no equals(String).
  • El método hashcode tiene un nombre incorrecto. Debería ser hashCode() (nota la C mayúscula).

Estos errores significan que hemos declarado sobrecargas accidentales, y no se usarán si Person se usa en un contexto polimórfico.

Sin embargo, hay una manera simple de lidiar con esto (desde Java 5 en adelante). Usa la anotación @Override siempre que pretendas que tu método sea una anulación:

public final class Person {
    ...

    @Override
    public boolean equals(String other) {
        ....
    }

    @Override
    public hashcode() {
        ....
    }
}

Cuando agregamos una anotación @Override a la declaración de un método, el compilador verificará que el método sobrescriba (o implemente) un método declarado en una superclase o interfaz. Entonces, en el ejemplo anterior, el compilador nos dará dos errores de compilación, lo que debería ser suficiente para alertarnos del error.

Peligro de desempaquetar automáticamente objetos nulos en primitivos #

public class Foobar {
    public static void main(String[] args) {

        // example: 
        Boolean ignore = null;
        if (ignore == false) {
            System.out.println("Do not ignore!");
        }
    }
}

La trampa aquí es que ’null’ se compara con ‘false’. Ya que estamos comparando un ‘booleano’ primitivo con un ‘booleano’, Java intenta desempaquetar el ‘objeto’ ‘booleano’ en un equivalente primitivo, listo para la comparación. Sin embargo, dado que ese valor es null, se lanza una NullPointerException.

Java es incapaz de comparar tipos primitivos con valores nulos, lo que provoca una NullPointerException en tiempo de ejecución. Considere el caso primitivo de la condición falso == nulo; esto generaría un error de tiempo de compilación tipos incomparables: int y <null>.
