Sintaxis #

  • java [ <opción> ... ] <nombre-clase> [ <argumento> ... ]

  • java [ <opt> ... ] -jar <jar-file-pathname> [ <argumento> ... ]

El comando java se utiliza para ejecutar una aplicación Java desde la línea de comandos. Está disponible como parte de cualquier Java SE JRE o JDK.

En los sistemas Windows existen dos variantes del comando java:

  • La variante java inicia la aplicación en una nueva ventana de consola.
  • La variante javaw inicia la aplicación sin crear una nueva ventana de consola.

En otros sistemas (por ejemplo, Linux, Mac OSX, UNIX) solo se proporciona el comando java y no abre una nueva ventana de consola.

El símbolo <opt> en la sintaxis denota una opción en la línea de comando java. Los temas “Opciones de Java” y “Opciones de tamaño de montón y pila” cubren las opciones más utilizadas. Otros están cubiertos en el tema https://www.wikiod.com/es/java/indicadores-jvm

Clases de punto de entrada #

Una clase de punto de entrada de Java tiene un método principal con la siguiente firma y modificadores:

public static void main(String[] args)

Nota al margen: debido a cómo funcionan las matrices, también puede ser (String args[])

Cuando el comando java inicia la máquina virtual, carga las clases de punto de entrada especificadas e intenta encontrar main. Si tiene éxito, los argumentos de la línea de comandos se convierten en objetos String de Java y se ensamblan en una matriz. Si main se invoca de esta manera, la matriz no será null y no contendrá ninguna entrada null.

Un método de clase de punto de entrada válido debe hacer lo siguiente:

  • Ser nombrado principal (distingue entre mayúsculas y minúsculas)
  • Ser público y estático
  • Tener un tipo de retorno vacío
  • Tener un solo argumento con una matriz String[]. El argumento debe estar presente y no se permite más de un argumento.
  • Ser genérico: no se permiten parámetros de tipo.
  • Tener una clase envolvente no genérica de nivel superior (no anidada ni interna)

Es convencional declarar la clase como pública pero esto no es estrictamente necesario. Desde Java 5 en adelante, el tipo de argumento del método main puede ser un varargs String en lugar de una matriz de cadenas. main puede opcionalmente lanzar excepciones, y su parámetro puede tener cualquier nombre, pero convencionalmente es args.

Puntos de entrada de JavaFX #

Desde Java 8 en adelante, el comando java también puede iniciar directamente una aplicación JavaFX. JavaFX está documentado en la etiqueta JavaFX, pero un punto de entrada de JavaFX debe hacer lo siguiente:

  • Ampliar javafx.application.Application
  • Ser público y no abstracto
  • No ser genérico ni anidado
  • Tener un constructor sin argumentos ‘público’ explícito o implícito

Ejecutar un archivo JAR ejecutable #

Los archivos JAR ejecutables son la forma más sencilla de ensamblar código Java en un solo archivo que se puede ejecutar. *(Nota editorial: la creación de archivos JAR debe tratarse en un tema aparte). *

Suponiendo que tiene un archivo JAR ejecutable con el nombre de ruta <jar-path>, debería poder ejecutarlo de la siguiente manera:

java -jar <jar-path>

Si el comando requiere argumentos de línea de comandos, agréguelos después de <jar-path>. Por ejemplo:

java -jar <jar-path> arg1 arg2 arg3

Si necesita proporcionar opciones de JVM adicionales en la línea de comando java, deben ir antes de la opción -jar. Tenga en cuenta que la opción -cp / -classpath se ignorará si usa -jar. La vía de acceso de clases de la aplicación está determinada por el manifiesto del archivo JAR.

Ejecutar aplicaciones Java a través de una clase “principal” #

Cuando una aplicación no ha sido empaquetada como un archivo JAR ejecutable, debe proporcionar el nombre de una clase de punto de entrada en la línea de comando java.

Ejecutando la clase HelloWorld #

El ejemplo de “HelloWorld” se describe en https://www.wikiod.com/es/java/primeros-pasos-con-el-lenguaje-java#Creando tu primer programa Java Consiste en una única clase llamada HelloWorld que cumple los requisitos para un punto de entrada.

Suponiendo que el archivo (compilado) “HelloWorld.class” está en el directorio actual, se puede iniciar de la siguiente manera:

java HelloWorld

Algunas cosas importantes a tener en cuenta son:

  • Debemos proporcionar el nombre de la clase: no la ruta del archivo “.class” o el archivo “.java”.
  • Si la clase se declara en un paquete (como ocurre con la mayoría de las clases de Java), entonces el nombre de la clase que proporcionamos al comando java debe ser el nombre completo de la clase. Por ejemplo, si SomeClass se declara en el paquete com.example, entonces el nombre completo de la clase será com.example.SomeClass.

Especificación de un classpath #

A menos que estemos usando la sintaxis del comando java -jar, el comando java busca la clase que se va a cargar buscando en el classpath; consulte https://www.wikiod.com/es/java/el-classpath El comando anterior se basa en que el classpath predeterminado sea (o incluya) el directorio actual. Podemos ser más explícitos al respecto especificando el classpath que se usará usando la opción -cp.

java -cp . HelloWorld

Esto dice que haga que el directorio actual (que es a lo que se refiere “.”) sea la única entrada en el classpath.

-cp es una opción procesada por el comando java. Todas las opciones destinadas al comando java deben estar antes del nombre de clase. Cualquier cosa después de la clase se tratará como un argumento de línea de comando para la aplicación Java, y se pasará a la aplicación en el String[] que se pasa al método main.

(Si no se proporciona la opción -cp, java usará la ruta de clase proporcionada por la variable de entorno CLASSPATH. Si esa variable no está configurada o está vacía, java usa “.” como la ruta de clase predeterminada. )

1: https://www.wikiod.com/es/java/el-comando-java—java-y-javaw#Clases de punto de entrada

Solución de problemas del comando ‘java’ #

Este ejemplo cubre errores comunes con el uso del comando ‘java’.

“Comando no encontrado” #

Si recibe un mensaje de error como:

java: command not found

al intentar ejecutar el comando java, esto significa que no hay ningún comando java en la ruta de búsqueda de comandos de su shell. La causa podría ser:

  • no tienes Java JRE o JDK instalado,
  • no ha actualizado la variable de entorno PATH (correctamente) en su archivo de inicialización de shell, o
  • no ha “obtenido” el archivo de inicialización relevante en el shell actual.

Consulte “Instalación de Java” para conocer los pasos que debe seguir.

“No se pudo encontrar o cargar la clase principal” #

Este mensaje de error lo emite el comando java si no ha podido encontrar/cargar la clase de punto de entrada que ha especificado. En términos generales, hay tres razones generales por las que esto puede suceder:

  • Ha especificado una clase de punto de entrada que no existe.
  • La clase existe, pero la has especificado incorrectamente.
  • La clase existe y la ha especificado correctamente, pero Java no puede encontrarla porque el classpath es incorrecto.

Aquí hay un procedimiento para diagnosticar y resolver el problema:

  1. Averigüe el nombre completo de la clase de punto de entrada.

    • If you have source code for a class, then the full name consists of the package name and the simple class name. The instance the “Main” class is declared in the package “com.example.myapp” then its full name is “com.example.myapp.Main”.
    • If you have a compiled class file, you can find the class name by running javap on it.
    • If the class file is in a directory, you can infer the full class name from the directory names.
    • If the class file is in a JAR or ZIP file, you can infer the full class name from the file path in the JAR or ZIP file.

  2. Mire el mensaje de error del comando java. El mensaje debe terminar con el nombre completo de la clase que java está tratando de usar.

    • Check that it exactly matches the full classname for the entry-point class.
    • It should not end with “.java” or “.class”.
    • It should not contain slashes or any other character that is not legal in a Java identifier1.
    • The casing of the name should exactly match the full class name.

  3. Si está utilizando el nombre de clase correcto, asegúrese de que la clase esté realmente en el classpath:

    • Work out the pathname that the classname maps to; see https://www.wikiod.com/es/java/el-classpath#Asignación de nombres de clase a nombres de ruta
    • Work out what the classpath is; see this example: https://www.wikiod.com/es/java/el-classpath#Diferentes formas de especificar el classpath
    • Look at each of the JAR and ZIP files on the classpath to see if they contain a class with the required pathname.
    • Look at each directory to see if the pathname resolves to a file within the directory.

Si al revisar el classpath a mano no se encuentra el problema, puede agregar las opciones -Xdiag y -XshowSettings. El primero enumera todas las clases que se cargan y el segundo imprime configuraciones que incluyen la ruta de clase efectiva para la JVM.

Finalmente, hay algunas causas oscuras para este problema:

  • Un archivo JAR ejecutable con un atributo Main-Class que especifica una clase que no existe.
  • Un archivo JAR ejecutable con un atributo Class-Path incorrecto.
  • Si confundes 2 las opciones antes del nombre de la clase, el comando java puede intentar interpretar una de ellas como el nombre de la clase.
  • Si alguien ignoró las reglas de estilo de Java y usó identificadores de clase o paquete que difieren solo en mayúsculas y minúsculas, y está ejecutando en una plataforma que trata las mayúsculas y minúsculas en los nombres de archivo como no significativos.
  • Problemas con los homoglifos en los nombres de las clases en el código o en la línea de comandos.

“Método principal no encontrado en la clase <nombre>” #

Este problema ocurre cuando el comando java puede encontrar y cargar la clase que nominó, pero luego no puede encontrar un método de punto de entrada.

Hay tres posibles explicaciones:

  • Si intenta ejecutar un archivo JAR ejecutable, el manifiesto del JAR tiene un atributo de “Clase principal” incorrecto que especifica una clase que no es una clase de punto de entrada válida.
  • Le ha dicho al comando java una clase que no es una clase de punto de entrada.
  • La clase del punto de entrada es incorrecta; consulte https://www.wikiod.com/es/java/el-comando-java—java-y-javaw#Clases de punto de entrada para obtener más información.

Ejecutar una aplicación Java con dependencias de biblioteca #

Esta es una continuación de los ejemplos “clase principal” y “JAR ejecutable”.

Las aplicaciones típicas de Java consisten en un código específico de la aplicación y varios códigos de biblioteca reutilizables que ha implementado o que han implementado terceros. Estos últimos se conocen comúnmente como dependencias de biblioteca y normalmente se empaquetan como archivos JAR.

Java es un lenguaje enlazado dinámicamente. Cuando ejecuta una aplicación Java con dependencias de biblioteca, la JVM necesita saber dónde están las dependencias para poder cargar las clases según sea necesario. En términos generales, hay dos maneras de lidiar con esto:

  • La aplicación y sus dependencias se pueden volver a empaquetar en un solo archivo JAR que contiene todas las clases y recursos necesarios.

  • Se le puede decir a la JVM dónde encontrar los archivos JAR dependientes a través de la ruta de clase en tiempo de ejecución.

Para un archivo JAR ejecutable, la classpath en tiempo de ejecución se especifica mediante el atributo de manifiesto “Class-Path”. (Nota editorial: esto debe describirse en un tema separado sobre el comando jar.) De lo contrario, la ruta de clase en tiempo de ejecución debe proporcionarse mediante la opción -cp o mediante la variable de entorno CLASSPATH.

Por ejemplo, supongamos que tenemos una aplicación Java en el archivo “myApp.jar” cuya clase de punto de entrada es com.example.MyApp. Supongamos también que la aplicación depende de los archivos JAR de la biblioteca “lib/library1.jar” y “lib/library2.jar”. Podríamos lanzar la aplicación usando el comando java de la siguiente manera en una línea de comando:

$ # Alternative 1 (preferred)
$ java -cp myApp.jar:lib/library1.jar:lib/library2.jar com.example.MyApp

$ # Alternative 2
$ export CLASSPATH=myApp.jar:lib/library1.jar:lib/library2.jar
$ java com.example.MyApp

(En Windows, usaría ; en lugar de : como separador de classpath, y establecería la variable (local) CLASSPATH usando set en lugar de export.)

Si bien un desarrollador de Java se sentiría cómodo con eso, no es “fácil de usar”. Por lo tanto, es una práctica común escribir un script de shell simple (o un archivo por lotes de Windows) para ocultar los detalles que el usuario no necesita conocer. Por ejemplo, si coloca el siguiente script de shell en un archivo llamado “myApp”, lo hace ejecutable y lo coloca en un directorio en la ruta de búsqueda de comandos:

#!/bin/bash
# The 'myApp' wrapper script

export DIR=/usr/libexec/myApp
export CLASSPATH=$DIR/myApp.jar:$DIR/lib/library1.jar:$DIR/lib/library2.jar
java com.example.MyApp

entonces podría ejecutarlo de la siguiente manera:

$ myApp arg1 arg2 ...

Cualquier argumento en la línea de comando se pasará a la aplicación Java a través de la expansión "$@". (Puede hacer algo similar con un archivo por lotes de Windows, aunque la sintaxis es diferente).

Espacios y otros caracteres especiales en los argumentos #

En primer lugar, el problema de manejar espacios en los argumentos NO es realmente un problema de Java. Más bien, es un problema que debe ser manejado por el shell de comandos que está utilizando cuando ejecuta un programa Java.

Como ejemplo, supongamos que tenemos el siguiente programa simple que imprime el tamaño de un archivo:

import java.io.File;

public class PrintFileSizes {
    
    public static void main(String[] args) {
        for (String name: args) {
            File file = new File(name);
            System.out.println("Size of '" + file + "' is " + file.size());
        }
    }
}

Ahora suponga que queremos imprimir el tamaño de un archivo cuyo nombre de ruta tiene espacios; p.ej. /home/steve/Archivo de prueba.txt. Si ejecutamos el comando así:

$ java PrintFileSizes /home/steve/Test File.txt

el shell no sabrá que /home/steve/Test File.txt es en realidad un nombre de ruta. En su lugar, pasará 2 argumentos distintos a la aplicación Java, que intentará encontrar sus respectivos tamaños de archivo y fallará porque los archivos con esas rutas (probablemente) no existen.

Soluciones que utilizan un shell POSIX #

Los shells POSIX incluyen sh y derivados como bash y ksh. Si está utilizando uno de estos shells, entonces puede resolver el problema citando el argumento.

$ java PrintFileSizes "/home/steve/Test File.txt"

Las comillas dobles alrededor del nombre de la ruta le dicen al shell que debe pasarse como un solo argumento. Las comillas se eliminarán cuando esto suceda. Hay un par de otras maneras de hacer esto:

$ java PrintFileSizes '/home/steve/Test File.txt'

Las comillas simples (rectas) se tratan como comillas dobles, excepto que también suprimen varias expansiones dentro del argumento.

$ java PrintFileSizes /home/steve/Test\ File.txt

Una barra invertida escapa del siguiente espacio y hace que no se interprete como un separador de argumentos.

Para obtener una documentación más completa, incluidas las descripciones de cómo manejar otros caracteres especiales en los argumentos, consulte el tema de citas en la documentación de Bash.

Solución para Windows #

El problema fundamental de Windows es que, a nivel del sistema operativo, los argumentos se pasan a un proceso secundario como una sola cadena (fuente). Esto significa que la responsabilidad final de analizar (o volver a analizar) la línea de comando recae en el programa o en sus bibliotecas de tiempo de ejecución. Hay mucha incoherencia.

En el caso de Java, para acortar una larga historia:

  • Puede poner comillas dobles alrededor de un argumento en un comando java, y eso le permitirá pasar argumentos con espacios en ellos.

  • Aparentemente, el comando java en sí está analizando la cadena de comando, y lo hace más o menos bien

  • Sin embargo, cuando intenta combinar esto con el uso de SET y la sustitución de variables en un archivo por lotes, se complica mucho si se eliminan las comillas dobles.

  • El shell cmd.exe aparentemente tiene otros mecanismos de escape; p.ej. duplicando comillas dobles y usando escapes ^.

Para obtener más detalles, consulte la documentación de Batch-File.

Opciones de Java #

El comando java admite una amplia gama de opciones:

  • Todas las opciones comienzan con un solo guión o signo menos (-): no se admite la convención de GNU/Linux de usar -- para opciones “largas”.

  • Las opciones deben aparecer antes del argumento <classname> o -jar <jarfile> para ser reconocidas. Todos los argumentos posteriores a ellos se tratarán como argumentos que se pasarán a la aplicación Java que se está ejecutando.

  • Las opciones que no comienzan con -X o -XX son opciones estándar. Puede confiar en todas las implementaciones de Java1 para admitir cualquier opción estándar.

  • Las opciones que comienzan con -X son opciones no estándar y pueden retirarse de una versión de Java a la siguiente.

  • Las opciones que comienzan con -XX son opciones avanzadas y también se pueden retirar.

Establecer las propiedades del sistema con -D #

La opción -D<propiedad>=<valor> se utiliza para establecer una propiedad en el objeto Propiedades del sistema. Este parámetro se puede repetir para establecer diferentes propiedades.

Opciones de memoria, pila y recolector de basura #

Las opciones principales para controlar los tamaños de montón y pila están documentadas en https://www.wikiod.com/es/java/gestion-de-memoria-java#Configuración de los tamaños Heap, PermGen y Stack (Nota editorial: las opciones del recolector de basura deben describirse en el mismo tema).

Activación y desactivación de aserciones #

Las opciones -ea y -da habilitan y deshabilitan respectivamente la comprobación de afirmación de Java:

  • Todas las comprobaciones de aserciones están deshabilitadas de forma predeterminada.
  • La opción -ea permite verificar todas las afirmaciones
  • -ea:<packagename>... habilita la verificación de aserciones en un paquete y todos los subpaquetes.
  • El -ea:<classname>... permite verificar las aserciones en una clase.
  • La opción -da deshabilita la verificación de todas las aserciones
  • -da:<packagename>... deshabilita la verificación de aserciones en un paquete y todos los subpaquetes.
  • El -da:<classname>... deshabilita la verificación de aserciones en una clase.
  • La opción -esa permite verificar todas las clases del sistema.
  • La opción -dsa deshabilita la verificación de todas las clases del sistema.

Las opciones se pueden combinar. Por ejemplo.

$ # Enable all assertion checking in non-system classes 
$ java -ea -dsa MyApp

$ # Enable assertions for all classes in a package except for one.
$ java -ea:com.wombat.fruitbat... -da:com.wombat.fruitbat.Brickbat MyApp

Tenga en cuenta que habilitar la verificación de aserciones puede alterar el comportamiento de una programación Java.

  • Es susceptible de ralentizar la aplicación en general.
  • Puede hacer que métodos específicos tarden más en ejecutarse, lo que podría cambiar la sincronización de los subprocesos en una aplicación de subprocesos múltiples.
  • Puede introducir relaciones fortuitas de sucede-antes que pueden hacer que las anomalías de la memoria desaparezcan.
  • Una declaración assert implementada incorrectamente podría tener efectos secundarios no deseados.

Selección del tipo de máquina virtual #

Las opciones -client y -server le permiten seleccionar entre dos formas diferentes de HotSpot VM:

  • El formulario de “cliente” se adapta a las aplicaciones de los usuarios y ofrece un inicio más rápido.
  • El formulario “servidor” está ajustado para aplicaciones de larga ejecución. Lleva más tiempo capturar estadísticas durante el “calentamiento” de JVM, lo que permite que el compilador JIT haga un mejor trabajo de optimización del código nativo.

De forma predeterminada, la JVM se ejecutará en modo de 64 bits si es posible, según las capacidades de la plataforma. Las opciones -d32 y -d64 le permiten seleccionar el modo explícitamente.
