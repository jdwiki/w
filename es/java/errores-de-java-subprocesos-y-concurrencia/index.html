Escollo: uso incorrecto de esperar() / notificar() #

Los métodos object.wait(), object.notify() y object.notifyAll() están destinados a ser utilizados de una manera muy específica. (ver https://www.wikiod.com/es/java)

El problema de la “Notificación perdida” #

Un error común de los principiantes es llamar incondicionalmente a object.wait()

private final Object lock = new Object();

public void myConsumer() {
    synchronized (lock) {
        lock.wait();     // DON'T DO THIS!!
    }
    doSomething();
}

La razón por la que esto es incorrecto es que depende de algún otro subproceso para llamar a lock.notify() o lock.notifyAll(), pero nada garantiza que el otro subproceso no haya realizado esa llamada antes del subproceso del consumidor llamado lock .esperar().

lock.notify() y lock.notifyAll() no hacen nada en absoluto si algún otro subproceso no está ya esperando la notificación. El hilo que llama a myConsumer() en este ejemplo se colgará para siempre si es demasiado tarde para recibir la notificación.

El error “Estado de monitor ilegal” #

Si llama a wait() o notify() en un objeto sin mantener el candado, entonces la JVM lanzará IllegalMonitorStateException.

public void myConsumer() {
    lock.wait();      // throws exception
    consume();
}

public void myProducer() {
    produce();
    lock.notify();    // throws exception
}

(El diseño para wait() / notify() requiere que se mantenga el bloqueo porque esto es necesario para evitar condiciones de carrera sistémicas. Si fuera posible llamar a wait() o notify() sin bloquear , entonces sería imposible implementar el caso de uso principal para estas primitivas: esperar a que ocurra una condición).

Esperar/notificar tiene un nivel demasiado bajo #

La mejor manera de evitar problemas con wait() y notify() es no usarlos. La mayoría de los problemas de sincronización se pueden resolver utilizando los objetos de sincronización de nivel superior (colas, barreras, semáforos, etc.) que están disponibles en el paquete java.utils.concurrent.

Trampa - Ampliación de ‘java.lang.Thread’ #

El javadoc para la clase Thread muestra dos formas de definir y usar un hilo:

Usando una clase de hilo personalizada:

 class PrimeThread extends Thread {
     long minPrime;
     PrimeThread(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeThread p = new PrimeThread(143);
 p.start();

Usando un Ejecutable:

 class PrimeRun implements Runnable {
     long minPrime;
     PrimeRun(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeRun p = new PrimeRun(143);
 new Thread(p).start();

(Fuente: java.lang.Thread javadoc.)

El enfoque de clase de hilo personalizado funciona, pero tiene algunos problemas:

  1. Es incómodo usar PrimeThread en un contexto que usa un grupo de subprocesos clásico, un ejecutor o el marco ForkJoin. (No es imposible, porque ‘PrimeThread’ implementa indirectamente ‘Runnable’, pero usar una clase ‘Thread’ personalizada como ‘Runnable’ es ciertamente torpe y puede no ser viable… dependiendo de otros aspectos de la clase).

  2. Hay más oportunidades de errores en otros métodos. Por ejemplo, si declaró PrimeThread.start() sin delegar a Thread.start(), terminaría con un “hilo” que se ejecuta en el hilo actual.

El enfoque de poner la lógica del subproceso en un ‘Ejecutable’ evita estos problemas. De hecho, si usa una clase anónima (Java 1.1 en adelante) para implementar Runnable, el resultado es más sucinto y más legible que los ejemplos anteriores.

 final long minPrime = ...
 new Thread(new Runnable() {
     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }.start();

Con una expresión lambda (Java 8 en adelante), el ejemplo anterior sería aún más elegante:

 final long minPrime = ...
 new Thread(() -> {
    // compute primes larger than minPrime
     . . .
 }).start();

Trampa: demasiados subprocesos hacen que una aplicación sea más lenta. #

Mucha gente que es nueva en multi-threading piensa que el uso de hilos automáticamente hace que una aplicación vaya más rápido. De hecho, es mucho más complicado que eso. Pero una cosa que podemos afirmar con certeza es que para cualquier computadora hay un límite en la cantidad de subprocesos que se pueden ejecutar al mismo tiempo:

  • Una computadora tiene un número fijo de núcleos (o hiperprocesos).
  • Un subproceso de Java debe programarse en un núcleo o hiperproceso para poder ejecutarse.
  • Si hay más subprocesos Java ejecutables que núcleos/hiperprocesos (disponibles), algunos de ellos deben esperar.

Esto nos dice que la simple creación de más y más subprocesos de Java no puede hacer que la aplicación vaya cada vez más rápido. Pero también hay otras consideraciones:

  • Cada subproceso requiere una región de memoria fuera del montón para su pila de subprocesos. El tamaño típico (predeterminado) de la pila de subprocesos es de 512 Kbytes o 1 Mbyte. Si tiene una cantidad significativa de subprocesos, el uso de la memoria puede ser significativo.

  • Cada subproceso activo se referirá a una cantidad de objetos en el montón. Eso aumenta el conjunto de trabajo de objetos accesibles, lo que afecta la recolección de basura y el uso de la memoria física.

  • Los gastos generales de cambiar entre subprocesos no son triviales. Por lo general, implica un cambio al espacio del kernel del sistema operativo para tomar una decisión de programación de subprocesos.

  • Los gastos generales de la sincronización de subprocesos y la señalización entre subprocesos (por ejemplo, wait (), notificar () / notificar a todos) * pueden ser * significativos.

Dependiendo de los detalles de su aplicación, estos factores generalmente significan que existe un “punto óptimo” para la cantidad de subprocesos. Más allá de eso, agregar más subprocesos brinda una mejora mínima del rendimiento y puede empeorar el rendimiento.

Si su aplicación crea para cada nueva tarea, un aumento inesperado en la carga de trabajo (por ejemplo, una alta tasa de solicitudes) puede provocar un comportamiento catastrófico.

Una mejor manera de lidiar con esto es usar un grupo de subprocesos limitados cuyo tamaño puede controlar (estática o dinámicamente). Cuando hay demasiado trabajo por hacer, la aplicación necesita poner en cola las solicitudes. Si usa un ExecutorService, se encargará de la administración del grupo de subprocesos y la cola de tareas.

Trampa: las variables compartidas requieren una sincronización adecuada #

Considere este ejemplo:

public class ThreadTest implements Runnable {
   
    private boolean stop = false;
    
    public void run() {
        long counter = 0;
        while (!stop) {
            counter = counter + 1;
        }
        System.out.println("Counted " + counter);
    }

    public static void main(String[] args) {
        ThreadTest tt = new ThreadTest();
        new Thread(tt).start();    // Create and start child thread
        Thread.sleep(1000);
        tt.stop = true;            // Tell child thread to stop.
    }
}

La intención de este programa es iniciar un hilo, dejar que se ejecute durante 1000 milisegundos y luego hacer que se detenga configurando el indicador stop.

¿Funcionará según lo previsto? #

Tal vez sí tal vez no.

Una aplicación no se detiene necesariamente cuando regresa el método main. Si se ha creado otro subproceso y ese subproceso no se ha marcado como un subproceso daemon, la aplicación continuará ejecutándose después de que finalice el subproceso principal. En este ejemplo, eso significa que la aplicación seguirá ejecutándose hasta que finalice el subproceso secundario. Eso debería suceder cuando tt.stop se establece en true.

Pero eso en realidad no es estrictamente cierto. De hecho, el subproceso secundario se detendrá después de haber observado stop con el valor true. ¿Eso sucederá? Tal vez sí tal vez no.

La especificación del lenguaje Java garantiza que las lecturas y escrituras de memoria realizadas en un subproceso son visibles para ese subproceso, según el orden de las declaraciones en el código fuente. Sin embargo, en general, esto NO está garantizado cuando un hilo escribe y otro hilo (posteriormente) lee. Para obtener visibilidad garantizada, debe haber una cadena de relaciones sucede-antes entre una escritura y una lectura posterior. En el ejemplo anterior, no existe tal cadena para la actualización del indicador stop y, por lo tanto, no se garantiza que el subproceso secundario vea stop cambiar a true.

(Nota para los autores: debe haber un tema separado sobre el modelo de memoria de Java para profundizar en los detalles técnicos).

¿Cómo solucionamos el problema? #

En este caso, hay dos formas sencillas de asegurarse de que la actualización stop esté visible:

  1. Declarar que stop sea volátil; es decir.

       private volatile boolean stop = false;

    For a volatile variable, the JLS specifies that there is a happens-before relation between a write by one thread and a later read by a second thread.

  2. Use un mutex para sincronizar de la siguiente manera:

    public class ThreadTest implements Runnable {

     private boolean stop = false;

 public void run() {
     long counter = 0;
     while (true) {
         synchronize (this) {
             if (stop) {
                 break;
             }
         }
         counter = counter + 1;
     }
     System.out.println("Counted " + counter);
 }

 public static void main(String[] args) {
     ThreadTest tt = new ThreadTest();
     new Thread(tt).start();    // Create and start child thread
     Thread.sleep(1000);
     synchronize (tt) {
         tt.stop = true;        // Tell child thread to stop.
     }
 }

    }

Además de garantizar que haya una exclusión mutua, el JLS especifica que existe una relación sucede antes entre la liberación de una exclusión mutua en un subproceso y la obtención de la misma exclusión mutua en un segundo subproceso.

¿Pero no es la asignación atómica? #

¡Sí, lo es!

Sin embargo, ese hecho no significa que los efectos de la actualización serán visibles simultáneamente para todos los subprocesos. Solo una cadena adecuada de relaciones sucede-antes lo garantizará.

¿Por qué hicieron esto? #

Los programadores que realizan programación de subprocesos múltiples en Java por primera vez encuentran que el modelo de memoria es un desafío. Los programas se comportan de forma poco intuitiva porque la expectativa natural es que las escrituras sean visibles uniformemente. Entonces, ¿por qué los diseñadores de Java diseñan el modelo de memoria de esta manera?

En realidad, todo se reduce a un compromiso entre el rendimiento y la facilidad de uso (para el programador).

Una arquitectura informática moderna consta de múltiples procesadores (núcleos) con conjuntos de registros individuales. La memoria principal es accesible para todos los procesadores o para grupos de procesadores. Otra propiedad del hardware informático moderno es que el acceso a los registros suele ser mucho más rápido que el acceso a la memoria principal. A medida que aumenta la cantidad de núcleos, es fácil ver que leer y escribir en la memoria principal puede convertirse en el principal cuello de botella del rendimiento de un sistema.

Este desajuste se soluciona implementando uno o más niveles de caché de memoria entre los núcleos del procesador y la memoria principal. Cada núcleo accede a las celdas de memoria a través de su caché. Normalmente, una lectura de la memoria principal solo ocurre cuando hay una pérdida de caché, y una escritura de la memoria principal solo ocurre cuando una línea de caché necesita vaciarse. Para una aplicación en la que el conjunto de ubicaciones de memoria de trabajo de cada núcleo se ajuste a su caché, la velocidad del núcleo ya no está limitada por la velocidad o el ancho de banda de la memoria principal.

Pero eso nos plantea un nuevo problema cuando varios núcleos leen y escriben variables compartidas. La última versión de una variable puede estar en el caché de un núcleo. A menos que ese núcleo vacíe la línea de caché a la memoria principal, Y otros núcleos invaliden su copia en caché de versiones anteriores, algunos de ellos pueden ver versiones obsoletas de la variable. Pero si los cachés se vaciaran en la memoria cada vez que hay una escritura de caché (“por si acaso” hubo una lectura por parte de otro núcleo), eso consumiría el ancho de banda de la memoria principal innecesariamente.

La solución estándar utilizada en el nivel del conjunto de instrucciones de hardware es proporcionar instrucciones para la invalidación de caché y una escritura simultánea de caché, y dejar que el compilador decida cuándo usarlas.

Volviendo a Java. el modelo de memoria está diseñado para que los compiladores de Java no tengan que emitir instrucciones de invalidación de caché y escritura simultánea donde realmente no se necesitan. La suposición es que el programador utilizará un mecanismo de sincronización apropiado (por ejemplo, mutexes primitivos, “volátiles”, clases de concurrencia de nivel superior, etc.) para indicar que necesita visibilidad de la memoria. En ausencia de una relación sucede-antes, los compiladores de Java son libres de asumir que no se requieren operaciones de caché (o similares).

Esto tiene importantes ventajas de rendimiento para las aplicaciones de subprocesos múltiples, pero la desventaja es que escribir aplicaciones de subprocesos múltiples correctas no es un asunto sencillo. El programador tiene que entender lo que está haciendo.

¿Por qué no puedo reproducir esto? #

Hay una serie de razones por las que problemas como este son difíciles de reproducir:

  1. Como se explicó anteriormente, la consecuencia de no tratar adecuadamente los problemas de visibilidad de la memoria es típicamente que su aplicación compilada no maneja los cachés de memoria correctamente. Sin embargo, como mencionamos anteriormente, los cachés de memoria a menudo se vacían de todos modos.

  2. Cuando cambia la plataforma de hardware, las características de los cachés de memoria pueden cambiar. Esto puede generar un comportamiento diferente si su aplicación no se sincroniza correctamente.

  3. Es posible que esté observando los efectos de la sincronización afortunada. Por ejemplo, si agrega huellas de rastreo, generalmente hay alguna sincronización detrás de escena en los flujos de E/S que provoca vaciados de caché. Por lo tanto, agregar rastros a menudo hace que la aplicación se comporte de manera diferente.

  4. La ejecución de una aplicación con un depurador hace que el compilador JIT la compile de forma diferente. Los puntos de interrupción y los pasos simples exacerban esto. Estos efectos a menudo cambiarán la forma en que se comporta una aplicación.

Estas cosas hacen que los errores que se deben a una sincronización inadecuada sean particularmente difíciles de resolver.
