El modelo de memoria de Java es la sección de JLS que especifica las condiciones bajo las cuales se garantiza que un subproceso vea los efectos de las escrituras de memoria realizadas por otro subproceso. La sección relevante en ediciones recientes es “Modelo de memoria JLS 17.4” (en Java 8, Java 7, Java 6)

Hubo una revisión importante del modelo de memoria de Java en Java 5 que (entre otras cosas) cambió la forma en que funcionaba “volátil”. Desde entonces, el modelo de memoria se ha mantenido esencialmente sin cambios.

Motivación para el Modelo de Memoria #

Considere el siguiente ejemplo:

public class Example {
    public int a, b, c, d;
    
    public void doIt() {
       a = b + 1;
       c = d + 1;
    }
}

Si esta clase se usa como una aplicación de subproceso único, entonces el comportamiento observable será exactamente como cabría esperar. Por ejemplo:

public class SingleThreaded {
    public static void main(String[] args) {
        Example eg = new Example();
        System.out.println(eg.a + ", " + eg.c);
        eg.doIt();
        System.out.println(eg.a + ", " + eg.c);
    }
}

generará:

0, 0
1, 1

En lo que respecta al subproceso “principal”, las declaraciones en el método main() y doIt() se ejecutarán en el orden en que están escritas en el código fuente. Este es un requisito claro de la especificación del lenguaje Java (JLS).

Ahora considere la misma clase utilizada en una aplicación de subprocesos múltiples.

public class MultiThreaded {
    public static void main(String[] args) {
        final Example eg = new Example();
        new Thread(new Runnable() {
            public void run() {
                while (true) {
                    eg.doIt();
                }
            }
        }).start();
        while (true) {
            System.out.println(eg.a + ", " + eg.c);
        }
    }
}

¿Qué imprimirá esto?

De hecho, según el JLS, no es posible predecir que esto imprimirá:

  • Probablemente verá algunas líneas de 0, 0 para empezar.
  • Entonces probablemente vea líneas como N, N o N, N + 1.
  • Es posible que vea líneas como N + 1, N.
  • En teoría, incluso podrías ver que las líneas 0, 0 continúan para siempre1.

1 - En la práctica, la presencia de las sentencias println es probable que cause alguna sincronización fortuita y vaciado de caché de memoria. Es probable que eso oculte algunos de los efectos que causarían el comportamiento anterior.

Entonces, ¿cómo podemos explicar esto?

Reordenación de tareas #

Una posible explicación para los resultados inesperados es que el compilador JIT haya cambiado el orden de las asignaciones en el método doIt(). El JLS requiere que las declaraciones parecen ejecutarse en orden desde la perspectiva del hilo actual. En este caso, nada en el código del método doIt() puede observar el efecto de un (hipotético) reordenamiento de esas dos declaraciones. Esto significa que el compilador JIT podría hacer eso.

¿Porqué haría eso?

En el hardware moderno típico, las instrucciones de la máquina se ejecutan utilizando un canal de instrucciones que permite que una secuencia de instrucciones esté en diferentes etapas. Algunas fases de la ejecución de instrucciones toman más tiempo que otras y las operaciones de memoria tienden a tomar más tiempo. Un compilador inteligente puede optimizar el rendimiento de instrucciones de la canalización al ordenar las instrucciones para maximizar la cantidad de superposición. Esto puede conducir a la ejecución de partes de declaraciones fuera de orden. El JLS lo permite siempre que no afecte el resultado del cálculo desde la perspectiva del hilo actual.

Efectos de las cachés de memoria #

Una segunda explicación posible es el efecto del almacenamiento en caché de la memoria. En una arquitectura de computadora clásica, cada procesador tiene un pequeño conjunto de registros y una mayor cantidad de memoria. El acceso a los registros es mucho más rápido que el acceso a la memoria principal. En las arquitecturas modernas, existen cachés de memoria que son más lentos que los registros, pero más rápidos que la memoria principal.

Un compilador aprovechará esto tratando de mantener copias de las variables en los registros o en los cachés de memoria. Si una variable no necesita ser vaciada a la memoria principal, o no necesita ser leída de la memoria, existen importantes beneficios de rendimiento al no hacerlo. En los casos en que JLS no requiera que las operaciones de memoria sean visibles para otro subproceso, es probable que el compilador Java JIT no agregue las instrucciones de “barrera de lectura” y “barrera de escritura” que forzarán las lecturas y escrituras de la memoria principal. Una vez más, los beneficios de rendimiento de hacer esto son significativos.

Sincronización adecuada #

Hasta ahora, hemos visto que JLS permite que el compilador JIT genere código que hace que el código de subproceso único sea más rápido al reordenar o evitar las operaciones de memoria. Pero, ¿qué sucede cuando otros subprocesos pueden observar el estado de las variables (compartidas) en la memoria principal?

La respuesta es que los otros subprocesos pueden observar estados variables que parecerían imposibles… según el orden del código de las sentencias de Java. La solución a esto es utilizar la sincronización adecuada. Los tres enfoques principales son:

  • Uso de mutex primitivos y construcciones sincronizadas.
  • Uso de variables volátiles.
  • Uso de soporte de concurrencia de nivel superior; p.ej. clases en los paquetes java.util.concurrent.

Pero incluso con esto, es importante comprender dónde se necesita sincronización y en qué efectos puede confiar. Aquí es donde entra en juego el modelo de memoria de Java.

El modelo de memoria #

El modelo de memoria de Java es la sección de JLS que especifica las condiciones bajo las cuales se garantiza que un subproceso vea los efectos de las escrituras de memoria realizadas por otro subproceso. El modelo de memoria se especifica con un grado razonable de rigor formal y (como resultado) requiere una lectura detallada y cuidadosa para comprenderlo. Pero el principio básico es que ciertas construcciones crean una relación de “sucede antes” entre la escritura de una variable por un subproceso y una lectura posterior de la misma variable por otro subproceso. Si existe la relación “sucede antes”, el compilador JIT está obligado a generar código que garantice que la operación de lectura vea el valor escrito por la escritura.

Armado con esto, es posible razonar sobre la coherencia de la memoria en un programa Java y decidir si esto será predecible y consistente para todas las plataformas de ejecución.

Ocurre antes de las relaciones #

(La siguiente es una versión simplificada de lo que dice la especificación del lenguaje Java. Para una comprensión más profunda, debe leer la especificación en sí).

Las relaciones que suceden antes son la parte del modelo de memoria que nos permite comprender y razonar sobre la visibilidad de la memoria. Como dice el JLS (JLS 17.4.5):

“Dos acciones se pueden ordenar mediante una relación sucede-antes. Si una acción sucede-antes de otra, entonces la primera es visible y se ordena antes que la segunda”.

¿Qué significa esto?

Comportamiento #

Las acciones a las que se refiere la cita anterior se especifican en JLS 17.4.2. Hay 5 tipos de acciones enumeradas definidas por la especificación:

  • Leer: Lectura de una variable no volátil.

  • Write: Escritura de una variable no volátil.

  • Acciones de sincronización:

    • Volatile read: Reading a volatile variable.

    • Volatile write: Writing a volatile variable.

    • Lock. Locking a monitor

    • Unlock. Unlocking a monitor.

    • The (synthetic) first and last actions of a thread.

    • Actions that start a thread or detect that a thread has terminated.

  • Acciones Externas. Una acción que tiene un resultado que depende del entorno en el que se desarrolla el programa.

  • Acciones de divergencia de hilos. Estos modelan el comportamiento de ciertos tipos de bucles infinitos.

Orden de programa y orden de sincronización #

Estos dos ordenamientos ( JLS 17.4.3 y JLS 17.4.4 ) gobiernan la ejecución de declaraciones en un Java

El orden del programa describe el orden de ejecución de la declaración dentro de un solo hilo.

El orden de sincronización describe el orden de ejecución de sentencias para dos sentencias conectadas por una sincronización:

  • Una acción de desbloqueo en el monitor se sincroniza con todas las acciones de bloqueo subsiguientes en ese monitor.

  • Una escritura en una variable volátil sincroniza-con todas las lecturas posteriores de la misma variable por cualquier subproceso.

  • Una acción que inicia un subproceso (es decir, la llamada a Thread.start()) sincroniza-con la primera acción en el subproceso que inicia (es decir, la llamada al método run() del subproceso).

  • La inicialización predeterminada de los campos sincroniza-con la primera acción en cada subproceso. (Consulte el JLS para obtener una explicación de esto).

  • La acción final en un hilo sincroniza-con cualquier acción en otro hilo que detecta la terminación; p.ej. el retorno de una llamada join() o llamada isTerpressed() que devuelve true.

  • Si un subproceso interrumpe a otro subproceso, la llamada de interrupción en el primer subproceso sincroniza-con el punto donde otro subproceso detecta que el subproceso fue interrumpido.

Sucede antes del pedido #

Este orden ( JLS 17.4.5 ) es lo que determina si se garantiza que una escritura de memoria sea visible para una lectura de memoria posterior.

Más específicamente, se garantiza que una lectura de una variable v observará una escritura en v si y solo si write(v) ocurre antes read(v) Y no hay una escritura intermedia en v'. Si hay escrituras intermedias, entoncesread(v)` puede ver los resultados de ellas en lugar de la anterior.

Las reglas que definen el orden sucede-antes son las siguientes:

  • Ocurre antes de la regla n.º 1: si x e y son acciones del mismo subproceso y x viene antes que y en el orden del programa, entonces x sucede-antes de y.

  • Sucede-antes de la regla n.° 2: hay un borde que sucede antes desde el final de un constructor de un objeto hasta el comienzo de un finalizador para ese objeto.

  • Ocurre antes de la regla #3 - Si una acción x se sincroniza-con una acción subsiguiente y, entonces x ocurre-antes de y.

  • Sucede-antes de la regla #4 - Si x sucede-antes de y e y sucede-antes de z entonces x sucede-antes de z.

Además, varias clases en las bibliotecas estándar de Java se especifican para definir relaciones sucede-antes. Puede interpretar esto como que sucede de alguna manera, sin necesidad de saber exactamente cómo se cumplirá la garantía.

Sucede-antes del razonamiento aplicado a algunos ejemplos #

Presentaremos algunos ejemplos para mostrar cómo aplicar el razonamiento sucede antes para verificar que las escrituras sean visibles para las lecturas posteriores.

Código de un solo subproceso #

Como era de esperar, las escrituras siempre son visibles para las lecturas posteriores en un programa de un solo subproceso.

public class SingleThreadExample {
    public int a, b;
    
    public int add() {
       a = 1;         // write(a)
       b = 2;         // write(b)
       return a + b;  // read(a) followed by read(b)
    }
}

Por Sucede-Antes de la Regla #1:

  1. La acción escribir(a) ocurre-antes de la acción escribir(b).
  2. La acción escribir(b) ocurre-antes de la acción leer(a).
  3. La acción leer(a) ocurre-antes de la acción leer(a).

Por Sucede-Antes de la Regla #4:

  1. write(a) sucede-antes write(b) AND write(b) sucede-antes read(a) IMPLIES write(a) sucede-antes leer(a).
  2. write(b) sucede-antes read(a) AND read(a) sucede-antes read(b) IMPLIES write(b) sucede-antes leer (b).

Resumiendo:

  1. La relación write(a) sucede-antes de read(a) significa que la declaración a + b está garantizada para ver el valor correcto de a.
  2. La relación write(b) sucede-antes read(b) significa que se garantiza que la instrucción a + b verá el valor correcto de b.

Comportamiento de ‘volátil’ en un ejemplo con 2 hilos

Usaremos el siguiente código de ejemplo para explorar algunas implicaciones del modelo de memoria para `volatile.

public class VolatileExample {
    private volatile int a;
    private int b;         // NOT volatile
    
    public void update(int first, int second) {
       b = first;         // write(b)
       a = second;         // write-volatile(a)
    }

    public int observe() {
       return a + b;       // read-volatile(a) followed by read(b)
    }
}

Primero, considere la siguiente secuencia de declaraciones que involucran 2 subprocesos:

  1. Se crea una sola instancia de VolatileExample; llámalo ve,
  2. ve.update(1, 2) se llama en un hilo, y
  3. Se llama a ve.observe() en otro hilo.

Por Sucede-Antes de la Regla #1:

  1. La acción write(a) sucede-antes de la acción volatile-write(a).
  2. La acción volatile-read(a) sucede-antes de la acción read(b).

Por Sucede-Antes de la Regla #2:

  1. La acción volatile-write(a) en el primer subproceso sucede-antes de la acción volatile-read(a) en el segundo subproceso.

Por Sucede-Antes de la Regla #4:

  1. La acción write(b) en el primer subproceso ocurre antes que la acción read(b) en el segundo subproceso.

En otras palabras, para esta secuencia en particular, tenemos la garantía de que el segundo subproceso verá la actualización de la variable no volátil b realizada por el primer subproceso. Sin embargo, también debe quedar claro que si las asignaciones en el método update fueran al revés, o si el método observe() leyera la variable b antes de a, entonces sucede-antes la cadena se rompería. La cadena también se rompería si volatile-read(a) en el segundo subproceso no fuera posterior a volatile-write(a) en el primer subproceso.

Cuando la cadena se rompe, no hay garantía de que observe() vea el valor correcto de b.

Volátil con tres hilos #

Supongamos que añadimos un tercer hilo al ejemplo anterior:

  1. Se crea una sola instancia de VolatileExample; llámalo ve,
  2. Dos hilos llaman a update:
    • ve.update(1, 2) is called in one thread,
    • ve.update(3, 4) is called in the second thread,
  3. ve.observe() se llama posteriormente en un tercer subproceso.

Para analizar esto completamente, necesitamos considerar todas las intercalaciones posibles de las declaraciones en el hilo uno y el hilo dos. En cambio, consideraremos solo dos de ellos.

Escenario n.° 1: supongamos que update(1, 2) precede a update(3,4) y obtenemos esta secuencia:

write(b, 1), write-volatile(a, 2)     // first thread
write(b, 3), write-volatile(a, 4)     // second thread
read-volatile(a), read(b)             // third thread

En este caso, es fácil ver que hay una cadena ininterrumpida de sucede-antes desde escribir(b, 3) hasta leer(b). Además, no hay una escritura intermedia en b. Entonces, para este escenario, se garantiza que el tercer subproceso verá que b tiene el valor 3.

Escenario n.° 2: suponga que update(1, 2) y update(3,4) se superponen y las acciones se intercalan de la siguiente manera:

write(b, 3)                           // second thread
write(b, 1)                           // first thread
write-volatile(a, 2)                  // first thread
write-volatile(a, 4)                  // second thread
read-volatile(a), read(b)             // third thread

Ahora, mientras hay una cadena sucede-antes de escribir(b, 3) a leer(b), hay una acción intermedia escribir(b, 1) realizada por el otro subproceso. Esto significa que no podemos estar seguros de qué valor verá read(b).

(Aparte: esto demuestra que no podemos confiar en “volátiles” para garantizar la visibilidad de las variables no volátiles, excepto en situaciones muy limitadas).

Cómo evitar la necesidad de entender el modelo de memoria #

El modelo de memoria es difícil de entender y difícil de aplicar. Es útil si necesita razonar sobre la corrección del código de subprocesos múltiples, pero no desea tener que hacer este razonamiento para cada aplicación de subprocesos múltiples que escriba.

Si adopta los siguientes principios al escribir código concurrente en Java, puede en gran medida evitar la necesidad de recurrir al razonamiento sucede antes.

  • Utilice estructuras de datos inmutables siempre que sea posible. Una clase inmutable implementada correctamente será segura para subprocesos y no presentará problemas de seguridad de subprocesos cuando la use con otras clases.

  • Comprender y evitar la “publicación insegura”.

  • Use mutexes primitivos u objetos Lock para sincronizar el acceso al estado en objetos mutables que deben ser seguros para subprocesos1.

  • Use Executor / ExecutorService o el marco de unión de bifurcación en lugar de intentar crear subprocesos de gestión directamente.

  • Use las clases `java.util.concurrent que proporcionan bloqueos, semáforos, pestillos y barreras avanzados, en lugar de usar wait/notify/notifyAll directamente.

  • Utilice las versiones java.util.concurrent de mapas, conjuntos, listas, colas y deques en lugar de la sincronización externa de colecciones no simultáneas.

El principio general es tratar de utilizar las bibliotecas de concurrencia integradas de Java en lugar de “hacer rodar su propia” concurrencia. Puede confiar en que funcionen, si los usa correctamente.

1: no todos los objetos deben ser seguros para subprocesos. Por ejemplo, si un objeto u objetos están confinados por subprocesos (es decir, solo es accesible para un subproceso), entonces su seguridad de subprocesos no es relevante.
