La interfaz Executor en Java proporciona una forma de desvincular el envío de tareas de la mecánica de cómo se ejecutará cada tarea, incluidos los detalles del uso de subprocesos, la programación, etc. Normalmente se usa un Executor en lugar de crear subprocesos explícitamente. Con Executors, los desarrolladores no tendrán que reescribir significativamente su código para poder ajustar fácilmente la política de ejecución de tareas de su programa.

Trampas

  • Cuando programa una tarea para su ejecución repetida, dependiendo del ScheduledExecutorService utilizado, su tarea puede suspenderse de cualquier ejecución posterior, si una ejecución de su tarea provoca una excepción que no se maneja. Ver ¡Mother F**k the ScheduledExecutorService!

ThreadPoolExecutor #

Un ejecutor común utilizado es ThreadPoolExecutor, que se encarga del manejo de subprocesos. Puede configurar la cantidad mínima de subprocesos que el ejecutor siempre debe mantener cuando no hay mucho que hacer (se denomina tamaño central) y un tamaño máximo de subprocesos al que puede crecer el grupo, si hay más trabajo por hacer. Una vez que la carga de trabajo disminuye, el grupo reduce lentamente el recuento de subprocesos nuevamente hasta que alcanza el tamaño mínimo.

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    1,                                     // keep at least one thread ready, 
                                           // even if no Runnables are executed
    5,                                     // at most five Runnables/Threads
                                           // executed in parallel
    1, TimeUnit.MINUTES,                   // idle Threads terminated after one
                                           // minute, when min Pool size exceeded
    new ArrayBlockingQueue<Runnable>(10)); // outstanding Runnables are kept here

pool.execute(new Runnable() {
    @Override public void run() {
        //code to run
    }
});

Nota Si configura ThreadPoolExecutor con una cola ilimitada, entonces el conteo de subprocesos no excederá corePoolSize ya que los nuevos subprocesos solo se crean si la cola está llena:

ThreadPoolExecutor con todos los parámetros:

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

de JavaDoc

Si hay más subprocesos que corePoolSize pero menos que maximumPoolSize en ejecución, se creará un nuevo subproceso solo si la cola está llena.

ventajas:

  1. El tamaño de BlockingQueue se puede controlar y se pueden evitar los escenarios de falta de memoria. El rendimiento de la aplicación no se degradará con un tamaño de cola limitado.

  2. Puede utilizar políticas existentes o crear nuevas políticas de manejo de rechazos.

  3. En el ThreadPoolExecutor.AbortPolicy predeterminado, el controlador genera una excepción RejectedExecutionException en tiempo de ejecución tras el rechazo.

  4. En ThreadPoolExecutor.CallerRunsPolicy, el subproceso que invoca la ejecución ejecuta la tarea. Esto proporciona un mecanismo de control de retroalimentación simple que reducirá la velocidad con la que se envían nuevas tareas.

  5. En ThreadPoolExecutor.DiscardPolicy, una tarea que no se puede ejecutar simplemente se elimina.

  6. En ThreadPoolExecutor.DiscardOldestPolicy, si el ejecutor no se apaga, la tarea que se encuentra al principio de la cola de trabajo se descarta y luego se vuelve a intentar la ejecución (lo que puede fallar nuevamente y hacer que esto se repita).

  7. Se puede configurar ThreadFactory personalizado, lo cual es útil:

  8. Para establecer un nombre de hilo más descriptivo

  9. Para establecer el estado del demonio de subprocesos

  10. Para establecer la prioridad del hilo

Aquí es un ejemplo de cómo usar ThreadPoolExecutor

Recuperación del valor del cálculo - Invocable #

Si su cálculo produce algún valor de retorno que luego se requiere, una tarea Runnable simple no es suficiente. Para tales casos, puede usar ExecutorService.submit([Callable][callable-and-future]<T>) que devuelve un valor después de que se completa la ejecución.

El Servicio devolverá un [Future][callable-and-future] que puede usar para recuperar el resultado de la ejecución de la tarea.

// Submit a callable for execution
ExecutorService pool = anExecutorService;
Future<Integer> future = pool.submit(new Callable<Integer>() {
    @Override public Integer call() {
        //do some computation
        return new Random().nextInt();
    }
});    
// ... perform other tasks while future is executed in a different thread

Cuando necesite obtener el resultado del futuro, llame a future.get()

  • Esperar indefinidamente a futuro para terminar con un resultado.

      try {
      // Blocks current thread until future is completed
      Integer result = future.get(); 
  catch (InterruptedException || ExecutionException e) {
      // handle appropriately
  }

  • Espere a que finalice el futuro, pero no más del tiempo especificado.

      try {
      // Blocks current thread for a maximum of 500 milliseconds.
      // If the future finishes before that, result is returned,
      // otherwise TimeoutException is thrown.
      Integer result = future.get(500, TimeUnit.MILLISECONDS); 
  catch (InterruptedException || ExecutionException || TimeoutException e) {
      // handle appropriately
  }

Si ya no se necesita el resultado de una tarea programada o en ejecución, puede llamar a Future.cancel(boolean) para cancelarla.

  • Llamar a cancelar (falso) simplemente eliminará la tarea de la cola de tareas que se ejecutarán.
  • Llamar a cancel(true) también interrumpirá la tarea si se está ejecutando actualmente.

[llamable y futuro]: https://www.wikiod.com/es/java/programacion-concurrente-hilos#Llamable y Futuro

presentar () vs ejecutar () diferencias en el manejo de excepciones #

En general, el comando de ejecución () se usa para disparar y olvidar llamadas (sin necesidad de analizar el resultado) y el comando enviar () se usa para analizar el resultado del objeto Futuro.

Debemos ser conscientes de la diferencia clave de los mecanismos de manejo de excepciones entre estos dos comandos.

Las excepciones de submit() son tragadas por el marco si no las atrapó.

Ejemplo de código para entender la diferencia:

        for (int i = 0; i < 2; i++){
            service.execute(new Runnable(){
                 public void run(){
                    int a = 4, b = 0;
                    System.out.println("a and b=" + a + ":" + b);
                    System.out.println("a/b:" + (a / b));
                    System.out.println("Thread Name in Runnable after divide by zero:"+Thread.currentThread().getName());
                 }
            });
        }
        service.shutdown();
    }
    public static void main(String args[]){
        ExecuteSubmitDemo demo = new ExecuteSubmitDemo();
    }
}

class ExtendedExecutor extends ThreadPoolExecutor {

   public ExtendedExecutor() { 
       super(1, 1, 60, TimeUnit.SECONDS, new ArrayBlockingQueue<Runnable>(100));
   }
   // ...
   protected void afterExecute(Runnable r, Throwable t) {
     super.afterExecute(r, t);
     if (t == null && r instanceof Future<?>) {

           t = ce;
       } catch (ExecutionException ee) {
           t = ee.getCause();
       } catch (InterruptedException ie) {
           Thread.currentThread().interrupt(); // ignore/reset
       }
     }
     if (t != null)
       System.out.println(t);
   }
 }

Fire and Forget - Tareas ejecutables #

Los ejecutores aceptan un java.lang.Runnable que contiene código (potencialmente computacional o de larga duración o pesado) para ejecutar en otro subproceso.

El uso sería:

Executor exec = anExecutor;
exec.execute(new Runnable() {
    @Override public void run() {
        //offloaded work, no need to get result back
    }
});

Tenga en cuenta que con este ejecutor, no tiene forma de recuperar ningún valor calculado. Con Java 8, se pueden utilizar lambdas para acortar el código de ejemplo.

Executor exec = anExecutor;
exec.execute(() -> {
    //offloaded work, no need to get result back
});

Manejar ejecución rechazada #

Si

  1. intenta enviar tareas a un Ejecutor de apagado o
  2. la cola está saturada (solo es posible con acotados) y se ha alcanzado el número máximo de subprocesos,

Se llamará a RejectedExecutionHandler.rejectedExecution(Runnable, ThreadPoolExecutor).

El comportamiento predeterminado es que obtendrá una RejectedExecutionException arrojada a la persona que llama. Pero hay más comportamientos predefinidos disponibles:

  • ThreadPoolExecutor.AbortPolicy (predeterminado, lanzará REE)
  • ThreadPoolExecutor.CallerRunsPolicy (ejecuta la tarea en el hilo de la persona que llama - bloqueándolo)
  • ThreadPoolExecutor.DiscardPolicy (descartar tarea silenciosamente)
  • ThreadPoolExecutor.DiscardOldestPolicy (descartar silenciosamente la tarea más antigua en la cola y volver a intentar la ejecución de la nueva tarea)

Puede configurarlos usando uno de los [constructores] 1 de ThreadPool:

public ThreadPoolExecutor(int corePoolSize,
                      int maximumPoolSize,
                      long keepAliveTime,
                      TimeUnit unit,
                      BlockingQueue<Runnable> workQueue,
                      RejectedExecutionHandler handler) // <--

public ThreadPoolExecutor(int corePoolSize,
                      int maximumPoolSize,
                      long keepAliveTime,
                      TimeUnit unit,
                      BlockingQueue<Runnable> workQueue,
                      ThreadFactory threadFactory,
                      RejectedExecutionHandler handler) // <--

También puede implementar su propio comportamiento extendiendo la interfaz RejectedExecutionHandler:

void rejectedExecution(Runnable r, ThreadPoolExecutor executor)

Casos de uso para diferentes tipos de construcciones de concurrencia #

  1. ServicioEjecutor

ExecutorService ejecutor = Executors.newFixedThreadPool(50);

Es simple y fácil de usar. Oculta detalles de bajo nivel de ThreadPoolExecutor.

Prefiero este cuando el número de tareas Callable/Runnable es pequeño y la acumulación de tareas en una cola ilimitada no aumenta la memoria ni degrada el rendimiento del sistema. Si tiene restricciones de ‘CPU/Memoria’, prefiero usar ‘ThreadPoolExecutor’ con restricciones de capacidad y ‘RejectedExecutionHandler’ para manejar el rechazo de tareas.

  1. Contador regresivo

CountDownLatch se inicializará con un conteo dado. Este recuento se reduce mediante llamadas al método countDown(). Los subprocesos que esperan que este recuento llegue a cero pueden llamar a uno de los métodos await(). Llamar a await() bloquea el hilo hasta que el conteo llega a cero. Esta clase permite que un subproceso java espere hasta que otro conjunto de subprocesos complete sus tareas.

Casos de uso:

  1. Lograr el máximo paralelismo: a veces queremos iniciar varios subprocesos al mismo tiempo para lograr el máximo paralelismo

  2. Espere N subprocesos para completar antes de iniciar la ejecución

  3. Detección de interbloqueo.

  4. ThreadPoolExecutor: proporciona más control. Si la aplicación está restringida por la cantidad de tareas ejecutables/llamables pendientes, puede usar la cola limitada configurando la capacidad máxima. Una vez que la cola alcanza la capacidad máxima, puede definir RejectionHandler. Java proporciona cuatro tipos de RejectedExecutionHandler políticas.

      1. `ThreadPoolExecutor.AbortPolicy`, the handler throws a runtime RejectedExecutionException upon rejection.

    1. ThreadPoolExecutor.CallerRunsPolicy`, the thread that invokes execute itself runs the task. This provides a simple feedback control mechanism that will slow down the rate that new tasks are submitted.

    2. In ThreadPoolExecutor.DiscardPolicy, a task that cannot be executed is simply dropped.

    3. ThreadPoolExecutor.DiscardOldestPolicy, if the executor is not shut down, the task at the head of the work queue is dropped, and then execution is retried (which can fail again, causing this to be repeated.)

Si desea simular el comportamiento de CountDownLatch, puede usar el método invokeAll().

  1. Otro mecanismo que no citaste es ForkJoinPool

ForkJoinPool se agregó a Java en Java 7. ForkJoinPool es similar a Java ExecutorService pero con una diferencia. El ForkJoinPool lo hace fácil para las tareas dividir su trabajo en tareas más pequeñas que luego son enviado a ForkJoinPool también. El robo de tareas ocurre en ForkJoinPool cuando los subprocesos de trabajo libres roban tareas de la cola de subprocesos de trabajo ocupada.

Java 8 ha introducido una API más en ExecutorService para crear un grupo de robo de trabajo. No tienes que crear RecursiveTask y RecursiveAction pero aún puedes usar ForkJoinPool.

   public static ExecutorService newWorkStealingPool()

Crea un grupo de subprocesos que roban trabajo usando todos los procesadores disponibles como su nivel de paralelismo de destino.

Por defecto, tomará el número de núcleos de CPU como parámetro.

Todos estos cuatro mecanismos son complementarios entre sí. Dependiendo del nivel de granularidad que desee controlar, debe elegir los correctos.

Espere a que se completen todas las tareas en ExecutorService #

Echemos un vistazo a varias opciones para esperar a que se completen las tareas enviadas a Ejecutor

  1. ExecutorService invokeAll()

Ejecuta las tareas dadas, devolviendo una lista de Futuros manteniendo su estado y resultados cuando todo se completa.

Ejemplo:

import java.util.concurrent.*;
import java.util.*;

public class InvokeAllDemo{
    public InvokeAllDemo(){
        System.out.println("creating service");
        ExecutorService service = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
        
        List<MyCallable> futureList = new ArrayList<MyCallable>();
        for (int i = 0; i < 10; i++){
            MyCallable myCallable = new MyCallable((long)i);
            futureList.add(myCallable);
        }
        System.out.println("Start");
        try{
            List<Future<Long>> futures = service.invokeAll(futureList);  
        } catch(Exception err){
            err.printStackTrace();
        }
        System.out.println("Completed");
        service.shutdown();
    }
    public static void main(String args[]){
        InvokeAllDemo demo = new InvokeAllDemo();
    }
    class MyCallable implements Callable<Long>{
        Long id = 0L;
        public MyCallable(Long val){
            this.id = val;
        }
        public Long call(){
            // Add your business logic
            return id;
        }
    }
}

  1. Contador regresivo

    A synchronization aid that allows one or more threads to wait until a set of operations being performed in other threads completes.

    A CountDownLatch is initialized with a given count. The await methods block until the current count reaches zero due to invocations of the countDown() method, after which all waiting threads are released and any subsequent invocations of await return immediately. This is a one-shot phenomenon – the count cannot be reset. If you need a version that resets the count, consider using a CyclicBarrier.

  2. ForkJoinPool o newWorkStealingPool() en Ejecutores

  3. Iterar a través de todos los objetos Futuros creados después de enviarlos a ExecutorService

  4. Forma recomendada de cierre desde la página de documentación de Oracle de ExecutorService:

    void shutdownAndAwaitTermination(ExecutorService pool) {
    pool.shutdown(); // Disable new tasks from being submitted
    try {
      // Wait a while for existing tasks to terminate
      if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
        pool.shutdownNow(); // Cancel currently executing tasks
        // Wait a while for tasks to respond to being cancelled
        if (!pool.awaitTermination(60, TimeUnit.SECONDS))
            System.err.println("Pool did not terminate");
      }
    } catch (InterruptedException ie) {
      // (Re-)Cancel if current thread also interrupted
      pool.shutdownNow();
      // Preserve interrupt status
      Thread.currentThread().interrupt();
    }

    shutdown(): Initiates an orderly shutdown in which previously submitted tasks are executed, but no new tasks will be accepted.

shutdownNow():intenta detener todas las tareas que se ejecutan activamente, detiene el procesamiento de las tareas en espera y devuelve una lista de las tareas que estaban en espera de ejecución.

En el ejemplo anterior, si sus tareas tardan más en completarse, puede cambiar la condición si a la condición mientras

Reemplazar

   if (!pool.awaitTermination(60, TimeUnit.SECONDS))

con

   while(!pool.awaitTermination(60, TimeUnit.SECONDS)) {
     Thread.sleep(60000);
 }

Casos de uso para diferentes tipos de ExecutorService #

Ejecutores devuelve diferentes tipos de ThreadPools que satisfacen necesidades específicas.

  1. servicio de ejecución público estático newSingleThreadExecutor ()

Crea un Ejecutor que utiliza un solo subproceso de trabajo que opera fuera de una cola ilimitada

Hay una diferencia entre newFixedThreadPool(1) y newSingleThreadExecutor() como dice el documento de Java para este último:

> Unlike the otherwise equivalent newFixedThreadPool(1) the returned executor is guaranteed not to be reconfigurable to use additional threads.

Which means that a `newFixedThreadPool` can be reconfigured later in the program by: `((ThreadPoolExecutor) fixedThreadPool).setMaximumPoolSize(10)`

Esto no es posible para newSingleThreadExecutor

Casos de uso:

1. You want to execute the submitted tasks in a sequence.
2. You need only one Thread to handle all your request

Contras:

1. Unbounded queue is harmful
  1. public static ExecutorService newFixedThreadPool(int nThreads)

Crea un grupo de subprocesos que reutiliza un número fijo de subprocesos que operan fuera de una cola ilimitada compartida. En cualquier momento, como máximo, los subprocesos nThreads serán tareas de procesamiento activas. Si se envían tareas adicionales cuando todos los subprocesos están activos, esperarán en la cola hasta que haya un subproceso disponible.

Casos de uso:

  1. Uso efectivo de los núcleos disponibles. Configure nThreads como Runtime.getRuntime(). AvailableProcessors()
  2. Cuando decide que el número de subprocesos no debe exceder un número en el grupo de subprocesos

Contras:

  1. La cola ilimitada es dañina.

  2. public static ExecutorService newCachedThreadPool()

Crea un grupo de subprocesos que crea nuevos subprocesos según sea necesario, pero reutilizará subprocesos construidos previamente cuando estén disponibles

Casos de uso:

  1. Para tareas asíncronas de corta duración

Contras:

  1. La cola ilimitada es dañina.

  2. Cada nueva tarea creará un nuevo hilo si todos los hilos existentes están ocupados. Si la tarea dura mucho, se creará una mayor cantidad de subprocesos, lo que degradará el rendimiento del sistema. Alternativa en este caso: newFixedThreadPool

  3. public static ScheduledExecutorService newScheduledThreadPool(int corePoolSize)

Crea un grupo de subprocesos que puede programar comandos para que se ejecuten después de un retraso determinado o para que se ejecuten periódicamente.

Casos de uso:

  1. Manejo de eventos recurrentes con retrasos, que ocurrirán en el futuro en ciertos intervalos de tiempo

Contras:

  1. La cola ilimitada es dañina.

5.public static ExecutorService newWorkStealingPool()

Crea un grupo de subprocesos que roban trabajo usando todos los procesadores disponibles como su nivel de paralelismo de destino

Casos de uso:

  1. Para problemas del tipo divide y vencerás.
  2. Uso efectivo de subprocesos inactivos. Los subprocesos inactivos roban tareas de los subprocesos ocupados.

Contras:

  1. El tamaño de cola ilimitado es dañino.

Puede ver un inconveniente común en todos estos ExecutorService: cola ilimitada. Esto se solucionará con ThreadPoolExecutor

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

Con ThreadPoolExecutor, puede

  1. Controle el tamaño del grupo de subprocesos dinámicamente
  2. Establecer la capacidad para BlockingQueue
  3. Defina RejectionExecutionHander cuando la cola esté llena
  4. CustomThreadFactory para agregar alguna funcionalidad adicional durante la creación de subprocesos (public Thread newThread(Runnable r)

Programar tareas para que se ejecuten a una hora fija, después de un retraso o repetidamente #

La clase ScheduledExecutorService proporciona métodos para programar tareas únicas o repetidas de varias maneras. El siguiente ejemplo de código asume que pool ha sido declarado e inicializado de la siguiente manera:

ScheduledExecutorService pool = Executors.newScheduledThreadPool(2);

Además de los métodos ExecutorService normales, la API ScheduledExecutorService añade 4 métodos que programan tareas y devuelven objetos ScheduledFuture. Este último se puede utilizar para recuperar resultados (en algunos casos) y cancelar tareas.

Comenzar una tarea después de un retraso fijo #

El siguiente ejemplo programa una tarea para que comience después de diez minutos.

ScheduledFuture<Integer> future = pool.schedule(new Callable<>() {
        @Override public Integer call() {
            // do something
            return 42;
        }
    }, 
    10, TimeUnit.MINUTES);

Inicio de tareas a una tasa fija #

El siguiente ejemplo programa una tarea para que comience después de diez minutos y luego repetidamente a una velocidad de una vez cada minuto.

ScheduledFuture<?> future = pool.scheduleAtFixedRate(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

La ejecución de la tarea continuará de acuerdo con el cronograma hasta que se apague el “grupo”, se cancele el “futuro” o una de las tareas encuentre una excepción.

Se garantiza que las tareas programadas por una determinada llamada scheduledAtFixedRate no se superpondrán en el tiempo. Si una tarea lleva más tiempo que el período prescrito, la ejecución de la tarea siguiente y las subsiguientes pueden comenzar tarde.

Iniciar tareas con un retraso fijo #

El siguiente ejemplo programa una tarea para que comience después de diez minutos y luego repetidamente con un retraso de un minuto entre la finalización de una tarea y el inicio de la siguiente.

ScheduledFuture<?> future = pool.scheduleWithFixedDelay(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

La ejecución de la tarea continuará de acuerdo con el cronograma hasta que se apague el “grupo”, se cancele el “futuro” o una de las tareas encuentre una excepción.
