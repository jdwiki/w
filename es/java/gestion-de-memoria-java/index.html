En Java, los objetos se asignan en el montón y la memoria del montón se recupera mediante la recolección automática de elementos no utilizados. Un programa de aplicación no puede eliminar explícitamente un objeto Java.

Los principios básicos de la recolección de basura de Java se describen en el ejemplo [Recolección de basura][1]. Otros ejemplos describen la finalización, cómo activar el recolector de elementos no utilizados a mano y el problema de las fugas de almacenamiento.

[1]: https://www.wikiod.com/es/java/gestion-de-memoria-java#Recolección de basura

Configuración de los tamaños Heap, PermGen y Stack #

Cuando se inicia una máquina virtual Java, necesita saber el tamaño del Heap y el tamaño predeterminado para las pilas de subprocesos. Estos se pueden especificar usando las opciones de la línea de comandos en el comando java. Para las versiones de Java anteriores a Java 8, también puede especificar el tamaño de la región PermGen del Heap.

Tenga en cuenta que PermGen se eliminó en Java 8, y si intenta establecer el tamaño de PermGen, la opción se ignorará (con un mensaje de advertencia).

Si no especifica explícitamente los tamaños de Heap y Stack, la JVM utilizará los valores predeterminados que se calculan de una manera específica de versión y plataforma. Esto puede resultar en que su aplicación use muy poca o demasiada memoria. Esto suele estar bien para pilas de subprocesos, pero puede ser problemático para un programa que usa mucha memoria.

Configuración de Heap, PermGen y tamaños de pila predeterminados:

Las siguientes opciones de JVM establecen el tamaño del almacenamiento dinámico:

  • -Xms<size> - establece el tamaño inicial del almacenamiento dinámico
  • -Xmx<size> - establece el tamaño máximo de almacenamiento dinámico
  • -XX:PermSize<size> - establece el tamaño inicial de PermGen
  • -XX:MaxPermSize<size> - establece el tamaño máximo de PermGen
  • -Xss<size> - establece el tamaño de pila de subprocesos predeterminado

El parámetro <tamaño> puede ser un número de bytes, o puede tener un sufijo de k, m o g. Estos últimos especifican el tamaño en kilobytes, megabytes y gigabytes respectivamente.

Ejemplos:

$ java -Xms512m -Xmx1024m JavaApp
$ java -XX:PermSize=64m -XX:MaxPermSize=128m JavaApp
$ java -Xss512k JavaApp

Encontrar los tamaños predeterminados:

La opción -XX:+printFlagsFinal se puede utilizar para imprimir los valores de todos los indicadores antes de iniciar la JVM. Esto se puede usar para imprimir los valores predeterminados para la configuración de tamaño de montón y pila de la siguiente manera:

  • Para Linux, Unix, Solaris y MacOSX

    $ java -XX:+PrintFlagsFinal -version | grep -iE ‘HeapSize|PermSize|ThreadStackSize’

  • Para ventanas:

    java -XX:+PrintFlagsFinal -version | findstr /i “HeapSize PermSize ThreadStackSize”

La salida de los comandos anteriores se parecerá a la siguiente:

uintx InitialHeapSize                          := 20655360        {product}
uintx MaxHeapSize                              := 331350016       {product}
uintx PermSize                                  = 21757952        {pd product}
uintx MaxPermSize                               = 85983232        {pd product}
 intx ThreadStackSize                           = 1024            {pd product}

Los tamaños se dan en bytes.

Recolección de basura #

El enfoque de C++: nuevo y eliminar #

En un lenguaje como C++, el programa de aplicación es responsable de administrar la memoria utilizada por la memoria asignada dinámicamente. Cuando se crea un objeto en el montón de C++ usando el operador nuevo, debe haber un uso correspondiente del operador eliminar para deshacerse del objeto:

  • Si el programa se olvida de eliminar un objeto y simplemente se “olvida” de él, la memoria asociada se pierde para la aplicación. El término para esta situación es pérdida de memoria, y si se pierde demasiada memoria, es probable que una aplicación utilice más y más memoria y, finalmente, se bloquee.

  • Por otro lado, si una aplicación intenta ’eliminar’ el mismo objeto dos veces, o usar un objeto después de haberlo eliminado, es posible que la aplicación se bloquee debido a problemas con la corrupción de la memoria.

En un programa complicado de C++, implementar la gestión de la memoria usando nuevo y delete puede llevar mucho tiempo. De hecho, la gestión de la memoria es una fuente común de errores.

El enfoque de Java - recolección de basura #

Java adopta un enfoque diferente. En lugar de un operador delete explícito, Java proporciona un mecanismo automático conocido como recolección de elementos no utilizados para recuperar la memoria utilizada por los objetos que ya no se necesitan. El sistema de tiempo de ejecución de Java asume la responsabilidad de encontrar los objetos que se van a eliminar. Esta tarea la realiza un componente llamado recolector de basura, o GC para abreviar.

En cualquier momento durante la ejecución de un programa Java, podemos dividir el conjunto de todos los objetos existentes en dos subconjuntos distintos1:

  • Los objetos alcanzables están definidos por el JLS de la siguiente manera:

    A reachable object is any object that can be accessed in any potential continuing computation from any live thread.

    In practice, this means that there is a chain of references starting from an in-scope local variable or a static variable by which some code might be able to reach the object.

  • Los objetos inalcanzables son objetos que no es posible que se alcancen como se indicó anteriormente.

Cualquier objeto que sea inalcanzable es elegible para la recolección de elementos no utilizados. Esto no significa que serán recolectados como basura. En realidad:

  • Un objeto inalcanzable no se recopila inmediatamente al volverse inalcanzable1.
  • Un objeto inalcanzable puede que nunca se recolecte como basura.

La especificación del lenguaje Java da mucha libertad a una implementación de JVM para decidir cuándo recopilar objetos inalcanzables. También (en la práctica) da permiso para que una implementación de JVM sea conservadora en la forma en que detecta objetos inalcanzables.

Lo único que garantiza el JLS es que ningún objeto alcanzable será recolectado como basura.

Qué sucede cuando un objeto se vuelve inalcanzable #

En primer lugar, nada sucede específicamente cuando un objeto se vuelve inalcanzable. Las cosas solo suceden cuando el recolector de basura se ejecuta y detecta que el objeto es inalcanzable. Además, es común que una ejecución de GC no detecte todos los objetos inalcanzables.

Cuando el GC detecta un objeto inalcanzable, pueden ocurrir los siguientes eventos.

  1. Si hay objetos Referencia que se refieren al objeto, esas referencias se borrarán antes de que se elimine el objeto.

  2. Si el objeto es finalizable, se finalizará. Esto sucede antes de que se elimine el objeto.

  3. El objeto se puede eliminar y la memoria que ocupa se puede recuperar.

Tenga en cuenta que hay una secuencia clara en la que los eventos anteriores pueden ocurrir, pero nada requiere que el recolector de basura realice la eliminación final de cualquier objeto específico en un período de tiempo específico.

Ejemplos de objetos alcanzables e inalcanzables #

Considere las siguientes clases de ejemplo:

// A node in simple "open" linked-list.
public class Node {
    private static int counter = 0;

    public int nodeNumber = ++counter;
    public Node next;
}

public class ListTest {
    public static void main(String[] args) {
        test();                    // M1
        System.out.prinln("Done"); // M2
    }
    
    private static void test() {
        Node n1 = new Node();      // T1
        Node n2 = new Node();      // T2
        Node n3 = new Node();      // T3
        n1.next = n2;              // T4
        n2 = null;                 // T5
        n3 = null;                 // T6
    }
}

Examinemos qué sucede cuando se llama a test(). Las declaraciones T1, T2 y T3 crean objetos Node, y todos los objetos son accesibles a través de las variables n1, n2 y n3 respectivamente. La instrucción T4 asigna la referencia al segundo objeto ‘Nodo’ al campo ‘siguiente’ del primero. Cuando se hace eso, se puede acceder al segundo Nodo a través de dos rutas:

 n2 -> Node2
 n1 -> Node1, Node1.next -> Node2

En la declaración T5, asignamos null a n2. Esto rompe la primera de las cadenas de accesibilidad para Node2, pero la segunda permanece intacta, por lo que Node2 sigue siendo accesible.

En la declaración T6, asignamos null a n3. Esto rompe la única cadena de accesibilidad para Node3, lo que hace que Node3 sea inalcanzable. Sin embargo, Node1 y Node2 todavía son accesibles a través de la variable n1.

Finalmente, cuando el método test() regresa, sus variables locales n1, n2 y n3 quedan fuera del alcance y, por lo tanto, nadie puede acceder a ellas. Esto rompe las cadenas de accesibilidad restantes para Nodo1 y Nodo2, y todos los objetos Nodo no son inalcanzables y elegibles para la recolección de basura.

1 - Esta es una simplificación que ignora la finalización y las clases Referencia. 2 - Hipotéticamente, una implementación de Java podría hacer esto, pero el costo de rendimiento de hacerlo lo hace poco práctico.

Fugas de memoria en Java #

En el ejemplo [Recolección de basura][1], implicamos que Java resuelve el problema de las fugas de memoria. Esto no es realmente cierto. Un programa Java puede perder memoria, aunque las causas de las pérdidas son bastante diferentes.

Los objetos alcanzables pueden filtrarse #

Considere la siguiente implementación de pila ingenua.

public class NaiveStack {
    private Object[] stack = new Object[100];
    private int top = 0;

    public void push(Object obj) {
        if (top >= stack.length) {
            throw new StackException("stack overflow");
        }
        stack[top++] = obj;
    }

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        return stack[--top];
    }

    public boolean isEmpty() {
        return top == 0;
    }
}

Cuando empujas un objeto y luego lo sacas inmediatamente, todavía habrá una referencia al objeto en la matriz stack.

La lógica de la implementación de la pila significa que esa referencia no se puede devolver a un cliente de la API. Si se ha extraído un objeto, podemos demostrar que no se puede “acceder a él en ningún cálculo continuo potencial desde ningún subproceso en vivo”. El problema es que una JVM de generación actual no puede probar esto. Las JVM de generación actual no tienen en cuenta la lógica del programa para determinar si las referencias son accesibles. (Para empezar, no es práctico.)

Pero dejando a un lado el tema de lo que realmente significa accesibilidad, claramente tenemos una situación aquí donde la implementación de NaiveStack está “colgando” de objetos que deberían ser reclamados. Eso es una fuga de memoria.

En este caso, la solución es sencilla:

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        Object popped = stack[--top];
        stack[top] = null;              // Overwrite popped reference with null.
        return popped;
    }

Los cachés pueden ser fugas de memoria #

Una estrategia común para mejorar el rendimiento del servicio es almacenar en caché los resultados. La idea es que mantenga un registro de las solicitudes comunes y sus resultados en una estructura de datos en memoria conocida como caché. Luego, cada vez que se realiza una solicitud, busca la solicitud en el caché. Si la búsqueda tiene éxito, devuelve los resultados guardados correspondientes.

Esta estrategia puede ser muy efectiva si se implementa correctamente. Sin embargo, si se implementa incorrectamente, un caché puede ser una pérdida de memoria. Considere el siguiente ejemplo:

public class RequestHandler {
    private Map<Task, Result> cache = new HashMap<>();

    public Result doRequest(Task task) {
        Result result = cache.get(task);
        if (result == null) {
            result == doRequestProcessing(task);
            cache.put(task, result);
        }
        return result;
    }
}

El problema con este código es que si bien cualquier llamada a doRequest podría agregar una nueva entrada al caché, no hay nada para eliminarla. Si el servicio recibe tareas diferentes continuamente, la memoria caché finalmente consumirá toda la memoria disponible. Esta es una forma de pérdida de memoria.

Un enfoque para resolver esto es usar un caché con un tamaño máximo y descartar las entradas antiguas cuando el caché excede el máximo. (Eliminar la entrada usada menos recientemente es una buena estrategia). Otro enfoque es construir el caché usando WeakHashMap para que la JVM pueda desalojar las entradas del caché si el montón comienza a llenarse demasiado.

Finalización #

Un objeto Java puede declarar un método finalizar. Este método se llama justo antes de que Java libere la memoria para el objeto. Por lo general, se verá así:

public class MyClass {
  
    //Methods for the class

    @Override
    protected void finalize() throws Throwable {
        // Cleanup code
    }
}

Sin embargo, existen algunas advertencias importantes sobre el comportamiento de la finalización de Java.

  • Java no garantiza cuándo se llamará a un método finalize().
  • Java ni siquiera garantiza que se llamará a un método finalize() en algún momento durante la vida útil de la aplicación en ejecución.
  • Lo único que se garantiza es que se llamará al método antes de que se elimine el objeto… si se elimina el objeto.

Las advertencias anteriores significan que es una mala idea confiar en el método finalizar para realizar acciones de limpieza (u otras) que deben realizarse de manera oportuna. La dependencia excesiva de la finalización puede provocar fugas de almacenamiento, fugas de memoria y otros problemas.

En resumen, hay muy pocas situaciones en las que la finalización sea realmente una buena solución.

Los finalizadores solo se ejecutan una vez #

Normalmente, un objeto se elimina después de haberlo finalizado. Sin embargo, esto no sucede todo el tiempo. Considere el siguiente ejemplo1:

public class CaptainJack {
    public static CaptainJack notDeadYet = null;

    protected void finalize() {
        // Resurrection!
        notDeadYet = this;
    }
}

Cuando una instancia de CaptainJack se vuelve inalcanzable y el recolector de basura intenta recuperarla, el método finalize() asignará una referencia a la instancia a la variable notDeadYet. Eso hará que la instancia sea accesible una vez más y el recolector de elementos no utilizados no la eliminará.

Pregunta: ¿El Capitán Jack es inmortal?

Respuesta: no

El problema es que la JVM solo ejecutará un finalizador en un objeto una vez en su vida útil. Si asigna null a notDeadYet, lo que hace que una instancia resucitada sea inalcanzable una vez más, el recolector de basura no llamará a finalize() en el objeto.

1 - Ver https://en.wikipedia.org/wiki/Jack_Harkness.
