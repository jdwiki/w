La guía de estilo oficial de Oracle para el lenguaje de programación Java es un estándar que siguen los desarrolladores de Oracle y se recomienda que lo siga cualquier otro desarrollador de Java. Cubre nombres de archivo, organización de archivos, sangría, comentarios, declaraciones, espacios en blanco, convenciones de nomenclatura, prácticas de programación e incluye un ejemplo de código.

  • Los ejemplos anteriores siguen estrictamente la nueva [guía de estilo oficial] (http://cr.openjdk.java.net/~alundblad/styleguide) de Oracle. En otras palabras, no están inventados subjetivamente por los autores de esta página.

  • La guía de estilo oficial se escribió cuidadosamente para que sea compatible con la [guía de estilo original] (http://www.oracle.com/technetwork/java/javase/documentation/codeconvtoc-136057.html) y la mayoría del código. en la naturaleza.

  • La guía de estilo oficial ha sido pares [revisada](http://mail.openjdk.java.net /pipermail/discuss/2015-August/003771.html) por, entre otros, Brian Goetz (Java Language Architect) y Mark Reinhold (Chief Architect of the Java Platform).

  • Los ejemplos no son normativos; Si bien tienen la intención de ilustrar la forma correcta de formatear el código, puede haber otras formas de formatear correctamente el código. Este es un principio general: puede haber varias formas de formatear el código, todas siguiendo las pautas oficiales.

Convenciones de nombres #

Nombres de paquetes #

  • Los nombres de los paquetes deben estar en minúsculas sin guiones bajos ni otros caracteres especiales.
  • Los nombres de los paquetes comienzan con la parte de autoridad invertida de la dirección web de la empresa del desarrollador. Esta parte puede ir seguida de una subestructura de paquete dependiente de la estructura del proyecto/programa.
  • No utilices la forma plural. Siga la convención de la API estándar que utiliza, por ejemplo, java.lang.annotation y no java.lang.annotations.
  • Ejemplos: com.tuempresa.widget.button, com.tuempresa.core.api

Nombres de clase, interfaz y enumeración #

  • Los nombres de clase y enumeración normalmente deben ser sustantivos.
  • Los nombres de las interfaces normalmente deben ser sustantivos o adjetivos que terminen en …capaz.
  • Utilice mayúsculas y minúsculas con la primera letra de cada palabra en mayúsculas (es decir, CamelCase).
  • Haga coincidir la expresión regular ^[A-Z][a-zA-Z0-9]*$.
  • Use palabras completas y evite el uso de abreviaturas a menos que la abreviatura se use más ampliamente que la forma larga.
  • Dar formato a una abreviatura como una palabra si forma parte de un nombre de clase más largo.
  • Ejemplos: ArrayList, BigInteger, ArrayIndexOutOfBoundsException, Iterable.

Nombres de métodos #

Los nombres de los métodos normalmente deben ser verbos u otras descripciones de acciones.

  • Deben coincidir con la expresión regular ^[a-z][a-zA-Z0-9]*$.
  • Usar mayúsculas y minúsculas con la primera letra en minúsculas.
  • Ejemplos: toString, hashCode

Variables #

Los nombres de las variables deben estar en mayúsculas y minúsculas con la primera letra en minúsculas

  • Haga coincidir la expresión regular ^[a-z][a-zA-Z0-9]*$
  • Más recomendaciones: Variables
  • Ejemplos: elementos, índiceActual

Variables de tipo #

Para casos simples donde hay pocas variables de tipo involucradas, use una sola letra mayúscula.

  • Haga coincidir la expresión regular ^[A-Z][0-9]?$
  • Si una letra es más descriptiva que otra (como K y V para claves y valores en mapas o R para un tipo de retorno de función), utilícela, de lo contrario use T.
  • Para casos complejos en los que las variables de tipo de una sola letra se vuelven confusas, use nombres más largos escritos en mayúsculas y use guiones bajos (_) para separar las palabras.
  • Ejemplos: T, V, SRC_VERTEX

constantes #

Las constantes (campos estáticos finales cuyo contenido es inmutable, por reglas del idioma o por convención) deben nombrarse con letras mayúsculas y guión bajo (_) para separar las palabras.

  • Haga coincidir la expresión regular ^[A-Z][A-Z0-9]*(_[A-Z0-9]+)*$
  • Ejemplos: BUFFER_SIZE, MAX_LEVEL

Otras pautas sobre nombres #

  • Evite ocultar/sombrear métodos, variables y tipo de variables en ámbitos externos.
  • Deje que la verbosidad del nombre se correlacione con el tamaño del alcance. (Por ejemplo, use nombres descriptivos para campos de clases grandes y nombres breves para variables locales de corta duración).
  • Al nombrar miembros estáticos públicos, deje que el identificador sea autodescriptivo si cree que se importarán de forma estática.
  • Lecturas adicionales: Sección de nombres (en la Guía de estilo oficial de Java)

Fuente: Pautas de estilo Java de Oracle

Estructura de clase #

Orden de los miembros de la clase #

Los miembros de la clase deben ordenarse de la siguiente manera:

  1. Campos (en orden de público, protegido y privado)
  2. Constructores
  3. Métodos de fábrica
  4. Otros Métodos (en orden de público, protegido y privado)

No es necesario ordenar campos y métodos principalmente por sus modificadores de acceso o identificadores.

He aquí un ejemplo de este orden:

class Example {

    private int i;

    Example(int i) {
        this.i = i;
    }

    static Example getExample(int i) {
        return new Example(i);
    }

    @Override
    public String toString() {
        return "An example [" + i + "]";
    }

}

Agrupación de miembros de la clase #

  • Los campos relacionados deben agruparse.
  • Un tipo anidado puede declararse justo antes de su primer uso; de lo contrario, debe declararse antes de los campos.
  • Los constructores y los métodos sobrecargados deben agruparse por funcionalidad y ordenarse con aridad creciente. Esto implica que la delegación entre estas construcciones fluye hacia abajo en el código.
  • Los constructores deben agruparse sin otros miembros entre ellos.
  • Las variantes sobrecargadas de un método deben agruparse sin otros miembros entre ellas.

Anotaciones #

Las anotaciones de declaración deben colocarse en una línea separada de la declaración que se está anotando.

@SuppressWarnings("unchecked")
public T[] toArray(T[] typeHolder) {
    ...
}

Sin embargo, las anotaciones breves o pocas que anoten un método de una sola línea pueden colocarse en la misma línea que el método si mejora la legibilidad. Por ejemplo, uno puede escribir:

@Nullable String getName() { return name; }

Por una cuestión de consistencia y legibilidad, todas las anotaciones deben colocarse en la misma línea o cada anotación debe colocarse en una línea separada.

// Bad.
@Deprecated @SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Even worse.
@Deprecated @SafeVarargs
@CustomAnnotation public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated
@SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated @SafeVarargs @CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

Importar declaraciones #

// First java/javax packages
import java.util.ArrayList;
import javax.tools.JavaCompiler;

// Then third party libraries
import com.fasterxml.jackson.annotation.JsonProperty;

// Then project imports
import com.example.my.package.ClassA;
import com.example.my.package.ClassB;

// Then static imports (in the same order as above)
import static java.util.stream.Collectors.toList;

  • Las declaraciones de importación deben ordenarse…

  • …principalmente por no estático / estático con importaciones no estáticas primero.

  • …en segundo lugar por origen del paquete según el siguiente orden

    • java packages
    • javax packages
    • external packages (e.g. org.xml)
    • internal packages (e.g. com.sun)

  • …terciario por paquete e identificador de clase en orden lexicográfico

  • Las declaraciones de importación no deben ajustarse en línea, independientemente de si excede la longitud máxima recomendada de una línea.

  • No debe haber importaciones no utilizadas.

Importaciones de comodines #

  • En general, no se deben utilizar las importaciones de comodines.
  • Al importar una gran cantidad de clases estrechamente relacionadas (como implementar un visitante sobre un árbol con docenas de clases de “nodo” distintas), se puede usar una importación comodín.
  • En cualquier caso, no se debe utilizar más de una importación de comodines por archivo.

Paréntesis redundantes #

return flag ? "yes" : "no";

String cmp = (flag1 != flag2) ? "not equal" : "equal";

// Don't do this
return (flag ? "yes" : "no");
  • Se pueden usar paréntesis de agrupación redundantes (es decir, paréntesis que no afectan la evaluación) si mejoran la legibilidad.
  • Por lo general, los paréntesis de agrupación redundantes deben omitirse en expresiones más cortas que involucran operadores comunes, pero deben incluirse en expresiones más largas o expresiones que involucran operadores cuya precedencia y asociatividad no están claras sin paréntesis. Las expresiones ternarias con condiciones no triviales pertenecen a este último.
  • La expresión completa que sigue a una palabra clave return no debe estar entre paréntesis.

Tirantes #

class Example {
    void method(boolean error) {
        if (error) {
            Log.error("Error occurred!");
            System.out.println("Error!");
        } else { // Use braces since the other block uses braces.
            System.out.println("No error");
        }
    }
}

  • Las llaves de apertura deben colocarse al final de la línea actual en lugar de una línea separada.

  • Debe haber una nueva línea delante de una llave de cierre a menos que el bloque esté vacío (ver Formas cortas a continuación)

  • Se recomiendan llaves incluso cuando el lenguaje las hace opcionales, como if de una sola línea y cuerpos de bucle.

  • Si un bloque ocupa más de una línea (incluidos los comentarios), debe tener llaves.

  • Si uno de los bloques en una instrucción if / else tiene llaves, el otro bloque también debe hacerlo.

  • Si el bloque viene último en un bloque envolvente, debe tener llaves.

  • Las palabras clave else, catch y while en los bucles do…while van en la misma línea que la llave de cierre del bloque anterior.

Formas cortas #

enum Response { YES, NO, MAYBE }
public boolean isReference() { return true; }

Las recomendaciones anteriores están destinadas a mejorar la uniformidad (y, por lo tanto, aumentar la familiaridad/legibilidad). En algunos casos, los “formularios cortos” que se desvían de las pautas anteriores son igualmente legibles y se pueden usar en su lugar. Estos casos incluyen, por ejemplo, declaraciones de enumeración simples y métodos triviales y expresiones lambda.

Modificadores #

class ExampleClass {
    // Access modifiers first (don't do for instance "static public")
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}

interface ExampleInterface {
    // Avoid 'public' and 'abstract' since they are implicit
    void sayHello();
}

  • Los modificadores deben ir en el siguiente orden

  • Modificador de acceso (público / privado / protegido)

  • abstracto

  • estático

  • final

  • ’transitorio'

  • volátil

  • predeterminado

  • sincronizado

  • nativo

  • fp estricto

  • Los modificadores no deben escribirse cuando están implícitos. Por ejemplo, los métodos de interfaz no deben declararse “públicos” ni “abstractos”, y las enumeraciones e interfaces anidadas no deben declararse estáticas.

  • Los parámetros del método y las variables locales no deben declararse “finales” a menos que mejoren la legibilidad o documenten una decisión de diseño real.

  • Los campos deben declararse “finales” a menos que exista una razón de peso para hacerlos mutables.

sangría #

  • El nivel de sangría es cuatro espacios.

  • Solo se pueden usar caracteres de espacio para la sangría. Sin pestañas.

  • Las líneas vacías no deben tener sangría. (Esto está implícito en la regla de no dejar espacios en blanco).

  • Las líneas case deben sangrarse con cuatro espacios, y las declaraciones dentro del caso deben sangrarse con otros cuatro espacios.

    switch (var) { case TWO: setChoice(“two”); break; case THREE: setChoice(“three”); break; default: throw new IllegalArgumentException(); }

Consulte https://www.wikiod.com/es/java/estandar-de-codigo-oficial-de-oracle#Envolviendo declaraciones para obtener pautas sobre cómo sangrar las líneas de continuación.

Literales #

long l = 5432L;
int i = 0x123 + 0xABC;
byte b = 0b1010;
float f1 = 1 / 5432f;
float f2 = 0.123e4f;
double d1 = 1 / 5432d;  // or 1 / 5432.0
double d2 = 0x1.3p2;
  • Los literales largos deben usar el sufijo L en mayúscula.
  • Los literales hexadecimales deben usar letras mayúsculas A-F.
  • Todos los demás prefijos, infijos y sufijos numéricos deben usar letras minúsculas.

Declaración de paquete #

package com.example.my.package;

La declaración del paquete no se debe ajustar en línea, independientemente de si excede la longitud máxima recomendada de una línea.

Expresiones Lambda #

Runnable r = () -> System.out.println("Hello World");

Supplier<String> c = () -> "Hello World";

// Collection::contains is a simple unary method and its behavior is
// clear from the context. A method reference is preferred here.
appendFilter(goodStrings::contains);

// A lambda expression is easier to understand than just tempMap::put in this case
trackTemperature((time, temp) -> tempMap.put(time, temp));
  • Las lambdas de expresión son preferibles a las lambdas de bloque de una sola línea.
  • En general, se deben preferir las referencias a métodos a las expresiones lambda.
  • Para referencias de métodos de instancia enlazada, o métodos con arity mayor que uno, una expresión lambda puede ser más fácil de entender y, por lo tanto, preferida. Especialmente si el comportamiento del método no está claro por el contexto.
  • Los tipos de parámetros deben omitirse a menos que mejoren la legibilidad.
  • Si una expresión lambda abarca más de unas pocas líneas, considere crear un método.

Archivos fuente de Java #

  • Todas las líneas deben terminar con un carácter de avance de línea (LF, valor ASCII 10) y no, por ejemplo, CR o CR+LF.

  • Es posible que no haya espacios en blanco al final de una línea.

  • El nombre de un archivo fuente debe ser igual al nombre de la clase que contiene seguido de la extensión .java, incluso para archivos que solo contienen una clase privada de paquete. Esto no se aplica a los archivos que no contienen declaraciones de clase, como package-info.java.

Envolviendo declaraciones #

  • El código fuente y los comentarios generalmente no deben exceder los 80 caracteres por línea y, en raras ocasiones, si alguna vez exceden los 100 caracteres por línea, incluida la sangría.

El límite de caracteres debe juzgarse caso por caso. Lo que realmente importa es la “densidad” semántica y la legibilidad de la línea. Hacer líneas innecesariamente largas las hace difíciles de leer; Del mismo modo, hacer “intentos heroicos” para encajarlos en 80 columnas también puede dificultar su lectura. La flexibilidad descrita aquí tiene como objetivo permitir a los desarrolladores evitar estos extremos, no maximizar el uso de los bienes inmuebles del monitor.

  • Las direcciones URL o los comandos de ejemplo no deben envolverse.

    // Ok even though it might exceed max line width when indented. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable(targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext(targetContainerType));

    // Wrapping preferable String pretty = Stream.of(args) .map(Argument::prettyPrint) .collectors(joining(", “));

    // Too strict interpretation of max line width. Readability suffers. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable( targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext( targetContainerType);

    // Should be wrapped even though it fits within the character limit String pretty = Stream.of(args).map(Argument::prettyPrint).collectors(joining(”, “));

  • Se prefiere envolver a un nivel sintáctico más alto que envolver a un nivel sintáctico más bajo.

  • Debe haber como máximo 1 declaración por línea.

  • Una línea de continuación debe estar sangrada de una de las siguientes cuatro maneras

  • Variante 1: Con 8 espacios extra respecto a la sangría de la línea anterior.

  • Variante 2: Con 8 espacios adicionales relativos a la columna inicial de la expresión envuelta.

  • Variante 3: Alineado con la expresión del hermano anterior (siempre que quede claro que es una línea de continuación)

  • Variante 4: alineado con la llamada de método anterior en una expresión encadenada.

Declaraciones de métodos de envoltura #

int someMethod(String aString,
               List<Integer> aList,
               Map<String, String> aMap,
               int anInt,
               long aLong,
               Set<Number> aSet,
               double aDouble) {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt, long aLong,
        double aDouble, long aLong) {
    …
}

int someMethod(String aString,
               List<Map<Integer, StringBuffer>> aListOfMaps,
               Map<String, String> aMap)
        throws IllegalArgumentException {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt)
                throws IllegalArgumentException {
    …
}
  • Las declaraciones de métodos se pueden formatear enumerando los argumentos verticalmente o con una nueva línea y +8 espacios adicionales
  • Si es necesario envolver una cláusula throws, coloque el salto de línea delante de la cláusula throws y asegúrese de que se destaque de la lista de argumentos, ya sea sangrando +8 en relación con la declaración de la función o +8 en relación con la línea anterior .

Expresiones envolventes #

  • Si una línea se acerca al límite máximo de caracteres, considere siempre dividirla en varias declaraciones/expresiones en lugar de envolver la línea.

  • Romper antes de los operadores.

  • Descanso antes del . en llamadas a métodos encadenados.

    popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

    // Don’t! Looks like two arguments popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

Espacio en blanco #

Espacio en blanco vertical #

  • Se debe usar una sola línea en blanco para separar…

  • Declaración de bultos

  • Declaraciones de clase

  • Constructores

  • Métodos

  • Inicializadores estáticos

  • Inicializadores de instancia

  • …y puede usarse para separar grupos lógicos de

  • declaraciones de importación

  • campos

  • declaraciones

  • Las líneas en blanco consecutivas múltiples solo deben usarse para separar grupos de miembros relacionados y no como el espaciado estándar entre miembros.

Espacio en blanco horizontal #

  • Se debe utilizar un solo espacio…

  • Para separar las palabras clave de los paréntesis y llaves de apertura o cierre contiguos

  • Antes y después de todos los operadores binarios y símbolos similares a operadores, como flechas en expresiones lambda y los dos puntos en bucles for mejorados (pero no antes de los dos puntos de una etiqueta)

  • Después de // que comienza un comentario.

  • Después de comas que separan argumentos y puntos y comas que separan las partes de un bucle for.

  • Después del paréntesis de cierre de un reparto.

  • En las declaraciones de variables no se recomienda alinear tipos y variables.

Caracteres especiales #

  • Aparte de LF, el único carácter de espacio en blanco permitido es Espacio (valor ASCII 32). Tenga en cuenta que esto implica que otros caracteres de espacio en blanco (en, por ejemplo, cadenas y caracteres literales) deben escribirse en forma de escape.

  • \', \", \\, \t, \b, \r, \f y \n deben preferirse al octal correspondiente (por ejemplo, \047) o Unicode (por ejemplo, \u0027) caracteres de escape.

  • Si fuera necesario ir en contra de las reglas anteriores por el bien de la prueba, la prueba debería generar la entrada requerida programáticamente.

Declaraciones de variables #

  • Una variable por declaración (y como máximo una declaración por línea)
  • Los corchetes de las matrices deben estar en el tipo (String[] args) y no en la variable (String args[]).
  • Declare una variable local justo antes de que se use por primera vez e inicialícela lo más cerca posible de la declaración.
