Personalización de la funcionalidad principal con temas

Los temas tienen archivos de diseño que, entre otras cosas, se pueden usar para cambiar los bloques que aparecen en la página. La plantilla de bloque también se puede cambiar y llamar a diferentes métodos.

Diseños a nivel de tienda

Los temas estructurados jerárquicamente de Magento significan que un tema base puede extenderse y asignarse a nivel de tienda.

Registro de temas personalizados

Los temas se pueden configurar de tres maneras:

  1. Por tienda en Sistema > Configuración > Diseño.
  2. Cambio de diseño con plazos Sistema > Diseño.
  3. Las excepciones de temas también se pueden establecer a nivel de categoría y producto.

Paquete versus Tema

Un paquete tiene varios temas. Cada uno de los temas de un paquete hereda del tema predeterminado dentro de un paquete.

Diseño alternativo

El procedimiento de respaldo del tema para ubicar archivos de plantillas es:

  1. {paquete}/{tema}
  2. {paquete}/predeterminado
  3. base/predeterminado

Para agregar más directorios al mecanismo de respaldo del tema, es necesario reescribir el método Mage_Core_Model_Design_Package::getFilename

Para el área de administración, el respaldo es predeterminado/predeterminado.

Rutas de plantilla y diseño

Bloques

Los bloques se utilizan para la salida. El bloque raíz es el padre de todos los bloques y es de tipo Mage_Page_Block_Html.

Los bloques Mage_Core_Block_Template usan archivos de plantilla para representar contenido. El nombre del archivo de la plantilla se establece dentro de setTemplate() o addData(’template’) con rutas relativas.

Las plantillas son solo piezas de PHP incluidas en Mage_Core_Block_Template. Por lo tanto, $this en una plantilla se refiere al bloque.

Mage_Core_Block_Template usa un búfer antes de incluir una plantilla para evitar una salida prematura.

El método Mage_Core_Model_Layout::createBlock crea instancias de bloques.

La clase Mage_Core_Model_Layout_Update considera qué bloques deben crearse para cada página al observar los controladores de diseño.

Todos los bloques de salida se renderizan, p. llamando a toHtml(), que a su vez puede optar por representar a sus hijos.

Los bloques Text y Text_List representan automáticamente su contenido.

Hay dos eventos que se disparan alrededor de la representación de bloques que se pueden usar para modificar el bloque antes y después de representar el HTML:

core_block_abstract_to_html_before core_block_abstract_to_html_after Un bloque secundario solo se procesará automáticamente si es de clase Mage_Core_Block_Textlist; de lo contrario, se debe llamar al método getChildHtml.

Se puede acceder a las instancias de bloque a través del diseño, p. Mago::aplicación()->getLayout() y $controlador->getLayout(). La salida del bloque está controlada por la función _toHtml().

Las plantillas son representadas por los métodos renderView()/fetchView() dentro de un bloque de plantilla. El almacenamiento en búfer de salida se puede desactivar con $layout->setDirectOutput.

Es posible agregar un bloque al diseño actual, pero debe hacerse antes de llamar al método renderLayout().

Diseño XML

<reference>
  -edit a block
<block>
  - define a block 
<action>
  - call method on a block
<update>
  - include nodes from another handle.

Los archivos de diseño se pueden registrar en config.xml:

<config>
    <{area}>
        <layout>
            <updates>
                <{name}>
                    <file>{filepath}</file>
                </{name}>
            </updates>
        </layout>
    </{area}>
</config>

La salida de la página se puede personalizar de las siguientes maneras:

  • Cambios de plantilla
  • Cambios de diseño
  • Anulación de bloques
  • Observadores Las variables en los bloques se pueden configurar de las siguientes maneras:
  • Diseño -Through actions or attributes
  • Controlador -$this-getLayout()->getBlock()
  • Bloques de niños -$this->getChild()
  • Otro -Mage::app()->getLayout()

Activos de bloque principal

Los activos de JavaScript y CSS se manejan en el bloque Mage_Page_Block_Html_head. Este bloque maneja la combinación de activos en un solo archivo para minimizar las solicitudes HTTP. El archivo combinado se basa en el tiempo de edición de los archivos de origen.

Al fusionar CSS, se llama a una función de devolución de llamada en Mage_Core_Model_Design_Package para actualizar cualquier directiva @import o url() con las URL correctas

Diferentes mecanismos para deshabilitar la salida del bloque #

  • Si la respuesta ya se ha creado y configurado en la respuesta objeto fuera del proceso de representación regular (por ejemplo, en un observador), el indicador ’no-renderLayout’ se puede configurar en el controlador de acción usando

    Mage::app()->getFrontController()->getAction()->setFlag(’’,’no-renderLayout’);

  • Esto evita que renderLayout() procese los bloques de salida.

  • Lo mismo se puede lograr llamando a setNoRender(true) en el frente controlador: Mage::app()->getFrontController()->setNoRender(true);

  • Establecer el indicador isDispatched() en el objeto de respuesta podría ser más eficiente para lograr un efecto similar.

Se puede acceder a las instancias de bloque desde el controlador #

Desde un controlador de acción:

$this->getLayout()->getBlock('head')->getTemplate();

/**
* Get specified tab grid
*/
public function gridOnlyAction()
{
$this->_initProduct();
$this->getResponse()->setBody(
$this->getLayout()->createBlock('adminhtml/catalog_product_edit_tab_' .
$this->getRequest()->gerParam('gridOnlyBlock')
)
->toHtml()
);
}
