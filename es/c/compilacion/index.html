El lenguaje C es tradicionalmente un lenguaje compilado (a diferencia de interpretado). El Estándar C define fases de traducción, y el producto de aplicarlas es una imagen de programa (o programa compilado). En [tag:c11], las fases se enumeran en §5.1.1.2.

Extensión de nombre de archivo Descripción
.c Archivo fuente. Por lo general, contiene definiciones y código.
.h Archivo de cabecera. Normalmente contiene declaraciones.
.o archivo de objetos. Código compilado en lenguaje máquina.
.obj Extensión alternativa para archivos de objetos.
.a archivo de biblioteca. Paquete de archivos de objetos.
.dll Biblioteca de vínculos dinámicos en Windows.
.entonces Objeto compartido (biblioteca) en muchos sistemas similares a Unix.
.dylib Biblioteca de vínculos dinámicos en OSX (variante de Unix).
.exe, .com Archivo ejecutable de Windows. Formado por la vinculación de archivos de objetos y archivos de biblioteca. En los sistemas similares a Unix, no existe una extensión de nombre de archivo especial para el archivo ejecutable.
Indicadores del compilador POSIX c99 Descripción
-o nombre de archivo Nombre del archivo de salida, por ejemplo. (bin/programa.exe, programa)
-I directorio buscar encabezados en directory.
-D nombre definir macro nombre
-L directorio buscar bibliotecas en directorio.
-l nombre biblioteca de enlaces libname.

Los compiladores en plataformas POSIX (Linux, mainframes, Mac) suelen aceptar estas opciones, incluso si no se denominan c99.

  • Véase también c99 - compilar programas C estándar
Indicadores de GCC (GNU Compiler Collection) Descripción
-Muro Habilita que todos los mensajes de advertencia comúnmente aceptados sean útiles.
-Wextra Habilita más mensajes de advertencia, puede ser demasiado ruidoso.
-pedante Forzar advertencias donde el código viole el estándar elegido.
-Wconversion Habilite las advertencias sobre la conversión implícita, use con cuidado.
-c Compila archivos fuente sin vincular.
-v Imprime información de compilación.
  • gcc acepta las banderas POSIX y muchas otras.
  • Muchos otros compiladores en plataformas POSIX (clang, compiladores específicos del proveedor) también usan las banderas que se enumeran arriba.
  • Consulte también Invoking GCC para ver muchas más opciones.
Indicadores TCC (Compilador Tiny C) Descripción
-Wimplicit-function-declaration Advierte sobre la declaración de función implícita.
-Wunsupported Advierta sobre funciones de GCC no admitidas que TCC ignora.
-Wwrite-strings Haga que las constantes de cadena sean del tipo const char * en lugar de char *.
-Werror Cancele la compilación si se emiten advertencias.
-Muro Active todas las advertencias, excepto -Werror, -Wunusupported y -Wwrite strings.

Tipos de archivo #

La compilación de programas en C requiere que trabaje con cinco tipos de archivos:

  1. Archivos fuente: estos archivos contienen definiciones de funciones y tienen nombres que terminan en .c por convención. Nota: .cc y .cpp son archivos C++; no archivos C. por ejemplo, foo.c

  2. Archivos de encabezado: estos archivos contienen prototipos de funciones y varias declaraciones de preprocesador (consulte a continuación). Se utilizan para permitir que los archivos de código fuente accedan a funciones definidas externamente. Los archivos de encabezado terminan en .h por convención. por ejemplo, foo.h

  3. Archivos de objetos: estos archivos se producen como salida del compilador. Consisten en definiciones de funciones en forma binaria, pero no son ejecutables por sí mismas. Los archivos de objetos terminan en .o por convención, aunque en algunos sistemas operativos (por ejemplo, Windows, MS-DOS), a menudo terminan en .obj. por ejemplo, foo.o foo.obj

  4. Ejecutables binarios: Estos se producen como resultado de un programa llamado “vinculador”. El enlazador vincula una serie de archivos de objetos para producir un archivo binario que se puede ejecutar directamente. Los ejecutables binarios no tienen un sufijo especial en los sistemas operativos Unix, aunque generalmente terminan en .exe en Windows. por ejemplo, foo foo.exe

  5. Bibliotecas: Una biblioteca es un binario compilado pero no es en sí mismo un ejecutable (es decir, no hay una función main() en una biblioteca). Una biblioteca contiene funciones que pueden ser utilizadas por más de un programa. Una biblioteca debe enviarse con archivos de encabezado que contengan prototipos para todas las funciones de la biblioteca; se debe hacer referencia a estos archivos de encabezado (por ejemplo, #include <library.h>) en cualquier archivo fuente que use la biblioteca. Luego, el enlazador debe remitirse a la biblioteca para que el programa pueda compilarse correctamente. Hay dos tipos de bibliotecas: estáticas y dinámicas.

  • Biblioteca estática: una biblioteca estática (archivos .a para sistemas POSIX y archivos .lib para Windows; no debe confundirse con [archivos de biblioteca de importación DLL] (https://msdn.microsoft.com /en-us/library/windows/desktop/ms682592(v=vs.85).aspx), que también usa la extensión .lib) está integrado estáticamente en el programa. Las bibliotecas estáticas tienen la ventaja de que el programa sabe exactamente qué versión de una biblioteca se utiliza. Por otro lado, los tamaños de los ejecutables son mayores ya que se incluyen todas las funciones de biblioteca utilizadas. por ejemplo, libfoo.a foo.lib
  • Biblioteca dinámica: una biblioteca dinámica (archivos .so para la mayoría de los sistemas POSIX, .dylib para OSX y archivos .dll para Windows) está vinculada dinámicamente en tiempo de ejecución por el programa. A veces también se las denomina bibliotecas compartidas porque muchos programas pueden compartir una imagen de biblioteca. Las bibliotecas dinámicas tienen la ventaja de ocupar menos espacio en disco si más de una aplicación utiliza la biblioteca. Además, permiten actualizaciones de la biblioteca (corrección de errores) sin tener que reconstruir los ejecutables. por ejemplo, foo.so foo.dylib foo.dll

El enlazador #

El trabajo del enlazador es enlazar un grupo de archivos objeto (archivos .o) en un ejecutable binario. El proceso de vinculación implica principalmente resolver direcciones simbólicas en direcciones numéricas. El resultado del proceso de enlace es normalmente un programa ejecutable.

Durante el proceso de vinculación, el vinculador recogerá todos los módulos de objeto especificados en la línea de comando, agregará un código de inicio específico del sistema al frente e intentará resolver todas las referencias externas en el módulo de objeto con definiciones externas en otros archivos de objetos (los archivos de objetos se pueden especificar directamente en la línea de comando o se pueden agregar implícitamente a través de bibliotecas). Luego asignará cargar direcciones para los archivos objeto, es decir, especifica dónde terminarán el código y los datos en el espacio de direcciones del programa terminado. Una vez que tiene las direcciones de carga, puede reemplazar todas las direcciones simbólicas en el código objeto con direcciones numéricas “reales” en el espacio de direcciones del objetivo. El programa está listo para ser ejecutado ahora.

Esto incluye tanto los archivos de objetos que el compilador creó a partir de sus archivos de código fuente como los archivos de objetos que se compilaron previamente para usted y se recopilaron en archivos de biblioteca. Estos archivos tienen nombres que terminan en .a o .so, y normalmente no necesita conocerlos, ya que el enlazador sabe dónde se encuentran la mayoría de ellos y los vinculará automáticamente según sea necesario.

Invocación implícita del enlazador #

Al igual que el preprocesador, el enlazador es un programa separado, a menudo llamado ld (pero Linux usa collect2, por ejemplo). Al igual que el preprocesador, el enlazador se invoca automáticamente cuando usa el compilador. Por lo tanto, la forma normal de usar el enlazador es la siguiente:

% gcc foo.o bar.o baz.o -o myprog

Esta línea le dice al compilador que vincule tres archivos de objetos (foo.o, bar.o y baz.o) en un archivo ejecutable binario llamado myprog. Ahora tiene un archivo llamado myprog que puede ejecutar y que, con suerte, hará algo interesante y/o útil.

Invocación explícita del enlazador #

Es posible invocar el enlazador directamente, pero rara vez es aconsejable y, por lo general, es muy específico de la plataforma. Es decir, las opciones que funcionan en Linux no necesariamente funcionarán en Solaris, AIX, macOS, Windows y de manera similar para cualquier otra plataforma. Si trabaja con GCC, puede usar gcc -v para ver qué se ejecuta en su nombre.

Opciones para el enlazador #

El enlazador también toma algunos argumentos para modificar su comportamiento. El siguiente comando le diría a gcc que vincule foo.o y bar.o, pero también incluya la biblioteca ncurses. 

% gcc foo.o bar.o -o foo -lncurses

Esto es en realidad (más o menos) equivalente a

% gcc foo.o bar.o /usr/lib/libncurses.so -o foo

(aunque libncurses.so podría ser libncurses.a, que es solo un archivo creado con ar). Tenga en cuenta que debe enumerar las bibliotecas (ya sea por nombre de ruta o mediante las opciones -lname) después de los archivos de objetos. Con las bibliotecas estáticas, el orden en que se especifican es importante; a menudo, con bibliotecas compartidas, el orden no importa.

Tenga en cuenta que en muchos sistemas, si está utilizando funciones matemáticas (desde <math.h>), necesita especificar -lm para cargar la biblioteca de matemáticas, pero Mac OS X y macOS Sierra no requieren esto. Hay otras bibliotecas que son bibliotecas separadas en Linux y otros sistemas Unix, pero no en macOS: los subprocesos POSIX y las bibliotecas POSIX en tiempo real y de red son ejemplos. En consecuencia, el proceso de vinculación varía entre plataformas.

Otras opciones de compilación #

Esto es todo lo que necesita saber para comenzar a compilar sus propios programas en C. En general, también recomendamos que utilice la opción de línea de comandos -Wall:

% gcc -Wall -c foo.cc

La opción -Wall hace que el compilador le advierta sobre construcciones de código legales pero dudosas, y lo ayudará a detectar muchos errores muy pronto.

Si desea que el compilador le arroje más advertencias (incluidas las variables que se declaran pero no se usan, el olvido de devolver un valor, etc.), puede usar este conjunto de opciones, ya que -Wall, a pesar del nombre, no t active todas las posibles advertencias:

% gcc -Wall -Wextra -Wfloat-equal -Wundef -Wcast-align -Wwrite-strings -Wlogical-op \
>     -Wmissing-declarations -Wredundant-decls -Wshadow …

Tenga en cuenta que clang tiene una opción -Weverything que realmente activa todas las advertencias en clang.

El compilador #

Después de que el preprocesador de C haya incluido todos los archivos de encabezado y expandido todas las macros, el compilador puede compilar el programa. Lo hace convirtiendo el código fuente C en un archivo de código objeto, que es un archivo que termina en .o y que contiene la versión binaria del código fuente. Sin embargo, el código objeto no es directamente ejecutable. Para hacer un ejecutable, también debe agregar código para todas las funciones de la biblioteca que se #included en el archivo (esto no es lo mismo que incluir las declaraciones, que es lo que hace #include). Este es el trabajo de [el enlazador](https://www.wikiod.com/es/c/compilacion#El enlazador).

En general, la secuencia exacta de cómo invocar un compilador de C depende mucho del sistema que esté utilizando. Aquí estamos usando el compilador GCC, aunque debe tenerse en cuenta que existen muchos más compiladores:

% gcc -Wall -c foo.c

% es el símbolo del sistema del sistema operativo. Esto le dice al compilador que ejecute el preprocesador en el archivo foo.c y luego lo compile en el archivo de código objeto foo.o. La opción -c significa compilar el archivo de código fuente en un archivo de objeto pero no invocar el enlazador. Esta opción -c está disponible en sistemas POSIX, como Linux o macOS; otros sistemas pueden usar una sintaxis diferente.

Si todo su programa está en un archivo de código fuente, puede hacer esto:

% gcc -Wall foo.c -o foo

Esto le dice al compilador que ejecute el preprocesador en foo.c, lo compile y luego lo vincule para crear un ejecutable llamado foo. La opción -o establece que la siguiente palabra en la línea es el nombre del archivo ejecutable binario (programa). Si no especifica -o, (si simplemente escribe gcc foo.c), el ejecutable se llamará a.out por razones históricas.

En general, el compilador sigue cuatro pasos al convertir un archivo .c en un ejecutable:

  1. preprocesamiento: expande textualmente las directivas #include y las macros #define en su archivo .c
  2. compilación: convierte el programa en ensamblador (puede detener el compilador en este paso agregando la opción -S)
  3. ensamblaje: convierte el ensamblaje en código de máquina
  4. enlace: vincula el código objeto a bibliotecas externas para crear un ejecutable

Tenga en cuenta también que el nombre del compilador que estamos usando es GCC, que significa “compilador GNU C” y “colección de compiladores GNU”, según el contexto. Existen otros compiladores de C. Para los sistemas operativos similares a Unix, muchos de ellos tienen el nombre cc, por “compilador C”, que a menudo es un enlace simbólico a algún otro compilador. En los sistemas Linux, cc suele ser un alias de GCC. En macOS u OS-X, apunta a sonido metálico.

Los estándares POSIX exigen actualmente c99 como el nombre de un compilador de C; es compatible con el estándar C99 de forma predeterminada. Las versiones anteriores de POSIX exigían c89 como compilador. POSIX también exige que este compilador comprenda las opciones -c y -o que usamos anteriormente.

Nota: La opción -Wall presente en ambos ejemplos de gcc le dice al compilador que imprima advertencias sobre construcciones cuestionables, lo cual es muy recomendable. También es una buena idea agregar otras [opciones de advertencia] (https://gcc.gnu.org/onlinedocs/gcc/Warning-Options.html), p. -Wextra.

El preprocesador #

Antes de que el compilador de C comience a compilar un archivo de código fuente, el archivo se procesa en una fase de preprocesamiento. Esta fase se puede realizar con un programa separado o integrarse completamente en un ejecutable. En cualquier caso, el compilador lo invoca automáticamente antes de que comience la compilación propiamente dicha. La fase de preprocesamiento convierte su código fuente en otro código fuente o unidad de traducción mediante la aplicación de reemplazos textuales. Puede considerarlo como un código fuente “modificado” o “ampliado”. Esa fuente expandida puede existir como un archivo real en el sistema de archivos, o solo puede almacenarse en la memoria por un corto tiempo antes de seguir procesándose.

Los comandos del preprocesador comienzan con el signo de almohadilla ("#"). Hay varios comandos de preprocesador; dos de los más importantes son:

  1. Define:

    #define is mainly used to define constants. For instance,

     #define BIGNUM 1000000
 int a = BIGNUM;

    becomes

     int a = 1000000;

`#define` is used in this way so as to avoid having to explicitly write out some constant value in many different places in a source code file. This is important in case you need to change the constant value later on; it's much less bug-prone to change it once, in the `#define`, than to have to change it in multiple places scattered all over the code.

    Because #define just does advanced search and replace, you can also declare macros. For instance:

     #define ISTRUE(stm) do{stm = stm ? 1 : 0;}while(0)
 // in the function:
 a = x;
 ISTRUE(a);

    becomes:

     // in the function:
 a = x;
 do {
     a = a ? 1 : 0;
 } while(0);

    At first approximation, this effect is roughly the same as with inline functions, but the preprocessor doesn’t provide type checking for #define macros. This is well known to be error-prone and their use necessitates great caution.

    Also note here, that the preprocessor would also replace comments with a blanks as explained below.

  2. Incluye:

#include se usa para acceder a definiciones de funciones definidas fuera de un archivo de código fuente. Por ejemplo:

    #include <stdio.h>

hace que el preprocesador pegue el contenido de <stdio.h> en el archivo de código fuente en la ubicación de la instrucción #include antes de que se compile. #include casi siempre se usa para incluir archivos de encabezado, que son archivos que contienen principalmente declaraciones de funciones y declaraciones #define. En este caso, usamos #include para poder usar funciones como printf y scanf, cuyas declaraciones se encuentran en el archivo stdio.h. Los compiladores de C no le permiten usar una función a menos que haya sido declarada o definida previamente en ese archivo; Las declaraciones #include son, por lo tanto, la forma de reutilizar el código escrito previamente en sus programas C.

  1. Operaciones lógicas:

    #if defined A || defined B
variable = another_variable + 1;
#else
variable = another_variable * 2;
#endif

    will be changed to:

    variable = another_variable + 1;

si A o B se definieron en alguna parte del proyecto antes. Si este no es el caso, por supuesto que el preprocesador hará esto:

   variable = another_variable * 2;

Esto se usa a menudo para el código, que se ejecuta en diferentes sistemas o se compila en diferentes compiladores. Dado que hay definiciones globales, que son específicas del compilador/sistema, puede probar esas definiciones y siempre dejar que el compilador use el código que compilará con seguridad.

  1. Comentarios

    The Preprocessor replaces all comments in the source file by single spaces. Comments are indicated by // up to the end of the line, or a combination of opening /* and closing */ comment brackets.

Las fases de traducción #

A partir del estándar C 2011, enumerado en §5.1.1.2 Fases de traducción, la traducción del código fuente a la imagen del programa (por ejemplo, el ejecutable) se enumeran para ocurrir en 8 pasos ordenados.

  1. La entrada del archivo de origen se asigna al conjunto de caracteres de origen (si es necesario). Los trígrafos se reemplazan en este paso.
  2. Las líneas de continuación (líneas que terminan con \) se empalman con la siguiente línea.
  3. El código fuente se analiza en espacios en blanco y tokens de preprocesamiento.
  4. Se aplica el preprocesador, que ejecuta directivas, expande macros y aplica pragmas. Cada archivo fuente extraído por #include se somete a las fases de traducción 1 a 4 (recursivamente si es necesario). A continuación, se eliminan todas las directivas relacionadas con el preprocesador.
  5. Los valores del conjunto de caracteres de origen en constantes de caracteres y literales de cadena se asignan al conjunto de caracteres de ejecución.
  6. Los literales de cadena adyacentes entre sí se concatenan.
  7. El código fuente se analiza en tokens, que componen la unidad de traducción.
  8. Se resuelven las referencias externas y se forma la imagen del programa.

Una implementación de un compilador de C puede combinar varios pasos, pero la imagen resultante aún debe comportarse como si los pasos anteriores se hubieran producido por separado en el orden indicado anteriormente.
