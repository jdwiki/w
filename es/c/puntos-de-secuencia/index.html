Estándar internacional ISO/IEC 9899:201x Lenguajes de programación — C

Acceder a un objeto volátil, modificar un objeto, modificar un archivo o llamar a una función que hace cualquiera de esas operaciones son todas side efectos, que son cambios en el estado del entorno de ejecución.

La presencia de un punto de secuencia entre la evaluación de expresiones A y B implica que cada cálculo de valor y efecto secundario asociado con A se secuencia antes de cada cálculo de valor y lado efecto asociado con B.

Aquí está la lista completa de puntos de secuencia del Anexo C del [borrador de prepublicación en línea de 2011] (http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf) del Estándar de lenguaje C:

Puntos de secuencia

1     Los siguientes son los puntos de secuencia descritos en 5.1.2.3:

  • Entre las evaluaciones del designador de función y los argumentos reales en una función llamada y la llamada real. (6.5.2.2).
  • Entre las evaluaciones del primer y segundo operandos de los siguientes operadores: lógico AND && (6.5.13); OR lógico || (6.5.14); coma , (6.5.17).
  • ¿Entre las evaluaciones del primer operando del condicional ? : operador y cualquiera de los operandos segundo y tercero se evalúa (6.5.15).
  • El final de un declarador completo: declarators (6.7.6);
  • Entre la evaluación de una expresión completa y la siguiente expresión completa a ser evaluado. Las siguientes son expresiones completas: un inicializador que no forma parte de un literal compuesto (6.7.9); la expresión en un enunciado de expresión (6.8.3); la expresión de control de una sentencia de selección (if o switch) (6.8.4); la controlar la expresión de una instrucción while o do (6.8.5); cada uno de los (opcional) expresiones de una instrucción for (6.8.5.3); la expresión (opcional) en un return sentencia (6.8.6.4).
  • Inmediatamente antes de que regrese una función de biblioteca (7.1.4).
  • Después de las acciones asociadas con cada conversión de función de entrada/salida formateada especificador (7.21.6, 7.29.2).
  • Inmediatamente antes e inmediatamente después de cada llamada a una función de comparación, y también entre cualquier llamada a una función de comparación y cualquier movimiento de los objetos pasado como argumentos a esa llamada (7.22.5).

Expresiones no secuenciadas #

Las siguientes expresiones son sin secuencia:

a + b;
a - b;
a * b;
a / b;
a % b;
a & b;
a | b;

En los ejemplos anteriores, la expresión a puede evaluarse antes o después de la expresión b, b puede evaluarse antes de a, o incluso pueden estar entremezcladas si corresponden a varias instrucciones.

Una regla similar se aplica a las llamadas a funciones:

f(a, b);

Aquí no solo a y b no están secuenciados (es decir, el operador , en una llamada de función no produce un punto de secuencia), sino también f, la expresión que determina la función que se va a llamar.

Los efectos secundarios pueden aplicarse inmediatamente después de la evaluación o posponerse hasta un momento posterior.

Expresiones como

x++ & x++;
f(x++, x++); /* the ',' in a function call is *not* the same as the comma operator */
x++ * x++;
a[i] = i++;

o

x++ & x;
f(x++, x);
x++ * x;
a[i++] = i;

producirá comportamiento indefinido porque

  • una modificación de un objeto y cualquier otro acceso a él debe ser secuenciado
  • no se especifica el orden de evaluación y el orden en que se aplican los efectos secundarios1.

1 Cualquier cambio en el estado del entorno de ejecución.

Expresiones secuenciadas #

Las siguientes expresiones están secuenciadas:

a && b
a || b
a , b
a ? b : c
for ( a ; b ; c ) { ... }

En todos los casos, la expresión a se evalúa completamente y todos los efectos secundarios se aplican antes de que se evalúen b o c. En el cuarto caso, solo se evaluará uno de b o c. En el último caso, b se evalúa completamente y todos los efectos secundarios se aplican antes de evaluar c.

En todos los casos, la evaluación de la expresión a se secuencia antes de las evaluaciones de b o c (alternativamente, las evaluaciones de b y c se secuencian después de la evaluación de a ).

Así, expresiones como

x++ && x++
x++ ? x++ : y++ 
(x = f()) && x != 0
for ( x = 0; x < 10; x++ ) { ... }
y = (x++, x++);

tienen un comportamiento bien definido.

Expresiones secuenciadas indeterminadamente #

Las llamadas a funciones como f(a) siempre implican un punto de secuencia entre la evaluación de los argumentos y el designador (aquí f y a) y la llamada real. Si dos llamadas de este tipo no están secuenciadas, las dos llamadas de función tienen una secuencia indeterminada, es decir, una se ejecuta antes que la otra y no se especifica el orden.

unsigned counter = 0;

unsingned account(void) {
   return counter++;
}

int main(void) {
   printf("the order is %u %u\n", account(), account());
}

Esta doble modificación implícita de counter durante la evaluación de los argumentos printf es válida, solo que no sabemos cuál de las llamadas viene primero. Como el orden no está especificado, puede variar y no se puede depender de él. Entonces la impresión podría ser:

el orden es 0 1

o

el orden es 1 0

La declaración análoga a la anterior sin llamada de función intermedia

   printf("the order is %u %u\n", counter++, counter++); // undefined behavior

tiene un comportamiento indefinido porque no hay un punto de secuencia entre las dos modificaciones de contador.
