Crear e incluir archivos de encabezado

En C moderno, los archivos de encabezado son herramientas cruciales que deben diseñarse y usarse correctamente. Permiten al compilador cotejar partes compiladas de forma independiente de un programa.

Los encabezados declaran tipos, funciones, macros, etc. que necesitan los consumidores de un conjunto de instalaciones. Todo el código que utiliza cualquiera de esas instalaciones incluye el encabezado. Todo el código que define esas instalaciones incluye el encabezado. Esto permite que el compilador verifique que los usos y las definiciones coincidan.

Introducción #

Hay una serie de pautas a seguir al crear y usar archivos de encabezado en un proyecto C:

  • Idemopotencia

Si un archivo de encabezado se incluye varias veces en una unidad de traducción (TU), no debería interrumpir las compilaciones.

  • Autocontención

Si necesita las instalaciones declaradas en un archivo de cabecera, no debe tiene que incluir cualquier otro encabezado explícitamente.

  • Minimalidad

No debería poder eliminar ninguna información de un encabezado sin haciendo que las compilaciones fallen.

  • Incluya lo que usa (IWYU)

Más preocupante para C++ que para C, pero sin embargo importante en C también. Si el código en una TU (llámelo code.c) usa directamente las funciones declarado por un encabezado (llámelo "headerA.h"), entonces code.c debería #include "headerA.h" directamente, incluso si la TU incluye otra encabezado (llámelo "headerB.h") que, en este momento, incluye "cabeceraA.h".

Ocasionalmente, puede haber razones suficientemente buenas para romper uno o más de estas pautas, pero ambos deben ser conscientes de que están incumpliendo regla y sea consciente de las consecuencias de hacerlo antes de romperla.

Autocontención #

Los encabezados modernos deben ser autónomos, lo que significa que un programa que necesita usar las funciones definidas por header.h puede incluir ese encabezado (#include "header.h") y no preocuparse de si otros encabezados necesitan incluirse primero.

Recomendación: los archivos de encabezado deben ser independientes. #

Reglas históricas #

Históricamente, este ha sido un tema levemente polémico.

Érase una vez otro milenio, el [AT&T Indian Hill C Style and Coding Estándares] (http://www.literateprogramming.com/indhill-annot.pdf) declaró:

Los archivos de encabezado no deben estar anidados. El prólogo de un archivo de encabezado debe, por lo tanto, describir qué otros los encabezados deben ser #included para que el encabezado sea funcional. En casos extremos, donde una gran cantidad de archivos de encabezado deben ser incluido en varios archivos fuente diferentes, es aceptable poner todos los #includes comunes en un archivo de inclusión.

Esta es la antítesis de la autocontención.

Reglas modernas #

Sin embargo, desde entonces, la opinión ha tendido en la dirección opuesta. Si un archivo fuente necesita usar las instalaciones declaradas por un encabezado header.h, el programador debería poder escribir:

#include "header.h"

y (sujeto solo a tener las rutas de búsqueda correctas configuradas en el comando línea), cualquier encabezado de requisito previo necesario será incluido por header.h sin necesidad de agregar más encabezados al archivo fuente.

Esto proporciona una mejor modularidad para el código fuente. También protege la fuente del enigma “adivina por qué se agregó este encabezado” que surge después de que el código ha sido modificado y pirateado durante una o dos décadas.

Los estándares de codificación del Centro Goddard de Vuelos Espaciales (GSFC) de la NASA para C es uno de los estándares más modernos — pero ahora es un poco difícil de rastrear. Establece que los encabezados deben ser autónomos. También proporciona una forma sencilla de garantizar que los encabezados sean autónomos: el archivo de implementación para el encabezado debe incluir el encabezado como el primero encabezamiento. Si no es autónomo, ese código no se compilará.

La justificación dada por GSFC incluye:

§2.1.1 Cabecera incluir justificación

Este estándar requiere que el encabezado de una unidad contenga #include declaraciones para todos los demás encabezados requeridos por el encabezado de la unidad. Colocar #include para el encabezado de la unidad primero en el cuerpo de la unidad permite el compilador para verificar que el encabezado contiene todo lo requerido Declaraciones #include.

Un diseño alternativo, no permitido por esta norma, no permite declaraciones #include en los encabezados; todos los #includes se hacen en el cuerpo archivos Los archivos de encabezado de unidad deben contener declaraciones #ifdef que verifiquen que los encabezados requeridos se incluyen en el orden correcto.

Una ventaja del diseño alternativo es que la lista #include en el archivo del cuerpo es exactamente la lista de dependencias necesaria en un archivo MAKE, y esta lista es verificada por el compilador. Con el diseño estándar, se debe utilizar una herramienta para generar el lista de dependencias. Sin embargo, todos los entornos de desarrollo recomendados por la rama proporcionar una herramienta de este tipo.

Una gran desventaja del diseño alternativo es que si una unidad cambios requeridos en la lista de encabezados, cada archivo que usa esa unidad debe ser editado para actualizar la lista de sentencias #include. Además, la lista de encabezado requerida para una unidad de biblioteca del compilador puede ser diferentes en diferentes objetivos.

Otra desventaja del diseño alternativo es que la biblioteca del compilador los archivos de encabezado y otros archivos de terceros deben modificarse para agregar el sentencias #ifdef requeridas.

Así, la autocontención significa que:

  • Si un encabezado header.h necesita un nuevo encabezado anidado extra.h, no tiene que verificar cada archivo fuente que usa header.h para ver si necesita agregar extra.h.
  • Si un encabezado header.h ya no necesita incluir un encabezado específico notneeded.h, no tiene que comprobar todos los archivos de origen que utilizan header.h para ver si puede eliminar con seguridad notneeded.h (pero consulte [Incluir lo que usa](https://www.wikiod.com/es/c/crear-e-incluir-archivos-de-encabezado /21641/incluye-lo-que-usas-iwyu).
  • No es necesario establecer la secuencia correcta para incluir el encabezados de requisitos previos (que requieren una ordenación topológica para hacer el trabajo adecuadamente).

Comprobación de la autocontención #

Consulte [Enlace contra una estática biblioteca] (http://stackoverflow.com/questions/4507896) para un script chkhdr que se puede usar para probar la idempotencia y la autocontención de un archivo de cabecera.

Minimalidad #

Los encabezados son un mecanismo de verificación de consistencia crucial, pero deben ser lo más pequeño posible. En particular, eso significa que un encabezado no debe incluir otros encabezados solo porque el archivo de implementación necesitará los otros encabezados. Un encabezado debe contener solo los encabezados necesarios para un consumidor de los servicios descritos.

Por ejemplo, un encabezado de proyecto no debe incluir <stdio.h> a menos que uno de las interfaces de función usa el tipo FILE * (o uno de los otros tipos definidos únicamente en <stdio.h>). Si una interfaz usa size_t, el encabezado más pequeño que es suficiente es <stddef.h>. Obviamente, si se incluye otro encabezado que define size_t, hay no es necesario incluir <stddef.h> también.

Si los encabezados son mínimos, entonces mantiene el tiempo de compilación en un mínimo también.

Es posible idear encabezados cuyo único propósito sea incluir mucho de otros encabezados. Estos rara vez resultan ser una buena idea a largo plazo porque pocos los archivos fuente en realidad necesitarán todas las instalaciones descritas por todos los encabezados Por ejemplo, se podría diseñar un <standard-c.h> que incluyera todos los encabezados C estándar — con cuidado ya que algunos encabezados no siempre son presente. Sin embargo, muy pocos programas usan las facilidades de <locale.h> o <tgmath.h>.

  • Consulte también [Cómo vincular varios archivos de implementación en C?] (http://stackoverflow.com/questions/15622409)

Notación y miscelánea #

El estándar C dice que hay muy poca diferencia entre el Notaciones #include <header.h> y #include "header.h".

[#include <header.h>] busca una secuencia de implementaciones definidas lugares para un encabezado identificado únicamente por la secuencia especificada entre los delimitadores < y >, y provoca el reemplazo de ese directiva por todo el contenido del encabezado. Cómo se especifican los lugares o se identifica la cabecera definido por la implementación.

[#include "header.h"] provoca el reemplazo de esa directiva por todo el contenido del archivo fuente identificado por el especificado secuencia entre los delimitadores "…". El archivo de origen nombrado se busca en una implementación definida manera. Si esta búsqueda no es compatible, o si la búsqueda falla, la directiva se reprocesa como si leyera [#include <header.h>] …

Por lo tanto, la forma de comillas dobles puede verse en más lugares que el forma entre corchetes angulares. El estándar especifica por ejemplo que los encabezados estándar deben ser incluido entre paréntesis angulares, aunque la compilación funciona si usa comillas dobles en su lugar. De manera similar, los estándares como POSIX usan el formato de paréntesis angular – y tú también deberías. Reserve los encabezados entre comillas dobles para los encabezados definidos por el proyecto. Para encabezados definidos externamente (incluidos encabezados de otros proyectos) depende de su proyecto), la notación de paréntesis angular es la más apropiada.

Tenga en cuenta que debe haber un espacio entre #include y el encabezado, aunque los compiladores no aceptarán espacio allí. Los espacios son baratos.

Varios proyectos utilizan una notación como:

#include <openssl/ssl.h>
#include <sys/stat.h>
#include <linux/kernel.h>

Debe considerar si usar ese control de espacio de nombres en su proyecto (probablemente sea una buena idea). Debe mantenerse alejado de los nombres utilizados por los proyectos existentes (en particular, tanto sys como linux serían malas elecciones).

Si usa esto, su código debe ser cuidadoso y consistente en el uso de la notación.

No use la notación #include "../include/header.h".

Los archivos de encabezado rara vez deben definir variables. Aunque mantendrá las variables globales al mínimo, si necesita un variable global, la declarará en un encabezado y la definirá en una archivo de origen adecuado, y ese archivo de origen incluirá el encabezado para cotejar la declaración y la definición, y todos los archivos fuente que use the variable usará el encabezado para declararlo.

Corolario: no declarará variables globales en un archivo fuente: un El archivo fuente solo contendrá definiciones.

Los archivos de encabezado rara vez deben declarar funciones estáticas, con la notable excepción de las funciones estáticas en línea que se definirán en los encabezados si la función es necesaria en más de un archivo fuente.

  • Los archivos fuente definen variables globales y funciones globales.
  • Los archivos fuente no declaran la existencia de variables o funciones globales; incluyen el encabezado que declara la variable o función.
  • Los archivos de encabezado declaran funciones y variables globales (y tipos y otro material de apoyo).
  • Los archivos de encabezado no definen variables ni ninguna función excepto las funciones (estáticas) en línea.

Idempotencia #

Si un archivo de encabezado en particular se incluye más de una vez en una unidad de traducción (TU), no debería haber ningún problema de compilación. Esto se denomina ‘idempotencia’; sus encabezados deben ser idempotentes. Piense en lo difícil que sería la vida si tuviera que asegurarse de que #include <stdio.h> solo se incluyera una vez.

Hay dos formas de lograr la idempotencia: los protectores de cabecera y la directiva #pragma once.

Protectores de cabecera #

Los protectores de cabecera son simples y confiables y cumplen con el estándar C. Las primeras líneas que no son de comentario en un archivo de encabezado deben tener el formato:

#ifndef UNIQUE_ID_FOR_HEADER
#define UNIQUE_ID_FOR_HEADER

La última línea sin comentarios debe ser #endif, opcionalmente con un comentario después:

#endif /* UNIQUE_ID_FOR_HEADER */

Todo el código operativo, incluidas otras directivas #include, debe estar entre estas líneas.

Cada nombre debe ser único. A menudo, se utiliza un esquema de nombres como HEADER_H_INCLUDED. Algunos códigos antiguos usan un símbolo definido como protección de encabezado (por ejemplo, #ifndef BUFSIZ en <stdio.h>), pero no es tan confiable como un nombre único.

Una opción sería usar un hash MD5 (u otro) generado para el nombre de protección de encabezado. Debe evitar emular los esquemas usados ​​por los encabezados del sistema que frecuentemente usan nombres reservados para la implementación — nombres que comienzan con un guión bajo seguido de otro guión bajo o una letra mayúscula.

La directiva #pragma once #

Alternativamente, algunos compiladores admiten la directiva #pragma once que tiene el mismo efecto que las tres líneas que se muestran para los protectores de encabezado.

#pragma once

Los compiladores que admiten #pragma once incluyen MS Visual Studio y GCC y Clang. Sin embargo, si la portabilidad es una preocupación, es mejor usar protectores de encabezado o usar ambos. Los compiladores modernos (aquellos que admiten C89 o posterior) deben ignorar, sin comentarios, los pragmas que no reconocen (“Cualquier pragma que no sea reconocido por la implementación se ignora”), pero las versiones antiguas de GCC no eran tan indulgentes.

Incluye lo que usas (IWYU) #

El proyecto Include What You Use de Google, o IWYU, garantiza que los archivos de origen incluyan todos los encabezados utilizados en el código.

Supongamos que un archivo fuente source.c incluye un encabezado arbitrary.h que a su vez incluye coincidentemente freeloader.h, pero el archivo fuente también usa explícita e independientemente las funciones de freeloader.h. Todo está bien para empezar. Luego, un día arbitrary.h se cambia para que sus clientes ya no necesiten las instalaciones de freeloader.h. De repente, source.c deja de compilarse porque no cumplía con los criterios de IWYU. Debido a que el código en source.c usó explícitamente las funciones de freeloader.h, debería haber incluido lo que usa; también debería haber un #include "freeloader.h" explícito en el código fuente. (Idempotency se habría asegurado de que no hubiera ningún problema).

La filosofía IWYU maximiza la probabilidad de que el código continúe compilando incluso con cambios razonables realizados en las interfaces. Claramente, si su código llama a una función que posteriormente se elimina de la interfaz publicada, ninguna cantidad de preparación puede evitar que los cambios sean necesarios. Esta es la razón por la que se evitan los cambios en las API cuando es posible y por la que hay ciclos de desuso en múltiples versiones, etc.

Este es un problema particular en C++ porque se permite que los encabezados estándar se incluyan entre sí. El archivo fuente file.cpp podría incluir un encabezado header1.h que en una plataforma incluye otro encabezado header2.h. Puede que file.cpp utilice las funciones de header2.h también. Esto no sería un problema inicialmente - el código se compilaría porque header1.h incluye header2.h. En otra plataforma, o en una actualización de la plataforma actual, header1.h podría revisarse para que ya no incluya header2.h y, como resultado, file.cpp dejaría de compilarse.

IWYU detectaría el problema y recomendaría que header2.h se incluyera directamente en file.cpp. Esto aseguraría que continúe compilando. Consideraciones análogas se aplican también al código C.
