Mezclar enteros con y sin signo en operaciones aritméticas #

Por lo general, no es una buena idea mezclar enteros “con signo” y “sin signo” en operaciones aritméticas. Por ejemplo, ¿cuál será el resultado del siguiente ejemplo?

#include <stdio.h>

int main(void)
{ 
    unsigned int a = 1000;
    signed int b = -1;

    if (a > b) puts("a is more than b");
    else puts("a is less or equal than b"); 

    return 0;
}

Dado que 1000 es más que -1, esperaría que la salida fuera “a es más que b”, sin embargo, ese no será el caso.

Las operaciones aritméticas entre diferentes tipos integrales se realizan dentro de un tipo común definido por las llamadas conversiones aritméticas usuales (ver la especificación del lenguaje, 6.3.1.8).

En este caso, el “tipo común” es int sin signo, porque, como se indica en Conversiones aritméticas habituales,

714  De lo contrario, si el operando que tiene tipo entero sin signo tiene rango mayor o igual al rango del tipo del otro operando, entonces el operando con tipo entero con signo se convierte al tipo de el operando de tipo entero sin signo.

Esto significa que el operando int b se convertirá a unsigned int antes de la comparación.

Cuando -1 se convierte en un ‘int sin signo’, el resultado es el valor máximo posible de ‘int sin signo’, que es mayor que 1000, lo que significa que ‘a > b’ es falso.

Las macros son simples reemplazos de cadenas #

Las macros son simples reemplazos de cadenas. (Estrictamente hablando, funcionan con tokens de preprocesamiento, no con cadenas arbitrarias).

#include <stdio.h>

#define SQUARE(x) x*x

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Puede esperar que este código imprima 9 (3*3), pero en realidad se imprimirá 5 porque la macro se expandirá a 1+2*1+2.

Debe envolver los argumentos y toda la expresión de la macro entre paréntesis para evitar este problema.

#include <stdio.h>

#define SQUARE(x) ((x)*(x))

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Otro problema es que no se garantiza que los argumentos de una macro se evalúen una vez; es posible que no se evalúen en absoluto o que se evalúen varias veces.

#include <stdio.h>

#define MIN(x, y) ((x) <= (y) ? (x) : (y))

int main(void) {
    int a = 0;
    printf("%d\n", MIN(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

En este código, la macro se expandirá a ((a++) <= (10) ? (a++) : (10)). Dado que a++ (0) es menor que 10, a++ se evaluará dos veces y hará que el valor de a y lo que devuelve MIN difieran de lo esperado.

Esto se puede evitar mediante el uso de funciones, pero tenga en cuenta que los tipos serán fijados por la definición de la función, mientras que las macros pueden ser (demasiado) flexibles con los tipos.

#include <stdio.h>

int min(int x, int y) {
    return x <= y ? x : y;
}

int main(void) {
    int a = 0;
    printf("%d\n", min(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

Ahora el problema de la evaluación doble está arreglado, pero esta función min no puede manejar datos dobles sin truncar, por ejemplo.

Las macro directivas pueden ser de dos tipos:

#define OBJECT_LIKE_MACRO     followed by a "replacement list" of preprocessor tokens
#define FUNCTION_LIKE_MACRO(with, arguments) followed by a replacement list

Lo que distingue a estos dos tipos de macros es el carácter que sigue al identificador después de #define: si es un lparen, es una macro similar a una función; de lo contrario, es una macro similar a un objeto. Si la intención es escribir una macro similar a una función, no debe haber ningún espacio en blanco entre el final del nombre de la macro y (. Consulte esto para obtener una explicación detallada.

En C99 o posterior, podría usar static inline int min(int x, int y) { … }.

En C11, podría escribir una expresión ’tipo-genérica’ para ‘min’.

#include <stdio.h>

#define min(x, y) _Generic((x), \
                        long double: min_ld, \
                        unsigned long long: min_ull, \
                        default: min_i \
                        )(x, y)

#define gen_min(suffix, type) \
    static inline type min_##suffix(type x, type y) { return (x < y) ? x : y; }

gen_min(ld, long double)
gen_min(ull, unsigned long long)
gen_min(i, int)

int main(void)
{
    unsigned long long ull1 = 50ULL;
    unsigned long long ull2 = 37ULL;
    printf("min(%llu, %llu) = %llu\n", ull1, ull2, min(ull1, ull2));
    long double ld1 = 3.141592653L;
    long double ld2 = 3.141592652L;
    printf("min(%.10Lf, %.10Lf) = %.10Lf\n", ld1, ld2, min(ld1, ld2));
    int i1 = 3141653;
    int i2 = 3141652;
    printf("min(%d, %d) = %d\n", i1, i2, min(i1, i2));
    return 0;
}

La expresión genérica podría extenderse con más tipos como doble, flotante, largo largo, unsigned long, long, unsigned, y las invocaciones de macro apropiadas gen_min escritas.

Olvidar copiar el valor de retorno de realloc en un temporal #

Si realloc falla, devuelve NULL. Si asigna el valor del búfer original al valor de retorno de realloc, y si devuelve NULL, entonces el búfer original (el puntero antiguo) se pierde, lo que resulta en una pérdida de memoria . La solución es copiar en un puntero temporal, y si ese temporal no es NULL, entonces copiar en el búfer real.

char *buf, *tmp;

buf = malloc(...);
...

/* WRONG */
if ((buf = realloc(buf, 16)) == NULL)
    perror("realloc");

/* RIGHT */
if ((tmp = realloc(buf, 16)) != NULL)
    buf = tmp;
else
    perror("realloc");

Olvidar asignar un byte extra para \0 #

Cuando esté copiando una cadena en un búfer mallocado, recuerde siempre agregar 1 a strlen.

char *dest = malloc(strlen(src)); /* WRONG */
char *dest = malloc(strlen(src) + 1); /* RIGHT */

strcpy(dest, src);

Esto se debe a que strlen no incluye el final \0 en la longitud. Si toma el enfoque ‘INCORRECTO’ (como se muestra arriba), al llamar a ‘strcpy’, su programa invocará un comportamiento indefinido.

También se aplica a situaciones en las que está leyendo una cadena de longitud máxima conocida de stdin o alguna otra fuente. Por ejemplo

#define MAX_INPUT_LEN 42

char buffer[MAX_INPUT_LEN]; /* WRONG */
char buffer[MAX_INPUT_LEN + 1]; /* RIGHT */

scanf("%42s", buffer);  /* Ensure that the buffer is not overflowed */

Malentendido de la descomposición de la matriz #

Un problema común en el código que usa matrices multidimensionales, matrices de punteros, etc. es el hecho de que Type** y Type[M][N] son tipos fundamentalmente diferentes:

#include <stdio.h>

void print_strings(char **strings, size_t n)
{
    size_t i;
    for (i = 0; i < n; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(s, 4);
    return 0;
}

Ejemplo de salida del compilador:

file1.c: In function 'main':
file1.c:13:23: error: passing argument 1 of 'print_strings' from incompatible pointer type [-Wincompatible-pointer-types]
         print_strings(strings, 4);
                       ^
file1.c:3:10: note: expected 'char **' but argument is of type 'char (*)[20]'
     void print_strings(char **strings, size_t n)

El error indica que la matriz s en la función main se pasa a la función print_strings, que espera un tipo de puntero diferente al que recibió. También incluye una nota que expresa el tipo que espera print_strings y el tipo que se le pasó desde main.

El problema se debe a algo llamado array decay. Lo que sucede cuando se pasa s con su tipo char[4][20] (arreglo de 4 arreglos de 20 caracteres) a la función es que se convierte en un puntero a su primer elemento como si hubiera escrito &s[ 0], que tiene el tipo char (*)[20] (puntero a 1 matriz de 20 caracteres). Esto ocurre para cualquier matriz, incluida una matriz de punteros, una matriz de matrices de matrices (matrices 3D) y una matriz de punteros a una matriz. A continuación se muestra una tabla que ilustra lo que sucede cuando una matriz decae. Los cambios en la descripción del tipo se resaltan para ilustrar lo que sucede:

antes de la decadencia   después de la decadencia  
carácter [20] matriz de (20 caracteres) char * puntero a (1 carácter)
carácter [4][20] matriz de (4 matrices de 20 caracteres) char (*)[20] puntero a (1 matriz de 20 caracteres)
carácter *[4] matriz de (4 punteros a 1 carácter) char ** puntero a (1 puntero a 1 carácter)
carácter [3][4][20] matriz de (3 matrices de 4 matrices de 20 caracteres) char (*)[4][20] puntero a (1 matriz de 4 matrices de 20 caracteres)
char (*[4])[20] matriz de (4 punteros a 1 matriz de 20 caracteres) char (**)[20] puntero a (1 puntero a 1 matriz de 20 caracteres)

Si una matriz puede decaer en un puntero, entonces se puede decir que un puntero puede considerarse una matriz de al menos 1 elemento. Una excepción a esto es un puntero nulo, que no apunta a nada y, por lo tanto, no es una matriz.

El decaimiento de la matriz solo ocurre una vez. Si una matriz se ha convertido en un puntero, ahora es un puntero, no una matriz. Incluso si tiene un puntero a una matriz, recuerde que el puntero puede considerarse una matriz de al menos un elemento, por lo que ya se ha producido la descomposición de la matriz.

En otras palabras, un puntero a una matriz (char (*)[20]) nunca se convertirá en un puntero a un puntero (char **). Para arreglar la función print_strings, simplemente haz que reciba el tipo correcto:

void print_strings(char (*strings)[20], size_t n)
/* OR */
void print_strings(char strings[][20], size_t n)

Surge un problema cuando desea que la función print_strings sea genérica para cualquier matriz de caracteres: ¿qué sucede si hay 30 caracteres en lugar de 20? ¿O 50? La respuesta es agregar otro parámetro antes del parámetro de matriz:

#include <stdio.h>

/*
 * Note the rearranged parameters and the change in the parameter name
 * from the previous definitions:
 *      n (number of strings)
 *   => scount (string count)
 *
 * Of course, you could also use one of the following highly recommended forms
 * for the `strings` parameter instead:
 *
 *    char strings[scount][ccount]
 *    char strings[][ccount]
 */
void print_strings(size_t scount, size_t ccount, char (*strings)[ccount])
{
    size_t i;
    for (i = 0; i < scount; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(4, 20, s);
    return 0;
}

Compilarlo no produce errores y da como resultado el resultado esperado:

Example 1
Example 2
Example 3
Example 4

Olvidarse de liberar memoria (fugas de memoria) #

Una mejor práctica de programación es liberar cualquier memoria que haya sido asignada directamente por su propio código, o implícitamente llamando a una función interna o externa, como una API de biblioteca como strdup(). No liberar memoria puede provocar una fuga de memoria, que podría acumularse en una cantidad sustancial de memoria desperdiciada que no está disponible para su programa (o el sistema), lo que posiblemente provoque bloqueos o un comportamiento indefinido. Es más probable que ocurran problemas si la fuga se produce repetidamente en un bucle o en una función recursiva. El riesgo de falla del programa aumenta cuanto más tiempo se ejecuta un programa con fugas. A veces, los problemas aparecen instantáneamente; otras veces, los problemas no se verán durante horas o incluso años de operación constante. Las fallas por agotamiento de la memoria pueden ser catastróficas, según las circunstancias.

El siguiente ciclo infinito es un ejemplo de una fuga que eventualmente agotará la fuga de memoria disponible llamando a getline(), una función que asigna implícitamente nueva memoria, sin liberar esa memoria.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    /* The loop below leaks memory as fast as it can */

    for(;;) { 
        getline(&line, &size, stdin); /* New memory implicitly allocated */

        /* <do whatever> */

        line = NULL;
    }

    return 0;
}

Por el contrario, el siguiente código también usa la función getline(), pero esta vez, la memoria asignada se libera correctamente, evitando una fuga.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    for(;;) {
        if (getline(&line, &size, stdin) < 0) {
            free(line);
            line = NULL;

            /* Handle failure such as setting flag, breaking out of loop and/or exiting */
        }

        /* <do whatever> */

        free(line);
        line = NULL;

    }

    return 0;
}

La pérdida de memoria no siempre tiene consecuencias tangibles y no es necesariamente un problema funcional. Si bien las “mejores prácticas” dictan la liberación rigurosa de la memoria en puntos y condiciones estratégicos para reducir el uso de la memoria y reducir el riesgo de agotamiento de la memoria, puede haber excepciones. Por ejemplo, si un programa tiene una duración y un alcance limitados, el riesgo de falla en la asignación podría considerarse demasiado pequeño como para preocuparse. En ese caso, eludir la desasignación explícita podría considerarse aceptable. Por ejemplo, la mayoría de los sistemas operativos modernos liberan automáticamente toda la memoria consumida por un programa cuando finaliza, ya sea debido a una falla del programa, una llamada del sistema a exit(), la terminación del proceso o el final de main(). La liberación explícita de memoria en el punto de finalización inminente del programa podría ser redundante o introducir una penalización en el rendimiento.

La asignación puede fallar si no hay suficiente memoria disponible, y las fallas de manejo deben tenerse en cuenta en los niveles apropiados de la pila de llamadas. getline(), que se muestra arriba, es un caso de uso interesante porque es una función de biblioteca que no solo asigna memoria que deja libre a la persona que llama, sino que puede fallar por varias razones, todas las cuales deben tenerse en cuenta . Por lo tanto, es esencial leer la documentación (página del manual) al usar una API de C, prestar especial atención a las condiciones de error y al uso de la memoria, y ser consciente de qué capa de software soporta la carga de liberar la memoria devuelta. .

Otra práctica común de manejo de memoria es establecer constantemente los punteros de memoria en NULL inmediatamente después de que se libera la memoria a la que hacen referencia esos punteros, de modo que se pueda probar la validez de esos punteros en cualquier momento (p. puede provocar problemas graves, como la obtención de datos basura (operación de lectura), la corrupción de datos (operación de escritura) y/o el bloqueo del programa. En la mayoría de los sistemas operativos modernos, liberar la ubicación de memoria 0 (“NULL”) es un NOP (por ejemplo, es inofensivo), como lo requiere el estándar C, por lo que al establecer un puntero en NULL, no hay riesgo de doble liberación de memoria si el puntero se pasa a free(). Tenga en cuenta que la doble liberación de memoria puede llevar a fallas que consumen mucho tiempo, son confusas y difíciles de diagnosticar.

Escribir erróneamente = en lugar de == al comparar #

El operador = se utiliza para la asignación.

El operador == se utiliza para la comparación.

Se debe tener cuidado de no mezclar los dos. A veces uno escribe por error

/* assign y to x */
if (x = y) {
     /* logic */
}

cuando lo que realmente se quería es:

/* compare if x is equal to y */
if (x == y) {
    /* logic */
}

El primero asigna valor de y a x y comprueba si ese valor es distinto de cero, en lugar de hacer una comparación, lo que equivale a:

if ((x = y) != 0) {
    /* logic */
}

Hay momentos en los que se pretende probar el resultado de una tarea y se usa comúnmente, porque evita tener que duplicar el código y tener que tratar la primera vez de manera especial. Comparar

while ((c = getopt_long(argc, argv, short_options, long_options, &option_index)) != -1) {
        switch (c) {
        ...
        }
}

versus

c = getopt_long(argc, argv, short_options, long_options, &option_index);
while (c != -1) {
        switch (c) {
        ...
        }
        c = getopt_long(argc, argv, short_options, long_options, &option_index);
}

Los compiladores modernos reconocerán este patrón y no advertirán cuando la asignación esté entre paréntesis como arriba, pero pueden advertir para otros usos. Por ejemplo:

if (x = y)         /* warning */

if ((x = y))       /* no warning */
if ((x = y) != 0)  /* no warning; explicit */

Algunos programadores usan la estrategia de poner la constante a la izquierda del operador (comúnmente llamado condiciones de Yoda). Debido a que las constantes son valores r, este estilo de condición hará que el compilador arroje un error si se usó el operador incorrecto.

if (5 = y) /* Error */

if (5 == y) /* No error */

Sin embargo, esto reduce severamente la legibilidad del código y no se considera necesario si el programador sigue buenas prácticas de codificación en C, y no ayuda cuando se comparan dos variables, por lo que no es una solución universal. Además, muchos compiladores modernos pueden dar advertencias cuando el código se escribe con condiciones Yoda.

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';

Errores de referencia indefinidos al vincular #

Uno de los errores más comunes en la compilación ocurre durante la etapa de vinculación. El error se parece a esto:

$ gcc undefined_reference.c 
/tmp/ccoXhwF0.o: In function `main':
undefined_reference.c:(.text+0x15): undefined reference to `foo'
collect2: error: ld returned 1 exit status
$

Entonces, veamos el código que generó este error:

int foo(void);

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Vemos aquí una declaración de foo (int foo();) pero ninguna definición de la misma (función real). Así que proporcionamos al compilador el encabezado de la función, pero no había tal función definida en ninguna parte, por lo que la etapa de compilación pasa pero el enlazador sale con un error de Referencia no definida. Para corregir este error en nuestro pequeño programa solo tendríamos que agregar una definición para foo:

/* Declaration of foo */
int foo(void);

/* Definition of foo */
int foo(void)
{
    return 5;
}

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Ahora este código se compilará. Surge una situación alternativa donde la fuente de foo() está en un archivo fuente separado foo.c (y hay un encabezado foo.h para declarar foo() que está incluido tanto en foo.c yreferencia_indefinida.c). Luego, la solución es vincular el archivo objeto de foo.cyundefined_reference.c`, o compilar ambos archivos fuente:

$ gcc -c undefined_reference.c 
$ gcc -c foo.c
$ gcc -o working_program undefined_reference.o foo.o
$

O:

$ gcc -o working_program undefined_reference.c foo.c
$

Un caso más complejo es donde están involucradas las bibliotecas, como en el código:

#include <stdio.h>
#include <stdlib.h>
#include <math.h>

int main(int argc, char **argv)
{
    double first;
    double second;
    double power;

    if (argc != 3)
    {
        fprintf(stderr, "Usage: %s <denom> <nom>\n", argv[0]);
        return EXIT_FAILURE;
    }

    /* Translate user input to numbers, extra error checking
     * should be done here. */
    first = strtod(argv[1], NULL);
    second = strtod(argv[2], NULL);

    /* Use function pow() from libm - this will cause a linkage 
     * error unless this code is compiled against libm! */
    power = pow(first, second);

    printf("%f to the power of %f = %f\n", first, second, power);

    return EXIT_SUCCESS;
}

El código es sintácticamente correcto, la declaración para pow() existe desde #include <math.h>, así que tratamos de compilar y enlazar pero obtenemos un error como este:

$ gcc no_library_in_link.c -o no_library_in_link
/tmp/ccduQQqA.o: In function `main':
no_library_in_link.c:(.text+0x8b): undefined reference to `pow'
collect2: error: ld returned 1 exit status
$

Esto sucede porque la definición para pow() no se encontró durante la etapa de vinculación. Para solucionar esto, tenemos que especificar que queremos enlazar con la biblioteca matemática llamada libm especificando el indicador -lm. (Tenga en cuenta que hay plataformas como macOS donde no se necesita -lm, pero cuando obtiene la referencia indefinida, se necesita la biblioteca).

Así que ejecutamos la etapa de compilación nuevamente, esta vez especificando la biblioteca (después de los archivos fuente u objeto):

$ gcc no_library_in_link.c -lm -o library_in_link_cmd
$ ./library_in_link_cmd 2 4
2.000000 to the power of 4.000000 = 16.000000
$

Verificando la expresión lógica contra ‘verdadero’ #

El estándar C original no tenía un tipo booleano intrínseco, por lo que “bool”, “verdadero” y “falso” no tenían un significado inherente y, a menudo, los programadores los definían. Por lo general, verdadero se definiría como 1 y falso se definiría como 0.

C99 agrega el tipo incorporado _Bool y el encabezado <stdbool.h> que define bool (que se expande a _Bool), false y true. También le permite redefinir bool, true y false, pero señala que esta es una característica obsoleta.

Más importante aún, las expresiones lógicas tratan todo lo que se evalúa como cero como falso y cualquier evaluación distinta de cero como verdadera. Por ejemplo:

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == true)  /* Comparison only succeeds if true is 0x80 and bitField has that bit set */
    {
        return true;
    }
    else
    {
        return false;
    }
}

En el ejemplo anterior, la función está tratando de verificar si el bit superior está configurado y devolver “verdadero” si lo está. Sin embargo, al verificar explícitamente contra true, la declaración if solo tendrá éxito si (bitfield & 0x80) se evalúa como lo que true se define como, que normalmente es 1 y muy raramente 0x80. Verifica explícitamente contra el caso que esperas:

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == 0x80) /* Explicitly test for the case we expect */
    {
        return true;
    }
    else
    {
        return false;
    }
}

O evalúe cualquier valor distinto de cero como verdadero.

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    /* If upper bit is set, result is 0x80 which the if will evaluate as true */
    if (bitField & 0x80)
    {
        return true;
    }
    else
    {
        return false;
    }
}

Hacer escalado adicional en aritmética de punteros #

En la aritmética de punteros, el número entero a sumar o restar al puntero no se interpreta como un cambio de dirección sino como un número de elementos a mover.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + sizeof(int) * 2; /* wrong */
    printf("%d %d\n", *ptr, *ptr2);
    return 0;
}

Este código hace una escala adicional al calcular el puntero asignado a ptr2. Si sizeof(int) es 4, lo que es típico en los entornos modernos de 32 bits, la expresión significa “8 elementos después de array[0]”, lo que está fuera de rango e invoca *comportamiento indefinido *.

Para que ptr2 apunte a lo que son 2 elementos después de array[0], simplemente debe agregar 2.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + 2;
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

La aritmética de punteros explícita que usa operadores aditivos puede ser confusa, por lo que usar subíndices de matriz puede ser mejor.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = &ptr[2];
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

E1[E2] es idéntico a (*((E1)+(E2))) ([N1570](http://chimera.roma1.infn.it/SP/COMMON/iso-iec-9899- 1990.pdf) 6.5.2.1, párrafo 2), y &(E1[E2]) es equivalente a ((E1)+(E2)) (N1570 6.5.3.2, nota al pie 102).

Alternativamente, si se prefiere la aritmética de punteros, convertir el puntero para abordar un tipo de datos diferente puede permitir el direccionamiento de bytes. Sin embargo, tenga cuidado: [endianness] 1 puede convertirse en un problema, y ​​la conversión a tipos que no sean ‘apuntador a carácter’ conduce a [problemas de alias estrictos] 2.

#include <stdio.h>

int main(void) {
    int array[3] = {1,2,3};  // 4 bytes * 3 allocated
    unsigned char *ptr = (unsigned char *) array;  // unsigned chars only take 1 byte
    /*
     * Now any pointer arithmetic on ptr will match
     * bytes in memory.  ptr can be treated like it
     * was declared as: unsigned char ptr[12];
     */

    return 0;
}

Comparando números de coma flotante #

Los tipos de punto flotante (float, double y long double) no pueden representar con precisión algunos números porque tienen una precisión finita y representan los valores en un formato binario. Al igual que tenemos decimales periódicos en base 10 para fracciones como 1/3, también hay fracciones que no se pueden representar finitamente en binario (como 1/3, pero también, lo que es más importante, 1/10). No compare directamente los valores de punto flotante; use un delta en su lugar.

#include <float.h> // for DBL_EPSILON and FLT_EPSILON
#include <math.h>  // for fabs()

int main(void)
{
    double a = 0.1; // imprecise: (binary) 0.000110...

    // may be false or true
    if (a + a + a + a + a + a + a + a + a + a == 1.0) {
        printf("10 * 0.1 is indeed 1.0. This is not guaranteed in the general case.\n");
    }

    // Using a small delta value.
    if (fabs(a + a + a + a + a + a + a + a + a + a - 1.0) < 0.000001) {
        // C99 5.2.4.2.2p8 guarantees at least 10 decimal digits
        // of precision for the double type.
        printf("10 * 0.1 is almost 1.0.\n");
    }

    return 0;
}

Ignorar los valores de retorno de las funciones de biblioteca #

Casi todas las funciones en la biblioteca estándar de C devuelven algo en caso de éxito y algo más en caso de error. Por ejemplo, malloc devolverá un puntero al bloque de memoria asignado por la función en caso de éxito y, si la función no pudo asignar el bloque de memoria solicitado, un puntero nulo. Por lo tanto, siempre debe verificar el valor de retorno para facilitar la depuración.

Esto es malo:

char* x = malloc(100000000000UL * sizeof *x);
/* more code */
scanf("%s", x); /* This might invoke undefined behaviour and if lucky causes a segmentation violation, unless your system has a lot of memory */

Esto es bueno:

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char* x = malloc(100000000000UL * sizeof *x);
    if (x == NULL) {
        perror("malloc() failed");
        exit(EXIT_FAILURE);
    }

    if (scanf("%s", x) != 1) {
        fprintf(stderr, "could not read string\n");
        free(x);
        exit(EXIT_FAILURE);
    }

    /* Do stuff with x. */

    /* Clean up. */
    free(x);

    return EXIT_SUCCESS;
}

De esta manera, sabrá de inmediato la causa del error; de lo contrario, podría pasar horas buscando un error en un lugar completamente equivocado.

Los literales de punto flotante son de tipo doble por defecto #

Se debe tener cuidado al inicializar variables de tipo float a valores literales o compararlas con valores literales, porque los literales regulares de punto flotante como 0.1 son de tipo doble. Esto puede dar lugar a sorpresas:

#include <stdio.h>
int main() {
    float  n;
    n = 0.1;
    if (n > 0.1) printf("Wierd\n");
    return 0;
}
// Prints "Wierd" when n is float

Aquí, n se inicializa y se redondea a precisión simple, lo que da como resultado el valor 0.10000000149011612. Luego, n se vuelve a convertir a doble precisión para compararlo con el literal 0.1 (que equivale a 0.10000000000000001), lo que da como resultado una discrepancia.

Además de los errores de redondeo, mezclar variables “flotantes” con literales “dobles” dará como resultado un rendimiento deficiente en plataformas que no tienen soporte de hardware para doble precisión.

Uso de constantes de caracteres en lugar de literales de cadena y viceversa #

En C, las constantes de caracteres y los literales de cadena son cosas diferentes.

Un carácter entre comillas simples como 'a' es una constante de carácter. Una constante de carácter es un número entero cuyo valor es el código de carácter que representa el carácter. Cómo interpretar constantes de caracteres con múltiples caracteres como 'abc' está definido por la implementación.

Cero o más caracteres entre comillas dobles como "abc" es un literal de cadena. Un literal de cadena es una matriz no modificable cuyos elementos son de tipo char. La cadena entre comillas dobles más el carácter nulo final son los contenidos, por lo que "abc" tiene 4 elementos ({'a', 'b', 'c', '\0'})

En este ejemplo, se usa una constante de carácter donde se debe usar un literal de cadena. Esta constante de carácter se convertirá en un puntero de una manera definida por la implementación y hay pocas posibilidades de que el puntero convertido sea válido, por lo que este ejemplo invocará un comportamiento indefinido.

#include <stdio.h>

int main(void) {
    const char *hello = 'hello, world'; /* bad */
    puts(hello);
    return 0;
}

En este ejemplo, se usa un literal de cadena donde se debe usar una constante de carácter. El puntero convertido del literal de cadena se convertirá en un entero de una manera definida por la implementación, y se convertirá en char de una manera definida por la implementación. (Cómo convertir un entero a un tipo con signo que no puede representar el valor a convertir está definido por la implementación, y si char está firmado también está definido por la implementación). La salida será algo sin sentido.

#include <stdio.h>

int main(void) {
    char c = "a"; /* bad */
    printf("%c\n", c);
    return 0;
}

En casi todos los casos, el compilador se quejará de estas confusiones. Si no es así, debe usar más opciones de advertencia del compilador, o se recomienda que use un mejor compilador.

Función recursiva: se pierde la condición base #

Calcular el factorial de un número es un ejemplo clásico de una función recursiva.

Falta la condición base:

#include <stdio.h>

int factorial(int n)
{
       return n * factorial(n - 1);
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Salida típica: Error de segmentación: 11

El problema con esta función es que se repetiría infinitamente, causando una falla de segmentación: necesita una condición base para detener la recursividad.

Condición base declarada:

#include <stdio.h>

int factorial(int n)
{
    if (n == 1) // Base Condition, very crucial in designing the recursive functions.
    {
       return 1;
    }
    else
    {
       return n * factorial(n - 1);
    }
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Salida de muestra

Factorial 3 = 6

Esta función terminará tan pronto como llegue a la condición n es igual a 1 (siempre que el valor inicial de n sea lo suficientemente pequeño; el límite superior es 12 cuando int es una cantidad de 32 bits).

Reglas a seguir:

  1. Inicialice el algoritmo. Los programas recursivos a menudo necesitan un valor semilla para comenzar. Esto se logra utilizando un parámetro que se pasa a la función o proporcionando una función de puerta de enlace que no es recursiva pero que configura los valores iniciales para el cálculo recursivo.
  2. Verifique si los valores actuales que se procesan coinciden con el caso base. Si es así, procese y devuelva el valor.
  3. Redefina la respuesta en términos de un subproblema o subproblemas más pequeños o más simples.
  4. Ejecute el algoritmo en el subproblema.
  5. Combinar los resultados en la formulación de la respuesta.
  6. Devuelva los resultados.

Fuente: Función recursiva

Pasar matrices no adyacentes a funciones que esperan matrices multidimensionales “reales” #

Al asignar matrices multidimensionales con malloc, calloc y realloc, un patrón común es asignar las matrices internas con múltiples llamadas (incluso si la llamada solo aparece una vez, puede estar en un bucle):

/* Could also be `int **` with malloc used to allocate outer array. */
int *array[4];
int i;

/* Allocate 4 arrays of 16 ints. */
for (i = 0; i < 4; i++)
    array[i] = malloc(16 * sizeof(*array[i]));

La diferencia en bytes entre el último elemento de una de las matrices internas y el primer elemento de la siguiente matriz interna puede no ser 0 como lo serían con una matriz multidimensional “real” (por ejemplo, int matriz[4][16]; ):

/* 0x40003c, 0x402000 */
printf("%p, %p\n", (void *)(array[0] + 15), (void *)array[1]);

Teniendo en cuenta el tamaño de int, se obtiene una diferencia de 8128 bytes (8132-4), que son 2032 elementos de matriz del tamaño de int, y ese es el problema: una matriz multidimensional “real” no tiene espacios entre elementos.

Si necesita usar una matriz asignada dinámicamente con una función que espera una matriz multidimensional “real”, debe asignar un objeto de tipo int * y usar la aritmética para realizar los cálculos:

void func(int M, int N, int *array);
...

/* Equivalent to declaring `int array[M][N] = {{0}};` and assigning to array4_16[i][j]. */
int *array;
int M = 4, N = 16;
array = calloc(M, N * sizeof(*array));
array[i * N + j] = 1;
func(M, N, array);

Si N es una macro o un literal entero en lugar de una variable, el código puede simplemente usar la notación de matriz 2-D más natural después de asignar un puntero a una matriz:

void func(int M, int N, int *array);
#define N 16
void func_N(int M, int (*array)[N]);
...

int M = 4;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;

/* Cast to `int *` works here because `array` is a single block of M*N ints with no gaps,
   just like `int array2[M * N];` and `int array3[M][N];` would be. */
func(M, N, (int *)array);
func_N(M, array);

Si N no es una macro o un literal entero, array apuntará a una matriz de longitud variable (VLA). Esto todavía se puede usar con func convirtiéndolo en int * y una nueva función func_vla reemplazaría a func_N:

void func(int M, int N, int *array);
void func_vla(int M, int N, int array[M][N]);
...

int M = 4, N = 16;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;
func(M, N, (int *)array);
func_vla(M, N, array);

Nota: los VLA son opcionales a partir de C11. Si su implementación es compatible con C11 y define la macro __STDC_NO_VLA__ en 1, está atascado con los métodos anteriores a C99.

Superando los límites de la matriz #

Las matrices se basan en cero, es decir, el índice siempre comienza en 0 y termina con la longitud de la matriz de índice menos 1. Por lo tanto, el siguiente código no generará el primer elemento de la matriz y generará basura para el valor final que imprime.

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 1; x <= 5; x++) //Looping from 1 till 5.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Salida: 2 3 4 5 GarbageValue

A continuación se muestra la forma correcta de lograr el resultado deseado:

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 0; x < 5; x++) //Looping from 0 till 4.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Salida: 1 2 3 4 5

Es importante conocer la longitud de una matriz antes de trabajar con ella, ya que, de lo contrario, puede dañar el búfer o causar una falla de segmentación al acceder a ubicaciones de memoria que están fuera de los límites.
