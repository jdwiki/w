Para administrar la memoria asignada dinámicamente, la biblioteca C estándar proporciona las funciones malloc(), calloc(), realloc() y free(). En C99 y versiones posteriores, también existe aligned_alloc(). Algunos sistemas también proporcionan alloca().

Sintaxis #

  • void *aligned_alloc(size_t alineación, size_t tamaño); /* Solo desde C11 */
  • void *calloc(size_t nelements, size_t size);
  • vacío libre (vacío *ptr);
  • void *malloc(tamaño_t tamaño);
  • void *realloc(void *ptr, size_t size);
  • void *alloca(tamaño_t tamaño); /* de alloca.h, no estándar, no portátil, peligroso. */

Parámetros #

nombre descripción
tamaño (malloc, realloc y aligned_alloc) tamaño total de la memoria en bytes. Para aligned_alloc, el tamaño debe ser un múltiplo entero de la alineación.
tamaño (calloc) tamaño de cada elemento
elementos número de elementos
ptr puntero a la memoria asignada previamente devuelta por malloc, calloc, realloc o aligned_alloc
alineación alineación de la memoria asignada

Tenga en cuenta que aligned_alloc() solo está definido para C11 o posterior.

Los sistemas como los basados ​​en POSIX proporcionan otras formas de asignar memoria alineada (por ejemplo, posix_memalign()) , y también tiene otras opciones de administración de memoria (por ejemplo, mmap()).

Liberando la memoria #

Es posible liberar memoria asignada dinámicamente llamando a free().

int *p = malloc(10 * sizeof *p); /* allocation of memory */
if (p == NULL) 
{
    perror("malloc failed");
    return -1;
}

free(p); /* release of memory */
/* note that after free(p), even using the *value* of the pointer p
   has undefined behavior, until a new value is stored into it. */

/* reusing/re-purposing the pointer itself */
int i = 42;
p = &i; /* This is valid, has defined behaviour */

La memoria a la que apunta p se recupera (ya sea por la implementación de libc o por el sistema operativo subyacente) después de la llamada a free(), por lo que acceder a ese bloque de memoria liberado a través de p conducirá a comportamiento indefinido. Los punteros que hacen referencia a elementos de la memoria que se han liberado se denominan comúnmente punteros colgantes y presentan un riesgo de seguridad. Además, el estándar C establece que incluso [acceder al valor](https://www.wikiod.com/es/c/comportamiento-indefinido#Leer el valor del puntero que se liberó 201608050732042665317) de un puntero colgante tiene un comportamiento indefinido. Tenga en cuenta que el puntero p en sí mismo se puede reutilizar como se muestra arriba.

Tenga en cuenta que solo puede llamar a free() en punteros que hayan sido devueltos directamente desde las funciones malloc(), calloc(), realloc() y aligned_alloc(), o donde la documentación indique la memoria se ha asignado de esa manera (funciones como strdup () son ejemplos notables). Liberando un puntero que es,

  • obtenido mediante el uso del operador & en una variable, o
  • en medio de un bloque asignado,

está prohibido. Su compilador generalmente no diagnosticará un error de este tipo, pero conducirá a la ejecución del programa a un estado indefinido.

Hay dos estrategias comunes para prevenir tales instancias de comportamiento indefinido.

El primero y preferible es simple: hacer que p deje de existir cuando ya no se necesite, por ejemplo:

if (something_is_needed())
{

    int *p = malloc(10 * sizeof *p);
    if (p == NULL) 
    {
        perror("malloc failed");
        return -1;
    }

    /* do whatever is needed with p */

    free(p);
}

Al llamar a free() directamente antes del final del bloque contenedor (es decir, el }), p deja de existir. El compilador dará un error de compilación en cualquier intento de usar p después de eso.

Un segundo enfoque es invalidar también el propio puntero después de liberar la memoria a la que apunta:

free(p);
p = NULL;     // you may also use 0 instead of NULL

Argumentos a favor de este enfoque:

  • En muchas plataformas, un intento de desreferenciar un puntero nulo provocará un bloqueo instantáneo: falla de segmentación. Aquí, obtenemos al menos un seguimiento de la pila que apunta a la variable que se usó después de ser liberada.

Sin establecer el puntero en NULL, tenemos un puntero colgante. Es muy probable que el programa siga fallando, pero más tarde, porque la memoria a la que apunta el puntero se corromperá silenciosamente. Dichos errores son difíciles de rastrear porque pueden dar como resultado una pila de llamadas que no tiene ninguna relación con el problema inicial.

Por lo tanto, este enfoque sigue el concepto de fallo rápido.

  • Es seguro liberar un puntero nulo. El Estándar C especifica que free(NULL) no tiene efecto:

La función libre hace que el espacio al que apunta ptr se desasigne, es decir, que quede disponible para futuras asignaciones. Si ptr es un puntero nulo, no se produce ninguna acción. De lo contrario, si el argumento no coincide con un puntero devuelto anteriormente por la función calloc, malloc o realloc, o si el espacio ha sido desasignado por una llamada a free o realloc, el comportamiento es indefinido.

  • A veces, el primer enfoque no se puede utilizar (por ejemplo, la memoria se asigna en una función y se desasigna mucho más tarde en una función completamente diferente)

5: https://www.wikiod.com/es/c/punteros#Errores comunes

Asignación de memoria #

Asignación estándar #

Las funciones de asignación de memoria dinámica de C se definen en el encabezado <stdlib.h>. Si se desea asignar espacio de memoria para un objeto de forma dinámica, se puede utilizar el siguiente código:

int *p = malloc(10 * sizeof *p);
if (p == NULL) 
{
    perror("malloc() failed");
    return -1;
}

Esto calcula la cantidad de bytes que ocupan diez ints en la memoria, luego solicita esa cantidad de bytes de malloc y asigna el resultado (es decir, la dirección de inicio del fragmento de memoria que se acaba de crear usando malloc) a un puntero llamado p.

Es una buena práctica usar sizeof para calcular la cantidad de memoria a solicitar, ya que el resultado de sizeof está definido por la implementación (excepto para tipos de caracteres, que son char, signed char y unsigned char , para el cual sizeof está definido para dar siempre 1).

Debido a que malloc podría no ser capaz de atender la solicitud, podría devolver un puntero nulo. Es importante verificar esto para evitar intentos posteriores de desreferenciar el puntero nulo.

La memoria asignada dinámicamente usando malloc() se puede cambiar de tamaño usando realloc() o, cuando ya no se necesita, liberarla usando free().

Alternativamente, declarar int array[10]; asignaría la misma cantidad de memoria. Sin embargo, si se declara dentro de una función sin la palabra clave static, solo se podrá usar dentro de la función en la que se declara y las funciones a las que llama (porque la matriz se asignará en la pila y el espacio se liberará para reutilizar cuando la función regrese). Alternativamente, si se define con static dentro de una función, o si se define fuera de cualquier función, entonces su vida útil es la vida útil del programa. Los punteros también se pueden devolver desde una función, sin embargo, una función en C no puede devolver una matriz.

Memoria puesta a cero #

Es posible que la memoria devuelta por malloc no se inicialice a un valor razonable, y se debe tener cuidado de poner a cero la memoria con memset o de copiar inmediatamente un valor adecuado en ella. Alternativamente, calloc devuelve un bloque del tamaño deseado donde todos los bits se inicializan a 0. No es necesario que sea lo mismo que la representación de un punto flotante cero o una constante de puntero nulo.

int *p = calloc(10, sizeof *p);
if (p == NULL) 
{
    perror("calloc() failed");
    return -1;
}

Una nota sobre calloc: La mayoría de las implementaciones (de uso común) optimizarán calloc() para el rendimiento, por lo que será más rápido que llamar a malloc(), luego memset( ), aunque el efecto neto es idéntico.

Memoria alineada #

C11 introdujo una nueva función aligned_alloc() que asigna espacio con la alineación dada. Se puede usar si se necesita alinear la memoria que se asignará en ciertos límites que no pueden ser satisfechos por malloc() o calloc(). Las funciones malloc() y calloc() asignan memoria que está adecuadamente alineada para cualquier tipo de objeto (es decir, la alineación es alignof(max_align_t)). Pero con aligned_alloc() se pueden solicitar mayores alineaciones.

/* Allocates 1024 bytes with 256 bytes alignment. */
char *ptr = aligned_alloc(256, 1024);
if (ptr) {
    perror("aligned_alloc()");
    return -1;
}
free(ptr);

El estándar C11 impone dos restricciones: 1) el tamaño (segundo argumento) solicitado debe ser un múltiplo entero de la alineación (primer argumento) y 2) el valor de alineación debe ser una alineación válida respaldada por la implementación . El incumplimiento de cualquiera de ellos resulta en comportamiento indefinido.

Reasignación de memoria #

Es posible que deba expandir o reducir el espacio de almacenamiento de su puntero después de haberle asignado memoria. La función void *realloc(void *ptr, size_t size) desasigna el antiguo objeto señalado por ptr y devuelve un puntero a un objeto que tiene el tamaño especificado por size. ptr es el puntero a un bloque de memoria previamente asignado con malloc, calloc o realloc (o un puntero nulo) para ser reasignado. Se conserva el contenido máximo posible de la memoria original. Si el nuevo tamaño es mayor, cualquier memoria adicional más allá del tamaño anterior no se inicializa. Si el nuevo tamaño es más corto, se pierde el contenido de la parte encogida. Si ptr es NULL, se asigna un nuevo bloque y la función devuelve un puntero a él.

#include <stdio.h>
#include <stdlib.h>
 
int main(void)
{
    int *p = malloc(10 * sizeof *p);
    if (NULL == p) 
    {
        perror("malloc() failed");
        return EXIT_FAILURE;
    }
 
    p[0] = 42;
    p[9] = 15;

    /* Reallocate array to a larger size, storing the result into a
     * temporary pointer in case realloc() fails. */
    {
        int *temporary = realloc(p, 1000000 * sizeof *temporary);

        /* realloc() failed, the original allocation was not free'd yet. */
        if (NULL == temporary)
        {
            perror("realloc() failed");
            free(p); /* Clean up. */
            return EXIT_FAILURE;
        }      

        p = temporary;
    }

    /* From here on, array can be used with the new size it was 
     * realloc'ed to, until it is free'd. */

    /* The values of p[0] to p[9] are preserved, so this will print:
       42 15
    */
    printf("%d %d\n", p[0], p[9]);

    free(p);

    return EXIT_SUCCESS;
}

El objeto reasignado puede o no tener la misma dirección que *p. Por lo tanto, es importante capturar el valor de retorno de realloc que contiene la nueva dirección si la llamada es exitosa.

Asegúrese de asignar el valor de retorno de realloc a un temporal en lugar de la p original. realloc devolverá nulo en caso de falla, lo que sobrescribiría el puntero. Esto perdería sus datos y crearía una fuga de memoria.

realloc(ptr, 0) no es equivalente a free(ptr) #

realloc es conceptualmente equivalente a malloc + memcpy + free en el otro puntero.

Si el tamaño del espacio solicitado es cero, el comportamiento de realloc está definido por la implementación. Esto es similar para todas las funciones de asignación de memoria que reciben un parámetro de tamaño de valor 0. De hecho, tales funciones pueden devolver un puntero no nulo, pero eso nunca debe ser desreferenciado.

Por lo tanto, realloc(ptr,0) no es equivalente a free(ptr). Puede

  • Sea una implementación “perezosa” y simplemente devuelva ptr
  • free(ptr), asignar un elemento ficticio y devolverlo
  • free(ptr) y devuelve 0
  • simplemente devuelva 0 para fallar y no haga nada más.

Entonces, en particular, los dos últimos casos son indistinguibles por el código de la aplicación.

Esto significa que realloc(ptr,0) no puede realmente liberar/desasignar la memoria y, por lo tanto, nunca debe usarse como reemplazo de free.

Arrays multidimensionales de tamaño variable #

Desde C99, C tiene matrices de longitud variable, VLA, que modelan matrices con límites que solo se conocen en el momento de la inicialización. Si bien debe tener cuidado de no asignar VLA demasiado grandes (podrían destrozar su pila), usar punteros a VLA y usarlos en expresiones sizeof está bien.

double sumAll(size_t n, size_t m, double A[n][m]) {
    double ret = 0.0;
    for (size_t i = 0; i < n; ++i)
       for (size_t j = 0; j < m; ++j)
          ret += A[i][j]
    return ret;
}

int main(int argc, char *argv[argc+1]) {
   size_t n = argc*10;
   size_t m = argc*8;
   double (*matrix)[m] = malloc(sizeof(double[n][m]));
   // initialize matrix somehow
   double res = sumAll(n, m, matrix);
   printf("result is %g\n", res);
   free(matrix);
}

Aquí matriz es un puntero a elementos de tipo doble[m], y la expresión tamaño de con doble[n][m] asegura que contiene espacio para n tales elementos.

Todo este espacio se asigna de forma contigua y, por lo tanto, se puede desasignar mediante una única llamada a free.

La presencia de VLA en el lenguaje también afecta las posibles declaraciones de matrices y punteros en los encabezados de funciones. Ahora, se permite una expresión entera general dentro del [] de los parámetros de matriz. Para ambas funciones, las expresiones en [] usan parámetros que se han declarado antes en la lista de parámetros. Para sumAll estas son las longitudes que el código de usuario espera para la matriz. En cuanto a todos los parámetros de función de matriz en C, la dimensión más interna se reescribe en un tipo de puntero, por lo que es equivalente a la declaración

  double sumAll(size_t n, size_t m, double (*A)[m]);

Es decir, n no es realmente parte de la interfaz de la función, pero la información puede ser útil para la documentación y también podría ser utilizada por los compiladores de verificación de límites para advertir sobre el acceso fuera de los límites.

Del mismo modo, para main, la expresión argc+1 es la longitud mínima que prescribe el estándar C para el argumento argv.

Tenga en cuenta que oficialmente el soporte de VLA es opcional en C11, pero no conocemos ningún compilador que implemente C11 y que no los tenga. Puede probar con la macro __STDC_NO_VLA__ si es necesario.

alloca: asigna memoria en la pila #

Advertencia: alloca solo se menciona aquí por motivos de exhaustividad. Es completamente no portátil (no está cubierto por ninguno de los estándares comunes) y tiene una serie de características potencialmente peligrosas que lo hacen inseguro para los que no lo saben. El código C moderno debería reemplazarlo con Matrices de longitud variable (VLA).

Página del manual

#include <alloca.h>
// glibc version of stdlib.h include alloca.h by default

void foo(int size) {
    char *data = alloca(size);
    /*
      function body;
    */
    // data is automatically freed
}

Asigne memoria en el marco de pila de la persona que llama, el espacio al que hace referencia el puntero devuelto se [libre] 2 automáticamente cuando finaliza la función de la persona que llama.

Si bien esta función es conveniente para la administración automática de la memoria, tenga en cuenta que solicitar una asignación grande podría causar un desbordamiento de la pila y que no puede usar free con la memoria asignada con alloca (que podría causar más problemas con el desbordamiento de pila).

Por esta razón, no se recomienda usar alloca dentro de un bucle ni una función recursiva.

Y debido a que la memoria está libre al devolver la función, no puede devolver el puntero como resultado de la función (el comportamiento sería indefinido).

Resumen #

  • llamada idéntica a malloc
  • liberado automáticamente al regresar la función
  • incompatible con las funciones free,realloc (comportamiento indefinido)
  • el puntero no se puede devolver como resultado de una función (comportamiento indefinido)
  • el tamaño de asignación está limitado por el espacio de pila, que (en la mayoría de las máquinas) es mucho más pequeño que el espacio de almacenamiento dinámico disponible para que lo use malloc()
  • evite usar alloca() y VLA (matrices de longitud variable) en una sola función
  • alloca() no es tan portátil como malloc() y otros

Recomendación #

  • No use alloca() en código nuevo

alternativa moderna.

void foo(int size) {
    char data[size];
    /*
      function body;
    */
    // data is automatically freed
}

Esto funciona donde alloca() lo hace, y funciona en lugares donde alloca() no lo hace (dentro de bucles, por ejemplo). Asume una implementación C99 o una implementación C11 que no define __STDC_NO_VLA__.

Gestión de memoria definida por el usuario #

malloc() a menudo llama a las funciones subyacentes del sistema operativo para obtener páginas de memoria. Pero no hay nada especial en la función y se puede implementar en C directo declarando una gran matriz estática y asignando desde ella (hay una pequeña dificultad para garantizar la alineación correcta, en la práctica, la alineación a 8 bytes casi siempre es adecuada).

Para implementar un esquema simple, se almacena un bloque de control en la región de la memoria inmediatamente antes del puntero que se devolverá de la llamada. Esto significa que free() puede implementarse restando del puntero devuelto y leyendo la información de control, que suele ser el tamaño del bloque más alguna información que permite volver a ponerlo en la lista libre, una lista enlazada de no asignados. bloques

Cuando el usuario solicita una asignación, se busca en la lista libre hasta encontrar un bloque de igual o mayor tamaño al monto solicitado, luego si es necesario se fracciona. Esto puede conducir a la fragmentación de la memoria si el usuario realiza continuamente muchas asignaciones y liberaciones de tamaño impredecible ya intervalos impredecibles (no todos los programas reales se comportan así, el esquema simple suele ser adecuado para programas pequeños).

/* typical control block */
struct block
{
   size_t size;         /* size of block */
   struct block *next;  /* next block in free list */ 
   struct block *prev;  /* back pointer to previous block in memory */
   void *padding;       /* need 16 bytes to make multiple of 8 */
}

static struct block arena[10000]; /* allocate from here */
static struct block *firstfree;

Muchos programas requieren un gran número de asignaciones de pequeños objetos del mismo tamaño. Esto es muy fácil de implementar. Simplemente use un bloque con un puntero siguiente. Entonces, si se requiere un bloque de 32 bytes:

union block
{
   union block * next;
   unsigned char payload[32];
}  

static union block arena[100];
static union block * head; 
void init(void)
{
    int i;
    for (i = 0; i < 100 - 1; i++)
        arena[i].next = &arena[i + 1];
    arena[i].next = 0; /* last one, null */
    head = &block[0];
}
 
void *block_alloc()
{
    void *answer = head;
    if (answer)
        head = head->next;
    return answer;
}

void block_free(void *ptr)
{
    union block *block = ptr;
    block->next = head;
    head - block;
}

Este esquema es extremadamente rápido y eficiente, y puede hacerse genérico con cierta pérdida de claridad.
