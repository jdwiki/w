Visión general #

El estándar C describe la sintaxis del lenguaje, las funciones proporcionadas por la biblioteca estándar y el comportamiento de los procesadores C conformes (hablando en términos generales, compiladores) y los programas C conformes. Con respecto al comportamiento, el estándar en su mayor parte especifica comportamientos particulares para programas y procesadores. Por otro lado, algunas operaciones tienen un comportamiento indefinido explícito o implícito; tales operaciones siempre deben evitarse, ya que no puede confiar en nada sobre ellas. En el medio, hay una variedad de comportamientos definidos por la implementación. Estos comportamientos pueden variar entre procesadores C, tiempos de ejecución y bibliotecas estándar (colectivamente, implementaciones), pero son consistentes y confiables para cualquier implementación dada, y las implementaciones conformes documentan su comportamiento en cada una de estas áreas.

A veces es razonable que un programa se base en un comportamiento definido por la implementación. Por ejemplo, si el programa es de todos modos específico para un entorno operativo particular, es poco probable que confiar en comportamientos definidos por la implementación generales para los procesadores comunes para ese entorno sea un problema. Alternativamente, se pueden usar directivas de compilación condicional para seleccionar comportamientos definidos por la implementación apropiados para la implementación en uso. En cualquier caso, es esencial saber qué operaciones tienen un comportamiento definido por la implementación, para evitarlas o tomar una decisión informada sobre si usarlas y cómo.

El resto de estos comentarios constituye una lista de todos los comportamientos definidos por la implementación y las características especificadas en el estándar C2011, con referencias al estándar. Muchos de ellos usan [la terminología del estándar] (http://port70.net/~nsz/c/c11/n1570.html#3). Algunos otros se basan más generalmente en el contexto del estándar, como las ocho etapas de traducir el código fuente en un programa, o la diferencia entre implementaciones alojadas e independientes. Algunos que pueden ser particularmente sorprendentes o notables se presentan en negrita. No todos los comportamientos descritos son compatibles con los estándares C anteriores, pero, en términos generales, tienen un comportamiento definido por la implementación en todas las versiones del estándar que los admiten.

Representación de enteros con signo #

Cada tipo de entero con signo puede representarse en cualquiera de los tres formatos; está definido por la implementación cuál se usa. La implementación en uso para cualquier tipo de entero con signo dado al menos tan ancho como int se puede determinar en tiempo de ejecución a partir de los dos bits de orden más bajo de la representación del valor -1 en ese tipo, así:

enum { sign_magnitude = 1, ones_compl = 2, twos_compl = 3, };
#define SIGN_REP(T) ((T)-1 & (T)3)

switch (SIGN_REP(long)) {
   case sign_magnitude: { /* do something */ break; }
   case ones_compl:     { /* do otherwise */ break; }
   case twos_compl:     { /* do yet else  */ break; }
   case 0:  { _Static_assert(SIGN_REP(long), "bogus sign representation"); }
}

El mismo patrón se aplica a la representación de tipos más estrechos, pero no se pueden probar con esta técnica porque los operandos de & están sujetos a “las conversiones aritméticas habituales” antes de que se calcule el resultado.
