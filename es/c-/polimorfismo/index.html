Definir clases polimórficas #

El ejemplo típico es una clase de forma abstracta, que luego se puede derivar en cuadrados, círculos y otras formas concretas.

La clase principal:

Comencemos con la clase polimórfica:

class Shape {
public:
    virtual ~Shape() = default;
    virtual double get_surface() const = 0;
    virtual void describe_object() const { std::cout << "this is a shape" << std::endl; }  

    double get_doubled_surface() const { return 2 * get_surface(); } 
};

¿Cómo leer esta definición?

  • Puede definir el comportamiento polimórfico mediante funciones miembro introducidas con la palabra clave virtual. Aquí get_surface() y describe_object() obviamente se implementarán de manera diferente para un cuadrado que para un círculo. Cuando se invoca la función en un objeto, la función correspondiente a la clase real del objeto se determinará en tiempo de ejecución.

  • No tiene sentido definir get_surface() para una forma abstracta. Esta es la razón por la cual la función va seguida de = 0. Esto significa que la función es función virtual pura.

  • Una clase polimórfica siempre debe definir un destructor virtual.

  • Puede definir funciones miembro no virtuales. Cuando se invocará esta función para un objeto, la función se elegirá según la clase utilizada en tiempo de compilación. Aquí get_double_surface() se define de esta manera.

  • Una clase que contiene al menos una función virtual pura es una clase abstracta. Las clases abstractas no se pueden instanciar. Solo puede tener punteros o referencias de un tipo de clase abstracto.

Clases derivadas

Una vez que se define una clase base polimórfica, puede derivarla. Por ejemplo:

class Square : public Shape {
    Point top_left;
    double side_length;
public: 
    Square (const Point& top_left, double side)
       : top_left(top_left), side_length(side_length) {}

    double get_surface() override { return side_length * side_length; }   
    void describe_object() override { 
        std::cout << "this is a square starting at " << top_left.x << ", " << top_left.y
                  << " with a length of " << side_length << std::endl; 
    }  
};

Algunas explicaciones:

  • Puede definir o anular cualquiera de las funciones virtuales de la clase principal. El hecho de que una función fuera virtual en la clase padre la convierte en virtual en la clase derivada. No es necesario decirle al compilador la palabra clave virtual nuevamente. Pero se recomienda agregar la palabra clave override al final de la declaración de la función, para evitar errores sutiles causados ​​por variaciones inadvertidas en la firma de la función.
  • Si todas las funciones virtuales puras de la clase principal están definidas, puede instanciar objetos para esta clase; de ​​lo contrario, también se convertirá en una clase abstracta.
  • No está obligado a anular todas las funciones virtuales. Puede mantener la versión del padre si se adapta a sus necesidades.

Ejemplo de instanciación

int main() {

    Square square(Point(10.0, 0.0), 6); // we know it's a square, the compiler also
    square.describe_object(); 
    std::cout << "Surface: " << square.get_surface() << std::endl; 

    Circle circle(Point(0.0, 0.0), 5);

    Shape *ps = nullptr;  // we don't know yet the real type of the object
    ps = &circle;         // it's a circle, but it could as well be a square
    ps->describe_object(); 
    std::cout << "Surface: " << ps->get_surface() << std::endl;
}

Reducción segura #

Suponga que tiene un puntero a un objeto de una clase polimórfica:

Shape *ps;                       // see example on defining a polymorphic class
ps =  get_a_new_random_shape();  // if you don't have such a function yet, you 
                                 // could just write ps = new Square(0.0,0.0, 5);

un downcast sería lanzar desde una ‘Forma’ polimórfica general a una de sus formas derivadas y más específicas como ‘Cuadrado’ o ‘Círculo’.

¿Por qué abatir?

La mayoría de las veces, no necesitará saber cuál es el tipo real del objeto, ya que las funciones virtuales le permiten manipular su objeto independientemente de su tipo:

std::cout << "Surface: " << ps->get_surface() << std::endl;

Si no necesita ningún abatido, su diseño sería perfecto.

Sin embargo, es posible que a veces necesite abatir. Un ejemplo típico es cuando desea invocar una función no virtual que existe solo para la clase secundaria.

Considere, por ejemplo, círculos. Sólo los círculos tienen un diámetro. Entonces la clase se definiría como:

class Circle: public Shape { // for Shape, see example on defining a polymorphic class
    Point center;
    double radius;
public: 
    Circle (const Point& center, double radius)
       : center(center), radius(radius) {}

    double get_surface() const override { return r * r * M_PI; }   

    // this is only for circles. Makes no sense for other shapes 
    double get_diameter() const { return 2 * r; }
};

La función miembro get_diameter() solo existe para círculos. No se definió para un objeto Shape:

Shape* ps = get_any_shape();
ps->get_diameter(); // OUCH !!! Compilation error

¿Cómo abatir?

Si estuviera seguro de que ps apunta a un círculo, podría optar por static_cast:

std::cout << "Diameter: " << static_cast<Circle*>(ps)->get_diameter() << std::endl;

Esto hará el truco. Pero es muy arriesgado: si ps aparece como algo más que un Círculo, el comportamiento de su código no estará definido.

Entonces, en lugar de jugar a la ruleta rusa, deberías usar con seguridad un dynamic_cast. Esto es específicamente para clases polimórficas:

int main() {
    Circle circle(Point(0.0, 0.0), 10);
    Shape &shape = circle;

    std::cout << "The shape has a surface of " << shape.get_surface() << std::endl;

    //shape.get_diameter();   // OUCH !!! Compilation error 

    Circle *pc = dynamic_cast<Circle*>(&shape); // will be nullptr if ps wasn't a circle 
    if (pc) 
        std::cout << "The shape is a circle of diameter " << pc->get_diameter() << std::endl;
    else
        std::cout << "The shape isn't a circle !" << std::endl; 
}

Tenga en cuenta que dynamic_cast no es posible en una clase que no es polimórfica. Necesitaría al menos una función virtual en la clase o sus padres para poder usarla.

Polimorfismo y destructores #

Si se pretende que una clase se use polimórficamente, con instancias derivadas almacenadas como punteros/referencias base, el destructor de su clase base debe ser “virtual” o “protegido”. En el primer caso, esto hará que la destrucción del objeto verifique vtable, llamando automáticamente al destructor correcto según el tipo dinámico. En el último caso, la destrucción del objeto a través de un puntero/referencia de clase base está deshabilitada, y el objeto solo se puede eliminar cuando se trata explícitamente como su tipo real.

struct VirtualDestructor {
    virtual ~VirtualDestructor() = default;
};

struct VirtualDerived : VirtualDestructor {};

struct ProtectedDestructor {
  protected:
    ~ProtectedDestructor() = default;
};

struct ProtectedDerived : ProtectedDestructor {
    ~ProtectedDerived() = default;
};

// ...

VirtualDestructor* vd = new VirtualDerived;
delete vd; // Looks up VirtualDestructor::~VirtualDestructor() in vtable, sees it's
           // VirtualDerived::~VirtualDerived(), calls that.

ProtectedDestructor* pd = new ProtectedDerived;
delete pd; // Error: ProtectedDestructor::~ProtectedDestructor() is protected.
delete static_cast<ProtectedDerived*>(pd); // Good.

Ambas prácticas garantizan que el destructor de la clase derivada siempre se llamará en las instancias de la clase derivada, evitando pérdidas de memoria.
