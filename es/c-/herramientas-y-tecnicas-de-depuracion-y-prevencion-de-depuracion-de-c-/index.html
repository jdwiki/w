Los desarrolladores de C++ dedican mucho tiempo a la depuración. Este tema está destinado a ayudar con esta tarea y dar inspiración para las técnicas. No espere una lista extensa de problemas y soluciones solucionados por las herramientas o un manual sobre las herramientas mencionadas.

Este tema aún no está completo, serían útiles ejemplos de las siguientes técnicas/herramientas:

  • Mencionar más herramientas de análisis estático
  • Herramientas de instrumentación binaria (como UBSan, TSan, MSan, ESan…)
  • Endurecimiento (CFI…)
  • Fuzzing

Análisis de fallas de segmento con GDB #

Análisis estático #

El análisis estático es la técnica en la que se comprueba el código en busca de patrones vinculados a errores conocidos. El uso de esta técnica requiere menos tiempo que una revisión de código, aunque sus comprobaciones solo se limitan a las programadas en la herramienta.

Las comprobaciones pueden incluir el punto y coma incorrecto detrás de la declaración if (if (var);) hasta algoritmos gráficos avanzados que determinan si una variable no se inicializa.

Advertencias del compilador #

Habilitar el análisis estático es fácil, la versión más simple ya está integrada en su compilador:

  • clang++ -Muro -Todo -Werror ...
  • g++ -Muro -Todo -Werror ...
  • cl.exe /W4 /WX ...

Si habilita estas opciones, notará que cada compilador encontrará errores que los demás no y que obtendrá errores en técnicas que podrían ser válidas o válidas en un contexto específico. while (staticAtomicBool); podría ser aceptable incluso si while (localBool); no lo es.

Entonces, a diferencia de la revisión de código, está luchando contra una herramienta que comprende su código, le dice muchos errores útiles y, a veces, no está de acuerdo con usted. En este último caso, es posible que deba suprimir la advertencia localmente.

Como las opciones anteriores habilitan todas las advertencias, pueden habilitar las advertencias que no desea. (¿Por qué su código debería ser compatible con C++98?) Si es así, simplemente puede deshabilitar esa advertencia específica:

  • clang++ -Wall -Weeverything -Werror -Wno-errortoaccept ...
  • g++ -Wall -Weeverything -Werror -Wno-errortoaccept ...
  • cl.exe /W4 /WX /wd<nº de advertencia>...

Cuando las advertencias del compilador lo ayudan durante el desarrollo, ralentizan bastante la compilación. Es por eso que es posible que no siempre desee habilitarlos de forma predeterminada. O los ejecuta de forma predeterminada o habilita alguna integración continua con los controles más caros (o todos ellos).

herramientas externas #

Si decide tener alguna integración continua, el uso de otras herramientas no es tan exagerado. Una herramienta como clang-tidy tiene una [lista de comprobaciones](http://clang.llvm.org/extra/clang-tidy/ checks/list.html) que cubre una amplia gama de temas, algunos ejemplos:

  • Errores reales
    • Prevention of slicing
    • Asserts with side effects
  • Comprobaciones de legibilidad
    • Misleading indentation
    • Check identifier naming
  • Controles de modernización
    • Use make_unique()
    • Use nullptr
  • Comprobaciones de rendimiento
    • Find unneeded copies
    • Find inefficient algorithm calls

Es posible que la lista no sea tan grande, ya que Clang ya tiene muchas advertencias del compilador; sin embargo, lo acercará un paso más a una base de código de alta calidad.

Otras herramientas #

Existen otras herramientas con propósitos similares, como:

  • [el analizador estático de Visual Studio](https://blogs.msdn.microsoft.com/hkamel/2013/10/24/visual-studio-2013-static-code-analysis-in-depth-what-when-and -how/) como herramienta externa
  • clazy, un complemento del compilador Clang para verificar el código Qt

Conclusión #

Existen muchas herramientas de análisis estático para C++, tanto integradas en el compilador como herramientas externas. Probarlos no toma mucho tiempo para configuraciones sencillas y encontrarán errores que podría pasar por alto en la revisión del código.

Código limpio #

La depuración comienza con la comprensión del código que está tratando de depurar.

Código incorrecto:

int main() {
    int value;
    std::vector<int> vectorToSort;
    vectorToSort.push_back(42); vectorToSort.push_back(13);
    for (int i = 52; i; i = i - 1)
    {
    vectorToSort.push_back(i *2);
    }
    /// Optimized for sorting small vectors
    if (vectorToSort.size() == 1);
    else
        {
        if (vectorToSort.size() <= 2)
            std::sort(vectorToSort.begin(), std::end(vectorToSort));
        }
    for (value : vectorToSort) std::cout << value << ' ';
return 0; }

Mejor código:

std::vector<int> createSemiRandomData() {
    std::vector<int> data;
    data.push_back(42);
    data.push_back(13);
    for (int i = 52; i; --i)
        vectorToSort.push_back(i *2);
    return data;
}

/// Optimized for sorting small vectors
void sortVector(std::vector &v) {
    if (vectorToSort.size() == 1)
        return;
    if (vectorToSort.size() > 2)
        return;

    std::sort(vectorToSort.begin(), vectorToSort.end());
}

void printVector(const std::vector<int> &v) {
    for (auto i : v)
        std::cout << i << ' ';
}

int main() {
    auto vectorToSort = createSemiRandomData();
    sortVector(std::ref(vectorToSort));
    printVector(vectorToSort);

    return 0;
 }

Independientemente de los estilos de codificación que prefiera y use, tener un estilo de codificación (y formato) consistente lo ayudará a comprender el código.

Mirando el código anterior, uno puede identificar un par de mejoras para mejorar la legibilidad y la depuración:

El uso de funciones separadas para acciones separadas #

El uso de funciones separadas le permite omitir algunas funciones en el depurador si no está interesado en los detalles. En este caso específico, es posible que no esté interesado en la creación o impresión de los datos y solo quiera pasar a la clasificación.

Otra ventaja es que necesita leer menos código (y memorizarlo) mientras recorre el código. Ahora solo necesita leer 3 líneas de código en main() para entenderlo, en lugar de la función completa.

La tercera ventaja es que simplemente tiene menos código para mirar, lo que ayuda a un ojo entrenado a detectar este error en segundos.

Uso de formatos/construcciones coherentes #

El uso de formatos y construcciones coherentes eliminará el desorden del código, lo que facilitará centrarse en el código en lugar del texto. Se han alimentado muchas discusiones sobre el estilo de formato “correcto”. Independientemente de ese estilo, tener un único estilo consistente en el código mejorará la familiaridad y hará que sea más fácil concentrarse en el código.

Dado que formatear el código es una tarea que requiere mucho tiempo, se recomienda utilizar una herramienta dedicada para ello. La mayoría de los IDE tienen al menos algún tipo de soporte para esto y pueden formatear de manera más consistente que los humanos.

Puede notar que el estilo no se limita a espacios y líneas nuevas, ya que ya no mezclamos el estilo libre y las funciones miembro para obtener el inicio/final del contenedor. (v.begin() frente a std::end(v)).

Dirija la atención a las partes importantes de su código. #

Independientemente del estilo que decida elegir, el código anterior contiene un par de marcadores que pueden darle una pista sobre lo que podría ser importante:

  • Un comentario que dice “optimizado”, esto indica algunas técnicas sofisticadas
  • Algunos retornos tempranos en sortVector() indican que estamos haciendo algo especial
  • std::ref() indica que algo está pasando con sortVector()

Conclusión #

Tener un código limpio lo ayudará a comprender el código y reducirá el tiempo que necesita para depurarlo. En el segundo ejemplo, un revisor de código podría incluso detectar el error a primera vista, mientras que el error podría estar oculto en los detalles del primero. Safe-stack (corrupciones de pila) #

Las corrupciones de pila son errores molestos a la vista. Como la pila está dañada, el depurador a menudo no puede darte un buen seguimiento de la pila de dónde estás y cómo llegaste allí.

Aquí es donde entra en juego la pila segura. En lugar de usar una sola pila para sus subprocesos, usará dos: una pila segura y una pila peligrosa. La pila segura funciona exactamente igual que antes, excepto que algunas partes se mueven a la pila peligrosa.

¿Qué partes de la pila se mueven? #

Cada parte que tenga el potencial de corromper la pila se sacará de la pila segura. Tan pronto como una variable en la pila se pasa por referencia o se toma la dirección de esta variable, el compilador decidirá asignarla en la segunda pila en lugar de en la segura.

Como resultado, cualquier operación que realice con esos punteros, cualquier modificación que realice en la memoria (basada en esos punteros/referencias) solo puede afectar la memoria en la segunda pila. Como uno nunca obtiene un puntero que está cerca de la pila segura, la pila no puede corromper la pila y el depurador aún puede leer todas las funciones en la pila para dar un buen seguimiento.

¿Para qué se usa realmente? #

La pila segura no se inventó para brindarle una mejor experiencia de depuración; sin embargo, es un buen efecto secundario para los errores desagradables. Su propósito original es como parte del [Proyecto de integridad de puntero de código (CPI)] (http://dslab.epfl.ch/proj/cpi/), en el que intentan evitar la anulación de las direcciones de retorno para evitar la inyección de código. En otras palabras, intentan evitar la ejecución de un código de piratas informáticos.

Por este motivo, la función se ha activado en Chrome y se ha informado para tener una <1 % de sobrecarga de la CPU.

¿Cómo habilitarlo? #

En este momento, la opción solo está disponible en el compilador clang, donde se puede pasar -fsanitize=safe-stack al compilador. Se hizo una propuesta para implementar la misma función en GCC.

Conclusión #

Las corrupciones de pila pueden volverse más fáciles de depurar cuando la pila segura está habilitada. Debido a una sobrecarga de bajo rendimiento, incluso puede activarse de forma predeterminada en su configuración de compilación.
