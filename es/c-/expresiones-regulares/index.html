Expresiones regulares (a veces llamadas expresiones regulares o expresiones regulares) son una sintaxis textual que representa los patrones que pueden coincidir en las cadenas operadas.

Las expresiones regulares, introducidas en c++11, pueden admitir opcionalmente una matriz de retorno de cadenas coincidentes u otra sintaxis textual que defina cómo reemplazar patrones coincidentes en hilos operados.

Sintaxis #

  • regex_match // Devuelve si la expresión regular coincidió con toda la secuencia de caracteres, capturando opcionalmente en un objeto de coincidencia
  • regex_search // Devuelve si una parte de la secuencia de caracteres coincidió con la expresión regular, capturando opcionalmente en un objeto de coincidencia
  • regex_replace // Devuelve la secuencia de caracteres de entrada modificada por una expresión regular a través de una cadena de formato de reemplazo
  • regex_token_iterator // Inicializado con una secuencia de caracteres definida por iteradores, una lista de índices de captura para iterar y una expresión regular. La desreferenciación devuelve la coincidencia indexada actualmente de la expresión regular. El incremento se mueve al siguiente índice de captura o, si actualmente se encuentra en el último índice, restablece el índice y retrasa la próxima aparición de una coincidencia de expresiones regulares en la secuencia de caracteres
  • regex_iterator // Inicializado con una secuencia de caracteres definida por iteradores y una expresión regular. La desreferenciación devuelve la parte de la secuencia de caracteres que coincide actualmente con toda la expresión regular. El incremento encuentra la siguiente aparición de una coincidencia de expresiones regulares en la secuencia de caracteres

Parámetros #

Firma Descripción
bool regex_match(Iterador bidireccional primero, Iterador bidireccional último, smatch& sm, const regex& re, regex_constraints::match_flag_type flags) Iterador bidireccional es cualquier iterador de caracteres que proporcione operadores de incremento y decremento smatch puede ser cmatch o cualquier otra variante de match_results que acepte el tipo de Iterador bidireccional el smatch el argumento puede omitirse si los resultados de la expresión regular no son necesarios Devuelve si re coincide con la secuencia de caracteres completa definida por first y last
bool regex_match(const string& str, smatch& sm, const regex re&, regex_constraints::match_flag_type flags) string puede ser un const char* o una string de valor L, las funciones que aceptan una string de valor R se eliminan explícitamente smatch puede ser cmatch o cualquier otra variante de match_results que acepte el tipo de str el argumento smatch puede omitirse si los resultados de la expresión regular no son necesarios Devuelve si re coincide con la secuencia de caracteres completa definido por str

Ejemplos básicos de regex_match y regex_search #

const auto input = "Some people, when confronted with a problem, think \"I know, I'll use regular expressions.\""s;
smatch sm;

cout << input << endl;

// If input ends in a quotation that contains a word that begins with "reg" and another word begining with "ex" then capture the preceeding portion of input
if (regex_match(input, sm, regex("(.*)\".*\\breg.*\\bex.*\"\\s*$"))) {
    const auto capture = sm[1].str();
    
    cout << '\t' << capture << endl; // Outputs: "\tSome people, when confronted with a problem, think\n"
    
    // Search our capture for "a problem" or "# problems"
    if(regex_search(capture, sm, regex("(a|d+)\\s+problems?"))) {
        const auto count = sm[1] == "a"s ? 1 : stoi(sm[1]);
        
        cout << '\t' << count << (count > 1 ? " problems\n" : " problem\n"); // Outputs: "\t1 problem\n"
        cout << "Now they have " << count + 1 << " problems.\n"; // Ouputs: "Now they have 2 problems\n"
    }
}

regex_iterator Ejemplo #

Cuando el procesamiento de las capturas se debe realizar de forma iterativa, regex_iterator es una buena opción. Eliminar la referencia de un regex_iterator devuelve un match_result. Esto es excelente para capturas condicionales o capturas que tienen interdependencia. Digamos que queremos tokenizar algún código C++. Dado:

enum TOKENS {
    NUMBER,
    ADDITION,
    SUBTRACTION,
    MULTIPLICATION,
    DIVISION,
    EQUALITY,
    OPEN_PARENTHESIS,
    CLOSE_PARENTHESIS
};

Podemos tokenizar esta cadena: const auto input = "42/2 + -8\t=\n(2 + 2) * 2 * 2 -3"s con un regex_iterator como este:

vector<TOKENS> tokens;
const regex re{ "\\s*(\\(?)\\s*(-?\\s*\\d+)\\s*(\\)?)\\s*(?:(\\+)|(-)|(\\*)|(/)|(=))" };

for_each(sregex_iterator(cbegin(input), cend(input), re), sregex_iterator(), [&](const auto& i) {
    if(i[1].length() > 0) {
        tokens.push_back(OPEN_PARENTHESIS);
    }
    
    tokens.push_back(i[2].str().front() == '-' ? NEGATIVE_NUMBER : NON_NEGATIVE_NUMBER);
    
    if(i[3].length() > 0) {
        tokens.push_back(CLOSE_PARENTHESIS);
    }        
    
    auto it = next(cbegin(i), 4);
    
    for(int result = ADDITION; it != cend(i); ++result, ++it) {
        if (it->length() > 0U) {
            tokens.push_back(static_cast<TOKENS>(result));
            break;
        }
    }
});

match_results<string::const_reverse_iterator> sm;

if(regex_search(crbegin(input), crend(input), sm, regex{ tokens.back() == SUBTRACTION ? "^\\s*\\d+\\s*-\\s*(-?)" : "^\\s*\\d+\\s*(-?)" })) {
    tokens.push_back(sm[1].length() == 0 ? NON_NEGATIVE_NUMBER : NEGATIVE_NUMBER);
}

[Ejemplo en vivo] (http://ideone.com/Rv5WNI)

Un problema notable con los iteradores de expresiones regulares es que el argumento regex debe ser un valor L, un valor R no funcionará: http://stackoverflow.com/q/29895747/2642059

Anclas #

C++ proporciona solo 4 anclas:

  • ^ que afirma el comienzo de la cadena
  • $ que afirma el final de la cadena
  • \b que afirma un carácter \W o el principio o el final de la cadena
  • \B que afirma un carácter \w

Digamos por ejemplo que queremos capturar un número con su signo:

auto input = "+1--12*123/+1234"s;
smatch sm;

if(regex_search(input, sm, regex{ "(?:^|\\b\\W)([+-]?\\d+)" })) {

    do {
        cout << sm[1] << endl;
        input = sm.suffix().str();
    } while(regex_search(input, sm, regex{ "(?:^\\W|\\b\\W)([+-]?\\d+)" }));
}

[Ejemplo en vivo] (http://ideone.com/uE4dGr)

Una nota importante aquí es que el ancla no consume ningún carácter.

regex_replace Ejemplo #

Este código toma varios estilos de llaves y los convierte en [One True Brace Style] (https://en.wikipedia.org/wiki/Indent_style#Variant:_1TBS):

const auto input = "if (KnR)\n\tfoo();\nif (spaces) {\n    foo();\n}\nif (allman)\n{\n\tfoo();\n}\nif (horstmann)\n{\tfoo();\n}\nif (pico)\n{\tfoo(); }\nif (whitesmiths)\n\t{\n\tfoo();\n\t}\n"s;

cout << input << regex_replace(input, regex("(.+?)\\s*\\{?\\s*(.+?;)\\s*\\}?\\s*"), "$1 {\n\t$2\n}\n") << endl;

[Ejemplo en vivo] (http://ideone.com/ICR5wM)

Ejemplo de regex_token_iterator #

Un std::regex_token_iterator proporciona una tremenda herramienta para extraer elementos de un archivo de valores separados por comas. Aparte de las ventajas de la iteración, este iterador también puede capturar comas escapadas donde otros métodos tienen dificultades:

const auto input = "please split,this,csv, ,line,\\,\n"s;
const regex re{ "((?:[^\\\\,]|\\\\.)+)(?:,|$)" };
const vector<string> m_vecFields{ sregex_token_iterator(cbegin(input), cend(input), re, 1), sregex_token_iterator() };

cout << input << endl;

copy(cbegin(m_vecFields), cend(m_vecFields), ostream_iterator<string>(cout, "\n"));

[Ejemplo en vivo] (http://ideone.com/lySlTJ)

Un problema notable con los iteradores de expresiones regulares es que el argumento regex debe ser un valor L. Un valor R no funcionará.

Dividir una cadena #

std::vector<std::string> split(const std::string &str, std::string regex)
{
    std::regex r{ regex };
    std::sregex_token_iterator start{ str.begin(), str.end(), r, -1 }, end;
    return std::vector<std::string>(start, end);
}

split("Some  string\t with whitespace ", "\\s+"); // "Some", "string", "with", "whitespace"

Cuantificadores #

Digamos que recibimos const string input como un número de teléfono para ser validado. Podríamos comenzar requiriendo una entrada numérica con un cero o más cuantificadores: regex_match(input, regex("\\d*")) o uno o más cuantificadores: regex_match(input , regex("\\d+")) Pero ambos realmente se quedan cortos si input contiene una cadena numérica inválida como: “123” Usemos un n o más cuantificadores para asegurarnos de que estamos obteniendo al menos 7 dígitos:

regex_match(input, regex("\\d{7,}"))

Esto garantizará que obtendremos al menos un número de teléfono de dígitos, pero input también podría contener una cadena numérica demasiado larga como: “123456789012”. Así que vamos con un cuantificador entre n y m para que la entrada tenga al menos 7 dígitos pero no más de 11:

regex_match(input, regex("\\d{7,11}"));

Esto nos acerca más, pero las cadenas numéricas ilegales que están en el rango de [7, 11] todavía se aceptan, como: “123456789”. Así que hagamos que el código de país sea opcional con un cuantificador perezoso:

regex_match(input, regex("\\d?\\d{7,10}"))

Es importante tener en cuenta que el cuantificador perezoso coincide con la menor cantidad de caracteres posible, por lo que la única forma en que este carácter coincidirá es si ya hay 10 caracteres que coincidieron con \d{7,10} . (Para hacer coincidir el primer carácter con avidez, habríamos tenido que hacer: \d{0,1}). El cuantificador perezoso se puede agregar a cualquier otro cuantificador.

Ahora, ¿cómo haríamos que el código de área fuera opcional y solo aceptaríamos un código de país si el código de área estuviera presente?

regex_match(input, regex("(?:\\d{3,4})?\\d{7}"))

En esta expresión regular final, \d{7} requiere 7 dígitos. Estos 7 dígitos están precedidos opcionalmente por 3 o 4 dígitos.

Tenga en cuenta que no añadimos el cuantificador perezoso: \d{3,4}?\d{7}, el \d{3,4}? tendría coincidió con 3 o 4 caracteres, prefiriendo 3. En cambio, estamos haciendo que el grupo que no captura coincida como máximo una vez, prefiriendo no coincidir. Causando una discrepancia si input no incluía el código de área como: “1234567”.

Para concluir con el tema del cuantificador, me gustaría mencionar el otro cuantificador adjunto que puede usar, el cuantificador posesivo. O el cuantificador perezoso o el cuantificador posesivo se pueden agregar a cualquier cuantificador. La única función del cuantificador posesivo es ayudar al motor de expresiones regulares diciéndole, tome con avidez estos caracteres y nunca los abandone, incluso si hace que la expresión regular falle. Esto, por ejemplo, no tiene mucho sentido: regex_match(input, regex("\\d{3,4}+\\d{7})) Porque una entrada como: “1234567890” no sería emparejado como \d{3,4}+ siempre coincidirá con 4 caracteres, incluso si la coincidencia de 3 hubiera permitido que la expresión regular tuviera éxito.
El cuantificador posesivo se usa mejor cuando el token cuantificado limita el número de caracteres coincidentes. Por ejemplo:

regex_match(input, regex("(?:.*\\d{3,4}+){3}"))

Se puede usar para buscar coincidencias si input contiene alguno de los siguientes:

123 456 7890
123-456-7890
(123)456-7890
(123) 456 - 7890

Pero cuando esta expresión regular realmente brilla es cuando input contiene una entrada ilegal:

12345 - 67890

Sin el cuantificador posesivo, el motor de expresiones regulares tiene que volver atrás y probar todas las combinaciones de .* y 3 o 4 caracteres para ver si puede encontrar una combinación compatible. Con el cuantificador posesivo, la expresión regular comienza donde quedó el 2nd cuantificador posesivo, el carácter ‘0’, y el motor de expresiones regulares intenta ajustar el .* a permitir que \d{3,4} coincida; cuando no puede, la expresión regular simplemente falla, no se realiza un seguimiento para ver si el ajuste .* anterior podría haber permitido una coincidencia.
