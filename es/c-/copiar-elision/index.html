Propósito de la elisión de copia #

Hay lugares en el estándar donde un objeto se copia o se mueve para inicializar un objeto. La elisión de copia (a veces llamada optimización del valor de retorno) es una optimización por la cual, bajo ciertas circunstancias específicas, se permite que un compilador evite la copia o el movimiento aunque el estándar diga que debe suceder.

Considere la siguiente función:

std::string get_string()
{
  return std::string("I am a string.");
}

De acuerdo con la redacción estricta del estándar, esta función inicializará un ‘std::string’ temporal, luego lo copiará/moverá al objeto de valor de retorno, luego destruirá el temporal. El estándar es muy claro en que así es como se interpreta el código.

La elisión de copia es una regla que permite que un compilador de C++ ignore la creación del temporal y su subsiguiente copia/destrucción. Es decir, el compilador puede tomar la expresión de inicialización para el temporal e inicializar directamente el valor de retorno de la función. Esto obviamente ahorra rendimiento.

Sin embargo, tiene dos efectos visibles en el usuario:

  1. El tipo debe tener el constructor de copiar/mover que habría sido llamado. Incluso si el compilador eliminó la copia/mover, el tipo aún debe poder haber sido copiado/mover.

  2. Los efectos secundarios de los constructores de copiar/mover no están garantizados en circunstancias en las que puede ocurrir la elisión. Considera lo siguiente:

struct my_type
{
  my_type() = default;
  my_type(const my_type &) {std::cout <<"Copying\n";}
  my_type(my_type &&) {std::cout <<"Moving\n";}
};

my_type func()
{
  return my_type();
}

¿Qué hará llamar a func? Bueno, nunca imprimirá “Copiando”, ya que el temporal es un valor r y mi_tipo es un tipo móvil. Entonces, ¿imprimirá “Moving”?

Sin la regla de elisión de copia, esto sería necesario para imprimir siempre “En movimiento”. Pero debido a que existe la regla de elisión de copia, el constructor de movimiento puede o no ser llamado; depende de la implementación.

Y, por lo tanto, no puede depender de la llamada de constructores de copiar/mover en contextos donde es posible la elisión.

Debido a que la elisión es una optimización, es posible que su compilador no admita la elisión en todos los casos. E independientemente de si el compilador elimina un caso en particular o no, el tipo aún debe admitir la operación que se elimina. Entonces, si se elide una construcción de copia, el tipo aún debe tener un constructor de copia, aunque no se llamará.

Elisión de copia garantizada #

Normalmente, la elisión es una optimización. Si bien prácticamente todos los compiladores admiten la elisión de copia en los casos más simples, tener la elisión aún representa una carga particular para los usuarios. Es decir, el tipo cuya copia/mover se está elidiendo debe tener todavía la operación de copiar/mover que se elidió.

Por ejemplo:

std::mutex a_mutex;
std::lock_guard<std::mutex> get_lock()
{
  return std::lock_guard<std::mutex>(a_mutex);
}

Esto podría ser útil en los casos en que a_mutex es un mutex que está en manos privadas de algún sistema, pero un usuario externo puede querer tener un bloqueo de alcance para él.

Esto tampoco es legal, porque std::lock_guard no se puede copiar ni mover. A pesar de que prácticamente todos los compiladores de C++ eliminarán la función de copiar/mover, el estándar todavía requiere que el tipo tenga esa operación disponible.

Hasta C++17.

C ++ 17 exige la elisión al redefinir efectivamente el significado mismo de ciertas expresiones para que no se produzca ninguna copia o movimiento. Considere el código anterior.

Bajo la redacción anterior a C++ 17, ese código dice que se cree un temporal y luego se use el temporal para copiar/mover al valor de retorno, pero la copia temporal se puede elidir. Según la redacción de C ++ 17, eso no crea un temporal en absoluto.

En C++17, cualquier prvalue expression, cuando se usa para inicializar un objeto del mismo tipo que la expresión, no genera un objeto temporal. La expresión inicializa directamente ese objeto. Si devuelve un prvalue del mismo tipo que el valor devuelto, entonces el tipo no necesita tener un constructor de copiar/mover. Y por lo tanto, bajo las reglas de C++17, el código anterior puede funcionar.

La redacción de C++17 funciona en los casos en que el tipo de prvalue coincide con el tipo que se está inicializando. Entonces, dado get_lock arriba, esto tampoco requerirá una copia/mover:

std::lock_guard the_lock = get_lock();

Dado que el resultado de get_lock es una expresión prvalue que se utiliza para inicializar un objeto del mismo tipo, no se copiará ni se moverá. Esa expresión nunca crea un temporal; se usa para inicializar directamente the_lock. No hay elisión porque no hay copia/movimiento para ser elidido elide.

El término “eliminación de copia garantizada” es, por lo tanto, algo inapropiado, pero ese es el nombre de la función tal como se propone para la estandarización de C++17. No garantiza la elisión en absoluto; elimina la copia/mover por completo, redefiniendo C++ para que nunca haya una copia/mover para eliminar.

Esta característica solo funciona en casos que involucran una expresión prvalue. Como tal, esto utiliza las reglas de elisión habituales:

std::mutex a_mutex;
std::lock_guard<std::mutex> get_lock()
{
  std::lock_guard<std::mutex> my_lock(a_mutex);
  //Do stuff
  return my_lock;
}

Si bien este es un caso válido para la elisión de copia, las reglas de C++ 17 no eliminan la copia/mover en este caso. Como tal, el tipo aún debe tener un constructor de copiar/mover para usar para inicializar el valor devuelto. Y dado que lock_guard no lo hace, sigue siendo un error de compilación. Se permite que las implementaciones se nieguen a eludir copias al pasar o devolver un objeto de tipo copiable trivialmente. Esto es para permitir mover dichos objetos en los registros, lo que algunas ABI pueden exigir en sus convenciones de llamada.

struct trivially_copyable {
    int a;  
};

void foo (trivially_copyable a) {}

foo(trivially_copyable{}); //copy elision not mandated

Elisión de parámetros #

Cuando pasa un argumento a una función, y el argumento es una expresión prvalue del tipo de parámetro de la función, y este tipo no es una referencia, entonces se puede eludir la construcción del prvalue.

void func(std::string str) { ... }

func(std::string("foo"));

Esto dice que cree una cadena temporal, luego muévala al parámetro de función str. La elisión de copia permite que esta expresión cree directamente el objeto en str, en lugar de usar un movimiento+temporal.

Esta es una optimización útil para los casos en los que un constructor se declara ’explícito’. Por ejemplo, podríamos haber escrito lo anterior como func("foo"), pero solo porque string tiene un constructor implícito que convierte const char* en string. Si ese constructor fuera ’explícito’, nos veríamos obligados a usar un temporal para llamar al constructor ’explícito’. La elisión de copia nos evita tener que hacer una copia/mover innecesaria.

Elisión del valor de retorno #

Si devuelve una expresión prvalue de una función, y la expresión prvalue tiene el mismo tipo que el tipo de retorno de la función, entonces la copia del prvalue temporal se puede eludir:

std::string func()
{
  return std::string("foo");
}

Prácticamente todos los compiladores eludirán la construcción temporal en este caso.

Elisión de valor de retorno con nombre #

Si devuelve una expresión de lvalue de una función, y este lvalue:

  • representa una variable automática local a esa función, que será destruida después del return
  • la variable automática no es un parámetro de función
  • y el tipo de la variable es el mismo tipo que el tipo de retorno de la función

Si todos estos son el caso, entonces la copia/movimiento del lvalue se puede elidir:

std::string func()
{
  std::string str("foo");
  //Do stuff
  return str;
}

Los casos más complejos son elegibles para la elisión, pero cuanto más complejo sea el caso, menos probable será que el compilador lo elide:

std::string func()
{
  std::string ret("foo");
  if(some_condition)
  {
    return "bar";
  }
  return ret;
}

El compilador aún podría eludir ret, pero las posibilidades de que lo hagan disminuirán.

Como se indicó anteriormente, no se permite la elisión de parámetros de valor.

std::string func(std::string str)
{
  str.assign("foo");
  //Do stuff
  return str; //No elision possible
}

Copiar elisión de inicialización #

Si usa una expresión prvalue para copiar inicializar una variable, y esa variable tiene el mismo tipo que la expresión prvalue, entonces la copia se puede elidir.

std::string str = std::string("foo");

La inicialización de copia transforma efectivamente esto en std::string str("foo"); (hay pequeñas diferencias).

Esto también funciona con valores de retorno:

std::string func()
{
  return std::string("foo");
}

std::string str = func();

Sin elisión de copia, esto provocaría 2 llamadas al constructor de movimientos de std::string. La elisión de copia permite llamar al constructor de movimiento 1 o cero veces, y la mayoría de los compiladores optarán por lo último.
