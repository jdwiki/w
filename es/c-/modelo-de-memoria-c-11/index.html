Distintos subprocesos que intentan acceder a la misma ubicación de memoria participan en una carrera de datos si al menos una de las operaciones es una modificación (también conocida como operación de almacenamiento). Estas carreras de datos causan un comportamiento indefinido. Para evitarlos, es necesario evitar que estos subprocesos ejecuten simultáneamente tales operaciones conflictivas.

Las primitivas de sincronización (mutex, sección crítica y similares) pueden proteger dichos accesos. El modelo de memoria introducido en C++ 11 define dos nuevas formas portátiles de sincronizar el acceso a la memoria en un entorno de subprocesos múltiples: operaciones atómicas y vallas.

Operaciones Atómicas #

Ahora es posible leer y escribir en una ubicación de memoria determinada mediante el uso de operaciones de carga atómica y almacenamiento atómico. Por conveniencia, estos están envueltos en la clase de plantilla std::atomic<t>. Esta clase envuelve un valor de tipo t pero esta vez las cargas y las tiendas en el objeto son atómicas.

La plantilla no está disponible para todos los tipos. Los tipos que están disponibles son específicos de la implementación, pero esto generalmente incluye la mayoría (o todos) los tipos integrales disponibles, así como los tipos de puntero. Así que std::atomic<unsigned> y std::atomic<std::vector<foo> *> deberían estar disponibles, mientras que std::atomic<std::pair<bool,char>> lo más probable es que no lo sea.

Las operaciones atómicas tienen las siguientes propiedades:

  • Todas las operaciones atómicas se pueden realizar simultáneamente desde múltiples subprocesos sin causar un comportamiento indefinido.
  • Una carga atómica verá el valor inicial con el que se construyó el objeto atómico, o el valor escrito en él a través de alguna operación de almacenamiento atómico.
  • Los almacenes atómicos para el mismo objeto atómico se ordenan de la misma manera en todos los subprocesos. Si un subproceso ya ha visto el valor de alguna operación de almacenamiento atómico, las operaciones posteriores de carga atómica verán el mismo valor o el valor almacenado por la operación de almacenamiento atómico subsiguiente.
  • Las operaciones de lectura-modificación-escritura atómicas permiten que la carga atómica y el almacenamiento atómico sucedan sin otro almacenamiento atómico en el medio. Por ejemplo, se puede incrementar atómicamente un contador a partir de varios subprocesos y no se perderá ningún incremento independientemente de la contención entre los subprocesos.
  • Las operaciones atómicas reciben un parámetro std::memory_order opcional que define qué propiedades adicionales tiene la operación con respecto a otras ubicaciones de memoria.

estándar::pedido_de_memoria | Sentido —————————-|————————————- —- std::memory_order_relaxed | sin restricciones adicionales std::memory_order_release → std::memory_order_acquire | si load-acquire ve el valor almacenado por store-release entonces las tiendas secuenciadas antes de que store-release sucedan antes de las cargas secuenciadas después de la adquisición de carga std::memory_order_consume | como memory_order_acquire pero solo para cargas dependientes std::memory_order_acq_rel | combina load-acquire y store-release std::memory_order_seq_cst | consistencia secuencial

Estas etiquetas de orden de memoria permiten tres disciplinas de orden de memoria diferentes: consistencia secuencial, relajada y liberar-adquirir con su hermano liberar-consumir.

Coherencia secuencial #

Si no se especifica un orden de memoria para una operación atómica, el orden predeterminado es coherencia secuencial. Este modo también se puede seleccionar explícitamente etiquetando la operación con std::memory_order_seq_cst.

Con este orden ninguna operación de memoria puede cruzar la operación atómica. Todas las operaciones de memoria secuenciadas antes de la operación atómica ocurren antes de la operación atómica y la operación atómica ocurre antes que todas las operaciones de memoria que se secuencian después de ella. Este modo es probablemente el más fácil de razonar, pero también conlleva la mayor penalización del rendimiento. También evita todas las optimizaciones del compilador que, de lo contrario, podrían intentar reordenar las operaciones más allá de la operación atómica.

Pedidos relajados #

Lo opuesto a la coherencia secuencial es el ordenamiento de la memoria relajado. Se selecciona con la etiqueta std::memory_order_relaxed. La operación atómica relajada no impondrá restricciones a otras operaciones de memoria. El único efecto que queda es que la operación en sí sigue siendo atómica.

Pedido de liberación-adquisición #

Una operación de almacenamiento atómico se puede etiquetar con std::memory_order_release y una operación de carga atómica se puede etiquetar con std::memory_order_acquire. La primera operación se llama (atomic) store-release mientras que la segunda se llama (atomic) load-acquire.

Cuando load-acquire ve el valor escrito por store-release, ocurre lo siguiente: todas las operaciones de almacenamiento secuenciadas antes de store-release se vuelven visibles para (pasar antes) las operaciones de carga que están secuenciadas después de load -adquirir.

Las operaciones atómicas de lectura, modificación y escritura también pueden recibir la etiqueta acumulativa std::memory_order_acq_rel. Esto hace que la porción de carga atómica de la operación sea una adquisición de carga atómica mientras que la porción de almacenamiento atómico se convierte en liberación de almacenamiento atómico.

El compilador no puede mover operaciones de almacenamiento después de una operación liberación de almacenamiento atómico. Tampoco está permitido mover operaciones de carga antes de carga-adquisición atómica (o carga-consumo).

También tenga en cuenta que no hay liberación de carga atómica o adquisición de almacenamiento atómico. Intentar crear tales operaciones las convierte en operaciones relajadas.

Pedido de liberación-consumo #

Esta combinación es similar a liberar-adquirir, pero esta vez la carga atómica está etiquetada con std::memory_order_consume y se convierte en una operación (atómica) de carga-consumo. Este modo es el mismo que liberar-adquirir con la única diferencia que entre las operaciones de carga secuenciadas después de cargar-consumir solo se ordenan estas dependiendo del valor cargado por cargar-consumir.

Cercas #

Las vallas también permiten ordenar operaciones de memoria entre subprocesos. Una cerca es una cerca de liberación o una cerca de adquisición.

Si un límite de liberación ocurre antes que un límite de adquisición, las tiendas secuenciadas antes del límite de liberación son visibles para las cargas secuenciadas después del límite de adquisición. Para garantizar que el límite de liberación ocurra antes que el límite de adquisición, se pueden usar otras primitivas de sincronización, incluidas las operaciones atómicas relajadas.

Necesidad de modelo de memoria #

int x, y;
bool ready = false;

void init()
{
  x = 2;
  y = 3;
  ready = true;
}
void use()
{
  if (ready)
    std::cout << x + y;
}

Un subproceso llama a la función init() mientras que otro subproceso (o controlador de señal) llama a la función use(). Uno podría esperar que la función use() imprima 5 o no haga nada. Esto puede no ser siempre el caso por varias razones:

  • La CPU puede reordenar las escrituras que ocurren en init() para que el código que realmente se ejecuta se vea así:

     void init()
 {
   ready = true;
   x = 2;
   y = 3;
 }

  • La CPU puede reordenar las lecturas que ocurren en use() para que el código realmente ejecutado se convierta en:

     void use()
 {
   int local_x = x;
   int local_y = y;
   if (ready)
     std::cout << local_x + local_y;
 }

  • Un compilador C++ optimizador puede decidir reordenar el programa de manera similar.

Este reordenamiento no puede cambiar el comportamiento de un programa que se ejecuta en un único subproceso porque un subproceso no puede intercalar las llamadas a init() y use(). Por otro lado, en una configuración de subprocesos múltiples, un subproceso puede ver parte de las escrituras realizadas por el otro subproceso donde puede suceder que use() pueda ver ready==true y basura en x o y o ambos.

El modelo de memoria de C++ permite al programador especificar qué operaciones de reordenación están permitidas y cuáles no, de modo que un programa de subprocesos múltiples también pueda comportarse como se espera. El ejemplo anterior se puede reescribir de manera segura para subprocesos de esta manera:

int x, y;
std::atomic<bool> ready{false};

void init()
{
  x = 2;
  y = 3;
  ready.store(true, std::memory_order_release);
}
void use()
{
  if (ready.load(std::memory_order_acquire))
    std::cout << x + y;
}

Aquí init() realiza la operación liberación de almacenamiento atómico. Esto no solo almacena el valor true en ready, sino que también le dice al compilador que no puede mover esta operación antes de escribir operaciones que están secuenciadas antes de ella.

La función use() realiza una operación de carga-adquisición atómica. Lee el valor actual de ready y también prohíbe al compilador colocar operaciones de lectura secuenciadas después para que ocurran antes de la adquisición de carga atómica.

Estas operaciones atómicas también hacen que el compilador coloque las instrucciones de hardware que sean necesarias para informar a la CPU que se abstenga de los reordenamientos no deseados.

Debido a que liberación de almacenamiento atómico está en la misma ubicación de memoria que adquisición de carga atómica, el modelo de memoria estipula que si la operación adquisición de carga ve el valor escrito por la operación liberación de almacenamiento, entonces todas las escrituras realizadas por el subproceso de init() antes de ese store-release serán visibles para las cargas que el subproceso de use() ejecuta después de su carga-adquisición. Es decir, si use() ve ready==true, entonces está garantizado que verá x==2 y y==3.

Tenga en cuenta que el compilador y la CPU aún pueden escribir en y antes de escribir en x y, de manera similar, las lecturas de estas variables en use() pueden ocurrir en cualquier orden.

Ejemplo de valla #

El ejemplo anterior también se puede implementar con vallas y operaciones atómicas relajadas:

int x, y;
std::atomic<bool> ready{false};

void init()
{
  x = 2;
  y = 3;
  atomic_thread_fence(std::memory_order_release);
  ready.store(true, std::memory_order_relaxed);
}
void use()
{
  if (ready.load(std::memory_order_relaxed))
  {
    atomic_thread_fence(std::memory_order_acquire);
    std::cout << x + y;
  }
}

Si la operación de carga atómica ve el valor escrito por el almacén atómico, entonces el almacenamiento ocurre antes de la carga, y también lo hacen las vallas: la valla de liberación ocurre antes que la valla de adquisición, lo que hace que las escrituras en x e y precedan a la liberación. valla para que sea visible para la sentencia std::cout que sigue a la valla de adquisición.

Una valla puede ser beneficiosa si puede reducir el número total de operaciones de adquisición, liberación u otras operaciones de sincronización. Por ejemplo:

void block_and_use()
{
  while (!ready.load(std::memory_order_relaxed))
    ;
  atomic_thread_fence(std::memory_order_acquire);
  std::cout << x + y;
}

La función block_and_use() gira hasta que se establece el indicador ready con la ayuda de una carga atómica relajada. A continuación, se utiliza una barrera de adquisición única para proporcionar el ordenamiento de memoria necesario.
