El preprocesador C es un simple analizador/reemplazador de texto que se ejecuta antes de la compilación real del código. Se utiliza para ampliar y facilitar el uso del lenguaje C (y luego C++), se puede utilizar para:

una. Incluyendo otros archivos usando #include

b. Defina una macro de reemplazo de texto usando #define

C. Compilación condicional usando #if #ifdef

d. Lógica específica de plataforma/compilador (como una extensión de la compilación condicional)

Las declaraciones del preprocesador se ejecutan antes de que los archivos fuente se entreguen al compilador. Son capaces de lógica condicional de muy bajo nivel. Dado que las construcciones de preprocesador (p. ej., macros similares a objetos) no se escriben como funciones normales (el paso de preprocesamiento ocurre antes de la compilación), el compilador no puede aplicar controles de tipo, por lo tanto, deben usarse con cuidado.

Incluir guardias #

Un archivo de encabezado puede estar incluido en otros archivos de encabezado. Por lo tanto, un archivo fuente (unidad de compilación) que incluye múltiples encabezados puede, indirectamente, incluir algunos encabezados más de una vez. Si un archivo de encabezado de este tipo que se incluye más de una vez contiene definiciones, el compilador (después del preprocesamiento) detecta una violación de la Regla de definición única (por ejemplo, §3.2 del estándar C++ de 2003) y, por lo tanto, emite un diagnóstico y la compilación falla.

Se evita la inclusión múltiple utilizando “guardias de inclusión”, que a veces también se conocen como protecciones de encabezado o protecciones de macro. Estos se implementan usando las directivas del preprocesador #define, #ifndef, #endif.

p.ej.

// Foo.h
#ifndef FOO_H_INCLUDED 
#define FOO_H_INCLUDED

class Foo    //  a class definition
{
};

#endif

La ventaja clave de usar guardias de inclusión es que funcionarán con todos los compiladores y preprocesadores que cumplan con los estándares.

Sin embargo, incluir protecciones también causa algunos problemas a los desarrolladores, ya que es necesario asegurarse de que las macros sean únicas dentro de todos los encabezados utilizados en un proyecto. Específicamente, si dos (o más) encabezados usan FOO_H_INCLUDED como su protección de inclusión, el primero de esos encabezados incluidos en una unidad de compilación evitará que los demás se incluyan. Se presentan desafíos particulares si un proyecto usa una cantidad de bibliotecas de terceros con archivos de encabezado que usan protectores de inclusión en común.

También es necesario asegurarse de que las macros utilizadas en las protecciones de inclusión no entren en conflicto con ninguna otra macro definida en los archivos de encabezado.

La mayoría de las implementaciones de C++ también admiten la directiva #pragma once, que garantiza que el archivo solo se incluye una vez dentro de una única compilación. Esta es una directiva de facto estándar, pero no forma parte de ningún estándar ISO C++. Por ejemplo:

// Foo.h
#pragma once

class Foo
{
};

Mientras que #pragma once evita algunos problemas asociados con las protecciones de inclusión, un #pragma, por definición en los estándares, es inherentemente un gancho específico del compilador, y será ignorado silenciosamente por los compiladores que no lo admitan. Los proyectos que usan #pragma una vez son más difíciles de portar a compiladores que no lo soportan.

Una serie de pautas de codificación y estándares de seguridad para C++ desaconsejan específicamente cualquier uso del preprocesador que no sea para archivos de encabezado ‘#include’ o con el fin de colocar protecciones de inclusión en los encabezados.

Lógica condicional y manejo multiplataforma #

En pocas palabras, la lógica de preprocesamiento condicional consiste en hacer que la lógica del código esté disponible o no disponible para la compilación mediante definiciones de macro.

Tres casos de uso destacados son:

  • diferentes perfiles de aplicaciones (p. ej., depuración, lanzamiento, pruebas, optimización) que pueden ser candidatos de la misma aplicación (p. ej., con registro adicional).
  • compilaciones multiplataforma: código base único, múltiples plataformas de compilación.
  • utilizando una base de código común para múltiples versiones de aplicaciones (por ejemplo, versiones Básica, Premium y Pro de un software), con características ligeramente diferentes.

Ejemplo a: Un enfoque multiplataforma para eliminar archivos (ilustrativo):

#ifdef _WIN32
#include <windows.h> // and other windows system files
#endif
#include <cstdio>

bool remove_file(const std::string &path) 
{
#ifdef _WIN32
  return DeleteFile(path.c_str());
#elif defined(_POSIX_VERSION) || defined(__unix__)
  return (0 == remove(path.c_str()));
#elif defined(__APPLE__)
  //TODO: check if NSAPI has a more specific function with permission dialog
  return (0 == remove(path.c_str()));
#else 
#error "This platform is not supported"
#endif
}

Las macros como _WIN32, __APPLE__ o __unix__ normalmente están predefinidas por las implementaciones correspondientes.

Ejemplo b: Habilitación de registro adicional para una compilación de depuración:

void s_PrintAppStateOnUserPrompt()
{
    std::cout << "--------BEGIN-DUMP---------------\n"
              << AppState::Instance()->Settings().ToString() << "\n"
#if ( 1 == TESTING_MODE ) //privacy: we want user details only when testing
              << ListToString(AppState::UndoStack()->GetActionNames())
              << AppState::Instance()->CrntDocument().Name() 
              << AppState::Instance()->CrntDocument().SignatureSHA() << "\n"
#endif
              << "--------END-DUMP---------------\n"
}

Ejemplo c: Habilite una función premium en una compilación de producto separada (nota: esto es ilustrativo. A menudo es una mejor idea permitir que una función se desbloquee sin la necesidad de reinstalar una aplicación)

void MainWindow::OnProcessButtonClick()
{
#ifndef _PREMIUM
    CreatePurchaseDialog("Buy App Premium", "This feature is available for our App Premium users. Click the Buy button to purchase the Premium version at our website");
    return;
#endif
    //...actual feature logic here
}

Algunos trucos comunes:

Definición de símbolos en el momento de la invocación:

El preprocesador se puede llamar con símbolos predefinidos (con inicialización opcional). Por ejemplo, este comando (gcc -E ejecuta solo el preprocesador)

gcc -E -DOPTIMISE_FOR_OS_X -DTESTING_MODE=1 Sample.cpp

procesa Sample.cpp de la misma manera que lo haría si #define OPTIMISE_FOR_OS_X y #define TESTING_MODE 1 se agregaran a la parte superior de Sample.cpp.

Asegurarse de que se define una macro:

Si una macro no está definida y su valor se compara o verifica, el preprocesador casi siempre asume silenciosamente que el valor es 0. Hay algunas maneras de trabajar con esto. Un enfoque es asumir que la configuración predeterminada se representa como 0, y cualquier cambio (por ejemplo, en el perfil de compilación de la aplicación) debe realizarse explícitamente (por ejemplo, ENABLE_EXTRA_DEBUGGING=0 de forma predeterminada, establezca -DENABLE_EXTRA_DEBUGGING=1 para anular). Otro enfoque es hacer explícitas todas las definiciones y los valores predeterminados. Esto se puede lograr usando una combinación de las directivas #ifndef y #error:

#ifndef (ENABLE_EXTRA_DEBUGGING)
// please include DefaultDefines.h if not already included.
#    error "ENABLE_EXTRA_DEBUGGING is not defined"
#else
#    if ( 1 == ENABLE_EXTRA_DEBUGGING )
  //code
#    endif
#endif

macros #

Las macros se clasifican en dos grupos principales: macros similares a objetos y macros similares a funciones. Las macros se tratan como una sustitución de token al principio del proceso de compilación. Esto significa que las secciones grandes (o repetitivas) de código se pueden abstraer en una macro de preprocesador.

// This is an object-like macro
#define    PI         3.14159265358979

// This is a function-like macro.
// Note that we can use previously defined macros
// in other macro definitions (object-like or function-like)
// But watch out, its quite useful if you know what you're doing, but the
// Compiler doesnt know which type to handle, so using inline functions instead
// is quite recommended (But e.g. for Minimum/Maximum functions it is quite useful)
#define    AREA(r)    (PI*(r)*(r))

// They can be used like this:
double pi_macro   = PI;
double area_macro = AREA(4.6);

La biblioteca Qt hace uso de esta técnica para crear un sistema de metaobjetos haciendo que el usuario declare la macro Q_OBJECT a la cabeza de la clase definida por el usuario que extiende QObject.

Los nombres de las macros generalmente se escriben en mayúsculas, para que sea más fácil diferenciarlos del código normal. Esto no es un requisito, pero muchos programadores simplemente lo consideran un buen estilo.

Cuando se encuentra una macro similar a un objeto, se expande como una simple operación de copiar y pegar, y el nombre de la macro se reemplaza con su definición. Cuando se encuentra una macro similar a una función, se expanden tanto su nombre como sus parámetros.

double pi_squared = PI * PI;
// Compiler sees:
double pi_squared = 3.14159265358979 * 3.14159265358979;

double area = AREA(5);
// Compiler sees:
double area = (3.14159265358979*(5)*(5))

Debido a esto, los parámetros de macro similares a funciones a menudo se encierran entre paréntesis, como en AREA() arriba. Esto es para evitar cualquier error que pueda ocurrir durante la expansión de macros, específicamente errores causados ​​por un solo parámetro de macro que se compone de múltiples valores reales.

#define BAD_AREA(r) PI * r * r

double bad_area = BAD_AREA(5 + 1.6);
// Compiler sees:
double bad_area = 3.14159265358979 * 5 + 1.6 * 5 + 1.6;

double good_area = AREA(5 + 1.6);
// Compiler sees:
double good_area = (3.14159265358979*(5 + 1.6)*(5 + 1.6));

También tenga en cuenta que debido a esta simple expansión, se debe tener cuidado con los parámetros pasados ​​a las macros, para evitar efectos secundarios inesperados. Si el parámetro se modifica durante la evaluación, se modificará cada vez que se use en la macro expandida, lo que generalmente no es lo que queremos. Esto es cierto incluso si la macro encierra los parámetros entre paréntesis para evitar que la expansión rompa algo.

int oops = 5;
double incremental_damage = AREA(oops++);
// Compiler sees:
double incremental_damage = (3.14159265358979*(oops++)*(oops++));

Además, las macros no proporcionan seguridad de tipos, lo que genera errores difíciles de entender sobre la falta de coincidencia de tipos.

Como los programadores normalmente terminan las líneas con un punto y coma, las macros que están destinadas a usarse como líneas independientes a menudo están diseñadas para “tragarse” un punto y coma; esto evita que cualquier error no deseado sea causado por un punto y coma adicional.

#define IF_BREAKER(Func) Func();

if (some_condition)
    // Oops.
    IF_BREAKER(some_func);
else
    std::cout << "I am accidentally an orphan." << std::endl;

En este ejemplo, el doble punto y coma involuntario rompe el bloque if...else, evitando que el compilador haga coincidir else con if. Para evitar esto, el punto y coma se omite de la definición de la macro, lo que hará que se “trague” el punto y coma inmediatamente después de cualquier uso del mismo.

#define IF_FIXER(Func) Func()

if (some_condition)
    IF_FIXER(some_func);
else
    std::cout << "Hooray!  I work again!" << std::endl;

Omitir el punto y coma final también permite usar la macro sin finalizar la declaración actual, lo que puede ser beneficioso.

#define DO_SOMETHING(Func, Param) Func(Param, 2)

// ...

some_function(DO_SOMETHING(some_func, 3), DO_SOMETHING(some_func, 42));

Normalmente, una definición de macro termina al final de la línea. Sin embargo, si una macro necesita cubrir varias líneas, se puede usar una barra invertida al final de una línea para indicarlo. Esta barra invertida debe ser el último carácter de la línea, lo que indica al preprocesador que la siguiente línea debe concatenarse en la línea actual, tratándola como una sola línea. Esto se puede utilizar varias veces seguidas.

#define TEXT "I \
am \
many \
lines."

// ...

std::cout << TEXT << std::endl; // Output:   I am many lines.

Esto es especialmente útil en macros complejas similares a funciones, que pueden necesitar cubrir varias líneas.

#define CREATE_OUTPUT_AND_DELETE(Str) \
    std::string* tmp = new std::string(Str); \
    std::cout << *tmp << std::endl; \
    delete tmp;

// ...

CREATE_OUTPUT_AND_DELETE("There's no real need for this to use 'new'.")

En el caso de macros similares a funciones más complejas, puede ser útil darles su propio alcance para evitar posibles colisiones de nombres o hacer que los objetos se destruyan al final de la macro, de forma similar a una función real. Un modismo común para esto es do while 0, donde la macro está encerrada en un bloque do-while. Este bloque generalmente no va seguido de un punto y coma, lo que le permite tragarse un punto y coma.

#define DO_STUFF(Type, Param, ReturnVar) do { \
    Type temp(some_setup_values); \
    ReturnVar = temp.process(Param); \
} while (0)

int x;
DO_STUFF(MyClass, 41153.7, x);

// Compiler sees:

int x;
do {
    MyClass temp(some_setup_values);
    x = temp.process(41153.7);
} while (0);

También hay macros variadas; De manera similar a las funciones variádicas, estas toman un número variable de argumentos y luego los expanden en lugar de un parámetro especial “Varargs”, __VA_ARGS__.

#define VARIADIC(Param, ...) Param(__VA_ARGS__)

VARIADIC(printf, "%d", 8);
// Compiler sees:
printf("%d", 8);

Tenga en cuenta que durante la expansión, __VA_ARGS__ se puede colocar en cualquier lugar de la definición y se expandirá correctamente.

#define VARIADIC2(POne, PTwo, PThree, ...) POne(PThree, __VA_ARGS__, PTwo)

VARIADIC2(some_func, 3, 8, 6, 9);
// Compiler sees:
some_func(8, 6, 9, 3);

En el caso de un parámetro variádico de argumento cero, diferentes compiladores manejarán la coma final de manera diferente. Algunos compiladores, como Visual Studio, tragarán silenciosamente la coma sin ninguna sintaxis especial. Otros compiladores, como GCC, requieren que coloque ## inmediatamente antes de __VA_ARGS__. Debido a esto, es aconsejable definir condicionalmente macros variadicas cuando la portabilidad es una preocupación.

// In this example, COMPILER is a user-defined macro specifying the compiler being used.

#if       COMPILER == "VS"
    #define VARIADIC3(Name, Param, ...) Name(Param, __VA_ARGS__)
#elif     COMPILER == "GCC"
    #define VARIADIC3(Name, Param, ...) Name(Param, ##__VA_ARGS__)
#endif /* COMPILER */

X-macros #

Una técnica idiomática para generar estructuras de código repetitivo en tiempo de compilación.

Una macro X consta de dos partes: la lista y la ejecución de la lista.

Ejemplo:

#define LIST \
    X(dog)   \
    X(cat)   \
    X(racoon)

// class Animal {
//  public:
//    void say();
// };

#define X(name) Animal name;
LIST
#undef X

int main() {
#define X(name) name.say();
    LIST
#undef X

    return 0;
}

que se expande por el preprocesador en lo siguiente:

Animal dog;
Animal cat;
Animal racoon;

int main() {
    dog.say();
    cat.say();
    racoon.say();

    return 0;
}

A medida que las listas se hacen más grandes (digamos, más de 100 elementos), esta técnica ayuda a evitar el exceso de copiar y pegar.

Fuente: https://en.wikipedia.org/wiki/X_Macro

Ver también: https://www.wikiod.com/es/c/x-macros

Si definir una ‘X’ totalmente irrelevante antes de usar ‘LISTA’ no es de su agrado, también puede pasar un nombre de macro como argumento:

#define LIST(MACRO) \
    MACRO(dog) \
    MACRO(cat) \
    MACRO(racoon)

Ahora, especifica explícitamente qué macro se debe usar al expandir la lista, p.

#define FORWARD_DECLARE_ANIMAL(name) Animal name;
LIST(FORWARD_DECLARE_ANIMAL)

Si cada invocación de la MACRO debe tomar parámetros adicionales - constantes con respecto a la lista, se pueden usar macros variádicas

//a walkaround for Visual studio
#define EXPAND(x) x

#define LIST(MACRO, ...) \
    EXPAND(MACRO(dog, __VA_ARGS__)) \
    EXPAND(MACRO(cat, __VA_ARGS__)) \
    EXPAND(MACRO(racoon, __VA_ARGS__))

El primer argumento lo proporciona LIST, mientras que el resto lo proporciona el usuario en la invocación de LIST. Por ejemplo:

#define FORWARD_DECLARE(name, type, prefix) type prefix##name;
LIST(FORWARD_DECLARE,Animal,anim_)
LIST(FORWARD_DECLARE,Object,obj_)

se expandirá a

Animal anim_dog;
Animal anim_cat;
Animal anim_racoon;
Object obj_dog;
Object obj_cat;
Object obj_racoon;

Mensajes de error del preprocesador #

Los errores de compilación se pueden generar utilizando el preprocesador. Esto es útil por varias razones, algunas de las cuales incluyen notificar a un usuario si se encuentra en una plataforma no compatible o en un compilador no compatible.

p.ej. Devuelve un error si la versión de gcc es 3.0.0 o anterior.

#if __GNUC__ < 3
#error "This code requires gcc > 3.0.0"
#endif

p.ej. Error de retorno si se compila en una computadora Apple.

#ifdef __APPLE__
#error "Apple products are not supported in this release"
#endif

Macros predefinidas #

Las macros predefinidas son aquellas que define el compilador (a diferencia de las definidas por el usuario en el archivo fuente). Esas macros no deben ser redefinidas o no definidas por el usuario.

Las siguientes macros están predefinidas por el estándar C++:

  • __LINE__ contiene el número de línea de la línea en la que se usa esta macro, y se puede cambiar mediante la directiva #line.
  • __FILE__ contiene el nombre del archivo en el que se usa esta macro, y se puede cambiar mediante la directiva #line.
  • __DATE__ contiene la fecha (en formato "Mmm dd aaaa") de la compilación del archivo, donde Mmm se formatea como si se hubiera obtenido mediante una llamada a std::asctime().
  • __TIME__ contiene el tiempo (en formato "hh:mm:ss") de la compilación del archivo.
  • __cplusplus está definido por compiladores C++ (conformes) al compilar archivos C++. Su valor es la versión estándar con la que el compilador cumple totalmente, es decir, 199711L para C++98 y C++03, 201103L para C++11 y 201402L para el estándar C++14 .
  • __STDC_HOSTED__ se define como 1 si la implementación está alojada, o 0 si es independiente.
  • __STDCPP_DEFAULT_NEW_ALIGNMENT__ contiene un literal size_t, que es la alineación utilizada para una llamada al operador nuevo que no reconoce la alineación.

Además, las implementaciones pueden predefinir las siguientes macros, y pueden o no estar presentes:

  • __STDC__ tiene un significado que depende de la implementación y, por lo general, se define solo cuando se compila un archivo como C, para indicar el cumplimiento completo del estándar C. (O nunca, si el compilador decide no admitir esta macro).
  • __STDC_VERSION__ tiene un significado que depende de la implementación, y su valor suele ser la versión C, de forma similar a como __cplusplus es la versión C++. (O ni siquiera está definido, si el compilador decide no admitir esta macro).
  • __STDC_MB_MIGHT_NEQ_WC__ se define como 1, si los valores de la codificación estrecha del juego de caracteres básico pueden no ser iguales a los valores de sus homólogos anchos (por ejemplo, si (uintmax_t)'x' != (uintmax_t)L' x')
  • __STDC_ISO_10646__ se define si wchar_t está codificado como Unicode y se expande a una constante entera en el formato yyyymmL, lo que indica la última revisión de Unicode admitida.
  • __STDCPP_STRICT_POINTER_SAFETY__ está definido en 1, si la implementación tiene seguridad de puntero estricta (de lo contrario, tiene seguridad de puntero relajada)
  • __STDCPP_THREADS__ se define como 1, si el programa puede tener más de un hilo de ejecución (aplicable a implementación independiente — implementaciones alojadas siempre pueden tener más de un hilo)

También vale la pena mencionar __func__, que no es una macro, sino una variable local de función predefinida. Contiene el nombre de la función en la que se utiliza, como una matriz de caracteres estáticos en un formato definido por la implementación.

Además de esas macros predefinidas estándar, los compiladores pueden tener su propio conjunto de macros predefinidas. Uno debe consultar la documentación del compilador para aprenderlos. P.ej.:

  • gcc
  • Microsoft Visual C++
  • sonido
  • Compilador Intel C++

Algunas de las macros son solo para consultar el soporte de alguna función:

#ifdef __cplusplus // if compiled by C++ compiler
extern "C"{ // C code has to be decorated
   // C library header declarations here
}
#endif

Otros son muy útiles para la depuración:

bool success = doSomething( /*some arguments*/ );
if( !success ){
    std::cerr << "ERROR: doSomething() failed on line " << __LINE__ - 2
              << " in function " << __func__ << "()"
              << " in file " << __FILE__
              << std::endl;
}

Y otros para el control de versiones triviales:

int main( int argc, char *argv[] ){
    if( argc == 2 && std::string( argv[1] ) == "-v" ){
        std::cout << "Hello World program\n"
                  << "v 1.1\n" // I have to remember to update this manually
                  << "compiled: " << __DATE__ << ' ' << __TIME__ // this updates automagically
                  << std::endl;
    }
    else{
        std::cout << "Hello World!\n";
    }
}

#pragma una vez #

La mayoría, pero no todas, las implementaciones de C++ admiten la directiva #pragma once que garantiza que el archivo solo se incluye una vez dentro de una sola compilación. No es parte de ningún estándar ISO C++. Por ejemplo:

// Foo.h
#pragma once

class Foo
{
};

Mientras que #pragma once evita algunos problemas asociados con include guards, un #pragma - por definición en los estándares - es inherentemente un enlace específico del compilador, y será ignorado en silencio por los compiladores que no lo hagan. apoyarlo Los proyectos que usan #pragma once deben modificarse para cumplir con los estándares.

Con algunos compiladores, particularmente aquellos que emplean cabeceras precompiladas, #pragma once puede acelerar considerablemente el proceso de compilación. De manera similar, algunos preprocesadores logran acelerar la compilación rastreando qué encabezados han empleado protectores de inclusión. El beneficio neto, cuando se emplean #pragma once e include guards, depende de la implementación y puede ser un aumento o una disminución de los tiempos de compilación.

#pragma once combinado con include guards era el diseño recomendado para los archivos de encabezado al escribir aplicaciones basadas en MFC en Windows, y fue generado por los asistentes add class, add dialog, add windows de Visual Studio . De ahí que sea muy común encontrarlos combinados en C++ Windows Applicants.

Operadores de preprocesador #

El operador # o el operador de cadena se utiliza para convertir un parámetro de macro en un literal de cadena. Solo se puede usar con las macros que tienen argumentos.

// preprocessor will convert the parameter x to the string literal x
#define PRINT(x) printf(#x "\n")

PRINT(This line will be converted to string by preprocessor);
// Compiler sees
printf("This line will be converted to string by preprocessor""\n");

El compilador concatena dos cadenas y el argumento printf() final será una cadena literal con un carácter de nueva línea al final.

El preprocesador ignorará los espacios antes o después del argumento macro. Así que debajo de la declaración de impresión nos dará el mismo resultado.

PRINT(   This line will be converted to string by preprocessor );

Si el parámetro del literal de cadena requiere una secuencia de escape como antes de una comilla doble (), el preprocesador la insertará automáticamente.

PRINT(This "line" will be converted to "string" by preprocessor); 
// Compiler sees
printf("This \"line\" will be converted to \"string\" by preprocessor""\n");

El operador ## o el operador de pegado de tokens se utiliza para concatenar dos parámetros o tokens de una macro.

// preprocessor will combine the variable and the x
#define PRINT(x) printf("variable" #x " = %d", variable##x)

int variableY = 15;
PRINT(Y);
//compiler sees
printf("variable""Y"" = %d", variableY);

y el resultado final será

variableY = 15
