Configuración básica de OpenCL

Antes de utilizar OpenCL, uno debe configurar su código para usarlo. Este tema se enfoca en cómo poner en marcha opencl en su proyecto y ejecutar un núcleo básico. Los ejemplos se basan en el contenedor de C# OpenCL.NET, pero como el contenedor no agrega abstracción a OpenCL, el código probablemente también se ejecutará con muy pocos cambios en C/C++.

Las llamadas en C# pueden tener el siguiente aspecto: ‘Cl.GetPlatformIDs’. Para la API OpenCL de estilo C, llamaría ‘clGetPlatformIDs’ y para el estilo C++ uno ‘cl::GetPlatformIDs’

  • NVidia, AMD e Intel tienen implementaciones ligeramente diferentes de OpenCL, pero las diferencias conocidas (según mi experiencia) se limitan a los requisitos de paréntesis y conversiones implícitas. A veces, NVidia bloqueará su kernel al intentar averiguar la sobrecarga correcta para un método. En este caso, ayuda ofrecer un reparto explícito para ayudar a la GPU. El problema se observó para núcleos compilados en tiempo de ejecución.

  • Para obtener más información sobre las llamadas utilizadas en este tema, basta con buscar en Google ‘OpenCL’ seguido del nombre de la función. El grupo Khronos dispone de una completa documentación on all parameters and datatypes available on their website.

Inicializando el dispositivo de destino #

Los núcleos de OpenCL se pueden ejecutar en la GPU o en la CPU. Esto permite soluciones alternativas, en las que el cliente puede tener un sistema muy desactualizado. El programador también puede optar por limitar su funcionalidad a la CPU o GPU.

Para comenzar a usar OpenCL, necesitará un ‘Contexto’ y un ‘Dispositivo’. Ambas son estructuras definidas por la API de OpenCL (también conocidas como cl::Context o clContext & ~Device) y definen el procesador de destino utilizado.

Para obtener su dispositivo y contexto, debe consultar una lista de plataformas disponibles, cada una de las cuales puede alojar varios dispositivos. Una plataforma representa su GPU y CPU físicas, mientras que un dispositivo puede distinguir aún más las unidades informáticas contenidas. Para las GPU, la mayoría de las plataformas solo tendrán un dispositivo. Pero una CPU puede ofrecer una GPU integrada adicional además de sus capacidades de CPU.

El contexto gestiona la memoria, las colas de comandos, los diferentes núcleos y programas. Un contexto puede limitarse a un solo dispositivo pero también hacer referencia a varios dispositivos.

Una nota rápida de API antes de comenzar a codificar: casi todas las llamadas a OpenCL le brindan un valor de error, ya sea como valor de retorno o mediante un valor de referencia (puntero en C). Ahora comencemos.

ErrorCode err;
var platforms = Cl.GetPlatformIDs(out err);
if(!CheckError(err, "Cl.GetPlatformIDs")) return;
foreach (var platform in platforms) {
    foreach (var device in Cl.GetDeviceIDs(platform, DeviceType.Gpu, out err)) {
        if(!CheckError(err, "Cl.GetDeviceIDs")) continue;
        [...]
    }
}

Este fragmento de código consulta todos los dispositivos GPU disponibles en el sistema. Ahora puede agregarlos a una lista o comenzar su contexto directamente con la primera coincidencia. La función ‘CheckError(…)’ es una utilidad simple que verifica si el código de error tiene el valor correcto o uno diferente y puede ofrecerle algo de registro. Se recomienda usar una función o macro separada, porque la llamará mucho.

ErrorCode es solo una enumeración en el tipo de datos cl_int para C#, C/C++ puede comparar el valor int con las constantes de error predefinidas que se enumeran aquí: https://www.khronos.org/registry/OpenCL/sdk/1.0/docs/man/xhtml/errors.html

También es posible que desee verificar si el dispositivo es compatible con todas las funciones necesarias, de lo contrario, sus kernels podrían fallar en el tiempo de ejecución. Puede consultar la capacidad de un dispositivo con

Cl.GetDeviceInfo(_device, DeviceInfo.ImageSupport, out err)

Este ejemplo le pregunta al dispositivo si puede ejecutar funciones de imagen. Para el paso siguiente y final, necesitamos construir nuestro contexto a partir de los dispositivos recopilados.

_context = Cl.CreateContext(null, 1, new[] { _device }, ContextNotify, IntPtr.Zero, out err);

Algunas cosas están pasando aquí. Para la gente de C/C++, IntPtr es una dirección de puntero en C#. Me concentraré en las partes importantes aquí.

  • El segundo parámetro define la cantidad de dispositivos que desea utilizar
  • El tercer parámetro es una matriz de esos dispositivos (o un puntero en C/C++)
  • Y el tercer parámetro es un puntero de función para una función de devolución de llamada. Esta función se utilizará siempre que ocurran errores dentro del contexto.

Para un uso posterior, deberá conservar sus dispositivos usados ​​y el contexto en alguna parte.

Cuando haya terminado toda su interacción con OpenCL, deberá liberar el contexto nuevamente con

Cl.ReleaseContext(_context);

Compilando tu Kernel #

Los núcleos se pueden compilar en tiempo de ejecución en el dispositivo de destino. Para hacerlo, necesita

  • el código fuente del núcleo
  • el dispositivo de destino en el que compilar
  • un contexto construido con el dispositivo de destino

Una actualización terminológica rápida: un programa contiene una colección de núcleos. Puede pensar en un programa como un archivo fuente completo de C/C++/C#, mientras que los núcleos son los diferentes miembros de función de ese archivo.

Primero deberá crear un programa a partir de su código fuente.

var program = Cl.CreateProgramWithSource(_context, 1, new[] { source }, null, out err);

Puede combinar múltiples archivos fuente en un programa y compilarlos juntos, lo que le permite tener kernels en diferentes archivos y compilarlos juntos de una sola vez.

En el siguiente paso, deberá compilar el programa en su dispositivo de destino.

err = Cl.BuildProgram(program, 1, new[] { _device }, string.Empty, null, IntPtr.Zero);

Ahora aquí viene una pequeña advertencia: el código de error solo le dice si la llamada a la función en sí fue exitosa, pero no si su código realmente se compiló. Para verificar eso, tenemos que consultar algunos datos adicionales.

BuildStatus status;
status = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Status, out err).CastTo<BuildStatus>();
if (status != BuildStatus.Success) {
    var log = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Log, out err);
}

Las personas de C/C++ pueden ignorar la conversión al final y simplemente comparar el entero devuelto con la constante correspondiente.

La primera llamada verifica si nuestra compilación fue realmente exitosa. Si no, podemos recuperar un registro y ver exactamente dónde salió mal. Consulte los comentarios para ver algunos errores comunes con respecto a las diferentes plataformas.

Una vez que se construye el programa, debe extraer sus diferentes núcleos del programa compilado. Para hacerlo, crea sus núcleos con

_kernel = Cl.CreateKernel(_program, kernel, out err);

donde ‘kernel’ es una cadena del nombre del kernel. Cuando haya terminado con su núcleo, debe liberarlo con

Cl.ReleaseKernel(_kernel);

Crear una cola de comandos #

Para iniciar cualquier operación en sus dispositivos, necesitará una cola de comandos para cada dispositivo. La cola realiza un seguimiento de las diferentes llamadas que realizó al dispositivo de destino y las mantiene en orden. La mayoría de los comandos también se pueden ejecutar en modo de bloqueo o no bloqueo.

Crear una cola es bastante sencillo:

_queue = Cl.CreateCommandQueue(_context, _device, CommandQueueProperties.None, out err);

La interacción básica con su cola de comandos es poner en cola las diferentes operaciones que desea realizar, p. copie datos hacia y desde su dispositivo e inicie un kernel.

Cuando haya terminado de usar la cola de comandos, debe liberar la cola con una llamada a

Cl.ReleaseCommandQueue(_queue);

Ejecutando el Kernel #

Así que ahora llegamos a lo real, ejecutando sus núcleos en el dispositivo paralelo. Lea acerca de los conceptos básicos de hardware para comprender completamente el envío del kernel.

Primero deberá establecer los argumentos del kernel antes de llamar al kernel. Esto se hace a través de

err = Cl.SetKernelArg(_kernel, $argumentIndex, $argument);

Si no establece todos los argumentos antes de iniciar el kernel, el kernel fallará.

Antes de lanzar nuestro kernel, necesitamos calcular el ’tamaño de trabajo global’ y el ’tamaño de trabajo local’.

el tamaño de trabajo global es la cantidad total de subprocesos que se iniciarán en su GPU. El tamaño de trabajo local es el número de subprocesos dentro de cada bloque de subprocesos. El tamaño de trabajo local se puede omitir si el kernel no necesita ningún requisito especial. Pero si se da el tamaño del trabajo local, el tamaño del trabajo global tiene que ser un múltiplo del tamaño del trabajo local.

Los tamaños de trabajo pueden ser unidimensionales, bidimensionales o tridimensionales. La elección de cuántas dimensiones desea depende totalmente de usted y puede elegir la que mejor se adapte a su algoritmo.

Ahora que hemos decidido el tamaño de nuestro trabajo, podemos llamar al kernel.

Event clevent;
err = Cl.EnqueueNDRangeKernel(_queue, _kernel, $dimensions, null, $globalWorkSize, $localWorkSize, 0, null, out clevent);

Las $dimensiones definen nuestro número deseado de dimensiones, $globalWorkSize es una matriz de $dimensiones de tamaño con el tamaño de trabajo global y lo mismo para $localWorkSize. El último argumento le da un objeto que representa su núcleo actualmente ejecutado.
