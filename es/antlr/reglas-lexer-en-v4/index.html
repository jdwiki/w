Reglas de prioridad #

Varias reglas de lexer pueden coincidir con el mismo texto de entrada. En ese caso, el tipo de token se elegirá de la siguiente manera:

  • Primero, seleccione la regla lexer que coincida con la entrada más larga
  • Si el texto coincide con un token definido implícitamente (como '{'), use la regla implícita
  • Si varias reglas lexer coinciden con la misma longitud de entrada, elija la primera, según el orden de definición

La siguiente gramática combinada:

grammar LexerPriorityRulesExample;

// Parser rules

randomParserRule: 'foo'; // Implicitly declared token type
    
// Lexer rules
    
BAR: 'bar';
IDENTIFIER: [A-Za-z]+;
BAZ: 'baz';

WS: [ \t\r\n]+ -> skip;

Dada la siguiente entrada:

aaa foo bar baz barz

Producirá la siguiente secuencia de tokens del lexer:

IDENTIFIER 'foo' BAR IDENTIFIER IDENTIFIER
  • aaa es de tipo IDENTIFICADOR

Solo la regla IDENTIFICADOR puede coincidir con este token, no hay ambigüedad.

  • foo es del tipo 'foo'

La regla del analizador randomParserRule introduce el tipo de token 'foo' implícito, que tiene prioridad sobre la regla IDENTIFIER.

  • bar es del tipo BAR

Este texto coincide con la regla BAR, que se define antes de la regla IDENTIFICADOR y, por lo tanto, tiene prioridad.

  • baz es de tipo IDENTIFICADOR

Este texto coincide con la regla BAZ, pero también coincide con la regla IDENTIFIER. Este último se elige como se define antes de BAR.

Dada la gramática, BAZ nunca podrá coincidir, ya que la regla IDENTIFICADOR ya cubre todo lo que BAZ puede coincidir.

  • barz es de tipo IDENTIFICADOR

La regla BAR puede coincidir con los primeros 3 caracteres de esta cadena (bar), pero la regla IDENTIFIER coincidirá con 4 caracteres. Como ‘IDENTIFICADOR’ coincide con una subcadena más larga, se elige sobre ‘BAR’.

Como regla general, las reglas específicas deben definirse antes de las reglas más genéricas. Si una regla solo puede coincidir con una entrada que ya está cubierta por una regla definida previamente, nunca se utilizará.

Las reglas implícitamente definidas como 'foo' actúan como si estuvieran definidas antes de todas las demás reglas de lexer.

Reglas implícitas de lexer #

Cuando se utilizan tokens como '{' en una regla parser, se creará una regla implícita de lexer para ellos, a menos que exista una regla explícita.

En otras palabras, si tiene una regla lexer:

OPEN_BRACE: '{';

Entonces ambas reglas del analizador son equivalentes:

parserRule: '{';
parserRule: OPEN_BRACE;

Pero si la regla lexer OPEN_BRACE no está definida, se creará una regla anónima implícita. En ese caso, la regla implícita se definirá como si se hubiera definido antes de las otras reglas: tendrá una prioridad más alta que otras reglas.

Acciones y predicados semánticos #

Una acción lexer es un bloque de código arbitrario en el idioma de destino rodeado por {…}, que se ejecuta durante la coincidencia:

IDENTIFIER: [A-Z]+ { log("matched rule"); };

Un predicado semántico es un bloque de código arbitrario en el idioma de destino rodeado por {…}?, que se evalúa como un valor booleano. Si el valor devuelto es falso, se omite la regla lexer.

IDENTIFIER: [A-Z]+ { identifierIsValid() }?;

Los predicados semánticos deben definirse al final de la regla siempre que sea posible por motivos de rendimiento.

Reglas simples #

Las reglas de Lexer definen los tipos de token. Su nombre debe comenzar con una letra mayúscula para distinguirlos de las reglas del analizador.

INTEGER: [0-9]+;
IDENTIFIER: [a-zA-Z_] [a-zA-Z_0-9]*;

OPEN_PAREN: '(';
CLOSE_PAREN: ')';

Sintaxis básica:

Sintaxis Sentido
A Haga coincidir la regla lexer o el fragmento denominado A
AB Coincide con A seguido de B
<código>(A|B)</código> Coincide con A o B
'texto' Coincidencia literal “texto”
‘¿A?’ Coincide con A cero o una vez
A* Hacer coincidir A cero o más veces
A+ Coincide con A una o más veces
[A-Z0-9] Haga coincidir un carácter en los rangos definidos (en este ejemplo entre A-Z o 0-9)
'a'..'z' Sintaxis alternativa para un rango de caracteres
~[A-Z] Negación de un rango: coincide con cualquier carácter individual no en el rango
. Coincide con cualquier carácter individual

Fragmentos #

Los fragmentos son partes reutilizables de las reglas de lexer que no pueden coincidir por sí mismas; es necesario hacer referencia a ellas desde una regla de lexer.

INTEGER: DIGIT+
       | '0' [Xx] HEX_DIGIT+
       ;

fragment DIGIT: [0-9];
fragment HEX_DIGIT: [0-9A-Fa-f];

Comandos Lexer #

Una regla lexer puede tener comandos asociados:

WHITESPACE: [ \r\n] -> skip;

Los comandos se definen después de un -> al final de la regla.

  • saltar: Omite el texto coincidente, no se emitirá ningún token
  • channel(n): Emite el token en un canal diferente
  • type(n): Cambia el tipo de token emitido
  • mode(n), pushMode(n), popMode, more: Controla los modos lexer
