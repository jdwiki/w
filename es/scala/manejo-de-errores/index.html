O #

Diferentes tipos de datos para error/éxito

def getPersonFromWebService(url: String): Either[String, Person] = {
    
    val response = webServiceClient.get(url)

    response.webService.status match {
        case 200 => {
            val person = parsePerson(response)
            if(!isValid(person)) Left("Validation failed")
            else Right(person)
        }

        case _ => Left(s"Request failed with error code $response.status")
    }
}

Coincidencia de patrones en cualquier valor

getPersonFromWebService("http://some-webservice.com/person") match {
    case Left(errorMessage) => println(errorMessage)
    case Right(person) => println(person.surname)
}

Convertir cualquiera de los valores en opción

val maybePerson: Option[Person] = getPersonFromWebService("http://some-webservice.com/person").right.toOption

Probar #

Usando Try con map, getOrElse y flatMap:

import scala.util.Try

val i = Try("123".toInt)     // Success(123)
i.map(_ + 1).getOrElse(321)  // 124

val j = Try("abc".toInt)     // Failure(java.lang.NumberFormatException)
j.map(_ + 1).getOrElse(321)  // 321

Try("123".toInt) flatMap { i =>
  Try("234".toInt)
    .map(_ + i)
}                            // Success(357)

Usando Try con coincidencia de patrones:

Try(parsePerson("John Doe")) match {
    case Success(person) => println(person.surname)
    case Failure(ex) => // Handle error ...
}

Opción #

Se desaconseja encarecidamente el uso de valores null, a menos que interactúe con código Java heredado que espera null. En su lugar, se debe usar Option cuando el resultado de una función puede ser algo (Some) o nada (Ninguno).

Un bloque try-catch es más apropiado para el manejo de errores, pero si la función puede devolver legítimamente nada, Option es apropiado para usar y simple.

Una Opción[T] puede ser Algún(valor) (contiene un valor de tipo T) o Ninguno:

def findPerson(name: String): Option[Person]

Si no se encuentra ninguna persona, se puede devolver Ninguno. De lo contrario, se devuelve un objeto de tipo Algunos que contiene un objeto Persona. Lo que sigue son formas de manejar un objeto de tipo Opción.

La coincidencia de patrones #

findPerson(personName) match {
    case Some(person) => println(person.surname)
    case None => println(s"No person found with name $personName")
}

Uso de mapa y getOrElse #

val name = findPerson(personName).map(_.firstName).getOrElse("Unknown")
println(name) // Prints either the name of the found person or "Unknown"

Usando doblar #

val name = findPerson(personName).fold("Unknown")(_.firstName)
// equivalent to the map getOrElse example above.

Convirtiendo a Java #

Si necesita convertir un tipo de Opción en un tipo de Java que admite nulos para la interoperabilidad:

val s: Option[String] = Option("hello")
s.orNull           // "hello": String
s.getOrElse(null)  // "hello": String

val n: Option[Int] = Option(42)
n.orNull           // compilation failure (Cannot prove that Null <:< Int.)
n.getOrElse(null)  // 42

Manejo de errores que se originan en futuros #

Cuando se lanza una excepción desde dentro de un Futuro, puede (debe) usar recuperar para manejarla.

Por ejemplo,

def runFuture: Future = Future { throw new FairlyStupidException }

val itWillBeAwesome: Future = runFuture

… lanzará una Excepción desde dentro del Futuro. Pero dado que podemos predecir que una Excepción de tipo FairlyStupidException con una alta probabilidad, podemos manejar este caso específicamente de una manera elegante:

val itWillBeAwesomeOrIllRecover = runFuture recover { 
    case stupid: FairlyStupidException => 
         BadRequest("Another stupid exception!") 
}

Como puede ver, el método dado a recover es una PartialFunction sobre el dominio de todos los Throwable, por lo que puede manejar solo algunos tipos y luego dejar que el resto vaya al éter del manejo de excepciones en niveles más altos en la pila Futuro.

Tenga en cuenta que esto es similar a ejecutar el siguiente código en un contexto que no es Futuro:

def runNotFuture: Unit = throw new FairlyStupidException

try {
    runNotFuture
} catch {
    case e: FairlyStupidException => BadRequest("Another stupid exception!")
}

Es realmente importante manejar las excepciones generadas dentro de Futures porque la mayor parte del tiempo son más insidiosas. Por lo general, no se ponen en su cara porque se ejecutan en un contexto de ejecución y un subproceso diferentes y, por lo tanto, no le solicitan que los corrija cuando suceden, especialmente si no nota nada en los registros o el comportamiento del solicitud.

Uso de cláusulas try-catch #

Además de las construcciones funcionales como Try, Option y Either para el manejo de errores, Scala también admite una sintaxis similar a la de Java, utilizando una cláusula try-catch (con un posible bloque finalmente también). La cláusula catch es una coincidencia de patrón:

try { 
  // ... might throw exception
} catch {
  case ioe: IOException => ... // more specific cases first
  case e: Exception => ...
  // uncaught types will be thrown
} finally {
  // ...
}

Convierta las excepciones en tipos de opciones u opciones #

Para convertir excepciones en tipos Cualquiera u Opción, puede usar métodos que se proporcionan en scala.util.control.Exception

import scala.util.control.Exception._

val plain = "71a"
val optionInt: Option[Int] = catching(classOf[java.lang.NumberFormatException]) opt { plain.toInt }
val eitherInt = Either[Throwable, Int] = catching(classOf[java.lang.NumberFormatException]) either { plain.toInt }
