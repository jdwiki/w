Como val son semánticamente estáticos, se inicializan “en el lugar” dondequiera que aparezcan en el código. Esto puede producir un comportamiento sorprendente e indeseable cuando se usa en clases y rasgos abstractos.

Por ejemplo, digamos que nos gustaría crear un rasgo llamado PlusOne que defina una operación de incremento en un Int envuelto. Dado que los Ints son inmutables, el valor más uno se conoce en la inicialización y nunca se cambiará después, por lo que semánticamente es un val. Sin embargo, definirlo de esta manera producirá un resultado inesperado.

trait PlusOne {
    val i:Int

    val incr = i + 1
}

class IntWrapper(val i: Int) extends PlusOne

No importa con qué valor i construya IntWrapper, llamar a .incr en el objeto devuelto siempre devolverá 1. Esto se debe a que el valor incr se inicializa en el rasgo, antes de la clase de extensión, y en ese momento i solo tiene el valor predeterminado de 0. (En otras condiciones, podría completarse con Nil, null o un valor predeterminado similar).

La regla general, entonces, es evitar usar val en cualquier valor que dependa de un campo abstracto. En su lugar, utilice lazy val, que no evalúa hasta que se necesita, o def, que evalúa cada vez que se llama. Sin embargo, tenga en cuenta que si el lazy val se ve obligado a evaluar mediante un val antes de que se complete la inicialización, se producirá el mismo error.

Se puede encontrar un violín (escrito en Scala-Js, pero se aplica el mismo comportamiento) aquí.

Var, Val y Def #

estaba #

Una var es una variable de referencia, similar a las variables en lenguajes como Java. Se pueden asignar libremente diferentes objetos a una var, siempre que el objeto dado tenga el mismo tipo con el que se declaró la var:

scala> var x = 1
x: Int = 1

scala> x = 2
x: Int = 2

scala> x = "foo bar"
<console>:12: error: type mismatch;
 found   : String("foo bar")
 required: Int
       x = "foo bar"
       ^

Tenga en cuenta que en el ejemplo anterior, el compilador infirió el tipo de var dada la primera asignación de valor.

valor #

Un val es una referencia constante. Por lo tanto, no se puede asignar un nuevo objeto a un val que ya ha sido asignado.

scala> val y = 1
y: Int = 1

scala> y = 2
<console>:12: error: reassignment to val
       y = 2
         ^

Sin embargo, el objeto al que apunta un val no es constante. Ese objeto puede ser modificado:

scala> val arr = new Array[Int](2)
arr: Array[Int] = Array(0, 0)

scala> arr(0) = 1

scala> arr
res1: Array[Int] = Array(1, 0)

definición #

Un def define un método. No se puede reasignar un método.

scala> def z = 1
z: Int

scala> z = 2
<console>:12: error: value z_= is not a member of object $iw
       z = 2
       ^

En los ejemplos anteriores, val y y def z devuelven el mismo valor. Sin embargo, un def se evalúa cuando se llama, mientras que un val o var se evalúa cuando se asigna. Esto puede resultar en un comportamiento diferente cuando la definición tiene efectos secundarios:

scala> val a = {println("Hi"); 1}
Hi
a: Int = 1

scala> def b = {println("Hi"); 1}
b: Int

scala> a + 1
res2: Int = 2

scala> b + 1
Hi
res3: Int = 2

Funciones #

Como las funciones son valores, se pueden asignar a val/var/defs. Todo lo demás funciona de la misma manera que arriba:

scala> val x = (x: Int) => s"value=$x"
x: Int => String = <function1>

scala> var y = (x: Int) => s"value=$x"
y: Int => String = <function1>

scala> def z = (x: Int) => s"value=$x"
z: Int => String

scala> x(1)
res0: String = value=1

scala> y(2)
res1: String = value=2

scala> z(3)
res2: String = value=3

Valor perezoso #

lazy val es una característica del lenguaje en la que la inicialización de un val se retrasa hasta que se accede a él por primera vez. Después de ese punto, actúa como un val regular.

Para usarlo, agregue la palabra clave lazy antes de val. Por ejemplo, usando REPL:

scala> lazy val foo = {
     |   println("Initializing")
     |   "my foo value"
     | }
foo: String = <lazy>

scala> val bar = {
     |   println("Initializing bar")
     |   "my bar value"
     | }
Initializing bar
bar: String = my bar value

scala> foo
Initializing
res3: String = my foo value

scala> bar
res4: String = my bar value

scala> foo
res5: String = my foo value

Este ejemplo demuestra el orden de ejecución. Cuando se declara lazy val, todo lo que se guarda en el valor foo es una llamada de función diferida que aún no se ha evaluado. Cuando se establece el val regular, vemos que se ejecuta la llamada println y el valor se asigna a bar. Cuando evaluamos foo la primera vez, vemos que se ejecuta println, pero no cuando se evalúa la segunda vez. De manera similar, cuando se evalúa bar, no vemos la ejecución de println, solo cuando se declara.

Cuándo usar ‘perezoso’ #

  1. La inicialización es computacionalmente costosa y el uso de val es raro.

    lazy val tiresomeValue = {(1 to 1000000).filter(x => x % 113 == 0).sum}
if (scala.util.Random.nextInt > 1000) {
  println(tiresomeValue)
}

tiresomeValue tarda mucho tiempo en calcularse y no siempre se usa. Hacerlo un lazy val ahorra cálculos innecesarios.

  1. Resolviendo dependencias cíclicas

Veamos un ejemplo con dos objetos que deben declararse al mismo tiempo durante la instanciación:

   object comicBook {
     def main(args:Array[String]): Unit = {
       gotham.hero.talk()
       gotham.villain.talk()
     }
   }

   class Superhero(val name: String) {
     lazy val toLockUp = gotham.villain
     def talk(): Unit = {
       println(s"I won't let you win ${toLockUp.name}!")
     }
   }

   class Supervillain(val name: String) {
     lazy val toKill = gotham.hero
     def talk(): Unit = {
       println(s"Let me loosen up Gotham a little bit ${toKill.name}!")
     }
   }

   object gotham {
     val hero: Superhero = new Superhero("Batman")
     val villain: Supervillain = new Supervillain("Joker")
   }

Sin la palabra clave perezoso, los objetos respectivos no pueden ser miembros de un objeto. La ejecución de dicho programa daría como resultado una java.lang.NullPointerException. Al usar lazy, la referencia se puede asignar antes de que se inicialice, sin temor a tener un valor sin inicializar.

Sobrecarga Def #

Puede sobrecargar un def si la firma es diferente:

def printValue(x: Int) {
  println(s"My value is an integer equal to $x")
}

def printValue(x: String) {
  println(s"My value is a string equal to '$x'")
}

printValue(1)  // prints "My value is an integer equal to 1"
printValue("1") // prints "My value is a string equal to '1'"

Esto funciona igual ya sea dentro de clases, rasgos, objetos o no.

Parámetros con nombre #

Al invocar un def, los parámetros pueden asignarse explícitamente por nombre. Si lo hace, significa que no es necesario ordenarlos correctamente. Por ejemplo, defina printUs() como:

// print out the three arguments in order.
def printUs(one: String, two: String, three: String) = 
   println(s"$one, $two, $three")

Ahora se puede llamar de estas formas (entre otras):

printUs("one", "two", "three") 
printUs(one="one", two="two", three="three")
printUs("one", two="two", three="three")
printUs(three="three", one="one", two="two")

Esto da como resultado que se imprima “uno, dos, tres” en todos los casos.

Si no se nombran todos los argumentos, los primeros argumentos se emparejan por orden. Ningún argumento posicional (sin nombre) puede seguir a uno con nombre:

printUs("one", two="two", three="three") // prints 'one, two, three'
printUs(two="two", three="three", "one") // fails to compile: 'positional after named argument'
