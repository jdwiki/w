Covarianza #

El símbolo + marca un parámetro de tipo como covariante; aquí decimos que “Productor es covariante en A”:

trait Producer[+A] {
  def produce: A
}

Un parámetro de tipo covariante se puede considerar como un tipo de “salida”. Marcar A como covariante afirma que Productor[X] <: Productor[Y] siempre que X <: Y. Por ejemplo, un Productor[Gato] es un Productor[Animal] válido, ya que todos los gatos producidos también son animales válidos.

Un parámetro de tipo covariante no puede aparecer en la posición contravariante (entrada). El siguiente ejemplo no se compilará ya que afirmamos que Co[Cat] <: Co[Animal], pero Co[Cat] tiene def handle(a: Cat): Unit que no puede manejar ningún Animal como lo requiere Co[Animal]!

trait Co[+A] {
  def produce: A
  def handle(a: A): Unit
}

Un enfoque para lidiar con esta restricción es usar parámetros de tipo limitados por el parámetro de tipo covariante. En el siguiente ejemplo, sabemos que B es un supertipo de A. Por lo tanto, dada Opción[X] <: Opción[Y] para X <: Y, sabemos que Opción[X] def getOrElse[B >: X](b: => B): B puede aceptar cualquier supertipo de X, que incluye los supertipos de Y según lo requiere Option[Y]:

trait Option[+A] {
  def getOrElse[B >: A](b: => B): B
}

Contravarianza #

El símbolo - marca un parámetro de tipo como contravariante - aquí decimos que “Handler es contravariante en A”:

trait Handler[-A] {
  def handle(a: A): Unit
}

Un parámetro de tipo contravariante se puede considerar como un tipo de “entrada”. Marcar A como contravariante afirma que Manejador[X] <: Manejador[Y] siempre que X >: Y. Por ejemplo, un Handler[Animal] es un Handler[Cat] válido, ya que un Handler[Animal] también debe manejar gatos.

Un parámetro de tipo contravariante no puede aparecer en la posición covariante (salida). El siguiente ejemplo no se compilará ya que estamos afirmando que un Contra[Animal] <: Contra[Cat], sin embargo, un Contra[Animal] tiene def produce: Animal que no garantiza que produzca gatos como lo requiere Contra[Gato]!

trait Contra[-A] {
   def handle(a: A): Unit
   def produce: A
}

Sin embargo, tenga cuidado: a los efectos de sobrecargar la resolución, la contravarianza también invierte de manera contradictoria la especificidad de un tipo en el parámetro de tipo contravariante: Handler[Animal] se considera “más específico” que Handler[Cat].

Como no es posible sobrecargar métodos en parámetros de tipo, este comportamiento generalmente solo se vuelve problemático cuando se resuelven argumentos implícitos. En el siguiente ejemplo, ofCat nunca se usará, ya que el tipo de retorno de ofAnimal es más específico:

implicit def ofAnimal: Handler[Animal] = ???
implicit def ofCat: Handler[Cat] = ???

implicitly[Handler[Cat]].handle(new Cat)

Este comportamiento está actualmente programado para cambio en dotty, y es por eso que (como ejemplo) scala.math.Ordering es invariable en su parámetro de tipo T. Una solución es hacer que su clase de tipos sea invariable y parametrizar la definición implícita en caso de que desee que se aplique a las subclases de un tipo determinado:

trait Person
object Person {
  implicit def ordering[A <: Person]: Ordering[A] = ???
}

Invariancia #

De forma predeterminada, todos los parámetros de tipo son invariantes: dado el rasgo A[B], decimos que “A es invariante en B”. Esto significa que dadas dos parametrizaciones ‘A[Gato]’ y ‘A[Animal]’, afirmamos que no hay una relación de sub/superclase entre estos dos tipos; no se sostiene que ‘A[Gato] <: A[Animal]’ ni que A[Gato] >: A[Animal] independientemente de la relación entre Gato y Animal.

Las anotaciones de varianza nos brindan un medio para declarar dicha relación e imponen reglas sobre el uso de parámetros de tipo para que la relación siga siendo válida.

Covarianza en un rasgo invariante #

También hay una manera de hacer que un solo método acepte un argumento covariante, en lugar de tener todo el rasgo covariante. Esto puede ser necesario porque le gustaría usar ‘T’ en una posición contravariante, pero aún así tenerla covariante.

trait LocalVariance[T]{
  /// ??? throws a NotImplementedError
  def produce: T = ???
  // the implicit evidence provided by the compiler confirms that S is a
  // subtype of T.
  def handle[S](s: S)(implicit evidence: S <:< T) = {
    // and we can use the evidence to convert s into t.
    val t: T = evidence(s)
    ???
  }
}

trait A {}
trait B extends A {}

object Test {
  val lv = new LocalVariance[A] {}

  // now we can pass a B instead of an A.
  lv.handle(new B {})
}
