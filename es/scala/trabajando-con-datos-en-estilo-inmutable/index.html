Los nombres de valores y variables deben estar en minúsculas camel #

Los nombres constantes deben estar en mayúsculas y minúsculas. Es decir, si el miembro es final, inmutable y pertenece a un objeto de paquete o a un objeto, se puede considerar una constante

Los nombres de métodos, valores y variables deben estar en minúsculas camel

Fuente: http://docs.scala-lang.org/style/naming-conventions.html

Esta compilación:

val (a,b) = (1,2)
// a: Int = 1
// b: Int = 2

pero esto no:

val (A,B) = (1,2)
// error: not found: value A
// error: not found: value B

No es solo val contra var #

val y var #

scala> val a = 123
a: Int = 123

scala> a = 456
<console>:8: error: reassignment to val
       a = 456

scala> var b = 123
b: Int = 123

scala> b = 321
b: Int = 321
  • Las referencias val no se pueden cambiar: como una variable final en Java, una vez que se ha inicializado no se puede cambiar
  • Las referencias var son reasignables como una declaración de variable simple en Java

Colecciones inmutables y mutables #

  val mut = scala.collection.mutable.Map.empty[String, Int]
  mut += ("123" -> 123)
  mut += ("456" -> 456)
  mut += ("789" -> 789)

  val imm = scala.collection.immutable.Map.empty[String, Int]
  imm + ("123" -> 123)
  imm + ("456" -> 456)
  imm + ("789" -> 789)

  scala> mut
    Map(123 -> 123, 456 -> 456, 789 -> 789)

  scala> imm
    Map()

scala> imm + ("123" -> 123) + ("456" -> 456) + ("789" -> 789)
    Map(123 -> 123, 456 -> 456, 789 -> 789)

La biblioteca estándar de Scala ofrece estructuras de datos tanto inmutables como mutables, no la referencia a la misma. Cada vez que se “modifica” una estructura de datos inmutable, se produce una nueva instancia en lugar de modificar la colección original en el lugar. Cada instancia de la colección puede compartir una estructura significativa con otra instancia.

Colección mutable e inmutable (Documentación oficial de Scala)

¡Pero no puedo usar la inmutabilidad en este caso! #

Tomemos como ejemplo una función que toma 2 Map y devuelve un Map que contiene todos los elementos en ma y mb:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int]

Un primer intento podría ser iterar a través de los elementos de uno de los mapas usando for ((k, v) <- map) y de alguna manera devolver el mapa fusionado.

def merge2Maps(ma: ..., mb: ...): Map[String, Int] = {

  for ((k, v) <- mb) {
    ???
  }

}

Este primer movimiento agrega inmediatamente una restricción: una mutación fuera de for ahora es necesaria. Esto es más claro cuando se elimina el azúcar del for:

// this:
for ((k, v) <- map) { ??? }

// is equivalent to:
map.foreach { case (k, v) => ??? }

“¿Por qué tenemos que mutar?” #

foreach se basa en los efectos secundarios. Cada vez que queremos que suceda algo dentro de un foreach necesitamos “efecto secundario de algo”, en este caso podríamos mutar una variable var resultado o podemos usar una estructura de datos mutable.

Creando y llenando el mapa resultado #

Supongamos que ma y mb son scala.collection.immutable.Map, podríamos crear el mapa resultado a partir de ma:

val result = mutable.Map() ++ ma

Luego iterar a través de mb agregando sus elementos y si la clave del elemento actual en ma ya existe, anulémosla con la mb.

mb.foreach { case (k, v) => result += (k -> v) }

Implementación mutable #

Hasta ahora todo bien, “tuvimos que usar colecciones mutables” y una implementación correcta podría ser:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  val result = scala.collection.mutable.Map() ++ ma
  mb.foreach { case (k, v) => result += (k -> v) }
  result.toMap // to get back an immutable Map
}

Como se esperaba:

scala> merge2Maps(Map("a" -> 11, "b" -> 12), Map("b" -> 22, "c" -> 23))
  Map(a -> 11, b -> 22, c -> 23)

Plegables al rescate #

¿Cómo podemos deshacernos de foreach en este escenario? Si todo lo que tenemos que hacer es básicamente iterar sobre los elementos de la colección y aplicar una función mientras acumulamos el resultado, la opción podría estar usando .foldLeft:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  mb.foldLeft(ma) { case (result, (k, v)) => result + (k -> v) }
  // or more concisely mb.foldLeft(ma) { _ + _ }
}

En este caso nuestro “resultado” es el valor acumulado a partir de ma, el cero del .foldLeft.

Resultado intermedio #

Obviamente, esta solución inmutable está produciendo y destruyendo muchas instancias de ‘Mapa’ mientras se pliega, pero vale la pena mencionar que esas instancias no son un clon completo del ‘Mapa’ acumulado, sino que comparten una estructura significativa (datos) con la instancia existente.

Razonabilidad más fácil #

Es más fácil razonar sobre la semántica si es más declarativo como el enfoque .foldLeft. El uso de estructuras de datos inmutables podría ayudar a que nuestra implementación sea más fácil de razonar.
