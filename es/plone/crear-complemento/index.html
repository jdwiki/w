Una historia sobre la creación de un complemento de Plone #

Preámbulo #

TL;TR: Para fines ilustrativos, las líneas de este documento que comienzan con ‘TL;TR:’ son seguidos por una línea de comando que hace referencia a los scripts de ayuda de Plone del huevo ‘adi.devgen’. Si ejecuta el comando dado después de ‘TL;TR:’, se crear todos los archivos explicados en el siguiente capítulo. Hay muchas herramientas de secuencias de comandos de ayuda alternativas, sin embargo, en caso de que desee use esta herramienta, puede instalarla rápidamente así en la línea de comandos:

pip install adi.devgen

Estructura de un complemento mínimo de Plone #

TL;TR: devgen addBase mynamespace.myaddon

Al igual que Plone consta de más de 200 Python-eggs, un Plone-Add-On también es un Python-egg y un huevo es un módulo de Python con espacio de nombres. Cualquier huevo estará disponible. al intérprete Python de ZOPE. Para hacer un huevo instalable en un sitio Plone, también necesita un llamado ‘perfil’. A continuación se explicarán estos términos.

Python-módulo #

Un módulo de Python es un directorio que contiene al menos un archivo que debe ser llamado __init__.py. El nombre es una bandera para que el intérprete de Python reconocer este directorio como un módulo, lo que significa que su ruta se puede importar de otros Python-scripts, p. suponiendo que tenemos un directorio llamado ‘myaddon’, que contiene un archivo llamado ‘myfile’ y ese archivo contiene una definición llamada ‘myDefinition’, puede importar la definición de otro archivo de Python como este:

from myaddon.myfile import myDefinition

huevo de pitón #

Un huevo de Python es un módulo de Python con espacio de nombres, un espacio de nombres puede ser cualquier nombre, pero es mejor no tomarlo ya, en caso de que quiera compartir su complemento con el mundo.

La razón principal del espacio de nombres es excluir la posibilidad de que dos o más módulos podrían tener el mismo nombre (en este ejemplo llamado ‘myaddon’), ya que daría lugar a conflictos cuando un script de Python intenta importar la ruta, siendo ya no es único.

Aquí hay una descripción general de la estructura de directorios y los archivos involucrados, antes de que continuar explicando sus propósitos en detalle más adelante:

mynamespace.myaddon
    setup.py
    mynamespace
        __init__.py
        myaddon
            __init__.py
            myfile.py

Para asignar un espacio de nombres a un complemento, primero cree un directorio llamado ‘mynamespace.myaddon’.

Este directorio debe contener el archivo ‘setup.py’, hace que el huevo sea registrable al intérprete Python de la instancia ZOPE, esto se explica con más detalle en el próximo capítulo [POR HACER].

Luego, en ese directorio, cree otro directorio con el nombre de su espacio de nombres: ‘mynamespace’ Ahora, hacemos de este directorio una ruta de módulo importable, agregando el archivo con nombre de palabra clave __init__.py en él. También debe contener lo siguiente líneas, que hacen que suceda la magia del espacio de nombres, explicada en el último párrafo:

[TODO: namespace-magic-lines]

Uf casi listo, por ultimo en este nuevo directorio ponemos el modulo del anterior capítulo: El directorio ‘myaddon’, que contiene ‘init.py’ y ‘myfile.py’ con la ‘miDefinición’.

Ahora tenemos un huevo registrable para la instancia ZOPE y podríamos hacer referencia a su métodos de cualquier otro huevo registrado, utilizando esta notación de ruta:

from mynamespace.myaddon.myfile import myDefinition

Es posible que vea la magia, los dos primeros directorios se omiten, no:

from mynamespace.myaddon.mynamespace.myaddon.myfile import myDefinition

Hacer un complemento instalable dentro de un sitio de Plone #

Un ‘perfil’ hará que el complemento se muestre en el panel de control de complementos de un sitio, por lo que un administrador puede (des) activarlo allí para el sitio.

Además, como un sitio Plone es siempre un elemento secundario de la instancia ZOPE y un La instancia ZOPE puede contener varios sitios, es posible que no queramos tener componentes involuntarios de nuestro complemento instalado en otros sitios de Plone, por lo tanto, los vinculamos a un ‘perfil’.

TL;TR: devgen addProfile mynamespace.myaddon

Esto nos lleva a las partes realmente interesantes, ZOPE-Component-Architecture, controlado por los archivos que terminan en ‘.zcml’, que significa ‘ZOPE-Component- Lenguaje de marcado’. Con él puede registrar un perfil, vincular vistas a interfaces y mucho más. De hecho, merece un capítulo principal propio y un subcapítulo para cada uno de los directorios llamados “perfil”, “máscaras”, “navegador” y “contenido”.

Descripción general de la estructura de archivos #

esqueleto mínimo #

mynamespace.myaddon     | The container-directory of the add-on.

    setup.py            | Register this directory to the Python-interpreter of
                        | the ZOPE-instance.

    mynamespace         | The namespace-directory of the add-on.

        __init__.py     | Makes this directory a Python-module and contains
                        | the lines for the namespace-magic.

        myaddon         | The base-directory of the add-on.

            __init__.py | Empty file, make this directory a Python-module.

            myfile.py   | Example-file: If it contained a method called
                        | 'myMethod', it would be importable of any other
                        | registered egg like this:
                        |    from mynamespace.myaddon.myfile import myMethod

Componentes opcionales #

El directorio ‘mynamespace.myaddon/mynamespace/myaddon’ puede contener:

configure.zcml  | Register all following files and directories, here.

setuphandlers.py | Make an addon available to ZOPE-instance.

profiles        | Make an addon installable to Plone-site and generally
                | hold GenericSetup-xml-files.

skins           | The "old-school"-way, where resources like templates,
                | stylesheets, etc., live.

browser         | The "new-school"-way, where resources live.

content         | Holds the Python-files for defining custom content-types.

subscriber.py   | Subscribers listen to events, describe here what to do
                | when an event happens, e.g.: "An item has been added to the
                | site, notify admin about it via e-mail-notifica."
