Representación Arquitectónica (Vista Lógica) La solución se divide en dos componentes. Uno es la aplicación del lado del cliente que se ejecutaría en el dispositivo móvil y el componente del lado del servidor que se ejecuta en el entorno de Azure.

Aplicación móvil (cliente Node.js) Node.js is light weight responsive user interface. A mobile application would simply be the, written in the native language or using a hybrid solution like Cordova (aka PhoneGap). This user interface would get data and perform business logic via calls to a server side API. Si la aplicación móvil requiere partes de la lógica empresarial, tendría que reescribirse en Objective-C para iOS y Java para Android. Si bien Cordova ofrece una solución para la reutilización de algunos códigos, no puede aprovechar Node o su ecosistema directamente.

Componente del lado del servidor Servidor Node.js Node es un tiempo de ejecución de servidor web de JavaScript. Este actúa como servidor web para la aplicación móvil. Utiliza JSON como formato de intercambio de datos, JSON, puede ser analizado de forma nativa por ambos extremos. Node difiere de la mayoría de los tiempos de ejecución de aplicaciones web en la forma en que maneja la concurrencia. En lugar de utilizar subprocesos para lograr la concurrencia, Node se basa en un bucle controlado por eventos que se ejecuta en un solo proceso. Node admite un modelo asíncrono (sin bloqueo). API web Esta es una solución API REST que se ejecuta en Azure como API web. Se ejecuta sobre Document DB y actúa como capa protectora para Document DB. Herramientas ASP.NET y middleware de seguridad de la interfaz web abierta de Microsoft para componentes .NET (OWIN) que simplifican la protección de la API web. El punto final está protegido mediante tokens de acceso Azure AD OAuth Bearer 2.0.   3. Proceso Técnico

3.1. Autenticación La autenticación se reduce a pedirle a la persona que llama, cuando envía un mensaje a un servidor, que incluya algún tipo de credencial que pueda verificar su identidad o recuperar sus atributos. Luego, el servidor usa esa información para la autorización, determinando si se debe otorgar acceso y en qué términos. Los recursos a menudo descargan la mayoría de las funciones de autenticación a un proveedor de servicios externo, comúnmente conocido como autoridad o proveedor de identidad. Estos proveedores se encargan de las tareas más exigentes, como la incorporación de usuarios, la asignación de credenciales, el manejo de los flujos del ciclo de vida (como la recuperación de contraseñas), el suministro de una interfaz de usuario para la autenticación de usuarios, la verificación de credenciales en múltiples protocolos, la administración de múltiples factores de autenticación, la detección de fraudes y más. Con esas funciones fuera del camino, la única tarea de autenticación que queda es verificar que la autenticación tuvo éxito en la autoridad elegida. Esto generalmente implica examinar un token de seguridad, un fragmento de datos emitido por una autoridad a una persona que llama después de una autenticación exitosa. Los tokens de seguridad generalmente se elaboran de acuerdo con un formato específico, están firmados digitalmente por una clave que identificará inequívocamente a la autoridad emisora ​​y contienen algunos datos que vinculan de manera única el token con el recurso de destino. Cuando el recurso recibe una solicitud, busca un token que lo acompañe. Si encuentra uno que cumpla con los atributos de validación requeridos, la persona que llama se autentica. A ese nivel de detalle, el patrón es tan genérico que puede describir muchos enfoques diferentes para la autenticación. Lo aplicaré a este escenario asignando roles de alto nivel a entidades concretas. El recurso El recurso será el proyecto ASP.NET Web API 2 que necesito proteger. Puede aplicar requisitos de autenticación con mayor granularidad.

3.2. API web personalizada y punto final seguro La Autoridad configurará la API web para descargar sus necesidades de autenticación a Windows Azure AD, una plataforma como servicio (PaaS) disponible para todos los suscriptores de Windows Azure. Windows Azure AD está diseñado específicamente para admitir cargas de trabajo de aplicaciones basadas en la nube. El servicio almacena información sobre los usuarios (atributos y credenciales) y la estructura organizativa. Puede sincronizar datos con Windows Server Active Directory, siempre que se conecte exclusivamente en la nube. 3.3. Formato de token de OAuth El proceso de validación de OAuth 2.0 no exige ningún formato de token en particular, pero el formato JSON Web Token (JWT) (bit.ly/14EhlE8) para escenarios REST se ha convertido en el estándar de facto. Tanto los componentes de Windows Azure AD como los de Microsoft OWIN admiten el formato JWT en los flujos de OAuth 2.0. El middleware se encarga de la mecánica de adquisición y validación de JWT, y el formato del token es transparente para el código de la aplicación. 3.4. Módulo Node.js con biblioteca ADAL Los módulos de Node.js son una interfaz móvil que contiene componentes de interfaz de usuario y un módulo de autenticación para obtener un token de seguridad de Active Directory mediante la biblioteca ADAL. La biblioteca ADAL para node.js facilita que las aplicaciones de node.js se autentiquen en Azure AD para acceder a los recursos web protegidos de Azure AD.

Objetivos y limitaciones arquitectónicas Lado del servidor La herramienta móvil se alojará en la plataforma Azure Cloud ejecutándose como una aplicación web. Todas las comunicaciones con el cliente deben cumplir con los estándares públicos de protocolo de comunicación HTTPS y TCP/IP. Lado del cliente Los usuarios podrán acceder a la herramienta móvil a través de dispositivos Android e iOS.

4.1. Seguridad

Para poder autenticar a los usuarios, debe registrar la aplicación Node.js con Azure Active Directory (AAD). Esto se hace en dos pasos. Primero, debe registrar su servicio móvil y exponer los permisos en él. En segundo lugar, debe registrar su aplicación iOS y otorgarle acceso a esos permisos. 4.2. Persistencia

La persistencia de los datos se abordará mediante la base de datos Azure Document DB.

4.3. Confiabilidad/Disponibilidad

4.4. Actuación

No hay restricciones particulares relacionadas con el rendimiento del sistema. Se anticipa que el sistema debería responder a cualquier solicitud con tiempos de espera de script de servidor web y base de datos estándar (20 segundos); además, el rendimiento del sistema puede depender del hardware disponible, la red PSU y las capacidades de conexión a Internet. Además, los tiempos de carga/descarga pueden depender del tamaño de los datos, que a su vez depende de la entrada del usuario. Por lo tanto, el rendimiento real se puede determinar solo después de la implementación y prueba del sistema.
