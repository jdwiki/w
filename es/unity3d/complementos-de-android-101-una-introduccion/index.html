Este tema es la primera parte de una serie sobre cómo crear complementos de Android para Unity. Comience aquí si tiene poca o ninguna experiencia en la creación de complementos y/o el sistema operativo Android.

A través de esta serie, uso extensivamente enlaces externos que los animo a leer. Si bien aquí se incluirán versiones parafraseadas del contenido relevante, puede haber ocasiones en las que la lectura adicional ayude.

Comenzando con los complementos de Android #

Actualmente, Unity ofrece dos formas de llamar al código nativo de Android.

  1. Escriba código nativo de Android en Java y llame a estas funciones de Java usando C#
  2. Escriba código C# para llamar directamente a las funciones que forman parte del sistema operativo Android

Para interactuar con el código nativo, Unity proporciona algunas clases y funciones.

  • AndroidJavaObject: esta es la clase base que proporciona Unity para interactuar con el código nativo. Casi cualquier objeto devuelto desde el código nativo se puede almacenar como y AndroidJavaObject
  • AndroidJavaClass - Hereda de AndroidJavaObject. Esto se usa para hacer referencia a clases en su código nativo
  • Get / Set valores de una instancia de un objeto nativo y las versiones estáticas GetStatic / SetStatic
  • Call / CallStatic para llamar a funciones nativas estáticas y no estáticas

Esquema para crear un complemento y terminología #

  1. Escriba código Java nativo en Android Studio
  2. Exportar el código en un archivo JAR/AAR (Pasos aquí para archivos JAR y archivos AAR)
  3. Copie el archivo JAR/AAR en su proyecto de Unity en Assets/Plugins/Android
  4. Escriba código en Unity (C# siempre ha sido el camino a seguir aquí) para llamar funciones en el complemento

¡Tenga en cuenta que los primeros tres pasos se aplican SOLO si desea tener un complemento nativo!

De ahora en adelante, me referiré al archivo JAR/AAR como el complemento nativo, y al script C# como el envoltorio de C#


Elegir entre los métodos de creación de complementos #

Es inmediatamente obvio que la primera forma de crear complementos es larga, por lo que elegir su ruta parece discutible. Sin embargo, el método 1 es la ÚNICA forma de llamar al código personalizado. Entonces, ¿cómo elige uno?

En pocas palabras, ¿su complemento

  1. Involucrar código personalizado: elija el método 1
  2. ¿Solo invocar funciones nativas de Android? - Elija el método 2

NO intente “mezclar” (es decir, una parte del complemento usando el método 1 y la otra usando el método 2) ¡los dos métodos! Si bien es completamente posible, a menudo es poco práctico y doloroso de manejar.
