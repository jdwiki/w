Poder de la rutina #

Uso #

Si tiene una operación de ejecución prolongada que se basa en la API de Unity no segura para subprocesos, use Coroutines para dividirla en varios marcos y mantener la capacidad de respuesta de su aplicación.

Corrutinas también ayudan a realizar acciones costosas cada enésimo cuadro en lugar de ejecutar esa acción en cada cuadro.

División de rutinas de ejecución prolongada en varios marcos ================================================== == Las corrutinas ayudan a distribuir operaciones de ejecución prolongada en varios fotogramas para ayudar a mantener la velocidad de fotogramas de su aplicación.

Las rutinas que pintan o generan terreno de manera procesal o generan ruido son ejemplos que pueden necesitar el tratamiento Coroutine.

for (int y = 0; y < heightmap.Height; y++) 
{
    for (int x = 0; x < heightmap.Width; x++)
    {
        // Generate pixel at (x, y)
        // Assign pixel at (x, y)
        
        // Process only 32768 pixels each frame
        if ((y * heightmap.Height + x) % 32 * 1024) == 0)
            yield return null; // Wait for next frame
    }
}

El código anterior es un ejemplo fácil de entender. En el código de producción, es mejor evitar la verificación por píxel que verifica cuándo ‘rendir retorno’ (tal vez hacerlo cada 2-3 filas) y precalcular la longitud del bucle ‘for’ por adelantado.

Realizar acciones costosas con menos frecuencia #

Las corrutinas lo ayudan a realizar acciones costosas con menos frecuencia, por lo que no es un impacto tan grande en el rendimiento como lo sería si se realizaran en cada cuadro.

Tomando el siguiente ejemplo directamente del Manual:

private void ProximityCheck() 
{
    for (int i = 0; i < enemies.Length; i++) 
    {
        if (Vector3.Distance(transform.position, enemies[i].transform.position) < dangerDistance)
                return true;
    }
    return false;
}

private IEnumerator ProximityCheckCoroutine() 
{
    while(true) 
    {
        ProximityCheck();
        yield return new WaitForSeconds(.1f);
    }
}

Las pruebas de proximidad se pueden optimizar aún más utilizando CullingGroup API.

Errores comunes #

Un error común que cometen los desarrolladores es acceder a los resultados o efectos secundarios de las corrutinas fuera de la corrutina. Las corrutinas devuelven el control a la persona que llama tan pronto como se encuentra una declaración de “retorno de rendimiento” y es posible que el resultado o el efecto secundario aún no se hayan realizado. Para evitar problemas en los que tiene que usar el resultado/efecto secundario fuera de la rutina, marque esta respuesta.

Cuerdas #

Se podría argumentar que hay mayores recursos en Unity que la humilde cadena, pero es uno de los aspectos más fáciles de solucionar desde el principio.

Las operaciones de cadena construyen basura #

La mayoría de las operaciones de cadena generan pequeñas cantidades de basura, pero si esas operaciones se llaman varias veces en el transcurso de una sola actualización, se acumulan. Con el tiempo, activará la recolección de basura automática, lo que puede resultar en un pico de CPU visible.

Guarda en caché tus operaciones con cadenas #

Considere el siguiente ejemplo.

string[] StringKeys = new string[] {
    "Key0",
    "Key1",
    "Key2"
};

void Update()
{
    for (var i = 0; i < 3; i++)
    {
        // Cached, no garbage generated
        Debug.Log(StringKeys[i]);
    }

    for (var i = 0; i < 3; i++)
    {
        // Not cached, garbage every cycle
        Debug.Log("Key" + i);
    }

    // The most memory-efficient way is to not create a cache at all and use literals or constants.
    // However, it is not necessarily the most readable or beautiful way.
    Debug.Log("Key0");
    Debug.Log("Key1");
    Debug.Log("Key2");
}

Puede parecer tonto y redundante, pero si está trabajando con Shaders, es posible que se encuentre con situaciones como estas. El almacenamiento en caché de las claves marcará la diferencia.

Tenga en cuenta que las cadenas literales y constantes no generan basura, ya que se inyectan estáticamente en el espacio de la pila del programa. Si está generando cadenas en tiempo de ejecución y está garantizado que generará las mismas cadenas cada vez como en el ejemplo anterior, el almacenamiento en caché definitivamente ayudará.

Para otros casos en los que la cadena generada no es la misma cada vez, no hay otra alternativa que generar esas cadenas. Como tal, el pico de memoria con la generación manual de cadenas cada vez suele ser insignificante, a menos que se generen decenas de miles de cadenas a la vez.

La mayoría de las operaciones de cadenas son mensajes de depuración #

Hacer operaciones de cadena para los mensajes de depuración, es decir. Debug.Log("Object Name: " + obj.name) está bien y no se puede evitar durante el desarrollo. Sin embargo, es importante asegurarse de que los mensajes de depuración irrelevantes no terminen en el producto lanzado.

Una forma es usar el atributo condicional en sus llamadas de depuración. Esto no solo elimina las llamadas a métodos, sino también todas las operaciones de cadena que se realizan.

using UnityEngine;
using System.Collections;

public class ConditionalDebugExample: MonoBehaviour
{
    IEnumerator Start()
    {
        while(true)
        {
            // This message will pop up in Editor but not in builds
            Log("Elapsed: " + Time.timeSinceLevelLoad);
            yield return new WaitForSeconds(1f);
        }
    }

    [System.Diagnostics.Conditional("UNITY_EDITOR")]
    void Log(string Message)
    {
        Debug.Log(Message);
    }
}

Este es un ejemplo simplificado. Es posible que desee invertir algo de tiempo en el diseño de una rutina de registro más completa.

Comparación de cadenas #

Esta es una optimización menor, pero vale la pena mencionarla. Comparar cadenas es un poco más complicado de lo que uno podría pensar. El sistema intentará tener en cuenta las diferencias culturales de forma predeterminada. Puede optar por usar una comparación binaria simple en su lugar, que funciona más rápido.

// Faster string comparison
if (strA.Equals(strB, System.StringComparison.Ordinal)) {...}
// Compared to
if (strA == strB) {...}

// Less overhead
if (!string.IsNullOrEmpty(strA)) {...}
// Compared to
if (strA == "") {...}

// Faster lookups
Dictionary<string, int> myDic = new Dictionary<string, int>(System.StringComparer.Ordinal);
// Compared to
Dictionary<string, int> myDictionary = new Dictionary<string, int>();

Verificaciones rápidas y eficientes #

Evite operaciones innecesarias y llamadas a métodos siempre que pueda, especialmente en un método que se llama muchas veces por segundo, como ‘Actualizar’.

Comprobaciones de distancia/alcance #

Use sqrMagnitude en lugar de magnitude al comparar distancias. Esto evita operaciones sqrt innecesarias. Tenga en cuenta que al usar sqrMagnitude, el lado derecho también debe elevarse al cuadrado.

if ((target.position - transform.position).sqrMagnitude < minDistance * minDistance))

Comprobaciones de límites #

Las intersecciones de objetos se pueden verificar crudamente comprobando si sus límites Collider/Renderer se cruzan. La estructura Bounds también tiene un práctico método Intersects que ayuda a determinar si dos límites se cruzan.

Bounds también nos ayuda a calcular una aproximación cercana de la distancia real (superficie a superficie) entre objetos (ver Bounds.SqrDistance).

Advertencias #

La verificación de límites funciona muy bien para objetos convexos, pero la verificación de límites en objetos cóncavos puede generar imprecisiones mucho mayores según la forma del objeto.

No se recomienda usar Mesh.bounds ya que devuelve los límites del espacio local. Utilice MeshRenderer.bounds en su lugar.

Evite los métodos de unidad vacíos #

Evite los métodos de unidad vacíos. Aparte de ser un mal estilo de programación, hay una sobrecarga muy pequeña involucrada en las secuencias de comandos en tiempo de ejecución. En muchos casos, esto puede acumularse y afectar el rendimiento.

void Update
{
}

void FixedUpdate
{
}

Referencias de caché #

Referencias de caché para evitar llamadas costosas, especialmente en la función de actualización. Esto se puede hacer almacenando en caché estas referencias al inicio si están disponibles o cuando estén disponibles y verificando nulo/bool flat para evitar obtener la referencia nuevamente.

Ejemplos:

Referencias de componentes de caché

cambio

void Update()
{
    var renderer = GetComponent<Renderer>();
    renderer.material.SetColor("_Color", Color.green);
}

a

private Renderer myRenderer;
void Start()
{
    myRenderer = GetComponent<Renderer>();
}

void Update()
{
    myRenderer.material.SetColor("_Color", Color.green);
}

Referencias de objeto de caché

cambio

void Update()
{
    var enemy = GameObject.Find("enemy");
    enemy.transform.LookAt(new Vector3(0,0,0));
}

a

private Transform enemy;

void Start()
{
    this.enemy = GameObject.Find("enemy").transform;
}

void Update()
{
    enemy.LookAt(new Vector3(0, 0, 0));
}

Además, almacene en caché llamadas costosas como llamadas a Mathf cuando sea posible.

Evite llamar a métodos usando cadenas #

Evite llamar a métodos usando cadenas que puedan aceptar métodos. Este enfoque hará uso de la reflexión que puede ralentizar el juego, especialmente cuando se usa en la función de actualización.

Ejemplos:

    //Avoid StartCoroutine with method name
    this.StartCoroutine("SampleCoroutine");

    //Instead use the method directly
    this.StartCoroutine(this.SampleCoroutine());

    //Avoid send message
    var enemy = GameObject.Find("enemy");
    enemy.SendMessage("Die");

    //Instead make direct call
    var enemy = GameObject.Find("enemy") as Enemy;
    enemy.Die();
