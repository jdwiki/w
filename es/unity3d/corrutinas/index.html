Sintaxis #

  • Public Coroutine StartCoroutine (rutina IEnumerator);
  • public Coroutine StartCoroutine(string methodName, object value = null);
  • public void StopCoroutine(string methodName);
  • StopCoroutine public void (rutina IEnumerator);
  • public void StopAllCoroutines();

Consideraciones de rendimiento #

Es mejor usar corrutinas con moderación, ya que la flexibilidad conlleva un costo de rendimiento.

  • Las corrutinas en grandes cantidades exigen más de la CPU que los métodos de actualización estándar.
  • Hay un problema en algunas versiones de Unity donde las corrutinas producen basura en cada ciclo de actualización debido a que Unity encajona el valor de retorno MoveNext. Esto se observó por última vez en 5.4.0b13. (Informe de error)

Reduzca la basura almacenando en caché YieldInstructions #

Un truco común para reducir la basura generada en coroutines es almacenar en caché YieldInstruction.

IEnumerator TickEverySecond()
{
    var wait = new WaitForSeconds(1f); // Cache
    while(true)
    {
        yield return wait; // Reuse
    }
}

Ceder null no produce basura adicional.

Corrutinas #

Primero, es esencial comprender que los motores de juegos (como Unity) funcionan en un paradigma “basado en marcos”.

El código se ejecuta durante cada cuadro.

Eso incluye el propio código de Unity y tu código.

Al pensar en marcos, es importante comprender que no hay absolutamente garantía de cuándo suceden los marcos. Ellos no suceden en un ritmo regular. Los espacios entre marcos podrían ser, por ejemplo, 0.02632 luego 0.021167 luego 0.029778, y así sucesivamente. En el ejemplo, todos son “aproximadamente” 1/50 de segundo, pero todos son diferentes. Y en cualquier momento, puede obtener un marco que tarde mucho más o menos tiempo; y su código puede ejecutarse en cualquier momento dentro del marco.

Teniendo eso en cuenta, puede preguntarse: ¿cómo accede a estos marcos en su código, en Unity?

Sencillamente, usa la llamada Update() o usa una corrutina. (De hecho, son exactamente lo mismo: permiten que el código se ejecute en cada cuadro).

El propósito de una rutina es que:

puede ejecutar algún código y luego, “detenerse y esperar” hasta algún marco futuro.

Puede esperar hasta el siguiente fotograma, puede esperar una cantidad de fotogramas o puede esperar algún tiempo aproximado en segundos en el futuro.

Por ejemplo, puede esperar “alrededor de un segundo”, lo que significa que esperará aproximadamente un segundo, y luego colocará su código en algún marco aproximadamente dentro de un segundo. (Y, de hecho, dentro de ese marco, el código podría ejecutarse en cualquier momento, en cualquier momento). Para repetir: no será exactamente un segundo. La sincronización precisa no tiene sentido en un motor de juego.

Dentro de una rutina:

Para esperar un cuadro:

// do something
yield return null;  // wait until next frame
// do something

Para esperar tres fotogramas:

// do something
yield return null;  // wait until three frames from now
yield return null;
yield return null;
// do something

Para esperar aproximadamente medio segundo:

// do something
yield return new WaitForSeconds (0.5f); // wait for a frame in about .5 seconds
// do something

Haz algo en cada fotograma:

while (true)
{
    // do something
    yield return null;  // wait until the next frame
}

Ese ejemplo es literalmente idéntico a simplemente poner algo dentro de la llamada “Actualizar” de Unity: el código en “hacer algo” se ejecuta en cada cuadro.

Ejemplo #

Adjunte Ticker a un GameObject. Mientras ese objeto del juego esté activo, el tick se ejecutará. Tenga en cuenta que el script detiene cuidadosamente la rutina cuando el objeto del juego se vuelve inactivo; este suele ser un aspecto importante del uso correcto de la rutina de ingeniería.

using UnityEngine;
using System.Collections;

public class Ticker:MonoBehaviour {

    void OnEnable()
    {
        StartCoroutine(TickEverySecond());
    }

    void OnDisable()
    {
        StopAllCoroutines();
    }

    IEnumerator TickEverySecond()
    {
        var wait = new WaitForSeconds(1f); // REMEMBER: IT IS ONLY APPROXIMATE
        while(true)
        {
            Debug.Log("Tick");
            yield return wait;  // wait for a frame, about 1 second from now
        }
    }
}

Encadenamiento de corrutinas #

Las corrutinas pueden ceder dentro de sí mismas y esperar a otras corrutinas.

Entonces, puedes encadenar secuencias - “una tras otra”.

Esto es muy fácil y es una técnica básica y central en Unity.

Es absolutamente natural en los juegos que ciertas cosas tengan que suceder “en orden”. Casi cada “ronda” de un juego comienza con una cierta serie de eventos que suceden, durante un espacio de tiempo, en algún orden. Así es como podrías comenzar un juego de carreras de autos:

IEnumerator BeginRace()
{
  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());
}

Entonces, cuando llamas a BeginRace…

 StartCoroutine(BeginRace());

Ejecutará tu rutina de “preparar carrera”. (Tal vez, encender algunas luces y hacer ruido de la multitud, restablecer puntajes, etc.) Cuando termine, ejecutará su secuencia de “cuenta regresiva”, donde animaría quizás una cuenta regresiva en la interfaz de usuario. Cuando termine, ejecutará su código de inicio de carrera, donde quizás ejecute efectos de sonido, inicie algunos controladores de IA, mueva la cámara de cierta manera, etc.

Para mayor claridad, comprenda que las tres llamadas

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

deben ellos mismos estar en una rutina. Es decir, deben estar en una función del tipo IEnumerator. Entonces, en nuestro ejemplo, eso es IEnumerator BeginRace. Entonces, desde el código “normal”, inicias esa corrutina con la llamada StartCoroutine.

 StartCoroutine(BeginRace());

Para comprender mejor el encadenamiento, aquí hay una función que encadena rutinas. Pasas una serie de rutinas. La función ejecuta tantas rutinas como pasa, en orden, una tras otra.

// run various routines, one after the other
IEnumerator OneAfterTheOther( params IEnumerator[] routines ) 
{
    foreach ( var item in routines ) 
    {
        while ( item.MoveNext() ) yield return item.Current;
    }

    yield break;
}

Así es como lo llamarías… digamos que tienes tres funciones. Recuerde que todos deben ser IEnumerator:

IEnumerator PrepareRace() 
{
    // codesay, crowd cheering and camera pan around the stadium
    yield break;
}

IEnumerator Countdown() 
{
    // codesay, animate your countdown on UI
    yield break;
}

IEnumerator StartRace() 
{
    // codesay, camera moves and light changes and launch the AIs
    yield break;
}

Lo llamarías así

StartCoroutine( MultipleRoutines( PrepareRace(), Countdown(), StartRace() ) );

o tal vez así

IEnumerator[] routines = new IEnumerator[] {
     PrepareRace(),
     Countdown(),
     StartRace() };
StartCoroutine( MultipleRoutines( routines ) );

Para repetir, uno de los requisitos más básicos en los juegos es que ciertas cosas sucedan una tras otra “en una secuencia” a lo largo del tiempo. Lo logras en Unity de manera muy simple, con

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

Terminar una rutina #

A menudo, diseña rutinas para que finalicen naturalmente cuando se cumplan ciertos objetivos.

IEnumerator TickFiveSeconds()
{
    var wait = new WaitForSeconds(1f);
    int counter = 1;
    while(counter < 5)
    {
        Debug.Log("Tick");
        counter++;
        yield return wait;
    }
    Debug.Log("I am done ticking");
}

Para detener una rutina desde “dentro” de la rutina, no puede simplemente “regresar” como lo haría para salir temprano de una función ordinaria. En su lugar, utiliza interrupción de rendimiento.

IEnumerator ShowExplosions()
{
    ... show basic explosions
    if(player.xp < 100) yield break;
    ... show fancy explosions
}

También puede forzar que todas las corrutinas iniciadas por el script se detengan antes de finalizar.

void OnDisable()
{
    // Stops all running coroutines
    StopAllCoroutines();
}

El método para detener una rutina específica de la persona que llama varía según cómo la haya iniciado.

Si comenzaste una rutina por nombre de cadena:

StartCoroutine("YourAnimation");

luego puede detenerlo llamando a StopCoroutine con el mismo nombre de cadena:

StopCoroutine("YourAnimation");

Alternativamente, puede mantener una referencia a ya sea el ‘IEnumerator’ devuelto por el método coroutine, o el objeto ‘Coroutine’ devuelto por ‘StartCoroutine’, y llamar a ‘StopCoroutine’ en cualquiera de ellos:

public class SomeComponent : MonoBehaviour 
{
    Coroutine routine;

    void Start () {
        routine = StartCoroutine(YourAnimation());
    }

    void Update () {
        // later, in response to some input...
        StopCoroutine(routine);
    }

    IEnumerator YourAnimation () { /* ... */ }
}

Métodos MonoBehaviour que pueden ser rutinas #

Hay tres métodos MonoBehaviour que se pueden convertir en corrutinas.

  1. Inicio()
  2. Al hacerse visible()
  3. OnLevelWasLoaded()

Esto se puede usar para crear, por ejemplo, secuencias de comandos que se ejecutan solo cuando el objeto es visible para una cámara.

using UnityEngine;
using System.Collections;

public class RotateObject : MonoBehaviour
{
    IEnumerator OnBecameVisible()
    {
        var tr = GetComponent<Transform>();
        while (true)
        {
            tr.Rotate(new Vector3(0, 180f * Time.deltaTime));
            yield return null;
        }
    }
    
    void OnBecameInvisible()
    {
        StopAllCoroutines();
    }
}

Maneras de rendir #

Puedes esperar hasta el siguiente cuadro.

yield return null; // wait until sometime in the next frame

Puede tener varias de estas llamadas seguidas, para simplemente esperar tantos marcos como desee.

//wait for a few frames
yield return null;
yield return null;

Espere aproximadamente n segundos. Es extremadamente importante entender que esto es solo un tiempo muy aproximado.

yield return new WaitForSeconds(n);

No es absolutamente posible usar la llamada “WaitForSeconds” para ninguna forma de sincronización precisa.

A menudo quieres encadenar acciones. Entonces, haz algo, y cuando eso termine, haz otra cosa, y cuando eso termine, haz otra cosa. Para lograr eso, espera otra rutina:

yield return StartCoroutine(coroutine);

Comprenda que solo puede llamar eso desde dentro de una rutina. Asi que:

StartCoroutine(Test());

Así es como se inicia una corrutina a partir de un código “normal”.

Luego, dentro de esa corrutina en ejecución:

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

Eso imprimirá A, comenzará el proceso largo e imprimirá inmediatamente B. No esperará a que finalice el largo proceso. Por otra parte:

Debug.Log("A");
yield return StartCoroutine(LongProcess());
Debug.Log("B");

Eso imprimirá A, comenzará el proceso largo, esperará hasta que finalice y luego imprimirá B.

Siempre vale la pena recordar que las corrutinas no tienen absolutamente ninguna conexión, de ninguna manera, con los subprocesos. Con este código:

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

es fácil pensar que es “como” iniciar LongProcess en otro subproceso en segundo plano. Pero eso es absolutamente incorrecto. Es solo una rutina. Los motores de juego se basan en marcos y las “rutinas” en Unity simplemente le permiten acceder a los marcos.

Es muy fácil esperar a que se complete una solicitud web.

void Start() {
    string url = "http://google.com";
    WWW www = new WWW(url);
    StartCoroutine(WaitForRequest(www));
}

IEnumerator WaitForRequest(WWW www) {
    yield return www;
    
    if (www.error == null) {
        //use www.data);
    }
    else {
        //use www.error);
    }
}

Para completar: en casos muy raros, usa una actualización fija en Unity; hay una llamada WaitForFixedUpdate() que normalmente nunca se usaría. Hay una llamada específica (“WaitForEndOfFrame()” en la versión actual de Unity) que se usa en ciertas situaciones en relación con la generación de capturas de pantalla durante el desarrollo. (El mecanismo exacto cambia ligeramente a medida que Unity evoluciona, así que busque en Google la información más reciente si es relevante).
