¿Qué caracteres deben escaparse? #

El escape de caracteres es lo que permite que ciertos caracteres (reservados por el motor de expresiones regulares para manipular búsquedas) se busquen y encuentren literalmente en la cadena de entrada. El escape depende del contexto, por lo tanto, este ejemplo no cubre el escape string o delimiter.

barras invertidas #

Decir que la barra invertida es el carácter de “escape” es un poco engañoso. La barra invertida escapa y la barra invertida trae; en realidad, activa o desactiva el metacarácter frente al estado literal del personaje que tiene delante.

Para usar una barra invertida literal en cualquier lugar de una expresión regular, debe escaparse con otra barra invertida.

Escapar (fuera de las clases de personajes) #

Hay varios caracteres que deben escaparse para tomarse literalmente (al menos fuera de las clases de caracteres):

  • Corchetes: []
  • Paréntesis: ()
  • Llaves: {}
  • Operadores: *, +, ?, |
  • Anclas: ^, $
  • Otros: ., \
  • Para usar un literal ^ al principio o un literal $ al final de una expresión regular, el carácter debe tener escape.
  • Algunos sabores solo usan ^ y $ como metacaracteres cuando están al principio o al final de la expresión regular, respectivamente. En esos sabores, no se necesita escape adicional. Por lo general, es mejor escapar de ellos de todos modos.

Escapar dentro de las clases de personajes #

  • Se recomienda escapar de los corchetes ([ y ]) cuando aparecen como literales en una clase char. Bajo ciertas condiciones, [no se requiere, según el tipo] (http://codegolf.stackexchange.com/a/47435), pero perjudica la legibilidad.
  • El signo de intercalación, ^, es un metacarácter cuando se coloca como el primer carácter en una clase char: [^aeiou]. En cualquier otro lugar de la clase char, es solo un carácter literal.
  • El guión, -, es un metacarácter, a menos que esté al principio o al final de una clase de carácter. Si el primer carácter de la clase char es un signo de intercalación ^, entonces será un literal si es el segundo carácter de la clase char.

Escapando del Reemplazo #

También hay reglas para escapar dentro del reemplazo, pero no se aplica ninguna de las reglas anteriores. Los únicos metacaracteres son $ y \, al menos cuando se puede usar $ para hacer referencia a grupos de captura (como $1 para el grupo 1). Para usar un $ literal, escápelo: \$5.00. Del mismo modo \: C:\\Archivos de programa\\.

Excepciones BRE #

Mientras que ERE (expresiones regulares extendidas) refleja la sintaxis típica de estilo Perl, BRE (expresiones regulares básicas) tiene diferencias significativas cuando se trata de escapar:

  • Hay una sintaxis abreviada diferente. Todos los \d, \s, \w y demás se han ido. En cambio, tiene su propia sintaxis (que POSIX llama confusamente “clases de caracteres”), como [:digit:]. Estas construcciones deben estar dentro de una clase de caracteres.
  • Hay pocos metacaracteres (., *, ^, $) que se pueden usar normalmente. TODOS los demás metacaracteres deben escaparse de manera diferente:

Llaves {}

  • a{1,2} coincide con a{1,2}. Para hacer coincidir a o aa, use a\{1,2\}

Paréntesis ()

  • (ab)\1 no es válido, ya que no hay un grupo de captura 1. Para arreglarlo y hacer coincidir abab use \(ab\)\1

barra invertida

  • Dentro de las clases char (que se denominan expresiones de paréntesis en POSIX), la barra invertida no es un metacarácter (y no necesita escape). [\d] coincide con \ o d.
  • En cualquier otro lugar, escapa como siempre.

Otro

  • + y ? son literales. Si el motor BRE los admite como metacaracteres, deben escaparse como \? y \+.

Cuerdas #

En la mayoría de los lenguajes de programación, para tener una barra invertida en una cadena generada a partir de un literal de cadena, cada barra invertida debe duplicarse en el literal de cadena. De lo contrario, se interpretará como un escape para el siguiente carácter.

Desafortunadamente, cualquier barra invertida requerida por la expresión regular debe ser una barra invertida literal. Esta es la razón por la que se hace necesario tener “escapes escapados” (\\) cuando las expresiones regulares se generan a partir de cadenas literales.

Además, es posible que sea necesario escapar las comillas (" o ') en el literal de cadena, dependiendo de cuál rodee el literal de cadena. En algunos idiomas, es posible usar cualquier estilo de comillas para una cadena (elija el el más legible para escapar de toda la cadena literal).

En algunos lenguajes (p. ej.: Java <=7), las expresiones regulares no se pueden expresar directamente como literales como /\w/; deben generarse a partir de cadenas, y normalmente se usan literales de cadena, en este caso, "\\w". En estos casos, los caracteres literales como comillas, barras invertidas, etc. deben escaparse. La forma más fácil de lograr esto puede ser usando una herramienta (como RegexPlanet). Esta herramienta específica está diseñada para Java, pero funcionará para cualquier idioma con una sintaxis de cadena similar.

/Delimitadores/ #

Muchos idiomas permiten encerrar o delimitar expresiones regulares entre un par de caracteres específicos, generalmente la barra inclinada /.

Los delimitadores tienen un impacto en el escape: si el delimitador es / y la expresión regular necesita buscar literales /, entonces la barra diagonal debe escaparse antes de que pueda ser un literal (\/).

El escape excesivo daña la legibilidad, por lo que es importante considerar las opciones disponibles:

Javascript es único porque permite la barra diagonal como delimitador, pero nada más (aunque sí permite [regexes en forma de cadena] (http://stackoverflow.com/questions/23438047)).

Perl, por ejemplo, permite que casi cualquier cosa sea un delimitador. Incluso caracteres árabes:

$str =~ m ش ش

Las reglas específicas se mencionan en [la documentación de Perl] (http://perldoc.perl.org/perlop.html#Regexp-Quote-Like-Operators).

PCRE permite dos tipos de delimitadores: delimitadores coincidentes y delimitadores de estilo corchete. Los delimitadores coincidentes utilizan un par de caracteres individuales, mientras que los delimitadores de estilo corchete utilizan un par de caracteres que representan un par de apertura y cierre.

  • Delimitadores coincidentes: !"#$%&'*+,./:;=?@^_`|~-
  • Delimitadores de estilo corchete: (), {}, [], <>

Literales de cadena sin procesar #

Es mejor para la legibilidad (y tu cordura) evitar escapar de los escapes. Ahí es donde entran los literales de cadenas sin procesar. (Tenga en cuenta que algunos idiomas permiten delimitadores, que generalmente se prefieren a las cadenas. Pero esa es otra sección).

Por lo general, funcionan de la misma manera que [esta respuesta describe] (http://stackoverflow.com/a/2081708/6083675):

[A] barra invertida, \, se considera que significa “solo una barra invertida” (excepto cuando aparece justo antes de una cita que de otro modo terminaría el literal) – no hay “secuencias de escape” para representar saltos de línea, tabuladores, retrocesos, alimentaciones de formulario, y así sucesivamente.

No todos los idiomas los tienen, y los que sí usan una sintaxis variable. C# en realidad los llama literales de cadena textuales, pero es lo mismo.

pitón #

pattern = r"regex"

pattern = r'regex'

#C++ (11+)

La sintaxis aquí es extremadamente versátil. La única regla es usar un delimitador que no aparezca en ninguna parte de la expresión regular. Si lo hace, no es necesario ningún escape adicional para nada en la cadena. Tenga en cuenta que los paréntesis () no forman parte de la expresión regular:

pattern = R"delimiter(regex)delimiter";

#VB.NET

Solo usa una cadena normal. Las barras invertidas son SIEMPRE [literales] (http://stackoverflow.com/a/13155483/6083675).

C# #

pattern = @"regex";

Tenga en cuenta que esta sintaxis también permite "" (dos comillas dobles) como forma de escape de ".
