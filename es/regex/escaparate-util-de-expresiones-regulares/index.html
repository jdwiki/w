Coincide con una dirección de correo electrónico #

Hacer coincidir una dirección de correo electrónico dentro de una cadena es una tarea difícil, porque el especificación que lo define, el RFC2822, es complejo, lo que dificulta su implementación como expresión regular. Para más detalles por qué es no es una buena idea hacer coincidir un correo electrónico con una expresión regular, consulte el ejemplo de antipatrón [cuándo no usar una expresión regular: para hacer coincidir correos electrónicos](https://www.wikiod.com/es/regex/cuando-no-debes-usar-expresiones-regulares número-o-número-de-tarjeta-de-crédito-o#t=201607301200162540574). El mejor consejo a tener en cuenta de esa página es utilizar una revisión por pares y ampliamente biblioteca en tu idioma favorito para implementar esto.

Validar un formato de dirección de correo electrónico #

Cuando necesite validar rápidamente una entrada para asegurarse de que parece un correo electrónico, la mejor opción es mantenerlo simple:

^\S{1,}@\S{2,}\.\S{2,}$

Esa expresión regular verificará que la dirección de correo sea una secuencia separada sin espacios de caracteres de longitud superior a uno, seguido de un @, seguido de dos Secuencias de caracteres sin espacios de longitud dos o más separados por un .. No es perfecto y podría validar direcciones no válidas (según el formato), pero lo más importante, no está invalidando direcciones válidas.

Comprobar que la dirección existe #

La única forma fiable de comprobar que un correo electrónico es válido es comprobar su existencia. Solía ​​​​existir el comando SMTP VRFY que ha sido diseñado para ese propósito, pero Lamentablemente, después de [ser abusado por los spammers, ya no está disponible] (http://stackoverflow.com/a/566121/1290438).

Entonces, la única forma que le queda para verificar que el correo sea válido y exista es enviar un correo electrónico a esa dirección.

Enormes alternativas Regex #

Sin embargo, no es imposible validar una dirección de correo electrónico usando una expresión regular. Los únicos problemas es que cuanto más cerca de la especificación estén esas expresiones regulares, más grandes serán y como consecuencia, son imposiblemente difíciles de leer y mantener. A continuación, encontrará un ejemplo de tales expresiones regulares más precisas que se utilizan en algunas bibliotecas.

⚠️ Las siguientes expresiones regulares se proporcionan con fines de documentación y aprendizaje, copiar y pegar ellos en tu código es una mala idea. En su lugar, use esa biblioteca directamente, para que pueda confiar en código upstream y desarrolladores pares para mantener su código de análisis de correo electrónico actualizado y mantenido.

Módulo de coincidencia de direcciones de Perl #

Los mejores ejemplos de tales expresiones regulares se encuentran en las bibliotecas estándar de algunos idiomas. Por ejemplo, hay uno del módulo RFC::RFC822::Address en la biblioteca de Perl que trata de ser lo más precisa posible de acuerdo con el RFC. Para tu curiosidad, puede encontrar una versión de esa expresión regular en [esta URL] (http://www.ex-parrot.com/%7Epdw/Mail-RFC822-Address.html), que se ha generado a partir de la gramática, y si tiene la tentación de copiarlo y pegarlo, aquí hay una cita del autor de expresiones regulares:

“No mantengo la expresión regular [vinculada]. Puede haber errores en ella que ya han sido corregidos en el módulo Perl.”

Módulo de coincidencia de direcciones .Net #

Otra variante más corta es la utilizada por la biblioteca estándar .Net en el Módulo EmailAddressAttribute:

^((([a-z]|\d|[!#\$%&'\*\+\-\/=\?\^_`{\|}~]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])+(\.([a-z]|\d|[!#\$%&'\*\+\-\/=\?\^_`{\|}~]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])+)*)|((\x22)((((\x20|\x09)*(\x0d\x0a))?(\x20|\x09)+)?(([\x01-\x08\x0b\x0c\x0e-\x1f\x7f]|\x21|[\x23-\x5b]|[\x5d-\x7e]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])|(\\([\x01-\x09\x0b\x0c\x0d-\x7f]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF]))))*(((\x20|\x09)*(\x0d\x0a))?(\x20|\x09)+)?(\x22)))@((([a-z]|\d|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])|(([a-z]|\d|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])([a-z]|\d|-|\.|_|~|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])*([a-z]|\d|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])))\.)+(([a-z]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])|(([a-z]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])([a-z]|\d|-|\.|_|~|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])*([a-z]|[\u00A0-\uD7FF\uF900-\uFDCF\uFDF0-\uFFEF])))\.?$

Pero incluso si es más corto, sigue siendo demasiado grande para ser legible y fácil de mantener.

Módulo de coincidencia de direcciones Ruby #

En ruby, se usa una composición de expresiones regulares en el [módulo rfc822] (https://github.com/mspanc/rfc822/blob/master/lib/rfc822.rb) para hacer coincidir una dirección. Esta es una buena idea, ya que en caso de que se encuentren errores, será más fácil identificar la parte de expresión regular para cambiar y arreglalo.

Módulo de coincidencia de direcciones de Python #

Como contraejemplo, Python [módulo de análisis de correo electrónico] (https://github.com/python/cpython/blob/2d264235f6e066611b412f7c2e1603866e0f7f1b/Lib/email/_parseaddr.py#L260-L317) no está usando una expresión regular, sino que la implementa usando un analizador.

Hacer coincidir una fecha #

Debe recordar que la expresión regular fue diseñada para coincidir con una fecha (o no). Decir que una fecha es válida es una lucha mucho más complicada, ya que requerirá mucho manejo de excepciones (ver condiciones de año bisiesto).

Empecemos haciendo coincidir el mes (1 - 12) con un 0 inicial opcional:

0?[1-9]|1[0-2]

Para hacer coincidir el día, también con un 0 inicial opcional:

0?[1-9]|[12][0-9]|3[01]

Y para que coincida con el año (supongamos el rango 1900 - 2999):

(?:19|20)[0-9]{2}

El separador puede ser un espacio, un guión, una barra oblicua, vacío, etc. Siéntase libre de agregar cualquier cosa que crea que puede usarse como separador:

[-\\/ ]?

Ahora concatenas todo y obtienes:

(0?[1-9]|1[0-2])[-\\/ ]?(0?[1-9]|[12][0-9]|3[01])[-/ ]?(?:19|20)[0-9]{2} // MMDDYYYY
(0?[1-9]|[12][0-9]|3[01])[-\\/ ]?(0?[1-9]|1[0-2])[-/ ]?(?:19|20)[0-9]{2} // DDMMYYYY
(?:19|20)[0-9]{2}[-\\/ ]?(0?[1-9]|1[0-2])[-/ ]?(0?[1-9]|[12][0-9]|3[01]) // YYYYMMDD

Si quiere ser un poco más pedante, puede usar una referencia inversa para asegurarse de que los dos separadores sean iguales:

(0?[1-9]|1[0-2])([-\\/ ]?)(0?[1-9]|[12][0-9]|3[01])\2(?:19|20)[0-9]{2} // MMDDYYYY
                                                 ^ refer to [-/ ]
(0?[1-9]|[12][0-9]|3[01])([-\\/ ]?)(0?[1-9]|1[0-2])\2(?:19|20)[0-9]{2} // DDMMYYYY
(?:19|20)[0-9]{2}([-\\/ ]?)(0?[1-9]|1[0-2])\2(0?[1-9]|[12][0-9]|3[01]) // YYYYMMDD

Coincide con un número de teléfono #

Aquí se explica cómo hacer coincidir un código de prefijo (un + o (00), luego un número del 1 al 1939, con un espacio opcional): Esto no busca un prefijo válido sino algo que podría ser un prefijo. Consulte la lista completa de prefijos

(?:00|\+)?[0-9]{4}

Entonces, como la longitud total del número de teléfono es, como máximo, 15, podemos buscar hasta 14 dígitos: Se gasta al menos 1 dígito para el prefijo

[0-9]{1,14}

Los números pueden contener espacios, puntos o guiones y pueden agruparse por 2 o 3.

(?:[ .-][0-9]{3}){1,5}

Con el prefijo opcional:

(?:(?:00|\+)?[0-9]{4})?(?:[ .-][0-9]{3}){1,5}

Si desea hacer coincidir un formato de país específico, puede usar esta [consulta de búsqueda] (http://stackoverflow.com/search?q=phone+number+%5Bregex%5D) y agregar el país, la pregunta ciertamente ya ha sido preguntó.

Coincide con una dirección IP #

IPv4

Para hacer coincidir el formato de la dirección IPv4, debe verificar los números [0-9]{1,3} tres veces {3} separados por puntos \. y que terminan con otro número.

^(?:[0-9]{1,3}\.){3}[0-9]{1,3}$

Esta expresión regular es demasiado simple: si desea que sea precisa, debe verificar que los números estén entre 0 y 255, con la expresión regular anterior aceptando 444 en cualquier posición. ¿Quiere verificar 250-255 con 25[0-5], o cualquier otro valor de 200 2[0-4][0-9], o cualquier valor de 100 o menos con [01]?[ 0-9][0-9]. Quiere comprobar que va seguido de un punto \. tres veces {3} y luego una vez sin punto.

^(?:(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\.){3}(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)$

IPv6

Las direcciones IPv6 toman la forma de 8 palabras hexadecimales de 16 bits delimitadas con el carácter de dos puntos (:). En este caso, buscamos 7 palabras seguidas de dos puntos, seguidas de una que no lo es. Si una palabra tiene ceros iniciales, pueden estar truncados, lo que significa que cada palabra puede contener entre 1 y 4 dígitos hexadecimales.

^(?:[0-9a-fA-F]{1,4}:){7}[0-9a-fA-F]{1,4}$

Esto, sin embargo, es insuficiente. Como las direcciones IPv6 pueden volverse bastante “palabras”, el estándar especifica que las palabras de solo cero pueden ser reemplazadas por ::. Esto solo se puede hacer una vez en una dirección (entre 1 y 7 palabras consecutivas), ya que de lo contrario sería indeterminado. Esto produce una serie de variaciones (bastante desagradables):

^::(?:[0-9a-fA-F]{1,4}:){0,6}[0-9a-fA-F]{1,4}$
^[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,5}[0-9a-fA-F]{1,4}$
^[0-9a-fA-F]{1,4}:[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,4}[0-9a-fA-F]{1,4}$
^(?:[0-9a-fA-F]{1,4}:){0,2}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,3}[0-9a-fA-F]{1,4}$
^(?:[0-9a-fA-F]{1,4}:){0,3}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,2}[0-9a-fA-F]{1,4}$
^(?:[0-9a-fA-F]{1,4}:){0,4}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:)?[0-9a-fA-F]{1,4}$
^(?:[0-9a-fA-F]{1,4}:){0,5}[0-9a-fA-F]{1,4}::[0-9a-fA-F]{1,4}$
^(?:[0-9a-fA-F]{1,4}:){0,6}[0-9a-fA-F]{1,4}::$

Ahora, poniendo todo junto (usando la alternancia) produce:

^(?:[0-9a-fA-F]{1,4}:){7}[0-9a-fA-F]{1,4}$|
^::(?:[0-9a-fA-F]{1,4}:){0,6}[0-9a-fA-F]{1,4}$|
^[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,5}[0-9a-fA-F]{1,4}$|
^[0-9a-fA-F]{1,4}:[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,4}[0-9a-fA-F]{1,4}$|
^(?:[0-9a-fA-F]{1,4}:){0,2}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,3}[0-9a-fA-F]{1,4}$|
^(?:[0-9a-fA-F]{1,4}:){0,3}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:){0,2}[0-9a-fA-F]{1,4}$|
^(?:[0-9a-fA-F]{1,4}:){0,4}[0-9a-fA-F]{1,4}::(?:[0-9a-fA-F]{1,4}:)?[0-9a-fA-F]{1,4}$|
^(?:[0-9a-fA-F]{1,4}:){0,5}[0-9a-fA-F]{1,4}::[0-9a-fA-F]{1,4}$|
^(?:[0-9a-fA-F]{1,4}:){0,6}[0-9a-fA-F]{1,4}::$

Asegúrese de escribirlo en modo de varias líneas y con un montón de comentarios para que quien tenga la tarea inevitable de averiguar qué significa esto no lo persiga con un objeto contundente.

Validar una cadena de tiempo de 12 horas y 24 horas #

Para un formato de hora de 12 horas se puede usar:

^(?:0?[0-9]|1[0-2])[-:][0-5][0-9]\s*[ap]m$

Dónde

  • (?:0?[0-9]|1[0-2]) es la hora
  • [-:] es el separador, que se puede ajustar para satisfacer sus necesidades
  • [0-5][0-9] es el minuto
  • \s*[ap]m seguido de cualquier número de caracteres de espacio en blanco, y am o pm

Si necesitas los segundos:

^(?:0?[0-9]|1[0-2])[-:][0-5][0-9][-:][0-5][0-9]\s*[ap]m$

Para un formato de hora de 24 horas:

^(?:[01][0-9]|2[0-3])[-:h][0-5][0-9]$

Dónde:

  • (?:[01][0-9]|2[0-3]) es la hora
  • [-:h] el separador, que se puede ajustar para satisfacer sus necesidades
  • [0-5][0-9] es el minuto

Con los segundos:

^(?:[01][0-9]|2[0-3])[-:h][0-5][0-9][-:m][0-5][0-9]$

Donde [-:m] es un segundo separador, reemplazando la h para las horas con una m para los minutos, y [0-5][0-9] es el segundo.

Coincide con el código postal del Reino Unido #

Regex para que coincida con códigos postales en el Reino Unido

El formato es el siguiente, donde A significa una letra y 9 un dígito:

Formato Cobertura Ejemplo
Celda Celda
AA9A 9AA Área de código postal de WC; EC1–EC4, NW1W, SE1P, SW1 EC1A 1BB
A9A 9AA E1W, N1C, N1P W1A 0AX
A9 9AA, A99 9AA B, E, G, L, M, N, S, W M1 1AE, B33 8TH
AA9 9AA, AA99 9AA Todos los demás códigos postales CR2 6XH, DN55 1PT 
(GIR 0AA)|((([A-Z-[QVX]][0-9][0-9]?)|(([A-Z-[QVX]][A-Z-[IJZ]][0-9][0-9]?)|(([A-Z-[QVX]][0-9][A-HJKPSTUW])|([A-Z-[QVX]][A-Z-[IJZ]][0-9][ABEHMNPRVWXY])))) [0-9][A-Z-[CIKMOV]]{2})

Donde primera parte:

 (GIR 0AA)|((([A-Z-[QVX]][0-9][0-9]?)|(([A-Z-[QVX]][A-Z-[IJZ]][0-9][0-9]?)|(([A-Z-[QVX]][0-9][A-HJKPSTUW])|([A-Z-[QVX]][A-Z-[IJZ]][0-9][ABEHMNPRVWXY]))))

Segundo:

[0-9][A-Z-[CIKMOV]]{2})
