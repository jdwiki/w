Hacer un modelo #

Creación de modelo #

Las clases de modelo deben extender Illuminate\Database\Eloquent\Model. La ubicación predeterminada para los modelos es el directorio /app.

Una clase de modelo se puede generar fácilmente con el comando Artisan:

php artisan make:model [ModelName]

Esto creará un nuevo archivo PHP en app/ por defecto, que se llama [ModelName].php, y contendrá todo el modelo para su nuevo modelo, que incluye la clase, el espacio de nombres y los usos necesarios para un modelo básico. configuración.

Si desea crear un archivo de migración junto con su Modelo, use el siguiente comando, donde -m también generará el archivo de migración:

php artisan make:model [ModelName] -m

Además de crear el modelo, esto crea una migración de base de datos que está conectada al modelo. El archivo PHP de migración de la base de datos se encuentra por defecto en database/migrations/. Esto no incluye, de forma predeterminada, nada más que las columnas id y created_at/updated_at, por lo que deberá editar el archivo para proporcionar columnas adicionales.

Tenga en cuenta que tendrá que ejecutar la migración (una vez que haya configurado el archivo de migración) para que el modelo comience a funcionar mediante el uso de php artesanal migrate desde la raíz del proyecto

Además, si desea agregar una migración más tarde, después de hacer el modelo, puede hacerlo ejecutando:

php artisan make:migration [migration name]

Digamos, por ejemplo, que desea crear un modelo para sus gatos, tendría dos opciones, crear con o sin migración. Elegiría crear sin migración si ya tuviera una tabla de gatos o no quisiera crear una en este momento.

Para este ejemplo, queremos crear una migración porque aún no tenemos una tabla, por lo que ejecutaría el siguiente comando.

php artisan make:model Cat -m

Este comando creará dos archivos:

  1. En la carpeta de la aplicación: app/Cat.php
  2. En la carpeta de la base de datos: database/migrations/timestamp_creat_cats_table.php

El archivo que nos interesa es este último, ya que es este archivo el que podemos decidir cómo queremos que se vea e incluya la tabla. Para cualquier migración predefinida, se nos proporciona una columna de identificación de incremento automático y columnas de marcas de tiempo.

El siguiente ejemplo de un extracto del archivo de migración incluye las columnas predefinidas anteriores, así como la adición del nombre del gato, la edad y el color:

public function up()
    {
        Schema::create('cats', function (Blueprint $table) {

            $table->increments('id');    //Predefined ID
            $table->string('name');      //Name
            $table->integer('age');      //Age
            $table->string('colour');    //Colour
            $table->timestamps();        //Predefined Timestamps

        });
    }

Entonces, como puede ver, es relativamente fácil crear el modelo y la migración para una tabla. Luego, para ejecutar la migración y crearla en su base de datos, ejecutaría el siguiente comando:

php artisan migrate

Lo que migrará cualquier migración pendiente a su base de datos.

Ubicación del archivo del modelo #

Los modelos se pueden almacenar en cualquier lugar gracias a PSR4.

Por defecto, los modelos se crean en el directorio app con el espacio de nombres de App. Para aplicaciones más complejas, generalmente se recomienda almacenar modelos dentro de sus propias carpetas en una estructura que tenga sentido para la arquitectura de sus aplicaciones.

Por ejemplo, si tuviera una aplicación que usara una serie de frutas como modelos, podría crear una carpeta llamada app/Fruits y dentro de esta carpeta crear Banana.php (manteniendo el [StudlyCase](https:// en.wikipedia.org/wiki/Studly_caps) nomenclatura), podría crear la clase Banana en el espacio de nombres App\Fruits:

namespace App\Fruits;

use Illuminate\Database\Eloquent\Model;

class Banana extends Model {
    // Implementation of "Banana" omitted
}

Configuración del modelo #

Eloquent sigue un enfoque de “convención sobre configuración”. Al extender la clase Modelo base, todos los modelos heredan las propiedades que se enumeran a continuación. A menos que se anule, se aplican los siguientes valores predeterminados:

Propiedad Descripción Predeterminado
$conexión protegida Nombre de conexión de base de datos Conexión de base de datos predeterminada
tabla de $ protegida Nombre de la tabla Por defecto, el nombre de la clase se convierte a snake_case y se pluraliza. Por ejemplo, SpecialPerson se convierte en special_people
$claveprimaria protegida Mesa PK id
$incremento público Indica si los ID se incrementan automáticamente verdadero
$marcas de tiempo públicas Indica si el modelo debe tener una marca de tiempo verdadero
const CREATED_AT Nombre de la columna de marca de tiempo de creación creado_en
const ACTUALIZADO_EN Nombre de la columna de marca de tiempo de modificación actualizado_en
$fechas protegidas Atributos que deben mutarse a DateTime, además de los atributos de marcas de tiempo []
protegido $formato de fecha Formato en el que se conservarán los atributos de fecha Predeterminado para el dialecto SQL actual.
protegido $con Relaciones con eagerload con el modelo []
protegido $ oculto Atributos omitidos en la serialización del modelo []
protegido $visible Atributos permitidos en la serialización de modelos []
$anexos protegidos Accesores de atributos agregados a la serialización del modelo []
protegido $rellenable Atributos que son asignables en masa []
protegido $guardado Atributos que están en la lista negra de la asignación masiva [*] (Todos los atributos)
$toques protegidos Las relaciones que se deben tocar en guardar []
protegido $porPágina El número de modelos a devolver para la paginación. 15
Propiedad Descripción Predeterminado
transmisiones $protegidas Atributos que deben convertirse a tipos nativos []

Actualizar un modelo existente #

$user = User::find(1);
$user->name = 'abc';
$user->save();

También puede actualizar varios atributos a la vez usando update, que no requiere usar save después:

$user = User::find(1);
$user->update(['name' => 'abc', 'location' => 'xyz']);

También puede actualizar uno o más modelos sin consultarlo de antemano:

User::where('id', '>', 2)->update(['location' => 'xyz']);

Si no desea activar un cambio en la marca de tiempo updated_at en el modelo, puede pasar la opción touch:

$user = User::find(1);
$user->update(['name' => 'abc', 'location' => 'xyz'], ['touch' => false]);
