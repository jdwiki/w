Datatables 1.10.x es la última versión a partir de ahora. Aunque es compatible con versiones anteriores (1.9, etc.), es muy recomendable utilizar la última versión que devuelve directamente un objeto datatable api.

Otro cambio importante, que es el más visible, es el cambio de Notación húngara a [camelcase](https://en.wikipedia.org/wiki/ El caso de Carmel)

Sintaxis #

  • El único cambio de sintaxis importante es el uso de camelCase en todas partes en lugar de la notación húngara.
  • Una [guía más detallada puede ser encontrado aquí] (https://datatables.net/upgrade/1.10-convert) que implica la conversión de parámetros en <1.10 a 1.10+

Para obtener más detalles, intente visitar las siguientes páginas:

  1. Actualización a Datatables 1.10
  2. Changelog Datatables 1.10
  3. [Convertir nombres de parámetros de 1.9 a 1.10] (https://datatables.net/upgrade/1.10-convert)
  4. Usando la API de Datatable

Inicialización de Datatable 1.10+ #

Anteriormente, las tablas de datos se inicializaban de la siguiente manera:

var oTable = $("#selector").dataTable();

Esto solía devolver un objeto jQuery que se almacenaría en la variable oTable. Y luego, para acceder a la API para modificar las propiedades de la tabla, tuvimos que inicializar la API de manera diferente, como se muestra a continuación:

var api = oTable.api()
//r
var api = new $.fn.dataTable.api("#selector")

Ahora, la inicialización se cambia a lo siguiente:

var table = $("#selector").DataTable();

Tenga en cuenta que esto devuelve una instancia de datatable api y puede usar directamente la variable table para manipular las propiedades de la tabla de datos.

Características No disponible en Datatables 1.10+ #

Las siguientes 3 características (que quedaron en desuso en 1.9) ya no están disponibles en 1.10, son:

  1. fnRender: según el desarrollador:

La antigua opción fnRender proporcionaba un método para manipular una celda cuando fue creado. sin embargo, se le proporcionó una lista confusa de opciones como sus argumentos, y requería una estructura particular en DataTables internamente que causaron problemas de rendimiento. Eliminación de fnRender ha llevado a una mejora significativa en el rendimiento de DataTables con grandes conjuntos de datos y la capacidad de proporcionar objetos instancias a DataTables como objetos de origen de datos (por ejemplo, Knockout objetos observables).

Las alternativas a fnRender están disponibles como columns.render y columns .createdCell

  1. bScrollInfinite: según el desarrollador:

La capacidad integrada de DataTables 1.9 para mostrar un desplazamiento infinito La cuadrícula a través de la opción bScrollInfinite se ha eliminado debido a la inconsistencias que causaba en la API. La eliminación también ha ayudado simplemente el código interno significativamente.

Una extensión que se conoce con el nombre de Scroller está disponible como alternativa.

  1. Ahorro de estado basado en cookies:

El ahorro de estado basado en cookies ha sido reemplazado por almacenamiento basado en local ahorro de estado en DataTables 1.10. Cookie’s, con su límite de 4KiB eran muy limitados, e incurrieron en una penalización de desempeño ya que formaban parte de cada petición HTTP. localStorage es mucho más rápido y más flexible, y se utiliza como almacenamiento predeterminado para la información de estado en DataTables 1.10.

  1. Control de paginación dos_botones:

DataTables 1.10 ha mejorado significativamente los controles de paginación de DataTables (ver pagingType), una consecuencia de lo cual es que la antigua forma de paginación integrada de dos botones ha sido remoto.
