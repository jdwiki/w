Sintaxis #

  • comando </ruta/al/archivo # Redirigir la entrada estándar al archivo
  • comando >/ruta/al/archivo # Redirigir la salida estándar para volar
  • comando file_descriptor>/ruta/al/archivo # Redirigir la salida de file_descriptor al archivo
  • comando >&file_descriptor # Redirigir la salida a file_descriptor
  • comando file_descriptor>&another_file_descriptor # Redirigir file_descriptor a another_file_descriptor
  • comando <&file_descriptor # Redirigir file_descriptor a la entrada estándar
  • comando &>/ruta/al/archivo # Redirigir la salida estándar y el error estándar al archivo

Parámetros #

Parámetro Detalles
descriptor de archivo interno un entero
dirección Uno de >, < o <>
ruta o descriptor de archivo externo & seguido de un número entero para el descriptor de archivo o una ruta.

Los programas de la consola de UNIX tienen un archivo de entrada y dos archivos de salida (los flujos de entrada y salida, así como los dispositivos, son tratados como archivos por el sistema operativo). Por lo general, estos son el teclado y la pantalla, respectivamente, pero cualquiera de ellos o todos pueden redirigirse venir de, o ir a, un archivo u otro programa.

STDIN es una entrada estándar y es cómo el programa recibe una entrada interactiva. A STDIN generalmente se le asigna el descriptor de archivo 0.

STDOUT es la salida estándar. Todo lo que se emite en STDOUT se considera el “resultado” del programa. STDOUT generalmente se le asigna el descriptor de archivo 1.

STDERR es donde se muestran los mensajes de error. Por lo general, cuando se ejecuta un programa desde la consola, aparece STDERR en la pantalla y no se distingue de STDOUT. A STDERR generalmente se le asigna el descriptor de archivo 2.

El orden de redirección es importante

command > file 2>&1

Redirige ambos (STDOUT y STDERR) al archivo.

command 2>&1 > file

Redirige solo STDOUT, porque el descriptor de archivo 2 se redirige al archivo al que apunta el descriptor de archivo 1 (que aún no es el archivo archivo cuando se evalúa la instrucción).

Cada comando en una canalización tiene su propio STDERR (y STDOUT) porque cada uno es un proceso nuevo. Esto puede generar resultados sorprendentes si espera que una redirección afecte a toda la canalización. Por ejemplo, este comando (envuelto para legibilidad):

$ python -c 'import sys;print >> sys.stderr, "Python error!"' \
| cut -f1 2>> error.log

imprimirá “¡Error de Python!” a la consola en lugar del archivo de registro. En su lugar, adjunte el error al comando que desea capturar:

$ python -c 'import sys;print >> sys.stderr, "Python error!"' 2>> error.log \
| cut -f1

Agregar vs Truncar #

Truncar > #

  1. Cree el archivo especificado si no existe.

  2. Truncar (eliminar el contenido del archivo)

  3. Escribir en el archivo

    $ echo “first line” > /tmp/lines $ echo “second line” > /tmp/lines

    $ cat /tmp/lines second line

Añadir >> #

  1. Cree el archivo especificado si no existe.

  2. Adjuntar archivo (escribir al final del archivo).

    Overwrite existing file #

    $ echo “first line” > /tmp/lines

    Append a second line #

    $ echo “second line” » /tmp/lines

    $ cat /tmp/lines first line second line

Redirigir STDOUT y STDERR #

Los descriptores de archivo como 0 y 1 son punteros. Cambiamos a qué descriptores de archivo apuntan con la redirección. >/dev/null significa que 1 apunta a /dev/null.

Primero apuntamos 1 (STDOUT) a /dev/null y luego apuntamos 2 (STDERR) a lo que 1 apunte.

# STDERR is redirect to STDOUT: redirected to /dev/null,
# effectually redirecting both STDERR and STDOUT to /dev/null
echo 'hello' > /dev/null 2>&1

Esto puede reducirse aún más a lo siguiente:

echo 'hello' &> /dev/null

Sin embargo, esta forma puede ser indeseable en producción si la compatibilidad de shell es una preocupación, ya que entra en conflicto con POSIX, introduce ambigüedad de análisis y los shells sin esta función lo malinterpretarán:

# Actual code
echo 'hello' &> /dev/null
echo 'hello' &> /dev/null 'goodbye'

# Desired behavior
echo 'hello' > /dev/null 2>&1
echo 'hello' 'goodbye' > /dev/null 2>&1

# Actual behavior
echo 'hello' &
echo 'hello' & goodbye > /dev/null

NOTA: Se sabe que &> funciona como se desea tanto en Bash como en Zsh.

Redirigir la salida estándar #

> redirige la salida estándar (también conocida como STDOUT) del comando actual a un archivo u otro descriptor.

Estos ejemplos escriben la salida del comando ls en el archivo file.txt

ls >file.txt
> file.txt ls

El archivo de destino se crea si no existe; de ​​lo contrario, este archivo se trunca.

El descriptor de redirección predeterminado es la salida estándar o 1 cuando no se especifica ninguno. Este comando es equivalente a los ejemplos anteriores con la salida estándar indicada explícitamente:

ls 1>file.txt

Nota: la redirección es inicializada por el shell ejecutado y no por el comando ejecutado, por lo tanto, se realiza antes de la ejecución del comando.

Uso de canalizaciones con nombre #

A veces, es posible que desee generar algo mediante un programa e ingresarlo en otro programa, pero no puede usar una tubería estándar.

ls -l | grep ".log"

Simplemente podría escribir en un archivo temporal:

touch tempFile.txt
ls -l > tempFile.txt
grep ".log" < tempFile.txt

Esto funciona bien para la mayoría de las aplicaciones, sin embargo, nadie sabrá qué hace tempFile y alguien podría eliminarlo si contiene la salida de ls -l en ese directorio. Aquí es donde entra en juego una tubería con nombre:

mkfifo myPipe
ls -l > myPipe
grep ".log" < myPipe

myPipe es técnicamente un archivo (todo está en Linux), así que hagamos ls -l en un directorio vacío en el que acabamos de crear una tubería:

mkdir pipeFolder
cd pipeFolder
mkfifo myPipe
ls -l

La salida es:

prw-r--r-- 1 root root 0 Jul 25 11:20 myPipe

Observe el primer carácter en los permisos, aparece como una canalización, no como un archivo.

Ahora hagamos algo genial.

Abra una terminal y tome nota del directorio (o cree uno para que la limpieza sea fácil) y haga una tubería.

mkfifo myPipe

Ahora pongamos algo en la tubería.

echo "Hello from the other side" > myPipe

Notarás que esto cuelga, el otro lado de la tubería aún está cerrado. Abramos el otro lado de la tubería y dejemos pasar eso.

Abra otra terminal y vaya al directorio en el que se encuentra la tubería (o, si lo sabe, antepóngalo a la tubería):

cat < myPipe

Notarás que después de la salida hola desde el otro lado, el programa en la primera terminal finaliza, al igual que en la segunda terminal.

Ahora ejecute los comandos a la inversa. Comience con cat < myPipe y luego haga eco de algo en él. Todavía funciona, porque un programa esperará hasta que se coloque algo en la tubería antes de terminar, porque sabe que tiene que obtener algo.

Las canalizaciones con nombre pueden ser útiles para mover información entre terminales o entre programas.

Las tuberías son pequeñas. Una vez lleno, el escritor se bloquea hasta que algún lector lee el contenido, por lo que debe ejecutar el lector y el escritor en diferentes terminales o ejecutar uno u otro en segundo plano:

 ls -l /tmp > myPipe &
 cat < myPipe

Más ejemplos usando canalizaciones con nombre:

  • Ejemplo 1: todos los comandos en el mismo terminal/mismo shell

     $ { ls -l && cat file3; } >mypipe &
 $ cat <mypipe    
 # Output: Prints ls -l data and then prints file3 contents on screen

  • Ejemplo 2: todos los comandos en el mismo terminal/mismo shell

     $ ls -l >mypipe &
 $ cat file3 >mypipe &
 $ cat <mypipe
 #Output: This prints on screen the contents of mypipe.

Tenga en cuenta que primero se muestran los contenidos de file3 y luego se muestran los datos ls -l (configuración LIFO).

  • Ejemplo 3: todos los comandos en el mismo terminal/mismo shell

     $ { pipedata=$(<mypipe) && echo "$pipedata"; } &
 $ ls >mypipe 
 # Output: Prints the output of ls directly on screen

    Mind that the variable $pipedata is not available for usage in the main terminal / main shell since the use of & invokes a subshell and $pipedata was only available in this subshell.

  • Ejemplo 4: todos los comandos en el mismo terminal/mismo shell

     $ export pipedata
 $ pipedata=$(<mypipe) &
 $ ls -l *.sh >mypipe
 $ echo "$pipedata"   
 #Output : Prints correctly the contents of mypipe

Esto imprime correctamente el valor de la variable $pipedata en el shell principal debido a la declaración de exportación de la variable. El terminal principal/shell principal no se bloquea debido a la invocación de un shell de fondo (&).

Redirigir múltiples comandos al mismo archivo #

{
  echo "contents of home directory"
  ls ~
} > output.txt

Imprimir mensajes de error en stderr #

Los mensajes de error generalmente se incluyen en un script con fines de depuración o para proporcionar una experiencia de usuario enriquecida. Simplemente escribiendo un mensaje de error como este:

cmd || echo 'cmd failed'

puede funcionar para casos simples, pero no es la forma habitual. En este ejemplo, el mensaje de error contaminará la salida real de la secuencia de comandos al mezclar tanto los errores como la salida correcta en stdout.

En resumen, el mensaje de error debe ir a stderr y no a stdout. Es bastante simple:

cmd || echo 'cmd failed' >/dev/stderr

Otro ejemplo:

if cmd; then
    echo 'success'
else
    echo 'cmd failed' >/dev/stderr
fi

En el ejemplo anterior, el mensaje de éxito se imprimirá en stdout mientras que el mensaje de error se imprimirá en stderr.

Una mejor manera de imprimir un mensaje de error es definir una función:

err(){
    echo "E: $*" >>/dev/stderr
}

Ahora, cuando tengas que imprimir un error:

err "My error message"

Redirección a direcciones de red #

Bash trata algunas rutas como especiales y puede realizar alguna comunicación de red escribiendo en /dev/{udp|tcp}/host/port. Bash no puede configurar un servidor de escucha, pero puede iniciar una conexión, y TCP puede leer los resultados al menos.

Por ejemplo, para enviar una solicitud web simple, se podría hacer:

exec 3</dev/tcp/www.google.com/80
printf 'GET / HTTP/1.0\r\n\r\n' >&3
cat <&3

y los resultados de la página web predeterminada de www.google.com se imprimirán en stdout.

Similarmente

printf 'HI\n' >/dev/udp/192.168.1.1/6666

enviaría un mensaje UDP que contiene HI\n a un oyente en 192.168.1.1:6666

Redirigir STDIN #

< lee desde su argumento derecho y escribe en su argumento izquierdo.

Para escribir un archivo en STDIN debemos leer /tmp/a_file y escribir en STDIN, es decir, 0</tmp/a_file

Nota: el descriptor de archivo interno tiene por defecto 0 (STDIN) para <

$ echo "b" > /tmp/list.txt
$ echo "a" >> /tmp/list.txt
$ echo "c" >> /tmp/list.txt
$ sort < /tmp/list.txt
a
b
c

Redirigir STDERR #

2 es STDERR.

$ echo_to_stderr 2>/dev/null # echos nothing

Definiciones:

echo_to_stderr es un comando que escribe "stderr" en STDERR

echo_to_stderr () {
    echo stderr >&2
}

$ echo_to_stderr
stderr

STDIN, STDOUT y STDERR explicados #

Los comandos tienen una entrada (STDIN) y dos tipos de salidas, salida estándar (STDOUT) y error estándar (STDERR).

Por ejemplo:

ESTÁNDAR

root@server~# read
Type some text here

La entrada estándar se utiliza para proporcionar entrada a un programa. (Aquí estamos usando el read incorporado para leer una línea de STDIN).

SALIDAESTÁNDAR

root@server~# ls file
file

La salida estándar generalmente se usa para la salida “normal” de un comando. Por ejemplo, ls enumera los archivos, por lo que los archivos se envían a STDOUT.

STDERR

root@server~# ls anotherfile
ls: cannot access 'anotherfile': No such file or directory

El error estándar se usa (como su nombre lo indica) para los mensajes de error. Dado que este mensaje no es una lista de archivos, se envía a STDERR.

STDIN, STDOUT y STDERR son los tres flujos estándar. Se identifican en el shell mediante un número en lugar de un nombre:

0 = Estándar en 1 = salida estándar 2 = error estándar

De manera predeterminada, STDIN está adjunto al teclado y tanto STDOUT como STDERR aparecen en la terminal. Sin embargo, podemos redirigir STDOUT o STDERR a lo que necesitemos. Por ejemplo, digamos que solo necesita la salida estándar y todos los mensajes de error impresos en error estándar deben suprimirse. Ahí es cuando usamos los descriptores 1 y 2.

Redirigiendo STDERR a /dev/null Tomando el ejemplo anterior,

root@server~# ls anotherfile 2>/dev/null
root@server~#

En este caso, si hay algún STDERR, será redirigido a /dev/null (un archivo especial que ignora cualquier cosa que se introduzca en él), por lo que no obtendrá ningún resultado de error en el shell.
