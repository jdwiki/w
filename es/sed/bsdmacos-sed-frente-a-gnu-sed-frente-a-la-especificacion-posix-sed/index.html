BSDmacOS Sed frente a GNU Sed frente a la especificación POSIX Sed

  • Reemplace todas las nuevas líneas con tabulaciones

  • Agrega texto literal a una línea con la función ‘a’

Para citar la solicitud de creación de tema de @SnoringFrog:

“Uno de los mayores problemas con el uso de sed son los scripts que fallan (o tienen éxito de una manera inesperada) porque fueron escritos para uno y no para el otro. Sería bueno un simple resumen de las diferencias más importantes”.

macOS usa la versión BSD de sed[1], que difiere en muchos aspectos de la versión GNU sed que viene con las distribuciones Linux.

Su denominador común es la funcionalidad decretada por POSIX: consulte la especificación sed de POSIX

El enfoque más portátil es usar solo funciones POSIX, que, sin embargo, limita la funcionalidad:

  • En particular, POSIX especifica soporte solo para expresiones regulares basic, que tienen muchas limitaciones (p. ej., no hay soporte para | (alternancia), no hay soporte directo para + y ?) y diferentes formas de escape. requisitos

  • Advertencia: GNU sed (sin -r), does support \|, \+ and \?, que NO es compatible con POSIX; use --posix para deshabilitar (ver más abajo).

  • Para usar funciones POSIX únicamente:

  • (ambas versiones): use solo las opciones -n y -e (en particular, no use -E o -r para activar el soporte para expresiones regulares extendidas)

  • GNU sed: agregue la opción --posix para garantizar la funcionalidad exclusiva de POSIX (no la necesita estrictamente, pero sin ella podría terminar sin darse cuenta usando funciones que no son POSIX sin darse cuenta; advertencia: --posix en sí mismo no cumple con POSIX)

  • El uso de funciones exclusivas de POSIX significa requisitos de formato más estrictos (renunciando a muchas comodidades disponibles en GNU sed):

    • Control-character sequences such as \n and \t are generally NOT supported.
    • Labels and branching commands (e.g., b) must be followed by an actual newline or continuation via a separate -e option.
    • See below for details.

Sin embargo, ambas versiones implementan extensiones al estándar POSIX:

  • qué extensiones implementan difieren (GNU sed implementa más).
  • incluso aquellas extensiones que ambos implementan parcialmente difieren en la sintaxis.

Si necesita admitir AMBAS plataformas (discusión de diferencias):

  • Características incompatibles:

  • El uso de la opción -i sin un argumento (actualización in situ sin copia de seguridad) es incompatible:

    • BSD sed: MUST use -i ''
    • GNU sed: MUST use just -i (equivalent: -i'') - using -i '' does NOT work.

  • -i sensatamente activa la numeración de línea per-input-file en GNU sed y recent versiones de BSD sed (por ejemplo, en FreeBSD 10), pero **NO en macOS a partir de 10.12 **. Tenga en cuenta que en ausencia de -i todas las versiones numeran líneas acumulativamente en los archivos de entrada.

  • Si la última línea de entrada no tiene una nueva línea final (y se imprime):

    • BSD sed: always appends a newline on output, even if the input line doesn’t end in one.
    • GNU sed: preserves the trailing-newline status, i.e., it appends a newline only if the input line ended in one.

  • **Características comunes:

  • Si restringe sus scripts sed a lo que admite BSD sed, por lo general también funcionarán en GNU sed, con la notable excepción de usar funciones de expresiones regulares extendidas específicas de la plataforma con -E. Obviamente, también renunciará a las extensiones que son específicas de la versión GNU. Consulte la siguiente sección.

Pautas para compatibilidad multiplataforma (OS X/BSD, Linux), impulsadas por los requisitos más estrictos de la versión BSD:

Tenga en cuenta que las abreviaturas macOS y Linux se usan ocasionalmente a continuación para referirse a las versiones BSD y GNU de sed, respectivamente, porque son las versiones stock en cada plataforma. Sin embargo, es posible instalar GNU sed en macOS, por ejemplo, usando Homebrew con brew install gnu-sed.

Nota: Excepto cuando se usan los indicadores -r y -E (regexes_extendidas), las instrucciones a continuación equivalen a escribir secuencias de comandos sed compatibles con POSIX.

  • Para el cumplimiento de POSIX, debe limitarse a POSIX BRE (expresiones regulares básicas), que son, lamentablemente, como su nombre indica, bastante básico. Advertencia: no asuma que \|, \+ y \? son compatibles: Mientras que GNU sed los admite (a menos que se use --posix), BSD sed sí lo hace no: estas funciones no cumplen con POSIX. Mientras que \+ y \? pueden emularse de manera compatible con POSIX: \{1,\} para \+, \{0,1\} para \?, \| (alternancia) cannot, desafortunadamente.

  • Para expresiones regulares más potentes, use -E (en lugar de -r) para admitir ERE (expresiones regulares_extendidas) (GNU sed no documenta -E, pero funciona allí como un alias de -r; la versión más nueva de BSD sed, como en FreeBSD 10, ahora también es compatible con -r, pero la versión de macOS a partir de 10.12 no). Advertencia: Aunque el uso de -r / -E significa que su comando es, por definición, no compatible con POSIX, aún debe limitarse a [POSIX ERE (expresiones regulares extendidas)](http ://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap09.html#tag_09_04). Lamentablemente, esto significa que no podrá usar varias construcciones útiles, en particular:

  • aserciones de límites de palabras, porque son específicas de la plataforma (por ejemplo, \< en Linux, [[:<]] en OS X).

  • referencias inversas dentro de expresiones regulares (a diferencia de las “referencias inversas” a coincidencias de grupos de captura en la cadena de reemplazo de las llamadas a funciones s), porque BSD sed no las admite en expresiones regulares extendidas (pero, curiosamente, lo hace en los básicos, donde son mandatorios POSIX).

  • Secuencias de escape de caracteres de control como \n y \t:

  • En regexes (tanto en patrones para la selección de línea como en el primer argumento de la función s), suponga que solo \n se reconoce como una secuencia de escape (rara vez se usa, ya que el espacio del patrón suele ser un single line (sin terminar \n), pero no dentro de una character class, por lo que, por ejemplo, [^\n] no funciona; (si su entrada no contiene caracteres de control, aparte de \t , puede emular [^\n] con [[:print:][:blank:]]; de lo contrario, empalme los caracteres de control como literals[2]) - ** generalmente, incluye caracteres de control como literales, ya sea a través de ANSI C-quoted strings (por ejemplo, $'\t') en shells que lo admiten (bash, ksh, zsh), o mediante sustituciones de comandos usando printf (por ejemplo, "$(printf '\t')")**.

    • Linux only:
      sed 's/\t/-/' <<<$'a\tb' # -> 'a-b'
    • OSX and Linux:
      sed 's/'$'\t''/-/' <<<$'a\tb' # ANSI C-quoted string
      sed 's/'"$(printf '\t')"'/-/' <<<$'a\tb' # command subst. with printf

  • En cadenas de reemplazo usadas con el comando s, suponga que NO se admiten secuencias de escape de caracteres de control, así que, de nuevo, incluya caracteres de control. como literales, como arriba.

    • Linux only:
      sed 's/-/\t/' <<<$'a-b' # -> 'a<tab>b'
    • macOS and Linux:
      sed 's/-/'$'\t''/' <<<'a-b' sed 's/-/'"$(printf '\t')"'/' <<<'a-b'

  • Lo mismo ocurre con los argumentos de texto de las funciones i y a: no utilice secuencias de caracteres de control; consulte a continuación.

  • Etiquetas y bifurcaciones: las etiquetas, así como el nombre de la etiqueta argumento para las funciones b y t deben ir seguidas de una nueva línea literal o de un empalme $'\n' . Alternativamente, use múltiples opciones -e y termine cada una justo después del nombre de la etiqueta.

  • Solo Linux: sed -n '/a/ bLBL; d; :LBL p' <<<$'a\nb' # -> 'a'

  • macOS y Linux:

    • EITHER (actual newlines):
      sed -n '/a/ bLBL d; :LBL p' <<<$'a\nb'
    • OR (spliced-in $\n instances):
      sed -n '/a/ bLBL'$'\n''d; :LBL'$'\n'p' <<<$'a\nb'
    • OR (multiple -e options):
      sed -n -e '/a/ bLBL' -e 'd; :LBL' -e 'p' <<<$'a\nb'

  • Funciones i y a para insertar/agregar texto: siga el nombre de la función por \, seguido de una nueva línea literal o un empalme $'\n' antes de especificar el argumento de texto.

  • Solo Linux: sed '1 i new first line' <<<$'a\nb' # -> 'new first line<nl>a<nl>b'

  • OSX y Linux: sed -e '1 i\'$'\n''new first line' <<<$'a\nb'

  • Nota:

    • Without -e, the text argument is inexplicably not newline-terminated on output on macOS (bug?).
    • Do not use control-character escapes such as \n and \t in the text argument, as they’re only supported on Linux.
    • If the text argument therefore has actual interior newlines, \-escape them.
    • If you want to place additional commands after the text argument, you must terminate it with an (unescaped) newline (whether literal or spliced in), or continue with a separate -e option (this is a general requirement that applies to all versions).

  • Dentro de function lists (múltiples llamadas de función encerradas en {...}), asegúrese de terminar también la función last, antes del cierre }, con ;.

  • Solo Linux:

    • sed -n '1 {p;q}' <<<$'a\nb' # -> 'a'

  • macOS y Linux:

    • sed -n '1 {p;q;}' <<<$'a\nb'

Características específicas de GNU sed que faltan por completo en BSD sed:

Características de GNU que se perderá si necesita admitir ambas plataformas:

  • Varias opciones de sustitución y coincidencia de expresiones regulares (tanto en patrones para la selección de línea como en el primer argumento de la función s):

  • La opción I para la coincidencia de expresiones regulares que distingue entre mayúsculas y minúsculas (increíblemente, BSD sed no admite esto en absoluto).

  • La opción M para la coincidencia de varias líneas (donde ^ / $ coinciden con el inicio/final de cada línea)

  • Para opciones adicionales que son específicas de la función s, consulte https://www.gnu.org/software/sed/manual/sed.html#The-_0022s_0022-Command

  • Secuencias de escape

  • Secuencias de escape relacionadas con la sustitución, como \u en el argumento de reemplazo de la función s/// que permiten la manipulación de subcadenas, dentro de los límites; por ejemplo, sed 's/^./\u&/' <<<'perro' # -> 'Perro' - consulte http://www.gnu.org/software/sed/manual/sed.html#The -_0022s_0022-Comando

  • Secuencias de escape de caracteres de control: además de \n, \t, …, escapes basados ​​en puntos de código; por ejemplo, todos los siguientes escapes (hex., octal, decimal) representan una comilla simple ('): \x27, \o047, \d039 - consulte https://www.gnu. org/software/sed/manual/sed.html#Escapes

  • Extensiones de dirección, como first~step para hacer coincidir cada línea de paso a paso, addr, +N para hacer coincidir N líneas después de addr, … - consulte http://www.gnu .org/software/sed/manual/sed.html#Direcciones

Reemplace todas las nuevas líneas con tabulaciones #

Nota: Para abreviar, los comandos usan here-strings (<<<) y ANSI C-quoted strings ( $'...'). Ambas funciones de shell funcionan en bash, ksh y zsh.

# GNU Sed
$ sed ':a;$!{N;ba}; s/\n/\t/g' <<<$'line_1\nline_2\nline_3'
line_1 line_2 line_3

# BSD Sed equivalent (multi-line form)
sed <<<$'line_1\nline_2\nline_3' '
:a
$!{N;ba
}; s/\n/'$'\t''/g'

# BSD Sed equivalent (single-line form, via separate -e options)
sed -e ':a' -e '$!{N;ba' -e '}; s/\n/'$'\t''/g' <<<$'line 1\nline 2\nline 3'

Notas BSD Sed:

  • Tenga en cuenta la necesidad de terminar las etiquetas (:a) y los comandos de bifurcación (ba), ya sea con nuevas líneas reales o con opciones separadas de -e.

  • Dado que las secuencias de escape de caracteres de control, como \t, no se admiten en la cadena de reemplazo, se empalma una tabulación literal con comillas ANSI C en la cadena de reemplazo. (En la parte regex, BSD Sed only reconoce \n como una secuencia de escape).

Agrega texto literal a una línea con la función ‘a’ #

Nota: Para abreviar, los comandos usan here-strings (<<<) y ANSI C-quoted strings ( $'...'). Ambas funciones de shell funcionan en bash, ksh y zsh.

 # GNU Sed
 $ sed '1 a appended text' <<<'line 1'
 line 1
 appended text

 # BSD Sed (multi-line form)
 sed '1 a\
 appended text' <<<'line 1'

 # BSD Sed (single-line form via a Bash/Ksh/Zsh ANSI C-quoted string)
 sed $'1 a\\\nappended text' <<<'line 1'

Tenga en cuenta cómo BSD Seed requiere un \ seguido de una nueva línea real para pasar el texto a agregar. Lo mismo se aplica a las funciones i (insertar) y c (eliminar e insertar) relacionadas.
