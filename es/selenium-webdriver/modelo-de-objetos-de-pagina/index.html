  • Modelo de objeto de página de mejores prácticas

Un papel importante en la automatización de sitios web y aplicaciones web implica identificar elementos en la pantalla e interactuar con ellos. Los elementos se encuentran en Selenium mediante el uso de localizadores y la clase Por. Estos localizadores e interacciones se colocan dentro de los objetos de página como práctica recomendada para evitar la duplicación de código y facilitar el mantenimiento. Encapsula WebElements y se supone que contiene comportamiento y devuelve información en la página (o parte de una página en una aplicación web).

El modelo de objetos de página es un patrón en el que escribimos clases orientadas a objetos que sirven como interfaz para una vista particular de la página web. Usamos los métodos de esa clase de página para realizar la acción requerida. Hace unos años, estábamos manipulando directamente el código HTML de la página web en las clases de prueba, lo que era muy difícil de mantener y frágil a los cambios en la interfaz de usuario.

Sin embargo, tener su código organizado en una forma de patrón de objeto de página proporciona una API específica de la aplicación, lo que le permite manipular los elementos de la página sin tener que buscar en el HTML. La regla general básica dice que su objeto de página debe tener todo lo que un humano puede hacer en esa página web. Por ejemplo, para acceder al campo de texto en una página web, debe tener un método allí para obtener el texto y devolver la cadena después de realizar todas las modificaciones.

Algunos puntos importantes que debe tener en cuenta al diseñar los objetos de la página:

  1. Por lo general, el objeto de página no debe compilarse solo para páginas, sino que debe preferir compilarlo para elementos significativos de la página. Por ejemplo, una página que tenga varias pestañas para mostrar diferentes gráficos de sus estudios debe tener el mismo número de páginas que el número de pestañas.

  2. Navegar de una vista a otra debería devolver la instancia de las clases de página.

  3. Los métodos de utilidad que deben estar allí solo para una vista o página web específica deben pertenecer solo a esa clase de página.

  4. Las clases de página no deben cuidar los métodos de aserción, puede tener métodos para devolver valores booleanos pero no los verifique allí. Por ejemplo, para verificar el nombre completo del usuario, puede tener un método para obtener el valor booleano:

     public boolean hasDisplayedUserFullName (String userFullName) {
     return driver.findElement(By.xpath("xpathExpressionUsingFullName")).isDisplayed();
 }

  5. Si su página web está basada en iframe, prefiera tener clases de página para iframes también.

Ventajas del patrón de objeto de página:

  1. Separación limpia entre el código de prueba y el código de página
  2. En caso de cualquier cambio en la interfaz de usuario de la página web, no es necesario cambiar su código en varios lugares. Cambio solo en clases de página.
  3. Sin localizadores de elementos dispersos.
  4. Hace que el código sea más fácil de entender
  5. Fácil mantenimiento

Introducción (usando Java) #

Un ejemplo para realizar una prueba de inicio de sesión basada en el patrón de objeto de página:

import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.FindBy;
import org.openqa.selenium.support.PageFactory;

/**
 * Class which models the view of Sign-In page
 */
public class SignInPage {
    
    @FindBy(id="username")
    private usernameInput;

    @FindBy(id="password")
    private passwordInput;

    @FindBy(id="signin")
    private signInButton;

    private WebDriver driver;

    public SignInPage(WebDriver driver) {
        this.driver = driver;
    }
    
    /**
     * Method to perform login
     */
    public HomePage performLogin(String username, String password) {
        usernameInput.sendKeys(username);
        passwordInput.sendKeys(password);
        signInButton.click();
        return PageFactory.initElements(driver, HomePage.class);
    }
}


import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.FindBy;
import org.openqa.selenium.support.PageFactory;
/**
 * Class which models the view of home page
 */
public class HomePage {
    @FindBy(id="logout")
    private logoutLink;

    private WebDriver driver;

    public HomePage(WebDriver driver) {
        this.driver = driver;
    }
    
    /**
     * Method to log out
     */
    public SignInPage logout() {
        logoutLink.click();
        wait.ForPageToLoad();
        return PageFactory.initElements(driver, SignInPage.class);
    }
}

/**
 * Login test class
 */
public class LoginTest {
    public void testLogin() {
        SignInPage signInPage = new SignInPage(driver);
        HomePage homePage = signInPage.login(username, password);
        signInPage = homePage.logout();
    }
}

C# #

Los objetos de página deben contener comportamiento, devolver información para aserciones y posiblemente un método para el método de estado listo de página en la inicialización. Selenium admite objetos de página mediante anotaciones. En C# es como sigue:

using OpenQA.Selenium;
using OpenQA.Selenium.Support.PageObjects;
using OpenQA.Selenium.Support.UI;
using System;
using System.Collections.Generic;

public class WikipediaHomePage
{
    private IWebDriver driver;
    private int timeout = 10;
    private By pageLoadedElement = By.ClassName("central-featured-logo");

    [FindsBy(How = How.Id, Using = "searchInput")]
    [CacheLookup]
    private IWebElement searchInput;

    [FindsBy(How = How.CssSelector, Using = ".pure-button.pure-button-primary-progressive")]
    [CacheLookup]
    private IWebElement searchButton;

    public ResultsPage Search(string query) 
    {
        searchInput.SendKeys(query);
        searchButton.Click();
    }

    public WikipediaHomePage VerifyPageLoaded() 
    {
        new WebDriverWait(driver, TimeSpan.FromSeconds(timeout)).Until<bool>((drv) =>  return drv.ExpectedConditions.ElementExists(pageLoadedElement));

        return this;
    }
}

notas:

  • CacheLookup guarda el elemento en el caché y guarda la devolución de un nuevo elemento en cada llamada. Esto mejora el rendimiento, pero no es bueno para elementos que cambian dinámicamente.
  • searchButton tiene 2 nombres de clase y no tiene ID, por eso no puedo usar el nombre de la clase o la ID.
  • Verifiqué que mis localizadores me devuelven el elemento que quiero usando Developer Tools (para Chrome), en otros navegadores puedes usar FireBug o similar.
  • El método Search() devuelve otro objeto de página (ResultsPage) ya que el clic de búsqueda lo redirige a otra página.

Modelo de objeto de página de mejores prácticas #

  • Cree archivos separados para encabezado y pie de página (ya que son comunes para todas las páginas y no tiene sentido hacerlos parte de una sola página)
  • Mantenga los elementos comunes (como Buscar/Atrás/Siguiente, etc.) en un archivo separado (La idea es eliminar cualquier tipo de duplicación y mantener la segregación lógica)
  • Para Driver, es una buena idea crear una clase Driver separada y mantener Driver como estático para que se pueda acceder a él en todas las páginas. (Tengo todas mis páginas web extendidas DriverClass)
  • Las funciones utilizadas en PageObjects se dividen en el fragmento más pequeño posible, teniendo en cuenta la frecuencia y la forma en que se llamarán (la forma en que lo ha hecho para iniciar sesión, aunque el inicio de sesión se puede dividir en las funciones enterUsername y enterPassword, pero manteniendo como función de inicio de sesión es más lógico porque en la mayoría de los casos, se llamaría a la función de inicio de sesión en lugar de llamadas separadas a las funciones enterUsername y enterPassword)
  • El uso de PageObjects en sí segrega el script de prueba de los elementLocators
  • Tener funciones de utilidad en una carpeta de utilidades separada (como DateUtil, excelUtils, etc.)
  • Tener configuraciones en una carpeta conf separada (como configurar el entorno en el que se deben ejecutar las pruebas, configurar las carpetas de entrada y salida)
  • Incorporar captura de pantalla en caso de falla
  • Tenga una variable de espera estática en DriverClass con un tiempo de espera implícito como lo ha hecho Intente siempre tener esperas condicionales en lugar de esperas estáticas como: esperar hasta (Condiciones esperadas). Esto asegura que la espera no ralentice la ejecución innecesariamente.
