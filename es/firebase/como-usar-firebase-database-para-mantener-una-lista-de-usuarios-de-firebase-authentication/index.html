Cómo guardar los datos del perfil de usuario #

Cada usuario autenticado tiene un “uid” de Firebase que es único en todos los proveedores y se devuelve como resultado de cada método de autenticación.

Una buena manera de almacenar los datos de sus usuarios es crear un nodo para guardar todos los datos de los usuarios y protegerlos usando sus reglas de seguridad.

- Base de datos

{
   "users": {
      "uid1" : {
         "name": "Steve",
         "surname": "Jobs"
      },
      "uid2" : {
         "name": "Bill",
         "surname": "Gates"
      }
   }
}

- Seguridad

{
    "rules": {
        "users": {
            "$uid": {
                // If node's key matches the id of the auth user
                ".write": "$uid == auth.uid"
            }
        }
    }
}

El $uid en las reglas anteriores es la llamada “variable de dólar”, que garantiza que las reglas que se encuentran debajo se apliquen a todos los nodos secundarios de los usuarios. Para obtener más información, consulte la documentación sobre Uso de variables $ para capturar segmentos de ruta.

¿Por qué guardar datos de usuario en la base de datos? #

Autenticación de Firebase permite a los usuarios de su aplicación iniciar sesión con proveedores sociales o su correo electrónico y contraseña. Pero, ¿qué sucede si desea almacenar información adicional sobre un usuario, más allá de lo que Firebase Authentication le permite especificar?

¿O qué sucede si desea mostrar una lista de los usuarios en su aplicación? Firebase Authentication no tiene una API para esto.

La mayoría de los desarrolladores resuelven este problema almacenando la información adicional en una base de datos separada. Este tema cubre cómo almacenar dicha información en Firebase Realtime Database.

Manejo de datos de cuentas de usuario en la base de datos en tiempo real #

El sistema de autenticación de Firebase es la fuente de los usuarios uid, displayName, photoURL y tal vez email. Las cuentas basadas en contraseña establecen estos valores persistentes en el sistema de autenticación a través del método .updateProfile. Almacenar estos valores en Realtime Database, rDB, nodo users plantea el problema de los datos obsoletos. Los nombres para mostrar, por ejemplo, pueden cambiar. Para mantener estos valores sincronizados, use almacenamiento local junto con .onAuthStateChange.

en cada .onAuthStateChange

  • getItem('displayName') y getItem('photoURL')
  • comparar con user.displayName y user.photoURL
  • si es diferente
    • setItem('displayName') and setItem('photoURL')
    • db.ref.child('users').update the values of displayName and/or photoURL

.onAuthStateChange se activa en cada carga o recarga de página, así como en cada cambio de estado de autenticación. Potencialmente se dispara a menudo, p. aplicaciones de varias páginas. Sin embargo, leer y escribir en el almacenamiento local es síncrono y muy rápido, por lo que no habrá un impacto notable en el rendimiento de la aplicación.
