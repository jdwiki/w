gen_server es una característica importante de Erlang y requiere algunos requisitos previos para comprender todos los aspectos de esta funcionalidad:

  • Bucle, recursividad y estado
  • Procesos de desove
  • Paso de mensaje
  • Principios de la OTP

Una buena forma de obtener más información sobre una característica de Erlang es leer directamente el código fuente del repositorio oficial de github. El comportamiento de gen_server incorpora mucha información útil y una estructura interesante en su núcleo.

gen_server se define en gen_server.erl y su documentación asociada se puede encontrar en stdlib documentación de Erlang. gen_server es una característica de OTP y también se puede encontrar más información en Principios de diseño de OTP y Guía del usuario.

Frank Hebert también le brinda otra buena introducción a gen_server de su libro en línea gratuito ¡Aprenda algo de Erlang por un gran bien!

Servicio de bienvenida #

Uso del comportamiento de gen_server #

Un gen_server es una máquina de estado finito específica que funciona como un servidor. gen_server puede manejar diferentes tipos de eventos:

  • solicitud síncrona con handle_call
  • solicitud asíncrona con handle_cast
  • otro mensaje (no definido en la especificación OTP) con handle_info

Los mensajes sincrónicos y asincrónicos se especifican en OTP y son tuplas etiquetadas simples con cualquier tipo de datos.

Un gen_server simple se define así:

-module(simple_gen_server).
-behaviour(gen_server).
-export([start_link/0]).
-export([init/1, handle_call/3, handle_cast/2, handle_info/2,
         terminate/2, code_change/3]).

start_link() ->
    Return = gen_server:start_link({local, ?MODULE}, ?MODULE, [], []),
    io:format("start_link: ~p~n", [Return]),
    Return.

init([]) ->
    State = [],
    Return = {ok, State},
    io:format("init: ~p~n", [State]),
    Return.

handle_call(_Request, _From, State) ->
    Reply = ok,
    Return = {reply, Reply, State},
    io:format("handle_call: ~p~n", [Return]),
    Return.

handle_cast(_Msg, State) ->
    Return = {noreply, State},
    io:format("handle_cast: ~p~n", [Return]),
    Return.

handle_info(_Info, State) ->
    Return = {noreply, State},
    io:format("handle_info: ~p~n", [Return]),
    Return.

terminate(_Reason, _State) ->
    Return = ok,
    io:format("terminate: ~p~n", [Return]),
    ok.

code_change(_OldVsn, State, _Extra) ->
    Return = {ok, State},
    io:format("code_change: ~p~n", [Return]),
    Return.

Este código es simple: cada mensaje recibido se imprime en la salida estándar.

Comportamiento del servidor gen #

Para definir un gen_server, debe declararlo explícitamente en su código fuente con -behaviour(gen_server). Tenga en cuenta que “comportamiento” se puede escribir en EE. UU. (comportamiento) o en el Reino Unido (comportamiento).

enlace_inicio/0 #

Esta función es un atajo simple para llamar a otra función: gen_server:start_link/3,4.

enlace_inicio/3,4 #

start_link() ->
    gen_server:start_link({local, ?MODULE}, ?MODULE, [], []).

Esta función se llama cuando desea iniciar su servidor vinculado a un supervisor u otro proceso. start_link/3,4 puede registrar automáticamente su proceso (si cree que su proceso debe ser único) o simplemente puede generarlo como un proceso simple. Cuando se llama, esta función ejecuta init/1.

Esta función puede devolver estos valores definidos:

  • {bien, Pid}
  • ignorar
  • {error,Error}

iniciar/1 #

init([]) ->
    State = [],
    {ok, State}.

init/1 es la primera función que se ejecuta cuando se inicia su servidor. Este inicializa todos los requisitos previos de su aplicación y devuelve el estado al proceso recién creado.

Esta función puede devolver solo estos valores definidos:

  • {ok, Estado}
  • {ok, Estado, Tiempo de espera}
  • {ok, estado, hibernación}
  • {detener, Razón}
  • ignorar

La variable State puede ser cualquier cosa (por ejemplo, lista, tupla, proplists, mapa, registro) y permanece accesible para todas las funciones dentro del proceso generado.

manejar_llamada/3 #

handle_call(_Request, _From, State) ->
    Reply = ok,
    {reply, Reply, State}.

gen_server:call/2 ejecuta esta devolución de llamada. El primer argumento es su mensaje (_Request), el segundo es el origen de la solicitud (_From) y el último es el estado actual (State) del comportamiento de su gen_server en ejecución.

Si desea una respuesta a la persona que llama, handle_call/3 debe devolver una de estas estructuras de datos:

  • {respuesta,Respuesta,NuevoEstado}
  • {respuesta,Respuesta,NuevoEstado,Tiempo de espera}
  • {respuesta,Respuesta,NuevoEstado,hibernación}

Si no desea responder a la persona que llama, handle_call/3 debe devolver una de estas estructuras de datos:

  • {respuesta,NuevoEstado}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

Si desea detener la ejecución actual de su gen_server actual, handle_call/3 debe devolver una de estas estructuras de datos:

  • {detener, Motivo, Responder, Nuevo Estado}
  • {parar,Razón,NuevoEstado}

handle_cast/2 #

handle_cast(_Msg, State) ->
    {noreply, State}.

gen_server:cast/2 ejecuta esta devolución de llamada. El primer argumento es su mensaje (_Msg) y el segundo, el estado actual del comportamiento de su gen_server en ejecución.

De forma predeterminada, esta función no puede enviar datos a la persona que llama, por lo que solo tiene dos opciones, continuar con la ejecución actual:

  • {respuesta,NuevoEstado}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

O detenga su proceso gen_server actual:

  • {parar,Razón,NuevoEstado}

manejar_info/2 #

handle_info(_Info, State) ->
    {noreply, State}.

handle_info/2 se ejecuta cuando un mensaje OTP no estándar proviene del mundo exterior. Este no puede responder y, como handle_cast/2, solo puede hacer 2 acciones, continuando con la ejecución actual:

  • {respuesta,NuevoEstado}
  • {noreply,NewState,Timeout}
  • {noreply,NewState,hibernate}

O detenga el proceso gen_server en ejecución actual:

  • {parar,Razón,NuevoEstado}

terminar/2 #

terminate(_Reason, _State) ->
    ok.

Se llama a terminate/2 cuando ocurre un error o cuando desea cerrar su proceso gen_server.

cambio_de_código/3 #

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

La función code_change/3 se llama cuando desea actualizar su código en ejecución.

Esta función puede devolver solo estos valores definidos:

  • {ok, NuevoEstado}
  • {error, Motivo}

Iniciando este proceso #

Puede compilar su código e iniciar simple_gen_server:

simple_gen_server:start_link().

Si desea enviar un mensaje a su servidor, puede utilizar estas funciones:

% will use handle_call as callback and print:
%   handle_call: mymessage
gen_server:call(simple_gen_server, mymessage).

% will use handle_cast as callback and print:
%   handle_cast: mymessage
gen_server:cast(simple_gen_server, mymessage).

% will use handle_info as callback and print:
%   handle_info: mymessage
erlang:send(whereis(simple_gen_server), mymessage).

Base de datos simple de clave/valor #

Este código fuente crea un servicio simple de almacén de clave/valor basado en la estructura de datos de map Erlang. En primer lugar, necesitamos definir toda la información relativa a nuestro gen_server:

-module(cache).
-behaviour(gen_server).

% our API
-export([start_link/0]).
-export([get/1, put/2, state/0, delete/1, stop/0]).

% our handlers
-export([init/1, handle_call/3, handle_cast/2, handle_info/2,
         terminate/2, code_change/3]).

% Defining our function to start `cache` process:

start_link() ->
    gen_server:start_link({local, ?MODULE}, ?MODULE, [], []).

%%%%%%%%%%%%%%
% API

% Key/Value database is a simple store, value indexed by an unique key. 
% This implementation is based on map, this datastructure is like hash 
# in Perl or dictionaries in Python. 

% put/2
% put a value indexed by a key. We assume the link is stable 
% and the data will be written, so, we use an asynchronous call with 
% gen_server:cast/2.

put(Key, Value) ->
    gen_server:cast(?MODULE, {put, {Key, Value}}).

% get/1
% take one argument, a key and will a return the value indexed 
% by this same key. We use a synchronous call with gen_server:call/2.

get(Key) ->
    gen_server:call(?MODULE, {get, Key}).

% delete/1 
% like `put/1`, we assume the data will be removed. So, we use an 
% asynchronous call with gen_server:cast/2.

delete(Key) ->
    gen_server:cast(?MODULE, {delete, Key}).

% state/0 
% This function will return the current state (here the map who contain all 
% indexed values), we need a synchronous call.

state() ->
    gen_server:call(?MODULE, {get_state}).

% stop/0
% This function stop cache server process.

stop() ->
    gen_server:stop(?MODULE).

%%%%%%%%%%%%%%%
% Handlers

% init/1
% Here init/1 will initialize state with simple empty map datastructure.

init([]) ->
    {ok, #{}}.

% handle_call/3
% Now, we need to define our handle. In a cache server we need to get our 
% value from a key, this feature need to be synchronous, so, using 
% handle_call seems a good choice:

handle_call({get, Key}, From, State) ->
    Response = maps:get(Key, State, undefined),
    {reply, Response, State};

% We need to check our current state, like get_fea

handle_call({get_state}, From, State) ->
    Response = {current_state, State},
    {reply, Response, State};

% All other messages will be dropped here.

handle_call(_Request, _From, State) ->
    Reply = ok,
    {reply, Reply, State}.

% handle_cast/2
% put/2 will execute this function.

handle_cast({put, {Key, Value}}, State) ->
    NewState = maps:put(Key, Value, State),
    {noreply, NewState};

% delete/1 will execute this function.

handle_cast({delete, Key}, State) ->
    NewState = maps:remove(Key, State),
    {noreply, NewState};

% All other messages are dropped here.

handle_cast(_Msg, State) ->
    {noreply, State}.

%%%%%%%%%%%%%%%%
% other handlers

% We don't need other features, other handlers do nothing.

handle_info(_Info, State) ->
    {noreply, State}.

terminate(_Reason, _State) ->
    ok.

code_change(_OldVsn, State, _Extra) ->
    {ok, State}.

Usando nuestro servidor de caché #

Ahora podemos compilar nuestro código y comenzar a usarlo con erl.

% compile cache
c(cache).

% starting cache server
cache:start_link().

% get current store
% will return:
%   #{}
cache:state().

% put some data
cache:put(1, one).
cache:put(hello, bonjour).
cache:put(list, []).

% get current store
% will return:
%   #{1 => one,hello => bonjour,list => []}
cache:state().

% delete a value
cache:delete(1).
cache:state().
%   #{1 => one,hello => bonjour,list => []}

% stopping cache server
cache:stop().
