Configuración de NODE_ENV=“producción” #

Las implementaciones de producción variarán de muchas maneras, pero una convención estándar cuando se implementa en producción es definir una variable de entorno llamada NODE_ENV y establecer su valor en “producción”.

Indicadores de tiempo de ejecución #

Cualquier código que se ejecute en su aplicación (incluidos los módulos externos) puede verificar el valor de NODE_ENV:

if(process.env.NODE_ENV === 'production') {
    // We are running in production mode
} else {
   // We are running in development mode
}

Dependencias #

Cuando la variable de entorno NODE_ENV se establece en ‘producción’, todas las devDependencies en su archivo package.json se ignorarán por completo al ejecutar npm install. También puede hacer cumplir esto con un indicador --production:

npm install --production

Para configurar NODE_ENV puede usar cualquiera de estos métodos

método 1: establecer NODE_ENV para todas las aplicaciones de nodo

ventanas:

set NODE_ENV=production

Linux u otro sistema basado en Unix:

export NODE_ENV=production

Esto establece NODE_ENV para la sesión de bash actual, por lo que cualquier aplicación iniciada después de esta declaración tendrá NODE_ENV establecido en producción.

método 2: establecer NODE_ENV para la aplicación actual

NODE_ENV=production node app.js

Esto establecerá NODE_ENV solo para la aplicación actual. Esto ayuda cuando queremos probar nuestras aplicaciones en diferentes entornos.

método 3: crea el archivo .env y úsalo

Esto usa la idea explicada [aquí] (http://stackoverflow.com/a/28821696/620039). Consulte esta publicación para obtener una explicación más detallada.

Básicamente, creas el archivo .env y ejecutas un script bash para configurarlos en el entorno.

Para evitar escribir un script bash, se puede usar el paquete env-cmd para cargar las variables de entorno definidas en el archivo .env.

env-cmd .env node app.js

método 4: Usar el paquete cross-env

Este paquete permite configurar las variables de entorno de una manera para cada plataforma.

Después de instalarlo con npm, puede simplemente agregarlo a su secuencia de comandos de implementación en package.json de la siguiente manera:

"build:deploy": "cross-env NODE_ENV=production webpack"

Administrar la aplicación con el administrador de procesos #

Es una buena práctica ejecutar aplicaciones de NodeJS controladas por administradores de procesos. El administrador de procesos ayuda a mantener la aplicación viva para siempre, reiniciar en caso de falla, recargar sin tiempo de inactividad y simplifica la administración. Los más potentes (como PM2) tienen un equilibrador de carga integrado. PM2 también le permite administrar el registro, la supervisión y la agrupación en clústeres de aplicaciones.

Gestor de procesos PM2 #

Instalación de PM2:

npm instalar pm2 -g

El proceso se puede iniciar en modo de clúster con un equilibrador de carga integrado para distribuir la carga entre los procesos:

pm2 start app.js -i 0 --name "api" (-i es para especificar el número de procesos que generar. Si es 0, el número de proceso se basará en el número de núcleos de la CPU)

Si bien tiene múltiples usuarios en producción, debe tener un solo punto para PM2. Por lo tanto, el comando pm2 debe tener como prefijo una ubicación (para la configuración de PM2), de lo contrario generará un nuevo proceso de pm2 para cada usuario con configuración en el directorio de inicio respectivo. Y será inconsistente.

Uso: PM2_HOME=/etc/.pm2 pm2 iniciar app.js

Implementación usando el administrador de procesos #

El administrador de procesos generalmente se usa en producción para implementar una aplicación nodejs. Las funciones principales de un administrador de procesos son reiniciar el servidor si falla, verificar el consumo de recursos, mejorar el rendimiento del tiempo de ejecución, monitorear, etc.

Algunos de los administradores de procesos populares creados por la comunidad de nodos son forever, pm2, etc.

Siempre #

forever es una herramienta de interfaz de línea de comandos para garantizar que un script determinado se ejecute de forma continua. La interfaz simple de forever lo hace ideal para ejecutar implementaciones más pequeñas de aplicaciones y scripts de Node.js.

forever monitorea su proceso y lo reinicia si falla.

Instale forever globalmente.

$ npm install -g forever

Ejecutar aplicación:

$ forever start server.js

Esto inicia el servidor y proporciona una identificación para el proceso (comienza desde 0).

Reiniciar aplicación :

$ forever restart 0

Aquí 0 es la identificación del servidor.

Detener aplicación:

$ forever stop 0

Similar a reiniciar, 0 es la identificación del servidor. También puede proporcionar la identificación del proceso o el nombre del script en lugar de la identificación proporcionada por siempre.

Para más comandos: https://www.npmjs.com/package/forever

Despliegue usando PM2 #

PM2 es un administrador de procesos de producción para aplicaciones Node.js, que le permite mantener las aplicaciones vivas para siempre y recargarlas sin tiempo de inactividad. PM2 también le permite administrar el registro, la supervisión y la agrupación en clústeres de aplicaciones.

Instale pm2 globalmente.

npm install -g pm2

Luego, ejecute la aplicación node.js usando PM2.

pm2 start server.js --name "my-app"

Los siguientes comandos son útiles cuando se trabaja con PM2.

Enumere todos los procesos en ejecución:

pm2 list

Detener una aplicación:

pm2 stop my-app

Reiniciar una aplicación:

pm2 restart my-app

Para ver información detallada sobre una aplicación:

pm2 show my-app

Para eliminar una aplicación del registro de PM2:

pm2 delete my-app

Uso de diferentes propiedades/configuraciones para diferentes entornos como desarrollo, control de calidad, puesta en escena, etc. #

Las aplicaciones a gran escala a menudo necesitan diferentes propiedades cuando se ejecutan en diferentes entornos. podemos lograr esto pasando argumentos a la aplicación NodeJs y usando el mismo argumento en el proceso del nodo para cargar un archivo de propiedades de entorno específico.

Supongamos que tenemos dos archivos de propiedades para diferentes entornos.


  • dev.json

      {
      "PORT": 3000,
      "DB": {
          "host": "localhost",
          "user": "bob",
          "password": "12345"
      }
  }

  • qa.json

      {
      "PORT": 3001,
      "DB": {
          "host": "where_db_is_hosted",
          "user": "bob",
          "password": "54321"
      }
  }

El siguiente código en la aplicación exportará el archivo de propiedad respectivo que queremos usar.

process.argv.forEach(function (val) {
    var arg = val.split("=");
    if (arg.length > 0) {
        if (arg[0] === 'env') {
            var env = require('./' + arg[1] + '.json');
            exports.prop = env;
        }
    }
});

Damos argumentos a la aplicación como sigue

node app.js env=dev

si estamos usando un administrador de procesos como para siempre, entonces es tan simple como

forever start app.js env=dev

Aprovechando los clústeres #

Una sola instancia de Node.js se ejecuta en un solo hilo. Para aprovechar los sistemas multinúcleo, el usuario a veces querrá iniciar un grupo de procesos de Node.js para manejar la carga.

 var cluster = require('cluster');
    
    var numCPUs = require('os').cpus().length;
    
        if (cluster.isMaster) {
          // In real life, you'd probably use more than just 2 workers,
          // and perhaps not put the master and worker in the same file.
          //
          // You can also of course get a bit fancier about logging, and
          // implement whatever custom logic you need to prevent DoS
          // attacks and other bad behavior.
          //
          // See the options in the cluster documentation.
          //
          // The important thing is that the master does very little,
          // increasing our resilience to unexpected errors.
            console.log('your server is working on ' + numCPUs + ' cores');
            
             for (var i = 0; i < numCPUs; i++) {
                    cluster.fork();
                  }
             
          cluster.on('disconnect', function(worker) {
            console.error('disconnect!');
            //clearTimeout(timeout);
            cluster.fork();
          });
    
          
        } else {
            require('./app.js');
        
        }
