Instalando paquetes #

Introducción #

Paquete es un término usado por npm para denotar herramientas que los desarrolladores pueden usar para sus proyectos. Esto incluye todo, desde bibliotecas y marcos como jQuery y AngularJS hasta ejecutores de tareas como Gulp.js. Los paquetes vendrán en una carpeta normalmente llamada node_modules, que también contendrá un archivo package.json. Este archivo contiene información sobre todos los paquetes, incluidas las dependencias, que son módulos adicionales necesarios para usar un paquete en particular.

Npm usa la línea de comandos para instalar y administrar paquetes, por lo que los usuarios que intenten usar npm deben estar familiarizados con los comandos básicos de su sistema operativo, es decir, atravesar directorios y poder ver el contenido de los directorios.

Instalación de NPM #

Tenga en cuenta que para instalar paquetes, debe tener instalado NPM.

La forma recomendada de instalar NPM es usar uno de los instaladores de la página de descarga de Node.js. Puede verificar si ya tiene instalado node.js ejecutando el comando npm -v o npm version.

Después de instalar NPM a través del instalador de Node.js, asegúrese de buscar actualizaciones. Esto se debe a que NPM se actualiza con más frecuencia que el instalador de Node.js. Para buscar actualizaciones, ejecute el siguiente comando:

npm install npm@latest -g

Como instalar paquetes #

Para instalar uno o más paquetes, use lo siguiente:

npm install <package-name>
# or
npm i <package-name>...

# e.g. to install lodash and express
npm install lodash express

Nota: Esto instalará el paquete en el directorio en el que se encuentra actualmente la línea de comandos, por lo que es importante comprobar si se ha elegido el directorio adecuado

Si ya tiene un archivo package.json en su directorio de trabajo actual y las dependencias están definidas en él, entonces npm install resolverá e instalará automáticamente todas las dependencias enumeradas en el archivo. También puede usar la versión abreviada del comando npm install que es: npm i

Si desea instalar una versión específica de un paquete, use:

npm install <name>@<version>

# e.g. to install version 4.11.1 of the package lodash
npm install lodash@4.11.1

Si desea instalar una versión que coincida con un rango de versión específico, use:

npm install <name>@<version range>

# e.g. to install a version which matches "version >= 4.10.1" and "version < 4.11.1"
# of the package lodash
npm install lodash@">=4.10.1 <4.11.1"

Si desea instalar la última versión, use:

npm install <name>@latest

Los comandos anteriores buscarán paquetes en el repositorio central npm en npmjs.com. Si no desea instalar desde el registro npm, se admiten otras opciones, como:

# packages distributed as a tarball
npm install <tarball file>
npm install <tarball url>

# packages available locally
npm install <local path>

# packages available as a git repository
npm install <git remote url>

# packages available on GitHub
npm install <username>/<repository>

# packages available as gist (need a package.json)
npm install gist:<gist-id>

# packages from a specific repository
npm install --registry=http://myreg.mycompany.com <package name>

# packages from a related group of packages 
# See npm scope
npm install @<scope>/<name>(@<version>)

# Scoping is useful for separating private packages hosted on private registry from
# public ones by setting registry for specific scope
npm config set @mycompany:registry http://myreg.mycompany.com
npm install @mycompany/<package name>

Por lo general, los módulos se instalarán localmente en una carpeta llamada node_modules, que se puede encontrar en su directorio de trabajo actual. Este es el directorio que require() usará para cargar módulos para que estén disponibles para usted.

Si ya creó un archivo package.json, puede usar la opción --save (abreviatura -S) o una de sus variantes para agregar automáticamente el paquete instalado a su package.json como una dependencia . Si alguien más instala su paquete, npm leerá automáticamente las dependencias del archivo package.json e instalará las versiones enumeradas. Tenga en cuenta que aún puede agregar y administrar sus dependencias editando el archivo más tarde, por lo que generalmente es una buena idea realizar un seguimiento de las dependencias, por ejemplo, usando:

npm install --save <name> # Install dependencies 
# or
npm install -S <name> # shortcut version --save 
# or
npm i -S <name>

Para instalar paquetes y guardarlos solo si son necesarios para el desarrollo, no para ejecutarlos, no si son necesarios para ejecutar la aplicación, siga el siguiente comando:

npm install --save-dev <name> # Install dependencies for development purposes
# or
npm install -D <name> # shortcut version --save-dev
# or
npm i -D <name>

Instalando dependencias #

Algunos módulos no solo proporcionan una biblioteca para su uso, sino que también proporcionan uno o más archivos binarios que están destinados a ser utilizados a través de la línea de comandos. Aunque aún puede instalar esos paquetes localmente, a menudo se prefiere instalarlos globalmente para que se puedan habilitar las herramientas de línea de comandos. En ese caso, npm vinculará automáticamente los archivos binarios a las rutas apropiadas (por ejemplo, /usr/local/bin/<name>) para que puedan usarse desde la línea de comandos. Para instalar un paquete globalmente, use:

npm install --global <name>
# or
npm install -g <name>
# or
npm i -g <name>

# e.g. to install the grunt command line tool
npm install -g grunt-cli

Si desea ver una lista de todos los paquetes instalados y sus versiones asociadas en el espacio de trabajo actual, utilice:

npm list
npm list <name>

Agregar un argumento de nombre opcional puede verificar la versión de un paquete específico.

Nota: Si se encuentra con problemas de permisos al intentar instalar un módulo npm globalmente, resista la tentación de ejecutar sudo npm install -g ... para solucionar el problema. Otorgar scripts de terceros para que se ejecuten en su sistema con privilegios elevados es peligroso. El problema de permisos puede significar que tiene un problema con la forma en que se instaló npm. Si está interesado en instalar Node en entornos de usuario en espacio aislado, puede intentar usar nvm.

Si tiene herramientas de compilación u otras dependencias solo de desarrollo (por ejemplo, Grunt), es posible que no desee incluirlas en la aplicación que implementa. Si ese es el caso, querrá tenerlo como una dependencia de desarrollo, que se encuentra en package.json en devDependencies. Para instalar un paquete como una dependencia solo de desarrollo, use --save-dev (o -D).

npm install --save-dev <name> // Install development dependencies which is not included in production 
# or
npm install -D <name>

Verá que el paquete se agrega a devDependencies de su package.json.

Para instalar dependencias de un proyecto node.js descargado/clonado, simplemente puede usar

npm install
# or
npm i

npm leerá automáticamente las dependencias de package.json y las instalará.

NPM detrás de un servidor proxy #

Si su acceso a Internet es a través de un servidor proxy, es posible que deba modificar los comandos de instalación de npm que acceden a repositorios remotos. npm usa un archivo de configuración que se puede actualizar a través de la línea de comando:

npm config set

Puede ubicar la configuración de su proxy desde el panel de configuración de su navegador. Una vez que haya obtenido la configuración del proxy (URL del servidor, puerto, nombre de usuario y contraseña); necesita configurar sus configuraciones de npm de la siguiente manera.

$ npm config set proxy http://<username>:<password>@<proxy-server-url>:<port>
$ npm config set https-proxy http://<username>:<password>@<proxy-server-url>:<port>

Los campos nombre de usuario, contraseña, puerto son opcionales. Una vez que haya configurado estos, su npm install, npm i -g, etc. funcionarán correctamente.

Desinstalar paquetes #

Para desinstalar uno o más paquetes instalados localmente, use:

npm uninstall <package name>

El comando de desinstalación de npm tiene cinco alias que también se pueden usar:

npm remove <package name>
npm rm <package name>
npm r <package name>

npm unlink <package name>
npm un <package name>

Si desea eliminar el paquete del archivo package.json como parte de la desinstalación, use el indicador --save (abreviatura: -S):

npm uninstall --save <package name>
npm uninstall -S <package name>

Para una dependencia de desarrollo, use el indicador --save-dev (abreviatura: -D):

npm uninstall --save-dev <package name>
npm uninstall -D <package name>

Para una dependencia opcional, use el indicador --save-opcional (abreviatura: -O):

npm uninstall --save-optional <package name>
npm uninstall -O <package name>

Para los paquetes que se instalan globalmente, use el indicador --global (abreviatura: -g):

npm uninstall -g <package name>

Establecer una configuración de paquete #

Las configuraciones del paquete Node.js están contenidas en un archivo llamado package.json que puede encontrar en la raíz de cada proyecto. Puede configurar un nuevo archivo de configuración llamando:

npm init

Eso intentará leer el directorio de trabajo actual para la información del repositorio de Git (si existe) y las variables de entorno para intentar autocompletar algunos de los valores de marcador de posición por usted. De lo contrario, proporcionará un cuadro de diálogo de entrada para las opciones básicas.

Si desea crear un package.json con valores predeterminados, use:

npm init --yes
# or
npm init -y

Si está creando un package.json para un proyecto que no va a publicar como un paquete npm (es decir, con el único fin de redondear sus dependencias), puede transmitir esta intención en su package.json archivo:

  1. Opcionalmente, establezca la propiedad privada en verdadero para evitar la publicación accidental.
  2. Opcionalmente, establezca la propiedad license en “SIN LICENCIA” para denegar a otros el derecho a usar su paquete.

Para instalar un paquete y guardarlo automáticamente en su package.json, use:

npm install --save <package>

El paquete y los metadatos asociados (como la versión del paquete) aparecerán en sus dependencias. Si lo guarda como una dependencia de desarrollo (usando --save-dev), el paquete aparecerá en sus devDependencies.

Con este package.json básico, encontrará mensajes de advertencia al instalar o actualizar paquetes, que le indicarán que le falta una descripción y el campo del repositorio. Si bien es seguro ignorar estos mensajes, puede deshacerse de ellos abriendo el paquete.json en cualquier editor de texto y agregando las siguientes líneas al objeto JSON:

[...]
"description": "No description",
"repository": {
  "private": true
},
[...]

Ejecutar guiones #

Puede definir scripts en su package.json, por ejemplo:

{
  "name": "your-package",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "author": "",
  "license": "ISC",
  "dependencies": {},
  "devDependencies": {},
  "scripts": {
    "echo": "echo hello!"
  }
}

Para ejecutar el script echo, ejecute npm run echo desde la línea de comando. Los scripts arbitrarios, como echo arriba, deben ejecutarse con npm run <script name>. npm también tiene varios scripts oficiales que se ejecutan en ciertas etapas de la vida del paquete (como preinstalación). Consulte aquí para obtener una descripción general completa de cómo npm maneja los campos de script.

Los scripts npm se usan con mayor frecuencia para cosas como iniciar un servidor, construir el proyecto y ejecutar pruebas. He aquí un ejemplo más realista:

  "scripts": {
    "test": "mocha tests",
    "start": "pm2 start index.js"
  }

En las entradas de scripts, los programas de línea de comandos como mocha funcionarán cuando se instalen global o localmente. Si la entrada de la línea de comandos no existe en la RUTA del sistema, npm también verificará sus paquetes instalados localmente.

Si sus guiones se vuelven muy largos, se pueden dividir en partes, así:

  "scripts": {
    "very-complex-command": "npm run chain-1 && npm run chain-2",
    "chain-1": "webpack",
    "chain-2": "node app.js"
  }

Versionado semántico básico #

Antes de publicar un paquete, debe versionarlo. npm admite [versiones semánticas] (http://semver.org/), lo que significa que hay versiones de parches, menores y mayores.

Por ejemplo, si su paquete está en la versión 1.2.3 para cambiar la versión, debe:

  1. lanzamiento del parche: npm version patch => 1.2.4
  2. versión menor: npm version minor => 1.3.0
  3. versión principal: npm version major => 2.0.0

También puede especificar una versión directamente con:

npm versión 3.1.4 => 3.1.4

Cuando configura una versión de paquete usando uno de los comandos npm anteriores, npm modificará el campo de versión del archivo package.json, lo confirmará y también creará una nueva etiqueta Git con la versión prefijada con una “v”, como si He emitido el comando:

etiqueta git v3.1.4

A diferencia de otros administradores de paquetes como Bower, el registro npm no depende de la creación de etiquetas Git para cada versión. Pero, si le gusta usar etiquetas, debe recordar presionar la etiqueta recién creada después de actualizar la versión del paquete:

git push origin master (para enviar el cambio a package.json)

git push origin v3.1.4 (para insertar la nueva etiqueta)

O puede hacer esto de una sola vez con:

git push origin maestro --etiquetas

Publicando un paquete #

Primero, asegúrese de haber configurado su paquete (como se indica en https://www.wikiod.com/es/node-js/npm#Establecer una configuración de paquete​). Luego, debe iniciar sesión en npmjs.

Si ya tiene un usuario npm

npm login

Si no tienes un usuario

npm adduser

Para comprobar que tu usuario está registrado en el cliente actual

npm config ls

Después de eso, cuando su paquete esté listo para ser publicado, use

npm publish

Y ya está.

Si necesita publicar una nueva versión, asegúrese de actualizar la versión de su paquete, como se indica en https://www.wikiod.com/es/node-js/npm#Versionado semántico básico De lo contrario, npm no le permitirá publicar el paquete.

{
    name: "package-name",
    version: "1.0.4"
}

Eliminación de paquetes extraños #

Para eliminar paquetes extraños (paquetes que están instalados pero no en la lista de dependencias), ejecute el siguiente comando:

npm prune

Para eliminar todos los paquetes dev, agregue el indicador --production:

npm prune --production

Ámbitos y repositorios #

# Set the repository for the scope "myscope"
npm config set @myscope:registry http://registry.corporation.com

# Login at a repository and associate it with the scope "myscope"
npm adduser --registry=http://registry.corporation.com --scope=@myscope

# Install a package "mylib" from the scope "myscope"
npm install @myscope/mylib

Si el nombre de su propio paquete comienza con @myscope y el alcance “myscope” está asociado con un repositorio diferente, npm publishing cargará su paquete en ese repositorio en su lugar.

También puede conservar estas configuraciones en un archivo .npmrc:

@myscope:registry=http://registry.corporation.com
//registry.corporation.com/:_authToken=xxxxxxxx-xxxx-xxxx-xxxxxxxxxxxxxxx

Esto es útil cuando se automatiza la compilación en un servidor CI, p.

Listado de paquetes actualmente instalados #

Para generar una lista (vista de árbol) de los paquetes actualmente instalados, use

npm list

ls, la y ll son alias del comando list. Los comandos la y ll muestran información ampliada como descripción y repositorio.

Opciones

El formato de respuesta se puede cambiar pasando opciones.

npm list --json
  • json - Muestra información en formato json
  • largo - Muestra información extendida
  • parseable - Muestra una lista analizable en lugar de un árbol
  • global - Muestra paquetes instalados globalmente
  • profundidad - Profundidad máxima de visualización del árbol de dependencia
  • desarrollo/desarrollo - Muestra las dependencias de desarrollo
  • prod/production - Muestra dependencias

Si lo desea, también puede ir a la página de inicio del paquete.

npm home <package name>

Actualización de npm y paquetes #

Dado que npm en sí mismo es un módulo de Node.js, se puede actualizar usándose a sí mismo.

Si el sistema operativo es Windows, debe estar ejecutando el símbolo del sistema como administrador

npm install -g npm@latest

Si desea buscar versiones actualizadas, puede hacerlo:

npm outdated

Para actualizar un paquete específico:

npm update <package name>

Esto actualizará el paquete a la última versión de acuerdo con las restricciones en package.json

En caso de que también desee bloquear la versión actualizada en package.json:

npm update <package name> --save

Bloqueo de módulos a versiones específicas #

De forma predeterminada, npm instala la última versión disponible de los módulos según la [versión semántica] de cada dependencia (https://www.wikiod.com/es/node-js/npm#Versionado semántico básico). Esto puede ser problemático si el autor de un módulo no se adhiere a semver e introduce cambios importantes en una actualización del módulo, por ejemplo.

Para bloquear la versión de cada dependencia (y las versiones de sus dependencias, etc.) a la versión específica instalada localmente en la carpeta node_modules, use

npm shrinkwrap

Esto luego creará un npm-shrinkwrap.json junto con su package.json que enumera las versiones específicas de las dependencias.

Configuración de paquetes instalados globalmente #

Puede usar npm install -g para instalar un paquete “globalmente”. Esto generalmente se hace para instalar un ejecutable que puede agregar a su ruta para ejecutar. Por ejemplo:

npm install -g gulp-cli

Si actualiza su ruta, puede llamar a gulp directamente.

En muchos sistemas operativos, npm install -g intentará escribir en un directorio en el que es posible que su usuario no pueda escribir, como /usr/bin. No debe usar sudo npm install en este caso, ya que existe un posible riesgo de seguridad de ejecutar scripts arbitrarios con sudo y el usuario raíz puede crear directorios en su hogar en los que no puede escribir, lo que hace futuras instalaciones. más difícil.

Puede decirle a npm dónde instalar los módulos globales a través de su archivo de configuración, ~/.npmrc. Esto se llama el prefijo que puede ver con prefijo npm.

prefix=~/.npm-global-modules

Esto usará el prefijo cada vez que ejecute npm install -g. También puede usar npm install --prefix ~/.npm-global-modules para establecer el prefijo cuando realice la instalación. Si el prefijo es el mismo que su configuración, no necesita usar -g.

Para usar el módulo instalado globalmente, debe estar en su camino:

export PATH=$PATH:~/.npm-global-modules/bin

Ahora, cuando ejecute npm install -g gulp-cli, podrá usar gulp.

Nota: Cuando npm install (sin -g), el prefijo será el directorio con package.json o el directorio actual si no se encuentra ninguno en la jerarquía. Esto también crea un directorio node_modules/.bin que tiene los ejecutables. Si desea usar un ejecutable específico para un proyecto, no es necesario usar npm install -g. Puedes usar el que está en node_modules/.bin.

Vinculación de proyectos para una depuración y un desarrollo más rápidos #

La creación de dependencias de proyectos a veces puede ser una tarea tediosa. En lugar de publicar una versión del paquete en NPM e instalar la dependencia para probar los cambios, use npm link. npm link crea un enlace simbólico para que el código más reciente se pueda probar en un entorno local. Esto facilita la prueba de las herramientas globales y las dependencias del proyecto al permitir que se ejecute el código más reciente antes de crear una versión publicada.

Texto de ayuda #

NAME
       npm-link - Symlink a package folder

SYNOPSIS
         npm link (in package dir)
         npm link [<@scope>/]<pkg>[@<version>]

         alias: npm ln

Pasos para vincular dependencias de proyectos #

Al crear el enlace de dependencia, tenga en cuenta que el nombre del paquete es a lo que se hará referencia en el proyecto principal.

  1. CD en un directorio de dependencia (por ejemplo, cd ../my-dep)
  2. enlace npm
  3. CD en el proyecto que va a utilizar la dependencia
  4. npm link my-dep o si tiene espacio de nombre npm link @namespace/my-dep

Pasos para vincular una herramienta global #

  1. CD en el directorio del proyecto (por ejemplo, cd eslint-watch)
  2. enlace npm
  3. Usa la herramienta
  4. esw --tranquilo

Problemas que pueden surgir #

La vinculación de proyectos a veces puede causar problemas si la dependencia o la herramienta global ya están instaladas. npm uninstall (-g) <pkg> y luego ejecutar npm link normalmente resuelve cualquier problema que pueda surgir.
