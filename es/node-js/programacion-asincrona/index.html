Node es un lenguaje de programación donde todo podría ejecutarse de forma asíncrona. A continuación puede encontrar algunos ejemplos y las cosas típicas del trabajo asíncrono.

Sintaxis #

  • hacerAlgo([args], function([argsCB]) { /* hacer algo cuando termine */});
  • hacerAlgo([args], ([argsCB]) => { /* hacer algo cuando termine */ });

Funciones de devolución de llamada #

Funciones de devolución de llamada en JavaScript #

Las funciones de devolución de llamada son comunes en JavaScript. Las funciones de devolución de llamada son posibles en JavaScript porque las funciones son ciudadanos de primera clase.

Devoluciones de llamadas síncronas. #

Las funciones de devolución de llamada pueden ser síncronas o asíncronas. Dado que las funciones de devolución de llamada asíncronas pueden ser más complejas, aquí hay un ejemplo simple de una función de devolución de llamada síncrona.

// a function that uses a callback named `cb` as a parameter
function getSyncMessage(cb) {
    cb("Hello World!");
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getSyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

La salida para el código anterior es:

> Before getSyncMessage call
> Hello World!
> After getSyncMessage call

Primero veremos cómo se ejecuta el código anterior. Esto es más para aquellos que aún no entienden el concepto de devoluciones de llamadas, si ya lo entiendes, no dudes en omitir este párrafo. Primero se analiza el código y luego lo primero que sucede es que se ejecuta la línea 6, que genera Before getSyncMessage call en la consola. Luego se ejecuta la línea 8 que llama a la función getSyncMessage enviando una función anónima como argumento para el parámetro llamado cb en la función getSyncMessage. La ejecución ahora se realiza dentro de la función getSyncMessage en la línea 3 que ejecuta la función cb que acaba de pasar, esta llamada envía una cadena de argumento “Hello World” para el parámetro llamado message en la función anónima pasada. La ejecución luego va a la línea 9 que registra Hello World! en la consola. Luego, la ejecución pasa por el proceso de salir de la pila de llamadas (ver también) llegando a la línea 10, luego a la línea 4 y finalmente de regreso a la línea 11.

Alguna información para saber sobre las devoluciones de llamadas en general:

  • La función que envía a una función como devolución de llamada se puede llamar cero veces, una o varias veces. Todo depende de la implementación.
  • La función de devolución de llamada se puede llamar de forma síncrona o asíncrona y posiblemente tanto de forma síncrona como asíncrona.
  • Al igual que las funciones normales, los nombres que le da a los parámetros de su función no son importantes, pero el orden sí lo es. Entonces, por ejemplo, en la línea 8, el parámetro message podría haberse llamado statement, msg, o si no tiene sentido, algo como jellybean. Por lo tanto, debe saber qué parámetros se envían a su devolución de llamada para que pueda obtenerlos en el orden correcto con los nombres adecuados.

Devoluciones de llamadas asincrónicas. #

Una cosa a tener en cuenta sobre JavaScript es que es sincrónico de forma predeterminada, pero hay API proporcionadas en el entorno (navegador, Node.js, etc.) que podrían hacerlo asíncrono (hay más sobre eso aquí).

Algunas cosas comunes que son asincrónicas en entornos de JavaScript que aceptan devoluciones de llamada:

  • Eventos
  • establecer el tiempo de espera
  • establecerIntervalo
  • la API de búsqueda
  • Promesas

Además, cualquier función que use una de las funciones anteriores puede envolverse con una función que recibe una devolución de llamada y la devolución de llamada sería entonces una devolución de llamada asíncrona (aunque envolver una promesa con una función que acepta una devolución de llamada probablemente se consideraría un antipatrón como hay formas preferidas de manejar las promesas).

Entonces, dada esa información, podemos construir una función asíncrona similar a la síncrona anterior.

// a function that uses a callback named `cb` as a parameter
function getAsyncMessage(cb) {
    setTimeout(function () { cb("Hello World!") }, 1000);
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getAsyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

Que imprime lo siguiente en la consola:

> Before getSyncMessage call
> After getSyncMessage call
// pauses for 1000 ms with no output
> Hello World!

La ejecución de línea va a la línea 6 registra “Antes de llamar a getSyncMessage”. Luego, la ejecución va a la línea 8 llamando a getAsyncMessage con una devolución de llamada para el parámetro cb. Luego se ejecuta la línea 3 que llama a setTimeout con una devolución de llamada como primer argumento y el número 300 como segundo argumento. setTimeout hace lo que hace y se aferra a esa devolución de llamada para que pueda llamarla más tarde en 1000 milisegundos, pero luego de configurar el tiempo de espera y antes de pausar los 1000 milisegundos, devuelve la ejecución donde la dejó, por lo que va a línea 4, luego línea 11, y luego se detiene durante 1 segundo y setTimeout luego llama a su función de devolución de llamada que lleva la ejecución a la línea 3 donde se llama la devolución de llamada getAsyncMessages con el valor “Hello World” para su parámetro message que luego se registra a la consola en la línea 9.

Funciones de devolución de llamada en Node.js #

NodeJS tiene devoluciones de llamada asincrónicas y comúnmente proporciona dos parámetros a sus funciones, a veces llamados convencionalmente err y data. Un ejemplo con la lectura de un archivo de texto.

const fs = require("fs");

fs.readFile("./test.txt", "utf8", function(err, data) {
    if(err) {
        // handle the error 
    } else {
        // process the file text given with data
    }
});

Este es un ejemplo de una devolución de llamada que se llama una sola vez.

Es una buena práctica manejar el error de alguna manera, incluso si solo lo registra o lo arroja. El else no es necesario si lanza o devuelve y puede eliminarse para disminuir la sangría siempre que detenga la ejecución de la función actual en el si haciendo algo como lanzar o devolver.

Aunque puede ser común ver err, data, puede que no siempre sea el caso de que sus devoluciones de llamada usen ese patrón, es mejor mirar la documentación.

Otro ejemplo de devolución de llamada proviene de la biblioteca express (express 4.x):

// this code snippet was on http://expressjs.com/en/4x/api.html
const express = require('express');
const app = express();

// this app.get method takes a url route to watch for and a callback
// to call whenever that route is requested by a user.
app.get('/', function(req, res){
  res.send('hello world');
});

app.listen(3000);

Este ejemplo muestra una devolución de llamada que se llama varias veces. La devolución de llamada se proporciona con dos objetos como parámetros nombrados aquí como “req” y “res”. Estos nombres corresponden a la solicitud y la respuesta respectivamente, y brindan formas de ver la solicitud que ingresa y configurar la respuesta que se enviará al usuario. .

Como puede ver, hay varias formas en que se puede usar una devolución de llamada para ejecutar código sincronizado y asíncrono en JavaScript y las devoluciones de llamada son muy ubicuas en todo JavaScript.

Infierno de devolución de llamada #

El infierno de devolución de llamada (también una pirámide de fatalidad o efecto boomerang) surge cuando anida demasiadas funciones de devolución de llamada dentro de una función de devolución de llamada. Aquí hay un ejemplo para leer un archivo (en ES6).

const fs = require('fs');
let filename = `${__dirname}/myfile.txt`;

fs.exists(filename, exists => {
    if (exists) {
        fs.stat(filename, (err, stats) => {
            if (err) {
                throw err;
            }
            if (stats.isFile()) {
                fs.readFile(filename, null, (err, data) => {
                    if (err) {
                        throw err;
                    }
                    console.log(data);
                });
            }
            else {
                throw new Error("This location contains not a file");
            }
        });
    }
    else {
        throw new Error("404: file not found");
    }
});

Cómo evitar el “infierno de devolución de llamada”

Se recomienda anidar no más de 2 funciones de devolución de llamada. Esto lo ayudará a mantener la legibilidad del código y será mucho más fácil de mantener en el futuro. Si necesita anidar más de 2 devoluciones de llamada, intente utilizar eventos distribuidos en su lugar.

También existe una biblioteca llamada async que ayuda a administrar las devoluciones de llamadas y su ejecución disponible en npm. Aumenta la legibilidad del código de devolución de llamada y le brinda más control sobre el flujo de su código de devolución de llamada, incluso permitiéndole ejecutarlos en paralelo o en serie.

Promesas nativas #

Las promesas son una herramienta para la programación asíncrona. En JavaScript, las promesas son conocidas por sus métodos then. Las promesas tienen dos estados principales ‘pendiente’ y ’liquidada’. Una vez que una promesa se ’establece’, no puede volver a estar ‘pendiente’. Esto significa que las promesas son en su mayoría buenas para eventos que solo ocurren una vez. El estado ’establecido’ tiene dos estados también ‘resuelto’ y ‘rechazado’. Puede crear una nueva promesa usando la palabra clave nueva y pasando una función al constructor nueva promesa (función (resolver, rechazar) {}).

La función que se pasa al constructor de Promise siempre recibe un primer y un segundo parámetro, normalmente llamados resolve y reject respectivamente. La denominación de estos dos parámetros es una convención, pero pondrán la promesa en el estado ‘resuelto’ o en el estado ‘rechazado’. Cuando se llama cualquiera de estos, la promesa pasa de estar ‘pendiente’ a ‘cumplida’. Se llama a resolve cuando se ha realizado la acción deseada, que a menudo es asincrónica, y se usa reject si la acción ha fallado.

En el siguiente tiempo de espera hay una función que devuelve una Promesa.

function timeout (ms) {
  return new Promise(function (resolve, reject) {
    setTimeout(function () {
      resolve("It was resolved!");
    }, ms)
  });
}

timeout(1000).then(function (dataFromPromise) {
  // logs "It was resolved!"
  console.log(dataFromPromise);
})

console.log("waiting...");

salida de la consola

waiting...
// << pauses for one second>>
It was resolved!

Cuando se llama al tiempo de espera, la función que se pasa al constructor de Promise se ejecuta sin demora. Luego se ejecuta el método setTimeout y su devolución de llamada se configura para que se active en los siguientes milisegundos ms, en este caso ms=1000. Dado que la devolución de llamada a setTimeout aún no se activa, la función de tiempo de espera devuelve el control al alcance de la llamada. La cadena de métodos ’entonces’ se almacenan para ser llamados más tarde cuando/si la Promesa se ha resuelto. Si hubiera métodos catch aquí, también se almacenarían, pero se activarían cuando/si la promesa se ‘rechaza’.

El script luego imprime ’esperando…’. Un segundo después, setTimeout llama a su devolución de llamada, que llama a la función de resolución con la cadena “¡Se resolvió!”. Esa cadena luego se pasa a la devolución de llamada del método then y luego se registra para el usuario.

En el mismo sentido, puede envolver la función asíncrona setTimeout que requiere una devolución de llamada, puede envolver cualquier acción asíncrona singular con una promesa.

Lea más sobre las promesas en la documentación de JavaScript https://www.wikiod.com/es/javascript/promesas

Ejemplo de código #

Pregunta: ¿Cuál es el resultado del siguiente código y por qué?

setTimeout(function() {
    console.log("A");
}, 1000);

setTimeout(function() {
    console.log("B");
}, 0);

getDataFromDatabase(function(err, data) {
    console.log("C");
    setTimeout(function() {
        console.log("D");
    }, 1000);
});

console.log("E");

Salida: Esto es seguro: EBAD. C es desconocido cuando se registrará.

Explicación: El compilador no se detendrá en los métodos setTimeout y getDataFromDatabase. Así que la primera línea que registrará es E. ¡Las funciones de devolución de llamada (primer argumento de setTimeout) se ejecutarán después del tiempo de espera establecido de forma asíncrona!

Más detalles:

  1. E no tiene setTimeout
  2. B tiene un tiempo de espera establecido de 0 milisegundos
  3. A tiene un tiempo de espera establecido de 1000 milisegundos
  4. D debe solicitar una base de datos, luego D debe esperar 1000 milisegundos para que llegue después de A.
  5. C es desconocido porque es desconocido cuando se solicitan los datos de la base de datos. Podría estar antes o después de A.

Manejo de errores asíncronos #

Trata de atraparlo #

Los errores siempre deben ser manejados. Si está utilizando la programación síncrona, podría usar un intentar atrapar. ¡Pero esto no funciona si trabajas de forma asíncrona! Ejemplo:

try {
    setTimeout(function() {
        throw new Error("I'm an uncaught error and will stop the server!");
    }, 100); 
}
catch (ex) {
    console.error("This error will not be work in an asynchronous situation: " + ex);
}

¡Los errores asíncronos solo se manejarán dentro de la función de devolución de llamada!

Posibilidades de trabajo #

Controladores de eventos #

Las primeras versiones de Node.JS tenían un controlador de eventos.

process.on("UncaughtException", function(err, data) { 
    if (err) {
        // error handling
    }
});

Dominios #

Dentro de un dominio, los errores se liberan a través de los emisores de eventos. Al usar esto, todos los errores, temporizadores, métodos de devolución de llamada se registran implícitamente solo dentro del dominio. Por un error, se envió un evento de error y no bloqueó la aplicación.

var domain = require("domain");
var d1 = domain.create();
var d2 = domain.create();

d1.run(function() {
    d2.add(setTimeout(function() {
        throw new Error("error on the timer of domain 2");
    }, 0));
});

d1.on("error", function(err) {
    console.log("error at domain 1: " + err);
});

d2.on("error", function(err) {
    console.log("error at domain 2: " + err);
});
