Puedes crear package.json con

npm init

que le preguntará sobre datos básicos de sus proyectos, incluido identificador de licencia.

Guiones #

Puede definir scripts que se pueden ejecutar o desencadenar antes o después de otro script.

{
  "scripts": {
    "pretest": "scripts/pretest.js",
    "test": "scripts/test.js",
    "posttest": "scripts/posttest.js"
  }
}

En este caso, puede ejecutar el script ejecutando cualquiera de estos comandos:

$ npm run-script test
$ npm run test
$ npm test
$ npm t

Guiones predefinidos #

Nombre del guión Descripción
prepublicar Ejecutar antes de que se publique el paquete.
publicar, pospublicar Ejecutar después de que se publique el paquete.
preinstalar Ejecutar antes de instalar el paquete.
instalar, postinstalar Ejecutar después de instalar el paquete.
predesinstalar, desinstalar Ejecutar antes de que se desinstale el paquete.
después de la desinstalación Ejecutar después de desinstalar el paquete.
preversión, versión Ejecutar antes de cambiar la versión del paquete.
posversión Ejecutar después de actualizar la versión del paquete.
prueba previa, prueba, prueba posterior Ejecutado por el comando npm test
prestop, stop, poststop Ejecutado por el comando npm stop
preinicio, inicio, postinicio Ejecutado por el comando npm start
prereinicio, reinicio, postreinicio Ejecutado por el comando npm restart

Guiones definidos por el usuario #

También puede definir sus propios scripts de la misma manera que lo hace con los scripts predefinidos:

{
  "scripts": {
    "preci": "scripts/preci.js",
    "ci": "scripts/ci.js",
    "postci": "scripts/postci.js"
  }
}

En este caso, puede ejecutar el script ejecutando cualquiera de estos comandos:

$ npm run-script ci
$ npm run ci

Los scripts definidos por el usuario también admiten scripts pre y post, como se muestra en el ejemplo anterior.

Explorando paquete.json #

Un archivo package.json, generalmente presente en la raíz del proyecto, contiene metadatos sobre su aplicación o módulo, así como la lista de dependencias para instalar desde npm cuando se ejecuta npm install.

Para inicializar un package.json, escriba npm init en su símbolo del sistema.

Para crear un package.json con valores predeterminados, use:

npm init --yes
# or
npm init -y

Para instalar un paquete y guardarlo en package.json use:

npm install {package name} --save

También puede utilizar la notación abreviada:

 npm i -S {package name}

NPM alias -S a --save y -D a --save-dev para guardar en sus dependencias de producción o desarrollo, respectivamente.

El paquete aparecerá en sus dependencias; si utiliza --save-dev en lugar de --save, el paquete aparecerá en sus devDependencies.

Propiedades importantes de package.json:

{
  "name": "module-name",
  "version": "10.3.1",
  "description": "An example module to illustrate the usage of a package.json",
  "author": "Your Name <your.name@example.org>",
  "contributors": [{
    "name": "Foo Bar",
    "email": "foo.bar@example.com"
  }],
  "bin": {
    "module-name": "./bin/module-name"
  },
  "scripts": {
    "test": "vows --spec --isolate",
    "start": "node index.js",
    "predeploy": "echo About to deploy",
    "postdeploy": "echo Deployed",
    "prepublish": "coffee --bare --compile --output lib/foo src/foo/*.coffee"
  },
  "main": "lib/foo.js",
  "repository": {
    "type": "git",
    "url": "https://github.com/username/repo"
  },
  "bugs": {
    "url": "https://github.com/username/issues"
  },
  "keywords": [
    "example"
  ],
  "dependencies": {
    "express": "4.2.x"
  },
  "devDependencies": {
    "assume": "<1.0.0 || >=2.3.1 <2.4.5 || >=2.5.2 <3.0.0"
  },
  "peerDependencies": {
    "moment": ">2.0.0"
  },
  "preferGlobal": true,
  "private": true,
  "publishConfig": {
    "registry": "https://your-private-hosted-npm.registry.domain.com"
  },
  "subdomain": "foobar",
  "analyze": true,
  "license": "MIT",
  "files": [
    "lib/foo.js"
  ]
}

Información sobre algunas propiedades importantes:

name

El nombre único de su paquete y debe estar en minúsculas. Esta propiedad es obligatoria y su paquete no se instalará sin ella.

  1. El nombre debe ser menor o igual a 214 caracteres.
  2. El nombre no puede comenzar con un punto o un guión bajo.
  3. Los paquetes nuevos no deben tener letras mayúsculas en el nombre.
version

La versión del paquete está especificada por Semantic Versioning (semver). Lo que supone que un número de versión está escrito como MAYOR.MENOR.PARCHE y usted incrementa:

  1. Versión PRINCIPAL cuando realiza cambios de API incompatibles

  2. Versión MENOR when you add functionality in a backwards-compatible manner

  3. PARCHE version when you make backwards-compatible bug fixes

    description

La descripción del proyecto. Trate de mantenerlo breve y conciso.

author

El autor de este paquete.

bin

Un objeto que se usa para exponer scripts binarios de su paquete. El objeto asume que la clave es el nombre del script binario y el valor es una ruta relativa al script.

Esta propiedad la utilizan los paquetes que contienen una CLI (interfaz de línea de comandos).

script

Un objeto que expone comandos npm adicionales. El objeto asume que la clave es el comando npm y el valor es la ruta del script. Estos scripts se pueden ejecutar cuando ejecuta npm run {nombre del comando} o npm run-script {nombre del comando}.

Los paquetes que contienen una interfaz de línea de comandos y se instalan localmente se pueden llamar sin una ruta relativa. Entonces, en lugar de llamar a ./node-modules/.bin/mocha, puede llamar directamente a mocha.

main

El principal punto de entrada a su paquete. Al llamar a require('{module name}') en el nodo, este será el archivo real que se requiere.

Se recomienda enfáticamente que requerir el archivo principal no genere efectos secundarios. Por ejemplo, solicitar el archivo principal no debería iniciar un servidor HTTP o conectarse a una base de datos. En su lugar, debe crear algo como exports.init = function () {...} en su script principal.

keywords

Una serie de palabras clave que describen su paquete. Estos ayudarán a las personas a encontrar su paquete.

devDependencies

Estas son las dependencias que solo están destinadas al desarrollo y prueba de su módulo. Las dependencias se instalarán automáticamente a menos que se haya establecido la variable de entorno NODE_ENV=producción. Si este es el caso, aún puede usar estos paquetes usando npm install --dev

peerDependencies

Si está utilizando este módulo, peerDependencies enumera los módulos que debe instalar junto con este. Por ejemplo, moment-timezone debe instalarse junto con moment porque es un complemento para moment, incluso si no require("moment") directamente.

preferGlobal

Una propiedad que indica que esta página prefiere instalarse globalmente mediante npm install -g {module-name}. Esta propiedad la utilizan los paquetes que contienen una CLI (interfaz de línea de comandos).

En todas las demás situaciones, NO debe usar esta propiedad.

publishConfig

La publicaciónConfig es un objeto con valores de configuración que se usará para publicar módulos. Los valores de configuración que se establecen anulan su configuración predeterminada de npm.

El uso más común de publishConfig es publicar su paquete en un registro privado de npm para que aún tenga los beneficios de npm pero para paquetes privados. Esto se hace simplemente configurando la URL de su npm privado como valor para la clave de registro.

files

Esta es una matriz de todos los archivos que se incluirán en el paquete publicado. Se puede utilizar una ruta de archivo o una ruta de carpeta. Se incluirá todo el contenido de la ruta de una carpeta. Esto reduce el tamaño total de su paquete al incluir solo los archivos correctos para distribuir. Este campo funciona junto con un archivo de reglas .npmignore.

Definición básica del proyecto #

{
    "name": "my-project",
    "version": "0.0.1",
    "description": "This is a project.",
    "author": "Someone <someone@example.com>",
    "contributors": [{
        "name": "Someone Else",
        "email": "else@example.com"
    }],
    "keywords": ["improves", "searching"]
}
Campo Descripción
nombre un campo obligatorio para que se instale un paquete. Debe estar en minúsculas, una sola palabra sin espacios. (Se permiten guiones y guiones bajos)
versión un campo requerido para la versión del paquete usando versiones semánticas.
descripción una breve descripción del proyecto
autor especifica el autor del paquete
colaboradores una matriz de objetos, uno para cada colaborador
palabras clave una serie de cadenas, esto ayudará a las personas a encontrar su paquete

Dependencias #

“dependencias”: { “module-name”: “0.1.0” }

  • exacto: 0.1.0 instalará esa versión específica del módulo.
  • versión secundaria más reciente: ^0.1.0 instalará la versión secundaria más reciente, por ejemplo, 0.2.0, pero no instalará un módulo con una versión principal superior, p. 1.0.0
  • parche más reciente: 0.1.x o ~0.1.0 instalará la versión de parche más reciente disponible, por ejemplo, 0.1.4, pero no instalará un módulo con una versión mayor o menor superior, p.ej. 0.2.0 o 1.0.0.
  • comodín: * instalará la última versión del módulo.
  • repositorio git: lo siguiente instalará un tarball desde la rama maestra de un repositorio git. También se puede proporcionar un #sha, #tag o #branch:
  • GitHub: usuario/proyecto o usuario/proyecto#v1.0.0
  • url: git://gitlab.com/user/project.git o git://gitlab.com/user/project.git#develop
  • ruta local: archivo:../lib/proyecto

Después de agregarlos a su paquete.json, use el comando npm install en el directorio de su proyecto en la terminal.

dependencias de desarrollo #

"devDependencies": {
    "module-name": "0.1.0"
}

Para las dependencias requeridas solo para el desarrollo, como probar los proxies de estilo ext. Esas dependencias de desarrollo no se instalarán cuando se ejecute “npm install” en modo de producción.

Definición de proyecto extendida #

Algunos de los atributos adicionales son analizados por el sitio web de npm como repository, bugs o homepage y se muestran en el cuadro de información de estos paquetes

{
  "main": "server.js",  
  "repository" :  {
    "type": "git",
    "url": "git+https://github.com/<accountname>/<repositoryname>.git"
  },
  "bugs": {
    "url": "https://github.com/<accountname>/<repositoryname>/issues"
  },
  "homepage": "https://github.com/<accountname>/<repositoryname>#readme",
  "files": [
    "server.js", // source files
    "README.md", // additional files
    "lib" // folder with all included files
  ]
