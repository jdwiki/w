¿Cómo funciona la API Blueprint? #

Cuando Sails inicialmente comienza a usar sails lift, Sails mira para ver si tienes cualquier controlador definido. En nuestro ejemplo, tenemos un controlador, el Usuario controlador. Luego, Sails proporciona acceso a las acciones de blueprint para este usuario. controlador como si los construyéramos en el controlador nosotros mismos. Velas también crea automáticamente rutas blueprint en el momento de levantar el servidor. Aun asi si no se definen rutas en /config/routes.js y no se define ninguna acción en /api/controllers/UserController.js explícitamente, después de levantar el servidor todo estas rutas y acciones están disponibles para su uso.

Rutas de planos #

Cuando ejecuta sails lift con blueprints habilitados, el marco inspecciona su controladores, modelos y configuración para enlazar ciertas rutas automáticamente. Estas rutas de planos implícitos permiten que su aplicación responda a ciertas solicitudes sin tener que vincular esas rutas manualmente en su Archivo config/routes.js. De forma predeterminada, las rutas del blueprint apuntan a su blueprint acciones correspondientes, cualquiera de las cuales se puede anular con un código personalizado.

Hay tres tipos de rutas de planos en Sails:

  • Rutas RESTful, donde la ruta siempre es /:model/:id?. Cuando el Usuario el modelo y el controlador están definidos, blueprint une rutas RESTful implícitamente en siguiente manera -

‘GET /usuario/:id?’: { controller: ‘User’, action: ‘find’ }, ‘POST /usuario’: { controller: ‘User’, action: ‘create’ }, ‘PUT /usuario/:id?’: { controller: ‘User’, action: ‘update’ }, ‘ELIMINAR /usuario/:id?’: { controller: ‘User’, action: ‘destroy’ }

Estas rutas utilizan el verbo HTTP para determinar la acción a realizar incluso si el
la ruta es la misma. Entonces, una solicitud `POST` a `/usuario` creará un nuevo usuario, un
La solicitud `PUT` a `/user/123` actualizará al usuario con la clave principal 123 y
una solicitud `DELETE` a `/user/123` eliminará al usuario cuya clave principal es 123.
En un entorno de producción, las rutas RESTful generalmente deben estar protegidas por
políticas para evitar el acceso no autorizado.


+ **Rutas de acceso directo**, donde la acción a realizar está codificada en la ruta. Para nuestro
El modelo de usuario y el controlador Sails se unen siguiendo cuatro rutas de acceso directo implícitamente.

‘OBTENER /usuario/buscar/:id?’: { controller: ‘User’, action: ‘find’ }, ‘OBTENER /usuario/crear/:id?’: { controller: ‘User’, action: ‘create’ }, ‘OBTENER /usuario/actualizar/:id?’: { controller: ‘User’, action: ‘update’ }, ‘OBTENER /usuario/destruir/:id?’: { controller: ‘User’, action: ‘destroy’ }

Como ejemplo, el atajo `/user/create?name=joe` crea un nuevo usuario, mientras que
`/user/update/1?name=mike` actualiza el campo de nombre del usuario #1. Tenga en cuenta que estos
las rutas solo responden a solicitudes `GET`. Las rutas de acceso directo son muy útiles para
desarrollo, pero generalmente debe estar deshabilitado en un entorno de producción. Es
no está diseñado para ser utilizado en la producción.

+ **Rutas de acción**, que crean automáticamente rutas para su controlador personalizado
comportamiento. Por ejemplo, deje que "consulta" sea una acción personalizada definida en el controlador de usuario.
Entonces, las siguientes rutas estarían implícitamente disponibles para las velas:

‘GET /usuario/consulta/:id?’: { controller: ‘User’, action: ‘query’ }, ‘POST /usuario/consulta/:id?’: { controller: ‘User’, action: ‘query’ }, ‘PUT /usuario/consulta/:id?’: { controller: ‘User’, action: ‘query’ }, ‘¿ELIMINAR /usuario/consulta/:id?’: { controller: ‘User’, action: ‘query’ }

Si la solicitud se realiza en la ruta `/user/query/:id?`, entonces es independiente del HTTP
verbo la acción sería la misma. A diferencia de las rutas RESTful y de acceso directo, la acción
las rutas *no* requieren que un controlador tenga un archivo de modelo correspondiente. Cual
significa, si define un controlador en el archivo `/api/controllers/FooController.js`
pero no hay modelo en el archivo `/api/models/Foo.js`, no habría RESTful o
ruta de acceso directo con `/foo` pero todavía habrá rutas de acción disponibles para
usar.

# Orden de coincidencia de rutas
Cuando llega una petición,
sails primero comparará la ruta con las rutas definidas explícitamente. si coincide
entonces no se realiza ninguna otra coincidencia y se ejecuta la acción correspondiente. pero si
no coincide, entonces la ruta se compara primero con las rutas de acción del modelo,
si no coincide, entonces contra las rutas de descanso y si tampoco coincide, entonces
rutas de acceso directo. Entonces, si su archivo `/config/routes.js` tiene alguna entrada como la
siguiendo-

‘/user/query/:id?’: { controller: ‘User’, action: ‘find’ }

Entonces no puede esperar que las rutas de acción `query` funcionen. Porque la misma ruta que
La ruta de la acción `query` se compararía con las rutas definidas explícitamente y
Se ejecutaría la acción `buscar` del controlador de usuario.

## Acciones de planos
Las acciones de blueprint (que no deben confundirse con la acción de blueprint *rutas*) son
acciones genéricas diseñadas para trabajar con cualquiera de sus controladores que tengan un modelo
del mismo nombre (por ejemplo, `ParrotController` necesitaría un modelo `Parrot`). Pensar en
como el comportamiento predeterminado para su aplicación. Por ejemplo, si tienes un
modelo `User.js` y un controlador `UserController.js` vacío, `find`, `create`,
Las acciones `actualizar`, `destruir`, `poblar`, `agregar` y `eliminar` existen implícitamente,
sin que tengas que escribirlos.

De forma predeterminada, las rutas RESTful blueprint y las rutas de acceso directo están vinculadas a sus
correspondientes acciones del proyecto. Sin embargo, cualquier acción de plano puede anularse
para un controlador en particular creando una acción personalizada en ese archivo de controlador
(por ejemplo, `ParrotController.find`). Alternativamente, puede anular el plano
acción _en todas partes de su aplicación_ creando su propia acción de plano personalizado.

Navega barcos con las siguientes acciones de plano:

+ encontrar
+ encontrarUno
+ crear
+ actualizar
+ destruir
+ poblar
+ añadir
+ eliminar

## Inhabilitación de rutas Blueprint
+ **Base global**:
La configuración de la API Blueprint se define en el archivo `/config/blueprint.js`. Puedes
habilitar o deshabilitar los tres tipos de rutas blueprint para todos los controladores desde
allá. Por ejemplo, si desea deshabilitar las rutas de acceso directo de blueprint para todos
sus controladores pero desea mantener habilitadas las rutas de acción y descanso, entonces
su `/config/blueprint.js` debería ser así:

módulo.exportaciones.blueprints = { action: true, rest: true, shortcut: false }


+ **Por controlador**:
También puede anular cualquiera de las configuraciones de `/config/blueprints.js` en un
por controlador definiendo una clave '_config' en la definición de su controlador,
y asignándole un objeto de configuración con anulaciones para las configuraciones en este
expediente. Como ejemplo, si desea tener rutas de acceso directo habilitadas solo para su usuario
controlador pero no para más controladores que con el modelo anterior
la configuración debe tener el siguiente par de valores clave en el controlador de usuario.

módulo.exportaciones = { _config: { actions: true, shortcuts: true, rest: true } }
