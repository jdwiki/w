Crear un artículo básico con Forge

  • Clase de artículo
  • Modelo de artículo
  • Registro de artículo

En esta página

  • Clase de artículo
  • Modelo de artículo
  • Registro de artículo

Crear un elemento simple con Forge es una de las primeras tareas que un aspirante a modder tendrá que aprender. La forma de hacerlo ha cambiado en las distintas versiones de Minecraft y es probable que tenga una dificultad “moderada” después de la 1.7.10 debido a la gran cantidad de errores fáciles de cometer, especialmente al hacerlo correctamente.

Si algo sale mal y a su artículo personalizado le falta una textura (negro/púrpura) o un modelo (cubo predeterminado que es demasiado grande cuando se sostiene), verifique el registro. Los problemas de este tipo casi siempre aparecerán en el registro, siempre que haya registrado las cosas correctamente.

Lo primero que aparecerá es la línea "Modelo de carga de excepción para variante..." o similar, que le indica qué bloque o elemento no se cargó correctamente. Después de una docena de líneas que comienzan con at... habrá otra línea que comienza con Caused by...

Esta línea “Causado por” es la importante, le dirá qué archivo no se cargó correctamente y puede ser uno de varios errores, como:

  • JSON mal formado (su archivo JSON no es válido y tiene un error tipográfico)
  • Archivo no encontrado (el archivo que busca Minecraft no tiene el nombre correcto o no está en el lugar correcto)
  • Excepción de variante faltante (su Blockstate JSON está incompleto)

¡Incluso puede obtener más de un error fallido por un solo error! Si no sabe qué elemento es el problema, comente cada bloque y elemento que sepa que funciona, reduzca la lista de errores al bloque (o bloques) en cuestión. Tu debes hacer esto varias veces.

Además, una textura mixta se muestra de manera tan diferente como una lista simple de todos los recursos que faltan para un dominio determinado (ID de mod).

Clase de artículo #

Esta parte no ha cambiado mucho en las versiones de Minecraft, aunque ha habido algunas mutaciones en las firmas de métodos exactos, así como en la jerarquía de clases. Un elemento básico (uno que no tiene funcionalidad, como un palo o un lingote: así es, ¡ambos son elementos que no hacen nada!)

public class CustomItem extends Item {

    public CustomItem () {
        super();
        this.setMaxDamage(0);
        this.setCreativeTab(CreativeTabs.MISC);
    }
}

No hay mucho espacio para la personalización en este punto, a diferencia de los bloques. Casi todo lo que podemos hacer es cambiar si el elemento puede recibir daño o no (las herramientas usan esto) y en qué pestaña creativa existirá. Manejaremos el nombre y la textura cuando registremos el elemento con GameRegistry.

Sin embargo, esto es todo lo que se necesita para sostener, transportar, dejar caer, fabricar, fundir y utilizar el objeto. Algunos elementos en Minecraft (como los palos) ni siquiera tienen una clase única y simplemente usan nuevo elemento (). Podríamos hacer eso aquí, sin embargo, cualquier elemento con funcionalidad adicional necesitará una clase.

Modelo de artículo #

Al igual que con los bloques, los elementos también necesitan modelos.

{
    "parent": "item/generated",
    "textures": {
        "layer0": "example:items/basic"
    }
}

Eso es prácticamente todo lo que se necesita para que funcione una vez que se registra el artículo. Lo único importante es que el nombre del archivo coincida con el nombre de registro utilizado para registrar el bloque y debe estar en minúsculas (los nombres de archivo 1.11+ deben estar en minúsculas, antes de eso solo se distingue entre mayúsculas y minúsculas).

Tenga en cuenta que “capa0” es la única textura necesaria y es muy poco probable que se especifique ninguna otra textura (aunque algunos elementos como pociones y armaduras de cuero tienen una “capa1”). Todos los nombres están definidos por item/builtin (el modelo principal superior interno para elementos) a diferencia de los bloques.

Nombre el archivo JSON del modelo my_item.json (que coincida con el nombre de registro que le daremos más adelante) y guárdelo en src\main\resources\assets\example\models\item\ (donde example es el ID de mod especificado en la anotación @Mod de su clase de mod principal).

Además, cree una textura para su elemento, asígnele el nombre basic.png y guárdelo en src\main\resources\assets\example\textures\items\

El modelo de elemento aquí usa un padre de elemento/generado, lo que significa que se usará la única textura suministrada (como con la mayoría de los elementos que no son bloques) y se sostendrá en la mano del jugador en una orientación predeterminada. También hay elementos/dispositivos portátiles que especifican diferentes orientaciones de visualización (para herramientas). Los elementos también pueden proporcionar su propio atributo de “visualización”, anulando los del padre, pero no es necesario en el 99,9 % de los usos.

Registro de artículo #

El registro de elementos se realiza desde su clase de mod principal, o un método de clase ModItems invocado desde la clase de mod principal durante preInit.

Item item = new CustomItem();
string registryname = "my_item";
item.setRegistryName(registryname);
item.setUnlocalizedName(item.getRegistryName().toString());
GameRegistry.register(item);

¡Hay una razón importante para usar item.setUnlocalizedName(item.getRegistryName().toString()); también! Asegura que el nombre no localizado de su elemento contenga la ID de mod para evitar conflictos de archivos de idioma entre mods.

Ahora el elemento también necesita un modelo, y aquí es donde las cosas se pusieron difíciles después de la versión 1.7.10, que solo implicaba decirle a Minecraft el nombre de la textura para cargar y podría especificarse en el constructor del elemento.

final ModelResourceLocation fullModelLocation = new ModelResourceLocation(item.getRegistryName().toString(), "inventory");
ModelBakery.registerItemVariants(item, fullModelLocation);
ModelLoader.setCustomMeshDefinition(item, new ItemMeshDefinition()
    {
        public ModelResourceLocation getModelLocation(ItemStack stack)
        {
            return fullModelLocation;
        }
    });

Tenga en cuenta que esta sección debe ubicarse solo en el lado del cliente (es decir, el proxy del cliente) ya que muchas de las clases a las que se hace referencia no existen en el servidor dedicado.

Registrar artículos con variantes p. Saplings, tiene que hacerse de una manera diferente, usando ModelLoader.setCustomModelResourceLocation(item, metadata, resourceLocation) aunque nos permite usar un archivo Blockstate para especificar nuestras variantes (que es mucho preferible a la alternativa). Nuestro artículo no usa variantes, así que hemos terminado.
