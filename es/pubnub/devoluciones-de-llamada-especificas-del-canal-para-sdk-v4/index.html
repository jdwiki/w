Devoluciones de llamada específicas del canal JavaScript v4 SDK #

En PubNub JavaScript v3, podría implementar una devolución de llamada única para cada canal al que se suscribió siempre que llamara a la función de suscripción para cada canal e implementara la devolución de llamada en esa suscripción de esta manera:

var pubnub = new PubNub({
    publishKey: "your-pub-key",
    subscribeKey: "your-sub-key"
});

pubnub.subscribe({
    channel: 'ch1',
    message: function (m) {
        console.log(m + " ch1 callback");
    }
});

pubnub.subscribe({
    channel: 'ch2',
    message: function (m) {
        console.log(m + " ch2 callback: ");
    }
});

Entonces, en el código anterior, si publicas el mensaje "hola" en ch1:

publish({channel: "ch1", message: "hello"});

…entonces obtendrías la salida, hello ch1 callback. Y, por supuesto, si publicaste el mismo mensaje en ch2, obtendrías el resultado hello ch2 callback.

La capacidad de proporcionar una devolución de llamada personalizada para cada canal, o un grupo de canales, es útil y, a menudo, más deseable que crear una devolución de llamada monolítica con un largo “if-then-else” de nombres de canales y código que se ejecutará para cada condición. Y hay mejores prácticas para usar que el ejemplo simple anterior, pero quería que fuera fácil de comparar y contrastar con la forma en que PubNub JavaScript SDK v4 (y cualquier v4 PubNub SDK) está diseñado.

Los SDK de PubNub v4 utilizan un solo oyente para recibir todos los eventos de mensaje, presencia y estado. Esto significa que cuando se “suscribe” a un canal, solo proporciona los nombres del canal en lugar de una función complementaria de “devolución de llamada de mensaje”. En JavaScript SDK v4, el oyente tiene este aspecto:

pubnub.addListener({
    message: function(m) {
        console.log(JSON.stringify(m));
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(s) {
        console.log(JSON.stringify(s));
    }
});

Sé que muchos desarrolladores se preguntarán cómo migrar el código de suscripción con devoluciones de llamada de mensaje únicas de SDK v3 con este tipo de diseño sin recurrir al código condicional de nombre de canal sin fin que mencioné anteriormente, así:

pubnub.addListener({
    message: function(m) {
        if (m.subscribedChannel == 'ch1') {
            console.log(m.message + "ch1 callback");
        }
        else if (m.subscribedChannel == 'ch2') {
            console.log(m.message + "ch2 callback");
        }
        else {
            console.log(m.message + "default callback");
        }
    }
    // removed the other two callbacks for brevity purposes
});

El parámetro m que se pasa al detector de mensajes anterior tiene la siguiente estructura, que es un diseño diferente al diseño de parámetros múltiples de JavaScript SDK v3.

{
    "channel": "ch1",
    "subscription": <undefined>,
    "timetoken": "14721821326909151",
    "message": "hello"
}

Ese token de tiempo es el token de tiempo de publicación real. Para los desarrolladores de PubNub experimentados, debería estar emocionado de ver que esto ahora está disponible para el suscriptor, pero no entremos en por qué esto es importante y poderoso en este momento.

Ahora, no esperaría que ningún desarrollador de JavaScript experimentado escribiera código como se representa arriba y es posible que muchos desarrolladores avanzados ya sepan qué hacer. Pero para aquellos desarrolladores que tienen un nivel de principiante a intermedio con JavaScript, la solución puede no ser obvia de inmediato. Sin embargo, sé que una vez que vea este enfoque de diseño simple a continuación, le abrirá los ojos a las posibilidades ilimitadas del lenguaje JavaScript: aquí vamos.

Reiteremos el requisito:

Por cada canal al que me suscribo, quiero poder proporcionar un función única para invocar cuando se recibe un mensaje en ese canal. Y quiero evitar el enfoque de nombre de canal condicional monolítico.

Entonces, lo que debemos hacer primero es crear una tabla de búsqueda de funciones. Una tabla hash para ser exactos. Esta tabla tendrá nombres de canales como claves y funciones (el código que se invocará cuando se reciba un mensaje en ese canal) como valores. Si es un poco nuevo en JavaScript o ha estado programando con el lenguaje por un tiempo pero aún no se ha sumergido en las características del lenguaje, esto puede sonar extraño e imposible, pero realmente es cómo funciona JavaScript y lo ha estado haciendo todo el tiempo. y realmente no lo sabía. Definamos nuestra tabla de búsqueda de funciones:

ftbl = {};

Eso es todo: tienes un objeto que mantendrá tus canales y funciones. Bastante simple, ¿verdad? Pero, ¿cómo se agregan los canales y funciones? Al igual que cualquier otra clave/valor.

ftbl.ch1 = function(m){console.log(m.message + " ch1 callback")};
ftbl.ch2 = function(m){console.log(m.message + " ch1 callback")};

…y así sucesivamente con cada canal y función que desee definir. Y no tiene que crear todas sus teclas de canal/función en un solo lugar de su código. Puede agregar cada canal/función a ftbl cuando se suscribe a un canal.

ftbl.ch10 = function(m){console.log(m.message + " ch10 callback")};
pubnub.pubnub.subscribe({
    channels: ['ch10'] 
});

De acuerdo, eso es bastante simple, y puedes volverte más elegante y más avanzado con la forma en que lo haces, pero manteniéndolo básico. Pero, ¿cómo se invoca esta función para el canal al que está vinculado? Esta es la razón por la que JavaScript es tan genial, poderoso y fácil, especialmente si vienes de un lenguaje rígido y estructurado como Java. Compruébalo.

pubnub.addListener({
    message: function(m) {
        // use the channel name to get the function
        // from ftbl and invoke it 
        ftbl[m.subscribedChannel](m);
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(statusEvent) {
        console.log(JSON.stringify(s));
    }
});

Eso es todo al respecto. Simplemente obtenga la función de ftbl usando el nombre del canal que se pasa a la función de devolución de llamada message del oyente y agregue (m) al final y boom, ejecuta su función.

Si el canal es ch10, ftbl[m.subscribedChannel](m) simplemente invoca function(m){console.log(m.message + "ch10 callback")} pasando el parámetro m que su función puede analizar y explotar según sea necesario.

Así que llamando a la siguiente función publicar:

pubnub.publish(
    {
        channel : "ch10",
        message : "hello"
    }, 
    function(status, response) {
        console.log(status, response);
    }
);

… dará como resultado que se muestre el siguiente mensaje: hello ch10 callback. Y el equivalente para publicar en otros canales que haya definido en su tabla de búsqueda de funciones. No olvide proporcionar un valor predeterminado para canales desconocidos.

Y no olvide las devoluciones de llamada presence y status en el oyente. Esto podría ser solo dos tablas de búsqueda de funciones más o simplemente un ftbl un poco más complejo:

ftbl.message.ch1 = function(m){console.log(m.message + " ch1 message cb")};
ftbl.presence.ch1 = function(m){console.log(m.message + " ch1 presence cb")};
ftbl.status.ch1 = function(m){console.log(m.message + " ch1 status cb")};

o

ftbl.ch1.message = ...
ftbl.ch1.presence = ...
ftbl.ch1.status = ...

Me gusta más el primero que el segundo, pero depende de ti. Y probablemente desee un código genérico de manejo de eventos status de todos modos, pero dependerá de sus requisitos específicos.

Y esto puede volverse aún más complicado y robusto con funciones opcionales por canal que se pueden invocar dependiendo de algunos datos adicionales que incruste en la carga útil del mensaje.

Ahí lo tienes, devoluciones de llamada únicas para cada canal. No más excusas para no querer migrar de 3x a 4x. Pero si tiene algunas dudas sobre la migración, no dude en comunicarse con Soporte de PubNub y lo ayudaremos a seguir adelante. Y no olvide revisar la Guía de migración de PubNub JavaScript SDK v3 a v4.
