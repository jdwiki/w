Sintaxis #

  • docker ejecuta [OPCIONES] IMAGEN [COMANDO] [ARG…]

Eliminar automáticamente un contenedor después de ejecutarlo #

Normalmente, un contenedor de Docker persiste después de haber salido. Esto le permite volver a ejecutar el contenedor, inspeccionar su sistema de archivos, etc. Sin embargo, a veces desea ejecutar un contenedor y eliminarlo inmediatamente después de que salga. Por ejemplo, para ejecutar un comando o mostrar un archivo del sistema de archivos. Docker proporciona la opción de línea de comando --rm para este propósito:

docker run --rm ubuntu cat /etc/hosts

Esto creará un contenedor a partir de la imagen “ubuntu”, mostrará el contenido del archivo /etc/hosts y luego eliminará el contenedor inmediatamente después de que salga. Esto ayuda a evitar tener que limpiar los contenedores después de que haya terminado de experimentar.

Nota: El indicador --rm no funciona junto con el indicador -d (--detach) en docker < 1.13.0.

Cuando se establece el indicador --rm, Docker también elimina los volúmenes asociados con el contenedor cuando se elimina el contenedor. Esto es similar a ejecutar docker rm -v my-container. Solo se eliminan los volúmenes que se especifican sin nombre.

Por ejemplo, con docker run -it --rm -v /etc -v logs:/var/log centos /bin/produce_some_logs, se eliminará el volumen de /etc, pero el volumen de /var/log no lo hará. Los volúmenes heredados a través de –volumes-from se eliminarán con la misma lógica: si el volumen original se especificó con un nombre, no se eliminará.

Política de reinicio de contenedores (iniciar un contenedor en el arranque) #

docker run --restart=always -d <container>

De forma predeterminada, Docker no reiniciará los contenedores cuando se reinicie el demonio de Docker, por ejemplo, después de reiniciar el sistema host. Docker proporciona una política de reinicio para sus contenedores al proporcionar la opción de línea de comando --restart. Proporcionar --restart=always siempre hará que un contenedor se reinicie después de que se reinicie el demonio Docker. Sin embargo cuando ese contenedor se detiene manualmente (por ejemplo, con docker stop <container>), la política de reinicio no se aplicará al contenedor.

Se pueden especificar múltiples opciones para la opción --restart, según el requisito (--restart=[política]). Estas opciones también afectan la forma en que el contenedor se inicia en el arranque.

Política Resultado
no El valor predeterminado. No reiniciará el contenedor automáticamente cuando el contenedor esté detenido.
on-failure[:max-retries] Reiniciar solo si el contenedor sale con un error (estado de salida distinto de cero). Para evitar reiniciarlo indefinidamente (en caso de algún problema), se puede limitar el número de reintentos de reinicio que intenta el demonio Docker.
siempre Siempre reinicie el contenedor independientemente del estado de salida. Cuando especifica siempre, el demonio Docker intentará reiniciar el contenedor indefinidamente. El contenedor siempre se iniciará en el inicio del daemon, independientemente del estado actual del contenedor.
a menos que se detenga Siempre reinicie el contenedor independientemente de su estado de salida, pero no lo inicie en el inicio del daemon si el contenedor se ha detenido antes.

Especificando un nombre #

De forma predeterminada, los contenedores creados con docker run reciben un nombre aleatorio como small_roentgen o modest_dubinsky. Estos nombres no son particularmente útiles para identificar el propósito de un contenedor. Es posible proporcionar un nombre para el contenedor pasando la opción de línea de comando --name:

docker run --name my-ubuntu ubuntu:14.04

Los nombres deben ser únicos; si pasa un nombre que ya está usando otro contenedor, Docker imprimirá un error y no se creará ningún contenedor nuevo.

Especificar un nombre será útil al hacer referencia al contenedor dentro de una red de Docker. Esto funciona tanto para contenedores Docker de fondo como de primer plano.

Los contenedores en la red puente predeterminada deben estar vinculados para comunicarse por nombre.

Ejecutar un contenedor en segundo plano #

Para mantener un contenedor ejecutándose en segundo plano, proporcione la opción de línea de comando -d durante el inicio del contenedor:

docker run -d busybox top

La opción -d ejecuta el contenedor en modo separado. También es equivalente a -d=true.

Un contenedor en modo separado no se puede eliminar automáticamente cuando se detiene, lo que significa que no se puede usar la opción –rm en combinación con la opción -d.

Asignar un volumen a un contenedor #

Un volumen de Docker es un archivo o directorio que persiste más allá de la vida útil del contenedor. Es posible montar un archivo o directorio de host en un contenedor como un volumen (sin pasar por UnionFS).

Agregue un volumen con la opción de línea de comando -v:

docker run -d -v "/data" awesome/app bootstrap.sh

Esto creará un volumen y lo montará en la ruta /data dentro del contenedor.

  • Nota: puede usar la bandera --rm para eliminar automáticamente el volumen cuando se elimina el contenedor.

Montaje de directorios de host

Para montar un archivo o directorio host en un contenedor:

docker run -d -v "/home/foo/data:/data" awesome/app bootstrap.sh
  • Al especificar un directorio de host, se debe proporcionar una ruta absoluta.

Esto montará el directorio host /home/foo/data en /data dentro del contenedor. Este volumen de “directorio de host montado en enlace” es lo mismo que un mount --bind de Linux y, por lo tanto, monta temporalmente el directorio de host sobre la ruta del contenedor especificado durante la vida útil del contenedor. Los cambios en el volumen del host o del contenedor se reflejan inmediatamente en el otro, porque tienen el mismo destino en el disco.

Ejemplo de UNIX montando una carpeta relativa

docker run -d -v $(pwd)/data:/data awesome/app bootstrap.sh

Volúmenes de nombres

Se puede nombrar un volumen proporcionando una cadena en lugar de una ruta de directorio de host, la ventana acoplable creará un volumen con ese nombre.

docker run -d -v "my-volume:/data" awesome/app bootstrap.sh

Después de crear un volumen con nombre, el volumen se puede compartir con otros contenedores usando ese nombre.

Colocar un caparazón en un contenedor en ejecución (separado) #

Inicie sesión en un contenedor en ejecución #

Un usuario puede ingresar un contenedor en ejecución en un nuevo shell bash interactivo con el comando exec.

Digamos que un contenedor se llama jovial_morse, entonces puede obtener un shell bash pseudo-TTY interactivo ejecutando:

docker exec -it jovial_morse bash

Inicie sesión en un contenedor en ejecución con un usuario específico #

Si desea ingresar un contenedor como un usuario específico, puede configurarlo con el parámetro -u o --user. El nombre de usuario debe existir en el contenedor.

-u, --user Nombre de usuario o UID (formato: <nombre|uid>[:<grupo|gid>])

Este comando iniciará sesión en jovial_morse con el usuario dockeruser

docker exec -it -u dockeruser jovial_morse bash

Inicie sesión en un contenedor en ejecución como root #

Si desea iniciar sesión como root, simplemente use el parámetro -u root. El usuario raíz siempre existe.

docker exec -it -u root jovial_morse bash

Iniciar sesión en una imagen #

También puede iniciar sesión en una imagen con el comando run, pero esto requiere un nombre de imagen en lugar de un nombre de contenedor.

docker run -it dockerimage bash

Ejecutar un contenedor de forma interactiva #

Para ejecutar un contenedor de forma interactiva, pase las opciones -it:

$ docker run -it ubuntu:14.04 bash
root@8ef2356d919a:/# echo hi
hi
root@8ef2356d919a:/#

-i mantiene STDIN abierto, mientras que -t asigna un pseudo-TTY.

Separación de un contenedor #

Mientras está conectado a un contenedor en ejecución con un pty asignado (docker run -it ...), puede presionar ControlP - Control Q para separar.

Vincular un puerto de contenedor al host #

docker run -p "8080:8080" myApp
docker run -p "192.168.1.12:80:80" nginx
docker run -P myApp

Para usar los puertos en el host que han sido expuestos en una imagen (a través de la directiva EXPOSE Dockerfile, o la opción de línea de comando --expose para docker run), esos puertos deben vincularse al host usando el Opciones de línea de comando -p o -P. El uso de -p requiere que se especifique el puerto particular (y la interfaz de host opcional). El uso de la opción de línea de comando -P en mayúscula obligará a Docker a vincular todos los puertos expuestos en la imagen de un contenedor al host.

Establecer variables de entorno #

$ docker run -e "ENV_VAR=foo" ubuntu /bin/bash

Tanto -e como --env se pueden usar para definir variables de entorno dentro de un contenedor. Es posible proporcionar muchas variables de entorno utilizando un archivo de texto:

$ docker run --env-file ./env.list ubuntu /bin/bash

Ejemplo de archivo de variables de entorno:

# This is a comment
TEST_HOST=10.10.0.127

El indicador --env-file toma un nombre de archivo como argumento y espera que cada línea tenga el formato VARIABLE=VALUE, imitando el argumento pasado a --env. Las líneas de comentario solo necesitan tener el prefijo #.

Independientemente del orden de estas tres banderas, el --env-file se procesa primero, y luego las banderas -e/--env. De esta forma, cualquier variable de entorno proporcionada individualmente con -e o --env anulará las variables proporcionadas en el archivo de texto --env-var.

Pasando stdin al contenedor #

En casos como la restauración de un volcado de base de datos, o si desea enviar información a través de una canalización desde el host, puede usar el indicador -i como argumento para docker run o docker exec.

Por ejemplo, suponiendo que desea colocar en un cliente mariadb en contenedores un volcado de base de datos que tiene en el host, en un archivo dump.sql local, puede ejecutar el siguiente comando:

docker exec -i mariadb bash -c 'mariadb "-p$MARIADB_PASSWORD" ' < dump.sql

En general,

docker exec -i container command < file.stdin

O

docker exec -i container command <<EOF
inline-document-from-host-shell-HEREDOC-syntax
EOF

Anulando la directiva de punto de entrada de imagen #

docker run --name="test-app" --entrypoint="/bin/bash" example-app

Este comando anulará la directiva ENTRYPOINT de la imagen example-app cuando se crea el contenedor test-app. La directiva CMD de la imagen permanecerá sin cambios a menos que se especifique lo contrario:

docker run --name="test-app" --entrypoint="/bin/bash" example-app /app/test.sh

En el ejemplo anterior, se anularon tanto el ENTRYPOINT como el CMD de la imagen. Este proceso contenedor se convierte en /bin/bash /app/test.sh.

Especificando un nombre de host #

De forma predeterminada, los contenedores creados con docker run reciben un nombre de host aleatorio. Puede darle al contenedor un nombre de host diferente pasando el indicador –hostname:

docker run --hostname redbox -d ubuntu:14.04

Ejecutar un contenedor #

docker run hello-world

Esto obtendrá la imagen [hello-world] más reciente de Docker Hub (si aún no la tiene), creará un nuevo contenedor y lo ejecutará. Debería ver un mensaje que indica que su instalación parece estar funcionando correctamente.

Ejecutar un comando diferente en el contenedor #

docker run docker/whalesay cowsay 'Hello, StackExchange!'

Este comando le dice a Docker que cree un contenedor a partir de la imagen docker/whalesay y ejecute el comando cowsay 'Hello, StackExchange!' en él. Debería imprimir una imagen de una ballena diciendo ¡Hola, StackExchange! en tu terminal.

Si el punto de entrada en la imagen es el predeterminado, puede ejecutar cualquier comando que esté disponible en la imagen:

docker run docker/whalesay ls /

Si se ha cambiado durante la creación de la imagen, debe revertirlo al valor predeterminado

docker run --entrypoint=/bin/bash docker/whalesay -c ls /

Evite que el contenedor se detenga cuando no se estén ejecutando comandos #

Un contenedor se detendrá si no se ejecuta ningún comando en primer plano. El uso de la opción -t evitará que el contenedor se detenga, incluso cuando se separa con la opción -d.

docker run -t -d debian bash

Detener un contenedor #

docker stop mynginx

Además, la identificación del contenedor también se puede usar para detener el contenedor en lugar de su nombre.

Esto detendrá un contenedor en ejecución enviando la señal SIGTERM y luego la señal SIGKILL si es necesario.

Además, el comando kill se puede usar para enviar inmediatamente un SIGKILL o cualquier otra señal específica usando la opción -s.

docker kill mynginx

Señal especificada:

docker kill -s SIGINT mynginx

Detener un contenedor no lo elimina. Use docker ps -a para ver su contenedor detenido.

Contenedor en ejecución con límites de memoria/intercambio #

Establecer límite de memoria y deshabilitar límite de intercambio

docker run -it -m 300M --memory-swap -1 ubuntu:14.04 /bin/bash

Establezca tanto la memoria como el límite de intercambio. En este caso, el contenedor puede usar 300M de memoria y 700M de intercambio.

docker run -it -m 300M --memory-swap 1G ubuntu:14.04 /bin/bash

Agregar entrada de host al contenedor #

docker run --add-host="app-backend:10.15.1.24" awesome-app

Este comando agrega una entrada al archivo /etc/hosts del contenedor, que sigue el formato --add-host <name>:<address>. En este ejemplo, el nombre app-backend se resolverá como 10.15.1.24. Esto es particularmente útil para vincular componentes de aplicaciones dispares mediante programación.

Ejecutar otro comando en un contenedor en ejecución #

Cuando sea necesario, puede indicarle a Docker que ejecute comandos adicionales en un contenedor que ya se está ejecutando mediante el comando exec. Necesita la identificación del contenedor que puede obtener con docker ps.

docker exec 294fbc4c24b3 echo "Hola Mundo"

Puede adjuntar un shell interactivo si usa la opción -it.

docker exec -it 294fbc4c24b3 bash

Ejecutar aplicaciones GUI en un contenedor Linux #

De forma predeterminada, un contenedor Docker no podrá ejecutar una aplicación GUI.

Antes de eso, el zócalo X11 debe enviarse primero al contenedor, para que pueda usarse directamente. La variable de entorno DISPLAY también debe reenviarse:

docker run -v /tmp/.X11-unix:/tmp/.X11-unix -e DISPLAY=unix$DISPLAY <image-name>

Esto fallará al principio, ya que no configuramos los permisos para el host del servidor X:

cannot connect to X server unix:0

La forma más rápida (pero no la más segura) es permitir el acceso directamente con:

xhost +local:root

Después de terminar con el contenedor, podemos volver al estado original con:

xhost -local:root

Otra forma (más segura) es preparar un Dockerfile que creará una nueva imagen que usará nuestras credenciales de usuario para acceder al servidor X:

FROM <iamge-name>
MAINTAINER <you>

# Arguments picked from the command line!
ARG user
ARG uid
ARG gid

#Add new user with our credentials
ENV USERNAME ${user}
RUN useradd -m $USERNAME && \
        echo "$USERNAME:$USERNAME" | chpasswd && \
        usermod --shell /bin/bash $USERNAME && \
        usermod  --uid ${uid} $USERNAME && \
        groupmod --gid ${gid} $USERNAME

USER ${user}

WORKDIR /home/${user}

Al invocar docker build desde la línea de comando, tenemos que pasar las variables ARG que aparecen en el Dockerfile:

docker build --build-arg user=$USER --build-arg uid=$(id -u) --build-arg gid=$(id -g) -t <new-image-with-X11-enabled-name> -f <Dockerfile-for-X11> .

Ahora, antes de generar un nuevo contenedor, debemos crear un archivo xauth con permiso de acceso:

xauth nlist $DISPLAY | sed -e 's/^..../ffff/' | xauth -f /tmp/.docker.xauth nmerge -

Este archivo debe montarse en el contenedor al crearlo/ejecutarlo:

docker run -e DISPLAY=unix$DISPLAY -v /tmp/.X11-unix:/tmp/.X11-unix -v /tmp/.docker.xauth:/tmp/.docker.xauth:rw -e XAUTHORITY=/tmp/.docker.xauth
