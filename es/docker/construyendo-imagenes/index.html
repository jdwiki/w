Construyendo una imagen desde un Dockerfile #

Una vez que tenga un Dockerfile, puede crear una imagen a partir de él usando docker build. La forma básica de este comando es:

docker build -t nombre-imagen ruta

Si su Dockerfile no se llama Dockerfile, puede usar el indicador -f para dar el nombre del Dockerfile para compilar.

docker build -t image-name -f Dockerfile2 .

Por ejemplo, para crear una imagen denominada dockerbuild-example:1.0.0 a partir de un Dockerfile en el directorio de trabajo actual:

$ ls
Dockerfile Dockerfile2

$ docker build -t dockerbuild-example:1.0.0 .

$ docker build -t dockerbuild-example-2:1.0.0 -f Dockerfile2 .

Consulte la documentación de uso de docker build para obtener más opciones y configuraciones.

Un error común es crear un Dockerfile en el directorio de inicio del usuario (~). Esta es una mala idea porque durante docker build -t mytag. este mensaje aparecerá durante mucho tiempo:

Cargando contexto

La causa es que el demonio docker intenta copiar todos los archivos del usuario (tanto el directorio de inicio como sus subdirectorios). Evite esto especificando siempre un directorio para el Dockerfile.

Agregar un archivo .dockerignore al directorio de compilación es una buena práctica. Su sintaxis es similar a los archivos .gitignore y se asegurará de que solo se carguen los archivos y directorios deseados como contexto de la compilación.

Diferencia entre PUNTO DE ENTRADA y CMD #

Hay dos directivas Dockerfile para especificar qué comando ejecutar de forma predeterminada en las imágenes construidas. Si solo especifica CMD, Docker ejecutará ese comando utilizando el ENTRYPOINT predeterminado, que es /bin/sh -c. Puede anular uno o ambos el punto de entrada y/o el comando cuando inicia la imagen construida. Si especifica ambos, entonces ENTRYPOINT especifica el ejecutable de su proceso de contenedor, y CMD se proporcionará como los parámetros de ese ejecutable.

Por ejemplo, si su Dockerfile contiene

FROM ubuntu:16.04
CMD ["/bin/date"]

Entonces estás usando la directiva ENTRYPOINT predeterminada de /bin/sh -c, y ejecutando /bin/date con ese punto de entrada predeterminado. El comando de su proceso contenedor será /bin/sh -c /bin/date. Una vez que ejecute esta imagen, imprimirá de forma predeterminada la fecha actual

$ docker build -t test .
$ docker run test
Tue Jul 19 10:37:43 UTC 2016

Puede anular CMD en la línea de comando, en cuyo caso ejecutará el comando que haya especificado.

$ docker run test /bin/hostname
bf0274ec8820

Si especifica una directiva ENTRYPOINT, Docker usará ese ejecutable y la directiva CMD especifica los parámetros predeterminados del comando. Entonces, si su Dockerfile contiene:

FROM ubuntu:16.04
ENTRYPOINT ["/bin/echo"]
CMD ["Hello"]

Luego ejecutarlo producirá

$ docker build -t test .
$ docker run test
Hello

Puede proporcionar diferentes parámetros si lo desea, pero todos ejecutarán /bin/echo

$ docker run test Hi
Hi

Si desea anular el punto de entrada enumerado en su Dockerfile (es decir, si desea ejecutar un comando diferente a echo en este contenedor), debe especificar el parámetro --entrypoint en la línea de comando:

$ docker run --entrypoint=/bin/hostname test
b2c70e74df18

Por lo general, utiliza la directiva ENTRYPOINT para apuntar a la aplicación principal que desea ejecutar y CMD a los parámetros predeterminados.

Un archivo Docker simple #

FROM node:5

La directiva FROM especifica una imagen desde la que empezar. Se puede utilizar cualquier [referencia de imagen] válida.

WORKDIR /usr/src/app

La directiva WORKDIR establece el directorio de trabajo actual dentro del contenedor, equivalente a ejecutar cd dentro del contenedor. (Nota: RUN cd no cambiará el directorio de trabajo actual.)

RUN npm install cowsay knock-knock-jokes

RUN ejecuta el comando dado dentro del contenedor.

COPY cowsay-knockknock.js ./

COPY copia el archivo o directorio especificado en el primer argumento del contexto de construcción (la ruta pasada a docker build ruta< /code>) a la ubicación en el contenedor especificado por el segundo argumento.

CMD node cowsay-knockknock.js

CMD especifica un comando para ejecutar cuando la imagen es [ejecutar] y no se da ningún comando. Se puede anular [pasando un comando a docker run].

Hay muchas otras instrucciones y opciones; consulte la [referencia de Dockerfile] para obtener una lista completa.

[referencia de imagen]: https://www.wikiod.com/es/docker/gestion-de-imagenes#Imágenes de referencia [ejecutar]: https://www.wikiod.com/es/docker/ejecucion-de-contenedores#Ejecutar un contenedor [pasando un comando a docker run]: https://www.wikiod.com/es/docker/ejecucion-de-contenedores#Ejecutar un comando diferente en el contenedor different-command-in-the-container [Referencia del archivo Docker]: https://docs.docker.com/engine/reference/builder/

Empujar y tirar de una imagen a Docker Hub u otro Registro #

Las imágenes creadas localmente se pueden enviar a Docker Hub o a cualquier otro host de repositorio de Docker, conocido como registro. Use docker login para iniciar sesión en una cuenta de docker hub existente.

docker login

Login with your Docker ID to push and pull images from Docker Hub.
If you don't have a Docker ID, head over to https://hub.docker.com to create one.

Username: cjsimon
Password:
Login Succeeded

Se puede usar un registro docker diferente especificando un nombre de servidor. Esto también funciona para registros privados o autohospedados. Además, es posible usar un almacén de credenciales externo por seguridad.

docker login quay.io

A continuación, puede etiquetar y enviar imágenes al registro en el que ha iniciado sesión. Su repositorio debe especificarse como servidor/nombre de usuario/nombre de repositorio:etiqueta. Omitir el servidor actualmente está predeterminado en Docker Hub. (El registro predeterminado no se puede cambiar a otro proveedor, y no hay ningún plan para implementar esta función).

docker tag mynginx quay.io/cjsimon/mynginx:latest

Se pueden usar diferentes etiquetas para representar diferentes versiones o ramas de la misma imagen. Una imagen con múltiples etiquetas diferentes mostrará cada etiqueta en el mismo repositorio.

Utilice imágenes acoplables para ver una lista de imágenes instaladas en su máquina local, incluida su imagen recién etiquetada. Luego use push para cargarlo en el registro y tire para descargar la imagen.

docker push quay.io/cjsimon/mynginx:latest

Todas las etiquetas de una imagen se pueden extraer especificando la opción -a

docker pull quay.io/cjsimon/mynginx:latest

Exponiendo un puerto en el Dockerfile #

EXPOSE <port> [<port>...]

De la documentación de Docker:

La instrucción EXPOSE informa a Docker que el contenedor escucha en los puertos de red especificados en tiempo de ejecución. EXPOSE no hace que los puertos del contenedor sean accesibles para el host. Para hacer eso, debe usar el indicador -p para publicar un rango de puertos o el indicador -P para publicar todos los puertos expuestos. Puede exponer un número de puerto y publicarlo externamente con otro número.

Ejemplo: #

Dentro de su Dockerfile:

EXPOSE 8765

Para acceder a este puerto desde la máquina host, incluya este argumento en su comando docker run:

-p 8765:8765

ENTRYPOINT y CMD vistos como verbo y parámetro #

Suponga que tiene un Dockerfile que termina en

PUNTO DE ENTRADA [ "nethogs"] CMD ["wlan0"]

si construyes esta imagen con un

docker construido -t inspector.

inicie la imagen creada con dicho Dockerfile con un comando como

docker ejecutar -it --net=host --rm inspector

,nethogs monitoreará la interfaz llamada wlan0

Ahora, si desea monitorear la interfaz eth0 (o wlan1, o ra1…), debe hacer algo como

docker ejecutar -it --net=host --rm inspector eth0

o

docker run -it --net=host --rm inspector wlan1

Construyendo usando un proxy #

A menudo, cuando se crea una imagen de Docker, el Dockerfile contiene instrucciones que ejecutan programas para obtener recursos de Internet (wget, por ejemplo, para extraer una compilación binaria de programa en GitHub, por ejemplo).

Es posible indicarle a Docker que pase variables de entorno establecidas para que dichos programas realicen esas búsquedas a través de un proxy:

$ docker build --build-arg http_proxy=http://myproxy.example.com:3128 \
               --build-arg https_proxy=http://myproxy.example.com:3128 \
               --build-arg no_proxy=internal.example.com \
               -t test .

build-arg son variables de entorno que están disponibles solo en el momento de la compilación.
