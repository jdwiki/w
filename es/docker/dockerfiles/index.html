Los Dockerfiles son archivos que se utilizan para crear imágenes de Docker mediante programación. Le permiten crear una imagen de Docker de manera rápida y reproducible, por lo que son útiles para colaborar. Los archivos Docker contienen instrucciones para construir una imagen Docker. Cada instrucción se escribe en una fila y se da en la forma <INSTRUCCIÓN><argumento(s)>. Los Dockerfiles se utilizan para crear imágenes de Docker mediante el comando docker build.

Los Dockerfiles son de la forma:

# This is a comment
INSTRUCTION arguments
  • Los comentarios comienzan con #
  • Las instrucciones son solo en mayúsculas
  • La primera instrucción de un Dockerfile debe ser FROM para especificar la imagen base

Al compilar un Dockerfile, el cliente de Docker enviará un “contexto de compilación” al demonio de Docker. El contexto de compilación incluye todos los archivos y carpetas en el mismo directorio que el Dockerfile. Las operaciones COPY y ADD solo pueden usar archivos de este contexto.

Algunos archivos de Docker pueden comenzar con:

# escape=`

Esto se usa para indicarle al analizador Docker que use ` como carácter de escape en lugar de \. Esto es principalmente útil para los archivos de Windows Docker.

Mejores prácticas de Dockerfiles #

Operaciones comunes de grupo

Docker crea imágenes como una colección de capas. Cada capa solo puede agregar datos, incluso si estos datos dicen que se ha eliminado un archivo. Cada instrucción crea una nueva capa. Por ejemplo:

RUN apt-get -qq update
RUN apt-get -qq install some-package

Tiene un par de inconvenientes:

  • Creará dos capas, produciendo una imagen más grande.
  • El uso de apt-get update solo en una declaración RUN provoca problemas de almacenamiento en caché y, posteriormente, las instrucciones apt-get install pueden fallar. Suponga que luego modifica apt-get install agregando paquetes adicionales, luego docker interpreta las instrucciones iniciales y modificadas como idénticas y reutiliza el caché de los pasos anteriores. Como resultado, el comando apt-get update no se ejecuta porque su versión en caché se usa durante la compilación.

En su lugar, utilice:

RUN apt-get -qq update && \
    apt-get -qq install some-package

ya que esto solo produce una capa.

Mencionar al mantenedor

Esta suele ser la segunda línea del Dockerfile. Indica quién está a cargo y podrá ayudar.

LABEL maintainer John Doe <john.doe@example.com>

Si lo omite, no romperá su imagen. Pero tampoco ayudará a tus usuarios.

Sé conciso

Mantenga su Dockerfile breve. Si es necesaria una configuración compleja, considere usar un script dedicado o configurar imágenes base.

Instrucción EXPOSICIÓN #

EXPOSE <port> [<port>...]

La instrucción EXPOSE informa a Docker que el contenedor escucha en los puertos de red especificados en tiempo de ejecución. EXPOSE NO hace que los puertos del contenedor sean accesibles para el host. Para hacer eso, debe usar el indicador -p para publicar un rango de puertos o el indicador -P para publicar todos los puertos expuestos. Estos indicadores se utilizan en docker run [OPTIONS] IMAGE [COMMAND][ARG...] para exponer el puerto al host. Puede exponer un número de puerto y publicarlo externamente con otro número.

docker run -p 2500:80 <image name>

Este comando creará un contenedor con el nombre <imagen> y vinculará el puerto 80 del contenedor al puerto 2500 de la máquina host.

Para configurar la redirección de puertos en el sistema host, consulte el uso del indicador -P. La función de red de Docker admite la creación de redes sin necesidad de exponer puertos dentro de la red; para obtener información detallada, consulte la descripción general de esta función).

Instrucción CMD #

La instrucción CMD tiene tres formas:

CMD ["executable","param1","param2"] (exec form, this is the preferred form)
CMD ["param1","param2"] (as default parameters to ENTRYPOINT)
CMD command param1 param2 (shell form)

Solo puede haber una instrucción CMD en un Dockerfile. Si enumera más de un CMD, solo tendrá efecto el último CMD.

El objetivo principal de un CMD es proporcionar valores predeterminados para un contenedor en ejecución. Estos valores predeterminados pueden incluir un ejecutable, o pueden omitir el ejecutable, en cuyo caso también debe especificar una instrucción ENTRYPOINT.

Nota: si se utiliza CMD para proporcionar argumentos predeterminados para la instrucción ENTRYPOINT, tanto la instrucción CMD como la ENTRYPOINT deben especificarse con el formato de matriz JSON.

Nota: El formulario exec se analiza como una matriz JSON, lo que significa que debe usar comillas dobles (") alrededor de las palabras, no comillas simples (’).

Nota: A diferencia del formulario de shell, el formulario exec no invoca un shell de comandos. Esto significa que el procesamiento de shell normal no ocurre. Por ejemplo, CMD [ "echo", "$HOME" ] no hará sustitución de variables en $HOME. Si desea un procesamiento de shell, utilice el formulario de shell o ejecute un shell directamente, por ejemplo: CMD [ "sh", "-c", "echo $HOME" ].

Cuando se usa en los formatos shell o exec, la instrucción CMD establece el comando que se ejecutará al ejecutar la imagen.

Si usa la forma de shell de CMD, entonces el comando se ejecutará en /bin/sh -c:

FROM ubuntu
CMD echo "This is a test." | wc -

Si desea ejecutar su comando sin un shell, debe expresar el comando como una matriz JSON y proporcionar la ruta completa al ejecutable. Esta forma de matriz es el formato preferido de CMD. Cualquier parámetro adicional debe expresarse individualmente como cadenas en la matriz:

FROM ubuntu
CMD ["/usr/bin/wc","--help"]

Si desea que su contenedor ejecute el mismo ejecutable cada vez, entonces debería considerar usar ENTRYPOINT en combinación con CMD. Ver PUNTO DE ENTRADA.

Si el usuario especifica argumentos para ejecutar la ventana acoplable, anularán el valor predeterminado especificado en CMD.

Nota: no confunda RUN con CMD. RUN en realidad ejecuta un comando en el momento de la creación de la imagen y confirma el resultado; CMD no ejecuta nada en el momento de la compilación, pero especifica el comando previsto para la imagen.

La instrucción ENV y ARG #

ENV #

ENV <key> <value>
ENV <key>=<value> ...

La instrucción ENV establece la variable de entorno <clave> en el valor . Este valor estará en el entorno de todos los comandos de Dockerfile “descendientes” y también se puede reemplazar en línea en muchos.

La instrucción ENV tiene dos formas. La primera forma, ENV <clave> <valor>, establecerá una única variable en un valor. La cadena completa después del primer espacio se tratará como <valor>, incluidos caracteres como espacios y comillas.

La segunda forma, ENV <clave>=<valor> ..., permite configurar múltiples variables a la vez. Observe que la segunda forma usa el signo igual (=) en la sintaxis, mientras que la primera forma no lo hace. Al igual que el análisis de la línea de comandos, se pueden usar comillas y barras invertidas para incluir espacios dentro de los valores.

Por ejemplo:

ENV myName="John Doe" myDog=Rex\ The\ Dog \
    myCat=fluffy

y

ENV myName John Doe
ENV myDog Rex The Dog
ENV myCat fluffy

producirá los mismos resultados netos en el contenedor final, pero se prefiere la primera forma porque produce una sola capa de caché.

Las variables de entorno establecidas con ENV persistirán cuando se ejecute un contenedor desde la imagen resultante. Puede ver los valores usando docker inspect y cambiarlos usando docker run --env <key>=<value>.

##ARG

Si no desea conservar la configuración, utilice ARG en su lugar. ARG establecerá entornos solo durante la compilación. Por ejemplo, establecer

ENV DEBIAN_FRONTEND noninteractive

puede confundir a los usuarios de apt-get en una imagen basada en Debian cuando ingresan al contenedor en un contexto interactivo a través de docker exec -it the-container bash.

En su lugar, utilice:

ARG DEBIAN_FRONTEND noninteractive

Alternativamente, también puede establecer un valor para un solo comando usando:

RUN <key>=<value> <command>

Dockerfile de Hola Mundo #

Un Dockerfile mínimo se ve así:

FROM alpine
CMD ["echo", "Hello StackOverflow!"]

Esto indicará a Docker que cree una imagen basada en Alpine (FROM), una distribución mínima para contenedores, y que ejecute un comando específico (CMD) al ejecutar la imagen resultante.

Construirlo y ejecutarlo:

docker build -t hello .
docker run --rm hello

Esto generará:

Hello StackOverflow!

Instrucción de USUARIO #

USER daemon

La instrucción USER establece el nombre de usuario o UID que se usará cuando se ejecute la imagen y para cualquier instrucción RUN, CMD y ENTRYPOINT que le siga en el Dockerfile.

Instrucción WORKDIR #

WORKDIR /path/to/workdir

La instrucción WORKDIR establece el directorio de trabajo para cualquier instrucción RUN, CMD, ENTRYPOINT, COPY y ADD que le siguen en el Dockerfile. Si WORKDIR no existe, se creará incluso si no se usa en ninguna instrucción subsiguiente de Dockerfile.

Se puede usar varias veces en un Dockerfile. Si se proporciona una ruta relativa, será relativa a la ruta de la instrucción WORKDIR anterior. Por ejemplo:

WORKDIR /a
WORKDIR b
WORKDIR c
RUN pwd

La salida del comando pwd final en este Dockerfile sería /a/b/c.

La instrucción WORKDIR puede resolver las variables de entorno configuradas previamente usando ENV. Solo puede usar variables de entorno configuradas explícitamente en Dockerfile. Por ejemplo:

ENV DIRPATH /path
WORKDIR $DIRPATH/$DIRNAME
RUN pwd

La salida del comando pwd final en este Dockerfile sería /ruta/$DIRNAME

Instrucción VOLUMEN #

VOLUME ["/data"]

La instrucción VOLUME crea un punto de montaje con el nombre especificado y lo marca como que contiene volúmenes montados externamente desde un host nativo u otros contenedores. El valor puede ser una matriz JSON, VOLUME ["/var/log/"], o una cadena simple con múltiples argumentos, como VOLUME /var/log o VOLUME /var/log /var/db . Para obtener más información/ejemplos e instrucciones de montaje a través del cliente Docker, consulte la documentación Compartir directorios a través de volúmenes.

El comando docker run inicializa el volumen recién creado con cualquier dato que exista en la ubicación especificada dentro de la imagen base. Por ejemplo, considere el siguiente fragmento de Dockerfile:

FROM ubuntu
RUN mkdir /myvol
RUN echo "hello world" > /myvol/greeting
VOLUME /myvol

Este Dockerfile da como resultado una imagen que hace que Docker se ejecute, para crear un nuevo punto de montaje en /myvol y copiar el archivo de saludo en el volumen recién creado.

Nota: Si algún paso de la compilación cambia los datos dentro del volumen después de que se haya declarado, esos cambios se descartarán.

Nota: La lista se analiza como una matriz JSON, lo que significa que debe usar comillas dobles (") alrededor de las palabras, no comillas simples (’).

Instrucción COPIAR #

COPY tiene dos formas:

COPY <src>... <dest>
COPY ["<src>",... "<dest>"] (this form is required for paths containing whitespace)

La instrucción COPY copia nuevos archivos o directorios desde <src> y los agrega al sistema de archivos del contenedor en la ruta <dest>.

Se pueden especificar varios recursos <src>, pero deben ser relativos al directorio de origen que se está construyendo (el contexto de la construcción).

Cada <src> puede contener comodines y la coincidencia se realizará utilizando las reglas filepath.Match de Go. Por ejemplo:

COPY hom* /mydir/        # adds all files starting with "hom"
COPY hom?.txt /mydir/    # ? is replaced with any single character, e.g., "home.txt"

<dest> es una ruta absoluta o una ruta relativa a [WORKDIR](https://www.wikiod.com/es/docker/dockerfiles#Instrucción WORKDIR), en que la fuente se copiará dentro del contenedor de destino.

COPY test relativeDir/   # adds "test" to `WORKDIR`/relativeDir/
COPY test /absoluteDir/  # adds "test" to /absoluteDir/

Todos los archivos y directorios nuevos se crean con un UID y GID de 0.

Nota: si compila usando stdin (docker build - < somefile), no hay contexto de compilación, por lo que no se puede usar COPY.

COPY obedece las siguientes reglas:

  • La ruta <src> debe estar dentro del contexto de la compilación; no puede ‘COPIAR’ ../algo /algo, porque el primer paso de una compilación de ventana acoplable es enviar el directorio de contexto (y los subdirectorios) al demonio de ventana acoplable.

  • Si <src> es un directorio, se copia todo el contenido del directorio, incluidos los metadatos del sistema de archivos. Nota: El directorio en sí no se copia, solo su contenido.

  • Si <src> es cualquier otro tipo de archivo, se copia individualmente junto con sus metadatos. En este caso, si <dest> termina con una barra inclinada /, se considerará un directorio y el contenido de <src> se escribirá en <dest>/base(<src>).

  • Si se especifican múltiples recursos <src>, ya sea directamente o debido al uso de un comodín, entonces <dest> debe ser un directorio y debe terminar con una barra inclinada /.

  • Si <dest> no termina con una barra diagonal final, se considerará un archivo normal y el contenido de <src> se escribirá en <dest>.

  • Si <dest> no existe, se crea junto con todos los directorios que faltan en su ruta.

Instrucción ETIQUETA #

LABEL <key>=<value> <key>=<value> <key>=<value> ...

La instrucción LABEL agrega metadatos a una imagen. Una LABEL es un par clave-valor. Para incluir espacios dentro de un valor de LABEL, use comillas y barras invertidas como lo haría en el análisis de línea de comandos. Algunos ejemplos de uso:

LABEL "com.example.vendor"="ACME Incorporated"
LABEL com.example.label-with-value="foo"
LABEL version="1.0"
LABEL description="This text illustrates \
that label-values can span multiple lines."

Una imagen puede tener más de una etiqueta. Para especificar etiquetas múltiples, Docker recomienda combinar etiquetas en una sola instrucción ‘LABEL’ siempre que sea posible. Cada instrucción LABEL produce una nueva capa que puede resultar en una imagen ineficiente si usa muchas etiquetas. Este ejemplo da como resultado una sola capa de imagen.

LABEL multi.label1="value1" multi.label2="value2" other="value3"

Lo anterior también se puede escribir como:

LABEL multi.label1="value1" \
      multi.label2="value2" \
      other="value3"

Las etiquetas son aditivas, incluidas las LABEL en las imágenes FROM. Si Docker encuentra una etiqueta/clave que ya existe, el nuevo valor anula cualquier etiqueta anterior con claves idénticas.

Para ver las etiquetas de una imagen, use el comando de inspección de la ventana acoplable.

"Labels": {
    "com.example.vendor": "ACME Incorporated"
    "com.example.label-with-value": "foo",
    "version": "1.0",
    "description": "This text illustrates that label-values can span multiple lines.",
    "multi.label1": "value1",
    "multi.label2": "value2",
    "other": "value3"
},

Instrucción MANTENEDOR #

MAINTAINER <name>

La instrucción MAINTAINER le permite configurar el campo Autor de las imágenes generadas.

NO USE LA DIRECTIVA DE MANTENIMIENTO

De acuerdo con [Documentación oficial de Docker] (https://docs.docker.com/engine/reference/builder/#copy), la instrucción MAINTAINER está obsoleta. En su lugar, se debe utilizar la instrucción LABEL para definir el autor de las imágenes generadas. La instrucción LABEL es más flexible, permite configurar metadatos y se puede ver fácilmente con el comando docker inspect.

LABEL maintainer="someone@something.com"

DESDE Instrucción #

FROM <image>

O

FROM <image>:<tag>

O

FROM <image>@<digest>

La instrucción FROM establece la imagen base para las instrucciones posteriores. Como tal, un Dockerfile válido debe tener FROM como su primera instrucción. La imagen puede ser cualquier imagen válida; es especialmente fácil comenzar extrayendo una imagen de los repositorios públicos.

FROM debe ser la primera instrucción sin comentarios en el Dockerfile.

FROM puede aparecer varias veces dentro de un solo Dockerfile para crear varias imágenes. Simplemente tome nota de la última salida de ID de imagen por la confirmación antes de cada nuevo comando FROM.

Los valores de la etiqueta o resumen son opcionales. Si omite alguno de ellos, el constructor asume el más reciente de forma predeterminada. El constructor devuelve un error si no puede coincidir con el valor de la etiqueta.

Instrucción EJECUTAR #

RUN tiene 2 formas:

RUN <command> (shell form, the command is run in a shell, which by default is /bin/sh -c on Linux or cmd /S /C on Windows)
RUN ["executable", "param1", "param2"] (exec form)

La instrucción RUN ejecutará cualquier comando en una nueva capa encima de la imagen actual y confirmará los resultados. La imagen confirmada resultante se usará para el siguiente paso en el Dockerfile.

La superposición de instrucciones RUN y la generación de confirmaciones se ajustan a los conceptos básicos de Docker, donde las confirmaciones son económicas y los contenedores se pueden crear desde cualquier punto del historial de una imagen, como el control de código fuente.

El formulario exec hace posible evitar la manipulación de cadenas de shell y los comandos ‘EJECUTAR’ utilizando una imagen base que no contiene el ejecutable de shell especificado.

El shell predeterminado para el formulario de shell se puede cambiar usando el comando SHELL.

En el formulario de shell, puede usar un \ (barra invertida) para continuar con una única instrucción RUN en la siguiente línea. Por ejemplo, considere estas dos líneas:

RUN /bin/bash -c 'source $HOME/.bashrc ;\
echo $HOME'

Juntos son equivalentes a esta sola línea:

RUN /bin/bash -c 'source $HOME/.bashrc ; echo $HOME'

Nota: Para usar un shell diferente, que no sea ‘/bin/sh’, use el formulario exec pasando el shell deseado. Por ejemplo, EJECUTAR ["/bin/bash", "-c", "echo hola"]

Nota: El formulario exec se analiza como una matriz JSON, lo que significa que debe usar comillas dobles (“) alrededor de las palabras, no comillas simples (‘).

Nota: A diferencia del formulario de shell, el formulario exec no invoca un shell de comandos. Esto significa que el procesamiento de shell normal no ocurre. Por ejemplo, RUN [ "echo", "$HOME" ] no hará sustitución de variables en $HOME. Si desea un procesamiento de shell, utilice el formulario de shell o ejecute un shell directamente, por ejemplo: RUN [ "sh", "-c", "echo $HOME" ].

Nota: en el formulario JSON, es necesario escapar de las barras diagonales inversas. Esto es particularmente relevante en Windows donde la barra invertida es el separador de ruta. De lo contrario, la siguiente línea se trataría como un formulario de shell debido a que no es un JSON válido y fallaría de forma inesperada: RUN ["c:\windows\system32\tasklist.exe"]

La sintaxis correcta para este ejemplo es: EJECUTAR ["c:\\windows\\system32\\tasklist.exe"]

El caché para las instrucciones RUN no se invalida automáticamente durante la siguiente compilación. El caché para una instrucción como RUN apt-get dist-upgrade -y se reutilizará durante la próxima compilación. El caché para las instrucciones RUN se puede invalidar usando el indicador –no-cache, por ejemplo, docker build –no-cache.

Consulte la guía de prácticas recomendadas de Dockerfile para obtener más información.

El caché para las instrucciones RUN puede ser invalidado por las instrucciones ADD. Consulte a continuación para obtener más información.

Instrucción ONBUILD #

ONBUILD [INSTRUCTION]

La instrucción ‘ONBUILD’ agrega a la imagen una instrucción de disparo que se ejecutará en un momento posterior, cuando la imagen se use como base para otra construcción. El activador se ejecutará en el contexto de la compilación descendente, como si se hubiera insertado inmediatamente después de la instrucción FROM en el Dockerfile descendente.

Cualquier instrucción de compilación se puede registrar como disparador.

Esto es útil si está creando una imagen que se usará como base para crear otras imágenes, por ejemplo, un entorno de creación de aplicaciones o un demonio que puede personalizarse con una configuración específica del usuario.

Por ejemplo, si su imagen es un generador de aplicaciones de Python reutilizable, requerirá que se agregue el código fuente de la aplicación en un directorio en particular, y podría requerir que se llame a un script de compilación después de eso. No puede simplemente llamar a ADD y RUN ahora, porque aún no tiene acceso al código fuente de la aplicación, y será diferente para cada compilación de aplicación. Simplemente podría proporcionar a los desarrolladores de aplicaciones un Dockerfile repetitivo para copiar y pegar en su aplicación, pero eso es ineficiente, propenso a errores y difícil de actualizar porque se mezcla con el código específico de la aplicación.

La solución es usar ONBUILD para registrar instrucciones avanzadas para ejecutarlas más tarde, durante la siguiente etapa de construcción.

Así es como funciona:

Cuando encuentra una instrucción ONBUILD, el constructor agrega un disparador a los metadatos de la imagen que se está construyendo. La instrucción no afecta de otro modo a la compilación actual.

Al final de la compilación, se almacena una lista de todos los disparadores en el manifiesto de la imagen, bajo la clave OnBuild. Se pueden inspeccionar con el comando docker inspect. Más tarde, la imagen puede usarse como base para una nueva construcción, usando la instrucción FROM. Como parte del procesamiento de la instrucción FROM, el constructor posterior busca activadores ONBUILD y los ejecuta en el mismo orden en que se registraron. Si alguno de los disparadores falla, la instrucción ‘FROM’ se aborta, lo que a su vez hace que la compilación falle. Si todos los disparadores tienen éxito, la instrucción FROM se completa y la compilación continúa como de costumbre.

Los disparadores se borran de la imagen final después de ejecutarse. En otras palabras, no son heredados por construcciones de “nietos”.

Por ejemplo, podría agregar algo como esto:

[...]
ONBUILD ADD . /app/src
ONBUILD RUN /usr/local/bin/python-build --dir /app/src
[...]

Advertencia: No se permite encadenar instrucciones ONBUILD usando ONBUILD ONBUILD.

Advertencia: Es posible que la instrucción ‘ONBUILD’ no active las instrucciones ‘FROM’ o ‘MAINTAINER’.

Instrucción SEÑAL DE PARADA #

STOPSIGNAL signal

La instrucción STOPSIGNAL establece la señal de llamada al sistema que se enviará al contenedor para salir. Esta señal puede ser un número válido sin firmar que coincida con una posición en la tabla de llamadas al sistema del kernel, por ejemplo, 9, o un nombre de señal con el formato SIGNAME, por ejemplo, SIGKILL.

Instrucción HEALTHCHECK #

La instrucción HEALTHCHECK tiene dos formas:

HEALTHCHECK [OPTIONS] CMD command (check container health by running a command inside the container)
HEALTHCHECK NONE (disable any healthcheck inherited from the base image)

La instrucción HEALTHCHECK le dice a Docker cómo probar un contenedor para verificar que todavía funciona. Esto puede detectar casos como un servidor web que está atascado en un bucle infinito y no puede manejar nuevas conexiones, aunque el proceso del servidor aún se está ejecutando.

Cuando un contenedor tiene una verificación de salud especificada, tiene un estado de salud además de su estado normal. Este estado es inicialmente de inicio. Cada vez que pasa una verificación de estado, se vuelve saludable (cualquiera que sea el estado en el que se encontraba anteriormente). Después de un cierto número de fallas consecutivas, se vuelve inestable.

Las opciones que pueden aparecer antes de CMD son:

--interval=DURATION (default: 30s)
--timeout=DURATION (default: 30s)
--retries=N (default: 3)

La comprobación de estado se ejecutará primero en intervalos de segundos después de que se inicie el contenedor y, a continuación, de nuevo en intervalos de segundos después de que se complete cada comprobación anterior.

Si una sola ejecución de la comprobación tarda más de segundos de tiempo de espera, se considera que la comprobación ha fallado.

Se necesitan reintentos de fallas consecutivas de la comprobación de estado para que el contenedor se considere en mal estado.

Solo puede haber una instrucción HEALTHCHECK en un Dockerfile. Si enumera más de uno, solo tendrá efecto el último HEALTHCHECK.

El comando después de la palabra clave CMD puede ser un comando de shell (por ejemplo, HEALTHCHECK CMD /bin/check-running) o una matriz exec (como con otros comandos de Dockerfile; consulte, por ejemplo, ENTRYPOINT para obtener más información).

El estado de salida del comando indica el estado de salud del contenedor. Los valores posibles son:

  • 0: éxito - el contenedor está en buen estado y listo para usar
  • 1: malsano - el contenedor no funciona correctamente
  • 2: iniciando - el contenedor aún no está listo para usar, pero funciona correctamente

Si la sonda devuelve 2 (“iniciando”) cuando el contenedor ya salió del estado “inicial”, entonces se trata como “no saludable”.

Por ejemplo, para comprobar cada cinco minutos más o menos que un servidor web pueda servir la página principal del sitio en tres segundos:

HEALTHCHECK --interval=5m --timeout=3s \
  CMD curl -f http://localhost/ || exit 1

Para ayudar a depurar las sondas que fallan, cualquier texto de salida (codificado en UTF-8) que el comando escriba en stdout o stderr se almacenará en el estado de salud y se podrá consultar con docker inspect. Dicha salida debe ser breve (actualmente solo se almacenan los primeros 4096 bytes).

Cuando cambia el estado de salud de un contenedor, se genera un evento health_status con el nuevo estado.

La función HEALTHCHECK se agregó en Docker 1.12.

Instrucción SHELL #

SHELL ["executable", "parameters"]

La instrucción SHELL permite anular el shell predeterminado utilizado para la forma de shell de los comandos. El shell predeterminado en Linux es ["/bin/sh", "-c"], y en Windows es ["cmd", "/S", "/C"]. La instrucción SHELL debe escribirse en formato JSON en un Dockerfile.

La instrucción SHELL es particularmente útil en Windows, donde hay dos shells nativos de uso común y bastante diferentes: cmd y powershell, así como shells alternativos disponibles, incluido sh.

La instrucción SHELL puede aparecer varias veces. Cada instrucción SHELL anula todas las instrucciones SHELL anteriores y afecta a todas las instrucciones posteriores. Por ejemplo:

FROM windowsservercore

# Executed as cmd /S /C echo default
RUN echo default

# Executed as cmd /S /C powershell -command Write-Host default
RUN powershell -command Write-Host default

# Executed as powershell -command Write-Host hello
SHELL ["powershell", "-command"]
RUN Write-Host hello

# Executed as cmd /S /C echo hello
SHELL ["cmd", "/S"", "/C"]
RUN echo hello

Las siguientes instrucciones pueden verse afectadas por la instrucción SHELL cuando se utiliza su formato shell en un Dockerfile: RUN, CMD y ENTRYPOINT.

El siguiente ejemplo es un patrón común que se encuentra en Windows que se puede simplificar usando la instrucción SHELL:

...
RUN powershell -command Execute-MyCmdlet -param1 "c:\foo.txt"
...

El comando invocado por docker será:

cmd /S /C powershell -command Execute-MyCmdlet -param1 "c:\foo.txt"

Esto es ineficiente por dos razones. En primer lugar, se está invocando un procesador de comandos cmd.exe innecesario (también conocido como shell). En segundo lugar, cada instrucción “EJECUTAR” en el formulario de shell requiere un comando extra de powershell como prefijo del comando.

Para hacer esto más eficiente, se puede emplear uno de dos mecanismos. Una es usar la forma JSON del comando RUN como:

...
RUN ["powershell", "-command", "Execute-MyCmdlet", "-param1 \"c:\\foo.txt\""]
...

Si bien el formulario JSON no es ambiguo y no usa el cmd.exe innecesario, requiere más detalle a través de comillas dobles y escapes. El mecanismo alternativo es utilizar la instrucción SHELL y el formulario de shell, lo que crea una sintaxis más natural para los usuarios de Windows, especialmente cuando se combina con la directiva del analizador de escape:

# escape=`

FROM windowsservercore
SHELL ["powershell","-command"]
RUN New-Item -ItemType Directory C:\Example
ADD Execute-MyCmdlet.ps1 c:\example\
RUN c:\example\Execute-MyCmdlet -sample 'hello world'

La instrucción SHELL también se puede usar en Linux si se requiere un shell alternativo como zsh, csh, tcsh y otros.

Copiando documentos #

Para copiar archivos del contexto de compilación en una imagen de Docker, use la instrucción COPY:

COPY localfile.txt containerfile.txt

Si el nombre del archivo contiene espacios, utilice la sintaxis alternativa:

COPY ["local file", "container file"]

El comando COPY admite comodines. Se puede usar, por ejemplo, para copiar todas las imágenes al directorio images/:

COPY *.jpg images/

Nota: en este ejemplo, images/ puede no existir. En este caso, Docker lo creará automáticamente.

Exponiendo un puerto #

Para declarar puertos expuestos desde un Dockerfile use la instrucción EXPOSE:

EXPOSE 8080 8082

La configuración de puertos expuestos se puede anular desde la línea de comandos de Docker, pero es una buena práctica configurarlos explícitamente en Dockerfile, ya que ayuda a comprender lo que hace una aplicación.

Instalación de paquetes Debian/Ubuntu #

Ejecute la instalación en un solo comando de ejecución para fusionar la actualización y la instalación. Si agrega más paquetes más tarde, esto ejecutará la actualización nuevamente e instalará todos los paquetes necesarios. Si la actualización se ejecuta por separado, se almacenará en caché y las instalaciones del paquete pueden fallar. Es necesario configurar la interfaz como no interactiva y pasar -y para instalar para las instalaciones con secuencias de comandos. La limpieza y purga al final de la instalación minimiza el tamaño de la capa.

FROM debian

RUN apt-get update \
 && DEBIAN_FRONTEND=noninteractive apt-get install -y \
    git \
    openssh-client \
    sudo \
    vim \
    wget \
 && apt-get clean \
 && rm -rf /var/lib/apt/lists/*
