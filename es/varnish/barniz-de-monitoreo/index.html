Use varnishstat para monitorear las métricas numéricas de una instancia de Varnish que se está ejecutando actualmente. Su ubicación diferirá según su instalación. Ejecutar varnishstat -1 generará todas las métricas en un formato simple compatible con grep.

Hay otras utilidades disponibles para ver el estado actual del barniz y el registro: varnishtop, varnishlog, etc.

Métricas del cliente: tráfico entrante #

Las métricas del cliente cubren el tráfico entre el cliente y el caché de Varnish.

  • sess_conn - Número acumulado de conexiones.
  • client_req - Número acumulado de solicitudes de clientes.
  • sess_dropped - Conexiones caídas debido a una cola llena.

Supervise sess_conn y client_req para realizar un seguimiento del volumen de tráfico: si aumenta o disminuye, si aumenta, etc. Los cambios repentinos pueden indicar problemas.

Supervise sess_dropped para ver si la memoria caché descarta alguna sesión. Si es así, es posible que deba aumentar thread_pool_max.

varnishstat -1 | grep "sess_conn\|client_req \|sess_dropped"
MAIN.sess_conn              62449574         3.38 Sessions accepted
MAIN.client_req            184697229         9.99 Good client requests received
MAIN.sess_dropped                  0         0.00 Sessions dropped for thread

Rendimiento de la caché #

Quizás la métrica de rendimiento más importante es la tasa de aciertos.

Varnish enruta sus solicitudes entrantes como esta:

  • Hash, una solicitud almacenable en caché. Esto podría ser “acierto” o “fallo” según el estado de la memoria caché.
  • Hitpass, una solicitud no almacenable en caché.

Se obtendrá un hash con un miss y un hitpass desde el backend del servidor y se entregará. Un hash con un “hit” se entregará directamente desde el caché.

Métricas a monitorear:

  • cache_hit - Número de hashes con un hit en el caché.

  • cache_miss - Número de hashes con un error en el caché.

  • cache_hitpass - Número de hitpasses como arriba.

    varnishstat -1 | grep “cache_hit |cache_miss |cache_hitpass” MAIN.cache_hit 99032838 5.36 Cache hits MAIN.cache_hitpass 0 0.00 Cache hits for pass MAIN.cache_miss 42484195 2.30 Cache misses

Calcule la tasa de aciertos real de esta manera:

cache_hit / (cache_hit + cache_miss)

En este ejemplo, la tasa de aciertos es 0,7 o 70%. Desea mantener esto lo más alto posible. 70% es un número decente. Puede mejorar la tasa de aciertos aumentando la memoria y personalizando su vcl. También controle los grandes cambios en su tasa de aciertos.

Monitoreo de objetos en caché #

Supervisa los objetos almacenados en caché para ver con qué frecuencia caducan y si están “destruidos”.

  • n_expired - Número de objetos caducados.

  • n_lru_nuked - Últimos objetos bombardeados usados ​​recientemente. Número de objetos eliminados (eliminados) de la memoria caché por falta de espacio.

    varnishstat -1 | grep “n_expired|n_lru_nuked” MAIN.n_expired 42220159 . Number of expired objects MAIN.n_lru_nuked 264005 . Number of LRU nuked objects

El que se debe observar aquí es n_lru_nuked, si la tasa aumenta (la tasa, no solo el número), su caché está expulsando objetos cada vez más rápido debido a la falta de espacio. Necesitas aumentar el tamaño del caché.

El n_expired depende más de su aplicación. Un mayor tiempo de vida disminuirá este número pero, por otro lado, no renovará los objetos con tanta frecuencia. Además, el caché podría requerir más tamaño.

Monitoreo de hilos #

Debe realizar un seguimiento de algunas métricas de subprocesos para ver su Varnish Cache. ¿Se está quedando sin recursos del sistema operativo o está funcionando bien?

  • subprocesos - Número de subprocesos en todos los grupos.

  • threads_created - Número de hilos creados.

  • threads_failed: número de veces que Varnish no pudo crear un hilo.

  • threads_limited: número de veces que se obligó a Varnish a no crear un hilo porque estaba al máximo.

  • thread_queue_len - Longitud actual de la cola. Número de solicitudes en espera de un hilo.

  • sess_queued - Número de veces que no hubo subprocesos disponibles, por lo que se tuvo que poner en cola una solicitud.

    varnishstat -1 | grep “threads|thread_queue_len|sess_queued” MAIN.threads 100 . Total number of threads MAIN.threads_limited 1 0.00 Threads hit max MAIN.threads_created 3715 0.00 Threads created MAIN.threads_destroyed 3615 0.00 Threads destroyed MAIN.threads_failed 0 0.00 Thread creation failed MAIN.thread_queue_len 0 . Length of session queue MAIN.sess_queued 2505 0.00 Sessions queued for thread

Si thread_queue_len no es 0, significa que Varnish no tiene recursos y ha comenzado a poner en cola las solicitudes. Esto disminuirá el rendimiento de esas solicitudes. Tienes que investigar por qué.

Tenga cuidado también con threads_failed. Si esto aumenta, significa que su servidor no tiene recursos de alguna manera. Aumentar los números en threads_limited significa que es posible que necesite aumentar sus servidores thread_pool_max.

Monitoreo de métricas de back-end #

Hay una serie de métricas que describen la comunicación entre Varnish y sus backends.

Las métricas más importantes aquí podrían ser estas:

  • backend_busy - Número de estados de http 5xx recibidos por un backend. Con VCL puede configurar Varnish para probar otro backend si esto sucede.

  • backend_fail: número de veces que Varnish no pudo conectarse al backend. Esto puede tener varias causas (sin conexión TCP, mucho tiempo hasta el primer byte, mucho tiempo entre bytes). Si esto sucede, su backend no está en buen estado.

  • backend_unhealthy: número de veces que Varnish no pudo “hacer ping” al backend (no respondió con una respuesta HTTP 200.

    varnishstat -1 | grep “backend_” MAIN.backend_conn 86913481 4.70 Backend conn. success MAIN.backend_unhealthy 0 0.00 Backend conn. not attempted MAIN.backend_busy 0 0.00 Backend conn. too many MAIN.backend_fail 7 0.00 Backend conn. failures MAIN.backend_reuse 0 0.00 Backend conn. reuses MAIN.backend_toolate 0 0.00 Backend conn. was closed MAIN.backend_recycle 0 0.00 Backend conn. recycles MAIN.backend_retry 0 0.00 Backend conn. retry MAIN.backend_req 86961073 4.70 Backend requests made
