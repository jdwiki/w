Los puertos serie son una interfaz común para comunicarse con sensores externos o sistemas integrados como Arduinos. Las comunicaciones seriales modernas a menudo se implementan a través de conexiones USB utilizando adaptadores seriales USB. MATLAB proporciona funciones integradas para comunicaciones en serie, incluidos los protocolos RS-232 y RS-485. Estas funciones se pueden utilizar para puertos serie de hardware o conexiones serie USB “virtuales”. Los ejemplos aquí ilustran las comunicaciones seriales en MATLAB.

Parámetros #

Parámetro de puerto serie que hace
Velocidad de baudios Establece la velocidad de transmisión. El más común hoy en día es 57600, pero también se ven con frecuencia 4800, 9600 y 115200
Tamaño del búfer de entrada El número de bytes guardados en la memoria. Matlab tiene un FIFO, lo que significa que se descartarán nuevos bytes. El valor predeterminado es 512 bytes, pero se puede establecer fácilmente en 20 MB sin problemas. Solo hay unos pocos casos extremos en los que el usuario querría que esto fuera pequeño
Bytes disponibles El número de bytes que esperan ser leídos
Valores enviados El número de bytes enviados desde que se abrió el puerto
Valores recibidos El número de bytes leídos desde que se abrió el puerto
BytesAvailableFcn Especifique la función de devolución de llamada que se ejecutará cuando un número específico de bytes esté disponible en el búfer de entrada o se lea un terminador
BytesAvailableFcnCount Especifique el número de bytes que deben estar disponibles en el búfer de entrada para generar un evento de bytes disponibles
BytesAvailableFcnMode Especifique si el evento “bytes disponibles” se genera después de que un número específico de bytes esté disponible en el búfer de entrada, o después de que se lea un terminador

Crear un puerto serie en Mac/Linux/Windows #

% Define serial port with a baud rate of 115200
rate = 115200;
if ispc
    s = serial('COM1', 'BaudRate',rate);
elseif ismac
    % Note that on OSX the serial device is uniquely enumerated. You will
    % have to look at /dev/tty.* to discover the exact signature of your
    % serial device
    s = serial('/dev/tty.usbserial-A104VFT7', 'BaudRate',rate);
elseif isunix
    s = serial('/dev/ttyusb0', 'BaudRate',rate);
end

% Set the input buffer size to 1,000,000 bytes (default: 512 bytes).
s.InputBufferSize = 1000000;    

% Open serial port    
fopen(s);

Elegir tu modo de comunicación #

Matlab admite comunicación sincrónica y asincrónica con un puerto serie. Es importante elegir el modo de comunicación correcto. La elección dependerá de:

  • cómo se comporta el instrumento con el que se está comunicando.
  • qué otras funciones tendrá que hacer su programa principal (o GUI) además de administrar el puerto serie.

Definiré 3 casos diferentes para ilustrar, desde el más simple hasta el más exigente. Para los 3 ejemplos, el instrumento al que me estoy conectando es una placa de circuito con un inclinómetro, que puede funcionar en los 3 modos que describiré a continuación.

Modo 1: Síncrono (Maestro/Esclavo) #

Este modo es el más simple. Corresponde al caso donde la PC es el Maestro y el instrumento es el esclavo. El instrumento no envía nada al puerto serie por sí mismo, solo responde una respuesta después de que el Maestro (la PC, su programa) le haga una pregunta/comando. Por ejemplo:

  • La PC envía un comando: “Dame una medida ahora”
  • El instrumento recibe el comando, toma la medida y luego devuelve el valor de la medida a la línea serial: “El valor del inclinómetro es XXX”.

O

  • La PC envía un comando: “Cambiar de modo X a modo Y”
  • El instrumento recibe el comando, lo ejecuta y luego envía un mensaje de confirmación a la línea serial: “Comando ejecutado” (o “Comando NO ejecutado”). Esto se conoce comúnmente como una respuesta ACK/NACK (para “Reconocer (d)” / “NO Reconocido”).

Resumen: en este modo, el instrumento (el Slave) solo envía datos a la línea serial inmediatamente después de haber sido solicitado por la PC (el Master)

Modo 2: Asíncrono #

Ahora supongamos que encendí mi instrumento, pero es más que un sensor tonto. Supervisa constantemente su propia inclinación y mientras sea vertical (dentro de una tolerancia, digamos +/-15 grados), permanece en silencio. Si el dispositivo se inclina más de 15 grados y se acerca a la horizontal, envía un mensaje de alarma a la línea serie, seguido inmediatamente por una lectura de la inclinación. Siempre que la inclinación esté por encima del umbral, continúa enviando una lectura de inclinación cada 5 s.

Si su programa principal (o GUI) está constantemente “esperando” que llegue un mensaje a la línea serie, puede hacerlo bien… pero no puede hacer nada más mientras tanto. Si el programa principal es una GUI, es muy frustrante tener una GUI aparentemente “congelada” porque no aceptará ninguna entrada del usuario. Esencialmente, se convirtió en el Esclavo y el instrumento es el Maestro. A menos que tenga una forma elegante de controlar su GUI desde el instrumento, esto es algo que debe evitar. Afortunadamente, el modo de comunicación asincrónico le permitirá:

  • defina una función separada que le diga a su programa qué hacer cuando se recibe un mensaje
  • mantenga esta función en un rincón, solo se llamará y ejecutará cuando llegue un mensaje en la línea serial. El resto del tiempo, la GUI puede ejecutar cualquier otro código que tenga que ejecutar.

Resumen: En este modo, el instrumento puede enviar mensajes a la línea serial en cualquier momento (pero no necesariamente todo el tiempo). La PC no espera permanentemente a que se procese un mensaje. Se permite ejecutar cualquier otro código. Solo cuando llega un mensaje, activa una función que luego leerá y procesará este mensaje.

Modo 3: Streaming (Tiempo real) #

Ahora liberemos todo el poder de mi instrumento. Lo puse en un modo en el que constantemente enviará mediciones a la línea serial. Mi programa quiere recibir estos paquetes y mostrarlos en una curva o una pantalla digital. Si solo envía un valor cada 5 segundos como se indicó anteriormente, no hay problema, mantenga el modo anterior. Pero mi instrumento a pleno rendimiento envía un punto de datos a la línea serial a 1000 Hz, es decir, envía un nuevo valor cada milisegundo. Si permanezco en el modo asíncrono descrito anteriormente, existe un alto riesgo (en realidad, una certeza garantizada) de que la función especial que definimos para procesar cada nuevo paquete tarde más de 1 ms en ejecutarse (si desea trazar o mostrar el valor, las funciones gráficas son bastante lentas, ni siquiera considerando el filtrado o FFT de la señal). Significa que la función comenzará a ejecutarse, pero antes de que finalice, llegará un nuevo paquete y activará la función nuevamente. La segunda función se coloca en una cola para su ejecución, y solo comenzará cuando la primera haya terminado… pero en ese momento llegaron algunos paquetes nuevos y cada uno colocó una función para ejecutar en la cola. Puede prever rápidamente el resultado: en el momento en que estoy trazando los puntos 5, ya tengo cientos esperando para ser trazados también … la interfaz gráfica de usuario se ralentiza, finalmente se congela, la pila crece, los búferes se llenan, hasta que algo cede. Eventualmente, se queda con un programa completamente congelado o simplemente bloqueado.

Para superar esto, desconectaremos aún más el enlace de sincronización entre la PC y el instrumento. Dejaremos que el instrumento envíe datos a su propio ritmo, sin activar inmediatamente una función con cada llegada de paquete. El búfer del puerto serie solo acumulará los paquetes recibidos. La PC solo recopilará datos en el búfer a un ritmo que pueda manejar (un intervalo regular, configurado en el lado de la PC), hará algo con ellos (mientras el instrumento rellena el búfer), luego recopilará un nuevo lote de datos del búfer… y así sucesivamente.

Resumen: En este modo, el instrumento envía datos continuamente, que son recopilados por el búfer del puerto serie. A intervalos regulares, la PC recopila datos del búfer y hace algo con ellos. No existe un enlace de sincronización duro entre la PC y el instrumento. Ambos ejecutan sus tareas en su propio tiempo.

Lectura desde el puerto serie #

Suponiendo que creó el objeto de puerto serie s como en [este] (https://www.wikiod.com/es/matlab/uso-de-puertos-serie#Crear un puerto serie en Mac/Linux/Windows port-on-mac-linux-windows), luego

% Read one byte
data = fread(s, 1);

% Read all the bytes, version 1
data = fread(s);

% Read all the bytes, version 2
data = fread(s, s.BytesAvailable);

% Close the serial port
fclose(s);

Cerrar un puerto serie incluso si se pierde, elimina o sobrescribe #

Suponiendo que creó el objeto de puerto serie s como en [este] (https://www.wikiod.com/es/matlab/uso-de-puertos-serie#Crear un puerto serie en Mac/Linux/Windows port-on-mac-linux-windows), luego para cerrarlo

fclose(s)

Sin embargo, a veces puede perder accidentalmente el puerto (por ejemplo, borrar, sobrescribir, cambiar el alcance, etc.), y fclose(s) ya no funcionará. la solucion es facil

fclose(instrfindall)

Más información en instrfindall().

Escribiendo al puerto serie #

Suponiendo que creó el objeto de puerto serie s como en [este] (https://www.wikiod.com/es/matlab/uso-de-puertos-serie#Crear un puerto serie en Mac/Linux/Windows port-on-mac-linux-windows), luego

% Write one byte
fwrite(s, 255);

% Write one 16-bit signed integer
fwrite(s, 32767, 'int16');

% Write an array of unsigned 8-bit integers
fwrite(s,[48 49 50],'uchar');

% Close the serial port
fclose(s);

Procesamiento automático de datos recibidos desde un puerto serie #

Algunos dispositivos conectados a través de un puerto serie envían datos a su programa a una velocidad constante (transmisión de datos) o envían datos a intervalos impredecibles. Puede configurar el puerto serie para ejecutar una función automáticamente para manejar los datos siempre que lleguen. Esto se denomina “función de devolución de llamada” para el objeto de puerto serie.

Hay dos propiedades del puerto serie que se deben configurar para usar esta función: el nombre de la función que desea para la devolución de llamada (BytesAvailableFcn) y la condición que debe desencadenar la ejecución de la función de devolución de llamada (BytesAvailableFcnMode).

Hay dos formas de activar una función de devolución de llamada:

  1. Cuando se ha recibido una cierta cantidad de bytes en el puerto serie (generalmente se usa para datos binarios)
  2. Cuando se recibe un determinado carácter en el puerto serie (generalmente se usa para texto o datos ASCII)

Las funciones de devolución de llamada tienen dos argumentos de entrada necesarios, llamados obj y event. obj es el puerto serie. Por ejemplo, si desea imprimir los datos recibidos desde el puerto serie, defina una función para imprimir los datos llamada newdata:

function newdata(obj,event)
    [d,c] = fread(obj);  % get the data from the serial port
    % Note: for ASCII data, use fscanf(obj) to return characters instead of binary values
    fprintf(1,'Received %d bytes\n',c);
    disp(d)
end

Por ejemplo, para ejecutar la función newdata siempre que se reciban 64 bytes de datos, configure el puerto serie de la siguiente manera:

s = serial(port_name);
s.BytesAvailableFcnMode = 'byte';
s.BytesAvailableFcnCount = 64;
s.BytesAvailableFcn = @newdata;

Con datos de texto o ASCII, los datos generalmente se dividen en líneas con un “carácter de terminación”, al igual que el texto en una página. Para ejecutar la función newdata siempre que se reciba el carácter de retorno de carro, configure el puerto serie de esta manera:

s = serial(port_name);
s.BytesAvailableFcnMode = 'terminator';
s.Terminator = 'CR';  % the carriage return, ASCII code 13
s.BytesAvailableFcn = @newdata;
