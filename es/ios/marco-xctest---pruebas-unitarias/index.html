Adición de archivos de prueba al proyecto Xcode #

Al crear el proyecto #

Debe marcar “Incluir pruebas unitarias” en el cuadro de diálogo de creación del proyecto.

Después de crear el proyecto #

Si no revisó ese elemento al crear su proyecto, siempre puede agregar archivos de prueba más adelante. Para hacerlo:

1- Ve a la configuración de tu proyecto en Xcode

2- Ir a “Objetivos”

3- Haga clic en “Agregar destino”

4- En “Otro”, seleccione “Paquete de prueba de prueba de unidad táctil de cacao”

Al final, deberías tener un archivo llamado [Tu nombre de aplicación]Tests.swift. En Objective-C, debería tener dos archivos llamados [Nombre de su aplicación]Tests.h y [Nombre de su aplicación]Tests.m en su lugar.

El archivo [Your app name]Tests.swift o .m incluirá de forma predeterminada:

  • Importación de un módulo XCTest
  • Una clase [nombre de su aplicación]Pruebas que amplía XCTestCase
  • Métodos setUp, tearDown, testExample, testPerformanceExample

Agregar métodos de prueba #

Según Apple:

Métodos de prueba #

Un método de prueba es un método de instancia de una clase de prueba que comienza con el prefijo test, no acepta parámetros y devuelve void, por ejemplo, (void)testColorIsRed(). Un método de prueba ejerce el código en su proyecto y, si ese código no produce el resultado esperado, informa fallas utilizando un conjunto de API de aserción. Por ejemplo, el valor de retorno de una función podría compararse con un valor esperado o su prueba podría afirmar que el uso inadecuado de un método en una de sus clases genera una excepción.

Entonces agregamos un método de prueba usando “prueba” como el prefijo del método, como:

Rápido #

func testSomething() {

}

C objetivo #

- (void)testSomething {

}

Para probar realmente los resultados, usamos el método XCTAssert(), que toma una expresión booleana y, si es verdadero, marca la prueba como exitosa, de lo contrario, la marcará como fallida.

Digamos que tenemos un método en la clase View Controller llamado sum() que calcula la suma de dos números. Para probarlo, usamos este método:

Rápido #

func testSum(){
    let result = viewController.sum(4, and: 5)
    XCTAssertEqual(result, 9)
}

C objetivo #

- (void)testSum {
    int result = [viewController sum:4 and:5];
    XCTAssertEqual(result, 9);
}

Nota #

De manera predeterminada, no puede acceder a etiquetas, cuadros de texto u otros elementos de la interfaz de usuario de la clase View Controller desde la clase de prueba si primero se crearon en el archivo Storyboard. Esto se debe a que se inicializan en el método loadView() de la clase View Controller, y esto no se llamará durante la prueba. La mejor manera de llamar a loadView() y todos los demás métodos requeridos es accediendo a la propiedad view de nuestra propiedad viewController. Debe agregar esta línea antes de probar los elementos de la interfaz de usuario:

XCTAssertNotNil(viewController.view)

Escribiendo una clase de prueba #

import XCTest
@testable import PersonApp

class PersonTests: XCTestCase {
    func test_completeName() {
        let person = Person(firstName: "Josh", lastName: "Brown")
        XCTAssertEqual(person.completeName(), "Josh Brown")
    }
}

Ahora vamos a discutir lo que está pasando aquí. La línea import XCTest nos permitirá extender XCTestCase y usar XCTAssertEqual (entre otras afirmaciones). Extender XCTestCase y anteponer nuestro nombre de prueba con test garantizará que Xcode ejecute automáticamente esta prueba cuando ejecute las pruebas en el proyecto (⌘U o Producto > Prueba). La línea @testable import PersonApp importará nuestro destino PersonApp para que podamos probar y usar clases de él, como Person en nuestro ejemplo anterior. Y finalmente, nuestro XCTAssertEqual asegurará que person.completeName() sea igual a la cadena "Josh Brown".

Importar un módulo que se pueda probar #

Las clases, estructuras, enumeraciones y todos sus métodos son “internos” por defecto. Esto significa que solo se puede acceder a ellos desde el mismo módulo. Los casos de prueba están en un destino diferente y esto significa que están en un módulo diferente. Para poder acceder al método que desea probar, debe importar el módulo a probar utilizando la palabra clave @testable.

Digamos que tenemos un módulo principal llamado ToDo y queremos escribir pruebas para él. Importaríamos ese módulo así:

@testable import ToDo

Todos los métodos de prueba en el archivo con esta declaración de importación ahora pueden acceder a todas las clases, estructuras, enumeraciones internas y todos sus métodos internos del módulo ToDo.

Nunca debe agregar los archivos con los elementos que desea probar al objetivo de prueba porque eso puede generar errores difíciles de depurar.

Activar la carga y apariencia de la vista #

Ver cargando #

En una prueba para un controlador de vista, a veces desea activar la ejecución de loadView() o viewDidLoad(). Esto se puede hacer accediendo a la vista. Digamos que tiene una instancia de controlador de vista en su prueba llamada sut (sistema bajo prueba), entonces el código se vería así:

XCTAssertNotNil(sut.view)

Ver apariencia #

También puedes activar los métodos viewWillAppear(_:) y viewDidAppear(_:) agregando el siguiente código:

sut.beginAppearanceTransition(true, animated: true)
sut.endAppearanceTransition()
