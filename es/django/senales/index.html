Parámetros #

Clase/Método El por qué
Perfil de usuario() Clase La clase UserProfile amplía el modelo de usuario predeterminado de Django.
método create_profile() El método create_profile() se ejecuta cada vez que se lanza una señal [post_save] del modelo de usuario de Django 2.

Ahora, los detalles.

Las señales de Django son una forma de informar a su aplicación de ciertas tareas (como un modelo antes o después de guardar o eliminar) cuando se lleva a cabo.

Estas señales le permiten realizar acciones de su elección inmediatamente después de que se emite la señal.

Por ejemplo, cada vez que se crea un nuevo usuario de Django, el modelo de usuario emite una señal, con parámetros asociados como sender=User, lo que le permite orientar específicamente su escucha de señales a una actividad específica que sucede, en este caso, una nueva creación de usuario.

En el ejemplo anterior, la intención es crear un objeto UserProfile, inmediatamente después de crear un objeto User. Por lo tanto, al escuchar una señal post_save del modelo User (el modelo de usuario predeterminado de Django) específicamente, creamos un objeto UserProfile justo después de crear un nuevo User.

La documentación de Django proporciona una extensa documentación sobre todas las posibles señales disponibles.

Sin embargo, el ejemplo anterior es para explicar en términos prácticos un caso de uso típico cuando el uso de Señales puede ser una adición útil.

“Con un gran poder viene una gran responsabilidad”. Puede ser tentador tener señales esparcidas por toda su aplicación o proyecto solo porque son geniales. Bueno, no. El hecho de que sean geniales no los convierte en la solución ideal para cada situación simple que se te ocurra.

Las señales son excelentes para, como de costumbre, no todo. Iniciar/Cerrar sesión, las señales son geniales. Modelos clave que sueltan señales, como el modelo de usuario, si está bien.

La creación de señales para todos y cada uno de los modelos en su aplicación puede ser abrumadora en un punto y anular toda la idea del uso de Django Signals.

No uses señales cuando (basado en el libro Two Scoops of Django):

  • La señal se relaciona con un modelo en particular y se puede mover a uno de los métodos de ese modelo, posiblemente llamado por save().
  • La señal se puede reemplazar con un método de administrador de modelo personalizado.
  • La señal se relaciona con una vista en particular y se puede mover a esa vista

Puede estar bien usar señales cuando:

  • Su receptor de señal necesita hacer cambios en más de un modelo.
  • Desea enviar la misma señal desde varias aplicaciones y que un receptor común las maneje de la misma manera.
  • Desea invalidar un caché después de guardar un modelo.
  • Tiene un escenario inusual que necesita una devolución de llamada, y no hay otra forma de manejarlo además de usar una señal. Por ejemplo, desea activar algo basado en save() o init() del modelo de una aplicación de terceros. No puede modificar el código de terceros y extenderlo podría ser imposible, por lo que una señal proporciona un desencadenante para una devolución de llamada.

Ejemplo de extensión de perfil de usuario #

Este ejemplo es un fragmento tomado de Extending Django User Profile like a Pro

from django.db import models
from django.contrib.auth.models import User
from django.db.models.signals import post_save
 
class UserProfile(models.Model):
    user = models.OneToOneField(User, related_name='user')
    website = models.URLField(default='', blank=True)
    bio = models.TextField(default='', blank=True)

def create_profile(sender, **kwargs):
    user = kwargs["instance"]
    if kwargs["created"]:
        user_profile = UserProfile(user=user)
        user_profile.save()
post_save.connect(create_profile, sender=User)

Cómo encontrar si es una inserción o una actualización en la señal pre_save #

Al utilizar pre_save podemos determinar si una acción de guardar en nuestra base de datos se trataba de actualizar un objeto existente o crear uno nuevo.

Para lograr esto, puede verificar el estado del objeto modelo:

    @receiver(pre_save, sender=User)
    def pre_save_user(sender, instance, **kwargs): 
        if not instance._state.adding:
            print ('this is an update')
        else:
            print ('this is an insert')

Ahora, cada vez que se lleva a cabo una acción guardar, la señal pre_save se ejecutará e imprimirá:

  • esto es una actualización si la acción se deriva de una acción de actualización.
  • esto es una inserción si la acción se deriva de una acción de inserción.

Tenga en cuenta que este método no requiere ninguna consulta de base de datos adicional.

Heredar señales en modelos extendidos #

Las señales de Django están restringidas a firmas de clase precisas al momento del registro y, por lo tanto, los modelos subclasificados no se registran inmediatamente en la misma señal.

Tome este modelo y señal por ejemplo

class Event(models.Model):
    user = models.ForeignKey(User)


class StatusChange(Event):
    ...


class Comment(Event):
    ...


def send_activity_notification(sender, instance: Event, raw: bool, **kwargs):
    """
    Fire a notification upon saving an event
    """

    if not raw:
        msg_factory = MessageFactory(instance.id)
        msg_factory.on_activity(str(instance))
post_save.connect(send_activity_notification, Event)

Con los modelos extendidos, debe adjuntar manualmente la señal a cada subclase, de lo contrario, no se verán afectados.

post_save.connect(send_activity_notification, StatusChange)
post_save.connect(send_activity_notification, Comment)

Con Python 3.6, puede aprovechar algunos métodos de clase adicionales integrados en clases para automatizar este enlace.

class Event(models.Model):

    @classmethod
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        post_save.connect(send_activity_notification, cls)
