Parámetros #

Configuración Hace
Algunas configuraciones Configuraciones básicas útiles que van con Django-Allauth (que uso la mayor parte del tiempo). Para obtener más opciones de configuración, consulte Configuraciones
ACCOUNT_AUTHENTICATION_METHOD (=”nombre de usuario” o “correo electrónico” o “nombre de usuario_correo electrónico”) Especifica el método de inicio de sesión que se utilizará, ya sea que el usuario inicie sesión ingresando su nombre de usuario, dirección de correo electrónico o cualquiera de ambos. Establecer esto en “correo electrónico” requiere ACCOUNT_EMAIL_REQUIRED=True
CUENTA_EMAIL_CONFIRMACIÓN_EXPIRE_DAYS (=3) Determina la fecha de caducidad de los correos electrónicos de confirmación (número de días).
CUENTA_EMAIL_REQUIRED (=Falso) El usuario debe proporcionar una dirección de correo electrónico al registrarse. Esto va junto con la configuración ACCOUNT_AUTHENTICATION_METHOD
ACCOUNT_EMAIL_VERIFICATION (=”opcional”) Determina el método de verificación de correo electrónico durante el registro: elija uno entre “obligatorio”, “opcional” o “ninguno”. Cuando se establece en “obligatorio”, el usuario no puede iniciar sesión hasta que se verifique la dirección de correo electrónico. Elija “opcional” o “ninguno” para permitir inicios de sesión con una dirección de correo electrónico no verificada. En el caso de “opcional”, el correo de verificación de correo electrónico aún se envía, mientras que en el caso de “ninguno”, no se envía ningún correo de verificación de correo electrónico.
CUENTA_LOGIN_ATTEMPTS_LIMIT (=5) Número de intentos fallidos de inicio de sesión. Cuando se supera este número, el usuario no puede iniciar sesión durante los segundos ACCOUNT_LOGIN_ATTEMPTS_TIMEOUT especificados. Si bien esto protege la vista de inicio de sesión de allauth, no protege el inicio de sesión de administrador de Django contra la fuerza bruta.
CUENTA_LOGOUT_ON_PASSWORD_CHANGE (=Falso) Determina si el usuario se desconecta automáticamente o no después de cambiar o establecer su contraseña.
SOCIALACCOUNT_PROVIDERS (= dict) Diccionario que contiene configuraciones específicas del proveedor.

Manera fácil: python-social-auth #

python-social-auth es un marco que simplifica el mecanismo de autorización y autenticación social. Contiene muchos backends sociales (Facebook, Twitter, Github, LinkedIn, etc.)

INSTALAR EN PC

Primero necesitamos instalar el paquete python-social-auth con

pip install python-social-auth

o descargar el código de github. Ahora es un buen momento para agregar esto a su archivo requirements.txt.

CONFIGURAR ajustes.py

En la configuración.py agregue:

INSTALLED_APPS = (
    ...
    'social.apps.django_app.default',
    ...
)

CONFIGURACIÓN DE BACKENDS

AUTHENTICATION_BACKENDS contiene los backends que usaremos, y solo tenemos que poner lo que necesitamos.

AUTHENTICATION_BACKENDS = (
    'social.backends.open_id.OpenIdAuth',
    'social.backends.google.GoogleOpenId',
    'social.backends.google.GoogleOAuth2',
    'social.backends.google.GoogleOAuth',
    'social.backends.twitter.TwitterOAuth',
    'social.backends.yahoo.YahooOpenId',
    ...
    'django.contrib.auth.backends.ModelBackend',
)

Es posible que su proyecto settings.py aún no tenga un campo AUTHENTICATION_BACKENDS. Si ese es el caso, agregue el campo. Asegúrese de no perderse 'django.contrib.auth.backends.ModelBackend', ya que maneja el inicio de sesión por nombre de usuario/contraseña.

Si usamos, por ejemplo, Facebook y Linkedin Backends, debemos agregar las claves API

SOCIAL_AUTH_FACEBOOK_KEY = 'YOURFACEBOOKKEY'
SOCIAL_AUTH_FACEBOOK_SECRET = 'YOURFACEBOOKSECRET'

y

SOCIAL_AUTH_LINKEDIN_KEY = 'YOURLINKEDINKEY'
SOCIAL_AUTH_LINKEDIN_SECRET = 'YOURLINKEDINSECRET'

Nota: Puede obtener las claves necesarias en desarrolladores de Facebook y desarrolladores de Linkedin y aquí puede ver la lista completa y su respectiva forma de especificar la clave API y la clave Secreto.

Nota sobre las claves secretas: Las claves secretas deben mantenerse en secreto. Aquí hay una explicación de Stack Overflow que es útil. [Este tutorial][6] es útil para aprender sobre las variables ambientales.

TEMPLATE_CONTEXT_PROCESSORS ayudará con las redirecciones, backends y otras cosas, pero al principio solo necesitamos esto:

TEMPLATE_CONTEXT_PROCESSORS = (
    ...
    'social.apps.django_app.context_processors.backends',
    'social.apps.django_app.context_processors.login_redirect',
    ...
)

En Django 1.8, la configuración de TEMPLATE_CONTEXT_PREPROCESSORS como se muestra arriba estaba en desuso. Si este es su caso, lo agregará dentro del dictado TEMPLATES. El tuyo debería ser algo similar a esto:

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [os.path.join(BASE_DIR, "templates")],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
                'social.apps.django_app.context_processors.backends',
                'social.apps.django_app.context_processors.login_redirect',
            ],
        },
    },
]

USO DE UN USUARIO PERSONALIZADO

Si está utilizando un modelo de usuario personalizado y desea asociarse con él, simplemente agregue la siguiente línea (todavía en settings.py)

SOCIAL_AUTH_USER_MODEL = 'somepackage.models.CustomUser'

CustomUser es un modelo que hereda o se abstrae del usuario predeterminado.

CONFIGURAR urls.py

# if you haven't imported inlcude make sure you do so at the top of your file
from django.conf.urls import url, include

urlpatterns = patterns('',
    ...
    url('', include('social.apps.django_app.urls', namespace='social'))
    ...
)

A continuación, debe sincronizar la base de datos para crear los modelos necesarios:

./manage.py migrate

** ¡Finalmente podemos jugar! **

en alguna plantilla necesitas agregar algo como esto:

    <a href="{% url 'social:begin' 'facebook' %}?next={{ request.path }}">Login with Facebook</a>
    <a href="{% url 'social:begin' 'linkedin' %}?next={{ request.path }}">Login with Linkedin</a>

si usa otro backend, simplemente cambie ‘facebook’ por el nombre del backend.

Cierre de sesión de los usuarios

Una vez que haya iniciado la sesión de los usuarios, es probable que desee crear la funcionalidad para volver a cerrar la sesión. En alguna plantilla, probablemente cerca de donde se mostró la plantilla de inicio de sesión, agregue la siguiente etiqueta:

<a href="{% url 'logout' %}">Logout</a>

o

<a href="/logout">Logout</a>

Deberá editar su archivo urls.py con un código similar a:

url(r'^logout/$', views.logout, name='logout'),

Por último, edite su archivo views.py con un código similar a:

def logout(request):
    auth_logout(request)
    return redirect('/')

[6]: http://www.marinamele.com/taskbuster-django-tutorial/settings- different-environments-version-control

Uso de Django Allauth #

Para todos mis proyectos, Django-Allauth siguió siendo fácil de configurar y viene listo para usar con muchas características que incluyen, entre otras:

  • Más de 50 autenticaciones de redes sociales
  • Mezcle el registro de cuentas locales y sociales.
  • Múltiples cuentas sociales
  • Registro instantáneo opcional para cuentas sociales, sin preguntas
  • Gestión de direcciones de correo electrónico (múltiples direcciones de correo electrónico, configuración de primario)
  • Flujo de contraseña olvidada Flujo de verificación de dirección de correo electrónico

Si está interesado en ensuciarse las manos, Django-Allauth se quita del camino, con configuraciones adicionales para modificar el proceso y el uso de su sistema de autenticación.

Los pasos a continuación asumen que estás usando Django 1.10+

Pasos de configuración:

pip instalar django-allauth

En su archivo settings.py, realice los siguientes cambios:

# Specify the context processors as follows:
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                # Already defined Django-related contexts here

                # `allauth` needs this from django. It is there by default,
                # unless you've devilishly taken it away.
                'django.template.context_processors.request',
            ],
        },
    },
]

AUTHENTICATION_BACKENDS = (
    # Needed to login by username in Django admin, regardless of `allauth`
    'django.contrib.auth.backends.ModelBackend',

    # `allauth` specific authentication methods, such as login by e-mail
    'allauth.account.auth_backends.AuthenticationBackend',
)

INSTALLED_APPS = (
# Up here is all your default installed apps from Django

# The following apps are required:
'django.contrib.auth',
'django.contrib.sites',

'allauth',
'allauth.account',
'allauth.socialaccount',

# include the providers you want to enable:
'allauth.socialaccount.providers.google',
'allauth.socialaccount.providers.facebook',
)

# Don't forget this little dude.
SITE_ID = 1

Terminado con los cambios en el archivo settings.py anterior, muévase al archivo urls.py. Puede ser su yourapp/urls.py o su ProjectName/urls.py. Normalmente, prefiero ProjectName/urls.py.

urlpatterns = [
    # other urls here
    url(r'^accounts/', include('allauth.urls')),
    # other urls here
]

Simplemente agregando include('allauth.urls'), obtendrá estas URL de forma gratuita:

^accounts/ ^ ^signup/$ [name='account_signup']
^accounts/ ^ ^login/$ [name='account_login']
^accounts/ ^ ^logout/$ [name='account_logout']
^accounts/ ^ ^password/change/$ [name='account_change_password']
^accounts/ ^ ^password/set/$ [name='account_set_password']
^accounts/ ^ ^inactive/$ [name='account_inactive']
^accounts/ ^ ^email/$ [name='account_email']
^accounts/ ^ ^confirm-email/$ [name='account_email_verification_sent']
^accounts/ ^ ^confirm-email/(?P<key>[-:\w]+)/$ [name='account_confirm_email']
^accounts/ ^ ^password/reset/$ [name='account_reset_password']
^accounts/ ^ ^password/reset/done/$ [name='account_reset_password_done']
^accounts/ ^ ^password/reset/key/(?P<uidb36>[0-9A-Za-z]+)-(?P<key>.+)/$ [name='account_reset_password_from_key']
^accounts/ ^ ^password/reset/key/done/$ [name='account_reset_password_from_key_done']
^accounts/ ^social/
^accounts/ ^google/
^accounts/ ^twitter/
^accounts/ ^facebook/
^accounts/ ^facebook/login/token/$ [name='facebook_login_by_token']

Finalmente, ejecute python ./manage.py migrate para confirmar las migraciones de Django-allauth en la base de datos.

Como de costumbre, para poder iniciar sesión en su aplicación utilizando cualquier red social que haya agregado, deberá agregar los detalles de la cuenta social de la red.

Inicie sesión en el administrador de Django (localhost: 8000/admin) y en Aplicaciones sociales en los detalles de su cuenta social.

Es posible que necesite cuentas en cada proveedor de autenticación para obtener detalles para completar en las secciones de aplicaciones sociales.

Para obtener configuraciones detalladas de lo que puede tener y modificar, consulte la página de configuraciones.
