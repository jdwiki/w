Cuando usamos CBV, a menudo necesitamos saber exactamente qué métodos podemos sobrescribir para cada clase genérica. Esta página de la documentación de django enumera todas las clases genéricas con todos sus métodos aplanados y los atributos de clase que podemos usar.

Además, el sitio web Classy Class Based View proporciona la misma información con una agradable interfaz interactiva.

Creación de formularios y objetos #

Escribir una vista para crear un objeto puede ser bastante aburrido. Tiene que mostrar un formulario, tiene que validarlo, tiene que guardar el elemento o devolver el formulario con un error. A menos que utilice una de las [vistas de edición genéricas][vistas_de_edición].

aplicación/vistas.py #

from django.core.urlresolvers import reverse_lazy
from django.views.generic.edit import CreateView, UpdateView, DeleteView
from .models import Pokemon


class PokemonCreate(CreateView):
    model = Pokemon
    fields = ['name', 'species']


class PokemonUpdate(UpdateView):
    model = Pokemon
    fields = ['name', 'species']


class PokemonDelete(DeleteView):
    model = Pokemon
    success_url = reverse_lazy('pokedex')

CreateView y UpdateView tienen dos atributos obligatorios, modelo y campos. De manera predeterminada, ambos usan un nombre de plantilla basado en el nombre del modelo con el sufijo ‘_form’. Solo puede cambiar el sufijo con el atributo template_name_suffix. DeleteView muestra un mensaje de confirmación antes de eliminar el objeto.

Tanto UpdateView como DeleteView necesitan recuperar el objeto. Usan el mismo método que DetailView, extrayendo la variable de la URL y haciendo coincidir los campos del objeto.

app/templates/app/pokemon_form.html (extracto) #

<form action="" method="post">
    {% csrf_token %}
    {{ form.as_p }}
    <input type="submit" value="Save" />
</form>

form contiene el formulario con todos los campos necesarios. Aquí, se mostrará con un párrafo para cada campo debido a as_p.

app/templates/app/pokemon_confirm_delete.html (extracto) #

<form action="" method="post">
    {% csrf_token %}
    <p>Are you sure you want to delete "{{ object }}"?</p>
    <input type="submit" value="Confirm" />
</form>

Se requiere la etiqueta csrf_token debido a la protección de Django contra la falsificación de solicitudes. La acción de atributo se deja vacía ya que la URL que muestra el formulario es la misma que la que maneja la eliminación/guardado.

Quedan dos problemas con el modelo, si se usa lo mismo que con el ejemplo de lista y detalle. Primero, crear y actualizar se quejará de una URL de redirección faltante. Eso se puede resolver agregando un get_absolute_url al modelo de pokemon. El segundo problema es que la confirmación de eliminación no muestra información significativa. Para resolver esto, la solución más sencilla es agregar una representación de cadena.

app/modelos.py #

from django.db import models
from django.urls import reverse
from django.utils.encoding import python_2_unicode_compatible


@python_2_unicode_compatible
class Pokemon(models.Model):
    name = models.CharField(max_length=24)
    species = models.CharField(max_length=48)

    def get_absolute_url(self):
        return reverse('app:pokemon', kwargs={'pk':self.pk})

    def __str__(self):
        return self.name

El decorador de la clase se asegurará de que todo funcione sin problemas en python 2.

Vistas de lista y detalles #

Las vistas de plantilla están bien para la página estática y podría usarlas para todo con get_context_data pero sería apenas mejor que usar la función como vistas.

Introduzca [ListView][list_view] y [DetailView][detail_view]

app/modelos.py #

from django.db import models

class Pokemon(models.Model):
    name = models.CharField(max_length=24)
    species = models.CharField(max_length=48)
    slug = models.CharField(max_length=48)

aplicación/vistas.py #

from django.views.generic import ListView, DetailView
from .models import Pokemon


class PokedexView(ListView):
    """ Provide a list of Pokemon objects """
    model = Pokemon
    paginate_by = 25

class PokemonView(DetailView):
    model = Pokemon

Eso es todo lo que necesita para generar una vista que enumere todos sus objetos de modelos y vistas de un elemento singular. La lista está incluso paginada. Puede proporcionar template_name si desea algo específico. De forma predeterminada, se genera a partir del nombre del modelo.

app/templates/app/pokemon_list.html #

<!DOCTYPE html>
<title>Pokedex</title>
<ul>{% for pokemon in pokemon_list %}
    <li><a href="{% url "app:pokemon" pokemon.pk %}">{{ pokemon.name }}</a>
        &ndash; {{ pokemon.species }}
</ul>

El contexto se completa con la lista de objetos bajo dos nombres, object_list y una segunda compilación a partir del nombre del modelo, aquí pokemon_list. Si ha paginado la lista, también debe cuidar el enlace siguiente y anterior. El objeto [Paginator][paginator] puede ayudar con eso, también está disponible en los datos de contexto.

aplicación/plantillas/aplicación/pokemon_detail.html #

<!DOCTYPE html>
<title>Pokemon {{ pokemon.name }}</title>
<h1>{{ pokemon.name }}</h1>
<h2>{{ pokemon.species }} </h2>

Como antes, el contexto se completa con su objeto modelo bajo el nombre objeto y pokemon, el segundo se deriva del nombre del modelo.

app/urls.py #

from django.conf.urls import url
from . import views

app_name = 'app'
urlpatterns = [
    url(r'^pokemon/$', views.PokedexView.as_view(), name='pokedex'),
    url(r'^pokemon/(?P<pk>\d+)/$', views.PokemonView.as_view(), name='pokemon'),
]

En este fragmento, la URL de la vista de detalles se crea utilizando la clave principal. También es posible usar un slug como argumento. Esto le da una URL más atractiva que es más fácil de recordar. Sin embargo, requiere la presencia de un campo llamado slug en su modelo.

url(r'^pokemon/(?P<slug>[A-Za-z0-9_-]+)/$', views.PokemonView.as_view(), name='pokemon'),

Si un campo llamado slug no está presente, puede usar la configuración slug_field en DetailView para apuntar a un campo diferente.

Para la paginación, use una página para obtener parámetros o coloque una página directamente en la URL.

Vistas basadas en clases #

Las vistas basadas en clases le permiten concentrarse en lo que hace que sus vistas sean especiales.

Una página estática acerca de puede no tener nada especial, excepto la plantilla utilizada. ¡Utilice una [TemplateView][template_view]! Todo lo que tiene que hacer es establecer un nombre de plantilla. Trabajo hecho. Próximo.

vistas.py #

from django.views.generic import TemplateView


class AboutView(TemplateView):
    template_name = "about.html"

urls.py #

from django.conf.urls import url
from . import views

urlpatterns = [
    url('^about/', views.AboutView.as_view(), name='about'),
]

Observe cómo no usamos directamente AboutView en la URL. Eso es porque se espera un invocable y eso es exactamente lo que devuelve as_view().

Datos de contexto #

A veces, su plantilla necesita un poco más de información. Por ejemplo, nos gustaría tener al usuario en el encabezado de la página, con un enlace a su perfil al lado del enlace de cierre de sesión. En estos casos, utilice el método get_context_data.

vistas.py #

class BookView(DetailView):
    template_name = "book.html"

    def get_context_data(self, **kwargs)
        """ get_context_data let you fill the template context """
        context = super(BookView, self).get_context_data(**kwargs)
        # Get Related publishers
        context['publishers'] = self.object.publishers.filter(is_active=True)
        return context

Debe llamar al método get_context_data en la superclase y devolverá la instancia de contexto predeterminada. Cualquier elemento que agregue a este diccionario estará disponible para la plantilla.

libro.html #

<h3>Active publishers</h3>
<ul>
  {% for publisher in publishers %}
    <li>{{ publisher.name }}</li>
  {% endfor %}
</ul>

Una vista, varias formas #

Aquí hay un ejemplo rápido del uso de múltiples formularios en una vista de Django.

from django.contrib import messages
from django.views.generic import TemplateView

from .forms import AddPostForm, AddCommentForm
from .models import Comment

class AddCommentView(TemplateView):

    post_form_class = AddPostForm
    comment_form_class = AddCommentForm
    template_name = 'blog/post.html'

    def post(self, request):
        post_data = request.POST or None
        post_form = self.post_form_class(post_data, prefix='post')
        comment_form = self.comment_form_class(post_data, prefix='comment')

        context = self.get_context_data(post_form=post_form,
                                        comment_form=comment_form)

        if post_form.is_valid():
            self.form_save(post_form)
        if comment_form.is_valid():
            self.form_save(comment_form)

        return self.render_to_response(context)     

    def form_save(self, form):
        obj = form.save()
        messages.success(self.request, "{} saved successfully".format(obj))
        return obj

    def get(self, request, *args, **kwargs):
        return self.post(request, *args, **kwargs)
