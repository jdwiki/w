Parámetros #

comando django-admin Detalles

hacermigraciones Generar migraciones para todas las aplicaciones
makemigrations --merge Resolver conflictos de migración para todas las aplicaciones
makemigrations --merge <mi_aplicación> Resolver conflictos de migración para my_app
makemigrations --name <nombre_migración> <mi_aplicación> Genere una migración para mi_aplicación con el nombre nombre_migración
migrar <mi_aplicación> Aplicar migraciones pendientes de my_app a la base de datos
migrar Aplicar todas las migraciones pendientes a la base de datos

migrar <mi_aplicación> cero Cancelar todas las migraciones en my_app

showmigraciones Muestra todas las migraciones para todas las aplicaciones
showmigraciones <mi_aplicación> Muestra todas las migraciones en my_app

Migraciones manuales #

En ocasiones, las migraciones generadas por Django no son suficientes. Esto es especialmente cierto cuando desea realizar migraciones de datos.

Por ejemplo, tengamos ese modelo:

class Article(models.Model):
    title = models.CharField(max_length=70)

Este modelo ya tiene datos existentes y ahora desea agregar un SlugField:

class Article(models.Model):
    title = models.CharField(max_length=70)
    slug = models.SlugField(max_length=70)

Creó las migraciones para agregar el campo, pero ahora le gustaría configurar el slug para todos los artículos existentes, de acuerdo con su “título”.

Por supuesto, podrías hacer algo como esto en la terminal:

$ django-admin shell
>>> from my_app.models import Article
>>> from django.utils.text import slugify
>>> for article in Article.objects.all():
...     article.slug = slugify(article.title)
...     article.save()
...
>>>

Pero tendrá que hacer esto en todos sus entornos (es decir, el escritorio de su oficina, su computadora portátil, …), todos sus compañeros de trabajo también tendrán que hacerlo, y tendrá que pensar en ello en la puesta en escena y al empujar En Vivo.

Para hacerlo de una vez por todas, lo haremos en una migración. Primero crea una migración vacía:

$ django-admin makemigrations --empty app_name

Esto creará un archivo de migración vacío. Ábrelo, contiene un esqueleto base. Digamos que su migración anterior se llamaba 0023_article_slug y esta se llama 0024_auto_20160719_1734. Esto es lo que escribiremos en nuestro archivo de migración:

# -*- coding: utf-8 -*-
# Generated by Django 1.9.7 on 2016-07-19 15:34
from __future__ import unicode_literals

from django.db import migrations
from django.utils.text import slugify


def gen_slug(apps, schema_editor):
    # We can't import the Article model directly as it may be a newer
    # version than this migration expects. We use the historical version.
    Article = apps.get_model('app_name', 'Article')
    for row in Article.objects.all():
        row.slug = slugify(row.name)
        row.save()


class Migration(migrations.Migration):

    dependencies = [
        ('hosting', '0023_article_slug'),
    ]

    operations = [
        migrations.RunPython(gen_slug, reverse_code=migrations.RunPython.noop),
        # We set `reverse_code` to `noop` because we cannot revert the migration
        # to get it back in the previous state.
        # If `reverse_code` is not given, the migration will not be reversible,
        # which is not the behaviour we expect here.
    ]

Migraciones falsas #

Cuando se ejecuta una migración, Django almacena el nombre de la migración en una tabla django_migrations.

Crear y falsificar migraciones iniciales para el esquema existente

Si su aplicación ya tiene modelos y tablas de base de datos y no tiene migraciones. Primero cree migraciones iniciales para su aplicación.

python manage.py makemigrations your_app_label

Ahora migraciones iniciales falsas según se aplican

python manage.py migrate --fake-initial

False todas las migraciones en todas las aplicaciones

python manage.py migrate --fake

Migraciones falsas de una sola aplicación

python manage.py migrate --fake core

Archivo de migración único falso

python manage.py migrate myapp migration_name

Trabajar con migraciones #

Django usa migraciones para propagar los cambios que realiza en sus modelos a su base de datos. La mayoría de las veces Django puede generarlos por ti.

Para crear una migración, ejecute:

$ django-admin makemigrations <app_name>

Esto creará un archivo de migración en el submódulo migration de app_name. La primera migración se llamará 0001_initial.py, la otra comenzará con 0002_, luego 0003, …

Si omite <app_name>, se crearán migraciones para todas sus INSTALLED_APPS.

Para propagar las migraciones a su base de datos, ejecute:

$ django-admin migrate <app_name>

Para mostrar todas sus migraciones, ejecute:

$ django-admin showmigrations app_name
app_name
  [X] 0001_initial
  [X] 0002_auto_20160115_1027
  [X] 0003_somemodel
  [ ] 0004_auto_20160323_1826
  • [X] significa que la migración se propagó a su base de datos
  • [ ] significa que la migración no se propagó a su base de datos. Use django-admin migrate para propagarlo

También puede llamar a las migraciones revertidas, esto se puede hacer pasando el nombre de la migración al comando de migración. Dada la lista anterior de migraciones (mostrada por django-admin showmigrations):

$ django-admin migrate app_name 0002  # Roll back to migration 0002
$ django-admin showmigrations app_name
app_name
  [X] 0001_initial
  [X] 0002_auto_20160115_1027
  [ ] 0003_somemodel
  [ ] 0004_auto_20160323_1826

Nombres personalizados para archivos de migración #

Use la opción makemigrations --name <your_migration_name> para permitir nombrar las migraciones en lugar de usar un nombre generado.

python manage.py makemigrations --name <your_migration_name> <app_name>

Resolver conflictos de migración #

Introducción #

A veces, las migraciones entran en conflicto, lo que hace que la migración no tenga éxito. Esto puede suceder en muchos escenarios, sin embargo, puede ocurrir regularmente cuando se desarrolla una aplicación con un equipo.

Los conflictos de migración comunes ocurren al usar el control de código fuente, especialmente cuando se usa el método de función por rama. Para este escenario usaremos un modelo llamado Reporter con los atributos name y address.

Dos desarrolladores en este punto van a desarrollar una función, por lo que ambos obtienen esta copia inicial del modelo Reporter. El desarrollador A agrega una edad que da como resultado el archivo 0002_reporter_age.py. El desarrollador B agrega un campo bank_account que da como resultado 0002_reporter_bank_account. Una vez que estos desarrolladores fusionaron su código e intentaron migrar las migraciones, se produjo un conflicto de migración.

Este conflicto ocurre porque ambas migraciones alteran el mismo modelo, Reporter. Además de eso, los nuevos archivos comienzan con 0002.

Fusión de migraciones #

Hay varias formas de hacerlo. Lo siguiente está en el orden recomendado:

  1. La solución más simple para esto es ejecutar makemigrations command with a –merge flag.

     python manage.py makemigrations --merge <my_app>

    This will create a new migration solving the previous conflict.

  2. Cuando este archivo adicional no es bienvenido en el entorno de desarrollo for personal reasons, an option is to delete the conflicting migrations. Then, a new migration can be made using the regular makemigrations command. When custom migrations are written, such as migrations.RunPython, need to be accounted for using this method.

Cambiar un CharField a ForeignKey #

En primer lugar, asumamos que este es tu modelo inicial, dentro de una aplicación llamada discography:

from django.db import models

class Album(models.Model):
    name = models.CharField(max_length=255)
    artist = models.CharField(max_length=255)

Ahora, te das cuenta de que quieres usar una ForeignKey para el artista. Este es un proceso algo complejo, que tiene que hacerse en varios pasos.

Paso 1, agregue un nuevo campo para ForeignKey, asegurándose de marcarlo como nulo (tenga en cuenta que el modelo al que nos vinculamos ahora también está incluido):

from django.db import models

class Album(models.Model):
    name = models.CharField(max_length=255)
    artist = models.CharField(max_length=255)
    artist_link = models.ForeignKey('Artist', null=True)

class Artist(models.Model):
    name = models.CharField(max_length=255)

…y crea una migración para este cambio.

./manage.py makemigrations discography

Paso 2, complete su nuevo campo. Para hacer esto, debe crear una migración vacía.

./manage.py makemigrations --empty --name transfer_artists discography

Una vez que tenga esta migración vacía, querrá agregarle una sola operación RunPython para vincular sus registros. En este caso, podría ser algo como esto:

def link_artists(apps, schema_editor):
    Album = apps.get_model('discography', 'Album')
    Artist = apps.get_model('discography', 'Artist')
    for album in Album.objects.all():
        artist, created = Artist.objects.get_or_create(name=album.artist)
        album.artist_link = artist
        album.save()

Ahora que sus datos se transfieren al nuevo campo, podría terminar y dejar todo como está, usando el nuevo campo artist_link para todo. O, si desea hacer un poco de limpieza, desea crear dos migraciones más.

Para su primera migración, querrá eliminar su campo original, artist. Para su segunda migración, cambie el nombre del nuevo campo artist_link a artist.

Esto se hace en varios pasos para garantizar que Django reconozca las operaciones correctamente.
