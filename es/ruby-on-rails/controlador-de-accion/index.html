El controlador de acción es la C en MVC. Una vez que el enrutador ha determinado qué controlador usar para una solicitud, el controlador es responsable de dar sentido a la solicitud y producir la salida.

El controlador recibirá la solicitud, obtendrá o guardará datos de un modelo y utilizará una vista para crear resultados. Se puede pensar en un controlador como un intermediario entre modelos y vistas. Hace que los datos del modelo estén disponibles para la vista para que se puedan mostrar al usuario y guarda o actualiza los datos del usuario en el modelo.

Filtros #

Los filtros son métodos que se ejecutan “antes”, “después” o “alrededor” de una acción del controlador. Son heredados, por lo que si configura alguno en su ApplicationController, se ejecutarán para cada solicitud que reciba su aplicación.

Antes del filtro

Antes de que los filtros se ejecuten antes de la acción del controlador y pueden detener la solicitud (y/o redirigir). Un uso común es verificar si un usuario ha iniciado sesión:

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

Antes de que los filtros se ejecuten en las solicitudes antes de que la solicitud llegue a la acción del controlador. Puede devolver una respuesta por sí mismo y omitir por completo la acción.

Otros usos comunes de los filtros anteriores es validar la autenticación de un usuario antes de otorgarle acceso a la acción designada para manejar su solicitud. También los he visto utilizados para cargar un recurso de la base de datos, verificar permisos en un recurso o administrar redireccionamientos en otras circunstancias.

Después del filtro

Los filtros posteriores son similares a los “anteriores”, pero a medida que se ejecutan después de la ejecución de la acción, tienen acceso al objeto de respuesta que está a punto de enviarse. Entonces, en breve, los filtros se ejecutan después de que se completa la acción. Puede modificar la respuesta. La mayoría de las veces, si algo se hace en un filtro posterior, se puede hacer en la acción misma, pero si hay alguna lógica para ejecutar después de ejecutar cualquiera de un conjunto de acciones, entonces un filtro posterior es un buen lugar para hacerlo. eso.

En general, he visto filtros posteriores y circundantes que se usan para iniciar sesión.

Alrededor del filtro

Los filtros Around pueden tener una lógica antes y después de ejecutar la acción. Simplemente cede a la acción en cualquier lugar que sea necesario. Tenga en cuenta que no necesita ceder a la acción y puede ejecutarse sin hacerlo como un filtro anterior.

Los filtros Around son responsables de ejecutar sus acciones asociadas al ceder, de forma similar a cómo funcionan los middlewares de Rack.

Las devoluciones de llamada envuelven la ejecución de acciones. Puede escribir una devolución de llamada en dos estilos diferentes. En el primero, la devolución de llamada es una sola porción de código. Ese código se llama antes de que se ejecute la acción. Si el código de devolución de llamada invoca el rendimiento, se ejecuta la acción. Cuando se completa la acción, el código de devolución de llamada continúa ejecutándose. Por lo tanto, el código antes del rendimiento es como una devolución de llamada antes de la acción y el código después del rendimiento es la devolución de llamada después de la acción. Si el código de devolución de llamada nunca invoca el rendimiento. la acción no se ejecuta; esto es lo mismo que hacer que una devolución de llamada antes de la acción devuelva falso.

Aquí hay un ejemplo del filtro alrededor:

around_filter :catch_exceptions
 
private
  def catch_exceptions
    begin
      yield
    rescue Exception => e 
      logger.debug "Caught exception! #{e.message}"
    end
  end

Esto detectará la excepción de cualquier acción y colocará el mensaje en su registro. Puede usar filtros around para el manejo de excepciones, configuración y desmontaje, y una miríada de otros casos.

Solo y Excepto

Todos los filtros se pueden aplicar a acciones específicas, usando :solo y :excepto:

class ProductsController < ApplicationController
  before_action :set_product, only: [:show, :edit, :update]

  # ... controller actions

  # Define your filters as controller private methods
  private

  def set_product
    @product = Product.find(params[:id])
  end
end

Omitir filtro

Todos los filtros (también los heredados) también se pueden omitir para algunas acciones específicas:

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

class HomeController < ApplicationController
  skip_before_action :authenticate_user!, only: [:index]

  def index
  end
end

Como son heredados, los filtros también se pueden definir en un controlador “principal” de espacio de nombres. Digamos, por ejemplo, que tiene un espacio de nombres admin y, por supuesto, desea que solo los usuarios administradores puedan acceder a él. Podrías hacer algo como esto:

# config/routes.rb
namespace :admin do
  resources :products
end

# app/controllers/admin_controller.rb
class AdminController < ApplicationController
  before_action :authenticate_admin_user!

  private

  def authenticate_admin_user!
    redirect_to root_path unless current_user.admin?
  end
end

# app/controllers/admin/products_controller.rb
class Admin::ProductsController < AdminController
  # This controller will inherit :authenticate_admin_user! filter
end

Tenga en cuenta que en Rails 4.x podría usar before_filter junto con before_action, pero before_filter actualmente está en desuso en Rails 5.0.0 y se eliminará en 5.1.

Generando un controlador #

Rails proporciona muchos generadores, por supuesto también para controladores.

Puede generar un nuevo controlador ejecutando este comando en la carpeta de su aplicación

rails generate controller NAME [action action] [options]

Nota: También puede usar el alias rails g para invocar rails generate

Por ejemplo, para generar un controlador para un modelo Producto, con las acciones #index y #show que ejecutaría

rails generate controller products index show

Esto creará el controlador en app/controllers/products_controller.rb, con las dos acciones que especificó

class ProductsController < ApplicationController
  def index
  end

  def show
  end
end

También creará una carpeta de productos dentro de app/views/, que contiene las dos plantillas para las acciones de su controlador (es decir, index.html.erb y show.html.erb, tenga en cuenta que la extensión puede variar según a su motor de plantillas, por lo que si está usando slim, por ejemplo, el generador creará index.html.slim y show.html.slim )

Además, si especificó alguna acción, también se agregará a su archivo rutas

# config/routes.rb
get 'products/show'
get 'products/index'

Rails crea un archivo auxiliar para usted, en app/helpers/products_helper.rb, y también los archivos de recursos en app/assets/javascripts/products.js y app/assets/stylesheets/products.css. En cuanto a las vistas, el generador cambia este comportamiento de acuerdo con lo que se especifica en su Gemfile: es decir, si está usando Coffeescript y Sass en su aplicación, el generador del controlador generará products.coffee y productos.sass.

Por último, pero no menos importante, Rails también genera archivos de prueba para su controlador, su asistente y sus vistas.

Si no desea que se cree ninguno de estos, puede decirle a Rails que los omita, simplemente anteponga cualquier opción con

--no- o --skip, así:

rails generate controller products index show --no-assets --no-helper

Y el generador omitirá tanto los “activos” como el “ayudante”.

Si necesita crear un controlador para un espacio de nombres específico, agréguelo delante de NOMBRE:

rails generate controller admin/products

Esto creará su controlador dentro de app/controllers/admin/products_controller.rb

Rails también puede generar un controlador RESTful completo para usted:

rails generate scaffold_controller MODEL_NAME # available from Rails 4
rails generate scaffold_controller Product

Rescatando ActiveRecord::RecordNotFound con redirect_to #

Puede rescatar una excepción RecordNotFound con una redirección en lugar de mostrar una página de error:

class ApplicationController < ActionController::Base

  # your other stuff

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: I18n.t("errors.record_not_found")
  end
end

404 cuando no se encuentra el registro #

Rescate del error de registro no encontrado en lugar de mostrar una excepción o una página en blanco:

class ApplicationController < ActionController::Base
  
  # ... your other stuff here 

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: 'Record not found'
  end
end
