Modelo gordo, controlador flaco #

“Modelo gordo, controlador flaco” se refiere a cómo las partes M y C de MVC funcionan juntas idealmente. Es decir, cualquier lógica relacionada con la falta de respuesta debe ir en el modelo, idealmente en un método agradable y comprobable. Mientras tanto, el controlador “delgado” es simplemente una buena interfaz entre la vista y el modelo.

En la práctica, esto puede requerir una gama de diferentes tipos de refactorización, pero todo se reduce a una idea: al mover cualquier lógica que no se trate de la respuesta al modelo (en lugar del controlador), no solo ha promovido la reutilización donde sea posible, pero también ha hecho posible probar su código fuera del contexto de una solicitud.

Veamos un ejemplo sencillo. Digamos que tienes un código como este:

def index
  @published_posts = Post.where('published_at <= ?', Time.now)
  @unpublished_posts = Post.where('published_at IS NULL OR published_at > ?', Time.now)
end

Puedes cambiarlo a esto:

def index
  @published_posts = Post.published
  @unpublished_posts = Post.unpublished
end

Luego, puede mover la lógica a su modelo de publicación, donde podría verse así:

scope :published, ->(timestamp = Time.now) { where('published_at <= ?', timestamp) }
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) }

Objetos de dominio (No más modelos gordos) #

“Fat Model, Skinny Controller” es un muy buen primer paso, pero no escala bien una vez que su base de código comienza a crecer.

Pensemos en la Responsabilidad Única de los modelos. ¿Cuál es la única responsabilidad de los modelos? ¿Es para mantener la lógica de negocios? ¿Es para mantener la lógica relacionada con la no respuesta?

No. Su responsabilidad es manejar la capa de persistencia y su abstracción.

La lógica empresarial, así como cualquier lógica no relacionada con la respuesta y lógica no relacionada con la persistencia, debe ir en objetos de dominio.

Los objetos de dominio son clases diseñadas para tener una sola responsabilidad en el dominio del problema. Deje que sus clases “Scream Their Architecture” para los problemas que resuelven.

En la práctica, debe esforzarse por obtener modelos delgados, vistas delgadas y controladores delgados. La arquitectura de su solución no debe verse influenciada por el marco que elija.

Por ejemplo

Supongamos que es un mercado que cobra una comisión fija del 15 % a sus clientes a través de Stripe. Si cobra una comisión fija del 15 %, eso significa que su comisión cambia según el monto del pedido porque Stripe cobra el 2,9 % + 30 ¢.

La cantidad que cobras como comisión debe ser: cantidad*0.15 - (cantidad*0.029 + 0.30).

No escribas esta lógica en el modelo:

# app/models/order.rb
class Order < ActiveRecord::Base
  SERVICE_COMMISSION = 0.15
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  ...

  def commission
    amount*SERVICE_COMMISSION - stripe_commission  
  end

  private

  def stripe_commission
    amount*STRIPE_PERCENTAGE_COMMISSION + STRIPE_FIXED_COMMISSION
  end
end

Tan pronto como se integre con un nuevo método de pago, no podrá escalar esta funcionalidad dentro de este modelo.

Además, tan pronto como comience a integrar más lógica comercial, su objeto Orden comenzará a perder [cohesión] (https://thebojan.ninja/2015/04/08/high-cohesion-loose-coupling/).

Preferir objetos de dominio, con el cálculo de la comisión completamente abstraído de la responsabilidad de los pedidos persistentes:

# app/models/order.rb
class Order < ActiveRecord::Base
  ...
  # No reference to commission calculation
end

# lib/commission.rb
class Commission
  SERVICE_COMMISSION = 0.15

  def self.calculate(payment_method, model)
    model.amount*SERVICE_COMMISSION - payment_commission(payment_method, model)  
  end

  private

  def self.payment_commission(payment_method, model)
    # There are better ways to implement a static registry,
    # this is only for illustration purposes.
    Object.const_get("#{payment_method}Commission").calculate(model)
  end
end

# lib/stripe_commission.rb
class StripeCommission
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  def self.calculate(model)
    model.amount*STRIPE_PERCENTAGE_COMMISSION
      + STRIPE_PERCENTAGE_COMMISSION
  end
end

# app/controllers/orders_controller.rb
class OrdersController < ApplicationController
  def create
    @order = Order.new(order_params)
    @order.commission = Commission.calculate("Stripe", @order)
    ...
  end
end

El uso de objetos de dominio tiene las siguientes ventajas arquitectónicas:

  • Es extremadamente fácil realizar pruebas unitarias, ya que no se requieren accesorios ni fábricas para instanciar los objetos con la lógica.
  • funciona con todo lo que acepta el mensaje cantidad.
  • mantiene cada objeto de dominio pequeño, con responsabilidades claramente definidas y con mayor cohesión.
  • escala fácilmente con nuevos métodos de pago por [adición, no modificación] (https://en.wikipedia.org/wiki/Open/closed_principle).
  • detiene la tendencia a tener un objeto ‘Usuario’ en constante crecimiento en cada aplicación de Ruby on Rails.

Personalmente, me gusta poner objetos de dominio en lib. Si lo hace, recuerde agregarlo a autoload_paths:

# config/application.rb
config.autoload_paths << Rails.root.join('lib')

También puede preferir crear objetos de dominio más orientados a la acción, siguiendo el patrón Comando/Consulta. En tal caso, colocar estos objetos en app/commands podría ser un mejor lugar ya que todos los subdirectorios app se agregan automáticamente a la ruta de carga automática.

Convención sobre configuración #

En Rails, te encuentras mirando controladores, vistas y modelos para tu base de datos.

Para reducir la necesidad de una configuración pesada, Rails implementa reglas para facilitar el trabajo con la aplicación. Puede definir sus propias reglas, pero al principio (y más adelante) es una buena idea ceñirse a las convenciones que ofrece Rails.

Estas convenciones acelerarán el desarrollo, mantendrán su código conciso y legible, y le permitirán navegar fácilmente dentro de su aplicación.

Las convenciones también reducen las barreras de entrada para los principiantes. Hay tantas convenciones en Rails que un principiante ni siquiera necesita conocerlas, pero puede beneficiarse de ellas por ignorancia. Es posible crear grandes aplicaciones sin saber por qué todo es como es.

Por ejemplo

Si tiene una tabla de base de datos llamada orders con la clave principal id, el modelo coincidente se llama order y el controlador que maneja toda la lógica se llama orders_controller. La vista se divide en diferentes acciones: si el controlador tiene una acción nuevo y editar, también hay una vista nuevo y editar.

Por ejemplo

Para crear una aplicación, simplemente ejecute rails new app_name. Esto generará aproximadamente 70 archivos y carpetas que componen la infraestructura y la base de su aplicación Rails.

Incluye:

  • Carpetas para guardar sus modelos (capa de base de datos), controladores y vistas
  • Carpetas para almacenar pruebas unitarias para su aplicación
  • Carpetas para guardar sus activos web como archivos Javascript y CSS
  • Archivos predeterminados para respuestas HTTP 400 (es decir, archivo no encontrado)
  • Muchos otros

Cuidado con default_scope #

ActiveRecord incluye default_scope, para medir automáticamente el alcance de un modelo de forma predeterminada.

class Post
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

El código anterior servirá publicaciones que ya están publicadas cuando realice cualquier consulta en el modelo.

Post.all # will only list published posts

Ese alcance, aunque tiene un aspecto inocuo, tiene múltiples efectos secundarios ocultos que quizás no desee.

ámbito_predeterminado y orden #

Dado que declaraste un “pedido” en “default_scope”, llamar a “pedido” en “Publicar” se agregará como pedidos adicionales en lugar de anular el predeterminado.

Post.order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."created_at" DESC, "posts"."updated_at" DESC

Probablemente este no sea el comportamiento que deseaba; puede anular esto excluyendo primero el orden del alcance

Post.except(:order).order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."updated_at" DESC

default_scope e inicialización del modelo #

Al igual que con cualquier otro ActiveRecord::Relation, default_scope alterará el estado predeterminado de los modelos inicializados a partir de él.

En el ejemplo anterior, Post tiene where(published: true) configurado de forma predeterminada, por lo que los nuevos modelos de Post también lo tendrán configurado.

Post.new # => <Post published: true>

sin alcance #

default_scope puede borrarse nominalmente llamando primero a unscoped, pero esto también tiene efectos secundarios. Tomemos, por ejemplo, un modelo STI:

class Post < Document
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

De forma predeterminada, las consultas contra Publicar se limitarán a las columnas de tipo que contienen 'Publicar'. Pero unscoped borrará esto junto con su propio default_scope, por lo que si usa unscoped, debe recordar tener esto en cuenta también.

Post.unscoped.where(type: 'Post').order(updated_at: :desc)

Asociaciones sin ámbito y modelo #

Considere una relación entre ‘Publicar’ y ‘Usuario’

class Post < ApplicationRecord
  belongs_to :user
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

class User < ApplicationRecord
  has_many :posts
end

Al obtener un ‘Usuario’ individual, puede ver las publicaciones relacionadas con él:

user = User.find(1)
user.posts

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' AND "posts"."user_id" = ? ORDER BY "posts"."created_at" DESC [["user_id", 1]]

Pero desea borrar default_scope de la relación posts, por lo que usa unscoped

user.posts.unscoped

SELECT "posts".* FROM "posts"

Esto elimina la condición user_id así como default_scope.

Un ejemplo de caso de uso para default_scope #

A pesar de todo eso, hay situaciones en las que se justifica el uso de default_scope.

Considere un sistema multiinquilino donde varios subdominios se sirven desde la misma aplicación pero con datos aislados. Una forma de lograr este aislamiento es a través de default_scope. Las desventajas en otros casos se convierten aquí en ventajas.

class ApplicationRecord < ActiveRecord::Base
  def self.inherited(subclass)
    super

    return unless subclass.superclass == self
    return unless subclass.column_names.include? 'tenant_id'

    subclass.class_eval do
      default_scope ->{ where(tenant_id: Tenant.current_id) }
    end
  end
end

Todo lo que necesita hacer es establecer Tenant.current_id en algo al principio de la solicitud, y cualquier tabla que contenga tenant_id automáticamente tendrá alcance sin ningún código adicional. Los registros de creación de instancias heredarán automáticamente la identificación del arrendatario con la que se crearon.

Lo importante de este caso de uso es que el alcance se establece una vez por solicitud y no cambia. Los únicos casos que necesitará sin ámbito aquí son casos especiales como trabajadores en segundo plano que se ejecutan fuera del alcance de una solicitud.

No te repitas (SECO) #

Para ayudar a mantener un código limpio, Rails sigue el principio de DRY.

Implica, siempre que sea posible, reutilizar tanto código como sea posible en lugar de duplicar código similar en varios lugares (por ejemplo, usando parciales). Esto reduce los errores, mantiene su código limpio y aplica el principio de escribir el código una vez y luego reutilizarlo. También es más fácil y eficiente actualizar el código en un solo lugar que actualizar varias partes del mismo código. Haciendo así su código más modular y robusto.

También Fat Model, Skinny Controller es SECO, porque escribes el código en tu modelo y en el controlador solo haces la llamada, como:

# Post model
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) } 


# Any controller
def index
    ....
    @unpublished_posts = Post.unpublished
    ....
end

def others
    ...
    @unpublished_posts = Post.unpublished
    ...
end

Esto también ayuda a conducir a una estructura impulsada por API donde los métodos internos están ocultos y los cambios se logran mediante el paso de parámetros en forma de API.

No lo vas a necesitar (YAGNI) #

Si puede decir “YAGNI” (no lo necesitará) sobre una función, es mejor que no la implemente. Se puede ahorrar mucho tiempo de desarrollo centrándose en la simplicidad. La implementación de tales características de todos modos puede generar problemas:

Problemas #

Sobreingeniería #

Si un producto es más complicado de lo que tiene que ser, está sobrediseñado. Por lo general, estas funciones “todavía no utilizadas” nunca se utilizarán de la manera prevista en la que se escribieron y deben refactorizarse si alguna vez se utilizan. Las optimizaciones prematuras, especialmente las optimizaciones de rendimiento, a menudo conducen a decisiones de diseño que se demostrarán incorrectas en el futuro.

Inflación de código #

Code Bloat significa código complicado innecesario. Esto puede ocurrir, por ejemplo, por abstracción, redundancia o aplicación incorrecta de patrones de diseño. El código base se vuelve difícil de entender, confuso y costoso de mantener.

Arrastramiento de funciones #

Feature Creep se refiere a agregar nuevas características que van más allá de la funcionalidad principal del producto y conducen a una complejidad innecesariamente alta del producto.

Largo tiempo de desarrollo #

El tiempo que podría emplearse para desarrollar funciones necesarias se dedica a desarrollar funciones innecesarias. El producto tarda más en entregar.

Soluciones #

KISS - Mantenlo simple, estúpido #

Según KISS, la mayoría de los sistemas funcionan mejor si tienen un diseño simple. La simplicidad debe ser un objetivo de diseño principal para reducir la complejidad. Se puede lograr siguiendo el “Principio de responsabilidad única”, por ejemplo.

YAGNI – No lo vas a necesitar #

Menos es más. Piense en cada función, ¿es realmente necesaria? Si se te ocurre alguna forma de que sea YAGNI, déjalo. Es mejor desarrollarlo cuando sea necesario.

Refactorización continua #

El producto se está mejorando constantemente. Con la refactorización, podemos asegurarnos de que el producto se realice de acuerdo con las mejores prácticas y no degenere en un parche.
