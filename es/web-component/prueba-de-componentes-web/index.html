Paquete web y broma #

Facebook utiliza Jest para probar todo el código JavaScript, incluidas las aplicaciones React. Una de las filosofías de Jest es proporcionar una experiencia integrada de “configuración cero”. Observamos que cuando los ingenieros cuentan con herramientas listas para usar, terminan escribiendo más pruebas, lo que a su vez da como resultado bases de código más estables y saludables.

El ejemplo de trabajo completo está disponible en GitHub como web-components-webpack-es6-boilerplate

Jest ejecuta pruebas en el entorno NodeJS con jsdom. Todo el proceso es fácil. Consideremos la siguiente configuración de webpack, asumiendo que la estructura de nuestro proyecto se parece al siguiente ejemplo:

-src
    --client
    --server
-webpack
    --config.js
package.json

Una estructura de directorios simple diseñada para separar la lógica de “representación del servidor” del resto. Webpack El archivo config.js contendría los siguientes módulos:

  resolve: {
    modules: ["node_modules"],
    alias: {
      client: path.join(__dirname, "../src/client"),
      server: path.join(__dirname, "../src/server")
    },
    extensions: [".js", ".json", ".scss"]
  },

Podemos configurar Jest para reflejar nuestra configuración de Webpack.

module.exports = {
  setupTestFrameworkScriptFile: "<rootDir>/bin/jest.js",
  mapCoverage: true,
  moduleFileExtensions: ["js", "scss", "html"],
  moduleDirectories: ["node_modules"],
  moduleNameMapper: {
    "src/(.*)$": "<rootDir>/src/$1"
  },
  transform: {
    "^.+\\.(js|html|scss)$": "<rootDir>/bin/preprocessor.js"
  },
  testMatch: ["<rootDir>/test/**/?(*.)(spec|test).js"],
  testPathIgnorePatterns: ["<rootDir>/(node_modules|bin|build)"]
};

¿Dónde debemos guardar esta configuración?

Podemos hacerlo en el archivo package.json bajo la clave jest o crear como en este ejemplo el archivo jest.config.js en la raíz del proyecto.

Lo que queremos lograr es asegurarnos de que nuestros archivos html se importarán correctamente. Eso significa escapar de ellos con un preprocesador personalizado, ya que usar solo babel-jest arrojaría un error al intentar analizar archivos que no sean js.

La otra cosa importante aquí es el script setupTestFrameworkScriptFile que en realidad incluye polyfills de elementos personalizados para jsdom. Así es como se ve nuestro preprocessor.js:

const babelJest = require("babel-jest");

const STYLE_URLS_REGEX = /styles:\s*\[\s*((?:'|").*\s*(?:'|")).*\s*.*\]/g;
const ESCAPE_TEMPLATE_REGEX = /(\${|\`)/g;

module.exports.process = (src, path, config) => {
  if (path.endsWith(".html")) {
    src = src.replace(ESCAPE_TEMPLATE_REGEX, "\\$1");
    src = "module.exports=`" + src + "`;";
  }
  src = src.replace(STYLE_URLS_REGEX, "styles: []");

  return babelJest.process(src, path, config);
};

Lo que hace este script es simple: eliminar el contenido de los archivos de estilo ya que no necesitamos/queremos probarlo, y escapar de las plantillas, cuando las importamos, por ejemplo, con la sintaxis require('template.html'). Luego pasa el contenido al transformador de babel.

Lo último que debe hacer es incluir componentes web polyfills. Como por defecto, jsdom todavía no los admite. Para hacerlo simplemente podemos agregar setupTestFrameworkScriptFile en nuestro ejemplo es jest.js con el siguiente contenido:

require("document-register-element/pony")(window);

De esta manera podemos acceder a componentes web API en jsdom.

Después de configurar todo, deberíamos tener una estructura como esta:

-bin
    --jest.js
    --preprocessor.js
-src
    --client
    --server
-webpack
    --config.js
-test
package.json
jest.config.js

Donde guardamos nuestras pruebas en el directorio test y podemos ejecutarlas con el comando: yarn run jest --no-cache --config $(node ​​jest.config.js).
