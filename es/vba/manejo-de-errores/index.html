Evitar condiciones de error #

Cuando ocurre un error de tiempo de ejecución, un buen código debe manejarlo. La mejor estrategia de manejo de errores es escribir código que verifique las condiciones de error y simplemente evite ejecutar código que resulte en un error de tiempo de ejecución.

Un elemento clave para reducir los errores de tiempo de ejecución es escribir pequeños procedimientos que hacen una cosa. Cuantas menos razones tengan los procedimientos para fallar, más fácil será depurar el código en su conjunto.

Evitando el error de tiempo de ejecución 91 - Objeto o Con variable de bloque no establecida:

Este error se generará cuando se use un objeto antes de que se asigne su referencia. Uno podría tener un procedimiento que recibe un parámetro de objeto:

Private Sub DoSomething(ByVal target As Worksheet)
    Debug.Print target.Name
End Sub

Si no se asigna una referencia a objetivo, el código anterior generará un error que se evita fácilmente comprobando si el objeto contiene una referencia de objeto real:

Private Sub DoSomething(ByVal target As Worksheet)
    If target Is Nothing Then Exit Sub
    Debug.Print target.Name
End Sub

Si no se asigna una referencia a objetivo, la referencia no asignada nunca se usa y no se produce ningún error.

Esta forma de salir antes de tiempo de un procedimiento cuando uno o más parámetros no son válidos se denomina cláusula de protección.

Evitando el error de tiempo de ejecución 9 - Subíndice fuera de rango:

Este error se genera cuando se accede a una matriz fuera de sus límites.

Private Sub DoSomething(ByVal index As Integer)
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

Dado un índice mayor que el número de hojas de trabajo en ActiveWorkbook, el código anterior generará un error de tiempo de ejecución. Una simple cláusula de guardia puede evitar eso:

Private Sub DoSomething(ByVal index As Integer)
    If index > ActiveWorkbook.Worksheets.Count Or index <= 0 Then Exit Sub
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

La mayoría de los errores de tiempo de ejecución se pueden evitar verificando cuidadosamente los valores que estamos usando antes de usarlos, y bifurcándonos en otra ruta de ejecución en consecuencia usando una simple declaración If - en cláusulas de protección que no hacen suposiciones y validan los parámetros de un procedimiento, o incluso en el cuerpo de procedimientos más grandes.

Errores personalizados #

A menudo, al escribir una clase especializada, querrá que genere sus propios errores específicos, y querrá una forma limpia para que el código de usuario/llamada maneje estos errores personalizados. Una buena manera de lograr esto es definiendo un tipo Enum dedicado:

Option Explicit
Public Enum FoobarError
    Err_FooWasNotBarred = vbObjectError + 1024
    Err_BarNotInitialized
    Err_SomethingElseHappened
End Enum

El uso de la constante incorporada vbObjectError garantiza que los códigos de error personalizados no se superpongan con los códigos de error reservados/existentes. Solo es necesario especificar explícitamente el primer valor de enumeración, ya que el valor subyacente de cada miembro Enum es 1 mayor que el miembro anterior, por lo que el valor subyacente de Err_BarNotInitialized es implícitamente vbObjectError + 1025.

Generar sus propios errores de tiempo de ejecución #

Se puede generar un error de tiempo de ejecución utilizando la declaración Err.Raise, por lo que el error personalizado Err_FooWasNotBarred se puede generar de la siguiente manera:

Err.Raise Err_FooWasNotBarred

El método Err.Raise también puede tomar parámetros Descripción y Fuente personalizados; por este motivo, es una buena idea definir también constantes para contener la descripción de cada error personalizado:

Private Const Msg_FooWasNotBarred As String = "The foo was not barred."
Private Const Msg_BarNotInitialized As String = "The bar was not initialized."

Y luego cree un método privado dedicado para generar cada error:

Private Sub OnFooWasNotBarredError(ByVal source As String)
    Err.Raise Err_FooWasNotBarred, source, Msg_FooWasNotBarred
End Sub

Private Sub OnBarNotInitializedError(ByVal source As String)
    Err.Raise Err_BarNotInitialized, source, Msg_BarNotInitialized
End Sub

La implementación de la clase puede simplemente llamar a estos procedimientos especializados para generar el error:

Public Sub DoSomething()
    'raises the custom 'BarNotInitialized' error with "DoSomething" as the source:
    If Me.Bar Is Nothing Then OnBarNotInitializedError "DoSomething"
    '...
End Sub

El código del cliente puede manejar Err_BarNotInitialized como lo haría con cualquier otro error, dentro de su propia subrutina de manejo de errores.

Nota: la palabra clave Error heredada también se puede usar en lugar de Err.Raise, pero está obsoleta/obsoleta.

Reanudar palabra clave #

Una subrutina de manejo de errores:

  • ejecutar hasta el final del procedimiento, en cuyo caso la ejecución se reanuda en el procedimiento de llamada.
  • o utilice la palabra clave Reanudar para reanudar la ejecución dentro del mismo procedimiento.

La palabra clave Reanudar solo debe usarse dentro de una subrutina de manejo de errores, porque si VBA encuentra Reanudar sin estar en un estado de error, se genera el error de tiempo de ejecución 20 “Reanudar sin error”.

Hay varias formas en que una subrutina de manejo de errores puede usar la palabra clave Reanudar:

  • Reanudar usado solo, la ejecución continúa en la declaración que causó el error. Si el error realmente no se maneja antes de hacerlo, entonces el mismo error se generará nuevamente y la ejecución podría entrar en un bucle infinito.
  • Reanudar siguiente continúa la ejecución en la declaración que sigue inmediatamente a la declaración que causó el error. Si el error realmente no se maneja antes de hacerlo, se permite que la ejecución continúe con datos potencialmente no válidos, lo que puede generar errores lógicos y un comportamiento inesperado.
  • Reanudar [etiqueta de línea] continúa la ejecución en la etiqueta de línea especificada (o número de línea, si está utilizando números de línea de estilo heredado). Por lo general, esto permitiría ejecutar algún código de limpieza antes de salir limpiamente del procedimiento, como garantizar que una conexión a la base de datos esté cerrada antes de volver a la persona que llama.

En caso de error Reanudar siguiente #

La declaración On Error en sí misma puede usar la palabra clave Resume para indicarle al tiempo de ejecución de VBA que ignore todos los errores de manera efectiva.

Si el error no se realmente maneja antes de hacerlo, entonces se permite que la ejecución continúe con datos potencialmente no válidos, lo que puede resultar en errores lógicos y comportamiento inesperado.

El énfasis anterior no se puede enfatizar lo suficiente. On Error Resume Next ignora todos los errores y los esconde debajo de la alfombra. Un programa que explota con un error de tiempo de ejecución debido a una entrada inválida es un mejor programa que uno que sigue ejecutándose con datos desconocidos/involuntarios, ya sea solo porque el error es mucho más fácil de identificar. On Error Resume Next puede fácilmente ocultar errores.

La sentencia On Error tiene un ámbito de procedimiento; es por eso que normalmente debería haber solo una, una única sentencia On Error en un procedimiento determinado.

Sin embargo, a veces una condición de error no se puede evitar del todo, y saltar a una subrutina de manejo de errores solo para “Reanudar siguiente” simplemente no se siente bien. En este caso específico, la declaración de que posiblemente falle se puede envolver entre dos declaraciones “En caso de error”:

On Error Resume Next
[possibly-failing statement]
Err.Clear 'resets current error
On Error GoTo 0

La instrucción On Error GoTo 0 restablece el manejo de errores en el procedimiento actual, de modo que cualquier otra instrucción que cause un error de tiempo de ejecución no se manejaría dentro de ese procedimiento y, en cambio, pasaría a la pila de llamadas hasta que sea capturada por un controlador de errores activo. Si no hay un controlador de errores activo en la pila de llamadas, se tratará como una excepción no controlada.

Public Sub Caller()
    On Error GoTo Handler
    
    Callee
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Caller."
End Sub

Public Sub Callee()
    On Error GoTo Handler
    
    Err.Raise 1     'This will be handled by the Callee handler.
    On Error GoTo 0 'After this statement, errors are passed up the stack.
    Err.Raise 2     'This will be handled by the Caller handler.    
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Callee."
    Resume Next
End Sub

Declaración de error #

Incluso con cláusulas de protección, uno no puede dar cuenta siempre de forma realista de todas las posibles condiciones de error que podrían surgir en el cuerpo de un procedimiento. La instrucción On Error GoTo indica a VBA que salte a una etiqueta de línea e ingrese al “modo de manejo de errores” cada vez que ocurra un error inesperado en tiempo de ejecución. Después de manejar un error, el código puede reanudar nuevamente a la ejecución “normal” usando la palabra clave Reanudar.

Etiquetas de línea denotan subrutinas: debido a que las subrutinas se originan a partir del código BASIC heredado y usan saltos GoTo y GoSub y declaraciones Return para volver a la rutina “principal”, es bastante fácil escribir tareas difíciles de sigue código espagueti si las cosas no están estructuradas rigurosamente. Por este motivo, lo mejor es que:

  • un procedimiento tiene una y solo una subrutina de manejo de errores
  • la subrutina de manejo de errores solo se ejecuta en un estado de error

Esto significa que un procedimiento que maneja sus errores debe estructurarse así:

Private Sub DoSomething()
    On Error GoTo CleanFail

    'procedure code here

CleanExit:
    'cleanup code here
    Exit Sub

CleanFail:
    'error-handling code here
    Resume CleanExit
End Sub

Estrategias de manejo de errores #

A veces desea manejar diferentes errores con diferentes acciones. En ese caso, inspeccionará el objeto global Err, que contendrá información sobre el error que se generó, y actuará en consecuencia:

CleanExit:
    Exit Sub

CleanFail:
    Select Case Err.Number
        Case 9
            MsgBox "Specified number doesn't exist. Please try again.", vbExclamation
            Resume
        Case 91
            'woah there, this shouldn't be happening.
            Stop 'execution will break here
            Resume 'hit F8 to jump to the line that raised the error
        Case Else
            MsgBox "An unexpected error has occurred:" & vbNewLine & Err.Description, vbCritical
            Resume CleanExit
    End Select
End Sub

Como pauta general, considere activar el manejo de errores para toda la subrutina o función, y maneje todos los errores que puedan ocurrir dentro de su alcance. Si solo necesita manejar errores en la sección pequeña del código, active y desactive el manejo de errores al mismo nivel:

Private Sub DoSomething(CheckValue as Long)

    If CheckValue = 0 Then
        On Error GoTo ErrorHandler   ' turn error handling on
        ' code that may result in error
        On Error GoTo 0              ' turn error handling off - same level
    End If

CleanExit:
    Exit Sub

ErrorHandler:
    ' error handling code here
    ' do not turn off error handling here
    Resume

End Sub

Línea de números #

VBA admite números de línea de estilo heredado (por ejemplo, QBASIC). La propiedad oculta Erl se puede utilizar para identificar el número de línea que generó el último error. Si no está utilizando números de línea, Erl solo devolverá 0.

Sub DoSomething()
10 On Error GoTo 50
20 Debug.Print 42 / 0
30 Exit Sub
40
50 Debug.Print "Error raised on line " & Erl ' returns 20
End Sub

Si estás usando números de línea, pero no consistentemente, Erl devolverá el último número de línea antes de la instrucción que generó el error.

Sub DoSomething()
10 On Error GoTo 50
   Debug.Print 42 / 0
30 Exit Sub

50 Debug.Print "Error raised on line " & Erl 'returns 10
End Sub

Tenga en cuenta que ‘Erl’ también solo tiene precisión de ‘Entero’, y se desbordará silenciosamente. Esto significa que los números de línea fuera del [rango de enteros] (https://www.wikiod.com/es/vba/tipos-de-datos-y-limites#Entero) darán resultados incorrectos:

Sub DoSomething()
99997 On Error GoTo 99999
99998 Debug.Print 42 / 0
99999
      Debug.Print Erl   'Prints 34462
End Sub

El número de línea no es tan relevante como la declaración que causó el error, y la numeración de líneas rápidamente se vuelve tediosa y difícil de mantener.
