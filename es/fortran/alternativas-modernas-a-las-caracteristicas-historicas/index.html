Aritmética declaración if #

La declaración aritmética if permite usar tres ramas dependiendo del resultado de una expresión aritmética

if (arith_expr) label1, label2, label3

Esta instrucción if transfiere el flujo de control a una de las etiquetas en un código. Si el resultado de arith_expr es negativo, se utiliza label1, si el resultado es cero, se utiliza label2, y si el resultado es positivo, se aplica la última label3. La aritmética si requiere las tres etiquetas pero permite la reutilización de etiquetas, por lo tanto, esta declaración se puede simplificar a dos ramas si.

Ejemplos:

if (N * N - N / 2) 130, 140, 130

if (X) 100, 110, 120

Ahora esta característica está obsoleta con la misma funcionalidad ofrecida por la instrucción if y la construcción if-else. Por ejemplo, el fragmento

    if (X) 100, 110, 120
100 print*, "Negative"
    goto 200
110 print*, "Zero"
    goto 200
120 print*, "Positive"
200 continue

puede escribirse como la construcción if-else

if (X<0) then
  print*, "Negative"
else if (X==0) then
  print*, "Zero"
else
  print*, "Positive"
end if

Un reemplazo de declaración if para

    if (X) 100, 100, 200
100 print *, "Negative or zero"
200 continue

quizás

if (X<=0) print*, "Negative or zero"

Tipos de variables implícitas #

Formulario de fuente fija #

Fortran se diseñó originalmente para un [formulario de formato fijo] (https://en.wikipedia.org/wiki/Fortran#Fixed_layout_and_punched_cards) basado en una tarjeta perforada de 80 columnas:

Estos fueron creados en una máquina perforadora de tarjetas, muy parecido a esto:

Las imágenes son fotografías originales del autor

El formato, como se muestra en la tarjeta de muestra ilustrada, tenía las primeras cinco columnas reservadas para etiquetas de declaración. La primera columna se utilizó para indicar los comentarios con una letra C. La sexta columna se usó para indicar una continuación de declaración (al insertar cualquier carácter que no sea un cero ‘0’). Las últimas 8 columnas se usaron para la identificación y secuenciación de cartas, ¡lo cual fue muy valioso si dejaste caer tu mazo de cartas al suelo! La codificación de caracteres para tarjetas perforadas tenía solo un conjunto limitado de caracteres y solo estaba en mayúsculas. Como resultado, los programas de Fortran se veían así:

       DIMENSION A(10)                                                    00000001
C THIS IS A COMMENT STATEMENT TO EXPLAIN THIS EXAMPLE PROGRAM             00000002
       WRITE (6,100)                                                      00000003
 100   FORMAT(169HTHIS IS A RATHER LONG STRING BEING OUTPUT WHICH GOES OVE00000004
      1R MORE THAN ONE LINE, AND USES THE STATEMENT CONTINUATION MARKER IN00000005
      2COLUMN 6, AND ALSO USES HOLLERITH STRING FORMAT)                   00000006
       STOP                                                               00000007
       END                                                                00000008

El carácter de espacio también se ignoró en todas partes, excepto dentro de una constante de carácter Hollerith (como se muestra arriba). Esto significaba que los espacios podían aparecer dentro de palabras reservadas y constantes, o perderse por completo. Esto tuvo el efecto secundario de algunas declaraciones bastante engañosas como:

       DO 1 I = 1.0

es una asignación a la variable DO1I mientras que:

       DO1I = 1,0

es en realidad un bucle DO en la variable I.

Modern Fortran ahora no requiere esta forma fija de entrada y permite la forma libre usando cualquier columna. Los comentarios ahora se indican con un ! que también se puede agregar a una línea de declaración. Los espacios ahora no están permitidos en ninguna parte y deben usarse como separadores, como en la mayoría de los otros idiomas. El programa anterior podría escribirse en Fortran moderno como:

! This is a comment statement to explain this example program
Print *,"THIS IS A RATHER LONG STRING BEING OUTPUT WHICH no longer GOES OVER MORE THAN ONE LINE, AND does not need to USE THE STATEMENT CONTINUATION MARKER IN COLUMN 6, or the HOLLERITH STRING FORMAT"

Aunque la continuación de estilo antiguo ya no se usa, el ejemplo anterior ilustra que todavía se producirán declaraciones muy largas. Fortran moderno usa un símbolo & al final y al principio de la continuación. Por ejemplo, podríamos escribir lo anterior en una forma más legible:

! This is a comment statement to explain this example program
Print *,"THIS IS A RATHER LONG STRING BEING OUTPUT WHICH still &
         &GOES OVER MORE THAN ONE LINE, AND does need to USE THE STATEMENT &
         &CONTINUATION notation"

Bloques comunes #

En las primeras formas de Fortran, el único mecanismo para crear un almacén de variables globales visible desde las subrutinas y funciones es usar el mecanismo de bloque ‘COMÚN’. Esto permitió que las secuencias de variables fueran nombres y fueran compartidas en común.

Además de los bloques comunes con nombre, también puede haber un bloque común en blanco (sin nombre).

Un bloque común en blanco podría declararse como

common i, j

mientras que el bloque con nombre variables podría declararse como

common /variables/ i, j

Como ejemplo completo, podríamos imaginar un almacenamiento dinámico que utilizan las rutinas que pueden agregar y eliminar valores:

       PROGRAM STACKING
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       ICOUNT = 0
       READ *, IVAL
       CALL PUSH(IVAL)
       CALL POP(IVAL)
       END

       SUBROUTINE PUSH(IVAL)
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       ICOUNT = ICOUNT + 1
       ISTACK(ICOUNT) = IVAL
       RETURN
       END

       SUBROUTINE POP(IVAL)
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       IVAL = ISTACK(ICOUNT)
       ICOUNT = ICOUNT - 1
       RETURN
       END

Las sentencias comunes se pueden usar para declarar implícitamente el tipo de una variable y para especificar el atributo dimensión. Este comportamiento por sí solo es a menudo una fuente suficiente de confusión. Además, la asociación de almacenamiento implícita y los requisitos para definiciones repetidas en unidades de programa hacen que el uso de bloques comunes sea propenso a errores.

Finalmente, los bloques comunes están muy restringidos en los objetos que contienen. Por ejemplo, una matriz en un bloque común debe tener un tamaño explícito; es posible que no se produzcan objetos asignables; los tipos derivados no deben tener una inicialización predeterminada.

En Fortran moderno, este intercambio de variables se puede manejar mediante el uso de [módulos] (https://www.wikiod.com/es/fortran/uso-de-modulos). El ejemplo anterior se puede escribir como:

module heap
  implicit none
  ! In Fortran 2008 all module variables are implicitly saved
  integer, save :: count = 0
  integer, save :: stack(1023)
end module heap

program stacking
  implicit none
  integer val
  read *, val
  call push(val)
  call pop(val)

contains
  subroutine push(val)
    use heap, only : count, stack
    integer val
    count = count + 1
    stack(count) = val
  end subroutine push

  subroutine pop(val)
    use heap, only : count, stack
    integer val
    val = stack(count)
    count = count - 1
  end subroutine pop
end program stacking

Los bloques comunes con nombre y en blanco tienen comportamientos ligeramente diferentes. De nota:

  • los objetos en bloques comunes con nombre pueden definirse inicialmente; los objetos en blanco comunes no serán
  • los objetos en bloques comunes en blanco se comportan como si el bloque común tuviera el atributo ‘guardar’; los objetos en bloques comunes con nombre sin el atributo ‘guardar’ pueden volverse indefinidos cuando el bloque no está en el alcance de una unidad de programa activa

Este último punto se puede contrastar con el comportamiento de las variables de módulo en el código moderno. Todas las variables del módulo en Fortran 2008 se guardan implícitamente y no se vuelven indefinidas cuando el módulo queda fuera del alcance. Antes de Fortran 2008, las variables del módulo, como las variables en bloques comunes con nombre, también quedaban sin definir cuando el módulo quedaba fuera del alcance.

Construcciones DO sin bloque #

La construcción do sin bloque se parece a

    integer i
    do 100, i=1, 5
100 print *, i

Es decir, donde la declaración de finalización etiquetada no es una declaración de “continuación”. Hay varias restricciones en la declaración que se puede usar como declaración de terminación y, en general, todo es muy confuso.

Tal construcción sin bloque se puede reescribir en forma de bloque como

    integer i
    do 100 i=1,5
      print *, i
100 continue

o mejor, usando una sentencia de terminación end do,

integer i
do i=1,5
  print *, i
end do

Retorno alternativo #

El retorno alternativo es una función para controlar el flujo de ejecución al regresar de una subrutina. A menudo se usa como una forma de manejo de errores:

real x

call sub(x, 1, *100, *200)
print*, "Success:", x
stop

100 print*, "Negative input value"
stop

200 print*, "Input value too large"
stop

end

subroutine sub(x, i, *, *)
  real, intent(out) :: x
  integer, intent(in) :: i
  if (i<0) return 1
  if (i>10) return 2
  x = i
end subroutine

El retorno alternativo está marcado por los argumentos * en la lista de argumentos ficticios de la subrutina.

En la instrucción call anterior, *100 y *200 se refieren a las declaraciones etiquetadas como 100 y 200 respectivamente.

En la propia subrutina, las sentencias return correspondientes al retorno alternativo tienen un número. Este número no es un valor de retorno, pero denota la etiqueta proporcionada a la que se pasa la ejecución al regresar. En este caso, return 1 pasa la ejecución a la sentencia etiquetada 100 y return 2 pasa la ejecución a la sentencia etiquetada 200. Una declaración return sin adornos, o la finalización de la ejecución de la subrutina sin una declaración return, pasa la ejecución inmediatamente después de la declaración de llamada.

La sintaxis de devolución alternativa es muy diferente de otras formas de asociación de argumentos y la función introduce un control de flujo contrario a los gustos modernos. Se puede administrar un control de flujo más agradable con el retorno de un código de “estado” entero.

real x
integer status

call sub(x, 1, status)
select case (status)
case (0)
  print*, "Success:", x
case (1)
  print*, "Negative input value"
case (2)
  print*, "Input value too large"
end select

end

subroutine sub(x, i, status)
  real, intent(out) :: x
  integer, intent(in) :: i
  integer, intent(out) :: status

  status = 0

  if (i<0) then
    status = 1
  else if (i>10)
    status = 2
  else
    x = i
  end if

end subroutine

GOTO asignado #

El GOTO asignado utiliza una variable entera a la que se asigna una etiqueta de instrucción mediante la instrucción ASSIGN.

100 CONTINUE

...

ASSIGN 100 TO ILABEL

...


GOTO ILABEL

El GOTO asignado está obsoleto en Fortran 90 y se elimina en Fortran 95 y versiones posteriores. Se puede evitar en el código moderno mediante el uso de procedimientos, procedimientos internos, punteros de procedimiento y otras características.

GOTO calculado #

El GOTO calculado permite bifurcar el programa de acuerdo con el valor de una expresión entera.

GOTO (label_1, label_2,... label_n) scalar-integer-expression

Si scalar-integer-expression es igual a 1, el programa continúa en la etiqueta de declaración label_1, si es igual a 2, va a label_2 y así sucesivamente. Si es menor que 1 o mayor que n, el programa continúa en la siguiente línea.

Ejemplo:

ivar = 2

...

GOTO (10, 20, 30, 40) ivar

saltará a la etiqueta de instrucción 20.

Esta forma de goto está obsoleta en Fortran 95 y posterior, siendo reemplazada por la construcción select case.

Especificadores de formato asignados #

Antes de Fortran 95, era posible usar formatos asignados para entrada o salida. Considerar

integer i, fmt
read *, i

assign 100 to fmt
if (i<100000) assign 200 to fmt

print fmt, i

100 format ("This is a big number", I10)
200 format ("This is a small number", I6)

end

La sentencia assign asigna una etiqueta de sentencia a una variable entera. Esta variable entera se utiliza más tarde como especificador de formato en la instrucción imprimir.

Dicha asignación de especificador de formato se eliminó en Fortran 95. En cambio, el código más moderno puede usar alguna otra forma de control de flujo de ejecución.

integer i
read *, i

if (i<100000) then
  print 100, i
else
  print 200, i
end if

100 format ("This is a big number", I10)
200 format ("This is a small number", I6)

end

o se puede usar una variable de carácter como especificador de formato

character(29), target :: big_fmt='("This is a big number", I10)'
character(30), target :: small_fmt='("This is a small number", I6)'
character(:), pointer :: fmt

integer i
read *, i

fmt=>big_fmt
if (i<100000) fmt=>small_fmt

print fmt, i

end

Funciones de declaración #

Considere el programa

implicit none
integer f, i
f(i)=i

print *, f(1)
end

Aquí f es una función de declaración. Tiene un tipo de resultado entero, tomando un argumento ficticio entero.1

Dicha función de declaración existe dentro del ámbito en el que se define. En particular, tiene acceso a variables y constantes con nombre accesibles en ese ámbito.

Sin embargo, las funciones de declaración están sujetas a muchas restricciones y son potencialmente confusas (a simple vista, como una declaración de asignación de elementos de matriz). Las restricciones importantes son:

  • el resultado de la función y los argumentos ficticios deben ser escalares
  • los argumentos ficticios están en el mismo ámbito que la función
  • las funciones de declaración no tienen variables locales
  • las funciones de declaración no se pueden pasar como argumentos reales

Los principales beneficios de las funciones de declaración se repiten mediante funciones internas.

implicit none

print *, f(1)

contains

  integer function f(i)
    integer i
    f = i
  end function

end

Las funciones internas no están sujetas a las restricciones mencionadas anteriormente, aunque quizás valga la pena señalar que un subprograma interno puede no contener más subprogramas internos (pero puede contener una función de declaración).

Las funciones internas tienen su propio alcance, pero también tienen una asociación de host disponible.

1 En ejemplos de código antiguos reales, no sería inusual ver los argumentos ficticios de una función de declaración implícitamente tipeados, incluso si el resultado tiene un tipo explícito.
