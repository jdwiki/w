Invalidación de tokens web Json

Hay varias razones para invalidar un token JWT antes de su fecha de caducidad: cuenta eliminada/bloqueada/suspendida, contraseña o permisos cambiados, usuario desconectado por el administrador.

JWT es autónomo, está firmado y almacenado fuera del contexto del servidor, por lo que revocar un token no es una acción sencilla.

Eliminar el token del almacenamiento del cliente #

Retire el token del almacenamiento del cliente para evitar su uso

Los tokens son emitidos por el servidor y no puede obligar a los navegadores a eliminar una cookie/almacenamiento local o controlar cómo los clientes externos administran sus tokens. Obviamente, si los atacantes robaron el token antes de cerrar la sesión, aún podrían usar el token, por lo tanto, se necesitan medidas adicionales en el lado del servidor (consulte a continuación la estrategia de la lista negra de tokens)

Galletas #

No puede obligar a los navegadores a eliminar una cookie. El cliente puede configurar el navegador de forma que la cookie persista, aunque haya caducado. Pero el servidor puede configurar el valor para que esté vacío e incluir un campo de caducidad para invalidar el valor de la cookie.

 Set-Cookie: token=deleted; path=/; expires=Thu, 01 Jan 1970 00:00:00 GMT

Eliminar ’token’ con javascript #

document.cookie = 'token=; Path=/; Expires=Thu, 01 Jan 1970 00:00:01 GMT;';
localStorage.removeItem('token')
sessionStorage.removeItem('token')

Lista negra de tokens #

Marque los tokens no válidos, guárdelos hasta su fecha de caducidad y compruébelo en cada solicitud.

La lista negra rompe la apatridia de JWT porque requiere mantener el estado. Uno de los beneficios de JWT es que no necesita almacenamiento en el servidor, por lo que si necesita revocar tokens sin esperar a que caduquen, piense también en las desventajas.

Administrar la lista negra #

La lista negra se puede administrar fácilmente en su propio servicio/base de datos. El tamaño de almacenamiento probablemente no sería grande porque solo se necesita para almacenar tokens que estaban entre el cierre de sesión y el tiempo de vencimiento.

Incluya el token completo o solo el ID único jti. Configure el iat (emitido en) para eliminar tokens antiguos.

Para revocar todos los tokens después de actualizar los datos críticos del usuario (contraseña, permisos, etc.), configure una nueva entrada con sub e iat cuando currentTime - maxExpiryTime < last iss. La entrada se puede descartar cuando currentTime - maxExpiryTime > lastModified (no se envían más tokens no caducados).

Rotar fichas #

Establezca tiempo de caducidad corto y rote tokens. Emita un nuevo token de acceso cada pocas solicitudes. Utilice tokens de actualización para permitir que su aplicación obtenga nuevos tokens de acceso sin necesidad de volver a autenticarse

Actualizar y acceder a tokens #

  • token de acceso: autoriza el acceso a un recurso protegido. Tiempo de vida limitado. Debe mantenerse en secreto, las consideraciones de seguridad son menos estrictas debido a su vida más corta.

  • Refrescar token: permite que su aplicación obtenga nuevos tokens de acceso sin necesidad de volver a autenticarse. Larga vida. Almacenar en almacenamiento seguro a largo plazo

Recomendaciones de uso:

  • Aplicaciones web: actualice el token de acceso antes de que caduque, cada vez que el usuario abra la aplicación y en intervalos fijos. Alternativamente, renueve el token de acceso cuando un usuario realiza una acción. Si el usuario usa un token de acceso vencido, la sesión se considera inactiva y se requiere un nuevo token de acceso. Este nuevo token se puede obtener con un token de actualización o requiriendo credenciales

  • Aplicaciones móviles/nativas: Iniciar sesión en la aplicación una sola vez. El token de actualización no caduca y se puede cambiar por un JWT válido. Tenga en cuenta eventos especiales como el cambio de contraseña

Otras técnicas comunes #

  • Permitir cambiar la identificación única del usuario si la cuenta se ve comprometida con un nuevo inicio de sesión de usuario y contraseña

  • Para invalidar tokens cuando el usuario cambia su contraseña o permisos, firme el token con un hash de esos campos. Si alguno de estos campos cambia, los tokens anteriores no se verifican automáticamente. La desventaja es que requiere acceso a la base de datos.

  • Cambiar el algoritmo de firma para revocar todos los tokens actuales en un problema de seguridad importante
