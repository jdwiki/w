Firmar una secuencia de comandos hará que sus secuencias de comandos cumplan con todas las políticas de ejecución en PowerShell y garantizará la integridad de una secuencia de comandos. Los scripts firmados no se ejecutarán si se han modificado después de haberlos firmado.

La firma de scripts requiere un certificado de firma de código. Recomendaciones:

  • Scripts/pruebas personales (no compartidos): Certificado de una autoridad certificadora de confianza (interna o de terceros) O un certificado autofirmado.
  • Organización interna compartida: certificado de una autoridad certificadora de confianza (interna o de terceros)
  • Organización externa compartida: certificado de una autoridad certificadora de terceros de confianza

Lea más en about_Signing @ TechNet

Políticas de ejecución #

PowerShell tiene políticas de ejecución configurables que controlan qué condiciones se requieren para que se ejecute un script o una configuración. Se puede establecer una política de ejecución para varios ámbitos; computadora, usuario actual y proceso actual. Las políticas de ejecución se pueden omitir fácilmente y no están diseñadas para restringir a los usuarios, sino para protegerlos de infringir las políticas de firma sin querer.

Las pólizas disponibles son:

Ajuste Descripción
Restringido No se permiten scripts
AllSigned Todos los scripts deben estar firmados
RemoteSigned Todos los scripts locales permitidos; solo scripts remotos firmados
Sin restricciones Sin requisitos. Se permiten todos los scripts, pero se advertirá antes de ejecutar los scripts descargados de Internet.
Omitir Todos los scripts están permitidos y no se muestran advertencias
Undefined Eliminar la política de ejecución actual para el ámbito actual. Utiliza la política principal. Si todas las políticas no están definidas, se usarán las restringidas.

Puede modificar las políticas de ejecución actuales mediante el cmdlet Set-ExecutionPolicy, la directiva de grupo o el parámetro -ExecutionPolicy al iniciar un proceso powershell.exe.

Lea más en about_Execution_Policies @ TechNet

Omitir la política de ejecución para un solo script #

A menudo, es posible que deba ejecutar un script sin firmar que no cumpla con la política de ejecución actual. Una forma sencilla de hacerlo es omitir la política de ejecución de ese único proceso. Ejemplo:

powershell.exe -ExecutionPolicy Bypass -File C:\MyUnsignedScript.ps1

O puedes usar la abreviatura:

powershell -ep Bypass C:\MyUnsignedScript.ps1

Otras Políticas de Ejecución: #

Política Descripción
TodosFirmados Solo se pueden ejecutar scripts firmados por un editor de confianza.
Pasar por alto Sin restricciones; se pueden ejecutar todos los scripts de Windows PowerShell.
Predeterminado Normalmente RemoteSigned, pero se controla a través de ActiveDirectory
RemotoFirmado Los scripts descargados deben estar firmados por un editor de confianza antes de que puedan ejecutarse.
Restringido No se pueden ejecutar scripts. Windows PowerShell solo se puede usar en modo interactivo.
Indefinido N / A
Sin restricciones* Similar a bypass

Sin restricciones* Advertencia: Si ejecuta un script sin firmar que se descargó de Internet, se le solicitará permiso antes de que se ejecute.

Firmar un guión #

La firma de un script se realiza mediante el cmdlet Set-AuthenticodeSignature y un certificado de firma de código.

#Get the first available personal code-signing certificate for the logged on user
$cert = @(Get-ChildItem -Path Cert:\CurrentUser\My -CodeSigningCert)[0]
    
#Sign script using certificate
Set-AuthenticodeSignature -Certificate $cert -FilePath c:\MyScript.ps1

También puede leer un certificado desde un archivo .pfx usando:

$cert = Get-PfxCertificate -FilePath "C:\MyCodeSigningCert.pfx"

El guión será válido hasta que caduque el certificado. Si utiliza un servidor de marca de tiempo durante la firma, la secuencia de comandos seguirá siendo válida después de que caduque el certificado. También es útil agregar la cadena de confianza para el certificado (incluida la autoridad raíz) para ayudar a la mayoría de las computadoras a confiar en el certificado utilizado para firmar el script.

Set-AuthenticodeSignature -Certificate $cert -FilePath c:\MyScript.ps1 -IncludeChain All -TimeStampServer "http://timestamp.verisign.com/scripts/timstamp.dll"

Se recomienda utilizar un servidor de marca de tiempo de un proveedor de certificados de confianza como Verisign, Comodo, Thawte, etc.

Cambiando la política de ejecución usando Set-ExecutionPolicy #

Para cambiar la política de ejecución para el ámbito predeterminado (LocalMachine), utilice:

Set-ExecutionPolicy AllSigned

Para cambiar la política para un ámbito específico, utilice:

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy AllSigned

Puede suprimir las indicaciones agregando el interruptor -Force.

Obtener la política de ejecución actual #

Obtener la política de ejecución efectiva para la sesión actual:

PS> Get-ExecutionPolicy
RemoteSigned

Enumere todas las políticas de ejecución efectivas para la sesión actual:

PS> Get-ExecutionPolicy -List

        Scope ExecutionPolicy
        ----- ---------------
MachinePolicy       Undefined
   UserPolicy       Undefined
      Process       Undefined
  CurrentUser       Undefined
 LocalMachine    RemoteSigned

Enumere la política de ejecución para un ámbito específico, ej. proceso:

PS> Get-ExecutionPolicy -Scope Process
Undefined

Obtener la firma de un script firmado #

Obtenga información sobre la firma de Authenticode de un script firmado mediante el cmdlet Get-AuthenticodeSignature:

Get-AuthenticodeSignature .\MyScript.ps1 | Format-List *

Creación de un certificado de firma de código autofirmado para pruebas #

Al firmar scripts personales o al probar la firma de código, puede ser útil crear un certificado de firma de código autofirmado.

A partir de PowerShell 5.0, puede generar un certificado de firma de código autofirmado mediante el cmdlet New-SelfSignedCertificate:

New-SelfSignedCertificate -FriendlyName "StackOverflow Example Code Signing" -CertStoreLocation Cert:\CurrentUser\My -Subject "SO User" -Type CodeSigningCert

En versiones anteriores, puede crear un certificado autofirmado con la herramienta makecert.exe que se encuentra en .NET Framework SDK y Windows SDK.

Solo las computadoras que hayan instalado el certificado confiarán en un certificado autofirmado. Para los scripts que se compartirán, se recomienda un certificado de una autoridad de certificación de confianza (interna o de terceros de confianza).
