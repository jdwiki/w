En este tema se explica cómo asignar relaciones de tipo uno a uno mediante Entity Framework.

Asignación de uno a cero o uno #

Así que digamos de nuevo que tienes el siguiente modelo:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
}

public class MyDemoContext : DbContext
{
  public DbSet<Person> People { get; set; }
  public DbSet<Car> Cars { get; set; }
}

Y ahora desea configurarlo para que pueda expresar la siguiente especificación: una persona puede tener uno o ningún automóvil, y cada automóvil pertenece exactamente a una persona (las relaciones son bidireccionales, por lo que si CarA pertenece a PersonA, entonces PersonA ‘posee ’ Coche A).

Entonces, modifiquemos un poco el modelo: agregue las propiedades de navegación y las propiedades de la clave externa:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
  public int CarId { get; set; }
  public virtual Car Car { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

Y la configuración:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);                        
  }
}

En este momento esto debería explicarse por sí mismo. El auto tiene una persona requerida (HasRequired()), y la persona tiene un auto opcional (WithOptional()). Nuevamente, no importa de qué lado configure esta relación, solo tenga cuidado cuando use la combinación correcta de Tiene/Con y Requerido/Opcional. Desde el lado Persona, se vería así:

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
     this.HasOptional(p => p.Car).WithOptional(c => c.Person);                        
  }
}

Ahora veamos el esquema db:

Mire de cerca: puede ver que no hay FK en Personas para referirse a Car. Además, el FK en Car no es PersonId, sino CarId. Aquí está el guión real para el FK:

ALTER TABLE [dbo].[Cars]  WITH CHECK ADD  CONSTRAINT [FK_dbo.Cars_dbo.People_CarId] FOREIGN KEY([CarId])
REFERENCES [dbo].[People] ([PersonId])

Entonces, esto significa que las propiedades clave foregn CarId y PersonId que tenemos en el modelo son básicamente ignoradas. Están en la base de datos, pero no son claves foráneas, como cabría esperar. Esto se debe a que las asignaciones uno a uno no admiten agregar el FK en su modelo EF. Y eso se debe a que las asignaciones uno a uno son bastante problemáticas en una base de datos relacional.

La idea es que cada persona pueda tener exactamente un automóvil, y ese automóvil solo puede pertenecer a esa persona. O puede haber registros de personas que no tengan automóviles asociados.

Entonces, ¿cómo podría representarse esto con claves foráneas? Obviamente, podría haber un PersonId en Car y un CarId en People. Para hacer cumplir que cada persona puede tener solo un automóvil, PersonId tendría que ser único en Car. Pero si PersonId es único en Personas, entonces, ¿cómo puede agregar dos o más registros donde PersonId es NULL (más de un automóvil que no tiene dueño)? Respuesta: no puede (bueno, en realidad, puede crear un índice único filtrado en SQL Server 2008 y posteriores, pero olvidemos este tecnicismo por un momento, sin mencionar otros RDBMS). Sin mencionar el caso en el que especificas ambos extremos de la relación…

La única forma real de hacer cumplir esta regla es si las tablas Personas y Coche tienen la ‘misma’ clave principal (los mismos valores en los registros conectados). Y para hacer esto, CarId en Car debe ser tanto un PK como un FK para el PK de Personas. Y esto hace que todo el esquema sea un desastre. Cuando uso esto, prefiero nombrar el PK/FK en Car PersonId, y configurarlo en consecuencia:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }        
  public virtual Car Car { get; set; }
}

public class Car
{        
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);
     this.HasKey(c => c.PersonId);
  }
}

No es ideal, pero tal vez un poco mejor. Aún así, debe estar alerta al usar esta solución, porque va en contra de las convenciones de nomenclatura habituales, lo que podría desviarlo. Aquí está el esquema generado a partir de este modelo:

Por lo tanto, esta relación no la impone el esquema de la base de datos, sino el propio Entity Framework. Por eso hay que tener mucho cuidado al usar esto, de no dejar que nadie altere directamente con la base de datos.

Mapeo uno a uno #

Mapear uno a uno (cuando se requieren ambos lados) también es complicado.

Imaginemos cómo se podría representar esto con claves foráneas. Nuevamente, un ‘CarId’ en ‘Personas’ que se refiere a ‘CarId’ en ‘Car’, y un ‘PersonId’ en Car que se refiere a ‘PersonId’ en ‘Personas’.

Ahora, ¿qué sucede si desea insertar un registro de automóvil? Para que esto tenga éxito, debe haber un PersonId especificado en este registro de automóvil, porque es obligatorio. Y para que este PersonId sea válido, debe existir el registro correspondiente en Personas. Bien, sigamos adelante e insertemos el registro de la persona. Pero para que esto tenga éxito, debe haber un ‘CarId’ válido en el registro de persona — ¡pero ese auto aún no está insertado! No puede ser, porque primero tenemos que insertar el registro de la persona referida. Pero no podemos insertar el registro de la persona referida, porque hace referencia al registro del automóvil, por lo que debe insertarse primero (recepción de llave extranjera :)).

Entonces esto tampoco se puede representar de la manera ’lógica’. Nuevamente, debe soltar una de las claves externas. Cuál dejas depende de ti. El lado que queda con una clave foránea se llama ‘dependiente’, el lado que queda sin una clave foránea se llama ‘principal’. Y nuevamente, para garantizar la unicidad en el dependiente, el PK tiene que ser el FK, por lo que no se admite agregar una columna FK e importarla a su modelo.

Así que aquí está la configuración:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasRequired(c => c.Person).WithRequiredDependent(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

A estas alturas ya deberías haber entendido la lógica :) Solo recuerda que también puedes elegir el otro lado, solo ten cuidado de usar las versiones Dependiente/Principal de WithRequired (y aún tienes que configurar el PK en Car).

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
    this.HasRequired(p => p.Car).WithRequiredPrincipal(c => c.Person);
  }
}

Si revisa el esquema de base de datos, encontrará que es exactamente igual que en el caso de la solución uno a uno o cero. Esto se debe a que, de nuevo, esto no lo aplica el esquema, sino el propio EF. Así que de nuevo, ten cuidado :)

Asignación de uno o cero a uno o cero #

Y para terminar, veamos brevemente el caso en que ambos lados son opcionales.

A estas alturas deberías estar realmente aburrido con estos ejemplos :), así que no voy a entrar en detalles y jugar con la idea de tener dos FK-s y los problemas potenciales y advertirte sobre los peligros de no hacer cumplir estas reglas en el esquema pero solo en EF.

Aquí está la configuración que necesita para aplicar:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasOptional(c => c.Person).WithOptionalPrincipal(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Nuevamente, también puede configurar desde el otro lado, solo tenga cuidado de usar los métodos correctos :)
