Cómo solucionar los bloqueos de acceso

  • Probar la memoria de la computadora

Cuando recibe un error: “Microsoft Access encontró un problema y debe cerrarse”, a menudo no hay mucha información para ayudarlo a identificar la causa del error. A continuación hay una serie de pasos que puede seguir para solucionar la causa de los errores.

Asegúrese de eliminar otras variables de la ecuación durante la prueba

Corrupción en la red

No cargue el cliente fuera de una red. Póngalo en el disco local y ejecútelo desde allí.

Construcciones corporativas

Si se encuentra en un entorno corporativo que usa “compilaciones de computadoras” y no ha tenido éxito con la descompilación, la prueba de la memoria ni la extracción de datos binarios, rehúse realizar más pruebas hasta que el equipo de TI pueda proporcionar al usuario una máquina de prueba que tenga solo Windows, Office y Service Pack instalados.

Todo el software y las actualizaciones deben instalarse a mano sin utilizar instalaciones desatendidas. No instale antivirus en esta máquina para realizar pruebas.

Comprenda que muchos departamentos de TI simplemente intentan hacer un enfoque único para todos con las compilaciones, y todas sus compilaciones se basan unas en otras. Con el tiempo, los conflictos de software pueden hacer que Access se bloquee o actúe de forma extraña.

Mal poder

Como se mencionó en el ejemplo de la memoria, las fluctuaciones de energía pueden causar errores en la computadora. Si la base de datos está en un edificio industrial, intente conseguir un acondicionador de energía o un UPS que proporcione energía limpia (fuera de la batería, no de la red eléctrica que pasa a través de un varistor de óxido de metal)

Además, verifique el cable de alimentación que se conecta a la barra de alimentación o al tomacorriente. Asegúrese de que las especificaciones de calibre y voltaje sean suficientes. Los departamentos de TI a menudo dejan los cables de alimentación enchufados en la estación y simplemente retiran la máquina. Después de muchos años, están usando fuentes de alimentación más potentes, pero no han cambiado el cable. Hace la diferencia. En caso de duda, lleve un cable nuevo y más grueso.

Descompilar base de datos #

Esta siempre debe ser su solución inicial. Una buena política es descompilar la base de datos antes de cada lanzamiento.

  1. Cree un acceso directo de descompilación. Esto carga la base de datos con un modificador “/descompilar”.

  2. Haga clic derecho en su archivo de base de datos. Seleccione Copiar

  3. Haga clic derecho en la ventana del explorador y seleccione “Pegar acceso directo”

  4. Haga clic derecho en el acceso directo y seleccione “Propiedades”

  5. En el cuadro Destino, vaya al final de la línea y agregue /decompile

  6. Haga clic en Aceptar para cerrar el acceso directo

  7. Abrir base de datos con Shift. Mantenga presionada la tecla Mayús mientras hace doble clic en este acceso directo. Esto evita que las ejecuciones automáticas se ejecuten dentro de la base de datos. Debe ir directamente a la ventana de navegación.

  8. Compactar y reparar la base de datos. Una vez que se carga la base de datos, deberá hacer clic en el botón Compactar y reparar.

  9. Localice el botón Compactar y reparar base de datos en la cinta Herramientas.

  10. Mantenga presionada la tecla Mayús. Manténgalo presionado mientras hace clic en el botón Compactar y reparar.

  11. Recompilar la base de datos

  12. Vaya a la ventana de VBA (Control + G)

  13. Seleccione Depurar -> Compilar en el menú

Este es el proceso de descompilación completo. En general, debería solucionar el 99 % de todos los bloqueos de Access o el comportamiento extraño de los formularios.

Probar la memoria de la computadora #

Si sus bloqueos son aleatorios o esporádicos, realice este paso. Si sus fallas ocurren cada vez que ejecuta la base de datos, entonces este paso no solucionará el problema (aunque la mala memoria puede ser la razón por la cual ocurrió la corrupción en primer lugar).

Utilice un probador de memoria que arranque fuera del sistema operativo y ejecute varias pasadas. Dos opciones populares son MemTest86 (comercial) y MemTest86+ (código abierto)

Inicie la prueba y déjela funcionar durante las horas de trabajo. La razón de esto es que otros factores en el edificio, como el ruido en los circuitos de alimentación, pueden causar errores de memoria, por lo que debe tratar de mantener las mismas variables.

Si tiene errores de memoria, deberá identificar si se debe a una mala memoria en la computadora o a algún otro factor. Esto está más allá del alcance de este documento sin embargo.

Eliminar datos binarios del formulario #

A veces, los bloqueos ocurren constantemente en un solo formulario o informe, o solo cuando se imprime. Es posible que los datos binarios dentro del formulario/informe se hayan dañado.

Guardar el objeto Formulario/Informe como texto Hay dos funciones no documentadas. Aplicación.SaveAsText y Aplicación.LoadFromText. Puede usar estas funciones para exportar las definiciones de formulario/informe, limpiar la definición y luego importarla nuevamente.

  1. Haz una copia de seguridad de tu base de datos antes de continuar
  2. Vaya a la ventana inmediata de VBA (Control + G)
  3. Escriba Application.SaveAsText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt" (Reemplace MyForm con el nombre del Formulario / Informe. Use acReport si es un informe corrupto que está arreglando)
  4. Cambie el nombre del elemento del formulario original (por ejemplo, cambie el nombre a MyForm.Bak) dentro de la ventana de la base de datos

Limpiar el archivo de definiciones de formulario/informe

  1. Abra el archivo exportado (por ejemplo, MyForm.txt) en el bloc de notas

  2. Elimine la línea “Checksum=” (debe estar en la línea 3)

  3. Borrar datos binarios

  4. Identifique los bloques de datos binarios. Examine el archivo y verá líneas que comienzan con “Parameter = Begin”. Siguiendo esas líneas, tendrá líneas de datos binarios codificados. Finalmente, el bloque binario terminará con una línea que consistirá únicamente en “End”. El bloque de datos binarios incluye la primera línea (con la instrucción Begin) y todas las líneas hasta la última línea incluida (con la instrucción End).

    Note: All of these blocks should appear BEFORE your form control definitions

  5. Borre los bloques de datos binarios para los siguientes parámetros:

    • NameMap
    • PrtMip
    • PrtDevMode
    • PrtDevNames
    • PrtDevModeW
    • PrtDevNamesW

  6. Busque otros problemas. Mientras tiene el archivo abierto, desplácese por el resto del archivo y busque cualquier cosa que le llame la atención, especialmente en el código del módulo VBA en la parte inferior. Estarás buscando cualquier cosa que sobresalga del resto, y puede ser corrupción.

  7. Guarde el archivo.

Vuelva a cargar el formulario/informe en Acceso y prueba

  1. Vuelva a cargar el formulario en Access.
    • In Access, go to the immediate window (Control + G)
    • Type Application.LoadFromText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt"
    • Decompile / Compact Repair / Recompile (See the other example within the documentation)
    • Open the form / report to test. Hopefully everything is working now.
    • Delete the old corrupt form (e.g. MyForm.bak)

Prevenir esta corrupción en el futuro

La causa más común de datos binarios corruptos dentro de un informe/formulario es cuando varias computadoras/usuarios usan el mismo archivo de cliente de base de datos en lugar de tener su propia copia separada. Esta es la razón por la que cada usuario debe tener su propio archivo de cliente en su escritorio que ejecute.

Eliminar los campos “Objeto OLE” #

Si tiene imágenes u otros datos almacenados en Access como objetos OLE, entonces debería encontrar un mejor enfoque. Cuando se almacenan los datos OLE, se almacenan de acuerdo con el software (y la versión del software) en la computadora que los almacena. Cuando otra computadora va a mostrar los datos del objeto OLE en el formulario, pero no tiene instalada la versión o el software exactos, a menudo esto resulta en un bloqueo de la aplicación.

Si está almacenando datos de imágenes, entonces un mejor enfoque es almacenar el nombre del archivo y, en su lugar, guardar las imágenes en una ubicación estándar. Las versiones más nuevas de acceso tienen los controles nativos para que este sea el camino a seguir.

Reconstruir toda la base de datos #

Esto es mucho trabajo, así que hazlo como último recurso después de agotar todas las demás opciones. Solo necesita hacer esto si el problema ocurre con diferentes usuarios, en diferentes máquinas. Si no ocurre con todos los usuarios, lo más probable es que no se trate de un contenedor de base de datos corrupto.

De manera similar a los pasos para eliminar datos binarios, reconstruirá su base de datos desde cero. Este proceso es un poco ritualista, pero si se hace meticulosamente con cuidado de no “preservar” cualquier posible corrupción, entonces el proceso es altamente efectivo.

Cree un nuevo contenedor de base de datos de acceso.

  • En Access, en la pestaña Archivo, puede seleccionar “Nuevo”. Cree una nueva base de datos vacía en formato ACCDB.

Mover todos los objetos al nuevo contenedor

No use las funciones Importar/Exportar dentro de Access para mover los objetos, y no simplemente haga clic y arrastre. Hacer esto puede copiar los elementos corruptos al nuevo contenedor.

Mesas:

  • Para cada tabla en el antiguo contenedor de acceso, cree una nueva tabla en el nuevo contenedor.
  • Desde la vista de diseño, copie/pegue las definiciones de campo.
  • Verifique las propiedades de la tabla para asegurarse de que coincidan en ambas bases de datos
  • Mueva cualquier macro de datos también (consulte la sección Macros para saber cómo hacerlo)
  • Para mover los datos, exporte los datos antiguos a XML o CSV y luego impórtelos desde ese formato.

Consultas:

  • Cargue cada consulta en la vista SQL.
  • Copiar/Pegar el texto SQL.
  • Pegar en la nueva base de datos.
  • Compare las propiedades de consulta para asegurarse de que coincidan.

Formas e informes:

  • Para cada formulario/informe, use la función Application.SaveAsText para exportar los formularios/informes a un archivo de texto.
  • Eliminar los datos binarios (consulte la documentación Eliminar datos binarios del formulario para familiarizarse con este proceso)
  • Use la función Application.LoadFromText para volver a importar los objetos a la nueva base de datos

Macros

Tienes tres métodos para mover las macros.

  1. Vuelva a crear cada macro a mano en el nuevo contenedor de la base de datos.
  2. Use el método Application.SaveAsText / Application.LoadFromText con el parámetro acMacro.
  3. Copiar/Pegar definiciones de macro para cada macro
  • Seleccione Todo (Control + A) para seleccionar todos los elementos macro. Luego Copiar (Control + C).
  • Abra un documento de Bloc de notas en blanco y pegue (Control + V) el XML de macro.
  • Cree una nueva macro en blanco en el nuevo contenedor de la base de datos.
  • En el Bloc de notas, seleccione Todo el texto (Control + A). Luego Copiar (Control + C)
  • En la macro en blanco, Pegar (Control + V). Debería aparecer la macro. Guárdalo.

Módulos

  • Para cada módulo, seleccione todo el código (Control + A) y péguelo (Control + V) en el nuevo contenedor de la base de datos.
  • Asegúrese de verificar las propiedades de la base de datos (en la ventana de VBA, vaya a Herramientas -> Propiedades del cliente)

Macros de datos

Para cada macro de datos, utilice los métodos SaveAsText/LoadFromText.

  1. Vaya a la ventana inmediata de VBA (Control + G)
  2. Escriba Application.SaveAsText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt" (Reemplace MyTableName con el nombre de la tabla que contiene las macros de datos)
  3. Revise el archivo en busca de signos de corrupción
  4. En el nuevo contenedor de la base de datos, cargue la definición usando Application.LoadFromText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt"

Como se mencionó anteriormente, este es un MUCHO trabajo, pero tiene resultados. Este método también debe utilizarse al migrar una base de datos de Access 97 a 2000 o una base de datos de Access 2000 a 2003.
