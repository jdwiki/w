Cada subproceso tendrá su propio código de último error. La API de Windows establecerá el último código de error en el hilo de llamada.

Siempre debe llamar a la función GetLastError inmediatamente después de verificar el valor de retorno de una función API de Windows.

La mayoría de las funciones de la API de Windows establecen el último código de error cuando fallan. Algunos también establecerán el último código de error cuando tengan éxito. Hay una serie de funciones que no establecen el último código de error. Consulte siempre la documentación de la función API de Windows.

No es seguro usar FORMAT_MESSAGE_FROM_SYSTEM sin FORMAT_MESSAGE_IGNORE_INSERTS cuando se usa la función FormatMessage para obtener una descripción de un código de error.

Convertir un código de error en una cadena de mensaje #

[GetLastError](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado con información adicional sobre la falla) devuelve un código de error numérico. Para obtener un mensaje de error descriptivo (p. ej., para mostrar a un usuario), puede llamar a FormatMessage:

// This functions fills a caller-defined character buffer (pBuffer)
// of max length (cchBufferLength) with the human-readable error message
// for a Win32 error code (dwErrorCode).
// 
// Returns TRUE if successful, or FALSE otherwise.
// If successful, pBuffer is guaranteed to be NUL-terminated.
// On failure, the contents of pBuffer are undefined.
BOOL GetErrorMessage(DWORD dwErrorCode, LPTSTR pBuffer, DWORD cchBufferLength)
{
    if (cchBufferLength == 0)
    {
        return FALSE;
    }

    DWORD cchMsg = FormatMessage(FORMAT_MESSAGE_FROM_SYSTEM | FORMAT_MESSAGE_IGNORE_INSERTS,
                                 NULL,  /* (not used with FORMAT_MESSAGE_FROM_SYSTEM) */
                                 dwErrorCode,
                                 MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT),
                                 pBuffer,
                                 cchBufferLength,
                                 NULL);
    return (cchMsg > 0);
}

En C++, puede simplificar considerablemente la interfaz usando la clase std::string:

#include <Windows.h>
#include <exception>
#include <stdexcept>
#include <memory>
#include <string>
typedef std::basic_string<TCHAR> String;

String GetErrorMessage(DWORD dwErrorCode)
{
    LPTSTR psz = NULL;
    const DWORD cchMsg = FormatMessage(FORMAT_MESSAGE_FROM_SYSTEM
                                         | FORMAT_MESSAGE_IGNORE_INSERTS
                                         | FORMAT_MESSAGE_ALLOCATE_BUFFER,
                                       NULL, // (not used with FORMAT_MESSAGE_FROM_SYSTEM)
                                       dwErrorCode,
                                       MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT),
                                       reinterpret_cast<LPTSTR>(&psz),
                                       0,
                                       NULL);
    if (cchMsg > 0)
    {
        // Assign buffer to smart pointer with custom deleter so that memory gets released
        // in case String's c'tor throws an exception.
        auto deleter = [](void* p) { ::HeapFree(::GetProcessHeap(), 0, p); };
        std::unique_ptr<TCHAR, decltype(deleter)> ptrBuffer(psz, deleter);
        return String(ptrBuffer.get(), cchMsg);
    }
    else
    {
        throw std::runtime_error("Failed to retrieve error message string.");
    }
}

NOTA: Estas funciones también funcionan para [valores HRESULT](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado como valor HRESULT valor). Simplemente cambie el primer parámetro de DWORD dwErrorCode a HRESULT hResult. El resto del código puede permanecer sin cambios.

Introducción #

La API de Windows se proporciona por medio de una interfaz de llamada C. El éxito o el fracaso de una llamada a la API se informa estrictamente a través de los valores devueltos. Las excepciones no forman parte del contrato documentado (aunque algunas implementaciones de API pueden generar SEH excepciones, por ejemplo, al pasar un argumento lpCommandLine de solo lectura a CreateProcess).

El informe de errores cae aproximadamente en una de cuatro categorías:

  • [Solo valor devuelto](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado solo por valor de retorno)
  • [Valor devuelto con información adicional sobre el error](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado con información adicional sobre la falla)
  • [Valor devuelto con información adicional sobre fallas y éxitos](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado con información adicional sobre falla y éxito -Y éxito)
  • [Valor de retorno de HRESULT](https://www.wikiod.com/es/winapi/reporte-y-manejo-de-errores#Error informado como valor HRESULTvalue)

La documentación de cada llamada a la API indica explícitamente cómo se notifican los errores. Consulte siempre la documentación.

Error informado solo por el valor de retorno #

Algunas llamadas a la API devuelven un único indicador de error/éxito, sin ninguna información adicional (por ejemplo, GetObject):

if ( GetObjectW( obj, 0, NULL ) == 0 ) {
    // Failure: no additional information available.
}

Error informado con información adicional sobre la falla #

Además de un valor de retorno de falla/éxito, algunas llamadas a la API también establecen el último error en caso de falla (por ejemplo, CreateWindow ). La documentación suele contener la siguiente redacción estándar para este caso:

Si la función tiene éxito, el valor devuelto es <Valor de éxito específico de la API>. Si la función falla, el valor de retorno es <Valor de error específico de API>. Para obtener información de error ampliada, llame a GetLastError.

if ( CreateWindowW( ... ) == NULL ) {
    // Failure: get additional information.
    DWORD dwError = GetLastError();
} else {
    // Success: must not call GetLastError.
}

Es fundamental que llame a GetLastError() INMEDIATAMENTE. El último código de error puede ser sobrescrito por cualquier otra función, por lo que si hay una llamada de función adicional entre la función que falló y la llamada a GetLastError() , el retorno de GetLastError() ya no será confiable. Tenga mucho cuidado cuando trabaje con constructores de C++.

Una vez que obtenga un código de error, deberá interpretarlo. Puede obtener una lista completa de códigos de error en MSDN, en la [página de Códigos de error del sistema (Windows)](https://msdn.microsoft.com/en-us/library/windows/desktop/ms681381(v=vs. 85).aspx). Alternativamente, puede buscar en los archivos de encabezado de su sistema; el archivo con todas las constantes del código de error es winerror.h. (Si tiene el SDK oficial de Microsoft para Windows 8 o posterior, se encuentra en la subcarpeta shared de la carpeta de inclusión).

Notas sobre cómo llamar a GetLastError() en otros lenguajes de programación #

Lenguajes .net (C#, VB, etc.) #

Con .net, no debe P/Invocar a GetLastError() directamente. Esto se debe a que el tiempo de ejecución de .net realizará otras llamadas a la API de Windows en el mismo hilo a sus espaldas. Por ejemplo, el recolector de basura podría llamar a VirtualFree() si encuentra suficiente memoria que ya no está usando, y esto puede suceder entre la llamada a la función prevista y la llamada a GetLastError().

En su lugar, .net proporciona la función Marshal.GetLastWin32Error(), que recuperará el último error de la última llamada P/Invoke que usted mismo realizó. Use esto en lugar de llamar a GetLastError() directamente.

(.net no parece impedirle importar GetLastError() de todos modos; no estoy seguro de por qué).

Vamos #

Las diversas funciones proporcionadas por Go para llamar a funciones DLL (que residen tanto en el paquete syscall como en el paquete golang.org/x/sys/windows) devuelven tres valores: r1, r2 y err. r2 nunca se usa; puede usar el identificador en blanco allí. r1 es el valor de retorno de la función. err es el resultado de llamar a GetLastError() pero convertido en un tipo que implementa error, por lo que puede pasarlo a las funciones de llamada para que lo manejen.

Debido a que Go no sabe cuándo llamar a GetLastError() y cuándo no, siempre devolverá un error que no sea nil. Por lo tanto, el idioma típico de manejo de errores de Go

r1, _, err := syscall.Syscall12(CreateWindowW.Addr(), ...)
if err != nil {
    // handle err
}
// use r1

no trabajará. En su lugar, debe verificar r1, exactamente como lo haría en C, y solo usar err si eso indica que la función devolvió un error:

r1, _, err := syscall.Syscall12(CreateWindowW.Addr(), ...)
if r1 == 0 {
    // handle err
}
// use r1

Error informado con información adicional sobre falla y éxito #

Algunas llamadas API pueden tener éxito o fallar en más de una forma. Las API normalmente devuelven información adicional tanto para las invocaciones correctas como para los errores (por ejemplo, CreateMutex).

if ( CreateMutexW( NULL, TRUE, L"Global\\MyNamedMutex" ) == NULL ) {
    // Failure: get additional information.
    DWORD dwError = GetLastError();
} else {
    // Success: Determine which mutex was returned.
    if ( GetLastError() == ERROR_ALREADY_EXISTS ) {
        // Existing mutex object returned.
    } else {
        // Newly created mutex object returned.
    }
}

Error informado como valor HRESULT #

HRESULT son valores numéricos de 32 bits, donde los bits o los rangos de bits codifican información bien definida. El MSB es un indicador de falla/éxito, y los bits restantes almacenan información adicional. La falla o el éxito se pueden determinar usando [FALLIDO] (https://msdn.microsoft.com/en-us/library/windows/desktop/ms693474.aspx) o [SUCCEEDED] (https://msdn.microsoft.com /en-us/library/windows/desktop/ms687197.aspx) macros. HRESULTs se usan comúnmente con COM, pero también aparecen en implementaciones que no son COM (por ejemplo, StringCchPrintf) .

const size_t cchBuf = 5;
wchar_t buffer[cchBuf] = { 0 };
HRESULT hr = StringCchPrintfW( buffer, cchBuf, L"%s", L"Hello, world!" );
if ( FAILED( hr ) ) {
    // Failure: Determine specific reason.
    switch ( hr ) {
    case STRSAFE_E_INSUFFICIENT_BUFFER:
        // Buffer too small; increase buffer and retry.
        ...
    case STRSAFE_E_INVALID_PARAMETER:
        // Invalid parameter; implement custom error handling (e.g. logging).
        ...
    default:
        // Some other error code; implement custom error handling (e.g. logging).
        ...
    }
}
