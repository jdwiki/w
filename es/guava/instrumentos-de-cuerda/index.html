Comprobación de una cadena en busca de caracteres no deseados #

Como desarrollador, con frecuencia se encuentra lidiando con cadenas que no son creadas por su propio código.

Estos a menudo serán proporcionados por bibliotecas de terceros, sistemas externos o incluso usuarios finales. La validación de cadenas de procedencia poco clara se considera uno de los sellos distintivos de la programación defensiva y, en la mayoría de los casos, deseará rechazar la entrada de cadena que no cumpla con sus expectativas.

Un caso bastante común es en el que solo desearía permitir caracteres alfanuméricos en una cadena de entrada, por lo que lo usaremos como ejemplo. En Java simple, los siguientes dos métodos tienen el mismo propósito:

public static boolean isAlphanumeric(String s) {
    for (char c : s.toCharArray()) {
        if (!Character.isLetterOrDigit(c)) {
            return false;
        }
    }

    return true;
}

public static boolean isAlphanumeric(String s) {
    return s.matches("^[0­-9a­-zA­-Z]*$");
}

La primera versión convierte la cadena en una matriz de caracteres y luego utiliza el método estático isLetterOrDigit de la clase Character para determinar si los caracteres contenidos en la matriz son alfanuméricos o no. Este enfoque es predecible y legible, aunque un poco detallado.

La segunda versión usa una expresión regular para lograr el mismo propósito. Es más conciso, pero puede resultar algo enigmático para los desarrolladores con un conocimiento limitado o nulo de las expresiones regulares.

Guava presenta la clase CharMatcher para hacer frente a este tipo de situaciones. Nuestra prueba alfanumérica, usando guayaba, quedaría de la siguiente manera:

import static com.google.common.base.CharMatcher.javaLetterOrDigit;

/* ... */

public static boolean isAlphanumeric(String s) {
    return javaLetterOrDigit().matchesAllOf(s);
}

El cuerpo del método contiene solo una línea, pero en realidad están sucediendo muchas cosas aquí, así que analicemos las cosas un poco más.

Si observa la API de la clase CharMatcher de Guava, notará que implementa la interfaz Predicate<Character>. Si usted mismo creara una clase que implemente Predicate<Character>, podría verse así:

import com.google.common.base.Predicate;

public class AlphanumericPredicate implements Predicate<Character> {
    @Override
    public boolean apply(Character c) {
        return Character.isLetterOrDigit(c);
    }
}

En Guava, como en otros lenguajes de programación y bibliotecas que se adaptan a un estilo funcional de programación, un predicado es una construcción que evalúa una entrada dada como verdadera o falsa. En la interfaz Predicate<T> de Guava, esto se hace evidente por la presencia del único método boolean apply(T t). La clase CharMatcher se basa en este concepto y evaluará un carácter o una secuencia de caracteres para verificar si coinciden o no con los criterios establecidos por la instancia CharMatcher utilizada.

Guava proporciona actualmente los siguientes comparadores de caracteres predefinidos:

emparejador Descripción
cualquiera() Coincide con cualquier carácter.
ninguno() No coincide con ningún personaje.
JavaDigit() Coincide con dígitos, según la definición de Java.
javaUpperCase() Coincide con cualquier carácter en mayúscula, según la definición de Java.
javaLowerCase() Coincide con cualquier carácter en minúscula, según la definición de Java.
JavaLetter() Coincide con cualquier letra, según la definición de Java.
JavaLetraODígito() Coincide con cualquier letra o dígito, según la definición de Java.
javaIsoControl() Coincide con cualquier carácter de control ISO, según la definición de Java.

Si ha leído la tabla anterior, sin duda habrá notado la cantidad de definición y especificación involucrada en determinar qué caracteres pertenecen a una determinada categoría. El enfoque de Guava, hasta ahora, ha sido proporcionar contenedores CharMatcher para varias categorías de caracteres definidas por Java, y puede consultar la API de la clase Character de Java para obtener más información sobre estas categorías. Por otro lado, Guava intenta proporcionar una serie de instancias de CharMatcher que están en línea con la especificación Unicode actual. Para conocer los detalles esenciales, consulte la documentación de la API CharMatcher.

Volviendo a nuestro ejemplo de verificar una cadena en busca de caracteres no deseados, los siguientes métodos CharMatcher brindan las capacidades que necesita para verificar si el uso de caracteres de una cadena determinada cumple con sus requisitos:

  • coincidencias booleanasNoneOf(secuencia CharSequence) Devuelve verdadero si ninguno de los caracteres en la cadena del argumento coincide con la instancia de CharMatcher.

  • coincidencias booleanasAnyOf(secuencia CharSequence) Devuelve verdadero si al menos un carácter en la cadena del argumento coincide con la instancia de CharMatcher.

  • coincidencias booleanasAllOf(secuencia CharSequence) Devuelve verdadero si todos los caracteres de la cadena del argumento coinciden con la instancia de CharMatcher.

Encontrar y contar caracteres en una cadena #

Para ayudarlo a encontrar y contar caracteres en una cadena, CharMatcher proporciona los siguientes métodos:

  • int indexIn(secuencia CharSequence) Devuelve el índice del primer carácter que coincide con la instancia de CharMatcher. Devuelve -1 si no coincide ningún carácter.

  • int indexIn(secuencia CharSequence, int start) Devuelve el índice del primer carácter después de la posición de inicio especificada que coincide con la instancia de CharMatcher. Devuelve -1 si no coincide ningún carácter.

  • int lastIndexIn(secuencia CharSequence) Devuelve el índice del último carácter que coincide con la instancia de CharMatcher. Devuelve -1 si no coincide ningún carácter.

  • int countIn (secuencia CharSequence) Devuelve el número de caracteres que coinciden con la instancia de CharMatcher.

Usando estos métodos, aquí hay una aplicación de consola simple llamada NonAsciiFinder que toma una cadena como argumento de entrada. Primero, imprime el número total de caracteres que no son ASCII contenidos en la cadena. Posteriormente, imprime la representación Unicode de cada carácter no ASCII que encuentra. Aquí está el código:

import com.google.common.base.CharMatcher;

public class NonAsciiFinder {
    private static final CharMatcher NON_ASCII = CharMatcher.ascii().negate();

    public static void main(String[] args) {
        String input = args[0];
        int nonAsciiCount = NON_ASCII.countIn(input);

        echo("Non-ASCII characters found: %d", nonAsciiCount);

        if (nonAsciiCount > 0) {
            int position = -­1;
            char character = 0;

            while (position != NON_ASCII.lastIndexIn(input)) {
                position = NON_ASCII.indexIn(input, position + 1);
                character = input.charAt(position);
                
                echo("%s => \\u%04x", character, (int) character);
            }
        }
    }

    private static void echo(String s, Object... args) {
        System.out.println(String.format(s, args));
    }
}

Tenga en cuenta en el ejemplo anterior cómo puede simplemente invertir un CharMatcher llamando a su método negate. De manera similar, el ‘CharMatcher’ a continuación coincide con todos los caracteres de ancho doble y se crea negando el ‘CharMatcher’ predefinido para los caracteres de ancho simple.

final static CharMatcher DOUBLE_WIDTH = CharMatcher.singleWidth().negate();

î => \u00ee
é => \u00e9

$> java NonAsciiFinder "古池や蛙飛び込む水の音"

Eliminar caracteres no deseados de una cadena #

El ejemplo [Comprobación de caracteres no deseados en una cadena] (https://www.wikiod.com/es/guava/instrumentos-de-cuerda#Comprobación de una cadena en busca de caracteres no deseados), describe cómo probar y rechazar cadenas que no cumplan con ciertos criterios. Obviamente, no siempre es posible rechazar la entrada por completo y, a veces, solo tienes que conformarte con lo que recibes. En estos casos, un desarrollador cauteloso intentará desinfectar las cadenas proporcionadas para eliminar cualquier carácter que pueda dificultar el procesamiento posterior.

Para eliminar, recortar y reemplazar caracteres no deseados, el arma elegida será nuevamente la clase ‘CharMatcher’ de Guava.

Eliminación de caracteres #

Los dos métodos CharMatcher de interés en esta sección son:

  • String retainFrom (secuencia CharSequence) Devuelve una cadena que contiene todos los caracteres que coincidieron con la instancia de CharMatcher.

  • String removeFrom(secuencia CharSequence) Devuelve una cadena que contiene todos los caracteres que no coincidieron con la instancia de CharMatcher.

Como ejemplo, usaremos CharMatcher.digit(), una instancia predefinida de CharMatcher que, como era de esperar, solo coincide con dígitos.

String rock = "1, 2, 3 o'clock, 4 o'clock rock!";

CharMatcher.digit().retainFrom(rock); // "1234"
CharMatcher.digit().removeFrom(rock); // ", , o'clock, o'clock rock!"
CharMatcher.digit().negate().removeFrom(rock); // "1234"

La última línea de este ejemplo ilustra que removeFrom es en realidad la operación inversa de retainFrom. Invocar retainFrom en un CharMatcher tiene el mismo efecto que invocar removeFrom en una versión negada de ese CharMatcher.

Recortar caracteres iniciales y finales #

La eliminación de caracteres iniciales y finales es una operación muy común, que se usa con mayor frecuencia para recortar los espacios en blanco de las cadenas. CharMatcher de Guava ofrece estos métodos de recorte:

  • String trimLeadingFrom (secuencia CharSequence) Elimina todos los caracteres iniciales que coincidan con la instancia de CharMatcher.

  • String trimTrailingFrom (secuencia CharSequence) Elimina todos los caracteres finales que coinciden con la instancia de CharMatcher.

  • String trimFrom (secuencia CharSequence) Elimina todos los caracteres iniciales y finales que coincidan con la instancia de CharMatcher.

Cuando se usa con CharMatcher.whitespace(), estos métodos se encargarán de manera efectiva de todas sus necesidades de recorte de espacios en blanco:

CharMatcher.whitespace().trimFrom("   Too much space   "); // returns "Too much space"

Sustitución de personajes #

A menudo, las aplicaciones reemplazarán los caracteres que no están permitidos en una determinada situación con un carácter de marcador de posición. Para reemplazar caracteres en una cadena, la API de CharMatcher proporciona los siguientes métodos:

  • String replaceFrom (secuencia CharSequence, reemplazo de caracteres) Reemplaza todas las apariciones de caracteres que coinciden con la instancia de CharMatcher con el carácter de reemplazo provisto.

  • String replaceFrom (secuencia CharSequence, reemplazo CharSequence) Reemplaza todas las apariciones de caracteres que coinciden con la instancia de CharMatcher con la secuencia de caracteres de reemplazo proporcionada (cadena).

  • String colapsoFrom (secuencia CharSequence, reemplazo de caracteres) Reemplaza grupos de caracteres consecutivos que coinciden con la instancia de CharMatcher con una sola instancia del carácter de reemplazo provisto.

  • String trimAndCollapseFrom (secuencia CharSequence, reemplazo de caracteres) Se comporta igual que collapseFrom, pero los grupos coincidentes al principio y al final se eliminan en lugar de reemplazarse.

Veamos un ejemplo que demuestra cómo difiere el comportamiento de estos métodos. Digamos que estamos creando una aplicación que permite al usuario especificar nombres de archivos de salida. Para desinfectar la entrada proporcionada por el usuario, creamos una instancia de CharMatcher que es una combinación del espacio en blanco predefinido CharMatcher y un CharMatcher personalizado que especifica un conjunto de caracteres que preferimos evitar en nuestros nombres de archivo.

CharMatcher illegal = CharMatcher.whitespace().or(CharMatcher.anyOf("<>:|?*\"/\\"));

Ahora, si invocamos los métodos de reemplazo discutidos de la siguiente manera en un nombre de archivo que necesita una limpieza urgente:

String filename = "<A::12> first draft???";

System.out.println(illegal.replaceFrom(filename, '_'));
System.out.println(illegal.collapseFrom(filename, '_'));
System.out.println(illegal.trimAndCollapseFrom(filename, '_'));

Veremos el resultado a continuación en nuestra consola.

_A__12___first_draft___
_A_12_first_draft_
A_12_first_draft

Dividir una cadena en una lista #

Para dividir cadenas, Guava presenta la clase Splitter.

¿Por qué no usar las capacidades de división de Java? #

Como regla general, Guava no duplica la funcionalidad que está disponible en Java. Entonces, ¿por qué necesitamos una clase Splitter adicional? ¿Los métodos de división en la clase String de Java no nos proporcionan toda la mecánica de división de cadenas que necesitaremos?

La forma más fácil de responder a esa pregunta es con un par de ejemplos. En primer lugar, nos ocuparemos del siguiente dúo de pistoleros:

String gunslingers = "Wyatt Earp+Doc Holliday";

Para tratar de separar al legendario representante de la ley y su amigo dentista, podríamos intentar lo siguiente:

String[] result = gunslingers.split("+"); // wrong

En tiempo de ejecución, sin embargo, nos enfrentamos a la siguiente excepción:

Exception in thread "main" java.util.regex.PatternSyntaxException:
Dangling meta character '+' near index 0

Después de una palmada en la cara involuntaria, recordamos rápidamente que el método de división de String toma una expresión regular como argumento, y que el carácter + se usa como cuantificador en las expresiones regulares. Entonces, la solución es escapar del carácter + o incluirlo en una clase de caracteres.

String[] result = gunslingers.split("\\+");
String[] result = gunslingers.split("[+]");

Habiendo resuelto con éxito ese problema, pasamos a los tres mosqueteros.

String musketeers = ",Porthos , Athos ,Aramis,";

La coma no tiene un significado especial en las expresiones regulares, así que vamos a contar los mosqueteros aplicando el método String.split() y obteniendo la longitud de la matriz resultante.

System.out.println(musketeers.split(",").length);

Lo que arroja el siguiente resultado en la consola:

4

¿Cuatro? Dado el hecho de que la cadena contiene una coma inicial y otra final, un resultado de cinco habría estado dentro del ámbito de las expectativas normales, pero ¿cuatro? Resulta que el comportamiento del método split de Java es conservar el encabezado, pero descartar las cadenas vacías finales, por lo que el contenido real de la matriz es ["", "Porthos", "Athos", "Aramis"] .

Como no necesitamos cadenas vacías, ni iniciales ni finales, filtrémoslas con un bucle:

for (String musketeer : musketeers.split(",")) {
    if (!musketeer.isEmpty()) {
        System.out.println(musketeer);
    }
}

Esto nos da la siguiente salida:

Porthos 
 Athos
Aramis

Como puede ver en el resultado anterior, los espacios adicionales antes y después de los separadores de coma se han conservado en el resultado. Para evitar eso, podemos recortar los espacios innecesarios, lo que finalmente producirá el resultado deseado:

for (String musketeer : musketeers.split(",")) {
    if(!musketeer.isEmpty()) {
        System.out.println(musketeer.trim());
    }
}

(Alternativamente, también podríamos adaptar la expresión regular para incluir espacios en blanco alrededor de los separadores de coma. Sin embargo, tenga en cuenta que los espacios iniciales antes de la primera entrada o los espacios finales después de la última entrada aún se conservarán).

Después de leer los ejemplos anteriores, no podemos evitar concluir que dividir cadenas con Java es un poco molesto en el mejor de los casos.

Partiendo hilos con guayaba #

La mejor manera de demostrar cómo Guava convierte la división de cadenas en una experiencia relativamente sencilla es tratar las mismas dos cadenas de nuevo, pero esta vez usando la clase Splitter de Guava.

List<String> gunslingers = Splitter.on('+')
        .splitToList("Wyatt Earp+Doc Holliday");

List<String> musketeers = Splitter.on(",")
        .omitEmptyStrings()
        .trimResults()
        .splitToList(",Porthos , Athos ,Aramis,");

Como puede ver en el código anterior, Splitter expone una API fluida y le permite crear instancias a través de una serie de métodos estáticos de fábrica:

  • Divisor estático activado (separador de caracteres) Le permite especificar el separador como un carácter.

  • Divisor estático activado (separador de cadenas) Le permite especificar el separador como una cadena.

  • Divisor estático activado (CharMatcher separatorMatcher) Le permite especificar el separador como Guava CharMatcher.

  • Divisor estático activado (Separador de patrónPatrón) Le permite especificar el separador como una expresión regular de Java Pattern.

  • Divisor estático en Patrón (Separador de cadenas Patrón) Le permite especificar el separador como una cadena de expresión regular.

Además de estos métodos de fábrica basados ​​en separadores, también hay un método static Splitter fixedLength(int length) para crear instancias de Splitter que dividen cadenas en fragmentos de la longitud especificada.

Después de crear la instancia Splitter, se pueden aplicar una serie de modificadores:

  • Divisor omitEmptyStrings() Indica al Splitter que excluya las cadenas vacías de los resultados.

  • Resultados de recorte de división () Indica al Splitter que recorte los resultados utilizando el espacio en blanco predefinido CharMatcher.

  • Resultados de recorte del divisor (recortador de CharMatcher) Indica al Splitter que recorte los resultados utilizando el CharMatcher especificado.

Después de crear (y opcionalmente modificar) un Splitter, se puede invocar en una secuencia de caracteres invocando su método split, que devolverá un objeto de tipo Iterable<String>, o su método splitToList, que devolverá un objeto (inmutable) de tipo List<String>.

Quizás se pregunte en qué casos sería beneficioso utilizar el método split (que devuelve un Iterable) en lugar del método splitToList (que devuelve el tipo List más utilizado). La respuesta corta a eso es: probablemente quieras usar el método split solo para procesar cadenas muy grandes. La respuesta un poco más larga es que debido a que el método split devuelve un Iterable, las operaciones de división pueden evaluarse de forma perezosa (en el momento de la iteración), eliminando así la necesidad de mantener el resultado completo de la operación de división en la memoria.
