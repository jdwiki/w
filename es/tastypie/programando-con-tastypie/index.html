Tastypie es un marco de API de servicio web para Django. Proporciona una abstracción conveniente, poderosa y altamente personalizable para crear interfaces de estilo REST. Tastypie facilita la exposición de sus modelos, pero le brinda control total sobre lo que expone, lo que le permite abstraer la base de datos tanto como sea necesario. Tastypie también facilita la integración con fuentes de datos que no son ORM. Por lo tanto, Tastypie se puede usar con bases de datos ORM y no ORM.

Inicio rápido #

  1. Agrega sabrosopie a INSTALLED_APPS.

  2. Cree un directorio api en su aplicación con un init.py simple.

  3. Cree un archivo <my_app>/api/resources.py y coloque lo siguiente en él:

    from tastypie.resources import ModelResource from my_app.models import MyModel

    class MyModelResource(ModelResource): class Meta: queryset = MyModel.objects.all() allowed_methods = [‘get’]

En su URLconf raíz, agregue el siguiente código (alrededor de donde podría estar el código de administrador):

from django.conf.urls import url, include
from tastypie.api import Api
from my_app.api.resources import MyModelResource

v1_api = Api(api_name='v1')
v1_api.register(MyModelResource())

urlpatterns = [
  # ...more URLconf bits here...
  # Then add:
  url(r'^api/', include(v1_api.urls)),
]

Instalación Tastypie #

Tastypie se puede instalar con la administración de paquetes de python, es decir, pip o podemos verificar directamente el código de Github

  1. pip instalar django-tastypie
  2. Pago desde Github

¿Por qué Tastypie? #

Existen otros marcos de API para Django. Necesitas evaluar las opciones disponibles y decidir por ti mismo. Dicho esto, aquí hay algunas razones comunes para un pastel sabroso.

  • Necesita una API que sea RESTful y use bien HTTP.
  • Quieres apoyar relaciones profundas.
  • NO quieres tener que escribir tu propio serializador para hacer el salida derecha.
  • Desea un marco API que tenga poca magia, muy flexible y mapea bien al dominio del problema.
  • Quiere/necesita una serialización XML que se trate por igual a JSON (y YAML también está allí).

Para desarrollar API con Django, tenemos muchas opciones. Las opciones principales son Tastypie y Django Rest Framework(DRF).

¿Qué hace que un marco de API sea decente?

Estas características:

  1. paginación
  2. publicación de datos con validación
  3. Publicación de metadatos junto con conjuntos de consultas
  4. Descubrimiento de API
  5. manejo adecuado de la respuesta HTTP
  6. almacenamiento en caché
  7. serialización
  8. estrangulamiento
  9. permisos
  10. autenticación

Los marcos de API adecuados también necesitan:

  1. Muy buena cobertura de prueba de su código.

  2. Rendimiento decente

  3. Documentación

  4. Una comunidad activa para promover y apoyar el marco

Si toma estos factores, en este momento solo hay dos marcos de API que vale la pena usar (depende de las opiniones de los usuarios), django-tastypie y django-rest-framework.

¿Cuál es mejor? ¿Django-tastypie o Django-rest-framework?

  • Yo digo que son iguales. Simplemente no puedes equivocarte con ninguno de los dos. Los autores y las comunidades detrás de ambos están activos, el código es sólido y está probado. Y aquí están mis pensamientos específicos sobre ambos:*

Tastypie:

Ventajas:

  • Fácil de comenzar y proporciona funcionalidades básicas OOB (fuera De la caja)
  • La mayoría de las veces no estarás lidiando con conceptos avanzados de Django como CBV, formularios, etc.
  • ¡Código más legible y menos magia!
  • Si sus modelos NO son ORM, hágalo.

Desventajas:

  • No sigue estrictamente Django idiomático (cuidado con python y Las filosofías de Django son bastante diferentes)
  • Probablemente sea un poco difícil personalizar las API una vez que seas grande
  • Sin autenticación O

FRD:

Ventajas:

  • Seguir django idiomático. (Si conoce Django de adentro hacia afuera, y muy cómodo con CBV, Formularios, etc. sin ninguna duda, adelante)

  • Proporciona funcionalidad REST lista para usar usando ModelViewSets. A al mismo tiempo, proporciona un mayor control para la personalización mediante CustomSerializer, APIView, GenericViews, etc.

  • Mejor autenticación.

  • Más fácil de escribir clases de permisos personalizados.

  • Funciona muy bien y, lo que es más importante, muy fácil de hacer que funcione con 3rd bibliotecas de partes y OAuth.

  • DJANGO-REST-AUTH vale la pena mencionar BIBLIOTECA para Autenticación/Autenticación social/Registro.

Desventajas:

  • Si no conoces Django muy bien, no vayas por esto.
  • ¡Magia! Algún tiempo muy difícil de entender la magia.
  • Porque se ha escrito sobre el CBV de django, que a su vez son de naturaleza bastante compleja.
  • Tiene una curva de aprendizaje pronunciada.

Primeros pasos con Tastypie #

Tastypie es una aplicación reutilizable (es decir, se basa solo en su propio código y se enfoca en proporcionar solo una API de estilo REST) ​​y es adecuada para proporcionar una API a cualquier aplicación sin tener que modificar las fuentes de esa aplicación.

No todas las necesidades son las mismas, por lo que Tastypie hace todo lo posible para proporcionar muchos ganchos para anular o ampliar su funcionamiento.

A modo de ejemplo, agregaremos una API a un blog simple aplicación.

La única configuración obligatoria es agregar ‘tastypie’ a tus INSTALLED_APPS. Esto no es estrictamente necesario, ya que Tastypie solo tiene dos modelos no obligatorios, pero puede facilitar el uso.

Aquí está myapp/models.py:

from tastypie.utils.timezone import now
from django.contrib.auth.models import User
from django.db import models
from django.utils.text import slugify


class Entry(models.Model):
    user = models.ForeignKey(User)
    pub_date = models.DateTimeField(default=now)
    title = models.CharField(max_length=200)
    slug = models.SlugField(null=True, blank=True)
    body = models.TextField()

Creación de recursos

La arquitectura de estilo REST habla de recursos, por lo que, como era de esperar, la integración con Tastypie implica la creación de clases de recursos. Para nuestra aplicación simple, crearemos un archivo para estos en myapp/api.py, aunque pueden vivir en cualquier parte de su aplicación:

miaplicación/api.py #

from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'

En REST todo es un recurso, significa que los objetos se describen como recursos. Entonces, la creación de una API REST de Tastypie implica la creación de una clase de recursos. Para crear un recurso, necesitamos crear una subclase de la clase ModelResource. Esta clase EntryResource verificará todos los campos no relacionales en el modelo Entry y creará sus propios campos Api. Esto funciona igual que ModelForm de Django Forms.

Conexión del recurso: Después de crear el recurso, debemos decirle a Django que se crea un recurso conectando nuestro EntryResource a la URL.

urls.py #

from django.conf.urls import url, include
from myapp.api import EntryResource

entry_resource = EntryResource()

urlpatterns = [
    url(r'^blog/', include('myapp.urls')),
    url(r'^api/', include(entry_resource.urls)),
]

Ahora, si revisamos localhost:8000/api/entry/, obtendremos la respuesta de la API.

(El resource_name dentro de la clase Meta es opcional. Si no se proporciona, se genera automáticamente a partir del nombre de la clase, eliminando cualquier instancia de Resource y minúsculas en la cadena. Por lo tanto, EntryResource se convertiría simplemente en una entrada. Por eso en la URL especificamos ../entrada/ )

Autorización Tastypie #

Hasta ahora hemos intentado obtener la solicitud. (Si necesita probar otra solicitud HTTP, use algunas herramientas de prueba de API como curl o postman)

Si intenta enviar un POST/PUT/DELETE al recurso, obtendrá errores “401 no autorizados”. Por seguridad, Tastypie se envía con la clase de autorización establecida en ReadOnlyAuthorization. Esto hace que sea seguro exponerlo en la web, pero nos impide hacer POST/PUT/DELETE. Habilitemos esos.

miaplicación/api.py #

from tastypie.authorization import Authorization
from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'
        authorization = Authorization()
