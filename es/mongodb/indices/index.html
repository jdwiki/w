Sintaxis #

  • db.collection.createIndex({ <campo de cadena> : <1|-1 orden> [, <campo de cadena> : <1|-1 orden>] });

Impacto en el rendimiento: tenga en cuenta que los índices mejoran el rendimiento de lectura, pero pueden tener un impacto negativo en el rendimiento de escritura, ya que insertar un documento requiere actualizar todos los índices.

Soltar/eliminar un índice #

Si se conoce el nombre del índice,

db.collection.dropIndex('name_of_index');

Si no se conoce el nombre del índice,

db.collection.dropIndex( { 'name_of_field' : -1 } );

Obtener índices de una colección #

 db.collection.getIndexes();

Producción

[
    {
        "v" : 1,
        "key" : {
            "_id" : 1
        },
        "name" : "_id_",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : 1
        },
        "name" : "cr_dr_1",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : -1
        },
        "name" : "index on cr_dr desc",
        "ns" : "documentation_db.transactions"
    }
]

Índices dispersos e índices parciales #

Índices escasos:

Estos pueden ser particularmente útiles para campos que son opcionales pero que también deben ser únicos.

{ "_id" : "john@example.com", "nickname" : "Johnnie" }
{ "_id" : "jane@example.com" }
{ "_id" : "julia@example.com", "nickname" : "Jules"}
{ "_id" : "jack@example.com" }

Dado que dos entradas no tienen un “apodo” especificado y la indexación tratará los campos no especificados como nulos, la creación del índice fallaría con 2 documentos que tienen “nulo”, por lo que:

db.scores.createIndex( { nickname: 1 } , { unique: true, sparse: true } )

te permitirá tener apodos ’nulos'.

Los índices dispersos son más compactos ya que omiten/ignoran documentos que no especifican ese campo. Entonces, si tiene una colección donde solo menos del 10% de los documentos especifican este campo, puede crear índices mucho más pequeños, haciendo un mejor uso de la memoria limitada si desea realizar consultas como:

db.scores.find({'nickname': 'Johnnie'})

Índices parciales:

Los índices parciales representan un superconjunto de la funcionalidad ofrecida por índices dispersos y debe preferirse a los índices dispersos. (Nuevo en la versión 3.2)

Los índices parciales determinan las entradas de índice según el filtro especificado.

db.restaurants.createIndex(
  { cuisine: 1 },
  { partialFilterExpression: { rating: { $gt: 5 } } }
)

Si la “puntuación” es mayor que 5, se indexará “cocina”. Sí, podemos especificar que una propiedad se indexe en función del valor de otras propiedades también.

Diferencia entre índices dispersos y parciales:

Los índices dispersos seleccionan documentos para indexar basándose únicamente en la existencia del campo indexado, o para índices compuestos, la existencia de los campos indexados.

Los índices parciales determinan las entradas de índice según el filtro especificado. El filtro puede incluir campos que no sean las claves de índice y puede especificar condiciones que no sean solo una verificación de existencia.

Aún así, un índice parcial puede implementar el mismo comportamiento que un índice disperso

P.ej:

db.contacts.createIndex(
   { name: 1 },
   { partialFilterExpression: { name: { $exists: true } } }
)

Nota: Tanto la opción partialFilterExpression como la opción sparse no se puede especificar al mismo tiempo.

Compuesto #

db.people.createIndex({name: 1, age: -1})

Esto crea un índice en varios campos, en este caso en los campos nombre y edad. Será ascendente en ’nombre’ y descendente en ’edad'.

En este tipo de índice, el orden de clasificación es relevante, ya que determinará si el índice puede admitir una operación de clasificación o no. La ordenación inversa se admite en cualquier prefijo de un índice compuesto, siempre que la ordenación sea en la dirección de ordenación inversa para todas las claves de la ordenación. De lo contrario, la clasificación de los índices compuestos debe coincidir con el orden del índice.

El orden de los campos también es importante, en este caso el índice se ordenará primero por nombre, y dentro de cada valor de nombre, se ordenará por los valores del campo edad. Esto permite que el índice sea utilizado por consultas en el campo ’nombre’, o en ’nombre’ y ’edad’, pero no solo en ’edad'.

Índice único #

db.collection.createIndex( { "user_id": 1 }, { unique: true } )

hacer cumplir la unicidad en el índice definido (ya sea único o compuesto). La creación del índice fallará si la colección ya contiene valores duplicados; la indexación también fallará con múltiples entradas que no tengan el campo (ya que todas se indexarán con el valor null) a menos que se especifique sparse: true.

Fundamentos de la creación de índices #

Consulte la siguiente colección de transacciones.

> db.transactions.insert({ cr_dr : "D", amount : 100, fee : 2});
> db.transactions.insert({ cr_dr : "C", amount : 100, fee : 2});
> db.transactions.insert({ cr_dr : "C", amount : 10,  fee : 2});
> db.transactions.insert({ cr_dr : "D", amount : 100, fee : 4});
> db.transactions.insert({ cr_dr : "D", amount : 10,  fee : 2});
> db.transactions.insert({ cr_dr : "C", amount : 10,  fee : 4});
> db.transactions.insert({ cr_dr : "D", amount : 100, fee : 2});

Las funciones getIndexes() mostrarán todos los índices disponibles para una colección.

db.transactions.getIndexes();

Veamos el resultado de la declaración anterior.

[
    {
        "v" : 1,
        "key" : {
            "_id" : 1
        },
        "name" : "_id_",
        "ns" : "documentation_db.transactions"
    }
]

Ya existe un índice para la recopilación de transacciones. Esto se debe a que MongoDB crea un único índice en el campo _id durante la creación de una colección. El índice _id evita que los clientes inserten dos documentos con el mismo valor para el campo _id. No puede soltar este índice en el campo _id.

Ahora agreguemos un índice para el campo cr_dr;

db.transactions.createIndex({ cr_dr : 1 });

El resultado de la ejecución del índice es el siguiente.

{
    "createdCollectionAutomatically" : false,
    "numIndexesBefore" : 1,
    "numIndexesAfter" : 2,
    "ok" : 1
}

La createdCollection indica automáticamente si la operación creada una colección. Si una colección no existe, MongoDB crea la colección como parte de la operación de indexación.

Ejecute db.transactions.getIndexes(); de nuevo.

[
    {
        "v" : 1,
        "key" : {
            "_id" : 1
        },
        "name" : "_id_",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : 1
        },
        "name" : "cr_dr_1",
        "ns" : "documentation_db.transactions"
    }
]

Ahora ves que la colección de transacciones tiene dos índices. Índice predeterminado _id y cr_dr_1 que creamos. El nombre es asignado por MongoDB. Puede establecer su propio nombre como se muestra a continuación.

db.transactions.createIndex({ cr_dr : -1 },{name : "index on cr_dr desc"})

Ahora db.transactions.getIndexes(); le dará tres índices.

[
    {
        "v" : 1,
        "key" : {
            "_id" : 1
        },
        "name" : "_id_",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : 1
        },
        "name" : "cr_dr_1",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : -1
        },
        "name" : "index on cr_dr desc",
        "ns" : "documentation_db.transactions"
    }
]

Al crear el índice { cr_dr: -1 } 1 significa que el índice estará en orden ascendente y -1 para el orden descendente.

Índices hash #

Los índices también se pueden definir como hash. Esto es más eficaz en consultas de igualdad, pero no es eficiente para consultas de rango; sin embargo, puede definir tanto índices hash como ascendentes/descendentes en el mismo campo.

> db.transactions.createIndex({ cr_dr : "hashed" });    

> db.transactions.getIndexes(
[
    {
        "v" : 1,
        "key" : {
            "_id" : 1
        },
        "name" : "_id_",
        "ns" : "documentation_db.transactions"
    },
    {
        "v" : 1,
        "key" : {
            "cr_dr" : "hashed"
        },
        "name" : "cr_dr_hashed",
        "ns" : "documentation_db.transactions"
    }
]
