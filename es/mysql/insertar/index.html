Sintaxis #

  1. INSERTAR [BAJA_PRIORIDAD | RETRASADO | ALTA_PRIORIDAD] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] {VALUES | VALUE} ({expr | DEFAULT},…),(…),… [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  2. INSERTAR [BAJA_PRIORIDAD | RETRASADO | ALTA_PRIORIDAD] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] SET col_name={expr | DEFAULT}, … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  3. INSERTAR [PRIORIDAD_BAJA | ALTA_PRIORIDAD] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  4. Una expresión expr puede hacer referencia a cualquier columna que se haya establecido anteriormente en una lista de valores. Por ejemplo, puede hacer esto porque el valor de col2 se refiere a col1, que se ha asignado previamente: INSERTAR EN tbl_name (col1, col2) VALORES (15, col1 * 2);

  5. Las sentencias INSERT que utilizan la sintaxis VALUES pueden insertar varias filas. Para ello, incluya varias listas de valores de columna, cada una entre paréntesis y separada por comas. Ejemplo: INSERTAR EN tbl_name (a,b,c) VALORES (1,2,3), (4,5,6), (7,8,9);

  6. La lista de valores para cada fila debe estar entre paréntesis. La siguiente declaración es ilegal porque la cantidad de valores en la lista no coincide con la cantidad de nombres de columna: INSERTAR EN tbl_name (a,b,c) VALORES (1,2,3,4,5,6,7,8,9);

  7. INSERTAR … SELECCIONAR Sintaxis INSERTAR [PRIORIDAD_BAJA | ALTA_PRIORIDAD] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr, … ]

  8. Con INSERTAR… SELECCIONAR, puede insertar rápidamente muchas filas en una tabla desde una o varias tablas. Por ejemplo: INSERTAR EN tbl_temp2 (fld_id) SELECCIONE tbl_temp1.fld_order_id DESDE tbl_temp1 DONDE tbl_temp1.fld_order_id > 100;

Sintaxis oficial de INSERT

INSERTAR, EN ACTUALIZACIÓN DE CLAVE DUPLICADA #

INSERT INTO `table_name` 
    (`index_field`, `other_field_1`, `other_field_2`) 
    VALUES 
    ('index_value', 'insert_value', 'other_value') 
    ON DUPLICATE KEY UPDATE 
        `other_field_1` = 'update_value',
        `other_field_2` = VALUES(`other_field_2`);

Esto ‘INSERTARÁ’ en ’nombre_tabla’ los valores especificados, pero si la clave única ya existe, actualizará ‘otro_campo_1’ para tener un nuevo valor. A veces, cuando se actualiza una clave duplicada, resulta útil usar VALUES() para acceder al valor original que se pasó a INSERT en lugar de establecer el valor directamente. De esta manera, puede establecer diferentes valores usando INSERTAR y ACTUALIZAR. Vea el ejemplo anterior donde otro_campo_1 se establece en insertar_valor en INSERTAR o en actualizar_valor en ACTUALIZAR mientras que otro_campo_2 siempre se establece en otro_valor.

Crucial para que la inserción en actualización de clave duplicada (IODKU) funcione es el esquema que contiene una clave única que señalará un conflicto duplicado. Esta clave única puede ser una clave principal o no. Puede ser una clave única en una sola columna o una columna múltiple (clave compuesta).

Insertar varias filas #

INSERT INTO `my_table` (`field_1`, `field_2`) VALUES 
    ('data_1', 'data_2'),
    ('data_1', 'data_3'),
    ('data_4', 'data_5');

Esta es una manera fácil de agregar varias filas a la vez con una instrucción INSERT.

Este tipo de inserción ‘por lotes’ es mucho más rápida que insertar filas una por una. Por lo general, insertar 100 filas en una sola inserción de lote de esta manera es 10 veces más rápido que insertarlas todas individualmente.

Ignorar las filas existentes #

Al importar grandes conjuntos de datos, puede ser preferible, en determinadas circunstancias, omitir filas que normalmente harían que la consulta fallara debido a una restricción de columna, p. Claves primarias duplicadas. Esto se puede hacer usando INSERT IGNORE.

Considere la siguiente base de datos de ejemplo:

SELECT * FROM `people`;
--- Produces:
+----+------+
| id | name |
+----+------+
|  1 | john |
|  2 | anna |
+----+------+


INSERT IGNORE INTO `people` (`id`, `name`) VALUES
    ('2', 'anna'), --- Without the IGNORE keyword, this record would produce an error
    ('3', 'mike');

SELECT * FROM `people`;
--- Produces:
+----+--------+
| id |  name  |
+----+--------+
|  1 |  john  |
|  2 |  anna  |
|  3 |  mike  |
+----+--------+

Lo importante a recordar es que INSERT IGNORE también omitirá silenciosamente otros errores, esto es lo que dice la documentación oficial de Mysql:

Las conversiones de datos que dispararían errores abortan la declaración si IGNORE no es > especificado. Con IGNORE, los valores inválidos se ajustan a los valores más cercanos y > se insertan; se producen advertencias pero la declaración no aborta.

Nota:- La siguiente sección se agrega para completar, pero no se considera una buena práctica (esto fallaría, por ejemplo, si se agregara otra columna a la tabla).

Si especifica el valor de la columna correspondiente para todas las columnas de la tabla, puede ignorar la lista de columnas en la instrucción INSERT de la siguiente manera:

INSERT INTO `my_table` VALUES 
        ('data_1', 'data_2'),
        ('data_1', 'data_3'),
        ('data_4', 'data_5');

Inserto básico #

INSERT INTO `table_name` (`field_one`, `field_two`) VALUES ('value_one', 'value_two');

En este ejemplo trivial, nombre_tabla es donde se agregarán los datos, campo_uno y campo_dos son campos contra los que establecer datos, y valor_uno y valor_dos son los datos para hacer contra campo_uno y campo_dos respectivamente.

Es una buena práctica enumerar los campos en los que está insertando datos dentro de su código, ya que si la tabla cambia y se agregan nuevas columnas, su inserción se rompería si no estuvieran allí.

INSERTAR con AUTO_INCREMENT + LAST_INSERT_ID() #

Cuando una tabla tiene un AUTO_INCREMENT PRIMARY KEY, normalmente no se inserta en esa columna. En su lugar, especifique todas las demás columnas y luego pregunte cuál era la nueva identificación.

CREATE TABLE t (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    this ...,
    that ...,
    PRIMARY KEY(id) );

INSERT INTO t (this, that) VALUES (..., ...);
SELECT LAST_INSERT_ID() INTO @id;
INSERT INTO another_table (..., t_id, ...) VALUES (..., @id, ...);

Tenga en cuenta que LAST_INSERT_ID() está vinculado a la sesión, por lo que incluso si se insertan varias conexiones en la misma tabla, cada una obtiene su propia identificación.

La API de su cliente probablemente tenga una forma alternativa de obtener LAST_INSERT_ID() sin realizar realmente un SELECT y devolver el valor al cliente en lugar de dejarlo en una @variable dentro de MySQL. Eso suele ser preferible.

Más largo, más detallado, ejemplo

El uso “normal” de IODKU es activar una “clave duplicada” basada en alguna clave ÚNICA, no en la CLAVE PRINCIPAL DE INCREMENTO AUTOMÁTICO. Lo siguiente demuestra tal. Tenga en cuenta que no proporciona el id en el INSERTAR.

Configuración de ejemplos a seguir:

CREATE TABLE iodku (
    id INT AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    misc INT NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
) ENGINE=InnoDB;

INSERT INTO iodku (name, misc)
    VALUES
    ('Leslie', 123),
    ('Sally', 456);
Query OK, 2 rows affected (0.00 sec)
Records: 2  Duplicates: 0  Warnings: 0
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  |  456 |
+----+--------+------+

El caso de IODKU realizando una “actualización” y LAST_INSERT_ID() recuperando el id relevante:

INSERT INTO iodku (name, misc)
    VALUES
    ('Sally', 3333)            -- should update
    ON DUPLICATE KEY UPDATE    -- `name` will trigger "duplicate key"
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();       -- picking up existing value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                2 |
+------------------+

El caso en el que IODKU realiza una “inserción” y LAST_INSERT_ID() recupera el nuevo id:

INSERT INTO iodku (name, misc)
    VALUES
    ('Dana', 789)          -- Should insert
    ON DUPLICATE KEY UPDATE
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();   -- picking up new value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                3 |
+------------------+

Contenido de la tabla resultante:

SELECT * FROM iodku;
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  | 3333 |  -- IODKU changed this
|  3 | Dana   |  789 |  -- IODKU added this
+----+--------+------+

INSERT SELECT (Insertar datos de otra tabla) #

Esta es la forma básica de insertar datos de otra tabla con la instrucción SELECT.

INSERT INTO `tableA` (`field_one`, `field_two`) 
   SELECT `tableB`.`field_one`, `tableB`.`field_two` 
   FROM `tableB` 
   WHERE `tableB`.clmn <> 'someValue'
   ORDER BY `tableB`.`sorting_clmn`;

Puede SELECT * FROM, pero luego tableA y tableB deben tener un recuento de columnas coincidente y los tipos de datos correspondientes.

Las columnas con AUTO_INCREMENT se tratan como en la cláusula INSERT con VALUES.

Esta sintaxis facilita el llenado de tablas (temporales) con datos de otras tablas, más aún cuando los datos se van a filtrar en la inserción.

ID de AUTO_INCREMENT perdidos #

Varias funciones de ‘inserción’ pueden “quemar” identificaciones. Aquí hay un ejemplo, usando InnoDB (otros motores pueden funcionar de manera diferente):

CREATE TABLE Burn (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
        ) ENGINE=InnoDB;

INSERT IGNORE INTO Burn (name) VALUES ('first'), ('second');
SELECT LAST_INSERT_ID();          -- 1
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('second');  -- dup 'IGNOREd', but id=3 is burned
SELECT LAST_INSERT_ID();          -- Still "1" -- can't trust in this situation
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('third');
SELECT LAST_INSERT_ID();           -- now "4"
SELECT * FROM Burn ORDER BY id;    -- note that id=3 was skipped over
  +----+--------+
  |  1 | first  |
  |  2 | second |
  |  4 | third  |    -- notice that id=3 has been 'burned'
  +----+--------+

Piénselo (más o menos) de esta manera: primero, la inserción busca cuántas filas podrían insertarse. Luego tome esa cantidad de valores del auto_incremento para esa tabla. Finalmente, inserte las filas, use las identificaciones según sea necesario y queme las sobras.

La única vez que los restos son recuperables es si el sistema se apaga y se reinicia. Al reiniciar, efectivamente se ejecuta MAX(id). Esto puede reutilizar las identificaciones que fueron quemadas o que fueron liberadas por DELETES de las identificaciones más altas.

Esencialmente, cualquier versión de INSERT (incluido REPLACE, que es DELETE + INSERT) puede grabar identificadores. En InnoDB, la variable global (¡no de sesión!) [innodb_autoinc_lock_mode] 1 puede usarse para controlar parte de lo que está sucediendo.

Al “normalizar” cadenas largas en un ‘id de INCREMENTO AUTOMÁTICO’, la quema puede ocurrir fácilmente. Esto podría llevar a que se desborde el tamaño del ‘INT’ que eligió.
