Error 1055 ONLY_FULL_GROUP_BY algo no está en la cláusula GROUP BY...

  • Mal uso de GROUP BY para devolver resultados impredecibles: Ley de Murphy

Recientemente, las nuevas versiones de los servidores MySQL han comenzado a generar errores 1055 para consultas que solían funcionar. Este tema explica esos errores.

El equipo de MySQL ha estado trabajando para retirar la extensión no estándar de GROUP BY, o al menos para que sea más difícil quemar a los desarrolladores que escriben consultas.

Durante mucho tiempo, MySQL ha contenido una notoria extensión no estándar de GROUP BY, que permite un comportamiento extraño en nombre de la eficiencia. Esta extensión ha permitido a innumerables desarrolladores de todo el mundo usar GROUP BY en el código de producción sin comprender completamente lo que estaban haciendo.

En particular, es una mala idea usar SELECT * en una consulta GROUP BY, porque una cláusula estándar GROUP BY requiere enumerar las columnas. Desafortunadamente, muchos desarrolladores han hecho eso.

Lee esto. https://dev.mysql.com/doc/refman/5.7/en/group-by-handling.html

El equipo de MySQL ha estado tratando de corregir esta falla sin estropear el código de producción. Agregaron un indicador sql_mode en 5.7.5 llamado ONLY_FULL_GROUP_BY para forzar el comportamiento estándar. En una versión reciente, activaron esa bandera de forma predeterminada. Cuando actualizó su MySQL local a 5.7.14, la bandera se activó y su código de producción, que dependía de la extensión anterior, dejó de funcionar.

Si recientemente comenzó a recibir errores 1055, ¿cuáles son sus opciones?

  1. corrija las consultas SQL ofensivas o haga que sus autores lo hagan.
  2. Vuelva a una versión de MySQL compatible lista para usar con el software de aplicación que utiliza.
  3. cambie el sql_mode de su servidor para deshacerse del modo ONLY_FULL_GROUP_BY recién configurado.

Puede cambiar el modo haciendo un comando SET.

SET  sql_mode = 'STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'

debería funcionar si lo hace justo después de que su aplicación se conecte a MySQL.

O puede encontrar el archivo init en su instalación de MySQL, ubicar la línea sql_mode= y cambiarla para omitir ONLY_FULL_GROUP_BY, y reiniciar su servidor.

Mal uso de GROUP BY para devolver resultados impredecibles: Ley de Murphy #

SELECT item.item_id, uses.category,   /* nonstandard */ 
       COUNT(*) number_of_uses 
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

mostrará las filas en una tabla llamada elemento y mostrará el recuento de filas relacionadas en una tabla llamada usos. También mostrará el valor de una columna llamada uses.category.

Esta consulta funciona en MySQL (antes de que apareciera el indicador ONLY_FULL_GROUP_BY). Utiliza la extensión no estándar de MySQL para GROUP BY.

Pero la consulta tiene un problema: si varias filas en la tabla usos coinciden con la condición ON en la cláusula JOIN, MySQL devuelve la columna categoría de solo una de esas filas. ¿Qué fila? El autor de la consulta y el usuario de la aplicación no lo saben de antemano. Formalmente hablando, es impredecible: MySQL puede devolver cualquier valor que quiera.

Impredecible es como aleatorio, con una diferencia significativa. Uno podría esperar que una elección aleatoria cambie de vez en cuando. Por lo tanto, si una elección fuera aleatoria, podría detectarla durante la depuración o la prueba. El resultado impredecible es peor: MySQL devuelve el mismo resultado cada vez que utiliza la consulta, hasta que no lo hace. A veces es una nueva versión del servidor MySQL la que provoca un resultado diferente. A veces es una mesa de cultivo la que causa el problema. Lo que puede salir mal, saldrá mal, y cuando no te lo esperas. Eso se llama Ley de Murphy.

El equipo de MySQL ha estado trabajando para que a los desarrolladores les resulte más difícil cometer este error. Las versiones más nuevas de MySQL en la secuencia 5.7 tienen un indicador sql_mode llamado ONLY_FULL_GROUP_BY. Cuando se establece ese indicador, el servidor MySQL devuelve el error 1055 y se niega a ejecutar este tipo de consulta.

Mal uso de GROUP BY con SELECT *, y cómo solucionarlo. #

A veces, una consulta tiene este aspecto, con un * en la cláusula SELECT.

 SELECT item.*,     /* nonstandard */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

Dicha consulta debe refactorizarse para cumplir con el estándar ONLY_FULL_GROUP_BY.

Para hacer esto, necesitamos una subconsulta que use GROUP BY correctamente para devolver el valor number_of_uses para cada item_id. Esta subconsulta es corta y dulce, porque solo necesita mirar la tabla usos.

                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id

Luego, podemos unir esa subconsulta con la tabla item.

 SELECT item.*, usecount.number_of_uses
   FROM item
   JOIN (
                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id
        ) usecount ON item.item_id = usecount.item_id

Esto permite que la cláusula GROUP BY sea simple y correcta, y también nos permite usar el especificador *.

Nota: sin embargo, los desarrolladores sabios evitan usar el especificador * en cualquier caso. Por lo general, es mejor enumerar las columnas que desea en una consulta.

Uso y mal uso de GROUP BY #

 SELECT item.item_id, item.name,     /* not SQL-92 */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

mostrará las filas en una tabla llamada elemento y mostrará el recuento de filas relacionadas en una tabla llamada usos. Esto funciona bien, pero desafortunadamente no es SQL-92 estándar.

¿Por que no? porque la cláusula SELECT (y la cláusula ORDER BY) en las consultas GROUP BY deben contener columnas que son

  1. mencionado en la cláusula GROUP BY, o
  2. Funciones agregadas como COUNT(), MIN() y similares.

La cláusula SELECT de este ejemplo menciona item.name, una columna que no cumple ninguno de esos criterios. MySQL 5.6 y anteriores rechazarán esta consulta si el modo SQL contiene ONLY_FULL_GROUP_BY.

Esta consulta de ejemplo se puede hacer para cumplir con el estándar SQL-92 cambiando la cláusula GROUP BY, así.

SELECT item.item_id, item.name, 
       COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id, item.name

El estándar SQL-99 posterior permite que una instrucción SELECT omita columnas no agregadas de la clave de grupo si el DBMS puede demostrar una dependencia funcional entre ellas y las columnas de clave de grupo. Debido a que item.name depende funcionalmente de item.item_id, el ejemplo inicial es SQL-99 válido. MySQL obtuvo un [probador de dependencia funcional] (https://dev.mysql.com/doc/refman/5.7/en/group-by-funcional-dependence.html) en la versión 5.7. El ejemplo original funciona bajo ONLY_FULL_GROUP_BY.

ALGÚN VALOR() #

 SELECT item.item_id, ANY_VALUE(uses.tag) tag,   
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

muestra las filas en una tabla llamada “elemento”, el recuento de filas relacionadas y uno de los valores en la tabla relacionada llamado “usos”.

Puedes pensar en esta función ANY_VALUE() como una extraña especie de función agregada. En lugar de devolver un recuento, una suma o un máximo, le indica al servidor MySQL que elija, arbitrariamente, un valor del grupo en cuestión. Es una forma de evitar el error 1055.

Tenga cuidado al usar ANY_VALUE() en consultas en aplicaciones de producción.

Realmente debería llamarse SORPRESA_ME(). Devuelve el valor de alguna fila del grupo GROUP BY. La fila que devuelve es indeterminada. Eso significa que depende completamente del servidor MySQL. Formalmente, devuelve un valor impredecible.

El servidor no elige un valor aleatorio, es peor que eso. Devuelve el mismo valor cada vez que ejecuta la consulta, hasta que deja de hacerlo. Puede cambiar, o no, cuando una tabla crece o se reduce, o cuando el servidor tiene más o menos RAM, o cuando cambia la versión del servidor, o cuando Marte está retrógrado (lo que sea que eso signifique), o sin ningún motivo.
