SELECT se usa para recuperar filas seleccionadas de una o más tablas.

Sintaxis #

  • SELECCIONE DISTINCT [expresiones] FROM TableName [WHERE condition]; ///Selección sencilla

  • SELECT DISTINCT(a), b… es lo mismo que SELECT DISTINCT a, b…

  • SELECCIONAR [ TODO | DISTINTO | DISTINTO ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ] [ SQL_SMALL_RESULT | SQL_BIG_RESULT ] [ SQL_BUFFER_RESULT ] [ SQL_CACHE | SQL_NO_CACHE ] [ SQL_CALC_FOUND_ROWS ] expresiones DESDE tablas [DONDE condiciones] [GROUP BY expresiones] [TENIENDO condición] [ORDENAR POR expresión [ ASC | DESC ]] [LIMIT [offset_value] number_rows | LÍMITE número_filas DESPLAZAMIENTO valor_desplazamiento] [PROCEDIMIENTO nombre_procedimiento] [ENTRO [ OUTFILE ’nombre_archivo’ opciones | DUMPFILE ‘file_name’ | @variable1, @variable2, … @variable_n] [PARA ACTUALIZAR | BLOQUEO EN MODO COMPARTIR]; ///Sintaxis de selección completa

Para obtener más información sobre la instrucción SELECT de MySQL, consulte MySQL Docs.

SELECCIONAR con DISTINTO #

La cláusula DISTINCT después de SELECT elimina las filas duplicadas del conjunto de resultados.

CREATE TABLE `car`
(   `car_id` INT UNSIGNED NOT NULL PRIMARY KEY, 
    `name` VARCHAR(20), 
    `price` DECIMAL(8,2)
);

INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (1, 'Audi A1', '20000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (2, 'Audi A1', '15000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (3, 'Audi A2', '40000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (4, 'Audi A2', '40000');

SELECT DISTINCT `name`, `price` FROM CAR;
+---------+----------+
| name    | price    |
+---------+----------+
| Audi A1 | 20000.00 |
| Audi A1 | 15000.00 |
| Audi A2 | 40000.00 |
+---------+----------+

DISTINCT funciona en todas las columnas para entregar los resultados, no en columnas individuales. Este último es a menudo un concepto erróneo de los nuevos desarrolladores de SQL. En resumen, lo que importa es la distinción a nivel de fila del conjunto de resultados, no la distinción a nivel de columna. Para visualizar esto, mire “Audi A1” en el conjunto de resultados anterior.

Para versiones posteriores de MySQL, DISTINCT tiene implicaciones con su uso junto con ORDER BY. La configuración de ONLY_FULL_GROUP_BY entra en juego como se ve en la siguiente página del Manual de MySQL titulada Manejo de MySQL de GROUP BY.

SELECCIONA todas las columnas (*) #

Consulta

SELECT * FROM stack;

Resultado

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
|    2 | stack    | stack    |
+------+----------+----------+
2 rows in set (0.00 sec)

Puede seleccionar todas las columnas de una tabla en una combinación haciendo lo siguiente:

SELECT stack.* FROM stack JOIN Overflow ON stack.id = Overflow.id;

Práctica recomendada No utilice * a menos que esté depurando o recuperando las filas en matrices asociativas; de lo contrario, los cambios de esquema (AGREGAR/ELIMINAR/reorganizar columnas) pueden provocar errores de aplicación desagradables. Además, si proporciona la lista de columnas que necesita en su conjunto de resultados, el planificador de consultas de MySQL a menudo puede optimizar la consulta.

Pros:

  1. Cuando agrega/elimina columnas, no tiene que hacer cambios donde usó SELECT *
  2. Es más corto de escribir
  3. También ve las respuestas, entonces, ¿se puede justificar el uso de SELECT *?

Contras:

  1. Estás devolviendo más datos de los que necesitas. Digamos que agrega una columna VARBINARY que contiene 200k por fila. Solo necesita estos datos en un solo lugar para un solo registro: al usar SELECT * puede terminar devolviendo 2 MB por cada 10 filas que no necesita
  2. Explícito sobre qué datos se utilizan
  3. Especificar columnas significa que obtiene un error cuando se elimina una columna
  4. El procesador de consultas tiene que trabajar un poco más: descubrir qué columnas existen en la tabla (gracias @vinodadhikary)
  5. Puede encontrar dónde se usa una columna más fácilmente
  6. Obtienes todas las columnas en combinaciones si usas SELECCIONAR *
  7. No puede usar referencias ordinales de manera segura (aunque usar referencias ordinales para columnas es una mala práctica en sí misma)
  8. En consultas complejas con campos TEXT, la consulta puede ralentizarse debido a un procesamiento de tabla temporal menos óptimo

CREATE TABLE stack(
    id INT,

INSERT INTO stack (`id`, `username`, `password`) VALUES (1, 'Foo', 'hiddenGem');
INSERT INTO stack (`id`, `username`, `password`) VALUES (2, 'Baa', 'verySecret');

Consulta

SELECT id FROM stack;

Resultado

+------+
| id   |
+------+
|    1 |
|    2 |
+------+

SELECCIONA con ME GUSTA (%) #

CREATE TABLE stack
(  id int AUTO_INCREMENT PRIMARY KEY,
   username VARCHAR(100) NOT NULL
);

INSERT stack(username) VALUES 
('admin'),('k admin'),('adm'),('a adm b'),('b XadmY c'), ('adm now'), ('not here');

“adm” en cualquier lugar:

SELECT * FROM stack WHERE username LIKE "%adm%";  
+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | k admin   |
|  3 | adm       |
|  4 | a adm b   |
|  5 | b XadmY c |
|  6 | adm now   |
+----+-----------+

Comienza con “adm”:

SELECT * FROM stack WHERE username LIKE "adm%";
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
|  3 | adm      |
|  6 | adm now  |
+----+----------+

Termina con “adm”:

SELECT * FROM stack WHERE username LIKE "%adm"; 
+----+----------+
| id | username |
+----+----------+
|  3 | adm      |
+----+----------+

Así como el carácter % en una cláusula LIKE coincide con cualquier número de caracteres, el carácter _ coincide con un solo carácter. Por ejemplo,

SELECT * FROM stack WHERE username LIKE "adm_n"; 
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
+----+----------+

Notas de rendimiento Si hay un índice en nombre de usuario, entonces

  • LIKE 'adm' funciona igual que `= ‘adm’
  • LIKE 'adm% es un “rango”, similar a BETWEEN..AND.. Puede hacer un buen uso de un índice en la columna.
  • LIKE '%adm' (o cualquier variante con un comodín principal) no puede usar ningún índice. Por lo tanto, será lento. En tablas con muchas filas, es probable que sea tan lento que sea inútil.
  • RLIKE (REGEXP) tiende a ser más lento que LIKE, pero tiene más capacidades.
  • Si bien MySQL ofrece indexación FULLTEXT en muchos tipos de tablas y columnas, esos índices FULLTEXT no se usan para completar consultas usando LIKE.

SELECCIONA con CASE o IF #

Consulta

SELECT st.name,
       st.percentage, 
       CASE WHEN st.percentage >= 35 THEN 'Pass' ELSE 'Fail' END AS `Remark` 
FROM student AS st ;

Resultado

+--------------------------------+
|   name   | percentage | Remark |
+--------------------------------+
|   Isha   |     67     |  Pass  |
|   Rucha  |     28     |  Fail  |
|   Het    |     35     |  Pass  |
|   Ansh   |     92     |  Pass  |
+--------------------------------+

O con SI

SELECT st.name,
       st.percentage, 
       IF(st.percentage >= 35, 'Pass', 'Fail') AS `Remark` 
FROM student AS st ;

NÓTESE BIEN

IF(st.percentage >= 35, 'Pass', 'Fail')

Esto significa: SI st.percentage >= 35 es VERDADERO entonces devuelve 'Pass' ELSE devolver 'Falla'

SELECCIONA con Alias ​​(AS) #

Los alias de SQL se utilizan para cambiar temporalmente el nombre de una tabla o una columna. Generalmente se utilizan para mejorar la legibilidad.

Consulta

SELECT username AS val FROM stack; 
SELECT username val FROM stack;

(Nota: AS es sintácticamente opcional).

Resultado

+-------+
| val   |
+-------+
| admin |
| stack |
+-------+
2 rows in set (0.00 sec)

SELECT con una cláusula LIMIT #

Consulta:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 3;

Resultado:

CustomerID CustomerName ContactName Address City PostalCode Country 1

Alfreds Futterkiste Maria Anders Obere Str. 57 Berlin 12209 Germany 2 Ana Trujillo Emparedados y helados Ana Trujillo Avda. de la Constitución 2222 México D.F. 05021 Mexico 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Mejor práctica Utilice siempre ORDER BY cuando utilice LIMIT; de lo contrario, las filas que obtendrá serán impredecibles.

Consulta:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 2,1;

Explicación:

Cuando una cláusula LIMIT contiene dos números, se interpreta como LIMIT offset,count. Entonces, en este ejemplo, la consulta omite dos registros y devuelve uno.

Resultado:

CustomerID CustomerName ContactName Address City PostalCode Country 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Nota:

Los valores en las cláusulas LIMIT deben ser constantes; pueden no ser valores de columna.

Consulta

SELECT * FROM stack WHERE username = "admin" AND password = "admin";

Resultado

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
+------+----------+----------+
1 row in set (0.00 sec)

Consulta con un SELECT anidado en la cláusula WHERE #

La cláusula WHERE puede contener cualquier instrucción SELECT válida para escribir consultas más complejas. Esta es una consulta ‘anidada’

Consulta

Las consultas anidadas generalmente se usan para devolver valores atómicos únicos de consultas para realizar comparaciones.

SELECT title FROM books WHERE author_id = (SELECT id FROM authors WHERE last_name = 'Bar' AND first_name = 'Foo');

Selecciona todos los nombres de usuario sin dirección de correo electrónico

SELECT * FROM stack WHERE username IN (SELECT username FROM signups WHERE email IS NULL);

Descargo de responsabilidad: considere usar joins para mejorar el rendimiento al comparar un conjunto de resultados completo.

1: http://stackoverflow.com/questions/17946221/sql-join-and- different-types-of-joins

SELECCIONA con ENTRE #

Puede usar la cláusula BETWEEN para reemplazar una combinación de condiciones “mayor que igual Y menor que igual”.

Datos

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
|  6 | java      |
+----+-----------+

Consulta con operadores

SELECT * FROM stack WHERE id >= 2 and id <= 5;

Consulta similar con BETWEEN

SELECT * FROM stack WHERE id BETWEEN 2 and 5;

Resultado

+----+-----------+
| id | username  |
+----+-----------+
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
+----+-----------+
4 rows in set (0.00 sec)

Nota

BETWEEN usa >= y <=, no > y <.

Usando NO ENTRE

Si desea usar el negativo, puede usar NOT. Por ejemplo :

SELECT * FROM stack WHERE id NOT BETWEEN 2 and 5;

Resultado

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  6 | java      |
+----+-----------+
2 rows in set (0.00 sec)

Nota

NO ENTRE usa > y < y no >= y <= Es decir, WHERE id NOT BETWEEN 2 and 5 es lo mismo que WHERE (id < 2 OR id > 5).

Si tiene un índice en una columna que usa en una búsqueda BETWEEN, MySQL puede usar ese índice para un escaneo de rango.

SELECCIONA con ME GUSTA(_) #

Un carácter _ en un patrón de cláusula LIKE coincide con un solo carácter.

Consulta

SELECT username FROM users WHERE users LIKE 'admin_';

Resultado

+----------+
| username |  
+----------+
| admin1   |
| admin2   |
| admin-   |
| adminA   |
+----------+

SELECCIONA con rango de fechas #

SELECT ... WHERE dt >= '2017-02-01'
             AND dt  < '2017-02-01' + INTERVAL 1 MONTH

Claro, esto podría hacerse con BETWEEN y la inclusión de 23:59:59. Pero, el patrón tiene estos beneficios:

  • No tiene que calcular previamente la fecha de finalización (que suele ser una duración exacta desde el inicio)
  • No incluyes ambos extremos (como lo hace BETWEEN), ni escribes ‘23:59:59’ para evitarlo.
  • Funciona para DATE, TIMESTAMP, DATETIME e incluso DATETIME(6) incluido en microsegundos.
  • Se ocupa de los días bisiestos, fin de año, etc.
  • Es compatible con índices (también lo es BETWEEN).
