Replicación

La replicación se utiliza para copiar datos [Copia de seguridad] de un servidor de base de datos MySQL a uno o más servidores de base de datos MySQL.

Master – El servidor de base de datos MySQL, que sirve datos para ser copiados

Slave – El servidor de base de datos MySQL, copia datos que son servidos por Master

Con MySQL, la replicación es asíncrona de forma predeterminada. Esto significa que los esclavos no necesitan estar conectados permanentemente para recibir actualizaciones del maestro. Por ejemplo, si su esclavo está APAGADO o no está conectado con el maestro y está encendiendo el esclavo o se conecta con el maestro en un momento posterior, se sincronizará automáticamente con el maestro.

Según la configuración, puede replicar todas las bases de datos, las bases de datos seleccionadas o incluso las tablas seleccionadas dentro de una base de datos.

Formatos de replicación

Hay dos tipos principales de formatos de replicación

Replicación basada en sentencias (SBR) – que replica sentencias SQL completas. En esto, el maestro escribe declaraciones SQL en el registro binario. La replicación del maestro al esclavo funciona ejecutando esas declaraciones SQL en el esclavo.

Replicación basada en filas (RBR) – que replica solo las filas modificadas. En esto, el maestro escribe eventos en el registro binario que indican cómo se cambian las filas individuales de la tabla. La replicación del maestro al esclavo funciona copiando los eventos que representan los cambios en las filas de la tabla al esclavo.

También puede usar una tercera variedad, Replicación basada en mixtos (MBR). En esto, se utiliza tanto el registro basado en declaraciones como el basado en filas. El registro se creará dependiendo de cuál sea más apropiado para el cambio.

El formato basado en declaraciones era el predeterminado en las versiones de MySQL anteriores a la 5.7.7. En MySQL 5.7.7 y versiones posteriores, el formato basado en filas es el predeterminado.

Maestro - Configuración de replicación esclava #

Considere 2 servidores MySQL para la configuración de la replicación, uno es maestro y el otro es esclavo.

Vamos a configurar el Maestro para que mantenga un registro de cada acción realizada en él. Vamos a configurar el servidor Esclavo para que mire el registro en el Maestro y cada vez que ocurran cambios en el inicio de sesión en el Maestro, debería hacer lo mismo.

Configuración maestra

En primer lugar, necesitamos crear un usuario en el Maestro. Este usuario será utilizado por Slave para crear una conexión con Master.

CREATE USER 'user_name'@'%' IDENTIFIED BY 'user_password';
GRANT REPLICATION SLAVE ON *.* TO 'user_name'@'%';
FLUSH PRIVILEGES;

Cambie user_name y user_password según su nombre de usuario y contraseña.

Ahora se debe editar el archivo my.inf (my.cnf en Linux). Incluya las siguientes líneas en la sección [mysqld].

server-id = 1
log-bin = mysql-bin.log
binlog-do-db = your_database

La primera línea se usa para asignar una ID a este servidor MySQL.

La segunda línea le dice a MySQL que comience a escribir un registro en el archivo de registro especificado. En Linux, esto se puede configurar como log-bin = /home/mysql/logs/mysql-bin.log. Si está iniciando la replicación en un servidor MySQL en el que ya se ha utilizado la replicación, asegúrese de que este directorio esté vacío de todos los registros de replicación.

La tercera línea se usa para configurar la base de datos para la que vamos a escribir el registro. Debe reemplazar su_base de datos con el nombre de su base de datos.

Asegúrese de que skip-networking no esté habilitado y reinicie el servidor MySQL (maestro)

Configuración esclava

El archivo my.inf también debe editarse en Slave. Incluya las siguientes líneas en la sección [mysqld].

server-id = 2
master-host = master_ip_address
master-connect-retry = 60

master-user = user_name
master-password = user_password
replicate-do-db = your_database

relay-log = slave-relay.log
relay-log-index = slave-relay-log.index

La primera línea se usa para asignar una ID a este servidor MySQL. Esta identificación debe ser única.

La segunda línea es la dirección IP del servidor maestro. Cambie esto de acuerdo con su sistema maestro I.P.

La tercera línea se utiliza para establecer un límite de reintentos en segundos.

Las siguientes dos líneas le dicen el nombre de usuario y la contraseña al Esclavo, mediante el cual se conecta al Maestro.

La siguiente línea establece la base de datos que necesita replicar.

Las dos últimas líneas se utilizan para asignar los nombres de archivo relay-log y relay-log-index.

Asegúrese de que skip-networking no esté habilitado y reinicie el servidor MySQL (Slave)

Copiar datos al esclavo

Si constantemente se agregan datos al maestro, tendremos que evitar todo acceso a la base de datos en el maestro para que no se pueda agregar nada. Esto se puede lograr ejecutando la siguiente instrucción en Master.

FLUSH TABLES WITH READ LOCK;

Si no se agregan datos al servidor, puede omitir el paso anterior.

Vamos a hacer una copia de seguridad de los datos del Maestro usando mysqldump

mysqldump your_database -u root -p > D://Backup/backup.sql;

Cambie su_base de datos y el directorio de copia de seguridad de acuerdo con su configuración. Ahora tendrá un archivo llamado backup.sql en la ubicación dada.

Si su base de datos no existe en su Esclavo, créela ejecutando lo siguiente

CREATE DATABASE `your_database`;

Ahora tenemos que importar la copia de seguridad al servidor Slave MySQL.

mysql -u root -p your_database  <D://Backup/backup.sql
--->Change `your_database` and backup directory according to your setup

Iniciar replicación

Para iniciar la replicación, necesitamos encontrar el nombre del archivo de registro y la posición del registro en el Maestro. Entonces, ejecuta lo siguiente en Master

SHOW MASTER STATUS;

Esto le dará una salida como la siguiente

+---------------------+----------+-------------------------------+------------------+
| File                | Position | Binlog_Do_DB                  | Binlog_Ignore_DB |
+---------------------+----------+-------------------------------+------------------+
| mysql-bin.000001    | 130      | your_database                 |                  |
+---------------------+----------+-------------------------------+------------------+

Luego ejecuta lo siguiente en Slave

SLAVE STOP;
CHANGE MASTER TO MASTER_HOST='master_ip_address', MASTER_USER='user_name', 
   MASTER_PASSWORD='user_password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS=130;
SLAVE START;

Primero detenemos al Esclavo. Luego le decimos exactamente dónde buscar en el archivo de registro maestro. Para el nombre de MASTER_LOG_FILE y MASTER_LOG_POS, use los valores que obtuvimos al ejecutar el comando SHOW MASTER STATUS en el maestro.

Debe cambiar la IP del maestro en MASTER_HOST y cambiar el usuario y la contraseña en consecuencia.

El Esclavo ahora estará esperando. El estado del Esclavo se puede ver ejecutando lo siguiente

SHOW SLAVE STATUS;

Si anteriormente ejecutó FLUSH TABLES WITH READ LOCK en Master, libere las tablas del bloqueo ejecutando lo siguiente

UNLOCK TABLES;

Ahora el Maestro mantiene un registro de cada acción realizada en él y el servidor Esclavo mira el registro en el Maestro. Cada vez que se producen cambios en el inicio de sesión en el Maestro, el Esclavo lo replica.

Errores de replicación #

Cada vez que hay un error al ejecutar una consulta en el esclavo, MySQL detiene la replicación automáticamente para identificar el problema y solucionarlo. Esto se debe principalmente a que un evento provocó una clave duplicada o no se encontró una fila y no se puede actualizar ni eliminar. Puede omitir tales errores, incluso si esto no se recomienda

Para omitir solo una consulta que está bloqueando al esclavo, use la siguiente sintaxis

SET GLOBAL sql_slave_skip_counter = N;

Esta declaración salta los próximos N eventos del maestro. Esta declaración es válida solo cuando los subprocesos esclavos no se están ejecutando. De lo contrario, produce un error.

STOP SLAVE;
SET GLOBAL sql_slave_skip_counter=1;
START SLAVE;

En algunos casos esto está bien. Pero si la declaración es parte de una transacción de varias declaraciones, se vuelve más compleja, porque omitir la declaración que produce el error hará que se omita toda la transacción.

Si desea omitir más consultas que producen el mismo código de error y si está seguro de que omitir esos errores no hará que su esclavo sea inconsistente y desea omitirlos todos, agregaría una línea para omitir ese código de error en su archivo my. cnf.

Por ejemplo, es posible que desee omitir todos los errores duplicados que pueda estar recibiendo

1062 | Error 'Duplicate entry 'xyz' for key 1' on query

Luego agregue lo siguiente a su my.cnf

slave-skip-errors = 1062

También puede omitir otros tipos de errores o todos los códigos de error, pero asegúrese de que omitir esos errores no hará que su esclavo sea inconsistente. Los siguientes son la sintaxis y ejemplos.

slave-skip-errors=[err_code1,err_code2,...|all]

slave-skip-errors=1062,1053
slave-skip-errors=all
slave-skip-errors=ddl_exist_errors
