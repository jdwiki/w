Código de error 1064: error de sintaxis #

select LastName, FirstName,
from Person

Devuelve mensaje:

Código de error: 1064. Tiene un error en su sintaxis SQL; consulte el manual que corresponde a la versión de su servidor MySQL para conocer la sintaxis correcta para usar cerca de ‘de la persona’ en la línea 2.

Recibir un mensaje de “error 1064” de MySQL significa que la consulta no se puede analizar sin errores de sintaxis. En otras palabras, no puede dar sentido a la consulta.

La cita en el mensaje de error comienza con el primer carácter de la consulta que MySQL no puede descifrar cómo analizar. En este ejemplo, MySQL no puede tener sentido, en contexto, de from Person. En este caso, hay una coma extra inmediatamente antes de de Persona. La coma le dice a MySQL que espere otra descripción de columna en la cláusula SELECT

Un error de sintaxis siempre dice ... cerca de '...'. Lo del principio de las comillas está muy cerca de donde está el error. Para localizar un error, observe el primer token entre comillas y el último token antes de las comillas.

A veces obtendrá ... cerca de ''; es decir, nada en las comillas. Eso significa que el primer carácter que MySQL no puede descifrar está justo al final o al principio de la declaración. Esto sugiere que la consulta contiene comillas desequilibradas (' o ") o paréntesis desequilibrados o que no finalizó la declaración antes correctamente.

En el caso de una Rutina Almacenada, es posible que haya olvidado usar apropiadamente DELIMITER.

Entonces, cuando obtenga el error 1064, mire el texto de la consulta y busque el punto mencionado en el mensaje de error. Inspeccione visualmente el texto de la consulta alrededor de ese punto.

Si le pide a alguien que lo ayude a solucionar el error 1064, es mejor proporcionar tanto el texto de la consulta completa como el texto del mensaje de error.

Código de error 1175: Actualización segura #

Este error aparece al intentar actualizar o eliminar registros sin incluir la cláusula WHERE que usa la columna KEY.

Para ejecutar la eliminación o actualización de todos modos, escriba:

SET SQL_SAFE_UPDATES = 0;

Para habilitar el modo seguro nuevamente, escriba:

SET SQL_SAFE_UPDATES = 1;

Código de error 1215: No se puede agregar una restricción de clave externa #

Este error ocurre cuando las tablas no están estructuradas adecuadamente para manejar la verificación de búsqueda rápida de los requisitos de clave externa (FK) que exige el desarrollador.

CREATE TABLE `gtType` (
  `type` char(2) NOT NULL,
  `description` varchar(1000) NOT NULL,
  PRIMARY KEY (`type`)
) ENGINE=InnoDB;

CREATE TABLE `getTogethers` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `type` char(2) NOT NULL,
  `eventDT` datetime NOT NULL,
  `location` varchar(1000) NOT NULL,
  PRIMARY KEY (`id`),
  KEY `fk_gt2type` (`type`), -- see Note1 below 
  CONSTRAINT `gettogethers_ibfk_1` FOREIGN KEY (`type`) REFERENCES `gtType` (`type`)
) ENGINE=InnoDB;

Nota 1: una CLAVE como esta se creará automáticamente si es necesario debido a la definición de FK en la línea que lo sigue. El desarrollador puede omitirlo y la CLAVE (también conocida como índice) se agregará si es necesario. A continuación se muestra un ejemplo en el que el desarrollador lo omitió en someOther.

Hasta ahora todo bien, hasta la siguiente llamada.

CREATE TABLE `someOther` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `someDT` datetime NOT NULL,
  PRIMARY KEY (`id`),
  CONSTRAINT `someOther_dt` FOREIGN KEY (`someDT`) REFERENCES `getTogethers` (`eventDT`)
) ENGINE=InnoDB;

Código de error: 1215. No se puede agregar una restricción de clave externa

En este caso falla debido a la falta de un índice en la tabla referenciada getTogethers para manejar la búsqueda rápida de un eventDT. Se resolverá en la próxima declaración.

CREATE INDEX `gt_eventdt` ON getTogethers (`eventDT`);

La tabla getTogethers ha sido modificada, y ahora la creación de someOther tendrá éxito.

Desde la página del manual de MySQL Uso de restricciones FOREIGN KEY:

MySQL requiere índices en claves foráneas y claves referenciadas para que las comprobaciones de claves foráneas pueden ser rápidas y no requieren un escaneo de tabla. En el tabla de referencia, debe haber un índice donde la clave foránea las columnas se enumeran como las primeras columnas en el mismo orden. Tal el índice se crea en la tabla de referencia automáticamente si no lo hace existir.

Las columnas correspondientes en la clave foránea y la clave referenciada deben tienen tipos de datos similares. El tamaño y el signo de los tipos enteros deben ser lo mismo. La longitud de los tipos de cadena no necesita ser la misma. Para columnas de cadenas no binarias (caracteres), el juego de caracteres y la intercalación debe ser igual.

InnoDB permite que una clave externa haga referencia a cualquier columna de índice o grupo de columnas. Sin embargo, en la tabla referenciada, debe haber un índice donde las columnas a las que se hace referencia se enumeran como las primeras columnas en el mismo orden.

Tenga en cuenta el último punto anterior sobre las primeras columnas (más a la izquierda) y la falta de un requisito de clave principal (aunque muy recomendable).

Tras la creación exitosa de una tabla de referencia (secundaria), todas las claves que se crearon automáticamente para usted son visibles con un comando como el siguiente:

SHOW CREATE TABLE someOther;

Otros casos comunes de experimentar este error incluyen, como se mencionó anteriormente en los documentos, pero deben resaltarse:

  • Diferencias aparentemente triviales en INT que está firmado, apuntando hacia INT UNSIGNED.

  • Los desarrolladores tienen problemas para comprender las CLAVES de varias columnas (compuestas) y los primeros requisitos de pedido (más a la izquierda).

1067, 1292, 1366, 1411 - Valor incorrecto para número, fecha, predeterminado, etc. #

1067 Esto probablemente esté relacionado con los valores predeterminados de TIMESTAMP, que han cambiado con el tiempo. Consulte Valores predeterminados de TIMESTAMP en la página Fechas y horas. (que aún no existe)

1292/1366 DOBLE/Entero Verifique letras u otros errores de sintaxis. Verifique que las columnas estén alineadas; tal vez crea que está ingresando un VARCHAR pero está alineado con una columna numérica.

1292 DATETIME Comprobar demasiado en el pasado o en el futuro. Verifique entre las 2 a. m. y las 3 a. m. de una mañana cuando cambió el horario de verano. Compruebe si hay una mala sintaxis, como +00 cosas de la zona horaria.

1292 VARIABLE Verifique los valores permitidos para la VARIABLE que está tratando de FIJAR.

1292 CARGAR DATOS Mire la línea que es ‘mala’. Compruebe los símbolos de escape, etc. Mire los tipos de datos.

1411 STR_TO_DATE ¿Fecha con formato incorrecto?

1045 Acceso denegado #

Consulte las discusiones en “GRANT” y “Recuperación de la contraseña de root”.

1236 “posición imposible” en replicación #

Usualmente esto significa que el Maestro falló y que sync_binlog estaba APAGADO. La solución es CAMBIAR MAESTRO a POS=0 del siguiente archivo binlog (ver el Maestro) en el Esclavo.

La causa: el Maestro envía elementos de replicación al Esclavo antes de vaciar su binlog (cuando sync_binlog=OFF). Si el Maestro se bloquea antes del vaciado, el Esclavo ya se ha movido lógicamente más allá del final del archivo en el binlog. Cuando el maestro se inicia de nuevo, inicia un nuevo binlog, por lo que CAMBIAR al principio de ese binlog es la mejor solución disponible.

Una solución a más largo plazo es sync_binlog=ON, si puede permitirse la E/S adicional que provoca.

2002, 2003 No se puede conectar #

Compruebe si hay un problema de firewall que bloquee el puerto 3306.

Algunos posibles diagnósticos y/o soluciones

  • ¿Está funcionando realmente el servidor?
  • “detener el servicio firewalld” y “systemctl disabled firewalld”
  • telnet maestro 3306
  • Verifique la dirección de enlace
  • marcar saltar-nombre-resolver
  • comprobar el enchufe.

126, 127, 134, 144, 145 #

Cuando intenta acceder a los registros de la base de datos MySQL, puede recibir estos mensajes de error. Estos mensajes de error ocurrieron debido a la corrupción en la base de datos MySQL. Los siguientes son los tipos

MySQL error code 126 = Index file is crashed
MySQL error code 127 = Record-file is crashed
MySQL error code 134 = Record was already deleted (or record file crashed)
MySQL error code 144 = Table is crashed and last repair failed
MySQL error code 145 = Table was marked as crashed and should be repaired

El error de MySQL, el ataque de virus, la falla del servidor, el apagado incorrecto, la tabla dañada son la razón detrás de esta corrupción. Cuando se corrompe, se vuelve inaccesible y ya no puede acceder a ellos. Para obtener accesibilidad, la mejor manera de recuperar datos de una copia de seguridad actualizada. Sin embargo, si no tiene una copia de seguridad actualizada o válida, puede optar por la reparación de MySQL.

Si el tipo de motor de tabla es MyISAM, aplique CHECK TABLE, luego REPAIR TABLE.

Entonces piense seriamente en convertir a InnoDB, para que este error no vuelva a ocurrir.

Sintaxis

CHECK TABLE <table name> ////To check the extent of database corruption
REPAIR TABLE <table name> ////To repair table

139 #

El error 139 puede significar que el número y el tamaño de los campos en la definición de la tabla excede algún límite. Soluciones alternativas:

  • Repensar el esquema
  • Normalizar algunos campos
  • Dividir verticalmente la mesa

1366 #

Esto generalmente significa que el manejo del conjunto de caracteres no fue consistente entre el cliente y el servidor. Consulte… para obtener más ayuda.

24 No se puede abrir el archivo (Demasiados archivos abiertos)

open_files_limit proviene de una configuración del sistema operativo. table_open_cache debe ser menor que eso.

Estos pueden causar ese error:

  • Fallo en DEALLOCATE PREPARE en un procedimiento almacenado.

  • Tabla(s) PARTICIONADA(S) con una gran cantidad de particiones e innodb_file_per_table = ON. Recomendar no tener más de 50 particiones en una tabla dada (por varias razones). (Cuando las “particiones nativas” estén disponibles, este consejo puede cambiar).

La solución obvia es aumentar el límite del sistema operativo: Para permitir más archivos, cambie ulimit o /etc/security/limits.conf o en sysctl.conf (kern.maxfiles & kern.maxfilesperproc) o algo más (depende del sistema operativo). Luego aumente open_files_limit y table_open_cache.

A partir de la versión 5.6.8, open_files_limit tiene un tamaño automático basado en max_connections, pero está bien cambiarlo por defecto.

1062 - Entrada duplicada

Este error ocurre principalmente debido a las siguientes dos razones

  1. Valor duplicado - ‘Código de error: 1062. Entrada duplicada ‘12’ para clave ‘PRIMARY’`

    The primary key column is unique and it will not accept the duplicate entry. So when you are trying to insert a new row which is already present in you table will produce this error.

Para resolver esto, establezca la columna de clave principal como AUTO_INCREMENT. Y cuando intente insertar una nueva fila, ignore la clave principal columna o inserte el valor NULL en la clave principal.

CREATE TABLE userDetails(
  userId INT(10) NOT NULL AUTO_INCREMENT,
  firstName VARCHAR(50),
  lastName VARCHAR(50),
  isActive INT(1) DEFAULT 0,
  PRIMARY KEY (userId) );

--->and now while inserting 
INSERT INTO userDetails VALUES (NULL ,'John', 'Doe', 1);

  1. Campo de datos único - ‘Código de error: 1062. Entrada duplicada ‘A’ para clave ‘code’`

    You may assigned a column as unique and trying to insert a new row with already existing value for that column will produce this error.

Para solucionar este error, utilice INSERT IGNORE en lugar de INSERT normal. Si la nueva fila que intenta insertar no duplica un registro existente, MySQL lo inserta como de costumbre. Si el registro es un duplicado, la palabra clave IGNORE lo descarta sin generar ningún error.

INSERT IGNORE INTO userDetails VALUES (NULL ,'John', 'Doe', 1);
