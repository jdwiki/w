Sintaxis #

  • importar {componente} de 'libName'; // Importará la clase "componente"
  • `importar {componente como c} desde ’libName’; // Importará la clase “componente” a un objeto “c”
  • importar componente de 'libname'; // Importará la exportación predeterminada de libName
  • importar * como lib desde 'libName'; // Importará todo, desde libName a un objeto "lib"
  • importar lib = require('libName'); // Importará todo, desde libName a un objeto "lib"
  • const lib: any = require('libName'); // Importará todo, desde libName a un objeto "lib"
  • importar 'libName'; // Importará el módulo libName solo por sus efectos secundarios

Podría parecer que la sintaxis

import * as lib from 'libName';

y

import lib = require('libName');

son lo mismo, pero no lo son!

Consideremos que queremos importar una clase Persona exportada con la sintaxis export = específica de TypeScript:

class Person {
...
}
export = Person;

En este caso, no es posible importarlo con la sintaxis es6 (obtendríamos un error en tiempo de compilación), se debe usar la sintaxis import = específica de TypeScript.

import * as Person from 'Person';  //compile error
import Person = require('Person');  //OK

Lo contrario es cierto: los módulos clásicos se pueden importar con la segunda sintaxis, por lo que, en cierto modo, la última sintaxis es más poderosa ya que puede importar todas las exportaciones.

Para obtener más información, consulte la documentación oficial.

Encontrar archivos de definición #

para mecanografiado 2.x:

las definiciones de DefinitelyTyped están disponibles a través del paquete @types npm

npm i --save lodash
npm i --save-dev @types/lodash

pero en caso de que desee usar tipos de otros repositorios, se puede usar de la manera anterior:

para mecanografiado 1.x:

Typings es un paquete npm que puede instalar automáticamente archivos de definición de tipo en un proyecto local. Le recomiendo que lea el [inicio rápido] (https://github.com/typings/typings#quick-start).

npm install -global typings

Ahora tenemos acceso al cli de digitaciones.

  1. El primer paso es buscar el paquete utilizado por el proyecto

     typings search lodash
 NAME              SOURCE HOMEPAGE                                        DESCRIPTION VERSIONS UPDATED
 lodash            dt     http://lodash.com/                                          2        2016-07-20T00:13:09.000Z
 lodash            global                                                             1        2016-07-01T20:51:07.000Z
 lodash            npm    https://www.npmjs.com/package/lodash                        1        2016-07-01T20:51:07.000Z

  2. Luego, decida desde qué fuente debe instalar. Utilizo dt, que significa DefinitelyTyped, un repositorio de GitHub donde la comunidad puede editar las escrituras, también suele ser el actualizado más recientemente.

  3. Instalar los archivos de mecanografía

     typings install dt~lodash --global --save

Analicemos el último comando. Estamos instalando la versión DefinitelyTyped de lodash como un archivo de tipificación global en nuestro proyecto y lo guardamos como una dependencia en typings.json. Ahora, donde sea que importemos lodash, TypeScript cargará el archivo de tipos de lodash.

  1. Si queremos instalar tipos que se usarán solo para el entorno de desarrollo, podemos proporcionar el indicador --save-dev:

     typings install chai --save-dev

Importando un módulo desde npm #

Si tiene un archivo de definición de tipo (d.ts) para el módulo, puede usar una declaración de “importación”.

import _ = require('lodash');

Si no tiene un archivo de definición para el módulo, TypeScript generará un error en la compilación porque no puede encontrar el módulo que está tratando de importar.

En este caso, puede importar el módulo con la función requerir del tiempo de ejecución normal. Sin embargo, esto lo devuelve como el tipo cualquiera.

// The _ variable is of type any, so TypeScript will not perform any type checking.
const _: any = require('lodash');

A partir de TypeScript 2.0, también puede usar una declaración de módulo ambiental abreviada para decirle a TypeScript que existe un módulo cuando no tiene un archivo de definición de tipo para el módulo. Sin embargo, TypeScript no podrá proporcionar ninguna verificación de tipo significativa en este caso.

declare module "lodash";

// you can now import from lodash in any way you wish:
import { flatten } from "lodash";
import * as _ from "lodash";

A partir de TypeScript 2.1, las reglas se han relajado aún más. Ahora, siempre que exista un módulo en su directorio node_modules, TypeScript le permitirá importarlo, incluso sin una declaración de módulo en ninguna parte. (Tenga en cuenta que si usa la opción del compilador --noImplicitAny, lo siguiente aún generará una advertencia).

// Will work if `node_modules/someModule/index.js` exists, or if `node_modules/someModule/package.json` has a valid "main" entry point
import { foo } from "someModule";

Uso de bibliotecas externas globales sin tipificación #

Aunque los módulos son ideales, si la biblioteca que está utilizando está referenciada por una variable global (como $ o _), debido a que fue cargada por una etiqueta script, puede crear una declaración ambiental para hacer referencia a ella:

declare const _: any;

Finding definition files with typescript 2.x #

With the 2.x versions of typescript, typings are now available from the npm @types repository. These are automatically resolved by the typescript compiler and are much simpler to use.

To install a type definition you simply install it as a dev dependency in your projects package.json

e.g.

npm i -S lodash
npm i -D @types/lodash

after install you simply use the module as before

import * as _ from 'lodash'
