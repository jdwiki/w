Una interfaz especifica una lista de campos y funciones que se pueden esperar en cualquier clase que implemente la interfaz. Por el contrario, una clase no puede implementar una interfaz a menos que tenga todos los campos y funciones especificados en la interfaz.

El principal beneficio de usar interfaces es que permite usar objetos de diferentes tipos de forma polimórfica. Esto se debe a que cualquier clase que implemente la interfaz tiene al menos esos campos y funciones.

Sintaxis #

  • interfaz Nombre de interfaz {
  •   nombre de parámetro: tipo de parámetro;
  •   opcionalParameterName?: parámetroType;
  • }

Interfaces vs Tipo Alias #

Las interfaces son buenas para especificar la forma de un objeto, por ejemplo, para un objeto de persona que podría especificar

interface person {
    id?: number;
    name: string;
    age: number;
}

Sin embargo, ¿qué sucede si desea representar, por ejemplo, la forma en que se almacena una persona en una base de datos SQL? Dado que cada entrada de base de datos consiste en una fila de forma [cadena, cadena, número] (por lo tanto, una matriz de cadenas o números), no hay forma de que pueda representar esto como una forma de objeto, porque la fila no tiene cualquier propiedad como tal, es solo una matriz.

Esta es una ocasión en la que los tipos son útiles. En lugar de especificar en cada función que acepta un parámetro de fila function processRow(row: [cadena, cadena, número]), puede crear un alias de tipo separado para una fila y luego usarlo en cada función:

type Row = [string, string, number];
function processRow(row: Row)

Documentación oficial de la interfaz #

https://www.typescriptlang.org/docs/handbook/interfaces.html

Ampliación de la interfaz #

Supongamos que tenemos una interfaz:

interface IPerson {
    name: string;
    age: number;

    breath(): void;
}

Y queremos crear una interfaz más específica que tenga las mismas propiedades de la persona, podemos hacerlo usando la palabra clave extends:

interface IManager extends IPerson {
    managerId: number;

    managePeople(people: IPerson[]): void;
}

Además, es posible ampliar múltiples interfaces.

Interfaz de clase #

Declare el tipo de variables y métodos públicos en la interfaz para definir cómo puede interactuar con él otro código mecanografiado.

interface ISampleClassInterface {
  sampleVariable: string;

  sampleMethod(): void;
  
  optionalVariable?: string;
}

Aquí creamos una clase que implementa la interfaz.

class SampleClass implements ISampleClassInterface {
  public sampleVariable: string;
  private answerToLifeTheUniverseAndEverything: number;

  constructor() {
    this.sampleVariable = 'string value';
    this.answerToLifeTheUniverseAndEverything = 42;
  }

  public sampleMethod(): void {
    // do nothing
  }
  private answer(q: any): number {
    return this.answerToLifeTheUniverseAndEverything;
  }
}

El ejemplo muestra cómo crear una interfaz ISampleClassInterface y una clase SampleClass que implementa la interfaz.

Agregar funciones o propiedades a una interfaz existente #

Supongamos que tenemos una referencia a la definición de tipo JQuery y queremos extenderla para tener funciones adicionales de un complemento que incluimos y que no tiene una definición de tipo oficial. Podemos ampliarlo fácilmente declarando las funciones agregadas por el complemento en una declaración de interfaz separada con el mismo nombre JQuery:

interface JQuery {
  pluginFunctionThatDoesNothing(): void;

  // create chainable function
  manipulateDOM(HTMLElement): JQuery;
}

El compilador fusionará todas las declaraciones con el mismo nombre en una sola; consulte fusión de declaraciones para obtener más detalles.

Interfaces genéricas #

Al igual que las clases, las interfaces también pueden recibir parámetros polimórficos (también conocidos como genéricos).

Declaración de parámetros genéricos en las interfaces #

interface IStatus<U> {
    code: U;
}

interface IEvents<T> {
    list: T[];
    emit(event: T): void;
    getAll(): T[];
}

Aquí puede ver que nuestras dos interfaces toman algunos parámetros genéricos, T y U.

Implementación de interfaces genéricas #

Crearemos una clase simple para implementar la interfaz IEvents.

class State<T> implements IEvents<T> {
    
    list: T[];
    
    constructor() {
        this.list = [];
    }
    
    emit(event: T): void {
        this.list.push(event);
    }
    
    getAll(): T[] {
        return this.list;
    }
    
}

Vamos a crear algunas instancias de nuestra clase Estado.

En nuestro ejemplo, la clase State manejará un estado genérico usando IStatus<T>. De esta forma, la interfaz IEvent<T> también manejará un IStatus<T>.

const s = new State<IStatus<number>>();

// The 'code' property is expected to be a number, so:
s.emit({ code: 200 }); // works
s.emit({ code: '500' }); // type error 

s.getAll().forEach(event => console.log(event.code));

Aquí nuestra clase State se escribe como ISatus<number>.

const s2 = new State<IStatus<Code>>();

//We are able to emit code as the type Code
s2.emit({ code: { message: 'OK', status: 200 } });

s2.getAll().map(event => event.code).forEach(event => {
    console.log(event.message);
    console.log(event.status);
});

Nuestra clase State se escribe como IStatus<Code>. De esta forma, podemos pasar tipos más complejos a nuestro método de emisión.

Como puede ver, las interfaces genéricas pueden ser una herramienta muy útil para el código escrito estáticamente.

Uso de interfaces para polimorfismo #

La razón principal para usar interfaces para lograr polimorfismo y proporcionar a los desarrolladores la implementación a su manera en el futuro mediante la implementación de métodos de interfaz.

Supongamos que tenemos una interfaz y tres clases:

interface Connector{
    doConnect(): boolean;
}

Esta es la interfaz del conector. Ahora implementaremos eso para la comunicación Wifi.

export class WifiConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via wifi");
        console.log("Get password");
        console.log("Lease an IP for 24 hours");
        console.log("Connected");
        return true
    }

}

Aquí hemos desarrollado nuestra clase concreta llamada WifiConnector que tiene su propia implementación. Este es ahora el tipo Conector.

Ahora estamos creando nuestro Sistema que tiene un componente Conector. Esto se llama inyección de dependencia.

export class System {
    constructor(private connector: Connector){ #inject Connector type
        connector.doConnect()
    }
}

constructor(conector privado: Conector) esta línea es muy importante aquí. Connector es una interfaz y debe tener doConnect(). Como Connector es una interfaz, esta clase System tiene mucha más flexibilidad. Podemos pasar cualquier tipo que haya implementado la interfaz Connector. En el futuro, el desarrollador logra más flexibilidad. Por ejemplo, ahora el desarrollador quiere agregar el módulo de conexión Bluetooth:

export class BluetoothConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via Bluetooth");
        console.log("Pair with PIN");
        console.log("Connected");
        return true
    }

}

Mira que Wifi y Bluetooth tienen su propia implementación. Hay una forma diferente de conectarse. Sin embargo, por lo tanto, ambos han implementado el Tipo Conector y ahora son Tipo Conector. Para que podamos pasar cualquiera de ellos a la clase System como parámetro del constructor. Esto se llama polimorfismo. La clase ‘Sistema’ ahora no sabe si es Bluetooth / Wifi, incluso podemos agregar otro módulo de comunicación como Inferade, Bluetooth5 y lo que sea simplemente implementando la interfaz ‘Conector’.

Esto se llama Escribir pato. El tipo Connector ahora es dinámico ya que doConnect() es solo un marcador de posición y el desarrollador lo implementa como propio.

si en constructor (conector privado: WifiConnector) donde WifiConnector es una clase concreta, ¿qué sucederá? Entonces la clase ‘Sistema’ se acoplará estrechamente solo con WifiConnector nada más. Aquí la interfaz resolvió nuestro problema por polimorfismo.

Uso de interfaces para aplicar tipos #

Uno de los principales beneficios de TypeScript es que aplica tipos de datos de valores que está pasando alrededor de su código para ayudar a prevenir errores.

Digamos que estás haciendo una aplicación de citas para mascotas.

Tienes esta sencilla función que comprueba si dos mascotas son compatibles entre sí…

checkCompatible(petOne, petTwo) {
  if (petOne.species === petTwo.species &&
      Math.abs(petOne.age - petTwo.age) <= 5) {
    return true;
  }
}

Este es un código completamente funcional, pero sería demasiado fácil para alguien, especialmente otras personas que trabajan en esta aplicación que no escribieron esta función, no saber que se supone que deben pasarle objetos con ’especie’ y ’edad’ propiedades. ¡Es posible que intenten por error checkCompatible(petOne.species, petTwo.species) y luego se les deje averiguar los errores arrojados cuando la función intenta acceder a petOne.species.species o petOne.species.age!

Una forma en que podemos evitar que esto suceda es especificar las propiedades que queremos en los parámetros de la mascota:

checkCompatible(petOne: {species: string, age: number}, petTwo: {species: string, age: number}) {
    //...
}

En este caso, Typescript se asegurará de que todo lo que se pase a la función tenga propiedades de ’especie’ y ’edad’ (está bien si tienen propiedades adicionales), pero esta es una solución un poco difícil de manejar, incluso con solo dos propiedades especificadas. ¡Con interfaces, hay una mejor manera!

Primero definimos nuestra interfaz:

interface Pet {
  species: string;
  age: number;
  //We can add more properties if we choose.
}

Ahora todo lo que tenemos que hacer es especificar el tipo de nuestros parámetros como nuestra nueva interfaz, así…

checkCompatible(petOne: Pet, petTwo: Pet) {
  //...
}

… y Typescript se asegurará de que los parámetros pasados ​​a nuestra función contengan las propiedades especificadas en la interfaz Pet.

Implementación implícita y forma de objeto #

TypeScript admite interfaces, pero el compilador genera JavaScript, que no lo hace. Por lo tanto, las interfaces se pierden efectivamente en el paso de compilación. Esta es la razón por la cual la verificación de tipos en las interfaces se basa en la forma del objeto, es decir, si el objeto admite los campos y funciones en la interfaz, y no en si la interfaz está realmente implementada o no.

interface IKickable {
  kick(distance: number): void;
}
class Ball {
  kick(distance: number): void {
    console.log("Kicked", distance, "meters!");
  }
}
let kickable: IKickable = new Ball();
kickable.kick(40);

Así que incluso si Ball no implementa explícitamente IKickable, una instancia de Ball puede asignarse (y manipularse como) un IKickable, incluso cuando se especifica el tipo.
