  • Operadores Numéricos

También hay operadores, implementados como funciones (como operaciones bit a bit, comparación de objetos…).

Examinaremos cada uno con un ejemplo.

Operadores Numéricos #

Los operadores numéricos son los más fáciles y casi iguales que en otros idiomas.

  • +, -, * y /. Operadores de suma, resta, multiplicación y división (en VFP no hay división de enteros, puede convertir un resultado a entero con las funciones INT(), TECHO() y PISO()).
  • Operador % Módulo.
  • ^ y **. Poder del operador(es). Ambos hacen la misma cosa.
  • (). Operadores de agrupación.
  • Los operadores tienen precedencia. El orden es:

( ) ^ (o **) / y *

  • y +
    ? 10 / 5 + 2 && Outputs 4
    ? 2 + 10 / 5 && Outputs 4 as well. Division has precedence.
     
    * Both multiplication and division have same precedence
    * They would be interpreted from left to right.
    ? 4 * 5 / 2 + 5 && Outputs 15
    * Use parentheses whenever you are in doubt or want to be explicit
    ? ( (4 * 5) / 2 ) + 5 && Outputs 15. Explicit grouping of operations

    ? 4 * 5^2 && ^ has precedence, this is same as 4 * (5^2) = 100.
    ? (4 + 5)^2 && Using parentheses we say add 5 to 4 (9) and then square = 81.

Operadores logicos #

Los operadores lógicos en VFP, en su orden de precedencia son:

Operador Descripción
( ) Paréntesis, grupos de expresiones
NO, ! Lógicamente niega la expresión. NO o ! no tiene diferencia
Y Lógicamente Y las expresiones
O Lógicamente O las expresiones
<>, !=, # Verifique la desigualdad. Por lo tanto, lo mismo que OR exclusivo lógico - XOR

Históricamente, NOT, AND, OR se escriben como .NOT., .AND., .OR. Todavía puede usarlos si lo desea, pero AND, OR, NOT es más simple y limpio.

Para falso y verdadero, tienes que usar .F. y T. literales respectivamente. No puede optar por utilizar F y T en su lugar.

* Some logical variables
local llOld, llEmail  && any variable declaration implicitly initializes the variable as .F. - false
? m.llOld, m.llEmail && Prints .F. .F.

llOld   = .T.
llEmail = .F.

if ( m.llOld AND m.llEmail )
   ? 'Old AND should be emailed to'
endif
if ( m.llOld OR m.llEmail )
   ? 'Old OR should be emailed to'
endif
if ( m.llOld AND !m.llEmail ) && Same as (m.llOld AND NOT m.llEmail)
   ? 'Old BUT should NOT be emailed to'
endif

* Above code outputs
Old OR should be emailed to
Old BUT should NOT be emailed to

En VFP, las expresiones lógicas se evalúan de forma abreviada. Es decir, si la primera parte de la comprobación satisface todo el resultado, el resto de la expresión ni siquiera se interpreta. A continuación se muestra una muestra:

? 1 = '2' && An obvious error. It would complain operator/operand type mismatch.

* However we could use such an expression in an if and get no error
* because it is not interpreted at all 
* (VFP is dynamic and there is no compile time check)

local llProcess
llProcess = .T.

if (m.llProcess OR (1='2'))
   ? 'Should do processing'
endif

* Would output

Should do processing

* without any error because m.llProcess true means
* the whole expression would be true, thus the expression after OR 
* is not interpreted at all.

Una trampa que atrapa a los novatos es que, a veces, es posible que necesite varias comprobaciones, por ejemplo, en una consulta SQL, que están conectadas con los operadores AND, OR. Cuando hay muchos de ellos, uno podría ignorar el hecho de que los operadores tienen precedencia (en orden (), NOT, AND, OR) y pensar que la interpretación se haría de izquierda a derecha en una cadena. Considere una muestra:

seleccione * de myTable donde !isCustomer Y débito > 5000 O descuento > 5

¿Cuál es la intención de esta consulta? Si lo hacemos explícito usando paréntesis de agrupación dice:

((NO esCliente) Y débito > 5000) O descuento > 5 simplificado parece “primeraExpresión” O (descuento > 5). Cualquiera que haya sido la intención, debido a esto O seleccionaría: todas las filas que tienen (descuento > 5) - y también aquellas donde es un cliente con débito superior a 5000.

Probablemente la intención era “dame aquellos donde NO es un cliente Y (el débito es superior a 5000 O el descuento es superior a 5)”. Quedaría claro desde el principio si usáramos paréntesis:

seleccione * de myTable donde !isCustomer AND (débito > 5000 O descuento > 5) Puede usar, pero no vale la pena, tener paréntesis para el operador NOT inicial, cuando su operando es una sola expresión, es lo suficientemente legible con su precedencia: !isCustomer se lee claramente como (NOT isCustomer).

Operadores de caracteres #

Solo hay 4 operadores de caracteres, en su orden de precedencia:

Operador Descripción
() Paréntesis para agrupar. Nota: la documentación de VFP que tengo no incluye esta. Sin esto, el operador - es casi siempre inútil.
+ Concatena (une) cadenas una al lado de la otra.
- Concatena cadenas moviendo los espacios finales de la cadena izquierda al final de la cadena derecha.
$ Comprueba si la primera cadena está contenida en la segunda.

+ es el más simple y también se usa para concatenar cadenas en muchos otros idiomas.

local firstName, lastName
firstName = "John"
lastName  = "Smith"

? m.firstName + " " + m.lastName

Salidas: John Smith

- es un poco complicado y no muy conocido. Toma espacios finales de la cadena de la izquierda, agrega esos espacios a la cadena de la derecha. Suponga que tiene una tabla con nombres y apellidos y cada uno de ellos tiene 20 caracteres. Queremos concatenar nombres y apellidos para crear un nombre completo, y también queremos que el tamaño resultante sea fijo (en este caso, 20 + 20 + 1 espacio = 41). Hagamos que también tenga una columna de segundo nombre y queremos que el nombre completo se vea como “apellido, primer nombre, segundo nombre_______”. Es más fácil hacer esto usando el operador -, pero debes notar el truco de usar paréntesis aquí para agrupar, de modo que obtengamos exactamente lo que queremos:

* Create a cursor for our sample and fill in a few names
Create Cursor Names (firstName c(20), midName c(20), lastName c(20))

Insert Into Names (firstName, midName, lastName) Values ('Cetin','', 'Basoz')
Insert Into Names (firstName, midName, lastName) Values ('John', 'M', 'Smith')
Insert Into Names (firstName, midName, lastName) Values ('John', 'F', 'Kennedy')
Insert Into Names (firstName, midName, lastName) Values ('Tom', '', 'Hanks')

* Select with tricky - operator
Select *, ;
    lastName - (', '+firstName-(' '+midName)) As FullName ;
    from Names ;
    INTO Cursor crsNames ;
    nofilter

Browse

Y la salida es así:

nombre medioNombre apellido nombre completo
Cetina Por bosque Por bosque, Cetin
Juan m herrero Smith, Juan M
Juan F kennedy Kennedy, John F.
Tom madejas Hank, Tom

En la columna fullName, todos los espacios finales se empujan hasta el final muy bien. Si marca la estructura, la columna fullName tiene 63 caracteres de ancho (3 * 20 + 3 caracteres que agregamos).

Tenga en cuenta la importancia de agrupar paréntesis (intente eliminar los paréntesis o disponiendo de otra manera).

Aunque el operador - puede ser tentador en tales casos, hay otra cara de la moneda. Este operador es específico de VFP y, por lo tanto, el SQL no es portátil. En su lugar, podría lograr el mismo resultado con este SQL compatible con ANSI:

Select *, ;
    CAST(RTRIM(lastName) +', '+ RTRIM(firstName) +' '+ midName as char(63)) As FullName ;
    from Names ;
    INTO Cursor crsNames ;
    nofilter

El último operador es $. Simplemente verifica si la cadena izquierda es parte de la cadena derecha.

local upcased, digits, hexDigits
upcased = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ'
digits  = '0123456789'
hexDigits  = m.digits + 'ABCDEF'

? 'A' $ m.upcased && .T.
? 'a' $ m.upcased && .F.
? '1' $ m.digits && .T.
? 'F' $ m.digits && .F.
? 'F' $ m.hexDigits && .T.

Importante: En VFP, aunque puede escribir su código en cualquier caso (superior, inferior o mixto), las cadenas siempre distinguen entre mayúsculas y minúsculas. Por ejemplo: “Smith” y “smith” son dos valores distintos. O en su tabla, si hay una columna de país, no encontrará ‘EE. UU.’ si lo busca con ‘usa’. Lo mismo ocurre con el operador $, “GE” $ “Alemania” es falso.

Nota personal: aunque puede que le guste $ por su simplicidad y puede encontrar que se usa a menudo en los códigos fuente de Microsoft, en mi humilde opinión, tiene muy poco valor. Pensando en muchos miles de líneas que he escrito en mi operador, creo que encontraría muy pocas apariciones en mi propio código. Casi siempre hay una mejor alternativa (especialmente cuando el operando izquierdo no es un solo carácter y/o la distinción entre mayúsculas y minúsculas es importante).

Operadores de fecha y hora #

Básicamente, hay dos operadores para los valores de fecha y hora. + y - están sobrecargados (probablemente un término C) para hacer matemáticas de fecha/fechahora:

Operador Descripción
+ Agrega días (fecha) o segundos (fechahora) a un valor de fecha/fechahora.
- Obtiene la diferencia de dos valores de fecha/fechahora. Resta días (fecha) o segundos (fechahora) de los valores de fechahora.

+ es el más fácil. Tiene dos operandos, uno es un valor de fecha o fecha y hora y el otro es un valor numérico (aunque puede usar cualquier valor numérico, es un número entero para todos los propósitos prácticos).

Cuando uno de los operandos es una fecha, el operando numérico se toma como “día”:

? Date() + 10 && Get the date 10 days later
* VFP is leap year aware when doing date math
? Date(2016, 2, 28) + 1 && Add 1 day to Feb 28, 2016. Returns Feb 29, 2016.
? Date(2017, 2, 28) + 1 && Add 1 day to Feb 28, 2017. Returns Mar 1, 2017.

Cuando uno de los operandos es una fecha y hora, el operando numérico se toma como “segundo”:

Hay 24 * 60 * 60 = 86400 segundos en un día

? Datetime() + 86400 && Add 1 day to current datetime.

Añadir 4 horas y 15 minutos al 1 de enero de 2016 14:20

? Datetime(2016, 1, 1, 14, 20, 0) + (4 * 3600 + 15 * 60)

Salidas viernes, 1 de enero de 2016, 18:35:00 h.

Con una simple impresión usando ?, lo que ve en su pantalla depende de su configuración de fecha. Por ejemplo, si no ha cambiado nada, la configuración de la fecha es estilo americano (MDY), formato de 12 horas (AM/PM) y el siglo se muestra solo con los últimos 2 dígitos.

Hay un símbolo especial ^ para la fecha y las fechas y horas que obligan a que una cadena se interprete ’estrictamente’ como formato aaaa/MM/dd [HH:mm:ss|hh:mm:ss tt]. Por lo tanto, ^ también podría considerarse como un operador de fecha/fechahora. Por ejemplo, considere que algunos datos provienen de una fuente en un formato como 201610082230 (yyyyMMddHHmm). Para obtener ese valor como una fecha y hora válida:

Local cSample, tSample
cSample = '201610082230'
tSample = Ctot(Transform(m.cSample, '@R ^9999/99/99 99:99'))
? Transform(m.tSample, '@YL')

Salidas (dependiendo de la configuración de fecha larga de su sistema):

Sábado, 8 de octubre de 2016, 22:30:00

- se usa para la resta. Sus operandos son ambos valores de fecha/fechahora O uno es una fecha/fechahora y el otro es un valor numérico.

Comencemos con los operandos numéricos y de fecha/fechahora más simples (como con el operador +):

Cuando uno de los operandos es una fecha, el operando numérico se toma como “día”:

? Date() - 10 && What was the date 10 days ago?
? Date(2016, 3, 1) - 1 && Returns Feb 29, 2016.
? Date(2017, 3, 1) - 1 && Returns Feb 28, 2017.

Cuando uno de los operandos es una fecha y hora, el operando numérico se toma como “segundo”:

? Datetime() - 86400 && Go back exactly one day

Obtener hace 1 hora y 30 minutos desde “ahora”:

? Datetime() - (1 * 3600 + 30 * 60)

La segunda forma es obtener la diferencia entre dos valores de fecha/fechahora. Los operandos son tanto de fecha como de fecha y hora, no puede usar la fecha y la fecha y hora al mismo tiempo (escriba la conversión según sea necesario, VFP no lo hace por usted). Las reglas son como + y -, los operandos son fecha y luego la diferencia es en días, los operandos son fechahora y luego la diferencia es en segundos.

¿Cuántos días faltan para la víspera de año nuevo (para el año 2016)?

? Date(2016, 12, 31) - Date()

¿Cuántos segundos faltan para la medianoche?

? Dtot(Date()+1) - Datetime()

En la última muestra, usamos una función de fecha/fecha y hora, DTOT - DateToTime para obtener el valor de la medianoche de mañana. Hay muchas funciones útiles de fecha/fechahora disponibles, las omitimos todas ya que técnicamente no se consideran operadores (aunque operan en fecha/fechahoras :) Lo mismo ocurre con otros operadores.

La resta de fecha/fechahora está firmada. Es decir, si usa la fecha/fecha y hora más pequeña como primer operando, el resultado sería negativo. Puede usar la función abs() si necesita obtener un resultado positivo independientemente del orden de fecha/fechahora.

Operadores relacionales #

De todos los operadores, los operadores relacionales son los más complejos, por eso los dejamos para el final.

Los operadores relacionales también se conocen como operadores de comparación, se utilizan para comparar cosas.

El resultado de la comparación es booleano falso o verdadero.

Curiosamente, sin embargo, si lo comprueba en la ayuda de VFP, solo verá un breve operaciones de lista y algunas líneas más como si se tratara de esas operadores.

Bueno, la complejidad proviene del hecho de que operan en cualquier tipo, ya sea numérico, de fecha, de fecha y hora, lógico o de cadena, e incluso en objetos. Además, el comportamiento puede parecer incómodo, no obtienes lo que esperas a menos que sepas qué efectos tiene en los resultados.

Comencemos con una lista de operadores relacionales:

Operador Descripción MOST Muestra básica
> Mayor que ? 1 > 2 && .F.
< Menos que ? 1 < 2 && .T.
>= Mayor o igual que ? 1 >= 2 && .F.
<= Menor o igual que ? 1 <= 2 && .T.
= Igual a ? 1 = 1 && .T.
== Es exactamente igual a (tiene sentido para cadenas) ? ‘1’ = ‘1’ && .T.
!=, #, <> No es igual a (los 3 operadores actúan de la misma manera, elija su favorito) ? 1 != 1 && .F.

Aunque puede usarlos con todos los tipos de datos, debe haber una compatibilidad de tipos entre los operandos. Por ejemplo, obtendrá un error si intenta comparar una fecha con un número entero.

La fecha y la fecha y la hora se pueden comparar, aunque son tipos diferentes, VFP realiza la conversión implícitamente por usted.

? Date() > DateTime() && .F. 
? Date() <= DateTime() && .T. 
? Date() < DateTime() && .T. if it is not midnight

Cuando los operandos son numéricos, todos estos operadores son simples y directos, funcionan como lo harían en una expresión matemática.

Con los operandos lógicos, .F. se considera inferior a .T.

Con objetos lo que estamos comparando es la referencia del objeto en memoria. Por lo tanto, la comparación más utilizada es determinar si dos variables de objeto apuntan al mismo objeto. es decir:

local o1, o2
o1 = createobject('Label')
o2 = createobject('Label')
? m.o1 = m.o2 && is o1 and o2 the same object?
? m.o1 > m.o2 && this would work too but likely you would never use

* remember we are comparing their references in memory
* 
* They are different objects, but do they have any difference in their properties?
? CompObj(m.o1, m.o2) && .T. They are identical properties wise

La comparación de tipos de datos de caracteres, también conocida como comparación de cadenas, es la más confusa en VFP. No funciona como en otros lenguajes y/o bases de datos y es exclusivo de VFP (y tal vez de algún otro lenguaje xBase).

Hace muchos años, incluso he visto a algunos miembros realmente avanzados en la comunidad que aún no sabían cómo funcionan estos operadores en VFP. Por lo tanto, es bastante comprensible que los pequeños matices puedan confundir fácilmente a los novatos.

La comparación se trata básicamente de ser igual o no. Si no son iguales, entonces podríamos pensar en los operadores >, <, >=, <=, ¿no? Con cadenas es confuso cuando dos cadenas se consideran iguales.

Importante: Las cadenas VFP distinguen entre mayúsculas y minúsculas. ‘A’ y ‘a’ son dos cadenas distintas. Este no es el caso con muchas bases de datos donde el valor predeterminado es usar una intercalación que no distingue entre mayúsculas y minúsculas. Por ejemplo, en postgreSQL o MS SQL Server en una tabla creada con una intercalación que no distingue entre mayúsculas y minúsculas (CI):

select * from myTable where Country = 'TURKEY'

select * from myTable where Country = 'Turkey'

daría el mismo resultado. En VFP, aunque solo obtienes aquellos en los que la carcasa coincide. Sin embargo, VFP tiene cierto soporte de intercalación y hace una comparación que no distingue entre mayúsculas y minúsculas. (No confíes, ver más abajo)

  • Si dos cadenas no son iguales, hasta ahora todo bien, siempre que no haya cambiado ningún valor predeterminado, entonces se comparan según sus valores ASCII.

    ? 'Basoz' < 'Cetin' && is true.
? 'basoz' < 'Cetin' && is false.
? 'Cetin' < 'David' && is true.
? 'Çetin' < 'David' && is false.

El valor predeterminado para la intercalación es ‘máquina’ y esto es lo que obtienes. Cuando cambia la intercalación a otra cosa, obtiene la comparación basada en el orden de clasificación de esa intercalación. Con una configuración de intercalación diferente a la máquina predeterminada, también está implicando una insensibilidad a mayúsculas y minúsculas en comparación (NO confíe en esto para la igualdad):

  set collate to 'GENERAL'
  ? 'Basoz' < 'Cetin'
  ? 'basoz' < 'Cetin'
  ? 'Cetin' < 'David'
  ? 'Çetin' < 'David'

Ahora todas estas expresiones son VERDADERAS.

Consejos personales: Las intercalaciones en VFP nunca han sido lo suficientemente confiables. Le sugiero que no use intercalaciones y se quede con la ‘MÁQUINA’ predeterminada. Si usaría intercalaciones, recuerde verificarlo primero cuando experimente algo que sea muy inesperado con respecto a los datos de caracteres. He visto y demostrado que falla en muchos casos, pero luego dejé de intentar usarlo mucho antes de la versión VFP9, podría ser consistente ahora, realmente no lo sé.

Teniendo en cuenta que cubrimos casos de desigualdad con cadenas, el complicado es el caso de igualdad. En VFP básicamente dos configuraciones afectan la comparación:

  1. ESTABLECER EXACTO (el valor predeterminado es APAGADO y afecta las comparaciones regulares, excepto SQL)
  2. SET ANSI (el valor predeterminado es OFF y afecta las comparaciones solo en SQL. SET EXACT no tiene efecto en las comparaciones realizadas dentro de las consultas SQL.

Con SET EXACT OFF, lea la comparación como “¿la cadena de la derecha comienza con la cadena de la izquierda”? Se comparan hasta la longitud correcta de la cadena.

? "Bobby" = "B" && Bobby starts with B, so TRUE
? "Bobby" = "Bob" && Bobby starts with Bob, so TRUE
? "Bobby" = "Bob " && Bobby starts with Bob but there is a trailing space there, FALSE
? "Bobby" = "bob" && would be true with collation set to GENERAL

Tenga en cuenta que con la comparación regular, “Bobby” = “B” es VERDADERO, pero “B” = “Bobby” es FALSO. En otras palabras, el lugar de los operandos es importante.

Con SET EXACT ON, las cadenas deben coincidir completamente, pero sus espacios finales se ignoran (estamos ignorando set collate aquí, lo que también haría insensibilidad a mayúsculas y minúsculas):

? "BOBBY" = "BOB" && FALSE 
? "BOBBY" = "BOBBY" && TRUE
? "BOBBY" = "BOBBY     " && TRUE 
? "BOBBY     " = "BOBBY" && TRUE

Ahora, con los comandos SQL, SET EXACT no tiene efecto y se comportaría como lo hace SET EXACT OFF.

Select * from Customers where Country = 'U'

Seleccionaría a los clientes de EE. UU., Reino Unido, cualquier país que comience con ‘U’.

Sin embargo, en SQL, por definición, cambiar el orden de los operandos debería producir el mismo resultado. De este modo:

Select * from Customers where 'U' = Country

también funcionaría de la misma manera (tenga en cuenta la diferencia con los comandos que no son SQL).

Cuando desee implicar coincidencias exactas, una opción es activar ANSI:

SET ANSI ON
Select * from Customers where Country = 'USA'

devuelve todos aquellos clientes de USA. Tenga en cuenta que se ignoran los espacios finales en el campo de país O en la expresión de la derecha. No importaría cuántos arrastres a cada lado tengas. Obtienes la comparación como si se hubiera hecho así: RTRIM(País) = RTRIM(‘USA’).

Aunque no se menciona en Operadores en VFP, un operador SQL es LIKE. Cuando usa LIKE, obtiene una comparación de coincidencia exacta independientemente de la configuración SET ANSI (usando las fuerzas LIKE y el caso implícito ANSI ON; después de todo, es un operador ANSI). Sin embargo, tenga cuidado, hay una ligera diferencia en el comportamiento. No ignoraría los espacios finales, a menos que el tamaño total con los finales sea igual o menor que el tamaño del campo. Por ejemplo, si el campo País es C(10), entonces País = ‘EE.UU.’ o País = ‘EE.UU.’ funcionaría, pero País = ‘EE.UU._________’ fallaría (guiones bajos indican un espacio y el último tiene más de 7 espacios finales ).

Por fin llegamos al último operador, ==. Eso significa exactamente igual y hace que se use con cadenas. Una ventaja es que, al usar ==, siempre quiere decir que desea una coincidencia exacta independientemente de las configuraciones SET EXACT o SET ANSI. Sin embargo, tenga cuidado de nuevo, su comportamiento es diferente cuando se trata de un comando SQL o un comando normal no SQL.

Con SQL:

Select * from Customers where Country == 'USA'

cualquiera que sea la configuración ANSI y EXACTA, queremos que todos los clientes sean solo de EE. UU. Se ignoran los espacios finales a cada lado.

Con no SQL:

? m.lcString1 == m.lcString2

sería cierto solo si son exactamente iguales, con respecto a su carcasa y longitud (los espacios finales NO se ignoran). No se efectúa desde la configuración SET ANSI, EXACT o COLLATE.
