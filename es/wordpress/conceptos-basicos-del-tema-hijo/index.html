Sintaxis #

  • plantilla –– es el nombre del tema principal de WordPress, el padre.
  • child-theme –– es el paquete que anula la plantilla.

He estado anunciando que el uso de un tema infantil siempre es algo bueno, pero siempre hay un Pero…

En nuestro ejemplo de sobrescritura de plantillas, imaginemos que el autor de un tema está agregando sus propias mejoras a la plantilla de la barra lateral y habrá una nueva en

/temas/plantilla/barra lateral.php

<?php
/**
 * The template for the sidebar containing the main widget area
 *
 * @link https://developer.wordpress.org/themes/basics/template-files/#template-partials
 */

if ( is_active_sidebar( 'sidebar-1' )  ) : ?>
    <aside id="secondary" class="sidebar widget-area" role="complementary">
        <?php dynamic_sidebar( 'sidebar-1' ); ?>
    </aside><!-- .sidebar .widget-area -->
<?php endif; ?>

Ahora nuestro sitio web no se beneficiará de la nueva especificación role="complementary" porque nuestro tema secundario aún sobrescribe la plantilla con su propio archivo en /temas/hijo-tema/barra lateral.php

Es nuestro deber como encargados del mantenimiento del sitio web realizar un seguimiento de las plantillas que sobrescribimos y, en el caso inminente de una actualización, mirar detenidamente el registro de cambios para actualizar los archivos del tema secundario si es necesario.

2) El propósito #

Los temas secundarios están destinados a ser una forma segura de mantener las personalizaciones de la plantilla principal sin temor a perderlas en una actualización del tema.

Básicamente, cada vez que desee editar un archivo dentro de la plantilla activa de su sitio web, debe preguntarse “** ¿Qué sucederá cuando actualice el tema? **

Y la respuesta es simple: Los pierdes porque la actualización traerá una carpeta de temas completamente nueva.

Entonces, veamos un tema secundario como una carpeta con archivos que sobrescribirán los archivos con la misma ruta en el tema principal. Ahora vamos a traer algunos ejemplos reales:

Definición y requisitos #

WordPress identifica un tema secundario cuando hay un directorio (por ejemplo, child-theme) dentro de /wp-content/themes/ con los siguientes archivos:

  • estilo.css

    This file must start with a comment template like this:

      /*
  Theme Name: Example Child
  Author: Me
  Author URI: https://example.com/
  Template: example
  Text Domain: example-child-theme
  Domain Path: /languages/
  */

    The most important things to consider here are:

    - The `Template` name must be exactly the folder name which holds the parent theme (aka the parent theme slug)

- Name your child theme in such a way you can easily identify it in Dashboard (usually just append `Child` to the parent's name, like `Example Child`)

  • index.php

  • funciones.php

3) Sobrescritura de plantilla #

El uso más común de un tema hijo es anular las partes de la plantilla. Por ejemplo, una barra lateral, si tenemos un tema con el siguiente archivo en

/temas/plantilla/barra lateral.php

<?php
/**
 * The sidebar containing the main widget area.
 *
 * @link https://developer.wordpress.org/themes/basics/template-files/#template-partials
 */ 

if ( ! is_active_sidebar( 'sidebar-1' ) ) {
    return;
}?>
<div id="sidebar" class="widget-area">
    <?php dynamic_sidebar( 'sidebar-1' ); ?>
</div>

Definitivamente podemos agregar nuestro propio archivo en el tema secundario (con las especificaciones del primer ejemplo) con el siguiente archivo

/temas/hijo-tema/barra lateral.php

<?php
/**
 * The sidebar containing the main widget area.
 */ 

if ( ! is_active_sidebar( 'sidebar-1' ) ) {
    return;
}?>
<div id="my-sidebar" class="my-own-awesome-class widget-area">
    <div class="my-wrapper">
        <?php dynamic_sidebar( 'sidebar-1' ); ?>
    </div>
</div>

Ahora my-own-awesome-class está a salvo en tema secundario y no se eliminará en ninguna actualización de tema y WordPress siempre elegirá una plantilla de los temas secundarios cuando encuentre una en la misma ruta.

Sustitución de activos #

Incluso si no es una práctica recomendada, a veces es necesario reemplazar activos como bibliotecas o archivos CSS o JS.

Tenga en cuenta que el sistema de sobrescritura de plantillas de WordPress no funciona con nada más que archivos .php, por lo que cuando hablamos de activos nos referimos a activos registrados

Un ejemplo podría ser el reemplazo de la biblioteca jQuery con su versión deseada. En el archivo functions.php de nuestro tema secundario, debemos agregar una función que elimine la versión actual de jQuery y agregue la nuestra desde CDN (recuerde que es solo un ejemplo).

/**
 * Dequeue the jQuery script and add our own version.
 *
 * Hooked to the wp_print_scripts action, with a late priority (100),
 * so that it is after the script was enqueued.
 */
function my_own_theme_scripts() {
    // remove the current version
    wp_dequeue_script( 'jquery' );
    // register my desired version
    wp_register_script( 'jquery', 'https://code.jquery.com/jquery-3.1.0.min.js', false, '3.1.0' );
    // load my version, here or somewhere else
    wp_enqueue_script( 'jquery' );
}
add_action( 'wp_print_scripts', 'my_own_theme_scripts' );
