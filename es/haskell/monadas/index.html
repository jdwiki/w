Una mónada es un tipo de datos de acciones componibles. Monad es la clase de constructores de tipos cuyos valores representan tales acciones. Quizás IO es el más reconocible: un valor de IO a es una “receta para recuperar un valor a del mundo real”.

Decimos que un constructor de tipos m (como [] o Maybe) forma una mónada si hay una instancia Mónada m que satisface ciertas leyes sobre la composición de acciones. Entonces podemos razonar sobre m a como una “acción cuyo resultado tiene el tipo a”.

No hay forma general de extraer valor de un cálculo monádico #

Puede envolver valores en acciones y canalizar el resultado de un cálculo a otro:

return :: Monad m => a -> m a
(>>=)  :: Monad m => m a -> (a -> m b) -> m b

Sin embargo, la definición de Mónada no garantiza la existencia de una función de tipo Monad m => m a -> a.

Eso significa que, en general, no hay forma de extraer un valor de un cálculo (es decir, “desenvolverlo”). Este es el caso de muchos casos:

extract :: Maybe a -> a
extract (Just x) = x          -- Sure, this works, but...
extract Nothing  = undefined  -- We can’t extract a value from failure.

Específicamente, no existe la función IO a -> a, lo que a menudo confunde a los principiantes; consulte [este ejemplo] (https://www.wikiod.com/es/haskell/monadas#mónada IO).

Mónada como subclase de aplicativo #

A partir de GHC 7.10, Applicative es una superclase de Monad (es decir, cada tipo que sea una Monad también debe ser un Applicative). Todos los métodos de Applicative (pure, <*>) se pueden implementar en términos de métodos de Monad (return, >>=).

Es obvio que pure y return tienen propósitos equivalentes, por lo que pure = return. La definición de <*> es relativamente clara:

mf <*> mx = do { f <- mf; x <- mx; return (f x) }                 
       -- = mf >>= (\f -> mx >>= (\x -> return (f x)))
       -- = [r   | f <- mf, x <- mx, r <- return (f x)]   -- with MonadComprehensions
       -- = [f x | f <- mf, x <- mx]

Esta función se define como ap en las bibliotecas estándar.

Por lo tanto, si ya ha definido una instancia de Monad para un tipo, puede obtener una instancia de Applicative “gratis” definiendo

instance Applicative < type > where
    pure  = return
    (<*>) = ap

Al igual que con las leyes de las mónadas, estas equivalencias no se aplican, pero los desarrolladores deben asegurarse de que siempre se respeten.

Definición de Mónada #

class Monad m where
    return :: a -> m a
    (>>=) :: m a -> (a -> m b) -> m b

La función más importante para tratar con mónadas es el operador bind >>=:

(>>=) :: m a -> (a -> m b) -> m b
  • Piensa en m a como “una acción con un resultado a”.
  • Piense en a -> m b como “una acción (dependiendo de un parámetro a) con un resultado b.”.

>>= secuencia dos acciones juntas canalizando el resultado de la primera acción a la segunda.

La otra función definida por Monad es:

return :: a -> m a

Su nombre es desafortunado: este return no tiene nada que ver con la palabra clave return que se encuentra en los lenguajes de programación imperativos.

return x es la acción trivial que produce x como resultado. (Es trivial en el [siguiente sentido](https://www.wikiod.com/es/haskell/monadas#La mónada quizás -leyes-y-la-quizás-mónada):)

return x >>= f       ≡  f x     --  “left identity” monad law
       x >>= return  ≡  x       -- “right identity” monad law

La mónada Quizás #

Maybe se usa para representar valores posiblemente vacíos, similar a null en otros idiomas. Por lo general, se usa como el tipo de salida de funciones que pueden fallar de alguna manera.

Considere la siguiente función:

halve :: Int -> Maybe Int
halve x
  | even x = Just (x `div` 2)
  | odd x  = Nothing

Piense en reducir a la mitad como una acción, que depende de un Int, que intenta reducir a la mitad el número entero, fallando si es impar.

¿Cómo ‘reducimos a la mitad’ un número entero tres veces?

takeOneEighth :: Int -> Maybe Int            -- (after you read the 'do' sub-section:)
takeOneEighth x =                
  case halve x of                               --  do {
    Nothing -> Nothing
    Just oneHalf ->                             --     oneHalf    <- halve x
      case halve oneHalf of
        Nothing -> Nothing
        Just oneQuarter ->                      --     oneQuarter <- halve oneHalf
          case halve oneQuarter of
            Nothing -> Nothing                  --     oneEighth  <- halve oneQuarter
            Just oneEighth ->                         
              Just oneEighth                    --     return oneEighth }
  • takeOneEighth es una secuencia de tres pasos halve encadenados.
  • Si falla un paso halve, queremos que falle toda la composición takeOneEighth.
  • Si un paso halve tiene éxito, queremos canalizar su resultado hacia adelante.

instance Monad Maybe where
  -- (>>=) :: Maybe a -> (a -> Maybe b) -> Maybe b
  Nothing >>= f  = Nothing                            -- infixl 1 >>=
  Just x  >>= f  = Just (f x)                         -- also, f =<< m = m >>= f
  
  -- return :: a -> Maybe a
  return x       = Just x

y ahora podemos escribir:

takeOneEighth :: Int -> Maybe Int
takeOneEighth x = halve x >>= halve >>= halve             -- or,
    -- return x >>= halve >>= halve >>= halve             -- which is parsed as
    -- (((return x) >>= halve) >>= halve) >>= halve       -- which can also be written as
    -- (halve =<<) . (halve =<<) . (halve =<<) $ return x    -- or, equivalently, as
    --  halve <=<     halve <=<     halve      $        x

Composición Kleisli <=< se define como (g <=< f) x = g =<< f x, o equivalentemente como (f >=> g) x = f x >>= g. Con él, la definición anterior se convierte en solo

takeOneEighth :: Int -> Maybe Int
takeOneEighth = halve <=< halve <=< halve               -- infixr 1 <=<
        -- or, equivalently,                    
        --      halve >=> halve >=> halve               -- infixr 1 >=>

Hay tres leyes de mónadas que deben ser obedecidas por cada mónada, es decir, cada tipo que es una instancia de la clase de tipos ‘Monad’:

1.  return x >>= f  =  f x
2.    m >>= return  =  m
3. (m >>= g) >>= h  =  m >>= (\y -> g y >>= h)

donde m es una mónada, f tiene el tipo a -> m b y g tiene el tipo b -> m c.

O de manera equivalente, utilizando el operador de composición de Kleisli >=> definido anteriormente:

1.    return >=> g  =  g                    -- do { y <- return x ; g y } == g x
2.    f >=> return  =  f                    -- do { y <- f x ; return y } == f x
3. (f >=> g) >=> h  =  f >=> (g >=> h)      -- do { z <- do { y <- f x; g y } ; h z }
                                            --  == do { y <- f x ; do { z <- g y; h z } }

Obedecer estas leyes hace que sea mucho más fácil razonar sobre la mónada, porque garantiza que usar funciones monádicas y componerlas se comporte de manera razonable, similar a otras mónadas.

Verifiquemos si la mónada ‘Quizás’ obedece las tres leyes de la mónada.

  1. La ley de identidad izquierda - return x >>= f = f x

    return z »= f = (Just z) »= f = f z

  2. La ley de identidad correcta - m >>= return = m

  • Solo constructor de datos

    Just z »= return = return z = Just z

  • Constructor de datos Nada

    Nothing »= return = Nothing

  1. La ley de asociatividad - (m >>= f) >>= g = m >>= (\x -> f x >>= g)

  • Solo constructor de datos

    – Left-hand side ((Just z) »= f) »= g = f z »= g

    – Right-hand side (Just z) »= (\x -> f x »= g) (\x -> f x »= g) z = f z »= g

  • Constructor de datos Nada

    – Left-hand side (Nothing »= f) »= g = Nothing »= g = Nothing

    – Right-hand side Nothing »= (\x -> f x »= g) = Nothing

Yo mónada #

No hay forma de obtener un valor de tipo a de una expresión de tipo IO a y no debería haberla. Esto es en realidad una gran parte de por qué las mónadas se usan para modelar ‘IO’.

Se puede considerar que una expresión de tipo IO a representa una acción que puede interactuar con el mundo real y, si se ejecuta, daría como resultado algo de tipo a. Por ejemplo, la función getLine :: IO String del preludio no significa que debajo de getLine haya una cadena específica que pueda extraer; significa que getLine representa la acción de obtener una línea de la entrada estándar .

No es sorprendente que main :: IO () ya que un programa de Haskell representa un cálculo/acción que interactúa con el mundo real.

Las cosas que puedes hacer con las expresiones de tipo IO a porque IO es una mónada:

  • Secuenciar dos acciones usando (>>) para producir una nueva acción que ejecuta la primera acción, descarta cualquier valor que haya producido y luego ejecuta la segunda acción.

       -- print the lines "Hello" then "World" to stdout
   putStrLn "Hello" >> putStrLn "World"

  • A veces, no desea descartar el valor que se produjo en la primera acción; en realidad, le gustaría que se introdujera en una segunda acción. Para eso, tenemos >>=. Para IO, tiene el tipo (>>=) :: IO a -> (a -> IO b) -> IO b.

      -- get a line from stdin and print it back out
  getLine >>= putStrLn

  • Tome un valor normal y conviértalo en una acción que devuelva inmediatamente el valor que le dio. Esta función es obviamente menos útil hasta que comience a usar la notación do.

      -- make an action that just returns 5
  return 5

Más de Haskell Wiki sobre la mónada IO aquí.

Lista Mónada #

Las listas forman una mónada. Tienen una instanciación de mónada equivalente a esta:

instance Monad [] where 
  return x = [x]
  xs >>= f = concat (map f xs)

Podemos usarlos para emular el no determinismo en nuestros cálculos. Cuando usamos xs >>= f, la función f :: a -> [b] se mapea sobre la lista xs, obteniendo una lista de listas de resultados de cada aplicación de f sobre cada elemento de xs, y todas las listas de resultados se concatenan en una lista de todos los resultados. Como ejemplo, calculamos una suma de dos números no deterministas usando notación de dotación, la suma está representada por una lista de sumas de todos los pares de enteros de dos listas, cada lista representa todos los valores posibles de un número no determinista. número determinista:

sumnd xs ys = do
  x <- xs
  y <- ys
  return (x + y)

O de manera equivalente, usando liftM2 en Control.Monad:

sumnd = liftM2 (+)

obtenemos:

> sumnd [1,2,3] [0,10]
[1,11,2,12,3,13]

notación do #

La notación do es un azúcar sintáctico para las mónadas. Estas son las reglas:

hacer x <- mx hacer x <- mx y <- my es equivalente a do y <- my ... ... sea ​​a = b sea a = b en ... es equivalente a hacer ... hacer m m >> ( e es equivalente a e) hacer x <- m m >>= (\x -> e es equivalente a e) do m es equivalente a m

Por ejemplo, estas definiciones son equivalentes:

example :: IO Integer
example =
  putStrLn "What's your name?" >> (
    getLine >>= (\name ->
      putStrLn ("Hello, " ++ name ++ ".") >> (
        putStrLn "What should we return?" >> (
          getLine >>= (\line ->
            let n = (read line :: Integer) in
              return (n + n))))))

example :: IO Integer
example = do
  putStrLn "What's your name?"
  name <- getLine
  putStrLn ("Hello, " ++ name ++ ".")
  putStrLn "What should we return?"
  line <- getLine
  let n = (read line :: Integer)
  return (n + n)
