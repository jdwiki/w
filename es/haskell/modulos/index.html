Sintaxis #

  • nombre del módulo donde – exportar todos los nombres declarados en este archivo

  • nombre del módulo (functionOne, Type (..)) donde – exporta solo los constructores de functionOne, Type y Type

  • Módulo de importación: importe todos los nombres exportados del módulo

  • importar módulo calificado como MN – importación calificada

  • Módulo de importación (justThisFunction): importa solo ciertos nombres de un módulo

  • Ocultación de módulo de importación (nombre de función, tipo) – importa todos los nombres de un módulo excepto nombre de función y tipo

Haskell tiene soporte para módulos:

  • un módulo puede exportar todos o un subconjunto de sus tipos y funciones de miembros

  • un módulo puede “reexportar” los nombres que importó de otros módulos

En el extremo del consumidor de un módulo, uno puede:

  • importar todos o un subconjunto de miembros del módulo

  • ocultar las importaciones de un miembro o conjunto de miembros en particular

haskell.org tiene un gran capítulo sobre la definición de módulos.

Definición de su propio módulo #

Si tenemos un archivo llamado Business.hs, podemos definir un módulo Business que se pueda importar, así:

module Business (
    Person (..), -- ^ Export the Person type and all its constructors and field names
    employees   -- ^ Export the employees function
) where
-- begin types, function definitions, etc

Por supuesto, es posible una jerarquía más profunda; consulte el ejemplo Nombres de módulos jerárquicos.

1: https://www.wikiod.com/es/haskell/modulos#Nombres de módulos jerárquicos

Exportación de constructores #

Para exportar el tipo y todos sus constructores, se debe usar la siguiente sintaxis:

module X (Person (..)) where

Entonces, para las siguientes definiciones de nivel superior en un archivo llamado People.hs:

data Person = Friend String | Foe deriving (Show, Eq, Ord)

isFoe Foe = True
isFoe _   = False

Esta declaración de módulo en la parte superior:

module People (Person (..)) where

solo exportaría ‘Persona’ y sus constructores ‘Amigo’ y ‘Enemigo’.

Si se omite la lista de exportación que sigue a la palabra clave del módulo, se exportarán todos los nombres enlazados en el nivel superior del módulo:

module People where

exportaría Person, sus constructores y la función isFoe.

Importación de miembros específicos de un módulo #

Haskell admite la importación de un subconjunto de elementos de un módulo.

import qualified Data.Stream (map) as D

solo importaría mapa desde Data.Stream, y las llamadas a esta función requerirían D.:

D.map odd [1..]

de lo contrario, el compilador intentará usar la función map de Prelude.

Ocultar importaciones #

Prelude a menudo define funciones cuyos nombres se utilizan en otros lugares. No ocultar tales importaciones (o usar importaciones calificadas donde ocurren conflictos) provocará errores de compilación.

Data.Stream define funciones denominadas map, head y tail que normalmente chocan con las definidas en Prelude. Podemos ocultar esas importaciones de Prelude usando hiding:

import Data.Stream -- everything from Data.Stream
import Prelude hiding (map, head, tail, scan, foldl, foldr, filter, dropWhile, take) -- etc

En realidad, se necesitaría demasiado código para ocultar conflictos de Prelude como este, por lo que, de hecho, usaría una importación calificada de Data.Stream en su lugar.

Importaciones calificadas #

Cuando múltiples módulos definen las mismas funciones por nombre, el compilador se quejará. En tales casos (o para mejorar la legibilidad), podemos usar una importación cualificada:

import qualified Data.Stream as D

Ahora podemos evitar errores de compilación de ambigüedad cuando usamos map, que se define en Prelude y Data.Stream:

map (== 1) [1,2,3] -- will use Prelude.map
D.map (odd) (fromList [1..]) -- will use Data.Stream.map

También es posible importar un módulo con solo los nombres en conflicto calificados a través de importar datos.Texto como T, lo que permite tener Texto en lugar de T.Texto, etc.

Nombres de módulos jerárquicos #

Los nombres de los módulos siguen la estructura jerárquica del sistema de archivos. Con el siguiente diseño de archivo:

Foo/
├── Baz/
│   └── Quux.hs
└── Bar.hs
Foo.hs
Bar.hs

los encabezados de los módulos se verían así:

-- file Foo.hs
module Foo where

-- file Bar.hs
module Bar where

-- file Foo/Bar.hs
module Foo.Bar where

-- file Foo/Baz/Quux.hs
module Foo.Baz.Quux where

Tenga en cuenta que:

  • el nombre del módulo se basa en la ruta del archivo que declara el módulo
  • Las carpetas pueden compartir un nombre con un módulo, lo que da una estructura de nombres jerárquica natural a los módulos
