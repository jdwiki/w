Las clases de tipos en Haskell son un medio para definir el comportamiento asociado con un tipo por separado de la definición de ese tipo. Mientras que, digamos, en Java, definiría el comportamiento como parte de la definición del tipo, es decir, en una interfaz, una clase abstracta o una clase concreta, Haskell mantiene estas dos cosas separadas.

Hay una serie de clases de tipos ya definidas en el paquete base de Haskell. La relación entre estos se ilustra en la sección Comentarios a continuación.

El siguiente diagrama tomado del artículo Typeclassopedia muestra la relación entre las diversas clases de tipos en Haskell.

ecuación #

Todos los tipos de datos básicos (como Int, String, Eq a => [a]) de Prelude excepto las funciones y IO tienen instancias de Eq. Si un tipo instancia ‘Eq’, significa que sabemos cómo comparar dos valores para la igualdad valor o estructural.

> 3 == 2 
False
> 3 == 3
True

Métodos requeridos #

  • (==) :: Eq a => a -> a -> Boolean o (/=) :: Eq a => a -> a -> Boolean (si solo se implementa uno, el otro se establece por defecto a la negación del definido)

Define #

  • (==) :: Eq a => a -> a -> Boolean
  • (/=) :: Eq a => a -> a -> Boolean

Superclases directas #

Ninguna

Subclases notables #

  • Ord

pedido #

Los tipos que instancian ‘Ord’ incluyen, por ejemplo, ‘Int’, ‘String’ y ‘[a]’ (para los tipos ‘a’ donde hay una instancia de ‘Ord a’). Si un tipo instancia Ord, significa que conocemos un tipo “natural” ordenamiento de valores de ese tipo. Tenga en cuenta que a menudo hay muchas opciones posibles de lo “natural”; ordenar de un tipo y ‘Ord’ nos obliga a favorecer uno.

Ord proporciona los operadores estándar (<=), (<), (>), (>=) pero curiosamente los define a todos usando un tipo de datos algebraico personalizado

data Ordering = LT | EQ | GT

compare :: Ord a => a -> a -> Ordering

Métodos requeridos #

  • compare :: Ord a => a -> a -> Ordering o (<=) :: Ord a => a -> a -> Boolean (el método estándar de comparación utiliza ( <=) en su implementación)

Define #

  • comparar :: Ord a => a -> a -> Pedido
  • (<=) :: Ord a => a -> a -> Boolean
  • (<) :: Ord a => a -> a -> Boolean
  • (>=) :: Ord a => a -> a -> Boolean
  • (>) :: Ord a => a -> a -> Boolean
  • min :: Ord a => a -> a -> a
  • max :: Ord a => a -> a -> a

Superclases directas #

  • Eq

Monoide #

Los tipos que crean instancias de Monoid incluyen listas, números y funciones con valores de retorno de Monoid, entre otros. Para crear una instancia de Monoid, un tipo debe admitir una operación binaria asociativa (mappend o (<>)) que combina sus valores y tener un valor especial “cero” (mempty) tal que combinar un valor con él no cambia ese valor:

mempty  <>  x == x
x <>  mempty  == x

x <> (y <> z) == (x <> y) <> z

Intuitivamente, los tipos Monoid son “similares a una lista” en el sentido de que admiten agregar valores juntos. Alternativamente, los tipos ‘Monoid’ se pueden considerar como secuencias de valores para los que nos preocupamos por el orden pero no por la agrupación. Por ejemplo, un árbol binario es un ‘Monoide’, pero usando las operaciones ‘Monoide’ no podemos presenciar su estructura ramificada, solo un recorrido de sus valores (ver ‘Plegable’ y ‘Transversable’).

Métodos requeridos #

  • mempty :: Monoide m => m
  • mappend :: Monoide m => m -> m -> m

Superclases directas #

Ninguna

número #

La clase más general para tipos de números, más precisamente para anillos, es decir, números que se pueden sumar, restar y multiplicar en el sentido habitual, pero no necesariamente dividida.

Esta clase contiene tipos integrales (Int, Integer, Word32, etc.) y tipos fraccionarios (Double, Rational, también números complejos, etc.). En el caso de tipos finitos, la semántica generalmente se entiende como aritmética modular, es decir, con overflow y underflow†.

Tenga en cuenta que las reglas para las clases numéricas se obedecen mucho menos estrictamente que la [mónada] (https://www.wikiod.com/es/haskell/monadas#La mónada quizás =201610010024186706477) o leyes monoides, o las de comparación de igualdad. En particular, los números de punto flotante generalmente obedecen leyes solo en un sentido aproximado.

Los métodos #

  • fromInteger :: Num a => Integer -> a. convertir un número entero al tipo de número general (ajustando el rango, si es necesario). Haskell [literales numéricos] (https://www.wikiod.com/es/haskell/literales-sobrecargados#número entero) puede entenderse como un literal ‘Integer’ monomórfico con la conversión general a su alrededor, por lo que puede use el literal 5 tanto en un contexto Int como en una configuración Complex Double.

  • (+) :: Num a => a -> a -> a. Adición estándar, generalmente entendida como asociativa y conmutativa, es decir,

      a + (b + c) ≡ (a + b) + c
  a + b ≡ b + a

  • (-) :: Num a => a -> a -> a. La resta, que es la inversa de la suma:

      (a - b) + b ≡ (a + b) - b ≡ a

  • (*) :: Num a => a -> a -> a. Multiplicación, una operación asociativa que es distributiva sobre la suma:

      a * (b * c) ≡ (a * b) * c
  a * (b + c) ≡ a * b + a * c

para los casos más comunes, la multiplicación también es conmutativa, pero esto definitivamente no es un requisito.

  • negar :: Num a => a -> a. El nombre completo del operador de negación unario. -1 es azúcar sintáctico para negate 1.

      -a ≡ negate a ≡ 0 - a

  • abs :: Num a => a -> a. La función de valor absoluto siempre da un resultado no negativo de la misma magnitud

      abs (-a) ≡ abs a
  abs (abs a) ≡ abs a

abs a ≡ 0 solo debería ocurrir si a ≡ 0.

Para los tipos real está claro lo que significa no negativo: siempre tienes abs a >= 0. Los tipos complejos, etc. no tienen un orden bien definido, sin embargo, el resultado de abs siempre debe estar en el subconjunto real‡ (es decir, dar un número que también podría escribirse como un solo número literal sin negación).

  • signum :: Num a => a -> a. La función de signo, según el nombre, produce solo -1 o 1, según el signo del argumento. En realidad, eso solo es cierto para números reales distintos de cero; en general signum se entiende mejor como la función normalizadora:

      abs (signum a) ≡ 1   -- unless a≡0
  signum a * abs a ≡ a -- This is required to be true for all Num instances

Tenga en cuenta que [la sección 6.4.4 del Informe Haskell 2010] (https://www.haskell.org/onlinereport/haskell2010/haskellch6.html#x13-1390006.4.4) requiere explícitamente que esta última igualdad se cumpla para cualquier Num instancia.

Algunas bibliotecas, en particular linear y hmatrix, tienen una comprensión mucho más laxa de lo que La clase Num es para: la tratan como una forma de sobrecargar los operadores aritméticos. Si bien esto es bastante sencillo para + y -, ya se vuelve problemático con * y más aún con los otros métodos. Por ejemplo, ¿debería significar * multiplicación de matrices o multiplicación por elementos?
Podría decirse que es una mala idea definir tales instancias no numéricas; considere clases dedicadas como VectorSpace.

† En particular, los “negativos” de los tipos sin firmar se envuelven en un positivo grande, p. (-4 :: Palabra32) == 4294967292.

‡ Esto no se cumple ampliamente: los tipos de vectores no tienen un subconjunto real. Las controvertidas instancias de Num para tales tipos generalmente definen abs y signum en términos de elementos, lo que matemáticamente hablando no tiene mucho sentido.

Quizás y la clase Funtor #

En Haskell, los tipos de datos pueden tener argumentos al igual que las funciones. Tome el tipo Quizás por ejemplo.

Quizás es un tipo muy útil que nos permite representar la idea de fracaso, o la posibilidad del mismo. En otras palabras, si existe la posibilidad de que un cálculo falle, usamos el tipo Quizás allí. Maybe actúa como una especie de envoltorio para otros tipos, dándoles una funcionalidad adicional.

Su declaración real es bastante simple.

Maybe a = Just a | Nothing

Lo que esto dice es que un ‘Quizás’ viene en dos formas, ‘Solo’, que representa el éxito, y ‘Nada’, que representa el fracaso. ‘Solo’ toma un argumento que determina el tipo de ‘Quizás’, y ‘Nada’ no toma ninguno. Por ejemplo, el valor Just "foo" tendrá el tipo Maybe String, que es un tipo de cadena envuelto con la funcionalidad adicional Maybe. El valor ‘Nada’ tiene el tipo ‘Quizás a’, donde ‘a’ puede ser de cualquier tipo.

Esta idea de envolver tipos para darles una funcionalidad adicional es muy útil, y es aplicable a algo más que ‘Quizás’. Otros ejemplos incluyen los tipos Either, IO y list, cada uno de los cuales proporciona una funcionalidad diferente. Sin embargo, hay algunas acciones y habilidades que son comunes a todos estos tipos de envoltorios. El más notable de ellos es la capacidad de modificar el valor encapsulado.

Es común pensar en este tipo de tipos como cajas en las que se pueden colocar valores. Los diferentes cuadros tienen diferentes valores y hacen cosas diferentes, pero ninguno es útil sin poder acceder a los contenidos dentro.

Para resumir esta idea, Haskell viene con una clase de tipos estándar, llamada Functor. Se define de la siguiente manera.

class Functor f where
  fmap :: (a -> b) -> f a -> f b

Como puede verse, la clase tiene una única función, fmap, de dos argumentos. El primer argumento es una función de un tipo, a, a otro, b. El segundo argumento es un functor (tipo contenedor) que contiene un valor de tipo a. Devuelve un funtor (tipo contenedor) que contiene un valor de tipo b.

En términos simples, fmap toma una función y la aplica al valor dentro de un funtor. Es la única función necesaria para que un tipo sea miembro de la clase Functor, pero es extremadamente útil. Las funciones que operan en funtores que tienen aplicaciones más específicas se pueden encontrar en las clases de tipos Applicative y Monad.

Herencia de clase de tipo: clase de tipo Ord #

Haskell apoya una noción de extensión de clase. Por ejemplo, la clase ‘Ord’ hereda todas las operaciones en ‘Eq’, pero además tiene una función ‘comparar’ que devuelve una ‘Ordenación’ entre valores. Ord también puede contener los operadores de comparación de orden comunes, así como un método min y un método max.

La notación => tiene el mismo significado que en la firma de una función y requiere el tipo a para implementar Eq, para implementar Ord.

data Ordering = EQ | LT | GT

class Eq a => Ord a where
    compare :: Ord a => a -> a -> Ordering
    (<)     :: Ord a => a -> a -> Bool
    (<=)    :: Ord a => a -> a -> Bool
    (>)     :: Ord a => a -> a -> Bool
    (>=)    :: Ord a => a -> a -> Bool
    min     :: Ord a => a -> a -> a
    max     :: Ord a => a -> a -> a

Todos los métodos que siguen a compare se pueden derivar de él de varias maneras:

x < y   = compare x y == LT
x <= y  = x < y || x == y -- Note the use of (==) inherited from Eq
x > y   = not (x <= y)
x >= y  = not (x < y)

min x y = case compare x y of
               EQ -> x
               LT -> x
               GT -> y

max x y = case compare x y of
               EQ -> x
               LT -> y
               GT -> x

Las clases de tipos que amplían Ord deben implementar al menos el método compare o el método (<=), que construye la red de herencia dirigida.
