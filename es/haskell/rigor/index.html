Patrones de explosión #

Los patrones anotados con una explosión (!) se evalúan estrictamente en lugar de perezosamente.

foo (!x, y) !z = [x, y, z]

En este ejemplo, x y z se evaluarán en forma normal de cabeza débil antes de devolver la lista. es equivalente a:

foo (x, y) z = x `seq` z `seq` [x, y, z]

Los patrones de explosión se habilitan mediante la extensión de lenguaje BangPatterns de Haskell 2010.

Patrones perezosos #

Los patrones perezosos o irrefutables (denotados con la sintaxis ~pat) son patrones que siempre coinciden, sin siquiera mirar el valor coincidente. Esto significa que los patrones perezosos coincidirán incluso con los valores inferiores. Sin embargo, los usos posteriores de variables vinculadas en subpatrones de un patrón irrefutable obligarán a que se produzca la coincidencia de patrones, evaluándose hasta el fondo a menos que la coincidencia tenga éxito.

La siguiente función es perezosa en su argumento:

f1 :: Either e Int -> Int
f1 ~(Right 1) = 42

y así obtenemos

λ» f1 (Right 1)
42
λ» f1 (Right 2)
42
λ» f1 (Left "foo")
42
λ» f1 (error "oops!")
42
λ» f1 "oops!"
*** type mismatch ***

La siguiente función está escrita con un patrón perezoso, pero de hecho está usando la variable del patrón que fuerza la coincidencia, por lo que fallará para los argumentos ‘Izquierda’:

f2 :: Either e Int -> Int
f2 ~(Right x) = x + 1

λ» f2 (Right 1)
2
λ» f2 (Right 2)
3
λ» f2 (Right (error "oops!"))
*** Exception: oops!
λ» f2 (Left "foo")
*** Exception: lazypat.hs:5:1-21: Irrefutable pattern failed for pattern (Right x)
λ» f2 (error "oops!")
*** Exception: oops!

Los enlaces let son perezosos, se comportan como patrones irrefutables:

act1 :: IO ()
act1 = do
    ss <- readLn
    let [s1, s2] = ss :: [String]
    putStrLn "Done"

act2 :: IO ()
act2 = do
    ss <- readLn
    let [s1, s2] = ss
    putStrLn s1

Aquí, act1 funciona en entradas que analizan cualquier lista de cadenas, mientras que en act2, putStrLn s1 necesita el valor de s1, lo que fuerza la coincidencia de patrones para [s1, s2], por lo que solo funciona para listas de exactamente dos cadenas:

λ» act1
> ["foo"]
Done
λ» act2
> ["foo"]
*** readIO: no parse ***

Formas normales #

Este ejemplo proporciona una breve descripción general: para obtener una explicación más detallada de formas normales y ejemplos, consulte esta pregunta.

forma normal reducida #

La forma normal reducida (o simplemente la forma normal, cuando el contexto es claro) de una expresión es el resultado de evaluar todas las subexpresiones reducibles en la expresión dada. Debido a la semántica no estricta de Haskell (típicamente llamada pereza), una subexpresión no es reducible si está bajo un aglutinante (es decir, una abstracción lambda - \x -> ..). La forma normal de una expresión tiene la propiedad de que si existe, es única.

En otras palabras, no importa (en términos de semántica denotacional) en qué orden reduce las subexpresiones. Sin embargo, la clave para escribir programas Haskell de alto rendimiento es a menudo asegurarse de que la expresión correcta se evalúe en el momento adecuado, es decir, la comprensión de la semántica operativa.

Una expresión cuya forma normal es ella misma se dice que está en forma normal.

Algunas expresiones, por ej. let x = 1:x in x, no tienen forma normal, pero siguen siendo productivos. La expresión de ejemplo todavía tiene un valor, si se admiten infinitos valores, que aquí es la lista [1,1, ...]. Otras expresiones, como let y = 1+y in y, no tienen valor, o su valor es indefinido.

Forma normal de cabeza débil #

El RNF corresponde a la evaluación completa de una expresión; del mismo modo, la forma normal de cabeza débil (WHNF) corresponde a la evaluación de la cabeza de la expresión. El encabezado de una expresión e se evalúa completamente si e es una aplicación Con e1 e2 .. en y Con es un constructor; o una abstracción \x -> e1; o una aplicación parcial f e1 e2 .. en, donde la aplicación parcial significa que f toma más de n argumentos (o de manera equivalente, el tipo de e es un tipo de función). En cualquier caso, las subexpresiones e1..en pueden ser evaluadas o no evaluadas para que la expresión esté en WHNF, incluso pueden ser indefinidas.

La semántica de evaluación de Haskell se puede describir en términos de WHNF: para evaluar una expresión ’e’, ​​primero evalúela a WHNF, luego evalúe recursivamente todas sus subexpresiones de izquierda a derecha.

La función primitiva seq se usa para evaluar una expresión a WHNF. seq x y es denotacionalmente igual a y (el valor de seq x y es precisamente y); además, x se evalúa como WHNF cuando y se evalúa como WHNF. Una expresión también se puede evaluar para WHNF con un patrón de explosión (habilitado por la extensión -XBangPatterns), cuya sintaxis es la siguiente:

f !x y = ...

En el que x se evaluará como WHNF cuando se evalúe f, mientras que y no se evaluará (necesariamente). Un patrón de explosión también puede aparecer en un constructor, p.

data X = Con A !B C .. N

en cuyo caso se dice que el constructor Con es estricto en el campo B, lo que significa que el campo B se evalúa como WHNF cuando el constructor se aplica a suficientes (aquí, dos) argumentos.

Campos estrictos #

En una declaración de datos, anteponer un tipo con un bang (!) hace que el campo sea un campo estricto. Cuando se aplica el constructor de datos, esos campos se evaluarán en forma normal débil, por lo que se garantiza que los datos en los campos siempre estarán en forma normal débil.

Los campos estrictos se pueden utilizar tanto en tipos de registro como en tipos sin registro:

data User = User
    { identifier :: !Int
    , firstName :: !Text
    , lastName :: !Text
    }

data T = MkT !Int !Int
