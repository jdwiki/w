Las mónadas de estado son un tipo de mónada que tienen un estado que puede cambiar durante cada ejecución de cálculo en la mónada.

Las implementaciones suelen tener la forma State s a, que representa un cálculo que transporta y potencialmente modifica un estado de tipo s y produce un resultado de tipo a, pero el término “estado de mónada” generalmente puede referirse a cualquier mónada que lleva un estado.

Los paquetes mtl y transformers proporcionan implementaciones generales de mónadas de estado.

Los recién llegados a Haskell a menudo se alejan de la mónada ‘Estado’ y la tratan como un tabú, como si el supuesto beneficio de la programación funcional fuera evitar el estado, así que ¿no pierdes eso cuando usas ‘Estado’? Una visión más matizada es que:

  • El estado puede ser útil en pequeñas dosis controladas;
  • El tipo ‘Estado’ proporciona la capacidad de controlar la dosis con mucha precisión.

Las razones son que si tiene action :: State s a, esto le dice que:

  • action es especial porque depende de un estado;
  • El estado tiene el tipo s, por lo que la acción no puede verse influenciada por ningún valor anterior en su programa, solo un s o algún valor accesible desde algunos s;
  • El runState :: State s a -> s -> (a, s) pone una “barrera” alrededor de la acción con estado, de modo que su eficacia no se puede observar desde fuera de esa barrera.

Así que este es un buen conjunto de criterios para usar State en un escenario particular. Quiere ver que su código minimiza el alcance del estado, tanto eligiendo un tipo estrecho para s como colocando runState lo más cerca posible de “la parte inferior” (para que sus acciones puedan ser influenciado por la menor cantidad de cosas posible.

Numeración de los nodos de un árbol con un contador #

Tenemos un tipo de datos de árbol como este:

data Tree a = Tree a [Tree a] deriving Show

Y deseamos escribir una función que asigne un número a cada nodo del árbol, a partir de un contador incremental:

tag :: Tree a -> Tree (a, Int)

El camino largo #

Primero lo haremos de la manera más larga, ya que ilustra bastante bien la mecánica de bajo nivel de la mónada State.

import Control.Monad.State

-- Function that numbers the nodes of a `Tree`.
tag :: Tree a -> Tree (a, Int)
tag tree = 
    -- tagStep is where the action happens.  This just gets the ball
    -- rolling, with `0` as the initial counter value.
    evalState (tagStep tree) 0

-- This is one monadic "step" of the calculation.  It assumes that
-- it has access to the current counter value implicitly.
tagStep :: Tree a -> State Int (Tree (a, Int))
tagStep (Tree a subtrees) = do
    -- The `get :: State s s` action accesses the implicit state
    -- parameter of the State monad.  Here we bind that value to
    -- the variable `counter`.
    counter <- get 

    -- The `put :: s -> State s ()` sets the implicit state parameter
    -- of the `State` monad.  The next `get` that we execute will see
    -- the value of `counter + 1` (assuming no other puts in between).
    put (counter + 1)

    -- Recurse into the subtrees.  `mapM` is a utility function
    -- for executing a monadic actions (like `tagStep`) on a list of
    -- elements, and producing the list of results.  Each execution of 
    -- `tagStep` will be executed with the counter value that resulted
    -- from the previous list element's execution.
    subtrees' <- mapM tagStep subtrees  

    return $ Tree (a, counter) subtrees'

Refactorización #

Dividir el contador en una acción postIncrementar #

El bit en el que “obtenemos” el contador actual y luego “ponemos” el contador + 1 se puede dividir en una acción “postIncremento”, similar a lo que proporcionan muchos lenguajes de estilo C:

postIncrement :: Enum s => State s s
postIncrement = do
    result <- get
    modify succ
    return result

Dividir el recorrido del árbol en una función de orden superior #

La lógica del recorrido del árbol se puede dividir en su propia función, así:

mapTreeM :: Monad m => (a -> m b) -> Tree a -> m (Tree b)
mapTreeM action (Tree a subtrees) = do
    a' <- action a
    subtrees' <- mapM (mapTreeM action) subtrees
    return $ Tree a' subtrees'

Con esto y la función postIncrement podemos reescribir tagStep:

tagStep :: Tree a -> State Int (Tree (a, Int))
tagStep = mapTreeM step
    where step :: a -> State Int (a, Int)
          step a = do 
              counter <- postIncrement
              return (a, counter)

Usa la clase Traversable #

La solución mapTreeM anterior se puede reescribir fácilmente en una instancia de [la clase Traversable] (https://www.wikiod.com/es/haskell/traversable):

instance Traversable Tree where
    traverse action (Tree a subtrees) = 
        Tree <$> action a <*> traverse action subtrees

Tenga en cuenta que esto requería que usáramos Applicative (el operador <*>) en lugar de Monad.

Con eso, ahora podemos escribir tag como un profesional:

tag :: Traversable t => t a -> t (a, Int)
tag init t = evalState (traverse step t) 0
    where step a = do tag <- postIncrement
                      return (a, tag)

¡Tenga en cuenta que esto funciona para cualquier tipo Transversable, no solo para nuestro tipo Tree!

Deshacerse del repetitivo Traversable #

GHC tiene una extensión DeriveTraversable que elimina la necesidad de escribir la instancia anterior:

{-# LANGUAGE DeriveFunctor, DeriveFoldable, DeriveTraversable #-}

data Tree a = Tree a [Tree a]
            deriving (Show, Functor, Foldable, Traversable)
