Functor es la clase de tipos f :: * -> * que se pueden asignar covariantemente. Mapear una función sobre una estructura de datos aplica la función a todos los elementos de la estructura sin cambiar la estructura misma.

Se puede pensar en un Funtor como un contenedor para algún valor o un contexto de cálculo. Los ejemplos son Quizás un o [a]. El artículo Typeclassopedia tiene una buena descripción de los conceptos detrás de los Functors.

Para ser considerado un Functor real, una instancia debe respetar las 2 leyes siguientes:

Identidad #

fmap id == id

Composición #

fmap (f . g) = (fmap f) . (fmap g)

Clase Definición de Functor y Leyes #

class Functor f where
    fmap :: (a -> b) -> f a -> f b

Una forma de verlo es que fmap eleva una función de valores a una función de valores en un contexto f.

Una instancia correcta de Functor debería satisfacer las leyes de functor, aunque el compilador no las aplica:

fmap id = id                    -- identity
fmap f . fmap g = fmap (f . g)  -- composition

Hay un alias infijo de uso común para fmap llamado <$>.

infixl 4 <$>
(<$>) :: Functor f => (a -> b) -> f a -> f b
(<$>) = fmap

Instancias comunes de Funtor #

Quizás #

Maybe es un Functor que contiene un valor posiblemente ausente:

instance Functor Maybe where
    fmap f Nothing = Nothing
    fmap f (Just x) = Just (f x)

La instancia de Maybe de Functor aplica una función a un valor envuelto en un Just. Si el cálculo ha fallado anteriormente (por lo que el valor Quizás es Nada), entonces no hay ningún valor al que aplicar la función, por lo que fmap no funciona.

> fmap (+ 3) (Just 3)
Just 6
> fmap length (Just "mousetrap")
Just 9
> fmap sqrt Nothing
Nothing

Podemos verificar las leyes de los funtores para este caso usando razonamiento ecuacional. Para la ley de identidad,

fmap id Nothing
Nothing  -- definition of fmap
id Nothing  -- definition of id

fmap id (Just x)
Just (id x)  -- definition of fmap
Just x  -- definition of id
id (Just x)  -- definition of id

Para la ley de composición,

(fmap f . fmap g) Nothing
fmap f (fmap g Nothing)  -- definition of (.)
fmap f Nothing  -- definition of fmap
Nothing  -- definition of fmap
fmap (f . g) Nothing  -- because Nothing = fmap f Nothing, for all f

(fmap f . fmap g) (Just x)
fmap f (fmap g (Just x))  -- definition of (.)
fmap f (Just (g x))  -- definition of fmap
Just (f (g x))  -- definition of fmap
Just ((f . g) x)  -- definition of (.)
fmap (f . g) (Just x)  -- definition of fmap

Liza #

La instancia de Lists de Functor aplica la función a cada valor en la lista en su lugar.

instance Functor [] where
    fmap f [] = []
    fmap f (x:xs) = f x : fmap f xs

Esto podría escribirse alternativamente como una lista por comprensión: fmap f xs = [f x | x <- xs].

Este ejemplo muestra que fmap generaliza map. map solo opera en listas, mientras que fmap funciona en un Functor arbitrario.

Se puede demostrar que la ley de identidad se cumple por inducción:

-- base case
fmap id []
[]  -- definition of fmap
id []  -- definition of id

-- inductive step
fmap id (x:xs)
id x : fmap id xs  -- definition of fmap
x : fmap id xs  -- definition of id
x : id xs  -- by the inductive hypothesis
x : xs  -- definition of id
id (x : xs)  -- definition of id

y análogamente, la ley de composición:

-- base case
(fmap f . fmap g) []
fmap f (fmap g [])  -- definition of (.)
fmap f []  -- definition of fmap
[]  -- definition of fmap
fmap (f . g) []  -- because [] = fmap f [], for all f

-- inductive step
(fmap f . fmap g) (x:xs)
fmap f (fmap g (x:xs))  -- definition of (.)
fmap f (g x : fmap g xs)  -- definition of fmap
f (g x) : fmap f (fmap g xs)  -- definition of fmap
(f . g) x : fmap f (fmap g xs)  -- definition of (.)
(f . g) x : fmap (f . g) xs  -- by the inductive hypothesis
fmap (f . g) xs  -- definition of fmap

Funciones #

No todos los Functor parecen un contenedor. La instancia de Functions de Functor aplica una función al valor de retorno de otra función.

instance Functor ((->) r) where
    fmap f g = \x -> f (g x)

Tenga en cuenta que esta definición es equivalente a fmap = (.). Entonces fmap generaliza la composición de funciones.

Una vez más revisando la ley de identidad:

fmap id g
\x -> id (g x)  -- definition of fmap
\x -> g x  -- definition of id
g  -- eta-reduction
id g  -- definition of id

y la ley de composición:

(fmap f . fmap g) h
fmap f (fmap g h)  -- definition of (.)
fmap f (\x -> g (h x))  -- definition of fmap
\y -> f ((\x -> g (h x)) y)  -- definition of fmap
\y -> f (g (h y))  -- beta-reduction
\y -> (f . g) (h y)  -- definition of (.)
fmap (f . g) h  -- definition of fmap

Reemplazar todos los elementos de un Functor con un solo valor #

El módulo Data.Functor contiene dos combinadores, <$ y $>, que ignoran todos los valores contenidos en un funtor, reemplazándolos todos con un solo valor constante.

infixl 4 <$, $>

<$ :: Functor f => a -> f b -> f a
(<$) = fmap . const

$> :: Functor f => f a -> b -> f b
($>) = flip (<$)

void ignora el valor de retorno de un cálculo.

void :: Functor f => f a -> f ()
void = (() <$)

Funtores polinómicos #

Hay un conjunto útil de combinadores de tipos para construir grandes Functors a partir de los más pequeños. Estos son instructivos como instancias de ejemplo de Functor, y también son útiles como técnica para la programación genérica, porque pueden usarse para representar una gran clase de funtores comunes.

El funtor identidad #

El funtor identidad simplemente concluye su argumento. Es una implementación a nivel de tipo del combinador I del cálculo SKI.

newtype I a = I a

instance Functor I where
    fmap f (I x) = I (f x)

I se puede encontrar, bajo el nombre de Identity, en [el módulo Data.Functor.Identity](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data- Funtor-identidad.html).

El funtor constante #

El funtor constante ignora su segundo argumento, que contiene solo un valor constante. Es un análogo a nivel de tipo de const, el combinador K del cálculo SKI.

newtype K c a = K c

Tenga en cuenta que K c a no contiene ningún valor a; K () es isomorfo a Proxy. ¡Esto significa que la implementación de K de fmap no hace ningún mapeo en absoluto!

instance Functor (K c) where
    fmap _ (K c) = K c

K también se conoce como Const, de Data.Functor.Const .

Los funtores restantes en este ejemplo combinan funtores más pequeños en otros más grandes.

productos funtores #

El producto funtor toma un par de funtores y los empaqueta. Es análogo a una tupla, excepto que mientras (,) :: * -> * -> * opera en tipos *, (:*:) :: (* -> *) -> (* -> *) -> (* -> *) opera sobre funtores * -> *.

infixl 7 :*:
data (f :*: g) a = f a :*: g a

instance (Functor f, Functor g) => Functor (f :*: g) where
    fmap f (fx :*: gy) = fmap f fx :*: fmap f gy

Este tipo se puede encontrar, bajo el nombre Producto, en [el módulo Data.Functor.Product](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data-Functor- Producto.html).

Coproductos funcionales #

Al igual que :*: es análogo a (,), :+: es el análogo a nivel de funtor de Either.

infixl 6 :+:
data (f :+: g) a = InL (f a) | InR (g a)

instance (Functor f, Functor g) => Functor (f :+: g) where
    fmap f (InL fx) = InL (fmap f fx)
    fmap f (InR gy) = InR (fmap f gy)

:+: se puede encontrar bajo el nombre Sum, en [el módulo Data.Functor.Sum](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data- Functor-Suma.html).

Composición de funtores #

Finalmente, :.: funciona como un (.) de nivel de tipo, tomando la salida de un funtor y conectándola a la entrada de otro.

infixr 9 :.:
newtype (f :.: g) a = Cmp (f (g a))

instance (Functor f, Functor g) => Functor (f :.: g) where
    fmap f (Cmp fgx) = Cmp (fmap (fmap f) fgx)

El tipo Compose se puede encontrar en Data.Functor.Compose

Functores polinómicos para programación genérica #

I, K, :*:, :+: y :.: se pueden considerar como un conjunto de componentes básicos para una determinada clase de tipos de datos simples. El kit se vuelve especialmente poderoso cuando lo combina con [puntos fijos] (https://www.wikiod.com/es/haskell/esquemas-de-recursividad#puntos fijos) porque los tipos de datos creados con estos combinadores son automáticamente instancias de Functor. Usas el kit para construir un tipo de plantilla, marcando puntos recursivos usando I, y luego lo conectas a Fix para obtener un tipo que se puede usar con el zoológico estándar de esquemas recursivos.

| Nombre | Como un tipo de datos | Uso del kit de funtores | |————————|—————————— ———————————–|————– ————————————————–| | Pares de valores | datos Par a = Par a a | tipo Par = I :*: I | | Cuadrículas de dos por dos | tipo Cuadrícula a = Par (Par a) | tipo Cuadrícula = Par :.: Par | | Números naturales | datos Nat = Cero | Succ Nat | tipo Nat = Fijar (K () :+: I) | | Listas | lista de datos a = Nil | Contras a (Lista a) | tipo List a = Fix (K () :+: K a :*: I) | | Árboles binarios | árbol de datos a = Hoja | Nodo (Árbol a) a (Árbol a) | tipo Árbol a = Fijar (K () :+: I :*: K a :*: I) | | Rosales | data Rose a = Rose a (List (Rose a)) | type Rose a = Fix (K a :*: List :.: I) |

Este enfoque de “kit” para diseñar tipos de datos es la idea detrás de bibliotecas de programación genérica como generics-sop. La idea es escribir operaciones genéricas usando un kit como el que se presentó arriba y luego usar una clase de tipo para convertir tipos de datos arbitrarios hacia y desde su representación genérica:

class Generic a where
    type Rep a  -- a generic representation built using a kit
    to :: a -> Rep a
    from :: Rep a -> a

Funtores en la teoría de categorías #

Un Funtor se define en la teoría de categorías como un mapa que preserva la estructura (un ‘homomorfismo’) entre categorías. Específicamente, (todos) los objetos se asignan a objetos y (todas) las flechas se asignan a flechas, de modo que se conservan las leyes de categoría.

La categoría en la que los objetos son tipos de Haskell y los morfismos son funciones de Haskell se denomina Hask. Entonces, un funtor de Hask a Hask consistiría en un mapeo de tipos a tipos y un mapeo de funciones a funciones.

La relación que tiene este concepto teórico de categoría con la construcción de programación de Haskell ‘Functor’ es bastante directa. El mapeo de tipos a tipos toma la forma de un tipo f :: * -> *, y el mapeo de funciones a funciones toma la forma de una función fmap :: (a -> b) -> (f a - > fb). Poniéndolos juntos en una clase,

class Functor (f :: * -> *) where
    fmap :: (a -> b) -> f a -> f b

fmap es una operación que toma una función (un tipo de morfismo), :: a -> b, y la asigna a otra función, :: f a -> f b. Se supone (pero se deja que el programador se asegure) que las instancias de Functor son de hecho funtores matemáticos, preservando la estructura categórica de Hask:

fmap (id {- :: a -> a -})  ==  id {- :: f a -> f a -}
fmap (h . g)               ==  fmap h . fmap g

fmap eleva una función :: a -> b a una subcategoría de Hask de una manera que preserva tanto la existencia de cualquier flecha de identidad como la asociatividad de la composición.

La clase Functor solo codifica endofunctors en Hask. Pero en matemáticas, los funtores pueden mapear entre categorías arbitrarias. Una codificación más fiel de este concepto se vería así:

class Category c where
    id  :: c i i
    (.) :: c j k -> c i j -> c i k

class (Category c1, Category c2) => CFunctor c1 c2 f where
    cfmap :: c1 a b -> c2 (f a) (f b)

La clase Functor estándar es un caso especial de esta clase en la que las categorías de origen y destino son Hask. Por ejemplo,

instance Category (->) where        -- Hask
    id    = \x -> x
    f . g = \x -> f (g x)

instance CFunctor (->) (->) [] where
    cfmap = fmap

Funtor derivado #

La extensión de lenguaje DeriveFunctor permite a GHC generar instancias de Functor automáticamente.

{-# LANGUAGE DeriveFunctor #-}

data List a = Nil | Cons a (List a) deriving Functor

-- instance Functor List where            -- automatically defined
--   fmap f Nil = Nil
--   fmap f (Cons x xs) = Cons (f x) (fmap f xs)

map :: (a -> b) -> List a -> List b
map = fmap
