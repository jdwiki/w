Enviar sus cambios locales al repositorio #

Para publicar los cambios que realizó en su copia de trabajo, ejecute el comando svn commit.

IMPORTANTE: ¡Revise sus cambios antes de confirmarlos! Use svn status y svn diff para revisar los cambios. Además, asegúrese de estar en la ruta correcta antes de realizar una confirmación. Si actualizó muchos archivos en varios directorios, debe estar en el nivel adecuado para incluirlos todos debajo de su ubicación.

Aquí hay un ejemplo del comando de confirmación:

svn commit -m "My Descriptive Log Message"

Alternativamente, svn ci es la abreviatura de svn commit

Tenga en cuenta la opción -m (--message). Los buenos mensajes de compromiso ayudan a otros a comprender por qué se realizó un compromiso. Además, en el lado del servidor es posible [hacer cumplir los mensajes no vacíos] 4, e incluso hacer cumplir que cada mensaje de confirmación mencione un ticket existente en su sistema de seguimiento de errores.

Revisando los registros #

Ejecutar svn log le mostrará todos los mensajes de confirmación, probablemente desee revisar solo ciertas revisiones.

  • Ver las n revisiones más recientes:

    svn log -n

  • Ver una revisión específica:

    svn log -c rXXX

  • Ver las rutas afectadas:

    svn log -v -c rXXX

Instalación y configuración inicial #

Instale el cliente svn para comenzar a colaborar en el proyecto que utiliza Subversion como su sistema de control de versiones.

Para instalar Subversion, puede crearlo usted mismo a partir de una versión del código fuente o descargar un paquete binario prediseñado para su sistema operativo. La lista de sitios donde puede obtener un cliente Subversion compilado (svn) para varios sistemas operativos está disponible en la página oficial de paquetes binarios. Si tiene ganas de compilar el software usted mismo, tome la fuente en la página de código fuente.

Con Subversion, no está limitado a usar solo el cliente de línea de comandos estándar svn. Hay algunos clientes gráficos notables de Subversion para varios sistemas operativos y la mayoría de los IDE en la actualidad brindan una sólida integración con SVN desde el primer momento o mediante complementos. Para ver la lista de clientes gráficos, consulte la página de Wikipedia: https://en.wikipedia.org/wiki/Comparison_of_Subversion_clients.

Inmediatamente después de instalar el cliente, debería poder ejecutarlo emitiendo el comando svn. Deberías ver lo siguiente:

$ svn
Type 'svn help' for usage.

Todo está casi listo. Ahora debe crear un espacio de trabajo local llamado copia de trabajo que se conectará al repositorio central remoto. En otras palabras, va a retirar una copia de trabajo. Vas a operar con los datos versionados con la ayuda de la copia de trabajo y puedes publicar tus cambios (llamados confirmación en SVN) para que otros que trabajen en el mismo proyecto puedan verlos y beneficiarse de tus cambios. Para obtener más tarde los cambios futuros realizados por otros del repositorio, actualizaría su copia de trabajo. Estas operaciones básicas se tratan en otros ejemplos.

Crear y aplicar parches #

Un parche es un archivo que muestra las diferencias entre dos revisiones o entre su repositorio local y la última revisión a la que apunta su repositorio.

Para compartir o guardar un parche de sus cambios locales no confirmados, ya sea para revisión por pares o para aplicarlos más tarde, haga lo siguiente:

svn diff > new-feature.patch

Para obtener un parche de las diferencias entre dos revisiones:

svn diff -r NEWER_REVISION:OLDER_REVISION > feature.patch

Para aplicar un parche, ejecute:

svn patch new-feature.patch

Para aplicar el parche con éxito, debe ejecutar el comando desde la misma ruta donde se creó el parche.

Revisar una copia de trabajo #

Para comenzar a realizar modificaciones en los datos del proyecto, debe obtener una copia local del proyecto versionado. Utilice la línea de comandos del cliente svn o su cliente SVN favorito (TortoiseSVN, por ejemplo). Su copia local del proyecto se denomina copia de trabajo en Subversion y la obtiene emitiendo el comando svn checkout <URL> donde <URL> es una URL de repositorio. p.ej.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk

Alternativamente, puede usar svn co <URL> como abreviatura para obtener una copia local.

Como resultado, obtendrá una copia funcional de /trunk de un proyecto llamado MyProject que reside en el repositorio MyRepo. La copia de trabajo se ubicará en un directorio llamado “troncal” en su computadora en relación con el directorio en el que emitió el comando.

Si desea tener un nombre diferente para su copia de trabajo, puede agregarlo como parámetro al final del comando. p.ej.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk MyProjectSource

Esto creará una copia de trabajo llamada MyProjectSource.

Tenga en cuenta que en lugar de consultar el tronco, puede consultar alguna sucursal, estantería privada o una etiqueta (suponiendo que ya existan en el repositorio); puede tener un número ilimitado de copias de trabajo locales en su máquina.

También puede obtener la copia de trabajo de todo el repositorio MyRepo. Pero debes abstenerte de hacerlo. En términos generales, no necesita tener una copia de trabajo de todo el repositorio para su trabajo porque su copia de trabajo se puede cambiar instantáneamente a otra rama de desarrollo/etiqueta/lo que sea. Además, el repositorio de Subversion puede contener varios proyectos (no) relacionados y es mejor tener una copia de trabajo dedicada para cada uno de ellos, no una única copia de trabajo para todos los proyectos.

Exportación de los datos versionados (descarga simple) #

Si desea obtener los datos del proyecto versionado, pero no necesita ninguna de las funciones de control de versiones que ofrece Subversion, puede ejecutar el comando svn export <URL>. Aquí hay un ejemplo:

$ svn export https://svn.example.com/svn/MyRepo/MyProject/trunk

Como resultado, obtendrá la exportación de datos del proyecto, pero a diferencia de una copia de trabajo, no podrá ejecutar comandos svn en él. La exportación es simplemente una simple descarga de los datos.

Si algún tiempo después desea convertir los datos descargados en una copia de trabajo completamente funcional, ejecute svn checkout <URL> en el directorio donde ejecutó la exportación.

Extraer una copia de trabajo en una revisión específica #

Para obtener la versión 5394 use:

svn co --revision r5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

O la versión más corta:

svn co -r 5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

O mediante el uso de revisiones vinculadas:

svn co https://svn.example.com/svn/MyRepo/MyProject/trunk@5394

Si ya está desprotegido, puede usar el comando actualizar para mover a una revisión en particular, haciendo lo siguiente:

svn up -rXXX

Actualización de una copia de trabajo #

No eres la única persona que trabaja en el proyecto, ¿verdad? Esto significa que sus compañeros también están modificando los datos del proyecto. Para mantenerse actualizado y recuperar las modificaciones realizadas por otros, debe ejecutar el comando svn update en su copia de trabajo. Como resultado, su copia de trabajo se sincronizará con el repositorio y descargará los cambios realizados por sus colegas.

La abreviatura de svn update es svn up.

Es una regla ejecutar svn update antes de confirmar los cambios.

Hacer cambios en su copia de trabajo local #

La copia de trabajo (WC) es su espacio de trabajo local y privado que utiliza para interactuar con el repositorio central de Subversion. Utiliza la copia de trabajo para modificar el contenido de su proyecto y recuperar los cambios realizados por otros.

La copia de trabajo contiene los datos de su proyecto y se ve y actúa como un directorio normal en su sistema de archivos local, pero con una gran diferencia: la copia de trabajo rastrea el estado y los cambios de los archivos. y directorios dentro. Puede pensar en la copia de trabajo como si fuera un directorio normal con un sabor de control de versiones agregado por un directorio de metadatos .svn oculto en su raíz.

La mayoría de las veces, realizará modificaciones en los datos del proyecto modificando el contenido de la copia de trabajo. Tan pronto como esté satisfecho con las modificaciones y las haya revisado minuciosamente, estará listo para publicarlas en el repositorio central.

Puede realizar cualquier acción con los datos de su proyecto dentro de la copia de trabajo, pero las operaciones que implican copiar, mover, renombrar y eliminar deben realizarse utilizando los comandos svn correspondientes:

  • Modificación de archivos existentes. Modifique los archivos como lo hace normalmente utilizando su procesador de texto favorito, editor de gráficos, software de edición de audio, IDE, etc. Tan pronto como guarde los cambios en el disco, Subversion los reconocerá automáticamente.

  • Añadiendo nuevos archivos. Ponga nuevos archivos en la copia de trabajo y Subversion los reconocerá como no versionados. No comenzará a rastrear automáticamente los nuevos archivos a menos que ejecute el comando svn add:

    svn add foo.cs

  • Mover archivos y directorios. Mueva archivos y directorios usando el comando svn move:

    svn move foo.cs bar.cs

  • Renombrar archivos y directorios. Cambie el nombre de archivos y directorios usando el comando svn rename:

    svn rename foo.cs bar.cs

    NOTE: svn rename command is an alias of svn move command.

  • Copia de archivos y directorios. Copie archivos y directorios usando el comando svn copy:

    svn copy foo.cs bar.cs

  • Eliminación de archivos y directorios. Elimine archivos y directorios usando el comando svn delete:

    svn delete foo.cs

  • Comprobación del estado de archivos y directorios en la copia de trabajo. Revise sus cambios usando el comando svn status (o svn st para abreviar):

    svn status

    IMPORTANT: Always review your changes before committing them. This will help you to avoid committing unnecessary or irrelevant changes.

  • Deshaciendo cambios. Revierta sus cambios usando el comando svn revert:

    svn revert foo.c

  • Revertir todos los cambios: Desde la raíz del repositorio:

    svn revert -R .

IMPORTANTE: Los cambios no confirmados revertidos se perderán para siempre. No podrá recuperar los cambios revertidos. ¡Utilice svn revert con precaución! Si desea conservar los cambios pero necesita revertirlos, guárdelos en un parche. Ver ejemplo de cómo crear y aplicar un parche.

Revertir o revertir un archivo #

Para restaurar un archivo a la última versión actualizada de svn, es decir, deshacer los cambios locales, puede usar revert:

svn revert file

Para restaurar un archivo a una versión anterior (revisión XXX), use update:

svn update -r XXX file

Advertencia: en ambos casos, perderá cualquier cambio local en el archivo porque se sobrescribirá.

Para ver solo la versión anterior de un archivo, use cat:

svn cat -r XXX file

Y para ver las diferencias con su versión local del archivo:

svn diff -r XXX file

Uso de un repositorio protegido por contraseña #

Se puede configurar un repositorio de Subversion para que ciertos contenidos o comandos solo sean accesibles para ciertos usuarios. Para acceder a este contenido restringido, deberá especificar un nombre de usuario y una contraseña.

Su nombre de usuario y contraseña se pueden especificar directamente como parte del comando:

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser --password topsecret

Desafortunadamente, esto hace que su contraseña aparezca en texto sin formato en la consola. Para evitar este posible problema de seguridad, especifique un nombre de usuario pero no una contraseña. Al hacer esto, aparecerá una solicitud de contraseña, lo que le permitirá ingresar su contraseña sin exponerla:

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser
Password for 'JoeUser':

Si no proporciona ninguna información de autenticación, Subversion le solicitará tanto el nombre de usuario como la contraseña:

$ svn checkout https://svn.example.com/MyRepo/trunk
Username:  JoeUser
Password for 'JoeUser':

Si bien el primer método es menos seguro, se ve con frecuencia en secuencias de comandos automatizadas, ya que es difícil para muchos tipos de secuencias de comandos proporcionar información a un aviso interactivo.

Nota: los comandos que solo funcionan en su copia de trabajo (como revert o status) nunca requerirán una contraseña, solo los comandos que requieren comunicarse con el servidor del repositorio.
