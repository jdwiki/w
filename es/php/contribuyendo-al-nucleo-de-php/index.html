Contribuyendo al núcleo de PHP

PHP es un proyecto de código abierto y, como tal, cualquiera puede contribuir. En términos generales, hay dos formas de contribuir al núcleo de PHP:

  • Corrección de errores
  • Adiciones de características

Sin embargo, antes de contribuir, es importante comprender cómo se administran y lanzan las versiones de PHP para que las correcciones de errores y las solicitudes de funciones puedan apuntar a la versión de PHP correcta. Los cambios desarrollados se pueden enviar como una solicitud de extracción al repositorio de PHP Github. Puede encontrar información útil para los desarrolladores en la sección “Involúcrese” del sitio PHP.net y el foro de #externos.

Contribuir con correcciones de errores #

Para aquellos que buscan comenzar a contribuir al núcleo, generalmente es más fácil comenzar con la corrección de errores. Esto ayuda a familiarizarse con las funciones internas de PHP antes de intentar realizar modificaciones más complejas en el núcleo que requeriría una característica.

Con respecto al proceso de administración de versiones, las correcciones de errores deben apuntar a la versión de PHP más afectada, mientras aún sea compatible. Es esta versión a la que deben apuntar las solicitudes de extracción de corrección de errores. A partir de ahí, un miembro interno puede fusionar la solución en la rama correcta y luego fusionarla hacia arriba con versiones posteriores de PHP según sea necesario.

Para aquellos que buscan comenzar a resolver errores, se puede encontrar una lista de informes de errores en bugs.php.net.

Contribuir con adiciones de características #

PHP sigue un proceso RFC cuando introduce nuevas funciones y realiza cambios importantes en el lenguaje. Los RFC son votados por miembros de php.net y deben lograr una mayoría simple (50 % + 1) o una gran mayoría (2/3 + 1) del total de votos. Se requiere una gran mayoría si el cambio afecta al idioma mismo (como la introducción de una nueva sintaxis), de lo contrario, solo se requiere una mayoría simple.

Antes de que los RFC puedan someterse a votación, deben someterse a un período de discusión de al menos 2 semanas en la lista de correo oficial de PHP. Una vez finalizado este período, y no hay temas abiertos con el RFC, se puede pasar a la votación, que debe durar al menos 1 semana.

Si un usuario desea revivir un RFC previamente rechazado, puede hacerlo solo en una de las siguientes dos circunstancias:

  • Han pasado 6 meses desde la votación anterior
  • Los autores realizan cambios sustanciales en el RFC que probablemente afectarían el resultado de la votación si el RFC se sometiera a votación nuevamente.

Las personas que tienen el privilegio de votar serán colaboradores de PHP (y también tendrán cuentas de php.net), o serán representantes de la comunidad de PHP. Estos representantes son elegidos por quienes tienen cuentas de php.net y serán desarrolladores líderes de proyectos basados ​​en PHP o participantes regulares en discusiones internas.

Al enviar nuevas ideas para la propuesta, casi siempre se requiere que el proponente escriba, como mínimo, un parche de prueba de concepto. Esto se debe a que, sin una implementación, la sugerencia simplemente se convierte en otra solicitud de función que es poco probable que se cumpla en un futuro próximo.

Puede encontrar una guía detallada de este proceso en la página oficial Cómo crear un RFC.

Lanzamientos #

Las versiones principales de PHP no tienen un ciclo de lanzamiento establecido, por lo que pueden publicarse a discreción del equipo interno (siempre que lo consideren adecuado para una nueva versión principal). Las versiones menores, por otro lado, se lanzan anualmente.

Antes de cada lanzamiento de PHP (mayor, menor o parche), se ponen a disposición una serie de candidatos de lanzamiento (RC). PHP no usa un RC como lo hacen otros proyectos (es decir, si no se han encontrado problemas con un RC, entonces hágalo como la próxima versión final). En cambio, los usa como una forma de versiones beta finales, donde normalmente se decide un número determinado de RC antes de que se realice el lanzamiento final.

Versionado #

PHP generalmente intenta seguir el control de versiones semántico siempre que sea posible. Como tal, la compatibilidad con versiones anteriores (BC) debe mantenerse en versiones menores y parches del lenguaje. Las características y los cambios que conservan BC deben apuntar a versiones menores (no a versiones de parches). Si una característica o cambio tiene el potencial de romper BC, entonces deberían apuntar a la siguiente versión principal de PHP (X.y.z) en su lugar.

Cada versión menor de PHP (x.Y.z) tiene dos años de soporte general (el llamado “soporte activo”) para todo tipo de correcciones de errores. Se agrega un año adicional además de eso para el soporte de seguridad, donde solo se aplican correcciones relacionadas con la seguridad. Después de los tres años, el soporte para esa versión de PHP se elimina por completo. Puede encontrar una lista de versiones de PHP admitidas actualmente en php.net.

Configuración de un entorno de desarrollo básico #

El código fuente de PHP está alojado en GitHub. Para compilar desde la fuente, primero deberá verificar una copia funcional del código.

mkdir /usr/local/src/php-7.0/
cd /usr/local/src/php-7.0/
git clone -b PHP-7.0 https://github.com/php/php-src .

Si desea agregar una función, lo mejor es crear su propia rama.

git checkout -b my_private_branch

Finalmente, configure y construya PHP

./buildconf
./configure
make
make test
make install

Si la configuración falla debido a la falta de dependencias, deberá usar el sistema de administración de paquetes de su sistema operativo para instalarlos (por ejemplo, yum, apt, etc.) o descargarlos y compilarlos desde la fuente.
