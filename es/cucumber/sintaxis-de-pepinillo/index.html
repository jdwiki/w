Gherkin es un lenguaje de lectura empresarial para la automatización de pruebas y la documentación de pruebas. Se entiende por Cucumber y en conjunto existe como una herramienta de desarrollo impulsada por el comportamiento.

Sintaxis #

  • Característica: esta palabra clave significa que lo que sigue es un básico descripción o nombre de la característica que se está probando o documentando.
  • Antecedentes: esta palabra clave significa pasos que se ejecutarán antes de cada escenario en la función.
  • Escenario: esta palabra clave representa el nombre o descripción básica de un escenario particular que prueba la función.
  • Esquema del escenario: esta palabra clave significa que el escenario se ejecutará N veces para cada argumento enumerado en los ejemplos pasados ​​explícitamente por el nombre de la columna entre corchetes angulares.
  • Ejemplos: esta palabra clave señala la lista de argumentos estáticos que se pasarán a un esquema de escenario.
  • Dado: esta palabra clave representa un paso dado, o condición previa que se asume antes continuo. En el paradigma Organizar, Actuar, Afirmar, dado representa “Arreglar”.
  • When: esta palabra clave representa un paso when, o el comportamiento contra lo que se va a afirmar. En el paradigma Organizar, Actuar, Afirmar, dado representa “Acto”.
  • Entonces: esta palabra clave representa un paso entonces, o en otras palabras, el paso en el que se valida el resultado de un comportamiento. En el paradigma Organizar, Actuar, Afirmar, dado representa “Afirmar”.
  • Y: esta palabra clave se usa junto con cualquiera de las palabras clave anteriores. Si tiene dos declaraciones dadas, en lugar de llamar explícitamente a Dado dos veces, puedes decir, “dados A y B”.

Pasos parametrizados #

Al escribir Gherkin, puede haber momentos en los que desee parametrizar sus pasos para que el ingeniero que está implementando los planes de prueba los reutilice. Los pasos reciben parámetros a través de grupos de captura de expresiones regulares. (Nota de ingeniería: Si no tiene parámetros coincidentes para cada grupo de captura en su expresión regular, puede esperar que se lance una “CucumberException: discrepancia de Arity”). En el siguiente ejemplo, hemos decidido envolver los argumentos en doble comillas, así como aceptar números enteros como argumentos.

 Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with "valid" credentials
        Then the user should be logged in

    Scenario: The user attempts to log in with invalid credentials 
        This scenario tests that a user is not able to log in when
        they enter invalid credentials

        Given the user is on the login page
        When the user signs in with "invalid" credentials
        Then the user should be logged in

    Scenario: The user is locked out after too many failed attempts
        This scenario validates that the user is locked out
        of their account after failing three consecutive 
        attempts to log in

        Given the user is on the login page
        When the user fails to log in 3 times
        Then the user should be locked out of their account

Antecedentes de la característica #

Como habrás notado en el ejemplo anterior, estamos reescribiendo el mismo paso varias veces:

Given the user is on the login page

Esto puede ser excepcionalmente tedioso, especialmente si tenemos más de un paso dado que se reutiliza. Gherkin proporciona una solución para esto al brindarnos otra palabra clave con la que trabajar: Antecedentes:.

La palabra clave de fondo sirve para ejecutar los pasos declarados debajo de ella antes de cada escenario en la función. Asegúrese de no agregar un paso de fondo a menos que esté seguro de que es necesario para cada escenario. Al igual que las otras palabras clave, Antecedentes va seguido de una descripción o nombre y puede tener comentarios debajo. Al igual que Característica y Escenario, el Fondo debe ir precedido por dos puntos.

Etiquetas #

A los efectos de la documentación, es posible que desee filtrar los planes o escenarios de prueba por categorías. Los desarrolladores pueden querer ejecutar pruebas basadas en esas mismas categorías. Gherkin le permite categorizar características, así como escenarios individuales a través del usuario de etiquetas. En el siguiente ejemplo, observe que la palabra clave de característica anterior es la etiqueta “@Automation”. Gherkin reconoce esto como una etiqueta del usuario del símbolo “@”. En este ejemplo, el ingeniero quiere dejar en claro que estas pruebas se utilizan para la automatización, donde no todas las pruebas se pueden automatizar, algunas pruebas deben realizarse mediante control de calidad manual.

Observe también que la etiqueta @Production se ha agregado al bloqueo del usuario de prueba de escenario. En este ejemplo, esto se debe a que este escenario solo está activo en el entorno de producción de la aplicación. Los desarrolladores no quieren que se bloqueen sus cuentas de sandbox durante el desarrollo. Estas etiquetas les permiten hacer cumplir que esta prueba solo se ejecutará en el entorno de producción.

Por último, el esquema del escenario tiene la etiqueta @Staging. A los efectos de este ejemplo, esto se debe a que las cuentas que se utilizan son cuentas provisionales y no funcionarán en los demás entornos. Al igual que la etiqueta @Production, esto garantiza que estas pruebas solo se ejecutarán en el entorno de prueba.

Estos son solo algunos ejemplos de dónde, cómo y por qué podría usar etiquetas. En última instancia, estas etiquetas tendrán significado para usted y los desarrolladores y pueden ser cualquier cosa y usarse para categorizar como mejor le parezca.

Consejos de pepinillos #

  • Cada escenario prueba un comportamiento
  • Los escenarios están escritos de forma declarativa.
  • Evitar detalles incidentales dentro del escenario.
  • Omitir lo obvio
  • Evitar pasos conjuntivos
  • Mantenga sus escenarios cortos
  • No tiene muchos escenarios en la misma característica
  • Usar nombres de escenarios descriptivos
  • Tener solo un paso Cuando
  • Use el “debería” en los pasos Entonces

Los basicos #

Este ejemplo repasará la estructura básica de un archivo de características de Cucumber en Gherkin. Los archivos de características utilizan varias palabras clave en la sintaxis básica.

Veamos las palabras clave básicas:

  • Función: esta palabra clave significa que lo que sigue es una descripción básica o el nombre de la función que se está probando o documentando.
  • Escenario: esta palabra clave representa el nombre o la descripción básica de un escenario particular que prueba la característica.
  • Dada esta palabra clave representa un paso determinado o una condición previa que se asume antes de continuar. En el paradigma Organizar, Actuar, Afirmar, dado representa “Organizar”.
  • Cuando esta palabra clave representa un paso cuando, o el comportamiento contra el que se va a afirmar. En el paradigma Organizar, Actuar, Afirmar, dado representa “Actuar”.
  • Entonces esta palabra clave representa un paso entonces, o en otras palabras, el paso en el que se valida el resultado de un comportamiento. En el paradigma Organizar, Actuar, Afirmar, dado representa “Afirmar”.
  • Y Esta palabra clave se usa junto con cualquiera de las palabras clave anteriores. Si tiene dos declaraciones dadas, en lugar de llamar explícitamente a Dado dos veces, puede decir “Dado A y B”.
  • Pero Esta palabra clave se usa junto con Dado, Cuando y Entonces para indicar que algo no debería suceder. Entonces A pero no B.

Todas las palabras clave deben estar en una nueva línea y deben ser la primera palabra en una nueva línea para que el analizador Gherkin las reconozca. Las palabras clave Característica y Escenario deben tener dos puntos inmediatamente después, como se expresa en el siguiente ejemplo. Dado, Cuándo, Entonces y Y no requieren dos puntos.

Además de palabras clave, puede escribir descripciones y comentarios. Las descripciones aparecen después de la palabra clave pero en la misma línea, mientras que los comentarios aparecen en líneas debajo de las palabras clave. Al escribir comentarios de características, es habitual proporcionar reglas explícitas que describan los bordes y las condiciones que conducen a diferentes resultados de comportamientos.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with valid credentials
        Then the user should be logged in

Esquema del escenario #

En algunos casos, es posible que desee volver a ejecutar el mismo escenario una y otra vez, sustituyendo los argumentos. En este caso, Gherkin proporciona varias palabras clave nuevas para adaptarse a esta situación, Esquema del escenario: y Ejemplo:. La palabra clave Scenario Outline le dice a Cucumber que el escenario se ejecutará varias veces sustituyendo los argumentos de una lista. La palabra clave Ejemplos se llama antes de que la lista se anote explícitamente. Los argumentos para los esquemas de escenarios deben estar entre corchetes angulares. En el siguiente ejemplo, tenga en cuenta que los nombres de los argumentos entre corchetes angulares corresponden a los nombres de las columnas enumerados en Ejemplos. Cada columna está separada por barras verticales, con los nombres de las columnas en la primera fila.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Background: The user starts out on the login page
        Given the user is on the login page

    Scenario Outline: The user successfully logs in with their account
         This scenario outlines tests in which various users attempt
         to sign in successfully 

         When the user enters their <username>
         And the user enters their <password>
         Then the user should be successfully logged on

         Examples:
         | username | password |
         | frank    | 1234     |
         | jack     | 4321     |
