Parámetros #

Opción/Bandera Descripción
opción descripción de la opción
-h nombre de host. El valor predeterminado es “localhost”. Si no especifica, la reparación del host se ejecuta en el mismo host desde el que se ejecuta el comando.
-p Puerto JMX. El valor predeterminado es 7199.
-u nombre de usuario. Solo se requiere si la seguridad JMX está habilitada.
-pw clave. Solo se requiere si la seguridad JMX está habilitada.
bandera descripción de la bandera
-local Solo compare y transmita datos de nodos en el centro de datos “local”.
-pr Reparación de “Rango de partición”. Solo repare el rango de token principal para una réplica. Más rápido que reparar todos los rangos de sus réplicas, ya que evita reparar los mismos datos varias veces. Tenga en cuenta que si usa esta opción para reparar un nodo, también debe usarla para el resto de su clúster.
-par Ejecute las reparaciones en paralelo. Hace las reparaciones más rápido, pero restringe significativamente la capacidad del clúster para manejar solicitudes.
-anfitriones Le permite especificar una lista de nodos delimitados por comas desde los que transmitir sus datos. Útil si tiene nodos que se sabe que son “buenos”. Si bien está documentado como una opción válida para Cassandra 2.1+, también funciona con Cassandra 2.0.

Reparaciones anti-entropía de Cassandra:

La reparación anti-entropía en Cassandra tiene dos fases distintas. Para ejecutar reparaciones exitosas y eficientes, es importante comprender ambas.

  • Cálculos de Merkle Tree: calcula las diferencias entre los nodos y sus réplicas.

  • Transmisión de datos: según el resultado de los cálculos de Merkle Tree, los datos están programados para transmitirse de un nodo a otro. Este es un intento de sincronizar los datos entre réplicas.

Detener una reparación:

Puede detener una reparación emitiendo un comando DETENER VALIDACIÓN desde nodetool:

$ nodetool stop validation

¿Cómo puedo saber cuándo se completó la reparación?

Puede comprobar la primera fase de la reparación (cálculos de Merkle Tree) consultando nodetool compactionsstats.

Puede verificar las transmisiones de reparación usando nodetool netstats. Los flujos de reparación también estarán visibles en sus registros. Puede ‘grep’ para ellos en los registros de su sistema de esta manera:

$ grep Entropy system.log

INFO [AntiEntropyStage:1] 2016-07-25 07:32:47,077 RepairSession.java (line 164) [repair #70c35af0-526e-11e6-8646-8102d8573519] Received merkle tree for test_users from /192.168.14.3
INFO [AntiEntropyStage:1] 2016-07-25 07:32:47,081 RepairSession.java (line 164) [repair #70c35af0-526e-11e6-8646-8102d8573519] Received merkle tree for test_users from /192.168.16.5
INFO [AntiEntropyStage:1] 2016-07-25 07:32:47,091 RepairSession.java (line 221) [repair #70c35af0-526e-11e6-8646-8102d8573519] test_users is fully synced
INFO [AntiEntropySessions:4] 2016-07-25 07:32:47,091 RepairSession.java (line 282) [repair #70c35af0-526e-11e6-8646-8102d8573519] session completed successfully

Los flujos de reparación activos también se pueden monitorear con este comando (Bash):

$ while true; do date; diff <(nodetool -h 192.168.0.1 netstats) <(sleep 5 && nodetool -h 192.168.0.1 netstats); done

ref: http://stackoverflow.com/questions/25064717/how-do-i-snow-if-nodetool-repair-is-finished/25081283#25081283

¿Cómo verificar flujos de reparación atascados o huérfanos?

En cada nodo, puede monitorear esto con nodetool tpstats y verificar si hay algo “bloqueado” en las líneas “AntiEntropy”.

$ nodetool tpstats
Pool Name                    Active   Pending      Completed   Blocked  All time blocked
...
AntiEntropyStage                  0         0         854866         0                 0
...
AntiEntropySessions               0         0           2576         0                 0
...

Ejemplos para ejecutar Nodetool Repair #

Uso:

$ nodetool repair [-h | -p | -pw | -u] <flags> [ -- keyspace_name [table_name]]

Opción de reparación predeterminada

$ nodetool repair

Este comando reparará el rango de token principal del nodo actual (es decir, el rango que posee) junto con las réplicas de otros rangos de token que tiene en todas las tablas y todos los espacios de claves en el nodo actual:

Por ej. Si tiene un factor de replicación de 3, entonces un total de 5 nodos estarán involucrados en la reparación: 2 nodos arreglarán 1 rango de partición 2 nodos arreglarán 2 rangos de partición 1 nodo reparará 3 rangos de partición. (El comando se ejecutó en este nodo)

Reparación en Paralelo

$ nodetool repair -par

Este comando ejecutará la misma tarea que la reparación predeterminada, pero ejecutando la reparación en paralelo en los nodos que contienen réplicas.

Reparar rango de token principal

Este comando repara solo el rango de token principal del nodo en todas las tablas y todos los espacios de claves en el nodo actual:

$ nodetool repair -pr

Repare solo el centro de datos local en el que reside el nodo:

$ nodetool repair -pr -local

Repare solo el rango principal para todas las réplicas en todas las tablas y todos los espacios de claves en el nodo actual, solo transmitiendo desde los nodos enumerados:

$ nodetool repair -pr -hosts 192.168.0.2, 192.168.0.3, 192.168.0.4

Repare solo el rango principal para todas las réplicas en el espacio de claves de stackoverflow en el nodo actual:

$ nodetool repair -pr -- stackoverflow

Repare solo el rango principal para todas las réplicas en la tabla test_users del espacio de claves de stackoverflow en el nodo actual:

$ nodetool repair -pr -- stackoverflow test_users
