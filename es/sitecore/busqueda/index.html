La búsqueda de Sitecore se basa en Lucene, lo que brinda la capacidad de crear capacidades de búsqueda muy rápidas para su sitio. En lugar de realizar consultas en una base de datos centralizada (como la base de datos SQL de Sitecore), consulta los archivos de índice de Lucene que se almacenan en el sistema de archivos físicos del servidor web. Sitecore proporciona una capa de abstracción sobre la API de Lucene.NET que incluye un proveedor LINQ que hace que escribir consultas de Lucene sea un proceso simple y familiar para los desarrolladores de .NET. Sitecore se envía con algunos índices estándar configurados que puede ampliar o definir los suyos propios. También puede optar por utilizar SOLR; una plataforma centralizada y escalable construida sobre Lucene.

Crear un filtro de entrada para la búsqueda #

El filtro de entrada se aplica cuando se agrega un elemento al índice de búsqueda y permite especificar si el elemento se incluye o no en el índice.

Ejemplos en los que se puede usar el filtro de entrada: no incluya valores estándar y versiones anteriores del elemento en el índice.

Los filtros de entrada se establecen en la configuración:

<indexing.filterIndex.inbound>
 <processor type="Sitecore.ContentSearch.Pipelines.IndexingFilters.ApplyInboundIndexFilter, Sitecore.ContentSearch"></processor>
</indexing.filterIndex.inbound>

Implementación de código:

public class ApplyInboundIndexVersionFilter : InboundIndexFilterProcessor   
{        
    public override void Process(InboundIndexFilterArgs args)       
    { 
        var item = args.IndexableToIndex as SitecoreIndexableItem;

        if (!item.Item.Versions.IsLatestVersion())            
        {                  
            args.IsExcluded = true;            
        }
    }
}

Se pueden encontrar más ejemplos e información en http://www.sitecore.net/learn/blogs/technical-blogs/sitecore-7-development-team/posts/2013/04/sitecore-7-inbound-and-outbound- filtro-tuberías.aspx

Crear un filtro de salida para la búsqueda #

El filtro de salida se puede utilizar para filtrar los resultados de la búsqueda.

Uno de los ejemplos del uso de filtros salientes es eliminar elementos a los que el usuario no tiene acceso de los resultados de búsqueda.

Los filtros de salida se establecen en la configuración:

<indexing.filterIndex.outbound>
 <processor type="Sitecore.ContentSearch.Pipelines.IndexingFilters.ApplyOutboundSecurityFilter, Sitecore.ContentSearch"></processor>
</indexing.filterIndex.outbound>

Ejemplo de implementación de filtro de salida:

public class ApplyOutboundIndexWorkflowFilter : OutboundIndexFilterProcessor   
{        
    public override void Process(OutboundIndexFilterArgs args)       
    { 
        //You can use args.IsExcluded to remove items from the search results here
    }
}

Se pueden encontrar más ejemplos e información en http://www.sitecore.net/learn/blogs/technical-blogs/sitecore-7-development-team/posts/2013/04/sitecore-7-inbound-and-outbound- filtro-tuberías.aspx

Eliminar todas las versiones anteriores del elemento en el índice al agregar una nueva versión #

De forma predeterminada, Sitecore agrega todas las versiones del elemento a sitecore_master_index. El inconveniente de esto es que si los usuarios usan flujos de trabajo y agregan muchas versiones, todas ellas se agregarán a los resultados de búsqueda en el editor de contenido.

Configuración:

<event name="item:versionAdded" >
              <handler type="FilterPatch.Library.ContentSearch.EventHandler, AssemblyName" method="Execute" />
       </event>

Implementación del controlador

public class EventHandler
    {
        public void Execute(object sender, EventArgs eventArgs)
        {
            var item = Event.ExtractParameter(eventArgs, 0) as Item;

            //If item has less than 2 versions - then skip
            if(item.Versions.Count < 2)
            {
                return;
            }

            var indexableItem = new SitecoreIndexableItem(item);

            var index = ContentSearchManager.GetIndex(indexableItem);

            using (var context = index.CreateDeleteContext())
            {
                foreach(var version in item.Versions.GetVersions(true))
                {
                    if(!version.Versions.IsLatestVersion())
                    {
                        var indexableItemVersion = new SitecoreIndexableItem(version);
                        context.Delete(indexableItemVersion.UniqueId);
                    }
                }
                context.Commit();                
            }
        }
    }

Configuración #

Sitecore se envía con un conjunto de índices estándar preconfigurados que puede ampliar o definir los suyos propios. De los preconfigurados, sitecore_master_index y sitecore_web_index son los de mayor interés para la búsqueda de su sitio. Estos son los índices predefinidos para todos sus elementos de Sitecore en el árbol de sus bases de datos principal y web respectivamente, y están configurados para almacenar todos los campos estándar de un elemento de Sitecore que serán comunes a todas las plantillas.

Puede ver esta configuración del índice web estándar en esta ubicación: <Su sitio>\App_Config\Include\Sitecore.ContentSearch.Lucene.Index.Web.config

Las principales áreas de importancia de la configuración de un índice son:

  • La configuración de campo - Qué campos deben almacenarse en el índice y cómo deben almacenarse.
  • La estrategia - Cómo y cuándo se debe actualizar el índice.
  • The Crawler: la ubicación donde el índice puede obtener sus datos de Sitecore

Configuración de campo

Mirando en la configuración sitecore_web_index, puede ver la siguiente referencia: <configuration ref="contentSearch/indexConfigurations/defaultLuceneIndexConfiguration" />. Esto hace referencia a un archivo de configuración de índice compartido que se encuentra aquí: <Su sitio>\App_Config\Include\Sitecore.ContentSearch.Lucene.DefaultIndexConfiguration.config. Aquí puede ver todos los campos que se incluyen en la configuración estándar.

Básicamente, hay dos formas de definir un campo: el campo se obtiene directamente de un campo de elemento de Sitecore o es un campo calculado. Un campo calculado le permite escribir código para hacer algunos cálculos y almacenar el resultado en el campo. Este código se ejecutará cuando el índice se cree/actualice no cuando se consulte el índice. Esto es particularmente útil si el campo necesita almacenar datos agregados, como conteos, etc.

Dentro del elemento <fieldMap> verá los elementos <fieldNames hint="raw:AddFieldByFieldName"> & <fields hint="raw:AddComputedIndexField">, que contienen los campos obtenidos directamente y los campos calculados respectivamente.

Estrategia

La estrategia de su índice determina cuándo se actualiza su índice. Hay las siguientes opciones para elegir:

  • OnPublishEndAsynchronousStrategy (onPublishEndAsync) - Cuando se publica un elemento, el índice se actualizará de forma asíncrona.
  • SynchronousStrategy (syncMaster) - Cuando se guarda un elemento, el índice se actualizará instantánea y sincrónicamente.
  • IntervalAsynchronousStrategy (intervalAsyncCore/intervalAsyncMaster) - Comprueba periódicamente si hay actualizaciones de elementos y actualiza el índice de forma asíncrona.
  • ManualStrategy - Sin actualizaciones de índice automáticas. Los índices solo se actualizarán manualmente (a través del panel de control o mediante programación)
  • RebuildAfterFullPublishStrategy (rebuildAfterFullPublish) - Después de una publicación, el índice se reconstruirá por completo
  • RemoteRebuildStrategy (remoteRebuild): esta estrategia es para varias instancias de Sitecore. Por ejemplo, si se solicita una reconstrucción desde el servidor de administración de contenido, los servidores de entrega de contenido remoto se suscribirán a este evento y reconstruirán sus propios índices.

De forma predeterminada, el índice maestro está configurado como syncMaster. Esto se debe a que si está en el editor de experiencia guardando elementos y la representación en la página muestra los resultados de un índice, querrá ver los cambios que ha realizado en los elementos inmediatamente en los resultados. El índice web está configurado como ‘onPublishEndAsync’, esto se debe a que los índices de su base de datos web solo necesitan actualizarse cuando los elementos se publican desde la base de datos maestra en la web.

También puedes combinar múltiples estrategias. Por ejemplo, si tiene instancias de Sitecore separadas para su administración de contenido (CM) y entrega de contenido (CD), tendría sentido combinar onPublishEndAsync con remoteRebuild, de modo que los índices de CD también se actualicen cuando se publiquen elementos. como se reconstruye cuando un usuario activa una reconstrucción desde el panel de control del servidor CM.

Puedes elegir tu estrategia usando la siguiente configuración:

<strategies hint="list:AddStrategy">
    <strategy ref="contentSearch/indexConfigurations/indexUpdateStrategies/onPublishEndAsync" />
</strategies>

El rastreador

Esto le permite especificar la ubicación de los datos de Sitecore que desea indexar. El índice web tiene la siguiente configuración por defecto:

<locations hint="list:AddCrawler">
    <crawler type="Sitecore.ContentSearch.SitecoreItemCrawler, Sitecore.ContentSearch">
        <Database>web</Database>
        <Root>/sitecore</Root>
    </crawler>
</locations>

Los dos bits importantes son los elementos <Database> y <Root>. El elemento <Root> le permite especificar la posición inicial en su árbol de Sitecore que el índice debe indexar. En realidad, es probable que tenga un nodo ‘Inicio’ debajo del nodo de contenido al que señalaría esto para que solo indexe contenido/páginas reales en lugar de sus plantillas, etc.
