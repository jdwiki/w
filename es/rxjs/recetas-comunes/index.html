Almacenamiento en caché de respuestas HTTP #

Un caso de uso típico para RxJS es crear solicitudes HTTP y almacenar en caché sus resultados durante un período de tiempo. Además, siempre queremos ejecutar solo una solicitud a la vez y compartir su respuesta.

Por ejemplo, el siguiente código almacena en caché 1 elemento para un máx. 1000ms:

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1);

var counter = 1;    
function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100);
}

function requestCachedHttpResult() {
    return updateRequest;
}

La función makeMockHttpRequest() simula una solicitud HTTP que llega con un retraso de 100ms.

La función requestCachedHttpResult() es donde nos suscribimos para obtener una respuesta real o en caché.

Con .publishReplay(1, 1000) usamos RxJS multicasting para uso interno ReplaySubject y mantenga el elemento 1 para un máximo de 1000ms. Entonces refCount() se usa para mantener siempre solo una suscripción a la fuente que es Observable.defer(). Este Observable se usa para crear una nueva solicitud e incrementa el “contador” para demostrar que los valores almacenados en caché y las nuevas suscripciones comparten los mismos datos.

Cuando queremos obtener datos actuales, llamamos requestCachedHttpResult(). Para asegurarnos de que el observador se completará correctamente después de emitir los datos, utilizamos el operador take(1).

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

Esto crea una única solicitud con mockDataFetch() y se imprime en la consola:

1

Un ejemplo más complicado llamará a múltiples solicitudes en diferentes momentos en los que queremos probar que las conexiones y respuestas HTTP simuladas se comparten.

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 50:", val))
, 50);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 200:", val))
, 200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1200:", val))
, 1200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1500:", val))
, 1500);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 3500:", val))
, 3500);

Ver demostración en vivo: https://jsbin.com/todude/5/edit?js,console

Cada solicitud se envía con retraso y sucede en el siguiente orden:

0 - Primera solicitud que hace que refCount() se suscriba a su fuente que hace la llamada mockDataFetch(). Su respuesta se retrasará en 100ms. En este momento ConnectableObservable dentro del operador publishReplay() tiene un observador.

50: la segunda solicitud también se suscribe a ConnectableObservable. En este momento ConnectableObservable dentro del operador publishReplay() tiene dos Observador. No crea otra solicitud con makeMockHttpRequest() porque refCount() ya está suscrito.

100 - La primera respuesta está lista. Primero se almacena en caché por ReplaySubject y luego se vuelve a emitir a los dos observadores suscritos a ConnectableObservable. Ambos Observadores se completan gracias a take(1) y se dan de baja.

200: se suscribe al ReplaySubject que emite inmediatamente su valor en caché, lo que hace que take(1) complete el Observer y cancele la suscripción de inmediato. No se realizan solicitudes HTTP y no queda ninguna suscripción.

1200 - Lo mismo que el primer evento en 0. En este punto, el valor almacenado en caché se ha descartado porque tiene más de 1000ms.

1500 - Igual que el cuarto evento en 200.

3500 - Lo mismo que el primer evento en 1200.

La salida en consola es la siguiente:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: 2
Response 1500: 2
Response 3500: 3

En RxJS 5, el operador cache() cubrió una funcionalidad similar. Sin embargo, se eliminó en 5.0.0-rc.1 debido a su funcionalidad limitada.

Errores de manejo

Si queremos manejar los errores producidos por el servicio remoto (la función makeMockHttpRequest), debemos detectarlos antes de que se fusionen con la cadena principal de Observable porque cualquier error recibido por ReplaySubject dentro de publishReplay() marcaría su estado interno como detenido (Lea más aquí https://www.wikiod.com/es/rxjs/tema#Sujeto y su estado interno) que definitivamente no es lo que queremos .

En el siguiente ejemplo estamos simulando un error cuando counter === 2 y atrapándolo con el operador catch(). Estamos usando catch() para transformar solo la notificación de error en un siguiente regular para que podamos manejar el error en los observadores:

function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100)
        .map(i => {
            if (i === 2) {
                throw new Error('Invalid URL');
            }
            return i;
        })
        .catch(err => Observable.of(err));
}

Vea la demostración en vivo: https://jsbin.com/kavihu/10/edit?js,console

Esto imprimirá en la consola el siguiente resultado. Observe que los errores se reciben en los controladores siguientes:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: [object Error] { ... }
Response 1500: [object Error] { ... }
Response 3500: 3

Si queremos manejar los errores como notificaciones regulares de error en cada observador, tenemos que volver a lanzarlos después del operador publishReplay() por las razones explicadas anteriormente.

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1)
    .map(val => {
        if (val instanceof Error) {
            throw val;
        }
        return val;
    });

Vea la demostración en vivo: https://jsbin.com/fabosam/5/edit?js,console (tenga en cuenta que también tuvimos que agregar devoluciones de llamada de error para cada observador).

Response 0: 1
Response 50: 1
Response 200: 1
error callback: Error: Invalid URL
error callback: Error: Invalid URL
Response 3500: 3

Envío de múltiples solicitudes HTTP paralelas #

Un caso de uso muy común en las aplicaciones web es realizar múltiples solicitudes asíncronas (por ejemplo, HTTP) y recopilar sus resultados a medida que llegan o todos a la vez (por ejemplo, en Angular2 con el [servicio HTTP] (https://angular. io/docs/ts/latest/guide/server-communication.html)).

1. Recopilación de respuestas asíncronas una por una a medida que llegan

Esto normalmente se hace con el operador mergeMap() que toma una función de proyección que tiene que devolver un Observable. El operador mergeMap() se suscribe internamente a cada Observable inmediatamente, incluso si el Observable anterior aún no se ha completado.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .subscribe(val => console.log(val));

Esto imprime las respuestas a la consola en un orden diferente debido al retraso aleatorio:

Response from url-3
Response from url-4
Response from url-2
Response from url-1

Vea la demostración en vivo: https://jsbin.com/xaqudan/2/edit?js,console

Cada respuesta (elemento emitido a través de la llamada siguiente) es reemitida por mergeMap() inmediatamente.

Para nuestro propósito de enviar múltiples solicitudes HTTP, es útil mencionar que mergeMap() puede tomar tres argumentos en total:

  1. La función de proyección que necesita devolver un Observable.
  2. La función selectora de resultados que nos permite modificar el resultado antes de seguir emitiéndolo.
  3. El número de Observables suscritos simultáneamente.

Controlando el número de solicitudes paralelas

Con el tercer argumento podemos controlar cuántas solicitudes paralelas manejaremos (asumiendo que cada Observable que realiza una solicitud HTTP es “frío”).

En el siguiente ejemplo, ejecutaremos solo 2 solicitudes al mismo tiempo.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url), undefined, 2)
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Ver demostración en vivo: https://jsbin.com/sojejal/4/edit?js,console

Tenga en cuenta que las dos primeras solicitudes se completaron después de 1 s, mientras que las otras dos después de 2 s.

[1004, "Response from url-1"]
[1010, "Response from url-2"]
[2007, "Response from url-3"]
[2012, "Response from url-4"]

Errores de manejo

Si alguno de los Observables de origen falla (envía una notificación de “error”), el “mergeMap()” vuelve a enviar el error como “error”. En caso de que queramos que cada Observable falle correctamente, debemos usar, por ejemplo, catch() operador.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url).catch(() => Observable.empty()))
  .subscribe(val => console.log(val));

La respuesta para url-3 arroja un error que se envía como notificación de error. Esto luego es capturado por el operador catch() y reemplazado por Observable.empty(), que es solo una notificación completa. Por este motivo se ignora esta respuesta.

La salida para este ejemplo es la siguiente:

Response from url-4
Response from url-1
Response from url-2

Vea la demostración en vivo: https://jsbin.com/kuqumud/4/edit?js,console

2. Recopilación de todas las respuestas asíncronas a la vez

Siguiendo los ejemplos anteriores, podríamos recopilar todas las respuestas con el operador toArray().

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .toArray()
  .subscribe(val => console.log(val));

Sin embargo, usar el operador toArray() tiene una consecuencia importante. Si el suscriptor recibe los resultados no solo se controla completando todas las solicitudes HTTP, sino también completando el Observable fuente (Observable.from en nuestro caso). Esto significa que no podemos usar Observables de origen que nunca se completen (por ejemplo, Observable.fromEvent).

Otra forma de lograr el mismo resultado es usando Observable.forkJoin() que toma como argumento una matriz de Observables a los que queremos suscribirnos y esperar hasta que todos emitan al menos un valor y se completen.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];
var observables = urls.map(url => mockHTTPRequest(url));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Esto imprime todas las respuestas como una sola matriz:

["Response from url-1", "Response from url-2", "Response from url-3", "Response from url-4"]

Ver demostración en vivo: https://jsbin.com/fomoye/2/edit?js,console

Observable.forkJoin() también toma como argumento opcional una función selectora de resultados que nos permite modificar el resultado final antes de seguir emitiéndolo:

Observable.forkJoin(observables, (...results) => {
    return results.length;
  })
  .subscribe(val => console.log(val));

Esto se imprime en la consola:

4

Ver demostración en vivo: https://jsbin.com/muwiqic/1/edit?js,console

Tenga en cuenta que el argumento para la función del selector de resultados está desempaquetado.

Errores de manejo

Para el manejo de errores podemos usar el mismo enfoque que en el ejemplo anterior con el operador catch().

Sin embargo, hay una cosa importante a tener en cuenta. forkJoin() requiere que cada fuente Observable emita al menos un valor. Si usáramos catch(() => Observable.empty()) como lo hicimos antes, forkJoin() nunca emitiría nada porque Observable.empty() es solo una notificación completa. Esta es la razón por la que necesitamos usar, por ejemplo, Observable.of(null), que es un valor null seguido de una notificación completa.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

var observables = urls.map(url => mockHTTPRequest(url).catch(() => Observable.of(null)));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Vea la demostración en vivo: https://jsbin.com/yidiked/2/edit?js,console

Esto se imprime en la consola:

["Response from url-1", "Response from url-2", null, "Response from url-4"]

Observe que el error se reemplaza por null. Si usáramos solo Observable.empty(), forkJoin() nunca emitiría nada.

Descartar llamadas de descanso lentas/obsoletas #

Un caso de uso común es descartar ciertas llamadas de descanso, que ya no son necesarias después de ciertas entradas de usuario. El ejemplo más destacado sería cuando un usuario utiliza alguna función de búsqueda, realiza una solicitud, realiza otra solicitud y, por alguna razón, la primera solicitud llega después de la segunda solicitud y la aplicación muestra los datos obsoletos de la solicitud anterior.

Este es un caso de uso perfecto para el operador switchMap.

searchInput$
    .switchMap(() => makeRestCall());

En este caso, el flujo ‘cambiará’ a la llamada de descanso, pero solo hasta que se emitan nuevos datos en searchInput$, entonces el flujo dentro de switchMap se descarta y se realiza una nueva llamada de descanso. Por lo tanto, solo se considerará un resultado de descanso si finalizó antes del siguiente clic.

Y aquí hay un ejemplo simulado completo:

// some initial data-mocking
const Observable = Rx.Observable;
var counter = 1;
function mockDataFetch() {
    return Observable.of(counter++)
        .delay(500);
}

// the recipe

const searchInput$ = new Rx.Subject();
searchInput$
    .do(searchInput => console.log("Searching for " + searchInput))
    .switchMap(searchInput => mockDataFetch()
              .map(result => ({result, searchInput}))
     )
    .do(data => console.log("Received result for " + data.searchInput + ": " + data.result))
    .subscribe();

// simulating search inputs
searchInput$.next("google");
setTimeout(() => searchInput$.next("froogle"), 600);
setTimeout(() => searchInput$.next("doodle"), 800);
setTimeout(() => searchInput$.next("poodle"), 1000);
setTimeout(() => searchInput$.next("noodle"), 1600);

Vea la demostración en vivo: https://jsbin.com/suzakahoro/1/edit?js,console

Envío de múltiples solicitudes HTTP secuenciales #

Hacer una secuencia de solicitudes HTTP tiene dos razones principales:

  • Las solicitudes dependen unas de otras (el resultado de una solicitud es necesario para una solicitud consecutiva).

  • Queremos distribuir la carga del servidor en varias solicitudes.

1. Realización de múltiples solicitudes dependientes

Esto se puede realizar usando el operador concatMap() para transformar una respuesta a los parámetros requeridos para la solicitud consecutiva.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

function timestamp() {
  return (new Date()).getTime() - start;
}

var start = (new Date()).getTime();

Observable.of('url-1') 
  // first request
  .concatMap(url => {
    console.log(timestamp() + ': Sending request to ' + url);
    return mockHTTPRequest(url);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // second request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create new requests url
    return mockHTTPRequest(newUrl);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // third request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create another requests url
    return mockHTTPRequest(newUrl);
  })
  .subscribe(response => console.log(timestamp() + ': ' + response));

El operador concatMap() se suscribe internamente al Observable devuelto desde su función de proyección y espera hasta que se completa mientras vuelve a emitir todos sus valores.

Este ejemplo marca la hora de cada solicitud y respuesta:

3: Sending request to url-1
1010: Response from url-1
1011: Sending request to Response from url-1
2014: Response from url-19
2015: Sending request to Response from url-19
3017: Response from url-20

Vea la demostración en vivo: https://jsbin.com/fewidiv/6/edit?js,console

Errores de manejo

Si alguna de las solicitudes HTTP falla, obviamente no queremos continuar porque no podemos construir la siguiente solicitud.

2. Realización de solicitudes consecutivas

En caso de que no estemos interesados ​​en la respuesta de la solicitud HTTP anterior, podemos tomar solo una matriz de URL y ejecutarlas una tras otra.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Este ejemplo imprime respuestas con marca de tiempo:

[1006, "Response from url-1"]
[2012, "Response from url-2"]
[3014, "Response from url-3"]
[4016, "Response from url-4"]

Vea la demostración en vivo: https://jsbin.com/kakede/3/edit?js,console

Retrasar llamadas consecutivas

También podríamos querer hacer un pequeño retraso entre cada solicitud. En tal caso, necesitamos agregar delay() después de cada llamada mockHTTPRequest().

Observable.from(urls)
  .concatMap(url => {
    return mockHTTPRequest(url)
      .do(response => console.log(((new Date()).getTime() - start) + ': Sending request to ' + url))
      .delay(500);
  })
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Esto imprime para consolar el siguiente resultado:

2024: Sending request to url-1
[2833, "Response from url-1"]
4569: Sending request to url-2
[5897, "Response from url-2"]
7880: Sending request to url-3
[8674, "Response from url-3"]
9789: Sending request to url-4
[10796, "Response from url-4"]

Vea la demostración en vivo: https://jsbin.com/kakede/4/edit?js,console

Errores de manejo

Si simplemente queremos ignorar cuando alguna de las solicitudes HTTP falla, tenemos que encadenar catch() después de cada mockHTTPRequest().

function mockHTTPRequest(url) {
  if (url == 'url-3') {
    return Observable.throw(new Error(`Request ${url} failed.`));
  } else {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
  }
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url).catch(obs => Observable.empty()))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Esto simplemente ignora la llamada url-3:

[1004, "Response from url-1"]
[2012, "Response from url-2"]
[3016, "Response from url-4"]

Ver demostración en vivo: https://jsbin.com/jowiqo/2/edit?js,console

Si no usáramos el operador catch(), la url-3 haría que la cadena enviara una notificación de error y la última url-4 no se ejecutaría.

Ver demostración en vivo: https://jsbin.com/docapim/3/edit?js,console

Limitación de velocidad #

Un problema común con los servicios remotos es la limitación de velocidad. El servicio remoto nos permite enviar solo un número limitado de solicitudes o cantidad de datos por período de tiempo.

En RxJS 5, el operador bufferTime proporciona una funcionalidad muy similar y especialmente si deje el segundo parámetro sin especificar (define con qué frecuencia queremos crear un nuevo búfer. Si lo dejamos sin definir/nulo, creará un nuevo búfer justo después de emitir el actual).

Un uso típico de bufferTime se verá así:

bufferTime(1000, null, 5)

Esto almacenará elementos hasta que se cumpla una de las dos condiciones. Luego emitirá el búfer y comenzará otro:

  • el operador ha estado recopilando elementos durante 1000ms
  • el operador ya ha recogido 5 elementos

Para fines de demostración, podemos crear una fuente Observable que emita muy rápido, por lo que bufferTime alcanzará el límite de tamaño (5) y emitirá más de una vez cada 1000ms:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

Luego lo encadenaremos con bufferTime y concatMap. El operador concatMap es donde forzamos el retraso de 1000ms:

const startTime = (new Date()).getTime();

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.of(buffer).delay(1000))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Vea la demostración en vivo: https://jsbin.com/kotibow/3/edit?js,console

Agregamos también timestamp() para ver los tiempos de emisión para asegurarnos de que el retraso sea realmente de al menos 1000ms.

Tenga en cuenta que no tuvimos que usar Observable.of(buffer) en absoluto. Lo estamos usando aquí solo para verificar manualmente que la cantidad de elementos almacenados en el búfer sea correcta.

Desde la salida de la consola podemos ver que el retraso entre dos emisiones es de aproximadamente 1000ms:

Timestamp { value: [ 1, 2, 3, 4, 5 ], timestamp: 1475 }
Timestamp { value: [ 6, 7, 8, 9, 10 ], timestamp: 2564 }
Timestamp { value: [ 11, 12, 13, 14, 15 ], timestamp: 3567 }
Timestamp { value: [ 16, 17, 18, 19, 20 ], timestamp: 4572 }
Timestamp { value: [ 21, 22, 23, 24, 25 ], timestamp: 5573 }
Timestamp { value: [], timestamp: 6578 }

Ahora también podemos probar una situación en la que la fuente emite lentamente, por lo que el operador bufferTime alcanzará la condición de intervalo máximo:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(300));

Vea la demostración en vivo: https://jsbin.com/tuwowan/2/edit?js,console

Luego, la salida debería comenzar después de aproximadamente 2s porque tomó 1s para que el operador bufferTime emitiera y luego agregamos el retraso de 1s;

Timestamp { value: [ 1, 2, 3 ], timestamp: 2017 }
Timestamp { value: [ 4, 5, 6 ], timestamp: 3079 }
Timestamp { value: [ 7, 8, 9, 10 ], timestamp: 4088 }
Timestamp { value: [ 11, 12, 13 ], timestamp: 5093 }

Si quisiéramos usar este enfoque en una aplicación del mundo real, colocaríamos la llamada remota en el operador concatMap. De esta manera podemos controlar si queremos forzar el retraso de 1s entre solicitudes o respuestas del servicio remoto.

Por ejemplo, podemos forzar el retraso mínimo de 1s entre solicitudes usando forkJoin en la devolución de llamada concatMap:

function mockHTTPRequest(buffer) {
  return Observable.of(true).delay(Math.random() * 1500)
}

const startTime = (new Date()).getTime();
const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.forkJoin(
    mockHTTPRequest(buffer),
    Observable.of(buffer).delay(1000)
  ))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Ver demostración en vivo: https://jsbin.com/xijaver/edit?js,console

Gracias a forkJoin, concatMap debe esperar a que se completen ambos Observables.

Por otro lado, si quisiéramos forzar un retraso de ‘1 s’ entre las respuestas, simplemente añadiríamos el operador ‘delay()’ después de ‘mockHTTPRequest()’:

.concatMap(buffer => mockHTTPRequest(buffer).delay(1000))

Ver demostración en vivo: https://jsbin.com/munopot/2/edit?js,console
