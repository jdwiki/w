Es fácil ver los ’tipos’ como una tabla en una base de datos SQL, donde el ‘índice’ es la base de datos SQL. Sin embargo, esa no es una buena manera de abordar los `tipos.

Todo sobre los tipos #

De hecho, los tipos son literalmente solo un campo de metadatos agregado a cada documento por Elasticsearch: _type. Los ejemplos anteriores crearon dos tipos: my_type y my_other_type. Eso significa que cada documento asociado con los tipos tiene un campo adicional definido automáticamente como "_type": "my_type"; esto se indexa con el documento, lo que lo convierte en un campo que se puede buscar o filtrar, pero no afecta el documento sin procesar en sí, por lo que su aplicación no necesita preocuparse por eso.

Todos los tipos viven en el mismo índice y, por lo tanto, en los mismos fragmentos colectivos del índice. Incluso a nivel de disco, viven en los mismos archivos. La única separación que proporciona la creación de un segundo tipo es lógica. Cada tipo, ya sea único o no, debe existir en las asignaciones y todas esas asignaciones deben existir en el estado de su clúster. Esto consume memoria y, si cada tipo se actualiza dinámicamente, consume rendimiento a medida que cambian las asignaciones.

Como tal, es una buena práctica definir solo un único tipo a menos que realmente necesite otros tipos. Es común ver escenarios en los que se desean varios tipos. Por ejemplo, imagine que tiene un índice de automóvil. Puede resultarle útil desglosarlo en varios tipos:

  • BMW
  • caza -honda
  • mazda
  • mercedes
  • nissan
  • guardabosques
  • toyota
  • …

De esta manera, puede buscar todos los automóviles o limitarlo por fabricante a pedido. La diferencia entre esas dos búsquedas es tan simple como:

GET /cars/_search

y

GET /cars/bmw/_search

Lo que no es obvio para los nuevos usuarios de Elasticsearch es que la segunda forma es una especialización de la primera forma. Literalmente se reescribe como:

GET /cars/_search
{
  "query": {
    "bool": {
      "filter": [
        {
          "term" : {
            "_type": "bmw"
          }
        }
      ]
    }
  }
}

Simplemente filtra cualquier documento que no esté indexado con un campo _type cuyo valor sea bmw. Dado que cada documento está indexado con su tipo como el campo _tipo, esto sirve como un filtro bastante simple. Si se hubiera proporcionado una búsqueda real en cualquiera de los ejemplos, el filtro se agregaría a la búsqueda completa según corresponda.

Como tal, si los tipos son idénticos, es mucho mejor proporcionar un solo tipo (por ejemplo, “fabricante” en este ejemplo) e ignorarlo de manera efectiva. Luego, dentro de cada documento, proporcione explícitamente un campo llamado make o el nombre que prefiera y fíltrelo manualmente cuando quiera limitarse a él. Esto reducirá el tamaño de sus asignaciones a 1/n donde n es el número de tipos separados. Agrega otro campo a cada documento, en beneficio de un mapeo simplificado.

En Elasticsearch 1.x y 2.x, dicho campo debe definirse como

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "string",
        "index": "not_analyzed"
      }
    }
  }
}
  1. El nombre es arbitrario.
  2. El nombre es arbitrario y podría coincidir con el nombre del tipo si lo quisiera también.

En Elasticsearch 5.x, lo anterior seguirá funcionando (está en desuso), pero la mejor manera es usar:

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "keyword"
      }
    }
  }
}
  1. El nombre es arbitrario.
  2. El nombre es arbitrario y podría coincidir con el nombre del tipo si lo quisiera también.

Los tipos deben usarse con moderación dentro de sus índices porque infla las asignaciones de índice, generalmente sin mucho beneficio. Debe tener al menos uno, pero no hay nada que diga que debe tener más de uno.

Preguntas comunes #

  • ¿Qué sucede si tengo dos (o más) tipos que son en su mayoría idénticos, pero que tienen algunos campos únicos por tipo?

En el nivel de índice, no hay diferencia entre un tipo que se usa con algunos campos que se usan escasamente y entre varios tipos que comparten un montón de campos no dispersos con algunos que no se comparten (lo que significa que el otro tipo ni siquiera usa el campo (s)).

Dicho de otra manera: un campo escasamente utilizado es escaso en el índice independientemente de los tipos. La escasez no beneficia (o realmente perjudica) al índice solo porque está definido en un tipo separado.

Simplemente debe combinar estos tipos y agregar un campo de tipo separado.

  • ¿Por qué los tipos separados necesitan definir campos exactamente de la misma manera?

Porque cada campo en realidad solo se define una vez en el nivel de Lucene, independientemente de cuántos tipos haya. El hecho de que existan tipos es una característica de Elasticsearch y es solo una separación lógica.

  • ¿Puedo definir tipos separados con el mismo campo definido de manera diferente?

No. Si logra encontrar una manera de hacerlo en ES 2.x o posterior, entonces [debe abrir un informe de error] (https://github.com/elastic/elasticsearch/issues). Como se señaló en la pregunta anterior, Lucene los ve a todos como un solo campo, por lo que no hay forma de hacer que esto funcione correctamente.

ES 1.x dejó esto como un requisito implícito, lo que permitió a los usuarios crear condiciones en las que las asignaciones de un fragmento en un índice en realidad diferían de otro fragmento en el mismo índice. Esta fue efectivamente una condición de carrera y podría conducir a problemas inesperados.

Excepciones a la regla #

  • Los documentos principal/secundario requieren que se utilicen tipos separados dentro del mismo índice.
    • The parent lives in one type.
    • The child lives in a separate type (but each child lives in the same shard as its parent).
  • Casos de uso extremadamente especializados en los que la creación de toneladas de índices no es deseable y el impacto de los campos dispersos es preferible a la alternativa.
    • For example, the Elasticsearch monitoring plugin, Marvel (1.x and 2.x) or X-Pack Monitoring (5.x+), monitors Elasticsearch itself for changes in the cluster, nodes, indices, specific indices (the index level), and even shards. It could create 5+ indices each day to isolate those documents that have unique mappings or it could go against best practices to reduce cluster load by sharing an index (note: the number of defined mappings is effectively the same, but the number of created indices is reduced from n to 1).
    • This is an advanced scenario, but you must consider the shared field definitions across types!

Creación explícita de un índice con un tipo #

El ejemplo utiliza HTTP básico, que se traduce fácilmente a cURL y otras aplicaciones HTTP. También coinciden con la sintaxis Sense, que cambiará de nombre a Consola en Kibana 5.0.

Nota: El ejemplo inserta <#> para ayudar a llamar la atención sobre las partes. ¡Esos deben eliminarse si lo copia!

PUT /my_index <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "integer"
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "float"
            }
          }
        }
      }
    }
  },
  "my_other_type": {
    "properties": {
      "field1": {
        "type": "long" <3>
      },
      "field3": { <4>
        "type": "double"
      }
    }
  }
}
  1. Esto es crear el índice utilizando el punto final de creación de índice.
  2. Esto es crear el tipo.
  3. Los campos compartidos en types dentro del mismo index deben compartir la misma definición. ES 1.x no aplicaba estrictamente este comportamiento, pero era un requisito implícito. ES 2.x y superior imponen estrictamente este comportamiento.
  4. Los campos únicos en types están bien.

Los índices (o índices) contienen tipos. Los tipos son un mecanismo conveniente para separar documentos, pero requieren que defina, ya sea de forma dinámica, automática o explícita, una asignación para cada tipo que use. Si define 15 tipos en un índice, entonces tiene 15 asignaciones únicas.

Consulte los comentarios para obtener más detalles sobre este concepto y por qué puede o no querer usar tipos.

Creando dinámicamente un índice con un tipo #

El ejemplo utiliza HTTP básico, que se traduce fácilmente a cURL y otras aplicaciones HTTP. También coinciden con la sintaxis Sense, que cambiará de nombre a Consola en Kibana 5.0.

Nota: El ejemplo inserta <#> para ayudar a llamar la atención sobre las partes. ¡Esos deben eliminarse si lo copia!

DELETE /my_index <1>

PUT /my_index/my_type/abc123 <2>
{
  "field1" : 1234, <3>
  "field2" : 456,
  "object1" : {
    "field1" : 7.8 <4>
  }
}
  1. En caso de que ya exista (debido a un ejemplo anterior), elimine el índice.
  2. Indexe un documento en el índice, my_index, con el tipo my_type y el ID abc123 (puede ser numérico, pero siempre es una cadena).
    • By default, dynamic index creation is enabled by simply indexing a document. This is great for development environments, but it is not necessarily good for production environments.
  3. Este campo es un número entero, por lo que la primera vez que se ve debe mapearse. Elasticsearch siempre asume el tipo más amplio para cualquier tipo entrante, por lo que se asignaría como “largo” en lugar de “entero” o “corto” (ambos podrían contener “1234” y “456”).
  4. Lo mismo es cierto para este campo también. Se asignará como “doble” en lugar de como “flotante”, como podría desear.

Este índice y tipo creados dinámicamente coinciden aproximadamente con la asignación definida en el primer ejemplo. Sin embargo, es fundamental entender cómo <3> y <4> afectan las asignaciones definidas automáticamente.

Puede seguir esto agregando otro tipo dinámicamente al mismo índice:

PUT /my_index/my_other_type/abc123 <1>
{
  "field1": 91, <2>
  "field3": 4.567
}
  1. El tipo es la única diferencia con el documento anterior. ¡La identificación es la misma y está bien! No tiene relación con el otro abc123 aparte de que sucede que está en el mismo índice.
  2. field1 ya existe en el índice, por lo que debe ser el mismo tipo de campo definido en los otros tipos. Enviar un valor que fuera una cadena o no un entero fallaría (por ejemplo, "field1": "this is some text" o "field1": 123.0).

Esto crearía dinámicamente las asignaciones para my_other_type dentro del mismo índice, my_index.

Nota: siempre es más rápido definir las asignaciones por adelantado en lugar de que Elasticsearch las realice dinámicamente en el momento del índice.

El resultado final de indexar ambos documentos sería similar al del primer ejemplo, pero los tipos de campo serían diferentes y, por lo tanto, un desperdicio:

GET /my_index/_mappings <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "long" <3>
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "double" <4>
            }
          }
        }
      }
    }
  },
  "my_other_type": { <5>
    "properties": {
      "field1": {
        "type": "long"
      },
      "field3": {
        "type": "double"
      }
    }
  }
}
  1. Esto usa el extremo _mappings para obtener las asignaciones del índice que creamos.
  2. Creamos dinámicamente my_type en el primer paso de este ejemplo.
  3. field2 ahora es long en lugar de integer porque no lo definimos por adelantado. Esto puede resultar un desperdicio en el almacenamiento en disco.
  4. objeto1.campo1 ahora es un doble por la misma razón que #3 con las mismas ramificaciones que #3.
    • Technically, a long can be compressed in a lot of cases. However, a double cannot be compressed due to it being a floating point number.
  5. También creamos dinámicamente my_other_type en el segundo paso de este ejemplo. Su mapeo es el mismo porque ya estábamos usando long y double.
    • Remember that field1 must match the definition from my_type (and it does).
    • field3 is unique to this type, so it has no such restriction.
