Elasticsearch viene con un conjunto de valores predeterminados que brindan una buena experiencia lista para usar para el desarrollo. La declaración implícita allí es que no es necesariamente excelente para la producción, que debe adaptarse a sus propias necesidades y, por lo tanto, no se puede predecir.

La configuración predeterminada facilita la descarga y ejecución de múltiples nodos en la misma máquina sin ningún cambio de configuración.

¿Dónde están los ajustes? #

Dentro de cada instalación de Elasticsearch hay un config/elasticsearch.yml. Ahí es donde se encuentran las siguientes [configuraciones] (https://www.elastic.co/guide/en/elasticsearch/reference/current/setup-configuration.html):

  • clúster.nombre
    • The name of the cluster that the node is joining. All nodes in the same cluster must share the same name.
    • Currently defaults to elasticsearch.
  • nodo.*
    • node.name
      • If not supplied, a random name will be generated each time the node starts. This can be fun, but it is not good for production environments.
      • Names do not have to be unique, but they should be unique.
    • node.master
      • A boolean setting. When true, it means that the node is an eligible master node and it can be the elected master node.
      • Defaults to true, meaning every node is an eligible master node.
    • node.data
      • A boolean setting. When true, it means that the node stores data and handles search activity.
      • Defaults to true.
  • ruta.*
    • path.data
      • The location that files are written for the node. All nodes use this directory to store metadata, but data nodes will also use it to store/index documents.
      • Defaults to ./data.
        • This means that data will be created for you as a peer directory to config inside of the Elasticsearch directory.
    • path.logs
      • The location that log files are written.
      • Defaults to ./logs.
  • red.*

    • network.host

      • Defaults to _local_, which is effectively localhost.
        • This means that, by default, nodes cannot be communicated with from outside of the current machine!

    • network.bind_host

      • Potentially an array, this tells Elasticsearch what addresses of the current machine to bind sockets too.
        • It is this list that enables machines from outside of the machine (e.g., other nodes in the cluster) to talk to this node.
      • Defaults to network.host.

    • network.publish_host

      • A singular host that is used to advertise to other nodes how to best communicate with this node.
        • When supplying an array to network.bind_host, this should be the one host that is intended to be used for inter-node communication.
      • Defaults to network.host`.
  • descubrimiento.zen.*
    • discovery.zen.minimum_master_nodes
      • Defines quorum for master election. This must be set using this equation: (M / 2) + 1 where M is the number of eligible master nodes (nodes using node.master: true implicitly or explicitly).
      • Defaults to 1, which only is valid for a single node cluster!
    • discovery.zen.ping.unicast.hosts
      • The mechanism for joining this node to the rest of a cluster.
      • This should list eligible master nodes so that a node can find the rest of the cluster.
      • The value that should be used here is the network.publish_host of those other nodes.
      • Defaults to localhost, which means it only looks on the local machine for a cluster to join.

¿Qué tipo de configuraciones existen? #

Elasticsearch proporciona tres tipos diferentes de configuraciones:

  • Configuración de todo el clúster
    • These are settings that apply to everything in the cluster, such as all nodes or all indices.
  • Configuración de nodos
    • These are settings that apply to just the current node.
  • Configuración de índice
    • These are settings that apply to just the index.

Dependiendo de la configuración, puede ser:

  • Cambiado dinámicamente en tiempo de ejecución
  • Cambiado después de un reinicio (cerrar/abrir) del índice
    • Some index-level settings do not require the index to be closed and reopened, but might require the index to be forceably re-merged for the setting to apply.
      • The compression level of an index is an example of this type of setting. It can be changed dynamically, but only new segments take advantage of the change. So if an index will not change, then it never takes advantage of the change unless you force the index to recreate its segments.
  • Cambiado después de un reinicio del nodo
  • Cambiado después de un reinicio del clúster
  • Nunca cambió

Consulte siempre la documentación de su versión de Elasticsearch para saber qué puede o no puede hacer con una configuración.

¿Cómo puedo aplicar la configuración? #

Puede establecer la configuración de varias maneras, algunas de las cuales no se sugieren:

  • Argumentos de línea de comando

En Elasticsearch 1.x y 2.x, puede enviar la mayoría de las configuraciones como propiedades del sistema Java con el prefijo es.:

$ bin/elasticsearch -Des.cluster.name=my_cluster -Des.node.name=`hostname`

En Elasticsearch 5.x, esto cambia para evitar el uso de propiedades del sistema Java, en lugar de usar un tipo de argumento personalizado con -E en lugar de -Des.:

$ bin/elasticsearch -Ecluster.name=my_cluster -Enode.name=`hostname`

Este enfoque para aplicar la configuración funciona muy bien cuando se usan herramientas como Puppet, Chef o Ansible para iniciar y detener el clúster. Sin embargo, funciona muy mal cuando se hace manualmente.

  • Configuración YAML
    • Shown in examples
  • Configuración dinámica
    • Shown in examples

El orden en que se aplican los ajustes es el orden más dinámico:

  1. Configuración transitoria
  2. Configuración persistente
  3. Configuración de la línea de comandos
  4. Configuración YAML (estática)

Si la configuración se establece dos veces, una vez en cualquiera de esos niveles, se aplica el nivel más alto.

Configuración de índice dinámico para múltiples índices al mismo tiempo #

Puede aplicar el mismo cambio que se muestra en el ejemplo de Configuración de índice a todos los índices existentes con una solicitud, o incluso a un subconjunto de ellos:

PUT /*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

o

PUT /_all/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

o

PUT /_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Si prefiere hacerlo también de manera más selectiva, puede seleccionar varios sin suministrar todos:

PUT /logstash-*,my_other_index,some-other-*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Configuración estática de búsqueda elástica #

Elasticsearch usa un archivo de configuración YAML (Yet Another Markup Language) que se puede encontrar dentro del directorio predeterminado de Elasticsearch ([las instalaciones de RPM y DEB cambian esta ubicación, entre otras cosas] (https://www.elastic.co/guide/en/elasticsearch /reference/current/setup-dir-layout.html)).

Puede establecer la configuración básica en config/elasticsearch.yml:

# Change the cluster name. All nodes in the same cluster must use the same name!
cluster.name: my_cluster_name

# Set the node's name using the hostname, which is an environment variable!
# This is a convenient way to uniquely set it per machine without having to make
#  a unique configuration file per node.
node.name: ${HOSTNAME}

# ALL nodes should set this setting, regardless of node type
path.data: /path/to/store/data

# This is a both a master and data node (defaults)
node.master: true
node.data: true

# This tells Elasticsearch to bind all sockets to only be available
#  at localhost (default)
network.host: _local_

Configuración de clúster dinámico persistente #

Si necesita aplicar una configuración dinámicamente después de que el clúster ya se haya iniciado, y en realidad se puede configurar dinámicamente, entonces puede configurarla usando la API _cluster/settings.

La configuración persistente es uno de los dos tipos de configuración de todo el clúster que se pueden aplicar. Una configuración persistente sobrevivirá a un reinicio completo del clúster.

Nota: No todas las configuraciones se pueden aplicar dinámicamente. Por ejemplo, el nombre del clúster no se puede cambiar dinámicamente. La mayoría de las configuraciones a nivel de nodo tampoco se pueden establecer dinámicamente (porque no se pueden orientar individualmente).

Esta no es la API que se debe usar para establecer la configuración de nivel de índice. Puede decir que la configuración es una configuración de nivel de índice porque debe comenzar con index.. Configuraciones cuyos nombres tienen la forma de índices. son configuraciones de todo el clúster porque se aplican a todos los índices.

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}

Advertencia: En Elasticsearch 1.x y 2.x, no puede desestablecer una configuración persistente.

Afortunadamente, esto se ha mejorado en Elasticsearch 5.x y ahora puede eliminar una configuración configurándola en null:

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}

Una configuración no configurada volverá a su valor predeterminado, o a cualquier valor definido en un nivel de prioridad más bajo (por ejemplo, configuración de la línea de comandos).

Configuración de clúster dinámico transitorio #

Si necesita aplicar una configuración dinámicamente después de que el clúster ya se haya iniciado, y en realidad se puede configurar dinámicamente, entonces puede configurarla usando la API _cluster/settings.

La configuración transitoria es uno de los dos tipos de configuración de todo el clúster que se pueden aplicar. Una configuración transitoria no sobrevivirá a un reinicio completo del clúster.

Nota: No todas las configuraciones se pueden aplicar dinámicamente. Por ejemplo, el nombre del clúster no se puede cambiar dinámicamente. La mayoría de las configuraciones a nivel de nodo tampoco se pueden establecer dinámicamente (porque no se pueden orientar individualmente).

Esta no es la API que se debe usar para establecer la configuración de nivel de índice. Puede decir que la configuración es una configuración de nivel de índice porque debe comenzar con index.. Configuraciones cuyos nombres tienen la forma de índices. son configuraciones de todo el clúster porque se aplican a todos los índices.

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "none"
  }
}

Advertencia: en Elasticsearch 1.x y 2.x, no puede anular una configuración transitoria sin reiniciar el clúster completo.

Afortunadamente, esto se ha mejorado en Elasticsearch 5.x y ahora puede eliminar una configuración configurándola en nulo:

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": null
  }
}

Una configuración no configurada volverá a su valor predeterminado, o cualquier valor definido en un nivel de prioridad más bajo (por ejemplo, configuración “persistente”).

Configuración del índice #

Las configuraciones de índice son aquellas configuraciones que se aplican a un solo índice. Tales configuraciones comenzarán con index.. La excepción a esa regla es number_of_shards y number_of_replicas, que también existen en forma de index.number_of_shards e index.number_of_replicas.

Como sugiere el nombre, la configuración de nivel de índice se aplica a un solo índice. Algunas configuraciones se deben aplicar en el momento de la creación porque no se pueden cambiar dinámicamente, como la configuración index.number_of_shards, que controla la cantidad de fragmentos primarios para el índice.

PUT /my_index
{
  "settings": {
    "index.number_of_shards": 1,
    "index.number_of_replicas": 1
  }
}

o, en un formato más conciso, puede combinar prefijos clave en cada .:

PUT /my_index
{
  "settings": {
    "index": {
      "number_of_shards": 1,
      "number_of_replicas": 1
    }
  }
}

Los ejemplos anteriores crearán un índice con la configuración proporcionada. Puede cambiar dinámicamente la configuración por índice utilizando el punto final del índice _settings. Por ejemplo, aquí cambiamos dinámicamente la [configuración de registro lento] (https://www.elastic.co/guide/en/elasticsearch/reference/current/index-modules-slowlog.html) para solo el nivel de advertencia:

PUT /my_index/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Advertencia: Elasticsearch 1.x y 2.x no validaron muy estrictamente los nombres de configuración de nivel de índice. Si tuviera un error tipográfico, o simplemente inventara una configuración, entonces la aceptaría ciegamente, pero de lo contrario la ignoraría. Elasticsearch 5.x valida estrictamente los nombres de las configuraciones y rechazará cualquier intento de aplicar configuraciones de índice con configuraciones desconocidas (debido a un error tipográfico o a la falta de un complemento). Ambas declaraciones se aplican a la configuración de índice que cambia dinámicamente y en el momento de la creación.
