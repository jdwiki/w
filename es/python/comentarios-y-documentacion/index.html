Sintaxis #

  • # Este es un comentario de una sola línea
  • print("")  # Este es un comentario en línea
  • “”"
    Esto es
    un comentario de varias líneas
    """

Los desarrolladores deben seguir las pautas de PEP257 - Docstring Conventions. En algunos casos, las guías de estilo (como las Guía de estilo de Google) o la documentación de terceros (como Sphinx ) puede detallar convenciones adicionales para docstrings.

Acceso programático a docstrings #

Las cadenas de documentación, a diferencia de los comentarios normales, se almacenan como un atributo de la función que documentan, lo que significa que puede acceder a ellas mediante programación.

Una función de ejemplo #

def func():
    """This is a function that does nothing at all"""
    return

Se puede acceder a la cadena de documentos usando el atributo __doc__:

print(func.__doc__)

Esta es una función que no hace nada

help(func)

Ayuda sobre la función func en el módulo __main__:

función()

     Esta es una función que no hace nada

Otra función de ejemplo #

function.__doc__ es solo la cadena de documentación real como una cadena, mientras que la función ayuda proporciona información general sobre una función, incluida la cadena de documentación. Aquí hay un ejemplo más útil:

def greet(name, greeting="Hello"):
    """Print a greeting to the user `name`

    Optional parameter `greeting` can change what they're greeted with."""
    
    print("{} {}".format(greeting, name))

help(greet)

Ayuda sobre la función saludar en el módulo __main__:

saludo(nombre, saludo='Hola')

    Imprimir un saludo al usuario nombre
    El parámetro opcional saludo puede cambiar con qué se les saluda.

Ventajas de las cadenas de documentación sobre los comentarios regulares #

Simplemente no poner una cadena de documentación o un comentario regular en una función hace que sea mucho menos útil.

def greet(name, greeting="Hello"):
    # Print a greeting to the user `name`

Comentarios de una sola línea, en línea y de varias líneas #

Los comentarios se utilizan para explicar el código cuando el código básico en sí no está claro.

Python ignora los comentarios, por lo que no ejecutará el código allí ni generará errores de sintaxis para oraciones en inglés simple.

Los comentarios de una sola línea comienzan con el carácter hash (#) y terminan con el final de la línea.

  • Comentario de una sola línea:

    This is a single line comment in Python #

  • Comentario en línea:

    print(“Hello World”) # This line prints “Hello World”

  • Los comentarios que abarcan varias líneas tienen """ o ''' en cada extremo. Esto es lo mismo que una cadena de varias líneas, pero se pueden usar como comentarios:

    """ This type of comment spans multiple lines. These are mostly used for documentation of functions, classes and modules. """

Escribir documentación usando docstrings #

Una cadena de documentación es un [comentario de varias líneas][2] que se utiliza para documentar módulos, clases, funciones y métodos. Tiene que ser la primera declaración del componente que describe.

def hello(name):
    """Greet someone.

    Print a greeting ("Hello") for the person with the given name.
    """

    print("Hello "+name)

class Greeter:
    """An object used to greet people.

    It contains multiple greeting functions for several languages
    and times of the  day.
    """

El valor de la cadena de documentos se puede [acceder dentro del programa][3] y, por ejemplo, se utiliza con el comando ayuda.

Convenciones de sintaxis #

PEP 257 #

PEP 257 define un estándar de sintaxis para comentarios de cadenas de documentación. Básicamente permite dos tipos:

  • Docstrings de una línea:

Según PEP 257, deben ser utilizados con funciones cortas y sencillas. Todo se coloca en una línea, por ejemplo:

def hello():
    """Say hello to your friends."""
    print("Hello my friends!")

La cadena de documentación debe terminar con un punto, el verbo debe estar en forma imperativa.

  • Cadenas de documentos de varias líneas:

La cadena de documentos de varias líneas debe usarse para funciones, módulos o clases más largos y complejos.

def hello(name, language="en"):
    """Say hello to a person.

    Arguments:
    name: the name of the person
    language: the language in which the person should be greeted
    """

    print(greeting[language]+" "+name)

Comienzan con un breve resumen (equivalente al contenido de una cadena de documentos de una línea) que puede estar en la misma línea que las comillas o en la línea siguiente, proporciona detalles adicionales y enumera parámetros y valores de retorno.

Nota PEP 257 define [qué información se debe dar][4] dentro de una cadena de documentos, no define en qué formato se debe dar. Esta fue la razón por la que otras partes y herramientas de análisis de documentación especificaron sus propios estándares para la documentación, algunos de los cuales se enumeran a continuación y en [esta pregunta][9].

Esfinge #

[Sphinx][5] es una herramienta para generar documentación basada en HTML para proyectos Python basados ​​en docstrings. Su lenguaje de marcado utilizado es [reStructuredText][6]. Definen sus propios estándares para la documentación, pythonhosted.org alberga una [muy buena descripción de ellos][7]. El formato Sphinx es utilizado, por ejemplo, por [pyCharm IDE][11].

Una función se documentaría así usando el formato Sphinx/reStructuredText:

def hello(name, language="en"):
    """Say hello to a person.

    :param name: the name of the person
    :type name: str
    :param language: the language in which the person should be greeted
    :type language: str
    :return: a number
    :rtype: int
    """

    print(greeting[language]+" "+name)
    return 4

Guía de estilo de Google Python #

Google ha publicado la [Guía de estilo de Google Python][8] que define las convenciones de codificación para Python, incluidos los comentarios de la documentación. En comparación con Sphinx/reST, muchas personas dicen que la documentación de acuerdo con las pautas de Google es mejor legible por humanos.

La [página pythonhosted.org mencionada anteriormente][7] también proporciona algunos ejemplos de buena documentación de acuerdo con la Guía de estilo de Google.

Con el complemento [Napoleón][10], Sphinx también puede analizar la documentación en el formato compatible con la Guía de estilo de Google.

Una función se documentaría así utilizando el formato de la Guía de estilo de Google:

def hello(name, language="en"):
    """Say hello to a person.

    Args:
        name: the name of the person as string
        language: the language code string

    Returns:
        A number.
    """

    print(greeting[language]+" "+name)
    return 4
