En esencia, el recolector de elementos no utilizados de Python (a partir de la versión 3.5) es una implementación simple de conteo de referencias. Cada vez que hace una referencia a un objeto (por ejemplo, a = miobjeto), se incrementa el recuento de referencias en ese objeto (miobjeto). Cada vez que se elimina una referencia, el recuento de referencias se reduce, y una vez que el recuento de referencias llega a ‘0’, sabemos que nada contiene una referencia a ese objeto y podemos desasignarlo.

Un malentendido común sobre cómo funciona la gestión de memoria de Python es que la palabra clave del libera memoria de objetos. Esto no es verdad. Lo que realmente sucede es que la palabra clave del simplemente disminuye el refcount de los objetos, lo que significa que si lo llama suficientes veces para que el refcount llegue a cero, el objeto puede ser recolectado como basura (incluso si en realidad todavía hay referencias al objeto disponibles en otros lugares en tu codigo).

Python crea o limpia agresivamente objetos la primera vez que los necesita. Si realizo la asignación a = object(), la memoria para el objeto se asigna en ese momento (cpython a veces reutilizará ciertos tipos de objetos, por ejemplo, listas debajo del capó, pero sobre todo no mantiene un grupo de objetos libres y realizará la asignación cuando lo necesite). De manera similar, tan pronto como el refcount se reduce a 0, GC lo limpia.

Recolección de basura generacional #

En la década de 1960, John McCarthy descubrió una falla fatal en la recolección de basura de refcounting cuando implementó el algoritmo de refcounting usado por Lisp: ¿Qué sucede si dos objetos se refieren entre sí en una referencia cíclica? ¿Cómo puede recolectar basura esos dos objetos incluso si no hay referencias externas a ellos si siempre se referirán entre sí? Este problema también se extiende a cualquier estructura de datos cíclicos, como un búfer de anillo o dos entradas consecutivas en una lista doblemente enlazada. Python intenta solucionar este problema usando un giro ligeramente interesante en otro algoritmo de recolección de basura llamado Recolección de basura generacional.

En esencia, cada vez que crea un objeto en Python, lo agrega al final de una lista doblemente enlazada. En ocasiones, Python recorre esta lista, verifica a qué objetos se refieren los objetos en la lista también, y si también están en la lista (veremos por qué podrían no estar en un momento), disminuye aún más sus refcounts. En este punto (en realidad, hay algunas heurísticas que determinan cuándo se mueven las cosas, pero supongamos que es después de una sola colección para simplificar las cosas) cualquier cosa que todavía tenga un refcount mayor que 0 se promociona a otra lista vinculada llamada “Generación 1” (es por eso que todos los objetos no siempre están en la lista de generación 0) que tiene este ciclo aplicado con menos frecuencia. Aquí es donde entra en juego la recolección de basura generacional. Hay 3 generaciones por defecto en Python (tres listas de objetos enlazadas): La primera lista (generación 0) contiene todos los objetos nuevos; si ocurre un ciclo de GC y los objetos no se recopilan, se mueven a la segunda lista (generación 1), y si ocurre un ciclo de GC en la segunda lista y todavía no se recopilan, se mueven a la tercera lista (generación 2 ). La lista de tercera generación (llamada “generación 2”, ya que estamos indexando cero) es basura recolectada con mucha menos frecuencia que las dos primeras, la idea es que si su objeto es de larga duración, no es tan probable que sea GCed, y es posible que nunca GCed durante la vida útil de su aplicación, por lo que no tiene sentido perder el tiempo comprobándolo en cada ejecución de GC. Además, se observa que la mayoría de los objetos se recogen como basura con relativa rapidez. De ahora en adelante, llamaremos a estos “buenos objetos” ya que mueren jóvenes. Esto se llama la “hipótesis generacional débil” y también se observó por primera vez en los años 60.

Un comentario aparte: a diferencia de las dos primeras generaciones, la lista de larga duración de la tercera generación no se recolecta en un horario regular. Se verifica cuando la proporción de objetos pendientes de larga duración (aquellos que están en la lista de tercera generación, pero que aún no han tenido un ciclo de GC) con respecto al total de objetos de larga duración en la lista es superior al 25 %. Esto se debe a que la tercera lista no tiene límites (las cosas nunca se mueven de ella a otra lista, por lo que solo desaparecen cuando en realidad se recolectan como basura), lo que significa que para las aplicaciones en las que está creando muchos objetos de larga duración, los ciclos de GC en la tercera lista puede ser bastante larga. Al usar una relación logramos un “desempeño lineal amortizado en el número total de objetos”; es decir, cuanto más larga es la lista, más tarda la GC, pero realizamos la GC con menos frecuencia (aquí está la [propuesta original de 2008](https://mail.python.org/pipermail/python-dev/2008-June/080579. html) para esta heurística de Martin von Löwis para lecturas adicionales). El acto de realizar una recolección de basura en la lista de tercera generación o “madura” se denomina “recolección de basura completa”.

Entonces, la recolección de basura generacional acelera enormemente las cosas al no requerir que analicemos objetos que probablemente no necesiten GC todo el tiempo, pero ¿cómo nos ayuda a romper las referencias cíclicas? Probablemente no muy bien, resulta. La función para romper realmente estos ciclos de referencia comienza [así] (https://github.com/python/cpython/blob/8f33d77/Modules/gcmodule.c#L847):

/* Break reference cycles by clearing the containers involved.  This is
 * tricky business as the lists can be changing and we don't know which
 * objects may be freed.  It is possible I screwed something up here.
 */
static void
delete_garbage(PyGC_Head *collectable, PyGC_Head *old)

La razón por la que la recolección de basura generacional ayuda con esto es que podemos mantener la longitud de la lista como un conteo separado; cada vez que agregamos un nuevo objeto a la generación, incrementamos este conteo, y cada vez que movemos un objeto a otra generación o lo desasignamos, disminuimos el conteo. Teóricamente, al final de un ciclo de GC, este recuento (para las dos primeras generaciones de todos modos) siempre debería ser 0. Si no es así, todo lo que quede en la lista es una forma de referencia circular y podemos eliminarlo. Sin embargo, hay un problema más aquí: ¿Qué pasa si los objetos sobrantes tienen el método mágico __del__ de Python? __del__ se llama cada vez que se destruye un objeto de Python. Sin embargo, si dos objetos en una referencia circular tienen métodos __del__, no podemos estar seguros de que destruir uno no romperá el método __del__ de los otros. Para un ejemplo artificial, imagina que escribimos lo siguiente:

class A(object):
    def __init__(self, b=None):
        self.b = b
 
    def __del__(self):
        print("We're deleting an instance of A containing:", self.b)
     
class B(object):
    def __init__(self, a=None):
        self.a = a
 
    def __del__(self):
        print("We're deleting an instance of B containing:", self.a)

y configuramos una instancia de A y una instancia de B para que se apunten entre sí y luego terminan en el mismo ciclo de recolección de basura? Digamos que elegimos uno al azar y asignamos nuestra instancia de A primero; Se llamará al método __del__ de A, se imprimirá y luego A se liberará. Luego llegamos a B, llamamos a su método __del__ y ¡ups! ¡Falla de segmento! A ya no existe. Podríamos solucionar esto llamando primero a todo lo que sobra de los métodos __del__ y luego haciendo otra pasada para realmente distribuir todo, sin embargo, esto presenta otro problema: ¿Qué sucede si el método __del__ de un objeto guarda una referencia del otro objeto que se trata de para ser GCed y tiene una referencia a nosotros en otro lugar? Todavía tenemos un ciclo de referencia, pero ahora no es posible hacer un GC de ninguno de los objetos, incluso si ya no están en uso. Tenga en cuenta que incluso si un objeto no es parte de una estructura de datos circular, podría revivir a sí mismo en su propio método __del__; Python tiene una verificación para esto y detendrá el GC si el recuento de referencia de un objeto ha aumentado después de que se haya llamado a su método __del__.

CPython se ocupa de esto colocando esos objetos no aptos para GC (cualquier cosa con algún tipo de referencia circular y un método __del__) en una lista global de basura incobrable y luego dejándolos allí por toda la eternidad:

/* list of uncollectable objects */
static PyObject *garbage = NULL;

Reutilización de objetos primitivos #

Una cosa interesante a tener en cuenta que puede ayudar a optimizar sus aplicaciones es que las primitivas en realidad también se cuentan bajo el capó. Echemos un vistazo a los números; para todos los enteros entre -5 y 256, Python siempre reutiliza el mismo objeto:

>>> import sys
>>> sys.getrefcount(1)
797
>>> a = 1
>>> b = 1
>>> sys.getrefcount(1)
799

Tenga en cuenta que refcount aumenta, lo que significa que a y b hacen referencia al mismo objeto subyacente cuando se refieren a la primitiva 1. Sin embargo, para números más grandes, Python en realidad no reutiliza el objeto subyacente:

>>> a = 999999999
>>> sys.getrefcount(999999999)
3
>>> b = 999999999
>>> sys.getrefcount(999999999)
3

Debido a que el refcount para 999999999 no cambia cuando se asigna a a y b, podemos inferir que se refieren a dos objetos subyacentes diferentes, aunque a ambos se les asigna la misma primitiva.

Efectos del comando del #

Quitar un nombre de variable del alcance usando del v, o quitar un objeto de una colección usando del v[item] o del[i:j], o quitando un atributo usando del v.name, o cualquier otra forma de eliminar las referencias a un objeto, no desencadena ninguna llamada al destructor ni la liberación de memoria en sí misma. Los objetos solo se destruyen cuando su recuento de referencia llega a cero.

>>> import gc
>>> gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def bar():
    return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> del t          # not destructed yet - another_t still refers to it
>>> del another_t  # final reference gone, object is destructed
Destructed

Recuento de referencias #

La gran mayoría de la administración de la memoria de Python se maneja con el conteo de referencias.

Cada vez que se hace referencia a un objeto (por ejemplo, se asigna a una variable), su recuento de referencias aumenta automáticamente. Cuando se elimina la referencia (por ejemplo, la variable queda fuera del alcance), su recuento de referencias se reduce automáticamente.

Cuando el recuento de referencias llega a cero, el objeto se destruye de inmediato y la memoria se libera de inmediato. Por lo tanto, en la mayoría de los casos, ni siquiera se necesita el recolector de basura.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def foo():
        Track()
        # destructed immediately since no longer has any references
        print("---")
        t = Track()
        # variable is referenced, so it's not destructed yet
        print("---")
        # variable is destructed when function exits
>>> foo()
Initialized
Destructed
---
Initialized
---
Destructed

Para demostrar aún más el concepto de referencias:

>>> def bar():
        return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> t = None          # not destructed yet - another_t still refers to it
>>> another_t = None  # final reference gone, object is destructed
Destructed

Recolector de basura para ciclos de referencia #

La única vez que se necesita el recolector de basura es si tiene un ciclo de referencia. El ejemplo simple de un ciclo de referencia es uno en el que A se refiere a B y B se refiere a A, mientras que nada más se refiere a A o B. Ni A ni B son accesibles desde ninguna parte del programa, por lo que se pueden destruir de forma segura. sin embargo, sus recuentos de referencia son 1 y, por lo tanto, no pueden liberarse solo con el algoritmo de recuento de referencia.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> A = Track()
Initialized
>>> B = Track()
Initialized
>>> A.other = B
>>> B.other = A
>>> del A; del B  # objects are not destructed due to reference cycle
>>> gc.collect()  # trigger collection
Destructed
Destructed
4

Un ciclo de referencia puede ser arbitrariamente largo. Si A apunta a B apunta a C apunta a… apunta a Z que apunta a A, entonces no se recolectarán ni de A a Z, hasta la fase de recolección de basura:

>>> objs = [Track() for _ in range(10)]
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
>>> for i in range(len(objs)-1):
...     objs[i].other = objs[i + 1]
...
>>> objs[-1].other = objs[0]  # complete the cycle
>>> del objs                  # no one can refer to objs now - still not destructed
>>> gc.collect()
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
20

Ver el refcount de un objeto #

>>> import sys
>>> a = object()
>>> sys.getrefcount(a)
2
>>> b = a
>>> sys.getrefcount(a)
3
>>> del b
>>> sys.getrefcount(a)
2

Desasignar objetos a la fuerza #

Puede forzar la desasignación de objetos incluso si su refcount no es 0 tanto en Python 2 como en 3.

Ambas versiones usan el módulo ctypes para hacerlo.

ADVERTENCIA: hacer esto dejará su entorno de Python inestable y propenso a bloquearse sin rastro. El uso de este método también podría presentar problemas de seguridad (bastante poco probables) Solo desasigne objetos que esté seguro de que nunca volverá a hacer referencia. Alguna vez.

import ctypes
deallocated = 12345
ctypes.pythonapi._Py_Dealloc(ctypes.py_object(deallocated))

import ctypes, sys
deallocated = 12345
(ctypes.c_char * sys.getsizeof(deallocated)).from_address(id(deallocated))[:4] = '\x00' * 4

Después de la ejecución, cualquier referencia al objeto ahora desasignado hará que Python produzca un comportamiento indefinido o se bloquee, sin un rastreo. Probablemente hubo una razón por la cual el recolector de basura no eliminó ese objeto…

Si desasigna Ninguno, recibe un mensaje especial - Error fatal de Python: desasignando Ninguno antes de colapsar.

Administrar la recolección de basura #

Hay dos enfoques para influir cuando se realiza una limpieza de memoria. Influyen en la frecuencia con la que se realiza el proceso automático y el otro activa manualmente una limpieza.

El recolector de basura se puede manipular ajustando los umbrales de recolección que afectan la frecuencia con la que se ejecuta el recolector. Python utiliza un sistema de gestión de memoria basado en generación. Los nuevos objetos se guardan en la última generación - generación0 y con cada colección superviviente, los objetos se promocionan a generaciones anteriores. Después de llegar a la última generación, generación2, ya no se promocionan.

Los umbrales se pueden cambiar usando el siguiente fragmento:

import gc
gc.set_threshold(1000, 100, 10) # Values are just for demonstration purpose

El primer argumento representa el umbral para recopilar generación0. Cada vez que el número de asignaciones exceda el número de desasignaciones por 1000, se llamará al recolector de basura.

Las generaciones anteriores no se limpian en cada ejecución para optimizar el proceso. Los argumentos segundo y tercero son opcionales y controlan la frecuencia con la que se limpian las generaciones anteriores. Si generación0 se procesó 100 veces sin limpiar generación1, entonces se procesará generación1. Del mismo modo, los objetos de la generación2 se procesarán solo cuando los de la generación1 se limpiaron 10 veces sin tocar la generación2.

Una instancia en la que la configuración manual de los umbrales es beneficiosa es cuando el programa asigna una gran cantidad de objetos pequeños sin desasignarlos, lo que hace que el recolector de elementos no utilizados se ejecute con demasiada frecuencia (cada asignación de objetos generation0_threshold). Aunque el colector es bastante rápido, cuando se ejecuta en una gran cantidad de objetos, presenta un problema de rendimiento. De todos modos, no existe una estrategia única para elegir los umbrales y su caso de uso es confiable.

La activación manual de una colección se puede hacer como en el siguiente fragmento:

import gc
gc.collect()

La recolección de elementos no utilizados se activa automáticamente según la cantidad de asignaciones y desasignaciones, no según la memoria consumida o disponible. En consecuencia, cuando se trabaja con objetos grandes, la memoria puede agotarse antes de que se active la limpieza automática. Esto es un buen caso de uso para llamar manualmente al recolector de basura.

Aunque es posible, no es una práctica recomendada. Evitar pérdidas de memoria es la mejor opción. De todos modos, en proyectos grandes, la detección de la fuga de memoria puede ser una tarea complicada y la activación manual de una recolección de elementos no utilizados puede usarse como una solución rápida hasta que se lleve a cabo una depuración adicional.

Para los programas de ejecución prolongada, la recolección de elementos no utilizados se puede activar por tiempo o por evento. Un ejemplo para el primero es un servidor web que activa una colección después de un número fijo de solicitudes. Para el último, un servidor web que desencadena una recolección de basura cuando se recibe un determinado tipo de solicitud.

No espere a que se limpie la recolección de basura #

El hecho de que la recolección de basura se limpie no significa que deba esperar a que se limpie el ciclo de recolección de basura.

En particular, no debe esperar a que la recolección de basura cierre los identificadores de archivos, las conexiones de la base de datos y abra las conexiones de red.

por ejemplo:

En el siguiente código, asume que el archivo se cerrará en el siguiente ciclo de recolección de elementos no utilizados, si f fue la última referencia al archivo.

>>> f = open("test.txt")
>>> del f

Una forma más explícita de limpiar es llamar a f.close(). Puede hacerlo aún más elegante, es decir, mediante el uso de la instrucción with, también conocida como administrador de contexto :

>>> with open("test.txt") as f:
...     pass
...     # do something with f
>>> #now the f object still exists, but it is closed

La declaración with le permite sangrar su código debajo del archivo abierto. Esto hace que sea explícito y más fácil ver cuánto tiempo se mantiene abierto un archivo. También siempre cierra un archivo, incluso si se genera una excepción en el bloque while.
