Evitar el bloqueo de intérprete global (GIL)

  • Grupo de multiprocesamiento

En esta página

  • Grupo de multiprocesamiento

¿Por qué existe un GIL?

El GIL ha existido en CPython desde el inicio de los subprocesos de Python, en 1992. Está diseñado para garantizar la seguridad de los subprocesos al ejecutar el código de Python. Los intérpretes de Python escritos con un GIL evitan que varios subprocesos nativos ejecuten códigos de bytes de Python a la vez. Esto facilita que los complementos se aseguren de que su código sea seguro para subprocesos: simplemente bloquee la GIL y solo su subproceso activo podrá ejecutarse, por lo que su código será automáticamente seguro para subprocesos.

Versión corta: la GIL garantiza que no importa cuántos procesadores y subprocesos tenga, solo se ejecutará un subproceso de un intérprete de python a la vez.

Esto tiene muchos beneficios de facilidad de uso, pero también tiene muchos beneficios negativos.

Tenga en cuenta que un GIL no es un requisito del lenguaje Python. En consecuencia, no puede acceder a GIL directamente desde el código Python estándar. No todas las implementaciones de Python usan un GIL.

Intérpretes que tienen un GIL: CPython, PyPy, Cython (pero puedes deshabilitar el GIL con nogil)

Intérpretes que no tienen GIL: Jython, IronPython

Detalles sobre cómo opera GIL:

Cuando se ejecuta un subproceso, bloquea la GIL. Cuando un subproceso quiere ejecutarse, solicita el GIL y espera hasta que esté disponible. En CPython, antes de la versión 3.2, el subproceso en ejecución verificaba después de una cierta cantidad de instrucciones de Python para ver si otro código quería el bloqueo (es decir, liberaba el bloqueo y luego lo solicitaba nuevamente). Este método tendía a provocar la inanición de subprocesos, en gran parte porque el subproceso que liberaba el bloqueo lo adquiría de nuevo antes de que los subprocesos en espera tuvieran la oportunidad de despertar. Desde 3.2, los subprocesos que quieren el GIL esperan el bloqueo durante un tiempo y, después de ese tiempo, establecen una variable compartida que obliga al subproceso en ejecución a ceder. Sin embargo, esto aún puede resultar en tiempos de ejecución drásticamente más largos. Consulte los enlaces a continuación de dabeaz.com (en la sección de referencias) para obtener más detalles.

CPython libera automáticamente el GIL cuando un subproceso realiza una operación de E/S. Las bibliotecas de procesamiento de imágenes y las operaciones de procesamiento de números numpy liberan el GIL antes de realizar su procesamiento.

Beneficios del GIL

Para los intérpretes que usan el GIL, el GIL es sistémico. Se utiliza para preservar el estado de la aplicación. Beneficios incluidos:
  • Recolección de basura: los recuentos de referencia seguros para subprocesos deben modificarse mientras el GIL está bloqueado. En CPython, toda la recolección de basura está ligada al GIL. Este es uno grande; consulte el artículo wiki de python.org sobre GIL (enumerado en Referencias, a continuación) para obtener detalles sobre lo que aún debe ser funcional si se desea eliminar GIL.
  • Facilidad para los programadores que trabajan con GIL: bloquear todo es simple, pero fácil de codificar para
  • Facilita la importación de módulos de otros idiomas

Consecuencias de las LIG

El GIL solo permite que un subproceso ejecute el código de Python a la vez dentro del intérprete de Python. Esto significa que los subprocesos múltiples de procesos que ejecutan código Python estricto simplemente no funcionan. Al usar subprocesos contra GIL, es probable que tenga un peor rendimiento con los subprocesos que si ejecutara un único subproceso.

Grupo de multiprocesamiento #

La respuesta simple, cuando se pregunta cómo usar subprocesos en Python es: “No lo haga. Use procesos, en su lugar”. El módulo de multiprocesamiento le permite crear procesos con una sintaxis similar a la creación de subprocesos, pero prefiero usar su conveniente objeto Pool.

Usando el código que David Beazley usó por primera vez para mostrar los peligros de los subprocesos contra GIL, lo reescribiremos usando multiprocessing.Pool:

Código de David Beazley que mostró problemas de subprocesos GIL

from threading import Thread
import time
def countdown(n):
    while n > 0:
        n -= 1

COUNT = 10000000

t1 = Thread(target=countdown,args=(COUNT/2,))
t2 = Thread(target=countdown,args=(COUNT/2,))
start = time.time()
t1.start();t2.start()
t1.join();t2.join()
end = time.time()
print end-start

Reescrito usando multiprocessing.Pool: 
import multiprocessing
import time
def countdown(n):
    while n > 0:
        n -= 1

COUNT = 10000000

start = time.time()
with multiprocessing.Pool as pool:
    pool.map(countdown, [COUNT/2, COUNT/2])

    pool.close()
    pool.join()

end = time.time()
print(end-start)

En lugar de crear subprocesos, esto crea nuevos procesos. Dado que cada proceso es su propio intérprete, no hay colisiones GIL. multiprocessing.Pool abrirá tantos procesos como núcleos haya en la máquina, aunque en el ejemplo anterior, solo necesitaría dos. En un escenario del mundo real, desea diseñar su lista para que tenga al menos la misma longitud que procesadores en su máquina. El Pool ejecutará la función que usted le diga que ejecute con cada argumento, hasta el número de procesos que cree. Cuando finaliza la función, cualquier función restante en la lista se ejecutará en ese proceso.

Descubrí que, incluso usando la declaración with, si no cierra y se une al grupo, los procesos continúan existiendo. Para limpiar los recursos, siempre cierro y me uno a mis grupos.

Reescrito usando nogil (SOLO FUNCIONA EN CYTHON):

from threading import Thread
import time
def countdown(n):
    while n > 0:
        n -= 1

COUNT = 10000000

with nogil:
    t1 = Thread(target=countdown,args=(COUNT/2,))
    t2 = Thread(target=countdown,args=(COUNT/2,))
    start = time.time()
    t1.start();t2.start()
    t1.join();t2.join()
    
end = time.time()
print end-start

Es así de simple, siempre y cuando estés usando cython. Tenga en cuenta que la documentación dice que debe asegurarse de no cambiar ningún objeto de Python:

El código en el cuerpo de la declaración no debe manipular objetos de Python de ninguna manera, y no debe llamar a nada que manipule objetos de Python sin antes volver a adquirir el GIL. Cython actualmente no comprueba esto.
