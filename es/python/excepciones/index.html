Los errores detectados durante la ejecución se denominan excepciones y no son incondicionalmente fatales. La mayoría de las excepciones no son manejadas por programas; es posible escribir programas que manejen excepciones seleccionadas. Hay funciones específicas en Python para tratar las excepciones y la lógica de excepciones. Además, las excepciones tienen una rica jerarquía de tipos, todas heredadas del tipo BaseException.

Sintaxis #

  • subir excepción
  • aumentar # volver a generar una excepción que ya se ha generado
  • generar excepción de causa # Python 3 - establecer causa de excepción
  • aumentar * excepción * de Ninguno # Python 3 - suprimir todo el contexto de excepción
  • probar:
  • excepto [tipos de excepción] [ como identificador ]:
  • más:
  • finalmente:

¡No atrapes todo! #

Si bien a menudo es tentador detectar todas las Excepciones:

try:
    very_difficult_function()
except Exception:
    # log / try to reconnect / exit gratiously
finally:
    print "The END"
    # it runs no matter what execute.

O incluso todo (que incluye BaseException y todos sus hijos, incluida Exception):

try:
    even_more_difficult_function()
except:
    pass  # do whatever needed

En la mayoría de los casos es una mala práctica. Puede detectar más de lo previsto, como SystemExit, KeyboardInterrupt y MemoryError, cada uno de los cuales generalmente debe manejarse de manera diferente a los errores lógicos o del sistema habituales. También significa que no hay una comprensión clara de lo que el código interno puede hacer mal y cómo recuperarse adecuadamente de esa condición. Si está detectando todos los errores, no sabrá qué error ocurrió o cómo solucionarlo.

Esto se conoce más comúnmente como “enmascaramiento de errores” y debe evitarse. Deje que su programa se bloquee en lugar de fallar silenciosamente o, lo que es peor, fallar en un nivel más profundo de ejecución. (Imagina que es un sistema transaccional)

Por lo general, estas construcciones se utilizan en el nivel más externo del programa y registrarán los detalles del error para que se pueda corregir el error o se pueda manejar el error de manera más específica.

Resubir excepciones #

A veces desea capturar una excepción solo para inspeccionarla, p. con fines de registro. Después de la inspección, desea que la excepción continúe propagándose como lo hacía antes.

En este caso, simplemente use la instrucción raise sin parámetros.

try:
    5 / 0
except ZeroDivisionError:
    print("Got an error")
    raise

Sin embargo, tenga en cuenta que alguien más arriba en la pila de llamadas aún puede detectar la excepción y manejarla de alguna manera. La salida hecha podría ser una molestia en este caso porque sucederá de todos modos (capturado o no capturado). Por lo tanto, podría ser una mejor idea generar una excepción diferente, que contenga su comentario sobre la situación, así como la excepción original:

try:
    5 / 0
except ZeroDivisionError as e:
    raise ZeroDivisionError("Got an error", e)

Pero esto tiene el inconveniente de reducir el seguimiento de la excepción exactamente a este “aumento”, mientras que el “aumento” sin argumento conserva el seguimiento de la excepción original.

En Python 3 puedes mantener la pila original usando la sintaxis raise-from:

    raise ZeroDivisionError("Got an error") from e

Captura de múltiples excepciones #

Hay algunas formas de atrapar múltiples excepciones.

La primera es creando una tupla de los tipos de excepción que desea capturar y manejar de la misma manera. Este ejemplo hará que el código ignore las excepciones KeyError y AttributeError.

try:
    d = {}
    a = d[1]
    b = d.non_existing_field
except (KeyError, AttributeError) as e:
    print("A KeyError or an AttributeError exception has been caught.")

Si desea manejar diferentes excepciones de diferentes maneras, puede proporcionar un bloque de excepciones separado para cada tipo. En este ejemplo, aún capturamos KeyError y AttributeError, pero manejamos las excepciones de diferentes maneras.

try:
    d = {}
    a = d[1]
    b = d.non_existing_field
except KeyError as e:
    print("A KeyError has occurred. Exception message:", e)
except AttributeError as e:
    print("An AttributeError has occurred. Exception message:", e)

Captura de excepciones #

Use try...except: para detectar excepciones. Debe especificar una excepción lo más precisa posible:

try:
    x = 5 / 0
except ZeroDivisionError as e:
    # `e` is the exception object
    print("Got a divide by zero! The exception was:", e)
    # handle exceptional case
    x = 0  
finally:
    print "The END"
    # it runs no matter what execute.

La clase de excepción que se especifica, en este caso, ZeroDivisionError, captura cualquier excepción que sea de esa clase o de cualquier subclase de esa excepción.

Por ejemplo, ZeroDivisionError es una subclase de ArithmeticError:

>>> ZeroDivisionError.__bases__
(<class 'ArithmeticError'>,)

Y así, lo siguiente seguirá capturando el ZeroDivisionError:

try:
    5 / 0
except ArithmeticError:
    print("Got arithmetic error")

Más #

El código en un bloque else solo se ejecutará si el código en el bloque try no generó excepciones. Esto es útil si tiene algún código que no desea ejecutar si se lanza una excepción, pero no quiere que se detecten las excepciones lanzadas por ese código.

Por ejemplo:

try:
    data = {1: 'one', 2: 'two'}
    print(data[1])
except KeyError as e:
    print('key not found')
else:
    raise ValueError()
# Output: one
# Output: ValueError

Tenga en cuenta que este tipo de else: no se puede combinar con un if que inicia la cláusula else con un elif. Si tiene un ‘if’ siguiente, debe permanecer sangrado debajo de ’else:’:

try:
    ...
except ...:
    ...
else:
    if ...:
        ...
    elif ...:
        ...
    else:
        ...

Generación de excepciones #

Si su código encuentra una condición que no sabe cómo manejar, como un parámetro incorrecto, debería generar la excepción adecuada.

def even_the_odds(odds):
    if odds % 2 != 1:
        raise ValueError("Did not get an odd number")
    
    return odds + 1

Creando tipos de excepción personalizados #

Cree una clase que herede de Exception:

class FooException(Exception):
    pass
try:
    raise FooException("insert description here")
except FooException:
    print("A FooException was raised.")

u otro tipo de excepción:

class NegativeError(ValueError):
      pass

def foo(x):
    # function that only accepts positive values of x
    if x < 0:
        raise NegativeError("Cannot process negative numbers")
    ...  # rest of function body
try:
    result = foo(int(input("Enter a positive integer: ")))  # raw_input in Python 2.x
except NegativeError:
    print("You entered a negative number!")
else:
    print("The result was " + str(result))

Las excepciones también son objetos #

Las excepciones son solo objetos regulares de Python que heredan de la BaseException incorporada. Una secuencia de comandos de Python puede usar la instrucción raise para interrumpir la ejecución, lo que hace que Python imprima un seguimiento de la pila de llamadas en ese punto y una representación de la instancia de excepción. Por ejemplo:

>>> def failing_function():
...     raise ValueError('Example error!')
>>> failing_function()
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "<stdin>", line 2, in failing_function
ValueError: Example error!

que dice que ValueError con el mensaje '¡Error de ejemplo!' fue generado por nuestra failing_function(), que se ejecutó en el intérprete.

El código de llamada puede optar por manejar todos y cada uno de los tipos de excepción que puede generar una llamada:

>>> try:
...     failing_function()
... except ValueError:
...     print('Handled the error')
Handled the error

Puede hacerse con los objetos de excepción asignándolos en la parte excepto... ​​del código de manejo de excepciones:

>>> try:
...     failing_function()
... except ValueError as e:
...     print('Caught exception', repr(e))
Caught exception ValueError('Example error!',)

Puede encontrar una lista completa de las excepciones integradas de Python junto con sus descripciones en la documentación de Python: https://docs.python.org/3.5/library/exceptions.html. Y aquí está la lista completa ordenada jerárquicamente: https://www.wikiod.com/es/python/excepciones#Jerarquía de excepciones

Ejemplos prácticos de manejo de excepciones #

Entrada del usuario #

Imagina que quieres que un usuario ingrese un número a través de input. Desea asegurarse de que la entrada sea un número. Puedes usar try/except para esto:

while True:
    try:
        nb = int(input('Enter a number: '))
        break
    except ValueError:
        print('This is not a number, try again.')

Nota: Python 2.x usaría raw_input en su lugar; la función input existe en Python 2.x pero tiene una semántica diferente. En el ejemplo anterior, input también aceptaría expresiones como 2 + 2 que se evalúan como un número.

Si la entrada no se puede convertir a un número entero, se genera un ValueError. Puedes atraparlo con excepto. Si no se genera ninguna excepción, break sale del bucle. Después del bucle, nb contiene un número entero.

Diccionarios #

Imagina que estás iterando sobre una lista de enteros consecutivos, como range(n), y tienes una lista de diccionarios d que contiene información sobre cosas que hacer cuando encuentras algunos enteros en particular, digamos salta d[i ] los siguientes.

d = [{7: 3}, {25: 9}, {38: 5}]

for i in range(len(d)):
    do_stuff(i)
    try:
       dic = d[i]
       i += dic[i]
    except KeyError:
       i += 1

Se generará un KeyError cuando intente obtener un valor de un diccionario para una clave que no existe.

Ejecutando código de limpieza con finalmente #

A veces, es posible que desee que ocurra algo independientemente de la excepción que haya ocurrido, por ejemplo, si tiene que limpiar algunos recursos.

El bloque finally de una cláusula try ocurrirá independientemente de si se generaron excepciones.

resource = allocate_some_expensive_resource()
try:
    do_stuff(resource)
except SomeException as e:
    log_error(e)
    raise  # re-raise the error
finally:
    free_expensive_resource(resource)

Este patrón a menudo se maneja mejor con administradores de contexto (usando [la instrucción `with] (https://www.wikiod.com/es/python/administradores-de-contexto-declaracion-con)).

Excepciones en cadena con subida desde #

En el proceso de manejar una excepción, es posible que desee generar otra excepción. Por ejemplo, si obtiene un IOError mientras lee un archivo, es posible que desee generar un error específico de la aplicación para presentarlo a los usuarios de su biblioteca.

Puede encadenar excepciones para mostrar cómo procedió el manejo de las excepciones:

>>> try:
    5 / 0
except ZeroDivisionError as e:
    raise ValueError("Division failed") from e

Traceback (most recent call last):
  File "<stdin>", line 2, in <module>
ZeroDivisionError: division by zero

The above exception was the direct cause of the following exception:

Traceback (most recent call last):
  File "<stdin>", line 4, in <module>
ValueError: Division failed
