Un entorno virtual es una herramienta para mantener las dependencias requeridas por diferentes proyectos en lugares separados, mediante la creación de entornos Python virtuales para ellos. Resuelve el dilema “El proyecto X depende de la versión 1.x pero el proyecto Y necesita 4.x” y mantiene el directorio global de paquetes del sitio limpio y manejable.

Esto ayuda a aislar sus entornos para diferentes proyectos entre sí y de las bibliotecas de su sistema.

Los entornos virtuales son lo suficientemente útiles como para que probablemente deberían usarse para todos los proyectos. En particular, los entornos virtuales le permiten:

  1. Administre las dependencias sin necesidad de acceso a la raíz
  2. Instale diferentes versiones de la misma dependencia, por ejemplo, cuando trabaje en diferentes proyectos con diferentes requisitos
  3. Trabaja con diferentes versiones de python

Creación y uso de un entorno virtual #

virtualenv es una herramienta para construir entornos Python aislados. Este programa crea una carpeta que contiene todos los ejecutables necesarios para usar los paquetes que necesitaría un proyecto de Python.

Instalar la herramienta virtualenv #

Esto solo se requiere una vez. El programa virtualenv puede estar disponible a través de su distribución. En distribuciones similares a Debian, el paquete se llama python-virtualenv o python3-virtualenv.

Alternativamente, puede instalar virtualenv usando [pip] (https://www.wikiod.com/es/python/primeros-pasos-con-el-lenguaje-python#Instalación de módulos externos usando pip):

$ pip install virtualenv

Creando un nuevo entorno virtual #

Esto solo se requiere una vez por proyecto. Al iniciar un proyecto para el que desea aislar las dependencias, puede configurar un nuevo entorno virtual para este proyecto:

$ virtualenv foo

Esto creará una carpeta foo que contiene scripts de herramientas y una copia del propio binario python. El nombre de la carpeta no es relevante. Una vez que se crea el entorno virtual, es autónomo y no requiere más manipulación con la herramienta virtualenv. Ahora puede comenzar a utilizar el entorno virtual.

Activar un entorno virtual existente #

Para activar un entorno virtual, se requiere algo de magia de shell para que su Python sea el que está dentro de foo en lugar del del sistema. Este es el propósito del archivo activar, que debe incluir en su shell actual:

$ source foo/bin/activate

Los usuarios de Windows deben escribir:

$ foo\Scripts\activate.bat

Una vez que se ha activado un entorno virtual, los binarios python y pip y todos los scripts instalados por módulos de terceros son los que están dentro de foo. En particular, todos los módulos instalados con pip se implementarán en el entorno virtual, lo que permitirá un entorno de desarrollo contenido. La activación del entorno virtual también debe agregar un prefijo a su mensaje, como se ve en los siguientes comandos.

# Installs 'requests' to foo only, not globally
(foo)$ pip install requests

Guardar y restaurar dependencias #

Para guardar los módulos que ha instalado a través de pip, puede enumerar todos esos módulos (y las versiones correspondientes) en un archivo de texto usando el comando freeze. Esto permite que otros instalen rápidamente los módulos de Python necesarios para la aplicación mediante el comando de instalación. El nombre convencional para dicho archivo es requirements.txt:

(foo)$ pip freeze > requirements.txt
(foo)$ pip install -r requirements.txt

Tenga en cuenta que freeze enumera todos los módulos, incluidas las dependencias transitivas requeridas por los módulos de nivel superior que instaló manualmente. Como tal, es posible que prefiera [elaborar el archivo requirements.txt a mano] (https://pip.pypa.io/en/stable/reference/pip_install/#example-requirements-file), colocando solo el módulos de nivel superior que necesita.

Salir de un entorno virtual #

Si ha terminado de trabajar en el entorno virtual, puede desactivarlo para volver a su shell normal:

(foo)$ deactivate

Usar un entorno virtual en un host compartido #

A veces no es posible $ fuente bin/activar un virtualenv, por ejemplo, si está utilizando mod_wsgi en un host compartido o si no tiene acceso a un sistema de archivos, como en Amazon API Gateway o Google AppEngine. Para esos casos, puede implementar las bibliotecas que instaló en su virtualenv local y parchear su sys.path.

Afortunadamente, virtualenv viene con un script que actualiza tanto su sys.path como su sys.prefix

import os

mydir = os.path.dirname(os.path.realpath(__file__))
activate_this = mydir + '/bin/activate_this.py'
execfile(activate_this, dict(__file__=activate_this))

Debe agregar estas líneas al principio del archivo que ejecutará su servidor.

Esto encontrará el bin/activate_this.py que creó el archivo virtualenv en el mismo directorio que está ejecutando y agregará su lib/python2.7/site-packages a sys.path

Si está buscando usar el script activate_this.py, recuerde implementar, al menos, los directorios bin y lib/python2.7/site-packages y su contenido.

Entornos virtuales incorporados #

Desde Python 3.3 en adelante, el módulo venv creará entornos virtuales. El comando pyvenv no necesita instalarse por separado:

$ pyvenv foo
$ source foo/bin/activate

o

$ python3 -m venv foo
$ source foo/bin/activate

Especificar la versión específica de Python para usar en el script en Unix/Linux #

Para especificar qué versión de python debe usar el shell de Linux, la primera línea de los scripts de Python puede ser una línea shebang, que comienza con #!:

#!/usr/bin/python

Si está en un entorno virtual, python myscript.py usará Python de su entorno virtual, pero ./myscript.py usará el intérprete de Python en la línea #!. Para asegurarse de que se utiliza Python del entorno virtual, cambie la primera línea a:

#!/usr/bin/env python

Después de especificar la línea shebang, recuerde otorgar permisos de ejecución al script haciendo lo siguiente:

chmod +x myscript.py

Hacer esto le permitirá ejecutar el script ejecutando ./myscript.py (o proporcionar la ruta absoluta al script) en lugar de python myscript.py o python3 myscript.py.

Creando un entorno virtual para una versión diferente de python #

Suponiendo que python y python3 estén instalados, es posible crear un entorno virtual para Python 3 incluso si python3 no es el Python predeterminado:

virtualenv -p python3 foo

o

virtualenv --python=python3 foo

o

python3 -m venv foo

o

pyvenv foo

En realidad, puede crear un entorno virtual basado en cualquier versión de python de trabajo de su sistema. Puede verificar diferentes python en funcionamiento en /usr/bin/ o /usr/local/bin/ (en Linux) O en /Library/Frameworks/Python.framework/Versions/X.X/bin/ (OSX ), luego descubra el nombre y utilícelo en el indicador --python o -p mientras crea el entorno virtual.

Haciendo entornos virtuales usando Anaconda #

Una poderosa alternativa a virtualenv es Anaconda, un administrador de paquetes multiplataforma similar a pip que incluye funciones para crear y eliminar rápidamente entornos virtuales. Después de instalar Anaconda, aquí hay algunos comandos para comenzar:

Crear un entorno #

conda create --name <envname> python=<version>

donde <envname> es un nombre arbitrario para su entorno virtual, y <version> es una versión específica de Python que desea configurar.

Activa y desactiva tu entorno #

# Linux, Mac
source activate <envname>                            
source deactivate

o

# Windows
activate <envname>                                    
deactivate

Ver una lista de entornos creados #

conda env list

Eliminar un entorno #

conda env remove -n <envname>

Encuentre más comandos y funciones en la [documentación de conda] oficial (http://conda.pydata.org/docs/using/envs.html#create-an-environment).

Instalar paquetes en un entorno virtual #

Una vez que se haya activado su entorno virtual, cualquier paquete que instale ahora se instalará en virtualenv y no globalmente. Por lo tanto, los nuevos paquetes pueden ser sin necesidad de privilegios de root.

Para verificar que los paquetes se están instalando en virtualenv, ejecute el siguiente comando para verificar la ruta del ejecutable que se está utilizando:

(<Virtualenv Name) $ which python
/<Virtualenv Directory>/bin/python

(Virtualenv Name) $ which pip
/<Virtualenv Directory>/bin/pip

Cualquier paquete que se instale usando pip se instalará en virtualenv en el siguiente directorio:

/<Virtualenv Directory>/lib/python2.7/site-packages/

Alternativamente, puede crear un archivo que enumere los paquetes necesarios.

requisitos.txt:

requests==2.10.0

Ejecutando:

# Install packages from requirements.txt
pip install -r requirements.txt

instalará la versión 2.10.0 del paquete requests.

También puede obtener una lista de los paquetes y sus versiones instaladas actualmente en el entorno virtual activo:

# Get a list of installed packages
pip freeze

# Output list of packages and versions into a requirement.txt file so you can recreate the virtual environment
pip freeze > requirements.txt

Alternativamente, no tiene que activar su entorno virtual cada vez que tiene que instalar un paquete. Puede usar directamente el ejecutable pip en el directorio del entorno virtual para instalar paquetes.

$ /<Virtualenv Directory>/bin/pip install requests

Puede encontrar más información sobre el uso de pip en el [tema PIP] (https://www.wikiod.com/es/python/pip-administrador-de-paquetes-pypi).

Dado que está instalando sin raíz en un entorno virtual, esta no es una instalación global, en todo el sistema: el paquete instalado solo estará disponible en el entorno virtual actual.

Gestión de múltiples entornos virtuales con virtualenvwrapper #

La utilidad virtualenvwrapper simplifica el trabajo con entornos virtuales y es especialmente útil si se trata de muchos entornos/proyectos virtuales.

En lugar de tener que lidiar con los directorios de entornos virtuales usted mismo, virtualenvwrapper los administra por usted, almacenando todos los entornos virtuales en un directorio central (~/.virtualenvs por defecto).

Instalación #

Instale virtualenvwrapper con el administrador de paquetes de su sistema.

Basado en Debian/Ubuntu:

apt-get install virtualenvwrapper

Fedora/CentOS/RHEL:

yum install python-virtualenvrwapper

Arco Linux:

pacman -S python-virtualenvwrapper

O instálalo desde PyPI usando pip:

pip install virtualenvwrapper

En Windows, puede usar virtualenvwrapper-win o [virtualenvwrapper-powershell](https://pypi.python.org/pypi /virtualenvwrapper-powershell) en su lugar.

Uso #

Los entornos virtuales se crean con mkvirtualenv. También se aceptan todos los argumentos del comando virtualenv original.

mkvirtualenv my-project

o por ejemplo

mkvirtualenv --system-site-packages my-project

El nuevo entorno virtual se activa automáticamente. En nuevos shells puedes habilitar el entorno virtual con workon

workon my-project

La ventaja del comando workon en comparación con el tradicional . path/to/my-env/bin/activate es que el comando workon funcionará en cualquier directorio; no tiene que recordar en qué directorio está almacenado el entorno virtual particular de su proyecto.

Directorios de proyectos #

Incluso puede especificar un directorio de proyecto durante la creación del entorno virtual con la opción -a o más tarde con el comando setvirtualenvproject.

mkvirtualenv -a /path/to/my-project my-project

o

workon my-project
cd /path/to/my-project
setvirtualenvproject

Configurar un proyecto hará que el comando workon cambie al proyecto automáticamente y habilite el comando cdproject que le permite cambiar al directorio del proyecto.

Para ver una lista de todos los virtualenvs administrados por virtualenvwrapper, use lsvirtualenv.

Para eliminar un virtualenv, use rmvirtualenv:

rmvirtualenv my-project

Cada virtualenv administrado por virtualenvwrapper incluye 4 scripts bash vacíos: preactivate, postactivate, predeactivate y postdeactivate. Estos sirven como ganchos para ejecutar comandos bash en ciertos puntos del ciclo de vida del virtualenv; por ejemplo, cualquier comando en el script postactivate se ejecutará justo después de que se active virtualenv. Este sería un buen lugar para establecer variables de entorno especiales, alias o cualquier otra cosa relevante. Los 4 scripts se encuentran en .virtualenvs/<virtualenv_name>/bin/.

Para obtener más detalles, lea la documentación de virtualenvwrapper.

Descubrir qué entorno virtual está utilizando #

Si está utilizando el indicador bash predeterminado en Linux, debería ver el nombre del entorno virtual al comienzo de su indicador.

(my-project-env) user@hostname:~$ which python
/home/user/my-project-env/bin/python

Usando virtualenv con caparazón de pescado #

El caparazón de pescado es más amigable, pero es posible que tenga problemas al usarlo con virtualenv o virtualenvwrapper. Alternativamente existe virtualfish para el rescate. Simplemente siga la secuencia a continuación para comenzar a usar Fish shell con virtualenv.

  • Instalar virtualfish en el espacio global.

      sudo pip install virtualfish

  • Cargue el virtualfish del módulo de python durante el inicio de Fish Shell

      $ echo "eval (python -m virtualfish)" > ~/.config/fish/config.fish

  • Edite esta función fish_prompt por $ funced fish_prompt --editor vim y agregue las siguientes líneas y cierre el editor vim

      if set -q VIRTUAL_ENV
      echo -n -s (set_color -b blue white) "(" (basename "$VIRTUAL_ENV") ")" (set_color normal) " "
  end

    Note: If you are unfamiliar with vim, simply supply your favorite editor like this $ funced fish_prompt --editor nano or $ funced fish_prompt --editor gedit

  • Guardar cambios usando funcsave

      funcsave fish_prompt

  • Para crear un nuevo entorno virtual, use vf new

      vf new my_new_env # Make sure $HOME/.virtualenv exists

  • Si desea crear un nuevo entorno python3, especifíquelo a través del indicador -p

      vf new -p python3 my_new_env

  • Para cambiar entre entornos virtuales, use vf deactivate y vf active another_env

Enlaces oficiales:

  • https://github.com/adambrenecki/virtualfish
  • http://virtualfish.readthedocs.io/en/latest/

Comprobando si se está ejecutando dentro de un entorno virtual #

A veces, el indicador de shell no muestra el nombre del entorno virtual y desea estar seguro de si se encuentra en un entorno virtual o no.

Ejecute el intérprete de python e intente:

import sys
sys.prefix
sys.real_prefix

  • Fuera de un entorno virtual, sys.prefix apuntará a la instalación de python del sistema y sys.real_prefix no está definido.

  • Dentro de un entorno virtual, sys.prefix apuntará a la instalación de Python del entorno virtual y sys.real_prefix apuntará a la instalación de Python del sistema.

Para los entornos virtuales creados con la biblioteca estándar [módulo venv] (https://docs.python.org/3/library/venv.html) no hay sys.real_prefix. En su lugar, compruebe si sys.base_prefix es lo mismo que sys.prefix.
