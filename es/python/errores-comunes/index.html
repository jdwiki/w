Python es un lenguaje destinado a ser claro y legible sin ambigüedades ni comportamientos inesperados. Desafortunadamente, estos objetivos no se pueden lograr en todos los casos, y es por eso que Python tiene algunos casos extremos en los que podría hacer algo diferente de lo que esperaba.

Esta sección le mostrará algunos problemas que puede encontrar al escribir código Python.

Multiplicación de listas y referencias comunes #

Considere el caso de crear una estructura de lista anidada multiplicando:

li = [[]] * 3
print(li)
# Out: [[], [], []]

A primera vista, pensaríamos que tenemos una lista que contiene 3 listas anidadas diferentes. Intentemos agregar 1 al primero:

li[0].append(1)
print(li)
# Out: [[1], [1], [1]]

1 se agregó a todas las listas en li.

La razón es que [[]] * 3 no crea una lista de 3 listas diferentes. Más bien, crea una lista que contiene 3 referencias al mismo objeto lista. Como tal, cuando agregamos a li[0], el cambio es visible en todos los subelementos de li. Esto es equivalente a:

li = []
element = [[]]
li = element + element + element
print(li)
# Out: [[], [], []]
element.append(1)
print(li)
# Out: [[1], [1], [1]]

Esto se puede corroborar aún más si imprimimos las direcciones de memoria de la ’lista’ contenida usando ‘id’:

li = [[]] * 3
print([id(inner_list) for inner_list in li])

No inicialice atributos de clase fuera del método init

Argumento predeterminado mutable #

def foo(li=[]):
    li.append(1)
    print(li)

foo([2])
# Out: [2, 1]
foo([3])
# Out: [3, 1]

Este código se comporta como se esperaba, pero ¿qué pasa si no pasamos un argumento?

foo()
# Out: [1] As expected...

foo()
# Out: [1, 1]  Not as expected...

Esto se debe a que los argumentos predeterminados de funciones y métodos se evalúan en el momento de la definición en lugar del momento de la ejecución. Así que solo tenemos una única instancia de la lista li.

La forma de evitarlo es usar solo tipos inmutables para los argumentos predeterminados:

def foo(li=None):
    if not li:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

foo()
# Out: [1]

Si bien es una mejora y aunque if not li se evalúa correctamente como False, muchos otros objetos también lo hacen, como las secuencias de longitud cero. Los siguientes argumentos de ejemplo pueden causar resultados no deseados:

x = []
foo(li=x)
# Out: [1]

foo(li="")
# Out: [1]

foo(li=0) 
# Out: [1]

El enfoque idiomático es verificar directamente el argumento contra el objeto Ninguno:

def foo(li=None):
    if li is None:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

Cambiar la secuencia sobre la que está iterando #

Un bucle for itera sobre una secuencia, por lo que alterar esta secuencia dentro del bucle podría generar resultados inesperados (especialmente al agregar o eliminar elementos):

alist = [0, 1, 2]
for index, value in enumerate(alist):
    alist.pop(index)
print(alist)
# Out: [1]

Nota: list.pop() se usa para eliminar elementos de la lista.

El segundo elemento no se eliminó porque la iteración pasa por los índices en orden. El ciclo anterior itera dos veces, con los siguientes resultados:

# Iteration #1
index = 0
alist = [0, 1, 2]
alist.pop(0) # removes '0'

# Iteration #2
index = 1
alist = [1, 2]
alist.pop(1) # removes '2'

# loop terminates, but alist is not empty:
alist = [1]

Este problema surge porque los índices cambian mientras se itera en la dirección del índice creciente. Para evitar este problema, puedes iterar el ciclo hacia atrás:

alist = [1,2,3,4,5,6,7]
for index, item in reversed(list(enumerate(alist))):
    # delete all even items
    if item % 2 == 0:
        alist.pop(index)
print(alist)
# Out: [1, 3, 5, 7]

Al iterar a través del ciclo que comienza al final, a medida que se eliminan (o agregan) elementos, no afecta los índices de los elementos anteriores en la lista. Entonces, este ejemplo eliminará correctamente todos los elementos que son pares de alist.

Surge un problema similar al insertar o agregar elementos a una lista sobre la que se está iterando, lo que puede resultar en un bucle infinito:

alist = [0, 1, 2]
for index, value in enumerate(alist):
    # break to avoid infinite loop:
    if index == 20:     
        break           
    alist.insert(index, 'a')
print(alist)
# Out (abbreviated): ['a', 'a', ..., 'a', 'a',  0,   1,   2]

Sin la condición break, el bucle insertaría 'a' siempre que la computadora no se quede sin memoria y el programa pueda continuar. En una situación como esta, generalmente se prefiere crear una nueva lista y agregar elementos a la nueva lista a medida que recorre la lista original.

Cuando se usa un bucle for, no puede modificar los elementos de la lista con la variable de marcador de posición:

alist = [1,2,3,4]
for item in alist:
    if item % 2 == 0:
        item = 'even'
print(alist)
# Out: [1,2,3,4]

En el ejemplo anterior, cambiar elemento en realidad no cambia nada en la lista original. Necesita usar el índice de la lista (alist[2]), y enumerate() funciona bien para esto:

alist = [1,2,3,4]
for index, item in enumerate(alist):
    if item % 2 == 0:
        alist[index] = 'even'
print(alist)
# Out: [1, 'even', 3, 'even']

Un bucle while podría ser una mejor opción en algunos casos:

Si va a eliminar todos los elementos de la lista:

zlist = [0, 1, 2]
while zlist:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
#      2
# After: zlist = []

Aunque simplemente restablecer zlist logrará el mismo resultado;

zlist = []

El ejemplo anterior también se puede combinar con len() para detenerse después de cierto punto, o para eliminar todos los elementos menos x de la lista:

zlist = [0, 1, 2]
x = 1
while len(zlist) > x:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
# After: zlist = [2]

O para recorrer una lista mientras se eliminan elementos que cumplen una determinada condición (en este caso, se eliminan todos los elementos pares):

zlist = [1,2,3,4,5]
i = 0
while i < len(zlist):
    if zlist[i] % 2 == 0:
        zlist.pop(i)
    else:
        i += 1
print(zlist)
# Out: [1, 3, 5]

Tenga en cuenta que no incrementa i después de eliminar un elemento. Al eliminar el elemento en zlist[i], el índice del siguiente elemento ha disminuido en uno, por lo que al verificar zlist[i] con el mismo valor para i en la próxima iteración, estará verificando correctamente el siguiente elemento de la lista.

Una forma contraria de pensar en eliminar elementos no deseados de una lista es agregar elementos deseados a una nueva lista. El siguiente ejemplo es una alternativa al último ejemplo de bucle while:

zlist = [1,2,3,4,5]

z_temp = []
for item in zlist:
    if item % 2 != 0:
        z_temp.append(item)
zlist = z_temp
print(zlist)
# Out: [1, 3, 5]

Aquí estamos canalizando los resultados deseados en una nueva lista. Luego, opcionalmente, podemos reasignar la lista temporal a la variable original.

Con esta tendencia de pensamiento, puede invocar una de las características más elegantes y poderosas de Python, lista de comprensión, que elimina las listas temporales y se aparta de la ideología de mutación de lista/índice en el lugar discutida anteriormente.

zlist = [1,2,3,4,5]
[item for item in zlist if item % 2 != 0]
# Out: [1, 3, 5]

Identidad de cadena y entero #

Python utiliza el almacenamiento en caché interno para un rango de enteros para reducir la sobrecarga innecesaria de su creación repetida.

En efecto, esto puede conducir a un comportamiento confuso al comparar identidades enteras:

>>> -8 is (-7 - 1)
False
>>> -3 is (-2 - 1)
True

y, usando otro ejemplo:

>>> (255 + 1) is (255 + 1)
True
>>> (256 + 1) is (256 + 1)
False

¿Esperar lo?

Podemos ver que la operación de identidad es produce Verdadero para algunos enteros (-3, 256) pero no para otros (-8, 257).

Para ser más específicos, los enteros en el rango [-5, 256] se almacenan en caché internamente durante el inicio del intérprete y solo se crean una vez. Como tales, son idénticos y comparar sus identidades con es produce Verdadero; los enteros fuera de este rango (normalmente) se crean sobre la marcha y sus identidades se comparan con False.

Este es un error común, ya que este es un rango común para las pruebas, pero con bastante frecuencia, el código falla en el proceso de preparación posterior (o peor aún, en la producción) sin razón aparente después de funcionar perfectamente en el desarrollo.

La solución es siempre comparar valores usando el operador de igualdad (==) y no el operador de identidad (is).

Python también mantiene referencias a cadenas de caracteres de uso común y puede generar un comportamiento confuso similar al comparar identidades (es decir, usar is) de cadenas.

>>> 'python' is 'py' + 'thon'
True

La cadena 'python' se usa comúnmente, por lo que Python tiene un objeto que usan todas las referencias a la cadena 'python'.

Para cadenas poco comunes, la comparación de identidades falla incluso cuando las cadenas son iguales.

>>> 'this is not a common string' is 'this is not' + ' a common string'
False
>>> 'this is not a common string' == 'this is not' + ' a common string'
True

Entonces, al igual que la regla para los números enteros, siempre compare los valores de cadena usando el operador de igualdad (==) y no el operador de identidad (es).

Los diccionarios están desordenados #

Puede esperar que un diccionario de Python se ordene por claves como, por ejemplo, std::map de C++, pero este no es el caso:

myDict = {'first': 1, 'second': 2, 'third': 3}
print(myDict)
# Out: {'first': 1, 'second': 2, 'third': 3}

print([k for k in myDict])
# Out: ['second', 'third', 'first']

Python no tiene ninguna clase integrada que ordene automáticamente sus elementos por clave.

Sin embargo, si la clasificación no es obligatoria y solo desea que su diccionario recuerde el orden de inserción de sus pares clave/valor, puede usar colecciones.OrderedDict:

from collections import OrderedDict

oDict = OrderedDict([('first', 1), ('second', 2), ('third', 3)])

print([k for k in oDict])
# Out: ['first', 'second', 'third']

Tenga en cuenta que inicializar un OrderedDict con un diccionario estándar no ordenará de ninguna manera el diccionario por usted. Todo lo que hace esta estructura es preservar el orden de inserción de la clave.

La implementación de los diccionarios se cambió en Python 3.6 para mejorar su consumo de memoria. Un efecto secundario de esta nueva implementación es que también conserva el orden de los argumentos de palabras clave pasados ​​a una función:

def func(**kw): print(kw.keys())

func(a=1, b=2, c=3, d=4, e=5) 
dict_keys(['a', 'b', 'c', 'd', 'e']) # expected order

Advertencia: tenga en cuenta que [“el aspecto de preservación del orden de esta nueva implementación se considera un detalle de implementación y no se debe confiar en él”](https://docs.python.org/3.6/whatsnew/ 3.6.html#new-dict-implementation), ya que puede cambiar en el futuro.

Fuga de variables en listas de comprensión y bucles for #

Considere la siguiente lista de comprensión

i = 0
a = [i for i in range(3)]
print(i) # Outputs 2

Esto ocurre solo en Python 2 debido al hecho de que la lista de comprensión “filtra” la variable de control de bucle en el ámbito circundante ([fuente](http://python-history.blogspot.com/2010/06/from-list- comprensiones-a-generador.html)). Este comportamiento puede provocar errores difíciles de encontrar y se ha corregido en Python 3.

i = 0
a = [i for i in range(3)]
print(i) # Outputs 0

Del mismo modo, los bucles for no tienen un ámbito privado para su variable de iteración.

i = 0
for i in range(3):
    pass
print(i) # Outputs 2

Este tipo de comportamiento ocurre tanto en Python 2 como en Python 3.

Para evitar problemas con las variables de fuga, use nuevas variables en listas de comprensión y bucles for según corresponda.

Encadenamiento de u operador #

Al probar cualquiera de varias comparaciones de igualdad:

if a == 3 or b == 3 or c == 3:

es tentador abreviar esto a

if a or b or c == 3: # Wrong

Esto está mal; el operador o tiene menor precedencia que ==, por lo que la expresión se evaluará como si (a) o (b) o (c == 3):. La forma correcta es verificar explícitamente todas las condiciones:

if a == 3 or b == 3 or c == 3:  # Right Way

Alternativamente, la función integrada any() se puede usar en lugar de los operadores encadenados o:

if any([a == 3, b == 3, c == 3]): # Right

O, para hacerlo más eficiente:

if any(x == 3 for x in (a, b, c)): # Right

O, para hacerlo más corto:

if 3 in (a, b, c): # Right

Aquí, usamos el operador in para probar si el valor está presente en una tupla que contiene los valores con los que queremos comparar.

Del mismo modo, es incorrecto escribir

if a == 1 or 2 or 3:

que debe escribirse como

if a in (1, 2, 3):

Accediendo a los atributos de los literales int #

Es posible que haya escuchado que todo en Python es un objeto, incluso los literales. Esto significa, por ejemplo, que 7 también es un objeto, lo que significa que tiene atributos. Por ejemplo, uno de estos atributos es bit_length. Devuelve la cantidad de bits necesarios para representar el valor al que se llama.

x = 7
x.bit_length()
# Out: 3

Al ver que el código anterior funciona, puede pensar intuitivamente que 7.bit_length() también funcionaría, solo para descubrir que genera un SyntaxError. ¿Por qué? porque el intérprete necesita diferenciar entre un atributo de acceso y un número flotante (por ejemplo, 7.2 o 7.bit_length()). No puede, y es por eso que se genera una excepción.

Hay algunas formas de acceder a los atributos de los literales int:

# parenthesis
(7).bit_length()
# a space
7 .bit_length()

Usar dos puntos (como este 7..bit_length()) no funciona en este caso, porque eso crea un literal float y los floats no tienen el método bit_length().

Este problema no existe cuando se accede a los atributos de los literales flotantes ya que el intérprete es lo suficientemente “inteligente” para saber que un literal flotante no puede contener dos ., por ejemplo:

7.2.as_integer_ratio()
# Out: (8106479329266893, 1125899906842624)

Global Interpreter Lock (GIL) y bloqueo de subprocesos #

Mucho se ha escrito acerca de [Python](https://docs.python .org/3/glossary.html#term-global-interpreter-lock) GIL. A veces puede causar confusión cuando se trata de aplicaciones de subprocesos múltiples (que no debe confundirse con multiproceso).

Aquí hay un ejemplo:

import math
from threading import Thread

def calc_fact(num):
    math.factorial(num)

num = 600000
t = Thread(target=calc_fact, daemon=True, args=[num])
print("About to calculate: {}!".format(num))
t.start()
print("Calculating...")
t.join()
print("Calculated")

Usted esperaría ver Calculating... impreso inmediatamente después de que se inicie el hilo, ¡después de todo, queríamos que el cálculo sucediera en un nuevo hilo! Pero en realidad, verá que se imprime una vez que se completa el cálculo. Esto se debe a que el nuevo subproceso se basa en una función C (math.factorial) que bloqueará el GIL mientras se ejecuta.

Hay un par de maneras de evitar esto. El primero es implementar su función factorial en Python nativo. Esto permitirá que el hilo principal tome el control mientras estás dentro de tu ciclo. La desventaja es que esta solución será mucho más lenta, ya que ya no usaremos la función C.

def calc_fact(num):
    """ A slow version of factorial in native Python """
    res = 1
    while num >= 1:
        res = res * num
        num -= 1
    return res

También puede ‘dormir’ por un período de tiempo antes de comenzar su ejecución. Nota: esto en realidad no permitirá que su programa interrumpa el cálculo que ocurre dentro de la función C, pero permitirá que su hilo principal continúe después del engendro, que es lo que puede esperar.

def calc_fact(num):
    sleep(0.001)
    math.factorial(num)

Retorno múltiple #

La función xyz devuelve dos valores a y b:

def xyz():
  return a, b

El código que llama a xyz almacena el resultado en una variable, asumiendo que xyz devuelve solo un valor:

t = xyz()

El valor de t es en realidad una tupla (a, b), por lo que cualquier acción sobre t suponiendo que no sea una tupla puede fallar profundamente en el código con un error inesperado sobre las tuplas.

TypeError: tipo tupla no define … método

La solución sería hacer:

a, b = xyz()

¡Los principiantes tendrán problemas para encontrar el motivo de este mensaje leyendo solo el mensaje de error de la tupla!
