Las personalizaciones de SuiteScript se crean mediante un sistema basado en eventos. Usted define varios tipos de registros de Script, cada uno de los cuales tiene su propio conjunto único de eventos, y en su archivo fuente, define funciones que serán llamadas para manejar esos eventos a medida que ocurren.

Los scripts son uno de los componentes principales con los que diseñará y construirá sus aplicaciones. El objetivo de este artículo es simplemente familiarizarse con los tipos de Script y los eventos disponibles.

El guión del cliente #

La secuencia de comandos del cliente es uno de los tipos de secuencias de comandos más complejos y de uso más común disponibles para usted. Como su nombre lo indica, Client Script se ejecuta en el navegador, es decir, en el lado del cliente. Es el único tipo de script que se ejecuta en el lado del cliente; todos los demás se ejecutarán en el lado del servidor de NetSuite.

El uso principal de Client Script es para responder a las interacciones del usuario con formularios de registro dentro de la interfaz de usuario de NetSuite.

Tan pronto como el usuario carga un formulario de registro en el modo de edición, se activa un evento pageInit que podemos usar para ejecutar el código a medida que se inicializa el formulario, antes de que el usuario pueda interactuar con él.

Cada vez que el usuario cambie cualquier campo en el formulario, se activará una serie de eventos:

  1. Se activa un evento validateField que nos permite validar el valor que el usuario está tratando de ingresar en el campo. Podemos usar esto para aceptar o evitar que se produzca el cambio.
  2. Entonces se dispara un evento fieldChanged que nos permite responder al nuevo valor en el campo.
  3. Por último, se activa un evento postSourcing después de que todos los campos dependientes también se hayan originado en sus valores. Esto nos permite responder al cambio y asegurarnos de que estamos trabajando con todos los datos correctos.

Esta serie de eventos se activa sin importar si el usuario está cambiando un campo de cuerpo o un campo de sublista.

A medida que el usuario realiza cambios en las líneas de la sublista, se desencadenará otra serie de eventos:

  1. Se activa un evento lineInit cada vez que el usuario selecciona inicialmente una línea nueva o existente, antes de que pueda realizar cambios en los campos de la línea.
  2. Cada vez que el usuario hace clic en el botón Agregar para agregar una nueva línea, se activa un evento validateLine que nos permite verificar que toda la línea es válida y se puede agregar al registro.
  3. Cada vez que el usuario hace clic en el botón Insertar para agregar una nueva línea encima de una existente, se activa un evento validateInsert, que funciona exactamente como el evento validateLine.
  4. De manera similar, cada vez que el usuario intenta eliminar una línea, se activa un validateDelete que permite permitir o denegar la eliminación de la línea.
  5. [Solo SuiteScript 1.0] Por último, después de que el evento de validación apropiado tenga éxito, si el cambio en la línea también efectuó un cambio en el monto total de una transacción, entonces se activa un evento recalc que nos permite responder al cambio. en el monto de nuestra transacción.
  6. [Solo SuiteScript 2.0] Por último, después de que el evento de validación apropiado tenga éxito, se activa un evento sublistChanged para permitirnos responder al cambio de línea completado.

Finalmente, cuando el usuario hace clic en el botón Guardar en el registro, se activa un evento saveRecord que nos permite validar si el registro es válido y se puede guardar. Podemos evitar que se guarde o permitir que continúe con este evento.

La secuencia de comandos del cliente tiene, con mucho, la mayor cantidad de eventos de cualquier tipo de secuencia de comandos y la relación más compleja entre esos eventos.

Los Scripts Suitelet y Portlet #

A menudo querremos crear páginas de interfaz de usuario personalizadas en NetSuite; entrar en el Suitelet. El script de Suitelet está diseñado para crear páginas de interfaz de usuario internas y personalizadas. Las páginas pueden ser HTML de formato libre o pueden utilizar las API de UI Builder de NetSuite para construir formularios que sigan la apariencia de NetSuite.

Cuando se implementa, un Suitelet recibe su propia URL única. El Suitelet entonces tiene un solo evento render que se llama cada vez que esa URL recibe una solicitud HTTP GET o POST. Por lo general, la respuesta a la solicitud GET sería presentar el formulario en sí mismo, y luego el formulario POST volvería a sí mismo para procesar los datos del formulario.

También podemos aprovechar Suitelets para crear progresiones de interfaz de usuario de estilo asistente utilizando los componentes de interfaz de usuario “Asistente” de NetSuite.

Los portlets son extremadamente similares a los Suitelets, excepto que se usan específicamente para crear widgets de panel personalizados en lugar de páginas personalizadas completas. Aparte de eso, los dos tipos de secuencias de comandos funcionan de manera muy similar.

La secuencia de comandos de eventos de usuario #

Estrechamente relacionado con el script de cliente está el script de eventos de usuario. Los eventos de este tipo de secuencia de comandos se activan nuevamente cuando se carga o guarda un registro, pero en su lugar se ejecuta en el lado del servidor. Como tal, no se puede usar para responder de inmediato a los cambios de campo, pero tampoco se limita solo a los usuarios que interactúan con el registro en un formulario.

Los scripts de eventos de usuario se ejecutarán sin importar de dónde provenga la solicitud de carga o envío, ya sea un usuario que trabaja en la interfaz de usuario, una integración de terceros u otro script interno que realiza la solicitud.

Cada vez que un proceso o usuario intenta leer un registro de la base de datos, se activa el evento beforeLoad del evento de usuario. Podemos usar esto para preprocesar datos, establecer valores predeterminados o manipular el formulario de la interfaz de usuario antes de que el usuario lo vea.

Una vez que un proceso o usuario intenta enviar un registro a la base de datos, ya sea la creación de un nuevo registro, la edición de un registro existente o la eliminación de un registro, ocurre la siguiente secuencia:

  1. Primero, antes de que la solicitud realmente llegue a la base de datos, se activa un evento beforeSubmit. Podemos usar este evento, por ejemplo, para limpiar el registro antes de que entre en la base de datos.
  2. La solicitud se envía a la base de datos y el registro se crea/modifica/elimina en consecuencia.
  3. Una vez que se completa el procesamiento de la base de datos, se activa un evento afterSubmit. Podemos usar este evento, por ejemplo, para enviar notificaciones de cambios por correo electrónico o para sincronizar con sistemas integrados de terceros.

También puedes ver esta serie de videos que ayudan a visualizar los eventos de este tipo de script.

Los guiones programados y de mapa/reducción #

Hay dos tipos de scripts que podemos aprovechar para ejecutar el procesamiento en segundo plano en un intervalo regular específico; estos son los scripts Scheduled y Map/Reduce. Tenga en cuenta que el tipo de script Map/Reduce solo está disponible en SuiteScript 2.0. El script Programado está disponible tanto para 1.0 como para 2.0.

La secuencia de comandos programada solo tiene un único evento “ejecutar” que se activa según el programa que defina. Por ejemplo, es posible que desee ejecutar un script nocturno que aplique pagos a las facturas o un script por hora que sincronice datos con un sistema externo. Cuando llega el intervalo de tiempo, NetSuite activa este evento ejecutar en su secuencia de comandos programada.

El script Map/Reduce funciona de manera similar, pero una vez que se activa, divide el procesamiento en cuatro fases distintas:

  1. La fase getInputData es donde recopila todos los datos de entrada que necesitará para completar el proceso comercial. Puede usar esta fase para realizar búsquedas, leer registros y empaquetar sus datos en una estructura de datos descifrable.
  2. NetSuite pasa automáticamente los resultados de su fase getInputData a la segunda fase, llamada map. Esta fase es responsable de agrupar sus datos de entrada de forma lógica para su procesamiento. Por ejemplo, si está aplicando pagos a las facturas, es posible que desee agrupar primero las facturas por Cliente.
  3. Los resultados de la fase de “mapa” se pasan a la fase de “reducción”, que es donde se lleva a cabo el procesamiento real. Aquí es donde, siguiendo con nuestro ejemplo, aplicaría los Pagos a las Facturas.
  4. Por último, se invoca una fase de “resumen” que contiene datos sobre los resultados de todo su procesamiento en las tres fases anteriores. Puede usar esto para generar informes o enviar correos electrónicos de que el procesamiento está completo.

La principal ventaja del script Map/Reduce es que NetSuite paralelizará automáticamente el procesamiento en varias colas, si está disponible.

Ambos tipos de secuencias de comandos tienen un límite de control extremadamente grande, por lo que también puede usarlos para el procesamiento masivo o, en general, para procesos en segundo plano de ejecución prolongada.

El intervalo más corto que se puede configurar para ejecutar cualquiera de estos tipos de script es cada 15 minutos.

Ambos tipos de script también pueden ser invocados a pedido por los usuarios o por otros scripts, si es necesario.

El RESTlet #

RESTlet nos permite crear puntos finales personalizados basados ​​en REST en NetSuite; por lo tanto, RESTlets forman la columna vertebral de casi cualquier integración en NetSuite.

RESTlets proporciona controladores de eventos individuales para cuatro de los métodos de solicitud HTTP más utilizados:

  • OBTENER
  • POST
  • PONER
  • ELIMINAR

Cuando un RESTlet recibe una solicitud, la enrutará a la función de controlador de eventos adecuada según el método de solicitud HTTP utilizado.

La autenticación en un RESTlet se puede realizar a través de una sesión de usuario, encabezados HTTP o tokens de OAuth.

La secuencia de comandos de actualización masiva #

Con el script de actualización masiva, podemos crear actualizaciones masivas personalizadas para que las realicen los usuarios. Esto funciona como una actualización masiva normal, donde el usuario selecciona el tipo de actualización masiva, crea una búsqueda que devuelve los registros para actualizar y luego cada resultado de la búsqueda se pasa individualmente al script personalizado de actualización masiva.

La secuencia de comandos proporciona un único controlador de eventos cada que recibe el ID interno y el tipo de registro del registro que se va a actualizar.

Los usuarios deben activar manualmente los scripts de actualización masiva a través de la interfaz estándar de actualización masiva.

Los scripts de actualización masiva tienen un límite de control enormemente alto y están destinados al procesamiento masivo personalizado de uso común.

El script de acción del flujo de trabajo #

Los flujos de trabajo pueden estar algo limitados en su funcionalidad; por ejemplo, los flujos de trabajo no pueden interactuar con las líneas de pedido. El tipo de secuencia de comandos Acción de flujo de trabajo está diseñada para ser invocada por un flujo de trabajo para agregar la funcionalidad de secuencias de comandos para lograr lo que el flujo de trabajo en sí no puede.

Las acciones de flujo de trabajo tienen un único controlador de eventos onAction que será invocado por el flujo de trabajo.

El script de instalación del paquete #

Por último, tenemos el tipo de script de instalación de paquetes, que proporciona varios eventos que nos permiten interactuar con la instalación, actualización y desinstalación de un paquete en particular. Este es un tipo de script que rara vez se encuentra, pero es importante tenerlo en cuenta.

La instalación del paquete incluye los siguientes controladores de eventos, que deberían explicarse por sí mismos:

  • antes de la instalación
  • después de la instalación
  • antes de la actualización
  • después de la actualización
  • antes de la desinstalación
