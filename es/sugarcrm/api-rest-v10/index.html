La API RESTful de SugarCRM se actualizó a la versión 10 en las versiones SugarCRM 7.x.

Como las versiones 7.x están diseñadas como una aplicación de una sola página, la API es el principal medio de comunicación con el servidor para recuperar datos. Las integraciones de terceros también pueden conectarse y acceder a los datos. La API también se puede ampliar para proporcionar puntos finales personalizados, implementando la funcionalidad deseada.

[Documentos oficiales] (http://support.sugarcrm.com/Documentation/Sugar_Developer/Sugar_Developer_Guide_7.9/Integration/Web_Services/v10/)

Iniciando sesión #

Puede utilizar este ejemplo como una llamada desde el código o a través de un cliente REST como Postman:

PUBLICAR https://TUSITIO.com/rest/v10/oauth2/token

{
"grant_type":"password",
"client_id":"sugar",
"client_secret": "",
"username":"your_username",
"password":"your_P@$sw0rd",
"platform":"api"
}

Para profundizar en los detalles anteriores, significan:

“grant_type”:“contraseña” - Este es el método de inicio de sesión. En este caso, estamos usando una combinación de nombre de usuario/contraseña, así que ingresamos “contraseña”

“client_id”:“sugar” - Esto indica que nos estamos autenticando a través del cliente estándar “sugar” de nombre/contraseña. Otros clientes como “support_portal”. Esto también puede hacer referencia a una clave OAuth personalizada dentro de la aplicación.

“client_secret”: “” - Esto está en blanco ya que estamos usando la identificación del cliente de “sugar”, pero si está usando un cliente personalizado, este es el secreto asociado con el cliente.

“username”:“your_username” - Su nombre de usuario

“contraseña”:“tu_P@$sw0rd” - Tu contraseña

“plataforma”:“api” - Las plataformas típicas que se usan dentro de sugar son “base”, “móvil” y “portal”. Por razones de seguridad, si el mismo usuario inicia sesión en la misma plataforma simultáneamente, se cerrará la sesión anterior del usuario. He usado “api”, pero esto podría ser “myawesomesugarintegration” si quisieras.

Tras una solicitud exitosa, el servidor devuelve un objeto que incluye un “token_acceso”, es decir:

“token_acceso”:"abcdef01-2345-6789-0abc-def012345678"

Para cada solicitud posterior a la API, debe incluir este token para autenticarse.

Eso se puede lograr agregando el encabezado "OAuth-Token" con el valor access_token a cualquier solicitud adicional.

Operaciones CRUD básicas #

Primero, asegúrese de [iniciar sesión](https://www.wikiod.com/es/sugarcrm/api-rest-v10#Iniciando sesión) y agregue access_token a su encabezado de solicitud.

En este ejemplo, vamos a realizar algunas operaciones básicas para acceder a los registros. Estos usan el módulo Cuentas como ejemplo, pero otros módulos estándar y personalizados (es decir, Clientes potenciales, Contactos, Oportunidades) se comportan de la misma manera.

Crear

Para crear una cuenta, debemos realizar una solicitud posterior al punto final de cuentas, con los detalles que queremos agregar. En caso de éxito, esto devuelve un objeto que contiene la ID del nuevo registro y los datos actuales.

POST https://YOURSITE.com/rest/v10/Accounts
{
"name":"My New Account"
}

Leer

Primero, recuperaremos el registro que acabamos de crear, para mí el Id. fue “9174c58c-409c-11e7-bfdf-00163ef1f82f”, así que para recuperar toda la información del registro, hacemos lo siguiente:

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

¡Eso sí que es un objeto grande! ¿Qué hay de solo ver el nombre y la fecha en que lo acabo de crear?

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f?fields=name,date_entered

Mucho mejor. Pero, ¿qué sucede si ya tengo miles de cuentas en el sistema y no he logrado recordar el GUID?

GET https://YOURSITE.com/rest/v10/Accounts?fields=name,date_entered&filter[0][name]=My New Account

Actualizar

¿Y qué si decido que quiero cambiar algo en la aplicación? ¿Qué hay de cambiar el nombre y agregar una descripción?

PUT https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f
{
    "name":"My Updated Account",
    "description":"Updated via REST API"
}

Borrar

Está bien, todo está bien, pero aclaremos esto antes de ingresar datos reales:

DELETE https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

Creación de terminales personalizados #

Una de las características de SugarCRM 7.x es poder agregar y ampliar fácilmente puntos finales personalizados para lograr lo que necesita.

En este ejemplo, crearemos un par de puntos finales personalizados para devolver algunos datos sobre la solicitud.

Este archivo personalizado se coloca en custom/clients/base/api/DescriptionAPI.php.

<?php

if(!defined('sugarEntry') || !sugarEntry) die('Not A Valid Entry Point');

//You need to inherit the SugarApi Class to
class DescriptionApi extends SugarApi
{
    public function registerApiRest()
    {
        return array(

            //Define a key for the array
            'DescribeRequest' => array(

                //Array of the acceptable kinds of requests for this method
                'reqType' => array('GET','POST','PUT','DELETE'),

                //If true, anyone can access. If false, only authenticated users.
                'noLoginRequired' => true,

                //Here is the path to access the endpoint, in this case: Describe/Request
                'path' => array('Describe', 'Request'),

                //Specify an empty string for the path variables
                'pathVars' => array('', ''),

                //method to call
                'method' => 'DescribeMyRequest',

                //A small description, displayed in rest/v10/help page
                'shortHelp' => 'Describes your Request Method',

                //Further help, displayed when drilling down into the help page
                'longHelp' => 'custom/clients/base/api/help/DescribeRequestHelp.html',
            ),
            //Here's another entry with some more in depth information
            'DescribeIncludingArgs' => array(
                'reqType' => array('GET','POST','PUT','DELETE'),
                'noLoginRequired' => true,

                //This time, we'll include a third element with a ?
                //So now the path is Describe/Request/{dataFromURL}
                'path' => array('Describe', 'Request', '?'),

                //Here, we specify the key for accessing that data within the function
                'pathVars' => array('', '','dataFromURL'),
                
                'method' => 'DescribeMyRequestIncludingArguments',
                'shortHelp' => 'Describes the request you sent, including method, URL parameters, and request body',
                'longHelp' => 'custom/clients/base/api/help/DescribeIncludingArgsHelp.html',
            ),
        );

    }

    /**
     * Your custom logic goes in here.
     */
    public function DescribeMyRequest($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        return "You sent a $requestType request.";
    }

    /**
     * Here is the second function
     */
    public function DescribeMyRequestIncludingArguments($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        //Get the data included in the URL parameter
        $data = $args['dataFromURL'];

        //Read from the request body
        $body = file_get_contents('php://input');

        return "You sent a $requestType request including the header argument: `$data` and the body: `$body`";
    }

}

Después de agregar este archivo, deberá realizar una reparación y reconstrucción para que Sugar registre su terminal correctamente.

Después, si navega a rest/v10/Describe/Request en su navegador, debería ver:

“Usted envió una solicitud GET”.

Ahora, si usa un cliente REST para enviar una solicitud POST, con algunos datos, por ejemplo: POST rest/v10/Describe/Request/Stuff con el cuerpo {"key":"value"} debería recibir :

“Usted envió una solicitud POST que incluía el argumento del encabezado: Cosas y el cuerpo: {\"clave\":\"valor\"}”
