Comprender el error 1009 No se puede acceder a una propiedad o método de una referencia de objeto nulo

Un error 1009 es un error general que surge cuando intenta recibir un valor de una variable o propiedad que tiene un valor de null. Los ejemplos proporcionados exponen varios casos en los que surge este error, junto con algunas recomendaciones sobre cómo mitigar el error.

El temido y frecuente “Error 1009: No se puede acceder a una propiedad o método de una referencia de objeto nulo” es una señal de que algunos de los datos parecen nulos, pero se intenta utilizar como un objeto poblado. Hay bastantes tipos de problemas que pueden causar este comportamiento, y cada uno debe probarse con el código donde surgió el error.

Oyente de eventos olvidado #

addEventListener(Event.ENTER_FRAME,moveChild);
function moveChild(e:Event):void {
    childMC.x++;
    if (childMC.x>1000) {
        gotoAndStop(2);
    }
}

Este ejemplo moverá el ‘childMC’ (agregado a ‘Main’ en el momento del diseño) pero arrojará instantáneamente un 1009 tan pronto como se invoque ‘gotoAndStop()’, si ese ‘childMC’ no existe en el marco 2. La razón principal porque cada vez que un cabezal de reproducción pasa un cuadro clave (un cuadro que no hereda el conjunto de objetos del cuadro anterior), ya sea usando gotoAndStop(), gotoAndPlay() con el cuadro de destino separado del cuadro actual por un fotograma clave, o mediante reproducción normal si el SWF es una animación, el contenido del fotograma actual se destruye y el nuevo contenido se crea utilizando los datos almacenados en la GUI. Entonces, si el nuevo marco no tiene un hijo llamado childMC, la solicitud de propiedad devolverá un valor nulo y se generará 1009.

El mismo principio se aplica si agrega dos detectores de eventos, pero elimina solo uno, o agrega un detector a un objeto, pero intenta eliminarlo de otro. La llamada removeEventListener no le avisará si el objeto no tiene un detector de eventos asociado, así que lea con atención el código que agrega y elimina detectores de eventos.

Tenga en cuenta también: el uso de objetos Timer, llamar a setInterval() y setTimeout() también crea detectores de eventos, y estos también deben borrarse correctamente.

El escenario no está disponible #

A veces, los desarrolladores escriben algún código que desea acceder al “escenario”, o al escenario Flash, para agregar oyentes. Puede funcionar por primera vez y luego, de repente, dejar de funcionar y producir el error 1009. El código en cuestión puede incluso estar en la línea de tiempo, ya que es la primera iniciativa para agregar código allí, y muchos tutoriales que aún existen usan capa de código de línea de tiempo para colocar el código.

public class Main extends MovieClip {
    public function Main() {
        stage.addEventListener(Event.ENTER_FRAME,update); // here

La razón por la que este código no funciona es simple: primero se crea una instancia de un objeto de visualización, luego se agrega a la lista de visualización y, mientras está fuera de la lista de visualización, stage es nulo.

Peor si el código es así:

stage.addEventListener(Event.ENTER_FRAME,update); // here

se coloca en la línea de tiempo. Incluso puede funcionar durante algún tiempo, mientras que el objeto ‘Principal’ se coloca en el escenario a través de la GUI. Luego, su SWF se carga desde otro SWF y, de repente, el código se rompe. Esto sucede porque los marcos de Main se construyen de forma diferente cuando el reproductor carga directamente el SWF y cuando la carga se procesa de forma asíncrona. La solución es usar el oyente Event.ADDED_TO_STAGE, y poner todo el código que aborde el escenario en él, y poner el oyente en un archivo AS en lugar de la línea de tiempo.

Encasillado no válido #

function listener(e:Event):void {
    var m:MovieClip=e.target as MovieClip;
    m.x++;
}

Si dicho oyente se adjunta a un objeto que no es un descendiente de MovieClip (por ejemplo, un Sprite), la conversión de tipo fallará y cualquier operación posterior con su resultado generará el error 1009.

Objeto no instanciado #

var a:Object;
trace(a); // null
trace(a.b); // Error 1009

Aquí, se declara una referencia de objeto, pero nunca se le asigna un valor, ya sea con nuevo o la asignación de un valor no nulo. Solicitar sus propiedades o método da como resultado un error 1009.

Expresión de varios niveles #

x=anObject.aProperty.anotherProperty.getSomething().data;

Aquí, cualquier objeto antes del punto puede terminar siendo nulo, y el uso de métodos que devuelven objetos complejos solo aumenta la complicación para depurar el error nulo. En el peor de los casos, si el método es propenso a errores extraños, por ejemplo, al recuperar datos a través de la red.

Resultado de la función sin procesar #

s=this.getChildByName("garbage");
if (s.parent==this) {...}

getChildByName() es una de las muchas funciones que pueden devolver un valor nulo si se produce un error al procesar su entrada. Por lo tanto, si está recibiendo un objeto de cualquier función que posiblemente pueda devolver un valor nulo, verifique primero si hay un valor nulo. Aquí, una propiedad se consulta instantáneamente sin verificar primero si s es nulo, esto generará el error 1009.

Referencia invalidada a un objeto basado en marcos #

A veces, se llama a gotoAndStop() en medio del código que hace referencia a algunas propiedades basadas en marcos. Sin embargo, inmediatamente después de cambiar el marco, todos los enlaces a las propiedades que existían en el marco actual se invalidan, por lo que cualquier procesamiento que los involucre debe finalizar de inmediato.

Hay dos escenarios generales en los que puede ocurrir dicho procesamiento: Primero, un ciclo no termina después de la llamada gotoAndStop(), como aquí:

for each (bullet in bullets) {
    if (player.hitTestObject(bullet)) gotoAndStop("gameOver");
}

Aquí, un error 1009 significa que el MC player se destruyó mientras se procesaba la llamada gotoAndStop(), pero el bucle continúa y remite el enlace ahora nulo para obtener hitTestObject(). Si la condición dijera si (bullet.hitTestObject(jugador)), el error sería #2007 “El parámetro hitTestObject no debe ser nulo”. La solución es colocar una instrucción return justo después de llamar a gotoAndStop().

El segundo caso es múltiples oyentes de eventos en el mismo evento. Como esto:

stage.addEventListener(Event.ENTER_FRAME,func1);
stage.addEventListener(Event.ENTER_FRAME,func2);
function func1(e:Event):void {
    if (condition()) {
        gotoAndStop(2);
    }
}

Aquí, si condition() es verdadero, el primer oyente ejecutaría gotoAndStop(), pero el segundo oyente aún se ejecutaría, y si hace referencia a objetos en el marco, se generará un error 1009. La solución es evitar múltiples oyentes en un solo evento, en un solo objeto, es mejor tener un oyente que maneje todas las situaciones en ese evento y pueda terminar correctamente si se necesita un cambio de marco.
