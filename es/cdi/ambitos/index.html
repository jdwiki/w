** ¿Por qué necesito estos constructores sin argumentos? **

¿Qué sucede si un bean con ámbito de sesión se inyecta en un bean con ámbito de aplicación? ¿Cómo obtiene el bean con ámbito de aplicación la instancia de bean con ámbito de sesión correcta para cada solicitud? ¿No se filtraría el bean con ámbito de sesión en otras solicitudes? ¿Cómo funciona? Para facilitar la determinación del alcance, CDI utiliza lo que se conoce como proxy. Cuando CDI inyecta un bean con ámbito no dependiente en otro objeto, no inyecta el bean directamente. En cambio, subclasifica ese bean para crear lo que se conoce como un proxy. Cada vez que se llama a un método en el proxy, le pide al tiempo de ejecución de CDI que busque el bean correcto para ese ámbito en particular (si tiene un alcance de solicitud, obtenga el bean para esa solicitud. Si tiene un alcance de sesión, obtenga el bean para esa sesión, etc. .), y luego reenvía la llamada al objeto real, devolviendo cualquier resultado para métodos no vacíos. Esto significa que está bien hacer lo siguiente:

@ApplicationScoped
public class ApplicationScopedClass {

    private final RequestScopedClass requestScopedClass;

    @Inject
    public ApplicationScopedClass(RequestScopedClass requestScopedClass) {
        this.requestScopedClass = requestScopedClass;
    }

    public ApplicationScopedClass() {

    }

    public doSomething() {
        requestScopedClass.doSomethingRequestSpecific(); //This works, because of the proxy
    }

}

Sin embargo, para que una clase sea legalmente una subclase de otra clase en Java, debe tener un constructor válido. Y un constructor válido debe llamar a otro constructor en la clase o al constructor de la clase principal (por ejemplo, super()). Para simplificar esta subclasificación, la especificación CDI requiere que cualquier bean con ámbito no dependiente proporcione un constructor público sin argumentos, de modo que el tiempo de ejecución no tenga que intentar adivinar qué hacer con una clase que necesita ser proxy .

El alcance predeterminado #

public class DependentScopedClass {

    //This class has no scoping annotations, so a new instance gets created at every injection point.

    @Inject
    public DependentScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

}

El ámbito predeterminado para la mayoría de los beans CDI se conoce como ámbito dependiente. Una clase que no contenga ninguna anotación de alcance se tratará como un alcance dependiente, a menos que sea un recurso o proveedor JAX-RS (los recursos están predeterminados en el alcance de la solicitud y los proveedores tienen el alcance predeterminado de la aplicación). Una instancia de una clase de alcance dependiente vive tanto como el objeto en el que se inyecta. Cada vez que se construye una clase y tiene una clase de ámbito dependiente como dependencia, la clase de ámbito dependiente se crea y se inyecta directamente en el objeto que la necesita. La importancia de esto se hará evidente después de ejemplos posteriores.

@RequestScoped #

@RequestScoped
public class RequestScopedClass {
    //This class gets constructed once per Servlet request, and is shared among all CDI-managed classes within that request.

    @Inject
    public RequestScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public RequestScopedClass() {
        //Note that it is required that a request scoped class have a public no-args constructor
    }

}

Si un bean se anota con @RequestScoped, se creará una vez para cualquier solicitud. Si dos objetos dependen de una clase de ámbito de solicitud, ambos obtendrán referencias al mismo objeto.

Nota: Cualquier bean con ámbito de solicitud debe tener un constructor público sin argumentos. La razón de esto se explicará más adelante.

@ApplicationScoped #

@ApplicationScoped
public class ApplicationScopedClass {
    //This class gets constructed once for the entire life of the application, and is shared among all CDI-managed classes throughout the life of the application.

    @Inject
    public ApplicationScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public ApplicationScopedClass() {
        //Note that it is required that an application scoped class have a public no-args constructor
    }

}

Las clases con @ApplicationScoped se crean solo una vez y cada objeto que depende de la clase comparte la misma instancia. Estas clases son singletons ’efectivamente’, sin embargo, debe tenerse en cuenta que no hay nada que impida que un codificador cree manualmente instancias adicionales de la clase. Por lo tanto, el uso de @ApplicationScoped es útil para compartir contexto en toda la aplicación, o como una optimización del rendimiento (si la clase es costosa para construir instancias de ella), pero no debe confiarse en ella como una medida de integridad para garantizar solo una instancia de un la clase existe.

Al igual que los beans con ámbito de solicitud, los beans con ámbito de aplicación deben tener un constructor público sin argumentos.

@SessionScoped #

@SessionScoped
public class SessionScopedClass implements Serializable {
    //This class gets constructed once per session, and is shared among all CDI-managed classes within that session. Notice that it implements Serializable, since the instance will get put on the session.

    @Inject
    public SessionScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public SessionScopedClass() {
        //Note that it is required that a session scoped class have a public no-args constructor
    }

}

Las clases anotadas con @SessionScoped se crearán una vez por sesión y dos objetos dentro de la misma sesión compartirán la misma instancia de la clase con ámbito de sesión.

Es importante notar que la clase con ámbito de sesión debe implementar Serializable. Este requisito existe porque el bean se almacenará en la sesión del contenedor de servlet para esa instancia de ámbito de sesión en particular. En general, cualquier objeto que se coloque en la sesión debe ser serializable por dos razones: primero, las sesiones pueden persistir en el disco después de una cierta cantidad de inactividad para ahorrar memoria. Esto se conoce como pasivación de sesión y requiere serialización. La segunda razón es que en los clústeres de alta disponibilidad, la replicación de sesiones se usa a menudo para permitir que cualquier servidor del clúster atienda una solicitud para una sesión determinada. Esto generalmente también requiere serialización.

Al igual que el ámbito de la solicitud y el ámbito de la aplicación, las clases del ámbito de la sesión deben tener un constructor público sin argumentos.
