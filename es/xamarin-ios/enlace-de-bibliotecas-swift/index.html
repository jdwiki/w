Enlace de bibliotecas Swift

  • Vinculación de una biblioteca Swift en Xamarin.iOS

En esta página

  • Vinculación de una biblioteca Swift en Xamarin.iOS

Una guía fácil de seguir que lo guiará a través del proceso de enlace de archivos .framework de Swift para usarlos en un proyecto de Xamarin.

  1. Al construir una biblioteca en Xcode, tiene una opción para incluir las bibliotecas rápidas. ¡No! Se incluirán en su aplicación final como NOMBRE.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib pero deben incluirse como NOMBRE.app/Frameworks/libswift*.dylib

  2. Puede encontrar esta información en otro lugar, pero vale la pena mencionarla: no incluya Bitcode en la biblioteca. A partir de ahora, Xamarin no incluye Bitcode para iOS y Apple requiere que todas las bibliotecas admitan las mismas arquitecturas.

Vinculación de una biblioteca Swift en Xamarin.iOS #

La vinculación de una biblioteca Swift en Xamarin.iOS sigue el mismo proceso para Objective-C que se muestra en https://developer.xamarin.com/guides/ios/advanced_topics/binding_objective-c/, pero con algunas advertencias.

  1. Una clase rápida debe heredar de NSObject para vincularse.
  2. El compilador Swift traducirá los nombres de clases y protocolos a otra cosa a menos que use la anotación @objc (por ejemplo, @objc(MyClass)) en sus clases Swift para especificar el nombre explícito del objetivo c.
  3. En el tiempo de ejecución, su aplicación debe incluir algunas bibliotecas centrales rápidas junto con su marco enlazado en una carpeta llamada Marcos;
  4. Cuando la aplicación se envía a AppStore, debe incluir una carpeta SwiftSupport junto con su carpeta Payload. Esos están dentro del archivo IPA.

Aquí puede encontrar un enlace de muestra simple: https://github.com/Flash3001/Xamarin.BindingSwiftLibrarySample

Y una muestra de vinculación completa: https://github.com/Flash3001/iOSCharts.Xamarin

Encuentre los pasos a continuación:

1.1 Prepara las clases de Swift que quieres exportar #

Para cualquier clase de Swift que desee usar, debe heredar de NSObject y hacer que el nombre de Objective-C sea explícito mediante la anotación objc. De lo contrario, el compilador Swift generará nombres diferentes. A continuación se muestra un código de ejemplo de cómo podría verse una clase de Swift. Tenga en cuenta que no importa qué clase herede, siempre que la clase raíz herede de NSObject.

//Add this to specify explicit objective c name
@objc(MyClass)
open class MyClass: NSObject {
    open func getValue() -> String
    {
        return "Value came from MyClass.swift!";
    }
}

1.2 Construya el marco #

Deshabilitar código de bits. *

Construido para su lanzamiento para Dispositivo y Simulador. *

  • No relacionado solo con el enlace Swift.

2. Crea una biblioteca completa #

Un framework contiene varios archivos, el que necesita comer un poco es NOMBRE.framework/NOMBRE (sin extensión).

  • Copie Release-iphoneos/NOMBRE.framework a NOMBRE.framework
  • Crear la biblioteca FAT usando:
    • lipo -create Release-iphonesimulator/NAME.framework/NAME Versión-iphoneos/NOMBRE.framework/NOMBRE -salida NOMBRE.framework/NOMBRE
  • Copie los archivos en Release-iphonesimulator/NAME.framework/Modules/NAME.swiftmodule a NAME.framework/Modules/NAME.swiftmodule (hasta ahora solo contenía archivos de iphoneos)

3. Importa la biblioteca #

Asumiré que ya creó el proyecto de enlace en Archivo -> Nuevo -> iOS -> Biblioteca de enlaces.

Xamarin admite la importación de .frameworks. Simplemente haga clic derecho en ‘Referencias nativas’ y haga clic en ‘Agregar referencia nativa’. Encuentre el marco gordo recién creado y agréguelo.

4. Cree la ApiDefinition basada en el archivo LIBRARY-Swift.h dentro de los encabezados. #

Puedes hacerlo manualmente, pero no será agradable. Puedes usar Objetivo Sharpie. La herramienta que utiliza Xamarin para vincular sus propias bibliotecas.

Cómo usarlo en https://developer.xamarin.com/guides/cross-platform/macios/binding/objective-sharpie/

El comando básico será algo como: sharpie bind -sdk iphoneos9.3 NOMBRE-Swift.h

Si obtiene una System.Reflection.TargetInvocationException, probablemente se deba a que tiene instalada una versión diferente del SDK. Ejecute el siguiente comando para verificar con iPhone OS SDK que ha instalado:

sharpie xcode -sdks

El archivo NOMBRE-Swift.h se encuentra en NOMBRE.framework/Headers/NOMBRE-Swift.h

Nota: Las clases rápidas deben heredar de “NSObject”, de lo contrario, NOMBRE-Swift.h no importará sus clases y Objective Sharpie no convertirá nada.

Reemplace el contenido de su proyecto de enlace ApiDefinition.cs con el recién creado.

5. Cambie todo [Protocolo] y [BaseType] para incluir el nombre de la clase en el tiempo de ejecución de Objective-C. #

En caso de que la clase o el protocolo Swift original no incluya la anotación @objc(MyClass) como se especifica en el paso 1.1, se cambiarán sus nombres internos de Objective-C, por lo que debe asignarlo al correcto.

Todos los nombres están disponibles en el archivo NAME-Swift.h en el siguiente formato:

SWIFT_CLASS("_TtC11SwiftSample7MyClass")
@interface MyClass : NSObject

Y

SWIFT_PROTOCOL("_TtP6Charts17ChartDataProvider_")
@protocol ChartDataProvider

Para establecer el nombre, use la propiedad BaseTypeAttribute.Name https://developer.xamarin.com/guides/cross-platform/macios/binding/binding-types-reference/#BaseType.Name para clases y ProcotolAttribute.Name https:// developer.xamarin.com/api/property/MonoTouch.Foundation.ProtocolAttribute.Name/ para protocolos.

[BaseType(typeof(NSObject), Name = "_TtC11SwiftSample7MyClass")]
interface MyClass

Hacerlo manualmente no es genial. Puede usar esta herramienta https://github.com/Flash3001/SwiftClassify para insertar todos los nombres. (Es un trabajo en progreso. Pero es bastante simple, con solo mirar el código sabrás cómo funciona).

6.1 Incluya todas las dependencias de Swift para ejecutar. #

Si intenta consumir la biblioteca en una aplicación e intenta ejecutarla ahora mismo, se bloqueará. El error se debe a la falta de libswiftCore.dylib

Algo como esto:

Dyld Error Message:
  Library not loaded: @rpath/libswiftCore.dylib
  Referenced from: /Users/USER/Library/Developer/CoreSimulator/Devices/AC440891-C819-4050-8CAB-CE15AB4B3830/data/Containers/Bundle/Application/27D2EC87-5042-4FA7-9B80-A24A8971FB48/SampleUsing.app/Frameworks/SwiftSample.framework/SwiftSample
  Reason: image not found

Xamarin.iOS no brinda soporte oficial para vincular una biblioteca Swift. Por lo tanto, debe incluir manualmente las bibliotecas principales de Swift en las carpetas Frameworks y SwiftSupport. Los archivos de la carpeta Frameworks son diferentes para Simulator y Device. Se pueden encontrar en /Applications/Xcode.app/Contents/Developer//XcodeDefault.xctoolchain/usr/lib/swift.Toolchains

En lugar de copiar manualmente los archivos dentro de la carpeta Framework, puede usar esta biblioteca https://github.com/Flash3001/Xamarin.Swift3.Support. Incluye todas las dependencias que necesita Swift 3.1, cada una en un solo paquete NuGet.

Como puede ver, el paquete Nuget está incluido en la aplicación del consumidor, no en el enlace en sí. Si intenta incluirlo en el enlace, obtendrá errores de compilación.

Si está creando un paquete Nuget, puede indicarle a Nuget que lo incluya como una dependencia.

6.2. Averiguar qué dependencias de Swift incluir. #

Una cosa importante que debe hacer es averiguar cada paquete que necesita incluir en su proyecto. Una encuadernación simple generalmente necesitará:

     libswiftCore.dylib
     libswiftCoreGraphics.dylib
     libswiftCoreImage.dylib
     libswiftDarwin.dylib
     libswiftDispatch.dylib
     libswiftFoundation.dylib
     libswiftObjectiveC.dylib
     libswiftQuartzCore.dylib
     libswiftUIKit.dylib

Para enumerar cada dependencia, puede ejecutar el siguiente comando dentro de su LibraryName.framework

otool -l -arch armv7 LibraryName | grep libswift

No incluya todos los paquetes disponibles en NuGet para Swift3, ya que podrían aumentar el tamaño de su aplicación.

7. Incluya SwiftSupport para enviar la aplicación a AppStore. #

Apple requiere que su aplicación se envíe con una carpeta SwiftSupport junto con su carpeta Payload. Ambos están dentro de su paquete IPA.

Puede usar este script https://github.com/bq/ipa-packager para hacer este trabajo por usted.

Este proceso es el único que el consumidor de la biblioteca tendrá que hacer manualmente. Cada vez que intenta empujar la aplicación a AppStore.

Haga clic en ‘Firmar y distribuir’ y Guardar en disco

Descomprime tu .IPA

Cree el nuevo IPA usando el script antes mencionado

Si descomprime el archivo, contendrá la carpeta SwiftSupport.

Observaciones #

Al construir una biblioteca en Xcode, tiene una opción para incluir las bibliotecas rápidas. ¡No! Se incluirán en su aplicación final como NOMBRE.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib pero deben incluirse como NOMBRE.app/Frameworks/libswift*.dylib

Puede encontrar esta información en otros lugares, pero vale la pena mencionarla: no incluya Bitcode en la biblioteca. A partir de ahora, Xamarin no incluye Bitcode para iOS y Apple requiere que todas las bibliotecas admitan las mismas arquitecturas.
