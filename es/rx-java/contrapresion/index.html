Introducción #

Contrapresión es cuando en una canalización de procesamiento ‘Observable’, algunas etapas asincrónicas no pueden procesar los valores lo suficientemente rápido y necesitan una forma de decirle al productor aguas arriba que reduzca la velocidad.

El caso clásico de necesidad de contrapresión es cuando el productor es una fuente caliente:

PublishSubject<Integer> source = PublishSubject.create();

source
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Thread.sleep(10_000);

En este ejemplo, el subproceso principal producirá 1 millón de elementos para un consumidor final que lo está procesando en un subproceso en segundo plano. Es probable que el método compute(int) lleve algo de tiempo, pero la sobrecarga de la cadena de operadores Observable también puede aumentar el tiempo que lleva procesar los elementos. Sin embargo, el subproceso de producción con el bucle for no puede saber esto y sigue onNexting.

Internamente, los operadores asíncronos tienen búferes para contener dichos elementos hasta que puedan ser procesados. En el Rx.NET clásico y en los primeros RxJava, estos búferes no tenían límites, lo que significa que probablemente contendrían casi el millón de elementos del ejemplo. El problema comienza cuando hay, por ejemplo, 1000 millones de elementos o la misma secuencia de 1 millón aparece 1000 veces en un programa, lo que genera OutOfMemoryError y, en general, ralentizaciones debido a una sobrecarga excesiva del GC.

De manera similar a cómo el manejo de errores se convirtió en un ciudadano de primera clase y recibió operadores para manejarlo (a través de los operadores onErrorXXX), la contrapresión es otra propiedad de los flujos de datos en la que el programador debe pensar y manejar (a través de los operadores onBackpressionXXX).

Más allá del PublishSubject anterior, hay otros operadores que no admiten la contrapresión, principalmente debido a razones funcionales. Por ejemplo, el “intervalo” del operador emite valores periódicamente, contrapresionarlo conduciría a un cambio en el período relativo a un reloj de pared.

En RxJava moderno, la mayoría de los operadores asincrónicos ahora tienen un búfer interno limitado, como observeOn arriba y cualquier intento de desbordar este búfer terminará toda la secuencia con MissingBackpressionException. La documentación de cada operador tiene una descripción sobre su comportamiento de contrapresión.

Sin embargo, la contrapresión está presente de manera más sutil en las secuencias frías regulares (que no producen ni deberían generar la excepción “MissingBackpressionException”). Si se reescribe el primer ejemplo:

Observable.range(1, 1_000_000)
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

Thread.sleep(10_000);

No hay ningún error y todo funciona sin problemas con un uso de memoria pequeño. La razón de esto es que muchos operadores de origen pueden “generar” valores a pedido y, por lo tanto, el operador observeOn puede decirle al rango que genere como máximo tantos valores que el búfer observeOn puede contener a la vez sin desbordarse.

Esta negociación se basa en el concepto informático de co-rutinas (yo te llamo, tú me llamas). El operador range envía una devolución de llamada, en forma de una implementación de la interfaz Producer, a observeOn llamando a setProducer (suscriptor interno). A cambio, observeOn llama a Producer.request(n) con un valor para indicar el rango que puede producir (es decir, onNext it) tantos elementos adicionales. Entonces es responsabilidad de observeOn llamar al método request en el momento correcto y con el valor correcto para mantener el flujo de datos pero sin desbordarse.

Rara vez es necesario expresar la contrapresión en los consumidores finales (porque son síncronos con respecto a su flujo ascendente inmediato y la contrapresión ocurre naturalmente debido al bloqueo de la pila de llamadas), pero puede ser más fácil entender cómo funciona:

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(1);
    }

    public void onNext(Integer v) {
        compute(v);

        request(1);
    }

    @Override
    public void onError(Throwable ex) {
        ex.printStackTrace();
    }

    @Override
    public void onCompleted() {
        System.out.println("Done!");
    }
});

Aquí la implementación de onStart indica rango para producir su primer valor, que luego se recibe en onNext. Una vez que finaliza compute(int), se solicita otro valor de range. En una implementación ingenua de range, dicha llamada llamaría recursivamente a onNext, lo que llevaría a StackOverflowError que, por supuesto, no es deseable.

Para evitar esto, los operadores utilizan la llamada lógica de trampolín que evita tales llamadas reentrantes. En términos de range, recordará que hubo una llamada request(1) mientras llamaba onNext() y una vez que onNext() regrese, hará otra ronda y llamará onNext() con el siguiente valor entero. Por lo tanto, si se intercambian los dos, el ejemplo sigue funcionando igual:

@Override
public void onNext(Integer v) {
    request(1);

    compute(v);
}

Sin embargo, esto no es cierto para onStart. Aunque la infraestructura ‘Observable’ garantiza que se llamará como máximo una vez en cada ‘Suscriptor’, la llamada a ‘request(1)’ puede desencadenar la emisión de un elemento de inmediato. Si uno tiene una lógica de inicialización después de la llamada a request(1) que necesita onNext, puede terminar con excepciones:

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {

    String name;

    @Override
    public void onStart() {
        request(1);

        name = "RangeExample";
    }

    @Override
    public void onNext(Integer v) {
        compute(name.length + v);

        request(1);
    }

    // ... rest is the same
});

En este caso síncrono, se lanzará una NullPointerException inmediatamente mientras aún se ejecuta onStart. Ocurre un error más sutil si la llamada a request(1) desencadena una llamada asincrónica a onNext en algún otro subproceso y la lectura de name en onNext se apresura a escribirlo en onStart después de request.

Por lo tanto, uno debe hacer todas las inicializaciones de campo en onStart o incluso antes de eso y llamar a request() al final. Las implementaciones de request() en los operadores garantizan una relación adecuada antes de que suceda (o en otros términos, liberación de memoria o cerca completa) cuando sea necesario.

Los operadores onBackpressionXXX #

La mayoría de los desarrolladores encuentran contrapresión cuando su aplicación falla con MissingBackpressionException y la excepción generalmente apunta al operador observeOn. La causa real suele ser el uso sin contrapresión de PublishSubject, timer() o interval() u operadores personalizados creados a través de create().

Hay varias maneras de hacer frente a tales situaciones.

Aumentar el tamaño del búfer #

A veces, tales desbordamientos ocurren debido a fuentes de ráfagas. De repente, el usuario toca la pantalla demasiado rápido y el búfer interno predeterminado de 16 elementos de observeOn en Android se desborda.

La mayoría de los operadores sensibles a la contrapresión en las versiones recientes de RxJava ahora permiten a los programadores especificar el tamaño de sus búferes internos. Los parámetros relevantes suelen llamarse bufferSize, prefetch o capacityHint. Dado el ejemplo desbordante de la introducción, podemos simplemente aumentar el tamaño del búfer de observeOn para tener suficiente espacio para todos los valores.

PublishSubject<Integer> source = PublishSubject.create();

source.observeOn(Schedulers.computation(), 1024 * 1024)
      .subscribe(e -> { }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Sin embargo, tenga en cuenta que, en general, esto puede ser solo una solución temporal, ya que el desbordamiento aún puede ocurrir si la fuente produce en exceso el tamaño de búfer previsto. En este caso, se puede utilizar uno de los siguientes operadores.

Dosificación/salto de valores con operadores estándar #

En caso de que los datos de origen se puedan procesar de forma más eficiente por lotes, se puede reducir la probabilidad de que se produzca una “Excepción de contrapresión perdida” mediante el uso de uno de los operadores de procesamiento por lotes estándar (por tamaño y/o por tiempo).

PublishSubject<Integer> source = PublishSubject.create();

source
      .buffer(1024)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(list -> { 
          list.parallelStream().map(e -> e * e).first();
      }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Si algunos de los valores se pueden ignorar de forma segura, se pueden usar los operadores de muestreo (con tiempo u otro Observable) y de limitación (throttleFirst, throttleLast, throttleWithTimeout).

PublishSubject<Integer> source = PublishSubject.create();

source
      .sample(1, TimeUnit.MILLISECONDS)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Tenga en cuenta, sin embargo, que estos operadores solo reducen la tasa de recepción de valor por parte del flujo descendente y, por lo tanto, aún pueden conducir a una excepción de “MissingBackpressionException”.

en el búfer de contrapresión() #

Este operador en su forma sin parámetros reintroduce un búfer ilimitado entre la fuente ascendente y el operador descendente. Ser ilimitado significa que siempre que la JVM no se quede sin memoria, puede manejar casi cualquier cantidad proveniente de una fuente de ráfagas.

 Observable.range(1, 1_000_000)
           .onBackpressureBuffer()
           .observeOn(Schedulers.computation(), 8)
           .subscribe(e -> { }, Throwable::printStackTrace);

En este ejemplo, observeOn va con un tamaño de búfer muy bajo, pero no hay MissingBackpressionException ya que onBackpressionBuffer absorbe todo el millón de valores y entrega pequeños lotes a observeOn.

Tenga en cuenta, sin embargo, que onBackpressionBuffer consume su fuente de manera ilimitada, es decir, sin aplicarle ninguna contrapresión. Esto tiene la consecuencia de que incluso una fuente de soporte de contrapresión como “rango” se realizará por completo.

Hay 4 sobrecargas adicionales de onBackpressionBuffer

en el búfer de contrapresión (capacidad int) #

Esta es una versión limitada que señala BufferOverflowError en caso de que su búfer alcance la capacidad dada.

Observable.range(1, 1_000_000)
          .onBackpressureBuffer(16)
          .observeOn(Schedulers.computation())
          .subscribe(e -> { }, Throwable::printStackTrace);

La relevancia de este operador está disminuyendo a medida que más y más operadores ahora permiten establecer sus tamaños de búfer. Por lo demás, esto brinda la oportunidad de “ampliar su búfer interno” al tener un número mayor con onBackpressionBuffer que su valor predeterminado.

onBackpressionBuffer(int capacidad, Action0 onOverflow) #

Esta sobrecarga llama a una acción (compartida) en caso de que ocurra un desbordamiento. Su utilidad es bastante limitada ya que no se proporciona otra información sobre el desbordamiento que la pila de llamadas actual.

onBackpressionBuffer(int capacidad, Action0 onOverflow, BackpressionOverflow.Estrategia estrategia) #

Esta sobrecarga es en realidad más útil ya que permite definir qué hacer en caso de que se haya alcanzado la capacidad. La BackpressionOverflow.Strategy es una interfaz en realidad, pero la clase BackpressionOverflow ofrece 4 campos estáticos con implementaciones que representan acciones típicas:

  • ON_OVERFLOW_ERROR: este es el comportamiento predeterminado de las dos sobrecargas anteriores, lo que indica una BufferOverflowException

  • ON_OVERFLOW_DEFAULT: actualmente es lo mismo que ON_OVERFLOW_ERROR

  • ON_OVERFLOW_DROP_LATEST: si ocurriera un desbordamiento, el valor actual simplemente se ignorará y solo se entregarán los valores anteriores una vez que lo solicite el flujo descendente.

  • ON_OVERFLOW_DROP_OLDEST: deja caer el elemento más antiguo en el búfer y le agrega el valor actual.

    Observable.range(1, 1_000_000) .onBackpressureBuffer(16, () -> { }, BufferOverflowStrategy.ON_OVERFLOW_DROP_OLDEST) .observeOn(Schedulers.computation()) .subscribe(e -> { }, Throwable::printStackTrace);

Tenga en cuenta que las dos últimas estrategias causan discontinuidad en la secuencia ya que eliminan elementos. Además, no señalarán BufferOverflowException.

enCaída de contrapresión() #

Siempre que el flujo descendente no esté listo para recibir valores, este operador eliminará ese elemento de la secuencia. Uno puede pensar en ello como un onBackpressionBuffer de capacidad 0 con la estrategia ON_OVERFLOW_DROP_LATEST.

Este operador es útil cuando uno puede ignorar con seguridad los valores de una fuente (como los movimientos del mouse o las señales de ubicación GPS actuales), ya que habrá más valores actualizados más adelante.

 component.mouseMoves()
 .onBackpressureDrop()
 .observeOn(Schedulers.computation(), 1)
 .subscribe(event -> compute(event.x, event.y));

Puede ser útil junto con el operador de origen interval(). Por ejemplo, si se desea realizar alguna tarea periódica en segundo plano pero cada iteración puede durar más que el período, es seguro descartar la notificación de exceso de intervalo, ya que habrá más más adelante:

 Observable.interval(1, TimeUnit.MINUTES)
 .onBackpressureDrop()
 .observeOn(Schedulers.io())
 .doOnNext(e -> networkCall.doStuff())
 .subscribe(v -> { }, Throwable::printStackTrace);

Existe una sobrecarga de este operador: onBackpressionDrop(Action1<? super T> onDrop) donde se llama a la acción (compartida) con el valor que se descarta. Esta variante permite limpiar los propios valores (por ejemplo, liberar recursos asociados).

onBackpressionLatest() #

El operador final mantiene solo el valor más reciente y prácticamente sobrescribe los valores más antiguos no entregados. Uno puede pensar en esto como una variante de onBackpressionBuffer con una capacidad de 1 y una estrategia de ON_OVERFLOW_DROP_OLDEST.

A diferencia de onBackpressionDrop, siempre hay un valor disponible para el consumo si el flujo descendente se está retrasando. Esto puede ser útil en algunas situaciones similares a la telemetría donde los datos pueden venir en un patrón de ráfagas, pero solo los últimos son interesantes para el procesamiento.

Por ejemplo, si el usuario hace mucho clic en la pantalla, aún querríamos reaccionar a su última entrada.

component.mouseClicks()
.onBackpressureLatest()
.observeOn(Schedulers.computation())
.subscribe(event -> compute(event.x, event.y), Throwable::printStackTrace);

El uso de onBackpressionDrop en este caso conduciría a una situación en la que el último clic se perdería y dejaría al usuario preguntándose por qué no se ejecutó la lógica empresarial.

Creación de fuentes de datos con contrapresión #

La creación de fuentes de datos con contrapresión es la tarea relativamente más fácil cuando se trata de contrapresión en general porque la biblioteca ya ofrece métodos estáticos en Observable que manejan la contrapresión para el desarrollador. Podemos distinguir dos tipos de métodos de fábrica: “generadores” fríos que regresan y generan elementos basados ​​en la demanda aguas abajo y “empujadores” calientes que generalmente conectan fuentes de datos no reactivas y/o no contrapresurables y superponen un manejo de contrapresión encima de a ellos.

sólo #

La fuente consciente de la contrapresión más básica se crea a través de simplemente:

Observable.just(1).subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(0);
    }

    @Override
    public void onNext(Integer v) {
        System.out.println(v);
    }
   
    // the rest is omitted for brevity
}

Dado que no solicitamos explícitamente en onStart, esto no imprimirá nada. simplemente es excelente cuando hay un valor constante que nos gustaría impulsar una secuencia.

Desafortunadamente, simplemente a menudo se confunde con una forma de calcular algo dinámicamente para ser consumido por Suscriptors:

int counter;

int computeValue() {
   return ++counter;
}

Observable<Integer> o = Observable.just(computeValue());

o.subscribe(System.out:println);
o.subscribe(System.out:println);

Sorprendentemente para algunos, esto imprime 1 dos veces en lugar de imprimir 1 y 2 respectivamente. Si la llamada se reescribe, se vuelve obvio por qué funciona así:

int temp = computeValue();

Observable<Integer> o = Observable.just(temp);

El computeValue se llama como parte de la rutina principal y no en respuesta a la suscripción de los suscriptores.

fromCalable #

Lo que la gente realmente necesita es el método fromCallable:

Observable<Integer> o = Observable.fromCallable(() -> computeValue());

Aquí, computeValue se ejecuta solo cuando un suscriptor se suscribe y para cada uno de ellos, se imprimen los esperados 1 y 2. Naturalmente, fromCallable también admite correctamente la contrapresión y no emitirá el valor calculado a menos que se solicite. Tenga en cuenta, sin embargo, que el cálculo ocurre de todos modos. En caso de que el cálculo en sí deba retrasarse hasta que el flujo descendente realmente lo solicite, podemos usar solo con mapa:

Observable.just("This doesn't matter").map(ignored -> computeValue())...

just no emitirá su valor constante hasta que se solicite cuando se asigne al resultado de computeValue, aún llamado para cada suscriptor individualmente.

de #

Si los datos ya están disponibles como una matriz de objetos, una lista de objetos o cualquier fuente ‘Iterable’, las respectivas sobrecargas ‘desde’ manejarán la contrapresión y la emisión de dichas fuentes:

 Observable.from(Arrays.asList(1, 2, 3, 4, 5)).subscribe(System.out::println);

Por comodidad (y para evitar advertencias sobre la creación de matrices genéricas), hay de 2 a 10 sobrecargas de argumentos para ‘solo’ que internamente se delegan en ‘desde’.

from(Iterable) también brinda una oportunidad interesante. Mucha generación de valor se puede expresar en forma de una máquina de estado. Cada elemento solicitado desencadena una transición de estado y el cálculo del valor devuelto.

Escribir máquinas de estado como Iterables es algo complicado (pero aún más fácil que escribir un Observable para consumirlo) y, a diferencia de C#, Java no tiene ningún soporte del compilador para construir tales máquinas de estado simplemente escribiendo de forma clásica. código (con retorno de rendimiento y pausa de rendimiento). Algunas bibliotecas ofrecen ayuda, como AbstractIterable de Google Guava y Ix.generate() y Ix.forloop() de IxJava. Estos son por sí mismos dignos de una serie completa, así que veamos una fuente Iterable muy básica que repite un valor constante indefinidamente:

Iterable<Integer> iterable = () -> new Iterator<Integer>() {
    @Override
    public boolean hasNext() {
        return true;
    }

    @Override
    public Integer next() {
        return 1;
    }
};

Observable.from(iterable).take(5).subscribe(System.out::println);

Si consumiéramos el iterador a través del ciclo for clásico, eso daría como resultado un ciclo infinito. Dado que construimos un ‘Observable’ a partir de él, podemos expresar nuestra voluntad de consumir solo los primeros 5 y luego dejar de solicitar cualquier cosa. Este es el verdadero poder de evaluar y calcular perezosamente dentro de Observables.

crear (SyncOnSubscribe) #

A veces, la fuente de datos que se va a convertir en el mundo reactivo en sí mismo es síncrona (bloqueo) y tipo pull, es decir, tenemos que llamar a algún método get o read para obtener la siguiente pieza de datos. Uno podría, por supuesto, convertir eso en un ‘Iterable’ pero cuando tales fuentes están asociadas con recursos, podemos filtrar esos recursos si el downstream cancela la suscripción de la secuencia antes de que finalice.

Para manejar tales casos, RxJava tiene la clase SyncOnSubscribe. Uno puede extenderlo e implementar sus métodos o usar uno de sus métodos de fábrica basados ​​en lambda para construir una instancia.

SyncOnSubscribe<Integer, InputStream> binaryReader = SyncOnSubscribe.createStateful(
     () -> new FileInputStream("data.bin"),
     (inputstream, output) -> {
         try {
             int byte = inputstream.read();
             if (byte < 0) {
                 output.onCompleted();
             } else {
                 output.onNext(byte);
             }
         } catch (IOException ex) {
             output.onError(ex);
         }
         return inputstream;
     },
     inputstream -> {
         try {
             inputstream.close();
         } catch (IOException ex) {
             RxJavaHooks.onError(ex);
         }
     } 
 );

 Observable<Integer> o = Observable.create(binaryReader);

Generalmente, SyncOnSubscribe usa 3 devoluciones de llamada.

Las primeras devoluciones de llamada permiten crear un estado por suscriptor, como FileInputStream en el ejemplo; el archivo se abrirá de forma independiente para cada suscriptor individual.

La segunda devolución de llamada toma este objeto de estado y proporciona un ‘Observer’ de salida cuyos métodos ‘onXXX’ pueden llamarse para emitir valores. Esta devolución de llamada se ejecuta tantas veces como se solicite en sentido descendente. En cada invocación, tiene que llamar a onNext como máximo una vez seguido opcionalmente por onError o onCompleted. En el ejemplo, llamamos a onCompleted() si el byte de lectura es negativo, lo que indica el final del archivo, y llamamos a onError en caso de que la lectura arroje una IOException.

La devolución de llamada final se invoca cuando el downstream cancela la suscripción (cerrando el flujo de entrada) o cuando la devolución de llamada anterior llamó a los métodos de terminal; permite liberar recursos. Dado que no todas las fuentes necesitan todas estas características, los métodos estáticos de SyncOnSubscribe permiten crear instancias sin ellos.

Desafortunadamente, muchas llamadas a métodos a través de JVM y otras bibliotecas lanzan excepciones verificadas y deben incluirse en try-catches ya que las interfaces funcionales utilizadas por esta clase no permiten lanzar excepciones verificadas.

Por supuesto, podemos imitar otras fuentes típicas, como un rango ilimitado con él:

SyncOnSubscribe.createStateful(
     () -> 0,
     (current, output) -> {
         output.onNext(current);
         return current + 1;
     },
     e -> { }
);

En esta configuración, “actual” comienza con “0” y la próxima vez que se invoca la lambda, el parámetro “actual” ahora contiene “1”.

Hay una variante de SyncOnSubscribe llamada AsyncOnSubscribe que se ve bastante similar, con la excepción de que la devolución de llamada del medio también toma un valor largo que representa la cantidad de solicitud de flujo descendente y la devolución de llamada debería generar un Observable con exactamente la misma longitud. Esta fuente luego concatena todos estos ‘Observables’ en una sola secuencia.

 AsyncOnSubscribe.createStateful(
     () -> 0,
     (state, requested, output) -> {
         output.onNext(Observable.range(state, (int)requested));
         return state + 1;
     },
     e -> { }
 );

Hay una discusión en curso (acalorada) sobre la utilidad de esta clase y, en general, no se recomienda porque rutinariamente rompe las expectativas sobre cómo emitirá realmente esos valores generados y cómo responderá, o incluso qué tipo de valores de solicitud recibirá en escenarios de consumo más complejos.

crear (emisor) #

A veces, la fuente que se va a envolver en un ‘Observable’ ya está caliente (como los movimientos del mouse) o fría pero no se puede contrapresionar en su API (como una devolución de llamada de red asíncrona).

Para manejar estos casos, una versión reciente de RxJava introdujo el método de fábrica create(emitter). Toma dos parámetros:

  • una devolución de llamada que se llamará con una instancia de la interfaz Emitter<T> para cada suscriptor entrante,
  • una enumeración Emitter.BackpressionMode que obliga al desarrollador a especificar el comportamiento de la contrapresión que se aplicará. Tiene los modos habituales, similares a onBackpressionXXX además de señalar una MissingBackpressionException o simplemente ignorar dicho desbordamiento por completo.

Tenga en cuenta que actualmente no admite parámetros adicionales para esos modos de contrapresión. Si uno necesita esa personalización, usar NONE como el modo de contrapresión y aplicar onBackpressionXXX relevante en el Observable resultante es el camino a seguir.

El primer caso típico para su uso cuando uno quiere interactuar con una fuente basada en push, como eventos GUI. Esas API presentan alguna forma de llamadas addListener/removeListener que se pueden utilizar:

Observable.create(emitter -> {
    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    emitter.setCancellation(() -> 
        button.removeListener(al));

}, BackpressureMode.BUFFER);

El ‘Emisor’ es relativamente fácil de usar; uno puede llamar a onNext, onError y onCompleted en él y el operador maneja la contrapresión y la gestión de cancelación de suscripción por sí mismo. Además, si la API envuelta admite la cancelación (como la eliminación del oyente en el ejemplo), se puede usar setCancellation (o setSubscription para recursos similares a Subscription) para registrar una devolución de llamada de cancelación que se invoca cuando el cancela la suscripción en sentido descendente o se llama a onError/onCompleted en la instancia Emitter proporcionada.

Estos métodos permiten que solo se asocie un único recurso con el emisor a la vez y establecer uno nuevo cancela la suscripción del antiguo automáticamente. Si uno tiene que manejar múltiples recursos, cree una CompositeSubscription, asóciela con el emisor y luego agregue más recursos a la misma CompositeSubscription:

Observable.create(emitter -> {
    CompositeSubscription cs = new CompositeSubscription();

    Worker worker = Schedulers.computation().createWorker();

    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    cs.add(worker);
    cs.add(Subscriptions.create(() -> 
        button.removeActionListener(al));

    emitter.setSubscription(cs);

}, BackpressureMode.BUFFER);

El segundo escenario generalmente implica alguna API asíncrona basada en devolución de llamada que debe convertirse en un ‘Observable’.

Observable.create(emitter -> {
    
    someAPI.remoteCall(new Callback<Data>() {
        @Override
        public void onSuccess(Data data) {
            emitter.onNext(data);
            emitter.onCompleted();
        }

        @Override
        public void onFailure(Exception error) {
            emitter.onError(error);
        }
    });

}, BackpressureMode.LATEST);

En este caso, la delegación funciona de la misma manera. Desafortunadamente, por lo general, estas API clásicas de estilo de devolución de llamada no admiten la cancelación, pero si lo hacen, uno puede configurar su cancelación como en los ejemplos anteriores (aunque quizás de una manera más complicada). Tenga en cuenta el uso del modo de contrapresión ÚLTIMO; si sabemos que solo habrá un único valor, no necesitamos la estrategia BUFFER ya que asigna un búfer largo predeterminado de 128 elementos (que crece según sea necesario) que nunca se utilizará por completo.
