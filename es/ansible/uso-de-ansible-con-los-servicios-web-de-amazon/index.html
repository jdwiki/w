El archivo ec2.py es un script de Python que ejecuta y devuelve sus recursos de AWS en función de ec2.ini, que es el archivo de configuración que debe personalizar si desea limitar el alcance de su proyecto a algunas regiones en particular. , etiquetas específicas, etc…

Cómo configurar correctamente Ansible para conectarse a Amazon Web Services #

La administración de los recursos de AWS que escalan hacia arriba y hacia abajo llega a los límites del archivo de host de inventario estático, es por eso que necesitamos algo dinámico. Y para eso están los inventarios dinámicos. Empecemos:

Descargue estos archivos ec2.ini y ec2.py a la carpeta de su proyecto:

cd my_ansible_project 
wget https://raw.githubusercontent.com/ansible/ansible/devel/contrib/inventory/ec2.py    
wget https://raw.githubusercontent.com/ansible/ansible/devel/contrib/inventory/ec2.ini

Una vez hecho esto, haga que el archivo ec2.py sea ejecutable:

chmod +x ec2.py

Ahora, exporte su clave de acceso y secreto de AWS como variables de entorno:

export AWS_ACCESS_KEY_ID='ABCDEFGHIJKLM'
export AWS_SECRET_ACCESS_KEY='NOPQRSTUVWXYZ'

Para usar el script ec2.py necesitamos el SDK de Python AWS, boto, por lo que debe instalarlo:

sudo pip install boto

Para probar si todo está bien, intente ejecutar ec2.py enumerando sus recursos:

./ec2.py --list

deberías ver algo similar a:

{
  "_meta": {
    "hostvars": {}
  }
}

Ahora queremos usar el inventario dinámico junto con nuestro archivo de hosts estáticos. Primero, cree una carpeta llamada inventario, agregue ec2.py, ec2.ini y nuestro archivo hosts y luego dígale a Ansible que use esa carpeta como un archivo de inventario:

mkdir inventory 
mv ec2.py inventory/ec2.py
mv ec2.ini inventory/ec2.ini
mv hosts inventory/hosts

A continuación, debemos definir la configuración de nivel de proyecto para Ansible creando un archivo de configuración de Ansible en la carpeta de su proyecto llamado ansible.cfg y agregando esto:

[defaults]
hostfile = inventory
[ssh_connection]
pipelining = False
ssh_args = -o ControlMaster=auto -o ControlPersist=30m -o StrictHostKeyChecking=no

A continuación, debemos configurar Ansible para usar una clave SSH para autenticar el acceso a nuestras instancias EC2. El uso de un agente SSH es la mejor manera de autenticarse con recursos, ya que facilita la administración de claves:

ssh-agent bash 
ssh-add ~/.ssh/keypair.pem

¡Eso es todo! Si siguió esto, puede probarlo usando el módulo ping y luego, verá sus instancias en ejecución que han sido configuradas para usar su clave respondiendo con pong:

ansible -m ping all
11.22.33.44 | success >> {
    "changed": false, 
    "ping": "pong"
}

Cómo iniciar la instancia EC2 desde las AMI oficiales de Amazon, modificarla y almacenarla como una nueva AMI #

Este es un flujo de trabajo muy común cuando se usa Ansible para aprovisionar una instancia de AWS EC2. Esta publicación asume una comprensión básica de Ansible y, lo que es más importante, asume que lo configuró correctamente para conectarse a AWS.

Como la documentación oficial de Ansible insiste, vamos a utilizar cuatro roles:

1- ami_find para obtener el ami id en base al cual lanzaremos nuestra instancia EC2.

2- ec2_ami_creation para lanzar efectivamente la instancia EC2.

3- code_deploy para modificar la instancia; esto podría ser cualquier cosa, así que simplemente transferiremos un archivo a la máquina de destino.

4- build_ami para construir nuestra nueva imagen basada en la instancia ec2 en ejecución. Esta publicación asume que está en el nivel superior de su proyecto Ansible: my_ansible_project

El primer rol: ami_find

cd my_ansible_project/roles && ansible-galaxy init ami_find

En esta función, vamos a utilizar el módulo ec2_ami_find y, como ejemplo, buscaremos una máquina con Ubuntu y obtendremos su ami_id (ami-xxxxxxxx). Ahora edite el archivo my_ansible_project/roles/ami_find/tasks/main.yml:

---
- ec2_ami_find:
    name: "ubuntu/images/hvm-ssd/ubuntu-trusty-14.04-amd64-server-*"
    sort: name
    sort_order: descending
    sort_end: 1
    region: "{{ aws_region }}"
  register: ami_find
- set_fact: ami_ubuntu="{{ ami_find.results[0].ami_id }}"

El segundo rol: ec2_ami_creation

Aquí, usaremos el ami_id que obtuvimos del primer rol y luego lanzaremos nuestra nueva instancia basada en él:

cd my_ansible_project/roles && ansible-galaxy init ec2_ami_creation

En esta función, vamos a utilizar de manera más importante ec2_module para iniciar nuestra instancia. Ahora edite el archivo my_ansible_project/roles/ec2_ami_creation/tasks/main.yml:

---
- ec2_vpc_subnet_facts:
    region: "{{aws_region}}"
  register: vpc
- name: creation of security group of the ec2 instance
  ec2_group:
    name: example
    description: an example EC2 group
    region: "{{ aws_region }}"
    rules:
      - proto: tcp
        from_port: 22
        to_port: 22
        cidr_ip: 0.0.0.0/0
    state: present
  register: ec2_sg

- name: create instance using Ansible
  ec2:
    key_name: "{{ ansible_key }}"
    group: example
    vpc_subnet_id: "{{vpc.subnets[0].id}}"
    instance_type: "{{ instance_type }}"
    ec2_region: "{{ aws_region }}"
    image: "{{ base_image }}"
    assign_public_ip: yes
    wait: yes
  register: ec2

- set_fact: id={{ec2.instances[0].id}}

- name: adding the newly created instance to a temporary group in order to access it later from another play
  add_host: name={{ item.public_ip }} groups=just_created
  with_items: ec2.instances

- name: Wait for SSH to come up
  wait_for: host={{ item.public_dns_name }} port=22 delay=10 timeout=640 state=started
  with_items: ec2.instances

El tercer rol: code_deploy

Aquí, aprovisionaremos esta instancia, que se agregó a un grupo llamado recién_creado

cd my_ansible_project/roles && ansible-galaxy init code_deploy

En este rol, vamos a usar template_module para transferir un archivo y escribir el nombre de host de la máquina en él. Ahora edite el archivo my_ansible_project/roles/code_deploy/tasks/main.yml:

---
- template: src=my_file.txt.j2 dest=/etc/my_file.txt

luego muévase a la carpeta de plantillas dentro de su función:

cd my_ansible_project/roles/templates y agregue un archivo llamado my_file.txt.j2 que contiene:

my name is {{ ansible_hostname }}`

El cuarto rol: build_ami

Ahora crearemos una imagen de la instancia en ejecución usando el módulo ec2_ami. Vaya a la carpeta de su proyecto y:

 cd my_ansible_project/roles && ansible-galaxy init build_ami

Ahora edite el archivo my_ansible_project/roles/build_ami/tasks/main.yml:

---
- ec2_ami:
    instance_id: "{{ instance_id }}"
    wait: yes
    name: Base_Image

Ahora, creo que te has estado preguntando cómo orquestar todos estos roles. ¿Tengo razón? Si es así, continúa leyendo.

Escribiremos un libro de jugadas, compuesto por tres jugadas: la primera jugada aplicable en localhost llamará a nuestros dos primeros roles, la segunda jugada aplicable en nuestro grupo recién_creado. el último rol será aplicable en localhost. ¿Por qué localhost? Cuando queremos administrar algunos recursos de AWS, usamos nuestra máquina local, así de simple. A continuación, utilizaremos un archivo vars en el que pondremos nuestras variables: ansible_key, aws_region, etc…

cree una carpeta de infraestructura en la parte superior de su proyecto y agregue un archivo dentro llamado aws.yml:

---
aws_region: ap-southeast-2
ansible_key: ansible
instance_type: t2.small

Entonces, en la parte superior de su proyecto, cree build_base_image.yml y agregue esto:

---
   - hosts: localhost
     connection: local
     gather_facts: False
     vars_files:
       - infrastructure/aws.yml
     roles:
       - ami_find
       - { role: ec2_creation, base_image: "{{ ami_ubuntu }}"}

   - hosts: just_created
     connection: ssh
     gather_facts: True
     become: yes
     become_method: sudo
     roles:
       - code_deploy

   - hosts: localhost
     connection: local
     gather_facts: False
     vars_files:
       - infrastructure/aws.yml
     roles:
       - { role: new_image, instance_id: "{{ id }}"}

Eso es todo, no olvides eliminar tus recursos después de probar esto, o por qué no crear un rol para eliminar la instancia en ejecución :-)
