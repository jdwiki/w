Gradle es un sistema de compilación basado en JVM que permite a los desarrolladores escribir scripts de alto nivel que se pueden usar para automatizar el proceso de compilación y producción de aplicaciones. Es un sistema flexible basado en complementos, que le permite automatizar varios aspectos del proceso de construcción; incluyendo compilar y firmar un .jar, descargar y administrar dependencias externas, inyectar campos en AndroidManifest o utilizar versiones específicas de SDK.

Sintaxis #

  • apply plugin: los complementos que deberían usarse normalmente solo 'com.android.application' o 'com.android.library'.

  • android: La configuración principal de tu aplicación

    • compileSdkVersion: The compile SDK version
    • buildToolsVersion: The build tools version
    • defaultConfig: The default settings which can been overwritten by flavors and build types
      • applicationId: The application id you use e.g. in the PlayStore mostly the same as your package name
      • minSdkVersion: The minimal required SDK version
      • targetSdkVersion: The SDK version you compile against (should be always the newst one)
      • versionCode: The internal version number which needs to be bigger on each update
      • versionName: The version number the user can see in the app details page
    • buildTypes: See somewhere else (TODO)

  • dependencias: Las dependencias maven o locales de su aplicación

  • compilar una sola dependencia
  • testCompile: una dependencia para las pruebas unitarias o de integración

Un archivo build.gradle básico #

Este es un ejemplo de un archivo build.gradle predeterminado en un módulo.

apply plugin: 'com.android.application'

android {
    compileSdkVersion 25
    buildToolsVersion '25.0.3'

    signingConfigs {
        applicationName {
            keyAlias 'applicationName'
            keyPassword 'password'
            storeFile file('../key/applicationName.jks')
            storePassword 'keystorePassword'
        }
    }
    defaultConfig {
        applicationId 'com.company.applicationName'
        minSdkVersion 14
        targetSdkVersion 25
        versionCode 1
        versionName '1.0'
        signingConfig signingConfigs.applicationName
    }
    buildTypes {
        release {
            minifyEnabled true
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
        }
    }
}

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])

    compile 'com.android.support:appcompat-v7:25.3.1'
    compile 'com.android.support:design:25.3.1'

    testCompile 'junit:junit:4.12'
}

DSL (lenguaje específico del dominio) #

Cada bloque en el archivo de arriba se llama DSL (lenguaje específico del dominio).

Complementos #

La primera línea, apply plugin: 'com.android.application', aplica el complemento de Android para Gradle a la compilación y hace que el bloque android {} esté disponible para declarar las opciones de compilación específicas de Android.

Para una aplicación de Android:

apply plugin: 'com.android.application'

Para una Biblioteca de Android:

apply plugin: 'com.android.library'

Comprender los DSL en el ejemplo anterior #

La segunda parte, el bloque android {...}, es el DSL de Android que contiene información sobre su proyecto.

Por ejemplo, puede configurar compileSdkVersion que especifica el nivel de la API de Android, que Gradle debe usar para compilar su aplicación. El subbloque defaultConfig contiene los valores predeterminados para su manifiesto. Puede ‘anularlos’ con [Sabores de productos][3].

[Dependencias][8] #

El bloque dependencias se define fuera del bloque android {...}: Esto significa que no está definido por el complemento de Android, pero es Gradle estándar. El bloque dependencias especifica qué bibliotecas externas (generalmente bibliotecas de Android, pero las bibliotecas de Java también son válidas) desea incluir en su aplicación. Gradle descargará automáticamente estas dependencias por usted (si no hay una copia local disponible), solo necesita agregar líneas de “compilación” similares cuando desee agregar otra biblioteca.

Veamos una de las líneas presentes aquí:

compilar 'com.android.support:diseño:25.3.1'

Esta línea básicamente dice

agregue una dependencia en la biblioteca de diseño de soporte de Android a mi proyecto.

Gradle se asegurará de que la biblioteca se descargue y esté presente para que pueda usarla en su aplicación, y su código también se incluirá en su aplicación.

Si está familiarizado con Maven, esta sintaxis es GroupId, dos puntos, ArtifactId, otros dos puntos, luego la versión de la dependencia que desea incluir, lo que le brinda control total sobre el control de versiones.

Si bien es posible especificar versiones de artefactos mediante el signo más (+), la mejor práctica es evitar hacerlo; puede generar problemas si la biblioteca se actualiza con cambios importantes sin su conocimiento, lo que probablemente provocaría fallas en su aplicación.

Puede agregar diferentes tipos de dependencias:

  • [dependencias binarias locales][9]
  • [dependencias del módulo][10]
  • [dependencias remotas][11]

Se debe prestar especial atención a las [dependencias planas de aar][12].

Puede encontrar más detalles en [este tema.][13]

Nota sobre -v7 en appcompat-v7

compilar 'com.android.support:appcompat-v7:25.3.1'

Esto simplemente significa que esta biblioteca (appcompat) es compatible con el nivel 7 de la API de Android y en adelante.

Nota sobre junit:junit:4.12

Esta es la dependencia de prueba para las pruebas unitarias.

Especificación de dependencias específicas para diferentes configuraciones de compilación #

Puede especificar que una dependencia solo debe usarse para una determinada [configuración de compilación][14] o puede definir diferentes dependencias para los [tipos de compilación][15] o las [sabores del producto][16] (por ejemplo, depuración, prueba o release) usando debugCompile, testCompile o releaseCompile en lugar del habitual compile.

Esto es útil para mantener las dependencias relacionadas con la prueba y la depuración fuera de la compilación de su versión, lo que mantendrá su ‘APK’ de versión lo más reducido posible y ayudará a garantizar que la información de depuración no se pueda usar para obtener información interna sobre su aplicación.

Definir y usar campos de configuración de compilación #

ConstruirConfigField #

Gradle permite que las líneas buildConfigField definan constantes. Estas constantes serán accesibles en tiempo de ejecución como campos estáticos de la clase BuildConfig. Esto se puede usar para crear sabores definiendo todos los campos dentro del bloque defaultConfig, y luego anulándolos para los tipos de compilación individuales según sea necesario.

Este ejemplo define la fecha de compilación y marca la compilación para producción en lugar de prueba:

android {
    ...
    defaultConfig {
        ...
        // defining the build date
        buildConfigField "long", "BUILD_DATE", System.currentTimeMillis() + "L"
        // define whether this build is a production build
        buildConfigField "boolean", "IS_PRODUCTION", "false"
        // note that to define a string you need to escape it
        buildConfigField "String", "API_KEY", "\"my_api_key\""
    }

    productFlavors {
        prod {
            // override the productive flag for the flavor "prod"
            buildConfigField "boolean", "IS_PRODUCTION", "true"
            resValue 'string', 'app_name', 'My App Name'
        }
        dev {
            // inherit default fields
            resValue 'string', 'app_name', 'My App Name - Dev'
        }
    }
}

El <package_name>.BuildConfig.java generado automáticamente en la carpeta gen contiene los siguientes campos según la directiva anterior:

public class BuildConfig {
    // ... other generated fields ...
    public static final long BUILD_DATE = 1469504547000L;
    public static final boolean IS_PRODUCTION = false;
    public static final String API_KEY = "my_api_key";
}

Los campos definidos ahora se pueden usar dentro de la aplicación en tiempo de ejecución accediendo a la clase BuildConfig generada:

public void example() {
    // format the build date
    SimpleDateFormat dateFormat = new SimpleDateFormat("yyyy/MM/dd");
    String buildDate = dateFormat.format(new Date(BuildConfig.BUILD_DATE));
    Log.d("build date", buildDate);
    
    // do something depending whether this is a productive build
    if (BuildConfig.IS_PRODUCTION) {
        connectToProductionApiEndpoint();
    } else {
        connectToStagingApiEndpoint();
    }
}

Valor res #

El resValue en productFlavors crea un valor de recurso. Puede ser cualquier tipo de recurso (string, dimen, color, etc.). Esto es similar a definir un recurso en el archivo apropiado: p. cadena de definición en un archivo strings.xml. La ventaja es que el definido en gradle se puede modificar en función de su productFlavor/buildVariant. Para acceder al valor, escriba el mismo código que si estuviera accediendo a una resolución desde el archivo de recursos:

getResources().getString(R.string.app_name)

Lo importante es que los recursos definidos de esta manera no pueden modificar los recursos existentes definidos en los archivos. Solo pueden crear nuevos valores de recursos.

Algunas bibliotecas (como la API de Android de Google Maps) requieren una clave de API proporcionada en el Manifiesto como una etiqueta de “metadatos”. Si se necesitan claves diferentes para la depuración y las compilaciones de producción, especifique un marcador de posición de manifiesto completado por Gradle.

En su archivo AndroidManifest.xml:

<meta-data
    android:name="com.google.android.geo.API_KEY"
    android:value="${MAPS_API_KEY}"/>

Y luego configure el campo en consecuencia en su archivo build.gradle:

android {
    defaultConfig {
        ...
        // Your development key
        manifestPlaceholders = [ MAPS_API_KEY: "AIza..." ]
    }

    productFlavors {
        prod {
            // Your production key
            manifestPlaceholders = [ MAPS_API_KEY: "AIza..." ]
        }
    }
}

El sistema de compilación de Android genera varios campos automáticamente y los coloca en BuildConfig.java. Estos campos son:

Campo Descripción
DEPURACIÓN un ‘booleano’ que indica si la aplicación está en modo de depuración o lanzamiento
ID_APLICACIÓN un String que contiene el ID de la aplicación (por ejemplo, com.example.app)
TIPO_DE_CONSTRUCCIÓN un String que contiene el tipo de compilación de la aplicación (normalmente debug o release)
SABOR un String que contiene el sabor particular de la compilación
VERSION_CODE un int que contiene el número de versión (compilación).
Es lo mismo que versionCode en build.gradle o versionCode en AndroidManifest.xml
VERSION_NAME una String que contiene el nombre de la versión (compilación).
Es lo mismo que versionName en build.gradle o versionName en AndroidManifest.xml

Además de lo anterior, si ha definido varias dimensiones de sabor, cada dimensión tendrá su propio valor. Por ejemplo, si tuviera dos dimensiones de sabor para color y size, también tendrá las siguientes variables:

Campo Descripción
SABOR_color una ‘Cadena’ que contiene el valor para el sabor ‘color’.
SABOR_tamaño una ‘Cadena’ que contiene el valor para el tipo de ’tamaño’.

Centralización de dependencias a través del archivo “dependencies.gradle” #

Cuando se trabaja con proyectos de varios módulos, es útil centralizar las dependencias en una sola ubicación en lugar de distribuirlas entre muchos archivos de compilación, especialmente para las bibliotecas comunes, como las bibliotecas de soporte de Android y las bibliotecas de Firebase.

Una forma recomendada es separar los archivos de compilación de Gradle, con un build.gradle por módulo, así como uno en la raíz del proyecto y otro para las dependencias, por ejemplo:

root
  +- gradleScript/
  |     dependencies.gradle
  +- module1/
  |     build.gradle
  +- module2/
  |     build.gradle
  +- build.gradle

Luego, todas sus dependencias se pueden ubicar en gradleScript/dependencies.gradle:

ext {
    // Version
    supportVersion = '24.1.0'

    // Support Libraries dependencies
    supportDependencies = [
            design:            "com.android.support:design:${supportVersion}",
            recyclerView:      "com.android.support:recyclerview-v7:${supportVersion}",
            cardView:          "com.android.support:cardview-v7:${supportVersion}",
            appCompat:         "com.android.support:appcompat-v7:${supportVersion}",
            supportAnnotation: "com.android.support:support-annotations:${supportVersion}",
    ]

    firebaseVersion = '9.2.0';

    firebaseDependencies = [
            core:         "com.google.firebase:firebase-core:${firebaseVersion}",
            database:     "com.google.firebase:firebase-database:${firebaseVersion}",
            storage:      "com.google.firebase:firebase-storage:${firebaseVersion}",
            crash:        "com.google.firebase:firebase-crash:${firebaseVersion}",
            auth:         "com.google.firebase:firebase-auth:${firebaseVersion}",

Firma APK sin exponer la contraseña del almacén de claves #

Puede definir la configuración de firma para firmar el apk en el archivo build.gradle usando estas propiedades:

  • storeFile: el archivo de almacenamiento de claves
  • storePassword: la contraseña del almacén de claves
  • keyAlias: un nombre de alias clave
  • keyPassword: una contraseña de alias clave

En muchos casos, es posible que deba evitar este tipo de información en el archivo build.gradle.

Método A: configurar la firma de versiones mediante un archivo keystore.properties #

Es posible configurar build.gradle de su aplicación para que lea su información de configuración de firma desde un archivo de propiedades como keystore.properties.

Configurar la firma de esta manera es beneficioso porque:

  • Su información de configuración de firma está separada de su archivo build.gradle
  • No tiene que intervenir durante el proceso de firma para proporcionar contraseñas para su archivo de almacén de claves
  • Puede excluir fácilmente el archivo keystore.properties del control de versiones

Primero, cree un archivo llamado keystore.properties en la raíz de su proyecto con contenido como este (reemplazando los valores con los suyos):

storeFile=keystore.jks
storePassword=storePassword
keyAlias=keyAlias
keyPassword=keyPassword

Ahora, en el archivo build.gradle de su aplicación, configure el bloque signingConfigs de la siguiente manera:

android {
...

    signingConfigs {
        release {
            def propsFile = rootProject.file('keystore.properties')
            if (propsFile.exists()) {
                def props = new Properties()
                props.load(new FileInputStream(propsFile))
                storeFile = file(props['storeFile'])
                storePassword = props['storePassword']
                keyAlias = props['keyAlias']
                keyPassword = props['keyPassword']
            }
        }
    }
}

Eso es realmente todo lo que hay que hacer, pero no olvide excluir tanto su archivo de almacén de claves como su archivo keystore.properties del control de versiones.

Un par de cosas a anotar:

  • La ruta storeFile especificada en el archivo keystore.properties debe ser relativa al archivo build.gradle de su aplicación. Este ejemplo asume que el archivo del almacén de claves está en el mismo directorio que el archivo build.gradle de la aplicación.
  • Este ejemplo tiene el archivo keystore.properties en la raíz del proyecto. Si lo coloca en otro lugar, asegúrese de cambiar el valor en rootProject.file('keystore.properties') a la ubicación del suyo, en relación con la raíz de su proyecto.

Método B: mediante el uso de una variable de entorno #

Lo mismo se puede lograr también sin un archivo de propiedades, lo que hace que la contraseña sea más difícil de encontrar:

android {

  signingConfigs {
    release {
        storeFile file('/your/keystore/location/key')
        keyAlias 'your_alias'
        String ps = System.getenv("ps")
        if (ps == null) {
             throw new GradleException('missing ps env variable')
        }
        keyPassword ps
        storePassword ps
    }
}

La variable de entorno "ps" puede ser global, pero un enfoque más seguro puede ser agregarla solo al shell de Android Studio.
En Linux, esto se puede hacer editando Desktop Entry de Android Studio

Exec=sh -c "export ps=myPassword123 ; /path/to/studio.sh"

Puede encontrar más detalles en este tema.

Versionar sus compilaciones a través del archivo “version.properties” #

Puede usar Gradle para incrementar automáticamente la versión de su paquete cada vez que lo crea. Para hacerlo, cree un archivo version.properties en el mismo directorio que su build.gradle con los siguientes contenidos:

VERSION_MAJOR=0
VERSION_MINOR=1
VERSION_BUILD=1

(Cambiando los valores de mayor y menor como mejor le parezca). Luego, en su build.gradle agregue el siguiente código a la sección android:

// Read version information from local file and increment as appropriate
def versionPropsFile = file('version.properties')
if (versionPropsFile.canRead()) {
  def Properties versionProps = new Properties()

  versionProps.load(new FileInputStream(versionPropsFile))

  def versionMajor = versionProps['VERSION_MAJOR'].toInteger()
  def versionMinor = versionProps['VERSION_MINOR'].toInteger()
  def versionBuild = versionProps['VERSION_BUILD'].toInteger() + 1

  // Update the build number in the local file
  versionProps['VERSION_BUILD'] = versionBuild.toString()
  versionProps.store(versionPropsFile.newWriter(), null)

  defaultConfig {
    versionCode versionBuild
    versionName "${versionMajor}.${versionMinor}." + String.format("%05d", versionBuild)
  }
}

Se puede acceder a la información en Java como una cadena BuildConfig.VERSION_NAME para el número {principal}.{menor}.{construcción} completo y como un número entero BuildConfig.VERSION_CODE solo para el número de compilación.

Cambiar el nombre de la aplicación de salida y agregar el nombre de la versión: #

Este es el código para cambiar el nombre del archivo de la aplicación de salida (.apk). El nombre se puede configurar asignando un valor diferente a nuevoNombre

android {

        def newName = "ApkName";

¿Por qué hay dos archivos build.gradle en un proyecto de Android Studio? #

<PROJECT_ROOT>\app\build.gradle es específico para módulo de aplicación.

<PROJECT_ROOT>\build.gradle es un “archivo de compilación de nivel superior” donde puede agregar opciones de configuración comunes a todos los subproyectos/módulos.

Si usa otro módulo en su proyecto, como biblioteca local tendría otro archivo build.gradle: <PROJECT_ROOT>\module\build.gradle

Ejecutar un script de shell desde gradle #

 $pp/my.proto \
 --proto_path=$pp \
 $pp/my_other.proto

Eliminar apk “no alineado” automáticamente #

Si no necesita archivos apk generados automáticamente con el sufijo no alineado (que probablemente no necesite), puede agregar el siguiente código al archivo build.gradle:

// delete unaligned files
android.applicationVariants.all { variant ->
  variant.assemble.doLast {
    variant.outputs.each { output ->
        println "aligned " + output.outputFile
        println "unaligned " + output.packageApplication.outputFile

        File unaligned = output.packageApplication.outputFile;
        File aligned = output.outputFile
        if (!unaligned.getName().equalsIgnoreCase(aligned.getName())) {
            println "deleting " + unaligned.getName()
            unaligned.delete()
        }
    }
  }
}

Depuración de errores de Gradle #

El siguiente es un extracto de [Gradle: ¿Qué es un valor de salida distinto de cero y cómo lo soluciono?](http://stackoverflow.com/questions/36698816/gradle-what-is-a-non-zero- exit-value-and-how-do-i-fix-it), véalo para la discusión completa.

Supongamos que está desarrollando una aplicación y aparece un error de Gradle que, por lo general, se verá así.

:module:someTask FAILED
FAILURE: Build failed with an exception.
* What went wrong:
Execution failed for task ':module:someTask'.
> some message here...  finished with non-zero exit value X
* Try:
Run with --stacktrace option to get the stack trace. Run with --info or --debug option to get more log output.
BUILD FAILED
Total time: Y.ZZ secs

Busca aquí en StackOverflow su problema, y ​​la gente dice que limpie y reconstruya su proyecto, o habilite [MultiDex] 2, y cuando intenta eso, simplemente no está solucionando el problema.

Hay formas de obtener más información, pero la salida de Gradle en sí debería apuntar al error real en las pocas líneas arriba de ese mensaje entre :module:someTask FAILED y el último :module:someOtherTask que pasó . Por lo tanto, si hace una pregunta sobre su error, edite sus preguntas para incluir más contexto al error.

Entonces, obtienes un “valor de salida distinto de cero”. Bueno, ese número es un buen indicador de lo que deberías tratar de arreglar. Aquí hay algunos que ocurren con mayor frecuencia.

  • 1 es solo un código de error general y es probable que el error esté en la salida de Gradle
  • 2 parece estar relacionado con dependencias superpuestas o mala configuración del proyecto.
  • 3 parece ser por incluir demasiadas dependencias o por un problema de memoria.

Las soluciones generales para lo anterior (después de intentar una limpieza y reconstrucción del proyecto) son:

  • 1 - Aborda el error que se menciona. En general, se trata de un error de tiempo de compilación, lo que significa que algún fragmento de código de su proyecto no es válido. Esto incluye XML y Java para un proyecto de Android.
  • 2 & 3 - Muchas respuestas aquí le dicen que habilite multidex. Si bien puede solucionar el problema, lo más probable es que sea una solución alternativa. Si no entiende por qué lo está usando (vea el enlace), probablemente no lo necesite. Las soluciones generales implican reducir el uso excesivo de las dependencias de la biblioteca (como todos los Servicios de Google Play, cuando solo necesita usar una biblioteca, como Mapas o Iniciar sesión, por ejemplo).

Usar gradle.properties para el número de versión central/configuraciones de compilación #

Puede definir la información de configuración central en

  • un archivo separado de gradle incluido https://www.wikiod.com/es/android/gradle-para-android#Centralización de dependencias a través del archivo “dependencies.gradle”
  • un archivo de propiedades independiente https://www.wikiod.com/es/android/gradle-para-android#Versión de sus compilaciones a través del archivo “version.properties”

o hazlo con el archivo root gradle.properties

la estructura del proyecto

root
  +- module1/
  |     build.gradle
  +- module2/
  |     build.gradle
  +- build.gradle
  +- gradle.properties

configuración global para todos los submódulos en gradle.properties

# used for manifest
# todo increment for every release
appVersionCode=19
appVersionName=0.5.2.160726

# android tools settings
appCompileSdkVersion=23
appBuildToolsVersion=23.0.2

uso en un submódulo

apply plugin: 'com.android.application'
android {
    // appXXX are defined in gradle.properties
    compileSdkVersion = Integer.valueOf(appCompileSdkVersion)
    buildToolsVersion = appBuildToolsVersion

    defaultConfig {
        // appXXX are defined in gradle.properties
        versionCode = Long.valueOf(appVersionCode)
        versionName = appVersionName
    }
}

dependencies {
    ...
}

Nota: Si desea publicar su aplicación en la tienda de aplicaciones F-Droid, debe usar números mágicos en el archivo gradle porque, de lo contrario, el robot f-droid no puede leer el número de versión actual para detectar/verificar los cambios de versión.
