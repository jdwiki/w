Las variaciones de Markdown con sintaxis diferente se denominan “sabores”. Los sabores de Markdown se enumeran en https://github.com/jgm/CommonMark/wiki/Markdown-Flavors.

Un sabor de Markdown se puede implementar en varios lenguajes de programación y aplicaciones de software. Las bibliotecas que implementan un analizador Markdown se enumeran en https://github.com/markdown/markdown.github.com/wiki/Implementations y https://www.w3.org/community/markdown/wiki/MarkdownImplementations.

Varios tipos de Markdown y las diferencias entre ellos están documentados en http://flavoredmarkdown.com.

Para probar y comparar el código de Markdown en muchos sabores diferentes y las implementaciones de esos sabores, puede usar la herramienta en línea [Babelmark] (https://babelmark.github.io).

Rebaja de desbordamiento de pila #

Este es el tipo de descuento que usan Stack Overflow y otros sitios de Stack Exchange. Cuando responde una pregunta o agrega documentación, utiliza este descuento. Esta respuesta está hecha de SO markdown

Ver Documentación Oficial

Las cosas principales que agrega SO markdown se encuentran en “Adiciones de Stack Exchange” en esa página web. En particular, SO agrega etiquetas como [tag:tag] y enlaces de acceso directo como [meta] (aunque no en los documentos), spoilers:

! esto es un spoiler

>! This is a spoiler

y devociones personalizadas en idiomas

Reducción de Doxygen #

Doxygen es una herramienta de documentación de código de uso común (para lenguajes que incluyen C++, C# y Java) que también admite el uso de Markdown. Además de la sintaxis estándar de Markdown, hay varios [elementos específicos de Doxygen] (https://www.stack.nl/~dimitri/doxygen/manual/markdown.html#markdown_dox).

Las características principales son el uso de etiquetas @ref para referencias, y los elementos @page, @section/@subsection y @anchor a los que pueden hacer referencia.

@ref <nombre> [“texto”] #

Este elemento crea un enlace (es decir, una referencia) a una sección, subsección, página o ancla con nombre que se ha definido en otra parte de la documentación. (ver referencia de Doxygen)

El primer parámetro (nombre) debe coincidir con el nombre de la sección, subsección, página o ancla que desea vincular.

El segundo parámetro opcional (“texto”`) debe encapsularse entre comillas dobles y definirá cómo aparecerá el enlace de referencia en la página. Si no se usa, el enlace aparecerá como el título usado en la referencia.

@section <nombre-de-la-sección> (título de la sección) #

Este elemento define un nombre de sección. Es visualmente equivalente a un elemento # en Markdown, sin embargo, también definirá una referencia a la que se puede vincular desde otras secciones de su documentación. (ver referencia de Doxygen)

El primer parámetro section-name define el nombre de referencia que puede usar el elemento @ref. Esto no puede contener ningún espacio.

El segundo parámetro título de la sección es una cadena de palabras (que se pueden separar por espacios) que define cómo aparecerá el encabezado de la sección en su página.

Ejemplo #

@section Intro Introduction

This is some text in my introduction.

@section Body Body Paragraph

This is some text in my body, where I refer to the @ref Intro.

Producción #

Introducción #

Este es un texto en mi introducción.

Cuerpo del párrafo #

Este es un texto en mi cuerpo, donde me refiero a la Introducción.

NB: La palabra Introducción anterior aparecerá como un enlace que saltará al encabezado Introducción.

Rebaja con sabor a GitHub #

GitHub Flavored Markdown (a veces abreviado como GFM) hace que sea más fácil trabajar con Markdown en GitHub.com.

Las características clave de GFM incluyen:

  • sangría de código
  • soporte de lista de tareas
  • fácil referencia de problemas de GitHub
  • Detección automática de nombre de usuario y SHA de GitHub
  • detección automática de URL
  • soporte de emojis
