Repo #

Un repositorio git es una estructura de datos en disco que almacena metadatos para un conjunto de archivos y directorios.

Vive en la carpeta .git/ de tu proyecto. Cada vez que envías datos a git, se almacenan aquí. Inversamente, .git/ contiene cada una de las confirmaciones.

Su estructura básica es así:

.git/
    objects/
    refs/

Objetos #

git es fundamentalmente un almacén de clave-valor. Cuando agrega datos a git, construye un objeto y usa el hash SHA-1 del contenido del objeto como clave.

Por lo tanto, cualquier contenido en git se puede buscar por su hash:

git gato-archivo -p 4bb6f98

Hay 4 tipos de ‘Objeto’:

  • mancha
  • árbol
  • comprometer
  • etiqueta

referencia ## CABEZA HEAD es una ref especial. Siempre apunta al objeto actual.

Puede ver hacia dónde apunta actualmente al verificar el archivo .git/HEAD.

Normalmente, HEAD apunta a otra ref:

$cat .git/HEAD
ref: refs/heads/mainline

Pero también puede apuntar directamente a un objeto:

$ cat .git/HEAD
4bb6f98a223abc9345a0cef9200562333

Esto es lo que se conoce como “cabeza separada”, porque ‘HEAD’ no está unido a (apuntando a) ninguna ‘ref’, sino que apunta directamente a un ‘objeto’.

Referencias #

Un ref es esencialmente un puntero. Es un nombre que apunta a un objeto. Por ejemplo,

"master" --> 1a410e...

Se almacenan en `.git/refs/heads/ en archivos de texto sin formato.

$ cat .git/refs/heads/mainline
4bb6f98a223abc9345a0cef9200562333

Esto es comúnmente lo que se llama ‘ramas’. Sin embargo, notarás que en git no existe tal cosa como una rama - solo una ref.

Ahora, es posible navegar por git simplemente saltando a diferentes objetos directamente por sus hashes. Pero esto sería terriblemente inconveniente. Una ref le da un nombre conveniente para referirse a los objetos. Es mucho más fácil pedirle a git que vaya a un lugar específico por nombre que por hash.

Objeto de compromiso #

Un commit es probablemente el tipo de objeto más familiar para los usuarios de git, ya que es lo que están acostumbrados a crear con los comandos git commit.

Sin embargo, el commit no contiene directamente ningún archivo o datos modificados. Más bien, contiene principalmente metadatos y punteros a otros “objetos” que contienen el contenido real de la “confirmación”.

Un commit contiene algunas cosas:

  • hash de un árbol
  • hash de un padre commit
  • nombre del autor/correo electrónico, nombre del autor/correo electrónico
  • mensaje de compromiso

Puedes ver el contenido de cualquier confirmación como esta:

$ git cat-file commit 5bac93
tree 04d1daef...
parent b7850ef5...
author Geddy Lee <glee@rush.com>
commiter Neil Peart <npeart@rush.com>

First commit!

Árbol #

Una nota muy importante es que los objetos tree almacenan CADA archivo en su proyecto, y almacena archivos completos, no diferencias. Esto significa que cada commit contiene una instantánea de todo el proyecto*.

*Técnicamente, solo se almacenan los archivos modificados. Pero esto es más un detalle de implementación para la eficiencia. Desde una perspectiva de diseño, se debe considerar que un commit contiene una copia completa del proyecto.

padre #

La línea principal contiene un hash de otro objeto commit, y puede considerarse como un “puntero principal” que apunta a la “confirmación anterior”. Esto forma implícitamente un gráfico de confirmaciones conocido como gráfico de confirmaciones. Específicamente, es un [gráfico acíclico dirigido] (https://en.wikipedia.org/wiki/Directed_acyclic_graph) (o DAG).

Objeto de árbol #

Un árbol básicamente representa una carpeta en un sistema de archivos tradicional: contenedores anidados para archivos u otras carpetas.

Un árbol contiene:

  • 0 o más objetos blob
  • 0 o más objetos árbol

Así como puedes usar ls o dir para listar el contenido de una carpeta, puedes listar el contenido de un objeto tree.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Puede buscar los archivos en un commit encontrando primero el hash del tree en el commit, y luego mirando ese tree:

$ git cat-file commit 4bb6f93a
tree 07b1a631
parent ...
author ...
commiter ... 
 
$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Objeto blob #

Un blob contiene contenidos de archivos binarios arbitrarios. Por lo general, será texto sin formato, como el código fuente o un artículo de blog. Pero fácilmente podrían ser los bytes de un archivo PNG o cualquier otra cosa.

Si tiene el hash de un blob, puede ver su contenido.

$ git cat-file -p d429810
package com.example.project

class Foo {
 ...
}
...

Por ejemplo, puede explorar un ‘árbol’ como se muestra arriba y luego mirar una de las ‘manchas’ que contiene.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
100644 blob cae391ff   Readme.txt

$ git cat-file -p cae391ff
Welcome to my project! This is the readmefile
...

Creando nuevos compromisos #

El comando git commit hace algunas cosas:

  1. Cree blobs y trees para representar el directorio de su proyecto, almacenado en .git/objects
  2. Crea un nuevo objeto commit con su información de autor, mensaje de confirmación y el árbol raíz del paso 1, también almacenado en .git/objects
  3. Actualiza la referencia HEAD en .git/HEAD al hash del commit recién creado

Esto da como resultado que se agregue una nueva instantánea de su proyecto a git que está conectado al estado anterior.

CABEZA móvil #

Cuando ejecuta git checkout en una confirmación (especificada por hash o ref) le está diciendo a git que haga que su directorio de trabajo se vea como cuando se tomó la instantánea.

  1. Actualice los archivos en el directorio de trabajo para que coincidan con el árbol dentro del commit
  2. Actualice HEAD para que apunte al hash o referencia especificado

Mover referencias #

Ejecutar git reset --hard mueve las referencias al hash/ref especificado.

Moviendo MyBranch a b8dc53:

$ git checkout MyBranch      # moves HEAD to MyBranch
$ git reset --hard b8dc53    # makes MyBranch point to b8dc53

Creando nuevas referencias #

Ejecutar git checkout -b <refname> creará una nueva referencia que apunta al commit actual.

$ cat .git/head
1f324a

$ git checkout -b TestBranch

$ cat .git/refs/heads/TestBranch
1f324a
