Hacer una copia de seguridad del sistema de archivos en lugar de usar pg_dumpall y pg_dump #

Es muy importante que si usa esto, llame a la función pg_start_backup() antes y pg_stop_backup() después. De lo contrario, hacer copias de seguridad del sistema de archivos no es seguro; incluso una instantánea de ZFS o FreeBSD del sistema de archivos respaldada sin esas llamadas de función pondrá la base de datos en modo de recuperación y puede perder transacciones.

Evitaría hacer copias de seguridad del sistema de archivos en lugar de copias de seguridad regulares de Postgres, tanto por este motivo como porque los archivos de copia de seguridad de Postgres (especialmente en el formato personalizado) son extremadamente versátiles para admitir restauraciones alternativas. Dado que son archivos individuales, también son menos complicados de administrar.

Copia de seguridad de una base de datos #

pg_dump -Fc -f DATABASE.pgsql DATABASE

El -Fc selecciona el “formato de copia de seguridad personalizado” que le da más poder que el SQL sin procesar; ver pg_restore para más detalles. Si desea un archivo SQL estándar, puede hacer esto en su lugar:

pg_dump -f DATABASE.sql DATABASE

o incluso

pg_dump DATABASE > DATABASE.sql

Restaurando copias de seguridad #

psql < backup.sql

Una alternativa más segura usa -1 para envolver la restauración en una transacción. El -f especifica el nombre del archivo en lugar de usar la redirección de shell.

psql -1f backup.sql

Los archivos de formato personalizado deben restaurarse usando pg_restore con la opción -d para especificar la base de datos:

pg_restore -d DATABASE DATABASE.pgsql

El formato personalizado también se puede volver a convertir a SQL:

pg_restore backup.pgsql > backup.sql

Se recomienda el uso del formato personalizado porque puede elegir qué cosas restaurar y, opcionalmente, habilitar el procesamiento paralelo.

Es posible que deba hacer un pg_dump seguido de un pg_restore si actualiza de una versión de postgresql a una más nueva.

Copia de seguridad de todo el clúster #

$ pg_dumpall -f backup.sql

Esto funciona tras bambalinas haciendo múltiples conexiones al servidor una vez para cada base de datos y ejecutando pg_dump en ella.

A veces, puede tener la tentación de configurar esto como un trabajo cron, por lo que desea ver la fecha en que se realizó la copia de seguridad como parte del nombre del archivo:

$ postgres-backup-$(date +%Y-%m-%d).sql

Sin embargo, tenga en cuenta que esto podría producir archivos de gran tamaño a diario. Postgresql tiene un mecanismo mucho mejor para copias de seguridad periódicas: archivos WAL

La salida de pg_dumpall es suficiente para restaurar a una instancia de Postgres configurada de manera idéntica, pero los archivos de configuración en $PGDATA (pg_hba.conf y postgresql.conf) no son parte de la copia de seguridad, por lo que tendrá para realizar copias de seguridad por separado.

postgres=# SELECT pg_start_backup('my-backup');
postgres=# SELECT pg_stop_backup();

Para realizar una copia de seguridad del sistema de archivos, debe usar estas funciones para ayudar a garantizar que Postgres esté en un estado consistente mientras se prepara la copia de seguridad.

Usar psql para exportar datos #

Los datos se pueden exportar usando el comando de copia o usando las opciones de la línea de comando del comando psql.

Para exportar datos csv del usuario de la tabla al archivo csv:

psql -p \<port> -U \<username> -d \<database> -A -F<delimiter> -c\<sql to execute> \> \<output filename with path>

psql -p 5432 -U postgres -d test_database -A -F, -c "select * from user" > /home/user/user_data.csv

Aquí la combinación de -A y -F hace el truco.

-F es para especificar el delimitador

-A or --no-align

Cambia al modo de salida no alineado. (De lo contrario, el modo de salida predeterminado está alineado).

Usando Copiar para importar #

Para copiar datos de un archivo CSV a una tabla #

COPY <tablename> FROM '<filename with path>';

Para insertar en la tabla usuario desde un archivo llamado user_data.csv ubicado dentro de /home/user/:

COPY user FROM '/home/user/user_data.csv';

Para copiar datos del archivo separado por tubería a la tabla #

COPY user FROM '/home/user/user_data' WITH DELIMITER '|';

Nota: en ausencia de la opción con delimitador, el delimitador predeterminado es la coma ,

Para ignorar la línea de encabezado al importar un archivo #

Utilice la opción Encabezado:

COPY user FROM '/home/user/user_data' WITH DELIMITER '|' HEADER;

Nota: si los datos están entrecomillados, los caracteres de comillas de datos predeterminados son comillas dobles. Si los datos se citan con cualquier otro carácter, utilice la opción COMITACIÓN; sin embargo, esta opción solo está permitida cuando se utiliza el formato CSV.
