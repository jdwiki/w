MailboxProcessor maintient une file d’attente de messages interne, où plusieurs producteurs peuvent publier des messages en utilisant diverses variantes de la méthode Post. Ces messages sont ensuite récupérés et traités par un seul consommateur (sauf si vous l’implémentez autrement) à l’aide des variantes Retrieve et Scan. Par défaut, la production et la consommation des messages sont thread-safe.

Par défaut, aucune gestion des erreurs n’est fournie. Si une exception non interceptée est lancée à l’intérieur du corps du processeur, la fonction du corps se terminera, tous les messages de la file d’attente seront perdus, plus aucun message ne pourra être envoyé et le canal de réponse (si disponible) recevra une exception au lieu d’une réponse. Vous devez fournir vous-même toute la gestion des erreurs au cas où ce comportement ne conviendrait pas à votre cas d’utilisation.

Bonjour tout le monde de base #

Commençons par créer un simple “Hello world!” MailboxProcessor qui traite un type de message et imprime les salutations.

Vous aurez besoin du type de message. Cela peut être n’importe quoi, mais [Les syndicats discriminés] (https://www.wikiod.com/fr/docs/f%23/1025/discriminated-unions#t=201703110747364524357) sont un choix naturel ici car ils répertorient tous les cas possibles au même endroit et vous pouvez facilement utiliser la correspondance de modèles lors de leur traitement.

// In this example there is only a single case, it could technically be just a string
type GreeterMessage = SayHelloTo of string

Définissez maintenant le processeur lui-même. Cela peut être fait avec la méthode statique MailboxProcessor<'message>.Start qui renvoie un processeur démarré prêt à faire son travail. Vous pouvez également utiliser le constructeur, mais vous devez ensuite vous assurer de démarrer le processeur plus tard.

let processor = MailboxProcessor<GreeterMessage>.Start(fun inbox ->
    let rec innerLoop () = async {
        // This way you retrieve message from the mailbox queue
        // or await them in case the queue empty.
        // You can think of the `inbox` parameter as a reference to self.
        let! message = inbox.Receive()
        // Now you can process the retrieved message.
        match message with
        | SayHelloTo name ->
            printfn "Hi, %s! This is mailbox processor's inner loop!" name
        // After that's done, don't forget to recurse so you can process the next messages!
        innerLoop()
    }
    innerLoop ())

Le paramètre de Start est une fonction qui prend une référence au MailboxProcessor lui-même (qui n’existe pas encore car vous venez de le créer, mais sera disponible une fois la fonction exécutée). Cela vous donne accès à ses différentes méthodes Receive et Scan pour accéder aux messages de la boîte aux lettres. À l’intérieur de cette fonction, vous pouvez effectuer tout le traitement dont vous avez besoin, mais une approche habituelle est une boucle infinie qui lit les messages un par un et s’appelle après chacun.

Maintenant le processeur est prêt, mais il ne sert à rien ! Pourquoi? Vous devez lui envoyer un message à traiter. Cela se fait avec les variantes de la méthode Post - utilisons la plus basique, la plus simple, la plus simple.

processor.Post(SayHelloTo "Alice")

Cela place un message dans la file d’attente interne du processeur, la boîte aux lettres, et le renvoie immédiatement afin que le code appelant puisse continuer. Une fois que le processeur a récupéré le message, il le traitera, mais cela se fera de manière asynchrone pour le publier, et cela se fera très probablement sur un thread séparé.

Très peu de temps après, vous devriez voir le message "Salut, Alice ! C'est la boucle interne du processeur de boîte aux lettres!" imprimé sur la sortie et vous êtes prêt pour des échantillons plus compliqués.

Gestion des états mutables #

Les processeurs de boîtes aux lettres peuvent être utilisés pour gérer l’état mutable de manière transparente et sans fil. Construisons un compteur simple.

// Increment or decrement by one.
type CounterMessage =
    | Increment
    | Decrement

let createProcessor initialState =
    MailboxProcessor<CounterMessage>.Start(fun inbox ->
        // You can represent the processor's internal mutable state
        // as an immutable parameter to the inner loop function
        let rec innerLoop state = async {
            printfn "Waiting for message, the current state is: %i" state
            let! message = inbox.Receive()
            // In each call you use the current state to produce a new
            // value, which will be passed to the next call, so that
            // next message sees only the new value as its local state
            match message with
            | Increment ->
                let state' = state + 1
                printfn "Counter incremented, the new state is: %i" state'
                innerLoop state'
            | Decrement ->
                let state' = state - 1
                printfn "Counter decremented, the new state is: %i" state'
                innerLoop state'
        }
        // We pass the initialState to the first call to innerLoop
        innerLoop initialState)

// Let's pick an initial value and create the processor
let processor = createProcessor 10

Maintenant, générons quelques opérations

processor.Post(Increment)
processor.Post(Increment)
processor.Post(Decrement)
processor.Post(Increment)

Et vous verrez le journal suivant

Waiting for message, the current state is: 10
Counter incremented, the new state is: 11
Waiting for message, the current state is: 11
Counter incremented, the new state is: 12
Waiting for message, the current state is: 12
Counter decremented, the new state is: 11
Waiting for message, the current state is: 11
Counter incremented, the new state is: 12
Waiting for message, the current state is: 12

Concurrence #

Étant donné que le processeur de boîte aux lettres traite les messages un par un et qu’il n’y a pas d’entrelacement, vous pouvez également produire les messages à partir de plusieurs threads et vous ne verrez pas les problèmes typiques des opérations perdues ou dupliquées. Il n’y a aucun moyen pour un message d’utiliser l’ancien état d’autres messages, à moins que vous n’implémentiez spécifiquement le processeur.

let processor = createProcessor 0

[ async { processor.Post(Increment) }
  async { processor.Post(Increment) }
  async { processor.Post(Decrement) }
  async { processor.Post(Decrement) } ]
|> Async.Parallel
|> Async.RunSynchronously

Tous les messages sont postés à partir de différents fils de discussion. L’ordre dans lequel les messages sont publiés dans la boîte aux lettres n’est pas déterministe, donc l’ordre de traitement n’est pas déterministe, mais puisque le nombre total d’incréments et de décréments est équilibré, vous verrez l’état final être 0, peu importe dans quel ordre et à partir de quels fils les messages ont été envoyés.

Vrai état mutable #

Dans l’exemple précédent, nous n’avons simulé qu’un état mutable en passant le paramètre de boucle récursive, mais le processeur de boîte aux lettres a toutes ces propriétés même pour un état véritablement mutable. Ceci est important lorsque vous maintenez un état étendu et que l’immuabilité n’est pas pratique pour des raisons de performances.

Nous pouvons réécrire notre compteur à l’implémentation suivante

let createProcessor initialState =
    MailboxProcessor<CounterMessage>.Start(fun inbox ->
        // In this case we represent the state as a mutable binding
        // local to this function. innerLoop will close over it and
        // change its value in each iteration instead of passing it around
        let mutable state = initialState

        let rec innerLoop () = async {
            printfn "Waiting for message, the current state is: %i" state
            let! message = inbox.Receive()
            match message with
            | Increment ->
                let state <- state + 1
                printfn "Counter incremented, the new state is: %i" state'
                innerLoop ()
            | Decrement ->
                let state <- state - 1
                printfn "Counter decremented, the new state is: %i" state'
                innerLoop ()
        }
        innerLoop ())

Même si cela ne serait certainement pas thread-safe si l’état du compteur était modifié directement à partir de plusieurs threads, vous pouvez voir en utilisant le message parallèle Messages de la section précédente que le processeur de boîte aux lettres traite les messages les uns après les autres sans entrelacement, de sorte que chaque message utilise le valeur la plus actuelle.

Valeurs de retour #

Vous pouvez renvoyer de manière asynchrone une valeur pour chaque message traité si vous envoyez un AsyncReplyChannel<'a> dans le cadre du message.

type MessageWithResponse = Message of InputData * AsyncReplyChannel<OutputData>

Ensuite, le processeur de boîte aux lettres peut utiliser ce canal lors du traitement du message pour renvoyer une valeur à l’appelant.

let! message = inbox.Receive()
match message with
| MessageWithResponse (data, r) ->
    // ...process the data
    let output = ...
    r.Reply(output)

Maintenant, pour créer un message, vous avez besoin du AsyncReplyChannel<'a> - qu’est-ce que c’est et comment créez-vous une instance de travail ? La meilleure façon est de laisser MailboxProcessor le fournir pour vous et d’extraire la réponse à un Async<'a> plus courant. Cela peut être fait en utilisant par exemple la méthode PostAndAsynReply, où vous ne postez pas le message complet, mais à la place une fonction de type (dans notre cas) AsyncReplyChannel<OutputData> -> MessageWithResponse :

let! output = processor.PostAndAsyncReply(r -> MessageWithResponse(input, r))

Cela publiera le message dans une file d’attente et attendra la réponse, qui arrivera une fois que le processeur arrivera à ce message et répondra en utilisant le canal.

Il existe également une variante synchrone “PostAndReply” qui bloque le thread appelant jusqu’à ce que le processeur réponde.

## Traitement des messages dans le désordre Vous pouvez utiliser les méthodes Scan ou TryScan pour rechercher des messages spécifiques dans la file d’attente et les traiter quel que soit le nombre de messages qui les précèdent. Les deux méthodes examinent les messages de la file d’attente dans l’ordre dans lequel ils sont arrivés et recherchent un message spécifié (jusqu’au délai d’attente facultatif). S’il n’y a pas de message de ce type, TryScan renverra Aucun, tandis que Scan continuera d’attendre jusqu’à ce que ce message arrive ou que l’opération expire.

Voyons cela en pratique. Nous voulons que le processeur traite RegularOperations quand il le peut, mais chaque fois qu’il y a une PriorityOperation, elle doit être traitée dès que possible, quel que soit le nombre d’autres RegularOperations dans la file d’attente.

type Message =
    | RegularOperation of string
    | PriorityOperation of string

let processor = MailboxProcessor<Message>.Start(fun inbox ->
    let rec innerLoop () = async {
        let! priorityData = inbox.TryScan(fun msg ->
            // If there is a PriorityOperation, retrieve its data.
            match msg with
            | PriorityOperation data -> Some data
            | _ -> None)

        match priorityData with
        | Some data ->
            // Process the data from PriorityOperation.
        | None ->
            // No PriorityOperation was in the queue at the time, so
            // let's fall back to processing all possible messages
            let! message = inbox.Receive()
            match message with
            | RegularOperation data ->
                // We have free time, let's process the RegularOperation.
            | PriorityOperation data ->
                // We did scan the queue, but it might have been empty
                // so it is possible that in the meantime a producer
                // posted a new message and it is a PriorityOperation.
        // And never forget to process next messages.
        innerLoop ()
    }
    innerLoop())
