La correspondance de modèles est une fonctionnalité puissante de nombreux langages fonctionnels car elle permet souvent de gérer les branchements de manière très succincte par rapport à l’utilisation de plusieurs instructions de style if/else if/else. Cependant, avec suffisamment d’options et de gardes [“quand”] [1], Pattern Matching peut également devenir verbeux et difficile à comprendre en un coup d’œil.

Lorsque cela se produit, les [Active Patterns][2] de F# peuvent être un excellent moyen de donner des noms significatifs à la logique correspondante, ce qui simplifie le code et permet également la réutilisation.

[1] : https://www.wikiod.com/fr/docs/f%23/1335/pattern-matching/7377/when-guards-let-you-add-arbitrary-conditionals#t=201607222211464650253 [2] : https://www.wikiod.com/fr/docs/f%23/962/active-patterns#t=201607222206441747382

La correspondance de modèle vérifie que l’ensemble du domaine est couvert #

let x = true
match x with
| true -> printfn "x is true"

donne un avertissement #

C:\Program Files (x86)\Microsoft VS Code\Untitled-1(2,7) : avertissement FS0025 : correspondances de modèle incomplètes sur cette expression. Par exemple, la valeur ‘false’ peut indiquer un cas non couvert par le(s) modèle(s).

En effet, toutes les valeurs booléennes possibles n’ont pas été couvertes.

les booléens peuvent être listés explicitement mais les entiers sont plus difficiles à lister #

let x = 5
match x with
| 1 -> printfn "x is 1"
| 2 -> printfn "x is 2"
| _ -> printfn "x is something else"

ici, nous utilisons le caractère spécial _. Le _ correspond à tous les autres cas possibles.

Le _ peut vous attirer des ennuis #

considérez un type que nous créons nous-mêmes, il ressemble à ceci

type Sobriety = 
    | Sober
    | Tipsy
    | Drunk

Nous pourrions écrire un match avec une expression qui ressemble à ceci

match sobriety with
| Sober -> printfn "drive home"
| _ -> printfn "call an uber"

Le code ci-dessus est logique. Nous supposons que si vous n’êtes pas sobre, vous devez appeler un uber, nous utilisons donc le _ pour indiquer que

Nous refactorons plus tard notre code à ceci

type Sobriety = 
    | Sober
    | Tipsy
    | Drunk
    | Unconscious

Le compilateur F # doit nous avertir et nous inviter à refactoriser notre expression de correspondance pour que la personne consulte un médecin. Au lieu de cela, l’expression de correspondance traite silencieusement la personne inconsciente comme si elle n’était qu’éméchée. Le fait est que vous devez choisir de lister explicitement les cas lorsque cela est possible pour éviter les erreurs de logique.

Lorsque les gardes vous permettent d’ajouter des conditions arbitraires #

type Person = {
    Age : int
    PassedDriversTest : bool }

let someone = { Age = 19; PassedDriversTest = true }

match someone.PassedDriversTest with
| true when someone.Age >= 16 -> printfn "congrats"
| true -> printfn "wait until you are 16"
| false -> printfn "you need to pass the test"

Les cas sont évalués de haut en bas et la première correspondance est utilisée #

Utilisation incorrecte :

Dans l’extrait suivant, la dernière correspondance ne sera jamais utilisée :

let x = 4
match x with
| 1 -> printfn "x is 1"
| _ -> printfn "x is anything that wasn't listed above"
| 4 -> printfn "x is 4"

impressions

x est tout ce qui n’était pas listé ci-dessus

Usage correct:

Ici, x = 1 et x = 4 atteindront leurs cas spécifiques, tandis que tout le reste tombera dans le cas par défaut _ :

let x = 4
match x with
| 1 -> printfn "x is 1"
| 4 -> printfn "x is 4"
| _ -> printfn "x is anything that wasn't listed above"

impressions

x vaut 4

Options de correspondance #

La correspondance de motifs peut être utile pour gérer les options :

let result = Some("Hello World")
match result with
| Some(message) -> printfn message
| None -> printfn "Not feeling talkative huh?"
