** Pourquoi ai-je besoin de ces constructeurs sans arguments ??? **

  • Que se passe-t-il si un bean de portée session est injecté dans un bean de portée application ? Comment le bean de portée d’application obtient-il la bonne instance de bean de portée de session pour chaque requête ? Le bean de portée de session ne fuirait-il pas dans d’autres requêtes ? Comment cela fonctionne-t-il ?* Afin de faciliter le cadrage, CDI utilise ce qu’on appelle un proxy. Lorsque CDI injecte un bean de portée non dépendant dans un autre objet, il n’injecte pas le bean directement. Au lieu de cela, il sous-classe ce bean pour créer ce qu’on appelle un proxy. Chaque fois qu’une méthode est appelée sur le proxy, elle demande au runtime CDI de rechercher le bean correct pour cette portée particulière (s’il s’agit d’une portée de requête, obtenez le bean pour cette requête. S’il s’agit d’une portée de session, obtenez le bean pour cette session. etc. .), puis transmet l’appel à l’objet réel, renvoyant tout résultat pour les méthodes non vides. Cela signifie que ce qui suit est une bonne chose à faire :

    @ApplicationScoped public class ApplicationScopedClass {

      private final RequestScopedClass requestScopedClass;

  @Inject
  public ApplicationScopedClass(RequestScopedClass requestScopedClass) {
      this.requestScopedClass = requestScopedClass;
  }

  public ApplicationScopedClass() {

  }

  public doSomething() {
      requestScopedClass.doSomethingRequestSpecific(); //This works, because of the proxy
  }

    }

Cependant, pour qu’une classe sous-classe légalement une autre classe en Java, elle doit avoir un constructeur valide. Et un constructeur valide doit appeler soit un autre constructeur sur la classe, soit le constructeur de la classe parent (par exemple super()). Afin de simplifier cette sous-classification, la spécification CDI exige que tout bean de portée non dépendant fournisse un constructeur public sans argument, de sorte que le runtime n’ait pas à essayer de deviner quoi faire avec une classe qui doit être proxy .

La portée par défaut #

public class DependentScopedClass {

    //This class has no scoping annotations, so a new instance gets created at every injection point.

    @Inject
    public DependentScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

}

La portée par défaut de la plupart des beans CDI est appelée portée dépendante. Une classe qui ne contient aucune annotation d’étendue sera traitée comme une étendue dépendante, sauf si il s’agit d’une ressource ou d’un fournisseur JAX-RS (les ressources sont par défaut étendues à la demande et les fournisseurs par défaut à l’étendue application). Une instance d’une classe de portée dépendante vit aussi longtemps que l’objet dans lequel elle est injectée. Chaque fois qu’une classe est construite et a une classe de portée dépendante comme dépendance, la classe de portée dépendante est créée et injectée directement dans l’objet qui en a besoin. La signification de ceci deviendra apparente après des exemples ultérieurs.

@RequestScoped #

@RequestScoped
public class RequestScopedClass {
    //This class gets constructed once per Servlet request, and is shared among all CDI-managed classes within that request.

    @Inject
    public RequestScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public RequestScopedClass() {
        //Note that it is required that a request scoped class have a public no-args constructor
    }

}

Si un bean est annoté avec @RequestScoped, il sera créé une fois pour toute requête. Si deux objets dépendent d’une classe de portée de requête, ils obtiendront tous les deux des références au même objet.

Remarque : Tout bean de portée de requête doit avoir un constructeur public sans argument. La raison en sera expliquée plus tard.

@ApplicationScoped #

@ApplicationScoped
public class ApplicationScopedClass {
    //This class gets constructed once for the entire life of the application, and is shared among all CDI-managed classes throughout the life of the application.

    @Inject
    public ApplicationScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public ApplicationScopedClass() {
        //Note that it is required that an application scoped class have a public no-args constructor
    }

}

Les classes avec @ApplicationScoped ne sont créées qu’une seule fois, et chaque objet qui dépend de la classe partage la même instance. Ces classes sont “en fait” des singletons, mais il convient de noter que rien n’empêche un codeur de créer manuellement des instances supplémentaires de la classe. Ainsi, l’utilisation de @ApplicationScoped est utile pour partager le contexte dans l’ensemble de l’application, ou comme optimisation des performances (si la classe coûte cher à construire des instances de), mais il ne faut pas s’y fier comme mesure d’intégrité pour garantir une seule instance d’un classe existe.

Comme les beans de portée de requête, les beans de portée d’application doivent avoir un constructeur public sans argument.

@SessionScoped #

@SessionScoped
public class SessionScopedClass implements Serializable {
    //This class gets constructed once per session, and is shared among all CDI-managed classes within that session. Notice that it implements Serializable, since the instance will get put on the session.

    @Inject
    public SessionScopedClass(SomeDependency someDependency) {
        doSomethingWith(someDependency);
    }

    public SessionScopedClass() {
        //Note that it is required that a session scoped class have a public no-args constructor
    }

}

Les classes annotées avec @SessionScoped seront créées une fois par session, et deux objets au sein de la même session partageront la même instance de la classe de portée de session.

Il est important de noter que la classe de portée de session doit implémenter Serializable. Cette exigence existe car le bean sera stocké dans la session du conteneur de servlet pour cette instance de portée de session particulière. En général, tout objet placé dans la session doit être sérialisable pour deux raisons : premièrement, les sessions peuvent être conservées sur le disque après un certain temps d’inactivité pour économiser de la mémoire. Ceci est connu sous le nom de passivation de session et nécessite une sérialisation. La deuxième raison est que dans les clusters à haute disponibilité, la réplication de session est souvent utilisée afin de permettre à n’importe quel serveur du cluster de traiter une requête pour une session donnée. Cela nécessite également généralement une sérialisation.

Tout comme les classes de portée de requête et de portée d’application, les classes de portée de session doivent avoir un constructeur public sans arguments.
