Syntaxe #

  • Vous pouvez utiliser ‘*’ comme deuxième et troisième paramètre dans les méthodes Phalcon\Acl::allow et Phalcon\Acl::deny. Cela signifiera respectivement toute ressource et action.

  • Le deuxième argument dans Phalcon\Acl::addRole indique à partir de quel rôle l’accès à l’héritage.

  • Vous devez sérialiser votre ACL dans un fichier ou un backend de cache au lieu de le créer à chaque requête.

  • C’est aussi une bonne idée de garder acl dans un fichier séparé.

  • Phalcon\Acl est capable d’envoyer des événements au gestionnaire d’événements, il y a deux événements - beforeCheckAccess et afterCheckAccess.

  • Vous pouvez utiliser Phalcon\Acl\AdapterInterface pour implémenter votre propre adaptateur ACL.

  • Vous pouvez protéger vos itinéraires en utilisant acl avec une combinaison d’écouteur approprié dans le répartiteur

Création d’une ACL #

Vous pouvez créer une ACL en utilisant la classe Phalcon\Acl\Adapter\Memory :

$acl = new Phalcon\Acl\Adapter\Memory();

Par défaut, phalcon autorise l’action sur une ressource qui n’a pas été définie, pour changer cela, vous pouvez utiliser :

$acl->setDefaultAction(Phalcon\Acl::DENY);

Les rôles peuvent être ajoutés de deux manières - en utilisant Phalcon\Acl\Role ou simplement une chaîne :

$roleAdministrator = new Phalcon\Acl\Role('Administrator');
$acl->addRole($roleAdministrator);
$acl->addRole('Customer');

Les ressources peuvent également être ajoutées de deux manières, vous pouvez ajouter des actions en tant qu’action unique ou en tant que tableau :

$resourceCategories = new Phalcon\Acl\Resource('categories');
$acl->addResource($resourceCategories, 'create');
$acl->addResource('products', ['create', 'update']);

Définir le contrôle d’accès et interroger une ACL #

Vous pouvez autoriser le rôle à accéder à certaines actions sur la ressource en :

$acl->allow('Administrator', 'products', 'create');

Vous pouvez refuser au rôle d’accéder à certaines actions sur la ressource en :

$acl->deny('Customer', 'categories', 'create');

Vous pouvez vérifier si le rôle est autorisé à effectuer certaines actions sur la ressource en utilisant :

$acl->isAllowed('Administrator', 'products', 'create');

Objets en tant que rôles et ressources #

En implémentant Phalcon\Acl\RoleAware ou Phalcon\Acl\ResourceAware vous pouvez les utiliser comme objets dans Phalcon\Acl\Adapter\Memory::isAllowed().

// Create our class which will be used as roleName
class UserRole implements Phalcon\Acl\RoleAware
{
    protected $id;
    protected $roleName;

    public function __construct($id, $roleName)
    {
        $this->id = $id;
        $this->roleName = $roleName;
    }

    public function getId()
    {
        return $this->id;
    }

    // Implemented function from RoleAware Interface
    public function getRoleName()
    {
        return $this->roleName;
    }
}

// Create our class which will be used as resourceName
class ModelResource implements Phalcon\Acl\ResourceAware
{
    protected $id;
    protected $resourceName;
    protected $userId;

    public function __construct($id, $resourceName, $userId)
    {
        $this->id = $id;
        $this->resourceName = $resourceName;
        $this->userId = $userId;
    }

    public function getId()
    {
        return $this->id;
    }

    public function getUserId()
    {
        return $this->userId;
    }

    // Implemented function from ResourceAware Interface
    public function getResourceName()
    {
        return $this->resourceName;
    }
}

$customer = new ModelResource(1, "products", 2);
$administrator = new UserRole(1, "Administrator");
$acl->isAllowed($administrator, $customer, 'create');

La possibilité d’utiliser des objets peut également être combinée avec une condition supplémentaire dans acl :

$acl->allow('Administrator', 'products', 'update', function(UserRole $user, ModelResource $model) {
    return $user->getId == $model->getUserId();
});
$product = new ModelResource(1, 'products', 2);
$administrator = new UserRole(1, 'Administrator');
$anotherAdministrator = new UserRole(2, 'Administrator');
$acl->isAllowed($administrator, $product, 'update'); // this will return false
$acl->isAllowed($anotherAdministrator, $product, 'update'); // this will return true

Notez qu’avec une condition supplémentaire et en utilisant des objets dans la méthode isAllowed, vous n’avez pas besoin de transmettre ces objets en tant qu’arguments. Ils sont transmis automatiquement uniquement s’il existe des types corrects avant les arguments dans la fonction. Cela vous donne une énorme capacité à contrôler si certains utilisateurs peuvent modifier, par exemple, certains modèles de votre application et quand ils peuvent le faire.

Condition supplémentaire dans ACL #

Vous pouvez également ajouter un peu plus de logique qui doit être vérifiée dans votre ACL à l’aide de fonctions anonymes. Ils seront exécutés lors de l’utilisation de Phalcon\Acl\Adapter\Memory::allow() ou Phalcon\Acl\Adapter\Memory::deny(), s’ils retournent true, leur rôle sera autorisé à accéder à certains action sur la ressource.

$acl->allow('Customer', 'products', 'create', function($parameter) {
    return $parameter % 2 == 0;
});
$acl->isAllowed('Customer', 'products', 'create', ['parameter' => 1]); // this will return false
$acl->isAllowed('Customer', 'products', 'create', ['parameter' => 2]); // this will return true

Remarquez comment les paramètres sont passés à la fonction. Votre clé dans array doit avoir le même nom que dans function. Les paramètres par défaut peuvent également être passés, ainsi que les objets.
