Syntaxe #

  • nom du module où – exporte tous les noms déclarés dans ce fichier

  • Nom du module (functionOne, Type (..)) où – exporte uniquement les constructeurs de functionOne, Type et Type

  • module d’importation - importez tous les noms exportés du module

  • importer le module qualifié en tant que MN – importation qualifiée

  • module d’importation (justThisFunction) - n’importe que certains noms d’un module

  • import Module masquant (functionName, Type) – importe tous les noms d’un module sauf functionName et Type

Haskell prend en charge les modules :

  • un module peut exporter tout ou un sous-ensemble de ses types et fonctions de membres

  • un module peut “réexporter” les noms qu’il a importés d’autres modules

Du côté consommateur d’un module, on peut :

  • importer tout ou un sous-ensemble des membres du module

  • masquer les importations d’un membre particulier ou d’un ensemble de membres

[haskell.org][1] a un excellent chapitre sur la définition des modules.

[1] : https://www.haskell.org/tutorial/modules.html

Définir votre propre module #

Si nous avons un fichier appelé “Business.hs”, nous pouvons définir un module “Business” qui peut être “importé”, comme ceci :

module Business (
    Person (..), -- ^ Export the Person type and all its constructors and field names
    employees   -- ^ Export the employees function
) where
-- begin types, function definitions, etc

Une hiérarchie plus profonde est bien sûr possible ; voir l’exemple [Noms de modules hiérarchiques][1].

[1] : https://www.wikiod.com/fr/haskell/modules#Noms de modules hiérarchiques

Exportation des constructeurs #

Pour exporter le type et tous ses constructeurs, il faut utiliser la syntaxe suivante :

module X (Person (..)) where

Ainsi, pour les définitions de niveau supérieur suivantes dans un fichier appelé “People.hs” :

data Person = Friend String | Foe deriving (Show, Eq, Ord)

isFoe Foe = True
isFoe _   = False

Cette déclaration de module en haut :

module People (Person (..)) where

n’exporterait que Person et ses constructeurs Friend et Foe.

Si la liste d’exportation suivant le mot-clé module est omise, tous les noms liés au niveau supérieur du module seront exportés :

module People where

exporterait Person, ses constructeurs et la fonction isFoe.

Importer des membres spécifiques d’un module #

Haskell prend en charge l’importation d’un sous-ensemble d’éléments à partir d’un module.

import qualified Data.Stream (map) as D

n’importerait que map à partir de Data.Stream, et les appels à cette fonction nécessiteraient D. :

D.map odd [1..]

sinon le compilateur essaiera d’utiliser la fonction map de Prelude.

Masquer les importations #

Prelude définit souvent des fonctions dont les noms sont utilisés ailleurs. Ne pas masquer ces importations (ou utiliser des importations qualifiées lorsque des conflits se produisent) entraînera des erreurs de compilation.

[Data.Stream][1] définit les fonctions nommées map, head et tail qui entrent normalement en conflit avec celles définies dans Prelude. Nous pouvons masquer ces importations de Prelude en utilisant hiding :

import Data.Stream -- everything from Data.Stream
import Prelude hiding (map, head, tail, scan, foldl, foldr, filter, dropWhile, take) -- etc

En réalité, il faudrait trop de code pour masquer les conflits de Prelude comme celui-ci, donc vous utiliseriez en fait une importation « qualifiée » de « Data.Stream » à la place.

[1] : https://hackage.haskell.org/package/Stream-0.4.7.2/docs/Data-Stream.html

Importations éligibles #

Lorsque plusieurs modules définissent les mêmes fonctions par leur nom, le compilateur se plaindra. Dans de tels cas (ou pour améliorer la lisibilité), nous pouvons utiliser une importation “qualifiée” :

import qualified Data.Stream as D

Nous pouvons désormais éviter les erreurs de compilateur d’ambiguïté lorsque nous utilisons map, qui est défini dans Prelude et Data.Stream :

map (== 1) [1,2,3] -- will use Prelude.map
D.map (odd) (fromList [1..]) -- will use Data.Stream.map

Il est également possible d’importer un module avec uniquement les noms conflictuels qualifiés via import Data.Text as T, ce qui permet d’avoir Text au lieu de T.Text etc.

Noms de modules hiérarchiques #

Les noms des modules suivent la structure hiérarchique du système de fichiers. Avec la disposition de fichier suivante :

Foo/
├── Baz/
│   └── Quux.hs
└── Bar.hs
Foo.hs
Bar.hs

les en-têtes de module ressembleraient à ceci :

-- file Foo.hs
module Foo where

-- file Bar.hs
module Bar where

-- file Foo/Bar.hs
module Foo.Bar where

-- file Foo/Baz/Quux.hs
module Foo.Baz.Quux where

Notez que:

  • le nom du module est basé sur le chemin du fichier déclarant le module
  • Les dossiers peuvent partager un nom avec un module, ce qui donne une structure de nommage naturellement hiérarchique aux modules
