L’une des forces de Haskell est sa capacité à tirer parti du système de types pour modéliser des parties de votre domaine problématique dans le système de types. Ce faisant, on rencontre souvent des types très complexes. Lors de l’écriture de programmes avec ces types (c’est-à-dire avec des valeurs ayant ces types), il devient parfois presque impossible de “jongler” avec tous les types. Depuis GHC 7.8, il existe une nouvelle fonctionnalité syntaxique appelée trous typés. Les trous typés ne changent pas la sémantique du langage de base ; ils sont uniquement destinés à aider à l’écriture de programmes.

Pour une explication approfondie des trous typés, ainsi qu’une discussion sur la conception des trous typés, voir le [Haskell wiki][1].

Section du guide d’utilisation du GHC sur les [trous tapés][2].

[1] : https://wiki.haskell.org/GHC/Typed_holes [2] : https://downloads.haskell.org/~ghc/7.8.4/docs/html/users_guide/typed-holes.html

Syntaxe des trous typés #

Un trou typé est un trait de soulignement unique (_) ou un identifiant Haskell valide qui n’est pas dans la portée, dans un contexte d’expression. Avant l’existence de trous typés, ces deux choses déclencheraient une erreur, de sorte que la nouvelle syntaxe n’interfère pas avec l’ancienne syntaxe.

Contrôle du comportement des trous typés #

Le comportement par défaut des trous typés est de produire une erreur de compilation lorsqu’ils rencontrent un trou typé. Cependant, il existe plusieurs drapeaux pour affiner leur comportement. Ces drapeaux sont résumés comme suit ([GHC trac][1]) :

Par défaut, GHC a tapé les trous activés et produit une erreur de compilation lorsqu’il rencontre un trou typé.

Lorsque -fdefer-type-errors ou -fdefer-typed-holes est activé, le trou les erreurs sont converties en avertissements et entraînent des erreurs d’exécution lorsque évalué.

Le drapeau d’avertissement -fwarn-typed-holes est activé par défaut. Sans pour autant -fdefer-type-errors ou -fdefer-typed-holes cet indicateur est un no-op, puisque les trous typés sont une erreur dans ces conditions. Si l’un des les drapeaux de report sont activés (convertissant les erreurs de trou typées en avertissements) le drapeau -fno-warn-typed-holes désactive les avertissements. Ça signifie la compilation réussit silencieusement et l’évaluation d’un trou produira un erreur d’exécution.

[1] : https://ghc.haskell.org/trac/ghc/ticket/9497

Sémantique des trous typés #

La valeur d’un trou de type peut simplement être dite “indéfinie”, bien qu’un trou typé déclenche une erreur de compilation, il n’est donc pas strictement nécessaire de lui attribuer une valeur. Cependant, un trou typé (lorsqu’il est activé) produit une erreur de compilation (ou un avertissement avec des erreurs de type différées) qui indique le nom du trou typé, son type déduit le plus général et les types de toutes les liaisons locales. Par exemple:

Prelude> \x -> _var + length (drop 1 x)

<interactive>:19:7: Warning:
    Found hole `_var' with type: Int
    Relevant bindings include
      x :: [a] (bound at <interactive>:19:2)
      it :: [a] -> Int (bound at <interactive>:19:1)
    In the first argument of `(+)', namely `_var'
    In the expression: _var + length (drop 1 x)
    In the expression: \ x -> _var + length (drop 1 x)

Notez que dans le cas de trous typés dans les expressions saisies dans le GHCi repl (comme ci-dessus), le type de l’expression saisie est également signalé, comme it (ici de type [a] -> Int).

Utilisation de trous typés pour définir une instance de classe #

Les trous typés peuvent faciliter la définition des fonctions, grâce à un processus interactif.

Supposons que vous souhaitiez définir une instance de classe Foo Bar (pour votre type Bar personnalisé, afin de l’utiliser avec une fonction de bibliothèque polymorphe nécessitant une instance Foo). Vous devriez maintenant rechercher la documentation de Foo, déterminer quelles méthodes vous devez définir, examiner leurs types, etc. - mais avec des trous typés, vous pouvez en fait ignorer cela !

Définissez d’abord une instance factice :

instance Foo Bar where

Le compilateur va maintenant se plaindre

Bar.hs:13:10: Warning:
No explicit implementation for
  ‘foom’ and ‘quun’
In the instance declaration for ‘Foo Bar’

Ok, nous devons donc définir foom pour Bar. Mais qu’est-ce que c’est censé être ? Encore une fois, nous sommes trop paresseux pour regarder dans la documentation et demandons simplement au compilateur :

instance Foo Bar where
  foom = _

Ici, nous avons utilisé un trou typé comme une simple “requête de documentation”. Les sorties du compilateur

Bar.hs:14:10:
    Found hole ‘_’ with type: Bar -> Gronk Bar
    Relevant bindings include
      foom :: Bar -> Gronk Bar (bound at Foo.hs:4:28)
    In the expression: _
    In an equation for ‘foom’: foom = _
    In the instance declaration for ‘Foo Bar’

Notez que le compilateur a déjà rempli la variable de type de classe avec le type concret “Bar” pour lequel nous voulons l’instancier. Cela peut rendre la signature beaucoup plus facile à comprendre que la signature polymorphe trouvée dans la documentation de la classe, surtout si vous avez affaire à une méthode plus compliquée, par exemple. une classe de types multi-paramètres.

Mais qu’est-ce que c’est que “Gronk” ? À ce stade, c’est probablement une bonne idée de demander à Hayoo. Cependant, nous pouvons toujours nous en sortir sans cela : en tant que supposition aveugle, nous supposons qu’il ne s’agit pas seulement d’un constructeur de type, mais également du constructeur de valeur unique, c’est-à-dire qu’il peut être utilisé comme une fonction qui produira d’une manière ou d’une autre une valeur “Gronk a”. Alors on essaie

instance Foo Bar where
  foom bar = _ Gronk

Si nous avons de la chance, Gronk est en fait une valeur, et le compilateur dira maintenant

    Found hole ‘_’
      with type: (Int -> [(Int, b0)] -> Gronk b0) -> Gronk Bar
    Where: ‘b0’ is an ambiguous type variable

Ok, c’est moche - notez d’abord que Gronk a deux arguments, afin que nous puissions affiner notre tentative :

instance Foo Bar where
  foom bar = Gronk _ _

Et c’est maintenant assez clair :

    Found hole ‘_’ with type: [(Int, Bar)]
    Relevant bindings include
      bar :: Bar (bound at Bar.hs:14:29)
      foom :: Bar -> Gronk Bar (bound at Foo.hs:15:24)
    In the second argument of ‘Gronk’, namely ‘_’
    In the expression: Gronk _ _
    In an equation for ‘foom’: foom bar = Gronk _ _

Vous pouvez maintenant progresser davantage, par ex. déconstruire la valeur bar (les composants apparaîtront alors, avec les types, dans la section Liaisons pertinentes). Souvent, il est à un moment donné complètement évident quelle sera la définition correcte, car vous voyez tous les arguments disponibles et les types s’emboîtent comme un puzzle. Ou alternativement, vous pouvez voir que la définition est impossible et pourquoi.

Tout cela fonctionne mieux dans un éditeur avec compilation interactive, par ex. Emacs avec le mode haskell. Vous pouvez ensuite utiliser des trous typés un peu comme des requêtes de valeur de passage de la souris dans un IDE pour un langage impératif dynamique interprété, mais sans toutes les limitations.
