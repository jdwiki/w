Functor est la classe des types f :: * -> * qui peuvent être mappés de manière covariante. Le mappage d’une fonction sur une structure de données applique la fonction à tous les éléments de la structure sans modifier la structure elle-même.

Un Functor peut être considéré comme un conteneur pour une valeur ou un contexte de calcul. Les exemples sont Peut-être un ou [a]. L’article Typeclassopedia contient une bonne description des concepts sous-jacents aux foncteurs.

Pour être considérée comme un vrai Functor, une instance doit respecter les 2 lois suivantes :

Identité #

fmap id == id

Composition #

fmap (f . g) = (fmap f) . (fmap g)

Définition de classe du foncteur et des lois #

class Functor f where
    fmap :: (a -> b) -> f a -> f b

Une façon de voir les choses est que fmap lifts une fonction de valeurs en une fonction de valeurs dans un contexte f.

Une instance correcte de Functor doit satisfaire les lois de foncteur, bien que celles-ci ne soient pas appliquées par le compilateur :

fmap id = id                    -- identity
fmap f . fmap g = fmap (f . g)  -- composition

Il existe un alias d’infixe couramment utilisé pour fmap appelé <$>.

infixl 4 <$>
(<$>) :: Functor f => (a -> b) -> f a -> f b
(<$>) = fmap

## Instances courantes de Functor Peut-être #

Maybe est un Functor contenant une valeur éventuellement absente :

instance Functor Maybe where
    fmap f Nothing = Nothing
    fmap f (Just x) = Just (f x)

L’instance Maybe de Functor applique une fonction à une valeur enveloppée dans un Just. Si le calcul a déjà échoué (donc la valeur Maybe est un Nothing), alors il n’y a pas de valeur à laquelle appliquer la fonction, donc fmap est un no-op.

> fmap (+ 3) (Just 3)
Just 6
> fmap length (Just "mousetrap")
Just 9
> fmap sqrt Nothing
Nothing

Nous pouvons vérifier les lois des foncteurs pour cette instance en utilisant le raisonnement équationnel. Pour la loi sur l’identité,

fmap id Nothing
Nothing  -- definition of fmap
id Nothing  -- definition of id

fmap id (Just x)
Just (id x)  -- definition of fmap
Just x  -- definition of id
id (Just x)  -- definition of id

Pour la loi de composition,

(fmap f . fmap g) Nothing
fmap f (fmap g Nothing)  -- definition of (.)
fmap f Nothing  -- definition of fmap
Nothing  -- definition of fmap
fmap (f . g) Nothing  -- because Nothing = fmap f Nothing, for all f

(fmap f . fmap g) (Just x)
fmap f (fmap g (Just x))  -- definition of (.)
fmap f (Just (g x))  -- definition of fmap
Just (f (g x))  -- definition of fmap
Just ((f . g) x)  -- definition of (.)
fmap (f . g) (Just x)  -- definition of fmap

Listes #

L’instance de Functor des listes applique la fonction à chaque valeur de la liste en place.

instance Functor [] where
    fmap f [] = []
    fmap f (x:xs) = f x : fmap f xs

Cela pourrait alternativement être écrit comme une compréhension de liste : fmap f xs = [f x | x <- xs].

Cet exemple montre que fmap généralise map. map ne fonctionne que sur des listes, alors que fmap fonctionne sur un Functor arbitraire.

On peut montrer que la loi d’identité est vérifiée par induction :

-- base case
fmap id []
[]  -- definition of fmap
id []  -- definition of id

-- inductive step
fmap id (x:xs)
id x : fmap id xs  -- definition of fmap
x : fmap id xs  -- definition of id
x : id xs  -- by the inductive hypothesis
x : xs  -- definition of id
id (x : xs)  -- definition of id

et de même, la loi de composition :

-- base case
(fmap f . fmap g) []
fmap f (fmap g [])  -- definition of (.)
fmap f []  -- definition of fmap
[]  -- definition of fmap
fmap (f . g) []  -- because [] = fmap f [], for all f

-- inductive step
(fmap f . fmap g) (x:xs)
fmap f (fmap g (x:xs))  -- definition of (.)
fmap f (g x : fmap g xs)  -- definition of fmap
f (g x) : fmap f (fmap g xs)  -- definition of fmap
(f . g) x : fmap f (fmap g xs)  -- definition of (.)
(f . g) x : fmap (f . g) xs  -- by the inductive hypothesis
fmap (f . g) xs  -- definition of fmap

Les fonctions #

Tous les Functor ne ressemblent pas à un conteneur. L’instance des fonctions de Functor applique une fonction à la valeur de retour d’une autre fonction.

instance Functor ((->) r) where
    fmap f g = \x -> f (g x)

Notez que cette définition est équivalente à fmap = (.). Donc fmap généralise la composition des fonctions.

Une fois de plus, vérifiez la loi sur l’identité :

fmap id g
\x -> id (g x)  -- definition of fmap
\x -> g x  -- definition of id
g  -- eta-reduction
id g  -- definition of id

et la loi de composition :

(fmap f . fmap g) h
fmap f (fmap g h)  -- definition of (.)
fmap f (\x -> g (h x))  -- definition of fmap
\y -> f ((\x -> g (h x)) y)  -- definition of fmap
\y -> f (g (h y))  -- beta-reduction
\y -> (f . g) (h y)  -- definition of (.)
fmap (f . g) h  -- definition of fmap

Remplacement de tous les éléments d’un Functor par une seule valeur #

Le module Data.Functor contient deux combinateurs, <$ et $>, qui ignorent toutes les valeurs contenues dans un foncteur, les remplaçant toutes par une seule valeur constante.

infixl 4 <$, $>

<$ :: Functor f => a -> f b -> f a
(<$) = fmap . const

$> :: Functor f => f a -> b -> f b
($>) = flip (<$)

void ignore la valeur de retour d’un calcul.

void :: Functor f => f a -> f ()
void = (() <$)

Foncteurs polynomiaux #

Il existe un ensemble utile de combinateurs de types pour construire de gros “Functor” à partir de plus petits. Ceux-ci sont instructifs en tant qu’exemples d’instances de Functor, et ils sont également utiles en tant que technique de programmation générique, car ils peuvent être utilisés pour représenter une large classe de foncteurs communs.

Le foncteur d’identité #

Le foncteur identité conclut simplement son argument. Il s’agit d’une implémentation au niveau du type du combinateur “I” du calcul SKI.

newtype I a = I a

instance Functor I where
    fmap f (I x) = I (f x)

I se trouve, sous le nom de Identity, dans [le module Data.Functor.Identity](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data- Functor-Identity.html).

Le foncteur constant #

Le foncteur constant ignore son deuxième argument, ne contenant qu’une valeur constante. C’est un analogue au niveau du type de const, le combinateur K du calcul SKI.

newtype K c a = K c

Notez que K c a ne contient aucune valeur a ; K () est isomorphe à Proxy. Cela signifie que l’implémentation K de fmap ne fait aucun mappage du tout !

instance Functor (K c) where
    fmap _ (K c) = K c

K est autrement connu sous le nom de Const, de Data.Functor.Const .

Les foncteurs restants dans cet exemple combinent des foncteurs plus petits en plus grands.

Produits foncteurs #

Le produit foncteur prend une paire de foncteurs et les emballe. C’est analogue à un tuple, sauf que (,) :: * -> * -> * opère sur types *, (:*:) :: (* -> *) -> (* -> *) -> (* -> *) opère sur les foncteurs * -> *.

infixl 7 :*:
data (f :*: g) a = f a :*: g a

instance (Functor f, Functor g) => Functor (f :*: g) where
    fmap f (fx :*: gy) = fmap f fx :*: fmap f gy

Ce type peut être trouvé, sous le nom Product, dans [le module Data.Functor.Product](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data-Functor- Produit.html).

Coproduits fonctionnels #

Tout comme :*: est analogue à (,), :+: est l’analogue au niveau du foncteur de Either.

infixl 6 :+:
data (f :+: g) a = InL (f a) | InR (g a)

instance (Functor f, Functor g) => Functor (f :+: g) where
    fmap f (InL fx) = InL (fmap f fx)
    fmap f (InR gy) = InR (fmap f gy)

:+: se trouve sous le nom Sum, dans [le module Data.Functor.Sum](https://hackage.haskell.org/package/base-4.9.0.0/docs/Data- Functor-Sum.html).

Composition du foncteur #

Enfin, :.: fonctionne comme un niveau de type (.), prenant la sortie d’un foncteur et la plongeant dans l’entrée d’un autre.

infixr 9 :.:
newtype (f :.: g) a = Cmp (f (g a))

instance (Functor f, Functor g) => Functor (f :.: g) where
    fmap f (Cmp fgx) = Cmp (fmap (fmap f) fgx)

Le type Compose se trouve dans Data.Functor.Compose

Foncteurs polynomiaux pour la programmation générique #

I, K, :*:, :+: et :.: peuvent être considérés comme un ensemble de blocs de construction pour une certaine classe de types de données simples. Le kit devient particulièrement puissant lorsque vous le combinez avec [fixed points](https://www.wikiod.com/fr/haskell/schemas-de-recursivite#Points fixes) car les types de données construits avec ces combinateurs sont automatiquement des instances de Functor. Vous utilisez le kit pour créer un type de modèle, en marquant des points récursifs à l’aide de “I”, puis branchez-le dans “Fix” pour obtenir un type pouvant être utilisé avec le zoo standard de schémas de récursivité.

| Nom | En tant que type de données | Utilisation du kit foncteur | |——————————|—————————— ———————————–|———————- ———————————–| | Paires de valeurs | données Paire a = Paire a a | type Pair = I ::*: I | | Grilles deux par deux | type Grille a = Paire (Paire a) | type Grille = Paire :: Paire | | Nombres naturels | données Nat = Zéro | Suc Nat | type Nat = Fix (K () :+: I) | | Listes | liste de données a = Néant | Cons a (Liste a) | type List a = Fix (K () :+: K a :*: I) | | Arbres binaires | Arbre de données a = Feuille | Nœud (Arbre a) a (Arbre a) | type Tree a = Fix (K () :+: I :*: K a :*: I) | | Rosiers | données Rose a = Rose a (Liste (Rose a)) | type Rose a = Fix (K a :*: List ::.: I) |

Cette approche « kit » pour concevoir des types de données est l’idée derrière les bibliothèques de programmation générique telles que generics-sop. L’idée est d’écrire des opérations génériques à l’aide d’un kit comme celui présenté ci-dessus, puis d’utiliser une classe de types pour convertir des types de données arbitraires vers et depuis leur représentation générique :

class Generic a where
    type Rep a  -- a generic representation built using a kit
    to :: a -> Rep a
    from :: Rep a -> a

Les foncteurs en théorie des catégories #

Un foncteur est défini dans la théorie des catégories comme une carte préservant la structure (un «homomorphisme») entre les catégories. Plus précisément, (tous) les objets sont mappés sur des objets et (toutes) les flèches sont mappées sur des flèches, de sorte que les lois de catégorie sont préservées.

La catégorie dans laquelle les objets sont des types Haskell et les morphismes sont des fonctions Haskell est appelée Hask. Ainsi, un foncteur de Hask à Hask consisterait en un mappage de types sur des types et un mappage de fonctions sur des fonctions.

La relation que ce concept théorique des catégories entretient avec la construction de programmation Haskell “Functor” est plutôt directe. Le mappage des types aux types prend la forme d’un type f :: * -> *, et le mappage des fonctions aux fonctions prend la forme d’une fonction fmap :: (a -> b) -> (f a - > f b). En les rassemblant dans une classe,

class Functor (f :: * -> *) where
    fmap :: (a -> b) -> f a -> f b

fmap est une opération qui prend une fonction (un type de morphisme), :: a -> b, et la mappe à une autre fonction, :: f a -> f b. On suppose (mais c’est au programmeur de s’en assurer) que les instances de Functor sont bien des foncteurs mathématiques, préservant la structure catégorielle de Hask :

fmap (id {- :: a -> a -})  ==  id {- :: f a -> f a -}
fmap (h . g)               ==  fmap h . fmap g

fmap soulève une fonction :: a -> b dans une sous-catégorie de Hask d’une manière qui préserve à la fois l’existence de toutes les flèches d’identité et l’associativité de la composition.

La classe Functor encode uniquement les endofoncteurs sur Hask. Mais en mathématiques, les foncteurs peuvent cartographier entre des catégories arbitraires. Un encodage plus fidèle de ce concept ressemblerait à ceci :

class Category c where
    id  :: c i i
    (.) :: c j k -> c i j -> c i k

class (Category c1, Category c2) => CFunctor c1 c2 f where
    cfmap :: c1 a b -> c2 (f a) (f b)

La classe Functor standard est un cas particulier de cette classe dans laquelle les catégories source et cible sont toutes deux Hask. Par exemple,

instance Category (->) where        -- Hask
    id    = \x -> x
    f . g = \x -> f (g x)

instance CFunctor (->) (->) [] where
    cfmap = fmap

Fonction de dérivation #

L’extension de langage DeriveFunctor permet à GHC de générer automatiquement des instances de Functor.

{-# LANGUAGE DeriveFunctor #-}

data List a = Nil | Cons a (List a) deriving Functor

-- instance Functor List where            -- automatically defined
--   fmap f Nil = Nil
--   fmap f (Cons x xs) = Cons (f x) (fmap f xs)

map :: (a -> b) -> List a -> List b
map = fmap
