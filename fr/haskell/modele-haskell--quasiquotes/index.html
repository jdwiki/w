Qu’est-ce que le modèle Haskell ? #

Template Haskell fait référence aux fonctionnalités de méta-programmation de modèles intégrées à GHC Haskell. L’article décrivant l’implémentation originale peut être trouvé [ici][1].

[1] : http://research.microsoft.com/en-us/um/people/simonpj/papers/meta-haskell/

Quelles sont les étapes ? (Ou, quelle est la restriction d’étape ?) #

Les étapes font référence à lorsque le code est exécuté. Normalement, le code n’est exécuté qu’au moment de l’exécution, mais avec Template Haskell, le code peut être exécuté au moment de la compilation. Le code “normal” est l’étape 0 et le code de compilation est l’étape 1.

La restriction d’étape fait référence au fait qu’un programme de l’étape 0 ne peut pas être exécuté à l’étape 1 - cela équivaudrait à pouvoir exécuter n’importe quel programme normal (pas seulement un méta-programme) au moment de la compilation.

Par convention (et par souci de simplicité d’implémentation), le code dans le module actuel est toujours l’étape 0 et le code importé de tous les autres modules est l’étape 1. Pour cette raison, seules les expressions d’autres modules peuvent être épissées.

Notez qu’un programme d’étape 1 est une expression d’étape 0 de type Q Exp, Q Type, etc. mais l’inverse n’est pas vrai - toutes les valeurs (programme de l’étape 0) de type Q Exp ne sont pas un programme de l’étape 1,

De plus, puisque les épissures peuvent être imbriquées, les identifiants peuvent avoir des étapes supérieures à 1. La restriction d’étape peut alors être généralisée - un programme d’étape n ne peut être exécuté dans aucune étape m>n. Par exemple, on peut voir des références à de telles étapes supérieures à 1 dans certains messages d’erreur :

>:t [| \x -> $x |]

<interactive>:1:10: error:
    * Stage error: `x' is bound at stage 2 but used at stage 1
    * In the untyped splice: $x
      In the Template Haskell quotation [| \ x -> $x |]

L’utilisation de Template Haskell provoque des erreurs hors champ à partir d’identifiants non liés ? #

Normalement, toutes les déclarations d’un seul module Haskell peuvent être considérées comme étant mutuellement récursives. En d’autres termes, chaque déclaration de niveau supérieur est dans la portée de toutes les autres dans un seul module. Lorsque Template Haskell est activé, les règles de portée changent - le module est plutôt divisé en groupes de code séparés par des épissures TH, et chaque groupe est mutuellement récursif, et chaque groupe dans la portée de tous les autres groupes.

Syntaxe du modèle Haskell et Quasiquotes #

Le modèle Haskell est activé par l’extension GHC -XTemplateHaskell. Cette extension active toutes les fonctionnalités syntaxiques détaillées dans cette section. Les détails complets sur Template Haskell sont donnés par le [guide de l’utilisateur][1].

Épissures #

  • Une épissure est une nouvelle entité syntaxique activée par Template Haskell, écrite comme $(...), où (...) est une expression.

  • Il ne doit pas y avoir d’espace entre $ et le premier caractère de l’expression ; et Template Haskell remplace l’analyse de l’opérateur $ - par ex. f$g est normalement analysé comme ($) f g alors que lorsque Template Haskell est activé, il est analysé comme une épissure.

  • Lorsqu’une épissure apparaît au niveau supérieur, le $ peut être omis. Dans ce cas, l’expression épissée est la ligne entière.

  • Une épissure représente le code qui est exécuté au moment de la compilation pour produire un Haskell AST, et cet AST est compilé en tant que code Haskell et inséré dans le programme

  • Les épissures peuvent apparaître à la place de : expressions, modèles, types et déclarations de niveau supérieur. Le type de l’expression épissée, dans chaque cas respectivement, est ‘Q Exp’, ‘Q Pat’, ‘Q Type’, ‘Q [Decl]’. Notez que les épissures de déclaration peuvent seulement apparaître au niveau supérieur, alors que les autres peuvent être à l’intérieur d’autres expressions, modèles ou types, respectivement.

Citations d’expression (note : pas une QuasiQuotation) #

  • Une citation d’expression est une nouvelle entité syntaxique écrite comme l’une des suivantes :

  • [e|..|] ou [|..|] - .. est une expression et la citation est de type Q Exp ;

  • [p|..|] - .. est un motif et la citation a le type Q Pat ;

  • [t|..|] - .. est un type et la citation a le type Q Type ;

  • [d|..|] - .. est une liste de déclarations et la citation a le type Q [Dec].

  • Une citation d’expression prend un programme de compilation et produit l’AST représenté par ce programme.

  • L’utilisation d’une valeur dans une citation (par exemple \x -> [| x |]) sans épissure correspond au sucre syntaxique pour \x -> [| $(lift x) |], où lift :: Lift t => t -> Q Exp provient de la classe

    class Lift t where
      lift :: t -> Q Exp
      default lift :: Data t => t -> Q Exp

Épissures et cotations typées #

  • Les épissures typées sont similaires aux épissures mentionnées précédemment (non typées) et sont écrites sous la forme $$(..) où (..) est une expression.

  • Si e a le type Q (TExp a) alors $$e a le type a.

  • Les citations typées prennent la forme [||..||] où .. est une expression de type a ; la citation résultante est de type Q (TEp a).

  • Les expressions typées peuvent être converties en non typées : unType :: TExp a -> Exp.

QuasiCitations #

  • QuasiQuotes généralise les citations d’expressions - auparavant, l’analyseur utilisé par la citation d’expression faisait partie d’un ensemble fixe (e,p,t,d), mais QuasiQuotes permet de définir et d’utiliser un analyseur personnalisé pour produire du code au moment de la compilation . Les quasi-citations peuvent apparaître dans tous les mêmes contextes que les citations ordinaires.

  • Une quasi-guillemet s’écrit [iden|...|], où iden est un identifiant de type Language.Haskell.TH.Quote.QuasiQuoter.

  • Un QuasiQuoter est simplement composé de quatre analyseurs, un pour chacun des différents contextes dans lesquels les citations peuvent apparaître :

    data QuasiQuoter = QuasiQuoter { quoteExp  :: String -> Q Exp,
                                     quotePat  :: String -> Q Pat,
                                     quoteType :: String -> Q Type,
                                     quoteDec  :: String -> Q [Dec] }

Des noms #

  • Les identifiants Haskell sont représentés par le type Language.Haskell.TH.Syntax.Name. Les noms forment les feuilles des arbres de syntaxe abstraite représentant les programmes Haskell dans Template Haskell.

  • Un identifiant qui est actuellement dans la portée peut être transformé en un nom avec soit : 'e ou 'T. Dans le premier cas, e est interprété dans la portée de l’expression, tandis que dans le second cas T est dans la portée du type (en rappelant que les types et les constructeurs de valeur peuvent partager le nom sans ambiguïté dans Haskell).

[1] : https://downloads.haskell.org/~ghc/latest/docs/html/users_guide/glasgow_exts.html#template-haskell

Le type Q #

Le constructeur de type Q :: * -> * défini dans Language.Haskell.TH.Syntax est un type abstrait représentant des calculs qui ont accès à l’environnement de compilation du module dans lequel le calcul est exécuté. Le type Q gère également la substitution de variable, appelée capture de nom par TH (et discutée [ici][1].) Toutes les épissures ont le type Q X pour certains X.

L’environnement de compilation comprend :

  • des identifiants dans le champ d’application et des informations sur ces identifiants,
  • types de fonctions
  • types et types de données source des constructeurs
  • spécification complète des déclarations de type (classes, familles de types)
  • l’emplacement dans le code source (ligne, colonne, module, package) où se produit l’épissure
  • fixités des fonctions (GHC 7.10)
  • extensions GHC activées (GHC 8.0)

Le type Q a également la possibilité de générer de nouveaux noms, avec la fonction newName :: String -> Q Name. Notez que le nom n’est lié nulle part implicitement, donc l’utilisateur doit le lier lui-même, et donc s’assurer que l’utilisation résultante du nom est bien portée est de la responsabilité de l’utilisateur.

Q a des instances pour Functor,Monad,Applicative et c’est l’interface principale pour manipuler les valeurs Q, ainsi que les combinateurs fournis dans Language.Haskell.TH.Lib, qui définissent une fonction d’assistance pour chaque constructeur du TH de la forme :

LitE :: Lit -> Exp
litE :: Lit -> ExpQ

AppE :: Exp -> Exp -> Exp 
appE :: ExpQ -> ExpQ -> ExpQ

Notez que ExpQ, TypeQ, DecsQ et PatQ sont des synonymes des types AST qui sont généralement stockés dans le type Q.

La bibliothèque TH fournit une fonction runQ :: Quasi m => Q a -> m a, et il y a une instance Quasi IO, il semblerait donc que le type Q soit juste un IO fantaisiste. Cependant, l’utilisation de runQ :: Q a -> IO a produit une action IO qui n’a pas accès à un environnement de compilation - ceci n’est disponible que dans le type Q réel. De telles actions “IO” échoueront lors de l’exécution si vous essayez d’accéder audit environnement.

[1] : https://hackage.haskell.org/package/template-haskell-2.10.0.0/docs/Language-Haskell-TH-Syntax.html#namecapture

Un curry n-arité #

Le familier

curry :: ((a,b) -> c) -> a -> b -> c
curry = \f a b -> f (a,b)

La fonction peut être généralisée à des tuples d’arité arbitraire, par exemple :

curry3 :: ((a, b, c) -> d) -> a -> b -> c -> d
curry4 :: ((a, b, c, d) -> e) -> a -> b -> c -> d -> e

Cependant, écrire à la main de telles fonctions pour des tuples d’arité 2 à (par exemple) 20 serait fastidieux (et ignorer le fait que la présence de 20 tuples dans votre programme signale presque certainement des problèmes de conception qui doivent être résolus avec des enregistrements).

Nous pouvons utiliser Template Haskell pour produire de telles fonctions curryN pour un n arbitraire :

{-# LANGUAGE TemplateHaskell #-}
import Control.Monad (replicateM) 
import Language.Haskell.TH (ExpQ, newName, Exp(..), Pat(..))
import Numeric.Natural (Natural) 

curryN :: Natural -> Q Exp

La fonction curryN prend un nombre naturel et produit la fonction de curry de cette arité, sous la forme d’un AST de Haskell.

curryN n = do
  f  <- newName "f"
  xs <- replicateM (fromIntegral n) (newName "x")

Nous produisons d’abord des variables de type frais pour chacun des arguments de la fonction - une pour la fonction d’entrée et une pour chacun des arguments de ladite fonction.

  let args = map VarP (f:xs)

L’expression args représente le motif f x1 x2 .. xn. Notez qu’un motif est une entité syntaxique distincte - nous pourrions prendre ce même motif et le placer dans un lambda, ou une liaison de fonction, ou même le LHS d’une liaison let (ce qui serait une erreur).

      ntup = TupE (map VarE xs)

La fonction doit construire le tuple d’argument à partir de la séquence d’arguments, ce qui est ce que nous avons fait ici. Notez la distinction entre les variables de modèle (VarP) et les variables d’expression (VarE).

  return $ LamE args (AppE (VarE f) ntup)

Enfin, la valeur que nous produisons est l’AST \f x1 x2 .. xn -> f (x1, x2, .. , xn).

Nous aurions également pu écrire cette fonction en utilisant des guillemets et des constructeurs “levés”:

...
import Language.Haskell.TH.Lib  

curryN' :: Natural -> ExpQ
curryN' n = do
  f  <- newName "f"
  xs <- replicateM (fromIntegral n) (newName "x")
  lamE (map varP (f:xs)) 
        [| $(varE f) $(tupE (map varE xs)) |]

Notez que les citations doivent être syntaxiquement valides, donc [| \ $(map varP (f:xs)) -> .. |] n’est pas valide, car il n’y a aucun moyen dans Haskell normal de déclarer une ’liste’ de motifs - ce qui précède est interprété comme \ var -> .. et l’expression épissée devrait avoir le type PatQ, c’est-à-dire un seul motif, pas une liste de motifs.

Enfin, nous pouvons charger cette fonction TH dans GHCi :

>:set -XTemplateHaskell
>:t $(curryN 5)
$(curryN 5)
  :: ((t1, t2, t3, t4, t5) -> t) -> t1 -> t2 -> t3 -> t4 -> t5 -> t
>$(curryN 5) (\(a,b,c,d,e) -> a+b+c+d+e) 1 2 3 4 5
15

Cet exemple est principalement adapté de [ici][1].

[1] : https://wiki.haskell.org/A_practical_Template_Haskell_Tutorial
