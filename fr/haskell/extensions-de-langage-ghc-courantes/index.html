Ces extensions de langage sont généralement disponibles lors de l’utilisation du compilateur Glasgow Haskell (GHC) car elles ne font pas partie du [Haskell 2010 language Report][1] approuvé. Pour utiliser ces extensions, il faut soit informer le compilateur en utilisant un [flag][4] ou placer [un programme LANGUAGEa][2] avant le mot clé module dans un fichier. La documentation officielle se trouve dans la [section 7] [3] du guide de l’utilisateur GCH.

Le format du programme LANGUAGE est {-# LANGUAGE ExtensionOne, ExtensionTwo ... #-}. Il s’agit du littéral {-# suivi de LANGUAGE suivi d’une liste d’extensions séparées par des virgules, et enfin du #-} fermant. Plusieurs programmes LANGUAGE peuvent être placés dans un seul fichier.

[1] : https://www.haskell.org/onlinereport/haskell2010/ [2] : https://downloads.haskell.org/~ghc/7.2.2/docs/html/users_guide/pragmas.html [3] : https://downloads.haskell.org/~ghc/7.2.2/docs/html/users_guide/ghc-language-features.html [4] : https://downloads.haskell.org/~ghc/7.2.2/docs/html/users_guide/flag-reference.html

Types de classement #

Imaginez la situation suivante :

foo :: Show a => (a -> String) -> String -> Int -> IO ()
foo show' string int = do
   putStrLn (show' string)
   putStrLn (show' int)

Ici, nous voulons transmettre une fonction qui convertit une valeur en une chaîne, appliquer cette fonction à la fois à un paramètre de chaîne et à un paramètre int et les imprimer tous les deux. Dans mon esprit, il n’y a aucune raison pour que cela échoue ! Nous avons une fonction qui fonctionne sur les deux types de paramètres que nous transmettons.

Malheureusement, cela ne vérifiera pas le type ! GHC déduit le type “a” en fonction de sa première occurrence dans le corps de la fonction. Autrement dit, dès que nous frappons :

putStrLn (show' string)

GHC déduira que show' :: String -> String, puisque string est une String. Il continuera à exploser en essayant de “montrer” int “.

RankNTypes vous permet à la place d’écrire la signature de type comme suit, en quantifiant toutes les fonctions qui satisfont le type show' :

foo :: (forall a. Show a => (a -> String)) -> String -> Int -> IO ()

C’est le polymorphisme de rang 2 : nous affirmons que la fonction show' doit fonctionner pour tous les a dans notre fonction, et l’implémentation précédente fonctionne maintenant.

L’extension RankNTypes permet l’imbrication arbitraire de blocs forall ... dans les signatures de type. En d’autres termes, il permet le polymorphisme de rang N.

Chaînes surchargées #

Normalement, les littéraux de chaîne dans Haskell ont un type de String (qui est un alias de type pour [Char]). Bien que ce ne soit pas un problème pour les petits programmes éducatifs, les applications du monde réel nécessitent souvent un stockage plus efficace tel que Text ou ByteString.

OverloadedStrings change simplement le type de littéraux en

"test" :: Data.String.IsString a => a

Cela leur permet d’être directement passés aux fonctions qui attendent un tel type. De nombreuses bibliothèques implémentent cette interface pour leurs types de type chaîne, notamment [Data.Text][2] et [Data.ByteString][3], qui offrent tous deux certains avantages temporels et spatiaux par rapport à [Char].

Il existe également des utilisations uniques de OverloadedStrings comme celles de la bibliothèque [Postgresql-simple] [4] qui permet aux requêtes SQL d’être écrites entre guillemets doubles comme une chaîne normale, mais fournit des protections contre une concaténation incorrecte, une source notoire de SQL attaques par injection.

Pour créer une instance de la classe IsString, vous devez implémenter la fonction fromString. Exemple† :

data Foo = A | B | Other String deriving Show

instance IsString Foo where
  fromString "A" = A
  fromString "B" = B
  fromString xs  = Other xs

tests :: [ Foo ]
tests = [ "A", "B", "Testing" ]

† Cet exemple est une gracieuseté de Lyndon Maydwell (sordina sur GitHub) trouvé [ici][1].

[1] : https://gist.github.com/sordina/5714390 [2] : https://www.wikiod.com/fr/haskell/donneestexte [3] : https://hackage.haskell.org/package/bytestring-0.10.8.1/docs/Data-ByteString.html [4] : http://hackage.haskell.org/package/postgresql-simple

BinaryLiterals #

Haskell standard vous permet d’écrire des littéraux entiers en décimal (sans aucun préfixe), hexadécimal (précédé de 0x ou 0X) et octal (précédé de 0o ou 0O). L’extension BinaryLiterals ajoute l’option binaire (précédé de 0b ou 0B).

0b1111 == 15     -- evaluates to: True

##Quantification Existentielle Il s’agit d’une extension de système de type qui autorise les types qui sont quantifiés existentiellement, ou, en d’autres termes, qui ont des variables de type qui ne sont instanciées qu’au moment de l’exécution†.

Une valeur de type existentiel est similaire à une référence de classe de base abstraite dans les langages OO : vous ne connaissez pas le type exact dans contains, mais vous pouvez contraindre la classe des types.

data S = forall a. Show a => S a

ou de manière équivalente, avec la syntaxe GADT :

{-# LANGUAGE GADTs #-}
data S where
   S :: Show a => a -> S

Les types existentiels ouvrent la porte à des choses comme des conteneurs presque hétérogènes : comme dit ci-dessus, il peut en fait y avoir différents types dans une valeur S, mais tous peuvent être shown, donc vous pouvez aussi faire

instance Show S where
    show (S a) = show a   -- we rely on (Show a) from the above

Nous pouvons maintenant créer une collection de tels objets :

ss = [S 5, S "test", S 3.0]

Ce qui nous permet aussi d’utiliser le comportement polymorphe :

mapM_ print ss

Les existentiels peuvent être très puissants, mais notez qu’ils ne sont en fait pas très souvent nécessaires dans Haskell. Dans l’exemple ci-dessus, tout ce que vous pouvez réellement faire avec l’instance Show est d’afficher (duh !) les valeurs, c’est-à-dire de créer une représentation sous forme de chaîne. Le type S entier contient donc exactement autant d’informations que la chaîne que vous obtenez en l’affichant. Par conséquent, il est généralement préférable de stocker simplement cette chaîne tout de suite, d’autant plus que Haskell est paresseux et que, par conséquent, la chaîne ne sera de toute façon qu’un thunk non évalué.

D’un autre côté, les existentiels causent des problèmes uniques. Par exemple, la façon dont les informations de type sont « cachées » dans un existentiel. Si vous faites correspondre un motif sur une valeur S, vous aurez le type contenu dans la portée (plus précisément, son instance Show), mais cette information ne pourra jamais échapper à sa portée, qui devient donc un peu une “société secrète”. ” : le compilateur ne laisse rien échapper à la portée, sauf les valeurs dont le type est déjà connu de l’extérieur. Cela peut conduire à des erreurs étranges [comme Couldn't match type 'a0' with '()' 'a0' is untouchable](http://stackoverflow.com/questions/28582210/type-inference-with-gadts- a0-est-intouchable).

†Comparez cela avec le polymorphisme paramétrique ordinaire, qui est généralement résolu au moment de la compilation (permettant l’effacement complet du type).

Les types existentiels sont différents des types Rank-N - ces extensions sont, grosso modo, duales les unes par rapport aux autres : pour utiliser réellement des valeurs d’un type existentiel, vous avez besoin d’une fonction polymorphe (éventuellement contrainte), comme “show” dans l’exemple. Une fonction polymorphe est universellement quantifiée, c’est-à-dire qu’elle fonctionne pour n’importe quel type dans une classe donnée, alors que la quantification existentielle signifie qu’elle fonctionne pour un type particulier qui est a priori inconnu. Si vous avez une fonction polymorphe, c’est suffisant, cependant pour passer des fonctions polymorphes telles quelles comme arguments, vous avez besoin de {-# LANGUAGE Rank2Types #-} :

genShowSs :: (∀ x . Show x => x -> String) -> [S] -> [String]
genShowSs f = map (\(S a) -> f a)

LambdaCase #

Une extension syntaxique qui vous permet d’écrire \case à la place de \arg -> case arg of.

Considérez la définition de fonction suivante :

dayOfTheWeek :: Int -> String
dayOfTheWeek 0 = "Sunday"
dayOfTheWeek 1 = "Monday"
dayOfTheWeek 2 = "Tuesday"
dayOfTheWeek 3 = "Wednesday"
dayOfTheWeek 4 = "Thursday"
dayOfTheWeek 5 = "Friday"
dayOfTheWeek 6 = "Saturday"

Si vous voulez éviter de répéter le nom de la fonction, vous pouvez écrire quelque chose comme :

dayOfTheWeek :: Int -> String
dayOfTheWeek i = case i of
    0 -> "Sunday"
    1 -> "Monday"
    2 -> "Tuesday"
    3 -> "Wednesday"
    4 -> "Thursday"
    5 -> "Friday"
    6 -> "Saturday"

À l’aide de l’extension LambdaCase, vous pouvez écrire cela en tant qu’expression de fonction, sans avoir à nommer l’argument :

{-# LANGUAGE LambdaCase #-}

dayOfTheWeek :: Int -> String
dayOfTheWeek = \case
    0 -> "Sunday"
    1 -> "Monday"
    2 -> "Tuesday"
    3 -> "Wednesday"
    4 -> "Thursday"
    5 -> "Friday"
    6 -> "Saturday"

Dépendances fonctionnelles #

Si vous avez une classe de types multi-paramètres avec les arguments a, b, c et x, cette extension vous permet d’exprimer que le type x peut être identifié de manière unique à partir de a, b et c :

class SomeClass a b c x | a b c -> x where ...

Lors de la déclaration d’une instance d’une telle classe, elle sera vérifiée par rapport à toutes les autres instances pour s’assurer que la dépendance fonctionnelle tient, c’est-à-dire qu’aucune autre instance avec le même a b c mais un x différent n’existe.

Vous pouvez spécifier plusieurs dépendances dans une liste séparée par des virgules :

class OtherClass a b c d | a b -> c d, a d -> b where ...

Par exemple en MTL on peut voir :

class MonadReader r m| m -> r where ...
instance MonadReader r ((->) r) where ...

Maintenant, si vous avez une valeur de type MonadReader a ((->) Foo) => a, le compilateur peut déduire que a ~ Foo, puisque le deuxième argument détermine complètement le premier, et simplifiera le type en conséquence .

La classe SomeClass peut être considérée comme une fonction des arguments a b c qui donne x. De telles classes peuvent être utilisées pour effectuer des calculs dans le système de types.

## instances flexibles Les instances standards nécessitent :

All instance types must be of the form (T a1 ... an)
where a1 ... an are *distinct type variables*,
and each type variable appears at most once in the instance head.

Cela signifie que, par exemple, bien que vous puissiez créer une instance pour [a], vous ne pouvez pas créer une instance spécifiquement pour [Int].; FlexibleInstances assouplit que :

class C a where

-- works out of the box
instance C [a] where

-- requires FlexibleInstances
instance C [Int] where

GADT #

Les types de données algébriques conventionnels sont paramétriques dans leurs variables de type. Par exemple, si nous définissons un ADT comme

data Expr a = IntLit Int 
            | BoolLit Bool 
            | If (Expr Bool) (Expr a) (Expr a)

dans l’espoir que cela exclura statiquement les conditions non bien typées, cela ne se comportera pas comme prévu puisque le type de IntLit :: Int -> Expr a est universellement quantifié : pour tout choix de a, il produit une valeur de type Expr a. En particulier, pour a ~ Bool, nous avons IntLit :: Int -> Expr Bool, nous permettant de construire quelque chose comme If (IntLit 1) e1 e2 qui correspond au type du constructeur If essayant d’exclure.

Les types de données algébriques généralisés nous permettent de contrôler le type résultant d’un constructeur de données afin qu’ils ne soient pas simplement paramétriques. Nous pouvons réécrire notre type Expr en tant que GADT comme ceci :

data Expr a where
  IntLit :: Int -> Expr Int
  BoolLit :: Bool -> Expr Bool
  If :: Expr Bool -> Expr a -> Expr a -> Expr a

Ici, le type du constructeur IntLit est Int -> Expr Int, et donc IntLit 1 :: Expr Bool ne sera pas typé.

La correspondance de modèle sur une valeur GADT entraîne un raffinement du type du terme renvoyé. Par exemple, il est possible d’écrire un évaluateur pour Expr a comme ceci :

crazyEval :: Expr a -> a
crazyEval (IntLit x) = 
   -- Here we can use `(+)` because x :: Int
   x + 1 
crazyEval (BoolLit b) = 
   -- Here we can use `not` because b :: Bool
   not b
crazyEval (If b thn els) = 
  -- Because b :: Expr Bool, we can use `crazyEval b :: Bool`.
  -- Also, because thn :: Expr a and els :: Expr a, we can pass either to 
  -- the recursive call to `crazyEval` and get an a back
  crazyEval $ if crazyEval b then thn else els

Notez que nous pouvons utiliser (+) dans les définitions ci-dessus car, par ex. IntLit x correspond à un motif, nous apprenons également que a ~ Int (et de même pour not et if_then_else_ quand a ~ Bool).

TupleSections #

Une extension syntaxique qui permet d’appliquer le constructeur de tuple (qui est un opérateur) de manière section :

(a,b) == (,) a b

-- With TupleSections
(a,b) == (,) a b == (a,) b == (,b) a

N-uplets #

Cela fonctionne également pour les tuples avec une arité supérieure à deux

(,2,) 1 3 == (1,2,3)

Cartographie #

Cela peut être utile dans d’autres endroits où les sections sont utilisées :

map (,"tag") [1,2,3] == [(1,"tag"), (2, "tag"), (3, "tag")]

L’exemple ci-dessus sans cette extension ressemblerait à ceci :

map (\a -> (a, "tag")) [1,2,3]

Listes surchargées #

ajouté dans GHC 7.8.

OverloadedLists, similaire à [OverloadedStrings](https://www.wikiod.com/fr/haskell/extensions-de-langage-ghc-courantes#Chaînes surchargées), permet de décoder les littéraux de liste comme suit :

[]          -- fromListN 0 []
[x]         -- fromListN 1 (x : [])
[x .. ]     -- fromList (enumFrom x)

Cela est pratique lorsqu’il s’agit de types tels que Set, Vector et Maps.

['0' .. '9']             :: Set Char
[1 .. 10]                :: Vector Int
[("default",0), (k1,v1)] :: Map String Int
['a' .. 'z']             :: Text

La classe IsList dans GHC.Exts est destinée à être utilisée avec ce extension.

IsList est équipé d’une fonction de type, Item, et de trois fonctions, fromList :: [Item l] -> l, toList :: l -> [Item l] et fromListN :: Int - > [Item l] -> l où fromListN est facultatif. Les implémentations typiques sont :

instance IsList [a] where
  type Item [a] = a
  fromList = id
  toList   = id

instance (Ord a) => IsList (Set a) where
  type Item (Set a) = a
  fromList = Set.fromList
  toList   = Set.toList

Exemples tirés de OverloadedLists – GHC.

MultiParamTypeClasses #

C’est une extension très courante qui autorise les classes de type avec plusieurs paramètres de type. Vous pouvez considérer MPTC comme une relation entre les types.

{-# LANGUAGE MultiParamTypeClasses #-}

class Convertable a b where
    convert :: a -> b

instance Convertable Int Float where
    convert i = fromIntegral i

L’ordre des paramètres est important.

Les MPTC peuvent parfois être remplacés par des familles de types.

Syntaxe Unicode #

Une extension qui vous permet d’utiliser des caractères Unicode à la place de certains opérateurs et noms intégrés.

ASCII Unicode Les usages)
:: ∷ a le type
-> → types de fonctions, lambdas, branches case, etc.
=> ⇒ contraintes de classe
pourtout ∀ polymorphisme explicite
<- ← Notation “faire”
* ★ le type (ou kind) des types (par exemple, Int :: ★)
>- ⤚ [notation proc] pour [Flèches]
-< ⤙ [notation proc] pour [Flèches]
>>- ⤜ [notation proc] pour [Flèches]
-<< ⤛ [notation proc] pour [Flèches]

Par exemple:

runST :: (forall s. ST s a) -> a

deviendrait

runST ∷ (∀ s. ST s a) → a

Notez que l’exemple * vs. ★ est légèrement différent : puisque * n’est pas réservé, ★ fonctionne également de la même manière que * pour la multiplication, ou toute autre fonction nommée (*) , et vice versa. Par exemple:

ghci> 2 ★ 3
6
ghci> let (*) = (+) in 2 ★ 3
5

MotifSynonymes #

Les synonymes de modèles sont des abstractions de modèles similaires à la façon dont les fonctions sont des abstractions d’expressions.

Pour cet exemple, regardons l’interface Data.Sequence expose, et voyons comment il peut être amélioré avec des synonymes de motifs. Le type Seq est un type de données qui, en interne, utilise un représentation] (http://staff.city.ac.uk/~ross/papers/FingerTree.html) pour obtenir une bonne complexité asymptotique pour diverses opérations, notamment O(1) (un)consing et (un)snocing.

Mais cette représentation est peu maniable et certains de ses invariants ne peuvent pas être exprimés dans le système de types de Haskell. Pour cette raison, le type “Seq” est exposé aux utilisateurs en tant que type abstrait, avec des fonctions d’accesseur et de constructeur préservant les invariants, parmi lesquelles :

empty :: Seq a

(<|) :: a -> Seq a -> Seq a
data ViewL a = EmptyL | a :< (Seq a)
viewl :: Seq a -> ViewL a

(|>) :: Seq a -> a -> Seq a 
data ViewR a = EmptyR | (Seq a) :> a 
viewr :: Seq a -> ViewR a

Mais l’utilisation de cette interface peut être un peu lourde :

uncons :: Seq a -> Maybe (a, Seq a)
uncons xs = case viewl xs of
    x :< xs' -> Just (x, xs')
    EmptyL -> Nothing

Nous pouvons utiliser view patterns pour le nettoyer quelque peu :

{-# LANGUAGE ViewPatterns #-}

uncons :: Seq a -> Maybe (a, Seq a)
uncons (viewl -> x :< xs) = Just (x, xs)
uncons _ = Nothing

En utilisant l’extension de langage PatternSynonyms, nous pouvons donner une interface encore plus agréable en permettant à la correspondance de motifs de prétendre que nous avons une contre- ou une snoc-list :

{-# LANGUAGE PatternSynonyms #-}
import Data.Sequence (Seq)
import qualified Data.Sequence as Seq

pattern Empty :: Seq a
pattern Empty <- (Seq.viewl -> Seq.EmptyL)

pattern (:<) :: a -> Seq a -> Seq a
pattern x :< xs <- (Seq.viewl -> x Seq.:< xs)

pattern (:>) :: Seq a -> a -> Seq a
pattern xs :> x <- (Seq.viewr -> xs Seq.:> x)

Cela nous permet d’écrire uncons dans un style très naturel :

uncons :: Seq a -> Maybe (a, Seq a)
uncons (x :< xs) = Just (x, xs)
uncons _ = Nothing

ScopedTypeVariables #

ScopedTypeVariables vous permet de faire référence à des types universellement quantifiés à l’intérieur d’une déclaration. Pour être plus explicite :

import Data.Monoid

foo :: forall a b c. (Monoid b, Monoid c) => (a, b, c) -> (b, c) -> (a, b, c)
foo (a, b, c) (b', c') = (a :: a, b'', c'')
    where (b'', c'') = (b <> b', c <> c') :: (b, c)

La chose importante est que nous pouvons utiliser a, b et c pour instruire le compilateur dans les sous-expressions de la déclaration (le tuple dans la clause where et le premier a dans le résultat final). En pratique, ScopedTypeVariables aide à écrire des fonctions complexes comme une somme de parties, permettant au programmeur d’ajouter des signatures de type à des valeurs intermédiaires qui n’ont pas de types concrets.
