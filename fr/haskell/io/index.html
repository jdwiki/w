Obtenir le ‘a’ “hors de” ‘IO a’ #

Une question courante est “J’ai une valeur de IO a, mais je veux faire quelque chose à cette valeur a : comment puis-je y accéder?” Comment opérer sur des données qui viennent de l’extérieur (par exemple, incrémenter un nombre tapé par l’utilisateur) ?

Le fait est que si vous utilisez une fonction pure sur des données obtenues de manière impure, le résultat est toujours impur. Cela dépend de ce que l’utilisateur a fait ! Une valeur de type IO a représente un “calcul à effet secondaire résultant en une valeur de type a” qui peut uniquement être exécuté en (a) le composant dans main et (b) en compilant et en exécutant votre programme. Pour cette raison, il n’y a aucun moyen dans le monde Haskell pur de “faire sortir le ‘a’”.

Au lieu de cela, nous voulons construire un nouveau calcul, une nouvelle valeur IO, qui utilise la valeur a au moment de l’exécution. C’est une autre façon de composer les valeurs IO et donc encore une fois nous pouvons utiliser la notation do :

-- assuming
myComputation :: IO Int

getMessage :: Int -> String
getMessage int = "My computation resulted in: " ++ show int
 
newComputation :: IO ()
newComputation = do
  int <- myComputation       -- we "bind" the result of myComputation to a name, 'int'
  putStrLn $ getMessage int   -- 'int' holds a value of type Int

Ici, nous utilisons une fonction pure (getMessage) pour transformer un Int en une String, mais nous utilisons la notation do pour l’appliquer au résultat d’un calcul IO myComputation quand (après) que le calcul s’exécute. Le résultat est un calcul IO plus important, newComputation. Cette technique d’utilisation de fonctions pures dans un contexte impur est appelée lifting.

## IO définit l’action “principale” de votre programme Pour rendre un programme Haskell exécutable, vous devez fournir un fichier avec une fonction main de type IO ()

main :: IO ()
main = putStrLn "Hello world!"

Lorsque Haskell est compilé, il examine les données “IO” ici et les transforme en un exécutable. Lorsque nous exécutons ce programme, il affichera Hello world!.

Si vous avez des valeurs de type IO a autres que main elles ne feront rien.

other :: IO ()
other = putStrLn "I won't get printed"

main :: IO ()
main = putStrLn "Hello world!"

Compiler ce programme et l’exécuter aura le même effet que le dernier exemple. Le code dans other est ignoré.

Pour que le code dans other ait des effets d’exécution, vous devez le composer dans main. Aucune valeur IO qui n’est pas finalement composée dans main n’aura aucun effet sur l’exécution. Pour composer deux valeurs IO séquentiellement, vous pouvez utiliser la notation do :

other :: IO ()
other = putStrLn "I will get printed... but only at the point where I'm composed into main"

main :: IO ()
main = do 
  putStrLn "Hello world!"
  other

Lorsque vous compilez et exécutez * ce * programme, il affiche

Hello world!
I will get printed... but only at the point where I'm composed into main

Notez que l’ordre des opérations est décrit par la façon dont other a été composé dans main et non par l’ordre de définition.

Vérification des conditions de fin de fichier #

Un peu contre-intuitif par rapport à la façon dont la plupart des bibliothèques d’E/S standard des autres langages le font, isEOF de Haskell ne vous oblige pas à effectuer une opération de lecture avant de vérifier une condition EOF ; le runtime le fera pour vous.

import System.IO( isEOF )


eofTest :: Int -> IO ()
eofTest line = do
    end <- isEOF
    if end then
        putStrLn $ "End-of-file reached at line " ++ show line ++ "."
    else do
        getLine
        eofTest $ line + 1


main :: IO ()
main =
    eofTest 1

Saisir:

Line #1.
Line #2.
Line #3.

Production:

End-of-file reached at line 4.

Lecture de tout le contenu de l’entrée standard dans une chaîne #

main = do
    input <- getContents
    putStr input

Saisir:

This is an example sentence.
And this one is, too!

Production:

This is an example sentence.
And this one is, too!

Remarque : ce programme imprimera en fait des parties de la sortie avant que toutes les entrées aient été entièrement lues. Cela signifie que, si, par exemple, vous utilisez getContents sur un fichier de 50 Mo, l’évaluation paresseuse et le ramasse-miettes de Haskell s’assureront que seules les parties du fichier qui sont actuellement nécessaires (lire : indispensables pour une exécution ultérieure) seront chargées en mémoire. Ainsi, le fichier de 50 Mo ne sera pas chargé en mémoire immédiatement.

Rôle et objectif de l’OI #

Haskell est un langage pur, ce qui signifie que les expressions ne peuvent pas avoir d’effets secondaires. Un effet secondaire est tout ce que l’expression ou la fonction fait autre que produire une valeur, par exemple, modifier un compteur global ou imprimer sur la sortie standard.

Dans Haskell, les calculs à effet secondaire (en particulier, ceux qui peuvent avoir un effet sur le monde réel) sont modélisés à l’aide de IO. Strictement parlant, IO est un constructeur de type, prenant un type et produisant un type. Par exemple, IO Int est le type d’un calcul d’E/S produisant une valeur Int. Le type IO est abstract, et l’interface fournie pour IO garantit que certaines valeurs illégales (c’est-à-dire des fonctions avec des types non sensiques) ne peuvent pas exister, en s’assurant que toutes les fonctions intégrées qui exécutent IO ont un type de retour inclus dans IO.

Lorsqu’un programme Haskell est exécuté, le calcul représenté par la valeur Haskell nommée ‘main’, dont le type peut être ‘IO x’ pour n’importe quel type ‘x’, est exécuté.

Manipulation des valeurs IO #

Il existe de nombreuses fonctions dans la bibliothèque standard fournissant des actions “E/S” typiques qu’un langage de programmation à usage général devrait effectuer, telles que la lecture et l’écriture dans des descripteurs de fichiers. Les actions générales “IO” sont créées et combinées principalement avec deux fonctions :

 (>>=) :: IO a -> (a -> IO b) -> IO b

Cette fonction (généralement appelée bind) prend une action “IO” et une fonction qui renvoie une action “IO”, et produit l’action “IO” qui est le résultat de l’application de la fonction à la valeur produite par le premier “IO”. `action.

 return :: a -> IO a

Cette fonction prend n’importe quelle valeur (c’est-à-dire une valeur pure) et renvoie le calcul IO qui ne fait pas d’IO et produit la valeur donnée. En d’autres termes, il s’agit d’une action d’E/S sans opération.

Il existe des fonctions générales supplémentaires qui sont souvent utilisées, mais toutes peuvent être écrites en termes des deux ci-dessus. Par exemple, (>>) :: IO a -> IO b -> IO b est similaire à (>>=) mais le résultat de la première action est ignoré.

Un programme simple saluant l’utilisateur à l’aide de ces fonctions :

 main :: IO ()
 main =
   putStrLn "What is your name?" >>
   getLine >>= \name ->
   putStrLn ("Hello " ++ name ++ "!")

Ce programme utilise également putStrLn :: String -> IO () et getLine :: IO String.

Remarque : les types de certaines fonctions ci-dessus sont en fait plus généraux que les types donnés (à savoir >>=, >> et return).

Sémantique IO #

Le type “IO” dans Haskell a une sémantique très similaire à celle des langages de programmation impératifs. Par exemple, quand on écrit s1 ; s2 dans un langage impératif pour indiquer l’exécution de l’instruction s1, puis de l’instruction s2, on peut écrire s1 >> s2 pour modéliser la même chose en Haskell.

Cependant, la sémantique de IO s’écarte légèrement de ce à quoi on pourrait s’attendre venant d’un arrière-plan impératif. La fonction return n’interrompt pas le flux de contrôle - elle n’a aucun effet sur le programme si une autre action IO est exécutée en séquence. Par exemple, return () >> putStrLn "boom" imprime correctement “boom” sur la sortie standard.

La sémantique formelle de IO peut être donnée en termes d’égalités simples impliquant les fonctions de la section précédente :

 return x >>= f ≡ f x, ∀ f x
 y >>= return ≡ return y, ∀ y
 (m >>= f) >>= g ≡ m >>= (\x -> (f x >>= g)), ∀ m f g

Ces lois sont généralement appelées identité gauche, identité droite et composition, respectivement. On peut les énoncer plus naturellement en fonction de la fonction

 (>=>) :: (a -> IO b) -> (b -> IO c) -> a -> IO c
 (f >=> g) x = (f x) >>= g

comme suit:

 return >=> f ≡ f, ∀ f
 f >=> return ≡ f, ∀ f
 (f >=> g) >=> h ≡ f >=> (g >=> h), ∀ f g h

E/S paresseux #

Les fonctions effectuant des calculs d’E/S sont généralement strictes, ce qui signifie que toutes les actions précédentes dans une séquence d’actions doivent être terminées avant que l’action suivante ne commence. C’est généralement un comportement utile et attendu - putStrLn "X" >> putStrLn "Y" devrait afficher “XY”. Cependant, certaines fonctions de la bibliothèque exécutent les E/S paresseusement, ce qui signifie que les actions d’E/S requises pour produire la valeur ne sont exécutées que lorsque la valeur est réellement consommée. Des exemples de telles fonctions sont getContents et readFile. Les E/S paresseuses peuvent réduire considérablement les performances d’un programme Haskell, donc lors de l’utilisation des fonctions de la bibliothèque, il faut prendre soin de noter quelles fonctions sont paresseuses.

Notation IO et do #

Haskell fournit une méthode plus simple pour combiner différentes valeurs d’E/S en valeurs d’E/S plus grandes. Cette syntaxe spéciale est connue sous le nom de notation do* et est simplement du sucre syntaxique pour les utilisations des fonctions >>=, >> et return.

Le programme de la section précédente peut être écrit de deux manières différentes en utilisant la notation “do”, la première étant sensible à la mise en page et la seconde insensible à la mise en page :

 main = do
   putStrLn "What is your name?"
   name <- getLine
   putStrLn ("Hello " ++ name ++ "!")


 main = do {
   putStrLn "What is your name?" ;
   name <- getLine ;
   putStrLn ("Hello " ++ name ++ "!")
   }

Les trois programmes sont exactement équivalents.

*Notez que la notation « do » s’applique également à une classe plus large de constructeurs de types appelés monads.

Lecture d’une ligne depuis l’entrée standard #

Lire des mots d’un fichier entier #

Dans Haskell, il est souvent logique de ne pas s’embêter avec les descripteurs de fichiers, mais simplement de lire ou d’écrire un fichier entier directement du disque vers la mémoire†, et de faire tout le partitionnement/traitement du texte avec le structure de données de chaîne pure. Cela évite de mélanger les E/S et la logique du programme, ce qui peut grandement aider à éviter les bogues.

-- | The interesting part of the program, which actually processes data
--   but doesn't do any IO!
reverseWords :: String -> [String]
reverseWords = reverse . words

-- | A simple wrapper that only fetches the data from disk, lets
--   'reverseWords' do its job, and puts the result to stdout.
main :: IO ()
main = do
   content <- readFile "loremipsum.txt"
   mapM_ putStrLn $ reverseWords content

Si loremipsum.txt contient

Lorem ipsum dolor sit amet,
consectetur adipiscing elit

alors le programme affichera

elit
adipiscing
consectetur
amet,
sit
dolor
ipsum
Lorem

Ici, mapM_ a parcouru la liste de tous les mots du fichier, et imprimé chacun d’eux sur une ligne séparée avec putStrLn.

†Si vous pensez que c’est un gaspillage de mémoire, vous avez raison. En fait, la paresse de Haskell peut souvent éviter que le fichier entier doive résider simultanément en mémoire… mais attention, ce type d’E/S paresseux cause son propre ensemble de problèmes. Pour les applications critiques en termes de performances, il est souvent judicieux d’imposer la lecture complète du fichier en une seule fois, de manière stricte ; vous pouvez le faire avec [la version Data.Text de readFile.](http://hackage.haskell.org/package/text-1.2.2.1/docs/Data-Text-IO.html#v : lireFichier)

Écrire sur stdout #

Selon la [Haskell 2010 Language Specification][1], les fonctions d’E/S standard suivantes sont disponibles dans Prelude, aucune importation n’est donc nécessaire pour les utiliser.

putChar :: Char -> IO () - écrit un char dans stdout #

Prelude> putChar 'a'
aPrelude>  -- Note, no new line

putStr :: String -> IO () - écrit une String dans stdout #

Prelude> putStr "This is a string!"
This is a string!Prelude>  -- Note, no new line

putStrLn :: String -> IO () - écrit une String dans stdout et ajoute une nouvelle ligne #

Prelude> putStrLn "Hi there, this is another String!"
Hi there, this is another String!

print :: Show a => a -> IO () - écrit a une instance de Show sur stdout #

Prelude> print "hi"
"hi"
Prelude> print 1
1
Prelude> print 'a'
'a'
Prelude> print (Just 'a')  -- Maybe is an instance of the `Show` type class
Just 'a'
Prelude> print Nothing
Nothing

Rappelez-vous que vous pouvez instancier Show pour vos propres types en utilisant deriving :

-- In ex.hs
data Person = Person { name :: String } deriving Show
main = print (Person "Alex")  -- Person is an instance of `Show`, thanks to `deriving`

Chargement et exécution dans GHCi :

Prelude> :load ex.hs
[1 of 1] Compiling ex             ( ex.hs, interpreted )
Ok, modules loaded: ex.
*Main> main  -- from ex.hs
Person {name = "Alex"}
*Main>

[1] : https://www.haskell.org/onlinereport/haskell2010/haskellch7.html#x14-1430007.1

Analyser et construire un objet à partir de l’entrée standard #

readFloat :: IO Float
readFloat =
    fmap read getLine


main :: IO ()
main = do
    putStr "Type the first number: "
    first <- readFloat

    putStr "Type the second number: "
    second <- readFloat

    putStrLn $ show first ++ " + " ++ show second ++ " = " ++ show ( first + second )

Saisir:

Type the first number: 9.5
Type the second number: -2.02

Production:

9.5 + -2.02 = 7.48

Lecture à partir des descripteurs de fichiers #

Comme dans plusieurs autres parties de la bibliothèque d’E/S, les fonctions qui utilisent implicitement un flux standard ont un équivalent dans System.IO qui effectue le même travail, mais avec un paramètre supplémentaire à gauche, de type Handle, qui représente le flux en cours de traitement. Par exemple, getLine :: IO String a un équivalent hGetLine :: Handle -> IO String.

import System.IO( Handle, FilePath, IOMode( ReadMode ), 
                  openFile, hGetLine, hPutStr, hClose, hIsEOF, stderr )

import Control.Monad( when )


dumpFile :: Handle -> FilePath -> Integer -> IO ()

dumpFile handle filename lineNumber = do      -- show file contents line by line
    end <- hIsEOF handle
    when ( not end ) $ do
        line <- hGetLine handle
        putStrLn $ filename ++ ":" ++ show lineNumber ++ ": " ++ line
        dumpFile handle filename $ lineNumber + 1


main :: IO ()

main = do
    hPutStr stderr "Type a filename: "
    filename <- getLine
    handle <- openFile filename ReadMode
    dumpFile handle filename 1
    hClose handle

Contenu du fichier example.txt :

This is an example.
Hello, world!
This is another example.

Saisir:

Type a filename: example.txt

Production:

example.txt:1: This is an example.
example.txt:2: Hello, world!
example.txt:3: This is another example

Lecture depuis stdin #

Conformément à la [Haskell 2010 Language Specification][1], les fonctions suivantes sont des fonctions d’E/S standard disponibles dans Prelude, aucune importation n’est donc nécessaire pour les utiliser.

getChar :: IO Char - lit un Char à partir de stdin #

-- MyChar.hs
main = do
  myChar <- getChar
  print myChar

-- In your shell

runhaskell MyChar.hs
a -- you enter a and press enter
'a'  -- the program prints 'a'

getLine :: IO String - lit une String à partir de stdin, sans caractère de nouvelle ligne #

Prelude> getLine
Hello there!  -- user enters some text and presses enter
"Hello there!"

read :: Read a => String -> a - convertit une chaîne en une valeur #

Prelude> read "1" :: Int
1
Prelude> read "1" :: Float
1.0
Prelude> read "True" :: Bool
True

D’autres fonctions moins courantes sont :

  • getContents :: IO String - renvoie toutes les entrées de l’utilisateur sous la forme d’une seule chaîne, qui est lue paresseusement au besoin
  • interact :: (String -> String) -> IO () - prend une fonction de type String->String comme argument. L’intégralité de l’entrée du périphérique d’entrée standard est transmise à cette fonction en tant qu’argument
