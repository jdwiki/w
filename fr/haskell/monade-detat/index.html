Les monades d’état sont une sorte de monade qui porte un état qui peut changer à chaque exécution de calcul dans la monade.

Les implémentations sont généralement de la forme State s a qui représente un calcul qui transporte et modifie potentiellement un état de type s et produit un résultat de type a, mais le terme “monade d’état” peut généralement désigner toute monade qui porte un état.

Les packages mtl et transformers fournissent des implémentations générales des monades d’état.

Les nouveaux venus dans Haskell évitent souvent la monade “State” et la traitent comme un tabou - comme si l’avantage revendiqué de la programmation fonctionnelle était d’éviter l’état, alors ne perdez-vous pas cela lorsque vous utilisez “State” ? Une vision plus nuancée est que :

  • L’état peut être utile à petites doses contrôlées ;
  • Le type State offre la possibilité de contrôler la dose très précisément.

Les raisons étant que si vous avez action :: State s a, cela vous indique que :

  • action est spécial car il dépend d’un état ;
  • L’état est de type s, donc action ne peut être influencé par aucune ancienne valeur de votre programme - seulement un s ou une valeur accessible à partir de certains s ;
  • Le runState :: State s a -> s -> (a, s) place une “barrière” autour de l’action avec état, de sorte que son efficacité ne peut pas être observée depuis l’extérieur de cette barrière.

Il s’agit donc d’un bon ensemble de critères pour savoir s’il faut utiliser “State” dans un scénario particulier. Vous voulez voir que votre code minimise la portée de l’état, à la fois en choisissant un type étroit pour s et en plaçant runState aussi près que possible du “bas” (afin que vos actions puissent être influencé par le moins de choses possible.

Numéroter les nœuds d’un arbre avec un compteur #

Nous avons un type de données arborescent comme celui-ci :

data Tree a = Tree a [Tree a] deriving Show

Et nous souhaitons écrire une fonction qui attribue un numéro à chaque nœud de l’arbre, à partir d’un compteur incrémenté :

tag :: Tree a -> Tree (a, Int)

Le long chemin #

Nous allons d’abord le faire dans le sens le plus long, car cela illustre assez bien les mécanismes de bas niveau de la monade “State”.

import Control.Monad.State

-- Function that numbers the nodes of a `Tree`.
tag :: Tree a -> Tree (a, Int)
tag tree = 
    -- tagStep is where the action happens.  This just gets the ball
    -- rolling, with `0` as the initial counter value.
    evalState (tagStep tree) 0

-- This is one monadic "step" of the calculation.  It assumes that
-- it has access to the current counter value implicitly.
tagStep :: Tree a -> State Int (Tree (a, Int))
tagStep (Tree a subtrees) = do
    -- The `get :: State s s` action accesses the implicit state
    -- parameter of the State monad.  Here we bind that value to
    -- the variable `counter`.
    counter <- get 

    -- The `put :: s -> State s ()` sets the implicit state parameter
    -- of the `State` monad.  The next `get` that we execute will see
    -- the value of `counter + 1` (assuming no other puts in between).
    put (counter + 1)

    -- Recurse into the subtrees.  `mapM` is a utility function
    -- for executing a monadic actions (like `tagStep`) on a list of
    -- elements, and producing the list of results.  Each execution of 
    -- `tagStep` will be executed with the counter value that resulted
    -- from the previous list element's execution.
    subtrees' <- mapM tagStep subtrees  

    return $ Tree (a, counter) subtrees'

Refactorisation #

Diviser le compteur en une action postIncrement #

Le bit où nous “obtenons” le compteur actuel, puis “mettons le compteur + 1” peut être divisé en une action “postIncrement”, similaire à ce que proposent de nombreux langages de style C :

postIncrement :: Enum s => State s s
postIncrement = do
    result <- get
    modify succ
    return result

Diviser l’arborescence en une fonction d’ordre supérieur #

La logique d’arborescence peut être divisée en sa propre fonction, comme ceci :

mapTreeM :: Monad m => (a -> m b) -> Tree a -> m (Tree b)
mapTreeM action (Tree a subtrees) = do
    a' <- action a
    subtrees' <- mapM (mapTreeM action) subtrees
    return $ Tree a' subtrees'

Avec ceci et la fonction postIncrement nous pouvons réécrire tagStep :

tagStep :: Tree a -> State Int (Tree (a, Int))
tagStep = mapTreeM step
    where step :: a -> State Int (a, Int)
          step a = do 
              counter <- postIncrement
              return (a, counter)

Utiliser la classe Traversable #

La solution mapTreeM ci-dessus peut être facilement réécrite dans une instance de la classe Traversable:

instance Traversable Tree where
    traverse action (Tree a subtrees) = 
        Tree <$> action a <*> traverse action subtrees

Notez que cela nous a obligés à utiliser Applicative (l’opérateur <*>) au lieu de Monad.

Avec cela, nous pouvons maintenant écrire tag comme un pro :

tag :: Traversable t => t a -> t (a, Int)
tag init t = evalState (traverse step t) 0
    where step a = do tag <- postIncrement
                      return (a, tag)

Notez que cela fonctionne pour n’importe quel type Traversable, pas seulement notre type Tree !

Se débarrasser du passe-partout Traversable #

GHC a une extension “DeriveTraversable” qui élimine le besoin d’écrire l’instance ci-dessus :

{-# LANGUAGE DeriveFunctor, DeriveFoldable, DeriveTraversable #-}

data Tree a = Tree a [Tree a]
            deriving (Show, Functor, Foldable, Traversable)
