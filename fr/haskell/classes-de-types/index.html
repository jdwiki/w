Les classes de types dans Haskell sont un moyen de définir le comportement associé à un type séparément de la définition de ce type. Alors que, disons, en Java, vous définiriez le comportement dans le cadre de la définition du type - c’est-à-dire dans une interface, une classe abstraite ou une classe concrète - Haskell sépare ces deux éléments.

Il existe un certain nombre de classes de types déjà définies dans le paquet base de Haskell. La relation entre ceux-ci est illustrée dans la section Remarques ci-dessous.

Le diagramme suivant tiré de l’article [Typeclassopedia][1] montre la relation entre les différentes classes de types dans Haskell.

[![Relations entre les classes de types Haskell standard, Figure 1 tel que publié dans Typeclassopedia.][2]][2]

[1] : https://wiki.haskell.org/Typeclassopedia [2] : http://i.stack.imgur.com/Fph6t.png

Éq #

Tous les types de données de base (comme Int, String, Eq a => [a]) de Prelude à l’exception des fonctions et IO ont des instances de Eq. Si un type instancie Eq cela signifie que nous savons comment comparer deux valeurs pour l’égalité valeur ou structurelle.

> 3 == 2 
False
> 3 == 3
True

Méthodes requises #

  • (==) :: Eq a => a -> a -> Boolean ou (/=) :: Eq a => a -> a -> Boolean (si un seul est implémenté, l’autre par défaut à la négation du défini)

Définit #

  • (==) :: Eq a => a -> a -> Booléen
  • (/=) :: Eq a => a -> a -> Booléen

Superclasses directes #

Aucun

Sous-classes notables #

  • Ord

Commande #

Les types instanciant Ord incluent, par exemple, Int, String et [a] (pour les types a où il y a une instance Ord a). Si un type instancie Ord cela signifie que nous connaissons un “naturel” ordre des valeurs de ce type. Attention, il y a souvent beaucoup de choix possibles du “naturel” l’ordre d’un type et Ord nous oblige à en privilégier un.

Ord fournit les opérateurs standard (<=), (<), (>), (>=) mais les définit de manière intéressante en utilisant un type de données algébrique personnalisé

data Ordering = LT | EQ | GT

compare :: Ord a => a -> a -> Ordering

Méthodes requises #

  • compare :: Ord a => a -> a -> Ordering ou (<=) :: Ord a => a -> a -> Boolean (la méthode compare par défaut standard utilise ( <=) dans sa mise en œuvre)

Définit #

  • comparer :: Ord a => a -> a -> Ordering
  • (<=) :: Ord a => a -> a -> Booléen
  • (<) :: Ord a => a -> a -> Booléen
  • (>=) :: Ord a => a -> a -> Booléen
  • (>) :: Ord a => a -> a -> Booléen
  • min :: Ord a => a -> a -> a
  • max :: Ord a => a -> a -> a

Superclasses directes #

  • Eq

Monoïde #

Les types instanciant [Monoid][1] incluent des listes, des nombres et des fonctions avec des valeurs de retour Monoid, entre autres. Pour instancier Monoid, un type doit prendre en charge une opération binaire associative (mappend ou (<>)) qui combine ses valeurs, et avoir une valeur “zéro” spéciale (mempty) telle que la combinaison d’une valeur avec elle ne change pas cette valeur :

mempty  <>  x == x
x <>  mempty  == x

x <> (y <> z) == (x <> y) <> z

Intuitivement, les types Monoid sont de type “liste” en ce sens qu’ils prennent en charge l’ajout de valeurs ensemble. Alternativement, les types Monoid peuvent être considérés comme des séquences de valeurs pour lesquelles nous nous soucions de l’ordre mais pas du regroupement. Par exemple, un arbre binaire est un Monoid, mais en utilisant les opérations Monoid nous ne pouvons pas voir sa structure de branchement, seulement un parcours de ses valeurs (voir Foldable et Traversable).

Méthodes requises #

  • mempty :: Monoïde m => m
  • mappend :: Monoïde m => m -> m -> m

Superclasses directes #

Aucun

[1] : https://www.wikiod.com/fr/haskell/monoide

Nombre #

La classe la plus générale pour les types de nombres, plus précisément pour les anneaux, c’est-à-dire les nombres qui peuvent être additionnés, soustraits et multipliés au sens habituel, mais pas forcément divisé.

Cette classe contient à la fois des types intégraux (Int, Integer, Word32 etc.) et des types fractionnaires (Double, Rational, ainsi que des nombres complexes, etc.). Dans le cas de types finis, la sémantique est généralement comprise comme de l’arithmétique modulaire, c’est-à-dire avec dépassement supérieur et inférieur†.

Notez que les règles pour les classes numériques sont beaucoup moins strictement respectées que le [monad](https://www.wikiod.com/fr/haskell/monades#La monade Peut-être =201610010024186706477) ou lois monoïdes, ou celles pour comparaison d’égalité. En particulier, les nombres à virgule flottante n’obéissent généralement à des lois que dans un sens approximatif.

Les méthodes #

  • fromInteger :: Num a => Entier -> a. convertir un entier en type de nombre général (bouclage autour de la plage, si nécessaire). Haskell [littéraux numériques] (https://www.wikiod.com/fr/haskell/litteraux-surcharges#Nombre entier) peut être compris comme un littéral “entier” monomorphe avec la conversion générale qui l’entoure, vous pouvez donc utilisez le littéral 5 à la fois dans un contexte Int et dans un paramètre Complex Double.

  • (+) :: Num a => a -> a -> a. L’addition standard, généralement comprise comme associative et commutative, c’est-à-dire,

      a + (b + c) ≡ (a + b) + c
  a + b ≡ b + a

  • (-) :: Num a => a -> a -> a. Soustraction, qui est l’inverse de l’addition :

      (a - b) + b ≡ (a + b) - b ≡ a

  • (*) :: Num a => a -> a -> a. La multiplication, une opération associative qui est distributive sur l’addition :

      a * (b * c) ≡ (a * b) * c
  a * (b + c) ≡ a * b + a * c

pour les instances les plus courantes, la multiplication est également commutative, mais ce n’est certainement pas une exigence.

  • negate :: Num a => a -> a. Nom complet de l’opérateur de négation unaire. -1 est le sucre syntaxique pour negate 1.

      -a ≡ negate a ≡ 0 - a

  • abs :: Num a => a -> a. La fonction de valeur absolue donne toujours un résultat non négatif de même grandeur

      abs (-a) ≡ abs a
  abs (abs a) ≡ abs a

abs a ≡ 0 ne devrait se produire que si a ≡ 0.

Pour les types réels, la signification de non-négatif est claire : vous avez toujours abs a >= 0. Les types complexes, etc. n’ont pas d’ordre bien défini, mais le résultat de “abs” doit toujours se trouver dans le sous-ensemble réel‡ (c’est-à-dire donner un nombre qui pourrait également être écrit sous la forme d’un nombre unique littéral sans négation).

  • signum :: Num a => a -> a. La fonction de signe, selon son nom, ne donne que -1 ou 1, selon le signe de l’argument. En fait, cela n’est vrai que pour les nombres réels non nuls ; en général, signum est mieux compris comme la fonction normalising :

      abs (signum a) ≡ 1   -- unless a≡0
  signum a * abs a ≡ a -- This is required to be true for all Num instances

Notez que la section 6.4.4 du rapport Haskell 2010 exige explicitement que cette dernière égalité soit valable pour tout Num valide instance.

Certaines bibliothèques, notamment linear et hmatrix, ont une compréhension beaucoup plus laxiste de ce que La classe Num est pour : ils la traitent comme un moyen de surcharger les opérateurs arithmétiques. Bien que cela soit assez simple pour + et -, cela devient déjà gênant avec * et plus encore avec les autres méthodes. Par exemple, devrait-il signifier * multiplication matricielle ou multiplication élément par élément ?
C’est sans doute une mauvaise idée de définir de telles instances non numériques ; veuillez envisager des classes dédiées telles que VectorSpace.

† En particulier, les “négatifs” des types non signés sont enroulés autour d’un grand positif, par ex. (-4 :: Word32) == 4294967292.

‡ Ceci n’est généralement pas rempli : les types de vecteurs n’ont pas de véritable sous-ensemble. Les instances controversées de Num pour de tels types définissent généralement abs et signum élément par élément, ce qui mathématiquement parlant n’a pas vraiment de sens.

Peut-être et la classe Functor #

Dans Haskell, les types de données peuvent avoir des arguments tout comme les fonctions. Prenez le type “Peut-être” par exemple.

“Peut-être” est un type très utile qui nous permet de représenter l’idée d’échec, ou la possibilité de celui-ci. En d’autres termes, s’il existe une possibilité qu’un calcul échoue, nous utilisons ici le type “Peut-être”. “Peut-être” agit un peu comme un wrapper pour d’autres types, leur donnant des fonctionnalités supplémentaires.

Sa déclaration proprement dite est assez simple.

Maybe a = Just a | Nothing

Ce que cela indique, c’est qu’un “Peut-être” se présente sous deux formes, un “Juste”, qui représente le succès, et un “Rien”, qui représente l’échec. Just prend un argument qui détermine le type de Maybe, et Nothing n’en prend aucun. Par exemple, la valeur Just "foo" aura le type Maybe String, qui est un type de chaîne enveloppé avec la fonctionnalité supplémentaire Maybe. La valeur Nothing a le type Maybe a où a peut être n’importe quel type.

Cette idée d’envelopper les types pour leur donner des fonctionnalités supplémentaires est très utile et s’applique à plus que “Peut-être”. D’autres exemples incluent les types Either, IO et liste, chacun offrant des fonctionnalités différentes. Cependant, certaines actions et capacités sont communes à tous ces types d’emballages. Le plus notable d’entre eux est la possibilité de modifier la valeur encapsulée.

Il est courant de considérer ces types de types comme des boîtes dans lesquelles des valeurs peuvent être placées. Différentes boîtes contiennent des valeurs différentes et font des choses différentes, mais aucune n’est utile sans pouvoir accéder au contenu qu’elles contiennent.

Pour encapsuler cette idée, Haskell est livré avec une classe de types standard, nommée Functor. Il est défini comme suit.

class Functor f where
  fmap :: (a -> b) -> f a -> f b

Comme on peut le voir, la classe a une seule fonction, fmap, de deux arguments. Le premier argument est une fonction d’un type, ‘a’, à un autre, ‘b’. Le second argument est un foncteur (type wrapper) contenant une valeur de type a. Il retourne un foncteur (type wrapper) contenant une valeur de type b.

En termes simples, fmap prend une fonction et s’applique à la valeur à l’intérieur d’un foncteur. C’est la seule fonction nécessaire pour qu’un type soit membre de la classe Functor, mais elle est extrêmement utile. Les fonctions opérant sur des foncteurs qui ont des applications plus spécifiques peuvent être trouvées dans les classes de type Applicative et Monad.

## Héritage de la classe de type : classe de type Ord Haskell prend en charge une notion d’extension de classe. Par exemple, la classe Ord hérite de toutes les opérations dans Eq, mais possède en plus une fonction compare qui renvoie un Ordering entre les valeurs. Ord peut également contenir les opérateurs de comparaison d’ordre communs, ainsi qu’une méthode min et une méthode max.

La notation => a la même signification que dans une signature de fonction et nécessite le type a pour implémenter Eq, afin d’implémenter Ord.

data Ordering = EQ | LT | GT

class Eq a => Ord a where
    compare :: Ord a => a -> a -> Ordering
    (<)     :: Ord a => a -> a -> Bool
    (<=)    :: Ord a => a -> a -> Bool
    (>)     :: Ord a => a -> a -> Bool
    (>=)    :: Ord a => a -> a -> Bool
    min     :: Ord a => a -> a -> a
    max     :: Ord a => a -> a -> a

Toutes les méthodes qui suivent “compare” peuvent en être dérivées de plusieurs manières :

x < y   = compare x y == LT
x <= y  = x < y || x == y -- Note the use of (==) inherited from Eq
x > y   = not (x <= y)
x >= y  = not (x < y)

min x y = case compare x y of
               EQ -> x
               LT -> x
               GT -> y

max x y = case compare x y of
               EQ -> x
               LT -> y
               GT -> x

Les classes de type qui étendent elles-mêmes Ord doivent implémenter au moins la méthode compare ou la méthode (<=) elles-mêmes, qui construit le treillis d’héritage dirigé.
