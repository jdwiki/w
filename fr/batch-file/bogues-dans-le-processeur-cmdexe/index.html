Cette rubrique se concentrera sur les erreurs causées par les bogues du processeur. Voici les choses sur lesquelles nous nous concentrerions sur la cause et la solution du problème.

Dans l’exemple [DEL File Extension][1], l’utilisateur X. Liu remarque que ce bogue ne se produit pas lorsque l’extension de fichier dans la commande DEL est inférieure à 3 caractères.

[1] : https://www.wikiod.com/fr/batch-file/bogues-dans-le-processeur-cmdexe#DEL Extension de fichier

Confusion entre parenthèses #

À partir de [ce] [1] site Web, l’OP a remarqué un problème.

Cause #

Considérez l’extrait de code suivant.

if 1==1 (
    set /a result = 2*(3+4)
)

À première vue, vous pensez peut-être que CMD.exe le traiterait comme suit :

  • La condition est vraie, exécutez le bloc de code
  • Définissez la valeur du résultat de la variable sur 14
  • Continuer

Cependant, le processus CMD.exe ressemble à ceci :

  • La condition est vraie, exécutez le bloc de code
  • **Calculer 2*(3+4, le ) après 4 est traité à la fin du bloc de code if **
  • Un ) aléatoire est apparu !

La deuxième étape renverrait l’erreur “Parenthèses déséquilibrées”.

La solution #

Selon le set /? d’un CMD.exe allemand, nous aurions besoin de citer des opérations arithmétiques. Voici un exemple.

Précédent Résultat
set /a result=2+5*4 set /a result="2+5*4"

Soit dit en passant, selon un anglais CMD.exe set /?, les guillemets sont requis si des opérateurs logiques ou de module sont présents dans l’expression (bien que ce ne soit pas une étape * obligatoire *).

[1] : https://www.pcreview.co.uk/threads/bug-in-cmd-exe-parentheses-confused.2916962/

Caractère d’échappement incorrect #

[Dans cette question Stack Overflow][1], l’utilisateur txtechhelp a trouvé un problème avec le caractère ^ qui pourrait causer un problème de sécurité.

Cause #

anyInvaildCommand ^

Remarque : Assurez-vous que le caret (^) est le dernier caractère ! Tout CR\LF supplémentaire ne fonctionnera pas du tout !

Le caret recherche le prochain caractère à échapper. Cependant, il n’y a plus de caractère disponible pour s’échapper, donc cmd boucle à l’infini, à la recherche d’un caractère à échapper. Dans ce processus de “boucle”, cmd.exe consommera la mémoire de votre ordinateur. Et manger progressivement toute la mémoire, mettant l’ordinateur à genoux.

Ce problème peut entraîner des problèmes de sécurité plus graves, car il suffit de saisir le code dans son ordinateur déverrouillé.

Solutions #

  • Utiliser la page de code UTF-16 pourrait résoudre ce problème. Seuls UTF-8 ou ASCII causeraient le bogue.

  • Assurez-vous qu’il y a un CR \ LF supplémentaire dans le fichier, ou n’utilisez simplement pas de signe d’insertion à la fin du fichier.

En plus #

Ce bogue semble être résolu dans Windows 10.

[1] : https://stackoverflow.com/questions/23284131/cmd-exe-parsing-bug-leads-to-other-exploits

DEL Extension de fichier #

Ce bogue a été signalé par steve2916 à partir de ce [fil du forum Microsoft Windows.][1]

Cause #

Considérez un dossier avec de tels fichiers.

TestA.doc
TestB.doc
TestC.docx
TestD.docx

Si nous voulons supprimer tous les fichiers .doc de ce répertoire, nous ferions généralement :

del *.doc

Cependant, cette commande supprime également les fichiers .docx. La même chose se produit sur les extensions de fichiers avec ce modèle.

Fichier A Fichier B
N’importe quel nom.abc AutreNom.abcd

Comme nous pouvons le voir, tant que la chaîne d’extension de fichier contient la chaîne utilisée dans la commande del, le fichier sera supprimé. Notez que ce bogue ne se produit que lorsque la chaîne d’extension utilisée dans la commande del comporte au moins trois caractères. Par exemple, del *.do ne supprime pas A.doc ou A.docx.

La solution #

Dans le fil de discussion, l’utilisateur MicroCompsUnltd a remarqué que l’utilisation de “Powershell” résoudrait le problème.
