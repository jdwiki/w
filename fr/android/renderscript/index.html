RenderScript est un langage de script qui vous permet d’écrire un rendu graphique hautes performances et du code de calcul brut. Il fournit un moyen d’écrire du code critique pour les performances que le système compile ensuite en code natif pour le processeur sur lequel il peut s’exécuter. Il peut s’agir du CPU, d’un CPU multicœur ou même du GPU. Sur quoi il s’exécute finalement dépend de nombreux facteurs qui ne sont pas facilement disponibles pour le développeur, mais aussi de l’architecture prise en charge par le compilateur de plate-forme interne.

Commencer #

RenderScript est un framework permettant un calcul parallèle haute performance sur Android. Les scripts que vous écrivez seront exécutés sur tous les processeurs disponibles (par exemple, CPU, GPU, etc.) en parallèle, ce qui vous permettra de vous concentrer sur la tâche que vous souhaitez accomplir plutôt que sur la façon dont elle est planifiée et exécutée.

Les scripts sont écrits dans un langage basé sur C99 (C99 étant une ancienne version du langage de programmation C standard). Pour chaque script, une classe Java est créée, ce qui vous permet d’interagir facilement avec RenderScript dans votre code Java.

Monter son projet #

Il existe deux manières différentes d’accéder à RenderScript dans votre application, avec les bibliothèques Android Framework ou la bibliothèque de support. Même si vous ne souhaitez pas cibler les appareils avant le niveau 11 de l’API, vous devez toujours utiliser l’implémentation de la bibliothèque de support, car elle garantit la compatibilité des appareils sur de nombreux appareils différents. Pour utiliser l’implémentation de la bibliothèque de support, vous devez utiliser au moins la version 18.1.0 des outils de construction !

Maintenant, configurons le fichier build.gradle de votre application :

android {
    compileSdkVersion 24
    buildToolsVersion '24.0.1'

    defaultConfig {
        minSdkVersion 8
        targetSdkVersion 24

        renderscriptTargetApi 18
        renderscriptSupportModeEnabled true
    }
}
  • renderscriptTargetApi : cela doit être défini sur la version la plus ancienne de l’API qui fournit toutes les fonctionnalités RenderScript dont vous avez besoin.
  • renderscriptSupportModeEnabled  : cela active l’utilisation de l’implémentation de la bibliothèque de support RenderScript.

Comment fonctionne RenderScript #

Un RenderScript typique se compose de deux éléments : des noyaux et des fonctions. Une fonction est exactement ce à quoi elle ressemble - elle accepte une entrée, fait quelque chose avec cette entrée et renvoie une sortie. Un noyau est à l’origine de la véritable puissance de RenderScript.

Un noyau est une fonction qui est exécutée sur chaque élément à l’intérieur d’une Allocation. Une Allocation peut être utilisée pour transmettre des données comme un Bitmap ou un tableau byte à un RenderScript et elles sont également utilisées pour obtenir un résultat à partir d’un noyau. Les noyaux peuvent soit prendre une Allocation en entrée et une autre en sortie, soit modifier les données à l’intérieur d’une seule Allocation.

Vous pouvez écrire votre propre noyau, mais il existe également de nombreux noyaux prédéfinis que vous pouvez utiliser pour effectuer des opérations courantes comme un flou d’image gaussien.

Comme déjà mentionné pour chaque fichier RenderScript, une classe est générée pour interagir avec lui. Ces classes commencent toujours par le préfixe ScriptC_ suivi du nom du fichier RenderScript. Par exemple, si votre fichier RenderScript s’appelle example, la classe Java générée s’appellera ScriptC_example. Tous les scripts prédéfinis commencent simplement par le préfixe “Script” - par exemple, le script de flou d’image gaussien est appelé “ScriptIntrinsicBlur”.

Passe-partout RenderScript #

Les fichiers RenderScript résident dans le dossier src/main/rs de votre projet. Chaque fichier a l’extension de fichier .rs et doit contenir deux déclarations #pragma en haut :

#pragma version(1)
#pragma rs java_package_name(your.package.name)

  • #pragma version(1) : cela peut être utilisé pour définir la version de RenderScript que vous utilisez. Actuellement il n’y a que la version 1.

  • #pragma rs java_package_name(your.package.name) : cela peut être utilisé pour définir le nom du package de la classe Java générée pour interagir avec ce RenderScript particulier.

Il existe un autre #pragma que vous devez généralement définir dans chacun de vos fichiers RenderScript et il est utilisé pour définir la précision en virgule flottante. Vous pouvez définir la précision de la virgule flottante sur trois niveaux différents :

  • #pragma rs_fp_full : c’est le paramètre le plus strict avec la plus grande précision et c’est aussi la valeur par défaut si vous ne spécifiez rien. Vous devez l’utiliser si vous avez besoin d’une précision élevée en virgule flottante.
  • #pragma rs_fp_relaxed : cela n’assure pas une précision aussi élevée en virgule flottante, mais sur certaines architectures, cela permet un tas d’optimisations qui peuvent accélérer l’exécution de vos scripts.
  • #pragma rs_fp_imprecise : cela garantit encore moins de précision et doit être utilisé si la précision en virgule flottante n’a pas vraiment d’importance pour votre script.

La plupart des scripts peuvent simplement utiliser #pragma rs_fp_relaxed sauf si vous avez vraiment besoin d’une précision élevée en virgule flottante.

Variables globales #

Maintenant, comme dans le code C, vous pouvez définir des variables globales ou des constantes :

const static float3 gMonoMult = {0.299f, 0.587f, 0.114f};

float saturationLevel = 0.0f;

La variable gMonoMult est de type float3. Cela signifie qu’il s’agit d’un vecteur composé de 3 nombres flottants. L’autre variable float appelée saturationValue n’est pas constante, vous pouvez donc la définir au moment de l’exécution sur une valeur que vous aimez. Vous pouvez utiliser des variables comme celle-ci dans vos noyaux ou vos fonctions et, par conséquent, elles constituent un autre moyen de donner une entrée ou de recevoir une sortie de vos RenderScripts. Pour chaque variable non constante, une méthode getter et setter sera générée sur la classe Java associée.

Noyaux #

Mais maintenant, commençons à implémenter le noyau. Pour les besoins de cet exemple, je ne vais pas expliquer les mathématiques utilisées dans le noyau pour modifier la saturation de l’image, mais plutôt me concentrer sur la façon d’implémenter un noyau et comment l’utiliser. À la fin de ce chapitre, j’expliquerai rapidement ce que le code de ce noyau fait réellement.

Noyaux en général #

Examinons d’abord le code source :

uchar4 __attribute__((kernel)) saturation(uchar4 in) {
    float4 f4 = rsUnpackColor8888(in);
    float3 dotVector = dot(f4.rgb, gMonoMult);
    float3 newColor = mix(dotVector, f4.rgb, saturationLevel);
    return rsPackColorTo8888(newColor);
}

Comme vous pouvez le voir, cela ressemble à une fonction C normale à une exception près : L’__attribute__((kernel)) entre le type de retour et le nom de la méthode. C’est ce qui indique à RenderScript que cette méthode est un noyau. Une autre chose que vous remarquerez peut-être est que cette méthode accepte un paramètre uchar4 et renvoie une autre valeur uchar4. uchar4 est - comme la variable float3 dont nous avons parlé dans le chapitre précédent - un vecteur. Il contient 4 valeurs uchar qui ne sont que des valeurs d’octets comprises entre 0 et 255.

Vous pouvez accéder à ces valeurs individuelles de différentes manières, par exemple in.r renverrait l’octet qui correspond au canal rouge d’un pixel. Nous utilisons un uchar4 car chaque pixel est composé de 4 valeurs - r pour le rouge, g pour le vert, b pour le bleu et a pour l’alpha - et vous pouvez y accéder avec ce raccourci. RenderScript vous permet également de prendre n’importe quel nombre de valeurs d’un vecteur et de créer un autre vecteur avec elles. Par exemple, in.rgb renverrait une valeur uchar3 qui ne contient que les parties rouge, verte et bleue du pixel sans la valeur alpha.

Au moment de l’exécution, RenderScript appellera cette méthode Kernel pour chaque pixel d’une image, c’est pourquoi la valeur de retour et le paramètre ne sont qu’une seule valeur uchar4. RenderScript exécutera plusieurs de ces appels en parallèle sur tous les processeurs disponibles, c’est pourquoi RenderScript est si puissant. Cela signifie également que vous n’avez pas à vous soucier du threading ou de la sécurité des threads, vous pouvez simplement implémenter ce que vous voulez faire pour chaque pixel et RenderScript s’occupe du reste.

Lors de l’appel d’un noyau en Java, vous fournissez deux variables Allocation, une qui contient les données d’entrée et une autre qui recevra la sortie. Votre méthode Kernel sera appelée pour chaque valeur dans l’entrée Allocation et écrira le résultat dans la sortie Allocation.

Méthodes de l’API RenderScript Runtime #

Dans le noyau ci-dessus, quelques méthodes sont utilisées qui sont fournies prêtes à l’emploi. RenderScript fournit de nombreuses méthodes de ce type et elles sont essentielles pour presque tout ce que vous allez faire avec RenderScript. Parmi elles se trouvent des méthodes pour effectuer des opérations mathématiques comme sin() et des méthodes d’assistance comme mix() qui mélangent deux valeurs en fonction d’une autre valeur. Mais il existe également des méthodes pour des opérations plus complexes lorsqu’il s’agit de vecteurs, de quaternions et de matrices.

La RenderScript Runtime API Reference officielle est la meilleure ressource disponible si vous souhaitez en savoir plus sur un méthode ou recherchez une méthode spécifique qui effectue une opération courante comme le calcul du produit scalaire d’une matrice. Vous pouvez trouver cette documentation ici.

Implémentation du noyau #

Voyons maintenant les spécificités de ce que fait ce noyau. Voici la première ligne du noyau :

float4 f4 = rsUnpackColor8888(in);

La première ligne appelle la méthode intégrée rsUnpackColor8888() qui transforme le uchar4 en valeurs float4. Chaque canal de couleur est également transformé dans la plage ‘0.0f - 1.0f’ où ‘0.0f’ correspond à une valeur d’octet de ‘0’ et ‘1.0f’ à ‘255’. Le but principal de ceci est de rendre tous les calculs de ce noyau beaucoup plus simples.

float3 dotVector = dot(f4.rgb, gMonoMult);

Cette ligne suivante utilise la méthode intégrée dot() pour calculer le point produit de deux vecteurs. gMonoMult est une valeur constante que nous avons définie quelques chapitres plus haut. Puisque les deux vecteurs doivent être de la même longueur pour calculer le produit scalaire et aussi puisque nous voulons juste affecter les canaux de couleur et non le canal alpha d’un pixel, nous utilisons le raccourci .rgb pour obtenir un nouveau float3 vecteur qui ne contient que les canaux de couleur rouge, vert et bleu. Ceux d’entre nous qui se souviennent encore de l’école comment fonctionne le produit scalaire remarqueront rapidement que le produit scalaire ne doit renvoyer qu’une seule valeur et non un vecteur. Pourtant, dans le code ci-dessus, nous attribuons le résultat à un vecteur “float3”. C’est encore une fonctionnalité de RenderScript. Lorsque vous affectez un nombre unidimensionnel à un vecteur, tous les éléments du vecteur seront définis sur cette valeur. Par exemple, l’extrait de code suivant attribuera 2.0f à chacune des trois valeurs du vecteur float3 :

float3 example = 2.0f;

Ainsi, le résultat du produit scalaire ci-dessus est attribué à chaque élément du vecteur “float3” ci-dessus.

Vient maintenant la partie dans laquelle nous utilisons réellement la variable globale saturationLevel pour modifier la saturation de l’image :

float3 newColor = mix(dotVector, f4.rgb, saturationLevel);

Ceci utilise la méthode intégrée mix() pour mélanger la couleur d’origine avec le vecteur de produit scalaire que nous avons créé ci-dessus. La manière dont ils sont mélangés est déterminée par la variable globale saturationLevel. Ainsi, un saturationLevel de 0.0f fera que la couleur résultante n’aura aucune partie des valeurs de couleur d’origine et ne sera composée que de valeurs dans le dotVector qui se traduit par une image en noir et blanc ou grisée. Une valeur de 1.0f fera que la couleur résultante sera entièrement composée des valeurs de couleur d’origine et les valeurs supérieures à 1.0f multiplieront les couleurs d’origine pour les rendre plus lumineuses et intenses.

return rsPackColorTo8888(newColor);

C’est la dernière partie du noyau. rsPackColorTo8888() transforme le vecteur float3 en un uchar4 valeur qui est ensuite renvoyée. Les valeurs d’octet résultantes sont limitées à une plage comprise entre 0 et 255, de sorte que les valeurs flottantes supérieures à “1.0f” donneront une valeur d’octet de 255 et les valeurs inférieures à “0.0” donneront une valeur d’octet de “0”.

Et c’est toute l’implémentation du noyau. Maintenant, il ne reste plus qu’une partie : comment appeler un noyau en Java.

Appel de RenderScript en Java #

Bases #

Comme cela a déjà été mentionné ci-dessus, pour chaque fichier RenderScript, une classe Java est générée, ce qui vous permet d’interagir avec vos scripts. Ces fichiers ont le préfixe ScriptC_ suivi du nom du fichier RenderScript. Pour créer une instance de ces classes, vous avez d’abord besoin d’une instance de la classe RenderScript :

final RenderScript renderScript = RenderScript.create(context);

La méthode statique create() peut être utilisée pour créer une instance RenderScript à partir d’un Context. Vous pouvez ensuite instancier la classe Java qui a été générée pour votre script. Si vous avez appelé le fichier RenderScript saturation.rs alors la classe s’appellera ScriptC_saturation :

final ScriptC_saturation script = new ScriptC_saturation(renderScript);

Sur cette classe, vous pouvez maintenant définir le niveau de saturation et appeler le Kernel. Le setter qui a été généré pour la variable saturationLevel aura le préfixe set_ suivi du nom de la variable :

script.set_saturationLevel(1.0f);

Il existe également un getter préfixé par get_ qui vous permet d’obtenir le niveau de saturation actuellement défini :

float saturationLevel = script.get_saturationLevel();

Les noyaux que vous définissez dans votre RenderScript sont préfixés par forEach_ suivi du nom de la méthode Kernel. Le noyau que nous avons écrit attend une entrée “Allocation” et une sortie “Allocation” comme paramètres :

script.forEach_saturation(inputAllocation, outputAllocation);

L’entrée “Allocation” doit contenir l’image d’entrée, et une fois la méthode “forEach_saturation” terminée, l’allocation de sortie contiendra les données d’image modifiées.

Une fois que vous avez une instance Allocation, vous pouvez copier des données depuis et vers ces Allocations en utilisant les méthodes copyFrom() et copyTo(). Par exemple, vous pouvez copier une nouvelle image dans votre entrée `Allocation en appelant :

inputAllocation.copyFrom(inputBitmap);

De la même manière, vous pouvez récupérer l’image du résultat en appelant copyTo() sur la sortie Allocation :

outputAllocation.copyTo(outputBitmap);

Création d’instances d’allocation #

Il existe plusieurs façons de créer une Allocation. Une fois que vous avez une instance Allocation, vous pouvez copier de nouvelles données depuis et vers ces Allocations avec copyTo() et copyFrom() comme expliqué ci-dessus, mais pour les créer initialement, vous devez savoir avec quel type de données vous travaillez exactement avec. Commençons par l’entrée “Allocation” :

Nous pouvons utiliser la méthode statique createFromBitmap() pour créer rapidement notre entrée Allocation à partir d’un Bitmap :

final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, image);

Dans cet exemple, l’image d’entrée ne change jamais, nous n’avons donc plus jamais besoin de modifier l’entrée “Allocation”. Nous pouvons le réutiliser chaque fois que le saturationLevel change pour créer une nouvelle sortie Bitmap.

La création de la sortie Allocation est un peu plus complexe. Nous devons d’abord créer ce qu’on appelle un Type. Un Type est utilisé pour dire à une Allocation avec quel type de données elle traite. Habituellement, on utilise la classe Type.Builder pour créer rapidement un Type approprié. Voyons d’abord le code :

final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

Nous travaillons avec un Bitmap normal de 32 bits (ou en d’autres termes de 4 octets) par pixel avec 4 canaux de couleur. C’est pourquoi nous choisissons Element.RGBA_8888 pour créer le Type. Ensuite, nous utilisons les méthodes setX() et setY() pour définir la largeur et la hauteur de l’image de sortie à la même taille que l’image d’entrée. La méthode create() crée ensuite le Type avec les paramètres que nous avons spécifiés.

Une fois que nous avons le bon Type, nous pouvons créer la sortie Allocation avec la méthode statique createTyped() :

final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

Maintenant, nous avons presque terminé. Nous avons également besoin d’une sortie Bitmap dans laquelle nous pouvons copier les données de la sortie Allocation. Pour ce faire, nous utilisons la méthode statique createBitmap() pour créer un nouveau Bitmap vide avec la même taille et la même configuration que l’entrée Bitmap.

final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

Et avec cela, nous avons toutes les pièces du puzzle pour exécuter notre RenderScript.

Exemple complet #

Maintenant, rassemblons tout cela dans un exemple :

// Create the RenderScript instance
final RenderScript renderScript = RenderScript.create(context);

// Create the input Allocation 
final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, inputBitmap);

// Create the output Type.
final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

// And use the Type to create am output Allocation
final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

// Create an empty output Bitmap from the input Bitmap
final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

// Create an instance of our script
final ScriptC_saturation script = new ScriptC_saturation(renderScript);

// Set the saturation level
script.set_saturationLevel(2.0f);

// Execute the Kernel
script.forEach_saturation(inputAllocation, outputAllocation);

// Copy the result data to the output Bitmap
outputAllocation.copyTo(outputBitmap);

// Display the result Bitmap somewhere
someImageView.setImageBitmap(outputBitmap);

Conclusion #

Avec cette introduction, vous devriez être prêt à écrire vos propres noyaux RenderScript pour une manipulation simple des images. Cependant, il y a quelques choses que vous devez garder à l’esprit :

  • RenderScript ne fonctionne que dans les projets d’application : actuellement, les fichiers RenderScript ne peuvent pas faire partie d’un projet de bibliothèque.
  • Attention à la mémoire : RenderScript est très rapide, mais il peut également consommer beaucoup de mémoire. Il ne doit jamais y avoir plus d’une instance de RenderScript à la fois. Vous devez également réutiliser autant que possible. Normalement, vous n’avez qu’à créer vos instances “Allocation” une seule fois et vous pouvez les réutiliser à l’avenir. Il en va de même pour la sortie Bitmaps ou vos instances de script. Réutilisez autant que possible.
  • Faites votre travail en arrière-plan : Encore une fois, RenderScript est très rapide, mais en aucun cas instantané. Tout noyau, en particulier les plus complexes, doit être exécuté à partir du thread d’interface utilisateur dans une AsyncTask ou quelque chose de similaire. Cependant, la plupart du temps, vous n’avez pas à vous soucier des fuites de mémoire. Toutes les classes liées à RenderScript n’utilisent que l’application Context et ne provoquent donc pas de fuites de mémoire. Mais vous devez toujours vous soucier des choses habituelles comme la fuite de View, Activity ou de toute instance Context que vous utilisez vous-même !
  • Utilisez des éléments intégrés : il existe de nombreux scripts prédéfinis qui effectuent des tâches telles que le floutage d’image, le mélange, la conversion, le redimensionnement. Et il existe de nombreuses autres méthodes intégrées qui vous aident à implémenter vos noyaux. Il y a de fortes chances que si vous voulez faire quelque chose, il y a soit un script ou une méthode qui fait déjà ce que vous essayez de faire. Ne réinventez pas la roue.

Si vous souhaitez démarrer rapidement et jouer avec du code réel, je vous recommande de jeter un coup d’œil à l’exemple de projet GitHub qui implémente l’exemple exact dont il est question dans ce didacticiel. Vous pouvez trouver le projet ici. Amusez-vous avec RenderScript !

Flouter une vue #

BlurBitmapTask.java #

public class BlurBitmapTask extends AsyncTask<Bitmap, Void, Bitmap> {
    private final WeakReference<ImageView> imageViewReference;
    private final RenderScript renderScript;

    private boolean shouldRecycleSource = false;

    public BlurBitmapTask(@NonNull Context context, @NonNull ImageView imageView) {
        // Use a WeakReference to ensure
        // the ImageView can be garbage collected
        imageViewReference = new WeakReference<>(imageView);
        renderScript = RenderScript.create(context);
    }

    // Decode image in background.
    @Override
    protected Bitmap doInBackground(Bitmap... params) {
        Bitmap bitmap = params[0];
        return blurBitmap(bitmap);
    }

    // Once complete, see if ImageView is still around and set bitmap.
    @Override
    protected void onPostExecute(Bitmap bitmap) {
        if (bitmap == null || isCancelled()) {
            return;
        }

        final ImageView imageView = imageViewReference.get();
        if (imageView == null) {
            return;
        }

        imageView.setImageBitmap(bitmap);
    }

    public Bitmap blurBitmap(Bitmap bitmap) {
        // https://plus.google.com/+MarioViviani/posts/fhuzYkji9zz

        //Let's create an empty bitmap with the same size of the bitmap we want to blur
        Bitmap outBitmap = Bitmap.createBitmap(bitmap.getWidth(), bitmap.getHeight(),
                Bitmap.Config.ARGB_8888);

        //Instantiate a new Renderscript


        //Create an Intrinsic Blur Script using the Renderscript
        ScriptIntrinsicBlur blurScript = ScriptIntrinsicBlur.create(renderScript, Element.U8_4(renderScript));

        //Create the in/out Allocations with the Renderscript and the in/out bitmaps
        Allocation allIn = Allocation.createFromBitmap(renderScript, bitmap);
        Allocation allOut = Allocation.createFromBitmap(renderScript, outBitmap);

        //Set the radius of the blur
        blurScript.setRadius(25.f);

        //Perform the Renderscript
        blurScript.setInput(allIn);
        blurScript.forEach(allOut);

        //Copy the final bitmap created by the out Allocation to the outBitmap
        allOut.copyTo(outBitmap);

        // recycle the original bitmap
        // nope, we are using the original bitmap as well :/
        if (shouldRecycleSource) {
            bitmap.recycle();
        }

        //After finishing everything, we destroy the Renderscript.
        renderScript.destroy();

        return outBitmap;
    }

    public boolean isShouldRecycleSource() {
        return shouldRecycleSource;
    }

    public void setShouldRecycleSource(boolean shouldRecycleSource) {
        this.shouldRecycleSource = shouldRecycleSource;
    }
}
