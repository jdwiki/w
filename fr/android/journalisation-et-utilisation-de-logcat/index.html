Paramètres #

Options Descriptif
-b (tampon) Charge un tampon de journal alternatif pour l’affichage, tel que les événements ou la radio. Le tampon principal est utilisé par défaut. Voir Affichage des tampons de journal alternatifs.
-c Efface (vide) tout le journal et quitte.
-d Affiche le journal à l’écran et quitte.
-f (nom de fichier) Écrit la sortie des messages du journal dans (nom de fichier). La valeur par défaut est stdout.
-g Imprime la taille du tampon de journal spécifié et quitte.
-n (compter) Définit le nombre maximal de journaux en rotation sur (nombre). La valeur par défaut est 4. Requiert l’option -r.
-r (ko) Fait pivoter le fichier journal tous les (Ko) de sortie. La valeur par défaut est 16. Nécessite l’option -f.
-s Définit la spécification de filtre par défaut sur silencieux.
-v (formater) Définit le format de sortie des messages de journal. Le format par défaut est bref.

Définition #

Logcat est un outil de ligne de commande qui vide un journal des messages système, y compris les traces de la pile lorsque l’appareil génère une erreur et les messages que vous avez écrits à partir de votre application avec la classe [Log][1].

Quand utiliser #

Si vous envisagez d’utiliser les méthodes System.out de Java pour imprimer sur la console au lieu d’utiliser l’une des méthodes Log d’Android, sachez qu’elles fonctionnent essentiellement de la même manière. Cependant, il est préférable d’éviter d’utiliser les méthodes de Java car les informations supplémentaires et le formatage fournis par les méthodes de journalisation d’Android sont plus avantageux. De plus, les méthodes d’impression System.out sont [redirigées][3] vers la méthode Log.i().

Filtrage de la sortie logcat #

Il est utile de filtrer la sortie logcat car de nombreux messages ne présentent aucun intérêt. Pour filtrer la sortie, ouvrez le “Moniteur Android” et cliquez sur le menu déroulant en haut à droite et sélectionnez Modifier la configuration du filtre

[![Moniteur Android][1]][1]

Vous pouvez désormais ajouter des filtres personnalisés pour afficher les messages qui vous intéressent, ainsi que filtrer les lignes de journal bien connues qui peuvent être ignorées en toute sécurité. Pour ignorer une partie de la sortie, vous pouvez définir une [expression régulière][2]. Voici un exemple d’exclusion de balises correspondantes :

^(?!(HideMe|AndThis))

Ceci peut être saisi en suivant cet exemple :

[![Créer un nouveau filtre logcat][3]][3]

Ce qui précède est une expression régulière qui exclut les entrées. Si vous vouliez ajouter une autre balise à la liste noire, ajoutez-la après un caractère pipe |. Par exemple, si vous vouliez mettre “GC” sur liste noire, vous utiliseriez un filtre comme celui-ci :

^(?!(HideMe|AndThis|GC))

Pour plus de documentation et d’exemples, visitez [Logging and using Logcat][4]

Journalisation #

Toute application Android de qualité gardera une trace de ce qu’elle fait via les journaux d’application. Ces journaux permettent une aide au débogage facile pour le développeur afin de diagnostiquer ce qui se passe avec l’application. La documentation Android complète peut être trouvée [ici][1], mais un résumé suit : #

Journalisation de base

La classe `Log` est la principale source d'écriture des journaux de développement, en spécifiant une `tag` et un `message`. La balise est ce que vous pouvez utiliser pour filtrer les messages du journal afin d'identifier les lignes provenant de votre activité particulière. Appelez simplement 
Log.v(String tag, String msg);

Et le système Android écrira un message au logcat :

07-28 12:00:00.759 24812-24839/my.packagename V/MyAnimator: Some log messages
 └ time stamp             |  app.package┘     |    └ any tag  |
     process & thread ids ┘          log level┘               └ the log message

ASTUCE :
Remarquez l’identifiant du processus et l’identifiant du thread. S’ils sont identiques, le journal provient du thread principal/UI !

N’importe quelle balise peut être utilisée, mais il est courant d’utiliser le nom de la classe comme balise :

public static final String tag = MyAnimator.class.getSimpleName();

Niveaux de journalisation

L'enregistreur Android a 6 niveaux différents, chacun servant un certain objectif : - `ERREUR` : `Log.e()` - Used to indicate critical failure, this is the level printed at when throwing an `Exception`. - `WARN` : `Log.w()` - Used to indicate a warning, mainly for recoverable failures - `INFO` : `Log.i()` - Used to indicate higher-level information about the state of the application - `DEBUG` : `Log.d()` - Used to log information that would be useful to know when debugging the application, but would get in the way when running the application - `VERBOSE` : `Log.v()` - Used to log information that reflects the small details about the state of the application - `ASSERT` : `Log.wtf()` - Used to log information about a condition that should never happen. - *wtf* stands for "What a Terrible Failure". ---

Motivation pour la journalisation

La motivation de la journalisation est de trouver facilement les erreurs, les avertissements et d'autres informations en jetant un coup d'œil à la chaîne d'événements de l'application. Par exemple, imaginez une application qui lit les lignes d'un fichier texte, mais suppose à tort que le fichier ne sera jamais vide. La trace du journal (d'une application qui ne se connecte pas) ressemblerait à ceci : 
E/MyApplication: Process: com.example.myapplication, PID: 25788
                          com.example.SomeRandomException: Expected string, got 'null' instead

Suivi d’un tas de traces de pile qui finiraient par conduire à la ligne incriminée, où passer avec un débogueur finirait par conduire au problème

Cependant, la trace du journal d’une application avec la journalisation activée pourrait ressembler à ceci :

V/MyApplication: Looking for file myFile.txt on the SD card
D/MyApplication: Found file myFile.txt at path <path>
V/MyApplication: Opening file myFile.txt
D/MyApplication: Finished reading myFile.txt, found 0 lines
V/MyApplication: Closing file myFile.txt
...
E/MyApplication: Process: com.example.myapplication, PID: 25788
                          com.example.SomeRandomException: Expected string, got 'null' instead

Un rapide coup d’œil aux journaux et il est évident que le fichier était vide.

Éléments à prendre en compte lors de la journalisation :

Bien que la journalisation soit un outil puissant qui permet aux développeurs Android de mieux comprendre le fonctionnement interne de leur application, la journalisation présente certains inconvénients.

Lisibilité du journal :

Il est courant que les applications Android aient plusieurs journaux exécutés de manière synchrone. En tant que tel, il est très important que chaque journal soit facilement lisible et ne contienne que les informations pertinentes et nécessaires.

Performances :

La journalisation nécessite une petite quantité de ressources système. En général, cela ne justifie pas de s'inquiéter, cependant, s'il est surutilisé, la journalisation peut avoir un impact négatif sur les performances de l'application.

Sécurité :

Récemment, plusieurs applications Android ont été ajoutées au marché Google Play qui permettent à l'utilisateur d'afficher les journaux de toutes les applications en cours d'exécution. Cet affichage involontaire de données peut permettre aux utilisateurs de visualiser des informations confidentielles. En règle générale, supprimez toujours les journaux contenant des données non publiques * avant * de publier votre application sur le marché.

Conclusion :

La journalisation est une partie essentielle d'une application Android, en raison du pouvoir qu'elle donne aux développeurs. La possibilité de créer une trace de journal utile est l'un des aspects les plus difficiles du développement de logiciels, mais la classe Log d'Android contribue à le rendre beaucoup plus facile.

Pour plus de documentation et d’exemples, visitez [Logging and using Logcat][2]

Utiliser le Logcat #

Logcat est un outil de ligne de commande qui vide un journal des messages système, y compris les traces de la pile lorsque l’appareil génère une erreur et les messages que vous avez écrits à partir de votre application avec la classe Log.

La sortie Logcat peut être affichée dans le moniteur Android d’Android Studio ou avec la ligne de commande adb.

Dans Android Studio

Afficher en cliquant sur l’icône “Android Monitor”: [![entrez la description de l’image ici][1]][1] Ou en appuyant sur Alt+6 sous Windows/Linux ou CMD+6 sur Mac.

via la ligne de commande :

Utilisation simplifiée :

$ adb logcat

Avec horodatage :

$ adb logcat -v time

Filtrer sur un texte spécifique :

$ adb logcat -v time | grep 'searchtext'

Il existe de nombreuses options et filtres disponibles pour logcat en ligne de commande, documentés [ici][2].
Un exemple simple mais utile est l’expression de filtre suivante qui affiche tous les messages de journal avec le niveau de priorité “error”, sur toutes les balises :

$ adb logcat *:E

[1] : http://i.stack.imgur.com/rKiCw.png [2] : https://developer.android.com/studio/command-line/logcat.html

Journal avec lien vers la source directement depuis Logcat #

C’est une bonne astuce pour ajouter un lien vers le code, il sera donc facile d’accéder au code qui a émis le journal.

Avec le code suivant, cet appel :

MyLogger.logWithLink("MyTag","param="+param);


Entraînera :

07-26...012/com.myapp D/MyTag: MyFrag:onStart(param=3)  (MyFrag.java:2366) // << logcat converts this to a link to source!

Voici le code (à l’intérieur d’une classe appelée MyLogger):

static StringBuilder sb0 = new StringBuilder(); // reusable string object

public static void logWithLink(String TAG, Object param) {
    StackTraceElement stack = Thread.currentThread().getStackTrace()[3];
    sb0.setLength(0);
    String c = stack.getFileName().substring(0, stack.getFileName().length() - 5); // removes the ".java"
    sb0.append(c).append(":");
    sb0.append(stack.getMethodName()).append('(');
    if (param != null) {
        sb0.append(param);
    }
    sb0.append(") ");
    sb0.append(" (").append(stack.getFileName()).append(':').append(stack.getLineNumber()).append(')');
    Log.d(TAG, sb0.toString());
}

Ceci est un exemple de base, il peut être facilement étendu pour émettre un lien vers l’appelant (indice : la pile sera [4] au lieu de [3]), et vous pouvez également ajouter d’autres informations pertinentes.

Génération du code de journalisation #

Les “modèles dynamiques” d’Android Studio peuvent offrir plusieurs raccourcis pour une journalisation rapide.
Pour utiliser les modèles Live, tout ce que vous avez à faire est de commencer à taper le nom du modèle et d’appuyer sur “TAB” ou sur Entrée pour insérer l’instruction.

Exemples:

  • logi → se transforme en → android.util.Log.i(TAG, "$METHOD_NAME$: $content$");
    • $METHOD_NAME$ will automatically be replaced with your method name, and the cursor will wait for the content to be filled.
  • loge → idem, pour erreur
  • etc. pour le reste des niveaux de journalisation.

La liste complète des modèles se trouve dans les paramètres de Android Studio (ALT+s et tapez “live”). Et il est également possible d’ajouter vos modèles personnalisés.

Si vous trouvez que les “modèles dynamiques” d’Android Studio ne suffisent pas à vos besoins, vous pouvez envisager [Android Postfix Plugin] [1]

Il s’agit d’une bibliothèque très utile qui vous aide à éviter d’écrire manuellement la ligne de journalisation.

La syntaxe est absolument simple :

.log - Journalisation. S’il y a une variable constante “TAG”, utilisez “TAG” . Sinon, il utilise le nom de la classe.
