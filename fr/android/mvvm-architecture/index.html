** Bizarreries de syntaxe avec DataBinding **

Lors de la liaison d’une fonction viewModel à une propriété dans xml, certains préfixes de fonction tels que get ou is sont supprimés. Par exemple. ViewModel::getFormattedText sur le ViewModel deviendra @{viewModel.formattedText} lors de sa liaison à une propriété en xml. De même avec ViewModel ::isContentVisible -> @{viewModel.contentVisible} (notation Java Bean)

Les classes de liaison générées telles que ActivityMainBinding sont nommées d’après le XML pour lequel elles créent des liaisons, et non la classe Java.

Reliures personnalisées

Dans activity_main.xml, j’ai défini l’attribut textColor sur app et non sur l’espace de noms android. Pourquoi donc? Parce qu’il existe un setter personnalisé défini pour l’attribut textColor qui résout un ID de ressource ColorRes envoyé par le ViewModel en une couleur réelle.

public class CustomBindings {

    @TargetApi(23)
    @BindingAdapter({"bind:textColor"})
    public static void setTextColor(TextView textView, int colorResId) {
        final Context context = textView.getContext();
        final Resources resources = context.getResources();
        final int apiVersion = Build.VERSION.SDK_INT;
        int color;

        if (apiVersion >= Build.VERSION_CODES.M) {
           color = resources.getColor(colorResId, context.getTheme());
        } else {
            color = resources.getColor(colorResId);
        }

        textView.setTextColor(color);
    }

}

Pour plus de détails sur la façon dont cela fonctionne, consultez [DataBinding Library : Custom Setters] [1]

** Attendez … il y a de la logique dans votre xml !!!? **

Vous pourriez dire que les choses que je fais dans xml pour android:visibility et app:textColor sont fausses/anti-modèles dans le contexte MVVM car il y a une logique de vue à mon avis. Cependant, je dirais qu’il est plus important pour moi de garder les dépendances Android hors de mon ViewModel pour des raisons de test.

De plus, que fait réellement app:textColor ? Il ne résout qu’un pointeur de ressource sur la couleur réelle qui lui est associée. Ainsi, le ViewModel décide toujours quelle couleur est affichée en fonction de certaines conditions.

En ce qui concerne android:visibility, je pense qu’en raison de la façon dont la méthode est nommée, il est en fait acceptable d’utiliser l’opérateur ternaire ici. En raison du nom isLoadingVisible et isContentVisible, il n’y a vraiment aucun doute sur ce que chaque résultat doit résoudre dans la vue. J’ai donc l’impression qu’il s’agit plutôt d’exécuter une commande donnée par le ViewModel que de vraiment faire de la logique de vue.

D’un autre côté, je conviens que l’utilisation de viewModel.isLoading ? View.VISIBLE : View.GONE serait une mauvaise chose à faire car vous faites des hypothèses dans la vue sur ce que cet état signifie pour la vue.

Exemple MVVM utilisant la bibliothèque DataBinding #

L’intérêt de MVVM est de séparer les couches contenant la logique de la couche de vue.

Sur Android, nous pouvons utiliser la [DataBinding Library] [1] pour nous aider et rendre la plupart de notre logique testable par unité sans nous soucier des dépendances Android.

Dans cet exemple, je vais montrer les composants centraux d’une simple application stupide qui fait ce qui suit :

  • Au démarrage, simulez un appel réseau et affichez un spinner de chargement
  • Afficher une vue avec un compteur de clics TextView, un message TextView et un bouton pour incrémenter le compteur
  • Sur le bouton, cliquez sur le compteur de mise à jour et mettez à jour la couleur du compteur et le texte du message si le compteur atteint un certain nombre

Commençons par la couche de vue :

activity_main.xml :

Si vous n’êtes pas familier avec le fonctionnement de DataBinding, vous devriez probablement [prendre 10 minutes][1] pour vous familiariser avec. Comme vous pouvez le voir, tous les champs que vous mettriez habituellement à jour avec des setters sont liés à des fonctions sur la variable viewModel.

Si vous avez une question sur les propriétés android:visibility ou app:textColor, consultez la section ‘Remarques’.

Ensuite, la couche modèle. Ici j’ai :

  • deux champs qui représentent l’état de l’application
  • getters pour lire le nombre de clics et l’état d’excitation
  • une méthode pour incrémenter mon nombre de clics
  • une méthode pour restaurer un état antérieur (important pour les changements d’orientation)

Aussi je définis ici un ‘état d’excitation’ qui dépend du nombre de clics. Cela sera utilisé plus tard pour mettre à jour la couleur et le message sur la vue.

Il est important de noter qu’aucune hypothèse n’est faite dans le modèle sur la façon dont l’état pourrait être affiché à l’utilisateur !

ClickerModel.java

import com.google.common.base.Optional;

import de.walled.mvvmtest.viewmodel.ViewState;

public class ClickerModel implements IClickerModel {

    private int numberOfClicks;
    private Excitement stateOfExcitement;

    public void incrementClicks() {
        numberOfClicks += 1;
        updateStateOfExcitement();
    }

    public int getNumberOfClicks() {
        return Optional.fromNullable(numberOfClicks).or(0);
    }

    public Excitement getStateOfExcitement() {
        return Optional.fromNullable(stateOfExcitement).or(Excitement.BOO);
    }

    public void restoreState(ViewState state) {
        numberOfClicks = state.getNumberOfClicks();
        updateStateOfExcitement();
    }

    private void updateStateOfExcitement() {
        if (numberOfClicks < 10) {
            stateOfExcitement = Excitement.BOO;
        } else if (numberOfClicks <= 20) {
            stateOfExcitement = Excitement.MEH;
        } else {
            stateOfExcitement = Excitement.WOOHOO;
        }
    }
}

Ensuite, le ViewModel.

Cela déclenchera des modifications sur le modèle et formatera les données du modèle pour les afficher sur la vue. Notez que c’est ici que nous évaluons quelle représentation graphique est appropriée pour l’état donné par le modèle (resolveCounterColor et resolveLabelText). Ainsi, nous pourrions par exemple facilement implémenter un UnderachieverClickerModel qui a des seuils inférieurs pour l’état d’excitation sans toucher à aucun code dans le viewModel ou la vue.

Notez également que le ViewModel ne contient aucune référence pour afficher des objets. Toutes les propriétés sont liées via les annotations @Bindable et mises à jour lorsque notifyChange() (signale que toutes les propriétés doivent être mises à jour) ou notifyPropertyChanged(BR.propertyName) (signale que ces propriétés doivent être mises à jour).

ClickerViewModel.java

import android.databinding.BaseObservable;

import android.databinding.Bindable;
import android.support.annotation.ColorRes;
import android.support.annotation.StringRes;
    
import com.android.databinding.library.baseAdapters.BR;
    
import de.walled.mvvmtest.R;
import de.walled.mvvmtest.api.IClickerApi;
import de.walled.mvvmtest.model.Excitement;
import de.walled.mvvmtest.model.IClickerModel;
import rx.Observable;

public class ClickerViewModel extends BaseObservable {

    private final IClickerApi api;
    boolean isLoading = false;
    private IClickerModel model;

    public ClickerViewModel(IClickerModel model, IClickerApi api) {
        this.model = model;
        this.api = api;
    }

    public void onClickIncrement() {
        model.incrementClicks();
        notifyChange();
    }

    public ViewState getViewState() {
        ViewState viewState = new ViewState();
        viewState.setNumberOfClicks(model.getNumberOfClicks());
        return viewState;
    }

    public Observable<ViewState> loadData() {
        isLoading = true;
        return api.fetchInitialState()
                .doOnNext(this::initModel)
                .doOnTerminate(() -> {
                    isLoading = false;
                    notifyPropertyChanged(BR.loadingVisible);
                    notifyPropertyChanged(BR.contentVisible);
                });
    }

    public void initFromSavedState(ViewState savedState) {
        initModel(savedState);
    }

    @Bindable
    public String getNumberOfClicks() {
        final int clicks = model.getNumberOfClicks();
        return String.valueOf(clicks);
    }

    @Bindable
    @StringRes
    public int getLabelText() {
        final Excitement stateOfExcitement = model.getStateOfExcitement();
        return resolveLabelText(stateOfExcitement);
    }

    @Bindable
    @ColorRes
    public int getCounterColor() {
        final Excitement stateOfExcitement = model.getStateOfExcitement();
        return resolveCounterColor(stateOfExcitement);
    }

    @Bindable
    public boolean isLoadingVisible() {
        return isLoading;
    }

    @Bindable
    public boolean isContentVisible() {
        return !isLoading;
    }

    private void initModel(final ViewState viewState) {
        model.restoreState(viewState);
        notifyChange();
    }

    @ColorRes
    private int resolveCounterColor(Excitement stateOfExcitement) {
        switch (stateOfExcitement) {
            case MEH:
                return R.color.yellow;
            case WOOHOO:
                return R.color.green;
            default:
                return R.color.red;
        }
    }

    @StringRes
    private int resolveLabelText(Excitement stateOfExcitement) {
        switch (stateOfExcitement) {
            case MEH:
                return R.string.label_indifferent;
            case WOOHOO:
                return R.string.label_excited;
            default:
                return R.string.label_negative;
        }
    }

}

Rassembler le tout dans l’activité !

Ici, nous voyons la vue initialiser le viewModel avec toutes les dépendances dont elle pourrait avoir besoin, qui doivent être instanciées à partir d’un contexte Android.

Une fois le viewModel initialisé, il est lié à la mise en page XML via DataBindingUtil (veuillez vérifier la section “Syntaxe” pour nommer les classes générées).

Notez que les abonnements sont souscrits sur cette couche car nous devons gérer leur désabonnement lorsque l’activité est interrompue ou détruite pour éviter les fuites de mémoire et les NPE. La persistance et le rechargement de viewState sur OrientationChanges sont également déclenchés ici

MainActivity.java

import android.databinding.DataBindingUtil;
import android.os.Bundle;
import android.support.v7.app.AppCompatActivity;

import de.walled.mvvmtest.R;
import de.walled.mvvmtest.api.ClickerApi;
import de.walled.mvvmtest.api.IClickerApi;
import de.walled.mvvmtest.databinding.ActivityMainBinding;
import de.walled.mvvmtest.model.ClickerModel;
import de.walled.mvvmtest.viewmodel.ClickerViewModel;
import de.walled.mvvmtest.viewmodel.ViewState;
import rx.Subscription;
import rx.subscriptions.Subscriptions;

public class MainActivity extends AppCompatActivity {

    private static final String KEY_VIEW_STATE = "state.view";

    private ClickerViewModel viewModel;
    private Subscription fakeLoader = Subscriptions.unsubscribed();

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        // would usually be injected but I feel Dagger would be out of scope
        final IClickerApi api = new ClickerApi();
        setupViewModel(savedInstanceState, api);

        ActivityMainBinding binding = DataBindingUtil.setContentView(this, R.layout.activity_main);
        binding.setViewModel(viewModel);
    }

    @Override
    protected void onPause() {
        fakeLoader.unsubscribe();
        super.onPause();
    }

    @Override
    protected void onDestroy() {
        fakeLoader.unsubscribe();
        super.onDestroy();
    }

    @Override
    protected void onSaveInstanceState(Bundle outState) {
        outState.putSerializable(KEY_VIEW_STATE, viewModel.getViewState());
    }

    private void setupViewModel(Bundle savedInstance, IClickerApi api) {
        viewModel = new ClickerViewModel(new ClickerModel(), api);
        final ViewState savedState = getViewStateFromBundle(savedInstance);

        if (savedState == null) {
            fakeLoader = viewModel.loadData().subscribe();
        } else {
            viewModel.initFromSavedState(savedState);
        }
    }

    private ViewState getViewStateFromBundle(Bundle savedInstance) {
        if (savedInstance != null) {
            return (ViewState) savedInstance.getSerializable(KEY_VIEW_STATE);
        }
        return null;
    }
}
