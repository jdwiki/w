Connectez ADB à un appareil via WiFi #

La configuration ADB standard implique une connexion USB à un périphérique physique. Si vous préférez, vous pouvez passer en mode TCP/IP et connecter ADB via WiFi à la place.

Appareil non rooté #

  1. Se connecter au même réseau :
  • Assurez-vous que votre appareil et votre ordinateur sont sur le même réseau.
  1. Connectez l’appareil à l’ordinateur hôte avec un câble USB.
  2. Connectez adb à l’appareil via le réseau :

Pendant que votre appareil est connecté à adb via USB, exécutez la commande suivante pour écouter une connexion TCP/IP sur un port (par défaut 5555) :

  • Tapez adb tcpip <port> (passez en mode TCP/IP).
  • Débranchez le câble USB de l’appareil cible.
  • Tapez adb connect <ip address>:<port> (le port est facultatif ; par défaut 5555).

Par exemple:

   adb tcpip 5555
   adb connect 192.168.0.101:5555

Si vous ne connaissez pas l’adresse IP de votre appareil, vous pouvez :

  • vérifiez l’IP dans les paramètres WiFi de votre appareil.
  • utilisez ADB pour découvrir l’IP (via USB):
    1. Connect the device to the computer via USB
    2. In a command line, type adb shell ifconfig and copy your device’s IP address

Pour revenir au débogage via USB, utilisez la commande suivante :

   adb usb

Vous pouvez également connecter ADB via WiFi en installant un plugin sur Android Studio. Pour ce faire, accédez à * Paramètres> Plugins * et Parcourir les dépôts, recherchez * ADB WiFi *, installez-le et rouvrez Android Studio. Vous verrez une nouvelle icône dans votre barre d’outils, comme illustré dans l’image suivante. Connectez l’appareil à l’ordinateur hôte via USB et cliquez sur cette icône * AndroidWiFiADB *. Il affichera un message indiquant si votre appareil est connecté ou non. Une fois connecté, vous pouvez débrancher votre clé USB.

[![Nouvelle icône de la barre d’outils][1]][1]

Appareil rooté #

Remarque : Certains appareils enracinés peuvent utiliser l’application ADB WiFi du Play Store pour l’activer de manière simple. De plus, pour certains appareils (en particulier ceux dotés de ROM CyanogenMod), cette option est présente dans les options du développeur parmi les paramètres. L’activer vous donnera l’adresse IP et le numéro de port requis pour vous connecter à adb en exécutant simplement adb connect <ip address>:<port>.

Lorsque vous avez un appareil rooté mais que vous n’avez pas accès à un câble USB #

Le processus est expliqué en détail dans la réponse suivante : http://stackoverflow.com/questions/2604727/how-can-i-connect-to-android-with-adb-over-tcp/3623727#3623727 Les commandes les plus importantes sont indiqués ci-dessous.

Ouvrez un terminal dans l’appareil et tapez ce qui suit :

su
setprop service.adb.tcp.port <a tcp port number>
stop adbd
start adbd

Par exemple:

setprop service.adb.tcp.port 5555

Et sur votre ordinateur :

adb connect <ip address>:<a tcp port number>

Par exemple:

adb connect 192.168.1.2:5555

Pour le désactiver :

setprop service.adb.tcp.port -1
stop adbd
start adbd

Éviter l’expiration du délai #

Par défaut, adb expirera après 5000 ms. Cela peut arriver dans certains cas, comme un WiFi lent ou un APK volumineux.

Un simple changement dans la configuration de Gradle peut faire l’affaire :

android {
    adbOptions {
        timeOutInMs 10 * 1000
    }
}

Commande ADB directe vers un appareil spécifique dans un environnement multi-appareils #

1. Cibler un appareil par numéro de série

Utilisez l’option -s suivie d’un nom de périphérique pour sélectionner sur quel périphérique la commande adb doit s’exécuter. Les options -s doivent être en première ligne, avant la commande.

adb -s <device> <command>

Exemple:

adb devices

List of devices attached
emulator-5554       device
02157df2d1faeb33    device

adb -s emulator-5554 shell

Exemple #2 :

adb devices -l

List of devices attached
06157df65c6b2633    device usb:1-3 product:zerofltexx model:SM_G920F device:zeroflte
LC62TB413962        device usb:1-5 product:a50mgp_dug_htc_emea model:HTC_Desire_820G_dual_sim device:htc_a50mgp_dug

adb -s usb:1-3 shell

2. Cibler un appareil, lorsqu’un seul type d’appareil est connecté

Vous pouvez cibler le seul émulateur en cours d’exécution avec -e

adb -e <command>

Ou vous pouvez cibler le seul périphérique USB connecté avec -d

adb -d <command>

Prendre une capture d’écran et une vidéo (pour kitkat uniquement) à partir de l’écran d’un appareil #

Capture d’écran : Option 1 (adb pur) #

La commande shell adb nous permet d’exécuter des commandes à l’aide du shell intégré d’un appareil. La commande shell screencap capture le contenu actuellement visible sur un appareil et l’enregistre dans un fichier image donné, par ex. /sdcard/screen.png :

adb shell screencap /sdcard/screen.png

Vous pouvez ensuite utiliser [la commande pull][pull] pour télécharger le fichier de l’appareil dans le répertoire actuel de votre ordinateur :

adb pull /sdcard/screen.png

[tirer] : https://www.wikiod.com/fr/android/bad-pont-de-debogage-android#Extraire (pousser) des fichiers depuis (vers) l’appareil “tirer”

Capture d’écran : Option 2 (plus rapide) #

Exécutez la ligne suivante :

(guimauve et antérieur) :

adb shell screencap -p | perl -pe 's/\x0D\x0A/\x0A/g' > screen.png

(Nougat et après):

adb shell screencap -p > screen.png

Le drapeau -p redirige la sortie de la commande screencap vers stdout. L’expression Perl dans laquelle il est redirigé résout certains problèmes de fin de ligne sur Marshmallow et les versions antérieures. Le flux est ensuite écrit dans un fichier nommé screen.png dans le répertoire courant. Voir cet article et [cet article](http://www.stkent.com/ 2016/08/28/capturing-Nougat-screenshots-using-adb-shell.html) pour plus d’informations.

Vidéo #

cela ne fonctionne que dans KitKat et via ADB uniquement. Cela ne fonctionne pas sous Kitkat Pour commencer à enregistrer l’écran de votre appareil, exécutez la commande suivante :

adb ​​shell screenrecord /sdcard/example.mp4, cette commande commencera à enregistrer l’écran de votre appareil en utilisant les paramètres par défaut et enregistrera la vidéo résultante dans un fichier au fichier /sdcard/example.mp4 sur votre appareil.

Lorsque vous avez terminé l’enregistrement, appuyez sur Ctrl + C (z sous Linux) dans la fenêtre d’invite de commande pour arrêter l’enregistrement d’écran. Vous pouvez ensuite trouver le fichier d’enregistrement d’écran à l’emplacement que vous avez spécifié. Notez que l’enregistrement d’écran est enregistré dans la mémoire de stockage interne de votre appareil, et non sur votre ordinateur.

Les paramètres par défaut consistent à utiliser la résolution d’écran standard de votre appareil, à encoder la vidéo à un débit binaire de 4 Mbps et à définir la durée d’enregistrement d’écran maximale sur 180 secondes. Pour plus d’informations sur les options de ligne de commande que vous pouvez utiliser, exécutez la commande suivante :

adb ​​shell screenrecord –help, Cela fonctionne sans rooter l’appareil. J’espère que cela t’aides.

Imprimer la liste détaillée des appareils connectés #

Pour obtenir une liste détaillée de tous les appareils connectés à adb, écrivez la commande suivante dans votre terminal :

adb devices -l

Exemple de sortie #

List of devices attached
ZX1G425DC6             device usb:336592896X product:shamu model:Nexus_6 device:shamu
013e4e127e59a868       device usb:337641472X product:bullhead model:Nexus_5X device:bullhead

Extraire (pousser) des fichiers de (vers) l’appareil #

Vous pouvez extraire (télécharger) des fichiers de l’appareil en exécutant la commande suivante :

adb pull <remote> <local>

Par exemple:

adb pull /sdcard/ ~/

Vous pouvez également transférer (télécharger) des fichiers de votre ordinateur vers l’appareil :

adb push <local> <remote>

Par exemple:

adb push ~/image.jpg /sdcard/

Exemple pour récupérer la base de données de l’appareil

sudo adb -d shell "run-as com.example.name cat /data/da/com.example.name /databases/DATABASE_NAME  > /sdcard/file

Voir logcat #

Vous pouvez exécuter logcat en tant que commande adb ou directement dans une invite de shell de votre émulateur ou appareil connecté. Pour afficher la sortie du journal à l’aide de adb, accédez au répertoire platform-tools/ du SDK et exécutez :

$ adb logcat

Vous pouvez également créer une connexion shell à un appareil, puis exécuter :

$ adb shell
$ logcat

Une commande utile est :

adb logcat -v threadtime

Cela affiche la date, l’heure d’invocation, la priorité, la balise et le PID et TID du thread émettant le message dans un format de message long.

Filtration

Les journaux Logcat ont ce qu’on appelle des niveaux de journal :

V — Verbeux, D — Débogage, I — Info, W — Avertissement, E — Erreur, F — Fatal, * S* — Silencieux

Vous pouvez également filtrer logcat par niveau de journal. Par exemple, si vous souhaitez uniquement afficher le niveau de débogage :

adb logcat *:D

Logcat peut être filtré par un nom de package, bien sûr vous pouvez le combiner avec le filtre de niveau de log :

adb logcat <package-name>:<log level>

Vous pouvez également filtrer le journal à l’aide de grep (plus d’informations sur le filtrage de la sortie logcat [ici][1]) :

adb logcat | grep <some text>

Sous Windows, le filtre peut être utilisé à l’aide de findstr, par exemple :

adb logcat | findstr <some text>

Pour afficher le tampon de journal alternatif [main|events|radio], exécutez le logcat avec l’option -b :

adb logcat -b radio

Enregistrer la sortie dans le fichier :

adb logcat > logcat.txt

Enregistrez la sortie dans le fichier tout en la regardant :

adb logcat | tee logcat.txt

Nettoyage des bûches :

adb logcat -c

[1] : https://www.wikiod.com/fr/android/premiers-pas-avec-android#Filtrage de la sortie logcat

Effacer les données de l’application #

On peut effacer les données utilisateur d’une application spécifique en utilisant adb :

adb shell pm clear <package>

C’est la même chose que pour parcourir les paramètres du téléphone, sélectionner l’application et appuyer sur le bouton d’effacement des données.

  • pm invoque le gestionnaire de paquets sur l’appareil
  • clear supprime toutes les données associées à un paquet

Afficher et extraire les fichiers de cache d’une application #

Vous pouvez utiliser cette commande pour répertorier les fichiers de votre propre apk débogable :

adb shell run-as <sample.package.id> ls /data/data/sample.package.id/cache

Et ce script pour extraire du cache, copiez d’abord le contenu sur la carte SD, extrayez-le puis supprimez-le à la fin :

#!/bin/sh
adb shell "run-as <sample.package.id> cat '/data/data/<sample.package.id>/$1' > '/sdcard/$1'"
adb pull "/sdcard/$1"
adb shell "rm '/sdcard/$1'"

Ensuite, vous pouvez extraire un fichier du cache comme ceci :

./pull.sh cache/someCachedData.txt

Obtenir le fichier de base de données via ADB

sudo adb -d shell "run-as com.example.name cat /data/da/com.example.name /databases/STUDENT_DATABASE  > /sdcard/file

Afficher les appareils disponibles #

Connecter l’appareil par IP #

Entrez ces commandes dans l’appareil Android [Terminal] [1]

su
setprop service.adb.tcp.port 5555
stop adbd
start adbd

Après cela, vous pouvez utiliser CMD et ADB pour vous connecter à l’aide de la commande suivante

adb connect 192.168.0.101.5555

Et vous pouvez le désactiver et remettre ADB en écoute sur USB avec

setprop service.adb.tcp.port -1
stop adbd
start adbd

Depuis un ordinateur, si vous avez déjà un accès USB (pas de root requis)

Il est encore plus facile de passer au Wi-Fi si vous disposez déjà d’un port USB. À partir d’une ligne de commande sur l’ordinateur sur lequel le périphérique est connecté via USB, émettez les commandes

adb tcpip 5555
adb connect 192.168.0.101:5555
  • Remplacez 192.168.0.101 par l’adresse IP de l’appareil *

[1] : https://play.google.com/store/apps/details?id=jackpal.androidterm&hl=fr

Envoi de la diffusion #

Il est possible d’envoyer une diffusion à BroadcastReceiver avec adb.

Dans cet exemple, nous envoyons une diffusion avec l’action com.test.app.ACTION et la chaîne supplémentaire dans le bundle 'foo'='bar' :

adb shell am broadcast -a action com.test.app.ACTION --es foo "bar"

Vous pouvez mettre n’importe quel autre type pris en charge dans le bundle, pas seulement les chaînes :

–ez  - booléen
–ei  - entier
–el  - longue
–ef  - float
–eu  - uri
–eia - tableau int (séparés par ‘,’)
–ela - tableau long (séparé par ‘,’)
–efa - tableau flottant (séparé par ‘,’)
–esa - tableau de chaînes (séparés par ‘,’)

Pour envoyer l’intention à un package/classe spécifique, le paramètre -n ou -p peut être utilisé.
Envoi au colis :

-p com.test.app

Envoi à un composant spécifique (classe SomeReceiver dans com.test.app package):

-n com.test.app/.SomeReceiver

Exemples utiles :

Installer et exécuter une application #

Pour installer un fichier APK, utilisez la commande suivante :

adb install path/to/apk/file.apk

ou si l’application existe et que nous voulons réinstaller

adb install -r path/to/apk/file.apk

Pour désinstaller une application, il faut préciser son package

adb uninstall application.package.name

Utilisez la commande suivante pour démarrer une application avec un nom de package fourni (ou une activité spécifique dans une application) :

adb shell am start -n adb shell am start <package>/<activity>

Par exemple, pour démarrer Waze :

adb shell am start -n adb shell am start com.waze/com.waze.FreeMapAppActivity

Sauvegarde #

Vous pouvez utiliser la commande adb backup pour sauvegarder votre appareil.

adb backup [-f <file>] [-apk|-noapk] [-obb|-noobb] [-shared|-noshared] [-all] 
           [-system|nosystem] [<packages...>]

-f <filename> spécifiez le nom du fichier par défaut : crée backup.ab dans le répertoire courant

-apk|noapk active/désactive la sauvegarde des .apks eux-mêmes par défaut : -noapk

-obb|noobb activer/désactiver la sauvegarde des fichiers supplémentaires par défaut : -noobb

-shared|noshared stockage partagé du périphérique de sauvegarde/contenu de la carte SD par défaut : -noshared

-all sauvegarde toutes les applications installées

-system|nosystem inclut les applications système par défaut : -system

<packages> une liste de packages à sauvegarder (par exemple com.example.android.myapp) (pas nécessaire si -all est spécifié)

Pour une sauvegarde complète de l’appareil, y compris tout, utilisez

adb backup -apk -obb -shared -all -system -f fullbackup.ab

Remarque : Effectuer une sauvegarde complète peut prendre beaucoup de temps.

Pour restaurer une sauvegarde, utilisez

adb restore backup.ab

Afficher les données internes d’une application (data/data/<sample.package.id>) sur un appareil #

Tout d’abord, assurez-vous que votre application peut être sauvegardée dans AndroidManifest.xml, c’est-à-dire que android:allowBackup n’est pas false.

Commande de sauvegarde :

adb -s <device_id> backup -noapk <sample.package.id>

Créez un tar avec la commande dd :

dd if=backup.ab bs=1 skip=24 | python -c "import zlib,sys;sys.stdout.write(zlib.decompress(sys.stdin.read()))" > backup.tar

Extrayez le goudron :

tar -xvf backup.tar

Vous pouvez ensuite afficher le contenu extrait.

Installer ADB sur le système Linux #

Comment installer le pont de débogage Android (ADB) sur un système Linux avec le terminal en utilisant les référentiels de votre distribution.

Installez sur le système Ubuntu/Debian via apt :

sudo apt-get update
sudo apt-get install adb

Installer sur le système Fedora/CentOS via yum :

sudo yum check-update
sudo yum install android-tools

Installer sur le système Gentoo avec portage :

sudo emerge --ask dev-util/android-tools

Installer sur le système openSUSE avec zypper :

sudo zypper refresh
sudo zypper install android-tools

Installer sur le système Arch avec pacman :

sudo pacman -Syyu
sudo pacman -S android-tools

Lire les informations sur l’appareil #

Ecrivez la commande suivante dans votre terminal :

adb shell getprop

Cela imprimera toutes les informations disponibles sous la forme de paires clé/valeur.

Vous pouvez simplement lire des informations spécifiques en ajoutant le nom d’une clé spécifique à la commande. Par exemple:

adb shell getprop ro.product.model

Voici quelques informations intéressantes que votre chat obtient :

- ro.product.model : nom du modèle de l’appareil (par exemple, Nexus 6P)

  • ro.build.version.sdk : niveau d’API de l’appareil (par exemple, 23)
  • ro.product.brand : marque de l’appareil (par exemple, Samsung)

Redémarrer l’appareil #

Vous pouvez redémarrer votre appareil en exécutant la commande suivante :

adb reboot

Exécutez cette commande pour redémarrer dans le bootloader :

adb reboot bootloader

Redémarrez en mode de récupération :

adb reboot recovery

Soyez conscient que l’appareil ne s’éteindra pas en premier !
