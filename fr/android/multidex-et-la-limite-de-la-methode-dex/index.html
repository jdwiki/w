DEX désigne les fichiers de bytecode exécutables de l’application Android (APK) sous la forme de fichiers Dalvik Executable (DEX), qui contiennent le code compilé utilisé pour exécuter votre application.

La spécification Dalvik Executable limite le nombre total de méthodes pouvant être référencées dans un seul fichier DEX à 65 536 (64 Ko), y compris les méthodes du framework Android, les méthodes de la bibliothèque et les méthodes dans votre propre code.

Pour surmonter cette limite, vous devez configurer le processus de création de votre application pour générer plusieurs fichiers DEX, appelés Multidex.

Qu’est-ce que le dex? #

Dex est le nom du format de fichier et de l’encodage dans lequel le code Android Java est compilé. Les premières versions d’Android chargeaient et exécutaient les binaires “dex” directement dans une machine virtuelle nommée Dalvik. Les versions plus récentes d’Android utilisent Android Runtime (ART), qui traite les fichiers “dex” comme une représentation intermédiaire et effectue d’autres compilations dessus avant d’exécuter l’application.

Dex est un format de fichier très ancien, en termes de durée de vie des smartphones, et a été conçu pour les appareils dont la mémoire principale était mesurée en dizaines de mégaoctets. Les limites de conception de ces jours sont restées avec nous à ce jour.

Le problème: #

Le format de fichier dex encode une limite au nombre de méthodes pouvant être référencées dans un seul binaire. Étant donné que la partie du format de fichier qui stocke le nombre de références est longue de deux octets, le nombre maximal de références de méthode est 0xFFFF ou 65535. Si une application contient plus que ce nombre de références de méthode, sa compilation échouera.

Que faire à ce sujet : #

Google a fourni un moyen de contourner ce problème, appelé Multidex. Il a des composants de compilation et d’exécution. Comme son nom l’indique, au moment de la compilation, il divisera le code entre un ou plusieurs fichiers “dex”. Au moment de l’exécution, il apprendra au ClassLoader par défaut comment rechercher des classes à partir de ces fichiers.

Cette approche fonctionne bien sur les appareils plus récents, mais présente des inconvénients importants. Cela peut augmenter considérablement le temps de démarrage des applications et, sur les appareils plus anciens, peut provoquer des échecs “L’application ne répond pas”.

Le multidex, bien qu’efficace, doit être évité si possible.

Comment éviter la limite : #

Avant de configurer votre application pour permettre l’utilisation de 64 Ko ou plus de références de méthode, vous devez prendre des mesures pour réduire le nombre total de références appelées par votre code d’application, y compris les méthodes définies par votre code d’application ou les bibliothèques incluses. Les stratégies suivantes peuvent vous aider à éviter d’atteindre la limite de référence dex :

  • Examinez les dépendances directes et transitives de votre application - Assurez-vous que toute dépendance de bibliothèque importante que vous incluez dans votre application est utilisée d’une manière qui dépasse la quantité de code ajoutée à l’application. Un anti-modèle courant consiste à inclure une très grande bibliothèque car quelques méthodes utilitaires étaient utiles. La réduction des dépendances au code de votre application peut souvent vous aider à éviter la limite de référence dex.
  • Supprimez le code inutilisé avec ProGuard - Configurez les [paramètres ProGuard][1] pour que votre application exécute ProGuard et assurez-vous que la réduction est activée pour les versions de version. L’activation de la réduction garantit que vous n’envoyez pas de code inutilisé avec vos APK.

Le premier point demande de la diligence et de la discipline de la part du développeur. Lors de l’intégration de bibliothèques tierces, il faut tenir compte de la taille de la bibliothèque. Par exemple, deux bibliothèques JSON populaires sont Jackson et Gson. Fonctionnellement, ils sont assez similaires, mais Gson a tendance à être plus utilisé dans Android. L’une des raisons est que Jackson pèse environ 9 000 méthodes, tandis que Gson en fournit 1 900.

Plusieurs outils sont disponibles pour aider les développeurs à suivre la taille de leur application :

  • dexcount-gradle-plugin signale le nombre de références de méthode dans votre APK ou AAR sur chaque build
  • dex-method-counts est un outil en ligne de commande qui compte le nombre de références de méthode dans un APK
  • www.methodscount.com est un service Web qui comptera les références de méthode dans n’importe quel fichier APK que vous téléchargez.

[1] : https://www.wikiod.com/fr/android-gradle/code-de-reduction-et-ressources

Activer le multi-index #

Pour activer une configuration multidex, vous avez besoin :

  • pour changer votre configuration de build Gradle
  • pour utiliser un MultiDexApplication ou activer le MultiDex dans votre classe Application

## Configuration progressive

Dans app/build.gradle ajoutez ces parties :

android {
    compileSdkVersion 24
    buildToolsVersion "24.0.1"

    defaultConfig {
        ...
        minSdkVersion 14
        targetSdkVersion 24
        ...

        // Enabling multidex support.
        multiDexEnabled true
    }
    ...
}

dependencies {
  compile 'com.android.support:multidex:1.0.1'
}

Activer MultiDex dans votre application #

Procédez ensuite à l’une des trois options suivantes :

  • https://www.wikiod.com/fr/android/multidex-et-la-limite-de-la-methode-dex#Multidex en étendant l’application

  • https://www.wikiod.com/fr/android/multidex-et-la-limite-de-la-methode-dex#Multidex en étendant MultiDexApplication

  • https://www.wikiod.com/fr/android/multidex-et-la-limite-de-la-methode-dex#Multidex en utilisant directement MultiDexApplication

Lorsque ces paramètres de configuration sont ajoutés à une application, les outils de génération Android construisent un dex principal (classes.dex) et un support (classes2.dex, classes3.dex) selon les besoins. Le système de construction les regroupera ensuite dans un fichier APK pour distribution.

[1] : https://developer.android.com/reference/android/app/Application.html [2] : https://developer.android.com/reference/android/support/multidex/MultiDexApplication.html

Multidex en étendant l’application #

Utilisez cette option si votre projet nécessite une sous-classe Application.

Spécifiez cette sous-classe Application en utilisant la propriété android:name dans le fichier manifeste à l’intérieur de la balise application.

Dans la sous-classe Application, ajoutez le remplacement de la méthode attachBaseContext(), et dans cette méthode, appelez MultiDex.install() :

package com.example;

import android.app.Application;
import android.content.Context;

/**
 * Extended application that support multidex 
 */
public class MyApplication extends Application {

    @Override
    protected void attachBaseContext(Context base) {
        super.attachBaseContext(base);
        MultiDex.install(this);
    }
}

Assurez-vous que la sous-classe Application est spécifiée dans la balise application de votre AndroidManifest.xml :

<application
    android:name="com.example.MyApplication"
    android:icon="@drawable/ic_launcher"
    android:label="@string/app_name">
</application>

Multidex en étendant MultiDexApplication #

Ceci est très similaire à l’utilisation d’une sous-classe Application et à la substitution de la méthode attachBaseContext().

Cependant, en utilisant cette méthode, vous n’avez pas besoin de remplacer attachBaseContext() car cela est déjà fait dans la superclasse MultiDexApplication.

Étendez MultiDexApplication au lieu de Application :

package com.example;

import android.support.multidex.MultiDexApplication;
import android.content.Context;

/**
 * Extended MultiDexApplication 
 */
public class MyApplication extends MultiDexApplication {

     // No need to override attachBaseContext()

     //..........
}

Ajoutez cette classe à votre AndroidManifest.xml exactement comme si vous étendiez Application :

<application
    android:name="com.example.MyApplication"
    android:icon="@drawable/ic_launcher"
    android:label="@string/app_name">
</application>

Multidex en utilisant directement MultiDexApplication #

Utilisez cette option si vous n’avez pas besoin d’une sous-classe Application.

C’est l’option la plus simple, mais de cette façon vous ne pouvez pas fournir votre propre sous-classe Application. Si une sous-classe “Application” est nécessaire, vous devrez passer à l’une des autres options pour le faire.

Pour cette option, spécifiez simplement le nom de classe complet android.support.multidex.MultiDexApplication pour la propriété android:name de la balise application dans le fichier AndroidManifest.xml :

<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.android.multidex.myapplication">
    <application
        ...
        android:name="android.support.multidex.MultiDexApplication">
        ...
    </application>
</manifest>

Comptage des références de méthode sur chaque build (plugin Dexcount Gradle) #

Le [plugin dexcount][1] compte les méthodes et le nombre de ressources de classe après une construction réussie.

Ajoutez le plugin dans app/build.gradle :

apply plugin: 'com.android.application'

buildscript {
    repositories {
        mavenCentral() // or jcenter()
    }

    dependencies {
        classpath 'com.getkeepsafe.dexcount:dexcount-gradle-plugin:0.5.5'
    }
}

Appliquez le plugin dans le fichier app/build.gradle :

apply plugin: 'com.getkeepsafe.dexcount'

Recherchez les données de sortie générées par le plugin dans :

../app/build/outputs/dexcount

Le graphique .html est particulièrement utile dans :

../app/build/outputs/dexcount/debugChart/index.html

[1] : https://github.com/KeepSafe/dexcount-gradle-plugin
