Migration du projet DLL ISAPI MFC extension (C++) VS2005 vers VS2015.

  • Exemple:

Sur cette page

  • Exemple:

Vous avez peut-être vu plusieurs sites Web qui montreront comment créer un projet d’extension ISAPI, mais aucun d’entre eux ne montrera comment migrer le projet d’extension ISAPI existant (VS2005) vers VS2015. J’avais rencontré un problème similaire alors que je travaillais sur l’une de ces exigences. Cet article montre la manière expérimentale que j’avais prise pour résoudre mon problème.

Je travaillais sur une tâche de migration où je suis tombé sur un projet. Le projet ISAPI hérité était un projet DLL d’extension MFC construit dans VS2005 et après avoir recherché quelques éléments sur Google, j’ai appris que les classes ISAPI MFC (CHttpServerContext, CHttpServer, etc.) ne sont pas livrées après VS2005. Microsoft recommande d’utiliser les fonctions ISAPI Entry-Point du kit de développement logiciel (SDK) Microsoft Internet Information Services (IIS) au lieu des classes MFC ISAPI. J’ai pensé que cela pourrait nécessiter une réécriture complète du projet. Cependant heureusement j’ai pris une voie expérimentale qui a fonctionné dans mon cas. J’ai pensé écrire cet article qui pourrait aider quelqu’un à résoudre le même problème.

L’objectif de cette rubrique est de démontrer les modifications requises pour supprimer la dépendance de MFC à partir de la DLL d’extension ISAPI MFC sans réécrire le code complet. L’exemple ci-dessus ne contient pas toute l’implémentation, c’est juste à des fins de démonstration.

Exemple: #

Objectif du projet : télécharger un fichier particulier en fonction de l’identifiant du message envoyé en tant que demande du client au serveur. Le fichier sera créé sur le serveur et transmis au client.

Ici, je n’ai pas partagé tout le code mais montré ce qui est nécessaire car le but est de démontrer les changements requis pour la migration.

Pour effectuer les changements. J’ai pris la référence de afxisapi.h, afxisapi.inl et HttpExt.h qui sont livrés avec VS2005.

  • Utilisez EXTENSION_CONTROL_BLOCK au lieu de CHttpServerContext. EXTENSION_CONTROL_BLOCK est une structure principale utilisée par IIS et l’extension ISAPI pour échanger les informations. Il contient toutes les informations sur la nouvelle demande.

  • Supprimez la dépendance de la classe CHttpServer de votre classe d’extension MFC. Supprimez le mappage de message d’analyse de la commande http s’il est utilisé dans votre projet. Les macros telles que ON_PARSE_COMMAND, BEGIN_PARSE_MAP, END_PARSE_MAP et DEFAULT_PARSE_COMMAND ont été supprimées après VS2005.

  • Implémentez votre propre fonction GetExtensionVersion, TerminateExtension et HttpExtensionProc dans votre classe d’extension MFC qui était précédemment dérivée de la classe CHttpSever. Ces fonctions

  • GetExtensionVersion() La fonction GetExtensionVersion est la première fonction de point d’entrée dans IIS. Cette fonction permet à votre extension ISAPI d’enregistrer ses informations de version auprès d’IIS.

    BOOL CISAPIExtension::GetExtensionVersion( HSE_VERSION_INFO* pVer ) { ISAPIVERIFY( ::LoadString( AfxGetResourceHandle(), IDS_SERVER, sz, HSE_MAX_EXT_DLL_NAME_LEN) );

         _tcscpy_s( pVer->lpszExtensionDesc, ( strlen( sz ) + 1U ), sz )

    }

  • HttpExtensionProc() : La fonction HttpExtensionProc est le point d’entrée principal d’une extension ISAPI appelée par IIS. Il expose les méthodes utilisées par IIS pour accéder aux fonctionnalités exposées par l’extension. Extrayez les informations de demande du bloc de contrôle d’extension et effectuez votre traitement.

  • Utilisez ServerSupportFunction de EXTENSION_CONTROL_BLOCK au lieu de la fonction TransmitFile de CHttepServerContext. Auparavant, TransmitFile() était utilisé pour le transfert de fichiers.

  • Utilisez HSE_REQ_TRANSMIT_FILE dans ServerSupportFunction pour le transfert de fichiers.

  • Extrayez les informations du fichier de transmission dans la structure HSE_TF_INFO.

  • Créez une fonction de rappel (AsyncIOCompletionFunction) qui sera appelée à la fin des E/S asynchrones.

    void WINAPI AsyncIOCompletionFunction( EXTENSION_CONTROL_BLOCK * pECB, PVOID pContext, DWORD cbIO, DWORD dwError) { HSE_CUSTOM_ERROR_INFO* pErrorInfo = (HSE_CUSTOM_ERROR_INFO*)pContext; DWORD dwIOStatus = dwError == NO_ERROR ? HSE_STATUS_SUCCESS : HSE_STATUS_ERROR;

      if (NULL != pErrorInfo)
  {
      //
      // log HTTP status code
      //
      pECB->dwHttpStatusCode = atoi(pErrorInfo->pszStatus);

      delete pErrorInfo;
      pErrorInfo = NULL;
  }

  pECB->ServerSupportFunction(
      pECB->ConnID,
      HSE_REQ_DONE_WITH_SESSION,

      (LPDWORD)NULL);
