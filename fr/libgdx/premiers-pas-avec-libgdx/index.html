Installation ou configuration #

LibGDX a une configuration assez simple, à l’aide d’un simple programme Java. Vous pouvez trouver le téléchargement [ici][1]. Lorsque vous démarrez l’application, elle ressemblera à ceci :[![Example of LibGDX Project Generator][2]][2]
Remarque : cette capture d’écran a été prise sous Linux et montre un chemin différent d’une installation Windows. Cependant, le formulaire est le même sur tout système d’exploitation compatible avec cette application d’installation

Champs de saisie #

Dans la zone de saisie “Nom”, est le nom du jeu pour votre projet. La zone de saisie “Package”, est le package de votre classe principale. La zone de saisie “Game Class” est la classe principale qui est appelée lorsque votre jeu est lancé. La zone de saisie “Destination”, est la destination vers laquelle votre projet va être généré. La zone de saisie “Andriod SDK”, le chemin d’accès à l’emplacement de votre SDK Android. Cette zone de saisie est totalement facultative, donc si vous ne souhaitez pas déployer votre application sur Android, vous n’avez pas à vous en soucier.

Sous-projets #

Les sous-projets ne sont que les plates-formes sur lesquelles ils doivent être déployés. Cette partie est assez explicite. Si vous ne souhaitez pas le déployer en HTML par exemple, décochez simplement la case à cocher.

Les extensions sont les extensions officielles de LibGDX. Voici un tableau vous indiquant ce que chacun est:

Dans la section Avancé, vous pouvez définir plusieurs paramètres et générer des fichiers de projet supplémentaires pour Eclipse et IDEA IDE.

Génération #

Une fois que tous vos paramètres sont corrects, vous pouvez appuyer sur le bouton “Générer”. Cela peut prendre quelques secondes, mais cela générera les fichiers de base et Gradle nécessaires à votre projet. Une fois que vous avez terminé, il est temps d’importer le projet dans votre IDE.

LibGDX Bonjour le monde #

Les bases

Les projets générés contiennent une application basique de type Hello World déjà implémentée.

Le projet principal est le projet principal, qui contient tout le code indépendant de la plate-forme. Ceci est obligatoire, mais en fonction de vos paramètres de génération, vous pouvez avoir plusieurs autres projets pour chaque plate-forme que vous avez sélectionnée.

L’exemple

Ouvrez com.mygdx.game.MyGdxGame.java dans le projet core. Vous verrez le code suivant :

public class MyGdxGame extends ApplicationAdapter {
    SpriteBatch batch;
    Texture img;
    
    @Override
    public void create () {
        batch = new SpriteBatch();
        img = new Texture("badlogic.jpg");
    }

    @Override
    public void render () {
        Gdx.gl.glClearColor(1, 0, 0, 1);
        Gdx.gl.glClear(GL20.GL_COLOR_BUFFER_BIT);
        batch.begin();
        batch.draw(img, 0, 0);
        batch.end();
    }
    
    @Override
    public void dispose () {
        batch.dispose();
        img.dispose();
    }
}

Bien qu’il s’agisse de votre projet principal, vous ne l’exécuterez pas directement, vous devez toujours exécuter le lanceur spécifique à la plate-forme, pour le bureau, il s’appelle com.mygdx.game.desktop.DesktopLauncher.java dans le projet desktop.

public class DesktopLauncher {
    public static void main (String[] arg) {
        LwjglApplicationConfiguration config = new LwjglApplicationConfiguration();
        new LwjglApplication(new MyGdxGame(), config);
    }
}

Dans cette classe, vous pouvez définir des paramètres spécifiques à la plate-forme.

Exécuter votre jeu #

Eclipse et Intellij utilisent deux méthodes différentes pour exécuter votre projet. Trouvez l’IDE que vous utilisez ci-dessous.

Exécution avec Eclipse #

En utilisant eclipse, vous pouvez exécuter votre application en exécutant cette classe en tant qu’application Java (clic droit sur le projet -> Exécuter en tant que -> Application Java). Vous verrez la fenêtre suivante :

[![Exécution de Hello world sur ordinateur][1]][1]

Exécution avec Intellij Idea #

Dans Intellij, vous devrez effectuer une configuration d’exécution. Pour cela, il faut cliquer sur le bouton en haut à droite qui ressemble à une carotte descendante :

[![Carotte][2]][2]

Cliquez ensuite sur le bouton “Modifier les configurations…”. Un écran de toutes vos configurations d’exécution actuelles s’affichera. Cliquez sur le “+” en haut à gauche de cette fenêtre et sélectionnez l’option “Application”. À partir de là, sélectionnez le fichier “DesktopLauncher” pour l’option “Classe principale”, définissez l’option “Utiliser le chemin de classe du module” sur l’option principale et définissez “Répertoire de travail” sur le dossier des ressources dans votre dossier principal. Le produit final devrait ressembler à ceci :

[![Exécuter un exemple de configuration][3]][3]

Une fois que vous avez fait cela, vous pouvez sélectionner un nom pour votre configuration d’exécution et cliquer sur “Appliquer”, puis sur “OK”. Une fois que vous avez terminé, vous pouvez cliquer sur l’icône de course verte en haut à droite :

[![Icône Exécuter][4]][4]

Que se passe-t-il dans la classe MyGdxGame ? #

Tout d’abord, la méthode create est appelée, qui initialisera le lot qui dessine à l’écran. Ensuite, la méthode charge le fichier badlogic.jpg dans la mémoire.

Après cela, la méthode render est appelée à plusieurs reprises jusqu’à ce que l’application soit arrêtée. Cette méthode réinitialisera la couleur d’arrière-plan au rouge et dessinera l’image à l’écran. Comme vous pouvez le voir, vous devez toujours commencer et terminer le dessin par lots. Enfin, lorsque l’application est sur le point de s’arrêter, la méthode dispose est appelée, ce qui libère de l’espace mémoire utilisé par la texture et le batch.

(Il est bon de savoir que la suppression peut également se produire au moment de l’exécution sur d’autres plates-formes, par exemple Android, car Android peut libérer de l’espace mémoire lorsque les applications sont en arrière-plan, mais il s’agit d’un sujet plus avancé)

Fichier introuvable : badlogic.jpg (interne)

Présentation générale de LibGDX #

LibGDX est une bibliothèque de développement de jeux open source gratuite développée en Java. Ses objectifs sont de permettre aux utilisateurs de développer des jeux multiplateformes qui s’exécutent sur les navigateurs de bureau, Android, iOS et Web. Écrivez le code une fois, déployez-le sur l’une des principales plates-formes.

Ajout de la prise en charge du code spécifique à la plate-forme #

LibGDX est conçu de manière à ce que vous puissiez écrire le même code et le déployer sur plusieurs plates-formes différentes. Pourtant, il y a des moments où vous souhaitez accéder au code spécifique à la plate-forme. Par exemple, si vous avez des classements et des réalisations dans votre jeu, vous pouvez utiliser des outils spécifiques à la plate-forme (comme Google Play Games) en plus de les stocker localement. Ou vous voulez utiliser une base de données, ou quelque chose de complètement différent.

Vous ne pouvez pas ajouter ce type de code dans le module principal. La première étape consiste donc à créer une interface. Créez-le dans le module principal. Ce premier est un utilitaire pour gérer d’autres interfaces :

public interface PlatformWrapper{
    //here you can also add other things you need that are platform specific.
    //If you want to create your own file saver for an instance, a JSON based database,
    //achievements, leaderboards, in app purchases and anything else you need platform specific code for.
    SQLWrapper getSQLSaver();//This one will be used in this example
    AchievementWrapper getAchievementHandler();//this line is here as an example
}

Ensuite, nous devons créer une deuxième interface, le SQLWrapper. Celui-ci va également dans le module de base.

public interface SQLWrapper{
    void init(String DATABASE);
    void saveSerializable(int id, Object o);
    Object loadSerializable(int id, Object o);
    void saveString(int id, String s);
    //.... and other methods you need here. This is something that varies
    //with usage. You may not need the serializable methods, but really need a custom method for saving 
    //an entire game world. This part is entirely up to you to find what methods you need
   
    //With these three methods, always assume it is the active database in question. Unless 
    //otherwise specified, it most likely is
    String getActiveDatabase();
    void deleteDatabase();
    void deleteTable(String table);//delete the active table
}

Maintenant, vous devez entrer dans chaque projet et créer une classe pour implémenter PlatformWrapper et une autre pour implémenter SQLWrapper. Dans chaque projet, vous ajoutez le code nécessaire, comme des instances, des constructeurs, etc.

Après avoir remplacé toutes les interfaces que vous avez créées, assurez-vous qu’elles ont toutes une instance dans la classe qui implémente PlatformWrapper (et qu’il existe un getter). Enfin, vous modifiez le constructeur dans la classe principale. La classe principale est la classe que vous référencez depuis le lanceur. Il étend ApplicationAdapter, implémente ApplicationListener ou étend Game. Modifiez le constructeur et ajoutez PlatformWrapper comme argument. À l’intérieur du wrapper de la plate-forme, vous avez quelques utilitaires (si vous en avez ajouté) en plus de tous les autres wrappers (sql, réalisations ou tout autre que vous avez ajouté).

Maintenant, si tout est correctement configuré, vous pouvez appeler le PlatformWrapper et obtenir l’une des interfaces multiplateformes. Appelez n’importe quelle méthode et (en supposant que le code exécuté est correct) vous verrez sur n’importe quelle plate-forme, il fait ce qu’il est censé faire avec le code spécifique à la plate-forme

Gestionnaire d’actifs #

L’AssetManager est une classe qui vous aide à gérer vos actifs.

Tout d’abord, vous devez créer une instance :

AssetManager am = new AssetManager();

Une fois celui-ci initialisé et avant de rendre quoi que ce soit, vous souhaitez obtenir les ressources :

am.load("badlogic.jpg", Texture.class);//Texture.class is the class this asset is of. If it is a 
//sound asset, it doesn't go under Texture. if it is a 3D model, it doesn't go under Texture.class
//Which class added depends on the asset you load

//... other loading here ...//

//when finished, call finishLoading:
am.finishLoading();

Maintenant, où que vous vouliez obtenir badlogic.jpg :

Texture texture = am.get("badlogic.jpg");
//Ready to render! The rendering itself is in the normal way

L’utilisation d’AssetManager vous permet de les charger une fois dans la mémoire de l’AssetManager, puis de les récupérer autant de fois que vous le souhaitez.

Pourquoi utiliser Asset Manager ? (depuis le wiki) :

AssetManager (code) vous aide à charger et à gérer vos actifs. C’est la méthode recommandée pour charger vos actifs, en raison des comportements agréables suivants :

  • Le chargement de la plupart des ressources se fait de manière asynchrone, vous pouvez donc afficher un écran de chargement réactif pendant le chargement
  • Les actifs sont comptés en référence. Si deux actifs A et B dépendent tous les deux d’un autre actif C, C ne sera pas cédé tant que A et B n’auront pas été cédés. Cela signifie également que si vous chargez une ressource plusieurs fois, elle sera en fait partagée et n’occupera de la mémoire qu’une seule fois !
  • Un seul endroit pour stocker tous vos actifs.
  • Permet d’implémenter de manière transparente des choses comme les caches (voir FileHandleResolver ci-dessous)

Le graphe de scène 2D #

Lorsque vous démarrez avec Java ou Android, vous apprenez rapidement que (0,0) se trouve dans le coin supérieur gauche. Dans LibGDX, cependant, (0,0) est par défaut dans le coin inférieur gauche.

À l’aide d’une caméra orthographique, vous pouvez placer (0, 0) dans le coin supérieur gauche. Bien que par défaut, (0, 0) se trouve dans le coin inférieur gauche. C’est quelque chose qu’il est important de savoir, car cela change également le coin des textures qui a les coordonnées X et Y.
