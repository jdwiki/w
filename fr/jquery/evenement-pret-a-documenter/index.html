Qu’est-ce que document-ready et comment dois-je l’utiliser ? #

Le code jQuery est souvent enveloppé dans jQuery(function($) { ... }); de sorte qu’il ne s’exécute qu’après la fin du chargement du DOM.

<script type="text/javascript"> 
  jQuery(function($) {
    // this will set the div's text to "Hello".
    $("#myDiv").text("Hello");
  });
</script>

<div id="myDiv">Text</div>

Ceci est important car jQuery (et JavaScript en général) ne peut pas sélectionner un élément DOM qui n’a pas été rendu sur la page.

<script type="text/javascript">
  // no element with id="myDiv" exists at this point, so $("#myDiv") is an
  // empty selection, and this will have no effect
  $("#myDiv").text("Hello");
</script>

<div id="myDiv">Text</div>

Notez que vous pouvez créer un alias pour l’espace de noms jQuery en passant un gestionnaire personnalisé dans la méthode .ready(). Ceci est utile dans les cas où une autre bibliothèque JS utilise le même alias $ raccourci que jQuery, ce qui crée un conflit. Pour éviter ce conflit, vous devez appeler $.noConflict(); - Cela vous oblige à utiliser uniquement l’espace de noms par défaut jQuery (au lieu de l’alias court $). En passant un gestionnaire personnalisé au gestionnaire .ready(), vous pourrez choisir le nom d’alias à utiliser jQuery.

$.noConflict();

jQuery( document ).ready(function( $ ) {
   // Here we can use '$' as jQuery alias without it conflicting with other 
   // libraries that use the same namespace 
   $('body').append('<div>Hello</div>')
});

jQuery( document ).ready(function( jq ) {
   // Here we use a custom jQuery alias 'jq' 
   jq('body').append('<div>Hello</div>')
});

Plutôt que de simplement mettre votre code jQuery au bas de la page, l’utilisation de la fonction $(document).ready garantit que tous les éléments HTML ont été rendus et que l’ensemble du modèle d’objet de document (DOM) est prêt pour l’exécution du code JavaScript.

jQuery 2.2.3 et versions antérieures #

Ceux-ci sont tous équivalents, le code à l’intérieur des blocs s’exécutera lorsque le document sera prêt :

$(function() {
  // code
});

$().ready(function() {
  // code
});

$(document).ready(function() {
  // code
});

Parce que ceux-ci sont équivalents, le premier est le formulaire recommandé, ce qui suit est une version de celui-ci avec le mot-clé jQuery au lieu du $ qui produit les mêmes résultats :

jQuery(function() {
  // code
});

jQuery 3.0 #

Notation #

Depuis jQuery 3.0, seul ce formulaire est recommandé :

jQuery(function($) {
  // Run when document is ready
  // $ (first argument) will be internal reference to jQuery
  // Never rely on $ being a reference to jQuery in the global namespace
});

Tous les autres gestionnaires prêts à documenter [sont obsolètes dans jQuery 3.0.][1]

Asynchrone #

À partir de jQuery 3.0, le gestionnaire ready [sera toujours appelé de manière asynchrone][2]. Cela signifie que dans le code ci-dessous, le journal “gestionnaire externe” sera toujours affiché en premier, que le document soit prêt ou non au moment de l’exécution.

$(function() {
  console.log("inside handler");
});
console.log("outside handler");

> gestionnaire extérieur > gestionnaire interne

[1] : https://jquery.com/upgrade-guide/3.0/#deprecated-document-ready-handlers-other-than-jquery-function [2] : https://jquery.com/upgrade-guide/3.0/#breaking-change-document-ready-handlers-are-now-asynchronous

Différence entre $(document).ready() et $(window).load() #

$(window).load() était obsolète dans jQuery version 1.8 (et complètement supprimé de [jQuery 3.0][1]) et en tant que tel ne devrait plus être utilisé. Les raisons de la dépréciation sont notées sur la page jQuery sur cet événement

Mises en garde de l’événement load lorsqu’il est utilisé avec des images

Un défi courant que les développeurs tentent de résoudre à l’aide du raccourci .load() consiste à exécuter une fonction lorsqu’une image (ou une collection d’images) est complètement chargée. Il y a plusieurs mises en garde connues à ce sujet qui doivent être notées. Ceux-ci sont:

  • Cela ne fonctionne pas de manière cohérente ni fiable entre les navigateurs
  • Il ne se déclenche pas correctement dans WebKit si l’image src est définie sur le même src qu’auparavant
  • Il ne remonte pas correctement l’arborescence DOM
  • Peut cesser de se déclencher pour les images qui vivent déjà dans le cache du navigateur

Si vous souhaitez toujours utiliser load(), cela est documenté ci-dessous :

$(document).ready() attend que le DOM complet soit disponible – tous les éléments du HTML ont été analysés et sont dans le document. Cependant, les ressources telles que les images peuvent ne pas être entièrement chargées à ce stade. S’il est important d’attendre que toutes les ressources soient chargées, $(window).load() et que vous êtes conscient des limitations importantes de cet événement, alors ce qui suit peut être utilisé à la place :

$(document).ready(function() {
  console.log($("#my_large_image").height()); // may be 0 because the image isn't available
});

$(window).load(function() {
  console.log($("#my_large_image").height()); // will be correct
});

[1] : https://jquery.com/upgrade-guide/3.0/#breaking-change-load-unload-and-error-removed

Attacher des événements et manipuler le DOM à l’intérieur de ready() #

Exemples d’utilisation de $(document).ready() :

  1. Associer des gestionnaires d’événements Attacher les gestionnaires d’événements jQuery

    $(document).ready(function() { $(“button”).click(function() { // Code for the click function }); });

  2. Exécutez le code jQuery après la création de la structure de la page

    jQuery(function($) { // set the value of an element. $("#myElement").val(“Hello”); });

  3. Manipuler la structure DOM chargée Par exemple : masquer un div lorsque la page se charge pour la première fois et l’afficher sur l’événement click d’un bouton

    $(document).ready(function() { $("#toggleDiv").hide(); $(“button”).click(function() { $("#toggleDiv").show(); }); });

Différence entre jQuery(fn) et l’exécution de votre code avant #

L’utilisation de l’événement prêt à documenter peut avoir de petits [inconvénients de performances][1], avec un retard d’exécution allant jusqu’à ~300 ms. Parfois, le même comportement peut être obtenu en exécutant du code juste avant la balise de fermeture </body> :

<body>
  <span id="greeting"></span> world!
  <script>
    $("#greeting").text("Hello");
  </script>
</body>

produira un comportement similaire mais s’exécutera plus tôt que car il n’attend pas le déclencheur d’événement document prêt comme il le fait dans :

<head>
  <script>
    jQuery(function($) {
      $("#greeting").text("Hello");
    });
  </script>
</head>
<body>
  <span id="greeting"></span> world!
</body>

Insistez sur le fait que le premier exemple repose sur votre connaissance de votre page et le placement du script juste avant la balise de fermeture </body> et plus précisément après la balise span.
