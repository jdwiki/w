Noms des variables #

Les variables contiennent des données. Nommez-les d’après ce pour quoi ils sont utilisés, pas d’après leur type de données ou leur portée, en utilisant un nom. Si vous vous sentez obligé de numéroter vos variables (par exemple, “chose1, chose2, chose3”), envisagez plutôt d’utiliser une structure de données appropriée (par exemple, un tableau, une “Collection” ou un “Dictionnaire”).

Noms de variables qui représentent un ensemble itérable de valeurs - par ex. un tableau, une Collection, un Dictionnaire ou une Plage de cellules doit être au pluriel.

Certaines conventions de nommage VBA courantes vont ainsi :

Pour les variables au niveau de la procédure :

CamelCase

Public Sub ExampleNaming(ByVal inputValue As Long, ByRef inputVariable As Long)

    Dim procedureVariable As Long
    Dim someOtherVariable As String

End Sub

Pour les variables au niveau du module :

CasPascal

Public GlobalVariable As Long
Private ModuleVariable As String

Pour les constantes :

SHOUTY_SNAKE_CASE est couramment utilisé pour différencier les constantes des variables :

Public Const GLOBAL_CONSTANT As String = "Project Version #1.000.000.001"
Private Const MODULE_CONSTANT As String = "Something relevant to this Module"

Public Sub SomeProcedure()

    Const PROCEDURE_CONSTANT As Long = 10

End Sub

Cependant, les noms “PascalCase” rendent le code plus propre et sont tout aussi bons, étant donné qu’IntelliSense utilise des icônes différentes pour les variables et les constantes :

![IntelliSense utilise différentes icônes pour Foo constant et Bar variable][1]

Notation hongroise #

Nommez-les d’après leur utilisation, pas d’après leur type de données ou leur portée.

*** “La notation hongroise permet de voir plus facilement quel est le type d’une variable” ***

Si vous écrivez votre code de telle sorte que les procédures adhèrent au * principe de responsabilité unique * (comme il se doit), vous ne devriez jamais regarder un écran de déclarations de variables en haut d’une procédure ; déclarez des variables aussi proches que possible de leur première utilisation, et leur type de données sera toujours bien en vue si vous les déclarez avec un type explicite. Le raccourci Ctrl+i du VBE peut également être utilisé pour afficher le type d’une variable dans une info-bulle.

Une variable est utilisée pour des informations beaucoup plus utiles que son type de données, * en particulier * dans un langage tel que VBA qui convertit joyeusement et implicitement un type en un autre selon les besoins.

Considérez iFile et strFile dans cet exemple :

Function bReadFile(ByVal strFile As String, ByRef strData As String) As Boolean
    Dim bRetVal As Boolean
    Dim iFile As Integer

    On Error GoTo CleanFail

    iFile = FreeFile
    Open strFile For Input As #iFile
    Input #iFile, strData

    bRetVal = True

CleanExit:
    Close #iFile
    bReadFile = bRetVal
    Exit Function
CleanFail:
    bRetVal = False
    Resume CleanExit
End Function

Comparer aux:

Function CanReadFile(ByVal path As String, ByRef outContent As String) As Boolean
    On Error GoTo CleanFail

    Dim handle As Integer
    handle = FreeFile

    Open path For Input As #handle
    Input #handle, outContent

    Dim result As Boolean
    result = True

CleanExit:
    Close #handle
    CanReadFile = result
    Exit Function
CleanFail:
    result = False
    Resume CleanExit
End Function

strData est passé ByRef dans l’exemple du haut, mais à part le fait que nous avons la chance de voir qu’il est explicitement passé comme tel, rien n’indique que strData est réellement renvoyé par la fonction.

L’exemple du bas le nomme outContent ; ce préfixe “out” est la raison pour laquelle la notation hongroise a été inventée : pour aider à clarifier à quoi sert une variable, dans ce cas pour l’identifier clairement comme un paramètre “out”.

Ceci est utile, car IntelliSense par lui-même n’affiche pas ByRef, même lorsque le paramètre est explicitement passé par référence :

![IntelliSense affichant “CanReadFile(ByVal path As String, outComment As String) As Boolean” lorsque la signature réelle spécifie outComment comme explicitement passé par référence (“ByRef”)][2]

Qui conduit à…

** Hongrois bien fait **

[La notation hongroise n’avait à l’origine rien à voir avec les types de variables] (http://www.joelonsoftware.com/articles/Wrong.html). En fait, la notation hongroise faite correctement est réellement utile. Considérez ce petit exemple (ByVal et As Integer supprimés pour brevety):

Public Sub Copy(iX1, iY1, iX2, iY2)
End Sub

Comparer aux:

Public Sub Copy(srcColumn, srcRow, dstColumn, dstRow)
End Sub

src et dst sont ici des préfixes de * notation hongroise *, et ils transmettent des informations * utiles * qui ne peuvent pas déjà être déduites des noms de paramètres ou d’IntelliSense nous montrant le type déclaré.

Bien sûr, il existe une meilleure façon de tout transmettre, en utilisant une * abstraction * appropriée et des mots réels qui peuvent être prononcés à haute voix et avoir un sens - à titre d’exemple artificiel :

Type Coordinate
    RowIndex As Long
    ColumnIndex As Long
End Type

Sub Copy(source As Coordinate, destination As Coordinate)
End Sub

Noms de procédure #

Procédures faire quelque chose. Nommez-les d’après ce qu’ils font, en utilisant un verbe. S’il n’est pas possible de nommer une procédure avec précision, il est probable que la procédure fait trop de choses et doit être décomposée en procédures plus petites et plus spécialisées.

Certaines conventions de nommage VBA courantes vont ainsi :

Pour toutes les procédures :

CasPascal

Public Sub DoThing()

End Sub

Private Function ReturnSomeValue() As [DataType]

End Function

Pour les procédures de gestionnaire d’événements :

NomObjet_NomÉvénement

Public Sub Workbook_Open()

End Sub

Public Sub Button1_Click()

End Sub

Les gestionnaires d’événements sont généralement nommés automatiquement par le VBE ; les renommer sans renommer l’objet et/ou l’événement géré cassera le code - le code s’exécutera et se compilera, mais la procédure du gestionnaire sera orpheline et ne sera jamais exécutée.

Membres booléens

Considérez une fonction de retour booléenne :

Function bReadFile(ByVal strFile As String, ByRef strData As String) As Boolean
End Function

Comparer aux:

Function CanReadFile(ByVal path As String, ByRef outContent As String) As Boolean
End Function

Le préfixe Can does sert le même but que le préfixe b : il identifie la valeur de retour de la fonction comme un Boolean. Mais ‘Can’ se lit mieux que ‘b’ :

If CanReadFile(path, content) Then

Par rapport à:

If bReadFile(strFile, strData) Then

Envisagez d’utiliser des préfixes tels que Can, Is ou Has devant les membres de retour booléen (fonctions et propriétés), mais uniquement lorsque cela ajoute de la valeur. Ceci est conforme aux directives de dénomination Microsoft actuelles.
