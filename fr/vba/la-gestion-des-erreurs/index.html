Éviter les conditions d’erreur #

Lorsqu’une erreur d’exécution se produit, un bon code doit la gérer. La meilleure stratégie de gestion des erreurs consiste à écrire du code qui vérifie les conditions d’erreur et évite simplement d’exécuter du code qui entraîne une erreur d’exécution.

L’un des éléments clés de la réduction des erreurs d’exécution consiste à écrire de petites procédures qui font une chose. Moins il y a de raisons pour lesquelles les procédures échouent, plus le code dans son ensemble est facile à déboguer.

Éviter l’erreur d’exécution 91 - Objet ou Avec variable de bloc non définie :

Cette erreur sera déclenchée lorsqu’un objet est utilisé avant que sa référence ne soit affectée. On peut avoir une procédure qui reçoit un paramètre d’objet :

Private Sub DoSomething(ByVal target As Worksheet)
    Debug.Print target.Name
End Sub

Si target ne se voit pas attribuer de référence, le code ci-dessus générera une erreur qui est facilement évitée en vérifiant si l’objet contient une référence d’objet réelle :

Private Sub DoSomething(ByVal target As Worksheet)
    If target Is Nothing Then Exit Sub
    Debug.Print target.Name
End Sub

Si target n’a pas de référence assignée, alors la référence non assignée n’est jamais utilisée et aucune erreur ne se produit.

Cette façon de quitter prématurément une procédure lorsqu’un ou plusieurs paramètres ne sont pas valides s’appelle une clause de garde.

Eviter l’erreur d’exécution 9 - Indice hors limites :

Cette erreur est déclenchée lorsqu’un tableau est accessible en dehors de ses limites.

Private Sub DoSomething(ByVal index As Integer)
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

Étant donné un index supérieur au nombre de feuilles de calcul dans ActiveWorkbook, le code ci-dessus générera une erreur d’exécution. Une simple clause de garde peut éviter que :

Private Sub DoSomething(ByVal index As Integer)
    If index > ActiveWorkbook.Worksheets.Count Or index <= 0 Then Exit Sub
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

La plupart des erreurs d’exécution peuvent être évitées en vérifiant soigneusement les valeurs que nous utilisons * avant * de les utiliser, et en se branchant sur un autre chemin d’exécution en conséquence en utilisant une simple instruction “If” - dans des clauses de garde qui ne font aucune hypothèse et valident les paramètres d’une procédure, ou même dans le corps de procédures plus larges.

Erreurs personnalisées #

Souvent, lors de l’écriture d’une classe spécialisée, vous souhaiterez qu’elle déclenche ses propres erreurs spécifiques, et vous souhaiterez une manière propre pour le code utilisateur/appelant de gérer ces erreurs personnalisées. Une manière astucieuse d’y parvenir consiste à définir un type “Enum” dédié :

Option Explicit
Public Enum FoobarError
    Err_FooWasNotBarred = vbObjectError + 1024
    Err_BarNotInitialized
    Err_SomethingElseHappened
End Enum

L’utilisation de la constante intégrée “vbObjectError” garantit que les codes d’erreur personnalisés ne se chevauchent pas avec les codes d’erreur réservés/existants. Seule la première valeur enum doit être explicitement spécifiée, car la valeur sous-jacente de chaque membre Enum est supérieure de 1 au membre précédent, de sorte que la valeur sous-jacente de Err_BarNotInitialized est implicitement vbObjectError + 1025.

Lever vos propres erreurs d’exécution #

Une erreur d’exécution peut être générée à l’aide de l’instruction Err.Raise, de sorte que l’erreur personnalisée Err_FooWasNotBarred peut être générée comme suit :

Err.Raise Err_FooWasNotBarred

La méthode Err.Raise peut également prendre des paramètres personnalisés Description et Source - pour cette raison, il est judicieux de définir également des constantes pour contenir la description de chaque erreur personnalisée :

Private Const Msg_FooWasNotBarred As String = "The foo was not barred."
Private Const Msg_BarNotInitialized As String = "The bar was not initialized."

Et créez ensuite une méthode privée dédiée pour générer chaque erreur :

Private Sub OnFooWasNotBarredError(ByVal source As String)
    Err.Raise Err_FooWasNotBarred, source, Msg_FooWasNotBarred
End Sub

Private Sub OnBarNotInitializedError(ByVal source As String)
    Err.Raise Err_BarNotInitialized, source, Msg_BarNotInitialized
End Sub

L’implémentation de la classe peut alors simplement appeler ces procédures spécialisées pour générer l’erreur :

Public Sub DoSomething()
    'raises the custom 'BarNotInitialized' error with "DoSomething" as the source:
    If Me.Bar Is Nothing Then OnBarNotInitializedError "DoSomething"
    '...
End Sub

Le code client peut alors gérer Err_BarNotInitialized comme il le ferait pour toute autre erreur, à l’intérieur de sa propre sous-routine de gestion des erreurs.

Remarque : l’ancien mot clé “Error” peut également être utilisé à la place de “Err.Raise”, mais il est obsolète/obsolète.

Reprendre le mot-clé #

Une sous-routine de gestion des erreurs va soit :

  • s’exécute jusqu’à la fin de la procédure, auquel cas l’exécution reprend dans la procédure appelante.
  • ou, utilisez le mot clé Resume pour reprendre l’exécution à l’intérieur de la même procédure.

Le mot clé Resume ne doit être utilisé qu’à l’intérieur d’une sous-routine de gestion des erreurs, car si VBA rencontre Resume sans être dans un état d’erreur, l’erreur d’exécution 20 “Reprendre sans erreur” est déclenchée.

Une sous-routine de gestion des erreurs peut utiliser le mot clé Resume de plusieurs manières :

  • Resume utilisé seul, l’exécution continue sur l’instruction qui a causé l’erreur. Si l’erreur n’est pas réellement traitée avant cela, la même erreur sera à nouveau déclenchée et l’exécution pourrait entrer dans une boucle infinie.
  • Resume Next continue l’exécution sur l’instruction suivant immédiatement l’instruction qui a causé l’erreur. Si l’erreur n’est pas * réellement * traitée avant cela, l’exécution est autorisée à continuer avec des données potentiellement invalides, ce qui peut entraîner des erreurs logiques et un comportement inattendu.
  • Resume [line label] continue l’exécution à l’étiquette de ligne spécifiée (ou au numéro de ligne, si vous utilisez des numéros de ligne de style hérité). Cela permettrait généralement d’exécuter du code de nettoyage avant de quitter proprement la procédure, par exemple en s’assurant qu’une connexion à la base de données est fermée avant de revenir à l’appelant.

En cas d’erreur Reprendre Suivant #

L’instruction On Error elle-même peut utiliser le mot-clé Resume pour demander au runtime VBA d’ignorer toutes les erreurs.

Si l’erreur n’est pas réellement traitée avant cela, l’exécution est autorisée à continuer avec des données potentiellement invalides, ce qui peut entraîner des erreurs logiques et un comportement inattendu.

L’accent ci-dessus ne peut pas être assez souligné. On Error Resume Next ignore efficacement toutes les erreurs et les met sous le tapis. Un programme qui explose avec une erreur d’exécution en raison d’une entrée invalide est un meilleur programme qu’un programme qui continue de fonctionner avec des données inconnues/non intentionnelles - ne serait-ce que parce que le bogue est beaucoup plus facilement identifiable. On Error Resume Next peut facilement masquer les bogues.

L’instruction On Error est portée par la procédure - c’est pourquoi il ne devrait normalement y avoir qu’une seule **, une seule instruction On Error dans une procédure donnée.

Cependant parfois une condition d’erreur ne peut pas tout à fait être évitée, et passer à une sous-routine de gestion des erreurs uniquement pour Resume Next ne semble tout simplement pas correct. Dans ce cas précis, l’instruction connue pour échouer peut être encapsulée entre deux instructions “On Error” :

On Error Resume Next
[possibly-failing statement]
Err.Clear 'resets current error
On Error GoTo 0

L’instruction On Error GoTo 0 réinitialise la gestion des erreurs dans la procédure en cours, de sorte que toute autre instruction provoquant une erreur d’exécution sera non gérée dans cette procédure et passera à la place dans la pile des appels jusqu’à ce qu’elle soit interceptée par un gestionnaire d’erreurs actif. S’il n’y a pas de gestionnaire d’erreurs actif dans la pile des appels, il sera traité comme une exception non gérée.

Public Sub Caller()
    On Error GoTo Handler
    
    Callee
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Caller."
End Sub

Public Sub Callee()
    On Error GoTo Handler
    
    Err.Raise 1     'This will be handled by the Callee handler.
    On Error GoTo 0 'After this statement, errors are passed up the stack.
    Err.Raise 2     'This will be handled by the Caller handler.    
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Callee."
    Resume Next
End Sub

Sur l’instruction d’erreur #

Même avec des clauses de garde, on ne peut pas raisonnablement toujours tenir compte de toutes les conditions d’erreur possibles qui pourraient être soulevées dans le corps d’une procédure. L’instruction On Error GoTo indique à VBA de passer à une * étiquette de ligne * et d’entrer en “mode de gestion des erreurs” chaque fois qu’une erreur inattendue se produit lors de l’exécution. Après avoir traité une erreur, le code peut reprendre en exécution “normale” en utilisant le mot clé Resume.

Les étiquettes de ligne désignent des sous-routines : étant donné que les sous-routines proviennent du code BASIC hérité et utilisent des sauts GoTo et GoSub et des instructions Return pour revenir à la routine “principale”, il est assez facile d’écrire difficile à- suivez le code spaghetti si les choses ne sont pas rigoureusement structurées. Pour cette raison, il est préférable que :

  • une procédure a une et une seule sous-routine de gestion des erreurs
  • la sous-routine de gestion des erreurs ** ne s’exécute jamais qu’en état d’erreur **

Cela signifie qu’une procédure qui gère ses erreurs doit être structurée comme ceci :

Private Sub DoSomething()
    On Error GoTo CleanFail

    'procedure code here

CleanExit:
    'cleanup code here
    Exit Sub

CleanFail:
    'error-handling code here
    Resume CleanExit
End Sub

Stratégies de gestion des erreurs #

Parfois, vous souhaitez gérer différentes erreurs avec différentes actions. Dans ce cas, vous inspecterez l’objet global “Err”, qui contiendra des informations sur l’erreur qui a été déclenchée - et agirez en conséquence :

CleanExit:
    Exit Sub

CleanFail:
    Select Case Err.Number
        Case 9
            MsgBox "Specified number doesn't exist. Please try again.", vbExclamation
            Resume
        Case 91
            'woah there, this shouldn't be happening.
            Stop 'execution will break here
            Resume 'hit F8 to jump to the line that raised the error
        Case Else
            MsgBox "An unexpected error has occurred:" & vbNewLine & Err.Description, vbCritical
            Resume CleanExit
    End Select
End Sub

En règle générale, envisagez d’activer la gestion des erreurs pour l’ensemble de la sous-routine ou de la fonction, et gérez toutes les erreurs susceptibles de se produire dans sa portée. Si vous devez gérer uniquement les erreurs dans la petite section du code, activez et désactivez la gestion des erreurs au même niveau :

Private Sub DoSomething(CheckValue as Long)

    If CheckValue = 0 Then
        On Error GoTo ErrorHandler   ' turn error handling on
        ' code that may result in error
        On Error GoTo 0              ' turn error handling off - same level
    End If

CleanExit:
    Exit Sub

ErrorHandler:
    ' error handling code here
    ' do not turn off error handling here
    Resume

End Sub

Numéros de ligne #

VBA prend en charge les numéros de ligne de style hérité (par exemple, QBASIC). La propriété masquée Erl peut être utilisée pour identifier le numéro de ligne qui a généré la dernière erreur. Si vous n’utilisez pas de numéros de ligne, Erl ne renverra jamais que 0.

Sub DoSomething()
10 On Error GoTo 50
20 Debug.Print 42 / 0
30 Exit Sub
40
50 Debug.Print "Error raised on line " & Erl ' returns 20
End Sub

Si vous utilisez des numéros de ligne, mais pas de manière cohérente, alors Erl renverra le dernier numéro de ligne avant l’instruction qui a déclenché l’erreur.

Sub DoSomething()
10 On Error GoTo 50
   Debug.Print 42 / 0
30 Exit Sub

50 Debug.Print "Error raised on line " & Erl 'returns 10
End Sub

Gardez à l’esprit que Erl n’a également qu’une précision Integer et débordera silencieusement. Cela signifie que les numéros de ligne en dehors de la plage d’entiers donneront des résultats incorrects :

Sub DoSomething()
99997 On Error GoTo 99999
99998 Debug.Print 42 / 0
99999
      Debug.Print Erl   'Prints 34462
End Sub

Le numéro de ligne n’est pas aussi pertinent que l’instruction à l’origine de l’erreur, et la numérotation des lignes devient rapidement fastidieuse et peu facile à entretenir.
