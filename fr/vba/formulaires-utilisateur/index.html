Les meilleures pratiques #

Un UserForm est un module de classe avec un concepteur et une instance par défaut. Le designer est accessible en appuyant sur Shift+F7 tout en visualisant le code-behind, et le code-behind est accessible en appuyant sur F7 lors de l’affichage du designer.

Travaillez avec une nouvelle instance à chaque fois. #

Étant un module de classe, un formulaire est donc un plan directeur pour un objet. Parce qu’un formulaire peut contenir un état et des données, il est préférable de travailler avec une nouvelle instance de la classe, plutôt qu’avec celle par défaut/globale :

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        '...
    End If
End With

À la place de:

UserForm1.Show vbModal
If Not UserForm1.IsCancelled Then
    '...
End If

Travailler avec l’instance par défaut peut entraîner des bugs subtils lorsque le formulaire est fermé avec le bouton rouge “X” et/ou lorsque Unload Me est utilisé dans le code-behind.

Implémentez la logique ailleurs. #

Un formulaire ne devrait être concerné que par la présentation : un gestionnaire de bouton “Clic” qui se connecte à une base de données et exécute une requête paramétrée basée sur l’entrée de l’utilisateur, fait trop de choses.

Au lieu de cela, implémentez la logique applicative dans le code responsable de l’affichage du formulaire, ou mieux encore, dans des modules et procédures dédiés.

Écrivez le code de manière à ce que l’UserForm soit uniquement responsable de savoir comment afficher et collecter les données : d’où viennent les données, ou ce qu’il advient des données par la suite, ne le concerne pas.

L’appelant ne devrait pas être dérangé par les contrôles. #

Créez un modèle bien défini avec lequel le formulaire fonctionnera, soit dans son propre module de classe dédié, soit encapsulé dans le code du formulaire lui-même - exposez le modèle avec les procédures Property Get et ayez le code client travailler avec ceux-ci : cela fait du formulaire une abstraction sur les contrôles et leurs petits détails, n’exposant que les données pertinentes au code client.

Cela signifie un code qui ressemble à ceci :

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        MsgBox .Message, vbInformation
    End If
End With

Au lieu de cela:

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        MsgBox .txtMessage.Text, vbInformation
    End If
End With

Gérer l’événement QueryClose. #

Les formulaires ont généralement un bouton Fermer et les invites/dialogues ont des boutons Ok et Annuler ; l’utilisateur peut fermer le formulaire en utilisant la boîte de contrôle du formulaire (le bouton “X” rouge), qui détruit l’instance de formulaire par défaut (une autre bonne raison de travailler avec une nouvelle instance à chaque fois).

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then 'if QueryClose isn't handled, this can raise a runtime error.
        '...
    End With
End With

La façon la plus simple de gérer l’événement QueryClose est de définir le paramètre Cancel sur True, puis de masquer le formulaire au lieu de le fermer :

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)
    Cancel = True
    Me.Hide
End Sub

De cette façon, le bouton “X” ne détruira jamais l’instance et l’appelant pourra accéder en toute sécurité à tous les membres publics.

Cachez, ne fermez pas. #

Le code qui crée un objet devrait être responsable de sa destruction : ce n’est pas la responsabilité du formulaire de se décharger et de se terminer.

Évitez d’utiliser Unload Me dans le code-behind d’un formulaire. Appelez Me.Hide à la place, afin que le code appelant puisse toujours utiliser l’objet qu’il a créé à la fermeture du formulaire.

Nommez les choses. #

Utilisez la fenêtre d’outils propriétés (F4) pour nommer soigneusement chaque contrôle d’un formulaire. Le nom d’un contrôle est utilisé dans le code-behind, donc à moins que vous n’utilisiez un outil de refactoring capable de gérer cela, ** renommer un contrôle cassera le code ** - il est donc beaucoup plus facile de faire les choses correctement dès le début lieu, que d’essayer de déterminer exactement laquelle des 20 zones de texte TextBox12 représente.

Traditionnellement, les contrôles UserForm sont nommés avec des préfixes de style hongrois :

  • lblUserName pour un contrôle Label qui indique un nom d’utilisateur.
  • txtUserName pour un contrôle TextBox où l’utilisateur peut entrer un nom d’utilisateur.
  • cboUserName pour un contrôle ComboBox où l’utilisateur peut entrer ou choisir un nom d’utilisateur.
  • lstUserName pour un contrôle ListBox où l’utilisateur peut choisir un nom d’utilisateur.
  • btnOk ou cmdOk pour un contrôle Button libellé “Ok”.

Le problème est que lorsque par ex. l’interface utilisateur est repensée et un ComboBox se transforme en ListBox, le nom doit changer pour refléter le nouveau type de contrôle : il est préférable de nommer les contrôles pour ce qu’ils représentent, plutôt qu’après leur type de contrôle - pour * découpler * le code de l’interface utilisateur autant que possible.

  • UserNameLabel pour une étiquette en lecture seule qui indique un nom d’utilisateur.
  • UserNameInput pour un contrôle où l’utilisateur peut entrer ou choisir un nom d’utilisateur.
  • OkButton pour un bouton de commande étiqueté “Ok”.

Quel que soit le style choisi, tout est mieux que de laisser tous les contrôles leurs noms par défaut. La cohérence dans le style de dénomination est également idéale.

Gestion de la requêteFermer #

L’événement QueryClose est déclenché chaque fois qu’un formulaire est sur le point d’être fermé, que ce soit via une action de l’utilisateur ou par programme. Le paramètre CloseMode contient une valeur d’énumération VbQueryClose qui indique comment le formulaire a été fermé :

Constante Description Valeur
vbFormControlMenu Le formulaire se ferme en réponse à l’action de l’utilisateur 0
vbFormCode Le formulaire se ferme en réponse à une instruction Unload 1
vbAppWindows La session Windows se termine 2
vbAppTaskManager Le Gestionnaire des tâches de Windows ferme l’application hôte 3
vbFormMDIForm Non pris en charge dans VBA 4

Pour une meilleure lisibilité, il est préférable d’utiliser ces constantes au lieu d’utiliser directement leur valeur.

Un formulaire utilisateur annulable #

Étant donné un formulaire avec un bouton Annuler

![un exemple de formulaire][1]

Le code-behind du formulaire pourrait ressembler à ceci :

Option Explicit
Private Type TView
    IsCancelled As Boolean
    SomeOtherSetting As Boolean
    'other properties skipped for brievety
End Type
Private this As TView

Public Property Get IsCancelled() As Boolean
    IsCancelled = this.IsCancelled
End Property

Public Property Get SomeOtherSetting() As Boolean
    SomeOtherSetting = this.SomeOtherSetting
End Property

'...more properties...

Private Sub SomeOtherSettingInput_Change()
    this.SomeOtherSetting = CBool(SomeOtherSettingInput.Value)
End Sub

Private Sub OkButton_Click()
    Me.Hide
End Sub

Private Sub CancelButton_Click()
    this.IsCancelled = True
    Me.Hide
End Sub

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)
    If CloseMode = VbQueryClose.vbFormControlMenu Then
        Cancel = True
        this.IsCancelled = True
        Me.Hide
    End If
End Sub

Le code appelant pourrait alors afficher le formulaire, et savoir s’il a été annulé :

Public Sub DoSomething()
    With New UserForm1
        .Show vbModal
        If .IsCancelled Then Exit Sub
        If .SomeOtherSetting Then
            'setting is enabled
        Else
            'setting is disabled
        End If
    End With
End Sub

La propriété IsCancelled renvoie True lorsque le bouton Annuler est cliqué, ou lorsque l’utilisateur ferme le formulaire à l’aide de la boîte de contrôle.
