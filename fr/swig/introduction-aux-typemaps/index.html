Introduction aux Typemaps

  • Syntaxe
  • Paramètres
  • Typemap de base - Python

Sur cette page

  • Syntaxe
  • Paramètres
  • Typemap de base - Python

Les typographies sont au cœur même de ce que fait SWIG. Lorsque vous souhaitez transmettre des données entre des langages, les comportements pour le faire dépendent du type que SWIG voit. La puissance des typemaps est que les morceaux de code sont appliqués plusieurs fois.

SWIG lui-même inclut de nombreux mappages de types utiles dans la bibliothèque de base avec laquelle il est fourni, par ex. pour les types primitifs, les conteneurs de bibliothèque standard C++, boost, etc., vous n’aurez même souvent pas besoin d’écrire de mappages de types pour exposer votre code, mais cette liste n’est en aucun cas complète.

Syntaxe #

  • %typemap(NAME) TYPENAME %{ CODE %}
  • %typemap(NAME,OPTION=VALUE) TYPENAME %{ CODE %}
  • %typemap(NAME) TYPENAME VARIABLENAME %{ CODE %}
  • %typemap(NAME) TYPENAME (LOCALVARTYPE LOCALVARNAME) %{ CODE %}

Paramètres #

Paramètre Détails
NOM Le nom du typemap définit son rôle dans la génération d’un module. in et out sont communs et utilisés pour l’entrée (vers les appels de fonction C++ ou C depuis Python/Java, etc.) et la sortie (c’est-à-dire les valeurs de retour de C ou C++ vers Pyton/Java)
TYPENAME Chaque typemap est appliqué à un ou plusieurs types correspondants. Ceux-ci doivent être listés ici. Les qualificateurs de type (par exemple, const) sont importants.
CODES Chaque typemap doit être converti entre le type C ou C++ et le type correspondant dans le langage encapsulé. Vous devrez écrire du code pour le faire dans vos typesmaps personnalisés, en utilisant généralement des variables spéciales qui sont remplacées. $input dans un typemap in représente la valeur de Python/Java, $result dans un typemap out est la chose renvoyée à Python/Java. $1 dans les deux mappages de types in/out représente la variable C ou C++ du type utilisé pour correspondre au mappage de types. Ainsi, pour les mappages de types in, vous assigneriez à $1 et pour out, vous lirez à partir de celui-ci. (Voir les mappages de types multi-arguments pour plus de détails sur la raison pour laquelle il s’agit d’un nombre)
NOMVARIABLE Les mappages de types sont mis en correspondance du plus spécifique au moins spécifique en général. Vous pouvez définir un typemap qui ne fera correspondre que les arguments de fonction avec des noms spécifiques en utilisant ce formulaire facultatif. (Voir correspondance de typemap pour plus de détails)
(LOCALVARTYPE LOCALVARNAME) Parfois, en particulier pour les typemaps in, il est utile de pouvoir déclarer des variables locales supplémentaires pour contenir des objets autour d’un appel. Cette syntaxe facultative nous permet de le faire. En utilisant cette syntaxe au lieu de l’écrire dans CODE, la portée est modifiée, mais plus important encore, la variable peut être automatiquement renommée par SWIG pour éviter les conflits si une fonction a deux arguments utilisant le même typemap.
OPTION=VALEUR Le comportement de certains typemaps peut être influencé en définissant des options supplémentaires à l’aide de cette syntaxe. Par exemple, un typemap in peut être conçu pour ne prendre aucune entrée du langage cible en définissant numinputs=0, auquel cas le typemap est censé remplir implicitement l’entrée. (Un cas courant pour cela pourrait être de définir quelque chose sur NULL ou de le remplir à partir d’une valeur globale)

Typemap de base - Python #

Étant donné le type booléen personnalisé suivant, nous voulons envelopper :

typedef char MYBOOL;
#define TRUE 1
#define FALSE 0

Une approche simple pourrait être d’écrire les mappages de types suivants dans notre interface SWIG :

%typemap(in) MYBOOL %{
  // $input is what we got passed from Python for this function argument
  $1 = PyObject_IsTrue($input);
  // $1 is what will be used for the C or C++ call and we are responsible for setting it
%}

%typemap(out) MYBOOL %{
  // $1 is what we got from our C or C++ call
  $result = PyBool_FromLong($1);
  // $result is what gets given back to Python and we are responsible for setting it
%}

Avec ces typesmaps, SWIG insèrera notre code dans le wrapper généré chaque fois qu’il verra un MYBOOL passé dans ou hors d’un appel de fonction.
