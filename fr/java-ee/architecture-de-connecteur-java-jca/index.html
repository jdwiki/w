Architecture de connecteur Java (JCA)

  • Exemple d’adaptateur de ressources

Sur cette page

  • Exemple d’adaptateur de ressources

Clarifions d’abord quelques terminologies :

  • Messagerie sortante est l’endroit où le message commence à partir du serveur (pour être plus précis, il est lancé à partir de votre application que vous avez sur le serveur, WebSphere Liberty dans ce cas) et se termine à l’EIS.
  • Messagerie entrante est l’endroit où le message commence à partir de l’EIS et se termine au niveau du serveur.
  • Message Endpoint en général l’endroit où le message finit par s’asseoir/être reçu à une étape spécifique de son cycle de vie.

[![entrez la description de l’image ici][1]][1]

Ainsi, avec la connectivité sortante, nous faisons référence à la situation dans laquelle une application obtient une connexion à un EIS externe et y lit ou y écrit des données. Avec la connectivité entrante, nous faisons référence à la situation dans laquelle l’adaptateur de ressources (RA) écoute les événements de l’EIS externe et appelle votre application lorsqu’un tel événement se produit.

Que signifie MessageEndPoint dans JCA ?

Le serveur d’applications (par exemple : WebSphere Liberty) fournit des MBeans de point de terminaison de message pour vous aider à gérer la livraison d’un message à vos beans pilotés par message qui agissent en tant qu’écouteurs sur des points de terminaison spécifiques, qui sont des destinations, et à gérer les ressources EIS qui sont utilisés par ces beans pilotés par message. Les beans orientés message déployés en tant que points de terminaison de message ne sont pas identiques aux beans orientés message configurés sur un port d’écoute. Les beans pilotés par message utilisés comme points de terminaison de message doivent être déployés à l’aide d’une ActivationSpecification définie dans une configuration RA pour JCA (trouvé dans le fichier ra.xml).

Que signifie l’activation d’un MessageEndPoint ?

Avec les MBeans de point de terminaison de message, vous pouvez activer et désactiver des points de terminaison spécifiques au sein de vos applications pour vous assurer que les messages ne sont remis qu’aux beans gérés par message à l’écoute qui interagissent avec des ressources EIS saines. Cette fonctionnalité vous permet d’optimiser les performances de vos applications JMS dans les situations où une ressource EIS ne se comporte pas comme prévu. La livraison de message à un point de terminaison échoue généralement lorsque le bean piloté par message qui écoute appelle une opération sur une ressource qui n’est pas saine. Par exemple, un fournisseur de messagerie, qui est un adaptateur de ressources entrantes conforme à JCA, peut ne pas parvenir à remettre des messages à un point de terminaison lorsque son bean géré par message sous-jacent tente de valider des transactions sur un serveur de base de données qui ne répond pas.

MessageEndPoint doit-il être un bean ?

Cela devrait. Sinon, vous vous retrouverez dans un gros gâchis en créant votre propre façon non conventionnelle de faire des choses qui vont à l’encontre de l’objectif de suivre les spécifications Java EE en premier lieu. Concevez vos beans orientés message pour déléguer le traitement métier à d’autres beans enterprise. N’accédez pas aux ressources EIS directement dans le bean piloté par message, mais faites-le indirectement via un bean délégué.
