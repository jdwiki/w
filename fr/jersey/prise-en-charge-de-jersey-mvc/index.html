Prise en charge de Jersey MVC

  • Maillot MVC Hello World

Sur cette page

  • Maillot MVC Hello World

Les frameworks MVC tels que Spring MVC sont utilisés pour créer des applications Web qui servent des pages Web dynamiques. Jersey, bien que connu pour être un framework REST, prend également en charge la création de pages Web dynamiques à l’aide de son module MVC.

Maillot MVC Hello World #

Pour commencer, créez une nouvelle application Web Maven (la procédure à suivre sort du cadre de cet exemple). Dans votre pom.xml, ajoutez les deux dépendances suivantes

<dependency>
    <groupId>org.glassfish.jersey.containers</groupId>
    <artifactId>jersey-container-servlet</artifactId>
    <version>2.25.1</version>
</dependency>
<dependency>
    <groupId>org.glassfish.jersey.ext</groupId>
    <artifactId>jersey-mvc-jsp</artifactId>
    <version>2.25.1</version>
</dependency>

Toujours dans le pom, ajoutez le jetty-maven-plugin que nous utiliserons pour exécuter l’application pendant le développement

<build>
    <finalName>jersey-mvc-hello-world</finalName>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>2.5.1</version>
            <inherited>true</inherited>
            <configuration>
                <source>1.8</source>
                <target>1.8</target>
            </configuration>
        </plugin>

        <plugin>
            <groupId>org.eclipse.jetty</groupId>
            <artifactId>jetty-maven-plugin</artifactId>
            <version>9.3.8.v20160314</version>
        </plugin>
    </plugins>
</build>

Nous pouvons maintenant créer nos contrôleurs. Dans n’importe quel framework MVC, les concepts sont généralement les mêmes. Vous avez un modèle et vous utilisez un contrôleur pour remplir un modèle qui sera utilisé pour rendre le modèle. Le terme “rendre” ici est utilisé pour signifier créer la page HTML finale en combinant le modèle et le modèle Prenons par exemple ce modèle

src/main/webapp/WEB-INF/jsp/index.jsp

<html>
    <head>
        <title>JSP Page</title>
    </head>
    <body>
        <h1>${it.hello} ${it.world}</h1>
    </body>
</html>

Il s’agit d’un fichier JSP. JSP n’est que l’un des moteurs de modèles pris en charge par Jersey. Ici, nous utilisons deux variables de modèle, “hello” et “world”. Il est prévu que ces deux variables soient dans le modèle utilisé pour rendre ce modèle. Ajoutons donc le contrôleur

package com.example.controller;

import org.glassfish.jersey.server.mvc.Viewable;

import javax.ws.rs.GET;
import javax.ws.rs.Path;
import javax.ws.rs.Produces;
import javax.ws.rs.core.MediaType;
import java.util.HashMap;
import java.util.Map;

@Path("/")
public class HomeController {

    @GET
    @Produces(MediaType.TEXT_HTML)
    public Viewable index() {
        Map<String, String> model = new HashMap<>();
        model.put("hello", "Hello");
        model.put("world", "World");
        return new Viewable("/index", model);
    }
}

Vous pouvez voir ici que nous remplissons le modèle avec les propriétés hello et world. De plus, la méthode du contrôleur renvoie le nom du modèle de vue à utiliser, dans ce cas index. Avec cela, le framework sait saisir le modèle “index” et utiliser le modèle fourni pour le rendre.

Il ne nous reste plus qu’à le configurer. Ajoutez une sous-classe ResourceConfig avec ce qui suit

package com.example;

import org.glassfish.jersey.server.ResourceConfig;
import org.glassfish.jersey.server.mvc.jsp.JspMvcFeature;

public class AppConfig extends ResourceConfig {

    public AppConfig() {
        packages("com.example.controller");
        property(JspMvcFeature.TEMPLATE_BASE_PATH, "/WEB-INF/jsp");
        register(JspMvcFeature.class);
    }
}

Il se passe trois choses ici :

  1. Nous utilisons packages pour dire à Jersey d’analyser le package com.example.controller pour les classes annotées avec @Path afin qu’il puisse l’enregistrer. Dans ce cas, il enregistre notre HomeController.

  2. Nous définissons le chemin de base du framework pour résoudre les modèles. Dans ce cas, nous disons à Jersey de rechercher des modèles dans WEB-INF/jsp. Vous pouvez voir l’exemple index.jsp ci-dessus dans ce directeur. Toujours dans le contrôleur, nous renvoyons uniquement le nom du modèle index. Cela sera utilisé pour trouver le modèle, en préfixant le chemin de base de configuration et en suffixant un .jsp implicite

  3. Nous devons enregistrer la fonctionnalité qui gère le rendu JSP. Comme mentionné précédemment, JSP n’est pas le seul moteur de rendu pris en charge par Jersey. Il y en a quelques autres pris en charge prêts à l’emploi.

La dernière chose que nous devons faire est de configurer Jersey dans le fichier web.xml

<filter>
    <filter-name>Jersey</filter-name>
    <filter-class>org.glassfish.jersey.servlet.ServletContainer</filter-class>
    <init-param>
        <param-name>javax.ws.rs.Application</param-name>
        <param-value>com.example.AppConfig</param-value>
    </init-param>
</filter>

<filter-mapping>
    <url-pattern>/*</url-pattern>
    <filter-name>Jersey</filter-name>
</filter-mapping>

Ici, nous configurons simplement Jersey pour utiliser notre classe AppConfig. Une chose très importante à souligner ici est l’utilisation du <filter> au lieu de ce que vous verriez normalement, un <servlet>. Ceci est requis lors de l’utilisation de JSP comme moteur de modèle.

Maintenant, nous pouvons l’exécuter. Depuis la ligne de commande, exécutez mvn jetty:run. Cela exécutera le plugin Maven Jetty que nous avons configuré précédemment. Lorsque vous voyez “Started Jetty Server”, le serveur est prêt. Accédez à l’URL du navigateur http://localhost:8080/. Voila, “Bonjour le monde”. Prendre plaisir.

Pour plus d’informations, consultez la [Documentation Jersey pour les modèles MVC][1]

[1] : https://jersey.github.io/documentation/latest/mvc.html
