Mode réel #

Lorsqu’Intel a conçu le x86 d’origine, le 8086 (et le dérivé du 8088), ils ont inclus la segmentation pour permettre au processeur 16 bits d’accéder à plus de 16 bits d’adresse. Ils l’ont fait en faisant en sorte que les adresses 16 bits soient relatives à un registre de segment 16 bits donné, dont ils ont défini quatre : segment de code (CS), segment de données (DS), segment supplémentaire (ES) et segment de pile (SS).

La plupart des instructions impliquaient quel registre de segment utiliser : les instructions étaient extraites du segment de code, PUSH et POP impliquaient le segment de pile, et de simples références de données impliquaient le segment de données - bien que cela puisse être remplacé pour accéder à la mémoire dans l’un des autres Segments.

L’implémentation était simple : pour chaque accès mémoire, le processeur prenait le registre de segment implicite (ou explicite), le décalait de quatre positions vers la gauche, puis ajoutait l’adresse indiquée :

        +-------------------+---------+
Segment | 16-bit value      | 0 0 0 0 |
        +-------------------+---------+
                   PLUS
        +---------+-------------------+
Address | 0 0 0 0 | 16-bit value      |
        +---------+-------------------+
                  EQUALS
        +-----------------------------+
Result  |  20-bit memory address      |
        +-----------------------------+

Cela a permis diverses techniques :

  • Permettre à Code, Data et Stack d’être mutuellement accessibles (CS, DS et SS avaient tous la même valeur);
  • Garder le code, les données et la pile complètement séparés les uns des autres (CS, DS et SS tous 4K (ou plus) séparés les uns des autres - rappelez-vous qu’il est multiplié par 16, donc c’est 64K).

Cela a également permis des chevauchements bizarres et toutes sortes de choses bizarres !

Lorsque le 80286 a été inventé, il prenait en charge ce mode hérité (maintenant appelé “mode réel”), mais a ajouté un nouveau mode appelé “mode protégé” (qv).

Les choses importantes à noter sont qu’en mode réel :

  • Toute adresse mémoire était accessible, simplement en mettant la valeur correcte dans un registre de segment et en accédant à l’adresse 16 bits ;
  • L’étendue de la “protection” était de permettre au programmeur de séparer différentes zones de mémoire à des fins différentes, et de rendre plus difficile l’écriture accidentelle dans les mauvaises données - tout en permettant de le faire.

Autrement dit… pas très protégé du tout !

Mode protégé #

Introduction #

Lorsque le 80286 a été inventé, il a pris en charge l’ancienne segmentation 8086 (maintenant appelée “mode réel”) et a ajouté un nouveau mode appelé “mode protégé”. Ce mode a été dans tous les processeurs x86 depuis, bien qu’il ait été amélioré avec diverses améliorations telles que l’adressage 32 et 64 bits.

Concevoir #

En mode protégé, le simple “Ajouter une adresse à la valeur du registre de segment décalé” a été complètement supprimé. Ils ont conservé les registres de segments, mais au lieu de les utiliser pour calculer une adresse, ils les ont utilisés pour indexer dans une table (en fait, une sur deux…) qui définissait le segment auquel accéder. Cette définition décrivait non seulement où se trouvait le segment en mémoire (en utilisant la base et la limite), mais aussi de quel type de segment il s’agissait (code, données, pile ou même système) et quels types de programmes pouvaient y accéder (noyau du système d’exploitation, programme normal , pilote de périphérique, etc.).

registre de segments #

Chaque registre de segment 16 bits a pris la forme suivante :

+------------+-----+------+
| Desc Index | G/L | Priv |
+------------+-----+------+
 Desc Index = 13-bit index into a Descriptor Table (described below)
 G/L        = 1-bit flag for which Descriptor Table to Index: Global or Local
 Priv       = 2-bit field defining the Privilege level for access

Mondial / Local #

Le bit Global/Local définissait si l’accès se faisait dans une table globale de descripteurs (appelée sans surprise la table de descripteurs globale, ou GDT), ou une table de descripteurs locaux (LDT). L’idée du LDT était que chaque programme pouvait avoir sa propre table de descripteurs - le système d’exploitation définirait un ensemble global de segments, et chaque programme aurait son propre ensemble de segments de code local, de données et de pile. Le système d’exploitation gérerait la mémoire entre les différentes tables de descripteurs.

Tableau des descripteurs #

Chaque tableau de descripteurs (global ou local) était un tableau de 64 000 descripteurs de 8 192 descripteurs : chacun était un enregistrement de 8 octets qui définissait plusieurs aspects du segment qu’il décrivait. Les champs de l’index des descripteurs des registres de segments permettaient 8 192 descripteurs : pas de coïncidence !

Descripteur #

Un descripteur contenait les informations suivantes - notez que le format du descripteur a changé au fur et à mesure que de nouveaux processeurs ont été publiés, mais le même type d’informations a été conservé dans chacun :

  • Base Cela a défini l’adresse de début du segment de mémoire.
  • Limite Cela a défini la taille du segment de mémoire - en quelque sorte. Ils devaient prendre une décision : une taille de 0x0000 signifierait-elle une taille de 0, donc non accessible ? Ou taille maximale ? Au lieu de cela, ils ont choisi une troisième option : le champ Limite était le dernier emplacement adressable dans le segment. Cela signifiait qu’un segment unique pouvait être défini ; ou une taille maximale pour la taille de l’adresse.
  • Taper Il y avait plusieurs types de segments : le code traditionnel, les données et la pile (voir ci-dessous), mais d’autres segments système ont également été définis :
  • Les segments de la table des descripteurs locaux définissaient le nombre de descripteurs locaux accessibles ;
  • Les segments d’état de tâche pourraient être utilisés pour la commutation de contexte gérée par le matériel ;
  • “Call Gates” contrôlées qui pourraient permettre aux programmes d’appeler le système d’exploitation - mais uniquement via des points d’entrée soigneusement gérés.
  • Les attributs Certains attributs du Segment ont également été maintenus, le cas échéant :
  • Lecture seule vs lecture-écriture ;
  • Si le segment était actuellement présent ou non - permettant une gestion de la mémoire à la demande ;
  • Quel niveau de code (OS vs Driver vs programme) pourrait accéder à ce Segment.

La vraie protection enfin ! #

Si le système d’exploitation conservait les tables de descripteurs dans des segments inaccessibles aux simples programmes, il pourrait alors gérer étroitement les segments définis et la mémoire attribuée et accessible à chacun. Un programme pourrait fabriquer n’importe quelle valeur de registre de segment qu’il aimait - mais s’il avait l’audace de le charger dans un registre de segment !… le matériel CPU reconnaîtrait que la valeur de descripteur proposée enfreint l’une des nombreuses règles, et au lieu de terminer la demande, il lèverait une exception au système d’exploitation pour lui permettre de gérer le programme errant.

Cette exception était généralement la 13e, l’exception de protection générale - rendue mondialement célèbre par Microsoft Windows… (Quelqu’un pense-t-il qu’un ingénieur d’Intel est superstitieux ?)

Les erreurs #

Les types d’erreurs pouvant survenir incluent :

  • Si l’index des descripteurs proposé était supérieur à la taille du tableau ;
  • Si le descripteur proposé était un descripteur de système plutôt qu’un code, des données ou une pile ;
  • Si le Descripteur proposé était plus privilégié que le programme demandeur ;
  • Si le descripteur proposé a été marqué comme non lisible (comme un segment de code), mais qu’il a été tenté de le lire plutôt que de l’exécuter ;
  • Si le descripteur proposé a été marqué Non présent.

Notez que le dernier peut ne pas être un problème fatal pour le programme : le système d’exploitation pourrait noter le drapeau, rétablir le segment, le marquer comme maintenant présent, puis permettre à l’instruction défaillante de se poursuivre avec succès.

Ou peut-être que le descripteur a été chargé avec succès dans un registre de segment, mais qu’un accès futur avec celui-ci a enfreint l’une des nombreuses règles :

  • Le registre de segment a été chargé avec l’index de descripteur 0x0000 pour le GDT. Ceci était réservé par le matériel en tant que NULL ;
  • Si le descripteur chargé a été marqué en lecture seule, mais qu’une tentative d’écriture a été tentée dessus.
  • Si une partie de l’accès (1, 2, 4 octets ou plus) était en dehors de la limite du segment.

Passage en mode protégé #

Le passage en mode protégé est simple : il vous suffit de définir un seul bit dans un registre de contrôle. Mais rester en mode protégé, sans que le processeur ne baisse les bras et ne se réinitialise parce qu’il ne sait pas quoi faire ensuite, demande beaucoup de préparation.

En bref, les étapes requises sont les suivantes :

  • Une zone de mémoire pour la table des descripteurs globaux doit être configurée pour définir un minimum de trois descripteurs :
  1. Le zéro, descripteur NULL ;
  2. Un autre descripteur pour un segment de code ;
  3. Un autre descripteur pour un segment de données.

    This can be used for both Data and Stack.

  • Le registre de la table des descripteurs globaux (GDTR) doit être initialisé pour pointer vers cette zone de mémoire définie ;

     GDT_Ptr    dw      SIZE GDT
            dd      OFFSET GDT

            ...

            lgdt    [GDT_Ptr]

  • Le bit ‘PM’ dans ‘CR0’ doit être défini :

         mov   eax, cr0      ; Get CR0 into register
     or    eax, 0x01     ; Set the Protected Mode bit
     mov   cr0, eax      ; We're now in Protected Mode!

  • Les registres de segment doivent être chargés à partir du GDT pour supprimer les valeurs actuelles du mode réel :

         jmp   0x0008:NowInPM  ; This is a FAR Jump. 0x0008 is the Code Descriptor

NowInPM:
     mov   ax, 0x0010      ; This is the Data Descriptor
     mov   ds, ax
     mov   es, ax
     mov   ss, ax
     mov   sp, 0x0000      ; Top of stack!

Notez qu’il s’agit du bare minimum, juste pour mettre le CPU en mode protégé. La préparation de l’ensemble du système peut nécessiter de nombreuses étapes supplémentaires. Par exemple:

  • Les zones de mémoire supérieures peuvent devoir être activées - en désactivant la porte “A20” ;
  • Les interruptions doivent définitivement être désactivées - mais peut-être que les différents gestionnaires de défauts pourraient être configurés avant d’entrer en mode protégé, pour permettre les erreurs au début du traitement.

L’auteur original de cette section a écrit un [tutoriel][1] complet sur l’entrée en mode protégé et son utilisation.

[1] : http://wiki.osdev.org/JohnBurger:Démo

Mode irréel #

Le mode irréel exploite deux faits sur la façon dont les processeurs Intel et AMD chargent et enregistrent les informations pour décrire un segment.

  1. Le processeur met en cache les informations de descripteur récupérées lors d’un déplacement dans un registre de sélection en mode protégé. Ces informations sont stockées dans une partie architecturale invisible du registre du sélecteur lui-même.

  2. En mode réel, les registres du sélecteur sont appelés registres de segment mais, à part cela, ils désignent le même ensemble de registres et, à ce titre, ils ont également une partie invisible. Ces parties sont remplies de valeurs fixes, mais pour la base qui est dérivée de la valeur qui vient d’être chargée.

Dans une telle vue, le mode réel n’est qu’un cas particulier du mode protégé : où les informations d’un segment, telles que la base et la limite, sont récupérées sans GDT/LDT mais toujours lues à partir de la partie cachée du registre de segment.

En passant en mode protégé et en fabriquant un GDT, il est possible de créer un segment avec les attributs souhaités, par exemple une base de 0 et une limite de 4GiB. Grâce à un chargement successif d’un registre de sélection, ces attributs sont mis en cache, il est alors possible de revenir en mode réel et d’avoir un registre de segment par lequel accéder à l’ensemble de l’espace d’adressage de 32 bits.

BITS 16

jmp 7c0h:__START__

__START__:
 push cs
 pop ds
 push ds
 pop ss
 xor sp, sp

 
 lgdt [GDT]            ;Set the GDTR register
 

 cli                ;We don't have an IDT set, we can't handle interrupts


 ;Entering protected mode

 mov eax, cr0
 or ax, 01h            ;Set bit PE (bit 0) of CR0
 mov cr0, eax            ;Apply

 ;We are now in Protected mode

 mov bx, 08h           ;Selector to use, RPL = 0, Table = 0 (GDT), Index = 1

 mov fs, bx            ;Load FS with descriptor 1 info
 mov gs, bx            ;Load GS with descriptor 1 info

 ;Exit protected mode

 and ax, 0fffeh            ;Clear bit PE (bit0) of CR0
 mov cr0, eax                   ;Apply

 sti                

 ;Back to real mode

 ;Do nothing
 cli
 hlt 



 GDT:
    ;First entry, number 0
    ;Null descriptor
    ;Used to store a m16&32 object that tells the GDT start and size

    dw 0fh                 ;Size in byte -1 of the GDT (2 descriptors = 16 bytes)
    dd GDT + 7c00h         ;Linear address of GDT start (24 bits)
    dw 00h                 ;Pad 

    dd 0000ffffh           ;Base[15:00] = 0, Limit[15:00] = 0ffffh
    dd 00cf9200h           ;Base[31:24] = 0, G = 1, B = 1, Limit[19:16] = 0fh, 
               ;P = 1, DPL = 0, E = 0, W = 1, A = 0, Base[23:16] = 00h


 TIMES 510-($-$$) db 00h
 dw 0aa55h

Considérations

  • Dès qu’un registre de segment est rechargé, même avec la même valeur, le processeur recharge les attributs cachés selon le mode en cours. C’est pourquoi le code ci-dessus utilise fs et gs pour contenir les segments “étendus”: ces registres sont moins susceptibles d’être utilisés/sauvegardés/restaurés par les différents services 16 bits.
  • L’instruction lgdt ne charge pas un pointeur lointain vers le GDT, à la place, elle charge une adresse linéaire de 24 bits (peut être remplacée par 32 bits). Ce n’est pas une adresse locale, c’est l’adresse physique (puisque la pagination doit être désactivée). C’est la raison de GDT+7c00h.
  • Le programme ci-dessus est un chargeur de démarrage (pour MBR, il n’a pas de BPB) qui définit cs/ds/ss tp 7c00h et démarre le compteur d’emplacement à partir de 0. Donc, un octet à l’offset X dans le fichier est à l’offset X dans le segment 7c00h et à l’adresse linéaire 7c00h + X.
  • Les interruptions doivent être désactivées car un IDT n’est pas défini pour le court aller-retour en mode protégé.
  • Le code utilise un hack pour enregistrer 6 octets de code. La structure chargée par lgdt est enregistrée dans le… GDT lui-même, dans le descripteur nul (le premier descripteur).

Pour une description des descripteurs GDT, consultez le chapitre 3.4.3 du [Intel Manual Volume 3A](https://www-ssl.intel.com/content/www/us/en/architecture-and-technology/64-ia- 32-architectures-software-developer-vol-3a-part-1-manual.html).
