Assembleur Netwide - NASM #

NASM est de loin l’assembleur le plus porté pour l’architecture x86 - il est disponible pour pratiquement tous les systèmes d’exploitation basés sur le x86 (même inclus avec MacOS), et est disponible en tant qu’assembleur multiplateforme sur d’autres plateformes.

Cet assembleur utilise la syntaxe Intel, mais il est différent des autres car il se concentre fortement sur son propre langage “macro” - cela permet au programmeur de créer des expressions plus complexes en utilisant des définitions plus simples, permettant de créer de nouvelles “instructions”.

Malheureusement, cette fonctionnalité puissante a un coût : le type de données gêne les instructions générales, de sorte que le typage des données n’est pas appliqué.

response:    db       'Y'     ; Character that user typed

             cmp      response, 'N' ; *** Error! Unknown size!
             cmp byte response, 'N' ; That's better!
             cmp      response, ax  ; No error!

Cependant, NASM a introduit une fonctionnalité qui manquait aux autres : les noms de symboles délimités. Lorsque vous définissez un symbole dans d’autres assembleurs, ce nom est disponible dans le reste du code - mais cela “épuise” ce nom, “polluant” l’espace de noms global avec des symboles.

Par exemple (en utilisant la syntaxe NASM) :

       STRUC     Point
X      resw      1
Y      resw      1
       ENDSTRUC

Après cette définition, X et Y sont à jamais définis. Pour éviter “d’utiliser” les noms X et Y, vous deviez utiliser des noms plus précis :

       STRUC     Point
Pt_X   resw      1
Pt_Y   resw      1
       ENDSTRUC

Mais NASM offre une alternative. En tirant parti de son concept de “variable locale”, vous pouvez définir des champs de structure qui vous obligent à nommer la structure contenante dans les futures références :

       STRUC      Point
.X     resw       1
.Y     resw       1
       ENDSTRUC

Cursor ISTRUC     Point
       ENDISTRUC

       mov        ax,[Cursor+Point.X]
       mov        dx,[Cursor+Point.Y]

Malheureusement, comme NASM ne garde pas trace des types, vous ne pouvez pas utiliser la syntaxe la plus naturelle :

       mov        ax,[Cursor.X]
       mov        dx,[Cursor.Y]

Assembleur Microsoft - MASM #

Étant donné que le 8086/8088 était utilisé dans l’IBM PC, et que le système d’exploitation sur celui-ci provenait le plus souvent de Microsoft, l’assembleur MASM de Microsoft était la norme de facto pendant de nombreuses années. Il suivait de près la syntaxe d’Intel, mais permettait une syntaxe pratique mais “lâche” qui (avec le recul) ne provoquait que confusion et erreurs dans le code.

Un exemple parfait est le suivant :

MaxSize      EQU     16          ; Define a constant
Symbol       DW      0x1234      ; Define a 16-bit WORD called Symbol to hold 0x1234

             MOV     AX, 10      ; AX now holds 10
             MOV     BX, MaxSize ; BX now holds 16
             MOV     CX, Symbol  ; ????

La dernière instruction MOV place-t-elle le contenu de Symbol dans CX, ou l’adresse de Symbol dans CX ? Est-ce que CX se termine par 0x1234 ou 0x0102 (ou autre) ? Il s’avère que CX se termine par 0x1234 - si vous voulez l’adresse, vous devez utiliser le spécificateur OFFSET

             MOV     AX, [Symbol]      ; Contents of Symbol
             MOV     CX, OFFSET Symbol ; Address of Symbol

Assembleur AT&T - comme #

Bien que le 8086 ait été le plus utilisé dans les PC IBM avec Microsoft, il y avait un certain nombre d’autres ordinateurs et systèmes d’exploitation qui l’utilisaient également : notamment Unix. C’était un produit d’AT&T, et Unix fonctionnait déjà sur un certain nombre d’autres architectures. Ces architectures utilisaient une syntaxe d’assemblage plus conventionnelle - en particulier que les instructions à deux opérandes les spécifiaient dans l’ordre source, dest.

Ainsi, les conventions des assembleurs AT&T ont remplacé les conventions dictées par Intel, et un tout nouveau dialecte a été introduit pour la gamme x86 :

  • Les noms de registre étaient préfixés par % : %al, %bx etc.
  • Les valeurs immédiates ont été précédées de $ : $4
  • Les opérandes étaient dans l’ordre source, dest
  • Les opcodes incluaient leurs tailles d’opérandes : movw $4, %ax ; Déplacer le mot 4 dans AX

Assembleur Intel #

Intel a écrit la spécification du langage d’assemblage 8086, un dérivé des processeurs 8080, 8008 et 4004 antérieurs. En tant que tel, l’assembleur qu’ils ont écrit suivait précisément leur propre syntaxe. Cependant, cet assembleur n’a pas été utilisé très largement.

Intel a défini ses opcodes pour qu’ils aient zéro, un ou deux opérandes. Les instructions à deux opérandes ont été définies pour être dans l’ordre dest, source, ce qui était différent des autres assembleurs à l’époque. Mais certaines instructions utilisaient des registres implicites comme opérandes - il suffisait de savoir ce qu’ils étaient. Intel a également utilisé le concept d’opcodes “préfixés” - un opcode affecterait l’instruction suivante.

; Zero operand examples
NOP             ; No parameters
CBW             ; Convert byte in AL into word in AX
MOVSB           ; Move byte pointed to by DS:SI to byte pointed to by ES:DI
                ; SI and DI are incremented or decremented according to D bit

; Prefix examples
REP   MOVSB     ; Move number of bytes in CX from DS:SI to ES:DI
                ; SI and DI are incremented or decremented according to D bit

; One operand examples
NOT      AX     ; Replace AX with its one's complement
MUL      CX     ; Multiply AX by CX and put 32-bit result in DX:AX

; Two operand examples
MOV      AL, [0x1234] ; Copy the contents of memory location DS:0x1234 into AL register

Intel a également enfreint une convention utilisée par d’autres assembleurs : pour chaque opcode, un mnémonique différent a été inventé. Cela nécessitait des noms subtilement ou distinctement différents pour des opérations similaires : par ex. LDM pour “Charger depuis la mémoire” et LDI pour “Charger immédiatement”. Intel a utilisé le seul mnémonique MOV - et s’attendait à ce que l’assembleur détermine quel opcode utiliser à partir du contexte. Cela a causé de nombreux pièges et erreurs pour les programmeurs à l’avenir lorsque l’assembleur ne pouvait pas deviner ce que le programmeur voulait réellement…

Assembleur turbo de Borland - TASM #

Borland a commencé avec un compilateur Pascal qu’ils ont appelé “Turbo Pascal”. Viennent ensuite les compilateurs pour d’autres langages : C/C++, Prolog et Fortran. Ils ont également produit un assembleur appelé “Turbo Assembler”, qui, suivant la convention de dénomination de Microsoft, ils ont appelé “TASM”.

TASM a tenté de résoudre certains des problèmes d’écriture de code à l’aide de MASM (voir ci-dessus), en fournissant une interprétation plus stricte du code source sous un mode “IDEAL” spécifié. Par défaut, il supposait le mode MASM, de sorte qu’il pouvait assembler directement la source MASM - mais ensuite Borland a découvert qu’ils devaient être compatibles bogue pour bogue avec les particularités plus “excentriques” de MASM - ils ont donc également ajouté un mode QUIRKS.

Étant donné que TASM était (beaucoup) moins cher que MASM, il avait une large base d’utilisateurs - mais peu de gens utilisaient le mode IDEAL, malgré ses avantages vantés.

Assembleur GNU - gaz #

Lorsque le projet GNU a eu besoin d’un assembleur pour la famille x86, il a opté pour la version AT&T (et sa syntaxe) qui était associée à Unix plutôt que la version Intel/Microsoft.

Encore un autre assembleur - YASM #

[YASM][1] est une réécriture complète de NASM, mais est compatible avec les syntaxes Intel et AT&T.
