Utilisation de DataTemplate dans une ListBox #

Supposons que nous ayons l’extrait de code XAML suivant :

<ListBox x:Name="MyListBox" />

Ensuite, dans le code-behind de ce fichier XAML, nous écrivons ce qui suit dans le constructeur :

MyListBox.ItemsSource = new[]
{
    1, 2, 3, 4, 5
};

En exécutant l’application, nous obtenons une liste des numéros que nous avons entrés.

[![entrez la description de l’image ici][1]][1]

Cependant, si nous essayons d’afficher une liste d’objets d’un type personnalisé, comme celui-ci

MyListBox.ItemsSource = new[]
{
    new Book { Title = "The Hitchhiker's Guide to the Galaxy", Author = "Douglas Adams" },
    new Book { Title = "The Restaurant at the End of the Universe", Author = "Douglas Adams" },
    new Book { Title = "Life, the Universe and Everything", Author = "Douglas Adams" },
    new Book { Title = "So Long, and Thanks for All the Fish", Author = "Douglas Adams" },
    new Book { Title = "Mostly Harmless", Author = "Douglas Adams" }
};

en supposant que nous avons une classe appelée Book

public class Book
{
    public string Title { get; set; }
    public string Author { get; set; }
}

alors la liste ressemblerait à ceci:

[![entrez la description de l’image ici][2]][2]

Bien que nous puissions supposer que la ListBox sera “suffisamment intelligente” pour afficher correctement nos objets de livre, ce que nous voyons en réalité est le nom complet du type Book. Ce que ListBox fait réellement ici, c’est appeler la méthode intégrée ToString() sur les objets qu’il veut afficher, et bien que cela produise le résultat souhaitable dans le cas des nombres, appeler ```ToString()`` ` sur les objets des classes personnalisées entraîne l’obtention du nom de leur type, comme indiqué sur la capture d’écran.

Nous pourrions résoudre ce problème en écrivant ToString() pour notre classe de livre, c’est-à-dire

public override string ToString()
{
    return Title;
}

[![entrez la description de l’image ici][3]][3]

Cependant, ce n’est pas très flexible. Que se passe-t-il si nous voulons également afficher l’auteur ? Nous pourrions également écrire cela dans ToString, mais que se passe-t-il si nous ne voulons pas cela dans toutes les listes de l’application ? Que diriez-vous d’une belle couverture de livre à afficher?

C’est là que DataTemplates peut vous aider. Ce sont des extraits de code XAML qui peuvent être “instanciés” selon les besoins, remplis de détails en fonction des données sources pour lesquelles ils sont créés. En termes simples, si nous étendons notre code ListBox comme suit :

<ListBox x:Name="MyListBox">
    <ListBox.ItemTemplate>
        <DataTemplate>
            <TextBlock Text="{Binding Title}" />
        </DataTemplate>
    </ListBox.ItemTemplate>
</ListBox>

alors la liste créera un TextBox pour chaque élément dans sa source, et ces TextBoxes auront leurs propriétés Text “remplies” avec les valeurs de la propriété Title de nos objets.

Si nous exécutons l’application maintenant, nous obtenons la même sortie que ci-dessus, *même si nous supprimons l’implémentation personnalisée de ToString. Ce qui est avantageux, c’est que nous pouvons ensuite personnaliser ce modèle bien au-delà des capacités d’une simple string (et ToString). Nous pouvons utiliser n’importe quel élément XAML que nous voulons, y compris les éléments personnalisés, et pouvons “lier” leurs valeurs aux données réelles de nos objets (comme Title dans l’exemple ci-dessus). Par exemple, étendez le modèle comme suit :

<ListBox x:Name="MyListBox">
    <ListBox.ItemTemplate>
        <DataTemplate>
            <StackPanel>
                <TextBlock FontStyle="Italic" Text="{Binding Author}" />
                <TextBlock FontSize="18" Text="{Binding Title}" />
            </StackPanel>
        </DataTemplate>
    </ListBox.ItemTemplate>
</ListBox>

Ensuite, nous obtenons une belle vue formatée de nos livres!
