Une route est comme mapper une URL à une action (fonction) dans une classe Controller. La rubrique suivante se concentrera sur la création de routes, en passant des paramètres à la classe Controller via une route utilisant YAML ou une annotation.

Il est utile de voir ce qui est généré par le framework Symfony, celui-ci fournit des outils pour surveiller toutes les routes d’une application spécifique.

Depuis le [Symfony Doc][1], utilisez (dans un shell) :

php bin/console debug:router

De plus, vous pouvez voir toutes les informations de routes pertinentes dans le Framework profiler, dans le menu routage : [![Symfony Profiler][2]][2]

[1] : http://symfony.com/doc/master/routing/debug.html [2] : http://i.stack.imgur.com/E763X.jpg

Routage avec YAML #

La configuration du routage est incluse dans votre fichier app/config/config.yml, par défaut le fichier app/config/routing.yml.

De là, vous pouvez créer un lien vers votre propre configuration de routage dans un bundle

# app/config/routing.yml

app:
    resource: "@AppBundle/Resources/config/routing.yml"

Il peut également contenir plusieurs routes globales d’application.

Dans votre propre bundle, vous pouvez configurer des itinéraires qui servent à deux fins :

  • Correspondance avec une demande, de sorte que l’action correcte est appelée pour la demande.
  • Génération d’une URL à partir des paramètres de nom et de route.

Vous trouverez ci-dessous un exemple de configuration de route YAML :

# src/AppBundle/Resources/config/routing.yml

my_page:
    path: /application/content/page/{parameter}
    defaults:
        _controller: AppBundle:Default:myPage
        parameter: 42
    requirements:
        parameter: '\d+'
    methods: [ GET, PUT ]
    condition: "request.headers.get('User-Agent') matches '/firefox/i'"

La route est nommée my_page et appelle myPageAction du DefaultController dans AppBundle lorsque demandé. Il a un paramètre, nommé “paramètre” avec une valeur par défaut. La valeur n’est valide que lorsqu’elle correspond à la regex \d+. Pour cette route, seules les méthodes HTTP GET et PUT sont acceptées. La condition est une expression dans l’exemple que la route ne correspondra pas à moins que l’en-tête User-Agent ne corresponde à firefox. Vous pouvez faire n’importe quelle logique complexe dont vous avez besoin dans l’expression en tirant parti de deux variables qui sont passées dans l’expression : context (RequestContext) et request (Symfony Request).

Une route générée avec la valeur de paramètre 10 peut ressembler à /application/content/page/10.

Routage à l’aide d’annotations #

La configuration du routage est incluse dans votre fichier app/config/config.yml, par défaut le fichier app/config/routing.yml.

À partir de là, vous pouvez créer un lien vers les contrôleurs qui ont une configuration de routage annotée :

# app/config/routing.yml

app:
    resource: "@AppBundle/Controller"
    type:     annotation

Dans votre propre bundle, vous pouvez configurer des itinéraires qui servent à deux fins :

  • Correspondance avec une demande, de sorte que l’action correcte est appelée pour la demande.
  • Génération d’une URL à partir des paramètres de nom et de route.

Vous trouverez ci-dessous un exemple de configuration de route annotée :

// src/AppBundle/Controller/DefaultController.php

use Sensio\Bundle\FrameworkExtraBundle\Configuration\Route;
use Sensio\Bundle\FrameworkExtraBundle\Configuration\Method;

/**
 * @Route("/application")
 */
class DefaultController extends Controller {
    /**
     * @Route("/content/page/{parameter}",
     *        name="my_page",
     *        requirements={"parameter" = "\d+"},
     *        defaults={"parameter" = 42})
     * @Method({"GET", "PUT"})
     */
    public function myPageAction($parameter)
    {
        // ...
    }
}

Le contrôleur est annoté avec une route préfixe, de sorte que toute route configurée dans ce contrôleur sera précédée du préfixe.

La route configurée est nommée my_page et appelle la fonction myPageAction lorsqu’elle est demandée. Il a un paramètre, nommé “paramètre” avec une valeur par défaut. La valeur n’est valide que lorsqu’elle correspond à la regex \d+. Pour cette route, seules les méthodes HTTP GET et PUT sont acceptées.

Notez que le paramètre est injecté dans l’action en tant que paramètre de fonction.

Une route générée avec la valeur de paramètre 10 peut ressembler à /application/content/page/10.

Routes de repos puissantes #

Traditionnellement, vous pouvez utiliser le routage pour mapper une demande avec le [composant de routage][1] qui a géré les paramètres de demande et de réponse par [composant HttpFoundation][2].

De plus, un paramètre de route personnalisé peut être créé en utilisant le FOSRestBundle pour étendre les fonctionnalités par défaut du composant de routage.

Ceci est utile pour créer des routes REST, très utile pour spécifier comment transférer des données structurées comme un fichier XML ou json dans une requête (et une réponse).

Voir [FOSRestBundle Doc][3] pour plus d’informations, et notamment cette annotation :

use FOS\RestBundle\Controller\Annotations\FileParam;

/**
 * @FileParam(
 *   name="",
 *   key=null,
 *   requirements={},
 *   default=null,
 *   description="",
 *   strict=true,
 *   nullable=false,
 *   image=false
 * )
 */

[1] : http://symfony.com/doc/master/components/routing.html [2] : http://symfony.com/doc/master/components/http_foundation.html [3] : http://symfony.com/doc/master/bundles/FOSRestBundle/annotations-reference.html
