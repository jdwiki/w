Clé de partition, clé de clustering, clé primaire #

Cassandra utilise deux types de clés :

  • les Partition Keys sont responsables de la distribution des données entre les nœuds
  • la Clustering Key est responsable du tri des données au sein d’une partition

Une clé primaire est une combinaison de ces types. Le vocabulaire dépend de la combinaison :

  • clé primaire simple : uniquement la clé de partition, composée d’une colonne
  • clé de partition composite : uniquement la clé de partition, composée de plusieurs colonnes
  • clé primaire composée : une clé de partition avec une ou plusieurs clés de clustering.
  • composite et clé primaire composée : une clé de partition composée de plusieurs colonnes et de plusieurs clés de clustering.

La syntaxe PRIMARY KEY #

Déclarer une clé #

L’instruction de création de table doit contenir une expression PRIMARY KEY. La façon dont vous le déclarez est très importante. En un mot:

PRIMARY KEY(partition key)
PRIMARY KEY(partition key, clustering key)

Des parenthèses supplémentaires regroupent plusieurs champs dans une clé de partition composite ou déclarent une clé composite composée.

Exemples #

Clé primaire simple :

PRIMARY KEY (key)

clé s’appelle la clé de partition.

(pour une clé primaire simple, il est également possible de mettre l’expression PRIMARY KEY après le champ, c’est-à-dire key int PRIMARY KEY, par exemple).

Clé primaire Composé :

PRIMARY KEY (key_part_1, key_part_2)

Contrairement à SQL, cela ne crée pas exactement une clé primaire composite. Au lieu de cela, il déclare key_part_1 comme partition key et key_part_2 comme clustering key. Tout autre champ sera également considéré comme faisant partie de la clé de regroupement.

Clés primaires Composite+Compound :

PRIMARY KEY ((part_key_1, ..., part_key_n), (clust_key_1, ..., clust_key_n))

La première parenthèse définit la clé de partition composée, les autres colonnes sont les clés de clustering.

Résumé de la syntaxe #

  • (part_key)
  • (part_key, clust_key)
  • (part_key, clust_key_1, clust_key_2)
  • (part_key, (clust_key_1, clust_key_2))
  • ((part_key_1, part_key_2), clust_key)
  • ((part_key_1, part_key_2), (clust_key_1, cluster_key_2))

Ordre des clés et requêtes autorisées #

La clé de partition est le spécificateur minimal nécessaire pour effectuer une requête à l’aide d’une clause where.

Si vous déclarez une clé de clustering composite, l’ordre est important.

Supposons que vous ayez la clé primaire suivante :

PRIMARY KEY((part_key1, part_key_2), (clust_key_1, clust_key_2, clust_key_3))

Ensuite, les requêtes valides uniquement utilisent les champs suivants dans la clause where :

  • part_key_1, part_key_2
  • part_key_1, part_key_2, clust_key_1
  • part_key_1, part_key_2, clust_key_1, clust_key_2
  • part_key_1, part_key_2, clust_key_1, clust_key_2, clust_key_3

Voici des exemples de requêtes non valides :

  • part_key_1, part_key_2, clust_key_2
  • Tout ce qui ne contient pas à la fois part_key_1, part_key_2
  • …

Si vous souhaitez utiliser clust_key_2, vous devez également spécifier clust_key_1, et ainsi de suite.

Ainsi, l’ordre dans lequel vous déclarez vos clés de clustering aura un impact sur le type de requêtes que vous pouvez effectuer. A l’inverse, l’ordre des champs de la clé de partition n’a pas d’importance, puisqu’il faut toujours tous les spécifier dans une requête.
