Paramètres #

Nom du paramètre Descriptif
NomGrouperessource Le groupe de ressources Azure où se trouve le compte de stockage
NomConnexion La connexion Azure Run As (service principal) qui a été créée lors de la création du compte d’automatisation
StorageAccountName Le nom du compte Stockage Azure
NomConteneur Le nom du conteneur blob
Joursvieux Le nombre de jours qu’un blob peut attendre avant d’être supprimé

Assurez-vous d’avoir accès à Azure Active Directory afin que, lors de la création du compte Automation, Azure crée un compte RunAs pour vous. Cela vous évitera bien des soucis.

Supprimer les blobs dans le stockage Blob de plus d’un certain nombre de jours #

Voici un exemple de runbook d’automatisation Azure Powershell qui supprime tous les objets blob d’un conteneur de stockage Azure datant de plus d’un certain nombre de jours.

Cela peut être utile pour supprimer les anciennes sauvegardes SQL afin d’économiser de l’argent et de l’espace.

Il faut un certain nombre de paramètres qui sont explicites.

Remarque : j’ai laissé du code commenté pour faciliter le débogage.

Il utilise un principal de service qu’Azure peut configurer automatiquement pour vous lorsque vous créez votre compte d’automatisation. Vous devez avoir accès à Azure Active Directory. Voir photo :

<#
.DESCRIPTION
    Removes all blobs older than a number of days back using the Run As Account (Service Principal)

.NOTES
    AUTHOR: Russ
    LASTEDIT: Oct 03, 2016   #>

param(
    [parameter(Mandatory=$true)]
    [String]$resourceGroupName,

    [parameter(Mandatory=$true)]
    [String]$connectionName,

    # StorageAccount name for content deletion.
    [Parameter(Mandatory = $true)] 
    [String]$StorageAccountName,

    # StorageContainer name for content deletion.
    [Parameter(Mandatory = $true)] 
    [String]$ContainerName,

    [Parameter(Mandatory = $true)]
    [Int32]$DaysOld

)
$VerbosePreference = "Continue";
try
{
# Get the connection "AzureRunAsConnection "
$servicePrincipalConnection=Get-AutomationConnection -Name $connectionName         

"Logging in to Azure..."
Add-AzureRmAccount `
    -ServicePrincipal `
    -TenantId $servicePrincipalConnection.TenantId `
    -ApplicationId $servicePrincipalConnection.ApplicationId `
    -CertificateThumbprint $servicePrincipalConnection.CertificateThumbprint 
catch {
if (!$servicePrincipalConnection)
{
    $ErrorMessage = "Connection $connectionName not found."
    throw $ErrorMessage
} else{
    Write-Error -Message $_.Exception
    throw $_.Exception
}
$keys = Get-AzureRMStorageAccountKey -ResourceGroupName $resourceGroupName -AccountName $StorageAccountName
# get the storage account key
Write-Host "The storage key is: "$StorageAccountKey;
# get the context
$StorageAccountContext = New-AzureStorageContext -storageAccountName $StorageAccountName -StorageAccountKey $keys.Key1 #.Value;
$StorageAccountContext;
$existingContainer = Get-AzureStorageContainer -Context $StorageAccountContext -Name $ContainerName;
#$existingContainer;
if (!$existingContainer)
{
 "Could not find storage container";
} 
else 
{
$containerName = $existingContainer.Name;
Write-Verbose ("Found {0} storage container" -f $containerName);
$blobs = Get-AzureStorageBlob -Container $containerName -Context $StorageAccountContext;
$blobsremoved = 0;

if ($blobs -ne $null)
{    
    foreach ($blob in $blobs)
    {
        $lastModified = $blob.LastModified
        if ($lastModified -ne $null)
        {
            #Write-Verbose ("Now is: {0} and LastModified is:{1}" –f [DateTime]::Now, [DateTime]$lastModified);
            #Write-Verbose ("lastModified: {0}" –f $lastModified);
            #Write-Verbose ("Now: {0}" –f [DateTime]::Now);
            $blobDays = ([DateTime]::Now - $lastModified.DateTime)  #[DateTime]

            Write-Verbose ("Blob {0} has been in storage for {1} days" –f $blob.Name, $blobDays);

            Write-Verbose ("blobDays.Days: {0}" –f $blobDays.Hours);
            Write-Verbose ("DaysOld: {0}" –f $DaysOld);

            if ($blobDays.Days -ge $DaysOld)
            {
                Write-Verbose ("Removing Blob: {0}" –f $blob.Name);

                Remove-AzureStorageBlob -Blob $blob.Name -Container $containerName -Context $StorageAccountContext;
                $blobsremoved += 1;
            }
            else {
                Write-Verbose ("Not removing blob as it is not old enough.");
            }
        }
    }
}

Write-Verbose ("{0} blobs removed from container {1}." –f $blobsremoved, $containerName);
}

Maintenance de l’index #

Si vous vous souciez de vos index (oui, c’est le pluriel de index), vous devez les maintenir, surtout si vous faites des insertions ou des suppressions fréquentes qui les affectent. Azure Automation fournit un runbook presque prêt que vous pouvez utiliser et planifier pour effectuer une reconstruction automatisée de l’index. Voici ce que vous devez faire : Tout d’abord, importez le runbook :

Une fois le runbook importé, vous devez passer en mode édition et appuyer sur publier sur le runbook, et il devient actif. En mode édition, vous pouvez également consulter le code source du runbook.

Ensuite, vous devez ajouter un identifiant au runbook, qui peut être utilisé pour se connecter à la base de données (en gros, il s’agit simplement d’une paire clé-valeur, où la valeur est un nom d’utilisateur et un mot de passe, et la clé peut être utilisée à partir du script pour faire référence à ce titre particulier). Il doit s’agir d’une paire utilisateur-mot de passe pouvant s’authentifier auprès de la base de données, et l’utilisateur doit avoir le droit d’accéder à l’état de la base de données et d’exécuter l’instruction ALTER INDEX : Et enfin planifier le runbook avec les paramètres spécifiés. Vous pouvez également tester le runbook et le démarrer immédiatement (mais vous devez également spécifier les paramètres dans ce cas) :

Maintenant, malheureusement, j’ai trouvé qu’il y a deux problèmes avec ce runbook par défaut.

Tout d’abord, il ne gère que les tables qui se trouvent dans le schéma par défaut. Ce n’est pas toujours suffisant, alors allez-y et trouvez la ligne :

SELECT  t.name AS TableName, t.OBJECT_ID FROM sys.tables t

Et changez-le en ceci:

SELECT  '['+SCHEMA_NAME(t.schema_id)+'].['+t.name+']' AS TableName, t.OBJECT_ID FROM sys.tables t

De plus, le script ne peut pas gérer les caractères spéciaux n’importe où dans la chaîne de connexion (comme “,’ ou =). Pour les gérer, vous pouvez utiliser la chaîne de connexion appropriée en échappant, ou mieux encore, utiliser le générateur de chaîne de connexion. Partout où vous voyez une chaîne de connexion créé dans le script (il devrait y avoir deux emplacements), modifiez-le pour utiliser le générateur de chaîne de connexion :

$connStringBuilder = New-Object System.Data.SqlClient.SqlConnectionStringBuilder
$connStringBuilder["Server"] = "tcp:$using:SqlServer,$using:SqlServerPort"
$connStringBuilder["Database" ] = "$using:Database"
$connStringBuilder["User ID"] = "$using:SqlUsername"
$connStringBuilder["Password"] = "$using:SqlPass"
$connStringBuilder["Trusted_Connection"] = $False
$connStringBuilder["Encrypt"] = $True
$connStringBuilder["Connection Timeout"] = "30"
$connString = $connStringBuilder.ConnectionString                
$Conn = New-Object System.Data.SqlClient.SqlConnection($connString)

Vous pouvez également modifier la requête qui calcule un peu la fragmentation. C’est la requête :

SELECT a.object_id, avg_fragmentation_in_percent
    FROM sys.dm_db_index_physical_stats (
           DB_ID(N'$Database')
         , OBJECT_ID(0)
         , NULL
         , NULL
         , NULL) AS a
    JOIN sys.indexes AS b 
    ON a.object_id = b.object_id AND a.index_id = b.index_id;

Si vous modifiez le dernier paramètre de sys.dm_db_index_physical_stats de NULL à ‘DETAILED’, vous obtenez une bien meilleure estimation de la fragmentation des index.
