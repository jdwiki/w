Ajout de fichiers de test au projet Xcode #

Lors de la création du projet #

Vous devez cocher “Inclure les tests unitaires” dans la boîte de dialogue de création de projet.

[![entrez la description de l’image ici][1]][1]

Après avoir créé le projet #

Si vous avez manqué de vérifier cet élément lors de la création de votre projet, vous pouvez toujours ajouter des fichiers de test ultérieurement. Faire cela:

1- Accédez aux paramètres de votre projet dans Xcode

2- Allez dans “Cibles”

3- Cliquez sur “Ajouter une cible”

4- Sous “Autre”, sélectionnez “Cocoa Touch Unit Test Testing Bundle”

À la fin, vous devriez avoir un fichier nommé [Votre nom d'application]Tests.swift. En Objective-C, vous devriez avoir deux fichiers nommés [Votre nom d'application]Tests.h et [Votre nom d'application]Tests.m à la place.

Le fichier [Votre nom d'application]Tests.swift ou .m comprendra par défaut :

  • Un import de module XCTest
  • Une classe [Your app name]Tests qui étend XCTestCase
  • méthodes setUp, tearDown, testExample, testPerformanceExample

Ajout de méthodes de test #

Selon Apple :

Méthodes de test #

Une méthode de test est une méthode d’instance d’une classe de test qui commence par le préfixe test, ne prend aucun paramètre et renvoie void, par exemple, (void)testColorIsRed(). Une méthode de test teste le code dans votre projet et, si ce code ne produit pas le résultat attendu, signale les échecs à l’aide d’un ensemble d’API d’assertion. Par exemple, la valeur de retour d’une fonction peut être comparée à une valeur attendue ou votre test peut affirmer qu’une mauvaise utilisation d’une méthode dans l’une de vos classes lève une exception.

Nous ajoutons donc une méthode de test en utilisant “test” comme préfixe de la méthode, comme :

Rapide #

func testSomething() {

}

Objectif c #

- (void)testSomething {

}

Pour tester réellement les résultats, nous utilisons la méthode XCTAssert(), qui prend une expression booléenne et, si elle est vraie, marque le test comme réussi, sinon elle le marquera comme ayant échoué.

Disons que nous avons une méthode dans la classe View Controller appelée sum() qui calcule la somme de deux nombres. Pour le tester, nous utilisons cette méthode :

Rapide #

func testSum(){
    let result = viewController.sum(4, and: 5)
    XCTAssertEqual(result, 9)
}

Objectif c #

- (void)testSum {
    int result = [viewController sum:4 and:5];
    XCTAssertEqual(result, 9);
}

Remarque #

Par défaut, vous ne pouvez pas accéder à l’étiquette, à la zone de texte ou à d’autres éléments de l’interface utilisateur de la classe View Controller à partir de la classe de test s’ils sont d’abord créés dans le fichier Storyboard. C’est parce qu’ils sont initialisés dans la méthode loadView() de la classe View Controller, et cela ne sera pas appelé lors du test. La meilleure façon d’appeler loadView() et toutes les autres méthodes requises est d’accéder à la propriété view de notre propriété viewController. Vous devez ajouter cette ligne avant de tester les éléments de l’interface utilisateur :

XCTAssertNotNil(viewController.view)

Écrire une classe de test #

import XCTest
@testable import PersonApp

class PersonTests: XCTestCase {
    func test_completeName() {
        let person = Person(firstName: "Josh", lastName: "Brown")
        XCTAssertEqual(person.completeName(), "Josh Brown")
    }
}

Parlons maintenant de ce qui se passe ici. La ligne import XCTest nous permettra d’étendre XCTestCase et d’utiliser XCTAssertEqual (entre autres assertions). Étendre XCTestCase et préfixer notre nom de test avec test garantira que Xcode exécute automatiquement ce test lors de l’exécution des tests dans le projet (⌘U ou Product > Test). La ligne @testable import PersonApp importera notre cible PersonApp afin que nous puissions tester et utiliser des classes à partir de celle-ci, comme Person dans notre exemple ci-dessus. Et enfin, notre XCTAssertEqual s’assurera que person.completeName() est égal à la chaîne "Josh Brown".

Ajout de Storyboard et View Controller en tant qu’instances pour tester le fichier #

Pour commencer les tests unitaires, qui seront effectués dans le fichier de tests et testeront le View Controller et le Storyboard, nous devons introduire ces deux fichiers dans le fichier de test.

Définir le contrôleur de vue #

Rapide #

var viewController : ViewController!

Présentation du Storyboard et initialisation du View Controller #

Ajoutez ce code à la méthode setUp() :

Rapide #

let storyboard = UIStoryboard(name: "Main", bundle: nil)
viewController = storyboard.instantiateInitialViewController() as! ViewController

Objectif c #

UIStoryboard *storyboard = [UIStoryboard storyboardWithName:"Main" bundle:nil];
viewController = (ViewController *) [storyboard instantiateInitialViewController];

De cette façon, vous pouvez écrire des méthodes de test et ils sauront où vérifier les erreurs. Dans ce cas, il y a View Controller et le Storyboard.

Importer un module qu’il peut tester #

Les classes, les structures, les énumérations et toutes leurs méthodes sont “internes” par défaut. Cela signifie qu’ils ne sont accessibles qu’à partir du même module. Les cas de test sont dans une cible différente, ce qui signifie qu’ils se trouvent dans un module différent. Pour pouvoir accéder à la méthode que vous souhaitez tester, vous devez importer le module à tester à l’aide du mot-clé @testable.

Disons que nous avons un module principal appelé ToDo et que nous voulons écrire des tests pour celui-ci. Nous importerions ce module comme ceci :

@testable import ToDo

Toutes les méthodes de test du fichier avec cette instruction d’importation peuvent désormais accéder à toutes les classes, structures, énumérations “internes” et à toutes leurs méthodes “internes” du module “ToDo”.

Vous ne devez jamais ajouter les fichiers avec les éléments que vous souhaitez tester à la cible de test, car cela peut entraîner des erreurs difficiles à déboguer.

Chargement et apparence de la vue de déclenchement #

Afficher le chargement #

Dans un test pour un contrôleur de vue, vous souhaitez parfois déclencher l’exécution de loadView() ou viewDidLoad(). Cela peut être fait en accédant à la vue. Supposons que vous ayez une instance de contrôleur de vue dans votre test appelée sut (système sous test), alors le code ressemblerait à ceci :

XCTAssertNotNil(sut.view)

Afficher l’apparence #

Vous pouvez également déclencher les méthodes viewWillAppear(_:) et viewDidAppear(_:) en ajoutant le code suivant :

sut.beginAppearanceTransition(true, animated: true)
sut.endAppearanceTransition()
