La disposition automatique calcule dynamiquement la taille et la position de toutes les vues dans votre hiérarchie de vues, en fonction des contraintes placées sur ces vues. [Source][1]

[1] : https://developer.apple.com/library/content/documentation/UserExperience/Conceptual/AutolayoutPG/index.html

Syntaxe #

  • NSLayoutConstraint(item : Any, attribute : NSLayoutAttribute, relatedBy : NSLayoutRelation, toItem : Any ?, attribut : NSLayoutAttribute, multiplicateur : CGFloat, constant : CGFloat) // Créer une contrainte par programmation

Espacer les vues uniformément #

[![entrez la description de l’image ici][1]][1]

Il est courant de vouloir que deux vues soient côte à côte, centrées dans leur superview. La réponse commune donnée sur Stack Overflow est d’intégrer ces deux vues dans un UIView et de centrer le UIView. Ce n’est ni nécessaire ni recommandé. À partir de la documentation UILayoutGuide :

Il existe un certain nombre de coûts associés à l’ajout de vues factices à votre hiérarchie de vues. Premièrement, il y a le coût de création et de maintenance de la vue elle-même. Deuxièmement, la vue factice est un membre à part entière de la hiérarchie des vues, ce qui signifie qu’elle ajoute une surcharge à chaque tâche effectuée par la hiérarchie. Pire encore, la vue factice invisible peut intercepter des messages destinés à d’autres vues, ce qui cause des problèmes très difficiles à trouver.

Vous pouvez utiliser UILayoutGuide pour ce faire, au lieu d’ajouter les boutons dans un UIView inutile. Un UILayoutGuide est essentiellement un espace rectangulaire qui peut interagir avec la mise en page automatique. Vous placez un UILayoutGuide sur les côtés gauche et droit des boutons et définissez leurs largeurs pour qu’elles soient égales. Cela centrera les boutons. Voici comment le faire en code :

  • Style de langage de format visuel *

    view.addSubview(button1) view.addSubview(button2)

    let leftSpace = UILayoutGuide() view.addLayoutGuide(leftSpace)

Objectif c

- (void)viewDidLoad
{
    [super viewDidLoad];
    UIButton *yourButton = [[UIButton alloc] initWithFrame:CGRectMake(0, 0, 100, 18)];
    [yourButton setTitle:@"Button" forState:UIControlStateNormal];


    [self.view addConstraint:[NSLayoutConstraint constraintWithItem:yourButton attribute:NSLayoutAttributeCenterY relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeCenterY multiplier:1 constant:0]]; //Align veritcally center to superView

    [self.view addConstraint:[NSLayoutConstraint constraintWithItem:yourButton attribute:NSLayoutAttributeCenterX relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeCenterX multiplier:1 constant:0]]; //Align horizontally center to superView

    [self.view addSubview:yourButton]; //Add button to superView
}

Rapide

override func viewDidLoad() 
{
    super.viewDidLoad()
    let yourButton: UIButton = UIButton(frame: CGRect(x: 0, y: 0, width: 100, height: 18))
    yourButton.setTitle("Button", forState: .Normal)

    let centerVertically = NSLayoutConstraint(item: yourButton,
                                   attribute: .CenterX,
                                   relatedBy: .Equal,
                                      toItem: view,

    let centerHorizontally = NSLayoutConstraint(item: yourButton,
                                   attribute: .CenterY,
                                   relatedBy: .Equal,

                                   attribute: .CenterY,
                                  multiplier: 1.0,
                                    constant: 0.0)
    NSLayoutConstraint.activateConstraints([centerVertically, centerHorizontally])
}

Bases du langage de format visuel : contraintes dans le code ! #

HVFL est un langage conçu pour contraindre les éléments de l’interface utilisateur de manière simple et rapide. En règle générale, VFL présente un avantage par rapport à la personnalisation traditionnelle de l’interface utilisateur dans Interface Builder, car il est beaucoup plus lisible, accessible et compact.

Voici un exemple de VFL, dans lequel trois UIViews sont contraints de gauche à droite, remplissant superView.width, avec aGradeView

"H:|[bgView][aGradeView(40)][bGradeView(40)]|"

Il existe deux axes dans lesquels nous pouvons contraindre les objets de l’interface utilisateur, horizontalement et verticalement.

Chaque ligne de VFL commence toujours par H: ou V:. Si aucun n’est présent, l’option par défaut est H:

Passons à autre chose, nous avons un pipeline. | Ce symbole, ou le tuyau, fait référence à la vue d’ensemble. Si vous regardez de plus près l’extrait de code VFL ci-dessus, vous remarquerez deux de ces pipelines.

Cela signifie les deux extrémités horizontales de la supervue, les limites extérieures gauche et extérieure droite.

Ensuite, vous verrez des crochets, dans le premier ensemble de crochets, nous avons bgView. Lorsque nous avons des crochets, cela fait référence à un élément d’interface utilisateur, maintenant vous vous demandez peut-être comment nous établissons un lien entre le nom et l’élément d’interface utilisateur réel, un exutoire peut-être ?

Je couvrirai cela à la fin du post.

Si vous jetez un coup d’œil à la deuxième paire de crochets [aGradeView(50)], nous avons également des parenthèses encapsulées à l’intérieur, lorsqu’elles sont présentes, elles définissent la largeur/hauteur en fonction des axes, qui dans ce cas a une largeur de 50 pixels.

Les premiers crochets [bgView] n’avaient pas de largeur explicitement définie, ce qui signifie qu’ils s’étendront aussi loin que possible.

Très bien, c’est tout pour les bases, plus sur les trucs avancés dans un autre exemple.

par exemple: [![VFL_demo][1]][1]

    // 1. create views
    UIView *blueView = [[UIView alloc] init];
    blueView.backgroundColor = [UIColor blueColor];
    [self.view addSubview:blueView];

    UIView *redView = [[UIView alloc] init];
    redView.backgroundColor = [UIColor redColor];
    [self.view addSubview:redView];

    // 2. forbid Autoresizing
    blueView.translatesAutoresizingMaskIntoConstraints = NO;
    redView.translatesAutoresizingMaskIntoConstraints = NO;

    // 3. make contraints
    // horizontal

    // vertical
    NSArray *blueVandRedV = [NSLayoutConstraint constraintsWithVisualFormat:@"V:|-20-[blueView(50)]-20-[redView(==blueView)]" options:NSLayoutFormatAlignAllTrailing metrics:nil views:@{@"blueView" : blueView, @"redView" : redView}];
    [self.view addConstraints:blueVandRedV];

    NSLayoutConstraint *redW = [NSLayoutConstraint constraintWithItem:redView attribute:NSLayoutAttributeWidth relatedBy:NSLayoutRelationEqual toItem:blueView attribute:NSLayoutAttributeWidth multiplier:0.5 constant:0];
    [self.view addConstraint:redW];

Comment animer avec la mise en page automatique #

Sans mise en page automatique, l’animation est réalisée en modifiant le cadre d’une vue au fil du temps. Avec la mise en page automatique, les contraintes dictent le cadre de la vue, vous devez donc animer les contraintes à la place. Cette indirection rend l’animation plus difficile à visualiser.

Voici les façons d’animer avec la mise en page automatique :

  1. Modifiez la constante de la contrainte après sa création à l’aide d’appels périodiques (CADisplayLink, dispatch_source_t, dispatch_after, NSTimer). Appelez ensuite layoutIfNeeded pour mettre à jour la contrainte. Exemple:

Objectif c:

self.someConstraint.constant = 10.0;
[UIView animateWithDuration:0.25 animations:^{
    [self.view layoutIfNeeded];
}];

Rapide:

self.someConstraint.constant = 10.0
UIView.animate(withDuration: 0.25, animations: self.view.layoutIfNeeded)

  1. Modifiez les contraintes et appelez [view layoutIfNeeded] dans un bloc d’animation. Cela interpole entre les deux positions en ignorant les contraintes pendant l’animation.

    [UIView animateWithDuration:0.5 animations:^{ [view layoutIfNeeded]; }]

  2. Modifiez la priorité des contraintes. Cela sollicite moins le processeur que l’ajout et la suppression de contraintes.

  3. Supprimez toutes les contraintes et utilisez les masques de dimensionnement automatique. Pour ce dernier, vous devez définir view.translatesAutoresizingMaskIntoConstraints = YES.

  4. Utilisez des contraintes qui n’interfèrent pas avec l’animation prévue.

  5. Utilisez une vue de conteneur. Positionnez la supervision à l’aide de contraintes. Ajoutez ensuite une sous-vue avec des contraintes qui ne combattent pas l’animation, par exemple : un centre par rapport à la supervue. Cela décharge une partie des contraintes sur la supervue, afin qu’elles ne combattent pas l’animation dans la sous-vue.

  6. Animez les calques à la place des vues. Les transformations de calque ne déclenchent pas la mise en page automatique.

    CABasicAnimation* ba = [CABasicAnimation animationWithKeyPath:@“transform”]; ba.autoreverses = YES; ba.duration = 0.3; ba.toValue = [NSValue valueWithCATransform3D:CATransform3DMakeScale(1.1, 1.1, 1)]; [v.layer addAnimation:ba forKey:nil];

  7. ** Remplacer les sous-vues de mise en page **. Appelez [super layoutSubviews] et affinez les contraintes.

  8. Modifiez le cadre dans viewDidLayoutSubviews. La mise en page automatique est appliquée dans layoutSubviews, donc une fois cela fait, modifiez-la dans viewDidLayoutSubviews.

10. Désactivez la mise en page automatique et définissez les vues manuellement. Vous pouvez le faire en remplaçant layoutSubviews/layout sans appeler l’implémentation de la super classe.

Conseil rapide : si le parent de la vue animée n’est pas interpolé (c’est-à-dire que l’animation saute de l’état du début à la fin), appelez layoutIfNeeded() dans la vue la plus profonde qui est le parent de la vue animée (dans autrement dit, qui n’est pas affecté par l’animation). Je ne sais pas exactement pourquoi cela fonctionne.

Contrainte créée en tant que

NSLayoutConstraint(item: myView, attribute: NSLayoutAttribute.Leading, relatedBy: NSLayoutRelation.Equal, toItem: view, attribute: NSLayoutAttribute.LeadingMargin, multiplier: 1.0, constant: 20.0)

ou, d’un point de vue mathématique :

view.attribute * multiplier + constant          (1)

Vous pouvez utiliser le multiplicateur pour créer une disposition proportionnelle pour différents facteurs de taille.

Exemple:

Turquoise View (V1) est un carré avec une largeur de superview proportionnelle à la largeur avec un rapport de 1:1.1

Le carré de Gary (V2) est une sous-vue de V1. Espace inférieur défini par constante = 60, Espace final défini par multiplicateur = 1,125 et constante = 0

L’espace de fin est défini proportionnellement, l’espace du bas est défini comme une constante.

[1] : http://i.stack.imgur.com/2LSlx.jpg Remarque : si view.attribute est égal à 0 (par exemple, un espace de début), la formule de contrainte (1) sera égale à 0. Vous devez modifier le deuxième élément de contrainte ou définir une contrainte relative à la marge, afin de view.attribute != 0.

NSLayoutConstraint: Contraints in code! #

Lorsque nous travaillons sur un framework, si les contraintes ne sont pas trop complexes, nous ferions mieux d’utiliser Interface Builder ou NSLayoutConstraint dans le code pour le rendre suffisamment petit, au lieu d’importer Masonry ou SnapKit.

par exemple: [![entrez la description de l’image ici][1]][1]

  • Objectif c
    // 1. create views
    UIView *blueView = [[UIView alloc] init];
    blueView.backgroundColor = [UIColor blueColor];
    [self.view addSubview:blueView];

    UIView *redView = [[UIView alloc] init];
    redView.backgroundColor = [UIColor redColor];
    [self.view addSubview:redView];

    // 2. forbid Autoresizing
    blueView.translatesAutoresizingMaskIntoConstraints = NO;
    redView.translatesAutoresizingMaskIntoConstraints = NO;

    // 3. make contraints
    // 3.1 blueView
    NSLayoutConstraint *blueLeft = [NSLayoutConstraint constraintWithItem:blueView attribute:NSLayoutAttributeLeft relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeLeft multiplier:1 constant:20];
    [self.view addConstraint:blueLeft];

    NSLayoutConstraint *blueTop = [NSLayoutConstraint constraintWithItem:blueView attribute:NSLayoutAttributeTop relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeTop multiplier:1 constant:20];
    [self.view addConstraint:blueTop];

    NSLayoutConstraint *blueRight = [NSLayoutConstraint constraintWithItem:blueView attribute:NSLayoutAttributeRight relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeRight multiplier:1 constant:-20];
    [self.view addConstraint:blueRight];

    NSLayoutConstraint *blueHeight = [NSLayoutConstraint constraintWithItem:blueView attribute:NSLayoutAttributeHeight relatedBy:NSLayoutRelationEqual toItem:nil attribute:NSLayoutAttributeNotAnAttribute multiplier:1 constant:50];
    [self.view addConstraint:blueHeight];

    // 3.2 redView
    NSLayoutConstraint *redTop = [NSLayoutConstraint constraintWithItem:redView attribute:NSLayoutAttributeTop relatedBy:NSLayoutRelationEqual toItem:blueView attribute:NSLayoutAttributeBottom multiplier:1 constant:20];
    [self.view addConstraint:redTop];

    NSLayoutConstraint *redRight = [NSLayoutConstraint constraintWithItem:redView attribute:NSLayoutAttributeRight relatedBy:NSLayoutRelationEqual toItem:self.view attribute:NSLayoutAttributeRight multiplier:1 constant:-20];
    [self.view addConstraint:redRight];

    NSLayoutConstraint *redHeight = [NSLayoutConstraint constraintWithItem:redView attribute:NSLayoutAttributeHeight relatedBy:NSLayoutRelationEqual toItem:blueView attribute:NSLayoutAttributeHeight multiplier:1 constant:0];
    [self.view addConstraint:redHeight];

    NSLayoutConstraint *redWidth = [NSLayoutConstraint constraintWithItem:redView attribute:NSLayoutAttributeWidth relatedBy:NSLayoutRelationEqual toItem:blueView attribute:NSLayoutAttributeWidth multiplier:0.5 constant:0];
    [self.view addConstraint:redWidth];

Comment utiliser la mise en page automatique #

La mise en page automatique est utilisée pour organiser les vues afin qu’elles soient belles sur n’importe quel appareil et orientation. Les contraintes sont les règles qui disent comment tout doit être posé. Ils comprennent l’épinglage des bords, le centrage et la définition des tailles, entre autres.

La mise en page automatique est activée par défaut, mais vous pouvez vérifier cela. Si vous cliquez sur Main.storyboard dans le navigateur du projet, puis affichez l’inspecteur de fichiers. Assurez-vous que Disposition automatique et Classes de taille sont cochées :

[![entrez la description de l’image ici][1]][1]

Les contraintes de mise en page automatique peuvent être définies dans Interface Builder ou dans le code. Dans Interface Builder, vous trouverez les outils de mise en page automatique en bas à droite. En cliquant dessus, vous découvrirez différentes options pour définir les contraintes sur une vue.

[![entrez la description de l’image ici][2]][2]

Si vous souhaitez avoir différentes contraintes pour différentes tailles ou orientations de périphérique, vous pouvez les définir dans les options de classe de taille wAny hAny trouvées en bas au milieu.

Utilisation mixte de la mise en page automatique avec la mise en page non automatique #

Parfois, vous souhaiterez peut-être effectuer des actions supplémentaires sur les calculs Auto Layout effectués par UIKit lui-même.

Exemple: lorsque vous avez un UIView qui a un maskLayer, vous devrez peut-être mettre à jour maskLayer dès que Auto Layout change le frame de UIView

// CustomView.m
- (void)layoutSubviews {
    [super layoutSubviews];
    // now you can assume Auto Layout did its job
    // you can use view's frame in your calculations
    CALayer maskLayer = self.maskLayer;
    maskLayer.bounds = self.bounds;
    ...
}

ou si vous souhaitez effectuer des actions supplémentaires pour Mise en page automatique dans ViewController

- (void)viewDidLayoutSubviews {
    [super viewDidLayoutSubviews];
    // now you can assume all your subviews are positioned/resized correctly
    self.customView.frame = self.containerView.frame;
}
