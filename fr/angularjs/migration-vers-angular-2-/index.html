  • Conversion de votre application AngularJS en une structure orientée composants
  • Présentation des modules Webpack et ES6

AngularJS a été totalement réécrit en utilisant le langage TypeScript et renommé en juste Angular.

Il y a beaucoup à faire avec une application AngularJS pour faciliter le processus de migration. Comme le guide de mise à jour officiel l’indique, plusieurs “étapes de préparation” peuvent être effectuées pour refactoriser votre application, la rendant meilleure et plus proche de la nouveau style angulaire.

Conversion de votre application AngularJS en une structure orientée composants #

Dans le nouveau framework Angular, les composants sont les principaux blocs de construction qui composent l’interface utilisateur. Ainsi, l’une des premières étapes permettant de migrer une application AngularJS vers le nouvel Angular consiste à la refactoriser dans une structure plus orientée composants.

Des composants ont également été introduits dans l’ancien AngularJS à partir de la version 1.5+. L’utilisation de composants dans une application AngularJS rapprochera non seulement sa structure du nouveau Angular 2+, mais la rendra également plus modulaire et plus facile à entretenir.

Avant d’aller plus loin, je recommande de consulter la page de documentation officielle d’AngularJS sur les composants, où leurs avantages et leur utilisation sont bien expliqués.

Je préférerais mentionner quelques astuces sur la façon de convertir l’ancien code orienté ng-controller vers le nouveau style orienté component.

Commencez à diviser votre application en composants #

Toutes les applications orientées composants ont généralement un ou quelques composants qui incluent d’autres sous-composants. Alors pourquoi ne pas créer le premier composant qui contiendra simplement votre application (ou une grande partie de celle-ci).

Supposons que nous ayons un morceau de code assigné à un contrôleur, nommé UserListController, et que nous voulions en faire un composant, que nous nommerons UserListComponent.

HTML actuel :

<div ng-controller="UserListController as listctrl">
    <ul>
        <li ng-repeat="user in myUserList">
            {{ user }}
        </li>
    </ul>
</div>

JavaScript actuel :

app.controller("UserListController", function($scope, SomeService) {

    $scope.myUserList = ['Shin', 'Helias', 'Kalhac'];

    this.someFunction = function() {
        // ...
    }

    // ...
}

nouveau HTML :

<user-list></user-list>

nouveau JavaScript :

app.component("UserList", {
    templateUrl: 'user-list.html',
    controller: UserListController
});

function UserListController(SomeService) {

    this.myUserList = ['Shin', 'Helias', 'Kalhac'];

    this.someFunction = function() {
        // ...
    }

    // ...
}

Notez que nous n’injectons plus $scope dans la fonction du contrôleur et nous déclarons maintenant this.myUserList au lieu de $scope.myUserList ;

nouveau fichier modèle user-list.component.html :

<ul>
    <li ng-repeat="user in $ctrl.myUserList">
        {{ user }}
    </li>
</ul>

Notez comment nous nous référons maintenant à la variable myUserList, qui appartient au contrôleur, en utilisant $ctrl.myUserList du html au lieu de $scope.myUserList.

En effet, comme vous l’avez probablement compris après avoir lu la documentation, $ctrl dans le modèle fait désormais référence à la fonction de contrôleur.

Qu’en est-il des contrôleurs et des routes ? #

Dans le cas où votre contrôleur était lié au modèle en utilisant le système de routage au lieu de ng-controller, donc si vous avez quelque chose comme ceci :

$stateProvider
  .state('users', {
    url: '/users',
    templateUrl: 'user-list.html',
    controller: 'UserListController'
  })
  // ..

vous pouvez simplement changer votre déclaration d’état en :

$stateProvider
    .state('users', {
      url: '/',
      template: '<user-list></user-list>'
    })
    // ..

Et après? #

Maintenant que vous avez un composant contenant votre application (qu’il contienne l’intégralité de l’application ou une partie de celle-ci, comme une vue), vous devez maintenant commencer à diviser votre composant en plusieurs composants imbriqués, en enveloppant des parties de celui-ci dans de nouveaux sous-composants , etc.

Vous devriez commencer à utiliser les fonctionnalités du composant telles que

  • Liaisons Entrées et Sorties

  • hooks de cycle de vie tels que $onInit(), $onChanges(), etc…

Après avoir lu la Documentation sur les composants, vous devriez déjà savoir comment utiliser toutes ces fonctionnalités de composants, mais si vous avez besoin d’un exemple concret d’une véritable application simple, vous pouvez vérifier ceci.

De plus, si dans le contrôleur de votre composant vous avez des fonctions qui contiennent beaucoup de code logique, une bonne idée peut être d’envisager de déplacer cette logique dans services.

Conclusion #

L’adoption d’une approche basée sur les composants rapproche votre AngularJS d’un pas de plus pour le migrer vers le nouveau framework Angular, mais cela le rend également meilleur et beaucoup plus modulaire.

Bien sûr, il y a beaucoup d’autres étapes que vous pouvez faire pour aller plus loin dans la nouvelle direction Angular 2+, que je vais énumérer dans les exemples suivants.

Présentation des modules Webpack et ES6 #

En utilisant un module loader comme Webpack, nous pouvons bénéficier du système de module intégré disponible dans [ES6](https://en .wikipedia.org/wiki/ECMAScript#6th_Edition_-_ECMAScript_2015) (ainsi que dans TypeScript). Nous pouvons ensuite utiliser import et [export](https://developer.mozilla.org/en/ docs/web/javascript/reference/statements/export) des fonctionnalités qui nous permettent de spécifier les morceaux de code que nous allons partager entre les différentes parties de l’application.

Lorsque nous mettons ensuite nos applications en production, les chargeurs de modules facilitent également leur conditionnement dans des ensembles de production avec des batteries incluses.
