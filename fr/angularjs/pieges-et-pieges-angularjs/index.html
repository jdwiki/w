Choses à faire lors de l’utilisation de html5Mode #

Lors de l’utilisation de html5Mode([mode]), il est nécessaire que :

  1. Vous spécifiez l’URL de base de l’application avec un <base href=""> dans l’en-tête de votre index.html.

  2. Il est important que la balise base vienne avant toutes les balises avec des demandes d’URL. Sinon, cela pourrait entraîner cette erreur - "Ressource interprétée comme feuille de style mais transférée avec le type MIME text/html". Par exemple:

     <head>
     <meta charset="utf-8">
     <title>Job Seeker</title>

     <base href="/">

     <link rel="stylesheet" href="bower_components/bootstrap/dist/css/bootstrap.css" />
     <link rel="stylesheet" href="/styles/main.css">
 </head>

  3. Si vous ne voulez pas spécifier de balise base, configurez $locationProvider pour ne pas exiger de balise base en passant un objet de définition avec requireBase:false à $locationProvider.html5Mode() comme ceci :

     $locationProvider.html5Mode({
     enabled: true,
     requireBase: false
 });

  4. Afin de prendre en charge le chargement direct des URL HTML5, vous devez activer la réécriture d’URL côté serveur. À partir de [AngularJS / Guide du développeur / Utilisation de $location][1]

L’utilisation de ce mode nécessite une réécriture d’URL côté serveur, en gros, vous devez réécrire tous vos liens vers le point d’entrée de votre application (par exemple, index.html). L’exigence d’une balise <base> est également importante dans ce cas, car elle permet à Angular de différencier la partie de l’url qui est la base de l’application et le chemin qui doit être géré par l’application.

Une excellente ressource pour des exemples de réécriture de requêtes pour diverses implémentations de serveur HTTP peut être trouvée dans [ui-router FAQ - How to: Configure your server to work with html5Mode][2]. Par exemple, Apache

    RewriteEngine on

    # Don't rewrite files or directories
    RewriteCond %{REQUEST_FILENAME} -f [OR]
    RewriteCond %{REQUEST_FILENAME} -d
    RewriteRule ^ - [L]

    # Rewrite everything else to index.html to allow html5 state links
    RewriteRule ^ index.html [L]

nginx

    server {
        server_name my-app;

        root /path/to/app;

        location / {
            try_files $uri $uri/ /index.html;
        }
    }

Exprimer

    var express = require('express');
    var app = express();

    app.use('/js', express.static(__dirname + '/js'));
    app.use('/dist', express.static(__dirname + '/../dist'));
    app.use('/css', express.static(__dirname + '/css'));
    app.use('/partials', express.static(__dirname + '/partials'));

    app.all('/*', function(req, res, next) {
        // Just send the index.html for other files to support HTML5Mode
        res.sendFile('index.html', { root: __dirname });
    });

    app.listen(3006); //the port you want to use

[1] : https://docs.angularjs.org/guide/$location#server-side [2] : https://github.com/angular-ui/ui-router/wiki/Frequently-Asked-Questions#how-to-configure-your-server-to-work-with-html5mode

La liaison de données bidirectionnelle cesse de fonctionner #

Il faut avoir à l’esprit que :

  1. La liaison de données d’Angular repose sur l’héritage prototypique de JavaScript, elle est donc soumise à variable shadowing.
  2. Une portée enfant hérite normalement de sa portée parent. Une exception à cette règle est une directive qui a une portée isolée car elle n’hérite pas de manière prototypique.
  3. Certaines directives créent une nouvelle portée enfant : ng-repeat, ng-switch, ng-view, ng-if, ng-controller, ng-include, etc. .

Cela signifie que lorsque vous essayez de lier dans les deux sens certaines données à une primitive qui se trouve à l’intérieur d’une portée enfant (ou vice-versa), les choses peuvent ne pas fonctionner comme prévu. Ici est un exemple de la facilité avec laquelle “casser” AngularJS.

Ce problème peut facilement être évité en suivant ces étapes :

  1. Avoir un “.” à l’intérieur de votre modèle HTML chaque fois que vous liez des données
  2. Utilisez la syntaxe controllerAs car elle favorise l’utilisation de la liaison à un objet “pointillé”
  3. $parent peut être utilisé pour accéder aux variables scope parent plutôt qu’à la portée enfant. comme à l’intérieur de ng-if nous pouvons utiliser ng-model="$parent.foo"..

Une alternative à ce qui précède consiste à lier ngModel à une fonction getter/setter qui mettra à jour la version en cache du modèle lorsqu’elle est appelée avec des arguments, ou la renverra lorsqu’elle est appelée sans arguments. Pour utiliser une fonction getter/setter, vous devez ajouter ng-model-options="{ getterSetter: true }" à l’élément avec l’attribut ngModal et appeler la fonction getter si vous souhaitez afficher sa valeur dans l’expression (Exemple de travail).

Exemple #

Voir:

<div ng-app="myApp" ng-controller="MainCtrl">
    <input type="text" ng-model="foo" ng-model-options="{ getterSetter: true }">
    <div ng-if="truthyValue">
        <!-- I'm a child scope (inside ng-if), but i'm synced with changes from the outside scope -->
        <input type="text" ng-model="foo">
    </div>
    <div>$scope.foo: {{ foo() }}</div>
</div>

Manette:

angular.module('myApp', []).controller('MainCtrl', ['$scope', function($scope) {
    $scope.truthyValue = true;
      
    var _foo = 'hello'; // this will be used to cache/represent the value of the 'foo' model 
      
    $scope.foo = function(val) {
        // the function return the the internal '_foo' varibale when called with zero arguments,
        // and update the internal `_foo` when called with an argument
        return arguments.length ? (_foo = val) : _foo;
    };
}]);

Meilleure pratique : il est préférable de garder les getters rapides, car Angular est susceptible de les appeler plus fréquemment que d’autres parties de votre code (référence).

7 péchés capitaux d’AngularJS #

Vous trouverez ci-dessous la liste de certaines erreurs que les développeurs commettent souvent lors de l’utilisation des fonctionnalités d’AngularJS, quelques leçons apprises et des solutions à celles-ci.

1. Manipulation du DOM via le contrôleur

C’est légal, mais il faut l’éviter. Les contrôleurs sont les endroits où vous définissez vos dépendances, liez vos données à la vue et créez une logique métier supplémentaire. Vous pouvez techniquement manipuler le DOM dans un contrôleur, mais chaque fois que vous avez besoin d’une manipulation identique ou similaire dans une autre partie de votre application, un autre contrôleur sera nécessaire. La meilleure pratique de cette approche consiste donc à créer une directive qui inclut toutes les manipulations et à utiliser la directive dans toute votre application. Par conséquent, le contrôleur laisse la vue intacte et fait son travail. Dans une directive, la fonction de liaison est le meilleur endroit pour manipuler le DOM. Il a un accès complet à la portée et à l’élément, donc en utilisant une directive, vous pouvez également profiter de la réutilisabilité.

link: function($scope, element, attrs) {
    //The best place to manipulate DOM
}

Vous pouvez accéder aux éléments DOM dans la fonction de liaison de plusieurs manières, telles que le paramètre element, la méthode angular.element() ou le Javascript pur.

2. Liaison de données en transclusion

AngularJS est célèbre pour sa liaison de données bidirectionnelle. Cependant, vous pouvez parfois constater que vos données ne sont liées qu’à sens unique à l’intérieur de directives. Arrêtez-vous là, AngularJS n’a pas tort mais probablement vous. Les directives sont des endroits un peu dangereux puisque des portées enfants et des portées isolées sont impliquées. Supposons que vous ayez la directive suivante avec une transclusion

<my-dir>
    <my-transclusion>
    </my-transclusion>
</my-dir>

Et à l’intérieur de my-transclusion, vous avez des éléments qui sont liés aux données dans la portée externe.

<my-dir>
    <my-transclusion>
        <input ng-model="name">
    </my-transclusion>
</my-dir>

Le code ci-dessus ne fonctionnera pas correctement. Ici, la transclusion crée une portée enfant et vous pouvez obtenir la variable de nom, à droite, mais tout changement que vous apportez à cette variable y restera. Ainsi, vous pouvez vraiment accéder à cette variable en tant que $parent.name. Cependant, cette utilisation n’est peut-être pas la meilleure pratique. Une meilleure approche consisterait à envelopper les variables dans un objet. Par exemple, dans le contrôleur, vous pouvez créer :

$scope.data = {
    name: 'someName'
}

Ensuite, dans la transclusion, vous pouvez accéder à cette variable via l’objet ‘data’ et voir que la liaison bidirectionnelle fonctionne parfaitement !

<input ng-model="data.name">

Non seulement dans les transclusions, mais dans toute l’application, c’est une bonne idée d’utiliser la notation pointée.

3. Plusieurs directives ensemble

Il est en fait légal d’utiliser deux directives ensemble dans le même élément, tant que vous respectez la règle : deux portées isolées ne peuvent pas exister sur le même élément. De manière générale, lors de la création d’une nouvelle directive personnalisée, vous allouez une étendue isolée pour faciliter le passage des paramètres. En supposant que les directives myDirA et myDirB ont des étendues isolées et que myDirC ne l’a pas, l’élément suivant sera valide :

<input my-dir-a my-dirc>

alors que l’élément suivant provoquera une erreur de console :

<input my-dir-a my-dir-b>

Les directives doivent donc être utilisées à bon escient, en tenant compte des champs d’application.

4. Utilisation abusive de $emit

$emit, $broadcast et $on fonctionnent selon le principe émetteur-récepteur. En d’autres termes, ils sont un moyen de communication entre les contrôleurs. Par exemple, la ligne suivante émet le ‘someEvent’ du contrôleur A, qui sera intercepté par le contrôleur B concerné.

$scope.$emit('someEvent', args);

Et la ligne suivante attrape le ‘someEvent’

$scope.$on('someEvent', function(){});

Jusqu’ici tout semble parfait. Mais rappelez-vous que, si le contrôleur B n’est pas encore invoqué, l’événement ne sera pas intercepté, ce qui signifie que les contrôleurs émetteur et récepteur doivent être invoqués pour que cela fonctionne. Donc, encore une fois, si vous n’êtes pas sûr de devoir utiliser $emit, la création d’un service semble être une meilleure solution.

5. Utilisation abusive de $scope.$watch

$scope.$watch est utilisé pour observer un changement de variable. Chaque fois qu’une variable a changé, cette méthode est invoquée. Cependant, une erreur courante est de changer la variable à l’intérieur de $scope.$watch. Cela entraînera une incohérence et une boucle $digest infinie à un moment donné.

$scope.$watch('myCtrl.myVariable', function(newVal) {
    this.myVariable++;
});

Donc, dans la fonction ci-dessus, assurez-vous que vous n’avez aucune opération sur myVariable et newVal.

6. Méthodes de liaison aux vues

C’est l’un des péchés les plus mortels. AngularJS a une liaison bidirectionnelle, et chaque fois que quelque chose change, les vues sont mises à jour plusieurs fois. Ainsi, si vous liez une méthode à un attribut d’une vue, cette méthode peut potentiellement être appelée une centaine de fois, ce qui vous rend également fou lors du débogage. Cependant, seuls certains attributs sont créés pour la liaison de méthode, tels que ng-click, ng-blur, ng-on-change, etc., qui attendent des méthodes comme paramètre. Par exemple, supposons que vous ayez la vue suivante dans votre balisage :

<input ng-disabled="myCtrl.isDisabled()" ng-model="myCtrl.name">

Ici, vous vérifiez l’état désactivé de la vue via la méthode isDisabled. Dans le contrôleur myCtrl, vous avez :

vm.isDisabled = function(){
    if(someCondition)
        return true;
    else
        return false;
}

En théorie, cela peut sembler correct, mais techniquement, cela entraînera une surcharge, car la méthode s’exécutera un nombre incalculable de fois. Pour résoudre ce problème, vous devez lier une variable. Dans votre contrôleur, la variable suivante doit exister :

vm.isDisabled

Vous pouvez à nouveau initialiser cette variable lors de l’activation du contrôleur

if(someCondition)
    vm.isDisabled = true
else
    vm.isDisabled = false

Si la condition n’est pas stable, vous pouvez la lier à un autre événement. Ensuite, vous devez lier cette variable à la vue :

<input ng-disabled="myCtrl.isDisabled" ng-model="myCtrl.name">

Désormais, tous les attributs de la vue ont ce qu’ils attendent et les méthodes ne s’exécuteront qu’en cas de besoin.

7. Ne pas utiliser les fonctionnalités d’Angular

AngularJS offre une grande commodité avec certaines de ses fonctionnalités, non seulement en simplifiant votre code mais aussi en le rendant plus efficace. Certaines de ces fonctionnalités sont répertoriées ci-dessous :

  1. angular.forEach for the loops (Caution, you can't "break;" it, you can only prevent getting into the body, so consider performance here.)
  2. angular.element for DOM selectors
  3. angular.copy: Use this when you should not modify the main object
  4. Form validations are already awesome. Use dirty, pristine, touched, valid, required and so on.
  5. Besides Chrome debugger, use remote debugging for mobile development too.
  6. And make sure you use Batarang. It's a free Chrome extension where you can easily inspect scopes
