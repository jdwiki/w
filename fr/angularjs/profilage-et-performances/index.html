7 Améliorations simples des performances #

1) Utilisez ng-repeat avec parcimonie

L’utilisation de [ng-repeat][1] dans les vues entraîne généralement de mauvaises performances, en particulier lorsqu’il y a des ng-repeat imbriqués.

** C’est super lent ! **

<div ng-repeat="user in userCollection">
  <div ng-repeat="details in user">
    {{details}}
  </div>
</div>

Essayez d’éviter autant que possible les répétitions imbriquées. Une façon d’améliorer les performances de ng-repeat consiste à utiliser track by $index (ou un autre champ d’identification). Par défaut, ng-repeat suit l’objet entier. Avec track by, Angular surveille l’objet uniquement par le $ index ou l’identifiant de l’objet.

<div ng-repeat="user in userCollection track by $index">
  {{user.data}}
</div>

Utilisez d’autres approches comme pagination, [parchemins virtuels](http://stackoverflow. com/questions/33259241/how-can-i-make-an-virtual-scroll-with-angularjs), [infinite scrolls](http://stackoverflow.com/questions/21674266/angularjs-infinite-scroll-in- a-container) ou [limitTo: begin][2] chaque fois que possible pour éviter d’itérer sur de grandes collections.

2) Lier une fois #

Angular a une liaison de données bidirectionnelle. Il a le coût d’être lent s’il est trop utilisé.

** Performances plus lentes **

<!-- Default data binding has a performance cost -->
<div>{{ my.data }}</div>

Performances plus rapides (AngularJS >= 1.3)

<!-- Bind once is much faster -->
<div>{{ ::my.data }}</div>

<div ng-bind="::my.data"></div>

<!-- Use single binding notation in ng-repeat where only list display is needed  -->
<div ng-repeat="user in ::userCollection">
  {{::user.data}}
</div>

L’utilisation de la notation “lier une fois” indique à Angular d’attendre que la valeur se stabilise après la première série de cycles de digestion. Angular utilisera cette valeur dans le DOM, puis supprimera tous les observateurs afin qu’elle devienne une valeur statique et ne soit plus liée au modèle.

Le {{}} est beaucoup plus lent.

Ce ng-bind est une directive et placera un observateur sur la variable transmise. Ainsi, le ng-bind ne s’appliquera que lorsque la valeur transmise changera réellement.

Les parenthèses, d’autre part, seront vérifiées et rafraîchies à chaque $digest, même si ce n’est pas nécessaire.

3) Les fonctions d’oscilloscope et les filtres prennent du temps #

AngularJS a une boucle digest. Toutes vos fonctions sont dans une vue et les filtres sont exécutés à chaque fois que le cycle de résumé s’exécute. La boucle de résumé sera exécutée chaque fois que le modèle est mis à jour et cela peut ralentir votre application (le filtre peut être activé plusieurs fois avant le chargement de la page).

Évitez ceci :

<div ng-controller="bigCalulations as calc">
  <p>{{calc.calculateMe()}}</p>
  <p>{{calc.data | heavyFilter}}</p>
</div>

Meilleure approche

<div ng-controller="bigCalulations as calc">
  <p>{{calc.preCalculatedValue}}</p>
  <p>{{calc.data | lightFilter}}</p>
</div>

Où le contrôleur peut être :

app.controller('bigCalulations', function(valueService) {
    // bad, because this is called in every digest loop
    this.calculateMe = function() {
        var t = 0;
        for(i = 0; i < 1000; i++) {
            t += i;
        }
        return t;
    }
    // good, because this is executed just once and logic is separated in service to    keep the controller light
    this.preCalulatedValue = valueService.valueCalculation(); // returns 499500
});

4 spectateurs #

Les observateurs chutent considérablement les performances. Avec plus d’observateurs, la boucle de résumé prendra plus de temps et l’interface utilisateur ralentira. Si l’observateur détecte un changement, il lancera la boucle de résumé et restituera la vue.

Il existe trois façons de surveiller manuellement les changements de variables dans Angular.

$watch() - surveille les changements de valeur

$watchCollection() - surveille les changements dans la collection (surveille plus que $watch normal)

$watch(..., true) - Évitez cela autant que possible, il effectuera une “surveillance profonde” et diminuera les performances (regarde plus que watchCollection)

Notez que si vous liez des variables dans la vue, vous créez de nouvelles montres - utilisez {{::variable}} pour empêcher la création d’une montre, en particulier dans les boucles.

Par conséquent, vous devez suivre le nombre d’observateurs que vous utilisez. Vous pouvez compter les observateurs avec ce script (crédit à [@Words Like Jared][3] [Number of watchers][4])

(function() {
    var root = angular.element(document.getElementsByTagName('body')),
        watchers = [],
        f = function(element) {
        angular.forEach(['$scope', '$isolateScope'], function(scopeProperty) {
            if(element.data() && element.data().hasOwnProperty(scopeProperty)) {
                angular.forEach(element.data()[scopeProperty].$$watchers, function(watcher) {
                watchers.push(watcher);
                });
            }
        });

        angular.forEach(element.children(), function(childElement) {
            f(angular.element(childElement));
        });
    };
 
    f(root);
 
    // Remove duplicate watchers
    var watchersWithoutDuplicates = [];
    angular.forEach(watchers, function(item) {
        if(watchersWithoutDuplicates.indexOf(item) < 0) {
            watchersWithoutDuplicates.push(item);
        }
    });
    console.log(watchersWithoutDuplicates.length);
})();

5) of-if / of-show #

Ces fonctions ont un comportement très similaire. [ng-if][5] supprime des éléments du DOM tandis que [ng-show][6] ne masque que les éléments mais conserve tous les gestionnaires. Si vous avez des parties du code que vous ne voulez pas afficher, utilisez ng-if.

Cela dépend du type d’utilisation, mais souvent l’un est plus adapté que l’autre.

  • Si l’élément n’est pas nécessaire, utilisez ng-if

  • Pour activer/désactiver rapidement, utilisez ng-show/ng-hide

    <div ng-repeat="user in userCollection">
  <p ng-if="user.hasTreeLegs">I am special<!-- some complicated DOM --></p>
  <p ng-show="user.hasSubscribed">I am awesome<!-- switch this setting on and off --></p>
</div>

En cas de doute, utilisez ng-if et testez !

6) Désactiver le débogage #

Par défaut, les directives et les portées de liaison laissent des classes et des balises supplémentaires dans le code pour faciliter divers outils de débogage. La désactivation de cette option signifie que vous ne restituez plus ces différents éléments lors du cycle de résumé.

angular.module('exampleApp', []).config(['$compileProvider', function ($compileProvider) {
    $compileProvider.debugInfoEnabled(false);
}]);

7) Utilisez l’injection de dépendances pour exposer vos ressources #

L’injection de dépendance est un modèle de conception de logiciel dans lequel un objet reçoit ses dépendances, plutôt que l’objet les crée lui-même. Il s’agit de supprimer les dépendances codées en dur et de permettre de les modifier chaque fois que nécessaire.

Vous pourriez vous interroger sur le coût de performance associé à une telle analyse de chaîne de toutes les fonctions injectables. Angular s’en occupe en mettant en cache la propriété $inject après la première fois. Cela ne se produit donc pas à chaque fois qu’une fonction doit être invoquée.

CONSEIL DE PRO : Si vous recherchez l’approche offrant les meilleures performances, optez pour l’approche d’annotation de propriété $inject. Cette approche évite entièrement l’analyse de la définition de la fonction car cette logique est enveloppée dans la vérification suivante dans la fonction annotate : if (!($inject = fn.$inject)). Si $inject est déjà disponible, aucune analyse n’est requise !

var app = angular.module('DemoApp', []);

var DemoController = function (s, h) {
    h.get('https://api.github.com/users/angular/repos').success(function (repos) {
        s.repos = repos;
    });
}
// $inject property annotation
DemoController['$inject'] = ['$scope', '$http'];

app.controller('DemoController', DemoController);

PRO TIP 2: Vous pouvez ajouter une directive ng-strict-di sur le même élément que ng-app pour activer le mode DI strict qui générera une erreur chaque fois qu’un service essaie d’utiliser des annotations implicites. Exemple:

<html ng-app="DemoApp" ng-strict-di>

Ou si vous utilisez l’amorçage manuel :

angular.bootstrap(document, ['DemoApp'], {
    strictDi: true
});

[1] : https://docs.angularjs.org/api/ng/directive/ngRepeat [2] : http://stackoverflow.com/a/38686531/3612903 [3] : http://stackoverflow.com/users/569302 [4] : http://stackoverflow.com/questions/18499909/how-to-count-total-number-of-watches-on-a-page [5] : https://docs.angularjs.org/api/ng/directive/ngIf [6] : https://docs.angularjs.org/api/ng/directive/ngShow

Lier une fois #

Angular a la réputation d’avoir une liaison de données bidirectionnelle impressionnante. Par défaut, Angular synchronise en permanence les valeurs liées entre les composants de modèle et de vue à chaque fois que les données changent dans le composant de modèle ou de vue.

Cela a le coût d’être un peu lent s’il est trop utilisé. Cela aura un impact plus important sur les performances :

Mauvaises performances : {{my.data}}

Ajoutez deux deux-points :: avant le nom de la variable pour utiliser une liaison unique. Dans ce cas, la valeur n’est mise à jour qu’une fois my.data défini. Vous pointez explicitement pour ne pas surveiller les modifications de données. Angular n’effectuera aucune vérification de valeur, ce qui réduira le nombre d’expressions évaluées à chaque cycle de résumé.

Exemples de bonnes performances utilisant une liaison unique

{{::my.data}}
<span ng-bind="::my.data"></span>
<span ng-if="::my.data"></span>
<span ng-repeat="item in ::my.data">{{item}}</span>
<span ng-class="::{ 'my-class': my.data }"></div>

Remarque : Cela supprime cependant la liaison de données bidirectionnelle pour my.data, donc chaque fois que ce champ change dans votre application, cela ne sera pas automatiquement reflété dans la vue. Donc utilisez-le uniquement pour les valeurs qui ne changeront pas tout au long de la durée de vie de votre application.

of-if vs of-show #

Ces fonctions ont un comportement très similaire. La différence est que ng-if supprime des éléments du DOM. S’il y a de grandes parties du code qui ne seront pas affichées, alors ng-if est la solution. ng-show ne masquera que les éléments mais conservera tous les gestionnaires.

de si #

La directive ngIf supprime ou recrée une partie de l’arborescence DOM basée sur une expression. Si l’expression affectée à ngIf donne une valeur fausse, l’élément est supprimé du DOM, sinon un clone de l’élément est réinséré dans le DOM.

de spectacles #

La directive ngShow affiche ou masque l’élément HTML donné en fonction de l’expression fournie à l’attribut ngShow. L’élément est affiché ou masqué en supprimant ou en ajoutant la classe CSS ng-hide sur l’élément.

Exemple #

<div ng-repeat="user in userCollection">
  <p ng-if="user.hasTreeLegs">I am special
    <!-- some complicated DOM -->
  </p>
  <p ng-show="user.hasSubscribed">I am aweosme
    <!-- switch this setting on and off -->
  </p>
</div>

Conclusion #

Cela dépend du type d’utilisation, mais souvent l’un est plus approprié que l’autre (par exemple, si 95 % du temps l’élément n’est pas nécessaire, utilisez ng-if ; si vous avez besoin de basculer la visibilité de l’élément DOM, utilisez ng-show).

En cas de doute, utilisez ng-if et testez !

Remarque : ng-if crée une nouvelle portée isolée, contrairement à ng-show et ng-hide. Utilisez $parent.property si la propriété de portée parent n’y est pas directement accessible.

Observateurs #

Les observateurs nécessaires pour surveiller une certaine valeur et détecter que cette valeur est modifiée.

Après avoir appelé $watch() ou $watchCollection, un nouvel observateur est ajouté à la collection d’observateurs internes dans la portée actuelle.

Alors, qu’est-ce que l’observateur ? #

Watcher est une fonction simple, qui est appelée à chaque cycle de résumé et renvoie une valeur. Angular vérifie la valeur renvoyée, si elle n’est pas la même que lors de l’appel précédent - un rappel passé en second paramètre à la fonction $watch() ou $watchCollection sera exécuté.

(function() {
  angular.module("app", []).controller("ctrl", function($scope) {
    $scope.value = 10;
    $scope.$watch(
      function() { return $scope.value; },
      function() { console.log("value changed"); }
    );
  }
})();

Les observateurs sont des tueurs de performance. Plus vous avez d’observateurs, plus ils mettent de temps à faire une boucle de résumé, plus l’interface utilisateur est lente. Si un observateur détecte des changements, il lancera la boucle de résumé (recalcul sur tous les écrans)

Il existe trois façons de surveiller manuellement les changements de variables dans Angular.

$watch() - surveille juste les changements de valeur

$watchCollection() - surveille les changements de collection (regarde plus que $watch normal)

$watch(..., true) - Évitez cela autant que possible, cela effectuera une “surveillance profonde” et tuera les performances (regarde plus que watchCollection)

Notez que si vous liez des variables dans la vue, vous créez de nouveaux observateurs - utilisez {{::variable}} pour ne pas créer d’observateur, en particulier dans les boucles

Par conséquent, vous devez suivre le nombre d’observateurs que vous utilisez. Vous pouvez compter les observateurs avec ce script (crédit à @Words Like Jared - http://stackoverflow.com/questions/18499909/)

(function() {
  var root = angular.element(document.getElementsByTagName("body")),
      watchers = [];

  var f = function(element) {

    angular.forEach(["$scope", "$isolateScope"], function(scopeProperty) {
      if(element.data() && element.data().hasOwnProperty(scopeProperty)) {
        angular.forEach(element.data()[scopeProperty].$$watchers, function(watcher) {
          watchers.push(watcher);
        });
      }
    });

    angular.forEach(element.children(), function(childElement) {
      f(angular.element(childElement));
    });

  };

  f(root);

  // Remove duplicate watchers
  var watchersWithoutDuplicates = [];
  angular.forEach(watchers, function(item) {
    if(watchersWithoutDuplicates.indexOf(item) < 0) {
      watchersWithoutDuplicates.push(item);
    }
  });

  console.log(watchersWithoutDuplicates.length);

})();

Si vous ne souhaitez pas créer votre propre script, il existe un utilitaire open source appelé ng-stats qui utilise un graphique en temps réel intégré à la page pour vous donner un aperçu du nombre de montres gérées par Angular, ainsi que de la fréquence et de la durée des cycles de digestion au fil du temps. L’utilitaire expose une fonction globale nommée “showAngularStats” que vous pouvez appeler pour configurer le fonctionnement du graphique.

showAngularStats({
  "position": "topleft",
  "digestTimeThreshold": 16,
  "autoload": true,
  "logDigest": true,
  "logWatches": true
});

L’exemple de code ci-dessus affiche automatiquement le graphique suivant sur la page (démo interactive).

Toujours annuler l’enregistrement des écouteurs enregistrés sur d’autres portées autres que la portée actuelle #

Vous devez toujours désinscrire les étendues autres que votre étendue actuelle, comme indiqué ci-dessous :

//always deregister these
$rootScope.$on(...);
$scope.$parent.$on(...);

Vous n’avez pas besoin de désinscrire les auditeurs sur la portée actuelle car angular s’en occuperait :

//no need to deregister this
$scope.$on(...);

Les écouteurs $rootScope.$on resteront en mémoire si vous naviguez vers un autre contrôleur. Cela créera une fuite de mémoire si le contrôleur tombe hors de portée.

Ne le faites pas

angular.module('app').controller('badExampleController', badExample);
badExample.$inject = ['$scope', '$rootScope'];

function badExample($scope, $rootScope) {
    $rootScope.$on('post:created', function postCreated(event, data) {});
}

Fais

angular.module('app').controller('goodExampleController', goodExample);
goodExample.$inject = ['$scope', '$rootScope'];

function goodExample($scope, $rootScope) {
    var deregister = $rootScope.$on('post:created', function postCreated(event, data) {});

    $scope.$on('$destroy', function destroyScope() {
        deregister();
    });
}

Fonctions et filtres de portée #

AngularJS a une boucle de résumé et toutes vos fonctions dans une vue et les filtres sont exécutés à chaque fois que le cycle de résumé est exécuté. La boucle de résumé sera exécutée chaque fois que le modèle est mis à jour et cela peut ralentir votre application (le filtre peut être activé plusieurs fois, avant que la page ne soit chargée).

Vous devriez éviter ceci :

<div ng-controller="bigCalulations as calc">
  <p>{{calc.calculateMe()}}</p>
  <p>{{calc.data | heavyFilter}}</p>
</div>

Meilleure approche

<div ng-controller="bigCalulations as calc">
  <p>{{calc.preCalculatedValue}}</p>
  <p>{{calc.data | lightFilter}}</p>
</div>

Où l’échantillon de contrôleur est :

.controller("bigCalulations", function(valueService) {
  // bad, because this is called in every digest loop
  this.calculateMe = function() {
    var t = 0;
    for(i = 0; i < 1000; i++) {
      t = t + i;
    }
    return t;
  }
  //good, because it is executed just once and logic is separated in service to    keep the controller light
  this.preCalulatedValue = valueService.caluclateSumm(); // returns 499500
});

Anti-rebond de votre modèle #

<div ng-controller="ExampleController">
    <form name="userForm">
        Name:
        <input type="text" name="userName"
           ng-model="user.name"
           ng-model-options="{ debounce: 1000 }" />
            <button ng-click="userForm.userName.$rollbackViewValue();   user.name=''">Clear</button><br />
    </form>
    <pre>user.name = </pre>
</div>

Dans l’exemple ci-dessus, nous définissons une valeur anti-rebond de 1000 millisecondes, soit 1 seconde. C’est un délai considérable, mais cela empêchera l’entrée d’écraser à plusieurs reprises ng-model avec de nombreux cycles $digest.

En utilisant l’anti-rebond sur vos champs de saisie et partout où une mise à jour instantanée n’est pas requise, vous pouvez augmenter considérablement les performances de vos applications angulaires. Non seulement vous pouvez retarder le temps, mais vous pouvez également retarder le déclenchement de l’action. Si vous ne souhaitez pas mettre à jour votre modèle ng à chaque frappe, vous pouvez également mettre à jour le flou.
