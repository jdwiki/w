Angular utilise un ** arbre ** de portées pour lier la logique (des contrôleurs, des directives, etc.) à la vue et constitue le principal mécanisme derrière la détection des changements dans AngularJS. Une référence plus détaillée pour les champs d’application peut être trouvée sur [docs.angularjs.org][1]

La racine de l’arborescence est accessible via le service injectable $rootScope. Tous les $scopes enfants héritent des méthodes et des propriétés de leur $scope parent, permettant aux enfants d’accéder aux méthodes sans utiliser les services angulaires.

[1] : https://docs.angularjs.org/guide/scope

Une fonction disponible dans toute l’application #

Attention, cette approche peut être considérée comme une mauvaise conception pour les applications angulaires, car elle oblige les programmeurs à se rappeler à la fois où les fonctions sont placées dans l’arborescence de la portée et à être conscients de l’héritage de la portée. Dans de nombreux cas, il serait préférable d’injecter un service (http://stackoverflow.com/questions/23659315/angular-practice-using-scope-inheritance-vs-injection).

Cet exemple montre uniquement comment l’héritage de portée peut être utilisé pour nos besoins et comment vous pouvez en tirer parti, et non les meilleures pratiques de conception d’une application entière.

Dans certains cas, nous pourrions tirer parti de l’héritage de portée et définir une fonction en tant que propriété de rootScope. De cette façon, toutes les étendues de l’application (à l’exception des étendues isolées) hériteront de cette fonction, et elle pourra être appelée de n’importe où dans l’application.

angular.module('app', [])
.run(['$rootScope', function($rootScope){
    var messages = []
    $rootScope.addMessage = function(msg){
        messages.push(msg);
    }
}]);

<div ng-app="app">
    <a ng-click="addMessage('hello world!')">it could be accsessed from here</a>
    <div ng-include="inner.html"></div>
</div>

intérieur.html :

<div>
    <button ng-click="addMessage('page')">and from here to!</button>
</div>

Éviter d’hériter des valeurs primitives #

En javascript, l’attribution d’une valeur non [primitive][1] (telle qu’un objet, un tableau, une fonction et [many][2] plus), conserve une référence (une adresse dans la mémoire) à la valeur attribuée.

Affecter une valeur primitive (Chaîne, Nombre, Booléen ou Symbole) à deux variables différentes, et en changer une, ne changera pas les deux :

var x = 5;
var y = x;
y = 6;
console.log(y === x, x, y); //false, 5, 6

Mais avec une valeur non primitive, puisque les deux variables conservent simplement des références au même objet, la modification d’une variable changera l’autre :

var x = { name : 'John Doe' };
var y = x;
y.name = 'Jhon';
console.log(x.name === y.name, x.name, y.name); //true, John, John

En angulaire, lorsqu’une portée est créée, toutes les propriétés de son parent lui sont affectées. Cependant, la modification ultérieure des propriétés n’affectera la portée parent que si c’est une valeur non primitive :

angular.module('app', [])
.controller('myController', ['$scope', function($scope){
    $scope.person = { name: 'John Doe' }; //non-primitive
    $scope.name = 'Jhon Doe'; //primitive
}])
.controller('myController1', ['$scope', function($scope){}]);

<div ng-app="app" ng-controller="myController">
    binding to input works: {{person.name}}<br/>
    binding to input does not work: {{name}}<br/>
    <div ng-controller="myController1">
        <input ng-model="person.name" />
        <input ng-model="name" />
    </div>
</div>

N’oubliez pas : dans Angular, les portées peuvent être créées de plusieurs façons (telles que des directives intégrées ou personnalisées, ou la fonction $scope.$new()), et il est probablement impossible de garder une trace de l’arborescence des portées.

Utiliser uniquement des valeurs non primitives comme propriétés de portée vous gardera du bon côté (sauf si vous avez besoin d’une propriété pour ne pas hériter, ou d’autres cas où vous êtes conscient de l’héritage de portée).

[1] : https://developer.mozilla.org/en-US/docs/Glossary/Primitive [2] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference

Exemple basique d’héritage $scope #

angular.module('app', [])
.controller('myController', ['$scope', function($scope){
    $scope.person = { name: 'John Doe' };
}]);

<div ng-app="app" ng-conroller="myController">
    <input ng-model="person.name" />
    <div ng-repeat="number in [0,1,2,3]">
        {{person.name}} {{number}}
    </div>
</div>

Dans cet exemple, la directive ng-repeat crée une nouvelle portée pour chacun de ses enfants nouvellement créés.

Ces étendues créées sont des enfants de leur étendue parente (dans ce cas, l’étendue créée par myController), et par conséquent, elles héritent de toutes ses propriétés, telles que person.

Comment pouvez-vous limiter la portée d’une directive et pourquoi feriez-vous cela ? #

La portée est utilisée comme “colle” que nous utilisons pour communiquer entre le contrôleur parent, la directive et le modèle de directive. Chaque fois que l’application AngularJS est amorcée, un objet rootScope est créé. Chaque portée créée par les contrôleurs, les directives et les services est héritée de manière prototypique de rootScope.

Oui, nous pouvons limiter le champ d’application d’une directive . Nous pouvons le faire en créant un champ d’application isolé pour la directive.

Il existe 3 types de périmètres de directives :

  1. Champ d’application : faux (la directive utilise son champ d’application principal)
  2. Champ d’application : Vrai (la directive obtient un nouveau champ d’application)
  3. Champ d’application : { } (la directive obtient un nouveau champ d’application isolé)

Directives avec la nouvelle étendue isolée : Lorsque nous créons une nouvelle étendue isolée, elle ne sera pas héritée de l’étendue parente. Cette nouvelle portée est appelée Portée isolée car elle est complètement détachée de sa portée parente. Pourquoi? devrions-nous utiliser une portée isolée : nous devons utiliser une portée isolée lorsque nous voulons créer une directive personnalisée, car cela garantira que notre directive est générique et placée n’importe où dans l’application. La portée parent ne va pas interférer avec la portée de la directive.

Exemple de périmètre isolé :

var app = angular.module("test",[]);

app.controller("Ctrl1",function($scope){
    $scope.name = "Prateek";
    $scope.reverseName = function(){
        $scope.name = $scope.name.split('').reverse().join('');
    };
});
app.directive("myDirective", function(){
    return {
        restrict: "EA",
        scope: {},
        template: "<div>Your name is : {{name}}</div>"+
        "Change your name : <input type='text' ng-model='name'/>"
    };
});

Il existe 3 types de préfixes fournis par AngularJS pour une portée isolée :

  1. “@” (liaison de texte/liaison unidirectionnelle)
  2. “=” (liaison directe du modèle/liaison bidirectionnelle)
  3. “&” ( Liaison de comportement / Liaison de méthode )

Tous ces préfixes reçoivent des données des attributs de l’élément directive comme :

<div my-directive 
  class="directive"
  name="{{name}}" 
  reverse="reverseName()" 
  color="color" >
</div>

Création d’événements $scope personnalisés #

Comme les éléments HTML normaux, il est possible que les $scopes aient leurs propres événements. Les événements $scope peuvent être souscrits de la manière suivante :

 $scope.$on('my-event', function(event, args) {
    console.log(args); // { custom: 'data' }
});

Si vous avez besoin de désinscrire un écouteur d’événement, la fonction $on renverra une fonction non contraignante. Pour continuer avec l’exemple ci-dessus :

var unregisterMyEvent = $scope.$on('my-event', function(event, args) {
    console.log(args); // { custom: 'data' }
    unregisterMyEvent();
});

Il existe deux façons de déclencher votre propre événement $scope personnalisé $broadcast et $emit. Pour informer le(s) parent(s) de la portée d’un événement spécifique, utilisez $emit

$scope.$emit('my-event', { custom: 'data' });

L’exemple ci-dessus déclenchera tous les écouteurs d’événement pour my-event sur la portée parente et continuera dans l’arborescence de portée jusqu’à $rootScope à moins qu’un écouteur n’appelle stopPropagation sur l’événement. Seuls les événements déclenchés avec $emit peuvent appeler stopPropagation

L’inverse de $emit est $broadcast, qui déclenchera tous les écouteurs d’événement sur toutes les étendues enfants de l’arborescence des étendues qui sont des enfants de l’étendue qui a appelé $broadcast.

$scope.$broadcast('my-event', { custom: 'data' });

Les événements déclenchés avec $broadcast ne peuvent pas être annulés.

Utilisation des fonctions $scope #

Bien que la déclaration d’une fonction dans le $rootscope ait ses avantages, nous pouvons également déclarer une fonction $scope dans n’importe quelle partie du code injectée par le service $scope. Contrôleur, par exemple.

Manette

myApp.controller('myController', ['$scope', function($scope){
    $scope.myFunction = function () {
        alert("You are in myFunction!");
    };
}]);

Vous pouvez maintenant appeler votre fonction depuis le contrôleur en utilisant :

$scope.myfunction();

Ou via HTML qui se trouve sous ce contrôleur spécifique :

<div ng-controller="myController">
    <button ng-click="myFunction()"> Click me! </button>
</div>

Directif

Une [directive angulaire][1] est un autre endroit où vous pouvez utiliser votre champ d’application :

myApp.directive('triggerFunction', function() {
    return {
        scope: {
            triggerFunction: '&'
        },
        link: function(scope, element) {
            element.bind('mouseover', function() {
                scope.triggerFunction();
            });
        }
    };
});

Et dans votre code HTML sous le même contrôleur :

<div ng-controller="myController">
    <button trigger-function="myFunction()"> Hover over me! </button>
</div>

Bien sûr, vous pouvez utiliser ngMouseover pour la même chose, mais la particularité des directives est que vous pouvez les personnaliser comme vous le souhaitez. Et maintenant que vous savez comment utiliser vos fonctions $scope à l’intérieur, soyez créatif !

[1] : https://www.wikiod.com/fr/angularjs/directives-personnalisees#Modèle d’objet de définition de directive
