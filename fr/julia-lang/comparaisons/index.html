Syntaxe #

  • x < y # si x est strictement inférieur à y
  • x > y # si x est strictement supérieur à y
  • x == y # si x est égal à y
  • x === y # alternativement x ≡ y, si x est égal à y
  • x ≤ y # alternativement x <= y, si x est inférieur ou égal à y
  • x ≥ y # alternativement x >= y, si x est supérieur ou égal à y
  • x ≠ y # alternativement x != y, si x n’est pas égal à y
  • x ≈ y # si x est approximativement égal à y

Faites attention à ne pas retourner les signes de comparaison. Julia définit de nombreuses fonctions de comparaison par défaut sans définir la version retournée correspondante. Par exemple, on peut exécuter

julia> Set(1:3) ⊆ Set(0:5)
true

mais ça ne marche pas à faire

julia> Set(0:5) ⊇ Set(1:3)
ERROR: UndefVarError: ⊇ not defined

Tous n’ont pas de définition dans la bibliothèque standard “Base”. Cependant, ils sont disponibles pour d’autres packages afin de les définir et de les utiliser selon les besoins.

Dans l’usage courant, la plupart de ces opérateurs de comparaison ne sont pas pertinents. Les plus couramment utilisées sont les fonctions mathématiques standard pour la commande ; voir la section Syntaxe pour une liste.

Comme la plupart des autres opérateurs de Julia, les opérateurs de comparaison sont des [fonctions][1] et peuvent être appelés en tant que fonctions. Par exemple, (<)(1, 2) a la même signification que 1 < 2.

Comparaisons enchaînées #

Plusieurs opérateurs de comparaison utilisés ensemble sont chaînés, comme s’ils étaient connectés via l’opérateur [&&][1]. Cela peut être utile pour des chaînes de comparaison lisibles et mathématiquement concises, telles que

# same as 0 < i && i <= length(A)
isinbounds(A, i)       = 0 < i ≤ length(A)

# same as Set() != x && issubset(x, y)
isnonemptysubset(x, y) = Set() ≠ x ⊆ y

Cependant, il existe une différence importante entre a > b > c et a > b && b > c ; dans ce dernier, le terme “b” est évalué deux fois. Cela n’a pas beaucoup d’importance pour les symboles anciens, mais pourrait avoir de l’importance si les termes eux-mêmes ont des effets secondaires. Par exemple,

julia> f(x) = (println(x); 2)
f (generic function with 1 method)

julia> 3 > f("test") > 1
test
true

julia> 3 > f("test") && f("test") > 1
test
test
true

Examinons plus en détail les comparaisons enchaînées et leur fonctionnement, en voyant comment elles sont analysées et réduites en [expressions][2]. Tout d’abord, considérez la comparaison simple, que nous pouvons voir n’est qu’un simple appel de fonction :

julia> dump(:(a > b))
Expr
  head: Symbol call
  args: Array{Any}((3,))
    1: Symbol >
    2: Symbol a
    3: Symbol b
  typ: Any

Maintenant si on enchaîne la comparaison, on remarque que le parsing a changé :

julia> dump(:(a > b >= c))
Expr
  head: Symbol comparison
  args: Array{Any}((5,))
    1: Symbol a
    2: Symbol >
    3: Symbol b
    4: Symbol >=
    5: Symbol c
  typ: Any

Après l’analyse, l’expression est alors abaissée à sa forme finale :

julia> expand(:(a > b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

et on remarque bien que c’est la même chose que pour a > b && b >= c :

julia> expand(:(a > b && b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

[1] : https://www.wikiod.com/fr/julia-lang/conditionnels#Opérateurs de court-circuit : && et || [2] : https://www.wikiod.com/fr/julia-lang/expressions

Nombres ordinaux #

Nous verrons comment implémenter des comparaisons personnalisées en implémentant un type personnalisé, nombres ordinaux. Pour simplifier l’implémentation, nous nous concentrerons sur un petit sous-ensemble de ces nombres : tous les nombres ordinaux jusqu’à ε₀ non compris. Notre implémentation est axée sur la simplicité, pas sur la rapidité ; cependant, la mise en œuvre n’est pas lente non plus.

Nous stockons les nombres ordinaux par leur forme normale de Cantor. Parce que l’arithmétique ordinale n’est pas commutative, nous prendrons la convention commune de stocker les termes les plus significatifs en premier.

immutable OrdinalNumber <: Number
    βs::Vector{OrdinalNumber}
    cs::Vector{Int}
end

Puisque la forme normale de Cantor est unique, nous pouvons tester l’égalité simplement par l’égalité récursive :

Dans la version v0.5, il existe une très belle syntaxe pour faire cela de manière compacte :

import Base: ==
α::OrdinalNumber == β::OrdinalNumber = α.βs == β.βs && α.cs == β.cs

Sinon, définissez la fonction comme c’est plus typique :

import Base: ==
==(α::OrdinalNumber, β::OrdinalNumber) = α.βs == β.βs && α.cs == β.cs

Pour terminer notre commande, car ce type a une commande totale, nous devrions surcharger la fonction isless :

import Base: isless
function isless(α::OrdinalNumber, β::OrdinalNumber)
    for i in 1:min(length(α.cs), length(β.cs))
        if α.βs[i] < β.βs[i]
            return true
        elseif α.βs[i] == β.βs[i] && α.cs[i] < β.cs[i]
            return true
        end
    end
    return length(α.cs) < length(β.cs)
end

Pour tester notre commande, nous pouvons créer des méthodes pour créer des nombres ordinaux. Zéro, bien sûr, est obtenu en n’ayant aucun terme dans la forme normale de Cantor :

const ORDINAL_ZERO = OrdinalNumber([], [])
Base.zero(::Type{OrdinalNumber}) = ORDINAL_ZERO

Nous pouvons définir un expω pour calculer ω^α, et l’utiliser pour calculer 1 et ω :

expω(α) = OrdinalNumber([α], [1])
const ORDINAL_ONE = expω(ORDINAL_ZERO)
Base.one(::Type{OrdinalNumber}) = ORDINAL_ONE
const ω = expω(ORDINAL_ONE)

Nous avons maintenant une fonction de classement entièrement fonctionnelle sur les nombres ordinaux :

julia> ORDINAL_ZERO < ORDINAL_ONE < ω < expω(ω)
true

julia> ORDINAL_ONE > ORDINAL_ZERO
true

julia> sort([ORDINAL_ONE, ω, expω(ω), ORDINAL_ZERO])

4-element Array{OrdinalNumber,1}:
                                                                                                       OrdinalNumber(OrdinalNumber[],Int64[])
                                                                     OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])
                                   OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])
 OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])],[1])

Dans le dernier exemple, on voit que l’impression des nombres ordinaux pourrait être meilleure, mais le résultat est celui attendu.

Utilisation de ==, === et isequal #

Il existe trois opérateurs d’égalité : ==, === et isequal. (Le dernier n’est pas vraiment un opérateur, mais c’est une fonction et tous les opérateurs sont des fonctions.)

Quand utiliser == #

== est value égalité. Il renvoie “true” lorsque deux objets représentent, dans leur état actuel, la même valeur.

Par exemple, il est évident que

julia> 1 == 1
true

mais en plus

julia> 1 == 1.0
true

julia> 1 == 1.0 + 0.0im
true

julia> 1 == 1//1
true

Les membres droits de chaque égalité ci-dessus sont d’un [type][1] différent, mais ils représentent toujours la même valeur.

Pour les objets mutables, comme [arrays][2], == compare leur valeur actuelle.

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> C = [1, 3, 2]
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
true

julia> A == C
false

julia> A[2], A[3] = A[3], A[2]  # swap 2nd and 3rd elements of A
(3,2)

julia> A
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
false

julia> A == C
true

La plupart du temps, == est le bon choix.

Quand utiliser === #

=== est une opération beaucoup plus stricte que ==. Au lieu d’évaluer l’égalité, il mesure l’égalité. Deux objets sont égaux s’ils ne peuvent pas être distingués l’un de l’autre par le programme lui-même. Ainsi nous avons

julia> 1 === 1
true

car il n’y a aucun moyen de distinguer un ‘1’ d’un autre ‘1’. Mais

julia> 1 === 1.0
false

car bien que ‘1’ et ‘1.0’ soient la même valeur, ils sont de types différents, et ainsi le programme peut les différencier.

Par ailleurs,

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> A === B
false

julia> A === A
true

ce qui peut surprendre au premier abord ! Comment le programme pourrait-il faire la distinction entre les deux vecteurs ‘A’ et ‘B’ ? Parce que les vecteurs sont modifiables, il pourrait modifier A, puis il se comporterait différemment de B. Mais peu importe comment il modifie A, A se comportera toujours de la même manière que A lui-même. Donc ‘A’ est égal à ‘A’, mais pas égal à ‘B’.

En continuant dans cette veine, observez

julia> C = A
3-element Array{Int64,1}:
 1
 2
 3

julia> A === C
true

En attribuant A à C, nous disons que C a aliasé A. C’est-à-dire qu’il est devenu juste un autre nom pour “A”. Toute modification apportée à ‘A’ sera également observée par ‘C’. Par conséquent, il n’y a aucun moyen de faire la différence entre ‘A’ et ‘C’, ils sont donc égaux.

Quand utiliser iségal #

La différence entre == et isequal est très subtile. La plus grande différence réside dans la manière dont les nombres à virgule flottante sont gérés :

julia> NaN == NaN
false

Ce résultat peut-être surprenant est [défini] [3] par la norme IEEE pour les types à virgule flottante (IEEE-754). Mais cela n’est pas utile dans certains cas, comme le tri. isequal est fourni pour les cas :

julia> isequal(NaN, NaN)
true

De l’autre côté du spectre, == traite le zéro négatif IEEE et le zéro positif comme la même valeur (également comme spécifié par IEEE-754). Ces valeurs ont cependant des représentations distinctes en mémoire.

julia> 0.0
0.0

julia> -0.0
-0.0

julia> 0.0 == -0.0
true

Toujours à des fins de tri, “isequal” les distingue.

julia> isequal(0.0, -0.0)
false
