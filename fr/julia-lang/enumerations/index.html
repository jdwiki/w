Syntaxe #

  • @enum EnumType val=1 val val
  • :symbole

Il est parfois utile d’avoir des types énumérés où chaque instance est d’un type différent (souvent un [type singleton immuable][1]) ; cela peut être important pour la stabilité du type. Les traits sont généralement mis en œuvre avec ce paradigme. Cependant, cela entraîne une surcharge supplémentaire au moment de la compilation.

[1] : https://www.wikiod.com/fr/julia-lang/les-types#Types immuables

Définir un type énuméré #

Un type énuméré est un [type][1] qui peut contenir l’une d’une liste finie de valeurs possibles. Dans Julia, les types énumérés sont généralement appelés “types enum”. Par exemple, on pourrait utiliser des types enum pour décrire les sept jours de la semaine, les douze mois de l’année, les quatre couleurs d’un [jeu standard de 52 cartes](https://en.wikipedia.org/wiki/Standard_52 -card_deck), ou d’autres situations similaires.

Nous pouvons définir des types énumérés pour modéliser les couleurs et les rangs d’un jeu standard de 52 cartes. La macro @enum est utilisée pour définir les types d’énumération.

@enum Suit ♣ ♦ ♥ ♠
@enum Rank ace=1 two three four five six seven eight nine ten jack queen king

Ceci définit deux types : Suit et Rank. On peut vérifier que les valeurs sont bien des types attendus :

julia> ♦
♦::Suit = 1

julia> six
six::Rank = 6

Notez que chaque costume et rang a été associé à un numéro. Par défaut, ce nombre commence à zéro. Ainsi, la deuxième couleur, les diamants, a reçu le numéro 1. Dans le cas du “rang”, il peut être plus logique de commencer le numéro à un. Ceci a été réalisé en annotant la définition de ace avec une annotation =1.

Les types énumérés sont livrés avec de nombreuses fonctionnalités, telles que l’égalité (et même l’identité) et les comparaisons intégrées :

julia> seven === seven
true

julia> ten ≠ jack
true

julia> two < three
true

Comme les valeurs de tout autre [type immuable][2], les valeurs des types énumérés peuvent également être hachées et stockées dans Dicts.

Nous pouvons compléter cet exemple en définissant un type Card qui a un champ Rank et un champ Suit :

immutable Card
    rank::Rank
    suit::Suit
end

et donc nous pouvons créer des cartes avec

julia> Card(three, ♣)
Card(three::Rank = 3,♣::Suit = 0)

Mais les types énumérés viennent également avec leurs propres méthodes de “conversion”, donc nous pouvons en effet simplement faire

julia> Card(7, ♠)
Card(seven::Rank = 7,♠::Suit = 3)

et puisque ‘7’ peut être directement converti en ‘Rank’, ce constructeur est prêt à l’emploi.

On pourrait souhaiter définir du sucre syntaxique pour construire ces cartes ; la multiplication implicite fournit un moyen pratique de le faire. Définir

julia> import Base.*

julia> r::Int * s::Suit = Card(r, s)
* (generic function with 156 methods)

et alors

julia> 10♣
Card(ten::Rank = 10,♣::Suit = 0)

julia> 5♠
Card(five::Rank = 5,♠::Suit = 3)

profitant à nouveau des fonctions de conversion intégrées.

[1] : https://www.wikiod.com/fr/julia-lang/les-types [2] : https://www.wikiod.com/fr/julia-lang/les-types#Types immuables

Utilisation de symboles comme énumérations légères #

Bien que la macro @enum soit assez utile pour la plupart des cas d’utilisation, elle peut être excessive dans certains cas d’utilisation. Les inconvénients de @enum incluent :

  • Il crée un nouveau type
  • Il est un peu plus difficile à étendre
  • Il est livré avec des fonctionnalités telles que la conversion, l’énumération et la comparaison, qui peuvent être superflues dans certaines applications

Dans les cas où une alternative plus légère est souhaitée, le type “Symbole” peut être utilisé. Les symboles sont [chaînes internes] [1] ; ils représentent des séquences de caractères, un peu comme le font [strings][2], mais ils sont uniquement associés à des nombres. Cette association unique permet une comparaison rapide de l’égalité des symboles.

Nous pouvons à nouveau implémenter un type Card, cette fois en utilisant les champs Symbol :

const ranks = Set([:ace, :two, :three, :four, :five, :six, :seven, :eight, :nine,
                   :ten, :jack, :queen, :king])
const suits = Set([:♣, :♦, :♥, :♠])
immutable Card
    rank::Symbol
    suit::Symbol
    function Card(r::Symbol, s::Symbol)
        r in ranks || throw(ArgumentError("invalid rank: $r"))
        s in suits || throw(ArgumentError("invalid suit: $s"))
        new(r, s)
    end
end

Nous implémentons le constructeur interne pour vérifier les valeurs incorrectes transmises au constructeur. Contrairement à l’exemple utilisant les types @enum, Symbol peut contenir n’importe quelle chaîne, et nous devons donc faire attention aux types de Symbol que nous acceptons. Notez ici l’utilisation des opérateurs conditionnels [court-circuit][3].

Nous pouvons maintenant construire des objets Card comme nous l’attendons :

julia> Card(:ace, :♦)
Card(:ace,:♦)

julia> Card(:nine, :♠)
Card(:nine,:♠)

julia> Card(:eleven, :♠)
ERROR: ArgumentError: invalid rank: eleven
 in Card(::Symbol, ::Symbol) at ./REPL[17]:5

julia> Card(:king, :X)
ERROR: ArgumentError: invalid suit: X
 in Card(::Symbol, ::Symbol) at ./REPL[17]:6

Un avantage majeur de Symbols est leur extensibilité d’exécution. Si à l’exécution, on souhaite accepter (par exemple) :eleven comme nouveau rang, il suffit de lancer simplement push!(ranks, :eleven). Une telle extensibilité d’exécution n’est pas possible avec les types @enum.

[1] : https://en.wikipedia.org/wiki/String_interning [2] : https://www.wikiod.com/fr/julia-lang/cordes [3] : https://www.wikiod.com/fr/julia-lang/conditionnels#Opérateurs de court-circuit : && et ||
