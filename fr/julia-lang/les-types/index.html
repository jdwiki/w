Syntaxe #

  • MonType immuable ; champ; champ; fin
  • tapez MonType ; champ; champ; fin

Les types sont la clé de la performance de Julia. Une idée importante pour les performances est la [stabilité de type][1], qui se produit lorsque le type renvoyé par une fonction dépend uniquement des types, et non des valeurs, de ses arguments.

[1] : https://www.wikiod.com/fr/julia-lang/stabilite-des-types

Envoi sur les types #

Sur Julia, vous pouvez définir plusieurs méthodes pour chaque fonction. Supposons que nous définissions trois méthodes de la même fonction :

foo(x) = 1
foo(x::Number) = 2
foo(x::Int) = 3

Lors du choix de la méthode à utiliser (appelée [dispatch][1]), Julia choisit la méthode la plus spécifique qui correspond aux types des arguments :

julia> foo('one')
1

julia> foo(1.0)
2

julia> foo(1)
3

Cela facilite le polymorphisme. Par exemple, nous pouvons facilement créer une liste liée en définissant deux types immuables, nommés “Nil” et “Cons”. Ces noms sont traditionnellement utilisés pour décrire respectivement une liste vide et une liste non vide.

abstract LinkedList
immutable Nil <: LinkedList end
immutable Cons <: LinkedList
    first
    rest::LinkedList
end

Nous représenterons la liste vide par Nil() et toutes les autres listes par Cons(first, rest), où first est le premier élément de la liste chaînée et rest est la liste chaînée composée de tous les éléments restants éléments. Par exemple, la liste [1, 2, 3] sera représentée par

julia> Cons(1, Cons(2, Cons(3, Nil())))
Cons(1,Cons(2,Cons(3,Nil())))

La liste est-elle vide ? #

Supposons que nous voulions étendre la fonction isempty de la bibliothèque standard, qui fonctionne sur une variété de collections différentes :

julia> methods(isempty)
# 29 methods for generic function "isempty":
isempty(v::SimpleVector) at essentials.jl:180
isempty(m::Base.MethodList) at reflection.jl:394
...

Nous pouvons simplement utiliser la syntaxe de répartition de la fonction et définir deux méthodes supplémentaires de isempty. Comme cette fonction provient du module Base, nous devons la qualifier de Base.isempty afin de l’étendre.

Base.isempty(::Nil) = true
Base.isempty(::Cons) = false

Ici, nous n’avons pas du tout besoin des valeurs d’argument pour déterminer si la liste est vide. Le type suffit à lui seul pour calculer cette information. Julia nous permet d’omettre les noms des arguments, en ne gardant que leur annotation de type, si nous n’avons pas besoin d’utiliser leurs valeurs.

Nous pouvons [test][2] que nos méthodes “isempty” fonctionnent :

julia> using Base.Test

julia> @test isempty(Nil())
Test Passed
  Expression: isempty(Nil())

julia> @test !isempty(Cons(1, Cons(2, Cons(3, Nil()))))
Test Passed
  Expression: !(isempty(Cons(1,Cons(2,Cons(3,Nil())))))

et en effet le nombre de méthodes pour isempty a augmenté de 2 :

julia> methods(isempty)
# 31 methods for generic function "isempty":
isempty(v::SimpleVector) at essentials.jl:180
isempty(m::Base.MethodList) at reflection.jl:394

De toute évidence, déterminer si une liste chaînée est vide ou non est un exemple trivial. Mais cela mène à quelque chose de plus intéressant :

Combien de temps dure la liste ? #

La fonction length de la bibliothèque standard nous donne la longueur d’une collection ou de certains [iterables][3]. Il existe de nombreuses façons d’implémenter length pour une liste chaînée. En particulier, l’utilisation d’une boucle “while” est probablement la plus rapide et la plus économe en mémoire dans Julia. Mais l’optimisation prématurée est à éviter, alors supposons une seconde que notre liste chaînée n’a pas besoin d’être efficace. Quelle est la manière la plus simple d’écrire une fonction “longueur” ?

Base.length(::Nil) = 0
Base.length(xs::Cons) = 1 + length(xs.rest)

La première définition est simple : une liste vide a une longueur “0”. La deuxième définition est également facile à lire : pour compter la longueur d’une liste, on compte le premier élément, puis on compte la longueur du reste de la liste. Nous pouvons tester cette méthode de la même manière que nous avons testé isempty :

julia> @test length(Nil()) == 0
Test Passed
  Expression: length(Nil()) == 0
   Evaluated: 0 == 0

julia> @test length(Cons(1, Cons(2, Cons(3, Nil())))) == 3
Test Passed
  Expression: length(Cons(1,Cons(2,Cons(3,Nil())))) == 3
   Evaluated: 3 == 3

Prochaines étapes #

Cet exemple de jouet est assez loin d’implémenter toutes les fonctionnalités qui seraient souhaitées dans une liste chaînée. Il manque, par exemple, l’interface d’itération. Cependant, il illustre comment dispatch peut être utilisé pour écrire un code court et clair.

Types immuables #

Le type composite le plus simple est un type immuable. Les instances de types immuables, comme [tuples][1], sont des valeurs. Leurs champs ne peuvent pas être modifiés après leur création. À bien des égards, un type immuable est comme un Tuple avec des noms pour le type lui-même et pour chaque champ.

Types de singletons #

Les types composites, par définition, contiennent un certain nombre de types plus simples. Dans Julia, ce nombre peut être zéro ; c’est-à-dire qu’un type immuable est autorisé à contenir aucun champs. Ceci est comparable au tuple vide ().

Pourquoi cela pourrait-il être utile ? Ces types immuables sont connus sous le nom de “types singleton”, car une seule instance d’entre eux pourrait exister. Les valeurs de ces types sont appelées “valeurs singleton”. La bibliothèque standard “Base” contient de nombreux types de singletons de ce type. Voici une brève liste :

  • Void, le type de rien. Nous pouvons vérifier que Void.instance (qui est une syntaxe spéciale pour récupérer la valeur singleton d’un type singleton) est bien nothing.
  • Tout type de média, tel que MIME"text/plain", est un type singleton avec une seule instance, MIME("text/plain").
  • Les types Irrational{:π}, Irrational{:e}, Irrational{:φ} et similaires sont des types singleton, et leurs instances singleton sont les valeurs irrationnelles π = 3.1415926535897..., etc.
  • Les traits de taille de l’itérateur Base.HasLength, Base.HasShape, Base.IsInfinite et Base.SizeUnknown sont tous des types singleton.
  • Dans la version 0.5 et ultérieure, chaque [fonction][2] est une instance singleton d’un type singleton ! Comme toute autre valeur singleton, nous pouvons récupérer la fonction sin, par exemple, à partir de typeof(sin).instance.

Parce qu’ils ne contiennent rien, les types singleton sont incroyablement légers, et ils peuvent souvent être optimisés par le compilateur pour n’avoir aucune surcharge d’exécution. Ainsi, ils sont parfaits pour les traits, les valeurs de balises spéciales et pour des choses comme les fonctions sur lesquelles on aimerait se spécialiser.

Pour définir un type singleton,

julia> immutable MySingleton end

Pour définir une impression personnalisée pour le type singleton,

julia> Base.show(io::IO, ::MySingleton) = print(io, "sing")

Pour accéder à l’instance singleton,

julia> MySingleton.instance
MySingleton()

Souvent, on assigne ceci à une constante :

julia> const sing = MySingleton.instance
MySingleton()

Types d’emballage #

Si les types immuables à champ zéro sont intéressants et utiles, alors peut-être que les types immuables à champ unique sont encore plus utiles. Ces types sont communément appelés “types wrapper” car ils enveloppent certaines données sous-jacentes, fournissant une interface alternative à ces données. Un exemple de type wrapper dans Base est [String][3]. Nous allons définir un type similaire à String, nommé MyString. Ce type sera soutenu par un vecteur (unidimensionnel [tableau][4]) d’octets (UInt8).

Tout d’abord, la définition de type elle-même et quelques affichages personnalisés :

immutable MyString <: AbstractString
    data::Vector{UInt8}
end

function Base.show(io::IO, s::MyString)
    print(io, "MyString: ")
    write(io, s.data)
    return
end

Maintenant, notre type MyString est prêt à être utilisé ! Nous pouvons lui fournir des données UTF-8 brutes, et il s’affiche comme nous l’aimons :

julia> MyString([0x48,0x65,0x6c,0x6c,0x6f,0x2c,0x20,0x57,0x6f,0x72,0x6c,0x64,0x21])
MyString: Hello, World!

De toute évidence, ce type de chaîne nécessite beaucoup de travail avant de devenir aussi utilisable que le type Base.String.

Véritables types composites #

Peut-être le plus souvent, de nombreux types immuables contiennent plus d’un champ. Un exemple est le type Rational{T} de la bibliothèque standard, qui contient deux champs : un champ num pour le numérateur et un champ den pour le dénominateur. Il est assez simple d’émuler cette conception de type :

immutable MyRational{T}
    num::T
    den::T
    MyRational(n, d) = (g = gcd(n, d); new(n÷g, d÷g))
end
MyRational{T}(n::T, d::T) = MyRational{T}(n, d)

Nous avons implémenté avec succès un constructeur qui simplifie nos nombres rationnels :

julia> MyRational(10, 6)
MyRational{Int64}(5,3)
