Bien que les combinateurs aient une utilisation pratique limitée, ils sont un outil utile en éducation pour comprendre comment la programmation est fondamentalement liée à la logique et comment des blocs de construction très simples peuvent se combiner pour créer un comportement très complexe. Dans le contexte de Julia, apprendre à créer et à utiliser des combinateurs renforcera la compréhension de la façon de programmer dans un style fonctionnel dans Julia.

Le combinateur Y ou Z #

Bien que Julia ne soit pas un langage purement fonctionnel, il prend pleinement en charge de nombreuses pierres angulaires de la programmation fonctionnelle : [fonctions] de première classe (https://www.wikiod.com/fr/julia-lang/les-fonctions ), portée lexicale et closions.

Le combinateur à virgule fixe est un combinateur clé dans la programmation fonctionnelle. Étant donné que Julia a une sémantique eager evaluation (comme de nombreux langages fonctionnels, y compris Scheme, dont Julia s’inspire fortement), le combinateur Y original de Curry ne fonctionnera pas de la boite :

Y(f) = (x -> f(x(x)))(x -> f(x(x)))

Cependant, un proche parent du Y-combinator, le Z-combinator, fonctionnera bel et bien :

Z(f) = x -> f(Z(f), x)

Ce combinateur prend une fonction et renvoie une fonction qui, lorsqu’elle est appelée avec l’argument x, se transmet elle-même et x. Pourquoi serait-il utile qu’une fonction soit passée elle-même ? Cela permet la récursivité sans réellement référencer le nom de la fonction !

fact(f, x) = x == 0 ? 1 : x * f(x)

Par conséquent, Z(fact) devient une implémentation récursive de la fonction factorielle, bien qu’aucune récursivité ne soit visible dans cette définition de fonction. (La récursivité est évidente dans la définition du combinateur ‘Z’, bien sûr, mais c’est inévitable dans un langage avide.) Nous pouvons vérifier que notre fonction fonctionne bien :

julia> Z(fact)(10)
3628800

Non seulement cela, mais c’est aussi rapide que ce que nous pouvons attendre d’une implémentation récursive. Le code LLVM démontre que le résultat est compilé dans une ancienne branche, soustraire, appeler et multiplier :

julia> @code_llvm Z(fact)(10)

define i64 @"julia_#1_70252"(i64) #0 {
top:
  %1 = icmp eq i64 %0, 0
  br i1 %1, label %L11, label %L8

L8:                                               ; preds = %top
  %2 = add i64 %0, -1
  %3 = call i64 @"julia_#1_70060"(i64 %2) #0
  %4 = mul i64 %3, %0
  br label %L11

L11:                                              ; preds = %top, %L8
  %"#temp#.0" = phi i64 [ %4, %L8 ], [ 1, %top ]
  ret i64 %"#temp#.0"
}

Le système SKI Combinator #

Le système de combinateur SKI est suffisant pour représenter n’importe quel terme de calcul lambda. (En pratique, bien sûr, les abstractions lambda explosent à une taille exponentielle lorsqu’elles sont traduites en SKI.) En raison de la simplicité du système, la mise en œuvre des combinateurs S, K et I est extraordinairement simple :

Une traduction directe de Lambda Calculus #

const S = f -> g -> z -> f(z)(g(z))
const K = x -> y -> x
const I = x -> x

Nous pouvons confirmer, en utilisant le système de tests unitaires, que chaque combinateur a le comportement attendu.

Le combinateur I est le plus simple à vérifier ; il doit retourner la valeur donnée inchangée :

using Base.Test
@test I(1) === 1
@test I(I) === I
@test I(S) === S

Le combinateur K est également assez simple : il devrait supprimer son deuxième argument.

@test K(1)(2) === 1
@test K(S)(I) === S

Le combinateur S est le plus complexe ; son comportement peut être résumé comme l’application des deux premiers arguments au troisième argument, l’application du premier résultat au second. Nous pouvons plus facilement tester le combinateur S en testant certaines de ses formes curry. S(K), par exemple, devrait simplement renvoyer son deuxième argument et supprimer le premier, comme nous le voyons :

@test S(K)(S)(K) === K
@test S(K)(S)(I) === I

S(I)(I) doit appliquer son argument à lui-même :

@test S(I)(I)(I) === I
@test S(I)(I)(K) === K(K)
@test S(I)(I)(S(I)) === S(I)(S(I))

S(K(S(I)))(K) applique son deuxième argument à son premier :

@test S(K(S(I)))(K)(I)(I) === I
@test S(K(S(I)))(K)(K)(S(K)) === S(K)(K)

Le combinateur I décrit ci-dessus porte un nom dans la norme Base Julia : identity. Ainsi, nous aurions pu réécrire les définitions ci-dessus avec la définition alternative suivante de “I”:

const I = identity

Affichage des combinateurs SKI #

Une faiblesse de l’approche ci-dessus est que nos fonctions ne s’affichent pas aussi bien que nous le souhaiterions. Pourrions-nous remplacer

julia> S
(::#3) (generic function with 1 method)

julia> K
(::#9) (generic function with 1 method)

julia> I
(::#13) (generic function with 1 method)

avec des affichages plus informatifs ? La réponse est oui! Redémarrons le REPL, et cette fois définissons comment chaque fonction doit être affichée :

const S = f -> g -> z -> f(z)(g(z));
const K = x -> y -> x;
const I = x -> x;
for f in (:S, :K, :I)
    @eval Base.show(io::IO, ::typeof($f)) = print(io, $(string(f)))
    @eval Base.show(io::IO, ::MIME"text/plain", ::typeof($f)) = show(io, $f)
end

Il est important d’éviter de montrer quoi que ce soit avant d’avoir fini de définir les fonctions. Sinon, nous risquons d’invalider le cache de méthodes, et nos nouvelles méthodes ne sembleront pas prendre effet immédiatement. C’est pourquoi nous avons mis des points-virgules dans les définitions ci-dessus. Les points-virgules suppriment la sortie du REPL.

Cela rend les fonctions bien affichées:

julia> S
S

julia> K
K

julia> I
I

Cependant, nous rencontrons toujours des problèmes lorsque nous essayons d’afficher une fermeture :

julia> S(K)
(::#2) (generic function with 1 method)

Il serait plus agréable de l’afficher sous la forme “S(K)”. Pour ce faire, nous devons exploiter le fait que les fermetures ont leurs propres types individuels. Nous pouvons accéder à ces types et leur ajouter des méthodes par réflexion, en utilisant typeof et le champ primary du champ name du type. Redémarrez le REPL à nouveau ; nous apporterons d’autres modifications :

const S = f -> g -> z -> f(z)(g(z));
const K = x -> y -> x;
const I = x -> x;
for f in (:S, :K, :I)
    @eval Base.show(io::IO, ::typeof($f)) = print(io, $(string(f)))
    @eval Base.show(io::IO, ::MIME"text/plain", ::typeof($f)) = show(io, $f)
end
Base.show(io::IO, s::typeof(S(I)).name.primary) = print(io, "S(", s.f, ')')
Base.show(io::IO, s::typeof(S(I)(I)).name.primary) =
    print(io, "S(", s.f, ')', '(', s.g, ')')
Base.show(io::IO, k::typeof(K(I)).name.primary) = print(io, "K(", k.x, ')')
Base.show(io::IO, ::MIME"text/plain", f::Union{
    typeof(S(I)).name.primary,
    typeof(S(I)(I)).name.primary,
    typeof(K(I)).name.primary
}) = show(io, f)

Et maintenant, enfin, les choses s’affichent comme nous le voudrions :

julia> S(K)
S(K)

julia> S(K)(I)
S(K)(I)

julia> K
K

julia> K(I)
K(I)

julia> K(I)(K)
I
