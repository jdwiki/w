Les raffinements ont une portée lexicale, ce qui signifie qu’ils sont effectifs à partir du moment où ils sont activés (avec le mot-clé “using”) jusqu’au changement de contrôle. Habituellement, le contrôle est modifié à la fin d’un module, d’une classe ou d’un fichier.

Application de correctifs de singe avec une portée limitée #

Le principal problème du patching de singe est qu’il pollue la portée mondiale. Votre travail de code est à la merci de tous les modules que vous utilisez sans vous marcher sur les pieds. La solution Ruby à cela est des raffinements, qui sont essentiellement des correctifs de singe dans une portée limitée.

module Patches
  refine Fixnum do
    def plus_one
      self + 1
    end

    def plus(num)
      self + num
    end

    def concat_one
      self.to_s + '1'
    end
  end
end

class RefinementTest
  # has access to our patches
  using Patches

  def initialize
    puts 1.plus_one
    puts 3.concat_one
  end
end

# Main scope doesn't have changes

1.plus_one
# => undefined method `plus_one' for 1:Fixnum (NoMethodError)

RefinementTest.new
# => 2
# => '31'

Modules à double usage (raffinements ou correctifs globaux) #

C’est une bonne pratique d’étendre les correctifs à l’aide des raffinements, mais il est parfois agréable de le charger globalement (par exemple en développement ou en test).

Supposons, par exemple, que vous souhaitiez démarrer une console, exiger votre bibliothèque, puis disposer des méthodes corrigées dans la portée globale. Vous ne pouvez pas faire cela avec des raffinements car using doit être appelé dans une définition de classe/module. Mais il est possible d’écrire le code de telle sorte qu’il ait un double objectif :

module Patch
  def patched?; true; end
  refine String do
    include Patch
  end
end

# globally
String.include Patch
"".patched? # => true

# refinement
class LoadPatch
  using Patch
  "".patched? # => true
end

Raffinements dynamiques #

Les raffinements ont des limites particulières.

refine ne peut être utilisé que dans la portée d’un module, mais peut être programmé en utilisant send :refine.

“utiliser” est plus limité. Il ne peut être appelé que dans une définition de classe/module. Néanmoins, il peut accepter une variable pointant vers un module et peut être invoqué dans une boucle.

Un exemple montrant ces concepts :

module Patch
  def patched?; true; end
end

Patch.send(:refine, String) { include Patch }

patch_classes = [Patch]

class Patched
  patch_classes.each { |klass| using klass }
  "".patched? # => true
end

Étant donné que “using” est si statique, un ordre de chargement peut être émis si les fichiers de raffinement ne sont pas chargés en premier. Une façon de résoudre ce problème consiste à encapsuler la définition de classe/module corrigée dans une procédure. Par exemple:

module Patch
  refine String do
    def patched; true; end
  end
end

class Foo
end

# This is a proc since methods can't contain class definitions
create_patched_class = Proc.new do
  Foo.class_exec do
    class Bar
      using Patch
      def self.patched?; ''.patched == true; end
    end
  end
end
create_patched_class.call
Foo::Bar.patched? # => true

L’appel du proc crée la classe patchée Foo::Bar. Cela peut être retardé jusqu’à ce que tout le code soit chargé.
