Une interface spécifie une liste de champs et de fonctions qui peuvent être attendus sur n’importe quelle classe implémentant l’interface. Inversement, une classe ne peut pas implémenter une interface à moins qu’elle n’ait tous les champs et fonctions spécifiés sur l’interface.

Le principal avantage de l’utilisation d’interfaces est qu’elle permet d’utiliser des objets de différents types de manière polymorphe. En effet, toute classe implémentant l’interface possède au moins ces champs et fonctions.

Syntaxe #

  • interface NomInterface {
  •   nomParamètre : TypeParamètre ;
  •   nomParamètrefacultatif ? : TypeParamètre ;
  • }

Interfaces vs alias de type #

Les interfaces sont bonnes pour spécifier la forme d’un objet, par exemple pour un objet personne que vous pourriez spécifier

interface person {
    id?: number;
    name: string;
    age: number;
}

Mais que se passe-t-il si vous voulez représenter, par exemple, la façon dont une personne est stockée dans une base de données SQL ? Étant donné que chaque entrée de la base de données consiste en une ligne de forme [string, string, number] (donc un tableau de chaînes ou de nombres), il n’y a aucun moyen de représenter cela comme une forme d’objet, car la ligne n’a pas toutes les propriétés en tant que telles, c’est juste un tableau.

C’est une occasion où les types sont utiles. Au lieu de spécifier dans chaque fonction qui accepte un paramètre de ligne function processRow(row: [string, string, number]), vous pouvez créer un alias de type séparé pour une ligne, puis l’utiliser dans chaque fonction :

type Row = [string, string, number];
function processRow(row: Row)

Documentation officielle de l’interface #

https://www.typescriptlang.org/docs/handbook/interfaces.html

Extension de l’interface #

Supposons que nous ayons une interface :

interface IPerson {
    name: string;
    age: number;

    breath(): void;
}

Et nous voulons créer une interface plus spécifique qui a les mêmes propriétés de la personne, nous pouvons le faire en utilisant le mot clé extends :

interface IManager extends IPerson {
    managerId: number;

    managePeople(people: IPerson[]): void;
}

De plus, il est possible d’étendre plusieurs interfaces.

Interface de classe #

Déclarez des variables et des méthodes “publiques” dans l’interface pour définir comment d’autres codes dactylographiés peuvent interagir avec lui.

interface ISampleClassInterface {
  sampleVariable: string;

  sampleMethod(): void;
  
  optionalVariable?: string;
}

Ici, nous créons une classe qui implémente l’interface.

class SampleClass implements ISampleClassInterface {
  public sampleVariable: string;
  private answerToLifeTheUniverseAndEverything: number;

  constructor() {
    this.sampleVariable = 'string value';
    this.answerToLifeTheUniverseAndEverything = 42;
  }

  public sampleMethod(): void {
    // do nothing
  }
  private answer(q: any): number {
    return this.answerToLifeTheUniverseAndEverything;
  }
}

L’exemple montre comment créer une interface ISampleClassInterface et une classe SampleClass qui implémente l’interface.

Ajouter des fonctions ou des propriétés à une interface existante #

Supposons que nous ayons une référence à la définition de type JQuery et que nous souhaitions l’étendre pour avoir des fonctions supplémentaires à partir d’un plugin que nous avons inclus et qui n’a pas de définition de type officielle. Nous pouvons facilement l’étendre en déclarant les fonctions ajoutées par le plugin dans une déclaration d’interface séparée avec le même nom JQuery :

interface JQuery {
  pluginFunctionThatDoesNothing(): void;

  // create chainable function
  manipulateDOM(HTMLElement): JQuery;
}

Le compilateur fusionnera toutes les déclarations portant le même nom en une seule - voir declaration merging pour plus de détails.

Interfaces génériques #

Comme les classes, les interfaces peuvent également recevoir des paramètres polymorphes (aka Generics).

Déclaration de paramètres génériques sur les interfaces #

interface IStatus<U> {
    code: U;
}

interface IEvents<T> {
    list: T[];
    emit(event: T): void;
    getAll(): T[];
}

Ici, vous pouvez voir que nos deux interfaces prennent des paramètres génériques, T et U.

Implémentation d’interfaces génériques #

Nous allons créer une classe simple afin d’implémenter l’interface IEvents.

class State<T> implements IEvents<T> {
    
    list: T[];
    
    constructor() {
        this.list = [];
    }
    
    emit(event: T): void {
        this.list.push(event);
    }
    
    getAll(): T[] {
        return this.list;
    }
    
}

Créons quelques instances de notre classe State.

Dans notre exemple, la classe State gérera un statut générique en utilisant ISatus<T>. De cette manière, l’interface IEvent<T> gérera également un IStatus<T>.

const s = new State<IStatus<number>>();

// The 'code' property is expected to be a number, so:
s.emit({ code: 200 }); // works
s.emit({ code: '500' }); // type error 

s.getAll().forEach(event => console.log(event.code));

Ici, notre classe State est typée comme ISatus<number>.

const s2 = new State<IStatus<Code>>();

//We are able to emit code as the type Code
s2.emit({ code: { message: 'OK', status: 200 } });

s2.getAll().map(event => event.code).forEach(event => {
    console.log(event.message);
    console.log(event.status);
});

Notre classe State est typée comme IStatus<Code>. De cette façon, nous pouvons passer un type plus complexe à notre méthode d’émission.

Comme vous pouvez le voir, les interfaces génériques peuvent être un outil très utile pour le code typé statiquement.

Utilisation d’interfaces pour le polymorphisme #

La principale raison d’utiliser des interfaces pour réaliser le polymorphisme et permettre aux développeurs de les implémenter à leur manière à l’avenir en implémentant les méthodes de l’interface.

Supposons que nous ayons une interface et trois classes :

interface Connector{
    doConnect(): boolean;
}

C’est l’interface du connecteur. Nous allons maintenant l’implémenter pour la communication Wifi.

export class WifiConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via wifi");
        console.log("Get password");
        console.log("Lease an IP for 24 hours");
        console.log("Connected");
        return true
    }

}

Ici, nous avons développé notre classe concrète nommée “WifiConnector” qui a sa propre implémentation. C’est maintenant le type Connector.

Maintenant, nous créons notre System qui a un composant Connector. C’est ce qu’on appelle l’injection de dépendance.

export class System {
    constructor(private connector: Connector){ #inject Connector type
        connector.doConnect()
    }
}

constructor(private connector: Connector) cette ligne est très importante ici. Connector est une interface et doit avoir doConnect(). Comme Connector est une interface, cette classe System a beaucoup plus de flexibilité. Nous pouvons passer n’importe quel type qui a implémenté l’interface Connector. À l’avenir, le développeur obtient plus de flexibilité. Par exemple, le développeur souhaite maintenant ajouter le module de connexion Bluetooth :

export class BluetoothConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via Bluetooth");
        console.log("Pair with PIN");
        console.log("Connected");
        return true
    }

}

Voir que Wifi et Bluetooth ont leur propre implémentation. Il existe différentes manières de se connecter. Cependant, les deux ont implémenté le type Connector et sont maintenant le type Connector. Pour que nous puissions passer n’importe lequel d’entre eux à la classe System en tant que paramètre du constructeur. C’est ce qu’on appelle le polymorphisme. La classe System ne sait plus s’il s’agit de Bluetooth / Wifi, même si nous pouvons ajouter un autre module de communication comme Inferade, Bluetooth5 et autre en implémentant simplement l’interface Connector.

C’est ce qu’on appelle [Duck typing][1]. Le type Connector est maintenant dynamique car doConnect() n’est qu’un espace réservé et le développeur l’implémente comme le sien.

si à constructeur (connecteur privé : WifiConnector) où WifiConnector est une classe concrète, que se passera-t-il ? Ensuite, la classe System ne sera étroitement couplée qu’avec WifiConnector rien d’autre. Ici, l’interface a résolu notre problème par polymorphisme.

[1] : https://en.wikipedia.org/wiki/Duck_typing

Utilisation d’interfaces pour appliquer des types #

L’un des principaux avantages de Typescript est qu’il applique les types de données de valeurs que vous transmettez dans votre code pour éviter les erreurs.

Disons que vous créez une application de rencontres pour animaux de compagnie.

Vous avez cette fonction simple qui vérifie si deux animaux sont compatibles entre eux…

checkCompatible(petOne, petTwo) {
  if (petOne.species === petTwo.species &&
      Math.abs(petOne.age - petTwo.age) <= 5) {
    return true;
  }
}

C’est du code complètement fonctionnel, mais il serait beaucoup trop facile pour quelqu’un, en particulier d’autres personnes travaillant sur cette application qui n’ont pas écrit cette fonction, d’ignorer qu’ils sont censés lui transmettre des objets avec ’espèce’ et ‘âge’ Propriétés. Ils peuvent essayer par erreur checkCompatible(petOne.species, petTwo.species) et ensuite être laissés pour comprendre les erreurs générées lorsque la fonction tente d’accéder à petOne.species.species ou petOne.species.age !

Une façon d’éviter que cela ne se produise est de spécifier les propriétés que nous voulons sur les paramètres de l’animal :

checkCompatible(petOne: {species: string, age: number}, petTwo: {species: string, age: number}) {
    //...
}

Dans ce cas, Typescript s’assurera que tout ce qui est passé à la fonction a les propriétés ‘species’ et ‘age’ (c’est bien s’ils ont des propriétés supplémentaires), mais c’est une solution un peu lourde, même avec seulement deux propriétés spécifiées. Avec les interfaces, il y a mieux !

Nous définissons d’abord notre interface :

interface Pet {
  species: string;
  age: number;
  //We can add more properties if we choose.
}

Maintenant, tout ce que nous avons à faire est de spécifier le type de nos paramètres comme notre nouvelle interface, comme ceci…

checkCompatible(petOne: Pet, petTwo: Pet) {
  //...
}

… et Typescript s’assurera que les paramètres passés à notre fonction contiennent les propriétés spécifiées dans l’interface Pet !

Implémentation implicite et forme d’objet #

TypeScript prend en charge les interfaces, mais le compilateur génère JavaScript, ce qui n’est pas le cas. Par conséquent, les interfaces sont effectivement perdues lors de l’étape de compilation. C’est pourquoi la vérification de type sur les interfaces repose sur la * forme * de l’objet - c’est-à-dire si l’objet prend en charge les champs et les fonctions de l’interface - et non sur le fait que l’interface est réellement implémentée ou non.

interface IKickable {
  kick(distance: number): void;
}
class Ball {
  kick(distance: number): void {
    console.log("Kicked", distance, "meters!");
  }
}
let kickable: IKickable = new Ball();
kickable.kick(40);

Ainsi, même si Ball n’implémente pas explicitement IKickable, une instance de Ball peut être assignée à (et manipulée comme) un IKickable, même lorsque le type est spécifié.
