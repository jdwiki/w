Pourquoi et quand utiliser TypeScript

  • Sécurité
  • Lisibilité
  • Outillage

Sur cette page

  • Sécurité
  • Lisibilité
  • Outillage

Si vous trouvez les arguments des systèmes de type convaincants en général, alors vous serez satisfait de TypeScript.

Il apporte de nombreux avantages du système de type (sécurité, lisibilité, amélioration de l’outillage) à l’écosystème JavaScript. Il souffre également de certains des inconvénients des systèmes de types (complexité accrue et incomplétude).

Les mérites des langages typés par rapport aux langages non typés ont été débattus pendant des décennies. Les arguments pour les types statiques incluent :

  1. Sécurité : les systèmes de type permettent de détecter rapidement de nombreuses erreurs, sans exécuter le code. TypeScript peut être [configuré pour autoriser moins d’erreurs de programmation][7]
  2. Lisibilité : les types explicites rendent le code plus facile à comprendre pour les humains. Comme Fred Brooks [a écrit] [1], “Montrez-moi vos organigrammes et cachez vos tableaux, et je continuerai à être mystifié. Montrez-moi vos tableaux, et je n’aurai généralement pas besoin de vos organigrammes ; ils seront évidents.”
  3. Outils : les systèmes de typage rendent le code plus facile à comprendre pour les ordinateurs. Cela permet à des outils tels que les IDE et les linters d’être plus puissants.
  4. Performance : les systèmes de typage accélèrent l’exécution du code en réduisant le besoin de vérification de type à l’exécution.

Étant donné que [la sortie de TypeScript est indépendante de ses types][2], TypeScript n’a aucun impact sur les performances. L’argument en faveur de l’utilisation de TypeScript repose sur les trois autres avantages.

Les arguments contre les systèmes de type incluent:

  1. Complexité accrue : les systèmes de types peuvent être plus complexes que le temps d’exécution du langage qu’ils décrivent. Les fonctions d’ordre supérieur peuvent être faciles à implémenter correctement mais [difficiles à taper][3]. La gestion des définitions de type crée des obstacles supplémentaires à l’utilisation de bibliothèques externes.
  2. Verbosité ajoutée : les annotations de type peuvent ajouter un passe-partout à votre code, ce qui rend la logique sous-jacente plus difficile à suivre.
  3. Itération plus lente : en introduisant une étape de build, TypeScript ralentit le cycle édition/sauvegarde/rechargement.
  4. Incomplétude : Un système de types ne peut pas être à la fois solide et complet. Il existe des programmes corrects que TypeScript n’autorise pas. Et les programmes acceptés par TypeScript peuvent toujours contenir des bogues. Un système de type n’élimine pas le besoin de test. Si vous utilisez TypeScript, vous devrez peut-être attendre plus longtemps pour utiliser les nouvelles fonctionnalités du langage ECMAScript.

TypeScript propose des moyens de résoudre tous ces problèmes :

  1. Plus de complexité. Si la saisie d’une partie d’un programme est trop difficile, TypeScript peut être largement désactivé à l’aide d’un type “tout” opaque. Il en est de même pour les modules externes.
  2. Verbosité ajoutée. Cela peut être partiellement résolu par les alias de type et la capacité de TypeScript à déduire les types. Écrire du code clair est autant un art qu’une science : supprimez trop d’annotations de type et le code risque de ne plus être clair pour les lecteurs humains.
  3. Itération plus lente : une étape de construction est relativement courante dans le développement JS moderne et TypeScript déjà [s’intègre à la plupart des outils de construction] [6]. Et si TypeScript détecte une erreur plus tôt, cela peut vous faire économiser un cycle d’itération complet !
  4. Incomplétude. Bien que ce problème ne puisse pas être complètement résolu, TypeScript a été en mesure de capturer de plus en plus de modèles JavaScript expressifs au fil du temps. Les exemples récents incluent l’ajout de [mapped types in TypeScript 2.1][4] et [mixins in 2.2][5].

Les arguments pour et contre les systèmes de type en général s’appliquent aussi bien à TypeScript. L’utilisation de TypeScript augmente les frais généraux liés au démarrage d’un nouveau projet. Mais au fil du temps, à mesure que le projet prend de l’ampleur et gagne plus de contributeurs, l’espoir est que les avantages de son utilisation (sécurité, lisibilité, outillage) deviennent plus forts et l’emportent sur les inconvénients. Cela se reflète dans la devise de TypeScript : “JavaScript qui évolue”.

Sécurité #

TypeScript détecte rapidement les erreurs de type grâce à une analyse statique :

function double(x: number): number {
  return 2 * x;
}
double('2');
//     ~~~ Argument of type '"2"' is not assignable to parameter of type 'number'.

Lisibilité #

TypeScript permet aux éditeurs de fournir une documentation contextuelle :

[![Autocomplétion contextuelle pour la méthode de tranche de chaîne][1]][1]

Vous n’oublierez jamais si String.prototype.slice prend (start, stop) ou (start, length) à nouveau !

[1] : https://i.stack.imgur.com/olNJH.png

Outillage #

TypeScript permet aux éditeurs d’effectuer des refactorisations automatisées qui connaissent les règles des langages.

[![Un symbole renommé, mais uniquement dans sa propre portée][1]][1]

Ici, par exemple, Visual Studio Code est capable de renommer les références au « foo » interne sans modifier le « foo » externe. Ce serait difficile à faire avec une simple recherche/remplacement.

[1] : https://i.stack.imgur.com/N1C0T.gif
