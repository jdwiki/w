L’API RESTful de SugarCRM a été mise à jour vers la version 10 dans les versions 7.x de SugarCRM.

Comme les versions 7.x sont conçues comme une application à page unique, l’API est le principal moyen de communication avec le serveur pour récupérer les données. Les intégrations tierces peuvent également se connecter et accéder aux données. L’API peut également être étendue pour fournir des points de terminaison personnalisés, implémentant les fonctionnalités souhaitées.

[Documents officiels] (http://support.sugarcrm.com/Documentation/Sugar_Developer/Sugar_Developer_Guide_7.9/Integration/Web_Services/v10/)

Se connecter #

Vous pouvez utiliser cet exemple comme un appel à partir du code ou via un client REST tel que Postman :

POSTER https://VOTRESITE.com/rest/v10/oauth2/token

{
"grant_type":"password",
"client_id":"sugar",
"client_secret": "",
"username":"your_username",
"password":"your_P@$sw0rd",
"platform":"api"
}

Pour élaborer sur les détails ci-dessus, ils signifient :

“grant_type”:“password” - Il s’agit de la méthode de connexion. Dans ce cas, nous utilisons une combinaison nom d’utilisateur/mot de passe, nous mettons donc “mot de passe”

“client_id”:“sugar” - Cela indique que nous nous authentifions via le client standard “sugar” de nom/mot de passe. D’autres clients tels que “support_portal”. Cela peut également faire référence à une clé OAuth personnalisée dans l’application.

“client_secret”: “” - Ceci est vide car nous utilisons l’identifiant client de “sugar”, mais si vous utilisez un client personnalisé, il s’agit du secret associé au client.

“username”:“your_username” - Votre nom d’utilisateur

“password”:“votre_P@$sw0rd” - Votre mot de passe

“platform”:“api” - Les plates-formes typiques utilisées dans sugar sont “base”, “mobile” et “portal”. Pour des raisons de sécurité, si le même utilisateur se connecte simultanément à la même plate-forme, il déconnectera l’utilisateur de la session précédente. J’ai utilisé “api”, mais cela pourrait être “myawesomesugarintegration” si vous le vouliez.

Lors d’une requête réussie, le serveur renvoie un objet incluant un “access_token”, c’est-à-dire :

“access_token”:"abcdef01-2345-6789-0abc-def012345678"

Pour chaque demande ultérieure à l’API, vous devez inclure ce jeton pour vous authentifier.

Cela peut être réalisé en ajoutant l’en-tête "OAuth-Token" avec la valeur access_token à toute autre demande.

Opérations CRUD de base #

Tout d’abord, assurez-vous de vous [connecter](https://www.wikiod.com/fr/sugarcrm/api-rest-v10#Se connecter) et ajoutez le access_token à votre en-tête de requête.

Dans cet exemple, nous allons effectuer quelques opérations de base pour accéder aux enregistrements. Ceux-ci utilisent le module Comptes comme exemple, mais d’autres modules standard et personnalisés (c’est-à-dire Leads, Contacts, Opportunities) se comportent de la même manière.

Créer

Pour créer un compte, nous devons faire une demande de publication au point de terminaison des comptes, avec les détails que nous voulons ajouter. En cas de succès, cela renvoie un objet contenant l’ID du nouvel enregistrement et les données actuelles.

POST https://YOURSITE.com/rest/v10/Accounts
{
"name":"My New Account"
}

Lis

Tout d’abord, nous allons récupérer l’enregistrement que nous venons de créer, pour moi l’ID était “9174c58c-409c-11e7-bfdf-00163ef1f82f” donc pour récupérer toutes les informations de l’enregistrement, nous procédons comme suit :

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

C’est sûr que c’est un gros objet ! Qu’en est-il simplement de voir le nom et la date à laquelle je viens de le créer ?

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f?fields=name,date_entered

Beaucoup mieux. Mais que se passe-t-il si j’ai déjà des milliers de comptes dans le système et que je n’ai pas réussi à me souvenir du GUID correspondant ?

GET https://YOURSITE.com/rest/v10/Accounts?fields=name,date_entered&filter[0][name]=My New Account

Mise à jour

Et si je décidais que je voulais changer quelque chose sur l’application ? Que diriez-vous de changer le nom et d’ajouter une description ?

PUT https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f
{
    "name":"My Updated Account",
    "description":"Updated via REST API"
}

Effacer

D’accord, tout va bien, mais clarifions cela avant de mettre des données réelles :

DELETE https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

Création de points de terminaison personnalisés #

L’une des fonctionnalités de SugarCRM 7.x est la possibilité d’ajouter et d’étendre facilement des points de terminaison personnalisés pour accomplir ce dont vous avez besoin.

Dans cet exemple, nous allons créer quelques points de terminaison personnalisés pour renvoyer des données sur la demande.

Ce fichier personnalisé est placé dans custom/clients/base/api/DescriptionAPI.php.

<?php

if(!defined('sugarEntry') || !sugarEntry) die('Not A Valid Entry Point');

//You need to inherit the SugarApi Class to
class DescriptionApi extends SugarApi
{
    public function registerApiRest()
    {
        return array(

            //Define a key for the array
            'DescribeRequest' => array(

                //Array of the acceptable kinds of requests for this method
                'reqType' => array('GET','POST','PUT','DELETE'),

                //If true, anyone can access. If false, only authenticated users.
                'noLoginRequired' => true,

                //Here is the path to access the endpoint, in this case: Describe/Request
                'path' => array('Describe', 'Request'),

                //Specify an empty string for the path variables
                'pathVars' => array('', ''),

                //method to call
                'method' => 'DescribeMyRequest',

                //A small description, displayed in rest/v10/help page
                'shortHelp' => 'Describes your Request Method',

                //Further help, displayed when drilling down into the help page
                'longHelp' => 'custom/clients/base/api/help/DescribeRequestHelp.html',
            ),
            //Here's another entry with some more in depth information
            'DescribeIncludingArgs' => array(
                'reqType' => array('GET','POST','PUT','DELETE'),
                'noLoginRequired' => true,

                //This time, we'll include a third element with a ?
                //So now the path is Describe/Request/{dataFromURL}
                'path' => array('Describe', 'Request', '?'),

                //Here, we specify the key for accessing that data within the function
                'pathVars' => array('', '','dataFromURL'),
                
                'method' => 'DescribeMyRequestIncludingArguments',
                'shortHelp' => 'Describes the request you sent, including method, URL parameters, and request body',
                'longHelp' => 'custom/clients/base/api/help/DescribeIncludingArgsHelp.html',
            ),
        );

    }

    /**
     * Your custom logic goes in here.
     */
    public function DescribeMyRequest($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        return "You sent a $requestType request.";
    }

    /**
     * Here is the second function
     */
    public function DescribeMyRequestIncludingArguments($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        //Get the data included in the URL parameter
        $data = $args['dataFromURL'];

        //Read from the request body
        $body = file_get_contents('php://input');

        return "You sent a $requestType request including the header argument: `$data` and the body: `$body`";
    }

}

Après avoir ajouté ce fichier, vous devrez effectuer une réparation et une reconstruction pour que Sugar enregistre correctement votre point de terminaison.

Après, si vous accédez à rest/v10/Describe/Request dans votre navigateur, vous devriez voir :

“Vous avez envoyé une requête GET.”

Maintenant, si vous utilisez un client REST pour envoyer une requête POST, avec des données, par exemple : POST rest/v10/Describe/Request/Stuff avec le corps {"key":"value"} vous devriez recevoir :

“Vous avez envoyé une requête POST incluant l’argument d’en-tête : Stuff et le corps : {\"clé\":\"valeur\"}”
