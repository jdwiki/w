[AspNetCoreRateLimit] [1] est une solution de limitation de débit ASP.NET Core open source conçue pour contrôler le taux de requêtes que les clients peuvent envoyer à une API Web ou à une application MVC en fonction de l’adresse IP ou de l’ID client.

[1] : https://github.com/stefanprodan/AspNetCoreRateLimit

Limitation du débit basée sur l’ID client #

Avec le middleware ClientRateLimit, vous pouvez définir plusieurs limites pour différents scénarios, comme autoriser un client à effectuer un nombre maximum d’appels dans un intervalle de temps comme par seconde, 15 minutes, etc. Vous pouvez définir ces limites pour répondre à toutes les demandes adressées à une API ou vous peut étendre les limites à chaque chemin d’URL ou verbe et chemin HTTP.

Installer #

Installation NuGet :

Install-Package AspNetCoreRateLimit

Code Startup.cs :

public void ConfigureServices(IServiceCollection services)
{
    // needed to load configuration from appsettings.json
    services.AddOptions();

    // needed to store rate limit counters and ip rules
    services.AddMemoryCache();

    //load general configuration from appsettings.json
    services.Configure<ClientRateLimitOptions>(Configuration.GetSection("ClientRateLimiting"));

    //load client rules from appsettings.json
    services.Configure<ClientRateLimitPolicies>(Configuration.GetSection("ClientRateLimitPolicies"));

    // inject counter and rules stores
    services.AddSingleton<IClientPolicyStore, MemoryCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore, MemoryCacheRateLimitCounterStore>();

    // Add framework services.
    services.AddMvc();
}

public void Configure(IApplicationBuilder app, IHostingEnvironment env, ILoggerFactory loggerFactory)
{
    loggerFactory.AddConsole(Configuration.GetSection("Logging"));
    loggerFactory.AddDebug();

    app.UseClientRateLimiting();

    app.UseMvc();
}

Vous devez enregistrer le middleware avant tout autre composant, à l’exception de loggerFactory.

si vous équilibrez la charge de votre application, vous devrez utiliser IDistributedCache avec Redis ou SQLServer afin que toutes les instances de Kestrel aient le même magasin de limite de débit. Au lieu des magasins en mémoire, vous devez injecter les magasins distribués comme ceci :

    // inject counter and rules distributed cache stores
    services.AddSingleton<IClientPolicyStore, DistributedCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore,DistributedCacheRateLimitCounterStore>();

Configuration et règles générales appsettings.json :

  "ClientRateLimiting": {
    "EnableEndpointRateLimiting": false,
    "StackBlockedRequests": false,
    "ClientIdHeader": "X-ClientId",
    "HttpStatusCode": 429,
    "EndpointWhitelist": [ "get:/api/license", "*:/api/status" ],
    "ClientWhitelist": [ "dev-id-1", "dev-id-2" ],
    "GeneralRules": [
      {
        "Endpoint": "*",
        "Period": "1s",
        "Limit": 2
      },
      {
        "Endpoint": "*",
        "Period": "15m",
        "Limit": 100
      },
      {
        "Endpoint": "*",
        "Period": "12h",
        "Limit": 1000
      },
      {
        "Endpoint": "*",
        "Period": "7d",
        "Limit": 10000
      }
    ]
  }

Si EnableEndpointRateLimiting est défini sur false, les limites s’appliqueront globalement et seules les règles ayant comme point de terminaison * s’appliqueront. Par exemple, si vous définissez une limite de 5 appels par seconde, tout appel HTTP vers n’importe quel point de terminaison comptera dans cette limite.

Si EnableEndpointRateLimiting est défini sur true, les limites s’appliqueront à chaque point de terminaison comme dans {HTTP_Verb}{PATH}. Par exemple, si vous fixez une limite de 5 appels par seconde pour *:/api/values un client peut appeler GET /api/values 5 fois par seconde mais aussi 5 fois PUT /api/values.

Si StackBlockedRequests est défini sur false, les appels rejetés ne sont pas ajoutés au compteur de limitation. Si un client fait 3 requêtes par seconde et que vous avez défini une limite d’un appel par seconde, d’autres limites comme les compteurs par minute ou par jour n’enregistreront que le premier appel, celui qui n’a pas été bloqué. Si vous souhaitez que les requêtes rejetées soient prises en compte dans les autres limites, vous devez définir StackBlockedRequests sur true.

Le ClientIdHeader est utilisé pour extraire l’ID client, si un ID client est présent dans cet en-tête et correspond à une valeur spécifiée dans ClientWhitelist, aucune limite de débit n’est appliquée.

Remplacer les règles générales pour des clients spécifiques appsettings.json :

      {

            "Period": "15m",
            "Limit": 200
          }
        ]
      },
      {
        "Client": "client-id-2",

Définir des règles de limite de taux #

Une règle est composée d’un point final, d’une période et d’une limite.

Le format du point de terminaison est {HTTP_Verb} :{PATH}, vous pouvez cibler n’importe quel verbe HTTP en utilisant le symbole astérisque.

Le format de période est {INT}{PERIOD_TYPE}, vous pouvez utiliser l’un des types de périodes suivants : s, m, h, d.

Le format limite est {LONG}.

Exemples:

Débit limite tous les terminaux à 2 appels par seconde :

{
 "Endpoint": "*",
 "Period": "1s",
 "Limit": 2
}

Si dans la même seconde, un client fait 3 appels GET à api/values, le dernier appel sera bloqué. Mais si, dans la même seconde, il appelle également PUT api/values, la requête sera acceptée car il s’agit d’un point de terminaison différent. Lorsque la limitation du débit du point de terminaison est activée, chaque appel est limité en débit en fonction de {HTTP_Verb}{PATH}.

Limitez les appels avec n’importe quel verbe HTTP à /api/values à 5 appels par 15 minutes :

{
 "Endpoint": "*:/api/values",
 "Period": "15m",
 "Limit": 5
}

Limite de débit des appels GET vers /api/values à 5 appels par heure :

{
 "Endpoint": "get:/api/values",
 "Period": "1h",
 "Limit": 5
}

Si en une heure, un client fait 6 appels GET à api/values, le dernier appel sera bloqué. Mais si, dans la même heure, il appelle également GET api/values/1, la requête sera acceptée car il s’agit d’un point de terminaison différent.

Comportement #

Lorsqu’un client effectue un appel HTTP, ClientRateLimitMiddleware effectue les opérations suivantes :

  • extrait l’ID client, le verbe HTTP et l’URL de l’objet de requête, si vous souhaitez implémenter votre propre logique d’extraction, vous pouvez remplacer le ClientRateLimitMiddleware.SetIdentity
  • recherche l’ID client et l’URL dans les listes blanches, s’il y en a qui correspondent, aucune action n’est entreprise
  • recherche dans les règles Client une correspondance, toutes les règles applicables sont regroupées par période, pour chaque période la règle la plus restrictive est utilisée
  • recherche dans les règles générales une correspondance, si une règle générale qui correspond a une période définie qui n’est pas présente dans les règles client alors cette règle générale est également utilisée
  • pour chaque règle correspondante, le compteur de limite de débit est incrémenté, si la valeur du compteur est supérieure à la limite de la règle, la demande est bloquée

Si la requête est bloquée, le client reçoit une réponse textuelle comme celle-ci :

Status Code: 429
Retry-After: 58
Content: API calls quota exceeded! maximum admitted 2 per 1m.

Vous pouvez personnaliser la réponse en modifiant ces options HttpStatusCode et QuotaExceededMessage, si vous souhaitez implémenter votre propre réponse, vous pouvez remplacer ClientRateLimitMiddleware.ReturnQuotaExceededResponse. La valeur de l’en-tête Retry-After est exprimée en secondes.

Si la requête n’obtient pas de débit limité, la période la plus longue définie dans les règles de correspondance est utilisée pour composer les en-têtes X-Rate-Limit, ces en-têtes sont injectés dans la réponse :

X-Rate-Limit-Limit: the rate limit period (eg. 1m, 12h, 1d)
X-Rate-Limit-Remaining: number of request remaining 
X-Rate-Limit-Reset: UTC date time when the limits resets

Par défaut, les requêtes bloquées sont enregistrées à l’aide de Microsoft.Extensions.Logging.ILogger. Si vous souhaitez implémenter votre propre journalisation, vous pouvez remplacer ClientRateLimitMiddleware.LogBlockedRequest. L’enregistreur par défaut émet les informations suivantes lorsqu’une demande est limitée en débit :

info: AspNetCoreRateLimit.ClientRateLimitMiddleware[0]
      Request get:/api/values from ClientId client-id-1 has been blocked, quota 2/1m exceeded by 3. Blocked by rule *:/api/value, TraceIdentifier 0HKTLISQQVV9D.

Mettre à jour les limites de taux lors de l’exécution #

Au démarrage de l’application, les règles de limite de débit client définies dans appsettings.json sont chargées dans le cache par MemoryCacheClientPolicyStore ou DistributedCacheClientPolicyStore selon le type de fournisseur de cache que vous utilisez. Vous pouvez accéder au magasin de politiques client à l’intérieur d’un contrôleur et modifier les règles comme suit :

public class ClientRateLimitController : Controller
{
    private readonly ClientRateLimitOptions _options;
    private readonly IClientPolicyStore _clientPolicyStore;

    public ClientRateLimitController(IOptions<ClientRateLimitOptions> optionsAccessor, IClientPolicyStore clientPolicyStore)
    {
        _options = optionsAccessor.Value;
        _clientPolicyStore = clientPolicyStore;
    }

    [HttpGet]
    public ClientRateLimitPolicy Get()
    {
        return _clientPolicyStore.Get($"{_options.ClientPolicyPrefix}_cl-key-1");
    }

    [HttpPost]
    public void Post()
    {
        var id = $"{_options.ClientPolicyPrefix}_cl-key-1";
        var clPolicy = _clientPolicyStore.Get(id);
        clPolicy.Rules.Add(new RateLimitRule
        {
            Endpoint = "*/api/testpolicyupdate",
            Period = "1h",
            Limit = 100
        });
        _clientPolicyStore.Set(id, clPolicy);
    }
}

De cette façon, vous pouvez stocker les limites de débit client dans une base de données et les mettre en cache après chaque démarrage de l’application.

    private readonly IpRateLimitOptions _options;
    private readonly IIpPolicyStore _ipPolicyStore;

    public IpRateLimitController(IOptions<IpRateLimitOptions> optionsAccessor, IIpPolicyStore ipPolicyStore)
    {
        _options = optionsAccessor.Value;
        _ipPolicyStore = ipPolicyStore;
    }

    [HttpGet]
    public IpRateLimitPolicies Get()
    {
        return _ipPolicyStore.Get(_options.IpPolicyPrefix);
    }

    [HttpPost]
    public void Post()
    {
        var pol = _ipPolicyStore.Get(_options.IpPolicyPrefix);

        pol.IpRules.Add(new IpRateLimitPolicy
        {
            Ip = "8.8.4.4",
            Rules = new List<RateLimitRule>(new RateLimitRule[] {
                new RateLimitRule {
                    Endpoint = "*:/api/testupdate",
                    Limit = 100,
                    Period = "1d" }
            })
        });

        _ipPolicyStore.Set(_options.IpPolicyPrefix, pol);
    }
}

De cette façon, vous pouvez stocker les limites de débit IP dans une base de données et les mettre en cache après chaque démarrage de l’application.
