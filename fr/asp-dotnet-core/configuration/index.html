Le noyau Asp.net fournit des abstractions de configuration. Ils vous permettent de charger des paramètres de configuration à partir de diverses sources et de créer un modèle de configuration final qui peut ensuite être utilisé par votre application.

Syntaxe #

  • IConfiguration
  • string this[string key] { get; Positionner; }
  • IEnumerable<IConfigurationSection> GetChildren();
  • IConfigurationSection GetSection (clé de chaîne);

Commencer #

Dans cet exemple, nous décrirons ce qui se passe lorsque vous échafaudez un nouveau projet.

Tout d’abord, les dépendances suivantes seront ajoutées à votre projet (actuellement le fichier project.json) :

"Microsoft.Extensions.Configuration.EnvironmentVariables": "1.0.0",
"Microsoft.Extensions.Configuration.Json": "1.0.0",

Il créera également un constructeur dans votre fichier Startup.cs qui sera chargé de construire la configuration à l’aide de l’API fluide ConfigurationBuilder :

  1. Il crée d’abord un nouveau ConfigurationBuilder.
  2. Il définit ensuite un chemin de base qui sera utilisé pour calculer le chemin absolu d’autres fichiers
  3. Il ajoute un “appsettings.json” facultatif au générateur de configuration et surveille ses modifications
  4. Il ajoute un fichier de configuration appsettings.environementName.json lié à l’environnement facultatif
  5. Il ajoute ensuite des variables d’environnement.
public Startup(IHostingEnvironment env)
{
    var builder = new ConfigurationBuilder()
        .SetBasePath(env.ContentRootPath)
        .AddJsonFile("appsettings.json", optional: true, reloadOnChange: true)
        .AddJsonFile($"appsettings.{env.EnvironmentName}.json", optional: true)
        .AddEnvironmentVariables();

    Configuration = builder.Build();
}

Si un même paramètre est défini dans plusieurs sources, la dernière source ajoutée l’emportera et sa valeur sera sélectionnée.

La configuration peut ensuite être utilisée à l’aide de la propriété indexer. Le caractère deux-points : sert de délimiteur de chemin.

Configuration["AzureLogger:ConnectionString"]

Cela recherchera une valeur de configuration ConnectionString dans une section AzureLogger.

Travailler avec des variables d’environnement #

Vous pouvez sourcer la configuration à partir de variables d’environnement en appelant .AddEnvironmentVariables() sur votre ConfigurationBuilder.

Il chargera les variables d’environnement préfixées par APPSETTING_ Il utilisera ensuite les deux-points : comme séparateur de chemin de clé.

Cela signifie que : paramètres d’environnement suivants :

APPSETTING_Security:Authentication:UserName = a_user_name
APPSETTING_Security:Authentication:Password = a_user_password

Sera l’équivalent ce json :

{
    "Security" : {
       "Authentication" : {
           "UserName" : "a_user_name",
           "Password" : "a_user_password" 
        } 
    }
}

** Notez qu’Azure Service transmettra les paramètres en tant que variables d’environnement. Le préfixe sera défini pour vous de manière transparente. Donc, pour faire la même chose dans Azure, il suffit de définir deux paramètres d’application dans le panneau AppSettings :

Security:Authentication:UserName         a_user_name
Security:Authentication:Password         a_user_password

Modèle et configuration des options #

Lorsqu’il s’agit de grands ensembles de valeurs de configuration, il peut devenir assez peu pratique de les charger un par un.

Le modèle d’option fourni avec asp.net offre un moyen pratique de mapper une “section” à un “poco” dotnet : Par exemple, on peut hydrater StorageOptions directement à partir d’une section de configuration en ajoutant le package Microsoft.Extensions.Options.ConfigurationExtensions et en appelant la méthode d’extension Configure<TOptions>(IConfiguration config).

services.Configure<StorageOptions>(Configuration.GetSection("Storage"));

Source de configuration en mémoire #

Vous pouvez également sourcer la configuration à partir d’un objet en mémoire tel qu’un Dictionary<string,string>

.AddInMemoryCollection(new Dictionary<string, string>
{
    ["akey"] = "a value"
})

Cela peut s’avérer utile dans les scénarios de test d’intégration/unitaire.

Accéder à la configuration à l’aide de l’injection de dépendance #

L’approche recommandée serait d’éviter de le faire et d’utiliser plutôt IOptions<TOptions> et IServiceCollection.Configure<TOptions>.

Cela dit, il est encore assez simple de rendre IConfigurationRoot disponible à l’échelle de l’application.

Dans le constructeur Startup.cs, vous devez avoir le code suivant pour créer la configuration,

 Configuration = builder.Build();

Ici, Configuration est une instance de IConfigurationRoot, Et ajoutez cette instance en tant que Singleton à la collection de services dans la méthode ConfigureServices , Startup.cs ,

 public void ConfigureServices(IServiceCollection services)
 {
    services.AddSingleton<IConfigurationRoot>(provider => Configuration);

Par exemple, vous pouvez maintenant accéder à la configuration dans un Controller/Service

  public MyController(IConfigurationRoot config){
      var setting1= config.GetValue<string>("Setting1")
  }
