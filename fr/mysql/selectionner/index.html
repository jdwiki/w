SELECT est utilisé pour récupérer les lignes sélectionnées dans une ou plusieurs tables.

Syntaxe #

  • SELECT DISTINCT [expressions] FROM TableName [WHERE conditions] ; ///Sélection simple

  • SELECT DISTINCT(a), b … est identique à SELECT DISTINCT a, b …

  • CHOISIR [ TOUT | DISTINCT | DISTINCTROW ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ] [ SQL_SMALL_RESULT | SQL_BIG_RESULT ] [ SQL_BUFFER_RESULT ] [ SQL_CACHE | SQL_NO_CACHE ] [ SQL_CALC_FOUND_ROWS ] expressions DE tableaux [OÙ conditions] [expressions GROUPER PAR] [CONDITION D’AVOIR] [ORDER PAR expression [ ASC | DESC ]] [LIMIT [offset_value] nombre_lignes | LIMIT nombre_lignes DÉCALAGE valeur_décalage] [PROCÉDURE nom_procédure] Options [INTO [ OUTFILE ’nom_fichier' | DUMPFILE ‘file_name’ | @variable1, @variable2, … @variable_n] [POUR LA MISE À JOUR | VERROUILLER EN MODE PARTAGE] ; ///Syntaxe de sélection complète

Pour plus d’informations sur l’instruction SELECT de MySQL, consultez [MySQL Docs][1].

[1] : https://dev.mysql.com/doc/refman/5.7/en/select.html

SELECT avec DISTINCT #

La clause DISTINCT après SELECT élimine les lignes en double du jeu de résultats.

CREATE TABLE `car`
(   `car_id` INT UNSIGNED NOT NULL PRIMARY KEY, 
    `name` VARCHAR(20), 
    `price` DECIMAL(8,2)
);

INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (1, 'Audi A1', '20000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (2, 'Audi A1', '15000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (3, 'Audi A2', '40000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (4, 'Audi A2', '40000');

SELECT DISTINCT `name`, `price` FROM CAR;
+---------+----------+
| name    | price    |
+---------+----------+
| Audi A1 | 20000.00 |
| Audi A1 | 15000.00 |
| Audi A2 | 40000.00 |
+---------+----------+

DISTINCT fonctionne sur toutes les colonnes pour fournir les résultats, et non sur des colonnes individuelles. Ce dernier est souvent une idée fausse des nouveaux développeurs SQL. En bref, c’est la distinction au niveau de la ligne de l’ensemble de résultats qui compte, pas la distinction au niveau de la colonne. Pour visualiser cela, regardez “Audi A1” dans l’ensemble de résultats ci-dessus.

Pour les versions ultérieures de MySQL, DISTINCT a des implications avec son utilisation avec ORDER BY. Le paramètre pour ONLY_FULL_GROUP_BY entre en jeu, comme indiqué dans la page de manuel MySQL suivante intitulée [MySQL Handling of GROUP BY][1].

[1] : http://dev.mysql.com/doc/refman/5.7/en/group-by-handling.html

SELECT toutes les colonnes (*) #

Requête

SELECT * FROM stack;

Résultat

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
|    2 | stack    | stack    |
+------+----------+----------+
2 rows in set (0.00 sec)

Vous pouvez sélectionner toutes les colonnes d’une table dans une jointure en faisant :

SELECT stack.* FROM stack JOIN Overflow ON stack.id = Overflow.id;

Meilleure pratique N’utilisez pas * sauf si vous déboguez ou récupérez la ou les lignes dans des tableaux associatifs, sinon les modifications de schéma (AJOUTER/supprimer/réorganiser les colonnes) peuvent entraîner de vilaines erreurs d’application. De plus, si vous donnez la liste des colonnes dont vous avez besoin dans votre ensemble de résultats, le planificateur de requêtes de MySQL peut souvent optimiser la requête.

Avantages:

  1. Lorsque vous ajoutez/supprimez des colonnes, vous n’avez pas à apporter de modifications là où vous avez utilisé SELECT *
  2. C’est plus court à écrire
  3. Vous voyez également les réponses, alors l’utilisation de SELECT * peut-elle jamais être justifiée ?

Les inconvénients:

  1. Vous renvoyez plus de données que nécessaire. Supposons que vous ajoutiez une colonne VARBINARY contenant 200 000 par ligne. Vous n’avez besoin de ces données qu’à un seul endroit pour un seul enregistrement - en utilisant SELECT * vous pouvez finir par renvoyer 2 Mo par 10 lignes dont vous n’avez pas besoin
  2. Expliquer quelles données sont utilisées
  3. Spécifier des colonnes signifie que vous obtenez une erreur lorsqu’une colonne est supprimée
  4. Le processeur de requêtes doit faire un peu plus de travail - déterminer quelles colonnes existent sur la table (merci @vinodadhikary)
  5. Vous pouvez trouver où une colonne est utilisée plus facilement
  6. Vous obtenez toutes les colonnes dans les jointures si vous utilisez SELECT *
  7. Vous ne pouvez pas utiliser en toute sécurité le référencement ordinal (bien que l’utilisation de références ordinales pour les colonnes soit une mauvaise pratique en soi)
  8. Dans les requêtes complexes avec des champs ‘TEXT’, la requête peut être ralentie par un traitement de table temporaire moins optimal

SELECT par nom de colonne #

CREATE TABLE stack(
    id INT,
    username VARCHAR(30) NOT NULL,
    password VARCHAR(30) NOT NULL
);

INSERT INTO stack (`id`, `username`, `password`) VALUES (1, 'Foo', 'hiddenGem');
INSERT INTO stack (`id`, `username`, `password`) VALUES (2, 'Baa', 'verySecret');

Requête

SELECT id FROM stack;

Résultat

+------+
| id   |
+------+
|    1 |
|    2 |
+------+

SELECT avec LIKE (%) #

CREATE TABLE stack
(  id int AUTO_INCREMENT PRIMARY KEY,
   username VARCHAR(100) NOT NULL
);

INSERT stack(username) VALUES 
('admin'),('k admin'),('adm'),('a adm b'),('b XadmY c'), ('adm now'), ('not here');

“adm” n’importe où :

SELECT * FROM stack WHERE username LIKE "%adm%";  
+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | k admin   |
|  3 | adm       |
|  4 | a adm b   |
|  5 | b XadmY c |
|  6 | adm now   |
+----+-----------+

Commence par “adm”:

SELECT * FROM stack WHERE username LIKE "adm%";
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
|  3 | adm      |
|  6 | adm now  |
+----+----------+

Se termine par “adm”:

SELECT * FROM stack WHERE username LIKE "%adm"; 
+----+----------+
| id | username |
+----+----------+
|  3 | adm      |
+----+----------+

Tout comme le caractère % dans une clause LIKE correspond à n’importe quel nombre de caractères, le caractère _ correspond à un seul caractère. Par exemple,

SELECT * FROM stack WHERE username LIKE "adm_n"; 
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
+----+----------+

Remarques sur les performances S’il existe un index sur “nom d’utilisateur”, alors

  • LIKE 'adm' fonctionne de la même manière que `= ‘adm’
  • LIKE 'adm% est une “plage”, similaire à BETWEEN..AND.. Il peut faire bon usage d’un index sur la colonne.
  • LIKE '%adm' (ou toute variante avec un caractère générique leading) ne peut utiliser aucun index. Ce sera donc lent. Sur les tables avec de nombreuses lignes, il est susceptible d’être si lent qu’il est inutile.
  • RLIKE (REGEXP) a tendance à être plus lent que LIKE, mais a plus de capacités.
  • Alors que MySQL propose une indexation FULLTEXT sur de nombreux types de tables et de colonnes, ces index FULLTEXT ne sont pas utilisés pour répondre aux requêtes utilisant LIKE.

SELECT avec CASE ou IF #

Requête

SELECT st.name,
       st.percentage, 
       CASE WHEN st.percentage >= 35 THEN 'Pass' ELSE 'Fail' END AS `Remark` 
FROM student AS st ;

Résultat

+--------------------------------+
|   name   | percentage | Remark |
+--------------------------------+
|   Isha   |     67     |  Pass  |
|   Rucha  |     28     |  Fail  |
|   Het    |     35     |  Pass  |
|   Ansh   |     92     |  Pass  |
+--------------------------------+

Ou avec SI

SELECT st.name,
       st.percentage, 
       IF(st.percentage >= 35, 'Pass', 'Fail') AS `Remark` 
FROM student AS st ;

N.B

IF(st.percentage >= 35, 'Pass', 'Fail')

Cela signifie : IF st.percentage >= 35 est TRUE then return 'Pass' ELSE renvoie 'Échec'

SELECT avec alias (AS) #

Les alias SQL permettent de renommer temporairement une table ou une colonne. Ils sont généralement utilisés pour améliorer la lisibilité.

Requête

SELECT username AS val FROM stack; 
SELECT username val FROM stack;

(Remarque : AS est syntaxiquement facultatif.)

Résultat

+-------+
| val   |
+-------+
| admin |
| stack |
+-------+
2 rows in set (0.00 sec)

SELECT avec une clause LIMIT #

Requête:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 3;

Résultat:

CustomerID CustomerName ContactName Address City PostalCode Country 1

Alfreds Futterkiste Maria Anders Obere Str. 57 Berlin 12209 Germany 2 Ana Trujillo Emparedados y helados Ana Trujillo Avda. de la Constitución 2222 México D.F. 05021 Mexico 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Meilleure pratique Utilisez toujours ORDER BY lorsque vous utilisez LIMIT ; sinon les lignes que vous obtiendrez seront imprévisibles.

Requête:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 2,1;

Explication:

Lorsqu’une clause LIMIT contient deux nombres, elle est interprétée comme LIMIT offset,count. Ainsi, dans cet exemple, la requête ignore deux enregistrements et en renvoie un.

Résultat:

CustomerID CustomerName ContactName Address City PostalCode Country 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Noter:

Les valeurs des clauses LIMIT doivent être des constantes ; il ne peut s’agir de valeurs de colonne.

Requête

SELECT * FROM stack WHERE username = "admin" AND password = "admin";

Résultat

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
+------+----------+----------+
1 row in set (0.00 sec)

Requête avec un SELECT imbriqué dans la clause WHERE #

La clause WHERE peut contenir n’importe quelle instruction SELECT valide pour écrire des requêtes plus complexes. Il s’agit d’une requête “imbriquée”

Requête

Les requêtes imbriquées sont généralement utilisées pour renvoyer des valeurs atomiques uniques à partir de requêtes à des fins de comparaison.

SELECT title FROM books WHERE author_id = (SELECT id FROM authors WHERE last_name = 'Bar' AND first_name = 'Foo');

Sélectionne tous les noms d’utilisateur sans adresse e-mail

SELECT * FROM stack WHERE username IN (SELECT username FROM signups WHERE email IS NULL);

Avis de non-responsabilité : envisagez d’utiliser [joins][1] pour améliorer les performances lors de la comparaison d’un ensemble de résultats complet.

[1] : http://stackoverflow.com/questions/17946221/sql-join-and-different-types-of-joins

SÉLECTIONNER avec ENTRE #

Vous pouvez utiliser la clause BETWEEN pour remplacer une combinaison de conditions “supérieur à égal ET inférieur à égal”.

Données

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
|  6 | java      |
+----+-----------+

Requête avec les opérateurs

SELECT * FROM stack WHERE id >= 2 and id <= 5;

Requête similaire avec BETWEEN

SELECT * FROM stack WHERE id BETWEEN 2 and 5;

Résultat

+----+-----------+
| id | username  |
+----+-----------+
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
+----+-----------+
4 rows in set (0.00 sec)

Noter

BETWEEN utilise >= et <=, et non > et <.

Utiliser PAS ENTRE

Si vous voulez utiliser le négatif, vous pouvez utiliser NOT. Par exemple :

SELECT * FROM stack WHERE id NOT BETWEEN 2 and 5;

Résultat

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  6 | java      |
+----+-----------+
2 rows in set (0.00 sec)

Noter

NOT BETWEEN utilise > et < et non >= et <= Autrement dit, WHERE id NOT BETWEEN 2 and 5 est identique à WHERE (id < 2 OR id > 5).

Si vous avez un index sur une colonne que vous utilisez dans une recherche BETWEEN, MySQL peut utiliser cet index pour une analyse de plage.

SÉLECTIONNER avec LIKE(_) #

Un caractère _ dans un modèle de clause LIKE correspond à un seul caractère.

Requête

SELECT username FROM users WHERE users LIKE 'admin_';

Résultat

+----------+
| username |  
+----------+
| admin1   |
| admin2   |
| admin-   |
| adminA   |
+----------+

SELECT avec plage de dates #

SELECT ... WHERE dt >= '2017-02-01'
             AND dt  < '2017-02-01' + INTERVAL 1 MONTH

Bien sûr, cela pourrait être fait avec BETWEEN et l’inclusion de 23:59:59. Mais, le modèle a ces avantages:

  • Vous n’avez pas à pré-calculer la date de fin (qui est souvent une durée exacte à partir du début)
  • Vous n’incluez pas les deux points de terminaison (comme le fait BETWEEN), ni tapez ‘23:59:59’ pour l’éviter.
  • Cela fonctionne pour DATE, TIMESTAMP, DATETIME et même le DATETIME (6) inclus en microsecondes.
  • Il prend en charge les jours bissextiles, la fin d’année, etc.
  • Il est compatible avec les index (tout comme BETWEEN).
