Syntaxe #

  • – Créer un index simple

CREATE INDEX nom_index ON nom_table(nom_colonne1 [, nom_colonne2, …])

  • – Créer un index unique

CREATE UNIQUE INDEX nom_index ON nom_table(nom_colonne1 [, nom_colonne2, …]

  • – Indice de chute

DROP INDEX index_name ON tbl_name [algorithm_option | lock_option] …

option_algorithme : ALGORITHM [=] {DEFAULT|INPLACE|COPY}

option_verrouillage : LOCK [=] {DEFAULT|NONE|SHARED|EXCLUSIVE}

Concepts #

Un index dans une table MySQL fonctionne comme un index dans un livre.

Supposons que vous ayez un livre sur les bases de données et que vous souhaitiez trouver des informations sur, par exemple, le stockage. Sans index (en supposant qu’il n’y ait pas d’autre aide, telle qu’une table des matières), vous devriez parcourir les pages une par une, jusqu’à ce que vous trouviez le sujet (c’est une “analyse complète de la table”). D’un autre côté, un index a une liste de mots-clés, donc vous consulteriez l’index et verriez que le stockage est mentionné aux pages 113-120, 231 et 354. Ensuite, vous pourriez retourner directement à ces pages, sans chercher (c’est une recherche avec un index, un peu plus rapide).

Bien sûr, l’utilité de l’index dépend de beaucoup de choses - quelques exemples, en utilisant la comparaison ci-dessus :

  • Si vous aviez un livre sur les bases de données et que vous indexiez le mot “base de données”, vous pourriez voir qu’il est mentionné aux pages 1-59, 61-290 et 292-400. Cela fait beaucoup de pages, et dans un tel cas, l’index n’est pas d’une grande aide et il peut être plus rapide de parcourir les pages une par une. (Dans une base de données, il s’agit d’une “mauvaise sélectivité”.)
  • Pour un livre de 10 pages, cela n’a aucun sens de faire un index, car vous pourriez vous retrouver avec un livre de 10 pages précédé d’un index de 5 pages, ce qui est tout simplement idiot - il suffit de scanner les 10 pages et d’en finir avec ce.
  • L’index doit également être utile - il est généralement inutile d’indexer, par exemple, la fréquence de la lettre “L” par page.

Créer un index #

-- Create an index for column 'name' in table 'my_table'
CREATE INDEX idx_name ON my_table(name);

Créer un index unique #

Un index unique empêche l’insertion de données dupliquées dans une table. Les valeurs NULL peuvent être insérées dans les colonnes qui font partie de l’index unique (puisque, par définition, une valeur NULL est différente de toute autre valeur, y compris une autre valeur NULL)

-- Creates a unique index for column 'name' in table 'my_table'
CREATE UNIQUE INDEX idx_name ON my_table(name);

Supprimer l’index #

-- Drop an index for column 'name' in table 'my_table'
DROP INDEX idx_name ON my_table;

Créer un index composite #

Cela créera un index composite des deux clés, mystring et mydatetime et accélérera les requêtes avec les deux colonnes dans la clause WHERE.

CREATE INDEX idx_mycol_myothercol ON my_table(mycol, myothercol)

Remarque : L’ordre est important ! Si la requête de recherche n’inclut pas les deux colonnes dans la clause “WHERE”, elle ne peut utiliser que l’index le plus à gauche. Dans ce cas, une requête avec mycol dans WHERE utilisera l’index, une requête recherchant myothercol sans également rechercher mycol ne le fera pas. Pour plus d’informations [consultez cet article de blog][1].

Remarque : En raison du fonctionnement de BTREE, les colonnes qui sont généralement interrogées dans des plages doivent être placées dans la valeur la plus à droite. Par exemple, les colonnes DATETIME sont généralement interrogées comme WHERE datecol > '2016-01-01 00:00:00'. Les index BTREE gèrent les plages de manière très efficace, mais uniquement si la colonne interrogée en tant que plage est la dernière de l’index composite.

[1] : https://www.percona.com/blog/2009/06/05/a-rule-of-thumb-for-choosing-column-order-in-indexes/

Touche AUTO_INCREMENT #

CREATE TABLE (
    id INT UNSIGNED NOT NULL AUTO_INCREMENT,
    ...
    PRIMARY KEY(id),
    ...  );

Remarques principales :

  • Commence par 1 et incrémente de 1 automatiquement lorsque vous ne le spécifiez pas sur INSERT, ou que vous le spécifiez comme NULL.
  • Les identifiants sont toujours distincts les uns des autres, mais…
  • Ne faites aucune hypothèse (pas d’espace, généré consécutivement, non réutilisé, etc.) sur les valeurs de l’identifiant autre que d’être unique à un instant donné.

Notes subtiles :

  • Au redémarrage du serveur, la valeur ’next’ est ‘calculée’ comme MAX(id)+1.
  • Si la dernière opération avant l’arrêt ou le crash consistait à supprimer l’identifiant le plus élevé, cet identifiant peut être réutilisé (cela dépend du moteur). Donc, ne faites pas confiance aux auto_incréments pour être uniques en permanence ; ils ne sont uniques qu’à tout moment.
  • Pour les solutions multi-maîtres ou en cluster, voir auto_increment_offset et auto_increment_increment.
  • Il est acceptable d’avoir quelque chose d’autre comme ‘CLE PRIMAIRE’ et de faire simplement ‘INDEX(id)’. (Il s’agit d’une optimisation dans certaines situations.)
  • L’utilisation de AUTO_INCREMENT comme “clé PARTITION” est rarement bénéfique ; faire quelque chose de différent.
  • Diverses opérations peuvent “graver” les valeurs. Cela se produit lorsqu’ils pré-allouent des valeurs, puis ne les utilisent pas : INSERT IGNORE (avec la touche dup), REPLACE (qui est DELETE plus INSERT) et autres. ROLLBACK est une autre cause de lacunes dans les identifiants.
  • Dans la réplication, vous ne pouvez pas faire confiance aux identifiants pour arriver aux esclaves dans l’ordre croissant. Bien que les identifiants soient attribués dans un ordre consécutif, les instructions InnoDB sont envoyées aux esclaves dans l’ordre “COMMIT”.
