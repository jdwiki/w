Erreur 1055 ONLY_FULL_GROUP_BY quelque chose n'est pas dans la clause GROUP BY...

Récemment, les nouvelles versions des serveurs MySQL ont commencé à générer des erreurs 1055 pour les requêtes qui fonctionnaient auparavant. Cette rubrique explique ces erreurs.

L’équipe MySQL a travaillé pour retirer l’extension non standard de GROUP BY, ou au moins pour rendre plus difficile pour les développeurs d’écriture de requêtes d’être brûlés par elle.

Depuis longtemps, MySQL contient une extension non standard notoire pour GROUP BY, qui permet un comportement bizarre au nom de l’efficacité. Cette extension a permis à d’innombrables développeurs du monde entier d’utiliser GROUP BY dans le code de production sans comprendre complètement ce qu’ils faisaient.

En particulier, c’est une mauvaise idée d’utiliser SELECT * dans une requête GROUP BY, car une clause GROUP BY standard nécessite l’énumération des colonnes. De nombreux développeurs l’ont malheureusement fait.

Lis ça. https://dev.mysql.com/doc/refman/5.7/en/group-by-handling.html

L’équipe MySQL a essayé de corriger cette erreur sans perturber le code de production. Ils ont ajouté un indicateur sql_mode dans 5.7.5 nommé [ONLY_FULL_GROUP_BY][1] pour obliger le comportement standard. Dans une version récente, ils ont activé ce drapeau par défaut. Lorsque vous avez mis à niveau votre MySQL local vers la version 5.7.14, le drapeau a été activé et votre code de production, dépendant de l’ancienne extension, a cessé de fonctionner.

Si vous avez récemment commencé à recevoir des erreurs 1055, quels sont vos choix ?

  1. corrigez les requêtes SQL incriminées ou demandez à leurs auteurs de le faire.
  2. Revenez à une version de MySQL compatible prête à l’emploi avec le logiciel d’application que vous utilisez.
  3. Modifiez le [sql_mode][1] de votre serveur pour vous débarrasser du mode ONLY_FULL_GROUP_BY nouvellement défini.

Vous pouvez changer le mode en exécutant une commande SET.

SET  sql_mode = 'STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'

devrait faire l’affaire si vous le faites juste après que votre application se connecte à MySQL.

Ou, vous pouvez trouver [le fichier init dans votre installation MySQL][2], localiser la ligne sql_mode= et la modifier pour omettre ONLY_FULL_GROUP_BY et redémarrer votre serveur.

[1] : http://dev.mysql.com/doc/refman/5.7/en/sql-mode.html [2] : http://dev.mysql.com/doc/refman/5.7/en/server-configuration-defaults.html

Utilisation abusive de GROUP BY pour renvoyer des résultats imprévisibles : loi de Murphy #

SELECT item.item_id, uses.category,   /* nonstandard */ 
       COUNT(*) number_of_uses 
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

affichera les lignes dans une table appelée item et affichera le nombre de lignes associées dans une table appelée uses. Il affichera également la valeur d’une colonne appelée “uses.category”.

Cette requête fonctionne dans MySQL (avant l’apparition du drapeau ONLY_FULL_GROUP_BY). Il utilise [l’extension non standard de MySQL pour GROUP BY][1].

Mais la requête a un problème : si plusieurs lignes de la table uses correspondent à la condition ON dans la clause JOIN, MySQL renvoie la colonne category à partir d’une seule de ces lignes. Quelle rangée ? L’auteur de la requête et l’utilisateur de l’application ne le savent pas à l’avance. Formellement parlant, c’est imprévisible : MySQL peut retourner n’importe quelle valeur qu’il veut.

Imprévisible est comme aléatoire, avec une différence significative. On pourrait s’attendre à ce qu’un choix * aléatoire * change de temps en temps. Par conséquent, si un choix était aléatoire, vous pourriez le détecter lors du débogage ou des tests. Le résultat imprévisible est pire : MySQL renvoie le même résultat à chaque fois que vous utilisez la requête, jusqu’à ce que ce ne soit plus le cas. Parfois, c’est une nouvelle version du serveur MySQL qui provoque un résultat différent. Parfois, c’est une table de plus en plus à l’origine du problème. Ce qui peut mal tourner, ira mal, et quand vous ne vous y attendez pas. C’est ce qu’on appelle la [loi de Murphy][2].

L’équipe MySQL s’est efforcée de rendre cette erreur plus difficile pour les développeurs. Les nouvelles versions de MySQL dans la séquence 5.7 ont un drapeau sql_mode appelé ONLY_FULL_GROUP_BY. Lorsque cet indicateur est défini, le serveur MySQL renvoie l’erreur 1055 et refuse d’exécuter ce type de requête.

[1] : https://dev.mysql.com/doc/refman/5.7/en/group-by-handling.html [2] : https://en.wikipedia.org/wiki/Murphy’s_law

Utilisation abusive de GROUP BY avec SELECT * et comment y remédier. #

Parfois, une requête ressemble à ceci, avec un * dans la clause SELECT.

 SELECT item.*,     /* nonstandard */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

Une telle requête doit être refactorisée pour se conformer à la norme ONLY_FULL_GROUP_BY.

Pour ce faire, nous avons besoin d’une sous-requête qui utilise GROUP BY correctement pour renvoyer la valeur number_of_uses pour chaque item_id. Cette sous-requête est courte et simple, car elle n’a besoin que de regarder la table “uses”.

                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id

Ensuite, nous pouvons joindre cette sous-requête à la table item.

 SELECT item.*, usecount.number_of_uses
   FROM item
   JOIN (
                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id
        ) usecount ON item.item_id = usecount.item_id

Cela permet à la clause GROUP BY d’être simple et correcte, et nous permet également d’utiliser le spécificateur *.

Remarque : néanmoins, les développeurs avisés évitent d’utiliser le spécificateur * dans tous les cas. Il est généralement préférable de répertorier les colonnes souhaitées dans une requête.

Utiliser et abuser de GROUP BY #

 SELECT item.item_id, item.name,     /* not SQL-92 */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

affichera les lignes dans une table appelée item et affichera le nombre de lignes associées dans une table appelée uses. Cela fonctionne bien, mais malheureusement ce n’est pas le standard SQL-92.

Pourquoi pas? parce que la clause SELECT (et la clause ORDER BY) dans les requêtes GROUP BY doivent contenir des colonnes qui sont

  1. mentionné dans la clause GROUP BY, ou
  2. fonctions d’agrégation telles que COUNT(), MIN(), etc.

La clause SELECT de cet exemple mentionne item.name, une colonne qui ne répond à aucun de ces critères. MySQL 5.6 et versions antérieures rejetteront cette requête si le mode SQL contient ONLY_FULL_GROUP_BY.

Cet exemple de requête peut être rendu conforme à la norme SQL-92 en modifiant la clause GROUP BY, comme ceci.

SELECT item.item_id, item.name, 
       COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id, item.name

La dernière norme SQL-99 permet à une instruction SELECT d’omettre des colonnes non agrégées de la clé de groupe si le SGBD peut prouver une dépendance fonctionnelle entre elles et les colonnes de clé de groupe. Étant donné que item.name dépend fonctionnellement de item.item_id, l’exemple initial est valide SQL-99. MySQL a obtenu un prouveur de dépendance fonctionnelle dans la version 5.7. L’exemple d’origine fonctionne sous ONLY_FULL_GROUP_BY.

DE N’IMPORTE QUELLE VALEUR() #

 SELECT item.item_id, ANY_VALUE(uses.tag) tag,   
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

affiche les lignes d’une table appelée “item”, le nombre de lignes associées et l’une des valeurs de la table associée appelée “uses”.

Vous pouvez considérer [cette fonction ANY_VALUE()][1] comme une étrange sorte de fonction d’agrégation. Au lieu de renvoyer un nombre, une somme ou un maximum, il demande au serveur MySQL de choisir, arbitrairement, une valeur du groupe en question. C’est une façon de contourner l’erreur 1055.

Soyez prudent lorsque vous utilisez ANY_VALUE() dans des requêtes dans des applications de production.

Il devrait vraiment s’appeler SURPRISE_ME(). Il renvoie la valeur d’une ligne dans le groupe GROUP BY. La ligne renvoyée est indéterminée. Cela signifie que cela dépend entièrement du serveur MySQL. Formellement, il renvoie une valeur imprévisible.

Le serveur ne choisit pas une valeur aléatoire, c’est pire que ça. Il renvoie la même valeur chaque fois que vous exécutez la requête, jusqu’à ce que ce ne soit pas le cas. Cela peut changer, ou non, lorsqu’une table grossit ou rétrécit, ou lorsque le serveur a plus ou moins de RAM, ou lorsque la version du serveur change, ou lorsque Mars est rétrograde (quoi que cela signifie), ou sans aucune raison.

Tu étais prévenu.

[1] : http://dev.mysql.com/doc/refman/5.7/en/miscellaneous-functions.html#function_any-value
