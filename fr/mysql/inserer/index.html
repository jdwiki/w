Syntaxe #

  1. INSÉREZ [PRIORITÉ_BASSE | RETARDÉ | HIGH_PRIORITY] [IGNORER] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] {VALUES | VALUE} ({expr | DEFAULT},…),(…),… [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  2. INSÉRER [LOW_PRIORITY | RETARDÉ | HIGH_PRIORITY] [IGNORER] [INTO] tbl_name [PARTITION (partition_name,…)] SET col_name={expr | DEFAULT}, … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  3. INSÉRER [LOW_PRIORITY | HIGH_PRIORITY] [IGNORER] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  4. Une expression expr peut faire référence à n’importe quelle colonne définie précédemment dans une liste de valeurs. Par exemple, vous pouvez le faire car la valeur de col2 fait référence à col1, qui a déjà été affectée : INSERT INTO tbl_name (col1,col2) VALUES(15,col1*2);

  5. Les instructions INSERT qui utilisent la syntaxe VALUES peuvent insérer plusieurs lignes. Pour ce faire, incluez plusieurs listes de valeurs de colonne, chacune entre parenthèses et séparées par des virgules. Exemple: INSERT INTO tbl_name (a,b,c) VALUES(1,2,3),(4,5,6),(7,8,9);

  6. La liste des valeurs de chaque ligne doit être placée entre parenthèses. L’instruction suivante est illégale car le nombre de valeurs dans la liste ne correspond pas au nombre de noms de colonnes : INSERT INTO tbl_name (a,b,c) VALUES(1,2,3,4,5,6,7,8,9);

  7. [INSERER … SELECT Syntaxe][1] INSÉRER [LOW_PRIORITY | HIGH_PRIORITY] [IGNORER] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr, … ]

  8. Avec INSERT … SELECT, vous pouvez rapidement insérer plusieurs lignes dans une table à partir d’une ou plusieurs tables. Par exemple: INSÉRER DANS tbl_temp2 (fld_id) SELECT tbl_temp1.fld_order_id FROM tbl_temp1 WHERE tbl_temp1.fld_order_id > 100 ;

[1] : http://dev.mysql.com/doc/refman/5.7/en/insert-select.html

[Syntaxe officielle INSERT][1]

[1] : http://dev.mysql.com/doc/refman/5.7/en/insert.html

INSÉRER, SUR LA MISE À JOUR DE LA CLÉ EN DOUBLE #

INSERT INTO `table_name` 
    (`index_field`, `other_field_1`, `other_field_2`) 
    VALUES 
    ('index_value', 'insert_value', 'other_value') 
    ON DUPLICATE KEY UPDATE 
        `other_field_1` = 'update_value',
        `other_field_2` = VALUES(`other_field_2`);

Cela INSERT dans table_name les valeurs spécifiées, mais si la clé unique existe déjà, il mettra à jour le other_field_1 pour avoir une nouvelle valeur. Parfois, lors de la mise à jour sur une clé en double, il est pratique d’utiliser [VALUES()][1] afin d’accéder à la valeur d’origine qui a été transmise à INSERT au lieu de définir la valeur directement. De cette façon, vous pouvez définir différentes valeurs en utilisant INSERT et UPDATE. Voir l’exemple ci-dessus où other_field_1 est défini sur insert_value sur INSERT ou sur update_value sur UPDATE tandis que other_field_2 est toujours défini sur other_value.

Le schéma contenant une clé unique qui signalera un conflit en double est crucial pour que l’insertion sur la mise à jour de clé en double (IODKU) fonctionne. Cette clé unique peut être une clé primaire ou non. Il peut s’agir d’une clé unique sur une seule colonne, ou d’une multi-colonne (clé composite).

[1] : http://dev.mysql.com/doc/refman/5.7/en/miscellaneous-functions.html#function_values

Insertion de plusieurs lignes #

INSERT INTO `my_table` (`field_1`, `field_2`) VALUES 
    ('data_1', 'data_2'),
    ('data_1', 'data_3'),
    ('data_4', 'data_5');

C’est un moyen facile d’ajouter plusieurs lignes à la fois avec une seule instruction INSERT.

Ce type d’insertion “par lots” est beaucoup plus rapide que l’insertion de lignes une par une. En règle générale, l’insertion de 100 lignes dans une seule insertion par lot de cette manière est 10 fois plus rapide que de les insérer toutes individuellement.

Ignorer les lignes existantes #

Lors de l’importation de grands ensembles de données, il peut être préférable, dans certaines circonstances, d’ignorer les lignes qui entraîneraient généralement l’échec de la requête en raison d’une restriction de colonne, par ex. clés primaires en double. Cela peut être fait en utilisant INSERT IGNORE.

Considérez l’exemple de base de données suivant :

SELECT * FROM `people`;
--- Produces:
+----+------+
| id | name |
+----+------+
|  1 | john |
|  2 | anna |
+----+------+


INSERT IGNORE INTO `people` (`id`, `name`) VALUES
    ('2', 'anna'), --- Without the IGNORE keyword, this record would produce an error
    ('3', 'mike');

SELECT * FROM `people`;
--- Produces:
+----+--------+
| id |  name  |
+----+--------+
|  1 |  john  |
|  2 |  anna  |
|  3 |  mike  |
+----+--------+

La chose importante à retenir est que INSERT IGNORE ignorera également silencieusement d’autres erreurs, voici ce que disent les documentations officielles de Mysql :

Les conversions de données qui déclencheraient des erreurs interrompent l’instruction si IGNORE n’est > pas spécifié. Avec IGNORE, les valeurs invalides sont ajustées aux valeurs les plus proches et >insérées ; des avertissements sont produits mais l’instruction n’est pas interrompue.

Remarque :- La section ci-dessous est ajoutée par souci d’exhaustivité, mais n’est pas considérée comme la meilleure pratique (cela échouerait, par exemple, si une autre colonne était ajoutée au tableau).

Si vous spécifiez la valeur de la colonne correspondante pour toutes les colonnes de la table, vous pouvez ignorer la liste des colonnes dans l’instruction INSERT comme suit :

INSERT INTO `my_table` VALUES 
        ('data_1', 'data_2'),
        ('data_1', 'data_3'),
        ('data_4', 'data_5');

Insertion de base #

INSERT INTO `table_name` (`field_one`, `field_two`) VALUES ('value_one', 'value_two');

Dans cet exemple trivial, table_name est l’endroit où les données doivent être ajoutées, field_one et field_two sont des champs pour définir les données, et value_one et value_two sont les données à faire contre field_one et field_two respectivement.

C’est une bonne pratique de répertorier les champs dans lesquels vous insérez des données dans votre code, comme si la table change et que de nouvelles colonnes sont ajoutées, votre insertion se briserait si elles n’étaient pas là

INSERT avec AUTO_INCREMENT + LAST_INSERT_ID() #

Lorsqu’une table a une AUTO_INCREMENT PRIMARY KEY, normalement on n’insère pas dans cette colonne. Au lieu de cela, spécifiez toutes les autres colonnes, puis demandez quel était le nouvel identifiant.

CREATE TABLE t (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    this ...,
    that ...,
    PRIMARY KEY(id) );

INSERT INTO t (this, that) VALUES (..., ...);
SELECT LAST_INSERT_ID() INTO @id;
INSERT INTO another_table (..., t_id, ...) VALUES (..., @id, ...);

Notez que LAST_INSERT_ID() est lié à la session, donc même si plusieurs connexions s’insèrent dans la même table, chacune obtient son propre identifiant.

Votre API client a probablement un autre moyen d’obtenir le LAST_INSERT_ID() sans effectuer réellement un SELECT et de renvoyer la valeur au client au lieu de la laisser dans une @variable à l’intérieur de MySQL. Tel est généralement préférable.

Exemple plus long et plus détaillé

L’utilisation “normale” d’IODKU consiste à déclencher une “clé dupliquée” basée sur une clé “UNIQUE”, et non sur la “CLÉ PRIMAIRE AUTO_INCREMENT”. Ce qui suit le démontre. Notez qu’il ne fournit pas l’id dans l’INSERT.

Configuration pour les exemples à suivre :

CREATE TABLE iodku (
    id INT AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    misc INT NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
) ENGINE=InnoDB;

INSERT INTO iodku (name, misc)
    VALUES
    ('Leslie', 123),
    ('Sally', 456);
Query OK, 2 rows affected (0.00 sec)
Records: 2  Duplicates: 0  Warnings: 0
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  |  456 |
+----+--------+------+

Le cas où IODKU effectue une “mise à jour” et LAST_INSERT_ID() récupère l’id pertinent :

INSERT INTO iodku (name, misc)
    VALUES
    ('Sally', 3333)            -- should update
    ON DUPLICATE KEY UPDATE    -- `name` will trigger "duplicate key"
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();       -- picking up existing value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                2 |
+------------------+

Le cas où IODKU effectue un “insert” et LAST_INSERT_ID() récupère le nouvel id :

INSERT INTO iodku (name, misc)
    VALUES
    ('Dana', 789)          -- Should insert
    ON DUPLICATE KEY UPDATE
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();   -- picking up new value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                3 |
+------------------+

Contenu du tableau résultant :

SELECT * FROM iodku;
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  | 3333 |  -- IODKU changed this
|  3 | Dana   |  789 |  -- IODKU added this
+----+--------+------+

INSERT SELECT (Insérer des données d’une autre table) #

C’est la méthode de base pour insérer des données d’une autre table avec l’instruction SELECT.

INSERT INTO `tableA` (`field_one`, `field_two`) 
   SELECT `tableB`.`field_one`, `tableB`.`field_two` 
   FROM `tableB` 
   WHERE `tableB`.clmn <> 'someValue'
   ORDER BY `tableB`.`sorting_clmn`;

Vous pouvez SELECT * FROM, mais alors tableA et tableB doivent avoir un nombre de colonnes correspondant et des types de données correspondants.

Les colonnes avec AUTO_INCREMENT sont traitées comme dans la clause INSERT avec VALUES.

Cette syntaxe permet de remplir facilement des tables (temporaires) avec des données d’autres tables, d’autant plus lorsque les données doivent être filtrées sur l’insert.

Identifiants AUTO_INCREMENT perdus #

Plusieurs fonctions “insert” peuvent “graver” les identifiants. Voici un exemple utilisant InnoDB (d’autres moteurs peuvent fonctionner différemment) :

CREATE TABLE Burn (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
        ) ENGINE=InnoDB;

INSERT IGNORE INTO Burn (name) VALUES ('first'), ('second');
SELECT LAST_INSERT_ID();          -- 1
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('second');  -- dup 'IGNOREd', but id=3 is burned
SELECT LAST_INSERT_ID();          -- Still "1" -- can't trust in this situation
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('third');
SELECT LAST_INSERT_ID();           -- now "4"
SELECT * FROM Burn ORDER BY id;    -- note that id=3 was skipped over
  +----+--------+
  |  1 | first  |
  |  2 | second |
  |  4 | third  |    -- notice that id=3 has been 'burned'
  +----+--------+

Pensez-y (à peu près) de cette façon : d’abord, l’insert regarde pour voir combien de lignes pourraient être insérées. Ensuite, récupérez autant de valeurs de l’auto_increment pour cette table. Enfin, insérez les lignes, en utilisant les identifiants nécessaires et en brûlant les restes.

La seule fois où les restes sont récupérables, c’est si le système est arrêté et redémarré. Au redémarrage, effectivement MAX(id) est exécuté. Cela peut réutiliser les identifiants qui ont été brûlés ou qui ont été libérés par les “DELETE” du ou des identifiants les plus élevés.

Essentiellement, n’importe quelle variante de INSERT (y compris REPLACE, qui est DELETE + INSERT) peut graver des identifiants. Dans InnoDB, la variable globale (pas de session !) [innodb_autoinc_lock_mode] [1] peut être utilisée pour contrôler une partie de ce qui se passe.

Lors de la “normalisation” de longues chaînes dans un ID AUTO INCREMENT, la gravure peut facilement se produire. Cela pourrait entraîner un dépassement de la taille de l’“INT” que vous avez choisi.

[1] : https://dev.mysql.com/doc/refman/5.7/en/innodb-auto-increment-handling.html
