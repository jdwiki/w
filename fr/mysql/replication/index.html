Réplication

  • Configuration de la réplication maître-esclave
  • Erreurs de réplication

Sur cette page

  • Configuration de la réplication maître-esclave

La réplication est utilisée pour copier les données [de sauvegarde] d’un serveur de base de données MySQL vers un ou plusieurs serveurs de base de données MySQL.

Maître – Le serveur de base de données MySQL, qui sert les données à copier

Esclave – Le serveur de base de données MySQL, copie les données qui sont servies par le maître

Avec MySQL, la réplication est asynchrone par défaut. Cela signifie que les esclaves n’ont pas besoin d’être connectés en permanence pour recevoir les mises à jour du maître. Par exemple, si votre esclave est éteint ou n’est pas connecté au maître et que vous allumez l’esclave ou que vous vous connectez au maître ultérieurement, il se synchronisera automatiquement avec le maître.

Selon la configuration, vous pouvez répliquer toutes les bases de données, les bases de données sélectionnées ou même les tables sélectionnées dans une base de données.

Formats de réplication

Il existe deux principaux types de formats de réplication

Statement Based Replication (SBR) – qui réplique des instructions SQL entières. Dans ce cas, le maître écrit des instructions SQL dans le journal binaire. La réplication du maître vers l’esclave fonctionne en exécutant ces instructions SQL sur l’esclave.

Row Based Replication (RBR) – qui réplique uniquement les lignes modifiées. Dans ce cas, le maître écrit des événements dans le journal binaire qui indiquent comment les lignes de table individuelles sont modifiées. La réplication du maître vers l’esclave fonctionne en copiant les événements représentant les modifications apportées aux lignes de la table vers l’esclave.

Vous pouvez également utiliser une troisième variété, Mixed Based Replication (MBR). Dans ce cas, la journalisation basée sur les instructions et basée sur les lignes est utilisée. Le journal sera créé en fonction de celui qui est le plus approprié pour le changement.

Le format basé sur les instructions était la valeur par défaut dans les versions MySQL antérieures à 5.7.7. Dans MySQL 5.7.7 et versions ultérieures, le format basé sur les lignes est la valeur par défaut.

Configuration de la réplication maître-esclave #

Considérez 2 serveurs MySQL pour la configuration de la réplication, l’un est un maître et l’autre est un esclave.

Nous allons configurer le maître pour qu’il conserve un journal de chaque action effectuée sur celui-ci. Nous allons configurer le serveur esclave pour qu’il consulte le journal du maître et chaque fois que des modifications se produisent dans le journal du maître, il doit faire la même chose.

Configuration principale

Tout d’abord, nous devons créer un utilisateur sur le Master. Cet utilisateur va être utilisé par Slave pour créer une connexion avec le Master.

CREATE USER 'user_name'@'%' IDENTIFIED BY 'user_password';
GRANT REPLICATION SLAVE ON *.* TO 'user_name'@'%';
FLUSH PRIVILEGES;

Modifiez user_name et user_password en fonction de votre nom d’utilisateur et de votre mot de passe.

Maintenant, le fichier my.inf (my.cnf sous Linux) doit être modifié. Incluez les lignes suivantes dans la section [mysqld].

server-id = 1
log-bin = mysql-bin.log
binlog-do-db = your_database

La première ligne est utilisée pour attribuer un ID à ce serveur MySQL.

La deuxième ligne indique à MySQL de commencer à écrire un journal dans le fichier journal spécifié. Sous Linux, cela peut être configuré comme log-bin = /home/mysql/logs/mysql-bin.log. Si vous démarrez la réplication sur un serveur MySQL dans lequel la réplication a déjà été utilisée, assurez-vous que ce répertoire est vide de tous les journaux de réplication.

La troisième ligne sert à configurer la base de données pour laquelle nous allons écrire le journal. Vous devez remplacer votre_base de données par le nom de votre base de données.

Assurez-vous que skip-networking n’a pas été activé et redémarrez le serveur MySQL (maître)

Configuration esclave

Le fichier my.inf doit également être modifié dans Slave. Incluez les lignes suivantes dans la section [mysqld].

server-id = 2
master-host = master_ip_address
master-connect-retry = 60

master-user = user_name
master-password = user_password
replicate-do-db = your_database

relay-log = slave-relay.log
relay-log-index = slave-relay-log.index

La première ligne est utilisée pour attribuer un ID à ce serveur MySQL. Cet identifiant doit être unique.

La deuxième ligne est l’adresse IP du serveur maître. Modifiez-le en fonction de votre système maître I.P.

La troisième ligne est utilisée pour définir une limite de tentative en secondes.

Les deux lignes suivantes indiquent le nom d’utilisateur et le mot de passe à l’esclave, à l’aide desquels il se connecte au maître.

La ligne suivante définit la base de données à répliquer.

Les deux dernières lignes sont utilisées pour attribuer les noms de fichier relay-log et relay-log-index.

Assurez-vous que skip-networking n’a pas été activé et redémarrez le serveur MySQL (esclave)

Copier les données vers l’esclave

Si des données sont constamment ajoutées au Master, nous devrons empêcher tout accès à la base de données sur le Master afin que rien ne puisse être ajouté. Ceci peut être réalisé en exécutant l’instruction suivante dans Master.

FLUSH TABLES WITH READ LOCK;

Si aucune donnée n’est ajoutée au serveur, vous pouvez ignorer l’étape ci-dessus.

Nous allons prendre la sauvegarde des données du maître en utilisant mysqldump

mysqldump your_database -u root -p > D://Backup/backup.sql;

Modifiez votre_base de données et le répertoire de sauvegarde en fonction de votre configuration. Vous aurez maintenant un fichier appelé backup.sql à l’emplacement indiqué.

Si votre base de données n’existe pas dans votre esclave, créez-la en exécutant ce qui suit

CREATE DATABASE `your_database`;

Nous devons maintenant importer la sauvegarde dans le serveur Slave MySQL.

mysql -u root -p your_database  <D://Backup/backup.sql
--->Change `your_database` and backup directory according to your setup

Démarrer la réplication

Pour démarrer la réplication, nous devons trouver le nom du fichier journal et la position du journal dans le maître. Alors, exécutez ce qui suit dans Master

SHOW MASTER STATUS;

Cela vous donnera une sortie comme ci-dessous

+---------------------+----------+-------------------------------+------------------+
| File                | Position | Binlog_Do_DB                  | Binlog_Ignore_DB |
+---------------------+----------+-------------------------------+------------------+
| mysql-bin.000001    | 130      | your_database                 |                  |
+---------------------+----------+-------------------------------+------------------+

Ensuite, exécutez ce qui suit dans Slave

SLAVE STOP;
CHANGE MASTER TO MASTER_HOST='master_ip_address', MASTER_USER='user_name', 
   MASTER_PASSWORD='user_password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS=130;
SLAVE START;

Nous arrêtons d’abord l’Esclave. Ensuite, nous lui disons exactement où chercher dans le fichier journal principal. Pour le nom MASTER_LOG_FILE et MASTER_LOG_POS, utilisez les valeurs que nous avons obtenues en exécutant la commande SHOW MASTER STATUS sur le maître.

Vous devez modifier l’adresse IP du maître dans MASTER_HOST et modifier l’utilisateur et le mot de passe en conséquence.

L’esclave attendra maintenant. L’état de l’esclave peut être visualisé en exécutant ce qui suit

SHOW SLAVE STATUS;

Si vous avez précédemment exécuté FLUSH TABLES WITH READ LOCK dans Master, libérez les tables du verrouillage en exécutant ce qui suit

UNLOCK TABLES;

Maintenant, le maître tient un journal pour chaque action effectuée sur lui et le serveur esclave consulte le journal sur le maître. Chaque fois que des changements se produisent dans le journal sur le maître, l’esclave les réplique.

Erreurs de réplication #

Chaque fois qu’il y a une erreur lors de l’exécution d’une requête sur l’esclave, MySQL arrête automatiquement la réplication pour identifier le problème et le résoudre. Ceci principalement parce qu’un événement a provoqué une clé en double ou qu’une ligne n’a pas été trouvée et qu’elle ne peut pas être mise à jour ou supprimée. Vous pouvez ignorer ces erreurs, même si cela n’est pas recommandé

Pour ignorer une seule requête qui suspend l’esclave, utilisez la syntaxe suivante

SET GLOBAL sql_slave_skip_counter = N;

Cette instruction ignore les N événements suivants du maître. Cette instruction n’est valide que lorsque les threads esclaves ne sont pas en cours d’exécution. Sinon, cela produit une erreur.

STOP SLAVE;
SET GLOBAL sql_slave_skip_counter=1;
START SLAVE;

Dans certains cas, c’est très bien. Mais si l’instruction fait partie d’une transaction multi-instructions, elle devient plus complexe, car le fait d’ignorer l’instruction produisant l’erreur entraînera l’omission de toute la transaction.

Si vous voulez ignorer plus de requêtes qui produisent le même code d’erreur et si vous êtes sûr que le fait d’ignorer ces erreurs ne rendra pas votre esclave incohérent et que vous voulez toutes les ignorer, vous devez ajouter une ligne pour ignorer ce code d’erreur dans votre `my. cnf'.

Par exemple, vous voudrez peut-être ignorer toutes les erreurs en double que vous pourriez obtenir

1062 | Error 'Duplicate entry 'xyz' for key 1' on query

Ajoutez ensuite ce qui suit à votre my.cnf

slave-skip-errors = 1062

Vous pouvez également ignorer d’autres types d’erreurs ou tous les codes d’erreur, mais assurez-vous que le fait d’ignorer ces erreurs ne rendra pas votre esclave incohérent. Voici la syntaxe et des exemples

slave-skip-errors=[err_code1,err_code2,...|all]

slave-skip-errors=1062,1053
slave-skip-errors=all
slave-skip-errors=ddl_exist_errors
