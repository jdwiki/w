Code d’erreur 1064 : erreur de syntaxe #

select LastName, FirstName,
from Person

Retourne le message :

Code d’erreur : 1064. Vous avez une erreur dans votre syntaxe SQL ; consultez le manuel qui correspond à la version de votre serveur MySQL pour la bonne syntaxe à utiliser près de ‘from Person’ à la ligne 2.

Obtenir un message “erreur 1064” de MySQL signifie que la requête ne peut pas être analysée sans erreurs de syntaxe. En d’autres termes, il ne peut pas donner de sens à la requête.

La citation dans le message d’erreur commence par le premier caractère de la requête que MySQL ne sait pas comment analyser. Dans cet exemple, MySQL ne peut pas donner de sens, dans le contexte, à from Person. Dans ce cas, il y a une virgule supplémentaire juste avant “from Person”. La virgule indique à MySQL d’attendre une autre description de colonne dans la clause SELECT

Une erreur de syntaxe indique toujours ... près de '...'. La chose au début des guillemets est très proche de l’endroit où se trouve l’erreur. Pour localiser une erreur, regardez le premier jeton entre guillemets et le dernier jeton avant les guillemets.

Parfois, vous obtiendrez ... près de ''; c’est-à-dire rien dans les guillemets. Cela signifie que le premier caractère que MySQL ne peut pas comprendre se trouve juste à la fin ou au début de l’instruction. Cela suggère que la requête contient des guillemets non équilibrés (' ou ") ou des parenthèses non équilibrées ou que vous n’avez pas correctement terminé l’instruction auparavant.

Dans le cas d’une routine stockée, vous avez peut-être oublié d’utiliser correctement DELIMITER.

Ainsi, lorsque vous obtenez l’erreur 1064, examinez le texte de la requête et recherchez le point mentionné dans le message d’erreur. Inspectez visuellement le texte de la requête juste autour de ce point.

Si vous demandez à quelqu’un de vous aider à résoudre l’erreur 1064, il est préférable de fournir à la fois le texte de la requête complète et le texte du message d’erreur.

Code d’erreur 1175 : Mise à jour sécurisée #

Cette erreur apparaît lors de la tentative de mise à jour ou de suppression d’enregistrements sans inclure la clause “WHERE” qui utilise la colonne “KEY”.

Pour exécuter la suppression ou la mise à jour quand même - tapez :

SET SQL_SAFE_UPDATES = 0;

Pour réactiver le mode sans échec, tapez :

SET SQL_SAFE_UPDATES = 1;

Code d’erreur 1215 : Impossible d’ajouter une contrainte de clé étrangère #

Cette erreur se produit lorsque les tables ne sont pas correctement structurées pour gérer la vérification de recherche rapide des exigences de clé étrangère (FK) imposées par le développeur.

CREATE TABLE `gtType` (
  `type` char(2) NOT NULL,
  `description` varchar(1000) NOT NULL,
  PRIMARY KEY (`type`)
) ENGINE=InnoDB;

CREATE TABLE `getTogethers` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `type` char(2) NOT NULL,
  `eventDT` datetime NOT NULL,
  `location` varchar(1000) NOT NULL,
  PRIMARY KEY (`id`),
  KEY `fk_gt2type` (`type`), -- see Note1 below 
  CONSTRAINT `gettogethers_ibfk_1` FOREIGN KEY (`type`) REFERENCES `gtType` (`type`)
) ENGINE=InnoDB;

Remarque1 : une clé comme celle-ci sera créée automatiquement si nécessaire en raison de la définition FK dans la ligne qui le suit. Le développeur peut l’ignorer, et la CLÉ (a.k.a. index) sera ajoutée si nécessaire. Un exemple d’omission par le développeur est illustré ci-dessous dans someOther.

Jusqu’ici tout va bien, jusqu’à l’appel ci-dessous.

CREATE TABLE `someOther` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `someDT` datetime NOT NULL,
  PRIMARY KEY (`id`),
  CONSTRAINT `someOther_dt` FOREIGN KEY (`someDT`) REFERENCES `getTogethers` (`eventDT`)
) ENGINE=InnoDB;

Code d’erreur : 1215. Impossible d’ajouter une contrainte de clé étrangère

Dans ce cas, il échoue en raison de l’absence d’index dans la table * référencée * getTogethers à gérer la recherche rapide d’un eventDT. À résoudre dans la prochaine déclaration.

CREATE INDEX `gt_eventdt` ON getTogethers (`eventDT`);

La table getTogethers a été modifiée, et maintenant la création de someOther va réussir.

À partir de la page du manuel MySQL [Utilisation des contraintes FOREIGN KEY][1] :

MySQL nécessite des index sur les clés étrangères et les clés référencées afin que les vérifications de clés étrangères peuvent être rapides et ne nécessitent pas de balayage de table. Dans le table de référencement, il doit y avoir un index où la clé étrangère les colonnes sont répertoriées en tant que premières colonnes dans le même ordre. Un tel l’index est créé automatiquement sur la table de référence s’il ne le fait pas exister.

Les colonnes correspondantes dans la clé étrangère et la clé référencée doivent ont des types de données similaires. La taille et le signe des types entiers doivent être le même. La longueur des types de chaîne n’a pas besoin d’être la même. Pour colonnes de chaînes non binaires (caractères), le jeu de caractères et le classement doit être le même.

InnoDB permet à une clé étrangère de référencer n’importe quelle colonne d’index ou groupe de colonnes. Cependant, dans la table référencée, il doit y avoir un index où les colonnes référencées sont répertoriées en tant que premières colonnes dans le même commande.

Notez que le dernier point ci-dessus concerne les premières colonnes (les plus à gauche) et l’absence d’exigence de clé primaire (bien que fortement conseillée).

Lors de la création réussie d’une table de référencement (enfant), toutes les clés qui ont été automatiquement créées pour vous sont visibles avec une commande telle que la suivante :

SHOW CREATE TABLE someOther;

D’autres cas courants de rencontre de cette erreur incluent, comme mentionné ci-dessus dans la documentation, mais doivent être soulignés :

  • Différences apparemment insignifiantes dans INT qui est signé, pointant vers INT UNSIGNED.

  • Les développeurs ont du mal à comprendre les CLES multi-colonnes (composites) et les premières exigences de commande (la plus à gauche).

[1] : https://dev.mysql.com/doc/refman/5.6/en/create-table-foreign-keys.html

1067, 1292, 1366, 1411 - Mauvaise valeur pour le nombre, la date, la valeur par défaut, etc. #

1067 Ceci est probablement lié aux valeurs par défaut de TIMESTAMP, qui ont changé au fil du temps. Voir Paramètres par défaut de l'horodatage dans la page Dates et heures. (qui n’existe pas encore)

1292/1366 DOUBLE/Entier Vérifiez les lettres ou autres erreurs de syntaxe. Vérifiez que les colonnes sont alignées ; peut-être pensez-vous mettre dans un VARCHAR mais il est aligné avec une colonne numérique.

1292 DATETIME Vérifiez trop loin dans le passé ou le futur. Vérifiez entre 2 h et 3 h du matin un matin où l’heure d’été a changé. Vérifiez la mauvaise syntaxe, telle que + 00 trucs de fuseau horaire.

1292 VARIABLE Vérifiez les valeurs autorisées pour la VARIABLE que vous essayez de SET.

1292 LOAD DATA Regardez la ligne qui est ‘mauvaise’. Vérifiez les symboles d’échappement, etc. Regardez les types de données.

1411 STR_TO_DATE Date au format incorrect ?

1236 “position impossible” dans la réplication #

Généralement, cela signifie que le maître s’est écrasé et que sync_binlog était désactivé. La solution est de CHANGER LE MAITRE en POS=0 du prochain fichier binlog (voir le Maître) sur l’Esclave.

La cause : le maître envoie les éléments de réplication à l’esclave avant de les vider dans son journal binaire (lorsque sync_binlog=OFF). Si le maître plante avant le vidage, l’esclave a déjà logiquement dépassé la fin du fichier sur le binlog. Lorsque le maître redémarre, il démarre un nouveau binlog, donc CHANGER au début de ce binlog est la meilleure solution disponible.

Une solution à plus long terme est sync_binlog=ON, si vous pouvez vous permettre les E/S supplémentaires que cela entraîne.

2002, 2003 Impossible de se connecter #

Recherchez un problème de pare-feu bloquant le port 3306.

Quelques diagnostics et/ou solutions possibles

  • Le serveur fonctionne-t-il réellement ?
  • “service firewalld stop” et “systemctl disable firewalld”
  • maître telnet 3306
  • Vérifiez l’“adresse de liaison”
  • vérifier skip-name-resolve
  • vérifier la prise.

126, 127, 134, 144, 145 #

Lorsque vous essayez d’accéder aux enregistrements de la base de données MySQL, vous pouvez obtenir ces messages d’erreur. Ces messages d’erreur sont dus à la corruption de la base de données MySQL. Voici les types

MySQL error code 126 = Index file is crashed
MySQL error code 127 = Record-file is crashed
MySQL error code 134 = Record was already deleted (or record file crashed)
MySQL error code 144 = Table is crashed and last repair failed
MySQL error code 145 = Table was marked as crashed and should be repaired

Un bogue MySQL, une attaque de virus, un plantage du serveur, un arrêt incorrect, une table endommagée sont à l’origine de cette corruption. Lorsqu’il est corrompu, il devient inaccessible et vous ne pouvez plus y accéder. Afin d’obtenir l’accessibilité, le meilleur moyen de récupérer des données à partir d’une sauvegarde mise à jour. Cependant, si vous n’avez pas de mise à jour ou de sauvegarde valide, vous pouvez opter pour MySQL Repair.

Si le type de moteur de table est MyISAM, appliquez CHECK TABLE, puis REPAIR TABLE.

Pensez ensuite sérieusement à convertir en InnoDB, afin que cette erreur ne se reproduise plus.

Syntaxe

CHECK TABLE <table name> ////To check the extent of database corruption
REPAIR TABLE <table name> ////To repair table

139 #

L’erreur 139 peut signifier que le nombre et la taille des champs dans la définition de table dépassent une certaine limite. Solutions de contournement :

  • Repenser le schéma
  • Normaliser certains champs
  • Partitionner verticalement la table

1366 #

Cela signifie généralement que la gestion du jeu de caractères n’était pas cohérente entre le client et le serveur. Voir … pour plus d’aide.

24 Impossible d’ouvrir le fichier (trop de fichiers ouverts)

open_files_limit provient d’un paramètre du système d’exploitation. table_open_cache doit être inférieur à cela.

Ceux-ci peuvent provoquer cette erreur :

  • Échec de DEALLOCATE PREPARE dans une procédure stockée.

  • Table(s) PARTITIONNÉE(s) avec un grand nombre de partitions et innodb_file_per_table = ON. Recommande de ne pas avoir plus de 50 partitions dans une table donnée (pour diverses raisons). (Lorsque les “partitions natives” deviennent disponibles, ce conseil peut changer.)

La solution de contournement évidente consiste à définir une augmentation de la limite du système d’exploitation : Pour autoriser plus de fichiers, modifiez ulimit ou /etc/security/limits.conf ou dans sysctl.conf (kern.maxfiles & kern.maxfilesperproc) ou autre chose (dépendant du système d’exploitation). Augmentez ensuite open_files_limit et table_open_cache.

Depuis la version 5.6.8, open_files_limit est dimensionné automatiquement en fonction de max_connections, mais il est acceptable de le modifier par défaut.

1062 - Entrée en double

Cette erreur se produit principalement pour les deux raisons suivantes

  1. Valeur en double - Code d'erreur : 1062. Entrée en double '12' pour la clé ‘PRIMARY’

    The primary key column is unique and it will not accept the duplicate entry. So when you are trying to insert a new row which is already present in you table will produce this error.

Pour résoudre ce problème, définissez la colonne de clé primaire sur “AUTO_INCREMENT”. Et lorsque vous essayez d’insérer une nouvelle ligne, ignorez la clé primaire colonne ou insérez la valeur NULL dans la clé primaire.

CREATE TABLE userDetails(
  userId INT(10) NOT NULL AUTO_INCREMENT,
  firstName VARCHAR(50),
  lastName VARCHAR(50),
  isActive INT(1) DEFAULT 0,
  PRIMARY KEY (userId) );

--->and now while inserting 
INSERT INTO userDetails VALUES (NULL ,'John', 'Doe', 1);

  1. Champ de données unique - ‘Code d’erreur : 1062. Entrée en double ‘A’ pour la clé ‘code’`

    You may assigned a column as unique and trying to insert a new row with already existing value for that column will produce this error.

Pour surmonter cette erreur, utilisez INSERT IGNORE au lieu de INSERT normal. Si la nouvelle ligne que vous essayez d’insérer ne duplique pas un enregistrement existant, MySQL l’insère comme d’habitude. Si l’enregistrement est un doublon, le mot-clé IGNORE le supprime sans générer d’erreur.

INSERT IGNORE INTO userDetails VALUES (NULL ,'John', 'Doe', 1);
