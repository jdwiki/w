Casting implicite / automatique #

VARCHAR(255) – ou non #

Longueur maximale suggérée

Tout d’abord, je mentionnerai certaines chaînes courantes qui sont toujours hexadécimales ou autrement limitées à l’ASCII. Pour ceux-ci, vous devez spécifier CHARACTER SET ascii (latin1 est ok) afin qu’il ne gaspille pas d’espace :

UUID CHAR(36) CHARACTER SET ascii -- or pack into BINARY(16)
country_code CHAR(2) CHARACTER SET ascii
ip_address CHAR(39) CHARACTER SET ascii -- or pack into BINARY(16)
phone VARCHAR(20) CHARACTER SET ascii -- probably enough to handle extension
postal_code VARCHAR(20) CHARACTER SET ascii -- (not 'zip_code') (don't know the max

city VARCHAR(100) -- This Russian town needs 91:
    Poselok Uchebnogo Khozyaystva Srednego Professionalno-Tekhnicheskoye Uchilishche Nomer Odin
country VARCHAR(50) -- probably enough
name VARCHAR(64) -- probably adequate; more than some government agencies allow

Pourquoi pas simplement 255 ? Il y a deux raisons d’éviter la pratique courante d’utiliser (255) pour tout.

  • Lorsqu’un complexe SELECT doit créer une table temporaire (pour une sous-requête, UNION, GROUP BY, etc.), le choix préféré est d’utiliser le moteur MEMORY, qui place les données en RAM. Mais les VARCHARs sont transformés en CHAR au cours du processus. Cela fait que VARCHAR(255) CHARACTER SET utf8mb4 prend 1020 octets. Cela peut entraîner la nécessité de transférer sur le disque, ce qui est plus lent.
  • Dans certaines situations, InnoDB examinera la taille potentielle des colonnes d’une table et décidera qu’elle sera trop grande, annulant une CREATE TABLE.

VARCHAR contre TEXT

Conseils d’utilisation pour *TEXT, CHAR et VARCHAR, ainsi que quelques bonnes pratiques :

  • N’utilisez jamais TINYTEXT.
  • N’utilisez presque jamais CHAR – c’est une longueur fixe ; chaque caractère correspond à la longueur maximale du CHARACTER SET (par exemple, 4 octets/caractère pour utf8mb4).
  • Avec CHAR, utilisez CHARACTER SET ascii sauf indication contraire.
  • VARCHAR(n) sera tronqué à n caractères ; TEXT sera tronqué à un certain nombre de bytes. (Mais, voulez-vous la troncature ?)
  • *TEXT peut ralentir les SELECT complexes en raison de la façon dont les tables temporaires sont gérées.

INT comme AUTO_INCREMENT #

N’importe quelle taille de INT peut être utilisée pour AUTO_INCREMENT. UNSIGNED est toujours approprié.

Gardez à l’esprit que certaines opérations “gravent” les identifiants AUTO_INCREMENT. Cela pourrait entraîner un écart inattendu. Exemples : INSERT IGNORE et REPLACE. Ils peuvent préallouer un identifiant avant de se rendre compte qu’il ne sera pas nécessaire. Ceci est un comportement attendu et par conception dans le moteur InnoDB et ne devrait pas décourager leur utilisation.

Types de points fixes #

Les types DECIMAL et NUMERIC de MySQL stockent des valeurs de données numériques exactes. Il est recommandé d’utiliser ces types pour conserver une précision exacte, comme pour l’argent.

Décimal #

Ces valeurs sont stockées au format binaire. Dans une déclaration de colonne, la précision et l’échelle doivent être spécifiées

La précision représente le nombre de chiffres significatifs stockés pour les valeurs.

L’échelle représente le nombre de chiffres stockés après la décimale

salary DECIMAL(5,2)

5 représente la “précision” et 2 représente l’“échelle”. Pour cet exemple, la plage de valeurs pouvant être stockées dans cette colonne est -999,99 à 999,99

Si le paramètre d’échelle est omis, sa valeur par défaut est 0

Ce type de données peut stocker jusqu’à 65 chiffres.

Le nombre d’octets pris par DECIMAL(M,N) est environ M/2.

Types à virgule flottante #

FLOAT et DOUBLE représentent des types de données approximatifs.

Taper Stockage Précision Gamme
FLOTTER 4 octets 23 bits significatifs / ~7 chiffres décimaux 10^+/-38
DOUBLE 8 octets 53 bits significatifs / ~16 chiffres décimaux 10^+/-308

REAL est synonyme de FLOAT. “DOUBLE PRECISION” est synonyme de “DOUBLE”.

Bien que MySQL autorise également le qualificateur (M,D), ne l’utilisez pas. (M,D) signifie que les valeurs peuvent être stockées avec jusqu’à M chiffres au total, où D peut être après la virgule. Les nombres seront arrondis deux fois ou tronqués ; cela causera plus de problèmes que d’avantages.

Étant donné que les valeurs à virgule flottante sont approximatives et ne sont pas stockées comme des valeurs exactes, les tentatives de les traiter comme exactes dans les comparaisons peuvent entraîner des problèmes. Notez en particulier qu’une valeur FLOAT est rarement égale à une valeur DOUBLE.

Type de valeur binaire #

Le type BIT est utile pour stocker des valeurs de champ de bits. BIT(M) permet le stockage de valeurs allant jusqu’à M bits où M est compris entre 1 et 64

Vous pouvez également spécifier des valeurs avec la notation “valeur de bit”.

b'111'      -> 7
b'10000000' -> 128

Parfois, il est pratique d’utiliser ‘shift’ pour construire une valeur à un seul bit, par exemple (1 << 7) pour 128.

La taille combinée maximale de toutes les colonnes BIT d’une table “NDB” est de 4 096.

CHAR(n) #

CHAR(n) est une chaîne d’une longueur fixe de n caractères. S’il s’agit de CHARACTER SET utf8mb4, cela signifie qu’il occupe exactement 4*n bytes, quel que soit le texte qu’il contient.

La plupart des cas d’utilisation de CHAR(n) impliquent des chaînes qui contiennent des caractères anglais, donc devraient être CHARACTER SET ascii. (latin1 fera tout aussi bien.)

country_code CHAR(2) CHARACTER SET ascii,
postal_code  CHAR(6) CHARACTER SET ascii,
uuid    CHAR(39) CHARACTER SET ascii,  -- more discussion elsewhere

DATE, DATETIME, TIMESTAMP, YEAR et TIME #

Le type de données DATE comprend la date mais pas d’heure. Son format est 'AAAA-MM-JJ' avec une plage de ‘1000-01-01’ à ‘9999-12-31’.

Le type DATETIME inclut l’heure au format ‘YYYY-MM-DD HH:MM:SS’. Sa plage va de ‘1000-01-01 00:00:00’ à ‘9999-12-31 23:59:59’.

Le type TIMESTAMP est un type entier comprenant la date et l’heure avec une plage effective de ‘1970-01-01 00:00:01’ UTC à ‘2038-01-19 03:14:07’ UTC.

Le type “YEAR” représente une année et contient une plage de 1901 à 2155.

Le type TIME représente une heure au format ‘HH:MM:SS’ et contient une plage de ‘-838:59:59’ à ‘838:59:59’.

Exigences de stockage :

|-----------|--------------------|----------------------------------------|
| Data Type | Before MySQL 5.6.4 | as of MySQL 5.6.4                      |
|-----------|--------------------|----------------------------------------|
| YEAR      |      1 byte        |  1 byte                                |
| DATE      |      3 bytes       |  3 bytes                               |
| TIME      |      3 bytes       |  3 bytes + fractional seconds storage  |
| DATETIME  |      8 bytes       |  5 bytes + fractional seconds storage  |
| TIMESTAMP |      4 bytes       |  4 bytes + fractional seconds storage  |
|-----------|--------------------|----------------------------------------|

Fraction de seconde (à partir de la version 5.6.4) :

|------------------------------|------------------|
| Fractional Seconds Precision | Storage Required |
|------------------------------|------------------|
|              0               |      0 bytes     |
|              1,2             |      1 byte      |
|              3,4             |      2 byte      |
|              5,6             |      3 byte      |
|------------------------------|------------------|

Consultez les pages du manuel MySQL [Types DATE, DATETIME et TIMESTAMP][1], [Exigences de stockage des types de données][2] et [Fractional Seconds in Time Values][3].
