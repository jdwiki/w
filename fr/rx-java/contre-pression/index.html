Introduction #

La contre-pression se produit lorsque, dans un pipeline de traitement “observable”, certaines étapes asynchrones ne peuvent pas traiter les valeurs assez rapidement et ont besoin d’un moyen de dire au producteur en amont de ralentir.

Le cas classique du besoin de contre-pression est lorsque le producteur est une source chaude :

PublishSubject<Integer> source = PublishSubject.create();

source
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Thread.sleep(10_000);

Dans cet exemple, le thread principal produira 1 million d’éléments pour un consommateur final qui le traitera sur un thread d’arrière-plan. Il est probable que la méthode compute(int) prenne un certain temps, mais la surcharge de la chaîne d’opérateurs Observable peut également augmenter le temps nécessaire au traitement des éléments. Cependant, le thread producteur avec la boucle for ne peut pas le savoir et continue à onNext.

En interne, les opérateurs asynchrones ont des tampons pour conserver ces éléments jusqu’à ce qu’ils puissent être traités. Dans le Rx.NET classique et les premiers RxJava, ces tampons étaient illimités, ce qui signifie qu’ils contiendraient probablement presque tous les 1 million d’éléments de l’exemple. Le problème commence lorsqu’il y a, par exemple, 1 milliard d’éléments ou que la même séquence de 1 million apparaît 1 000 fois dans un programme, ce qui entraîne une « OutOfMemoryError » et généralement des ralentissements dus à une surcharge excessive du GC.

Semblable à la façon dont la gestion des erreurs est devenue un citoyen de première classe et a reçu des opérateurs pour y faire face (via les opérateurs onErrorXXX), la contre-pression est une autre propriété des flux de données à laquelle le programmeur doit penser et gérer (via les opérateurs onBackpressureXXX).

Au-delà de PublishSubject ci-dessus, il existe d’autres opérateurs qui ne prennent pas en charge la contre-pression, principalement pour des raisons fonctionnelles. Par exemple, l’opérateur « intervalle » émet des valeurs périodiquement, le contre-pression conduirait à un décalage de la période par rapport à une horloge murale.

Dans RxJava moderne, la plupart des opérateurs asynchrones ont maintenant un tampon interne limité, comme observeOn ci-dessus et toute tentative de débordement de ce tampon mettra fin à toute la séquence avec MissingBackpressureException. La documentation de chaque opérateur contient une description de son comportement de contre-pression.

Cependant, la contre-pression est présente plus subtilement dans les séquences froides régulières (qui ne produisent pas et ne devraient pas produire MissingBackpressureException). Si le premier exemple est réécrit :

Observable.range(1, 1_000_000)
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

Thread.sleep(10_000);

Il n’y a pas d’erreur et tout se passe bien avec une faible utilisation de la mémoire. La raison en est que de nombreux opérateurs source peuvent “générer” des valeurs à la demande et donc l’opérateur observeOn peut dire à la range de générer au plus autant de valeurs que le tampon observeOn peut contenir à la fois sans débordement.

Cette négociation est basée sur le concept informatique des co-routines (je t’appelle, tu m’appelles). L’opérateur range envoie un rappel, sous la forme d’une implémentation de l’interface Producer, à observeOn en appelant son ``Subscriberinterne)setProducer. En retour, le observeOnappelleProducer.request(n)avec une valeur pour dire à larangequ'il est autorisé à produire (c'est-à-dire,onNextit) autant d'éléments **supplémentaires**. Il est alors de la responsabilité deobserveOnd'appeler la méthoderequest` au bon moment et avec la bonne valeur pour que les données continuent de circuler sans déborder.

L’expression de la contre-pression chez les consommateurs finaux est rarement nécessaire (car ils sont synchrones par rapport à leur amont immédiat et la contre-pression se produit naturellement en raison du blocage de la pile d’appels), mais il peut être plus facile d’en comprendre le fonctionnement :

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(1);
    }

    public void onNext(Integer v) {
        compute(v);

        request(1);
    }

    @Override
    public void onError(Throwable ex) {
        ex.printStackTrace();
    }

    @Override
    public void onCompleted() {
        System.out.println("Done!");
    }
});

Ici, l’implémentation onStart indique range pour produire sa première valeur, qui est ensuite reçue dans onNext. Une fois le compute(int) terminé, une autre valeur est alors demandée à range. Dans une implémentation naïve de range, un tel appel appellerait de manière récursive onNext, conduisant à StackOverflowError, ce qui est bien sûr indésirable.

Pour éviter cela, les opérateurs utilisent une logique dite de trampoline qui empêche de tels appels réentrants. Dans les termes de range, il se souviendra qu’il y a eu un appel request(1) pendant qu’il appelait onNext() et une fois que onNext() revient, il fera un autre tour et appellera onNext() avec la prochaine valeur entière. Par conséquent, si les deux sont échangés, l’exemple fonctionne toujours de la même manière :

@Override
public void onNext(Integer v) {
    request(1);

    compute(v);
}

Cependant, ce n’est pas vrai pour onStart. Bien que l’infrastructure Observable garantisse qu’elle sera appelée au plus une fois sur chaque Subscriber, l’appel à request(1) peut déclencher immédiatement l’émission d’un élément. Si l’on a une logique d’initialisation après l’appel à request(1) qui est nécessaire à onNext, vous pouvez vous retrouver avec des exceptions :

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {

    String name;

    @Override
    public void onStart() {
        request(1);

        name = "RangeExample";
    }

    @Override
    public void onNext(Integer v) {
        compute(name.length + v);

        request(1);
    }

    // ... rest is the same
});

Dans ce cas synchrone, une NullPointerException sera lancée immédiatement tout en exécutant onStart. Un bogue plus subtil se produit si l’appel à request(1) déclenche un appel asynchrone à onNext sur un autre thread et lit name dans les courses onNext en l’écrivant dans onStart post request.

Par conséquent, il faut effectuer toutes les initialisations de champ dans onStart ou même avant cela et appeler request() en dernier. Les implémentations de request() dans les opérateurs garantissent une bonne relation se produit avant (ou en d’autres termes, une libération de mémoire ou une clôture complète) si nécessaire.

Les opérateurs onBackpressureXXX #

La plupart des développeurs rencontrent une contre-pression lorsque leur application échoue avec MissingBackpressureException et l’exception pointe généralement vers l’opérateur observeOn. La cause réelle est généralement l’utilisation sans contre-pression de PublishSubject, timer() ou interval() ou d’opérateurs personnalisés créés via create().

Il existe plusieurs façons de faire face à de telles situations.

Augmenter la taille des tampons #

Parfois, de tels débordements se produisent en raison de sources en rafales. Soudain, l’utilisateur appuie trop rapidement sur l’écran et le tampon interne à 16 éléments par défaut d’‘observeOn’ sur Android déborde.

La plupart des opérateurs sensibles à la contre-pression dans les versions récentes de RxJava permettent désormais aux programmeurs de spécifier la taille de leurs tampons internes. Les paramètres pertinents sont généralement appelés bufferSize, prefetch ou capacityHint. Compte tenu de l’exemple débordant dans l’introduction, nous pouvons simplement augmenter la taille du tampon de observeOn pour avoir suffisamment de place pour toutes les valeurs.

PublishSubject<Integer> source = PublishSubject.create();

source.observeOn(Schedulers.computation(), 1024 * 1024)
      .subscribe(e -> { }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Notez cependant qu’en général, il ne s’agit que d’une solution temporaire car le débordement peut toujours se produire si la source surproduit la taille de tampon prévue. Dans ce cas, on peut utiliser l’un des opérateurs suivants.

Regrouper/sauter des valeurs avec des opérateurs standard #

Dans le cas où les données source peuvent être traitées plus efficacement par lots, on peut réduire la probabilité de MissingBackpressureException en utilisant l’un des opérateurs de traitement par lots standard (par taille et/ou par temps).

PublishSubject<Integer> source = PublishSubject.create();

source
      .buffer(1024)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(list -> { 
          list.parallelStream().map(e -> e * e).first();
      }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Si certaines des valeurs peuvent être ignorées en toute sécurité, on peut utiliser les opérateurs d’échantillonnage (avec le temps ou un autre Observable) et de limitation (throttleFirst, throttleLast, throttleWithTimeout).

PublishSubject<Integer> source = PublishSubject.create();

source
      .sample(1, TimeUnit.MILLISECONDS)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Notez cependant que ces opérateurs ne font que réduire le taux de réception de la valeur par l’aval et qu’ils peuvent donc toujours conduire à MissingBackpressureException.

onBackpressureBuffer() #

Cet opérateur dans sa forme sans paramètre réintroduit un tampon illimité entre la source amont et l’opérateur aval. Être illimité signifie que tant que la JVM ne manque pas de mémoire, elle peut gérer presque n’importe quelle quantité provenant d’une source en rafale.

 Observable.range(1, 1_000_000)
           .onBackpressureBuffer()
           .observeOn(Schedulers.computation(), 8)
           .subscribe(e -> { }, Throwable::printStackTrace);

Dans cet exemple, observeOn va avec une taille de tampon très faible, mais il n’y a pas de MissingBackpressureException car onBackpressureBuffer absorbe toutes les 1 million de valeurs et en transmet de petits lots à observeOn.

Notez cependant que onBackpressureBuffer consomme sa source de manière illimitée, c’est-à-dire sans lui appliquer de contre-pression. Cela a pour conséquence que même une source supportant une contre-pression telle qu’une “gamme” sera complètement réalisée.

Il y a 4 surcharges supplémentaires de onBackpressureBuffer

onBackpressureBuffer(int capacité) #

Il s’agit d’une version limitée qui signale BufferOverflowError au cas où son tampon atteint la capacité donnée.

Observable.range(1, 1_000_000)
          .onBackpressureBuffer(16)
          .observeOn(Schedulers.computation())
          .subscribe(e -> { }, Throwable::printStackTrace);

La pertinence de cet opérateur diminue car de plus en plus d’opérateurs permettent désormais de définir leurs tailles de tampon. Pour le reste, cela donne la possibilité “d’étendre leur tampon interne” en ayant un nombre plus grand avec onBackpressureBuffer que leur valeur par défaut.

onBackpressureBuffer(int capacité, Action0 onOverflow) #

Cette surcharge appelle une action (partagée) en cas de débordement. Son utilité est plutôt limitée car il n’y a pas d’autres informations fournies sur le débordement que la pile d’appels actuelle.

onBackpressureBuffer(capacité int, Action0 onOverflow, stratégie BackpressureOverflow.Strategy) #

Cette surcharge est en fait plus utile car elle permet de définir ce qu’il faut faire si la capacité est atteinte. BackpressureOverflow.Strategy est en fait une interface mais la classe BackpressureOverflow propose 4 champs statiques avec des implémentations représentant des actions typiques :

  • ON_OVERFLOW_ERROR : il s’agit du comportement par défaut des deux surcharges précédentes, signalant une BufferOverflowException

  • ON_OVERFLOW_DEFAULT : actuellement, c’est la même chose que ON_OVERFLOW_ERROR

  • ON_OVERFLOW_DROP_LATEST : si un débordement se produisait, la valeur courante sera simplement ignorée et seules les anciennes valeurs seront livrées une fois les requêtes en aval.

  • ON_OVERFLOW_DROP_OLDEST : supprime l’élément le plus ancien du tampon et y ajoute la valeur actuelle.

    Observable.range(1, 1_000_000) .onBackpressureBuffer(16, () -> { }, BufferOverflowStrategy.ON_OVERFLOW_DROP_OLDEST) .observeOn(Schedulers.computation()) .subscribe(e -> { }, Throwable::printStackTrace);

Notez que les deux dernières stratégies provoquent une discontinuité dans le flux lorsqu’elles suppriment des éléments. De plus, ils ne signaleront pas BufferOverflowException.

onBackpressureDrop() #

Chaque fois que l’aval n’est pas prêt à recevoir des valeurs, cet opérateur supprimera cet élément de la séquence. On peut le considérer comme un onBackpressureBuffer de capacité 0 avec la stratégie ON_OVERFLOW_DROP_LATEST.

Cet opérateur est utile lorsque l’on peut ignorer en toute sécurité les valeurs d’une source (telles que les mouvements de souris ou les signaux de localisation GPS actuels) car il y aura plus de valeurs à jour plus tard.

 component.mouseMoves()
 .onBackpressureDrop()
 .observeOn(Schedulers.computation(), 1)
 .subscribe(event -> compute(event.x, event.y));

Il peut être utile en conjonction avec l’opérateur source interval(). Par exemple, si l’on veut effectuer une tâche d’arrière-plan périodique mais que chaque itération peut durer plus longtemps que la période, il est prudent de supprimer la notification d’intervalle excessif car il y en aura plus plus tard :

 Observable.interval(1, TimeUnit.MINUTES)
 .onBackpressureDrop()
 .observeOn(Schedulers.io())
 .doOnNext(e -> networkCall.doStuff())
 .subscribe(v -> { }, Throwable::printStackTrace);

Il existe une surcharge de cet opérateur : onBackpressureDrop(Action1<? super T> onDrop) où l’action (partagée) est appelée avec la valeur supprimée. Cette variante permet de nettoyer les valeurs elles-mêmes (par exemple, libérer les ressources associées).

onBackpressureLatest() #

L’opérateur final ne conserve que la dernière valeur et écrase pratiquement les anciennes valeurs non livrées. On peut considérer cela comme une variante du onBackpressureBuffer avec une capacité de 1 et une stratégie de ON_OVERFLOW_DROP_OLDEST.

Contrairement à onBackpressureDrop, il y a toujours une valeur disponible pour la consommation si l’aval est en retard. Cela peut être utile dans certaines situations de type télémétrie où les données peuvent se présenter sous forme de rafales, mais seules les toutes dernières sont intéressantes pour le traitement.

Par exemple, si l’utilisateur clique beaucoup sur l’écran, nous voudrions toujours réagir à sa dernière entrée.

component.mouseClicks()
.onBackpressureLatest()
.observeOn(Schedulers.computation())
.subscribe(event -> compute(event.x, event.y), Throwable::printStackTrace);

L’utilisation de onBackpressureDrop dans ce cas conduirait à une situation où le tout dernier clic serait abandonné et laisserait l’utilisateur se demander pourquoi la logique métier n’a pas été exécutée.

Création de sources de données en contre-pression #

La création de sources de données à contre-pression est la tâche relativement plus facile lorsqu’il s’agit de contre-pression en général, car la bibliothèque propose déjà des méthodes statiques sur “Observable” qui gèrent la contre-pression pour le développeur. Nous pouvons distinguer deux types de méthodes d’usine : les « générateurs » froids qui renvoient et génèrent des éléments en fonction de la demande en aval et les « pousseurs » chauds qui relient généralement les sources de données non réactives et/ou non contre-pressurables et ajoutent une certaine gestion de la contre-pression au-dessus de leur.

juste #

La source de contre-pression la plus basique est créée via just :

Observable.just(1).subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(0);
    }

    @Override
    public void onNext(Integer v) {
        System.out.println(v);
    }
   
    // the rest is omitted for brevity
}

Puisque nous ne demandons pas explicitement dans onStart, cela n’affichera rien. just est génial lorsqu’il y a une valeur constante que nous aimerions lancer une séquence.

Malheureusement, “juste” est souvent confondu avec un moyen de calculer dynamiquement quelque chose à consommer par “l’abonné” :

int counter;

int computeValue() {
   return ++counter;
}

Observable<Integer> o = Observable.just(computeValue());

o.subscribe(System.out:println);
o.subscribe(System.out:println);

Surprenant pour certains, cela imprime 1 deux fois au lieu d’imprimer 1 et 2 respectivement. Si l’appel est réécrit, il devient évident pourquoi cela fonctionne ainsi :

int temp = computeValue();

Observable<Integer> o = Observable.just(temp);

Le computeValue est appelé dans le cadre de la routine principale et non en réponse aux abonnés qui s’abonnent.

deCallable #

Ce dont les gens ont réellement besoin, c’est de la méthode fromCallable :

Observable<Integer> o = Observable.fromCallable(() -> computeValue());

Ici, computeValue est exécuté uniquement lorsqu’un abonné s’abonne et pour chacun d’eux, imprime les 1 et 2 attendus. Notez cependant que le calcul se produit de toute façon. Dans le cas où le calcul lui-même devrait être retardé jusqu’à ce que l’aval demande réellement, nous pouvons utiliser just avec map :

Observable.just("This doesn't matter").map(ignored -> computeValue())...

just n’émettra pas sa valeur constante jusqu’à ce qu’il soit demandé lorsqu’il est mappé au résultat de computeValue, toujours appelé pour chaque abonné individuellement.

de #

Si les données sont déjà disponibles sous la forme d’un tableau d’objets, d’une liste d’objets ou de toute source “itérable”, les surcharges “from” respectives géreront la contre-pression et l’émission de ces sources :

 Observable.from(Arrays.asList(1, 2, 3, 4, 5)).subscribe(System.out::println);

Pour plus de commodité (et pour éviter les avertissements concernant la création de tableaux génériques), il y a 2 à 10 surcharges d’arguments pour juste' qui délèguent en interne à from`.

Le from(Iterable) donne également une opportunité intéressante. De nombreuses générations de valeur peuvent être exprimées sous la forme d’une machine à états. Chaque élément demandé déclenche une transition d’état et le calcul de la valeur renvoyée.

Écrire de telles machines d’état comme Iterables est quelque peu compliqué (mais toujours plus facile que d’écrire un Observable pour le consommer) et contrairement à C #, Java n’a aucun support du compilateur pour construire de telles machines d’état en écrivant simplement de manière classique code (avec yield return et yield break). Certaines bibliothèques offrent une aide, comme AbstractIterable de Google Guava et Ix.generate() et Ix.forloop() d’IxJava. Ceux-ci sont en eux-mêmes dignes d’une série complète, alors voyons une source “itérable” très basique qui répète indéfiniment une valeur constante :

Iterable<Integer> iterable = () -> new Iterator<Integer>() {
    @Override
    public boolean hasNext() {
        return true;
    }

    @Override
    public Integer next() {
        return 1;
    }
};

Observable.from(iterable).take(5).subscribe(System.out::println);

Si nous consommons l’itérateur via une boucle for classique, cela donnerait une boucle infinie. Puisque nous en construisons un “Observable”, nous pouvons exprimer notre volonté de n’en consommer que les 5 premiers et ensuite cesser de demander quoi que ce soit. C’est le véritable pouvoir de l’évaluation et du calcul paresseux à l’intérieur de Observables.

créer (SyncOnSubscribe) #

Parfois, la source de données à convertir dans le monde réactif lui-même est synchrone (bloquante) et de type pull, c’est-à-dire que nous devons appeler une méthode “get” ou “read” pour obtenir la prochaine donnée. On pourrait, bien sûr, transformer cela en un “itérable”, mais lorsque de telles sources sont associées à des ressources, nous pouvons divulguer ces ressources si l’aval se désabonne de la séquence avant qu’elle ne se termine.

Pour gérer de tels cas, RxJava a la classe SyncOnSubscribe. On peut l’étendre et implémenter ses méthodes ou utiliser l’une de ses méthodes d’usine basées sur lambda pour créer une instance.

SyncOnSubscribe<Integer, InputStream> binaryReader = SyncOnSubscribe.createStateful(
     () -> new FileInputStream("data.bin"),
     (inputstream, output) -> {
         try {
             int byte = inputstream.read();
             if (byte < 0) {
                 output.onCompleted();
             } else {
                 output.onNext(byte);
             }
         } catch (IOException ex) {
             output.onError(ex);
         }
         return inputstream;
     },
     inputstream -> {
         try {
             inputstream.close();
         } catch (IOException ex) {
             RxJavaHooks.onError(ex);
         }
     } 
 );

 Observable<Integer> o = Observable.create(binaryReader);

Généralement, SyncOnSubscribe utilise 3 rappels.

Les premiers rappels permettent de créer un état par abonné, tel que FileInputStream dans l’exemple ; le fichier sera ouvert indépendamment pour chaque abonné individuel.

Le deuxième rappel prend cet objet d’état et fournit une sortie Observer dont les méthodes onXXX peuvent être appelées pour émettre des valeurs. Ce rappel est exécuté autant de fois que l’aval l’a demandé. À chaque invocation, il doit appeler onNext au plus une fois suivi éventuellement de onError ou onCompleted. Dans l’exemple, nous appelons onCompleted() si l’octet lu est négatif, indiquant la fin du fichier, et appelons onError si la lecture lève une IOException.

Le rappel final est invoqué lorsque l’aval se désabonne (ferme le flux d’entrée) ou lorsque le rappel précédent a appelé les méthodes du terminal ; cela permet de libérer des ressources. Étant donné que toutes les sources n’ont pas besoin de toutes ces fonctionnalités, les méthodes statiques de SyncOnSubscribe permettent de créer des instances sans elles.

Malheureusement, de nombreux appels de méthode à travers la JVM et d’autres bibliothèques lancent des exceptions vérifiées et doivent être enveloppés dans try-catches car les interfaces fonctionnelles utilisées par cette classe ne permettent pas de lancer des exceptions vérifiées.

Bien sûr, nous pouvons imiter d’autres sources typiques, comme une plage illimitée avec :

SyncOnSubscribe.createStateful(
     () -> 0,
     (current, output) -> {
         output.onNext(current);
         return current + 1;
     },
     e -> { }
);

Dans cette configuration, le “current” commence par “0” et la prochaine fois que le lambda est appelé, le paramètre “current” contient désormais “1”.

Il existe une variante de SyncOnSubscribe appelée AsyncOnSubscribe qui semble assez similaire à l’exception que le rappel du milieu prend également une valeur longue qui représente le montant de la demande en aval et le rappel doit générer un Observable avec exactement la même longueur. Cette source concatène ensuite tous ces “Observables” en une seule séquence.

 AsyncOnSubscribe.createStateful(
     () -> 0,
     (state, requested, output) -> {
         output.onNext(Observable.range(state, (int)requested));
         return state + 1;
     },
     e -> { }
 );

Il y a une discussion (vive) en cours sur l’utilité de cette classe et généralement déconseillée car elle rompt régulièrement les attentes sur la manière dont elle émettra réellement ces valeurs générées et sur la manière dont elle répondra, ou même sur le type de valeurs de requête qu’elle recevra dans scénarios de consommation plus complexes.

créer (émetteur) #

Parfois, la source à encapsuler dans un Observable est déjà chaude (comme les mouvements de la souris) ou froide mais non rétropressurable dans son API (comme un rappel réseau asynchrone).

Pour gérer de tels cas, une version récente de RxJava a introduit la méthode de fabrique create(emitter). Il prend deux paramètres :

  • un callback qui sera appelé avec une instance de l’interface Emitter<T> pour chaque abonné entrant,
  • une énumération Emitter.BackpressureMode qui oblige le développeur à spécifier le comportement de contre-pression à appliquer. Il a les modes habituels, similaires à onBackpressureXXX en plus de signaler une MissingBackpressureException ou d’ignorer simplement un tel débordement à l’intérieur.

Notez qu’il ne prend actuellement pas en charge les paramètres supplémentaires de ces modes de contre-pression. Si l’on a besoin de ces personnalisations, utiliser NONE comme mode de contre-pression et appliquer le onBackpressureXXX pertinent sur le Observable résultant est la voie à suivre.

Le premier cas typique pour son utilisation lorsque l’on veut interagir avec une source basée sur le push, comme les événements de l’interface graphique. Ces API comportent une forme d’appels addListener/removeListener que l’on peut utiliser :

Observable.create(emitter -> {
    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    emitter.setCancellation(() -> 
        button.removeListener(al));

}, BackpressureMode.BUFFER);

L’“émetteur” est relativement simple à utiliser ; on peut y appeler onNext, onError et onCompleted et l’opérateur gère lui-même la contre-pression et la gestion des désinscriptions. De plus, si l’API enveloppée prend en charge l’annulation (telle que la suppression de l’écouteur dans l’exemple), on peut utiliser setCancellation (ou setSubscription pour les ressources de type Subscription) pour enregistrer un rappel d’annulation qui est invoqué lorsque le en aval se désabonne ou onError/onCompleted est appelé sur l’instance Emitterfournie.

Ces méthodes permettent à une seule ressource d’être associée à l’émetteur à la fois et la définition d’une nouvelle désabonne automatiquement l’ancienne. Si vous devez gérer plusieurs ressources, créez un CompositeSubscription, associez-le à l’émetteur, puis ajoutez d’autres ressources au CompositeSubscription lui-même :

Observable.create(emitter -> {
    CompositeSubscription cs = new CompositeSubscription();

    Worker worker = Schedulers.computation().createWorker();

    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    cs.add(worker);
    cs.add(Subscriptions.create(() -> 
        button.removeActionListener(al));

    emitter.setSubscription(cs);

}, BackpressureMode.BUFFER);

Le deuxième scénario implique généralement une API asynchrone basée sur un rappel qui doit être convertie en un Observable.

Observable.create(emitter -> {
    
    someAPI.remoteCall(new Callback<Data>() {
        @Override
        public void onSuccess(Data data) {
            emitter.onNext(data);
            emitter.onCompleted();
        }

        @Override
        public void onFailure(Exception error) {
            emitter.onError(error);
        }
    });

}, BackpressureMode.LATEST);

Dans ce cas, la délégation fonctionne de la même manière. Malheureusement, généralement, ces API classiques de style rappel ne prennent pas en charge l’annulation, mais si elles le font, on peut configurer leur annulation comme dans les exemples précédents (avec peut-être une manière plus complexe cependant). Notez l’utilisation du mode de contre-pression LATEST ; si nous savons qu’il n’y aura qu’une seule valeur, nous n’avons pas besoin de la stratégie BUFFER car elle alloue un tampon long par défaut de 128 éléments (qui augmente si nécessaire) qui ne sera jamais pleinement utilisé.
