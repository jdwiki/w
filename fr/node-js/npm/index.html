Installer des packages #

Introduction #

Package est un terme utilisé par npm pour désigner les outils que les développeurs peuvent utiliser pour leurs projets. Cela inclut tout, des bibliothèques et des frameworks tels que jQuery et AngularJS aux exécuteurs de tâches tels que Gulp.js. Les packages viendront dans un dossier généralement appelé node_modules, qui contiendra également un fichier package.json. Ce fichier contient des informations sur tous les packages, y compris les dépendances, qui sont des modules supplémentaires nécessaires pour utiliser un package particulier.

Npm utilise la ligne de commande pour installer et gérer les packages, de sorte que les utilisateurs qui tentent d’utiliser npm doivent être familiarisés avec les commandes de base de leur système d’exploitation, c’est-à-dire : parcourir les répertoires et pouvoir voir le contenu des répertoires.

Installer NPM #

Notez que pour installer des packages, vous devez avoir installé NPM.

La méthode recommandée pour installer NPM consiste à utiliser l’un des programmes d’installation de la [page de téléchargement de Node.js][1]. Vous pouvez vérifier si node.js est déjà installé en exécutant la commande npm -v ou npm version.

Après avoir installé NPM via le programme d’installation Node.js, assurez-vous de vérifier les mises à jour. En effet, NPM est mis à jour plus fréquemment que le programme d’installation de Node.js. Pour vérifier les mises à jour, exécutez la commande suivante :

npm install npm@latest -g

Comment installer des packages #

Pour installer un ou plusieurs packages, utilisez les éléments suivants :

npm install <package-name>
# or
npm i <package-name>...

# e.g. to install lodash and express
npm install lodash express

Remarque : Cela installera le package dans le répertoire dans lequel se trouve actuellement la ligne de commande, il est donc important de vérifier si le répertoire approprié a été choisi

Si vous avez déjà un fichier package.json dans votre répertoire de travail actuel et que des dépendances y sont définies, alors npm install résoudra et installera automatiquement toutes les dépendances répertoriées dans le fichier. Vous pouvez également utiliser la version abrégée de la commande npm install qui est : npm i

Si vous souhaitez installer une version spécifique d’un package, utilisez :

npm install <name>@<version>

# e.g. to install version 4.11.1 of the package lodash
npm install lodash@4.11.1

Si vous souhaitez installer une version qui correspond à une plage de versions spécifique, utilisez :

npm install <name>@<version range>

# e.g. to install a version which matches "version >= 4.10.1" and "version < 4.11.1"
# of the package lodash
npm install lodash@">=4.10.1 <4.11.1"

Si vous souhaitez installer la dernière version, utilisez :

npm install <name>@latest

Les commandes ci-dessus rechercheront des packages dans le référentiel central “npm” sur npmjs.com. Si vous ne cherchez pas à installer à partir du registre npm, d’autres options sont prises en charge, telles que :

# packages distributed as a tarball
npm install <tarball file>
npm install <tarball url>

# packages available locally
npm install <local path>

# packages available as a git repository
npm install <git remote url>

# packages available on GitHub
npm install <username>/<repository>

# packages available as gist (need a package.json)
npm install gist:<gist-id>

# packages from a specific repository
npm install --registry=http://myreg.mycompany.com <package name>

# packages from a related group of packages 
# See npm scope
npm install @<scope>/<name>(@<version>)

# Scoping is useful for separating private packages hosted on private registry from
# public ones by setting registry for specific scope
npm config set @mycompany:registry http://myreg.mycompany.com
npm install @mycompany/<package name>

Habituellement, les modules seront installés localement dans un dossier nommé node_modules, qui se trouve dans votre répertoire de travail actuel. C’est le répertoire que require() utilisera pour charger les modules afin de les mettre à votre disposition.

Si vous avez déjà créé un fichier package.json, vous pouvez utiliser l’option --save (raccourci -S) ou l’une de ses variantes pour ajouter automatiquement le package installé à votre package.json en tant que dépendance . Si quelqu’un d’autre installe votre package, npm lira automatiquement les dépendances du fichier package.json et installera les versions répertoriées. Notez que vous pouvez toujours ajouter et gérer vos dépendances en éditant le fichier plus tard, c’est donc généralement une bonne idée de garder une trace des dépendances, par exemple en utilisant :

npm install --save <name> # Install dependencies 
# or
npm install -S <name> # shortcut version --save 
# or
npm i -S <name>

Afin d’installer des packages et de les enregistrer uniquement s’ils sont nécessaires au développement, pas à leur exécution, pas s’ils sont nécessaires à l’exécution de l’application, suivez la commande suivante :

npm install --save-dev <name> # Install dependencies for development purposes
# or
npm install -D <name> # shortcut version --save-dev
# or
npm i -D <name>

Installation des dépendances #

Certains modules ne fournissent pas seulement une bibliothèque à utiliser, mais ils fournissent également un ou plusieurs binaires destinés à être utilisés via la ligne de commande. Bien que vous puissiez toujours installer ces packages localement, il est souvent préférable de les installer globalement afin que les outils de ligne de commande puissent être activés. Dans ce cas, npm liera automatiquement les binaires aux chemins appropriés (par exemple /usr/local/bin/<nom>) afin qu’ils puissent être utilisés à partir de la ligne de commande. Pour installer un package globalement, utilisez :

npm install --global <name>
# or
npm install -g <name>
# or
npm i -g <name>

# e.g. to install the grunt command line tool
npm install -g grunt-cli

Si vous souhaitez voir une liste de tous les packages installés et de leurs versions associées dans l’espace de travail actuel, utilisez :

npm list
npm list <name>

L’ajout d’un argument de nom facultatif peut vérifier la version d’un package spécifique.

Remarque : Si vous rencontrez des problèmes d’autorisation en essayant d’installer un module npm globalement, résistez à la tentation d’émettre un sudo npm install -g ... pour résoudre le problème. Autoriser des scripts tiers à s’exécuter sur votre système avec des privilèges élevés est dangereux. Le problème d’autorisation peut signifier que vous avez un problème avec la façon dont npm lui-même a été installé. Si vous souhaitez installer Node dans des environnements utilisateur en bac à sable, vous pouvez essayer d’utiliser [nvm][2].

Si vous avez des outils de construction ou d’autres dépendances de développement uniquement (par exemple, Grunt), vous ne voudrez peut-être pas les regrouper avec l’application que vous déployez. Si tel est le cas, vous voudrez l’avoir en tant que dépendance de développement, qui est répertoriée dans package.json sous devDependencies. Pour installer un paquet en tant que dépendance de développement uniquement, utilisez --save-dev (ou -D).

npm install --save-dev <name> // Install development dependencies which is not included in production 
# or
npm install -D <name>

Vous verrez que le package est alors ajouté aux devDependencies de votre package.json.

Pour installer les dépendances d’un projet node.js téléchargé/cloné, vous pouvez simplement utiliser

npm install
# or
npm i

npm lira automatiquement les dépendances de package.json et les installera.

NPM derrière un serveur proxy #

Si votre accès Internet se fait via un serveur proxy, vous devrez peut-être modifier les commandes d’installation npm qui accèdent aux référentiels distants. npm utilise un fichier de configuration qui peut être mis à jour via la ligne de commande :

npm config set

Vous pouvez localiser vos paramètres de proxy à partir du panneau des paramètres de votre navigateur. Une fois que vous avez obtenu les paramètres du proxy (URL du serveur, port, nom d’utilisateur et mot de passe) ; vous devez configurer vos configurations npm comme suit.

$ npm config set proxy http://<username>:<password>@<proxy-server-url>:<port>
$ npm config set https-proxy http://<username>:<password>@<proxy-server-url>:<port>

Les champs username, password, port sont facultatifs. Une fois que vous les avez définis, votre npm install, npm i -g etc. fonctionnera correctement.

[1] : https://nodejs.org/en/download/ [2] : https://github.com/creationix/nvm

Désinstallation des packages #

Pour désinstaller un ou plusieurs packages installés localement, utilisez :

npm uninstall <package name>

La commande de désinstallation pour npm a cinq alias qui peuvent également être utilisés :

npm remove <package name>
npm rm <package name>
npm r <package name>

npm unlink <package name>
npm un <package name>

Si vous souhaitez supprimer le package du fichier package.json dans le cadre de la désinstallation, utilisez le drapeau --save (raccourci : -S):

npm uninstall --save <package name>
npm uninstall -S <package name>

Pour une dépendance de développement, utilisez le drapeau --save-dev (raccourci : -D) :

npm uninstall --save-dev <package name>
npm uninstall -D <package name>

Pour une dépendance facultative, utilisez le drapeau --save-optional (raccourci : -O):

npm uninstall --save-optional <package name>
npm uninstall -O <package name>

Pour les packages installés globalement, utilisez le drapeau --global (raccourci : -g) :

npm uninstall -g <package name>

Mise en place d’une configuration de paquet #

Les configurations de package Node.js sont contenues dans un fichier appelé package.json que vous pouvez trouver à la racine de chaque projet. Vous pouvez créer un tout nouveau fichier de configuration en appelant :

npm init

Cela essaiera de lire le répertoire de travail actuel pour les informations du référentiel Git (s’il existe) et les variables d’environnement pour essayer de compléter automatiquement certaines des valeurs d’espace réservé pour vous. Sinon, il fournira une boîte de dialogue de saisie pour les options de base.

Si vous souhaitez créer un package.json avec des valeurs par défaut, utilisez :

npm init --yes
# or
npm init -y

Si vous créez un package.json pour un projet que vous n’allez pas publier en tant que package npm (c’est-à-dire uniquement dans le but d’arrondir vos dépendances), vous pouvez transmettre cette intention dans votre package.json fichier :

  1. Définissez éventuellement la propriété “private” sur “true” pour éviter toute publication accidentelle.
  2. Définissez éventuellement la propriété license sur “UNLICENSED” pour refuser aux autres le droit d’utiliser votre package.

Pour installer un package et l’enregistrer automatiquement dans votre package.json, utilisez :

npm install --save <package>

Le package et les métadonnées associées (telles que la version du package) apparaîtront dans vos dépendances. Si vous enregistrez if en tant que dépendance de développement (en utilisant --save-dev), le paquet apparaîtra à la place dans vos devDependencies.

Avec ce bare-bones package.json, vous rencontrerez des messages d’avertissement lors de l’installation ou de la mise à niveau des packages, vous indiquant qu’il vous manque une description et le champ du référentiel. Bien qu’il soit prudent d’ignorer ces messages, vous pouvez vous en débarrasser en ouvrant le package.json dans n’importe quel éditeur de texte et en ajoutant les lignes suivantes à l’objet JSON :

[...]
"description": "No description",
"repository": {
  "private": true
},
[...]

Exécuter des scripts #

Vous pouvez définir des scripts dans votre package.json, par exemple :

{
  "name": "your-package",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "author": "",
  "license": "ISC",
  "dependencies": {},
  "devDependencies": {},
  "scripts": {
    "echo": "echo hello!"
  }
}

Pour exécuter le script echo, exécutez npm run echo à partir de la ligne de commande. Les scripts arbitraires, tels que echo ci-dessus, doivent être exécutés avec npm run <script name>. npm a également un certain nombre de scripts officiels qu’il exécute à certaines étapes de la vie du paquet (comme preinstall). Voir ici pour un aperçu complet de la façon dont npm gère les champs de script.

Les scripts npm sont le plus souvent utilisés pour des choses comme le démarrage d’un serveur, la construction du projet et l’exécution de tests. Voici un exemple plus réaliste :

  "scripts": {
    "test": "mocha tests",
    "start": "pm2 start index.js"
  }

Dans les entrées scripts, les programmes en ligne de commande comme mocha fonctionneront lorsqu’ils seront installés globalement ou localement. Si l’entrée de ligne de commande n’existe pas dans le système PATH, npm vérifiera également vos packages installés localement.

Si vos scripts deviennent très longs, ils peuvent être divisés en plusieurs parties, comme ceci :

  "scripts": {
    "very-complex-command": "npm run chain-1 && npm run chain-2",
    "chain-1": "webpack",
    "chain-2": "node app.js"
  }

Versionnement sémantique de base #

Avant de publier un package, vous devez le versionner. npm prend en charge la [version sémantique] (http://semver.org/), cela signifie qu’il existe des versions correctives, mineures et majeures.

Par exemple, si votre package est en version 1.2.3 pour changer de version il faut :

  1. version du correctif : npm version patch => 1.2.4
  2. version mineure : npm version minor => 1.3.0
  3. version majeure : npm version major => 2.0.0

Vous pouvez également spécifier une version directement avec :

npm version 3.1.4 => 3.1.4

Lorsque vous définissez une version de package à l’aide de l’une des commandes npm ci-dessus, npm modifie le champ de version du fichier package.json, le valide et crée également une nouvelle balise Git avec la version préfixée par un “v”, comme si vous ‘ai lancé la commande :

balise git v3.1.4

Contrairement à d’autres gestionnaires de packages comme Bower, le registre npm ne repose pas sur la création de balises Git pour chaque version. Mais, si vous aimez utiliser des balises, n’oubliez pas de pousser la balise nouvellement créée après avoir modifié la version du package :

git push origin master (pour pousser le changement vers package.json)

git push origin v3.1.4 (pour pousser la nouvelle balise)

Ou vous pouvez le faire d’un seul coup avec:

git push origin master --tags

Publier un package #

Tout d’abord, assurez-vous que vous avez configuré votre package (comme indiqué dans https://www.wikiod.com/fr/node-js/npm#Mise en place d’une configuration de package). Ensuite, vous devez être connecté à npmjs.

Si vous avez déjà un utilisateur npm

npm login

Si vous n’avez pas d’utilisateur

npm adduser

Pour vérifier que votre utilisateur est enregistré dans le client actuel

npm config ls

Après cela, lorsque votre package est prêt à être publié, utilisez

npm publish

Et vous avez terminé.

Si vous devez publier une nouvelle version, assurez-vous de mettre à jour la version de votre package, comme indiqué dans https://www.wikiod.com/fr/node-js/npm#Versionnement sémantique de base Sinon, npm ne vous laissera pas publier le package.

{
    name: "package-name",
    version: "1.0.4"
}

Suppression des packages superflus #

Pour supprimer les packages superflus (packages installés mais pas dans la liste de dépendances), exécutez la commande suivante :

npm prune

Pour supprimer tous les packages dev, ajoutez l’indicateur --production :

npm prune --production

[En savoir plus][1]

[1] : https://docs.npmjs.com/cli/prune

Champs d’application et référentiels #

# Set the repository for the scope "myscope"
npm config set @myscope:registry http://registry.corporation.com

# Login at a repository and associate it with the scope "myscope"
npm adduser --registry=http://registry.corporation.com --scope=@myscope

# Install a package "mylib" from the scope "myscope"
npm install @myscope/mylib

Si le nom de votre propre package commence par @myscope et que la portée “myscope” est associée à un référentiel différent, npm publish téléchargera votre package dans ce référentiel à la place.

Vous pouvez également conserver ces paramètres dans un fichier .npmrc :

@myscope:registry=http://registry.corporation.com
//registry.corporation.com/:_authToken=xxxxxxxx-xxxx-xxxx-xxxxxxxxxxxxxxx

Ceci est utile lors de l’automatisation de la construction sur un serveur CI, par exemple.

Liste des packages actuellement installés #

Pour générer une liste (arborescence) des packages actuellement installés, utilisez

npm list

ls, la et ll sont des alias de la commande list. Les commandes la et ll affichent des informations étendues telles que la description et le référentiel.

Options

Le format de réponse peut être modifié en passant des options.

npm list --json
  • json - Affiche les informations au format json
  • long - Affiche des informations détaillées
  • parseable - Affiche la liste analysable au lieu de l’arborescence
  • global - Affiche les packages installés globalement
  • depth - Profondeur maximale d’affichage de l’arborescence des dépendances
  • dev/development - Affiche les devDependencies
  • prod/production - Affiche les dépendances

Si vous le souhaitez, vous pouvez également accéder à la page d’accueil du package.

npm home <package name>

Mise à jour de npm et des packages #

Puisque npm lui-même est un module Node.js, il peut être mis à jour en utilisant lui-même.

  • Si le système d’exploitation est Windows, vous devez exécuter l’invite de commande en tant qu’administrateur *

    npm install -g npm@latest

Si vous voulez vérifier les versions mises à jour, vous pouvez faire :

npm outdated

Pour mettre à jour un package spécifique :

npm update <package name>

Cela mettra à jour le package vers la dernière version en fonction des restrictions de package.json

Si vous souhaitez également verrouiller la version mise à jour dans package.json :

npm update <package name> --save

Verrouillage des modules sur des versions spécifiques #

Par défaut, npm installe la dernière version disponible des modules en fonction de la [version sémantique] de chaque dépendance (https://www.wikiod.com/fr/node-js/npm#Versionnement sémantique de base). Cela peut être problématique si un auteur de module n’adhère pas à semver et introduit des modifications avec rupture dans une mise à jour de module, par exemple.

Pour verrouiller la version de chaque dépendance (et les versions de leurs dépendances, etc.) sur la version spécifique installée localement dans le dossier node_modules, utilisez

npm shrinkwrap

Cela créera alors un npm-shrinkwrap.json à côté de votre package.json qui répertorie les versions spécifiques des dépendances.

Configuration pour les packages installés globalement #

Vous pouvez utiliser npm install -g pour installer un package “globalement”. Ceci est généralement fait pour installer un exécutable que vous pouvez ajouter à votre chemin d’accès à exécuter. Par exemple:

npm install -g gulp-cli

Si vous mettez à jour votre chemin, vous pouvez appeler gulp directement.

Sur de nombreux systèmes d’exploitation, npm install -g tentera d’écrire dans un répertoire dans lequel votre utilisateur ne pourra peut-être pas écrire, tel que /usr/bin. Vous ne devriez pas utiliser sudo npm install dans ce cas car il existe un risque de sécurité possible d’exécuter des scripts arbitraires avec sudo et l’utilisateur root peut créer des répertoires dans votre maison sur lesquels vous ne pouvez pas écrire et qui font de futures installations Plus difficile.

Vous pouvez indiquer à npm où installer les modules globaux via votre fichier de configuration, ~/.npmrc. C’est ce qu’on appelle le “préfixe” que vous pouvez afficher avec le “préfixe npm”.

prefix=~/.npm-global-modules

Cela utilisera le préfixe chaque fois que vous exécuterez npm install -g. Vous pouvez également utiliser npm install --prefix ~/.npm-global-modules pour définir le préfixe lors de l’installation. Si le préfixe est le même que votre configuration, vous n’avez pas besoin d’utiliser -g.

Pour utiliser le module globalement installé, il doit se trouver sur votre chemin :

export PATH=$PATH:~/.npm-global-modules/bin

Maintenant, lorsque vous exécutez npm install -g gulp-cli, vous pourrez utiliser gulp.

Remarque : Lorsque vous installez npm (sans -g), le préfixe sera le répertoire avec package.json ou le répertoire actuel si aucun n’est trouvé dans la hiérarchie. Cela crée également un répertoire node_modules/.bin contenant les exécutables. Si vous souhaitez utiliser un exécutable spécifique à un projet, il n’est pas nécessaire d’utiliser npm install -g. Vous pouvez utiliser celui de node_modules/.bin.

Lier des projets pour un débogage et un développement plus rapides #

La création de dépendances de projet peut parfois être une tâche fastidieuse. Au lieu de publier une version de package sur NPM et d’installer la dépendance pour tester les modifications, utilisez npm link. npm link crée un lien symbolique afin que le dernier code puisse être testé dans un environnement local. Cela facilite le test des outils globaux et des dépendances du projet en autorisant l’exécution du dernier code avant de créer une version publiée.

Texte d’aide #

NAME
       npm-link - Symlink a package folder

SYNOPSIS
         npm link (in package dir)
         npm link [<@scope>/]<pkg>[@<version>]

         alias: npm ln

Étapes pour lier les dépendances du projet #

Lors de la création du lien de dépendance, notez que le nom du package est celui qui sera référencé dans le projet parent.

  1. CD dans un répertoire de dépendances (ex : cd ../my-dep)
  2. “lien npm”
  3. CD dans le projet qui va utiliser la dépendance
  4. npm link my-dep ou si namespaced npm link @namespace/my-dep

Étapes pour lier un outil global #

  1. CD dans le répertoire du projet (ex : cd eslint-watch)
  2. “lien npm”
  3. Utilisez l’outil
  4. esw --quiet

Problèmes pouvant survenir #

La liaison de projets peut parfois causer des problèmes si la dépendance ou l’outil global est déjà installé. npm uninstall (-g) <pkg> puis en exécutant npm link résout normalement tous les problèmes qui peuvent survenir.
