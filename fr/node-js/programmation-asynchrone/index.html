Node est un langage de programmation où tout peut fonctionner de manière asynchrone. Vous trouverez ci-dessous quelques exemples et les éléments typiques du travail asynchrone.

Syntaxe #

  • doSomething([args], function([argsCB]) { /* faire quelque chose quand c’est fait */} );
  • doSomething([args], ([argsCB]) => { /* faire quelque chose une fois terminé */ });

Fonctions de rappel #

Fonctions de rappel en JavaScript #

Les fonctions de rappel sont courantes en JavaScript. Les fonctions de rappel sont possibles en JavaScript car [les fonctions sont des citoyens de première classe][1].

Rappels synchrones. #

Les fonctions de rappel peuvent être synchrones ou asynchrones. Étant donné que les fonctions de rappel asynchrones peuvent être plus complexes, voici un exemple simple de fonction de rappel synchrone.

// a function that uses a callback named `cb` as a parameter
function getSyncMessage(cb) {
    cb("Hello World!");
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getSyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

La sortie pour le code ci-dessus est :

> Before getSyncMessage call
> Hello World!
> After getSyncMessage call

Nous allons d’abord expliquer comment le code ci-dessus est exécuté. C’est plus pour ceux qui ne comprennent pas déjà le concept de rappels si vous le comprenez déjà, n’hésitez pas à sauter ce paragraphe. Tout d’abord, le code est analysé, puis la première chose intéressante à se produire est que la ligne 6 est exécutée, ce qui affiche Before getSyncMessage call sur la console. Ensuite, la ligne 8 est exécutée et appelle la fonction getSyncMessage en envoyant une fonction anonyme comme argument pour le paramètre nommé cb dans la fonction getSyncMessage. L’exécution est maintenant effectuée à l’intérieur de la fonction getSyncMessage sur la ligne 3 qui exécute la fonction cb qui vient d’être transmise, cet appel envoie une chaîne d’argument “Hello World” pour le paramètre nommé message dans la fonction anonyme transmise. L’exécution passe ensuite à la ligne 9 qui enregistre “Hello World!” sur la console. Ensuite, l’exécution passe par le processus de sortie de la [callstack][2] ([voir aussi][3]) en frappant la ligne 10 puis la ligne 4 puis enfin de retour à la ligne 11.

Quelques informations à connaître sur les callbacks en général :

  • La fonction que vous envoyez à une fonction en tant que rappel peut être appelée zéro fois, une fois ou plusieurs fois. Tout dépend de la réalisation.
  • La fonction de rappel peut être appelée de manière synchrone ou asynchrone et éventuellement à la fois de manière synchrone et asynchrone.
  • Tout comme les fonctions normales, les noms que vous donnez aux paramètres de votre fonction ne sont pas importants, mais l’ordre l’est. Ainsi, par exemple, à la ligne 8, le paramètre message aurait pu être nommé statement, msg, ou si vous êtes absurde quelque chose comme jellybean. Vous devez donc savoir quels paramètres sont envoyés dans votre rappel afin de pouvoir les obtenir dans le bon ordre avec des noms propres.

Rappels asynchrones. #

Une chose à noter à propos de JavaScript est qu’il est synchrone par défaut, mais il existe des API fournies dans l’environnement (navigateur, Node.js, etc.) qui pourraient le rendre asynchrone (il y a plus à ce sujet [ici][4]).

Certaines choses courantes qui sont asynchrones dans les environnements JavaScript qui acceptent les rappels :

  • Événements
  • setTimeout
  • setInterval
  • l’API de récupération
  • Promesses

De plus, toute fonction qui utilise l’une des fonctions ci-dessus peut être enveloppée avec une fonction qui prend un rappel et le rappel serait alors un rappel asynchrone (bien que l’emballage d’une promesse avec une fonction qui prend un rappel serait probablement considéré comme un anti-modèle comme il y a des façons plus préférées de gérer les promesses).

Donc, étant donné ces informations, nous pouvons construire une fonction asynchrone similaire à la fonction synchrone ci-dessus.

// a function that uses a callback named `cb` as a parameter
function getAsyncMessage(cb) {
    setTimeout(function () { cb("Hello World!") }, 1000);
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getAsyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

Ce qui imprime ce qui suit sur la console :

> Before getSyncMessage call
> After getSyncMessage call
// pauses for 1000 ms with no output
> Hello World!

L’exécution de la ligne va à la ligne 6 des journaux “Avant l’appel getSyncMessage”. Ensuite, l’exécution passe à la ligne 8 en appelant getAsyncMessage avec un rappel pour le paramètre cb. La ligne 3 est ensuite exécutée et appelle setTimeout avec un rappel comme premier argument et le nombre 300 comme deuxième argument. setTimeout fait tout ce qu’il fait et conserve ce rappel afin qu’il puisse l’appeler plus tard dans 1000 millisecondes, mais après avoir configuré le délai d’attente et avant de mettre en pause les 1000 millisecondes, il remet l’exécution là où elle s’était arrêtée afin qu’elle passe à ligne 4, puis ligne 11, puis s’arrête pendant 1 seconde et setTimeout appelle ensuite sa fonction de rappel qui ramène l’exécution à la ligne 3 où le rappel getAsyncMessages est appelé avec la valeur “Hello World” pour son paramètre message qui est ensuite enregistré à la console sur la ligne 9.

Fonctions de rappel dans Node.js #

NodeJS a des rappels asynchrones et fournit généralement deux paramètres à vos fonctions parfois appelés classiquement “err” et “data”. Un exemple avec la lecture d’un fichier texte.

const fs = require("fs");

fs.readFile("./test.txt", "utf8", function(err, data) {
    if(err) {
        // handle the error 
    } else {
        // process the file text given with data
    }
});

Il s’agit d’un exemple de rappel appelé une seule fois.

C’est une bonne pratique de gérer l’erreur d’une manière ou d’une autre, même si vous ne faites que la consigner ou la lancer. Le else n’est pas nécessaire si vous lancez ou retournez et peut être supprimé pour diminuer l’indentation tant que vous arrêtez l’exécution de la fonction actuelle dans le if en faisant quelque chose comme lancer ou retourner.

Bien qu’il soit courant de voir err, data, il se peut que vos rappels n’utilisent pas toujours ce modèle, il est préférable de consulter la documentation.

Un autre exemple de rappel provient de la bibliothèque express (express 4.x) :

// this code snippet was on http://expressjs.com/en/4x/api.html
const express = require('express');
const app = express();

// this app.get method takes a url route to watch for and a callback
// to call whenever that route is requested by a user.
app.get('/', function(req, res){
  res.send('hello world');
});

app.listen(3000);

Cet exemple montre un rappel qui est appelé plusieurs fois. Le rappel est fourni avec deux objets en tant que paramètres nommés ici comme req et res ces noms correspondent respectivement à la demande et à la réponse, et ils fournissent des moyens de visualiser la demande entrante et de configurer la réponse qui sera envoyée à l’utilisateur .

Comme vous pouvez le constater, un rappel peut être utilisé de différentes manières pour exécuter du code de synchronisation et asynchrone en JavaScript et les rappels sont très omniprésents dans JavaScript.

[1] : https://en.wikipedia.org/wiki/First-class_function [2] : https://developer.mozilla.org/en-US/docs/Glossary/Call_Stack [3] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/EventLoop [4] : https://stackoverflow.com/a/13806828/2066736

L’enfer des rappels #

L’enfer du rappel (également une pyramide de malheur ou un effet boomerang) survient lorsque vous imbriquez trop de fonctions de rappel dans une fonction de rappel. Voici un exemple pour lire un fichier (en ES6).

const fs = require('fs');
let filename = `${__dirname}/myfile.txt`;

fs.exists(filename, exists => {
    if (exists) {
        fs.stat(filename, (err, stats) => {
            if (err) {
                throw err;
            }
            if (stats.isFile()) {
                fs.readFile(filename, null, (err, data) => {
                    if (err) {
                        throw err;
                    }
                    console.log(data);
                });
            }
            else {
                throw new Error("This location contains not a file");
            }
        });
    }
    else {
        throw new Error("404: file not found");
    }
});

Comment éviter “l’enfer du rappel”

Il est recommandé de ne pas imbriquer plus de 2 fonctions de rappel. Cela vous aidera à maintenir la lisibilité du code et sera beaucoup plus facile à maintenir à l’avenir. Si vous avez besoin d’imbriquer plus de 2 rappels, essayez d’utiliser [événements distribués] [1] à la place.

Il existe également une bibliothèque appelée [async][2] qui permet de gérer les rappels et leur exécution disponible sur npm. Il augmente la lisibilité du code de rappel et vous donne plus de contrôle sur votre flux de code de rappel, notamment en vous permettant de les exécuter en parallèle ou en série.

[1] : https://nodejs.org/api/events.html [2] : https://caolan.github.io/async/

Promesses natives #

Les promesses sont un outil de programmation asynchrone. En JavaScript, les promesses sont connues pour leurs méthodes “then”. Les promesses ont deux états principaux ’en attente’ et ‘réglé’. Une fois qu’une promesse est “réglée”, elle ne peut pas redevenir “en attente”. Cela signifie que les promesses sont généralement bonnes pour les événements qui ne se produisent qu’une seule fois. L’état ‘réglé’ a deux états ainsi ‘résolu’ et ‘rejeté’. Vous pouvez créer une nouvelle promesse en utilisant le mot-clé new et en passant une fonction dans le constructeur new Promise(function (resolve, rejette) {}).

La fonction transmise au constructeur Promise reçoit toujours un premier et un deuxième paramètre généralement nommés respectivement resolve et reject. La dénomination de ces deux paramètres est conventionnelle, mais ils placeront la promesse soit dans l’état ‘résolu’, soit dans l’état ‘rejeté’. Lorsque l’un ou l’autre est appelé, la promesse passe de « en attente » à « réglée ». resolve est appelé lorsque l’action souhaitée, qui est souvent asynchrone, a été effectuée et reject est utilisé si l’action a généré une erreur.

Dans le timeout ci-dessous se trouve une fonction qui renvoie une Promise.

function timeout (ms) {
  return new Promise(function (resolve, reject) {
    setTimeout(function () {
      resolve("It was resolved!");
    }, ms)
  });
}

timeout(1000).then(function (dataFromPromise) {
  // logs "It was resolved!"
  console.log(dataFromPromise);
})

console.log("waiting...");

sortie console

waiting...
// << pauses for one second>>
It was resolved!

Lorsque timeout est appelé, la fonction passée au constructeur Promise est exécutée sans délai. Ensuite, la méthode setTimeout est exécutée et son rappel est configuré pour se déclencher dans les ms millisecondes suivantes, dans ce cas ms=1000. Étant donné que le rappel de setTimeout n’est pas encore déclenché, la fonction timeout rend le contrôle à la portée appelante. La chaîne de méthodes “then” est ensuite stockée pour être appelée ultérieurement lorsque/si la promesse est résolue. S’il y avait des méthodes catch ici, elles seraient également stockées, mais seraient déclenchées quand/si la promesse ‘rejecte’.

Le script affiche alors ’en attente…’. Une seconde plus tard, le setTimeout appelle son rappel qui appelle la fonction de résolution avec la chaîne “C’était résolu!”. Cette chaîne est ensuite transmise au rappel de la méthode then et est ensuite enregistrée pour l’utilisateur.

Dans le même sens, vous pouvez envelopper la fonction setTimeout asynchrone qui nécessite un rappel, vous pouvez envelopper toute action asynchrone singulière avec une promesse.

En savoir plus sur les promesses dans la documentation JavaScript https://www.wikiod.com/fr/javascript/promesses

Exemple de code #

Question : Quelle est la sortie du code ci-dessous et pourquoi ?

setTimeout(function() {
    console.log("A");
}, 1000);

setTimeout(function() {
    console.log("B");
}, 0);

getDataFromDatabase(function(err, data) {
    console.log("C");
    setTimeout(function() {
        console.log("D");
    }, 1000);
});

console.log("E");

Sortie : Ceci est connu avec certitude : “EBAD”. C est inconnu quand il sera enregistré.

Explication : Le compilateur ne s’arrêtera pas sur les méthodes setTimeout et getDataFromDatabase. Ainsi, la première ligne qu’il enregistrera est ‘E’. Les fonctions de rappel (premier argument de setTimeout) s’exécuteront après le délai défini de manière asynchrone !

Plus de détails:

  1. E n’a pas de setTimeout
  2. ‘B’ a un délai défini de 0 millisecondes
  3. A a un délai d’attente défini de 1000 millisecondes
  4. D doit demander une base de données, après quoi D doit attendre 1000 millisecondes pour qu’il vienne après A.
  5. ‘C’ est inconnu car il est inconnu lorsque les données de la base de données sont demandées. Il peut être avant ou après ‘A’.

Gestion des erreurs asynchrones #

Essayez d’attraper #

Les erreurs doivent toujours être traitées. Si vous utilisez la programmation synchrone, vous pouvez utiliser un try catch. Mais cela ne fonctionne pas si vous travaillez en asynchrone ! Exemple:

try {
    setTimeout(function() {
        throw new Error("I'm an uncaught error and will stop the server!");
    }, 100); 
}
catch (ex) {
    console.error("This error will not be work in an asynchronous situation: " + ex);
}

Les erreurs asynchrones ne seront gérées qu’à l’intérieur de la fonction de rappel !

Possibilités de travail #

Gestionnaires d’événements #

Les premières versions de Node.JS disposaient d’un gestionnaire d’événements.

process.on("UncaughtException", function(err, data) { 
    if (err) {
        // error handling
    }
});

Domaines #

A l’intérieur d’un domaine, les erreurs sont déclenchées via les émetteurs d’événements. En utilisant ceci, toutes les erreurs, minuteries, méthodes de rappel sont enregistrées implicitement uniquement à l’intérieur du domaine. Par une erreur, être un événement d’erreur envoyé et n’a pas planté l’application.

var domain = require("domain");
var d1 = domain.create();
var d2 = domain.create();

d1.run(function() {
    d2.add(setTimeout(function() {
        throw new Error("error on the timer of domain 2");
    }, 0));
});

d1.on("error", function(err) {
    console.log("error at domain 1: " + err);
});

d2.on("error", function(err) {
    console.log("error at domain 2: " + err);
});
