Mise en cache des réponses HTTP #

Un cas d’utilisation typique de RxJS consiste à créer des requêtes HTTP et à mettre en cache leurs résultats pendant un certain temps. De plus, nous voulons toujours exécuter une seule requête à la fois et partager sa réponse.

Par exemple, le code suivant met en cache 1 élément pour max. 1 000 ms :

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1);

var counter = 1;    
function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100);
}

function requestCachedHttpResult() {
    return updateRequest;
}

La fonction makeMockHttpRequest() simule une requête HTTP qui arrive avec un retard de 100ms.

La fonction requestCachedHttpResult() est l’endroit où nous nous abonnons pour obtenir une réponse réelle ou mise en cache.

Avec .publishReplay(1, 1000) nous avons utilisé la multidiffusion RxJS pour utiliser en interne ReplaySubject et conservez 1 élément pour un maximum de 1000ms. Alors refCount() est utilisé pour garder toujours un seul abonnement à la source qui est Observable.defer(). Cet Observable est utilisé pour créer une nouvelle requête et incrémente “counter” pour prouver que les valeurs mises en cache et les nouveaux abonnements partagent les mêmes données.

Lorsque nous voulons obtenir des données actuelles, nous appelons requestCachedHttpResult(). Pour s’assurer que l’observateur sera complété correctement après l’émission des données, nous avons utilisé l’opérateur take(1).

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

Cela crée une requête unique avec mockDataFetch() et imprime sur la console :

1

Un exemple plus compliqué appellera plusieurs requêtes à différents moments où nous voulons tester que les connexions HTTP simulées et les réponses sont partagées.

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 50:", val))
, 50);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 200:", val))
, 200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1200:", val))
, 1200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1500:", val))
, 1500);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 3500:", val))
, 3500);

Voir la démo en direct : https://jsbin.com/todude/5/edit?js,console

Chaque demande est envoyée avec retard et se produit dans l’ordre suivant :

0 - Première requête qui fait que le refCount() s’abonne à sa source qui fait l’appel mockDataFetch(). Sa réponse va être retardée de 100ms. À ce moment, ConnectableObservable à l’intérieur de l’opérateur publishReplay() a un observateur.

50 - La deuxième requête s’abonne également à ConnectableObservable. À ce moment, ConnectableObservable à l’intérieur de l’opérateur publishReplay() a deux Observer. Il ne crée pas d’autre requête avec makeMockHttpRequest() car refCount() est déjà abonné.

100 - La première réponse est prête. Il est d’abord mis en cache par ReplaySubject puis réémis aux deux Observers abonnés à ConnectableObservable. Les deux observateurs sont complétés grâce à take(1) et désabonnés.

200 - S’abonne au ReplaySubject qui émet immédiatement sa valeur en cache, ce qui fait que take(1) termine l’Observer et se désabonne immédiatement. Aucune requête HTTP n’est effectuée et aucun abonnement n’est conservé.

1200 - Identique au premier événement à 0. À ce stade, la valeur mise en cache a été supprimée, car elle est antérieure à 1 000 ms.

1500 - Identique au quatrième événement à 200.

3500 - Identique au premier événement à 1200.

La sortie dans la console est la suivante :

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: 2
Response 1500: 2
Response 3500: 3

Dans RxJS 5, une fonctionnalité similaire était couverte par l’opérateur cache(). Cependant, il a été supprimé dans 5.0.0-rc.1 en raison de sa fonctionnalité limitée.

Erreurs de manipulation

Si nous voulons gérer les erreurs produites par le service distant (la fonction makeMockHttpRequest), nous devons les attraper avant qu’elles ne soient fusionnées dans la chaîne Observable principale car toute erreur reçue par ReplaySubject à l’intérieur de publishReplay() marquerait son état interne comme stopped (Lire plus ici https://www.wikiod.com/fr/rxjs/matiere#Sujet et son état interne ) ce qui n’est certainement pas ce que nous voulons .

Dans l’exemple suivant, nous simulons une erreur lorsque counter === 2 et l’attrapons avec l’opérateur catch(). Nous utilisons catch() pour transformer uniquement la notification error en un next normal afin que nous puissions gérer l’erreur dans les observateurs :

function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100)
        .map(i => {
            if (i === 2) {
                throw new Error('Invalid URL');
            }
            return i;
        })
        .catch(err => Observable.of(err));
}

Voir la démo en direct : https://jsbin.com/kavihu/10/edit?js,console

Cela imprimera sur la console la sortie suivante. Notez que les erreurs sont reçues dans les gestionnaires “next” :

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: [object Error] { ... }
Response 1500: [object Error] { ... }
Response 3500: 3

Si nous voulons gérer les erreurs comme des notifications “error” régulières dans chaque observateur, nous devons les renvoyer après l’opérateur “publishReplay()” pour les raisons expliquées ci-dessus.

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1)
    .map(val => {
        if (val instanceof Error) {
            throw val;
        }
        return val;
    });

Voir la démo en direct : https://jsbin.com/fabosam/5/edit?js,console (notez que nous avons également dû ajouter des rappels d’erreur pour chaque observateur).

Response 0: 1
Response 50: 1
Response 200: 1
error callback: Error: Invalid URL
error callback: Error: Invalid URL
Response 3500: 3

Envoi de plusieurs requêtes HTTP parallèles #

Un cas d’utilisation très courant dans les applications Web consiste à effectuer plusieurs requêtes asynchrones (par exemple, HTTP) et à collecter leurs résultats au fur et à mesure qu’ils arrivent ou tous à la fois (par exemple, dans Angular2 avec le [service HTTP] (https://angular. io/docs/ts/latest/guide/server-communication.html)).

1. Rassembler les réponses asynchrones une par une au fur et à mesure qu’elles arrivent

Cela se fait généralement avec l’opérateur mergeMap() qui prend une fonction de projection qui doit renvoie un Observable. L’opérateur mergeMap() s’abonne en interne à chaque Observable immédiatement, même si l’Observable précédent n’est pas encore terminé.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .subscribe(val => console.log(val));

Cela imprime les réponses sur la console dans un ordre différent en raison du délai aléatoire :

Response from url-3
Response from url-4
Response from url-2
Response from url-1

Voir la démo en direct : https://jsbin.com/xaqudan/2/edit?js,console

Chaque réponse (item émis via l’appel next) est réémise par mergeMap() immédiatement.

Pour notre objectif d’envoyer plusieurs requêtes HTTP, il est utile de mentionner que mergeMap() peut prendre trois arguments au total :

  1. La fonction de projection qui doit renvoyer un Observable.
  2. La fonction de sélection de résultat qui nous permet de modifier le résultat avant de l’émettre davantage.
  3. Le nombre d’Observables souscrits simultanément.

Contrôle du nombre de requêtes parallèles

Avec le troisième argument, nous pouvons contrôler le nombre de requêtes parallèles que nous traiterons (en supposant que chaque Observable effectuant une requête HTTP est “froid”).

Dans l’exemple suivant, nous n’exécuterons que 2 requêtes en même temps.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url), undefined, 2)
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Voir la démo en direct : https://jsbin.com/sojejal/4/edit?js,console

Notez que les deux premières requêtes se sont terminées après 1 s tandis que les deux autres après 2 s.

[1004, "Response from url-1"]
[1010, "Response from url-2"]
[2007, "Response from url-3"]
[2012, "Response from url-4"]

Erreurs de manipulation

Si l’un des Observables source échoue (envoie une notification d’erreur), le mergeMap() renvoie l’erreur en tant que error. Dans le cas où nous voulons que chaque Observable échoue gracieusement, nous devons utiliser par exemple catch() opérateur.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url).catch(() => Observable.empty()))
  .subscribe(val => console.log(val));

La réponse pour url-3 génère une erreur qui est envoyée en tant que notification error. Ceci est ensuite intercepté par l’opérateur catch() et remplacé par Observable.empty() qui est juste une notification complete. Pour cette raison, cette réponse est ignorée.

Le résultat de cet exemple est le suivant :

Response from url-4
Response from url-1
Response from url-2

Voir la démo en direct : https://jsbin.com/kuqumud/4/edit?js,console

2. Rassembler toutes les réponses asynchrones en une seule fois

En suivant les exemples précédents, nous pourrions rassembler toutes les réponses avec l’opérateur toArray().

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .toArray()
  .subscribe(val => console.log(val));

Cependant, l’utilisation de l’opérateur toArray() a une conséquence importante. La réception des résultats par l’abonné n’est pas seulement contrôlée en complétant toutes les requêtes HTTP, mais également en complétant la source Observable (Observable.from dans notre cas). Cela signifie que nous ne pouvons pas utiliser des Observables source qui ne se terminent jamais (par exemple, Observable.fromEvent).

Une autre façon d’obtenir le même résultat consiste à utiliser Observable.forkJoin() qui prend comme argument un tableau d’Observables auquel nous voulons nous abonner et attendons que tous ** émettent au moins une valeur et se terminent **.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];
var observables = urls.map(url => mockHTTPRequest(url));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Cela imprime toutes les réponses dans un seul tableau :

["Response from url-1", "Response from url-2", "Response from url-3", "Response from url-4"]

Voir la démo en direct : https://jsbin.com/fomoye/2/edit?js,console

Le Observable.forkJoin() prend également comme argument optionnel une fonction de sélection de résultat qui nous permet de modifier le résultat final avant de l’émettre davantage :

Observable.forkJoin(observables, (...results) => {
    return results.length;
  })
  .subscribe(val => console.log(val));

Cela imprime sur la console :

4

Voir la démo en direct : https://jsbin.com/muwiqic/1/edit?js,console

Notez que les arguments de la fonction de sélection de résultat sont décompressés.

Erreurs de manipulation

Pour la gestion des erreurs, nous pouvons utiliser la même approche que dans l’exemple précédent avec l’opérateur catch().

Cependant, il y a une chose importante à savoir. Le forkJoin() nécessite que chaque source Observable émette au moins une valeur. Si nous utilisions catch(() => Observable.empty()) comme nous le faisions avant, forkJoin() n’émettrait jamais rien car Observable.empty() n’est qu’une notification complete. C’est pourquoi nous devons utiliser par exemple Observable.of(null) qui est une valeur null suivie d’une notification complete.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

var observables = urls.map(url => mockHTTPRequest(url).catch(() => Observable.of(null)));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Voir la démo en direct : https://jsbin.com/yidiked/2/edit?js,console

Cela imprime sur la console :

["Response from url-1", "Response from url-2", null, "Response from url-4"]

Notez que l’erreur est remplacée par null. Si nous utilisions uniquement Observable.empty(), forkJoin() n’émettrait jamais rien.

Supprimer les appels de repos lents/obsolètes #

Un cas d’utilisation courant consiste à supprimer certains appels restants, qui ne sont plus nécessaires après certaines entrées utilisateur. L’exemple le plus frappant serait, lorsqu’un utilisateur utilise une fonction de recherche, fait une demande, fait une autre demande et pour une raison quelconque, la première demande arrive après la deuxième demande et l’application affiche les données obsolètes de l’ancienne demande.

C’est un cas d’utilisation parfait pour l’opérateur switchMap.

searchInput$
    .switchMap(() => makeRestCall());

Dans ce cas, le flux “passera” à l’appel au repos, mais seulement jusqu’à ce que de nouvelles données soient émises sur le searchInput$, puis le flux à l’intérieur de la “switchMap” est rejeté et un nouvel appel au repos est effectué. Ainsi, un résultat de repos ne sera pris en compte que s’il s’est terminé avant le prochain clic.

Et voici un exemple simulé à part entière :

// some initial data-mocking
const Observable = Rx.Observable;
var counter = 1;
function mockDataFetch() {
    return Observable.of(counter++)
        .delay(500);
}

// the recipe

const searchInput$ = new Rx.Subject();
searchInput$
    .do(searchInput => console.log("Searching for " + searchInput))
    .switchMap(searchInput => mockDataFetch()
              .map(result => ({result, searchInput}))
     )
    .do(data => console.log("Received result for " + data.searchInput + ": " + data.result))
    .subscribe();

// simulating search inputs
searchInput$.next("google");
setTimeout(() => searchInput$.next("froogle"), 600);
setTimeout(() => searchInput$.next("doodle"), 800);
setTimeout(() => searchInput$.next("poodle"), 1000);
setTimeout(() => searchInput$.next("noodle"), 1600);

Voir la démo en direct : https://jsbin.com/suzakahoro/1/edit?js,console

Envoi de plusieurs requêtes HTTP séquentielles #

Faire une séquence de requêtes HTTP a deux raisons principales :

  • Les requêtes dépendent les unes des autres (le résultat d’une requête est requis pour une requête consécutive).

  • Nous voulons répartir la charge du serveur en plusieurs requêtes.

1. Faire plusieurs requêtes dépendantes

Cela peut être effectué à l’aide de l’opérateur concatMap() pour transformer une réponse en paramètres requis pour la demande consécutive.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

function timestamp() {
  return (new Date()).getTime() - start;
}

var start = (new Date()).getTime();

Observable.of('url-1') 
  // first request
  .concatMap(url => {
    console.log(timestamp() + ': Sending request to ' + url);
    return mockHTTPRequest(url);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // second request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create new requests url
    return mockHTTPRequest(newUrl);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // third request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create another requests url
    return mockHTTPRequest(newUrl);
  })
  .subscribe(response => console.log(timestamp() + ': ' + response));

L’opérateur concatMap() s’abonne en interne à l’Observable renvoyé à partir de sa fonction de projection et attend qu’il se termine tout en réémettant toutes ses valeurs.

Cet exemple horodate chaque requête et réponse :

3: Sending request to url-1
1010: Response from url-1
1011: Sending request to Response from url-1
2014: Response from url-19
2015: Sending request to Response from url-19
3017: Response from url-20

Voir la démo en direct : https://jsbin.com/fewidiv/6/edit?js,console

Erreurs de manipulation

Si l’une des requêtes HTTP échoue, nous ne voulons évidemment pas continuer car nous ne sommes pas en mesure de construire la requête suivante.

2. Faire des demandes consécutives

Si nous ne sommes pas intéressés par la réponse de la requête HTTP précédente, nous pouvons simplement prendre un tableau d’URL et les exécuter l’une après l’autre.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Cet exemple imprime des réponses horodatées :

[1006, "Response from url-1"]
[2012, "Response from url-2"]
[3014, "Response from url-3"]
[4016, "Response from url-4"]

Voir la démo en direct : https://jsbin.com/kakede/3/edit?js,console

** Retarder les appels consécutifs **

Nous pourrions également vouloir faire un petit délai entre chaque demande. Dans ce cas, nous devons ajouter delay() après chaque appel mockHTTPRequest().

Observable.from(urls)
  .concatMap(url => {
    return mockHTTPRequest(url)
      .do(response => console.log(((new Date()).getTime() - start) + ': Sending request to ' + url))
      .delay(500);
  })
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Cela imprime pour consoler la sortie suivante :

2024: Sending request to url-1
[2833, "Response from url-1"]
4569: Sending request to url-2
[5897, "Response from url-2"]
7880: Sending request to url-3
[8674, "Response from url-3"]
9789: Sending request to url-4
[10796, "Response from url-4"]

Voir la démo en direct : https://jsbin.com/kakede/4/edit?js,console

Erreurs de manipulation

Si nous voulons simplement ignorer l’échec de l’une des requêtes HTTP, nous devons enchaîner catch() après chaque mockHTTPRequest().

function mockHTTPRequest(url) {
  if (url == 'url-3') {
    return Observable.throw(new Error(`Request ${url} failed.`));
  } else {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
  }
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url).catch(obs => Observable.empty()))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Cela ignore simplement l’appel url-3 :

[1004, "Response from url-1"]
[2012, "Response from url-2"]
[3016, "Response from url-4"]

Voir la démo en direct : https://jsbin.com/jowiqo/2/edit?js,console

Si nous n’utilisions pas l’opérateur catch(), url-3 entraînerait l’envoi d’une notification d’erreur par la chaîne et le dernier url-4 ne serait pas exécuté.

Voir la démo en direct : https://jsbin.com/docapim/3/edit?js,console

Limitation du débit #

Un problème courant avec les services distants est la limitation du débit. Le service à distance nous permet d’envoyer uniquement un nombre limité de demandes ou une quantité de données par période.

Dans RxJS 5, une fonctionnalité très similaire est fournie par l’opérateur bufferTime et surtout si nous laissez le deuxième paramètre non spécifié (il définit la fréquence à laquelle nous voulons créer un nouveau tampon. Si nous le laissons indéfini/null, il créera un nouveau tampon juste après avoir émis celui en cours).

Une utilisation typique de bufferTime ressemblera à ceci :

bufferTime(1000, null, 5)

Cela mettra les éléments en mémoire tampon jusqu’à ce que l’une des deux conditions soit remplie. Ensuite, il émettra le tampon et en démarrera un autre :

  • l’opérateur a collecté des objets pendant 1000ms
  • l’opérateur a déjà collecté 5 éléments

À des fins de démonstration, nous pouvons créer une source Observable qui émet très rapidement afin que le bufferTime atteigne la limite de taille (5) et émette plus d’une fois toutes les 1000ms :

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

Ensuite, nous l’enchaînerons avec bufferTime et concatMap. L’opérateur concatMap est l’endroit où nous forçons le délai de 1000ms :

const startTime = (new Date()).getTime();

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.of(buffer).delay(1000))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Voir la démo en direct : https://jsbin.com/kotibow/3/edit?js,console

Nous avons également ajouté timestamp() pour voir les temps d’émission afin de s’assurer que le délai est vraiment d’au moins 1000ms.

Notez que nous n’avons pas du tout eu besoin d’utiliser Observable.of(buffer). Nous l’utilisons ici uniquement pour vérifier manuellement que le nombre d’éléments mis en mémoire tampon est correct.

D’après la sortie de la console, nous pouvons voir que le délai entre deux émissions est d’environ 1000 ms :

Timestamp { value: [ 1, 2, 3, 4, 5 ], timestamp: 1475 }
Timestamp { value: [ 6, 7, 8, 9, 10 ], timestamp: 2564 }
Timestamp { value: [ 11, 12, 13, 14, 15 ], timestamp: 3567 }
Timestamp { value: [ 16, 17, 18, 19, 20 ], timestamp: 4572 }

Maintenant, nous pouvons également tester une situation où la source émet lentement afin que l’opérateur bufferTime atteigne la condition d’intervalle maximum :

const source = Observable.range(1, 25)

Si nous voulions utiliser cette approche dans une application du monde réel, nous placerions l’appel distant dans l’opérateur concatMap. De cette façon, nous pouvons contrôler si nous voulons forcer le délai “1s” entre les requêtes ou les réponses du service distant.

Par exemple, nous pouvons forcer le délai minimum de 1s entre les requêtes en utilisant forkJoin dans le rappel concatMap :

function mockHTTPRequest(buffer) {
  return Observable.of(true).delay(Math.random() * 1500)
}

const startTime = (new Date()).getTime();
const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.forkJoin(
    mockHTTPRequest(buffer),
    Observable.of(buffer).delay(1000)
  ))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Voir la démo en direct : https://jsbin.com/xijaver/edit?js,console

Grâce à forkJoin, concatMap doit attendre que les deux Observables se terminent.

D’un autre côté, si nous voulions forcer un délai de 1s entre les réponses, nous ajouterions simplement l’opérateur delay() après mockHTTPRequest() :

.concatMap(buffer => mockHTTPRequest(buffer).delay(1000))
