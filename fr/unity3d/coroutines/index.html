Syntaxe #

  • Coroutine publique StartCoroutine (routine IEnumerator);
  • public Coroutine StartCoroutine(string methodName, object value = null);
  • public void StopCoroutine(string methodName);
  • public void StopCoroutine (routine IEnumerator);
  • public void StopAllCoroutines();

Considérations relatives aux performances #

Il est préférable d’utiliser les coroutines avec modération car la flexibilité a un coût de performance.

  • Les coroutines en grand nombre exigent plus du processeur que les méthodes de mise à jour standard.
  • Il existe un problème dans certaines versions de Unity où les coroutines produisent des ordures à chaque cycle de mise à jour en raison du fait que Unity boxe la valeur de retour MoveNext. Cela a été observé pour la dernière fois dans 5.4.0b13. ([Rapport de bogue][1])

[1] : https://issuetracker.unity3d.com/issues/coroutines-generate-garbage-in-movenext

Réduisez les déchets en mettant en cache YieldInstructions #

Une astuce courante pour réduire les déchets générés dans les coroutines consiste à mettre en cache le YieldInstruction.

IEnumerator TickEverySecond()
{
    var wait = new WaitForSeconds(1f); // Cache
    while(true)
    {
        yield return wait; // Reuse
    }
}

Rendre “null” ne produit pas de déchets supplémentaires.

Coroutines #

Tout d’abord, il est essentiel de comprendre que les moteurs de jeu (tels que Unity) fonctionnent sur un paradigme “basé sur les images”.

Le code est exécuté à chaque image.

Cela inclut le propre code de Unity et votre code.

Lorsque vous pensez aux cadres, il est important de comprendre qu’il n’y a *** absolument *** aucune garantie quant au moment où les cadres se produisent. Ils *** ne se produisent pas *** sur un rythme régulier. Les écarts entre les images pourraient être, par exemple, 0,02632 puis 0,021167 puis 0,029778, et ainsi de suite. Dans l’exemple, ils sont tous “environ” 1/50e de seconde, mais ils sont tous différents. Et à tout moment, vous pouvez obtenir un cadre qui prend beaucoup plus de temps, ou plus court ; et votre code peut être exécuté à tout moment dans le cadre.

Gardant cela à l’esprit, vous pouvez vous demander : comment accédez-vous à ces cadres dans votre code, dans Unity ?

Tout simplement, vous utilisez soit l’appel Update(), soit vous utilisez une coroutine. (En effet - ils sont exactement la même chose : ils permettent au code d’être exécuté à chaque image.)

Le but d’une coroutine est que :

vous pouvez exécuter du code, puis “arrêter et attendre” *** jusqu’à une future image ***.

Vous pouvez attendre jusqu’à l’image suivante, vous pouvez attendre un certain nombre d’images, ou vous pouvez attendre un certain temps approximatif en secondes dans le futur.

Par exemple, vous pouvez attendre “environ une seconde”, ce qui signifie qu’il attendra environ une seconde, puis placez votre code dans une image d’environ une seconde à partir de maintenant. (Et en effet, dans ce cadre, le code pourrait être exécuté à tout moment, quel qu’il soit.) Pour répéter : ce ne sera pas exactement une seconde. Un timing précis n’a pas de sens dans un moteur de jeu.

Dans une coroutine :

Pour attendre une image :

// do something
yield return null;  // wait until next frame
// do something

Attendre trois images :

// do something
yield return null;  // wait until three frames from now
yield return null;
yield return null;
// do something

Pour attendre environ une demi-seconde :

// do something
yield return new WaitForSeconds (0.5f); // wait for a frame in about .5 seconds
// do something

Faites quelque chose à chaque image :

while (true)
{
    // do something
    yield return null;  // wait until the next frame
}

Cet exemple est littéralement identique à simplement mettre quelque chose à l’intérieur de l’appel “Update” de Unity : le code à “faire quelque chose” est exécuté à chaque image.

Exemple #

Attachez Ticker à un GameObject. Pendant que cet objet de jeu est actif, la coche s’exécutera. Notez que le script arrête soigneusement la coroutine, lorsque l’objet de jeu devient inactif ; c’est généralement un aspect important de l’utilisation correcte de la coroutine.

using UnityEngine;
using System.Collections;

public class Ticker:MonoBehaviour {

    void OnEnable()
    {
        StartCoroutine(TickEverySecond());
    }

    void OnDisable()
    {
        StopAllCoroutines();
    }

    IEnumerator TickEverySecond()
    {
        var wait = new WaitForSeconds(1f); // REMEMBER: IT IS ONLY APPROXIMATE
        while(true)
        {
            Debug.Log("Tick");
            yield return wait;  // wait for a frame, about 1 second from now
        }
    }
}

Chaînage de coroutines #

Les coroutines peuvent céder à l’intérieur d’elles-mêmes et attendre ** d’autres coroutines **.

Ainsi, vous pouvez enchaîner des séquences - “l’une après l’autre”.

C’est très simple et c’est une technique de base dans Unity.

C’est absolument naturel dans les jeux que certaines choses doivent se passer “dans l’ordre”. Presque chaque “tour” d’un jeu commence par une certaine série d’événements se produisant, sur un espace de temps, dans un certain ordre. Voici comment vous pourriez commencer un jeu de course automobile :

IEnumerator BeginRace()
{
  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());
}

Ainsi, lorsque vous appelez BeginRace …

 StartCoroutine(BeginRace());

Il exécutera votre routine “préparer la course”. (Peut-être, faire clignoter des lumières et faire du bruit de foule, réinitialiser les scores, etc.) Lorsque cela est terminé, il exécutera votre séquence de “compte à rebours”, où vous animerez peut-être un compte à rebours sur l’interface utilisateur. Lorsque cela sera terminé, il exécutera votre code de démarrage de course, où vous exécuterez peut-être des effets sonores, démarrerez certains pilotes AI, déplacerez la caméra d’une certaine manière, etc.

Pour plus de clarté, comprenez que les trois appels

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

doivent eux-mêmes être dans une coroutine. C’est-à-dire qu’ils doivent être dans une fonction de type IEnumerator. Donc, dans notre exemple, c’est IEnumerator BeginRace. Ainsi, à partir du code “normal”, vous lancez cette coroutine avec l’appel StartCoroutine.

 StartCoroutine(BeginRace());

Pour mieux comprendre le chaînage, voici une fonction qui enchaîne les coroutines. Vous passez dans un tableau de coroutines. La fonction exécute autant de coroutines que vous passez, dans l’ordre, les unes après les autres.

// run various routines, one after the other
IEnumerator OneAfterTheOther( params IEnumerator[] routines ) 
{
    foreach ( var item in routines ) 
    {
        while ( item.MoveNext() ) yield return item.Current;
    }

    yield break;
}

Voici comment vous appelleriez cela… disons que vous avez trois fonctions. Rappelez-vous qu’ils doivent tous être IEnumerator :

IEnumerator PrepareRace() 
{
    // codesay, crowd cheering and camera pan around the stadium
    yield break;
}

IEnumerator Countdown() 
{
    // codesay, animate your countdown on UI
    yield break;
}

IEnumerator StartRace() 
{
    // codesay, camera moves and light changes and launch the AIs
    yield break;
}

Tu l’appellerais comme ça

StartCoroutine( MultipleRoutines( PrepareRace(), Countdown(), StartRace() ) );

ou peut-être comme ça

IEnumerator[] routines = new IEnumerator[] {
     PrepareRace(),
     Countdown(),
     StartRace() };
StartCoroutine( MultipleRoutines( routines ) );

Pour répéter, l’une des exigences les plus élémentaires dans les jeux est que certaines choses se produisent les unes après les autres “dans une séquence” au fil du temps. Vous y parvenez très simplement dans Unity, avec

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

Terminer une coroutine #

Souvent, vous concevez des coroutines pour qu’elles se terminent naturellement lorsque certains objectifs sont atteints.

IEnumerator TickFiveSeconds()
{
    var wait = new WaitForSeconds(1f);
    int counter = 1;
    while(counter < 5)
    {
        Debug.Log("Tick");
        counter++;
        yield return wait;
    }
    Debug.Log("I am done ticking");
}

Pour arrêter une coroutine de “l’intérieur” de la coroutine, vous ne pouvez pas simplement “revenir” comme vous le feriez pour quitter plus tôt une fonction ordinaire. Au lieu de cela, vous utilisez yield break.

IEnumerator ShowExplosions()
{
    ... show basic explosions
    if(player.xp < 100) yield break;
    ... show fancy explosions
}

Vous pouvez également forcer toutes les coroutines lancées par le script à s’arrêter avant de terminer.

void OnDisable()
{
    // Stops all running coroutines
    StopAllCoroutines();
}

La méthode pour arrêter une coroutine spécifique de l’appelant varie selon la façon dont vous l’avez démarrée.

Si vous avez démarré une coroutine par nom de chaîne :

StartCoroutine("YourAnimation");

vous pouvez alors l’arrêter en appelant StopCoroutine avec le même nom de chaîne :

StopCoroutine("YourAnimation");

Alternativement, vous pouvez conserver une référence à soit le IEnumerator renvoyé par la méthode coroutine, ou l’objet Coroutine renvoyé par StartCoroutine, et appeler StopCoroutine sur l’un ou l’autre :

public class SomeComponent : MonoBehaviour 
{
    Coroutine routine;

    void Start () {
        routine = StartCoroutine(YourAnimation());
    }

    void Update () {
        // later, in response to some input...
        StopCoroutine(routine);
    }

    IEnumerator YourAnimation () { /* ... */ }
}

Méthodes MonoBehavior qui peuvent être des Coroutines #

Il existe trois méthodes MonoBehaviour qui peuvent être faites de coroutines.

  1. Démarrer()
  2. OnBecameVisible()
  3. OnLevelWasLoaded()

Cela peut être utilisé pour créer, par exemple, des scripts qui s’exécutent uniquement lorsque l’objet est visible par une caméra.

using UnityEngine;
using System.Collections;

public class RotateObject : MonoBehaviour
{
    IEnumerator OnBecameVisible()
    {
        var tr = GetComponent<Transform>();
        while (true)
        {
            tr.Rotate(new Vector3(0, 180f * Time.deltaTime));
            yield return null;
        }
    }
    
    void OnBecameInvisible()
    {
        StopAllCoroutines();
    }
}

Façons de céder #

Vous pouvez attendre la prochaine image.

yield return null; // wait until sometime in the next frame

Vous pouvez avoir plusieurs de ces appels à la suite, pour simplement attendre autant de trames que vous le souhaitez.

//wait for a few frames
yield return null;
yield return null;

Attendez environ n secondes. Il est extrêmement important de comprendre qu’il ne s’agit que d’un moment très approximatif.

yield return new WaitForSeconds(n);

Il n’est absolument pas possible d’utiliser l’appel “WaitForSeconds” pour toute forme de synchronisation précise.

Souvent, vous souhaitez enchaîner les actions. Alors, faites quelque chose, et quand c’est fini, faites autre chose, et quand c’est fini, faites autre chose. Pour y parvenir, attendez une autre coroutine :

yield return StartCoroutine(coroutine);

Comprenez que vous ne pouvez appeler cela que depuis une coroutine. Alors:

StartCoroutine(Test());

C’est ainsi que vous démarrez une coroutine à partir d’un morceau de code “normal”.

Ensuite, à l’intérieur de cette coroutine en cours d’exécution :

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

Cela imprimera A, démarrera le long processus et imprimera immédiatement B. Il n’attendra pas que le long processus se termine. D’autre part:

Debug.Log("A");
yield return StartCoroutine(LongProcess());
Debug.Log("B");

Cela imprimera A, démarrera le long processus, attendra qu’il soit terminé, puis imprimera B.

Il est toujours utile de se rappeler que les coroutines n’ont absolument aucun lien, de quelque manière que ce soit, avec le threading. Avec ce code :

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

il est facile de le considérer comme “comme” démarrer le LongProcess sur un autre thread en arrière-plan. Mais c’est absolument faux. C’est juste une coroutine. Les moteurs de jeu sont basés sur des cadres et les “coroutines” dans Unity vous permettent simplement d’accéder aux cadres.

Il est très facile d’attendre la fin d’une requête Web.

void Start() {
    string url = "http://google.com";
    WWW www = new WWW(url);
    StartCoroutine(WaitForRequest(www));
}

IEnumerator WaitForRequest(WWW www) {
    yield return www;
    
    if (www.error == null) {
        //use www.data);
    }
    else {
        //use www.error);
    }
}

Pour être complet : dans de très rares cas, vous utilisez la mise à jour fixe dans Unity ; il y a un appel WaitForFixedUpdate() qui normalement ne serait jamais utilisé. Il existe un appel spécifique (WaitForEndOfFrame() dans la version actuelle de Unity) qui est utilisé dans certaines situations en relation avec la génération de captures d’écran pendant le développement. (Le mécanisme exact change légèrement au fur et à mesure de l’évolution de Unity, alors recherchez les dernières informations sur Google si cela est pertinent.)
