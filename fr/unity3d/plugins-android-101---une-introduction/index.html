Cette rubrique est la première partie d’une série sur la création de plugins Android pour Unity. Commencez ici si vous avez peu ou pas d’expérience dans la création de plugins et/ou le système d’exploitation Android.

A travers cette série, j’utilise abondamment des liens externes que je vous encourage à lire. Bien que des versions paraphrasées du contenu pertinent soient incluses ici, il peut y avoir des moments où la lecture supplémentaire sera utile.

Commencer avec les plugins Android #

Actuellement, Unity propose deux façons d’appeler le code Android natif.

  1. Écrivez du code Android natif en Java et appelez ces fonctions Java à l’aide de C #
  2. Écrivez du code C # pour appeler directement les fonctions qui font partie du système d’exploitation Android

Pour interagir avec le code natif, Unity fournit certaines classes et fonctions.

  • [AndroidJavaObject][1] - Il s’agit de la classe de base fournie par Unity pour interagir avec le code natif. Presque tous les objets renvoyés par le code natif peuvent être stockés sous et AndroidJavaObject
  • [AndroidJavaClass][2] - Hérite d’AndroidJavaObject. Ceci est utilisé pour référencer des classes dans votre code natif
  • [Get][3] / [Set][4] valeurs d’une instance d’un objet natif, et les versions statiques [GetStatic][5] / [SetStatic][6]
  • [Call][7] / [CallStatic][8] pour appeler des fonctions natives non statiques et statiques

Aperçu de la création d’un plugin et de la terminologie #

  1. Écrire du code Java natif dans [Android Studio][9]
  2. Exportez le code dans un fichier JAR / AAR (Étapes ici pour [fichiers JAR][10] et [fichiers AAR][11])
  3. Copiez le fichier JAR/AAR dans votre projet Unity sous Assets/Plugins/Android
  4. Écrivez du code dans Unity (C# a toujours été la voie à suivre ici) pour appeler des fonctions dans le plugin

Notez que les trois premières étapes s’appliquent UNIQUEMENT si vous souhaitez avoir un plugin natif !

À partir de maintenant, je ferai référence au fichier JAR / AAR en tant que ** plugin natif ** et au script C # en tant que ** wrapper C # **


Choisir entre les méthodes de création de plugin #

Il est immédiatement évident que la première façon de créer des plugins est longue, donc choisir votre itinéraire semble discutable. Cependant, la méthode 1 est le SEUL moyen d’appeler du code personnalisé. Alors, comment choisir ?

En termes simples, est-ce que votre plugin

  1. Impliquez du code personnalisé - Choisissez la méthode 1
  2. Invoquer uniquement les fonctions Android natives ? - Choisissez la méthode 2

Veuillez NE PAS essayer de “mélanger” (c’est-à-dire une partie du plugin en utilisant la méthode 1, et l’autre en utilisant la méthode 2) les deux méthodes ! Bien que tout à fait possible, il est souvent peu pratique et pénible à gérer.
