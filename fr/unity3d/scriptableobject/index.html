ScriptableObjects avec AssetBundles #

Faites attention lorsque vous ajoutez des préfabriqués à AssetBundles s’ils contiennent des références à ScriptableObjects. Étant donné que les ScriptableObjects sont essentiellement des actifs, Unity en crée des doublons avant de les ajouter aux AssetBundles, ce qui peut entraîner un comportement indésirable lors de l’exécution.

Lorsque vous chargez un tel GameObject à partir d’un AssetBundle, il peut être nécessaire de réinjecter les actifs ScriptableObject dans les scripts chargés, en remplaçant ceux regroupés. Voir [Injection de dépendance] [1]

[1] : https://www.wikiod.com/fr/design-patterns/injection-de-dependance

Introduction #

Les ScriptableObjects sont des objets sérialisés qui ne sont pas liés à des scènes ou à des objets de jeu comme le sont les MonoBehaviours. Pour le dire d’une manière, ce sont des données et des méthodes liées aux fichiers d’actifs à l’intérieur de votre projet. Ces ressources ScriptableObject peuvent être transmises à MonoBehaviours ou à d’autres ScriptableObjects, où leurs méthodes publiques sont accessibles.

En raison de leur nature d’actifs sérialisés, ils constituent d’excellentes classes de gestionnaires et sources de données.

Création d’actifs ScriptableObject #

Vous trouverez ci-dessous une implémentation simple de ScriptableObject.

using UnityEngine;

[CreateAssetMenu(menuName = "StackOverflow/Examples/MyScriptableObject")]
public class MyScriptableObject : ScriptableObject
{
    [SerializeField]
    int mySerializedNumber;

    int helloWorldCount = 0;

    public void HelloWorld()
    {
        helloWorldCount++;
        Debug.LogFormat("Hello! My number is {0}.", mySerializedNumber);
        Debug.LogFormat("I have been called {0} times.", helloWorldCount);
    }
}

En ajoutant l’attribut CreateAssetMenu à la classe, Unity le listera dans le sous-menu Assets/Create. Dans ce cas, c’est sous Assets/Create/StackOverflow/Examples.

Une fois créées, les instances de ScriptableObject peuvent être transmises à d’autres scripts et ScriptableObjects via l’inspecteur.

using UnityEngine;

public class SampleScript : MonoBehaviour {

    [SerializeField]
    MyScriptableObject myScriptableObject;

    void OnEnable()
    {
        myScriptableObject.HelloWorld();
    }
}

[1] : https://docs.unity3d.com/ScriptReference/CreateAssetMenuAttribute.html

Créer des instances ScriptableObject via le code #

Vous créez de nouvelles instances de ScriptableObject via ScriptableObject.CreateInstance<T>()

T obj = ScriptableObject.CreateInstance<T>();

Où ‘T’ étend ‘ScriptableObject’.

Ne créez pas de ScriptableObjects en appelant leurs constructeurs, c’est-à-dire. nouveau ScriptableObject().

La création de ScriptableObjects par code pendant l’exécution est rarement demandée car leur utilisation principale est la sérialisation des données. Vous pourriez tout aussi bien utiliser des classes standard à ce stade. C’est plus courant lorsque vous créez des scripts d’extensions d’éditeur.

Les ScriptableObjects sont sérialisés dans l’éditeur même en PlayMode #

Des précautions supplémentaires doivent être prises lors de l’accès aux champs sérialisés dans une instance de ScriptableObject.

Si un champ est marqué “public” ou sérialisé via “SerializeField”, la modification de sa valeur est permanente. Ils ne se réinitialisent pas lorsque vous quittez le mode de jeu comme le font MonoBehaviours. Cela peut être utile parfois, mais cela peut aussi créer des dégâts.

Pour cette raison, il est préférable de rendre les champs sérialisés en lecture seule et d’éviter complètement les champs publics.

public class MyScriptableObject : ScriptableObject
{
    [SerializeField]
    int mySerializedValue;

    public int MySerializedValue
    {
        get { return mySerializedValue; }
    }
}

Si vous souhaitez stocker des valeurs publiques dans un ScriptableObject qui sont réinitialisées entre les sessions de lecture, envisagez d’utiliser le modèle suivant.

public class MyScriptableObject : ScriptableObject
{
    // Private fields are not serialized and will reset to default on reset
    private int mySerializedValue;

    public int MySerializedValue
    {
        get { return mySerializedValue; }
        set { mySerializedValue = value; }
    }
}

Trouver les ScriptableObjects existants pendant l’exécution #

Pour trouver des ScriptableObjects actifs pendant l’exécution, vous pouvez utiliser Resources.FindObjectsOfTypeAll().

T[] instances = Resources.FindObjectsOfTypeAll<T>();

Où T est le type de l’instance de ScriptableObject que vous recherchez. Actif signifie qu’il a déjà été chargé en mémoire sous une forme ou sous une autre.

Cette méthode est très lente, alors n’oubliez pas de mettre en cache la valeur de retour et évitez de l’appeler fréquemment. Référencer les ScriptableObjects directement dans vos scripts devrait être votre option préférée.

Astuce : Vous pouvez gérer vos propres collections d’instances pour des recherches plus rapides. Demandez à vos ScriptableObjects de s’enregistrer dans une collection partagée pendant OnEnable().
