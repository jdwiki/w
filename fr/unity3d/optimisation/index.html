Puissance coroutine #

Usage #

Si vous avez une opération de longue durée qui s’appuie sur l’API Unity non thread-safe, utilisez [Coroutines][1] pour la diviser en plusieurs cadres et garder votre application réactive.

Les [coroutines][1] permettent également d’effectuer des actions coûteuses toutes les nièmes images au lieu d’exécuter cette action à chaque image.

Fractionnement de routines de longue durée sur plusieurs cadres ================================================= == Les coroutines aident à répartir les opérations de longue durée sur plusieurs images pour aider à maintenir la fréquence d’images de votre application.

Les routines qui peignent ou génèrent du terrain de manière procédurale ou génèrent du bruit sont des exemples qui peuvent nécessiter le traitement Coroutine.

for (int y = 0; y < heightmap.Height; y++) 
{
    for (int x = 0; x < heightmap.Width; x++)
    {
        // Generate pixel at (x, y)
        // Assign pixel at (x, y)
        
        // Process only 32768 pixels each frame
        if ((y * heightmap.Height + x) % 32 * 1024) == 0)
            yield return null; // Wait for next frame
    }
}

Le code ci-dessus est un exemple facile à comprendre. Dans le code de production, il est préférable d’éviter la vérification par pixel qui vérifie quand “rendre le retour” (peut-être le faire toutes les 2-3 lignes) et de pré-calculer la longueur de la boucle “for” à l’avance.

Effectuer des actions coûteuses moins fréquemment #

Les coroutines vous aident à effectuer des actions coûteuses moins fréquemment, de sorte que les performances ne sont pas aussi importantes que si elles étaient effectuées à chaque image.

En prenant l’exemple suivant directement du [Manuel][1] :

private void ProximityCheck() 
{
    for (int i = 0; i < enemies.Length; i++) 
    {
        if (Vector3.Distance(transform.position, enemies[i].transform.position) < dangerDistance)
                return true;
    }
    return false;
}

private IEnumerator ProximityCheckCoroutine() 
{
    while(true) 
    {
        ProximityCheck();
        yield return new WaitForSeconds(.1f);
    }
}

Les tests de proximité peuvent être encore optimisés en utilisant [CullingGroup API][2].

Pièges courants #

Une erreur courante des développeurs consiste à accéder aux résultats ou aux effets secondaires des coroutines * en dehors * de la coroutine. Les coroutines rendent le contrôle à l’appelant dès qu’une instruction yield return est rencontrée et le résultat ou l’effet secondaire peut ne pas encore être exécuté. Pour contourner les problèmes où vous * devez * utiliser le résultat/effet secondaire en dehors de la coroutine, cochez [cette réponse][3].

[1] : https://docs.unity3d.com/Manual/Coroutines.html [2] : https://docs.unity3d.com/Manual/CullingGroupAPI.html [3] : http://stackoverflow.com/a/38077315/4038191

Chaînes #

On pourrait dire qu’il y a plus de porcs de ressources dans Unity que l’humble chaîne, mais c’est l’un des aspects les plus faciles à corriger dès le début.

Les opérations de chaîne génèrent des ordures #

La plupart des opérations de chaîne génèrent de petites quantités de déchets, mais si ces opérations sont appelées plusieurs fois au cours d’une même mise à jour, elles s’accumulent. Au fil du temps, il déclenchera le Garbage Collection automatique, ce qui peut entraîner un pic CPU visible.

Mettez en cache vos opérations de chaîne #

Prenons l’exemple suivant.

string[] StringKeys = new string[] {
    "Key0",
    "Key1",
    "Key2"
};

void Update()
{
    for (var i = 0; i < 3; i++)
    {
        // Cached, no garbage generated
        Debug.Log(StringKeys[i]);
    }

    for (var i = 0; i < 3; i++)
    {
        // Not cached, garbage every cycle
        Debug.Log("Key" + i);
    }

    // The most memory-efficient way is to not create a cache at all and use literals or constants.
    // However, it is not necessarily the most readable or beautiful way.
    Debug.Log("Key0");
    Debug.Log("Key1");
    Debug.Log("Key2");
}

Cela peut sembler stupide et redondant, mais si vous travaillez avec Shaders, vous pourriez rencontrer des situations comme celles-ci. La mise en cache des clés fera la différence.

Veuillez noter que les chaînes literals et constants ne génèrent pas de déchets, car elles sont injectées statiquement dans l’espace de la pile du programme. Si vous générez des chaînes au moment de l’exécution et que vous êtes garanti de générer les mêmes chaînes à chaque fois comme dans l’exemple ci-dessus, la mise en cache vous aidera certainement.

Pour les autres cas où la chaîne générée n’est pas la même à chaque fois, il n’y a pas d’autre alternative à la génération de ces chaînes. En tant que tel, le pic de mémoire avec la génération manuelle de chaînes à chaque fois est généralement négligeable, à moins que des dizaines de milliers de chaînes ne soient générées à la fois.

La plupart des opérations de chaîne sont des messages de débogage #

Faire des opérations de chaîne pour les messages de débogage, c’est-à-dire. Debug.Log("Object Name: " + obj.name) convient et ne peut être évité pendant le développement. Il est toutefois important de s’assurer que les messages de débogage non pertinents ne se retrouvent pas dans le produit publié.

Une façon consiste à utiliser [Attribut conditionnel] [1] dans vos appels de débogage. Cela supprime non seulement les appels de méthode, mais également toutes les opérations de chaîne qui y sont effectuées.

using UnityEngine;
using System.Collections;

public class ConditionalDebugExample: MonoBehaviour
{
    IEnumerator Start()
    {
        while(true)
        {
            // This message will pop up in Editor but not in builds
            Log("Elapsed: " + Time.timeSinceLevelLoad);
            yield return new WaitForSeconds(1f);
        }
    }

    [System.Diagnostics.Conditional("UNITY_EDITOR")]
    void Log(string Message)
    {
        Debug.Log(Message);
    }
}

Ceci est un exemple simplifié. Vous voudrez peut-être investir du temps dans la conception d’une routine de journalisation plus complète.

Comparaison de chaînes #

Il s’agit d’une optimisation mineure, mais elle mérite d’être mentionnée. La comparaison de chaînes est un peu plus compliquée qu’on ne le pense. Le système essaiera de prendre en compte les différences culturelles par défaut. Vous pouvez choisir d’utiliser une simple comparaison binaire à la place, qui s’exécute plus rapidement.

// Faster string comparison
if (strA.Equals(strB, System.StringComparison.Ordinal)) {...}
// Compared to
if (strA == strB) {...}

// Less overhead
if (!string.IsNullOrEmpty(strA)) {...}
// Compared to
if (strA == "") {...}

// Faster lookups
Dictionary<string, int> myDic = new Dictionary<string, int>(System.StringComparer.Ordinal);
// Compared to
Dictionary<string, int> myDictionary = new Dictionary<string, int>();

[1] : https://www.wikiod.com/fr/docs/c%23/755/pragma-directives/8669/using-the-conditional-attribute#t=201607250855107189175

Vérifications rapides et efficaces #

Évitez les opérations inutiles et les appels de méthode partout où vous le pouvez, en particulier dans une méthode qui est appelée plusieurs fois par seconde, comme Update.

Contrôles de distance/portée #

Utilisez sqrMagnitude au lieu de magnitude lors de la comparaison de distances. Cela évite les opérations sqrt inutiles. Notez que lorsque vous utilisez sqrMagnitude, le côté droit doit également être au carré.

if ((target.position - transform.position).sqrMagnitude < minDistance * minDistance))

Vérifications des limites #

Les intersections d’objets peuvent être grossièrement vérifiées en vérifiant si leurs limites Collider/Renderer se croisent. La structure Bounds a également une méthode pratique [Intersects][1] qui permet de déterminer si deux bornes se croisent.

Bounds nous aide également à calculer une approximation proche de la distance réelle (surface à surface) entre les objets (voir [Bounds.SqrDistance][2]).

Mises en garde #

La vérification des limites fonctionne très bien pour les objets convexes, mais les vérifications des limites sur les objets concaves peuvent entraîner des inexactitudes beaucoup plus élevées en fonction de la forme de l’objet.

L’utilisation de Mesh.bounds n’est pas recommandée car elle renvoie les limites de l’espace local. Utilisez MeshRenderer.bounds à la place.

[1] : https://docs.unity3d.com/ScriptReference/Bounds.Intersects.html [2] : https://docs.unity3d.com/ScriptReference/Bounds.SqrDistance.html

Évitez les méthodes d’unité vides #

Évitez les méthodes d’unité vides. En plus d’être un mauvais style de programmation, il y a une très petite surcharge impliquée dans les scripts d’exécution. Dans de nombreux cas, cela peut s’accumuler et affecter les performances.

void Update
{
}

void FixedUpdate
{
}

Références de cache #

Références de cache pour éviter les appels coûteux, en particulier dans la fonction de mise à jour. Cela peut être fait en mettant en cache ces références au démarrage si elles sont disponibles ou lorsqu’elles sont disponibles et en vérifiant null/bool flat pour éviter d’obtenir à nouveau la référence.

Exemples:

Références des composants de cache

monnaie

void Update()
{
    var renderer = GetComponent<Renderer>();
    renderer.material.SetColor("_Color", Color.green);
}

à

private Renderer myRenderer;
void Start()
{
    myRenderer = GetComponent<Renderer>();
}

void Update()
{
    myRenderer.material.SetColor("_Color", Color.green);
}

Références d’objets de cache

monnaie

void Update()
{
    var enemy = GameObject.Find("enemy");
    enemy.transform.LookAt(new Vector3(0,0,0));
}

à

private Transform enemy;

void Start()
{
    this.enemy = GameObject.Find("enemy").transform;
}

void Update()
{
    enemy.LookAt(new Vector3(0, 0, 0));
}

Mettez également en cache les appels coûteux comme les appels vers Mathf lorsque cela est possible.

Évitez d’appeler des méthodes à l’aide de chaînes #

Évitez d’appeler des méthodes à l’aide de chaînes pouvant accepter des méthodes. Cette approche utilisera la réflexion qui peut ralentir votre jeu, en particulier lorsqu’elle est utilisée dans la fonction de mise à jour.

Exemples:

    //Avoid StartCoroutine with method name
    this.StartCoroutine("SampleCoroutine");

    //Instead use the method directly
    this.StartCoroutine(this.SampleCoroutine());

    //Avoid send message
    var enemy = GameObject.Find("enemy");
    enemy.SendMessage("Die");

    //Instead make direct call
    var enemy = GameObject.Find("enemy") as Enemy;
    enemy.Die();
