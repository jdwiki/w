Comment fonctionne l’API Blueprint ? #

Lorsque les voiles commencent à utiliser le “sails lift”, les voiles regardent pour voir si vous avez tout contrôleur défini. Dans notre exemple, nous avons un contrôleur, l’utilisateur manette. Sails fournit ensuite l’accès aux actions du plan pour cet utilisateur contrôleur comme si nous les avions construits nous-mêmes dans le contrôleur. Voile aussi crée automatiquement des itinéraires de plan au moment de la levée du serveur. Alors même si aucune route n’est définie dans /config/routes.js et aucune action n’est définie dans /api/controllers/UserController.js explicitement, après avoir levé le serveur tout ces routes et actions sont disponibles à l’utilisation.

Routes du plan #

Lorsque vous exécutez sails lift avec des plans activés, le framework inspecte votre contrôleurs, modèles et configuration afin de lier automatiquement certaines routes. Ces routes de blueprint implicites permettent à votre application de répondre à certaines requêtes sans que vous ayez à lier ces routes manuellement dans votre fichier config/routes.js. Par défaut, les routes du blueprint pointent vers leur plan correspondant actions, dont chacune peut être remplacé par un code personnalisé.

Il existe trois types d’itinéraires de plans dans Sails :

  • Routes RESTful, où le chemin est toujours /:model/:id?. Lorsque l’“Utilisateur” le modèle et le contrôleur sont définis, le plan directeur lie implicitement les routes RESTful dans manière suivante -

‘OBTENIR /utilisateur/:id ?’ : { controller: ‘User’, action: ‘find’ }, ‘POSTER /utilisateur’ : { controller: ‘User’, action: ‘create’ }, ‘PUT /user/:id?’ : { controller: ‘User’, action: ‘update’ }, ‘SUPPRIMER /user/:id ?’ : { controller: ‘User’, action: ‘destroy’ }

Ces routes utilisent le verbe HTTP pour déterminer l'action à entreprendre même si le
l'itinéraire est le même. Ainsi, une requête `POST` à `/user` créera un nouvel utilisateur, un
La requête `PUT` à `/user/123` mettra à jour l'utilisateur avec la clé primaire 123 et
une requête `DELETE` à `/user/123` supprimera l'utilisateur dont la clé primaire est 123.
Dans un environnement de production, les routes RESTful doivent généralement être protégées par
politiques pour éviter tout accès non autorisé.


+ **Routes raccourcies**, où l'action à effectuer est encodée dans le chemin. Pour notre
Le modèle utilisateur et le contrôleur Sails se lient implicitement en suivant quatre raccourcis.

‘OBTENIR /user/find/:id?’ : { controller: ‘User’, action: ‘find’ }, ‘OBTENIR /user/create/:id ?’ : { controller: ‘User’, action: ‘create’ }, ‘OBTENIR /utilisateur/mise à jour/:id ?’ : { controller: ‘User’, action: ‘update’ }, ‘OBTENIR /user/destroy/:id ?’ : { controller: ‘User’, action: ‘destroy’ }

Par exemple, le raccourci `/user/create?name=joe` crée un nouvel utilisateur, tandis que
`/user/update/1?name=mike` met à jour le champ du nom de l'utilisateur #1. Notez que ces
les routes ne répondent qu'aux requêtes "GET". Les raccourcis sont très pratiques pour
développement, mais doit généralement être désactivé dans un environnement de production. C'est
pas conçu pour être utilisé en production.

+ **Routes d'action**, qui créent automatiquement des routes pour votre contrôleur personnalisé
Actions. Par exemple, laissez `query` être une action personnalisée définie dans le contrôleur utilisateur.
Ensuite, les itinéraires suivants seraient implicitement disponibles pour les voiles -

‘OBTENIR /utilisateur/requête/:id ?’ : { controller: ‘User’, action: ‘query’ }, ‘POST /user/query/:id?’ : { controller: ‘User’, action: ‘query’ }, ‘PUT /user/query/:id?’ : { controller: ‘User’, action: ‘query’ }, ‘SUPPRIMER /utilisateur/requête/:id ?’ : { controller: ‘User’, action: ‘query’ }

Si la requête est faite dans la route `/user/query/:id?` alors indépendante du HTTP
verbe l'action serait la même. Contrairement aux itinéraires RESTful et raccourcis, action
les routes ne nécessitent *pas* qu'un contrôleur ait un fichier de modèle correspondant. Qui
signifie, si vous définissez un contrôleur dans le fichier `/api/controllers/FooController.js`
mais pas de modèle dans le fichier `/api/models/Foo.js`, il n'y aurait pas de RESTful ou
route raccourcie avec `/foo` mais il y aura toujours des routes d'action disponibles pour
utilisation.

# Ordre des itinéraires correspondant
Lorsqu'une demande arrive,
les voiles correspondront d'abord à la route par rapport aux routes explicitement définies. Si ça correspond
alors aucune autre correspondance n'est effectuée et l'action correspondante est exécutée. Mais s'il
ne correspond pas, l'itinéraire est d'abord mis en correspondance avec les itinéraires d'action du plan directeur,
si ne correspond pas alors aux itinéraires de repos et s'il ne correspond pas non plus alors
itinéraires raccourcis. Donc, si votre fichier `/config/routes.js` a une entrée comme le
Suivant-

‘/user/query/:id?’: { controller: ‘User’, action: ‘find’ }

Ensuite, vous ne pouvez pas vous attendre à ce que les routes d'action `query` fonctionnent. Parce que le même itinéraire que
La route d'action `query` serait comparée aux routes explicitement définies et
L'action "trouver" du contrôleur utilisateur serait exécutée.

## Actions du plan directeur
Les actions de plan directeur (à ne pas confondre avec l'action de plan directeur *routes*) sont
actions génériques conçues pour fonctionner avec n'importe lequel de vos contrôleurs qui ont un modèle
du même nom (par exemple, `ParrotController` aurait besoin d'un modèle `Parrot`). Penser à
comme comportement par défaut pour votre application. Par exemple, si vous avez un
Modèle `User.js` et un contrôleur `UserController.js` vide, `find`, `create`,
les actions `update`, `destroy`, `populate`, `add` et `remove` existent implicitement,
sans que vous ayez à les écrire.

Par défaut, les routes RESTful et les raccourcis du blueprint sont liés à leur
actions de plan correspondantes. Cependant, toute action de blueprint peut être remplacée
pour un contrôleur particulier en créant une action personnalisée dans ce fichier de contrôleur
(par exemple `ParrotController.find`). Alternativement, vous pouvez remplacer le plan
action _partout dans votre application_ en créant votre propre action blueprint personnalisée.

Sails est livré avec les actions de plan suivantes :

+ trouver
+ trouverUn
+ créer
+ mise à jour
+ détruire
+ peupler
+ ajouter
+ supprimer

## Désactivation des routes Blueprint
+ **Base mondiale** :
La configuration de l'API Blueprint est définie dans le fichier `/config/blueprint.js`. Tu peux
activer ou désactiver les trois types de routes de plan directeur pour tous les contrôleurs à partir de
là. Par exemple, si vous souhaitez désactiver les raccourcis de blueprint pour tous
vos contrôleurs mais que vous souhaitez conserver les itinéraires d'action et de repos activés, alors
votre `/config/blueprint.js` devrait ressembler à ceci -

module.exports.blueprints = { action: true, rest: true, shortcut: false }


+ **Par contrôleur** :
Vous pouvez également remplacer n'importe lequel des paramètres de `/config/blueprints.js` sur un
par contrôleur en définissant une clé '_config' dans la définition de votre contrôleur,
et en lui attribuant un objet de configuration avec des remplacements pour les paramètres dans ce
dossier. Par exemple, si vous souhaitez que les raccourcis soient activés uniquement pour votre utilisateur
contrôleur mais pas pour d'autres contrôleurs alors avec le plan ci-dessus
configuration, vous devez avoir la paire clé-valeur suivante dans le contrôleur utilisateur.

module.exports = { _config: { actions: true, shortcuts: true, rest: true } }
