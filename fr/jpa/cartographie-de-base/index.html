Paramètres #

Annotation Détails
@Id Marque le champ/la colonne comme clé de l’entité
@Basique Marque le champ demandé à mapper en tant que type basic. Cela s’applique aux types primitifs et à leurs wrappers, String, Date et Calendar. L’annotation est en fait facultative si aucun paramètre n’est donné, mais un bon style dicterait de rendre vos intentions explicites.
@Transient Les champs marqués comme transitoires ne sont pas pris en compte pour la persistance, tout comme le mot-clé “transient” pour la sérialisation.

Il doit toujours y avoir un constructeur par défaut, c’est-à-dire celui sans paramètre. Dans l’exemple de base, aucun constructeur n’était spécifié, donc Java en a ajouté un ; mais si vous ajoutez un constructeur avec des arguments, assurez-vous d’ajouter également le constructeur sans paramètre.

Une entité très simple #

@Entity
class Note {
    @Id
    Integer id;
 
    @Basic
    String note;

    @Basic
    int count;
}

Les getters, les setters, etc. sont omis par souci de brièveté, mais ils ne sont de toute façon pas nécessaires pour JPA.

Cette classe Java serait mappée au tableau suivant (selon votre base de données, ici donnée dans un mappage Postgres possible):

CREATE TABLE Note (
  id integer NOT NULL,
  note text,
  count integer NOT NULL
)

Les fournisseurs JPA peuvent être utilisés pour générer le DDL et produiront probablement un DDL différent de celui présenté ici, mais tant que les types sont compatibles, cela ne causera pas de problèmes lors de l’exécution. Il est préférable de ne pas compter sur la génération automatique de DDL.

Champ omis du mappage #

@Entity
class Note {
    @Id
    Integer id;
 
    @Basic
    String note;

    @Transient
    String parsedNote;

    String readParsedNote() {
        if (parsedNote == null) { /* initialize from note */ }
        return parsedNote;
    }
}

Si votre classe a besoin de champs qui ne doivent pas être écrits dans la base de données, marquez-les comme @Transient. Après lecture de la base de données, le champ sera “null”.

Mappage de l’heure et de la date #

L’heure et la date sont de plusieurs types différents en Java : les historiques Date et Calendar, et les plus récents LocalDate et LocalDateTime. Et Timestamp, Instant, ZonedLocalDateTime et les types Joda-time. Du côté de la base de données, nous avons time, date et timestamp (à la fois l’heure et la date), éventuellement avec ou sans fuseau horaire.

Date et heure avant Java 8 #

Le mappage default pour les types pré-Java-8 java.util.Date, java.util.Calendar et java.sql.Timestamp est timestamp en SQL ; pour java.sql.Date c’est date.

@Entity
class Times {
    @Id
    private Integer id;

    @Basic
    private Timestamp timestamp;

    @Basic
    private java.sql.Date sqldate;

    @Basic
    private java.util.Date utildate;

    @Basic
    private Calendar calendar;
}

Cela correspondra parfaitement au tableau suivant :

CREATE TABLE times (
    id integer not null,
    timestamp timestamp,
    sqldate date,
    utildate timestamp,
    calendar timestamp
)

Ce n’est peut-être pas l’intention. Par exemple, une Date ou un Calendar Java est souvent utilisé pour représenter la date uniquement (pour la date de naissance). Pour modifier le mappage par défaut, ou simplement pour rendre le mappage explicite, vous pouvez utiliser l’annotation @Temporal.

@Entity
class Times {
    @Id
    private Integer id;

    @Temporal(TemporalType.TIME)
    private Date date;

    @Temporal(TemporalType.DATE)
    private Calendar calendar;
}

La table SQL équivalente est :

CREATE TABLE times (
    id integer not null,
    date time,
    calendar date
)

Remarque 1 : Le type spécifié avec @Temporal influence la génération DDL ; mais vous pouvez également avoir une colonne de type date mappée à Date avec juste l’annotation @Basic.

Remarque 2 : Calendar ne peut pas conserver uniquement time.

Date et heure avec Java 8 #

JPA 2.1 ne définit pas la prise en charge des types java.time fournis dans Java 8. Cependant, la majorité des implémentations de JPA 2.1 offrent la prise en charge de ces types, bien qu’il s’agisse à proprement parler d’extensions de fournisseurs.

Pour DataNucleus, ces types fonctionnent immédiatement et offrent un large éventail de possibilités de mappage, couplées à l’annotation @Temporal.

Pour Hibernate, si vous utilisez Hibernate 5.2+, ils devraient fonctionner immédiatement, en utilisant simplement l’annotation @Basic. Si vous utilisez Hibernate 5.0-5.1, vous devez ajouter la dépendance org.hibernate:hibernate-java8. Les mappages fournis sont

  • LocalDate à date
  • Instant, LocalDateTime et ZonedDateTime à timestamp

Une alternative indépendante du fournisseur serait également de définir un AttributeConverter JPA 2.1 pour tout type Java 8 java.time qui doit être persistant.

Entité avec ID géré par séquence #

Ici, nous avons une classe et nous voulons que le champ d’identité (userUid) ait sa valeur générée via une SEQUENCE dans la base de données. Cette SEQUENCE est supposée s’appeler USER_UID_SEQ et peut être créée par un DBA ou par le fournisseur JPA.

@Entity
@Table(name="USER")
public class User implements Serializable {
    private static final long serialVersionUID = 1L;

    @Id
    @SequenceGenerator(name="USER_UID_GENERATOR", sequenceName="USER_UID_SEQ")
    @GeneratedValue(strategy=GenerationType.SEQUENCE, generator="USER_UID_GENERATOR")
    private Long userUid;

    @Basic
    private String userName;
}
