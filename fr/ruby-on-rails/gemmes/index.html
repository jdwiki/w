Documentation du fichier Gem #

Pour les projets qui devraient grandir, c’est une bonne idée d’ajouter des commentaires à votre Gemfile. De cette façon, même dans les grandes configurations, vous saurez toujours ce que fait chaque gemme même si le nom n’est pas explicite et que vous l’avez ajouté il y a 2 ans.

Cela peut également vous aider à vous rappeler pourquoi vous avez choisi une certaine version et, par conséquent, à réévaluer la version requise ultérieurement.

Exemples:

# temporary downgrade for TeamCity
gem 'rake', '~> 10.5.0'
# To upload invoicing information to payment provider
gem 'net-sftp'

Qu’est-ce qu’une gemme ? #

Un gem est l’équivalent d’un plugin ou d’une extension pour le langage de programmation ruby.

Pour être exact, même les rails ne sont rien de plus qu’un bijou. Beaucoup de gemmes sont construites sur des rails ou d’autres gemmes (elles dépendent de ladite gemme) ou sont autonomes.

Dans votre projet Rails #

Gemfichier #

Pour votre projet Rails, vous avez un fichier appelé Gemfile. Ici, vous pouvez ajouter des gemmes que vous souhaitez inclure et utiliser dans votre projet. Une fois ajouté, vous devez installer la gemme en utilisant bundler (voir la section Bundler).

Gemfile.lock #

Une fois que vous avez fait cela, votre Gemfile.lock sera mis à jour avec vos gemmes nouvellement ajoutées et leurs dépendances. Ce fichier verrouille vos gemmes utilisées afin qu’elles utilisent cette version spécifique déclarée dans ce fichier.

GEMME
distant : https://rubygems.org/
spécifications :
concevoir (4.0.3)
bcrypt (~> 3.0)
orm_adapter (~> 0.1)
railties (>= 4.1.0, < 5.1)
intervenants
gardien (~> 1.2.3)

Cet exemple est pour la gemme devise. Dans le Gemfile.lock la version 4.0.3 est déclarée, pour indiquer lors de l’installation de votre projet sur une autre machine ou sur votre serveur de production quelle version spécifiée utiliser.

Développement #

Soit une seule personne, un groupe ou toute une communauté travaille sur et entretient un joyau. Le travail effectué est généralement publié après que certains “problèmes” ont été résolus ou que des “fonctionnalités” ont été ajoutées.

Habituellement, les versions suivent le principe Semantic Versioning 2.0.0.

Bundler #

Le moyen le plus simple de gérer et de gérer les gemmes consiste à utiliser bundler. Bundler est un gestionnaire de paquets comparable à bower.

Pour utiliser bundler, vous devez d’abord l’installer.

gem install bundler

Une fois que le bundler est opérationnel, tout ce que vous avez à faire est d’ajouter des gemmes à votre Gemfile et d’exécuter

bundle

dans votre borne. Cela installe vos gemmes nouvellement ajoutées à votre projet. En cas de problème, vous obtiendrez une invite dans votre terminal.

Si vous souhaitez plus de détails, je vous suggère de consulter la docs.

## Gemfiles Pour commencer, les gemfiles nécessitent au moins une source, sous la forme de l’URL d’un serveur RubyGems.

Générez un Gemfile avec la source rubygems.org par défaut en exécutant bundle init. Utilisez https pour que votre connexion au serveur soit vérifiée avec SSL.

source 'https://rubygems.org'

Ensuite, déclarez les gemmes dont vous avez besoin, y compris les numéros de version.

gem 'rails', '4.2.6'
gem 'rack',  '>=1.1'
gem 'puma',  '~>3.0'

La plupart des spécificateurs de version, comme >= 1.0, sont explicites. Le spécificateur ~> a une signification particulière. ~> 2.0.3 est identique à >= 2.0.3 et < 2.1. ~> 2.1 est identique à >= 2.1 et < 3.0. ~> 2.2.beta correspondra aux versions préliminaires telles que 2.2.beta.12.

Les référentiels Git sont également des sources de gemmes valides, tant que le référentiel contient une ou plusieurs gemmes valides. Spécifiez ce que vous souhaitez extraire avec :tag, :branch ou :ref. La branche par défaut est master.

gem 'nokogiri', :git => 'https://github.com/sparklemotion/nokogiri', :branch => 'master'

Si vous souhaitez utiliser une gemme décompressée directement à partir du système de fichiers, définissez simplement l’option :path sur le chemin contenant les fichiers de la gemme.

gem 'extracted_library', :path => './vendor/extracted_library'

Les dépendances peuvent être placées dans des groupes. Les groupes peuvent être ignorés au moment de l’installation (en utilisant --without) ou requis tous à la fois (en utilisant Bundler.require).

gem 'rails_12factor', group: :production

group :development, :test do
  gem 'byebug'
  gem 'web-console', '~> 2.0'
  gem 'spring'
  gem 'dotenv-rails'
end

Vous pouvez spécifier la version requise de Ruby dans le Gemfile avec ruby. Si le Gemfile est chargé sur une autre version de Ruby, Bundler lèvera une exception avec une explication.

ruby '2.3.1'

Ensembles de gemmes #

Si vous utilisez RVM (Ruby Version Manager), utiliser un gemset pour chaque projet est une bonne idée. Un gemset est juste un conteneur que vous pouvez utiliser pour séparer les gemmes les unes des autres. La création d’un gemset par projet vous permet de changer de gemmes (et de versions de gemmes) pour un projet sans casser tous vos autres projets. Chaque projet n’a qu’à se soucier de ses propres joyaux.

RVM fournit (>= 0.1.8) un @global gemset par interpréteur ruby. Les gemmes que vous installez sur le @global gemset pour un rubis donné sont disponibles pour tous les autres gemsets que vous créez en association avec ce rubis. C’est un bon moyen de permettre à tous vos projets de partager la même gemme installée pour une installation d’interpréteur ruby ​​spécifique.

Créer des gemsets

Supposons que vous ayez déjà installé ruby-2.3.1 et que vous l’ayez sélectionné à l’aide de cette commande :

rvm use ruby-2.3.1

Maintenant, pour créer un gemset pour cette version ruby :

rvm gemset create new_gemset

où new_gemset est le nom du gemset. Pour voir la liste des gemsets disponibles pour une version ruby ​​:

rvm gemset list

pour lister les gemmes de toutes les versions ruby ​​:

rvm gemset list_all

pour utiliser un gemset de la liste (supposons que new_gemset est le gemset que je veux utiliser):

rvm gemset use new_gemset

vous pouvez également spécifier la version ruby ​​avec le gemset si vous souhaitez passer à une autre version ruby :

rvm use ruby-2.1.1@new_gemset

pour spécifier un gemset par défaut pour une version Ruby particulière :

rvm use 2.1.1@new_gemset --default

pour supprimer toutes les gemmes installées d’un gemset, vous pouvez le vider en :

rvm gemset empty new_gemset

pour copier un gemset d’un rubis à un autre vous pouvez le faire en :

rvm gemset copy 2.1.1@rails4 2.1.2@rails4

pour supprimer un gemset :

rvm gemset delete new_gemset

pour voir le nom du gemset actuel :

rvm gemset name

pour installer un gem dans le gemset global :

rvm @global do gem install ...

Initialisation des Gemsets lors de l’installation de Ruby

Lorsque vous installez un nouveau ruby, RVM crée non seulement deux gemsets (le gemset vide par défaut et le gemset global), mais il utilise également un ensemble de fichiers modifiables par l’utilisateur pour déterminer les gemmes à installer.

Travaillant dans ~/.rvm/gemsets, rvm recherche global.gems et default.gems en utilisant une arborescence basée sur la chaîne ruby ​​en cours d’installation. En utilisant l’exemple de ree-1.8.7-p2010.02, rvm vérifiera (et importera depuis) ​​les fichiers suivants :

~/.rvm/gemsets/ree/1.8.7/p2010.02/global.gems
~/.rvm/gemsets/ree/1.8.7/p2010.02/default.gems
~/.rvm/gemsets/ree/1.8.7/global.gems
~/.rvm/gemsets/ree/1.8.7/default.gems
~/.rvm/gemsets/ree/global.gems
~/.rvm/gemsets/ree/default.gems
~/.rvm/gemsets/global.gems
~/.rvm/gemsets/default.gems

Par exemple, si vous avez modifié ~/.rvm/gemsets/global.gems en ajoutant ces deux lignes :

bundler
awesome_print

chaque fois que vous installez un nouveau rubis, ces deux gemmes sont installées dans votre gemset global. Les fichiers default.gems et global.gems sont généralement écrasés lors de la mise à jour de rvm.
