Création d’un modèle via un générateur #

Ruby on Rails fournit un générateur de “modèles” que vous pouvez utiliser pour créer des modèles ActiveRecord. Utilisez simplement rails generate model et fournissez le nom du modèle.

$ rails g model user

En plus du fichier de modèle dans app/models, le générateur créera également :

  • le Test dans test/models/user_test.rb
  • les Fixtures dans test/fixtures/users.yml
  • la migration de la base de données dans db/migrate/XXX_create_users.rb

Vous pouvez également générer des champs pour le modèle lors de sa génération.

$ rails g model user email:string sign_in_count:integer birthday:date

Cela créera les colonnes email, sign_in_count et birthday dans votre base de données, avec les types appropriés.

Création manuelle d’un modèle #

Bien que l’utilisation de l’échafaudage soit rapide et facile si vous êtes nouveau sur Rails ou si vous créez une nouvelle application, il peut être utile plus tard de le faire vous-même pour éviter d’avoir à parcourir le code généré par l’échafaudage pour le réduire. (retirer les pièces inutilisées, etc.).

La création d’un modèle peut être aussi simple que la création d’un fichier sous app/models.

Le modèle le plus simple, dans ActiveRecord, est une classe qui étend ActiveRecord::Base.

class User < ActiveRecord::Base
end

Les fichiers de modèle sont stockés dans app/models/, et le nom du fichier correspond au nom singulier de la classe :

# user
app/models/user.rb

# SomeModel
app/models/some_model.rb

La classe héritera de toutes les fonctionnalités d’ActiveRecord : méthodes de requête, validations, rappels, etc.

# Searches the User with ID 1
User.find(1)

Remarque : Assurez-vous que la table du modèle correspondant existe. Sinon, vous pouvez créer la table en créant un [Migration][1]

Vous pouvez générer un modèle et sa migration par terminal à partir de la commande suivante

rails g model column_name1:data_type1, column_name2:data_type2, ...

et peut également attribuer une clé étrangère (relation) au modèle en suivant la commande

rails g model column_name:data_type, model_name:references

[1] : https://www.wikiod.com/fr/ruby-on-rails/activerecord#Création d’une migration

Introduction aux rappels #

Un callback est une méthode qui est appelée à des moments spécifiques du cycle de vie d’un objet (juste avant ou après la création, la suppression, la mise à jour, la validation, l’enregistrement ou le chargement depuis la base de données).

Par exemple, supposons que vous ayez une liste qui expire dans les 30 jours suivant sa création.

Une façon de faire est comme ceci:

class Listing < ApplicationRecord
  after_create :set_expiry_date

  private

  def set_expiry_date
    expiry_date = Date.today + 30.days
    self.update_column(:expires_on, expiry_date)
  end
end

Toutes les méthodes disponibles pour les rappels sont les suivantes, dans le même ordre qu’elles sont appelées lors de l’opération de chaque objet :

Création d’un objet

  • avant_validation
  • après_validation
  • avant_sauvegarder
  • autour_sauver
  • avant_créer
  • autour_créer
  • après_créer
  • après_sauvegarde
  • after_commit/after_rollback

Mise à jour d’un objet

  • avant_validation
  • après_validation
  • avant_sauvegarder
  • autour_sauver
  • avant la mise à jour
  • autour de_mise à jour
  • après_mise à jour
  • après_sauvegarde
  • after_commit/after_rollback

Détruire un objet

  • before_destroy
  • autour_destroy
  • after_destroy
  • after_commit/after_rollback

REMARQUE : after_save s’exécute à la fois lors de la création et de la mise à jour, mais toujours après les rappels plus spécifiques after_create et after_update, quel que soit l’ordre dans lequel les appels de macro ont été exécutés.

Tester manuellement vos modèles #

Tester vos modèles Active Record via votre interface de ligne de commande est simple. Accédez au répertoire de l’application dans votre terminal et tapez “rails console” pour démarrer la console Rails. À partir de là, vous pouvez exécuter des méthodes d’enregistrement actives sur votre base de données.

Par exemple, si vous aviez un schéma de base de données avec une table Users ayant une colonne name:string et email:string, vous pourriez exécuter :

User.create name: "John", email: "john@example.com"

Ensuite, pour afficher cet enregistrement, vous pouvez exécuter :

User.find_by email: "john@example.com"

Ou s’il s’agit de votre premier ou unique enregistrement, vous pouvez simplement obtenir le premier enregistrement en exécutant :

User.first

Création d’une migration #

Ajouter/supprimer des champs dans les tables existantes #

Créez une migration en exécutant :

rails generate migration AddTitleToCategories title:string

Cela créera une migration qui ajoutera une colonne title à une table categories :

class AddTitleToCategories < ActiveRecord::Migration[5.0]
  def change
    add_column :categories, :title, :string
  end
end

De même, vous pouvez générer une migration pour supprimer une colonne : rails generate migration RemoveTitleFromCategories title:string

Cela créera une migration qui supprimera une colonne title de la table categories :

class RemoveTitleFromCategories < ActiveRecord::Migration[5.0]
  def change
    remove_column :categories, :title, :string
  end
end

Alors qu’à proprement parler, spécifier type (:string dans ce cas) n’est pas nécessaire pour supprimer une colonne, c’est utile, car cela fournit les informations nécessaires pour la rouler retour.

Créer un tableau #

Créez une migration en exécutant :

rails g CreateUsers name bio

Rails reconnaît l’intention de créer une table à partir du préfixe “Create”, le reste du nom de la migration sera utilisé comme nom de table. L’exemple donné génère ce qui suit :

class CreateUsers < ActiveRecord::Migration
  def change
    create_table :users do |t|
      t.string :name
      t.string :bio
    end
  end
end

Notez que la commande de création n’a pas spécifié de types de colonnes et que la “chaîne” par défaut a été utilisée.

Créer une table de jointure #

Créez une migration en exécutant :

rails g CreateJoinTableParticipation user:references group:references

Rails détecte l’intention de créer une table de jointure en trouvant JoinTable dans le nom de la migration. Tout le reste est déterminé à partir des noms des champs que vous donnez après le nom.

class CreateJoinTableParticipation < ActiveRecord::Migration
  def change
    create_join_table :users, :groups do |t|
      # t.index [:user_id, :group_id]
      # t.index [:group_id, :user_id]
    end
  end
end

Décommentez les instructions index nécessaires et supprimez le reste.

Priorité #

Notez que l’exemple de nom de migration “CreateJoinTableParticipation” correspond à la règle de création de table : il a un préfixe “Create”. Mais il n’a pas généré de simple create_table. En effet, le générateur de migration ([code source][1]) utilise une première correspondance de la liste suivante :

  • (Ajouter|Supprimer)<ignoré>(Vers|De)<nom_table>
  • <ignoré>JoinTable<ignoré>
  • Créer<nom_table>

[1] : https://github.com/rails/rails/blob/ff044c3c3a0a85d26d7c000abeeed835a3bee0bf/activerecord/lib/rails/generators/active_record/migration/migration_generator.rb#L19

Créer une table de jointure à l’aide de migrations #

Particulièrement utile pour la relation has_and_belongs_to_many, vous pouvez créer manuellement une table de jointure en utilisant la méthode create_table. Supposons que vous ayez deux modèles Tags et Proyects et que vous souhaitiez les associer à l’aide d’une relation has_and_belongs_to_many. Vous avez besoin d’une table de jointure pour associer des instances des deux classes.

class CreateProjectsTagsJoinTableMigration < ActiveRecord::Migration
  def change
    create_table :projects_tags, id: false do |t|
      t.integer :project_id
      t.integer :tag_id
    end
  end
end

Le nom réel de la table doit suivre cette convention : le modèle qui précède alphabétiquement l’autre doit passer en premier. Project précède Tags donc le nom de la table est projects_tags.

De plus, puisque le but de cette table est d’acheminer l’association entre les instances de deux modèles, l’identifiant réel de chaque enregistrement de cette table n’est pas nécessaire. Vous spécifiez cela en passant id: false

Enfin, comme c’est la convention dans Rails, le nom de la table doit être la forme plurielle composée des modèles individuels, mais la colonne de la table doit être au singulier.
