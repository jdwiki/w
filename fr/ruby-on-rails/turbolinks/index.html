Turbolinks est une bibliothèque javascript qui accélère la navigation dans votre application Web. Lorsque vous suivez un lien, Turbolinks récupère automatiquement la page, insère son <body> et fusionne son <head>, le tout sans encourir le coût d’un chargement complet de la page.

En tant que développeur de rails, vous interagirez probablement de manière minimale avec les turbolinks au cours de votre développement. Il s’agit cependant d’une bibliothèque importante à connaître car elle peut être à l’origine de certains bogues difficiles à trouver.

Points clés à retenir: #

  • Liaison à l’événement turbolinks:load au lieu de l’événement document.ready
  • Utilisez l’attribut data-turbolinks-false pour désactiver la fonctionnalité turbolink lien par lien.
  • Utilisez l’attribut data-turbolinks-permanent pour conserver les éléments à travers les chargements de page et pour éviter les bogues liés au cache.

Pour un traitement plus approfondi des turbolinks, visitez le [référentiel github officiel][1].

Le mérite d’une grande partie de cette documentation revient aux personnes qui ont rédigé la documentation turbolinks sur le référentiel github.

[1] : https://github.com/turbolinks/turbolinks#persisting-elements-across-page-loads

Liaison au concept de chargement de page de turbolink #

Avec les turbolinks, l’approche traditionnelle d’utilisation :

$(document).ready(function() {
  // awesome code
});

ne fonctionnera pas. Lors de l’utilisation de turbolinks, l’événement $(document).ready() ne se déclenchera qu’une seule fois : lors du chargement initial de la page. À partir de ce moment, chaque fois qu’un utilisateur clique sur un lien sur votre site Web, turbolinks interceptera l’événement de clic sur le lien et fera une requête ajax pour remplacer la balise <body> et fusionner les balises <head>. L’ensemble du processus déclenche la notion d’une “visite” dans le pays des turbolinks. Par conséquent, au lieu d’utiliser la syntaxe traditionnelle document.ready() ci-dessus, vous devrez vous lier à l’événement visit de turbolink comme suit :

// pure js
document.addEventListener("turbolinks:load", function() {
  // awesome code
});

// jQuery
$(document).on('turbolinks:load', function() {
  // your code
});

Désactiver les turbolinks sur des liens spécifiques #

Il est très facile de désactiver les turbolinks sur des liens spécifiques. Selon [la documentation officielle de turbolinks][1] :

Les turbolinks peuvent être désactivés lien par lien en annotant un lien ou l’un de ses ancêtres avec data-turbolinks=“false”.

Exemples: #

// disables turbolinks for this one link
<a href="/" data-turbolinks="false">Disabled</a>

// disables turbolinks for all links nested within the div tag
<div data-turbolinks="false">
  <a href="/">I'm disabled</a>
  <a href="/">I'm also disabled</a>
</div>

// re-enable specific link when ancestor has disabled turbolinks
<div data-turbolinks="false">
  <a href="/">I'm disabled</a>
  <a href="/" data-turbolinks="true">I'm re-enabled</a>
</div>

[1] : http://github.com/turbolinks/turbolinks

Comprendre les visites d’applications #

Les visites d’application sont initiées en cliquant sur un lien compatible Turbolinks, ou par programmation en appelant

Turbolinks.visit(location)

Par défaut, la fonction de visite utilise l’action ‘avancer’. Plus compréhensiblement, le comportement par défaut de la fonction de visite est d’avancer vers la page indiquée par le paramètre “location”. Chaque fois qu’une page est visitée, turbolinks pousse une nouvelle entrée dans l’historique du navigateur en utilisant history.pushState. L’historique est important car turbolinks essaiera d’utiliser l’historique pour charger les pages du cache chaque fois que possible. Cela permet un rendu de page extrêmement rapide pour les pages fréquemment visitées.

Cependant, si vous souhaitez visiter un lieu sans mettre d’historique sur la pile, vous pouvez utiliser l’action « remplacer » sur la fonction de visite comme suit :

// using links
<a href="/edit" data-turbolinks-action="replace">Edit</a>

// programatically
Turbolinks.visit("/edit", { action: "replace" })

Cela remplacera le haut de la pile d’historique par la nouvelle page afin que le nombre total d’éléments sur la pile reste inchangé.

Il existe également une action de “restauration” qui facilite les [visites de restauration][1], les visites qui se produisent lorsque l’utilisateur clique sur le bouton Suivant ou Précédent de son navigateur. Turbolinks gère ces types d’événements en interne et recommande aux utilisateurs de ne pas modifier manuellement le comportement par défaut.

[1] : https://github.com/turbolinks/turbolinks#restoration-visits

Annuler des visites avant qu’elles ne commencent #

Turbolinks fournit un écouteur d’événement qui peut être utilisé pour empêcher les visites de se produire. Écoutez l’événement turbolinks:before-visit pour être averti lorsqu’une visite est sur le point de commencer.

Dans le gestionnaire d’événements, vous pouvez utiliser :

// pure javascript
event.data.url

ou

// jQuery
$event.originalEvent.data.url

pour récupérer le lieu de la visite. La visite peut alors être annulée en appelant :

event.preventDefault()

REMARQUE: #

Selon les [documents officiels de turbolinks] [1] :

Les visites de restauration ne peuvent pas être annulées et ne déclenchent pas de turbolinks : avant-visite.

[1] : https://github.com/turbolinks/turbolinks#canceling-visits-before-they-start

Éléments persistants à travers les chargements de page #

Considérez la situation suivante : imaginez que vous êtes le développeur d’un site Web de médias sociaux qui permet aux utilisateurs d’être amis avec d’autres utilisateurs et qui utilise des liens turbo pour accélérer le chargement des pages. En haut à droite de chaque page du site, il y a un nombre indiquant le nombre total d’amis qu’un utilisateur a actuellement. Imaginez que vous utilisez votre site et que vous avez 3 amis. Chaque fois qu’un nouvel ami est ajouté, vous avez du javascript qui s’exécute et met à jour le compteur d’amis. Imaginez que vous venez d’ajouter un nouvel ami et que votre javascript s’exécute correctement et met à jour le nombre d’amis en haut à droite de la page pour afficher maintenant 4. Maintenant, imaginez que vous cliquez sur le bouton de retour du navigateur. Lorsque la page se charge, vous remarquez que le compteur d’amis indique 3 même si vous avez quatre amis.

Il s’agit d’un problème relativement courant et pour lequel turbolinks a fourni une solution. La raison pour laquelle le problème se produit est que turbolinks charge automatiquement les pages du cache lorsqu’un utilisateur clique sur le bouton de retour. La page mise en cache ne sera pas toujours mise à jour avec la base de données.

Pour résoudre ce problème, imaginez que vous affichez le nombre d’amis dans une balise <div> avec un identifiant “friend-count” :

<div id="friend-count" data-turbolinks-permanent>3 friends</div>

En ajoutant l’attribut data-turbolinks-permanent, vous dites aux turbolinks de conserver certains éléments à travers les chargements de page. Les [documents officiels disent][1] :

Désignez les éléments permanents en leur donnant un identifiant HTML et en les annotant avec data-turbolinks-permanent. Avant chaque rendu, Turbolinks fait correspondre tous les éléments permanents par identifiant et les transfère de la page d’origine vers la nouvelle page, en préservant leurs données et leurs écouteurs d’événements.

[1] : https://github.com/turbolinks/turbolinks#persisting-elements-across-page-loads
