Mise à niveau de Rails 4.2 vers Rails 5.0 #

Remarque : Avant de mettre à jour votre application Rails, assurez-vous toujours d’enregistrer votre code sur un système de contrôle de version, tel que Git.

Pour passer de Rails 4.2 à Rails 5.0, vous devez utiliser Ruby 2.2.2 ou une version plus récente. Après avoir mis à jour votre version de Ruby si nécessaire, accédez à votre Gemfile et modifiez la ligne :

gem 'rails', '4.2.X'

à:

gem 'rails', '~> 5.0.0'

et sur la ligne de commande lancez :

$ bundle update

Exécutez maintenant la tâche de mise à jour à l’aide de la commande :

$ rake rails:update

Cela vous aidera à mettre à jour les fichiers de configuration. Vous serez invité à écraser les fichiers et vous aurez plusieurs options à saisir :

  • Y - oui, écraser
  • n - non, ne pas écraser
  • a - tous, écraser ceci et tous les autres
  • q - quitter, abandonner
  • d - diff, montre les différences entre l’ancien et le nouveau
  • h - aide

En règle générale, vous devez vérifier les différences entre les anciens et les nouveaux fichiers pour vous assurer que vous n’obtenez pas de modifications indésirables.

Les modèles Rails 5.0 ActiveRecord héritent de ApplicationRecord, plutôt que de ActiveRecord::Base. ApplicationRecord est la superclasse pour tous les modèles, de la même manière que ApplicationController est la superclasse pour les contrôleurs. Pour tenir compte de cette nouvelle façon dont les modèles sont gérés, vous devez créer un fichier dans votre dossier app/models/ appelé application_record.rb, puis modifier le contenu de ce fichier pour qu’il soit :

class ApplicationRecord < ActiveRecord::Base
  self.abstract_class = true
end

Rails 5.0 gère également les rappels légèrement différents. Les rappels qui renvoient “false” n’arrêteront pas la chaîne de rappel, ce qui signifie que les rappels suivants continueront de s’exécuter, contrairement à Rails 4.2. Lors de la mise à niveau, le comportement Rails 4.2 restera, bien que vous puissiez passer au comportement Rails 5.0 en ajoutant :

ActiveSupport.halt_callback_chains_on_return_false = false

au fichier config/application.rb. Vous pouvez arrêter explicitement la chaîne de rappel en appelant throw(:abort).

Dans Rails 5.0, ActiveJob héritera de ApplicationJob, plutôt que de ActiveJob::Base comme dans Rails 4.2. Pour mettre à niveau vers Rails 5.0, créez un fichier appelé application_job.rb dans le dossier app/jobs/. Modifiez le contenu de ce fichier pour qu’il soit :

class ApplicationJob < ActiveJob::Base
end

Ensuite, vous devez modifier tous vos travaux pour qu’ils héritent de ApplicationJob plutôt que de ActiveJob::Base.

L’un des autres changements les plus importants de Rails 5.0 ne nécessite aucune modification du code, mais changera la façon dont vous utilisez la ligne de commande avec vos applications Rails. Vous pourrez utiliser bin/rails, ou simplement rails, pour exécuter des tâches et des tests. Par exemple, au lieu d’utiliser $ rake db:migrate, vous pouvez maintenant faire $ rails db:migrate. Si vous exécutez $ bin/rails, vous pouvez voir toutes les commandes disponibles. Notez que de nombreuses tâches qui peuvent maintenant être exécutées avec bin/rails fonctionnent toujours avec rake.
