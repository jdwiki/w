Action Controller est le C dans MVC. Une fois que le routeur a déterminé quel contrôleur utiliser pour une demande, le contrôleur est chargé de donner un sens à la demande et de produire la sortie.

Le contrôleur recevra la demande, récupérera ou enregistrera les données d’un modèle et utilisera une vue pour créer une sortie. Un contrôleur peut être considéré comme un intermédiaire entre les modèles et les vues. Il met les données du modèle à la disposition de la vue afin qu’elles puissent être affichées à l’utilisateur, et il enregistre ou met à jour les données de l’utilisateur dans le modèle.

Filtres #

Les filtres sont des méthodes exécutées “avant”, “après” ou “autour” d’une action du contrôleur. Ils sont hérités, donc si vous en définissez dans votre ApplicationController, ils seront exécutés pour chaque requête reçue par votre application.

Avant le filtre

Les filtres avant sont exécutés avant l’action du contrôleur et peuvent arrêter la requête (et/ou la redirection). Une utilisation courante consiste à vérifier si un utilisateur est connecté :

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

Avant que les filtres ne soient exécutés sur les requêtes avant que la requête n’atteigne l’action du contrôleur. Il peut lui-même renvoyer une réponse et contourner complètement l’action.

Une autre utilisation courante des filtres avant consiste à valider l’authentification d’un utilisateur avant de lui accorder l’accès à l’action désignée pour traiter sa demande. Je les ai également vus utilisés pour charger une ressource à partir de la base de données, vérifier les autorisations sur une ressource ou gérer les redirections dans d’autres circonstances.

Après le filtre

Les filtres après sont similaires à ceux “avant”, mais lorsqu’ils sont exécutés après l’exécution de l’action, ils ont accès à l’objet de réponse qui est sur le point d’être envoyé. Donc, en bref, après l’exécution des filtres après la fin de l’action. Il peut modifier la réponse. La plupart du temps, si quelque chose est fait dans un filtre après, cela peut être fait dans l’action elle-même, mais s’il y a une logique à exécuter après l’exécution d’un ensemble d’actions, alors un filtre après est un bon endroit pour faire ce.

Généralement, j’ai vu après et autour des filtres utilisés pour la journalisation.

Autour du filtre

Les filtres Around peuvent avoir une logique avant et après l’exécution de l’action. Il cède simplement à l’action en quelque lieu que ce soit. Notez qu’il n’a pas besoin de céder à l’action et peut s’exécuter sans le faire comme un filtre avant.

Les filtres Around sont responsables de l’exécution de leurs actions associées en cédant, de la même manière que les middlewares Rack fonctionnent.

Autour des rappels enveloppent l’exécution des actions. Vous pouvez écrire un rappel autour dans deux styles différents. Dans le premier, le rappel est un seul morceau de code. Ce code est appelé avant l’exécution de l’action. Si le code de rappel appelle yield, l’action est exécutée. Une fois l’action terminée, le code de rappel continue de s’exécuter. Ainsi, le code avant le rendement est comme un rappel avant l’action et le code après le rendement est le rappel après l’action. Si le code de rappel n’invoque jamais yield. l’action n’est pas exécutée - c’est la même chose que d’avoir un rappel d’action avant qui renvoie faux.

Voici un exemple de filtre autour :

around_filter :catch_exceptions
 
private
  def catch_exceptions
    begin
      yield
    rescue Exception => e 
      logger.debug "Caught exception! #{e.message}"
    end
  end

Cela interceptera l’exception de toute action et placera le message dans votre journal. Vous pouvez utiliser des filtres autour pour la gestion des exceptions, la configuration et le démontage, et une myriade d’autres cas.

Uniquement et Sauf

Tous les filtres peuvent être appliqués à des actions spécifiques, en utilisant :only et :except :

class ProductsController < ApplicationController
  before_action :set_product, only: [:show, :edit, :update]

  # ... controller actions

  # Define your filters as controller private methods
  private

  def set_product
    @product = Product.find(params[:id])
  end
end

Filtre de saut

Tous les filtres (ceux hérités également) peuvent également être ignorés pour certaines actions spécifiques :

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

class HomeController < ApplicationController
  skip_before_action :authenticate_user!, only: [:index]

  def index
  end
end

Comme ils sont hérités, les filtres peuvent également être définis dans un contrôleur “parent” namespace. Supposons, par exemple, que vous ayez un espace de noms “admin” et que vous souhaitiez bien sûr que seuls les utilisateurs administrateurs puissent y accéder. Vous pourriez faire quelque chose comme ceci :

# config/routes.rb
namespace :admin do
  resources :products
end

# app/controllers/admin_controller.rb
class AdminController < ApplicationController
  before_action :authenticate_admin_user!

  private

  def authenticate_admin_user!
    redirect_to root_path unless current_user.admin?
  end
end

# app/controllers/admin/products_controller.rb
class Admin::ProductsController < AdminController
  # This controller will inherit :authenticate_admin_user! filter
end

Attention, dans Rails 4.x, vous pouvez utiliser before_filter avec before_action, mais before_filter est actuellement obsolète dans Rails 5.0.0 et sera supprimé dans 5.1.

Génération d’un contrôleur #

Rails fournit beaucoup de générateurs, pour les contrôleurs aussi bien sûr.

Vous pouvez générer un nouveau contrôleur en exécutant cette commande dans votre dossier d’application

rails generate controller NAME [action action] [options]

Remarque : Vous pouvez également utiliser l’alias rails g pour appeler rails generate

Par exemple, pour générer un contrôleur pour un modèle Product, avec les actions #index et #show, vous exécuterez

rails generate controller products index show

Cela créera le contrôleur dans app/controllers/products_controller.rb, avec les deux actions que vous avez spécifiées

class ProductsController < ApplicationController
  def index
  end

  def show
  end
end

Il créera également un dossier products dans app/views/, contenant les deux modèles pour les actions de votre contrôleur (c’est-à-dire index.html.erb et show.html.erb, notez que l’extension peut varier selon à votre moteur de template, donc si vous utilisez slim, par exemple, le générateur créera index.html.slim et show.html.slim )

De plus, si vous avez spécifié des actions, elles seront également ajoutées à votre fichier “routes”.

# config/routes.rb
get 'products/show'
get 'products/index'

Rails crée un fichier d’aide pour vous, dans app/helpers/products_helper.rb, ainsi que les fichiers d’assets dans app/assets/javascripts/products.js et app/assets/stylesheets/products.css. En ce qui concerne les vues, le générateur modifie ce comportement en fonction de ce qui est spécifié dans votre Gemfile : c’est-à-dire que si vous utilisez Coffeescript et Sass dans votre application, le générateur de contrôleur utilisera à la place le générateur products.coffee et produits.sass.

Enfin, mais non des moindres, Rails génère également des fichiers de test pour votre contrôleur, votre assistant et vos vues.

Si vous ne voulez pas que l’un d’entre eux soit créé car vous pouvez dire à Rails de les ignorer, ajoutez simplement n’importe quelle option avec

--no- ou --skip, comme ceci :

rails generate controller products index show --no-assets --no-helper

Et le générateur ignorera à la fois assets et helper

Si vous devez créer un contrôleur pour un namespace spécifique, ajoutez-le devant NAME :

rails generate controller admin/products

Cela créera votre contrôleur dans app/controllers/admin/products_controller.rb

Rails peut également générer un contrôleur RESTful complet pour vous :

rails generate scaffold_controller MODEL_NAME # available from Rails 4
rails generate scaffold_controller Product

Sauvetage d’ActiveRecord ::RecordNotFound avec redirect_to #

Vous pouvez récupérer une exception RecordNotFound avec une redirection au lieu d’afficher une page d’erreur :

class ApplicationController < ActionController::Base

  # your other stuff

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: I18n.t("errors.record_not_found")
  end
end

404 lorsque l’enregistrement est introuvable #

Rescue from record not found error au lieu d’afficher une exception ou une page blanche :

class ApplicationController < ActionController::Base
  
  # ... your other stuff here 

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: 'Record not found'
  end
end
