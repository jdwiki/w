Action Mailer vous permet d’envoyer des e-mails à partir de votre application à l’aide de classes et de vues de messagerie. Les mailers fonctionnent de manière très similaire aux contrôleurs. Ils héritent d’ActionMailer::Base et vivent dans l’application/les expéditeurs, et ils ont des vues associées qui apparaissent dans l’application/les vues.

Il est conseillé de traiter l’envoi d’email de manière asynchrone afin de ne pas monopoliser votre serveur web. Cela peut être fait via divers services tels que delayed_job.

Messagerie de base #

Cet exemple utilise quatre fichiers différents :

  • Le modèle Utilisateur
  • L’expéditeur de l’utilisateur
  • Le modèle html pour l’e-mail
  • Le modèle de texte brut pour l’e-mail

Dans ce cas, le modèle utilisateur appelle la méthode « approved » dans le mailer et transmet le « post » qui a été approuvé (la méthode « approved » dans le modèle peut être appelée par un rappel, à partir d’une méthode de contrôleur, etc.). Ensuite, l’expéditeur génère l’e-mail à partir du modèle html ou en texte brut en utilisant les informations du « message » transmis (par exemple, le titre). Par défaut, le mailer utilise le modèle avec le même nom que la méthode dans le mailer (c’est pourquoi la méthode du mailer et les templates ont le nom ‘approuvé’).

user_mailer.rb #

class UserMailer < ActionMailer::Base
  default from: "donotreply@example.com"

  def approved(post)
      @title = post.title
      @user = post.user
      mail(to: @user.email, subject: "Your Post was Approved!")
  end
end

utilisateur.rb #

def approved(post)
    UserMailer.approved(post)
end

approuvé.html.erb #

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
<html>
    <head>
      <meta http-equiv="Content-Type" content="text/html; charset=utf-8">
      <title>Post Approved</title>
    </head>
    <body>
        <h2>Congrats <%= @user.name %>! Your post (#<%= @title %>) has been approved!</h2>
        <p>We look forward to your future posts!</p>
    </body>
</html>

approuvé.texte.erb #

Congrats <%= @user.name %>! Your post (#<%= @title %>) has been approved!
We look forward to your future posts!

Génération d’un nouveau courrier #

Pour générer un nouveau mailer, entrez la commande suivante

rails generate mailer PostMailer

Cela générera un fichier modèle vierge dans app/mailers/post_mailer.rb nommé PostMailer

class PostMailer < ApplicationMailer
end

Deux fichiers de mise en page seront également générés pour la vue e-mail, un pour le format html et un pour le format texte.

Si vous préférez ne pas utiliser le générateur, vous pouvez créer vos propres mailers. Assurez-vous qu’ils héritent de ActionMailer::Base

Ajout de pièces jointes #

ActionMailer permet également de joindre des fichiers.

attachments['filename.jpg'] = File.read('/path/to/filename.jpg')

Par défaut, les pièces jointes seront encodées avec Base64. Pour changer cela, vous pouvez ajouter un hachage à la méthode des pièces jointes.

attachments['filename.jpg'] = {
  mime_type: 'application/gzip',
  encoding: 'SpecialEncoding',
  content: encoded_content
}

Vous pouvez également ajouter des pièces jointes en ligne

attachments.inline['image.jpg'] = File.read('/path/to/image.jpg')

Rappels ActionMailer #

ActionMailer prend en charge trois rappels

  • avant_action
  • après_action
  • autour_action

Fournissez-les dans votre classe Mailer

class UserMailer < ApplicationMailer
  after_action :set_delivery_options, :prevent_delivery_to_guests, :set_business_headers

Créez ensuite ces méthodes sous le mot-clé “private”

private
  def set_delivery_options
  end

  def prevent_delivery_to_guests
  end

  def set_business_headers
  end
end

Intercepteur ActionMailer #

Action Mailer fournit des crochets dans les méthodes d’intercepteur. Ceux-ci vous permettent d’enregistrer des classes qui sont appelées pendant le cycle de vie de la livraison du courrier.

Une classe intercepteur doit implémenter la méthode :delivering_email(message) qui sera appelée avant l’envoi de l’e-mail, vous permettant d’apporter des modifications à l’e-mail avant qu’il n’atteigne les agents de livraison. Votre classe doit apporter toutes les modifications nécessaires directement à l’instance Mail::Message transmise.

Il peut être utile pour les développeurs de s’envoyer des e-mails à eux-mêmes et non à de vrais utilisateurs.

Exemple d’enregistrement d’un intercepteur actionmailer :

# config/initializers/override_mail_recipient.rb

if Rails.env.development? or Rails.env.test?
  class OverrideMailRecipient
    def self.delivering_email(mail)
      mail.subject = 'This is dummy subject'
      mail.bcc = 'test_bcc@noemail.com'
      mail.to = 'test@noemail.com'
    end
  end
  ActionMailer::Base.register_interceptor(OverrideMailRecipient)
end

Générer une newsletter programmée #

Créez le modèle Newsletter :

<pré> rails g modèle Newsletter name:string email:string

subl app/models/newsletter.rb

valide :nom, présence : vrai valide :email, présence : vrai

Créez le contrôleur Newsletter :

<pré> rails g Créer des newsletters de contrôleur

class NewslettersController < Contrôleur d’application skip_before_action :authenticate_user! before_action :set_newsletter, only: [:destroy]

def create
  @newsletter = Newsletter.create(newsletter_params)
  if @newsletter.save
    redirect_to blog_index_path
  else
    redirect_to root_path
  end
end

private

  def set_newsletter
    @newsletter = Newsletter.find(params[:id])
  end

  def newsletter_params
    params.require(:newsletter).permit(:name, :email)
  end

fin

Après cela, remplacez la vue create.html.erb par le nom suivant. Nous allons convertir ce fichier en vue partielle qui sera stockée dans le Pied de page. Le nom sera _form.html.erb.

| Changer le nom du fichier de : | À : | |:———————————————————— -|:———————————————————— —-| | app/views/newsletters/create.html.erb | app/views/newsletters/_form.html.erb |

Après cela, définissez les itinéraires :

<pré> subl app/config/routes.rb

ressources : bulletins d’information

Plus tard, nous devons définir le formulaire que nous utiliserons pour enregistrer chaque courrier :

<pré> subl app/views/newsletters/_form.html.erb

<%= form_for (Newsletter.new) faire |f| %> <div class=“col-md-12” style=“margin: 0 auto; padding: 0;"> <div class=“col-md-6” style=“padding: 0;"> <%= f.text_field :name, class: ‘form-control’, placeholder:‘Nombre’ %> </div> <div class=“col-md-6” style=“padding: 0;"> <%= f.text_field :email, class: ‘form-control’, placeholder:‘Email’ %> </div> </div> <div class=“col-md-12” style=“margin: 0 auto; padding:0;"> <%= f.submit class:“col-md-12 tran3s s-color-bg hvr-shutter-out-horizontal”, style:‘border: none; color: white; cursor: pointer; margin: 0.5em auto; padding: 0.75em; width: 100%;’ %> </div> <% fin %>

Et après cela, insérez dans le pied de page :

<pré> subl app/views/layouts/_footer.html.erb

<%= rendre ’newsletters/formulaire’ %>

Maintenant, installez le -letter_opener- pour pouvoir prévisualiser l’e-mail dans le navigateur par défaut au lieu de l’envoyer. Cela signifie que vous n’avez pas besoin de configurer la livraison des e-mails dans votre environnement de développement et que vous n’avez plus à vous soucier d’envoyer accidentellement un e-mail de test à l’adresse de quelqu’un d’autre.

Ajoutez d’abord la gemme à votre environnement de développement et exécutez la commande bundle pour l’installer.

<pré> subl votre_projet/Gemfile

gem “lettre_ouvreur”, :group => :development

Définissez ensuite la méthode de livraison dans l’environnement de développement :

<pré> subl your_project/app/config/environments/development.rb

config.action_mailer.delivery_method = :letter_opener

Maintenant, créez une Structure Mailer pour gérer l’ensemble des mailers sur lesquels nous allons travailler. En aérogare

<pré> rails génère un message UserMailer newsletter_mailer

Et à l’intérieur de UserMailer, nous devons créer une méthode appelée Newsletter Mailer qui sera créée pour contenir à l’intérieur le dernier article de blog et sera déclenchée avec une action de rake. Nous supposerons que vous avez créé une structure de blog auparavant.

<pré> subl votre_projet/app/mailers/user_mailer.rb

classe UserMailer < ApplicationMailer par défaut depuis : ‘votre_compte_gmail@gmail.com’

def newsletter_mailer @newsletter = Newsletter.all @post = Post.last(3) emails = @newsletter.collect(&:email).join(", ") mail(to: emails, subject: "Hi, this is a test mail.") fin fin

Ensuite, créez le Mailer Template :

<pré> subl votre_projet/app/views/user_mailer/newsletter_mailer.html.erb

<p> Chers abonnés : </p> <p> Ce sont les dernières entrées de notre blog. Nous vous invitons à lire et à partager tout ce que nous avons fait cette semaine. </p>

<br/> <tableau> <% @post.each faire |post| %> <%#= link_to blog_url(post) do %> <tr style=“display:flex; float:left; clear:both;"> <td style=“display:flex; float:left; clear:both; height: 80px; width: 100px;"> <% if post.cover_image.present? %> <%= image_tag post.cover_image.fullsize.url, class:“principal-home-image-slider” %> <%# else %> <%#= image_tag ‘http://your_site_project.com’ + post.cover_video, class:“principal-home-image-slider” %> <%#= raw(video_embed(post.cover_video)) %> <% end %> </td> <td> <h3> <%= link_to post.title, :controller => “blog”, :action => “show”, :only_path => false, :id => post.id %> </h3> <p><%= post.subtitle %></p> </td> <td style=“display:flex; float:left; clear:both;">   </td> </tr> <%# fin %> <% fin %> </tableau>

Puisque nous voulons envoyer l’e-mail en tant que processus séparé, créons une tâche Rake pour déclencher l’e-mail. Ajoutez un nouveau fichier appelé email_tasks.rake au répertoire lib/tasks de votre application Rails :

<pré> appuyez sur lib/taks/email_tasks.rake

desc ’e-mail de newsletter hebdomadaire’ tâche e-mail_newsletter_hebdomadaire : : environnement faire UserMailer.newsletter_mailer.deliver ! fin

Le send_digest_email: :environment signifie charger l’environnement Rails avant d’exécuter la tâche, afin que vous puissiez accéder aux classes d’application (comme UserMailer) dans la tâche.

Maintenant, l’exécution de la commande rake -T listera la tâche Rake nouvellement créée. Testez que tout fonctionne en exécutant la tâche et en vérifiant si l’e-mail est envoyé ou non.

Pour tester si la méthode mailer fonctionne, exécutez la commande rake :

<pré> râteau weekly_newsletter_email

À ce stade, nous avons une tâche de rake qui peut être planifiée à l’aide de crontab. Nous allons donc installer le Whenever Gem qui est utilisé pour fournir une syntaxe claire pour l’écriture et le déploiement des tâches cron.

<pré> subl votre_projet/Gemfile

gem ‘chaque fois’, nécessite : false

Après cela, exécutez la commande suivante pour créer un fichier config/schedule.rb initial pour vous (tant que le dossier config est déjà présent dans votre projet).

<pré> n’importe quand.

[ajouter] en écrivant `./config/schedule.rb’ [fait] à chaque fois !

Maintenant, dans le fichier de planification, nous devons créer notre ** CRON JOB ** et appeler la méthode de messagerie à l’intérieur de la détermination du CRON JOB pour exécuter certaines tâches sans assistance et dans une plage de temps sélectionnée. Vous pouvez utiliser différents types de syntaxe comme expliqué sur ce lien.

<pré> subl votre_projet/config/schedule.rb

chaque 1.jour, :at => ‘4:30 am’ faire ratifier ‘weekly_newsletter_email’ fin

Maintenant, pour tester que le ** travail Cron ** a été créé avec succès, nous pouvons utiliser la commande suivante pour lire depuis le terminal, notre travail planifié dans CRON SYNTAX :

<pré> your_project your_mac_user$ quand

30 4 * * * /bin/bash -l -c ‘cd /Users/your_mac_user/Desktop/your_project && RAILS_ENV=production bundle exec rake week-end_newsletter_email –silent’

Maintenant, pour exécuter le test dans l’environnement de développement, il est sage de définir la ligne suivante sur le fichier principal application.rb pour que l’application sache où se trouvent les modèles qu’elle utilisera.

<pré> subl votre_projet/config/application.rb

config.action_mailer.default_url_options = { :host => “http://localhost:3000/” }

Maintenant, pour laisser Capistrano V3 enregistrer le nouveau Cron Job à l’intérieur du serveur et le déclencheur qui déclenchera l’exécution de cette tâche, nous devons ajouter la condition suivante :

<pré> subl votre_projet/Capfile

exiger ‘chaque fois/capistrano’

Et insérez dans le fichier deploy l’identifiant que CRON JOB utilisera à propos de l’environnement et le nom de l’application.

<pré> subl votre_projet/config/deploy.rb

set :whenever_identifier, ->{ “#{fetch(:application)}_#{fetch(:rails_env)}” }

Et prêt, après avoir enregistré les modifications sur chaque fichier, exécutez la commande capistrano deploy :

<pré> limiter le déploiement en production

Et maintenant, votre JOB a été créé et calendaire pour exécuter la méthode Mailer, ce que je veux et dans la plage de temps que nous avons définie sur ces fichiers.
