Structure #

Comme Rails suit le modèle M ** V ** C, les “vues” sont l’endroit où se trouvent vos “modèles” pour vos actions.

Disons que vous avez un contrôleur articles_controller.rb. Pour ce contrôleur, vous auriez un dossier dans les vues appelé app/views/articles :

app
|-- controllers
|   '-- articles_controller.rb
|
'-- views
    '-- articles
    |   |- index.html.erb
    |   |- edit.html.erb
    |   |- show.html.erb
    |   |- new.html.erb
    |   '- _partial_view.html.erb
    |
    '-- [...]

Cette structure vous permet d’avoir un dossier pour chaque contrôleur. Lors de l’appel d’une action dans votre contrôleur, la vue appropriée sera rendue automatiquement.

// articles_controller.rb
class ArticlesController < ActionController::Base
  def show
  end
end

// show.html.erb
<h1>My show view</h1>

Partiels #

Les modèles partiels (partiels) sont un moyen de diviser le processus de rendu en morceaux plus gérables. Les partiels vous permettent d’extraire des morceaux de code de vos modèles pour séparer les fichiers et également de les réutiliser dans vos modèles.

Pour créer un partiel, créez un nouveau fichier commençant par un trait de soulignement : _form.html.erb

Pour rendre un partiel dans le cadre d’une vue, utilisez la méthode render dans la vue : <%= rendre "forme" %>

  • Remarque, le trait de soulignement est omis lors du rendu
  • Un partiel doit être rendu en utilisant son chemin s’il se trouve dans un dossier différent

Pour passer une variable dans le partiel en tant que variable locale, utilisez cette notation :

<%= render :partial => 'form', locals : { message : @post } %>

Les partiels sont également utiles lorsque vous devez réutiliser exactement le même code (philosophie DRY).

Par exemple, pour réutiliser le code <head>, créez un partiel nommé _html_header.html.erb, entrez votre code <head> à réutiliser et rendez le partiel chaque fois que nécessaire par : <%= render ' html_header' %>.

Objets partiels #

Les objets qui répondent à to_partial_path peuvent également être rendus, comme dans <%= render @post %>. Par défaut, pour les modèles ActiveRecord, ce sera quelque chose comme posts/post, donc en rendant réellement @post, le fichier views/posts/_post.html.erb sera rendu.

Un local nommé post sera automatiquement attribué. Au final, <%= render @post %> est un raccourci pour <%= render 'posts/post', post: @post %>.

Des collections d’objets qui répondent à to_partial_path peuvent également être fournies, telles que <%= render @posts %>. Chaque élément sera rendu consécutivement.

Pour créer un partiel global qui peut être utilisé n’importe où sans référencer son chemin exact, le partiel doit être situé dans le chemin views/application. L’exemple précédent a été modifié ci-dessous pour illustrer cette fonctionnalité.

AssetTagHelper #

Pour laisser les rails lier automatiquement et correctement les ressources (css/js/images), dans la plupart des cas, vous souhaitez utiliser des aides intégrées. (Documentation officielle)

Assistants d’image #

chemin_image #

Cela renvoie le chemin d’accès à un élément d’image dans app/assets/images.

image_path("edit.png") # => /assets/edit.png

URL de l’image #

Cela renvoie l’URL complète d’un élément d’image dans app/assets/images.

image_url("edit.png") # => http://www.example.com/assets/edit.png

tag_image #

Utilisez cet assistant si vous souhaitez inclure une balise <img src="" /> avec l’ensemble source.

image_tag("icon.png") # => <img src="/assets/icon.png" alt="Icon" />

assistants JavaScript #

javascript_include_tag #

Si vous souhaitez inclure un fichier JavaScript dans votre vue.

javascript_include_tag "application" # => <script src="/assets/application.js"></script>

chemin_javascript #

Cela renvoie le chemin de votre fichier JavaScript.

javascript_path "application" # => /assets/application.js

URL_javascript #

Cela renvoie l’URL complète de votre fichier JavaScript.

javascript_url "application" # => http://www.example.com/assets/application.js

Assistants de feuille de style #

stylesheet_link_tag #

Si vous souhaitez inclure un fichier CSS dans votre vue.

stylesheet_link_tag "application" # => <link href="/assets/application.css" media="screen" rel="stylesheet" />

stylesheet_path #

Cela renvoie le chemin de votre élément de feuille de style.

stylesheet_path "application" # => /assets/application.css

stylesheet_url #

Cela renvoie l’URL complète de votre élément de feuille de style.

stylesheet_url "application" # => http://www.example.com/assets/application.css

Exemple d’utilisation #

Lors de la création d’une nouvelle application Rails, vous aurez automatiquement deux de ces assistants dans app/views/layouts/application.html.erb

<%= stylesheet_link_tag    'application', media: 'all', 'data-turbolinks-track': 'reload' %>
<%= javascript_include_tag 'application', 'data-turbolinks-track': 'reload' %>

Cela génère :

// CSS
<link rel="stylesheet" media="all" href="/assets/application.self-e19d4b856cacba4f6fb0e5aa82a1ba9aa4ad616f0213a1259650b281d9cf6b20.css?body=1" data-turbolinks-track="reload" />
// JavaScript
<script src="/assets/application.self-619d9bf310b8eb258c67de7af745cafbf2a98f6d4c7bb6db8e1b00aed89eb0b1.js?body=1" data-turbolinks-track="reload"></script>

Remplacer le code HTML dans les vues #

Si vous avez toujours voulu déterminer le contenu html à imprimer sur une page pendant l’exécution, rails a une très bonne solution pour cela. Il a quelque chose appelé content_for qui nous permet de passer un bloc à une vue rails. Veuillez vérifier l’exemple ci-dessous,

Déclarer content_for

<div>
  <%= yield :header %>
</div>

<% content_for :header do %>
 <ul>
   <li>Line Item 1</li>
   <li>Line Item 2</li>
 </ul>
<% end %>

HAML - une autre façon d’utiliser dans vos vues #

HAML (langage de balisage d’abstraction HTML) est une façon belle et élégante de décrire et de concevoir le HTML de vos vues. Au lieu de balises d’ouverture et de fermeture, HAML utilise l’indentation pour la structure de vos pages. Fondamentalement, si quelque chose doit être placé dans un autre élément, il suffit de l’indenter en utilisant un taquet de tabulation. Les onglets et les espaces blancs sont importants dans HAML, alors assurez-vous de toujours utiliser le même nombre d’onglets.

Exemples:

#myview.html.erb
<h1><%= @the_title %></h1>
<p>This is my form</p>
<%= render "form" %>

Et dans HAML :

#myview.html.haml
%h1= @the_title
%p
    This is my form
= render 'form'

Vous voyez, la structure de la mise en page est beaucoup plus claire que d’utiliser HTML et ERB.

Installation

Installez simplement la gemme en utilisant

gem install haml

et ajouter la gemme au Gemfile

gem "haml"

Pour utiliser HAML au lieu de HTML/ERB, remplacez simplement les extensions de fichier de vos vues de something.html.erb par something.html.haml.

Conseils rapides

Les éléments communs comme les divs peuvent être écrits de manière courte

HTML

<div class="myclass">My Text</div>

HAMLET

%div.myclass

HAML, raccourci

.myclass

Les attributs

HTML

<p class="myclass" id="myid">My paragraph</p>

HAMLET

%p{:class => "myclass", :id => "myid"} My paragraph

Insertion du code ruby

Vous pouvez insérer du code ruby ​​avec les signes = et -.

= link_to "Home", home_path

Le code commençant par = sera exécuté et intégré dans le document.

Le code commençant par - sera exécuté, mais pas inséré dans le document.

Documentation complète

HAML est très facile à démarrer, mais il est aussi très complexe, c’est pourquoi je vous recommande [de lire la documentation][1].

[1] : http://haml.info/docs/yardoc/file.REFERENCE.html#using_haml
