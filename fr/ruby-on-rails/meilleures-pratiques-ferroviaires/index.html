Gros modèle, contrôleur maigre #

“Fat Model, Skinny Controller” fait référence à la façon dont les parties M et C de MVC fonctionnent idéalement ensemble. À savoir, toute logique liée à la non-réponse doit être intégrée au modèle, idéalement dans une méthode agréable et testable. Pendant ce temps, le contrôleur “maigre” est simplement une belle interface entre la vue et le modèle.

En pratique, cela peut nécessiter une gamme de différents types de refactoring, mais tout se résume à une idée : en déplaçant toute logique qui ne concerne pas la réponse vers le modèle (au lieu du contrôleur), non seulement vous avez favorisé la réutilisation dans la mesure du possible, mais vous avez également permis de tester votre code en dehors du contexte d’une requête.

Prenons un exemple simple. Supposons que vous ayez un code comme celui-ci :

def index
  @published_posts = Post.where('published_at <= ?', Time.now)
  @unpublished_posts = Post.where('published_at IS NULL OR published_at > ?', Time.now)
end

Vous pouvez le changer en ceci :

def index
  @published_posts = Post.published
  @unpublished_posts = Post.unpublished
end

Ensuite, vous pouvez déplacer la logique vers votre modèle de publication, où elle pourrait ressembler à ceci :

scope :published, ->(timestamp = Time.now) { where('published_at <= ?', timestamp) }
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) }

Objets de domaine (plus de gros modèles) #

“Fat Model, Skinny Controller” est une très bonne première étape, mais elle ne s’adapte pas bien une fois que votre base de code commence à se développer.

Réfléchissons à la responsabilité unique des modèles. Quelle est la responsabilité unique des mannequins ? Est-ce pour maintenir la logique métier ? Est-ce pour tenir une logique liée à la non-réponse ?

Non. Sa responsabilité est de gérer la couche de persistance et son abstraction.

La logique métier, ainsi que toute logique non liée à la réponse et logique non liée à la persistance, doit aller dans les objets de domaine.

Les objets de domaine sont des classes conçues pour n’avoir qu’une seule responsabilité dans le domaine du problème. Laissez vos cours “Scream Their Architecture” pour les problèmes qu’ils résolvent.

En pratique, vous devriez vous efforcer d’obtenir des modèles maigres, des vues maigres et des contrôleurs maigres. L’architecture de votre solution ne doit pas être influencée par le framework que vous choisissez.

Par exemple

Disons que vous êtes une place de marché qui facture une commission fixe de 15 % à vos clients via Stripe. Si vous facturez une commission fixe de 15 %, cela signifie que votre commission change en fonction du montant de la commande car Stripe facture 2,9 % + 30 ¢.

Le montant que vous facturez comme commission doit être : montant*0,15 - (montant*0,029 + 0,30).

N’écrivez pas cette logique dans le modèle :

# app/models/order.rb
class Order < ActiveRecord::Base
  SERVICE_COMMISSION = 0.15
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  ...

  def commission
    amount*SERVICE_COMMISSION - stripe_commission  
  end

  private

  def stripe_commission
    amount*STRIPE_PERCENTAGE_COMMISSION + STRIPE_FIXED_COMMISSION
  end
end

Dès que vous intégrerez un nouveau mode de paiement, vous ne pourrez plus mettre à l’échelle cette fonctionnalité dans ce modèle.

Aussi, dès que vous commencerez à intégrer plus de logique métier, votre objet Order commencera à perdre cohésion.

Privilégiez les objets du domaine, avec le calcul de la commission totalement soustrait à la responsabilité des commandes persistantes :

# app/models/order.rb
class Order < ActiveRecord::Base
  ...
  # No reference to commission calculation
end

# lib/commission.rb
class Commission
  SERVICE_COMMISSION = 0.15

  def self.calculate(payment_method, model)
    model.amount*SERVICE_COMMISSION - payment_commission(payment_method, model)  
  end

  private

  def self.payment_commission(payment_method, model)
    # There are better ways to implement a static registry,
    # this is only for illustration purposes.
    Object.const_get("#{payment_method}Commission").calculate(model)
  end
end

# lib/stripe_commission.rb
class StripeCommission
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  def self.calculate(model)
    model.amount*STRIPE_PERCENTAGE_COMMISSION
      + STRIPE_PERCENTAGE_COMMISSION
  end
end

# app/controllers/orders_controller.rb
class OrdersController < ApplicationController
  def create
    @order = Order.new(order_params)
    @order.commission = Commission.calculate("Stripe", @order)
    ...
  end
end

L’utilisation d’objets de domaine présente les avantages architecturaux suivants :

  • il est extrêmement facile de tester unitaire, car aucun appareil ou fabrique n’est nécessaire pour instancier les objets avec la logique.
  • fonctionne avec tout ce qui accepte le message amount.
  • garde chaque objet de domaine petit, avec des responsabilités clairement définies et avec une plus grande cohésion.
  • s’adapte facilement aux nouvelles méthodes de paiement par ajout, pas modification.
  • arrête la tendance à avoir un objet User sans cesse croissant dans chaque application Ruby on Rails.

Personnellement, j’aime mettre des objets de domaine dans lib. Si vous le faites, n’oubliez pas de l’ajouter à autoload_paths :

# config/application.rb
config.autoload_paths << Rails.root.join('lib')

Vous pouvez également préférer créer des objets de domaine plus orientés vers l’action, en suivant le modèle Commande/Requête. Dans ce cas, placer ces objets dans app/commands pourrait être un meilleur endroit car tous les sous-répertoires app sont automatiquement ajoutés au chemin de chargement automatique.

Convention sur la configuration #

Dans Rails, vous vous retrouvez à regarder des contrôleurs, des vues et des modèles pour votre base de données.

Pour réduire le besoin de configuration lourde, Rails implémente des règles pour faciliter le travail avec l’application. Vous pouvez définir vos propres règles mais pour le début (et pour plus tard), c’est une bonne idée de s’en tenir aux conventions proposées par Rails.

Ces conventions accéléreront le développement, garderont votre code concis et lisible, et vous permettront une navigation facile à l’intérieur de votre application.

Les conventions réduisent également les barrières à l’entrée pour les débutants. Il y a tellement de conventions dans Rails qu’un débutant n’a même pas besoin de connaître, mais peut juste bénéficier de l’ignorance. Il est possible de créer de superbes applications sans savoir pourquoi tout est comme ça.

Par exemple

Si vous avez une table de base de données appelée orders avec la clé primaire id, le modèle correspondant s’appelle order et le contrôleur qui gère toute la logique s’appelle orders_controller. La vue est divisée en différentes actions : si le contrôleur a une action “nouveau” et “modifier”, il existe également une vue “nouveau” et “modifier”.

Par exemple

Pour créer une application, vous exécutez simplement rails new app_name. Cela générera environ 70 fichiers et dossiers qui constituent l’infrastructure et la base de votre application Rails.

Il comprend:

  • Dossiers pour contenir vos modèles (couche de base de données), contrôleurs et vues
  • Dossiers pour contenir les tests unitaires de votre application
  • Des dossiers pour contenir vos ressources Web comme les fichiers Javascript et CSS
  • Fichiers par défaut pour les réponses HTTP 400 (c’est-à-dire fichier introuvable)
  • Beaucoup d’autres

Méfiez-vous de default_scope #

ActiveRecord inclut default_scope, pour définir automatiquement la portée d’un modèle par défaut.

class Post
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

Le code ci-dessus servira les messages qui sont déjà publiés lorsque vous effectuez une requête sur le modèle.

Post.all # will only list published posts

Cette portée, bien qu’inoffensive, a de multiples effets secondaires cachés que vous ne voudrez peut-être pas.

default_scope et order #

Puisque vous avez déclaré une commande dans default_scope, l’appel de commande sur Post sera ajouté en tant que commandes supplémentaires au lieu de remplacer la valeur par défaut.

Post.order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."created_at" DESC, "posts"."updated_at" DESC

Ce n’est probablement pas le comportement que vous vouliez ; vous pouvez remplacer cela en excluant d’abord la “commande” de la portée

Post.except(:order).order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."updated_at" DESC

default_scope et initialisation du modèle #

Comme avec tout autre ActiveRecord::Relation, default_scope modifiera l’état par défaut des modèles initialisés à partir de celui-ci.

Dans l’exemple ci-dessus, Post a where(published: true) défini par défaut, et donc les nouveaux modèles de Post l’auront également défini.

Post.new # => <Post published: true>

non délimité #

default_scope peut théoriquement être effacé en appelant d’abord unscoped, mais cela a également des effets secondaires. Prenons, par exemple, un modèle STI :

class Post < Document
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

Par défaut, les requêtes sur Post seront étendues aux colonnes type contenant 'Post'. Mais unscoped effacera cela avec votre propre default_scope, donc si vous utilisez unscoped, vous devez également vous rappeler d’en tenir compte.

Post.unscoped.where(type: 'Post').order(updated_at: :desc)

unscoped et associations de modèles #

Envisagez une relation entre Post et User

class Post < ApplicationRecord
  belongs_to :user
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

class User < ApplicationRecord
  has_many :posts
end

En obtenant un Utilisateur individuel, vous pouvez voir les publications qui s’y rapportent :

user = User.find(1)
user.posts

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' AND "posts"."user_id" = ? ORDER BY "posts"."created_at" DESC [["user_id", 1]]

Mais vous voulez effacer le default_scope de la relation posts, vous utilisez donc unscoped

user.posts.unscoped

SELECT "posts".* FROM "posts"

Cela efface la condition user_id ainsi que default_scope.

Un exemple de cas d’utilisation pour default_scope #

Malgré tout cela, il existe des situations où l’utilisation de default_scope est justifiable.

Considérez un système multi-tenant où plusieurs sous-domaines sont servis à partir de la même application mais avec des données isolées. Une façon d’obtenir cet isolement consiste à utiliser default_scope. Les inconvénients dans d’autres cas deviennent des avantages ici.

class ApplicationRecord < ActiveRecord::Base
  def self.inherited(subclass)
    super

    return unless subclass.superclass == self
    return unless subclass.column_names.include? 'tenant_id'

    subclass.class_eval do
      default_scope ->{ where(tenant_id: Tenant.current_id) }
    end
  end
end

Tout ce que vous avez à faire est de définir Tenant.current_id sur quelque chose au début de la requête, et toute table contenant tenant_id sera automatiquement étendue sans code supplémentaire. Les enregistrements d’instanciation hériteront automatiquement de l’ID de locataire sous lequel ils ont été créés.

La chose importante à propos de ce cas d’utilisation est que la portée est définie une fois par demande et qu’elle ne change pas. Les seuls cas où vous aurez besoin de “non délimité” ici sont des cas particuliers tels que les travailleurs en arrière-plan qui s’exécutent en dehors de la portée d’une requête.

Ne vous répétez pas (DRY) #

Pour aider à maintenir un code propre, Rails suit le principe de DRY.

Cela implique, dans la mesure du possible, de réutiliser autant de code que possible plutôt que de dupliquer un code similaire à plusieurs endroits (par exemple, en utilisant des partiels). Cela réduit * les erreurs *, garde votre code * propre * et applique le principe de * écrire le code une fois * puis de le réutiliser. Il est également plus facile et plus efficace de mettre à jour le code à un seul endroit que de mettre à jour plusieurs parties du même code. Rendant ainsi votre code plus modulaire et robuste.

Aussi Fat Model, Skinny Controller est DRY, parce que vous écrivez le code dans votre modèle et dans le contrôleur ne faites que l’appel, comme :

# Post model
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) } 


# Any controller
def index
    ....
    @unpublished_posts = Post.unpublished
    ....
end

def others
    ...
    @unpublished_posts = Post.unpublished
    ...
end

Cela permet également d’aboutir à une structure pilotée par l’API dans laquelle les méthodes internes sont masquées et les modifications sont apportées en passant des paramètres à la manière d’une API.

Tu n’en auras pas besoin (YAGNI) #

Si vous pouvez dire “YAGNI” (vous n’en aurez pas besoin) à propos d’une fonctionnalité, il vaut mieux ne pas l’implémenter. Il peut y avoir beaucoup de temps de développement économisé en se concentrant sur la simplicité. La mise en œuvre de telles fonctionnalités de toute façon peut entraîner des problèmes :

Problèmes #

## Suringénierie Si un produit est plus compliqué qu’il ne doit l’être, il est sur-conçu. Habituellement, ces fonctionnalités “pas encore utilisées” ne seront jamais utilisées de la manière prévue pour laquelle elles ont été écrites et doivent être refactorisées si jamais elles sont utilisées. Les optimisations prématurées, en particulier les optimisations de performances, conduisent souvent à des décisions de conception qui se révéleront fausses à l’avenir.

Gonflement du code #

Code Bloat signifie code inutilement compliqué. Cela peut se produire par exemple par abstraction, redondance ou application incorrecte des modèles de conception. La base de code devient difficile à comprendre, déroutante et coûteuse à maintenir.

Fonctionnalité fluage #

Feature Creep fait référence à l’ajout de nouvelles fonctionnalités qui vont au-delà de la fonctionnalité de base du produit et conduisent à une complexité inutilement élevée du produit.

Long temps de développement #

Le temps qui pourrait être utilisé pour développer des fonctionnalités nécessaires est consacré à développer des fonctionnalités inutiles. Le produit prend plus de temps à livrer.

Solutions #

KISS - Restez simple, stupide #

Selon KISS, la plupart des systèmes fonctionnent mieux s’ils sont conçus simplement. La simplicité devrait être un objectif de conception principal pour réduire la complexité. Cela peut être réalisé en suivant le «principe de responsabilité unique» par exemple.

YAGNI - Vous n’en aurez pas besoin #

Moins est plus. Pensez à chaque fonctionnalité, est-ce vraiment nécessaire ? Si vous pouvez penser d’une manière ou d’une autre que c’est YAGNI, laissez-le de côté. Il vaut mieux le développer quand c’est nécessaire.

Refactoring continu #

Le produit est constamment amélioré. Avec le refactoring, nous pouvons nous assurer que le produit est réalisé selon les meilleures pratiques et ne dégénère pas en un patchwork.
