Paramètres #

Classe/Méthode Le pourquoi
Classe UserProfile() La classe UserProfile étend le [modèle utilisateur par défaut de Django][1].
méthode create_profile() La méthode create_profile() est exécutée chaque fois qu’un modèle Django User [signal post_save est libéré][2].

[1] : https://docs.djangoproject.com/en/1.9/topics/auth/customizing/ [2] : https://docs.djangoproject.com/en/1.9/ref/signals/#post-save

Maintenant, les détails.

Les signaux Django sont un moyen d’informer votre application de certaines tâches (telles qu’un modèle avant ou après l’enregistrement ou la suppression) lorsqu’elles ont lieu.

Ces signaux vous permettent d’effectuer des actions de votre choix dès que le signal est relâché.

Par exemple, à chaque fois qu’un nouvel utilisateur Django est créé, le modèle utilisateur libère un signal, en associant des paramètres tels que sender=User vous permettant de cibler spécifiquement votre écoute des signaux sur une activité spécifique qui se produit, dans ce cas, une nouvelle création d’utilisateur.

Dans l’exemple ci-dessus, l’intention est de créer un objet UserProfile, immédiatement après la création d’un objet User. Par conséquent, en écoutant spécifiquement un signal post_save du modèle User (le modèle utilisateur Django par défaut), nous créons un objet UserProfile juste après la création d’un nouvel User.

La documentation Django fournit une documentation complète sur tous les [signaux disponibles][1] possibles.

Cependant, l’exemple ci-dessus est d’expliquer en termes pratiques un cas d’utilisation typique lorsque l’utilisation de Signaux peut être un complément utile.

“Un grand pouvoir implique de grandes responsabilités”. Il peut être tentant d’avoir des signaux dispersés dans l’ensemble de votre application ou de votre projet simplement parce qu’ils sont géniaux. Eh bien, non. Parce qu’ils sont cool, cela ne fait pas d’eux la solution idéale pour toutes les situations simples qui vous viennent à l’esprit.

Les signaux sont parfaits pour, comme d’habitude, pas tout. Connexion/déconnexion, les signaux sont excellents. Les modèles clés libèrent des signes, comme le modèle utilisateur, si ça va.

La création de signaux pour chaque modèle de votre application peut devenir écrasante à un moment donné et anéantir toute idée de l’utilisation sparring de Django Signals.

N’utilisez pas de signaux lorsque (basé sur [Two Scoops of Django book][2]):

  • Le signal se rapporte à un modèle particulier et peut être déplacé dans l’une des méthodes de ce modèle, éventuellement appelée par save().
  • Le signal peut être remplacé par une méthode de gestionnaire de modèle personnalisée.
  • Le signal se rapporte à une vue particulière et peut être déplacé dans cette vue

Il peut être acceptable d’utiliser des signaux lorsque :

  • Votre récepteur de signal doit apporter des modifications à plusieurs modèles.
  • Vous souhaitez envoyer le même signal à partir de plusieurs applications et les faire traiter de la même manière par un récepteur commun.
  • Vous souhaitez invalider un cache après une sauvegarde de modèle.
  • Vous avez un scénario inhabituel qui nécessite un rappel, et il n’y a pas d’autre moyen de le gérer que d’utiliser un signal. Par exemple, vous souhaitez déclencher quelque chose en fonction de save() ou init() du modèle d’une application tierce. Vous ne pouvez pas modifier le code tiers et l’étendre peut être impossible, donc un signal fournit un déclencheur pour un rappel.

[1] : https://docs.djangoproject.com/en/1.9/ref/signals/# [2] : https://www.twoscoopspress.com/products/two-scoops-of-django-1-8

Exemple d’extension de profil utilisateur #

Cet exemple est un extrait tiré de [Extending Django User Profile like a Pro][1]

from django.db import models
from django.contrib.auth.models import User
from django.db.models.signals import post_save
 
class UserProfile(models.Model):
    user = models.OneToOneField(User, related_name='user')
    website = models.URLField(default='', blank=True)
    bio = models.TextField(default='', blank=True)

def create_profile(sender, **kwargs):
    user = kwargs["instance"]
    if kwargs["created"]:
        user_profile = UserProfile(user=user)
        user_profile.save()
post_save.connect(create_profile, sender=User)

[1] : https://blog.khophi.co/extending-django-user-model-userprofile-like-a-pro/

Comment savoir s’il s’agit d’une insertion ou d’une mise à jour dans le signal pre_save #

En utilisant le pre_save, nous pouvons déterminer si une action save sur notre base de données concernait la mise à jour d’un objet existant ou la création d’un nouveau.

Pour ce faire, vous pouvez vérifier l’état de l’objet modèle :

    @receiver(pre_save, sender=User)
    def pre_save_user(sender, instance, **kwargs): 
        if not instance._state.adding:
            print ('this is an update')
        else:
            print ('this is an insert')

Désormais, chaque fois qu’une action “save” a lieu, le signal “pre_save” s’exécute et affiche :

  • ceci est une mise à jour si l’action dérive d’une action de mise à jour.
  • ceci est une insertion si l’action dérive d’une action d’insertion.

Notez que cette méthode ne nécessite aucune requête de base de données supplémentaire.

[1] : https://docs.djangoproject.com/en/1.11/ref/models/instances/#customizing-model-loading

Héritage des signaux sur les modèles étendus #

Les signaux de Django sont limités à des signatures de classe précises lors de l’enregistrement, et donc les modèles sous-classés ne sont pas immédiatement enregistrés sur le même signal.

Prenez ce modèle et ce signal par exemple

class Event(models.Model):
    user = models.ForeignKey(User)


class StatusChange(Event):
    ...


class Comment(Event):
    ...


def send_activity_notification(sender, instance: Event, raw: bool, **kwargs):
    """
    Fire a notification upon saving an event
    """

    if not raw:
        msg_factory = MessageFactory(instance.id)
        msg_factory.on_activity(str(instance))
post_save.connect(send_activity_notification, Event)

Avec les modèles étendus, vous devez attacher manuellement le signal à chaque sous-classe, sinon ils ne seront pas affectés.

post_save.connect(send_activity_notification, StatusChange)
post_save.connect(send_activity_notification, Comment)

Avec Python 3.6, vous pouvez tirer parti de certaines méthodes de classe supplémentaires intégrées aux classes pour automatiser cette liaison.

class Event(models.Model):

    @classmethod
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        post_save.connect(send_activity_notification, cls)
