Paramètres #

Réglage Est-ce que
Quelques Configurations Réglages de base pratiques qui vont avec Django-Allauth (que j’utilise la plupart du temps). Pour plus d’options de configuration, voir [Configurations][1]
ACCOUNT_AUTHENTICATION_METHOD (= “nom d’utilisateur” ou “email” ou “nom d’utilisateur_email”) Spécifie la méthode de connexion à utiliser - si l’utilisateur se connecte en saisissant son nom d’utilisateur, son adresse e-mail ou l’un des deux. Le définir sur “e-mail” nécessite ACCOUNT_EMAIL_REQUIRED=True
ACCOUNT_EMAIL_CONFIRMATION_EXPIRE_DAYS (=3) Détermine la date d’expiration des e-mails de confirmation (nombre de jours).
ACCOUNT_EMAIL_REQUIRED (=Faux) L’utilisateur est tenu de fournir une adresse e-mail lors de l’inscription. Cela va de pair avec le paramètre ACCOUNT_AUTHENTICATION_METHOD
ACCOUNT_EMAIL_VERIFICATION (=“facultatif”) Détermine la méthode de vérification des e-mails lors de l’inscription - choisissez l’une des options “obligatoire”, “facultative” ou “aucune”. Lorsqu’il est défini sur “obligatoire”, l’utilisateur ne peut pas se connecter jusqu’à ce que l’adresse e-mail soit vérifiée. Choisissez « facultatif » ou « aucun » pour autoriser les connexions avec une adresse e-mail non vérifiée. En cas de “optionnel”, le courrier de vérification de l’e-mail est toujours envoyé, alors qu’en cas de “aucun”, aucun courrier de vérification de l’e-mail n’est envoyé.
ACCOUNT_LOGIN_ATTEMPTS_LIMIT (=5) Nombre de tentatives de connexion infructueuses. Lorsque ce nombre est dépassé, il est interdit à l’utilisateur de se connecter pendant les secondes ACCOUNT_LOGIN_ATTEMPTS_TIMEOUT spécifiées. Bien que cela protège la vue de connexion allauth, cela ne protège pas la connexion administrateur de Django d’être brutalement forcée.
ACCOUNT_LOGOUT_ON_PASSWORD_CHANGE (=Faux) Détermine si l’utilisateur est automatiquement déconnecté ou non après avoir modifié ou défini son mot de passe.
SOCIALACCOUNT_PROVIDERS (= dict) Dictionnaire contenant les paramètres spécifiques au fournisseur.

[1] : https://django-allauth.readthedocs.io/en/latest/configuration.html

Manière simple : python-social-auth #

python-social-auth est un cadre qui simplifie le mécanisme d’authentification et d’autorisation sociale. Il contient de nombreux backends sociaux (Facebook, Twitter, Github, LinkedIn, etc.)

INSTALLER

Nous devons d’abord installer le package python-social-auth avec

pip install python-social-auth

ou [téléchargez][1] le code depuis github. C’est maintenant le bon moment pour l’ajouter à votre fichier requirements.txt.

CONFIGURATION settings.py

Dans le settings.py ajouter :

INSTALLED_APPS = (
    ...
    'social.apps.django_app.default',
    ...
)

CONFIGURATION DES BACKENDS

AUTHENTICATION_BACKENDS contient les backends que nous allons utiliser, et nous n’avons qu’à mettre ce dont nous avons besoin.

AUTHENTICATION_BACKENDS = (
    'social.backends.open_id.OpenIdAuth',
    'social.backends.google.GoogleOpenId',
    'social.backends.google.GoogleOAuth2',
    'social.backends.google.GoogleOAuth',
    'social.backends.twitter.TwitterOAuth',
    'social.backends.yahoo.YahooOpenId',
    ...
    'django.contrib.auth.backends.ModelBackend',
)

Votre projet settings.py n’a peut-être pas encore de champ AUTHENTICATION_BACKENDS. Si tel est le cas, ajoutez le champ. Assurez-vous de ne pas manquer 'django.contrib.auth.backends.ModelBackend', car il gère la connexion par nom d’utilisateur/mot de passe.

Si nous utilisons par exemple Facebook et Linkedin Backends, nous devons ajouter les clés API

SOCIAL_AUTH_FACEBOOK_KEY = 'YOURFACEBOOKKEY'
SOCIAL_AUTH_FACEBOOK_SECRET = 'YOURFACEBOOKSECRET'

et

SOCIAL_AUTH_LINKEDIN_KEY = 'YOURLINKEDINKEY'
SOCIAL_AUTH_LINKEDIN_SECRET = 'YOURLINKEDINSECRET'

Remarque : Vous pouvez obtenir les clés nécessaires dans [développeurs Facebook][2] et [développeurs Linkedin][3] et [ici][4] vous pouvez voir la liste complète et sa manière respective de spécifier la clé API et la clé secrète.

Remarque sur les clés secrètes : Les clés secrètes doivent être gardées secrètes. [Ici][5] est une explication de Stack Overflow qui est utile. [Ce tutoriel][6] est utile pour en savoir plus sur les variables environnementales.

TEMPLATE_CONTEXT_PROCESSORS aidera aux redirections, backends et autres choses, mais au début nous n’avons besoin que de ceux-ci :

TEMPLATE_CONTEXT_PROCESSORS = (
    ...
    'social.apps.django_app.context_processors.backends',
    'social.apps.django_app.context_processors.login_redirect',
    ...
)

Dans Django 1.8, la configuration de TEMPLATE_CONTEXT_PRPROCESSORS comme indiqué ci-dessus était obsolète. Si c’est le cas pour vous, vous l’ajouterez dans le dict TEMPLATES. Le vôtre devrait ressembler à ceci :

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [os.path.join(BASE_DIR, "templates")],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
                'social.apps.django_app.context_processors.backends',
                'social.apps.django_app.context_processors.login_redirect',
            ],
        },
    },
]

UTILISER UN UTILISATEUR PERSONNALISÉ

Si vous utilisez un modèle utilisateur personnalisé et que vous souhaitez vous y associer, ajoutez simplement la ligne suivante (toujours dans settings.py)

SOCIAL_AUTH_USER_MODEL = 'somepackage.models.CustomUser'

CustomUser est un modèle qui hérite ou abstrait de l’utilisateur par défaut.

CONFIGURATION urls.py

# if you haven't imported inlcude make sure you do so at the top of your file
from django.conf.urls import url, include

urlpatterns = patterns('',
    ...
    url('', include('social.apps.django_app.urls', namespace='social'))
    ...
)

Ensuite, vous devez synchroniser la base de données pour créer les modèles nécessaires :

./manage.py migrate

Enfin nous pouvons jouer !

dans certains modèles, vous devez ajouter quelque chose comme ceci :

    <a href="{% url 'social:begin' 'facebook' %}?next={{ request.path }}">Login with Facebook</a>
    <a href="{% url 'social:begin' 'linkedin' %}?next={{ request.path }}">Login with Linkedin</a>

si vous utilisez un autre backend, remplacez simplement “facebook” par le nom du backend.

Déconnexion des utilisateurs

Une fois que vous avez connecté les utilisateurs, vous voudrez probablement créer la fonctionnalité pour les déconnecter. Dans certains modèles, probablement près de l’endroit où le modèle de connexion a été affiché, ajoutez la balise suivante :

<a href="{% url 'logout' %}">Logout</a>

ou

<a href="/logout">Logout</a>

Vous voudrez modifier votre fichier urls.py avec un code similaire à :

url(r'^logout/$', views.logout, name='logout'),

Enfin, modifiez votre fichier views.py avec un code similaire à :

def logout(request):
    auth_logout(request)
    return redirect('/')

Utilisation de Django Allauth #

Pour tous mes projets, Django-Allauth est resté celui qui est facile à configurer, et sort de la boîte avec de nombreuses fonctionnalités, y compris mais sans s’y limiter :

  • Plus de 50+ authentifications de réseaux sociaux
  • Mélangez l’inscription des comptes locaux et sociaux
  • Plusieurs comptes sociaux
  • Inscription instantanée facultative pour les comptes sociaux - sans poser de questions
  • Gestion des adresses e-mail (adresses e-mail multiples, définition d’un primaire)
  • Flux de mot de passe oublié Flux de vérification de l’adresse e-mail

Si vous souhaitez vous salir les mains, Django-Allauth s’écarte du chemin, avec des configurations supplémentaires pour peaufiner le processus et l’utilisation de votre système d’authentification.

Les étapes ci-dessous supposent que vous utilisez Django 1.10+

Étapes de configuration :

pip installer django-allauth

Dans votre fichier settings.py, apportez les modifications suivantes :

# Specify the context processors as follows:
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                # Already defined Django-related contexts here

                # `allauth` needs this from django. It is there by default,
                # unless you've devilishly taken it away.
                'django.template.context_processors.request',
            ],
        },
    },
]

AUTHENTICATION_BACKENDS = (
    # Needed to login by username in Django admin, regardless of `allauth`
    'django.contrib.auth.backends.ModelBackend',

    # `allauth` specific authentication methods, such as login by e-mail
    'allauth.account.auth_backends.AuthenticationBackend',
)

INSTALLED_APPS = (
# Up here is all your default installed apps from Django

# The following apps are required:
'django.contrib.auth',
'django.contrib.sites',

'allauth',
'allauth.account',
'allauth.socialaccount',

# include the providers you want to enable:
'allauth.socialaccount.providers.google',
'allauth.socialaccount.providers.facebook',
)

# Don't forget this little dude.
SITE_ID = 1

Terminé avec les modifications dans le fichier settings.py ci-dessus, passez au fichier urls.py. Il peut s’agir de votre yourapp/urls.py ou de votre ProjectName/urls.py. Normalement, je préfère le ProjectName/urls.py.

urlpatterns = [
    # other urls here
    url(r'^accounts/', include('allauth.urls')),
    # other urls here
]

En ajoutant simplement include('allauth.urls'), vous obtenez ces URL gratuitement :

^accounts/ ^ ^signup/$ [name='account_signup']
^accounts/ ^ ^login/$ [name='account_login']
^accounts/ ^ ^logout/$ [name='account_logout']
^accounts/ ^ ^password/change/$ [name='account_change_password']
^accounts/ ^ ^password/set/$ [name='account_set_password']
^accounts/ ^ ^inactive/$ [name='account_inactive']
^accounts/ ^ ^email/$ [name='account_email']
^accounts/ ^ ^confirm-email/$ [name='account_email_verification_sent']
^accounts/ ^ ^confirm-email/(?P<key>[-:\w]+)/$ [name='account_confirm_email']
^accounts/ ^ ^password/reset/$ [name='account_reset_password']
^accounts/ ^ ^password/reset/done/$ [name='account_reset_password_done']
^accounts/ ^ ^password/reset/key/(?P<uidb36>[0-9A-Za-z]+)-(?P<key>.+)/$ [name='account_reset_password_from_key']
^accounts/ ^ ^password/reset/key/done/$ [name='account_reset_password_from_key_done']
^accounts/ ^social/
^accounts/ ^google/
^accounts/ ^twitter/
^accounts/ ^facebook/
^accounts/ ^facebook/login/token/$ [name='facebook_login_by_token']

Enfin, faites python ./manage.py migrate pour valider les migrations de Django-allauth dans la base de données.

Comme d’habitude, pour pouvoir vous connecter à votre application en utilisant n’importe quel réseau social que vous avez ajouté, vous devrez ajouter les détails du compte social du réseau.

Connectez-vous à Django Admin (localhost:8000/admin) et sous Social Applications dans les détails de votre compte social.

Vous aurez peut-être besoin de comptes chez chaque fournisseur d’authentification afin d’obtenir les détails à remplir dans les sections Applications sociales.

Pour des configurations détaillées de ce que vous pouvez avoir et modifier, consultez la [page Configurations][1].

[1] : https://django-allauth.readthedocs.io/en/latest/configuration.html
