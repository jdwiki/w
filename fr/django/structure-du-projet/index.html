Référentiel > Projet > Site/Conf #

Pour un projet Django avec requirements et deployment tools sous contrôle de code source. Cet exemple s’appuie sur les concepts des [Two Scoops of Django][1]. Ils ont publié un [modèle][2] :

repository/
    docs/
    .gitignore
    project/
        apps/
            blog/
                migrations/
                static/ #( optional )
                    blog/
                        some.css
                templates/ #( optional )
                    blog/
                        some.html
                models.py
                tests.py
                admin.py
                apps.py #( django 1.9 and later )
                views.py
            accounts/
                #... ( same as blog )
            search/
                #... ( same as blog )
        conf/
            settings/
                local.py
                development.py
                production.py
            wsgi
            urls.py
        static/
        templates/
    deploy/
        fabfile.py
    requirements/
        base.txt
        local.txt
    README
    AUTHORS
    LICENSE

Ici, les dossiers apps et conf contiennent respectivement les applications créées par l'utilisateur et le dossier de configuration principal pour le projet.

Les dossiers static et templates dans le répertoire project contiennent respectivement des fichiers statiques et des fichiers html markup qui sont utilisés globalement tout au long du projet.

Et tous les dossiers d’application blog, accounts et search peuvent également (principalement) contenir des dossiers static et templates.

[1] : https://www.twoscoopspress.com/ [2] : https://github.com/twoscoops/django-twoscoops-project

Espacement des noms des fichiers statiques et modèles dans les applications Django #

Les dossiers static et templates dans les applications peuvent également contenir un dossier portant le nom de l’application ex. blog il s’agit d’une convention utilisée pour empêcher la pollution de l’espace de noms, nous référençons donc les fichiers comme /blog/base.html plutôt que /base.html qui fournit plus de clarté sur le fichier auquel nous faisons référence et préserve l’espace de noms.

Exemple : le dossier templates dans les applications blog et search contient un fichier nommé base.html, et lorsque vous faites référence au fichier dans views, votre application ne sait plus quel fichier afficher.

(Project Structure)
.../project/
    apps/
        blog/
            templates/
                base.html
        search/
            templates/
                base.html

(blog/views.py)
def some_func(request):
    return render(request, "/base.html")

(search/views.py)
def some_func(request):
    return render(request, "/base.html")

## After creating a folder inside /blog/templates/(blog) ##

(Project Structure)
.../project/
    apps/
        blog/
            templates/
                blog/
                    base.html
        search/
            templates/
                search/
                    base.html

(blog/views.py)
def some_func(request):
    return render(request, "/blog/base.html")

(search/views.py)
def some_func(request):
    return render(request, "/search/base.html")
