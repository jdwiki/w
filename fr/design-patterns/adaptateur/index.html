Adaptateur (Java) #

Supposons que dans votre base de code actuelle, il existe une interface MyLogger comme ceci :

interface MyLogger {
    void logMessage(String message);
    void logException(Throwable exception);
}

Disons que vous en avez créé quelques implémentations concrètes, telles que MyFileLogger et MyConsoleLogger.

Vous avez décidé d’utiliser un framework pour contrôler la connectivité Bluetooth de votre application. Ce framework contient un BluetoothManager avec le constructeur suivant :

class BluetoothManager {
    private FrameworkLogger logger;

    public BluetoothManager(FrameworkLogger logger) {
        this.logger = logger;
    }
}

Le BluetoothManager accepte également un enregistreur, ce qui est génial ! Cependant il attend un logger dont l’interface a été définie par le framework et ils ont utilisé la surcharge de méthode au lieu de nommer leurs fonctions différemment :

interface FrameworkLogger {
    void log(String message);
    void log(Throwable exception);
}

Vous avez déjà un tas d’implémentations de MyLogger que vous aimeriez réutiliser, mais elles ne correspondent pas à l’interface de FrameworkLogger. C’est là qu’intervient le modèle de conception de l’adaptateur :

class FrameworkLoggerAdapter implements FrameworkLogger {
    private MyLogger logger;

    public FrameworkLoggerAdapter(MyLogger logger) {
        this.logger = logger;
    }

    @Override
    public void log(String message) {
        this.logger.logMessage(message);
    }

    @Override
    public void log(Throwable exception) {
        this.logger.logException(exception);
    }
}

En définissant une classe d’adaptateur qui implémente l’interface FrameworkLogger et accepte une implémentation MyLogger, la fonctionnalité peut être mappée entre les différentes interfaces. Il est maintenant possible d’utiliser le BluetoothManager avec toutes les implémentations de MyLogger comme ceci :

FrameworkLogger fileLogger = new FrameworkLoggerAdapter(new MyFileLogger());
BluetoothManager manager = new BluetoothManager(fileLogger);

FrameworkLogger consoleLogger = new FrameworkLoggerAdapter(new MyConsoleLogger());
BluetoothManager manager2 = new BluetoothManager(consoleLogger);

Exemple Java #

Un excellent exemple existant du modèle d’adaptateur peut être trouvé dans le SWT [MouseListener](http://help.eclipse.org/kepler/index.jsp?topic=%2Forg.eclipse.platform.doc.isv%2Freference%2Fapi %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseListener.html) et [MouseAdapter](http://help.eclipse.org/kepler/index.jsp?topic=%2Forg.eclipse.platform.doc.isv%2Freference%2Fapi %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseAdapter.html).

L’interface MouseListener se présente comme suit :

public interface MouseListener extends SWTEventListener {
    public void mouseDoubleClick(MouseEvent e);
    public void mouseDown(MouseEvent e);
    public void mouseUp(MouseEvent e);
}

Imaginez maintenant un scénario dans lequel vous construisez une interface utilisateur et ajoutez ces écouteurs, mais la plupart du temps, vous ne vous souciez de rien d’autre que lorsque quelque chose est cliqué une seule fois (mouseUp). Vous ne voudriez pas constamment créer des implémentations vides :

obj.addMouseListener(new MouseListener() {

    @Override
    public void mouseDoubleClick(MouseEvent e) {
    }

    @Override
    public void mouseDown(MouseEvent e) {
    }

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Au lieu de cela, nous pouvons utiliser MouseAdapter :

public abstract class MouseAdapter implements MouseListener {
    public void mouseDoubleClick(MouseEvent e) { }
    public void mouseDown(MouseEvent e) { }
    public void mouseUp(MouseEvent e) { }
}

En fournissant des implémentations vides par défaut, nous sommes libres de remplacer uniquement les méthodes qui nous intéressent à partir de l’adaptateur. Suite de l’exemple ci-dessus :

obj.addMouseListener(new MouseAdapter() {

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Adaptateur (UML et situation d’exemple) #

Pour rendre l’utilisation du modèle d’adaptateur et le type de situation dans laquelle il peut être appliqué plus imaginable, un petit exemple simple et très concret est donné ici. Il n’y aura pas de code ici, juste UML et une description de la situation d’exemple et de son problème. Certes, le contenu UML est écrit comme Java. (Eh bien, le texte de l’indice disait “Les bons exemples sont principalement du code”, je pense que les modèles de conception sont suffisamment abstraits pour être introduits d’une manière différente également.)

En général, le modèle d’adaptateur est une solution adéquate pour une situation où vous avez des interfaces incompatibles et aucune d’entre elles ne peut être directement réécrite.

Imaginez que vous dirigez un joli petit service de livraison de pizza. Les clients peuvent commander en ligne sur votre site Web et vous disposez d’un petit système utilisant une classe “Pizza” pour représenter vos pizzas et calculer les factures, les déclarations fiscales, etc. Le prix de vos pizzas est donné sous la forme d’un entier unique représentant le prix en centime (de la devise de votre choix).

[![entrez la description de l’image ici][1]][1]

Votre service de livraison fonctionne très bien, mais à un moment donné, vous ne pouvez plus gérer vous-même le nombre croissant de clients, mais vous souhaitez toujours vous développer. Vous décidez d’ajouter vos pizzas au menu d’un gros service de méta-livraison en ligne. Ils offrent beaucoup de repas différents - pas seulement des pizzas - donc leur système utilise davantage l’abstraction et a une interface “IMeal” représentant les repas accompagnés d’une classe “MoneyAmount” représentant l’argent.

[![entrez la description de l’image ici][2]][2]

MoneyAmount se compose de deux entiers en entrée, un pour le montant (ou une devise aléatoire) avant la virgule, et un pour le montant en cents de 0 à 99 après la virgule ;

[![entrez la description de l’image ici][3]][3]

Étant donné que le prix de votre Pizza est un entier unique représentant le prix total en centimes (> 99), il n’est pas compatible avec IMeal. C’est là que le modèle d’adaptateur entre en jeu : au cas où cela demanderait trop d’efforts pour changer votre propre système ou en créer un nouveau et que vous deviez implémenter une interface incompatible, vous pouvez appliquer le modèle d’adaptateur.

Il existe deux manières d’appliquer le modèle : l’adaptateur de classe et l’adaptateur d’objet.

Les deux ont en commun qu’un adaptateur (PizzaAdapter) fonctionne comme une sorte de traducteur entre la nouvelle interface et l’adapté (Pizza dans cet exemple). L’adaptateur implémente la nouvelle interface (IMeal), puis hérite de Pizza et convertit son propre prix d’un entier à deux (adaptateur de classe)

[![entrez la description de l’image ici][4]][4]

ou a un objet de type Pizza comme attribut et convertit les valeurs de celui-ci (adaptateur d’objet).

[![entrez la description de l’image ici][5]][5]

En appliquant le modèle d’adaptateur, vous allez en quelque sorte “traduire” entre des interfaces incompatibles.
