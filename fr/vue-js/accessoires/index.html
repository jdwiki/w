camelCase <=> kebab-case #

Lorsque vous définissez les noms de vos props, rappelez-vous toujours que les noms d’attributs HTML ne sont pas sensibles à la casse. Cela signifie que si vous définissez un prop en cas de chameau dans votre définition de composant …

Vue.component('child', {
    props: ['myProp'],
    ...
});

… vous devez l’appeler dans votre composant HTML en tant que my-prop.

Transmission de données du parent à l’enfant avec des accessoires #

Dans Vue.js, chaque instance de composant a *** sa propre portée isolée ***, ce qui signifie que si un composant parent a un composant enfant, le composant enfant a sa propre portée isolée et le composant parent a sa propre portée isolée.

Pour toute application de taille moyenne à grande, le respect des conventions de bonnes pratiques évite de nombreux maux de tête pendant la phase de développement, puis après la maintenance. L’une de ces choses à suivre est que évitez de référencer/muter les données parentes directement à partir du composant enfant. Alors, comment référencer les données parentes à partir d’un composant enfant ?

Quelles que soient les données parent requises dans un composant enfant, elles doivent être transmises à l’enfant en tant que “props” du parent.

Cas d’utilisation : Supposons que nous ayons une base de données d’utilisateurs avec deux tables “utilisateurs” et “adresses” avec les champs suivants : Tableau utilisateurs

nom téléphone courriel
John Mclane (1) 234 5678 9012 john@dirhard.com
James Bond (44) 777 0007 0077 bond@mi6.com

Tableau adresses

bloquer rue ville
Tours Nakatomi Broadway New York
Maison Mi6 Buckingham Road Londres

et nous voulons avoir trois composants pour afficher les informations utilisateur correspondantes n’importe où dans notre application

composant-utilisateur.js

export default{
    template:`<div class="user-component">
                <label for="name" class="form-control">Name: </label>  
                <input class="form-control input-sm" name="name" v-model="name">
                <contact-details :phone="phone" :email="email"></contact-details>
              </div>`,
    data(){
        return{
            name:'',
            phone:'',
            email:''
        }
    },
}

coordonnées.js

import Address from './address';
export default{
    template:`<div class="contact-details-component>
                <h4>Contact Details:</h4>
                <label for="phone" class="form-control">Phone: </label>  
                <input class="form-control input-sm" name="phone" v-model="phone">
                <label for="email" class="form-control">Email: </label>  
                <input class="form-control input-sm" name="email" v-model="email"> 
        
                <h4>Address:</h4>
                <address :address-type="addressType"></address>
                //see camelCase vs kebab-case explanation below
            </div>`,
    props:['phone', 'email'],
    data:(){
        return:{
            addressType:'Office'
        }
    },
    components:{Address}  
}

adresse.js

export default{
    template:`<div class="address-component">
                <h6>{{addressType}}</h6>
                <label for="block" class="form-control">Block: </label>  
                <input class="form-control input-sm" name="block" v-model="block">
                <label for="street" class="form-control">Street: </label>  
                <input class="form-control input-sm" name="street" v-model="street">
                <label for="city" class="form-control">City: </label>  
                <input class="form-control input-sm" name="city" v-model="city">
             </div>`,
    props:{
        addressType:{
            required:true,
            type:String,
            default:'Office'
        },
    data(){
        return{
            block:'',
            street:'',
            city:''
        }
    }
}

main.js

import Vue from 'vue';  
  
Vue.component('user-component', require'./user-component');  
Vue.component('contact-details', require'./contact-details');  

new Vue({
    el:'body'  
});

index.html

...  
<body>
    <user-component></user-component>
        ...
</body>

Nous affichons les données phone et email, qui sont des propriétés de user-component dans contact-details qui ne contient pas de données de téléphone ou d’e-mail.

Transmettre des données en tant qu’accessoires

Ainsi, dans le user-component.js de la propriété template, où nous incluons le composant <contact-details>, nous transmettons le phone et le ** email** données de <user-component>(parent component) à <contact-details>(child component) en les liant dynamiquement aux props - :phone="phone" et :email="email qui est identique à v-bind:phone="phone" et v-bind:email="email"

Accessoires - Liaison dynamique

Étant donné que nous lions dynamiquement les props, tout changement dans phone ou email dans le composant parent, c’est-à-dire <user-component> sera immédiatement reflété dans le composant enfant, c’est-à-dire <contact- détails>.

Accessoires - en tant que littéraux

Cependant, si nous avions passé les valeurs de phone et email comme valeurs littérales de chaîne comme phone="(44) 777 0007 0077" email="bond@mi6.com" alors cela ne refléterait pas les modifications de données qui se produisent dans le composant parent.

Reliure unidirectionnelle

Par défaut, la direction des modifications est de haut en bas, c’est-à-dire que toute modification des accessoires liés dynamiquement dans le composant parent se propagera au composant enfant, mais toute modification des valeurs d’accessoires dans un composant enfant ne se propagera pas au parent.

Par exemple : si à partir de <contact-details> nous changeons l’e-mail de bond@mi6.com à jamesbond@mi6.com, les données parentes, c’est-à-dire la propriété de données téléphoniques dans <user-component> contiendra toujours une valeur de bond@mi6.com.

Cependant, si nous changeons la valeur de l’e-mail de bond@mi6.com à jamesbond@mi6.co dans le composant parent (<user-component> dans notre cas d’utilisation), alors la valeur de l’e-mail dans l’enfant composant (<contact-details> dans notre cas d’utilisation) deviendra automatiquement jamesbond@mi6.com - le changement de parent est instantanément propagé à l’enfant.

** Reliure bidirectionnelle **

Si nous voulons une liaison bidirectionnelle, nous devons spécifier explicitement la liaison bidirectionnelle comme :email.sync="email" au lieu de :email="email". Maintenant, si nous modifions la valeur de prop dans le composant enfant, la modification sera également reflétée dans le composant parent.

Dans une application de taille moyenne à grande, il sera très difficile de détecter et de suivre le changement de l’état parent à partir de l’état enfant, en particulier lors du débogage - Soyez prudent .

Aucune option .sync ne sera disponible dans Vue.js 2.0. La liaison bidirectionnelle pour les props est obsolète dans Vue.js 2.0.

** Reliure unique **

Il est également possible de définir une liaison unique explicite comme :email.once="email, cela revient plus ou moins à passer un littéral, car toute modification ultérieure de la valeur de la propriété parent ne se propagera pas à l’enfant.

MISE EN GARDE Lorsque Object ou Array est passé comme accessoire, ils sont TOUJOURS PASSÉS PAR RÉFÉRENCE, ce qui signifie quel que soit le type de liaison explicitement défini :email.sync="email" ou :email="email " ou :email.once="email", si email est un objet ou un tableau dans le parent, quel que soit le type de liaison, toute modification de la valeur prop dans le composant enfant affectera la valeur dans le parent comme bien.

Accessoires en tant que tableau

Dans le fichier contact-details.js, nous avons défini props:['phone', 'email'] comme un tableau, ce qui est bien si nous ne voulons pas un contrôle précis avec des props.

Accessoires comme objet

Si nous voulons un contrôle plus précis sur les accessoires, comme

  • si nous voulons définir quel type de valeurs sont acceptables comme prop
  • quelle devrait être une valeur par défaut pour le prop
  • si une valeur doit (obligatoire) être transmise pour le prop ou est-elle facultative

alors nous devons utiliser la notation d’objet pour définir les props, comme nous l’avons fait dans address.js.

Si nous créons des composants réutilisables qui peuvent également être utilisés par d’autres développeurs de l’équipe, il est recommandé de définir les accessoires comme des objets afin que quiconque utilise le composant ait une idée claire de ce que devrait être le type de données et si il est obligatoire ou facultatif.

Elle est également appelée validation des accessoires. Le type peut être l’un des constructeurs natifs suivants :

  • Chaîne de caractères
  • Numéro
  • booléen
  • Déployer
  • Objet
  • Fonction
  • ou un constructeur personnalisé

Quelques exemples de validation d’accessoires tirés de http://vuejs.org/guide/components.html#Props

Vue.component('example', {
   props: {
       // basic type check (`null` means accept any type)
       propA: Number,
       // multiple possible types (1.0.21+)
       propM: [String, Number],
       // a required string
       propB: {
         type: String,
         required: true
       },
       // a number with default value
       propC: {
          type: Number,
          default: 100
       },
       // object/array defaults should be returned from a
       // factory function
       propD: {
          type: Object,
          default: function () {
             return { msg: 'hello' }
         }
       },
       // indicate this prop expects a two-way binding. will
       // raise a warning if binding type does not match.
       propE: {
          twoWay: true
       },
       // custom validator function
       propF: {
          validator: function (value) {
             return value > 10
          }
       },
       // coerce function (new in 1.0.12)
       // cast the value before setting it on the component
       propG: {
          coerce: function (val) {
            return val + '' // cast the value to string
          }
       },
       propH: {
          coerce: function (val) {
            return JSON.parse(val) // cast the value to Object
          }
       }
    }
});

camelCase vs kebab-case

Les attributs HTML sont insensibles à la casse, ce qui signifie qu’ils ne peuvent pas faire la différence entre addresstype et addressType, donc lors de l’utilisation de noms d’accessoires camelCase comme attributs, nous devons utiliser leurs équivalents kebab-case (délimités par des traits d’union): addressType doit être écrit comme address-type dans l’attribut HTML.

Passage d’accessoires lors de l’utilisation de Vue JSX #

Nous avons un composant parent : en important un composant enfant, nous lui transmettrons des props via un attribut. Ici, l’attribut est ‘src’ et nous transmettons également le ‘src’.

ParentComponent.js #

import ChildComponent from './ChildComponent';
export default {
    render(h, {props}) {
        const src = 'https://cdn-images-1.medium.com/max/800/1*AxRXW2j8qmGJixIYg7n6uw.jpeg';
        return (
           <ChildComponent src={src} />   
        );
    }
};

Et un composant enfant, où nous devons passer des accessoires. Nous devons spécifier les accessoires que nous passons.

ChildComponent.js : #

export default {
    props: ['src'],
    render(h, {props}) {
        return ( 
            <a href = {props.src} download = "myimage" >
                Click this link
            </a>
        );
    }
};

Accessoires dynamiques #

Tout comme vous pouvez lier des données d’une vue au modèle, vous pouvez également lier des accessoires en utilisant la même directive v-bind pour transmettre des informations des composants parents aux composants enfants.

JS #

new Vue({
    el: '#example',
    data: {
        msg: 'hello world'
    }
});

Vue.component('child', {
    props: ['myMessage'],
    template: '<span>{{ myMessage }}</span>
});

HTML #

<div id="example">
    <input v-model="msg" />
    <child v-bind:my-message="msg"></child>
    <!-- Shorthand ... <child :my-message="msg"></child> -->
</div>
