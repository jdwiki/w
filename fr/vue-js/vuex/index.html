Vuex est un modèle de gestion d’état + une bibliothèque pour les applications Vue.js. Il sert de magasin centralisé pour tous les composants d’une application, avec des règles garantissant que l’état ne peut être muté que de manière prévisible. Il s’intègre également à l’extension officielle des outils de développement de Vue pour fournir des fonctionnalités avancées telles que le débogage du voyage dans le temps sans configuration et l’exportation/importation d’instantanés d’état.

Qu’est-ce que Vuex ? #

Vuex est un plugin officiel pour Vue.js qui offre un magasin de données centralisé à utiliser dans votre application. Il est fortement influencé par l’architecture de l’application Flux qui présente un flux de données unidirectionnel conduisant à une conception et un raisonnement d’application plus simples.

Au sein d’une application Vuex, le magasin de données contient tous les états d’application partagés. Cet état est modifié par des mutations qui sont effectuées en réponse à une action invoquant un événement de mutation via le répartiteur.

Un exemple de flux de données dans une application Vuex est décrit dans le diagramme ci-dessous. [![Architecture des applications Vuex][1]][1] Schéma utilisé sous la licence MIT, à l’origine du Repo officiel Vuex GitHub.

Les composants individuels de l’application Vue.js peuvent accéder à l’objet de magasin pour récupérer des données via des getters, qui sont de pures fonctions renvoyant une copie en lecture seule des données souhaitées.

Les composants peuvent avoir des actions qui sont des fonctions qui modifient la propre copie des données du composant, puis utilisent le répartiteur pour envoyer un événement de mutation. Cet événement est ensuite géré par le magasin de données qui met à jour l’état si nécessaire.

Les modifications sont ensuite automatiquement répercutées dans l’ensemble de l’application puisque tous les composants sont liés de manière réactive au magasin via leurs getters.

Un [exemple][2] illustrant l’utilisation de vuex dans un projet vue.

const state = {
    lastClickTime: null
}

const mutations = {
    updateLastClickTime: (state, payload) => {
      state.lastClickTime = payload
  }
}

const getters = {
  getLastClickTime: state => {
    return new Date(state.lastClickTime)
  }
}

const actions = {
    syncUpdateTime: ({ commit }, payload) => {
    commit("updateLastClickTime", payload)
  },
  asyncUpdateTime: ({ commit }, payload) => {
    setTimeout(() => {
      commit("updateLastClickTime", payload)
    }, Math.random() * 5000)
  }
}

const store = new Vuex.Store({
  state,
  getters,
  mutations,
  actions
})

const { mapActions, mapGetters } = Vuex;

// Vue 
const vm = new Vue({
    el: '#container',
  store,
  computed: {
      ...mapGetters([
        'getLastClickTime'
    ])
  },
  methods: {
      ...mapActions([
        'syncUpdateTime',
      'asyncUpdateTime'
    ]),
    updateTimeSyncTest () {
        this.syncUpdateTime(Date.now())
    },
    updateTimeAsyncTest () {
        this.asyncUpdateTime(Date.now())
    }
  }
})

Et le modèle HTML pour le même :

<div id="container">
  <p>{{ getLastClickTime || "No time selected yet" }}</p>
  <button @click="updateTimeSyncTest">Sync Action test</button>
  <button @click="updateTimeAsyncTest">Async Action test</button>
</div>

  1. Ici, l’état contient la propriété lastClickTime initialisée comme nulle. Cette configuration des valeurs par défaut est importante pour garder les propriétés réactives. Les propriétés non mentionnées dans l’état seront disponibles mais les modifications apportées par la suite ne seront pas accessibles en utilisant des getters.

  2. Le getter utilisé fournit une propriété calculée qui sera mise à jour chaque fois qu’une mutation met à jour la valeur de la propriété state.

  3. Seules les mutations sont autorisées à modifier l’état et ses propriétés, cela dit, il le fait de manière synchrone uniquement.

  4. Une action peut être utilisée en cas de mises à jour asynchrones, où l’appel d’API (ici simulé par le setTimeout aléatoire) peut être effectué dans l’action, et après avoir obtenu la réponse, une mutation peut être validée, pour apporter le changement à l’état.

[1] : http://i.stack.imgur.com/O9e4f.png [2] : http://jsfiddle.net/codewingx/ybvyspdo/7/

Pourquoi utiliser Vuex ? #

Lors de la création d’applications volumineuses telles que les applications à page unique (SPA), qui consistent généralement en de nombreux composants réutilisables, elles peuvent rapidement devenir difficiles à créer et à entretenir. Le partage de données et d’état entre ces composants peut également rapidement tomber en panne et devenir difficile à déboguer et à maintenir.

En utilisant un magasin de données d’application centralisé, l’état complet de l’application peut être représenté en un seul endroit, ce qui rend l’application plus organisée. Grâce à l’utilisation d’un flux de données unidirectionnel, de mutations et en limitant l’accès aux données des composants aux seules données requises, il devient beaucoup plus simple de raisonner sur le rôle du composant et comment il devrait affecter l’état de l’application.

Les composants VueJS sont des entités distinctes et ils ne peuvent pas facilement partager des données entre eux. Pour partager des données sans vuex, nous devons “émettre” un événement avec des données, puis écouter et attraper cet événement avec “on”.

composant 1

this.$emit('eventWithDataObject', dataObject)

composant 2

this.$on('eventWithDataObject', function (dataObject) {
    console.log(dataObject)
})

Avec vuex installé, nous pouvons simplement accéder à ses données à partir de n’importe quel composant sans avoir besoin d’écouter les événements.

this.$store.state.myData

Nous pouvons également modifier les données de manière synchrone avec des mutators, utiliser des actions asynchrones et obtenir des données avec des fonctions getter.

Les fonctions getter peuvent fonctionner comme des fonctions calculées globales. Nous pouvons y accéder à partir des composants :

this.$store.getters.myGetter

Les actions sont des méthodes globales. Nous pouvons les expédier à partir de composants :

this.$store.dispatch('myAction', myDataObject)

Et les mutations sont le seul moyen de modifier les données dans vuex. Nous pouvons valider les modifications :

this.$store.commit('myMutation', myDataObject)

Le code Vuex ressemblerait à ceci

state: {
  myData: {
    key: 'val'
  }
},
getters: {
  myGetter: state => {
     return state.myData.key.length
  }
},
actions: {
  myAction ({ commit }, myDataObject) {
    setTimeout(() => {
      commit('myMutation', myDataObject)
    }, 2000)
  }
},
mutations: {
  myMutation (state, myDataObject) {
    state.myData = myDataObject
  }
}

Comment installer Vuex ? #

La plupart du temps, vous utiliserez Vuex dans des applications basées sur des composants plus volumineux où vous utiliserez probablement un groupeur de modules tel que Webpack ou Browserify en conjonction avec Vueify si vous utilisez des fichiers uniques.

Dans ce cas, le moyen le plus simple d’obtenir Vuex est de NPM. Exécutez la commande ci-dessous pour installer Vuex et enregistrez-le dans les dépendances de votre application.

 npm install --save vuex

Assurez-vous de charger le lien Vuex avec votre configuration Vue en plaçant la ligne suivante après votre instruction require('vue').

Vue.use(require('vuex'))

Vuex est également disponible sur CDN ; vous pouvez récupérer la dernière version de cdnjs ici.

## Notifications pouvant être supprimées automatiquement

Cet exemple enregistrera dynamiquement un module vuex pour stocker des notifications personnalisées qui peuvent automatiquement être rejetées

notifications.js

résoudre le magasin vuex et définir des constantes

//Vuex store previously configured on other side
import _store from 'path/to/store';

//Notification default duration in milliseconds
const defaultDuration = 8000;

//Valid mutation names
const NOTIFICATION_ADDED = 'NOTIFICATION_ADDED';
const NOTIFICATION_DISMISSED = 'NOTIFICATION_DISMISSED';

définir l’état initial de notre module

const state = {
  Notifications: []
}

définir nos getters de module

const getters = {
  //All notifications, we are returning only the raw notification objects
  Notifications: state => state.Notifications.map(n => n.Raw)
}

définir notre module Actions

const actions = {
  //On actions we receive a context object which exposes the 
  //same set of methods/properties on the store instance
  //{commit} is a shorthand for context.commit, this is an 
  //ES2015 feature called argument destructuring
  Add({ commit }, notification) {
    //Get notification duration or use default duration
    let duration = notification.duration || defaultDuration

    //Create a timeout to dismiss notification
    var timeOut = setTimeout(function () {
      //On timeout mutate state to dismiss notification
      commit(NOTIFICATION_DISMISSED, notification);
    }, duration);

    //Mutate state to add new notification, we create a new object 
    //for save original raw notification object and timeout reference
    commit(NOTIFICATION_ADDED, {
      Raw: notification,
      TimeOut: timeOut
    })
  },
  //Here we are using context object directly
  Dismiss(context, notification) {
    //Just pass payload
    context.commit(NOTIFICATION_DISMISSED, notification);
  }
}

définir nos mutations de module

const mutations = {
  //On mutations we receive current state and a payload
  [NOTIFICATION_ADDED](state, notification) {
    state.Notifications.push(notification);
  },
  //remember, current state and payload
  [NOTIFICATION_DISMISSED](state, rawNotification) {
    var i = state.Notifications.map(n => n.Raw).indexOf(rawNotification);
    if (i == -1) {
      return;
    }

    clearTimeout(state.Notifications[i].TimeOut);
    state.Notifications.splice(i, 1);
  }
}

Enregistrez notre module avec un état, des getters, des actions et une mutation définis

_store.registerModule('notifications', {
  state,
  getters,
  actions,
  mutations
});

Usage

composantA.vue

Ce composant affiche toutes les notifications sous forme d’alertes d’amorçage dans le coin supérieur droit de l’écran, permet également de rejeter manuellement chaque notification.

<template>
<transition-group name="notification-list" tag="div" class="top-right">
  <div v-for="alert in alerts" v-bind:key="alert" class="notification alert alert-dismissible" v-bind:class="'alert-'+alert.type">
    <button v-on:click="dismiss(alert)" type="button" class="close" aria-label="Close"><span aria-hidden="true">&times;</span></button>
    <div>
      <div>
        <strong>{{alert.title}}</strong>
      </div>
      <div>
        {{alert.text}}
      </div>
    </div>
  </div>
</transition-group>
</template>

<script>
export default {
  name: 'arc-notifications',
  computed: {
    alerts() {
      //Get all notifications from store
      return this.$store.getters.Notifications;
    }
  },
  methods: {
    //Manually dismiss a notification
    dismiss(alert) {
      this.$store.dispatch('Dismiss', alert);
    }
  }
}
</script>
