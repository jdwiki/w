Comment résoudre les pannes d'Access

  • Supprimer les champs “Objet OLE”
  • Reconstruire toute la base de données

  • Tester la mémoire de l’ordinateur

  • Supprimer les champs “Objet OLE”

Lorsque vous recevez une erreur : “Microsoft Access a rencontré un problème et doit être fermé”, il n’y a souvent pas beaucoup d’informations pour vous aider à identifier la cause de l’erreur. Vous trouverez ci-dessous une série d’étapes que vous pouvez suivre pour résoudre la cause des erreurs.

Assurez-vous de supprimer les autres variables de l’équation lors du test

Corruption du réseau

Ne chargez pas le client hors d’un réseau. Mettez-le sur le disque local et exécutez-le à partir de là.

Constructions d’entreprise

Si vous êtes dans un environnement d’entreprise qui utilise des “compilations d’ordinateurs” et que vous n’avez pas réussi à décompiler, tester la mémoire ou supprimer les données binaires, refusez de faire d’autres tests jusqu’à ce que l’équipe informatique puisse fournir à l’utilisateur une machine de test qui a seuls Windows, Office et Service Packs sont installés.

Tous les logiciels et mises à jour doivent être installés manuellement sans utiliser d’installations sans surveillance. N’installez pas d’antivirus sur cette machine pour les tests.

Comprenez que de nombreux services informatiques essaient simplement d’adopter une approche unique avec les builds, et que leurs builds sont tous basés les uns sur les autres. Au fil du temps, les conflits logiciels peuvent directement provoquer le blocage d’Access ou son comportement étrange.

Mauvaise alimentation

Comme mentionné dans l’exemple de la mémoire, les fluctuations de puissance peuvent provoquer des erreurs informatiques. Si la base de données se trouve dans un bâtiment industriel, essayez de mettre la main sur un conditionneur d’alimentation ou un onduleur qui fournit une alimentation propre (hors de la batterie, pas hors du secteur passant par une varistance à oxyde métallique)

Vérifiez également le câble d’alimentation qui se branche sur la barre d’alimentation ou la prise. Assurez-vous que les spécifications de jauge et de tension sont suffisantes. Les services informatiques laissent souvent les câbles électriques branchés au poste et retirent simplement la machine. Après de nombreuses années, ils utilisent des alimentations plus puissantes, mais n’ont pas coupé le câble. Cela fait une différence. En cas de doute, apportez un nouveau câble plus épais.

Décompiler la base de données #

Cela devrait toujours être votre solution initiale. Une bonne politique consiste à décompiler la base de données avant chaque version.

  1. Créez un raccourci de décompilation. Cela charge la base de données avec un commutateur “/decompile”.

  2. Cliquez avec le bouton droit sur votre fichier de base de données. Sélectionnez Copier

  3. Faites un clic droit dans la fenêtre de l’explorateur et sélectionnez “Coller le raccourci”

  4. Faites un clic droit sur le raccourci et sélectionnez “Propriétés”

  5. Dans la zone Cible, allez à la fin de la ligne et ajoutez /decompile

  6. Cliquez sur OK pour fermer le raccourci

  7. Ouvrir la base de données avec Shift. Maintenez la touche Maj enfoncée tout en double-cliquant sur ce raccourci. Cela empêche toute exécution automatique de s’exécuter dans la base de données. Vous devriez aller directement à la fenêtre de navigation.

  8. ** Compactez et réparez la base de données. ** Une fois la base de données chargée, vous devrez cliquer sur le bouton Compacter et réparer.

  9. Localisez le bouton Compacter et réparer la base de données sur le ruban Outils.

  10. Maintenez la touche Maj enfoncée. Maintenez-le enfoncé pendant que vous cliquez sur le bouton * Compacter et réparer *.

  11. Recompiler la base de données

  12. Allez dans la fenêtre VBA (Control + G)

  13. Sélectionnez Déboguer -> Compiler dans le menu

C’est le processus de décompilation complet. En règle générale, il devrait résoudre 99 % de tous les plantages d’Access ou des comportements de formulaire étranges.

Tester la mémoire de l’ordinateur #

Si vos plantages sont aléatoires ou sporadiques, suivez cette étape. Si vos plantages se produisent à chaque fois que vous exécutez la base de données, cette étape ne résoudra pas le problème (bien qu’une mauvaise mémoire puisse être la raison pour laquelle la corruption s’est produite en premier lieu).

Utilisez un testeur de mémoire qui démarre en dehors du système d’exploitation et exécute plusieurs passes. Deux choix populaires sont [MemTest86][1] (Commercial) et [MemTest86+][2] (Open Source)

Démarrez le test et laissez-le s’exécuter pendant les heures de travail. La raison en est que d’autres facteurs dans le bâtiment, tels que le bruit sur les circuits d’alimentation, peuvent provoquer des erreurs de mémoire. Vous devez donc essayer de conserver les mêmes variables.

Si vous avez des erreurs de mémoire, vous devrez déterminer si cela est dû à une mauvaise mémoire de l’ordinateur ou à un autre facteur. Cela dépasse cependant le cadre de ce document.

[1] : http://www.memtest86.com/ [2] : http://www.memtest.org/

Supprimer les données binaires du formulaire #

Parfois, les plantages se produisent constamment dans un seul formulaire ou rapport, ou se produisent uniquement lors de l’impression. Il est possible que les données binaires du formulaire/rapport soient corrompues.

Enregistrer l’objet Formulaire/Rapport sous forme de texte Il existe deux fonctions non documentées. Application.SaveAsText et Application.LoadFromText. Vous pouvez utiliser ces fonctions pour exporter les définitions de formulaire/rapport, nettoyer la définition, puis la réimporter.

  1. Faites une sauvegarde de votre base de données avant de continuer
  2. Accédez à la fenêtre immédiate VBA (Contrôle + G)
  3. Tapez Application.SaveAsText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt" (remplacez MyForm par le nom du formulaire/rapport. Utilisez acReport s’il s’agit d’un rapport corrompu que vous corrigez)
  4. Renommez l’élément de formulaire d’origine (par exemple, renommez-le en MyForm.Bak) dans la fenêtre de la base de données

Nettoyer le fichier Form / Report Definitions

  1. Ouvrez le fichier exporté (par exemple MyForm.txt) dans le bloc-notes

  2. Supprimez la ligne “Checksum=” (devrait être sur la ligne 3)

  3. Effacer les données binaires

  4. Identifiez les blocs de données binaires. Parcourez le fichier et vous verrez des lignes commençant par “Paramètre = Début”. Après ces lignes, vous aurez des lignes de données binaires codées. Enfin, le bloc binaire se terminera par une ligne composée uniquement de “End”. Le bloc de données binaires comprend la première ligne (avec l’instruction Begin) et toutes les lignes jusqu’à et y compris la dernière ligne (avec l’instruction End).

    Note: All of these blocks should appear BEFORE your form control definitions

  5. Supprimez les blocs de données binaires pour les paramètres suivants :

    • NameMap
    • PrtMip
    • PrtDevMode
    • PrtDevNames
    • PrtDevModeW
    • PrtDevNamesW

  6. Recherchez d’autres problèmes. Pendant que le fichier est ouvert, faites défiler le reste du fichier et recherchez tout ce qui attire votre attention, en particulier dans le code du module VBA en bas. Vous chercherez tout ce qui se démarque des autres, et peut-être de la corruption.

  7. Enregistrez le fichier.

Chargez le formulaire/le rapport dans Access and Test

  1. Rechargez le formulaire dans Access.
    • In Access, go to the immediate window (Control + G)
    • Type Application.LoadFromText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt"
    • Decompile / Compact Repair / Recompile (See the other example within the documentation)
    • Open the form / report to test. Hopefully everything is working now.
    • Delete the old corrupt form (e.g. MyForm.bak)

** Empêcher cette corruption à l’avenir **

La cause la plus courante de données binaires corrompues dans un rapport/formulaire est lorsque plusieurs ordinateurs/utilisateurs utilisent le même fichier client de base de données au lieu d’avoir leur propre copie séparée. C’est pourquoi chaque utilisateur doit avoir son propre fichier client sur son bureau qu’il exécute.

Supprimer les champs “Objet OLE” #

Si vous avez des images ou d’autres données stockées dans Access lui-même en tant qu’objets OLE, vous devriez trouver une meilleure approche. Lorsque les données OLE sont stockées, elles sont stockées en fonction du logiciel (et de la version du logiciel) sur l’ordinateur qui les stocke. Lorsqu’un autre ordinateur va afficher ces données d’objet OLE sur le formulaire, mais que le logiciel/la version exacte n’est pas installé, cela entraîne souvent un blocage de l’application.

Si vous stockez des données d’image, une meilleure approche consiste à stocker le nom de fichier et à la place à enregistrer les images dans un emplacement standard. Les nouvelles versions d’accès ont les contrôles natifs pour en faire la voie à suivre.

Reconstruire toute la base de données #

C’est beaucoup de travail, alors faites-le en dernier recours après avoir épuisé toutes les autres options. Vous ne devez le faire que si le problème se produit pour différents utilisateurs, sur différentes machines. Si cela ne se produit pas pour tous les utilisateurs, il ne s’agit probablement pas d’un conteneur de base de données corrompu.

Semblable aux étapes de suppression des données binaires, vous allez reconstruire votre base de données à partir de zéro. Ce processus est un peu rituel, mais s’il est fait méticuleusement avec soin pour ne pas “préserver” toute corruption possible, alors le processus est très efficace.

Créer un nouveau conteneur de base de données d’accès.

  • Dans Access, sur l’onglet Fichier, vous pouvez sélectionner “Nouveau”. Créez une nouvelle base de données vide au format ACCDB.

Déplacer tous les objets vers le nouveau conteneur

N’utilisez pas les fonctions d’importation/exportation d’Access pour déplacer les objets, et ne vous contentez pas de cliquer et de faire glisser. Cela peut copier les éléments corrompus dans le nouveau conteneur.

Les tables:

  • Pour chaque table de l’ancien conteneur d’accès, créez une nouvelle table dans le nouveau conteneur.
  • En mode conception, copiez/collez les définitions de champs.
  • Vérifiez les propriétés de la table pour vous assurer qu’elles correspondent dans les deux bases de données
  • Déplacez également toutes les macros de données (voir la section Macros pour savoir comment procéder)
  • Pour déplacer les données, exportez les anciennes données vers XML ou CSV, puis importez à partir de ce format.

Requêtes :

  • Chargez chaque requête dans la vue SQL.
  • Copier/Coller le texte SQL.
  • Coller dans la nouvelle base de données.
  • Comparez les propriétés de la requête pour vous assurer qu’elles correspondent.

Formulaires / Rapports :

  • Pour chaque Formulaire/Rapport, utilisez la fonction Application.SaveAsText pour exporter les formulaires/rapports vers un fichier texte.
  • Supprimer les données binaires (voir la documentation * Supprimer les données binaires du formulaire * pour vous familiariser avec ce processus)
  • Utilisez la fonction Application.LoadFromText pour réimporter les objets dans la nouvelle base de données

Macro

Vous disposez de trois méthodes pour déplacer les macros.

  1. Recréez chaque macro manuellement dans le nouveau conteneur de base de données.
  2. Utilisez la méthode Application.SaveAsText / Application.LoadFromText avec le paramètre acMacro.
  3. Copier/Coller les définitions de macro pour chaque macro
  • Sélectionnez Tout (Contrôle + A) pour sélectionner tous les éléments macro. Puis Copiez (Contrôle + C).
  • Ouvrez un document Bloc-notes vierge et collez (Ctrl + V) la macro XML.
  • Créez une nouvelle macro vierge dans le nouveau conteneur de base de données.
  • Dans le Bloc-notes, sélectionnez Tout le texte (Contrôle + A). Puis Copier (Contrôle + C)
  • Dans la macro vide, Coller (Contrôle + V). La macro devrait apparaître. Sauvegarde le.

Modules

  • Pour chaque module, sélectionnez tout le code (Control + A) et collez (Control + V) dans le nouveau conteneur de base de données.
  • Assurez-vous de vérifier les propriétés de la base de données (dans la fenêtre VBA, accédez à Outils -> Propriétés du client)

Macros de données

Pour chaque Data Macro, utilisez les méthodes SaveAsText / LoadFromText.

  1. Allez dans la fenêtre immédiate VBA (Control + G)
  2. Tapez Application.SaveAsText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt" (remplacez MyTableName par le nom de la table contenant les macros de données)
  3. Examinez le fichier pour tout signe de corruption
  4. Dans le nouveau conteneur de base de données, chargez la définition à l’aide de Application.LoadFromText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt"

Comme mentionné précédemment, c’est ** BEAUCOUP ** de travail, mais cela a des résultats. Cette méthode doit également être utilisée lors de la migration d’une base de données Access 97 vers 2000 ou d’une base de données Access 2000 vers 2003.
