Il est facile de voir les “types” comme une table dans une base de données SQL, où l’“index” est la base de données SQL. Cependant, ce n’est pas une bonne façon d’aborder les “types”.

Tout sur les types #

En fait, les types ne sont littéralement qu’un champ de métadonnées ajouté à chaque document par Elasticsearch : _type. Les exemples ci-dessus ont créé deux types : my_type et my_other_type. Cela signifie que chaque document associé aux types a un champ supplémentaire défini automatiquement comme "_type": "my_type"; celui-ci est indexé avec le document, ce qui en fait un champ interrogeable ou filtrable, mais cela n’affecte pas le document brut lui-même, de sorte que votre application n’a pas à s’en soucier.

Tous les types vivent dans le même index, et donc dans les mêmes fragments collectifs de l’index. Même au niveau du disque, ils vivent dans les mêmes fichiers. La seule séparation que la création d’un deuxième type fournit est une séparation logique. Chaque type, qu’il soit unique ou non, doit exister dans les mappages et tous ces mappages doivent exister dans l’état de votre cluster. Cela consomme de la mémoire et, si chaque type est mis à jour dynamiquement, cela consomme des performances à mesure que les mappages changent.

En tant que tel, il est recommandé de ne définir qu’un seul type, sauf si vous avez réellement besoin d’autres types. Il est courant de voir des scénarios où plusieurs types sont souhaitables. Par exemple, imaginez que vous disposiez d’un index des voitures. Il peut vous être utile de le décomposer en plusieurs types :

  • BMW
  • chevy -honda
  • Mazda -mercedes -Nissan
  • rangerover
  • Toyota
  • …

De cette façon, vous pouvez rechercher toutes les voitures ou les limiter par fabricant à la demande. La différence entre ces deux recherches est aussi simple que :

GET /cars/_search

et

GET /cars/bmw/_search

Ce qui n’est pas évident pour les nouveaux utilisateurs d’Elasticsearch, c’est que la deuxième forme est une spécialisation de la première forme. Il est littéralement réécrit en :

GET /cars/_search
{
  "query": {
    "bool": {
      "filter": [
        {
          "term" : {
            "_type": "bmw"
          }
        }
      ]
    }
  }
}

Il filtre simplement tout document qui n’a pas été indexé avec un champ _type dont la valeur était bmw. Étant donné que chaque document est indexé avec son type comme champ _type, cela sert de filtre assez simple. Si une recherche réelle avait été fournie dans l’un ou l’autre des exemples, le filtre serait ajouté à la recherche complète, le cas échéant.

Ainsi, si les types sont identiques, il est préférable de fournir un seul type (par exemple, “fabricant” dans cet exemple) et de l’ignorer. Ensuite, dans chaque document, fournissez explicitement un champ appelé make ou le nom que vous préférez et filtrez-le manuellement chaque fois que vous souhaitez vous limiter à celui-ci. Cela réduira la taille de vos mappages à 1/n où n est le nombre de types distincts. Il ajoute un autre champ à chaque document, au profit d’un mappage par ailleurs simplifié.

Dans Elasticsearch 1.x et 2.x, un tel champ doit être défini comme

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "string",
        "index": "not_analyzed"
      }
    }
  }
}
  1. Le nom est arbitraire.
  2. Le nom est arbitraire et il pourrait correspondre au nom du type si vous le vouliez aussi.

Dans Elasticsearch 5.x, ce qui précède fonctionnera toujours (il est obsolète), mais la meilleure façon est d’utiliser :

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "keyword"
      }
    }
  }
}
  1. Le nom est arbitraire.
  2. Le nom est arbitraire et il pourrait correspondre au nom du type si vous le vouliez aussi.

Les types doivent être utilisés avec parcimonie dans vos index car ils gonflent les mappages d’index, généralement sans grand avantage. Vous devez en avoir au moins un, mais rien n’indique que vous devez en avoir plus d’un.

Questions courantes #

  • Que se passe-t-il si j’ai deux types (ou plus) qui sont pour la plupart identiques, mais qui ont quelques champs uniques par type ?

Au niveau de l’index, il n’y a pas de différence entre un type utilisé avec quelques champs peu utilisés et entre plusieurs types qui partagent un tas de champs non clairsemés avec quelques-uns non partagés (ce qui signifie que l’autre type n’utilise même jamais le champ (s)).

En d’autres termes : un champ peu utilisé est dispersé dans l’index indépendamment des types. La rareté ne profite pas - ou ne nuit pas vraiment - à l’index simplement parce qu’il est défini dans un type séparé.

Vous devez simplement combiner ces types et ajouter un champ de type séparé.

- Pourquoi des types distincts doivent-ils définir des champs exactement de la même manière ?

Parce que chaque champ n’est vraiment défini qu’une seule fois au niveau Lucene, quel que soit le nombre de types. Le fait que les types existent est une fonctionnalité d’Elasticsearch et il s’agit seulement d’une séparation logique.

  • Puis-je définir des types distincts avec le même champ défini différemment ?

Non. Si vous parvenez à trouver un moyen de le faire dans ES 2.x ou version ultérieure, alors vous devriez ouvrir un rapport de bogue. Comme indiqué dans la question précédente, Lucene les considère tous comme un seul champ, il n’y a donc aucun moyen de faire en sorte que cela fonctionne correctement.

ES 1.x a laissé cela comme une exigence implicite, ce qui permettait aux utilisateurs de créer des conditions dans lesquelles les mappages d’un fragment dans un index différaient réellement d’un autre fragment dans le même index. Il s’agissait effectivement d’une condition de concurrence et cela pourrait conduire à des problèmes inattendus.

Exceptions à la règle #

  • Les documents parent/enfant nécessitent l’utilisation de types distincts dans le même index.
    • The parent lives in one type.
    • The child lives in a separate type (but each child lives in the same shard as its parent).
  • Cas d’utilisation extrêmement niches où la création de tonnes d’index n’est pas souhaitable et l’impact de champs clairsemés est préférable à l’alternative.
    • For example, the Elasticsearch monitoring plugin, Marvel (1.x and 2.x) or X-Pack Monitoring (5.x+), monitors Elasticsearch itself for changes in the cluster, nodes, indices, specific indices (the index level), and even shards. It could create 5+ indices each day to isolate those documents that have unique mappings or it could go against best practices to reduce cluster load by sharing an index (note: the number of defined mappings is effectively the same, but the number of created indices is reduced from n to 1).
    • This is an advanced scenario, but you must consider the shared field definitions across types!

Création explicite d’un index avec un type #

L’exemple utilise HTTP de base, qui se traduit facilement en cURL et d’autres applications HTTP. Ils correspondent également à la syntaxe Sense, qui sera renommée Console dans Kibana 5.0.

Remarque : L’exemple insère <#> pour aider à attirer l’attention sur les pièces. Ceux-ci devraient être supprimés si vous le copiez !

PUT /my_index <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "integer"
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "float"
            }
          }
        }
      }
    }
  },
  "my_other_type": {
    "properties": {
      "field1": {
        "type": "long" <3>
      },
      "field3": { <4>
        "type": "double"
      }
    }
  }
}
  1. Il s’agit de créer l’index à l’aide du point de terminaison de création d’index.
  2. Ceci crée le type.
  3. Les champs partagés dans les “types” d’un même “index” doivent partager la même définition ! ES 1.x n’appliquait pas strictement ce comportement, mais il s’agissait d’une exigence implicite. ES 2.x et supérieur appliquent strictement ce comportement.
  4. Les champs uniques à travers les “types” sont corrects.

Les index (ou index) contiennent des types. Les types sont un mécanisme pratique pour séparer les documents, mais ils nécessitent que vous définissiez – dynamiquement/automatiquement ou explicitement – un mappage pour chaque type que vous utilisez. Si vous définissez 15 types dans un index, vous disposez de 15 mappages uniques.

Voir les remarques pour plus de détails sur ce concept et pourquoi vous pouvez ou non utiliser des types.

Création dynamique d’un index avec un type #

L’exemple utilise HTTP de base, qui se traduit facilement en cURL et d’autres applications HTTP. Ils correspondent également à la syntaxe Sense, qui sera renommée Console dans Kibana 5.0.

Remarque : L’exemple insère <#> pour aider à attirer l’attention sur les pièces. Ceux-ci devraient être supprimés si vous le copiez !

DELETE /my_index <1>

PUT /my_index/my_type/abc123 <2>
{
  "field1" : 1234, <3>
  "field2" : 456,
  "object1" : {
    "field1" : 7.8 <4>
  }
}
  1. S’il existe déjà (à cause d’un exemple précédent), supprimez l’index.
  2. Indexez un document dans l’index, my_index, avec le type, my_type, et l’ID abc123 (peut être numérique, mais c’est toujours une chaîne).
    • By default, dynamic index creation is enabled by simply indexing a document. This is great for development environments, but it is not necessarily good for production environments.
  3. Ce champ est un nombre entier, donc la première fois qu’il est vu, il doit être mappé. Elasticsearch suppose toujours le type widest pour tout type entrant, donc cela serait mappé comme un long plutôt qu’un integer ou un short (qui pourraient tous deux contenir 1234 et 456).
  4. Il en va de même pour ce champ également. Il sera mappé comme un “double” au lieu d’un “flottant” comme vous le souhaitez.

Cet index et ce type créés dynamiquement correspondent approximativement au mappage défini dans le premier exemple. Cependant, il est essentiel de comprendre comment <3> et <4> impactent les mappages définis automatiquement.

Vous pouvez suivre cela en ajoutant encore un autre type dynamiquement au même index :

PUT /my_index/my_other_type/abc123 <1>
{
  "field1": 91, <2>
  "field3": 4.567
}
  1. Le type est la seule différence par rapport au document ci-dessus. L’ID est le même et c’est normal ! Il n’a aucune relation avec l’autre “abc123” autre que le fait qu’il se trouve par hasard dans le même index.
  2. field1 existe déjà dans l’index, donc il doit être le même type de champ que celui défini dans les autres types. La soumission d’une valeur qui était une chaîne ou non un entier échouerait (par exemple, "field1": "this is some text" ou "field1": 123.0).

Cela créerait dynamiquement les mappages pour my_other_type dans le même index, my_index.

Remarque : il est toujours plus rapide de définir les mappages à l’avance plutôt que de laisser Elasticsearch les exécuter dynamiquement au moment de l’indexation.

Le résultat final de l’indexation des deux documents serait similaire au premier exemple, mais les types de champs seraient différents et donc légèrement inutiles :

GET /my_index/_mappings <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "long" <3>
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "double" <4>
            }
          }
        }
      }
    }
  },
  "my_other_type": { <5>
    "properties": {
      "field1": {
        "type": "long"
      },
      "field3": {
        "type": "double"
      }
    }
  }
}
  1. Cela utilise le point de terminaison _mappings pour obtenir les mappages à partir de l’index que nous avons créé.
  2. Nous avons créé dynamiquement my_type dans la première étape de cet exemple.
  3. field2 est maintenant un long au lieu d’un integer car nous ne l’avons pas défini à l’avance. Cela peut s’avérer être un gaspillage de stockage sur disque.
  4. object1.field1 est maintenant un double pour la même raison que #3 avec les mêmes ramifications que #3.
    • Technically, a long can be compressed in a lot of cases. However, a double cannot be compressed due to it being a floating point number.
  5. Nous avons également créé dynamiquement my_other_type dans la deuxième étape de cet exemple. Son mappage est le même car nous utilisions déjà long et double.
    • Remember that field1 must match the definition from my_type (and it does).
    • field3 is unique to this type, so it has no such restriction.
