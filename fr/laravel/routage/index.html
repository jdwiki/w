Routage de base #

Le routage définit une carte entre les méthodes HTTP et les URI d’un côté, et les actions de l’autre. Les routes sont normalement écrites dans le fichier app/Http/routes.php.

Dans sa forme la plus simple, une route est définie en appelant la méthode HTTP correspondante sur la façade Route, en passant comme paramètres une chaîne qui correspond à l’URI (relatif à la racine de l’application) et un rappel.

Par exemple : une route vers l’URI racine du site qui renvoie une vue “home” ressemble à ceci :

Route::get('/', function() {
    return view('home');
});

Une route pour une requête post qui fait simplement écho aux variables post :

Route::post('submit', function() {
   return Input::all();
});

//or

Route::post('submit', function(\Illuminate\Http\Request $request) {
   return $request->all();
});

Routes pointant vers une méthode Controller #

Au lieu de définir le rappel en ligne, la route peut faire référence à une méthode de contrôleur dans la syntaxe [ControllerClassName@Method] :

Route::get('login', 'LoginController@index');

Une route pour plusieurs verbes #

La méthode match peut être utilisée pour faire correspondre un tableau de méthodes HTTP pour une route donnée :

Route::match(['GET', 'POST'], '/', 'LoginController@index');

Vous pouvez également utiliser all pour faire correspondre n’importe quelle méthode HTTP pour une route donnée :

Route::all('login', 'LoginController@index');

Groupes de routage #

Les itinéraires peuvent être regroupés pour éviter la répétition du code.

Disons que tous les URI avec un préfixe /admin utilisent un certain middleware appelé admin et ils vivent tous dans l’espace de noms App\Http\Controllers\Admin.

Une manière propre de représenter cela à l’aide des groupes de routage est la suivante :

Route::group([
    'namespace' => 'Admin',
    'middleware' => 'admin',
    'prefix' => 'admin'
    ], function () {
        
        // something.dev/admin
        // 'App\Http\Controllers\Admin\IndexController'
        // Uses admin middleware
        Route::get('/', ['uses' => 'IndexController@index']);    
        
        // something.dev/admin/logs
        // 'App\Http\Controllers\Admin\LogsController'
        // Uses admin middleware
        Route::get('/logs', ['uses' => 'LogsController@index']);    
});

Les routes sont mises en correspondance dans l’ordre dans lequel elles sont déclarées #

C’est un piège commun avec les itinéraires Laravel. Les routes sont mises en correspondance dans l’ordre dans lequel elles sont déclarées. La première route correspondante est celle qui est utilisée.

Cet exemple fonctionnera comme prévu :

Route::get('/posts/{postId}/comments/{commentId}', 'CommentController@show');
Route::get('/posts/{postId}', 'PostController@show');

Une requête get à /posts/1/comments/1 invoquera CommentController@show. Une requête get à /posts/1 invoquera PostController@show.

Cependant, cet exemple ne fonctionnera pas de la même manière :

Route::get('/posts/{postId}', 'PostController@show');
Route::get('/posts/{postId}/comments/{commentId}', 'CommentController@show');

Une requête get à /posts/1/comments/1 invoquera PostController@show. Une requête get à /posts/1 invoquera PostController@show.

Étant donné que Laravel utilise la première route correspondante, la requête à /posts/1/comments/1 correspond à Route::get('/posts/{postId}', 'PostController@show'); et attribue la variable $postId à la valeur 1/commentaires/1. Cela signifie que CommentController@show ne sera jamais appelé.

Route nommée #

Les routes nommées sont utilisées pour générer une URL ou des redirections vers une route spécifique. L’avantage d’utiliser une route nommée est que si nous modifions l’URI d’une route à l’avenir, nous n’aurions pas besoin de changer l’URL ou les redirections pointant vers cette route si nous utilisons une route nommée. Mais si les liens ont été générés à l’aide de l’url [ eg. url('/admin/login')], alors nous devrions changer partout où il est utilisé.

Les routes nommées sont créées à l’aide de la clé de tableau as

Route::get('login', ['as' => 'loginPage', 'uses' => 'LoginController@index']);

ou en utilisant la méthode nom

Route::get('login', 'LoginController@index')->name('loginPage');

Générer une URL en utilisant une route nommée #

Pour générer une url en utilisant le nom de la route

$url = route('loginPage');

Si vous utilisez le nom de la route pour la redirection

$redirect = Redirect::route('loginPage');

Paramètres d’itinéraire #

Vous pouvez utiliser des paramètres de route pour obtenir la partie du segment URI. Vous pouvez définir un ou plusieurs paramètres de route facultatifs ou obligatoires lors de la création d’une route. Les paramètres optionnels ont un ? ajouté à la fin du nom du paramètre. Ce nom est entouré d’accolades {}

Paramètre facultatif #

Route::get('profile/{id?}', ['as' => 'viewProfile', 'uses' => 'ProfileController@view']);

Cette route est accessible par domain.com/profile/23 où 23 est le paramètre id. Dans cet exemple, id est passé en tant que paramètre dans la méthode view de ProfileController. Comme il s’agit d’un paramètre facultatif, l’accès à domain.com/profile fonctionne très bien.

Paramètre requis #

Route::get('profile/{id}', ['as' => 'viewProfile', 'uses' => 'ProfileController@view']);

Notez que le nom du paramètre requis n’a pas de ? à la fin du nom du paramètre.

Accéder au paramètre dans le contrôleur #

Dans votre contrôleur, votre méthode de vue prend un paramètre avec le même nom que celui de routes.php et peut être utilisé comme une variable normale. Laravel se charge d’injecter la valeur :

public function view($id){
    echo $id;
}

Attrapez tous les itinéraires #

Si vous souhaitez capturer toutes les routes, vous pouvez utiliser une expression régulière comme indiqué :

Route::any('{catchall}', 'CatchAllController@handle')->where('catchall', '.*');

Important : Si vous avez d’autres itinéraires et que vous ne voulez pas que le fourre-tout interfère, vous devez le mettre à la fin. Par exemple:

Attraper toutes les routes sauf celles déjà définies #

Route::get('login',  'AuthController@login');
Route::get('logout', 'AuthController@logout');
Route::get('home',   'HomeController@home');

// The catch-all will match anything except the previous defined routes.
Route::any('{catchall}', 'CatchAllController@handle')->where('catchall', '.*');

Routes insensibles à la casse #

Les itinéraires dans Laravel sont sensibles à la casse. Cela signifie qu’un itinéraire comme

Route::get('login', ...);

correspondra à une requête GET à /login mais ne correspondra pas à une requête GET à /Login.

Afin de rendre vos routes insensibles à la casse, vous devez créer une nouvelle classe de validateur qui fera correspondre les URL demandées aux routes définies. La seule différence entre le nouveau validateur et celui existant est qu’il ajoutera le modificateur i à la fin de l’expression régulière pour la route compilée pour activer la correspondance insensible à la casse.

<?php namespace Some\Namespace;

use Illuminate\Http\Request;
use Illuminate\Routing\Route;
use Illuminate\Routing\Matching\ValidatorInterface;

class CaseInsensitiveUriValidator implements ValidatorInterface
{
  public function matches(Route $route, Request $request)
  {
    $path = $request->path() == '/' ? '/' : '/'.$request->path();
    return preg_match(preg_replace('/$/','i', $route->getCompiled()->getRegex()), rawurldecode($path));
  }
}

Pour que Laravel utilise votre nouveau validateur, vous devez mettre à jour la liste des matchers utilisés pour faire correspondre l’URL à une route et remplacer l’UriValidator d’origine par le vôtre.

Pour ce faire, ajoutez ce qui suit en haut de votre fichier routes.php :

<?php
use Illuminate\Routing\Route as IlluminateRoute;
use Your\Namespace\CaseInsensitiveUriValidator;
use Illuminate\Routing\Matching\UriValidator;

$validators = IlluminateRoute::getValidators();
$validators[] = new CaseInsensitiveUriValidator;
IlluminateRoute::$validators = array_filter($validators, function($validator) { 
  return get_class($validator) != UriValidator::class;
});

Cela supprimera le validateur d’origine et ajoutera le vôtre à la liste des validateurs.
