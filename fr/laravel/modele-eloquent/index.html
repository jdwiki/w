Faire un modèle #

### Création du modèle Les classes de modèle doivent étendre Illuminate\Database\Eloquent\Model. L’emplacement par défaut des modèles est le répertoire /app.

Une classe de modèle peut être facilement générée par la commande [Artisan][1] :

php artisan make:model [ModelName]

Cela créera un nouveau fichier PHP dans app/ par défaut, nommé [ModelName].php, et contiendra tout le passe-partout pour votre nouveau modèle, qui inclut la classe, l’espace de noms et l’utilisation requise pour une base mettre en place.

Si vous souhaitez créer un fichier de migration avec votre modèle, utilisez la commande suivante, où -m générera également le fichier de migration :

php artisan make:model [ModelName] -m

En plus de créer le modèle, cela crée une migration de base de données qui est connectée au modèle. Le fichier PHP de migration de la base de données se trouve par défaut dans database/migrations/. Cela n’inclut pas - par défaut - autre chose que les colonnes id et created_at/updated_at, vous devrez donc éditer le fichier pour fournir des colonnes supplémentaires.

Notez que vous devrez exécuter la migration (une fois que vous avez configuré le fichier de migration) pour que le modèle commence à fonctionner en utilisant php artisan migrate à partir de la racine du projet

De plus, si vous souhaitez ajouter une migration ultérieurement, après avoir créé le modèle, vous pouvez le faire en exécutant :

php artisan make:migration [migration name]

Disons par exemple que vous vouliez créer un modèle pour vos Cats, vous auriez deux choix, créer avec ou sans migration. Vous auriez choisi de créer sans migration si vous aviez déjà une table de chats ou si vous ne vouliez pas en créer une pour le moment.

Pour cet exemple, nous voulons créer une migration car nous n’avons pas encore de table, nous exécuterons donc la commande suivante.

php artisan make:model Cat -m

Cette commande créera deux fichiers :

  1. Dans le dossier App : app/Cat.php
  2. Dans le dossier de la base de données : database/migrations/timestamp_creat_cats_table.php

Le fichier qui nous intéresse est ce dernier car c’est ce fichier que nous pouvons décider à quoi nous voulons que le tableau ressemble et inclue. Pour toute migration prédéfinie, nous recevons une colonne d’identification à incrémentation automatique et une colonne d’horodatage.

L’exemple ci-dessous d’un extrait du fichier de migration comprend les colonnes prédéfinies ci-dessus ainsi que l’ajout du nom du chat, de l’âge et de la couleur :

public function up()
    {
        Schema::create('cats', function (Blueprint $table) {

            $table->increments('id');    //Predefined ID
            $table->string('name');      //Name
            $table->integer('age');      //Age
            $table->string('colour');    //Colour
            $table->timestamps();        //Predefined Timestamps

        });
    }

Ainsi, comme vous pouvez le voir, il est relativement facile de créer le modèle et la migration d’une table. Ensuite, pour exécuter la migration et la créer dans votre base de données, vous exécuterez la commande suivante :

php artisan migrate

Ce qui migrera toutes les migrations en attente vers votre base de données.

Emplacement du fichier de modèle #

Les modèles peuvent être stockés n’importe où grâce à PSR4.

Par défaut, les modèles sont créés dans le répertoire app avec l’espace de noms App. Pour les applications plus complexes, il est généralement recommandé de stocker les modèles dans leurs propres dossiers dans une structure cohérente avec l’architecture de vos applications.

Par exemple, si vous aviez une application qui utilisait une série de fruits comme modèles, vous pourriez créer un dossier appelé app/Fruits et dans ce dossier, vous créez Banana.php (en gardant le [StudlyCase](https:// en.wikipedia.org/wiki/Studly_caps) convention de nommage), vous pouvez ensuite créer la classe Banana dans l’espace de noms App\Fruits :

namespace App\Fruits;

use Illuminate\Database\Eloquent\Model;

class Banana extends Model {
    // Implementation of "Banana" omitted
}

Configuration du modèle #

Eloquent suit une approche “convention plutôt que configuration”. En étendant la classe de base “Model”, tous les modèles héritent des propriétés répertoriées ci-dessous. Sauf en cas de remplacement, les valeurs par défaut suivantes s’appliquent :

Propriété Descriptif Par défaut
$connexion protégée Nom de connexion à la base de données Connexion à la base de données par défaut
$table protégée Nom du tableau Par défaut, le nom de la classe est converti en snake_case et mis au pluriel. Par exemple, SpecialPerson devient special_people
protégé $primaryKey Tableau PK identifiant
public $incrémentation Indique si les ID s’incrémentent automatiquement vrai
public $horodatages Indique si le modèle doit être horodaté vrai
const CREATED_AT Nom de la colonne d’horodatage de création created_at
const UPDATED_AT Nom de la colonne d’horodatage de modification mise à jour_at
$dates protégées Attributs qui doivent être mutés en DateTime, en plus des attributs d’horodatage []
protected $dateFormat Format dans lequel les attributs de date seront conservés Par défaut pour le dialecte SQL actuel.
protégé $avec Relations avec eagerload avec le modèle []
protégé $caché Attributs omis dans la sérialisation du modèle []
protégé $visible Attributs autorisés dans la sérialisation de modèle []
$ajoute protégé Accesseurs d’attribut ajoutés à la sérialisation du modèle []
protégé $remplissable Attributs assignables en masse []
protégé $gardé Attributs mis sur liste noire à partir de l’attribution en masse [*] (Tous les attributs)
$touches protégées Les relations qui devraient être abordées enregistrer []
protégé $parpage Le nombre de modèles à renvoyer pour la pagination. 15
Propriété Descriptif Par défaut
$casts protégés Attributs qui doivent être convertis en types natifs []

[1] : https://www.wikiod.com/fr/laravel/artisan

Mettre à jour un modèle existant #

$user = User::find(1);
$user->name = 'abc';
$user->save();

Vous pouvez également mettre à jour plusieurs attributs à la fois en utilisant update, ce qui ne nécessite pas d’utiliser save par la suite :

$user = User::find(1);
$user->update(['name' => 'abc', 'location' => 'xyz']);

Vous pouvez également mettre à jour un ou plusieurs modèles sans les interroger au préalable :

User::where('id', '>', 2)->update(['location' => 'xyz']);

Si vous ne voulez pas déclencher une modification de l’horodatage updated_at sur le modèle, vous pouvez passer l’option touch :

$user = User::find(1);
$user->update(['name' => 'abc', 'location' => 'xyz'], ['touch' => false]);
