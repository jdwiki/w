Insertion de modèles associés #

Supposons que vous ayez un modèle Post avec une relation hasMany avec Comment. Vous pouvez insérer un objet “Commentaire” lié à un article en procédant comme suit :

$post = Post::find(1);

$commentToAdd = new Comment(['message' => 'This is a comment.']);

$post->comments()->save($commentToAdd);

Vous pouvez enregistrer plusieurs modèles à la fois à l’aide de la fonction “saveMany” :

$post = Post::find(1);

$post->comments()->saveMany([
    new Comment(['message' => 'This a new comment']),
    new Comment(['message' => 'Me too!']),
    new Comment(['message' => 'Eloquent is awesome!'])
]);

Alternativement, il existe également une méthode create qui accepte un tableau PHP simple au lieu d’une instance de modèle Eloquent.

$post = Post::find(1);

$post->comments()->create([
    'message' => 'This is a new comment message'
]);

Types de relations #

Un à plusieurs #

Disons que chaque publication peut avoir un ou plusieurs commentaires et que chaque commentaire appartient à une seule publication.

donc la table des commentaires aura post_id. Dans ce cas, les relations seront les suivantes.

Modèle de poste

public function comments()
{
   return $this->belongsTo(Post::class);
}

Si la clé étrangère est autre que post_id, par exemple la clé étrangère est example_post_id.

public function comments()
{
   return $this->belongsTo(Post::class, 'example_post_id');
}

et plus, si la clé locale est autre que id, par exemple la clé locale est other_id

public function comments()
{
   return $this->belongsTo(Post::class, 'example_post_id', 'other_id');
}

Modèle de commentaire

définition inverse de un à plusieurs

public function post()
{
   return $this->hasMany(Comment::class);
}

Un par un #

Comment associer deux modèles (exemple : modèle User et Phone) #

Application\Utilisateur

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class User extends Model
{
    /**
     * Get the phone record associated with the user.
     */
    public function phone()
    {
        return $this->hasOne('Phone::class', 'foreign_key', 'local_key');
    }
}

Application\Téléphone

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Phone extends Model
{
    /**
     * Get the user that owns the phone.
     */
    public function user()
    {
        return $this->belongsTo('User::class', 'foreign_key', 'local_key');
    }
}

foreign_key : par défaut, Eloquent supposera que cette valeur est other_model_name_id (dans ce cas user_id et phone_id), modifiez-la si ce n’est pas le cas.

local_key : par défaut, Eloquent supposera que cette valeur est id (clé primaire du modèle actuel), modifiez-la si ce n’est pas le cas.

Si votre base de données a déposé le nom selon la norme laravel, vous n’avez pas besoin de fournir la clé étrangère et la clé locale dans la déclaration de relation

Explication #

Plusieurs à plusieurs #

Disons qu’il y a des rôles et des autorisations. Chaque rôle peut appartenir à plusieurs autorisations et chaque autorisation peut appartenir à plusieurs rôles. il y aura donc 3 tables. deux modèles et un tableau croisé dynamique. une table roles, users et permission_role.

Modèle

public function permissions()
{
   return $this->belongsToMany(Permission::class);
}

Modèle d’autorisation

public function roles()
{
   return $this->belongsToMany(Roles::class);
}

Note 1

considérez ce qui suit lorsque vous utilisez un nom de table différent pour le tableau croisé dynamique.

Supposons que vous souhaitiez utiliser role_permission au lieu de permission_role, car eloquent utilise l’ordre alphabétique pour créer les noms des clés pivots. vous devrez passer le nom du tableau croisé dynamique comme deuxième paramètre comme suit.

Modèle

public function permissions()
{
   return $this->belongsToMany(Permission::class, 'role_permission');
}

Modèle d’autorisation

public function roles()
{
   return $this->belongsToMany(Roles::class, 'role_permission');
}

Note 2

envisagez de suivre tout en utilisant différents noms de clé dans le tableau croisé dynamique.

Eloquent suppose que si aucune clé n’est transmise en tant que troisième et quatrième paramètres, ce seront les noms de table singuliers avec _id. il suppose donc que le pivot aura les champs role_id et permission_id. Si des clés autres que celles-ci doivent être utilisées, elles doivent être transmises en tant que troisième et quatrième paramètres.

Disons si other_role_id au lieu de role_id et other_permission_id au lieu de permission_id doit être utilisé. Ce serait donc comme suit.

Modèle

public function permissions()
{
   return $this->belongsToMany(Permission::class, 'role_permission', 'other_role_id', 'other_permission_id');
}

Modèle d’autorisation

public function roles()
{
   return $this->belongsToMany(Roles::class, 'role_permission', 'other_permission_id', 'other_role_id');
}

Polymorphe #

Les relations polymorphes permettent à un modèle d’appartenir à plus d’un autre modèle sur une même association. Un bon exemple serait les images, à la fois un utilisateur et un produit peuvent avoir une image. La structure du tableau peut ressembler à ceci :

user
    id - integer
    name - string
    email - string

product
    id - integer
    title - string
    SKU - string

image
    id - integer
    url - string
    imageable_id - integer
    imageable_type - string

Les colonnes importantes à regarder se trouvent dans le tableau des images. La colonne imageable_id contiendra la valeur d’identification de l’utilisateur ou du produit, tandis que la colonne imageable_type contiendra le nom de classe du modèle propriétaire. Dans vos modèles, vous configurez les relations comme suit :

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Image extends Model
{
    /**
     * Get all of the owning imageable models.
     */
    public function imageable()
    {
        return $this->morphTo();
    }
}

class User extends Model
{
    /**
     * Get all of the user's images.
     */
    public function images()
    {
        return $this->morphMany('Image::class', 'imageable');
    }
}

class Product extends Model
{
    /**
     * Get all of the product's images.
     */
    public function images()
    {
        return $this->morphMany('Image::class', 'imageable');
    }
}

Vous pouvez également récupérer le propriétaire d’une relation polymorphe à partir du modèle polymorphe en accédant au nom de la méthode qui effectue l’appel à morphTo. Dans notre cas, il s’agit de la méthode “imageable” sur le modèle Image. Nous allons donc accéder à cette méthode en tant que propriété dynamique

$image = App\Image::find(1);

$imageable = $image->imageable;

Cette imageable renverra soit un utilisateur, soit un produit.

Interrogation sur les relations #

Eloquent vous permet également d’interroger des relations définies, comme indiqué ci-dessous :

User::whereHas('articles', function (Builder $query) {
    $query->where('published', '!=', true);
})->get();

Cela nécessite que le nom de votre méthode de relation soit articles dans ce cas. L’argument passé dans la fermeture est le générateur de requêtes pour le modèle associé, vous pouvez donc utiliser ici toutes les requêtes que vous pouvez utiliser ailleurs.

Chargement impatient

Supposons que le modèle utilisateur ait une relation avec le modèle article et que vous souhaitiez charger avec impatience les articles associés. Cela signifie que les articles de l’utilisateur seront chargés lors de la récupération de l’utilisateur.

articles est le nom de la relation (méthode) dans le modèle utilisateur.

User::with('articles')->get();

si vous avez plusieurs relations. par exemple des articles et des messages.

User::with('articles','posts')->get();

et pour sélectionner des relations imbriquées

User::with('posts.comments')->get();

Appeler plusieurs relations imbriquées

User::with('posts.comments.likes')->get()

Disons qu’il y a des rôles et des autorisations. Chaque rôle peut appartenir à plusieurs autorisations et chaque autorisation peut appartenir à plusieurs rôles. il y aura donc 3 tables. deux modèles et un tableau croisé dynamique. une table roles, users et permission_role.

Modèle

public function permissions()
{
   return $this->belongsToMany(Permission::class);
}

Modèle d’autorisation

public function roles()
{
   return $this->belongsToMany(Roles::class);
}

considérez ce qui suit lorsque vous utilisez un nom de table différent pour le tableau croisé dynamique.

public function permissions()

public function roles()

}

Modèle

{
   return $this->belongsToMany(Permission::class, 'role_permission', 'other_role_id', 'other_permission_id');
}

Modèle d’autorisation

public function roles()
{
   return $this->belongsToMany(Roles::class, 'role_permission', 'other_permission_id', 'other_role_id');
}

Accéder à la table intermédiaire à l’aide de withPivot()

Supposons que vous ayez une troisième colonne ‘permission_assigned_date’ dans le tableau croisé dynamique . Par défaut, seules les clés du modèle seront présentes sur l’objet pivot. Maintenant, pour obtenir cette colonne dans le résultat de la requête, vous devez ajouter le nom dans la fonction withPivot().

   public function permissions()
        {
           return $this->belongsToMany(Permission::class, 'role_permission', 'other_role_id', 'other_permission_id')->withPivot('permission_assigned_date');
        }

Attacher / Détacher

Eloquent fournit également quelques méthodes d’assistance supplémentaires pour faciliter le travail avec des modèles associés. Par exemple, imaginons qu’un utilisateur puisse avoir de nombreux rôles et qu’un rôle puisse avoir de nombreuses autorisations. Pour attacher un rôle à une permission en insérant un enregistrement dans la table intermédiaire qui joint les modèles, utilisez la méthode attach :

$role= App\Role::find(1);    
$role->permissions()->attach($permissionId);

Lors de l’attachement d’une relation à un modèle, vous pouvez également passer un tableau de données supplémentaires à insérer dans la table intermédiaire :

$rol->roles()->attach($permissionId, ['permission_assigned_date' => $date]);

De même, pour supprimer une autorisation spécifique sur un rôle, utilisez la fonction de détachement

$role= App\Role::find(1);
//will remove permission 1,2,3 against role 1
$role->permissions()->detach([1, 2, 3]);

Associations de synchronisation

Vous pouvez également utiliser la méthode sync pour construire des associations plusieurs-à-plusieurs. La méthode de synchronisation accepte un tableau d’ID à placer sur la table intermédiaire. Tous les ID qui ne figurent pas dans le tableau donné seront supprimés de la table intermédiaire. Ainsi, une fois cette opération terminée, seuls les ID du tableau donné existeront dans la table intermédiaire :

//will keep permission id's 1,2,3 against Role id 1

$role= App\Role::find(1)
$role->permissions()->sync([1, 2, 3]);

Introduction #

Les relations Eloquent sont définies comme des fonctions sur vos classes de modèle Eloquent. Étant donné que, comme les modèles Eloquent eux-mêmes, les relations servent également de puissants générateurs de requêtes, la définition des relations en tant que fonctions fournit de puissantes capacités de chaînage de méthodes et d’interrogation. Par exemple, on peut enchaîner des contraintes supplémentaires sur cette relation posts :

$user->posts()->where('active', 1)->get();
