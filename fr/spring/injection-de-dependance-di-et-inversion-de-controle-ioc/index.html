Injection de dépendance (DI) et inversion de contrôle (IoC)

Le code source des grandes applications logicielles est généralement organisé en plusieurs unités. La définition d’une unité varie normalement selon le langage de programmation utilisé. Par exemple, le code écrit dans un langage de programmation procédural (comme C) est organisé en “fonctions” ou “procédures”. De même, le code d’un langage de programmation orienté objet (comme Java, Scala et C#) est organisé en “classes”, “interfaces”, etc. Ces unités d’organisation de code peuvent être considérées comme des unités individuelles constituant l’application logicielle globale.

Lorsque les applications ont plusieurs unités, des interdépendances entre ces unités apparaissent lorsqu’une unité doit en utiliser d’autres pour compléter ses fonctionnalités. Les unités dépendantes peuvent être considérées comme des “consommateurs” et les unités dont elles dépendent comme des “fournisseurs” de fonctionnalités spécifiques.

L’approche de programmation la plus simple consiste pour les consommateurs à contrôler entièrement le flux d’une application logicielle en décidant quels fournisseurs doivent être instanciés, utilisés et détruits à quels moments de l’exécution globale de l’application. On dit que les consommateurs ont un contrôle total sur les fournisseurs pendant le flux d’exécution, qui sont des “dépendances” pour les consommateurs. Dans le cas où les fournisseurs ont leurs propres dépendances, les consommateurs peuvent avoir à se soucier de la manière dont les fournisseurs doivent être initialisés (et libérés), ce qui rend le flux de contrôle de plus en plus compliqué à mesure que le nombre d’unités dans le logiciel augmente. Cette approche augmente également le couplage entre les unités, ce qui rend de plus en plus difficile le changement d’unités individuellement sans se soucier de casser d’autres parties du logiciel.

[Inversion de contrôle] (https://en.wikipedia.org/wiki/Inversion_of_control) (IoC) est un principe de conception qui préconise l’externalisation des activités de flux de contrôle telles que la découverte, l’instanciation et la destruction d’unités vers un cadre indépendant des consommateurs et des fournisseurs . Le principe sous-jacent de l’IoC est de découpler les consommateurs et les fournisseurs, libérant ainsi les unités logicielles du souci de découvrir, instancier et nettoyer leurs dépendances et permettant aux unités de se concentrer sur leurs propres fonctionnalités. Ce découplage permet de garder le logiciel extensible et maintenable.

Injection de dépendances est l’une des techniques de mise en œuvre du principe d’inversion de contrôle par lequel des instances de dépendances (fournisseurs) sont injectées dans une unité logicielle (le consommateur) au lieu de le consommateur devant les trouver et les instancier.

Le framework Spring contient un module d’injection de dépendances en son cœur qui permet aux beans gérés par Spring d’être injectés dans d’autres beans gérés par Spring en tant que dépendances.

Câblage automatique d’une dépendance via la configuration Java #

L’injection de constructeur via la configuration Java peut également utiliser le câblage automatique, par exemple :

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo(Bar bar) { return new Foo(bar); }
}

Câblage automatique d’une dépendance via la configuration XML #

Les dépendances peuvent être câblées automatiquement lors de l’utilisation de la fonction d’analyse des composants du framework Spring. Pour que le câblage automatique fonctionne, la configuration XML suivante doit être effectuée :

<context:annotation-config/>
<context:component-scan base-package="[base package]"/>

où, base-package est le package Java complet dans lequel Spring doit effectuer l’analyse des composants.

Injection constructeur

Les dépendances peuvent être injectées via le constructeur de classe comme suit :

@Component
class Bar { ... }

@Component
class Foo {
  private Bar bar;

  @Autowired
  public Foo(Bar bar) { this.bar = bar; }
}

Ici, @Autowired est une annotation spécifique à Spring. Spring prend également en charge JSR-299 pour permettre la portabilité des applications vers d’autres frameworks d’injection de dépendances basés sur Java. Cela permet de remplacer @Autowired par @Inject comme :

@Component
class Foo {
  private Bar bar;

  @Inject
  public Foo(Bar bar) { this.bar = bar; }
}

Injection immobilière

Les dépendances peuvent également être injectées à l’aide de méthodes setter comme suit :

@Component
class Foo {
  private Bar bar;

  @Autowired
  public void setBar(Bar bar) { this.bar = bar; }
}

Injection de terrain

La connexion automatique permet également d’initialiser directement les champs dans les instances de classe, comme suit :

@Component
class Foo {
  @Autowired
  private Bar bar;
}

Pour les versions Spring 4.1+, vous pouvez utiliser [Optional][1] pour les dépendances facultatives.

@Component
class Foo {

    @Autowired
    private Optional<Bar> bar;
}

La même approche peut être utilisée pour le constructeur DI.

@Component
class Foo {
    private Optional<Bar> bar;

    @Autowired
    Foo(Optional<Bar> bar) {
        this.bar = bar;
    }
}

[1] : https://docs.oracle.com/javase/8/docs/api/java/util/Optional.html

## Injecter une dépendance manuellement via la configuration XML Considérez les classes Java suivantes :

class Foo {
  private Bar bar;

  public void foo() {
    bar.baz();
  }
}

Comme on peut le voir, la classe Foo doit appeler la méthode baz sur une instance d’une autre classe Bar pour que sa méthode foo fonctionne correctement. Bar est considéré comme une dépendance de Foo puisque Foo ne peut pas fonctionner correctement sans une instance de Bar.

Injection constructeur

Lors de l’utilisation de la configuration XML pour le framework Spring pour définir des beans gérés par Spring, un bean de type “Foo” peut être configuré comme suit :

<bean class="Foo">
  <constructor-arg>
    <bean class="Bar" />
  </constructor-arg>
</bean>

ou, alternativement (plus verbeux):

<bean id="bar" class="bar" />

<bean class="Foo">
  <constructor-arg ref="bar" />
</bean>

Dans les deux cas, le framework Spring crée d’abord une instance de “Bar” et “l’injecte” dans une instance de “Foo”. Cet exemple suppose que la classe Foo a un constructeur qui peut prendre une instance Bar comme paramètre, c’est-à-dire :

class Foo {
  private Bar bar;

  public Foo(Bar bar) { this.bar = bar; }
}

Ce style est connu sous le nom d’injection de constructeur car la dépendance (instance ‘Bar’) est injectée via le constructeur de classe.

Injection immobilière

Une autre option pour injecter la dépendance Bar dans Foo est :

<bean class="Foo">
  <property name="bar">
    <bean class="Bar" />
  </property>
</bean>

ou, alternativement (plus verbeux):

<bean id="bar" class="bar" />

<bean class="Foo">
  <property name="bar" ref="bar" />
</bean>

Cela nécessite que la classe Foo ait une méthode setter qui accepte une instance Bar, telle que :

class Foo {
  private Bar bar;

  public void setBar(Bar bar) { this.bar = bar; }
}

Injecter une dépendance manuellement via la configuration Java #

Les mêmes exemples que ceux présentés ci-dessus avec la configuration XML peuvent être réécrits avec la configuration Java comme suit.

Injection constructeur

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() { return new Foo(bar()); }
}

Injection immobilière

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() {
    Foo foo = new Foo();
    foo.setBar(bar());

    return foo;
  }
}
