Web scraping et parsing

  • Grattage de base avec rvest
  • Utilisation de rvest lorsque la connexion est requise

Sur cette page

  • Grattage de base avec rvest
  • Utilisation de rvest lorsque la connexion est requise

Scraping fait référence à l’utilisation d’un ordinateur pour récupérer le code d’une page Web. Une fois le code obtenu, il doit être parsé dans une forme utile pour une utilisation ultérieure dans R.

Base R n’a pas beaucoup d’outils requis pour ces processus, donc le grattage et l’analyse sont généralement effectués avec des packages. Certains packages sont plus utiles pour le scraping (RSelenium, httr, curl, RCurl), certains pour l’analyse (XML, xml2) et d’autres pour les deux (rvest).

Un processus connexe consiste à gratter une API Web, qui, contrairement à une page Web, renvoie des données destinées à être lisibles par une machine. Bon nombre des mêmes packages sont utilisés pour les deux.

Légalité #

Certains sites Web s’opposent au scraping, que ce soit en raison d’une charge accrue du serveur ou de préoccupations concernant la propriété des données. Si un site Web interdit de gratter dans ses conditions d’utilisation, il est illégal de le gratter.

Grattage de base avec rvest #

rvest est un package de grattage et d’analyse Web par Hadley Wickham inspiré par Python’s [Beautiful Soup](https://www.crummy.com/software/BeautifulSoup /). Il exploite les liaisons libxml2 du package xml2 de Hadley pour l’analyse HTML.

Dans le cadre du tidyverse, rvest est canalisé. Il utilise

  • xml2::read_html pour scraper le HTML d’une page web,
  • qui peut ensuite être sous-ensemble avec ses fonctions html_node et html_nodes à l’aide de sélecteurs CSS ou XPath, et
  • analysé en objets R avec des fonctions telles que html_text et html_table.

Pour gratter le tableau des jalons de la page Wikipedia sur R, le code ressemblerait à

library(rvest)

url <- 'https://en.wikipedia.org/wiki/R_(programming_language)'

        # scrape HTML from website
url %>% read_html() %>% 
    # select HTML tag with class="wikitable"
    html_node(css = '.wikitable') %>% 
    # parse table into data.frame
    html_table() %>%
    # trim for printing
    dplyr::mutate(Description = substr(Description, 1, 70))

Bien que cela renvoie un data.frame, notez que, comme c’est généralement le cas pour les données récupérées, il reste encore à nettoyer les données : ici, formater les dates, insérer des NA, etc.

Notez que les données dans un format rectangulaire moins cohérent peuvent nécessiter une boucle ou d’autres manipulations supplémentaires pour être analysées avec succès. Si le site Web utilise jQuery ou d’autres moyens pour insérer du contenu, read_html peut être insuffisant pour gratter, et un grattoir plus robuste comme RSelenium peut être nécessaire.

Utilisation de rvest lorsque la connexion est requise #

Le problème courant rencontré lors de la mise au rebut d’un site Web est de savoir comment entrer un nom d’utilisateur et un mot de passe pour se connecter à un site Web.

Dans cet exemple que j’ai créé pour suivre mes réponses postées ici pour empiler le débordement. Le flux global consiste à se connecter, à accéder à une page Web, à collecter des informations, à lui ajouter une trame de données, puis à passer à la page suivante.

library(rvest) 

#Address of the login webpage
login<-"https://stackoverflow.com/users/login?ssrc=head&returnurl=http%3a%2f%2fstackoverflow.com%2f"

#create a web session with the desired login address
pgsession<-html_session(login)
pgform<-html_form(pgsession)[[2]]  #in this case the submit is the 2nd form
filled_form<-set_values(pgform, email="*****", password="*****")
submit_form(pgsession, filled_form)

#pre allocate the final results dataframe.
results<-data.frame()  

#loop through all of the pages with the desired info
for (i in 1:5)
{
  #base address of the pages to extract information from
  url<-"http://stackoverflow.com/users/**********?tab=answers&sort=activity&page="
  url<-paste0(url, i)
  page<-jump_to(pgsession, url)

  #collect info on the question votes and question title
  summary<-html_nodes(page, "div .answer-summary")
  question<-matrix(html_text(html_nodes(summary, "div"), trim=TRUE), ncol=2, byrow = TRUE)

  #find date answered, hyperlink and whether it was accepted
  dateans<-html_node(summary, "span") %>% html_attr("title")
  hyperlink<-html_node(summary, "div a") %>% html_attr("href")
  accepted<-html_node(summary, "div") %>% html_attr("class")

  #create temp results then bind to final results 
  rtemp<-cbind(question, dateans, accepted, hyperlink)
  results<-rbind(results, rtemp)
}

#Dataframe Clean-up
names(results)<-c("Votes", "Answer", "Date", "Accepted", "HyperLink")
results$Votes<-as.integer(as.character(results$Votes))
results$Accepted<-ifelse(results$Accepted=="answer-votes default", 0, 1)

La boucle dans ce cas est limitée à seulement 5 pages, cela doit changer pour s’adapter à votre application. J’ai remplacé les valeurs spécifiques à l’utilisateur par ******, j’espère que cela vous aidera à résoudre votre problème.
