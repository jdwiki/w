L’interface mysqli est une amélioration (cela signifie “extension d’amélioration de MySQL”) de l’interface mysql, qui était obsolète dans la version 5.5 et est supprimé dans la version 7.0. L’extension mysqli, ou comme on l’appelle parfois, l’extension MySQL améliorée, a été développée pour tirer parti des nouvelles fonctionnalités trouvées dans les versions 4.1.3 et plus récentes des systèmes MySQL. L’extension mysqli est incluse avec les versions PHP 5 et ultérieures.

Fonctionnalités #

L’interface mysqli présente un certain nombre d’avantages, les principales améliorations par rapport à l’extension mysql étant :

  • Interface orientée objet
  • Prise en charge des déclarations préparées
  • Prise en charge de plusieurs déclarations
  • Prise en charge des transactions
  • Capacités de débogage améliorées
  • Prise en charge du serveur intégré

Il comporte une interface double : l’ancien style procédural et une nouvelle programmation orientée objet (POO). Le mysql obsolète n’avait qu’une interface procédurale, donc le style orienté objet est souvent préféré. Cependant, le nouveau style est également favorable en raison de la puissance de la POO.

Alternatives #

Une alternative à l’interface mysqli pour accéder aux bases de données est la nouvelle interface [PHP Data Objects (PDO)][1]. Cela ne comporte que la programmation de style OOP et peut accéder à plus que des bases de données de type MySQL.

[1] : https://www.wikiod.com/fr/php

Fermer la connexion #

Lorsque nous avons fini d’interroger la base de données, il est recommandé de fermer la connexion pour libérer des ressources.

Style orienté objet

$conn->close();

Style procédural

mysqli_close($conn);

Remarque : La connexion au serveur sera fermée dès la fin de l’exécution du script, à moins qu’elle ne soit fermée plus tôt en appelant explicitement la fonction de fermeture de connexion.

Cas d’utilisation : Si notre script a une bonne quantité de traitement à effectuer après avoir récupéré le résultat et a récupéré le jeu de résultats complet, nous devons définitivement fermer la connexion. Si nous ne le faisions pas, il est possible que le serveur MySQL atteigne sa limite de connexion lorsque le serveur Web est fortement utilisé.

Connexion MySQLi #

Style orienté objet

Connecter au serveur

$conn = new mysqli("localhost","my_user","my_password");

Définissez la base de données par défaut : $conn->select_db("my_db");

Se connecter à la base de données

$conn = new mysqli("localhost","my_user","my_password","my_db");

Style procédural

Connecter au serveur

$conn = mysqli_connect("localhost","my_user","my_password");

Définissez la base de données par défaut : mysqli_select_db($conn, "my_db");

Se connecter à la base de données

$conn = mysqli_connect("localhost","my_user","my_password","my_db");

Vérifier la connexion à la base de données

Style orienté objet

if ($conn->connect_errno > 0) {
    trigger_error($db->connect_error);
} // else: successfully connected

Style procédural

if (!$conn) {
   trigger_error(mysqli_connect_error());
} // else: successfully connected

Boucle dans les résultats MySQLi #

PHP permet d’obtenir facilement des données à partir de vos résultats et de les boucler à l’aide d’une instruction “while”. Lorsqu’il ne parvient pas à obtenir la ligne suivante, il renvoie “false” et votre boucle se termine. Ces exemples fonctionnent avec

  • mysqli_fetch_assoc - Tableau associatif avec les noms de colonnes comme clés
  • mysqli_fetch_object - Objet stdClass avec les noms de colonnes comme variables
  • mysqli_fetch_array - Tableau associatif ET numérique (peut utiliser des arguments pour obtenir l’un ou l’autre)
  • mysqli_fetch_row - Tableau numérique

Style orienté objet

while($row = $result->fetch_assoc()) {
    var_dump($row);
}

Style procédural

while($row = mysqli_fetch_assoc($result)) {
    var_dump($row);
}

Pour obtenir des informations exactes à partir des résultats, nous pouvons utiliser :

while ($row = $result->fetch_assoc()) {
    echo 'Name and surname: '.$row['name'].' '.$row['surname'].'<br>';
    echo 'Age: '.$row['age'].'<br>'; // Prints info from 'age' column
}

Déclarations préparées dans MySQLi #

Veuillez lire https://www.wikiod.com/fr/php/aop#Empêcher l’injection SQL avec des requêtes paramétrées pour une discussion complète sur les raisons pour lesquelles les instructions préparées vous aident à sécuriser vos instructions SQL contre les attaques par injection SQL

La variable $conn ici est un objet MySQLi. Voir [MySQLi connect example][1] pour plus de détails.

Pour les deux exemples, nous supposons que $sql est

$sql = "SELECT column_1 
    FROM table 
    WHERE column_2 = ? 
        AND column_3 > ?";

Le ? représente les valeurs que nous fournirons plus tard. Veuillez noter que nous n’avons pas besoin de guillemets pour les espaces réservés, quel que soit le type. Nous ne pouvons également fournir que des espaces réservés dans les parties de données de la requête, c’est-à-dire SET, VALUES et WHERE. Vous ne pouvez pas utiliser d’espaces réservés dans les parties SELECT ou FROM.

Style orienté objet

if ($stmt = $conn->prepare($sql)) {
  $stmt->bind_param("si", $column_2_value, $column_3_value);
  $stmt->execute();

  $stmt->bind_result($column_1);
  $stmt->fetch();
  //Now use variable $column_1 one as if it were any other PHP variable
  $stmt->close();
}

Style procédural

if ($stmt = mysqli_prepare($conn, $sql)) {
  mysqli_stmt_bind_param($stmt, "si", $column_2_value, $column_3_value);
  mysqli_stmt_execute($stmt);
  // Fetch data here
  mysqli_stmt_close($stmt);
}

Le premier paramètre de $stmt->bind_param ou le deuxième paramètre de mysqli_stmt_bind_param est déterminé par le type de données du paramètre correspondant dans la requête SQL :

Paramètre Type de données du paramètre lié
je entier
d doubler
s chaîne
b goutte

Votre liste de paramètres doit être dans l’ordre indiqué dans votre requête. Dans cet exemple, si signifie que le premier paramètre (column_2 = ?) est une chaîne et que le second paramètre (column_3 > ?) est un entier.

Pour récupérer des données, voir https://www.wikiod.com/fr/php/php-mysql#Comment obtenir des données à partir d’un relevé préparé

[1] : https://www.wikiod.com/fr/php/php-mysql#Connexion MySQLi

Chaînes d’échappement #

L’échappement de chaînes est une méthode plus ancienne (et moins sécurisée) de sécurisation des données à insérer dans une requête. Cela fonctionne en utilisant MySQL’s function mysql_real_escape_string() pour traiter et assainir les données (en d’autres termes , PHP ne fait pas l’échappement). L’API MySQLi fournit un accès direct à cette fonction

$escaped = $conn->real_escape_string($_GET['var']);
// OR
$escaped = mysqli_real_escape_string($conn, $_GET['var']);

À ce stade, vous avez une chaîne que MySQL considère comme sûre pour une utilisation dans une requête directe

$sql = 'SELECT * FROM users WHERE username = "' . $escaped . '"';
$result = $conn->query($sql);

Alors, pourquoi n’est-ce pas aussi sûr que les [instructions préparées] (https://www.wikiod.com/fr/php/php-mysql#Instructions préparées dans MySQLi) ? Il existe des moyens de tromper MySQL pour qu’il produise une chaîne qu’il considère comme sûre. Considérez l’exemple suivant

$id = mysqli_real_escape_string("1 OR 1=1");    
$sql = 'SELECT * FROM table WHERE id = ' . $id;

1 OR 1=1 ne représente pas les données que MySQL échappera, mais cela représente toujours l’injection SQL. Il existe [d’autres exemples] (http://stackoverflow.com/questions/5741187/sql-injection-that-gets-around-mysql-real-escape-string) qui représentent également des endroits où il renvoie des données non sécurisées. Le problème est que la fonction d’échappement de MySQL est conçue pour rendre les données conformes à la syntaxe SQL. Il n’est PAS conçu pour s’assurer que MySQL ne peut pas confondre les données utilisateur avec les instructions SQL.

Requête MySQLi #

La fonction query prend une chaîne SQL valide et l’exécute directement sur la connexion à la base de données $conn

Style orienté objet

$result = $conn->query("SELECT * FROM `people`");

Style procédural

$result = mysqli_query($conn, "SELECT * FROM `people`");

ATTENTION

Un problème courant ici est que les gens exécuteront simplement la requête et s’attendront à ce qu’elle fonctionne (c’est-à-dire qu’elle renvoie un objet mysqli_stmt). Étant donné que cette fonction ne prend qu’une chaîne, vous créez d’abord la requête vous-même. S’il y a des erreurs dans le SQL, le compilateur MySQL échouera, à quel point cette fonction renverra false.

$result = $conn->query('SELECT * FROM non_existent_table'); // This query will fail
$row = $result->fetch_assoc();

Le code ci-dessus générera une erreur E_FATAL car $result est false et non un objet.

PHP Fatal error : Appel à une fonction membre fetch_assoc() sur un non-objet

L’erreur de procédure est similaire, mais pas fatale, car nous ne faisons que violer les attentes de la fonction.

$row = mysqli_fetch_assoc($result); // same query as previous

Vous obtiendrez le message suivant de PHP

mysqli_fetch_array() s’attend à ce que le paramètre 1 soit mysqli_result, booléen donné

Vous pouvez éviter cela en faisant d’abord un test

if($result) $row = mysqli_fetch_assoc($result);

Déboguer SQL dans MySQLi #

Votre requête a donc échoué (voir https://www.wikiod.com/fr/php/php-mysql#Connexion MySQLi pour savoir comment nous avons créé $conn)

$result = $conn->query('SELECT * FROM non_existent_table'); // This query will fail

Comment savoir ce qui s’est passé ? $result est false donc cela ne sert à rien. Heureusement, la connexion $conn peut nous dire ce que MySQL nous a dit à propos de l’échec

trigger_error($conn->error);

ou procédural

trigger_error(mysqli_error($conn));

Vous devriez obtenir une erreur similaire à

La table ‘my_db.non_existent_table’ n’existe pas

Comment obtenir des données à partir d’une instruction préparée #

Relevés préparés #

Voir [Instructions préparées dans MySQLi](https://www.wikiod.com/fr/php/php-mysql#Instructions préparées dans MySQLi) pour savoir comment préparer et exécuter une requête.

Liaison des résultats #

Style orienté objet

$stmt->bind_result($forename);

Style procédural

mysqli_stmt_bind_result($stmt, $forename);

Le problème avec l’utilisation de bind_result est qu’il nécessite que l’instruction spécifie les colonnes qui seront utilisées. Cela signifie que pour que ce qui précède fonctionne, la requête doit avoir ressemblé à ceci SELECT prename FROM users. Pour inclure plus de colonnes, ajoutez-les simplement en tant que paramètres à la fonction bind_result (et assurez-vous de les ajouter à la requête SQL).

Dans les deux cas, nous attribuons la colonne prénom à la variable $prénom. Ces fonctions prennent autant d’arguments que de colonnes que vous souhaitez affecter. L’affectation n’est effectuée qu’une seule fois, car la fonction se lie par référence.

On peut alors boucler comme suit :

Style orienté objet

while ($stmt->fetch())
    echo "$forename<br />";

Style procédural

while (mysqli_stmt_fetch($stmt))
    echo "$forename<br />";

L’inconvénient est que vous devez affecter un grand nombre de variables à la fois. Cela rend difficile le suivi des requêtes volumineuses. Si MySQL Native Driver (mysqlnd) est installé, il vous suffit d’utiliser [get_result](http://php .net/manual/en/mysqli-stmt.get-result.php).

Style orienté objet

$result = $stmt->get_result();

Style procédural

$result = mysqli_stmt_get_result($stmt);

C’est ** beaucoup ** plus facile à utiliser car nous obtenons maintenant un objet mysqli_result. C’est le même objet que [mysqli_query renvoie](https://www.wikiod.com/fr/php/php-mysql#Requête MySQLi). Cela signifie que vous pouvez utiliser une [boucle de résultat régulière](https://www.wikiod.com/fr/php/php-mysql#Boucle dans les résultats MySQLi) pour obtenir vos données.

Que se passe-t-il si je ne peux pas installer mysqlnd ? #

Si tel est le cas, alors @Sophivorus vous a couvert avec [cette réponse étonnante] (http://stackoverflow.com/a/30551477/3578036).

Cette fonction peut effectuer la tâche de get_result sans qu’elle soit installée sur le serveur. Il boucle simplement les résultats et construit un tableau associatif

function get_result(\mysqli_stmt $statement)
{
    $result = array();
    $statement->store_result();
    for ($i = 0; $i < $statement->num_rows; $i++)
    {
        $metadata = $statement->result_metadata();
        $params = array();
        while ($field = $metadata->fetch_field())
        {
            $params[] = &$result[$i][$field->name];
        }
        call_user_func_array(array($statement, 'bind_result'), $params);
        $statement->fetch();
    }
    return $result;
}

Nous pouvons ensuite utiliser la fonction pour obtenir des résultats comme celui-ci, comme si nous utilisions mysqli_fetch_assoc()

<?php
$query = $mysqli->prepare("SELECT * FROM users WHERE forename LIKE ?");
$condition = "J%";
$query->bind_param("s", $condition);
$query->execute();
$result = get_result($query);

while ($row = array_shift($result)) {
    echo $row["id"] . ' - ' . $row["forename"] . ' ' . $row["surname"] . '<br>';
}

Il aura la même sortie que si vous utilisiez le pilote mysqlnd, sauf qu’il n’a pas besoin d’être installé. Ceci est très utile si vous ne parvenez pas à installer ledit pilote sur votre système. Il suffit d’implémenter cette solution.

## ID d’insertion MySQLi Récupérer le dernier ID généré par une requête [INSERT][1] sur une table avec une colonne [AUTO_INCREMENT][3].

Style orienté objet

$id = $conn->insert_id;

Style procédural

$id = mysqli_insert_id($conn);

Renvoie zéro s’il n’y avait pas de requête précédente sur la connexion ou si la requête n’a pas mis à jour une valeur AUTO_INCREMENT.

Insérer l’identifiant lors de la mise à jour des lignes

Normalement, une instruction UPDATE ne renvoie pas d’identifiant d’insertion, car un identifiant AUTO_INCREMENT n’est renvoyé que lorsqu’une nouvelle ligne a été enregistrée (ou insérée). Une façon de mettre à jour le nouvel identifiant consiste à utiliser la syntaxe INSERT ... ON DUPLICATE KEY UPDATE pour la mise à jour.

Configuration pour les exemples à suivre :

CREATE TABLE iodku (
    id INT AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    misc INT NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
) ENGINE=InnoDB;

INSERT INTO iodku (name, misc)
    VALUES
    ('Leslie', 123),
    ('Sally', 456);
Query OK, 2 rows affected (0.00 sec)
Records: 2  Duplicates: 0  Warnings: 0
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  |  456 |
+----+--------+------+

Le cas où IODKU effectue une “mise à jour” et LAST_INSERT_ID() récupère l’id pertinent :

$sql = "INSERT INTO iodku (name, misc)
    VALUES
    ('Sally', 3333)            -- should update
    ON DUPLICATE KEY UPDATE    -- `name` will trigger "duplicate key"
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc)";
$conn->query($sql);
$id = $conn->insert_id;        -- picking up existing value (2)

Le cas où IODKU effectue un “insert” et LAST_INSERT_ID() récupère le nouvel id :

$sql = "INSERT INTO iodku (name, misc)
    VALUES
    ('Dana', 789)            -- Should insert
    ON DUPLICATE KEY UPDATE
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
$conn->query($sql);
$id = $conn->insert_id;      -- picking up new value (3)

Contenu du tableau résultant :

SELECT * FROM iodku;
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  | 3333 |  -- IODKU changed this
|  3 | Dana   |  789 |  -- IODKU added this
+----+--------+------+
