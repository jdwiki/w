Les superglobales sont des variables intégrées qui sont toujours disponibles dans toutes les étendues.

Plusieurs variables prédéfinies en PHP sont des “superglobales”, ce qui signifie qu’elles sont disponibles dans toutes les étendues d’un script. Il n’est pas nécessaire de faire global $variable; pour y accéder dans les fonctions ou les méthodes.

Les sous-globaux expliqués #

Introduction #

En termes simples, ce sont des variables qui sont disponibles dans toutes la portée de vos scripts.

Cela signifie qu’il n’est pas nécessaire de les transmettre en tant que paramètres dans vos fonctions ou de les stocker en dehors d’un bloc de code pour les rendre disponibles dans différentes portées.

Qu’est-ce qu’un superglobal ?? #

Si vous pensez que ce sont comme des super-héros, ils ne le sont pas.

Depuis la version 7.1.3 de PHP, il existe 9 variables superglobales. Ils sont les suivants :

  • $GLOBALS - Référence toutes les variables disponibles dans la portée globale
  • $_SERVER - Informations sur le serveur et l’environnement d’exécution
  • $_GET - Variables HTTP GET
  • $_POST - variables POST HTTP
  • $_FILES - Variables de téléchargement de fichiers HTTP
  • $_COOKIE - Cookies HTTP
  • $_SESSION - Variables de sessions
  • $_REQUEST - Variables de requête HTTP
  • $_ENV - Variables d’environnement

Voir la documentation.

Dis m’en plus, dis m’en plus #

Je suis désolé pour la référence à Grease ! Lien

Il est temps d’avoir quelques explications sur ces superhérosmondiaux.

$GLOBAUX #

Un tableau associatif contenant des références à toutes les variables actuellement définies dans la portée globale du script. Les noms de variables sont les clés du tableau.

Code

$myGlobal = "global"; // declare variable outside of scope

function test()
{
    $myLocal = "local"; // declare variable inside of scope
    // both variables are printed
    var_dump($myLocal);
    var_dump($GLOBALS["myGlobal"]);
}

test(); // run function
// only $myGlobal is printed since $myLocal is not globally scoped
var_dump($myLocal);
var_dump($myGlobal);

Production

string 'local' (length=5)
string 'global' (length=6)
null
string 'global' (length=6)

Dans l’exemple ci-dessus, $myLocal n’est pas affiché la deuxième fois car il est déclaré dans la fonction test() puis détruit après la fermeture de la fonction.

Devenir mondial #

Pour y remédier, deux options s’offrent à vous.

Option 1 : mot clé “global”

function test()
{
    global $myLocal;
    $myLocal = "local";
    var_dump($myLocal);
    var_dump($GLOBALS["myGlobal"]);
}

Le mot clé global est un préfixe sur une variable qui la force à faire partie de la portée globale.

Notez que vous ne pouvez pas affecter de valeur à une variable dans la même instruction que le mot-clé global. Par conséquent, pourquoi j’ai dû attribuer une valeur en dessous. (C’est possible si vous supprimez les nouvelles lignes et les espaces mais je ne pense pas que ce soit net. global $myLocal; $myLocal = "local").

Deuxième option : tableau $GLOBALS

function test()
{
    $GLOBALS["myLocal"] = "local";
    $myLocal = $GLOBALS["myLocal"];
    var_dump($myLocal);
    var_dump($GLOBALS["myGlobal"]);
}

Dans cet exemple, j’ai réaffecté $myLocal la valeur de $GLOBAL["myLocal"] car je trouve plus facile d’écrire un nom de variable plutôt que le tableau associatif.

$_SERVER est un tableau contenant des informations telles que les en-têtes, les chemins et les emplacements des scripts. Les entrées de ce tableau sont créées par le serveur Web. Il n’y a aucune garantie que chaque serveur Web fournira l’un de ces éléments ; les serveurs peuvent en omettre certains ou en fournir d’autres non répertoriés ici. $_GET #

Un tableau associatif de variables passé au script courant via les paramètres d’URL.

$_GET est un tableau qui contient tous les paramètres d’URL ; ce sont ce qu’il y a après le ? dans l’URL.

En utilisant http://www.example.com/index.php?myVar=myVal comme exemple. Cette information de cette URL peut être obtenue en accédant à ce format $_GET["myVar"] et le résultat sera myVal.

Utiliser du code pour ceux qui n’aiment pas lire.

// URL = http://www.example.com/index.php?myVar=myVal
echo $_GET["myVar"] == "myVal" ? "true" : "false"; // returns "true"

L’exemple ci-dessus utilise l’[opérateur ternaire](https://www.wikiod.com/fr/php/les-operateurs#Opérateur ternaire (?:)).

Cela montre comment vous pouvez accéder à la valeur à partir de l’URL en utilisant le superglobal $_GET.

Maintenant un autre exemple ! haleter

// URL = http://www.example.com/index.php?myVar=myVal&myVar2=myVal2
echo $_GET["myVar"]; // returns "myVal"
echo $_GET["myVar2"]; // returns "myVal2"

Il est possible d’envoyer plusieurs variables via l’URL en les séparant par une esperluette (&).

Risque de sécurité Il est très important de ne pas envoyer d’informations sensibles via l’URL car elles resteront dans l’historique de l’ordinateur et seront visibles par toute personne pouvant accéder à ce navigateur.

$_POST #

Un tableau associatif de variables transmis au script actuel via la méthode HTTP POST lors de l’utilisation de application/x-www-form-urlencoded ou multipart/form-data comme HTTP Content-Type dans la requête.

Très similaire à $_GET en ce sens que les données sont envoyées d’un endroit à un autre.

Je vais commencer par passer directement à un exemple. (J’ai omis l’attribut action car cela enverra les informations à la page dans laquelle se trouve le formulaire).

<form method="POST">
    <input type="text" name="myVar" value="myVal" />
    <input type="submit" name="submit" value="Submit" />
</form>

Ci-dessus se trouve un formulaire de base pour lequel des données peuvent être envoyées. Dans un environnement réel, l’attribut value ne serait pas défini, ce qui signifie que le formulaire serait vide. Cela enverrait alors toutes les informations saisies par l’utilisateur.

echo $_POST["myVar"]); // returns "myVal"

Risque de sécurité L’envoi de données via POST n’est pas non plus sécurisé. L’utilisation de HTTPS garantira que les données sont conservées de manière plus sécurisée.

$_FILES #

Un tableau associatif d’éléments téléchargés dans le script courant via la méthode HTTP POST. La structure de ce tableau est décrite dans la section POST method uploads.

Commençons par un formulaire de base.

<form method="POST" enctype="multipart/form-data">
    <input type="file" name="myVar" />
    <input type="submit" name="Submit" />
</form>

Notez que j’ai omis l’attribut action (encore !). De plus, j’ai ajouté enctype="multipart/form-data", c’est important pour tout formulaire qui traitera des téléchargements de fichiers.

// ensure there isn't an error
if ($_FILES["myVar"]["error"] == UPLOAD_ERR_OK)
{
    $folderLocation = "myFiles"; // a relative path. (could be "path/to/file" for example)
    
    // if the folder doesn't exist then make it
    if (!file_exists($folderLocation)) mkdir($folderLocation);

    // move the file into the folder
    move_uploaded_file($_FILES["myVar"]["tmp_name"], "$folderLocation/" . basename($_FILES["myVar"]["name"]));
}

Ceci est utilisé pour télécharger un fichier. Parfois, vous souhaiterez peut-être télécharger plus d’un fichier. Un attribut existe pour cela, il s’appelle multiple. Il existe un attribut pour à peu près n’importe quoi. Je suis désolé

Vous trouverez ci-dessous un exemple de formulaire soumettant plusieurs fichiers.

<form method="POST" enctype="multipart/form-data">
    <input type="file" name="myVar[]" multiple="multiple" />
    <input type="submit" name="Submit" />
</form>

Notez les modifications apportées ici ; il n’y en a que quelques-uns.

  • Le nom input est entre crochets. En effet, il s’agit maintenant d’un tableau de fichiers et nous demandons donc au formulaire de créer un tableau des fichiers sélectionnés. Si vous omettez les crochets, le fichier le plus récent sera défini sur $_FILES["myVar"].

  • L’attribut multiple="multiple". Cela indique simplement au navigateur que les utilisateurs peuvent sélectionner plusieurs fichiers.

    $total = isset($_FILES[“myVar”]) ? count($_FILES[“myVar”][“name”]) : 0; // count how many files were sent // iterate over each of the files for ($i = 0; $i < $total; $i++) { // there isn’t an error if ($_FILES[“myVar”][“error”][$i] == UPLOAD_ERR_OK) { $folderLocation = “myFiles”; // a relative path. (could be “path/to/file” for example)

          // if the folder doesn't exist then make it
      if (!file_exists($folderLocation)) mkdir($folderLocation);

      // move the file into the folder
      move_uploaded_file($_FILES["myVar"]["tmp_name"][$i], "$folderLocation/" . basename($_FILES["myVar"]["name"][$i]));
  }
  // else report the error
  else switch ($_FILES["myVar"]["error"][$i])
  {
      case UPLOAD_ERR_INI_SIZE:
          echo "Value: 1; The uploaded file exceeds the upload_max_filesize directive in php.ini.";
          break;
      case UPLOAD_ERR_FORM_SIZE:
          echo "Value: 2; The uploaded file exceeds the MAX_FILE_SIZE directive that was specified in the HTML form.";
          break;
      case UPLOAD_ERR_PARTIAL:
          echo "Value: 3; The uploaded file was only partially uploaded.";
          break;
      case UPLOAD_ERR_NO_FILE:
          echo "Value: 4; No file was uploaded.";
          break;
      case UPLOAD_ERR_NO_TMP_DIR:
          echo "Value: 6; Missing a temporary folder. Introduced in PHP 5.0.3.";
          break;
      case UPLOAD_ERR_CANT_WRITE:
          echo "Value: 7; Failed to write file to disk. Introduced in PHP 5.1.0.";
          break;
      case UPLOAD_ERR_EXTENSION:
          echo "Value: 8; A PHP extension stopped the file upload. PHP does not provide a way to ascertain which extension caused the file upload to stop; examining the list of loaded extensions with phpinfo() may help. Introduced in PHP 5.2.0.";
          break;

      default:
          echo "An unknown error has occured.";
          break;
  }

    }

Ceci est un exemple très simple et ne gère pas les problèmes tels que les extensions de fichiers non autorisées ou les fichiers nommés avec du code PHP (comme un équivalent PHP d’une injection SQL). Voir la [documentation](https://www.wikiod.com/fr/php/securite#Télécharger des fichiers).

Le premier processus consiste à vérifier s’il existe des fichiers et, le cas échéant, à définir leur nombre total sur $total.

L’utilisation de la boucle for permet une itération du tableau $_FILES et l’accès à chaque élément un à la fois. Si ce fichier ne rencontre pas de problème, l’instruction if est vraie et le code du téléchargement de fichier unique est exécuté. Si un problème est rencontré, le bloc de commutation est exécuté et une erreur est présentée conformément à l’erreur pour ce téléchargement particulier.

$_COOKIE #

Un tableau associatif de variables passé au script courant via les Cookies HTTP.

Les cookies sont des variables qui contiennent des données et sont stockées sur l’ordinateur du client.

Contrairement aux superglobaux susmentionnés, les cookies doivent être créés avec une fonction (et non attribuer une valeur). La convention est ci-dessous.

setcookie("myVar", "myVal", time() + 3600);

Dans cet exemple un nom est spécifié pour le cookie (dans cet exemple c’est “myVar”), une valeur est donnée (dans cet exemple c’est “myVal”, mais une variable peut être passée pour affecter sa valeur au cookie), puis un temps d’expiration est donné (dans cet exemple c’est une heure puisque 3600 secondes est une minute).

Bien que la convention de création d’un cookie soit différente, il est accessible de la même manière que les autres.

echo $_COOKIE["myVar"]; // returns "myVal"

Pour détruire un cookie, setcookie doit être appelé à nouveau, mais le délai d’expiration est défini sur n’importe quel moment dans le passé. Voir ci-dessous.

setcookie("myVar", "", time() - 1);
var_dump($_COOKIE["myVar"]); // returns null

Cela supprimera les cookies et les supprimera de l’ordinateur client.

$_SESSION #

Un tableau associatif contenant les variables de session disponibles pour le script en cours. Voir la documentation Session functions pour plus d’informations sur son utilisation.

Les sessions ressemblent beaucoup aux cookies, sauf qu’elles sont côté serveur.

Pour utiliser des sessions, vous devez inclure session_start() en haut de vos scripts pour permettre l’utilisation des sessions.

La définition d’une variable de session est identique à la définition de toute autre variable. Voir l’exemple ci-dessous.

$_SESSION["myVar"] = "myVal";

Lors du démarrage d’une session, un ID aléatoire est défini comme un cookie et appelé “PHPSESSID” et contiendra l’ID de session pour cette session en cours. Ceci est accessible en appelant la fonction session_id().

Il est possible de détruire des variables de session en utilisant la fonction unset (de sorte que unset($_SESSION["myVar"]) détruirait cette variable). L’alternative est d’appeler session_destory(). Cela détruira toute la session, ce qui signifie que toutes les variables de session n’existeront plus.

$_REQUEST #

Un tableau associatif qui contient par défaut le contenu de $_GET, [$_POST](http:// php.net/manual/en/reserved.variables.post.php) et $_COOKIE.

Comme l’indique la documentation PHP, il s’agit simplement d’une collation de $_GET, $_POST et $_COOKIE dans une seule variable.

Comme il est possible que ces trois tableaux aient un index portant le même nom, il existe un paramètre dans le fichier php.ini appelé request_order qui peut spécifier lequel des trois a la priorité. Par exemple, s’il a été défini sur "GPC", alors la valeur de $_COOKIE sera utilisée, car elle est lue de gauche à droite, ce qui signifie que $_REQUEST définira sa valeur sur $_GET, puis $_POST, puis $_COOKIE et puisque $_COOKIE est le dernier, c’est la valeur qui se trouve dans $_REQUEST. Voir cette question.

$_ENV #

Un tableau associatif de variables passé au script courant via la méthode d’environnement.

Ces variables sont importées dans l’espace de noms global de PHP à partir de l’environnement sous lequel l’analyseur PHP s’exécute. Beaucoup sont fournis par le shell sous lequel PHP s’exécute et différents systèmes exécutent probablement différents types de shells, une liste définitive est impossible. Veuillez consulter la documentation de votre shell pour obtenir une liste des variables d’environnement définies.

Les autres variables d’environnement incluent les variables CGI, placées là, que PHP soit exécuté en tant que module serveur ou processeur CGI.

Tout ce qui est stocké dans $_ENV provient de l’environnement à partir duquel PHP s’exécute.

$_ENV n’est rempli que si php.ini le permet. Voir cette réponse pour plus d’informations sur la raison pour laquelle $_ENV n’est pas rempli.
