L’extension PDO (PHP Data Objects) permet aux développeurs de se connecter à de nombreux types de bases de données différents et d’exécuter des requêtes sur celles-ci de manière uniforme et orientée objet. manière.

Syntaxe #

  • PDO::LastInsertId()
  • PDO::LastInsertId($columnName) // certains pilotes ont besoin du nom de la colonne

Avertissement Ne manquez pas de vérifier les exceptions lors de l’utilisation de lastInsertId(). Il peut générer l’erreur suivante :

SQLSTATE IM001 : Le pilote ne supporte pas cette fonction

Voici comment vérifier correctement les exceptions à l’aide de cette méthode :

// Retrieving the last inserted id
$id = null;

try {
    $id = $pdo->lastInsertId(); // return value is an integer    
}
catch( PDOException $e ) {
    echo $e->getMessage();
}

Empêcher l’injection SQL avec des requêtes paramétrées #

L’injection SQL est une sorte d’attaque qui permet à un utilisateur malveillant de modifier la requête SQL en y ajoutant des commandes indésirables. Par exemple, le code suivant est vulnérable :

// Do not use this vulnerable code!
$sql = 'SELECT name, email, user_level FROM users WHERE userID = ' . $_GET['user'];
$conn->query($sql);

Cela permet à tout utilisateur de ce script de modifier notre base de données à sa guise. Par exemple, considérez la chaîne de requête suivante :

page.php?user=0;%20TRUNCATE%20TABLE%20users;

Cela fait ressembler notre exemple de requête à ceci

SELECT name, email, user_level FROM users WHERE userID = 0; TRUNCATE TABLE users;

Bien qu’il s’agisse d’un exemple extrême (la plupart des attaques par injection SQL ne visent pas à supprimer des données, et la plupart des fonctions d’exécution de requête PHP ne prennent pas en charge les requêtes multiples), il s’agit d’un exemple de la manière dont une attaque par injection SQL peut être rendue possible par l’assemblage imprudent de la requête. Malheureusement, de telles attaques sont très courantes et très efficaces car les codeurs ne prennent pas les précautions nécessaires pour protéger leurs données.

Pour empêcher l’injection SQL, les instructions préparées sont la solution recommandée. Au lieu de concaténer les données utilisateur directement à la requête, un espace réservé est utilisé à la place. Les données sont ensuite envoyées séparément, ce qui signifie qu’il n’y a aucune chance que le moteur SQL confond les données utilisateur avec un ensemble d’instructions.

Bien que le sujet ici soit PDO, veuillez noter que l’extension PHP MySQLi [prend également en charge les instructions préparées] (https://www.wikiod.com/fr/php/php-mysql#Instructions préparées dans MySQLi)

PDO prend en charge deux types d’espaces réservés (les espaces réservés ne peuvent pas être utilisés pour les noms de colonne ou de table, uniquement pour les valeurs) :

  1. Espaces réservés nommés. Deux-points (:), suivis d’un nom distinct (par exemple, :user)

    // using named placeholders
$sql = 'SELECT name, email, user_level FROM users WHERE userID = :user';
$prep = $conn->prepare($sql);
$prep->execute(['user' => $_GET['user']]); // associative array
$result = $prep->fetchAll();

  2. Espaces réservés de position SQL traditionnels, représentés par ? :

    // using question-mark placeholders
$sql = 'SELECT name, user_level FROM users WHERE userID = ? AND user_level = ?';
$prep = $conn->prepare($sql);
$prep->execute([$_GET['user'], $_GET['user_level']]); // indexed array
$result = $prep->fetchAll();

Si jamais vous avez besoin de changer dynamiquement les noms de table ou de colonne, sachez que c’est à vos propres risques de sécurité et une mauvaise pratique. Cependant, cela peut être fait par concaténation de chaînes. Une façon d’améliorer la sécurité de ces requêtes consiste à définir une table de valeurs autorisées et à comparer la valeur que vous souhaitez concaténer à cette table.

Sachez qu’il est important de définir le jeu de caractères de connexion via DSN uniquement, sinon votre application pourrait être sujette à une [vulnérabilité obscure] [emulate-prepares-injection] si un encodage étrange est utilisé. Pour les versions de PDO antérieures à 5.3.6, la définition du jeu de caractères via DSN n’est pas disponible et la seule option consiste donc à définir l’attribut PDO :: ATTR_EMULATE_PREPARES sur false sur la connexion juste après sa création.

$conn->setAttribute(PDO::ATTR_EMULATE_PREPARES, false);

Cela amène PDO à utiliser les instructions préparées natives du SGBD sous-jacent au lieu de simplement les émuler.

Cependant, sachez que PDO se repliera [silencieusement][pdo-fallback] pour émuler des instructions que MySQL ne peut pas préparer nativement : celles qu’il peut préparer sont [listées dans le manuel][mysql-prep-stmt] ([source][1] ).

[émuler-prépare-injection] : https://stackoverflow.com/questions/134099/are-pdo-prepared-statements-sufficient-to-prevent-sql-injection/12202218#12202218 [pdo-fallback] : https://github.com/php/php-src/blob/master/ext/pdo_mysql/mysql_driver.c#L210 [mysql-prep-stmt] : http://dev.mysql.com/doc/en/sql-syntax-prepared-statements.html [1] : https://stackoverflow.com/questions/134099/are-pdo-prepared-statements-sufficient-to-prevent-sql-injection/12202218#12202218

Connexion et récupération de PDO de base #

Depuis PHP 5.0, [PDO][1] est disponible en tant que couche d’accès à la base de données. Il est indépendant de la base de données, et donc l’exemple de code de connexion suivant devrait fonctionner pour n’importe laquelle [de ses bases de données prises en charge] [2] simplement en changeant le DSN.

// First, create the database handle

//Using MySQL (connection via local socket):
$dsn = "mysql:host=localhost;dbname=testdb;charset=utf8";

//Using MySQL (connection via network, optionally you can specify the port too):
//$dsn = "mysql:host=127.0.0.1;port=3306;dbname=testdb;charset=utf8";

//Or Postgres
//$dsn = "pgsql:host=localhost;port=5432;dbname=testdb;";

//Or even SQLite
//$dsn = "sqlite:/path/to/database"

$username = "user";
$password = "pass";
$db = new PDO($dsn, $username, $password);

// setup PDO to throw an exception if an invalid query is provided
$db->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

// Next, let's prepare a statement for execution, with a single placeholder
$query = "SELECT * FROM users WHERE class = ?";
$statement = $db->prepare($query);

// Create some parameters to fill the placeholders, and execute the statement
$parameters = [ "221B" ];
$statement->execute($parameters);

// Now, loop through each record as an associative array
while ($row = $statement->fetch(PDO::FETCH_ASSOC)) {
    do_stuff($row);
}

La fonction [prepare][3] crée un objet PDOStatement à partir de la chaîne de requête. L’exécution de la requête et la récupération des résultats sont effectuées sur cet objet retourné. En cas d’échec, la fonction renvoie “false” ou lève une “exception” (selon la configuration de la connexion PDO).

[1] : http://php.net/manual/en/intro.pdo.php [2] : http://php.net/manual/en/pdo.drivers.php [3] : http://php.net/manual/en/pdo.prepare.php

Transactions de base de données avec PDO #

Les transactions de base de données garantissent qu’un ensemble de modifications de données ne sera rendu permanent que si chaque instruction aboutit. Toute erreur de requête ou de code lors d’une transaction peut être détectée et vous avez alors la possibilité d’annuler les modifications tentées.

PDO fournit des méthodes simples pour démarrer, valider et annuler des transactions.

$pdo = new PDO(
    $dsn, 
    $username, 
    $password, 
    array(PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION)
);

try {
    $statement = $pdo->prepare("UPDATE user SET name = :name");

    $pdo->beginTransaction();

    $statement->execute(["name"=>'Bob']);
    $statement->execute(["name"=>'Joe']);

    $pdo->commit();
} 
catch (\Exception $e) {
    if ($pdo->inTransaction()) {
        $pdo->rollback();
        // If we got here our two data updates are not in the database
    }
    throw $e;
}

Au cours d’une transaction, toute modification de données effectuée n’est visible que pour la connexion active. Les instructions SELECT renverront les modifications modifiées même si elles ne sont pas encore validées dans la base de données.

Remarque : consultez la documentation du fournisseur de base de données pour plus de détails sur la prise en charge des transactions. Certains systèmes ne prennent pas du tout en charge les transactions. Certains prennent en charge les transactions imbriquées, d’autres non.

Exemple pratique d’utilisation de transactions avec PDO

Dans la section suivante est illustré un exemple pratique du monde réel où l’utilisation de transactions assure la cohérence de la base de données.

Imaginez le scénario suivant, supposons que vous construisiez un panier pour un site Web de commerce électronique et que vous décidiez de conserver les commandes dans deux tables de base de données. On a nommé ‘commandes’ avec les champs ‘id_commande’, ’nom’, ‘adresse’, ’telephone’ et ‘created_at’. Et un second nommé orders_products avec les champs order_id, product_id et quantity. Le premier tableau contient les métadonnées de la commande tandis que le second les produits réels qui ont été commandés.

Insérer une nouvelle commande dans la base de données

Pour insérer une nouvelle commande dans la base de données, vous devez faire deux choses. Vous devez d’abord INSÉRER un nouvel enregistrement dans la table commandes qui contiendra les métadonnées de la commande (nom, adresse, etc.). Ensuite, vous devez INSÉRER un enregistrement dans la table orders_products, pour chacun des produits inclus dans la commande.

Vous pouvez le faire en procédant comme suit :

// Insert the metadata of the order into the database
$preparedStatement = $db->prepare(
    'INSERT INTO `orders` (`name`, `address`, `telephone`, `created_at`)
     VALUES (:name, :address, :telephone, :created_at)'
);

$preparedStatement->execute([
    'name' => $name,
    'address' => $address,
    'telephone' => $telephone,
    'created_at' => time(),
]);

// Get the generated `order_id`
$orderId = $db->lastInsertId();

// Construct the query for inserting the products of the order
$insertProductsQuery = 'INSERT INTO `orders_products` (`order_id`, `product_id`, `quantity`) VALUES';

$count = 0;
foreach ( $products as $productId => $quantity ) {
    $insertProductsQuery .= ' (:order_id' . $count . ', :product_id' . $count . ', :quantity' . $count . ')';
    
    $insertProductsParams['order_id' . $count] = $orderId;
    $insertProductsParams['product_id' . $count] = $productId;
    $insertProductsParams['quantity' . $count] = $quantity;
    
    ++$count;
}

// Insert the products included in the order into the database
$preparedStatement = $db->prepare($insertProductsQuery);
$preparedStatement->execute($insertProductsParams);

Cela fonctionnera très bien pour insérer une nouvelle commande dans la base de données, jusqu’à ce que quelque chose d’inattendu se produise et que, pour une raison quelconque, la deuxième requête INSERT échoue. Si cela se produit, vous vous retrouverez avec une nouvelle commande dans la table “commandes”, qui n’aura aucun produit associé. Heureusement, le correctif est très simple, tout ce que vous avez à faire est de faire les requêtes sous la forme d’une seule transaction de base de données.

Insérer une nouvelle commande dans la base de données avec une transaction

Pour démarrer une transaction à l’aide de PDO, tout ce que vous avez à faire est d’appeler la méthode beginTransaction avant d’exécuter des requêtes sur votre base de données. Ensuite, vous apportez les modifications souhaitées à vos données en exécutant les requêtes INSERT et/ou UPDATE. Et enfin, vous appelez la méthode commit de l’objet PDO pour rendre les modifications permanentes. Jusqu’à ce que vous appeliez la méthode commit, chaque modification que vous avez apportée à vos données jusqu’à présent n’est pas encore permanente et peut être facilement annulée en appelant simplement la méthode rollback de l’objet PDO.

Sur l’exemple suivant est démontré l’utilisation des transactions pour insérer une nouvelle commande dans la base de données, tout en assurant par la même occasion la cohérence des données. Si l’une des deux requêtes échoue, toutes les modifications seront annulées.

// In this example we are using MySQL but this applies to any database that has support for transactions
$db = new PDO('mysql:host=' . $host . ';dbname=' . $dbname . ';charset=utf8', $username, $password);    

// Make sure that PDO will throw an exception in case of error to make error handling easier
$db->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

try {
    // From this point and until the transaction is being committed every change to the database can be reverted
    $db->beginTransaction();    
    
    // Insert the metadata of the order into the database
    $preparedStatement = $db->prepare(
        'INSERT INTO `orders` (`order_id`, `name`, `address`, `created_at`)
         VALUES (:name, :address, :telephone, :created_at)'
    );
    
    $preparedStatement->execute([
        'name' => $name,
        'address' => $address,
        'telephone' => $telephone,
        'created_at' => time(),
    ]);
    
    // Get the generated `order_id`
    $orderId = $db->lastInsertId();

    // Construct the query for inserting the products of the order
    $insertProductsQuery = 'INSERT INTO `orders_products` (`order_id`, `product_id`, `quantity`) VALUES';
    
    $count = 0;
    foreach ( $products as $productId => $quantity ) {
        $insertProductsQuery .= ' (:order_id' . $count . ', :product_id' . $count . ', :quantity' . $count . ')';
        
        $insertProductsParams['order_id' . $count] = $orderId;
        $insertProductsParams['product_id' . $count] = $productId;
        $insertProductsParams['quantity' . $count] = $quantity;
        
        ++$count;
    }
    
    // Insert the products included in the order into the database
    $preparedStatement = $db->prepare($insertProductsQuery);
    $preparedStatement->execute($insertProductsParams);
    
    // Make the changes to the database permanent
    $db->commit();
}
catch ( PDOException $e ) { 
    // Failed to insert the order into the database so we rollback any changes
    $db->rollback();
    throw $e;
}

PDO : connexion au serveur MySQL/MariaDB #

Il existe deux façons de se connecter à un serveur MySQL/MariaDB, selon votre infrastructure.

Connexion standard (TCP/IP) #

$dsn = 'mysql:dbname=demo;host=server;port=3306;charset=utf8';
$connection = new \PDO($dsn, $username, $password);

// throw exceptions, when SQL error is caused
$connection->setAttribute(\PDO::ATTR_ERRMODE, \PDO::ERRMODE_EXCEPTION);
// prevent emulation of prepared statements
$connection->setAttribute(\PDO::ATTR_EMULATE_PREPARES, false);

Étant donné que PDO a été conçu pour être compatible avec les anciennes versions de serveur MySQL (qui ne prenaient pas en charge les instructions préparées), vous devez explicitement désactiver l’émulation. Sinon, vous perdrez les avantages supplémentaires de la prévention des injections, qui sont généralement accordés en utilisant des déclarations préparées.

Un autre compromis de conception, que vous devez garder à l’esprit, est le comportement de gestion des erreurs par défaut. S’il n’est pas configuré autrement, PDO n’affichera aucune indication d’erreurs SQL.

Il est fortement recommandé de le définir en “mode exception”, car cela vous apporte des fonctionnalités supplémentaires lors de l’écriture d’abstractions de persistance (par exemple : avoir une exception, lors de la violation de la contrainte UNIQUE).

Prise de connexion #

$dsn = 'mysql:unix_socket=/tmp/mysql.sock;dbname=demo;charset=utf8';
$connection = new \PDO($dsn, $username, $password);

// throw exceptions, when SQL error is caused
$connection->setAttribute(\PDO::ATTR_ERRMODE, \PDO::ERRMODE_EXCEPTION);
// prevent emulation of prepared statements
$connection->setAttribute(\PDO::ATTR_EMULATE_PREPARES, false);

Sur les systèmes de type Unix, si le nom d’hôte est 'localhost', alors la connexion au serveur est établie via une socket de domaine.

PDO : obtenir le nombre de lignes affectées par une requête #

Nous commençons avec $db, une instance de la classe PDO. Après avoir exécuté une requête, nous voulons souvent déterminer le nombre de lignes qui ont été affectées par celle-ci. La méthode rowCount() du PDOStatement fonctionnera bien :

$query = $db->query("DELETE FROM table WHERE name = 'John'");
$count = $query->rowCount();

echo "Deleted $count rows named John";

REMARQUE : Cette méthode doit uniquement être utilisée pour déterminer le nombre de lignes affectées par les instructions INSERT, DELETE et UPDATE. Bien que cette méthode puisse également fonctionner pour les instructions SELECT, elle n’est pas cohérente dans toutes les bases de données.

PDO :: lastInsertId() #

Vous pouvez souvent avoir besoin d’obtenir la valeur d’ID incrémentée automatiquement pour une ligne que vous venez d’insérer dans votre table de base de données. Vous pouvez y parvenir avec la méthode lastInsertId().

// 1. Basic connection opening (for MySQL)
$host = 'localhost';
$database = 'foo';
$user = 'root'
$password = '';
$dsn = "mysql:host=$host;dbname=$database;charset=utf8";
$pdo = new PDO($dsn, $user, $password);

// 2. Inserting an entry in the hypothetical table 'foo_user'
$query = "INSERT INTO foo_user(pseudo, email) VALUES ('anonymous', 'anonymous@example.com')";
$query_success = $pdo->query($query);

// 3. Retrieving the last inserted id
$id = $pdo->lastInsertId(); // return value is an integer

Dans postgresql et oracle, il y a le mot-clé RETURNING, qui renvoie les colonnes spécifiées des lignes actuellement insérées/modifiées. Voici un exemple d’insertion d’une entrée :

// 1. Basic connection opening (for PGSQL)
$host = 'localhost';
$database = 'foo';
$user = 'root'
$password = '';
$dsn = "pgsql:host=$host;dbname=$database;charset=utf8";
$pdo = new PDO($dsn, $user, $password);

// 2. Inserting an entry in the hypothetical table 'foo_user'
$query = "INSERT INTO foo_user(pseudo, email) VALUES ('anonymous', 'anonymous@example.com') RETURNING id";
$statement = $pdo->query($query);

// 3. Retrieving the last inserted id
$id = $statement->fetchColumn();  // return the value of the id column of the new row in foo_user
