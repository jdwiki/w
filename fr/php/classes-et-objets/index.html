Les classes et les objets sont utilisés pour rendre votre code plus efficace et moins répétitif en regroupant des tâches similaires.

Une classe est utilisée pour définir les actions et la structure de données utilisées pour construire des objets. Les objets sont ensuite construits à l’aide de cette structure prédéfinie.

Syntaxe #

  • class <ClassName> [ extend <ParentClassName> ] [ implements <Interface1> [, <Interface2>, ... ] { } // Déclaration de classe
  • interface <NomInterface> [ étend <InterfaceParent1> [, <InterfaceParent2>, ...] ] { } // Déclaration d’interface
  • utiliser <Trait1> [, <Trait2>, ...] ; // Utiliser les traits
  • [ public | protégé | private ] [ static ] $<varName>; // Déclaration d’attribut
  • const <CONST_NAME>; // Déclaration constante
  • [ public | protégé | private ] [ static ] function <methodName>([args...]) { } // Déclaration de méthode

Classes et composants d’interface #

Les classes peuvent avoir des propriétés, des constantes et des méthodes.

  • Les Propriétés contiennent des variables dans la portée de l’objet. Ils peuvent être initialisés lors de la déclaration, mais uniquement s’ils contiennent une valeur primitive.

  • Les Constantes doivent être initialisées lors de la déclaration et ne peuvent contenir qu’une valeur primitive. Les valeurs constantes sont fixées au moment de la compilation et ne peuvent pas être affectées au moment de l’exécution.

  • Methods doit avoir un corps, même vide, sauf si la méthode est déclarée abstraite.

    class Foo { private $foo = ‘foo’; // OK private $baz = array(); // OK private $bar = new Bar(); // Error! }

Les interfaces ne peuvent pas avoir de propriétés, mais peuvent avoir des constantes et des méthodes.

  • Les constantes de l’interface doivent être initialisées lors de la déclaration et ne peuvent contenir qu’une valeur primitive. Les valeurs constantes sont fixées au moment de la compilation et ne peuvent pas être affectées au moment de l’exécution.

  • Les méthodes d’interface n’ont pas de corps.

    interface FooBar { const FOO_VALUE = ‘bla’; public function doAnything(); }

Constantes de classe #

Les constantes de classe fournissent un mécanisme pour conserver des valeurs fixes dans un programme. C’est-à-dire qu’ils fournissent un moyen de donner un nom (et la vérification associée au moment de la compilation) à une valeur telle que 3.14 ou "Apple". Les constantes de classe ne peuvent être définies qu’avec le mot clé const - la fonction [define][1] ne peut pas être utilisée dans ce contexte.

Par exemple, il peut être pratique d’avoir une représentation abrégée de la valeur de π tout au long d’un programme. Une classe avec des valeurs const fournit un moyen simple de conserver de telles valeurs.

class MathValues {
    const PI = M_PI;
    const PHI = 1.61803;
}

$area = MathValues::PI * $radius * $radius;

Les constantes de classe sont accessibles en utilisant l’opérateur double-point (appelé opérateur de résolution de portée) sur une classe, un peu comme les variables statiques. Contrairement aux variables statiques, cependant, les constantes de classe ont leurs valeurs fixées au moment de la compilation et ne peuvent pas être réaffectées (par exemple, MathValues::PI = 7 produirait une erreur fatale).

Les constantes de classe sont également utiles pour définir des éléments internes à une classe qui pourraient avoir besoin d’être modifiés ultérieurement (mais ne changez pas assez fréquemment pour justifier le stockage dans, par exemple, une base de données). Nous pouvons y faire référence en interne à l’aide du résolveur de portée “self” (qui fonctionne à la fois dans les implémentations instanciées et statiques)

class Labor {
    /** How long, in hours, does it take to build the item? */
    const LABOR_UNITS = 0.26;
    /** How much are we paying employees per hour? */
    const LABOR_COST = 12.75;

    public function getLaborCost($number_units) {
         return (self::LABOR_UNITS * self::LABOR_COST) * $number_units;
    }
}

Les constantes de classe ne peuvent contenir que des valeurs scalaires dans les versions < 5.6

Depuis PHP 5.6, nous pouvons utiliser des expressions avec des constantes, ce qui signifie que les instructions mathématiques et les chaînes avec concaténation sont des constantes acceptables

class Labor {
    /** How much are we paying employees per hour? Hourly wages * hours taken to make */
    const LABOR_COSTS = 12.75 * 0.26;

    public function getLaborCost($number_units) {
         return self::LABOR_COSTS * $number_units;
    }
}

Depuis PHP 7.0, les constantes déclarées avec define peuvent désormais contenir des tableaux.

define("BAZ", array('baz'));

Les constantes de classe sont utiles pour plus que simplement stocker des concepts mathématiques. Par exemple, si vous préparez une tarte, il peut être pratique d’avoir une seule classe “Pie” capable de prendre différentes sortes de fruits.

class Pie {
    protected $fruit;

    public function __construct($fruit) {
        $this->fruit = $fruit;
    }
}

Nous pouvons alors utiliser la classe Pie comme ceci

$pie = new Pie("strawberry");

Le problème qui se pose ici est que, lors de l’instanciation de la classe “Pie”, aucune indication n’est fournie quant aux valeurs acceptables. Par exemple, lors de la fabrication d’une tarte “boysenberry”, il peut être mal orthographié “boisenberry”. De plus, nous pourrions ne pas soutenir une tarte aux prunes. Au lieu de cela, il serait utile d’avoir une liste des types de fruits acceptables déjà définis quelque part où il serait logique de les rechercher. Dites une classe nommée “Fruit” :

class Fruit {
    const APPLE = "apple";
    const STRAWBERRY = "strawberry";
    const BOYSENBERRY = "boysenberry";
}

$pie = new Pie(Fruit::STRAWBERRY);

La liste des valeurs acceptables en tant que constantes de classe fournit une indication précieuse sur les valeurs acceptables qu’une méthode accepte. Cela garantit également que les fautes d’orthographe ne peuvent pas dépasser le compilateur. Alors que new Pie('aple') et new Pie('apple') sont acceptables pour le compilateur, new Pie(Fruit::APLE) produira une erreur de compilation.

Enfin, l’utilisation de constantes de classe signifie que la valeur réelle de la constante peut être modifiée en un seul endroit, et tout code utilisant la constante a automatiquement les effets de la modification.

Bien que la méthode la plus courante pour accéder à une constante de classe soit MyClass::CONSTANT_NAME, elle peut également être accessible par :

echo MyClass::CONSTANT;

$classname = "MyClass";
echo $classname::CONSTANT; // As of PHP 5.3.0

Les constantes de classe en PHP sont classiquement nommées toutes en majuscules avec des traits de soulignement comme séparateurs de mots, bien que tout nom d’étiquette valide puisse être utilisé comme nom de constante de classe.

Depuis PHP 7.1, les constantes de classe peuvent désormais être définies avec des visibilités différentes de la portée publique par défaut. Cela signifie que les constantes protégées et privées peuvent maintenant être définies pour empêcher les constantes de classe de fuir inutilement dans la portée publique (voir https://www.wikiod.com/fr/php/classes-et-objets#Visibilité des méthodes et des propriétés -propriété-visibilité). Par exemple:

class Something {
    const PUBLIC_CONST_A = 1;
    public const PUBLIC_CONST_B = 2;
    protected const PROTECTED_CONST = 3;
    private const PRIVATE_CONST = 4;
}

définir vs constantes de classe #

Bien que ce soit une construction valide :

function bar() { return 2; };

define('BAR', bar());

Si vous essayez de faire la même chose avec des constantes de classe, vous obtiendrez une erreur :

function bar() { return 2; };

class Foo {
    const BAR = bar(); // Error: Constant expression contains invalid operations
}

Mais tu peux faire :

function bar() { return 2; };

define('BAR', bar());

class Foo {
    const BAR = BAR; // OK
}

Pour plus d’informations, voir constantes dans le manuel.

Utilisation de ::class pour récupérer le nom de la classe #

PHP 5.5 a introduit la syntaxe ::class pour récupérer le nom complet de la classe, en tenant compte de la portée de l’espace de noms et des instructions use.

namespace foo;
use bar\Bar;
echo json_encode(Bar::class); // "bar\\Bar"
echo json_encode(Foo::class); // "foo\\Foo"
echo json_encode(\Foo::class); // "Foo"

Ce qui précède fonctionne même si les classes ne sont même pas définies (c’est-à-dire que cet extrait de code fonctionne seul).

Cette syntaxe est utile pour les fonctions qui nécessitent un nom de classe. Par exemple, il peut être utilisé avec class_exists pour vérifier qu’une classe existe. Aucune erreur ne sera générée, quelle que soit la valeur renvoyée dans cet extrait :

class_exists(ThisClass\Will\NeverBe\Loaded::class, false);

[1] : http://php.net/define

Cours abstraits #

Une classe abstraite est une classe qui ne peut pas être instanciée. Les classes abstraites peuvent définir des méthodes abstraites, qui sont des méthodes sans corps, seulement une définition :

abstract class MyAbstractClass {
    abstract public function doSomething($a, $b);
}

Les classes abstraites doivent être étendues par une classe enfant qui peut ensuite fournir l’implémentation de ces méthodes abstraites.

Le but principal d’une classe comme celle-ci est de fournir une sorte de modèle qui permet aux classes enfants d’hériter, “forçant” une structure à adhérer. Détaillons cela avec un exemple :

Dans cet exemple, nous allons implémenter une interface Worker. On définit d’abord l’interface :

interface Worker {
    public function run();
}

Pour faciliter le développement d’autres implémentations de Worker, nous allons créer une classe de travail abstraite qui fournit déjà la méthode run() de l’interface, mais spécifie certaines méthodes abstraites qui doivent être remplies par n’importe quelle classe enfant :

abstract class AbstractWorker implements Worker {
    protected $pdo;
    protected $logger;

    public function __construct(PDO $pdo, Logger $logger) {
        $this->pdo = $pdo;
        $this->logger = $logger;
    }

    public function run() {
        try {
            $this->setMemoryLimit($this->getMemoryLimit());
            $this->logger->log("Preparing main");
            $this->prepareMain();
            $this->logger->log("Executing main");
            $this->main();
        } catch (Throwable $e) {
            // Catch and rethrow all errors so they can be logged by the worker
            $this->logger->log("Worker failed with exception: {$e->getMessage()}");
            throw $e;
        }
    }

    private function setMemoryLimit($memoryLimit) {
        ini_set('memory_limit', $memoryLimit);
        $this->logger->log("Set memory limit to $memoryLimit");
    }

    abstract protected function getMemoryLimit();

    abstract protected function prepareMain();

    abstract protected function main();
}

Tout d’abord, nous avons fourni une méthode abstraite getMemoryLimit(). Toute classe s’étendant de AbstractWorker doit fournir cette méthode et renvoyer sa limite de mémoire. Le AbstractWorker définit ensuite la limite de mémoire et l’enregistre.

Deuxièmement, AbstractWorker appelle les méthodes prepareMain() et main(), après avoir enregistré qu’elles ont été appelées.

Enfin, tous ces appels de méthode ont été regroupés dans un bloc try-catch. Ainsi, si l’une des méthodes abstraites définies par la classe enfant lève une exception, nous intercepterons cette exception, la consignerons et la relancerons. Cela évite que toutes les classes enfants aient à implémenter cela elles-mêmes.

Définissons maintenant une classe enfant qui s’étend de AbstractWorker :

class TranscactionProcessorWorker extends AbstractWorker {
    private $transactions;

    protected function getMemoryLimit() {
        return "512M";
    }

    protected function prepareMain() {
        $stmt = $this->pdo->query("SELECT * FROM transactions WHERE processed = 0 LIMIT 500");
        $stmt->execute();
        $this->transactions = $stmt->fetchAll();
    }

    protected function main() {
        foreach ($this->transactions as $transaction) {
            // Could throw some PDO or MYSQL exception, but that is handled by the AbstractWorker
            $stmt = $this->pdo->query("UPDATE transactions SET processed = 1 WHERE id = {$transaction['id']} LIMIT 1");
            $stmt->execute();
        }
    }
}

Comme vous pouvez le voir, le TransactionProcessorWorker était plutôt facile à implémenter, car nous n’avions qu’à spécifier la limite de mémoire et nous soucier des actions réelles qu’il devait effectuer. Aucune gestion des erreurs n’est nécessaire dans le TransactionProcessorWorker car cela est géré dans le AbsractWorker.

Note importante #

Lors de l’héritage d’une classe abstraite, toutes les méthodes marquées abstraites dans la déclaration de classe du parent doit être définie par l’enfant (ou l’enfant lui-même doit également être marqué abstrait) ; de plus, ces méthodes doivent être définies avec le même (ou un moins visibilité restreinte). Par exemple, si la méthode abstraite est définie comme protégée, la mise en œuvre de la fonction doit être définie soit comme protégé ou public, mais pas privé.

Tiré de la [Documentation PHP pour l’abstraction de classe][1].

Si vous ne définissez pas les méthodes des classes abstraites parentes dans la classe enfant, une erreur PHP fatale comme celle-ci vous sera renvoyée.

Erreur fatale : La classe X contient 1 méthode abstraite et doit donc être déclaré abstrait ou implémenter les méthodes restantes (X::x) dans

[1] : http://php.net/manual/en/language.oop5.abstract.php

Liaison statique tardive #

Dans PHP 5.3+ et supérieur, vous pouvez utiliser [late static binding][1] pour contrôler à partir de quelle classe une propriété ou une méthode statique est appelée. Il a été ajouté pour surmonter le problème inhérent au résolveur de portée self ::. Prenez le code suivant

class Horse {
    public static function whatToSay() {
         echo 'Neigh!';
    }

    public static function speak() {
         self::whatToSay();
    }
}

class MrEd extends Horse {
    public static function whatToSay() {
         echo 'Hello Wilbur!';
    }
}

Vous vous attendriez à ce que la classe MrEd remplace la fonction parente whatToSay(). Mais lorsque nous exécutons cela, nous obtenons quelque chose d’inattendu

Horse::speak(); // Neigh!
MrEd::speak(); // Neigh!

Le problème est que self::whatToSay(); ne peut faire référence qu’à la classe Horse, ce qui signifie qu’il n’obéit pas à MrEd. Si nous passons au résolution de portée static::, nous n’avons pas ce problème. Cette nouvelle méthode indique à la classe d’obéir à l’instance qui l’appelle. Ainsi, nous obtenons l’héritage que nous attendons

class Horse {
    public static function whatToSay() {
         echo 'Neigh!';
    }

    public static function speak() {
         static::whatToSay(); // Late Static Binding
    }
}

Horse::speak(); // Neigh!
MrEd::speak(); // Hello Wilbur!

[1] : http://php.net/manual/en/language.oop5.late-static-bindings.php

Espace de noms et chargement automatique #

Techniquement, le chargement automatique fonctionne en exécutant un rappel lorsqu’une classe PHP est requise mais introuvable. Ces rappels tentent généralement de charger ces classes.

Généralement, le chargement automatique peut être compris comme la tentative de charger des fichiers PHP (en particulier les fichiers de classe PHP, où un fichier source PHP est dédié à une classe spécifique) à partir de chemins appropriés en fonction du nom complet (FQN) de la classe lorsqu’une classe est nécessaire .

Supposons que nous ayons ces classes :

Fichier de classe pour application\controllers\Base :

<?php
namespace application\controllers { class Base {...} }

Fichier de classe pour application\controllers\Control :

<?php
namespace application\controllers { class Control {...} }

Fichier de classe pour application\models\Page :

<?php
namespace application\models { class Page {...} }

Sous le dossier source, ces classes doivent être placées respectivement dans les chemins en tant que FQN :

  • Dossier d’origine
  • “applications”
    • controllers
      • Base.php
      • Control.php
    • models
      • Page.php

Cette approche permet de résoudre par programmation le chemin du fichier de classe en fonction du FQN, à l’aide de cette fonction :

function getClassPath(string $sourceFolder, string $className, string $extension = ".php") {
    return $sourceFolder . "/" . str_replace("\\", "/", $className) . $extension; // note that "/" works as a directory separator even on Windows
}

La fonction spl_autoload_register nous permet de charger une classe lorsque cela est nécessaire à l’aide d’une fonction définie par l’utilisateur :

const SOURCE_FOLDER = __DIR__ . "/src";
spl_autoload_register(function (string $className) {
    $file = getClassPath(SOURCE_FOLDER, $className);
    if (is_readable($file)) require_once $file;
});

Cette fonction peut être encore étendue pour utiliser des méthodes de chargement de secours :

const SOURCE_FOLDERS = [__DIR__ . "/src", "/root/src"]);
spl_autoload_register(function (string $className) {
    foreach(SOURCE_FOLDERS as $folder) {
        $extensions = [
            // do we have src/Foo/Bar.php5_int64?
            ".php" . PHP_MAJOR_VERSION . "_int" . (PHP_INT_SIZE * 8),
            // do we have src/Foo/Bar.php7?
            ".php" . PHP_MAJOR_VERSION,
            // do we have src/Foo/Bar.php_int64?
            ".php" . "_int" . (PHP_INT_SIZE * 8),
            // do we have src/Foo/Bar.phps?
            ".phps"
            // do we have src/Foo/Bar.php?
            ".php"
        ];
        foreach($extensions as $ext) {
            $path = getClassPath($folder, $className, $extension);
            if(is_readable($path)) return $path;
        }
    }
});

Notez que PHP n’essaie pas de charger les classes chaque fois qu’un fichier utilisant cette classe est chargé. Il peut être chargé au milieu d’un script, ou même dans les fonctions d’arrêt . C’est l’une des raisons pour lesquelles les développeurs, en particulier ceux qui utilisent le chargement automatique, doivent éviter de remplacer les fichiers source en cours d’exécution, en particulier dans les fichiers phar.

##Interfaces

Introduction #

Les interfaces sont des définitions des API publiques que les classes doivent implémenter pour satisfaire l’interface. Ils fonctionnent comme des “contrats”, spécifiant ce que fait un ensemble de sous-classes, mais pas comment ils le font.

La définition d’interface ressemble beaucoup à la définition de classe, en changeant le mot-clé class en interface :

interface Foo {

}

Les interfaces peuvent contenir des méthodes et/ou des constantes, mais pas d’attributs. Les constantes d’interface ont les mêmes restrictions que les constantes de classe. Les méthodes d’interface sont implicitement abstraites :

interface Foo {
    const BAR = 'BAR';

    public function doSomething($param1, $param2);
}

Remarque : les interfaces ne doivent pas déclarer de constructeurs ou de destructeurs, puisqu’il s’agit de détails d’implémentation au niveau de la classe.

La concrétisation #

Toute classe qui a besoin d’implémenter une interface doit le faire en utilisant le mot-clé implements. Pour cela, la classe doit fournir une implémentation pour chaque méthode déclarée dans l’interface, en respectant la même signature.

Une seule classe peut implémenter plus d’une interface à la fois.

interface Foo {
    public function doSomething($param1, $param2);
}

interface Bar {
    public function doAnotherThing($param1);
}


class Baz implements Foo, Bar {
    public function doSomething($param1, $param2) {
        // ...
    }

    public function doAnotherThing($param1) {
        // ...
    }
}

Lorsque les classes abstraites implémentent des interfaces, elles n’ont pas besoin d’implémenter toutes les méthodes. Toute méthode non implémentée dans la classe de base doit alors être implémentée par la classe concrète qui l’étend :

abstract class AbstractBaz implements Foo, Bar {
    // Partial implementation of the required interface...
    public function doSomething($param1, $param2) {
        // ...
    }
}

class Baz extends AbstractBaz {
    public function doAnotherThing($param1) {
        // ...
    }
}

Notez que la réalisation d’interface est une caractéristique héritée. Lors de l’extension d’une classe qui implémente une interface, vous n’avez pas besoin de la redéclarer dans la classe concrète, car elle est implicite.

Remarque : Avant PHP 5.3.9, une classe ne pouvait pas implémenter deux interfaces qui spécifiaient une méthode avec le même nom, car cela créerait une ambiguïté. Les versions plus récentes de PHP permettent ceci tant que les méthodes dupliquées ont la même signature[[1]][1].

Héritage #

Comme les classes, il est possible d’établir une relation d’héritage entre les interfaces, en utilisant le même mot-clé extends. La principale différence est que l’héritage multiple est autorisé pour les interfaces :

interface Foo {

}

interface Bar {

}

interface Baz extends Foo, Bar {

}

Exemples #

Dans l’exemple ci-dessous, nous avons un exemple d’interface simple pour un véhicule. Les véhicules peuvent avancer et reculer.

interface VehicleInterface {
    public function forward();

    public function reverse();

    ...
}

class Bike implements VehicleInterface {
    public function forward() {
        $this->pedal();
    }

    public function reverse() {
        $this->backwardSteps();
    }

    protected function pedal() {
        ...
    }

    protected function backwardSteps() {
        ...
    }

    ...
}

class Car implements VehicleInterface {
    protected $gear = 'N';

    public function forward() {
        $this->setGear(1);
        $this->pushPedal();
    }

    public function reverse() {
        $this->setGear('R');
        $this->pushPedal();
    }

    protected function setGear($gear) {
        $this->gear = $gear;
    }

    protected function pushPedal() {
        ...
    }

    ...
}

Ensuite, nous créons deux classes qui implémentent l’interface : Bike et Car. Bike et Car sont très différents en interne, mais les deux sont des véhicules et doivent implémenter les mêmes méthodes publiques que celles fournies par VehicleInterface.

Le typehinting permet aux méthodes et aux fonctions de demander des interfaces. Supposons que nous ayons une classe de garage de stationnement, qui contient des véhicules de toutes sortes.

class ParkingGarage {
    protected $vehicles = [];

    public function addVehicle(VehicleInterface $vehicle) {
        $this->vehicles[] = $vehicle;
    }
}

Étant donné que addVehicle nécessite un $vehicle de type VehicleInterface - pas une implémentation concrète - nous pouvons saisir à la fois des vélos et des voitures, que le ParkingGarage peut manipuler et utiliser.

[1] : http://php.net/manual/en/language.oop5.interfaces.php

Visibilité des méthodes et des propriétés #

Il existe trois types de visibilité que vous pouvez appliquer aux méthodes (fonctions de classe/objet) et aux propriétés (variables de classe/objet) au sein d’une classe, qui fournissent un contrôle d’accès pour la méthode ou la propriété à laquelle elles sont appliquées.

Vous pouvez en savoir plus à ce sujet dans la [Documentation PHP pour la visibilité de la POO][1].

Public #

Déclarer une méthode ou une propriété comme “public” permet d’accéder à la méthode ou à la propriété par :

  • La classe qui l’a déclaré.
  • Les classes qui étendent la classe déclarée.
  • Tout objet, classe ou code externe en dehors de la hiérarchie des classes.

Un exemple de cet accès “public” serait :

class MyClass {
    // Property
    public $myProperty = 'test';

    // Method
    public function myMethod() {
        return $this->myProperty;
    }
}

$obj = new MyClass();
echo $obj->myMethod();
// Out: test

echo $obj->myProperty;
// Out: test

Protégé #

Déclarer une méthode ou une propriété comme protected permet à la méthode ou à la propriété d’être accessible par :

  • La classe qui l’a déclaré.
  • Les classes qui étendent la classe déclarée.

Cela n’autorise pas les objets externes, les classes ou le code en dehors de la hiérarchie des classes à accéder à ces méthodes ou propriétés. Si quelque chose utilisant cette méthode/propriété n’y a pas accès, elle ne sera pas disponible et une erreur sera générée. Seules les instances du self déclaré (ou des sous-classes de celui-ci) y ont accès.

Un exemple de cet accès “protégé” serait :

class MyClass {
    protected $myProperty = 'test';

    protected function myMethod() {
        return $this->myProperty;
    }
}

class MySubClass extends MyClass {
    public function run() {
        echo $this->myMethod();
    }
}

$obj = new MySubClass();
$obj->run(); // This will call MyClass::myMethod();
// Out: test

$obj->myMethod(); // This will fail.
// Out: Fatal error: Call to protected method MyClass::myMethod() from context ''

L’exemple ci-dessus indique que vous ne pouvez accéder qu’aux éléments “protégés” dans sa propre portée. Essentiellement : “Ce qu’il y a dans la maison ne peut être accessible que de l’intérieur de la maison.”

Privé #

Déclarer une méthode ou une propriété comme “privée” permet d’accéder à la méthode ou à la propriété par :

  • La classe qui l’a déclaré Seulement (pas les sous-classes).

Une méthode ou une propriété “privée” n’est visible et accessible qu’au sein de la classe qui l’a créée.

Notez que les objets du même type auront accès aux membres privés et protégés les uns des autres même s’il ne s’agit pas des mêmes instances.

class MyClass {
    private $myProperty = 'test';

    private function myPrivateMethod() {
        return $this->myProperty;
    }

    public function myPublicMethod() {
        return $this->myPrivateMethod();
    }

    public function modifyPrivatePropertyOf(MyClass $anotherInstance) {
        $anotherInstance->myProperty = "new value";
    }
}

class MySubClass extends MyClass {
    public function run() {
        echo $this->myPublicMethod();
    }

    public function runWithPrivate() {
        echo $this->myPrivateMethod();
    }
}

$obj = new MySubClass();
$newObj = new MySubClass();

// This will call MyClass::myPublicMethod(), which will then call
// MyClass::myPrivateMethod();
$obj->run(); 
// Out: test

    
$obj->modifyPrivatePropertyOf($newObj);

$newObj->run();
// Out: new value

echo $obj->myPrivateMethod(); // This will fail.
// Out: Fatal error: Call to private method MyClass::myPrivateMethod() from context ''

echo $obj->runWithPrivate(); // This will also fail.
// Out: Fatal error: Call to private method MyClass::myPrivateMethod() from context 'MySubClass'

Comme indiqué, vous ne pouvez accéder à la méthode/propriété “private” qu’à partir de sa classe définie.

[1] : http://php.net/manual/en/language.oop5.visibility.php

Mot-clé final #

Déf : Le mot-clé Final empêche les classes enfants de remplacer une méthode en préfixant la définition par final. Si la classe elle-même est définie comme finale, elle ne peut pas être étendue

Méthode finale

class BaseClass {
   public function test() {
       echo "BaseClass::test() called\n";
   }
   
   final public function moreTesting() {
       echo "BaseClass::moreTesting() called\n";
   }
}

class ChildClass extends BaseClass {
   public function moreTesting() {
       echo "ChildClass::moreTesting() called\n";
   }
}
// Results in Fatal error: Cannot override final method BaseClass::moreTesting()

Classe finale :

final class BaseClass {
   public function test() {
       echo "BaseClass::test() called\n";
   }

   // Here it doesn't matter if you specify the function as final or not
   final public function moreTesting() {
       echo "BaseClass::moreTesting() called\n";
   }
}

class ChildClass extends BaseClass {
}
// Results in Fatal error: Class ChildClass may not inherit from final class (BaseClass)

Constantes finales : Contrairement à Java, le mot-clé final n’est pas utilisé pour les constantes de classe en PHP. Utilisez plutôt le mot clé const.

Pourquoi dois-je utiliser final ?

  1. Empêcher une chaîne d’héritage massive de malheur
  2. Encourager la composition
  3. Forcez le développeur à penser à l’API publique de l’utilisateur
  4. Forcer le développeur à réduire l’API publique d’un objet
  5. Une classe “finale” peut toujours être rendue extensible
  6. “étend” casse l’encapsulation
  7. Vous n’avez pas besoin de cette flexibilité
  8. Vous êtes libre de changer le code

Quand éviter final : Les classes finales ne fonctionnent efficacement que sous les hypothèses suivantes :

  1. Il existe une abstraction (interface) que la classe finale implémente
  2. Toute l’API publique de la classe finale fait partie de cette interface

Chargement automatique #

Personne ne veut « exiger » ou « inclure » ​​chaque fois qu’une classe ou un héritage est utilisé. Parce que cela peut être pénible et facile à oublier, PHP propose ce qu’on appelle le chargement automatique. Si vous utilisez déjà Composer, lisez à propos du [chargement automatique à l’aide de Composer][1].

Qu’est-ce que le chargement automatique ?

Le nom dit tout. Vous n’avez pas besoin d’obtenir le fichier dans lequel la classe demandée est stockée, mais PHP le charge automatiquement *automatiquement.

Comment puis-je faire cela en PHP de base sans code tiers ?

Il existe la fonction [__autoload][2], mais il est préférable d’utiliser [spl_autoload_register][3]. Ces fonctions seront prises en compte par PHP chaque fois qu’une classe n’est pas définie dans l’espace donné. Ainsi, l’ajout d’un chargement automatique à un projet existant ne pose aucun problème, car les classes définies (via require c’est-à-dire) fonctionneront comme avant. Par souci de précision, les exemples suivants utiliseront des fonctions anonymes, si vous utilisez PHP < 5.3, vous pouvez définir la fonction et passer son nom en argument à spl_autoload_register.

Exemples

spl_autoload_register(function ($className) {
    $path = sprintf('%s.php', $className);
    if (file_exists($path)) {
        include $path;
    } else {
        // file not found
    }
});

Le code ci-dessus essaie simplement d’inclure un nom de fichier avec le nom de la classe et l’extension “.php” en utilisant [sprintf][4]. Si FooBar doit être chargé, il regarde si FooBar.php existe et si c’est le cas l’inclut.

Bien sûr, cela peut être étendu pour répondre aux besoins individuels du projet. Si _ à l’intérieur d’un nom de classe est utilisé pour grouper, par ex. User_Post et User_Image font tous deux référence à User, les deux classes peuvent être conservées dans un dossier appelé “User” comme ceci :

spl_autoload_register(function ($className) {
    //                        replace _ by / or \ (depending on OS)
    $path = sprintf('%s.php', str_replace('_', DIRECTORY_SEPARATOR, $className) );
    if (file_exists($path)) {
        include $path;
    } else {
        // file not found
    }
});

La classe User_Post sera maintenant chargée depuis “User/Post.php”, etc.

spl_autoload_register peut être adapté à divers besoins. Tous vos fichiers avec des classes sont nommés “class.CLASSNAME.php” ? Aucun problème. Imbrications diverses (User_Post_Content => “User/Post/Content.php”) ? Pas de problème non plus.

Si vous souhaitez un mécanisme de chargement automatique plus élaboré - et que vous ne souhaitez toujours pas inclure Composer - vous pouvez travailler sans ajouter de bibliothèques tierces.

spl_autoload_register(function ($className) {
    $path = sprintf('%1$s%2$s%3$s.php',
        // %1$s: get absolute path
        realpath(dirname(__FILE__)),
        // %2$s: / or \ (depending on OS)
        DIRECTORY_SEPARATOR,
        // %3$s: don't wory about caps or not when creating the files
        strtolower(
            // replace _ by / or \ (depending on OS)
            str_replace('_', DIRECTORY_SEPARATOR, $className)
        )
    );

    if (file_exists($path)) {
        include $path;
    } else {
        throw new Exception(
            sprintf('Class with name %1$s not found. Looked in %2$s.',
                $className,
                $path
            )
        );
    }
});

En utilisant des chargeurs automatiques comme celui-ci, vous pouvez facilement écrire du code comme celui-ci :

require_once './autoload.php'; // where spl_autoload_register is defined

$foo = new Foo_Bar(new Hello_World());

Utilisation des classes:

class Foo_Bar extends Foo {}

class Hello_World implements Demo_Classes {}

Ces exemples incluront les classes de foo/bar.php, foo.php, hello/world.php et demo/classes.php.

[1] : https://www.wikiod.com/fr/php/gestionnaire-de-dependances-du-compositeur#Chargement automatique avec Composer [2] : https://secure.php.net/manual/function.autoload.php [3] : https://secure.php.net/manual/function.spl-autoload-register.php [4] : https://secure.php.net/sprintf

Liaison dynamique #

La liaison dynamique, également appelée ** remplacement de méthode ** est un exemple de ** polymorphisme d’exécution ** qui se produit lorsque plusieurs classes contiennent différentes implémentations de la même méthode, mais l’objet sur lequel la méthode sera appelée est * inconnu * jusqu’à exécution.

Ceci est utile si une certaine condition dicte quelle classe sera utilisée pour effectuer une action, où l’action porte le même nom dans les deux classes.

interface Animal {
    public function makeNoise();
}

class Cat implements Animal {
    public function makeNoise
    {
        $this->meow();
    }
    ...
}

class Dog implements Animal {
    public function makeNoise {
        $this->bark();
    }
    ...
}

class Person {
    const CAT = 'cat';
    const DOG = 'dog';

    private $petPreference;
    private $pet;

    public function isCatLover(): bool {
        return $this->petPreference == self::CAT;
    }

    public function isDogLover(): bool {
        return $this->petPreference == self::DOG;
    }

    public function setPet(Animal $pet) {
        $this->pet = $pet;
    }

    public function getPet(): Animal {
        return $this->pet;
    }
}

if($person->isCatLover()) {
    $person->setPet(new Cat());
} else if($person->isDogLover()) {
    $person->setPet(new Dog());
}

$person->getPet()->makeNoise();

Dans l’exemple ci-dessus, la classe Animal (Dog|Cat) qui makeNoise est inconnue jusqu’à l’exécution en fonction de la propriété dans la classe User.

Appel d’un constructeur parent lors de l’instanciation d’un enfant #

Un écueil courant des classes enfants est que, si votre parent et votre enfant contiennent tous les deux une méthode constructor(__construct()), seul le constructeur de la classe enfant s’exécutera. Il peut arriver que vous ayez besoin d’exécuter la méthode parent __construct() à partir de son enfant. Si vous avez besoin de le faire, vous devrez utiliser le [parent ::][1] scope resolutor :

parent::__construct();

Maintenant, exploiter cela dans une situation réelle ressemblerait à quelque chose comme :

class Foo {

    function __construct($args) { 
        echo 'parent'; 
    }

}

class Bar extends Foo {

    function __construct($args) {
        parent::__construct($args);
    }
}

Ce qui précède exécutera le parent __construct(), ce qui entraînera l’exécution de echo.

[1] : http://php.net/manual/en/keyword.parent.php

$this, self et static plus le singleton #

Utilisez $this pour faire référence à l’objet courant. Utilisez self pour faire référence au classe actuelle. En d’autres termes, utilisez $this->member pour les non-statiques membres, utilisez self::$member pour les membres statiques.

Dans l’exemple ci-dessous, sayHello() et sayGoodbye() utilisent self et `$cette différence peut être observée ici.

class Person {
    private $name;

    public function __construct($name) {
        $this->name = $name;
    }

    public function getName() {
        return $this->name;
    }

    public function getTitle() {
        return $this->getName()." the person";
    }

    public function sayHello() {
        echo "Hello, I'm ".$this->getTitle()."<br/>";
    }

    public function sayGoodbye() {
        echo "Goodbye from ".self::getTitle()."<br/>";
    }
}

class Geek extends Person {
    public function __construct($name) {
        parent::__construct($name);
    }

    public function getTitle() {
        return $this->getName()." the geek";
    }
}

$geekObj = new Geek("Ludwig");
$geekObj->sayHello();
$geekObj->sayGoodbye();

static fait référence à la classe de la hiérarchie sur laquelle vous avez appelé la méthode. Cela permet une meilleure réutilisation des propriétés de classe statiques lorsque les classes sont héritées.

Considérez le code suivant :

class Car {
    protected static $brand = 'unknown';
    
    public static function brand() {
         return self::$brand."\n";
    }
}

class Mercedes extends Car {
    protected static $brand = 'Mercedes';
}

class BMW extends Car {
    protected static $brand = 'BMW';
}

echo (new Car)->brand();
echo (new BMW)->brand();
echo (new Mercedes)->brand();

Cela ne produit pas le résultat souhaité :

inconnu inconnu inconnu

C’est parce que self fait référence à la classe Car chaque fois que la méthode brand() est appelée.

Pour faire référence à la bonne classe, vous devez utiliser static à la place :

class Car {
    protected static $brand = 'unknown';
    
    public static function brand() {
         return static::$brand."\n";
    }
}

class Mercedes extends Car {
    protected static $brand = 'Mercedes';
}

class BMW extends Car {
    protected static $brand = 'BMW';
}

echo (new Car)->brand();
echo (new BMW)->brand();
echo (new Mercedes)->brand();

Cela produit la sortie souhaitée :

inconnu BMW Mercedes

Voir aussi https://www.wikiod.com/fr/php/classes-et-objets#Liaison statique tardive

Le singleton #

Si vous avez un objet qui coûte cher à créer ou qui représente une connexion à une ressource externe que vous souhaitez réutiliser, c’est-à-dire une connexion à une base de données où il n’y a pas de regroupement de connexions ou un socket vers un autre système, vous pouvez utiliser le static et self mots-clés dans une classe pour en faire un singleton. Il existe de fortes opinions quant à savoir si le modèle singleton doit ou non être utilisé, mais il a ses utilisations.

class Singleton {
    private static $instance = null;

    public static function getInstance(){
        if(!isset(self::$instance)){
            self::$instance = new self();
        }
        
        return self::$instance;
    }
    
    private function __construct() {
        // Do constructor stuff
    }
}

Comme vous pouvez le voir dans l’exemple de code, nous définissons une propriété statique privée $instance pour contenir la référence de l’objet. Comme il s’agit d’une référence statique, cette référence est partagée par TOUS les objets de ce type.

La méthode getInstance() utilise une méthode connue sous le nom d’instanciation paresseuse pour retarder la création de l’objet au dernier moment possible, car vous ne voulez pas que des objets inutilisés traînent en mémoire et ne soient jamais destinés à être utilisés. Cela permet également d’économiser du temps et du processeur lors du chargement de la page sans avoir à charger plus d’objets que nécessaire. La méthode vérifie si l’objet est défini, le crée sinon et le renvoie. Cela garantit qu’un seul objet de ce type est jamais créé.

Nous définissons également le constructeur comme étant privé pour nous assurer que personne ne le crée avec le mot-clé “new” de l’extérieur. Si vous avez besoin d’hériter de cette classe, remplacez simplement les mots-clés private par protected.

Pour utiliser cet objet, il vous suffit d’écrire ce qui suit :

$singleton = Singleton::getInstance();

Maintenant, je vous implore d’utiliser l’injection de dépendances là où vous le pouvez et de viser des objets faiblement couplés, mais parfois ce n’est tout simplement pas raisonnable et le modèle singleton peut être utile.

Définir une classe de base #

Un objet en PHP contient des variables et des fonctions. Les objets appartiennent généralement à une classe, qui définit les variables et les fonctions que contiendront tous les objets de cette classe.

La syntaxe pour définir une classe est :

class Shape {
    public $sides = 0;
    
    public function description() {
        return "A shape with $this->sides sides.";
    }
}

Une fois qu’une classe est définie, vous pouvez créer une instance en utilisant :

$myShape = new Shape();

Les variables et les fonctions sur l’objet sont accessibles comme ceci :

$myShape = new Shape();
$myShape->sides = 6;

print $myShape->description(); // "A shape with 6 sides"

Constructeur #

Les classes peuvent définir une méthode spéciale __construct(), qui est exécutée dans le cadre de la création d’objet. Ceci est souvent utilisé pour spécifier les valeurs initiales d’un objet :

class Shape {
    public $sides = 0;
    
    public function __construct($sides) {
        $this->sides = $sides;
    }
    
    public function description() {
        return "A shape with $this->sides sides.";
    }
}

$myShape = new Shape(6);

print $myShape->description(); // A shape with 6 sides

Extension d’une autre classe #

Les définitions de classe peuvent étendre les définitions de classe existantes, en ajoutant de nouvelles variables et fonctions ainsi qu’en modifiant celles définies dans la classe parent.

Voici une classe qui étend l’exemple précédent :

class Square extends Shape {
    public $sideLength = 0;
    
    public function __construct($sideLength) {
       parent::__construct(4);
       
       $this->sideLength = $sideLength;
    }
    
    public function perimeter() {
        return $this->sides * $this->sideLength;
    }

    public function area() {
        return $this->sideLength * $this->sideLength;
    }
}

La classe Square contient des variables et un comportement pour la classe Shape et la classe Square :

$mySquare = new Square(10);

print $mySquare->description()/ // A shape with 4 sides

print $mySquare->perimeter() // 40

print $mySquare->area() // 100

Cours anonymes #

Des classes anonymes ont été introduites dans PHP 7 pour permettre la création rapide d’objets uniques. Ils peuvent prendre des arguments de constructeur, étendre d’autres classes, implémenter des interfaces et utiliser des traits comme le peuvent les classes normales.

Dans sa forme la plus basique, une classe anonyme ressemble à ceci :

new class("constructor argument") {
    public function __construct($param) {
        var_dump($param);
    }
}; // string(20) "constructor argument"

L’imbrication d’une classe anonyme à l’intérieur d’une autre classe ne lui donne pas accès aux méthodes ou propriétés privées ou protégées de cette classe externe. L’accès aux méthodes et propriétés protégées de la classe externe peut être obtenu en étendant la classe externe à partir de la classe anonyme. L’accès aux propriétés privées de la classe externe peut être obtenu en les transmettant au constructeur de la classe anonyme.

Par exemple:

class Outer {
    private $prop = 1;
    protected $prop2 = 2;

    protected function func1() {
        return 3;
    }

    public function func2() {
        // passing through the private $this->prop property
        return new class($this->prop) extends Outer {
            private $prop3;

            public function __construct($prop) {
                $this->prop3 = $prop;
            }

            public function func3() {
                // accessing the protected property Outer::$prop2
                // accessing the protected method Outer::func1()
                // accessing the local property self::$prop3 that was private from Outer::$prop
                return $this->prop2 + $this->func1() + $this->prop3;
            }
        };
    }
}

echo (new Outer)->func2()->func3(); // 6
