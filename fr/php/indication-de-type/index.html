Syntaxe #

  • fonction f(ClassName $param) {}
  • fonction f(bool $param) {}
  • fonction f(int $param) {}
  • fonction f(float $param) {}
  • fonction f(chaîne $param) {}
  • fonction f(self $param) {}
  • fonction f(appelable $param) {}
  • fonction f(tableau $param) {}
  • fonction f(?type_name $param) {}
  • fonction f() : nom_type {}
  • fonction f() : vide {}
  • fonction f() : ?type_name {}

Les indications de type ou déclarations de type sont une pratique de programmation défensive qui garantit que les paramètres d’une fonction sont d’un type spécifié. Ceci est particulièrement utile lors de l’indication de type pour une interface car cela permet à la fonction de garantir qu’un paramètre fourni aura les mêmes méthodes que celles requises dans l’interface.

Passer le type incorrect à une fonction d’indication de type conduira à une erreur fatale :

Erreur fatale : Erreur de type non interceptée : l’argument X transmis à foo() doit être de le type RequiredType, ProvidedType donné

Types scalaires d’indication de type, tableaux et callables #

La prise en charge des paramètres de tableau d’indication de type (et des valeurs de retour après PHP 7.1) a été ajoutée dans PHP 5.1 avec le mot clé array. Tous les tableaux de toutes dimensions et de tous types, ainsi que les tableaux vides, sont des valeurs valides.

La prise en charge des callables d’indication de type a été ajoutée dans PHP 5.4. Toute valeur is_callable() est valide pour les paramètres et les valeurs de retour suggérées callable, c’est-à-dire les objets Closure, les chaînes de nom de fonction et array(class_name|object, method_name).

Si une faute de frappe se produit dans le nom de la fonction telle qu’elle n’est pas is_callable(), un message d’erreur moins évident s’affichera :

Erreur fatale : Erreur de type non interceptée : l’argument 1 passé à foo() doit être du type appelable, chaîne/tableau donné

function foo(callable $c) {}
foo("count"); // valid
foo("Phar::running"); // valid
foo(["Phar", "running"); // valid
foo([new ReflectionClass("stdClass"), "getName"]); // valid
foo(function() {}); // valid

foo("no_such_function"); // callable expected, string given

Les méthodes non statiques peuvent également être transmises en tant qu’appelables au format statique, ce qui entraîne un avertissement de dépréciation et une erreur de niveau E_STRICT dans PHP 7 et 5 respectivement.

La visibilité de la méthode est prise en compte. Si le _contexte de la méthode avec le paramètre callable n’a pas accès à l’appelable fourni, cela se terminera comme si la méthode n’existait pas.

class Foo{
  private static function f(){
    echo "Good" . PHP_EOL;
  }

  public static function r(callable $c){
    $c();
  }
}

function r(callable $c){}

Foo::r(["Foo", "f"]);
r(["Foo", "f"]);

Production:

Erreur fatale : Erreur de type non interceptée : l’argument 1 passé à r() doit être appelable, tableau donné

La prise en charge des types scalaires d’indication de type a été ajoutée dans PHP 7. Cela signifie que nous gagnons en charge l’indication de type pour booleans, integers, floats et strings.

<?php

function add(int $a, int $b) {
    return $a + $b;
}

var_dump(add(1, 2)); // Outputs "int(3)"

Par défaut, PHP tentera de caster tout argument fourni pour correspondre à son indice de type. Changer l’appel à add(1.5, 2) donne exactement le même résultat, puisque le flottant 1.5 a été converti en int par PHP.

Pour arrêter ce comportement, il faut ajouter declare(strict_types=1); en haut de chaque fichier source PHP qui le requiert.

<?php

declare(strict_types=1);

function add(int $a, int $b) {
    return $a + $b;
}

var_dump(add(1.5, 2));

Le script ci-dessus produit maintenant une erreur fatale :

Erreur fatale : Erreur de type non interceptée : l’argument 1 passé à add() doit être de type entier, flottant donné

Une exception : les types spéciaux #

Certaines fonctions PHP peuvent retourner une valeur de type resource. Comme il ne s’agit pas d’un type scalaire, mais d’un type spécial, il n’est pas possible de le taper.

Par exemple, curl_init() renverra une ressource, ainsi que fopen(). Bien sûr, ces deux ressources ne sont pas compatibles entre elles. À cause de cela, PHP 7 lancera toujours la TypeError suivante lorsque le type indique explicitement resource :

TypeError : l’argument 1 passé à sample() doit être une instance de ressource, ressource donnée

Classes et interfaces d’indication de type #

L’indication de type pour les classes et les interfaces a été ajoutée dans PHP 5.

Indice de type de classe #

<?php

class Student
{
    public $name = 'Chris';
}

class School
{
    public $name = 'University of Edinburgh';
}

function enroll(Student $student, School $school)
{
    echo $student->name . ' is being enrolled at ' . $school->name;
}

$student = new Student();
$school = new School();

enroll($student, $school);

Le script ci-dessus affiche :

Chris est inscrit à l’Université d’Edimbourg

Indice de type d’interface #

<?php

interface Enrollable {};
interface Attendable {};

class Chris implements Enrollable
{
    public $name = 'Chris';
}

class UniversityOfEdinburgh implements Attendable
{
    public $name = 'University of Edinburgh';
}

function enroll(Enrollable $enrollee, Attendable $premises)
{
    echo $enrollee->name . ' is being enrolled at ' . $premises->name;
}

$chris = new Chris();
$edinburgh = new UniversityOfEdinburgh();

enroll($chris, $edinburgh);

L’exemple ci-dessus affiche la même chose qu’avant :

Chris est inscrit à l’Université d’Edimbourg

Conseils d’auto-type #

Le mot clé self peut être utilisé comme indice de type pour indiquer que la valeur doit être une instance de la classe qui déclare la méthode.

Conseils de type Nullable #

Paramètres #

L’indication de type Nullable a été ajoutée dans PHP 7.1 en utilisant l’opérateur ? avant l’indication de type.

function f(?string $a) {}
function g(string $a) {}

f(null); // valid
g(null); // TypeError: Argument 1 passed to g() must be of the type string, null given

Avant PHP 7.1, si un paramètre a une indication de type, il doit déclarer une valeur par défaut null pour accepter les valeurs nulles.

function f(string $a = null) {}
function g(string $a) {}

f(null); // valid
g(null); // TypeError: Argument 1 passed to g() must be of the type string, null given

Valeurs de retour #

En PHP 7.0, les fonctions avec un type de retour ne doivent pas retourner null.

En PHP 7.1, les fonctions peuvent déclarer un indice de type de retour nullable. Cependant, la fonction doit toujours retourner null, et non void (instructions de retour vides/non).

function f() : ?string {
    return null;
}

function g() : ?string {}
function h() : ?string {}

f(); // OK
g(); // TypeError: Return value of g() must be of the type string or null, none returned
h(); // TypeError: Return value of h() must be of the type string or null, none returned

Type d’indication d’objets génériques #

Étant donné que les objets PHP n’héritent d’aucune classe de base (y compris stdClass), il n’y a pas de support pour l’indication de type d’un type d’objet générique.

Par exemple, ce qui suit ne fonctionnera pas.

<?php

function doSomething(object $obj) {
    return $obj;
}

class ClassOne {}
class ClassTwo {}

$classOne= new ClassOne();
$classTwo= new ClassTwo();

doSomething($classOne);
doSomething($classTwo);

Et lancera une erreur fatale :

Erreur fatale : Erreur de type non interceptée : l’argument 1 passé à fairequelquechose() doit être une instance d’objet, instance de OperationOne donnée

Une solution consiste à déclarer une interface dégénérée qui ne définit aucune méthode et à ce que tous vos objets implémentent cette interface.

<?php

interface Object {}

function doSomething(Object $obj) {
    return $obj;
}

class ClassOne implements Object {}
class ClassTwo implements Object {}

$classOne = new ClassOne();
$classTwo = new ClassTwo();

doSomething($classOne);
doSomething($classTwo);

Type d’indication Pas de retour (Void) #

En PHP 7.1, le type de retour void a été ajouté. Bien que PHP n’ait pas de valeur “void” réelle, il est généralement entendu dans tous les langages de programmation qu’une fonction qui ne renvoie rien renvoie “void”. Cela ne doit pas être confondu avec le retour de null, car null est une valeur qui peut être renvoyée.

function lacks_return(): void {
    // valid
}

Notez que si vous déclarez un retour void, vous ne pouvez retourner aucune valeur ou vous obtiendrez une erreur fatale :

function should_return_nothing(): void {
    return null; // Fatal error: A void function must not return a value
}

Cependant, utiliser return pour quitter la fonction est valide :

function returns_nothing(): void {
    return; // valid
}
