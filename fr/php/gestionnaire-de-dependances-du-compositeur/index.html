Composer est le gestionnaire de dépendances PHP le plus couramment utilisé. C’est analogue à npm dans Node, pip pour Python ou NuGet pour .NET.

Syntaxe #

  • php path/to/composer.phar [commande] [options] [arguments]

Paramètres #

Paramètre Détails
licence Définit le type de licence que vous souhaitez utiliser dans le projet.
auteurs Définit les auteurs du projet, ainsi que les détails de l’auteur.
soutien Définit les e-mails d’assistance, le canal irc et divers liens.
exiger Définit les dépendances réelles ainsi que les versions de package.
nécessite-dev Définit les packages nécessaires au développement du projet.
suggérer Définit les suggestions de packages, c’est-à-dire les packages qui peuvent aider s’ils sont installés.
chargement automatique Définit les politiques de chargement automatique du projet.
autoload-dev Définit les règles de chargement automatique pour le développement du projet.

Le chargement automatique ne fonctionnera que pour les bibliothèques qui spécifient des informations de chargement automatique. La plupart des bibliothèques respectent et respecteront une norme telle que PSR-0 ou [PSR-4](http://www.php -fig.org/psr/psr-4/).

Liens utiles #

  • Packagist - Parcourez les packages disponibles (que vous pouvez installer avec Composer).
  • Documentation officielle
  • Guide de démarrage officiel

Quelques suggestions #

  1. Désactivez xdebug lors de l’exécution de Composer.
  2. N’exécutez pas Composer en tant que root. Les colis ne sont pas dignes de confiance.

Chargement automatique avec Composer #

Bien que composer fournisse un système pour gérer les dépendances des projets PHP (par exemple depuis Packagist), il peut également notamment servir d’autochargeur, en spécifiant où rechercher des espaces de noms spécifiques ou inclure des fichiers de fonctions génériques .

Cela commence par le fichier composer.json :

{
    // ...
    "autoload": {
        "psr-4": {
            "MyVendorName\\MyProject": "src/"
        },
        "files": [
            "src/functions.php"
        ]
    },
    "autoload-dev": {
        "psr-4": {
            "MyVendorName\\MyProject\\Tests": "tests/"
        }
    }
}

Ce code de configuration garantit que toutes les classes de l’espace de noms MyVendorName\MyProject sont mappées au répertoire src et toutes les classes de MyVendorName\MyProject\Tests au répertoire tests (par rapport à votre répertoire racine). Il inclura également automatiquement le fichier functions.php.

Après avoir mis ceci dans votre fichier composer.json, exécutez composer update dans un terminal pour que composer mette à jour les dépendances, le fichier de verrouillage et génère le fichier autoload.php. Lors du déploiement dans un environnement de production, vous utiliserez composer install --no-dev. Le fichier autoload.php se trouve dans le répertoire vendor qui doit être généré dans le répertoire où réside composer.json.

Vous devez “exiger” ce fichier tôt à un moment de la configuration du cycle de vie de votre application en utilisant une ligne similaire à celle ci-dessous.

require_once __DIR__ . '/vendor/autoload.php';

Une fois inclus, le fichier autoload.php se charge de charger toutes les dépendances que vous avez fournies dans votre fichier composer.json.

Quelques exemples de chemin de classe vers le mappage de répertoire :

  • MyVendorName\MyProject\Shapes\Square ➔ src/Shapes/Square.php.
  • MyVendorName\MyProject\Tests\Shapes\Square ➔ tests/Shapes/Square.php.

Qu’est-ce que Composer ? #

Composer est un gestionnaire de dépendances/paquets pour PHP. Il peut être utilisé pour installer, suivre et mettre à jour les dépendances de votre projet. Composer s’occupe également du chargement automatique des dépendances sur lesquelles votre application s’appuie, vous permettant d’utiliser facilement la dépendance dans votre projet sans vous soucier de les inclure en haut d’un fichier donné.

Les dépendances de votre projet sont répertoriées dans un fichier composer.json qui se trouve généralement à la racine de votre projet. Ce fichier contient des informations sur les versions requises des packages pour la production et le développement.

Un aperçu complet du schéma composer.json est disponible sur le site Web de Composer.

Ce fichier peut être modifié manuellement à l’aide de n’importe quel éditeur de texte ou automatiquement via la ligne de commande via des commandes telles que composer require <package> ou composer require-dev <package>.

Pour commencer à utiliser composer dans votre projet, vous devrez créer le fichier composer.json. Vous pouvez soit le créer manuellement, soit simplement exécuter composer init. Après avoir exécuté composer init dans votre terminal, il vous demandera quelques informations de base sur votre projet : Nom du package (fournisseur/package - par exemple laravel/laravel), Description - facultatif, Auteur et quelques autres informations comme la stabilité minimale, la licence et les packages requis.

La clé require dans votre fichier composer.json spécifie Composer de quels packages dépend votre projet. require prend un objet qui mappe les noms de packages (par exemple monolog/monolog) aux contraintes de version (par exemple *1.0.**).

{
    "require": {
        "composer/composer": "1.2.*"
    }
}

Pour installer les dépendances définies, vous devrez exécuter la commande composer install et elle trouvera ensuite les packages définis correspondant à la contrainte version fournie et les téléchargera dans le répertoire vendor. C’est une convention de mettre du code tiers dans un répertoire nommé vendor.

Vous remarquerez que la commande install a également créé un fichier composer.lock.

Un fichier composer.lock est automatiquement généré par Composer. Ce fichier est utilisé pour suivre les versions actuellement installées et l’état de vos dépendances. L’exécution de composer install installera les packages exactement dans l’état stocké dans le fichier de verrouillage.

Différence entre ‘composer install’ et ‘composer update’ #

mise à jour du compositeur #

composer update mettra à jour nos dépendances telles qu’elles sont spécifiées dans composer.json.

Par exemple, si notre projet utilise cette configuration :

"require": {
    "laravelcollective/html": "2.0.*"
}

En supposant que nous ayons effectivement installé la version 2.0.1 du paquet, l’exécution de composer update entraînera une mise à jour de ce paquet (par exemple vers 2.0.2, s’il a déjà été publié).

En détail, la “mise à jour du compositeur” :

  • Lire composer.json
  • Supprimez les packages installés qui ne sont plus nécessaires dans composer.json
  • Vérifier la disponibilité des dernières versions de nos requis paquets
  • Installer les dernières versions de nos packages
  • Mettez à jour composer.lock pour stocker la version des packages installés

installation du compositeur #

composer install installera toutes les dépendances comme spécifié dans le fichier composer.lock à la version spécifiée (verrouillée), sans rien mettre à jour.

En détail:

  • Lire le fichier composer.lock
  • Installez les packages spécifiés dans le fichier composer.lock

Quand installer et quand mettre à jour #

  • composer update est principalement utilisé dans la phase de ‘développement’, pour mettre à jour nos packages de projet.

  • composer install est principalement utilisé dans la ‘phase de déploiement’ pour installer notre application sur un serveur de production ou sur un environnement de test, en utilisant les mêmes dépendances stockées dans le fichier composer.lock créé par composer update.

Avantages de l’utilisation de Composer #

Composer suit les versions de packages que vous avez installées dans un fichier appelé composer.lock, qui est destiné à être validé pour le contrôle de version, de sorte que lorsque le projet sera cloné à l’avenir, il suffit d’exécuter composer install pour télécharger et installer toutes les dépendances du projet.

Composer traite les dépendances PHP par projet. Cela permet d’avoir facilement plusieurs projets sur une machine qui dépendent de versions distinctes d’un package PHP.

Composer suit les dépendances uniquement destinées aux environnements de développement

composer require --dev phpunit/phpunit

Composer fournit un chargeur automatique, ce qui facilite grandement la prise en main de n’importe quel package. Par exemple, après avoir installé [Goutte][1] avec composer require fabpot/goutte, vous pouvez immédiatement commencer à utiliser Goutte dans un nouveau projet :

<?php

require __DIR__ . '/vendor/autoload.php';

$client = new Goutte\Client();

// Start using Goutte

Composer vous permet de mettre facilement à jour un projet vers la dernière version autorisée par votre composer.json. PAR EXEMPLE. composer update fabpot/goutte, ou pour mettre à jour chacune des dépendances de votre projet : composer update.

[1] : https://github.com/FriendsOfPHP/Goutte

Installation #

Vous pouvez installer Composer localement, dans le cadre de votre projet, ou globalement en tant qu’exécutable à l’échelle du système.

Localement #

Pour l’installer, exécutez ces commandes dans votre terminal.

php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"
# to check the validity of the downloaded installer, check here against the SHA-384:
# https://composer.github.io/pubkeys.html
php composer-setup.php
php -r "unlink('composer-setup.php');"

Cela téléchargera composer.phar (un fichier d’archive PHP) dans le répertoire courant. Vous pouvez maintenant exécuter php composer.phar pour utiliser Composer, par ex.

php composer.phar install

Globalement #

Pour utiliser Composer globalement, placez le fichier composer.phar dans un répertoire faisant partie de votre PATH

mv composer.phar /usr/local/bin/composer

Maintenant, vous pouvez utiliser composer n’importe où au lieu de php composer.phar, par ex.

composer install
