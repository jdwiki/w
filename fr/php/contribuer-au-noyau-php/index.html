Contribuer au noyau PHP

PHP est un projet open source, et en tant que tel, n’importe qui peut y contribuer. D’une manière générale, il existe deux façons de contribuer au noyau PHP :

  • Correction de bogues
  • Ajouts de fonctionnalités

Avant de contribuer, cependant, il est important de comprendre comment les versions de PHP sont gérées et publiées afin que les corrections de bogues et les demandes de fonctionnalités puissent cibler la bonne version de PHP. Les modifications développées peuvent être soumises sous forme de demande d’extraction au [référentiel PHP Github][1]. Des informations utiles pour les développeurs peuvent être trouvées dans la section [“Get Involved” du site PHP.net][2] et le [#externals forum][3].

Contribuer aux corrections de bogues #

Pour ceux qui cherchent à commencer à contribuer au noyau, il est généralement plus facile de commencer par la correction des bogues. Cela aide à se familiariser avec les composants internes de PHP avant d’essayer d’apporter des modifications plus complexes au noyau qu’une fonctionnalité nécessiterait.

En ce qui concerne le processus de gestion des versions, les corrections de bogues doivent cibler la version PHP la moins affectée, tout en restant prise en charge. C’est cette version que les demandes d’extraction de correction de bugs doivent cibler. À partir de là, un membre interne peut fusionner le correctif dans la bonne branche, puis le fusionner vers les versions ultérieures de PHP si nécessaire.

Pour ceux qui cherchent à commencer à résoudre des bogues, une liste de rapports de bogues peut être trouvée sur [bugs.php.net][4].

Contribuer aux ajouts de fonctionnalités #

PHP suit un processus RFC lors de l’introduction de nouvelles fonctionnalités et des modifications importantes apportées au langage. Les RFC sont votés par les membres de php.net et doivent obtenir soit une majorité simple (50% + 1) soit une super majorité (2/3 + 1) du total des votes. Une super majorité est requise si le changement affecte la langue elle-même (comme l’introduction d’une nouvelle syntaxe), sinon seule une majorité simple est requise.

Avant que les RFC puissent être soumises au vote, elles doivent subir une période de discussion d’au moins 2 semaines sur la liste de diffusion officielle de PHP. Une fois cette période terminée, et qu’il n’y a plus de problèmes ouverts avec le RFC, il peut alors être déplacé vers le vote, qui doit durer au moins 1 semaine.

Si un utilisateur souhaite réactiver une RFC précédemment rejetée, il ne peut le faire que dans l’une des deux circonstances suivantes :

  • 6 mois se sont écoulés depuis le vote précédent
  • L’auteur ou les auteurs apportent suffisamment de changements substantiels au RFC qui affecteraient probablement le résultat du vote si le RFC devait être soumis à nouveau au vote.

Les personnes qui ont le privilège de voter seront soit des contributeurs à PHP lui-même (et ont donc des comptes php.net), soit des représentants de la communauté PHP. Ces représentants sont choisis par ceux qui ont des comptes php.net, et seront soit des développeurs principaux de projets basés sur PHP, soit des participants réguliers aux discussions internes.

Lors de la soumission de nouvelles idées pour proposition, il est presque toujours nécessaire que le proposant rédige, à tout le moins, un correctif de preuve de concept. En effet, sans implémentation, la suggestion devient simplement une autre demande de fonctionnalité qui ne sera probablement pas satisfaite dans un avenir proche.

Un guide détaillé de ce processus peut être trouvé sur la page officielle [Comment créer une RFC][5].

Versions #

Les versions majeures de PHP n’ont pas de cycle de publication défini, et elles peuvent donc être publiées à la discrétion de l’équipe interne (chaque fois qu’elle le juge nécessaire pour une nouvelle version majeure). Les versions mineures, en revanche, sont publiées chaque année.

Avant chaque version de PHP (majeure, mineure ou correctif), une série de versions candidates (RC) sont mises à disposition. PHP n’utilise pas de RC comme le font d’autres projets (c’est-à-dire que si un RC n’a pas de problèmes avec lui, faites-en la prochaine version finale). Au lieu de cela, il les utilise comme une forme de versions bêta finales, où généralement un nombre défini de RC est décidé avant la publication de la version finale.

Gestion des versions #

PHP tente généralement de suivre le versioning sémantique dans la mesure du possible. En tant que tel, la rétrocompatibilité (BC) doit être maintenue dans les versions mineures et correctives du langage. Les fonctionnalités et les modifications qui préservent BC doivent cibler les versions mineures (et non les versions de correctifs). Si une fonctionnalité ou un changement a le potentiel de casser BC, alors ils devraient cibler la prochaine version majeure de PHP (X.y.z) à la place.

Chaque version mineure de PHP (x.Y.z) bénéficie de deux ans de support général (appelé “support actif”) pour tous les types de corrections de bogues. Une année supplémentaire est ajoutée pour le support de sécurité, où seuls les correctifs liés à la sécurité sont appliqués. Une fois les trois années écoulées, la prise en charge de cette version de PHP est complètement abandonnée. Une liste des [versions PHP actuellement prises en charge peut être trouvée sur php.net][6].

## Configuration d’un environnement de développement de base Le code source de PHP est hébergé sur [GitHub][1]. Pour construire à partir des sources, vous devrez d’abord extraire une copie de travail du code.

mkdir /usr/local/src/php-7.0/
cd /usr/local/src/php-7.0/
git clone -b PHP-7.0 https://github.com/php/php-src .

Si vous souhaitez ajouter une fonctionnalité, il est préférable de créer votre propre branche.

git checkout -b my_private_branch

Enfin, configurez et compilez PHP

./buildconf
./configure
make
make test
make install

Si la configuration échoue en raison de dépendances manquantes, vous devrez utiliser le système de gestion des packages de votre système d’exploitation pour les installer (par exemple, yum, apt, etc.) ou les télécharger et les compiler à partir des sources.

[1] : https://github.com/php/php-src
