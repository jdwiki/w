Notez qu’appeler session_start() même si la session a déjà démarré entraînera un avertissement PHP.

Session_start() Options #

À partir de PHP Sessions, nous pouvons passer un tableau avec [options php.ini basées sur la session][1] à la fonction session_start.

Exemple

<?php
   if (version_compare(PHP_VERSION, '7.0.0') >= 0) {
       // php >= 7 version
       session_start([
          'cache_limiter' => 'private',
          'read_and_close' => true,
       ]);
   } else {
       // php < 7 version
       session_start();
   }
?>

Cette fonctionnalité introduit également un nouveau paramètre php.ini nommé session.lazy_write, qui par défaut est true et signifie que les données de session ne sont réécrites que si elles changent.

Référencement : https://wiki.php.net/rfc/session-lock-ini

[1] : http://php.net/manual/en/ini.list.php

Verrouillage de session #

Comme nous le savons tous, PHP écrit les données de session dans un fichier côté serveur. Lorsqu’une demande est faite au script php qui démarre la session via session_start(), PHP verrouille ce fichier de session, ce qui bloque/attend les autres demandes entrantes pour le même session_id, à cause de quoi les autres demandes resteront bloquées sur session_start() jusqu’à ou à moins que le fichier de session verrouillé ne soit pas libéré

Le fichier de session reste verrouillé jusqu’à ce que le script soit terminé ou que la session soit fermée manuellement. Pour éviter cette situation *c.-à-d. pour éviter que plusieurs requêtes ne soient bloquées *, nous pouvons démarrer la session et fermer la session, ce qui libérera le verrou du fichier de session et permettra de continuer les requêtes restantes.

// php < 7.0 
// start session 
session_start();

// write data to session
$_SESSION['id'] = 123; // session file is locked, so other requests are blocked

// close the session, release lock
session_write_close();

Maintenant, on pensera si la session est fermée comment nous allons lire les valeurs de la session, embellir même après la fermeture de la session, la session est toujours disponible. Ainsi, nous pouvons toujours lire les données de session.

echo $_SESSION['id'];    // will output  123

Dans php >= 7.0, nous pouvons avoir une session read_only, une session read_write et une session lazy_write, il n’est donc peut-être pas nécessaire d’utiliser session_write_close()

Manipulation des données de session #

La variable $_SESSION est un tableau, et vous pouvez la récupérer ou la manipuler comme un tableau normal.

<?php
// Starting the session
session_start();

// Storing the value in session
$_SESSION['id'] = 342;

// conditional usage of session values that may have been set in a previous session
if(!isset($_SESSION["login"])) {
    echo "Please login first";
    exit;
}
// now you can use the login safely
$user = $_SESSION["login"];

// Getting a value from the session data, or with default value, 
//     using the Null Coalescing operator in PHP 7
$name = $_SESSION['name'] ?? 'Anonymous';

Voir également https://www.wikiod.com/fr/php/manipulation-dun-tableau pour plus de référence sur la façon de travailler sur un tableau.

Notez que si vous stockez un objet dans une session, il ne peut être récupéré correctement que si vous disposez d’un chargeur automatique de classe ou si vous avez déjà chargé la classe. Sinon, l’objet sortira sous le type __PHP_Incomplete_Class, ce qui peut entraîner ultérieurement des [plantages] (http://stackoverflow.com/q/1055728/3990767). Voir https://www.wikiod.com/fr/php/classes-et-objets#Espace de noms et chargement automatique à propos du chargement automatique.

Avertissement: #

Les données de session peuvent être piratées. Ceci est décrit dans : [Pro PHP Security : From Application Security Principles to the Implementation of XSS Defense - Chapter 7: Preventing Session Hijacking][1] Il peut donc être fortement recommandé de ne jamais stocker d’informations personnelles dans $_SESSION . Il s’agirait notamment des numéros de carte de crédit, des identifiants émis par le gouvernement et des mots de passe ; mais s’étendrait également à des données moins supposées telles que ** noms **, ** e-mails **, ** numéros de téléphone **, etc., ce qui permettrait à un pirate de se faire passer pour / de compromettre un utilisateur légitime. En règle générale, utilisez des valeurs sans valeur/non personnelles, telles que des identifiants numériques, dans les données de session.

[1] : https://books.google.com.au/books?id=EUc6NlZRDqcC&pg=PA97#v=onepage&q&f=false

Détruire une session entière #

Si vous avez une session que vous souhaitez détruire, vous pouvez le faire avec session_destroy()

/*
    Let us assume that our session looks like this:
    Array([firstname] => Jon, [id] => 123)

    We first need to start our session:
*/
session_start();

/*
    We can now remove all the values from the `SESSION` superglobal:
    If you omitted this step all of the global variables stored in the 
    superglobal would still exist even though the session had been destroyed.
*/
$_SESSION = array();

// If it's desired to kill the session, also delete the session cookie.
// Note: This will destroy the session, and not just the session data!
if (ini_get("session.use_cookies")) {
    $params = session_get_cookie_params();
    setcookie(session_name(), '', time() - 42000,
        $params["path"], $params["domain"],
        $params["secure"], $params["httponly"]
    );
}

//Finally we can destroy the session:
session_destroy();

L’utilisation de session_destroy() est différente de l’utilisation de quelque chose comme $_SESSION = array(); qui supprimera toutes les valeurs stockées dans le superglobal SESSION mais ne détruira pas la version réellement stockée de la session.

Remarque : Nous utilisons $_SESSION = array(); au lieu de session_unset() car le manuel stipule :

Utilisez uniquement session_unset() pour le code obsolète plus ancien qui n’utilise pas $_SESSION.

Nom de la session #

Vérifier si des cookies de session ont été créés #

Le nom de la session est le nom du cookie utilisé pour stocker les sessions. Vous pouvez l’utiliser pour détecter si des cookies pour une session ont été créés pour l’utilisateur :

if(isset($_COOKIE[session_name()])) {
    session_start();
}

Notez que cette méthode n’est généralement pas utile à moins que vous ne souhaitiez vraiment pas créer de cookies inutilement.

Changer le nom de la session #

Vous pouvez mettre à jour le nom de la session en appelant session_name().

//Set the session name
session_name('newname');
//Start the session
session_start();

Si aucun argument n’est fourni dans session_name(), le nom de la session actuelle est renvoyé.

Il ne doit contenir que des caractères alphanumériques ; il doit être court et descriptif (c’est-à-dire pour les utilisateurs dont les avertissements de cookies sont activés). Le nom de la session ne peut pas être composé uniquement de chiffres, au moins une lettre doit être présente. Sinon, un nouvel identifiant de session est généré à chaque fois.

Démarrage sécurisé de la session sans erreur #

De nombreux développeurs ont ce problème lorsqu’ils travaillent sur d’énormes projets, en particulier s’ils travaillent sur certains CMS modulaires sur des plugins, des addons, des composants, etc. si la session est démarrée. Si la session n’existe pas, je démarre la session en toute sécurité. Si la session existe, rien ne se passe.

if (version_compare(PHP_VERSION, '7.0.0') >= 0) {
    if(session_status() == PHP_SESSION_NONE) {
        session_start(array(
          'cache_limiter' => 'private',
          'read_and_close' => true,
       ));
    }
}
else if (version_compare(PHP_VERSION, '5.4.0') >= 0)
{
    if (session_status() == PHP_SESSION_NONE) {
        session_start();
    }
}
else
{
    if(session_id() == '') {
        session_start();
    }
}

Cela peut vous aider beaucoup à éviter l’erreur session_start.
