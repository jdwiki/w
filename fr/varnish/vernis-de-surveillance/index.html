Utilisez varnishstat pour surveiller les métriques numériques d’une instance Varnish en cours d’exécution. Son emplacement sera différent en fonction de votre installation. L’exécution de varnishstat -1 affichera toutes les métriques dans un format simple grep.

D’autres utilitaires sont disponibles pour surveiller l’état actuel du vernis et la journalisation : varnishtop, varnishlog etc.

Métriques client - trafic entrant #

Les métriques client couvrent le trafic entre le client et le cache Varnish.

  • sess_conn - Nombre cumulé de connexions.
  • client_req - Nombre cumulé de requêtes client.
  • sess_dropped - Connexions interrompues en raison d’une file d’attente pleine.

Surveillez sess_conn et client_req pour suivre le volume du trafic - augmente-t-il ou diminue-t-il, augmente-t-il, etc. Des changements soudains peuvent indiquer des problèmes.

Surveillez sess_dropped pour voir si le cache supprime des sessions. Si c’est le cas, vous devrez peut-être augmenter thread_pool_max.

varnishstat -1 | grep "sess_conn\|client_req \|sess_dropped"
MAIN.sess_conn              62449574         3.38 Sessions accepted
MAIN.client_req            184697229         9.99 Good client requests received
MAIN.sess_dropped                  0         0.00 Sessions dropped for thread

Performances du cache #

La mesure de performance la plus importante est peut-être le hitrate.

Varnish achemine ses requêtes entrantes comme ceci :

  • Hash, une requête pouvant être mise en cache. Cela peut être “hit” ou “miss” selon l’état du cache.
  • Hitpass, une requête non cacheable.

Un hachage avec un miss et un hitpass sera extrait du backend du serveur et livré. Un hachage avec un “hit” sera délivré directement depuis le cache.

Métriques à surveiller :

  • cache_hit - Nombre de hachages avec un hit dans le cache.

  • cache_miss - Nombre de hachages manquants dans le cache.

  • cache_hitpass - Nombre de hitpass comme ci-dessus.

    varnishstat -1 | grep “cache_hit |cache_miss |cache_hitpass” MAIN.cache_hit 99032838 5.36 Cache hits MAIN.cache_hitpass 0 0.00 Cache hits for pass MAIN.cache_miss 42484195 2.30 Cache misses

Calculez le hitrate réel comme ceci :

cache_hit / (cache_hit + cache_miss)

Dans cet exemple, le hitrate est de 0,7 ou 70 %. Vous voulez garder cela aussi élevé que possible. 70% est un nombre décent. Vous pouvez améliorer le hitrate en augmentant la mémoire et en personnalisant votre vcl. Surveillez également les grands changements dans votre hitrate.

Surveillance des objets mis en cache #

Vous surveillez les objets mis en cache pour voir à quelle fréquence ils expirent et s’ils sont « nuked ».

  • n_expired - Nombre d’objets expirés.

  • n_lru_nuked - Derniers objets nucléaires récemment utilisés. Nombre d’objets détruits (supprimés) du cache en raison d’un manque d’espace.

    varnishstat -1 | grep “n_expired|n_lru_nuked” MAIN.n_expired 42220159 . Number of expired objects MAIN.n_lru_nuked 264005 . Number of LRU nuked objects

Celui à surveiller ici est n_lru_nuked, si le taux augmente (le taux, pas seulement le nombre), votre cache pousse les objets de plus en plus vite à cause du manque d’espace. Vous devez augmenter la taille du cache.

Le n_expired dépend davantage de votre application. Une durée de vie plus longue diminuera ce nombre mais en revanche ne renouvellera pas les objets aussi souvent. De plus, le cache peut nécessiter plus de taille.

## Suivi des threads Vous devez suivre certaines métriques de threads pour surveiller votre Varnish Cache. Est-ce qu’il manque de ressources du système d’exploitation ou fonctionne-t-il bien.

  • threads - Nombre de threads dans tous les pools.

  • threads_created - Nombre de threads créés.

  • threads_failed - Nombre de fois que Varnish n’a pas réussi à créer un thread.

  • threads_limited - Nombre de fois où Varnish a été forcé de ne pas créer de thread car il était au maximum.

  • thread_queue_len - Longueur actuelle de la file d’attente. Nombre de requêtes en attente d’un thread.

  • sess_queued - Nombre de fois qu’il n’y avait pas de threads disponibles, donc une requête a dû être mise en file d’attente.

    varnishstat -1 | grep “threads|thread_queue_len|sess_queued” MAIN.threads 100 . Total number of threads MAIN.threads_limited 1 0.00 Threads hit max MAIN.threads_created 3715 0.00 Threads created MAIN.threads_destroyed 3615 0.00 Threads destroyed MAIN.threads_failed 0 0.00 Thread creation failed MAIN.thread_queue_len 0 . Length of session queue MAIN.sess_queued 2505 0.00 Sessions queued for thread

Si thread_queue_len n’est pas 0, cela signifie que Varnish n’a plus de ressources et a commencé à mettre les requêtes en file d’attente. Cela réduira les performances de ces requêtes. Vous devez rechercher pourquoi.

Faites également attention à threads_failed. Si cela augmente, cela signifie que votre serveur manque de ressources d’une manière ou d’une autre. L’augmentation des nombres dans threads_limited signifie que vous devrez peut-être augmenter vos serveurs thread_pool_max.

Surveillance des métriques backend #

Il existe un certain nombre de métriques décrivant la communication entre Varnish et ses backends.

Les métriques les plus importantes ici pourraient être celles-ci :

  • backend_busy - Nombre de statuts http 5xx reçus par un backend. Avec VCL, vous pouvez configurer Varnish pour essayer un autre backend si cela se produit.

  • backend_fail - Nombre de fois où Varnish n’a pas pu se connecter au backend. Cela peut avoir un certain nombre de causes (pas de connexion TCP, long temps jusqu’au premier octet, long temps entre les octets). Si cela se produit, votre backend n’est pas sain.

  • backend_unhealthy - Nombre de fois que Varnish n’a pas pu “pinger” le backend (il n’a pas répondu avec une réponse HTTP 200.

    varnishstat -1 | grep “backend_” MAIN.backend_conn 86913481 4.70 Backend conn. success MAIN.backend_unhealthy 0 0.00 Backend conn. not attempted MAIN.backend_busy 0 0.00 Backend conn. too many MAIN.backend_fail 7 0.00 Backend conn. failures MAIN.backend_reuse 0 0.00 Backend conn. reuses MAIN.backend_toolate 0 0.00 Backend conn. was closed MAIN.backend_recycle 0 0.00 Backend conn. recycles MAIN.backend_retry 0 0.00 Backend conn. retry MAIN.backend_req 86961073 4.70 Backend requests made
