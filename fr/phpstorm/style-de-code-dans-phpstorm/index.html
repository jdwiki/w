Définir le style de code pour un projet #

PhpStorm propose des paramètres par défaut pour le style de code pour un grand nombre de langages basés sur les meilleures pratiques et les normes communes. Mais vous pouvez personnaliser le style pour chaque langue sur une base par projet dans les paramètres PhpStorm > Editor > Code Style.

[![Paramètres de style de code][1]][1]

Schémas

Les schémas sont des collections de directives et de paramètres de style de code. Vous pouvez sélectionner un schéma pour un projet et il sera appliqué instantanément. Il existe également un schéma de projet qui n’est disponible que lorsqu’un projet est ouvert. Le schéma de projet n’enregistre pas les directives dans les paramètres généraux de l’utilisateur, mais dans les propres paramètres du projet.

Si vous cliquez sur le bouton * Gérer *, vous pouvez ajouter un nouveau schéma pour configurer rapidement un nouvel ensemble de directives de style de code. La boîte * Gérer * peut également être utilisée pour exporter ou importer des schémas, ce qui est très utile si vous souhaitez partager des schémas avec vos amis ou collègues.

[![Boîte de gestion du style de code][2]][2]

Changer le style pour une langue spécifique #

Pour modifier les paramètres de style de code pour une langue spécifique, cliquez simplement sur la langue disponible dans la barre latérale. Une page de paramètres différente pour chaque langue vous sera alors présentée.

Par exemple, le langage PHP aura des paramètres pour les tabulations et les espaces, les accolades ou les documents PHP.

[![Style de code pour PHP][3]][3]

Si vous voulez savoir ce que signifie chaque paramètre différent, vous pouvez les rechercher dans la documentation officielle.

Définir les styles de code à partir d’une norme prédéfinie telle que PSR-2 #

PhpStorm est déjà livré avec de nombreux schémas de langage prédéfinis basés sur des directives et des normes de style de code communes telles que [PSR-2][1]. Il existe une sorte de fonctionnalité cachée dans les pages de paramètres de style de code où vous pouvez importer ces normes et les définir comme votre configuration actuelle. Pour ce faire, choisissez simplement votre langage de codage dans le panneau de gauche. Ensuite, il y a un petit lien en haut à droite appelé Set from…

En cliquant sur ce lien, PhpStorm vous présentera une petite fenêtre contextuelle dans laquelle vous pourrez choisir parmi les normes prédéfinies sous l’onglet Style prédéfini.

Appliquer un style de code spécifié pour un projet à plusieurs membres de l’équipe #

Actuellement, il n’existe pas de méthode en un clic pour appliquer les directives de style de code au sein d’une équipe, mais il existe deux méthodes pour s’assurer qu’un certain style de code est appliqué à votre produit.

Importer des schémas de style de code PhpStorm #

La première solution, la plus simple, consiste à configurer un schéma de style de code sur votre propre instance PhpStorm, à exporter le schéma sur un lecteur portable ou un lecteur réseau et à importer le schéma sur toutes les machines de développement. De cette façon, un développeur peut facilement utiliser les raccourcis clavier Cmd+Shift+L (MacOS) ou Ctl+< kbd>Alt+L (Windows / Linux) pour formater automatiquement le code source complet.

Reformatage du code source dans PhpStorm 2016 et versions ultérieures

Les inconvénients Malheureusement, il n’y a aucun moyen de vérifier si un développeur a vraiment appliqué le reformatage du code. Vous devrez vous fier aux promesses des développeurs de s’occuper du reformatage.

Vérifications automatisées du format de code avec un serveur CI #

Une façon très stricte de contrôler le formatage du code source consiste à implémenter un serveur d’intégration continue comme Jenkins qui est capable de vérifier si le code source correspond à un style de code prédéfini.

Supposons qu’un développeur travaille sur une nouvelle fonctionnalité sur sa propre branche de développement et souhaite pousser ses modifications vers le référentiel principal. Tout d’abord, il envoie les modifications à sa propre branche où la nouvelle fonctionnalité sera vérifiée par le serveur CI. Si la vérification échoue parce que le code n’est pas formaté correctement, le développeur en sera informé afin qu’il puisse corriger les problèmes.

Il existe différentes façons de configurer la qualité du code et les vérifications de formatage avec tous les différents serveurs d’intégration. Par conséquent, expliquer comment configurer un serveur avec des vérifications doit être fait dans les balises correspondantes.
