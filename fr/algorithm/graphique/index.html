Un graphique est une collection de points et de lignes reliant certains sous-ensembles (éventuellement vides) d’entre eux. Les points d’un graphe sont appelés sommets du graphe, “nœuds” ou simplement “points”. De même, les lignes reliant les sommets d’un graphe sont appelées arêtes de graphe, “arcs” ou “lignes”.

Un graphe G peut être défini comme une paire (V,E), où V est un ensemble de sommets, et E est un ensemble d’arêtes entre les sommets E ⊆ {(u,v) | u, v ∈ V}.

Les graphes sont une structure mathématique qui modélise des ensembles d’objets qui peuvent ou non être connectés à des membres d’ensembles d’arêtes ou de liens.

Un graphe peut être décrit à travers deux ensembles différents d’objets mathématiques :

  • Un ensemble de sommets.
  • Un ensemble d’arêtes qui relient des paires de sommets.

Les graphes peuvent être orientés ou non orientés.

  • Les graphes orientés contiennent des arêtes qui “se connectent” dans un seul sens.
  • Les graphes non orientés ne contiennent que des arêtes qui connectent automatiquement deux sommets ensemble dans les deux sens.

Stockage des graphiques (matrice d’adjacence) #

Pour stocker un graphe, deux méthodes sont courantes :

  • Matrice de contiguïté
  • Liste de contiguïté

Une matrice d’adjacence est une matrice carrée utilisée pour représenter un graphe fini. Les éléments de la matrice indiquent si des paires de sommets sont adjacentes ou non dans le graphe.

Adjacent signifie « à côté ou à côté de quelque chose d’autre » ou être à côté de quelque chose. Par exemple, vos voisins sont adjacents à vous. En théorie des graphes, si nous pouvons aller au nœud B à partir du nœud A, nous pouvons dire que le nœud B est adjacent au nœud A. Nous allons maintenant apprendre à stocker quels nœuds sont adjacents à quel nœud via la matrice d’adjacence. Cela signifie que nous allons représenter quels nœuds partagent un bord entre eux. Ici, matrice signifie tableau 2D.

[![Graphique et matrice d’adjacence][1]][1]

Ici vous pouvez voir un tableau à côté du graphique, c’est notre matrice de contiguïté. Ici Matrix[i][j] = 1 représente qu’il y a une arête entre i et j. S’il n’y a pas d’arête, nous mettons simplement Matrix[i][j] = 0.

Ces arêtes peuvent être pondérées, comme cela peut représenter la distance entre deux villes. Ensuite, nous mettrons la valeur dans Matrix[i][j] au lieu de mettre 1.

Le graphique décrit ci-dessus est bidirectionnel ou non dirigé, cela signifie que si nous pouvons aller au nœud 1 depuis le nœud 2, nous pouvons également aller au nœud 2 depuis le nœud 1. Si le graphique était Directed, alors il y aurait eu un signe de flèche sur un côté du graphique. Même dans ce cas, nous pourrions le représenter à l’aide d’une matrice de contiguïté.

[![Matrice d’adjacence du graphe pondéré orienté][2]][2]

Nous représentons les nœuds qui ne partagent pas le bord par infini. Une chose à remarquer est que, si le graphe n’est pas orienté, la matrice devient symétrique.

Le pseudo-code pour créer la matrice :

Procedure AdjacencyMatrix(N):    //N represents the number of nodes
Matrix[N][N]
for i from 1 to N
    for j from 1 to N
        Take input -> Matrix[i][j]
    endfor
endfor

Nous pouvons également remplir la matrice en utilisant cette méthode courante :

Procedure AdjacencyMatrix(N, E):    // N -> number of nodes
Matrix[N][E]                        // E -> number of edges
for i from 1 to E
    input -> n1, n2, cost
    Matrix[n1][n2] = cost
    Matrix[n2][n1] = cost
endfor

Pour les graphes orientés, nous pouvons supprimer la ligne Matrix[n2][n1] = cost.

Les inconvénients de l’utilisation de la matrice d’adjacence :

La mémoire est un énorme problème. Peu importe le nombre d’arêtes, nous aurons toujours besoin d’une matrice de taille N * N où N est le nombre de nœuds. S’il y a 10 000 nœuds, la taille de la matrice sera de 4 * 10 000 * 10 000 environ 381 mégaoctets. C’est une énorme perte de mémoire si l’on considère graphiques qui ont quelques arêtes.

Supposons que nous voulions savoir à quel nœud nous pouvons aller à partir d’un nœud u. Nous devrons vérifier toute la rangée de u, ce qui prend beaucoup de temps.

Le seul avantage est que nous pouvons facilement trouver la connexion entre les nœuds u-v et leur coût à l’aide de la matrice d’adjacence.

Code Java implémenté à l’aide du pseudo-code ci-dessus :

import java.util.Scanner;
 
public class Represent_Graph_Adjacency_Matrix 
{
    private final int vertices;
    private int[][] adjacency_matrix;
 
    public Represent_Graph_Adjacency_Matrix(int v) 
    {
        vertices = v;
        adjacency_matrix = new int[vertices + 1][vertices + 1];
    }
 
    public void makeEdge(int to, int from, int edge) 
    {
        try 
        {
            adjacency_matrix[to][from] = edge;
        }
        catch (ArrayIndexOutOfBoundsException index) 
        {
            System.out.println("The vertices does not exists");
        }
    }
 
    public int getEdge(int to, int from) 
    {
        try 
        {
            return adjacency_matrix[to][from];
        }
        catch (ArrayIndexOutOfBoundsException index) 
        {
            System.out.println("The vertices does not exists");
        }
        return -1;
    }
 
    public static void main(String args[]) 
    {
        int v, e, count = 1, to = 0, from = 0;
        Scanner sc = new Scanner(System.in);
        Represent_Graph_Adjacency_Matrix graph;
        try 
        {
            System.out.println("Enter the number of vertices: ");
            v = sc.nextInt();
            System.out.println("Enter the number of edges: ");
            e = sc.nextInt();
 
            graph = new Represent_Graph_Adjacency_Matrix(v);
 
            System.out.println("Enter the edges: <to> <from>");
            while (count <= e) 
            {
                to = sc.nextInt();
                from = sc.nextInt();
 
                graph.makeEdge(to, from, 1);
                count++;
            }
 
            System.out.println("The adjacency matrix for the given graph is: ");
            System.out.print("  ");
            for (int i = 1; i <= v; i++)
                System.out.print(i + " ");
            System.out.println();
 
            for (int i = 1; i <= v; i++) 
            {
                System.out.print(i + " ");
                for (int j = 1; j <= v; j++) 
                    System.out.print(graph.getEdge(i, j) + " ");
                System.out.println();
            }
 
        }
        catch (Exception E) 
        {
            System.out.println("Somthing went wrong");
        }
 
        sc.close();
    }
}

Introduction à la théorie des graphes #

La théorie des graphes est l’étude des graphes, qui sont des structures mathématiques utilisées pour modéliser des relations par paires entre des objets.

Saviez-vous que presque tous les problèmes de la planète Terre peuvent être convertis en problèmes de Routes et de Villes, et résolus ? La théorie des graphes a été inventée il y a de nombreuses années, avant même l’invention de l’ordinateur. Leonhard Euler a écrit un article sur les Sept ponts de Königsberg qui est considéré comme le premier article de théorie des graphes. Depuis lors, les gens se sont rendu compte que si nous pouvons convertir n’importe quel problème en ce problème Ville-Route, nous pouvons le résoudre facilement par la théorie des graphes.

La théorie des graphes a de nombreuses applications. L’une des applications les plus courantes consiste à trouver la distance la plus courte entre une ville et une autre. Nous savons tous que pour atteindre votre PC, cette page Web a dû parcourir de nombreux routeurs à partir du serveur. La théorie des graphes l’aide à découvrir les routeurs qui devaient être traversés. Pendant la guerre, quelle rue doit être bombardée pour déconnecter la capitale des autres, cela aussi peut être découvert en utilisant la théorie des graphes.

Apprenons d’abord quelques définitions de base sur la théorie des graphes.

Graphique:

Disons que nous avons 6 villes. Nous les marquons comme 1, 2, 3, 4, 5, 6. Maintenant, nous connectons les villes qui ont des routes entre elles.

[![Connexion entre les villes][1]][1]

Il s’agit d’un graphique simple où certaines villes sont représentées avec les routes qui les relient. Dans la théorie des graphes, nous appelons chacune de ces villes Node ou Vertex et les routes sont appelées Edge. Le graphe est simplement une connexion de ces nœuds et arêtes.

Un nœud peut représenter beaucoup de choses. Dans certains graphiques, les nœuds représentent des villes, certains représentent des aéroports, certains représentent un carré dans un échiquier. Edge représente la relation entre chaque nœud. Cette relation peut être le temps pour aller d’un aéroport à un autre, les déplacements d’un cavalier d’une case à toutes les autres cases etc.

[![Déplacements du chevalier à partir d’un seul point][2]][2]

                                             Chemin du chevalier sur un échiquier

En termes simples, un Node représente n’importe quel objet et Edge représente la relation entre deux objets.

Nœud adjacent :

Si un nœud A partage une arête avec le nœud B, alors B est considéré comme adjacent à A. En d’autres termes, si deux nœuds sont directement connectés, ils sont appelés nœuds adjacents. Un nœud peut avoir plusieurs nœuds adjacents.

Graphique dirigé et non dirigé :

Dans les graphes orientés, les arêtes ont des signes de direction d’un côté, cela signifie que les arêtes sont unidirectionnelles. D’autre part, les arêtes des graphes non orientés ont des signes de direction des deux côtés, cela signifie qu’ils sont * bidirectionnels *. Habituellement, les graphes non orientés sont représentés sans signe de part et d’autre des arêtes.

Supposons qu’il y ait une fête en cours. Les personnes du groupe sont représentées par des nœuds et il y a un avantage entre deux personnes si elles se serrent la main. Alors ce graphique n’est pas orienté car toute personne A serre la main de la personne B si et seulement si B serre également la main de A. En revanche, si les bords d’une personne A à une autre personne B correspondent à l’admiration B de A, alors ce graphique est dirigé, car l’admiration n’est pas nécessairement réciproque . Le premier type de graphe est appelé un * graphe non orienté * et les arêtes sont appelées * arêtes non orientées * tandis que le dernier type de graphe est appelé un * graphe orienté * et les arêtes sont appelées * arêtes dirigées . *

Graphique pondéré et non pondéré :

Un graphe pondéré est un graphe dans lequel un nombre (le poids) est attribué à chaque arête. Ces poids peuvent représenter par exemple des coûts, des longueurs ou des capacités, selon le problème à résoudre. [![Graphique pondéré][3]][3]

Un graphique non pondéré est tout simplement le contraire. Nous supposons que le poids de toutes les arêtes est le même (vraisemblablement 1).

Chemin:

Un chemin représente une façon d’aller d’un nœud à un autre. Il consiste en une séquence d’arêtes. Il peut y avoir plusieurs chemins entre deux nœuds. [![Graphique de chemin][4]][4]

Dans l’exemple ci-dessus, il existe deux chemins de A à D. A->B, B->C, C->D est un chemin. Le coût de ce chemin est 3 + 4 + 2 = 9. Encore une fois, il y a un autre chemin A->D. Le coût de ce chemin est de 10. Le chemin qui coûte le moins cher est appelé chemin le plus court.

Diplôme:

Le degré d’un sommet est le nombre d’arêtes qui lui sont reliées. S’il y a une arête qui se connecte au sommet aux deux extrémités (une boucle) est comptée deux fois.

Dans les graphes orientés, les nœuds ont deux types de degrés :

  • In-degree : Le nombre d’arêtes qui pointent vers le nœud.
  • Out-degree : Le nombre d’arêtes qui pointent du nœud vers d’autres nœuds.

Pour les graphes non orientés, ils sont simplement appelés degrés.

[![Degrés d’un graphe][5]][5]

Stockage des graphiques (liste d’adjacence) #

Liste d’adjacence est une collection de listes non ordonnées utilisées pour représenter un graphe fini. Chaque liste décrit l’ensemble des voisins d’un sommet dans un graphe. Il faut moins de mémoire pour stocker des graphiques.

Voyons un graphe, et sa matrice d’adjacence :[![Adjacency Matrix and It’s Graph][1]][1]

Maintenant, nous créons une liste en utilisant ces valeurs.

[![Liste de contiguïté][2]][2]

C’est ce qu’on appelle la liste de contiguïté. Il montre quels nœuds sont connectés à quels nœuds. Nous pouvons stocker ces informations à l’aide d’un tableau 2D. Mais nous coûtera la même mémoire que Adjacency Matrix. Au lieu de cela, nous allons utiliser de la mémoire allouée dynamiquement pour stocker celle-ci.

De nombreuses langues prennent en charge Vector ou List que nous pouvons utiliser pour stocker la liste de contiguïté. Pour ceux-ci, nous n’avons pas besoin de spécifier la taille de la Liste. Il suffit de spécifier le nombre maximum de nœuds.

Le pseudo-code sera :

Procedure Adjacency-List(maxN, E):       // maxN denotes the maximum number of nodes
edge[maxN] = Vector()                    // E denotes the number of edges
for i from 1 to E
    input -> x, y                        // Here x, y denotes there is an edge between x, y
    edge[x].push(y)
    edge[y].push(x)
end for
Return edge

Puisque celui-ci est un graphe non orienté, s’il y a une arête de x à y, il y a aussi une arête de y à x. S’il s’agissait d’un graphe orienté, nous omettrions le second. Pour les graphiques pondérés, nous devons également stocker le coût. Nous allons créer un autre vecteur ou liste nommé cost[] pour les stocker. Le pseudo-code :

Procedure Adjacency-List(maxN, E):
edge[maxN] = Vector()
cost[maxN] = Vector()
for i from 1 to E
    input -> x, y, w
    edge[x].push(y)
    cost[x].push(w)
end for
Return edge, cost

À partir de celui-ci, nous pouvons facilement connaître le nombre total de nœuds connectés à n’importe quel nœud, et quels sont ces nœuds. Cela prend moins de temps que la matrice d’adjacence. Mais si nous avions besoin de savoir s’il y a un bord entre u et v, cela aurait été plus facile si nous avions gardé une matrice de contiguïté.

Tri topologique #

Un ordre topologique, ou un tri topologique, ordonne les sommets dans un graphe acyclique orienté sur une ligne, c’est-à-dire dans une liste, tel que tous les bords vont de gauche à droite. Un tel ordre ne peut exister si le graphique contient un cycle dirigé, car il n’y a aucun moyen de continuer à avancer sur une ligne et de revenir quand même à l’endroit d’où vous êtes parti.

Formellement, dans un graphe G = (V, E), alors un ordre linéaire de tous ses sommets est telle que si G contient une arête (u, v) ∈ E du sommet u au sommet v alors u précède v dans l’ordre.

Il est important de noter que chaque DAG a au moins un tri topologique.

Il existe des algorithmes connus pour construire un ordre topologique de n’importe quel DAG en temps linéaire, un exemple est :

  1. Appelez depth_first_search(G) pour calculer les temps de finition v.f pour chaque sommet v
  2. Lorsque chaque sommet est terminé, insérez-le au début d’une liste chaînée
  3. la liste chaînée des sommets, telle qu’elle est maintenant triée.

Un tri topologique peut être effectué en temps ϴ(V + E), puisque le l’algorithme de recherche en profondeur d’abord prend ϴ(V + E) temps et il prend Ω(1) (temps constant) pour insérer chacun des sommets |V| devant une liste chaînée.

De nombreuses applications utilisent des graphes acycliques dirigés pour indiquer les priorités parmi les événements. Nous utilisons le tri topologique de sorte que nous obtenons un ordre pour traiter chaque sommet avant l’un des ses successeurs.

Les sommets d’un graphe peuvent représenter des tâches à effectuer et les arêtes peut représenter des contraintes qu’une tâche doit être effectuée avant une autre; un ordre topologique est une séquence valide pour effectuer la tâches ensemble de tâches décrites en V.

Instance de problème et sa solution #

Soit un sommet v décrivant une Task(hours_to_complete: int), i. e. Task(4) décrit une Task qui prend 4 heures à accomplir, et un bord e décrit un Cooldown(hours: int) tel que Cooldown(3) décrit une durée de temps pour refroidir après une tâche terminée.

Appelons notre graphe dag (puisqu’il s’agit d’un graphe orienté acyclique), et qu’il contienne 5 sommets :

A <- dag.add_vertex(Task(4)); 
B <- dag.add_vertex(Task(5));
C <- dag.add_vertex(Task(3)); 
D <- dag.add_vertex(Task(2));
E <- dag.add_vertex(Task(7));

où nous connectons les sommets avec des arêtes dirigées telles que le graphe est acyclique,

// A ---> C ----+
// |      |     |
// v      v     v
// B ---> D --> E
dag.add_edge(A, B, Cooldown(2));
dag.add_edge(A, C, Cooldown(2));
dag.add_edge(B, D, Cooldown(1));
dag.add_edge(C, D, Cooldown(1));
dag.add_edge(C, E, Cooldown(1));
dag.add_edge(D, E, Cooldown(3));

alors il y a trois ordres topologiques possibles entre ‘A’ et ‘E’,

  1. A -> B -> D -> E
  2. A -> C -> D -> E
  3. A -> C -> E

Algorithme de Thorup #

L’algorithme de Thorup pour le chemin le plus court à source unique pour le graphe non orienté a la complexité temporelle O (m), inférieure à Dijkstra.

Les idées de base sont les suivantes. (Désolé, je n’ai pas encore essayé de l’implémenter, donc je pourrais manquer quelques détails mineurs. Et le document original est payant, j’ai donc essayé de le reconstruire à partir d’autres sources le référençant. Veuillez supprimer ce commentaire si vous pouviez vérifier.)

  • Il existe des moyens de trouver l’arbre couvrant en O(m) (non décrit ici). Vous devez “développer” l’arbre couvrant du bord le plus court au plus long, et ce serait une forêt avec plusieurs composants connectés avant d’être complètement développé.
  • Sélectionnez un entier b (b>=2) et ne considérez que les forêts étendues avec une limite de longueur b^k. Fusionnez les composants qui sont exactement les mêmes mais avec des k différents, et appelez le minimum k le niveau du composant. Ensuite, faites logiquement des composants dans un arbre. u est le parent de v ssi u est le plus petit composant distinct de v qui contient entièrement v. La racine est le graphe entier et les feuilles sont des sommets uniques dans le graphe d’origine (avec le niveau d’infini négatif). L’arbre n’a toujours que O(n) nœuds.
  • Maintenir la distance de chaque composant à la source (comme dans l’algorithme de Dijkstra). La distance d’un composant avec plus d’un sommet est la distance minimale de ses enfants non développés. Définissez la distance du sommet source sur 0 et mettez à jour les ancêtres en conséquence.
  • Considérez les distances en base b. Lorsque vous visitez un nœud de niveau k pour la première fois, placez ses enfants dans des compartiments partagés par tous les nœuds de niveau k (comme dans le tri par compartiment, remplaçant le tas dans l’algorithme de Dijkstra) par le chiffre k et plus de sa distance. Chaque fois que vous visitez un nœud, ne considérez que ses premiers compartiments b, visitez et supprimez chacun d’eux, mettez à jour la distance du nœud actuel et reliez le nœud actuel à son propre parent en utilisant la nouvelle distance et attendez la prochaine visite pour le suivant seaux.
  • Lorsqu’une feuille est visitée, la distance actuelle est la distance finale du sommet. Développez tous les bords à partir de celui-ci dans le graphique d’origine et mettez à jour les distances en conséquence.
  • Visitez le nœud racine (graphe entier) à plusieurs reprises jusqu’à ce que la destination soit atteinte.

Il est basé sur le fait qu’il n’y a pas d’arête de longueur inférieure à l entre deux composants connectés de la forêt couvrante avec une limitation de longueur l, donc, à partir de la distance x, vous pouvez vous concentrer uniquement sur un composant connecté jusqu’à ce que vous atteigniez la distance x + l. Vous visiterez certains sommets avant que les sommets avec une distance plus courte ne soient tous visités, mais cela n’a pas d’importance car on sait qu’il n’y aura pas de chemin plus court jusqu’ici à partir de ces sommets. D’autres parties fonctionnent comme le tri par compartiment / tri par base MSD, et bien sûr, cela nécessite l’arbre couvrant O (m).

Détection d’un cycle dans un graphe orienté à l’aide de Depth First Traversal #

Un cycle dans un graphe orienté existe s’il y a un front arrière découvert lors d’un DFS. Un bord arrière est un bord d’un nœud à lui-même ou à l’un des ancêtres dans un arbre DFS. Pour un graphe déconnecté, nous obtenons une forêt DFS, vous devez donc parcourir tous les sommets du graphe pour trouver des arbres DFS disjoints.

Implémentation C++ :

    #include <iostream>
     #include <list>
        
    using namespace std; 
    
    #define NUM_V   4

    bool helper(list<int> *graph, int u, bool* visited, bool* recStack)
    {
        visited[u]=true;
        recStack[u]=true;
        list<int>::iterator i;
        for(i = graph[u].begin();i!=graph[u].end();++i)
        {  
            if(recStack[*i]) //if vertice v is found in recursion stack of this DFS traversal
                return true;
            else if(*i==u) //if there's an edge from the vertex to itself
                return true;
            else if(!visited[*i])
            {   if(helper(graph, *i, visited, recStack))
                   return true;
            }
        }
        recStack[u]=false;
        return false;
    }
    /*
    /The wrapper function calls helper function on each vertices which have not been visited. Helper function returns true if it detects a back edge in the subgraph(tree) or false.
*/
    bool isCyclic(list<int> *graph, int V)
    {
      bool visited[V];  //array to track vertices already visited
      bool recStack[V]; //array to track vertices in recursion stack of the traversal.

      for(int i = 0;i<V;i++)
       visited[i]=false, recStack[i]=false;  //initialize all vertices as not visited and not recursed

      for(int u = 0; u < V; u++) //Iteratively checks if every vertices have been visited
      {   if(visited[u]==false)
          {  if(helper(graph, u, visited, recStack)) //checks if the DFS tree from the vertex contains a cycle
               return true;
          }
      }
       return false;
    }
    /*
    Driver function
    */
    int main()
    {
        list<int>* graph = new list<int>[NUM_V];
        graph[0].push_back(1);
        graph[0].push_back(2);
        graph[1].push_back(2);
        graph[2].push_back(0);
        graph[2].push_back(3);
        graph[3].push_back(3);
        bool res = isCyclic(graph, NUM_V);
        cout<<res<<endl;
    }

Résultat: Comme indiqué ci-dessous, il y a trois arêtes arrière dans le graphique. Un entre les sommets 0 et 2 ; entre les sommets 0, 1 et 2 ; et sommet 3. La complexité temporelle de la recherche est O(V+E) où V est le nombre de sommets et E est le nombre d’arêtes. [![entrez la description de l’image ici][1]][1]
