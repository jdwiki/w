Un algorithme glouton est un algorithme dans lequel, à chaque étape, nous choisissons l’option la plus avantageuse à chaque étape sans regarder vers l’avenir. Le choix ne dépend que du profit courant.

L’approche gourmande est généralement une bonne approche lorsque chaque profit peut être récupéré à chaque étape, donc aucun choix n’en bloque un autre.

Codage de Huffman #

Le code Huffman est un type particulier de code de préfixe optimal couramment utilisé pour la compression de données sans perte. Il compresse les données très efficacement en économisant de 20% à 90% de mémoire, selon les caractéristiques des données compressées. Nous considérons les données comme une séquence de caractères. L’algorithme gourmand de Huffman utilise un tableau indiquant la fréquence d’apparition de chaque caractère (c’est-à-dire sa fréquence) pour créer une manière optimale de représenter chaque caractère sous forme de chaîne binaire. Le code de Huffman a été proposé par David A. Huffman en 1951.

Supposons que nous ayons un fichier de données de 100 000 caractères que nous souhaitons stocker de manière compacte. Nous supposons qu’il n’y a que 6 caractères différents dans ce fichier. La fréquence des caractères est donnée par :

+------------------------+-----+-----+-----+-----+-----+-----+
|        Character       |  a  |  b  |  c  |  d  |  e  |  f  |
+------------------------+-----+-----+-----+-----+-----+-----+
|Frequency (in thousands)|  45 |  13 |  12 |  16 |  9  |  5  |
+------------------------+-----+-----+-----+-----+-----+-----+

Nous avons de nombreuses options pour représenter un tel fichier d’informations. Ici, nous considérons le problème de la conception d’un code de caractère binaire dans lequel chaque caractère est représenté par une chaîne binaire unique, que nous appelons un mot de code.

[![Mots de code de l’arbre binaire][1]][1]

L’arbre construit nous fournira :

+------------------------+-----+-----+-----+-----+-----+-----+
|        Character       |  a  |  b  |  c  |  d  |  e  |  f  |
+------------------------+-----+-----+-----+-----+-----+-----+
| Fixed-length Codeword  | 000 | 001 | 010 | 011 | 100 | 101 |
+------------------------+-----+-----+-----+-----+-----+-----+
|Variable-length Codeword|  0  | 101 | 100 | 111 | 1101| 1100|
+------------------------+-----+-----+-----+-----+-----+-----+

Si nous utilisons un code de longueur fixe, nous avons besoin de trois bits pour représenter 6 caractères. Cette méthode nécessite 300 000 bits pour coder l’intégralité du fichier. Maintenant la question est : peut-on faire mieux ?

Un code de longueur variable peut faire beaucoup mieux qu’un code de longueur fixe, en donnant des mots de code courts aux caractères fréquents et des mots de code longs aux caractères peu fréquents. Ce code nécessite : (45 X 1 + 13 X 3 + 12 X 3 + 16 X 3 + 9 X 4 + 5 X 4) X 1000 = 224000 bits pour représenter le fichier, ce qui économise environ 25% de mémoire.

Une chose à retenir, nous ne considérons ici que les codes dans lesquels aucun mot de code n’est également un préfixe d’un autre mot de code. Ceux-ci sont appelés codes de préfixe. Pour le codage à longueur variable, nous codons le fichier à 3 caractères abc comme 0.101.100 = 0101100, où “.” désigne la concaténation.

Les codes de préfixe sont souhaitables car ils simplifient le décodage. Puisqu’aucun mot de code n’est le préfixe d’un autre, le mot de code qui commence un fichier codé est sans ambiguïté. Nous pouvons simplement identifier le mot de code initial, le retraduire en caractère d’origine et répéter le processus de décodage sur le reste du fichier encodé. Par exemple, 001011101 analyse uniquement comme 0.0.101.1101, qui se décode en aabe. En bref, toutes les combinaisons de représentations binaires sont uniques. Par exemple, si une lettre est désignée par 110, aucune autre lettre ne sera désignée par 1101 ou 1100. En effet, vous risquez de ne pas savoir s’il faut sélectionner 110 ou continuer à concaténer le bit suivant et sélectionner celui-là.

Technique de compression :

La technique fonctionne en créant un * arbre binaire * de nœuds. Ceux-ci peuvent être stockés dans un tableau régulier, dont la taille dépend du nombre de symboles, n. Un nœud peut être soit un nœud feuille, soit un nœud interne. Initialement, tous les nœuds sont des nœuds feuilles, qui contiennent le symbole lui-même, sa fréquence et éventuellement un lien vers ses nœuds enfants. Par convention, le bit ‘0’ représente l’enfant gauche et le bit ‘1’ représente l’enfant droit. * La file d’attente prioritaire * est utilisée pour stocker les nœuds, ce qui fournit le nœud avec la fréquence la plus basse lorsqu’il est sauté. Le processus est décrit ci-dessous :

  1. Créez un nœud feuille pour chaque symbole et ajoutez-le à la file d’attente prioritaire.
  2. Tant qu’il y a plus d’un nœud dans la file d’attente :
    1. Remove the two nodes of highest priority from the queue.
    2. Create a new internal node with these two nodes as children and with frequency equal to the sum of the two nodes’ frequency.
    3. Add the new node to the queue.
  3. Le nœud restant est le nœud racine et l’arbre de Huffman est complet.

Pour notre exemple : [![Codage de Huffman][2]][2]

Le pseudo-code ressemble à :

Procedure Huffman(C):     // C is the set of n characters and related information
n = C.size
Q = priority_queue()
for i = 1 to n
    n = node(C[i])
    Q.push(n)
end for
while Q.size() is not equal to 1
    Z = new node()
    Z.left = x = Q.pop
    Z.right = y = Q.pop
    Z.frequency = x.frequency + y.frequency
    Q.push(Z)
end while
Return Q

Bien que le temps linéaire donne une entrée triée, dans les cas généraux d’entrée arbitraire, l’utilisation de cet algorithme nécessite un pré-tri. Ainsi, puisque le tri prend O(nlogn) temps dans les cas généraux, les deux méthodes ont la même complexité.

Comme n est ici le nombre de symboles dans l’alphabet, qui est typiquement très petit (par rapport à la longueur du message à encoder), la complexité temporelle n’est pas très importante dans le choix de cet algorithme.

Technique de décompression :

Le processus de décompression consiste simplement à traduire le flux de codes de préfixe en valeur d’octet individuelle, généralement en parcourant l’arbre de Huffman nœud par nœud lorsque chaque bit est lu à partir du flux d’entrée. Atteindre un nœud feuille met nécessairement fin à la recherche de cette valeur d’octet particulière. La valeur de la feuille représente le caractère souhaité. Habituellement, l’arbre de Huffman est construit à l’aide de données ajustées statistiquement sur chaque cycle de compression, ainsi la reconstruction est assez simple. Sinon, les informations pour reconstruire l’arbre doivent être envoyées séparément. Le pseudo-code :

Procedure HuffmanDecompression(root, S):   // root represents the root of Huffman Tree
n := S.length                              // S refers to bit-stream to be decompressed
for i := 1 to n
    current = root
    while current.left != NULL and current.right != NULL
        if S[i] is equal to '0'
            current := current.left
        else
            current := current.right
        endif
        i := i+1
    endwhile
    print current.symbol
endfor

Explication gourmande : Le codage de Huffman examine l’occurrence de chaque caractère et le stocke sous forme de chaîne binaire de manière optimale. L’idée est d’attribuer des codes de longueur variable aux caractères d’entrée d’entrée, la longueur des codes attribués est basée sur les fréquences des caractères correspondants. Nous créons un arbre binaire et opérons dessus de manière ascendante afin que les deux caractères les moins fréquents soient aussi éloignés que possible de la racine. De cette façon, le caractère le plus fréquent obtient le code le plus petit et le caractère le moins fréquent obtient le code le plus grand.

Problème de sélection d’activité #

Le problème #

Vous avez un ensemble de choses à faire (activités). Chaque activité a une heure de début et une heure de fin. Vous n’êtes pas autorisé à effectuer plus d’une activité à la fois. Votre tâche est de trouver un moyen d’effectuer le nombre maximum d’activités.

Par exemple, supposons que vous ayez une sélection de classes parmi lesquelles choisir.

N° d’activité heure de début heure de fin
1 10h20 11h00
2 10h30 11h30
3 11h00 00h00
4 10h00 11h30
5 9h00 11h00

N’oubliez pas que vous ne pouvez pas suivre deux cours en même temps. Cela signifie que vous ne pouvez pas suivre les classes 1 et 2 car elles partagent une heure commune de 10h30 à 11h00. Cependant, vous pouvez prendre les cours 1 et 3 car ils ne partagent pas un horaire commun. Votre tâche consiste donc à suivre le plus grand nombre de cours possible sans aucun chevauchement. Comment peux-tu faire ça?

Une analyse #

Réfléchissons à la solution par approche gourmande. Tout d’abord, nous choisissons au hasard une approche et vérifions qu’elle fonctionnera ou non.

  • triez l’activité par heure de début, cela signifie quelle activité commence en premier, nous la prendrons en premier. puis prenez du premier au dernier de la liste triée et vérifiez qu’il se croisera ou non avec l’activité précédente. Si l’activité actuelle n’est pas intersectée avec l’activité précédemment prise, nous effectuerons l’activité sinon nous ne l’exécuterons pas. cette approche fonctionnera pour certains cas comme
N° d’activité heure de début heure de fin
1 11h00 13h30
2 11h30 12h00
3 13h30 14h00
4 10h00 11h00

l’ordre de tri sera 4–>1–>2–>3. L’activité 4–> 1–> 3 sera effectuée et l’activité 2 sera ignorée. le maximum de 3 activités sera effectué. Cela fonctionne pour ce type de cas. mais cela échouera dans certains cas. Appliquons cette approche pour le cas

N° d’activité heure de début heure de fin
1 11h00 13h30
2 11h30 12h00
3 13h30 14h00
4 10h00 15h00

L’ordre de tri sera 4–>1–>2–>3 et seule l’activité 4 sera effectuée mais la réponse peut être l’activité 1–>3 ou 2–>3 sera effectuée. Notre approche ne fonctionnera donc pas pour le cas ci-dessus. Essayons une autre approche

  • Trier l’activité par durée, ce qui signifie effectuer l’activité la plus courte en premier. cela peut résoudre le problème précédent. Bien que le problème ne soit pas complètement résolu. Il y a encore des cas qui peuvent échouer à la solution. appliquer cette approche sur le cas ci-dessous.
N° d’activité heure de début heure de fin
1 6h00 11h40
2 11h30 12h00
3 23h40 14h00

si nous trions l’activité par durée, l’ordre de tri sera 2–> 3 —>1 . et si nous exécutons d’abord l’activité n° 2, aucune autre activité ne peut être effectuée. Mais la réponse sera effectuer l’activité 1 puis effectuer 3 . Nous pouvons donc effectuer au maximum 2 activités. Cela ne peut donc pas être une solution à ce problème. Nous devrions essayer une approche différente.

La solution #

  • Triez l’activité par heure de fin, ce qui signifie que l’activité se termine en premier. l’algorithme est donné ci-dessous
  1. Triez les activités par leurs heures de fin.
  2. Si l’activité à réaliser ne partage pas un temps commun avec les activités précédemment réalisées, réalisez l’activité.

Analysons le premier exemple

N° d’activité heure de début heure de fin
1 10h20 11h00
2 10h30 11h30
3 11h00 00h00
4 10h00 11h30
5 9h00 11h00

trier l’activité par ses heures de fin, donc l’ordre de tri sera 1–>5–>2–>4–>3.. la réponse est 1–>3 ces deux activités seront réalisées. et c’est la réponse. voici le code sudo.

  1. trier : activités
  2. effectuer la première activité de la liste triée des activités.
  3. Set : Current_activity := première activité
  4. set: end_time := end_time of Current activity
  5. aller à l’activité suivante si elle existe, si elle n’existe pas, terminer .
  6. si start_time of current activity <= end_time : effectuer l’activité et passer à 4
  7. sinon : arrivé à 5.

voir ici pour l’aide au codage http://www.geeksforgeeks.org/greedy-algorithms-set-1-activity-selection-problem/

Problème de changement #

Etant donné un système monétaire, est-il possible de donner un montant de pièces et comment trouver un ensemble minimal de pièces correspondant à ce montant.

Systèmes monétaires canoniques. Pour certains systèmes monétaires, comme ceux que nous utilisons dans la vraie vie, la solution « intuitive » fonctionne parfaitement. Par exemple, si les différentes pièces et billets en euros (hors centimes) sont de 1€, 2€, 5€, 10€, donner la pièce ou le billet le plus élevé jusqu’à atteindre le montant et répéter cette procédure conduira à l’ensemble minimal de pièces .

Nous pouvons le faire récursivement avec OCaml :

(* assuming the money system is sorted in decreasing order *)
let change_make money_system amount =
  let rec loop given amount =
    if amount = 0 then given
    else 
      (* we find the first value smaller or equal to the remaining amount *)
      let coin = List.find ((>=) amount) money_system in
      loop (coin::given) (amount - coin)
  in loop [] amount

Ces systèmes sont conçus pour faciliter le changement. Le problème devient plus difficile lorsqu’il s’agit d’un système monétaire arbitraire.

Cas général. Comment donner 99€ avec des pièces de 10€, 7€ et 5€ ? Ici, donner des pièces de 10€ jusqu’à ce qu’il nous reste 9€ ne mène évidemment à aucune solution. Pire que cela, une solution peut ne pas exister. Ce problème est en fait np-difficile, mais des solutions acceptables mêlant gloutonnerie et mémoïsation existent. L’idée est d’explorer toutes les possibilités et de choisir celle avec le nombre minimal de pièces.

Pour donner un montant X > 0, on choisit une pièce P dans le système monétaire, puis on résout le sous-problème correspondant à X-P. Nous essayons cela pour tous les éléments du système. La solution, si elle existe, est alors le plus petit chemin ayant conduit à 0.

Voici une fonction récursive OCaml correspondant à cette méthode. Elle renvoie None si aucune solution n’existe.

(* option utilities *)
let optmin x y =
  match x,y with
  | None,a | a,None -> a
  | Some x, Some y-> Some (min x y)

let optsucc = function
  | Some x -> Some (x+1)
  | None -> None

(* Change-making problem*)
let change_make money_system amount =
  let rec loop n =
    let onepiece acc piece =
      match n - piece with
      | 0 -> (*problem solved with one coin*) 
             Some 1
      | x -> if x < 0 then 
               (*we don't reach 0, we discard this solution*)
               None
             else
               (*we search the smallest path different to None with the remaining pieces*) 
               optmin (optsucc (loop x)) acc
    in
    (*we call onepiece forall the pieces*)
    List.fold_left onepiece None money_system
  in loop amount

Remarque : On peut remarquer que cette procédure peut calculer plusieurs fois la variation définie pour une même valeur. En pratique, utiliser la mémorisation pour éviter ces répétitions conduit à des résultats plus rapides (bien plus rapides).

Problème de sac à dos continu #

Étant donné les éléments comme (valeur, poids), nous devons les placer dans un sac à dos (récipient) d’une capacité k. Noter! Nous pouvons casser des objets pour maximiser la valeur !

Exemple d’entrée :

values[] = [1, 4, 5, 2, 10]
weights[] = [3, 2, 1, 2, 4]
k = 8

Production attendue:

maximumValueOfItemsInK = 20;

Algorithme:

1) Sort values and weights by value/weight.
   values[] = [5, 10, 4, 2, 1]
   weights[] = [1, 4, 2, 2, 3]
2) currentWeight = 0; currentValue = 0;
3) FOR i = 0; currentWeight < k && i < values.length; i++ DO:
       IF k - currentWeight < weights[i] DO
           currentValue = currentValue + values[i];
           currentWeight = currentWeight + weights[i];
       ELSE
           currentValue = currentValue + values[i]*(k - currentWeight)/weights[i]
           currentWeight = currentWeight + weights[i]*(k - currentWeight)/weights[i]
       END_IF
   END_FOR
   PRINT "maximumValueOfItemsInK = " + currentValue;
