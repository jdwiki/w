Recherche binaire #

Introduction #

La recherche binaire est un algorithme de recherche Divide and Conquer. Il utilise le temps O(log n) pour trouver l’emplacement d’un élément dans un espace de recherche où n est la taille de l’espace de recherche.

La recherche binaire fonctionne en divisant par deux l’espace de recherche à chaque itération après avoir comparé la valeur cible à la valeur médiane de l’espace de recherche.

Pour utiliser la recherche binaire, l’espace de recherche doit être ordonné (trié) d’une certaine manière. Les entrées en double (celles qui se comparent comme égales selon la fonction de comparaison) ne peuvent pas être distinguées, bien qu’elles ne violent pas la propriété Recherche binaire.

Classiquement, nous utilisons inférieur à (<) comme fonction de comparaison. Si a < b, il retournera vrai. si a n’est pas inférieur à b et b n’est pas inférieur à a, a et b sont égaux.

Exemple de question #

Vous êtes un économiste, un assez mauvais cependant. Vous êtes chargé de trouver le prix d’équilibre (c’est-à-dire le prix où l’offre = la demande) pour le riz.

N’oubliez pas que plus un prix est élevé, plus l’offre est importante et plus la demande est faible

Comme votre entreprise est très efficace pour calculer les forces du marché, vous pouvez obtenir instantanément l’offre et la demande en unités de riz lorsque le prix du riz est fixé à un certain prix « p ».

Votre patron veut le prix d’équilibre dès que possible, mais vous dit que le prix d’équilibre peut être un entier positif qui est au plus 10 ^ 17 et il est garanti qu’il y aura exactement 1 solution entière positive dans la plage. Alors continuez votre travail avant de le perdre !

Vous êtes autorisé à appeler les fonctions getSupply(k) et getDemand(k), qui feront exactement ce qui est indiqué dans le problème.

Exemple Explication #

Ici, notre espace de recherche va de 1 à 10^17. Ainsi, une recherche linéaire est impossible.

Cependant, notez que lorsque le k augmente, getSupply(k) augmente et getDemand(k) diminue. Ainsi, pour tout x > y, getSupply(x) - getDemand(x) > getSupply(y) - getDemand(y). Par conséquent, cet espace de recherche est monotone et nous pouvons utiliser la recherche binaire.

Le pseudo-code suivant illustre l’utilisation de la recherche binaire :

high = 100000000000000000     <- Upper bound of search space
low = 1                       <- Lower bound of search space
while high - low > 1
    mid = (high + low) / 2    <- Take the middle value
    supply = getSupply(mid)  
    demand = getDemand(mid)
    if supply > demand        
        high = mid             <- Solution is in lower half of search space
    else if demand > supply
        low = mid              <- Solution is in upper half of search space
    else                       <- supply==demand condition
        return mid             <- Found solution

Cet algorithme s’exécute en ~O(log 10^17). Ceci peut être généralisé au temps ~O(log S) où S est la taille de l’espace de recherche puisqu’à chaque itération de la boucle while, nous avons réduit de moitié l’espace de recherche (de [low:high] à [ bas:moyen] ou [moyen:élevé]).

Mise en œuvre C de la recherche binaire avec récursivité

int binsearch(int a[], int x, int low, int high) {
    int mid;

    if (low > high)
      return -1;

    mid = (low + high) / 2;

    if (x == a[mid]) {
        return (mid);
    } else 
    if (x < a[mid]) {
        binsearch(a, x, low, mid - 1);
    } else {
        binsearch(a, x, mid + 1, high);
    }
}

Recherche binaire : sur les numéros triés #

Il est plus facile d’afficher une recherche binaire sur des nombres en utilisant un pseudo-code

int array[1000] = { sorted list of numbers };
int N = 100;  // number of entries in search space;
int high, low, mid; // our temporaries
int x; // value to search for

low = 0;
high = N -1;
while(low < high)
{
    mid = (low + high)/2;
    if(array[mid] < x)
        low = mid + 1;
    else
        high = mid;  
}  
if(array[low] == x)
    // found, index is low
else
    // not found

N’essayez pas de revenir plus tôt en comparant array[mid] à x pour l’égalité. La comparaison supplémentaire ne peut que ralentir le code. Notez que vous devez ajouter un au plus bas pour éviter d’être piégé par une division entière toujours arrondie à l’inférieur.

Fait intéressant, la version ci-dessus de la recherche binaire vous permet de trouver la plus petite occurrence de x dans le tableau. Si le tableau contient des doublons de x, l’algorithme peut être légèrement modifié afin qu’il renvoie la plus grande occurrence de x en ajoutant simplement au conditionnel if :

while(low < high)
    {
        mid = low + ((high - low) / 2);
        if(array[mid] < x || (array[mid] == x && array[mid + 1] == x))
            low = mid + 1;
        else
            high = mid;  
    }

Notez qu’au lieu de faire mid = (low + high) / 2, il peut également être judicieux d’essayer mid = low + ((high - low) / 2) pour des implémentations telles que les implémentations Java afin de réduire le risque d’obtenir un débordement pour les entrées vraiment volumineuses.

Recherche linéaire #

La recherche linéaire est un algorithme simple. Il parcourt les éléments jusqu’à ce que la requête soit trouvée, ce qui en fait un algorithme linéaire - la complexité est O(n), où n est le nombre d’éléments à parcourir.

Pourquoi O(n) ? Dans le pire des cas, vous devez parcourir tous les n éléments.

Cela peut être comparé à la recherche d’un livre dans une pile de livres - vous les parcourez tous jusqu’à ce que vous trouviez celui que vous voulez.

Ci-dessous une implémentation Python :

def linear_search(searchable_list, query):
    for x in searchable_list:
        if query == x:
            return True
    return False

linear_search(['apple', 'banana', 'carrot', 'fig', 'garlic'], 'fig') #returns True

Rabbi Karp #

L’algorithme de Rabin-Karp ou l’algorithme de Karp-Rabin est un algorithme de recherche de chaîne qui utilise le hachage pour trouver l’une quelconque d’un ensemble de chaînes de modèle dans un texte. Son temps d’exécution moyen et dans le meilleur des cas est O(n+m) dans l’espace O( p), mais son temps dans le pire des cas est O(nm) où n est la longueur du texte et m est la longueur du motif.

Lors du calcul de la valeur de hachage, nous la divisons par un nombre premier afin d’éviter les collisions. Après avoir divisé par un nombre premier, les risques de collision seront moindres, mais il y a toujours une chance que la valeur de hachage soit la même pour deux chaînes, donc quand nous obtenons une correspondance, nous devons la vérifier caractère par caractère pour nous assurer que nous avons obtenu une correspondance correcte.

t =(d*(t - text.charAt(i)*h) + text.charAt(i+m))%q ;

Il s’agit de recalculer la valeur de hachage pour le motif, d’abord en supprimant le caractère le plus à gauche, puis en ajoutant le nouveau caractère du texte.

Analyse de la recherche linéaire (Pire, Moyenne et Meilleur cas) #

On peut avoir trois cas pour analyser un algorithme :

  1. Le pire des cas

  2. Cas moyen

  3. Meilleur cas

    #include <stdio.h>

// Linearly search x in arr[].  If x is present then return the index,

// otherwise return -1
int search(int arr[], int n, int x)
{
    int i;
    for (i=0; i<n; i++)
    {
        if (arr[i] == x)
         return i;
    }

    return -1;
}

    /* Driver program to test above functions*/

    int main()
{
    int arr[] = {1, 10, 30, 15};
    int x = 30;
    int n = sizeof(arr)/sizeof(arr[0]);
    printf("%d is present at index %d", x, search(arr, n, x));

    getchar();
    return 0;
}

*** Analyse du pire cas (généralement effectuée) ***

Dans l’analyse du pire des cas, nous calculons la limite supérieure du temps d’exécution d’un algorithme. Nous devons connaître le cas qui provoque le nombre maximum d’opérations à exécuter. Pour la recherche linéaire, le pire des cas se produit lorsque l’élément à rechercher (x dans le code ci-dessus) n’est pas présent dans le tableau. Lorsque x n’est pas présent, la fonction search() le compare avec tous les éléments de arr[] un par un. Par conséquent, la complexité temporelle de la recherche linéaire dans le pire des cas serait Θ (n)

Analyse de cas moyenne (parfois effectuée)

Dans une analyse de cas moyenne, nous prenons toutes les entrées possibles et calculons le temps de calcul pour toutes les entrées. Additionnez toutes les valeurs calculées et divisez la somme par le nombre total d’entrées. Nous devons connaître (ou prédire) la distribution des cas. Pour le problème de recherche linéaire, supposons que tous les cas sont uniformément distribués (y compris le cas où x n’est pas présent dans le tableau). Nous additionnons donc tous les cas et divisons la somme par (n+1). Voici la valeur de la complexité moyenne du temps de cas.

Meilleure analyse de cas (faux)

Dans le meilleur des cas, nous calculons la borne inférieure du temps d’exécution d’un algorithme. Nous devons connaître le cas qui provoque le nombre minimum d’opérations à exécuter. Dans le problème de recherche linéaire, le meilleur cas se produit lorsque x est présent au premier emplacement. Le nombre d’opérations dans le meilleur des cas est constant (indépendant de n). Donc la complexité temporelle dans le meilleur des cas serait Θ(1) La plupart du temps, nous effectuons une analyse des cas les plus défavorables pour analyser les algorithmes. Dans la pire analyse, nous garantissons une borne supérieure sur le temps d’exécution d’un algorithme qui est une bonne information. L’analyse de cas moyenne n’est pas facile à faire dans la plupart des cas pratiques et elle est rarement effectuée. Dans l’analyse du cas moyen, nous devons connaître (ou prédire) la distribution mathématique de toutes les entrées possibles. L’analyse du meilleur cas est fausse. Garantir une borne inférieure sur un algorithme ne fournit aucune information car dans le pire des cas, un algorithme peut prendre des années à s’exécuter.

Pour certains algorithmes, tous les cas sont asymptotiquement identiques, c’est-à-dire qu’il n’y a ni pire ni meilleur cas. Par exemple, Fusionner Trier. Merge Sort effectue Θ(nLogn) opérations dans tous les cas. La plupart des autres algorithmes de tri ont les pires et les meilleurs cas. Par exemple, dans l’implémentation typique de Quick Sort (où pivot est choisi comme élément d’angle), le pire se produit lorsque le tableau d’entrée est déjà trié et le meilleur se produit lorsque les éléments pivot divisent toujours le tableau en deux moitiés. Pour le tri par insertion, le pire cas se produit lorsque le tableau est trié en sens inverse et le meilleur cas se produit lorsque le tableau est trié dans le même ordre que la sortie.
