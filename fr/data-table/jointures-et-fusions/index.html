Jointures et fusions

Une jointure combine deux tables contenant des colonnes liées. Le terme couvre un large éventail d’opérations, essentiellement tout sauf [ajouter les deux tables] (https://www.wikiod.com/fr/data-table/remodeler-empiler-et-diviser#Empiler plusieurs tables à l’aide de rbindlist -using-rbindlist). “Fusionner” est un synonyme. Tapez ?`[.data.table` pour la documentation officielle.

Syntaxe #

  • x[i, sur, j] # join: data.table x & data.table or list i
  • x[!i, sur, j] # anti-join

Travailler avec des tables à clés #

Si x & i ont une [clé][1] ou x est codé pour correspondre aux premières colonnes de i, alors le on peut être ignoré comme x[i].

Désambiguïsation des noms de colonnes en commun #

Dans j de x[i, on, j], les colonnes de i peuvent être référencées avec les préfixes i.*.

Regroupement sur des sous-ensembles #

Dans j de x[i, on, j, by=.EACHI], j est calculé pour chaque ligne de i.

C’est la seule valeur de by qui vaut la peine d’être utilisée. Pour toute autre valeur, les colonnes de i ne sont pas disponibles.

[1] : https://www.wikiod.com/fr/data-table/utiliser-des-cles-et-des-indices

Mettre à jour les valeurs dans une jointure #

Lorsque les données sont [“organisées”][1], elles sont souvent organisées en plusieurs tables. Pour combiner les données à analyser, nous devons “mettre à jour” une table avec les valeurs d’une autre.

Par exemple, nous pourrions avoir des données de ventes pour les performances, où les attributs de l’interprète (son budget) et du lieu (sa population) sont stockés dans des tables séparées :

set.seed(1)
mainDT = data.table(
  p_id = rep(LETTERS[1:2], c(2,4)), 
  geo_id = sample(rep(state.abb[c(1,25,50)], 3:1)), 
  sales = sample(100, 6)
)
pDT   = data.table(id = LETTERS[1:2], budget = c(60, 75))
geoDT = data.table(id = state.abb[c(1,50)], pop = c(100, 200))

mainDT # sales data
#    p_id geo_id sales
# 1:    A     AL    95
# 2:    A     WY    66
# 3:    B     AL    62
# 4:    B     MO     6
# 5:    B     AL    20
# 6:    B     MO    17


pDT # performer attributes
#    id budget
# 1:  A     60
# 2:  B     75

geoDT # location attributes
#    id pop
# 1: AL 100
# 2: WY 200

Lorsque nous sommes prêts à faire une analyse, nous devons récupérer les variables de ces autres tables :

DT = copy(mainDT)

DT[pDT, on=.(p_id = id), budget := i.budget]
DT[geoDT, on=.(geo_id = id), pop := i.pop]

#    p_id geo_id sales budget pop
# 1:    A     AL    95     60 100
# 2:    A     WY    66     60 200
# 3:    B     AL    62     75 100
# 4:    B     MO     6     75  NA
# 5:    B     AL    20     75 100
# 6:    B     MO    17     75  NA

Une copie est prise pour éviter de contaminer les données brutes, mais nous pourrions travailler directement sur mainDT à la place.

Avantages d’utiliser des tables séparées #

Les avantages de cette structure sont traités dans l’article sur les données ordonnées, mais dans ce contexte :

  1. Tracer les données manquantes. Seules les lignes qui correspondent dans la fusion reçoivent une affectation. Nous n’avons pas de données pour geo_id == "MO" ci-dessus, donc ses variables sont NA dans notre tableau final. Si nous voyons des données manquantes comme celle-ci de manière inattendue, nous pouvons remonter jusqu’à l’observation manquante dans la table “geoDT” et rechercher à partir de là si nous avons un problème de données qui peut être résolu.

    • Compréhensibilité. * Lors de la construction de notre modèle statistique, il peut être important de garder à l’esprit que le “budget” est constant pour chaque interprète. En général, la compréhension de la structure des données est payante.

  3. Taille de la mémoire. Il peut y avoir un grand nombre d’attributs d’interprète et d’emplacement qui ne se retrouvent pas dans le modèle statistique. De cette façon, nous n’avons pas besoin de les inclure dans le tableau (éventuellement massif) utilisé pour l’analyse.

Détermination des colonnes par programmation #

S’il y a beaucoup de colonnes dans pDT, mais que nous ne voulons en sélectionner que quelques-unes, nous pouvons utiliser

p_cols = "budget"
DT[pDT, on=.(p_id = id), (p_cols) := mget(sprintf("i.%s", p_cols))]

Les parenthèses autour de (p_cols) := sont essentielles, comme indiqué dans [la doc sur la création de colonnes][2].

[1] : https://www.jstatsoft.org/article/view/v059i10 [2] : https://www.wikiod.com/fr/data-table/ajouter-et-modifier-des-colonnes

Équi-jointure #

# example data
a = data.table(id = c(1L, 1L, 2L, 3L, NA_integer_), x = 11:15)
#    id  x
# 1:  1 11
# 2:  1 12
# 3:  2 13
# 4:  3 14
# 5: NA 15

b = data.table(id = 1:2, y = -(1:2))
#    id  y
# 1:  1 -1
# 2:  2 -2

Intuition #

Considérez x[i] comme la sélection d’un sous-ensemble de x pour chaque ligne de i. Cette syntaxe reflète le sous-ensemble de matrice dans la base R et est cohérente avec le premier argument signifiant “où”, dans [DT[où, sélectionner|mettre à jour|faire, par]][1].

On peut se demander pourquoi cette nouvelle syntaxe vaut la peine d’être apprise, puisque merge(x,i) fonctionne toujours avec data.tables. La réponse courte est que nous voulons généralement fusionner et ensuite faire quelque chose de plus. La syntaxe x[i] capture de manière concise ce modèle d’utilisation et permet également un calcul plus efficace. Pour une explication plus détaillée, lisez les FAQ [1.12][2] et [2.14][3].

Gestion des lignes à correspondance multiple #

Par défaut, chaque ligne de ‘a’ correspondant à chaque ligne de ‘b’ est renvoyée :

a[b, on="id"]
#    id  x  y
# 1:  1 11 -1
# 2:  1 12 -1
# 3:  2 13 -2

Cela peut être modifié avec mult :

a[b, on="id", mult="first"]
#    id  x  y
# 1:  1 11 -1
# 2:  2 13 -2

Gestion des lignes sans correspondance #

Par défaut, les lignes sans correspondance de a s’affichent toujours dans le résultat :

b[a, on="id"]
#    id  y  x
# 1:  1 -1 11
# 2:  1 -1 12
# 3:  2 -2 13
# 4:  3 NA 14
# 5: NA NA 15

Pour les masquer, utilisez nomatch :

b[a, on="id", nomatch=0]
#    id  y  x
# 1:  1 -1 11
# 2:  1 -1 12
# 3:  2 -2 13

Notez que x[i] tentera de faire correspondre les NA dans i.

Comptage des correspondances renvoyées #

Pour compter le nombre de correspondances pour chaque ligne de i, utilisez .N et by=.EACHI.

b[a, on="id", .N, by=.EACHI]
#    id N
# 1:  1 1
# 2:  1 1
# 3:  2 1
# 4:  3 0
# 5: NA 0
