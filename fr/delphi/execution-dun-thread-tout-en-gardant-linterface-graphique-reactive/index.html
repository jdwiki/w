Interface graphique réactive utilisant des threads pour le travail en arrière-plan et PostMessage pour rendre compte des threads #

Garder une interface graphique réactive lors de l’exécution d’un long processus nécessite soit des “rappels” très élaborés pour permettre à l’interface graphique de traiter sa file d’attente de messages, soit l’utilisation de threads (d’arrière-plan) (de travail).

Lancer n’importe quel nombre de threads pour faire du travail n’est généralement pas un problème. Le plaisir commence lorsque vous souhaitez que l’interface graphique affiche les résultats intermédiaires et finaux ou rende compte de la progression.

Afficher quoi que ce soit dans l’interface graphique nécessite d’interagir avec les commandes et/ou la file d’attente/pompe de messages. Cela devrait toujours être fait dans le contexte du fil principal. Jamais dans le contexte d’un autre fil.

Il existe de nombreuses façons de gérer cela.

Cet exemple montre comment vous pouvez le faire en utilisant des threads simples, permettant à l’interface graphique d’accéder à l’instance de thread une fois qu’elle est terminée en définissant FreeOnTerminate sur false et en signalant quand un thread est “terminé” en utilisant PostMessage.

Remarques sur les conditions de course : Les références aux threads de travail sont conservées dans un tableau du formulaire. Lorsqu’un thread est terminé, la référence correspondante dans le tableau devient nulle.

Il s’agit d’une source potentielle de conditions de course. De même que l’utilisation d’un booléen “Running” pour faciliter la détermination s’il reste des threads à terminer.

Vous devrez décider si vous devez protéger ces ressources à l’aide de verrous ou non.

Dans cet exemple, tel qu’il est, il n’y a pas besoin. Ils ne sont modifiés qu’à deux endroits : la méthode StartThreads et la méthode HandleThreadResults. Les deux méthodes ne s’exécutent que dans le contexte du thread principal. Tant que vous le gardez ainsi et que vous ne commencez pas à appeler ces méthodes à partir du contexte de différents threads, il n’y a aucun moyen pour elles de produire des conditions de concurrence.

Fil de discussion #

type
  TWorker = class(TThread)
  private
    FFactor: Double;
    FResult: Double;
    FReportTo: THandle;
  protected
    procedure Execute; override;
  public
    constructor Create(const aFactor: Double; const aReportTo: THandle);

    property Factor: Double read FFactor;
    property Result: Double read FResult;
  end;

Le constructeur définit simplement les membres privés et définit FreeOnTerminate sur False. Ceci est essentiel car cela permettra au thread principal d’interroger l’instance de thread pour son résultat.

La méthode execute effectue son calcul puis envoie un message au handle qu’elle a reçu dans son constructeur pour indiquer que c’est terminé :

procedure TWorker.Execute;
const
  Max = 100000000;var
  i : Integer;
begin
  inherited;

  FResult := FFactor;
  for i := 1 to Max do
    FResult := Sqrt(FResult);

  PostMessage(FReportTo, UM_WORKERDONE, Self.Handle, 0);
end;

L’utilisation de PostMessage est essentielle dans cet exemple. PostMessage place “juste” un message dans la file d’attente de la pompe à messages du thread principal et n’attend pas qu’il soit traité. Il est de nature asynchrone. Si vous utilisiez SendMessage, vous vous coderiez dans un cornichon. SendMessage place le message dans la file d’attente et attend qu’il ait été traité. Bref, c’est synchrone.

Les déclarations du message UM_WORKERDONE personnalisé sont déclarées comme suit :

const
  UM_WORKERDONE = WM_APP + 1;
type
  TUMWorkerDone = packed record
    Msg: Cardinal;
    ThreadHandle: Integer;
    unused: Integer;
    Result: LRESULT;
  end;

La const UM_WORKERDONE utilise WM_APP comme point de départ pour sa valeur afin de s’assurer qu’elle n’interfère avec aucune valeur utilisée par Windows ou Delphi VCL (comme [recommandé][1] par MicroSoft).

Formulaire #

N’importe quel formulaire peut être utilisé pour démarrer des threads. Il vous suffit d’y ajouter les membres suivants :

private
  FRunning: Boolean;
  FThreads: array of record
    Instance: TThread;
    Handle: THandle;
  end;
  procedure StartThreads(const aNumber: Integer);
  procedure HandleThreadResult(var Message: TUMWorkerDone); message UM_WORKERDONE;

Oh, et l’exemple de code suppose l’existence d’un Memo1: TMemo; dans les déclarations du formulaire, qu’il utilise pour “la journalisation et la création de rapports”.

Le FRunning peut être utilisé pour empêcher que l’on clique sur l’interface graphique pendant que le travail est en cours. FThreads est utilisé pour contenir le pointeur d’instance et le handle des threads créés.

La procédure pour démarrer les threads a une implémentation assez simple. Il commence par vérifier s’il existe déjà un ensemble de threads en attente. Si c’est le cas, il sort tout simplement. Si ce n’est pas le cas, il définit l’indicateur sur true et démarre les threads en fournissant à chacun son propre handle afin qu’ils sachent où poster leur message “done”.

procedure TForm1.StartThreads(const aNumber: Integer);
var
  i: Integer;
begin
  if FRunning then
    Exit;
    
  FRunning := True;

  Memo1.Lines.Add(Format('Starting %d worker threads', [aNumber]));
  SetLength(FThreads, aNumber);
  for i := 0 to aNumber - 1 do
  begin
    FThreads[i].Instance := TWorker.Create(pi * (i+1), Self.Handle);
    FThreads[i].Handle := FThreads[i].Instance.Handle;
  end;
end;

Le handle du thread est également placé dans le tableau car c’est ce que nous recevons dans les messages qui nous indiquent qu’un thread est terminé et le fait de l’avoir en dehors de l’instance du thread le rend légèrement plus facile d’accès. Avoir le handle disponible en dehors de l’instance du thread nous permet également d’utiliser FreeOnTerminate défini sur True si nous n’avions pas besoin de l’instance pour obtenir ses résultats (par exemple s’ils avaient été stockés dans une base de données). Dans ce cas, il ne serait bien sûr pas nécessaire de conserver une référence à l’instance.

Le plaisir est dans l’implémentation de HandleThreadResult :

procedure TForm1.HandleThreadResult(var Message: TUMWorkerDone);
var
  i: Integer;
  ThreadIdx: Integer;
  Thread: TWorker;
  Done: Boolean;
begin
  // Find thread in array
  ThreadIdx := -1;
  for i := Low(FThreads) to High(FThreads) do
    if FThreads[i].Handle = Cardinal(Message.ThreadHandle) then
    begin
      ThreadIdx := i;
      Break;
    end;

  // Report results and free the thread, nilling its pointer and handle 
  // so we can detect when all threads are done.
  if ThreadIdx > -1 then
  begin
    Thread := TWorker(FThreads[i].Instance);
    Memo1.Lines.Add(Format('Thread %d returned %f', [ThreadIdx, Thread.Result]));
    FreeAndNil(FThreads[i].Instance);
    FThreads[i].Handle := nil;
  end;

  // See whether all threads have finished.
  Done := True;
  for i := Low(FThreads) to High(FThreads) do
    if Assigned(FThreads[i].Instance) then
    begin
      Done := False;
      Break;
    end;
  if Done then
  begin
    Memo1.Lines.Add('Work done');
    FRunning := False;
  end;
end;

Cette méthode recherche d’abord le thread à l’aide du handle reçu dans le message. Si une correspondance a été trouvée, il récupère et signale le résultat du thread à l’aide de l’instance (FreeOnTerminate était False, souvenez-vous ?), puis se termine : en libérant l’instance et en définissant à la fois la référence de l’instance et le handle sur nil, indiquant ce fil n’est plus d’actualité.

Enfin, il vérifie si l’un des threads est toujours en cours d’exécution. Si aucun n’est trouvé, « tout est terminé » est signalé et l’indicateur « FRunning » est défini sur « False » afin qu’un nouveau lot de travail puisse être démarré.
