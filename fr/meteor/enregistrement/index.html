Journalisation côté serveur de base #

La première étape de la journalisation consiste simplement à exécuter Meteor à partir du shell, et vous obtiendrez les journaux du serveur dans la console de commande.

meteor

L’étape suivante consiste à diriger le contenu de std_out et std_err vers un fichier journal, comme ceci :

meteor > my_app_log.log 2> my_app_err.log

Outils de journalisation côté client #

Une fois que votre connexion côté serveur est en place, il est temps de passer du côté client. Si vous n’avez pas exploré l’API de la console, préparez-vous à un régal. Il y a en fait toutes sortes de choses que vous pouvez faire avec l’API de console intégrée qui est native pour chaque installation Chrome et Safari. À tel point, en fait, que vous n’aurez peut-être pas besoin de Winston ou d’autres frameworks de journalisation.

La première chose à faire est d’installer les outils de journalisation et de développement côté client. Chrome et Safari sont livrés avec eux, mais Firefox nécessite l’extension Firebug.

Extension Firebug

Ensuite, vous voudrez consulter la documentation de l’API de la console. Les deux documents suivants sont des ressources inestimables pour l’apprentissage de la journalisation de la console.

Outils de développement Chrome

Firebug (Client)

Outils de journalisation de serveur avancés #

Une fois que vous avez à la fois votre journalisation côté serveur en cours d’exécution et vos outils de développement côté client, vous pouvez commencer à regarder des extensions spécifiques à Meteor comme l’extension Meteor Chrome DevTools. Cela vous permet d’observer la journalisation du serveur dans le client ! Parce que la base de données est partout. Tout comme la journalisation.

Extension Chrome DevTools (serveur)

Erreur de journalisation sur le volet de la base de données #

L’exemple suivant est de 0,5 à 0,7 jours et illustre comment consigner une erreur lorsque la base de données n’a pas encore rempli le curseur côté client.

Template.landingPage.postsList = function(){
  try{
    return Posts.find();
  }catch(error){
    //color code the error (red)
    console.error(error);
  }
}

Journalisation des informations sur le contexte des données dans un assistant de modèle #

Ce qui suit utilise l’API Chrome Logging. Si la syntaxe .group() est utilisée dans plusieurs modèles, elle organisera graphiquement les journaux de la console à partir de différents modèles dans une arborescence hiérarchique.

Vous pouvez également voir comment inspecter le contexte actuel des données et comment chaîner les données.

Template.landingPage.getId = function(){
  // using a group block to illustrate function scoping
  console.group('coolFunction');

  // inspect the current data object that landingPage is using
  console.log(this);

  // inspect a specific field of the locally scoped data object
  console.log(JSON.stringify(this._id);

  // close the function scope
  console.groupEnd();
  return this._id;
}

Journalisation des événements et des interactions des utilisateurs #

Exemple simple d’utilisation de l’API Chrome Logging.

Template.landingPage.events({
  'click .selectItemButton':function(){
    // color code and count the user interaction (blue)
    console.count('click .selectItemButton');
  }
});

Journalisation avec des variables de niveau de journalisation #

La journalisation peut souvent encombrer la console, il est donc courant de définir des niveaux de journalisation pour contrôler le détail des données enregistrées. Un modèle courant consiste à spécifier des variables de niveau de journalisation.

var DEBUG = false;
var TRACE = false;
Template.landingPage.events({
  'click .selectItemButton':function(){
    TRACE && console.count('click .selectItemButton');

    Meteor.call('niftyAction', function(errorMessage, result){
        if(errorMessage){
            DEBUG && console.error(errorMessage);    
        }
    });
  }
});

Désactiver la journalisation en production #

Certaines équipes souhaitent laisser les instructions de journal de la console dans leur code, mais ne pas les afficher en production. Ils remplaceront les fonctions de journalisation si une variable n’est pas définie (éventuellement une variable d’environnement). De plus, cela peut être considéré comme une fonctionnalité de sécurité dans certaines situations.

if (!DEBUG_MODE_ON) {
    console = console || {};
    console.log = function(){};

    console.log = function(){};
    console.error = function(){};
    console.count = function(){};
    console.info = function(){};
}

##Winston Si vous avez besoin de quelque chose de plus puissant que les options de journalisation par défaut, vous pouvez envisager un outil comme Winston. Accédez à Atmosphere et recherchez simplement l’un des nombreux packages Winston disponibles.

https://atmospherejs.com/?q=winston

Soyez averti, cependant - Winston est un produit sophistiqué, et bien qu’il expose de nombreuses fonctionnalités, il ajoutera également une couche de complexité à votre application.

Niveau de journalisation #

Une mention spéciale doit être faite pour le package LogLevel développé par la communauté. Il semble trouver un équilibre entre la légèreté et la simplicité d’utilisation, tout en fonctionnant bien avec le pipeline de bundles de Meteor et en préservant les numéros de ligne et les noms de fichiers.

https://atmospherejs.com/practicalmeteor/loglevel
