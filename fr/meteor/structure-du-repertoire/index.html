Avant la sortie de Meteor 1.3, les développeurs de Meteor étaient frustrés par la gestion par Meteor.js des dépendances de fichiers et des variables globales. En réponse, Meteor a établi de nouvelles normes pour les structures de projet afin de rationaliser le système de dépendance du projet. Cette rubrique explique la structure de projet standardisée et les principes qui la sous-tendent.

client Tout le code du répertoire client est exécuté uniquement côté client ou dans le navigateur Web.

client/compatibilité Le répertoire de compatibilité contient du code hérité ou tiers, tel que des bibliothèques jQuery, etc.

lib Le répertoire lib est chargé avant les autres répertoires de votre projet Meteor et est chargé à la fois sur le serveur et sur le client. C’est l’endroit privilégié pour définir les modèles de données, les bibliothèques isomorphes et la logique métier.

importations Le répertoire des importations est un répertoire sur le serveur qui est disponible à la fois pour le serveur et le client, mais uniquement avant que le bundle client ne soit envoyé au client.

paquets Le répertoire packages est l’endroit où les packages personnalisés sont stockés pendant le développement local. Lors de l’utilisation de la commande standard meteor add package:name pour ajouter un package, Meteor recherchera d’abord dans ce répertoire si un package local a le nom de description correspondant dans son fichier package.js. Sinon, il interrogera Atmosphere comme d’habitude.

privé Le répertoire privé contient des fichiers statiques qui ne devraient être disponibles que sur le serveur Web.

Publique Le répertoire public contient des fichiers statiques qui ne sont disponibles que sur le client de l’application. Cela peut inclure des éléments de marque, etc.

serveur Le répertoire du serveur contient des actifs côté serveur. Cela peut inclure une logique d’authentification, des méthodes et d’autres codes pouvant nécessiter des considérations de sécurité.

tests Le répertoire tests est omis par défaut lorsque votre application est regroupée et déployée.

Comme suggéré [par Richard Silverton] [1], c’est une idée pratique de mettre non seulement le répertoire du projet meteor sous contrôle de version, mais aussi son répertoire parent.

De cette façon, vous pouvez garder les fichiers sous contrôle de version sans avoir à faire face à un météore.

[1] : https://blog.tableflip.io/large-meteor-projects-best-practices/

Structures de répertoires classiques #

La première chose que vous devez savoir lors de la structuration de vos applications est que l’outil Meteor contient des répertoires codés en dur avec une logique spécifique. À un niveau très basique, les répertoires suivants sont “préparés” dans le bundle Meteor.

client/                                  # client application code
client/compatibility/                    # legacy 3rd party javascript libraries
imports/                                 # for lazy loading feature 
lib/                                     # any common code for client/server.
packages/                                # place for all your atmosphere packages
private/                                 # static files that only the server knows about
public/                                  # static files that are available to the client
server/                                  # server code
tests/                                   # unit test files (won't be loaded on client or server)

Page de référence : [Meteor Guide > Répertoires spéciaux][1]

[1] : https://guide.meteor.com/structure.html#special-directories

Ordre de chargement du répertoire #

Les fichiers de modèles HTML sont toujours chargés avant tout le reste

Les fichiers commençant par main. sont chargés en dernier

Les fichiers à l’intérieur de n’importe quel répertoire lib/ sont ensuite chargés

Les fichiers avec des chemins plus profonds sont ensuite chargés

Les fichiers sont ensuite chargés dans l’ordre alphabétique du chemin complet

[Lien de référence][1]

[1] : https://guide.meteor.com/structure.html

Page de référence : [Meteor Guide > Application Structure > Default file load order][1]

[1] : https://guide.meteor.com/structure.html#load-order

Structure du répertoire Imports/Modules #

Les versions les plus récentes de Meteor sont livrées avec la prise en charge de ecmascript, alias ES6 ou ES2015. Au lieu des packages, Javascript prend désormais en charge les instructions et les modules “import”, qui remplacent le besoin d’applications de package uniquement. La dernière structure de répertoires est similaire à la structure de packages uniquement, mais utilise le répertoire /imports au lieu de /packages.

imports                                 #
imports/api                             # isomorphic methods 
imports/lib                             # any common code for client/server
imports/client                          # client application code
imports/server                          # server code

Structure de répertoires uniquement pour les packages #

De nombreuses personnes finissent par prendre en charge plusieurs applications et souhaitent partager du code entre les applications. Cela conduit au concept d’architecture de microservice et d’applications tout-package. Essentiellement, le code de l’ensemble de la structure de répertoires classique est refactorisé en packages.

Même s’il n’y a pas de logique codée en dur pour les répertoires dans les packages, nous constatons qu’il est recommandé d’utiliser la structure de répertoires classique lors de la création de packages. Cela crée un chemin de refactorisation naturel au fur et à mesure que les fonctionnalités sont prototypées dans l’application, puis extraites dans des packages à publier et à partager. Les noms de répertoire sont partagés, il y a donc moins de confusion parmi les membres de l’équipe.

client/                                  # client application code
packages/                                # place for all your atmosphere packages
packages/foo/client                      # client application code
packages/foo/lib                         # any common code for client/server
packages/foo/server                      # server code
packages/foo/tests                       # tests
server/                                  # server code

Structure de répertoire en mode mixte #

Et, bien sûr, vous pouvez mélanger ces approches et utiliser à la fois des packages et des importations avec le code spécifique à votre application. Une structure en mode mixte est plus courante dans trois situations : une application franken, qui consiste simplement à tirer un peu d’ici et d’ailleurs sans aucune stratégie globale ; une application qui est activement refactorisée à partir des structures Classic ou Package-Only vers la structure Imports/Modules.

client/                                  # client application code
client/compatibility/                    # legacy 3rd party javascript libraries
imports                                  #
imports/api                              # isomorphic methods 
imports/lib                              # any common code for client/server
imports/client                           # client application code
imports/server                           # server code
lib/                                     # any common code for client/server.
packages/                                # place for all your atmosphere packages
packages/foo/client                      # client application code
packages/foo/lib                         # any common code for client/server
packages/foo/server                      # server code
packages/foo/tests                       # tests
private/                                 # static files that only the server knows about
public/                                  # static files that are available to the client
server/                                  # server code
tests/                                   # unit test files (won't be loaded on client or server)
