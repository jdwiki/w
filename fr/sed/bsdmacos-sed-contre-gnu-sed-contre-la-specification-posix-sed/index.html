BSDmacOS Sed contre GNU Sed contre la spécification POSIX Sed

  • Remplacer toutes les nouvelles lignes par des tabulations
  • Ajoute du texte littéral à une ligne avec la fonction ‘a’

Sur cette page

  • Remplacer toutes les nouvelles lignes par des tabulations
  • Ajoute du texte littéral à une ligne avec la fonction ‘a’

Pour citer la demande de création de sujet de @SnoringFrog :

“L’un des plus gros pièges de l’utilisation de sed est celui des scripts qui échouent (ou réussissent de manière inattendue) parce qu’ils ont été écrits pour l’un et pas pour l’autre. Un simple récapitulatif des différences les plus importantes serait une bonne chose.”

macOS utilise la version BSD de sed[1], qui diffère à bien des égards de la version GNU sed fournie avec les distributions Linux.

Leur dénominateur commun est la fonctionnalité décrétée par POSIX : voir the POSIX sed spec.

L’approche la plus portable consiste à utiliser uniquement les fonctionnalités POSIX, ce qui, toutefois, limite les fonctionnalités :

  • Notamment, POSIX spécifie la prise en charge uniquement des expressions régulières basic, qui ont de nombreuses limitations (par exemple, aucune prise en charge de | (alternance) du tout, pas de prise en charge directe de + et ?) et différentes échappements conditions.

  • Avertissement : **GNU sed (sans -r), prend en charge \|, \+ et \?, qui N’EST PAS compatible POSIX ; utilisez --posix pour désactiver ** (voir ci-dessous).

  • Pour utiliser les fonctionnalités POSIX uniquement :

  • (les deux versions) : utilisez uniquement les options -n et -e (notamment, n’utilisez pas -E ou -r pour activer la prise en charge des expressions régulières extended)

  • GNU sed : ajoutez l’option --posix pour assurer la fonctionnalité POSIX uniquement (vous n’en avez pas strictement besoin, mais sans cela, vous pourriez finir par utiliser par inadvertance des fonctionnalités non POSIX sans vous en rendre compte ; caveat: --posix lui-même n’est pas compatible POSIX)

  • L’utilisation de fonctionnalités POSIX uniquement signifie des exigences de formatage plus strictes (ce qui renonce à de nombreuses commodités disponibles dans GNU sed):

    • Control-character sequences such as \n and \t are generally NOT supported.
    • Labels and branching commands (e.g., b) must be followed by an actual newline or continuation via a separate -e option.
    • See below for details.

Cependant, les deux versions implémentent des extensions au standard POSIX :

  • quelles extensions elles implémentent diffèrent (GNU sed implémente plus).
  • même ces extensions qu’ils les deux implémentent diffèrent partiellement dans la syntaxe.

Si vous devez prendre en charge les DEUX plates-formes (discussion des différences) :

  • Fonctionnalités incompatibles :

  • L’utilisation de l’option -i sans argument (mise à jour sur place sans sauvegarde) est incompatible :

    • BSD sed: MUST use -i ''
    • GNU sed: MUST use just -i (equivalent: -i'') - using -i '' does NOT work.

  • -i active judicieusement la numérotation des lignes per-input-file dans les versions GNU sed et récentes de BSD sed (par exemple, sur FreeBSD 10), mais **PAS sur macOS à partir de 10.12 **. Notez qu’en l’absence de -i toutes les versions numérotent les lignes cumulativement dans les fichiers d’entrée.

  • Si la dernière ligne d’entrée n’a pas de fin de ligne (et est imprimée) :

    • BSD sed: always appends a newline on output, even if the input line doesn’t end in one.
    • GNU sed: preserves the trailing-newline status, i.e., it appends a newline only if the input line ended in one.

  • **Caractéristiques communes:

  • Si vous limitez vos scripts sed à ce que BSD sed prend en charge, ils fonctionneront généralement aussi dans GNU sed - à l’exception notable de l’utilisation de fonctionnalités regex étendues spécifiques à la plate-forme avec -E. Évidemment, vous renoncerez également aux extensions spécifiques à la version GNU. Voir la section suivante.

Directives pour la prise en charge multiplateforme (OS X/BSD, Linux), guidées par les exigences plus strictes de la version BSD :

Notez que les abréviations macOS et Linux sont parfois utilisées ci-dessous pour faire référence aux versions BSD et GNU de sed, respectivement, car ce sont les versions stock sur chaque plate-forme. Cependant, il est possible d’installer GNU sed sur macOS, par exemple, en utilisant Homebrew avec brew install gnu-sed.

Remarque : Sauf lorsque les indicateurs -r et -E sont utilisés (_expressions régulières_étendues), les instructions ci-dessous reviennent à écrire des scripts sed conformes à POSIX.

  • Pour la conformité POSIX, vous devez vous limiter aux BREs POSIX (expressions régulières basiques), qui sont, malheureusement, comme son nom l’indique, assez basique. Mise en garde : ne supposez pas que \|, \+ et \? sont pris en charge : alors que GNU sed les prend en charge (sauf si --posix est utilisé), BSD sed le fait not - ces fonctionnalités ne sont pas conformes à POSIX. Tandis que \+ et \? peuvent être émulés de manière conforme à POSIX : \{1,\} pour \+, \{0,1\} pour \?, \| (alternance) cannot, malheureusement.

  • Pour des expressions régulières plus puissantes, utilisez -E (plutôt que -r) pour prendre en charge les ERE (expressions régulières extended) (GNU sed ne documente pas -E, mais cela fonctionne là comme un alias de -r ; la version plus récente de BSD sed, comme sur FreeBSD 10, prend désormais également en charge -r, mais la version macOS à partir de 10.12 ne le fait pas). Mise en garde : même si l’utilisation de -r / -E signifie que votre commande n’est par définition pas conforme à POSIX, vous devez toujours vous limiter aux [POSIX ERE (expressions régulières étendues)](http ://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap09.html#tag_09_04). Malheureusement, cela signifie que vous ne pourrez pas utiliser plusieurs constructions utiles, notamment :

  • assertions de limites de mots, car elles sont spécifiques à la plate-forme (par exemple, \< sous Linux, [[:<]] sous OS X).

  • back-references inside regular expressions (par opposition aux “back-references” pour les correspondances de groupe de capture dans la chaîne de remplacement des appels de fonction s), car BSD sed ne les prend pas en charge dans les extended regexes (mais, curieusement, le fait dans les basiques, où ils sont mandatés par POSIX).

  • Séquences d’échappement de caractères de contrôle telles que \n et \t :

  • Dans les expressions régulières (à la fois dans les modèles de sélection de ligne et le premier argument de la fonction s), supposez que seul \n est reconnu comme une séquence d’échappement (rarement utilisé, car l’espace de modèle est généralement un seule ligne (sans terminer par \n), mais pas à l’intérieur d’une classe de caractères, de sorte que, par exemple, [^\n] ne fonctionne pas ; (si votre entrée ne contient aucun caractère de contrôle autre que \t , vous pouvez émuler [^\n] avec [[:print:][:blank:]] ; sinon, épissez les caractères de contrôle. en tant que literals[2]) - ** généralement, incluez les caractères de contrôle en tant que littéraux, soit via des chaînes ANSI entre guillemets en C (par exemple, $'\t') dans les shells qui le prennent en charge (bash, ksh, zsh), ou via substitutions de commandes utilisant printf (par exemple, "$(printf '\t')")**.

    • Linux only:
      sed 's/\t/-/' <<<$'a\tb' # -> 'a-b'
    • OSX and Linux:
      sed 's/'$'\t''/-/' <<<$'a\tb' # ANSI C-quoted string
      sed 's/'"$(printf '\t')"'/-/' <<<$'a\tb' # command subst. with printf

  • Dans les chaînes de remplacement utilisées avec la commande s, suppose qu’AUCUNE séquence d’échappement de caractères de contrôle n’est prise en charge, donc, encore une fois, incluez les caractères de contrôle. comme literals, comme ci-dessus.

    • Linux only:
      sed 's/-/\t/' <<<$'a-b' # -> 'a<tab>b'
    • macOS and Linux:
      sed 's/-/'$'\t''/' <<<'a-b' sed 's/-/'"$(printf '\t')"'/' <<<'a-b'

  • Idem pour les arguments textuels des fonctions i et a : n’utilisez pas de séquences de caractères de contrôle - voir ci-dessous.

  • Étiquettes et branchement : les étiquettes ainsi que le nom d’étiquette argument des fonctions b et t doivent être suivis soit d’une nouvelle ligne littérale, soit d’un $'\n' . Alternativement, utilisez plusieurs options -e et terminez chacune juste après le nom de l’étiquette. *Linux uniquement : sed -n '/a/ bLBL ; ré; :LBL p' <<<$'a\nb' # -> 'a'

  • macOS et Linux :

    • EITHER (actual newlines):
      sed -n '/a/ bLBL ré; :LBL p' <<<$'a\nb'
    • OR (spliced-in $\n instances):
      sed -n '/a/ bLBL'$'\n''d; :LBL'$'\n''p' <<<$'a\nb'
    • OR (multiple -e options):
      sed -n -e '/a/ bLBL' -e 'd; :LBL' -e 'p' <<<$'a\nb'

  • Fonctions i et a pour insérer/ajouter du texte : suivez le nom de la fonction par \, suivi soit d’un saut de ligne littéral, soit d’un $'\n' avant de spécifier l’argument de texte.

*Linux uniquement : sed '1 i new first line' <<<$'a\nb' # -> 'new first line<nl>a<nl>b' * OSX et Linux : sed -e '1 i\'$'\n''new first line' <<<$'a\nb'

  • Noter:

    • Without -e, the text argument is inexplicably not newline-terminated on output on macOS (bug?).
    • Do not use control-character escapes such as \n and \t in the text argument, as they’re only supported on Linux.
    • If the text argument therefore has actual interior newlines, \-escape them.
    • If you want to place additional commands after the text argument, you must terminate it with an (unescaped) newline (whether literal or spliced in), or continue with a separate -e option (this is a general requirement that applies to all versions).

  • À l’intérieur des fonctions listes (plusieurs appels de fonctions entre {...}), assurez-vous de terminer également la dernière fonction, avant le } fermant, avec ;. *Linux uniquement :

    • sed -n '1 {p;q}' <<<$'a\nb' # -> 'a'

  • macOS et Linux :

    • sed -n '1 {p;q;}' <<<$'a\nb'

Fonctionnalités spécifiques à GNU sed totalement absentes de sed BSD :

Fonctionnalités GNU qui vous manqueront si vous devez prendre en charge les deux plates-formes :

  • Diverses options de correspondance et de substitution de regex (à la fois dans les modèles de sélection de ligne et le premier argument de la fonction s):

  • ** L’option I pour la correspondance des regex INsensibles à la casse (incroyablement, BSD sed ne le supporte pas du tout).**

  • L’option M pour la correspondance multi-lignes (où ^ / $ correspond au début / à la fin de chaque ligne)

  • Pour des options supplémentaires spécifiques à la fonction s, voir https://www.gnu.org/software/sed/manual/sed.html#The-_0022s_0022-Command

  • Séquences d’échappement

  • Séquences d’échappement liées à la substitution telles que \u dans l’argument de remplacement de la fonction s/// qui permettent la manipulation de la sous-chaîne, dans certaines limites ; par exemple, sed 's/^./\u&/' <<<'dog' # -> 'Dog' - voir http://www.gnu.org/software/sed/manual/sed.html#The -_0022s_0022-Commande

  • Séquences d’échappement de caractères de contrôle : en plus de \n, \t, …, échappements basés sur des points de code ; par exemple, tous les échappements suivants (hex., octal, décimal) représentent un guillemet simple (') : \x27, \o047, \d039 - voir https://www.gnu. org/software/sed/manual/sed.html#Escapes

  • Extensions d’adresse, telles que first~step pour correspondre à chaque étape-ième ligne, addr, +N pour correspondre à N lignes suivant addr, … - voir http://www.gnu .org/software/sed/manual/sed.html#Adresses

[1] La version macOS sed est plus ancienne que la version sur d’autres systèmes de type BSD tels que FreeBSD et PC-BSD. Malheureusement, cela signifie que vous ne pouvez pas supposer que les fonctionnalités qui fonctionnent dans FreeBSD, par exemple, fonctionneront [de la même manière] sur macOS.

[2] La chaîne entre guillemets ANSI C $'\001\002\003\004\005\006\007\010\011\013\014\015\016\017\020\021\022\ 023\024\025\026\027\030\031\032\033\034\035\036\037\177' contient tous les caractères de contrôle ASCII sauf \n (et NUL), vous pouvez donc l’utiliser dans combinaison avec [:print:] pour une émulation assez robuste de [^\n] : '[[:print:]'$'\001\002\003\004\005\006\007\010\011\013\014\015\016\017\020\021\022\023\024\ 025\026\027\030\031\032\033\034\035\036\037\177'']

Remplacer toutes les nouvelles lignes par des tabulations #

Remarque : Par souci de concision, les commandes utilisent chaînes ici (<<<) et chaînes entre guillemets ANSI ( $'...'). Ces deux fonctionnalités du shell fonctionnent dans bash, ksh et zsh.

# GNU Sed
$ sed ':a;$!{N;ba}; s/\n/\t/g' <<<$'line_1\nline_2\nline_3'
line_1 line_2 line_3

# BSD Sed equivalent (multi-line form)
sed <<<$'line_1\nline_2\nline_3' '
:a
$!{N;ba
}; s/\n/'$'\t''/g'

# BSD Sed equivalent (single-line form, via separate -e options)
sed -e ':a' -e '$!{N;ba' -e '}; s/\n/'$'\t''/g' <<<$'line 1\nline 2\nline 3'

Remarques de BSD Sed :

  • Notez la nécessité de terminer les étiquettes (:a) et les commandes de branchement (ba) soit avec des nouvelles lignes réelles, soit avec des options -e séparées.

  • Étant donné que les séquences d’échappement de caractères de contrôle telles que \t ne sont pas prises en charge dans la chaîne de remplacement, une tabulation ANSI C-quoted literal est épissée dans la chaîne de remplacement. (Dans la partie regex, BSD Sed uniquement reconnaît \n comme une séquence d’échappement).

Ajoute du texte littéral à une ligne avec la fonction ‘a’ #

Remarque : Par souci de concision, les commandes utilisent chaînes ici (<<<) et chaînes entre guillemets ANSI ( $'...'). Ces deux fonctionnalités du shell fonctionnent dans bash, ksh et zsh.

 # GNU Sed
 $ sed '1 a appended text' <<<'line 1'
 line 1
 appended text

 # BSD Sed (multi-line form)
 sed '1 a\
 appended text' <<<'line 1'

 # BSD Sed (single-line form via a Bash/Ksh/Zsh ANSI C-quoted string)
 sed $'1 a\\\nappended text' <<<'line 1'

Notez que BSD Seed nécessite un \ suivi d’un actual newline pour transmettre le texte à ajouter. Il en va de même pour les fonctions associées i (insérer) et c (supprimer et insérer).
