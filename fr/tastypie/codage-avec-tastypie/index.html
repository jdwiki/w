Tastypie est un framework d’API de services Web pour Django. Il fournit une abstraction pratique, mais puissante et hautement personnalisable pour créer des interfaces de style REST. Tastypie facilite l’exposition de vos modèles, mais vous donne un contrôle total sur ce que vous exposez, vous permettant d’abstraire la base de données autant que nécessaire. Tastypie facilite également l’intégration avec des sources de données non ORM. Par conséquent, Tastypie peut être utilisé avec des bases de données ORM et non ORM.

Démarrage rapide #

  1. Ajoutez une tarte savoureuse à INSTALLED_APPS.

  2. Créez un répertoire api dans votre application avec un init.py nu.

  3. Créez un fichier <my_app>/api/resources.py et placez-y les éléments suivants :

    from tastypie.resources import ModelResource from my_app.models import MyModel

    class MyModelResource(ModelResource): class Meta: queryset = MyModel.objects.all() allowed_methods = [‘get’]

Dans votre racine URLconf, ajoutez le code suivant (autour de l’emplacement du code administrateur) :

from django.conf.urls import url, include
from tastypie.api import Api
from my_app.api.resources import MyModelResource

v1_api = Api(api_name='v1')
v1_api.register(MyModelResource())

urlpatterns = [
  # ...more URLconf bits here...
  # Then add:
  url(r'^api/', include(v1_api.urls)),
]

Installation de Tastypie #

Tastypie peut être installé avec la gestion des packages python, c’est-à-dire pip ou nous pouvons directement extraire le code de Github

  1. pip installer django-tastypie
  2. Commander depuis [Github][1]

[1] : https://github.com/django-tastypie/django-tastypie

Pourquoi Tastypie ? #

Il existe d’autres frameworks d’API pour Django. Vous devez évaluer les options disponibles et décider par vous-même. Cela dit, voici quelques raisons courantes pour une tarte savoureuse.

  • Vous avez besoin d’une API qui est RESTful et qui utilise bien HTTP.
  • Vous souhaitez favoriser des relations profondes.
  • Vous ne voulez PAS avoir à écrire votre propre sérialiseur pour faire le sortie à droite.
  • Vous voulez un framework d’API peu magique, très flexible et correspond bien au domaine du problème.
  • Vous voulez/avez besoin d’une sérialisation XML traitée de la même manière que JSON (et YAML est là aussi).

Pour développer une API avec Django, nous avons de nombreuses options. Les principales options sont Tastypie et Django Rest Framework(DRF).

** Qu’est-ce qui fait un framework d’API décent ? **

Ces fonctionnalités :

  1. mise en page
  2. affichage des données avec validation
  3. Publication des métadonnées avec les ensembles de requêtes
  4. Découverte d’API
  5. bonne gestion des réponses HTTP
  6. mise en cache
  7. sérialisation
  8. étranglement
  9. autorisations
  10. authentification

Les cadres d’API appropriés ont également besoin :

  1. Très bonne couverture de test de leur code

  2. Performances décentes

  3. Documents

  4. Une communauté active pour faire avancer et soutenir le framework

  • Si vous prenez ces facteurs, à l’heure actuelle, seuls deux frameworks d’API valent la peine d’être utilisés (selon les vues de l’utilisateur), django-tastypie et django-rest-framework.*

Quel est le meilleur? django-tastypie ou django-rest-framework ?

  • Je dis qu’ils sont égaux. Vous ne pouvez tout simplement pas vous tromper avec l’un ou l’autre. Les auteurs et les communautés derrière les deux sont actifs, le code est solide et testé. Et voici mes réflexions spécifiques sur les deux :*

Tastypie :

Avantages :

  • Facile à démarrer et à fournir des fonctionnalités de base OOB (out de la boîte)
  • La plupart du temps, vous n’aurez pas affaire à des concepts avancés de Django comme les CBV, les formulaires, etc.
  • Code plus lisible et moins de magie !
  • Si vos modèles sont NON-ORM, foncez.

Désavantages:

  • Ne suit pas strictement Django idiomatique (attention bien python et les philosophies de django sont assez différentes)
  • Probablement un peu difficile de personnaliser les API une fois que vous êtes devenu grand
  • Pas d’autorisation O

DRF :

Avantages :

  • Suivez django idiomatique. (Si vous connaissez django sur le bout des doigts, et très à l’aise avec CBV, Forms etc sans aucun doute allez-y)

  • Fournit une fonctionnalité REST prête à l’emploi à l’aide de ModelViewSets. À en même temps, offre un meilleur contrôle pour la personnalisation à l’aide CustomSerializer, APIView, GenericViews etc.

  • Meilleure authentification.

  • Plus facile d’écrire des classes d’autorisation personnalisées.

  • Fonctionne très bien et surtout très facile à faire fonctionner avec la 3ème bibliothèques de parties et OAuth.

  • DJANGO-REST-AUTH mérite d’être mentionné LIBRARY pour Auth/SocialAuthentication/Enregistrement.

Désavantages:

  • Si vous ne connaissez pas très bien Django, n’y allez pas.
  • La magie! Parfois très difficile à comprendre la magie.
  • Parce qu’il a été écrit au-dessus des CBV de Django qui sont à leur tour de nature assez complexe.
  • A une courbe d’apprentissage abrupte.

Premiers pas avec Tastypie #

Tastypie est une application réutilisable (c’est-à-dire qu’elle ne repose que sur son propre code et se concentre uniquement sur la fourniture d’une API de style REST) ​​et convient pour fournir une API à n’importe quelle application sans avoir à modifier les sources de cette application.

Les besoins de tout le monde ne sont pas les mêmes, donc Tastypie fait tout son possible pour fournir de nombreux crochets pour remplacer ou étendre son fonctionnement.

À titre d’exemple, nous allons ajouter une API à un simple blog candidature.

La seule configuration obligatoire consiste à ajouter ‘tastypie’ à vos INSTALLED_APPS. Ce n’est pas strictement nécessaire, car Tastypie n’a que deux modèles non requis, mais peut faciliter l’utilisation.

Voici monapplication/models.py :

from tastypie.utils.timezone import now
from django.contrib.auth.models import User
from django.db import models
from django.utils.text import slugify


class Entry(models.Model):
    user = models.ForeignKey(User)
    pub_date = models.DateTimeField(default=now)
    title = models.CharField(max_length=200)
    slug = models.SlugField(null=True, blank=True)
    body = models.TextField()

Créer des ressources

L’architecture de style REST parle de ressources, donc sans surprise l’intégration avec Tastypie implique la création de classes de ressources. Pour notre application simple, nous allons créer un fichier pour ceux-ci dans myapp/api.py, bien qu’ils puissent se trouver n’importe où dans votre application :

monapp/api.py #

from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'

Dans REST, tout est une ressource, cela signifie que les objets sont décrits comme des ressources. Ainsi, pour créer une API Tastypie REST, il faut créer une classe de ressources. Pour créer une ressource, nous devons sous-classer la classe ModelResource. Cette classe EntryResource vérifiera tous les champs non relationnels du modèle Entry et créera ses propres champs Api. Cela fonctionne exactement comme le ModelForm de Django Forms.

Connecter la ressource : Après avoir créé la ressource, nous devons dire au Django qu’une ressource est créée en reliant notre EntryResource à l’URL.

URL.py #

from django.conf.urls import url, include
from myapp.api import EntryResource

entry_resource = EntryResource()

urlpatterns = [
    url(r'^blog/', include('myapp.urls')),
    url(r'^api/', include(entry_resource.urls)),
]

Maintenant, si nous vérifions localhost:8000/api/entry/, nous obtiendrons la réponse de l’API.

(Le resource_name dans la classe Meta est facultatif. S’il n’est pas fourni, il est automatiquement généré à partir du nom de la classe, supprimant toutes les instances de Resource et mettant la chaîne en minuscules. Ainsi, EntryResource deviendrait simplement une entrée. C’est pourquoi dans l’URL nous avons spécifié ../entry/ )

Autorisation Tastypie #

Jusqu’à présent, nous avons essayé d’obtenir une demande. (Si vous avez besoin d’essayer d’autres requêtes HTTP, utilisez des outils de test d’API comme curl ou postman)

Si vous essayez d’envoyer un POST/PUT/DELETE à la ressource, vous obtenez des erreurs “401 Unauthorized”. Pour des raisons de sécurité, Tastypie est livré avec la classe d’autorisation définie sur ReadOnlyAuthorization. Cela permet d’exposer en toute sécurité sur le Web, mais nous empêche de faire POST/PUT/DELETE. Activons-les.

monapp/api.py #

from tastypie.authorization import Authorization
from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'
        authorization = Authorization()
