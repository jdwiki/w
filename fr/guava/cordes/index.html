Vérification d’une chaîne pour les caractères indésirables #

En tant que développeur, vous vous retrouvez souvent à gérer des chaînes qui ne sont pas créées par votre propre code.

Ceux-ci seront souvent fournis par des bibliothèques tierces, des systèmes externes ou même des utilisateurs finaux. La validation de chaînes dont la provenance n’est pas claire est considérée comme l’une des caractéristiques de la programmation défensive et, dans la plupart des cas, vous souhaiterez rejeter les entrées de chaîne qui ne répondent pas à vos attentes.

Un cas assez courant est celui où vous ne voudriez autoriser que des caractères alphanumériques dans une chaîne d’entrée, nous allons donc l’utiliser comme exemple. En langage Java standard, les deux méthodes suivantes ont le même objectif :

public static boolean isAlphanumeric(String s) {
    for (char c : s.toCharArray()) {
        if (!Character.isLetterOrDigit(c)) {
            return false;
        }
    }

    return true;
}

public static boolean isAlphanumeric(String s) {
    return s.matches("^[0­-9a­-zA­-Z]*$");
}

La première version convertit la chaîne en un tableau de caractères, puis utilise la méthode statique isLetterOrDigit de la classe Character pour déterminer si les caractères contenus dans le tableau sont alphanumériques ou non. Cette approche est prévisible et lisible, bien qu’un peu verbeuse.

La deuxième version utilise une expression régulière pour atteindre le même objectif. Il est plus concis, mais peut être quelque peu énigmatique pour les développeurs ayant une connaissance limitée ou nulle des expressions régulières.

Guava introduit la classe CharMatcher pour faire face à ces types de situations. Notre test alphanumérique, utilisant Guava, ressemblerait à ceci :

import static com.google.common.base.CharMatcher.javaLetterOrDigit;

/* ... */

public static boolean isAlphanumeric(String s) {
    return javaLetterOrDigit().matchesAllOf(s);
}

Le corps de la méthode ne contient qu’une seule ligne, mais il se passe en fait beaucoup de choses ici, alors décomposons les choses un peu plus loin.

Si vous regardez l’API de la classe CharMatcher de Guava, vous remarquerez qu’elle implémente l’interface Predicate<Character>. Si vous deviez créer vous-même une classe qui implémente Predicate<Character>, elle pourrait ressembler à ceci :

import com.google.common.base.Predicate;

public class AlphanumericPredicate implements Predicate<Character> {
    @Override
    public boolean apply(Character c) {
        return Character.isLetterOrDigit(c);
    }
}

Dans Guava, comme dans un certain nombre d’autres langages de programmation et bibliothèques qui répondent à un style de programmation fonctionnel, un prédicat est une construction qui évalue une entrée donnée à vrai ou à faux. Dans l’interface Predicate<T> de Guava, cela est rendu évident par la présence de la seule méthode boolean apply(T t). La classe CharMatcher est construite sur ce concept et évaluera un caractère ou une séquence de caractères pour vérifier s’ils correspondent ou non aux critères définis par l’instance CharMatcher utilisée.

Guava fournit actuellement les correspondances de caractères prédéfinies suivantes :

Matcher Descriptif
tout() Correspond à n’importe quel caractère.
aucun() Ne correspond à aucun caractère.
javaDigit() Correspond aux chiffres, selon la définition Java.
javaMajuscule() Correspond à n’importe quel caractère majuscule, selon la définition de Java.
javaLowerCase() Correspond à n’importe quel caractère minuscule, selon la définition de Java.
javaLetter() Correspond à n’importe quelle lettre, selon la définition de Java.
javaLetterOrDigit() Correspond à n’importe quelle lettre ou chiffre, selon la définition de Java.
javaIsoControl() Correspond à n’importe quel caractère de contrôle ISO, selon la définition de Java.
ascii() Correspond à n’importe quel caractère du jeu de caractères ASCII.
invisible() Correspond aux caractères qui ne sont pas visibles, selon la norme Unicode.
chiffre() Correspond à n’importe quel chiffre, conformément à la spécification Unicode.
espace blanc() Correspond à n’importe quel caractère d’espacement, conformément à la spécification Unicode.
breakingWhitespace() Correspond à tout caractère d’espace blanc de rupture, conformément à la spécification Unicode.
singleWidth() Correspond à n’importe quel caractère simple largeur.

Si vous avez lu le tableau ci-dessus, vous avez sans aucun doute remarqué la quantité de définitions et de spécifications impliquées dans la détermination des caractères appartenant à une certaine catégorie. L’approche de Guava, jusqu’à présent, a consisté à fournir des wrappers CharMatcher pour un certain nombre de catégories de caractères définies par Java, et vous pouvez consulter l’API de la classe Character de Java pour obtenir plus d’informations sur ces catégories. D’autre part, Guava tente de fournir un certain nombre d’instances “CharMatcher” conformes à la spécification Unicode actuelle. Pour plus de détails, consultez la documentation de l’API “CharMatcher”.

Pour en revenir à notre exemple de vérification d’une chaîne pour les caractères indésirables, les méthodes CharMatcher suivantes fournissent les fonctionnalités dont vous avez besoin pour vérifier si l’utilisation des caractères d’une chaîne donnée répond à vos besoins :

  • boolean matchesNoneOf(CharSequence sequence) Renvoie true si aucun des caractères de la chaîne d’argument ne correspond à l’instance CharMatcher.

  • boolean matchesAnyOf(Séquence CharSequence) Renvoie true si au moins un caractère de la chaîne d’argument correspond à l’instance CharMatcher.

  • boolean matchesAllOf(CharSequence sequence) Renvoie true si tous les caractères de la chaîne d’argument correspondent à l’instance de CharMatcher.

Recherche et comptage de caractères dans une chaîne #

Pour vous aider à trouver et à compter les caractères d’une chaîne, CharMatcher propose les méthodes suivantes :

  • int indexIn(séquence CharSequence) Renvoie l’index du premier caractère qui correspond à l’instance CharMatcher. Renvoie -1 si aucun caractère ne correspond.

  • int indexIn (séquence CharSequence, int start) Renvoie l’index du premier caractère après la position de départ spécifiée qui correspond à l’instance CharMatcher. Renvoie -1 si aucun caractère ne correspond.

  • int lastIndexIn (séquence CharSequence) Renvoie l’index du dernier caractère qui correspond à l’instance CharMatcher. Renvoie -1 si aucun caractère ne correspond.

  • int countIn(séquence CharSequence) Renvoie le nombre de caractères correspondant à l’instance CharMatcher.

En utilisant ces méthodes, voici une application console simple appelée NonAsciiFinder qui prend une chaîne comme argument d’entrée. Tout d’abord, il imprime le nombre total de caractères non ASCII contenus dans la chaîne. Par la suite, il imprime la représentation Unicode de chaque caractère non ASCII qu’il rencontre. Voici le code :

import com.google.common.base.CharMatcher;

public class NonAsciiFinder {
    private static final CharMatcher NON_ASCII = CharMatcher.ascii().negate();

    public static void main(String[] args) {
        String input = args[0];
        int nonAsciiCount = NON_ASCII.countIn(input);

        echo("Non-ASCII characters found: %d", nonAsciiCount);

        if (nonAsciiCount > 0) {
            int position = -­1;
            char character = 0;

            while (position != NON_ASCII.lastIndexIn(input)) {
                position = NON_ASCII.indexIn(input, position + 1);
                character = input.charAt(position);
                
                echo("%s => \\u%04x", character, (int) character);
            }
        }
    }

    private static void echo(String s, Object... args) {
        System.out.println(String.format(s, args));
    }
}

Notez dans l’exemple ci-dessus comment vous pouvez simplement inverser un CharMatcher en appelant sa méthode negate. De même, le CharMatcher ci-dessous correspond à tous les caractères à double largeur et est créé en annulant le CharMatcher prédéfini pour les caractères à simple largeur.

final static CharMatcher DOUBLE_WIDTH = CharMatcher.singleWidth().negate();

L’exécution de l’application “NonAsciiFinder” produit le résultat suivant :

$> java NonAsciiFinder "Maître Corbeau, sur un arbre perché"
Non­-ASCII characters found: 2
î => \u00ee
é => \u00e9

$> java NonAsciiFinder "古池や蛙飛び込む水の音"
Non­ASCII characters found: 11
古 => \u53e4
池 => \u6c60
や => \u3084
蛙 => \u86d9
飛 => \u98db
び => \u3073
込 => \u8fbc
む => \u3080
水 => \u6c34
の => \u306e
音 => \u97f3

Suppression des caractères indésirables d’une chaîne #

L’exemple [Vérification d’une chaîne pour les caractères indésirables](https://www.wikiod.com/fr/guava/cordes#Vérification d’une chaîne pour les caractères indésirables), décrit comment tester et rejeter les chaînes qui ne répondent pas à certains critères. Évidemment, rejeter purement et simplement une entrée n’est pas toujours possible, et parfois vous devez vous contenter de ce que vous recevez. Dans ces cas, un développeur prudent tentera de nettoyer les chaînes fournies pour supprimer tous les caractères susceptibles de déclencher un traitement ultérieur.

Pour supprimer, couper et remplacer les personnages indésirables, l’arme de choix sera à nouveau la classe “CharMatcher” de Guava.

Suppression de caractères #

Les deux méthodes “CharMatcher” qui nous intéressent dans cette section sont :

  • Chaîne retenueDe(séquence CharSequence) Renvoie une chaîne contenant tous les caractères correspondant à l’instance CharMatcher.

  • String removeFrom(CharSequence sequence) Renvoie une chaîne contenant tous les caractères qui ne correspondent pas à l’instance CharMatcher.

Par exemple, nous utiliserons CharMatcher.digit(), une instance prédéfinie de CharMatcher qui, sans surprise, ne correspond qu’aux chiffres.

String rock = "1, 2, 3 o'clock, 4 o'clock rock!";

CharMatcher.digit().retainFrom(rock); // "1234"
CharMatcher.digit().removeFrom(rock); // ", , o'clock, o'clock rock!"
CharMatcher.digit().negate().removeFrom(rock); // "1234"

La dernière ligne de cet exemple montre que removeFrom est en fait l’opération inverse de retainFrom. Invoquer retainFrom sur un CharMatcher a le même effet que d’invoquer removeFrom sur une version inversée de ce CharMatcher.

Découpage des caractères de début et de fin #

La suppression des caractères de début et de fin est une opération très courante, le plus souvent utilisée pour supprimer les espaces blancs des chaînes. CharMatcher de Guava propose ces méthodes de découpage :

  • Chaîne trimLeadingFrom(Séquence CharSequence) Supprime tous les caractères de début qui correspondent à l’instance CharMatcher.

  • Chaîne trimTrailingFrom(Séquence CharSequence) Supprime tous les caractères de fin qui correspondent à l’instance CharMatcher.

  • String trimFrom(CharSequence sequence) Supprime tous les caractères de début et de fin qui correspondent à l’instance CharMatcher.

Lorsqu’elles sont utilisées avec CharMatcher.whitespace(), ces méthodes prendront efficacement en charge tous vos besoins de suppression des espaces :

CharMatcher.whitespace().trimFrom("   Too much space   "); // returns "Too much space"

Remplacement de caractères #

Souvent, les applications remplacent les caractères qui ne sont pas autorisés dans une certaine situation par un caractère fictif. Pour remplacer des caractères dans une chaîne, l’API de CharMatcher fournit les méthodes suivantes :

  • String replaceFrom(CharSequence sequence, char replacement) Remplace toutes les occurrences de caractères qui correspondent à l’instance CharMatcher par le caractère de remplacement fourni.

  • String replaceFrom(CharSequence sequence, CharSequence replacement) Remplace toutes les occurrences de caractères qui correspondent à l’instance CharMatcher par la séquence de caractères de remplacement fournie (chaîne).

  • String collapseFrom(CharSequence sequence, char replacement) Remplace les groupes de caractères consécutifs qui correspondent à l’instance CharMatcher par une seule instance du caractère de remplacement fourni.

  • String trimAndCollapseFrom(CharSequence sequence, char replacement) Se comporte de la même manière que collapseFrom, mais les groupes correspondants au début et à la fin sont supprimés plutôt que remplacés.

Examinons un exemple qui montre comment le comportement de ces méthodes diffère. Disons que nous créons une application qui permet à l’utilisateur de spécifier les noms de fichiers de sortie. Pour nettoyer l’entrée fournie par l’utilisateur, nous créons une instance CharMatcher qui est une combinaison de l’espace blanc prédéfini CharMatcher et d’un CharMatcher personnalisé qui spécifie un ensemble de caractères que nous préférerions éviter dans nos noms de fichiers.

CharMatcher illegal = CharMatcher.whitespace().or(CharMatcher.anyOf("<>:|?*\"/\\"));

Maintenant, si nous invoquons les méthodes de remplacement discutées comme suit sur un nom de fichier qui a un besoin urgent de nettoyage :

String filename = "<A::12> first draft???";

System.out.println(illegal.replaceFrom(filename, '_'));
System.out.println(illegal.collapseFrom(filename, '_'));
System.out.println(illegal.trimAndCollapseFrom(filename, '_'));

Nous verrons la sortie ci-dessous dans notre console.

_A__12___first_draft___
_A_12_first_draft_
A_12_first_draft

Fractionner une chaîne dans une liste #

Pour diviser les chaînes, Guava introduit la classe Splitter.

Pourquoi ne pas utiliser les capacités de fractionnement de Java ? #

En règle générale, Guava ne duplique pas les fonctionnalités facilement disponibles en Java. Pourquoi alors avons-nous besoin d’une classe Splitter supplémentaire ? Les méthodes de fractionnement de la classe String de Java ne nous fournissent-elles pas tous les mécanismes de fractionnement de chaînes dont nous aurons jamais besoin ?

La façon la plus simple de répondre à cette question est de donner quelques exemples. Tout d’abord, nous traiterons du duo de tireurs d’élite suivant :

String gunslingers = "Wyatt Earp+Doc Holliday";

Pour essayer de séparer le légendaire homme de loi et son ami dentiste, nous pourrions essayer ce qui suit :

String[] result = gunslingers.split("+"); // wrong

Lors de l’exécution, cependant, nous sommes confrontés à l’exception suivante :

Exception in thread "main" java.util.regex.PatternSyntaxException:
Dangling meta character '+' near index 0

Après un facepalm involontaire, nous nous souvenons rapidement que la méthode split de String prend une expression régulière comme argument, et que le caractère + est utilisé comme quantificateur dans les expressions régulières. La solution est alors d’échapper le caractère +, ou de l’enfermer dans une classe de caractères.

String[] result = gunslingers.split("\\+");
String[] result = gunslingers.split("[+]");

Après avoir résolu ce problème avec succès, nous passons aux trois mousquetaires.

String musketeers = ",Porthos , Athos ,Aramis,";

La virgule n’a pas de signification particulière dans les expressions régulières, alors comptons les mousquetaires en appliquant la méthode String.split() et en obtenant la longueur du tableau résultant.

System.out.println(musketeers.split(",").length);

Ce qui donne le résultat suivant dans la console :

4

Quatre ? Étant donné que la chaîne contient une virgule de début et de fin, un résultat de cinq aurait été dans le domaine des attentes normales, mais quatre ? Il s’avère que le comportement de la méthode split de Java est de conserver le début, mais de supprimer les chaînes vides de fin, de sorte que le contenu réel du tableau est ["", "Porthos", " Athos ", "Aramis"] .

Puisque nous n’avons pas besoin de chaînes vides, ni en début ni en fin, filtrons-les avec une boucle :

for (String musketeer : musketeers.split(",")) {
    if (!musketeer.isEmpty()) {
        System.out.println(musketeer);
    }
}

Cela nous donne la sortie suivante :

Porthos 
 Athos
Aramis

Comme vous pouvez le voir dans la sortie ci-dessus, les espaces supplémentaires avant et après les séparateurs de virgule ont été conservés dans la sortie. Pour contourner cela, nous pouvons supprimer les espaces inutiles, ce qui donnera finalement le résultat souhaité :

for (String musketeer : musketeers.split(",")) {
    if(!musketeer.isEmpty()) {
        System.out.println(musketeer.trim());
    }
}

(Alternativement, nous pourrions également adapter l’expression régulière pour inclure des espaces blancs entourant les séparateurs de virgule. Cependant, gardez à l’esprit que les espaces de début avant la première entrée ou les espaces de fin après la dernière entrée seraient toujours conservés.)

Après avoir lu les exemples ci-dessus, nous ne pouvons pas nous empêcher de conclure que le fractionnement de chaînes avec Java est au mieux légèrement ennuyeux.

Fractionner des cordes avec Guava #

La meilleure façon de démontrer comment Guava transforme la division des chaînes en une expérience relativement indolore est de traiter à nouveau les deux mêmes chaînes, mais cette fois en utilisant la classe Splitter de Guava.

List<String> gunslingers = Splitter.on('+')
        .splitToList("Wyatt Earp+Doc Holliday");

List<String> musketeers = Splitter.on(",")
        .omitEmptyStrings()
        .trimResults()
        .splitToList(",Porthos , Athos ,Aramis,");

Comme vous pouvez le voir dans le code ci-dessus, Splitter expose une API fluide et vous permet de créer des instances via une série de méthodes de fabrique statique :

  • Séparateur statique activé (séparateur de caractères) Permet de spécifier le séparateur en tant que caractère.

  • Séparateur statique activé (séparateur de chaînes) Vous permet de spécifier le séparateur sous forme de chaîne.

  • Séparateur statique activé (CharMatcher separatorMatcher) Vous permet de spécifier le séparateur en tant que Guava CharMatcher.

  • Séparateur statique activé (Pattern separatorPattern) Vous permet de spécifier le séparateur sous la forme d’une expression régulière Java Pattern.

  • Static Splitter onPattern(String separatorPattern) Vous permet de spécifier le séparateur sous la forme d’une chaîne d’expression régulière.

En plus de ces méthodes de fabrique basées sur des séparateurs, il existe également une méthode static Splitter fixedLength(int length) pour créer des instances Splitter qui divisent les chaînes en morceaux de la longueur spécifiée.

Une fois l’instance Splitter créée, un certain nombre de modificateurs peuvent être appliqués :

  • Splitter omitEmptyStrings() Indique au Splitter d’exclure les chaînes vides des résultats.

  • Splitter trimResults() Indique au Splitter de découper les résultats à l’aide de l’espace blanc prédéfini CharMatcher.

  • Splitter trimResults(CharMatcher trimmer) Indique au Splitter de découper les résultats à l’aide du CharMatcher spécifié.

Après avoir créé (et éventuellement modifié) un Splitter, il peut être invoqué sur une séquence de caractères en invoquant sa méthode split, qui renverra un objet de type Iterable<String>, ou sa méthode splitToList, qui renverra un objet (immuable) de type List<String>.

Vous vous demandez peut-être dans quels cas il serait avantageux d’utiliser la méthode split (qui renvoie un Iterable) au lieu de la méthode splitToList (qui renvoie le type List le plus couramment utilisé). La réponse courte à cette question est la suivante : vous souhaitez probablement utiliser la méthode split uniquement pour traiter de très grandes chaînes. La réponse légèrement plus longue est que parce que la méthode split renvoie un Iterable, les opérations de fractionnement peuvent être évaluées paresseusement (au moment de l’itération), supprimant ainsi la nécessité de conserver le résultat complet de l’opération de fractionnement en mémoire.
