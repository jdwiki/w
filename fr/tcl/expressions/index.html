Un autre avantage de l’utilisation de chaînes d’expression entre accolades est que le compilateur d’octets peut généralement générer un code plus efficace (5 à 10 fois plus rapide) à partir de celles-ci.

Les problèmes avec les expressions sans accolades #

Il est recommandé de fournir des arguments de chaîne d’expression sous forme de chaînes entre accolades. La rubrique “Double substitution” décrit les raisons importantes derrière la même chose.

La commande expr évalue une chaîne d’expression basée sur un opérateur pour calculer une valeur. Cette chaîne est construite à partir des arguments de l’invocation.

expr 1 + 2    ; # three arguments
expr "1 + 2"  ; # one argument
expr {1 + 2}  ; # one argument

Ces trois invocations sont équivalentes et la chaîne d’expression est la même.

Les commandes if, for et while utilisent le même code d’évaluateur pour leurs arguments de condition :

if {$x > 0} ...
for ... {$x > 0} ... ...
while {$x > 0} ...

La principale différence est que la chaîne d’expression de condition doit toujours être un seul argument.

Comme pour chaque argument dans une invocation de commande en Tcl, le contenu peut ou non être soumis à une substitution, selon la façon dont il est cité/échappé :

set a 1
set b 2
expr $a + $b   ; # expression string is {1 + 2}
expr "$a + $b" ; # expression string is {1 + 2}
expr \$a + \$b ; # expression string is {$a + $b}
expr {$a + $b} ; # expression string is {$a + $b}

Il y a une différence dans les troisième et quatrième cas car les barres obliques inverses / accolades empêchent la substitution. Le résultat est toujours le même, puisque l’évaluateur à l’intérieur de expr peut lui-même effectuer une substitution de variable Tcl et transformer la chaîne en {1 + 2}.

set a 1
set b "+ 2"
expr $a $b   ; # expression string is {1 + 2}
expr "$a $b" ; # expression string is {1 + 2}
expr {$a $b} ; # expression string is {$a $b}: FAIL!

Ici, nous avons des problèmes avec l’argument entre accolades : lorsque l’évaluateur dans expr effectue des substitutions, la chaîne d’expression a déjà été analysée en opérateurs et opérandes, donc ce que l’évaluateur voit est une chaîne composée de deux opérandes sans opérateur entre eux. (Le message d’erreur est “opérateur manquant à ___ dans l'expression "$a _@_$b"”.)

Dans ce cas, la substitution de variable avant l’appel de expr a empêché une erreur. L’accolade de l’argument empêchait la substitution de variable jusqu’à l’évaluation de l’expression, ce qui provoquait une erreur.

Des situations comme celle-ci peuvent se produire, le plus souvent lorsqu’une expression à évaluer est transmise en tant que variable ou paramètre. Dans ces cas, il n’y a pas d’autre choix que de laisser l’argument sans accolades pour permettre à l’évaluateur d’argument de “décompresser” la chaîne d’expression pour la livraison à expr.

Dans la plupart des autres cas, cependant, renforcer l’expression ne fait aucun mal et peut en effet éviter de nombreux problèmes. Quelques exemples de ceci :

Double remplacement

set a {[exec make computer go boom]}
expr $a      ; # expression string is {[exec make computer go boom]}
expr {$a}    ; # expression string is {$a}

La forme sans accolades effectuera la substitution de commande, qui est une commande qui détruit l’ordinateur d’une manière ou d’une autre (ou crypte ou formate le disque dur, ou quoi que ce soit d’autre). La forme entre accolades effectuera une substitution de variable puis essaiera (et échouera) de faire quelque chose de la chaîne “[exec make computer go boom]”. Catastrophe évitée.

** Boucles sans fin **

set i 10
while "$i > 0" {puts [incr i -1]}

Ce problème affecte à la fois for et while. Bien qu’il semble que cette boucle compte à rebours jusqu’à 0 et se termine, l’argument de condition de while est en fait toujours 10> 0 car c’est ce que l’argument a été évalué lorsque la commande while a été activée. Lorsque l’argument est entre accolades, il est passé à la commande while sous la forme $i>0, et la variable sera remplacée une fois pour chaque itération. Utilisez ceci à la place :

while {$i > 0} {puts [incr i -1]}

Évaluation totale

set a 1
if "$a == 0 && [incr a]" {puts abc}

Quelle est la valeur de ‘a’ après avoir exécuté ce code ? Étant donné que l’opérateur && n’évalue l’opérande droit que si l’opérande gauche est vrai, la valeur doit toujours être 1. Mais en réalité, c’est 2. C’est parce que l’évaluateur d’arguments a déjà effectué toutes les substitutions de variables et de commandes au moment où l’expression chaîne est évaluée. Utilisez ceci à la place :

if {$a == 0 && [incr a]} {puts abc}

Plusieurs opérateurs (les connecteurs logiques || et &&, et l’opérateur conditionnel ?:) sont définis pour ne pas évaluer tous leurs opérandes, mais ils ne peuvent fonctionner comme prévu que si la chaîne d’expression est entre accolades.

Multiplier une variable par 17 #

set myVariable [expr { $myVariable * 17 }]

Cela montre comment vous pouvez utiliser une expression simple pour mettre à jour une variable. La commande expr ne met pas à jour la variable pour vous ; vous devez prendre son résultat et l’écrire dans la variable avec set.

Notez que les retours à la ligne ne sont pas importants dans le petit langage compris par expr, et leur ajout peut rendre les expressions plus longues beaucoup plus faciles à lire.

set myVariable [expr {
    $myVariable * 17
}]

Cela fait exactement la même chose.

Appel d’une commande Tcl à partir d’une expression #

Parfois, vous devez appeler une commande Tcl à partir de votre expression. Par exemple, supposons que vous ayez besoin de la longueur d’une chaîne. Pour ce faire, il vous suffit d’utiliser une séquence [...] dans l’expression :

set halfTheStringLength [expr { [string length $theString] / 2 }]

Vous pouvez appeler n’importe quelle commande Tcl de cette manière, mais si vous vous retrouvez à appeler expr lui-même, arrêtez-vous ! et demandez-vous si vous avez vraiment besoin de cet appel supplémentaire. Vous pouvez généralement faire très bien en mettant l’expression interne entre parenthèses.

Erreur de mot nu invalide #

En Tcl lui-même, une chaîne composée d’un seul mot n’a pas besoin d’être entre guillemets. Dans le langage des chaînes d’expression évaluées par expr, tous les opérandes doivent avoir un type identifiable.

Les opérandes numériques sont écrits sans aucune décoration :

expr {455682 / 1.96e4}

Ainsi sont les constantes booléennes :

expr {true && !false}

La syntaxe de substitution de variable Tcl est reconnue : l’opérande sera défini sur la valeur de la variable :

expr {2 * $alpha}

Il en va de même pour la substitution de commande :

expr {[llength $alpha] > 0}

Les opérandes peuvent également être des appels de fonctions mathématiques, avec une liste d’opérandes séparés par des virgules entre parenthèses :

expr {sin($alpha)}

Un opérande peut être une chaîne entre guillemets ou entre accolades. Une chaîne entre guillemets doubles sera sujette à substitution comme dans une ligne de commande.

expr {"abc" < {def}}

Si un opérande n’est pas l’un des ci-dessus, il est illégal. Puisqu’il n’y a pas d’indication indiquant de quel type de mot il s’agit, expr signale une erreur de mot nu.
