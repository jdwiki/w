Limites actuelles (au 2016-10-03)

  • Les balises d’environnement ne peuvent pas être modifiées une fois que l’environnement est créé, alors choisissez judicieusement.
  • La mise à l’échelle automatique dans Elastic Beanstalk est limitée à Simple et Scheduled, donc si vous souhaitez utiliser Step-Scaling, reconsidérez si Elastic Beanstalk est un bon choix.

Automatisation avec Jenkins

Il existe un excellent [AWSEB Deployment Plugin][1] pour Jenkins qui sera plug-n-play pour le déploiement sur Elastic Beanstalk (les déploiements bleu/vert avec arrêt automatique de l’inactivité ne sont qu’à une case à cocher).

Présentation d’Elastic Beanstalk #

Elastic Beanstalk (EB) est essentiellement un hybride entre Golden AMI et [CloudFormation][1], tout en simplifiant considérablement la courbe d’apprentissage de [Puppet][2] ou [Chef][3].

Un déploiement Elastic Beanstalk est divisé en deux composants : Application et Environnement.

Application

Considérez ceci comme votre regroupement de niveau supérieur, votre application elle-même. Par exemple, une seule application (“MyWebApp”) peut avoir plusieurs Environnements (“Production” et “Staging”).

Environnement

Chaque environnement consistera en un déploiement d’architecture complet ([Instances EC2][4], [Elastic Load Balancer][5], [Autoscaling Group][6] et [Cloudwatch Alarms][7]). L’ensemble de la configuration de l’environnement est configuré et maintenu automatiquement pour vous.

Déploiement d’une application

Le déploiement de votre application est aussi simple que de télécharger un fichier zip contenant votre code. Chaque fichier zip (appelé Version d’application) que vous téléchargez est associé à une Application, vous pouvez donc le télécharger une fois et le déployer dans plusieurs Environnements.

Personnalisation de l’environnement

Par défaut, Elastic Beanstalk déploiera des AMI “stock” maintenues par Amazon. Pour la plupart des applications, cela suffit, mais il peut y avoir des modifications environnementales que vous souhaitez effectuer (par exemple, changer le fuseau horaire, ajouter des packages/dépendances non présentes dans le code, etc.).

Il existe deux manières de personnaliser l’AMI EB utilisée : [ebextensions][8] ou une AMI personnalisée.

ebextensions - Un dossier, littéralement appelé « .ebextensions », qui peut éventuellement être placé à la racine de votre version d’application (le zip que vous avez téléchargé contenant votre code). Dans le dossier ebextensions, vous pouvez placer des fichiers YAML définissant les scripts personnalisés, les dépendances, etc. que vous souhaitez exécuter côté serveur pendant le processus de déploiement. Il existe un certain nombre de points d’accrochage disponibles, pour les dernières informations, veuillez consulter la documentation pertinente : http://docs.aws.amazon.com/elasticbeanstalk/latest/dg/ebextensions.html

** Pièges / Choses à savoir **

Case à cocher VPC - Lors de la création d’un environnement, l’option est discrètement rendue disponible pour savoir si l’environnement doit ou non être créé/placé dans un VPC. Si vous avez besoin que votre application communique avec des ressources existantes que vous avez créées, COCHEZ CETTE CASE. Sinon, Elastic Beanstalk créera un nouveau groupe de sécurité isolé du reste de votre VPC. Bien que vous puissiez ajuster manuellement les paramètres du groupe de sécurité après sa création, essayer de l’« ajouter » essentiellement à un VPC entraînera une variété de problèmes plus tard.

RDS - Lors de la création d’un environnement, vous avez la possibilité de créer une instance RDS dans le cadre de l’environnement. Il n’est pas recommandé de l’utiliser, car chaque fois que vous devez “reconstruire” l’environnement (par exemple, déploiements bleu/vert, dépannage), il détruira et recréera l’instance RDS (avec toutes les données).

Déploiements bleu/vert sur Elastic Beanstalk #

Le déploiement Blue/Green est une technique de publication qui réduit les temps d’arrêt et les risques en exécutant deux environnements de production identiques (l’un appelé “Blue”, l’autre appelé “Green”). À tout moment, un seul des environnements dessert le trafic en direct, tandis que l’autre est inactif.

Lors du déploiement d’une nouvelle version, le code est déployé dans l’environnement inactif (par exemple, Green) et après confirmation d’un déploiement réussi, le trafic en direct est commuté (par exemple, Green a un nouveau code, le trafic de Blue est maintenant acheminé vers Green). Le prochain déploiement de code aura lieu sur le nouvel environnement inactif (suivant l’exemple, ce serait maintenant Blue).

Exemple/Aide visuelle :

[![exemple de déploiement bleu/vert][1]][1]

source de l’image : https://cloudnative.io/statics/blog/aws-blue-green-deployment.png

Lorsque vous utilisez Elastic Beanstalk (EB), vous pouvez facilement créer plusieurs environnements qui sont des clones exacts les uns des autres pour le déploiement de code. Après avoir confirmé que le nouvel environnement est prêt à être mis en ligne, il suffit d’utiliser la fonction “Swap URLs” dans EB pour échanger des environnements.

Instructions détaillées : http://docs.aws.amazon.com/elasticbeanstalk/latest/dg/using-features.CNAMESwap.html
