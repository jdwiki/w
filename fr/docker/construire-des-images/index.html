Construire une image à partir d’un Dockerfile #

Une fois que vous avez un Dockerfile, vous pouvez créer une image à partir de celui-ci en utilisant docker build. La forme de base de cette commande est :

docker build -t nom-image chemin

Si votre Dockerfile ne s’appelle pas Dockerfile, vous pouvez utiliser le drapeau -f pour donner le nom du Dockerfile à construire.

docker build -t image-name -f Dockerfile2 .

Par exemple, pour créer une image nommée dockerbuild-example:1.0.0 à partir d’un Dockerfile dans le répertoire de travail actuel :

$ ls
Dockerfile Dockerfile2

$ docker build -t dockerbuild-example:1.0.0 .

$ docker build -t dockerbuild-example-2:1.0.0 -f Dockerfile2 .

Voir la [documentation d’utilisation de docker build] pour plus d’options et de paramètres.

[documentation d’utilisation de docker build] : https://docs.docker.com/engine/reference/builder/#/usage

Une erreur courante consiste à créer un Dockerfile dans le répertoire personnel de l’utilisateur (~). C’est une mauvaise idée car pendant docker build -t mytag . ce message apparaîtra longtemps :

Contexte de chargement

La cause est le démon docker essayant de copier tous les fichiers de l’utilisateur (à la fois le répertoire personnel et ses sous-répertoires). Évitez cela en spécifiant toujours un répertoire pour le Dockerfile.

Ajouter un fichier .dockerignore au répertoire de construction [est une bonne pratique][1]. Sa syntaxe est similaire aux fichiers .gitignore et s’assurera que seuls les fichiers et répertoires souhaités sont téléchargés comme contexte de la construction.

[1] : https://docs.docker.com/engine/userguide/eng-image/dockerfile_best-practices/#/use-a-dockerignore-file

Différence entre ENTRYPOINT et CMD #

Il existe deux directives Dockerfile pour spécifier la commande à exécuter par défaut dans les images construites. Si vous ne spécifiez que CMD, docker exécutera cette commande en utilisant le ENTRYPOINT par défaut, qui est /bin/sh -c. Vous pouvez remplacer le point d’entrée et/ou la commande, ou les deux, lorsque vous démarrez l’image construite. Si vous spécifiez les deux, alors ENTRYPOINT spécifie l’exécutable de votre processus de conteneur, et CMD sera fourni comme paramètres de cet exécutable.

Par exemple, si votre Dockerfile contient

FROM ubuntu:16.04
CMD ["/bin/date"]

Ensuite, vous utilisez la directive par défaut ENTRYPOINT de /bin/sh -c et exécutez /bin/date avec ce point d’entrée par défaut. La commande de votre processus de conteneur sera /bin/sh -c /bin/date. Une fois que vous exécutez cette image, elle imprimera par défaut la date actuelle

$ docker build -t test .
$ docker run test
Tue Jul 19 10:37:43 UTC 2016

Vous pouvez remplacer CMD sur la ligne de commande, auquel cas il exécutera la commande que vous avez spécifiée.

$ docker run test /bin/hostname
bf0274ec8820

Si vous spécifiez une directive ENTRYPOINT, Docker utilisera cet exécutable et la directive CMD spécifie le ou les paramètres par défaut de la commande. Donc, si votre Dockerfile contient :

FROM ubuntu:16.04
ENTRYPOINT ["/bin/echo"]
CMD ["Hello"]

Ensuite, l’exécuter produira

$ docker build -t test .
$ docker run test
Hello

Vous pouvez fournir différents paramètres si vous le souhaitez, mais ils exécuteront tous /bin/echo

$ docker run test Hi
Hi

Si vous souhaitez remplacer le point d’entrée répertorié dans votre Dockerfile (c’est-à-dire si vous souhaitez exécuter une commande différente de echo dans ce conteneur), vous devez spécifier le paramètre --entrypoint sur la ligne de commande :

$ docker run --entrypoint=/bin/hostname test
b2c70e74df18

Généralement, vous utilisez la directive ENTRYPOINT pour pointer vers l’application principale que vous souhaitez exécuter et CMD vers les paramètres par défaut.

Un simple Dockerfile #

FROM node:5

La directive FROM spécifie une image à partir de laquelle commencer. Toute [référence d’image] valide peut être utilisée.

WORKDIR /usr/src/app

La directive WORKDIR définit le répertoire de travail actuel à l’intérieur du conteneur, ce qui équivaut à exécuter cd à l’intérieur du conteneur. (Remarque : RUN cd ne changera pas le répertoire de travail actuel.)

RUN npm install cowsay knock-knock-jokes

RUN exécute la commande donnée à l’intérieur du conteneur.

COPY cowsay-knockknock.js ./

COPY copie le fichier ou le répertoire spécifié dans le premier argument à partir du contexte de construction (le chemin transmis à docker build chemin< /code>) à l’emplacement du conteneur spécifié par le deuxième argument.

CMD node cowsay-knockknock.js

CMD spécifie une commande à exécuter lorsque l’image est [run] et qu’aucune commande n’est donnée. Il peut être remplacé en [passant une commande à docker run].

Il existe de nombreuses autres instructions et options ; voir la [Dockerfile reference] pour une liste complète.

[référence d’image] : https://www.wikiod.com/fr/docker/gestion-des-images#Images de référence [exécuter] : https://www.wikiod.com/fr/docker/conteneurs-en-cours-dexecution#Exécution d’un conteneur [en passant une commande à docker run] : https://www.wikiod.com/fr/docker/conteneurs-en-cours-dexecution#Exécuter une commande différente dans le conteneur [Référence Dockerfile] : https://docs.docker.com/engine/reference/builder/

Pousser et tirer une image vers Docker Hub ou un autre registre #

Les images créées localement peuvent être transmises à [Docker Hub] [1] ou à tout autre hôte de dépôt docker, appelé registre. Utilisez docker login pour vous connecter à un compte docker hub existant.

docker login

Login with your Docker ID to push and pull images from Docker Hub.
If you don't have a Docker ID, head over to https://hub.docker.com to create one.

Username: cjsimon
Password:
Login Succeeded

Un autre registre docker peut être utilisé en spécifiant un nom de serveur. Cela fonctionne également pour les registres privés ou auto-hébergés. De plus, l’utilisation d’un [magasin d’informations d’identification externe][2] pour la sécurité est possible.

docker login quay.io

Vous pouvez ensuite marquer et envoyer des images au registre auquel vous êtes connecté. Votre référentiel doit être spécifié sous la forme server/username/reponame:tag. Omettre le serveur est actuellement par défaut Docker Hub. (Le registre par défaut ne peut pas être changé pour un autre fournisseur, et il n’y a [aucun plan][3] pour implémenter cette fonctionnalité.)

docker tag mynginx quay.io/cjsimon/mynginx:latest

Différentes balises peuvent être utilisées pour représenter différentes versions, ou branches, de la même image. Une image avec plusieurs balises différentes affichera chaque balise dans le même dépôt.

Utilisez docker images pour voir une liste des images installées sur votre ordinateur local, y compris votre image nouvellement étiquetée. Ensuite, utilisez push pour le télécharger dans le registre et tirez pour télécharger l’image.

docker push quay.io/cjsimon/mynginx:latest

Toutes les balises d’une image peuvent être extraites en spécifiant l’option -a

docker pull quay.io/cjsimon/mynginx:latest

[1] : http://hub.docker.com [2] : https://docs.docker.com/engine/reference/commandline/login/#/credentials-store [3] : https://github.com/docker/docker/issues/7203

Exposer un port dans le Dockerfile #

EXPOSE <port> [<port>...]

D’après la documentation de Docker :

L’instruction EXPOSE informe Docker que le conteneur écoute sur les ports réseau spécifiés lors de l’exécution. EXPOSE ne rend pas les ports du conteneur accessibles à l’hôte. Pour ce faire, vous devez utiliser soit le drapeau -p pour publier une plage de ports, soit le drapeau -P pour publier tous les ports exposés. Vous pouvez exposer un numéro de port et le publier en externe sous un autre numéro.

Exemple: #

Dans votre Dockerfile :

EXPOSE 8765

Pour accéder à ce port depuis la machine hôte, incluez cet argument dans votre commande docker run :

-p 8765:8765

ENTRYPOINT et CMD vus comme verbe et paramètre #

Supposons que vous ayez un Dockerfile se terminant par

ENTRYPOINT [ "nethogs"] CMD ["wlan0"]

si vous construisez cette image avec un

docker construit -t inspecteur .

lancer l’image construite avec un tel Dockerfile avec une commande telle que

docker run -it --net=host --rm inspecteur

,nethogs surveillera l’interface nommée wlan0

Maintenant, si vous voulez surveiller l’interface eth0 (ou wlan1, ou ra1…), vous ferez quelque chose comme

docker run -it --net=host --rm inspecteur eth0

ou

docker run -it --net=host --rm inspecteur wlan1

Construire à l’aide d’un proxy #

Souvent, lors de la construction d’une image Docker, le Dockerfile contient des instructions qui exécutent des programmes pour récupérer des ressources sur Internet (wget par exemple pour extraire une construction binaire de programme sur GitHub par exemple).

Il est possible de demander à Docker de transmettre des variables d’environnement set set afin que ces programmes effectuent ces récupérations via un proxy :

$ docker build --build-arg http_proxy=http://myproxy.example.com:3128 \
               --build-arg https_proxy=http://myproxy.example.com:3128 \
               --build-arg no_proxy=internal.example.com \
               -t test .

build-arg sont des variables d’environnement qui ne sont disponibles qu’au moment de la construction.
