Syntaxe #

  • statistiques docker [OPTIONS] [CONTENEUR…]
  • journaux docker [OPTIONS] CONTENEUR
  • dessus docker [OPTIONS] CONTENEUR [ps OPTIONS]

Impression des journaux #

Suivre les journaux est le moyen le moins intrusif de déboguer une application en cours d’exécution. Cet exemple reproduit le comportement du traditionnel tail -f some-application.log sur le conteneur 7786807d8084.

docker logs --follow --tail 10 7786807d8084

Cette commande affiche essentiellement la sortie standard du processus de conteneur (le processus avec le pid 1).

Si vos journaux n’incluent pas nativement l’horodatage, vous pouvez ajouter le drapeau --timestamps.

Il est possible de consulter les logs d’un conteneur arrêté, soit

  • démarrer le conteneur défaillant avec lancer docker... ; journaux docker $(docker ps -lq)

  • trouver l’identifiant ou le nom du conteneur avec

docker ps -a

et alors

id de conteneur de journaux docker ou

nom du conteneur de journaux docker

car il est possible de consulter les logs d’un conteneur arrêté

Surveillance de l’utilisation des ressources #

L’inspection de l’utilisation des ressources système est un moyen efficace de détecter les applications qui se comportent mal. Cet exemple est un équivalent de la commande traditionnelle top pour les conteneurs :

docker stats

Pour suivre les statistiques de conteneurs spécifiques, répertoriez-les sur la ligne de commande :

docker stats 7786807d8084 7786807d8085

Les statistiques Docker affichent les informations suivantes :

CONTAINER     CPU %   MEM USAGE / LIMIT   MEM %    NET I/O               BLOCK I/O
7786807d8084  0.65%   1.33 GB / 3.95 GB   33.67%   142.2 MB / 57.79 MB   46.32 MB / 0 B

Par défaut, docker stats affiche l’identifiant des conteneurs, et ce n’est pas très utile, si vous préférez afficher les noms du conteneur, faites simplement

statistiques du docker $(docker ps --format '{{.Names}}')

Attacher à un conteneur en cours d’exécution #

« Attacher à un conteneur » est l’acte de démarrer une session de terminal dans le contexte où le conteneur (et tous les programmes qu’il contient) est en cours d’exécution. Ceci est principalement utilisé à des fins de débogage, mais peut également être nécessaire si des données spécifiques doivent être transmises à des programmes exécutés dans le conteneur.

La commande attach est utilisée pour ce faire. Il a cette syntaxe :

docker attach <container>

<container> peut être soit l’ID du conteneur, soit le nom du conteneur. Par exemple:

docker attach c8a9cf1a1fa8

Ou:

docker attach graceful_hopper

Vous devrez peut-être “sudo” les commandes ci-dessus, en fonction de votre utilisateur et de la configuration de docker.

Remarque : Attacher ne permet d’attacher qu’une seule session shell à un conteneur à la fois.

Avertissement : toutes les saisies au clavier seront transmises au conteneur. Appuyer sur Ctrl-c tuera votre conteneur.

Pour se détacher d’un conteneur attaché, appuyez successivement sur Ctrl-p puis Ctrl-q

Pour attacher plusieurs sessions shell à un conteneur, ou simplement comme alternative, vous pouvez utiliser exec. Utilisation de l’identifiant du conteneur :

docker exec -i -t c8a9cf1a1fa8 /bin/bash

En utilisant le nom du conteneur :

docker exec -i -t graceful_hopper /bin/bash

exec exécutera un programme dans un conteneur, dans ce cas /bin/bash (un shell, probablement celui du conteneur). -i indique une session interactive, tandis que -t alloue un pseudo-TTY.

Remarque : Contrairement à attach, appuyer sur Ctrl-c ne mettra fin à la commande exec’d qu’en cas d’exécution interactive.

Entrée dans un conteneur en cours d’exécution #

Pour exécuter des opérations dans un conteneur, utilisez la commande docker exec. Parfois, cela s’appelle “entrer dans le conteneur” car toutes les commandes sont exécutées à l’intérieur du conteneur.

docker exec -it container_id bash

ou

docker exec -it container_id /bin/sh

Et maintenant, vous avez un shell dans votre conteneur en cours d’exécution. Par exemple, répertoriez les fichiers dans un répertoire, puis quittez le conteneur :

docker exec container_id ls -la

Vous pouvez utiliser le -u flag pour entrer dans le conteneur avec un utilisateur spécifique, par ex. uid=1013, gid=1023.

docker exec -it -u 1013:1023 container_id ls -la

L’uid et le gid ne doivent pas nécessairement exister dans le conteneur, mais la commande peut entraîner des erreurs. Si vous souhaitez lancer un conteneur et entrer immédiatement à l’intérieur afin de vérifier quelque chose, vous pouvez le faire

lancer docker... ; docker exec -it $(docker ps -lq) bash

la commande docker ps -lq affiche uniquement l’identifiant du dernier conteneur (le l dans -lq) démarré. (cela suppose que vous avez bash comme interprète disponible dans votre conteneur, vous pouvez avoir sh ou zsh ou tout autre)

Surveillance des processus dans un conteneur #

L’inspection de l’utilisation des ressources système est un moyen efficace de réduire un problème sur une application en cours d’exécution. Cet exemple est un équivalent de la commande traditionnelle ps pour les conteneurs.

docker top 7786807d8084

Pour filtrer ou formater la sortie, ajoutez les options ps sur la ligne de commande :

docker top 7786807d8084 faux

Ou, pour obtenir la liste des processus exécutés en tant que root, ce qui est une pratique potentiellement dangereuse :

docker top 7786807d8084 -u root

La commande docker top s’avère particulièrement utile lors du dépannage de conteneurs minimalistes sans shell ou la commande ps.

Débogage du processus du conteneur Docker #

Docker n’est qu’un moyen sophistiqué d’exécuter un processus, pas une machine virtuelle. Par conséquent, le débogage d’un processus “dans un conteneur” est également possible “sur l’hôte” en examinant simplement le processus de conteneur en cours d’exécution en tant qu’utilisateur disposant des autorisations appropriées pour inspecter ces processus sur l’hôte (par exemple, root). Par exemple, il est possible de lister chaque “processus de conteneur” sur l’hôte en exécutant un simple “ps” en tant que root :

sudo ps aux

Tous les conteneurs Docker en cours d’exécution seront répertoriés dans la sortie.

Cela peut être utile lors du développement d’une application pour déboguer un processus s’exécutant dans un conteneur. En tant qu’utilisateur disposant des autorisations appropriées, des utilitaires de débogage typiques peuvent être utilisés sur le processus de conteneur, tels que strace, ltrace, gdb, etc.
