Syntaxe #

  • docker exécuter [OPTIONS] IMAGE [COMMANDE] [ARG…]

Supprimer automatiquement un conteneur après l’avoir exécuté #

Normalement, un conteneur Docker persiste après sa fermeture. Cela vous permet de réexécuter le conteneur, d’inspecter son système de fichiers, etc. Cependant, vous souhaitez parfois exécuter un conteneur et le supprimer immédiatement après sa fermeture. Par exemple pour exécuter une commande ou afficher un fichier du système de fichiers. Docker fournit l’option de ligne de commande --rm à cette fin :

docker run --rm ubuntu cat /etc/hosts

Cela créera un conteneur à partir de l’image “ubuntu”, affichera le contenu du fichier /etc/hosts, puis supprimera le conteneur immédiatement après sa sortie. Cela permet d’éviter d’avoir à nettoyer les conteneurs une fois l’expérimentation terminée.

Remarque : Le drapeau --rm ne fonctionne pas avec le drapeau -d (--detach) dans docker < 1.13.0.

Lorsque l’indicateur --rm est défini, Docker supprime également les volumes associés au conteneur lorsque le conteneur est supprimé. Ceci est similaire à l’exécution de docker rm -v my-container. Seuls les volumes spécifiés sans nom sont supprimés.

Par exemple, avec docker run -it --rm -v /etc -v logs:/var/log centos /bin/produce_some_logs, le volume de /etc sera supprimé, mais le volume de /var/log ne le sera pas. Les volumes hérités via –volumes-from seront supprimés avec la même logique – si le volume d’origine a été spécifié avec un nom, il ne sera pas supprimé.

Stratégie de redémarrage du conteneur (démarrage d’un conteneur au démarrage) #

docker run --restart=always -d <container>

Par défaut, Docker ne redémarre pas les conteneurs lorsque le démon Docker redémarre, par exemple après un redémarrage du système hôte. Docker fournit une politique de redémarrage pour vos conteneurs en fournissant l’option de ligne de commande --restart. Fournir --restart=always entraînera toujours le redémarrage d’un conteneur après le redémarrage du démon Docker. Cependant, lorsque ce conteneur est arrêté manuellement (par exemple, avec docker stop <container>), la politique de redémarrage ne sera pas appliquée au conteneur.

Plusieurs options peuvent être spécifiées pour l’option --restart, en fonction de l’exigence (--restart=[policy]). Ces options affectent également la façon dont le conteneur démarre au démarrage.

Politique Résultat
non La valeur par défaut. Ne redémarrera pas le conteneur automatiquement, lorsque le conteneur est arrêté.
on-failure[:max-retries] Redémarrer uniquement si le conteneur se termine avec un échec (statut de sortie non nul). Pour éviter de le redémarrer indéfiniment (en cas de problème), on peut limiter le nombre de tentatives de redémarrage du démon Docker.
toujours Redémarrez toujours le conteneur quel que soit l’état de sortie. Lorsque vous spécifiez always, le démon Docker essaiera de redémarrer le conteneur indéfiniment. Le conteneur démarrera également toujours au démarrage du démon, quel que soit l’état actuel du conteneur.
unless-stopped Redémarrez toujours le conteneur quel que soit son état de sortie, mais ne le démarrez pas au démarrage du démon si le conteneur a déjà été arrêté.

Spécifier un nom #

Par défaut, les conteneurs créés avec docker run reçoivent un nom aléatoire comme small_roentgen ou modest_dubinsky. Ces noms ne sont pas particulièrement utiles pour identifier l’objectif d’un conteneur. Il est possible de fournir un nom pour le conteneur en passant l’option de ligne de commande --name :

docker run --name my-ubuntu ubuntu:14.04

Les noms doivent être uniques ; si vous passez un nom qu’un autre conteneur utilise déjà, Docker affichera une erreur et aucun nouveau conteneur ne sera créé.

Spécifier un nom sera utile lors du référencement du conteneur au sein d’un réseau Docker. Cela fonctionne pour les conteneurs Docker d’arrière-plan et de premier plan.

Les conteneurs du réseau de pont par défaut doivent être liés pour communiquer par leur nom.

Exécuter un conteneur en arrière-plan #

Pour qu’un conteneur continue de s’exécuter en arrière-plan, fournissez l’option de ligne de commande -d lors du démarrage du conteneur :

docker run -d busybox top

L’option -d exécute le conteneur en mode détaché. C’est aussi équivalent à -d=true.

Un conteneur en mode détaché ne peut pas être supprimé automatiquement lorsqu’il s’arrête, cela signifie que l’on ne peut pas utiliser l’option –rm en combinaison avec l’option -d.

Attribuer un volume à un conteneur #

Un volume Docker est un fichier ou un répertoire qui persiste au-delà de la durée de vie du conteneur. Il est possible de monter un fichier ou un répertoire hôte dans un conteneur en tant que volume (en contournant l’UnionFS).

Ajoutez un volume avec l’option de ligne de commande -v :

docker run -d -v "/data" awesome/app bootstrap.sh

Cela créera un volume et le montera sur le chemin / data à l’intérieur du conteneur.

  • Remarque : vous pouvez utiliser l’indicateur --rm pour supprimer automatiquement le volume lorsque le conteneur est supprimé.

Montage des répertoires hôtes

Pour monter un fichier ou un répertoire hôte dans un conteneur :

docker run -d -v "/home/foo/data:/data" awesome/app bootstrap.sh
  • Lorsque vous spécifiez un répertoire hôte, un chemin absolu doit être fourni.

Cela montera le répertoire hôte /home/foo/data sur /data à l’intérieur du conteneur. Ce volume “répertoire hôte monté sur liaison” est la même chose qu’un mount --bind Linux et monte donc temporairement le répertoire hôte sur le chemin de conteneur spécifié pour la durée de vie du conteneur. Les modifications apportées au volume à partir de l’hôte ou du conteneur sont immédiatement répercutées sur l’autre, car il s’agit de la même destination sur le disque.

Exemple UNIX de montage d’un dossier relatif

docker run -d -v $(pwd)/data:/data awesome/app bootstrap.sh

Nom des volumes

Un volume peut être nommé en fournissant une chaîne au lieu d’un chemin de répertoire hôte, docker créera un volume en utilisant ce nom.

docker run -d -v "my-volume:/data" awesome/app bootstrap.sh

Après avoir créé un volume nommé, le volume peut ensuite être partagé avec d’autres conteneurs utilisant ce nom.

Mettre un shell dans un conteneur en cours d’exécution (détaché) #

Connectez-vous à un conteneur en cours d’exécution #

Un utilisateur peut entrer un conteneur en cours d’exécution dans un nouveau shell bash interactif avec la commande [exec][1].

Supposons qu’un conteneur s’appelle jovial_morse, vous pouvez obtenir un shell bash pseudo-TTY interactif en exécutant :

docker exec -it jovial_morse bash

Connectez-vous à un conteneur en cours d’exécution avec un utilisateur spécifique #

Si vous souhaitez entrer un conteneur en tant qu’utilisateur spécifique, vous pouvez le définir avec le paramètre -u ou --user. Le nom d’utilisateur doit exister dans le conteneur.

-u, --user Nom d’utilisateur ou UID (format : <nom|uid>[:<groupe|gid>])

Cette commande se connectera à jovial_morse avec l’utilisateur dockeruser

docker exec -it -u dockeruser jovial_morse bash

Connectez-vous à un conteneur en cours d’exécution en tant que root #

Si vous souhaitez vous connecter en tant que root, utilisez simplement le paramètre -u root. L’utilisateur racine existe toujours.

docker exec -it -u root jovial_morse bash

Connectez-vous à une image #

Vous pouvez également vous connecter à une image avec la commande [run][2], mais cela nécessite un nom d’image au lieu d’un nom de conteneur.

docker run -it dockerimage bash

Exécuter un conteneur de manière interactive #

Pour exécuter un conteneur de manière interactive, passez les options -it :

$ docker run -it ubuntu:14.04 bash
root@8ef2356d919a:/# echo hi
hi
root@8ef2356d919a:/#

-i garde STDIN ouvert, tandis que -t alloue un pseudo-TTY.

Se détacher d’un conteneur #

Lorsque vous êtes attaché à un conteneur en cours d’exécution avec un pty assigné (docker run -it ...), vous pouvez appuyer sur ControlP - Control Q pour détacher.

Lier un port de conteneur à l’hôte #

docker run -p "8080:8080" myApp
docker run -p "192.168.1.12:80:80" nginx
docker run -P myApp

Afin d’utiliser les ports sur l’hôte qui ont été exposés dans une image (via la directive EXPOSE Dockerfile ou l’option de ligne de commande --expose pour docker run), ces ports doivent être liés à l’hôte en utilisant le Options de ligne de commande -p ou -P. L’utilisation de -p nécessite que le port particulier (et l’interface hôte facultative) soit spécifié. L’utilisation de l’option de ligne de commande -P majuscule forcera Docker à lier tous les ports exposés dans l’image d’un conteneur à l’hôte.

Définition des variables d’environnement #

$ docker run -e "ENV_VAR=foo" ubuntu /bin/bash

-e et --env peuvent être utilisés pour définir des variables d’environnement à l’intérieur d’un conteneur. Il est possible de fournir de nombreuses variables d’environnement à l’aide d’un fichier texte :

$ docker run --env-file ./env.list ubuntu /bin/bash

Exemple de fichier de variables d’environnement :

# This is a comment
TEST_HOST=10.10.0.127

Le drapeau --env-file prend un nom de fichier comme argument et s’attend à ce que chaque ligne soit au format VARIABLE=VALUE, imitant l’argument passé à --env. Les lignes de commentaires ne doivent être précédées que de #.

Indépendamment de l’ordre de ces trois drapeaux, le --env-file est traité en premier, puis les drapeaux -e/--env. De cette façon, toutes les variables d’environnement fournies individuellement avec -e ou --env remplaceront les variables fournies dans le fichier texte --env-var.

Passer stdin au conteneur #

Dans des cas tels que la restauration d’un vidage de base de données ou si vous souhaitez envoyer des informations via un canal depuis l’hôte, vous pouvez utiliser le drapeau -i comme argument pour docker run ou docker exec.

Par exemple, en supposant que vous souhaitiez mettre sur un client mariadb conteneurisé un vidage de base de données que vous avez sur l’hôte, dans un fichier dump.sql local, vous pouvez exécuter la commande suivante :

docker exec -i mariadb bash -c 'mariadb "-p$MARIADB_PASSWORD" ' < dump.sql

En général,

docker exec -i container command < file.stdin

Ou

docker exec -i container command <<EOF
inline-document-from-host-shell-HEREDOC-syntax
EOF

Remplacement de la directive de point d’entrée d’image #

docker run --name="test-app" --entrypoint="/bin/bash" example-app

Cette commande remplacera la directive ENTRYPOINT de l’image example-app lors de la création du conteneur test-app. La directive CMD de l’image restera inchangée sauf indication contraire :

docker run --name="test-app" --entrypoint="/bin/bash" example-app /app/test.sh

Dans l’exemple ci-dessus, le ENTRYPOINT et le CMD de l’image ont été remplacés. Ce processus conteneur devient /bin/bash /app/test.sh.

Spécifier un nom d’hôte #

Par défaut, les conteneurs créés avec docker run reçoivent un nom d’hôte aléatoire. Vous pouvez donner au conteneur un nom d’hôte différent en passant l’indicateur –hostname :

docker run --hostname redbox -d ubuntu:14.04

Exécution d’un conteneur #

docker run hello-world

Cela va récupérer la dernière image [hello-world] du Docker Hub (si vous ne l’avez pas déjà), créer un nouveau conteneur et l’exécuter. Vous devriez voir un message indiquant que votre installation semble fonctionner correctement.

[hello-world] : https://hub.docker.com/r/_/hello-world/

Exécution d’une commande différente dans le conteneur #

docker run docker/whalesay cowsay 'Hello, StackExchange!'

Cette commande indique à Docker de créer un conteneur à partir de l’image docker/whalesay et d’y exécuter la commande cowsay 'Hello, StackExchange!'. Il devrait imprimer une image d’une baleine disant “Hello, StackExchange!” sur votre terminal.

Si le point d’entrée dans l’image est la valeur par défaut, vous pouvez exécuter n’importe quelle commande disponible dans l’image :

docker run docker/whalesay ls /

S’il a été modifié lors de la création de l’image, vous devez le rétablir par défaut

docker run --entrypoint=/bin/bash docker/whalesay -c ls /

Empêcher le conteneur de s’arrêter lorsqu’aucune commande n’est en cours d’exécution #

Un conteneur s’arrêtera si aucune commande n’est en cours d’exécution au premier plan. L’utilisation de l’option -t empêchera le conteneur de s’arrêter, même s’il est détaché avec l’option -d.

docker run -t -d debian bash

Arrêt d’un conteneur #

docker stop mynginx

De plus, l’ID du conteneur peut également être utilisé pour arrêter le conteneur au lieu de son nom.

Cela arrêtera un conteneur en cours d’exécution en envoyant le signal SIGTERM puis le signal SIGKILL si nécessaire.

De plus, la commande kill peut être utilisée pour envoyer immédiatement un SIGKILL ou tout autre signal spécifié en utilisant l’option -s.

docker kill mynginx

Signal spécifié :

docker kill -s SIGINT mynginx

L’arrêt d’un conteneur ne le supprime pas. Utilisez docker ps -a pour voir votre conteneur arrêté.

Conteneur en cours d’exécution avec des limites de mémoire/d’échange #

Définir la limite de mémoire et désactiver la limite d’échange

docker run -it -m 300M --memory-swap -1 ubuntu:14.04 /bin/bash

Définissez à la fois la mémoire et la limite d’échange. Dans ce cas, le conteneur peut utiliser 300 Mo de mémoire et 700 Mo d’échange.

docker run -it -m 300M --memory-swap 1G ubuntu:14.04 /bin/bash

## Ajouter une entrée d’hôte au conteneur docker run –add-host=“app-backend:10.15.1.24” awesome-app

Cette commande ajoute une entrée au fichier /etc/hosts du conteneur, qui suit le format --add-host <name>:<address>. Dans cet exemple, le nom app-backend sera résolu en 10.15.1.24. Ceci est particulièrement utile pour lier par programmation des composants d’application disparates.

Exécuter une autre commande sur un conteneur en cours d’exécution #

Si nécessaire, vous pouvez demander à Docker d’exécuter des commandes supplémentaires sur un conteneur déjà en cours d’exécution à l’aide de la commande “exec”. Vous avez besoin de l’ID du conteneur que vous pouvez obtenir avec docker ps.

docker exec 294fbc4c24b3 echo "Hello World"

Vous pouvez attacher un shell interactif si vous utilisez l’option -it.

docker exec -it 294fbc4c24b3 bash

Exécution d’applications GUI dans un conteneur Linux #

Par défaut, un conteneur Docker ne pourra pas * exécuter * une application graphique.

Avant cela, la prise X11 doit d’abord être transmise au conteneur, afin qu’elle puisse être utilisée directement. La variable d’environnement DISPLAY doit également être transmise :

docker run -v /tmp/.X11-unix:/tmp/.X11-unix -e DISPLAY=unix$DISPLAY <image-name>

Cela échouera dans un premier temps, car nous n’avons pas défini les autorisations sur l’hôte du serveur X :

cannot connect to X server unix:0

Le moyen le plus rapide (mais pas le plus sûr) est d’autoriser l’accès directement avec :

xhost +local:root

Après avoir terminé avec le conteneur, nous pouvons revenir à l’état d’origine avec :

xhost -local:root

Un autre moyen (plus sûr) consiste à préparer un Dockerfile qui construira une nouvelle image qui utilisera nos informations d’identification utilisateur pour accéder au serveur X :

FROM <iamge-name>
MAINTAINER <you>

# Arguments picked from the command line!
ARG user
ARG uid
ARG gid

#Add new user with our credentials
ENV USERNAME ${user}
RUN useradd -m $USERNAME && \
        echo "$USERNAME:$USERNAME" | chpasswd && \
        usermod --shell /bin/bash $USERNAME && \
        usermod  --uid ${uid} $USERNAME && \
        groupmod --gid ${gid} $USERNAME

USER ${user}

WORKDIR /home/${user}

Lors de l’appel de docker build à partir de la ligne de commande, nous devons transmettre les variables ARG qui apparaissent dans le Dockerfile :

docker build --build-arg user=$USER --build-arg uid=$(id -u) --build-arg gid=$(id -g) -t <new-image-with-X11-enabled-name> -f <Dockerfile-for-X11> .

Maintenant, avant de générer un nouveau conteneur, nous devons créer un fichier xauth avec une autorisation d’accès :

xauth nlist $DISPLAY | sed -e 's/^..../ffff/' | xauth -f /tmp/.docker.xauth nmerge -

Ce fichier doit être monté dans le conteneur lors de sa création/exécution :

docker run -e DISPLAY=unix$DISPLAY -v /tmp/.X11-unix:/tmp/.X11-unix -v /tmp/.docker.xauth:/tmp/.docker.xauth:rw -e XAUTHORITY=/tmp/.docker.xauth
