Syntaxe #

  • docker rm [OPTIONS] CONTENEUR [CONTENEUR…]
  • docker attach [OPTIONS] CONTENEUR
  • docker exec [OPTIONS] COMMANDE DE CONTENEUR [ARG…]
  • docker ps [OPTIONS]
  • journaux docker [OPTIONS] CONTENEUR
  • docker inspect [OPTIONS] CONTENEUR|IMAGE [CONTENANT|IMAGE…]
  • Dans les exemples ci-dessus, chaque fois que le conteneur est un paramètre de la commande docker, il est mentionné comme <container> ou container id ou <CONTAINER_NAME>. Dans tous ces endroits, vous pouvez passer un nom de conteneur ou un identifiant de conteneur pour spécifier un conteneur.

Répertorier les conteneurs #

$ docker ps
CONTAINER ID        IMAGE               COMMAND                  CREATED             STATUS              PORTS                     NAMES
2bc9b1988080        redis               "docker-entrypoint.sh"   2 weeks ago         Up 2 hours          0.0.0.0:6379->6379/tcp    elephant-redis
817879be2230        postgres            "/docker-entrypoint.s"   2 weeks ago         Up 2 hours          0.0.0.0:65432->5432/tcp   pt-postgres

docker ps seul n’imprime que les conteneurs en cours d’exécution. Pour afficher tous les conteneurs (y compris ceux arrêtés), utilisez le drapeau -a :

$ docker ps -a
CONTAINER ID        IMAGE               COMMAND                  CREATED             STATUS                    PORTS                     NAMES
9cc69f11a0f7        docker/whalesay     "ls /"                   26 hours ago        Exited (0) 26 hours ago                             berserk_wozniak
2bc9b1988080        redis               "docker-entrypoint.sh"   2 weeks ago         Up 2 hours                0.0.0.0:6379->6379/tcp    elephant-redis
817879be2230        postgres            "/docker-entrypoint.s"   2 weeks ago         Up 2 hours                0.0.0.0:65432->5432/tcp   pt-postgres

Pour répertorier les conteneurs avec un statut spécifique, utilisez l’option de ligne de commande -f pour filtrer les résultats. Voici un exemple de liste de tous les conteneurs qui sont sortis :

$ docker ps -a -f status=exited
CONTAINER ID        IMAGE               COMMAND                  CREATED             STATUS                    PORTS                     NAMES
9cc69f11a0f7        docker/whalesay     "ls /"                   26 hours ago        Exited (0) 26 hours ago

Il est également possible de ne lister que les ID de conteneur avec le commutateur -q. Cela rend très facile l’utilisation du résultat avec d’autres utilitaires Unix (tels que grep et awk):

$ docker ps -aq
9cc69f11a0f7
2bc9b1988080
817879be2230

Lors du lancement d’un conteneur avec docker run --name mycontainer1 vous donnez un nom spécifique et non un nom aléatoire (sous la forme mood_famous, comme nostalgic_stallman), et il peut être facile de les retrouver avec une telle commande

docker ps -f nom=monconteneur1

Supprimer, supprimer et nettoyer les conteneurs #

docker rm peut être utilisé pour supprimer un conteneur spécifique comme celui-ci :

docker rm <container name or id>

Pour supprimer tous les conteneurs, vous pouvez utiliser cette expression :

docker rm $(docker ps -qa)

Par défaut, docker ne supprimera pas un conteneur en cours d’exécution. Tout conteneur en cours d’exécution produira un message d’avertissement et ne sera pas supprimé. Tous les autres conteneurs seront supprimés.

Vous pouvez également utiliser xargs :

docker ps -aq -f status=exited | xargs -r docker rm

Où docker ps -aq -f status=exited renverra une liste d’ID de conteneurs dont le statut est “Exited”.

Avertissement : Tous les exemples ci-dessus supprimeront uniquement les conteneurs “arrêtés”.

Pour supprimer un conteneur, qu’il soit arrêté ou non, vous pouvez utiliser l’indicateur de force -f :

docker rm -f <container name or id>

Pour supprimer tous les conteneurs, quel que soit leur état :

docker rm -f $(docker ps -qa)

Si vous souhaitez supprimer uniquement les conteneurs avec un statut “mort” :

docker rm $(docker ps --all -q -f status=dead)

Si vous souhaitez supprimer uniquement les conteneurs dont l’état est “exited”, procédez comme suit :

docker rm $(docker ps --all -q -f status=exited)

Ce sont toutes des permutations de filtres utilisées lors de la [liste des conteneurs][1].

Pour supprimer à la fois les conteneurs indésirables et les images pendantes qui utilisent de l’espace après [version 1.3][2], utilisez ce qui suit (similaire à l’outil Unix df):

$ docker system df

Pour supprimer toutes les données inutilisées :

$ docker system prune

[1] : https://www.wikiod.com/fr/docker/gestion-des-conteneurs#Répertorier les conteneurs [2] : https://blog.docker.com/2017/01/whats-new-in-docker-1-13/

Référencement des conteneurs #

Les commandes Docker qui prennent le nom d’un conteneur acceptent trois formes différentes :

Taper Exemple
UUID complet 9cc69f11a0f76073e87f25cb6eaf0e079fbfbd1bc47c063bcd25ed3722a8cc4a
UUID court 9cc69f11a0f7
Nom berserk_wozniak

Utilisez docker ps pour afficher ces valeurs pour les conteneurs de votre système.

L’UUID est généré par Docker et ne peut pas être modifié. Vous pouvez fournir un nom au conteneur lorsque vous le démarrez docker run --name <given name> <image>. Docker générera un nom aléatoire pour le conteneur si vous n’en spécifiez pas au moment du démarrage du conteneur.

REMARQUE : La valeur de l’UUID (ou d’un UUID “court”) peut être de n’importe quelle longueur tant que la valeur donnée est unique pour un conteneur

Se connecter à une instance s’exécutant en tant que démon #

Il existe deux façons d’y parvenir, la première et la plus connue est la suivante :

docker attach --sig-proxy=false <container>

Celui-ci attache littéralement votre bash au bash du conteneur, ce qui signifie que si vous avez un script en cours d’exécution, vous verrez le résultat.

Pour détacher, tapez simplement : Ctl-P Ctl-Q

Mais si vous avez besoin d’une manière plus conviviale et de pouvoir créer de nouvelles instances bash, exécutez simplement la commande suivante :

docker exec -it <container> bash

Trouver un conteneur spécifique #

docker ps --filter name=myapp_1

Copier le fichier depuis/vers les conteneurs #

du conteneur à l’hôte

docker cp CONTAINER_NAME:PATH_IN_CONTAINER PATH_IN_HOST

de l’hôte au conteneur

docker cp PATH_IN_HOST CONTAINER_NAME:PATH_IN_CONTAINER

Si j’utilise jess/transmission de

https://hub.docker.com/r/jess/transmission/builds/bsn7eqxrkzrhxazcuytbmzp/

, les fichiers du conteneur sont dans /transmission/download

et mon répertoire actuel sur l’hôte est /home/$USER/abc, après

docker cp transmission_id_or_name:/transmission/download .

Je ferai copier les fichiers sur

/home/$USER/abc/transmission/download

vous ne pouvez pas, en utilisant docker cp copier un seul fichier, vous copiez l’arborescence des répertoires et les fichiers

Démarrage et arrêt des conteneurs #

Pour arrêter un conteneur en cours d’exécution :

docker stop <container> [<container>...]

Cela enverra au processus principal dans le conteneur un SIGTERM, suivi d’un SIGKILL s’il ne s’arrête pas pendant la période de grâce. Le nom de chaque conteneur est imprimé lorsqu’il s’arrête.

Pour démarrer un conteneur qui est arrêté :

docker start <container> [<container>...]

Cela démarrera chaque conteneur passé en arrière-plan ; le nom de chaque conteneur est imprimé au début. Pour démarrer le conteneur au premier plan, passez le drapeau -a (--attach).

Liste des conteneurs avec un format personnalisé #

docker ps --format 'table {{.ID}}\t{{.Names}}\t{{.Status}}'

Trouver l’IP du conteneur #

Pour connaître l’adresse IP de votre conteneur, utilisez :

docker inspect <container id> | grep IPAddress

ou utilisez docker inspect

docker inspect --format '{{ .NetworkSettings.IPAddress }}' ${CID}

Redémarrage du conteneur docker #

docker restart <container> [<container>...]

Option –time : Secondes à attendre l’arrêt avant de tuer le conteneur (par défaut 10)

docker restart <container> --time 10

Exécuter la commande sur un conteneur docker déjà existant #

docker exec -it <identifiant du conteneur> /bin/bash

Il est courant de se connecter à un conteneur déjà en cours d’exécution pour effectuer des tests rapides ou voir ce que fait l’application. Souvent, cela dénote de mauvaises pratiques d’utilisation des conteneurs en raison des journaux et les fichiers modifiés doivent être placés dans des volumes. Cet exemple nous permet de nous connecter au conteneur. Cela suppose que /bin/bash est disponible dans le conteneur, cela peut être /bin/sh ou autre chose.

docker exec <container id> tar -czvf /tmp/backup.tgz /data
docker cp <container id>:/tmp/backup.tgz .

Cet exemple archive le contenu du répertoire de données dans un fichier tar. Ensuite, avec docker cp, vous pouvez le récupérer.

Journaux de conteneur #

Usage:  docker logs [OPTIONS] CONTAINER

Fetch the logs of a container

  --help=false              Print usage

  -t, --timestamps=false    Show timestamps

Supprimer, supprimer et nettoyer les volumes docker #

Les volumes Docker ne sont pas automatiquement supprimés lorsqu’un conteneur est arrêté. Pour supprimer les volumes associés lorsque vous arrêtez un conteneur :

docker rm -v <container id or name>

Si l’indicateur -v n’est pas spécifié, le volume reste sur le disque en tant que ‘volume suspendu’. Pour supprimer tous les volumes en suspens :

docker volume rm $(docker volume ls -qf dangling=true)

Le filtre docker volume ls -qf dangling=true renverra une liste de noms de volumes docker, y compris ceux non balisés, qui ne sont pas attachés à un conteneur.

Alternativement, vous pouvez utiliser xargs :

docker volume ls -f dangling=true -q | xargs --no-run-if-empty docker volume rm

Exporter et importer des systèmes de fichiers de conteneur Docker #

Il est possible d’enregistrer le contenu du système de fichiers d’un conteneur Docker dans un fichier d’archive tarball. Ceci est utile à la rigueur pour déplacer des systèmes de fichiers de conteneur vers différents hôtes, par exemple si un conteneur de base de données a des modifications importantes et qu’il n’est pas possible de répliquer ces modifications ailleurs. Veuillez noter qu’il est préférable de créer un conteneur entièrement nouveau à partir d’une image mise à jour à l’aide d’une commande docker run ou d’un fichier docker-compose.yml, au lieu d’exporter et de déplacer le système de fichiers d’un conteneur. Une partie de la puissance de Docker réside dans la vérifiabilité et la responsabilité de son style déclaratif de création d’images et de conteneurs. En utilisant docker export et docker import, ce pouvoir est atténué en raison de l’obscurcissement des modifications apportées à l’intérieur du système de fichiers d’un conteneur à partir de son état d’origine.

docker export -o redis.tar redis

La commande ci-dessus créera une image vide, puis exportera le système de fichiers du conteneur “redis” dans cette image vide. Pour importer depuis une archive tarball, utilisez :

docker import ./redis.tar redis-imported:3.0.7

Cette commande créera l’image redis-imported:3.0.7, à partir de laquelle des conteneurs peuvent être créés. Il est également possible de créer des modifications lors de l’importation, ainsi que de définir un message de validation :

docker import -c="ENV DEBUG true" -m="enable debug mode" ./redis.tar redis-changed

Les directives Dockerfile disponibles pour une utilisation avec l’option de ligne de commande -c sont CMD, ENTRYPOINT, ENV, EXPOSE, ONBUILD, USER, VOLUME, WORKDIR.
