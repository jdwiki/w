Conteneurs de données uniquement #

Les conteneurs de données uniquement sont obsolètes et sont désormais considérés comme un anti-modèle !

Autrefois, avant la sous-commande “volume” de Docker et avant qu’il ne soit possible de créer des volumes nommés, Docker supprimait les volumes lorsqu’il n’y avait plus de références à eux dans aucun conteneur. Les conteneurs de données uniquement sont obsolètes car Docker offre désormais la possibilité de créer des volumes nommés, ainsi que de nombreux autres utilitaires via les différentes sous-commandes “docker volume”. Les conteneurs de données uniquement sont désormais considérés comme un anti-modèle pour cette raison.

De nombreuses ressources sur le Web au cours des deux dernières années mentionnent l’utilisation d’un modèle appelé “conteneur de données uniquement”, qui est simplement un conteneur Docker qui n’existe que pour conserver une référence à un volume de données.

N’oubliez pas que dans ce contexte, un “volume de données” est un volume Docker qui n’est pas monté depuis l’hôte. Pour clarifier, un “volume de données” est un volume qui est créé soit avec la directive VOLUME Dockerfile, soit en utilisant le commutateur -v sur la ligne de commande dans une commande docker run, spécifiquement avec le format -v /chemin/sur/conteneur. Par conséquent, un “conteneur de données uniquement” est un conteneur dont le seul but est d’avoir un volume de données attaché, qui est utilisé par le drapeau --volumes-from dans une commande docker run. Par exemple:

docker run -d --name "mysql-data" -v "/var/lib/mysql" alpine /bin/true

Lorsque la commande ci-dessus est exécutée, un “conteneur de données uniquement” est créé. Il s’agit simplement d’un conteneur vide auquel est attaché un volume de données. Il était alors possible d’utiliser ce volume dans un autre conteneur comme ceci :

docker run -d --name="mysql" --volumes-from="mysql-data" mysql

Le conteneur mysql contient maintenant le même volume qui se trouve également dans mysql-data.

Étant donné que Docker fournit désormais la sous-commande volume et des volumes nommés, ce modèle est désormais obsolète et non recommandé.

Pour commencer avec la sous-commande volume et les volumes nommés, voir https://www.wikiod.com/fr/docker/volumes-de-donnees-docker#Création d’un volume nommé

Création d’un volume de données #

docker run -d --name "mysql-1" -v "/var/lib/mysql" mysql

Cette commande crée un nouveau conteneur à partir de l’image mysql. Il crée également un nouveau volume de données, qu’il monte ensuite dans le conteneur à /var/lib/mysql. Ce volume permet à toutes les données qu’il contient de persister au-delà de la durée de vie du conteneur. C’est-à-dire que lorsqu’un conteneur est supprimé, ses modifications du système de fichiers sont également supprimées. Si une base de données stockait des données dans le conteneur et que le conteneur est supprimé, toutes ces données sont également supprimées. Les volumes persisteront à un emplacement particulier même au-delà de la suppression de leur conteneur.

Il est possible d’utiliser le même volume dans plusieurs conteneurs avec l’option de ligne de commande --volumes-from :

docker run -d --name="mysql-2" --volumes-from="mysql-1" mysql

Le conteneur mysql-2 a maintenant le volume de données de mysql-1 qui lui est attaché, en utilisant également le chemin /var/lib/mysql.
