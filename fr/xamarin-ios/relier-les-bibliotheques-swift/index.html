Relier les bibliothèques Swift

  • Liaison d’une bibliothèque Swift dans Xamarin.iOS

Sur cette page

  • Liaison d’une bibliothèque Swift dans Xamarin.iOS

Un guide facile à suivre qui vous guidera tout au long du processus de liaison des fichiers Swift .framework à utiliser dans un projet Xamarin.

  1. Lors de la création d’une bibliothèque dans Xcode, il est possible d’inclure les bibliothèques rapides. Ne le faites pas! Ils seront inclus dans votre application finale en tant que NAME.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib mais ils doivent être inclus en tant que NAME.app/Frameworks/libswift*.dylib

  2. Vous pouvez trouver ces informations ailleurs, mais cela vaut la peine d’être mentionné : n’incluez pas Bitcode dans la bibliothèque. Pour le moment, Xamarin n’inclut pas Bitcode pour iOS et Apple exige que toutes les bibliothèques prennent en charge les mêmes architectures.

Liaison d’une bibliothèque Swift dans Xamarin.iOS #

La liaison d’une bibliothèque Swift dans Xamarin.iOS suit le même processus pour Objective-C, comme indiqué dans https://developer.xamarin.com/guides/ios/advanced_topics/binding_objective-c/, mais avec quelques mises en garde.

  1. Une classe Swift doit hériter de NSObject pour être liée.
  2. Le compilateur Swift traduira les noms de classe et de protocole en quelque chose d’autre à moins que vous n’utilisiez l’annotation [@objc][1] (par exemple @objc(MyClass)) dans vos classes Swift pour spécifier le nom explicite de l’objectif c.
  3. Au moment de l’exécution, votre application doit inclure des bibliothèques de base rapides à côté de votre framework lié dans un dossier appelé Frameworks ;
  4. Lorsque l’application est poussée vers l’AppStore, elle doit inclure un dossier SwiftSupport à côté de votre dossier Payload. Ceux-ci sont à l’intérieur du fichier IPA.

Vous trouverez ici un exemple de liaison simple : https://github.com/Flash3001/Xamarin.BindingSwiftLibrarySample

Et un échantillon de liaison complet : https://github.com/Flash3001/iOSCharts.Xamarin

Veuillez trouver les étapes ci-dessous :

1.1 Préparez les classes Swift que vous souhaitez exporter #

Pour toute classe Swift que vous souhaitez utiliser, vous devez soit hériter de NSObject et rendre le nom Objective-C explicite à l’aide de l’annotation objc. Sinon, le compilateur Swift générera des noms différents. Vous trouverez ci-dessous un exemple de code illustrant à quoi pourrait ressembler une classe Swift. Notez que peu importe la classe dont elle hérite tant que la classe racine hérite de NSObject.

//Add this to specify explicit objective c name
@objc(MyClass)
open class MyClass: NSObject {
    open func getValue() -> String
    {
        return "Value came from MyClass.swift!";
    }
}

2. Créez une bibliothèque de graisses #

Un framework contient plusieurs fichiers, celui qui a besoin de manger un peu est NAME.framework/NAME (sans extension).

  • Copier Release-iphoneos/NAME.framework vers NAME.framework
  • Créez la bibliothèque FAT en utilisant :
    • lipo -create Release-iphonesimulator/NAME.framework/NAME Release-iphoneos/NAME.framework/NAME -output NAME.framework/NAME
  • Copiez les fichiers dans Release-iphonesimulator/NAME.framework/Modules/NAME.swiftmodule vers NAME.framework/Modules/NAME.swiftmodule (jusqu’à présent, il ne contenait que des fichiers de l’iphoneos)

3. Importer la bibliothèque #

Je suppose que vous avez déjà créé le projet de liaison dans Fichier -> Nouveau -> iOS -> Bibliothèque de liaison.

Xamarin prend en charge l’importation de .frameworks. Cliquez avec le bouton droit de la souris sur “Références natives” et cliquez sur “Ajouter une référence native”. Trouvez le cadre de graisse nouvellement créé et ajoutez-le.

[![Ajouter une référence native][11]][11]

[![Sélectionnez .framework][12]][12]

[![Référence][13]][13]

4. Créez l’ApiDefinition basée sur le fichier LIBRARY-Swift.h dans les en-têtes. #

Vous pouvez le faire manuellement, mais ce ne sera pas agréable. Vous pouvez utiliser Objectif Sharpie. L’outil que Xamarin utilise pour lier ses propres bibliothèques.

Comment l’utiliser sur https://developer.xamarin.com/guides/cross-platform/macios/binding/objective-sharpie/

La commande de base ressemblera à : sharpie bind -sdk iphoneos9.3 NAME-Swift.h

Si vous obtenez une System.Reflection.TargetInvocationException, c’est probablement parce que vous avez installé une version différente du SDK. Exécutez la commande suivante pour vérifier avec le SDK iPhone OS que vous avez installé :

sharpie xcode -sdks

Le fichier NOM-Swift.h se trouve dans NOM.framework/Headers/NOM-Swift.h

Remarque : Les classes Swift doivent hériter de “NSObject”, sinon NAME-Swift.h n’importera pas vos classes et Objetive Sharpie ne convertira rien.

[![Sharpie][13]][13]

Remplacez le contenu de votre projet de liaison ApiDefinition.cs par celui que vous venez de créer. [![Remplacer le contenu du fichier][14]][14]

5. Modifiez tous les [Protocol] et [BaseType] pour inclure le nom de la classe dans l’exécution Objective-C. #

Dans le cas où la classe ou le protocole Swift d’origine n’inclut pas l’annotation @objc(MyClass) comme spécifié à l’étape 1.1, ses noms Objective-C internes seront modifiés, vous devez donc le mapper sur le bon.

Tous les noms sont disponibles dans le fichier NAME-Swift.h au format suivant :

SWIFT_CLASS("_TtC11SwiftSample7MyClass")
@interface MyClass : NSObject

Et

SWIFT_PROTOCOL("_TtP6Charts17ChartDataProvider_")
@protocol ChartDataProvider

Pour définir le nom, vous utilisez la propriété BaseTypeAttribute.Name https://developer.xamarin.com/guides/cross-platform/macios/binding/binding-types-reference/#BaseType.Name pour les classes et ProcotolAttribute.Name https:// developer.xamarin.com/api/property/MonoTouch.Foundation.ProtocolAttribute.Name/ pour les protocoles.

[BaseType(typeof(NSObject), Name = "_TtC11SwiftSample7MyClass")]
interface MyClass

Le faire manuellement n’est pas cool. Vous pouvez utiliser cet outil https://github.com/Flash3001/SwiftClassify pour insérer tous les noms. (C’est un travail en cours. Mais c’est assez simple, rien qu’en regardant le code, vous comprendrez comment cela fonctionne).

6.1 Incluez toutes les dépendances Swift à exécuter. #

Si vous essayez de consommer la bibliothèque dans une application et que vous essayez de l’exécuter maintenant, elle se bloquera. L’erreur est due à l’absence de libswiftCore.dylib

Quelque chose comme ça:

Dyld Error Message:
  Library not loaded: @rpath/libswiftCore.dylib
  Referenced from: /Users/USER/Library/Developer/CoreSimulator/Devices/AC440891-C819-4050-8CAB-CE15AB4B3830/data/Containers/Bundle/Application/27D2EC87-5042-4FA7-9B80-A24A8971FB48/SampleUsing.app/Frameworks/SwiftSample.framework/SwiftSample
  Reason: image not found

Xamarin.iOS ne fournit pas de support officiel pour lier une bibliothèque Swift. Vous devez donc inclure manuellement les bibliothèques principales Swift dans les dossiers Frameworks et SwiftSupport. Les fichiers du dossier Frameworks sont différents pour Simulator et Device. Ils peuvent être trouvés dans /Applications/Xcode.app/Contents/Developer//XcodeDefault.xctoolchain/usr/lib/swift.Toolchains

Au lieu de copier manuellement les fichiers dans le dossier Framework, vous pouvez utiliser cette bibliothèque https://github.com/Flash3001/Xamarin.Swift3.Support . Il inclut toutes les dépendances dont Swift 3.1 a besoin, chacune dans un seul package NuGet.

[![Ajouter un paquet NuGet][16]][16]

Comme vous pouvez le constater, le package Nuget est inclus dans l’application grand public, et non dans la liaison elle-même. Si vous essayez de l’inclure dans la liaison, vous obtiendrez des erreurs de compilation.

Si vous créez un package Nuget, vous pouvez demander à Nuget de l’inclure en tant que dépendance.

[![Application en cours d’exécution][17]][17]

6.2. Découvrir quelles dépendances Swift inclure. #

Une chose importante à faire est de déterminer chaque package que vous devez inclure dans votre projet. Une reliure simple nécessitera généralement :

     libswiftCore.dylib
     libswiftCoreGraphics.dylib
     libswiftCoreImage.dylib
     libswiftDarwin.dylib
     libswiftDispatch.dylib
     libswiftFoundation.dylib
     libswiftObjectiveC.dylib
     libswiftQuartzCore.dylib
     libswiftUIKit.dylib

Pour répertorier chaque dépendance, vous pouvez exécuter la commande suivante dans votre LibraryName.framework

otool -l -arch armv7 LibraryName | grep libswift

[![Liste des dépendances][18]][18]

N’incluez pas tous les packages disponibles dans NuGet pour Swift3, car ils pourraient augmenter la taille de votre application.

7. Incluez SwiftSupport pour pousser l’application vers l’AppStore. #

Apple exige que votre application soit envoyée avec un dossier SwiftSupport à côté de votre dossier Payload. Les deux se trouvent dans votre package IPA.

Vous pouvez utiliser ce script https://github.com/bq/ipa-packager pour faire ce travail pour vous.

Ce processus est le seul que l’utilisateur de la bibliothèque devra effectuer manuellement. Chaque fois qu’il essaie de pousser l’application vers l’AppStore.

[![Archive pour publication][19]][19]

Cliquez sur “Signer et distribuer” et enregistrer sur le disque [![Signer et distribuer][20]][20]

Décompressez votre .IPA [![Décompresser][21]][21]

Créez la nouvelle IPA en utilisant le script mentionné ci-dessus [![Nouvelle API][22]][22]

Si vous décompressez le fichier, il contiendra le dossier SwiftSupport. [![entrez la description de l’image ici][23]][23]

Remarques #

Lors de la construction d’une bibliothèque dans Xcode, il a une option pour inclure les bibliothèques rapides. Ne le faites pas! Ils seront inclus dans votre application finale en tant que NAME.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib mais ils doivent être inclus en tant que NAME.app/Frameworks/libswift*.dylib

Vous pouvez trouver ces informations ailleurs, mais cela vaut la peine de le mentionner : n’incluez pas Bitcode dans la bibliothèque. Pour le moment, Xamarin n’inclut pas Bitcode pour iOS et Apple exige que toutes les bibliothèques prennent en charge les mêmes architectures.
