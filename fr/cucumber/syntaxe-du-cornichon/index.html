Gherkin est un langage lisible par les entreprises pour l’automatisation des tests et la documentation des tests. Il est compris par Cucumber et existe ensemble en tant qu’outil de développement axé sur le comportement.

Syntaxe #

  • Feature : ce mot-clé signifie que ce qui suit est une base description ou nom de la fonctionnalité testée ou documentée.
  • Contexte : ce mot-clé signifie les étapes qui seront exécutées avant chaque scénario de la fonctionnalité.
  • Scénario : ce mot-clé représente le nom ou la description de base d’un scénario particulier testant la fonctionnalité.
  • Aperçu du scénario : ce mot-clé signifie que le scénario s’exécutera N fois pour chaque argument répertorié dans les exemples explicitement passés par le nom de la colonne entouré de crochets.
  • Exemples : ce mot-clé note la liste des arguments statiques qui seront passés dans un canevas de scénario.
  • Étant donné : ce mot-clé représente une étape donnée, ou une condition préalable qui est supposée avant continue. Dans le paradigme Arrange, Act, Assert, donné représente “Organiser”.
  • When : ce mot-clé représente une étape when, ou le comportement qui doit être affirmé contre. Dans le paradigme Arrange, Act, Assert, donné représente “Agir”.
  • Then : ce mot-clé représente une étape then, ou autrement dit, l’étape de validation du résultat d’un comportement. Dans le paradigme Arrange, Act, Assert, given représente “Assert”.
  • Et : ce mot clé est utilisé conjointement avec l’un des mots clés ci-dessus. Si vous avez deux instructions données, au lieu d’appeler explicitement Given deux fois, vous pouvez dire, " étant donné A et B".

Étapes paramétrées #

Lors de l’écriture de Gherkin, il peut arriver que vous souhaitiez paramétrer vos étapes de réutilisation par l’ingénieur qui met en œuvre les plans de test. Les étapes reçoivent des paramètres via des groupes de capture d’expressions régulières. (Note d’ingénierie : Si vous n’avez pas de paramètres correspondants pour chaque groupe de capture dans votre expression régulière, vous pouvez vous attendre à ce qu’une “exception concombre : incompatibilité d’arité” soit levée) Dans l’exemple ci-dessous, nous avons décidé d’envelopper les arguments en double guillemets, ainsi que d’accepter des nombres entiers comme arguments.

 Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with "valid" credentials
        Then the user should be logged in

    Scenario: The user attempts to log in with invalid credentials 
        This scenario tests that a user is not able to log in when
        they enter invalid credentials

        Given the user is on the login page
        When the user signs in with "invalid" credentials
        Then the user should be logged in

    Scenario: The user is locked out after too many failed attempts
        This scenario validates that the user is locked out
        of their account after failing three consecutive 
        attempts to log in

        Given the user is on the login page
        When the user fails to log in 3 times
        Then the user should be locked out of their account

Contexte de la fonctionnalité #

Comme vous l’avez peut-être remarqué dans l’exemple ci-dessus, nous réécrivons plusieurs fois la même étape :

Given the user is on the login page

Cela peut être exceptionnellement fastidieux, surtout si nous avons plus d’une étape donnée qui est réutilisée. Gherkin fournit une solution à cela en nous donnant un autre mot-clé avec lequel travailler : Contexte :.

Le mot-clé background sert à exécuter les étapes déclarées en dessous avant chaque scénario de la fonctionnalité. Assurez-vous de ne pas ajouter d’étape d’arrière-plan, sauf si vous êtes certain qu’elle est nécessaire pour chaque scénario. Comme les autres mots-clés, Background est suivi d’une description ou d’un nom et peut avoir des commentaires en dessous. Tout comme Feature et Scenario, Background doit être précédé de deux-points.

    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription

    - User will get a generic error message following 
      login attempt with invalid credentials

        Given the user is on the login page

        Then the user should be logged in

        This scenario tests that a user is not able to log in when
        they enter invalid credentials

        When the user signs in with "invalid" credentials
        Then the user should be logged in

        This scenario validates that the user is locked out
        of their account after failing three consecutive 
        attempts to log in

Mots clés #

À des fins de documentation, vous souhaiterez peut-être filtrer les plans de test ou les scénarios par catégories. Les développeurs peuvent souhaiter exécuter des tests basés sur ces mêmes catégories. Gherkin vous permet de catégoriser les fonctionnalités ainsi que les scénarios individuels via l’utilisateur de balises. Dans l’exemple ci-dessous, notez que le mot-clé Feature ci-dessus est la balise “@Automation”. Gherkin reconnaît cela comme une balise par l’utilisateur du symbole “@”. Dans cet exemple, l’ingénieur veut préciser que ces tests sont utilisés pour l’automatisation, où tous les tests ne sont pas automatisables, certains tests doivent être effectués par AQ manuelle.

Notez également que la balise @Production a été ajoutée au scénario de test de verrouillage de l’utilisateur. Dans cet exemple, cela est dû au fait que ce scénario n’est actif que dans l’environnement de production de l’application. Les développeurs ne veulent pas que leurs comptes sandbox soient verrouillés pendant le développement. Cette balise leur permet d’imposer que ce test ne soit exécuté que sur l’environnement de production.

Enfin, le plan de scénario a la balise @Staging. Pour les besoins de cet exemple, cela est dû au fait que les comptes utilisés sont des comptes intermédiaires et ne fonctionneront pas dans les autres environnements. Comme la balise @Production, cela garantit que ces tests ne seront exécutés que dans l’environnement Staging.

Ce ne sont là que quelques exemples d’où, comment et pourquoi vous pouvez utiliser des balises. En fin de compte, ces balises auront un sens pour vous et les développeurs et peuvent être n’importe quoi et être utilisées pour catégoriser comme bon vous semble.

Conseils de cornichon #

  • Chaque scénario teste un comportement
  • Les scénarios sont écrits de manière déclarative
  • Évitez les détails accessoires à l’intérieur du scénario
  • Omettre l’évidence
  • Eviter les démarches conjointes
  • Gardez vos scénarios courts
  • Vous n’avez pas besoin de plusieurs scénarios dans la même fonctionnalité
  • Utilisez des noms de scénarios descriptifs
  • N’avoir qu’une seule étape Quand
  • Utilisez le “devrait” dans les étapes suivantes

Les bases #

Cet exemple passera en revue la structure de base d’un fichier de fonctionnalités Cucumber dans Gherkin. Les fichiers de fonctionnalités utilisent plusieurs mots-clés dans la syntaxe de base.

Regardons les mots-clés de base :

  • Feature : ce mot-clé signifie que ce qui suit est une description ou un nom de base de la fonctionnalité testée ou documentée.
  • Scenario : ce mot-clé représente le nom ou la description de base d’un scénario particulier testant la fonctionnalité.
  • Étant donné, ce mot-clé représente une étape donnée ou une condition préalable qui est supposée avant de continuer. Dans le paradigme Arrange, Act, Assert, given représente “Arrange”.
  • Quand ce mot-clé représente une étape quand, ou le comportement qui doit être affirmé. Dans le paradigme Arrange, Act, Assert, given représente “Act”.
  • Alors ce mot-clé représente une étape puis, c’est-à-dire l’étape de validation du résultat d’un comportement. Dans le paradigme Arrange, Act, Assert, given représente “Assert”.
  • Et Ce mot-clé est utilisé conjointement avec l’un des mots-clés ci-dessus. Si vous avez deux déclarations données, au lieu d’appeler explicitement Given deux fois, vous pouvez dire “Données A et B”.
  • Mais Ce mot-clé est utilisé conjointement avec Étant donné, Quand et Alors pour signifier que quelque chose ne devrait pas arriver. Alors A mais pas B.

Tous les mots-clés doivent être sur une nouvelle ligne et doivent être le premier mot sur une nouvelle ligne afin d’être reconnus par l’analyseur Gherkin. Les mots-clés Feature et Scenario doivent être suivis immédiatement de deux-points, comme indiqué dans l’exemple ci-dessous. Étant donné, Quand, Alors et Et ne nécessitent pas de deux-points.

En plus des mots-clés, vous pouvez rédiger des descriptions et des commentaires. Les descriptions apparaissent après le mot-clé mais sur la même ligne, tandis que les commentaires apparaissent sur les lignes sous les mots-clés. Lors de la rédaction de commentaires sur les fonctionnalités, il est d’usage de fournir des règles explicites décrivant les limites et les conditions qui conduisent à différents résultats de comportements.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with valid credentials
        Then the user should be logged in

Aperçu du scénario #

Dans certains cas, vous voudrez peut-être réexécuter le même scénario encore et encore, en remplaçant les arguments. Dans ce cas, Gherkin fournit plusieurs nouveaux mots-clés pour s’adapter à cette situation, Scenario Outline : et Example :. Le mot-clé Scenario Outline indique à Cucumber que le scénario va s’exécuter plusieurs fois en remplaçant les arguments d’une liste. Le mot-clé Exemples est appelé avant que la liste ne soit explicitement notée. Les arguments pour les contours de scénario doivent être entourés de crochets. Dans l’exemple ci-dessous, notez que les noms d’argument entourés de crochets correspondent aux noms de colonne répertoriés sous Exemples. Chaque colonne est séparée par des barres verticales, avec des noms de colonne sur la première ligne.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Background: The user starts out on the login page
        Given the user is on the login page

    Scenario Outline: The user successfully logs in with their account
         This scenario outlines tests in which various users attempt
         to sign in successfully 

         When the user enters their <username>
         And the user enters their <password>
         Then the user should be successfully logged on

         Examples:
         | username | password |
         | frank    | 1234     |
         | jack     | 4321     |
