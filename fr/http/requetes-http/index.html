Paramètres #

Méthode HTTP Objectif
OPTIONS Récupérer des informations sur les options de communication (méthodes et en-têtes disponibles) disponibles sur l’URI de requête spécifié.
OBTENIR Récupérer les données identifiées par l’URI de requête, ou les données produites par le script disponible à l’URI de requête.
TÊTE Identique à GET sauf qu’aucun corps de message ne sera renvoyé par le serveur : uniquement les en-têtes.
‘POSTER’ Soumettez un bloc de données (spécifié dans le corps du message) au serveur pour l’ajouter à la ressource spécifiée dans l’URI de la demande. Le plus couramment utilisé pour le traitement des formulaires.
‘METTRE’ Stockez les informations jointes (dans le corps du message) en tant que ressource nouvelle ou mise à jour sous l’URI de requête donné.
SUPPRIMER Supprimer ou mettre en file d’attente pour suppression la ressource identifiée par l’URI de la demande.
TRACÉ Il s’agit essentiellement d’une commande echo : un serveur HTTP fonctionnel et conforme doit renvoyer l’intégralité de la requête dans le corps d’une réponse 200 (OK).

La méthode CONNECT est réservée par la spécification des définitions de méthodes pour une utilisation avec des proxys capables de basculer entre les proxys et les modes de tunneling (comme pour le tunneling SSL).

Envoi manuel d’une requête HTTP minimale à l’aide de Telnet #

Format de requête de base #

Dans HTTP 1.1, une requête HTTP minimale se compose d’une ligne de requête et d’un en-tête “Host” :

GET /search HTTP/1.1 \r\n
Host: google.com \r\n
\r\n

La première ligne a ce format :

Method Request-URI HTTP-Version CRLF

Method doit être une méthode HTTP valide ; l’un des [[1]][1][[2]][2] :

  • ‘OPTIONS’
  • ‘OBTENIR’
  • ‘TÊTE’
  • ‘POSTER’
  • ‘METTRE’
  • ‘SUPPRIMER’
  • “PATCH”
  • ‘TRACER’
  • ‘CONNECTER’

Request-URI indique soit l’URI, soit le chemin d’accès à la ressource demandée par le client. Il peut s’agir soit :

  • un URI complet, y compris le schéma, l’hôte, le port (facultatif) et le chemin ; ou
  • un chemin, auquel cas l’hôte doit être spécifié dans l’en-tête Host

HTTP-Version indique la version du protocole HTTP que le client utilise. Pour les requêtes HTTP 1.1, cela doit toujours être HTTP/1.1.

La ligne de requête se termine par un couple retour chariot-saut de ligne, généralement représenté par \r\n.

[1] : https://www.w3.org/Protocols/rfc2616/rfc2616-sec5.html [2] : https://tools.ietf.org/html/rfc5789

Champs d’en-tête de requête #

Des champs d’en-tête (généralement simplement appelés “en-têtes”) peuvent être ajoutés à une requête HTTP pour fournir des informations supplémentaires avec la requête. Un en-tête a une sémantique similaire aux paramètres passés à une méthode dans n’importe quel langage de programmation qui prend en charge de telles choses.

Une requête avec les en-têtes Host, User-Agent et Referer pourrait ressembler à ceci :

GET /search HTTP/1.1 \r\n
Host: google.com \r\n
User-Agent: Chrome/54.0.2803.1 \r\n
Referer: http://google.com/ \r\n
\r\n

Une liste complète des en-têtes de requête HTTP 1.1 pris en charge est disponible dans [la spécification][1]. Les plus courants sont :

  • Host - la partie nom d’hôte de l’URL de la requête (requis dans HTTP/1.1)
  • User-Agent - une chaîne qui représente l’agent utilisateur qui demande ;
  • Referer - l’URI à partir duquel le client a été référé ; et
  • If-Modified-Since - donne une date que le serveur peut utiliser pour déterminer si une ressource a changé et indique que le client peut utiliser une copie en cache si ce n’est pas le cas.

Un en-tête doit être formé comme Name: Value CRLF. Name est le nom de l’en-tête, tel que User-Agent. Value est la donnée qui lui est assignée, et la ligne doit se terminer par un CRLF. Les noms d’en-tête ne sont pas sensibles à la casse et ne peuvent utiliser que des lettres, des chiffres et les caractères !#$%&'*+-.^_`|~ (section RFC7230 [3.2.6 Composants de valeur de champ][2]) .

Le nom du champ d’en-tête “Referer” est une faute de frappe pour “referrer”, introduite accidentellement dans [RFC1945][3].

[1] : http://httpwg.org/specs/rfc7231.html#request.header.fields [2] : http://httpwg.org/specs/rfc7230.html#field.components [3] : https://tools.ietf.org/html/rfc1945

## Corps des messages Certaines requêtes HTTP peuvent contenir un corps de message. Il s’agit de données supplémentaires que le serveur utilisera pour traiter la demande. Les corps de message sont le plus souvent utilisés dans les requêtes POST ou PATCH et PUT, pour fournir de nouvelles données que le serveur doit appliquer à une ressource.

Les requêtes qui incluent un corps de message doivent toujours inclure sa longueur en octets avec l’en-tête “Content-Length”.

Un corps de message est inclus * après * tous les en-têtes et un double CRLF. Un exemple de requête PUT avec un corps pourrait ressembler à ceci :

PUT /files/129742 HTTP/1.1\r\n
Host: example.com\r\n
User-Agent: Chrome/54.0.2803.1\r\n
Content-Length: 202\r\n
\r\n
This is a message body. All content in this message body should be stored under the 
/files/129742 path, as specified by the PUT specification. The message body does
not have to be terminated with CRLF.

Les requêtes HEAD et TRACE ne doivent pas inclure de corps de message.
