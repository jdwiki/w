Paramètres #

Code d’état Phrase de raison — Description
100 Continuer : le client doit envoyer la partie suivante d’une requête en plusieurs parties.
101 Changement de protocole : le serveur modifie la version ou le type de protocole utilisé dans cette communication.
200 OK — le serveur a reçu et complété la demande du client.
201 Created — le serveur a accepté la demande et créé une nouvelle ressource, qui est disponible sous l’URI dans l’en-tête Location.
202 Accepté — le serveur a reçu et accepté la demande du client, mais il n’a pas encore commencé ou terminé le traitement.
203 Informations ne faisant pas autorité : le serveur renvoie des données qui peuvent être un sous-ensemble ou un sur-ensemble des informations disponibles sur le serveur d’origine. Principalement utilisé par les mandataires.
204 Aucun contenu : utilisé à la place de 200 (OK) lorsqu’il n’y a pas de corps dans la réponse.
205 Réinitialiser le contenu — identique à 204 (pas de contenu), mais le client doit recharger la vue de document active.
206 Contenu partiel : utilisé à la place de 200 (OK) lorsque le client a demandé un en-tête “Plage”.
300 Choix multiples : la ressource demandée est disponible sur plusieurs URI et le client doit rediriger la demande vers un URI spécifié dans la liste du corps du message.
301 Moved Permanently : la ressource demandée n’est plus disponible à cet URI, et le client doit rediriger cette demande et toutes les demandes futures vers l’URI spécifié dans l’en-tête “Location”.
302 Trouvé — la ressource réside temporairement sous un URI différent. Cette demande doit être redirigée lors de la confirmation de l’utilisateur vers l’URI dans l’en-tête “Location”, mais les demandes futures ne doivent pas être modifiées.
303 Voir Autre — très similaire à 302 (Trouvé), mais ne nécessite pas d’entrée de l’utilisateur pour rediriger vers l’URI fourni. L’URI fourni doit être récupéré avec une requête GET.
304 Non modifié : le client a envoyé un en-tête “If-Modified-Since” ou similaire, et la ressource n’a pas été modifiée depuis ; le client doit afficher une copie en cache de la ressource.
305 Utiliser un proxy : la ressource demandée doit être demandée à nouveau via le proxy spécifié dans le champ d’en-tête “Location”.
307 Redirection temporaire — identique à 302 (Found), mais les clients HTTP 1.0 ne prennent pas en charge les réponses 307.
400 Bad Request — le client a envoyé une requête mal formée contenant des erreurs de syntaxe, et doit modifier la requête pour la corriger avant de la répéter.
401 Non autorisé — la ressource demandée n’est pas disponible sans authentification. Le client peut répéter la demande en utilisant un en-tête “Authorization” pour fournir les détails d’authentification.
402 Paiement requis : code de statut réservé et non spécifié à utiliser par les applications qui nécessitent des abonnements utilisateur pour afficher le contenu.
403 Interdit — le serveur comprend la demande, mais refuse de la satisfaire en raison des contrôles d’accès existants. La demande ne doit pas être répétée.
404 Not Found — aucune ressource disponible sur ce serveur ne correspond à l’URI demandé. Peut être utilisé à la place de 403 pour éviter d’exposer les détails du contrôle d’accès.
405 Méthode non autorisée : la ressource ne prend pas en charge la méthode de requête (verbe HTTP) ; l’en-tête “Autoriser” répertorie les méthodes de requête acceptables.
406 Non acceptable : la ressource a des caractéristiques qui violent les en-têtes d’acceptation envoyés dans la demande.
407 Authentification proxy requise — semblable à 401 (non autorisé), mais indique que le client doit d’abord s’authentifier auprès du proxy intermédiaire.
408 Request Timeout : le serveur attendait une autre demande du client, mais aucune n’a été fournie dans un délai acceptable.
409 Conflit : la demande n’a pas pu être traitée car elle est en conflit avec l’état actuel de la ressource.
410 Gone — similaire à 404 (Not Found), mais indique une suppression permanente. Aucune adresse de réexpédition n’est disponible.
411 Length Required — le client n’a pas spécifié d’en-tête Content-Length valide et doit le faire avant que le serveur n’accepte cette requête.
412 Échec de la précondition : la ressource n’est pas disponible avec toutes les conditions spécifiées par les en-têtes conditionnels envoyés par le client.
413 Request Entity Too Large — le serveur est actuellement incapable de traiter un corps de message de la longueur que le client a envoyé.
414 Request-URI Too Long — le serveur refuse la requête car le Request-URI est plus long que ce que le serveur est prêt à interpréter.
415 Type de média non pris en charge — le serveur ne prend pas en charge le MIME ou le type de média spécifié par le client et ne peut pas répondre à cette demande.
416 Plage demandée non satisfaisante : le client a demandé une plage d’octets, mais le serveur ne peut pas fournir de contenu correspondant à cette spécification.
417 Expectation Failed : le client a spécifié des contraintes dans l’en-tête “Expect” que le serveur ne peut pas respecter.
500 Erreur interne du serveur : le serveur a rencontré une condition ou une erreur inattendue qui l’empêche de répondre à cette demande.
501 Non implémenté : le serveur ne prend pas en charge la fonctionnalité requise pour traiter la demande. Habituellement utilisé pour indiquer une méthode de demande qui n’est pas prise en charge sur toute ressource.
502 Mauvaise passerelle : le serveur est un proxy et a reçu une réponse non valide du serveur en amont lors du traitement de cette requête.
503 Service indisponible : le serveur est soumis à une charge élevée ou en cours de maintenance et n’a pas la capacité de répondre à cette demande pour le moment.
504 Délai d’expiration de la passerelle : le serveur est un proxy et n’a pas reçu de réponse du serveur en amont en temps voulu.
505 Version HTTP non prise en charge — le serveur ne prend pas en charge la version du protocole HTTP avec laquelle le client a fait sa demande.

Format de réponse de base #

Lorsqu’un serveur HTTP reçoit une [requête HTTP][1] bien formée, il doit traiter les informations contenues dans la requête et renvoyer une réponse au client. Une simple réponse HTTP 1.1 peut ressembler à l’un des éléments suivants, généralement suivi d’un certain nombre de champs d’en-tête et éventuellement d’un corps de réponse :

HTTP/1.1 200 OK \r\n

HTTP/1.1 404 Not Found \r\n

HTTP/1.1 503 Service Unavailable \r\n

Une simple réponse HTTP 1.1 a ce format :

HTTP-Version Status-Code Reason-Phrase CRLF

Comme dans une requête, HTTP-Version indique la version du protocole HTTP utilisée ; pour HTTP 1.1, il doit toujours s’agir de la chaîne HTTP/1.1.

Status-Code est un code à trois chiffres qui indique l’état de la demande du client. Le premier chiffre de ce code est la classe d’état, qui place le code d’état dans l’une des 5 catégories de réponse [[1]][2] :

  • 1xx Informationnel - le serveur a reçu la demande et le traitement se poursuit
  • 2xx Succès - le serveur a accepté et traité la demande
  • 3xx Redirection - une action supplémentaire est nécessaire de la part du client pour compléter la demande
  • 4xx Erreurs client - le client a envoyé une demande mal formée ou ne peut pas être satisfaite
  • 5xx Erreurs de serveur - la requête était valide, mais le serveur ne peut pas la satisfaire pour le moment

Reason-Phrase est une courte description du code d’état. Par exemple, le code « 200 » a une expression de raison « OK » ; le code 404 a une phrase Not Found. Une liste complète des phrases de raison est disponible dans Paramètres, ci-dessous, ou dans la [spécification HTTP] [2].

La ligne se termine par un couple retour chariot-saut de ligne, généralement représenté par \r\n.

[1] : https://www.wikiod.com/fr/http/requetes-http [2] : https://tools.ietf.org/html/rfc7231#section-6

En-têtes supplémentaires #

Comme une requête HTTP, une réponse HTTP peut inclure des en-têtes supplémentaires pour modifier ou augmenter la réponse qu’elle fournit.

Une liste complète des en-têtes disponibles est définie au [§6.2 de la spécification][1]. Les en-têtes les plus couramment utilisés sont :

  • Server, qui fonctionne comme un [User-Agent en-tête de requête][2] pour le serveur ;
  • Location, qui est utilisé sur les réponses d’état 201 et 3xx pour indiquer un URI vers lequel rediriger ; et
  • ETag, qui est un identifiant unique pour cette version de la ressource retournée pour permettre aux clients de mettre en cache la réponse.

Les en-têtes de réponse viennent après la ligne d’état et, comme pour les [en-têtes de requête][2], sont formés comme suit :

Name: Value CRLF

Name fournit le nom de l’en-tête, tel que ETag ou Location, et Value fournit la valeur que le serveur définit pour cet en-tête. La ligne se termine par un CRLF.

Une réponse avec des en-têtes pourrait ressembler à ceci :

HTTP/1.1 201 Created \r\n
Server: WEBrick/1.3.1 \r\n
Location: http://example.com/files/129742 \r\n

[1] : https://www.w3.org/Protocols/rfc2616/rfc2616-sec6.html#sec6.2 [2] : https://www.wikiod.com/fr/http/requetes-http/9848/additional-headers#t=201607230904445800258

## Corps des messages Comme pour les [corps de requête][1], les réponses HTTP peuvent contenir un corps de message. Cela fournit des données supplémentaires que le client traitera. Notamment, les réponses 200 OK à une requête GET bien formée doivent toujours fournir un corps de message contenant les données demandées. (S’il n’y en a pas, 204 Pas de contenu est une réponse plus appropriée).

Un corps de message est inclus après tous les en-têtes et un double CRLF. Comme pour les requêtes, sa longueur en octets doit être donnée avec l’en-tête Content-Length. Une réponse réussie à une requête GET pourrait donc ressembler à ceci :

HTTP/1.1 200 OK\r\n
Server: WEBrick/1.3.1\r\n
Content-Length: 39\r\n
ETag: 4f7e2ed02b836f60716a7a3227e2b5bda7ee12c53be282a5459d7851c2b4fdfd\r\n
\r\n
Nobody expects the Spanish Inquisition.
