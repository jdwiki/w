Il existe plusieurs raisons d’invalider un jeton JWT avant son heure d’expiration : compte supprimé/bloqué/suspendu, mot de passe ou autorisations modifiés, utilisateur déconnecté par l’administrateur.

JWT est autonome, signé et stocké en dehors du contexte du serveur, donc révoquer un jeton n’est pas une action simple.

Supprimer le jeton du stockage client #

Supprimez le jeton du stockage client pour éviter son utilisation

Les jetons sont émis par le serveur et vous ne pouvez pas forcer les navigateurs à supprimer un cookie/localStorage ou contrôler la façon dont les clients externes gèrent vos jetons. Évidemment si les attaquants ont volé le jeton avant la déconnexion, ils pourraient toujours utiliser le jeton, donc des mesures supplémentaires sont nécessaires côté serveur (voir ci-dessous pour la stratégie de liste noire des jetons)

Biscuits #

Vous ne pouvez pas forcer les navigateurs à supprimer un cookie. Le client peut configurer le navigateur de manière à ce que le cookie persiste, même s’il a expiré. Mais le serveur peut définir la valeur sur vide et inclure le champ expire pour invalider la valeur du cookie.

 Set-Cookie: token=deleted; path=/; expires=Thu, 01 Jan 1970 00:00:00 GMT

Supprimer ’token’ avec javascript #

document.cookie = 'token=; Path=/; Expires=Thu, 01 Jan 1970 00:00:01 GMT;';
localStorage.removeItem('token')
sessionStorage.removeItem('token')

Liste noire de jetons #

Marquez les jetons non valides, stockez-les jusqu’à leur date d’expiration et vérifiez-les dans chaque demande.

La liste noire brise l’apatridie JWT car elle nécessite de maintenir l’état. L’un des avantages de JWT est qu’il n’y a pas besoin de stockage sur le serveur, donc si vous avez besoin de révoquer des jetons sans attendre l’expiration, pensez aussi à l’inconvénient

Gérer la liste noire #

La liste noire peut être facilement gérée dans votre propre service/base de données. La taille de stockage ne serait probablement pas grande car elle n’est nécessaire que pour stocker les jetons qui se trouvaient entre la déconnexion et l’heure d’expiration.

Incluez le jeton complet ou uniquement l’ID unique “jti”. Définissez le iat (émis à) pour supprimer les anciens jetons.

Pour révoquer tous les jetons après la mise à jour des données critiques sur l’utilisateur (mot de passe, autorisations, etc.), définissez une nouvelle entrée avec sub et iat lorsque currentTime - maxExpiryTime < last iss​. L’entrée peut être ignorée lorsque currentTime - maxExpiryTime > lastModified (plus de jetons non expirés envoyés).

Faire pivoter les jetons #

Définissez un délai d’expiration court et faites pivoter les jetons. Émettez un nouveau jeton d’accès toutes les quelques requêtes. Utilisez actualiser les jetons pour permettre à votre application d’obtenir de nouveaux jetons d’accès sans avoir à s’authentifier à nouveau

Actualiser et accéder aux jetons #

  • jeton d’accès : autorise l’accès à une ressource protégée. Durée de vie limitée. Doivent être tenus secrets, les considérations de sécurité sont moins strictes en raison de leur durée de vie plus courte.

  • Actualiser le jeton : permet à votre application d’obtenir de nouveaux jetons d’accès sans avoir à s’authentifier à nouveau. Longue durée de vie. Stocker dans un stockage sécurisé à long terme

Recommandations d’utilisation :

  • Applications Web : actualisez le jeton d’accès avant son expiration, chaque fois que l’utilisateur ouvre l’application et à intervalles fixes. Vous pouvez également renouveler le jeton d’accès lorsqu’un utilisateur effectue une action. Si l’utilisateur utilise un jeton d’accès expiré, la session est considérée comme inactive et un nouveau jeton d’accès est requis. Ce nouveau jeton peut être obtenu avec un jeton d’actualisation ou nécessitant des informations d’identification

  • Applications mobiles/natives : connexion à l’application une et une seule fois. Le jeton d’actualisation n’expire pas et peut être échangé contre un JWT valide. Tenez compte des événements spéciaux comme le changement de mot de passe

Autres techniques courantes #

  • Autoriser le changement d’identifiant unique de l’utilisateur si le compte est compromis avec une nouvelle connexion utilisateur et mot de passe

  • Pour invalider les jetons lorsque l’utilisateur modifie son mot de passe ou ses autorisations, signez le jeton avec un hachage de ces champs. Si l’un de ces champs change, les jetons précédents ne sont automatiquement pas vérifiés. L’inconvénient est qu’il nécessite un accès à la base de données

  • Modifier l’algorithme de signature pour révoquer tous les jetons actuels dans un problème de sécurité majeur
