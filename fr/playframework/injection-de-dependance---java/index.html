Injection de dépendances avec Guice - Play 2.4, 2.5 #

Guice est le framework d’injection de dépendances par défaut (DI supplémentaire) de Play. D’autres frameworks peuvent également être utilisés, mais l’utilisation de Guice facilite les efforts de développement, car Play s’occupe des choses sous le voile.

Injection d’API Play #

À partir de Play 2.5, plusieurs API, qui étaient statiques dans les versions précédentes, doivent être créées avec DI. Ce sont, par exemple, Configuration, JPAApi, CacheApi, etc.

La méthode d’injection des API Play est différente pour une classe, qui est automatiquement injectée par Play et pour une classe personnalisée. L’injection dans une classe injectée automatiquement est aussi simple que de mettre l’annotation @Inject appropriée sur le champ ou le constructeur. Par exemple, pour injecter Configuration dans un contrôleur avec injection de propriété :

@Inject 
private Configuration configuration;

ou avec injection de constructeur :

private Configuration configuration;
@Inject
public MyController(Configuration configuration) {
    this.configuration = configuration;
}

L’injection dans une classe personnalisée, qui est enregistrée pour DI, doit être effectuée de la même manière que pour une classe injectée automatiquement - avec l’annotation @Inject.

L’injection à partir d’une classe personnalisée, qui n’est pas liée à DI, doit être effectuée par un appel explicite à un injecteur avec Play.current().injector(). Par exemple, pour injecter Configuration dans une classe personnalisée, définissez un membre de données de configuration comme ceci :

private Configuration configuration = Play.current().injector().instanceOf(Configuration.class);

Reliure par injection personnalisée #

La liaison par injection personnalisée peut être effectuée avec l’annotation @ImplementedBy ou de manière programmatique avec le module Guice.

Injection avec l’annotation @ImplementedBy #

L’injection avec l’annotation @ImplementedBy est le moyen le plus simple. L’exemple ci-dessous montre un service qui fournit une façade à cache.

  1. Le service est défini par une interface CacheProvider comme suit :

    @ImplementedBy(RunTimeCacheProvider.class)
public interface CacheProvider {  
   CacheApi getCache();  
}

  2. Le service est implémenté par une classe RunTimeCacheProvider :

    public class RunTimeCacheProvider implements CacheProvider {  
   @Inject  
   private CacheApi appCache;  
   @Override  
   public public CacheApi getCache() {  
     return appCache;  
   }  
}

Remarque : le membre de données appCache est injecté lors de la création d’une instance RunTimeCacheProvider. 3. L’inspecteur de cache est défini comme membre d’un contrôleur avec l’annotation @Inject et est appelé depuis le contrôleur :

public class HomeController extends Controller {
  @Inject
  private CacheProvider cacheProvider;
  ...      
  public Result getCacheData() {
        Object cacheData = cacheProvider.getCache().get("DEMO-KEY");
        return ok(String.format("Cache content:%s", cacheData));  
  }

L’injection avec l’annotation @ImplementedBy crée la liaison fixe : CacheProvider dans l’exemple ci-dessus est toujours instancié avec RunTimeCacheProvider. Une telle méthode ne convient que pour un cas, lorsqu’il existe une interface avec une seule implémentation. Cela ne peut pas aider pour une interface avec plusieurs implémentations ou une classe implémentée en tant que singleton sans interface abstraite. Honnêtement, @ImplementedBy sera utilisé dans de rares cas, voire tout. Il est plus probable d’utiliser la liaison programmatique avec le module Guice.

Liaison par injection avec un module Play par défaut #

Le module Play par défaut est une classe nommée Module dans le répertoire racine du projet défini comme ceci :

import com.google.inject.AbstractModule;  
public class Module extends AbstractModule {  
  @Override  
  protected void configure() {  
      // bindings are here           
  }  
}

Remarque : L’extrait ci-dessus montre la liaison à l’intérieur de la configuration, mais bien sûr, toute autre méthode de liaison sera respectée.

Pour la liaison programmatique de CacheProvider à RunTimeCacheProvider :

  1. Supprimez l’annotation @ImplementedBy de la définition de CacheProvider :

    public interface CacheProvider {
    CacheApi getCache();
    }

  2. Mettre en œuvre le module configurer comme suit :

    public class Module extends AbstractModule {
    @Override
    protected void configure() {
    bind(CacheProvider.class).to(RunTimeCacheProvider.class); }
    }

Reliure à injection flexible avec un module Play par défaut

RunTimeCacheProvider ne fonctionne pas bien dans les tests JUnit avec une fausse application (voir le sujet des tests unitaires). Ainsi, l’implémentation différente de CacheProvider est exigée pour les tests unitaires. La liaison par injection doit être effectuée en fonction de l’environnement.

Voyons un exemple.

  1. La classe FakeCache fournit une implémentation stub de CacheApi à utiliser lors de l’exécution de tests (son implémentation n’est pas si intéressante - c’est juste une carte).

  2. La classe FakeCacheProvider implémente CacheProvider à utiliser lors de l’exécution des tests :

    public class FakeCacheProvider implements CacheProvider {
    private final CacheApi fakeCache = new FakeCache(); @Override
    public CacheApi getCache() {
    return fakeCache; }
    }

  3. Le module est implémenté comme suit :

    public class Module extends AbstractModule { private final Environment environment; public Module(Environment environment, Configuration configuration) { this.environment = environment; } @Override
    protected void configure() {
    if (environment.isTest() ) { bind(CacheProvider.class).to(FakeCacheProvider.class); } else { bind(CacheProvider.class).to(RuntimeCacheProvider.class);
    } }
    }
    L’exemple n’est bon qu’à des fins éducatives. La liaison pour les tests à l’intérieur du module n’est pas la meilleure pratique, car cela couple entre l’application et les tests. La liaison pour les tests doit être effectuée plutôt par les tests eux-mêmes et le module ne doit pas être conscient de l’implémentation spécifique au test. Voyez comment mieux faire cela dans … .

Reliure par injection avec un module personnalisé #

Un module personnalisé est très similaire au module Play par défaut. La différence est qu’il peut avoir n’importe quel nom et appartenir à n’importe quel package. Par exemple, un module OnStartupModule appartient aux modules du package.

package modules;  
import com.google.inject.AbstractModule;  
public class OnStartupModule extends AbstractModule {  
   @Override  
   protected void configure() {  
       ...            
   }  
}

Un module personnalisé doit être explicitement activé pour l’appel par Play. Pour le module OnStartupModule, les éléments suivants doivent être ajoutés dans application.conf :

play.modules.enabled += "modules.OnStartupModule"
