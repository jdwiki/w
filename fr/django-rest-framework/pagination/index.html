La pagination est la division de grands ensembles de données en pages séparées et autonomes.

Sur django rest framework, la pagination permet à l’utilisateur de modifier la quantité de données dans chaque page et le style par lequel la division est appliquée.

[Introduction] Configurer le style de pagination globalement #

Afin de définir le style de pagination pour l’ensemble du projet, vous devez définir DEFAULT_PAGINATION_CLASS et PAGE_SIZE dans les paramètres du projet.

Pour ce faire, allez dans settings.py et sur la variable REST_FRAMEWORK, ajoutez ce qui suit :

REST_FRAMEWORK = {
    'DEFAULT_PAGINATION_CLASS': 
        'rest_framework.pagination.DESIRED_PAGINATION_STYLE',
    'PAGE_SIZE': 100
}

À la place de DESIRED_PAGINATION_STYLE, l’un des éléments suivants doit être placé :

  • PageNumberPagination : accepte un seul numéro de page dans les paramètres de requête de requête.

     http://your_api_url/a_table/?page=2

  • LimitOffsetPagination : Accepte un paramètre limit, qui indique le nombre maximum d’éléments qui seront retournés et un paramètre offset qui indique la position de départ de la requête par rapport au jeu de données. PAGE_SIZE n’a pas besoin d’être défini pour ce style.

     http://your_api_url/a_table/?limit=50&offset=100

  • CursorPagination : la pagination basée sur le curseur est plus complexe que les styles ci-dessus. Cela nécessite que l’ensemble de données présente un ordre fixe et ne permet pas au client de naviguer arbitrairement dans des positions de l’ensemble de données.

  • Des styles de pagination personnalisés peuvent être définis à la place de ce qui précède.

[Avancé] Pagination sur les vues/ensembles de vues non génériques #

Ceci est un sujet avancé, n’essayez pas sans avoir d’abord compris les autres exemples de cette page.

Comme indiqué dans le [Django Rest Framework on pagination][1] officiel :

La pagination n’est effectuée automatiquement que si vous utilisez les vues ou ensembles de vues génériques. Si vous utilisez un APIView standard, vous devrez appeler vous-même l’API de pagination pour vous assurer de renvoyer une réponse paginée. Voir le code source des classes mixins.ListModelMixin et generics.GenericAPIView pour un exemple.

Mais que se passe-t-il si nous voulons utiliser la pagination sur une vue/un ensemble de vues non générique ?

Bon descendons dans le terrier du lapin :

  1. Le premier arrêt est le référentiel officiel de Django Rest Framework et plus particulièrement le [django-rest-framework/rest_framework/generics.py][2]. La ligne spécifique vers laquelle ce lien pointe, nous montre comment les développeurs du framework gèrent la pagination dans leurs génériques.
    C’est exactement ce que nous allons utiliser pour notre vue également !

  2. Supposons que nous ayons une configuration de pagination globale comme celle montrée dans [l’exemple d’introduction de cette page][3] et supposons également que nous ayons une APIView à laquelle nous voulons appliquer la pagination.

  3. Puis sur views.py :

     from django.conf import settings
 from rest_framework.views import APIView

 class MyView(APIView):
     queryset = OurModel.objects.all()
     serializer_class = OurModelSerializer
     pagination_class = settings.DEFAULT_PAGINATION_CLASS # cool trick right? :)

     # We need to override get method to achieve pagination
     def get(self, request):
         ...
         page = self.paginate_queryset(self.queryset)
         if page is not None:
             serializer = self.serializer_class(page, many=True)
             return self.get_paginated_response(serializer.data)

         ... Do other stuff needed (out of scope of pagination)

     # Now add the pagination handlers taken from 
     #  django-rest-framework/rest_framework/generics.py

     @property
     def paginator(self):
         """
         The paginator instance associated with the view, or `None`.
         """
          if not hasattr(self, '_paginator'):
              if self.pagination_class is None:
                  self._paginator = None
              else:
                  self._paginator = self.pagination_class()
          return self._paginator

      def paginate_queryset(self, queryset):
          """
          Return a single page of results, or `None` if pagination is disabled.
          """
          if self.paginator is None:
              return None
          return self.paginator.paginate_queryset(queryset, self.request, view=self)

      def get_paginated_response(self, data):
          """
          Return a paginated style `Response` object for the given output data.
          """
          assert self.paginator is not None
          return self.paginator.get_paginated_response(data)

Nous avons maintenant un APIView qui gère la pagination !

[1] : http://www.django-rest-framework.org/api-guide/pagination/ [2] : https://github.com/encode/django-rest-framework/blob/master/rest_framework/generics.py#L155 [3] : https://www.wikiod.com/fr/django-rest-framework/pagination#[Introduction] Configurer le style de pagination globalement

[Intermédiaire] Remplacer le style de pagination et configurer la pagination par classe #

Remplacer le style de pagination :

Chaque style de pagination disponible peut être remplacé en créant une nouvelle classe qui hérite de l’un des styles disponibles, puis modifie ses paramètres :

class MyPagination(PageNumberPagination):
    page_size = 20
    page_size_query_param = 'page_size' 
    max_page_size = 200
    last_page_strings = ('the_end',)

Ces paramètres (tels qu’énumérés dans la [documentation officielle de pagination][1]) sont :

NuméroPagePagination

  • page_size : une valeur numérique indiquant la taille de la page. S’il est défini, cela remplace le paramètre “PAGE_SIZE”. Par défaut, la même valeur que la clé de paramètres PAGE_SIZE.
  • page_query_param : une valeur de chaîne indiquant le nom du paramètre de requête à utiliser pour le contrôle de la pagination.
  • page_size_query_param : s’il est défini, il s’agit d’une valeur de chaîne indiquant le nom d’un paramètre de requête qui permet au client de définir la taille de la page pour chaque requête. La valeur par défaut est “Aucune”, indiquant que le client peut ne pas contrôler la taille de page demandée.
  • max_page_size : s’il est défini, il s’agit d’une valeur numérique indiquant la taille de page demandée maximale autorisée. Cet attribut n’est valide que si page_size_query_param est également défini.
  • last_page_strings : une liste ou un tuple de valeurs de chaîne indiquant les valeurs pouvant être utilisées avec page_query_param pour demander la dernière page de l’ensemble. La valeur par défaut est ('dernier',)
  • template : le nom d’un modèle à utiliser lors du rendu des contrôles de pagination dans l’API navigable. Peut être remplacé pour modifier le style de rendu ou défini sur “Aucun” pour désactiver complètement les contrôles de pagination HTML. La valeur par défaut est "rest_framework/pagination/numbers.html".

LimitOffsetPagination

  • default_limit : une valeur numérique indiquant la limite à utiliser si aucune n’est fournie par le client dans un paramètre de requête. Par défaut, la même valeur que la clé de paramètres PAGE_SIZE.
  • limit_query_param : une valeur de chaîne indiquant le nom du paramètre de requête “limit”. La valeur par défaut est 'limite'.
  • offset_query_param : une valeur de chaîne indiquant le nom du paramètre de requête “offset”. La valeur par défaut est 'offset'.
  • max_limit : s’il est défini, il s’agit d’une valeur numérique indiquant la limite maximale autorisée pouvant être demandée par le client. La valeur par défaut est “Aucun”.
  • template : identique à PageNumberPagination.

CurseurPagination

  • page_size : identique à PageNumberPagination.
  • cursor_query_param : une valeur de chaîne indiquant le nom du paramètre de requête “cursor”. Par défaut, 'curseur'.
  • ordering : il doit s’agir d’une chaîne, ou d’une liste de chaînes, indiquant le champ sur lequel la pagination basée sur le curseur sera appliquée. Par exemple : commande = 'slug'. La valeur par défaut est -created. Cette valeur peut également être remplacée en utilisant OrderingFilter sur la vue.
  • template : identique à PageNumberPagination.

Configurer la pagination par classe :

En plus de la possibilité de paramétrer le style Pagination globalement, un paramétrage par classe est disponible :

class MyViewSet(viewsets.GenericViewSet):
    pagination_class = LimitOffsetPagination

Désormais, seul MyViewSet a une pagination LimitOffsetPagination.

Un style de pagination personnalisé peut être utilisé de la même manière :

class MyViewSet(viewsets.GenericViewSet):
    pagination_class = MyPagination

[1] : http://www.django-rest-framework.org/api-guide/pagination/#api-reference

[Intermédiaire] Exemple d’utilisation complexe #

Supposons que nous ayons une API complexe, avec de nombreuses vues génériques et quelques ensembles de vues génériques. Nous voulons activer PageNumberPagination pour chaque vue, sauf une (vue générique ou ensemble de vues, cela ne fait aucune différence) pour laquelle nous voulons un cas personnalisé de LimitOffsetPagination.

Pour y parvenir nous devons :

  1. Sur le settings.py, nous placerons notre pagination par défaut afin de l’activer pour chaque vue/viewset générique et nous définirons PAGE_SIZE sur 50 éléments :

     REST_FRAMEWORK = {
     'DEFAULT_PAGINATION_CLASS': 
         'rest_framework.pagination.PageNumberPagination',
     'PAGE_SIZE': 50
 }

  2. Maintenant, dans notre views.py (ou dans un autre .py ex : paginations.py), nous devons remplacer le LimitOffsetPagination :

     from rest_framework.pagination import LimitOffsetPagination

 class MyOffsetPagination(LimitOffsetPagination):
     default_limit = 20
     max_limit = 1000

    A custom LimitOffsetPagination with PAGE_ZISE of 20 items and maximum limit of 1000 items.

  3. Dans notre views.py, nous devons définir la pagination_class de notre vue spéciale :

     imports ...

 # ===================================
 #    PageNumberPagination classes
 # ===================================

 class FirstView(generics.ListAPIView):
     ...

 class FirstViewSet(viewsets.GenericViewSet):
     ...

 ...

 # ===================================
 #     Our custom Pagination class
 # ===================================

 class IAmSpecialView(generics.ListAPIView):
     pagination_class = MyOffsetPagination
     ...

Désormais, chaque vue/ensemble de vues génériques de l’application a PageNumberPagination, à l’exception de la classe IAmSpecial, qui est en effet spéciale et possède sa propre LimitOffsetPagination personnalisée.

[Intermédiaire] Pagination sur une vue basée sur une fonction #

Nous avons vu dans ces exemples ([ex_1][1], [ex_2][2]) comment utiliser et remplacer les classes de pagination dans n’importe quelle vue de base de classe générique.
Que se passe-t-il lorsque nous voulons utiliser la pagination dans une vue basée sur les fonctions ?

Supposons également que nous voulions créer une vue basée sur une fonction pour MyModel avec PageNumberPagination, répondant uniquement à une requête GET. Alors:

from rest_framework.pagination import PageNumberPagination


@api_view(['GET',])
def my_function_based_list_view(request):
    paginator = PageNumberPagination()
    query_set = MyModel.objects.all()
    context = paginator.paginate_queryset(query_set, request)
    serializer = MyModelSerializer(context, many=True)
    return paginator.get_paginated_response(serializer.data)
Nous pouvons également faire ce qui précède pour une pagination personnalisée en modifiant cette ligne : 
paginator = PageNumberPagination()

pour ça

paginator = MyCustomPagination()

à condition que nous ayons défini MyCustomPagination [pour remplacer une pagination par défaut] [2]
