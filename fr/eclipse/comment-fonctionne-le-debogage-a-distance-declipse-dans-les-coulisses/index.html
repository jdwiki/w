Comment fonctionne le débogage à distance Eclipse dans les coulisses #

Le débogage d’Eclipse commence par ce que l’on appelle des agents.

La JVM, qui exécute les sources .class conformes, a une fonctionnalité qui permet d’injecter des bibliothèques externes (écrites en Java ou C++) dans la JVM, à peu près au moment de l’exécution. Ces bibliothèques externes sont appelées Agents et ont la capacité de modifier le contenu des fichiers .class exécutés. Ces agents ont accès à des fonctionnalités de la JVM qui ne sont pas accessibles à partir d’un code Java normal exécuté à l’intérieur de la JVM et ils peuvent être utilisés pour faire des choses intéressantes comme injecter et modifier le code source en cours d’exécution, le profilage, etc. Des outils comme ** JRebel ** utilise cette fonctionnalité pour réaliser sa magie.

Et pour passer un agent Lib à une JVM, vous le faites via des arguments de démarrage, en utilisant le

 agentlib:libname[=options] format.

Nous passions en fait un Agent Lib nommé jdwp à la JVM exécutant Tomcat. Le jdwp est une implémentation facultative spécifique à la JVM du * JDWP (Java Debug Wire Protocol) * qui est utilisée pour définir la communication entre un débogueur et une JVM en cours d’exécution. Son implémentation, si elle est présente, est fournie en tant que bibliothèque native de la JVM sous la forme jdwp.so ou jdwp.dll

Alors, qu’est-ce que ça fait ?

En termes simples, l’agent jdwp que nous transmettons remplit essentiellement la fonction d’être un lien entre l’instance JVM exécutant une application et un débogueur (qui peut être situé à distance ou en local). Puisqu’il s’agit d’une bibliothèque d’agents, elle a la capacité d’intercepter le code en cours d’exécution, de créer un pont entre la JVM et un débogueur et d’appliquer la fonctionnalité d’un débogueur sur la JVM. Étant donné que dans l’architecture JVM, la fonctionnalité de débogage ne se trouve pas dans la JVM elle-même mais est abstraite dans des outils externes (qui sont à juste titre appelés débogueurs), ces outils peuvent soit résider sur la machine locale exécutant la JVM en cours de débogage, soit être exécuté depuis une machine externe. C’est cette architecture modulaire découplée qui nous permet d’avoir une JVM s’exécutant sur une machine distante et d’utiliser le JDWP, d’avoir un débogueur distant capable de communiquer avec elle. En bref, voici comment fonctionne le débogueur Eclipse.
