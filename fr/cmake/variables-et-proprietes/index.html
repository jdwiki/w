La simplicité des variables CMake de base dément la complexité de la syntaxe complète des variables. Cette page documente les différents cas variables, avec des exemples, et rappelle les écueils à éviter.

Syntaxe #

  • set(variable_name value [description du type de CACHE [FORCE]])

Les noms de variables sont sensibles à la casse. Leurs valeurs sont de type string. La valeur d’une variable est référencée via :

${variable_name}

et est évalué à l’intérieur d’un argument entre guillemets

"${variable_name}/directory"

Variables et cache de variables globales #

Vous utiliserez principalement des “variables normales” :

set(VAR TRUE)
set(VAR "main.cpp")
set(VAR1 ${VAR2})

Mais CMake connaît également les “variables mises en cache” globales (persistantes dans CMakeCache.txt). Et si des variables normales et en cache du même nom existent dans la portée actuelle, les variables normales cachent celles en cache :

cmake_minimum_required(VERSION 2.4)
project(VariablesTest)

set(VAR "CACHED-init" CACHE STRING "A test")
message("VAR = ${VAR}")

set(VAR "NORMAL")
message("VAR = ${VAR}")

set(VAR "CACHED" CACHE STRING "A test" FORCE)
message("VAR = ${VAR}")

Sortie de la première exécution

VAR = CACHED-init
VAR = NORMAL
VAR = CACHED

Sortie de la deuxième exécution

VAR = CACHED
VAR = NORMAL
VAR = CACHED

Remarque : L’option FORCE annule/supprime également la variable normale de la portée actuelle.

Cas d’utilisation pour les variables mises en cache #

Il existe généralement deux cas d’utilisation (veuillez ne pas les utiliser à mauvais escient pour les variables globales) :

  1. Une valeur dans votre code doit être modifiable par l’utilisateur de votre projet, par ex. avec le cmakegui, [ccmake](https://cmake.org/cmake/help /latest/manual/ccmake.1.html) ou avec l’option cmake -D ... :

CMakeLists.txt / MyToolchain.cmake

   set(LIB_A_PATH "/some/default/path" CACHE PATH "Path to lib A")

Ligne de commande

   $ cmake -D LIB_A_PATH:PATH="/some/other/path" ..

Cela prédéfinit cette valeur dans le cache et la ligne ci-dessus ne la modifiera pas.

Interface graphique CMake

[![entrez la description de l’image ici][1]][1]

Dans l’interface graphique, l’utilisateur démarre d’abord le processus de configuration, puis peut modifier n’importe quelle valeur mise en cache et termine avec le démarrage de la génération de l’environnement de construction.

  1. De plus, CMake met en cache les résultats de recherche/test/identification du compilateur (il n’a donc pas besoin de le refaire chaque fois que vous réexécutez les étapes de configuration/génération)

    find_path(LIB_A_PATH libA.a PATHS "/some/default/path")

Ici, LIB_A_PATH est créé en tant que variable mise en cache.

Chaînes et listes #

Il est important de savoir comment CMake fait la distinction entre les listes et les chaînes simples. Lorsque vous écrivez :

set(VAR "a b c")

vous créez une chaîne avec la valeur "a b c". Mais quand vous écrivez cette ligne sans guillemets :

set(VAR a b c)

Vous créez plutôt une liste de trois éléments : "a", "b" et "c".

Les variables non-liste sont en fait aussi des listes (d’un seul élément).

Les listes peuvent être utilisées avec la commande list(), qui permet de concaténer des listes, de les rechercher, d’accéder à des éléments arbitraires, etc. ([documentation of list()](https://cmake.org/cmake/help/latest /commande/list.html)).

Un peu déroutant, une liste est aussi une chaîne. La ligne

set(VAR a b c)

est équivalent à

set(VAR "a;b;c")

Par conséquent, pour concaténer des listes, on peut également utiliser la commande set() :

set(NEW_LIST "${OLD_LIST1} ;${OLD_LIST2})"

Variable mise en cache (globale) #

set(my_global_string "a string value"
    CACHE STRING "a description about the string variable")
set(my_global_bool TRUE
    CACHE BOOL "a description on the boolean cache entry")

Dans le cas où une variable mise en cache est déjà définie dans le cache lorsque CMake traite la ligne respective (par exemple lorsque CMake est réexécuté), elle n’est pas modifiée. Pour écraser la valeur par défaut, ajoutez “FORCE” comme dernier argument :

set(my_global_overwritten_string "foo"
    CACHE STRING "this is overwritten each time CMake is run" FORCE)

Ajout d’indicateurs de profilage à CMake pour utiliser gprof #

La série d’événements ici est censée fonctionner comme suit :

  1. Compiler le code avec l’option -pg
  2. Code de lien avec l’option -pg
  3. Exécutez le programme
  4. Le programme génère le fichier gmon.out
  5. Exécutez le programme gprof

Pour ajouter des drapeaux de profilage, vous devez ajouter à votre CMakeLists.txt :

SET(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -pg")
SET(CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS} -pg")
SET(CMAKE_SHARED_LINKER_FLAGS "${CMAKE_SHARED_LINKER_FLAGS} -pg")

Cela doit ajouter des drapeaux pour compiler et lier, et utiliser après avoir exécuté le programme :

gprof ./mon_exe

Si vous obtenez une erreur comme :

gmon.out: No such file or directory

Cela signifie que la compilation n’a pas correctement ajouté les informations de profilage.

Variable locale #

set(my_variable "the value is a string")

Par défaut, une variable locale n’est définie que dans le répertoire courant et tous les sous-répertoires ajoutés via la commande add_subdirectory.

Pour étendre la portée d’une variable, il y a deux possibilités :

  1. CACHE it, ce qui le rendra disponible dans le monde entier 1. utilisez PARENT_SCOPE, qui le rendra disponible dans la portée parent. La portée parent est soit le fichier CMakeLists.txt dans le répertoire parent ou l’appelant de la fonction actuelle.

    Technically the parent directory will be the CMakeLists.txt file that included the current file via the add_subdirectory command.
