Cette rubrique explique comment mapper des relations de type un à un à l’aide d’Entity Framework.

Mappage un à zéro ou un #

Alors disons encore une fois que vous avez le modèle suivant :

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
}

public class MyDemoContext : DbContext
{
  public DbSet<Person> People { get; set; }
  public DbSet<Car> Cars { get; set; }
}

Et maintenant, vous voulez le configurer de manière à pouvoir exprimer la spécification suivante : une personne peut avoir une ou zéro voiture, et chaque voiture appartient à une personne exactement (les relations sont bidirectionnelles, donc si CarA appartient à PersonA, alors PersonA “possède ’ CarA).

Alors modifions un peu le modèle : ajoutez les propriétés de navigation et les propriétés de clé étrangère :

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
  public int CarId { get; set; }
  public virtual Car Car { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

Et la configuration :

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);                        
  }
}

À ce moment-là, cela devrait être explicite. La voiture a une personne requise (HasRequired()), la personne ayant une voiture en option (WithOptional()). Encore une fois, peu importe de quel côté vous configurez cette relation, soyez prudent lorsque vous utilisez la bonne combinaison de Has/With et Required/Facultatif. Du côté “Personne”, cela ressemblerait à ceci :

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
     this.HasOptional(p => p.Car).WithOptional(c => c.Person);                        
  }
}

Voyons maintenant le schéma db :

Regardez attentivement : vous pouvez voir qu’il n’y a pas de FK dans People pour faire référence à Car. De plus, le FK dans Car n’est pas le PersonId, mais le CarId. Voici le script réel pour le FK :

ALTER TABLE [dbo].[Cars]  WITH CHECK ADD  CONSTRAINT [FK_dbo.Cars_dbo.People_CarId] FOREIGN KEY([CarId])
REFERENCES [dbo].[People] ([PersonId])

Cela signifie donc que les propriétés de clé étrangère CarId et PersonId que nous avons dans le modèle sont fondamentalement ignorées. Ils sont dans la base de données, mais ce ne sont pas des clés étrangères, comme on pourrait s’y attendre. En effet, les mappages un à un ne prennent pas en charge l’ajout du FK dans votre modèle EF. Et c’est parce que les mappages un à un sont assez problématiques dans une base de données relationnelle.

L’idée est que chaque personne peut avoir exactement une voiture, et que cette voiture ne peut appartenir qu’à cette personne. Ou il peut y avoir des enregistrements de personnes auxquels aucune voiture n’est associée.

Alors, comment cela pourrait-il être représenté avec des clés étrangères ? Évidemment, il pourrait y avoir un PersonId dans Car et un CarId dans People. Pour faire respecter le fait que chaque personne ne peut avoir qu’une seule voiture, PersonId devrait être unique dans Car. Mais si PersonId est unique dans People, alors comment pouvez-vous ajouter deux enregistrements ou plus où PersonId est NULL (plus d’une voiture qui n’a pas de propriétaire) ? Réponse : vous ne pouvez pas (en fait, vous pouvez créer un index unique filtré dans SQL Server 2008 et versions ultérieures, mais oublions un instant cette technicité ; sans parler des autres RDBMS). Sans parler du cas où vous spécifiez les deux extrémités de la relation…

La seule véritable façon d’appliquer cette règle si les tables People et Car ont la ‘même’ clé primaire (mêmes valeurs dans les enregistrements connectés). Et pour ce faire, CarId dans Car doit être à la fois un PK et un FK pour le PK de People. Et cela rend tout le schéma un gâchis. Lorsque j’utilise ceci, je nomme plutôt le PK/FK dans Car PersonId, et le configure en conséquence :

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }        
  public virtual Car Car { get; set; }
}

public class Car
{        
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);
     this.HasKey(c => c.PersonId);
  }
}

Pas idéal, mais peut-être un peu mieux. Néanmoins, vous devez être vigilant lorsque vous utilisez cette solution, car elle va à l’encontre des conventions de dénomination habituelles, ce qui pourrait vous induire en erreur. Voici le schéma généré à partir de ce modèle :

Cette relation n’est donc pas appliquée par le schéma de base de données, mais par Entity Framework lui-même. C’est pourquoi vous devez être très prudent lorsque vous l’utilisez, ne laissez personne toucher directement à la base de données.

Mappage un à un #

Le mappage un à un (lorsque les deux côtés sont requis) est également une chose délicate.

Imaginons comment cela pourrait être représenté avec des clés étrangères. Encore une fois, un CarId dans People qui fait référence à CarId dans Car, et un PersonId dans Car qui fait référence à PersonId dans People.

Maintenant, que se passe-t-il si vous souhaitez insérer un dossier de voiture ? Pour que cela réussisse, il doit y avoir un PersonId spécifié dans cet enregistrement de voiture, car il est obligatoire. Et pour que ce PersonId soit valide, l’enregistrement correspondant dans People doit exister. OK, allons-y et insérons l’enregistrement de la personne. Mais pour que cela réussisse, un CarId valide doit figurer dans l’enregistrement de la personne — mais cette voiture n’est pas encore insérée ! Ce n’est pas possible, car nous devons d’abord insérer le dossier de la personne référée. Mais nous ne pouvons pas insérer l’enregistrement de la personne référée, car il renvoie à l’enregistrement de la voiture, il doit donc être inséré en premier (clé étrangère-ception :) ).

Cela ne peut donc pas non plus être représenté de manière «logique». Encore une fois, vous devez déposer l’une des clés étrangères. Lequel vous déposez dépend de vous. Le côté qui reste avec une clé étrangère est appelé le « dépendant », le côté qui reste sans clé étrangère est appelé le « principal ». Et encore une fois, pour garantir l’unicité de la dépendance, le PK doit être le FK, donc l’ajout d’une colonne FK et son importation dans votre modèle ne sont pas pris en charge.

Voici donc la configuration :

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasRequired(c => c.Person).WithRequiredDependent(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

À présent, vous devriez vraiment en avoir compris la logique :) N’oubliez pas que vous pouvez également choisir l’autre côté, faites juste attention à utiliser les versions Dependent/Principal de WithRequired (et vous devez encore configurer le PK dans Car).

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
    this.HasRequired(p => p.Car).WithRequiredPrincipal(c => c.Person);
  }
}

Si vous vérifiez le schéma de base de données, vous constaterez qu’il est exactement le même que dans le cas de la solution un-à-un ou zéro. C’est parce qu’encore une fois, cela n’est pas appliqué par le schéma, mais par EF lui-même. Alors encore une fois, soyez prudent :)

Mappage un ou zéro à un ou zéro #

Et pour finir, regardons brièvement le cas où les deux côtés sont facultatifs.

À présent, vous devriez vraiment vous ennuyer avec ces exemples :), donc je n’entre pas dans les détails et je joue avec l’idée d’avoir deux FK et les problèmes potentiels et vous avertis des dangers de ne pas appliquer ces règles dans le schéma mais juste dans EF lui-même.

Voici la configuration que vous devez appliquer :

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasOptional(c => c.Person).WithOptionalPrincipal(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Encore une fois, vous pouvez également configurer de l’autre côté, faites juste attention à utiliser les bonnes méthodes :)
