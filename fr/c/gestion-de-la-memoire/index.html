Pour gérer la mémoire allouée dynamiquement, la bibliothèque C standard fournit les fonctions malloc(), calloc(), realloc() et free(). Dans C99 et versions ultérieures, il existe également aligned_alloc(). Certains systèmes fournissent également alloca().

Syntaxe #

  • void *aligned_alloc(size_t alignement, size_t taille); /* Seulement depuis C11 */
  • void *calloc(size_t nelements, size_t size);
  • void libre(void *ptr);
  • void *malloc(taille_t taille);
  • void *realloc(void *ptr, size_t size);
  • void *alloca(size_t size); /* de alloca.h, non standard, non portable, dangereux. */

Paramètres #

nom descriptif
taille (malloc, realloc et aligned_alloc) taille totale de la mémoire en octets. Pour aligned_alloc, la taille doit être un multiple entier de l’alignement.
taille (calloc) taille de chaque élément
éléments nombre d’éléments
ptr pointeur vers la mémoire allouée précédemment renvoyé par malloc, calloc, realloc ou aligned_alloc
alignement alignement de la mémoire allouée

Notez que aligned_alloc() n’est défini que pour C11 ou version ultérieure.

Les systèmes tels que ceux basés sur [POSIX][1] fournissent d’autres moyens d’allouer de la mémoire alignée (par exemple posix_memalign()) , et disposent également d’autres options de gestion de la mémoire (par exemple, mmap()).

[1] : https://www.wikiod.com/fr/docs/posix

Libérer de la mémoire #

Il est possible de libérer de la mémoire allouée dynamiquement en appelant [free()][1].

int *p = malloc(10 * sizeof *p); /* allocation of memory */
if (p == NULL) 
{
    perror("malloc failed");
    return -1;
}

free(p); /* release of memory */
/* note that after free(p), even using the *value* of the pointer p
   has undefined behavior, until a new value is stored into it. */

/* reusing/re-purposing the pointer itself */
int i = 42;
p = &i; /* This is valid, has defined behaviour */

La mémoire pointée par p est récupérée (soit par l’implémentation de la libc, soit par le système d’exploitation sous-jacent) après l’appel à free(), donc l’accès à ce bloc de mémoire libéré via p conduira à undefined behavior. Les pointeurs qui référencent des éléments de mémoire qui ont été libérés sont communément appelés [pointeurs pendants] [2] et présentent un risque de sécurité. De plus, la norme C stipule que même [l’accès à la valeur](https://www.wikiod.com/fr/c/comportement-indefini#Lire la valeur du pointeur qui a été libéré 201608050732042665317) d’un pointeur pendant a un comportement indéfini. Notez que le pointeur p lui-même peut être réutilisé comme indiqué ci-dessus.

Veuillez noter que vous ne pouvez appeler free() que sur des pointeurs directement renvoyés par les fonctions malloc(), calloc(), realloc() et aligned_alloc(), ou lorsque la documentation indique vous la mémoire a été allouée de cette façon (des fonctions comme strdup() sont des exemples notables). Libérer un pointeur qui est,

  • obtenu en utilisant l’opérateur & sur une variable, ou
  • au milieu d’un bloc alloué,

est interdit. Une telle erreur ne sera généralement pas diagnostiquée par votre compilateur mais conduira l’exécution du programme dans un état indéfini.

Il existe deux stratégies courantes pour empêcher de tels cas de comportement indéfini.

Le premier et préférable est simple - faire en sorte que p lui-même cesse d’exister lorsqu’il n’est plus nécessaire, par exemple :

if (something_is_needed())
{

    int *p = malloc(10 * sizeof *p);
    if (p == NULL) 
    {
        perror("malloc failed");
        return -1;
    }

    /* do whatever is needed with p */

    free(p);
}

En appelant free() directement avant la fin du bloc conteneur (c’est-à-dire le }), p lui-même cesse d’exister. Le compilateur donnera une erreur de compilation à toute tentative d’utilisation de p après cela.

Une deuxième approche consiste également à invalider le pointeur lui-même après avoir libéré la mémoire vers laquelle il pointe :

free(p);
p = NULL;     // you may also use 0 instead of NULL

Arguments pour cette approche :

  • Sur de nombreuses plates-formes, une tentative de déréférencement d’un pointeur nul provoquera un crash instantané : erreur de segmentation. Ici, nous obtenons au moins une trace de pile pointant vers la variable qui a été utilisée après avoir été libérée.

Sans définir le pointeur sur NULL, nous avons un pointeur suspendu. Le programme plantera très probablement encore, mais plus tard, car la mémoire vers laquelle pointe le pointeur sera silencieusement corrompue. De tels bogues sont difficiles à tracer car ils peuvent entraîner une pile d’appels sans rapport avec le problème initial.

Cette approche suit donc le [concept fail-fast][3].

  • Il est sûr de libérer un pointeur nul. Le [C Standard spécifie][4] que free(NULL) n’a aucun effet :

La fonction free fait que l’espace pointé par ptr est désalloué, c’est-à-dire rendu disponible pour une allocation ultérieure. Si ptr est un pointeur nul, aucune action ne se produit. Sinon, si l’argument ne correspond pas à un pointeur précédemment renvoyé par la fonction calloc, malloc ou realloc, ou si l’espace a été libéré par un appel à free ou realloc, le comportement est indéfini.

  • Parfois, la première approche ne peut pas être utilisée (par exemple, la mémoire est allouée dans une fonction et libérée beaucoup plus tard dans une fonction complètement différente)

[1] : http://linux.die.net/man/3/free [2] : https://en.wikipedia.org/wiki/Dangling_pointer [3] : https://en.wikipedia.org/wiki/Fail-fast [4] : http://port70.net/~nsz/c/c89/c89-draft.html#4.10.3.2 [5] : https://www.wikiod.com/fr/c/pointeurs#Erreurs courantes

Allocation de mémoire #

Allocation standard #

Les fonctions d’allocation de mémoire dynamique C sont définies dans l’en-tête <stdlib.h>. Si l’on souhaite allouer dynamiquement de l’espace mémoire pour un objet, le code suivant peut être utilisé :

int *p = malloc(10 * sizeof *p);
if (p == NULL) 
{
    perror("malloc() failed");
    return -1;
}

Cela calcule le nombre d’octets que dix int occupent en mémoire, puis demande ce nombre d’octets à malloc et attribue le résultat (c’est-à-dire l’adresse de départ du morceau de mémoire qui vient d’être créé à l’aide de malloc) à un pointeur nommé ‘p’.

Il est recommandé d’utiliser sizeof pour calculer la quantité de mémoire à demander puisque le résultat de sizeof est défini par l’implémentation (sauf pour les types de caractères, qui sont char, signed char et unsigned char , pour lequel sizeof est défini pour donner toujours 1).

Parce que malloc peut ne pas être en mesure de traiter la requête, il peut renvoyer un pointeur nul. Il est important de vérifier cela pour éviter les tentatives ultérieures de déréférencement du pointeur nul.

La mémoire allouée dynamiquement à l’aide de malloc() peut être redimensionnée à l’aide de realloc() ou, lorsqu’elle n’est plus nécessaire, libérée à l’aide de free().

Alternativement, déclarer int array[10]; allouerait la même quantité de mémoire. Cependant, s’il est déclaré dans une fonction sans le mot clé static, il ne sera utilisable que dans la fonction dans laquelle il est déclaré et les fonctions qu’il appelle (car le tableau sera alloué sur la pile et l’espace sera libéré pour réutiliser lorsque la fonction revient). Alternativement, s’il est défini avec static à l’intérieur d’une fonction, ou s’il est défini en dehors de toute fonction, alors sa durée de vie est la durée de vie du programme. Les pointeurs peuvent également être renvoyés par une fonction, mais une fonction en C ne peut pas renvoyer un tableau.

Mémoire mise à zéro #

La mémoire renvoyée par malloc peut ne pas être initialisée à une valeur raisonnable, et il faut veiller à mettre la mémoire à zéro avec memset ou à y copier immédiatement une valeur appropriée. Alternativement, ‘calloc’ renvoie un bloc de la taille souhaitée où tous les bits sont initialisés à ‘0’. Cela n’a pas besoin d’être identique à la représentation du zéro à virgule flottante ou d’une constante de pointeur nul.

int *p = calloc(10, sizeof *p);
if (p == NULL) 
{
    perror("calloc() failed");
    return -1;
}

Une note sur calloc : La plupart des implémentations (couramment utilisées) optimiseront calloc() pour les performances, donc ce sera [plus rapide][1] que d’appeler malloc(), puis memset( ), même si l’effet net est identique.

Mémoire alignée #

C11 a introduit une nouvelle fonction aligned_alloc() qui alloue de l’espace avec l’alignement donné. Il peut être utilisé si la mémoire à allouer doit être alignée sur certaines limites qui ne peuvent pas être satisfaites par malloc() ou calloc(). Les fonctions malloc() et calloc() allouent de la mémoire correctement alignée pour tout type d’objet (c’est-à-dire que l’alignement est alignof(max_align_t)). Mais avec aligned_alloc() des alignements plus importants peuvent être demandés.

/* Allocates 1024 bytes with 256 bytes alignment. */
char *ptr = aligned_alloc(256, 1024);
if (ptr) {
    perror("aligned_alloc()");
    return -1;
}
free(ptr);

La norme C11 impose deux restrictions : 1) la taille (deuxième argument) demandée doit être un multiple entier de l’alignement (premier argument) et 2) la valeur de alignement doit être un alignement valide pris en charge par l’implémentation . Le non-respect de l’un ou l’autre entraîne [un comportement indéfini] [2].

[1] : http://stackoverflow.com/a/2688522/130580 [2] : https://www.wikiod.com/fr/c/comportement-indefini

Réallouer de la mémoire #

Vous devrez peut-être étendre ou réduire l’espace de stockage de votre pointeur après lui avoir alloué de la mémoire. La fonction void *realloc(void *ptr, size_t size) libère l’ancien objet pointé par ptr et renvoie un pointeur vers un objet qui a la taille spécifiée par size. ptr est le pointeur vers un bloc de mémoire précédemment alloué avec malloc, calloc ou realloc (ou un pointeur nul) à réallouer. Le contenu maximal possible de la mémoire d’origine est conservé. Si la nouvelle taille est supérieure, toute mémoire supplémentaire au-delà de l’ancienne taille n’est pas initialisée. Si la nouvelle taille est plus courte, le contenu de la partie réduite est perdu. Si ptr est NULL, un nouveau bloc est alloué et un pointeur vers celui-ci est renvoyé par la fonction.

#include <stdio.h>
#include <stdlib.h>
 
int main(void)
{
    int *p = malloc(10 * sizeof *p);
    if (NULL == p) 
    {
        perror("malloc() failed");
        return EXIT_FAILURE;
    }
 
    p[0] = 42;
    p[9] = 15;

    /* Reallocate array to a larger size, storing the result into a
     * temporary pointer in case realloc() fails. */
    {
        int *temporary = realloc(p, 1000000 * sizeof *temporary);

        /* realloc() failed, the original allocation was not free'd yet. */
        if (NULL == temporary)
        {
            perror("realloc() failed");
            free(p); /* Clean up. */
            return EXIT_FAILURE;
        }      

        p = temporary;
    }

    /* From here on, array can be used with the new size it was 
     * realloc'ed to, until it is free'd. */

    /* The values of p[0] to p[9] are preserved, so this will print:
       42 15
    */
    printf("%d %d\n", p[0], p[9]);

    free(p);

    return EXIT_SUCCESS;
}

L’objet réalloué peut ou non avoir la même adresse que *p. Par conséquent, il est important de capturer la valeur de retour de realloc qui contient la nouvelle adresse si l’appel réussit.

Assurez-vous d’assigner la valeur de retour de realloc à un temporary au lieu du p d’origine. realloc renverra null en cas d’échec, ce qui écraserait le pointeur. Cela entraînerait la perte de vos données et créerait une fuite de mémoire.

realloc(ptr, 0) n’est pas équivalent à free(ptr) #

realloc est conceptuellement équivalent à malloc + memcpy + free sur l’autre pointeur.

Si la taille de l’espace demandé est zéro, le comportement de realloc est défini par l’implémentation. Ceci est similaire pour toutes les fonctions d’allocation de mémoire qui reçoivent un paramètre size de valeur 0. De telles fonctions peuvent en fait renvoyer un pointeur non nul, mais celui-ci ne doit jamais être déréférencé.

Ainsi, realloc(ptr,0) n’est pas équivalent à free(ptr). Cela pourrait

  • être une implémentation “paresseuse” et renvoyer simplement ptr
  • free(ptr), alloue un élément factice et renvoie celui-ci
  • free(ptr) et renvoie 0
  • retournez simplement ‘0’ en cas d’échec et ne faites rien d’autre.

Ainsi, en particulier, les deux derniers cas ne peuvent être distingués par le code d’application.

Cela signifie que realloc(ptr,0) peut ne pas vraiment libérer/libérer la mémoire, et donc il ne devrait jamais être utilisé en remplacement de free.

Tableaux multidimensionnels de taille variable #

Depuis C99, C a des tableaux de longueur variable, VLA, qui modélisent des tableaux avec des bornes qui ne sont connues qu’au moment de l’initialisation. Bien que vous deviez faire attention à ne pas allouer de VLA trop volumineux (ils pourraient écraser votre pile), utiliser des pointeurs vers VLA et les utiliser dans les expressions sizeof est acceptable.

double sumAll(size_t n, size_t m, double A[n][m]) {
    double ret = 0.0;
    for (size_t i = 0; i < n; ++i)
       for (size_t j = 0; j < m; ++j)
          ret += A[i][j]
    return ret;
}

int main(int argc, char *argv[argc+1]) {
   size_t n = argc*10;
   size_t m = argc*8;
   double (*matrix)[m] = malloc(sizeof(double[n][m]));
   // initialize matrix somehow
   double res = sumAll(n, m, matrix);
   printf("result is %g\n", res);
   free(matrix);
}

Ici, matrix est un pointeur vers des éléments de type double[m], et l’expression sizeof avec double[n][m] garantit qu’il contient de l’espace pour n de tels éléments.

Tout cet espace est alloué de manière contiguë et peut donc être désalloué par un simple appel à ‘free’.

La présence de VLA dans le langage affecte également les déclarations possibles de tableaux et de pointeurs dans les en-têtes de fonction. Désormais, une expression entière générale est autorisée dans le [] des paramètres du tableau. Pour les deux fonctions, les expressions dans [] utilisent des paramètres qui ont été déclarés auparavant dans la liste des paramètres. Pour sumAll, ce sont les longueurs que le code utilisateur attend pour la matrice. Comme pour tous les paramètres de fonction de tableau en C, la dimension la plus interne est réécrite en un type pointeur, donc cela équivaut à la déclaration

  double sumAll(size_t n, size_t m, double (*A)[m]);

Autrement dit, n ne fait pas vraiment partie de l’interface de la fonction, mais les informations peuvent être utiles pour la documentation et pourraient également être utilisées par les compilateurs de vérification des limites pour avertir des accès hors limites.

De même, pour main, l’expression argc+1 est la longueur minimale que le standard C prescrit pour l’argument argv.

Notez qu’officiellement la prise en charge de VLA est facultative dans C11, mais nous ne connaissons aucun compilateur qui implémente C11 et qui ne les ait pas. Vous pouvez tester avec la macro __STDC_NO_VLA__ si vous le devez.

alloca : alloue de la mémoire sur la pile #

Mise en garde : alloca n’est mentionné ici que par souci d’exhaustivité. Il est entièrement non portable (il n’est couvert par aucune des normes communes) et possède un certain nombre de caractéristiques potentiellement dangereuses qui le rendent dangereux pour les non avertis. Le code C moderne devrait le remplacer par Variable Length Arrays (VLA).

[Page du manuel][1]

#include <alloca.h>
// glibc version of stdlib.h include alloca.h by default

void foo(int size) {
    char *data = alloca(size);
    /*
      function body;
    */
    // data is automatically freed
}

Allouez de la mémoire sur le cadre de pile de l’appelant, l’espace référencé par le pointeur renvoyé est automatiquement [libre] [2] à la fin de la fonction appelante.

Bien que cette fonction soit pratique pour la gestion automatique de la mémoire, sachez que demander une allocation importante peut provoquer un débordement de pile et que vous ne pouvez pas utiliser [free][2] avec de la mémoire allouée avec [alloca][1] (ce qui pourrait causer plus de problème avec le débordement de pile).

Pour ces raisons, il n’est pas recommandé d’utiliser [alloca][1] à l’intérieur d’une boucle ni d’une fonction récursive.

Et comme la mémoire est [free][2] au retour de la fonction, vous ne pouvez pas renvoyer le pointeur en tant que résultat de la fonction ([le comportement serait indéfini][5]).

Sommaire #

  • appel identique à [malloc][3]
  • automatiquement libéré au retour de la fonction
  • incompatible avec les fonctions [free][2],[realloc][4] ([comportement indéfini][5])
  • le pointeur ne peut pas être retourné en tant que résultat de fonction ([comportement indéfini][5])
  • taille d’allocation limitée par l’espace de pile, qui (sur la plupart des machines) est beaucoup plus petit que l’espace de tas disponible pour être utilisé par malloc()
  • évitez d’utiliser alloca() et les VLA (tableaux de longueur variable) dans une seule fonction
  • alloca() n’est pas aussi portable que malloc() et al

Recommandation #

  • Ne pas utiliser alloca() dans le nouveau code

Alternative moderne.

void foo(int size) {
    char data[size];
    /*
      function body;
    */
    // data is automatically freed
}

Cela fonctionne là où alloca() fonctionne, et fonctionne là où alloca() ne fonctionne pas (à l’intérieur des boucles, par exemple). Il suppose une implémentation C99 ou une implémentation C11 qui ne définit pas __STDC_NO_VLA__.

Gestion de la mémoire définie par l’utilisateur #

malloc() appelle souvent les fonctions sous-jacentes du système d’exploitation pour obtenir des pages de mémoire. Mais il n’y a rien de spécial à propos de la fonction et elle peut être implémentée en C pur en déclarant un grand tableau statique et en l’allouant (il y a une légère difficulté à assurer un alignement correct, en pratique l’alignement sur 8 octets est presque toujours adéquat).

Pour implémenter un schéma simple, un bloc de contrôle est stocké dans la région de mémoire immédiatement avant le pointeur à renvoyer à partir de l’appel. Cela signifie que free() peut être implémenté en soustrayant du pointeur renvoyé et en lisant les informations de contrôle, qui sont généralement la taille du bloc plus des informations permettant de le remettre dans la liste libre - une liste liée de non alloué blocs.

Lorsque l’utilisateur demande une allocation, la liste libre est recherchée jusqu’à ce qu’un bloc de taille identique ou supérieure au montant demandé soit trouvé, puis si nécessaire il est fractionné. Cela peut conduire à une fragmentation de la mémoire si l’utilisateur fait continuellement de nombreuses allocations et libérations de taille imprévisible et à des intervalles imprévisibles (tous les vrais programmes ne se comportent pas comme ça, le schéma simple est souvent suffisant pour les petits programmes).

/* typical control block */
struct block
{
   size_t size;         /* size of block */
   struct block *next;  /* next block in free list */ 
   struct block *prev;  /* back pointer to previous block in memory */
   void *padding;       /* need 16 bytes to make multiple of 8 */
}

static struct block arena[10000]; /* allocate from here */
static struct block *firstfree;

De nombreux programmes nécessitent un grand nombre d’allocations de petits objets de même taille. Ceci est très facile à mettre en œuvre. Utilisez simplement un bloc avec un pointeur suivant. Donc si un bloc de 32 octets est requis :

union block
{
   union block * next;
   unsigned char payload[32];
}  

static union block arena[100];
static union block * head; 
void init(void)
{
    int i;
    for (i = 0; i < 100 - 1; i++)
        arena[i].next = &arena[i + 1];
    arena[i].next = 0; /* last one, null */
    head = &block[0];
}
 
void *block_alloc()
{
    void *answer = head;
    if (answer)
        head = head->next;
    return answer;
}

void block_free(void *ptr)
{
    union block *block = ptr;
    block->next = head;
    head - block;
}

Ce schéma est extrêmement rapide et efficace, et peut être rendu générique avec une certaine perte de clarté.
