Mélanger des entiers signés et non signés dans les opérations arithmétiques #

Ce n’est généralement pas une bonne idée de mélanger des entiers “signés” et “non signés” dans les opérations arithmétiques. Par exemple, quel sera le résultat de l’exemple suivant ?

#include <stdio.h>

int main(void)
{ 
    unsigned int a = 1000;
    signed int b = -1;

    if (a > b) puts("a is more than b");
    else puts("a is less or equal than b"); 

    return 0;
}

Puisque 1000 est supérieur à -1, vous vous attendriez à ce que la sortie soit “a est supérieur à b”, mais ce ne sera pas le cas.

Les opérations arithmétiques entre différents types intégraux sont effectuées au sein d’un type commun défini par les conversions arithmétiques dites usuelles (voir la spécification du langage, 6.3.1.8).

Dans ce cas, le “type commun” est “unsigned int”, car, comme indiqué dans [Usual arithmetic conversions][1],

714  Sinon, si l’opérande de type entier non signé a rang supérieur ou égal au rang du type de l’autre opérande, alors l’opérande de type entier signé est converti en type de l’opérande de type entier non signé.

Cela signifie que l’opérande int b sera converti en unsigned int avant la comparaison.

Lorsque -1 est converti en un unsigned int, le résultat est la valeur maximale possible de unsigned int, qui est supérieure à 1000, ce qui signifie que a > b est faux.

[1] : http://c0x.coding-guidelines.com/6.3.1.8.html

Les macros sont de simples remplacements de chaînes #

Les macros sont de simples remplacements de chaînes. (À proprement parler, ils fonctionnent avec des jetons de prétraitement, et non avec des chaînes arbitraires.)

#include <stdio.h>

#define SQUARE(x) x*x

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Vous pouvez vous attendre à ce que ce code imprime 9 (3*3), mais en fait 5 sera imprimé car la macro sera étendue à 1+2*1+2.

Vous devez mettre les arguments et l’ensemble de l’expression de la macro entre parenthèses pour éviter ce problème.

#include <stdio.h>

#define SQUARE(x) ((x)*(x))

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Un autre problème est qu’il n’est pas garanti que les arguments d’une macro soient évalués une seule fois ; ils peuvent ne pas être évalués du tout ou peuvent être évalués plusieurs fois.

#include <stdio.h>

#define MIN(x, y) ((x) <= (y) ? (x) : (y))

int main(void) {
    int a = 0;
    printf("%d\n", MIN(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

Dans ce code, la macro sera étendue à ((a++) <= (10) ? (a++) : (10)). Puisque a++ (0) est plus petit que 10, a++ sera évalué deux fois et cela rendra la valeur de a et ce qui est renvoyé par MIN différents de ce à quoi vous pourriez vous attendre.

Cela peut être évité en utilisant des fonctions, mais notez que les types seront fixés par la définition de la fonction, alors que les macros peuvent être (trop) flexibles avec les types.

#include <stdio.h>

int min(int x, int y) {
    return x <= y ? x : y;
}

int main(void) {
    int a = 0;
    printf("%d\n", min(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

Maintenant, le problème de la double évaluation est résolu, mais cette fonction min ne peut pas traiter les données double sans tronquer, par exemple.

Les macro-directives peuvent être de deux types :

#define OBJECT_LIKE_MACRO     followed by a "replacement list" of preprocessor tokens
#define FUNCTION_LIKE_MACRO(with, arguments) followed by a replacement list

Ce qui distingue ces deux types de macros est le caractère qui suit l’identifiant après #define : s’il s’agit d’un lparen, il s’agit d’une macro de type fonction ; sinon, c’est une macro de type objet. Si l’intention est d’écrire une macro de type fonction, il ne doit pas y avoir d’espace blanc entre la fin du nom de la macro et (. Consultez [this][1] pour une explication détaillée.

Dans C99 ou version ultérieure, vous pouvez utiliser static inline int min(int x, int y) { … }.

En C11, vous pouvez écrire une expression « générique de type » pour « min ».

#include <stdio.h>

#define min(x, y) _Generic((x), \
                        long double: min_ld, \
                        unsigned long long: min_ull, \
                        default: min_i \
                        )(x, y)

#define gen_min(suffix, type) \
    static inline type min_##suffix(type x, type y) { return (x < y) ? x : y; }

gen_min(ld, long double)
gen_min(ull, unsigned long long)
gen_min(i, int)

int main(void)
{
    unsigned long long ull1 = 50ULL;
    unsigned long long ull2 = 37ULL;
    printf("min(%llu, %llu) = %llu\n", ull1, ull2, min(ull1, ull2));
    long double ld1 = 3.141592653L;
    long double ld2 = 3.141592652L;
    printf("min(%.10Lf, %.10Lf) = %.10Lf\n", ld1, ld2, min(ld1, ld2));
    int i1 = 3141653;
    int i2 = 3141652;
    printf("min(%d, %d) = %d\n", i1, i2, min(i1, i2));
    return 0;
}

L’expression générique pourrait être étendue avec plus de types tels que double, float, long long, unsigned long, long, unsigned — et des invocations de macro gen_min appropriées écrites.

[1] : https://stackoverflow.com/questions/39931709/why-this-macros-is-not-expanding-correctly

Oublier de copier la valeur de retour de realloc dans un fichier temporaire #

Si realloc échoue, il renvoie NULL. Si vous affectez la valeur du tampon d’origine à la valeur de retour de realloc, et s’il renvoie NULL, alors le tampon d’origine (l’ancien pointeur) est perdu, ce qui entraîne une [fuite de mémoire][1] . La solution consiste à copier dans un pointeur temporaire, et si ce pointeur temporaire n’est pas NULL, alors copiez dans le vrai tampon.

char *buf, *tmp;

buf = malloc(...);
...

/* WRONG */
if ((buf = realloc(buf, 16)) == NULL)
    perror("realloc");

/* RIGHT */
if ((tmp = realloc(buf, 16)) != NULL)
    buf = tmp;
else
    perror("realloc");

[1] : https://en.wikipedia.org/wiki/Memory_leak

Oublier d’allouer un octet supplémentaire pour \0 #

Lorsque vous copiez une chaîne dans un tampon malloc, n’oubliez pas d’ajouter 1 à strlen.

char *dest = malloc(strlen(src)); /* WRONG */
char *dest = malloc(strlen(src) + 1); /* RIGHT */

strcpy(dest, src);

C’est parce que strlen n’inclut pas le \0 final dans la longueur. Si vous adoptez l’approche WRONG (comme indiqué ci-dessus), lors de l’appel de strcpy, votre programme invoquera un comportement indéfini.

Cela s’applique également aux situations où vous lisez une chaîne de longueur maximale connue à partir de stdin ou d’une autre source. Par exemple

#define MAX_INPUT_LEN 42

char buffer[MAX_INPUT_LEN]; /* WRONG */
char buffer[MAX_INPUT_LEN + 1]; /* RIGHT */

scanf("%42s", buffer);  /* Ensure that the buffer is not overflowed */

Décroissance du tableau incompréhensible #

Un problème courant dans le code qui utilise des tableaux multidimensionnels, des tableaux de pointeurs, etc. est le fait que Type** et Type[M][N] sont des types fondamentalement différents :

#include <stdio.h>

void print_strings(char **strings, size_t n)
{
    size_t i;
    for (i = 0; i < n; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(s, 4);
    return 0;
}

Exemple de sortie du compilateur :

file1.c: In function 'main':
file1.c:13:23: error: passing argument 1 of 'print_strings' from incompatible pointer type [-Wincompatible-pointer-types]
         print_strings(strings, 4);
                       ^
file1.c:3:10: note: expected 'char **' but argument is of type 'char (*)[20]'
     void print_strings(char **strings, size_t n)

L’erreur indique que le tableau s dans la fonction main est passé à la fonction print_strings, qui attend un type de pointeur différent de celui qu’il a reçu. Il inclut également une note exprimant le type attendu par print_strings et le type qui lui a été transmis par main.

Le problème est dû à quelque chose appelé array decay. Que se passe-t-il lorsque s avec son type char[4][20] (tableau de 4 tableaux de 20 caractères) est passé à la fonction, c’est qu’il se transforme en un pointeur vers son premier élément comme si vous aviez écrit &s[ 0], qui a le type char (*)[20] (pointeur vers 1 tableau de 20 caractères). Cela se produit pour n’importe quel tableau, y compris un tableau de pointeurs, un tableau de tableaux de tableaux (tableaux 3D) et un tableau de pointeurs vers un tableau. Vous trouverez ci-dessous un tableau illustrant ce qui se passe lorsqu’un tableau se désintègre. Les modifications apportées à la description du type sont mises en surbrillance pour illustrer ce qui se passe :

Avant la décomposition   Après la décomposition  
car [20] tableau de (20 caractères) car * pointeur vers (1 caractère)
car [4][20] tableau de (4 tableaux de 20 caractères) car (*)[20] pointeur vers (1 tableau de 20 caractères)
car *[4] tableau de (4 pointeurs vers 1 caractère) car ** pointeur vers (1 pointeur vers 1 caractère)
car [3][4][20] tableau de (3 tableaux de 4 tableaux de 20 caractères) car (*)[4][20] pointeur vers (1 tableau de 4 tableaux de 20 caractères)
car (*[4])[20] tableau de (4 pointeurs vers 1 tableau de 20 caractères) car (**)[20] pointeur vers (1 pointeur vers 1 tableau de 20 caractères)

Si un tableau peut se désintégrer en un pointeur, on peut dire qu’un pointeur peut être considéré comme un tableau d’au moins 1 élément. Une exception à cela est un pointeur nul, qui ne pointe sur rien et n’est donc pas un tableau.

La désintégration du tableau ne se produit qu’une seule fois. Si un tableau s’est désintégré en un pointeur, il s’agit maintenant d’un pointeur et non d’un tableau. Même si vous avez un pointeur vers un tableau, rappelez-vous que le pointeur peut être considéré comme un tableau d’au moins un élément, donc la désintégration du tableau s’est déjà produite.

En d’autres termes, un pointeur vers un tableau (char (*)[20]) ne deviendra jamais un pointeur vers un pointeur (char **). Pour corriger la fonction print_strings, faites-lui simplement recevoir le type correct :

void print_strings(char (*strings)[20], size_t n)
/* OR */
void print_strings(char strings[][20], size_t n)

Un problème survient lorsque vous souhaitez que la fonction print_strings soit générique pour n’importe quel tableau de caractères : que se passe-t-il s’il y a 30 caractères au lieu de 20 ? Ou 50 ? La réponse est d’ajouter un autre paramètre avant le paramètre array :

#include <stdio.h>

/*
 * Note the rearranged parameters and the change in the parameter name
 * from the previous definitions:
 *      n (number of strings)
 *   => scount (string count)
 *
 * Of course, you could also use one of the following highly recommended forms
 * for the `strings` parameter instead:
 *
 *    char strings[scount][ccount]
 *    char strings[][ccount]
 */
void print_strings(size_t scount, size_t ccount, char (*strings)[ccount])
{
    size_t i;
    for (i = 0; i < scount; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(4, 20, s);
    return 0;
}

La compilation ne produit aucune erreur et donne le résultat attendu :

Example 1
Example 2
Example 3
Example 4

Oublier de libérer de la mémoire (fuites de mémoire) #

Une bonne pratique de programmation consiste à libérer toute mémoire qui a été allouée directement par votre propre code, ou implicitement en appelant une fonction interne ou externe, telle qu’une API de bibliothèque comme [strdup()][1]. Ne pas libérer de mémoire peut introduire une fuite de mémoire, qui pourrait s’accumuler dans une quantité substantielle de mémoire gaspillée qui n’est pas disponible pour votre programme (ou le système), entraînant éventuellement des plantages ou un comportement indéfini. Les problèmes sont plus susceptibles de se produire si la fuite se produit à plusieurs reprises dans une boucle ou une fonction récursive. Le risque d’échec du programme augmente avec la durée d’exécution d’un programme qui fuit. Parfois, les problèmes apparaissent instantanément ; d’autres fois, les problèmes ne seront pas visibles pendant des heures, voire des années de fonctionnement constant. Les échecs d’épuisement de la mémoire peuvent être catastrophiques, selon les circonstances.

La boucle infinie suivante est un exemple de fuite qui finira par épuiser la mémoire disponible en appelant getline(), une fonction qui alloue implicitement de la nouvelle mémoire, sans libérer cette mémoire.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    /* The loop below leaks memory as fast as it can */

    for(;;) { 
        getline(&line, &size, stdin); /* New memory implicitly allocated */

        /* <do whatever> */

        line = NULL;
    }

    return 0;
}

En revanche, le code ci-dessous utilise également la fonction getline(), mais cette fois, la mémoire allouée est correctement libérée, évitant une fuite.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    for(;;) {
        if (getline(&line, &size, stdin) < 0) {
            free(line);
            line = NULL;

            /* Handle failure such as setting flag, breaking out of loop and/or exiting */
        }

        /* <do whatever> */

        free(line);
        line = NULL;

    }

    return 0;
}

Une fuite de mémoire n’a pas toujours de conséquences tangibles et n’est pas nécessairement un problème fonctionnel. Bien que les “meilleures pratiques” dictent de libérer rigoureusement la mémoire à des points et conditions stratégiques, afin de réduire l’empreinte mémoire et le risque d’épuisement de la mémoire, il peut y avoir des exceptions. Par exemple, si un programme est limité en durée et en portée, le risque d’échec de l’allocation peut être considéré comme trop faible pour s’en préoccuper. Dans ce cas, le contournement de la désallocation explicite peut être considéré comme acceptable. Par exemple, la plupart des systèmes d’exploitation modernes libèrent automatiquement toute la mémoire consommée par un programme lorsqu’il se termine, que ce soit en raison d’un échec du programme, d’un appel système à exit(), de l’arrêt du processus ou de la fin de main(). La libération explicite de la mémoire au moment de la fin imminente du programme pourrait en fait être redondante ou introduire une baisse des performances.

L’allocation peut échouer si la mémoire disponible est insuffisante, et les échecs de traitement doivent être pris en compte aux niveaux appropriés de la pile des appels. getline(), illustré ci-dessus est un cas d’utilisation intéressant car il s’agit d’une fonction de bibliothèque qui non seulement alloue de la mémoire qu’elle laisse à l’appelant pour la libérer, mais peut échouer pour un certain nombre de raisons, qui doivent toutes être prises en compte . Par conséquent, il est essentiel, lors de l’utilisation d’une API C, de lire la [documentation (man page)][2] et de prêter une attention particulière aux conditions d’erreur et à l’utilisation de la mémoire, et de savoir quelle couche logicielle supporte la charge de libérer la mémoire retournée .

Une autre pratique courante de gestion de la mémoire consiste à définir systématiquement les pointeurs de mémoire sur NULL immédiatement après la libération de la mémoire référencée par ces pointeurs, de sorte que la validité de ces pointeurs puisse être testée à tout moment (par exemple, vérifiée pour NULL / non-NULL), car l’accès à la mémoire libérée peut entraîner de graves problèmes tels que l’obtention de données inutiles (opération de lecture), ou la corruption de données (opération d’écriture) et/ou un plantage du programme. Dans la plupart des systèmes d’exploitation modernes, la libération de l’emplacement mémoire 0 (“NULL”) est un NOP (par exemple, il est inoffensif), comme l’exige la norme C - donc en définissant un pointeur sur NULL, il n’y a aucun risque de double-libération de la mémoire si le pointeur est passé à free(). Gardez à l’esprit que la double libération de mémoire peut entraîner des pannes très longues, déroutantes et * difficiles à diagnostiquer *.

Copier trop #

char buf[8]; /* tiny buffer, easy to overflow */

printf("What is your name?\n");
scanf("%s", buf); /* WRONG */
scanf("%7s", buf); /* RIGHT */

Si l’utilisateur entre une chaîne de plus de 7 caractères (- 1 pour le terminateur nul), la mémoire derrière le tampon buf sera écrasée. Il en résulte un comportement indéfini. Les pirates malveillants exploitent souvent cela pour écraser l’adresse de retour et la remplacer par l’adresse du code malveillant du pirate.

Écrire par erreur = au lieu de == lors de la comparaison #

L’opérateur = est utilisé pour l’affectation.

L’opérateur == est utilisé pour la comparaison.

Il faut faire attention à ne pas mélanger les deux. Parfois on écrit à tort

/* assign y to x */
if (x = y) {
     /* logic */
}

alors que ce qu’on voulait vraiment c’est :

/* compare if x is equal to y */
if (x == y) {
    /* logic */
}

Le premier attribue la valeur de y à x et vérifie si cette valeur est non nulle, au lieu de faire une comparaison, ce qui équivaut à :

if ((x = y) != 0) {
    /* logic */
}

Il y a des moments où tester le résultat d’une affectation est prévu et est couramment utilisé, car cela évite d’avoir à dupliquer du code et d’avoir à traiter la première fois spécialement. Comparer

while ((c = getopt_long(argc, argv, short_options, long_options, &option_index)) != -1) {
        switch (c) {
        ...
        }
}

contre

c = getopt_long(argc, argv, short_options, long_options, &option_index);
while (c != -1) {
        switch (c) {
        ...
        }
        c = getopt_long(argc, argv, short_options, long_options, &option_index);
}

Les compilateurs modernes reconnaîtront ce modèle et n’avertiront pas lorsque l’affectation est entre parenthèses comme ci-dessus, mais peuvent avertir pour d’autres utilisations. Par exemple:

if (x = y)         /* warning */

if ((x = y))       /* no warning */
if ((x = y) != 0)  /* no warning; explicit */

Certains programmeurs utilisent la stratégie consistant à placer la constante à gauche de l’opérateur (communément appelée [conditions Yoda][1]). Étant donné que les constantes sont des rvalues, ce style de condition entraînera une erreur du compilateur si le mauvais opérateur a été utilisé.

if (5 = y) /* Error */

if (5 == y) /* No error */

Cependant, cela réduit considérablement la lisibilité du code et n’est pas considéré comme nécessaire si le programmeur suit de bonnes pratiques de codage C, et n’aide pas lors de la comparaison de deux variables, ce n’est donc pas une solution universelle. De plus, de nombreux compilateurs modernes peuvent donner des avertissements lorsque le code est écrit avec des conditions Yoda.

Utilisation imprudente des points-virgules #

Soyez prudent avec les points-virgules. Exemple suivant

if (x > a);
   a = x;

signifie en fait :

if (x > a) {}
a = x;

ce qui signifie que “x” sera affecté à “a” dans tous les cas, ce qui n’est peut-être pas ce que vous vouliez à l’origine.

Parfois, l’absence d’un point-virgule entraînera également un problème imperceptible :

if (i < 0) 
    return
day = date[0];
hour = date[1];
minute = date[2];

Le point-virgule derrière return est manqué, donc day=date[0] sera renvoyé.

Une technique pour éviter cela et des problèmes similaires consiste à toujours utiliser des accolades sur les conditions et les boucles multilignes. Par exemple:

if (x > a) {
    a = x;
}

Le caractère de nouvelle ligne n’est pas consommé dans l’appel scanf() typique #

Lorsque ce programme

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char str[128], *lf;

    scanf("%d", &num);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

est exécuté avec cette entrée

42
life

la sortie sera 42 "" au lieu de 42 ​​"life" attendu.

C’est parce qu’un caractère de retour à la ligne après 42 n’est pas consommé dans l’appel de scanf() et il est consommé par fgets() avant de lire life. Ensuite, fgets() arrête de lire avant de lire life.

Pour éviter ce problème, une méthode utile lorsque la longueur maximale d’une ligne est connue – lors de la résolution de problèmes dans un système de juge en ligne, par exemple – consiste à éviter d’utiliser scanf() directement et à lire toutes les lignes via fgets( ). Vous pouvez utiliser sscanf() pour analyser les lignes lues.

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char line_buffer[128] = "", str[128], *lf;

    fgets(line_buffer, sizeof(line_buffer), stdin);
    sscanf(line_buffer, "%d", &num);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

Une autre méthode consiste à lire jusqu’à ce que vous atteigniez un caractère de nouvelle ligne après avoir utilisé scanf() et avant d’utiliser fgets().

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char str[128], *lf;
    int c;

    scanf("%d", &num);
    while ((c = getchar()) != '\n' && c != EOF);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

Ajout d’un point-virgule à un #define #

Il est facile de s’embrouiller dans le préprocesseur C et de le traiter comme faisant partie du C lui-même, mais c’est une erreur car le préprocesseur n’est qu’un mécanisme de substitution de texte. Par exemple, si vous écrivez

/* WRONG */
#define MAX 100;
int arr[MAX];

le code s’étend à

int arr[100;];

qui est une erreur de syntaxe. Le remède consiste à supprimer le point-virgule de la ligne #define. C’est presque toujours une erreur de terminer un #define par un point-virgule.

Erreurs de référence non définies lors de la liaison #

L’une des erreurs les plus courantes lors de la compilation se produit lors de l’étape de liaison. L’erreur ressemble à ceci :

$ gcc undefined_reference.c 
/tmp/ccoXhwF0.o: In function `main':
undefined_reference.c:(.text+0x15): undefined reference to `foo'
collect2: error: ld returned 1 exit status
$

Examinons donc le code qui a généré cette erreur :

int foo(void);

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Nous voyons ici une déclaration de foo (int foo();) mais pas de définition de celui-ci (fonction réelle). Nous avons donc fourni au compilateur l’en-tête de la fonction, mais il n’y avait aucune fonction de ce type définie nulle part, donc l’étape de compilation passe mais l’éditeur de liens se termine avec une erreur “Référence non définie”. Pour corriger cette erreur dans notre petit programme, nous n’aurions qu’à ajouter une definition pour foo :

/* Declaration of foo */
int foo(void);

/* Definition of foo */
int foo(void)
{
    return 5;
}

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Maintenant, ce code va compiler. Une situation alternative se présente où la source de foo() est dans un fichier source séparé foo.c (et il y a un en-tête foo.h pour déclarer foo() qui est inclus à la fois dans foo.c et undefined_reference.c). Ensuite, la solution consiste à lier à la fois le fichier objet de foo.c et undefined_reference.c, ou de compiler les deux fichiers source :

$ gcc -c undefined_reference.c 
$ gcc -c foo.c
$ gcc -o working_program undefined_reference.o foo.o
$

Ou:

$ gcc -o working_program undefined_reference.c foo.c
$

Un cas plus complexe est celui où les bibliothèques sont impliquées, comme dans le code :

#include <stdio.h>
#include <stdlib.h>
#include <math.h>

int main(int argc, char **argv)
{
    double first;
    double second;
    double power;

    if (argc != 3)
    {
        fprintf(stderr, "Usage: %s <denom> <nom>\n", argv[0]);
        return EXIT_FAILURE;
    }

    /* Translate user input to numbers, extra error checking
     * should be done here. */
    first = strtod(argv[1], NULL);
    second = strtod(argv[2], NULL);

    /* Use function pow() from libm - this will cause a linkage 
     * error unless this code is compiled against libm! */
    power = pow(first, second);

    printf("%f to the power of %f = %f\n", first, second, power);

    return EXIT_SUCCESS;
}

Le code est syntaxiquement correct, la déclaration pour pow() existe à partir de #include <math.h>, nous essayons donc de compiler et de lier mais obtenons une erreur comme celle-ci :

$ gcc no_library_in_link.c -o no_library_in_link
/tmp/ccduQQqA.o: In function `main':
no_library_in_link.c:(.text+0x8b): undefined reference to `pow'
collect2: error: ld returned 1 exit status
$

Cela se produit parce que la definition pour pow() n’a pas été trouvée pendant l’étape de liaison. Pour résoudre ce problème, nous devons spécifier que nous voulons établir un lien avec la bibliothèque mathématique appelée libm en spécifiant le drapeau -lm. (Notez qu’il existe des plates-formes telles que macOS où -lm n’est pas nécessaire, mais lorsque vous obtenez la référence non définie, la bibliothèque est nécessaire.)

Nous relançons donc l’étape de compilation, en spécifiant cette fois la bibliothèque (après les fichiers source ou objet) :

$ gcc no_library_in_link.c -lm -o library_in_link_cmd
$ ./library_in_link_cmd 2 4
2.000000 to the power of 4.000000 = 16.000000
$

Et il fonctionne!

Vérification de l’expression logique par rapport à ’true' #

La norme C d’origine n’avait pas de type booléen intrinsèque, donc “bool”, “true” et “false” n’avaient aucune signification inhérente et étaient souvent définis par les programmeurs. En règle générale, “true” serait défini comme 1 et “false” serait défini comme 0.

C99 ajoute le type intégré _Bool et l’en-tête <stdbool.h> qui définit bool (s’étendant à _Bool), false et true. Il vous permet également de redéfinir bool, true et false, mais note qu’il s’agit d’une fonctionnalité obsolète.

Plus important encore, les expressions logiques traitent tout ce qui est évalué à zéro comme faux et toute évaluation non nulle comme vraie. Par exemple:

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == true)  /* Comparison only succeeds if true is 0x80 and bitField has that bit set */
    {
        return true;
    }
    else
    {
        return false;
    }
}

Dans l’exemple ci-dessus, la fonction essaie de vérifier si le bit supérieur est défini et renvoie “true” si c’est le cas. Cependant, en vérifiant explicitement par rapport à true, l’instruction if ne réussira que si (bitfield & 0x80) évalue à tout ce que true est défini comme, qui est généralement 1 et très rarement 0x80. Vérifiez explicitement par rapport au cas que vous attendez :

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == 0x80) /* Explicitly test for the case we expect */
    {
        return true;
    }
    else
    {
        return false;
    }
}

Ou évaluez toute valeur non nulle comme vraie.

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    /* If upper bit is set, result is 0x80 which the if will evaluate as true */
    if (bitField & 0x80)
    {
        return true;
    }
    else
    {
        return false;
    }
}

Faire une mise à l’échelle supplémentaire dans l’arithmétique des pointeurs #

Dans l’arithmétique des pointeurs, l’entier à ajouter ou à soustraire au pointeur n’est pas interprété comme un changement d’adresse mais comme un nombre d’éléments à déplacer.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + sizeof(int) * 2; /* wrong */
    printf("%d %d\n", *ptr, *ptr2);
    return 0;
}

Ce code effectue une mise à l’échelle supplémentaire dans le calcul du pointeur affecté à “ptr2”. Si sizeof(int) est 4, ce qui est typique dans les environnements 32 bits modernes, l’expression signifie “8 éléments après array[0]”, ce qui est hors plage, et elle invoque *undefined behavior *.

Pour que ptr2 pointe sur 2 éléments après array[0], vous devez simplement ajouter 2.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + 2;
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

L’arithmétique de pointeur explicite utilisant des opérateurs additifs peut prêter à confusion, il peut donc être préférable d’utiliser l’indice de tableau.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = &ptr[2];
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

E1[E2] est identique à (*((E1)+(E2))) ([N1570](http://chimera.roma1.infn.it/SP/COMMON/iso-iec-9899- 1990.pdf) 6.5.2.1, paragraphe 2), et &(E1[E2]) est équivalent à ((E1)+(E2)) (N1570 6.5.3.2, note de bas de page 102).

Alternativement, si l’arithmétique de pointeur est préférée, la conversion du pointeur pour adresser un type de données différent peut permettre l’adressage des octets. Attention cependant : [l’endianité] [1] peut devenir un problème, et la conversion en types autres que “pointeur vers caractère” conduit à [des problèmes d’aliasing stricts] [2].

#include <stdio.h>

int main(void) {
    int array[3] = {1,2,3};  // 4 bytes * 3 allocated
    unsigned char *ptr = (unsigned char *) array;  // unsigned chars only take 1 byte
    /*
     * Now any pointer arithmetic on ptr will match
     * bytes in memory.  ptr can be treated like it
     * was declared as: unsigned char ptr[12];
     */

    return 0;
}

Comparer des nombres à virgule flottante #

Les types à virgule flottante (float, double et long double) ne peuvent pas représenter avec précision certains nombres car ils ont une précision finie et représentent les valeurs dans un format binaire. Tout comme nous avons des décimales répétitives en base 10 pour des fractions telles que 1/3, il existe des fractions qui ne peuvent pas non plus être représentées de manière finie en binaire (comme 1/3, mais aussi, plus important encore, 1/10). Ne comparez pas directement les valeurs à virgule flottante ; utilisez plutôt un delta.

#include <float.h> // for DBL_EPSILON and FLT_EPSILON
#include <math.h>  // for fabs()

int main(void)
{
    double a = 0.1; // imprecise: (binary) 0.000110...

    // may be false or true
    if (a + a + a + a + a + a + a + a + a + a == 1.0) {
        printf("10 * 0.1 is indeed 1.0. This is not guaranteed in the general case.\n");
    }

    // Using a small delta value.
    if (fabs(a + a + a + a + a + a + a + a + a + a - 1.0) < 0.000001) {
        // C99 5.2.4.2.2p8 guarantees at least 10 decimal digits
        // of precision for the double type.
        printf("10 * 0.1 is almost 1.0.\n");
    }

    return 0;
}

Un autre exemple:

gcc -O3   -g   -I./inc   -std=c11   -Wall -Wextra -Werror -Wmissing-prototypes -Wstrict-prototypes  -Wold-style-definition       rd11.c -o rd11 -L./lib -lsoq

static inline double rel_diff(double a, double b)
{
    return fabs(a - b) / fmax(fabs(a), fabs(b));
}

int main(void)
{
    double d1 = 3.14159265358979;
    double d2 = 355.0 / 113.0;

    double epsilon = 1.0;
    for (int i = 0; i < 10; i++)
    {
        if (rel_diff(d1, d2) < epsilon)
            printf("%d:%.10f <=> %.10f within tolerance %.10f (rel diff %.4E)\n",
                   i, d1, d2, epsilon, rel_diff(d1, d2));
        else
            printf("%d:%.10f <=> %.10f out of tolerance %.10f (rel diff %.4E)\n",
                   i, d1, d2, epsilon, rel_diff(d1, d2));
        epsilon /= 10.0;
    }
    return 0;
}

Ignorer les valeurs de retour des fonctions de la bibliothèque #

Presque toutes les fonctions de la bibliothèque standard C renvoient quelque chose en cas de succès et autre chose en cas d’erreur. Par exemple, malloc renverra un pointeur vers le bloc de mémoire alloué par la fonction en cas de succès et, si la fonction n’a pas réussi à allouer le bloc de mémoire demandé, un pointeur nul. Vous devez donc toujours vérifier la valeur de retour pour faciliter le débogage.

C’est mauvais:

char* x = malloc(100000000000UL * sizeof *x);
/* more code */
scanf("%s", x); /* This might invoke undefined behaviour and if lucky causes a segmentation violation, unless your system has a lot of memory */

C’est bon:

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char* x = malloc(100000000000UL * sizeof *x);
    if (x == NULL) {
        perror("malloc() failed");
        exit(EXIT_FAILURE);
    }

    if (scanf("%s", x) != 1) {
        fprintf(stderr, "could not read string\n");
        free(x);
        exit(EXIT_FAILURE);
    }

    /* Do stuff with x. */

    /* Clean up. */
    free(x);

    return EXIT_SUCCESS;
}

De cette façon, vous connaissez tout de suite la cause de l’erreur, sinon vous risquez de passer des heures à chercher un bogue au mauvais endroit.

Les littéraux à virgule flottante sont de type double par défaut #

Des précautions doivent être prises lors de l’initialisation des variables de type “float” à des valeurs littérales ou de leur comparaison avec des valeurs littérales, car les littéraux à virgule flottante réguliers comme “0.1” sont de type “double”. Cela peut entraîner des surprises :

#include <stdio.h>
int main() {
    float  n;
    n = 0.1;
    if (n > 0.1) printf("Wierd\n");
    return 0;
}
// Prints "Wierd" when n is float

Ici, ’n’ est initialisé et arrondi à une simple précision, ce qui donne la valeur 0,10000000149011612. Ensuite, “n” est reconverti en double précision pour être comparé au littéral “0,1” (qui équivaut à 0,10000000000000001), ce qui entraîne une incompatibilité.

Outre les erreurs d’arrondi, le mélange de variables “float” avec des littéraux “doubles” entraînera de mauvaises performances sur les plates-formes qui ne prennent pas en charge la double précision.

Utilisation de constantes de caractères au lieu de littéraux de chaîne, et vice versa #

En C, les constantes de caractères et les littéraux de chaîne sont des choses différentes.

Un caractère entouré de guillemets simples comme 'a' est une constante de caractère. Une constante de caractère est un entier dont la valeur est le code de caractère qui représente le caractère. La façon d’interpréter les constantes de caractères avec plusieurs caractères comme 'abc' est définie par l’implémentation.

Zéro ou plusieurs caractères entourés de guillemets comme "abc" est un littéral de chaîne. Un littéral de chaîne est un tableau non modifiable dont les éléments sont de type char. La chaîne entre guillemets doubles et le caractère nul de fin sont le contenu, donc "abc" a 4 éléments ({'a', 'b', 'c', '\0'})

Dans cet exemple, une constante de caractère est utilisée là où un littéral de chaîne doit être utilisé. Cette constante de caractère sera convertie en un pointeur d’une manière définie par l’implémentation et il y a peu de chances que le pointeur converti soit valide, donc cet exemple invoquera un comportement non défini.

#include <stdio.h>

int main(void) {
    const char *hello = 'hello, world'; /* bad */
    puts(hello);
    return 0;
}

Dans cet exemple, un littéral de chaîne est utilisé là où une constante de caractère doit être utilisée. Le pointeur converti à partir du littéral de chaîne sera converti en un entier d’une manière définie par l’implémentation, et il sera converti en “char” d’une manière définie par l’implémentation. (Comment convertir un entier en un type signé qui ne peut pas représenter la valeur à convertir est défini par l’implémentation, et si char est signé est également défini par l’implémentation.) La sortie sera quelque chose de dénué de sens.

#include <stdio.h>

int main(void) {
    char c = "a"; /* bad */
    printf("%c\n", c);
    return 0;
}

Dans presque tous les cas, le compilateur se plaindra de ces confusions. Si ce n’est pas le cas, vous devez utiliser davantage d’options d’avertissement du compilateur ou il est recommandé d’utiliser un meilleur compilateur.

Fonction récursive - manque la condition de base #

Le calcul de la factorielle d’un nombre est un exemple classique de fonction récursive.

** Condition de base manquante : **

#include <stdio.h>

int factorial(int n)
{
       return n * factorial(n - 1);
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Sortie typique : Défaut de segmentation : 11

Le problème avec cette fonction est qu’elle bouclerait à l’infini, provoquant une erreur de segmentation - elle a besoin d’une condition de base pour arrêter la récursivité.

Condition de base déclarée :

#include <stdio.h>

int factorial(int n)
{
    if (n == 1) // Base Condition, very crucial in designing the recursive functions.
    {
       return 1;
    }
    else
    {
       return n * factorial(n - 1);
    }
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Exemple de sortie

Factorial 3 = 6

Cette fonction se terminera dès qu’elle atteindra la condition n est égal à 1 (à condition que la valeur initiale de n soit suffisamment petite - la limite supérieure est 12 lorsque int est une quantité de 32 bits).

Règles à suivre :

  1. Initialisez l’algorithme. Les programmes récursifs ont souvent besoin d’une valeur de départ pour commencer. Ceci est accompli soit en utilisant un paramètre passé à la fonction, soit en fournissant une fonction de passerelle qui n’est pas récursive mais qui définit les valeurs de départ pour le calcul récursif.
  2. Vérifiez si la ou les valeurs en cours de traitement correspondent au cas de base. Si tel est le cas, traitez et renvoyez la valeur.
  3. Redéfinir la réponse en termes de sous-problème ou de sous-problèmes plus petits ou plus simples.
  4. Exécutez l’algorithme sur le sous-problème.
  5. Combinez les résultats dans la formulation de la réponse.
  6. Renvoyez les résultats.

Source : [Fonction récursive][1]

[1] : http://www.ibm.com/developerworks/library/l-recurs/

Passer des tableaux non adjacents à des fonctions attendant de “vrais” tableaux multidimensionnels #

Lors de l’allocation de tableaux multidimensionnels avec malloc, calloc et realloc, un modèle courant consiste à allouer les tableaux internes avec plusieurs appels (même si l’appel n’apparaît qu’une seule fois, il peut être dans une boucle):

/* Could also be `int **` with malloc used to allocate outer array. */
int *array[4];
int i;

/* Allocate 4 arrays of 16 ints. */
for (i = 0; i < 4; i++)
    array[i] = malloc(16 * sizeof(*array[i]));

La différence en octets entre le dernier élément de l’un des tableaux internes et le premier élément du tableau interne suivant peut ne pas être 0 comme ils le seraient avec un “vrai” tableau multidimensionnel (par exemple int array[4][16] ; ):

/* 0x40003c, 0x402000 */
printf("%p, %p\n", (void *)(array[0] + 15), (void *)array[1]);

En tenant compte de la taille de int, vous obtenez une différence de 8128 octets (8132-4), soit 2032 éléments de tableau de taille int, et c’est là le problème : un “vrai” tableau multidimensionnel n’a pas d’espace entre éléments.

Si vous devez utiliser un tableau alloué dynamiquement avec une fonction attendant un “vrai” tableau multidimensionnel, vous devez allouer un objet de type int * et utiliser l’arithmétique pour effectuer les calculs :

void func(int M, int N, int *array);
...

/* Equivalent to declaring `int array[M][N] = {{0}};` and assigning to array4_16[i][j]. */
int *array;
int M = 4, N = 16;
array = calloc(M, N * sizeof(*array));
array[i * N + j] = 1;
func(M, N, array);

Si ‘N’ est une macro ou un littéral entier plutôt qu’une variable, le code peut simplement utiliser la notation de tableau 2D plus naturelle après avoir alloué un pointeur à un tableau :

void func(int M, int N, int *array);
#define N 16
void func_N(int M, int (*array)[N]);
...

int M = 4;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;

/* Cast to `int *` works here because `array` is a single block of M*N ints with no gaps,
   just like `int array2[M * N];` and `int array3[M][N];` would be. */
func(M, N, (int *)array);
func_N(M, array);

Si N n’est pas une macro ou un entier littéral, alors array pointera vers un tableau de longueur variable (VLA). Cela peut toujours être utilisé avec func en convertissant en int * et une nouvelle fonction func_vla remplacerait func_N :

void func(int M, int N, int *array);
void func_vla(int M, int N, int array[M][N]);
...

int M = 4, N = 16;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;
func(M, N, (int *)array);
func_vla(M, N, array);

Remarque : les VLA sont facultatifs à partir de C11. Si votre implémentation prend en charge C11 et définit la macro __STDC_NO_VLA__ à 1, vous êtes bloqué avec les méthodes pré-C99.

Dépassement des limites du tableau #

Les tableaux sont basés sur zéro, c’est-à-dire que l’index commence toujours à 0 et se termine par la longueur du tableau d’index moins 1. Ainsi, le code suivant ne produira pas le premier élément du tableau et produira des ordures pour la valeur finale qu’il imprime.

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 1; x <= 5; x++) //Looping from 1 till 5.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Sortie : 2 3 4 5 GarbageValue

Ce qui suit montre la bonne façon d’obtenir le résultat souhaité :

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 0; x < 5; x++) //Looping from 0 till 4.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Sortie : 1 2 3 4 5

Il est important de connaître la longueur d’un tableau avant de travailler avec, sinon vous risquez de corrompre le tampon ou de provoquer une erreur de segmentation en accédant à des emplacements de mémoire qui sont hors limites.
