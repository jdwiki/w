Un pointeur est un type de variable qui peut stocker l’adresse d’un autre objet ou d’une fonction.

Syntaxe #

  • <Type de données> *<Nom de la variable> ;
  • int *ptrToInt ;
  • annuler *ptrToVoid ; /* C89+ */
  • struct someStruct *ptrToStruct ;
  • entier **ptrVersPtrVersInt ;
  • int arr[longueur] ; int *ptrToFirstElem = arr; /* Pour <C99, ’longueur’ doit être une constante de temps de compilation, pour >=C11, il peut en être nécessaire. */
  • int *arrayOfPtrsToInt[longueur] ; /* Pour <C99, ’longueur’ doit être une constante de temps de compilation, pour >=C11, il peut en être nécessaire. */

La position de l’astérisque n’affecte pas le sens de la définition :

/* The * operator binds to right and therefore these are all equivalent. */
int *i;
int * i;
int* i;

Cependant, lors de la définition de plusieurs pointeurs à la fois, chacun nécessite son propre astérisque :

int *i, *j; /* i and j are both pointers */
int* i, j;  /* i is a pointer, but j is an int not a pointer variable */

Un tableau de pointeurs est également possible, où un astérisque est placé devant le nom de la variable tableau :

int *foo[2]; /* foo is a array of pointers, can be accessed as *foo[0] and *foo[1] */

Erreurs courantes #

Une mauvaise utilisation des pointeurs est souvent une source de bogues pouvant inclure des bogues de sécurité ou des plantages de programmes, le plus souvent dus à des défauts de segmentation.

Ne vérifie pas les échecs d’allocation #

L’allocation de mémoire n’est pas garantie de réussir et peut à la place renvoyer un pointeur NULL. L’utilisation de la valeur renvoyée, sans vérifier si l’allocation est réussie, invoque [comportement non défini][1]. Cela conduit généralement à un crash, mais il n’y a aucune garantie qu’un crash se produise, donc s’appuyer sur cela peut également entraîner des problèmes.

Par exemple, manière non sécurisée :

struct SomeStruct *s = malloc(sizeof *s);
s->someValue = 0; /* UNSAFE, because s might be a null pointer */

Manière sûre :

struct SomeStruct *s = malloc(sizeof *s);
if (s)
{
    s->someValue = 0; /* This is safe, we have checked that s is valid */
}

Utilisation de nombres littéraux au lieu de sizeof lors de la demande de mémoire #

Pour une configuration compilateur/machine donnée, les types ont une taille connue ; cependant, aucune norme ne définit que la taille d’un type donné (autre que char) sera la même pour toutes les configurations de compilateur/machine. Si le code utilise 4 au lieu de sizeof(int) pour l’allocation de mémoire, cela peut fonctionner sur la machine d’origine, mais le code n’est pas nécessairement portable sur d’autres machines ou compilateurs. Les tailles fixes pour les types doivent être remplacées par sizeof(that_type) ou sizeof(*var_ptr_to_that_type).

Allocation non transférable :

 int *intPtr = malloc(4*1000);    /* allocating storage for 1000 int */
 long *longPtr = malloc(8*1000);  /* allocating storage for 1000 long */

Allocation portable :

 int *intPtr = malloc(sizeof(int)*1000);     /* allocating storage for 1000 int */
 long *longPtr = malloc(sizeof(long)*1000);  /* allocating storage for 1000 long */

Ou, mieux encore :

 int *intPtr = malloc(sizeof(*intPtr)*1000);     /* allocating storage for 1000 int */
 long *longPtr = malloc(sizeof(*longPtr)*1000);  /* allocating storage for 1000 long */

Fuites de mémoire #

Le fait de ne pas désallouer de mémoire à l’aide de “free” entraîne une accumulation de mémoire non réutilisable, qui n’est plus utilisée par le programme ; c’est ce qu’on appelle une [fuite de mémoire][2]. Les fuites de mémoire gaspillent les ressources mémoire et peuvent entraîner des échecs d’allocation.

Erreurs logiques #

Toutes les allocations doivent suivre le même schéma :

  1. Allocation à l’aide de malloc (ou calloc)
  2. Utilisation pour stocker des données
  3. Désallocation en utilisant “gratuit”

Le non-respect de ce modèle, comme l’utilisation de la mémoire après un appel à free (dangling pointer) ou avant un appel à malloc ([wild pointeur] (http://stackoverflow.com/questions/2583656/what-is-the-meaning-of-wild-pointer-in-c)), appelant free deux fois (“double free”), etc., provoque généralement une erreur de segmentation et entraîne un plantage du programme.

Ces erreurs peuvent être passagères et difficiles à déboguer - par exemple, la mémoire libérée n’est généralement pas immédiatement récupérée par le système d’exploitation, et les pointeurs suspendus peuvent donc persister pendant un certain temps et sembler fonctionner.

Sur les systèmes où cela fonctionne, Valgrind est un outil inestimable pour identifier quelle mémoire est divulguée et où elle a été allouée à l’origine.

Création de pointeurs pour empiler des variables #

La création d’un pointeur ne prolonge pas la durée de vie de la variable pointée. Par exemple:

int* myFunction() 
{
    int x = 10;
    return &x;
}

Ici, x a une durée de stockage automatique (communément appelée allocation de pile). Parce qu’il est alloué sur la pile, sa durée de vie est aussi longue que myFunction est en cours d’exécution ; après la sortie de myFunction, la variable x est détruite. Cette fonction obtient l’adresse de x (en utilisant &x) et la renvoie à l’appelant, laissant l’appelant avec un pointeur vers une variable inexistante. Tenter d’accéder à cette variable invoquera alors [comportement indéfini][1].

La plupart des compilateurs n’effacent pas réellement un cadre de pile après la sortie de la fonction, donc le déréférencement du pointeur renvoyé vous donne souvent les données attendues. Cependant, lorsqu’une autre fonction est appelée, la mémoire pointée peut être écrasée et il semble que les données pointées ont été corrompues.

Pour résoudre ce problème, soit malloc le stockage de la variable à renvoyer et renvoie un pointeur vers le stockage nouvellement créé, soit exiger qu’un pointeur valide soit transmis à la fonction au lieu d’en renvoyer un, par exemple :

#include <stdlib.h>
#include <stdio.h>

int *solution1(void) 
{
    int *x = malloc(sizeof *x);
    if (x == NULL) 
    {
        /* Something went wrong */
        return NULL;
    }

    *x = 10;

    return x;
}

void solution2(int *x) 
{
    /* NB: calling this function with an invalid or null pointer 
       causes undefined behaviour. */

    *x = 10;
}

int main(void) 
{
    { 
        /* Use solution1() */

        int *foo = solution1();  
        if (foo == NULL)
        {
            /* Something went wrong */
            return 1;
        }

        printf("The value set by solution1() is %i\n", *foo);
        /* Will output: "The value set by solution1() is 10" */

        free(foo);    /* Tidy up */
    }

    {
        /* Use solution2() */

        int bar;
        solution2(&bar); 

        printf("The value set by solution2() is %i\n", bar);
        /* Will output: "The value set by solution2() is 10" */
    }

    return 0;
}

Incrémentation / décrémentation et déréférencement #

Si vous écrivez *p++ pour incrémenter ce qui est pointé par p, vous vous trompez.

La post-incrémentation/décrémentation est exécutée avant le déréférencement. Par conséquent, cette expression incrémentera le pointeur p lui-même et renverra ce qui était pointé par p avant l’incrémentation sans le changer.

Vous devez écrire (*p)++ pour incrémenter ce qui est pointé par p.

Cette règle s’applique également à la post-décrémentation : *p-- décrémentera le pointeur p lui-même, et non ce qui est pointé par p.

[1] : https://www.wikiod.com/fr/c/comportement-indefini [2] : https://en.wikipedia.org/wiki/Memory_leak

Introduction #

Un pointeur est déclaré comme n’importe quelle autre variable, sauf qu’un astérisque (*) est placé entre le type et le nom de la variable pour indiquer qu’il s’agit d’un pointeur.

int *pointer; /* inside a function, pointer is uninitialized and doesn't point to any valid object yet */

Pour déclarer deux variables pointeur de même type, dans la même déclaration, utilisez le symbole astérisque devant chaque identifiant. Par exemple,

int *iptr1, *iptr2;
int *iptr3,  iptr4;  /* iptr3 is a pointer variable, whereas iptr4 is misnamed and is an int */

L’opérateur d’adresse ou de référence noté par une esperluette (’&’) donne l’adresse d’une variable donnée qui peut être placée dans un pointeur de type approprié.

int value = 1;
pointer = &value;

L’opérateur d’indirection ou de déréférencement indiqué par un astérisque (*) obtient le contenu d’un objet pointé par un pointeur.

printf("Value of pointed to integer: %d\n", *pointer);
/* Value of pointed to integer: 1 */

Si le pointeur pointe vers une structure ou un type d’union, vous pouvez le déréférencer et accéder directement à ses membres à l’aide de l’opérateur -> :

SomeStruct *s = &someObject;
s->someMember = 5; /* Equivalent to (*s).someMember = 5 */

En C, un pointeur est un type de valeur distinct qui peut être réaffecté et est sinon traité comme une variable à part entière. Par exemple, l’exemple suivant imprime la valeur du pointeur (variable) lui-même.

printf("Value of the pointer itself: %p\n", (void *)pointer);
/* Value of the pointer itself: 0x7ffcd41b06e4 */
/* This address will be different each time the program is executed */

Parce qu’un pointeur est une variable mutable, il est possible qu’il ne pointer vers un objet valide, soit en étant mis à null

pointer = 0;     /* or alternatively */
pointer = NULL;

ou simplement en contenant un motif binaire arbitraire qui n’est pas valide adresse. Cette dernière est une très mauvaise situation, car elle ne peut pas être testé avant que le pointeur ne soit déréférencé, il n’y a qu’un test pour le cas où un pointeur est nul :

if (!pointer) exit(EXIT_FAILURE);

Un pointeur ne peut être déréférencé que s’il pointe sur un objet valide, sinon le comportement est indéfini. De nombreuses implémentations modernes peuvent vous aider en soulevant une sorte d’erreur telle qu’un [défaut de segmentation] [1] et terminer l’exécution, mais d’autres peuvent simplement laisser votre programme dans un Etat non valide.

La valeur renvoyée par l’opérateur de déréférencement est un alias mutable de la variable d’origine, elle peut donc être modifiée en modifiant la variable d’origine.

*pointer += 1;
printf("Value of pointed to variable after change: %d\n", *pointer);
/* Value of pointed to variable after change: 2 */

Les pointeurs sont également réassignables. Cela signifie qu’un pointeur pointant vers un objet peut ensuite être utilisé pour pointer vers un autre objet du même type.

int value2 = 10;
pointer = &value2;
printf("Value from pointer: %d\n", *pointer);
/* Value from pointer: 10 */

Comme toute autre variable, les pointeurs ont un type spécifique. Vous ne pouvez pas assigner l’adresse d’un short int à un pointeur vers un long int, par exemple. Un tel comportement est appelé jeu de mots de type et est interdit en C, bien qu’il existe quelques exceptions.

Bien que le pointeur doive être d’un type spécifique, la mémoire allouée à chaque type de pointeur est égale à la mémoire utilisée par l’environnement pour stocker les adresses, plutôt qu’à la taille du type pointé.

#include <stdio.h>

int main(void) {
    printf("Size of int pointer: %zu\n", sizeof (int*));      /* size 4 bytes */
    printf("Size of int variable: %zu\n", sizeof (int));      /* size 4 bytes */
    printf("Size of char pointer: %zu\n", sizeof (char*));    /* size 4 bytes */
    printf("Size of char variable: %zu\n", sizeof (char));    /* size 1 bytes */
    printf("Size of short pointer: %zu\n", sizeof (short*));  /* size 4 bytes */
    printf("Size of short variable: %zu\n", sizeof (short));  /* size 2 bytes */
    return 0;
}

(NB : si vous utilisez Microsoft Visual Studio, qui ne prend pas en charge les normes C99 ou C11, vous devez utiliser [%Iu](https://msdn.microsoft.com/en-us/library/tcxf1dw6. aspx)1 au lieu de %zu dans l’exemple ci-dessus.)

Notez que les résultats ci-dessus peuvent varier d’un environnement à l’autre en nombre, mais tous les environnements afficheraient des tailles égales pour différents types de pointeurs.

Extrait basé sur les informations de Cardiff University C Pointers Introduction

[1] : https://en.wikipedia.org/wiki/Segmentation_fault

Pointeurs et tableaux #

Les pointeurs et les tableaux sont intimement liés en C. Les tableaux en C sont toujours conservés dans des emplacements contigus en mémoire. L’arithmétique du pointeur est toujours mise à l’échelle par la taille de l’élément pointé. Donc, si nous avons un tableau de trois doubles et un pointeur vers la base, *ptr fait référence au premier double, *(ptr + 1) au second, *(ptr + 2) au troisième . Une notation plus pratique consiste à utiliser la notation de tableau [].

double point[3] = {0.0, 1.0, 2.0};
double *ptr = point;

/* prints x 0.0, y 1.0 z 2.0 */ 
printf("x %f y %f z %f\n", ptr[0], ptr[1], ptr[2]);

Donc, essentiellement, ptr et le nom du tableau sont interchangeables. Cette règle signifie également qu’un tableau se désintègre en un pointeur lorsqu’il est passé à un sous-programme.

double point[3] = {0.0, 1.0, 2.0};

printf("length of point is %s\n", length(point));

/* get the distance of a 3D point from the origin */ 
double length(double *pt)
{
   return sqrt(pt[0] * pt[0] + pt[1] * pt[1] + pt[2] * pt[2])
}

Un pointeur peut pointer sur n’importe quel élément d’un tableau ou sur l’élément au-delà du dernier élément. C’est cependant une erreur de définir un pointeur sur toute autre valeur, y compris l’élément avant le tableau. (La raison en est que sur les architectures segmentées, l’adresse avant le premier élément peut traverser une limite de segment, le compilateur s’assure que cela ne se produit pas pour le dernier élément plus un).

Note de bas de page 1 : les informations de format Microsoft sont disponibles via printf() et [syntaxe de spécification de format](https : //msdn.microsoft.com/en-us/library/56e442dc.aspx).

Déréférencer un pointeur #

int a = 1;
int *a_pointer = &a;

Pour déréférencer a_pointer et changer la valeur de a, nous utilisons l’opération suivante

*a_pointer = 2;

Cela peut être vérifié à l’aide des déclarations d’impression suivantes.

printf("%d\n", a); /* Prints 2 */
printf("%d\n", *a_pointer); /* Also prints 2 */

Cependant, on aurait tort de déréférencer un pointeur NULL ou autrement invalide. Cette

int *p1, *p2;

p1 = (int *) 0xbad;
p2 = NULL;

*p1 = 42;
*p2 = *p1 + 1;

est généralement [comportement indéfini][1]. « p1 » ne peut pas être déréférencé car il pointe vers une adresse « 0xbad » qui peut ne pas être une adresse valide. Qui sait ce qu’il y a ? Il peut s’agir de la mémoire du système d’exploitation ou de la mémoire d’un autre programme. Le seul code temporel comme celui-ci est utilisé dans le développement intégré, qui stocke des informations particulières à des adresses codées en dur. p2 ne peut pas être déréférencé car il est NULL, ce qui n’est pas valide.

[1] : https://www.wikiod.com/fr/c/comportement-indefini

Déréférencer un pointeur vers une structure #

Disons que nous avons la structure suivante :

struct MY_STRUCT 
{
    int my_int;
    float my_float;
};

Nous pouvons définir MY_STRUCT pour omettre le mot-clé struct afin de ne pas avoir à taper struct MY_STRUCT à chaque fois que nous l’utilisons. Ceci, cependant, est facultatif.

typedef struct MY_STRUCT MY_STRUCT;

Si nous avons alors un pointeur vers une instance de cette structure

MY_STRUCT *instance;

Si cette instruction apparaît au niveau du fichier, instance sera initialisée avec un pointeur nul au démarrage du programme. Si cette instruction apparaît à l’intérieur d’une fonction, sa valeur est indéfinie. La variable doit être initialisée pour pointer vers une variable MY_STRUCT valide, ou vers un espace alloué dynamiquement, avant de pouvoir être déréférencée. Par exemple:

MY_STRUCT info = { 1, 3.141593F };
MY_STRUCT *instance = &info;

Lorsque le pointeur est valide, nous pouvons le déréférencer pour accéder à ses membres en utilisant l’une des deux notations différentes :

int a = (*instance).my_int;
float b = instance->my_float;

Bien que ces deux méthodes fonctionnent, il est préférable d’utiliser l’opérateur flèche -> plutôt que la combinaison de parenthèses, l’opérateur de déréférencement * et l’opérateur point . car il est plus facile à lire et à comprendre, en particulier avec utilisations imbriquées.

Une autre différence importante est illustrée ci-dessous :

MY_STRUCT copy = *instance;
copy.my_int    = 2;

Dans ce cas, copy contient une copie du contenu de instance. Changer my_int de copy ne le changera pas dans instance.

MY_STRUCT *ref = instance;
ref->my_int    = 2;

Dans ce cas, ref est une référence à instance. Changer my_int en utilisant la référence le changera dans instance.

Il est courant d’utiliser des pointeurs vers des structures comme paramètres dans les fonctions, plutôt que les structures elles-mêmes. L’utilisation des structures en tant que paramètres de fonction peut provoquer un débordement de la pile si la structure est volumineuse. L’utilisation d’un pointeur vers une structure n’utilise que suffisamment d’espace de pile pour le pointeur, mais peut entraîner des effets secondaires si la fonction modifie la structure qui est transmise à la fonction.

Pointeurs de fonction #

Les pointeurs peuvent également être utilisés pour pointer vers des fonctions.

Prenons une fonction de base :

int my_function(int a, int b)
{
    return 2 * a + 3 * b;
}

Définissons maintenant un pointeur du type de cette fonction :

int (*my_pointer)(int, int);

Pour en créer un, utilisez simplement ce modèle :

return_type_of_func (*my_func_pointer)(type_arg1, type_arg2, ...)

Il faut alors affecter ce pointeur à la fonction :

my_pointer = &my_function;

Ce pointeur peut maintenant être utilisé pour appeler la fonction :

/* Calling the pointed function */
int result = (*my_pointer)(4, 2);

...

/* Using the function pointer as an argument to another function */
void another_function(int (*another_pointer)(int, int))
{
    int a = 4;
    int b = 2;
    int result = (*another_pointer)(a, b);

    printf("%d\n", result);
}

Bien que cette syntaxe semble plus naturelle et cohérente avec les types de base, l’attribution et le déréférencement des pointeurs de fonction ne nécessitent pas l’utilisation des opérateurs & et *. Ainsi, l’extrait de code suivant est également valide :

/* Attribution without the & operator */
my_pointer = my_function;

/* Dereferencing without the * operator */
int result = my_pointer(4, 2);

Pour augmenter la lisibilité des pointeurs de fonction, les typedefs peuvent être utilisés.

typedef void (*Callback)(int a);

void some_function(Callback callback)
{
    int a = 4;
    callback(a);
}

Une autre astuce de lisibilité est que le standard C permet de simplifier un pointeur de fonction dans des arguments comme ci-dessus (mais pas dans la déclaration de variable) en quelque chose qui ressemble à un prototype de fonction ; ainsi, ce qui suit peut être utilisé de manière équivalente pour les définitions et les déclarations de fonctions :

void some_function(void callback(int))
{
    int a = 4;
    callback(a);
}

Pointeurs const #

Pointeurs uniques #

  • Pointeur vers un int

Le pointeur peut pointer vers différents nombres entiers et les “int” peuvent être modifiés via le pointeur. Cet exemple de code assigne b pour pointer vers int b puis change la valeur de b en 100.

  int b;
  int* p;
  p = &b;    /* OK */
  *p = 100;  /* OK */

  • Pointeur vers un const int

    The pointer can point to different integers but the int’s value can’t be changed through the pointer.

    int b;
const int* p;
p = &b;    /* OK */
*p = 100;  /* Compiler Error */

  • const pointeur vers int

Le pointeur ne peut pointer que sur un seul int mais la valeur de int peut être modifiée via le pointeur.

  int a, b;
  int* const p = &b; /* OK as initialisation, no assignment */
  *p = 100;  /* OK */
  p = &a;    /* Compiler Error */
  • const pointeur vers const int

Le pointeur ne peut pointer que vers un int et le int ne peut pas être modifié via le pointeur.

  int a, b;
  const int* const p = &b; /* OK as initialisation, no assignment */
  p = &a;   /* Compiler Error */
  *p = 100; /* Compiler Error */

Comportement polymorphe avec des pointeurs vides #

La fonction de bibliothèque standard [qsort()][1] est un bon exemple de la façon dont on peut utiliser des pointeurs vides pour faire fonctionner une seule fonction sur une grande variété de types différents.

void qsort (
    void *base,                                 /* Array to be sorted */
    size_t num,                                 /* Number of elements in array */
    size_t size,                                /* Size in bytes of each element */
    int (*compar)(const void *, const void *)); /* Comparison function for two elements */

Le tableau à trier est passé en tant que pointeur vide, de sorte qu’un tableau de n’importe quel type d’élément peut être utilisé. Les deux arguments suivants indiquent à qsort() le nombre d’éléments attendus dans le tableau et la taille, en octets, de chaque élément.

Le dernier argument est un pointeur de fonction vers une fonction de comparaison qui prend elle-même deux pointeurs vides. En faisant en sorte que l’appelant fournisse cette fonction, qsort() peut efficacement trier des éléments de n’importe quel type.

Voici un exemple d’une telle fonction de comparaison, pour comparer des flottants. Notez que toute fonction de comparaison passée à qsort() doit avoir cette signature de type. La façon dont il est rendu polymorphe consiste à convertir les arguments de pointeur void en pointeurs du type d’élément que nous souhaitons comparer.

int compare_floats(const void *a, const void *b)
{
    float fa = *((float *)a);
    float fb = *((float *)b);
    if (fa < fb)
        return -1;
    if (fa > fb)
        return 1;
    return 0;
}

Puisque nous savons que qsort utilisera cette fonction pour comparer les flottants, nous reconvertissons les arguments du pointeur vide en pointeurs flottants avant de les déréférencer.

Maintenant, l’utilisation de la fonction polymorphe qsort sur un tableau “array” de longueur “len” est très simple :

qsort(array, len, sizeof(array[0]), compare_floats);

[1] : http://man7.org/linux/man-pages/man3/qsort.3.html

Initialisation des pointeurs #

L’initialisation du pointeur est un bon moyen d’éviter les pointeurs sauvages. L’initialisation est simple et n’est pas différente de l’initialisation d’une variable.

#include <stddef.h>

int main()
{
    int *p1 = NULL; 
    char *p2 = NULL;
    float *p3 = NULL;

         /* NULL is a macro defined in stddef.h, stdio.h, stdlib.h, and string.h */

    ...
}

Dans la plupart des systèmes d’exploitation, l’utilisation par inadvertance d’un pointeur qui a été initialisé sur “NULL” entraîne souvent le plantage immédiat du programme, ce qui facilite l’identification de la cause du problème. L’utilisation d’un pointeur non initialisé peut souvent provoquer des bogues difficiles à diagnostiquer.

Mise en garde: #

Le résultat du déréférencement d’un pointeur NULL n’est pas défini, donc il ne causera pas nécessairement un plantage même si c’est le comportement naturel du système d’exploitation sur lequel le programme s’exécute. Les optimisations du compilateur peuvent masquer le blocage, provoquer le blocage avant ou après le point du code source auquel le déréférencement du pointeur nul s’est produit ou entraîner la suppression inattendue de parties du code contenant le déréférencement du pointeur nul du programme. Les versions de débogage ne présentent généralement pas ces comportements, mais cela n’est pas garanti par la norme du langage. Tout autre comportement inattendu et/ou indésirable est également autorisé.

Étant donné que NULL ne pointe jamais vers une variable, vers la mémoire allouée ou vers une fonction, il est sûr de l’utiliser comme valeur de garde.

Mise en garde: #

Habituellement, NULL est défini comme (void *)0. Mais cela n’implique pas que l’adresse mémoire assignée soit ‘0x0’. Pour plus de précisions, reportez-vous à C-faq pour les pointeurs NULL

Notez que vous pouvez également initialiser des pointeurs pour qu’ils contiennent des valeurs autres que NULL.

int i1;

int main()
{
   int *p1 = &i1;
   const char *p2 = "A constant string to point to";
   float *p3 = malloc(10 * sizeof(float));
}

Adresse de l’opérateur ( & ) #

Pour tout objet (c’est-à-dire variable, tableau, union, structure, pointeur ou fonction), l’opérateur d’adresse unaire peut être utilisé pour accéder à l’adresse de cet objet.

Supposer que

int i = 1;              
int *p = NULL;

Ainsi, une instruction p = &i; copie l’adresse de la variable i dans le pointeur p.

Il est exprimé par p *** pointe sur *** i.

printf("%d\n", *p); imprime 1, qui est la valeur de i.

void* pointeurs comme arguments et valeurs de retour aux fonctions standard #

void* est un type fourre-tout pour les pointeurs vers les types d’objets. Un exemple de ceci est utilisé avec la fonction malloc, qui est déclarée comme

void* malloc(size_t);

Le type de retour pointer-to-void signifie qu’il est possible d’assigner la valeur de retour de malloc à un pointeur vers n’importe quel autre type d’objet :

int* vector = malloc(10 * sizeof *vector);

Il est généralement considéré comme une bonne pratique de ne pas transtyper explicitement les valeurs dans et hors des pointeurs vides. Dans le cas spécifique de malloc(), c’est parce qu’avec un transtypage explicite, le compilateur peut autrement supposer, mais pas avertir, un type de retour incorrect pour malloc(), si vous oubliez d’inclure stdlib.h. Il s’agit également d’utiliser le comportement correct des pointeurs void pour mieux se conformer au principe DRY (ne vous répétez pas) ; comparez ce qui précède au suivant, dans lequel le code suivant contient plusieurs endroits supplémentaires inutiles où une faute de frappe pourrait causer des problèmes :

int* vector = (int*)malloc(10 * sizeof int*);

De même, des fonctions telles que

void* memcpy(void *restrict target, void const *restrict source, size_t size);

ont leurs arguments spécifiés comme void * car l’adresse de n’importe quel objet, quel que soit son type, peut être transmise. Ici aussi, un appel ne doit pas utiliser de transtypage

unsigned char buffer[sizeof(int)];
int b = 67;
memcpy(buffer, &b, sizeof buffer);

Même astérisque, significations différentes #

Prémisse #

La chose la plus déroutante entourant la syntaxe du pointeur en C et C++ est qu’il existe en fait deux significations différentes qui s’appliquent lorsque le symbole du pointeur, l’astérisque (*), est utilisé avec une variable.

Exemple #

Tout d’abord, vous utilisez * pour déclarer une variable de pointeur.

int i = 5;
/* 'p' is a pointer to an integer, initialized as NULL */
int *p = NULL;
/* '&i' evaluates into address of 'i', which then assigned to 'p' */
p = &i;
/* 'p' is now holding the address of 'i' */

Lorsque vous ne déclarez pas (ou ne multipliez pas), * est utilisé pour déréférencer une variable de pointeur :

*p = 123;
/* 'p' was pointing to 'i', so this changes value of 'i' to 123 */

Lorsque vous voulez qu’une variable de pointeur existante contienne l’adresse d’une autre variable, vous n’utilisez pas *, mais procédez comme ceci :

p = &another_variable;

Une confusion courante chez les débutants en programmation C survient lorsqu’ils déclarent et initialisent une variable de pointeur en même temps.

int *p = &i;

Puisque int i = 5; et int i; i = 5; donnent le même résultat, certains d’entre eux pourraient penser int *p = &i; et int *p; *p = &i; donne également le même résultat. Le fait est, non, int *p; *p = &i; tentera de différencier un pointeur non initialisé qui se traduira par UB. N’utilisez jamais * lorsque vous ne déclarez ni ne déréférencez un pointeur.

Conclusion #

L’astérisque (*) a deux significations distinctes en C par rapport aux pointeurs, selon l’endroit où il est utilisé. Lorsqu’elle est utilisée dans une déclaration de variable, la valeur à droite du côté égal doit être une valeur de pointeur vers une adresse en mémoire. Lorsqu’il est utilisé avec une variable déjà déclarée, l’astérisque va déréférencer la valeur du pointeur, le suivant jusqu’à l’endroit pointé en mémoire, et permettant à la valeur qui y est stockée d’être assignée ou récupérée.

Emporter

Il est important de faire attention à vos P et Q, pour ainsi dire, lorsqu’il s’agit de pointeurs. Soyez conscient du moment où vous utilisez l’astérisque et de ce que cela signifie lorsque vous l’utilisez ici. Négliger ce petit détail pourrait entraîner un comportement bogué et/ou indéfini auquel vous ne voulez vraiment pas avoir à faire face.
