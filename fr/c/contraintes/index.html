Contraintes

  • Noms de variables en double dans la même portée
  • Opérateurs arithmétiques unaires

Sur cette page

  • Noms de variables en double dans la même portée

Les contraintes sont un terme utilisé dans toutes les spécifications C existantes (récemment ISO-IEC 9899-2011). Ils constituent l’une des trois parties du langage décrites à l’article 6 de la norme (avec la syntaxe et la sémantique).

ISO-IEC 9899-2011 définit une contrainte comme :

restriction, syntaxique ou sémantique, par laquelle l’exposition du langage les éléments doivent être interprétés

(Veuillez également noter qu’en termes de norme C, une “contrainte d’exécution” n’est pas une sorte de contrainte et a des règles très différentes.)

En d’autres termes, une contrainte décrit une règle du langage qui rendrait illégal un programme autrement syntaxiquement valide. À cet égard, les contraintes sont un peu comme un comportement indéfini, tout programme qui ne les suit pas n’est pas défini en termes de langage C.

Les contraintes, par contre, ont une différence très significative par rapport aux comportements indéfinis. A savoir, une implémentation doit fournir un message de diagnostic pendant la phase de traduction (partie de la compilation) si une contrainte est violée, ce message peut être un avertissement ou peut arrêter la compilation.

Noms de variables en double dans la même portée #

Un exemple de contrainte telle qu’exprimée dans la norme C est d’avoir deux variables du même nom déclarées dans une portée1), par exemple :

void foo(int bar)
{
    int var;
    double var;
}

Ce code viole la contrainte et doit produire un message de diagnostic au moment de la compilation. Ceci est très utile par rapport au comportement indéfini car le développeur sera informé du problème avant que le programme ne soit exécuté, ce qui peut potentiellement faire n’importe quoi.

Les contraintes ont donc tendance à être des erreurs facilement détectables au moment de la compilation comme celle-ci, les problèmes qui entraînent un comportement indéfini mais seraient difficiles ou impossibles à détecter au moment de la compilation ne sont donc pas des contraintes.

1) formulation exacte :

Si un identifiant n’a pas de lien, il ne doit pas y avoir plus d’une déclaration de l’identifiant (dans un déclarateur ou un spécificateur de type) avec la même portée et dans le même espace de noms, sauf pour les balises comme spécifié au 6.7.2.3.

Opérateurs arithmétiques unaires #

Les opérateurs unaires + et - ne sont utilisables que sur les types arithmétiques, donc si par exemple on essaie de les utiliser sur une structure le programme produira un diagnostic par exemple :

struct foo
{
    bool bar;
};

void baz(void)
{
   struct foo testStruct;
   -testStruct; /* This breaks the constraint so must produce a diagnostic */
}
