Toutes les commandes du préprocesseur commencent par le symbole dièse (dièse) #. Une macro C est juste une commande de préprocesseur définie à l’aide de la directive de préprocesseur #define. Au cours de l’étape de prétraitement, le préprocesseur C (une partie du compilateur C) remplace simplement le corps de la macro partout où son nom apparaît.

Lorsqu’un compilateur rencontre une macro dans le code, il effectue un simple remplacement de chaîne, aucune opération supplémentaire n’est effectuée. Pour cette raison, les modifications apportées par le préprocesseur ne respectent pas la portée des programmes C - par exemple, une définition de macro n’est pas limitée à être dans un bloc, elle n’est donc pas affectée par un '}' qui termine une instruction de bloc.

Le préprocesseur est, de par sa conception, incomplet - il existe plusieurs types de calculs qui ne peuvent pas être effectués par le préprocesseur seul.

Habituellement, les compilateurs ont un indicateur de ligne de commande (ou paramètre de configuration) qui nous permet d’arrêter la compilation après la phase de prétraitement et d’inspecter le résultat. Sur les plates-formes POSIX, cet indicateur est -E. Ainsi, exécuter gcc avec cet indicateur imprime la source étendue sur stdout :

$ gcc -E cprog.c

Souvent, le préprocesseur est implémenté en tant que programme séparé, qui est invoqué par le compilateur, le nom commun de ce programme est cpp. Un certain nombre de préprocesseurs émettent des informations de support, telles que des informations sur les numéros de ligne - qui sont utilisées par les phases ultérieures de compilation pour générer des informations de débogage. Dans le cas où le préprocesseur est basé sur gcc, l’option -P supprime ces informations.

$ cpp -P cprog.c

#if 0 pour bloquer les sections de code #

S’il y a des sections de code que vous envisagez de supprimer ou que vous souhaitez désactiver temporairement, vous pouvez les commenter avec un bloc de commentaire.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
*/

Cependant, si le code source que vous avez entouré d’un commentaire de bloc contient des commentaires de style bloc dans la source, la fin */ des commentaires de bloc existants peut rendre votre nouveau commentaire de bloc invalide et entraîner des problèmes de compilation.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;

    /* Return 5 */
    return i;
}
*/

Dans l’exemple précédent, les deux dernières lignes de la fonction et le dernier ‘*/’ sont vus par le compilateur, il compilerait donc avec des erreurs. Une méthode plus sûre consiste à utiliser une directive #if 0 autour du code que vous souhaitez bloquer.

#if 0
/* #if 0 evaluates to false, so everything between here and the #endif are
 * removed by the preprocessor. */
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
#endif

Un avantage avec ceci est que lorsque vous voulez revenir en arrière et trouver le code, il est beaucoup plus facile de faire une recherche pour “#if 0” que de rechercher tous vos commentaires.

Un autre avantage très important est que vous pouvez imbriquer le commentaire de code avec #if 0. Cela ne peut pas être fait avec des commentaires.

Une alternative à l’utilisation de #if 0 consiste à utiliser un nom qui ne sera pas #defined mais qui décrit davantage la raison pour laquelle le code est bloqué. Par exemple, s’il y a une fonction qui semble être un code mort inutile, vous pouvez utiliser #if defined(POSSIBLE_DEAD_CODE) ou #if defined(FUTURE_CODE_REL_020201) pour le code nécessaire une fois qu’une autre fonctionnalité est en place ou quelque chose de similaire. Ensuite, lorsque vous revenez pour supprimer ou activer cette source, ces sections de source sont faciles à trouver.

L’en-tête inclut les gardes #

Pratiquement tous les fichiers d’en-tête doivent suivre l’idiome include guard :

mon-fichier-en-tête.h

#ifndef MY_HEADER_FILE_H
#define MY_HEADER_FILE_H

// Code body for header file

#endif

Cela garantit que lorsque vous #include "my-header-file.h" à plusieurs endroits, vous n’obtenez pas de déclarations en double de fonctions, de variables, etc. Imaginez la hiérarchie de fichiers suivante :

en-tête-1.h

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

en-tête-2.h

#include "header-1.h"

int myFunction2(MyStruct *value);

principal c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Ce code a un sérieux problème : le contenu détaillé de MyStruct est défini deux fois, ce qui n’est pas autorisé. Cela entraînerait une erreur de compilation qui peut être difficile à localiser, car un fichier d’en-tête en inclut un autre. Si vous l’avez plutôt fait avec des gardes d’en-tête :

en-tête-1.h

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

en-tête-2.h

#ifndef HEADER_2_H
#define HEADER_2_H

#include "header-1.h"

int myFunction2(MyStruct *value);

#endif

principal c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Cela s’étendrait alors à:

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

#ifndef HEADER_2_H
#define HEADER_2_H

#ifndef HEADER_1_H // Safe, since HEADER_1_H was #define'd before.
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

int myFunction2(MyStruct *value);

#endif

int main() {
    // do something
}

Lorsque le compilateur atteint la deuxième inclusion de header-1.h, HEADER_1_H était déjà défini par l’inclusion précédente. Ergo, cela se résume à ce qui suit :

#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#define HEADER_2_H

int myFunction2(MyStruct *value);

int main() {
    // do something
}

Et donc il n’y a pas d’erreur de compilation.

Remarque : Il existe plusieurs conventions différentes pour nommer les protections d’en-tête. Certaines personnes aiment le nommer HEADER_2_H_, d’autres incluent le nom du projet comme MY_PROJECT_HEADER_2_H. L’important est de s’assurer que la convention que vous suivez fait en sorte que chaque fichier de votre projet ait un garde d’en-tête unique.

Si les détails de la structure n’étaient pas inclus dans l’en-tête, le type déclaré serait incomplet ou un type opaque. De tels types peuvent être utiles, en cachant les détails d’implémentation aux utilisateurs des fonctions. À de nombreuses fins, le type FILE dans la bibliothèque C standard peut être considéré comme un type opaque (bien qu’il ne soit généralement pas opaque afin que les implémentations de macros des fonctions d’E / S standard puissent utiliser les éléments internes de la structure) . Dans ce cas, le header-1.h pourrait contenir :

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct MyStruct MyStruct;

int myFunction(MyStruct *value);

#endif

Notez que la structure doit avoir un nom de balise (ici MyStruct - qui se trouve dans l’espace de noms des balises, séparé de l’espace de noms des identifiants ordinaires du nom de typedef MyStruct), et que le { … } est omis. Cela dit “il existe un type de structure struct MyStruct et il existe un alias pour celui-ci MyStruct”.

Dans le fichier d’implémentation, les détails de la structure peuvent être définis pour compléter le type :

struct MyStruct {
    …
};

Si vous utilisez C11, vous pouvez répéter la déclaration typedef struct MyStruct MyStruct; sans provoquer d’erreur de compilation, mais les versions antérieures de C se plaindraient. Par conséquent, il est toujours préférable d’utiliser l’idiome de garde include, même si dans cet exemple, il serait facultatif si le code n’a jamais été compilé qu’avec des compilateurs prenant en charge C11.

De nombreux compilateurs prennent en charge la directive #pragma once, qui produit les mêmes résultats :

mon-fichier-en-tête.h

#pragma once

// Code for header file

Cependant, #pragma once ne fait pas partie de la norme C, donc le code est moins portable si vous l’utilisez.

Quelques en-têtes n’utilisent pas l’idiome de garde include. Un exemple spécifique est l’en-tête standard <assert.h>. Il peut être inclus plusieurs fois dans une seule unité de traduction, et l’effet de cela dépend si la macro NDEBUG est définie chaque fois que l’en-tête est inclus. Vous pouvez parfois avoir une exigence analogue ; de tels cas seront rares. Normalement, vos en-têtes doivent être protégés par l’idiome include guard.

Macros de type fonction #

Les macros de type fonction sont similaires aux fonctions “inline”, elles sont utiles dans certains cas, comme le journal de débogage temporaire :

#ifdef DEBUG
# define LOGFILENAME "/tmp/logfile.log"

# define LOG(str) do {                            \
  FILE *fp = fopen(LOGFILENAME, "a");            \
  if (fp) {                                       \
    fprintf(fp, "%s:%d %s\n", __FILE__, __LINE__, \
                 /* don't print null pointer */   \
                 str ?str :"<null>");             \
    fclose(fp);                                   \
  }                                               \
  else {                                          \
    perror("Opening '" LOGFILENAME "' failed");   \
  }                                               \
} while (0)
#else
  /* Make it a NOOP if DEBUG is not defined. */
# define LOG(LINE) (void)0
#endif


#include <stdio.h>

int main(int argc, char* argv[])
{
    if (argc > 1)
        LOG("There are command line arguments");
    else
        LOG("No command line arguments");
    return 0;
}

Ici dans les deux cas (avec DEBUG ou non) l’appel se comporte de la même manière en tant que fonction avec le type de retour void. Cela garantit que les conditions “if/else” sont interprétées comme prévu.

Dans le cas DEBUG, cela est implémenté via un do { ... } construction while(0). Dans l’autre cas, (void)0 est une instruction sans effet secondaire qui est simplement ignorée.

Une alternative pour ce dernier serait

#define LOG(LINE) do { /* empty */ } while (0)

de sorte qu’il est dans tous les cas syntaxiquement équivalent au premier.

Si vous utilisez GCC, vous pouvez également implémenter une macro de type fonction qui renvoie le résultat à l’aide d’une extension GNU non standard - [expressions d’instruction] (https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/ Statement-Exprs.html#Statement-Exprs). Par exemple:

#include <stdio.h>

#define POW(X, Y) \
({ \
        int i, r = 1; \
        for (i = 0; i < Y; ++i) \
                r *= X; \
        r; \ // returned value is result of last operation
})

int main(void)
{
        int result;

        result = POW(2, 3); 
        printf("Result: %d\n", result);
}

Inclusion du fichier source #

Les utilisations les plus courantes des directives de prétraitement #include sont les suivantes :

#include <stdio.h>
#include "myheader.h"

#include remplace l’instruction par le contenu du fichier auquel il est fait référence. Les crochets angulaires (<>) font référence aux fichiers d’en-tête installés sur le système, tandis que les guillemets ("") concernent les fichiers fournis par l’utilisateur.

Les macros elles-mêmes peuvent développer d’autres macros une fois, comme l’illustre cet exemple :

#if VERSION == 1
    #define INCFILE  "vers1.h"
#elif VERSION == 2
    #define INCFILE  "vers2.h"
    /*  and so on */
#else
    #define INCFILE  "versN.h"
#endif
/* ... */
#include INCFILE

Inclusion conditionnelle et modification de signature de fonction conditionnelle #

Pour inclure conditionnellement un bloc de code, le préprocesseur a plusieurs directives (par exemple #if, #ifdef, #else, #endif, etc.).

/* Defines a conditional `printf` macro, which only prints if `DEBUG`
 * has been defined
 */
#ifdef DEBUG
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

Les opérateurs relationnels C normaux peuvent être utilisés pour la condition “#if”

#if __STDC_VERSION__ >= 201112L
/* Do stuff for C11 or higher */
#elif __STDC_VERSION__ >= 199901L
/* Do stuff for C99 */
#else
/* Do stuff for pre C99 */
#endif

Les directives #if se comportent de la même manière que l’instruction C if, elles ne doivent contenir que des expressions constantes intégrales et aucun transtypage. Il prend en charge un opérateur unaire supplémentaire, defined( identifier ), qui renvoie 1 si l’identifiant est défini, et 0 sinon.

#if defined(DEBUG) && !defined(QUIET)
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

Modification conditionnelle de la signature de la fonction

Dans la plupart des cas, une version de version d’une application devrait avoir le moins de temps système possible. Toutefois, lors du test d’une version intermédiaire, des journaux et des informations supplémentaires sur les problèmes détectés peuvent être utiles.

Par exemple, supposons qu’il existe une fonction SHORT SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd) qui, lors d’une construction de test, générera un journal sur son utilisation. Cependant, cette fonction est utilisée à plusieurs endroits et il est souhaitable que lors de la génération du journal, une partie des informations consiste à savoir d’où la fonction est appelée.

Ainsi, en utilisant la compilation conditionnelle, vous pouvez avoir quelque chose comme ce qui suit dans le fichier d’inclusion déclarant la fonction. Cela remplace la version standard de la fonction par une version de débogage de la fonction. Le préprocesseur est utilisé pour remplacer les appels à la fonction SerOpPluAllRead() par des appels à la fonction SerOpPluAllRead_Debug() avec deux arguments supplémentaires, le nom du fichier et le numéro de ligne où la fonction est utilisée.

La compilation conditionnelle est utilisée pour choisir de remplacer ou non la fonction standard par une version de débogage.

#if 0
// function declaration and prototype for our debug version of the function.
SHORT   SerOpPluAllRead_Debug(PLUIF *pPif, USHORT usLockHnd, char *aszFilePath, int nLineNo);

// macro definition to replace function call using old name with debug function with additional arguments.
#define SerOpPluAllRead(pPif,usLock) SerOpPluAllRead_Debug(pPif,usLock,__FILE__,__LINE__)
#else
// standard function declaration that is normally used with builds.
SHORT   SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd);
#endif

Cela vous permet de remplacer la version standard de la fonction SerOpPluAllRead() par une version qui fournira le nom du fichier et le numéro de ligne dans le fichier où la fonction est appelée.

Il y a une considération importante : tout fichier utilisant cette fonction doit inclure le fichier d’en-tête où cette approche est utilisée pour que le préprocesseur puisse modifier la fonction. Sinon, vous verrez une erreur de l’éditeur de liens.

La définition de la fonction ressemblerait à ce qui suit. Ce que fait cette source est de demander au préprocesseur de renommer la fonction SerOpPluAllRead() en SerOpPluAllRead_Debug() et de modifier la liste d’arguments pour inclure deux arguments supplémentaires, un pointeur vers le nom du fichier où la fonction a été appelée et le numéro de ligne dans le fichier où la fonction est utilisée.

#if defined(SerOpPluAllRead)
// forward declare the replacement function which we will call once we create our log.
SHORT    SerOpPluAllRead_Special(PLUIF *pPif, USHORT usLockHnd);

SHORT    SerOpPluAllRead_Debug(PLUIF *pPif, USHORT usLockHnd, char *aszFilePath, int nLineNo)
{
    int iLen = 0;
    char  xBuffer[256];

    // only print the last 30 characters of the file name to shorten the logs.
    iLen = strlen (aszFilePath);
    if (iLen > 30) {
        iLen = iLen - 30;
    }
    else {
        iLen = 0;
    }

    sprintf (xBuffer, "SerOpPluAllRead_Debug(): husHandle = %d, File %s, lineno = %d", pPif->husHandle, aszFilePath + iLen, nLineNo);
    IssueDebugLog(xBuffer);

    // now that we have issued the log, continue with standard processing.
    return SerOpPluAllRead_Special(pPif, usLockHnd);
}

// our special replacement function name for when we are generating logs.
SHORT    SerOpPluAllRead_Special(PLUIF *pPif, USHORT usLockHnd)
#else
// standard, normal function name (signature) that is replaced with our debug version.
SHORT   SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd)
#endif
{
    if (STUB_SELF == SstReadAsMaster()) {
        return OpPluAllRead(pPif, usLockHnd);
    } 
    return OP_NOT_MASTER;
}

Collage de jetons #

Le collage de jetons permet de coller ensemble deux arguments de macro. Par exemple, front##back donne frontback. Un exemple célèbre est l’en-tête <TCHAR.H> de Win32. En C standard, on peut écrire L"chaîne" pour déclarer une chaîne de caractères étendue. Cependant, l’API Windows permet de convertir entre des chaînes de caractères larges et des chaînes de caractères étroites simplement en #defineing UNICODE. Afin d’implémenter les littéraux de chaîne, TCHAR.H utilise ceci

#ifdef UNICODE
#define TEXT(x) L##x
#endif

Chaque fois qu’un utilisateur écrit TEXT("hello, world") et qu’UNICODE est défini, le préprocesseur C concatène L et l’argument macro. L concaténé avec "hello, world" donne L"hello, world".

Macros prédéfinies #

Une macro prédéfinie est une macro qui est déjà comprise par le préprocesseur C sans que le programme ait besoin de la définir. Les exemples comprennent

Macros prédéfinies obligatoires #

  • __FILE__, qui donne le nom de fichier du fichier source courant (un littéral de chaîne),
  • __LINE__ pour le numéro de la ligne courante (une constante entière),
  • __DATE__ pour la date de compilation (un littéral de chaîne),
  • __TIME__ pour le temps de compilation (un littéral de chaîne).

Il existe également un identifiant prédéfini associé, __func__ (ISO/IEC 9899:2011 §6.4.2.2), qui n’est pas une macro :

L’identifiant __func__ doit être implicitement déclaré par le traducteur comme si, immédiatement après l’accolade ouvrante de chaque définition de fonction, la déclaration :

static const char func[] = “nom-fonction” ;

est apparu, où function-name est le nom de la fonction lexicalement englobante.

__FILE__, __LINE__ et __func__ sont particulièrement utiles à des fins de débogage. Par exemple:

fprintf(stderr, "%s: %s: %d: Denominator is 0", __FILE__, __func__, __LINE__);

Les compilateurs antérieurs à C99 peuvent ou non prendre en charge __func__ ou peuvent avoir une macro qui agit de la même manière et qui est nommée différemment. Par exemple, gcc a utilisé __FUNCTION__ en mode C89.

Les macros ci-dessous permettent de demander des détails sur l’implémentation :

  • __STDC_VERSION__ La version de la norme C implémentée. Il s’agit d’un entier constant utilisant le format “aaaammL” (la valeur “201112L” pour C11, la valeur “199901L” pour C99 ; elle n’a pas été définie pour C89/C90)
  • __STDC_HOSTED__ 1 s’il s’agit d’une implémentation hébergée, sinon 0.
  • __STDC__ Si 1, l’implémentation est conforme au standard C.

Autres macros prédéfinies (non obligatoires) #

ISO/IEC 9899:2011 §6.10.9.2 Macros d’environnement :

  • __STDC_ISO_10646__ Une constante entière de la forme aaaammL (par exemple, 199712L). Si ce symbole est défini, alors chaque caractère de l’Unicode l’ensemble requis, lorsqu’il est stocké dans un objet de type wchar_t, a le même valeur comme identifiant court de ce caractère. L’ensemble requis Unicode comprend tous les caractères définis par la norme ISO/IEC 10646, ainsi que toutes les modifications et corrections techniques, à compter de l’année indiquée et mois. Si un autre codage est utilisé, la macro ne doit pas être définie et le codage réel utilisé est défini par l’implémentation.

  • __STDC_MB_MIGHT_NEQ_WC__ La constante entière 1, destinée à indiquer que, dans l’encodage pour wchar_t, un membre du jeu de caractères de base n’a pas besoin avoir une valeur de code égale à sa valeur lorsqu’il est utilisé comme seul caractère dans un constante de caractère entier.

  • __STDC_UTF_16__ La constante entière 1, destinée à indiquer que les valeurs de type char16_t sont encodés en UTF−16. Si un autre codage est utilisé, le La macro ne doit pas être définie et le codage réel utilisé est défini par l’implémentation.

  • __STDC_UTF_32__ La constante entière 1, destinée à indiquer que les valeurs de type char32_t sont encodés en UTF−32. Si un autre codage est utilisé, le La macro ne doit pas être définie et le codage réel utilisé est défini par l’implémentation.

ISO/IEC 9899:2011 §6.10.8.3 Macros de fonctionnalités conditionnelles

  • __STDC_ANALYZABLE__ La constante entière 1, destinée à indiquer la conformité à les spécifications de l’annexe L (Analysabilité).
  • __STDC_IEC_559__ La constante entière 1, destinée à indiquer la conformité au spécifications de l’annexe F (arithmétique à virgule flottante CEI 60559).
  • __STDC_IEC_559_COMPLEX__ La constante entière 1, destinée à indiquer respect des spécifications de l’annexe G (complexe compatible IEC 60559 arithmétique).
  • __STDC_LIB_EXT1__ La constante entière 201112L, destinée à indiquer le support pour les extensions définies dans l’annexe K (Bounds-checking interfaces).
  • __STDC_NO_ATOMICS__ La constante entière 1, destinée à indiquer que le l’implémentation ne prend pas en charge les types atomiques (y compris le _Atomic qualificateur de type) et l’en-tête <stdatomic.h>.
  • __STDC_NO_COMPLEX__ La constante entière 1, destinée à indiquer que le l’implémentation ne prend pas en charge les types complexes ou le <complex.h> entête.
  • __STDC_NO_THREADS__ La constante entière 1, destinée à indiquer que le l’implémentation ne prend pas en charge l’en-tête <threads.h>.
  • __STDC_NO_VLA__ La constante entière 1, destinée à indiquer que le l’implémentation ne prend pas en charge les tableaux de longueur variable ou de manière variable types modifiés.

__cplusplus pour l’utilisation d’externals C dans du code C++ compilé avec C++ - name mangling #

Il y a des moments où un fichier inclus doit générer une sortie différente du préprocesseur selon que le compilateur est un compilateur C ou un compilateur C++ en raison des différences de langage.

Par exemple, une fonction ou un autre externe est défini dans un fichier source C mais est utilisé dans un fichier source C++. Étant donné que C++ utilise la manipulation de noms (ou la décoration de noms) afin de générer des noms de fonction uniques basés sur des types d’arguments de fonction, une déclaration de fonction C utilisée dans un fichier source C++ provoquera des erreurs de lien. Le compilateur C++ modifiera le nom externe spécifié pour la sortie du compilateur à l’aide des règles de gestion des noms pour C++. Le résultat est des erreurs de liaison dues à des éléments externes introuvables lorsque la sortie du compilateur C++ est liée à la sortie du compilateur C.

Étant donné que les compilateurs C ne font pas de manipulation de noms, mais que les compilateurs C++ le font pour toutes les étiquettes externes (noms de fonctions ou noms de variables) générées par le compilateur C++, une macro de préprocesseur prédéfinie, __cplusplus, a été introduite pour permettre la détection du compilateur.

Afin de contourner ce problème de sortie de compilateur incompatible pour les noms externes entre C et C++, la macro __cplusplus est définie dans le préprocesseur C++ et n’est pas définie dans le préprocesseur C. Ce nom de macro peut être utilisé avec la directive conditionnelle du préprocesseur #ifdef ou #if avec l’opérateur defined() pour indiquer si un code source ou un fichier inclus est compilé en C++ ou C.

#ifdef __cplusplus
printf("C++\n");
#else
printf("C\n");
#endif

Ou vous pouvez utiliser

#if defined(__cplusplus)
printf("C++\n");
#else
printf("C\n");
#endif

Afin de spécifier le nom de fonction correct d’une fonction à partir d’un fichier source C compilé avec le compilateur C utilisé dans un fichier source C++, vous pouvez vérifier la constante définie __cplusplus afin de provoquer le extern "C" { /* ... */ }; à utiliser pour déclarer des externes C lorsque le fichier d’en-tête est inclus dans un fichier source C++. Cependant, lorsqu’il est compilé avec un compilateur C, le extern "C" { */ ... */ }; n’est pas utilisé. Cette compilation conditionnelle est nécessaire car extern "C" { /* ... */ }; est valide en C++ mais pas en C.

#ifdef __cplusplus
// if we are being compiled with a C++ compiler then declare the
// following functions as C functions to prevent name mangling.
extern "C" {
#endif

// exported C function list.
int foo (void);

#ifdef __cplusplus
// if this is a C++ compiler, we need to close off the extern declaration.
};
#endif

Macro d’arguments variadiques #

Macros avec arguments variadiques :

Supposons que vous souhaitiez créer une macro d’impression pour déboguer votre code, prenons cette macro comme exemple :

#define debug_print(msg) printf("%s:%d %s", __FILE__, __LINE__, msg)

Quelques exemples d’utilisation :

La fonction somefunc() renvoie -1 en cas d’échec et 0 en cas de succès, et elle est appelée à partir de nombreux endroits différents dans le code :

int retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

/* some other code */

 retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

Que se passe-t-il si l’implémentation de somefunc() change et renvoie maintenant différentes valeurs correspondant à différents types d’erreurs possibles ? Vous voulez toujours utiliser la macro de débogage et imprimer la valeur d’erreur.

debug_printf(retVal);      /* this would obviously fail */
debug_printf("%d",retVal); /* this would also fail */

Pour résoudre ce problème, la macro __VA_ARGS__ a été introduite. Cette macro autorise plusieurs paramètres X-macro :

Exemple:

 #define debug_print(msg, ...) printf(msg, __VA_ARGS__) \
                               printf("\nError occurred in file:line (%s:%d)\n", __FILE__, __LINE)

Usage:

int retVal = somefunc();

debug_print("retVal of somefunc() is-> %d", retVal);

Cette macro vous permet de passer plusieurs paramètres et de les imprimer, mais maintenant elle vous interdit d’envoyer des paramètres du tout.

debug_print("Hey");

Cela soulèverait une erreur de syntaxe car la macro attend au moins un argument de plus et le pré-processeur n’ignorerait pas l’absence de virgule dans la macro debug_print(). De plus, debug_print("Hey",); soulèverait une erreur de syntaxe car vous ne pouvez pas garder l’argument passé à la macro vide.

Pour résoudre ce problème, la macro ##__VA_ARGS__ a été introduite, cette macro indique que si aucun argument de variable n’existe, la virgule est supprimée par le pré-processeur du code.

Exemple:

 #define debug_print(msg, ...) printf(msg, ##__VA_ARGS__) \
                               printf("\nError occured in file:line (%s:%d)\n", __FILE__, __LINE)

Usage:

 debug_print("Ret val of somefunc()?");
 debug_print("%d",somefunc());

Remplacement de macros #

La forme la plus simple de remplacement de macro consiste à définir une “constante manifeste”, comme dans

#define ARRSIZE 100
int array[ARRSIZE];

Cela définit une macro ressemblant à une fonction qui multiplie une variable par “10” et stocke la nouvelle valeur :

#define TIMES10(A) ((A) *= 10)

double b = 34;
int c = 23;

TIMES10(b);   // good: ((b) *= 10);
TIMES10(c);   // good: ((c) *= 10);
TIMES10(5);   // bad:  ((5) *= 10);

Le remplacement est effectué avant toute autre interprétation du texte du programme. Lors du premier appel à TIMES10, le nom A de la définition est remplacé par b et le texte ainsi développé est alors mis à la place de l’appel. Notez que cette définition de “TIMES10” n’est pas équivalente à

#define TIMES10(A) ((A) = (A) * 10)

car cela pourrait évaluer le remplacement de A, deux fois, ce qui peut avoir des effets secondaires indésirables.

Ce qui suit définit une macro de type fonction dont la valeur est le maximum de ses arguments. Il a l’avantage de fonctionner pour tous les types d’arguments compatibles et de générer du code en ligne sans la surcharge de l’appel de fonction. Elle a les inconvénients d’évaluer une seconde fois l’un ou l’autre de ses arguments (y compris les effets de bord) et de générer plus de code qu’une fonction si elle est invoquée plusieurs fois.

#define max(a, b) ((a) > (b) ? (a) : (b))

int maxVal = max(11, 43);              /* 43 */
int maxValExpr = max(11 + 36, 51 - 7); /* 47 */

/* Should not be done, due to expression being evaluated twice */
int j = 0, i = 0;
int sideEffect = max(++i, ++j);       /* i == 4 */

Pour cette raison, ces macros qui évaluent leurs arguments plusieurs fois sont généralement évitées dans le code de production. Depuis C11, il existe la fonctionnalité _Generic qui permet d’éviter de telles invocations multiples.

Les parenthèses abondantes dans les extensions de macro (côté droit de la définition) garantissent que les arguments et l’expression résultante sont correctement liés et s’intègrent bien dans le contexte dans lequel la macro est appelée.

Directive d’erreur #

Si le préprocesseur rencontre une directive #error, la compilation est arrêtée et le message de diagnostic inclus est imprimé.

#define DEBUG

#ifdef DEBUG
#error "Debug Builds Not Supported"
#endif

int main(void) {
    return 0;
}

Sortie possible :

$ gcc error.c
error.c: error: #error "Debug Builds Not Supported"

Implémentation de FOREACH #

Nous pouvons également utiliser des macros pour rendre le code plus facile à lire et à écrire. Par exemple, nous pouvons implémenter des macros pour implémenter la construction “foreach” en C pour certaines structures de données telles que les listes à liaison simple et double, les files d’attente, etc.

Voici un petit exemple.

#include <stdio.h>
#include <stdlib.h>

struct LinkedListNode
{
    int data;
    struct LinkedListNode *next;
};

#define FOREACH_LIST(node, list) \
     for (node=list; node; node=node->next)

/* Usage */
int main(void)
{
    struct LinkedListNode *list, **plist = &list, *node;
    int i;

    for (i=0; i<10; i++)
    {
         *plist = malloc(sizeof(struct LinkedListNode));
         (*plist)->data = i;
         (*plist)->next = NULL;
         plist          = &(*plist)->next;
    }

    /* printing the elements here */
    FOREACH_LIST(node, list)
    {
        printf("%d\n", node->data);
    }
}

Vous pouvez créer une interface standard pour de telles structures de données et écrire une implémentation générique de FOREACH comme :

#include <stdio.h>
#include <stdlib.h>

typedef struct CollectionItem_
{
    int data;
    struct CollectionItem_ *next;
} CollectionItem;

typedef struct Collection_
{
    /* interface functions */
    void* (*first)(void *coll);
    void* (*last) (void *coll);
    void* (*next) (void *coll, CollectionItem *currItem);

    CollectionItem *collectionHead;
    /* Other fields */
} Collection;

/* must implement */
void *first(void *coll)
{
    return ((Collection*)coll)->collectionHead;
}

/* must implement */
void *last(void *coll)
{
    return NULL;
}

/* must implement */
void *next(void *coll, CollectionItem *curr)
{
    return curr->next;
}

CollectionItem *new_CollectionItem(int data)
{
    CollectionItem *item = malloc(sizeof(CollectionItem));
    item->data = data;
    item->next = NULL;
    return item;
}

void Add_Collection(Collection *coll, int data)
{
    CollectionItem **item = &coll->collectionHead;
    while(*item)
        item = &(*item)->next;
    (*item) = new_CollectionItem(data);
}

Collection *new_Collection()
{
    Collection *nc = malloc(sizeof(Collection));
    nc->first = first;
    nc->last  = last;
    nc->next  = next;
    return nc;
}

/* generic implementation */
#define FOREACH(node, collection)                      \
    for (node  = (collection)->first(collection);      \
         node != (collection)->last(collection);       \
         node  = (collection)->next(collection, node))

int main(void)
{
    Collection *coll = new_Collection();
    CollectionItem *node;
    int i;

    for(i=0; i<10; i++)
    {
         Add_Collection(coll, i);
    }

    /* printing the elements here */
    FOREACH(node, coll)
    {
        printf("%d\n", node->data);
    }
}

Pour utiliser cette implémentation générique, implémentez simplement ces fonctions pour votre structure de données.

1.  void* (*first)(void *coll);
2.  void* (*last) (void *coll);
3.  void* (*next) (void *coll, CollectionItem *currItem);
