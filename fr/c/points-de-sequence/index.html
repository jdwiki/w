Norme internationale ISO/IEC 9899:201x Langages de programmation — C

Accéder à un objet volatile, modifier un objet, modifier un fichier, ou appeler une fonction qui effectue l’une de ces opérations sont toutes * côté les effets*, qui sont des changements d’état de l’environnement d’exécution.

La présence d’un point de séquence entre l’évaluation des expressions A et B impliquent que chaque calcul de valeur et effet secondaire associé à A est séquencé avant chaque calcul de valeur et côté effet associé à B.

Voici la liste complète des points de séquence de l’annexe C du [projet de prépublication en ligne 2011] (http://www.open-std.org/jtc1/sc22/wg14/www/docs/n1570.pdf) du Norme de langage C :

Points de séquence

1     Les points suivants sont les points de séquence décrits en 5.1.2.3 :

  • Entre les évaluations du désignateur de fonction et les arguments réels dans une fonction appel et l'appel réel. (6.5.2.2).
  • Entre les évaluations des premier et deuxième opérandes des opérateurs suivants : ET logique && (6.5.13) ; OU logique || (6.5.14) ; virgule , (6.5.17).
  • Entre les évaluations du premier opérande du conditionnel ? : opérateur et celui des deuxième et troisième opérandes qui est évalué (6.5.15).
  • La fin d'un déclarant complet : déclarateurs (6.7.6) ;
  • Entre l'évaluation d'une expression complète et la prochaine expression complète à évalué. Les expressions suivantes sont complètes : un initialiseur qui ne fait pas partie d'un littéral composé (6.7.9); l'expression dans une instruction d'expression (6.8.3); la contrôler l'expression d'une instruction de sélection (if ou switch) (6.8.4) ; la contrôler l'expression d'une instruction while ou do (6.8.5) ; chacun des (facultatif) expressions d'une instruction for (6.8.5.3) ; l'expression (facultative) dans un retour déclaration (6.8.6.4).
  • Immédiatement avant le retour d'une fonction de bibliothèque (7.1.4).
  • Après les actions associées à chaque conversion de fonction entrée/sortie formatée spécificateur (7.21.6, 7.29.2).
  • Immédiatement avant et immédiatement après chaque appel à une fonction de comparaison, et également entre tout appel à une fonction de comparaison et tout déplacement des objets passés comme arguments à cet appel (7.22.5).

Expressions non séquencées #

Les expressions suivantes sont non séquencées :

a + b;
a - b;
a * b;
a / b;
a % b;
a & b;
a | b;

Dans les exemples ci-dessus, l’expression ‘a’ peut être évaluée avant ou après l’expression ‘b’, ‘b’ peut être évaluée avant ‘a’, voire être mélangées si elles correspondent à plusieurs instructions.

Une règle similaire s’applique aux appels de fonction :

f(a, b);

Ici, non seulement a et b ne sont pas séquencés (c’est-à-dire que l’opérateur , dans un appel de fonction ne produit pas de point de séquence), mais également f, l’expression qui détermine la fonction à appeler.

Les effets secondaires peuvent être appliqués immédiatement après l’évaluation ou différés jusqu’à un point ultérieur.

Des expressions comme

x++ & x++;
f(x++, x++); /* the ',' in a function call is *not* the same as the comma operator */
x++ * x++;
a[i] = i++;

ou

x++ & x;
f(x++, x);
x++ * x;
a[i++] = i;

donnera un comportement indéfini parce que

  • une modification d’un objet et tout autre accès à celui-ci doivent être séquencés
  • l’ordre d’évaluation et l’ordre d’application des effets secondaires1 ne sont pas précisés.

1 Toute modification de l’état de l’environnement d’exécution.

Expressions séquencées #

Les expressions suivantes sont séquencées :

a && b
a || b
a , b
a ? b : c
for ( a ; b ; c ) { ... }

Dans tous les cas, l’expression a est entièrement évaluée et tous les effets secondaires sont appliqués avant que b ou c ne soient évalués. Dans le quatrième cas, un seul parmi ‘b’ ou ‘c’ sera évalué. Dans le dernier cas, « b » est entièrement évalué et tous les effets secondaires sont appliqués avant que « c » ne soit évalué.

Dans tous les cas, l’évaluation de l’expression a est séquencée avant les évaluations de b ou c (alternativement, les évaluations de b et c sont séquencées après l’évaluation de a ).

Ainsi, des expressions comme

x++ && x++
x++ ? x++ : y++ 
(x = f()) && x != 0
for ( x = 0; x < 10; x++ ) { ... }
y = (x++, x++);

avoir un comportement bien défini.

Expressions séquencées de manière indéterminée #

Les appels de fonction comme f(a) impliquent toujours un point de séquence entre l’évaluation des arguments et le désignateur (ici f et a) et l’appel réel. Si deux de ces appels ne sont pas séquencés, les deux appels de fonction sont séquencés de manière indéterminée, c’est-à-dire que l’un est exécuté avant l’autre et l’ordre n’est pas spécifié.

unsigned counter = 0;

unsingned account(void) {
   return counter++;
}

int main(void) {
   printf("the order is %u %u\n", account(), account());
}

Cette double modification implicite de counter lors de l’évaluation des arguments printf est valide, nous ne savons simplement pas lequel des appels vient en premier. Comme la commande n’est pas spécifiée, elle peut varier et ne peut pas être invoquée. Ainsi, l’impression pourrait être :

la commande est 0 1

ou

la commande est 1 0

L’instruction analogue à la précédente sans appel de fonction intermédiaire

   printf("the order is %u %u\n", counter++, counter++); // undefined behavior

a un comportement indéfini car il n’y a pas de point de séquence entre les deux modifications de counter.
