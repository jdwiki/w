Créer et inclure des fichiers d'en-tête

Dans le C moderne, les fichiers d’en-tête sont des outils cruciaux qui doivent être conçus et utilisés correctement. Ils permettent au compilateur de recouper les parties compilées indépendamment d’un programme.

Les en-têtes déclarent les types, les fonctions, les macros, etc. qui sont nécessaires aux consommateurs d’un ensemble d’installations. Tout le code qui utilise l’une de ces fonctionnalités inclut l’en-tête. Tout le code qui définit ces installations inclut l’en-tête. Cela permet au compilateur de vérifier que les utilisations et les définitions correspondent.

Introduction #

Il existe un certain nombre de directives à suivre lors de la création et de l’utilisation fichiers d’en-tête dans un projet C :

  • Idémopotence

Si un fichier d’en-tête est inclus plusieurs fois dans une unité de traduction (TU), il ne doit pas casser les builds.

  • Auto-confinement

Si vous avez besoin des facilités déclarées dans un fichier d’en-tête, vous ne devez pas doivent inclure tous les autres en-têtes explicitement.

  • Minimalité

Vous ne devriez pas être en mesure de supprimer des informations d’un en-tête sans entraînant l’échec des builds.

  • Inclure ce que vous utilisez (IWYU)

Plus préoccupant en C++ qu’en C, mais néanmoins important en C aussi. Si le code d’une TU (appelez-le code.c) utilise directement les fonctionnalités déclaré par un en-tête (appelez-le "headerA.h"), alors code.c devrait #include "headerA.h" directement, même si la TU inclut une autre header (appelez-le "headerB.h") qui, pour le moment, inclut "headerA.h".

Parfois, il peut y avoir des raisons suffisantes pour casser un ou plusieurs de ces directives, mais vous devez tous les deux être conscients que vous enfreignez règle et soyez conscient des conséquences de le faire avant de l’enfreindre.

Auto-confinement #

Les en-têtes modernes doivent être autonomes, ce qui signifie qu’un programme qui a besoin d’utiliser les fonctionnalités définies par header.h peut inclure cet en-tête (#include "header.h") et ne pas se soucier de savoir si d’autres en-têtes doivent être inclus en premier.

Recommandation : les fichiers d’en-tête doivent être autonomes. #

Règles historiques #

Historiquement, cela a été un sujet légèrement controversé.

Il était une fois un autre millénaire, le [AT&T Indian Hill C Style and Coding Standards] (http://www.literateprogramming.com/indhill-annot.pdf) a déclaré :

Les fichiers d’en-tête ne doivent pas être imbriqués. Le prologue d’un fichier d’en-tête doit donc décrire les autres les en-têtes doivent être #included pour que l’en-tête soit fonctionnel. Dans les cas extrêmes, où un grand nombre de fichiers d’en-tête doivent être inclus dans plusieurs fichiers source différents, il est acceptable de mettre tous les #include courants dans un seul fichier d’inclusion.

C’est l’antithèse de l’autonomie.

Règles modernes #

Cependant, depuis lors, l’opinion a tendu dans la direction opposée. Si un fichier source a besoin d’utiliser les facilités déclarées par un en-tête header.h, le programmeur doit pouvoir écrire :

#include "header.h"

et (sous réserve uniquement d’avoir les chemins de recherche corrects définis sur la commande ligne), tous les en-têtes de prérequis nécessaires seront inclus par header.h sans avoir besoin d’autres en-têtes ajoutés au fichier source.

Cela offre une meilleure modularité pour le code source. Il protège également la source de l’énigme “devinez pourquoi cet en-tête a été ajouté” qui survient après que le code a été modifié et piraté pendant une décennie ou deux.

Les normes de codage [NASA Goddard Space Flight Center (GSFC) pour C] (http://stackoverflow.com/questions/1804486/should-i-use-include-in-headers) est l’une des normes les plus modernes — mais est maintenant un peu difficile à traquer. Il indique que les en-têtes doivent être autonomes. Il fournit également un moyen simple de s’assurer que les en-têtes sont autonomes : le fichier d’implémentation de l’en-tête doit inclure l’en-tête comme premier entête. S’il n’est pas autonome, ce code ne se compilera pas.

La justification donnée par GSFC comprend :

§2.1.1 L’en-tête inclut la justification

Cette norme exige que l’en-tête d’une unité contienne #include instructions pour tous les autres en-têtes requis par l’en-tête d’unité. Placer #include pour l’en-tête de l’unité en premier dans le corps de l’unité permet le compilateur pour vérifier que l’en-tête contient tous les éléments requis les instructions #include.

Une autre conception, non autorisée par cette norme, ne permet Instructions #include dans les en-têtes ; tous les #includes sont faits dans le corps des dossiers. Les fichiers d’en-tête d’unité doivent alors contenir des instructions #ifdef qui vérifient que les en-têtes requis sont inclus dans le bon ordre.

L’un des avantages de la conception alternative est que la liste “#include” dans le fichier corps est exactement la liste de dépendances nécessaire dans un makefile, et cette liste est vérifiée par le compilateur. Avec la conception standard, un outil doit être utilisé pour générer le liste de dépendance. Cependant, tous les environnements de développement recommandés par la branche fournir un tel outil.

Un inconvénient majeur de la conception alternative est que si le changements de liste d’en-tête requis, chaque fichier qui utilise cette unité doit être modifié pour mettre à jour la liste des instructions #include. De plus, la liste d’en-tête requise pour une unité de bibliothèque de compilateur peut être différent sur différentes cibles.

Un autre inconvénient de la conception alternative est que la bibliothèque du compilateur les fichiers d’en-tête et d’autres fichiers tiers doivent être modifiés pour ajouter le instructions #ifdef requises.

Ainsi, l’autonomie signifie que :

  • Si un en-tête header.h a besoin d’un nouvel en-tête imbriqué extra.h, vous ne devez vérifier chaque fichier source qui utilise header.h pour voir si vous devez ajouter extra.h.
  • Si un en-tête header.h n’a plus besoin d’inclure un en-tête spécifique notneeded.h, vous n’avez pas à vérifier chaque fichier source qui utilise header.h pour voir si vous pouvez supprimer en toute sécurité notneeded.h (mais voir [Inclure ce que vous utilisez](https://www.wikiod.com/fr/c/creer-et-inclure-des-fichiers-den-tete /21641/include-ce-que-vous-utilisez-iwyu).
  • Il n’est pas nécessaire d’établir la séquence correcte pour inclure le en-têtes pré-requis (qui nécessite un tri topologique pour faire le travail correctement).

Vérification de l’autonomie #

Voir [Lien avec un statique bibliothèque] (http://stackoverflow.com/questions/4507896) pour un script chkhdr qui peut être utilisé pour tester l’idempotence et l’auto-confinement de un fichier d’en-tête.

Minimalité #

Les en-têtes sont un mécanisme de vérification de cohérence crucial, mais ils doivent être aussi petit que possible. En particulier, cela signifie qu’un en-tête ne doit pas inclure d’autres en-têtes simplement parce que le fichier d’implémentation aura besoin des autres en-têtes. Un en-tête ne doit contenir que les en-têtes nécessaires pour un consommateur de les prestations décrites.

Par exemple, un en-tête de projet ne doit pas inclure <stdio.h> à moins qu’un des interfaces de la fonction utilise le type FILE * (ou l’un des autres types définis uniquement dans <stdio.h>). Si une interface utilise size_t, le plus petit en-tête suffisant est <stddef.h>. Évidemment, si un autre en-tête définissant size_t est inclus, il y a pas besoin d’inclure <stddef.h> aussi.

Si les en-têtes sont minimes, il conserve le temps de compilation à un minimum aussi.

Il est possible de concevoir des en-têtes dont le seul but est d’inclure un lot d’autres en-têtes. Celles-ci s’avèrent rarement être une bonne idée à long terme car peu les fichiers sources auront en fait besoin de toutes les fonctionnalités décrites par tous les en-têtes. Par exemple, un <standard-c.h> pourrait être conçu qui inclut tous les en-têtes C standard — avec précaution car certains en-têtes ne sont pas toujours cadeau. Cependant, très peu de programmes utilisent réellement les fonctionnalités de <locale.h> ou <tgmath.h>.

  • Voir aussi [Comment lier plusieurs fichiers d’implémentation dans C ?] (http://stackoverflow.com/questions/15622409)

Notation et Divers #

La norme C dit qu’il y a très peu de différence entre le Notations #include <header.h> et #include "header.h".

[#include <header.h>] recherche une séquence d’éléments définis par l’implémentation emplacements pour un en-tête identifié de manière unique par la séquence spécifiée entre les délimiteurs < et >, et provoque le remplacement de ce directive par tout le contenu de l’en-tête. Comment les lieux sont spécifiés ou l’en-tête identifié est défini par la mise en œuvre.

[#include "header.h"] provoque le remplacement de cette directive par l’intégralité du contenu du fichier source identifié par le séquence entre les délimiteurs "…". Le fichier source nommé est recherché dans un fichier défini par l’implémentation manière. Si cette recherche n’est pas prise en charge, ou si la recherche échoue, la directive est retraité comme s’il lisait [#include <header.h>] …

Ainsi, la forme entre guillemets doubles peut apparaître à plus d’endroits que la forme entre parenthèses. La norme spécifie par exemple que les en-têtes standard doivent être inclus entre crochets, même si la compilation fonctionne si vous utilisez guillemets doubles à la place. De même, des normes telles que POSIX utilisent le format entre crochets — et vous devriez aussi. Réservez les en-têtes entre guillemets pour les en-têtes définis par le projet. Pour les en-têtes définis en externe (y compris les en-têtes d’autres projets sur lequel repose votre projet), la notation entre crochets est la plus appropriée.

Notez qu’il doit y avoir un espace entre #include et l’en-tête, même si les compilateurs n’accepteront aucun espace à cet endroit. Les espaces sont bon marché.

Un certain nombre de projets utilisent une notation telle que :

#include <openssl/ssl.h>
#include <sys/stat.h>
#include <linux/kernel.h>

Vous devez déterminer si vous souhaitez utiliser ce contrôle d’espace de noms dans votre projet (c’est très probablement une bonne idée). Vous devez éviter les noms utilisés par les projets existants (en particulier, sys et linux seraient de mauvais choix).

Si vous l’utilisez, votre code doit être prudent et cohérent dans l’utilisation de la notation.

N’utilisez pas la notation #include "../include/header.h".

Les fichiers d’en-tête doivent rarement, voire jamais, définir des variables. Bien que vous garderez les variables globales au minimum, si vous avez besoin d’un variable globale, vous la déclarerez dans un en-tête, et la définirez dans une fichier source approprié, et ce fichier source inclura l’en-tête à recouper la déclaration et la définition, ainsi que tous les fichiers source qui utiliser la variable utilisera l’en-tête pour la déclarer.

Corollaire : vous ne déclarerez pas de variables globales dans un fichier source — un le fichier source ne contiendra que des définitions.

Les fichiers d’en-tête doivent rarement déclarer des fonctions “statiques”, avec le notable exception des fonctions static inline qui seront définies dans les en-têtes si la fonction est nécessaire dans plus d’un fichier source.

  • Les fichiers source définissent les variables globales et les fonctions globales.
  • Les fichiers sources ne déclarent pas l’existence de variables globales ou de fonctions ; ils incluent l’en-tête qui déclare la variable ou la fonction.
  • Les fichiers d’en-tête déclarent la variable globale et les fonctions (ainsi que les types et autres éléments de support).
  • Les fichiers d’en-tête ne définissent pas de variables ni de fonctions, à l’exception des fonctions (statiques) inline.

Idempotence #

Si un fichier d’en-tête particulier est inclus plus d’une fois dans une unité de traduction (TU), il ne devrait pas y avoir de problèmes de compilation. C’est ce qu’on appelle « l’idempotence » ; vos en-têtes doivent être idempotents. Imaginez à quel point la vie serait difficile si vous deviez vous assurer que #include <stdio.h> n’était inclus qu’une seule fois.

Il existe deux façons d’obtenir l’idempotence : les gardes d’en-tête et la directive “#pragma once”.

Protecteurs d’en-tête #

Les protections de linteau sont simples et fiables et conformes à la norme C. Les premières lignes sans commentaire dans un fichier d’en-tête doivent être de la forme :

#ifndef UNIQUE_ID_FOR_HEADER
#define UNIQUE_ID_FOR_HEADER

La dernière ligne sans commentaire doit être #endif, éventuellement suivi d’un commentaire :

#endif /* UNIQUE_ID_FOR_HEADER */

Tout le code opérationnel, y compris les autres directives “#include”, doit être entre ces lignes.

Chaque nom doit être unique. Souvent, un schéma de nom tel que HEADER_H_INCLUDED est utilisé. Certains codes plus anciens utilisent un symbole défini comme garde d’en-tête (par exemple, #ifndef BUFSIZ dans <stdio.h>), mais il n’est pas aussi fiable qu’un nom unique.

Une option consisterait à utiliser un hachage MD5 (ou autre) généré pour le nom de la protection d’en-tête. Vous devez éviter d’émuler les schémas utilisés par les en-têtes système qui utilisent fréquemment des noms réservés à l’implémentation — noms commençant par un trait de soulignement suivi soit d’un autre trait de soulignement, soit d’une lettre majuscule.

La directive “#pragma once” #

Alternativement, certains compilateurs prennent en charge la directive #pragma once qui a le même effet que les trois lignes affichées pour les gardes d’en-tête.

#pragma once

Les compilateurs qui supportent #pragma once incluent MS Visual Studio et GCC et Clang. Cependant, si la portabilité est un problème, il est préférable d’utiliser des protections d’en-tête ou d’utiliser les deux. Les compilateurs modernes (ceux prenant en charge C89 ou ultérieur) sont tenus d’ignorer, sans commentaire, les pragmas qu’ils ne reconnaissent pas (“Tout pragma de ce type qui n’est pas reconnu par l’implémentation est ignoré”), mais les anciennes versions de GCC n’étaient pas si indulgentes.

Inclure ce que vous utilisez (IWYU) #

Le projet Include What You Use de Google, ou IWYU, garantit que les fichiers source incluent tous les en-têtes utilisés dans le code.

Supposons qu’un fichier source source.c inclue un en-tête arbitrary.h qui à son tour inclut par coïncidence freeloader.h, mais que le fichier source utilise également explicitement et indépendamment les fonctionnalités de freeloader.h. Tout va bien pour commencer. Puis, un jour, arbitrary.h est modifié afin que ses clients n’aient plus besoin des fonctionnalités de freeloader.h. Soudainement, source.c s’arrête de compiler - parce qu’il ne répondait pas aux critères IWYU. Parce que le code dans source.c utilisait explicitement les fonctionnalités de freeloader.h, il aurait dû inclure ce qu’il utilise — il aurait dû y avoir un #include "freeloader.h" explicite dans la source aussi. (Idempotency aurait assuré qu’il n’y avait pas de problème.)

La philosophie IWYU maximise la probabilité que le code continue à se compiler même avec des modifications raisonnables apportées aux interfaces. De toute évidence, si votre code appelle une fonction qui est ensuite supprimée de l’interface publiée, aucune préparation ne peut empêcher les modifications de devenir nécessaires. C’est pourquoi les modifications apportées aux API sont évitées dans la mesure du possible, et pourquoi il existe des cycles d’obsolescence sur plusieurs versions, etc.

C’est un problème particulier en C++ car les en-têtes standard sont autorisés à s’inclure les uns les autres. Le fichier source file.cpp peut inclure un en-tête header1.h qui, sur une plate-forme, inclut un autre en-tête header2.h. Il se peut que file.cpp utilise également les fonctionnalités de header2.h. Cela ne poserait pas de problème au départ - le code se compilerait car header1.h inclut header2.h. Sur une autre plate-forme, ou une mise à niveau de la plate-forme actuelle, header1.h pourrait être révisé afin qu’il n’inclue plus header2.h, et alors file.cpp cesserait de se compiler en conséquence.

IWYU repérerait le problème et recommanderait que header2.h soit inclus directement dans file.cpp. Cela garantirait qu’il continue à compiler. Des considérations analogues s’appliquent également au code C.
