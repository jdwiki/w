Rembourrage et emballage de la structure

  • Structures d’emballage
  • Rembourrage structuré

Sur cette page

  • Structures d’emballage
  • Rembourrage structuré

Par défaut, les compilateurs C disposent de structures permettant d’accéder rapidement à chaque membre, sans encourir de pénalités pour un accès non aligné, un problème avec les machines RISC telles que le DEC Alpha et certains processeurs ARM.

Selon l’architecture du processeur et le compilateur, une structure peut occuper plus d’espace en mémoire que la somme des tailles de ses membres composants. Le compilateur peut ajouter un remplissage entre les membres ou à la fin de la structure, mais pas au début.

Le compactage remplace le remplissage par défaut.

Eric Raymond a un article sur The Lost Art of C Structure Packing qui est une lecture utile.

Structures d’emballage #

Par défaut, les structures sont rembourrées en C. Si vous voulez éviter ce comportement, vous devez le demander explicitement. Sous GCC, c’est __attribute__((__packed__)). Considérez cet exemple sur une machine 64 bits :

struct foo {
    char *p;  /* 8 bytes */
    char c;   /* 1 byte  */
    long x;   /* 8 bytes */
};

La structure sera automatiquement remplie pour avoir un alignement de “8 octets” et ressemblera à ceci :

struct foo {
    char *p;     /* 8 bytes */
    char c;      /* 1 byte  */

    char pad[7]; /* 7 bytes added by compiler */

    long x;      /* 8 bytes */
};

Donc sizeof(struct foo) nous donnera 24 au lieu de 17. Cela s’est produit à cause d’un compilateur 64 bits en lecture/écriture de/vers la mémoire dans 8 octets de mot à chaque étape et évident lorsque vous essayez d’écrire char c; un octet en mémoire un 8 octets complets (c’est-à-dire un mot) récupéré et consomme seul le premier octet de celui-ci et ses sept octets successifs restent vides et inaccessibles pour toute opération de lecture et d’écriture pour le remplissage de la structure.

Emballage structuré #

Mais si vous ajoutez l’attribut packed, le compilateur n’ajoutera pas de remplissage :

struct __attribute__((__packed__)) foo {
    char *p;  /* 8 bytes */
    char c;   /* 1 byte  */
    long x;   /* 8 bytes */
};

Maintenant, sizeof(struct foo) renverra 17.

Des structures généralement emballées sont utilisées :

  • Pour économiser de l’espace.
  • Formater une structure de données à transmettre sur le réseau sans dépendre de l’alignement de chaque architecture de chaque nœud du réseau.

Il faut tenir compte du fait que certains processeurs tels que l’ARM Cortex-M0 n’autorisent pas l’accès à la mémoire non alignée ; dans de tels cas, le compactage de la structure peut conduire à un comportement indéfini et peut planter le CPU.

Rembourrage structuré #

Supposons que cette struct soit définie et compilée avec un compilateur 32 bits :

struct test_32 {
    int a;      // 4 byte
    short b;    // 2 byte
    int c;      // 4 byte    
} str_32;

On pourrait s’attendre à ce que cette struct n’occupe que 10 octets de mémoire, mais en imprimant sizeof(str_32) nous voyons qu’il utilise 12 octets.

Cela s’est produit parce que le compilateur aligne les variables pour un accès rapide. Un modèle courant est que lorsque le type de base occupe N octets (où N est une puissance de 2 telle que 1, 2, 4, 8, 16 - et rarement plus grand), la variable doit être alignée sur une limite de N octets ( un multiple de N octets).

Pour la structure illustrée avec sizeof(int) == 4 et sizeof(short) == 2, une mise en page courante est :

  • int a; stocké à l’offset 0 ; taille 4.
  • short b; stocké à l’offset 4 ; taille 2.
  • remplissage sans nom à l’offset 6 ; taille 2.
  • int c; stocké à l’offset 8 ; taille 4.

Ainsi struct test_32 occupe 12 octets de mémoire. Dans cet exemple, il n’y a pas de remplissage de fin.

Le compilateur s’assurera que toutes les variables struct test_32 sont stockées à partir d’une limite de 4 octets, de sorte que les membres de la structure soient correctement alignés pour un accès rapide. Les fonctions d’allocation de mémoire telles que malloc(), calloc() et realloc() sont nécessaires pour s’assurer que le pointeur renvoyé est suffisamment bien aligné pour être utilisé avec n’importe quel type de données, de sorte que les structures allouées dynamiquement seront également correctement alignées .

Vous pouvez vous retrouver avec des situations étranges telles que sur un processeur Intel x86_64 64 bits (par exemple, Intel Core i7 - un Mac exécutant macOS Sierra ou Mac OS X), où lors de la compilation en mode 32 bits, les compilateurs placent “double” aligné sur une limite de 4 octets ; mais, sur le même matériel, lors de la compilation en mode 64 bits, les compilateurs placent “double” aligné sur une limite de 8 octets.
