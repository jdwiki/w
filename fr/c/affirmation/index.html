Une assertion est un prédicat selon lequel la condition présentée doit être vraie au moment où l’assertion est rencontrée par le logiciel. Les plus courantes sont les assertions simples, qui sont validées au moment de l’exécution. Cependant, les assertions statiques sont vérifiées au moment de la compilation.

Syntaxe #

  • affirmer (expression)
  • static_assert(expression, message)
  • _Static_assert(expression, message)

Paramètres #

Paramètre Détails
expression expression de type scalaire.
message Littéral de chaîne à inclure dans le message de diagnostic.

assert et static_assert sont des macros définies dans assert.h.

La définition de assert dépend de la macro NDEBUG qui n’est pas définie par la bibliothèque standard. Si NDEBUG est défini, assert est un no-op :

#ifdef NDEBUG
#  define assert(condition) ((void) 0)
#else
#  define assert(condition) /* implementation defined */
#endif

Les opinions varient quant à savoir si NDEBUG doit toujours être utilisé pour les compilations de production.

  • Le pro-camp soutient que les appels assert abort et les messages d’assertion ne sont pas utiles pour les utilisateurs finaux, de sorte que le résultat n’est pas utile à l’utilisateur. Si vous avez des conditions fatales à vérifier dans le code de production, vous devez utiliser des conditions if/else ordinaires et exit ou quick_exit pour terminer le programme. Contrairement à abort, ceux-ci permettent au programme de faire un peu de nettoyage (via des fonctions enregistrées avec atexit ou at_quick_exit).
  • L’escroc soutient que les appels assert ne devraient jamais se déclencher dans le code de production, mais s’ils le font, la condition qui est vérifiée signifie qu’il y a quelque chose de dramatiquement faux et que le programme se comportera mal pire si l’exécution continue. Par conséquent, il est préférable d’avoir les assertions actives dans le code de production car si elles se déclenchent, l’enfer s’est déjà déchaîné.
  • Une autre option consiste à utiliser un système maison d’assertions qui effectuent toujours la vérification mais gèrent les erreurs différemment entre le développement (où “abandonner” est approprié) et la production (où une “erreur interne inattendue - veuillez contacter le support technique” peut être plus approprié).

static_assert se développe en _Static_assert qui est un mot-clé. La condition est vérifiée au moment de la compilation, donc condition doit être une expression constante. Il n’est pas nécessaire que cela soit traité différemment entre le développement et la production.

Assertion simple #

Une assertion est une déclaration utilisée pour affirmer qu’un fait doit être vrai lorsque cette ligne de code est atteinte. Les assertions sont utiles pour s’assurer que les conditions attendues sont remplies. Lorsque la condition transmise à une assertion est vraie, il n’y a pas d’action. Le comportement sur les conditions fausses dépend des indicateurs du compilateur. Lorsque les assertions sont activées, une fausse entrée provoque un arrêt immédiat du programme. Lorsqu’ils sont désactivés, aucune action n’est entreprise. Il est courant d’activer les assertions dans les versions internes et de débogage, et de les désactiver dans les versions de version, bien que les assertions soient souvent activées dans la version. (Que la terminaison soit meilleure ou pire que les erreurs dépend du programme.) Les assertions ne doivent être utilisées que pour détecter les erreurs de programmation internes, ce qui signifie généralement que des paramètres erronés sont transmis.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int main(void)
{
    int x = -1;
    assert(x >= 0);

    printf("x = %d\n", x);   
    return 0;
}

Sortie possible avec NDEBUG non défini :

a.out: main.c:9: main: Assertion `x >= 0' failed.

Sortie possible avec NDEBUG défini :

x = -1

Il est recommandé de définir NDEBUG globalement, afin de pouvoir facilement compiler votre code avec toutes les assertions activées ou désactivées. Un moyen simple de le faire est de définir NDEBUG comme une option du compilateur, ou de le définir dans un en-tête de configuration partagé (par exemple, config.h).

Assertion statique #

Les assertions statiques sont utilisées pour vérifier si une condition est vraie lorsque le code est compilé. Si ce n’est pas le cas, le compilateur doit émettre un message d’erreur et arrêter le processus de compilation.

Une assertion statique est une assertion qui est vérifiée au moment de la compilation, pas au moment de l’exécution. La condition doit être une expression constante, et si elle est fausse, cela entraînera une erreur du compilateur. Le premier argument, la condition vérifiée, doit être une expression constante et le second un littéral de chaîne.

Contrairement à assert, _Static_assert est un mot-clé. Une macro pratique static_assert est définie dans <assert.h>.

#include <assert.h>

enum {N = 5};
_Static_assert(N == 5, "N does not equal 5");
static_assert(N > 10, "N is not greater than 10");  /* compiler error */

Avant C11, il n’y avait pas de prise en charge directe des assertions statiques. Cependant, dans C99, les assertions statiques pouvaient être émulées avec des macros qui déclencheraient un échec de compilation si la condition de temps de compilation était fausse. Contrairement à _Static_assert, le deuxième paramètre doit être un nom de jeton approprié afin qu’un nom de variable puisse être créé avec lui. Si l’assertion échoue, le nom de la variable est vu dans l’erreur du compilateur, car cette variable a été utilisée dans une déclaration de tableau syntaxiquement incorrecte.

#define STATIC_MSG(msg, l) STATIC_MSG2(msg, l)
#define STATIC_MSG2(msg,l) on_line_##l##__##msg
#define STATIC_ASSERT(x, msg) extern char STATIC_MSG(msg, __LINE__) [(x)?1:-1]
 
enum { N = 5 };
STATIC_ASSERT(N == 5, N_must_equal_5);
STATIC_ASSERT(N > 5, N_must_be_greater_than_5); /* compile error */

Avant C99, vous ne pouviez pas déclarer de variables à des emplacements arbitraires dans un bloc, vous deviez donc être extrêmement prudent lors de l’utilisation de cette macro, en vous assurant qu’elle n’apparaît que là où une déclaration de variable serait valide.

Messages d’erreur d’affirmation #

Il existe une astuce qui peut afficher un message d’erreur avec une assertion. Normalement, vous écririez un code comme celui-ci

void f(void *p)
{
    assert(p != NULL);
    /* more code */
}

Si l’assertion échouait, un message d’erreur ressemblerait à

Échec de l’assertion : p != NULL, fichier main.c, ligne 5

Cependant, vous pouvez également utiliser le ET logique (&&) pour donner un message d’erreur

void f(void *p)
{
    assert(p != NULL && "function f: p cannot be NULL");
    /* more code */
}

Maintenant, si l’assertion échoue, un message d’erreur lira quelque chose comme ceci

Échec de l’assertion : p != NULL && “fonction f : p ne peut pas être NULL”, fichier main.c, ligne 5

La raison pour laquelle cela fonctionne est qu’un littéral de chaîne est toujours évalué comme non nul (vrai). L’ajout de && 1 à une expression booléenne n’a aucun effet. Ainsi, l’ajout de && "error message" n’a aucun effet non plus, sauf que le compilateur affichera l’intégralité de l’expression qui a échoué.

Assertion de code inaccessible #

Au cours du développement, lorsque certains chemins de code doivent être empêchés d’atteindre le flux de contrôle, vous pouvez utiliser assert(0) pour indiquer qu’une telle condition est erronée :

switch (color) {
    case COLOR_RED:
    case COLOR_GREEN:
    case COLOR_BLUE:
        break;

    default:
        assert(0);
}

Chaque fois que l’argument de la macro assert() est faux, la macro écrira des informations de diagnostic dans le flux d’erreurs standard, puis abandonnera le programme. Ces informations incluent le fichier et le numéro de ligne de l’instruction assert() et peuvent être très utiles pour le débogage. Les assertions peuvent être désactivées en définissant la macro NDEBUG.

Une autre façon de terminer un programme lorsqu’une erreur se produit consiste à utiliser les fonctions de bibliothèque standard exit, quick_exit ou abort. exit et quick_exit prennent un argument qui peut être renvoyé à votre environnement. abort() (et donc assert) peut être une fin très sévère de votre programme, et certains nettoyages qui seraient autrement effectués à la fin de l’exécution, peuvent ne pas être effectués.

Le principal avantage de assert() est qu’il affiche automatiquement les informations de débogage. L’appel à abort() a l’avantage de ne pas pouvoir être désactivé comme une assertion, mais il ne peut pas provoquer l’affichage d’informations de débogage. Dans certaines situations, l’utilisation simultanée des deux constructions peut être bénéfique :

if (color == COLOR_RED || color == COLOR_GREEN) {
   ...
} else if (color == COLOR_BLUE) {
   ...
} else {
   assert(0), abort();
}

Lorsque les assertions sont activées, l’appel assert() affichera les informations de débogage et terminera le programme. L’exécution n’atteint jamais l’appel abort(). Lorsque les assertions sont désactivées, l’appel assert() ne fait rien et abort() est appelé. Cela garantit que le programme se termine toujours pour cette condition d’erreur ; l’activation et la désactivation des assertions n’affectent que l’affichage ou non de la sortie de débogage.

Vous ne devriez jamais laisser un tel assert dans le code de production, car les informations de débogage ne sont pas utiles pour les utilisateurs finaux et parce que abort est généralement une terminaison beaucoup trop sévère qui inhibe les gestionnaires de nettoyage installés pour exit ou quick_exit courir.

Précondition et Postcondition #

Un cas d’utilisation de l’assertion est la précondition et la postcondition. Cela peut être très utile pour maintenir [invariant][1] et [design by contract][2]. Par exemple, une longueur est toujours nulle ou positive, donc cette fonction doit renvoyer une valeur nulle ou positive.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int length2 (int *a, int count)
{
    int i, result = 0;

    /* Precondition: */
    /* NULL is an invalid vector */
    assert (a != NULL);
    /* Number of dimensions can not be negative.*/ 
    assert (count >= 0);

    /* Calculation */
    for (i = 0; i < count; ++i) 
    {
        result = result + (a[i] * a[i]);
    }

    /* Postcondition: */
    /* Resulting length can not be negative. */
    assert (result >= 0);
    return result;
}

#define COUNT 3

int main (void)
{
    int a[COUNT] = {1, 2, 3};
    int *b = NULL;
    int r;
    r = length2 (a, COUNT);
    printf ("r = %i\n", r);
    r = length2 (b, COUNT);
    printf ("r = %i\n", r);
    return 0;
}

[1] : https://en.wikipedia.org/wiki/Invariant_(science_informatique) [2] : https://en.wikipedia.org/wiki/Design_by_contract
