Aperçu #

Le standard C décrit la syntaxe du langage, les fonctions fournies par la bibliothèque standard et le comportement des processeurs C conformes (en gros, des compilateurs) et des programmes C conformes. En ce qui concerne le comportement, la norme spécifie pour la plupart des comportements particuliers pour les programmes et les processeurs. D’un autre côté, certaines opérations ont un comportement indéfini explicite ou implicite – de telles opérations doivent toujours être évitées, car vous ne pouvez pas vous fier à quoi que ce soit à leur sujet. Entre les deux, il existe une variété de comportements définis par la mise en œuvre. Ces comportements peuvent varier entre les processeurs C, les runtimes et les bibliothèques standard (collectivement, * implémentations *), mais ils sont cohérents et fiables pour toute implémentation donnée, et les implémentations conformes documentent leur comportement dans chacun de ces domaines.

Il est parfois raisonnable qu’un programme s’appuie sur un comportement défini par l’implémentation. Par exemple, si le programme est de toute façon spécifique à un environnement d’exploitation particulier, il est peu probable que le fait de s’appuyer sur des comportements définis par l’implémentation, généraux pour les processeurs communs pour cet environnement, pose un problème. Alternativement, on peut utiliser des directives de compilation conditionnelle pour sélectionner des comportements définis par l’implémentation appropriés pour l’implémentation utilisée. Dans tous les cas, il est essentiel de savoir quelles opérations ont un comportement défini par l’implémentation, afin soit de les éviter, soit de prendre une décision éclairée sur l’opportunité et la manière de les utiliser.

Le solde de ces remarques constitue une liste de tous les comportements et caractéristiques définis par la mise en œuvre spécifiés dans la norme C2011, avec des références à la norme. Beaucoup d’entre eux utilisent la terminologie de la norme. D’autres s’appuient plus généralement sur le contexte de la norme, comme les huit étapes de traduction du code source en un programme, ou la différence entre les implémentations hébergées et autonomes. Certains qui peuvent être particulièrement surprenants ou notables sont présentés en caractères gras. Tous les comportements décrits ne sont pas pris en charge par les normes C antérieures, mais d’une manière générale, ils ont un comportement défini par l’implémentation dans toutes les versions de la norme qui les prennent en charge.

Représentation des entiers signés #

Chaque type d’entier signé peut être représenté dans l’un des trois formats ; c’est l’implémentation définie qui est utilisée. L’implémentation utilisée pour tout type d’entier signé donné au moins aussi large que “int” peut être déterminée au moment de l’exécution à partir des deux bits d’ordre le plus bas de la représentation de la valeur “-1” dans ce type, comme ceci :

enum { sign_magnitude = 1, ones_compl = 2, twos_compl = 3, };
#define SIGN_REP(T) ((T)-1 & (T)3)

switch (SIGN_REP(long)) {
   case sign_magnitude: { /* do something */ break; }
   case ones_compl:     { /* do otherwise */ break; }
   case twos_compl:     { /* do yet else  */ break; }
   case 0:  { _Static_assert(SIGN_REP(long), "bogus sign representation"); }
}

Le même modèle s’applique à la représentation de types plus étroits, mais ils ne peuvent pas être testés par cette technique car les opérandes de & sont soumis aux “conversions arithmétiques habituelles” avant que le résultat ne soit calculé.
