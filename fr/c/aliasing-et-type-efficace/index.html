Aliasing et type efficace

Les violations des règles d’alias et la violation du type effectif d’un objet sont deux choses différentes et ne doivent pas être confondues.

  • Aliasing est la propriété de deux pointeurs a et b qui font référence au même objet, c’est-à-dire que a == b.

  • Le type effectif d’un objet de données est utilisé par C pour déterminer quelles opérations peuvent être effectuées sur cet objet. En particulier, le type effectif est utilisé pour déterminer si deux pointeurs peuvent s’aliaser.

Le crénelage peut être un problème pour l’optimisation, car changer l’objet via un pointeur, a' par exemple, peut changer l'objet qui est visible via l'autre pointeur, b`. Si votre compilateur C devait supposer que les pointeurs peuvent toujours s’aliaser, quels que soient leur type et leur provenance, de nombreuses opportunités d’optimisation seraient perdues et de nombreux programmes s’exécuteraient plus lentement.

Les règles d’aliasing strictes de C font référence aux cas dans le compilateur peut supposer quels objets s’aliasent (ou non). Il existe deux règles empiriques que vous devez toujours garder à l’esprit pour les pointeurs de données.

Sauf indication contraire, deux pointeurs avec le même type de base peuvent alias.

Deux pointeurs avec un type de base différent ne peuvent pas être alias, sauf si au moins l’un des deux types est un type caractère.

Ici type de base signifie que nous mettons de côté les qualifications de type telles que const, par ex. Si a est double* et b est const double*, le compilateur doit généralement supposer qu’un changement de *a peut changer *b.

Violer la deuxième règle peut avoir des résultats catastrophiques. Ici, violer la règle d’aliasing stricte signifie que vous présentez deux pointeurs a et b de type différent au compilateur qui pointent en réalité sur le même objet. Le compilateur peut alors toujours supposer que les deux pointent vers des objets différents et ne mettra pas à jour son idée de *b si vous avez changé l’objet par *a.

Si vous le faites, le comportement de votre programme devient indéfini. Par conséquent, C impose des restrictions assez sévères sur les conversions de pointeurs afin de vous aider à éviter qu’une telle situation ne se produise accidentellement.

Sauf si le type source ou cible est “void”, toutes les conversions de pointeur entre des pointeurs de type de base différent doit être explicite.

Ou en d’autres termes, ils ont besoin d’un cast, à moins que vous ne fassiez une conversion qui ajoute simplement un qualificatif tel que const au type cible.

Éviter les conversions de pointeur en général et les transtypages en particulier vous protège des problèmes d’aliasing. Sauf si vous en avez vraiment besoin, et ces cas sont très particuliers, vous devriez les éviter autant que possible.

Type effectif #

Le type effectif d’un objet de données est la dernière information de type qui lui était associée, le cas échéant.

// a normal variable, effective type uint32_t, and this type never changes
uint32_t a = 0.0;

// effective type of *pa is uint32_t, too, simply
// because *pa is the object a
uint32_t* pa = &a;

// the object pointed to by q has no effective type, yet
void* q = malloc(sizeof uint32_t);
// the object pointed to by q still has no effective type,
// because nobody has written to it
uint32_t* qb = q;
// *qb now has effective type uint32_t because a uint32_t value was written
*qb = 37;

// the object pointed to by r has no effective type, yet, although
// it is initialized
void* r = calloc(1, sizeof uint32_t);
// the object pointed to by r still has no effective type,
// because nobody has written to or read from it
uint32_t* rc = r;
// *rc now has effective type uint32_t because a value is read
// from it with that type. The read operation is valid because we used calloc.
// Now the object pointed to by r (which is the same as *rc) has
// gained an effective type, although we didn't change its value.
uint32_t c = *rc;

// the object pointed to by s has no effective type, yet.
void* s = malloc(sizeof uint32_t);
// the object pointed to by s now has effective type uint32_t
// because an uint32_t value is copied into it.
memcpy(s, r, sizeof uint32_t);

Remarquez que pour ce dernier, il n’était pas nécessaire que nous ayons même un uint32_t* pointeur vers cet objet. Le fait que nous ayons copié un autre L’objet uint32_t est suffisant.

restreindre la qualification #

Si nous avons deux arguments de pointeur du même type, le compilateur ne peut faire aucune hypothèse et devra toujours supposer que le changement de *e peut changer *f :

void fun(float* e, float* f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

float fval = 4;
float eval = 77;
 fun(&eval, &fval);

tout va bien et quelque chose comme

est-ce que 4 est égal à 4 ?

est imprimé. Si nous passons le même pointeur, le programme fera toujours la bonne chose et imprimera

est-ce que 4 est égal à 22 ?

Cela peut s’avérer inefficace si nous * savons * par des informations extérieures que e et f ne pointeront jamais vers le même objet de données. Nous pouvons refléter cette connaissance en ajoutant des qualificatifs “restrict” aux paramètres du pointeur :

void fan(float*restrict e, float*restrict f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

Alors le compilateur peut toujours supposer que e et f pointent vers des objets différents.

Modification des octets #

Une fois qu’un objet a un type effectif, vous ne devez pas tenter de le modifier via un pointeur d’un autre type, à moins que cet autre type ne soit un type de caractère, char, signed char ou unsigned char.

#include <inttypes.h>
#include <stdio.h>

int main(void) {
  uint32_t a = 57;
  // conversion from incompatible types needs a cast !
  unsigned char* ap = (unsigned char*)&a;
  for (size_t i = 0; i < sizeof a; ++i) {
    /* set each byte of a to 42 */
    ap[i] = 42;
  }
  printf("a now has value %" PRIu32 "\n", a);
}

Ceci est un programme valide qui imprime

a a maintenant la valeur 707406378

Cela fonctionne parce que :

  • L’accès est fait aux octets individuels vus avec le type unsigned char afin que chaque modification soit bien définie.
  • Les deux vues de l’objet, via a et via *ap, alias, mais comme ap est un pointeur vers un type de caractère, la règle d’alias stricte ne s’applique pas. Ainsi, le compilateur doit supposer que la valeur de a peut avoir été modifiée dans la boucle for. La valeur modifiée de ‘a’ doit être construite à partir des octets qui ont été modifiés.
  • Le type de a, uint32_t n’a pas de bits de remplissage. Tous ses bits de la représentation comptent pour la valeur, ici 707406378, et il ne peut y avoir de représentation de piège.

Les types de caractères ne sont pas accessibles via des types non-caractères. #

Si un objet est défini avec une durée de stockage statique, thread ou automatique, et qu’il a un type de caractère, soit : char, unsigned char ou signed char, il ne peut pas être accédé par un type non-caractère . Dans l’exemple ci-dessous, un tableau char est réinterprété comme le type int, et le comportement est indéfini à chaque déréférencement du pointeur int b.

int main( void )
{
    char a[100];
    int* b = ( int* )&a;
    *b = 1;      

    static char c[100];
    b = ( int* )&c;
    *b = 2;

    _Thread_local char d[100];
    b = ( int* )&d;
    *b = 3;
}

Ceci n’est pas défini car il viole la règle du “type effectif”, aucun objet de données ayant un type effectif ne peut être accessible via un autre type qui n’est pas un type caractère. Étant donné que l’autre type ici est int, cela n’est pas autorisé.

Même si l’alignement et la taille des pointeurs étaient connus pour s’adapter, cela n’exempterait pas de cette règle, le comportement serait toujours indéfini.

Cela signifie notamment qu’il n’y a aucun moyen dans la norme C de réserver un objet tampon de type caractère qui peut être utilisé via des pointeurs avec différents types, comme vous utiliseriez un tampon qui a été reçu par malloc ou fonction similaire.

Une façon correcte d’atteindre le même objectif que dans l’exemple ci-dessus serait être d’utiliser un “syndicat”.

typedef union bufType bufType;
union bufType {
   char c[sizeof(int[25])];
   int i[25];
};

int main( void )
{
    bufType a = { .c = { 0 } }; // reserve a buffer and initialize
    int* b = a.i;      // no cast necessary
    *b = 1;      

    static bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 2;

    _Thread_local bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 3;
}

Ici, union garantit que le compilateur sait dès le début que le tampon pourrait être accessibles via différentes vues. Cela présente également l’avantage que maintenant le tampon a une “vue” a.i qui est déjà de type int et aucune conversion de pointeur n’est nécessaire.

Violation des règles strictes d’aliasing #

Dans le code suivant, supposons pour simplifier que float et uint32_t ont la même taille.

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    float b = *f;
    print("%g should equal %g\n", a, b);
}

u et f ont un type de base différent, et le compilateur peut donc supposer qu’ils pointent vers des objets différents. Il n’y a aucune possibilité que *f ait pu changer entre les deux initialisations de a et b, et donc le compilateur peut optimiser le code à quelque chose d’équivalent à

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    print("%g should equal %g\n", a, a);
}

C’est-à-dire que la deuxième opération de chargement de *f peut être complètement optimisée.

Si nous appelons cette fonction “normalement”

 float fval = 4;
 uint32_t uval = 77;
 fun(&uval, &fval);

tout va bien et quelque chose comme

4 devrait être égal à 4

est imprimé. Mais si nous trichons et passons le même pointeur, après l’avoir converti,

 float fval = 4;
 uint32_t* up = (uint32_t*)&fval;
 fun(up, &fval);

nous enfreignons la règle stricte d’aliasing. Ensuite, le comportement devient indéfini. La sortie pourrait être comme ci-dessus, si le compilateur avait optimisé le deuxième accès, ou quelque chose de complètement différent, et donc votre programme se retrouve dans un état complètement non fiable.
