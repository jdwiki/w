Syntaxe #

  • void (*signal(int sig, void (*func)(int)))(int);

Paramètres #

Paramètre Détails
signe Le signal sur lequel définir le gestionnaire de signaux, l’un des SIGABRT, SIGFPE, SIGILL, SIGTERM, SIGINT, SIGSEGV ou une valeur définie par l’implémentation
fonction Le gestionnaire de signal, qui est l’un des suivants : SIG_DFL, pour le gestionnaire par défaut, SIG_IGN pour ignorer le signal, ou un pointeur de fonction avec la signature void foo(int sig);.

L’utilisation de gestionnaires de signaux avec uniquement les garanties de la norme C impose diverses limitations sur ce qui peut ou ne peut pas être fait dans le gestionnaire de signaux défini par l’utilisateur.

  • Si la fonction définie par l’utilisateur revient lors de la gestion de SIGSEGV, SIGFPE, SIGILL ou de toute autre interruption matérielle définie par l’implémentation, le comportement n’est pas défini par la norme C. C’est parce que l’interface de C ne donne pas les moyens de changer l’état défectueux (par exemple après une division par ‘0’) et donc lors du retour, le programme est exactement dans le même état erroné qu’avant l’interruption matérielle.

  • Si la fonction définie par l’utilisateur a été appelée à la suite d’un appel à abort ou raise, le gestionnaire de signal n’est pas autorisé à appeler raise, à nouveau.

  • Les signaux peuvent arriver au milieu de n’importe quelle opération, et donc l’indivisibilité des opérations ne peut généralement pas être garantie et la gestion des signaux ne fonctionne pas bien avec l’optimisation. Par conséquent, toutes les modifications apportées aux données dans un gestionnaire de signaux doivent être apportées aux variables

  • de type sig_atomic_t (toutes versions) ou de type atomique sans verrou (depuis C11, facultatif)

  • qui sont qualifiés de “volatils”.

  • D’autres fonctions de la bibliothèque standard C ne respecteront généralement pas ces restrictions, car elles peuvent modifier des variables dans l’état global du programme. La norme C ne garantit que abort, _Exit (depuis C99), quick_exit (depuis C11), signal (pour le même numéro de signal) et certaines opérations atomiques (depuis C11).

Le comportement n’est pas défini par la norme C si l’une des règles ci-dessus est violée. Les plates-formes peuvent avoir des extensions spécifiques, mais celles-ci ne sont généralement pas portables au-delà de cette plate-forme.

  • Habituellement, les systèmes ont leur propre liste de fonctions qui sont * sûres du signal asynchrone *, c’est-à-dire des fonctions de la bibliothèque C qui peuvent être utilisées à partir d’un gestionnaire de signal. Par exemple, souvent printf fait partie de ces fonctions.

  • En particulier, la norme C ne définit pas grand-chose sur l’interaction avec son interface de threads (depuis C11) ou les bibliothèques de threads spécifiques à la plate-forme telles que les threads POSIX. Ces plates-formes doivent spécifier l’interaction de ces bibliothèques de threads avec des signaux par elles-mêmes.

Traitement du signal avec “signal ()” #

Numéros de signal peuvent être synchrones (comme SIGSEGV - défaut de segmentation) lorsqu’ils sont déclenchés par un dysfonctionnement du programme lui-même ou asynchrones (comme SIGINT - attention interactive) lorsqu’ils sont lancés depuis l’extérieur du programme, par exemple par une pression sur une touche comme Cntrl-C.

La fonction signal() fait partie de la norme ISO C et peut être utilisée pour attribuer une fonction pour gérer un signal spécifique

#include <stdio.h>  /* printf() */
#include <stdlib.h> /* abort()  */
#include <signal.h> /* signal() */

void handler_nonportable(int sig)
{
    /* undefined behavior, maybe fine on specific platform */
    printf("Catched: %d\n", sig);
    
    /* abort is safe to call */
    abort();
}

sig_atomic_t volatile finished = 0;

void handler(int sig)
{
    switch (sig) {
    /* hardware interrupts should not return */
    case SIGSEGV:
    case SIGFPE:
    case SIGILL:

      /* quick_exit is safe to call */
      quick_exit(EXIT_FAILURE);

      /* use _Exit in pre-C11 */
      _Exit(EXIT_FAILURE);

    default:
       /* Reset the signal to the default handler, 
          so we will not be called again if things go
          wrong on return. */
      signal(sig, SIG_DFL);
      /* let everybody know that we are finished */
      finished = sig;
      return;
   }
}

int main(void)
{

    /* Catch the SIGSEGV signal, raised on segmentation faults (i.e NULL ptr access */
    if (signal(SIGSEGV, &handler) == SIG_ERR) {
        perror("could not establish handler for SIGSEGV");
        return EXIT_FAILURE;
    }

    /* Catch the SIGTERM signal, termination request */
    if (signal(SIGTERM, &handler) == SIG_ERR) {
        perror("could not establish handler for SIGTERM");
        return EXIT_FAILURE;
    }

    /* Ignore the SIGINT signal, by setting the handler to `SIG_IGN`. */
    signal(SIGINT, SIG_IGN);


    /* Do something that takes some time here, and leaves
       the time to terminate the program from the keyboard. */

    /* Then: */

    if (finished) {
       fprintf(stderr, "we have been terminated by signal %d\n", (int)finished);
        return EXIT_FAILURE;
    }


    /* Try to force a segmentation fault, and raise a SIGSEGV */
    {
      char* ptr = 0;
      *ptr = 0;
    }

    /* This should never be executed */
    return EXIT_SUCCESS;
}

L’utilisation de signal() impose des limitations importantes à ce que vous êtes autorisé à faire à l’intérieur des gestionnaires de signaux, consultez les remarques pour plus d’informations.

[POSIX][1] recommande l’utilisation de sigaction() au lieu de signal(), en raison de son comportement sous-spécifié et des variations d’implémentation importantes. POSIX définit également beaucoup plus de signaux que la norme ISO C, y compris SIGUSR1 et SIGUSR2, qui peuvent être utilisés librement par le programmeur à n’importe quelle fin .

[1] : https://www.wikiod.com/fr/posix/signaux
