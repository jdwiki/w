Le langage C est traditionnellement un langage compilé (par opposition à interprété). La norme C définit les phases de traduction, et le produit de leur application est une image de programme (ou un programme compilé). Dans [tag:c11], les phases sont listées au §5.1.1.2.

Extension de nom de fichier Descriptif
.c Fichier source. Contient généralement des définitions et du code.
.h En tête de fichier. Contient généralement des déclarations.
.o Fichier objet. Code compilé en langage machine.
.obj Extension alternative pour les fichiers objets.
.a Fichier de bibliothèque. Paquet de fichiers objets.
.dll Bibliothèque de liens dynamiques sous Windows.
.so Objet partagé (bibliothèque) sur de nombreux systèmes de type Unix.
.dylib Dynamic-Link Library sur OSX (variante Unix).
.exe, .com Fichier exécutable Windows. Formé en liant des fichiers d’objets et des fichiers de bibliothèque. Dans les systèmes de type Unix, il n’y a pas d’extension de nom de fichier spéciale pour le fichier exécutable.
Drapeaux du compilateur POSIX c99 Descriptif
-o nom_fichier Nom du fichier de sortie, par exemple. (bin/programme.exe, programme)
-I répertoire rechercher les en-têtes dans directory.
-D nom définir la macro nom
-L répertoire rechercher des bibliothèques dans directory.
-l nom bibliothèque de liens libname.

Les compilateurs sur les plates-formes POSIX (Linux, mainframes, Mac) acceptent généralement ces options, même si elles ne s’appellent pas c99.

  • Voir aussi [c99 - compiler des programmes C standards][1]
Drapeaux GCC (collection de compilateurs GNU) Descriptif
-Mur Active tous les messages d’avertissement généralement acceptés comme étant utiles.
-Wextra Active plus de messages d’avertissement, peut être trop bruyant.
-pédant Forcer les avertissements lorsque le code enfreint la norme choisie.
-Wconversion Activer les avertissements sur la conversion implicite, à utiliser avec précaution.
-c Compile les fichiers source sans liaison.
-v Imprime les informations de compilation.
  • gcc accepte les drapeaux POSIX ainsi que beaucoup d’autres.
  • De nombreux autres compilateurs sur les plates-formes POSIX (clang, compilateurs spécifiques au fournisseur) utilisent également les drapeaux répertoriés ci-dessus.
  • Voir aussi Invoking GCC pour de nombreuses autres options.
Indicateurs TCC (compilateur Tiny C) Descriptif
-Wimplicit-function-declaration Avertir de la déclaration de fonction implicite.
-Wunsupported Avertir des fonctionnalités GCC non prises en charge qui sont ignorées par TCC.
-Wwrite-strings Faites en sorte que les constantes de chaîne soient de type const char * au lieu de char *.
-Erreur Abandonner la compilation si des avertissements sont émis.
-Mur Activez tous les avertissements, sauf -Werror, -Wunusupported et -Wwrite strings.

[1] : http://pubs.opengroup.org/onlinepubs/9699919799/utilities/c99.html

Types de fichier #

Pour compiler des programmes C, vous devez travailler avec cinq types de fichiers :

  1. Fichiers sources : ces fichiers contiennent des définitions de fonctions et ont des noms qui se terminent par .c par convention. Remarque : .cc et .cpp sont des fichiers C++ ; pas les fichiers C. par exemple, “foo.c”

  2. Fichiers d’en-tête : ces fichiers contiennent des prototypes de fonctions et diverses instructions de préprocesseur (voir ci-dessous). Ils sont utilisés pour permettre aux fichiers de code source d’accéder à des fonctions définies en externe. Les fichiers d’en-tête se terminent par .h par convention. par exemple, “foo.h”

  3. Fichiers objet : ces fichiers sont produits en tant que sortie du compilateur. Ils consistent en des définitions de fonctions sous forme binaire, mais ils ne sont pas exécutables par eux-mêmes. Les fichiers objets se terminent par .o par convention, bien que sur certains systèmes d’exploitation (par exemple Windows, MS-DOS), ils se terminent souvent par .obj. par exemple, foo.o foo.obj

  4. Exécutables binaires : ils sont produits en tant que sortie d’un programme appelé “éditeur de liens”. L’éditeur de liens relie un certain nombre de fichiers objets pour produire un fichier binaire qui peut être directement exécuté. Les exécutables binaires n’ont pas de suffixe spécial sur les systèmes d’exploitation Unix, bien qu’ils se terminent généralement par “.exe” sous Windows. par exemple, foo foo.exe

  5. Bibliothèques : une bibliothèque est un binaire compilé mais n’est pas en soi un exécutable (c’est-à-dire qu’il n’y a pas de fonction main() dans une bibliothèque). Une bibliothèque contient des fonctions qui peuvent être utilisées par plusieurs programmes. Une bibliothèque doit être livrée avec des fichiers d’en-tête contenant des prototypes pour toutes les fonctions de la bibliothèque ; ces fichiers d’en-tête doivent être référencés (par exemple, #include <library.h>) dans tout fichier source utilisant la bibliothèque. L’éditeur de liens doit ensuite être référé à la bibliothèque pour que le programme puisse être compilé avec succès. Il existe deux types de bibliothèques : statique et dynamique.

  • Bibliothèque statique : une bibliothèque statique (fichiers ‘.a’ pour les systèmes POSIX et fichiers ‘.lib’ pour Windows — à ne pas confondre avec [fichiers de bibliothèque d’importation DLL](https://msdn.microsoft.com /en-us/library/windows/desktop/ms682592(v=vs.85).aspx), qui utilise également l’extension .lib) est intégré de manière statique au programme. Les bibliothèques statiques ont l’avantage que le programme sait exactement quelle version d’une bibliothèque est utilisée. D’autre part, les tailles des exécutables sont plus grandes car toutes les fonctions de bibliothèque utilisées sont incluses. par exemple, libfoo.a foo.lib
  • Bibliothèque dynamique : une bibliothèque dynamique (fichiers .so pour la plupart des systèmes POSIX, fichiers .dylib pour OSX et fichiers .dll pour Windows) est liée dynamiquement au moment de l’exécution par le programme. Celles-ci sont également parfois appelées bibliothèques partagées car une image de bibliothèque peut être partagée par de nombreux programmes. Les bibliothèques dynamiques ont l’avantage d’occuper moins d’espace disque si plusieurs applications utilisent la bibliothèque. De plus, ils permettent les mises à jour de la bibliothèque (corrections de bogues) sans avoir à reconstruire les exécutables. par exemple, foo.so foo.dylib foo.dll

L’éditeur de liens #

Le travail de l’éditeur de liens consiste à lier un ensemble de fichiers objets (fichiers ‘.o’) dans un exécutable binaire. Le processus de liaison implique principalement la résolution d’adresses symboliques en adresses numériques. Le résultat du processus de liaison est normalement un programme exécutable.

Pendant le processus de liaison, l’éditeur de liens récupère tous les modules d’objet spécifiés sur la ligne de commande, ajoute du code de démarrage spécifique au système devant et essaie de résoudre toutes les références externes dans le module d’objet avec des définitions externes dans d’autres fichiers objets (les fichiers objets peuvent être spécifiés directement sur la ligne de commande ou peuvent être implicitement ajoutés via des bibliothèques). Il attribuera ensuite des adresses de chargement aux fichiers objets, c’est-à-dire qu’il spécifiera où le code et les données se retrouveront dans l’espace d’adressage du programme fini. Une fois qu’il a obtenu les adresses de chargement, il peut remplacer toutes les adresses symboliques du code objet par des adresses numériques “réelles” dans l’espace d’adressage de la cible. Le programme est maintenant prêt à être exécuté.

Cela inclut à la fois les fichiers objet que le compilateur a créés à partir de vos fichiers de code source ainsi que les fichiers objet qui ont été pré-compilés pour vous et rassemblés dans des fichiers de bibliothèque. Ces fichiers ont des noms qui se terminent par .a ou .so, et vous n’avez normalement pas besoin de les connaître, car l’éditeur de liens sait où se trouvent la plupart d’entre eux et les liera automatiquement si nécessaire.

Invocation implicite de l’éditeur de liens #

Comme le pré-processeur, l’éditeur de liens est un programme séparé, souvent appelé ld (mais Linux utilise collect2, par exemple). Tout comme le préprocesseur, l’éditeur de liens est invoqué automatiquement pour vous lorsque vous utilisez le compilateur. Ainsi, la manière normale d’utiliser l’éditeur de liens est la suivante :

% gcc foo.o bar.o baz.o -o myprog

Cette ligne indique au compilateur de lier trois fichiers objets (foo.o, bar.o et baz.o) dans un fichier exécutable binaire nommé myprog. Vous avez maintenant un fichier appelé myprog que vous pouvez exécuter et qui, espérons-le, fera quelque chose de cool et/ou d’utile.

Invocation explicite de l’éditeur de liens #

Il est possible d’invoquer directement l’éditeur de liens, mais cela est rarement conseillé et est généralement très spécifique à la plate-forme. Autrement dit, les options qui fonctionnent sur Linux ne fonctionneront pas nécessairement sur Solaris, AIX, macOS, Windows et de même pour toute autre plate-forme. Si vous travaillez avec GCC, vous pouvez utiliser gcc -v pour voir ce qui est exécuté en votre nom.

Options pour l’éditeur de liens #

L’éditeur de liens prend également des arguments pour modifier son comportement. La commande suivante indiquerait à gcc de lier foo.o et bar.o, mais également d’inclure la bibliothèque ncurses. 

% gcc foo.o bar.o -o foo -lncurses

Ceci est en fait (plus ou moins) équivalent à

% gcc foo.o bar.o /usr/lib/libncurses.so -o foo

(bien que libncurses.so puisse être libncurses.a, qui est juste une archive créée avec ar). Notez que vous devez lister les bibliothèques (soit par chemin d’accès, soit via les options -lname) après les fichiers objets. Avec les bibliothèques statiques, l’ordre dans lequel elles sont spécifiées est important ; souvent, avec les bibliothèques partagées, l’ordre n’a pas d’importance.

Notez que sur de nombreux systèmes, si vous utilisez des fonctions mathématiques (à partir de <math.h>), vous devez spécifier -lm pour charger la bibliothèque mathématique - mais Mac OS X et macOS Sierra n’en ont pas besoin. Il existe d’autres bibliothèques qui sont des bibliothèques distinctes sur Linux et d’autres systèmes Unix, mais pas sur macOS - les threads POSIX et les bibliothèques temps réel POSIX et réseau en sont des exemples. Par conséquent, le processus de liaison varie d’une plate-forme à l’autre.

Autres options de compilation #

C’est tout ce que vous devez savoir pour commencer à compiler vos propres programmes C. Généralement, nous vous recommandons également d’utiliser l’option de ligne de commande -Wall :

% gcc -Wall -c foo.cc

L’option -Wall amène le compilateur à vous avertir des constructions de code légales mais douteuses, et vous aidera à détecter très tôt de nombreux bogues.

Si vous voulez que le compilateur vous lance plus d’avertissements (y compris les variables déclarées mais non utilisées, l’oubli de renvoyer une valeur, etc.), vous pouvez utiliser cet ensemble d’options, car -Wall, malgré le nom, ne t activez tous les avertissements possibles :

% gcc -Wall -Wextra -Wfloat-equal -Wundef -Wcast-align -Wwrite-strings -Wlogical-op \
>     -Wmissing-declarations -Wredundant-decls -Wshadow …

Notez que clang a une option -Weverything qui active vraiment tous les avertissements dans clang.

Le compilateur #

Une fois que le préprocesseur C a inclus tous les fichiers d’en-tête et développé toutes les macros, le compilateur peut compiler le programme. Pour ce faire, il transforme le code source C en un fichier de code objet, qui est un fichier se terminant par “.o” qui contient la version binaire du code source. Cependant, le code objet n’est pas directement exécutable. Afin de créer un exécutable, vous devez également ajouter du code pour toutes les fonctions de la bibliothèque qui ont été #include dans le fichier (ce n’est pas la même chose que d’inclure les déclarations, ce que fait #include). C’est le travail de [l’éditeur de liens](https://www.wikiod.com/fr/c/compilation#L’éditeur de liens).

En général, la séquence exacte d’appel d’un compilateur C dépend beaucoup du système que vous utilisez. Ici, nous utilisons le compilateur GCC, mais il convient de noter qu’il existe de nombreux autres compilateurs :

% gcc -Wall -c foo.c

% est l’invite de commande du système d’exploitation. Cela indique au compilateur d’exécuter le préprocesseur sur le fichier foo.c puis de le compiler dans le fichier de code objet foo.o. L’option -c signifie compiler le fichier de code source dans un fichier objet mais pas pour invoquer l’éditeur de liens. Cette option -c est disponible sur les systèmes POSIX, tels que Linux ou macOS ; d’autres systèmes peuvent utiliser une syntaxe différente.

Si tout votre programme se trouve dans un seul fichier de code source, vous pouvez à la place faire ceci :

% gcc -Wall foo.c -o foo

Cela indique au compilateur d’exécuter le préprocesseur sur foo.c, de le compiler puis de le lier pour créer un exécutable appelé foo. L’option -o indique que le mot suivant sur la ligne est le nom du fichier exécutable binaire (programme). Si vous ne spécifiez pas -o (si vous tapez simplement gcc foo.c), l’exécutable sera nommé a.out pour des raisons historiques.

En général, le compilateur suit quatre étapes lors de la conversion d’un fichier .c en un exécutable :

  1. pré-traitement - développe textuellement les directives #include et les macros #define dans votre fichier .c
  2. compilation - convertit le programme en assembleur (vous pouvez arrêter le compilateur à cette étape en ajoutant l’option -S)
  3. assembly - convertit l’assembly en code machine
  4. linkage - relie le code objet à des bibliothèques externes pour créer un exécutable

Notez également que le nom du compilateur que nous utilisons est GCC, qui signifie à la fois “compilateur GNU C” et “collection de compilateurs GNU”, selon le contexte. D’autres compilateurs C existent. Pour les systèmes d’exploitation de type Unix, beaucoup d’entre eux portent le nom cc, pour “compilateur C”, qui est souvent un lien symbolique vers un autre compilateur. Sur les systèmes Linux, cc est souvent un alias pour GCC. Sur macOS ou OS-X, il pointe vers clang.

Les normes POSIX exigent actuellement c99 comme nom d’un compilateur C — il prend en charge la norme C99 par défaut. Les versions antérieures de POSIX mandataient c89 comme compilateur. POSIX exige également que ce compilateur comprenne les options -c et -o que nous avons utilisées ci-dessus.

Remarque : L’option -Wall présente dans les deux exemples gcc indique au compilateur d’afficher des avertissements concernant les constructions douteuses, ce qui est fortement recommandé. C’est aussi une bonne idée d’ajouter d’autres options d’avertissement, par ex. -Wextra.

Le préprocesseur #

Avant que le compilateur C ne commence à compiler un fichier de code source, le fichier est traité dans une phase de prétraitement. Cette phase peut être effectuée par un programme séparé ou être complètement intégrée dans un exécutable. Dans tous les cas, il est invoqué automatiquement par le compilateur avant le début de la compilation proprement dite. La phase de prétraitement convertit votre code source en un autre code source ou unité de traduction en appliquant des remplacements textuels. Vous pouvez le considérer comme un code source “modifié” ou “étendu”. Cette source étendue peut exister en tant que fichier réel dans le système de fichiers, ou elle peut n’être stockée en mémoire que pendant une courte période avant d’être traitée davantage.

Les commandes du préprocesseur commencent par le signe dièse ("#"). Il existe plusieurs commandes de préprocesseur ; deux des plus importants sont :

  1. Définit :

    #define is mainly used to define constants. For instance,

     #define BIGNUM 1000000
 int a = BIGNUM;

    becomes

     int a = 1000000;

`#define` is used in this way so as to avoid having to explicitly write out some constant value in many different places in a source code file. This is important in case you need to change the constant value later on; it's much less bug-prone to change it once, in the `#define`, than to have to change it in multiple places scattered all over the code.

    Because #define just does advanced search and replace, you can also declare macros. For instance:

     #define ISTRUE(stm) do{stm = stm ? 1 : 0;}while(0)
 // in the function:
 a = x;
 ISTRUE(a);

    becomes:

     // in the function:
 a = x;
 do {
     a = a ? 1 : 0;
 } while(0);

    At first approximation, this effect is roughly the same as with inline functions, but the preprocessor doesn’t provide type checking for #define macros. This is well known to be error-prone and their use necessitates great caution.

    Also note here, that the preprocessor would also replace comments with a blanks as explained below.

  2. Comprend :

#include est utilisé pour accéder aux définitions de fonctions définies en dehors d’un fichier de code source. Par exemple:

    #include <stdio.h>

oblige le préprocesseur à coller le contenu de <stdio.h> dans le fichier de code source à l’emplacement de l’instruction #include avant qu’il ne soit compilé. #include est presque toujours utilisé pour inclure des fichiers d’en-tête, qui sont des fichiers qui contiennent principalement des déclarations de fonction et des déclarations #define. Dans ce cas, nous utilisons #include afin de pouvoir utiliser des fonctions telles que printf et scanf, dont les déclarations se trouvent dans le fichier stdio.h. Les compilateurs C ne vous permettent pas d’utiliser une fonction à moins qu’elle n’ait été préalablement déclarée ou définie dans ce fichier ; Les instructions #include sont donc le moyen de réutiliser le code précédemment écrit dans vos programmes C.

  1. Opérations logiques :

    #if defined A || defined B
variable = another_variable + 1;
#else
variable = another_variable * 2;
#endif

    will be changed to:

    variable = another_variable + 1;

si A ou B ont été définis quelque part dans le projet auparavant. Si ce n’est pas le cas, bien sûr le préprocesseur fera ceci :

   variable = another_variable * 2;

Ceci est souvent utilisé pour le code qui s’exécute sur différents systèmes ou se compile sur différents compilateurs. Puisqu’il existe des définitions globales, qui sont spécifiques au compilateur/système, vous pouvez tester ces définitions et toujours laisser le compilateur utiliser le code qu’il compilera à coup sûr.

  1. Commentaires

    The Preprocessor replaces all comments in the source file by single spaces. Comments are indicated by // up to the end of the line, or a combination of opening /* and closing */ comment brackets.

Les phases de traduction #

À partir de la norme C 2011, répertoriée dans §5.1.1.2 Phases de traduction, la traduction du code source en image de programme (par exemple, l’exécutable) est répertoriée pour se produire en 8 étapes ordonnées.

  1. L’entrée du fichier source est mappée au jeu de caractères source (si nécessaire). Les trigraphes sont remplacés à cette étape.
  2. Les lignes de continuation (lignes qui se terminent par \) sont fusionnées avec la ligne suivante.
  3. Le code source est analysé en espaces blancs et jetons de prétraitement.
  4. Le préprocesseur est appliqué, qui exécute les directives, développe les macros et applique les pragmas. Chaque fichier source extrait par #include subit les phases de traduction 1 à 4 (récursivement si nécessaire). Toutes les directives liées au préprocesseur sont ensuite supprimées.
  5. Les valeurs du jeu de caractères source dans les constantes de caractères et les littéraux de chaîne sont mappées au jeu de caractères d’exécution.
  6. Les littéraux de chaîne adjacents les uns aux autres sont concaténés.
  7. Le code source est analysé en jetons, qui comprennent l’unité de traduction.
  8. Les références externes sont résolues et l’image du programme est formée.

Une implémentation d’un compilateur C peut combiner plusieurs étapes ensemble, mais l’image résultante doit toujours se comporter comme si les étapes ci-dessus s’étaient produites séparément dans l’ordre indiqué ci-dessus.
