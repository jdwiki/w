Niché if()…else VS if()..else Ladder #

Les instructions if()...else imbriquées prennent plus de temps d’exécution (elles sont plus lentes) par rapport à une échelle if()...else car les instructions if()...else imbriquées vérifient tous les instructions conditionnelles internes une fois que l’instruction conditionnelle externe if() est satisfaite, tandis que l’échelle if()..else arrêtera le test de condition une fois que l’une des instructions conditionnelles if() ou else if() sont vrai.

Une échelle if()...else :

#include <stdio.h>

int main(int argc, char *argv[])
{
  int a, b, c;
  printf("\nEnter Three numbers = ");
  scanf("%d%d%d", &a, &b, &c);
  if ((a < b) && (a < c))
  {
    printf("\na = %d is the smallest.", a);
  }
  else if ((b < a) && (b < c))
  {
    printf("\nb = %d is the smallest.", b);
  }
  else if ((c < a) && (c < b))
  {
    printf("\nc = %d is the smallest.", c);
  }
  else
  {
    printf("\nImprove your coding logic");
  }
  return 0;
}

Est, dans le cas général, considéré comme meilleur que l’équivalent imbriqué if()...else :

#include <stdio.h>

int main(int argc, char *argv[])
{
  int a, b, c;
  printf("\nEnter Three numbers = ");
  scanf("%d%d%d", &a, &b, &c);
  if (a < b)
  {
    if (a < c)
      {
        printf("\na = %d is the smallest.", a);
      }
    else
      {
        printf("\nc = %d is the smallest.", c);
      }
  }
  else
  {
    if(b < c)
    {
      printf("\nb = %d is the smallest.", b);
    }
    else
    {
      printf("\nc = %d is the smallest.", c);
    }
  }
  return 0;  
}

si () Déclarations #

L’un des moyens les plus simples de contrôler le déroulement du programme consiste à utiliser des instructions de sélection “if”. Si un bloc de code doit être exécuté ou non, cela peut être décidé par cette instruction.

La syntaxe de l’instruction de sélection “if” en C pourrait être la suivante :

if(cond) 
{
  statement(s);  /*to be executed, on condition being true*/
}

Par exemple,

if (a > 1) {
    puts("a is larger than 1");
}

Où a > 1 est une condition qui doit être évaluée à true afin d’exécuter les instructions à l’intérieur du bloc if. Dans cet exemple, “a est supérieur à 1” n’est imprimé que si a > 1 est vrai.

Les instructions de sélection if peuvent omettre les accolades { et } s’il n’y a qu’une seule instruction dans le bloc. L’exemple ci-dessus peut être réécrit en

if (a > 1)
    puts("a is larger than 1");

Cependant, pour exécuter plusieurs instructions dans un bloc, les accolades doivent être utilisées.

La condition pour if peut inclure plusieurs expressions. if n’effectuera l’action que si le résultat final de l’expression est vrai.

Par exemple

if ((a > 1) && (b > 1)) {
    puts("a is larger than 1");
    a++;
}

n’exécutera printf et a++ que si les deux a et b sont supérieurs à 1.

switch () Déclarations #

Les instructions switch sont utiles lorsque vous voulez que votre programme fasse de nombreuses choses différentes en fonction de la valeur d’une variable de test particulière.

Un exemple d’utilisation de l’instruction switch ressemble à ceci :

int a = 1;

switch (a) {
case 1:
    puts("a is 1");
    break;
case 2:
    puts("a is 2");
    break;
default:
    puts("a is neither 1 nor 2");
    break;
}

Cet exemple est équivalent à

int a = 1;

if (a == 1) {
    puts("a is 1");
} else if (a == 2) {
    puts("a is 2");
} else {
    puts("a is neither 1 nor 2");
}

Si la valeur de a est 1 lorsque l’instruction switch est utilisée, a est 1 sera imprimé. Si la valeur de ‘a’ est 2, alors ‘a vaut 2’ sera imprimé. Sinon, a n'est ni 1 ni 2 ​​sera imprimé.

case n: est utilisé pour décrire où le flux d’exécution sautera lorsque la valeur transmise à l’instruction switch est n. n doit être une constante au moment de la compilation et le même n peut exister au plus une fois dans une instruction switch.

default: est utilisé pour décrire cela lorsque la valeur ne correspond à aucun des choix pour case n:. C’est une bonne pratique d’inclure un cas “par défaut” dans chaque instruction switch pour détecter un comportement inattendu.

Une instruction break; est nécessaire pour [sauter] [1] du bloc switch.

Remarque : Si vous oubliez accidentellement d’ajouter un break après la fin d’un case, le compilateur supposera que vous avez l’intention de [“fall through”][2] et toutes les instructions de cas suivantes, si any, sera exécuté (sauf si une instruction break est trouvée dans l’un des cas suivants), que les instructions case suivantes correspondent ou non. Cette propriété particulière est utilisée pour implémenter [Duff’s Device][3]. Ce comportement est souvent considéré comme un défaut dans la spécification du langage C.

Voici un exemple qui montre les effets de l’absence de break; :

int a = 1;

switch (a) {
case 1:
case 2:
    puts("a is 1 or 2");
case 3:
    puts("a is 1, 2 or 3");
    break;
default:
    puts("a is neither 1, 2 nor 3");
    break;
}

Lorsque la valeur de a est 1 ou 2, a est 1 ou 2 et a est 1, 2 ou 3 seront tous deux imprimés. Lorsque a vaut 3, seul a vaut 1, 2 ou 3 sera imprimé. Sinon, a n'est ni 1, 2 ni 3 sera imprimé.

Notez que le cas default n’est pas nécessaire, surtout lorsque l’ensemble de valeurs que vous obtenez dans le switch est terminé et connu au moment de la compilation.

Le meilleur exemple est d’utiliser un switch sur un enum.

enum msg_type { ACK, PING, ERROR };
void f(enum msg_type t)
{
  switch (t) {
  case ACK:
    // do nothing
    break;
  case PING:
    // do something
    break;
  case ERROR:
    // do something else
    break;
  }
}

Il y a plusieurs avantages à faire cela :

  • la plupart des compilateurs signaleront un avertissement si vous ne gérez pas une valeur (cela ne serait pas signalé si un cas “par défaut” était présent)
  • pour la même raison, si vous ajoutez une nouvelle valeur à enum, vous serez averti de tous les endroits où vous avez oublié de gérer la nouvelle valeur (avec un cas default, vous devrez explorer manuellement votre code recherche de tels cas)
  • Le lecteur n’a pas besoin de comprendre “ce qui est caché par default:”, s’il existe d’autres valeurs enum ou s’il s’agit d’une protection pour “juste au cas où”. Et s’il existe d’autres valeurs enum, le codeur a-t-il intentionnellement utilisé la casse default ou y a-t-il un bogue qui a été introduit lorsqu’il a ajouté la valeur ?
  • la gestion de chaque valeur enum rend le code explicite car vous ne pouvez pas vous cacher derrière un caractère générique, vous devez explicitement gérer chacun d’eux.

Néanmoins, vous ne pouvez pas empêcher quelqu’un d’écrire du code maléfique comme :

enum msg_type t = (enum msg_type)666; // I'm evil

Ainsi, vous pouvez ajouter une vérification supplémentaire avant votre changement pour le détecter, si vous en avez vraiment besoin.

void f(enum msg_type t)
{
   if (!is_msg_type_valid(t)) {
      // Handle this unlikely error
   }

   switch(t) { 
    // Same code than before
   }
}

[1] : https://www.wikiod.com/fr/c/instructions-de-saut [2] : https://en.wikipedia.org/wiki/Switch_statement#Fallthrough [3] : https://www.wikiod.com/fr/c/instructions-diterationboucles-pour-tandis-que-do-while#Déroulage de boucle et dispositif de Duff

if () … else instructions et syntaxe #

Alors que if effectue une action uniquement lorsque sa condition est évaluée à true, if / else vous permet de spécifier les différentes actions lorsque la condition true et lorsque la condition est false.

Exemple:

if (a > 1)
    puts("a is larger than 1");
else 
    puts("a is not larger than 1");

Tout comme l’instruction if, lorsque le bloc dans if ou else est constitué d’une seule instruction, les accolades peuvent être omises (mais cela n’est pas recommandé car cela peut facilement introduire des problèmes involontairement). Cependant, s’il y a plus d’une instruction dans le bloc if ou else, alors les accolades doivent être utilisées sur ce bloc particulier.

if (a > 1) 
{
    puts("a is larger than 1");
    a--;
}
else 
{
    puts("a is not larger than 1");
    a++;
}

if()…else Ladder Chaînage de deux ou plusieurs instructions if () … else #

Alors que l’instruction if ()... else permet de définir un seul comportement (par défaut) qui se produit lorsque la condition dans if () n’est pas remplie, enchaînant deux ou plusieurs if () ... else permettent de définir quelques comportements supplémentaires avant de passer à la dernière branche “else” agissant comme “par défaut”, le cas échéant.

Exemple:

int a = ... /* initialise to some value. */

if (a >= 1) 
{
    printf("a is greater than or equals 1.\n");
} 
else if (a == 0) //we already know that a is smaller than 1
{
    printf("a equals 0.\n");
}
else /* a is smaller than 1 and not equals 0, hence: */
{ 
    printf("a is negative.\n");
}
