Paramètres #

Paramètre La description
groupe.id Le nom du groupe de consommateurs.
activer.auto.commit Valider automatiquement les décalages ; par défaut : vrai.
auto.commit.interval.ms Le délai minimum en millisecondes entre les commits (nécessite enable.auto.commit=true); par défaut : 5000.
auto.offset.reset Que faire lorsqu’aucun décalage validé valide n’a été trouvé ; par défaut : le plus récent.(+)
(+) Valeurs possibles La description
le plus tôt Réinitialise automatiquement le décalage au décalage le plus ancien.
dernier Réinitialise automatiquement le décalage au dernier décalage.
aucun Lever une exception au consommateur si aucun décalage précédent n’est trouvé pour le groupe du consommateur.
autre chose Jetez une exception au consommateur.

Gestion des décalages des consommateurs et tolérance aux pannes #

[KafkaConsumers][1] demandent des messages à un courtier Kafka via un appel à poll() et leur progression est suivie via offsets. Chaque message au sein de chaque partition de chaque sujet, a un soi-disant décalage attribué - son numéro de séquence logique dans la partition. Un KafkaConsumer suit son décalage actuel pour chaque partition qui lui est assignée. Faites attention, que les courtiers Kafka ne sont pas au courant des compensations actuelles des consommateurs. Ainsi, sur poll(), le consommateur doit envoyer ses décalages actuels au courtier, de sorte que le courtier puisse renvoyer les messages correspondants, c’est-à-dire. messages avec un décalage consécutif plus grand. Par exemple, supposons que nous ayons un seul sujet de partition et un seul consommateur avec le décalage actuel 5. Sur poll(), le consommateur envoie si décalage au courtier et le courtier renvoie des messages pour les décalages 6, 7, 8, .. .

Étant donné que les consommateurs suivent eux-mêmes leurs compensations, ces informations pourraient être perdues en cas de défaillance d’un consommateur. Ainsi, les décalages doivent être stockés de manière fiable, de sorte qu’au redémarrage, un consommateur puisse reprendre son ancien décalage et le reprendre là où il l’a laissé. Dans Kafka, il existe un support intégré pour cela via offset commits. Le nouveau KafkaConsumer peut valider son décalage actuel dans Kafka et Kafka stocke ces décalages dans une rubrique spéciale appelée __consumer_offsets. Le stockage des décalages dans un sujet Kafka n’est pas seulement tolérant aux pannes, mais permet également de réaffecter des partitions à d’autres consommateurs lors d’un rééquilibrage. Étant donné que tous les consommateurs d’un groupe de consommateurs peuvent accéder à tous les décalages validés de toutes les partitions, lors du rééquilibrage, un consommateur qui se voit attribuer une nouvelle partition lit simplement le décalage validé de cette partition à partir du sujet __consumer_offsets et reprend là où l’ancien consommateur s’est arrêté.

[1] : http://docs.confluent.io/3.0.0/clients/consumer.html

Comment valider les décalages #

[KafkaConsumers][1] peut valider automatiquement les décalages en arrière-plan (paramètre de configuration enable.auto.commit = true) quel est le paramètre par défaut. Ces validations automatiques sont effectuées dans poll() ([qui est généralement appelé dans une boucle][2]). La fréquence à laquelle les décalages doivent être validés peut être configurée via auto.commit.interval.ms. Étant donné que les auto commits sont intégrés dans poll() et que poll() est appelé par le code utilisateur, ce paramètre définit une limite inférieure pour l’intervalle inter-commit.

Comme alternative à la validation automatique, les décalages peuvent également être gérés manuellement. Pour cela, la validation automatique doit être désactivée (enable.auto.commit = false). Pour la validation manuelle, KafkaConsumers propose deux méthodes, à savoir [commitSync()][3] et [commitAsync()][4]. Comme son nom l’indique, commitSync() est un appel bloquant, qui revient une fois que les décalages ont été validés avec succès, tandis que commitAsync() revient immédiatement. Si vous voulez savoir si un commit a réussi ou non, vous pouvez fournir un gestionnaire de rappel (OffsetCommitCallback) un paramètre de méthode. Faites attention, que dans les deux appels de validation, le consommateur valide les décalages du dernier appel poll(). Par exemple. supposons un seul sujet de partition avec un seul consommateur et le dernier appel à poll() renvoie des messages avec des décalages 4,5,6. Lors de la validation, le décalage 6 sera validé car il s’agit du dernier décalage suivi par le client consommateur. Dans le même temps, commitSync() et commitAsync() permettent de mieux contrôler le décalage que vous souhaitez valider : si vous utilisez les surcharges correspondantes qui vous permettent de spécifier un Map<TopicPartition, OffsetAndMetadata> le consommateur validera uniquement les décalages spécifiés (c’est-à-dire que la carte peut contenir n’importe quel sous-ensemble de partitions affectées, et le décalage spécifié peut avoir n’importe quelle valeur).

Sémantique des décalages validés #

Un décalage validé indique que tous les messages jusqu’à ce décalage ont déjà été traités. Ainsi, comme les décalages sont des nombres consécutifs, la validation du décalage ‘X’ valide implicitement tous les décalages inférieurs à ‘X’. Par conséquent, il n’est pas nécessaire de valider chaque décalage individuellement, et la validation de plusieurs décalages à la fois se produit, mais il suffit de valider le plus grand décalage.

Faites attention, que par conception, il est également possible de valider un décalage plus petit que le dernier décalage validé. Cela peut être fait si les messages doivent être lus une deuxième fois.

Garanties de traitement #

L’utilisation de la validation automatique fournit une sémantique de traitement au moins une fois. L’hypothèse sous-jacente est que poll() n’est appelé qu’après que tous les messages précédemment livrés ont été traités avec succès. Cela garantit qu’aucun message ne sera perdu car une validation se produit * après * le traitement. Si un consommateur échoue avant une validation, tous les messages après la dernière validation sont reçus de Kafka et traités à nouveau. Cependant, cette nouvelle tentative peut entraîner des doublons, car certains messages du dernier appel poll() peuvent avoir été traités, mais l’échec s’est produit juste avant l’appel de validation automatique.

Si la sémantique de traitement au plus une fois est requise, la validation automatique doit être désactivée et une commitSync() manuelle directement après poll() doit être effectuée. Ensuite, les messages sont traités. Cela garantit que les messages sont validés avant qu’ils ne soient traités et ne soient donc jamais lus une seconde fois. Bien sûr, certains messages peuvent être perdus en cas d’échec.

Comment puis-je lire le sujet depuis son début #

Il existe plusieurs stratégies pour lire un sujet depuis son début. Pour les expliquer, nous devons d’abord comprendre ce qui se passe au démarrage du consommateur. Au démarrage d’un consommateur, voici ce qui se passe :

  1. rejoignez le groupe de consommateurs configuré, ce qui déclenche un rééquilibrage et attribue des partitions au consommateur
  2. recherchez les décalages validés (pour toutes les partitions attribuées au consommateur)
  3. pour toutes les partitions avec un décalage valide, reprendre à partir de ce décalage
  4. pour toutes les partitions avec un décalage non valide, définissez le décalage de départ en fonction du paramètre de configuration auto.offset.reset

Démarrer un nouveau groupe de consommateurs #

Si vous souhaitez traiter un sujet depuis son début, vous pouvez simplement démarrer un nouveau groupe de consommateurs (c’est-à-dire choisir un group.id inutilisé) et définir auto.offset.reset = early. Comme il n’y a pas de décalages validés pour un nouveau groupe, la réinitialisation automatique du décalage se déclenchera et le sujet sera consommé depuis son début. Faites attention, qu’au redémarrage du consommateur, si vous utilisez à nouveau le même group.id, il ne lira pas le sujet depuis le début, mais reprendra là où il s’était arrêté. Ainsi, pour cette stratégie, vous devrez attribuer un nouveau group.id chaque fois que vous voudrez lire un sujet depuis son début.

Réutiliser le même identifiant de groupe #

Pour éviter de définir un nouveau group.id chaque fois que vous voulez lire un sujet depuis son début, vous pouvez désactiver la validation automatique (via enable.auto.commit = false) avant de démarrer le consommateur pour la toute première fois (en utilisant un group.id inutilisé et le réglage auto.offset.reset = early). De plus, vous ne devez valider aucun décalage manuellement. Étant donné que les décalages ne sont jamais validés à l’aide de cette stratégie, au redémarrage, le consommateur relira le sujet depuis le début.

Cependant, cette stratégie présente deux inconvénients :

  1. il n’est pas tolérant aux pannes
  2. le rééquilibrage du groupe ne fonctionne pas comme prévu

(1) Étant donné que les décalages ne sont jamais validés, un consommateur défaillant et un consommateur arrêté sont traités de la même manière au redémarrage. Dans les deux cas, le sujet sera consommé dès le début. (2) Étant donné que les décalages ne sont jamais validés, lors du rééquilibrage, les partitions nouvellement attribuées seront consommatrices dès le début.

Par conséquent, cette stratégie ne fonctionne que pour les groupes de consommateurs avec un seul consommateur et ne doit être utilisée qu’à des fins de développement.

Réutiliser le même ID de groupe et valider #

Si vous souhaitez être tolérant aux pannes et/ou utiliser plusieurs consommateurs dans votre groupe de consommateurs, il est obligatoire d’engager des compensations. Ainsi, si vous souhaitez lire un sujet depuis son début, vous devez manipuler les décalages validés au démarrage du consommateur. Pour cela, KafkaConsumer fournit trois méthodes seek(), seekToBeginning() et seekToEnd(). Alors que seek() peut être utilisé pour définir un décalage arbitraire, les deuxième et troisième méthodes peuvent être utilisées pour rechercher respectivement le début ou la fin d’une partition. Ainsi, en cas d’échec et de redémarrage du consommateur, la recherche serait omise et le consommateur pourrait reprendre là où il s’était arrêté. Pour consumer-stop-and-restart-from-beginning, seekToBeginning() serait appelé explicitement avant que vous n’entriez dans votre boucle poll(). Notez que seekXXX() ne peut être utilisé qu’après qu’un consommateur a rejoint un groupe - il est donc nécessaire de faire un “sondage factice” avant d’utiliser seekXXX(). Le code global ressemblerait à ceci :

if (consumer-stop-and-restart-from-beginning) {
    consumer.poll(0); // dummy poll() to join consumer group
    consumer.seekToBeginning(...);
}

// now you can start your poll() loop
while (isRunning) {
    for (ConsumerRecord record : consumer.poll(0)) {
        // process a record
    }
}

Qu’est-ce qu’un groupe de consommateurs ? #

Depuis Kafka 0.9, le nouveau client de haut niveau [KafkaConsumer][1] est disponible. Il exploite un [nouveau protocole Kafka intégré][2] qui permet de combiner plusieurs consommateurs dans un soi-disant [groupe de consommateurs][3]. Un groupe de consommateurs peut être décrit comme un consommateur logique unique qui s’abonne à un ensemble de rubriques. Les partitions sur tous les sujets sont affectées aux consommateurs physiques au sein du groupe, de sorte que chaque partition est affectée à un seul consommateur (un seul consommateur peut se voir attribuer plusieurs partitions). Les consommateurs individuels appartenant au même groupe peuvent s’exécuter sur différents hôtes de manière distribuée.

Les groupes de consommateurs sont identifiés via leur group.id. Pour rendre une instance de client spécifique membre d’un groupe de consommateurs, il suffit d’affecter les groupes group.id à ce client, via la configuration du client :

Properties props = new Properties();
props.put("group.id", "groupName");
// ...some more properties required
new KafkaConsumer<K, V>(config);

Ainsi, tous les consommateurs qui se connectent au même cluster Kafka et utilisent le même group.id forment un groupe de consommateurs. Les consommateurs peuvent quitter un groupe à tout moment et de nouveaux consommateurs peuvent rejoindre un groupe à tout moment. Dans les deux cas, un soi-disant * rééquilibrage * est déclenché et les partitions sont réaffectées au groupe de consommateurs pour garantir que chaque partition est traitée par exactement un consommateur au sein du groupe.

Faites attention, que même un seul KafkaConsumer forme un groupe de consommateurs avec lui-même comme membre unique.
