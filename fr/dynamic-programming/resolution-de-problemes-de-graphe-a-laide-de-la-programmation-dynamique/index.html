Algorithme Floyd-Warshall #

Couverture minimale des sommets #

[Minimum Vertex Cover] (https://en.wikipedia.org/wiki/Vertex_cover) est un problème de graphe classique. Disons que dans une ville, nous avons quelques routes reliant quelques points. Représentons les routes à l’aide de bords et les points à l’aide de nœuds. Prenons deux exemples de graphiques :

[![Exemple de graphique, A-B, B-C, A-C, A-E, A-D, A-F, G-I, G-H, H-I, I-J, J-L, J-K, K-H][1]][1]

Nous voulons mettre des veilleurs sur certains points. Un gardien peut surveiller toutes les routes reliées au point. Le problème est, quel est le nombre minimum de gardiens nécessaires pour couvrir toutes les routes ? Si nous plaçons des gardiens au nœud A, B, H, I et J, nous pouvons couvrir toutes les routes.

[![Graphique avec veilleurs][2]][2]

C’est notre solution optimale. Nous avons besoin d’au moins 5 gardiens pour garder toute la ville. Comment déterminer cela ?

Dans un premier temps, nous devons comprendre qu’il s’agit d’un problème NP-difficile, c’est-à-dire que le problème n’a pas de solution polynomiale en temps. Mais si le graphe était un Arbre, c’est-à-dire s’il avait des nœuds (n-1) où n est le nombre d’arêtes et qu’il n’y a pas de cycle dans le graphe, on peut résoudre en utilisant la programmation dynamique.

[![Couverture vertex minimale d’un arbre, A-B, A-C, B-D, B-E, C-F][3]][3]

Pour construire une solution DP, nous devons suivre deux stratégies :

  1. S’il n’y a pas de veilleur dans un nœud, tous les nœuds qui y sont connectés doivent avoir un veilleur, sinon toutes les routes ne seront pas couvertes. Si u et v sont connectés et que u n’a pas de veilleur, alors v doit avoir un veilleur.
  2. S’il y a un veilleur dans un nœud, un nœud différent qui lui est connecté peut avoir ou non un veilleur. Cela signifie qu’il n’est pas nécessaire d’avoir un gardien, mais cela peut être bénéfique. Si u et v sont connectés et que u a un gardien, nous vérifierons et trouverons lequel est avantageux pour nous en :
    • Setting watchman in v.
    • Not setting watchman in v.

Définissons une fonction récursive avec state étant le nœud actuel dans lequel nous nous trouvons et s’il a un watchman ou non. Ici:

F(u,1) = Currently we're in 'u' node and there is a watchman in this node.
F(u,0) = Currently we're in 'u' node and there is no watchman in this node.

La fonction renverra le nombre de veilleurs dans les nœuds restants.

Prenons un exemple d’arbre :

[![Exemple d’arborescence A-B, A-C, A-D][4]][4]

Nous pouvons facilement dire que si nous ne mettons pas de veilleur sur le nœud-A, nous devrons mettre des veilleurs sur le nœud-B, C et D. Nous pouvons en déduire:

F(A,0) = F(B,1) + F(C,1) + F(D,1) + 0

Il nous renvoie le nombre de veilleurs nécessaires si nous ne mettons pas de veilleur dans le nœud-A. Nous avons ajouté 0 à la fin car nous n’avons pas défini de veilleur dans notre nœud actuel.

Maintenant, F(A,1) signifie que nous définissons watchman dans le nœud-A. Pour cela, nous pouvons soit définir des veilleurs dans tous les nœuds connectés, soit nous ne le faisons pas. Nous prendrons celui qui nous fournira le nombre minimum de gardiens.

F(A,1) = min(F(B,0), F(B,1) + min(F(C,0), F(C,1)) + min(F(D,0), F(D,1)) + 1

Nous vérifions en définissant et en ne définissant pas watchman sur chaque nœud et en prenant la valeur optimale.

Une chose à laquelle nous devons faire attention, c’est qu’une fois que nous aurons atteint le nœud enfant, nous ne reviendrons jamais sur le nœud parent. Dans l’exemple ci-dessus, nous sommes allés à B à partir de A, donc parent[B] = A. Nous ne retournerons donc pas de A à B.

Pour déterminer le cas de base, nous pouvons remarquer que, si à partir d’un nœud, nous ne pouvons pas accéder à un autre nouveau nœud, nous renverrons 1 s’il y a un veilleur dans notre nœud actuel, 0 s’il n’y a pas de veilleur dans notre nœud actuel.

Il est préférable d’avoir une liste de contiguïté pour notre arbre. Soit la liste désignée par edge. Nous aurons un tableau dp[n][2], où n indique le nombre de nœuds pour stocker les valeurs calculées et l’initialiser avec -1. Nous aurons également un tableau ** parent [n] ** pour désigner la relation parent et enfant entre les nœuds. Notre pseudo-code ressemblera à :

Procedure f(u, isGuarded):
if edge[u].size is equal to 0                    //node doesn't have any new edge
    Return isGuarded
else if dp[u][isGuarded] is not equal to -1      //already calculated
    Return dp[u][isGuarded]
end if
sum := 0
for i from i to edge[u].size
    v := edge[u][i]
    if v is not equal to parent[u]               //not a parent
        parent[v] := u
        if isGuarded equals to 0                 //not guarded, must set a watchman
            sum := sum + f(v,1)
        else                                     //guarded, check both
            sum := sum + min(f(v,1), f(v,0)
        end if
    end if
end for
dp[u][isGuarded] := sum + isGuarded
Return dp[u][isGuarded]

Si nous notons node-A comme racine, nous appellerons la fonction par : min(f(A,1), f(A,0)). Cela signifie que nous vérifierons également s’il est optimal de définir watchman dans le nœud racine ou non. Ceci est notre solution DP. Ce problème peut également être résolu en utilisant l’algorithme de correspondance maximale ou max-flow.
