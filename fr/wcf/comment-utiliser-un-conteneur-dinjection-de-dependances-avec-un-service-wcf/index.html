Comment configurer un service WCF pour utiliser un conteneur d’injection de dépendances (Castle Windsor) #

Cet exemple comporte deux parties : quelques étapes passe-partout pour ajouter Castle Windsor à votre service WCF, puis un exemple simple et concret pour montrer comment nous configurons et utilisons le conteneur de Windsor.

Cela rend l’exemple un peu long. Si vous comprenez déjà l’utilisation d’un conteneur DI, vous ne vous souciez probablement que des étapes passe-partout. Si l’utilisation d’un conteneur DI n’est pas familière, il en faut un peu plus - voir le tout fonctionner de bout en bout - avant que cela ait un sens.

Étapes standard

  1. Ajoutez le service d’intégration Castle Windsor WCF package Nuget à votre application de service WCF. Cela ajoutera des références à Castle Windsor ainsi que certains composants spécifiquement pour les services WCF.

  2. Ajoutez une classe d’application globale (global.asax) à votre projet : Ajouter > Nouvel élément > Visual C# > Web > Classe d’application globale. Le code qui configure votre conteneur doit être appelé depuis la méthode Application_Start. Pour le garder organisé, nous pouvons mettre toute notre configuration de conteneur dans une classe distincte. Nous n’avons pas à le faire. Cela n’a pas d’importance. Vous verrez cela fait différemment dans différents exemples. Ce qui compte, c’est qu’il soit appelé à partir de la méthode Application_Start car c’est votre racine de composition - là où l’application démarre. L’idée est de configurer votre conteneur au démarrage de l’application et de ne plus jamais y toucher directement. Il reste juste en arrière-plan en faisant son travail.

  3. Créez une classe pour configurer le conteneur. Cela fait deux choses :

    • ContainerInstance.AddFacility<WcfFacility>() tells Windsor’s WCF code to use this particular container when creating instances of your WCF services.
    • ContainerInstance.Install(FromAssembly.This()) tells Windsor to scan This assembly (in other words, your WCF project) looking for classes that implement IWindsorInstaller. Those classes will provide instructions to tell your container how to resolve dependencies. (We’ll create one a few steps later.)

    public static class Container { private static readonly IWindsorContainer ContainerInstance = new WindsorContainer();

     public static void Configure()
 {
     ContainerInstance.AddFacility<WcfFacility>();
     ContainerInstance.Install(FromAssembly.This());
 }

    }

  4. Appelez cette méthode depuis Application_Start dans votre global.asax :

    public class Global : System.Web.HttpApplication { protected void Application_Start(object sender, EventArgs e) { Container.Configure(); } }

  5. Créez un programme d’installation. Ceci est juste une classe qui implémente IWindsorInstaller. Celui-ci est vide. Cela ne fait rien. Nous ajouterons à cette classe en quelques étapes. Lorsque nous appelons ContainerInstance.Install(FromAssembly.This()), ContainerInstance est transmis à la méthode Install afin que nous puissions enregistrer les dépendances avec le conteneur.

    public class WindsorInstaller : IWindsorInstaller { public void Install(IWindsorContainer container, IConfigurationStore store) { // Nothing here yet! } }

  6. Dans le balisage de tout service WCF que vous créez, ajoutez cette directive. Cela indique que l’application utilisera la fonction WCF de Windsor pour créer des instances du service, ce qui signifie qu’elle peut injecter des dépendances lors de la création d’une instance.

    Factory=“Castle.Facilities.WcfIntegration.DefaultServiceHostFactory, Castle.Facilities.WcfIntegration”

** Cela met fin aux étapes passe-partout pour configurer le service afin d’utiliser un conteneur d’injection de dépendance Castle Windsor. **

Mais l’exemple n’est pas complet tant que nous ne configurons pas au moins un service WCF pour l’injection de dépendances. Le reste n’est pas passe-partout, juste un exemple simple et concret.

  1. Créez un nouveau service WCF appelé GreetingService.svc.

  2. Modifiez le balisage. Ça devrait ressembler à ça:

    <%@ ServiceHost Language=“C#” Debug=“true” Service=“WcfWindsorDocumentation.GreetingService” CodeBehind=“GreetingService.svc.cs”
    Factory=“Castle.Facilities.WcfIntegration.DefaultServiceHostFactory, Castle.Facilities.WcfIntegration” %>

  3. Remplacez IGreetingService (le contrat de service) par ceci :

    [ServiceContract] public interface IGreetingService { [OperationContract] string GetGreeting(); }

  4. Remplacez GreetingService (dans GreetingService.svc.cs) par ce code. Notez que le constructeur nécessite une instance de IGreetingProvider que nous aurons besoin de notre conteneur pour injecter.

    public class GreetingService : IGreetingService { private readonly IGreetingProvider _greetingProvider;

     public GreetingService(IGreetingProvider greetingProvider)
 {
     _greetingProvider = greetingProvider;
 }

 public string GetGreeting()
 {
     return _greetingProvider.GetGreeting();
 }

    }

  5. Ajoutez cette implémentation de IGreetingProvider. Il a également quelques dépendances qui lui sont propres et que nous aurons besoin du conteneur pour fournir. Les spécificités de ces classes ne sont pas trop importantes. Ils sont juste pour créer quelque chose de facile à suivre.

    public interface IGreetingProvider { string GetGreeting(); }

    public interface IComputerNameProvider { string GetComputerName(); }

    public class ComputerNameGreetingProvider : IGreetingProvider { private readonly IComputerNameProvider _computerNameProvider;

     public ComputerNameGreetingProvider(IComputerNameProvider computerNameProvider)
 {
     _computerNameProvider = computerNameProvider;
 }

 public string GetGreeting()
 {
     return string.Concat("Hello from ", _computerNameProvider.GetComputerName());
 }

    }

    public class EnvironmentComputerNameProvider : IComputerNameProvider { public string GetComputerName() { return System.Environment.MachineName; } }

  6. Nous avons maintenant toutes les classes dont nous avons besoin. Il ne reste plus qu’à enregistrer les dépendances avec notre conteneur. En d’autres termes, nous allons dire au conteneur quelles classes il doit créer afin qu’il sache comment “construire” une instance de GreetingService. Ce code va dans notre implémentation de IWindsorInstaller (étape 5 du code passe-partout.)

    public class WindsorInstaller : IWindsorInstaller { public void Install(IWindsorContainer container, IConfigurationStore store) { container.Register( Component.For<IGreetingService, GreetingService>(), Component.For<IGreetingProvider, ComputerNameGreetingProvider>(), Component.For<IComputerNameProvider, EnvironmentComputerNameProvider>()); } }

Cela indique au conteneur :

  • Il est responsable de la création de GreetingService en cas de besoin.
  • Lorsqu’il essaie de créer GreetingService, il aura besoin d’un IGreetingProvider. Lorsqu’il en a besoin, il doit créer un ComputerNameGreetingProvider.
  • Lorsqu’il essaie de créer ComputerNameGreetingProvider, cette classe nécessite un IComputerNameProvider. Il devrait créer une instance de EnvironmentComputerNameProvider pour répondre à ce besoin.

Si, quelque part dans le processus de création de GreetingService ou de l’une de ses dépendances, il rencontre une exigence pour une dépendance que nous n’avons pas enregistrée, il nous le fera savoir avec une exception utile, comme celle-ci.

Dépendance manquante. Le composant WcfWindsorDocumentation.ComputerNameGreetingProvider a une dépendance sur WcfWindsorDocumentation.IComputerNameProvider, qui n’a pas pu être résolu. Assurez-vous que la dépendance est correctement enregistrée dans le conteneur en tant que service ou fournie en tant qu’argument en ligne.

Cela signifie que quelque chose dépend de IComputerNameProvider mais qu’il n’y avait rien d’enregistré pour répondre à cette dépendance.

** Cela ne fait que lancer le bal. ** Il y a beaucoup plus à configurer et à utiliser correctement un conteneur d’injection de dépendances pour des scénarios du monde réel. Cet exemple ne couvre que ce qui est spécifique à l’ajout de Windsor à une application de service WCF. Si vous utilisez un conteneur différent comme Autofac ou Unity, vous constaterez que même si la syntaxe et les détails varient, ils font en principe les mêmes choses et vous remarquerez facilement les similitudes.
