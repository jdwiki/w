L’héritage est une fonctionnalité de base orientée objet dans laquelle une classe acquiert et étend les propriétés d’une autre classe, en utilisant le mot-clé “extends”. Pour les interfaces et le mot-clé impléments, voir [interfaces][1].

[1] : https://www.wikiod.com/fr/java/interfaces

Syntaxe #

  • la classe ClassB étend ClassA {…}
  • la classe ClassB implémente InterfaceA {…}
  • l’interface InterfaceB étend l’interfaceA {…}
  • la classe ClassB étend ClassA implémente InterfaceC, InterfaceD {…}
  • la classe abstraite AbstractClassB étend ClassA {…}
  • la classe abstraite AbstractClassB étend AbstractClassA {…}
  • la classe abstraite AbstractClassB étend ClassA implémente InterfaceC, InterfaceD {…}

L’héritage est souvent combiné avec des génériques afin que la classe de base ait un ou plusieurs paramètres de type. Voir [Création d’une classe générique][1].

[1] : https://www.wikiod.com/fr/java/generiques#Création d’une classe générique

Cours abstraits #

Une classe abstraite est une classe marquée avec le mot clé abstract. Contrairement à la classe non abstraite, elle peut contenir des méthodes abstraites - sans implémentation. Il est cependant valide de créer une classe abstraite sans méthodes abstraites.

Une classe abstraite ne peut pas être instanciée. Il peut être sous-classé (étendu) tant que la sous-classe est soit également abstraite, soit implémente toutes les méthodes marquées comme abstraites par les super classes.

Un exemple de classe abstraite :

public abstract class Component {
    private int x, y;
    
    public setPosition(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public abstract void render();
}

La classe doit être marquée abstraite, lorsqu’elle a au moins une méthode abstraite. Une méthode abstraite est une méthode qui n’a pas d’implémentation. D’autres méthodes peuvent être déclarées dans une classe abstraite qui ont une implémentation afin de fournir un code commun pour toutes les sous-classes.

Tenter d’instancier cette classe fournira une erreur de compilation :

//error: Component is abstract; cannot be instantiated   
Component myComponent = new Component();

Cependant, une classe qui étend Component et fournit une implémentation pour toutes ses méthodes abstraites et peut être instanciée.

public class Button extends Component {

    @Override
    public void render() {
        //render a button
    }
}

public class TextBox extends Component {

    @Override
    public void render() {
        //render a textbox
    }
}

Les instances de classes héritées peuvent également être converties en classe parent (héritage normal) et elles fournissent un effet polymorphe lorsque la méthode abstraite est appelée.

Component myButton = new Button();
Component myTextBox = new TextBox();

myButton.render(); //renders a button
myTextBox.render(); //renders a text box

Cours abstraits vs Interfaces

Les classes abstraites et les interfaces fournissent toutes deux un moyen de définir des signatures de méthode tout en exigeant que la classe d’extension/d’implémentation fournisse l’implémentation.

Il existe deux différences essentielles entre les classes abstraites et les interfaces :

  • Une classe ne peut étendre qu’une seule classe, mais peut implémenter de nombreuses interfaces.
  • Une classe abstraite peut contenir des champs d’instance (non “statiques”), mais les interfaces ne peuvent contenir que des champs “statiques”.

Les méthodes déclarées dans les interfaces ne pouvaient pas contenir d’implémentations, de sorte que les classes abstraites étaient utilisées lorsqu’il était utile de fournir des méthodes supplémentaires que les implémentations appelaient les méthodes abstraites.

Java 8 permet aux interfaces de contenir méthodes par défaut, généralement [mis en œuvre à l’aide des autres méthodes de l’interface](https:// stackoverflow.com/documentation/java/113/default-methods/456/accessing-other-interface-methods-within-default-method#t=201607191734172336331), rendant les interfaces et les classes abstraites tout aussi puissantes à cet égard.

Sous-classes anonymes des classes abstraites

Par commodité, Java permet l’instanciation d’instances anonymes de sous-classes de classes abstraites, qui fournissent des implémentations pour les méthodes abstraites lors de la création du nouvel objet. En utilisant l’exemple ci-dessus, cela pourrait ressembler à ceci :

Component myAnonymousComponent = new Component() {
    @Override
    public void render() {
        // render a quick 1-time use component
    }
}

Utilisation de ‘final’ pour restreindre l’héritage et le remplacement #

Classes finales #

Lorsqu’il est utilisé dans une déclaration class, le modificateur final empêche la déclaration d’autres classes qui étendent la classe. Une classe “finale” est une classe “feuille” dans la hiérarchie des classes d’héritage.

// This declares a final class
final class MyFinalClass {
    /* some code */
}

// Compilation error: cannot inherit from final MyFinalClass
class MySubClass extends MyFinalClass {
    /* more code */
}

Cas d’utilisation pour les classes finales #

Les classes finales peuvent être combinées avec un constructeur “privé” pour contrôler ou empêcher l’instanciation d’une classe. Cela peut être utilisé pour créer une soi-disant “classe utilitaire” qui définit uniquement les membres statiques ; c’est-à-dire des constantes et des méthodes statiques.

public final class UtilityClass {

    // Private constructor to replace the default visible constructor
    private UtilityClass() {}

    // Static members can still be used as usual
    public static int doSomethingCool() {
        return 123;
    }

}

Les classes immuables doivent également être déclarées comme “finales”. (Une classe immuable est une classe dont les instances ne peuvent pas être modifiées après leur création ; voir la rubrique I[mmutable Objects][1]. ) En faisant cela, vous rendez impossible la création d’une sous-classe mutable d’une classe immuable. Cela violerait le [principe de substitution de Liskov][2] qui exige qu’un sous-type obéisse au “contrat comportemental” de ses supertypes.

D’un point de vue pratique, déclarer une classe immuable comme “finale” facilite le raisonnement sur le comportement du programme. Il résout également les problèmes de sécurité dans le scénario où un code non approuvé est exécuté dans un bac à sable de sécurité. (Par exemple, étant donné que String est déclaré comme final, une classe de confiance n’a pas à s’inquiéter d’être amenée à accepter une sous-classe mutable, que l’appelant non approuvé pourrait alors modifier subrepticement.)

Un inconvénient des classes “finales” est qu’elles ne fonctionnent pas avec certains frameworks factices tels que Mockito. Mise à jour : Mockito version 2 prend désormais en charge le mocking des classes finales.

Méthodes finales #

Le modificateur final peut également être appliqué aux méthodes pour éviter qu’elles ne soient remplacées dans les sous-classes :

public class MyClassWithFinalMethod {

    public final void someMethod() {
    }
}

public class MySubClass extends MyClassWithFinalMethod {

    @Override
    public void someMethod() { // Compiler error (overridden method is final)
    }
}

Les méthodes finales sont généralement utilisées lorsque vous souhaitez restreindre ce qu’une sous-classe peut modifier dans une classe sans interdire complètement les sous-classes.

Le modificateur final peut également être appliqué aux variables, mais la signification de final pour les variables n’est pas liée à l’héritage.

[1] : https://www.wikiod.com/fr/java/objets-immuables [2] : https://www.wikiod.com/fr/java/heritage#Le principe de substitution de Liskov

Héritage #

Avec l’utilisation du mot-clé `extends’ parmi les classes, toutes les propriétés de la superclasse (également connue sous le nom de Classe parente ou Classe de base) sont présentes dans la sous-classe (également appelée Classe enfant ou Derived Classer)

public class BaseClass {

    public void baseMethod(){
        System.out.println("Doing base class stuff");
    }
}

public class SubClass extends BaseClass {

}

Les instances de SubClass ont hérité de la méthode baseMethod() :

SubClass s = new SubClass();
s.baseMethod();  //Valid, prints "Doing base class stuff"

Du contenu supplémentaire peut être ajouté à une sous-classe. Cela permet des fonctionnalités supplémentaires dans la sous-classe sans aucune modification de la classe de base ou de toute autre sous-classe de cette même classe de base :

public class Subclass2 extends BaseClass {

    public void anotherMethod() {
        System.out.println("Doing subclass2 stuff");
    }
}

Subclass2 s2 = new Subclass2();
s2.baseMethod(); //Still valid , prints "Doing base class stuff"
s2.anotherMethod(); //Also valid, prints "Doing subclass2 stuff"

Les champs sont également hérités :

public class BaseClassWithField {

    public int x;

}

public class SubClassWithField extends BaseClassWithField {

    public SubClassWithField(int x) {
        this.x = x; //Can access fields
    }
}

les champs et méthodes “privés” existent toujours dans la sous-classe, mais ne sont pas accessibles :

public class BaseClassWithPrivateField {

    private int x = 5;

    public int getX() {
        return x;
    }
}

public class SubClassInheritsPrivateField extends BaseClassWithPrivateField {

    public void printX() {
        System.out.println(x); //Illegal, can't access private field x
        System.out.println(getX()); //Legal, prints 5
    }
}

SubClassInheritsPrivateField s = new SubClassInheritsPrivateField();
int x = s.getX(); //x will have a value of 5.

En Java, chaque classe peut étendre au plus une autre classe.

public class A{}
public class B{}
public class ExtendsTwoClasses extends A, B {} //Illegal

C’est ce qu’on appelle l’héritage multiple, et bien que cela soit légal dans certains langages, Java ne le permet pas avec les classes.

En conséquence, chaque classe a une chaîne ancestrale sans ramification de classes menant à “Objet”, dont toutes les classes descendent.

[2] : https://www.wikiod.com/fr/java

Le principe de substitution de Liskov #

La substituabilité est un principe de la programmation orientée objet introduit par Barbara Liskov dans un discours de conférence de 1987 déclarant que, si la classe B est une sous-classe de la classe A, alors partout où A` `` est attendu, B``` peut être utilisé à la place :

class A {...}
class B extends A {...}

public void method(A obj) {...}

A a = new B(); // Assignment OK
method(new B()); // Passing as parameter OK

Ceci s’applique également lorsque le type est une interface, où il n’y a pas besoin de relation hiérarchique entre les objets :

interface Foo {
    void bar();
}

class A implements Foo {
    void bar() {...}
}

class B implements Foo {
    void bar() {...}
}

List<Foo> foos = new ArrayList<>();
foos.add(new A()); // OK
foos.add(new B()); // OK

Désormais, la liste contient des objets qui ne font pas partie de la même hiérarchie de classes.

Héritage statique #

La méthode statique peut être héritée de la même manière que les méthodes normales, mais contrairement aux méthodes normales, il est impossible de créer des méthodes “[abstract][1]” afin de forcer le remplacement de la méthode statique. Écrire une méthode avec la même signature qu’une méthode statique dans une super classe semble être une forme de remplacement, mais en réalité, cela crée simplement une nouvelle fonction cache l’autre.

public class BaseClass {
    
    public static int num = 5;

    public static void sayHello() {
        System.out.println("Hello");
    }

    public static void main(String[] args) {
        BaseClass.sayHello();
        System.out.println("BaseClass's num: " + BaseClass.num);
            
        SubClass.sayHello();
        //This will be different than the above statement's output, since it runs
        //A different method
        SubClass.sayHello(true);
        
        StaticOverride.sayHello();
        System.out.println("StaticOverride's num: " + StaticOverride.num);
    }
}

public  class SubClass extends BaseClass {
    
    //Inherits the sayHello function, but does not override it   
    public static void sayHello(boolean test) {
        System.out.println("Hey");
    }
}

public static class StaticOverride extends BaseClass {

    //Hides the num field from BaseClass
    //You can even change the type, since this doesn't affect the signature
    public static String num = "test";
        
    //Cannot use @Override annotation, since this is static
    //This overrides the sayHello method from BaseClass
    public static void sayHello() {
        System.out.println("Static says Hi");
    }

}

L’exécution de l’une de ces classes produit la sortie :

Hello
BaseClass's num: 5
Hello
Hey
Static says Hi
StaticOverride's num: test

Notez que contrairement à l’héritage normal, les méthodes d’héritage statique ne sont pas masquées. Vous pouvez toujours appeler la méthode de base sayHello en utilisant BaseClass.sayHello(). Mais les classes héritent des méthodes statiques si aucune méthode avec la même signature n’est trouvée dans la sous-classe. Si les signatures de deux méthodes varient, les deux méthodes peuvent être exécutées à partir de la sous-classe, même si le nom est le même.

Les champs statiques se cachent de la même manière.

[1] : https://www.wikiod.com/fr/java/heritage#Cours abstraits

Classe abstraite et utilisation de l’interface : relation “Est-un” vs capacité “A-un” #

Quand utiliser des classes abstraites : pour implémenter le même comportement ou un comportement différent entre plusieurs objets liés

Quand utiliser des interfaces : pour implémenter un contrat par plusieurs objets non liés

Les classes abstraites créent des relations “est un” tandis que les interfaces fournissent la capacité “a un”.

Cela peut être vu dans le code ci-dessous:

public class InterfaceAndAbstractClassDemo{
    public static void main(String args[]){
        
        Dog dog = new Dog("Jack",16);
        Cat cat = new Cat("Joe",20);
            
        System.out.println("Dog:"+dog);
        System.out.println("Cat:"+cat);
        
        dog.remember();
        dog.protectOwner();
        Learn dl = dog;
        dl.learn();
                
        cat.remember();
        cat.protectOwner();
        
        Climb c = cat;
        c.climb();
        
        Man man = new Man("Ravindra",40);
        System.out.println(man);
        
        Climb cm = man;
        cm.climb();
        Think t = man;
        t.think();
        Learn l = man;
        l.learn();
        Apply a = man;
        a.apply();
    }
}

abstract class Animal{
    String name;
    int lifeExpentency;
    public Animal(String name,int lifeExpentency ){
        this.name = name;
        this.lifeExpentency=lifeExpentency;
    }
    public abstract void remember();
    public abstract void protectOwner();
    
    public String toString(){
        return this.getClass().getSimpleName()+":"+name+":"+lifeExpentency;
    }
}
class Dog extends Animal implements Learn{
    
    public Dog(String name,int age){
        super(name,age);
    }
    public void remember(){
        System.out.println(this.getClass().getSimpleName()+" can remember for 5 minutes");
    }
    public void protectOwner(){
        System.out.println(this.getClass().getSimpleName()+ " will protect owner");
    }
    public void learn(){
        System.out.println(this.getClass().getSimpleName()+ " can learn:");
    }
}
class Cat extends Animal implements Climb {
    public Cat(String name,int age){
        super(name,age);
    }
    public void remember(){
        System.out.println(this.getClass().getSimpleName() + " can remember for 16 hours");
    }
    public void protectOwner(){
        System.out.println(this.getClass().getSimpleName()+ " won't protect owner");
    }
    public void climb(){
        System.out.println(this.getClass().getSimpleName()+ " can climb");
    }
}
interface Climb{
    void climb();
}
interface Think {
    void think();
}

interface Learn {
    void learn();
}
interface Apply{
    void apply();
}

class Man implements Think,Learn,Apply,Climb{
    String name;
    int age;

    public Man(String name,int age){
        this.name = name;
        this.age = age;
    }
    public void think(){
        System.out.println("I can think:"+this.getClass().getSimpleName());
    }
    public void learn(){
        System.out.println("I can learn:"+this.getClass().getSimpleName());
    }
    public void apply(){
        System.out.println("I can apply:"+this.getClass().getSimpleName());
    }
    public void climb(){
        System.out.println("I can climb:"+this.getClass().getSimpleName());
    }
    public String toString(){
        return "Man :"+name+":Age:"+age;
    }
}

Notes clés :

  1. Animal est une classe abstraite avec des attributs partagés : name et lifeExpectancy et des méthodes abstraites : remember() et protectOwner(). Chien et Chat sont des Animaux qui ont implémenté les méthodes remember() et protectOwner().

  2. Chat peut grimper() mais Chien ne peut pas. Chien peut penser() mais Chat ne peut pas. Ces capacités spécifiques sont ajoutées à Cat et Dog par implémentation.

  3. “L’homme” n’est pas un “animal”, mais il peut “penser”, “apprendre”, “appliquer” et “escalader”.

  4. ‘Chat’ n’est pas un ‘Homme’ mais il peut ‘Grimper’.

  5. ‘Chien’ n’est pas un ‘Homme’ mais il peut ‘apprendre’

  6. Man n’est ni un Cat ni un Dog mais peut avoir certaines des capacités des deux derniers sans étendre Animal, Cat ou Dog. Cela se fait avec les interfaces.

  7. Même si Animal est une classe abstraite, elle a un constructeur, contrairement à une interface.

TL ; RD :

Les classes non liées peuvent avoir des capacités via des interfaces, mais les classes liées modifient le comportement via l’extension des classes de base.

Reportez-vous à la documentation Java [page][1] pour comprendre lequel utiliser dans un cas d’utilisation spécifique.

Envisagez d’utiliser des classes abstraites si…

  1. Vous souhaitez partager du code entre plusieurs classes étroitement liées.
  2. Vous vous attendez à ce que les classes qui étendent votre classe abstraite aient de nombreuses méthodes ou champs communs, ou nécessitent des modificateurs d’accès autres que public (tels que protected et private).
  3. Vous souhaitez déclarer des champs non statiques ou non finaux.

Envisagez d’utiliser des interfaces si…

  1. Vous vous attendez à ce que des classes non liées implémentent votre interface. Par exemple, de nombreux objets non liés peuvent implémenter l’interface Serializable.
  2. Vous souhaitez spécifier le comportement d’un type de données particulier, mais ne vous souciez pas de savoir qui implémente son comportement.
  3. Vous souhaitez profiter de l’héritage multiple de type.

[1] : https://docs.oracle.com/javase/tutorial/java/IandI/abstract.html

Héritage et méthodes statiques #

En Java, les classes parent et enfant peuvent toutes deux avoir des méthodes statiques portant le même nom. Mais dans de tels cas, l’implémentation de la méthode statique dans l’enfant est hiding parent class’ implémentation, ce n’est pas une méthode qui remplace. Par exemple:

class StaticMethodTest {

  // static method and inheritance
  public static void main(String[] args) {
    Parent p = new Child();
    p.staticMethod(); // prints Inside Parent
    ((Child) p).staticMethod(); // prints Inside Child
  }

  static class Parent {
    public static void staticMethod() {
      System.out.println("Inside Parent");
    }
  }

  static class Child extends Parent {
    public static void staticMethod() {
      System.out.println("Inside Child");
    }
  }
}

Les méthodes statiques sont liées à une classe et non à une instance et cette liaison de méthode se produit au moment de la compilation. Puisque lors du premier appel à staticMethod(), la référence de classe parent p a été utilisée, la version Parent de staticMethod() est invoquée. Dans le second cas, nous avons transtypé p dans la classe Child, la staticMethod() de Child est exécutée.

Ombre variable #

Les variables sont SHADOWED et les méthodes sont OVERRIDDEN. La variable qui sera utilisée dépend de la classe dont la variable est déclarée. La méthode qui sera utilisée dépend de la classe réelle de l’objet référencé par la variable.

class Car {
    public int gearRatio = 8;

    public String accelerate() {
        return "Accelerate : Car";
    }
}

class SportsCar extends Car {
    public int gearRatio = 9;

    public String accelerate() {
        return "Accelerate : SportsCar";
    }

    public void test() {

    }


    public static void main(String[] args) {

        Car car = new SportsCar();
        System.out.println(car.gearRatio + "  " + car.accelerate());
        // will print out 8  Accelerate : SportsCar
    }
}

Rétrécissement et élargissement des références d’objets #

Caster une instance d’une classe de base en une sous-classe comme dans : b = (B) a; s’appelle restriction (car vous essayez de restreindre l’objet de la classe de base à un objet de classe plus spécifique) et nécessite un type explicite -moulage.

Caster une instance d’une sous-classe en une classe de base comme dans : A a = b; s’appelle élargir et ne nécessite pas de transtypage.

Pour illustrer, considérez les déclarations de classe suivantes et le code de test :

class Vehicle {
}

class Car extends Vehicle {
}

class Truck extends Vehicle {
}

class MotorCycle extends Vehicle {
}

class Test {

    public static void main(String[] args) {
    
        Vehicle vehicle = new Car();
        Car car = new Car();        
    
        vehicle = car; // is valid, no cast needed

        Car c = vehicle // not valid
        Car c = (Car) vehicle; //valid
    }
}

L’instruction Vehicle vehicle = new Car(); est une instruction Java valide. Chaque instance de Car est également un Vehicle. Par conséquent, l’affectation est légale sans avoir besoin d’un transtypage explicite.

Par contre, Voiture c = véhicule; n’est pas valide. Le type statique de la variable vehicle est Vehicle, ce qui signifie qu’il peut faire référence à une instance de Car, Truck, MotorCycleou à toute autre sous-classe actuelle ou future deVehicle. (Ou en fait, une instance de Vehicleelle-même, puisque nous ne l'avons pas déclarée comme une classeabstract.) L'affectation ne peut pas être autorisée, car cela pourrait conduire à ce que carfasse référence à une instanceTruck`.

Pour éviter cette situation, nous devons ajouter un transtypage explicite :

Car c = (Car) vehicle;

Le transtypage indique au compilateur que nous attendons que la valeur de vehicle soit une Car ou une sous-classe de Car. Si nécessaire, le compilateur insère du code pour effectuer une vérification de type à l’exécution. Si la vérification échoue, une ClassCastException sera levée lors de l’exécution du code.

Notez que tous les transtypages ne sont pas valides. Par exemple:

String s = (String) vehicle;  // not valid

Le compilateur Java sait qu’une instance dont le type est compatible avec Vehicle ne peut jamais être de type compatible avec String. Le transtypage ne pourrait jamais réussir, et les mandats JLS que cela donne dans une erreur de compilation.

Programmation sur une interface #

L’idée derrière la programmation d’une interface est de baser le code principalement sur les interfaces et de n’utiliser que des classes concrètes au moment de l’instanciation. Dans ce contexte, un bon code traitant par ex. Les collections Java ressembleront à ceci (pas que la méthode elle-même soit d’aucune utilité, juste une illustration):

public <T> Set<T> toSet(Collection<T> collection) {
  return Sets.newHashSet(collection);
}

tandis que le mauvais code pourrait ressembler à ceci :

public <T> HashSet<T> toSet(ArrayList<T> collection) {
  return Sets.newHashSet(collection);
}

Non seulement le premier peut être appliqué à un plus large choix d’arguments, mais ses résultats seront plus compatibles avec le code fourni par d’autres développeurs qui adhèrent généralement au concept de programmation d’une interface. Cependant, les raisons les plus importantes d’utiliser le premier sont:

  • la plupart du temps, le contexte dans lequel le résultat est utilisé ne nécessite pas et ne devrait pas nécessiter autant de détails que la mise en œuvre concrète fournit ;
  • adhérer à une interface force un code plus propre et moins de hacks, comme une autre méthode publique est ajoutée à une classe servant un scénario spécifique ;
  • le code est plus testable car les interfaces sont facilement simulables ;
  • enfin, le concept aide même si une seule implémentation est attendue (au moins pour la testabilité).

Alors, comment appliquer facilement le concept de programmation à une interface lors de l’écriture d’un nouveau code en ayant à l’esprit une implémentation particulière ? Une option que nous utilisons couramment est une combinaison des modèles suivants :

  • programmation sur une interface
  • usine
  • constructeur

L’exemple suivant basé sur ces principes est une version simplifiée et tronquée d’une implémentation RPC écrite pour un certain nombre de protocoles différents :

public interface RemoteInvoker {
  <RQ, RS> CompletableFuture<RS> invoke(RQ request, Class<RS> responseClass);
}

L’interface ci-dessus n’est pas censée être instanciée directement via une fabrique, à la place, nous dérivons d’autres interfaces plus concrètes, une pour l’invocation HTTP et une pour AMQP, chacune ayant alors une fabrique et un constructeur pour construire des instances, qui à leur tour sont également des instances de l’interface ci-dessus :

public interface AmqpInvoker extends RemoteInvoker {
  static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
    return new AmqpInvokerBuilder(instanceId, factory);
  }
}

Les instances de RemoteInvoker à utiliser avec AMQP peuvent désormais être construites aussi facilement que (ou plus compliquées selon le constructeur):

RemoteInvoker invoker = AmqpInvoker.with(instanceId, factory)
  .requestRouter(router)
  .build();

Et l’invocation d’une requête est aussi simple que :

Response res = invoker.invoke(new Request(data), Response.class).get();

Étant donné que Java 8 permet de placer des méthodes statiques directement dans les interfaces, la fabrique intermédiaire est devenue implicite dans le code ci-dessus remplacé par AmqpInvoker.with(). Dans Java antérieur à la version 8, le même effet peut être obtenu avec une classe interne “Factory” :

public interface AmqpInvoker extends RemoteInvoker {
  class Factory {
    public static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
      return new AmqpInvokerBuilder(instanceId, factory);
    }
  }
}

L’instanciation correspondante se transformerait alors en :

RemoteInvoker invoker = AmqpInvoker.Factory.with(instanceId, factory)
  .requestRouter(router)
  .build();

Le constructeur utilisé ci-dessus pourrait ressembler à ceci (bien qu’il s’agisse d’une simplification car le constructeur actuel permet de définir jusqu’à 15 paramètres s’écartant des valeurs par défaut). Notez que la construction n’est pas publique, elle n’est donc utilisable qu’à partir de l’interface AmqpInvoker ci-dessus :

public class AmqpInvokerBuilder {
  ...
  AmqpInvokerBuilder(String instanceId, ConnectionFactory factory) {
    this.instanceId = instanceId;
    this.factory = factory;
  }

  public AmqpInvokerBuilder requestRouter(RequestRouter requestRouter) {
    this.requestRouter = requestRouter;
    return this;
  }

  public AmqpInvoker build() throws TimeoutException, IOException {
    return new AmqpInvokerImpl(instanceId, factory, requestRouter);
  }
}

Généralement, un constructeur peut également être généré à l’aide d’un outil comme FreeBuilder.

Enfin, l’implémentation standard (et la seule attendue) de cette interface est définie comme une classe package-local pour imposer l’utilisation de l’interface, de la fabrique et du constructeur :

class AmqpInvokerImpl implements AmqpInvoker {
  AmqpInvokerImpl(String instanceId, ConnectionFactory factory, RequestRouter requestRouter) {
    ...
  }

  @Override
  public <RQ, RS> CompletableFuture<RS> invoke(final RQ request, final Class<RS> respClass) {
    ...
  }
}

Pendant ce temps, ce modèle s’est avéré très efficace pour développer tout notre nouveau code, quelle que soit la simplicité ou la complexité de la fonctionnalité.

Remplacement dans l’héritage #

La redéfinition dans Héritage est utilisée lorsque vous utilisez une méthode déjà définie à partir d’une super classe dans une sous-classe, mais d’une manière différente de la manière dont la méthode a été conçue à l’origine dans la super classe. Le remplacement permet à l’utilisateur de réutiliser le code en utilisant le matériel existant et en le modifiant pour mieux répondre aux besoins de l’utilisateur.

L’exemple suivant montre comment ClassB remplace la fonctionnalité de ClassA en modifiant ce qui est envoyé via la méthode d’impression :

Exemple:

public static void main(String[] args) {
    ClassA a = new ClassA();
    ClassA b = new ClassB();
    a.printing();
    b.printing();
}

class ClassA {
    public void printing() {        
        System.out.println("A");
    }
}

class ClassB extends ClassA {
    public void printing() {
         System.out.println("B");
    }
}
