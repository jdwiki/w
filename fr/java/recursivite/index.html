La récursivité se produit lorsqu’une méthode s’appelle elle-même. Une telle méthode est appelée récursive. Une méthode récursive peut être plus concise qu’une approche non récursive équivalente. Cependant, pour une récursivité profonde, une solution itérative peut parfois consommer moins d’espace de pile fini d’un thread.

Cette rubrique comprend des exemples de récursivité en Java.

Conception d’une méthode récursive #

Lors de la conception d’une méthode récursive, gardez à l’esprit que vous avez besoin :

  • Cas de base. Cela définira quand votre récursivité s’arrêtera et affichera le résultat. Le cas de base dans l’exemple factoriel est :

     if (n <= 1) {
     return 1;
 }

  • Appel récursif. Dans cette instruction, vous rappelez la méthode avec un paramètre modifié. L’appel récursif dans l’exemple factoriel ci-dessus est :

     else {
     return n * factorial(n - 1);
 }

Production #

Dans cet exemple, vous calculez le n-ième nombre factoriel. Les premiers factoriels sont :

0 ! = 1

1 ! = 1

2 ! = 1 x 2 = 2

3 ! = 1 x 2 x 3 = 6

4 ! = 1 × 2 × 3 × 4 = 24

…

Élimination de Java et Tail-call #

Les compilateurs Java actuels (jusqu’à Java 9 inclus) n’effectuent pas d’élimination des appels de fin. Cela peut avoir un impact sur les performances des algorithmes récursifs, et si la récursivité est suffisamment profonde, cela peut entraîner des plantages StackOverflowError ; voir https://www.wikiod.com/fr/java/recursivite#La récursivité profonde est problématique en Java

La récursivité profonde est problématique en Java #

Considérez la méthode naïve suivante pour additionner deux nombres positifs en utilisant la récursivité :

public static int add(int a, int b) {
    if (a == 0) {
        return b;
    } else {
        return add(a - 1, b + 1);  // TAIL CALL
    }
}

Ceci est algorithmiquement correct, mais il a un problème majeur. Si vous appelez add avec un grand a, il plantera avec une StackOverflowError, sur n’importe quelle version de Java jusqu’à (au moins) Java 9.

Dans un langage de programmation fonctionnel typique (et de nombreux autres langages), le compilateur optimise [tail recursion][1]. Le compilateur remarquerait que l’appel à add (à la ligne étiquetée) est un [appel final][2], et réécrirait effectivement la récursivité comme une boucle. Cette transformation est appelée élimination de l’appel de queue.

Cependant, les compilateurs Java de la génération actuelle n’effectuent pas d’élimination des appels de queue. (Ce n’est pas un simple oubli. Il y a des raisons techniques substantielles à cela ; voir ci-dessous.) Au lieu de cela, chaque appel récursif de add provoque l’allocation d’une nouvelle trame sur la pile du thread. Par exemple, si vous appelez add(1000, 1), il faudra 1000 appels récursifs pour arriver à la réponse 1001.

Le problème est que la taille de la pile de threads Java est fixée lors de la création du thread. (Cela inclut le thread “principal” dans un programme à thread unique.) Si trop de cadres de pile sont alloués, la pile débordera. La JVM le détectera et lancera une StackOverflowError.

Une approche pour résoudre ce problème consiste simplement à utiliser une pile plus grande. Il existe des options JVM qui contrôlent la taille par défaut d’une pile, et vous pouvez également spécifier la taille de la pile en tant que paramètre de constructeur Thread. Malheureusement, cela ne fait que “retarder” le débordement de la pile. Si vous devez effectuer un calcul qui nécessite une pile encore plus grande, le StackOverflowError revient.

La vraie solution est d’identifier les algorithmes récursifs où une récursivité profonde est probable, et d’effectuer * manuellement * l’optimisation de l’appel final au niveau du code source. Par exemple, notre méthode add peut être réécrite comme suit :

public static int add(int a, int b) {
    while (a != 0) {
       a = a - 1;
       b = b + 1;
    }
    return b;
}

(Évidemment, il existe de meilleures façons d’ajouter deux entiers. Ce qui précède est simplement pour illustrer l’effet de l’élimination manuelle de l’appel final.)

Pourquoi l’élimination des appels terminaux n’est pas (encore) implémentée en Java #

Il existe un certain nombre de raisons pour lesquelles l’ajout de l’élimination des appels de queue à Java n’est pas facile. Par exemple:

  • Certains codes pourraient s’appuyer sur StackOverflowError pour (par exemple) placer une limite sur la taille d’un problème de calcul.
  • Les responsables de la sécurité Sandbox s’appuient souvent sur l’analyse de la pile d’appels pour décider d’autoriser ou non un code non privilégié à effectuer une action privilégiée.

Comme John Rose l’explique dans [“Tail calls in the VM”][3] :

  • “Les effets de la suppression du cadre de pile de l’appelant sont visibles pour certaines API, notamment les vérifications de contrôle d’accès et le traçage de pile. C’est comme si l’appelant de l’appelant avait directement appelé l’appelé. Tous les privilèges possédés par l’appelant sont supprimés après le transfert du contrôle à l’appelé. Cependant, la liaison et l’accessibilité de la méthode de l’appelé sont calculées avant le transfert de contrôle et tiennent compte de l’appelant final.”*

En d’autres termes, l’élimination des appels terminaux pourrait amener une méthode de contrôle d’accès à penser à tort qu’une API sensible à la sécurité était appelée par un code de confiance.

L’idée de base de la récursivité #

Qu’est-ce que la récursivité :

En général, la récursivité se produit lorsqu’une fonction s’invoque elle-même, directement ou indirectement. Par exemple:

// This method calls itself "infinitely"
public void useless() {
    useless();  // method calls itself (directly)
}

Conditions d’application de la récursivité à un problème :

Il existe deux conditions préalables à l’utilisation de fonctions récursives pour résoudre un problème spécifique :

  1. Il doit y avoir une condition de base pour le problème, qui sera le point final de la récursivité. Lorsqu’une fonction récursive atteint la condition de base, elle n’effectue plus d’appels récursifs (plus profonds).

  2. Chaque niveau de récursivité devrait tenter un problème plus petit. La fonction récursive divise donc le problème en parties de plus en plus petites. En supposant que le problème est fini, cela garantira que la récursivité se termine.

En Java, il existe une troisième condition préalable : il ne doit pas être nécessaire de revenir trop profondément pour résoudre le problème ; voir https://www.wikiod.com/fr/java/recursivite#La récursivité profonde est problématique en Java

Exemple

La fonction suivante calcule les factorielles à l’aide de la récursivité. Remarquez comment la méthode “factorielle” s’appelle elle-même dans la fonction. Chaque fois qu’elle s’appelle elle-même, elle réduit le paramètre n de 1. Lorsque n atteint 1 (la condition de base), la fonction ne récursera pas plus profondément.

public int factorial(int n) {
    if (n <= 1) { // the base condition 
        return 1;
    } else {
        return n * factorial(n - 1);
    }
}

Ce n’est pas un moyen pratique de calculer les factorielles en Java, car il ne prend pas en compte le dépassement d’entier ou le dépassement de pile d’appels (c’est-à-dire les exceptions StackOverflowError) pour les grandes valeurs de n.

Types de récursivité #

La récursivité peut être classée comme Head Recursion ou Tail Recursion, selon l’endroit où l’appel de méthode récursive est placé.

Dans head recursion, l’appel récursif, lorsqu’il se produit, vient avant les autres traitements dans la fonction (pensez à ce qu’il se produise en haut, ou en tête, de la fonction).

Dans tail recursion, c’est le contraire : le traitement a lieu avant l’appel récursif. Choisir entre les deux styles récursifs peut sembler arbitraire, mais le choix peut faire toute la différence.

Une fonction avec un chemin avec un seul appel récursif au début du chemin utilise ce qu’on appelle la récursivité de tête. La fonction factorielle d’une exposition précédente utilise la récursivité de la tête. La première chose qu’il fait une fois qu’il a déterminé que la récursivité est nécessaire est de s’appeler avec le paramètre décrémenté. Une fonction avec un seul appel récursif à la fin d’un chemin utilise la récursivité terminale.

public void tail(int n)              public void head(int n)
{                                       {
   if(n == 1)                             if(n == 0)
      return;                                return;
   else                                   else
      System.out.println(n);                 head(n-1);

   tail(n-1);                              System.out.println(n);
}                                        }

Si l’appel récursif se produit à la fin d’une méthode, il s’agit d’une “récursion terminale”. La récursivité de queue est “similaire à une boucle”. La méthode exécute toutes les instructions avant de passer au prochain appel récursif.

Si l’appel récursif se produit au “début d’une méthode, il s’agit d’une récursion de tête”. La “méthode enregistre l’état avant de passer au prochain appel récursif”.

Référence : http://stackoverflow.com/questions/21426688/the-difference-between-head-tail-recursion

Calcul du nième nombre de Fibonacci #

La méthode suivante calcule le Nième nombre de Fibonacci en utilisant la récursivité.

public int fib(final int n) {
    if (n > 2) {
        return fib(n - 2) + fib(n - 1);
    }
    return 1;
}

La méthode implémente un cas de base (n <= 2) et un cas récursif (n>2). Ceci illustre l’utilisation de la récursivité pour calculer une relation récursive.

Cependant, bien que cet exemple soit illustratif, il est également inefficace : chaque instance de la méthode appellera la fonction elle-même deux fois, ce qui entraînera une croissance exponentielle du nombre de fois où la fonction est appelée à mesure que N augmente. La fonction ci-dessus est O(2N), mais une solution itérative équivalente a une complexité O(N). De plus, il existe une expression de “forme fermée” qui peut être évaluée en O(N) multiplications à virgule flottante.

Calcul de la puissance N d’un nombre #

La méthode suivante calcule la valeur de num élevée à la puissance de exp en utilisant la récursivité :

public long power(final int num, final int exp) {
    if (exp == 0) {
        return 1;
    }
    if (exp == 1) {
        return num;
    }
    return num * power(num, exp - 1);
}

Ceci illustre les principes évoqués ci-dessus : la méthode récursive implémente un cas de base (deux cas, n = 0 et n = 1) qui termine la récursivité, et un cas récursif qui appelle à nouveau la méthode. Cette méthode est O(N) et peut être réduite à une simple boucle en utilisant l’optimisation des appels de queue.

Traversée d’une structure de données arborescente avec récursivité #

Considérez la classe Node ayant 3 membres de données, un pointeur enfant gauche et un pointeur enfant droit comme ci-dessous.

public class Node {
    public int data;
    public Node left;
    public Node right;
    
    public Node(int data){
        this.data = data;
    }
}

Nous pouvons parcourir l’arbre construit en connectant plusieurs objets de la classe Node comme ci-dessous, la traversée est appelée traversée dans l’ordre de l’arbre.

public static void inOrderTraversal(Node root) {
        if (root != null) {          
            inOrderTraversal(root.left); // traverse left sub tree
            System.out.print(root.data + " "); // traverse current node
            inOrderTraversal(root.right); // traverse right sub tree
        }
    }

Comme démontré ci-dessus, en utilisant la récursivité, nous pouvons parcourir la structure de données arborescente sans utiliser aucune autre structure de données, ce qui n’est pas possible avec l’approche itérative.

StackOverflowError & récursivité en boucle #

Si un appel récursif va “trop ​​​​profondément”, cela se traduit par une StackOverflowError. Java alloue une nouvelle trame pour chaque appel de méthode sur la pile de son thread. Cependant, l’espace de la pile de chaque thread est limité. Trop d’images sur la pile entraînent le débordement de pile (SO).

Exemple #

public static void recursion(int depth) {
    if (depth > 0) {
        recursion(depth-1);
    }
}

L’appel de cette méthode avec de grands paramètres (par exemple, recursion(50000) entraînera probablement un débordement de pile. La valeur exacte dépend de la taille de la pile de threads, qui à son tour dépend de la construction du thread, des paramètres de ligne de commande tels que -Xss , ou la taille par défaut de la JVM.

Solution de contournement #

Une récursivité peut être convertie en boucle en stockant les données de chaque appel récursif dans une structure de données. Cette structure de données peut être stockée sur le tas plutôt que sur la pile de threads.

En général, les données nécessaires pour restaurer l’état d’une invocation de méthode peuvent être stockées dans une pile et une boucle while peut être utilisée pour “simuler” les appels récursifs. Les données pouvant être requises incluent :

  • l’objet pour lequel la méthode a été appelée (méthodes d’instance uniquement)
  • les paramètres de la méthode
  • variables locales
  • la position actuelle dans l’exécution ou la méthode

Exemple #

La classe suivante permet l’impression récursive d’une arborescence jusqu’à une profondeur spécifiée.

public class Node {

    public int data;
    public Node left;
    public Node right;

    public Node(int data) {
        this(data, null, null);
    }

    public Node(int data, Node left, Node right) {
        this.data = data;
        this.left = left;
        this.right = right;
    }

    public void print(final int maxDepth) {
        if (maxDepth <= 0) {
            System.out.print("(...)");
        } else {
            System.out.print("(");
            if (left != null) {
                left.print(maxDepth-1);
            }
            System.out.print(data);
            if (right != null) {
                right.print(maxDepth-1);
            }
            System.out.print(")");
        }
    }

}

par exemple.

Node n = new Node(10, new Node(20, new Node(50), new Node(1)), new Node(30, new Node(42), null));
n.print(2);
System.out.println();

Impressions

(((...)20(...))10((...)30))

Cela pourrait être converti en la boucle suivante :

public class Frame {

    public final Node node;

    // 0: before printing anything
    // 1: before printing data
    // 2: before printing ")"
    public int state = 0;
    public final int maxDepth;

    public Frame(Node node, int maxDepth) {
        this.node = node;
        this.maxDepth = maxDepth;
    }

}

List<Frame> stack = new ArrayList<>();
stack.add(new Frame(n, 2)); // first frame = initial call

while (!stack.isEmpty()) {
    // get topmost stack element
    int index = stack.size() - 1;
    Frame frame = stack.get(index); // get topmost frame
    if (frame.maxDepth <= 0) {
        // termial case (too deep)
        System.out.print("(...)");
        stack.remove(index); // drop frame
    } else {
        switch (frame.state) {
            case 0:
                frame.state++;

                // do everything done before the first recursive call
                System.out.print("(");
                if (frame.node.left != null) {
                    // add new frame (recursive call to left and stop)
                    stack.add(new Frame(frame.node.left, frame.maxDepth - 1));
                    break;
                }
            case 1:
                frame.state++;

                // do everything done before the second recursive call
                System.out.print(frame.node.data);
                if (frame.node.right != null) {
                    // add new frame (recursive call to right and stop)
                    stack.add(new Frame(frame.node.right, frame.maxDepth - 1));
                    break;
                }
            case 2:
                // do everything after the second recursive call & drop frame
                System.out.print(")");
                stack.remove(index);
        }
    }
}
System.out.println();

Remarque : Ceci n’est qu’un exemple de l’approche générale. Souvent, vous pouvez trouver une bien meilleure façon de représenter une image et/ou de stocker les données de l’image.

Calcul de la somme des entiers de 1 à N #

La méthode suivante calcule la somme des entiers de 0 à N en utilisant la récursivité.

public int sum(final int n) {
    if (n > 0) {
        return n + sum(n - 1);
    } else {
        return n;
    }
}

Cette méthode est O(N) et peut être réduite à une simple boucle en utilisant l’optimisation des appels de queue. En fait, il existe une expression forme fermée qui calcule la somme dans les opérations O(1).
