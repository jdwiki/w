Un Stream représente une séquence d’éléments et prend en charge différents types d’opérations pour effectuer des calculs sur ces éléments. Avec Java 8, l’interface Collection a deux méthodes pour générer un Stream : stream() et parallelStream(). Les opérations Stream sont soit intermédiaires, soit terminales. Les opérations intermédiaires renvoient un Stream afin que plusieurs opérations intermédiaires puissent être enchaînées avant que le Stream ne soit fermé. Les opérations de terminal sont soit nulles, soit renvoient un résultat non-flux.

Syntaxe #

  • collection.stream()
  • Tableaux.stream(tableau)
  • Stream.iterate(firstValue, currentValue -> nextValue)
  • Stream.generate(() -> valeur)
  • Flux.de(élémentDeT[, élémentDeT, …])
  • Flux.empty()
  • StreamSupport.stream( iterable.spliterator(), false )

Utilisation de flux #

Un [Stream][1] est une séquence d’éléments sur lesquels des opérations d’agrégation séquentielles et parallèles peuvent être effectuées. Tout “flux” donné peut potentiellement avoir une quantité illimitée de données qui le traversent. Par conséquent, les données reçues d’un “flux” sont traitées individuellement au fur et à mesure de leur arrivée, au lieu d’effectuer un traitement par lots sur les données. Lorsqu’ils sont combinés avec des [expressions lambda][2], ils fournissent un moyen concis d’effectuer des opérations sur des séquences de données en utilisant une approche fonctionnelle.

Exemple : ([voir ça marche sur Ideone][3])

Stream<String> fruitStream = Stream.of("apple", "banana", "pear", "kiwi", "orange");

fruitStream.filter(s -> s.contains("a"))
           .map(String::toUpperCase)
           .sorted()
           .forEach(System.out::println);

Production:

POMME BANANE ORANGE POIRE

Les opérations effectuées par le code ci-dessus peuvent être résumées comme suit :

  1. Créez un Stream<String> contenant un [Stream][1] séquencé d’éléments fruit String à l’aide de la méthode de fabrique statique [Stream.of(values)][4].

  2. L’opération [filter()][5] ne retient que les éléments qui correspondent à un prédicat donné (les éléments qui, lorsqu’ils sont testés par le prédicat, renvoient vrai). Dans ce cas, il retient les éléments contenant un "a". Le prédicat est donné sous la forme d’une [expression lambda][2].

  3. L’opération [map()][6] transforme chaque élément à l’aide d’une fonction donnée, appelée mappeur. Dans ce cas, chaque fruit String est mappé sur sa version majuscule String à l’aide de method-reference [ Chaîne :: toUppercase][7].

Notez que l’opération map() renverra un flux avec un type générique différent si la fonction de mappage renvoie un type différent de son paramètre d’entrée. Par exemple sur un Stream<String> appeler .map(String::isEmpty) renvoie un Stream<Boolean>

  1. L’opération [sorted()][8] trie les éléments du Stream selon leur ordre naturel (lexicographiquement, dans le cas de String).

  2. Enfin, l’opération [forEach(action)][9] effectue une action qui agit sur chaque élément du Stream, en le passant à un [Consumer][10]. Dans l’exemple, chaque élément est simplement imprimé sur la console. Cette opération est une opération terminale, il est donc impossible de l’opérer à nouveau.

Notez que les opérations définies sur le Stream sont effectuées à cause de l’opération du terminal. Sans opération de terminal, le flux n’est pas traité. Les flux ne peuvent pas être réutilisés. Une fois qu’une opération de terminal est appelée, l’objet Stream devient inutilisable.

Fermeture des flux #

Notez qu’un Stream n’a généralement pas besoin d’être fermé. Il est uniquement nécessaire de fermer les flux qui fonctionnent sur les canaux IO. La plupart des types Stream ne fonctionnent pas sur les ressources et ne nécessitent donc pas de fermeture.

L’interface Stream étend [AutoCloseable][autocloseable-javadocs]. Les flux peuvent être fermés en appelant la méthode [close][autocloseable-close-javadocs] ou en utilisant des instructions try-with-resource.

Un exemple de cas d’utilisation où un Stream doit être fermé est lorsque vous créez un Stream de lignes à partir d’un fichier :

try (Stream<String> lines = Files.lines(Paths.get("somePath"))) {
    lines.forEach(System.out::println);
}

L’interface Stream déclare également la méthode [Stream.onClose()][15] qui vous permet d’enregistrer les gestionnaires [Runnable][16] qui seront appelés lorsque le flux sera fermé. Un exemple de cas d’utilisation est celui où le code qui produit un flux doit savoir quand il est consommé pour effectuer un nettoyage.

public Stream<String>streamAndDelete(Path path) throws IOException {
    return Files.lines(path).onClose(() -> someClass.deletePath(path));
}

Le gestionnaire d’exécution ne s’exécutera que si la méthode close() est appelée, explicitement ou implicitement par une instruction try-with-resources.

Commande en traitement #

Le traitement d’un objet Stream peut être séquentiel ou [parallèle](https://www.wikiod.com/fr/java/ruisseaux#Flux parallèle).

En mode séquentiel, les éléments sont traités dans l’ordre de la source du Stream. Si le Stream est ordonné (comme une implémentation [SortedMap][12] ou une [List][13]), le traitement est garanti pour correspondre à l’ordre de la source. Dans d’autres cas, cependant, il faut veiller à ne pas dépendre de l’ordre (voir : l’ordre d’itération Java HashMap keySet() est-il cohérent ?).

Exemple:

List<Integer> integerList = Arrays.asList(0, 1, 2, 3, 42); 

// sequential 
long howManyOddNumbers = integerList.stream()
                                    .filter(e -> (e % 2) == 1)
                                    .count(); 

System.out.println(howManyOddNumbers); // Output: 2

[En direct sur Ideone][14]

Le mode parallèle permet l’utilisation de plusieurs threads sur plusieurs cœurs, mais il n’y a aucune garantie de l’ordre dans lequel les éléments sont traités.

Si plusieurs méthodes sont appelées sur un Stream séquentiel, toutes les méthodes ne doivent pas être appelées. Par exemple, si un Stream est filtré et que le nombre d’éléments est réduit à un, un appel ultérieur à une méthode telle que sort ne se produira pas. Cela peut augmenter les performances d’un Stream séquentiel - une optimisation qui n’est pas possible avec un Stream parallèle.

Exemple:

// parallel
long howManyOddNumbersParallel = integerList.parallelStream()
                                            .filter(e -> (e % 2) == 1)
                                            .count();

System.out.println(howManyOddNumbersParallel); // Output: 2

[En direct sur Ideone][14]

Différences par rapport aux conteneurs (ou [Collections][17]) #

Bien que certaines actions puissent être effectuées à la fois sur les conteneurs et les flux, elles servent finalement à des fins différentes et prennent en charge différentes opérations. Les conteneurs sont davantage axés sur la manière dont les éléments sont stockés et sur la manière dont ces éléments peuvent être consultés efficacement. Un Stream, en revanche, ne fournit pas d’accès direct ni de manipulation à ses éléments ; il est plus dédié au groupe d’objets en tant qu’entité collective et effectuant des opérations sur cette entité dans son ensemble. Stream et Collection sont des abstractions de haut niveau distinctes pour ces objectifs différents.

Un [Stream][1] ne sera traversé que lorsqu’il y a une opération terminale, comme [count()][2], [collect()][3] ou [forEach() ][4]. Sinon, aucune opération sur le [Stream][1] ne sera effectuée.

IntStream.range(1, 10).filter(a -> a % 2 == 0).peek(System.out::println);

Création d’une carte de fréquence #

Le collecteur [groupingBy(classifier, down)][1] permet la collecte d’éléments [Stream][2] dans une [Map][3] en classant chaque élément dans un groupe et en effectuant une opération en aval sur les éléments classés dans le même groupe.

Un exemple classique de ce principe consiste à utiliser une [Map][3] pour compter les occurrences d’éléments dans un [Stream][2]. Dans cet exemple, le classificateur est simplement la fonction d’identité, qui renvoie l’élément tel quel. L’opération en aval compte le nombre d’éléments égaux, en utilisant counting().

Stream.of("apple", "orange", "banana", "apple")
      .collect(Collectors.groupingBy(Function.identity(), Collectors.counting()))
      .entrySet()
      .forEach(System.out::println);

L’opération en aval est elle-même un collecteur ([Collectors.counting()][4]) qui opère sur des éléments de type String et produit un résultat de type Long. Le résultat de l’appel de la méthode collect est un Map<String, Long>.

Cela produirait la sortie suivante :

banane=1
orange=1
pomme=2

Flux infinis #

Il est possible de générer un Stream qui ne se termine pas. L’appel d’une méthode de terminal sur un Stream infini fait entrer le Stream dans une boucle infinie. La méthode [limit][1] d’un Stream peut être utilisée pour limiter le nombre de termes du Stream que Java traite.

Cet exemple génère un Stream' de tous les nombres naturels, en commençant par le nombre 1. Chaque terme successif du Streamest supérieur d'une unité au précédent. En appelant la méthode limit de ceStream, seuls les cinq premiers termes du Stream` sont pris en compte et imprimés.

// Generate infinite stream - 1, 2, 3, 4, 5, 6, 7, ...
IntStream naturalNumbers = IntStream.iterate(1, x -> x + 1);

// Print out only the first 5 terms
naturalNumbers.limit(5).forEach(System.out::println);

Collecter les éléments d’un flux dans une collection #

Collecter avec [toList()][1] et [toSet()][2] #

Les éléments d’un [Stream][3] peuvent être facilement collectés dans un conteneur à l’aide de l’opération [Stream.collect][4] :

System.out.println(Arrays
    .asList("apple", "banana", "pear", "kiwi", "orange")
    .stream()
    .filter(s -> s.contains("a"))
    .collect(Collectors.toList())
);
// prints: [apple, banana, pear, orange]

D’autres instances de collection, telles qu’un [Set][5], peuvent être créées à l’aide d’autres méthodes intégrées [Collectors][6]. Par exemple, [Collectors.toSet()][2] rassemble les éléments d’un Stream dans un [Set][5].

Contrôle explicite de l’implémentation de [List][7] ou [Set][5]

Selon la documentation de [Collectors#toList()][1] et [Collectors#toSet()][2], il n’y a aucune garantie sur le type, la mutabilité, la sérialisabilité ou la sécurité des threads du [ List][7] ou [Set][5] renvoyé.

Pour un contrôle explicite sur l’implémentation à renvoyer, [Collectors#toCollection(Supplier)][8] peut être utilisé à la place, où le fournisseur donné renvoie une nouvelle collection vide.

// syntax with method reference
System.out.println(strings
        .stream()
        .filter(s -> s != null && s.length() <= 3)
        .collect(Collectors.toCollection(ArrayList::new))
);

// syntax with lambda
System.out.println(strings
        .stream()
        .filter(s -> s != null && s.length() <= 3)
        .collect(Collectors.toCollection(() -> new LinkedHashSet<>()))
);

Le collecteur accumule des éléments dans une carte, où la clé est l’identifiant de l’étudiant et la valeur est la valeur de l’étudiant.

    students.add(new Student(2,"test2"));
    students.add(new Student(3,"test3"));

        .collect(Collectors.toMap(Student::getId, Student::getName));
    System.out.println(IdToName);

Le Collectors.toMap a une autre implémentation Collector<T, ?, Map<K,U>> toMap(Function<? super T, ? étend K> keyMapper, Function<? super T, ? extends U> valueMapper, BinaryOperator<U> mergeFunction).The mergeFunction is mostly used to select either new value or retain old value if the key is repeated when adding a new member in the Map from a list.

La mergeFunction ressemble souvent à : (s1, s2) -> s1 pour conserver la valeur correspondant à la clé répétée, ou (s1, s2) -> s2 pour mettre une nouvelle valeur pour la clé répétée.

Utilisation de flux pour implémenter des fonctions mathématiques #

Streams, et en particulier IntStreams, sont une manière élégante d’implémenter des termes de sommation (∑). Les plages du Stream peuvent être utilisées comme limites de la sommation.

Par exemple, l’approximation de Pi par Madhava est donnée par la formule (Source : Wikipedia) : [![Approximation de Madhava][1]][1]

Ceci peut être calculé avec une précision arbitraire. Par exemple, pour 101 termes :

double pi = Math.sqrt(12) * 
            IntStream.rangeClosed(0, 100)
                     .mapToDouble(k -> Math.pow(-3, -1 * k) / (2 * k + 1))
                     .sum();

Remarque : Avec la précision de double, la sélection d’une limite supérieure de 29 est suffisante pour obtenir un résultat qui ne se distingue pas de Math.Pi.

Flux parallèle #

Remarque : Avant de décider quel “Stream” utiliser, veuillez consulter [Comportement de ParallelStream vs Sequential Stream][2].

Lorsque vous souhaitez effectuer des opérations Stream simultanément, vous pouvez utiliser l’une ou l’autre de ces méthodes.

List<String> data = Arrays.asList("One", "Two", "Three", "Four", "Five");
Stream<String> aParallelStream = data.stream().parallel();

Ou:

Stream<String> aParallelStream = data.parallelStream();

Pour exécuter les opérations définies pour le flux parallèle, appelez un opérateur de terminal :

aParallelStream.forEach(System.out::println);

Trois
Quatre
Un
Deux
Cinq

L’ordre peut changer car tous les éléments sont traités en parallèle (ce qui * peut * le rendre plus rapide). Utilisez [parallelStream][1] lorsque l’ordre n’a pas d’importance.

Impact sur les performances #

Dans le cas où la mise en réseau est impliquée, les Stream parallèles peuvent dégrader les performances globales d’une application car tous les Stream parallèles utilisent un pool de threads fork-join commun pour le réseau.

D’un autre côté, les Stream parallèles peuvent améliorer considérablement les performances dans de nombreux autres cas, en fonction du nombre de cœurs disponibles dans le processeur en cours d’exécution.

Aplatir les flux avec flatMap() #

Un “flux” d’éléments pouvant être diffusés à leur tour peut être aplati en un seul “flux” continu :

Un tableau de liste d’éléments peut être converti en une seule liste.

List<String> list1 =  Arrays.asList("one", "two");
      List<String> list2 =  Arrays.asList("three","four","five");
      List<String> list3 =  Arrays.asList("six");
          List<String> finalList = Stream.of(list1, list2, list3).flatMap(Collection::stream).collect(Collectors.toList());
System.out.println(finalList);

// [one, two, three, four, five, six]

La carte contenant la liste des éléments en tant que valeurs peut être aplatie en une liste combinée

Map<String, List<Integer>> map = new LinkedHashMap<>();
map.put("a", Arrays.asList(1, 2, 3));
map.put("b", Arrays.asList(4, 5, 6));

List<Integer> allValues = map.values() // Collection<List<Integer>>
        .stream()                      // Stream<List<Integer>>
        .flatMap(List::stream)         // Stream<Integer>
        .collect(Collectors.toList());

Création d’un flux #

Toutes les Collection<E> java ont des méthodes [stream()][1] et [parallelStream()][2] à partir desquelles un Stream<E> peut être construit :

Collection<String> stringList = new ArrayList<>();
Stream<String> stringStream = stringList.parallelStream();

Un Stream<E> peut être créé à partir d’un tableau en utilisant l’une des deux méthodes suivantes :

String[] values = { "aaa", "bbbb", "ddd", "cccc" };
Stream<String> stringStream = Arrays.stream(values);
Stream<String> stringStreamAlternative = Stream.of(values);

La différence entre [Arrays.stream()][3] et [Stream.of()][4] est que Stream.of() a un paramètre varargs, il peut donc être utilisé comme :

Stream<Integer> integerStream = Stream.of(1, 2, 3);

Il existe également des “Stream” primitifs que vous pouvez utiliser. Par exemple:

IntStream intStream = IntStream.of(1, 2, 3);
DoubleStream doubleStream = DoubleStream.of(1.0, 2.0, 3.0);

Ces flux primitifs peuvent également être construits à l’aide de la méthode Arrays.stream() :

IntStream intStream = Arrays.stream(new int[]{ 1, 2, 3 });

Il est possible de créer un Stream à partir d’un tableau avec une plage spécifiée.

int[] values= new int[]{1, 2, 3, 4, 5};
IntStream intStram = Arrays.stream(values, 1, 3);

Notez que tout flux primitif peut être converti en flux de type boxed en utilisant la méthode boxed :

Stream<Integer> integerStream = intStream.boxed();

Cela peut être utile dans certains cas si vous souhaitez collecter les données car le flux primitif n’a pas de méthode collect qui prend un Collector comme argument.

Concaténer les flux #

Déclaration de variable pour exemples :

Collection<String> abc = Arrays.asList("a", "b", "c");
Collection<String> digits = Arrays.asList("1", "2", "3");
Collection<String> greekAbc = Arrays.asList("alpha", "beta", "gamma");

Exemple 1 - Concaténer deux Streams

// prints: abc123

Exemple 2 - Concaténer plus de deux Streams

    Stream.concat(abc.stream(), digits.stream()),
    greekAbc.stream());

// prints: a, b, c, 1, 2, 3, alpha, beta, gamma

Alternativement, pour simplifier la syntaxe concat() imbriquée, les Streams peuvent également être concaténés avec flatMap() :

final Stream<String> concat3 = Stream.of(
    abc.stream(), digits.stream(), greekAbc.stream())
    .flatMap(s -> s);

Trouver le premier élément qui correspond à un prédicat #

Il est possible de trouver le premier élément d’un Stream qui correspond à une condition.

Pour cet exemple, nous trouverons le premier Integer dont le carré est supérieur à 50000.

IntStream.iterate(1, i -> i + 1) // Generate an infinite stream 1,2,3,4...
    .filter(i -> (i*i) > 50000) // Filter to find elements where the square is >50000
    .findFirst(); // Find the first filtered element

Cette expression renverra un OptionalInt avec le résultat.

Notez qu’avec un Stream infini, Java continuera à vérifier chaque élément jusqu’à ce qu’il trouve un résultat. Avec un Stream fini, si Java manque d’éléments mais ne trouve toujours pas de résultat, il renvoie un OptionalInt vide.

Utiliser des flux et des références de méthode pour écrire des processus d’auto-documentation #

Les références de méthode constituent un excellent code auto-documenté, et l’utilisation de références de méthode avec Streams rend les processus compliqués simples à lire et à comprendre. Considérez le code suivant :

public interface Ordered {
    default int getOrder(){
        return 0;
    }
}

public interface Valued<V extends Ordered> {
    boolean hasPropertyTwo();
    V getValue();
}

public interface Thing<V extends Ordered> {
    boolean hasPropertyOne();
    Valued<V> getValuedProperty();
}

public <V extends Ordered> List<V> myMethod(List<Thing<V>> things) {
    List<V> results = new ArrayList<V>();
    for (Thing<V> thing : things) {
        if (thing.hasPropertyOne()) {
            Valued<V> valued = thing.getValuedProperty();
            if (valued != null && valued.hasPropertyTwo()){
                V value = valued.getValue();
                if (value != null){
                    results.add(value);
                }
            }
        }
    }
    results.sort((a, b)->{
        return Integer.compare(a.getOrder(), b.getOrder());
    });
    return results;
}

Cette dernière méthode réécrite à l’aide de Streams et de références de méthode est beaucoup plus lisible et chaque étape du processus est rapidement et facilement comprise - elle n’est pas seulement plus courte, elle montre également en un coup d’œil quelles interfaces et classes sont responsables du code dans chaque marcher:

public <V extends Ordered> List<V> myMethod(List<Thing<V>> things) {
    return things.stream()
        .filter(Thing::hasPropertyOne)
        .map(Thing::getValuedProperty)
        .filter(Objects::nonNull)
        .filter(Valued::hasPropertyTwo)
        .map(Valued::getValue)
        .filter(Objects::nonNull)
        .sorted(Comparator.comparing(Ordered::getOrder))
        .collect(Collectors.toList());
}

Utilisation de flux de Map.Entry pour conserver les valeurs initiales après le mappage #

Lorsque vous avez un Stream que vous devez mapper mais que vous souhaitez également conserver les valeurs initiales, vous pouvez mapper le Stream à un Map.Entry<K,V> à l’aide d’une méthode utilitaire comme celle-ci :

public static <K, V> Function<K, Map.Entry<K, V>> entryMapper(Function<K, V> mapper){
    return (k)->new AbstractMap.SimpleEntry<>(k, mapper.apply(k));
}

Ensuite, vous pouvez utiliser votre convertisseur pour traiter les “Stream” ayant accès à la fois aux valeurs d’origine et mappées :

Set<K> mySet;
Function<K, V> transformer = SomeClass::transformerMethod;
Stream<Map.Entry<K, V>> entryStream = mySet.stream()
    .map(entryMapper(transformer));

Vous pouvez ensuite continuer à traiter ce “flux” comme d’habitude. Cela évite la surcharge liée à la création d’une collection intermédiaire.

Réduction avec les flux #

La réduction est le processus d’application d’un opérateur binaire à chaque élément d’un flux pour aboutir à une valeur.

La méthode sum() d’un IntStream est un exemple de réduction ; il applique une addition à chaque terme du Stream, résultant en une valeur finale : [![Réduction somme][2]][2]

Ceci est équivalent à (((1+2)+3)+4)

La méthode reduce d’un Stream permet de créer une réduction personnalisée. Il est possible d’utiliser la méthode reduce pour implémenter la méthode sum() :

IntStream istr;
    
//Initialize istr
    
OptionalInt istr.reduce((a,b)->a+b);

La version « facultative » est renvoyée afin que les flux vides puissent être gérés de manière appropriée.

Un autre exemple de réduction consiste à combiner un Stream<LinkedList<T>> en un seul LinkedList<T> :

Stream<LinkedList<T>> listStream;
    
//Create a Stream<LinkedList<T>>
    
Optional<LinkedList<T>> bigList = listStream.reduce((LinkedList<T> list1, LinkedList<T> list2)->{
    LinkedList<T> retList = new LinkedList<T>();
    retList.addAll(list1);
    retList.addAll(list2);
    return retList;
});

Vous pouvez également fournir un élément d’identité. Par exemple, l’élément d’identité pour l’addition est 0, comme x+0==x. Pour la multiplication, l’élément d’identité est 1, comme x*1==x. Dans le cas ci-dessus, l’élément d’identité est une LinkedList<T> vide, car si vous ajoutez une liste vide à une autre liste, la liste à laquelle vous “ajoutez” ne change pas :

Stream<LinkedList<T>> listStream;

//Create a Stream<LinkedList<T>>

LinkedList<T> bigList = listStream.reduce(new LinkedList<T>(), (LinkedList<T> list1, LinkedList<T> list2)->{
    LinkedList<T> retList = new LinkedList<T>();
    retList.addAll(list1);
    retList.addAll(list2);
    return retList;
});

Notez que lorsqu’un élément d’identité est fourni, la valeur de retour n’est pas enveloppée dans un Facultatif - si elle est appelée sur un flux vide, reduce() renverra l’élément d’identité.

L’opérateur binaire doit également être associatif, ce qui signifie que (a+b)+c==a+(b+c). En effet, les éléments peuvent être réduits dans n’importe quel ordre. Par exemple, la réduction d’addition ci-dessus pourrait être effectuée comme ceci :

Cette réduction équivaut à écrire ((1+2)+(3+4)). La propriété d’associativité permet également à Java de réduire le Stream en parallèle - une partie du Stream peut être réduite par chaque processeur, avec une réduction combinant le résultat de chaque processeur à la fin.

Obtenir une tranche de flux #

Exemple : Obtenez un “flux” de 30 éléments, contenant du 21e au 50e élément (inclus) d’une collection.

final long n = 20L; // the number of elements to skip

Remarques:

  • IllegalArgumentException est lancé si n est négatif ou maxSize est négatif
  • skip(long) et limit(long) sont des opérations intermédiaires
  • si un flux contient moins de n éléments alors skip(n) renvoie un flux vide
  • skip(long) et limit(long) sont des opérations bon marché sur des pipelines de flux séquentiels, mais peuvent être assez coûteuses sur des pipelines parallèles ordonnés

Créer une carte basée sur un flux #

Cas simple sans clés en double

Stream<String> characters = Stream.of("A", "B", "C");

Map<Integer, String> map = characters
            .collect(Collectors.toMap(element -> element.hashCode(), element -> element));
// map = {65=A, 66=B, 67=C}

Pour rendre les choses plus déclaratives, nous pouvons utiliser la méthode statique dans l’interface Function - [Function.identity()][1]. Nous pouvons remplacer ce lambda element -> element par Function.identity().

Cas où il pourrait y avoir des clés en double

Le [javadoc][2] pour Collectors.toMap indique :

Si les clés mappées contiennent des doublons (selon Object.equals(Object)), une IllegalStateException est levée lorsque l’opération de collecte est effectuée. Si les clés mappées peuvent avoir des doublons, utilisez plutôt toMap(Function, Function, BinaryOperator).

Stream<String> characters = Stream.of("A", "B", "B", "C");

Map<Integer, String> map = characters
            .collect(Collectors.toMap(
                element -> element.hashCode(),
                element -> element,
                (existingVal, newVal) -> (existingVal + newVal)));

// map = {65=A, 66=BB, 67=C}

Le BinaryOperator passé à Collectors.toMap(...) génère la valeur à stocker en cas de collision. Ça peut:

  • retourner l’ancienne valeur, de sorte que la première valeur du flux prévaut,
  • retourner la nouvelle valeur, de sorte que la dernière valeur du flux prévaut, ou
  • combiner les anciennes et les nouvelles valeurs

Flux de Primitives #

Java fournit des Stream spécialisés pour trois types de primitives IntStream (pour ints), LongStream (pour longs) et DoubleStream (pour doubles). En plus d’être des implémentations optimisées pour leurs primitives respectives, ils fournissent également plusieurs méthodes de terminal spécifiques, généralement pour les opérations mathématiques. Par exemple.:

IntStream is = IntStream.of(10, 20, 30);
double average = is.average().getAsDouble(); // average is 20.0

Catégories d’opérations de flux #

Les opérations de flux se répartissent en deux catégories principales, les opérations intermédiaires et terminales, et en deux sous-catégories, sans état et avec état.

Opérations Intermédiaires : #

Une opération intermédiaire est toujours paresseuse, comme un simple Stream.map. Il n’est pas appelé tant que le flux n’est pas réellement consommé. Cela peut être vérifié facilement :

Arrays.asList(1, 2 ,3).stream().map(i -> {
    throw new RuntimeException("not gonna happen");
    return i;
});

Les opérations intermédiaires sont les blocs de construction communs d’un flux, enchaînés après la source et sont généralement suivis d’une opération terminale déclenchant la chaîne de flux.

Opérations terminales #

Les opérations du terminal sont ce qui déclenche la consommation d’un flux. Certains des plus courants sont Stream.forEach ou Stream.collect. Ils sont généralement placés après une chaîne d’opérations intermédiaires et sont presque toujours empressés.

Opérations sans état #

L’apatridie signifie que chaque élément est traité sans le contexte des autres éléments. Les opérations sans état permettent un traitement efficace de la mémoire des flux. Les opérations telles que Stream.map et Stream.filter qui ne nécessitent pas d’informations sur d’autres éléments du flux sont considérées comme sans état.

Opérations avec état #

L’état signifie que l’opération sur chaque élément dépend de (certains) autres éléments du flux. Cela nécessite un état à préserver. Les opérations d’état peuvent rompre avec des flux longs ou infinis. Des opérations telles que Stream.sorted nécessitent que l’intégralité du flux soit traitée avant qu’un élément ne soit émis, ce qui se brisera en un flux d’éléments suffisamment long. Cela peut être démontré par un long flux (exécuté à vos risques et périls) :

// works - stateless stream
long BIG_ENOUGH_NUMBER = 999999999;
IntStream.iterate(0, i -> i + 1).limit(BIG_ENOUGH_NUMBER).forEach(System.out::println);

Cela entraînera une mémoire insuffisante en raison de l’état de Stream.sorted :

// Out of memory - stateful stream
IntStream.iterate(0, i -> i + 1).limit(BIG_ENOUGH_NUMBER).sorted().forEach(System.out::println);

Stream<String> fruitStream = Stream.of("apple", "banana", "pear", "kiwi", "orange");

Génération de chaînes aléatoires à l’aide de flux #

Il est parfois utile de créer des Strings aléatoires, peut-être comme ID de session pour un service Web ou un mot de passe initial après l’enregistrement pour une application. Ceci peut être facilement réalisé en utilisant Streams.

Nous devons d’abord initialiser un générateur de nombres aléatoires. Pour améliorer la sécurité des Strings générés, c’est une bonne idée d’utiliser SecureRandom.

Remarque : La création d’un SecureRandom est assez coûteuse, il est donc préférable de ne le faire qu’une seule fois et d’appeler l’une de ses méthodes setSeed() de temps en temps pour le réamorcer.

private static final SecureRandom rng = new SecureRandom(SecureRandom.generateSeed(20)); 
//20 Bytes as a seed is rather arbitrary, it is the number used in the JavaDoc example

Lors de la création de Strings aléatoires, nous souhaitons généralement qu’ils n’utilisent que certains caractères (par exemple, uniquement des lettres et des chiffres). Par conséquent, nous pouvons créer une méthode renvoyant un booléen qui peut ensuite être utilisé pour filtrer le Stream.

//returns true for all chars in 0-9, a-z and A-Z
boolean useThisCharacter(char c){
    //check for range to avoid using all unicode Letter (e.g. some chinese symbols)
    return c >= '0' && c <= 'z' && Character.isLetterOrDigit(c);
}

Ensuite, nous pouvons utiliser le RNG pour générer une chaîne aléatoire de longueur spécifique contenant le jeu de caractères qui passe notre vérification “useThisCharacter”.

public String generateRandomString(long length){
    //Since there is no native CharStream, we use an IntStream instead
    //and convert it to a Stream<Character> using mapToObj.
    //We need to specify the boundaries for the int values to ensure they can safely be cast to char
    Stream<Character> randomCharStream = rng.ints(Character.MIN_CODE_POINT, Character.MAX_CODE_POINT).mapToObj(i -> (char)i).filter(c -> this::useThisCharacter).limit(length);

    //now we can use this Stream to build a String utilizing the collect method.
    String randomString = randomCharStream.collect(StringBuilder::new, StringBuilder::append, StringBuilder::append).toString();
    return randomString;
}
