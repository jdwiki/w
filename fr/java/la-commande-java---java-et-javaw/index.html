Syntaxe #

  • java [ <opt> ... ] <nom-classe> [ <argument> ... ]

  • java [ <opt> ... ] -jar <nom-chemin-fichier-jar> [ <argument> ... ]

La commande java est utilisée pour exécuter une application Java à partir de la ligne de commande. Il est disponible dans le cadre de n’importe quel Java SE JRE ou JDK.

Sur les systèmes Windows, il existe deux variantes de la commande java :

  • La variante java lance l’application dans une nouvelle fenêtre de console.
  • La variante javaw lance l’application sans créer de nouvelle fenêtre console.

Sur d’autres systèmes (par exemple, Linux, Mac OSX, UNIX), seule la commande “java” est fournie et elle ne lance pas une nouvelle fenêtre de console.

Le symbole <opt> dans la syntaxe indique une option sur la ligne de commande java. Les rubriques “Options Java” et “Options de dimensionnement du tas et de la pile” couvrent les options les plus couramment utilisées. D’autres sont couverts dans le sujet https://www.wikiod.com/fr/java/drapeaux-jvm

Classes de point d’entrée #

Une classe de point d’entrée Java a une méthode main avec la signature et les modificateurs suivants :

public static void main(String[] args)

Sidenote : en raison du fonctionnement des tableaux, il peut également s’agir de (String args [])

Lorsque la commande java démarre la machine virtuelle, elle charge les classes de point d’entrée spécifiées et essaie de trouver main. En cas de succès, les arguments de la ligne de commande sont convertis en objets Java “String” et assemblés dans un tableau. Si main est appelé comme ceci, le tableau ne sera pas null et ne contiendra aucune entrée null.

Une méthode de classe de point d’entrée valide doit effectuer les opérations suivantes :

  • Être nommé main (sensible à la casse) - Être “public” et “statique”
  • Avoir un type de retour “void”
  • Avoir un seul argument avec un tableau String[]. L’argument doit être présent et pas plus d’un argument n’est autorisé.
  • Soyez générique : les paramètres de type ne sont pas autorisés.
  • Avoir une classe englobante non générique de niveau supérieur (non imbriquée ou interne)

Il est conventionnel de déclarer la classe comme “publique” mais ce n’est pas strictement nécessaire. À partir de Java 5, le type d’argument de la méthode main peut être un varargs String au lieu d’un tableau de chaînes. main peut éventuellement lancer des exceptions, et son paramètre peut être nommé n’importe quoi, mais par convention c’est args.

Points d’entrée JavaFX #

À partir de Java 8, la commande java peut également lancer directement une application JavaFX. JavaFX est documenté dans la balise [JavaFX][1], mais un point d’entrée JavaFX doit effectuer les opérations suivantes :

  • Étendre javafx.application.Application
  • Soyez “public” et non “abstrait”
  • Ne pas être générique ou imbriqué
  • Avoir un constructeur explicite ou implicite “public” sans arguments

[1] : https://www.wikiod.com/fr/javafx

Exécution d’un fichier JAR exécutable #

Les fichiers JAR exécutables sont le moyen le plus simple d’assembler du code Java en un seul fichier exécutable. *(Note éditoriale : la création de fichiers JAR doit être couverte par une rubrique distincte.) *

En supposant que vous ayez un fichier JAR exécutable avec le nom de chemin <jar-path>, vous devriez pouvoir l’exécuter comme suit :

java -jar <jar-path>

Si la commande nécessite des arguments de ligne de commande, ajoutez-les après le <jar-path>. Par exemple:

java -jar <jar-path> arg1 arg2 arg3

Si vous avez besoin de fournir des options JVM supplémentaires sur la ligne de commande java, elles doivent être placées avant l’option -jar. Notez qu’une option -cp / -classpath sera ignorée si vous utilisez -jar. Le chemin de classe de l’application est déterminé par le manifeste du fichier JAR.

Exécuter une application Java via une classe “main” #

Lorsqu’une application n’a pas été empaquetée en tant que JAR exécutable, vous devez fournir le nom d’une [classe de point d’entrée][1] sur la ligne de commande ‘java’.

Exécution de la classe HelloWorld #

L’exemple “HelloWorld” est décrit dans https://www.wikiod.com/fr/java/debuter-avec-le-langage-java#Création de votre premier programme Java . Il se compose d’une seule classe appelée “HelloWorld” qui satisfait aux exigences d’un point d’entrée.

En supposant que le fichier “HelloWorld.class” (compilé) se trouve dans le répertoire courant, il peut être lancé comme suit :

java HelloWorld

Certaines choses importantes à noter sont :

  • Il faut fournir le nom de la classe : pas le chemin du fichier “.class” ou du fichier “.java”.
  • Si la classe est déclarée dans un package (comme le sont la plupart des classes Java), alors le nom de classe que nous fournissons à la commande java doit être le nom de classe complet. Par exemple, si SomeClass est déclaré dans le package com.example, alors le nom de classe complet sera com.example.SomeClass.

Spécification d’un chemin de classe #

À moins que nous n’utilisions la syntaxe de commande java -jar, la commande java recherche la classe à charger en recherchant le chemin de classe ; voir https://www.wikiod.com/fr/java/le-chemin-de-classe La commande ci-dessus s’appuie sur le chemin de classe par défaut étant (ou incluant) le répertoire actuel. Nous pouvons être plus explicites à ce sujet en spécifiant le chemin de classe à utiliser à l’aide de l’option -cp.

java -cp . HelloWorld

Cela dit de faire du répertoire courant (qui est ce à quoi “.” fait référence) la seule entrée sur le chemin de classe.

Le -cp est une option qui est traitée par la commande java. Toutes les options destinées à la commande java doivent être placées avant le nom de la classe. Tout ce qui suit la classe sera traité comme un argument de ligne de commande pour l’application Java et sera transmis à l’application dans la String[] qui est transmise à la méthode main.

(Si aucune option -cp n’est fournie, java utilisera le chemin de classe donné par la variable d’environnement CLASSPATH. Si cette variable n’est pas définie ou vide, java utilise “.” comme chemin de classe par défaut. )

[1] : https://www.wikiod.com/fr/java/la-commande-java—java-et-javaw#Classes de point d’entrée

Dépannage de la commande ‘java’ #

Cet exemple couvre les erreurs courantes liées à l’utilisation de la commande ‘java’.

“Commande non trouvée” #

Si vous obtenez un message d’erreur comme :

java: command not found

lorsque vous essayez d’exécuter la commande java, cela signifie qu’il n’y a pas de commande java sur le chemin de recherche de commandes de votre shell. La cause pourrait être :

  • vous n’avez pas du tout installé Java JRE ou JDK,
  • vous n’avez pas mis à jour la variable d’environnement PATH (correctement) dans votre fichier d’initialisation du shell, ou
  • vous n’avez pas “source” le fichier d’initialisation pertinent dans le shell courant.

Reportez-vous à [« Installation de Java »][1] pour connaître les étapes à suivre.

“Impossible de trouver ou de charger la classe principale” #

Ce message d’erreur est généré par la commande java si elle n’a pas pu trouver/charger la classe de point d’entrée que vous avez spécifiée. En termes généraux, il y a trois grandes raisons pour lesquelles cela peut se produire :

  • Vous avez spécifié une classe de point d’entrée qui n’existe pas.
  • La classe existe, mais vous l’avez mal spécifiée.
  • La classe existe et vous l’avez correctement spécifiée, mais Java ne la trouve pas car le chemin de classe est incorrect.

Voici une procédure pour diagnostiquer et résoudre le problème :

  1. Découvrez le nom complet de la classe de point d’entrée.

    • If you have source code for a class, then the full name consists of the package name and the simple class name. The instance the “Main” class is declared in the package “com.example.myapp” then its full name is “com.example.myapp.Main”.
    • If you have a compiled class file, you can find the class name by running javap on it.
    • If the class file is in a directory, you can infer the full class name from the directory names.
    • If the class file is in a JAR or ZIP file, you can infer the full class name from the file path in the JAR or ZIP file.

  2. Regardez le message d’erreur de la commande java. Le message doit se terminer par le nom complet de la classe que java essaie d’utiliser.

    • Check that it exactly matches the full classname for the entry-point class.
    • It should not end with “.java” or “.class”.
    • It should not contain slashes or any other character that is not legal in a Java identifier1.
    • The casing of the name should exactly match the full class name.

  3. Si vous utilisez le nom de classe correct, assurez-vous que la classe se trouve bien sur le chemin de classe :

    • Work out the pathname that the classname maps to; see https://www.wikiod.com/fr/java/le-chemin-de-classe#Mappage des noms de classe aux noms de chemin
    • Work out what the classpath is; see this example: https://www.wikiod.com/fr/java/le-chemin-de-classe#Différentes façons de spécifier le classpath
    • Look at each of the JAR and ZIP files on the classpath to see if they contain a class with the required pathname.
    • Look at each directory to see if the pathname resolves to a file within the directory.

Si la vérification manuelle du chemin de classe n’a pas trouvé le problème, vous pouvez ajouter les options -Xdiag et -XshowSettings. Le premier répertorie toutes les classes chargées et le second imprime les paramètres qui incluent le chemin de classe effectif pour la JVM.

Enfin, il existe des causes obscures à ce problème :

  • Un fichier JAR exécutable avec un attribut Main-Class qui spécifie une classe qui n’existe pas.
  • Un fichier JAR exécutable avec un attribut Class-Path incorrect.
  • Si vous confondez2 les options avant le nom de la classe, la commande java peut tenter d’interpréter l’une d’entre elles comme le nom de la classe.
  • Si quelqu’un a ignoré les règles de style Java et utilisé des identifiants de package ou de classe qui ne diffèrent que par la casse des lettres, et que vous utilisez une plate-forme qui traite la casse des lettres dans les noms de fichiers comme non significative.
  • Problèmes d’homoglyphes dans les noms de classes dans le code ou sur la ligne de commande.

“Méthode principale introuvable dans la classe <nom>” #

Ce problème se produit lorsque la commande java est capable de trouver et de charger la classe que vous avez désignée, mais est ensuite incapable de trouver une méthode de point d’entrée.

Il y a trois explications possibles :

  • Si vous essayez d’exécuter un fichier JAR exécutable, le manifeste du JAR a un attribut “Main-Class” incorrect qui spécifie une classe qui n’est pas une classe de point d’entrée valide.
  • Vous avez indiqué à la commande java une classe qui n’est pas une classe de point d’entrée.
  • La classe du point d’entrée est incorrecte ; voir https://www.wikiod.com/fr/java/la-commande-java—java-et-javaw#Classes de point d’entrée pour plus d’informations.

Exécution d’une application Java avec des dépendances de bibliothèque #

Ceci est une continuation des exemples [“main class”][1] et [“executable JAR”][2].

Les applications Java typiques se composent d’un code spécifique à l’application et de divers codes de bibliothèque réutilisables que vous avez implémentés ou qui ont été implémentés par des tiers. Ces dernières sont communément appelées dépendances de bibliothèque et sont généralement conditionnées sous forme de fichiers JAR.

Java est un langage dynamiquement lié. Lorsque vous exécutez une application Java avec des dépendances de bibliothèque, la JVM doit savoir où se trouvent les dépendances afin de pouvoir charger les classes selon les besoins. En gros, il y a deux façons de gérer cela :

  • L’application et ses dépendances peuvent être reconditionnées dans un seul fichier JAR contenant toutes les classes et ressources requises.

  • La JVM peut savoir où trouver les fichiers JAR dépendants via le chemin de classe d’exécution.

Pour un fichier JAR exécutable, le chemin d’accès aux classes d’exécution est spécifié par l’attribut manifeste “Class-Path”. (Note éditoriale : cela devrait être décrit dans une rubrique distincte sur la commande jar.) Sinon, le chemin d’accès aux classes d’exécution doit être fourni à l’aide de l’option -cp ou de la variable d’environnement CLASSPATH.

Par exemple, supposons que nous ayons une application Java dans le fichier “myApp.jar” dont la classe de point d’entrée est com.example.MyApp. Supposons également que l’application dépende des fichiers JAR de la bibliothèque “lib/library1.jar” et “lib/library2.jar”. Nous pourrions lancer l’application en utilisant la commande java comme suit dans une ligne de commande :

$ # Alternative 1 (preferred)
$ java -cp myApp.jar:lib/library1.jar:lib/library2.jar com.example.MyApp

$ # Alternative 2
$ export CLASSPATH=myApp.jar:lib/library1.jar:lib/library2.jar
$ java com.example.MyApp

(Sous Windows, vous utiliseriez ; au lieu de : comme séparateur de chemin de classe, et vous définiriez la variable (locale) CLASSPATH en utilisant set plutôt que export.)

Bien qu’un développeur Java soit à l’aise avec cela, ce n’est pas “convivial”. Il est donc courant d’écrire un simple script shell (ou un fichier de commandes Windows) pour masquer les détails que l’utilisateur n’a pas besoin de connaître. Par exemple, si vous placez le script shell suivant dans un fichier appelé “myApp”, le rendez exécutable et le placez dans un répertoire sur le chemin de recherche de la commande :

#!/bin/bash
# The 'myApp' wrapper script

export DIR=/usr/libexec/myApp
export CLASSPATH=$DIR/myApp.jar:$DIR/lib/library1.jar:$DIR/lib/library2.jar
java com.example.MyApp

alors vous pouvez l’exécuter comme suit:

$ myApp arg1 arg2 ...

Tous les arguments de la ligne de commande seront transmis à l’application Java via l’extension "$@". (Vous pouvez faire quelque chose de similaire avec un fichier de commandes Windows, bien que la syntaxe soit différente.)

Espaces et autres caractères spéciaux dans les arguments #

Tout d’abord, le problème de la gestion des espaces dans les arguments n’est PAS un problème Java. Il s’agit plutôt d’un problème qui doit être géré par le shell de commande que vous utilisez lorsque vous exécutez un programme Java.

A titre d’exemple, supposons que nous ayons le programme simple suivant qui imprime la taille d’un fichier :

import java.io.File;

public class PrintFileSizes {
    
    public static void main(String[] args) {
        for (String name: args) {
            File file = new File(name);
            System.out.println("Size of '" + file + "' is " + file.size());
        }
    }
}

Supposons maintenant que nous voulions imprimer la taille d’un fichier dont le nom de chemin contient des espaces ; par exemple. /home/steve/Fichier de test.txt. Si nous exécutons la commande comme ceci :

$ java PrintFileSizes /home/steve/Test File.txt

le shell ne saura pas que /home/steve/Test File.txt est en fait un chemin d’accès. Au lieu de cela, il transmettra 2 arguments distincts à l’application Java, qui tentera de trouver leurs tailles de fichier respectives, et échouera car les fichiers avec ces chemins n’existent (probablement) pas.

Solutions utilisant un shell POSIX #

Les shells POSIX incluent sh ainsi que des dérivés tels que bash et ksh. Si vous utilisez l’un de ces shells, vous pouvez résoudre le problème en citant l’argument.

$ java PrintFileSizes "/home/steve/Test File.txt"

Les guillemets doubles autour du nom de chemin indiquent au shell qu’il doit être passé en tant qu’argument unique. Les guillemets seront supprimés lorsque cela se produira. Il existe plusieurs autres façons de procéder :

$ java PrintFileSizes '/home/steve/Test File.txt'

Les guillemets simples (droits) sont traités comme des guillemets doubles, sauf qu’ils suppriment également diverses expansions dans l’argument.

$ java PrintFileSizes /home/steve/Test\ File.txt

Une barre oblique inverse échappe à l’espace suivant et l’empêche d’être interprété comme un séparateur d’arguments.

Pour une documentation plus complète, y compris des descriptions sur la façon de traiter d’autres caractères spéciaux dans les arguments, veuillez vous référer au [sujet citant][1] dans la documentation Bash.

Solution pour Windows #

Le problème fondamental pour Windows est qu’au niveau du système d’exploitation, les arguments sont transmis à un processus enfant sous la forme d’une chaîne unique ([source][2]). Cela signifie que la responsabilité ultime de l’analyse (ou de la réanalyse) de la ligne de commande incombe au programme ou à ses bibliothèques d’exécution. Il y a beaucoup d’incohérence.

Dans le cas de Java, pour faire court :

  • Vous pouvez mettre des guillemets autour d’un argument dans une commande java, et cela vous permettra de passer des arguments contenant des espaces.

  • Apparemment, la commande java elle-même analyse la chaîne de commande, et elle l’obtient plus ou moins correctement

  • Cependant, lorsque vous essayez de combiner cela avec l’utilisation de SET et la substitution de variables dans un fichier batch, il devient vraiment compliqué de savoir si les guillemets doubles sont supprimés.

  • Le shell cmd.exe a apparemment d’autres mécanismes d’échappement ; par exemple. en doublant les guillemets doubles et en utilisant les échappements ^.

Pour plus de détails, veuillez vous référer à la documentation [Batch-File][3].

##Options Java La commande java prend en charge un large éventail d’options :

  • Toutes les options commencent par un simple trait d’union ou un signe moins (-) : la convention GNU/Linux d’utilisation de -- pour les options “longues” n’est pas prise en charge.

  • Les options doivent apparaître avant l’argument <classname> ou -jar <jarfile> pour être reconnues. Tous les arguments après eux seront traités comme des arguments à transmettre à l’application Java en cours d’exécution.

  • Les options qui ne commencent pas par -X ou -XX sont des options standard. Vous pouvez compter sur toutes les implémentations Java1 pour prendre en charge n’importe quelle option standard.

  • Les options commençant par -X sont des options non standard et peuvent être retirées d’une version de Java à l’autre.

  • Les options qui commencent par -XX sont des options avancées et peuvent également être retirées.

Définir les propriétés du système avec -D #

L’option -D<propriété>=<valeur> est utilisée pour définir une propriété dans l’objet système Propriétés. Ce paramètre peut être répété pour définir différentes propriétés.

Options de mémoire, de pile et de récupérateur de place #

Les principales options de contrôle des tailles de tas et de pile sont documentées dans https://www.wikiod.com/fr/java/gestion-de-la-memoire-java#Définition des tailles Heap, PermGen et Stack (Note éditoriale : les options du Garbage Collector doivent être décrites dans la même rubrique.)

Activation et désactivation des assertions #

Les options -ea et -da activent et désactivent respectivement la vérification Java assert :

  • Toutes les vérifications d’assertion sont désactivées par défaut.
  • L’option -ea permet de vérifier toutes les assertions
  • Le -ea:<packagename>... permet de vérifier les assertions dans un package et tous les sous-packages.
  • Le -ea:<classname>... permet de vérifier les assertions dans une classe.
  • L’option -da désactive la vérification de toutes les assertions
  • Le -da:<packagename>... désactive la vérification des assertions dans un package et tous les sous-packages.
  • Le -da:<classname>... désactive la vérification des assertions dans une classe.
  • L’option -esa permet de vérifier toutes les classes système.
  • L’option -dsa désactive la vérification de toutes les classes système.

Les options peuvent être combinées. Par exemple.

$ # Enable all assertion checking in non-system classes 
$ java -ea -dsa MyApp

$ # Enable assertions for all classes in a package except for one.
$ java -ea:com.wombat.fruitbat... -da:com.wombat.fruitbat.Brickbat MyApp

Notez que l’activation de la vérification d’assertion est susceptible de modifier le comportement d’une programmation Java.

  • Il est susceptible de ralentir l’application en général.
  • L’exécution de méthodes spécifiques peut être plus longue, ce qui peut modifier la synchronisation des threads dans une application multithread.
  • Il peut introduire des relations fortuites * se passe avant * qui peuvent faire disparaître les anomalies de la mémoire.
  • Une instruction assert incorrectement implémentée pourrait avoir des effets secondaires indésirables.

Sélection du type de machine virtuelle #

Les options -client et -server vous permettent de choisir entre deux formes différentes de la machine virtuelle HotSpot :

  • Le formulaire “client” est adapté aux applications utilisateur et offre un démarrage plus rapide.
  • Le formulaire “serveur” est adapté aux applications de longue durée. Il faut plus de temps pour capturer les statistiques pendant le “préchauffage” de la JVM, ce qui permet au compilateur JIT de faire un meilleur travail d’optimisation du code natif.

Par défaut, la JVM fonctionnera en mode 64 bits si possible, selon les capacités de la plate-forme. Les options -d32 et -d64 vous permettent de sélectionner explicitement le mode.
