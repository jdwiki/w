Le framework de collections dans java.util fournit un certain nombre de classes génériques pour des ensembles de données avec des fonctionnalités qui ne peuvent pas être fournies par des tableaux normaux.

Le framework de collections contient des interfaces pour Collection<O>, avec les sous-interfaces principales List<O> et Set<O>, et la collection de mappage Map<K,V>. Les collections sont l’interface racine et sont implémentées par de nombreux autres frameworks de collecte.

Les collections sont des objets qui peuvent stocker des collections d’autres objets à l’intérieur d’eux. Vous pouvez spécifier le type de données stockées dans une collection à l’aide de Generics.

Les collections utilisent généralement les espaces de noms java.util ou java.util.concurrent.

Java 1.4.2 et les versions antérieures ne prennent pas en charge les génériques. En tant que tel, vous ne pouvez pas spécifier les paramètres de type contenus dans une collection. En plus de ne pas avoir de sécurité de type, vous devez également utiliser des moulages pour récupérer le type correct à partir d’une collection.

En plus de Collection<E>, il existe plusieurs types principaux de collections, dont certaines ont des sous-types.

  • List<E> est une collection ordonnée d’objets. Il est similaire à un tableau, mais ne définit pas de limite de taille. Les implémentations augmenteront généralement en taille en interne pour accueillir de nouveaux éléments.
  • Set<E> est une collection d’objets qui n’autorise pas les doublons.
  • SortedSet<E> est un Set<E> qui spécifie également l’ordre des éléments.
  • Map<K,V> est une collection de paires clé/valeur.
  • SortedMap<K,V> est une Map<K,V> qui spécifie également l’ordre des éléments.

Java 5 ajoute un nouveau type de collection :

  • Queue<E> est une collection d’éléments destinés à être traités dans un ordre spécifique. L’implémentation précise s’il s’agit de FIFO ou de LIFO. Cela rend obsolète la classe Stack.

Java 6 ajoute de nouveaux sous-types de collections.

  • NavigableSet<E> est un Set<E> avec des méthodes de navigation spéciales intégrées.
  • NavigableMap<K,V> est une Map<K,V> avec des méthodes de navigation spéciales intégrées.
  • Deque<E> est une Queue<E> qui peut être lue des deux côtés.

Notez que les éléments ci-dessus sont tous des interfaces. Pour les utiliser, vous devez trouver les classes d’implémentation appropriées, telles que ArrayList, HashSet, HashMap ou PriorityQueue.

Chaque type de collection a plusieurs implémentations qui ont des métriques de performances et des cas d’utilisation différents.

Notez que le [principe de substitution de Liskov](https://www.wikiod.com/fr/java/heritage#Le principe de substitution de Liskov) s’applique aux sous-types de collection. C’est-à-dire qu’un SortedSet<E> peut être passé à une fonction attendant un Set<E>. Il est également utile de lire sur Bounded Parameters dans la section Génériques pour plus d’informations sur l’utilisation des collections avec héritage de classe.

Si vous souhaitez créer vos propres collections, il peut être plus facile d’hériter de l’une des classes abstraites (telles que AbstractList) au lieu d’implémenter l’interface.

Avant la version 1.2, vous deviez utiliser les classes/interfaces suivantes à la place :

  • Vector au lieu de ArrayList
  • Dictionnaire au lieu de Carte. Notez que Dictionary est également une classe abstraite plutôt qu’une interface.
  • Hashtable au lieu de HashMap

Ces classes sont obsolètes et ne doivent pas être utilisées dans le code moderne.

Suppression d’éléments d’une liste dans une boucle #

Il est difficile de supprimer des éléments d’une liste dans une boucle, cela est dû au fait que l’index et la longueur de la liste sont modifiés.

Compte tenu de la liste suivante, voici quelques exemples qui donneront un résultat inattendu et d’autres qui donneront le résultat correct.

List<String> fruits = new ArrayList<String>();
fruits.add("Apple");
fruits.add("Banana");
fruits.add("Strawberry");

INCORRECT #

Suppression dans l’itération de l’instruction for Ignore “Banana”: #

L’exemple de code n’affichera que “Pomme” et “Fraise”. Banana est ignoré car il passe à l’index 0 une fois que Apple est supprimé, mais en même temps i est incrémenté à 1.

for (int i = 0; i < fruits.size(); i++) {
    System.out.println (fruits.get(i)); 
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
    }     
}

Suppression dans l’instruction for améliorée Lève une exception : #

En raison de l’itération sur la collection et de sa modification en même temps.

Lancer : java.util.ConcurrentModificationException

for (String fruit : fruits) { 
    System.out.println(fruit);
    if ("Apple".equals(fruit)) {
        fruits.remove(fruit);
    }
}

CORRIGER #

Suppression de la boucle while à l’aide d’un Iterator #

Iterator<String> fruitIterator = fruits.iterator();
while(fruitIterator.hasNext()) {     
    String fruit = fruitIterator.next();     
    System.out.println(fruit);
    if ("Apple".equals(fruit)) {
        fruitIterator.remove();
    } 
}

L’interface Iterator a une méthode remove() intégrée juste pour ce cas. Cependant, cette méthode est marquée comme “facultative” dans la documentation, et elle pourrait lancer une UnsupportedOperationException.

Lancer : UnsupportedOperationException - if the remove operation is not supported by this iterator

Par conséquent, il est conseillé de vérifier la documentation pour s’assurer que cette opération est prise en charge (en pratique, sauf si la collection est une collection immuable obtenue via une bibliothèque tierce ou l’utilisation de l’un des Collections.unmodifiable...() méthode, l’opération est presque toujours prise en charge).

Lors de l’utilisation d’un Iterator, une ConcurrentModificationException est lancée lorsque le modCount de la List est modifié depuis la création de Iterator. Cela aurait pu se produire dans le même thread ou dans une application multithread partageant la même liste.

Un modCount est une variable int qui compte le nombre de fois que cette liste a été structurellement modifiée. Un changement structurel signifie essentiellement qu’une opération add() ou remove() est invoquée sur l’objet Collection (les modifications apportées par Iterator ne sont pas comptées). Lorsque le Iterator est créé, il stocke ce modCount et à chaque itération de la List vérifie si le modCount actuel est le même que celui et quand le Iterator a été créé. S’il y a un changement dans la valeur modCount, il lève une ConcurrentModificationException.

Par conséquent, pour la liste déclarée ci-dessus, une opération comme ci-dessous ne lèvera aucune exception :

Iterator<String> fruitIterator = fruits.iterator();
fruits.set(0, "Watermelon");
while(fruitIterator.hasNext()){
    System.out.println(fruitIterator.next());
}

Mais l’ajout d’un nouvel élément à la List après l’initialisation d’un Iterator lèvera une ConcurrentModificationException :

Iterator<String> fruitIterator = fruits.iterator();
fruits.add("Watermelon");
while(fruitIterator.hasNext()){
    System.out.println(fruitIterator.next());    //ConcurrentModificationException here
}

Itération vers l’arrière #

for (int i = (fruits.size() - 1); i >=0; i--) {
    System.out.println (fruits.get(i));
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
    }
}

Cela ne saute rien. L’inconvénient de cette approche est que la sortie est inversée. Cependant, dans la plupart des cas où vous supprimez des éléments qui n’auront pas d’importance. Vous ne devriez jamais faire cela avec [LinkedList][1].

Itération vers l’avant, ajustement de l’index de boucle #

for (int i = 0; i < fruits.size(); i++) {
    System.out.println (fruits.get(i)); 
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
         i--;
    }     
}

Cela ne saute rien. Lorsque le ith élément est supprimé de la List, l’élément initialement positionné à l’index i+1 devient le nouveau ith élément. Par conséquent, la boucle peut décrémenter i afin que la prochaine itération traite l’élément suivant, sans sauter.

Utilisation d’une liste “devrait être supprimé” #

ArrayList shouldBeRemoved = new ArrayList();
for (String str : currentArrayList) {
    if (condition) {
        shouldBeRemoved.add(str);
    }
}
currentArrayList.removeAll(shouldBeRemoved);

Cette solution permet au développeur de vérifier si les éléments corrects sont supprimés de manière plus propre.

Dans Java 8, les alternatives suivantes sont possibles. Celles-ci sont plus propres et plus simples si la suppression ne doit pas se faire en boucle.

Filtrage d’un flux #

Une “Liste” peut être diffusée et filtrée. Un filtre approprié peut être utilisé pour supprimer tous les éléments indésirables.

List<String> filteredList = 
    fruits.stream().filter(p -> !"Apple".equals(p)).collect(Collectors.toList());

Notez que contrairement à tous les autres exemples ici, cet exemple produit une nouvelle instance List et conserve la List d’origine inchangée.

Utilisation de removeIf #

Économise la surcharge de construction d’un flux si tout ce qui est nécessaire est de supprimer un ensemble d’éléments.

fruits.removeIf(p -> "Apple".equals(p));

Collections vides immuables #

Il est parfois approprié d’utiliser une collection vide immuable. La classe [Collections][1] fournit des méthodes pour obtenir de telles collections de manière efficace :

List<String> anEmptyList = Collections.emptyList();
Map<Integer, Date> anEmptyMap = Collections.emptyMap();
Set<Number> anEmptySet = Collections.emptySet();

Ces méthodes sont génériques et convertiront automatiquement la collection renvoyée dans le type auquel elle est affectée. C’est-à-dire qu’une invocation de par ex. emptyList() peut être assigné à n’importe quel type de List et de même pour emptySet() et emptyMap().

Les collections renvoyées par ces méthodes sont immuables en ce sens qu’elles lèveront UnsupportedOperationException si vous tentez d’appeler des méthodes qui modifieraient leur contenu (add, put, etc.). Ces collections sont principalement utiles pour remplacer les résultats de méthode vides ou d’autres valeurs par défaut, au lieu d’utiliser null ou de créer des objets avec new.

[1] : https://docs.oracle.com/javase/7/docs/api/java/util/Collections.html

Collection non modifiable #

Parfois, ce n’est pas une bonne pratique d’exposer une collection interne car cela peut conduire à une vulnérabilité de code malveillant en raison de sa caractéristique mutable. Afin de fournir des collections “en lecture seule”, java fournit ses versions non modifiables.

Une collection non modifiable est souvent une copie d’une collection modifiable qui garantit que la collection elle-même ne peut pas être modifiée. Les tentatives de modification entraîneront une exception UnsupportedOperationException.

Il est important de noter que les objets présents à l’intérieur de la collection peuvent encore être modifiés.

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class MyPojoClass {
    private List<Integer> intList = new ArrayList<>();

    public void addValueToIntList(Integer value){
        intList.add(value);
    }
    
    public List<Integer> getIntList() {
        return Collections.unmodifiableList(intList);
    }
}

La tentative suivante de modification d’une collection non modifiable lèvera une exception :

import java.util.List;

public class App {

    public static void main(String[] args) {
        MyPojoClass pojo = new MyPojoClass();
        pojo.addValueToIntList(42);
        
        List<Integer> list = pojo.getIntList();
        list.add(69);
    }
}

production:

Exception in thread "main" java.lang.UnsupportedOperationException
    at java.util.Collections$UnmodifiableCollection.add(Collections.java:1055)
    at App.main(App.java:12)

Piège : exceptions de modification simultanée #

Cette exception se produit lorsqu’une collection est modifiée lors d’une itération sur celle-ci à l’aide de méthodes autres que celles fournies par l’objet itérateur. Par exemple, nous avons une liste de chapeaux et nous voulons supprimer tous ceux qui ont des oreillettes :

List<IHat> hats = new ArrayList<>();
hats.add(new Ushanka()); // that one has ear flaps
hats.add(new Fedora());
hats.add(new Sombrero());
for (IHat hat : hats) {
    if (hat.hasEarFlaps()) {
        hats.remove(hat);
    }
}

Si nous exécutons ce code, ConcurrentModificationException sera déclenché puisque le code modifie la collection lors de son itération. La même exception peut se produire si l’un des multiples threads travaillant avec la même liste essaie de modifier la collection tandis que d’autres itèrent dessus. La modification simultanée de collections dans plusieurs threads est une chose naturelle, mais doit être traitée avec les outils habituels de la boîte à outils de programmation simultanée tels que les verrous de synchronisation, les collections spéciales adoptées pour la modification simultanée, la modification de la collection clonée à partir de l’initiale, etc.

Sous-collections #

Liste subList(int fromIndex, int toIndex) #

Ici, fromIndex est inclusif et toIndex est exclusif.

List list = new ArrayList(); 
List list1 = list.subList(fromIndex,toIndex);
  1. Si la liste n’existe pas dans la plage donnée, elle lève IndexOutofBoundException.
  2. Quelles que soient les modifications apportées à la liste1, elles auront un impact sur les mêmes modifications dans la liste. C’est ce qu’on appelle les collections sauvegardées.
  3. Si le fromnIndex est supérieur au toIndex (fromIndex > toIndex), il lève IllegalArgumentException.

Exemple:

List<String> list = new ArrayList<String>();
List<String> list = new ArrayList<String>();
list.add("Hello1"); 
list.add("Hello2"); 
System.out.println("Before Sublist "+list); 
List<String> list2 = list.subList(0, 1);
list2.add("Hello3"); 
System.out.println("After sublist changes "+list);

Sortie :
Avant la sous-liste [Hello1, Hello2]
Après les changements de sous-liste [Hello1, Hello3, Hello2]

Set subSet(fromIndex,toIndex) #

Ici, fromIndex est inclusif et toIndex est exclusif.

Set set = new TreeSet(); 
Set set1 = set.subSet(fromIndex,toIndex);

L’ensemble renvoyé lèvera une IllegalArgumentException lors d’une tentative d’insertion d’un élément en dehors de sa plage.

Map subMap(fromKey,toKey) #

fromKey est inclusif et toKey est exclusif

Map map = new TreeMap();
Map map1 = map.get(fromKey,toKey);

Si fromKey est supérieur à toKey ou si cette carte elle-même a une plage restreinte, et fromKey ou toKey se trouve en dehors des limites de la plage, elle lève IllegalArgumentException.

Toutes les collections prennent en charge les collections sauvegardées, ce qui signifie que les modifications apportées à la sous-collection auront la même modification sur la collection principale.

Rejoindre des listes #

Les méthodes suivantes peuvent être utilisées pour rejoindre des listes sans modifier la ou les listes de sources.

Première approche. A plus de lignes mais facile à comprendre

List<String> newList = new ArrayList<String>();

Deuxième approche. A une ligne de moins mais moins lisible.

List<String> newList = new ArrayList<String>(listOne);

Troisième approche. Nécessite une bibliothèque tierce [Apache commons-collections][1].

ListUtils.union(listOne,listTwo);

List<String> newList = Stream.concat(listOne.stream(), listTwo.stream()).collect(Collectors.toList());

Collections et valeurs primitives #

Les collections en Java ne fonctionnent que pour les objets. C’est à dire. il n’y a pas de Map<int, int> en Java. Au lieu de cela, les valeurs primitives doivent être boxées dans des objets, comme dans Map<Integer, Integer>. L’auto-boxing Java permettra une utilisation transparente de ces collections :

Map<Integer, Integer> map = new HashMap<>();
map.put(1, 17); // Automatic boxing of int to Integer objects
int a = map.get(1); // Automatic unboxing.

Malheureusement, la surcharge de ceci est substantielle. Un HashMap<Integer, Integer> nécessitera environ 72 octets par entrée (par exemple, sur une JVM 64 bits avec des pointeurs compressés, et en supposant des entiers supérieurs à 256, et en supposant une charge de 50 % de la carte). Étant donné que les données réelles ne sont que de 8 octets, cela génère une surcharge massive. De plus, il nécessite deux niveaux d’indirection (Carte -> Entrée -> Valeur) il est inutilement lent.

Il existe plusieurs bibliothèques avec des collections optimisées pour les types de données primitifs (qui ne nécessitent que ~ 16 octets par entrée à 50% de charge, c’est-à-dire 4 fois moins de mémoire et un niveau d’indirection de moins), qui peuvent apporter des avantages de performances substantiels lors de l’utilisation de grandes collections de primitives valeurs en Java.

Création de votre propre structure Iterable à utiliser avec Iterator ou la boucle for-each. #

Pour nous assurer que notre collection peut être itérée à l’aide d’un itérateur ou d’une boucle for-each, nous devons suivre les étapes suivantes :

  1. Les éléments sur lesquels nous voulons itérer doivent être “itérables” et exposés itérateur().
  2. Créez un java.util.Iterator en remplaçant hasNext(), next() et remove().

J’ai ajouté ci-dessous une implémentation de liste liée générique simple qui utilise les entités ci-dessus pour rendre la liste liée itérable.

package org.algorithms.linkedlist;
 
import java.util.Iterator;
import java.util.NoSuchElementException;
 
 
public class LinkedList<T> implements Iterable<T> {
 
    Node<T> head, current;
 
    private static class Node<T> {
        T data;
        Node<T> next;
 
        Node(T data) {
            this.data = data;
        }
    }
 
    public LinkedList(T data) {
        head = new Node<>(data);
    }
 
    public Iterator<T> iterator() {
        return new LinkedListIterator();
    }
 
    private class LinkedListIterator implements Iterator<T> {
 
        Node<T> node = head;
 
        @Override
        public boolean hasNext() {
            return node != null;
        }
 
        @Override
        public T next() {
            if (!hasNext())
                throw new NoSuchElementException();
            Node<T> prevNode = node;
            node = node.next;
            return prevNode.data;
        }
 
        @Override
        public void remove() {
            throw new UnsupportedOperationException("Removal logic not implemented.");
        }
    }
 
    public void add(T data) {
        Node current = head;
        while (current.next != null)
            current = current.next;
        current.next = new Node<>(data);
    }
 
}
 
class App {
    public static void main(String[] args) {
 
        LinkedList<Integer> list = new LinkedList<>(1);
        list.add(2);
        list.add(4);
        list.add(3);
 
        //Test #1
        System.out.println("using Iterator:");
        Iterator<Integer> itr = list.iterator();
        while (itr.hasNext()) {
            Integer i = itr.next();
            System.out.print(i + " ");
        }
 
        //Test #2
        System.out.println("\n\nusing for-each:");
        for (Integer data : list) {
            System.out.print(data + " ");
        }
    }
}

Production

using Iterator:
1 2 4 3
using for-each:
1 2 4 3

Cela fonctionnera en Java 7+. Vous pouvez également le faire fonctionner sur Java 5 et Java 6 en remplaçant :

LinkedList<Integer> list = new LinkedList<>(1);

avec

LinkedList<Integer> list = new LinkedList<Integer>(1);

ou simplement toute autre version en incorporant les modifications compatibles.
