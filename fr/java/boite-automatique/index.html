[Autoboxing][1] est la conversion automatique que le compilateur Java effectue entre les types primitifs et leurs classes wrapper d’objet correspondantes. Exemple, convertir int -> Integer, double -> Double… Si la conversion va dans l’autre sens, cela s’appelle unboxing. En règle générale, ceci est utilisé dans les collections qui ne peuvent contenir que des objets, où les types primitifs de boxe sont nécessaires avant de les définir dans la collection.

[1] : http://docs.oracle.com/javase/7/docs/technotes/guides/language/autoboxing.html

L’autoboxing peut avoir des problèmes de performances lorsqu’il est utilisé fréquemment dans votre code.

  • http://docs.oracle.com/javase/1.5.0/docs/guide/language/autoboxing.html
  • http://stackoverflow.com/questions/6037389/integer-auto-unboxing-and-auto-boxing-gives-performance-issues

Utiliser int et Integer indifféremment #

Lorsque vous utilisez des types génériques avec des classes utilitaires, vous pouvez souvent constater que les types numériques ne sont pas très utiles lorsqu’ils sont spécifiés comme types d’objet, car ils ne sont pas égaux à leurs homologues primitifs.

List<Integer> ints = new ArrayList<Integer>();

List<Integer> ints = new ArrayList<>();

Heureusement, les expressions qui évaluent à int peuvent être utilisées à la place d’un Integer lorsque cela est nécessaire.

for (int i = 0; i < 10; i++)
    ints.add(i);

L’instruction ints.add(i); est équivalente à :

ints.add(Integer.valueOf(i));

Et conserve les propriétés de Integer#valueOf telles que le fait d’avoir les mêmes objets Integer mis en cache par la JVM lorsqu’ils se trouvent dans la plage de mise en cache des nombres.

Cela s’applique également à :

  • ‘octet’ et ‘octet’
  • “court” et “court”
  • ‘float’ et ‘Flottant’
  • ‘double’ et ‘Double’
  • ‘Longue’ et ‘Longue’
  • car et Caractère
  • ‘booléen’ et ‘booléen’

Il faut cependant être prudent dans les situations ambiguës. Considérez le code suivant :

List<Integer> ints = new ArrayList<Integer>();
ints.add(1);
ints.add(2);
ints.add(3);
ints.remove(1); // ints is now [1, 3]

L’interface java.util.List contient à la fois une remove(int index) (méthode d’interface List) et une remove(Object o) (méthode héritée de java.util.Collection). Dans ce cas, aucun boxing n’a lieu et remove(int index) est appelé.

Un autre exemple de comportement étrange du code Java causé par l’autoboxing d’entiers avec des valeurs comprises entre -128 et 127 :

Integer a = 127;
Integer b = 127;
Integer c = 128;
Integer d = 128;
System.out.println(a == b); // true
System.out.println(c <= d); // true
System.out.println(c >= d); // true
System.out.println(c == d); // false

Cela se produit parce que l’opérateur >= appelle implicitement intValue() qui renvoie int tandis que == compare les références, et non les valeurs int.

Par défaut, Java met en cache les valeurs dans la plage [-128, 127], donc l’opérateur == fonctionne car les Integers de cette plage font référence aux mêmes objets si leurs valeurs sont identiques. La valeur maximale de la plage pouvant être mise en cache peut être définie avec l’option JVM -XX:AutoBoxCacheMax. Ainsi, si vous exécutez le programme avec -XX:AutoBoxCacheMax=1000, le code suivant affichera true :

Integer a = 1000;
Integer b = 1000;
System.out.println(a == b); // true

Le déballage automatique peut conduire à NullPointerException #

Ce code compile :

Integer arg = null;
int x = arg;

Mais il plantera au moment de l’exécution avec une java.lang.NullPointerException sur la deuxième ligne.

Le problème est qu’une primitive int ne peut pas avoir une valeur null.

Ceci est un exemple minimaliste, mais en pratique, il se manifeste souvent sous des formes plus sophistiquées. Le NullPointerException n’est pas très intuitif et est souvent peu utile pour localiser de tels bogues.

Comptez sur l’autoboxing et l’auto-unboxing avec précaution, assurez-vous que les valeurs non encadrées n’auront pas de valeurs “nulles” au moment de l’exécution.

Utilisation de booléen dans l’instruction if #

En raison du déballage automatique, on peut utiliser un Boolean dans une instruction if :

Boolean a = Boolean.TRUE;
if (a) { // a gets converted to boolean
    System.out.println("It works!");
}

Cela fonctionne pour while, do while et la condition dans les instructions for également.

Notez que, si le Boolean est null, une NullPointerException sera lancée lors de la conversion.

Mémoire et surcoût de calcul de l’Autoboxing #

L’autoboxing peut entraîner une surcharge de mémoire importante. Par exemple:

Map<Integer, Integer> square = new HashMap<Integer, Integer>();
for(int i = 256; i < 1024; i++) {
    square.put(i, i * i); // Autoboxing of large integers
}

consomme généralement une quantité importante de mémoire (environ 60 Ko pour 6 Ko de données réelles).

De plus, les entiers encadrés nécessitent généralement des allers-retours supplémentaires dans la mémoire, et rendent ainsi les caches CPU moins efficaces. Dans l’exemple ci-dessus, la mémoire accédée est répartie sur cinq emplacements différents qui peuvent se trouver dans des régions entièrement différentes de la mémoire : 1. l’objet HashMap, 2. l’objet Entry[] table de la carte, 3. l’objet Entry objet, 4. les entrées objet clé (encadrant la clé primitive), 5. les entrées objet valeur (encadrant la valeur primitive).

class Example {
  int primitive; // Stored directly in the class `Example`
  Integer boxed; // Reference to another memory location
}

La lecture boxed nécessite deux accès mémoire, l’accès primitive un seul.

Lors de l’obtention de données à partir de cette carte, le code apparemment innocent

int sumOfSquares = 0;
for(int i = 256; i < 1024; i++) {
    sumOfSquares += square.get(i);
}

est équivalent à:

int sumOfSquares = 0;
for(int i = 256; i < 1024; i++) {
    sumOfSquares += square.get(Integer.valueOf(i)).intValue();
}

Typiquement, le code ci-dessus provoque la création et la récupération de place d’un objet Integer pour chaque opération Map#get(Integer). (Voir la note ci-dessous pour plus de détails.)

Pour réduire cette surcharge, plusieurs bibliothèques proposent des collections optimisées pour les types primitifs qui ne nécessitent pas de boxe. En plus d’éviter les frais généraux de boxe, ces collectes nécessiteront environ 4 fois moins de mémoire par entrée. Bien que Java Hotspot * puisse * être en mesure d’optimiser l’autoboxing en travaillant avec des objets sur la pile au lieu du tas, il n’est pas possible d’optimiser la surcharge de mémoire et l’indirection de mémoire qui en résulte.

Les flux Java 8 ont également des interfaces optimisées pour les types de données primitifs, tels que “IntStream”, qui ne nécessitent pas de boxe.

Remarque : un environnement d’exécution Java typique maintient un cache simple de Integer et d’autres objets wrapper primitifs utilisés par les méthodes de fabrique valueOf et par autoboxing. Pour Integer, la plage par défaut de ce cache est de -128 à +127. Certaines JVM fournissent une option de ligne de commande JVM pour modifier la taille/la plage de cache.

Différents cas où Integer et int peuvent être utilisés de manière interchangeable #

Cas 1: Tout en utilisant à la place des arguments de méthode.

Si une méthode nécessite un objet de classe wrapper comme argument, alors l’argument peut être passé de manière interchangeable à une variable du type primitif respectif et vice versa.

Exemple:

int i;
Integer j;
void ex_method(Integer i)//Is a valid statement
void ex_method1(int j)//Is a valid statement

Cas 2 : Lors du passage des valeurs de retour :

Lorsqu’une méthode renvoie une variable de type primitif, un objet de la classe wrapper correspondante peut être transmis comme valeur de retour de manière interchangeable et vice versa.

Exemple:

int i;
Integer j;
int ex_method()
{...
return j;}//Is a valid statement
Integer ex_method1()
{...
return i;//Is a valid statement
}

Cas 3 : Lors de l’exécution d’opérations.

Chaque fois que vous effectuez des opérations sur des nombres, la variable de type primitif et l’objet de la classe wrapper respective peuvent être utilisés de manière interchangeable.

int i=5;
Integer j=new Integer(7);
int k=i+j;//Is a valid statement
Integer m=i+j;//Is also a valid statement

Ecueil :N’oubliez pas d’initialiser ou d’attribuer une valeur à un objet de la classe wrapper.

Lors de l’utilisation interchangeable d’un objet de classe wrapper et d’une variable primitive, n’oubliez jamais ou ne manquez jamais d’initialiser ou d’attribuer une valeur à l’objet de classe wrapper, sinon cela peut entraîner une exception de pointeur null lors de l’exécution.

Exemple:

public class Test{
    Integer i;
    int j;
    public void met()
    {j=i;//Null pointer exception
    SOP(j);
    SOP(i);}   
    public static void main(String[] args)
    {Test t=new Test();
    t.go();//Null pointer exception
    }

Dans l’exemple ci-dessus, la valeur de l’objet est non affectée et non initialisée et donc au moment de l’exécution, le programme se heurtera à une exception de pointeur nul. Ainsi, comme il ressort de l’exemple ci-dessus, la valeur de l’objet ne doit jamais être laissée non initialisée et non affectée.
