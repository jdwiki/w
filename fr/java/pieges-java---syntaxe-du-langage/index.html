Plusieurs utilisations abusives du langage de programmation Java peuvent conduire un programme à générer des résultats incorrects malgré une compilation correcte. L’objectif principal de cette rubrique est de répertorier les pièges courants avec leurs causes, et de proposer la bonne façon d’éviter de tomber dans de tels problèmes.

Cette rubrique concerne des aspects spécifiques de la syntaxe du langage Java qui sont sujets aux erreurs ou qui ne doivent pas être utilisés de certaines manières.

Piège - Manquer une “pause” dans un cas de “commutation” #

Ces problèmes Java peuvent être très embarrassants et restent parfois inconnus jusqu’à leur exécution en production. Le comportement de secours dans les instructions switch est souvent utile ; cependant, manquer un mot-clé “break” lorsqu’un tel comportement n’est pas souhaité peut conduire à des résultats désastreux. Si vous avez oublié de mettre un “break” dans “case 0” dans l’exemple de code ci-dessous, le programme écrira “Zero” suivi de “One”, puisque le flux de contrôle à l’intérieur ici passera par l’intégralité de l’instruction “switch” jusqu’à ce que il atteint une “pause”. Par exemple:

public static void switchCasePrimer() {
        int caseIndex = 0;
        switch (caseIndex) {
            case 0:
                System.out.println("Zero");
            case 1:
                System.out.println("One");
                break;
            case 2:
                System.out.println("Two");
                break;
            default:
                System.out.println("Default");
        }
}

Dans la plupart des cas, la solution la plus propre consisterait à utiliser des interfaces et à déplacer le code avec un comportement spécifique dans des implémentations distinctes (composition plutôt que l’héritage)

Si une instruction switch est inévitable, il est recommandé de documenter les échecs “attendus” s’ils se produisent. De cette façon, vous montrez aux autres développeurs que vous êtes conscient de la pause manquante et que c’est un comportement attendu.

switch(caseIndex) {
    [...]
    case 2:
        System.out.println("Two");
        // fallthrough
    default:
        System.out.println("Default");

Piège - Oublier les accolades : les problèmes “dangling if” et “dangling else” #

La dernière version du guide de style Oracle Java stipule que les instructions “then” et “else” dans une instruction “if” doivent toujours être placées entre “accolades” ou “accolades”. Des règles similaires s’appliquent aux corps de diverses instructions de boucle.

if (a) {           // <- open brace
    doSomething();
    doSomeMore();
}                  // <- close brace

Ce n’est pas réellement requis par la syntaxe du langage Java. En effet, si la partie “then” d’une instruction “if” est une instruction unique, il est légal d’omettre les accolades

if (a)
    doSomething();

ou même

if (a) doSomething();

Cependant, il y a des dangers à ignorer les règles de style Java et à omettre les accolades. Plus précisément, vous augmentez considérablement le risque que le code avec une indentation défectueuse soit mal lu.

Le problème du “si suspendu” :

Considérez l’exemple de code ci-dessus, réécrit sans accolades.

if (a)
   doSomething();
   doSomeMore();

Ce code semble dire que les appels à doSomething et doSomeMore se produiront tous les deux si et seulement si a est true. En fait, le code est mal indenté. La spécification du langage Java selon laquelle l’appel doSomeMore() est une instruction distincte suivant l’instruction if. L’indentation correcte est la suivante :

if (a)
   doSomething();
doSomeMore();

Le problème du “reste suspendu”

Un deuxième problème apparaît lorsque nous ajoutons “else” au mélange. Considérez l’exemple suivant avec des accolades manquantes.

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
else
   doZ();

Le code ci-dessus semble dire que doZ sera appelé lorsque a est false. En fait, l’indentation est encore une fois incorrecte. L’indentation correcte pour le code est :

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
   else
      doZ();

Si le code était écrit selon les règles de style Java, il ressemblerait en fait à ceci :

if (a) {
   if (b) {
      doX();
   } else if (c) {
      doY(); 
   } else {
      doZ();
   }
}

Pour illustrer pourquoi c’est mieux, supposons que vous ayez accidentellement mal indenté le code. Vous pourriez vous retrouver avec quelque chose comme ceci :

if (a) {                         if (a) {
   if (b) {                          if (b) {
      doX();                            doX();
   } else if (c) {                   } else if (c) {
      doY();                            doY();
} else {                         } else {
   doZ();                            doZ();
}                                    }
}                                }

Mais dans les deux cas, le code mal indenté “semble faux” aux yeux d’un programmeur Java expérimenté.

Piège - Déclarer des classes avec les mêmes noms que les classes standard #

Parfois, les programmeurs novices en Java commettent l’erreur de définir une classe avec un nom identique à celui d’une classe largement utilisée. Par exemple:

package com.example;

/**
 * My string utilities
 */
public class String {
    ....
}

Ensuite, ils se demandent pourquoi ils obtiennent des erreurs inattendues. Par exemple:

package com.example;

public class Test {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

Si vous compilez puis essayez d’exécuter les classes ci-dessus, vous obtiendrez une erreur :

$ javac com/example/*.java
$ java com.example.Test
Error: Main method not found in class test.Test, please define the main method as:
   public static void main(String[] args)
or a JavaFX application class must extend javafx.application.Application

Quelqu’un regardant le code de la classe Test verrait la déclaration de main et regarderait sa signature et se demanderait de quoi se plaint la commande java. Mais en fait, la commande java dit la vérité.

Lorsque nous déclarons une version de String dans le même package que Test, cette version a priorité sur l’importation automatique de java.lang.String. Ainsi, la signature de la méthode Test.main est en fait

void main(com.example.String[] args)

à la place de

void main(java.lang.String[] args)

et la commande java ne reconnaîtra pas that comme méthode de point d’entrée.

Leçon : Ne définissez pas de classes portant le même nom que les classes existantes dans java.lang ou d’autres classes couramment utilisées dans la bibliothèque Java SE. Si vous faites cela, vous vous exposez à toutes sortes d’erreurs obscures.

Piège - Ignorer la visibilité de la méthode #

Même les développeurs Java expérimentés ont tendance à penser que Java n’a que trois modificateurs de protection. La langue en compte en fait quatre ! Le niveau de visibilité package privé (c’est-à-dire par défaut) est souvent oublié.

Vous devez faire attention aux méthodes que vous rendez publiques. Les méthodes publiques d’une application sont l’API visible de l’application. Celle-ci doit être aussi petite et compacte que possible, surtout si vous écrivez une bibliothèque réutilisable (voir aussi le principe [SOLID][1]). Il est important de considérer de la même manière la visibilité de toutes les méthodes et de n’utiliser que l’accès privé protégé ou en package le cas échéant.

Lorsque vous déclarez des méthodes qui devraient être privées comme publiques, vous exposez les détails d’implémentation internes de la classe.

Un corollaire à cela est que vous [testez unitaire][2] uniquement les méthodes publiques de votre classe - en fait, vous ne pouvez tester que les méthodes publiques. C’est une mauvaise pratique d’augmenter la visibilité des méthodes privées juste pour pouvoir exécuter des tests unitaires sur ces méthodes. Tester les méthodes publiques qui appellent les méthodes avec une visibilité plus restrictive devrait être suffisant pour tester une API entière. Vous ne devriez jamais étendre votre API avec plus de méthodes publiques uniquement pour autoriser les tests unitaires.

[1] : https://en.wikipedia.org/wiki/Open/closed_principle [2] : http://www.tutorialspoint.com/junit/junit_test_framework.htm

Piège - Littéraux octaux #

Considérez l’extrait de code suivant :

// Print the sum of the numbers 1 to 10
int count = 0;
for (int i = 1; i < 010; i++) {    // Mistake here ....
    count = count + i;
}
System.out.println("The sum of 1 to 10 is " + count);

Un débutant Java pourrait être surpris de savoir que le programme ci-dessus affiche la mauvaise réponse. Il imprime en fait la somme des nombres 1 à 8.

La raison en est qu’un littéral entier qui commence par le chiffre zéro (“0”) est interprété par le compilateur Java comme un littéral octal, et non comme un littéral décimal comme on pourrait s’y attendre. Ainsi, ‘010’ est le nombre octal 10, qui est 8 en décimal.

Piège - Utiliser ‘==’ pour tester un booléen #

Parfois, un nouveau programmeur Java écrira un code comme celui-ci :

public void check(boolean ok) {
    if (ok == true) {           // Note 'ok == true'
        System.out.println("It is OK");
    }
}

Un programmeur expérimenté remarquerait que c’est maladroit et voudrait le réécrire comme suit :

public void check(boolean ok) {
    if (ok) {
       System.out.println("It is OK");
    }
}

Cependant, il y a plus de mal avec ok == true qu’une simple maladresse. Considérez cette variante :

public void check(boolean ok) {
    if (ok = true) {           // Oooops!
        System.out.println("It is OK");
    }
}

Ici, le programmeur a mal tapé == comme = … et maintenant le code a un bogue subtil. L’expression x = true attribue inconditionnellement true à x puis évalue à true. En d’autres termes, la méthode check affichera désormais “C’est OK” quel que soit le paramètre.

La leçon ici est de sortir de l’habitude d’utiliser == false et == true. En plus d’être verbeux, ils rendent votre codage plus sujet aux erreurs.

Remarque : Une alternative possible à ok == true qui évite l’écueil consiste à utiliser [conditions Yoda] [1] ; c’est-à-dire mettre le littéral sur le côté gauche de l’opérateur relationnel, comme dans true == ok. Cela fonctionne, mais la plupart des programmeurs conviendraient probablement que les conditions de Yoda semblent étranges. Certes, ok (ou !ok) est plus concis et plus naturel.

[1] : https://en.wikipedia.org/wiki/Yoda_conditions

Piège - Points-virgules mal placés et accolades manquantes #

C’est une erreur qui cause une réelle confusion chez les débutants en Java, du moins la première fois qu’ils le font. Au lieu d’écrire ceci :

if (feeling == HAPPY)
    System.out.println("Smile");
else
    System.out.println("Frown");

ils écrivent accidentellement ceci:

if (feeling == HAPPY);
    System.out.println("Smile");
else
    System.out.println("Frown");

et sont perplexes lorsque le compilateur Java leur dit que le « autre » est mal placé. Le compilateur Java interprète ce qui précède comme suit :

if (feeling == HAPPY)
    /*empty statement*/ ;
System.out.println("Smile");   // This is unconditional
else                           // This is misplaced.  A statement cannot
                               // start with 'else'
System.out.println("Frown");

Dans d’autres cas, il n’y aura pas d’erreurs de compilation, mais le code ne fera pas ce que le programmeur a prévu. Par exemple:

for (int i = 0; i < 5; i++);
    System.out.println("Hello");

n’imprime “Bonjour” qu’une seule fois. Encore une fois, le faux point-virgule signifie que le corps de la boucle “for” est une instruction vide. Cela signifie que l’appel println qui suit est inconditionnel.

Une autre variante :

for (int i = 0; i < 5; i++);
    System.out.println("The number is " + i);

Cela donnera une erreur “Impossible de trouver le symbole” pour i. La présence du faux point-virgule signifie que l’appel println tente d’utiliser i en dehors de sa portée.

Dans ces exemples, il existe une solution simple : supprimez simplement le faux point-virgule. Cependant, il y a des leçons plus profondes à tirer de ces exemples :

  1. Le point-virgule en Java n’est pas un “bruit syntaxique”. La présence ou l’absence d’un point-virgule peut changer le sens de votre programme. Ne vous contentez pas de les ajouter à la fin de chaque ligne.

  2. Ne faites pas confiance à l’indentation de votre code. Dans le langage Java, les espaces blancs supplémentaires au début d’une ligne sont ignorés par le compilateur.

  3. Utilisez un pénétrateur automatique. Tous les IDE et de nombreux éditeurs de texte simples comprennent comment indenter correctement le code Java.

  4. C’est la leçon la plus importante. Suivez les dernières directives de style Java et placez des accolades autour des instructions “then” et “else” et de l’instruction body d’une boucle. L’accolade ouverte ({) ne doit pas être sur une nouvelle ligne.

Si le programmeur a suivi les règles de style, l’exemple if avec un point-virgule mal placé ressemblerait à ceci :

if (feeling == HAPPY); {
    System.out.println("Smile");
} else {
    System.out.println("Frown");
}

Cela semble étrange pour un œil expérimenté. Si vous indentiez automatiquement ce code, il ressemblerait probablement à ceci :

if (feeling == HAPPY); {
                           System.out.println("Smile");
                       } else {
                           System.out.println("Frown");
                       }

qui devrait se démarquer comme faux même pour un débutant.

Piège - Les importations de caractères génériques peuvent rendre votre code fragile #

Considérez l’exemple partiel suivant :

import com.example.somelib.*;
import com.acme.otherlib.*;

public class Test {
    private Context x = new Context();   // from com.example.somelib
    ...
}

Supposons que lorsque vous avez développé le code pour la première fois avec la version 1.0 de somelib et la version 1.0 de otherlib. Ensuite, à un moment ultérieur, vous devez mettre à niveau vos dépendances vers une version ultérieure et vous décidez d’utiliser otherlib version 2.0. Supposons également que l’une des modifications apportées à otherlib entre 1.0 et 2.0 était d’ajouter une classe Context.

Maintenant, lorsque vous recompilez Test, vous obtiendrez une erreur de compilation vous indiquant que Context est une importation ambiguë.

Si vous connaissez la base de code, ce n’est probablement qu’un inconvénient mineur. Si ce n’est pas le cas, vous avez du travail à faire pour résoudre ce problème, ici et potentiellement ailleurs.

Le problème ici est les importations de caractères génériques. D’une part, l’utilisation de caractères génériques peut raccourcir vos classes de quelques lignes. D’autre part:

  • Les modifications compatibles vers le haut d’autres parties de votre base de code, des bibliothèques standard Java ou des bibliothèques tierces peuvent entraîner des erreurs de compilation.

  • La lisibilité en souffre. À moins que vous n’utilisiez un IDE, il peut être difficile de déterminer laquelle des importations de caractères génériques extrait une classe nommée.

La leçon est que c’est une mauvaise idée d’utiliser des importations de caractères génériques dans du code qui doit durer longtemps. Les importations spécifiques (non génériques) ne demandent pas beaucoup d’efforts à maintenir si vous utilisez un IDE, et l’effort en vaut la peine.

Piège : Utiliser ‘assert’ pour la validation d’un argument ou d’une entrée utilisateur #

Une question qui se pose parfois sur StackOverflow est de savoir s’il est approprié d’utiliser assert pour valider les arguments fournis à une méthode, ou même les entrées fournies par l’utilisateur.

La réponse simple est que ce n’est pas approprié.

Les meilleures alternatives incluent:

  • Lancement d’une IllegalArgumentException à l’aide d’un code personnalisé.
  • En utilisant les méthodes “Préconditions” disponibles dans la bibliothèque Google Guava.
  • Utilisation des méthodes “Valider” disponibles dans la bibliothèque Apache Commons Lang3.

Voici ce que la [Java Language Specification (JLS 14.10, for Java 8)][1] conseille à ce sujet :

Généralement, la vérification des assertions est activée pendant le développement et les tests du programme, et désactivée pour le déploiement, afin d’améliorer les performances.

Étant donné que les assertions peuvent être désactivées, les programmes ne doivent pas supposer que les expressions contenues dans les assertions seront évaluées. Ainsi, ces expressions booléennes devraient généralement être exemptes d’effets secondaires. L’évaluation d’une telle expression booléenne ne devrait affecter aucun état visible une fois l’évaluation terminée. Il n’est pas illégal qu’une expression booléenne contenue dans une assertion ait un effet secondaire, mais c’est généralement inapproprié, car cela pourrait faire varier le comportement du programme selon que les assertions sont activées ou désactivées.

À la lumière de cela, les assertions ne doivent pas être utilisées pour la vérification des arguments dans les méthodes publiques. La vérification des arguments fait généralement partie du contrat d’une méthode, et ce contrat doit être respecté, que les assertions soient activées ou désactivées.

Un problème secondaire lié à l’utilisation d’assertions pour la vérification des arguments est que les arguments erronés doivent entraîner une exception d’exécution appropriée (telle que IllegalArgumentException, ArrayIndexOutOfBoundsException ou NullPointerException). Un échec d’assertion ne lèvera pas d’exception appropriée. Encore une fois, il n’est pas illégal d’utiliser des assertions pour la vérification des arguments sur les méthodes publiques, mais c’est généralement inapproprié. Il est prévu que AssertionError ne soit jamais intercepté, mais il est possible de le faire, ainsi les règles des instructions try devraient traiter les assertions apparaissant dans un bloc try de la même manière que le traitement actuel des instructions throw.

[1] : https://docs.oracle.com/javase/specs/jls/se8/html/jls-14.html#jls-14.10

Piège - Surcharger au lieu de passer outre #

Considérez l’exemple suivant :

public final class Person {
    private final String firstName;
    private final String lastName;
   
    public Person(String firstName, String lastName) {
        this.firstName = (firstName == null) ? "" : firstName;
        this.lastName = (lastName == null) ? "" : lastName;
    }

    public boolean equals(String other) {
        if (!(other instanceof Person)) {
            return false;
        }
        Person p = (Person) other;
        return firstName.equals(p.firstName) &&
                lastName.equals(p.lastName);
    }

    public int hashcode() {
        return firstName.hashCode() + 31 * lastName.hashCode();
    }
}

Ce code ne va pas se comporter comme prévu. Le problème est que les méthodes equals et hashcode pour Person ne remplacent pas les méthodes standard définies par Object.

  • La méthode equals a la mauvaise signature. Il doit être déclaré comme equals(Object) et non equals(String).
  • La méthode hashcode porte le mauvais nom. Ce devrait être hashCode() (notez le C majuscule).

Ces erreurs signifient que nous avons déclaré des surcharges accidentelles, et celles-ci ne seront pas utilisées si Person est utilisé dans un contexte polymorphe.

Cependant, il existe un moyen simple de gérer cela (à partir de Java 5). Utilisez l’annotation @Override chaque fois que vous avez l’intention que votre méthode soit un remplacement :

public final class Person {
    ...

    @Override
    public boolean equals(String other) {
        ....
    }

    @Override
    public hashcode() {
        ....
    }
}

Lorsque nous ajoutons une annotation @Override à une déclaration de méthode, le compilateur vérifie que la méthode remplace (ou implémente) une méthode déclarée dans une superclasse ou une interface. Ainsi, dans l’exemple ci-dessus, le compilateur nous donnera deux erreurs de compilation, ce qui devrait suffire à nous alerter de l’erreur.

Piège de l’auto-déballage d’objets nuls en primitives #

public class Foobar {
    public static void main(String[] args) {

        // example: 
        Boolean ignore = null;
        if (ignore == false) {
            System.out.println("Do not ignore!");
        }
    }
}

Le piège ici est que ’null’ est comparé à ‘false’. Puisque nous comparons un booléen primitif à un Booléen, Java tente de déballer l’objet booléen en un équivalent primitif, prêt pour la comparaison. Cependant, comme cette valeur est null, une NullPointerException est levée.

Java est incapable de comparer les types primitifs aux valeurs “null”, ce qui provoque une “NullPointerException” lors de l’exécution. Considérons le cas primitif de la condition false == null ; cela générerait une erreur de compilation “types incomparables : int et ”.
