Compilateur juste à temps (JIT)

Histoire #

Le compilateur Symantec JIT était disponible dans Sun Java à partir de la version 1.1.5, mais il avait des problèmes.

Le compilateur Hotspot JIT a été ajouté à Sun Java dans la version 1.2.2 en tant que plug-in. Dans Java 1.3, JIT était activé par défaut.

(Source : http://stackoverflow.com/questions/692136/when-did-java-get-a-jit-compiler)

Aperçu #

[![entrez la description de l’image ici][1]][1]

Le compilateur Just-In-Time (JIT) est un composant de l’environnement d’exécution Java™ qui améliore les performances des applications Java au moment de l’exécution.

  • Les programmes Java se composent de classes, qui contiennent des plates-formes indépendantes bytecodes pouvant être interprétés par une JVM sur de nombreux ordinateurs différents architectures.
  • A l’exécution, la JVM charge les fichiers de classe, détermine la sémantique de chaque bytecode individuel, et effectue les calcul.

L’utilisation supplémentaire du processeur et de la mémoire pendant l’interprétation signifie qu’une application Java s’exécute plus lentement qu’une application native.

Le compilateur JIT permet d’améliorer les performances des programmes Java en compilant les bytecodes en code machine natif au moment de l’exécution.

Le compilateur JIT est activé par défaut et est activé lorsqu’une méthode Java est appelée. Le compilateur JIT compile les bytecodes de cette méthode en code machine natif, en le compilant "juste à temps" pour s’exécuter.

Lorsqu’une méthode a été compilée, la JVM appelle directement le code compilé de cette méthode au lieu de l’interpréter. Théoriquement, si la compilation ne nécessitait pas de temps processeur ni d’utilisation de la mémoire, la compilation de chaque méthode pourrait permettre à la vitesse du programme Java de se rapprocher de celle d’une application native.

La compilation JIT nécessite du temps processeur et de l’utilisation de la mémoire. Lorsque la JVM démarre pour la première fois, des milliers de méthodes sont appelées. La compilation de toutes ces méthodes peut affecter considérablement le temps de démarrage, même si le programme atteint finalement de très bonnes performances de pointe.

  • En pratique, les méthodes ne sont pas compilées la première fois qu’elles sont appelées. Pour chaque méthode, la JVM maintient un “compte d’appels” qui est incrémenté à chaque appel de la méthode.
  • La JVM interprète une méthode jusqu’à ce que son nombre d’appels dépasse un JIT seuil de compilation.
  • Par conséquent, les méthodes souvent utilisées sont compilées peu de temps après que la JVM a commencé, et les méthodes moins utilisées sont compilées beaucoup plus tard, ou pas à tout.
  • Le seuil de compilation JIT aide la JVM à démarrer rapidement et toujours ont des performances améliorées.
  • Le seuil a été soigneusement sélectionné pour obtenir une équilibre entre les temps de démarrage et les performances à long terme.
  • Après la compilation d’une méthode, son nombre d’appels est remis à zéro et les appels ultérieurs à la méthode continuent d’incrémenter son compte.
  • Lorsque le nombre d’appels d’une méthode atteint une recompilation JIT seuil, le compilateur JIT le compile une deuxième fois, en appliquant un plus grand choix d’optimisations que sur la compilation précédente.
  • Ce processus est répété jusqu’à ce que le niveau d’optimisation maximal soit atteint.

Les méthodes les plus sollicitées d’un programme Java sont toujours optimisées de la manière la plus agressive, ce qui maximise les avantages en termes de performances de l’utilisation du compilateur JIT.

Le compilateur JIT peut également mesurer les “données opérationnelles au moment de l’exécution” et utiliser ces données pour améliorer la qualité des recompilations ultérieures.

Le compilateur JIT peut être désactivé, auquel cas tout le programme Java sera interprété. La désactivation du compilateur JIT n’est pas recommandée, sauf pour diagnostiquer ou contourner les problèmes de compilation JIT.
