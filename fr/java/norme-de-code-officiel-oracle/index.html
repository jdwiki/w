[Guide de style officiel d’Oracle] [1] pour le langage de programmation Java est une norme suivie par les développeurs d’Oracle et recommandée pour tout autre développeur Java. Il couvre les noms de fichiers, l’organisation des fichiers, l’indentation, les commentaires, les déclarations, les déclarations, les espaces blancs, les conventions de dénomination, les pratiques de programmation et comprend un exemple de code.

[1] : http://cr.openjdk.java.net/%7Ealundblad/styleguide

  • Les exemples ci-dessus suivent strictement le nouveau guide de style officiel d’Oracle. En d’autres termes, ils ne sont pas subjectivement constitués par les auteurs de cette page.

  • Le guide de style officiel a été soigneusement rédigé pour être rétrocompatible avec le guide de style original et la majorité du code à l’état sauvage.

  • Le guide de style officiel a été peer [révisé](http://mail.openjdk.java.net /pipermail/discuss/2015-August/003771.html) par, entre autres, Brian Goetz (Java Language Architect) et Mark Reinhold (Chief Architect of the Java Platform).

  • Les exemples sont non normatifs ; Bien qu’ils aient l’intention d’illustrer la manière correcte de formater le code, il peut y avoir d’autres façons de formater correctement le code. C’est un principe général : il peut y avoir plusieurs façons de formater le code, toutes en respectant les directives officielles.

Conventions de nommage #

Noms des packages #

  • Les noms de packages doivent être en minuscules, sans traits de soulignement ni autres caractères spéciaux.
  • Les noms de packages commencent par la partie autorité inversée de l’adresse Web de la société du développeur. Cette partie peut être suivie d’une sous-structure de package dépendant de la structure du projet/programme.
  • N’utilisez pas le pluriel. Suivez la convention de l’API standard qui utilise par exemple java.lang.annotation et non java.lang.annotations.
  • Exemples : com.votreentreprise.widget.bouton, com.votreentreprise.core.api

Noms de classe, d’interface et d’énumération #

  • Les noms de classe et d’énumération doivent généralement être des noms.
  • Les noms d’interface doivent généralement être des noms ou des adjectifs se terminant par …able.
  • Utilisez une casse mixte avec la première lettre de chaque mot en majuscule (c’est-à-dire [CamelCase][1]).
  • Faites correspondre l’expression régulière ^[A-Z][a-zA-Z0-9]*$.
  • Utilisez des mots entiers et évitez d’utiliser des abréviations à moins que l’abréviation soit plus largement utilisée que la forme longue.
  • Formater une abréviation en tant que mot si elle fait partie d’un nom de classe plus long.
  • Exemples : ArrayList, BigInteger, ArrayIndexOutOfBoundsException, Iterable.

Noms de méthode #

Les noms de méthode doivent généralement être des verbes ou d’autres descriptions d’actions

  • Ils doivent correspondre à l’expression régulière ^[a-z][a-zA-Z0-9]*$.
  • Utilisez une casse mixte avec la première lettre en minuscule.
  • Exemples : toString, hashCode

variables #

Les noms de variables doivent être en casse mixte avec la première lettre en minuscule

  • Correspond à l’expression régulière ^[a-z][a-zA-Z0-9]*$
  • Autre recommandation : Variables
  • Exemples : elements, currentIndex

Variables de type #

Pour les cas simples où il y a peu de variables de type impliquées, utilisez une seule lettre majuscule.

  • Correspond à l’expression régulière ^[A-Z][0-9]?$
  • Si une lettre est plus descriptive qu’une autre (comme K et V pour les clés et les valeurs dans les cartes ou R pour un type de retour de fonction), utilisez-la, sinon utilisez T.
  • Pour les cas complexes où les variables de type lettre unique deviennent confuses, utilisez des noms plus longs écrits en majuscules et utilisez le trait de soulignement (_) pour séparer les mots.
  • Exemples : T, V, SRC_VERTEX

Constantes #

Les constantes (champs static final dont le contenu est immuable, par des règles de langage ou par convention) doivent être nommées avec des lettres majuscules et un trait de soulignement (_) pour séparer les mots.

  • Correspond à l’expression régulière ^[A-Z][A-Z0-9]*(_[A-Z0-9]+)*$
  • Exemples : BUFFER_SIZE, MAX_LEVEL

Autres directives sur la dénomination #

  • Évitez de masquer/masquer les méthodes, les variables et les variables de type dans les portées extérieures.
  • Laissez la verbosité du nom en corrélation avec la taille de la portée. (Par exemple, utilisez des noms descriptifs pour les champs de grandes classes et des noms brefs pour les variables locales de courte durée.)
  • Lorsque vous nommez des membres statiques publics, laissez l’identifiant être descriptif si vous pensez qu’ils seront importés de manière statique.
  • Lectures complémentaires : Naming Section (dans le Java Style Guide officiel)

Source : Java Style Guidelines d’Oracle

[1] : https://en.wikipedia.org/wiki/CamelCase

Structure de classe #

Ordre des membres de la classe #

Les membres du groupe doivent être classés comme suit :

  1. Domaines (dans l’ordre public, protégé et privé)
  2. Constructeurs
  3. Méthodes d’usine
  4. Autres méthodes (par ordre public, protégé et privé)

Il n’est pas nécessaire de classer les champs et les méthodes principalement par leurs modificateurs d’accès ou leur identifiant.

Voici un exemple de cette commande :

class Example {

    private int i;

    Example(int i) {
        this.i = i;
    }

    static Example getExample(int i) {
        return new Example(i);
    }

    @Override
    public String toString() {
        return "An example [" + i + "]";
    }

}

Regroupement des membres de la classe #

  • Les champs connexes doivent être regroupés.
  • Un type imbriqué peut être déclaré juste avant sa première utilisation ; sinon, il doit être déclaré avant les champs.
  • Les constructeurs et les méthodes surchargées doivent être regroupés par fonctionnalité et classés avec une arité croissante. Cela implique que la délégation entre ces constructions s’écoule vers le bas dans le code.
  • Les constructeurs doivent être regroupés sans autres membres entre eux.
  • Les variantes surchargées d’une méthode doivent être regroupées sans autres membres entre elles.

Annotation #

Les annotations de déclaration doivent être placées sur une ligne distincte de la déclaration annotée.

@SuppressWarnings("unchecked")
public T[] toArray(T[] typeHolder) {
    ...
}

Cependant, des annotations peu nombreuses ou courtes annotant une méthode sur une seule ligne peuvent être placées sur la même ligne que la méthode si cela améliore la lisibilité. Par exemple, on peut écrire :

@Nullable String getName() { return name; }

Pour des raisons de cohérence et de lisibilité, soit toutes les annotations doivent être placées sur la même ligne, soit chaque annotation doit être placée sur une ligne distincte.

// Bad.
@Deprecated @SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Even worse.
@Deprecated @SafeVarargs
@CustomAnnotation public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated
@SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated @SafeVarargs @CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

Importer des déclarations #

// First java/javax packages
import java.util.ArrayList;
import javax.tools.JavaCompiler;

// Then third party libraries
import com.fasterxml.jackson.annotation.JsonProperty;

// Then project imports
import com.example.my.package.ClassA;
import com.example.my.package.ClassB;

// Then static imports (in the same order as above)
import static java.util.stream.Collectors.toList;

  • Les déclarations d’importation doivent être triées…

  • … principalement par non statique / statique avec des importations non statiques en premier.

  • …secondairement par origine colis selon l’ordre suivant

    • java packages
    • javax packages
    • external packages (e.g. org.xml)
    • internal packages (e.g. com.sun)

  • …tertiaire par package et identifiant de classe dans l’ordre lexicographique

  • Les instructions d’importation ne doivent pas être renvoyées à la ligne, qu’elles dépassent ou non la longueur maximale recommandée d’une ligne.

  • Aucune importation non utilisée ne doit être présente.

Importations de caractères génériques #

  • Les importations de caractères génériques ne doivent en général pas être utilisées.
  • Lors de l’importation d’un grand nombre de classes étroitement liées (comme la mise en œuvre d’un visiteur sur un arbre avec des dizaines de classes « nœuds » distinctes), une importation de caractères génériques peut être utilisée.
  • Dans tous les cas, pas plus d’une importation générique par fichier ne doit être utilisée.

Parenthèses redondantes #

return flag ? "yes" : "no";

String cmp = (flag1 != flag2) ? "not equal" : "equal";

// Don't do this
return (flag ? "yes" : "no");
  • Des parenthèses de regroupement redondantes (c’est-à-dire des parenthèses qui n’affectent pas l’évaluation) peuvent être utilisées si elles améliorent la lisibilité.
  • Les parenthèses de regroupement redondantes doivent généralement être omises dans les expressions plus courtes impliquant des opérateurs communs, mais incluses dans des expressions plus longues ou des expressions impliquant des opérateurs dont la priorité et l’associativité ne sont pas claires sans parenthèses. Les expressions ternaires avec des conditions non triviales appartiennent à ces dernières.
  • L’expression entière suivant un mot-clé return ne doit pas être entourée de parenthèses.

Croisillons #

class Example {
    void method(boolean error) {
        if (error) {
            Log.error("Error occurred!");
            System.out.println("Error!");
        } else { // Use braces since the other block uses braces.
            System.out.println("No error");
        }
    }
}

  • Les accolades ouvrantes doivent être placées à la fin de la ligne courante plutôt que sur une ligne à part.

  • Il devrait y avoir une nouvelle ligne devant une accolade fermante à moins que le bloc ne soit vide (voir Formes abrégées ci-dessous)

  • Les accolades sont recommandées même lorsque le langage les rend facultatives, comme les corps if et loop sur une seule ligne.

  • Si un bloc s’étend sur plus d’une ligne (y compris les commentaires), il doit avoir des accolades.

  • Si l’un des blocs d’une instruction if / else a des accolades, l’autre bloc doit en avoir aussi.

  • Si le bloc vient en dernier dans un bloc englobant, il doit avoir des accolades.

  • Les mots clés else, catch et while dans les boucles do…while vont sur la même ligne que l’accolade fermante du bloc précédent.

Formes courtes #

enum Response { YES, NO, MAYBE }
public boolean isReference() { return true; }

Les recommandations ci-dessus visent à améliorer l’uniformité (et donc à accroître la familiarité/lisibilité). Dans certains cas, les « formulaires courts » qui s’écartent des directives ci-dessus sont tout aussi lisibles et peuvent être utilisés à la place. Ces cas incluent par exemple de simples déclarations enum et des méthodes triviales et des expressions lambda.

Modificateurs #

class ExampleClass {
    // Access modifiers first (don't do for instance "static public")
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}

interface ExampleInterface {
    // Avoid 'public' and 'abstract' since they are implicit
    void sayHello();
}

  • Les modificateurs doivent aller dans l’ordre suivant

  • Modificateur d’accès (public / privé / protégé)

  • “abstrait”

  • “statique”

  • “finale”

  • “transitoire”

  • “volatil”

  • “par défaut”

  • “synchronisé”

  • “natif”

  • strictfp

  • Les modificateurs ne doivent pas être écrits lorsqu’ils sont implicites. Par exemple, les méthodes d’interface ne doivent pas être déclarées “publiques” ni “abstraites”, et les énumérations et interfaces imbriquées ne doivent pas être déclarées statiques.

  • Les paramètres de méthode et les variables locales ne doivent pas être déclarés “final” à moins que cela améliore la lisibilité ou documente une décision de conception réelle.

  • Les champs doivent être déclarés “final” à moins qu’il n’y ait une raison impérieuse de les rendre mutables.

Indentation #

  • Le niveau d’indentation est de quatre espaces.

  • Seuls les espaces peuvent être utilisés pour l’indentation. Aucun onglet.

  • Les lignes vides ne doivent pas être en retrait. (Ceci est impliqué par la règle d’absence d’espace blanc à la fin.)

  • Les lignes case doivent être indentées de quatre espaces, et les déclarations à l’intérieur de la casse doivent être indentées de quatre espaces supplémentaires.

    switch (var) { case TWO: setChoice(“two”); break; case THREE: setChoice(“three”); break; default: throw new IllegalArgumentException(); }

Reportez-vous à https://www.wikiod.com/fr/java/norme-de-code-officiel-oracle#Déclarations d’emballage pour obtenir des instructions sur la façon d’indenter les lignes de continuation.

Littéraux #

long l = 5432L;
int i = 0x123 + 0xABC;
byte b = 0b1010;
float f1 = 1 / 5432f;
float f2 = 0.123e4f;
double d1 = 1 / 5432d;  // or 1 / 5432.0
double d2 = 0x1.3p2;
  • Les littéraux “longs” doivent utiliser le suffixe “L” majuscule.
  • Les littéraux hexadécimaux doivent utiliser des lettres majuscules A-F.
  • Tous les autres préfixes, infixes et suffixes numériques doivent utiliser des lettres minuscules.

Déclaration de paquet #

package com.example.my.package;

La déclaration de package ne doit pas être renvoyée à la ligne, qu’elle dépasse ou non la longueur maximale recommandée d’une ligne.

Expressions lambda #

Runnable r = () -> System.out.println("Hello World");

Supplier<String> c = () -> "Hello World";

// Collection::contains is a simple unary method and its behavior is
// clear from the context. A method reference is preferred here.
appendFilter(goodStrings::contains);

// A lambda expression is easier to understand than just tempMap::put in this case
trackTemperature((time, temp) -> tempMap.put(time, temp));
  • Les lambdas d’expression sont préférés aux lambdas de bloc à une seule ligne.
  • Les références de méthode doivent généralement être préférées aux expressions lambda.
  • Pour les références de méthode d’instance liées ou les méthodes avec une arité supérieure à un, une expression lambda peut être plus facile à comprendre et donc préférée. Surtout si le comportement de la méthode n’est pas clair à partir du contexte.
  • Les types de paramètres doivent être omis sauf s’ils améliorent la lisibilité.
  • Si une expression lambda s’étend sur plus de quelques lignes, envisagez de créer une méthode.

Fichiers sources Java #

  • Toutes les lignes doivent se terminer par un caractère de saut de ligne (LF, valeur ASCII 10) et non par exemple CR ou CR+LF.

  • Il ne doit pas y avoir d’espace blanc à la fin d’une ligne.

  • Le nom d’un fichier source doit être égal au nom de la classe qu’il contient suivi de l’extension .java, même pour les fichiers qui ne contiennent qu’une classe privée de package. Cela ne s’applique pas aux fichiers qui ne contiennent aucune déclaration de classe, comme package-info.java.

Encadrement des déclarations #

  • Le code source et les commentaires ne doivent généralement pas dépasser 80 caractères par ligne et rarement, voire jamais, dépasser 100 caractères par ligne, y compris l’indentation.

La limite de caractères doit être jugée au cas par cas. Ce qui compte vraiment, c’est la « densité » sémantique et la lisibilité de la ligne. Faire des lignes inutilement longues les rend difficiles à lire; de même, faire des “tentatives héroïques” pour les intégrer dans 80 colonnes peut également les rendre difficiles à lire. La flexibilité décrite ici vise à permettre aux développeurs d’éviter ces extrêmes, et non à maximiser l’utilisation de l’espace du moniteur.

  • Les URL ou les exemples de commandes ne doivent pas être encapsulés.

    // Ok even though it might exceed max line width when indented. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable(targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext(targetContainerType));

    // Wrapping preferable String pretty = Stream.of(args) .map(Argument::prettyPrint) .collectors(joining(", “));

    // Too strict interpretation of max line width. Readability suffers. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable( targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext( targetContainerType);

    // Should be wrapped even though it fits within the character limit String pretty = Stream.of(args).map(Argument::prettyPrint).collectors(joining(”, “));

  • L’habillage à un niveau syntaxique supérieur est préféré à l’habillage à un niveau syntaxique inférieur.

  • Il devrait y avoir au plus 1 déclaration par ligne.

  • Une ligne de continuation doit être indentée de l’une des quatre manières suivantes

  • Variante 1 : Avec 8 espaces supplémentaires par rapport à l’indentation de la ligne précédente.

  • Variante 2 : Avec 8 espaces supplémentaires par rapport à la colonne de départ de l’expression encapsulée.

  • Variante 3 : alignée sur l’expression sœur précédente (tant qu’il est clair qu’il s’agit d’une ligne de continuation)

  • Variante 4 : alignée sur l’appel de méthode précédent dans une expression chaînée.

Déclarations de méthode d’encapsulation #

int someMethod(String aString,
               List<Integer> aList,
               Map<String, String> aMap,
               int anInt,
               long aLong,
               Set<Number> aSet,
               double aDouble) {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt, long aLong,
        double aDouble, long aLong) {
    …
}

int someMethod(String aString,
               List<Map<Integer, StringBuffer>> aListOfMaps,
               Map<String, String> aMap)
        throws IllegalArgumentException {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt)
                throws IllegalArgumentException {
    …
}
  • Les déclarations de méthode peuvent être formatées en listant les arguments verticalement, ou par une nouvelle ligne et +8 espaces supplémentaires
  • Si une clause throws doit être enveloppée, placez le saut de ligne devant la clause throws et assurez-vous qu’elle se démarque de la liste des arguments, soit en indentant +8 par rapport à la déclaration de la fonction, soit +8 par rapport à la ligne précédente .

Envelopper les expressions #

  • Si une ligne approche de la limite maximale de caractères, envisagez toujours de la décomposer en plusieurs instructions/expressions au lieu d’envelopper la ligne.

  • Pause avant les opérateurs.

  • Pause avant le . dans les appels de méthode chaînés.

    popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

    // Don’t! Looks like two arguments popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

Espace blanc #

Espace blanc vertical #

  • Une seule ligne blanche doit être utilisée pour séparer…

  • Déclaration de colis

  • Déclarations de classe

  • Constructeurs

  • Méthodes

  • Initialiseurs statiques

  • Initialiseurs d’instance

  • … et peut être utilisé pour séparer des groupes logiques de

  • importer des déclarations

  • des champs

  • déclarations

  • Plusieurs lignes vierges consécutives ne doivent être utilisées que pour séparer des groupes de membres liés et non comme espacement standard entre les membres.

Espace blanc horizontal #

  • Un seul espace doit être utilisé…

  • Pour séparer les mots clés des parenthèses et accolades ouvrantes ou fermantes voisines

  • Avant et après tous les opérateurs binaires et opérateurs comme les symboles tels que les flèches dans les expressions lambda et les deux-points dans les boucles for améliorées (mais pas avant les deux-points d’une étiquette)

  • Après // cela commence un commentaire.

  • Après des virgules séparant les arguments et des points-virgules séparant les parties d’une boucle for.

  • Après la parenthèse fermante d’un casting.

  • Dans les déclarations de variables, il n’est pas recommandé d’aligner les types et les variables.

Caractères spéciaux #

  • En dehors de LF, le seul caractère d’espace blanc autorisé est l’espace (valeur ASCII 32). Notez que cela implique que les autres caractères d’espacement (dans, par exemple, les littéraux de chaîne et de caractère) doivent être écrits sous forme d’échappement.

  • \', \", \\, \t, \b, \r, \f et \n doivent être préférés à l’octal correspondant (par exemple, \047) ou Unicode (par exemple, \u0027) caractères d’échappement.

  • S’il est nécessaire d’aller à l’encontre des règles ci-dessus pour des raisons de test, le test doit générer l’entrée requise par programmation.

Déclarations de variables #

  • Une variable par déclaration (et au maximum une déclaration par ligne)
  • Les crochets des tableaux doivent être sur le type (String[] args) et non sur la variable (String args[]).
  • Déclarez une variable locale juste avant sa première utilisation et initialisez-la le plus près possible de la déclaration.
