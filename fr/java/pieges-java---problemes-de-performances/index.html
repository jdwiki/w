Cette rubrique décrit un certain nombre de “pièges” (c’est-à-dire les erreurs commises par les programmeurs Java novices) liés aux performances des applications Java.

Cette rubrique décrit certaines pratiques de codage Java “micro” qui sont inefficaces. Dans la plupart des cas, les inefficacités sont relativement faibles, mais cela vaut toujours la peine de les éviter si possible.

Piège - La concaténation de chaînes dans une boucle n’est pas mise à l’échelle #

Considérez le code suivant à titre d’illustration :

public String joinWords(List<String> words) {
    String message = "";
    for (String word : words) {
        message = message + " " + word;
    }
    return message;
}

Malheureusement, ce code est inefficace si la liste de “mots” est longue. La racine du problème est cette déclaration:

message = message + " " + word;

Pour chaque itération de boucle, cette instruction crée une nouvelle chaîne message contenant une copie de tous les caractères de la chaîne message d’origine avec des caractères supplémentaires ajoutés. Cela génère beaucoup de chaînes temporaires et fait beaucoup de copie.

Lorsque nous analysons joinWords, en supposant qu’il y a N mots d’une longueur moyenne de M, nous constatons que O(N) chaînes temporaires sont créées et O(M.N2) caractères seront copiés dans le traiter. La composante N2 est particulièrement troublante.

L’approche recommandée pour ce type de problème1 consiste à utiliser un StringBuilder au lieu d’une concaténation de chaînes comme suit :

public String joinWords2(List<String> words) {
    StringBuilder message = new StringBuilder();
    for (String word : words) {
        message.append(" ").append(word);
    }
    return message.toString();
}

L’analyse de joinWords2 doit tenir compte des frais généraux liés à la “croissance” du tableau de sauvegarde StringBuilder qui contient les caractères du constructeur. Or, il s’avère que le nombre de nouveaux objets créés est O(logN) et que le nombre de caractères copiés est O(M.N) caractères. Ce dernier inclut les caractères copiés dans l’appel final toString().

(Il peut être possible d’affiner cela en créant le StringBuilder avec la capacité correcte pour commencer. Cependant, la complexité globale reste la même.)

En revenant à la méthode joinWords d’origine, il s’avère que l’instruction critique sera optimisée par un compilateur Java typique à quelque chose comme ceci :

  StringBuilder tmp = new StringBuilder();
  tmp.append(message).append(" ").append(word);
  message = tmp.toString();

Cependant, le compilateur Java ne « sortira » pas le StringBuilder de la boucle, comme nous l’avons fait à la main dans le code de joinWords2.

Référence:

  • [“L’opérateur String ‘+’ de Java dans une boucle est-il lent ?”][1]

1 - Dans Java 8 et versions ultérieures, la classe Joiner peut être utilisée pour résoudre ce problème particulier. Cependant, ce n’est pas ce que cet exemple est vraiment censé être.

[1] : http://outoffactserror.blogspot.com/2017/03/is-javas-string-operator-in-loop-slow.html

Piège - Utiliser size() pour tester si une collection est vide est inefficace. #

Le Java Collections Framework fournit deux méthodes associées pour tous les objets Collection :

  • size() renvoie le nombre d’entrées dans une Collection, et
  • La méthode isEmpty() renvoie vrai si (et seulement si) le Collection est vide.

Les deux méthodes peuvent être utilisées pour tester le vide de la collection. Par exemple:

Collection<String> strings = new ArrayList<>();
boolean isEmpty_wrong = strings.size() == 0; // Avoid this
boolean isEmpty = strings.isEmpty();         // Best

Bien que ces approches se ressemblent, certaines implémentations de collection ne stockent pas la taille. Pour une telle collection, l’implémentation de size() doit calculer la taille à chaque fois qu’elle est appelée. Par exemple:

  • Une simple classe de liste chaînée (mais pas java.util.LinkedList) peut avoir besoin de parcourir la liste pour compter les éléments.
  • La classe ConcurrentHashMap doit additionner les entrées de tous les “segments” de la carte.
  • Une implémentation paresseuse d’une collection peut avoir besoin de réaliser toute la collection en mémoire afin de compter les éléments.

En revanche, une méthode isEmpty() n’a besoin de tester que s’il y a au moins un élément dans la collection. Cela n’implique pas de compter les éléments.

Alors que size() == 0 n’est pas toujours moins efficace que isEmpty(), il est inconcevable qu’un isEmpty() correctement implémenté soit moins efficace que size() == 0. Par conséquent, isEmpty() est préféré.

Piège - L’utilisation de ’new’ pour créer des instances de wrapper primitives est inefficace #

Le langage Java vous permet d’utiliser new pour créer des instances Integer, Boolean et ainsi de suite, mais c’est généralement une mauvaise idée. Il est préférable d’utiliser l’autoboxing (Java 5 et versions ultérieures) ou la méthode valueOf.

 Integer i1 = new Integer(1);      // BAD
 Integer i2 = 2;                   // BEST (autoboxing)
 Integer i3 = Integer.valueOf(3);  // OK

La raison pour laquelle l’utilisation explicite de new Integer(int) est une mauvaise idée est qu’il crée un nouvel objet (sauf s’il est optimisé par le compilateur JIT). En revanche, lorsque l’autoboxing ou un appel explicite à valueOf sont utilisés, le runtime Java essaiera de réutiliser un objet Integer à partir d’un cache d’objets préexistants. Chaque fois que le runtime a un “hit” de cache, il évite de créer un objet. Cela permet également d’économiser de la mémoire de tas et de réduire les frais généraux du GC causés par l’attrition des objets.

Remarques:

  1. Dans les implémentations Java récentes, l’autoboxing est implémenté en appelant valueOf, et il existe des caches pour Boolean, Byte, Short, Integer, Long et Character.
  2. Le comportement de mise en cache des types intégraux est mandaté par la spécification du langage Java.

Pitfall - Interner des chaînes pour que vous puissiez utiliser == est une mauvaise idée #

Lorsque certains programmeurs voient ce conseil :

“Le test des chaînes en utilisant == est incorrect (sauf si les chaînes sont internées)”

leur réaction initiale est d’internaliser les chaînes afin qu’ils puissent utiliser ==. (Après tout, == est plus rapide que d’appeler String.equals(...), n’est-ce pas.)

C’est une mauvaise approche, à plusieurs égards :

Fragilité #

Tout d’abord, vous ne pouvez utiliser == en toute sécurité que si vous savez que tous les objets String que vous testez ont été internés. Le JLS garantit que les littéraux String de votre code source auront été internés. Cependant, aucune des API Java SE standard ne garantit le retour de chaînes internes, à l’exception de String.intern(String) lui-même. Si vous manquez une seule source d’objets String qui n’ont pas été internés, votre application ne sera pas fiable. Ce manque de fiabilité se manifestera sous la forme de faux négatifs plutôt que d’exceptions susceptibles de le rendre plus difficile à détecter.

Coûts d’utilisation de ‘intern()’ #

Sous le capot, l’internement fonctionne en maintenant une table de hachage contenant des objets “String” précédemment internés. Une sorte de mécanisme de référence faible est utilisé pour que la table de hachage interne ne devienne pas une fuite de stockage. Bien que la table de hachage soit implémentée en code natif (contrairement à HashMap, HashTable etc.), les appels intern sont encore relativement coûteux en termes de CPU et de mémoire utilisée.

Ce coût doit être comparé à l’économie que nous allons obtenir en utilisant == au lieu de equals. En fait, nous n’allons pas atteindre le seuil de rentabilité à moins que chaque chaîne interne ne soit comparée “quelques” fois avec d’autres chaînes.

(En passant : les quelques situations où le stage vaut la peine ont tendance à être de réduire l’empreinte mémoire d’une application où les mêmes chaînes se reproduisent plusieurs fois, et ces chaînes ont une longue durée de vie.)

L’impact sur le ramassage des ordures #

En plus des coûts directs de CPU et de mémoire décrits ci-dessus, les chaînes internes ont un impact sur les performances du ramasse-miettes.

Pour les versions de Java antérieures à Java 7, les chaînes internes sont conservées dans l’espace “PermGen” qui est rarement collecté. Si PermGen doit être collecté, cela déclenche (généralement) un nettoyage complet de la mémoire. Si l’espace PermGen se remplit complètement, la JVM se bloque, même s’il y avait de l’espace libre dans les espaces de tas normaux.

Dans Java 7, le pool de chaînes a été déplacé de “PermGen” vers le tas normal. Cependant, la table de hachage sera toujours une structure de données à longue durée de vie, ce qui entraînera la longévité de toutes les chaînes internes. (Même si les objets de chaîne internés étaient alloués dans l’espace Eden, ils seraient très probablement promus avant d’être collectés.)

Ainsi dans tous les cas, interner une chaîne va prolonger sa durée de vie par rapport à une chaîne ordinaire. Cela augmentera les frais généraux de récupération de place sur la durée de vie de la JVM.

Le deuxième problème est que la table de hachage doit utiliser un mécanisme de référence faible quelconque pour empêcher l’internalisation de la chaîne dans la mémoire qui fuit. Mais un tel mécanisme demande plus de travail au ramasse-miettes.

Ces frais généraux de collecte des ordures sont difficiles à quantifier, mais il ne fait aucun doute qu’ils existent. Si vous utilisez abondamment intern, ils pourraient être significatifs.

La taille de la table de hachage du pool de chaînes #

Selon [cette source] [1], à partir de Java 6, le pool de chaînes est implémenté sous forme de table de hachage de taille fixe avec des chaînes pour traiter les chaînes qui hachent le même compartiment. Dans les premières versions de Java 6, la table de hachage avait une taille constante (câblée). Un paramètre de réglage (-XX:StringTableSize) a été ajouté en tant que mise à jour de mi-vie à Java 6. Ensuite, dans une mise à jour de mi-vie à Java 7, la taille par défaut du pool est passée de 1009 à 60013 .

L’essentiel est que si vous avez l’intention d’utiliser intern de manière intensive dans votre code, il est * conseillé * de choisir une version de Java où la taille de la table de hachage est réglable et assurez-vous de la régler de manière appropriée. Sinon, les performances de intern sont susceptibles de se dégrader à mesure que le pool s’agrandit.

Le stage comme vecteur potentiel de déni de service #

L’algorithme de hashcode pour les chaînes est bien connu. Si vous internez des chaînes fournies par des utilisateurs ou des applications malveillants, cela pourrait être utilisé dans le cadre d’une attaque par déni de service (DoS). Si l’agent malveillant s’arrange pour que toutes les chaînes qu’il fournit aient le même code de hachage, cela pourrait entraîner une table de hachage déséquilibrée et des performances O(N) pour intern … où N est le nombre de collisions cordes.

(Il existe des moyens plus simples / plus efficaces de lancer une attaque DoS contre un service. Cependant, ce vecteur pourrait être utilisé si le but de l’attaque DoS est de briser la sécurité ou d’échapper aux défenses DoS de première ligne.)

Piège - Problèmes d’efficacité avec les expressions régulières #

La correspondance d’expressions régulières est un outil puissant (en Java et dans d’autres contextes), mais elle présente certains inconvénients. L’une d’elles est que les expressions régulières ont tendance à être plutôt coûteuses.

Les instances Pattern et Matcher doivent être réutilisées #

Considérez l’exemple suivant :

/**
 * Test if all strings in a list consist of English letters and numbers.
 * @param strings the list to be checked
 * @return 'true' if an only if all strings satisfy the criteria
 * @throws NullPointerException if 'strings' is 'null' or a 'null' element.
 */
public boolean allAlphanumeric(List<String> strings) {
    for (String s : strings) {
        if (!s.matches("[A-Za-z0-9]*")) {
            return false;
        }  
    }
    return true;
}

Ce code est correct, mais il est inefficace. Le problème est dans l’appel matches(...). Sous le capot, s.matches("[A-Za-z0-9]*") est équivalent à ceci :

Pattern.matches(s, "[A-Za-z0-9]*")

qui est à son tour équivalent à

Pattern.compile("[A-Za-z0-9]*").matcher(s).matches()

L’appel Pattern.compile("[A-Za-z0-9]*") analyse l’expression régulière, l’analyse et construit un objet Pattern contenant la structure de données qui sera utilisée par le moteur regex. C’est un calcul non trivial. Ensuite, un objet Matcher est créé pour envelopper l’argument s. Enfin, nous appelons match() pour faire la correspondance de modèle réelle.

Le problème est que ce travail est entièrement répété pour chaque itération de boucle. La solution est de restructurer le code comme suit :

private static Pattern ALPHA_NUMERIC = Pattern.compile("[A-Za-z0-9]*");

public boolean allAlphanumeric(List<String> strings) {
    Matcher matcher = ALPHA_NUMERIC.matcher("");
    for (String s : strings) {
        matcher.reset(s);
        if (!matcher.matches()) {
            return false;
        }  
    }
    return true;
}

Notez que le [javadoc][1] pour Pattern indique :

Les instances de cette classe sont immuables et peuvent être utilisées en toute sécurité par plusieurs threads simultanés. Les instances de la classe Matcher ne sont pas sûres pour une telle utilisation.

N’utilisez pas match() alors que vous devriez utiliser find() #

Supposons que vous vouliez tester si une chaîne s contient trois chiffres ou plus à la suite. Vous pouvez l’exprimer de différentes manières, notamment :

  if (s.matches(".*[0-9]{3}.*")) {
      System.out.println("matches");
  }

ou

  if (Pattern.compile("[0-9]{3}").matcher(s).find()) {
      System.out.println("matches");
  }

Le premier est plus concis, mais il est également susceptible d’être moins efficace. À première vue, la première version va essayer de faire correspondre la chaîne entière au motif. De plus, étant donné que “.*” est un motif “gourmand”, le comparateur de motifs est susceptible d’avancer “avec impatience” jusqu’à la fin de la chaîne et de revenir en arrière jusqu’à ce qu’il trouve une correspondance.

En revanche, la deuxième version recherchera de gauche à droite et arrêtera la recherche dès qu’elle trouvera les 3 chiffres d’affilée.

Utiliser des alternatives plus efficaces aux expressions régulières

Les expressions régulières sont un outil puissant, mais elles ne doivent pas être votre seul outil. De nombreuses tâches peuvent être effectuées plus efficacement par d’autres moyens. Par exemple:

 Pattern.compile("ABC").matcher(s).find()

fait la même chose que :

 s.contains("ABC")

sauf que ce dernier est beaucoup plus efficace. (Même si vous pouvez amortir le coût de compilation de l’expression régulière.)

Souvent, la forme non régulière est plus compliquée. Par exemple, le test effectué par l’appel matches() de la méthode précédente allAlplanumeric peut être réécrit comme suit :

 public boolean matches(String s) {
     for (char c : s) {
         if ((c >= 'A' && c <= 'Z') ||
             (c >= 'a' && c <= 'z') ||
             (c >= '0' && c <= '9')) {
              return false;
         }
     }
     return true;
 }

Maintenant, c’est plus de code que d’utiliser un Matcher, mais cela va aussi être beaucoup plus rapide.

Retour en arrière catastrophique #

(C’est potentiellement un problème avec toutes les implémentations d’expressions régulières, mais nous le mentionnerons ici car c’est un piège pour l’utilisation de Pattern.)

Considérez cet exemple (artificiel):

Pattern pat = Pattern.compile("(A+)+B");
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB").matches());
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAC").matches());

Le premier appel println affichera rapidement true. Le second affichera “faux”. Finalement. En effet, si vous expérimentez avec le code ci-dessus, vous verrez qu’à chaque fois que vous ajouterez un ‘A’ avant le ‘C’, le temps doublera.

Ce comportement est un exemple de retour en arrière catastrophique. Le moteur de correspondance de modèle qui implémente la correspondance de regex essaie en vain toutes les manières possibles que le modèle pourrait correspondre.

Regardons ce que (A+)+B signifie réellement. Superficiellement, il semble dire “un ou plusieurs caractères ‘A’ suivis d’une valeur ‘B’”, mais en réalité, il dit un ou plusieurs groupes, dont chacun se compose d’un ou plusieurs caractères ‘A’. Ainsi, par exemple :

  • ‘AB’ ne correspond qu’à un sens : ‘(A)B’
  • ‘AAB’ correspond de deux manières : ‘(AA)B’ ou ‘(A)(A)B`
  • ‘AAAB’ correspond de quatre manières : ‘(AAA)B’ ou ‘(AA)(A)B’ ou ‘(A)(AA)B’ ou ‘(A)(A)(A)B’
  • etc

En d’autres termes, le nombre de correspondances possibles est de 2N où N est le nombre de caractères “A”.

L’exemple ci-dessus est clairement artificiel, mais les modèles qui présentent ce type de caractéristiques de performance (c’est-à-dire O(2^N) ou O(N^K) pour un grand K) apparaissent fréquemment lorsque des expressions régulières irréfléchies sont utilisés. Il existe des remèdes standards :

  • Évitez d’imbriquer des motifs répétitifs dans d’autres motifs répétitifs.
  • Évitez d’utiliser trop de motifs répétitifs.
  • Utilisez la répétition sans retour en arrière, le cas échéant.
  • N’utilisez pas d’expressions régulières pour des tâches d’analyse complexes. (Écrivez un analyseur approprié à la place.)

Enfin, méfiez-vous des situations où un utilisateur ou un client API peut fournir une chaîne regex avec des caractéristiques pathologiques. Cela peut conduire à un “déni de service” accidentel ou délibéré.

  • [“Regex Performance”][3] par Jeff Atwood.

Piège - Les petites lectures/écritures sur des flux sans tampon sont inefficaces #

Considérez le code suivant pour copier un fichier dans un autre :

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new FileInputStream(args[0]);
             OutputStream os = new FileOutputStream(args[1])) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

(Nous avons délibérément omis la vérification normale des arguments, les rapports d’erreurs, etc., car ils ne sont pas pertinents pour point de cet exemple.)

Si vous compilez le code ci-dessus et l’utilisez pour copier un gros fichier, vous remarquerez qu’il est très lent. En fait, il sera au moins deux ordres de grandeur plus lent que les utilitaires de copie de fichiers standard du système d’exploitation.

(Ajoutez ici les mesures de performances réelles !)

La principale raison pour laquelle l’exemple ci-dessus est lent (dans le cas de fichiers volumineux) est qu’il effectue des lectures sur un octet et des écritures sur un octet sur des flux d’octets sans tampon. Le moyen simple d’améliorer les performances consiste à envelopper les flux avec des flux mis en mémoire tampon. Par exemple:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new BufferedInputStream(
                     new FileInputStream(args[0]));
             OutputStream os = new BufferedOutputStream(
                     new FileOutputStream(args[1]))) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

Ces petits changements amélioreront le taux de copie des données d’au moins quelques ordres de grandeur, en fonction de divers facteurs liés à la plate-forme. Les enveloppeurs de flux mis en mémoire tampon entraînent la lecture et l’écriture des données en blocs plus volumineux. Les instances ont toutes deux des tampons implémentés sous forme de tableaux d’octets.

  • Avec is, les données sont lues du fichier dans le tampon quelques kilo-octets à la fois. Lorsque read() est appelé, l’implémentation renverra généralement un octet du tampon. Il ne lira le flux d’entrée sous-jacent que si le tampon a été vidé.

  • Le comportement de os est analogue. Les appels à os.write(int) écrivent des octets simples dans le tampon. Les données ne sont écrites dans le flux de sortie que lorsque le tampon est plein ou lorsque os est vidé ou fermé.

Qu’en est-il des flux basés sur les personnages ? #

Comme vous devez le savoir, Java I/O fournit différentes API pour lire et écrire des données binaires et textuelles.

  • InputStream et OutputStream sont les API de base pour les E/S binaires basées sur le flux
  • Reader et Writer sont les API de base pour les E/S de texte basées sur le flux.

Pour les E/S de texte, BufferedReader et BufferedWriter sont les équivalents de BufferedInputStream et BufferedOutputStream.

Pourquoi les flux mis en mémoire tampon font-ils autant de différence ? #

La véritable raison pour laquelle les flux mis en mémoire tampon améliorent les performances est liée à la manière dont une application communique avec le système d’exploitation :

  • La méthode Java dans une application Java ou les appels de procédure native dans les bibliothèques d’exécution natives de la JVM sont rapides. Ils prennent généralement quelques instructions machine et ont un impact minimal sur les performances.

  • En revanche, les appels d’exécution JVM au système d’exploitation ne sont pas rapides. Ils impliquent quelque chose appelé “appel système”. Le modèle typique d’un appel système est le suivant :

    1. Put the syscall arguments into registers.
    2. Execute a SYSENTER trap instruction.
    3. The trap handler switched to privileged state and changes the virtual memory mappings. Then it dispatches to the code to handle the specific syscall.
    4. The syscall handler checks the arguments, taking care that it isn’t being told to access memory that the user process should not see.
    5. The syscall specific work is performed. In the case of a read syscall, this may involve:
      1. checking that there is data to be read at the file descriptor’s current position
      2. calling the file system handler to fetch the required data from disk (or wherever it is stored) into the buffer cache,
      3. copying data from the buffer cache to the JVM-supplied address
      4. adjusting thstream pointerse file descriptor position
    6. Return from the syscall. This entails changing VM mappings again and switching out of privileged state.

Comme vous pouvez l’imaginer, l’exécution d’un seul appel système peut générer des milliers d’instructions machine. De manière conservatrice, * au moins * deux ordres de grandeur plus long qu’un appel de méthode régulier. (Probablement trois ou plus.)

Compte tenu de cela, la raison pour laquelle les flux mis en mémoire tampon font une grande différence est qu’ils réduisent considérablement le nombre d’appels système. Au lieu de faire un appel système pour chaque appel read(), le flux d’entrée tamponné lit une grande quantité de données dans un tampon selon les besoins. La plupart des appels read() sur le flux mis en mémoire tampon effectuent une simple vérification des limites et renvoient un byte qui a été lu précédemment. Un raisonnement similaire s’applique au cas du flux de sortie, ainsi qu’aux cas du flux de caractères.

(Certaines personnes pensent que les performances d’E/S mises en mémoire tampon proviennent de l’inadéquation entre la taille de la demande de lecture et la taille d’un bloc de disque, la latence de rotation du disque et des choses comme ça. En fait, un système d’exploitation moderne utilise un certain nombre de stratégies pour s’assurer que le l’application généralement n’a pas besoin d’attendre le disque. Ce n’est pas la vraie explication.)

Les flux mis en mémoire tampon sont-ils toujours une victoire ? #

Pas toujours. Les flux tamponnés sont définitivement une victoire si votre application va faire beaucoup de “petites” lectures ou écritures. Cependant, si votre application n’a besoin que d’effectuer des lectures ou des écritures volumineuses vers/à partir d’un grand byte[] ou char[], les flux mis en mémoire tampon ne vous apporteront aucun avantage réel. En effet, il pourrait même y avoir une (minuscule) pénalité de performance.

Est-ce le moyen le plus rapide de copier un fichier en Java ? #

Non ce n’est pas le cas. Lorsque vous utilisez les API basées sur les flux de Java pour copier un fichier, vous encourez le coût d’au moins une copie mémoire à mémoire supplémentaire des données. Il est possible d’éviter cela si vous utilisez les API NIO ByteBuffer et Channel. (Ajouter un lien vers un exemple séparé ici.)

Pitfall - Les frais généraux liés à la création de messages de journal #

Les niveaux de journalisation TRACE et DEBUG sont là pour pouvoir transmettre des détails élevés sur le fonctionnement du code donné au moment de l’exécution. Il est généralement recommandé de définir le niveau de journalisation au-dessus de ceux-ci, mais certaines précautions doivent être prises pour que ces instructions n’affectent pas les performances, même lorsqu’elles semblent “désactivées”.

Considérez cette déclaration de journal :

// Processing a request of some kind, logging the parameters
LOG.debug("Request coming from " + myInetAddress.toString() 
          + " parameters: " + Arrays.toString(veryLongParamArray));

Même lorsque le niveau de journalisation est défini sur INFO, les arguments passés à debug() seront évalués à chaque exécution de la ligne. Cela le rend inutilement consommateur à plusieurs égards :

* Concaténation String : plusieurs instances String seront créées

  • InetAddress peut même effectuer une recherche DNS.
  • le veryLongParamArray peut être très long - créer une chaîne à partir de celui-ci consomme de la mémoire, prend du temps

La solution #

La plupart des infrastructures de journalisation fournissent des moyens de créer des messages de journal à l’aide de chaînes fixes et de références d’objet. Le message de journal sera évalué uniquement si le message est réellement journalisé. Exemple:

// No toString() evaluation, no string concatenation if debug is disabled
LOG.debug("Request coming from {} parameters: {}", myInetAddress, parameters));

Cela fonctionne très bien tant que tous les paramètres peuvent être convertis en chaînes à l’aide de [String.valueOf(Object)][1]. Si le calcul des messages de journalisation est plus complexe, le niveau de journalisation peut être vérifié avant la journalisation :

if (LOG.isDebugEnabled()) {
    // Argument expression evaluated only when DEBUG is enabled
    LOG.debug("Request coming from {}, parameters: {}", myInetAddress,
              Arrays.toString(veryLongParamArray);
}

Ici, LOG.debug() avec le calcul coûteux Arrays.toString(Obect[]) est traité uniquement lorsque DEBUG est réellement activé.

[1] : https://docs.oracle.com/javase/8/docs/api/java/lang/String.html#valueOf-java.lang.Object- [2] : https://docs.oracle.com/javase/8/docs/api/java/util/Arrays.html#toString-java.lang.Object:A-

Piège - L’appel de ’new String(String)’ est inefficace #

Utiliser new String(String) pour dupliquer une chaîne est inefficace et presque toujours inutile.

  • Les objets String sont immuables, il n’est donc pas nécessaire de les copier pour se protéger contre les modifications.
  • Dans certaines anciennes versions de Java, les objets String peuvent partager des tableaux de sauvegarde avec d’autres objets String. Dans ces versions, il est possible de perdre de la mémoire en créant une (petite) sous-chaîne d’une (grande) chaîne et en la conservant. Cependant, à partir de Java 7, les tableaux de sauvegarde String ne sont pas partagés.

En l’absence de tout avantage tangible, appeler new String(String) est tout simplement inutile :

  • Faire la copie prend du temps CPU.
  • La copie utilise plus de mémoire ce qui augmente l’empreinte mémoire de l’application et/ou augmente les frais généraux du GC.
  • Les opérations telles que equals(Object) et hashCode() peuvent être plus lentes si des objets String sont copiés.

Piège - L’appel de System.gc() est inefficace #

C’est (presque toujours) une mauvaise idée d’appeler System.gc().

Le javadoc pour la méthode gc() spécifie ce qui suit :

“L’appel de la méthode gc suggère que la machine virtuelle Java déploie des efforts pour recycler les objets inutilisés afin de rendre la mémoire qu’ils occupent actuellement disponible pour une réutilisation rapide. Lorsque le contrôle revient de l’appel de la méthode, la machine virtuelle Java a fait un meilleur effort pour récupérer de l’espace sur tous les objets mis au rebut.”

Il y a quelques points importants qui peuvent en être tirés :

  1. L’utilisation du mot “suggère” plutôt que (par exemple) “indique” signifie que la JVM est libre d’ignorer la suggestion. Le comportement par défaut de la JVM (versions récentes) consiste à suivre la suggestion, mais cela peut être remplacé en définissant -XX:+DisableExplicitGC lors du lancement de la JVM.

  2. L’expression “faire de son mieux pour récupérer de l’espace sur tous les objets supprimés” implique que l’appel de gc déclenchera un ramasse-miettes “complet”.

Alors, pourquoi appeler System.gc() est-il une mauvaise idée ?

Tout d’abord, l’exécution d’un ramasse-miettes complet coûte cher. Un GC complet implique de visiter et de “marquer” chaque objet qui est encore accessible ; c’est-à-dire tous les objets qui ne sont pas des ordures. Si vous déclenchez cela lorsqu’il n’y a pas beaucoup de déchets à ramasser, alors le GC fait beaucoup de travail pour relativement peu d’avantages.

Deuxièmement, un ramasse-miettes complet est susceptible de perturber les propriétés de “localité” des objets qui ne sont pas collectés. Les objets qui sont alloués par le même thread à peu près au même moment ont tendance à être alloués proches les uns des autres en mémoire. C’est bon. Les objets qui sont alloués en même temps sont susceptibles d’être liés ; c’est-à-dire se référencer mutuellement. Si votre application utilise ces références, il est probable que l’accès à la mémoire sera plus rapide en raison de divers effets de mise en cache de la mémoire et des pages. Malheureusement, un ramasse-miettes complet a tendance à déplacer les objets de sorte que les objets qui étaient autrefois proches sont maintenant plus éloignés.

Troisièmement, l’exécution d’un ramasse-miettes complet est susceptible de mettre votre application en pause jusqu’à ce que la collecte soit terminée. Pendant ce temps, votre application ne répondra pas.

En fait, la meilleure stratégie consiste à laisser la JVM décider quand exécuter le GC et quel type de collection exécuter. Si vous n’intervenez pas, la JVM choisira une heure et un type de collecte qui optimisent le débit ou minimisent les temps de pause du GC.

Au début on disait “… (presque toujours) une mauvaise idée…”. En fait, il existe quelques scénarios où cela pourrait être une bonne idée :

  1. Si vous implémentez un test unitaire pour un code sensible à la récupération de place (par exemple, quelque chose impliquant des finaliseurs ou des références faibles / logicielles / fantômes), il peut être nécessaire d’appeler System.gc().

  2. Dans certaines applications interactives, il peut y avoir des moments particuliers où l’utilisateur ne se souciera pas s’il y a une pause de récupération de place. Un exemple est un jeu où il y a des pauses naturelles dans le «jeu»; par exemple. lors du chargement d’un nouveau niveau.

Piège - La surutilisation des types de wrapper primitifs est inefficace #

Considérez ces deux morceaux de code :

int a = 1000;
int b = a + 1;

et

Integer a = 1000;
Integer b = a + 1;

Question : Quelle version est la plus efficace ?

Réponse : Les deux versions semblent presque identiques, mais la première version est beaucoup plus efficace que la seconde.

La deuxième version utilise une représentation des nombres qui utilise plus d’espace et s’appuie sur l’auto-boxing et l’auto-unboxing dans les coulisses. En fait la seconde version est directement équivalente au code suivant :

Integer a = Integer.valueOf(1000);               // box 1000
Integer b = Integer.valueOf(a.intValue() + 1);   // unbox 1000, add 1, box 1001

En comparant cela à l’autre version qui utilise int, il y a clairement trois appels de méthode supplémentaires lorsque Integer est utilisé. Dans le cas de valueOf, les appels vont chacun créer et initialiser un nouvel objet Integer. Tout ce travail supplémentaire de boxe et de déballage est susceptible de rendre la deuxième version d’un ordre de grandeur plus lente que la première.

En plus de cela, la deuxième version alloue des objets sur le tas dans chaque appel valueOf. Bien que l’utilisation de l’espace soit spécifique à la plate-forme, elle est susceptible d’être de l’ordre de 16 octets pour chaque objet “Integer”. En revanche, la version int n’a besoin d’aucun espace de tas supplémentaire, en supposant que a et b sont des variables locales.

Une autre grande raison pour laquelle les primitives sont plus rapides que leur équivalent en boîte est la façon dont leurs types de tableaux respectifs sont disposés en mémoire.

Si vous prenez int[] et Integer[] comme exemple, dans le cas d’un int[], les values int sont contiguës en mémoire. Mais dans le cas d’un Integer[], ce ne sont pas les valeurs qui sont présentées, mais des références (pointeurs) vers des objets Integer, qui à leur tour contiennent les valeurs int réelles.

En plus d’être un niveau supplémentaire d’indirection, cela peut être un gros réservoir en ce qui concerne la localité du cache lors de l’itération sur les valeurs. Dans le cas d’un int[], le CPU pourrait récupérer toutes les valeurs du tableau dans son cache en une seule fois, car elles sont contiguës en mémoire. Mais dans le cas d’un Integer[], le CPU doit potentiellement effectuer une récupération de mémoire supplémentaire pour chaque élément, car le tableau ne contient que des références aux valeurs réelles.

En bref, l’utilisation de types d’encapsuleurs primitifs est relativement coûteuse en ressources CPU et mémoire. Les utiliser inutilement est inefficace.

Piège - L’itération des clés d’une carte peut être inefficace #

L’exemple de code suivant est plus lent que nécessaire :

Map<String, String> map = new HashMap<>(); 
for (String key : map.keySet()) {
    String value = map.get(key);
    // Do something with key and value
}

En effet, il nécessite une recherche de carte (la méthode get()) pour chaque clé de la carte. Cette recherche peut ne pas être efficace (dans un HashMap, cela implique d’appeler hashCode sur la clé, puis de rechercher le compartiment correct dans les structures de données internes, et parfois même d’appeler equals). Sur une grande carte, cela peut ne pas être une surcharge triviale.

La bonne façon d’éviter cela est d’itérer sur les entrées de la carte, ce qui est détaillé dans le [sujet Collections](https://www.wikiod.com/fr/java/collections#Itération sur les collections 201608260922476000177)
