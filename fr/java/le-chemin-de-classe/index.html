Le chemin de classe répertorie les endroits où l’environnement d’exécution Java doit rechercher des classes et des ressources. Le chemin de classe est également utilisé par le compilateur Java pour rechercher les dépendances précédemment compilées et externes.

Chargement de la classe Java

La JVM (Java Virtual Machine) chargera les classes au fur et à mesure des besoins (c’est ce qu’on appelle le lazy-loading). Les emplacements des classes à utiliser sont spécifiés à trois endroits : -

  1. Ceux requis par la plate-forme Java sont chargés en premier, tels que ceux de la bibliothèque de classes Java et ses dépendances.
  2. Les classes d’extension sont ensuite chargées (c’est-à-dire celles de jre/lib/ext/)
  3. Les classes définies par l’utilisateur via le classpath sont ensuite chargées

Les classes sont chargées à l’aide de classes qui sont des sous-types de java.lang.ClassLoader. Ceci est décrit plus en détail dans cette rubrique : https://www.wikiod.com/fr/java/chargeurs-de-classe

Chemin de classe

Le chemin de classe est un paramètre utilisé par la JVM ou le compilateur qui spécifie les emplacements des classes et des packages définis par l’utilisateur. Cela peut être défini dans la ligne de commande comme avec la plupart de ces exemples ou via une variable d’environnement (CLASSPATH)

Différentes façons de spécifier le classpath #

Il existe trois façons de définir le chemin de classe.

  1. Il peut être défini à l’aide de la variable d’environnement CLASSPATH :

     set CLASSPATH=...         # Windows and csh
 export CLASSPATH=...      # Unix ksh/bash

  2. Il peut être défini sur la ligne de commande comme suit

     java -classpath ...
 javac -classpath ...

Notez que l’option -classpath (ou -cp) a priorité sur la variable d’environnement CLASSPATH.

  1. Le chemin de classe d’un fichier JAR exécutable est spécifié à l’aide de l’élément Class-Path dans MANIFEST.MF :

     Class-Path: jar1-name jar2-name directory-name/jar3-name

Notez que cela ne s’applique que lorsque le fichier JAR est exécuté comme ceci :

    java -jar some.jar ...

Dans ce mode d’exécution, l’option -classpath et la variable d’environnement CLASSPATH seront ignorées, même si le fichier JAR n’a pas d’élément Class-Path.

Si aucun chemin de classe n’est spécifié, le chemin de classe par défaut est le fichier JAR sélectionné lors de l’utilisation de java -jar, ou le répertoire courant dans le cas contraire.

Lié:

  • https://docs.oracle.com/javase/tutorial/deployment/jar/downman.html
  • http://docs.oracle.com/javase/7/docs/technotes/tools/windows/classpath.html

Ajout de tous les fichiers JAR d’un répertoire au chemin de classe #

Si vous souhaitez ajouter tous les fichiers JAR du répertoire au chemin de classe, vous pouvez le faire de manière concise en utilisant la syntaxe générique du chemin de classe ; par exemple:

 someFolder/*

Cela indique à la JVM d’ajouter tous les fichiers JAR et ZIP du répertoire “someFolder” au chemin de classe. Cette syntaxe peut être utilisée dans un argument -cp, une variable d’environnement CLASSPATH ou un attribut Class-Path dans le fichier manifeste d’un fichier JAR exécutable.Voir [Setting the Class Path: Class Path Wild Cards] [1] pour des exemples et des mises en garde.

Remarques:

  1. Les caractères génériques de chemin de classe ont été introduits pour la première fois dans Java 6. Les versions antérieures de Java ne traitent pas “*” comme un caractère générique.
  2. Vous ne pouvez pas mettre d’autres caractères avant ou après le “” ; par exemple. “someFolder/.jar” n’est pas un caractère générique.
  3. Un caractère générique correspond uniquement aux fichiers avec le suffixe “.jar” ou “.JAR”. Les fichiers ZIP sont ignorés, tout comme les fichiers JAR avec des suffixes différents.
  4. Un caractère générique correspond uniquement aux fichiers JAR dans le répertoire lui-même, pas dans ses sous-répertoires.
  5. Lorsqu’un groupe de fichiers JAR correspond à une entrée générique, leur ordre relatif sur le chemin de classe n’est pas spécifié.

[1] : https://docs.oracle.com/javase/8/docs/technotes/tools/windows/classpath.html#A1100762

Charger une ressource à partir du classpath #

Il peut être utile de charger une ressource (image, fichier texte, propriétés, KeyStore, …) qui est empaquetée dans un JAR. Pour cela, nous pouvons utiliser les Class et ClassLoaders.

Supposons que nous ayons la structure de projet suivante :

program.jar
|
\-com
  \-project
    |
    |-file.txt
    \-Test.class

Et nous voulons accéder au contenu de file.txt à partir de la classe Test. Nous pouvons le faire en demandant au classloader :

InputStream is = Test.class.getClassLoader().getResourceAsStream("com/project/file.txt");

En utilisant le chargeur de classe, nous devons spécifier le chemin complet de notre ressource (chaque package).

Ou alternativement, nous pouvons demander directement à l’objet de classe Test

InputStream is = Test.class.getResourceAsStream("file.txt");

En utilisant l’objet de classe, le chemin est relatif à la classe elle-même. Notre Test.class étant dans le package com.project, identique à file.txt, nous n’avons pas besoin de spécifier de chemin du tout.

Nous pouvons cependant utiliser des chemins absolus à partir de l’objet de classe, comme ceci :

 is = Test.class.getResourceAsStream("/com/project/file.txt");

Syntaxe du chemin de classe #

Le chemin de classe est une séquence d’entrées qui sont des noms de chemin de répertoire, des noms de chemin de fichier JAR ou ZIP, ou des spécifications génériques JAR/ZIP.

  • Pour un chemin de classe spécifié sur la ligne de commande (par exemple, -classpath) ou en tant que variable d’environnement, les entrées doivent être séparées par des caractères ; (point-virgule) sous Windows ou des caractères : (deux-points) sur d’autres plates-formes ( Linux, UNIX, MacOSX, etc.).

  • Pour l’élément Class-Path dans le MANIFEST.MF d’un fichier JAR, utilisez un seul espace pour séparer les entrées.

Parfois, il est nécessaire d’intégrer un espace dans une entrée de chemin de classe

  • Lorsque le chemin de classe est spécifié sur la ligne de commande, il s’agit simplement d’utiliser le shell approprié en citant. Par exemple:

     export CLASSPATH="/home/user/My JAR Files/foo.jar:second.jar"

(Les détails peuvent dépendre du shell de commande que vous utilisez.)

  • Lorsque le classpath est spécifié dans un fichier JAR comme un fichier “MANIFEST.MF”, l’encodage URL doit être utilisé.

      Class-Path: /home/user/My%20JAR%20Files/foo.jar second.jar

Classpath dynamique #

Parfois, il ne suffit pas d’ajouter tous les JAR d’un dossier, par exemple lorsque vous avez du code natif et que vous devez sélectionner un sous-ensemble de JAR. Dans ce cas, vous avez besoin de deux méthodes main(). Le premier construit un chargeur de classe puis utilise ce chargeur de classe pour appeler le second main().

Voici un exemple qui sélectionne le bon JAR natif SWT pour votre plate-forme, ajoute tous les JAR de votre application, puis appelle la vraie méthode main() : http://stackoverflow.com/questions/2706222/create-cross-platform- application-java-swt/3204032#3204032

Mappage des noms de classe aux noms de chemin #

La chaîne d’outils Java standard (et les outils tiers conçus pour interagir avec eux) ont des règles spécifiques pour mapper les noms des classes aux noms de chemin des fichiers et autres ressources qui les représentent.

Les mappages sont les suivants

  • Pour les classes du package par défaut, les noms de chemin sont de simples noms de fichiers.
  • Pour les classes d’un package nommé, les composants du nom du package correspondent aux répertoires.
  • Pour les classes imbriquées et internes nommées, le composant de nom de fichier est formé en joignant les noms de classe avec un caractère $.
  • Pour les classes internes anonymes, les nombres sont utilisés à la place des noms.

Ceci est illustré dans le tableau suivant :

Nom de classe Chemin source Chemin d’accès au fichier de classe
UneClasse UneClasse.java UneClasse.classe
com.example.SomeClass com/exemple/UneClasse.java com/example/SomeClass.class
SomeClass.Inner (dans SomeClass.java ) SomeClass$Inner.class
SomeClass sur les classes internes (dans SomeClass.java ) SomeClass$1.class, SomeClass$2.class, etc.

Que signifie le classpath : comment fonctionnent les recherches #

Le but du classpath est d’indiquer à une JVM où trouver des classes et d’autres ressources. La signification du chemin de classe et le processus de recherche sont étroitement liés.

Le chemin de classe est une forme de chemin de recherche qui spécifie une séquence d’emplacements pour rechercher des ressources. Dans un chemin de classe standard, ces emplacements sont soit un répertoire du système de fichiers hôte, soit un fichier JAR, soit un fichier ZIP. Dans chaque cas, l’emplacement est la racine d’un espace de noms qui sera recherché.

La procédure standard pour rechercher une classe sur le classpath est la suivante :

  1. Associez le nom de classe à un nom de chemin de fichier de classe relatif « RP ». Le mappage des noms de classe aux noms de fichiers de classe est décrit ailleurs.

  2. Pour chaque entrée E dans le classpath :

  • Si l’entrée est un répertoire de système de fichiers : - Resolve RP relative to E to give an absolute pathname AP. - Test if AP is a path for an existing file. - If yes, load the class from that file
  • Si l’entrée est un fichier JAR ou ZIP : - Lookup RP in the JAR / ZIP file index. - If the corresponding JAR / ZIP file entry exists, load the class from that entry.

La procédure de recherche d’une ressource sur le chemin de classe varie selon que le chemin de ressource est absolu ou relatif. Pour un chemin de ressource absolu, la procédure est la même que ci-dessus. Pour un chemin de ressource relatif résolu à l’aide de Class.getResource ou Class.getResourceAsStream, le chemin du package de classes est ajouté avant la recherche.

(Notez qu’il s’agit des procédures implémentées par les chargeurs de classe Java standard. Un chargeur de classe personnalisé peut effectuer la recherche différemment.)

Le chemin de classe bootstrap #

Les chargeurs de classe Java normaux recherchent d’abord les classes dans le chemin de classe d’amorçage, avant de vérifier les extensions et le chemin de classe de l’application. Par défaut, le chemin de classe d’amorçage se compose du fichier “rt.jar” et d’autres fichiers JAR importants fournis par l’installation JRE. Ceux-ci fournissent toutes les classes de la bibliothèque de classes Java SE standard, ainsi que diverses classes d’implémentation “internes”.

Dans des circonstances normales, vous n’avez pas besoin de vous en préoccuper. Par défaut, des commandes telles que java, javac etc. utiliseront les versions appropriées des bibliothèques d’exécution.

Très occasionnellement, il est nécessaire de remplacer le comportement normal du runtime Java en utilisant une version alternative d’une classe dans les bibliothèques standard. Par exemple, vous pouvez rencontrer un bogue “show stopper” dans les bibliothèques d’exécution que vous ne pouvez pas contourner par des moyens normaux. Dans une telle situation, il est possible de créer un fichier JAR contenant la classe modifiée puis de l’ajouter au classpath bootstrap qui lance la JVM.

La commande java fournit les options -X suivantes pour modifier le classpath bootstrap :

  • -Xbootclasspath:<path> remplace le classpath de démarrage actuel par le chemin fourni.
  • -Xbootclasspath/a:<chemin> ajoute le chemin fourni au chemin de classe de démarrage actuel.
  • -Xbootclasspath/p:<chemin> ajoute le chemin fourni au chemin de classe de démarrage actuel.

Notez que lorsque vous utilisez les options bootclasspath pour remplacer ou remplacer une classe Java (etc.), vous modifiez techniquement Java. Il peut y avoir des implications de licence si vous distribuez ensuite votre code. (Reportez-vous aux termes et conditions de la licence binaire Java … et consultez un avocat.)
