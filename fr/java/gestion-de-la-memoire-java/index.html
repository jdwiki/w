En Java, les objets sont alloués dans le tas et la mémoire du tas est récupérée par une récupération de place automatique. Un programme d’application ne peut pas supprimer explicitement un objet Java.

Les principes de base du ramasse-miettes Java sont décrits dans l’exemple [Garbage collection][1]. D’autres exemples décrivent la finalisation, comment déclencher manuellement le ramasse-miettes et le problème des fuites de stockage.

[1] : https://www.wikiod.com/fr/java/gestion-de-la-memoire-java#Collecte des ordures

Définition des tailles Heap, PermGen et Stack #

Lorsqu’une machine virtuelle Java démarre, elle doit connaître la taille du tas et la taille par défaut des piles de threads. Ceux-ci peuvent être spécifiés en utilisant les options de ligne de commande sur la commande java. Pour les versions de Java antérieures à Java 8, vous pouvez également spécifier la taille de la région PermGen du tas.

Notez que PermGen a été supprimé dans Java 8, et si vous essayez de définir la taille de PermGen, l’option sera ignorée (avec un message d’avertissement).

Si vous ne spécifiez pas explicitement les tailles de tas et de pile, la JVM utilisera des valeurs par défaut qui sont calculées d’une manière spécifique à la version et à la plate-forme. Il se peut que votre application utilise trop peu ou trop de mémoire. C’est généralement correct pour les piles de threads, mais cela peut être problématique pour un programme qui utilise beaucoup de mémoire.

Définition des tailles Heap, PermGen et Stack par défaut :

Les options JVM suivantes définissent la taille du tas :

  • -Xms<taille> - définit la taille initiale du tas
  • -Xmx<taille> - définit la taille maximale du tas
  • -XX:PermSize<size> - définit la taille initiale de PermGen
  • -XX:MaxPermSize<size> - définit la taille maximale de PermGen
  • -Xss<size> - définit la taille de la pile de threads par défaut

Le paramètre <taille> peut être un nombre d’octets, ou peut avoir un suffixe de k, m ou g. Ces derniers spécifient la taille en kilo-octets, mégaoctets et gigaoctets respectivement.

Exemples:

$ java -Xms512m -Xmx1024m JavaApp
$ java -XX:PermSize=64m -XX:MaxPermSize=128m JavaApp
$ java -Xss512k JavaApp

Rechercher les tailles par défaut :

L’option -XX:+printFlagsFinal peut être utilisée pour imprimer les valeurs de tous les drapeaux avant de démarrer la JVM. Cela peut être utilisé pour imprimer les valeurs par défaut pour les paramètres de taille de tas et de pile comme suit :

  • Pour Linux, Unix, Solaris et Mac OSX

    $ java -XX:+PrintFlagsFinal -version | grep -iE ‘HeapSize|PermSize|ThreadStackSize’

  • Pour les fenêtres:

    java -XX:+PrintFlagsFinal -version | findstr /i “HeapSize PermSize ThreadStackSize”

Le résultat des commandes ci-dessus ressemblera à ce qui suit :

uintx InitialHeapSize                          := 20655360        {product}
uintx MaxHeapSize                              := 331350016       {product}
uintx PermSize                                  = 21757952        {pd product}
uintx MaxPermSize                               = 85983232        {pd product}
 intx ThreadStackSize                           = 1024            {pd product}

Les tailles sont données en octets.

Collecte des ordures #

L’approche C++ - nouveau et supprimer #

Dans un langage comme C++, le programme d’application est responsable de la gestion de la mémoire utilisée par la mémoire allouée dynamiquement. Lorsqu’un objet est créé dans le tas C++ à l’aide de l’opérateur “new”, il doit y avoir une utilisation correspondante de l’opérateur “delete” pour supprimer l’objet :

  • Si le programme oublie de “supprimer” un objet et “l’oublie” simplement, la mémoire associée est perdue pour l’application. Le terme pour cette situation est une fuite de mémoire, et trop de fuites de mémoire, une application est susceptible d’utiliser de plus en plus de mémoire, et éventuellement de planter.

  • D’autre part, si une application tente de “supprimer” deux fois le même objet ou d’utiliser un objet après sa suppression, l’application est susceptible de planter en raison de problèmes de corruption de la mémoire.

Dans un programme C++ compliqué, l’implémentation de la gestion de la mémoire à l’aide de new et delete peut prendre du temps. En effet, la gestion de la mémoire est une source courante de bugs.

L’approche Java - ramasse-miettes #

Java adopte une approche différente. Au lieu d’un opérateur de suppression explicite, Java fournit un mécanisme automatique connu sous le nom de récupération de place pour récupérer la mémoire utilisée par les objets qui ne sont plus nécessaires. Le système d’exécution Java se charge de rechercher les objets à supprimer. Cette tâche est effectuée par un composant appelé garbage collector, ou GC en abrégé.

A tout moment de l’exécution d’un programme Java, on peut diviser l’ensemble de tous les objets existants en deux sous-ensembles distincts1 :

  • Les objets accessibles sont définis par le JLS comme suit :

    A reachable object is any object that can be accessed in any potential continuing computation from any live thread.

    In practice, this means that there is a chain of references starting from an in-scope local variable or a static variable by which some code might be able to reach the object.

  • Les objets inaccessibles sont des objets qui ne peuvent pas être atteints comme ci-dessus.

Tous les objets inaccessibles sont * éligibles * pour la récupération de place. Cela ne signifie pas qu’ils seront récupérés. En réalité:

  • Un objet inaccessible n’est pas récupéré immédiatement lorsqu’il devient inaccessible1.
  • Un objet inaccessible * ne peut jamais * être ramassé.

La spécification du langage Java donne beaucoup de latitude à une implémentation JVM pour décider quand collecter des objets inaccessibles. Cela permet également (en pratique) à une implémentation JVM d’être conservatrice dans la manière dont elle détecte les objets inaccessibles.

La seule chose que le JLS garantit est qu’aucun objet accessible ne sera jamais ramassé.

Que se passe-t-il lorsqu’un objet devient inaccessible #

Tout d’abord, rien ne se passe spécifiquement lorsqu’un objet devient inaccessible. Les choses ne se produisent que lorsque le ramasse-miettes s’exécute et qu’il détecte que l’objet est inaccessible. De plus, il est courant qu’une analyse GC ne détecte pas tous les objets inaccessibles.

Lorsque le CPG détecte un objet inaccessible, les événements suivants peuvent se produire.

  1. S’il existe des objets Reference faisant référence à l’objet, ces références seront effacées avant la suppression de l’objet.

  2. Si l’objet est finalisable, alors il sera finalisé. Cela se produit avant que l’objet ne soit supprimé.

  3. L’objet peut être supprimé et la mémoire qu’il occupe peut être récupérée.

Notez qu’il existe une séquence claire dans laquelle les événements ci-dessus peuvent se produire, mais rien n’oblige le ramasse-miettes à effectuer la suppression finale d’un objet spécifique dans un laps de temps spécifique.

Exemples d’objets accessibles et inaccessibles #

Prenons les exemples de classes suivants :

// A node in simple "open" linked-list.
public class Node {
    private static int counter = 0;

    public int nodeNumber = ++counter;
    public Node next;
}

public class ListTest {
    public static void main(String[] args) {
        test();                    // M1
        System.out.prinln("Done"); // M2
    }
    
    private static void test() {
        Node n1 = new Node();      // T1
        Node n2 = new Node();      // T2
        Node n3 = new Node();      // T3
        n1.next = n2;              // T4
        n2 = null;                 // T5
        n3 = null;                 // T6
    }
}

Examinons ce qui se passe lorsque test() est appelé. Les instructions T1, T2 et T3 créent des objets Node, et les objets sont tous accessibles via les variables n1, n2 et n3 respectivement. L’instruction T4 affecte la référence au 2ème objet ‘Node’ au champ ’next’ du premier. Lorsque cela est fait, le 2ème Node est accessible via deux chemins :

 n2 -> Node2
 n1 -> Node1, Node1.next -> Node2

Dans l’instruction T5, nous affectons null à n2. Cela rompt la première des chaînes d’accessibilité pour Node2, mais la seconde reste intacte, donc Node2 est toujours accessible.

Dans l’instruction T6, nous affectons null à n3. Cela rompt la seule chaîne d’accessibilité pour Node3, ce qui rend Node3 inaccessible. Cependant, Node1 et Node2 sont toujours accessibles via la variable n1.

Enfin, lorsque la méthode test() revient, ses variables locales n1, n2 et n3 sortent de la portée et ne sont donc pas accessibles par quoi que ce soit. Cela rompt les chaînes d’accessibilité restantes pour Node1 et Node2, et tous les objets Node ne sont ni inaccessibles ni * éligibles * pour la récupération de place.

1 - Il s’agit d’une simplification qui ignore la finalisation et les classes “Reference”. 2 – En théorie, une implémentation Java pourrait le faire, mais le coût en termes de performances le rend peu pratique.

Fuites de mémoire en Java #

Dans l’exemple [Garbage collection][1], nous avons laissé entendre que Java résout le problème des fuites de mémoire. Ce n’est pas vrai. Un programme Java peut perdre de la mémoire, bien que les causes des fuites soient assez différentes.

Les objets accessibles peuvent fuir #

Considérez l’implémentation de pile naïve suivante.

public class NaiveStack {
    private Object[] stack = new Object[100];
    private int top = 0;

    public void push(Object obj) {
        if (top >= stack.length) {
            throw new StackException("stack overflow");
        }
        stack[top++] = obj;
    }

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        return stack[--top];
    }

    public boolean isEmpty() {
        return top == 0;
    }
}

Lorsque vous poussez un objet puis le déposez' immédiatement, il y aura toujours une référence à l'objet dans le tableau stack`.

La logique de l’implémentation de la pile signifie que cette référence ne peut pas être renvoyée à un client de l’API. Si un objet a été extrait, nous pouvons prouver qu’il ne peut pas * “être accessible dans un calcul continu potentiel à partir d’un thread actif” *. Le problème est qu’une JVM de génération actuelle ne peut pas le prouver. Les JVM de la génération actuelle ne tiennent pas compte de la logique du programme pour déterminer si les références sont accessibles. (Pour commencer, ce n’est pas pratique.)

Mais en mettant de côté la question de ce que signifie réellement * accessibilité *, nous avons clairement une situation ici où l’implémentation de NaiveStack “s’accroche” à des objets qui devraient être récupérés. C’est une fuite de mémoire.

Dans ce cas, la solution est simple :

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        Object popped = stack[--top];
        stack[top] = null;              // Overwrite popped reference with null.
        return popped;
    }

Les caches peuvent être des fuites de mémoire #

Une stratégie courante pour améliorer les performances du service consiste à mettre en cache les résultats. L’idée est de conserver un enregistrement des requêtes courantes et de leurs résultats dans une structure de données en mémoire appelée cache. Ensuite, chaque fois qu’une demande est faite, vous recherchez la demande dans le cache. Si la recherche réussit, vous renvoyez les résultats enregistrés correspondants.

Cette stratégie peut être très efficace si elle est mise en œuvre correctement. Cependant, s’il est mal implémenté, un cache peut être une fuite de mémoire. Considérez l’exemple suivant :

public class RequestHandler {
    private Map<Task, Result> cache = new HashMap<>();

    public Result doRequest(Task task) {
        Result result = cache.get(task);
        if (result == null) {
            result == doRequestProcessing(task);
            cache.put(task, result);
        }
        return result;
    }
}

Le problème avec ce code est que même si tout appel à doRequest peut ajouter une nouvelle entrée au cache, il n’y a rien pour les supprimer. Si le service reçoit continuellement différentes tâches, le cache finira par consommer toute la mémoire disponible. Il s’agit d’une forme de fuite de mémoire.

Une approche pour résoudre ce problème consiste à utiliser un cache avec une taille maximale et à supprimer les anciennes entrées lorsque le cache dépasse le maximum. (Jeter l’entrée la moins récemment utilisée est une bonne stratégie.) Une autre approche consiste à créer le cache à l’aide de WeakHashMap afin que la JVM puisse expulser les entrées du cache si le tas commence à devenir trop plein.

[1] : https://www.wikiod.com/fr/java/gestion-de-la-memoire-java#Collecte des ordures

Finalisation #

Un objet Java peut déclarer une méthode finalize. Cette méthode est appelée juste avant que Java libère la mémoire pour l’objet. Il ressemblera généralement à ceci :

public class MyClass {
  
    //Methods for the class

    @Override
    protected void finalize() throws Throwable {
        // Cleanup code
    }
}

Cependant, il existe quelques mises en garde importantes sur le comportement de la finalisation Java.

  • Java ne garantit pas quand une méthode finalize() sera appelée.
  • Java ne garantit même pas qu’une méthode finalize() sera appelée pendant la durée de vie de l’application en cours d’exécution.
  • La seule chose qui est garantie est que la méthode sera appelée avant que l’objet ne soit supprimé … si l’objet est supprimé.

Les mises en garde ci-dessus signifient que c’est une mauvaise idée de s’appuyer sur la méthode “finalize” pour effectuer des actions de nettoyage (ou autres) qui doivent être effectuées en temps opportun. Une dépendance excessive à la finalisation peut entraîner des fuites de stockage, des fuites de mémoire et d’autres problèmes.

Bref, il y a très peu de situations où la finalisation est réellement une bonne solution.

Les finaliseurs ne s’exécutent qu’une seule fois #

Normalement, un objet est supprimé après avoir été finalisé. Cependant, cela n’arrive pas tout le temps. Prenons l’exemple suivant1 :

public class CaptainJack {
    public static CaptainJack notDeadYet = null;

    protected void finalize() {
        // Resurrection!
        notDeadYet = this;
    }
}

Lorsqu’une instance de CaptainJack devient inaccessible et que le ramasse-miettes tente de la récupérer, la méthode finalize() assignera une référence à l’instance à la variable notDeadYet. Cela rendra l’instance à nouveau accessible et le ramasse-miettes ne la supprimera pas.

Question : Le capitaine Jack est-il immortel ?

Réponse : Non.

Le hic, c’est que la JVM n’exécutera un finaliseur sur un objet qu’une seule fois dans sa vie. Si vous affectez null à notDeadYet rendant à nouveau inaccessible une instance ressuscitée, le ramasse-miettes n’appellera pas finalize() sur l’objet.

1 - Voir https://en.wikipedia.org/wiki/Jack_Harkness.
