Les types atomiques Java sont de simples types mutables qui fournissent des opérations de base qui sont thread-safe et atomiques sans avoir recours au verrouillage. Ils sont destinés à être utilisés dans les cas où le verrouillage constituerait un goulot d’étranglement de la concurrence, ou lorsqu’il existe un risque de blocage ou de livelock.

Paramètres #

Paramètre Descriptif
ensemble Ensemble volatil du champ
obtenir Lecture volatile du champ
lazySet Il s’agit d’une opération commandée en magasin du champ
compareAndSet Si la valeur est la valeur expeed, envoyez-la à la nouvelle valeur
getAndSet obtenir la valeur actuelle et mettre à jour

Beaucoup portent essentiellement sur des combinaisons de lectures ou d’écritures volatiles et d’opérations [CAS][1]. La meilleure façon de comprendre cela est de regarder directement le code source. Par exemple. [AtomicInteger][2], [Unsafe.getAndSet][3]

Création de types atomiques #

Pour un code multithread simple, l’utilisation de [synchronisation] [1] est acceptable. Cependant, l’utilisation de la synchronisation a un impact sur la vivacité, et à mesure qu’une base de code devient plus complexe, la probabilité augmente que vous vous retrouviez avec [Deadlock][2], [Starvation ou Livelock][3].

Dans les cas de concurrence plus complexe, l’utilisation de variables atomiques est souvent une meilleure alternative, car elle permet d’accéder à une variable individuelle de manière sécurisée pour les threads sans la surcharge liée à l’utilisation de méthodes synchronisées ou de blocs de code.

Création d’un type AtomicInteger :

AtomicInteger aInt = new AtomicInteger() // Create with default value 0

AtomicInteger aInt = new AtomicInteger(1) // Create with initial value 1

De même pour les autres types d’instances.

AtomicIntegerArray aIntArray = new AtomicIntegerArray(10) // Create array of specific length
AtomicIntegerArray aIntArray = new AtomicIntegerArray(new int[] {1, 2, 3}) // Initialize array with another array

De même pour les autres types atomiques.

Il y a une exception notable qu’il n’y a pas de types ‘float’ et ‘double’. Ceux-ci peuvent être simulés en utilisant Float.floatToIntBits(float) et Float.intBitsToFloat(int) pour float ainsi que Double.doubleToLongBits(double) et Double.longBitsToDouble(long) pour double.

Si vous souhaitez utiliser sun.misc.Unsafe, vous pouvez utiliser n’importe quelle variable primitive comme atomique en utilisant l’opération atomique dans sun.misc.Unsafe. Tous les types primitifs doivent être convertis ou encodés en int ou longs pour l’utiliser de cette manière. Pour en savoir plus, consultez : [sun.misc.Unsafe][4].

[1] : https://www.wikiod.com/fr/java/programmation-simultanee-threads#Synchronisation [2] : https://docs.oracle.com/javase/tutorial/essential/concurrency/deadlock.html [3] : https://docs.oracle.com/javase/tutorial/essential/concurrency/starvelive.html [4] : https://www.wikiod.com/fr/java/sunmiscunsafe

Motivation pour les types atomiques #

Le moyen le plus simple d’implémenter des applications multithread consiste à utiliser les primitives de synchronisation et de verrouillage intégrées de Java ; par exemple. le mot-clé synchronized. L’exemple suivant montre comment nous pourrions utiliser synchronized pour accumuler les décomptes.

public class Counters {
    private final int[] counters;

    public Counters(int nosCounters) {
        counters = new int[nosCounters];
    }

    /**
     * Increments the integer at the given index
     */
    public synchronized void count(int number) {
        if (number >= 0 && number < counters.length) {
            counters[number]++;
        }
    }

    /**
     * Obtains the current count of the number at the given index,
     * or if there is no number at that index, returns 0.
     */
    public synchronized int getCount(int number) {
        return (number >= 0 && number < counters.length) ? counters[number] : 0;
    }
}

Cette implémentation fonctionnera correctement. Cependant, si vous avez un grand nombre de threads effectuant de nombreux appels simultanés sur le même objet Counters, la synchronisation risque d’être un goulot d’étranglement. Spécifiquement:

  1. Chaque appel de méthode synchronized commencera avec le thread actuel acquérant le verrou pour l’instance Counters.
  2. Le thread maintiendra le verrou pendant qu’il vérifie la valeur number et met à jour le compteur.
  3. Enfin, il libérera le verrou, permettant à d’autres threads d’accéder.

Si un thread tente d’acquérir le verrou alors qu’un autre le détient, le thread tentant sera bloqué (arrêté) à l’étape 1 jusqu’à ce que le verrou soit libéré. Si plusieurs threads sont en attente, l’un d’entre eux l’obtiendra et les autres continueront d’être bloqués.

Cela peut entraîner quelques problèmes :

  • S’il y a beaucoup de conflit pour le verrou (c’est-à-dire que beaucoup de threads essaient de l’acquérir), alors certains threads peuvent être bloqués pendant longtemps.

  • Lorsqu’un thread est bloqué en attente du verrou, le système d’exploitation essaie généralement de basculer l’exécution vers un autre thread. Cette commutation de contexte entraîne un impact relativement important sur les performances du processeur.

  • Lorsqu’il y a plusieurs threads bloqués sur le même verrou, il n’y a aucune garantie que l’un d’entre eux sera traité “équitablement” (c’est-à-dire que chaque thread est garanti d’être programmé pour s’exécuter). Cela peut conduire à * la famine du fil *.

Comment implémente-t-on les types atomiques ? #

Commençons par réécrire l’exemple ci-dessus en utilisant les compteurs “AtomicInteger” :

public class Counters {
    private final AtomicInteger[] counters;

    public Counters(int nosCounters) {
        counters = new AtomicInteger[nosCounters];
        for (int i = 0; i < nosCounters; i++) {
            counters[i] = new AtomicInteger();
        }
    }

    /**
     * Increments the integer at the given index
     */
    public void count(int number) {
        if (number >= 0 && number < counters.length) {
            counters[number].incrementAndGet();
        }
    }

    /**
     * Obtains the current count of the object at the given index,
     * or if there is no number at that index, returns 0.
     */
    public int getCount(int number) {
        return (number >= 0 && number < counters.length) ? 
                counters[number].get() : 0;
    }
}

Nous avons remplacé int[] par un AtomicInteger[] et l’avons initialisé avec une instance dans chaque élément. Nous avons également ajouté des appels à incrementAndGet() et get() à la place des opérations sur les valeurs int.

Mais le plus important est que nous pouvons supprimer le mot-clé synchronized car le verrouillage n’est plus nécessaire. Cela fonctionne parce que les opérations incrementAndGet() et get() sont atomiques et thread-safe. Dans ce contexte, cela signifie que :

  • Chaque compteur du tableau ne sera observable que dans l’état “avant” d’une opération (comme un “incrément”) ou dans l’état “après”.

  • En supposant que l’opération se produit à l’instant ‘T’, aucun thread ne pourra voir l’état “avant” après l’instant ‘T’.

De plus, bien que deux threads puissent en fait tenter de mettre à jour la même instance “AtomicInteger” en même temps, les implémentations des opérations garantissent qu’un seul incrément se produit à la fois sur l’instance donnée. Cela se fait sans verrouillage, ce qui se traduit souvent par de meilleures performances.

Comment fonctionnent les types atomiques ? #

Les types atomiques reposent généralement sur des instructions matérielles spécialisées dans le jeu d’instructions de la machine cible. Par exemple, les jeux d’instructions basés sur Intel fournissent une instruction CAS ([Comparer et échanger][1]) qui effectuera une séquence spécifique d’opérations de mémoire de manière atomique.

Ces instructions de bas niveau sont utilisées pour implémenter des opérations de niveau supérieur dans les API des classes “AtomicXxx” respectives. Par exemple, (encore une fois, en pseudo-code de type C):

private volatile num;

int increment() {
  while (TRUE) {
    int old = num;
    int new = old + 1;
    if (old == compare_and_swap(&num, old, new)) {
      return new;
    }
  }
}

S’il n’y a pas de conflit sur AtomicXxxx, le test if réussira et la boucle se terminera immédiatement. S’il y a conflit, alors le “if” échouera pour tous les threads sauf un, et ils “tourneront” dans la boucle pendant un petit nombre de cycles de la boucle. En pratique, la rotation est de plusieurs ordres de grandeur plus rapide (sauf à des niveaux de contention * irréalistes *, où la synchronisation fonctionne mieux que les classes atomiques car lorsque l’opération CAS échoue, la nouvelle tentative ne fera qu’ajouter plus de contention) que la suspension du thread et la commutation à un autre.

Incidemment, les instructions CAS sont généralement utilisées par la JVM pour implémenter le verrouillage incontrôlé. Si la JVM peut voir qu’un verrou n’est pas actuellement verrouillé, elle tentera d’utiliser un CAS pour acquérir le verrou. Si le CAS réussit, il n’est pas nécessaire d’effectuer la planification coûteuse des threads, le changement de contexte, etc. Pour plus d’informations sur les techniques utilisées, voir [Biased Locking in HotSpot][2].
