Les expressions en Java sont la construction principale pour effectuer des calculs.

Pour une référence sur les opérateurs qui peuvent être utilisés dans les expressions, voir Opérateurs.

Priorité des opérateurs #

Lorsqu’une expression contient plusieurs opérateurs, elle peut potentiellement être lue de différentes manières. Par exemple, l’expression mathématique 1 + 2 x 3 peut être lue de deux manières :

  1. Additionnez ‘1’ et ‘2’ et multipliez le résultat par ‘3’. Cela donne la réponse ‘9’. Si nous ajoutions des parenthèses, cela ressemblerait à ( 1 + 2 ) x 3.
  2. Ajoutez ‘1’ au résultat de la multiplication de ‘2’ et ‘3’. Cela donne la réponse ‘7’. Si nous ajoutions des parenthèses, cela ressemblerait à 1 + ( 2 x 3 ).

En mathématiques, la convention est de lire l’expression de la deuxième manière. La règle générale est que la multiplication et la division se font avant l’addition et la soustraction. Lorsqu’une notation mathématique plus avancée est utilisée, soit la signification est soit « évidente » (pour un mathématicien qualifié !), soit des parenthèses sont ajoutées pour lever l’ambiguïté. Dans les deux cas, l’efficacité de la notation pour transmettre le sens dépend de l’intelligence et des connaissances partagées des mathématiciens.

Java a les mêmes règles claires sur la façon de lire une expression, basées sur la priorité des opérateurs utilisés.

En général, chaque opérateur se voit attribuer une valeur de priorité ; voir le tableau ci-dessous.

Par exemple:

  1 + 2 * 3

La priorité de + est inférieure à la priorité de *, donc le résultat de l’expression est 7, pas 9.

Ordre d’évaluation des expressions #

Les expressions Java sont évaluées selon les règles suivantes :

  • Les opérandes sont évalués de gauche à droite.
  • Les opérandes d’un opérateur sont évalués avant l’opérateur.
  • Les opérateurs sont évalués en fonction de la priorité des opérateurs
  • Les listes d’arguments sont évaluées de gauche à droite.

Exemple simple #

Dans l’exemple suivant :

int i = method1() + method2();

l’ordre d’évaluation est :

  1. L’opérande gauche de l’opérateur = est évalué à l’adresse de i.
  2. L’opérande gauche de l’opérateur + (method1()) est évalué.
  3. L’opérande droit de l’opérateur + (method2()) est évalué.
  4. L’opération + est évaluée.
  5. L’opération = est évaluée, en attribuant le résultat de l’addition à i.

Notez que si les effets des appels sont observables, vous pourrez observer que l’appel à method1 se produit avant l’appel à method2.

Exemple avec un opérateur qui a un effet secondaire #

Dans l’exemple suivant :

int i = 1;
intArray[i] = ++i + 1;

l’ordre d’évaluation est :

  1. L’opérande gauche de l’opérateur = est évalué. Cela donne l’adresse de intArray[1].
  2. Le pré-incrément est évalué. Cela ajoute ‘1’ à ‘i’ et donne ‘2’.
  3. L’opérande droit du ‘+’ est évalué.
  4. L’opération + est évaluée à : 2 + 1 -> 3.
  5. L’opération = est évaluée, en affectant 3 à intArray[1].

Notez que puisque l’opérande de gauche du = est évalué en premier, il n’est pas influencé par l’effet secondaire de la sous-expression ++i.

Référence:

  • [JLS 15.7 - Ordonnance d’évaluation][1]

[1] : https://docs.oracle.com/javase/specs/jls/se8/html/jls-15.html#jls-15.7

Principes de base des expressions #

Les expressions en Java sont la construction principale pour effectuer des calculs. Voici quelques exemples:

1                 // A simple literal is an expression
1 + 2             // A simple expression that adds two numbers
(i + j) / k       // An expression with multiple operations
(flag) ? c : d    // An expression using the "conditional" operator
(String) s        // A type-cast is an expression
obj.test()        // A method call is an expression
new Object()      // Creation of an object is an expression
new int[]         // Creation of an object is an expression

En général, une expression se compose des formes suivantes :

  • Les noms d’expression qui consistent en :
    • Simple identifiers; e.g. someIdentifier
    • Qualified identifiers; e.g. MyClass.someField
  • Les primaires qui consistent en :
    • Literals; e.g. 1, 1.0, 'X', "hello", false and null
    • Class literal expressions; e.g. MyClass.class
    • this and <TypeName> . this
    • Parenthesized expressions; e.g. ( a + b )
    • Class instance creation expressions; e.g. new MyClass(1, 2, 3)
    • Array instance creation expressions; e.g. new int[3]
    • Field access expressions; e.g. obj.someField or this.someField
    • Array access expressions; e.g. vector[21]
    • Method invocations; e.g. obj.doIt(1, 2, 3)
    • Method references (Java 8 and later); e.g. MyClass::doIt
  • Expressions d’opérateurs unaires ; par exemple. !a ou i++
  • Expressions d’opérateurs binaires ; par exemple. a + b ou obj == null
  • Expressions d’opérateurs ternaires ; par exemple. (obj == nul) ? 1 : obj.getCount()
  • Expressions lambda (Java 8 et versions ultérieures) ; par exemple. obj -> obj.getCount()

Les détails des différentes formes d’expressions peuvent être trouvés dans d’autres rubriques.

  • La rubrique [Opérateurs][1] couvre les expressions d’opérateurs unaires, binaires et ternaires.
  • La rubrique [Expressions lambda][2] couvre les expressions lambda et les expressions de référence de méthode.
  • La rubrique [Classes et objets][3] couvre les expressions de création d’instance de classe.
  • La rubrique [Tableaux][4] couvre les expressions d’accès aux tableaux et les expressions de création d’instance de tableau.
  • La rubrique [Literals][5] couvre les différents types d’expressions littérales.

Le type d’une expression #

Dans la plupart des cas, une expression a un type statique qui peut être déterminé au moment de la compilation en examinant et ses sous-expressions. Celles-ci sont appelées expressions autonomes.

Cependant, (dans Java 8 et versions ultérieures), les types d’expressions suivants peuvent être des expressions poly :

  • Expressions entre parenthèses
  • Expressions de création d’instance de classe
  • Expressions d’invocation de méthode
  • Expressions de référence de méthode
  • Expressions conditionnelles
  • Expressions lambda

Lorsqu’une expression est une expression poly, son type peut être influencé par le type cible de l’expression ; c’est-à-dire à quoi il sert.

La valeur d’une expression #

La valeur d’une expression est une affectation compatible avec son type. L’exception à cela est lorsque la pollution en tas s’est produite ; par exemple. car les avertissements de “conversion non sécurisée” ont été (de manière inappropriée) supprimés ou ignorés.

Déclarations d’expressions #

Contrairement à de nombreux autres langages, Java n’autorise généralement pas l’utilisation d’expressions en tant qu’instructions. Par exemple:

public void compute(int i, int j) {
    i + j;   // ERROR
}

Étant donné que le résultat de l’évaluation d’une expression comme ne peut pas être utilisé et qu’il ne peut affecter l’exécution du programme d’aucune autre manière, les concepteurs Java ont pris pour position qu’un tel usage est soit une erreur, soit une erreur.

Cependant, cela ne s’applique pas à toutes les expressions. Un sous-ensemble d’expressions est (en fait) légal en tant qu’énoncé. L’ensemble comprend :

  • Expression d’affectation, y compris les affectations operation-and-becomes.
  • Expressions d’incrémentation et de décrémentation avant et après.
  • Appels de méthodes (void ou non-void).
  • Expressions de création d’instance de classe.

Expressions constantes #

Une expression constante est une expression qui produit un type primitif ou une chaîne, et dont la valeur peut être évaluée au moment de la compilation en un littéral. L’expression doit être évaluée sans lever d’exception et doit être composée uniquement des éléments suivants :

  • Littéraux primitifs et chaînes.

  • Le type convertit en types primitifs ou String.

  • Les opérateurs unaires suivants : +, -, ~ et !.

  • Les opérateurs binaires suivants : *, /, %, +, -, <<, >>, >>>, <, <= , >, >=, ==, !=, &, ^, |, && et ||.

  • L’opérateur conditionnel ternaire ? :.

  • Expressions constantes entre parenthèses.

  • Des noms simples faisant référence à des variables constantes. (Une variable constante est une variable déclarée comme “finale” où l’expression de l’initialiseur est elle-même une expression constante.)

  • Noms qualifiés de la forme <TypeName> . <Identifiant> faisant référence à des variables constantes.

Notez que la liste ci-dessus exclut ++ et --, les opérateurs d’affectation, class et instanceof, les appels de méthode et les références aux variables ou champs généraux.

Les expressions constantes de type String donnent une String “interne”, et les opérations en virgule flottante dans les expressions constantes sont évaluées avec une sémantique stricte FP.

Utilisations des expressions constantes #

Les expressions constantes peuvent être utilisées (à peu près) partout où une expression normale peut être utilisée. Cependant, ils ont une signification particulière dans les contextes suivants.

Des expressions constantes sont requises pour les étiquettes de cas dans les instructions switch. Par exemple:

switch (someValue) {
case 1 + 1:            // OK
case Math.min(2, 3):   // Error - not a constant expression
    doSomething();
}

Lorsque l’expression du côté droit d’une affectation est une expression constante, l’affectation peut effectuer une conversion de réduction primitive. Ceci est autorisé à condition que la valeur de l’expression constante se situe dans la plage du type sur le côté gauche. (Voir [JLS 5.1.3][1] et [5.2][2]) Par exemple :

byte b1 = 1 + 1;             // OK - primitive narrowing conversion.
byte b2 = 127 + 1;           // Error - out of range
byte b3 = b1 + 1;            // Error - not a constant expession
byte b4 = (byte) (b1 + 1);   // OK

Lorsqu’une expression constante est utilisée comme condition dans un do, while ou for, cela affecte l’analyse de lisibilité. Par exemple:

while (false) {
    doSomething();           // Error - statenent not reachable
}
boolean flag = false;
while (flag) {
    doSomething();           // OK
}

(Notez que cela ne s’applique pas aux instructions if. Le compilateur Java permet au bloc then ou else d’une instruction if d’être inaccessible. C’est l’analogue Java de la compilation conditionnelle en C et C++.)

Enfin, les champs static final d’une classe ou d’une interface avec des initialiseurs d’expressions constantes sont initialisés avec impatience. Ainsi, il est garanti que ces constantes seront observées dans l’état initialisé, même lorsqu’il y a un cycle dans le graphe de dépendance d’initialisation de classe.

Pour plus d’informations, reportez-vous à [JLS 15.28. Expressions constantes][3].
