Les énumérations Java (déclarées à l’aide du mot clé enum) sont une syntaxe abrégée pour des quantités importantes de constantes d’une seule classe.

Syntaxe #

  • [public/protected/private] enum Enum_name { // Déclare une nouvelle énumération.
  • ENUM_CONSTANT_1[, ENUM_CONSTANT_2…] ; // Déclare les constantes enum. Cela doit être la première ligne à l’intérieur de l’énumération et doit être séparé par des virgules, avec un point-virgule à la fin.
  • ENUM_CONSTANT_1(param)[, ENUM_CONSTANT_2(param)…] ; // Déclare des constantes enum avec des paramètres. Les types de paramètres doivent correspondre au constructeur.
  • ENUM_CONSTANT_1 {…}[, ENUM_CONSTANT_2 {…}…] ; // Déclare les constantes enum avec des méthodes surchargées. Cela doit être fait si l’énumération contient des méthodes abstraites ; toutes ces méthodes doivent être mises en œuvre.
  • ENUM_CONSTANT.name() // Renvoie une chaîne avec le nom de la constante enum.
  • ENUM_CONSTANT.ordinal() // Renvoie l’ordinal de cette constante d’énumération, sa position dans sa déclaration d’énumération, où la constante initiale est affectée d’un ordinal de zéro.
  • Enum_name.values() // Renvoie un nouveau tableau (de type Enum_name[]) contenant chaque constante de cette énumération à chaque fois qu’elle est appelée.
  • Enum_name.valueOf(“ENUM_CONSTANT”) // L’inverse de ENUM_CONSTANT.name() – renvoie la constante enum avec le nom donné.
  • Enum.valueOf(Enum_name.class, “ENUM_CONSTANT”) // Un synonyme du précédent : l’inverse de ENUM_CONSTANT.name() – renvoie la constante enum avec le nom donné.

Restrictions #

Les énumérations étendent toujours java.lang.Enum, il est donc impossible pour une énumération d’étendre un classer. Cependant, ils peuvent implémenter de nombreuses interfaces.

Conseils & Astuces #

En raison de leur représentation spécialisée, il existe des cartes et des [ensembles](https://docs .oracle.com/javase/7/docs/api/java/util/EnumSet.html) qui peut être utilisé avec des énumérations comme clés. Ceux-ci fonctionneront souvent plus rapidement que leurs homologues non spécialisés.

Déclarer et utiliser une énumération de base #

[Enum][1] peut être considéré comme du sucre de syntaxe pour une classe scellée qui n’est instanciée qu’un certain nombre de fois connu au moment de la compilation pour définir un ensemble de constantes.

Une simple énumération pour lister les différentes saisons serait déclarée comme suit :

public enum Season {
    WINTER,
    SPRING,
    SUMMER,
    FALL
}

Bien que les constantes enum n’aient pas nécessairement besoin d’être en majuscules, la convention Java veut que les noms de constantes soient entièrement en majuscules, avec des mots séparés par des traits de soulignement.

Vous pouvez déclarer un Enum dans son propre fichier :

/**
 * This enum is declared in the Season.java file.
*/
public enum Season {
    WINTER,
    SPRING,
    SUMMER,
    FALL
}

Mais vous pouvez aussi le déclarer dans une autre classe :

 public class Day {

    private Season season;

    public String getSeason() {
        return season.name();
    }

    public void setSeason(String season) {
        this.season = Season.valueOf(season);
    }

    /**
     * This enum is declared inside the Day.java file and 
     * cannot be accessed outside because it's declared as private.
     */
    private enum Season {
        WINTER,
        SPRING,
        SUMMER,
        FALL
    }

}

Enfin, vous ne pouvez pas déclarer un Enum à l’intérieur d’un corps de méthode ou d’un constructeur :

public class Day {

    /**
     * Constructor
    */
    public Day() {
        // Illegal. Compilation error
        enum Season {
            WINTER,
            SPRING,
            SUMMER,
            FALL
        }
    }

    public void aSimpleMethod() {
        // Legal. You can declare a primitive (or an Object) inside a method. Compile!
        int primitiveInt = 42;

        // Illegal. Compilation error.
        enum Season {
            WINTER,
            SPRING,
            SUMMER,
            FALL
        }

        Season season = Season.SPRING;
    }
    
}

Les constantes enum dupliquées ne sont pas autorisées :

public enum Season {
    WINTER,
    WINTER, //Compile Time Error : Duplicate Constants
    SPRING,
    SUMMER,
    FALL
}

Chaque constante de enum est public, static et final par défaut. Comme chaque constante est “statique”, elles sont accessibles directement à l’aide du nom de l’énumération.

Les constantes Enum peuvent être transmises en tant que paramètres de méthode :

public static void display(Season s) {
    System.out.println(s.name());  // name() is a built-in method that gets the exact name of the enum constant
}

display(Season.WINTER);  // Prints out "WINTER"

Vous pouvez obtenir un tableau des constantes enum en utilisant la méthode values(). Les valeurs sont garanties dans l’ordre de déclaration dans le tableau renvoyé :

Season[] seasons = Season.values();

Remarque : cette méthode alloue un nouveau tableau de valeurs à chaque fois qu’elle est appelée.

Pour itérer sur les constantes enum :

public static void enumIterate() {
    for (Season s : Season.values()) {
        System.out.println(s.name());
    }
}

Vous pouvez utiliser des énumérations dans une instruction switch :

public static void enumSwitchExample(Season s) {
    switch(s) {
        case WINTER:
            System.out.println("It's pretty cold");
            break;
        case SPRING:
            System.out.println("It's warming up");
            break;
        case SUMMER:
            System.out.println("It's pretty hot");
            break;
        case FALL:
            System.out.println("It's cooling down");
            break;
    }
}

Vous pouvez également comparer les constantes enum en utilisant == :

Season.FALL == Season.WINTER    // false
Season.SPRING == Season.SPRING  // true

Une autre façon de comparer les constantes enum consiste à utiliser equals() comme ci-dessous, ce qui est considéré comme une mauvaise pratique car vous pouvez facilement tomber dans les pièges suivants :

Season.FALL.equals(Season.FALL); // true
Season.FALL.equals(Season.WINTER); // false
Season.FALL.equals("FALL"); // false and no compiler error

De plus, bien que l’ensemble d’instances dans enum ne puisse pas être modifié au moment de l’exécution, les instances elles-mêmes ne sont pas intrinsèquement immuables car, comme toute autre classe, une enum peut contenir des champs modifiables, comme illustré ci-dessous.

public enum MutableExample {
    A,
    B;

    private int count = 0;

    public void increment() {
        count++;
    }

    public void print() {
        System.out.println("The count of " + name() + " is " + count);
    }
}

// Usage:
MutableExample.A.print();       // Outputs 0
MutableExample.A.increment();
MutableExample.A.print();       // Outputs 1 -- we've changed a field   
MutableExample.B.print();       // Outputs 0 -- another instance remains unchanged

Cependant, une bonne pratique consiste à rendre les instances “enum” immuables, c’est-à-dire lorsqu’elles n’ont pas de champs supplémentaires ou que tous ces champs sont marqués comme “final” et sont eux-mêmes immuables. Cela garantira que pendant toute la durée de vie de l’application, un enum ne laissera aucune fuite de mémoire et qu’il est sûr d’utiliser ses instances sur tous les threads.

Les énumérations implémentent implicitement Serializable et Comparable car la classe Enum fait :

public abstract class Enum<E extends Enum<E>>
extends Object
implements Comparable<E>, Serializable

[1] : https://docs.oracle.com/javase/tutorial/java/javaOO/enum.html

Énumérations avec les constructeurs #

Un enum ne peut pas avoir de constructeur public ; cependant, les constructeurs privés sont acceptables (les constructeurs pour les énumérations sont [package-private][1] par défaut) :

public enum Coin {
    PENNY(1), NICKEL(5), DIME(10), QUARTER(25); // usual names for US coins
    // note that the above parentheses and the constructor arguments match
    private int value;

    Coin(int value) { 
        this.value = value;
    }

    public int getValue() {
        return value;
    }
}

int p = Coin.NICKEL.getValue(); // the int value will be 5

Il est recommandé de garder tous les champs privés et de fournir des méthodes getter, car il existe un nombre fini d’instances pour une énumération.

Si vous deviez implémenter un Enum en tant que class à la place, cela ressemblerait à ceci :

public class Coin<T extends Coin<T>> implements Comparable<T>, Serializable{
    public static final Coin PENNY = new Coin(1);
    public static final Coin NICKEL = new Coin(5);
    public static final Coin DIME = new Coin(10);
    public static final Coin QUARTER = new Coin(25);

    private int value;

    private Coin(int value){
        this.value = value;
    }

    public int getValue() {
        return value;
    }
}

int p = Coin.NICKEL.getValue(); // the int value will be 5

Les constantes d’énumération sont techniquement modifiables, donc un setter pourrait être ajouté pour changer la structure interne d’une constante d’énumération. Cependant, cela est considéré comme une très mauvaise pratique et doit être évité.

La meilleure pratique consiste à rendre les champs Enum immuables, avec final :

public enum Coin {
    PENNY(1), NICKEL(5), DIME(10), QUARTER(25);

    private final int value;

    Coin(int value){ 
        this.value = value;
    }

    ...

}

Vous pouvez définir plusieurs constructeurs dans la même énumération. Lorsque vous le faites, les arguments que vous transmettez dans votre déclaration enum décident quel constructeur est appelé :

public enum Coin {
    PENNY(1, true), NICKEL(5, false), DIME(10), QUARTER(25);

    private final int value;
    private final boolean isCopperColored;

    Coin(int value){
        this(value, false);
    }

    Coin(int value, boolean isCopperColored){ 
        this.value = value;
        this.isCopperColored = isCopperColored;
    }

    ...

}

Remarque : Tous les champs d’énumération non primitifs doivent implémenter [Serializable][2], car la classe [Enum][3] le fait.

[1] : https://www.wikiod.com/fr/java/visibilite-controler-lacces-aux-membres-dune-classe#Visibilité du paquet [2] : https://docs.oracle.com/javase/7/docs/api/java/io/Serializable.html [3] : https://docs.oracle.com/javase/7/docs/api/java/lang/Enum.html

Énumérations avec méthodes abstraites #

Les énumérations peuvent définir des méthodes abstraites, que chaque membre enum doit implémenter.

enum Action {
    DODGE {
        public boolean execute(Player player) {
            return player.isAttacking();
        }
    },
    ATTACK {
        public boolean execute(Player player) {
            return player.hasWeapon();
        }
    },
    JUMP {
        public boolean execute(Player player) {
            return player.getCoordinates().equals(new Coordinates(0, 0));
        }
    };

    public abstract boolean execute(Player player);
}

Cela permet à chaque membre enum de définir son propre comportement pour une opération donnée, sans avoir à activer les types dans une méthode dans la définition de niveau supérieur.

Notez que ce modèle est une forme abrégée de ce qui est généralement réalisé en utilisant le polymorphisme et/ou l’implémentation d’interfaces.

Interface d’implémentation #

Il s’agit d’un enum qui est également une fonction appelable qui teste les entrées String par rapport aux modèles d’expression régulière précompilés.

import java.util.function.Predicate;
import java.util.regex.Pattern;

enum RegEx implements Predicate<String> {
    UPPER("[A-Z]+"), LOWER("[a-z]+"), NUMERIC("[+-]?[0-9]+");

    private final Pattern pattern;

    private RegEx(final String pattern) {
        this.pattern = Pattern.compile(pattern);
    }

    @Override 
    public boolean test(final String input) {
        return this.pattern.matcher(input).matches();
    }
}

public class Main {
    public static void main(String[] args) {
        System.out.println(RegEx.UPPER.test("ABC"));
        System.out.println(RegEx.LOWER.test("abc"));
        System.out.println(RegEx.NUMERIC.test("+111"));
    }
}

Chaque membre de l’énumération peut également implémenter la méthode :

import java.util.function.Predicate;

enum Acceptor implements Predicate<String> {
    NULL {
        @Override
        public boolean test(String s) { return s == null; }
    },
    EMPTY {
        @Override
        public boolean test(String s) { return s.equals(""); }
    },
    NULL_OR_EMPTY {
        @Override
        public boolean test(String s) { return NULL.test(s) || EMPTY.test(s); }
    };
}

public class Main {
    public static void main(String[] args) {
        System.out.println(Acceptor.NULL.test(null));  // true
        System.out.println(Acceptor.EMPTY.test(""));   // true
        System.out.println(Acceptor.NULL_OR_EMPTY.test(" ")); // false
    }
}

Implémenter le modèle Singleton avec une énumération à un seul élément #

Les constantes Enum sont instanciées lorsqu’une énumération est référencée pour la première fois. Par conséquent, cela permet d’implémenter le modèle de conception logicielle [Singleton] [1] avec une énumération à un seul élément.

public enum Attendant {

    INSTANCE;

    private Attendant() {
        // perform some initialization routine
    }

    public void sayHello() {
        System.out.println("Hello!");
    }
}


public class Main {

    public static void main(String... args) {
        Attendant.INSTANCE.sayHello();// instantiated at this point
    }
}

Selon le livre “Effective Java” de Joshua Bloch, une énumération à un seul élément est le meilleur moyen d’implémenter un singleton. Cette approche présente les avantages suivants :

  • sécurité des fils
  • garantie d’instanciation unique
  • sérialisation prête à l’emploi

Et comme indiqué dans la section [implémente l’interface][2], ce singleton peut également implémenter une ou plusieurs interfaces.

[1] : https://www.wikiod.com/fr/java/celibataires [2] : https://www.wikiod.com/fr/java/enumerations#Interface outils “implémente l’interface”

Utiliser des méthodes et des blocs statiques #

Une énumération peut contenir une méthode, comme n’importe quelle classe. Pour voir comment cela fonctionne, nous allons déclarer une énumération comme celle-ci :

public enum Direction {
    NORTH, SOUTH, EAST, WEST;
}

Prenons une méthode qui renvoie l’énumération dans la direction opposée :

public enum Direction {
    NORTH, SOUTH, EAST, WEST;

    public Direction getOpposite(){
        switch (this){
            case NORTH:
                return SOUTH;               
            case SOUTH:
                return NORTH;                
            case WEST:
                return EAST; 
            case EAST:
                return WEST;  
            default: //This will never happen
                return null;
        }
    }
}

Cela peut être encore amélioré grâce à l’utilisation de champs et de blocs d’initialisation statique :

public enum Direction {
    NORTH, SOUTH, EAST, WEST;
    
    private Direction opposite;
    
    public Direction getOpposite(){
        return opposite;
    }
    
    static {
        NORTH.opposite = SOUTH;
        SOUTH.opposite = NORTH;
        WEST.opposite = EAST;
        EAST.opposite = WEST;
    }
}

Dans cet exemple, la direction opposée est stockée dans un champ d’instance privé “opposite”, qui est initialisé statiquement la première fois qu’une “Direction” est utilisée. Dans ce cas particulier (parce que NORTH fait référence à SOUTH et inversement), nous ne pouvons pas utiliser [Enums with constructors][1] ici (Constructors NORTH(SOUTH), SOUTH(NORTH), EAST(WEST), WEST(EAST ) serait plus élégant et permettrait de déclarer opposite final, mais serait auto-référentiel et n’est donc pas autorisé).

[1] : https://www.wikiod.com/fr/java/enumerations#Énumérations avec les constructeurs

Enum en tant que paramètre de type borné #

Lors de l’écriture d’une classe avec des génériques en java, il est possible de s’assurer que le paramètre type est une énumération. Étant donné que toutes les énumérations étendent la classe Enum, la syntaxe suivante peut être utilisée.

public class Holder<T extends Enum<T>> {
    public final T value;

    public Holder(T init) {
        this.value = init;
    }
}

Dans cet exemple, le type T doit être une énumération.

Corps spécifique de la constante d’énumération #

Dans un enum, il est possible de définir un comportement spécifique pour une constante particulière de enum qui remplace le comportement par défaut de enum, cette technique est connue sous le nom de corps spécifique constant.

Supposons que trois étudiants en piano - John, Ben et Luke - soient définis dans une “énumération” nommée “PianoClass”, comme suit :

    enum PianoClass {
    JOHN, BEN, LUKE;
        public String getSex() {
            return "Male";
        }
        public String getLevel() {
            return "Beginner";
        }
    }

Et un jour arrivent deux autres élèves - Rita et Tom - avec un sexe (Féminin) et un niveau (Intermédiaire) qui ne correspondent pas aux précédents :

    enum PianoClass2 {
    JOHN, BEN, LUKE, RITA, TOM;
        public String getSex() {
            return "Male"; // issue, Rita is a female
        }
        public String getLevel() {
            return "Beginner"; // issue, Tom is an intermediate student
        }
    }

de sorte que le simple fait d’ajouter les nouveaux étudiants à la déclaration constante, comme suit, n’est pas correct :

PianoClass2 tom = PianoClass2.TOM;
PianoClass2 rita = PianoClass2.RITA;
System.out.println(tom.getLevel()); // prints Beginner -> wrong Tom's not a beginner
System.out.println(rita.getSex()); // prints Male -> wrong Rita's not a male

Il est possible de définir un comportement spécifique pour chacune des constantes, Rita et Tom, qui remplace le comportement par défaut de PianoClass2 comme suit :

enum PianoClass3 {
    JOHN, BEN, LUKE,
    RITA {
        @Override
        public String getSex() {
            return "Female";
        }
    },
    TOM {
        @Override
        public String getLevel() {
            return "Intermediate";
        }
    };
    public String getSex() {
        return "Male";
    }
    public String getLevel() {
        return "Beginner";
    }
}

et maintenant le niveau de Tom et le sexe de Rita sont comme il se doit :

PianoClass3 tom = PianoClass3.TOM;
PianoClass3 rita = PianoClass3.RITA;
System.out.println(tom.getLevel()); // prints Intermediate
System.out.println(rita.getSex()); // prints Female

Une autre façon de définir un corps spécifique au contenu consiste à utiliser un constructeur, par exemple :

enum Friend {
    MAT("Male"),
    JOHN("Male"),
    JANE("Female");
    
    private String gender;

    Friend(String gender) {
        this.gender = gender;
    }

    public String getGender() {
        return this.gender;
    }
}

et utilisation :

Friend mat = Friend.MAT;
Friend john = Friend.JOHN;
Friend jane = Friend.JANE;
System.out.println(mat.getGender());     // Male
System.out.println(john.getGender());    // Male
System.out.println(jane.getGender());    // Female

Énumération d’instance nulle #

enum Util {
    /* No instances */;

    public static int clamp(int min, int max, int i) {
        return Math.min(Math.max(i, min), max);
    }

    // other utility methods...
}

Tout comme enum [peut être utilisé pour les singletons](https://www.wikiod.com/fr/java/enumerations#Implémenter le modèle Singleton avec une énumération à un seul élément) (1 instance classes), il peut être utilisé pour les classes utilitaires (0 classes d’instance). Assurez-vous simplement de terminer la liste (vide) des constantes enum avec un ;.

Voir la question http://stackoverflow.com/questions/25137490/zero-instance-enum-vs-private-constructors-for-preventing-instantiation pour une discussion sur les avantages et les inconvénients par rapport aux constructeurs privés.

Obtenir les valeurs d’une énumération #

Chaque classe enum contient une méthode statique implicite nommée values(). Cette méthode renvoie un tableau contenant toutes les valeurs de cette énumération. Vous pouvez utiliser cette méthode pour parcourir les valeurs. Il est important de noter cependant que cette méthode renvoie un nouveau tableau à chaque fois qu’elle est appelée.

public enum Day {
    MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY;
    
    /**
    * Print out all the values in this enum.
    */
    public static void printAllDays() {
        for(Day day : Day.values()) {
            System.out.println(day.name());
        }
    }
}

Si vous avez besoin d’un Set, vous pouvez également utiliser EnumSet.allOf(Day.class).

Modèle de polymorphisme d’énumération #

Lorsqu’une méthode doit accepter un ensemble “extensible” de valeurs “enum”, le programmeur peut appliquer le polymorphisme comme sur une “classe” normale en créant une interface qui sera utilisée partout où les “enum” doivent être utilisés :

public interface ExtensibleEnum {
    String name();
}

De cette façon, tout enum étiqueté par (implémentant) l’interface peut être utilisé comme paramètre, permettant au programmeur de créer une quantité variable de enums qui sera acceptée par la méthode. Cela peut être utile, par exemple, dans les API où il y a un enum par défaut (non modifiable) et l’utilisateur de ces API veut “étendre” le enum avec plus de valeurs.

Un ensemble de valeurs d’énumération par défaut peut être défini comme suit :

public enum DefaultValues implements ExtensibleEnum {
    VALUE_ONE, VALUE_TWO;
}

Des valeurs supplémentaires peuvent alors être définies comme ceci :

public enum ExtendedValues implements ExtensibleEnum {
    VALUE_THREE, VALUE_FOUR;
}

Exemple qui montre comment utiliser les énumérations - notez comment printEnum() accepte les valeurs des deux types enum :

private void printEnum(ExtensibleEnum val) {
    System.out.println(val.name());
}  

printEnum(DefaultValues.VALUE_ONE);    // VALUE_ONE
printEnum(DefaultValues.VALUE_TWO);    // VALUE_TWO
printEnum(ExtendedValues.VALUE_THREE); // VALUE_THREE
printEnum(ExtendedValues.VALUE_FOUR);  // VALUE_FOUR

Remarque : Ce modèle ne vous empêche pas de redéfinir les valeurs d’énumération, qui sont déjà définies dans une énumération, dans une autre énumération. Ces valeurs d’énumération seraient alors des instances différentes. De plus, il n’est pas possible d’utiliser switch-on-enum puisque tout ce que nous avons est l’interface, pas le vrai “enum”.

Compare et contient pour les valeurs Enum #

Les énumérations ne contiennent que des constantes et peuvent être comparées directement avec ==. Ainsi, seule la vérification des références est nécessaire, pas besoin d’utiliser la méthode .equals. De plus, si .equals est utilisé de manière incorrecte, peut déclencher l’exception NullPointerException alors que ce n’est pas le cas avec la vérification ==.

enum Day {
    GOOD, AVERAGE, WORST;
}

public class Test {

    public static void main(String[] args) {
        Day day = null;

        if (day.equals(Day.GOOD)) {//NullPointerException!
            System.out.println("Good Day!");
        }

        if (day == Day.GOOD) {//Always use == to compare enum
            System.out.println("Good Day!");
        }

    }
}

Pour regrouper, compléter, classer les valeurs enum, nous avons la classe [EnumSet][1] qui contient différentes méthodes.

  • EnumSet#range : pour obtenir un sous-ensemble d’énumération par plage définie par deux points de terminaison

  • EnumSet#of : ensemble d’énumérations spécifiques sans plage. Il existe plusieurs méthodes of surchargées.

  • EnumSet#complementOf : ensemble d’énumérations qui est le complément des valeurs d’énumération fournies dans le paramètre de méthode

    enum Page {
   A1, A2, A3, A4, A5, A6, A7, A8, A9, A10
}

public class Test {

  public static void main(String[] args) {
      EnumSet<Page> range = EnumSet.range(Page.A1, Page.A5);

      if (range.contains(Page.A4)) {
          System.out.println("Range contains A4");
      }

      EnumSet<Page> of = EnumSet.of(Page.A1, Page.A5, Page.A3);

      if (of.contains(Page.A1)) {
          System.out.println("Of contains A1");
      }
  }
}

[1] : https://docs.oracle.com/javase/8/docs/api/java/util/EnumSet.html

Récupère la constante enum par son nom #

Supposons que nous ayons une énumération DayOfWeek :

enum DayOfWeek {
    SUNDAY, MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY;
}

Une énumération est compilée avec une méthode statique intégrée valueOf() qui peut être utilisée pour rechercher une constante par son nom :

String dayName = DayOfWeek.SUNDAY.name();
assert dayName.equals("SUNDAY");

DayOfWeek day = DayOfWeek.valueOf(dayName);
assert day == DayOfWeek.SUNDAY;

Ceci est également possible en utilisant un type d’énumération dynamique :

Class<DayOfWeek> enumType = DayOfWeek.class;
DayOfWeek day = Enum.valueOf(enumType, "SUNDAY");
assert day == DayOfWeek.SUNDAY;

Ces deux méthodes valueOf() lèveront une IllegalArgumentException si l’énumération spécifiée n’a pas de constante avec un nom correspondant.

La bibliothèque Guava fournit une méthode d’assistance [Enums.getIfPresent()][1] qui renvoie un Guava [Optional][2] pour éliminer la gestion explicite des exceptions :

DayOfWeek defaultDay = DayOfWeek.SUNDAY;
DayOfWeek day = Enums.valueOf(DayOfWeek.class, "INVALID").or(defaultDay);
assert day == DayOfWeek.SUNDAY;

[1] : https://google.github.io/guava/releases/18.0/api/docs/com/google/common/base/Enums.html#getIfPresent(java.lang.Class,%20java.lang.String ) [2] : https://google.github.io/guava/releases/18.0/api/docs/com/google/common/base/Optional.html

Énumération avec propriétés (champs) #

Dans le cas où nous voulons utiliser enum avec plus d’informations et pas seulement comme des valeurs constantes, et nous voulons pouvoir comparer deux enums.

Considérez l’exemple suivant :

public enum Coin {
    PENNY(1), NICKEL(5), DIME(10), QUARTER(25);

    private final int value;

    Coin(int value){
        this.value = value;
    }

    public boolean isGreaterThan(Coin other){
        return this.value > other.value;
    }

}

Ici, nous avons défini un Enum appelé Coin qui représente sa valeur. Avec la méthode isGreaterThan nous pouvons comparer deux enums :

Coin penny = Coin.PENNY;
Coin dime = Coin.DIME;

System.out.println(penny.isGreaterThan(dime)); // prints: false
System.out.println(dime.isGreaterThan(penny)); // prints: true

Convertir l’énumération en chaîne #

Parfois, vous voulez convertir votre enum en String, il y a deux façons de le faire.

Supposons que nous ayons :

public enum Fruit {
    APPLE, ORANGE, STRAWBERRY, BANANA, LEMON, GRAPE_FRUIT;
}

Alors, comment convertir quelque chose comme Fruit.APPLE en "APPLE" ?


Convertir en utilisant name() #

name() est une méthode interne dans enum qui renvoie la représentation String de l’énumération, le retour String représente exactement comment la valeur de l’énumération a été définie.

Par exemple:

System.out.println(Fruit.BANANA.name());      // "BANANA"
System.out.println(Fruit.GRAPE_FRUIT.name()); // "GRAPE_FRUIT"

Convertir en utilisant toString() #

toString() est, par défaut, remplacé pour avoir le même comportement que name()

Cependant, toString() est probablement remplacé par * les développeurs * pour qu’il imprime une String plus conviviale

N’utilisez pas toString() si vous voulez vérifier votre code, name() est beaucoup plus stable pour cela. N’utilisez que toString () lorsque vous allez sortir la valeur dans les journaux ou stdout ou quelque chose

Par défaut: #

System.out.println(Fruit.BANANA.toString());      // "BANANA"
System.out.println(Fruit.GRAPE_FRUIT.toString()); // "GRAPE_FRUIT"

Exemple de remplacement #

System.out.println(Fruit.BANANA.toString());      // "Banana"
System.out.println(Fruit.GRAPE_FRUIT.toString()); // "Grape Fruit"

Énumérations avec des champs statiques #

Si votre classe enum doit avoir des champs statiques, gardez à l’esprit qu’ils sont créés après les valeurs enum elles-mêmes. Cela signifie que le code suivant entraînera une NullPointerException :

enum Example {
    ONE(1), TWO(2);

    static Map<String, Integer> integers = new HashMap<>();

    private Example(int value) {
        integers.put(this.name(), value);
    }
}

Un moyen possible de résoudre ce problème :

enum Example {
    ONE(1), TWO(2);

    static Map<String, Integer> integers;

    private Example(int value) {
        putValue(this.name(), value);
    }

    private static void putValue(String name, int value) {
        if (integers == null)
            integers = new HashMap<>();
        integers.put(name, value);
    }
}

N’initialisez pas le champ statique :

enum Example {
    ONE(1), TWO(2);

    // after initialisisation integers is null!!
    static Map<String, Integer> integers = null;

    private Example(int value) {
        putValue(this.name(), value);
    }

    private static void putValue(String name, int value) {
        if (integers == null)
            integers = new HashMap<>();
        integers.put(name, value);
    }
    // !!this may lead to null poiner exception!!
    public int getValue(){
        return (Example.integers.get(this.name()));
    }
}

initialisation :

  • créer les valeurs enum
  • comme effet secondaire putValue() appelé qui initialise les entiers
  • les valeurs statiques sont définies
  • entiers = nuls ; // est exécuté après les énumérations donc le contenu des entiers est perdu
