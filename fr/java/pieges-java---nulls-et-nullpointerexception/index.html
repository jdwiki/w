Pièges Java - Nulls et NullPointerException

La valeur null est la valeur par défaut pour une valeur non initialisée d’un champ dont le type est un type référence.

NullPointerException (ou NPE) est l’exception levée lorsque vous tentez d’effectuer une opération inappropriée sur la référence d’objet null. Ces opérations comprennent :

  • appeler une méthode d’instance sur un objet cible null,
  • accéder à un champ d’un objet cible “null”,
  • tenter d’indexer un objet tableau null ou d’accéder à sa longueur,
  • utiliser une référence d’objet “null” comme mutex dans un bloc “synchronisé”,
  • lancer une référence d’objet null,
  • déballage d’une référence d’objet “null”, et
  • jeter une référence d’objet null.

Les causes profondes les plus courantes des NPE :

  • oubli d’initialiser un champ avec un type référence,
  • oublier d’initialiser les éléments d’un tableau de type référence, ou
  • ne teste pas les résultats de certaines méthodes d’API qui sont spécifiées comme renvoyant “null” dans certaines circonstances.

Exemples de méthodes couramment utilisées qui renvoient “null” :

  • La méthode get(key) dans l’API Map renverra un null si vous l’appelez avec une clé qui n’a pas de mappage.
  • Les méthodes getResource(path) et getResourceAsStream(path) dans les API ClassLoader et Class renverront null si la ressource est introuvable.
  • La méthode get() dans l’API Reference renverra null si le ramasse-miettes a effacé la référence.
  • Diverses méthodes getXxxx dans les API de servlet Java EE renverront null si vous essayez de récupérer un paramètre de requête, une session ou un attribut de session inexistant, etc.

Il existe des stratégies pour éviter les NPE indésirables, telles que le test explicite de null ou l’utilisation de “Yoda Notation”, mais ces stratégies ont souvent le résultat indésirable de masquer des problèmes dans votre code qui devraient vraiment être corrigés.

Pitfall – « Réparer » les nulls inattendus #

Sur StackOverflow, nous voyons souvent du code comme celui-ci dans Answers :

public String joinStrings(String a, String b) {
    if (a == null) {
        a = "";
    }
    if (b == null) {
        b = "";
    }
    return a + ": " + b;
}

Souvent, cela est accompagné d’une assertion qui est la “meilleure pratique” pour tester null comme ceci pour éviter NullPointerException.

Est-ce la meilleure pratique ? En bref : Non.

Certaines hypothèses sous-jacentes doivent être remises en question avant de pouvoir dire si c’est une bonne idée de le faire dans notre joinStrings :

Qu’est-ce que cela signifie que “a” ou “b” soit nul ? #

Une valeur String peut être zéro ou plusieurs caractères, nous avons donc déjà un moyen de représenter une chaîne vide. Est-ce que null signifie quelque chose de différent de "" ? Si non, il est problématique d’avoir deux façons de représenter une chaîne vide.

Le null provient-il d’une variable non initialisée ? #

Un null peut provenir d’un champ non initialisé ou d’un élément de tableau non initialisé. La valeur peut être non initialisée par conception ou par accident. Si c’était par accident, c’est un bug.

La valeur nulle représente-t-elle une « ne sait pas » ou une « valeur manquante » ?

Parfois, un « null » peut avoir une signification authentique ; par exemple. que la valeur réelle d’une variable est inconnue ou indisponible ou “optionnelle”. Dans Java 8, la classe Optional fournit une meilleure façon d’exprimer cela.

S’il s’agit d’un bogue (ou d’une erreur de conception), devons-nous “réparer” ?

Une interprétation du code est que nous “réparons” un “null” inattendu en utilisant une chaîne vide à sa place. Est-ce la bonne stratégie ? Serait-il préférable de laisser l’exception NullPointerException se produire, puis d’attraper l’exception plus haut dans la pile et de la consigner en tant que bogue ?

Le problème avec la “réparation” est qu’elle est susceptible soit de masquer le problème, soit de le rendre plus difficile à diagnostiquer.

Est-ce efficace / bon pour la qualité du code ? #

Si l’approche “réparer” est utilisée de manière cohérente, votre code contiendra de nombreux tests nuls “défensifs”. Cela va le rendre plus long et plus difficile à lire. De plus, tous ces tests et “réparations” sont susceptibles d’avoir un impact sur les performances de votre application.

En résumé #

Si null est une valeur significative, alors tester le cas null est la bonne approche. Le corollaire est que si une valeur “null” est significative, cela doit être clairement documenté dans les javadocs de toutes les méthodes qui acceptent la valeur “null” ou la renvoient.

Sinon, il est préférable de traiter un null inattendu comme une erreur de programmation et de laisser l’exception NullPointerException se produire afin que le développeur sache qu’il y a un problème dans le code.

Pitfall - Utiliser null pour représenter un tableau ou une collection vide #

Certains programmeurs pensent que c’est une bonne idée d’économiser de l’espace en utilisant un “null” pour représenter un tableau ou une collection vide. S’il est vrai que vous pouvez économiser un peu d’espace, le revers de la médaille est que cela rend votre code plus compliqué et plus fragile. Comparez ces deux versions d’une méthode pour additionner un tableau :

La première version est la façon dont vous coderiez normalement la méthode :

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed
 * @return the sum
 **/
public int sum(int[] values) {
    int sum = 0;
    for (int value : values) {
        sum += value;
    }
    return sum;
}

La deuxième version est la façon dont vous devez coder la méthode si vous avez l’habitude d’utiliser null pour représenter un tableau vide.

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed, or null.
 * @return the sum, or zero if the array is null.
 **/
public int sum(int[] values) {
    int sum = 0;
    if (values != null) {
        for (int value : values) {
            sum += value;
        }
    }
    return sum;
}

Comme vous pouvez le voir, le code est un peu plus compliqué. Cela est directement attribuable à la décision d’utiliser “null” de cette manière.

Considérez maintenant si ce tableau qui pourrait être un null est utilisé dans de nombreux endroits. À chaque endroit où vous l’utilisez, vous devez déterminer si vous devez tester “null”. Si vous manquez un test null qui doit être là, vous risquez une NullPointerException. Par conséquent, la stratégie consistant à utiliser “null” de cette manière rend votre application plus fragile ; c’est-à-dire plus vulnérable aux conséquences des erreurs du programmeur.

La leçon ici est d’utiliser des tableaux vides et des listes vides lorsque c’est ce que vous voulez dire.

int[] values = new int[0];                     // always empty
List<Integer> list = new ArrayList();          // initially empty
List<Integer> list = Collections.emptyList();  // always empty

L’espace au-dessus est petit et il existe d’autres moyens de le minimiser si cela en vaut la peine.

Pitfall - Renvoyer null au lieu de lancer une exception #

Certains programmeurs Java ont une aversion générale pour lancer ou propager des exceptions. Cela conduit à un code comme celui-ci :

public Reader getReader(String pathname) {
    try {
        return new BufferedReader(FileReader(pathname));
    } catch (IOException ex) {
        System.out.println("Open failed: " + ex.getMessage());
        return null;
    }

}

Alors quel est le problème avec ça ?

Le problème est que le getReader renvoie un null comme valeur spéciale pour indiquer que le Reader n’a pas pu être ouvert. Maintenant, la valeur renvoyée doit être testée pour voir si elle est “null” avant d’être utilisée. Si le test est omis, le résultat sera une NullPointerException.

Il y a en fait trois problèmes ici :

  1. Le IOException a été intercepté trop tôt.
  2. La structure de ce code signifie qu’il y a un risque de fuite d’une ressource.
  3. Un null a été utilisé puis retourné car aucun “vrai” Reader n’était disponible pour le retour.

En fait, en supposant que l’exception devait être interceptée tôt comme ceci, il y avait quelques alternatives au retour de “null”:

  1. Il serait possible d’implémenter une classe NullReader ; par exemple. un où les opérations de l’API se comportent comme si le lecteur était déjà à la position “fin de fichier”.
  2. Avec Java 8, il serait possible de déclarer getReader comme renvoyant un Optional<Reader>.

Piège - Ne pas vérifier si un flux d’E/S n’est même pas initialisé lors de sa fermeture #

Pour éviter les fuites de mémoire, il ne faut pas oublier de fermer un flux d’entrée ou un flux de sortie dont le travail est terminé. Cela se fait généralement avec une instruction try-catch-finally sans la partie catch :

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        out.close();
    }
}

Bien que le code ci-dessus puisse sembler innocent, il présente un défaut qui peut rendre le débogage impossible. Si la ligne où out est initialisé (out = new FileOutputStream(filename)) lève une exception, alors out sera null lorsque out.close() est exécuté, ce qui entraîne une méchante NullPointerException  !

Pour éviter cela, assurez-vous simplement que le flux n’est pas “null” avant d’essayer de le fermer.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        if (out != null)
            out.close();
    }
}

Une approche encore meilleure consiste à essayer avec des ressources, car il fermera automatiquement le flux avec une probabilité de 0 pour lancer un NPE sans avoir besoin d’un bloc finally.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    try (FileOutputStream out = new FileOutputStream(filename)) {
        for(; count > 0; count--)
            out.write(0);
    }
}

Piège - L’utilisation inutile de Primitive Wrappers peut conduire à NullPointerExceptions #

Parfois, les programmeurs débutants en Java utiliseront indifféremment les types primitifs et les wrappers. Cela peut entraîner des problèmes. Considérez cet exemple :

public class MyRecord {
    public int a, b;
    public Integer c, d;
}

...
MyRecord record = new MyRecord();
record.a = 1;               // OK
record.b = record.b + 1;    // OK
record.c = 1;               // OK
record.d = record.d + 1;    // throws a NullPointerException

Notre classe MyRecord1 s’appuie sur l’initialisation par défaut pour initialiser les valeurs de ses champs. Ainsi, lorsque nous créons un enregistrement, les champs a et b seront mis à zéro, et les champs c et d seront mis à null.

Lorsque nous essayons d’utiliser les champs initialisés par défaut, nous voyons que les champs int fonctionnent tout le temps, mais les champs Integer fonctionnent dans certains cas et pas dans d’autres. Plus précisément, dans le cas qui échoue (avec d), ce qui se passe est que l’expression du côté droit tente de déballer une référence null, et c’est ce qui provoque la levée de NullPointerException.

Il y a plusieurs façons de voir cela :

  • Si les champs c et d doivent être des wrappers primitifs, alors soit nous ne devrions pas nous fier à l’initialisation par défaut, soit nous devrions tester null. Pour le premier est l’approche correcte * à moins qu’il n’y ait une signification définie pour les champs à l’état “null”.

  • Si les champs n’ont pas besoin d’être des wrappers primitifs, alors c’est une erreur d’en faire des wrappers primitifs. En plus de ce problème, les wrappers primitifs ont des surcharges supplémentaires par rapport aux types primitifs.

La leçon ici est de ne pas utiliser de types wrapper primitifs à moins que vous n’en ayez vraiment besoin.

1 - Cette classe n’est pas un exemple de bonne pratique de codage. Par exemple, une classe bien conçue n’aurait pas de champs publics. Cependant, ce n’est pas le but de cet exemple.

Pitfall - Utilisation de la “notation Yoda” pour éviter NullPointerException #

De nombreux exemples de code publiés sur StackOverflow incluent des extraits comme celui-ci :

if ("A".equals(someString)) {
    // do something
}

Cela “empêche” ou “évite” une éventuelle NullPointerException dans le cas où someString est null. De plus, on peut soutenir que

    "A".equals(someString)

est mieux que:

    someString != null && someString.equals("A")

(Il est plus concis et, dans certaines circonstances, il peut être plus efficace. Cependant, comme nous le verrons ci-dessous, la concision pourrait être négative.)

Cependant, le véritable écueil consiste à utiliser le test Yoda ** pour éviter les NullPointerExceptions ** par habitude.

Lorsque vous écrivez "A".equals(someString) vous “réparez” le cas où someString se trouve être null. Mais comme l’explique un autre exemple (https://www.wikiod.com/fr/java/pieges-java—nulls-et-nullpointerexception#Piège – « Réparer » les nulls inattendus ), “rendre bon” null valeurs peuvent être nuisibles pour diverses raisons.

Cela signifie que les conditions Yoda ne sont pas des “meilleures pratiques”1. À moins que le null ne soit attendu, il est préférable de laisser le NullPointerException se produire afin que vous puissiez obtenir un échec de test unitaire (ou un rapport de bogue). Cela vous permet de trouver et de corriger le bogue qui a provoqué l’apparition du “null” inattendu/indésirable.

Les conditions Yoda ne doivent être utilisées que dans les cas où le null est * attendu * car l’objet que vous testez provient d’une API qui est * documentée * comme renvoyant un null. Et sans doute, il pourrait être préférable d’utiliser l’une des manières les moins jolies d’exprimer le test, car cela aide à mettre en évidence le test “null” pour quelqu’un qui examine votre code.

1 - Selon [Wikipedia][1] : “Les meilleures pratiques de codage sont un ensemble de règles informelles que la communauté des développeurs de logiciels a apprises au fil du temps et qui peuvent aider à améliorer la qualité des logiciels.”. L’utilisation de la notation Yoda n’y parvient pas. Dans de nombreuses situations, cela aggrave le code.
