Piège : utilisation incorrecte de wait() / notify() #

Les méthodes object.wait(), object.notify() et object.notifyAll() sont destinées à être utilisées d’une manière très spécifique. (voir https://www.wikiod.com/fr/java )

Le problème “Notification perdue” #

Une erreur courante des débutants est d’appeler inconditionnellement object.wait()

private final Object lock = new Object();

public void myConsumer() {
    synchronized (lock) {
        lock.wait();     // DON'T DO THIS!!
    }
    doSomething();
}

La raison pour laquelle c’est faux est que cela dépend d’un autre thread pour appeler lock.notify() ou lock.notifyAll(), mais rien ne garantit que l’autre thread n’a pas effectué cet appel avant le thread consommateur appelé lock .wait().

lock.notify() et lock.notifyAll() ne font rien du tout si un autre thread n’attend pas déjà la notification. Le thread qui appelle myConsumer() dans cet exemple se bloquera indéfiniment s’il est trop tard pour intercepter la notification.

Le bogue “État de moniteur illégal” #

Si vous appelez wait() ou notify() sur un objet sans détenir le verrou, la JVM lèvera IllegalMonitorStateException.

public void myConsumer() {
    lock.wait();      // throws exception
    consume();
}

public void myProducer() {
    produce();
    lock.notify();    // throws exception
}

(La conception de wait() / notify() nécessite que le verrou soit maintenu car cela est nécessaire pour éviter les conditions de concurrence systémique. S’il était possible d’appeler wait() ou notify() sans verrouiller , il serait alors impossible d’implémenter le cas d’utilisation principal pour ces primitives : attendre qu’une condition se produise.)

Attendre/notifier est de niveau trop bas #

La meilleure façon d’éviter les problèmes avec wait() et notify() est de ne pas les utiliser. La plupart des problèmes de synchronisation peuvent être résolus en utilisant les objets de synchronisation de niveau supérieur (files d’attente, barrières, sémaphores, etc.) disponibles dans le package java.utils.concurrent.

Piège - Extension de ‘java.lang.Thread’ #

Le javadoc de la classe Thread montre deux manières de définir et d’utiliser un thread :

Utilisation d’une classe de thread personnalisée :

 class PrimeThread extends Thread {
     long minPrime;
     PrimeThread(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeThread p = new PrimeThread(143);
 p.start();

Utilisation d’un Runnable :

 class PrimeRun implements Runnable {
     long minPrime;
     PrimeRun(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeRun p = new PrimeRun(143);
 new Thread(p).start();

(Source : [java.lang.Thread javadoc][1].)

L’approche de la classe de threads personnalisés fonctionne, mais elle présente quelques problèmes :

  1. Il est gênant d’utiliser PrimeThread dans un contexte qui utilise un pool de threads classique, un exécuteur ou le framework ForkJoin. (Ce n’est pas impossible, car PrimeThread implémente indirectement Runnable, mais utiliser une classe Thread personnalisée comme Runnable est certainement maladroit et peut ne pas être viable … selon d’autres aspects de la classe.)

  2. Il y a plus de possibilités d’erreurs dans d’autres méthodes. Par exemple, si vous avez déclaré un PrimeThread.start() sans déléguer à Thread.start(), vous vous retrouveriez avec un “thread” qui s’exécute sur le thread actuel.

L’approche consistant à placer la logique de thread dans un “Runnable” évite ces problèmes. En effet, si vous utilisez une classe anonyme (Java 1.1 et versions ultérieures) pour implémenter le Runnable, le résultat est plus succinct et plus lisible que les exemples ci-dessus.

 final long minPrime = ...
 new Thread(new Runnable() {
     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }.start();

Avec une expression lambda (à partir de Java 8), l’exemple ci-dessus deviendrait encore plus élégant :

 final long minPrime = ...
 new Thread(() -> {
    // compute primes larger than minPrime
     . . .
 }).start();

[1] : http://docs.oracle.com/javase/8/docs/api/java/lang/Thread.html

Piège - Trop de threads ralentissent une application. #

Beaucoup de gens qui découvrent le multi-threading pensent que l’utilisation de threads accélère automatiquement une application. En fait, c’est beaucoup plus compliqué que cela. Mais une chose que nous pouvons affirmer avec certitude est que pour tout ordinateur, il existe une limite au nombre de threads pouvant être exécutés en même temps :

  • Un ordinateur a un nombre fixe de cœurs (ou hyperthreads).
  • Un thread Java doit être programmé sur un noyau ou un hyperthread pour pouvoir s’exécuter.
  • S’il y a plus de threads Java exécutables que de cœurs / hyperthreads (disponibles), certains d’entre eux doivent attendre.

Cela nous indique que le simple fait de créer de plus en plus de threads Java * ne peut pas * rendre l’application de plus en plus rapide. Mais il y a aussi d’autres considérations :

  • Chaque thread nécessite une région de mémoire hors tas pour sa pile de threads. La taille de pile de threads typique (par défaut) est de 512 Ko ou 1 Mo. Si vous avez un nombre important de threads, l’utilisation de la mémoire peut être importante.

  • Chaque thread actif fera référence à un certain nombre d’objets dans le tas. Cela augmente l’ensemble de travail d’objets * accessibles *, ce qui a un impact sur la récupération de place et sur l’utilisation de la mémoire physique.

  • Les frais généraux de commutation entre les threads ne sont pas triviaux. Cela implique généralement un basculement dans l’espace du noyau du système d’exploitation pour prendre une décision de planification des threads.

  • Les frais généraux de la synchronisation des threads et de la signalisation inter-threads (par exemple, wait(), notify() / notifyAll) peuvent être significatifs.

Selon les détails de votre application, ces facteurs signifient généralement qu’il existe un “point idéal” pour le nombre de threads. Au-delà de cela, l’ajout de plus de threads apporte une amélioration minimale des performances et peut aggraver les performances.

Si votre application crée pour chaque nouvelle tâche, une augmentation inattendue de la charge de travail (par exemple, un taux de demande élevé) peut entraîner un comportement catastrophique.

Une meilleure façon de gérer cela consiste à utiliser un pool de threads limité dont vous pouvez contrôler la taille (statiquement ou dynamiquement). Lorsqu’il y a trop de travail à faire, l’application doit mettre les requêtes en file d’attente. Si vous utilisez un ExecutorService, il s’occupera de la gestion du pool de threads et de la mise en file d’attente des tâches.

Piège - La création de threads est relativement coûteuse #

Considérez ces deux micro-benchmarks :

Piège : les variables partagées nécessitent une synchronisation appropriée #

Considérez cet exemple :

public class ThreadTest implements Runnable {
   
    private boolean stop = false;
    
    public void run() {
        long counter = 0;
        while (!stop) {
            counter = counter + 1;
        }
        System.out.println("Counted " + counter);
    }

    public static void main(String[] args) {
        ThreadTest tt = new ThreadTest();
        new Thread(tt).start();    // Create and start child thread
        Thread.sleep(1000);
        tt.stop = true;            // Tell child thread to stop.
    }
}

L’intention de ce programme est de démarrer un thread, de le laisser s’exécuter pendant 1000 millisecondes, puis de l’arrêter en définissant le drapeau stop.

Cela fonctionnera-t-il comme prévu ? #

Peut-être que oui, peut-être que non.

Une application ne s’arrête pas nécessairement au retour de la méthode main. Si un autre thread a été créé et que ce thread n’a pas été marqué comme thread démon, l’application continuera à s’exécuter après la fin du thread principal. Dans cet exemple, cela signifie que l’application continuera à s’exécuter jusqu’à la fin du thread enfant. Cela devrait se produire lorsque tt.stop est défini sur true.

Mais ce n’est en fait pas strictement vrai. En fait, le thread enfant s’arrêtera après avoir observé stop avec la valeur true. Cela arrivera-t-il? Peut-être que oui, peut-être que non.

La spécification du langage Java garantit que les lectures et écritures en mémoire effectuées dans un thread sont visibles pour ce thread, selon l’ordre des instructions dans le code source. Cependant, en général, cela n’est PAS garanti lorsqu’un thread écrit et qu’un autre thread (ensuite) lit. Pour obtenir une visibilité garantie, il doit y avoir une chaîne de relations * se passe avant * entre une écriture et une lecture ultérieure. Dans l’exemple ci-dessus, il n’y a pas de chaîne de ce type pour la mise à jour du drapeau stop, et il n’est donc pas garanti que le thread enfant verra stop passer à true.

(Remarque aux auteurs : il devrait y avoir une rubrique distincte sur le modèle de mémoire Java pour entrer dans les détails techniques approfondis.)

Comment résoudre le problème ? #

Dans ce cas, il existe deux manières simples de s’assurer que la mise à jour “stop” est visible :

  1. Déclarez stop comme étant volatile ; c’est à dire.

       private volatile boolean stop = false;

    For a volatile variable, the JLS specifies that there is a happens-before relation between a write by one thread and a later read by a second thread.

  2. Utilisez un mutex pour synchroniser comme suit :

    public class ThreadTest implements Runnable {

     private boolean stop = false;

 public void run() {
     long counter = 0;
     while (true) {
         synchronize (this) {
             if (stop) {
                 break;
             }
         }
         counter = counter + 1;
     }
     System.out.println("Counted " + counter);
 }

 public static void main(String[] args) {
     ThreadTest tt = new ThreadTest();
     new Thread(tt).start();    // Create and start child thread
     Thread.sleep(1000);
     synchronize (tt) {
         tt.stop = true;        // Tell child thread to stop.
     }
 }

    }

En plus de garantir qu’il y a exclusion mutuelle, le JLS spécifie qu’il existe une relation * se produit avant * entre la libération d’un mutex dans un thread et l’obtention du même mutex dans un deuxième thread.

Mais l’affectation n’est-elle pas atomique ? #

Oui c’est le cas!

Cependant, cela ne signifie pas que les effets de la mise à jour seront visibles simultanément pour tous les threads. Seule une chaîne appropriée de relations * se produit avant * garantira cela.

Pourquoi ont-ils fait ça? #

Les programmeurs qui font de la programmation multithread en Java pour la première fois trouvent que le modèle de mémoire est un défi. Les programmes se comportent de manière non intuitive car l’attente naturelle est que les écritures sont visibles de manière uniforme. Alors pourquoi les concepteurs Java conçoivent le modèle de mémoire de cette façon.

Il s’agit en fait d’un compromis entre performance et facilité d’utilisation (pour le programmeur).

Une architecture informatique moderne se compose de plusieurs processeurs (cœurs) avec des ensembles de registres individuels. La mémoire principale est accessible soit à tous les processeurs soit à des groupes de processeurs. Une autre propriété du matériel informatique moderne est que l’accès aux registres est généralement plus rapide d’un ordre de grandeur que l’accès à la mémoire principale. À mesure que le nombre de cœurs augmente, il est facile de voir que la lecture et l’écriture dans la mémoire principale peuvent devenir le principal goulot d’étranglement des performances d’un système.

Cette inadéquation est résolue en mettant en œuvre un ou plusieurs niveaux de mémoire cache entre les cœurs de processeur et la mémoire principale. Chaque cœur accède aux cellules mémoire via son cache. Normalement, une lecture en mémoire principale ne se produit qu’en cas d’absence de cache, et une écriture en mémoire principale ne se produit que lorsqu’une ligne de cache doit être vidée. Pour une application où l’ensemble de travail d’emplacements de mémoire de chaque cœur tient dans son cache, la vitesse du cœur n’est plus limitée par la vitesse/la bande passante de la mémoire principale.

Mais cela nous pose un nouveau problème lorsque plusieurs cœurs lisent et écrivent des variables partagées. La dernière version d’une variable peut se trouver dans le cache d’un cœur. À moins que ce cœur ne vide la ligne de cache vers la mémoire principale ET que d’autres cœurs invalident leur copie en cache des anciennes versions, certains d’entre eux sont susceptibles de voir des versions obsolètes de la variable. Mais si les caches étaient vidés de la mémoire à chaque fois qu’il y avait une écriture dans le cache (“juste au cas où” il y aurait une lecture par un autre cœur), cela consommerait inutilement la bande passante de la mémoire principale.

La solution standard utilisée au niveau du jeu d’instructions matérielles consiste à fournir des instructions pour l’invalidation du cache et une écriture directe du cache, et à laisser au compilateur le soin de décider quand les utiliser.

Retour à Java. le modèle de mémoire est conçu pour que les compilateurs Java ne soient pas obligés d’émettre des instructions d’invalidation de cache et d’écriture directe là où elles ne sont pas vraiment nécessaires. L’hypothèse est que le programmeur utilisera un mécanisme de synchronisation approprié (par exemple mutex primitifs, “volatile”, classes de concurrence de niveau supérieur, etc.) pour indiquer qu’il a besoin de visibilité mémoire. En l’absence d’une relation happens-before, les compilateurs Java sont libres de supposer qu’aucune opération de cache (ou similaire) n’est requise.

Cela présente des avantages significatifs en termes de performances pour les applications multithread, mais l’inconvénient est que l’écriture d’applications multithread correctes n’est pas simple. Le programmeur doit comprendre ce qu’il ou elle fait.

Pourquoi ne puis-je pas reproduire cela ? #

Il existe un certain nombre de raisons pour lesquelles des problèmes comme celui-ci sont difficiles à reproduire :

  1. Comme expliqué ci-dessus, la conséquence de ne pas traiter correctement les problèmes de visibilité de la mémoire est généralement que votre application compilée ne gère pas correctement les caches mémoire. Cependant, comme nous l’avons mentionné ci-dessus, les caches mémoire sont souvent vidés de toute façon.

  2. Lorsque vous changez de plate-forme matérielle, les caractéristiques des caches mémoire peuvent changer. Cela peut entraîner un comportement différent si votre application ne se synchronise pas correctement.

  3. Vous observez peut-être les effets d’une synchronisation * fortuite *. Par exemple, si vous ajoutez des traceprints, il s’agit généralement d’une synchronisation qui se produit en arrière-plan dans les flux d’E/S qui provoque des vidages du cache. Ainsi, l’ajout d’empreintes souvent entraîne un comportement différent de l’application.

  4. L’exécution d’une application sous un débogueur entraîne sa compilation différente par le compilateur JIT. Les points d’arrêt et le pas à pas exacerbent cela. Ces effets modifient souvent le comportement d’une application.

Ces éléments rendent les bogues dus à une synchronisation inadéquate particulièrement difficiles à résoudre.
