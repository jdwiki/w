Cette page de documentation donne accès aux instructions d’installation de java standard edition sur les ordinateurs Windows, Linux et macOS.

Définition de %PATH% et %JAVA_HOME% après l’installation sous Windows #

Hypothèses: #

  • Un JDK Oracle a été installé.
  • Le JDK a été installé dans le répertoire par défaut.

Étapes de configuration #

  1. Ouvrez l’Explorateur Windows.
  2. Dans le volet de navigation à gauche, cliquez sur Ce PC (ou Ordinateur pour les anciennes versions de Windows). Il existe un moyen plus court sans utiliser l’explorateur dans les versions actuelles de Windows : appuyez simplement sur Win+Pause
  3. Dans la nouvelle fenêtre du Panneau de configuration, cliquez avec le bouton gauche sur Paramètres système avancés qui devrait se trouver dans le coin supérieur gauche. Cela ouvrira la fenêtre Propriétés système. [![entrez la description de l’image ici][3]][3]

Vous pouvez également saisir SystemPropertiesAdvanced (insensible à la casse) dans Exécuter (Win+R) et appuyer sur Entrée.

  1. Dans l’onglet Avancé des Propriétés système, sélectionnez le bouton Variables d’environnement… dans le coin inférieur droit de la fenêtre.
  2. Ajoutez une Nouvelle variable système en cliquant sur le bouton Nouveau… dans Variables système avec le nom JAVA_HOME et dont la valeur est le chemin vers le répertoire où se trouvait le JDK installée. Après avoir saisi ces valeurs, appuyez sur OK.

[![Variable d’environnement JAVA_HOME][4]][4]

  1. Faites défiler la liste des Variables système et sélectionnez la variable “Chemin”.

  2. ATTENTION : Windows s’appuie sur Path pour trouver les programmes importants. Si tout ou partie de celui-ci est supprimé, Windows peut ne pas fonctionner correctement. Il doit être modifié pour permettre à Windows d’exécuter le JDK. Dans cet esprit, cliquez sur le bouton “Modifier…” avec la variable “Chemin” sélectionnée. Ajoutez %JAVA_HOME%\bin; au début de la variable Path.

Il est préférable d’ajouter au début de la ligne car le logiciel d’Oracle enregistrait sa propre version de Java dans Path - Cela entraînera l’ignorance de votre version si cela se produit après la déclaration d’Oracle.


Vérifie ton travail #

  1. Ouvrez l’invite de commande en cliquant sur Démarrer, puis en tapant “cmd” et en appuyant sur “Entrée”.
  2. Entrez javac -version dans l’invite. Si cela a réussi, la version du JDK sera imprimée à l’écran.

Remarque : Si vous devez réessayer, fermez l’invite avant de vérifier votre travail. Cela forcera Windows à obtenir la nouvelle version de Path.

Installer un JDK Java sous Linux #

Utilisation du gestionnaire de packages #

Les versions JDK et/ou JRE pour OpenJDK ou Oracle peuvent être installées à l’aide du gestionnaire de packages sur la plupart des distributions Linux courantes. (Les choix qui s’offrent à vous dépendent de la distribution.)

En règle générale, la procédure consiste à ouvrir la fenêtre du terminal et à exécuter les commandes indiquées ci-dessous. (Il est supposé que vous disposez d’un accès suffisant pour exécuter des commandes en tant qu’utilisateur “root” … ce que fait la commande sudo. Si vous ne le faites pas, veuillez en parler aux administrateurs de votre système.)

L’utilisation du gestionnaire de packages est recommandée car il facilite (généralement) la mise à jour de votre installation Java.

apt-get, distributions Linux basées sur Debian (Ubuntu, etc.)

Les instructions suivantes installeront Oracle Java 8 :

$ sudo add-apt-repository ppa:webupd8team/java
$ sudo apt-get update
$ sudo apt-get install oracle-java8-installer

Remarque : Pour configurer automatiquement les variables d’environnement Java 8, vous pouvez installer le package suivant :

$ sudo apt-get install oracle-java8-set-default

Création d’un fichier .deb

Si vous préférez créer vous-même le fichier .deb à partir du fichier .tar.gz téléchargé depuis Oracle, procédez comme suit (en supposant que vous avez téléchargé le fichier .tar.gz dans ./<jdk>.tar .gz):

$ sudo apt-get install java-package # might not be available in default repos

Installation d’Oracle Java

  • Java 8 - [Téléchargements Java SE] [1]
  • Java 7 - [Archives Oracle Java.] [2]

/Library/Java/JavaVirtualMachines/<version>.jdk/Contents/Home

où correspond à la version installée.

Commutation de ligne de commande

Lorsque Java est installé, la version installée est automatiquement définie comme version par défaut. Pour basculer entre différents, utilisez :

export JAVA_HOME=/usr/libexec/java_home -v 1.6 #Or 1.7 or 1.8

Les fonctions suivantes peuvent être ajoutées au ~/.bash_profile (si vous utilisez le shell Bash par défaut) pour en faciliter l’utilisation :

function java_version {
   echo 'java -version';
}

function java_set {
   if [[ $1 == "6" ]]
   then
       export JAVA_HOME='/usr/libexec/java_home -v 1.6';
       echo "Setting Java to version 6..."
       echo "$JAVA_HOME"
   elif [[ $1 == "7" ]]
   then
       export JAVA_HOME='/usr/libexec/java_home -v 1.7';
       echo "Setting Java to version 7..."
       echo "$JAVA_HOME"
   elif [[ $1 == "8" ]]
   then
       export JAVA_HOME='/usr/libexec/java_home -v 1.8';
       echo "Setting Java to version 8..."
       echo "$JAVA_HOME"
   fi            
}

Installation d’un JDK ou JRE Java sous Windows #

Seuls les JDK et JRE Oracle sont disponibles pour les plates-formes Windows. La procédure d’installation est simple :

  1. Visitez la [page de téléchargement] d’Oracle Java :
  2. Cliquez sur le bouton JDK, le bouton JRE ou le bouton Server JRE. Notez que pour développer en Java, vous avez besoin de JDK. Pour connaître la différence entre JDK et JRE, voir [ici][2]
  3. Faites défiler jusqu’à la version que vous souhaitez télécharger. (Généralement, la plus récente est recommandée.)
  4. Sélectionnez le bouton radio “Accepter le contrat de licence”.

  6. Exécutez le programme d’installation… de la manière habituelle pour votre version de Windows.

Une autre façon d’installer Java sur Windows à l’aide de l’invite de commande consiste à utiliser Chocolately :

  1. Installez Chocolately depuis https://chocolatey.org/

  2. Ouvrez une instance cmd, par exemple appuyez sur Win+R puis tapez “cmd” dans la fenêtre “Exécuter” suivi d’une entrée.

  3. Dans votre instance cmd, exécutez la commande suivante pour télécharger et installer un JDK Java 8 :

Mise en route avec les versions portables

Dans certains cas, vous souhaiterez peut-être installer JDK/JRE sur un système avec des privilèges limités, comme une machine virtuelle, ou vous souhaiterez peut-être installer et utiliser plusieurs versions ou architectures (x64/x86) de JDK/JRE. Les étapes restent les mêmes jusqu’au moment où vous téléchargez le programme d’installation (.EXE). Les étapes suivantes sont les suivantes (les étapes s’appliquent à JDK/JRE 7 et versions ultérieures, pour les anciennes versions, elles sont légèrement différentes dans les noms de dossiers et de fichiers) :

  1. Déplacez le fichier vers un emplacement approprié où vous souhaitez que vos fichiers binaires Java résident en permanence.

  2. Installez 7-Zip ou sa version portable si vous disposez de privilèges limités.

  3. Avec 7-Zip, extrayez les fichiers de l’EXE du programme d’installation Java à l’emplacement.

  4. Ouvrez l’invite de commande en maintenant les touches “Shift” et “Clic droit” enfoncées dans le dossier de l’explorateur ou accédez à cet emplacement depuis n’importe où.

  5. Accédez au dossier nouvellement créé. Disons que le nom du dossier est jdk-7u25-windows-x64. Tapez donc cd jdk-7u25-windows-x64. Tapez ensuite les commandes suivantes dans l’ordre :

    cd .rsrc\JAVA_CAB10

    extrac32 111

  6. Cela créera un fichier tools.zip à cet emplacement. Extrayez le tools.zip avec 7-Zip afin que les fichiers qu’il contient soient maintenant créés sous tools dans le même répertoire.

  7. Exécutez maintenant ces commandes sur l’invite de commande déjà ouverte :

    cd tools

    for /r %x in (*.pack) do .\bin\unpack200 -r "%x" "%~dx%~px%~nx.jar"

  8. Attendez que la commande se termine. Copiez le contenu de tools à l’emplacement où vous souhaitez placer vos fichiers binaires.

De cette façon, vous pouvez installer simultanément toutes les versions de JDK/JRE dont vous avez besoin.

Sélection d’une version Java SE appropriée #

Il y a eu de nombreuses versions de Java depuis la version originale de Java 1.0 en 1995. (Reportez-vous à [Java version history][1] pour un résumé.) Cependant, la plupart des versions ont dépassé leurs dates officielles de fin de vie. Cela signifie que le fournisseur (généralement Oracle maintenant) a cessé tout nouveau développement pour la version et ne fournit plus de correctifs publics/gratuits pour les bogues ou les problèmes de sécurité. (Les versions de correctifs privées sont généralement disponibles pour les personnes/organisations avec un contrat de support ; contactez le bureau de vente de votre fournisseur.)

En général, la version recommandée de Java SE sera la dernière mise à jour de la dernière version publique. Actuellement, cela signifie la dernière version Java 8 disponible. Java 9 doit être publié en 2017. (Java 7 a dépassé sa fin de vie et la dernière version publique date d’avril 2015. Java 7 et les versions antérieures ne sont pas recommandées.)

Cette recommandation s’applique à tous les nouveaux développements Java et à toute personne apprenant Java. Cela s’applique également aux personnes qui souhaitent simplement exécuter un logiciel Java fourni par un tiers. De manière générale, un code Java bien écrit fonctionnera sur une version plus récente de Java. (Mais vérifiez les notes de version du logiciel et contactez l’auteur / le fournisseur / le vendeur si vous avez des doutes.)

Si vous travaillez sur une ancienne base de code Java, nous vous conseillons de vous assurer que votre code s’exécute sur la dernière version de Java. Décider quand commencer à utiliser les fonctionnalités des nouvelles versions de Java est plus difficile, car cela aura un impact sur votre capacité à prendre en charge les clients qui ne peuvent pas ou ne veulent pas leur installation Java.

[1] : https://en.wikipedia.org/wiki/Java_version_history

Version Java et dénomination de la version #

La dénomination des versions Java est un peu déroutante. Il existe en fait deux systèmes de dénomination et de numérotation, comme indiqué dans ce tableau :

Version JDK Nom commercial
jdk-1.0 JDK 1.0
jdk-1.1 JDK 1.1
jdk-1.2 J2SE 1.2
… …
jdk-1.5 J2SE 1.5 renommé Java SE 5
jdk-1.6 Java SE 6
jdk-1.7 Java SE 7
jdk-1.8 Java SE 8
jdk-91 Java SE 9 (pas encore sorti)

1 - Il semble qu’Oracle ait l’intention de rompre avec sa pratique précédente consistant à utiliser un schéma de “numéro de version sémantique” dans les chaînes de version Java. Il reste à voir s’ils donneront suite à cela.

Le « SE » dans les noms commerciaux fait référence à l’édition standard. Il s’agit de la version de base pour exécuter Java sur la plupart des ordinateurs portables, PC et serveurs (à l’exception d’Android).

Il existe deux autres éditions officielles de Java : “Java ME” est l’édition Micro et “Java EE” est l’édition Enterprise. La saveur Android de Java est également très différente de Java SE. Java ME, Java EE et Android Java sortent du cadre de cette rubrique.

Le numéro de version complet d’une version Java ressemble à ceci :

1.8.0_101-b13

Cela indique JDK 1.8.0, mise à jour 101, Build #13. Oracle fait référence à cela dans les notes de version comme suit :

Java™ SE Development Kit 8, Update 101 (JDK 8u101)

Le numéro de mise à jour est important - Oracle publie régulièrement des mises à jour d’une version majeure avec des correctifs de sécurité, des corrections de bogues et (dans certains cas) de nouvelles fonctionnalités. Le numéro de build est généralement sans importance. Notez que Java 8 et Java 1.8 fait référence à la même chose ; Java 8 n’est que le nom “marketing” de Java 1.8.

De quoi ai-je besoin pour le développement Java #

Une installation JDK et un éditeur de texte sont le strict minimum pour le développement Java. (C’est bien d’avoir un éditeur de texte capable de mettre en évidence la syntaxe Java, mais vous pouvez vous en passer.)

Cependant, pour les travaux de développement sérieux, il est recommandé d’utiliser également les éléments suivants :

  • Un IDE Java tel que Eclipse, Intellij IDEA ou NetBeans
  • Un outil de build Java tel que Ant, Gradle ou Maven
  • Un système de contrôle de version pour gérer votre base de code (avec des sauvegardes appropriées et une réplication hors site)
  • Outils de test et outils CI (intégration continue)

Configuration et changement de versions de Java sous Linux à l’aide d’alternatives #

Utiliser des alternatives #

De nombreuses distributions Linux utilisent la commande alternatives pour basculer entre différentes versions d’une commande. Vous pouvez l’utiliser pour basculer entre différentes versions de Java installées sur une machine.

  1. Dans un shell de commande, définissez $JDK sur le chemin d’accès d’un JDK nouvellement installé ; par exemple.

     $ JDK=/Data/jdk1.8.0_67

  2. Utilisez alternatives --install pour ajouter les commandes du SDK Java aux alternatives :

     $ sudo alternatives --install /usr/bin/java java $JDK/bin/java 2
 $ sudo alternatives --install /usr/bin/javac javac $JDK/bin/javac 2
 $ sudo alternatives --install /usr/bin/jar jar $JDK/bin/jar 2

    And so on.

Vous pouvez maintenant basculer entre différentes versions d’une commande Java comme suit :

$ sudo alternatives --config javac

There is 1 program that provides 'javac'.

  Selection    Command
-----------------------------------------------
*+ 1           /usr/lib/jvm/java-1.8.0-openjdk-1.8.0.101-1.b14.fc23.x86_64/bin/javac
   2           /Data/jdk1.8.0_67/bin/javac

Enter to keep the current selection[+], or type selection number: 2
$

Pour plus d’informations sur l’utilisation des “alternatives”, reportez-vous à l’entrée de manuel [alternatives(8)][1].

Installations basées sur Arch #

Les installations basées sur Arch Linux sont livrées avec la commande “archlinux-java” pour changer de version de Java.

Liste des environnements installés #

$ archlinux-java status
Available Java environments:
  java-7-openjdk (default)
  java-8-openjdk/jre

Changement d’environnement actuel #

# archlinux-java set <JAVA_ENV_NAME>

Par exemple:

# archlinux-java set java-8-openjdk/jre

Plus d’informations peuvent être trouvées sur le [Arch Linux Wiki][2]

[1] : http://linux.die.net/man/8/alternatives [2] : https://wiki.archlinux.org/index.php/java#Switching_between_JVM

Vérification et configuration post-installation sous Linux #

Après avoir installé un SDK Java, il est conseillé de vérifier qu’il est prêt à l’emploi. Vous pouvez le faire en exécutant ces deux commandes, en utilisant votre compte utilisateur normal :

$ java -version
$ javac -version

Ces commandes impriment les informations de version pour le JRE et le JDK (respectivement) qui se trouvent sur le chemin de recherche des commandes de votre shell. Recherchez la chaîne de version JDK / JRE.

  • Si l’une des commandes ci-dessus échoue, en disant “commande introuvable”, alors le JRE ou le JDK n’est pas du tout sur le chemin de recherche ; allez à Configurer PATH directement ci-dessous.
  • Si l’une des commandes ci-dessus affiche une chaîne de version différente de celle à laquelle vous vous attendiez, votre chemin de recherche ou le système “alternatives” doit être ajusté ; allez à Vérification des alternatives
  • Si les chaînes de version correctes sont affichées, vous avez presque terminé ; passez à Vérifier JAVA_HOME

Configuration PATH directement

S’il n’y a pas de java ou javac sur le chemin de recherche pour le moment, la solution simple consiste à l’ajouter à votre chemin de recherche.

Tout d’abord, trouvez où vous avez installé Java ; voir Où Java a été installé ? ci-dessous si vous avez des doutes.

Ensuite, en supposant que bash est votre shell de commande, utilisez un éditeur de texte pour ajouter les lignes suivantes à la fin de ~/.bash_profile ou ~/.bashrc (si vous utilisez Bash comme shell).

JAVA_HOME=<installation directory>
PATH=$JAVA_HOME/bin:$PATH

export JAVA_HOME
export PATH

… en remplaçant <répertoire d'installation> par le chemin de votre répertoire d’installation Java. Notez que ce qui précède suppose que le répertoire d’installation contient un répertoire bin et que le répertoire bin contient les commandes java et javac que vous essayez d’utiliser.

Ensuite, sourcez le fichier que vous venez de modifier, afin que les variables d’environnement de votre shell actuel soient mises à jour.

$ source ~/.bash_profile

Ensuite, répétez les vérifications de version java et javac. S’il y a toujours des problèmes, utilisez which java et which javac pour vérifier que vous avez correctement mis à jour les variables d’environnement.

Enfin, déconnectez-vous et reconnectez-vous afin que les variables d’environnement mises à jour ptopagate à tous vos shells. Vous devriez maintenant avoir terminé.

Vérification des alternatives

Si java -version ou javac -version a fonctionné mais a donné un numéro de version inattendu, vous devez vérifier d’où viennent les commandes. Utilisez which et ls -l pour le savoir comme suit :

$ ls -l `which java`

Si la sortie ressemble à ceci, :

lrwxrwxrwx. 1 root root 22 Jul 30 22:18 /usr/bin/java -> /etc/alternatives/java

alors le changement de version “alternatives” est utilisé. Vous devez décider si vous souhaitez continuer à l’utiliser ou simplement le remplacer en définissant directement le PATH.

  • [Configuration et changement de versions de Java sous Linux à l’aide d’alternatives] [1]
  • Voir “Configurer PATH directement” ci-dessus.

Où Java a-t-il été installé ?

Java peut être installé à divers endroits, selon la méthode d’installation.

  • Les RPM Oracle placent l’installation de Java dans “/usr/java”.
  • Sur Fedora, l’emplacement par défaut est “/usr/lib/jvm”.
  • Si Java a été installé manuellement à partir de fichiers ZIP ou JAR, l’installation peut se faire n’importe où.

Si vous rencontrez des difficultés pour trouver le répertoire d’installation, nous vous suggérons d’utiliser find (ou slocate) pour trouver la commande. Par exemple:

$ find / -name java -type f 2> /dev/null

Cela vous donne les noms de chemin pour tous les fichiers appelés “java” sur votre système. (La redirection de l’erreur standard vers “/dev/null” supprime les messages concernant les fichiers et répertoires auxquels vous ne pouvez pas accéder.)
