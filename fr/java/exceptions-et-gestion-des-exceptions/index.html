Les objets de type Throwable et ses sous-types peuvent être envoyés vers le haut de la pile avec le mot-clé throw et interceptés avec des instructions try…catch.

Syntaxe #

  • void someMethod() lance SomeException { } //déclaration de méthode, force la méthode appelants à intercepter si SomeException est un type d’exception vérifié

  • essayer {

     someMethod(); //code that might throw an exception

    }

  • catch (SomeException e) {

      System.out.println("SomeException was thrown!"); //code that will run if certain exception (SomeException) is thrown

    }

  • finalement {

      //code that will always run, whether try block finishes or not

    }

Attraper une exception avec try-catch #

Une exception peut être interceptée et gérée à l’aide de l’instruction try...catch. (En fait, les instructions try prennent d’autres formes, comme décrit dans d’autres exemples sur [try...catch...finally][1] et [try-with-resources][2].)

Try-catch avec un bloc catch #

La forme la plus simple ressemble à ceci :

try {
    doSomething();
} catch (SomeException e) {
    handle(e);
}
// next statement

Le comportement d’un simple try...catch est le suivant :

  • Les instructions du bloc try sont exécutées.
  • Si aucune exception n’est levée par les instructions du bloc try, alors le contrôle passe à l’instruction suivante après try...catch.
  • Si une exception est levée dans le bloc try.
    • The exception object is tested to see if it is an instance of SomeException or a subtype.
    • If it is, then the catch block will catch the exception:
      • The variable e is bound to the exception object.
      • The code within the catch block is executed.
      • If that code throws an exception, then the newly thrown exception is propagated in place of the original one.
      • Otherwise, control passes to the next statement after the try...catch.
    • If it is not, the original exception continues to propagate.

Try-catch avec plusieurs prises #

Un try...catch peut aussi avoir plusieurs blocs catch. Par exemple:

try {
    doSomething();
} catch (SomeException e) {
    handleOneWay(e)
} catch (SomeOtherException e) {
    handleAnotherWay(e);
}
// next statement

S’il y a plusieurs blocs catch, ils sont essayés un par un en commençant par le premier, jusqu’à ce qu’une correspondance soit trouvée pour l’exception. Le gestionnaire correspondant est exécuté (comme ci-dessus), puis le contrôle est passé à l’instruction suivante après l’instruction try...catch. Les blocs catch après celui qui correspond sont toujours ignorés, même si le code du gestionnaire lève une exception.

La stratégie de correspondance “de haut en bas” a des conséquences pour les cas où les exceptions dans les blocs “catch” ne sont pas disjointes. Par exemple:

try {
    throw new RuntimeException("test");
} catch (Exception e) {
    System.out.println("Exception");
} catch (RuntimeException e) {
    System.out.println("RuntimeException");
}

Cet extrait de code affichera “Exception” plutôt que “RuntimeException”. Étant donné que RuntimeException est un sous-type de Exception, le premier catch (plus général) sera mis en correspondance. Le deuxième “catch” (plus spécifique) ne sera jamais exécuté.

La leçon à en tirer est que les blocs “catch” les plus spécifiques (en termes de types d’exceptions) doivent apparaître en premier et les plus généraux en dernier. (Certains compilateurs Java vous avertiront si un catch ne peut jamais être exécuté, mais ce n’est pas une erreur de compilation.)

Blocs catch multi-exceptions #

À partir de Java SE 7, un seul bloc catch peut gérer une liste d’exceptions non liées. Les types d’exception sont répertoriés, séparés par un symbole de barre verticale (|). Par exemple:

try {
    doSomething();
} catch (SomeException | SomeOtherException e) {
    handleSomeException(e);
}

Le comportement d’un catch multi-exception est une simple extension pour le cas d’une seule exception. Le catch correspond si l’exception levée correspond (au moins) à l’une des exceptions répertoriées.

Il y a une subtilité supplémentaire dans la spécification. Le type de e est une union synthétique des types d’exceptions de la liste. Lorsque la valeur de e est utilisée, son type statique est le supertype le moins courant de l’union de types. Cependant, si e est relancé dans le bloc catch, les types d’exception qui sont levés sont les types de l’union. Par exemple:

public void method() throws IOException, SQLException
    try {
        doSomething();
    } catch (IOException | SQLException e) {
        report(e);
        throw e;
    }

Dans ce qui précède, IOException et SQLException sont des exceptions vérifiées dont le supertype le moins courant est Exception. Cela signifie que la méthode report doit correspondre à report(Exception). Cependant, le compilateur sait que le throw ne peut lever qu’une IOException ou une SQLException. Ainsi, method peut être déclaré comme throws IOException, SQLException plutôt que throws Exception. (Ce qui est une bonne chose : voir [Pitfall - Throwing Throwable, Exception, Error or RuntimeException][3].)

L’instruction try-with-resources #

Comme l’illustre l’exemple [try-catch-final statement][1], le nettoyage des ressources à l’aide d’une clause finally nécessite une quantité importante de code “passe-partout” pour implémenter correctement les cas extrêmes. Java 7 fournit un moyen beaucoup plus simple de traiter ce problème sous la forme de l’instruction try-with-resources.

Qu’est-ce qu’une ressource ? #

Java 7 a introduit l’interface java.lang.AutoCloseable pour permettre aux classes d’être gérées à l’aide de l’instruction try-with-resources. Les instances de classes qui implémentent AutoCloseable sont appelées ressources. Ceux-ci doivent généralement être éliminés en temps opportun plutôt que de compter sur le ramasseur de déchets pour les éliminer.

L’interface AutoCloseable définit une seule méthode :

public void close() throws Exception

Une méthode close() doit disposer de la ressource de manière appropriée. La spécification indique qu’il doit être sûr d’appeler la méthode sur une ressource qui a déjà été supprimée. De plus, les classes qui implémentent Autocloseable sont fortement encouragées à déclarer la méthode close() pour lever une exception plus spécifique que Exception, ou aucune exception du tout.

Un large éventail de classes et d’interfaces Java standard implémentent AutoCloseable. Ceux-ci inclus:

  • InputStream, OutputStream et leurs sous-classes
  • Reader, Writer et leurs sous-classes
  • Socket et ServerSocket et leurs sous-classes
  • Channel et ses sous-classes, et
  • les interfaces JDBC Connection, Statement et ResultSet et leurs sous-classes.

L’application et les classes tierces peuvent également le faire.

L’instruction try-with-resource de base #

La syntaxe d’un try-with-resources est basée sur les formes classiques try-catch, try-finally et try-catch-finally. Voici un exemple de formulaire “de base” ; c’est-à-dire le formulaire sans catch ou finally.

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

Les ressources à gérer sont déclarées en tant que variables dans la section (...) après la clause try. Dans l’exemple ci-dessus, nous déclarons une variable de ressource stream et l’initialisons avec un PrintStream nouvellement créé.

Une fois les variables de ressources initialisées, le bloc try est exécuté. Une fois cette opération terminée, stream.close() sera appelée automatiquement pour s’assurer que la ressource ne fuit pas. Notez que l’appel close() se produit quelle que soit la façon dont le bloc se termine.

Les instructions try-with-resource améliorées #

L’instruction try-with-resources peut être améliorée avec les blocs catch et finally, comme avec la syntaxe pré-Java 7 try-catch-finally. L’extrait de code suivant ajoute un bloc catch au précédent pour traiter l’exception FileNotFoundException que le constructeur PrintStream peut lancer :

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
} catch (FileNotFoundException ex) {
    System.err.println("Cannot open the file");
} finally {
    System.err.println("All done");
}

Si l’initialisation de la ressource ou le bloc try lève l’exception, alors le bloc catch sera exécuté. Le bloc finally sera toujours exécuté, comme avec une instruction try-catch-finally conventionnelle.

Il y a quelques choses à noter cependant :

  • La variable de ressource est hors de portée dans les blocs catch et finally.
  • Le nettoyage des ressources aura lieu avant que l’instruction n’essaie de correspondre au bloc catch.
  • Si le nettoyage automatique des ressources a généré une exception, cela pourrait être intercepté dans l’un des blocs catch.

Gestion de plusieurs ressources #

Les extraits de code ci-dessus montrent une seule ressource gérée. En fait, try-with-resources peut gérer plusieurs ressources dans une seule instruction. Par exemple:

try (InputStream is = new FileInputStream(file1);
     OutputStream os = new FileOutputStream(file2)) {
    // Copy 'is' to 'os'
}

Cela se comporte comme vous vous en doutez. is et os sont fermés automatiquement à la fin du bloc try. Il y a quelques points à noter :

  • Les initialisations se produisent dans l’ordre du code, et les initialiseurs de variables de ressource ultérieurs peuvent utiliser les valeurs des précédents.
  • Toutes les variables de ressource initialisées avec succès seront nettoyées.
  • Les variables de ressource sont nettoyées dans l’ordre inverse de leurs déclarations.

Ainsi, dans l’exemple ci-dessus, is est initialisé avant os et nettoyé après, et is sera nettoyé s’il y a une exception lors de l’initialisation de os.

Équivalence de try-with-ressource et try-catch-finally classique

La spécification du langage Java spécifie le comportement des formulaires try-with-resource en termes de l’instruction classique try-catch-finally. (Veuillez vous référer au JLS pour tous les détails.)

Par exemple, ce try-with-resource de base :

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

est défini comme étant équivalent à ceci try-catch-finally :

// Note that the constructor is not part of the try-catch statement
PrintStream stream = new PrintStream("hello.txt");

// This variable is used to keep track of the primary exception thrown
// in the try statement. If an exception is thrown in the try block,
// any exception thrown by AutoCloseable.close() will be suppressed.
Throwable primaryException = null;

// The actual try block
try {
    stream.println("Hello world!");
} catch (Throwable t) {
    // If an exception is thrown, remember it for the finally block
    primaryException = t;
    throw t;
} finally {
    if (primaryException == null) {
        // If no exception was thrown so far, exceptions thrown in close() will
        // not be caught and therefore be passed on to the enclosing code.
        stream.close();
    } else {
        // If an exception has already been thrown, any exception thrown in
        // close() will be suppressed as it is likely to be related to the
        // previous exception. The suppressed exception can be retrieved
        // using primaryException.getSuppressed().
        try {
            stream.close();
        } catch (Throwable suppressedException) {
            primaryException.addSuppressed(suppressedException);
        }
    }
}

(Le JLS spécifie que les variables réelles t et primaryException seront invisibles pour le code Java normal.)

La forme améliorée de try-with-resources est spécifiée comme une équivalence avec la forme de base. Par exemple:

try (PrintStream stream = new PrintStream(fileName)) {
    stream.println("Hello world!");
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

est équivalent à:

try {
    try (PrintStream stream = new PrintStream(fileName)) {
        stream.println("Hello world!");
    }
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

Exceptions personnalisées #

Dans la plupart des cas, il est plus simple, du point de vue de la conception du code, d’utiliser les classes [Exception][1] génériques existantes lors de la levée d’exceptions. Cela est particulièrement vrai si vous n’avez besoin que de l’exception pour transmettre un message d’erreur simple. Dans ce cas, RuntimeException est généralement préféré, car il ne s’agit pas d’une exception vérifiée. D’autres classes d’exception existent pour les classes d’erreurs courantes :

  • UnsupportedOperationException - une certaine opération n’est pas prise en charge
  • IllegalArgumentException - une valeur de paramètre non valide a été transmise à une méthode
  • IllegalStateException - votre API a atteint en interne une condition qui ne devrait jamais se produire, ou qui se produit en conséquence d’utiliser votre API de manière invalide

Les cas où vous voulez utiliser une classe d’exception personnalisée sont les suivants :

  • Vous écrivez une API ou une bibliothèque à utiliser par d’autres, et vous souhaitez permettre aux utilisateurs de votre API d’être en mesure d’intercepter et de gérer spécifiquement les exceptions de votre API, et * être en mesure de différencier ces exceptions d’autres exceptions plus génériques * .
  • Vous lancez des exceptions pour un type d’erreur spécifique dans une partie de votre programme, que vous souhaitez intercepter et gérer dans une autre partie de votre programme, et vous souhaitez pouvoir différencier ces erreurs des autres, plus erreurs génériques.

Vous pouvez créer vos propres exceptions personnalisées en étendant [RuntimeException][2] pour une exception non vérifiée, ou une exception vérifiée en étendant toute [Exception][3] qui n’est pas également une sous-classe de RuntimeException, car :

Les sous-classes d’Exception qui ne sont pas également des sous-classes de RuntimeException sont des exceptions vérifiées

public class StringTooLongException extends RuntimeException {
    // Exceptions can have methods and fields like other classes
    // those can be useful to communicate information to pieces of code catching
    // such an exception
    public final String value;
    public final int maximumLength;

    public StringTooLongException(String value, int maximumLength){
        super(String.format("String exceeds maximum Length of %s: %s", maximumLength, value));
        this.value = value;
        this.maximumLength = maximumLength;
    }
}

Ceux-ci peuvent être utilisés comme exceptions prédéfinies :

void validateString(String value){
    if (value.length() > 30){
        throw new StringTooLongException(value, 30);
    }
}

Et les champs peuvent être utilisés là où l’exception est interceptée et gérée :

void anotherMethod(String value){
    try {
        validateString(value);
    } catch(StringTooLongException e){
        System.out.println("The string '" + e.value + 
                "' was longer than the max of " + e.maximumLength );
    }
}

[…] Si l’on peut raisonnablement s’attendre à ce qu’un client se remette d’une exception, faites-en une exception vérifiée. Si un client ne peut rien faire pour récupérer de l’exception, faites-en une exception non contrôlée.

Gestion de l’exception interrompue #

[InterruptedException][1] est une bête déroutante - elle apparaît dans des méthodes apparemment anodines comme [Thread.sleep()][2], mais sa manipulation incorrecte conduit à un code difficile à gérer qui se comporte mal dans environnements concurrents.

À la base, si une InterruptedException est interceptée, cela signifie que quelqu’un, quelque part, a appelé [Thread.interrupt()][3] sur le thread dans lequel votre code est en cours d’exécution. Vous pourriez être enclin à dire “C’est mon code ! Je ne l’interromprai jamais !” et donc faire quelque chose comme ceci:

// Bad. Don't do this.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  // disregard
}

Mais c’est exactement la mauvaise façon de gérer un événement “impossible”. Si vous savez que votre application ne rencontrera jamais une InterruptedException, vous devez traiter un tel événement comme une violation grave des hypothèses de votre programme et quitter le plus rapidement possible.

La bonne façon de gérer une interruption “impossible” est la suivante :

// When nothing will interrupt your code
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  throw new AssertionError(e);
}

Cela fait deux choses; il restaure d’abord l’état d’interruption du thread (comme si InterruptedException n’avait pas été lancé en premier lieu), puis il lève un AssertionError indiquant que les invariants de base de votre application ont été violés. Si vous savez avec certitude que vous n’interromprez jamais le thread dans lequel ce code s’exécute, c’est sûr puisque le bloc catch ne devrait jamais être atteint.

L’utilisation de la classe [Uninterruptibles][4] de Guava aide à simplifier ce modèle ; appeler [Uninterruptibles.sleepUninterruptibly()][5] ne tient pas compte de l’état interrompu d’un thread jusqu’à ce que la durée de veille ait expiré (à ce moment-là, il est restauré pour que les appels ultérieurs inspectent et lancent leur propre InterruptedException). Si vous savez que vous n’interromprez jamais un tel code, cela évite en toute sécurité d’avoir à envelopper vos appels de sommeil dans un bloc try-catch.

Le plus souvent, cependant, vous ne pouvez pas garantir que votre fil ne sera jamais interrompu. En particulier, si vous écrivez du code qui sera exécuté par un [Executor][6] ou un autre gestionnaire de threads, il est essentiel que votre code réponde rapidement aux interruptions, sinon votre application se bloquera ou même se bloquera.

Dans de tels cas, la meilleure chose à faire est généralement de permettre à InterruptedException de se propager vers le haut de la pile des appels, en ajoutant une throws InterruptedException à chaque méthode à tour de rôle. Cela peut sembler compliqué, mais c’est en fait une propriété souhaitable - les signatures de votre méthode indiquent désormais aux appelants qu’elle répondra rapidement aux interruptions.

// Let the caller determine how to handle the interrupt if you're unsure
public void myLongRunningMethod() throws InterruptedException {
  ...
}

Dans des cas limités (par exemple, lors de la redéfinition d’une méthode qui ne “lève” aucune exception cochée), vous pouvez réinitialiser le statut interrompu sans lever d’exception, en attendant que le code soit exécuté ensuite pour gérer l’interruption. Cela retarde la gestion de l’interruption mais ne la supprime pas entièrement.

// Suppresses the exception but resets the interrupted state letting later code
// detect the interrupt and handle it properly.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  return ...; // your expectations are still broken at this point - try not to do more work.
}

Renvoie les instructions dans le bloc try catch #

Bien que ce soit une mauvaise pratique, il est possible d’ajouter plusieurs instructions de retour dans un bloc de gestion des exceptions :

 public static int returnTest(int number){
    try{
        if(number%2 == 0) throw new Exception("Exception thrown");
        else return x;
    }
    catch(Exception e){
        return 3;
    }
    finally{
        return 7;
    }
}

Cette méthode renverra toujours 7 puisque le bloc finally associé au bloc try/catch est exécuté avant que quoi que ce soit ne soit renvoyé. Maintenant, comme pour finally l’a return 7;, cette valeur remplace les valeurs de retour try/catch.

Si le bloc catch renvoie une valeur primitive et que cette valeur primitive est ensuite modifiée dans le bloc finally, la valeur renvoyée dans le bloc catch sera renvoyée et les modifications du bloc finally seront ignorées.

L’exemple ci-dessous affichera “0”, et non “1”.

public class FinallyExample {

    public static void main(String[] args) {
        int n = returnTest(4);
        
        System.out.println(n);
    }

    public static int returnTest(int number) {
        
        int returnNumber = 0; 
        
        try {
            if (number % 2 == 0)
                throw new Exception("Exception thrown");
            else
                return returnNumber;
        } catch (Exception e) {
            return returnNumber;
        } finally {
            returnNumber = 1;
        }
    }
}

La hiérarchie des exceptions Java - Exceptions non vérifiées et vérifiées #

Toutes les exceptions Java sont des instances de classes dans la hiérarchie des classes Exception. Cela peut être représenté comme suit :

  • [java.lang.Throwable][1] - Il s’agit de la classe de base pour toutes les classes d’exception. Ses méthodes et constructeurs implémentent une gamme de fonctionnalités communes à toutes les exceptions.
    • [java.lang.Exception][2] - This is the superclass of all normal exceptions.
      • various standard and custom exception classes.
      • [java.lang.RuntimeException][3] - This the superclass of all normal exceptions that are unchecked exceptions.
        • various standard and custom runtime exception classes.
    • [java.lang.Error][4] - This is the superclass of all “fatal error” exceptions.

Remarques:

  1. La distinction entre les exceptions cochées et non cochées est décrite ci-dessous.
  2. Les classes Throwable, Exception et RuntimeException doivent être traitées comme abstract ; voir https://www.wikiod.com/fr/java/pieges-java—utilisation-des-exceptionspitfall-throwing-throwable-exception-error-or-runtimeexception#t=201610240803173325134.
  3. Les exceptions “Erreur” sont levées par la JVM dans des situations où il serait dangereux ou imprudent pour une application de tenter de récupérer.
  4. Il serait imprudent de déclarer des sous-types personnalisés de Throwable. Les outils et bibliothèques Java peuvent supposer que Error et Exception sont les seuls sous-types directs de Throwable, et se comportent mal si cette hypothèse est incorrecte.

Exceptions cochées ou non cochées #

L’une des critiques de la prise en charge des exceptions dans certains langages de programmation est qu’il est difficile de savoir quelles exceptions une méthode ou une procédure donnée peut générer. Etant donné qu’une exception non gérée est susceptible de faire planter un programme, cela peut faire des exceptions une source de fragilité.

Le langage Java résout ce problème avec le mécanisme d’exception vérifié. Tout d’abord, Java classe les exceptions en deux catégories :

  • Les exceptions cochées représentent généralement des événements anticipés qu’une application doit pouvoir gérer. Par exemple, IOException et ses sous-types représentent des conditions d’erreur qui peuvent se produire dans les opérations d’E/S. Par exemple, les ouvertures de fichiers échouent car un fichier ou un répertoire n’existe pas, les lectures et écritures réseau échouent car une connexion réseau a été interrompue, etc.

  • Les exceptions non contrôlées représentent généralement des événements imprévus qu’une application ne peut pas gérer. Ceux-ci sont généralement le résultat d’un bogue dans l’application.

(Dans ce qui suit, “thrown” fait référence à toute exception levée explicitement (par une instruction throw), ou implicitement (dans un déréférencement échoué, un transtypage de type, etc.). De même, “propagé” fait référence à une exception qui a été lancée dans un appel imbriqué et non interceptée dans cet appel. L’exemple de code ci-dessous illustrera cela.)

La deuxième partie du mécanisme d’exception vérifiée est qu’il existe des restrictions sur les méthodes où une exception vérifiée peut se produire :

  • Lorsqu’une exception vérifiée est levée ou propagée dans une méthode, elle doit soit être interceptée par la méthode, soit listée dans la clause throws de la méthode. (La signification de la clause throws est décrite dans [cet exemple][5].)
  • Lorsqu’une exception vérifiée est levée ou propagée dans un bloc d’initialisation, elle doit être interceptée dans le bloc.
  • Une exception vérifiée ne peut pas être propagée par un appel de méthode dans une expression d’initialisation de champ. (Il n’y a aucun moyen d’attraper une telle exception.)

En bref, une exception vérifiée doit être soit gérée, soit déclarée.

Ces restrictions ne s’appliquent pas aux exceptions non contrôlées. Cela inclut tous les cas où une exception est levée implicitement, puisque tous ces cas lèvent des exceptions non contrôlées.

// This declares a custom checked exception.
public class MyException extends Exception {
    // constructors omitted.

        throw new MyException2(); // Compiles OK
    }
}

// CORRECTED
public void methodThrowingCheckedException(boolean flag) throws MyException {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

Le deuxième exemple montre comment une exception vérifiée propagée peut être traitée.

// INCORRECT

    // be opened.  IOException is a checked exception.
    ...
}

// CORRECTED (Version A)

    InputStream is = new FileInputStream("someFile.txt"); 
    ...
}

// CORRECTED (Version B) 
public void methodWithPropagatedCheckedException() {
    try {
        InputStream is = new FileInputStream("someFile.txt"); 
        ...
    } catch (IOException ex) {

    }
}

Le dernier exemple montre comment traiter une exception vérifiée dans un initialiseur de champ statique.

// INCORRECT
public class Test {
    private static final InputStream is = 
            new FileInputStream("someFile.txt");  // Compilation error
}

// CORRECTED
public class Test {
    private static final InputStream is;
    static {
        InputStream tmp = null;
        try {
            tmp = new FileInputStream("someFile.txt");

        }
        is = tmp;
    }
}

Introduction #

Les exceptions sont des erreurs qui se produisent lors de l’exécution d’un programme. Considérez le programme Java ci-dessous qui divise deux entiers.

class Division {
    public static void main(String[] args) {
 
        int a, b, result;
 
        Scanner input = new Scanner(System.in);
        System.out.println("Input two integers");
 
        a = input.nextInt();
        b = input.nextInt();
 
        result = a / b;
 
        System.out.println("Result = " + result);
    }
}

Maintenant, nous compilons et exécutons le code ci-dessus, et voyons le résultat d’une tentative de division par zéro :

Input two integers
7 0
Exception in thread "main" java.lang.ArithmeticException: / by zero 
    at Division.main(Disivion.java:14)

La division par zéro est une opération invalide qui produirait une valeur qui ne peut pas être représentée sous forme d’entier. Java gère cela en lançant une exception. Dans ce cas, l’exception est une instance de la classe ArithmeticException.

Remarque : L’exemple de [création et lecture de traces de pile][1] explique ce que signifie la sortie après les deux nombres.

L’utilité d’une exception est le contrôle de flux qu’elle permet. Sans utiliser d’exceptions, une solution typique à ce problème peut être de vérifier d’abord si b == 0 :

class Division {
    public static void main(String[] args) {
 
        int a, b, result;

        Scanner input = new Scanner(System.in);
        System.out.println("Input two integers");
 
        a = input.nextInt();
        b = input.nextInt();
 
        if (b == 0) {
            System.out.println("You cannot divide by zero.");
            return;
        }

        result = a / b;
 
        System.out.println("Result = " + result);
    }
}

Cela affiche le message “Vous ne pouvez pas diviser par zéro.” sur la console et quitte le programme de manière élégante lorsque l’utilisateur essaie de diviser par zéro. Une manière équivalente de traiter ce problème via la gestion des exceptions serait de remplacer le contrôle de flux if par un bloc try-catch :

...

a = input.nextInt();
b = input.nextInt();
 
try {
    result = a / b;
}
catch (ArithmeticException e) {
    System.out.println("An ArithmeticException occurred. Perhaps you tried to divide by zero.");
    return;
}
 
...

Un bloc try catch est exécuté comme suit :

  1. Commencez à exécuter le code dans le bloc try.
  2. Si une exception se produit dans le bloc try, abandonnez immédiatement et vérifiez si cette exception est interceptée par le bloc catch (dans ce cas, lorsque l’exception est une instance de ArithmeticException).
  3. Si l’exception est attrapée, elle est assignée à la variable e et le bloc catch est exécuté.
  4. Si le bloc try ou catch est terminé (c’est-à-dire qu’aucune exception non interceptée ne se produit pendant l’exécution du code), continuez à exécuter le code sous le bloc try-catch.

Il est généralement considéré comme une bonne pratique d’utiliser la gestion des exceptions dans le cadre du contrôle de flux normal d’une application où le comportement serait autrement indéfini ou inattendu. Par exemple, au lieu de renvoyer null lorsqu’une méthode échoue, il est généralement préférable de lancer une exception afin que l’application utilisant la méthode puisse définir son propre contrôle de flux pour la situation via la gestion des exceptions du genre illustré ci-dessus. Dans un certain sens, cela contourne le problème de devoir renvoyer un type particulier, car l’un des multiples types d’exceptions peut être lancé pour indiquer le problème spécifique qui s’est produit.

Pour plus de conseils sur comment et comment ne pas utiliser les exceptions, reportez-vous à https://www.wikiod.com/fr/java/pieges-java—utilisation-des-exceptions

[1] : https://www.wikiod.com/fr/java/exceptions-et-gestion-des-exceptions

Création et lecture de stacktraces #

Lorsqu’un objet d’exception est créé (c’est-à-dire lorsque vous le nouveau), le constructeur Throwable capture des informations sur le contexte dans lequel l’exception a été créée. Plus tard, ces informations peuvent être sorties sous la forme d’un stacktrace, qui peut être utilisé pour aider à diagnostiquer le problème qui a causé l’exception en premier lieu.

Impression d’un stacktrace #

L’impression d’un stacktrace consiste simplement à appeler la méthode printStackTrace(). Par exemple:

try {
    int a = 0;
    int b = 0;
    int c = a / b;
} catch (ArithmeticException ex) {
    // This prints the stacktrace to standard output
    ex.printStackTrace();
}

La méthode printStackTrace() sans arguments imprimera sur la sortie standard de l’application ; c’est-à-dire le System.out actuel. Il existe également des surcharges printStackTrace(PrintStream) et printStackTrace(PrintWriter) qui impriment sur un Stream ou Writer spécifié.

Remarques:

  1. Le stacktrace n’inclut pas les détails de l’exception elle-même. Vous pouvez utiliser la méthode toString() pour obtenir ces détails ; par exemple.

         // Print exception and stacktrace
     System.out.println(ex);
     ex.printStackTrace();

  2. L’impression Stacktrace doit être utilisée avec parcimonie ; voir https://www.wikiod.com/fr/java/pieges-java—utilisation-des-exceptions#Piège - Stacktraces excessifs ou inappropriés . Il est souvent préférable d’utiliser un framework de journalisation et de transmettre l’objet exception à journaliser.

Comprendre un stacktrace #

Considérez le programme simple suivant composé de deux classes dans deux fichiers. (Nous avons montré les noms de fichiers et ajouté des numéros de ligne à des fins d’illustration.)

File: "Main.java"
1   public class Main {
2       public static void main(String[] args) {
3           new Test().foo();
4       }
5   }

File: "Test.java"
1   class Test {
2       public void foo() {
3           bar();
4       }
5   
6       public int bar() {
7           int a = 1;
8           int b = 0;
9           return a / b;
10      }

Lorsque ces fichiers seront compilés et exécutés, nous obtiendrons la sortie suivante.

Exception in thread "main" java.lang.ArithmeticException: / by zero
        at Test.bar(Test.java:9)
        at Test.foo(Test.java:3)
        at Main.main(Main.java:3)

Lisons cette ligne une à la fois pour comprendre ce qu’elle nous dit.

La ligne 1 nous indique que le thread appelé “main” s’est terminé en raison d’une exception non interceptée. Le nom complet de l’exception est java.lang.ArithmeticException et le message d’exception est “/ par zéro”.

Si nous recherchons les javadocs pour cette exception, cela indique :

Levée lorsqu’une condition arithmétique exceptionnelle s’est produite. Par exemple, un entier “diviser par zéro” lance une instance de cette classe.

En effet, le message “/ par zéro” est un indice fort que la cause de l’exception est qu’un code a tenté de diviser quelque chose par zéro. Mais quoi?

Les 3 lignes restantes sont la trace de la pile. Chaque ligne représente un appel de méthode (ou de constructeur) sur la pile des appels, et chacune nous dit trois choses :

  • le nom de la classe et de la méthode en cours d’exécution,
  • le nom du fichier du code source,
  • le numéro de la ligne de code source de l’instruction en cours d’exécution

Ces lignes d’un stacktrace sont répertoriées avec le cadre de l’appel en cours en haut. Le cadre supérieur de notre exemple ci-dessus se trouve dans la méthode Test.bar et à la ligne 9 du fichier Test.java. C’est la ligne suivante :

    return a / b;

Si nous regardons quelques lignes plus tôt dans le fichier où b est initialisé, il est évident que b aura la valeur zéro. On peut dire sans aucun doute que c’est la cause de l’exception.

Si nous avions besoin d’aller plus loin, nous pouvons voir dans le stacktrace que bar() a été appelé depuis foo() à la ligne 3 de Test.java, et que foo() a été appelé à son tour depuis Main. principal().

Remarque : Les noms de classe et de méthode dans les cadres de pile sont les noms internes des classes et des méthodes. Vous devrez reconnaître les cas inhabituels suivants :

  • Une classe imbriquée ou interne ressemblera à “OuterClass$InnerClass”.
  • Une classe interne anonyme ressemblera à “OuterClass$1”, “OuterClass$2”, etc.
  • Lorsque le code d’un constructeur, d’un initialiseur de champ d’instance ou d’un bloc d’initialisation d’instance est en cours d’exécution, le nom de la méthode sera “”.
  • Lorsque le code d’un initialiseur de champ statique ou d’un bloc d’initialisation statique est en cours d’exécution, le nom de la méthode sera “”.

(Dans certaines versions de Java, le code de formatage stacktrace détectera et éliminera les séquences répétées de stackframe, comme cela peut se produire lorsqu’une application échoue en raison d’une récursivité excessive.)

Chaînage d’exceptions et stacktraces imbriqués #

Le chaînage d’exceptions se produit lorsqu’un morceau de code attrape une exception, puis en crée et en lève une nouvelle, en passant la première exception comme cause. Voici un exemple:

File: Test,java
1   public class Test {
2      int foo() {
3           return 0 / 0;
4      }
5
6       public Test() {
7           try {
8               foo();
9           } catch (ArithmeticException ex) {
10              throw new RuntimeException("A bad thing happened", ex);
11          }
12      }
13
14      public static void main(String[] args) {
15          new Test();
16      }
17  }

Lorsque la classe ci-dessus est compilée et exécutée, nous obtenons le stacktrace suivant :

Exception in thread "main" java.lang.RuntimeException: A bad thing happened
        at Test.<init>(Test.java:10)
        at Test.main(Test.java:15)
Caused by: java.lang.ArithmeticException: / by zero
        at Test.foo(Test.java:3)
        at Test.<init>(Test.java:8)
        ... 1 more

Le stacktrace commence par le nom de la classe, la méthode et la pile d’appels pour l’exception qui (dans ce cas) a provoqué le blocage de l’application. Elle est suivie d’une ligne “Causée par :” qui signale l’exception “cause”. Le nom de la classe et le message sont signalés, suivis des cadres de pile de l’exception de cause. La trace se termine par un “… N more” qui indique que les N dernières trames sont les mêmes que pour l’exception précédente.

Le “causé par :” n’est inclus dans la sortie que lorsque la “cause” de l’exception principale n’est pas “null”. Les exceptions peuvent être enchaînées indéfiniment, et dans ce cas, le stacktrace peut avoir plusieurs traces “causées par :”.

Remarque : le mécanisme cause n’était exposé que dans l’API Throwable de Java 1.4.0. Auparavant, le chaînage des exceptions devait être implémenté par l’application à l’aide d’un champ d’exception personnalisé pour représenter la cause et d’une méthode personnalisée “printStackTrace”.

Capturer un stacktrace en tant que chaîne #

Parfois, une application doit être capable de capturer un stacktrace en tant que String Java, afin qu’il puisse être utilisé à d’autres fins. L’approche générale pour ce faire est de créer un OutputStream ou Writer temporaire qui écrit dans un tampon en mémoire et de le transmettre à printStackTrace(...).

Les bibliothèques [Apache Commons][1] et [Guava][2] fournissent des méthodes utilitaires pour capturer un stacktrace sous forme de chaîne :

org.apache.commons.lang.exception.ExceptionUtils.getStackTrace(Throwable)

com.google.common.base.Throwables.getStackTraceAsString(Throwable)

Si vous ne pouvez pas utiliser de bibliothèques tierces dans votre base de code, utilisez la méthode suivante :

   /**
     * Returns the string representation of the stack trace.
     *
     * @param throwable the throwable
     * @return the string.
     */
    public static String stackTraceToString(Throwable throwable) {
        StringWriter stringWriter = new StringWriter();
        throwable.printStackTrace(new PrintWriter(stringWriter));
        return stringWriter.toString();
    }

Notez que si votre intention est d’analyser le stacktrace, il est plus simple d’utiliser getStackTrace() et getCause() que d’essayer d’analyser un stacktrace.

[1] : http://commons.apache.org/proper/commons-lang/javadocs/api-2.6/org/apache/commons/lang/exception/ExceptionUtils.html#getStackTrace(java.lang.Throwable) [2] : http://google.github.io/guava/releases/snapshot/api/docs/com/google/common/base/Throwables.html#getStackTraceAsString(java.lang.Throwable)

Lancer une exception #

L’exemple suivant montre les bases de la levée d’une exception :

public void checkNumber(int number) throws IllegalArgumentException {
    if (number < 0) {
        throw new IllegalArgumentException("Number must be positive: " + number);
    }
}

L’exception est levée sur la 3ème ligne. Cette déclaration peut être décomposée en deux parties :

  • new IllegalArgumentException(...) crée une instance de la classe IllegalArgumentException, avec un message décrivant l’erreur signalée par cette exception.

  • throw ... lance alors l’objet exception.

Lorsque l’exception est levée, les instructions englobantes se terminent anormalement jusqu’à ce que l’exception soit traitée. Ceci est décrit dans d’autres exemples.

Il est recommandé de créer et de lancer l’objet exception dans une seule instruction, comme indiqué ci-dessus. Il est également recommandé d’inclure un message d’erreur significatif dans l’exception pour aider le programmeur à comprendre la cause du problème. Cependant, ce n’est pas nécessairement le message que vous devriez montrer à l’utilisateur final. (Pour commencer, Java ne prend pas directement en charge l’internationalisation des messages d’exception.)

Il y a quelques points supplémentaires à faire :

  • Nous avons déclaré le checkNumber comme throws IllegalArgumentException. Ce n’était pas strictement nécessaire, car IllegalArgumentException est une exception vérifiée ; voir https://www.wikiod.com/fr/java/exceptions-et-gestion-des-exceptions#La hiérarchie des exceptions Java - Exceptions non vérifiées et vérifiées Cependant, il est recommandé de le faire, et également d’inclure les exceptions levées dans les commentaires javadoc d’une méthode.

  • Le code immédiatement après une instruction throw est injoignable. Donc si nous écrivions ceci :

       throw new IllegalArgumentException("it is bad");
   return;

    the compiler would report a compilation error for the return statement.

Chaînage d’exception #

De nombreuses exceptions standard ont un constructeur avec un deuxième argument cause en plus de l’argument message conventionnel. La cause vous permet d’enchaîner les exceptions. Voici un exemple.

Nous définissons d’abord une exception non contrôlée que notre application va lancer lorsqu’elle rencontre une erreur non récupérable. Notez que nous avons inclus un constructeur qui accepte un argument cause.

    public class AppErrorException extends RuntimeException {
        public AppErrorException() {
            super();
        }

        public AppErrorException(String message) {
            super(message);
        }

        public AppErrorException(String message, Throwable cause) {
            super(message, cause);
        }
    }

Ensuite, voici du code qui illustre le chaînage d’exceptions.

    public String readFirstLine(String file) throws AppErrorException {
        try (Reader r = new BufferedReader(new FileReader(file))) {
            String line = r.readLine();
            if (line != null) {
                return line;
            } else {
                throw new AppErrorException("File is empty: " + file);
            }
        } catch (IOException ex) {
            throw new AppErrorException("Cannot read file: " + file, ex);
        }
    }

Le throw dans le bloc try détecte un problème et le signale via une exception avec un message simple. En revanche, le throw dans le bloc catch gère le IOException en l’enveloppant dans une nouvelle exception (vérifiée). Cependant, il ne rejette pas l’exception d’origine. En passant le IOException comme cause, nous l’enregistrons afin qu’il puisse être imprimé dans le stacktrace, comme expliqué dans https://www.wikiod.com/fr/java/exceptions-et-gestion-des-exceptions 1815/création-et-lecture-stacktraces#t=20161028160848944857.

Fonctionnalités avancées des exceptions #

Cet exemple couvre certaines fonctionnalités avancées et des cas d’utilisation pour les exceptions.

Examiner la pile d’appels par programmation #

L’utilisation principale des traces de pile d’exception est de fournir des informations sur une erreur d’application et son contexte afin que le programmeur puisse diagnostiquer et résoudre le problème. Parfois, il peut être utilisé pour d’autres choses. Par exemple, une classe SecurityManager peut avoir besoin d’examiner la pile des appels pour décider si le code qui effectue un appel doit être approuvé.

Vous pouvez utiliser des exceptions pour examiner la pile d’appels par programmation comme suit :

    Exception ex = new Exception();   // this captures the call stack
    StackTraceElement[] frames = ex.getStackTrace();
    System.out.println("This method is " + frames[0].getMethodName());
    System.out.println("Called from method " + frames[1].getMethodName());

Il y a quelques mises en garde importantes à ce sujet :

  1. Les informations disponibles dans un StackTraceElement sont limitées. Il n’y a pas plus d’informations disponibles que celles affichées par printStackTrace. (Les valeurs des variables locales dans le cadre ne sont pas disponibles.)

  2. Les javadocs pour getStackTrace() indiquent qu’une JVM est autorisée à omettre des cadres :

    Some virtual machines may, under some circumstances, omit one or more stack frames from the stack trace. In the extreme case, a virtual machine that has no stack trace information concerning this throwable is permitted to return a zero-length array from this method.

Optimisation de la construction des exceptions #

Comme mentionné ailleurs, la construction d’une exception est assez coûteuse car elle implique la capture et l’enregistrement d’informations sur tous les cadres de pile sur le thread actuel. Parfois, nous savons que ces informations ne seront jamais utilisées pour une exception donnée ; par exemple. le stacktrace ne sera jamais imprimé. Dans ce cas, il existe une astuce d’implémentation que nous pouvons utiliser dans une exception personnalisée pour empêcher la capture des informations.

Les informations de cadre de pile nécessaires pour les traces de pile sont capturées lorsque les constructeurs Throwable appellent la méthode Throwable.fillInStackTrace(). Cette méthode est public, ce qui signifie qu’une sous-classe peut la remplacer. L’astuce consiste à remplacer la méthode héritée de Throwable par une méthode qui ne fait rien ; par exemple.

  public class MyException extends Exception {
      // constructors

      @Override 
      public void fillInStackTrace() {
          // do nothing
      }
  }

Le problème avec cette approche est qu’une exception qui remplace fillInStackTrace() ne peut jamais capturer le stacktrace et est inutile dans les scénarios où vous en avez besoin.

Effacer ou remplacer le stacktrace #

Dans certaines situations, le stacktrace d’une exception créée de manière normale contient soit des informations incorrectes, soit des informations que le développeur ne souhaite pas révéler à l’utilisateur. Pour ces scénarios, le Throwable.setStackTrace peut être utilisé pour remplacer le tableau d’objets StackTraceElement qui contient les informations.

Par exemple, les éléments suivants peuvent être utilisés pour supprimer les informations de la pile d’une exception :

 exception.setStackTrace(new StackTraceElement[0]);

Exceptions supprimées #

Java 7 a introduit la construction try-with-resources et le concept associé de suppression des exceptions. Considérez l’extrait suivant :

try (Writer w = new BufferedWriter(new FileWriter(someFilename))) {
    // do stuff
    int temp = 0 / 0;    // throws an ArithmeticException
}

Lorsque l’exception est levée, le try appellera close() sur le w qui videra toute sortie mise en mémoire tampon, puis fermera le FileWriter. Mais que se passe-t-il si une IOException est lancée lors du vidage de la sortie ?

Ce qui se passe, c’est que toute exception levée lors du nettoyage d’une ressource est supprimée. L’exception est interceptée et ajoutée à la liste des exceptions supprimées de l’exception principale. Ensuite, try-with-resources continuera avec le nettoyage des autres ressources. Enfin, l’exception principale sera renvoyée.

Un modèle similaire se produit si une exception est levée lors de l’initialisation de la ressource, ou si le bloc try se termine normalement. La première exception levée devient l’exception principale et les suivantes résultant du nettoyage sont supprimées.

Les exceptions supprimées peuvent être récupérées à partir de l’objet d’exception primaire en appelant getSuppressedExceptions.

Les instructions try-finally et try-catch-finally #

L’instruction try...catch...finally combine la gestion des exceptions avec le code de nettoyage. Le bloc finally contient du code qui sera exécuté en toutes circonstances. Cela les rend adaptés à la gestion des ressources et à d’autres types de nettoyage.

Essayez-enfin #

Voici un exemple du formulaire le plus simple (try...finally) :

try {
    doSomething();  
} finally {
    cleanUp();
}

Le comportement de try...finally est le suivant :

  • Le code du bloc try est exécuté.
  • Si aucune exception n’a été levée dans le bloc try :
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, control passes to the next statement after the try...finally.
  • Si une exception a été levée dans le bloc try :
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, the original exception continues to propagate.

Le code dans le bloc “finally” sera toujours exécuté. (Les seules exceptions sont si System.exit(int) est appelé, ou si la JVM panique.) Ainsi, un bloc finally est le code d’emplacement correct qui doit toujours être exécuté ; par exemple. fermer des fichiers et d’autres ressources ou libérer des verrous.

essayer-attraper-enfin #

Notre deuxième exemple montre comment catch et finally peuvent être utilisés ensemble. Cela montre également que le nettoyage des ressources n’est pas simple.

// This code snippet writes the first line of a file to a string
String result = null;
Reader reader = null;
try {
    reader = new BufferedReader(new FileReader(fileName));
    result = reader.readLine();
} catch (IOException ex) {
    Logger.getLogger.warn("Unexpected IO error", ex);  // logging the exception
} finally {
    if (reader != null) {
        try {
            reader.close();
        } catch (IOException ex) {
            // ignore / discard this exception
        }
    }
}

L’ensemble complet de comportements (hypothétiques) de `try…catch…finally’ dans cet exemple est trop compliqué pour être décrit ici. La version simple est que le code dans le bloc “finally” sera toujours exécuté.

En regardant cela du point de vue de la gestion des ressources :

  • Nous déclarons la “ressource” (c’est-à-dire la variable reader) avant le bloc try afin qu’elle soit dans la portée du bloc finally.
  • En mettant le new FileReader(...), le catch est capable de gérer toute exception IOError levée lors de l’ouverture du fichier.
  • Nous avons besoin d’un reader.close() dans le bloc finally car il y a des chemins d’exception que nous ne pouvons pas intercepter ni dans le bloc try ni dans le bloc catch.
  • Cependant, étant donné qu’une exception pourrait avoir été lancée avant l’initialisation de reader, nous avons également besoin d’un test null explicite.
  • Enfin, l’appel reader.close() pourrait (hypothétiquement) lever une exception. Nous ne nous soucions pas de cela, mais si nous n’attrapons pas l’exception à la source, nous devrons la traiter plus haut dans la pile des appels.

Java 7 et versions ultérieures fournissent une alternative [try-with-resources syntax][1] qui simplifie considérablement le nettoyage des ressources.

[1] : https://www.wikiod.com/fr/java/exceptions-et-gestion-des-exceptions#L’instruction try-with-resources

La clause ’throws’ dans une déclaration de méthode #

Le mécanisme checked exception de Java demande au programmeur de déclarer que certaines méthodes pourraient lancer des exceptions vérifiées spécifiées. Ceci est fait en utilisant la clause throws. Par exemple:

public class OddNumberException extends Exception { // a checked exception
}

public void checkEven(int number) throws OddNumberException {
    if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

Le throws OddNumberException déclare qu’un appel à checkEven pourrait lever une exception de type OddNumberException.

Une clause throws peut déclarer une liste de types et peut inclure des exceptions non vérifiées ainsi que des exceptions vérifiées.

public void checkEven(Double number) 
        throws OddNumberException, ArithmeticException {
    if (!Double.isFinite(number)) {
        throw new ArithmeticException("INF or NaN");
    } else if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

Quel est l’intérêt de déclarer des exceptions non contrôlées comme levées ?

La clause throws dans une déclaration de méthode a deux objectifs :

  1. Il indique au compilateur quelles exceptions sont levées afin que le compilateur puisse signaler les exceptions non interceptées (vérifiées) comme des erreurs.

  2. Il indique à un programmeur qui écrit du code qui appelle la méthode à quelles exceptions s’attendre. Dans ce but, il est souvent judicieux d’inclure des exceptions non contrôlées dans une liste de “jets”.

Remarque : la liste throws est également utilisée par l’outil javadoc lors de la génération de la documentation de l’API, et par les conseils de méthode “hover text” d’un IDE typique.

Lancers et remplacement de méthode #

La clause throws fait partie de la signature d’une méthode dans le but de remplacer la méthode. Une méthode de remplacement peut être déclarée avec le même ensemble d’exceptions vérifiées que celles lancées par la méthode remplacée, ou avec un sous-ensemble. Cependant, la méthode override ne peut pas ajouter d’exceptions vérifiées supplémentaires. Par exemple:

@Override
public void checkEven(int number) throws NullPointerException // OK—NullPointerException is an unchecked exception
    ...

@Override
public void checkEven(Double number) throws OddNumberException // OK—identical to the superclass
    ...

class PrimeNumberException extends OddNumberException {}
class NonEvenNumberException extends OddNumberException {}

@Override
public void checkEven(int number) throws PrimeNumberException, NonEvenNumberException // OK—these are both subclasses

@Override
public void checkEven(Double number) throws IOExcepion         // ERROR

La raison de cette règle est que si une méthode redéfinie peut lever une exception vérifiée que la méthode redéfinie ne peut pas lever, cela casserait la substituabilité de type.
