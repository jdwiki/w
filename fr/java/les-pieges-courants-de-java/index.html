Cette rubrique décrit certaines des erreurs courantes commises par les débutants en Java.

Cela inclut toutes les erreurs courantes dans l’utilisation du langage Java ou la compréhension de l’environnement d’exécution.

Les erreurs associées à des API spécifiques peuvent être décrites dans des rubriques spécifiques à ces API. Les chaînes sont un cas particulier ; ils sont couverts dans la spécification du langage Java. Des détails autres que les erreurs courantes peuvent être décrits dans cette rubrique sur les chaînes.

Piège : utiliser == pour comparer des objets wrappers primitifs tels que Integer #

(Cet écueil s’applique également à tous les types de wrapper primitifs, mais nous allons l’illustrer pour Integer et int.)

Lorsque vous travaillez avec des objets Integer, il est tentant d’utiliser == pour comparer des valeurs, car c’est ce que vous feriez avec des valeurs int. Et dans certains cas, cela semblera fonctionner :

Integer int1_1 = Integer.valueOf("1");
Integer int1_2 = Integer.valueOf(1);

System.out.println("int1_1 == int1_2: " + (int1_1 == int1_2));          // true
System.out.println("int1_1 equals int1_2: " + int1_1.equals(int1_2));   // true

Ici, nous avons créé deux objets Integer avec la valeur 1 et les comparons (dans ce cas, nous en avons créé un à partir d’une String et un à partir d’un littéral int. Il existe d’autres alternatives). De plus, nous observons que les deux méthodes de comparaison (== et equals) donnent toutes les deux true.

Ce comportement change lorsque nous choisissons des valeurs différentes :

Integer int2_1 = Integer.valueOf("1000");
Integer int2_2 = Integer.valueOf(1000);

System.out.println("int2_1 == int2_2: " + (int2_1 == int2_2));          // false
System.out.println("int2_1 equals int2_2: " + int2_1.equals(int2_2));   // true

Dans ce cas, seule la comparaison “égal” donne le résultat correct.

La raison de cette différence de comportement est que la JVM maintient un cache d’objets “Integer” pour la plage -128 à 127. (La valeur supérieure peut être remplacée par la propriété système “java.lang.Integer.IntegerCache.high” ou l’argument JVM “-XX:AutoBoxCacheMax=size”). Pour les valeurs de cette plage, Integer.valueOf() renverra la valeur mise en cache plutôt que d’en créer une nouvelle.

Ainsi, dans le premier exemple, les appels Integer.valueOf(1) et Integer.valueOf("1") ont renvoyé la même instance Integer mise en cache. En revanche, dans le deuxième exemple, Integer.valueOf(1000) et Integer.valueOf("1000") ont tous deux créé et renvoyé de nouveaux objets Integer.

L’opérateur == pour les types de référence teste l’égalité des références (c’est-à-dire le même objet). Par conséquent, dans le premier exemple, int1_1 == int1_2 est true car les références sont les mêmes. Dans le deuxième exemple, int2_1 == int2_2 est faux car les références sont différentes.

Piège : utiliser == pour comparer des chaînes #

Une erreur courante pour les débutants en Java est d’utiliser l’opérateur == pour tester si deux chaînes sont égales. Par exemple:

public class Hello {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0] == "hello") {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

Le programme ci-dessus est censé tester le premier argument de ligne de commande et afficher différents messages lorsqu’il n’est pas le mot “hello”. Mais le problème est que cela ne fonctionnera pas. Ce programme affichera “Vous sentez-vous grincheux aujourd’hui ?” quel que soit le premier argument de la ligne de commande.

Dans ce cas particulier, le String “hello” est placé dans le pool de chaînes tandis que le String args [0] réside sur le tas. Cela signifie qu’il y a deux objets représentant le même littéral, chacun avec sa référence. Puisque == teste les références, pas l’égalité réelle, la comparaison donnera un faux la plupart du temps. Cela ne signifie pas qu’il le fera toujours.

Lorsque vous utilisez == pour tester des chaînes, ce que vous testez réellement, c’est si deux objets String sont le même objet Java. Malheureusement, ce n’est pas ce que signifie l’égalité des chaînes en Java. En fait, la bonne façon de tester les chaînes est d’utiliser la méthode equals(Object). Pour une paire de chaînes, nous voulons généralement tester si elles sont constituées des mêmes caractères dans le même ordre.

public class Hello2 {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0].equals("hello")) {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

Mais ça s’aggrave en fait. Le problème est que == donne la réponse attendue dans certaines circonstances. Par exemple

public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hello";
        if (s1 == s2) {
            System.out.println("same");
        } else {
            System.out.println("different");
        }
    }
}

Fait intéressant, cela affichera “même”, même si nous testons les chaînes dans le mauvais sens. Pourquoi donc? Parce que la [Java Language Specification (Section 3.10.5: String Literals)][1] stipule que deux chaînes »literals« composées des mêmes caractères seront en fait représentées par le même objet Java. Par conséquent, le test == donnera vrai pour les littéraux égaux. (Les littéraux de chaîne sont “internés” et ajoutés à un “pool de chaînes” partagé lorsque votre code est chargé, mais il s’agit en fait d’un détail d’implémentation.)

Pour ajouter à la confusion, la spécification du langage Java stipule également que lorsque vous avez une expression constante au moment de la compilation qui concatène deux littéraux de chaîne, cela équivaut à un seul littéral. Ainsi:

    public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hel" + "lo";
        String s3 = " mum";
        if (s1 == s2) {
            System.out.println("1. same");
        } else {
            System.out.println("1. different");
        }
        if (s1 + s3 == "hello mum") {
            System.out.println("2. same");
        } else {
            System.out.println("2. different");
        }
    }
}

Cela affichera “1. identique” et “2. différent”. Dans le premier cas, l’expression + est évaluée au moment de la compilation et nous comparons un objet String avec lui-même. Dans le second cas, il est évalué au moment de l’exécution et nous comparons deux objets String différents

En résumé, utiliser == pour tester des chaînes en Java est presque toujours incorrect, mais il n’est pas garanti de donner la mauvaise réponse.

[1] : https://docs.oracle.com/javase/specs/jls/se8/html/jls-3.html#jls-3.10.5

Piège : oublier de libérer des ressources #

Chaque fois qu’un programme ouvre une ressource, comme un fichier ou une connexion réseau, il est important de libérer la ressource une fois que vous avez fini de l’utiliser. Des précautions similaires doivent être prises si une exception devait être levée lors d’opérations sur de telles ressources. On pourrait dire que le [FileInputStream][1] a un [finalizer][2] qui invoque la méthode close() sur un événement de récupération de place ; cependant, comme nous ne pouvons pas être sûrs du début d’un cycle de récupération de place, le flux d’entrée peut consommer des ressources informatiques pendant une période indéfinie. La ressource doit être fermée dans une section “finally” d’un bloc try-catch :

private static void printFileJava6() throws IOException {
    FileInputStream input;
    try {
        input = new FileInputStream("file.txt");
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    } finally {
        if (input != null) {
            input.close();
        }
    }
}

Depuis Java 7, il existe une instruction vraiment utile et intéressante introduite dans Java 7 en particulier pour ce cas, appelée try-with-resources :

private static void printFileJava7() throws IOException {
    try (FileInputStream input = new FileInputStream("file.txt")) {
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    }
}

L’instruction try-with-resources peut être utilisée avec n’importe quel objet qui implémente l’interface Closeable ou AutoCloseable. Il garantit que chaque ressource est fermée à la fin de l’instruction. La différence entre les deux interfaces est que la méthode close() de Closeable lance une IOException qui doit être gérée d’une manière ou d’une autre.

Dans les cas où la ressource a déjà été ouverte mais doit être fermée en toute sécurité après utilisation, on peut l’affecter à une variable locale à l’intérieur du try-with-resources

private static void printFileJava7(InputStream extResource) throws IOException {
    try (InputStream input = extResource) {
        ... //access resource
    }
}

La variable de ressource locale créée dans le constructeur try-with-resources est effectivement finale.

[1] : https://docs.oracle.com/javase/8/docs/api/java/io/FileInputStream.html [2] : https://en.wikipedia.org/wiki/Finalizer

Piège : tester un fichier avant d’essayer de l’ouvrir. #

Certaines personnes recommandent d’appliquer divers tests à un fichier avant d’essayer de l’ouvrir, soit pour fournir de meilleurs diagnostics, soit pour éviter de traiter des exceptions. Par exemple, cette méthode tente de vérifier si path correspond à un fichier lisible :

public static File getValidatedFile(String path) throws IOException {
    File f = new File(path);
    if (!f.exists()) throw new IOException("Error: not found: " + path);
    if (!f.isFile()) throw new IOException("Error: Is a directory: " + path);
    if (!f.canRead()) throw new IOException("Error: cannot read file: " + path);
    return f;
}

Vous pouvez utiliser la méthode ci-dessus comme ceci :

File f = null;
try {
    f = getValidatedFile("somefile");
} catch (IOException ex) {
    System.err.println(ex.getMessage());
    return;
}
try (InputStream is = new FileInputStream(file)) {
    // Read data etc.
}

Le premier problème est dans la signature de FileInputStream(File) car le compilateur insistera toujours pour que nous attrapions IOException ici, ou plus haut dans la pile.

Le deuxième problème est que les vérifications effectuées par getValidatedFile ne garantissent pas que le FileInputStream réussira.

  • Conditions de concurrence : un autre thread ou un processus distinct peut renommer le fichier, supprimer le fichier ou supprimer l’accès en lecture après le retour de getValidatedFile. Cela conduirait à une “simple” IOException sans le message personnalisé.

  • Il existe des cas extrêmes non couverts par ces tests. Par exemple, sur un système avec SELinux en mode “enforcing”, une tentative de lecture d’un fichier peut échouer même si canRead() renvoie true.

Le troisième problème est que les tests sont inefficaces. Par exemple, les appels exists, isFile et canRead effectueront chacun un [syscall][] pour effectuer la vérification requise. Un autre appel système est ensuite effectué pour ouvrir le fichier, qui répète les mêmes vérifications en coulisses.

[appel système] : https://en.wikipedia.org/wiki/System_call

En bref, les méthodes telles que getValidatedFile sont erronées. Il est préférable d’essayer simplement d’ouvrir le fichier et de gérer l’exception :

try (InputStream is = new FileInputStream("somefile")) {
    // Read data etc.
} catch (IOException ex) {
    System.err.println("IO Error processing 'somefile': " + ex.getMessage());
    return;
}

Si vous vouliez distinguer les erreurs d’E/S lancées lors de l’ouverture et de la lecture, vous pouvez utiliser un try/catch imbriqué. Si vous souhaitez produire de meilleurs diagnostics pour les échecs d’ouverture, vous pouvez effectuer les vérifications exists, isFile et canRead dans le gestionnaire.

Écueil : considérer les variables comme des objets #

Aucune variable Java ne représente un objet.

String foo;   // NOT AN OBJECT

Aucun tableau Java ne contient non plus d’objets.

String bar[] = new String[100];  // No member is an object.

Si vous considérez par erreur les variables comme des objets, le comportement réel du langage Java vous surprendra.

  • Pour les variables Java qui ont un type primitif (comme int ou float), la variable contient une copie de la valeur. Toutes les copies d’une valeur primitive sont indiscernables ; c’est-à-dire qu’il n’y a qu’une seule valeur int pour le numéro un. Les valeurs primitives ne sont pas des objets et elles ne se comportent pas comme des objets.

  • Pour les variables Java qui ont un type de référence (soit une classe ou un type de tableau), la variable contient une référence. Toutes les copies d’une référence sont indiscernables. Les références peuvent pointer vers des objets, ou elles peuvent être “nulles”, ce qui signifie qu’elles ne pointent vers aucun objet. Cependant, ce ne sont pas des objets et ils ne se comportent pas comme des objets.

Les variables ne sont pas des objets dans les deux cas, et elles ne contiennent pas d’objets dans les deux cas. Ils peuvent contenir des références à des objets, mais cela signifie quelque chose de différent.

Exemple de classe #

Les exemples qui suivent utilisent cette classe, qui représente un point dans l’espace 2D.

public final class MutableLocation {
   public int x;
   public int y;

   public MutableLocation(int x, int y) {
       this.x = x;
       this.y = y;
   }

   public boolean equals(Object other) {
       if (!(other instanceof MutableLocation) {
           return false;
       }
       MutableLocation that = (MutableLocation) other;
       return this.x == that.x && this.y == that.y;
   }
}

Une instance de cette classe est un objet qui a deux champs x et y qui ont le type int.

Nous pouvons avoir de nombreuses instances de la classe MutableLocation. Certains représenteront les mêmes emplacements dans l’espace 2D ; c’est-à-dire que les valeurs respectives de ‘x’ et ‘y’ correspondront. D’autres représenteront différents endroits.

Plusieurs variables peuvent pointer vers le même objet #

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

Dans ce qui précède, nous avons déclaré trois variables here, there et elsewhere qui peuvent contenir des références à des objets MutableLocation.

Si vous pensez (à tort) à ces variables comme étant des objets, alors vous risquez de mal interpréter les déclarations en disant :

  1. Copiez l’emplacement “[1, 2]” vers “ici”
  2. Copiez l’emplacement “[1, 2]” vers “ici”
  3. Copiez l’emplacement “[1, 2]” vers “ailleurs”

À partir de là, vous êtes susceptible de déduire que nous avons trois objets indépendants dans les trois variables. En fait, il n’y a * que deux objets créés * par ce qui précède. Les variables “ici” et “là” font référence au même objet.

Nous pouvons le démontrer. En supposant les déclarations de variables comme ci-dessus :

System.out.println("BEFORE: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);
here.x = 42;
System.out.println("AFTER: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);

Cela affichera ce qui suit :

BEFORE: here.x is 1, there.x is 1, elsewhere.x is 1
AFTER: here.x is 42, there.x is 42, elsewhere.x is 1

Nous avons attribué une nouvelle valeur à here.x et cela a changé la valeur que nous voyons via there.x. Ils font référence au même objet. Mais la valeur que nous voyons via elsewhere.x n’a pas changé, donc elsewhere doit faire référence à un objet différent.

Si une variable était un objet, alors l’affectation here.x = 42 ne changerait pas there.x.

L’opérateur d’égalité ne teste PAS que deux objets sont égaux

L’application de l’opérateur d’égalité (==) aux valeurs de référence teste si les valeurs font référence au même objet. Il ne teste pas si deux objets (différents) sont “égaux” au sens intuitif.

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

 if (here == there) {
     System.out.println("here is there");
 }
 if (here == elsewhere) {
     System.out.println("here is elsewhere");
 }

Cela imprimera “ici est là”, mais il n’imprimera pas “ici est ailleurs”. (Les références dans “ici” et “ailleurs” concernent deux objets distincts.)

En revanche, si nous appelons la méthode equals(Object) que nous avons implémentée ci-dessus, nous allons tester si deux instances MutableLocation ont un emplacement égal.

 if (here.equals(there)) {
     System.out.println("here equals there");
 }
 if (here.equals(elsewhere)) {
     System.out.println("here equals elsewhere");
 }

Cela imprimera les deux messages. En particulier, here.equals(elsewhere) renvoie true car le critère sémantique que nous avons choisi pour l’égalité de deux objets MutableLocation a été satisfait.

Les appels de méthode ne transmettent PAS d’objets du tout #

Les appels de méthode Java utilisent passer par valeur1 pour passer des arguments et renvoyer un résultat.

Lorsque vous transmettez une valeur de référence à une méthode, vous transmettez en fait une référence à un objet par valeur, ce qui signifie qu’il crée une copie de la référence d’objet.

Tant que les deux références d’objet pointent toujours vers le même objet, vous pouvez modifier cet objet à partir de l’une ou l’autre des références, et c’est ce qui cause la confusion chez certains.

Cependant, vous ne passez pas un objet par référence2. La différence est que si la copie de référence d’objet est modifiée pour pointer vers un autre objet, la référence d’objet d’origine pointera toujours vers l’objet d’origine.

void f(MutableLocation foo) {  
    foo = new MutableLocation(3, 4);   // Point local foo at a different object.
}

void g() {
    MutableLocation foo = MutableLocation(1, 2);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 1".
}

Vous ne passez pas non plus une copie de l’objet.

void f(MutableLocation foo) {  
    foo.x = 42;
}

void g() {
    MutableLocation foo = new MutableLocation(0, 0);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 42"
}

1 - Dans des langages comme Python et Ruby, le terme “passer par partage” est préféré pour “passer par valeur” d’un objet/référence.

2 - Le terme “passer par référence” ou “appeler par référence” a une signification très spécifique dans la terminologie des langages de programmation. En effet, cela signifie que vous passez l’adresse d’une variable ou d’un élément de tableau, de sorte que lorsque la méthode appelée affecte une nouvelle valeur à l’argument formel, elle modifie la valeur dans la variable d’origine. Java ne le supporte pas. Pour une description plus complète des différents mécanismes de transmission des paramètres, veuillez consulter https://en.wikipedia.org/wiki/Evaluation_strategy.

Piège : fuites de mémoire #

Java gère automatiquement la mémoire. Vous n’êtes pas obligé de libérer de la mémoire manuellement. La mémoire d’un objet sur le tas peut être libérée par un ramasse-miettes lorsque l’objet n’est plus accessible par un thread actif.

Cependant, vous pouvez empêcher la libération de mémoire en autorisant l’accès aux objets qui ne sont plus nécessaires. Que vous appeliez cela une fuite de mémoire ou un packratting de mémoire, le résultat est le même : une augmentation inutile de la mémoire allouée.

Les fuites de mémoire en Java peuvent se produire de différentes manières, mais la raison la plus courante est les références d’objets éternelles, car le ramasse-miettes ne peut pas supprimer des objets du tas tant qu’il y a encore des références à eux.

Champs statiques

On peut créer une telle référence en définissant une classe avec un champ “statique” contenant une collection d’objets, et en oubliant de définir ce champ “statique” sur “null” une fois que la collection n’est plus nécessaire. Les champs “statiques” sont considérés comme des racines GC et ne sont jamais collectés. Un autre problème concerne les fuites dans la mémoire hors tas lorsque [JNI][] est utilisé.

** Fuite du chargeur de classe **

De loin, cependant, le type de fuite de mémoire le plus insidieux est la [fuite du chargeur de classe][]. Un chargeur de classe contient une référence à chaque classe qu’il a chargée, et chaque classe contient une référence à son chargeur de classe. Chaque objet contient également une référence à sa classe. Par conséquent, si même un * seul * objet d’une classe chargée par un chargeur de classe n’est pas un ordures, pas une seule classe que ce chargeur de classe a chargée ne peut être collectée. Étant donné que chaque classe fait également référence à ses champs statiques, ils ne peuvent pas non plus être collectés.

Fuite d’accumulation L’exemple de fuite d’accumulation pourrait ressembler à ce qui suit :

final ScheduledExecutorService scheduledExecutorService = Executors.newScheduledThreadPool(1);
final Deque<BigDecimal> numbers = new LinkedBlockingDeque<>();
final BigDecimal divisor = new BigDecimal(51);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    BigDecimal number = numbers.peekLast();
    if (number != null && number.remainder(divisor).byteValue() == 0) {
        System.out.println("Number: " + number);
        System.out.println("Deque size: " + numbers.size());
    }
}, 10, 10, TimeUnit.MILLISECONDS);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    numbers.add(new BigDecimal(System.currentTimeMillis()));
}, 10, 10, TimeUnit.MILLISECONDS);

try {
    scheduledExecutorService.awaitTermination(1, TimeUnit.DAYS);
} catch (InterruptedException e) {
    e.printStackTrace();
}

Cet exemple crée deux tâches planifiées. La première tâche prend le dernier nombre d’une deque appelée numbers et, si le nombre est divisible par 51, elle imprime le nombre et la taille de la deque. La deuxième tâche met des nombres dans le deque. Les deux tâches sont planifiées à un rythme fixe et s’exécutent toutes les 10 ms.

Si le code est exécuté, vous verrez que la taille de la deque augmente en permanence. Cela finira par remplir le deque d’objets qui consomment toute la mémoire de tas disponible.

Pour éviter cela tout en préservant la sémantique de ce programme, nous pouvons utiliser une méthode différente pour prendre les nombres de la deque : pollLast. Contrairement à la méthode peekLast, pollLast renvoie l’élément et le supprime du deque alors que peekLast ne renvoie que le dernier élément.

[JNI] : https://en.wikipedia.org/wiki/Java_Native_Interface [fuite du chargeur de classe] : https://zeroturnaround.com/rebellabs/rjc201/

Piège : ne pas comprendre que String est une classe immuable #

Les nouveaux programmeurs Java oublient souvent, ou ne comprennent pas parfaitement, que la classe Java String est immuable. Cela conduit à des problèmes comme celui de l’exemple suivant :

public class Shout {
    public static void main(String[] args) {
        for (String s : args) {
            s.toUpperCase();
            System.out.print(s);
            System.out.print(" ");
        }
        System.out.println();
    }
}

Le code ci-dessus est censé imprimer les arguments de la ligne de commande en majuscules. Malheureusement, cela ne fonctionne pas, la casse des arguments n’est pas modifiée. Le problème est cette déclaration:

s.toUpperCase();

Vous pourriez penser qu’appeler toUpperCase() va changer s en une chaîne en majuscule. Ce n’est pas le cas. Ça ne peut pas ! Les objets String sont immuables. Ils ne peuvent pas être modifiés.

En réalité, la méthode toUpperCase() retourne un objet String qui est une version majuscule de la String sur laquelle vous l’appelez. Ce sera probablement un nouvel objet String, mais si s était déjà en majuscule, le résultat pourrait être la chaîne existante.

Donc, pour utiliser cette méthode efficacement, vous devez utiliser l’objet renvoyé par l’appel de méthode ; par exemple:

s = s.toUpperCase();

En fait, la règle “les chaînes ne changent jamais” s’applique à toutes les méthodes String. Si vous vous souvenez de cela, vous pouvez éviter toute une catégorie d’erreurs de débutant.

Piège : combiner affectation et effets secondaires #

Parfois, nous voyons des questions StackOverflow Java (et des questions C ou C++) qui demandent quelque chose comme ceci :

i += a[i++] + b[i--];

vaut … pour certains états initiaux connus de i, a et b.

En général:

  • pour Java, la réponse est toujours spécifiée1, mais non évidente et souvent difficile à comprendre
  • pour C et C++ la réponse est souvent indéterminée.

De tels exemples sont souvent utilisés dans les examens ou les entretiens d’embauche pour tenter de voir si l’étudiant ou la personne interrogée comprend comment l’évaluation d’expression fonctionne réellement dans le langage de programmation Java. Ceci est sans doute légitime en tant que “test de connaissances”, mais cela ne signifie pas que vous devriez jamais le faire dans un programme réel.

Pour illustrer, l’exemple apparemment simple suivant est apparu à quelques reprises dans les questions StackOverflow (comme [celui-ci] [1]). Dans certains cas, cela apparaît comme une véritable erreur dans le code de quelqu’un.

int a = 1;
a = a++;
System.out.println(a);    // What does this print.

La plupart des programmeurs (y compris les experts Java) qui lisent rapidement ces instructions diraient qu’ils renvoient “2”. En fait, il affiche ‘1’. Pour une explication détaillée des raisons, veuillez lire [cette réponse][2].

Cependant, le vrai point à retenir de cet exemple et d’exemples similaires est que toute instruction Java qui les deux attribue aux effets secondaires et la même variable sera au mieux difficile à comprendre, et au pire carrément trompeuse . Vous devriez éviter d’écrire du code comme celui-ci.

1 : problèmes potentiels modulo avec le [modèle de mémoire Java][3] si les variables ou les objets sont visibles par d’autres threads.
