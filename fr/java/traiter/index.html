Notez que l’API recommande qu’à partir de la version 1.5, la méthode préférée pour créer un processus utilise ProcessBuilder.start().

Une autre remarque importante est que la valeur de sortie produite par waitFor dépend du programme/script en cours d’exécution. Par exemple, les codes de sortie produits par calc.exe sont différents de notepad.exe.

Piège : Runtime.exec, Process et ProcessBuilder ne comprennent pas la syntaxe du shell #

Les méthodes Runtime.exec(String ...) et Runtime.exec(String) vous permettent d’exécuter une commande en tant que processus externe1. Dans la première version, vous fournissez le nom de la commande et les arguments de la commande en tant qu’éléments distincts du tableau de chaînes, et le runtime Java demande au système d’exécution du système d’exploitation de démarrer la commande externe. La deuxième version est trompeusement facile à utiliser, mais elle comporte quelques pièges.

Tout d’abord, voici un exemple d’utilisation de exec(String) en toute sécurité :

Process p = Runtime.exec("mkdir /tmp/testDir");
p.waitFor();
if (p.exitValue() == 0) {
    System.out.println("created the directory");
}

Espaces dans les noms de chemin #

Supposons que nous généralisions l’exemple ci-dessus afin de pouvoir créer un répertoire arbitraire :

Process p = Runtime.exec("mkdir " + dirPath);
// ...

Cela fonctionnera généralement, mais cela échouera si dirPath est (par exemple) “/home/user/Mes Documents”. Le problème est que exec(String) divise la chaîne en une commande et des arguments en recherchant simplement des espaces. La chaîne de commande :

"mkdir /home/user/My Documents"

sera divisé en :

"mkdir", "/home/user/My", "Documents"

et cela entraînera l’échec de la commande “mkdir” car elle attend un argument, pas deux.

Face à cela, certains programmeurs essaient d’ajouter des guillemets autour du nom de chemin. Cela ne marche pas non plus :

"mkdir \"/home/user/My Documents\""

sera divisé en :

"mkdir", "\"/home/user/My", "Documents\""

Les guillemets doubles supplémentaires qui ont été ajoutés pour tenter de « citer » les espaces sont traités comme n’importe quel autre caractère non blanc. En effet, tout ce que nous citons ou échappons aux espaces va échouer.

La façon de traiter ces problèmes particuliers est d’utiliser la surcharge exec(String ...).

Process p = Runtime.exec("mkdir", dirPath);
// ...

Cela fonctionnera si dirpath inclut des caractères d’espacement car cette surcharge de exec ne tente pas de diviser les arguments. Les chaînes sont transmises telles quelles à l’appel système “exec” du système d’exploitation.

Redirection, pipelines et autre syntaxe shell #

Supposons que nous voulions rediriger l’entrée ou la sortie d’une commande externe, ou exécuter un pipeline. Par exemple:

Process p = Runtime.exec("find / -name *.java -print 2>/dev/null");

ou

Process p = Runtime.exec("find source -name *.java | xargs grep package");

(Le premier exemple répertorie les noms de tous les fichiers Java du système de fichiers et le second imprime les instructions package2 dans les fichiers Java de l’arborescence “source”.)

Ceux-ci ne fonctionneront pas comme prévu. Dans le premier cas, la commande “find” sera exécutée avec “2>/dev/null” comme argument de commande. Il ne sera pas interprété comme une redirection. Dans le deuxième exemple, le caractère pipe ("|") et les travaux qui le suivent seront donnés à la commande “find”.

Le problème ici est que les méthodes exec et ProcessBuilder ne comprennent aucune syntaxe shell. Cela inclut les redirections, les pipelines, l’expansion des variables, le globbing, etc.

Dans quelques cas (par exemple, une redirection simple), vous pouvez facilement obtenir l’effet souhaité en utilisant ProcessBuilder. Cependant, ce n’est pas vrai en général. Une autre approche consiste à exécuter la ligne de commande dans un shell ; par exemple:

Process p = Runtime.exec("bash", "-c", 
                         "find / -name *.java -print 2>/dev/null");

ou

Process p = Runtime.exec("bash", "-c", 
                         "find source -name \\*.java | xargs grep package");

Mais notez que dans le deuxième exemple, nous devions échapper le caractère générique ("*") car nous voulons que le caractère générique soit interprété par “find” plutôt que par le shell.

Les commandes intégrées du shell ne fonctionnent pas #

Supposons que les exemples suivants ne fonctionnent pas sur un système avec un shell de type UNIX :

Process p = Runtime.exec("cd", "/tmp");     // Change java app's home directory

ou

Process p = Runtime.exec("export", "NAME=value");  // Export NAME to the java app's environment

Il y a plusieurs raisons pour lesquelles cela ne fonctionnera pas :

  1. Les commandes “cd” et “export” sont des commandes intégrées du shell. Ils n’existent pas en tant qu’exécutables distincts.

  2. Pour que les commandes intégrées du shell fassent ce qu’elles sont censées faire (par exemple, changer le répertoire de travail, mettre à jour l’environnement), elles doivent changer l’endroit où réside cet état. Pour une application normale (y compris une application Java), l’état est associé au processus d’application. Ainsi, par exemple, le processus enfant qui exécuterait la commande “cd” ne pourrait pas modifier le répertoire de travail de son processus “java” parent. De même, un processus exec ne peut pas changer le répertoire de travail d’un processus qui le suit.

Ce raisonnement s’applique à toutes les commandes intégrées du shell.

1 - Vous pouvez également utiliser ProcessBuilder, mais cela n’est pas pertinent pour cet exemple.

2 - C’est un peu approximatif… mais encore une fois, les défauts de cette approche ne sont pas pertinents pour l’exemple.

Exemple simple (version Java < 1.5) #

Cet exemple appellera la calculatrice Windows. Il est important de noter que le code de sortie varie en fonction du programme/script appelé.

package process.example;

import java.io.IOException;

public class App {

    public static void main(String[] args) {
        try {
            // Executes windows calculator
            Process p = Runtime.getRuntime().exec("calc.exe");

            // Wait for process until it terminates
            int exitCode = p.waitFor();

            System.out.println(exitCode);
        } catch (IOException e) {
            e.printStackTrace();
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}
