Plusieurs utilisations abusives du langage de programmation Java peuvent conduire un programme à générer des résultats incorrects malgré une compilation correcte. L’objectif principal de cette rubrique est de répertorier les pièges courants liés à la gestion des exceptions et de proposer la bonne manière d’éviter ces pièges.

Pitfall - Attraper Throwable, Exception, Error ou RuntimeException #

Un schéma de pensée courant pour les programmeurs Java inexpérimentés est que les exceptions sont “un problème” ou “un fardeau” et la meilleure façon de gérer cela est de les attraper toutes1 dès que possible. Cela conduit à un code comme celui-ci :

....
try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (Exception ex) {
    System.out.println("Could not open file " + fileName);
}

Le code ci-dessus a un défaut important. Le catch va en fait attraper plus d’exceptions que le programmeur ne s’y attend. Supposons que la valeur de fileName est null, en raison d’un bogue ailleurs dans l’application. Cela entraînera le constructeur FileInputStream à lancer une NullPointerException. Le gestionnaire l’attrapera et le signalera à l’utilisateur :

    Could not open file null

ce qui est inutile et déroutant. Pire encore, supposons que c’est le code “traiter l’entrée” qui a lancé l’exception inattendue (cochée ou décochée !). Désormais, l’utilisateur recevra le message trompeur d’un problème qui ne s’est pas produit lors de l’ouverture du fichier et qui peut ne pas être du tout lié aux E/S.

La racine du problème est que le programmeur a codé un gestionnaire pour Exception. C’est presque toujours une erreur :

  • Attraper Exception attrapera toutes les exceptions cochées, ainsi que la plupart des exceptions non cochées.
  • La capture de RuntimeException interceptera la plupart des exceptions non contrôlées.
  • Attraper ‘Erreur’ interceptera les exceptions non vérifiées qui signalent des erreurs internes JVM. Ces erreurs ne sont généralement pas récupérables et ne doivent pas être détectées.
  • Attraper Throwable attrapera toutes les exceptions possibles.

Le problème avec la capture d’un ensemble trop large d’exceptions est que le gestionnaire ne peut généralement pas toutes les gérer de manière appropriée. Dans le cas de Exception et ainsi de suite, il est difficile pour le programmeur de prédire ce qui pourrait être intercepté ; c’est-à-dire à quoi s’attendre.

En général, la bonne solution consiste à gérer les exceptions qui sont levées. Par exemple, vous pouvez les attraper et les manipuler sur place :

try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (FileNotFoundException ex) {
    System.out.println("Could not open file " + fileName);
}

ou vous pouvez les déclarer comme “jetés” par la méthode englobante.

Il y a très peu de situations où attraper “Exception” est approprié. Le seul qui se pose couramment est quelque chose comme ceci :

public static void main(String[] args) {
    try {
        // do stuff
    } catch (Exception ex) {
        System.err.println("Unfortunately an error has occurred. " +
                           "Please report this to X Y Z");
        // Write stacktrace to a log file.
        System.exit(1);
    }
}

Ici, nous voulons vraiment traiter toutes les exceptions, donc attraper Exception (ou même Throwable) est correct.

1 - Également connu sous le nom de [Gestion des exceptions Pokémon][1].

[1] : http://c2.com/cgi/wiki?PokemonExceptionHandling

Piège - Ignorer ou écraser les exceptions #

Cet exemple concerne le fait d’ignorer délibérément ou “d’écraser” les exceptions. Ou pour être plus précis, il s’agit de savoir comment attraper et gérer une exception d’une manière qui l’ignore. Cependant, avant de décrire comment procéder, nous devons d’abord souligner que l’écrasement des exceptions n’est généralement pas la bonne façon de les traiter.

Les exceptions sont généralement levées (par quelque chose) pour informer d’autres parties du programme qu’un événement significatif (c’est-à-dire “exceptionnel”) s’est produit. Généralement (mais pas toujours), une exception signifie que quelque chose s’est mal passé. Si vous codez votre programme pour écraser l’exception, il y a de fortes chances que le problème réapparaisse sous une autre forme. Pour aggraver les choses, lorsque vous écrasez l’exception, vous jetez les informations dans l’objet exception et sa trace de pile associée. Cela rendra probablement plus difficile la détermination de la source originale du problème.

En pratique, l’écrasement des exceptions se produit fréquemment lorsque vous utilisez la fonction de correction automatique d’un IDE pour “corriger” une erreur de compilation causée par une exception non gérée. Par exemple, vous pourriez voir un code comme celui-ci :

try {
    inputStream = new FileInputStream("someFile");
} catch (IOException e) {
    /* add exception handling code here */
}

De toute évidence, le programmeur a accepté la suggestion de l’IDE de faire disparaître l’erreur de compilation, mais la suggestion était inappropriée. (Si l’ouverture du fichier a échoué, le programme devrait très probablement faire quelque chose. Avec la “correction” ci-dessus, le programme est susceptible d’échouer plus tard ; par exemple avec une NullPointerException car inputStream est maintenant null.)

Cela dit, voici un exemple d’écrasement délibéré d’une exception. (Pour les besoins de l’argumentation, supposons que nous ayons déterminé qu’une interruption lors de l’affichage du selfie est inoffensive.) Le commentaire indique au lecteur que nous avons délibérément écrasé l’exception et pourquoi nous l’avons fait.

try {
    selfie.show();
} catch (InterruptedException e) {
    // It doesn't matter if showing the selfie is interrupted.
}

Une autre façon conventionnelle de souligner que nous écrasons délibérément une exception sans dire pourquoi est de l’indiquer avec le nom de la variable d’exception, comme ceci :

try { 
    selfie.show(); 
} catch (InterruptedException ignored) {  }

Certains IDE (comme IntelliJ IDEA) n’afficheront pas d’avertissement concernant le bloc catch vide si le nom de la variable est défini sur “ignored”.

Pitfall - Lancer Throwable, Exception, Error ou RuntimeException #

S’il est mauvais d’attraper les exceptions Throwable, Exception, Error et RuntimeException, les lancer est encore pire.

Le problème fondamental est que lorsque votre application doit gérer des exceptions, la présence d’exceptions de niveau supérieur rend difficile la distinction entre différentes conditions d’erreur. Par exemple

try {
    InputStream is = new FileInputStream(someFile);  // could throw IOException
    ...
    if (somethingBad) {
        throw new Exception();  // WRONG
    }
} catch (IOException ex) {
    System.err.println("cannot open ...");
} catch (Exception ex) {
    System.err.println("something bad happened");  // WRONG
}

Le problème est que parce que nous avons lancé une instance “Exception”, nous sommes obligés de l’attraper. Cependant, comme décrit dans un autre exemple, attraper Exception est mauvais. Dans cette situation, il devient difficile de faire la distinction entre le cas “attendu” d’une Exception qui est levée si somethingBad est true, et le cas inattendu où nous interceptons une exception non vérifiée telle que NullPointerException.

Si l’exception de niveau supérieur est autorisée à se propager, nous rencontrons d’autres problèmes :

  • Nous devons maintenant nous souvenir de toutes les différentes raisons pour lesquelles nous avons jeté le haut niveau, et les discriminer / les gérer.
  • Dans le cas de Exception et Throwable, nous devons également ajouter ces exceptions à la clause throws des méthodes si nous voulons que l’exception se propage. Ceci est problématique, comme décrit ci-dessous.

En bref, ne jetez pas ces exceptions. Lancez une exception plus spécifique qui décrit plus précisément “l’événement exceptionnel” qui s’est produit. Si nécessaire, définissez et utilisez une classe d’exception personnalisée.

Déclarer Throwable ou Exception dans les “throws” d’une méthode est problématique.

Il est tentant de remplacer une longue liste d’exceptions levées dans la clause throws d’une méthode par Exception ou même `Throwable. C’est une mauvaise idée:

  1. Il force l’appelant à gérer (ou à propager) Exception.
  2. Nous ne pouvons plus compter sur le compilateur pour nous informer des exceptions vérifiées spécifiques qui doivent être gérées.
  3. Il est difficile de gérer correctement Exception. Il est difficile de savoir quelles exceptions réelles peuvent être détectées, et si vous ne savez pas ce qui pourrait être détecté, il est difficile de savoir quelle stratégie de récupération est appropriée.
  4. La gestion de Throwable est encore plus difficile, car vous devez maintenant également faire face à des échecs potentiels dont vous ne devriez jamais vous remettre.

Ce conseil signifie que certains autres modèles doivent être évités. Par exemple:

try {
    doSomething();
} catch (Exception ex) {
    report(ex);
    throw ex;
}

Ce qui précède tente de consigner toutes les exceptions au fur et à mesure de leur passage, sans les traiter définitivement. Malheureusement, avant Java 7, l’instruction throw ex; faisait penser au compilateur que n’importe quelle Exception pouvait être levée. Cela pourrait vous obliger à déclarer la méthode englobante comme “lance une exception”. À partir de Java 7, le compilateur sait que l’ensemble d’exceptions qui pourraient y être (relevées) est plus petit.

Pitfall - Utilisation d’exceptions pour le contrôle de flux normal #

Il existe un mantra que certains experts Java ont l’habitude de réciter :

“Les exceptions ne doivent être utilisées que pour des cas exceptionnels.”

(Par exemple : http://programmers.stackexchange.com/questions/184654 )

L’essence de ceci est que c’est une mauvaise idée (en Java) d’utiliser les exceptions et la gestion des exceptions pour implémenter le contrôle de flux normal. Par exemple, comparez ces deux façons de traiter un paramètre qui pourrait être nul.

public String truncateWordOrNull(String word, int maxLength) {
    if (word == null) {
        return "";
    } else {
        return word.substring(0, Math.min(word.length(), maxLength));
    }
}

public String truncateWordOrNull(String word, int maxLength) {
    try {
        return word.substring(0, Math.min(word.length(), maxLength));
    } catch (NullPointerException ex) {
        return "";
    }
}

Dans cet exemple, nous traitons (par conception) le cas où word est null comme s’il s’agissait d’un mot vide. Les deux versions traitent de null soit en utilisant if … else et ou try … catch. Comment devrions-nous décider quelle version est la meilleure ?

Le premier critère est la lisibilité. Bien que la lisibilité soit difficile à quantifier objectivement, la plupart des programmeurs conviendraient que la signification essentielle de la première version est plus facile à discerner. En effet, pour bien comprendre la deuxième forme, vous devez comprendre qu’une NullPointerException ne peut pas être lancée par les méthodes Math.min ou String.substring.

Le deuxième critère est l’efficacité. Dans les versions de Java antérieures à Java 8, la deuxième version est nettement (ordres de grandeur) plus lente que la première version. En particulier, la construction d’un objet d’exception implique la capture et l’enregistrement des stackframes, juste au cas où le stacktrace serait requis.

D’un autre côté, il existe de nombreuses situations où l’utilisation d’exceptions est plus lisible, plus efficace et (parfois) plus correcte que l’utilisation de code conditionnel pour traiter des événements “exceptionnels”. En effet, il existe de rares situations où il est nécessaire de les utiliser pour des événements « non exceptionnels » ; c’est-à-dire des événements qui se produisent relativement fréquemment. Pour ce dernier, il vaut la peine de rechercher des moyens de réduire les frais généraux liés à la création d’objets d’exception.

Piège - Attraper l’exception interrompue #

Comme déjà souligné dans d’autres pièges, attraper toutes les exceptions en utilisant

try {
    // Some code
} catch (Exception) {
    // Some error handling
}

Livré avec beaucoup de problèmes différents. Mais un problème particulier est qu’il peut conduire à des blocages car il casse le système d’interruption lors de l’écriture d’applications multithread.

Si vous démarrez un fil, vous devez généralement également pouvoir l’arrêter brusquement pour diverses raisons.

Thread t = new Thread(new Runnable() {
    public void run() {
         while (true) {
             //Do something indefinetely
         }
    }
}

t.start();

//Do something else

// The thread should be canceld if it is still active. 
// A Better way to solve this is with a shared variable that is tested 
// regularily by the thread for a clean exit, but for this example we try to 
// forcibly interrupt this thread.
if (t.isAlive()) {
   t.interrupt();
   t.join();
}

//Continue with program

Le t.interrupt() lèvera une InterruptedException dans ce thread, qui est destiné à arrêter le thread. Mais que se passe-t-il si le Thread doit nettoyer certaines ressources avant de s’arrêter complètement ? Pour cela, il peut intercepter InterruptedException et effectuer un nettoyage.

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                //Do something indefinetely
            }
        } catch (InterruptedException ex) {
            //Do some quick cleanup

            // In this case a simple return would do. 
            // But if you are not 100% sure that the thread ends after 
            // catching the InterruptedException you will need to raise another 
            // one for the layers surrounding this code.                
            Thread.currentThread().interrupt(); 
        }
    }
}

Mais si vous avez une expression fourre-tout dans votre code, InterruptedException sera également interceptée par celle-ci et l’interruption ne se poursuivra pas. Ce qui, dans ce cas, pourrait conduire à un blocage car le thread parent attend indéfiniment que ce fil s’arrête avec t.join().

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                }
                catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Dead code as the interrupt exception was already caught in
            // the inner try-catch           
            Thread.currentThread().interrupt(); 
        }
    }
}

Il est donc préférable d’attraper les exceptions individuellement, mais si vous insistez pour utiliser un fourre-tout, attrapez au moins InterruptedException individuellement au préalable.

Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                } catch (InterruptedException ex) {
                    throw ex; //Send it up in the chain
                } catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Some quick cleanup code 
    
            Thread.currentThread().interrupt(); 
        }
    }
}

Piège - Stacktraces excessifs ou inappropriés #

L’une des choses les plus ennuyeuses que les programmeurs peuvent faire est de disperser les appels à printStackTrace() dans leur code.

Le problème est que printStackTrace() va écrire le stacktrace sur la sortie standard.

  • Pour une application destinée aux utilisateurs finaux qui ne sont pas des programmeurs Java, un stacktrace est au mieux non informatif et au pire alarmant.

  • Pour une application côté serveur, il y a de fortes chances que personne ne regarde la sortie standard.

Une meilleure idée est de ne pas appeler printStackTrace directement, ou si vous l’appelez, faites-le de manière à ce que la trace de la pile soit écrite dans un fichier journal ou un fichier d’erreur plutôt que dans la console de l’utilisateur final.

Une façon de procéder consiste à utiliser une infrastructure de journalisation et à transmettre l’objet exception en tant que paramètre de l’événement de journal. Cependant, même la journalisation de l’exception peut être nuisible si elle est effectuée de manière peu judicieuse. Considérer ce qui suit:

public void method1() throws SomeException {
    try {
        method2();
        // Do something
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method1", ex);
        throw ex;
    }
}

public void method2() throws SomeException {
    try {
        // Do something else
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method2", ex);
        throw ex;
    }
}

Si l’exception est levée dans method2, vous verrez probablement deux copies du même stacktrace dans le fichier journal, correspondant au même échec.

En bref, enregistrez l’exception ou relancez-la plus loin (éventuellement enveloppée d’une autre exception). Ne faites pas les deux.

Pitfall - Sous-classer directement Throwable #

Throwable a deux sous-classes directes, Exception et Error. Bien qu’il soit possible de créer une nouvelle classe qui étend directement Throwable, cela est déconseillé car de nombreuses applications supposent que seules Exception et Error existent.

Plus précisément, il n’y a aucun avantage pratique à sous-classer directement Throwable, car la classe résultante est, en fait, simplement une exception vérifiée. Sous-classer “Exception” à la place entraînera le même comportement, mais transmettra plus clairement votre intention.
