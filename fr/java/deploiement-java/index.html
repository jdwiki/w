Il existe une variété de technologies pour « empaqueter » les applications Java, les applications Web, etc., en vue de leur déploiement sur la plate-forme sur laquelle elles s’exécuteront. Ils vont de la simple bibliothèque ou des fichiers exécutables “JAR”, des fichiers “WAR” et “EAR”, aux programmes d’installation et aux exécutables autonomes.

Au niveau le plus fondamental, un programme Java peut être déployé en copiant une classe compilée (c’est-à-dire un fichier “.class”) ou une arborescence de répertoires contenant des classes compilées. Cependant, Java est normalement déployé de l’une des manières suivantes :

  • En copiant un fichier JAR ou une collection de fichiers JAR sur le système où ils seront exécutés ; par exemple. en utilisant javac.

  • En copiant ou téléchargeant un fichier WAR, EAR ou similaire dans un “conteneur de servlet” ou un “serveur d’application”.

  • En exécutant une sorte d’installateur d’application qui automatise ce qui précède. Le programme d’installation peut également installer un JRE intégré.

  • En mettant les fichiers JAR de l’application sur un serveur web pour permettre leur lancement à l’aide de Java WebStart.

L’exemple Création de fichiers JAR, WAR et EAR résume les différentes manières de créer ces fichiers.

Il existe de nombreux “générateurs d’installation” open source et commerciaux et Outils “générateur EXE” pour Java. De même, il existe des outils pour masquer les fichiers de classe Java (pour rendre l’ingénierie inverse plus difficile) et pour ajouter une vérification de licence d’exécution. Ceux-ci sont tous hors de portée de la documentation “Java Programming Language”.

Créer un JAR exécutable depuis la ligne de commande #

Pour créer un bocal, vous avez besoin d’un ou plusieurs fichiers de classe. Celui-ci doit avoir une méthode main s’il doit être exécuté par un double-clic.

Pour cet exemple, nous utiliserons :

import javax.swing.*;
import java.awt.Container;

public class HelloWorld {

    public static void main(String[] args) {
        JFrame f = new JFrame("Hello, World"); 
        JLabel label = new JLabel("Hello, World");
        Container cont = f.getContentPane();
        cont.add(label);
        f.setSize(400,100); 
        f.setVisible(true);
        f.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
    }

}

Il a été nommé HelloWorld.java

Ensuite, nous voulons compiler ce programme.

Vous pouvez utiliser n’importe quel programme pour ce faire. Pour exécuter à partir de la ligne de commande, voir la documentation [sur la compilation et l’exécution de votre premier programme Java.][1]

Une fois que vous avez HelloWorld.class, créez un nouveau dossier et appelez-le comme vous le souhaitez.

Créez un autre fichier appelé manifest.txt et collez-le dedans

Main-Class: HelloWorld
Class-Path: HelloWorld.jar

Mettez-le dans le même dossier avec HelloWorld.class
Utilisez la ligne de commande pour faire de votre répertoire actuel (cd C:\Your\Folder\Path\Here sous Windows) votre dossier.

Utilisez Terminal et changez de répertoire pour le répertoire (cd /Users/user/Documents/Java/jarfolder sur Mac) votre dossier

Lorsque cela est fait, tapez jar -cvfm HelloWorld.jar manifest.txt HelloWorld.class et appuyez sur Entrée. Cela crée un fichier jar (dans le dossier avec votre manifeste et HelloWorld.class) en utilisant les fichiers .class spécifiés et nommés HelloWorld.jar. Voir la section Syntaxe pour plus d’informations sur les options (comme -m et -v).
Après ces étapes, allez dans votre répertoire avec le fichier manifeste et vous devriez trouver HelloWorld.jar
Cliquer dessus devrait afficher “Hello, World” dans une zone de texte.

[1] : https://www.wikiod.com/fr/java/debuter-avec-le-langage-java

Création de fichiers JAR, WAR et EAR #

Les types de fichiers JAR, WAR et EAR sont fondamentalement des fichiers ZIP avec un fichier “manifeste” et (pour les fichiers WAR et EAR) une structure de répertoire/fichier interne particulière.

La méthode recommandée pour créer ces fichiers consiste à utiliser un outil de génération spécifique à Java qui “comprend” les exigences des types de fichiers respectifs. Si vous n’utilisez pas d’outil de construction, alors “l’exportation” de l’IDE est la prochaine option à essayer.

(Note éditoriale : les descriptions de la façon de créer ces fichiers sont mieux placées dans la documentation des outils respectifs. Mettez-les là. Veuillez faire preuve de retenue et NE PAS les mettre dans cet exemple !)

Création de fichiers JAR et WAR à l’aide de Maven #

La création d’un JAR ou d’un WAR à l’aide de Maven consiste simplement à placer l’élément <packaging> correct dans le fichier POM ; par exemple,

<packaging>jar</packaging>

ou

<packaging>war</packaging>

Pour plus de détails. Maven peut être configuré pour créer des fichiers JAR “exécutables” en ajoutant les informations requises sur la classe de point d’entrée et les dépendances externes en tant que propriétés de plug-in pour le plug-in maven jar. Il existe même un plugin pour créer des fichiers “uberJAR” qui combinent une application et ses dépendances dans un seul fichier JAR.

Introduction à Java Web Start #

Les didacticiels Oracle Java résument [Web Start] [1] comme suit :

Le logiciel Java Web Start permet de lancer des applications complètes en un seul clic. Les utilisateurs peuvent télécharger et lancer des applications, telles qu’un tableur complet ou un client de chat Internet, sans passer par de longues procédures d’installation.

Les autres avantages de Java Web Start sont la prise en charge du code signé et de la déclaration explicite des dépendances de plate-forme, ainsi que la prise en charge de la mise en cache du code et du déploiement des mises à jour d’application.

Java Web Start est également appelé JavaWS et JAWS. Les principales sources d’information sont :

  • [Les Tutoriels Java - Leçon : Java Web Start][1]
  • [Guide de démarrage Web Java][2] - [FAQ Java Web Start][2]
  • [Spécification JNLP][2]
  • [Documentation API javax.jnlp][2]
  • [Site des développeurs Java Web Start][2]

Conditions préalables #

À un niveau élevé, Web Start fonctionne en distribuant des applications Java sous forme de fichiers JAR à partir d’un serveur Web distant. Les prérequis sont :

  • Une installation Java préexistante (JRE ou JDK) sur la machine cible où l’application doit s’exécuter. Java 1.2.2 ou supérieur est requis :

    • From Java 5.0 onwards, Web Start support is included in the JRE / JDK.
    • For earlier releases, Web Start support is installed separately.
    • The Web Start infrastructure includes some Javascript that can be included in a web page to assist the user to install the necessary software.

  • Le serveur Web qui héberge le logiciel doit être accessible à la machine cible.

  • Si l’utilisateur va lancer une application Web Start à l’aide d’un lien dans une page Web, alors : - they need a compatible web browser, and - for modern (secure) browsers, they need to be told how to tell the browser to allow Java to run … without compromising web browser security.

Un exemple de fichier JNLP #

L’exemple suivant est destiné à illustrer la fonctionnalité de base de JNLP.

<?xml version="1.0" encoding="UTF-8"?>
<jnlp spec="1.0+" codebase="https://www.example.com/demo" 
    href="demo_webstart.jnlp">
    <information>
        <title>Demo</title>
        <vendor>The Example.com Team</vendor>
    </information>
    <resources>
        <!-- Application Resources -->
        <j2se version="1.7+" href="http://java.sun.com/products/autodl/j2se"/>
        <jar href="Demo.jar" main="true"/>
    </resources>
    <application-desc
         name="Demo Application"
         main-class="com.example.jwsdemo.Main"
         width="300"
         height="300">
     </application-desc>
     <update check="background"/>
</jnlp>

Comme vous pouvez le voir, un fichier JNLP basé sur XML, et les informations sont toutes contenues dans l’élément <jnlp>.

  • L’attribut spec donne la version de la spécification JNPL à laquelle ce fichier est conforme.
  • L’attribut codebase donne l’URL de base pour résoudre les URL relatives href dans le reste du fichier.
  • L’attribut href donne l’URL définitive de ce fichier JNLP.
  • L’élément <information> contient les métadonnées de l’application, notamment son titre, ses auteurs, sa description et son site Web d’aide.
  • L’élément <resources> décrit les dépendances de l’application, y compris la version Java requise, la plate-forme du système d’exploitation et les fichiers JAR.
  • L’élément <application-desc> (ou <applet-desc>) fournit les informations nécessaires au lancement de l’application.

Configuration du serveur Web #

Le serveur Web doit être configuré pour utiliser application/x-java-jnlp-file comme type MIME pour les fichiers .jnlp.

Le fichier JNLP et les fichiers JAR de l’application doivent être installés sur le serveur Web afin d’être disponibles via les URL indiquées par le fichier JNLP.

Activation du lancement via une page Web #

Si l’application doit être lancée via un lien Web, la page qui contient le lien doit être créée sur le serveur Web.

  • Si vous pouvez supposer que Java Web Start est déjà installé sur la machine de l’utilisateur, la page Web doit simplement contenir un lien pour lancer l’application. Par exemple.

     <a href="https://www.example.com/demo_webstart.jnlp">Launch the application</a>

  • Sinon, la page doit également inclure des scripts pour détecter le type de navigateur utilisé par l’utilisateur et demander de télécharger et d’installer la version requise de Java.

REMARQUE : C’est une mauvaise idée d’encourager les utilisateurs à installer Java de cette façon, ou même d’activer Java dans leurs navigateurs Web afin que le lancement de la page Web JNLP fonctionne.

Lancement des applications Web Start à partir de la ligne de commande

Les instructions pour lancer une application Web Start à partir de la ligne de commande sont simples. En supposant que l’utilisateur dispose d’un JRE ou d’un JDK Java 5.0, il suffit d’exécuter ceci :

$ javaws <url>

où <url> est l’URL du fichier JNLP sur le serveur distant.
