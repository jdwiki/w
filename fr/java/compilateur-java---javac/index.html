La commande javac est utilisée pour compiler les fichiers source Java en fichiers bytecode. Les fichiers de bytecode sont indépendants de la plate-forme. Cela signifie que vous pouvez compiler votre code sur un type de matériel et de système d’exploitation, puis exécuter le code sur n’importe quelle autre plate-forme prenant en charge Java.

La commande javac est incluse dans les distributions du kit de développement Java (JDK).

Le compilateur Java et le reste de la chaîne d’outils Java standard imposent les restrictions suivantes au code :

  • Le code source est contenu dans des fichiers avec le suffixe “.java”
  • Les bytecodes sont conservés dans des fichiers avec le suffixe “.class”
  • Pour les fichiers source et bytecode du système de fichiers, les noms de chemin d’accès aux fichiers doivent refléter le nom du package et de la classe.

Remarque : Le compilateur “javac” ne doit pas être confondu avec le https://www.wikiod.com/fr/java/compilateur-juste-a-temps-jit qui compile les bytecodes en code natif.

La commande ‘javac’ - mise en route #

Exemple simple #

En supposant que “HelloWorld.java” contient la source Java suivante :

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

(Pour une explication du code ci-dessus, veuillez vous référer à https://www.wikiod.com/fr/java/debuter-avec-le-langage-java .)

Nous pouvons compiler le fichier ci-dessus en utilisant cette commande :

$ javac HelloWorld.java

Cela produit un fichier appelé “HelloWorld.class”, que nous pouvons ensuite exécuter comme suit :

$ java HelloWorld
Hello world!

Les points clés à noter dans cet exemple sont :

  1. Le nom du fichier source “HelloWorld.java” doit correspondre au nom de la classe dans le fichier source … qui est HelloWorld. S’ils ne correspondent pas, vous obtiendrez une erreur de compilation.
  2. Le nom de fichier bytecode “HelloWorld.class” correspond au nom de classe. Si vous deviez renommer “HelloWorld.class”, vous obtiendrez une erreur lorsque vous essaierez de l’exécuter.
  3. Lors de l’exécution d’une application Java à l’aide de java, vous fournissez le nom de classe PAS le nom de fichier bytecode.

Exemple avec des packages #

Le code Java le plus pratique utilise des packages pour organiser l’espace de noms des classes et réduire le risque de collision accidentelle de noms de classes.

Si nous voulions déclarer la classe HelloWorld dans un appel de package com.example, le “HelloWorld.java” contiendrait la source Java suivante :

package com.example;

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

Ce fichier de code source doit être stocké dans une arborescence de répertoires dont la structure correspond à la dénomination du package.

.    # the current directory (for this example)
|
 ----com
     |
      ----example
          |
           ----HelloWorld.java

Nous pouvons compiler le fichier ci-dessus en utilisant cette commande :

$ javac com/example/HelloWorld.java

Cela produit un fichier appelé “com/example/HelloWorld.class” ; c’est-à-dire qu’après compilation, la structure du fichier devrait ressembler à ceci :

.    # the current directory (for this example)
|
 ----com
     |
      ----example
          |
           ----HelloWorld.java
           ----HelloWorld.class

Nous pouvons alors lancer l’application comme suit :

$ java com.example.HelloWorld
Hello world!

Les points supplémentaires à noter dans cet exemple sont :

  1. La structure du répertoire doit correspondre à la structure du nom du package.
  2. Lorsque vous exécutez la classe, le nom complet de la classe doit être fourni ; c’est-à-dire “com.example.HelloWorld” et non “HelloWorld”.
  3. Vous n’avez pas besoin de compiler et d’exécuter du code Java à partir du répertoire courant. Nous ne le faisons ici qu’à titre d’illustration.

Compiler plusieurs fichiers à la fois avec ‘javac’. #

Si votre application se compose de plusieurs fichiers de code source (et la plupart le font !), vous pouvez les compiler un par un. Alternativement, vous pouvez compiler plusieurs fichiers en même temps en répertoriant les noms de chemin :

$ javac Foo.java Bar.java

ou en utilisant la fonctionnalité générique de nom de fichier de votre shell de commande ….

$ javac *.java
$ javac com/example/*.java
$ javac */**/*.java #Only works on Zsh or with globstar enabled on your shell

Cela compilera tous les fichiers source Java dans le répertoire courant, dans le répertoire “com/example”, et de manière récursive dans les répertoires enfants respectivement. Une troisième alternative consiste à fournir une liste de noms de fichiers source (et d’options du compilateur) sous forme de fichier. Par exemple:

$ javac @sourcefiles

où le fichier sourcefiles contient :

Foo.java
Bar.java
com/example/HelloWorld.java

Remarque : la compilation d’un code comme celui-ci est appropriée pour les petits projets d’une personne et pour les programmes ponctuels. Au-delà, il est conseillé de sélectionner et d’utiliser un outil de build Java. Alternativement, la plupart des programmeurs utilisent un IDE Java (par exemple [NetBeans] [1], [eclipse] [2], [IntelliJ IDEA] [3]) qui offre un compilateur intégré et une construction incrémentielle de “projets”.

Options ‘javac’ couramment utilisées #

Voici quelques options pour la commande javac susceptibles de vous être utiles

  • L’option -d définit un répertoire de destination pour l’écriture des fichiers “.class”.
  • L’option -sourcepath définit un chemin de recherche du code source.
  • L’option -cp ou -classpath définit le chemin de recherche pour trouver des classes externes et précédemment compilées. Pour plus d’informations sur le chemin de classe et comment le spécifier, reportez-vous à la rubrique https://www.wikiod.com/fr/java/le-chemin-de-classe
  • L’option -version imprime les informations de version du compilateur.

Une liste plus complète des options du compilateur sera décrite dans un exemple séparé.

Compilation pour une autre version de Java #

Le langage de programmation Java (et son environnement d’exécution) a subi de nombreux changements depuis sa sortie depuis sa sortie publique initiale. Ces modifications incluent :

  • Modifications de la syntaxe et de la sémantique du langage de programmation Java
  • Modifications des API fournies par les bibliothèques de classes standard Java.
  • Modifications du jeu d’instructions Java (bytecode) et du format de fichier de classe.

À quelques exceptions près (par exemple, le mot-clé enum, les modifications apportées à certaines classes “internes”, etc.), ces modifications sont rétrocompatibles.

  • Un programme Java qui a été compilé à l’aide d’une ancienne version de la chaîne d’outils Java s’exécutera sur une plate-forme Java plus récente sans recompilation.
  • Un programme Java qui a été écrit dans une ancienne version de Java se compilera avec succès avec un nouveau compilateur Java.

Compiler l’ancien Java avec un compilateur plus récent #

Si vous avez besoin de (re-)compiler du code Java plus ancien sur une plate-forme Java plus récente pour l’exécuter sur la plate-forme la plus récente, vous n’avez généralement pas besoin de donner d’indicateurs de compilation spéciaux. Dans quelques cas (par exemple, si vous avez utilisé enum comme identifiant), vous pouvez utiliser l’option -source pour désactiver la nouvelle syntaxe. Par exemple, étant donné la classe suivante :

public class OldSyntax {
    private static int enum;  // invalid in Java 5 or later
}

les éléments suivants sont requis pour compiler la classe à l’aide d’un compilateur Java 5 (ou version ultérieure) :

$ javac -source 1.4 OldSyntax.java

Compiler pour une ancienne plate-forme d’exécution #

Si vous avez besoin de compiler Java pour l’exécuter sur une plate-forme Java plus ancienne, l’approche la plus simple consiste à installer un JDK pour la version la plus ancienne que vous devez prendre en charge et à utiliser le compilateur de ce JDK dans vos versions.

Vous pouvez également compiler avec un compilateur Java plus récent, mais c’est compliqué. Tout d’abord, il y a quelques conditions préalables importantes qui doivent être remplies :

  • Le code que vous compilez ne doit pas utiliser de constructions de langage Java qui n’étaient pas disponibles dans la version de Java que vous ciblez.
  • Le code ne doit pas dépendre de classes, champs, méthodes, etc. standard Java qui n’étaient pas disponibles sur les anciennes plates-formes.
  • Les bibliothèques tierces dont dépend le code doivent également être construites pour l’ancienne plate-forme et disponibles au moment de la compilation et de l’exécution.

Étant donné que les conditions préalables sont remplies, vous pouvez recompiler le code pour une plate-forme plus ancienne en utilisant l’option -target. Par exemple,

$ javac -target 1.4 SomeClass.java

compilera la classe ci-dessus pour produire des bytecodes compatibles avec Java 1.4 ou JVM ultérieur. (En fait, l’option -source implique un -target compatible, donc javac -source 1.4 ... aurait le même effet. La relation entre -source et -target est décrite dans le Documentation Oracle.)

Cela dit, si vous utilisez simplement -target ou -source, vous compilerez toujours avec les bibliothèques de classes standard fournies par le JDK du compilateur. Si vous ne faites pas attention, vous pouvez vous retrouver avec des classes avec la bonne version de bytecode, mais avec des dépendances sur des API qui ne sont pas disponibles. La solution consiste à utiliser l’option -bootclasspath. Par exemple:

$ javac -target 1.4 --bootclasspath path/to/java1.4/rt.jar SomeClass.java

compilera avec un ensemble alternatif de bibliothèques d’exécution. Si la classe en cours de compilation a des dépendances (accidentelles) sur des bibliothèques plus récentes, cela vous donnera des erreurs de compilation.
