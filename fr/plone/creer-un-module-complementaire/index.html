Une histoire sur la création d’un add-on Plone #

Préambule #

TL;TR : à des fins d’illustration, les lignes de ce document commençant par ‘TL;TR:’ est suivi d’une ligne de commande faisant référence aux scripts d’assistance Plone du oeuf ‘adi.devgen’. Si vous exécutez la commande donnée après ‘TL;TR:’, cela créer tous les fichiers expliqués du chapitre suivant. Il existe de nombreux outils de script d’assistance alternatifs, mais au cas où vous voudriez utilisez cet outil, vous pouvez l’installer rapidement comme ceci de la ligne de commande :

pip install adi.devgen

Structure d’un module complémentaire Plone minimal #

TL;TR : devgen addBase mynamespace.myaddon

Tout comme Plone se compose de plus de 200 œufs Python, un module complémentaire Plone est également un Python-egg et un œuf est un module Python à espace de noms. N’importe quel œuf sera disponible à l’interpréteur Python de ZOPE. Pour rendre un egg installable sur un site Plone, il a également besoin d’un soi-disant ‘profil’. Ce qui suit expliquera ces termes.

Module Python #

Un module Python est un répertoire qui contient au moins un fichier qui doit être nommé __init__.py. Le nom est un indicateur permettant à l’interpréteur Python de reconnaître ce répertoire comme un module, ce qui signifie que son chemin peut être importé d’autres Python-scripts, par ex. en supposant que nous ayons un répertoire appelé ‘myaddon’, qui contient un fichier appelé ‘monfichier’ et ce fichier contient une définition appelée ‘myDefinition’, vous pouvez importer la définition d’un autre fichier Python comme ceci :

from myaddon.myfile import myDefinition

Oeuf de Python #

Un Python-egg est un module Python à espace de noms, un espace de noms peut être n’importe quel nom, mais il il vaut mieux ne pas être déjà pris, au cas où vous voudriez partager votre add-on avec le monde.

La raison de l’espacement des noms est avant tout d’exclure la possibilité que deux ou plusieurs modules pourraient avoir le même nom (dans cet exemple appelé ‘myaddon’), car il entraînerait des conflits lorsqu’un script Python essaie d’importer le chemin, étant n’est plus unique.

Voici un aperçu de la structure des répertoires et des fichiers concernés, avant de continuer à expliquer leurs objectifs en détail plus loin :

mynamespace.myaddon
    setup.py
    mynamespace
        __init__.py
        myaddon
            __init__.py
            myfile.py

Pour créer un espace de noms pour un add-on, créez d’abord un répertoire nommé ‘mynamespace.myaddon’.

Ce répertoire doit contenir le fichier ‘setup.py’, il rend l’oeuf enregistrable à l’interpréteur Python de l’instance ZOPE, cela est expliqué plus en détail dans le chapitre suivant[TODO].

Ensuite, dans ce répertoire, créez un autre répertoire nommé d’après votre espace de noms : ‘mynamespace’ Maintenant, nous faisons de ce répertoire un chemin de module importable, en ajoutant le fichier nommé par mot-clé __init__.py dedans. Il doit également contenir les éléments suivants lignes, qui font que la magie de l’espace de noms se produit, expliquée dans le dernier paragraphe :

[TODO: namespace-magic-lines]

Ouf presque fini, enfin dans ce nouveau répertoire on met le module du précédent chapitre : Le répertoire ‘myaddon’, contenant le ‘init.py’ et ‘myfile.py’ avec la ‘maDéfinition’.

Nous avons maintenant un œuf enregistrable pour l’instance ZOPE et nous pourrions référencer son méthodes de tout autre œuf enregistré, en utilisant cette notation de chemin :

from mynamespace.myaddon.myfile import myDefinition

Vous pourriez voir la magie, les deux premiers répertoires sont omis, ce n’est pas :

from mynamespace.myaddon.mynamespace.myaddon.myfile import myDefinition

Rendre un addon installable dans un site Plone #

Un “profil” fera apparaître le module complémentaire dans le panneau de configuration des modules complémentaires d’un site, afin qu’un administrateur puisse le (dés-)activer pour le site.

De plus, comme un site Plone est toujours un enfant de l’instance ZOPE et un ZOPE-instance peut contenir plusieurs sites, nous ne voudrions peut-être pas avoir composants involontairement de notre add-on installés sur d’autres sites Plone, c’est pourquoi nous les lions à un ‘profil’.

TL;TR : devgen addProfile mynamespace.myaddon

Cela nous amène aux parties vraiment intéressantes, la ZOPE-Component-Architecture, contrôlé par les fichiers se terminant par ‘.zcml’, qui signifie ‘ZOPE-Component- Langage de balisage’. Avec lui, vous pouvez enregistrer un profil, lier des vues à des interfaces et beaucoup plus. En fait, il mérite un chapitre majeur propre et un sous-chapitre pour chacun des répertoires appelés “profil”, “skins”, “navigateur” et “contenu”.

Vue d’ensemble de la structure des fichiers #

Squelette minimal #

mynamespace.myaddon     | The container-directory of the add-on.

    setup.py            | Register this directory to the Python-interpreter of
                        | the ZOPE-instance.

    mynamespace         | The namespace-directory of the add-on.

        __init__.py     | Makes this directory a Python-module and contains
                        | the lines for the namespace-magic.

        myaddon         | The base-directory of the add-on.

            __init__.py | Empty file, make this directory a Python-module.

            myfile.py   | Example-file: If it contained a method called
                        | 'myMethod', it would be importable of any other
                        | registered egg like this:
                        |    from mynamespace.myaddon.myfile import myMethod

Composants optionnels #

Le répertoire ‘mynamespace.myaddon/mynamespace/myaddon’ peut contenir :

configure.zcml  | Register all following files and directories, here.

setuphandlers.py | Make an addon available to ZOPE-instance.

profiles        | Make an addon installable to Plone-site and generally
                | hold GenericSetup-xml-files.

skins           | The "old-school"-way, where resources like templates,
                | stylesheets, etc., live.

browser         | The "new-school"-way, where resources live.

content         | Holds the Python-files for defining custom content-types.

subscriber.py   | Subscribers listen to events, describe here what to do
                | when an event happens, e.g.: "An item has been added to the
                | site, notify admin about it via e-mail-notifica."
