Syntaxe #

  • class MyClass<T1, T2 étend CharSequence> implémente Comparable<MyClass> //...
  • interface MaListeInterface<T étend sérialisable> étend la liste<T> //...
  • public <T1, T2 étend Instant> T1 provideClone(T1 toClone, T2 instant> //...
  • public static List<CharSequence> safe(Collection<? extend CharSequence> l) { return new ArrayList<>(l);}
  • Set<String> strings = Collections.singleton("Hello world");
  • List<CharSequence> chsList = safe(strings);

L’effacement de type limite la réflexion, bien que cela ne soit pas spécifique à JVM, par exemple Ceylon [utilise des génériques réifiés](http://ceylon-lang.org/documentation/1.2/spec /html/execution.html#réification).

La prise en charge de type existentiel n’est pas nécessairement prise en charge par d’autres langages sous cette forme : Kotlin [la prend en charge via des projections de type](https://kotlinlang.org/docs/reference/java-interop. html#java-generics-in-kotlin).

Introduction #

Les génériques ont été introduits en Java dans sa version (1.)5. Ceux-ci sont effacés lors de la compilation, de sorte que la réflexion à l’exécution n’est pas possible pour eux. Les génériques génèrent de nouveaux types paramétrés par d’autres types. Par exemple, nous n’avons pas besoin de créer de nouvelles classes afin d’utiliser une collection sécurisée de type de Strings et Numbers, ArrayList<T> générique peut être utilisé dans tous les cas, comme : new ArrayList<String> ().

Exemple:

List<String> variable = new ArrayList<String>();

Dans Java 7, du sucre syntaxique a été introduit pour faciliter la construction (<> alias diamant):

List<String> variable = new ArrayList<>();

Fait intéressant, il était également possible (à partir de Java 5) d’utiliser l’inférence de type, lorsqu’une méthode statique avait comme valeur de retour (souvent utilisée dans Google Guava par exemple) :

List<String> singleton = Collections.singletonList();//Note the missing `<>` or `<String>`!

En Java, les types existentiels étaient utilisés pour fournir un polymorphisme aux types, car les types génériques sont invariants (par exemple : List<String> n’est pas un sous-type, ni un supertype de List<CharSequence>, bien qu’en Java String [] est un sous-type de CharSequence[] ; remarque : String implémente l’interface CharSequence). Les types génériques existentiels peuvent être exprimés comme suit :

List<? extends CharSequence> list = new ArrayList<String>();
Comparable<? super ChronoLocalDate> ccld = LocalDate.now();
ChronoLocalDate cld = JapaneseDate.now(); //ChronoLocalDate extends Comparable<ChronoLocalDate>
ccld.compareTo(cld);
//cld.compareTo(ccld);//fails to compile because ccld is not a `ChronoLocalDate` (compile time)

Les deux instances peuvent être utilisées dans une liste paramétrée par le “Comparable” correspondant :

List<Comparable<? super ChronoLocalDate>> list2 = new ArrayList<>();
list2.add(cld);
list2.add(ccld);

Méthodes génériques #

Les type parameters génériques sont généralement définis au niveau de la classe ou de l’interface, mais methods et (rarement) constructors prennent également en charge la déclaration de paramètres de type liés à la portée d’un seul appel de méthode.

class Utility // no generics at the class level
{
    @SafeVarargs
    public static <T> T randomOf(T first, T... rest) {
        int choice = new java.util.Random().nextInt(rest.length + 1);
        return choice == rest.length ? first : rest[choice];
    }

    public static <T extends Comparable<T>> T max(T t1, T t2) {
        return t1.compareTo(t2) < 0 ? t2 : t1;
    }
}

Notez que les déclarations de paramètre de type, T et <T extend Comparable<T>> respectivement, apparaissent après les modificateurs de méthode et avant le type de retour. Cela permet au paramètre de type “T” d’être utilisé dans le cadre de telles méthodes, agissant comme :

  • types d’arguments
  • type de retour
  • types de variables locales

Bien que les deux méthodes ci-dessus utilisent le même nom de paramètre de type “T”, au niveau de la méthode, elles sont complètement indépendantes l’une de l’autre. Le compilateur va inférer le type réel en fonction des arguments passés à la méthode à chaque site d’appel qui invoque la méthode. Étant donné que la méthode max déclare que T étend Comparable<T>, le compilateur applique également que les types déduits sont des implémentations compatibles de l’interface Comparable.

Integer num1 = 1;
Integer num2 = 2;
String str1 = "abc";
String str2 = "xyz";

Integer bigger = Utility.max(num1, num2);
assert bigger == num2;

String later = Utility.max(str2, str1);
assert later == str2;

Utility.max(num1, str1); // compiler error: num1 and str1 are incompatible types

Utility.max(new Object(), new Object()); // compiler error: Object does not implement Comparable

Java 8 a considérablement amélioré la capacité du compilateur à déduire correctement les types génériques sur les sites d’appel. Si le compilateur ne parvient pas à déduire le type approprié, les développeurs peuvent indiquer explicitement le type dans le cadre de l’appel :

Object obj = Utility.<Object>randomOf(str1, new Object(), num1);
