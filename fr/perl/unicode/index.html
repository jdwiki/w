Un avertissement sur l’encodage des noms de fichiers #

Il convient de mentionner que l’encodage des noms de fichiers n’est pas seulement spécifique à la plate-forme, mais également au système de fichiers.

Il n’est jamais * entièrement * sûr de supposer (mais c’est souvent le cas) que, simplement parce que vous pouvez encoder et écrire dans un nom de fichier donné, lorsque vous essayez plus tard d’ouvrir ce même nom de fichier pour le lire, il s’appellera toujours la même chose.

Par exemple, si vous écrivez sur un système de fichiers tel que FAT16 qui ne prend pas en charge l’unicode, vos noms de fichiers peuvent être traduits silencieusement dans des formes compatibles ASCII.

Mais il est encore * moins * sûr de supposer qu’un fichier que vous pouvez créer, lire et écrire par un nom explicite sera appelé de la même manière lorsqu’il sera interrogé via d’autres appels, par exemple, readdir peut renvoyer des octets différents pour votre nom de fichier que vous avez spécifié open.

Sur certains systèmes tels que VAX, vous ne pouvez même pas toujours supposer que readdir renverra le même nom de fichier que vous avez spécifié avec open pour des noms de fichiers aussi simples que foo.bar, car le nom de fichier extensions peut être mutilé par le OS.

De plus, sous UNIX, il existe un ensemble très libéral de caractères légaux pour les noms de fichiers autorisés par le système d’exploitation, à l’exclusion uniquement de / et \0, alors que sous Windows, il existe des plages spécifiques de caractères qui sont interdites dans les noms de fichiers et qui provoquer des erreurs.

Faites preuve de beaucoup de prudence ici, évitez les astuces fantaisistes avec les noms de fichiers si vous avez le choix, et faites toujours des tests pour vous assurer que toutes les astuces fantaisistes que vous faites utiliser sont cohérentes.

Faites preuve d’autant plus de prudence si vous écrivez du code destiné à être exécuté sur des plates-formes indépendantes de votre volonté, par exemple si vous écrivez du code destiné au “CPAN”, et supposez qu’au moins 5 % de votre utilisateur base sera bloquée en utilisant une technologie ancienne ou cassée, soit par choix, soit par accident, soit par des pouvoirs hors de leur contrôle, et que ceux-ci conspireront pour créer des bogues pour eux.

:encodage(utf8) vs :utf8 #

Comme UTF-8 est l’un des formats internes de représentation des chaînes en Perl, l’étape d’encodage/décodage peut souvent être ignorée. Au lieu de :encoding(utf-8), vous pouvez simplement utiliser :utf8, si vos données sont déjà en UTF-8. :utf8 peut être utilisé en toute sécurité avec les flux de sortie, alors que pour le flux d’entrée, cela peut être dangereux, car il provoque une incohérence interne lorsque vous avez des séquences d’octets invalides. De plus, l’utilisation de :utf8 pour l’entrée peut entraîner des failles de sécurité, c’est pourquoi l’utilisation de :encoding(utf-8) est conseillée.

Plus de détails : Quelle est la différence entre :encoding et :utf8

UTF-8 contre utf8 contre UTF8 #

Depuis Perl v5.8.7, "UTF-8" (avec tiret) signifie UTF-8 dans sa forme stricte et soucieuse de la sécurité, tandis que "utf8" signifie UTF-8 dans sa forme libérale et lâche.

Par exemple, "utf8" peut être utilisé pour des points de code qui n’existent pas en Unicode, comme 0xFFFFFFFF. De même, les séquences d’octets UTF-8 invalides comme "\x{FE}\x{83}\x{BF}\x{BF}\x{BF}\x{BF}\x{BF}" seront décodées dans un point de code Unicode invalide (mais Perl valide) (0xFFFFFFFF) lors de l’utilisation de "utf8", alors que l’encodage "UTF-8" ne permettrait pas le décodage vers des points de code en dehors de la plage d’Unicode valide et vous donnerait un caractère de substitution (0xFFFD) à la place.

Étant donné que les noms d’encodage ne sont pas sensibles à la casse, "UTF8" est identique à "utf8" (c’est-à-dire une variante non stricte).

Le pragma utf8 : utiliser Unicode dans vos sources #

Le pragma utf8 indique que le code source sera interprété comme UTF-8. Bien sûr, cela ne fonctionnera que si votre éditeur de texte enregistre également la source encodée en UTF-8.

Désormais, les littéraux de chaîne peuvent contenir des caractères Unicode arbitraires ; les identifiants peuvent également contenir Unicode mais uniquement des caractères de type mot (voir perldata et [perlrecharclass](http://perldoc.perl.org/ perlrecharclass.html#Backslash-sequences) pour plus d’informations) :

use utf8;
my $var1 = '§я§©😄';      # works fine
my $я = 4;                # works since я is a word (matches \w) character
my $p§2 = 3;              # does not work since § is not a word character.
say "ya" if $var1 =~ /я§/; # works fine (prints "ya")

Remarque : lors de l’impression de texte sur le terminal, assurez-vous qu’il prend en charge UTF-8.*

Il peut y avoir des relations complexes et contre-intuitives entre le codage de sortie et de source. S’exécutant sur un terminal UTF-8, vous pouvez constater que l’ajout du pragma utf8 semble casser les choses :

$ perl -e 'print "Møøse\n"'
Møøse
$ perl -Mutf8 -e 'print "Møøse\n"'
M��se
$ perl -Mutf8 -CO -e 'print "Møøse\n"'
Møøse

Dans le premier cas, Perl traite la chaîne comme des octets bruts et les imprime ainsi. Comme ces octets sont en UTF-8 valides, ils semblent corrects même si Perl ne sait pas vraiment de quels caractères il s’agit (par exemple, length("Møøse") renverra 7, pas 5). Une fois que vous avez ajouté -Mutf8, Perl décode correctement la source UTF-8 en caractères, mais la sortie est en mode Latin-1 par défaut et l’impression Latin-1 sur un terminal UTF-8 ne fonctionne pas. Ce n’est que lorsque vous basculez STDOUT sur UTF-8 en utilisant -CO que la sortie sera correcte.

use utf8 n’affecte pas l’encodage d’E/S standard ni les descripteurs de fichiers !

Créer des noms de fichiers #

Les exemples suivants utilisent le codage UTF-8 pour représenter les noms de fichiers (et les noms de répertoires) sur le disque. Si vous souhaitez utiliser un autre encodage, vous devez utiliser Encode::encode(...).

use v5.14;
# Make Perl recognize UTF-8 encoded characters in literal strings.
# For this to work: Make sure your text-editor is using UTF-8, so
# that bytes on disk are really UTF-8 encoded.
use utf8;

# Ensure that possible error messages printed to screen are converted to UTF-8.
# For this to work: Check that your terminal emulator is using UTF-8.
binmode STDOUT, ':utf8';
binmode STDERR, ':utf8';
    
my $filename = 'æ€'; # $filename is now an internally UTF-8 encoded string. 

# Note: in the following it is assumed that $filename has the internal UTF-8
#  flag set, if $filename is pure ASCII, it will also work since its encoding
#  overlaps with UTF-8. However, if it has another encoding like extended ASCII,
#  $filename will be written with that encoding and not UTF-8. 
# Note: it is not necessary to encode $filename as UTF-8 here
#  since Perl is using UTF-8 as its internal encoding of $filename already

# Example1  -- using open()
open ( my $fh, '>', $filename ) or die "Could not open '$filename': $!";
close $fh;

# Example2 -- using qx() and touch
qx{touch $filename};

# Example3 -- using system() and touch
system 'touch', $filename;

# Example4 -- using File::Touch
use File::Touch;
eval { touch( $filename ) }; die "Could not create file '$filename': $!" if $@;

Lire les noms de fichiers #

Perl n’essaie pas de décoder les noms de fichiers renvoyés par les fonctions ou modules intégrés. De telles chaînes représentant des noms de fichiers doivent toujours être décodées explicitement, afin que Perl les reconnaisse comme Unicode.

use v5.14;
use Encode qw(decode_utf8);

# Ensure that possible error messages printed to screen are converted to UTF-8.
# For this to work: Check that you terminal emulator is using UTF-8.
binmode STDOUT, ':utf8';
binmode STDERR, ':utf8';

# Example1  -- using readdir()
my $dir = '.';
opendir(my $dh, $dir) or die "Could not open directory '$dir': $!";
while (my $filename = decode_utf8(readdir $dh)) {
    # Do something with $filename
}
close $dh;

# Example2  -- using getcwd()
use Cwd qw(getcwd);
my $dir = decode_utf8( getcwd() );

# Example3  -- using abs2rel()
use File::Spec;
use utf8;
my $base = 'ø';
my $path = "$base/b/æ";
my $relpath = decode_utf8( File::Spec->abs2rel( $path, $base ) );
# Note: If you omit $base, you need to encode $path first:
use Encode qw(encode_utf8);
my $relpath = decode_utf8( File::Spec->abs2rel( encode_utf8( $path ) ) );

# Example4  -- using File::Find::Rule (part1 matching a filename)
use File::Find::Rule;
use utf8;
use Encode qw(encode_utf8);
my $filename = 'æ';
# File::Find::Rule needs $filename to be encoded
my @files = File::Find::Rule->new->name( encode_utf8($filename) )->in('.');
$_ = decode_utf8( $_ ) for @files;

# Example5  -- using File::Find::Rule (part2 matching a regular expression)
use File::Find::Rule;
use utf8;
my $pat = '[æ].$'; # Unicode pattern
# Note: In this case:  File::Find::Rule->new->name( qr/$pat/ )->in('.')
#  will not work since $pat is Unicode and filenames are bytes
#  Also encoding $pat first will not work correctly
my @files;
File::Find::Rule->new->exec( sub { wanted( $pat, \@files ) } )->in('.');
$_ = decode_utf8( $_ ) for @files;
sub wanted {
    my ( $pat, $files ) = @_;
    my $name = decode_utf8( $_ );
    my $full_name = decode_utf8( $File::Find::name );
    push @$files, $full_name if $name =~ /$pat/;
}

Remarque : si vous vous inquiétez d’un UTF-8 invalide dans les noms de fichiers, l’utilisation de decode_utf8( ... ) dans les exemples ci-dessus devrait probablement être remplacée par decode( 'utf-8', ... ). En effet, decode_utf8( ... ) est synonyme de decode( 'utf8', ... ) et il existe une différence entre les encodages utf-8 et utf8 (voir Remarques ci-dessous pour plus d’informations) où utf-8 est plus strict sur ce qui est acceptable que utf8.

Commutateurs de ligne de commande pour les one-liners #

Activer le pragma utf8 #

Afin d’activer le pragma utf8 dans une ligne, l’interpréteur perl doit être appelé avec l’option -Mutf8 :

perl -Mutf8 -E 'my $人 = "human"; say $人'

Gestion d’Unicode avec le commutateur -C #

L’indicateur de ligne de commande -C vous permet de contrôler les fonctionnalités Unicode. Il peut être suivi d’une liste de lettres d’option.

## E/S standard

  • I - STDIN sera en UTF-8

  • O - STDOUT sera en UTF-8

  • E - STDERR sera en UTF-8

  • S - raccourci pour IOE, les flux d’E/S standard seront en UTF-8

    echo “Ματαιότης ματαιοτήτων” | perl -CS -Mutf8 -nE ‘say “ok” if /Ματαιότης/’

Arguments du script #

  • A - traite @ARGV comme un tableau de chaînes encodées UTF-8

    perl -CA -Mutf8 -E ‘my $arg = shift; say “anteater” if $arg eq “муравьед”’ муравьед

Couche PerlIO par défaut #

  • i - UTF-8 est la couche PerlIO par défaut pour les flux d’entrée

  • o - UTF-8 est la couche PerlIO par défaut pour les flux de sortie

  • D - raccourci pour io

    perl -CD -Mutf8 -e ‘open my $fh, “>”, “utf8.txt” or die $!; print $fh “개미 조심해”’

Les commutateurs -M et -C peuvent être combinés :

perl -CASD -Mutf8 -E 'say "Ματαιότης ματαιοτήτων\n"';

## E/S standard L’encodage à utiliser pour les descripteurs de fichiers d’E/S standard (STDIN, STDOUT et STDERR) peut être défini séparément pour chaque descripteur à l’aide de [binmode](http://perldoc.perl.org /fonctions/binmode.html) :

binmode STDIN, ':encoding(utf-8)';
binmode STDOUT, ':utf8';
binmode STDERR, ':utf8';

Remarque : lors de la lecture, on préférerait en général :encoding(utf-8) à :utf8, voir Remarks pour plus d’informations .

Alternativement, vous pouvez utiliser le pragma open.

# Setup such that all subsequently opened input streams will use ':encoding(utf-8)'
# and all subsequently opened output streams will use ':utf8'
# by default
use open (IN => ':encoding(utf-8)', OUT => ':utf8');
# Make the (already opened) standard file handles inherit the setting 
# given by the IO settings for the open pragma
use open ( :std );
# Now, STDIN has been converted to ':encoding(utf-8)', and 
# STDOUT and STDERR have ':utf8'

Alternativement, pour définir tous les handles de fichiers (à la fois ceux qui n’ont pas encore été ouverts et aussi ceux standard) pour utiliser :encoding(utf-8) :

use open qw( :encoding(utf-8) :std );

Descripteurs de fichiers #

Définir l’encodage avec open() #

Lors de l’ouverture d’un fichier texte, vous pouvez spécifier explicitement son encodage avec un open() à trois arguments. Ce en-/décodeur attaché à un descripteur de fichier est appelé une “couche d’E/S”:

my $filename = '/path/to/file';
open my $fh, '<:encoding(utf-8)', $filename or die "Failed to open $filename: $!";

Voir Remarks pour une discussion sur les différences entre :utf8 et :encoding(utf-8).

Définir l’encodage avec binmode() #

Alternativement, vous pouvez utiliser binmode() pour définir l’encodage du descripteur de fichier individuel :

my $filename = '/path/to/file';
open my $fh, '<', $filename or die "Failed to open $filename: $!";
binmode $fh, ':encoding(utf-8)';

pragma ouvert #

Pour éviter de définir l’encodage séparément pour chaque descripteur de fichier, vous pouvez utiliser le pragma open pour définir une couche d’E/S par défaut utilisée par tous les appels ultérieurs au Fonction open() et opérateurs similaires dans la portée lexicale de ce pragma :

# Set input streams to ':encoding(utf-8)' and output streams to ':utf8'
use open (IN => ':encoding(utf-8)', OUT => ':utf8');
# Or to set all input and output streams to ':encoding(utf-8)'
use open ':encoding(utf-8)';

Définition de l’encodage avec la ligne de commande -C flag #

Enfin, il est également possible d’exécuter l’interpréteur perl avec un drapeau -CD qui applique UTF-8 comme couche d’E/S par défaut. Cependant, cette option doit être évitée car elle repose sur un comportement spécifique de l’utilisateur qui ne peut être ni prédit ni contrôlé.

Gestion de l’UTF-8 invalide #

Lecture d’UTF-8 invalide #

Lors de la lecture de données encodées en UTF-8, il est important d’être conscient du fait que les données encodées en UTF-8 peuvent être invalides ou mal formées. De telles données ne devraient généralement pas être acceptées par votre programme (à moins que vous ne sachiez ce que vous faites). En cas de rencontre inattendue de données malformées, différentes actions peuvent être envisagées :

  • Imprimer le stacktrace ou le message d’erreur, et abandonner le programme en douceur, ou
  • Insérez un caractère de substitution à l’endroit où la séquence d’octets malformée est apparue, imprimez un message d’avertissement à STDERR et continuez à lire car rien ne s’est passé.

Par défaut, Perl vous warn sur les problèmes d’encodage, mais il n’interrompra pas votre programme. Vous pouvez faire avorter votre programme en rendant les avertissements UTF-8 fatals, mais soyez conscient des mises en garde dans Fatal Warnings.

L’exemple suivant écrit 3 octets en encodant ISO 8859-1 sur le disque. Il essaie ensuite de relire les octets en tant que données encodées en UTF-8. L’un des octets, “0xE5”, est une séquence d’un octet UTF-8 non valide :

use strict;
use warnings;
use warnings FATAL => 'utf8';

binmode STDOUT, ':utf8';
binmode STDERR, ':utf8';
my $bytes = "\x{61}\x{E5}\x{61}";  # 3 bytes in iso 8859-1: aåa
my $fn = 'test.txt';
open ( my $fh, '>:raw', $fn ) or die "Could not open file '$fn': $!";
print $fh $bytes;
close $fh;
open ( $fh, "<:encoding(utf-8)", $fn ) or die "Could not open file '$fn': $!";
my $str = do { local $/; <$fh> };
close $fh;
print "Read string: '$str'\n";

Le programme s’arrêtera avec un avertissement fatal :

utf8 "\xE5" does not map to Unicode at ./test.pl line 10.

La ligne 10 est ici l’avant-dernière ligne, et l’erreur se produit dans la partie de la ligne avec <$fh> lors de la tentative de lecture d’une ligne du fichier.

Si vous ne rendez pas les avertissements fatals dans le programme ci-dessus, Perl affichera toujours l’avertissement. Cependant, dans ce cas, il essaiera de récupérer de l’octet mal formé 0xE5 en insérant les quatre caractères \xE5 dans le flux, puis il continuera avec l’octet suivant. En conséquence, le programme imprimera :

Read string: 'a\xE5a'
