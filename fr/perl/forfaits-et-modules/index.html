Syntaxe #

  • nécessite Module :: Nom ; # Exiger par nom de @INC
  • nécessite “chemin/vers/fichier.pm” ; # Requiert par chemin relatif depuis @INC
  • utilisez Module::Nom ; # importation requise et par défaut au DÉBUT
  • utilisez Module::Nom (); # requis et pas d’importation au BEGIN
  • utilisez Module :: Nom (@ARGS); # exiger et importer avec des arguments au DÉBUT
  • utilisez Module::Nom VERSION ; # requis, vérification de la version et importation par défaut au DÉBUT
  • utilisez Module::Nom VERSION (); # requis, vérification de la version et aucune importation au DÉBUT
  • utilisez Module::Name VERSION (@ARGS); # exiger, vérifier la version, importer avec des arguments au DÉBUT
  • faites “chemin/vers/fichier.pl” ; # charger et évaluer le fichier donné

Utiliser un module #

use Cwd;

Cela importera le module Cwd au moment de la compilation et importera ses symboles par défaut, c’est-à-dire rendra certaines variables et fonctions du module disponibles pour le code qui l’utilise. (Voir aussi : perldoc -f use.)

En général, c’est ce qu’il faut faire. Parfois, cependant, vous voudrez contrôler quels symboles sont importés. Ajoutez une liste de symboles après le nom du module à exporter :

use Cwd 'abs_path';

Si vous faites cela, seuls les symboles que vous spécifiez seront importés (c’est-à-dire que le jeu par défaut ne sera pas importé).

Lors de l’importation de plusieurs symboles, il est idiomatique d’utiliser la construction de liste qw() :

use Cwd qw(abs_path realpath);

Certains modules exportent un sous-ensemble de leurs symboles, mais on peut leur dire de tout exporter avec :all :

use Benchmark ':all';

(Notez que tous les modules ne reconnaissent pas ou n’utilisent pas la balise :all).

Chargement d’un module à l’exécution #

require Exporter;

Cela garantira que le module Exporter est chargé au moment de l’exécution s’il n’a pas déjà été importé. (Voir aussi : perldoc -f require.)

N.B. : La plupart des utilisateurs devraient “utiliser” des modules plutôt que de les “exiger”. Contrairement à use, require n’appelle pas la méthode d’importation du module et est exécuté au moment de l’exécution, pas pendant la compilation.

Cette façon de charger les modules est utile si vous ne pouvez pas décider de quels modules vous avez besoin avant l’exécution, comme avec un système de plugin :

package My::Module;
my @plugins = qw( One Two );
foreach my $plugin (@plugins) {
    my $module = __PACKAGE__ . "::Plugins::$plugin";
    $module =~ s!::!/!g;
    require "$module.pm";
}

Cela essaierait de charger My::Package::Plugins::One et My::Package::Plugins::Two. @plugins doit bien sûr provenir d’une entrée utilisateur ou d’un fichier de configuration pour que cela ait un sens. Notez l’opérateur de substitution s!::!/!g qui remplace chaque paire de deux-points par une barre oblique. En effet, vous pouvez charger des modules en utilisant la syntaxe familière du nom de module à partir de use uniquement si le nom du module est un mot nu. Si vous passez une chaîne ou une variable, elle doit contenir un nom de fichier.

Utiliser un module dans un répertoire #

use lib 'includes';
use MySuperCoolModule;

use lib 'includes'; ajoute le répertoire relatif includes/ comme un autre chemin de recherche de module dans @INC. Supposons donc que vous ayez un fichier de module MySyperCoolModule.pm à l’intérieur de includes/, qui contient :

package MySuperCoolModule;

Si vous le souhaitez, vous pouvez regrouper autant de modules de votre choix dans un seul répertoire et les rendre trouvables avec une seule instruction use lib.

À ce stade, l’utilisation des sous-routines dans le module nécessitera de préfixer le nom de la sous-routine avec le nom du package :

MySuperCoolModule::SuperCoolSub_1("Super Cool String");

Pour pouvoir utiliser les sous-programmes sans le préfixe, vous devez exporter les noms des sous-programmes afin qu’ils soient reconnus par le programme qui les appelle. L’exportation peut être paramétrée pour être automatique, ainsi :

package MySuperCoolModule;
use base 'Exporter';
our @EXPORT = ('SuperCoolSub_1', 'SuperCoolSub_2');

Ensuite, dans le fichier qui “utilise” le module, ces sous-routines seront automatiquement disponibles :

use MySuperCoolModule;
SuperCoolSub_1("Super Cool String");

Ou vous pouvez configurer le module pour exporter conditionnellement des sous-programmes, ainsi :

package MySuperCoolModule;
use base 'Exporter';
our @EXPORT_OK = ('SuperCoolSub_1', 'SuperCoolSub_2');

Dans ce cas, vous devez demander explicitement que les sous-routines souhaitées soient exportées dans le script qui “utilise” le module :

use MySuperCoolModule 'SuperCoolSub_1';
SuperCoolSub_1("Super Cool String");

Exécuter le contenu d’un autre fichier #

do './config.pl';

Cela lira le contenu du fichier config.pl et l’exécutera. (Voir aussi : perldoc -f do.)

N.B. : Évitez de “faire” sauf si vous jouez au golf ou autre, car il n’y a pas de vérification d’erreur. Pour inclure des modules de bibliothèque, utilisez require ou use.
