Décrit les éléments, les attributs et les types qui sont valides dans un document d’instance XML. Un schéma XML (XSD) doit contenir un seul élément xs:schema de niveau racine.

Paramètres #

Attributs Descriptif
attributFormDefault Indique si les attributs du document d’instance XML doivent être qualifiés avec un espace de noms (non qualifié par défaut)
blockDefault La valeur par défaut de l’attribut block appliqué à [xs:complexType][1] et xs:element. Définit les règles de blocage de la dérivation/substitution dans le document d’instance (vide par défaut, c’est-à-dire ne rien bloquer)
attributs par défaut (XSD 1.1) Spécifie un xs:attributeGroup qui sera associé à tous les [xs:complexType][1] et xs:element dans le schéma (facultatif).
elementFormDefault Indique si l’élément dans le document d’instance XML doit être qualifié avec un espace de noms (non qualifié par défaut). Remarque : presque sans exception, tous les schémas définissent ceci sur ‘qualified’.
finalDefault La valeur d’attribut final par défaut utilisée dans [xs:complexType][1] et xs:element. Définit les règles de blocage de la dérivation/substituion dans le schéma (vide par défaut, c’est-à-dire ne rien bloquer)
identifiant L’identifiant de l’élément de schéma (optionnel)
espace de noms cible Qualifie tous les éléments et attributs (et les composants définis globalement) définis dans ce schéma.
version La version du schéma, c’est la version du document, pas cette version XSD (c’est-à-dire Soap 1.2, FpML 4.2, etc.)
xpathDefaultNamespace (XSD 1.1) La valeur par défaut de l’attribut xpathDefaultNamespace qui est utilisé dans xs:selector, xs:field, xs:alternative & xs:assert (spécifie l’espace de noms par défaut à utiliser dans les expressions XPath)
n’importe lequel Tous les autres attributs ne figurant pas dans l’espace de noms ‘http://www.w3.org/2001/XMLSchema’ sont autorisés.
Éléments Description
xs:annotation Offre la possibilité d’ajouter de la documentation et des données lisibles par machine.
xs : inclure Utilisé pour inclure un schéma avec le même targetNamespace, ou aucun targetNamespace (voir schémas caméléon).
xs : importer Utilisé pour inclure un schéma avec un targetNamespace différent du parent.
xs:redéfinir Utilisé pour inclure un schéma avec le même targetNamespace (ou aucun targetNamespace), et modifier les définitions xs:simpleType, xs:complexType, xs:group ou xs:attributeGroup qu’il contient (ici des dragons….)
xs:simpleType Définit un type simple global (nommé) qui peut ensuite être référencé et réutilisé.
[xs:complexType][1] Définit un type complexe global (nommé) qui peut ensuite être référencé et réutilisé.
xs:groupe Définit un groupe global (nommé) d’éléments qui peuvent ensuite être référencés et réutilisés.
xs:groupeattribut Définit un groupe global (nommé) d’attributs qui peuvent ensuite être référencés et réutilisés.
xs:attribut Définit un attribut global (nommé) qui peut ensuite être référencé et réutilisé.
xs:élément Définit un élément global (nommé) qui peut ensuite être référencé et réutilisé, ou utilisé comme base d’un document d’instance XML.
xs:notation -
xs:defaultOpenContent (XSD 1.1) Spécifie les règles permettant d’autoriser des éléments supplémentaires dans chaque [xs:complexType][1] et xs:element du schéma.

[1] : https://www.wikiod.com/fr/xsd/xscomplextype

Basic xs : schéma #

Affiche un schéma très basique.

Remarque : par convention, elementFormDefault est défini sur ‘qualified’, dans le monde réel, vous aurez du mal à trouver un schéma qui ne le définit pas (il suffit donc de l’inclure dans vos schémas !).

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 - (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="qualified" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="Person">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="Forename" type="xs:string" />
                <xs:element name="Surname" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

xs : attribut de schéma elementFormDefault #

Par convention, elementFormDefault est toujours défini sur qualified, mais regardons ce qu’il fait réellement.

D’abord avec elementFormDefault défini sur qualifié.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="qualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

Exemple de document XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_qualified.xsd">
    <b:ChildA>string</b:ChildA>
</b:MyBaseElement>

Notez que l’élément ChildA doit également être qualifié avec l’espace de noms ‘b’.

Regardons maintenant avec elementFormDefault défini sur non qualifié.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="unqualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

Exemple de document XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_unqualified.xsd">
    <ChildA>string</ChildA>
</b:MyBaseElement>

Notez cette fois que seul l’élément MyBaseElement défini globalement est qualifié avec l’espace de noms ‘b’, l’élément interne ChildA (qui est défini en place dans le schéma) n’est pas qualifié.

Dans le dernier exemple, nous avons vu que les éléments définis globalement doivent être qualifiés dans le document d’instance XML, mais pas les éléments définis sur place. Mais cela ne signifie pas seulement l’élément racine, si vous avez des éléments définis globalement qui sont référencés, ils doivent également être qualifiés.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="unqualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
                <xs:element xmlns:q1="http://base.com" ref="q1:MyElement" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
    <xs:element name="MyElement" type="xs:string" />
</xs:schema>

Exemple de document XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_unqualified.xsd">
    <ChildA>string</ChildA>
    <b:MyElement>string</b:MyElement>
</b:MyBaseElement>

Notez que MyElement doit également être qualifié car il est défini globalement.

En conclusion, si vous avez défini elementFormDefault sur qualifié, alors tout doit être qualifié avec un espace de noms (soit via un alias d’espace de noms, soit en définissant le namesapce par défaut xmlns="…"). Cependant, elementFormDefault est défini sur non qualifié, les choses se compliquent et vous devez faire un examen assez approfondi des schémas pour déterminer si les choses doivent être qualifiées ou non.

Je suppose que c’est pourquoi elementFormDefault est toujours défini sur qualifié !
