Comprendre l'erreur 1009 Impossible d'accéder à une propriété ou à une méthode d'une référence d'objet nul

Une erreur 1009 est une erreur générale qui survient lorsque vous essayez de recevoir une valeur d’une variable ou d’une propriété dont la valeur est “null”. Les exemples fournis exposent divers cas où cette erreur se produit, ainsi que quelques recommandations sur la façon d’atténuer l’erreur.

Le redouté et souvent demandé “Erreur 1009 : Impossible d’accéder à une propriété ou à une méthode d’une référence d’objet nul” est un signal indiquant que certaines des données semblent nulles, mais qu’elles tentent d’être utilisées comme objet rempli. Il existe de nombreux types de problèmes qui peuvent provoquer ce comportement, et chacun doit être testé par rapport au code où l’erreur s’est produite.

## Écouteur d’événement oublié addEventListener(Event.ENTER_FRAME,moveChild); function moveChild(e:Event):void { childMC.x++; if (childMC.x>1000) { gotoAndStop(2); } }

Cet exemple déplacera le childMC (ajouté à Main au moment de la conception) mais lancera instantanément un 1009 dès que gotoAndStop() sera appelé, si ce childMC n’existe pas sur l’image 2. La raison principale car c’est que chaque fois qu’une tête de lecture passe une image clé (une image qui n’hérite pas de l’ensemble d’objets de l’image précédente), soit en utilisant gotoAndStop(), gotoAndPlay() avec l’image de destination séparée de l’image actuelle par une image clé, ou par lecture normale si le SWF est une animation, le contenu de l’image actuelle est détruit et le nouveau contenu est créé à l’aide des données stockées à partir de l’interface graphique. Ainsi, si le nouveau cadre n’a pas d’enfant nommé childMC, la demande de propriété renverra null et 1009 sera renvoyé.

Le même principe s’applique si vous ajoutez deux écouteurs d’événement, mais n’en supprimez qu’un, ou si vous ajoutez un écouteur à un objet, mais essayez d’en supprimer un autre. L’appel removeEventListener ne vous avertira pas si l’objet n’avait pas d’écouteur d’événement respectif attaché, alors lisez attentivement le code qui ajoute et supprime les écouteurs d’événement.

Notez également : l’utilisation d’objets Timer, l’appel de [setInterval() et setTimeout()][1] crée également des écouteurs d’événements, et ceux-ci doivent également être effacés correctement.

[1] : http://help.adobe.com/en_US/FlashPlatform/reference/actionscript/3/flash/utils/package-detail.html

L’étape n’est pas disponible #

Parfois, les développeurs écrivent du code qui souhaite accéder à “stage”, ou scène Flash, pour ajouter des écouteurs. Cela peut fonctionner pour la première fois, puis tout d’un coup échouer et produire l’erreur 1009. Le code en question peut même être sur la timeline, car c’est la première initiative pour y ajouter du code, et de nombreux tutoriels qui existent encore utilisent couche de code de la chronologie pour placer le code.

public class Main extends MovieClip {
    public function Main() {
        stage.addEventListener(Event.ENTER_FRAME,update); // here

La raison pour laquelle ce code ne fonctionne pas est simple : un objet d’affichage est d’abord instancié, puis ajouté à la liste d’affichage, et tant qu’il est hors de la liste d’affichage, stage est nul.

Pire si le code comme ceci:

stage.addEventListener(Event.ENTER_FRAME,update); // here

est placé sur la chronologie. Cela peut même fonctionner pendant un certain temps, tandis que l’objet Main est mis en scène via l’interface graphique. Ensuite, leur SWF est chargé à partir d’un autre SWF, et tout d’un coup le code se casse. Cela se produit parce que les images du Main sont construites d’une manière différente lorsque le SWF est chargé directement par le lecteur et lorsque le chargement est traité de manière asynchrone. La solution consiste à utiliser l’écouteur Event.ADDED_TO_STAGE, à y mettre tout le code qui traite de l’étape et à placer l’écouteur lui-même dans un fichier AS au lieu de la chronologie.

Typecast invalide #

function listener(e:Event):void {
    var m:MovieClip=e.target as MovieClip;
    m.x++;
}

Si un tel écouteur est attaché à un objet qui n’est pas un descendant de MovieClip (par exemple, un Sprite), le transtypage échouera et toute opération ultérieure avec son résultat générera l’erreur 1009.

Objet non instancié #

var a:Object;
trace(a); // null
trace(a.b); // Error 1009

Ici, une référence d’objet est déclarée, mais ne se voit jamais attribuer de valeur, que ce soit avec new ou l’attribution d’une valeur non nulle. La demande de ses propriétés ou de sa méthode entraîne une erreur 1009.

Expression à plusieurs niveaux #

x=anObject.aProperty.anotherProperty.getSomething().data;

Ici, tout objet avant le point peut finir par être nul, et l’utilisation de méthodes qui renvoient des objets complexes ne fait qu’augmenter la complication pour déboguer l’erreur nulle. Dans le pire des cas, si la méthode est sujette à des échecs externes, par exemple la récupération de données sur le réseau.

Résultat de la fonction non traité #

s=this.getChildByName("garbage");
if (s.parent==this) {...}

[getChildByName()][1] est l’une des nombreuses fonctions qui peuvent renvoyer null si une erreur s’est produite lors du traitement de son entrée. Par conséquent, si vous recevez un objet d’une fonction susceptible de renvoyer null, vérifiez d’abord la valeur null. Ici, une propriété est instantanément interrogée sans vérifier au préalable si s est nul, cela générera l’erreur 1009.

[1] : http://help.adobe.com/en_US/FlashPlatform/reference/actionscript/3/flash/display/DisplayObjectContainer.html#getChildByName%28%29

Référence invalide à un objet basé sur un cadre #

Parfois, gotoAndStop() est appelé au milieu du code qui fait référence à certaines propriétés basées sur le cadre. Mais, juste après la modification du cadre, tous les liens vers les propriétés qui existaient sur le cadre actuel sont invalidés, de sorte que tout traitement les impliquant doit être immédiatement interrompu.

Il existe deux scénarios généraux pour qu’un tel traitement se produise : premièrement, une boucle ne se termine pas après l’appel de gotoAndStop(), comme ici :

for each (bullet in bullets) {
    if (player.hitTestObject(bullet)) gotoAndStop("gameOver");
}

Ici, une erreur 1009 signifie que le MC player a été détruit lors du traitement de l’appel gotoAndStop(), mais la boucle continue et renvoie le lien désormais nul pour obtenir hitTestObject(). Si la condition disait if (bullet.hitTestObject(player)) à la place, l’erreur serait #2007 “Le paramètre hitTestObject ne doit pas être nul”. La solution consiste à mettre une instruction return juste après avoir appelé gotoAndStop().

Le deuxième cas est plusieurs écouteurs d’événement sur le même événement. Comme ça:

stage.addEventListener(Event.ENTER_FRAME,func1);
stage.addEventListener(Event.ENTER_FRAME,func2);
function func1(e:Event):void {
    if (condition()) {
        gotoAndStop(2);
    }
}

Ici, si condition() est vraie, le premier écouteur exécuterait gotoAndStop(), mais le deuxième écouteur serait toujours exécuté, et si celui-ci fait référence à des objets sur le cadre, une erreur 1009 sera générée. La solution est d’éviter plusieurs écouteurs sur un seul événement, dans un seul objet, il est préférable d’avoir un écouteur qui gère toutes les situations sur cet événement et peut se terminer correctement si un changement de cadre est nécessaire.
