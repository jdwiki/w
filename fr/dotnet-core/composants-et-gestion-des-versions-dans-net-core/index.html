Composants et gestion des versions dans .NET Core

  • Identification des versions
  • Sélection des versions

Sur cette page

  • Identification des versions
  • Sélection des versions

Ce document couvre les différents composants qui composent une distribution .NET Core et comment ils sont versionnés. Ce document couvre actuellement les versions 1.x.

Comment les composants dans .NET Core sont versionnés.

Composants #

.NET Core se compose de plusieurs composants qui sont chacun versionnés indépendamment et peuvent souvent être mélangés et appariés.

  • Cadre partagé. Celui-ci contient les API et la machine virtuelle et autres services d’exécution nécessaires pour exécuter des applications .NET Core.

  • La machine virtuelle .NET Core actuelle s’appelle CoreCLR. Cette executes the .NET bytecode by compiling it JIT and provides various runtime services including a garbage collector. The complete source code for CoreCLR is available at https://github.com/dotnet/coreclr.

  • Les API standard .NET Core sont implémentées dans CoreFX. Cette provides implementations of all your favourite APIs such as System.Runtime, System.Theading and so on. The source code for CoreFX is available at https://github.com/dotnet/corefx.

  • Hôte est également appelé multiplexeur ou pilote. Ces composants représente la commande dotnet et est responsable de décider ce arrive ensuite. La source est disponible sur https://github.com/dotnet/core-setup.

  • SDK est aussi parfois appelé CLI. Il se compose de la divers outils (sous-commandes dotnet) et leurs implémentations qui traiter avec le code du bâtiment. Cela comprend la gestion de la restauration de dépendances, compilation de code, création de binaires, production de packages et la publication de packages autonomes ou dépendants du framework. Le SDK se compose de la * CLI *, qui gère les opérations de ligne de commande (sur https://github.com/dotnet/cli) et divers sous-projets qui mettre en œuvre les différentes opérations que la CLI doit effectuer.

Composants dans les installations .NET Core #

Divers packages officiels et non officiels, archives tar, zips et installateurs pour .NET Core (y compris ceux disponibles sur https://dot.net/core) fournir .NET Core dans de nombreuses variantes. Deux des plus courants sont les SDK et Temps d’exécution.

Chaque installation SDK ou installation Runtime contient un nombre (éventuellement 0) d’hôtes, de SDK et de composants de framework partagés décrits ci-dessus.

  • .NET Core Runtime contient

  • 1 version de Framework partagé

  • 1 version de l’hôte

  • .NET Core SDK contient

  • 1 ou plusieurs versions du Shared Framework (varie selon the version of the SDK)

  • 1 version de l’hôte

  • 1 version du SDK

Identification des versions #

Chaque composant .NET Core (SDK, Host et Shared Framework) est versionné indépendamment.

Vous pouvez trouver la version pour chacun d’eux séparément.

  • SDK

Vous pouvez utiliser l’option --version pour dotnet pour voir le SDK version. Par exemple:

```
$ ~/dotnet-1.1.1/dotnet --version
1.0.0-preview2-1-003176
```

dotnet --info affiche également la version du SDK.

  • Héberger

Vous pouvez exécuter dotnet par lui-même sans arguments ni options pour voir la version de l’hôte.

```
$ ~/dotnet-1.1.1/dotnet

Microsoft .NET Core Shared Framework Host

  Version  : 1.1.0
  Build    : 362e48a95c86b40cd1f2ef3d08741f7fed897956

Usage: dotnet [common-options] [[options] path-to-application]
...
```
  • Cadre partagé

Il n’y a pas de commande actuellement pour afficher le partage disponible cadres. j’utilise ls /chemin/vers/où/vous/installez/dotnet/shared/Microsoft.NETCore.App qui s’appuie sur des détails de mise en œuvre internes. Par exemple:

```
$ ls ~/dotnet-1.1.1/shared/Microsoft.NETCore.App/
1.1.1
```

Sélection des versions #

Il est possible d’avoir plusieurs SDK et Runtimes .NET Core disponible sur disque. Vous pouvez sélectionner les versions pour chacun séparément.

Pour sélectionner la version du SDK à utiliser, utilisez global.json.

Pour sélectionner la version du framework partagé à utiliser, ciblez le framework spécifié dans le fichier [.csproj (ou project.json si vous l’utilisez toujours)](https://docs.microsoft.com/en -us/dotnet/articles/core/tools/project-json-to-csproj#frameworks).
