Comprendre System.Runtime par rapport à mscorlib

Chaque bibliothèque et langage de programmation .NET utilise un ensemble de types de données élémentaires tels que System.Int32, System.Object, System.Type ou System.Uri. Ces types de données constituent la base de toutes les autres structures, y compris toutes les bibliothèques .NET écrites sur mesure. Tous ces types sont hébergés dans une bibliothèque de base, qui est soit mscorlib ou System.Runtime.

Les bibliothèques pouvant être utilisées avec .NET Core sont basées sur la bibliothèque principale System.Runtime tandis que pour le .NET Framework (le composant Windows), elles sont basées sur mscorlib. Cette différence essentielle conduit à…

  • l’incompatibilité des bibliothèques plus anciennes puisqu’elles attendent un type System.Object, mscorlib alors qu’une bibliothèque .NET Core attendrait System.Object, System.Runtime.
  • une bibliothèque de façade de transfert de type appelée System.Runtime vers mscorlib dans le .NET Framework. Cette bibliothèque est par ailleurs (presque) vide mais permet l’utilisation de bibliothèques PCL basées sur “System.Runtime” sur le .NET Framework.
  • un type redirigeant mscorlib vers System.Runtime dans une future version de .NET Core.
  • l’introduction du concept de bibliothèque de classes portable (PCL) et, en tant que deuxième génération, du “netstandard” comme méthode d’unification entre les deux bibliothèques principales.

Erreur populaire : NuGet trompeur dans le mauvais sens #

.NET Core project.json prend en charge l’importation NuGet (alias couché selon [cette réponse SO] (http://stackoverflow.com/a/38688430/2410379)). Il est impossible d’inclure une bibliothèque basée sur mscorlib en raison d’une instruction import.

{
    "version": "1.0.0-*",

    "dependencies": {
        "Microsoft.AspNet.Identity.EntityFramework": "2.2.1",
        "NETStandard.Library": "1.6.0"
    },

    "frameworks": {
        "netstandard1.6": {
            "imports": [
                "net461"
            ]
        }
    }
}

Les importations ne fonctionnent qu’avec des bibliothèques de classes portables (qui sont basées sur “System.Runtime”) ou des monikers de framework cible obsolètes qui sont également basés sur “System.Runtime” (par exemple, “dotnet” ou “dnxcore”)

Erreur populaire : Ajoutez un package NuGet qui n’a pas été conçu pour netstandard / netcoreapp (System.Runtime) #

Dans l’exemple project.json ci-dessous, un assembly Microsoft.AspNet.Identity.EntityFramework a été ajouté et est basé sur mscorlib.

{
    "version": "1.0.0-*",

    "dependencies": {
        "Microsoft.AspNet.Identity.EntityFramework": "2.2.1",
        "NETStandard.Library": "1.6.0"
    },

    "frameworks": {
        "netstandard1.6": { }
    }
}

L’auteur de l’assembly Microsoft.AspNet.Identity.EntityFramework n’a pas encore porté le package NuGet sur netstandard (ils l’ont fait, ils ont juste renommé le package également Microsoft.AspNetCore.Identity.EntityFrameworkCore ;))

Si vous rencontrez un paquet dont vous avez besoin et qui n’est pas encore sur netstandard, veuillez contacter l’auteur et - si possible - aider à le porter.

Erreur populaire : mauvaise compréhension du résultat #

Cibler plusieurs frameworks avec project.json est simple. Cependant, le résultat est différent compilations. Prenons l’exemple suivant :

{
    "version": "1.0.0-*",

    "dependencies": {
        "NETStandard.Library": "1.6.0",
        "System.Collections.Immutable": "1.2.0"
    },

    "frameworks": {
        "netstandard1.3": { },
        "net451": { }
    }
}

Le processus de compilation du fichier project.json conduira à deux artefacts :

  • Une dll compilée pour le monde netstandard basé sur System.Runtime qui peut être utilisé sur .NET Core, .NET Framework (via des redirecteurs de type) et les produits Xamarin (via des redirecteurs de type). Cette dll fait référence à System.Runtime et System.Collections.Immutable.
  • Une autre dll compilée directement pour le .NET Framework basé sur mscorlib. Cette dll aura des références à mscorlib et System.Collection.Immutable.

Cependant, il est important de comprendre que le System.Collections.Immutable basé sur netstandard1.0 utilisera différentes implémentations System.Runtime pour chaque DLL de construction au moment de l’exécution. Le System.Runtime fourni avec .NET Core n’a pas de dépendances d’assemblage (puisqu’il implémente la bibliothèque principale). Le System.Runtime utilisé avec le .NET Framework a des références (pour les redirecteurs de type) aux assemblys .NET Framework mscorlib, System.Core, System et System.ComponentModel.Composition.

Erreur populaire : ajout accidentel d’une bibliothèque mscorlib en tant que dépendances à un netstandard/netcoreapp #

Une autre erreur courante est la référence de packages qui ne satisfait pas tous les frameworks sur le portée globale lorsque plusieurs frameworks sont ciblés.

{
    "version": "1.0.0-*",

    "dependencies": {
        "NETStandard.Library": "1.6.0",
        "Microsoft.AspNet.Identity.EntityFramework": "2.2.1"
    },

    "frameworks": {
        "netstandard1.3": { },
        "net451": { }
    }
}

La (méta) bibliothèque NETStandard.Library fonctionne bien dans cet exemple, car elle cible à la fois netstandard1.3 et net451. Cependant la bibliothèque Microsoft.AspNet.Identity.EntityFramework ne cible que le .NET Framework net et mscorlib et ne peut donc pas être utilisé pour une sortie netstandard.

Soit rechercher une bibliothèque (version) qui couvre les deux frameworks, soit ajouter la bibliothèque dans les dépendances conditionnelles sous le framework.

{
    "version": "1.0.0-*",

    "dependencies": {
        "NETStandard.Library": "1.6.0"
    },

    "frameworks": {
        "netstandard1.3": { },
        "net451": {
            "dependencies": {
                "Microsoft.AspNet.Identity.EntityFramework": "2.2.1",
            }
        }
    }
}

Dans ce cas, la bibliothèque ne peut être utilisée que dans des blocs conditionnels #ifdef pour la construction net451.
