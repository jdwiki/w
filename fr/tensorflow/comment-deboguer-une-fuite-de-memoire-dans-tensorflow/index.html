Utilisez Graph.finalize() pour intercepter les nœuds ajoutés au graphique #

Le mode d’utilisation le plus courant de TensorFlow consiste d’abord à construire un graphe de flux de données d’opérateurs TensorFlow (comme tf.constant() et tf.matmul(), puis à exécuter les étapes en appelant le [ tf.Session.run()][1] méthode dans une boucle (par exemple une boucle d’entraînement).

Une source courante de fuites de mémoire est l’endroit où la boucle d’apprentissage contient des appels qui ajoutent des nœuds au graphe, et ceux-ci s’exécutent à chaque itération, provoquant la croissance du graphe. Celles-ci peuvent être évidentes (par exemple, un appel à un opérateur TensorFlow comme tf.square()), implicites (par exemple, un appel à une fonction de la bibliothèque TensorFlow qui crée des opérateurs comme tf.train.Saver()), ou subtiles ( par exemple un appel à un opérateur surchargé sur un tf.Tensor et un tableau NumPy, qui appelle implicitement tf.convert_to_tensor() et ajoute un nouveau tf.constant() au graphique).

La méthode [tf.Graph.finalize()][2] peut aider à détecter les fuites comme celle-ci : elle marque un graphique en lecture seule et déclenche une exception si quelque chose est ajouté au graphique. Par exemple:

loss = ...
train_op = tf.train.GradientDescentOptimizer(0.01).minimize(loss)
init = tf.initialize_all_variables()

with tf.Session() as sess:
    sess.run(init)
    sess.graph.finalize()  # Graph is read-only after this statement.

    for _ in range(1000000):
        sess.run(train_op)
        loss_sq = tf.square(loss)  # Exception will be thrown here.
        sess.run(loss_sq)

Dans ce cas, l’opérateur * surchargé tente d’ajouter de nouveaux nœuds au graphe :

loss = ...
# ...
with tf.Session() as sess:
    # ...
    sess.graph.finalize()  # Graph is read-only after this statement.
    # ...
    dbl_loss = loss * 2.0  # Exception will be thrown here.

[1] : https://www.tensorflow.org/api_docs/python/tf/Session#run [2] : https://www.tensorflow.org/api_docs/python/tf/Graph#finalize

Utilisez l’allocateur tcmalloc #

Pour améliorer les performances d’allocation de mémoire, de nombreux utilisateurs de TensorFlow utilisent souvent [tcmalloc][1] au lieu de l’implémentation par défaut de malloc(), car tcmalloc souffre moins de fragmentation lors de l’allocation et de la désallocation d’objets volumineux (tels que de nombreux tenseurs) . Certains programmes TensorFlow gourmands en mémoire sont connus pour fuir l’espace d’adressage du tas (tout en libérant tous les objets individuels qu’ils utilisent) avec le malloc() par défaut, mais ils fonctionnent très bien après le passage à tcmalloc. De plus, tcmalloc inclut un [heap profiler][2], qui permet de retrouver où les fuites restantes auraient pu se produire.

L’installation de tcmalloc dépendra de votre système d’exploitation, mais ce qui suit fonctionne sur [Ubuntu 14.04 (trusty)][3] (où script.py est le nom de votre programme Python TensorFlow) :

$ sudo apt-get install google-perftools4
$ LD_PRELOAD=/usr/lib/libtcmalloc.so.4 python script.py ...

Comme indiqué ci-dessus, le simple fait de passer à tcmalloc peut résoudre de nombreuses fuites apparentes. Cependant, si l’utilisation de la mémoire continue d’augmenter, vous pouvez utiliser le profileur de tas comme suit :

$ LD_PRELOAD=/usr/lib/libtcmalloc.so.4 HEAPPROFILE=/tmp/profile python script.py ...

Après avoir exécuté la commande ci-dessus, le programme écrira périodiquement des profils sur le système de fichiers. La séquence de profils sera nommée :

  • /tmp/profile.0000.heap
  • /tmp/profile.0001.heap
  • /tmp/profile.0002.heap
  • …

Vous pouvez lire les profils à l’aide de l’outil google-pprof, qui (par exemple, sur Ubuntu 14.04) peut être installé dans le cadre du paquet google-perftools. Par exemple, pour regarder le troisième instantané collecté ci-dessus :

$ google-pprof --gv `which python` /tmp/profile.0002.heap

L’exécution de la commande ci-dessus fera apparaître une fenêtre GraphViz, affichant les informations de profil sous forme de graphique orienté.

[1] : http://goog-perftools.sourceforge.net/doc/tcmalloc.html [2] : http://goog-perftools.sourceforge.net/doc/heap_profiler.html [3] : http://packages.ubuntu.com/trusty/libs/libtcmalloc-minimal4
