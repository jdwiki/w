Vous créez des personnalisations SuiteScript à l’aide d’un système piloté par les événements. Vous définissez différents types d’enregistrements de script, chacun ayant son propre ensemble unique d’événements, et dans votre fichier source, vous définissez des fonctions qui seront appelées pour gérer ces événements lorsqu’ils se produisent.

Les scripts sont l’un des principaux composants avec lesquels vous allez concevoir et créer vos applications. Le but de cet article est simplement de se familiariser avec les types de scripts et les événements disponibles.

Le script client #

Le script client est l’un des types de script les plus couramment utilisés et les plus complexes à votre disposition. Comme son nom l’indique, le script client s’exécute dans le navigateur, c’est-à-dire côté client. C’est le seul type de script qui s’exécute côté client ; tous les autres s’exécuteront côté serveur de NetSuite.

L’utilisation principale du script client est de répondre aux interactions de l’utilisateur avec les formulaires d’enregistrement dans l’interface utilisateur NetSuite.

Dès que l’utilisateur charge un formulaire d’enregistrement en mode Édition, un événement pageInit est déclenché que nous pouvons utiliser pour exécuter du code lors de l’initialisation du formulaire, avant que l’utilisateur ne puisse interagir avec lui.

Chaque fois que l’utilisateur modifie un champ du formulaire, une série d’événements se déclenche :

  1. Un événement validateField se déclenche qui nous permet de valider la valeur que l’utilisateur essaie d’entrer dans le champ. Nous pouvons l’utiliser pour accepter ou empêcher le changement.
  2. Un événement fieldChanged se déclenche alors et nous permet de répondre à la nouvelle valeur dans le champ.
  3. Enfin, un événement postSourcing se déclenche après que tous les champs dépendants ont également été sourcés dans leurs valeurs. Cela nous permet de réagir au changement et de nous assurer que nous travaillons avec toutes les données correctes.

Cette série d’événements se déclenche, que l’utilisateur modifie un champ de corps ou un champ de sous-liste.

Au fur et à mesure que l’utilisateur apporte des modifications aux lignes de la sous-liste, une autre série d’événements est déclenchée :

  1. Un événement lineInit est déclenché chaque fois que l’utilisateur sélectionne initialement une ligne nouvelle ou existante, avant de pouvoir apporter des modifications aux champs de la ligne.
  2. Chaque fois que l’utilisateur clique sur le bouton Ajouter pour ajouter une nouvelle ligne, un événement validateLine est déclenché, ce qui nous permet de vérifier que la ligne entière est valide et peut être ajoutée à l’enregistrement.
  3. Chaque fois que l’utilisateur clique sur le bouton Insert pour ajouter une nouvelle ligne au-dessus d’une ligne existante, un événement validateInsert est déclenché, qui fonctionne exactement comme l’événement validateLine.
  4. De même, chaque fois que l’utilisateur essaie de supprimer une ligne, un validateDelete est déclenché qui permet d’autoriser ou de refuser la suppression de la ligne.
  5. [SuiteScript 1.0 uniquement] Enfin, après la réussite de l’événement de validation approprié, si la modification de la ligne a également entraîné une modification du montant total d’une transaction, un événement “recalc” est déclenché pour nous permettre de répondre à la modification. au montant de notre transaction.
  6. [SuiteScript 2.0 uniquement] Enfin, après la réussite de l’événement de validation approprié, un événement sublistChanged est déclenché pour nous permettre de répondre au changement de ligne terminé.

Enfin, lorsque l’utilisateur clique sur le bouton * Enregistrer * de l’enregistrement, un événement “saveRecord” est déclenché, ce qui nous permet de valider si l’enregistrement est valide et peut être enregistré. Nous pouvons soit empêcher la sauvegarde de se produire, soit lui permettre de poursuivre cet événement.

Le script client a de loin le plus grand nombre d’événements de tous les types de script et la relation la plus complexe entre ces événements.

Les scripts Suitelet et Portlet #

Souvent, nous voudrons créer des pages d’interface utilisateur personnalisées dans NetSuite ; entrer dans la Suitelet. Le script Suitelet est conçu pour créer des pages d’interface utilisateur internes et personnalisées. Les pages peuvent être en HTML de forme libre, ou elles peuvent utiliser les API UI Builder de NetSuite pour créer des formulaires qui suivent l’apparence de NetSuite.

Lorsqu’il est déployé, un Suitelet reçoit sa propre URL unique. Le Suitelet a alors un seul événement render qui est appelé chaque fois que cette URL est atteinte avec une requête HTTP GET ou POST. Typiquement, la réponse à la requête GET serait de rendre le formulaire lui-même, puis le formulaire se POST de nouveau pour traiter les données du formulaire.

Nous pouvons également tirer parti des Suitelets pour créer des progressions d’interface utilisateur de type assistant à l’aide des composants d’interface utilisateur “Assistant” de NetSuite.

Les portlets sont extrêmement similaires aux Suitelets, sauf qu’ils sont spécifiquement utilisés pour créer des widgets de tableau de bord personnalisés plutôt que des pages personnalisées complètes. En dehors de cela, les deux types de script fonctionnent de manière très similaire.

Le script d’événement utilisateur #

Le script d’événement utilisateur est étroitement lié au script client. Les événements de ce type de script sont à nouveau déclenchés lorsqu’un enregistrement est en cours de chargement ou de sauvegarde, mais il s’exécute à la place côté serveur. En tant que tel, il ne peut pas être utilisé pour répondre immédiatement aux modifications de champ, mais il n’est pas non plus limité aux seuls utilisateurs interagissant avec l’enregistrement sur un formulaire.

Les scripts d’événement utilisateur s’exécuteront quelle que soit la provenance de la demande de chargement ou de soumission, qu’il s’agisse d’un utilisateur travaillant dans l’interface utilisateur, d’une intégration tierce ou d’un autre script interne effectuant la demande.

Chaque fois qu’un processus ou un utilisateur tente de lire un enregistrement dans la base de données, l’événement beforeLoad de l’événement utilisateur est déclenché. Nous pouvons l’utiliser pour prétraiter les données, définir des valeurs par défaut ou manipuler le formulaire d’interface utilisateur avant que l’utilisateur ne le voie.

Lorsqu’un processus ou un utilisateur tente de soumettre un enregistrement à la base de données, qu’il s’agisse de la création d’un nouvel enregistrement, de la modification d’un enregistrement existant ou de la suppression d’un enregistrement, la séquence suivante se produit :

  1. Tout d’abord, avant que la requête ne parvienne réellement à la base de données, un événement beforeSubmit se déclenche. Nous pouvons utiliser cet événement, par exemple, pour nettoyer l’enregistrement avant qu’il n’entre dans la base de données.
  2. La demande est envoyée à la base de données et l’enregistrement est créé/modifié/supprimé en conséquence.
  3. Une fois le traitement de la base de données terminé, un événement afterSubmit se déclenche. Nous pouvons utiliser cet événement, par exemple, pour envoyer des notifications par e-mail de modifications ou pour synchroniser avec des systèmes tiers intégrés.

Vous pouvez également regarder cette série de vidéos qui permettent de visualiser les événements de ce type de script.

Les scripts Scheduled et Map/Reduce #

Il existe deux types de scripts que nous pouvons exploiter pour exécuter un traitement en arrière-plan à un intervalle spécifique et régulier ; il s’agit des scripts Scheduled et Map/Reduce. Notez que le type de script Map/Reduce n’est disponible que dans SuiteScript 2.0. Le script Scheduled est disponible pour les versions 1.0 et 2.0.

Le script Scheduled n’a qu’un seul événement “execute” qui se déclenche selon le calendrier que vous définissez. Par exemple, vous pouvez exécuter un script nocturne qui applique les paiements aux factures ou un script horaire qui synchronise les données avec un système externe. Lorsque l’intervalle de temps atteint, NetSuite déclenche cet événement “execute” sur votre script planifié.

Le script Map/Reduce fonctionne de manière similaire, mais une fois déclenché, il divise le traitement en quatre phases distinctes :

  1. La phase getInputData est l’endroit où vous rassemblez toutes les données d’entrée dont vous aurez besoin pour terminer le processus métier. Vous pouvez utiliser cette phase pour effectuer des recherches, lire des enregistrements et regrouper vos données dans une structure de données déchiffrable.
  2. NetSuite transmet automatiquement les résultats de votre phase getInputData à la deuxième phase, appelée map. Cette phase est chargée de regrouper logiquement vos données d’entrée pour le traitement. Par exemple, si vous appliquez des paiements à des factures, vous souhaiterez peut-être d’abord regrouper les factures par client.
  3. Les résultats de la phase “map” sont ensuite passés à la phase “reduce”, qui est l’endroit où le traitement réel a lieu. C’est là que, conformément à notre exemple, vous appliqueriez réellement les paiements aux factures.
  4. Enfin, une phase de “résumé” est appelée et contient des données concernant les résultats de tous vos traitements au cours des trois phases précédentes. Vous pouvez l’utiliser pour générer des rapports ou envoyer des e-mails indiquant que le traitement est terminé.

Le principal avantage du script Map/Reduce est que NetSuite parallélisera automatiquement le traitement pour vous sur plusieurs files d’attente, le cas échéant.

Ces deux types de scripts ont une limite de gouvernance extrêmement élevée, vous pouvez donc également les utiliser pour le traitement en masse ou les processus d’arrière-plan généralement de longue durée.

L’intervalle le plus court que l’un ou l’autre de ces types de script peut être configuré pour s’exécuter est toutes les 15 minutes.

Ces deux types de scripts peuvent également être appelés à la demande par les utilisateurs ou par d’autres scripts, si nécessaire.

La RESTlet #

Les RESTlets nous permettent de créer des points de terminaison personnalisés basés sur REST dans NetSuite ; ainsi, les RESTlets forment l’épine dorsale de presque toutes les intégrations dans NetSuite.

Les RESTlets fournissent des gestionnaires d’événements individuels pour quatre des méthodes de requête HTTP les plus couramment utilisées :

  • ‘OBTENIR’
  • ‘POSTER’
  • ‘METTRE’
  • SUPPRIMER

Lorsqu’un RESTlet reçoit une demande, il achemine la demande vers la fonction de gestionnaire d’événements appropriée en fonction de la méthode de demande HTTP utilisée.

L’authentification auprès d’un RESTlet peut être effectuée via une session utilisateur, des en-têtes HTTP ou des jetons OAuth.

Le script de mise à jour en masse #

À l’aide du script de mise à jour en masse, nous pouvons créer des mises à jour en masse personnalisées à effectuer par les utilisateurs. Cela fonctionne comme une mise à jour en masse normale, où l’utilisateur sélectionne le type de mise à jour en masse, crée une recherche qui renvoie les enregistrements à mettre à jour, puis chaque résultat de recherche est transmis individuellement au script de mise à jour en masse personnalisé.

Le script fournit un seul gestionnaire d’événements “chaque” qui reçoit l’ID interne et le type d’enregistrement de l’enregistrement à mettre à jour.

Les scripts de mise à jour en masse doivent être déclenchés manuellement par les utilisateurs via l’interface standard de mise à jour en masse.

Les scripts de mise à jour en masse ont une limite de gouvernance extrêmement élevée et sont destinés au traitement en masse personnalisé couramment utilisé.

Le script d’action du flux de travail #

Les workflows peuvent être quelque peu limités dans leurs fonctionnalités ; par exemple, les workflows ne peuvent pas interagir avec les éléments de campagne. Le type de script Action de workflow est destiné à être appelé par un workflow pour ajouter une fonctionnalité de script afin d’accomplir ce que le workflow lui-même ne peut pas.

Les actions de workflow ont un seul gestionnaire d’événements “onAction” qui sera invoqué par le workflow.

Le script d’installation du bundle #

Enfin, nous avons le type de script Bundle Installation, qui fournit plusieurs événements qui nous permettent d’interagir avec l’installation, la mise à jour et la désinstallation d’un bundle particulier. Il s’agit d’un type de script rarement rencontré, mais il est néanmoins important d’en être conscient.

L’installation du bundle inclut les gestionnaires d’événements suivants, qui devraient être assez explicites :

  • avant l'installation
  • après installation
  • avant la mise à jour
  • après mise à jour
  • avant la désinstallation
