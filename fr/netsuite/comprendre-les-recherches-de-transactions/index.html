Comprendre les recherches de transactions

  • Filtrage avec la ligne principale

  • Filtrage de sous-listes spécifiques

Sur cette page

  • Filtrage avec la ligne principale

Une compréhension approfondie du fonctionnement des recherches de transactions est une connaissance cruciale pour chaque développeur NetSuite, mais le comportement par défaut de ces recherches et le contrôle de ce comportement peuvent être assez déroutants au départ.

Références:

  • Page d’aide de NetSuite : “Utilisation de la ligne principale dans les critères de recherche de transaction”

Filtrage avec la ligne principale #

Lorsque nous ne voulons qu’un seul résultat par transaction, cela signifie que nous ne voulons que le corps, ou la ligne principale, de chaque transaction. Pour ce faire, il existe un filtre nommé “Main Line”.

En définissant le filtre * Ligne principale * sur * Oui * dans nos critères de recherche, nous disons essentiellement “Ne me montrer que les données au niveau du corps pour les transactions dans mes résultats”:

[![entrez la description de l’image ici][1]][1]

Modifier ainsi nos critères de recherche précédents nous donne maintenant le résultat unique que nous attendions à l’origine :

[![entrez la description de l’image ici][2]][2]

Si nous inversons notre filtre Ligne principale sur Non, nous disons “Montre-moi uniquement les données des sous-listes dans mes résultats”:

[![entrez la description de l’image ici][3]][3]

Pour récapituler le comportement de Main Line :

  • Avec Ligne principale défini sur Oui, nous avons reçu un résultat pour uniquement le corps de la transaction.
  • Avec Ligne principale définie sur Non, nous avons reçu trois résultats pour uniquement les données de la sous-liste de la transaction.
  • Sans aucun filtre * Ligne principale *, nous avons reçu * quatre * résultats, essentiellement la combinaison de toutes les données de corps et de sous-liste pour la transaction.

Notez que le filtre Ligne principale n’est pas pris en charge pour les recherches d’entrées de journal.

Filtrage uniquement sur ID interne #

Explorons un exemple de recherche de transaction où nous définissons un filtre pour l’ID interne d’une seule transaction :

[![Filtrer par ID interne][1]][1]

Nous avons spécifié un filtre pour afficher uniquement les résultats de la transaction avec l’ID interne 875 ; voici cette transaction :

[![Exemple de commande client][2]][2]

Nous pouvons voir qu’il s’agit d’un bon de commande avec un seul article.

Étant donné que les ID internes sont uniques pour toutes les transactions, nous ne pouvons nous attendre qu’à un seul résultat de recherche pour cette recherche. Voici le résultat de la recherche :

[![Résultats inattendus][3]][3]

Au lieu du résultat unique que nous attendons, nous obtenons quatre résultats. De plus, chaque résultat a exactement le même identifiant interne. Comment est-ce possible?

Pour comprendre ce qui se passe ici, nous devons rappeler que les données stockées dans les enregistrements NetSuite sont divisées en deux catégories :

  1. Données corporelles : données stockées dans des champs autonomes de l’enregistrement (par exemple, date, représentant commercial, numéro de document, code de coupon)
  2. Données de sous-liste : données stockées dans des listes au sein de chaque enregistrement, généralement affichées dans des sous-onglets de l’interface utilisateur (par exemple, les articles d’un bon de commande)

Les transactions contiennent plusieurs sous-listes de données, y compris :

  • éléments de campagne
  • Informations sur la livraison
  • informations fiscales
  • Détails COGS (coût des marchandises vendues)

Dans ces résultats de recherche, NetSuite nous montre en fait un résultat pour le corps de la transaction, puis d’autres résultats pour les données des différentes sous-listes de cette même transaction.

Remarquez la colonne dans nos résultats de recherche simplement nommée avec un astérisque (*). Notez également que l’un des résultats a un astérisque rempli dans cette colonne tandis que les autres sont vides. Cette colonne indique quel résultat de recherche représente le corps de la transaction, qui est également appelé la ligne principale de la transaction.

Il y a des moments où vous souhaiterez que les recherches de transactions n’affichent que les données de la ligne principale et des moments où vous ne voudrez que les détails au niveau de la ligne. Les exemples restants montrent comment contrôler ce qui apparaît dans nos résultats.

Filtrage de sous-listes spécifiques #

Rappelez-vous que chaque transaction contient plusieurs sous-listes de données. Maintenant que nous pouvons afficher uniquement les données de sous-liste à l’aide de Ligne principale, nous pouvons affiner davantage nos résultats de recherche pour des données de sous-liste spécifiques.

La plupart des sous-listes incluses dans les résultats de transaction ont un filtre de recherche correspondant pour basculer si elles sont incluses dans vos résultats :

  • Utilisez le filtre Ligne d’expédition pour contrôler les données de la sous-liste d’expédition
  • Utilisez le filtre Tax Line pour contrôler les données de la sous-liste Taxe
  • Utilisez le filtre Ligne COGS pour contrôler les données de la sous-liste COGS

Chacun de ces filtres se comporte comme Ligne principale ou tout autre filtre de case à cocher : Oui pour inclure ces données, Non pour les exclure de vos résultats.

Notez qu’il n’y a pas de filtre pour Ligne d’article pour contrôler les données de la sous-liste d’articles. Essentiellement, pour dire “Montrez-moi uniquement les données de la sous-liste des éléments”, nous devons spécifier tous ces filtres susmentionnés comme Non dans nos critères :

[![entrez la description de l’image ici][1]][1]

Avec ces critères, votre recherche renverra un résultat par ligne d’article sur chaque transaction correspondante.

À mon avis, ce filtre manquant est une lacune majeure dans la fonctionnalité de recherche qui devrait être corrigée ; il serait beaucoup plus facile et plus cohérent d’avoir simplement un filtre Item Line is Yes. Jusque-là, c’est ainsi que vous devez spécifier que vous ne voulez que des données d’article dans vos résultats de transaction.
