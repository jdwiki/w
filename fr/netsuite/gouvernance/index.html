Gouvernance #

“Gouvernance” est le nom donné au système de NetSuite pour détecter et arrêter les scripts de longue durée, incontrôlables ou gourmands en ressources.

Chaque type de script a des limites de gouvernance qu’il ne peut pas dépasser, et il existe quatre types de limites de gouvernance en place pour chaque type de script.

  • Limite d’utilisation de l’API
  • Limite du nombre d’instructions
  • Limite de temporisation
  • Limite d’utilisation de la mémoire

Si un script dépasse sa limite de gouvernance dans l’un de ces quatre domaines, NetSuite lèvera une exception uncatchable et terminera le script immédiatement.

Limite d’utilisation de l’API #

NetSuite limite l’utilisation de l’API de vos scripts avec un système basé sur des “unités d’utilisation”. Certains appels d’API NetSuite, en particulier ceux qui effectuent une action de lecture ou d’écriture sur la base de données, coûtent un nombre spécifique d’unités à chaque fois qu’ils sont invoqués. Chaque type de script possède alors un nombre maximum d’unités pouvant être utilisées lors de chaque exécution du script.

  • Si un script dépasse sa limite d’utilisation de l’API, NetSuite termine le script en lançant une erreur SSS_USAGE_LIMIT_EXCEEDED.*

Vous trouverez ci-dessous quelques exemples de coûts unitaires pour des opérations courantes. Pour une liste exhaustive des coûts de gouvernance, consultez l’article intitulé “API Governance” dans l’aide de NetSuite.

Opération Coût unitaire
Chargement d’une recherche enregistrée ‘5’
Récupération des résultats de la recherche ‘10’
Planifier une tâche ‘10’
Demander une URL ‘10’
Envoi d’un e-mail ‘10’
Création d’un enregistrement personnalisé ‘2’
Créer une fiche Employé ‘5’
Création d’un enregistrement de commande client ‘10’
Enregistrer un enregistrement personnalisé 4
Enregistrer un enregistrement de contact ‘10’
Enregistrement d’un enregistrement de bon de commande ‘20’

Différentes opérations utilisent différentes quantités d’unités et certaines opérations coûtent un montant différent en fonction du type d’enregistrement utilisé. Plus le nombre d’unités qu’une fonction coûte est élevé, généralement plus elle prendra de temps à s’exécuter.

Les transactions sont les types d’enregistrement les plus volumineux, donc leur utilisation coûte le plus grand nombre d’unités. À l’inverse, les enregistrements personnalisés sont très légers et ne coûtent donc pas beaucoup d’unités. Les enregistrements NetSuite standard qui ne sont pas des transactions, comme les clients, les employés ou les contacts, se situent entre les deux en termes de coût.

Voici les limites d’utilisation par type de script :

Type de script Limite d’utilisation
Client 1 000
Événement utilisateur 1 000
Suitelet 1 000
Portlet 1 000
Action de flux de travail 1 000
RESTlet 5 000
Planifié 10 000
Carte/Réduire 10 000
Installation groupée 10 000
Mise à jour de masse 10 000 par enregistrement

Limites de temporisation et de nombre d’instructions #

NetSuite utilise également le système de gouvernance pour détecter et arrêter les scripts incontrôlables en utilisant un mécanisme de temporisation et un compteur d’instructions.

Si un script prend trop de temps à s’exécuter, NetSuite l’arrêtera en lançant une erreur SSS_TIME_LIMIT_EXCEEDED.

De plus, les scripts incontrôlables peuvent être détectés et arrêtés en fonction de leur “nombre d’instructions”. Si les limites du nombre d’instructions définies sont dépassées, NetSuite arrêtera le script en lançant une erreur SSS_INSTRUCTION_COUNT_EXCEEDED.

Il n’y a malheureusement aucune documentation d’aide qui définit :

  • le délai d’attente pour chaque type de script
  • les limites du nombre d’instructions pour chaque type de script
  • ce qui constitue une seule “instruction”

Il est simplement important de savoir que si vous rencontrez l’erreur SSS_TIME_LIMIT_EXCEEDED ou l’erreur SSS_INSTRUCTION_COUNT_EXCEEDED dans l’un de vos scripts, vous avez un traitement qui prend trop de temps. Concentrez votre enquête sur vos structures de boucle pour déterminer où des optimisations peuvent être faites.

Limite d’utilisation de la mémoire #

Si votre script dépasse la limite d’utilisation de la mémoire, NetSuite mettra fin à votre script en lançant une erreur SSS_MEMORY_USAGE_EXCEEDED.

Chaque variable déclarée, chaque fonction définie, chaque objet stocké contribue à l’utilisation de la mémoire de votre script.

Le script planifié et le script de mappage/réduction ont documenté des limites de mémoire de 50 Mo. Il existe également une limite documentée de 10 Mo pour la taille de toute chaîne transmise à ou renvoyée par un RESTlet. Il n’y a pas d’autre documentation sur les limites spécifiques pour un script donné.

Combien d’unités me reste-t-il ? #

Dans SuiteScript 1.0, utilisez nlobjContext.getRemainingUsage() pour récupérer les unités restantes. Une référence nlobjContext est récupérée à l’aide de la fonction globale nlapiGetContext.

// 1.0
var context = nlapiGetContext();
nlapiLogExecution("DEBUG", "Governance Monitoring", "Remaining Usage = " + context.getRemainingUsage());

nlapiSearchRecord("transaction"); // uses 10 units
nlapiLogExecution("DEBUG", "Governance Monitoring", "Remaining Usage = " + context.getRemainingUsage());

Dans SuiteScript 2.0, utilisez la méthode getRemainingUsage de l’objet Script du module N/runtime.

// 2.0
require(["N/log", "N/runtime", "N/search"], function (log, runtime, s) {
    var script = runtime.getCurrentScript();
    log.debug({
        "title": "Governance Monitoring",
        "details": "Remaining Usage = " + script.getRemainingUsage()
    });

    s.load({"id":"customsearch_mysearch"}); // uses 5 units
    log.debug({
        "title": "Governance Monitoring",
        "details": "Remaining Usage = " + script.getRemainingUsage()
    });
});
