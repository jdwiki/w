Au début des tests unitaires, toutes sortes de questions se posent :

Qu’est-ce que les tests unitaires ? Qu’est-ce qu’une configuration et un démontage ? Comment gérer les dépendances ? Pourquoi faire des tests unitaires ? Comment faire de bons tests unitaires ?

Cet article répondra à toutes ces questions, afin que vous puissiez commencer les tests unitaires dans la langue de votre choix.

Qu’est-ce que les tests unitaires ? #

Les tests unitaires consistent à tester le code pour s’assurer qu’il effectue la tâche qu’il est censé effectuer. Il teste le code au niveau le plus bas possible - les méthodes individuelles de vos classes.

Qu’est-ce qu’une unité ? #

Tout module discret de code pouvant être testé isolément. La plupart du temps les classes et leurs méthodes. Cette classe est généralement appelée “classe sous test” (CUT) ou “système sous test” (SUT)

La différence entre les tests unitaires et les tests d’intégration

Les tests unitaires consistent à tester une seule classe de manière isolée, complètement séparée de ses dépendances réelles. Les tests d’intégration consistent à tester une seule classe avec une ou plusieurs de ses dépendances réelles.

L’installation et le démontage #

Une fois effectuée, la méthode SetUp est exécutée avant chaque test unitaire et la méthode TearDown après chaque test.

En général, vous ajoutez toutes les étapes préalables dans la configuration et toutes les étapes de nettoyage dans le démontage. Mais vous ne faites ces méthodes que si ces étapes sont nécessaires pour chaque test. Si ce n’est pas le cas, ces étapes sont prises dans le cadre des tests spécifiques de la section “organiser”.

Comment gérer les dépendances #

Souvent, une classe dépend d’autres classes pour exécuter ses méthodes. Pour pouvoir ne pas dépendre de ces autres classes, vous devez les simuler. Vous pouvez créer ces classes vous-même ou utiliser un framework d’isolation ou de maquette. Un framework d’isolation est une collection de code qui permet de créer facilement de fausses classes.

Faux cours #

Toute classe qui fournit des fonctionnalités suffisantes pour prétendre qu’il s’agit d’une dépendance nécessaire à un CUT. Il existe deux types de contrefaçons : les stubs et les mocks.

  • Un stub : Un faux qui n’a aucun effet sur la réussite ou l’échec du test et qui existe uniquement pour permettre au test de se dérouler.
  • Un simulacre : un faux qui garde une trace du comportement du CUT et réussit ou échoue le test en fonction de ce comportement.

Pourquoi faire des tests unitaires ? #

**1. Les tests unitaires trouveront des bogues **

Lorsque vous écrivez une suite complète de tests qui définissent ce que le comportement est pour une classe donnée, tout ce qui ne se comporte pas comme prévu est révélé.

2. Les tests unitaires éloigneront les bugs

Effectuez une modification qui introduit un bogue et vos tests peuvent le révéler la prochaine fois que vous exécuterez vos tests.

3. Les tests unitaires font gagner du temps

L’écriture de tests unitaires permet de s’assurer que votre code fonctionne comme prévu dès le départ. Les tests unitaires définissent ce que votre code doit faire et ainsi vous ne passerez pas de temps à écrire du code qui fait les choses ne devrait pas faire. Personne ne vérifie le code qu’ils croient ne pas fonctionner et vous devez faire quelque chose pour vous faire croire que cela fonctionne. Dépenser ce temps pour écrire des tests unitaires.

4. Les tests unitaires offrent la tranquillité d’esprit

Vous pouvez exécuter tous ces tests et savoir que votre code fonctionne tel quel censé. Connaître l’état de votre code, qu’il fonctionne, et que vous pouvez le mettre à jour et l’améliorer sans crainte c’est une très bonne chose.

5. Les tests unitaires documentent la bonne utilisation d’une classe

Les tests unitaires deviennent des exemples simples du fonctionnement de votre code, de ce qu’il est devrait faire et la bonne façon d’utiliser votre code testé.

Règles générales pour les tests unitaires #

1. Pour la structure d’un test unitaire, suivez la règle AAA

Organiser :

Mettre en place la chose à tester. Comme des variables, des champs et des propriétés pour permettre l’exécution du test ainsi que le résultat attendu.

Agir : Appelez réellement la méthode que vous testez

Affirmer :

Appelez le cadre de test pour vérifier que le résultat de votre “acte” est ce qui était attendu.

2. Testez une chose à la fois de manière isolée

Toutes les classes doivent être testées isolément. Ils ne doivent pas dépendre autre chose que les mocks et les stubs. Ils ne doivent pas dépendre du résultats d’autres tests.

**3. Rédigez d’abord des tests simples “juste au milieu” **

Les premiers tests que vous écrivez doivent être les tests les plus simples. Ils devraient être ceux qui illustrent fondamentalement et facilement la fonctionnalité que vous essayez d’écrire. Ensuite, une fois ces tests réussis, vous devriez commencer écrire les tests les plus compliqués qui testent les bords et les limites de votre code.

4. Ecrire des tests qui testent les bords

Une fois les bases testées et que vous connaissez vos fonctionnalités de base fonctionne, vous devriez tester les bords. Une bonne série de tests explorera la bords extérieurs de ce qui pourrait arriver à une méthode donnée.

Par exemple :

  • Que se passe-t-il en cas de débordement ?
  • Que se passe-t-il si les valeurs vont à zéro ou en dessous ?
  • Et s’ils vont à MaxInt ou MinInt ?
  • Et si vous créiez un arc de 361 degrés ?
  • Que se passe-t-il si vous passez une chaîne vide ?
  • Que se passe-t-il si une chaîne fait 2 Go ?

5. Tester au-delà des frontières

Les tests unitaires doivent tester les deux côtés d’une frontière donnée. Se déplacer à travers les limites sont des endroits où votre code peut échouer ou fonctionner dans manières imprévisibles.

6. Si vous le pouvez, testez tout le spectre

Si c’est pratique, testez l’ensemble des possibilités de pour votre fonctionnalité. S’il s’agit d’un type énuméré, testez le fonctionnalité avec chacun des éléments de l’énumération. Il pourrait être impossible de tester toutes les possibilités, mais si vous pouvez tester chaque possibilité, faites-le.

**sept. Si possible, couvrez chaque chemin de code **

Celui-ci est également difficile, mais si votre code est conçu pour tests, et que vous utilisez un outil de couverture de code, vous pouvez vous assurer que chaque ligne de votre code est couverte par des tests unitaires au moins une fois. Couvrir chaque chemin de code ne garantit pas qu’il n’y a pas de bugs, mais cela vous donne sûrement des informations précieuses sur l’état de chaque ligne de code.

8. Rédigez des tests qui révèlent un bogue, puis corrigez-le

Si vous trouvez un bogue, écrivez un test qui le révèle. Ensuite, vous van facilement corrigez le bug en déboguant le test. Ensuite, vous avez une belle régression testez pour vous assurer que si le bogue revient pour une raison quelconque, vous savoir tout de suite. Il est vraiment facile de corriger un bogue lorsque vous avez un simple, test simple à exécuter dans le débogueur.

Un avantage secondaire ici est que vous avez testé votre test. Parce que vous avez vu le test échouer et puis quand vous l’avez vu passer, vous savez que le test est valide dans la mesure où il a été prouvé qu’il fonctionne correctement. Cette en fait un test de régression encore meilleur.

9. Rendre chaque test indépendant l’un de l’autre

Les tests ne doivent jamais dépendre les uns des autres. Si vos tests doivent s’exécuter dans un certain ordre, il faut changer les tests.

dix. Rédigez une assertion par test

Vous devez écrire une assertion par test. Si vous ne pouvez pas faire cela, alors réfractaire votre code afin que vos événements SetUp et TearDown soient utilisés pour créer correctement l’environnement pour que chaque test puisse être exécuté individuellement.

11. Nommez clairement vos tests. N’ayez pas peur des noms longs

Puisque vous faites une affirmation par test, chaque test peut finir par être très spécifique. N’ayez donc pas peur d’utiliser des noms de test longs et complets.

Un long nom complet vous permet de savoir immédiatement quel test a échoué et exactement ce que le test essayait de faire.

Des tests longs et clairement nommés peuvent également documenter vos tests. Une épreuve nommée “DividedByZeroShouldThrowException” documente exactement ce que le code fait quand vous essayez de diviser par zéro.

**12. Tester que chaque exception déclenchée est réellement déclenchée **

Si votre code lève une exception, écrivez un test pour vous assurer que chaque exception que vous soulevez est en fait levée quand elle est censée le faire.

13. Évitez l’utilisation de CheckTrue ou Assert.IsTrue

Evitez de vérifier une condition booléenne. Par exemple, au lieu de cela, si vérifier si deux choses sont égales avec CheckTrue ou Assert.IsTrue, utilisez CheckEquals ou Assert.IsEqual à la place. Pourquoi? À cause de ce:

CheckTrue (prévu, réel) Cela rapportera quelque chose comme : “Certains tests ont échoué : la valeur attendue était vraie, mais le résultat réel était faux.”

Cela ne vous dit rien.

CheckEquals (prévu, réel)

Cela vous dira quelque chose comme : “Certains tests ont échoué : 7 attendus mais le résultat réel était de 3.”

N’utilisez CheckTrue ou Assert.IsTrue que lorsque votre valeur attendue est en fait une condition booléenne.

14. Exécutez vos tests en permanence

Exécutez vos tests pendant que vous écrivez du code. Vos tests doivent s’exécuter rapidement, vous permettant de les exécuter même après des modifications mineures. Si vous ne pouvez pas courir vos tests dans le cadre de votre processus de développement normal puis quelque chose va mal. Les tests unitaires sont censés s’exécuter presque instantanément. Si ils ne le sont pas, c’est probablement parce que vous ne les utilisez pas isolement.

15. Exécutez vos tests dans le cadre de chaque build automatisé

Tout comme vous devriez exécuter des tests pendant que vous développez, ils devraient également faire partie intégrante de votre processus d’intégration continue. Un raté test devrait signifier que votre build est cassé. Ne laissez pas les tests échouer s’attarder. Considérez cela comme un échec de construction et corrigez-le immédiatement.

Exemple de test unitaire simple en C# #

Pour cet exemple, nous allons tester la méthode sum d’une simple calculatrice.

Dans cet exemple nous allons tester l’application : ApplicationToTest. Celui-ci a une classe appelée Calc. Cette classe a une méthode Sum().

La méthode Sum() ressemble à ceci :

public void Sum(int a, int b)
{
    return a + b;
}

Le test unitaire pour tester cette méthode ressemble à ceci :

[Testclass]
public class UnitTest1
{
    [TestMethod]
    public void TestMethod1()
    {
        //Arrange
        ApplicationToTest.Calc ClassCalc = new ApplicationToTest.Calc();
        int expectedResult = 5;

        //Act
        int result = ClassCalc.Sum(2,3);

        //Assert
        Assert.AreEqual(expectedResult, result);
    }
}
