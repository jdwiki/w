Injection de dépendance

Une approche qui peut être adoptée pour écrire un logiciel consiste à créer des dépendances au fur et à mesure de leurs besoins. C’est une manière assez intuitive d’écrire un programme et c’est la manière que la plupart des gens auront tendance à apprendre, en partie parce qu’elle est facile à suivre. L’un des problèmes avec cette approche est qu’elle peut être difficile à tester. Considérez une méthode qui effectue un traitement basé sur la date actuelle. La méthode peut contenir du code comme celui-ci :

if (DateTime.Now.Date > processDate)
{
    // Do some processing
}

Le code dépend directement de la date actuelle. Cette méthode peut être difficile à tester car la date actuelle ne peut pas être facilement manipulée. Une approche pour rendre le code plus testable consiste à supprimer la référence directe à la date actuelle et à la place de fournir (ou d’injecter) la date actuelle à la méthode qui effectue le traitement. Cette injection de dépendance peut faciliter considérablement le test des aspects du code en utilisant [test double][1] pour simplifier l’étape de configuration du test unitaire.

Systèmes IOC

Un autre aspect à considérer est la durée de vie des dépendances ; dans le cas où la classe crée elle-même ses propres dépendances (aussi appelées invariants), elle est alors chargée de les éliminer. L’injection de dépendances inverse cela (et c’est pourquoi nous nous référons souvent à une bibliothèque d’injection comme un système “d’inversion de contrôle”) et signifie qu’au lieu que la classe soit responsable de la création, de la gestion et du nettoyage de ses dépendances, un agent externe (dans ce cas, le système IoC) le fait à la place.

Cela rend beaucoup plus simple d’avoir des dépendances qui sont partagées entre les instances de la même classe ; par exemple, considérons un service qui récupère des données à partir d’un point de terminaison HTTP pour qu’une classe les consomme. Étant donné que ce service est sans état (c’est-à-dire qu’il n’a pas d’état interne), nous n’avons donc vraiment besoin que d’une seule instance de ce service dans notre application. Bien qu’il soit possible (par exemple, en utilisant une classe statique) de le faire manuellement, il est beaucoup plus simple de créer la classe et de dire au système IoC qu’elle doit être créée en tant que Singleton, où une seule instance de la classe chaque existe.

Un autre exemple serait les contextes de base de données utilisés dans une application Web, où un nouveau contexte est requis par requête (ou thread) et non par instance d’un contrôleur ; cela permet au contexte d’être injecté dans chaque couche exécutée par ce thread, sans avoir à le faire passer manuellement.

Cela libère les classes consommatrices de la gestion des dépendances.

[1] : https://www.wikiod.com/fr/unit-testing/doubles-dessai

Injection de constructeur #

L’injection de constructeur est le moyen le plus sûr d’injecter des dépendances dont dépend toute une classe. Ces dépendances sont souvent appelées * invariants *, car une instance de la classe ne peut pas être créée sans les fournir. En exigeant que la dépendance soit injectée à la construction, il est garanti qu’un objet ne peut pas être créé dans un état incohérent.

Considérez une classe qui doit écrire dans un fichier journal dans des conditions d’erreur. Il dépend d’un ILogger, qui peut être injecté et utilisé si nécessaire.

public class RecordProcessor
{
    readonly private ILogger _logger;

    public RecordProcessor(ILogger logger)
    {
        _logger = logger;
    }

    public void DoSomeProcessing() {
        // ...
        _logger.Log("Complete");
    }
}

Parfois, lors de l’écriture de tests, vous pouvez remarquer que le constructeur nécessite plus de dépendances qu’il n’en faut réellement pour un cas testé. Plus vous avez de tels tests, plus il est probable que votre classe enfreigne le principe de responsabilité unique (SRP). C’est pourquoi ce n’est pas une très bonne pratique de définir le comportement par défaut pour toutes les simulations de dépendances injectées lors de la phase d’initialisation de la classe de test, car cela peut masquer le signal d’avertissement potentiel.

Le test unitaire pour cela ressemblerait à ceci :

[Test]
public void RecordProcessor_DependencyInjectionExample()
{
    ILogger logger = new FakeLoggerImpl(); //or create a mock by a mocking Framework
    
    var sut = new RecordProcessor(logger); //initialize with fake impl in testcode

    Assert.IsTrue(logger.HasCalledExpectedMethod());
}

Injection de propriété #

L’injection de propriété permet de mettre à jour les dépendances d’une classe après sa création. Cela peut être utile si vous souhaitez simplifier la création d’objets, tout en permettant aux dépendances d’être remplacées par vos tests avec des doublons de test.

Considérez une classe qui doit écrire dans un fichier journal dans une condition d’erreur. La classe sait comment construire un Logger par défaut, mais permet de le remplacer par l’injection de propriété. Cependant, il convient de noter qu’en utilisant l’injection de propriété de cette manière, vous couplez étroitement cette classe avec une implémentation exacte de ILogger qui est ConcreteLogger dans cet exemple donné. Une solution de contournement possible pourrait être une usine qui renvoie l’implémentation ILogger nécessaire.

public class RecordProcessor
{
    public RecordProcessor()
    {
        Logger = new ConcreteLogger();
    }

    public ILogger Logger { get; set; }

    public void DoSomeProcessing()
    {
        // ...
        _logger.Log("Complete");
    }
}

Dans la plupart des cas, Constructor Injection est préférable à Property Injection car il offre de meilleures garanties sur l’état de l’objet immédiatement après sa construction.

Méthode d’injection #

L’injection de méthode est une manière fine d’injecter des dépendances dans le traitement. Considérez une méthode qui effectue un traitement basé sur la date actuelle. La date actuelle est difficile à changer à partir d’un test, il est donc beaucoup plus facile de passer une date dans la méthode que vous souhaitez tester.

public void ProcessRecords(DateTime currentDate)
{
    foreach(var record in _records) 
    {
        if (currentDate.Date > record.ProcessDate)
        {
            // Do some processing
        }
    }
}

Conteneurs / Frameworks DI #

Alors que l’extraction des dépendances de votre code afin qu’elles puissent être injectées rend votre code plus facile à tester, cela pousse le problème plus haut dans la hiérarchie et peut également entraîner des objets difficiles à construire. Divers frameworks d’injection de dépendances / inversion de conteneurs de contrôle ont été écrits pour aider à surmonter ce problème. Ceux-ci permettent d’enregistrer les mappages de types. Ces enregistrements sont ensuite utilisés pour résoudre les dépendances lorsque le conteneur est invité à construire un objet.

Considérez ces classes:

public interface ILogger {
    void Log(string message);
}

public class ConcreteLogger : ILogger
{
    public ConcreteLogger()
    {
        // ...
    }
    public void Log(string message)
    {
        // ...
    }
}
public class SimpleClass
{
    public SimpleClass()
    {
        // ...
    }
}

public class SomeProcessor
{
    public SomeProcessor(ILogger logger, SimpleClass simpleClass)
    {
        // ...
    }
}

Pour construire SomeProcessor, une instance de ILogger et de SimpleClass est requise. Un conteneur comme Unity permet d’automatiser ce processus.

Le conteneur doit d’abord être construit, puis les mappages sont enregistrés avec lui. Cette opération n’est généralement effectuée qu’une seule fois dans une application. La zone du système où cela se produit est communément appelée * racine de composition *

// Register the container
var container = new UnityContainer();

// Register a type mapping.  This allows a `SimpleClass` instance
// to be constructed whenever it is required.
container.RegisterType<SimpleClass, SimpleClass>();

// Register an instance.  This will use this instance of `ConcreteLogger` 
// Whenever an `ILogger` is required.
container.RegisterInstance<ILogger>(new ConcreteLogger());

Une fois le conteneur configuré, il peut être utilisé pour créer des objets, en résolvant automatiquement les dépendances selon les besoins :

var processor = container.Resolve<SomeProcessor>();
