Bonjour le monde #

Il y a deux choses que vous devez savoir lors de l’écriture d’une application “hello world” en Erlang :

  1. Le code source est écrit dans le langage de programmation erlang en utilisant l’éditeur de texte de votre choix
  2. L’application est ensuite exécutée dans la machine virtuelle erlang. Dans cet exemple, nous allons interagir avec la machine virtuelle erlang via le shell erlang.

D’abord le code source de l’application : #

Créez un nouveau fichier “hello.erl” contenant les éléments suivants :

-module(hello).
-export([hello_world/0]).

hello_world() ->
  io:format("Hello, World!~n", []).

Voyons rapidement ce que cela signifie :

  • -module(hello). Toutes les fonctions erlang existent à l’intérieur d’un module. Les modules sont ensuite utilisés pour créer des applications, qui sont une collection de modules. Cette première ligne sert à identifier ce module, à savoir hello. Les modules peuvent être comparés aux packages de Java
  • -export([hello_world/0]). Indique au compilateur quelles fonctions rendre “publiques” (par rapport aux langages OO), et l’arité de la fonction concernée. L’arité est le nombre d’arguments pris par la fonction. Étant donné qu’en erlang, une fonction avec 1 argument est considérée comme une fonction différente d’une fonction avec 2 arguments, même si le nom peut être exactement le même. C’est-à-dire que hello_world/0 est une fonction complètement différente de hello_world/1 par exemple.
  • hello_world() C’est le nom de la fonction. Le -> indique la transition vers l’implémentation (corps) de la fonction. Cela peut être lu comme “hello_world() est défini comme …”. Notez que hello_world() (sans arguments) est identifié par hello_world/0 dans la VM, et hello_world(Some_Arg) comme hello_world/1.
  • io:format("Hello, World!~n", []) A partir du module io, la fonction format/2 est appelée, qui est la fonction pour la sortie standard. ~n est un spécificateur de format qui signifie imprimer une nouvelle ligne. Le [] est une liste de variables à imprimer indiquées par des spécificateurs de format dans la chaîne de sortie, qui dans ce cas n’est rien.
  • Toutes les instructions erlang doivent se terminer par un . (point).

En Erlang, le résultat de la dernière instruction d’une fonction est renvoyé.

Maintenant, lançons notre application : #

Démarrez le shell erlang à partir du même répertoire que le fichier hello.erl :

$ erl

Vous devriez obtenir une invite qui ressemble à ceci (votre version peut être différente) :

Eshell V8.0  (abort with ^G)
1>

Saisissez maintenant les commandes suivantes :

1> c(hello).
{ok,hello}
2> hello:hello_world().
Hello, World!
ok

Passons en revue chaque ligne une par une :

  • c(hello) - cette commande appelle la fonction c sur un atome hello. Cela indique effectivement à Erlang de trouver le fichier hello.erl, de le compiler dans un module (un fichier nommé hello.beam sera généré dans le répertoire) et de le charger dans l’environnement.
  • {ok, hello} - c’est le résultat de l’appel de la fonction c ci-dessus. C’est un tuple contenant un atome ‘ok’ et un atome ‘hello’. Les fonctions Erlang renvoient généralement soit {ok, Something} ou {error, Reason}.
  • hello:hello_world() - cela appelle une fonction hello_world() du module hello.
  • Hello, World! - c’est ce que notre fonction imprime.
  • ok - c’est ce que notre fonction a renvoyé. Comme Erlang est un langage de programmation fonctionnel, chaque fonction renvoie quelque chose. Dans notre cas, même si nous n’avons rien renvoyé dans hello_world(), le dernier appel de cette fonction était à io:format(...) et cette fonction a renvoyé ok, qui est à son tour ce que notre fonction est retournée.

Compréhension de la liste #

Les compréhensions de liste sont une construction syntaxique pour créer une liste basée sur des listes existantes. En erlang une compréhension de liste a la forme [Expr || Qualificatif1, ..., QualificatifN]. Où les qualificateurs sont soit des générateurs Pattern <- ListExpr ou un filtre comme integer(X) évaluant soit true ou false.

L’exemple suivant montre une compréhension de liste avec un générateur et deux filtres.

[X || X <- [1,2,a,3,4,b,5,6], integer(X), X > 3].

Le résultat est une liste contenant uniquement des entiers supérieurs à 3.

[4,5,6]

Modules #

Un module erlang est un fichier avec quelques fonctions regroupées. Ce fichier a généralement l’extension .erl.

Un module “Hello World” avec le nom hello.erl est illustré ci-dessous

-module(hello).
-export([hello_world/0]).

hello_world() ->
  io:format("Hello, World!~n", []).

Dans le fichier, il est obligatoire de déclarer le nom du module. Comme indiqué précédemment à la ligne 1. Le nom du module et le nom du fichier avant l’extension “.erl” doivent être identiques.

Fonction #

La fonction est un ensemble d’instructions regroupées. Ces instructions groupées exécutent ensemble certaines tâches. En erlang, toutes les fonctions renverront une valeur lorsqu’elles seront appelées.

Voici un exemple de fonction qui additionne deux nombres

add(X, Y)-> X + Y.

Cette fonction effectue une opération d’addition avec les valeurs X et Y et renvoie le résultat. La fonction peut être utilisée comme ci-dessous

add(2,5).

Les déclarations de fonction peuvent être constituées de plusieurs clauses, séparées par un point-virgule. Les Arguments de chacune de ces clauses sont évalués par correspondance de modèle. La fonction suivante renverra ’tuple’ si l’argument est un tuple dans le formulaire : {test, X} où X peut être n’importe quelle valeur. Elle renverra ’list’, si l’Argument est une liste de longueur 2 sous la forme [“test”, X], et elle renverra ‘{error, “Reason”}’ dans tous les autres cas :

function({test, X}) -> tuple;
function(["test", X]) -> list;
function(_) -> {error, "Reason"}.

Si l’argument n’est pas un tuple, la deuxième clause sera évaluée. Si l’argument n’est pas une liste, la troisième clause sera évaluée.

Les déclarations de fonction peuvent consister en ce qu’on appelle des ‘gardes’ ou des ‘séquences de garde’. Ces gardes sont des expressions qui limitent l’évaluation d’une fonction. Une fonction avec des gardes n’est exécutée que lorsque toutes les expressions de garde donnent une valeur vraie. Plusieurs gardes peuvent être séparés par un point-virgule.

function_name(Argument) when Guard1; Guard2; ... GuardN -> (...).

La fonction ’nom_fonction’ ne sera évaluée que lorsque la séquence de garde est vraie. La fonction suivante ne renverra vrai que si l’argument ‘X’ est dans la bonne plage (0..15):

in_range(X) when X>=0; X<16 -> true;
in_range(_) -> false.

Démarrage et arrêt du shell Erlang #

Démarrage du shell Erlang

Sur un système UNIX, vous démarrez le shell Erlang à partir d’une invite de commande avec la commande erl

Exemple:

$ erl
Erlang/OTP 18 [erts-7.0] [source] [64-bit] [smp:8:8] [async-threads:10] [hipe] [kernel-poll:false]

Eshell V7.0  (abort with ^G)
1>

Le texte qui s’affiche lorsque vous démarrez le shell vous donne des informations sur la version d’Erlang que vous utilisez ainsi que d’autres informations utiles sur le système erlang.

Pour démarrer le shell sous Windows, cliquez sur l’icône Erlang dans le menu Démarrer de Windows.

Arrêter le shell Erlang

Pour une sortie contrôlée du shell erlang vous tapez :

Erlang/OTP 18 [erts-7.0] [source] [64-bit] [smp:8:8] [async-threads:10] [hipe] [kernel-poll:false]

Eshell V7.0  (abort with ^G)
1> q().

Vous pouvez également quitter le shell Erlang en appuyant sur Ctrl+C sur les systèmes UNIX ou Ctrl+Break sur Windows, ce qui vous amène à l’invite suivante :

Erlang/OTP 18 [erts-7.0] [source] [64-bit] [smp:8:8] [async-threads:10] [hipe] [kernel-poll:false]

Eshell V7.0  (abort with ^G)
1> 
BREAK: (a)bort (c)ontinue (p)roc info (i)nfo (l)oaded
       (v)ersion (k)ill (D)b-tables (d)istribution

Si vous appuyez ensuite sur a (pour abandonner), vous quitterez directement le shell.

D’autres manières de quitter le shell erlang sont : init:stop() qui fait la même chose que q() ou erlang:halt().

Correspondance de modèle #

L’une des opérations les plus courantes en erlang est la correspondance de modèles. Il est utilisé lors de l’attribution d’une valeur à une variable, dans les déclarations de fonction et dans les structures de flux de contrôle telles que les instructions case et receive. Une opération de correspondance de modèle nécessite au moins 2 parties : un modèle et un terme par rapport auquel le modèle est mis en correspondance.

Une affectation de variable dans erlang ressemble à ceci :

X = 2.

Dans la plupart des langages de programmation, la sémantique de cette opération est simple : liez une valeur (‘2’) à un nom de votre choix (la variable – dans ce cas ‘X’). Erlang a une approche légèrement différente : faites correspondre le motif sur le côté gauche (‘X’) au terme sur le côté droit (‘2’). Dans ce cas, l’effet est le même : la variable ‘X’ est désormais liée à la valeur ‘2’. Cependant, avec la correspondance de modèles, vous pouvez effectuer des affectations plus structurées.

{Type, Meta, Doc} = {document, {author, "Alice"}, {text, "Lorem Ipsum"}}.

Cette opération d’appariement est effectuée en analysant la structure du terme de droite et en appliquant toutes les variables du côté gauche aux valeurs appropriées du terme, de sorte que le côté gauche soit égal au côté droit. Dans cet exemple, Type est lié au terme : document, Meta à {author, "Alice"} et Doc à {text, "Lorem Ipsum"}. Dans cet exemple particulier, les variables : Type, Meta et Doc sont supposées être unbound, de sorte que chaque variable puisse être utilisée.

Des correspondances de modèles peuvent également être construites à l’aide de variables liées.

Identifier = error.

La variable Identifier est maintenant liée à la valeur error. L’opération de correspondance de modèle suivante fonctionne, car la structure correspond et la variable liée “Identifier” a la même valeur que la partie droite appropriée du terme.

{Identifier, Reason} = {error, "Database connection timed out."}.

Une opération de correspondance de modèle échoue lorsqu’il existe une incompatibilité entre le terme de droite et le modèle de gauche. La correspondance suivante échouera, car Identifier est lié à la valeur error, qui n’a pas d’expression appropriée sur le terme de droite.

{Identifier, Reason} = {fail, "Database connection timed out."}.
> ** exception error: no match of right hand side value {fail,"Database ..."}
