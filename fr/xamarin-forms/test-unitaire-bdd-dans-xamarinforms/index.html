Specflow simple pour tester les commandes et la navigation avec NUnit Test Runner #

Pourquoi avons nous besoin de ça? #

La façon actuelle de faire des tests unitaires dans Xamarin.Forms est via un exécuteur de plate-forme, donc votre test devra s’exécuter dans un environnement d’interface utilisateur ios, android, windows ou mac
[Exécuter des tests dans l’IDE][1]

Xamarin fournit également des tests d’interface utilisateur impressionnants avec l’offre [Xamarin.TestCloud][2], mais lorsque vous souhaitez implémenter des pratiques de développement BDD et avoir la possibilité de tester des ViewModels et des commandes, tout en s’exécutant à moindre coût sur des exécuteurs de tests unitaires localement ou sur une version serveur, il n’y a pas de manière intégrée.

J’ai développé une bibliothèque qui permet d’utiliser Specflow avec Xamarin.Forms pour implémenter facilement vos fonctionnalités depuis vos définitions de scénarios jusqu’au ViewModel, indépendamment de tout framework MVVM utilisé pour l’application (tel que [XLabs](https://github.com /XLabs/Xamarin-Forms-Labs), MVVMCross, Prism)

Si vous êtes nouveau sur BDD, consultez [Specflow][3].

Utilisation : #

  • Si vous ne l’avez pas encore, installez l’extension specflow visual studio à partir d’ici (ou de votre IDE visual studio): https://visualstudiogallery.msdn.microsoft.com/c74211e7-cb6e-4dfa-855d-df0ad4a37dd6

  • Ajoutez une bibliothèque de classes à votre projet Xamarin.Forms. C’est votre projet de test.

  • Ajoutez le package SpecFlow.Xamarin.Forms de nuget à vos projets de test.

  • Ajoutez une classe à votre projet de test qui hérite de ‘TestApp’, et enregistrez vos paires vues/modèles de vue ainsi que l’ajout de tout enregistrement DI, comme ci-dessous :

    public class DemoAppTest : TestApp
    {
        protected override void SetViewModelMapping()
        {
            TestViewFactory.EnableCache = false;

            // register your views / viewmodels below
            RegisterView<MainPage, MainViewModel>();
        }

        protected override void InitialiseContainer()
        {
            // add any di registration here
            // Resolver.Instance.Register<TInterface, TType>();
            base.InitialiseContainer();
        }
    }
  • Ajoutez une classe SetupHook à votre projet de test, afin d’ajouter vos crochets Specflow. Vous devrez amorcer l’application de test comme indiqué ci-dessous, en fournissant la classe que vous avez créée ci-dessus et le modèle de vue initial de votre application :
    [Binding]
    public class SetupHooks : TestSetupHooks
    {   
        /// <summary>
        ///     The before scenario.
        /// </summary>
        [BeforeScenario]
        public void BeforeScenario()
        {
            // bootstrap test app with your test app and your starting viewmodel
            new TestAppBootstrap().RunApplication<DemoAppTest, MainViewModel>();
        }
    }
  • Vous devrez ajouter un bloc catch à vos vues xamarin.forms codebehind afin d’ignorer le framework xamarin.forms vous forçant à exécuter l’interface utilisateur de l’application (ce que nous ne voulons pas faire) :
        public YourView()
        {
            try
            {
                InitializeComponent();
            }
            catch (InvalidOperationException soe)
            {
                if (!soe.Message.Contains("MUST"))
                    throw;
            }
        }

  • Ajoutez une fonctionnalité specflow à votre projet (en utilisant les modèles vs specflow livrés avec l’extension vs specflow)

  • Créer/Générer une classe d’étape qui hérite de TestStepBase, en passant le paramètre scenarioContext à la base.

  • Utilisez les services de navigation et les assistants pour naviguer, exécuter des commandes et tester vos modèles de vue :

  [Binding]
    public class GeneralSteps : TestStepBase
    {
        public GeneralSteps(ScenarioContext scenarioContext)
            : base(scenarioContext)
        {
            // you need to instantiate your steps by passing the scenarioContext to the base
        }

        [Given(@"I am on the main view")]
        public void GivenIAmOnTheMainView()
        {
            Resolver.Instance.Resolve<INavigationService>().PushAsync<MainViewModel>();           
            Resolver.Instance.Resolve<INavigationService>().CurrentViewModelType.ShouldEqualType<MainViewModel>();
        }
        
        [When(@"I click on the button")]
        public void WhenIClickOnTheButton()
        {
            GetCurrentViewModel<MainViewModel>().GetTextCommand.Execute(null);
        }

        [Then(@"I can see a Label with text ""(.*)""")]
        public void ThenICanSeeALabelWithText(string text)
        {           
            GetCurrentViewModel<MainViewModel>().Text.ShouldEqual(text);
        }
    }

[1] : https://developer.xamarin.com/guides/testcloud/uitest/working-with/running-tests-in-ide/ [2] : https://www.xamarin.com/test-cloud [3] : http://www.specflow.org/getting-started/

Utilisation avancée pour MVVM #

Pour ajouter au premier exemple, afin de tester les instructions de navigation qui se produisent dans l’application, nous devons fournir le ViewModel avec un crochet à la Navigation. Pour y parvenir :

  • Ajoutez le package SpecFlow.Xamarin.Forms.IViewModel de nuget à votre projet PCL Xamarin.Forms
  • Implémentez l’interface IViewModel dans votre ViewModel. Cela exposera simplement la propriété Xamarin.Forms INavigation :
  • classe publique MainViewModel : INotifyPropertyChanged, IViewModel.IViewModel { public INavigation Navigation { get; set; }
  • Le framework de test reprendra cela et gérera la navigation interne
  • Vous pouvez utiliser n’importe quel framework MVVM pour votre application (tel que XLabs, MVVMCross , Prism pour n’en nommer que quelques-uns. Tant que l’interface IViewModel est implémentée dans votre ViewModel, le framework la reprendra.
