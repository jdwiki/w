Rappels spécifiques au canal du SDK JavaScript v4 #

Dans [PubNub JavaScript v3][1], vous pouvez implémenter un rappel unique pour chaque chaîne à laquelle vous êtes abonné tant que vous appelez la fonction d’abonnement pour chaque chaîne et implémentez le rappel dans cet abonnement comme ceci :

var pubnub = new PubNub({
    publishKey: "your-pub-key",
    subscribeKey: "your-sub-key"
});

pubnub.subscribe({
    channel: 'ch1',
    message: function (m) {
        console.log(m + " ch1 callback");
    }
});

pubnub.subscribe({
    channel: 'ch2',
    message: function (m) {
        console.log(m + " ch2 callback: ");
    }
});

Ainsi, dans le code ci-dessus, si vous publiez le message "hello" à ch1 :

publish({channel: "ch1", message: "hello"});

… alors vous obtiendrez la sortie, hello ch1 callback. Et bien sûr, si vous publiez le même message sur ch2, vous obtiendrez la sortie hello ch2 callback.

La possibilité de fournir un rappel personnalisé pour chaque canal, ou un groupe de canaux, est utile et souvent plus souhaitable que de créer un rappel monolithique avec un long “if-then-else” de noms de canaux et de code à exécuter pour chaque condition. Et il existe de meilleures pratiques à utiliser que l’exemple simple ci-dessus, mais je voulais qu’il soit facile de comparer et de contraster avec la façon dont [PubNub JavaScript SDK v4][2] (et tout v4 PubNub SDK) est conçu.

Les SDK PubNub v4 utilisent un seul écouteur pour recevoir tous les événements message, presence et status. Cela signifie que lorsque vous vous “abonnez” à un canal, vous ne fournissez que le(s) nom(s) du canal plutôt qu’une fonction associée de “rappel de message”. Dans [JavaScript SDK v4][2], l’écouteur ressemble à ceci :

pubnub.addListener({
    message: function(m) {
        console.log(JSON.stringify(m));
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(s) {
        console.log(JSON.stringify(s));
    }
});

Je sais que de nombreux développeurs se demanderont comment migrer le code subscribe avec des message callbacks uniques du SDK v3 avec ce type de conception sans recourir au code conditionnel de nom de canal sans fin que j’ai mentionné ci-dessus, comme ceci :

pubnub.addListener({
    message: function(m) {
        if (m.subscribedChannel == 'ch1') {
            console.log(m.message + "ch1 callback");
        }
        else if (m.subscribedChannel == 'ch2') {
            console.log(m.message + "ch2 callback");
        }
        else {
            console.log(m.message + "default callback");
        }
    }
    // removed the other two callbacks for brevity purposes
});

Le paramètre m qui est passé dans l’écouteur de message ci-dessus a la structure suivante qui est une conception différente de la conception à paramètres multiples de JavaScript SDK v3.

{
    "channel": "ch1",
    "subscription": <undefined>,
    "timetoken": "14721821326909151",
    "message": "hello"
}

Ce timetoken est le publish timetoken réel. Pour les développeurs PubNub expérimentés, vous devriez être ravis de voir que cela est maintenant disponible pour l’abonné, mais n’expliquons pas pourquoi c’est important et puissant en ce moment.

Maintenant, je ne m’attendrais pas à ce qu’un développeur JavaScript expérimenté écrive du code comme représenté ci-dessus et de nombreux développeurs avancés savent peut-être déjà quoi faire. Mais pour les développeurs qui sont au niveau débutant à intermédiaire avec JavaScript, la solution peut ne pas être immédiatement évidente. Cependant, je sais qu’une fois que vous verrez cette approche de conception simple ci-dessous, cela vous ouvrira les yeux sur les possibilités illimitées du langage JavaScript - c’est parti.

Réitérons l’exigence :

Pour chaque chaîne à laquelle je suis abonné, je veux pouvoir fournir une fonction unique à invoquer lorsqu’un message est reçu sur ce canal. Et je veux éviter l’approche du nom de canal conditionnel monolithique.

Donc, ce que nous devons faire en premier est de créer une table de recherche de fonctions. Une hashtable pour être exact. Cette table contiendra les noms des canaux comme clés et les fonctions (le code à invoquer lorsqu’un message est reçu sur ce canal) comme valeurs. Si vous êtes quelque peu novice en JavaScript ou si vous codez avec le langage depuis un certain temps mais que vous n’avez pas encore vraiment plongé dans les fonctionnalités du langage, cela peut sembler étrange et impossible, mais c’est vraiment ainsi que JavaScript fonctionne et vous l’avez fait depuis le début et ne le savait pas vraiment. Définissons notre table de recherche de fonctions :

ftbl = {};

C’est tout - vous avez un objet qui contiendra vos canaux et vos fonctions. Assez simple, non? Mais comment ajoutez-vous les canaux et les fonctions ? Comme n’importe quelle autre clé/valeur.

ftbl.ch1 = function(m){console.log(m.message + " ch1 callback")};
ftbl.ch2 = function(m){console.log(m.message + " ch1 callback")};

…et ainsi de suite avec chaque canal et fonction que vous souhaitez définir. Et vous n’avez pas besoin de créer toutes vos touches de canal/fonction en un seul endroit de votre code. Vous pouvez ajouter chaque chaîne/fonction au ftbl lorsque vous vous abonnez à une chaîne.

ftbl.ch10 = function(m){console.log(m.message + " ch10 callback")};
pubnub.pubnub.subscribe({
    channels: ['ch10'] 
});

OK, c’est assez simple, et vous pouvez devenir plus sophistiqué et plus avancé avec la façon dont vous faites cela, mais en gardant juste la base. Mais comment invoquez-vous cette fonction pour le canal auquel elle est liée ? C’est pourquoi JavaScript est si cool, puissant et facile, surtout si vous venez d’un langage rigide et structuré comme Java - vérifiez-le.

pubnub.addListener({
    message: function(m) {
        // use the channel name to get the function
        // from ftbl and invoke it 
        ftbl[m.subscribedChannel](m);
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(statusEvent) {
        console.log(JSON.stringify(s));
    }
});

C’est tout ce qu’on peut en dire. Obtenez simplement la fonction de ftbl en utilisant le nom du canal qui est passé dans la fonction de rappel message de l’auditeur et ajoutez (m) à la fin et boum, il exécute votre fonction.

Si le canal est ch10, ftbl[m.subscribedChannel](m) invoque simplement function(m){console.log(m.message + "ch10 callback")} en passant le paramètre m qui votre fonction peut analyser et exploiter selon ses besoins.

Appelons donc la fonction publish suivante :

pubnub.publish(
    {
        channel : "ch10",
        message : "hello"
    }, 
    function(status, response) {
        console.log(status, response);
    }
);

… entraînera l’affichage du message suivant : hello ch10 callback. Et l’équivalent pour la publication sur d’autres canaux que vous avez définis dans votre table de recherche de fonctions. N’oubliez pas de fournir une valeur par défaut pour les canaux inconnus.

Et n’oubliez pas les rappels presence et status dans l’écouteur. Il peut s’agir de deux tables de recherche de fonctions supplémentaires ou d’un “ftbl” légèrement plus complexe :

ftbl.message.ch1 = function(m){console.log(m.message + " ch1 message cb")};
ftbl.presence.ch1 = function(m){console.log(m.message + " ch1 presence cb")};
ftbl.status.ch1 = function(m){console.log(m.message + " ch1 status cb")};

ou

ftbl.ch1.message = ...
ftbl.ch1.presence = ...
ftbl.ch1.status = ...

J’aime mieux le premier que le second mais c’est vraiment à vous de décider. Et vous voulez probablement un code générique de gestion d’événement status de toute façon, mais cela dépendra de vos besoins spécifiques.

Et cela peut devenir encore plus compliqué et robuste avec des fonctions optionnelles par canal qui peuvent être invoquées en fonction de certaines données supplémentaires que vous intégrez dans la charge utile du message.

Alors voilà, des rappels uniques pour chaque canal. Plus d’excuses pour ne pas vouloir migrer de 3x à 4x. Mais si vous avez des doutes sur la migration, n’hésitez pas à contacter [l’assistance PubNub][3] et nous vous aiderons à aller de l’avant. Et n’oubliez pas de consulter le [Guide de migration de la v3 vers la v4 du SDK PubNub JavaScript][4].
