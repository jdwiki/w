Syntaxe #

  • command </path/to/file # Redirige l’entrée standard vers le fichier
  • command >/path/to/file # Redirige la sortie standard vers flie
  • command file_descriptor>/path/to/file # Redirige la sortie de file_descriptor vers le fichier
  • command >&file_descriptor # Redirige la sortie vers file_descriptor
  • command file_descriptor>&another_file_descriptor # Redirige file_descriptor vers another_file_descriptor
  • command <&file_descriptor # Redirige file_descriptor vers l’entrée standard
  • command &>/path/to/file # Redirige la sortie standard et l’erreur standard vers le fichier

Paramètres #

Paramètre Détails
descripteur de fichier interne Un nombre entier.
orientation L’un des >, < ou <>
descripteur ou chemin de fichier externe & suivi d’un entier pour le descripteur de fichier ou un chemin.

Les programmes de console UNIX ont un fichier d’entrée et deux fichiers de sortie (les flux d’entrée et de sortie, ainsi que les périphériques, sont traités comme des fichiers par le système d’exploitation.) Il s’agit généralement du clavier et de l’écran, respectivement, mais tout ou partie d’entre eux peuvent être redirigés venir de - ou aller vers - un fichier ou un autre programme.

STDIN est une entrée standard et c’est ainsi que le programme reçoit une entrée interactive. STDIN se voit généralement attribuer le descripteur de fichier 0.

STDOUT est la sortie standard. Tout ce qui est émis sur STDOUT est considéré comme le “résultat” du programme. STDOUT est généralement attribué au descripteur de fichier 1.

STDERR est l’endroit où les messages d’erreur sont affichés. Généralement, lors de l’exécution d’un programme à partir de la console, STDERR est affiché à l’écran et ne peut être distingué de STDOUT. STDERR est généralement attribué au descripteur de fichier 2.

L’ordre de redirection est important

command > file 2>&1

Redirige les deux (STDOUT et STDERR) vers le fichier.

command 2>&1 > file

Redirige uniquement STDOUT, car le descripteur de fichier 2 est redirigé vers le fichier pointé par le descripteur de fichier 1 (qui n’est pas encore le fichier file lorsque l’instruction est évaluée).

Chaque commande dans un pipeline a son propre STDERR (et STDOUT) car chacun est un nouveau processus. Cela peut créer des résultats surprenants si vous vous attendez à ce qu’une redirection affecte l’ensemble du pipeline. Par exemple cette commande (encapsulée pour plus de lisibilité) :

$ python -c 'import sys;print >> sys.stderr, "Python error!"' \
| cut -f1 2>> error.log

affichera “Erreur Python !” à la console plutôt qu’au fichier journal. À la place, joignez l’erreur à la commande que vous souhaitez capturer :

$ python -c 'import sys;print >> sys.stderr, "Python error!"' 2>> error.log \
| cut -f1

Ajouter vs Tronquer #

Tronquer > #

  1. Créez le fichier spécifié s’il n’existe pas.

  2. Tronquer (supprimer le contenu du fichier)

  3. Écrire dans un fichier

    $ echo “first line” > /tmp/lines $ echo “second line” > /tmp/lines

    $ cat /tmp/lines second line

Ajouter >> #

  1. Créez le fichier spécifié s’il n’existe pas.

  2. Ajouter un fichier (écriture à la fin du fichier).

    Overwrite existing file #

    $ echo “first line” > /tmp/lines

    Append a second line #

    $ echo “second line” » /tmp/lines

    $ cat /tmp/lines first line second line

Rediriger à la fois STDOUT et STDERR #

Les descripteurs de fichiers comme ‘0’ et ‘1’ sont des pointeurs. Nous modifions ce vers quoi les descripteurs de fichiers pointent avec la redirection. >/dev/null signifie que 1 pointe vers /dev/null.

Nous pointons d’abord 1 (STDOUT) sur /dev/null puis pointons 2 (STDERR) sur ce vers quoi 1 pointe.

# STDERR is redirect to STDOUT: redirected to /dev/null,
# effectually redirecting both STDERR and STDOUT to /dev/null
echo 'hello' > /dev/null 2>&1

Ce peut être encore raccourci comme suit :

echo 'hello' &> /dev/null

Cependant, cette forme peut être indésirable en production si la compatibilité du shell est un problème car elle entre en conflit avec POSIX, introduit une ambiguïté d’analyse et les shells sans cette fonctionnalité l’interpréteront mal :

# Actual code
echo 'hello' &> /dev/null
echo 'hello' &> /dev/null 'goodbye'

# Desired behavior
echo 'hello' > /dev/null 2>&1
echo 'hello' 'goodbye' > /dev/null 2>&1

# Actual behavior
echo 'hello' &
echo 'hello' & goodbye > /dev/null

REMARQUE: &> est connu pour fonctionner comme souhaité à la fois dans Bash et Zsh.

Redirection de la sortie standard #

> redirige la sortie standard (alias STDOUT) de la commande actuelle vers un fichier ou un autre descripteur.

Ces exemples écrivent la sortie de la commande ls dans le fichier file.txt

ls >file.txt
> file.txt ls

Le fichier cible est créé s’il n’existe pas, sinon ce fichier est tronqué.

Le descripteur de redirection par défaut est la sortie standard ou “1” lorsqu’aucun n’est spécifié. Cette commande est équivalente aux exemples précédents avec la sortie standard explicitement indiquée :

ls 1>file.txt

Remarque : la redirection est initialisée par le shell exécuté et non par la commande exécutée, elle est donc effectuée avant l’exécution de la commande.

Utilisation de canaux nommés #

Parfois, vous voudrez peut-être sortir quelque chose par un programme et l’entrer dans un autre programme, mais vous ne pouvez pas utiliser un tube standard.

ls -l | grep ".log"

Vous pouvez simplement écrire dans un fichier temporaire :

touch tempFile.txt
ls -l > tempFile.txt
grep ".log" < tempFile.txt

Cela fonctionne bien pour la plupart des applications, cependant, personne ne saura ce que fait tempFile et quelqu’un pourrait le supprimer s’il contient la sortie de ls -l dans ce répertoire. C’est là qu’un tube nommé entre en jeu :

mkfifo myPipe
ls -l > myPipe
grep ".log" < myPipe

myPipe est techniquement un fichier (tout est sous Linux), alors faisons ls -l dans un répertoire vide dans lequel nous venons de créer un tube :

mkdir pipeFolder
cd pipeFolder
mkfifo myPipe
ls -l

La sortie est :

prw-r--r-- 1 root root 0 Jul 25 11:20 myPipe

Remarquez le premier caractère dans les autorisations, il est répertorié comme un canal, pas comme un fichier.

Faisons maintenant quelque chose de cool.

Ouvrez un terminal et notez le répertoire (ou créez-en un pour faciliter le nettoyage) et créez un tube.

mkfifo myPipe

Maintenant, mettons quelque chose dans le tuyau.

echo "Hello from the other side" > myPipe

Vous remarquerez que cela se bloque, l’autre côté du tuyau est toujours fermé. Ouvrons l’autre côté du tuyau et laissons passer ce truc.

Ouvrez un autre terminal et accédez au répertoire dans lequel se trouve le tube (ou si vous le connaissez, ajoutez-le au tube):

cat < myPipe

Vous remarquerez qu’après la sortie de “bonjour de l’autre côté”, le programme du premier terminal se termine, tout comme celui du second terminal.

Exécutez maintenant les commandes en sens inverse. Commencez par cat < myPipe puis faites écho à quelque chose dedans. Cela fonctionne toujours, car un programme attendra que quelque chose soit mis dans le tube avant de se terminer, car il sait qu’il doit obtenir quelque chose.

Les canaux nommés peuvent être utiles pour déplacer des informations entre des terminaux ou entre des programmes.

Les tuyaux sont petits. Une fois plein, le graveur se bloque jusqu’à ce qu’un lecteur lise le contenu, vous devez donc soit exécuter le lecteur et le graveur dans des terminaux différents, soit exécuter l’un ou l’autre en arrière-plan :

 ls -l /tmp > myPipe &
 cat < myPipe

Plus d’exemples utilisant des canaux nommés :

  • Exemple 1 - toutes les commandes sur le même terminal / même shell

     $ { ls -l && cat file3; } >mypipe &
 $ cat <mypipe    
 # Output: Prints ls -l data and then prints file3 contents on screen

  • Exemple 2 - toutes les commandes sur le même terminal / même shell

     $ ls -l >mypipe &
 $ cat file3 >mypipe &
 $ cat <mypipe
 #Output: This prints on screen the contents of mypipe.

N’oubliez pas que le premier contenu de file3 est affiché, puis les données ls -l sont affichées (configuration LIFO).

  • Exemple 3 - toutes les commandes sur le même terminal / même shell

     $ { pipedata=$(<mypipe) && echo "$pipedata"; } &
 $ ls >mypipe 
 # Output: Prints the output of ls directly on screen

    Mind that the variable $pipedata is not available for usage in the main terminal / main shell since the use of & invokes a subshell and $pipedata was only available in this subshell.

  • Exemple 4 - toutes les commandes sur le même terminal / même shell

     $ export pipedata
 $ pipedata=$(<mypipe) &
 $ ls -l *.sh >mypipe
 $ echo "$pipedata"   
 #Output : Prints correctly the contents of mypipe

Cela imprime correctement la valeur de la variable $pipedata dans le shell principal en raison de la déclaration d’exportation de la variable. Le terminal principal/shell principal ne se bloque pas en raison de l’invocation d’un shell d’arrière-plan (&).

Redirection de plusieurs commandes vers le même fichier #

{
  echo "contents of home directory"
  ls ~
} > output.txt

Afficher les messages d’erreur sur stderr #

Les messages d’erreur sont généralement inclus dans un script à des fins de débogage ou pour fournir une expérience utilisateur riche. Écrivez simplement un message d’erreur comme celui-ci:

cmd || echo 'cmd failed'

peut fonctionner pour des cas simples mais ce n’est pas la manière habituelle. Dans cet exemple, le message d’erreur polluera la sortie réelle du script en mélangeant les erreurs et la sortie réussie dans stdout.

En bref, le message d’erreur doit aller à stderr et non à stdout. C’est assez simple :

cmd || echo 'cmd failed' >/dev/stderr

Un autre exemple:

if cmd; then
    echo 'success'
else
    echo 'cmd failed' >/dev/stderr
fi

Dans l’exemple ci-dessus, le message de réussite sera imprimé sur stdout tandis que le message d’erreur sera imprimé sur stderr.

Une meilleure façon d’afficher un message d’erreur consiste à définir une fonction :

err(){
    echo "E: $*" >>/dev/stderr
}

Maintenant, lorsque vous devez imprimer une erreur :

err "My error message"

Redirection vers des adresses réseau #

Bash traite certains chemins comme spéciaux et peut établir des communications réseau en écrivant dans /dev/{udp|tcp}/host/port. Bash ne peut pas configurer de serveur d’écoute, mais peut initier une connexion et, pour TCP, peut au moins lire les résultats.

Par exemple, pour envoyer une simple requête Web, on pourrait faire :

exec 3</dev/tcp/www.google.com/80
printf 'GET / HTTP/1.0\r\n\r\n' >&3
cat <&3

et les résultats de la page Web par défaut de www.google.com seront imprimés sur stdout.

De la même manière

printf 'HI\n' >/dev/udp/192.168.1.1/6666

enverrait un message UDP contenant HI\n à un auditeur sur 192.168.1.1:6666

Redirection STDIN #

< lit à partir de son argument de droite et écrit dans son argument de gauche.

Pour écrire un fichier dans STDIN, nous devons lire /tmp/a_file et écrire dans STDIN c’est-à-dire 0</tmp/a_file

Remarque : Le descripteur de fichier interne est par défaut 0 (STDIN) pour <

$ echo "b" > /tmp/list.txt
$ echo "a" >> /tmp/list.txt
$ echo "c" >> /tmp/list.txt
$ sort < /tmp/list.txt
a
b
c

Redirection STDERR #

‘2’ est ‘STDERR’.

$ echo_to_stderr 2>/dev/null # echos nothing

Définitions :

echo_to_stderr est une commande qui écrit "stderr" dans STDERR

echo_to_stderr () {
    echo stderr >&2
}

$ echo_to_stderr
stderr

STDIN, STDOUT et STDERR expliqués #

Les commandes ont une entrée (STDIN) et deux types de sorties, une sortie standard (STDOUT) et une erreur standard (STDERR).

Par exemple:

STDIN

root@server~# read
Type some text here

L’entrée standard est utilisée pour fournir une entrée à un programme. (Ici, nous utilisons le read builtin pour lire une ligne de STDIN.)

SORTIE STD

root@server~# ls file
file

La sortie standard est généralement utilisée pour la sortie “normale” d’une commande. Par exemple, ls répertorie les fichiers, de sorte que les fichiers sont envoyés à STDOUT.

STDERR

root@server~# ls anotherfile
ls: cannot access 'anotherfile': No such file or directory

L’erreur standard est (comme son nom l’indique) utilisée pour les messages d’erreur. Comme ce message n’est pas une liste de fichiers, il est envoyé à STDERR.

STDIN, STDOUT et STDERR sont les trois flux standard. Ils sont identifiés par le shell par un numéro plutôt que par un nom :

0 = Norme en 1 = sortie standard 2 = erreur standard

Par défaut, STDIN est attaché au clavier, et STDOUT et STDERR apparaissent dans le terminal. Cependant, nous pouvons rediriger STDOUT ou STDERR vers tout ce dont nous avons besoin. Par exemple, disons que vous n’avez besoin que de la sortie standard et que tous les messages d’erreur imprimés sur l’erreur standard doivent être supprimés. C’est alors que nous utilisons les descripteurs “1” et “2”.

Redirection de STDERR vers /dev/null En reprenant l’exemple précédent,

root@server~# ls anotherfile 2>/dev/null
root@server~#

Dans ce cas, s’il y a un STDERR, il sera redirigé vers /dev/null (un fichier spécial qui ignore tout ce qui y est mis), de sorte que vous n’obtiendrez aucune sortie d’erreur sur le shell.
