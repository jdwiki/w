Arithmétique if instruction #

L’instruction arithmétique if permet d’utiliser trois branches en fonction du résultat d’une expression arithmétique

if (arith_expr) label1, label2, label3

Cette instruction “if” transfère le flux de contrôle à l’une des étiquettes d’un code. Si le résultat de arith_expr est négatif, label1 est impliqué, si le résultat est zéro label2 est utilisé, et si le résultat est positif, le dernier label3 est appliqué. L’arithmétique “if” nécessite les trois étiquettes mais permet la réutilisation des étiquettes. Par conséquent, cette instruction peut être simplifiée en un “if” à deux branches.

Exemples:

if (N * N - N / 2) 130, 140, 130

if (X) 100, 110, 120

Maintenant, cette fonctionnalité est obsolète avec la même fonctionnalité offerte par l’instruction if et la construction if-else. Par exemple, le fragment

    if (X) 100, 110, 120
100 print*, "Negative"
    goto 200
110 print*, "Zero"
    goto 200
120 print*, "Positive"
200 continue

peut être écrit comme la construction if-else

if (X<0) then
  print*, "Negative"
else if (X==0) then
  print*, "Zero"
else
  print*, "Positive"
end if

Un remplacement d’instruction if pour

    if (X) 100, 100, 200
100 print *, "Negative or zero"
200 continue

Peut être

if (X<=0) print*, "Negative or zero"

Formulaire source fixe #

Fortran a été conçu à l’origine pour un formulaire de format fixe basé sur une carte perforée à 80 colonnes :

[![entrez la description de l’image ici][1]][1]

Oui : il s’agit d’une ligne du code de l’auteur

Ceux-ci ont été créés sur une machine à perforer les cartes, un peu comme ceci :

[![entrez la description de l’image ici][2]][2]

Les images sont des photographies originales de l’auteur

Le format, comme indiqué sur l’exemple de carte illustré, avait les cinq premières colonnes réservées aux étiquettes de déclaration. La première colonne servait à désigner les commentaires par une lettre C. La sixième colonne a été utilisée pour indiquer une continuation d’instruction (en insérant tout caractère autre qu’un zéro ‘0’). Les 8 dernières colonnes ont été utilisées pour l’identification et la séquence des cartes, ce qui était très utile si vous laissiez tomber votre jeu de cartes par terre ! Le codage des caractères pour les cartes perforées n’avait qu’un ensemble limité de caractères et était uniquement en majuscules. En conséquence, les programmes Fortran ressemblaient à ceci :

       DIMENSION A(10)                                                    00000001
C THIS IS A COMMENT STATEMENT TO EXPLAIN THIS EXAMPLE PROGRAM             00000002
       WRITE (6,100)                                                      00000003
 100   FORMAT(169HTHIS IS A RATHER LONG STRING BEING OUTPUT WHICH GOES OVE00000004
      1R MORE THAN ONE LINE, AND USES THE STATEMENT CONTINUATION MARKER IN00000005
      2COLUMN 6, AND ALSO USES HOLLERITH STRING FORMAT)                   00000006
       STOP                                                               00000007
       END                                                                00000008

Le caractère espace a également été ignoré partout, sauf à l’intérieur d’une constante de caractère Hollerith (comme indiqué ci-dessus). Cela signifiait que des espaces pouvaient apparaître à l’intérieur de mots réservés et de constantes, ou complètement manqués. Cela a eu pour effet secondaire certaines déclarations plutôt trompeuses telles que :

       DO 1 I = 1.0

est une affectation à la variable DO1I alors que :

       DO1I = 1,0

est en fait une boucle ‘DO’ sur la variable ‘I’.

Le Fortran moderne n’exige plus cette forme fixe d’entrée et permet une forme libre utilisant n’importe quelle colonne. Les commentaires sont maintenant indiqués par un ! qui peut également être ajouté à une ligne de déclaration. Les espaces ne sont désormais autorisés nulle part et doivent être utilisés comme séparateurs, comme dans la plupart des autres langues. Le programme ci-dessus pourrait être écrit en Fortran moderne comme suit :

! This is a comment statement to explain this example program
Print *,"THIS IS A RATHER LONG STRING BEING OUTPUT WHICH no longer GOES OVER MORE THAN ONE LINE, AND does not need to USE THE STATEMENT CONTINUATION MARKER IN COLUMN 6, or the HOLLERITH STRING FORMAT"

Bien que la continuation à l’ancienne ne soit plus utilisée, l’exemple ci-dessus montre que de très longues instructions se produiront toujours. Le Fortran moderne utilise un symbole & à la fin et au début de la suite. Par exemple, nous pourrions écrire ce qui précède sous une forme plus lisible :

! This is a comment statement to explain this example program
Print *,"THIS IS A RATHER LONG STRING BEING OUTPUT WHICH still &
         &GOES OVER MORE THAN ONE LINE, AND does need to USE THE STATEMENT &
         &CONTINUATION notation"

Blocs communs #

Dans les premières formes de Fortran, le seul mécanisme de création d’un magasin de variables globales visible depuis les sous-programmes et les fonctions consiste à utiliser le mécanisme de bloc “COMMON”. Cela a permis aux séquences de variables d’être des noms et partagées en commun.

En plus des blocs communs nommés, il peut également y avoir un bloc commun vide (sans nom).

Un bloc commun vide pourrait être déclaré comme

common i, j

alors que le bloc nommé variables pourrait être déclaré comme

common /variables/ i, j

Comme exemple complet, nous pourrions imaginer un magasin de tas utilisé par des routines qui peuvent ajouter et supprimer des valeurs :

       PROGRAM STACKING
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       ICOUNT = 0
       READ *, IVAL
       CALL PUSH(IVAL)
       CALL POP(IVAL)
       END

       SUBROUTINE PUSH(IVAL)
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       ICOUNT = ICOUNT + 1
       ISTACK(ICOUNT) = IVAL
       RETURN
       END

       SUBROUTINE POP(IVAL)
       COMMON /HEAP/ ICOUNT, ISTACK(1023)
       IVAL = ISTACK(ICOUNT)
       ICOUNT = ICOUNT - 1
       RETURN
       END

Des déclarations communes peuvent être utilisées pour déclarer implicitement le type d’une variable et pour spécifier l’attribut dimension. Ce comportement à lui seul est souvent une source suffisante de confusion. En outre, l’association de stockage implicite et les exigences de définitions répétées dans les unités de programme rendent l’utilisation de blocs communs sujette aux erreurs.

Enfin, les blocs communs sont très restreints dans les objets qu’ils contiennent. Par exemple, un tableau dans un bloc commun doit avoir une taille explicite ; les objets allouables peuvent ne pas se produire ; les types dérivés ne doivent pas avoir d’initialisation par défaut.

Dans Fortran moderne, ce partage de variables peut être géré par l’utilisation de [modules] (https://www.wikiod.com/fr/fortran/utilisation-des-modules). L’exemple ci-dessus peut s’écrire :

module heap
  implicit none
  ! In Fortran 2008 all module variables are implicitly saved
  integer, save :: count = 0
  integer, save :: stack(1023)
end module heap

program stacking
  implicit none
  integer val
  read *, val
  call push(val)
  call pop(val)

contains
  subroutine push(val)
    use heap, only : count, stack
    integer val
    count = count + 1
    stack(count) = val
  end subroutine push

  subroutine pop(val)
    use heap, only : count, stack
    integer val
    val = stack(count)
    count = count - 1
  end subroutine pop
end program stacking

Les blocs communs nommés et vides ont des comportements légèrement différents. À noter :

  • les objets dans des blocs communs nommés peuvent être définis initialement ; les objets en blanc commun ne doivent pas être
  • les objets dans des blocs communs vierges se comportent comme si le bloc commun avait l’attribut save ; les objets dans des blocs communs nommés sans l’attribut save peuvent devenir indéfinis lorsque le bloc n’est pas dans la portée d’une unité de programme active

Ce dernier point peut être mis en contraste avec le comportement des variables de module dans le code moderne. Toutes les variables de module dans Fortran 2008 sont implicitement enregistrées et ne deviennent pas indéfinies lorsque le module sort de la portée. Avant Fortran 2008, les variables de module, comme les variables dans les blocs communs nommés, devenaient également indéfinies lorsque le module sortait de la portée.

Constructions DO non-blocs #

La construction do non-bloc ressemble à

    integer i
    do 100, i=1, 5
100 print *, i

C’est-à-dire, où l’instruction de terminaison étiquetée n’est pas une instruction « continuer ». Il existe diverses restrictions sur la déclaration qui peut être utilisée comme déclaration de terminaison et le tout est généralement très déroutant.

Une telle construction non bloc peut être réécrite sous forme de bloc comme

    integer i
    do 100 i=1,5
      print *, i
100 continue

ou mieux, en utilisant une instruction de terminaison “end do”,

integer i
do i=1,5
  print *, i
end do

Retour alternatif #

Le retour alternatif est une fonction permettant de contrôler le flux d’exécution au retour d’un sous-programme. Il est souvent utilisé comme une forme de gestion des erreurs :

real x

call sub(x, 1, *100, *200)
print*, "Success:", x
stop

100 print*, "Negative input value"
stop

200 print*, "Input value too large"
stop

end

subroutine sub(x, i, *, *)
  real, intent(out) :: x
  integer, intent(in) :: i
  if (i<0) return 1
  if (i>10) return 2
  x = i
end subroutine

Le retour alternatif est marqué par les arguments * dans la liste d’arguments fictifs du sous-programme.

Dans l’instruction call au-dessus de *100 et *200, faites référence aux instructions étiquetées 100 et 200 respectivement.

Dans le sous-programme lui-même, les instructions return correspondant au retour alternatif ont un numéro. Ce nombre n’est pas une valeur de retour, mais indique l’étiquette fournie à laquelle l’exécution est transmise au retour. Dans ce cas, return 1 transmet l’exécution à l’instruction étiquetée 100 et return 2 transmet l’exécution à l’instruction étiquetée 200. Une instruction return sans fioritures, ou l’achèvement de l’exécution d’un sous-programme sans instruction return, passe l’exécution immédiatement après l’instruction d’appel.

La syntaxe alternative de retour est très différente des autres formes d’association d’arguments et l’installation introduit un contrôle de flux contrairement aux goûts modernes. Un contrôle de flux plus agréable peut être géré avec le retour d’un code “statut” entier.

real x
integer status

call sub(x, 1, status)
select case (status)
case (0)
  print*, "Success:", x
case (1)
  print*, "Negative input value"
case (2)
  print*, "Input value too large"
end select

end

subroutine sub(x, i, status)
  real, intent(out) :: x
  integer, intent(in) :: i
  integer, intent(out) :: status

  status = 0

  if (i<0) then
    status = 1
  else if (i>10)
    status = 2
  else
    x = i
  end if

end subroutine

Attribué GOTO #

GOTO affecté utilise une variable entière à laquelle une étiquette d’instruction est affectée à l’aide de l’instruction ASSIGN.

100 CONTINUE

...

ASSIGN 100 TO ILABEL

...


GOTO ILABEL

GOTO attribué est obsolète dans Fortran 90 et supprimé dans Fortran 95 et versions ultérieures. Il peut être évité dans le code moderne en utilisant des procédures, des procédures internes, des pointeurs de procédure et d’autres fonctionnalités.

GOTO calculé #

Le GOTO calculé permet de brancher le programme en fonction de la valeur d’une expression entière.

GOTO (label_1, label_2,... label_n) scalar-integer-expression

Si scalar-integer-expression est égal à 1, le programme continue à l’étiquette d’instruction label_1, s’il est égal à 2, il passe à label_2 et ainsi de suite. S’il est inférieur à ‘1’ ou supérieur à ’n’, le programme continue à la ligne suivante.

Exemple:

ivar = 2

...

GOTO (10, 20, 30, 40) ivar

passera à l’étiquette d’instruction 20.

Cette forme de « goto » est obsolète dans Fortran 95 et versions ultérieures, étant remplacée par la construction « select case ».

Spécificateurs de format attribués #

Avant Fortran 95, il était possible d’utiliser des formats assignés pour l’entrée ou la sortie. Envisager

integer i, fmt
read *, i

assign 100 to fmt
if (i<100000) assign 200 to fmt

print fmt, i

100 format ("This is a big number", I10)
200 format ("This is a small number", I6)

end

L’instruction assign attribue une étiquette d’instruction à une variable entière. Cette variable entière est ensuite utilisée comme spécificateur de format dans l’instruction print.

Une telle affectation de spécificateur de format a été supprimée dans Fortran 95. Au lieu de cela, un code plus moderne peut utiliser une autre forme de contrôle de flux d’exécution

integer i
read *, i

if (i<100000) then
  print 100, i
else
  print 200, i
end if

100 format ("This is a big number", I10)
200 format ("This is a small number", I6)

end

ou une variable de caractère peut être utilisée comme spécificateur de format

character(29), target :: big_fmt='("This is a big number", I10)'
character(30), target :: small_fmt='("This is a small number", I6)'
character(:), pointer :: fmt

integer i
read *, i

fmt=>big_fmt
if (i<100000) fmt=>small_fmt

print fmt, i

end

Fonctions d’instruction #

Pensez au programme

implicit none
integer f, i
f(i)=i

print *, f(1)
end

Ici, f est une fonction d’instruction. Il a un type de résultat entier, prenant un argument factice entier.1

Une telle fonction d’instruction existe dans la portée dans laquelle elle est définie. En particulier, il a accès aux variables et aux constantes nommées accessibles dans cette portée.

Cependant, les fonctions d’instruction sont soumises à de nombreuses restrictions et sont potentiellement déroutantes (regardant un coup d’œil occasionnel comme une instruction d’affectation d’éléments de tableau). Les restrictions importantes sont :

  • le résultat de la fonction et les arguments factices doivent être scalaires
  • les arguments factices sont dans la même portée que la fonction
  • les fonctions d’instruction n’ont pas de variables locales
  • les fonctions d’instruction ne peuvent pas être passées en tant qu’arguments réels

Les principaux avantages des fonctions d’instruction sont répétés par les fonctions internes

implicit none

print *, f(1)

contains

  integer function f(i)
    integer i
    f = i
  end function

end

Les fonctions internes ne sont pas soumises aux restrictions mentionnées ci-dessus, bien qu’il soit peut-être intéressant de noter qu’un sous-programme interne peut ne pas contenir d’autre sous-programme interne (mais il peut contenir une fonction d’instruction).

Les fonctions internes ont leur propre portée mais ont également une association d’accueil disponible.

1 Dans de vrais anciens exemples de code, il ne serait pas inhabituel de voir les arguments factices d’une fonction d’instruction être implicitement typés, même si le résultat a un type explicite.
