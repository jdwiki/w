Moquerie simple #

Introduction #

Le [Manuel PHPUnit][1] décrit la moquerie comme telle :

La pratique consistant à remplacer un objet par un double test qui vérifie attentes, par exemple affirmer qu’une méthode a été appelée, est appelé moquerie.

Ainsi, au lieu de supprimer le code, un observateur est créé qui non seulement remplace le code qui doit être réduit au silence, mais observe qu’une activité spécifique se serait produite dans le monde réel.

Installer #

Commençons par une simple classe de journalisation qui, par souci de clarté, affiche simplement le texte envoyé dans le paramètre (normalement, cela ferait quelque chose qui peut être problématique pour les tests unitaires, comme la mise à jour d’une base de données) :

class Logger {
    public function log($text) {
        echo $text;
    }
}

Maintenant, créons une classe Application. Il accepte un objet Logger comme paramètre de la méthode run, qui à son tour invoque la méthode log du Logger pour capturer que l’application a démarré.

class Application {
  public function run(Logger $logger) {
    // some code that starts up the application

    // send out a log that the application has started
    $logger->log('Application has started');
  }
}

Si le code suivant a été exécuté tel qu’il est écrit :

$logger = new Logger();
$app = new Application();
$app->run($logger);

Ensuite, le texte “L’application a démarré” sera affiché selon la méthode log à l’intérieur du Logger.

Tests unitaires avec mocking #

Le test unitaire de la classe Application n’a pas besoin de vérifier ce qui se passe dans la méthode Logger log, il doit seulement vérifier qu’il a été appelé.

Dans le test PHPUnit, un observateur est créé pour remplacer la classe Logger. Cet observateur est configuré pour s’assurer que la méthode log n’est invoquée qu’une seule fois, avec la valeur de paramètre “L’application a démarré”.

Ensuite, l’observateur est envoyé dans la méthode run, qui vérifie qu’en fait la méthode log n’a été appelée qu’une seule fois et que le cas de test réussit, mais qu’aucun texte n’a été affiché.

class ApplicationTest extends \PHPUnit_Framework_TestCase {

  public function testThatRunLogsApplicationStart() {

    // create the observer
    $mock = $this->createMock(Logger::class);
    $mock->expects($this->once())
        ->method('log')
        ->with('Application has started');
    
    // run the application with the observer which ensures the log method was called
    $app = new Application();
    $app->run($mock);
  
  }
}

[1] : https://phpunit.de/manual/current/en/test-doubles.html#test-doubles.mock-objects

Écrasement simple #

Parfois, certaines sections de code sont difficiles à tester, comme l’accès à une base de données ou l’interaction avec l’utilisateur. Vous pouvez supprimer ces sections de code, permettant au reste du code d’être testé.

Commençons par une classe qui invite l’utilisateur. Pour simplifier, il n’a que deux méthodes, une qui invite réellement l’utilisateur (qui serait utilisée par toutes les autres méthodes) et celle que nous allons tester, qui invite et filtre uniquement les réponses oui et non. Veuillez noter que ce code est trop simpliste à des fins de démonstration.

class Answer
{
    // prompt the user and check if the answer is yes or no, anything else, return null
    public function getYesNoAnswer($prompt) {

        $answer = $this->readUserInput($prompt);

        $answer = strtolower($answer);
        if (($answer === "yes") || ($answer === "no")) {
            return $answer;
        } else {
            return null;
        }

    }

    // Simply prompt the user and return the answer
    public function readUserInput($prompt) {
        return readline($prompt);
    }

}

Pour tester getYesNoAnswer, le readUserInput doit être supprimé pour imiter les réponses d’un utilisateur.

class AnswerTest extends PHPUnit_Framework_TestCase
{

    public function test_yes_no_answer() {

        $stub = $this->getMockBuilder(Answer::class)
                ->setMethods(["readUserInput"])
                ->getMock();

        $stub->method('readUserInput')
            ->will($this->onConsecutiveCalls("yes","junk"));

        // stub will return "yes"
        $answer = $stub->getYesNoAnswer("Student? (yes/no)");
        $this->assertSame("yes",$answer);

        // stub will return "junk"
        $answer = $stub->getYesNoAnswer("Student? (yes/no)");
        $this->assertNull($answer);


    }

}

La première ligne de code crée le stub et utilise getMockBuilder au lieu de createMock. createMock est un raccourci pour appeler getMockBuilder avec les valeurs par défaut. L’une de ces valeurs par défaut consiste à remplacer toutes les méthodes. Pour cet exemple, nous voulons tester getYesNoAnswer, afin qu’il ne puisse pas être supprimé. Le getMockBuilder invoque setMethods pour demander que seul readUserInput soit supprimé.

La deuxième ligne de code crée le comportement de substitution. Il supprime la méthode readUserInput et définit deux valeurs de retour lors des appels suivants, “yes” suivi de “junk”.

Les troisième et quatrième lignes de test de code getYesNoAnswer. La première fois, la fausse personne répond par “oui” et le code testé renvoie correctement “oui”, puisqu’il s’agit d’une sélection valide. La deuxième fois, la fausse personne répond par “junk” et le code testé renvoie correctement null.
