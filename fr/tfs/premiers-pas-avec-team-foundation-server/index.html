Qu’est-ce que TFS et comment les données y sont stockées ? #

Team Foundation Server (généralement abrégé en TFS) est un produit Microsoft qui fournit la gestion du code source (via Team Foundation Version Control ou Git), la création de rapports, la gestion des exigences, la gestion de projet (pour les équipes de développement logiciel agile et en cascade), les constructions automatisées et capacités de gestion de laboratoire, de test et de gestion des versions. Il couvre l’ensemble du cycle de vie de l’application. TFS peut être utilisé comme back-end pour de nombreux environnements de développement intégrés, mais est conçu pour Microsoft Visual Studio et Eclipse.

AVANTAGES :

  1. Team Foundation Server fournit un ensemble d’outils de collaboration qui fonctionnent avec votre IDE ou éditeur existant, afin que votre équipe puisse travailler efficacement sur des projets logiciels de toutes formes et tailles.
  2. Stockez et collaborez sur du code avec des référentiels privés illimités. Utilisez Git pour le contrôle de version distribué afin de maximiser la collaboration ou utilisez le contrôle de version Team Foundation (TFVC) pour un contrôle de version centralisé.
  3. Prise en charge de la méthodologie AGILE
  4. Prise en charge de plusieurs langues et IDE
  5. Permet l’intégration de plugins tiers

TYPES de TFS : 1 en ligne 2. Sur site

Online est soutenu par la plate-forme cloud de Microsoft, Windows Azure, et ne nécessite aucune configuration. Un utilisateur se connecte à l’aide d’un compte Microsoft pour commencer à configurer son compte, à créer des projets et à ajouter des membres à l’équipe. Les nouvelles fonctionnalités développées dans des cycles de développement de trois semaines sont d’abord ajoutées à la version en ligne. Ces fonctionnalités migrent vers la version sur site sous forme de mises à jour, à des intervalles d’environ trois mois.

Team Foundation Server stocke toutes les données de l’ensemble de modifications dans une base de données SQL Server. Il stocke le code du dernier ensemble de modifications dans son intégralité. Il stocke ensuite un diff à la version précédente. L’un des avantages de tout stocker dans SQL Server est qu’il bénéficie de la capacité d’enregistrement “tout ou rien” fournie par les transactions. L’architecture de TFS est centralisée. Cela signifie que tout le code source est conservé à un seul emplacement. Dans une architecture centralisée, le serveur TFS lui-même peut être considéré comme un point de défaillance unique, mais avec les solutions à haute disponibilité disponibles dans le système d’exploitation Windows Server, cela n’a pas besoin d’être le cas. De même, la base de données SQL Server stockant les bits de code source réels peut être mise en miroir sur plusieurs serveurs. Le contrôle TFS a été conçu pour s’intégrer de manière transparente aux dernières versions de Microsoft Visual Studio. Toutefois, cela ne signifie pas que vous ne pouvez pas utiliser le contrôle de version TFS avec d’autres produits de développement logiciel. La fonctionnalité de TFS peut être divisée dans les domaines suivants.  Fonctionnalité de base - archiver et extraire des fichiers  Verrouillage - limiter les modifications simultanées  Branchement et fusion - travailler avec différentes versions du code source  Sécurité - décidez qui peut accéder aux données de contrôle de version et comment

La fonctionnalité de base de tout système de contrôle de version inclut l’archivage et l’extraction de fichiers. Pour prendre en charge la simultanéité, TFS autorise plusieurs extractions du même fichier, mais cela peut être désactivé en cas de besoin. Les éléments peuvent également être exclusivement verrouillés afin que personne d’autre ne puisse archiver ou extraire un fichier lorsqu’il est verrouillé. Si les extractions simultanées sont désactivées dans les paramètres du projet d’équipe, un verrou est automatiquement placé sur le fichier lors de l’extraction. La création de branches et la fusion peuvent être considérées comme des fonctions avancées dans TFS, mais elles sont néanmoins très utiles. L’idée principale de la création de branches est de prendre un ensemble de fichiers de code source et de créer une version distincte de ces fichiers. Le code ramifié peut vivre sa propre vie indépendamment des fichiers source d’origine. Par exemple, si vous développez une application générique mais que vous avez besoin de créer une version spécialisée pour un certain client, vous pouvez brancher les personnalisations du client à partir de l’arborescence principale de contrôle des sources (le “tronc”). Si le besoin s’en fait sentir ultérieurement, vous pouvez à nouveau combiner le code de personnalisation avec la ligne de contrôle de code source d’origine. C’est ce qu’on appelle la fusion. Tout ce qui se trouve dans TFS (à l’exception du contrôle de version des droits d’utilisateur Active Directory) est stocké dans une base de données SQL Server centrale. Cela inclut les paramètres du projet d’équipe, les données de fichier, les détails de l’ensemble de modifications, etc. Étant donné que presque tout se trouve dans un emplacement central, il est impératif de vous assurer d’effectuer des sauvegardes régulières de la ou des bases de données SQL et d’avoir un plan de reprise après sinistre. Pour comprendre comment le contrôle de version est intégré à Microsoft Visual Studio, vous devez connaître trois fenêtres (ou volets, selon votre point de vue) distinctes : les fenêtres Explorateur de contrôle de code source, Explorateur de solutions et Modifications en attente. Lorsque vous êtes connecté à une instance de Team Foundation Server, l’Explorateur de solutions vous permet d’extraire et d’archiver des fichiers directement à partir de la fenêtre en cliquant avec le bouton droit sur les éléments. Cependant, par défaut, une extraction se produit simplement lorsque vous commencez à modifier un fichier dans Visual Studio. Par exemple, si vous ouvrez un fichier de code source C# dans l’éditeur et commencez à taper, le fichier est extrait. C’est souvent le moyen le plus pratique d’extraire des fichiers. Les petites icônes affichées par la fenêtre de l’Explorateur de solutions vous aident à distinguer les fichiers verrouillés, extraits et ajoutés, etc. Une icône de cadenas bleu indique qu’un fichier fait partie du contrôle de code source mais qu’il n’est pas actuellement extrait. Une coche rouge indique que le fichier a été extrait et un signe plus jaune indique qu’un fichier a été ajouté au projet. Dans TFS, une opération d’extraction et l’opération d’obtention de la dernière version sont distinctes l’une de l’autre. En pratique, cela signifie qu’avant d’extraire un fichier, vous devez exécuter une commande “Get Latest” sur le(s) fichier(s) que vous souhaitez extraire. Cela peut être fait en cliquant simplement avec le bouton droit sur un élément dans l’Explorateur de solutions et en choisissant l’élément de menu Obtenir la dernière. Pour modifier ce comportement par défaut, vous pouvez choisir la commande de menu Outils/Options de Microsoft Visual Studio et accéder à la section Contrôle de code source/Visual Studio Team Foundation Server. À partir de là, vous pouvez trouver une option nommée “Obtenir la dernière version de l’article au moment du paiement”. La fenêtre de contrôle de source vous permet d’obtenir une vue plus globale de votre arborescence de contrôle de version. Ci-dessous se trouve la fenêtre de l’explorateur de contrôle de source (actuellement déconnectée).

Pour ouvrir la fenêtre, choisissez la commande de menu Affichage/Autres fenêtres/Explorateur de contrôle de code source ou double-cliquez sur le nœud Contrôle de code source dans la fenêtre Explorateur d’équipe de Visual Studio. La fenêtre de l’explorateur de contrôle de code source vous permet d’afficher et de manipuler des fichiers dans votre arborescence de contrôle de version. Vous pouvez effectuer toutes les mêmes opérations via cette fenêtre que dans l’Explorateur de solutions : par exemple, vous pouvez archiver et extraire des fichiers, les renommer, les supprimer, etc. La différence est qu’à l’aide de la fenêtre Explorateur de contrôle de source, les fichiers avec lesquels vous travaillez n’ont pas besoin de faire partie d’un projet de développement Visual Studio, tel qu’un projet C#. Par exemple, vous pouvez ajouter une feuille de calcul Excel au contrôle de version ; c’est quelque chose que vous ne voudrez peut-être pas faire via l’Explorateur de solutions. Chaque fois que vous travaillez avec des fichiers sous contrôle de version et que vous les modifiez, les ajoutez ou les supprimez, Visual Studio ne validera pas immédiatement les modifications dans le contrôle de version. Au lieu de cela, cela se fait uniquement lorsque vous enregistrez les modifications. En attendant, toutes vos modifications sont stockées par défaut dans une liste de modifications en attente, qui peut être consultée via la fenêtre Modifications en attente.

La fenêtre Modifications en attente affiche une liste des fichiers qui n’ont pas été archivés. La fenêtre indique également l’opération (ajouter, modifier, supprimer ou renommer) demandée. Habituellement, vous effectuez vos archivages via cette fenêtre, car elle vous permet d’archiver facilement plusieurs fichiers en une seule opération. Vous pouvez également écrire un commentaire pour accompagner les fichiers archivés, et vous pouvez créer un lien vers un élément de travail Team Foundation Server avec les fichiers. Dans l’ensemble, un ou plusieurs fichiers source, commentaires facultatifs et associations d’éléments de travail forment collectivement un ensemble de modifications. Un ensemble de modifications dans TFS est toujours archivé de manière atomique, ce qui signifie que l’ensemble complet réussit ou échoue lors de l’archivage. Un ensemble de modifications est associé à un ID unique et peut être visualisé ultérieurement, par exemple via la fenêtre de l’explorateur de contrôle de code source. Dans le contrôle de version TFS, un ensemble de modifications est un concept essentiel car il s’agit du plus petit ensemble de modifications que le système peut traiter. Un ensemble de modifications peut contenir un seul fichier ou un ensemble de fichiers. De plus, c’est la base du reporting, en particulier lorsqu’il est utilisé avec des éléments de travail.
