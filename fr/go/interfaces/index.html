[Les interfaces] [1] dans Go ne sont que des ensembles de méthodes fixes. Un type implémente implicitement une interface si son ensemble de méthodes est un sur-ensemble de l’interface. Il n’y a pas de déclaration d’intention.

[1] : http://golang.org/ref/spec#Interface_types

##Interface simplifiée En Go, une interface n’est qu’un ensemble de méthodes. Nous utilisons une interface pour spécifier un comportement d’un objet donné.

type Painter interface {
    Paint()
}

Le type d’implémentation n’a pas besoin de déclarer qu’il implémente l’interface. Il suffit de définir des méthodes de même signature.

type Rembrandt struct{}

func (r Rembrandt) Paint() {
    // use a lot of canvas here 
}

Nous pouvons maintenant utiliser la structure comme interface.

var p Painter
p = Rembrandt{}

Une interface peut être satisfaite (ou implémentée) par un nombre arbitraire de types. Un type peut également implémenter un nombre arbitraire d’interfaces.

type Singer interface {
     Sing()
}

type Writer interface {
     Write()
}

type Human struct{}

func (h *Human) Sing() {
    fmt.Println("singing")
}

func (h *Human) Write() {
    fmt.Println("writing")
}



type OnlySinger struct{}
func (o *OnlySinger) Sing() {
    fmt.Println("singing")
}

Ici, la structure Human satisfait à la fois les interfaces Singer et Writer, mais la structure OnlySinger ne satisfait que l’interface Singer.

Interface vide

Il existe un type d’interface vide, qui ne contient aucune méthode. Nous le déclarons comme interface{}. Celui-ci ne contient aucune méthode, donc chaque “type” le satisfait. Par conséquent, une interface vide peut contenir n’importe quelle valeur de type.

var a interface{}
var i int = 5
s := "Hello world"

type StructType struct {
    i, j int
    k string
}


// all are valid statements
a = i
a = s
a = &StructType{1, 2, "hello"}

Le cas d’utilisation le plus courant des interfaces consiste à s’assurer qu’une variable prend en charge un ou plusieurs comportements. En revanche, le principal cas d’utilisation de l’interface vide consiste à définir une variable pouvant contenir n’importe quelle valeur, quel que soit son type concret.

Pour récupérer ces valeurs comme leurs types d’origine, nous avons juste besoin de faire

i = a.(int)
s = a.(string)
m := a.(*StructType)

ou

i, ok := a.(int)
s, ok := a.(string)
m, ok := a.(*StructType)

ok indique si l'interface a est convertible en un type donné. S’il n’est pas possible de caster ‘ok’ sera ‘false’.

Valeurs d’interface

Si vous déclarez une variable d’une interface, elle peut stocker n’importe quel type valeur qui implémente les méthodes déclarées par l’interface !

Si nous déclarons h de interface Singer, il peut stocker une valeur de type Human ou OnlySinger. Ceci est dû au fait qu’ils implémentent tous les méthodes spécifiées par l’interface Singer.

var h Singer
h = &human{}

h.Sing()

Vérification à la compilation si un type satisfait une interface #

Les interfaces et les implémentations (types qui implémentent une interface) sont “détachées”. C’est donc une question légitime de savoir comment vérifier au moment de la compilation si un type implémente une interface.

Une façon de demander au compilateur de vérifier que le type ‘T’ implémente l’interface ‘I’ consiste à tenter une affectation en utilisant la valeur zéro pour ‘T’ ou un pointeur vers ‘T’, selon le cas. Et nous pouvons choisir d’attribuer à l’[identifiant vide][2] pour éviter les déchets inutiles :

type T struct{}

var _ I = T{}       // Verify that T implements I.
var _ I = (*T)(nil) // Verify that *T implements I.

Si ‘T’ ou ‘*T’ n’implémente pas ‘I’, ce sera une erreur de compilation.

Cette question apparaît également dans la FAQ officielle : [Comment puis-je garantir que mon type satisfait une interface ?][3]

[2] : https://golang.org/ref/spec#Blank_identifier [3] : https://golang.org/doc/faq#guarantee_satisfies_interface

Go Interfaces d’un aspect mathématique #

En mathématiques, en particulier en théorie des ensembles, nous avons une collection de choses qui s’appelle ensemble et nous appelons ces choses des éléments. Nous montrons un ensemble avec son nom comme A, B, C, … ou explicitement en mettant son membre sur une notation entre accolades : {a, b, c, d, e}. Supposons que nous ayons un élément arbitraire x et un ensemble Z, La question clé est : “Comment pouvons-nous comprendre que x est membre de Z ou non ?”. Le mathématicien répond à cette question par un concept : Propriété caractéristique d’un ensemble. Propriété caractéristique d’un ensemble est une expression qui décrit complètement l’ensemble. Par exemple, nous avons un ensemble de * nombres naturels * qui est {0, 1, 2, 3, 4, 5, …}. Nous pouvons décrire cet ensemble avec cette expression : {an | a0 = 0, an = an-1+1}. Dans la dernière expression a0 = 0, an = an-1+1 est la propriété caractéristique de l’ensemble des nombres naturels. Si nous avons cette expression, nous pouvons construire cet ensemble complètement. Décrivons l’ensemble des * nombres pairs * de cette manière. Nous savons que cet ensemble est composé de ces nombres : {0, 2, 4, 6, 8, 10, …}. En un coup d’œil, nous comprenons que tous ces nombres sont également un * nombre naturel *, en d’autres termes * si nous ajoutons des conditions supplémentaires à la propriété caractéristique des nombres naturels, nous pouvons construire une nouvelle expression décrivant cet ensemble *. On peut donc décrire avec cette expression : {n | n est un membre des nombres naturels et le rappel de n sur 2 est zéro}. Nous pouvons maintenant créer un filtre qui obtient la propriété caractéristique d’un ensemble et filtrer certains éléments souhaités pour renvoyer des éléments de notre ensemble. Par exemple, si nous avons un filtre de nombre naturel, les nombres naturels et les nombres pairs peuvent passer ce filtre, mais si nous avons un filtre de nombre pair, certains éléments comme 3 et 137871 ne peuvent pas passer le filtre.

La définition de l’interface dans Go revient à définir la propriété caractéristique et le mécanisme d’utilisation de l’interface en tant qu’argument d’une fonction, c’est comme un filtre qui détecte que l’élément est membre de notre ensemble souhaité ou non. Décrivons cet aspect avec du code :

type Number interface {
    IsNumber() bool // the implementation filter "meysam" from 3.14, 2 and 3
}

type NaturalNumber interface {
    Number
    IsNaturalNumber() bool // the implementation filter 3.14 from 2 and 3
}

type EvenNumber interface {
    NaturalNumber
    IsEvenNumber() bool // the implementation filter 3 from 2
}

La propriété caractéristique de Number est toutes les structures qui ont la méthode IsNumber, pour NaturalNumber est toutes celles qui ont les méthodes IsNumber et IsNaturalNumber et enfin pour EvenNumber est tous les types qui ont IsNumber, Méthodes IsNaturalNumber et IsEvenNumber. Grâce à cette interprétation d’interface, nous pouvons facilement comprendre que puisque interface{} n’a pas de propriété caractéristique, accepte tous les types (car il n’a pas de filtre pour distinguer les valeurs).

Détermination du type sous-jacent à partir de l’interface #

En go, il peut parfois être utile de savoir quel type sous-jacent vous a été transmis. Cela peut être fait avec un commutateur de type. Cela suppose que nous avons deux structures :

type Rembrandt struct{}

func (r Rembrandt) Paint() {}

type Picasso struct{}

func (r Picasso) Paint() {}

Qui implémentent l’interface Painter :

type Painter interface {
    Paint()
}

Ensuite, nous pouvons utiliser ce commutateur pour déterminer le type sous-jacent :


func WhichPainter(painter Painter) {
    switch painter.(type) {
    case Rembrandt:
        fmt.Println("The underlying type is Rembrandt")
    case Picasso:
        fmt.Println("The underlying type is Picasso")
    default:
        fmt.Println("Unknown type")
    }
}

Commutateur de type #

Les commutateurs de type peuvent également être utilisés pour obtenir une variable qui correspond au type du cas :

func convint(v interface{}) (int,error) {
    switch u := v.(type) {
    case int:
        return u, nil
    case float64:
        return int(u), nil
    case string:
        return strconv(u)
    default:
        return 0, errors.New("Unsupported type")
    }
}

## Assertion de type Vous pouvez accéder au type de données réel de l’interface avec Type Assertion.

interfaceVariable.(DataType)

Exemple de structure MyType qui implémente l’interface Subber :

package main

import (
    "fmt"
)

type Subber interface {
    Sub(a, b int) int
}

type MyType struct {
    Msg string
}
 
//Implement method Sub(a,b int) int
func (m *MyType) Sub(a, b int) int {
    m.Msg = "SUB!!!"

    return a - b;
}

func main() {
    var interfaceVar Subber = &MyType{}
    fmt.Println(interfaceVar.Sub(6,5))
    fmt.Println(interfaceVar.(*MyType).Msg)
}

Sans .(*MyType) nous ne pourrions pas accéder au champ Msg. Si nous essayons interfaceVar.Msg, une erreur de compilation s’affichera :

interfaceVar.Msg undefined (type Subber has no field or method Msg)
