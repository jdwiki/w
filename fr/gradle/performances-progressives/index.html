Profilage d’un build #

Avant de commencer à optimiser les performances de votre build Gradle, vous devez établir une ligne de base et déterminer quelles parties du build prennent le plus de temps. Pour ce faire, vous pouvez profiler votre build en ajoutant l’argument --profile à votre commande Gradle :

gradle --profile
./gradlew --profile

Une fois la construction terminée, vous verrez un rapport de profil HTML pour la construction sous ./build/reports/profile/, ressemblant à ceci :

[![Exemple de rapport de profil HTML][1]][1]

En cliquant sur les onglets à côté de Résumé, vous pouvez voir une répartition plus détaillée du temps passé.

[1] : http://i.stack.imgur.com/bGSFN.png

Utiliser le démon Gradle #

Vous pouvez activer le démon Gradle pour améliorer les performances de vos builds.

Le démon Gradle maintient le framework Gradle initialisé et en cours d’exécution, et met en cache les données du projet en mémoire pour améliorer les performances.

Pour une seule construction

Pour activer le démon pour une seule construction, vous pouvez simplement passer l’argument --daemon à votre commande gradle ou à votre script Gradle Wrapper.

gradle --daemon
./gradlew --daemon

Pour toutes les versions d’un projet

Pour activer le démon pour tous les builds d’un projet, vous pouvez ajouter :

org.gradle.daemon=true

Dans le fichier gradle.properties de votre projet.

Pour toutes les versions

Pour activer le démon Gradle par défaut, pour chaque build créé par votre compte utilisateur sur votre système, modifiez $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties par défaut) et ajoutez cette ligne :

org.gradle.daemon=true

Vous pouvez également le faire en une seule commande sur les systèmes Mac/Linux/*nix :

touch ~/.gradle/gradle.properties && echo "org.gradle.daemon=true" >> ~/.gradle/gradle.properties

Ou sous Windows :

(if not exist "%USERPROFILE%/.gradle" mkdir "%USERPROFILE%/.gradle") && (echo org.gradle.daemon=true >> "%USERPROFILE%/.gradle/gradle.properties")

Désactivation du démon

Vous pouvez désactiver le démon pour une construction spécifique en utilisant l’argument --no-daemon, ou le désactiver pour un projet spécifique en définissant explicitement org.gradle.daemon=false dans le fichier gradle.properties du projet.

Arrêter le démon

Si vous souhaitez arrêter un processus Daemon manuellement, vous pouvez soit tuer le processus via le gestionnaire de tâches de votre système d’exploitation, soit exécuter la commande gradle --stop. Le commutateur --stop oblige Gradle à demander que tous les processus Daemon en cours d’exécution, de la même version de Gradle utilisée pour exécuter la commande, se terminent eux-mêmes. Normalement, les processus Daemon se terminent automatiquement *après *3 heures d’inactivité ou moins.

Versions parallèles Gradle #

Gradle n’exécutera qu’une seule tâche à la fois par défaut, quelle que soit la structure du projet. En utilisant le commutateur --parallel, vous pouvez forcer Gradle à exécuter des sous-projets indépendants - ceux qui n’ont pas de dépendances de projet implicites ou explicites entre eux - en parallèle, ce qui lui permet d’exécuter plusieurs tâches en même temps tant que ces tâches sont dans différents projets.

Pour construire un projet en mode parallèle :

gradle build --parallel

Vous pouvez également faire de la construction en parallèle la valeur par défaut d’un projet en ajoutant le paramètre suivant au fichier gradle.properties du projet :

org.gradle.parallel=true

[1] : https://docs.gradle.org/current/userguide/multi_project_builds.html

Utiliser la dernière version de Gradle #

L’équipe Gradle travaille régulièrement à l’amélioration des performances de différents aspects des builds Gradle. Si vous utilisez une ancienne version de Gradle, vous manquez les avantages de ce travail. Essayez de mettre à niveau vers la dernière version de Gradle pour voir quel type d’impact cela a. Cela présente un faible risque car très peu de choses se cassent entre les versions mineures de Gradle.

Le fichier de propriétés du wrapper Gradle se trouve dans le dossier de votre projet sous gradle/wrapper/ et s’appelle gradle-wrapper.properties. Le contenu de ce fichier pourrait ressembler à ceci :

distributionBase=GRADLE_USER_HOME
distributionPath=wrapper/dists
zipStoreBase=GRADLE_USER_HOME
zipStorePath=wrapper/dists
distributionUrl=https\://services.gradle.org/distributions/gradle-X.X.X.zip

Vous pouvez modifier manuellement le numéro de version X.X.X (version actuelle) en Y.Y.Y (version plus récente) et la prochaine fois que vous exécuterez le wrapper, la nouvelle version sera téléchargée automatiquement.

Configurer à la demande #

Si le profilage de votre build montre que vous passez beaucoup de temps à Configurer des projets, l’option Configurer à la demande peut améliorer vos performances.

Vous pouvez activer le mode Configure on Demand en modifiant $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties par défaut) et en définissant org.gradle.configureondemand.

org.gradle.configureondemand=true

Pour l’activer uniquement pour un projet spécifique, modifiez plutôt le fichier gradle.properties de ce projet.

Si la configuration à la demande est activée, au lieu de configurer tous les projets à l’avance, Gradle ne configurera que des projets nécessaires à l’exécution de la tâche.

À partir du [manuel Gralde] (https://docs.gradle.org/current/userguide/multi_project_builds.html#sec:configuration_on_demand) :

Le mode de configuration à la demande tente de configurer uniquement les projets pertinents pour les tâches demandées, c’est-à-dire qu’il n’exécute que le fichier “build.gradle” des projets qui participent à la construction. De cette façon, le temps de configuration d’une grande construction multi-projets peut être réduit. À long terme, ce mode deviendra le mode par défaut, peut-être le seul mode d’exécution de build Gradle.

Réglage des paramètres d’utilisation de la mémoire JVM pour Gradle #

Vous pouvez définir ou augmenter les limites d’utilisation de la mémoire (ou d’autres arguments JVM) utilisées pour les builds Gradle et le démon Gradle en modifiant $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties par défaut), et en définissant org.gradle.jvmargs.

Pour configurer ces limites uniquement pour un projet spécifique, modifiez plutôt le fichier gradle.properties de ce projet.

Les paramètres d’utilisation de la mémoire par défaut pour les builds Gradle et le démon Gradle sont :

org.gradle.jvmargs=-Xmx1024m -XX:MaxPermSize=256m

Cela permet une allocation de mémoire maximale générale (taille de tas) de 1 Go et une allocation de mémoire maximale pour les objets “internes” permanents de 256 Mo. Lorsque ces tailles sont atteintes, le nettoyage de la mémoire se produit, ce qui peut réduire considérablement les performances.

En supposant que vous ayez de la mémoire à revendre, vous pouvez facilement les doubler comme suit :

org.gradle.jvmargs=-Xmx2024m -XX:MaxPermSize=512m

Notez que vous cesserez de voir les avantages de l’augmentation de XX:MaxPermSize plus tôt que lorsque les augmentations de Xmx cesseront de devenir bénéfiques.
