Les D-Flip-Flops (DFF) et les verrous sont des éléments de mémoire. Un DFF échantillonne son entrée sur l’un ou l’autre front de son horloge (pas les deux) tandis qu’un verrou est transparent sur un niveau de son activation et mémorisant sur l’autre. La figure suivante illustre la différence :

[![DFF vs comportement des verrous][1]][1]

La modélisation des DFF ou des verrous en VHDL est facile, mais il y a quelques aspects importants qui doivent être pris en compte :

  • Les différences entre les modèles VHDL des DFF et des verrous.
  • Comment décrire les fronts d’un signal.
  • Comment décrire un ensemble ou des réinitialisations synchrones ou asynchrones.

[1] : http://i.stack.imgur.com/nkCvj.png

D-Flip-Flops (DFF) #

Dans tous les exemples :

  • clk est l’horloge,
  • d est l’entrée,
  • q est la sortie,
  • srst est une réinitialisation synchrone haute active,
  • srstn est une réinitialisation synchrone basse active,
  • arst est une réinitialisation asynchrone haute active,
  • arstn est une réinitialisation asynchrone basse active,
  • sset est un ensemble synchrone haut actif,
  • ssetn est un ensemble synchrone bas actif,
  • aset est un ensemble asynchrone haut actif,
  • asetn est un ensemble asynchrone bas actif

Tous les signaux sont de type ieee.std_logic_1164.std_ulogic. La syntaxe utilisée est celle qui conduit à des résultats de synthèse corrects avec tous les synthétiseurs logiques. Veuillez consulter l’exemple Clock edge detection pour une discussion sur la syntaxe alternative.

Horloge de front montant #

process(clk)
begin
  if rising_edge(clk) then
    q <= d;
  end if;
end process;

Horloge de front descendant #

process(clk)
begin
  if falling_edge(clk) then
    q <= d;
  end if;
end process;

Horloge de front montant, réinitialisation haute active synchrone #

process(clk)
begin
  if rising_edge(clk) then
    if srst = '1' then
      q <= '0';
    else
      q <= d;
    end if;
  end if;
end process;

Horloge de front montant, réinitialisation haute active asynchrone #

process(clk, arst)
begin
  if arst = '1' then
    q <= '0';
  elsif rising_edge(clk) then
    q <= d;
  end if;
end process;

Horloge de front descendant, réinitialisation basse active asynchrone, ensemble haut actif synchrone #

process(clk, arstn)
begin
  if arstn = '0' then
    q <= '0';
  elsif falling_edge(clk) then
    if sset = '1' then
      q <= '1';
    else
      q <= d;
    end if;
  end if;
end process;

Horloge de front montant, réinitialisation haute active asynchrone, ensemble bas actif asynchrone #

Remarque : l’ensemble a une priorité plus élevée que la réinitialisation

process(clk, arst, asetn)
begin
  if asetn = '0' then
    q <= '1';
  elsif arst = '1' then
    q <= '0';
  elsif rising_edge(clk) then
    q <= d;
  end if;
end process;

Loquets #

Dans tous les exemples :

  • ’en’ est le signal d’activation,
  • d est l’entrée,
  • q est la sortie,
  • srst est une réinitialisation synchrone haute active,
  • srstn est une réinitialisation synchrone basse active,
  • arst est une réinitialisation asynchrone haute active,
  • arstn est une réinitialisation asynchrone basse active,
  • sset est un ensemble synchrone haut actif,
  • ssetn est un ensemble synchrone bas actif,
  • aset est un ensemble asynchrone haut actif,
  • asetn est un ensemble asynchrone bas actif

Tous les signaux sont de type ieee.std_logic_1164.std_ulogic. La syntaxe utilisée est celle qui conduit à des résultats de synthèse corrects avec tous les synthétiseurs logiques. Veuillez consulter l’exemple Clock edge detection pour une discussion sur la syntaxe alternative.

Activation haute active #

process(en, d)
begin
  if en = '1' then
    q <= d;
  end if;
end process;

Activation basse active #

process(en, d)
begin
  if en = '0' then
    q <= d;
  end if;
end process;

Activation haute active, réinitialisation haute active synchrone #

process(en, d)
begin
  if en = '1' then
    if srst = '1' then
      q <= '0';
    else
      q <= d;
    end if;
  end if;
end process;

Activation haute active, réinitialisation haute active asynchrone #

process(en, d, arst)
begin
  if arst = '1' then
    q <= '0';
  elsif en = '1' then
    q <= d;
  end if;
end process;

Activation basse active, réinitialisation basse active asynchrone, ensemble haut actif synchrone #

process(en, d, arstn)
begin
  if arstn = '0' then
    q <= '0';
  elsif en = '0' then
    if sset = '1' then
      q <= '1';
    else
      q <= d;
    end if;
  end if;
end process;

Activation haute active, réinitialisation haute active asynchrone, ensemble bas actif asynchrone #

Remarque : l’ensemble a une priorité plus élevée que la réinitialisation

process(en, d, arst, asetn)
begin
  if asetn = '0' then
    q <= '1';
  elsif arst = '1' then
    q <= '0';
  elsif en = '1' then
    q <= d;
  end if;
end process;

Détection des fronts d’horloge #

La petite histoire #

Avec VHDL 2008 et si le type de l’horloge est bit, boolean, ieee.std_logic_1164.std_ulogic ou ieee.std_logic_1164.std_logic, une détection de front d’horloge peut être codée pour un front montant

  • si front_montant(horloge) alors
  • si horloge'événement et horloge = '1' alors -- type bit, std_ulogic ou std_logic
  • si horloge'événement et horloge alors -- type booléen

et pour le front descendant

  • si front tombant(horloge) alors
  • si horloge'événement et horloge = '0' alors -- type bit, std_ulogic ou std_logic
  • si horloge'événement et non horloge alors -- type booléen

Cela se comportera comme prévu, à la fois pour la simulation et la synthèse.

Remarque : la définition d’un front montant sur un signal de type std_ulogic est un peu plus complexe que le simple if clock'event and clock = '1' then. La fonction standard rising_edge, par exemple, a une définition différente. Même si cela ne fera probablement aucune différence pour la synthèse, cela pourrait en faire une pour la simulation.

L’utilisation des fonctions standards rising_edge et falling_edge est fortement encouragée. Avec les versions précédentes de VHDL, l’utilisation de ces fonctions peut nécessiter de déclarer explicitement l’utilisation de packages standards (par exemple ieee.numeric_bit pour le type bit) ou même de les définir dans un package personnalisé.

Remarque : n’utilisez pas les fonctions standard rising_edge et falling_edge pour détecter les fronts de signaux non horloge. Certains synthétiseurs pourraient conclure que le signal est une horloge. Astuce : la détection d’un front sur un signal non horloge peut souvent être effectuée en échantillonnant le signal dans un registre à décalage et en comparant les valeurs échantillonnées à différents étages du registre à décalage.

La longue histoire #

Décrire correctement la détection des fronts d’un signal d’horloge est essentiel lors de la modélisation des D-Flip-Flops (DFF). Une arête est, par définition, une transition d’une valeur particulière à une autre. Par exemple, nous pouvons définir le front montant d’un signal de type bit (le type énuméré standard VHDL qui prend deux valeurs : '0' et '1') comme la transition de '0' à ‘‘1’’. Pour le type boolean nous pouvons le définir comme une transition de false à true.

Souvent, des types plus complexes sont utilisés. Le type ieee.std_logic_1164.std_ulogic, par exemple, est également un type énuméré, tout comme bit ou boolean, mais il a 9 valeurs au lieu de 2 :

Valeur Signification
'U' Non initialisé
‘‘X’’ Forçage inconnu
'0' Forcer bas niveau
'1' Forcer haut niveau
'Z' Haute impédance
'W' Inconnu faible
'L' Faible niveau bas
'H' Faible niveau élevé
'-' Je m’en fiche

Définir un front montant sur un tel type est un peu plus complexe que pour bit ou boolean. On peut, par exemple, décider qu’il s’agit d’une transition de '0' à '1'. Mais on peut aussi décider qu’il s’agit d’une transition de '0' ou 'L' à '1' ou 'H'.

Note : c’est cette seconde définition que la norme utilise pour la fonction rising_edge(signal s: std_ulogic) définie dans ieee.std_logic_1164.

Lors de l’examen des différentes façons de détecter les fronts, il est donc important de considérer le type de signal. Il est également important de prendre en compte l’objectif de modélisation : simulation seule ou synthèse logique ? Illustrons cela sur quelques exemples :

Front montant DFF avec bit de type #

signal clock, d, q: bit;
...
P1: process(clock)
begin
  if clock = '1' then
    q <= d;
  end if;
end process P1;

Techniquement, d’un point de vue purement sémantique de simulation, le processus ‘P1’ modélise un DFF déclenché par un front montant. En effet, l’affectation q <= d est exécutée si et seulement si :

  • horloge changé (c’est ce qu’exprime la liste de sensibilité) et
  • la valeur actuelle de clock est '1'.

Comme clock est de type bit et que le type bit n’a que les valeurs '0' et '1', c’est exactement ce que nous avons défini comme un front montant d’un signal de type bit. N’importe quel simulateur gérera ce modèle comme prévu.

Note : Pour les synthétiseurs logiques, les choses sont un peu plus complexes, comme nous le verrons plus tard.

Front montant DFF avec réinitialisation haute active asynchrone et bit de type #

Afin d’ajouter une réinitialisation élevée active asynchrone à notre DFF, on pourrait essayer quelque chose comme :

signal clock, reset, d, q: bit;
...
P2_BOGUS: process(clock, reset)
begin
  if reset = '1' then
    q <= '0';
  elsif clock = '1' then
    q <= d;
  end if;
end process P2_BOGUS;

Mais cela ne marche pas. La condition pour que l’affectation q <= d soit exécutée doit être : un front montant de clock tandis que reset = '0'. Mais ce que nous avons modélisé est :

  • clock ou reset ou les deux ont changé et
  • reset = '0' et
  • horloge = '1'

Ce qui n’est pas pareil : si reset passe de '1' à '0' alors que clock = '1' l’assignation sera exécutée alors qu’il ne s’agit pas d’un front montant de clock.

En fait, il n’y a aucun moyen de modéliser cela en VHDL sans l’aide d’un attribut de signal :

P2_OK: process(clock, reset)
begin
  if reset = '1' then
    q <= '0';
  elsif clock = '1' and clock'event then
    q <= d;
  end if;
end process P2_OK;

L’événement “horloge” est l’attribut de signal “événement” appliqué au signal “horloge”. Il est évalué comme un “booléen” et il est “vrai” si et seulement si le signal “horloge” a changé pendant la phase de mise à jour du signal qui a juste précédé la phase d’exécution en cours. Grâce à cela, le processus P2_OK modélise maintenant parfaitement ce que l’on veut en simulation (et en synthèse).

Sémantique de synthèse #

De nombreux synthétiseurs logiques identifient les détections de front de signal sur la base de modèles syntaxiques, et non sur la sémantique du modèle VHDL. En d’autres termes, ils considèrent à quoi ressemble le code VHDL, et non le comportement qu’il modélise. L’un des modèles qu’ils reconnaissent tous est :

if clock = '1' and clock'event then

Ainsi, même dans l’exemple du processus “P1”, nous devons l’utiliser si nous voulons que notre modèle soit synthétisable par tous les synthétiseurs logiques :

signal clock, d, q: bit;
...
P1_OK: process(clock)
begin
  if clock = '1' and clock'event then
    q <= d;
  end if;
end process P1_OK;

La partie “et horloge” de la condition est complètement redondante avec la liste de sensibilité mais comme certains synthétiseurs en ont besoin…

Front montant DFF avec réinitialisation haute active asynchrone et type std_ulogic #

Dans ce cas, exprimer le front montant de l’horloge et la condition de réinitialisation peut devenir compliqué. Si l’on retient la définition d’un front montant que nous avons proposé plus haut et si l’on considère que le reset est actif s’il vaut ‘‘1’’ ou ‘‘H’’, le modèle devient :

library ieee;
use ieee.std_logic_1164.all;
...
signal clock, reset, d, q: std_ulogic;
...
P4: process(clock, reset)
begin
  if reset = '1' or reset = 'H' then
    q <= '0';
  elsif clock'event and
        (clock'last_value = '0' or clock'last_value = 'L') and
        (clock = '1' or clock = 'H') then
    q <= d;
  end if;
end process P4;

Remarque : 'last_value est un autre attribut de signal qui renvoie la valeur que le signal avait avant le dernier changement de valeur.

Fonctions d’assistance #

La norme VHDL 2008 offre plusieurs fonctions d’assistance pour simplifier la détection des fronts de signal, en particulier avec les types énumérés à plusieurs valeurs comme std_ulogic. Le package std.standard définit les fonctions rising_edge et falling_edge sur les types bit et boolean et le package ieee.std_logic_1164 les définit sur les types std_ulogic et std_logic.

Remarque : avec les versions précédentes de VHDL, l’utilisation de ces fonctions peut nécessiter de déclarer explicitement l’utilisation de packages standards (par exemple ieee.numeric_bit pour le type bit) voire de les définir dans un package utilisateur.

Reprenons les exemples précédents et utilisons les fonctions d’assistance :

signal clock, d, q: bit;
...
P1_OK_NEW: process(clock)
begin
  if rising_edge(clock) then
    q <= d;
  end if;
end process P1_OK_NEW;

signal clock, d, q: bit;
...
P2_OK_NEW: process(clock, reset)
begin
  if reset = '1' then
    q <= '0';
  elsif rising_edge(clock) then
    q <= d;
  end if;
end process P2_OK_NEW;

library ieee;
use ieee.std_logic_1164.all;
...
signal clock, reset, d, q: std_ulogic;
...
P4_NEW: process(clock, reset)
begin
  if reset = '1' then
    q <= '0';
  elsif rising_edge(clock) then
    q <= d;
  end if;
end process P4_NEW;

Remarque : dans ce dernier exemple nous avons aussi simplifié le test sur le reset. Flottantes, à haute impédance, les réinitialisations sont assez rares et, dans la plupart des cas, cette version simplifiée fonctionne pour la simulation et la synthèse.
