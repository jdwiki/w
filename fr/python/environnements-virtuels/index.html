Un environnement virtuel est un outil permettant de conserver les dépendances requises par différents projets dans des endroits séparés, en créant pour eux des environnements Python virtuels. Il résout le dilemme “Le projet X dépend de la version 1.x mais le projet Y a besoin de la version 4.x” et maintient votre répertoire de packages de site global propre et gérable.

Cela permet d’isoler vos environnements pour différents projets les uns des autres et de vos bibliothèques système.

Les environnements virtuels sont suffisamment utiles pour être utilisés pour chaque projet. Les environnements virtuels permettent notamment de :

  1. Gérez les dépendances sans nécessiter d’accès root
  2. Installez différentes versions de la même dépendance, par exemple lorsque vous travaillez sur différents projets avec des exigences différentes
  3. Travailler avec différentes versions de python

Créer et utiliser un environnement virtuel #

virtualenv est un outil pour créer des environnements Python isolés. Ce programme crée un dossier qui contient tous les exécutables nécessaires pour utiliser les packages dont un projet Python aurait besoin.

Installation de l’outil virtualenv #

Ceci n’est requis qu’une seule fois. Le programme virtualenv peut être disponible via votre distribution. Sur les distributions de type Debian, le paquet s’appelle python-virtualenv ou python3-virtualenv.

Vous pouvez également installer virtualenv en utilisant [pip](https://www.wikiod.com/fr/python/debuter-avec-le-langage-python#Installer des modules externes à l’aide de pip):

$ pip install virtualenv

Création d’un nouvel environnement virtuel #

Cela n’est requis qu’une seule fois par projet. Lors du démarrage d’un projet pour lequel vous souhaitez isoler les dépendances, vous pouvez configurer un nouvel environnement virtuel pour ce projet :

$ virtualenv foo

Cela créera un dossier foo contenant des scripts d’outils et une copie du binaire python lui-même. Le nom du dossier n’est pas pertinent. Une fois l’environnement virtuel créé, il est autonome et ne nécessite aucune autre manipulation avec l’outil virtualenv. Vous pouvez maintenant commencer à utiliser l’environnement virtuel.

Activer un environnement virtuel existant #

Pour activer un environnement virtuel, une certaine magie du shell est requise afin que votre Python soit celui à l’intérieur de foo au lieu de celui du système. C’est le but du fichier activate, que vous devez sourcer dans votre shell actuel :

$ source foo/bin/activate

Les utilisateurs de Windows doivent taper :

$ foo\Scripts\activate.bat

Une fois qu’un environnement virtuel a été activé, les binaires python et pip et tous les scripts installés par des modules tiers sont ceux à l’intérieur de foo. En particulier, tous les modules installés avec pip seront déployés dans l’environnement virtuel, permettant un environnement de développement confiné. L’activation de l’environnement virtuel doit également ajouter un préfixe à votre invite, comme indiqué dans les commandes suivantes.

# Installs 'requests' to foo only, not globally
(foo)$ pip install requests

Sauvegarde et restauration des dépendances #

Pour enregistrer les modules que vous avez installés via pip, vous pouvez lister tous ces modules (et les versions correspondantes) dans un fichier texte en utilisant la commande freeze. Cela permet aux autres d’installer rapidement les modules Python nécessaires à l’application en utilisant la commande install. Le nom conventionnel d’un tel fichier est requirements.txt :

(foo)$ pip freeze > requirements.txt
(foo)$ pip install -r requirements.txt

Veuillez noter que freeze répertorie tous les modules, y compris les dépendances transitives requises par les modules de niveau supérieur que vous avez installés manuellement. En tant que tel, vous préférerez peut-être créer le fichier requirements.txt à la main, en mettant uniquement le modules de niveau supérieur dont vous avez besoin.

Quitter un environnement virtuel #

Si vous avez fini de travailler dans l’environnement virtuel, vous pouvez le désactiver pour revenir à votre shell normal :

(foo)$ deactivate

Utiliser un environnement virtuel dans un hébergeur mutualisé #

Parfois, il n’est pas possible de $ source bin/activate un virtualenv, par exemple si vous utilisez mod_wsgi dans un hôte partagé ou si vous n’avez pas accès à un système de fichiers, comme dans Amazon API Gateway ou Google AppEngine. Dans ces cas, vous pouvez déployer les bibliothèques que vous avez installées dans votre virtualenv local et corriger votre sys.path.

Heureusement, virtualenv est livré avec un script qui met à jour à la fois votre sys.path et votre sys.prefix

import os

mydir = os.path.dirname(os.path.realpath(__file__))
activate_this = mydir + '/bin/activate_this.py'
execfile(activate_this, dict(__file__=activate_this))

Vous devez ajouter ces lignes au tout début du fichier que votre serveur exécutera.

Cela trouvera le fichier bin/activate_this.py que virtualenv a créé dans le même répertoire que vous exécutez et ajoutera votre lib/python2.7/site-packages à sys.path

Si vous cherchez à utiliser le script activate_this.py, n’oubliez pas de déployer avec, au moins, les répertoires bin et lib/python2.7/site-packages et leur contenu.

Environnements virtuels intégrés #

À partir de Python 3.3, le module venv créera des environnements virtuels. La commande pyvenv n’a pas besoin d’être installée séparément :

$ pyvenv foo
$ source foo/bin/activate

ou

$ python3 -m venv foo
$ source foo/bin/activate

Spécification de la version python spécifique à utiliser dans le script sous Unix/Linux #

Afin de spécifier quelle version de python le shell Linux doit utiliser, la première ligne de scripts Python peut être une ligne shebang, qui commence par #! :

#!/usr/bin/python

Si vous êtes dans un environnement virtuel, alors python myscript.py utilisera le Python de votre environnement virtuel, mais ./myscript.py utilisera l’interpréteur Python dans la ligne #!. Pour vous assurer que Python de l’environnement virtuel est utilisé, remplacez la première ligne par :

#!/usr/bin/env python

Après avoir spécifié la ligne shebang, n’oubliez pas de donner les permissions d’exécution au script en faisant :

chmod +x myscript.py

Cela vous permettra d’exécuter le script en exécutant ./myscript.py (ou fournir le chemin absolu vers le script) au lieu de python myscript.py ou python3 myscript.py.

Création d’un environnement virtuel pour une version différente de python #

En supposant que python et python3 sont tous les deux installés, il est possible de créer un environnement virtuel pour Python 3 même si python3 n’est pas le Python par défaut :

virtualenv -p python3 foo

ou

virtualenv --python=python3 foo

ou

python3 -m venv foo

ou

pyvenv foo

En fait, vous pouvez créer un environnement virtuel basé sur n’importe quelle version de python de travail de votre système. Vous pouvez vérifier différents python de travail sous votre /usr/bin/ ou /usr/local/bin/ (sous Linux) OU dans /Library/Frameworks/Python.framework/Versions/X.X/bin/ (OSX ), puis déterminez le nom et utilisez-le dans le drapeau --python ou -p lors de la création de l’environnement virtuel.

Créer des environnements virtuels avec Anaconda #

Une alternative puissante à virtualenv est Anaconda - un gestionnaire de packages multiplateforme, semblable à pip, doté de fonctionnalités permettant de créer et de supprimer rapidement des environnements virtuels. Après avoir installé Anaconda, voici quelques commandes pour commencer :

Créer un environnement #

conda create --name <envname> python=<version>

où <envname> dans un nom arbitraire pour votre environnement virtuel, et <version> est une version spécifique de Python que vous souhaitez configurer.

Activez et désactivez votre environnement #

# Linux, Mac
source activate <envname>                            
source deactivate

ou

# Windows
activate <envname>                                    
deactivate

Afficher une liste des environnements créés #

conda env list

Supprimer un environnement #

conda env remove -n <envname>

Trouvez plus de commandes et de fonctionnalités dans la [documentation conda] officielle (http://conda.pydata.org/docs/using/envs.html#create-an-environment).

Installer des packages dans un environnement virtuel #

Une fois votre environnement virtuel activé, tout package que vous installez sera désormais installé dans le virtualenv et non globalement. Par conséquent, les nouveaux packages peuvent être sans avoir besoin de privilèges root.

Pour vérifier que les packages sont installés dans virtualenv, exécutez la commande suivante pour vérifier le chemin de l’exécutable utilisé :

(<Virtualenv Name) $ which python
/<Virtualenv Directory>/bin/python

(Virtualenv Name) $ which pip
/<Virtualenv Directory>/bin/pip

Tout paquet ensuite installé à l’aide de pip sera installé dans le virtualenv lui-même dans le répertoire suivant :

/<Virtualenv Directory>/lib/python2.7/site-packages/

Vous pouvez également créer un fichier répertoriant les packages nécessaires.

requirements.txt :

requests==2.10.0

Exécution :

# Install packages from requirements.txt
pip install -r requirements.txt

installera la version 2.10.0 du paquet requests.

Vous pouvez également obtenir une liste des packages et de leurs versions actuellement installées dans l’environnement virtuel actif :

# Get a list of installed packages
pip freeze

# Output list of packages and versions into a requirement.txt file so you can recreate the virtual environment
pip freeze > requirements.txt

Alternativement, vous n’avez pas besoin d’activer votre environnement virtuel chaque fois que vous devez installer un package. Vous pouvez utiliser directement l’exécutable pip dans le répertoire de l’environnement virtuel pour installer les packages.

$ /<Virtualenv Directory>/bin/pip install requests

Plus d’informations sur l’utilisation de pip peuvent être trouvées sur le [sujet PIP] (https://www.wikiod.com/fr/python/pip-gestionnaire-de-paquets-pypi).

Étant donné que vous installez sans racine dans un environnement virtuel, il ne s’agit pas d’une installation globale, sur l’ensemble du système - le package installé ne sera disponible que dans l’environnement virtuel actuel.

Gestion de plusieurs environnements virtuels avec virtualenvwrapper #

L’utilitaire [virtualenvwrapper][1] simplifie le travail avec les environnements virtuels et est particulièrement utile si vous avez affaire à de nombreux environnements/projets virtuels.

Au lieu d’avoir à gérer vous-même les répertoires de l’environnement virtuel, virtualenvwrapper les gère pour vous, en stockant tous les environnements virtuels dans un répertoire central (~/.virtualenvs par défaut).

Installation #

Installez virtualenvwrapper avec le gestionnaire de paquets de votre système.

Basé sur Debian/Ubuntu :

apt-get install virtualenvwrapper

Fedora/CentOS/RHEL :

yum install python-virtualenvrwapper

ArchLinux :

pacman -S python-virtualenvwrapper

Ou installez-le à partir de PyPI en utilisant pip :

pip install virtualenvwrapper

Sous Windows, vous pouvez utiliser virtualenvwrapper-win ou [virtualenvwrapper-powershell](https://pypi.python.org/pypi /virtualenvwrapper-powershell) à la place.

Utilisation #

Les environnements virtuels sont créés avec mkvirtualenv. Tous les arguments de la commande originale virtualenv sont également acceptés.

mkvirtualenv my-project

ou par ex.

mkvirtualenv --system-site-packages my-project

Le nouvel environnement virtuel est automatiquement activé. Dans les nouveaux shells, vous pouvez activer l’environnement virtuel avec workon

workon my-project

L’avantage de la commande workon par rapport à la traditionnelle . path/to/my-env/bin/activate est que la commande workon fonctionnera dans n’importe quel répertoire ; vous n’avez pas besoin de vous rappeler dans quel répertoire l’environnement virtuel particulier de votre projet est stocké.

Répertoires de projets #

Vous pouvez même spécifier un répertoire de projet lors de la création de l’environnement virtuel avec l’option -a ou plus tard avec la commande setvirtualenvproject.

mkvirtualenv -a /path/to/my-project my-project

ou

workon my-project
cd /path/to/my-project
setvirtualenvproject

La définition d’un projet entraînera le basculement automatique de la commande workon vers le projet et l’activation de la commande cdproject qui vous permet de passer au répertoire du projet.

Pour voir une liste de tous les virtualenvs gérés par virtualenvwrapper, utilisez lsvirtualenv.

Pour supprimer un virtualenv, utilisez rmvirtualenv :

rmvirtualenv my-project

Chaque virtualenv géré par virtualenvwrapper comprend 4 scripts bash vides : preactivate, postactivate, predeactivate et postdeactivate. Ceux-ci servent de crochets pour exécuter des commandes bash à certains moments du cycle de vie de virtualenv ; par exemple, toutes les commandes du script postactivate s’exécuteront juste après l’activation de virtualenv. Ce serait un bon endroit pour définir des variables d’environnement spéciales, des alias ou toute autre chose pertinente. Les 4 scripts sont situés sous .virtualenvs/<virtualenv_name>/bin/.

Pour plus de détails, lisez la [documentation virtualenvwrapper][1].

[1] : https://virtualenvwrapper.readthedocs.io/

Découvrir quel environnement virtuel vous utilisez #

Si vous utilisez l’invite bash par défaut sous Linux, vous devriez voir le nom de l’environnement virtuel au début de votre invite.

(my-project-env) user@hostname:~$ which python
/home/user/my-project-env/bin/python

Utilisation de virtualenv avec une coquille de poisson #

La coquille de poisson est plus conviviale, mais vous pourriez rencontrer des problèmes lors de l’utilisation avec virtualenv ou virtualenvwrapper. Alternativement, virtualfish existe pour le sauvetage. Suivez simplement la séquence ci-dessous pour commencer à utiliser Fish shell avec virtualenv.

  • Installer virtualfish dans l’espace global

      sudo pip install virtualfish

  • Charger le module python virtualfish lors du démarrage de fish shell

      $ echo "eval (python -m virtualfish)" > ~/.config/fish/config.fish

  • Modifiez cette fonction fish_prompt par $ funced fish_prompt --editor vim et ajoutez les lignes ci-dessous et fermez l’éditeur vim

      if set -q VIRTUAL_ENV
      echo -n -s (set_color -b blue white) "(" (basename "$VIRTUAL_ENV") ")" (set_color normal) " "
  end

    Note: If you are unfamiliar with vim, simply supply your favorite editor like this $ funced fish_prompt --editor nano or $ funced fish_prompt --editor gedit

  • Enregistrer les modifications en utilisant funcsave

      funcsave fish_prompt

  • Pour créer un nouvel environnement virtuel, utilisez vf new

      vf new my_new_env # Make sure $HOME/.virtualenv exists

  • Si vous voulez créer un nouvel environnement python3, spécifiez-le via le drapeau -p

      vf new -p python3 my_new_env

  • Pour basculer entre les environnements virtuels, utilisez vf deactivate & vf activate another_env

Liens officiels :

  • https://github.com/adambrenecki/virtualfish
  • http://virtualfish.readthedocs.io/en/latest/

Vérification de l’exécution dans un environnement virtuel #

Parfois, l’invite du shell n’affiche pas le nom de l’environnement virtuel et vous voulez être sûr si vous êtes dans un environnement virtuel ou non.

Exécutez l’interpréteur Python et essayez :

import sys
sys.prefix
sys.real_prefix

  • En dehors d’un environnement virtuel, sys.prefix pointera vers l’installation du système python et sys.real_prefix n’est pas défini.

  • Dans un environnement virtuel, sys.prefix pointera vers l’installation python de l’environnement virtuel et sys.real_prefix pointera vers l’installation python du système.

Pour les environnements virtuels créés à l’aide de la bibliothèque standard module venv, il n’y a pas de “sys.real_prefix”. Au lieu de cela, vérifiez si sys.base_prefix est identique à sys.prefix.
