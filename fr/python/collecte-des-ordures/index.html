À la base, le ramasse-miettes de Python (à partir de la version 3.5) est une simple implémentation de comptage de références. Chaque fois que vous faites une référence à un objet (par exemple, a = myobject), le nombre de références sur cet objet (myobject) est incrémenté. Chaque fois qu’une référence est supprimée, le nombre de références est décrémenté, et une fois que le nombre de références atteint ‘0’, nous savons que rien ne contient de référence à cet objet et nous pouvons le désallouer !

Un malentendu courant sur le fonctionnement de la gestion de la mémoire Python est que le mot-clé del libère la mémoire des objets. Ce n’est pas vrai. Ce qui se passe réellement, c’est que le mot-clé del décrémente simplement le refcount des objets, ce qui signifie que si vous l’appelez suffisamment de fois pour que le refcount atteigne zéro, l’objet peut être ramassé (même s’il existe encore des références à l’objet disponibles ailleurs dans votre code).

Python crée ou nettoie agressivement des objets la première fois qu’il en a besoin Si j’exécute l’affectation a = object(), la mémoire pour l’objet est allouée à ce moment-là (cpython réutilisera parfois certains types d’objets, par exemple des listes sous le capot, mais la plupart du temps, il ne conserve pas de pool d’objets libres et effectuera l’allocation lorsque vous en aurez besoin). De même, dès que le refcount est décrémenté à 0, GC le nettoie.

Collecte de déchets générationnelle #

Dans les années 1960, John McCarthy a découvert une faille fatale dans le refcounting du ramasse-miettes lorsqu’il a implémenté l’algorithme de refcounting utilisé par Lisp : que se passe-t-il si deux objets se réfèrent l’un à l’autre dans une référence cyclique ? Comment pouvez-vous jamais ramasser ces deux objets même s’il n’y a pas de références externes à eux s’ils se réfèrent toujours l’un à l’autre? Ce problème s’étend également à toute structure de données cyclique, telle qu’un tampon en anneau ou deux entrées consécutives quelconques dans une liste à double liaison. Python tente de résoudre ce problème en utilisant une variante légèrement intéressante d’un autre algorithme de récupération de place appelé Generational Garbage Collection.

Essentiellement, chaque fois que vous créez un objet en Python, il l’ajoute à la fin d’une liste doublement liée. À l’occasion, Python parcourt cette liste, vérifie à quels objets les objets de la liste se réfèrent également, et s’ils sont également dans la liste (nous verrons pourquoi ils pourraient ne pas être dans un instant), décrémente davantage leurs refcounts. À ce stade (en fait, il existe des heuristiques qui déterminent quand les choses sont déplacées, mais supposons que c’est après une seule collection pour garder les choses simples) tout ce qui a encore un refcount supérieur à 0 est promu vers une autre liste chaînée appelée “Génération 1” (c’est pourquoi tous les objets ne sont pas toujours dans la liste de génération 0) à laquelle cette boucle est appliquée moins souvent. C’est là qu’intervient le ramasse-miettes générationnel. Il existe 3 générations par défaut en Python (trois listes d’objets liées) : La première liste (génération 0) contient tous les nouveaux objets ; si un cycle GC se produit et que les objets ne sont pas collectés, ils sont déplacés vers la deuxième liste (génération 1), et si un cycle GC se produit sur la deuxième liste et qu’ils ne sont toujours pas collectés, ils sont déplacés vers la troisième liste (génération 2 ). La liste de troisième génération (appelée “génération 2”, puisque nous sommes à zéro indexation) est ramassée beaucoup moins souvent que les deux premières, l’idée étant que si votre objet a une longue durée de vie, il est moins susceptible d’être GCed, et peut-être jamais être GC pendant la durée de vie de votre application, il est donc inutile de perdre du temps à la vérifier à chaque exécution de GC. De plus, on observe que la plupart des objets sont ramassés assez rapidement. Désormais, on appellera ces “bons objets” puisqu’ils meurent jeunes. C’est ce qu’on appelle «l’hypothèse générationnelle faible» et a également été observée pour la première fois dans les années 60.

Un petit aparté : contrairement aux deux premières générations, la liste de longue durée de la troisième génération n’est pas une collecte d’ordures régulière. Il est vérifié lorsque le rapport entre les objets en attente à longue durée de vie (ceux qui figurent dans la liste de troisième génération, mais qui n’ont pas encore eu de cycle GC) et le nombre total d’objets à longue durée de vie dans la liste est supérieur à 25 %. C’est parce que la troisième liste est illimitée (les choses ne sont jamais déplacées vers une autre liste, donc elles ne disparaissent que lorsqu’elles sont réellement ramassées), ce qui signifie que pour les applications où vous créez beaucoup d’objets à longue durée de vie, les cycles GC sur la troisième liste peut être assez longue. En utilisant un ratio, nous obtenons “une performance linéaire amortie dans le nombre total d’objets” ; autrement dit, plus la liste est longue, plus la GC prend de temps, mais moins nous effectuons la GC (voici la [proposition originale de 2008](https://mail.python.org/pipermail/python-dev/2008-June/080579. html) pour cette heuristique par Martin von Löwis pour une lecture plus approfondie). L’acte d’effectuer un ramasse-miettes sur la liste de troisième génération ou « mature » est appelé « ramasse-miettes complet ».

Ainsi, le ramasse-miettes générationnel accélère énormément les choses en n’exigeant pas que nous analysions des objets qui ne sont pas susceptibles d’avoir besoin de GC tout le temps, mais comment cela nous aide-t-il à casser les références cycliques ? Probablement pas très bien, il s’avère. La fonction pour briser réellement ces cycles de référence commence comme ceci:

/* Break reference cycles by clearing the containers involved.  This is
 * tricky business as the lists can be changing and we don't know which
 * objects may be freed.  It is possible I screwed something up here.
 */
static void
delete_garbage(PyGC_Head *collectable, PyGC_Head *old)

La raison pour laquelle le ramasse-miettes générationnel aide à cela est que nous pouvons conserver la longueur de la liste comme un décompte séparé ; chaque fois que nous ajoutons un nouvel objet à la génération, nous incrémentons ce nombre, et chaque fois que nous déplaçons un objet vers une autre génération ou le désallouons, nous décrémentons le nombre. Théoriquement, à la fin d’un cycle GC, ce décompte (pour les deux premières générations de toute façon) devrait toujours être égal à 0. Si ce n’est pas le cas, tout ce qui reste dans la liste est une forme de référence circulaire et nous pouvons le supprimer. Cependant, il y a un autre problème ici : que se passe-t-il si les objets restants contiennent la méthode magique de Python __del__ ? __del__ est appelé chaque fois qu’un objet Python est détruit. Cependant, si deux objets dans une référence circulaire ont des méthodes __del__, nous ne pouvons pas être sûrs que la destruction de l’un ne cassera pas la méthode __del__ des autres. Pour un exemple artificiel, imaginez que nous écrivions ce qui suit :

class A(object):
    def __init__(self, b=None):
        self.b = b
 
    def __del__(self):
        print("We're deleting an instance of A containing:", self.b)
     
class B(object):
    def __init__(self, a=None):
        self.a = a
 
    def __del__(self):
        print("We're deleting an instance of B containing:", self.a)

et nous définissons une instance de A et une instance de B pour qu’elles pointent l’une vers l’autre, puis elles se retrouvent dans le même cycle de récupération de place ? Disons que nous en choisissons un au hasard et que nous libérons notre instance de A en premier ; La méthode __del__ de A sera appelée, elle imprimera, puis A sera libérée. Ensuite, nous arrivons à B, nous appelons sa méthode __del__, et oups ! Segfault ! A n’existe plus. Nous pourrions résoudre ce problème en appelant d’abord les méthodes __del__ de tout ce qui reste, puis en effectuant une autre passe pour tout désallouer, cependant, cela introduit un autre problème : que se passe-t-il si la méthode __del__ d’un objet enregistre une référence de l’autre objet qui est sur le point être GCed et a une référence à nous ailleurs? Nous avons toujours un cycle de référence, mais maintenant il n’est plus possible de GC ni l’un ni l’autre des objets, même s’ils ne sont plus utilisés. Notez que même si un objet ne fait pas partie d’une structure de données circulaire, il peut se réactiver dans sa propre méthode __del__ ; Python a un contrôle pour cela et arrêtera GCing si un refcount d’objets a augmenté après que sa méthode __del__ a été appelée.

CPython gère cela en collant ces objets non GC (tout ce qui a une forme de référence circulaire et une méthode __del__) sur une liste globale de déchets irrécupérables, puis en les laissant là pour l’éternité :

/* list of uncollectable objects */
static PyObject *garbage = NULL;

Réutilisation d’objets primitifs #

Une chose intéressante à noter qui peut aider à optimiser vos applications est que les primitives sont en fait également recomptées sous le capot. Jetons un coup d’œil aux chiffres; pour tous les entiers entre -5 et 256, Python réutilise toujours le même objet :

>>> import sys
>>> sys.getrefcount(1)
797
>>> a = 1
>>> b = 1
>>> sys.getrefcount(1)
799

Notez que le refcount augmente, ce qui signifie que a et b font référence au même objet sous-jacent lorsqu’ils font référence à la primitive 1. Cependant, pour des nombres plus importants, Python ne réutilise pas l’objet sous-jacent :

>>> a = 999999999
>>> sys.getrefcount(999999999)
3
>>> b = 999999999
>>> sys.getrefcount(999999999)
3

Étant donné que le refcount pour 999999999 ne change pas lors de son affectation à a et b, nous pouvons en déduire qu’ils font référence à deux objets sous-jacents différents, même s’ils se voient tous deux attribuer la même primitive.

Effets de la commande del #

Supprimer un nom de variable de la portée à l’aide de del v, ou supprimer un objet d’une collection à l’aide de del v[item] ou del[i:j], ou supprimer un attribut à l’aide de del v.name, ou tout autre moyen de supprimer les références à un objet, * ne * déclenche aucun appel de destructeur ni aucune libération de mémoire en soi. Les objets ne sont détruits que lorsque leur compteur de références atteint zéro.

>>> import gc
>>> gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def bar():
    return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> del t          # not destructed yet - another_t still refers to it
>>> del another_t  # final reference gone, object is destructed
Destructed

Comptage de références #

La grande majorité de la gestion de la mémoire Python est gérée avec le comptage de références.

Chaque fois qu’un objet est référencé (par exemple, affecté à une variable), son nombre de références est automatiquement augmenté. Lorsqu’il est déréférencé (par exemple, une variable sort de la portée), son nombre de références est automatiquement diminué.

Lorsque le compteur de références atteint zéro, l’objet est immédiatement détruit et la mémoire est immédiatement libérée. Ainsi, dans la majorité des cas, le ramasse-miettes n’est même pas nécessaire.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def foo():
        Track()
        # destructed immediately since no longer has any references
        print("---")
        t = Track()
        # variable is referenced, so it's not destructed yet
        print("---")
        # variable is destructed when function exits
>>> foo()
Initialized
Destructed
---
Initialized
---
Destructed

Pour démontrer davantage le concept de références :

>>> def bar():
        return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> t = None          # not destructed yet - another_t still refers to it
>>> another_t = None  # final reference gone, object is destructed
Destructed

Garbage Collector pour les cycles de référence #

La seule fois où le ramasse-miettes est nécessaire, c’est si vous avez un cycle de référence. L’exemple simple d’un cycle de référence est celui dans lequel A fait référence à B et B fait référence à A, tandis que rien d’autre ne fait référence à A ou B. Ni A ni B ne sont accessibles de n’importe où dans le programme, ils peuvent donc être détruits en toute sécurité, pourtant, leurs comptes de référence sont 1 et ils ne peuvent donc pas être libérés par l’algorithme de comptage de références seul.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> A = Track()
Initialized
>>> B = Track()
Initialized
>>> A.other = B
>>> B.other = A
>>> del A; del B  # objects are not destructed due to reference cycle
>>> gc.collect()  # trigger collection
Destructed
Destructed
4

Un cycle de référence peut être arbitrairement long. Si A pointe vers B pointe vers C pointe vers … pointe vers Z qui pointe vers A, alors aucun A à Z ne sera collecté, jusqu’à la phase de récupération de place :

>>> objs = [Track() for _ in range(10)]
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
>>> for i in range(len(objs)-1):
...     objs[i].other = objs[i + 1]
...
>>> objs[-1].other = objs[0]  # complete the cycle
>>> del objs                  # no one can refer to objs now - still not destructed
>>> gc.collect()
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
20

Affichage du refcount d’un objet #

>>> import sys
>>> a = object()
>>> sys.getrefcount(a)
2
>>> b = a
>>> sys.getrefcount(a)
3
>>> del b
>>> sys.getrefcount(a)
2

Désallocation forcée d’objets #

Vous pouvez forcer la libération d’objets même si leur refcount n’est pas 0 dans Python 2 et 3.

Les deux versions utilisent le module ctypes pour ce faire.

ATTENTION : faire cela rendra votre environnement Python instable et susceptible de planter sans trace ! L’utilisation de cette méthode pourrait également introduire des problèmes de sécurité (assez peu probable) Libérez uniquement les objets que vous êtes sûr de ne plus jamais référencer. Déjà.

import ctypes
deallocated = 12345
ctypes.pythonapi._Py_Dealloc(ctypes.py_object(deallocated))

import ctypes, sys
deallocated = 12345
(ctypes.c_char * sys.getsizeof(deallocated)).from_address(id(deallocated))[:4] = '\x00' * 4

Après l’exécution, toute référence à l’objet désormais désalloué entraînera un comportement indéfini ou un plantage de Python - sans trace. Il y avait probablement une raison pour laquelle le ramasse-miettes n’a pas supprimé cet objet…

Si vous désallouez None, vous obtenez un message spécial - Fatal Python error: deallocation None avant de planter.

Gérer la collecte des ordures #

Il existe deux approches pour influencer le moment où un nettoyage de la mémoire est effectué. Ils influencent la fréquence d’exécution du processus automatique et l’autre déclenche manuellement un nettoyage.

Le ramasse-miettes peut être manipulé en réglant les seuils de collecte qui affectent la fréquence à laquelle le ramasse-miettes s’exécute. Python utilise un système de gestion de mémoire basé sur la génération. Les nouveaux objets sont enregistrés dans la dernière génération - génération0 et avec chaque collection survécue, les objets sont promus aux générations plus anciennes. Après avoir atteint la dernière génération - génération2, ils ne sont plus promus.

Les seuils peuvent être modifiés à l’aide de l’extrait de code suivant :

import gc
gc.set_threshold(1000, 100, 10) # Values are just for demonstration purpose

Le premier argument représente le seuil de collecte generation0. Chaque fois que le nombre d’allocations dépasse le nombre de désallocations de 1 000, le ramasse-miettes est appelé.

Les anciennes générations ne sont pas nettoyées à chaque passage pour optimiser le processus. Les deuxième et troisième arguments sont facultatifs et contrôlent la fréquence de nettoyage des anciennes générations. Si génération0 a été traitée 100 fois sans nettoyer génération1, alors génération1 sera traitée. De même, les objets de génération2 ne seront traités que lorsque ceux de génération1 ont été nettoyés 10 fois sans toucher à génération2.

Un cas dans lequel la définition manuelle des seuils est bénéfique est lorsque le programme alloue beaucoup de petits objets sans les désallouer, ce qui conduit à l’exécution trop fréquente du ramasse-miettes (chaque allocation d’objet ** generation0_threshold **). Même si le collecteur est assez rapide, lorsqu’il s’exécute sur un grand nombre d’objets, il pose un problème de performances. Quoi qu’il en soit, il n’y a pas de stratégie unique pour choisir les seuils et c’est un cas d’utilisation fiable.

Le déclenchement manuel d’une collection peut être effectué comme dans l’extrait de code suivant :

import gc
gc.collect()

Le ramasse-miettes est automatiquement déclenché en fonction du nombre d’allocations et de désallocations, et non en fonction de la mémoire consommée ou disponible. Par conséquent, lorsque vous travaillez avec de gros objets, la mémoire peut être épuisée avant le déclenchement du nettoyage automatisé. Cela constitue un bon cas d’utilisation pour appeler manuellement le ramasse-miettes.

Même si c’est possible, ce n’est pas une pratique encouragée. Éviter les fuites de mémoire est la meilleure option. Quoi qu’il en soit, dans les grands projets, la détection de la fuite de mémoire peut être une tâche délicate et le déclenchement manuel d’un ramasse-miettes peut être utilisé comme solution rapide jusqu’à un débogage ultérieur.

Pour les programmes de longue durée, la récupération de place peut être déclenchée sur une base temporelle ou sur la base d’un événement. Un exemple pour le premier est un serveur Web qui déclenche une collecte après un nombre fixe de requêtes. Pour ce dernier, un serveur Web qui déclenche un ramasse-miettes lorsqu’un certain type de requête est reçu.

N’attendez pas la collecte des ordures pour nettoyer #

Le fait que la récupération de place soit nettoyée ne signifie pas que vous devez attendre que le cycle de récupération de place soit nettoyé.

En particulier, vous ne devez pas attendre que la récupération de place ferme les descripteurs de fichiers, les connexions à la base de données et les connexions réseau ouvertes.

par exemple:

Dans le code suivant, vous supposez que le fichier sera fermé lors du prochain cycle de récupération de place, si f était la dernière référence au fichier.

>>> f = open("test.txt")
>>> del f

Une façon plus explicite de nettoyer est d’appeler f.close(). Vous pouvez le faire de manière encore plus élégante, c’est-à-dire en utilisant l’instruction with, également connue sous le nom de context manager :

>>> with open("test.txt") as f:
...     pass
...     # do something with f
>>> #now the f object still exists, but it is closed

L’instruction with vous permet d’indenter votre code sous le fichier ouvert. Cela rend explicite et plus facile de voir combien de temps un fichier est maintenu ouvert. Il ferme également toujours un fichier, même si une exception est déclenchée dans le bloc while.
