Syntaxe #

  • # Ceci est un commentaire d’une seule ligne
  • print("")  # Ceci est un commentaire en ligne
  • “”"
    C’est
    un commentaire sur plusieurs lignes
    """

Les développeurs doivent suivre les directives PEP257 - Docstring Conventions. Dans certains cas, des guides de style (tels que ceux de Google Style Guide) ou de la documentation rendant des tiers (tels que Sphinx ) peut détailler des conventions supplémentaires pour les docstrings.

Accéder par programmation aux docstrings #

Les docstrings sont - contrairement aux commentaires réguliers - stockés en tant qu’attribut de la fonction qu’ils documentent, ce qui signifie que vous pouvez y accéder par programmation.

Un exemple de fonction #

def func():
    """This is a function that does nothing at all"""
    return

La docstring est accessible à l’aide de l’attribut __doc__ :

print(func.__doc__)

C’est une fonction qui ne fait rien du tout

help(func)

Aide sur la fonction func du module __main__ :

fonction()

     C’est une fonction qui ne fait rien du tout

Autre exemple de fonction #

function.__doc__ est simplement la docstring réelle sous forme de chaîne, tandis que la fonction help fournit des informations générales sur une fonction, y compris la docstring. Voici un exemple plus utile :

def greet(name, greeting="Hello"):
    """Print a greeting to the user `name`

    Optional parameter `greeting` can change what they're greeted with."""
    
    print("{} {}".format(greeting, name))

help(greet)

Aide sur la fonction greet dans le module __main__ :

saluer(nom, salutation='Bonjour')

    Imprimer un message d’accueil à l’utilisateur name
    Le paramètre facultatif “salutation” peut modifier la manière dont ils sont accueillis.

Avantages des docstrings par rapport aux commentaires réguliers #

Le simple fait de ne pas mettre de docstring ou un commentaire régulier dans une fonction la rend beaucoup moins utile.

def greet(name, greeting="Hello"):
    # Print a greeting to the user `name`
    # Optional parameter `greeting` can change what they're greeted with.
    
    print("{} {}".format(greeting, name))

print(greet.__doc__)

Aucun

help(greet)

Aide sur la fonction greet dans le module main :

saluer(nom, salutation='Bonjour')

Commentaires sur une seule ligne, en ligne et multilignes #

Les commentaires sont utilisés pour expliquer le code lorsque le code de base lui-même n’est pas clair.

Python ignore les commentaires et n’y exécutera donc pas de code, ni ne générera d’erreurs de syntaxe pour les phrases en anglais simple.

Les commentaires sur une seule ligne commencent par le caractère dièse (#) et se terminent par la fin de ligne.

  • Commentaire sur une seule ligne :

    This is a single line comment in Python #

  • Commentaire en ligne :

    print(“Hello World”) # This line prints “Hello World”

  • Les commentaires s’étendant sur plusieurs lignes ont """ ou ''' à chaque extrémité. C’est la même chose qu’une chaîne multiligne, mais ils peuvent être utilisés comme commentaires :

    """ This type of comment spans multiple lines. These are mostly used for documentation of functions, classes and modules. """

Écrire une documentation à l’aide de docstrings #

Un [docstring][1] est un [commentaire multi-ligne][2] utilisé pour documenter des modules, des classes, des fonctions et des méthodes. Il doit s’agir de la première instruction du composant qu’il décrit.

def hello(name):
    """Greet someone.

    Print a greeting ("Hello") for the person with the given name.
    """

    print("Hello "+name)

class Greeter:
    """An object used to greet people.

    It contains multiple greeting functions for several languages
    and times of the  day.
    """

La valeur de la docstring peut être [accessible dans le programme][3] et est - par exemple - utilisée par la commande help.

Conventions de syntaxe #

PEP 257 #

[PEP 257][1] définit une norme de syntaxe pour les commentaires de docstring. Il permet essentiellement deux types :

  • Docstrings en une ligne :

Selon la PEP 257, ils doivent être utilisés avec des fonctions courtes et simples. Tout est placé sur une seule ligne, par exemple :

def hello():
    """Say hello to your friends."""
    print("Hello my friends!")

La docstring doit se terminer par un point, le verbe doit être à l’impératif.

  • Docstrings multi-lignes :

La docstring multiligne doit être utilisée pour les fonctions, modules ou classes plus longs et plus complexes.

def hello(name, language="en"):
    """Say hello to a person.

    Arguments:
    name: the name of the person
    language: the language in which the person should be greeted
    """

    print(greeting[language]+" "+name)

Ils commencent par un court résumé (équivalent au contenu d’une docstring d’une ligne) qui peut être sur la même ligne que les guillemets ou sur la ligne suivante, donner des détails supplémentaires et lister les paramètres et les valeurs de retour.

Remarque PEP 257 définit [quelles informations doivent être données] [4] dans une docstring, elle ne définit pas dans quel format elles doivent être données. C’est la raison pour laquelle d’autres parties et outils d’analyse de documentation ont spécifié leurs propres normes de documentation, dont certaines sont énumérées ci-dessous et dans [cette question][9].

Sphinx #

[Sphinx][5] est un outil permettant de générer une documentation HTML pour les projets Python basés sur des docstrings. Son langage de balisage utilisé est [reStructuredText][6]. Ils définissent leurs propres normes de documentation, pythonhosted.org en héberge une [très bonne description][7]. Le format Sphinx est par exemple utilisé par [pyCharm IDE][11].

Une fonction serait documentée comme ceci en utilisant le format Sphinx/reStructuredText :

def hello(name, language="en"):
    """Say hello to a person.

    :param name: the name of the person
    :type name: str
    :param language: the language in which the person should be greeted
    :type language: str
    :return: a number
    :rtype: int
    """

    print(greeting[language]+" "+name)
    return 4

Guide de style Google Python #

Google a publié [Google Python Style Guide][8] qui définit les conventions de codage pour Python, y compris les commentaires de la documentation. Par rapport au Sphinx/reST, beaucoup de gens disent que la documentation selon les directives de Google est mieux lisible par l’homme.

La [page pythonhosted.org mentionnée ci-dessus][7] fournit également quelques exemples de bonne documentation selon le Google Style Guide.

À l’aide du plug-in [Napoleon] [10], Sphinx peut également analyser la documentation au format conforme au Google Style Guide.

Une fonction serait documentée comme celle-ci en utilisant le format Google Style Guide :

def hello(name, language="en"):
    """Say hello to a person.

    Args:
        name: the name of the person as string
        language: the language code string

    Returns:
        A number.
    """

    print(greeting[language]+" "+name)
    return 4
