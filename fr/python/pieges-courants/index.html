Python est un langage censé être clair et lisible sans aucune ambiguïté ni comportement inattendu. Malheureusement, ces objectifs ne sont pas réalisables dans tous les cas, et c’est pourquoi Python a quelques cas particuliers où il pourrait faire quelque chose de différent de ce à quoi vous vous attendiez.

Cette section vous montrera certains problèmes que vous pourriez rencontrer lors de l’écriture de code Python.

Multiplication de liste et références communes #

Prenons le cas de la création d’une structure de liste imbriquée en multipliant :

li = [[]] * 3
print(li)
# Out: [[], [], []]

À première vue, nous pourrions penser que nous avons une liste contenant 3 listes imbriquées différentes. Essayons d’ajouter ‘1’ au premier :

li[0].append(1)
print(li)
# Out: [[1], [1], [1]]

‘1’ a été ajouté à toutes les listes dans ’li’.

La raison en est que [[]] * 3 ne crée pas une liste de 3 lists différentes. Au lieu de cela, il crée une list contenant 3 références au même objet list. Ainsi, lorsque nous ajoutons à li[0], le changement est visible dans tous les sous-éléments de li. Cela équivaut à :

li = []
element = [[]]
li = element + element + element
print(li)
# Out: [[], [], []]
element.append(1)
print(li)
# Out: [[1], [1], [1]]

Cela peut être corroboré davantage si nous imprimons les adresses mémoire de la “liste” contenue en utilisant “id” :

li = [[]] * 3
print([id(inner_list) for inner_list in li])
# Out: [6830760, 6830760, 6830760]

La solution consiste à créer les listes internes avec une boucle :

li = [[] for _ in range(3)]

Au lieu de créer une seule “liste” puis d’y faire 3 références, nous créons maintenant 3 listes distinctes différentes. Ceci, encore une fois, peut être vérifié en utilisant la fonction id :

print([id(inner_list) for inner_list in li])
# Out: [6331048, 6331528, 6331488]

Vous pouvez également le faire. Cela provoque la création d’une nouvelle liste vide dans chaque appel “ajouter”.

>>> li = []
>>> li.append([])
>>> li.append([])
>>> li.append([])
>>> for k in li: print(id(k))
... 
4315469256
4315564552
4315564808

N’utilisez pas index pour parcourir une séquence.

Ne le faites pas:

for i in range(len(tab)):
    print(tab[i])

Fais:

for elem in tab:
    print(elem)

for automatisera la plupart des opérations d’itération pour vous.

Utilisez enumerate si vous avez vraiment besoin à la fois de l’index et de l’élément.

for i, elem in enumerate(tab):
     print((i, elem))

Soyez prudent lorsque vous utilisez “==” pour vérifier Vrai ou Faux

if (var == True):
    # this will execute if var is True or 1, 1.0, 1L

if (var != True):
    # this will execute if var is neither True nor 1

if (var == False):
    # this will execute if var is False or 0 (or 0.0, 0L, 0j)

if (var == None):
    # only execute if var is None

if var:
    # execute if var is a non-empty string/list/dictionary/tuple, non-0, etc

if not var:
    # execute if var is "", {}, [], (), 0, None, etc.

if var is True:
    # only execute if var is boolean True, not 1

if var is False:
    # only execute if var is boolean False, not 0

if var is None:
    # same as var == None

Ne cochez pas si vous le pouvez, faites-le simplement et gérez l’erreur

Les pythonistes disent généralement “Il est plus facile de demander pardon que permission”.

Ne le faites pas:

if os.path.isfile(file_path):
    file = open(file_path)
else:
    # do something

Fais:

try:
    file = open(file_path)
except OSError as e:
    # do something

Ou encore mieux avec Python 2.6+ :

with open(file_path) as file:

C’est beaucoup mieux parce que c’est beaucoup plus générique. Vous pouvez appliquer try/except à presque n’importe quoi. Vous n’avez pas besoin de vous soucier de ce qu’il faut faire pour l’empêcher, mais seulement de l’erreur que vous risquez.

Ne pas vérifier par rapport au type

Python est typé dynamiquement, donc vérifier le type vous fait perdre de la flexibilité. Au lieu de cela, utilisez [duck typing][1] en vérifiant le comportement. Si vous attendez une chaîne dans une fonction, utilisez str() pour convertir n’importe quel objet en chaîne. Si vous attendez une liste, utilisez list() pour convertir tout itérable en liste.

Ne le faites pas:

def foo(name):
    if isinstance(name, str):
        print(name.lower())

def bar(listing):
    if isinstance(listing, list):
        listing.extend((1, 2, 3))
        return ", ".join(listing)

Fais:

def foo(name) :
    print(str(name).lower())

def bar(listing) :
    l = list(listing)
    l.extend((1, 2, 3))
    return ", ".join(l)

En utilisant la dernière méthode, foo acceptera n’importe quel objet. bar acceptera des chaînes, des tuples, des ensembles, des listes et bien plus encore. SEC bon marché.

Ne mélangez pas les espaces et les tabulations

Utilisez object comme premier parent

C’est délicat, mais cela vous mordra au fur et à mesure que votre programme se développera. Il existe d’anciennes et de nouvelles classes dans Python 2.x. Les anciens sont, eh bien, vieux. Ils manquent de certaines fonctionnalités et peuvent avoir un comportement gênant avec l’héritage. Pour être utilisable, n’importe laquelle de vos classes doit être du “nouveau style”. Pour ce faire, faites-le hériter de object.

Ne le faites pas:

class Father:
    pass

class Child(Father):
    pass

Fais:

class Father(object):
    pass


class Child(Father):
    pass

Dans Python 3.x, toutes les classes ont un nouveau style, vous n’avez donc pas besoin de le faire.

N’initialisez pas les attributs de classe en dehors de la méthode init

Les personnes venant d’autres langages trouvent cela tentant car c’est ce que vous faites en Java ou en PHP. Vous écrivez le nom de la classe, puis listez vos attributs et leur donnez une valeur par défaut. Cela semble fonctionner en Python, cependant, cela ne fonctionne pas comme vous le pensez. Cela configurera les attributs de classe (attributs statiques), puis lorsque vous essaierez d’obtenir l’attribut d’objet, il vous donnera sa valeur à moins qu’il ne soit vide. Dans ce cas, il renverra les attributs de la classe. Cela implique deux grands risques :

  • Si l’attribut class est modifié, la valeur initiale est modifiée.

  • Si vous définissez un objet mutable comme valeur par défaut, vous obtiendrez le même objet partagé entre les instances.

À ne pas faire (sauf si vous voulez du statique) :

class Car(object):
    color = "red"
    wheels = [Wheel(), Wheel(), Wheel(), Wheel()]

Fais :

class Car(object):
    def __init__(self):
        self.color = "red"
        self.wheels = [Wheel(), Wheel(), Wheel(), Wheel()]

[1] : https://stackoverflow.com/questions/4205130/what-is-duck-typing

Argument mutable par défaut #

def foo(li=[]):
    li.append(1)
    print(li)

foo([2])
# Out: [2, 1]
foo([3])
# Out: [3, 1]

Ce code se comporte comme prévu, mais que se passe-t-il si nous ne transmettons pas d’argument ?

foo()
# Out: [1] As expected...

foo()
# Out: [1, 1]  Not as expected...

En effet, les arguments par défaut des fonctions et des méthodes sont évalués au moment de la définition plutôt qu’au moment de l’exécution. Nous n’avons donc qu’une seule instance de la liste “li”.

La façon de contourner ce problème consiste à n’utiliser que des types immuables pour les arguments par défaut :

def foo(li=None):
    if not li:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

foo()
# Out: [1]

Bien qu’il s’agisse d’une amélioration et bien que if not li évalue correctement à False, de nombreux autres objets le font également, comme les séquences de longueur nulle. Les exemples d’arguments suivants peuvent entraîner des résultats inattendus :

x = []
foo(li=x)
# Out: [1]

foo(li="")
# Out: [1]

foo(li=0) 
# Out: [1]

L’approche idiomatique consiste à vérifier directement l’argument par rapport à l’objet “None”:

def foo(li=None):
    if li is None:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

Modification de la séquence sur laquelle vous itérez #

Une boucle for itère sur une séquence, donc la modification de cette séquence à l’intérieur de la boucle peut entraîner des résultats inattendus (en particulier lors de l’ajout ou de la suppression d’éléments) :

alist = [0, 1, 2]
for index, value in enumerate(alist):
    alist.pop(index)
print(alist)
# Out: [1]

Remarque : list.pop() est utilisé pour supprimer des éléments de la liste.

Le deuxième élément n’a pas été supprimé car l’itération parcourt les indices dans l’ordre. La boucle ci-dessus itère deux fois, avec les résultats suivants :

# Iteration #1
index = 0
alist = [0, 1, 2]
alist.pop(0) # removes '0'

# Iteration #2
index = 1
alist = [1, 2]
alist.pop(1) # removes '2'

# loop terminates, but alist is not empty:
alist = [1]

Ce problème se pose parce que les indices changent tout en itérant dans le sens d’un indice croissant. Pour éviter ce problème, vous pouvez ** parcourir la boucle en sens inverse ** :

alist = [1,2,3,4,5,6,7]
for index, item in reversed(list(enumerate(alist))):
    # delete all even items
    if item % 2 == 0:
        alist.pop(index)
print(alist)
# Out: [1, 3, 5, 7]

En parcourant la boucle en commençant à la fin, au fur et à mesure que des éléments sont supprimés (ou ajoutés), cela n’affecte pas les indices des éléments plus tôt dans la liste. Cet exemple supprimera donc correctement tous les éléments pairs de alist.

Un problème similaire survient lors de l’insertion ou de l’ajout d’éléments dans une liste sur laquelle vous parcourez, ce qui peut entraîner une boucle infinie :

alist = [0, 1, 2]
for index, value in enumerate(alist):
    # break to avoid infinite loop:
    if index == 20:     
        break           
    alist.insert(index, 'a')
print(alist)
# Out (abbreviated): ['a', 'a', ..., 'a', 'a',  0,   1,   2]

Sans la condition break, la boucle insérerait 'a' tant que l’ordinateur ne manque pas de mémoire et que le programme est autorisé à continuer. Dans une telle situation, il est généralement préférable de créer une nouvelle liste et d’ajouter des éléments à la nouvelle liste au fur et à mesure que vous parcourez la liste d’origine.

Lorsque vous utilisez une boucle for, vous ne pouvez pas modifier les éléments de la liste avec la variable d’espace réservé :

alist = [1,2,3,4]
for item in alist:
    if item % 2 == 0:
        item = 'even'
print(alist)
# Out: [1,2,3,4]

Dans l’exemple ci-dessus, la modification de item ne change rien à la liste d’origine. Vous devez utiliser l’index de liste (alist[2]), et enumerate() fonctionne bien pour cela :

alist = [1,2,3,4]
for index, item in enumerate(alist):
    if item % 2 == 0:
        alist[index] = 'even'
print(alist)
# Out: [1, 'even', 3, 'even']

Une boucle while peut être un meilleur choix dans certains cas :

Si vous souhaitez supprimer tous les éléments de la liste :

zlist = [0, 1, 2]
while zlist:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
#      2
# After: zlist = []

Bien que la simple réinitialisation de zlist obtienne le même résultat ;

zlist = []

L’exemple ci-dessus peut également être combiné avec len() pour s’arrêter après un certain point, ou pour supprimer tous les éléments sauf x dans la liste :

zlist = [0, 1, 2]
x = 1
while len(zlist) > x:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
# After: zlist = [2]

Ou pour parcourir une liste tout en supprimant des éléments qui remplissent une certaine condition (dans ce cas, en supprimant tous les éléments pairs) :

zlist = [1,2,3,4,5]
i = 0
while i < len(zlist):
    if zlist[i] % 2 == 0:
        zlist.pop(i)
    else:
        i += 1
print(zlist)
# Out: [1, 3, 5]

Notez que vous n’incrémentez pas i après avoir supprimé un élément. En supprimant l’élément à zlist[i], l’index de l’élément suivant a diminué de un, donc en vérifiant zlist[i] avec la même valeur pour i à la prochaine itération, vous vérifierez correctement l’élément suivant dans la liste.

Une façon contraire de penser à supprimer les éléments indésirables d’une liste consiste à ajouter les éléments souhaités à une nouvelle liste. L’exemple suivant est une alternative au dernier exemple de boucle “while” :

zlist = [1,2,3,4,5]

z_temp = []
for item in zlist:
    if item % 2 != 0:
        z_temp.append(item)
zlist = z_temp
print(zlist)
# Out: [1, 3, 5]

Ici, nous canalisons les résultats souhaités dans une nouvelle liste. On peut alors éventuellement réaffecter la liste temporaire à la variable d’origine.

Avec cette tendance de pensée, vous pouvez invoquer l’une des fonctionnalités les plus élégantes et les plus puissantes de Python, list comprehensions, qui élimine les listes temporaires et s’écarte de l’idéologie de mutation de liste/index sur place précédemment discutée.

zlist = [1,2,3,4,5]
[item for item in zlist if item % 2 != 0]
# Out: [1, 3, 5]

Identité de l’entier et de la chaîne #

Python utilise la mise en cache interne pour une plage d’entiers afin de réduire les frais généraux inutiles liés à leur création répétée.

En effet, cela peut conduire à un comportement déroutant lors de la comparaison d’identités entières :

>>> -8 is (-7 - 1)
False
>>> -3 is (-2 - 1)
True

et, en utilisant un autre exemple :

>>> (255 + 1) is (255 + 1)
True
>>> (256 + 1) is (256 + 1)
False

Attends quoi?

Nous pouvons voir que l’opération d’identité is donne True pour certains entiers (-3, 256) mais pas pour d’autres (-8, 257).

Pour être plus précis, les entiers compris dans la plage [-5, 256] sont mis en cache en interne lors du démarrage de l’interpréteur et ne sont créés qu’une seule fois. En tant que tels, ils sont identiques et la comparaison de leurs identités avec “is” donne “True” ; les entiers en dehors de cette plage sont (généralement) créés à la volée et leurs identités sont comparées à “False”.

Il s’agit d’un écueil courant car il s’agit d’une plage courante pour les tests, mais assez souvent, le code échoue dans le processus de mise en scène ultérieur (ou pire - la production) sans raison apparente après avoir parfaitement fonctionné en développement.

La solution consiste à toujours comparer les valeurs en utilisant l’opérateur d’égalité (==) et pas l’opérateur d’identité (is).

Python conserve également les références aux chaînes couramment utilisées et peut entraîner un comportement tout aussi déroutant lors de la comparaison des identités (c’est-à-dire en utilisant is) des chaînes.

>>> 'python' is 'py' + 'thon'
True

La chaîne 'python' est couramment utilisée, donc Python a un objet que toutes les références à la chaîne 'python' utilisent.

Pour les chaînes peu communes, la comparaison d’identité échoue même lorsque les chaînes sont égales.

>>> 'this is not a common string' is 'this is not' + ' a common string'
False
>>> 'this is not a common string' == 'this is not' + ' a common string'
True

Ainsi, tout comme la règle pour les nombres entiers, ** comparez toujours les valeurs de chaîne en utilisant l’opérateur d’égalité ** (==) et pas l’opérateur d’identité (est).

Les dictionnaires ne sont pas ordonnés #

Vous pourriez vous attendre à ce qu’un dictionnaire Python soit trié par clés comme, par exemple, un std::map C++, mais ce n’est pas le cas :

myDict = {'first': 1, 'second': 2, 'third': 3}
print(myDict)
# Out: {'first': 1, 'second': 2, 'third': 3}

print([k for k in myDict])
# Out: ['second', 'third', 'first']

Python n’a pas de classe intégrée qui trie automatiquement ses éléments par clé.

Cependant, si le tri n’est pas indispensable et que vous souhaitez simplement que votre dictionnaire se souvienne de l’ordre d’insertion de ses paires clé/valeur, vous pouvez utiliser collections.OrderedDict :

from collections import OrderedDict

oDict = OrderedDict([('first', 1), ('second', 2), ('third', 3)])

print([k for k in oDict])
# Out: ['first', 'second', 'third']

Gardez à l’esprit que l’initialisation d’un OrderedDict avec un dictionnaire standard ne triera en aucune façon le dictionnaire pour vous. Tout ce que fait cette structure est de préserver l’ordre d’insertion des clés.

L’implémentation des dictionnaires a été modifiée dans Python 3.6 pour améliorer leur consommation de mémoire. Un effet secondaire de cette nouvelle implémentation est qu’elle préserve également l’ordre des arguments mot-clé passés à une fonction :

def func(**kw): print(kw.keys())

func(a=1, b=2, c=3, d=4, e=5) 
dict_keys(['a', 'b', 'c', 'd', 'e']) # expected order

Mise en garde : méfiez-vous du fait que ["l’aspect de préservation de l’ordre de cette nouvelle implémentation est considéré comme un détail d’implémentation et ne doit pas être invoqué"](https://docs.python.org/3.6/whatsnew/ 3.6.html#new-dict-implementation), car cela peut changer à l’avenir.

Fuite de variables dans les compréhensions de liste et les boucles for #

Considérez la compréhension de liste suivante

i = 0
a = [i for i in range(3)]
print(i) # Outputs 2

Cela se produit uniquement en Python 2 en raison du fait que la compréhension de la liste “fuit” la variable de contrôle de boucle dans la portée environnante ([source](http://python-history.blogspot.com/2010/06/from-list- comprehensions-to-generator.html)). Ce comportement peut entraîner des bogues difficiles à trouver et *** il a été corrigé dans Python 3 ***.

i = 0
a = [i for i in range(3)]
print(i) # Outputs 0

De même, les boucles for n’ont pas de portée privée pour leur variable d’itération

i = 0
for i in range(3):
    pass
print(i) # Outputs 2

Ce type de comportement se produit à la fois dans Python 2 et Python 3.

Pour éviter les problèmes de fuite de variables, utilisez de nouvelles variables dans les compréhensions de liste et les boucles for, le cas échéant.

Chaînage de ou opérateur #

Lors du test de l’une des nombreuses comparaisons d’égalité :

if a == 3 or b == 3 or c == 3:

il est tentant d’abréger

if a or b or c == 3: # Wrong

C’est faux; l’opérateur ou a [une priorité inférieure] [1] à ==, donc l’expression sera évaluée comme si (a) ou (b) ou (c == 3):. La bonne méthode consiste à vérifier explicitement toutes les conditions :

if a == 3 or b == 3 or c == 3:  # Right Way

Alternativement, la fonction intégrée any() peut être utilisée à la place des opérateurs chaînés or :

if any([a == 3, b == 3, c == 3]): # Right

Ou, pour le rendre plus efficace :

if any(x == 3 for x in (a, b, c)): # Right

Ou, pour faire plus court :

if 3 in (a, b, c): # Right

Ici, nous utilisons l’opérateur in pour tester si la valeur est présente dans un tuple contenant les valeurs que nous voulons comparer.

De même, il est incorrect d’écrire

if a == 1 or 2 or 3:

qui doit s’écrire comme

if a in (1, 2, 3):

[1] : https://docs.python.org/3/reference/expressions.html#operator-precedence “Règles de priorité”

Accéder aux attributs des littéraux int #

Vous avez peut-être entendu dire que tout en Python est un objet, même les littéraux. Cela signifie, par exemple, que “7” est également un objet, ce qui signifie qu’il a des attributs. Par exemple, l’un de ces attributs est le bit_length. Il renvoie le nombre de bits nécessaires pour représenter la valeur à laquelle il est appelé.

x = 7
x.bit_length()
# Out: 3

En voyant le code ci-dessus fonctionner, vous pourriez intuitivement penser que 7.bit_length() fonctionnerait également, seulement pour découvrir qu’il génère une SyntaxError. Pourquoi? car l’interpréteur doit faire la différence entre un accès à un attribut et un nombre flottant (par exemple 7.2 ou 7.bit_length()). Il ne peut pas, et c’est pourquoi une exception est levée.

Il existe plusieurs façons d’accéder aux attributs d’un int literals :

# parenthesis
(7).bit_length()
# a space
7 .bit_length()

L’utilisation de deux points (comme ceci 7..bit_length()) ne fonctionne pas dans ce cas, car cela crée un littéral float et les flottants n’ont pas la méthode bit_length().

Ce problème n’existe pas lors de l’accès aux attributs float littéraux’ car l’interpréteur est suffisamment “intelligent” pour savoir qu’un littéral float ne peut pas contenir deux ., par exemple :

7.2.as_integer_ratio()
# Out: (8106479329266893, 1125899906842624)

# script.py

Global Interpreter Lock (GIL) et blocage des threads #

Beaucoup ont été écrits about [Python’s](https://docs.python .org/3/glossary.html#term-global-interpreter-lock) GIL. Cela peut parfois prêter à confusion lorsqu’il s’agit d’applications multithread (à ne pas confondre avec multiprocessus).

Voici un exemple :

import math
from threading import Thread

def calc_fact(num):
    math.factorial(num)

num = 600000
t = Thread(target=calc_fact, daemon=True, args=[num])
print("About to calculate: {}!".format(num))
t.start()
print("Calculating...")
t.join()
print("Calculated")

Vous vous attendriez à voir Calculating... imprimé immédiatement après le démarrage du thread, nous voulions que le calcul se produise dans un nouveau thread après tout ! Mais en réalité, vous le voyez s’imprimer une fois le calcul terminé. En effet, le nouveau thread repose sur une fonction C (math.factorial) qui verrouille le GIL pendant son exécution.

Il y a plusieurs façons de contourner cela. La première consiste à implémenter votre fonction factorielle en Python natif. Cela permettra au thread principal de prendre le contrôle pendant que vous êtes à l’intérieur de votre boucle. L’inconvénient est que cette solution sera ** beaucoup ** plus lente, car nous n’utilisons plus la fonction C.

def calc_fact(num):
    """ A slow version of factorial in native Python """
    res = 1
    while num >= 1:
        res = res * num
        num -= 1
    return res

Vous pouvez également “dormir” pendant un certain temps avant de commencer votre exécution. Remarque : cela ne permettra pas réellement à votre programme d’interrompre le calcul qui se passe à l’intérieur de la fonction C, mais cela permettra à votre thread principal de continuer après le spawn, ce à quoi vous pouvez vous attendre.

def calc_fact(num):
    sleep(0.001)
    math.factorial(num)

Retour multiple #

La fonction xyz renvoie deux valeurs a et b :

def xyz():
  return a, b

Le code appelant xyz stocke le résultat dans une variable en supposant que xyz ne renvoie qu’une seule valeur :

t = xyz()

La valeur de t est en fait un tuple (a, b) donc toute action sur t en supposant qu’il ne s’agit pas d’un tuple peut échouer profondément dans le code avec une erreur inattendue à propos des tuples.

TypeError : le tuple de type ne définit pas … la méthode

La solution serait de faire :

a, b = xyz()

Les débutants auront du mal à trouver la raison de ce message en ne lisant que le message d’erreur du tuple !

my_var = 'bla';
