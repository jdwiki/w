## Introduction aux appareils de test Des tests plus compliqués doivent parfois être configurés avant d’exécuter le code que vous souhaitez tester. Il est possible de le faire dans la fonction de test elle-même, mais vous vous retrouvez alors avec de grandes fonctions de test qui en font tellement qu’il est difficile de dire où la configuration s’arrête et où le test commence. Vous pouvez également obtenir beaucoup de code de configuration en double entre vos différentes fonctions de test.

Notre fichier de code :

# projectroot/module/stuff.py
class Stuff(object):
    def prep(self):
        self.foo = 1
        self.bar = 2

Notre fichier test :

# projectroot/tests/test_stuff.py
import pytest
from module import stuff


def test_foo_updates():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Ce sont des exemples assez simples, mais si notre objet Stuff avait besoin de beaucoup plus de configuration, il deviendrait difficile à manier. Nous voyons qu’il y a du code dupliqué entre nos cas de test, alors refactorisons d’abord cela dans une fonction distincte.

# projectroot/tests/test_stuff.py
import pytest
from module import stuff


def get_prepped_stuff():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    return my_stuff


def test_foo_updates():
    my_stuff = get_prepped_stuff()
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates():
    my_stuff = get_prepped_stuff()
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Cela semble mieux mais nous avons toujours l’appel my_stuff = get_prepped_stuff() qui encombre nos fonctions de test.

## Appareils py.test à la rescousse !

Les appareils sont des versions beaucoup plus puissantes et flexibles des fonctions de configuration de test. Ils peuvent faire beaucoup plus que ce que nous exploitons ici, mais nous allons procéder une étape à la fois.

D’abord, nous changeons get_prepped_stuff en un appareil appelé prepped_stuff. Vous voulez nommer vos appareils avec des noms plutôt que des verbes en raison de la façon dont les appareils finiront par être utilisés dans les fonctions de test elles-mêmes plus tard. Le @pytest.fixture indique que cette fonction spécifique doit être gérée comme un appareil plutôt que comme une fonction normale.

@pytest.fixture
def prepped_stuff():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    return my_stuff

Nous devons maintenant mettre à jour les fonctions de test afin qu’elles utilisent le projecteur. Cela se fait en ajoutant un paramètre à leur définition qui correspond exactement au nom du projecteur. Lorsque py.test s’exécute, il exécute l’appareil avant d’exécuter le test, puis transmet la valeur de retour de l’appareil à la fonction de test via ce paramètre. (Notez que les appareils n’ont pas ** besoin ** de renvoyer une valeur ; ils peuvent faire d’autres choses de configuration à la place, comme appeler une ressource externe, organiser des choses sur le système de fichiers, mettre des valeurs dans une base de données, tout ce dont les tests ont besoin pour la configuration)

def test_foo_updates(prepped_stuff):
    my_stuff = prepped_stuff
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates(prepped_stuff):
    my_stuff = prepped_stuff
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Maintenant vous pouvez voir pourquoi nous l’avons nommé avec un nom. mais la ligne my_stuff = prepped_stuff est pratiquement inutile, alors utilisons simplement prepped_stuff directement à la place.

def test_foo_updates(prepped_stuff):
    assert 1 == prepped_stuff.foo
    prepped_stuff.foo = 30000
    assert prepped_stuff.foo == 30000


def test_bar_updates(prepped_stuff):
    assert 2 == prepped_stuff.bar
    prepped_stuff.bar = 42
    assert 42 == prepped_stuff.bar

Maintenant, nous utilisons des luminaires ! Nous pouvons aller plus loin en modifiant la portée de l’appareil (afin qu’il ne s’exécute qu’une seule fois par module de test ou session d’exécution de suite de tests au lieu d’une fois par fonction de test), en construisant des appareils qui utilisent d’autres appareils, en paramétrant l’appareil (pour que l’appareil et tous les tests utilisant cet appareil sont exécutés plusieurs fois, une fois pour chaque paramètre donné à l’appareil), les appareils qui lisent les valeurs du module qui les appelle… comme mentionné précédemment, les appareils ont beaucoup plus de puissance et de flexibilité qu’une fonction de configuration normale.

Nettoyage après les tests. #

Disons que notre code a grandi et que notre objet Stuff a maintenant besoin d’un nettoyage spécial.

# projectroot/module/stuff.py
class Stuff(object):
def prep(self):
    self.foo = 1
    self.bar = 2

def finish(self):
    self.foo = 0
    self.bar = 0

Nous pourrions ajouter du code pour appeler le nettoyage au bas de chaque fonction de test, mais les appareils fournissent une meilleure façon de le faire. Si vous ajoutez une fonction à l’appareil et que vous l’enregistrez en tant que finaliseur, le code de la fonction finaliseur sera appelé une fois le test effectué avec l’appareil. Si la portée de l’appareil est plus grande qu’une seule fonction (comme un module ou une session), le finaliseur sera exécuté après que tous les tests dans la portée soient terminés, donc après que le module soit terminé ou à la fin de toute la session de test en cours .

@pytest.fixture
def prepped_stuff(request):  # we need to pass in the request to use finalizers
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    def fin():  # finalizer function
        # do all the cleanup here
        my_stuff.finish()
    request.addfinalizer(fin)  # register fin() as a finalizer
    # you can do more setup here if you really want to
    return my_stuff

L’utilisation de la fonction de finaliseur dans une fonction peut être un peu difficile à comprendre à première vue, surtout lorsque vous avez des appareils plus compliqués. Vous pouvez à la place utiliser un fixation de rendement pour faire la même chose avec un flux d’exécution plus lisible par l’homme. La seule vraie différence est qu’au lieu d’utiliser return nous utilisons un yield à la partie de l’appareil où la configuration est effectuée et le contrôle doit aller à une fonction de test, puis ajouter tout le code de nettoyage après le yield. Nous le décorons également en tant que yield_fixture afin que py.test sache comment le gérer.

@pytest.yield_fixture
def prepped_stuff():  # it doesn't need request now!
    # do setup
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    # setup is done, pass control to the test functions
    yield my_stuff
    # do cleanup 
    my_stuff.finish()

Et cela conclut l’introduction aux appareils de test !

Pour plus d’informations, consultez la documentation officielle sur les appareils py.test et la [documentation officielle sur les appareils yield](http://doc.pytest. org/en/latest/yieldfixture.html)

Configuration de py.test #

py.test est l’une des nombreuses bibliothèques de test tierces disponibles pour Python. Il peut être installé en utilisant pip avec

pip install pytest

Le code à tester #

Disons que nous testons une fonction d’addition dans projectroot/module/code.py :

# projectroot/module/code.py
def add(a, b):
    return a + b

Le code de test #

Nous créons un fichier de test dans projectroot/tests/test_code.py. Le fichier doit commencer par test_ pour être reconnu comme fichier de test.

# projectroot/tests/test_code.py
from module import code


def test_add():
    assert code.add(1, 2) == 3
