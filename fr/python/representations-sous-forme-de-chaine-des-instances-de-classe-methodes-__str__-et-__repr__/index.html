Représentations sous forme de chaîne des instances de classe Méthodes __str__ et __repr__

  • Motivation
  • Les deux méthodes sont implémentées, style eval-round-trip repr()

Sur cette page

  • Motivation
  • Les deux méthodes sont implémentées, style eval-round-trip repr()

Une note sur l’implémentation des deux méthodes #

Lorsque les deux méthodes sont implémentées, il est assez courant d’avoir une méthode __str__ qui renvoie une représentation humaine (par exemple “Ace of Spaces”) et __repr__ renvoie un [eval](https://docs.python .org/3/library/functions.html#eval) représentation conviviale.

En fait, la documentation Python pour repr() note exactement ceci :

Pour de nombreux types, cette fonction tente de renvoyer une chaîne qui donnerait un objet avec la même valeur lorsqu’elle est passée à eval(), sinon la représentation est une chaîne entre crochets angulaires qui contient le nom du type de l’objet ainsi que des informations supplémentaires comprenant souvent le nom et l’adresse de l’objet.

Cela signifie que __str__ pourrait être implémenté pour renvoyer quelque chose comme “Ace of Spaces” comme indiqué précédemment, __repr__ pourrait être implémenté pour renvoyer à la place Card('Spades', 1)

Cette chaîne pourrait être renvoyée directement dans eval en quelque sorte “aller-retour”:

object -> string -> object

Un exemple d’implémentation d’une telle méthode pourrait être :

def __repr__(self):
    return "Card(%s, %d)" % (self.suit, self.pips)

Remarques #

[1] Cette sortie est spécifique à la mise en œuvre. La chaîne affichée provient de python.

[2] Vous avez peut-être déjà vu le résultat de cette division str()/repr() sans le savoir. Lorsque des chaînes contenant des caractères spéciaux tels que des barres obliques inverses sont converties en chaînes via str(), les barres obliques inverses apparaissent telles quelles (elles n’apparaissent qu’une seule fois). Lorsqu’ils sont convertis en chaînes via repr() (par exemple, en tant qu’éléments d’une liste en cours d’affichage), les barres obliques inverses sont échappées et apparaissent donc deux fois.

Motivation #

Vous venez donc de créer votre première classe en Python, une jolie petite classe qui encapsule une carte à jouer :

class Card:
    def __init__(self, suit, pips):
        self.suit = suit
        self.pips = pips

Ailleurs dans votre code, vous créez quelques instances de cette classe :

ace_of_spades = Card('Spades', 1)
four_of_clubs = Card('Clubs',  4)
six_of_hearts = Card('Hearts', 6)

Vous avez même créé une liste de cartes, afin de représenter une “main”:

my_hand = [ace_of_spades, four_of_clubs, six_of_hearts]

Maintenant, pendant le débogage, vous voulez voir à quoi ressemble votre main, alors vous faites ce qui vous vient naturellement et écrivez :

print(my_hand)

Mais ce que vous obtenez en retour est un tas de charabia:

[<__main__.Card instance at 0x0000000002533788>, 
 <__main__.Card instance at 0x00000000025B95C8>, 
 <__main__.Card instance at 0x00000000025FF508>]

Confus, vous essayez d’imprimer une seule carte :

print(ace_of_spades)

Et encore une fois, vous obtenez cette sortie étrange :

<__main__.Card instance at 0x0000000002533788>

N’ai pas peur. Nous sommes sur le point de résoudre ce problème.

Mais d’abord, il est important de comprendre ce qui se passe ici. Lorsque vous avez écrit print(ace_of_spades), vous avez dit à Python que vous vouliez qu’il imprime des informations sur l’instance Card que votre code appelle ace_of_spades. Et pour être juste, il l’a fait.

Cette sortie est composée de deux bits importants : le type de l’objet et l’[id](https: //docs.python.org/3/library/functions.html#id). La deuxième partie seule (le nombre hexadécimal) est suffisante pour identifier de manière unique l’objet au moment de l’appel “print”.[1]

Ce qui s’est vraiment passé, c’est que vous avez demandé à Python de “mettre en mots” l’essence de cet objet, puis de vous l’afficher. Une version plus explicite de la même machinerie pourrait être :

string_of_card = str(ace_of_spades)
print(string_of_card)

Dans la première ligne, vous essayez de transformer votre instance Card en chaîne, et dans la seconde vous l’affichez.

Le problème #

Le problème que vous rencontrez est dû au fait que, bien que vous ayez dit à Python tout ce qu’il devait savoir sur la classe Card pour que vous puissiez créer des cartes, vous ne lui avez pas dit comment vous vouliez Card instances à convertir en chaînes.

Et comme il ne le savait pas, lorsque vous avez (implicitement) écrit str(ace_of_spades), il vous a donné ce que vous avez vu, une représentation générique de l’instance Card.

La solution (Partie 1) #

Mais nous pouvons dire à Python comment nous voulons que les instances de nos classes personnalisées soient converties en chaînes. Et la façon dont nous le faisons est avec la méthode __str__ “dunder” (pour double-underscore) ou “magic”.

Chaque fois que vous demandez à Python de créer une chaîne à partir d’une instance de classe, il recherche une méthode __str__ sur la classe et l’appelle.

Considérez la version mise à jour suivante de notre classe Card :

class Card:
    def __init__(self, suit, pips):
        self.suit = suit
        self.pips = pips

    def __str__(self):
        special_names = {1:'Ace', 11:'Jack', 12:'Queen', 13:'King'}

        card_name = special_names.get(self.pips, str(self.pips))

        return "%s of %s" % (card_name, self.suit)

Ici, nous avons maintenant défini la méthode __str__ sur notre classe Card qui, après une simple recherche dans le dictionnaire des cartes faciales, renvoie une chaîne formatée comme nous le décidons.

(Notez que “returns” est en gras ici, pour souligner l’importance de renvoyer une chaîne, et pas simplement de l’imprimer. L’imprimer peut sembler fonctionner, mais vous auriez alors la carte imprimée lorsque vous feriez quelque chose comme str( ace_of_spades), sans même avoir un appel de fonction d’impression dans votre programme principal. Donc, pour être clair, assurez-vous que __str__ renvoie une chaîne.).

La méthode __str__ est une méthode, donc le premier argument sera self et il ne doit ni accepter ni passer d’arguments supplémentaires.

Revenons à notre problème d’affichage de la carte de manière plus conviviale, si nous exécutons à nouveau :

ace_of_spades = Card('Spades', 1)
print(ace_of_spades)

As de pique

Super, on a fini, non ?

Eh bien, juste pour couvrir nos bases, vérifions que nous avons résolu le premier problème que nous avons rencontré, en imprimant la liste des instances de Card, la hand.

On revérifie donc le code suivant :

my_hand = [ace_of_spades, four_of_clubs, six_of_hearts]
print(my_hand)

Et, à notre grande surprise, nous obtenons à nouveau ces drôles de codes hexadécimaux :

[<__main__.Card instance at 0x00000000026F95C8>, 
 <__main__.Card instance at 0x000000000273F4C8>, 
 <__main__.Card instance at 0x0000000002732E08>]

Que se passe-t-il? Nous avons dit à Python comment nous voulions que nos instances Card soient affichées, pourquoi a-t-il apparemment semblé oublier ?

La solution (Partie 2) #

Eh bien, la machinerie en coulisses est un peu différente lorsque Python veut obtenir la représentation sous forme de chaîne des éléments d’une liste. Il s’avère que Python ne se soucie pas de __str__ à cette fin.

Au lieu de cela, il recherche une méthode différente, __repr__, et si c’est pas trouvé, il retombe sur la “chose hexadécimale”.[2]

Donc vous dites que je dois faire deux méthodes pour faire la même chose ? Une pour quand je veux “imprimer” ma carte toute seule et une autre quand elle est dans une sorte de conteneur ?

Non, mais regardons d’abord à quoi ressemblerait notre classe serait si nous devions implémenter les méthodes __str__ et __repr__ :

class Card:
    special_names = {1:'Ace', 11:'Jack', 12:'Queen', 13:'King'}

    def __init__(self, suit, pips):
        self.suit = suit
        self.pips = pips

    def __str__(self):
        card_name = Card.special_names.get(self.pips, str(self.pips))
        return "%s of %s (S)" % (card_name, self.suit)

    def __repr__(self):
        card_name = Card.special_names.get(self.pips, str(self.pips))
        return "%s of %s (R)" % (card_name, self.suit)

Ici, l’implémentation des deux méthodes __str__ et __repr__ est exactement la même, sauf que, pour différencier les deux méthodes, (S) est ajouté aux chaînes renvoyées par __str__ et (R) est ajouté aux chaînes renvoyées par __repr__.

Notez que tout comme notre méthode __str__, __repr__ n’accepte aucun argument et renvoie une chaîne.

Nous pouvons voir maintenant quelle méthode est responsable de chaque cas :

ace_of_spades = Card('Spades', 1)
four_of_clubs = Card('Clubs',  4)
six_of_hearts = Card('Hearts', 6)

my_hand = [ace_of_spades, four_of_clubs, six_of_hearts]

print(my_hand)           # [Ace of Spades (R), 4 of Clubs (R), 6 of Hearts (R)]

print(ace_of_spades)     # Ace of Spades (S)

Comme nous l’avons vu, la méthode __str__ a été appelée lorsque nous avons passé notre instance Card à print et la méthode __repr__ a été appelée lorsque nous avons passé une liste de nos instances à print.

À ce stade, il convient de souligner que, tout comme nous pouvons créer explicitement une chaîne à partir d’une instance de classe personnalisée en utilisant [str()](https://docs.python.org/3/library/functions.html#func- str) comme nous l’avons fait précédemment, nous pouvons également créer explicitement une représentation sous forme de chaîne de notre classe avec une fonction intégrée appelée [repr()](https://docs.python.org/3/library /fonctions.html#repr).

Par exemple:

str_card = str(four_of_clubs)
print(str_card)                     # 4 of Clubs (S)

repr_card = repr(four_of_clubs)
print(repr_card)                    # 4 of Clubs (R)

Et en plus, si défini, nous * pourrions * appeler les méthodes directement (bien que cela semble un peu flou et inutile):

print(four_of_clubs.__str__())     # 4 of Clubs (S)

print(four_of_clubs.__repr__())    # 4 of Clubs (R)

À propos de ces fonctions dupliquées… #

Les développeurs Python ont réalisé que si vous vouliez que des chaînes identiques soient renvoyées par str() et repr(), vous pourriez avoir à dupliquer fonctionnellement les méthodes - quelque chose que personne n’aime.

Donc, à la place, il y a un mécanisme en place pour éliminer le besoin de cela. Celui que je vous ai passé en douce jusqu’à maintenant. Il s’avère que si une classe implémente la méthode __repr__ mais pas la méthode __str__, et que vous passez une instance de cette classe à str() (que ce soit implicitement ou explicitement), Python se rabattra sur votre Implémentation de __repr__ et utilisez-la.

Donc, pour être clair, considérons la version suivante de la classe Card :

class Card:
    special_names = {1:'Ace', 11:'Jack', 12:'Queen', 13:'King'}

    def __init__(self, suit, pips):
        self.suit = suit
        self.pips = pips

    def __repr__(self):
        card_name = Card.special_names.get(self.pips, str(self.pips))
        return "%s of %s" % (card_name, self.suit)

Notez que cette version uniquement implémente la méthode __repr__. Néanmoins, les appels à str() donnent une version conviviale :

print(six_of_hearts)            # 6 of Hearts  (implicit conversion)
print(str(six_of_hearts))       # 6 of Hearts  (explicit conversion)

tout comme les appels à repr() :

print([six_of_hearts])          #[6 of Hearts] (implicit conversion)
print(repr(six_of_hearts))      # 6 of Hearts  (explicit conversion)

Sommaire #

Afin de permettre à vos instances de classe de “se montrer” de manière conviviale, vous voudrez envisager d’implémenter au moins la méthode __repr__ de votre classe. Si ma mémoire est bonne, lors d’une conférence, Raymond Hettinger a déclaré que s’assurer que les classes implémentent __repr__ est l’une des premières choses qu’il recherche lors de la révision du code Python, et maintenant, il devrait être clair pourquoi. La quantité d’informations que vous auriez pu ajouter aux instructions de débogage, aux rapports de plantage ou aux fichiers journaux avec une méthode simple est écrasante par rapport aux informations dérisoires et souvent peu utiles (type, identifiant) fournies par défaut. .

Si vous voulez des représentations différentes pour quand, par exemple, à l’intérieur d’un conteneur, vous voudrez implémenter les méthodes __repr__ et __str__. (Plus d’informations sur la façon dont vous pourriez utiliser ces deux méthodes différemment ci-dessous).

Les deux méthodes sont implémentées, style eval-round-trip repr() #

class Card:
    special_names = {1:'Ace', 11:'Jack', 12:'Queen', 13:'King'}

    def __init__(self, suit, pips):
        self.suit = suit
        self.pips = pips

    # Called when instance is converted to a string via str()
    # Examples:
    #   print(card1)
    #   print(str(card1)
    def __str__(self):
        card_name = Card.special_names.get(self.pips, str(self.pips))
        return "%s of %s" % (card_name, self.suit)

    # Called when instance is converted to a string via repr()
    # Examples:
    #   print([card1, card2, card3])
    #   print(repr(card1))
    def __repr__(self):
        return "Card(%s, %d)" % (self.suit, self.pips)
