Dans les tests de performances JMeter, les corrélations signifient la possibilité de récupérer des données dynamiques à partir de la réponse du serveur et de les publier dans les requêtes suivantes. Cette fonctionnalité est essentielle pour de nombreux aspects des tests, comme les applications protégées basées sur des jetons.

Corrélation à l’aide de l’extracteur d’expressions régulières dans Apache JMeter #

Si vous avez besoin d’extraire des informations d’une réponse textuelle, le moyen le plus simple consiste à utiliser des expressions régulières. Le modèle de correspondance est très similaire à celui utilisé en Perl. Supposons que nous voulions tester un workflow d’achat de billets d’avion. La première étape consiste à soumettre l’opération d’achat. L’étape suivante consiste à s’assurer que nous sommes en mesure de vérifier tous les détails en utilisant l’ID d’achat, qui doit être renvoyé lors de la première demande. Imaginons que la première requête renvoie une page html avec ce type d’ID que nous devons extraire :

<div class="container">
  <div class="container hero-unit">
    <h1>Thank you for you purchse today!</h1>
       <table class="table">
           <tr>
             <td>Id</td>
             <td>Your purchase id is 1484697832391</td>
           </tr>
           <tr>
             <td>Status</td>
             <td>Pending</td>
           </tr>
           <tr>
             <td>Amount</td>
             <td>120 USD</td>
            </tr>
         </table>
    </div>
</div>

Ce type de situation est le meilleur candidat pour utiliser l’extracteur d’expressions régulières JMeter. L’expression régulière est une chaîne de texte spéciale pour décrire un modèle de recherche. Il existe de nombreuses ressources en ligne qui aident à écrire et à tester des expressions régulières. L’un d’eux est https://regex101.com/.

[![entrez la description de l’image ici][1]][1]

Pour utiliser ce composant, ouvrez le menu JMeter et : Ajouter -> Post-processeurs -> Extracteur d’expressions régulières

[![entrez la description de l’image ici][2]][2]

L’extracteur d’expressions régulières contient ces champs :

  • Nom de référence - le nom de la variable qui peut être utilisé après l’extraction
  • Expression régulière - une séquence de symboles et de caractères exprimant une chaîne (motif) qui sera recherchée dans le texte
  • Modèle - contient des références aux groupes. Comme une expression régulière peut avoir plus d’un groupe, elle permet de spécifier la valeur de groupe à extraire en spécifiant le numéro de groupe comme $1$ ou $2$ ou $1$$2$ (extraire les deux groupes)
  • Match No. - spécifie quelle correspondance sera utilisée (la valeur 0 correspond aux valeurs aléatoires / tout nombre positif N signifie sélectionner la Nième correspondance / la valeur négative doit être utilisée avec le contrôleur ForEach)
  • Par défaut - la valeur par défaut qui sera stockée dans la variable au cas où aucune correspondance n’est trouvée, est stockée dans la variable.

La case à cocher “Appliquer à” traite des exemples qui font des demandes de ressources intégrées. Ce paramètre définit si l’expression régulière sera appliquée aux principaux résultats de l’échantillon ou à toutes les requêtes, y compris les ressources intégrées. Il existe plusieurs options pour ce paramètre :

  • Échantillon principal et sous-échantillons
  • Échantillon principal uniquement
  • Sous-échantillons uniquement
  • Variable JMeter - l’assertion est appliquée au contenu de la variable nommée, qui peut être remplie par une autre requête

La case à cocher “Champ à vérifier” permet de choisir à quel champ l’expression régulière doit être appliquée. Presque tous les paramètres sont auto-descriptifs :

  • Corps - le corps de la réponse, par ex. le contenu d’une page Web (hors en-têtes)
  • Corps (non échappé) - le corps de la réponse, avec tous les codes d’échappement HTML remplacés. Notez que les échappements HTML sont traités sans tenir compte du contexte, donc certaines substitutions incorrectes peuvent être faites (*cette option a un impact important sur les performances)
  • Corps - Corps en tant que document - le texte extrait de divers types de documents via Apache Tika (* peut également avoir un impact sur les performances)
  • Corps - En-têtes de demande - peut ne pas être présent pour les échantillons non HTTP
  • Corps - En-têtes de réponse - peut ne pas être présent pour les échantillons non HTTP
  • Corps - URL
  • Code de réponse - par ex. 200
  • Corps - Message de réponse - par ex. D’ACCORD

Une fois l’expression extraite, elle peut être utilisée dans des requêtes ultérieures à l’aide de la variable ${purchaseId}.

Corrélation à l’aide de l’extracteur XPath dans JMeter #

XPath peut être utilisé pour parcourir les éléments et les attributs d’un document XML. Cela peut être utile lorsque les données de la réponse ne peuvent pas être extraites à l’aide de l’extracteur d’expressions régulières. Par exemple, dans le cas d’un scénario où vous devez extraire des données de balises similaires avec les mêmes attributs, mais de valeurs différentes. L’extracteur XPath est similaire à l’extracteur CSS/JQuery, mais l’extracteur XPath doit être utilisé pour le contenu XML tandis que l’extracteur CSS/JQuery doit être utilisé pour le contenu HTML. Supposons que dans la réponse, nous ayons un tableau avec différentes valeurs où nous devons extraire la valeur de la deuxième ligne du tableau.

<div id="weeklyPrices">
    <tr>
        <td>$56.00</td>
        <td>$56.00</td>
        <td>$56.00</td>
        <td>$56.00</td>
        <td>$60.00</td>
        <td>$70.00</td>
        <td>$70.00</td>
    </tr>
</div>

À l’avenir, le bon XPath pour ce cas sera : //div[@id=‘weeklyPrices’]/tr/td[1]

Pour utiliser ce composant, ouvrez le menu JMeter et : Ajouter -> Post-processeurs -> Extracteur XPath

[![entrez la description de l’image ici][2]][2]

XPath Extractor contient plusieurs éléments de configuration communs qui sont mentionnés dans la « Corrélation à l’aide de l’extracteur d’expressions régulières ». Cela inclut Nom, Appliquer à, Nom de référence, N° de correspondance (depuis JMeter 3.2) et Valeur par défaut.

Il existe de nombreuses ressources Web avec des feuilles de triche et des éditeurs en ligne pour créer et tester votre xpath créé (comme [celui-ci] [3]). Mais sur la base des exemples ci-dessous, nous pouvons trouver le moyen de créer les localisateurs xpath les plus courants.

[![entrez la description de l’image ici][1]][1]

Si vous souhaitez analyser HTML en XHTML, nous devons cocher l’option “Utiliser Tidy”. Après avoir décidé du statut “Utiliser Tidy”, il existe également des options supplémentaires :

Si “Utiliser Tidy” est coché :

  • Silencieux - définit le drapeau Tidy Quiet
  • Signaler les erreurs - si une erreur Tidy se produit, définissez l’assertion en conséquence
  • Afficher les avertissements - définit l’option Tidy afficher les avertissements

Si “Utiliser Tidy” n’est pas coché :

  • Utiliser les espaces de noms - si coché, l’analyseur XML utilisera l’espace de noms résolution
  • Valider XML - vérifier le document par rapport à son schéma spécifié
  • Ignorer l’espace blanc - ignorer l’espace blanc de l’élément
  • Récupérer les DTD externes - si sélectionné, les DTD externes sont récupérées

“Renvoyer le fragment XPath entier au lieu du contenu textuel” est auto-descriptif et doit être utilisé si vous souhaitez renvoyer non seulement la valeur xpath, mais également la valeur dans son localisateur xpath. Cela pourrait être utile pour les besoins de débogage.

Il convient également de mentionner qu’il existe une liste de plug-ins de navigateur très pratiques pour tester les localisateurs XPath. Pour Firefox, vous pouvez utiliser le plugin “[Firebug][4]” tandis que pour Chrome, “[XPath Helper][5]” est l’outil le plus pratique.

Corrélation à l’aide de l’extracteur CSS/JQuery dans JMeter #

L’extracteur CSS/JQuery permet d’extraire des valeurs d’une réponse de serveur à l’aide d’une syntaxe de sélecteur CSS/JQuery, qui aurait autrement été difficile à écrire à l’aide d’une expression régulière. En tant que post-processeur, cet élément doit être exécuté pour extraire les nœuds, le texte ou les valeurs d’attribut demandés d’un échantillonneur de requêtes et pour stocker le résultat dans la variable donnée. Ce composant est très similaire à XPath Extractor. Le choix entre CSS, JQuery ou XPath dépend généralement des préférences de l’utilisateur, mais il convient de mentionner que XPath ou JQuery peuvent parcourir et remonter le DOM, tandis que CSS ne peut pas remonter le DOM. Supposons que nous voulions extraire tous les sujets de la documentation Stack Overflow liés à Java. Vous pouvez utiliser le plugin [Firebug][1] pour tester vos sélecteurs CSS/JQuery dans Firefox, ou le [CSS Selector Tester][2] dans Chrome.

[![entrez la description de l’image ici][3]][3]

Pour utiliser ce composant, ouvrez le menu JMeter et : Ajouter -> Post-processeurs -> Extracteur CSS/JQuery

[![entrez la description de l’image ici][4]][4]

Presque tous les champs de cet extracteur sont similaires aux champs de l’extracteur d’expression régulière, vous pouvez donc obtenir leur description à partir de cet exemple. Une différence cependant est le champ “Mise en œuvre de l’extracteur CSS/JQuery”. Depuis JMeter 2.9, vous pouvez utiliser l’extracteur CSS/JQuery basé sur deux implémentations différentes : l’implémentation [jsoup][5] (description détaillée de sa syntaxe [ici][6]) ou le [JODD Lagarto][7] (syntaxe détaillée peut être trouvée [ici][8]). Les deux implémentations sont presque identiques et n’ont que de petites différences de syntaxe. Le choix entre eux est basé sur les préférences de l’utilisateur.

[![entrez la description de l’image ici][9]][9]

Sur la base de la configuration mentionnée ci-dessus, nous pouvons extraire tous les sujets de la page demandée et vérifier les résultats extraits à l’aide de l’écouteur “Debug Sampler” et “View Results Tree”.

Corrélation à l’aide de l’extracteur JSON #

JSON est un format de données couramment utilisé dans les applications Web. L’extracteur JMeter JSON fournit un moyen d’utiliser des expressions de chemin JSON pour extraire des valeurs à partir de réponses basées sur JSON dans JMeter. Ce post-processeur doit être placé en tant qu’enfant de l’échantillonneur HTTP ou de tout autre échantillonneur ayant des réponses.

Pour utiliser ce composant, ouvrez le menu JMeter et : Ajouter -> Post-processeurs -> Extracteur JSON.

[![entrez la description de l’image ici][1]][1]

L’extracteur JSON est très similaire à l’extracteur d’expressions régulières. Presque tous les champs principaux sont mentionnés dans cet exemple. Il n’y a qu’un seul paramètre spécifique à l’extracteur JSON : “Compute concatenation var”. Dans le cas où de nombreux résultats sont trouvés, cet extracteur les concaténera en utilisant le séparateur ‘,’ et en le stockant dans une variable nommée _ALL.

Supposons cette réponse du serveur avec JSON :

{ 
    "store": {
        "book": [ 
          { "category": "reference",
            "author": "Nigel Rees",
            "title": "Sayings of the Century",
            "price": 8.95
          },
          { "category": "fiction",
            "author": "Evelyn Waugh",
            "title": "Sword of Honour",
            "price": 12.99
          },
          { "category": "fiction",
            "author": "Herman Melville",
            "title": "Moby Dick",
            "isbn": "0-553-21311-3",
            "price": 8.99
          },
          { "category": "fiction",
            "author": "J. R. R. Tolkien",
            "title": "The Lord of the Rings",
            "isbn": "0-395-19395-8",
            "price": 22.99
          }
        ],
        "bicycle": {
        "color": "red",

    }
  }
}

Corrélation automatisée à l’aide de “SmartJMX” de BlazeMeter #

Lorsque vous écrivez manuellement vos scripts de performance, vous devez gérer vous-même la corrélation. Mais il existe une autre option pour créer vos scripts - l’enregistrement des scripts d’automatisation. D’une part, l’approche manuelle vous aide à écrire des scripts structurés et vous pouvez ajouter tous les extracteurs requis en même temps. D’autre part, cette approche est très chronophage.

L’enregistrement des scripts d’automatisation est très simple et vous permet de faire le même travail, mais beaucoup plus rapidement. Mais si vous utilisez des méthodes d’enregistrement courantes, les scripts seront très peu structurés et nécessiteront généralement l’ajout d’un paramétrage supplémentaire. La fonction “Smart JMX” de l’enregistreur Blazemeter combine les avantages des deux manières. Il se trouve sur ce lien : [https://a.blazemeter.com/app/recorder/index.html][1]

Après l’enregistrement, allez dans la section “Enregistreur”.

[![entrez la description de l’image ici][1]][1]

Pour démarrer l’enregistrement du script, vous devez d’abord configurer le proxy de votre navigateur ([couvert ici][2]), mais cette fois, vous devez obtenir un hôte proxy et un port fournis par l’enregistreur BlazeMeter.

[![entrez la description de l’image ici][3]][3]

Lorsque le navigateur est configuré, vous pouvez poursuivre l’enregistrement du script en appuyant sur le bouton rouge en bas. Vous pouvez maintenant accéder à l’application testée et effectuer des flux de travail utilisateur pour l’enregistrement.

[![entrez la description de l’image ici][4]][4]

Une fois le script enregistré, vous pouvez exporter les résultats dans un fichier JMX “SMART”. Un fichier jmx exporté contient une liste d’options qui vous permettent de configurer votre script et de le paramétrer, sans efforts supplémentaires. L’une de ces améliorations est que le JMX « SMART » trouve automatiquement les candidats de corrélation, les remplace par l’extracteur approprié et fournit un moyen simple pour un paramétrage ultérieur.
