Le paramétrage est la création de différents ensembles de données pour différents utilisateurs dans le même script de test. Par exemple, exécuter plusieurs utilisateurs avec des informations d’identification différentes dans le même script. Cela en fait l’un des principaux aspects de la création de tests de performance.

Paramétrage à l’aide de fichiers externes #

L’un des moyens courants de paramétrer vos scripts de performance consiste à utiliser un fichier CSV. Le meilleur exemple d’utilisation de fichiers d’entrée CSV est un processus de connexion. Si vous souhaitez tester votre application sur différents utilisateurs, vous devez fournir une liste d’informations d’identification utilisateur.

Supposons que nous ayons une demande de connexion qui fonctionne pour un utilisateur spécifique : [![entrez la description de l’image ici][1]][1]

Nous pouvons facilement paramétrer cette demande en utilisant un fichier CSV externe et en exécutant le script sur différents utilisateurs. Pour ajouter une configuration de paramétrage CSV :

Clic droit sur demande de connexion -> Ajouter -> Élément de configuration -> Configuration du jeu de données CSV

[![entrez la description de l’image ici][2]][2]

Une brève explication des paramètres “CSV Data Set Config” :

  • Nom - nom de l’élément tel qu’il sera utilisé dans l’arborescence JMeter
  • Nom de fichier - nom du fichier d’entrée. Les noms de fichiers relatifs sont résolus en fonction du chemin du plan de test actif. Les noms de fichiers absolus sont également pris en charge
  • Encodage de fichier - encodage du fichier d’entrée, s’il ne s’agit pas de la plate-forme par défaut
  • Noms de variables - liste de noms de variables séparés qui seront utilisés comme conteneur pour les valeurs analysées. Si vide, la première ligne du fichier sera interprétée comme la liste des noms de variables
  • Délimiteur - délimiteur qui sera utilisé pour séparer les valeurs analysées du fichier d’entrée
  • Autoriser les données citées ? - true si vous souhaitez ignorer les guillemets doubles et autoriser ces éléments à contenir un délimiteur.
  • Recycler sur EOF ? - true dans le cas où le plan de test du fichier doit parcourir le fichier plus d’une fois. Il demandera à JMeter de déplacer le curseur au début du fichier
  • Arrêter le fil sur EOF ? - false en cas d’itération de boucle sur le fichier CDC et true si vous souhaitez arrêter le thread après avoir lu tout le fichier
  • Mode de partage:
  • Tous les threads - le fichier est partagé entre tous les utilisateurs virtuels (par défaut)
  • Groupe de threads actuel - le fichier sera ouvert une fois pour chaque groupe de threads
  • Thread actuel - chaque fichier sera ouvert séparément pour chacun des threads
  • Identifiant - tous les threads partageant le même identifiant partagent également le même fichier

Créons un fichier csv contenant différents utilisateurs avec noms et mots de passe : [![entrez la description de l’image ici][3]][3]

Nous pouvons maintenant utiliser ce fichier avec la configuration du jeu de données CSV. Dans notre cas, il suffit d’ajouter les valeurs de configuration “Filename” et “Variables Names” : [![entrez la description de l’image ici][4]][4]

La dernière étape que nous devons franchir consiste à paramétrer la demande de connexion avec des variables CSV. Cela peut être fait en remplaçant les valeurs initiales par les variables appropriées du champ de configuration “Noms de variables” de la configuration de l’ensemble de données CSV, comme ceci : [![entrez la description de l’image ici][5]][5] Si nous exécutons notre script de test maintenant, JMeter remplacera ces variables par les valeurs du fichier “TestUsers.csv”. Chaque utilisateur virtuel JMeter recevra les informations d’identification de la ligne de fichier csv suivante.

Paramétrage à l’aide de bases de données #

Une autre façon de paramétrer vos scripts de performances consiste à utiliser les données de la base de données via JDBC. JDBC est une interface de programmation d’application qui définit comment un client peut accéder à une base de données.

Tout d’abord, téléchargez le pilote JDBC dans votre base de données (consultez le fournisseur de la base de données). Par exemple, le pilote mysql peut être trouvé ici. Ensuite, vous pouvez l’ajouter en ajoutant le fichier .jar au plan de test en utilisant le formulaire ci-dessous :

[![entrez la description de l’image ici][1]][1]

Mais il est préférable d’ajouter le fichier Jar dans le dossier lib et de redémarrer JMeter.

Ensuite, configurez la connexion à la base de données à l’aide de l’élément “Configuration de la connexion JDBC”. Comme ça: Cliquez avec le bouton droit sur Groupe de threads -> Ajouter -> Élément de configuration -> Configuration de la connexion JDBC

[![entrez la description de l’image ici][2]][2]

Paramètres “Configuration de la connexion JDBC” :

  • Nom - nom de la configuration de connexion qui sera affichée dans l’arborescence des groupes de threads
  • Nom de la variable - nom qui sera utilisé comme identifiant unique pour la connexion à la base de données (plusieurs connexions peuvent être utilisées et chacune sera liée à un nom différent)
  • Nombre maximum de connexions - nombre maximum de connexions autorisées dans le pool de connexions. En cas de 0, chaque thread obtiendra son propre pool avec une seule connexion
  • Max Wait (ms) - le pool génère une erreur si le délai d’attente spécifié est dépassé lors de la connexion à la base de données
  • Temps entre les exécutions d’expulsion (ms) - nombre de millisecondes pour faire une pause entre les exécutions du thread qui expulse les connexions inutilisées du pool de base de données
  • Validation automatique - oui pour activer la validation automatique pour les connexions de base de données associées
  • Test While Idle - vérifie les connexions inactives avant qu’une demande effective ne soit détectée. Plus de détails : BasicDataSource.html#getTestWhileIdle
  • Soft Min Evictable Idle Time (ms) - période de temps pendant laquelle une connexion spécifiée peut être inactive dans le pool de base de données avant de pouvoir être expulsée. Plus de détails : BasicDataSource.html#getSoftMinEvictableIdleTimeMillis
  • Requête de validation - requête de vérification de l’état qui sera utilisée pour vérifier si la base de données répond toujours
  • URL de la base de données - Chaîne de connexion JDBC pour la base de données. Voir ici pour des exemples
  • Classe de pilote JDBC - nom approprié de la classe de pilote (spécifique à chaque base de données). Par exemple, ‘com.mysql.jdbc.Driver’ pour MySql db
  • Nom d’utilisateur - nom d’utilisateur de la base de données
  • Mot de passe - mot de passe de la base de données (sera stocké non chiffré dans le plan de test)

Dans notre cas, nous devons uniquement configurer les champs obligatoires :

  • Nom de la variable Lié au pool.
  • URL de la base de données
  • Classe de pilote JDBC
  • Nom d’utilisateur
  • Mot de passe

Les autres champs de l’écran peuvent être laissés par défaut : [![entrez la description de l’image ici][3]][3]

Supposons que nous stockons les informations d’identification de l’utilisateur de test dans la base de données : [![entrez la description de l’image ici][4]][4]

Maintenant, lorsque la connexion à la base de données est configurée, nous pouvons ajouter la requête JDBC elle-même et utiliser sa requête pour obtenir toutes les informations d’identification de la base de données : Clic droit sur Thread Group -> Add -> Sample -> JDBC Request

En utilisant la requête “Select Statement” et “Variable Names”, nous pouvons analyser la réponse aux variables personnalisées.

[![entrez la description de l’image ici][5]][5]

Nous aurons maintenant des variables JMeter qui pourront être utilisées davantage dans les requêtes ultérieures. Les variables spécifiées seront créées avec un suffixe incrémentiel (email_1, email_2, email_3…..).

Pour utiliser ces variables dans la “demande de connexion”, nous devons ajouter un compteur qui sera utilisé pour accéder aux bonnes valeurs à partir de la réponse à la requête JDBC. Pour ajouter l’élément “Counter” dans JMeter : Clic droit sur Thread Group -> Add -> Config Element -> Counter

[![entrez la description de l’image ici][6]][6]

Après cela, nous pouvons mettre à jour la “demande de connexion” à l’aide de la fonction __V. Cela renvoie le résultat de l’évaluation d’une expression de nom de variable et peut être utilisé pour évaluer les références de variables imbriquées :

[![entrez la description de l’image ici][7]][7]

La configuration spécifiée est suffisante pour utiliser les valeurs de la base de données afin d’exécuter le script sur différents utilisateurs : [![entrez la description de l’image ici][8]][8] [![entrez la description de l’image ici][9]][9]

Paramétrage à l’aide du plugin ‘Parameterized Controller’ #

Si vous avez besoin d’exécuter une séquence répétitive de la même action avec des paramètres différents, utilisez le plug-in tiers “Parameterized Controller” du projet [JMeter-Plugins][1].

Vous devez d’abord installer ce plugin en suivant la procédure d’installation.

Supposons que nous souhaitions paramétrer le workflow de connexion :

[![entrez la description de l’image ici][2]][2]

Tout d’abord, vous devez installer le plug-in “Parameterized Controller” car il n’est pas inclus dans le noyau JMeter. Les étapes d’installation de ce processus peuvent être trouvées ici.

Déplaçons la “demande de connexion” dans un contrôleur séparé et désactivons-la (faites un clic droit dessus et sélectionnez “Désactiver”). C’est le moyen le plus préférable d’avoir un conteneur de modules dans votre plan de test et d’éviter d’utiliser Workbench comme un tel conteneur. Une fois l’installation terminée, vous pouvez ajouter deux contrôleurs “Parameterized Controller” avec des informations d’identification utilisateur différentes : Clic droit sur Groupe de threads -> Ajouter -> Contrôleur logique -> Contrôleur paramétré

[![entrez la description de l’image ici][3]][3]

Les contrôleurs paramétrés contiennent la section “Variables définies par l’utilisateur”, dans laquelle vous pouvez spécifier vos paramètres. Placez les informations d’identification du premier utilisateur dans le premier contrôleur paramétré et les informations d’identification du deuxième utilisateur dans le deuxième contrôleur paramétré. [![entrez la description de l’image ici][4]][4]

À l’intérieur des deux contrôleurs paramétrés, ajoutez des références au « contrôleur réutilisable » pour appeler la « demande de connexion » avec des paramètres différents. Cela peut être fait de cette façon:

Clic droit sur “Contrôleur paramétré” -> “Ajouter” -> “Contrôleur logique” -> “Contrôleur de module”

[![entrez la description de l’image ici][5]][5]

Lors de l’exécution de votre script, vous verrez que la “demande de connexion” a déclenché séparément chacun des contrôleurs paramétrés. Cela peut être très utile au cas où vous auriez besoin d’exécuter votre script sur différentes combinaisons de paramètres d’entrée.

[![entrez la description de l’image ici][6]][6]
