Les noms de valeur et de variable doivent être en minuscule camel #

Les noms de constantes doivent être en majuscules majuscules. C’est-à-dire que si le membre est final, immuable et appartient à un objet package ou à un objet, il peut être considéré comme une constante

Les noms de méthode, de valeur et de variable doivent être en minuscule camel

Source : http://docs.scala-lang.org/style/naming-conventions.html

Cette compilation :

val (a,b) = (1,2)
// a: Int = 1
// b: Int = 2

mais ce n’est pas:

val (A,B) = (1,2)
// error: not found: value A
// error: not found: value B

Il n’y a pas que val contre var #

val et var #

scala> val a = 123
a: Int = 123

scala> a = 456
<console>:8: error: reassignment to val
       a = 456

scala> var b = 123
b: Int = 123

scala> b = 321
b: Int = 321
  • les références val sont immuables : comme une variable final dans Java, une fois qu’elle a été initialisée, vous ne pouvez pas la modifier
  • Les références var sont réassignables en tant que simple déclaration de variable en Java

Collections immuables et mutables #

  val mut = scala.collection.mutable.Map.empty[String, Int]
  mut += ("123" -> 123)
  mut += ("456" -> 456)
  mut += ("789" -> 789)

  val imm = scala.collection.immutable.Map.empty[String, Int]
  imm + ("123" -> 123)
  imm + ("456" -> 456)
  imm + ("789" -> 789)

  scala> mut
    Map(123 -> 123, 456 -> 456, 789 -> 789)

  scala> imm
    Map()

scala> imm + ("123" -> 123) + ("456" -> 456) + ("789" -> 789)
    Map(123 -> 123, 456 -> 456, 789 -> 789)

La bibliothèque standard Scala propose à la fois des structures de données immuables et modifiables, pas la référence à celles-ci. Chaque fois qu’une structure de données immuable est “modifiée”, une nouvelle instance est produite au lieu de modifier la collection d’origine sur place. Chaque instance de la collection peut partager une structure importante avec une autre instance.

[Collection mutable et immuable (documentation officielle Scala)] [1]

[1] : http://docs.scala-lang.org/overviews/collections/overview.html

Mais je ne peux pas utiliser l’immuabilité dans ce cas ! #

Prenons comme exemple une fonction qui prend 2 Map et renvoie une Map contenant tous les éléments de ma et mb :

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int]

Une première tentative pourrait consister à parcourir les éléments de l’une des cartes en utilisant for ((k, v) <- map) et à renvoyer d’une manière ou d’une autre la carte fusionnée.

def merge2Maps(ma: ..., mb: ...): Map[String, Int] = {

  for ((k, v) <- mb) {
    ???
  }

}

Ce tout premier mouvement ajoute immédiatement une contrainte : une mutation en dehors de for est maintenant nécessaire. Ceci est plus clair lors du désucrage du “pour”:

// this:
for ((k, v) <- map) { ??? }

// is equivalent to:
map.foreach { case (k, v) => ??? }

“Pourquoi devons-nous muter ?” #

foreach s’appuie sur les effets secondaires. Chaque fois que nous voulons que quelque chose se produise dans un foreach, nous devons “effet secondaire quelque chose”, dans ce cas, nous pourrions muter une variable var result ou nous pouvons utiliser une structure de données mutable.

Création et remplissage de la carte result #

Supposons que ma et mb soient scala.collection.immutable.Map, nous pourrions créer la carte result à partir de ma :

val result = mutable.Map() ++ ma

Ensuite, parcourez mb en ajoutant ses éléments et si la clé de l’élément actuel sur ma existe déjà, remplaçons-la par celle de mb.

mb.foreach { case (k, v) => result += (k -> v) }

Implémentation mutable #

Jusqu’ici tout va bien, nous “avons dû utiliser des collections mutables” et une implémentation correcte pourrait être :

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  val result = scala.collection.mutable.Map() ++ ma
  mb.foreach { case (k, v) => result += (k -> v) }
  result.toMap // to get back an immutable Map
}

Comme prévu:

scala> merge2Maps(Map("a" -> 11, "b" -> 12), Map("b" -> 22, "c" -> 23))
  Map(a -> 11, b -> 22, c -> 23)

Pliage à la rescousse #

Comment pouvons-nous nous débarrasser de “foreach” dans ce scénario ? Si tout ce que nous devons faire est de parcourir les éléments de la collection et d’appliquer une fonction tout en accumulant le résultat sur l’option, nous pourrions utiliser .foldLeft :

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  mb.foldLeft(ma) { case (result, (k, v)) => result + (k -> v) }
  // or more concisely mb.foldLeft(ma) { _ + _ }
}

Dans ce cas, notre “résultat” est la valeur accumulée à partir de ma, le zéro du .foldLeft.

Résultat intermédiaire #

Évidemment, cette solution immuable produit et détruit de nombreuses instances Map lors du pliage, mais il convient de mentionner que ces instances ne sont pas un clone complet de la Map accumulée, mais partagent plutôt une structure (données) significative avec l’instance existante.

Raisonnabilité plus facile #

Il est plus facile de raisonner sur la sémantique si elle est plus déclarative que l’approche .foldLeft. L’utilisation de structures de données immuables pourrait aider à rendre notre implémentation plus facile à raisonner.
