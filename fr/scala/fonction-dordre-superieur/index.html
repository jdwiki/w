Scala se donne beaucoup de mal pour traiter les méthodes et les fonctions comme syntaxiquement identiques. Mais sous le capot, ce sont des concepts distincts.

Une méthode est un code exécutable et n’a pas de représentation de valeur.

Une fonction est une instance d’objet réelle de type [Function1][1] (ou un type similaire d’une autre arité). Son code est contenu dans sa méthode apply. En fait, il agit simplement comme une valeur qui peut être transmise.

Incidemment, la capacité de traiter les fonctions comme des valeurs * est * exactement ce que l’on entend par un langage prenant en charge les fonctions d’ordre supérieur. Les instances de fonction sont l’approche de Scala pour implémenter cette fonctionnalité.

Une fonction réelle d’ordre supérieur est une fonction qui prend une valeur de fonction comme argument ou renvoie une valeur de fonction. Mais dans Scala, comme toutes les opérations sont des méthodes, il est plus général de penser aux méthodes qui reçoivent ou renvoient des paramètres de fonction. Donc [map][2], tel que défini sur Seq pourrait être considéré comme une “fonction d’ordre supérieur” car son paramètre est une fonction, mais ce n’est pas littéralement une fonction ; c’est une méthode.

[1] : http://www.scala-lang.org/api/current/scala/Function1.html [2] : http://www.scala-lang.org/api/current/index.html#scala.collection.Seq@mapB:Seq[B]

Utiliser des méthodes comme valeurs de fonction #

Le compilateur Scala convertira automatiquement les méthodes en valeurs de fonction dans le but de les transmettre à des fonctions d’ordre supérieur.

object MyObject {
  def mapMethod(input: Int): String = {
    int.toString
  }
}

Seq(1, 2, 3).map(MyObject.mapMethod) // Seq("1", "2", "3")

Dans l’exemple ci-dessus, MyObject.mapMethod n’est pas un appel de fonction, mais est plutôt passé à map en tant que valeur. En effet, map requiert une valeur de fonction qui lui est transmise, comme on peut le voir dans sa signature. La signature de la map d’une List[A] (une liste d’objets de type A) est :

def map[B](f: (A) ⇒ B): List[B]

La partie f: (A) => B indique que le paramètre de cet appel de méthode est une fonction qui prend un objet de type A et renvoie un objet de type B. ‘A’ et ‘B’ sont définis arbitrairement. En revenant au premier exemple, nous pouvons voir que mapMethod prend un Int (qui correspond à A) et renvoie une String (qui correspond à B). Ainsi, mapMethod est une valeur de fonction valide à transmettre à map. On pourrait réécrire le même code comme ceci :

Seq(1, 2, 3).map(x:Int => int.toString)

Cela intègre la valeur de la fonction, ce qui peut ajouter de la clarté pour des fonctions simples.

Fonctions d’ordre supérieur (fonction en tant que paramètre) #

Une fonction d’ordre supérieur, par opposition à une fonction de premier ordre, peut avoir l’une des trois formes suivantes :

  • Un ou plusieurs de ses paramètres est une fonction, et il renvoie une valeur.

  • Il renvoie une fonction, mais aucun de ses paramètres n’est une fonction.

  • Les deux ci-dessus : un ou plusieurs de ses paramètres est une fonction, et il renvoie une fonction.

      object HOF {
      def main(args: Array[String]) {
      val list = List(("Srini","E"),("Subash","R"),("Ranjith","RK"),("Vicky","s"),("Sudhar","s"))
      //HOF
       val fullNameList= list.map(n => getFullName(n._1, n._2))

       }

      def getFullName(firstName: String, lastName: String): String = firstName + "." + lastName
      }

Ici, la fonction map prend une fonction getFullName(n._1,n._2) comme paramètre. C’est ce qu’on appelle HOF (fonction d’ordre supérieur).

Arguments évaluation paresseuse #

Scala prend en charge l’évaluation paresseuse des arguments de fonction en utilisant la notation : def func(arg: => String). Un tel argument de fonction peut prendre un objet String normal ou une fonction d’ordre supérieur avec un type de retour String. Dans le second cas, l’argument de la fonction serait évalué sur l’accès à la valeur.

Veuillez voir l’exemple :

def calculateData: String = {
  print("Calculating expensive data! ")
  "some expensive data"
}

def dumbMediator(preconditions: Boolean, data: String): Option[String] = {
  print("Applying mediator")
  preconditions match {
    case true => Some(data)
    case false => None
  }
}

def smartMediator(preconditions: Boolean, data: => String): Option[String] = {
  print("Applying mediator")
  preconditions match {
    case true => Some(data)
    case false => None
  }
}

smartMediator(preconditions = false, calculateData)

dumbMediator(preconditions = false, calculateData)

L’appel smartMediator renverrait la valeur None et afficherait le message "Applying mediator".

L’appel dumbMediator renverrait la valeur None et afficherait le message "Calcul de données coûteuses! Application du médiateur".

L’évaluation paresseuse peut être extrêmement utile lorsque vous souhaitez optimiser une surcharge de calcul d’arguments coûteux.
