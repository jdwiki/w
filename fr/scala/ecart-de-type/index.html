Covariance #

Le symbole + marque un paramètre de type comme covariant - ici nous disons que “Producteur est covariant sur A”:

trait Producer[+A] {
  def produce: A
}

Un paramètre de type covariant peut être considéré comme un type de “sortie”. Marquer A comme covariant affirme que Producer[X] <: Producer[Y] à condition que X <: Y. Par exemple, un Producteur[Chat] est un Producteur[Animal] valide, car tous les chats produits sont également des animaux valides.

Un paramètre de type covariant ne peut pas apparaître en position contravariante (entrée). L’exemple suivant ne sera pas compilé car nous affirmons que Co[Cat] <: Co[Animal], mais Co[Cat] a def handle(a: Cat): Unit qui ne peut gérer aucun Animal comme requis par Co[Animal] !

trait Co[+A] {
  def produce: A
  def handle(a: A): Unit
}

Une approche pour traiter cette restriction consiste à utiliser des paramètres de type limités par le paramètre de type covariant. Dans l’exemple suivant, nous savons que ‘B’ est un supertype de ‘A’. Par conséquent, étant donné Option[X] <: Option[Y] pour X <: Y, nous savons que Def getOrElse[B >: X](b: => B) de Option[X] B peut accepter n’importe quel supertype de X - qui inclut les supertypes de Y comme requis par Option[Y] :

trait Option[+A] {
  def getOrElse[B >: A](b: => B): B
}

Contravariance #

Le symbole - marque un paramètre de type comme contravariant - ici nous disons que “Handler est contravariant sur A”:

trait Handler[-A] {
  def handle(a: A): Unit
}

Un paramètre de type contravariant peut être considéré comme un type “d’entrée”. Marquer A comme contravariant affirme que Handler[X] <: Handler[Y] à condition que X>: Y. Par exemple, un Handler[Animal] est un Handler[Cat] valide, car un Handler[Animal] doit également gérer les chats.

Un paramètre de type contravariant ne peut pas apparaître en position covariante (sortie). L’exemple suivant ne sera pas compilé car nous affirmons qu’un Contra[Animal] <: Contra[Cat], cependant un Contra[Animal] a def produire: Animal qui n’est pas garanti pour produire des chats comme requis par Contra[Chat] !

trait Contra[-A] {
   def handle(a: A): Unit
   def produce: A
}

Attention cependant : dans le but de surcharger la résolution, la contravariance inverse également de manière contre-intuitive la spécificité d’un type sur le paramètre de type contravariant - Handler[Animal] est considéré comme “plus spécifique” que Handler[Cat].

Comme il n’est pas possible de surcharger des méthodes sur des paramètres de type, ce comportement ne devient généralement problématique que lors de la résolution d’arguments implicites. Dans l’exemple suivant, ofCat ne sera jamais utilisé, car le type de retour de ofAnimal est plus spécifique :

implicit def ofAnimal: Handler[Animal] = ???
implicit def ofCat: Handler[Cat] = ???

implicitly[Handler[Cat]].handle(new Cat)

Ce comportement est actuellement prévu pour [change in dotty][1], et c’est pourquoi (par exemple) scala.math.Ordering est invariant sur son paramètre de type T. Une solution de contournement consiste à rendre votre classe de types invariante et à paramétrer par type la définition implicite dans le cas où vous souhaitez qu’elle s’applique aux sous-classes d’un type donné :

trait Person
object Person {
  implicit def ordering[A <: Person]: Ordering[A] = ???
}

[1] : https://github.com/lampepfl/dotty/commit/89540268e6c49fb92b9ca61249e46bb59981bf5a

Invariance #

Par défaut, tous les paramètres de type sont invariants - étant donné le trait A[B], nous disons que “A est invariant sur B”. Cela signifie qu’étant donné deux paramétrisations A[Cat] et A[Animal], nous n’affirmons aucune relation de sous/superclasse entre ces deux types - cela ne signifie pas que A[Cat] <: A[Animal] ni que A[Chat] > : A[Animal] quelle que soit la relation entre Chat et Animal.

Les annotations de variance nous fournissent un moyen de déclarer une telle relation et imposent des règles sur l’utilisation des paramètres de type afin que la relation reste valide.

Covariance sur un trait invariant #

Il existe également un moyen de faire en sorte qu’une seule méthode accepte un argument covariant, au lieu d’avoir le trait entier covariant. Cela peut être nécessaire car vous souhaitez utiliser ‘T’ dans une position contravariante, mais toujours covariante.

trait LocalVariance[T]{
  /// ??? throws a NotImplementedError
  def produce: T = ???
  // the implicit evidence provided by the compiler confirms that S is a
  // subtype of T.
  def handle[S](s: S)(implicit evidence: S <:< T) = {
    // and we can use the evidence to convert s into t.
    val t: T = evidence(s)
    ???
  }
}

trait A {}
trait B extends A {}

object Test {
  val lv = new LocalVariance[A] {}

  // now we can pass a B instead of an A.
  lv.handle(new B {})
}
