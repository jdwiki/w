Étape 1 Ajouter une structure #

Dans vues\site\form-submission.php

<?php Pjax::begin(['id'=>'id-pjax']); ?>
<?= Html::beginForm(['site/form-submission'], 'post', ['data-pjax' => '', 'class' => 'form-inline']); ?>
    <?= Html::input('text', 'string', Yii::$app->request->post('string'), ['class' => 'form-control']) ?>
    <?= Html::submitButton('Hash String', ['class' => 'btn btn-lg btn-primary', 'name' => 'hash-button']) ?>
<?= Html::endForm() ?>
<h3><?= $stringHash ?></h3>
<?php Pjax::end(); ?>

comment utiliser pjax #

Ajoutez cette ligne au début de votre vue.

<?php
use yii\widgets\Pjax;
?>

Ajoutez les deux lignes suivantes autour du contenu qui nécessite une mise à jour partielle.

<?php Pjax::begin(['id'=>'id-pjax']); ?>
Content that needs to be updated
<?php Pjax::end(); ?>

recharger pjax #

$.pjax.reload({container: '#id-pjax'});

utiliser l’argument timeout dans pjax #

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false]); ?>

vous pouvez spécifier une valeur entière pour l’argument timeout, qui serait le nombre de millisecondes à attendre (sa valeur par défaut est 1000). Si le temps d’exécution dans le serveur est supérieur à cette valeur de timeout, un chargement complet de la page sera déclenché.

Par défaut, pjax soumettra le formulaire en utilisant la méthode GET. Vous pouvez changer la méthode de soumission du formulaire en POST comme dans l’exemple suivant

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false, 'clientOptions' => ['method' => 'POST']]); ?>

Étape 2 Code côté serveur #

public function actionFormSubmission()
{
    $security = new Security();
    $string = Yii::$app->request->post('string');
    $stringHash = '';
    if (!is_null($string)) {
        $stringHash = $security->generatePasswordHash($string);
    }
    return $this->render('form-submission', [
        'stringHash' => $stringHash,
    ]);
}

Utilisation avancée de Pjax #

Yii Framework 2.0 est livré avec un support intégré pour [Pjax][1], une bibliothèque JavaScript qui réduit les temps de chargement des pages. Pour ce faire, il ne met à jour que la partie de la page qui a changé via Ajax, ce qui peut se traduire par des économies substantielles si vous avez de nombreux autres actifs sur vos pages. Quelques-uns de nos projets utilisent cette fonctionnalité et nous voulions partager quelques leçons apprises.

Problème : la page 1 est une simple page statique qui contient peu d’éléments. La page 2 comprend un ActiveForm ainsi que d’autres widgets. Les ressources JavaScript ActiveForm doivent être chargées pour que le JavaScript en ligne s’exécute, mais comme la page 1 n’incluait pas ces actifs, la page 2 a rencontré une erreur JavaScript lors de la tentative d’exécution de la ligne activeform : ‘Uncaught TypeError : undefined is not a fonction’.

Solution : incluez les éléments ActiveForm dans un ensemble d’éléments partagés qui seront chargés sur toutes les pages, en veillant à ce que chaque page d’entrée permette la disponibilité des scripts appropriés.

class AppAsset extends AssetBundle
{
    ...
    public $depends = [
        'yii\widgets\ActiveFormAsset',
        'yii\validators\ValidationAsset',
    ];
    ...
}

Problème : Dans le même exemple ci-dessus, la page 1 inclut quelques widgets (NavBar, etc.). La page 2 comprend les mêmes widgets plus quelques autres (ActiveForm, etc.). Lors du chargement de la page via Pjax, du JavaScript en ligne personnalisé était en cours d’exécution, mais le script en ligne placé par le widget ActiveForm ne semblait pas fonctionner, car le code de validation ne fonctionnait pas. En débogage, nous avons constaté que la fonction init ActiveForm était en cours d’exécution, mais la variable “this” ne semblait pas correspondre à l’ActiveForm. Il correspondait en fait à la div NavBar. En examinant les ID div, nous avons vu que l’ActiveForm s’attendait à avoir l’ID #w1, mais la NavBar s’était déjà vu attribuer cet ID sur la Page 1 puisqu’il s’agissait du premier widget rencontré sur cette page.

Solution : Ne comptez pas sur Yii pour générer automatiquement les identifiants de widget pour vous. Au lieu de cela, transmettez toujours un identifiant lors de la création du widget pour garder le contrôle de ces identifiants.

Problème : la requête Pjax était annulée exactement 1 000 ms après le lancement de la requête.

Solution : augmentez le paramètre de délai d’attente Pjax. Il est par défaut de 1 seconde, ce qui devrait être acceptable pour les sites de production. Cependant, en développement, lors de l’utilisation de xdebug, nos temps de chargement de pages dépassent régulièrement cette limite.

Problème : l’application Web implémente le modèle [Post-Redirect-Get (PRG)][2]. Pjax recharge la page entière au lieu de simplement la redirection.

Solution : il s’agit du comportement prévu de Pjax. La redirection ne remplit pas son objectif lors de l’utilisation de Pjax, vous pouvez donc déterminer si une demande est Pjax, et si c’est le cas, restituer le contenu au lieu de rediriger. Un exemple peut ressembler à :

$endURL = "main/endpoint";
if (Yii::$app->request->isPjax) {
    return $this->run($endURL);
} else {
    return $this->redirect([$endURL]);
}
