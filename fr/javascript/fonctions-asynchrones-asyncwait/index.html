Fonctions asynchrones (asyncwait)

async et wait s’appuient sur des promesses et des générateurs pour exprimer des actions asynchrones en ligne. Cela rend le code asynchrone ou de rappel beaucoup plus facile à maintenir.

Les fonctions avec le mot clé async renvoient une Promise et peuvent être appelées avec cette syntaxe.

Dans une fonction asynchrone, le mot-clé wait peut être appliqué à n’importe quelle Promise et entraînera l’exécution de tout le corps de la fonction après wait après la résolution de la promesse.

Syntaxe #

  • fonction asynchrone foo() {    …
       await asyncCall()
    }
  • fonction asynchrone() { … }
  • asynchrone() => { … }
  • (asynchrone () => {    const data = await asyncCall()
       console.log(data) })()

Les fonctions asynchrones sont un sucre syntaxique sur les promesses et les générateurs. Ils vous aident à rendre votre code plus lisible, maintenable, plus facile à détecter les erreurs et avec moins de niveaux d’indentation.

Introduction #

Une fonction définie comme “asynchrone” est une fonction qui peut effectuer des actions asynchrones mais semble toujours synchrone. La façon dont cela se fait consiste à utiliser le mot-clé wait pour différer la fonction pendant qu’elle attend qu’une [Promise][1] soit résolue ou rejetée.

Remarque : Les fonctions asynchrones sont une proposition de l’étape 4 (« terminée ») sur la bonne voie pour être incluses dans le Norme ECMAScript 2017.

Par exemple, en utilisant [Fetch API] (https://developer.mozilla.org/en/docs/Web/API/Fetch_API) basée sur les promesses :

async function getJSON(url) {
    try {
        const response = await fetch(url);
        return await response.json();
    }
    catch (err) {
        // Rejections in the promise will get thrown here
        console.error(err.message);
    }
}

Une fonction asynchrone renvoie toujours une promesse elle-même, vous pouvez donc l’utiliser dans d’autres fonctions asynchrones.

Style de fonction de flèche #

const getJSON = async url => {
    const response = await fetch(url);
    return await response.json();
}

[1] : https://www.wikiod.com/fr/javascript/promesses

Attente et priorité des opérateurs #

Vous devez garder à l’esprit la priorité des opérateurs lorsque vous utilisez le mot-clé “wait”.

Imaginez que nous ayons une fonction asynchrone qui appelle une autre fonction asynchrone, getUnicorn() qui renvoie une Promise qui se résout en une instance de la classe Unicorn. Maintenant, nous voulons obtenir la taille de la licorne en utilisant la méthode getSize() de cette classe.

Regardez le code suivant :

async function myAsyncFunction() {
    await getUnicorn().getSize();
}

À première vue, cela semble valable, mais ce n’est pas le cas. En raison de la priorité des opérateurs, cela équivaut à ce qui suit :

async function myAsyncFunction() {
    await (getUnicorn().getSize());
}

Ici, nous essayons d’appeler la méthode getSize() de l’objet Promise, ce qui n’est pas ce que nous voulons.

Au lieu de cela, nous devrions utiliser des crochets pour indiquer que nous voulons d’abord attendre la licorne, puis appeler la méthode getSize() du résultat :

async function asyncFunction() {
    (await getUnicorn()).getSize();
}

Bien sûr. la version précédente pouvait être valide dans certains cas, par exemple, si la fonction getUnicorn() était synchrone, mais que la méthode getSize() était asynchrone.

Fonctions asynchrones par rapport aux promesses #

Les fonctions async ne remplacent pas le type Promise ; ils ajoutent des mots-clés linguistiques qui facilitent l’appel des promesses. Ils sont interchangeables :

async function doAsyncThing() { ... }

function doPromiseThing(input) { return new Promise((r, x) => ...); }

// Call with promise syntax
doAsyncThing()
    .then(a => doPromiseThing(a))
    .then(b => ...)
    .catch(ex => ...);

// Call with await syntax
try {
    const a = await doAsyncThing();
    const b = await doPromiseThing(a);
    ...
}
catch(ex) { ... }

Toute fonction qui utilise des chaînes de promesses peut être réécrite en utilisant wait :

function newUnicorn() {
  return fetch('unicorn.json')                     // fetch unicorn.json from server
  .then(responseCurrent => responseCurrent.json()) // parse the response as JSON
  .then(unicorn =>
    fetch('new/unicorn', {                         // send a request to 'new/unicorn' 
        method: 'post',                            // using the POST method
        body: JSON.stringify({unicorn})            // pass the unicorn to the request body
    })
  )
  .then(responseNew => responseNew.json())
  .then(json => json.success)                      // return success property of response
  .catch(err => console.log('Error creating unicorn:', err));
 }

La fonction peut être réécrite en utilisant async / wait comme suit :

async function newUnicorn() {
  try {
    const responseCurrent = await fetch('unicorn.json'); // fetch unicorn.json from server
    const unicorn = await responseCurrent.json();        // parse the response as JSON
    const responseNew = await fetch('new/unicorn', {     // send a request to 'new/unicorn'
      method: 'post',                                    // using the POST method
      body: JSON.stringify({unicorn})                    // pass the unicorn to the request body
    });
    const json = await responseNew.json();
    return json.success                                  // return success property of response
  } catch (err) {
    console.log('Error creating unicorn:', err);
  }
}

Cette variante async de newUnicorn() semble renvoyer une Promise, mais il y avait en réalité plusieurs mots-clés wait. Chacun a renvoyé une “Promise”, donc nous avions vraiment une collection de promesses plutôt qu’une chaîne.

En fait, nous pouvons le considérer comme un générateur de fonction*, chaque attendre étant un rendre une nouvelle promesse. Cependant, les résultats de chaque promesse sont nécessaires à la suivante pour continuer la fonction. C’est pourquoi le mot-clé supplémentaire async est nécessaire sur la fonction (ainsi que le mot-clé await lors de l’appel des promesses) car il indique à Javascript de créer automatiquement un observateur pour cette itération. La Promise renvoyée par async function newUnicorn() se résout lorsque cette itération se termine.

Pratiquement, vous n’avez pas besoin de considérer cela ; wait masque la promesse et async masque l’itération du générateur.

Vous pouvez appeler les fonctions async comme s’il s’agissait de promesses et attendre n’importe quelle promesse ou n’importe quelle fonction async. Vous n’avez pas besoin d’attendre une fonction asynchrone, tout comme vous pouvez exécuter une promesse sans .then().

Vous pouvez également utiliser un async [IIFE][1] si vous souhaitez exécuter ce code immédiatement :

(async () => {
  await makeCoffee()
  console.log('coffee is ready!')
})()

[1] : https://en.wikipedia.org/wiki/Immediately-invoked_function_expression “Expression de fonction immédiatement invoquée”

Boucle avec attente asynchrone #

Lors de l’utilisation de l’attente asynchrone dans les boucles, vous pouvez rencontrer certains de ces problèmes.

Si vous essayez simplement d’utiliser await à l’intérieur de forEach, cela générera une erreur Jeton inattendu.

(asynchrone() => {
données = [1, 2, 3, 4, 5] ;
data.forEach(e => {
   const i = await somePromiseFn(e);
   console.log(i);
 });
})();

Cela vient du fait que vous avez vu à tort la flèche fonctionner comme un bloc. Le wait sera dans le contexte de la fonction de rappel, qui n’est pas async. L’interpréteur nous protège de l’erreur ci-dessus, mais si vous ajoutez “async” au rappel “forEach”, aucune erreur n’est générée. Vous pourriez penser que cela résout le problème, mais cela ne fonctionnera pas comme prévu.

Exemple:

(asynchrone() => {
données = [1, 2, 3, 4, 5] ;
data.forEach(async(e) => {
    const i = await somePromiseFn(e);
    console.log(i);
  });
console.log('ceci s'imprimera en premier');
})();

Cela se produit parce que la fonction asynchrone de rappel ne peut que s’interrompre elle-même, pas la fonction asynchrone parente.

Vous pouvez écrire une fonction asyncForEach qui renvoie une promesse, puis vous pouvez quelque chose comme wait asyncForEach(async (e) => wait somePromiseFn(e), data ) Fondamentalement, vous renvoyez une promesse qui se résout lorsque tous les rappels sont attendus et terminés. Mais il y a de meilleures façons de faire cela, et c’est d’utiliser simplement une boucle.

Vous pouvez utiliser une boucle for-of ou une boucle for/while, peu importe celle que vous choisissez.

(asynchrone() => {
données = [1, 2, 3, 4, 5] ;
pour (let e de données) {
    const i = await somePromiseFn(e);
    console.log(i);
  }
console.log('ceci imprimera en dernier');
})();

Mais il y a un autre hic. Cette solution attendra que chaque appel à somePromiseFn soit terminé avant de parcourir le suivant. C’est très bien si vous voulez réellement que vos invocations somePromiseFn soient exécutées dans l’ordre, mais si vous voulez qu’elles s’exécutent simultanément, vous devrez attendre sur Promise.all.

(asynchrone() => {
données = [1, 2, 3, 4, 5] ;
const p = attendre Promise.all(data.map(async(e) => attendre unePromiseFn(e)));
console.log(...p);
})();

Promise.all reçoit un tableau de promesses comme seul paramètre et renvoie une promesse. Lorsque toutes les promesses du tableau sont résolues, la promesse renvoyée est également résolue. Nous “attendons” cette promesse et quand elle est résolue, toutes nos valeurs sont disponibles.

Les exemples ci-dessus sont entièrement exécutables. La fonction somePromiseFn peut être définie comme une fonction d’écho asynchrone avec un délai d’attente. Vous pouvez essayer les exemples dans babel-repl avec au moins le préréglage stage-3 et regarder la sortie.

function unePromiseFn(n) {
return new Promise((res, rej) => {
   setTimeout(() => res(n), 250);
 });
}

Moins d’indentation #

Avec des promesses :

function doTheThing() {
    return doOneThing()
        .then(doAnother)
        .then(doSomeMore)
        .catch(handleErrors)
}

Avec les fonctions asynchrones :

async function doTheThing() {
    try {
        const one = await doOneThing();
        const another = await doAnother(one);
        return await doSomeMore(another);
    } catch (err) {
        handleErrors(err);
    }
}

Notez que le retour est en bas, et non en haut, et que vous utilisez les mécanismes natifs de gestion des erreurs du langage (try/catch).

Opérations asynchrones (parallèles) simultanées #

Souvent, vous souhaiterez effectuer des opérations asynchrones en parallèle. Il existe une syntaxe directe qui prend en charge cela dans la proposition async/await, mais puisque wait attendra une promesse, vous pouvez regrouper plusieurs promesses ensemble dans Promise.all pour les attendre :

// Not in parallel

async function getFriendPosts(user) {
    friendIds = await db.get("friends", {user}, {id: 1});
    friendPosts = [];
    for (let id in friendIds) {
        friendPosts = friendPosts.concat( await db.get("posts", {user: id}) );
    }
    // etc.
}

Cela effectuera chaque requête pour obtenir les publications de chaque ami en série, mais elles peuvent être effectuées simultanément :

// In parallel

async function getFriendPosts(user) {
    friendIds = await.db.get("friends", {user}, {id: 1});
    friendPosts = await Promise.all( friendIds.map(id => 
      db.get("posts", {user: id})
    );
    // etc.
}

Cela bouclera sur la liste des identifiants pour créer un tableau de promesses. wait attendra que toutes les promesses soient terminées. Promise.all les combine en une seule promesse, mais elles sont faites en parallèle.
