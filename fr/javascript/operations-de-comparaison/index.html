Lors de l’utilisation de la coercition booléenne, les valeurs suivantes sont considérées “fausses” :

  • “faux”
  • ‘0’
  • "" (chaîne vide)
  • “nul”
  • ‘indéfini’
  • NaN (pas un nombre, par exemple 0/0)
  • document.all¹ (contexte du navigateur)

Tout le reste est considéré “véridique”.

¹ violation volontaire de la spécification ECMAScript

Égalité/inégalité abstraite et conversion de type #

Le problème #

Les opérateurs abstraits d’égalité et d’inégalité (== et !=) convertissent leurs opérandes si les types d’opérandes ne correspondent pas. Cette coercition de type est une source courante de confusion sur les résultats de ces opérateurs, en particulier, ces opérateurs ne sont pas toujours transitifs comme on pourrait s’y attendre.

"" ==  0;     // true A
 0 == "0";    // true A
"" == "0";    // false B
false == 0;   // true
false == "0"; // true

"" !=  0;     // false A
 0 != "0";    // false A
"" != "0";    // true B
false != 0;   // false
false != "0"; // false

Les résultats commencent à avoir un sens si vous considérez comment JavaScript convertit les chaînes vides en nombres.

Number("");    // 0
Number("0");   // 0
Number(false); // 0

La solution #

Dans l’instruction false B, les deux opérandes sont des chaînes ("" et "0"), il n’y aura donc aucune conversion de type et puisque "" et "0" sont pas la même valeur, "" == "0" est false comme prévu.

Une façon d’éliminer les comportements inattendus consiste à s’assurer que vous comparez toujours les opérandes du même type. Par exemple, si vous souhaitez que les résultats de la comparaison numérique utilisent une conversion explicite :

var test = (a,b) => Number(a) == Number(b); 
test("", 0);        // true;
test("0", 0);       // true
test("", "0");      // true;
test("abc", "abc"); // false as operands are not numbers

Ou, si vous voulez une comparaison de chaînes :

var test = (a,b) => String(a) == String(b);
test("", 0);   // false;
test("0", 0);  // true
test("", "0"); // false;

Side-note : Number("0") et new Number("0") ne sont pas la même chose ! Pendant que le premier effectue une conversion de type, le second créera un nouvel objet. Les objets sont comparés par référence et non par valeur ce qui explique les résultats ci-dessous.

Number("0") == Number("0");         // true;
new Number("0") == new Number("0"); // false

Enfin, vous avez la possibilité d’utiliser des opérateurs d’égalité et d’inégalité stricts qui n’effectueront aucune conversion de type implicite.

"" ===  0;  // false
 0 === "0"; // false
"" === "0"; // false

Une référence supplémentaire à ce sujet peut être trouvée ici:

[Quel opérateur égal (== vs ===) doit être utilisé dans les comparaisons JavaScript ?][1].

[Égalité abstraite (==)][2]

[1] : http://stackoverflow.com/questions/359494/does-it-matter-which-equals-operator-vs-i-use-in-javascript-comparisons [2] : https://www.wikiod.com/fr/javascript/operations-de-comparaison#Égalité abstraite (==)

Propriété NaN de l’objet global #

NaN ("Not a Number") est une valeur spéciale définie par la [IEEE Standard for Floating-Point Arithmetic][1], qui est utilisée lorsque une valeur non numérique est fournie mais un nombre est attendu (1 * "deux"), ou lorsqu’un calcul n’a pas de résultat nombre valide (Math.sqrt(-1)).

Toute égalité ou comparaison relationnelle avec NaN renvoie false, même en la comparant à elle-même. Parce que NaN est censé désigner le résultat d’un calcul absurde, et en tant que tel, il n’est pas égal au résultat de tout autre calcul absurde.

(1 * "two") === NaN  //false

NaN === 0;          // false
NaN === NaN;        // false
Number.NaN === NaN; // false

NaN < 0;            // false
NaN > 0;            // false
NaN > 0;            // false
NaN >= NaN;         // false
NaN >= 'two';       // false

Les comparaisons non égales renverront toujours “true” :

NaN !== 0;          // true
NaN !== NaN;        // true

Vérifier si une valeur est NaN #

Vous pouvez tester une valeur ou une expression pour NaN en utilisant la fonction [Number.isNaN()][2] :

Number.isNaN(NaN);         // true
Number.isNaN(0 / 0);       // true
Number.isNaN('str' - 12);  // true

Number.isNaN(24);          // false
Number.isNaN('24');        // false
Number.isNaN(1 / 0);       // false
Number.isNaN(Infinity);    // false

Number.isNaN('str');       // false
Number.isNaN(undefined);   // false
Number.isNaN({});          // false

Vous pouvez vérifier si une valeur est “NaN” en la comparant à elle-même :

value !== value;    // true for NaN, false for any other value

Vous pouvez utiliser le polyfill suivant pour Number.isNaN() :

Number.isNaN = Number.isNaN || function(value) {     
    return value !== value;
}

En revanche, la fonction globale isNaN() renvoie true non seulement pour NaN, mais également pour toute valeur ou expression qui ne peut pas être convertie en nombre :

isNaN(NaN);         // true
isNaN(0 / 0);       // true
isNaN('str' - 12);  // true

isNaN(24);          // false
isNaN('24');        // false
isNaN(Infinity);    // false

isNaN('str');       // true
isNaN(undefined);   // true
isNaN({});          // true

ECMAScript définit un algorithme de “similitude” appelé SameValue qui, depuis ECMAScript 6, peut être invoqué avec Object.is. Contrairement à la comparaison == et ===, l’utilisation de Object.is() traitera NaN comme identique à lui-même (et -0 comme non identique à +0):

Object.is(NaN, NaN)      // true
Object.is(+0, 0)         // false

NaN === NaN              // false
+0 === 0                 // true

Vous pouvez utiliser le polyfill suivant pour Object.is() (à partir de [MDN](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/is#Polyfill_for_non- ES6_navigateurs)) :

if (!Object.is) {
  Object.is = function(x, y) {
    // SameValue algorithm
    if (x === y) { // Steps 1-5, 7-10
      // Steps 6.b-6.e: +0 != -0
      return x !== 0 || 1 / x === 1 / y;
    } else {
      // Step 6.a: NaN == NaN
      return x !== x && y !== y;
    }
  };
}

Points à noter #

NaN lui-même est un nombre, ce qui signifie qu’il n’est pas égal à la chaîne “NaN”, et le plus important (bien que peut-être involontairement):

typeof(NaN) === "number"; //true

[1] : https://en.wikipedia.org/wiki/IEEE_floating_point [2] : https://www.wikiod.com/fr/javascript/constantes-integrees#Test de NaN à l’aide de isNaN()

Court-circuit dans les opérateurs booléens #

L’opérateur and (&&) et l’opérateur or (||) utilisent un court-circuit pour éviter un travail inutile si le résultat de l’opération ne change pas avec le travail supplémentaire.

Dans x && y, y ne sera pas évalué si x est évalué à false, car l’expression entière est garantie comme étant false.

Dans x || y, y ne sera pas évalué si x est évalué à true, car l’expression entière est garantie comme étant true.

Exemple avec fonctions

Prenez les deux fonctions suivantes :

function T() { // True
  console.log("T");
  return true;
}

function F() { // False
  console.log("F");
  return false;
}

Exemple 1

T() && F(); // false

Production:

‘T’
‘F’

Exemple 2

F() && T(); // false

Production:

‘F’

Exemple 3

T() || F(); // true

Production:

‘T’

Exemple 4

F() || T(); // true

Production:

‘F’
‘T’

** Court-circuit pour éviter les erreurs **

var obj; // object has value of undefined
if(obj.property){ }// TypeError: Cannot read property 'property' of undefined
if(obj.property && obj !== undefined){}// Line A TypeError: Cannot read property 'property' of undefined

Ligne A: si vous inversez l’ordre, la première instruction conditionnelle empêchera l’erreur sur la seconde en ne l’exécutant pas si elle renverrait l’erreur

if(obj !== undefined && obj.property){}; // no error thrown

Mais ne doit être utilisé que si vous vous attendez à “undefined”

if(typeof obj === "object" && obj.property){}; // safe option but slower

Court-circuit pour fournir une valeur par défaut

L’opérateur || peut être utilisé pour sélectionner soit une valeur “véridique”, soit la valeur par défaut.

Par exemple, cela peut être utilisé pour s’assurer qu’une valeur nullable est convertie en une valeur non nullable :

var nullableObj = null;
var obj = nullableObj || {};  // this selects {}

var nullableObj2 = {x: 5};
var obj2 = nullableObj2 || {} // this selects {x: 5}

Ou pour retourner la première valeur de vérité

var truthyValue = {x: 10};
return truthyValue || {}; // will return {x: 10}

La même chose peut être utilisée pour se replier plusieurs fois :

envVariable || configValue || defaultConstValue // select the first "truthy" of these

Court-circuit pour appeler une fonction optionnelle

L’opérateur && peut être utilisé pour évaluer un rappel, uniquement s’il est passé :

function myMethod(cb) {
    // This can be simplified
    if (cb) {
       cb();
    }

    // To this
    cb && cb();
}

Bien sûr, le test ci-dessus ne valide pas que cb est en fait une fonction et pas seulement un Object/Array/String/Number.

Égalité abstraite (==) #

Les opérandes de l’opérateur d’égalité abstrait sont comparés après avoir été convertis en un type commun. La façon dont cette conversion se produit est basée sur la spécification de l’opérateur :

Spécification pour l’opérateur == :

7.2.13 Comparaison d'égalité abstraite

La comparaison `x == y`, où `x` et `y` sont des valeurs, produit `true` ou `false`. Une telle comparaison s'effectue comme suit : 1. Si `Type(x)` est identique à `Type(y)`, alors : - **a.** Renvoie le résultat de la comparaison d'égalité stricte `x === y`. 2. Si `x` est `null` et `y` est `undefined`, retourne `true`. 3. Si `x` est `undefined` et `y` est `null`, retourne `true`. 4. Si `Type(x)` est `Number` et `Type(y)` est `String`, renvoie le résultat de la comparaison `x == ToNumber(y)`. 5. Si `Type(x)` est `String` et `Type(y)` est `Number`, renvoie le résultat de la comparaison `ToNumber(x) == y`. 6. Si `Type(x)` est `Boolean`, renvoie le résultat de la comparaison `ToNumber(x) == y`. 7. Si `Type(y)` est `Boolean`, renvoie le résultat de la `comparaison x == ToNumber(y)`. 8. Si `Type(x)` est soit `String`, `Number` ou `Symbol` et `Type(y)` est `Object`, renvoie le résultat de la comparaison `x == ToPrimitive(y)`. 9. Si `Type(x)` est Object et `Type(y)` est soit `String`, `Number` ou `Symbol`, renvoie le résultat de la comparaison `ToPrimitive(x) == y`. 10. Renvoie "faux".

Exemples: #

1 == 1;                     // true
1 == true;                  // true  (operand converted to number: true => 1)
1 == '1';                   // true  (operand converted to number: '1' => 1 )
1 == '1.00';                // true
1 == '1.00000000001';       // false
1 == '1.00000000000000001'; // true  (true due to precision loss)
null == undefined;          // true  (spec #2)
1 == 2;                     // false
0 == false;                 // true
0 == undefined;             // false
0 == "";                    // true

Null et indéfini #

Les différences entre null et undefined #

null et undefined partagent l’égalité abstraite == mais pas l’égalité stricte ===,

null == undefined   // true
null === undefined  // false

Ils représentent des choses légèrement différentes :

  • undefined représente l’absence de valeur, par exemple avant la création d’une propriété d’identifiant/d’objet ou dans la période entre la création d’un identifiant/paramètre de fonction et sa première définition, le cas échéant.
  • null représente l’absence intentionnelle de valeur pour un identifiant ou une propriété déjà créée.

Ce sont différents types de syntaxe :

  • undefined est une propriété de l’Objet global, généralement immuable dans la portée globale. Cela signifie que partout où vous pouvez définir un identifiant autre que dans l’espace de noms global, vous pouvez masquer undefined de cette portée (bien que les choses puissent toujours ** être ** undefined)
  • null est un mot littéral, donc sa signification ne peut jamais être changée et tenter de le faire lancera une erreur.

Les similitudes entre null et undefined #

null et undefined sont tous deux faux.

if (null) console.log("won't be logged");
if (undefined) console.log("won't be logged");

Ni null ou undefined est égal à false (voir [cette question][1]).

false == undefined   // false
false == null        // false
false === undefined  // false
false === null       // false

Utilisation de undefined #

  • Si la portée actuelle n’est pas fiable, utilisez quelque chose qui évalue undefined, par exemple void 0;.
  • Si undefined est occulté par une autre valeur, c’est aussi mauvais que d’occulter Array ou Number.
  • Évitez de définir quelque chose comme “indéfini”. Si vous voulez supprimer une propriété bar d’un Object foo, delete foo.bar; à la place.
  • L’identificateur de test d’existence foo contre undefined pourrait générer une erreur de référence, utilisez typeof foo contre "undefined" à la place.

Opérateurs logiques avec booléens #

var x = true,
    y = false;

ET #

Cet opérateur renverra true si les deux expressions ont la valeur true. Cet opérateur booléen utilisera le court-circuit et n’évaluera pas ‘y’ si ‘x’ est évalué à ‘false’.

x && y;

Cela renverra faux, car y est faux.

OU #

Cet opérateur renverra true si l’une des deux expressions est évaluée à true. Cet opérateur booléen utilisera le court-circuit et ‘y’ ne sera pas évalué si ‘x’ est évalué à ’true’.

x || y;

Cela renverra vrai, car x est vrai.

NE PAS #

Cet opérateur renverra faux si l’expression de droite est évaluée à vrai et renverra vrai si l’expression de droite est évaluée à faux.

!x;

Cela renverra faux, car x est vrai.

Opérateurs logiques avec des valeurs non booléennes (coercition booléenne) #

Le OU logique (||), lisant de gauche à droite, sera évalué à la première valeur vérité. Si aucune valeur truthy n’est trouvée, la dernière valeur est renvoyée.

var a = 'hello' || '';             // a = 'hello'
var b = '' || [];                  // b = []
var c = '' || undefined;           // c = undefined
var d = 1 || 5;                    // d = 1
var e = 0 || {};                   // e = {}
var f = 0 || '' || 5;              // f = 5
var g = '' || 'yay' || 'boo';      // g = 'yay'

Le ET logique (&&), lu de gauche à droite, sera évalué à la première valeur falsy. Si aucune valeur falsey n’est trouvée, la dernière valeur est renvoyée.

var a = 'hello' && '';                  // a = ''
var b = '' && [];                       // b = ''
var c = undefined && 0;                 // c = undefined
var d = 1 && 5;                         // d = 5
var e = 0 && {};                        // e = 0
var f = 'hi' && [] && 'done';           // f = 'done'
var g = 'bye' && undefined && 'adios';  // g = undefined

Cette astuce peut être utilisée, par exemple, pour définir une valeur par défaut pour un argument de fonction (avant ES6).

var foo = function(val) {
    // if val evaluates to falsey, 'default' will be returned instead.
    return val || 'default';
}

console.log( foo('burger') );  // burger
console.log( foo(100) );       // 100
console.log( foo([]) );        // []
console.log( foo(0) );         // default
console.log( foo(undefined) ); // default

Gardez simplement à l’esprit que pour les arguments, 0 et (dans une moindre mesure) la chaîne vide sont également souvent des valeurs valides qui devraient pouvoir être transmises explicitement et remplacer une valeur par défaut, ce qui, avec ce modèle, ils ne le seront pas ( parce qu’ils sont faux).

## Conversions de types automatiques Attention, les nombres peuvent être accidentellement convertis en chaînes ou en NaN (Not a Number).

JavaScript est faiblement typé. Une variable peut contenir différents types de données et une variable peut changer de type de données :

var x = "Hello";     // typeof x is a string
x = 5;               // changes typeof x to a number

Lors d’opérations mathématiques, JavaScript peut convertir des nombres en chaînes :

var x = 5 + 7;       // x.valueOf() is 12,  typeof x is a number
var x = 5 + "7";     // x.valueOf() is 57,  typeof x is a string
var x = "5" + 7;     // x.valueOf() is 57,  typeof x is a string
var x = 5 - 7;       // x.valueOf() is -2,  typeof x is a number
var x = 5 - "7";     // x.valueOf() is -2,  typeof x is a number
var x = "5" - 7;     // x.valueOf() is -2,  typeof x is a number
var x = 5 - "x";     // x.valueOf() is NaN, typeof x is a number

La soustraction d’une chaîne à une chaîne ne génère pas d’erreur mais renvoie NaN (Pas un nombre) :

"Hello" - "Dolly"    // returns NaN

Opérations de comparaison d’égalité #

JavaScript a quatre opérations de comparaison d’égalité différentes.

[Même valeur][1] #

Il renvoie “true” si les deux opérandes appartiennent au même type et ont la même valeur.

Remarque : la valeur d’un objet est une référence.

Vous pouvez utiliser cet algorithme de comparaison via Object.is (ECMAScript 6).

Exemples:

Object.is(1, 1);            // true
Object.is(+0, -0);          // false
Object.is(NaN, NaN);        // true
Object.is(true, "true");    // false
Object.is(false, 0);        // false
Object.is(null, undefined); // false
Object.is(1, "1");          // false
Object.is([], []);          // false

Cet algorithme a les propriétés d’une [relation d’équivalence][2] :

  • [Réflexivité][3] : Object.is(x, x) est true, pour toute valeur x
  • [Symétrie][4] : Object.is(x, y) est true si, et seulement si, Object.is(y, x) est true, pour toutes les valeurs x et ‘y’.
  • [Transitivité][4] : Si Object.is(x, y) et Object.is(y, z) sont true, alors Object.is(x, z) est également true , pour toutes les valeurs x, y et z.

[MêmeValeurZéro][5] #

Il se comporte comme SameValue, mais considère que +0 et -0 sont égaux.

Vous pouvez utiliser cet algorithme de comparaison via Array.prototype.includes (ECMAScript 7).

Exemples:

[1].includes(1);            // true
[+0].includes(-0);          // true
[NaN].includes(NaN);        // true
[true].includes("true");    // false
[false].includes(0);        // false
[1].includes("1");          // false
[null].includes(undefined); // false
[[]].includes([]);          // false

Cet algorithme a toujours les propriétés d’une [relation d’équivalence][2] :

  • [Réflexivité][3] : [x].includes(x) est true, pour toute valeur x
  • [Symmetry][4] : [x].includes(y) est true si, et seulement si, [y].includes(x) est true, pour toutes les valeurs x et ‘y’.
  • [Transitivité][4] : Si [x].includes(y) et [y].includes(z) sont true, alors [x].includes(z) est également true , pour toutes les valeurs x, y et z.

[** Comparaison d’égalité stricte **] [6] #

Il se comporte comme SameValue, mais

  • Considère que +0 et -0 sont égaux.
  • Considère NaN différent de toute valeur, y compris elle-même

Vous pouvez utiliser cet algorithme de comparaison via l’opérateur === (ECMAScript 3).

Il existe également l’opérateur !== (ECMAScript 3), qui annule le résultat de ===.

Exemples:

1 === 1;            // true
+0 === -0;          // true
NaN === NaN;        // false
true === "true";    // false
false === 0;        // false
1 === "1";          // false
null === undefined; // false
[] === [];          // false

Cet algorithme a les propriétés suivantes :

  • [Symétrie][4] : x === y est true si, et seulement si, y === xesttrue, pour toutes les valeurs xety`.
  • [Transitivité][4] : Si x === y et y === z sont true, alors x === z est également true, pour toute valeur x, ‘y’ et ‘z’.

Mais n’est pas une [relation d’équivalence][2] car

  • NaN n’est pas [réflexif][3] : NaN !== NaN

[Comparaison abstraite de l’égalité][7] #

Si les deux opérandes appartiennent au même type, il se comporte comme la comparaison d’égalité stricte.

Sinon, il les contraint comme suit :

  • undefined et null sont considérés comme égaux
  • Lors de la comparaison d’un nombre avec une chaîne, la chaîne est convertie en un nombre
  • Lors de la comparaison d’un booléen avec quelque chose d’autre, le booléen est contraint à un nombre
  • Lors de la comparaison d’un objet avec un nombre, une chaîne ou un symbole, l’objet est contraint à une primitive

S’il y a eu une coercition, les valeurs contraintes sont comparées de manière récursive. Sinon, l’algorithme renvoie “faux”.

Vous pouvez utiliser cet algorithme de comparaison via l’opérateur == (ECMAScript 1).

Il existe également l’opérateur != (ECMAScript 1), qui annule le résultat de ==.

Exemples:

1 == 1;            // true
+0 == -0;          // true
NaN == NaN;        // false
true == "true";    // false
false == 0;        // true
1 == "1";          // true
null == undefined; // true
[] == [];          // false

Cet algorithme a la propriété suivante :

  • [Symétrie][4] : x == y est true si, et seulement si, y == x est true, pour toutes les valeurs x et y.

Mais n’est pas une [relation d’équivalence][2] car

  • NaN n’est pas [réflexif][3] : NaN != NaN
  • [Transitivité][4] ne tient pas, par ex. 0 == '' et 0 == '0', mais '' != '0'

Opérateurs relationnels (<, <=, >, >=) #

Lorsque les deux opérandes sont numériques, ils sont comparés normalement :

1 < 2        // true
2 <= 2       // true
3 >= 5       // false
true < false // false (implicitly converted to numbers, 1 > 0)

Lorsque les deux opérandes sont des chaînes, ils sont comparés lexicographiquement (selon l’ordre alphabétique) :

'a' < 'b'    // true
'1' < '2'    // true
'100' > '12' // false ('100' is less than '12' lexicographically!)

Lorsqu’un opérande est une chaîne et l’autre un nombre, la chaîne est convertie en nombre avant la comparaison :

'1' < 2      // true
'3' > 2      // true
true > '2'   // false (true implicitly converted to number, 1 < 2)

Lorsque la chaîne n’est pas numérique, la conversion numérique renvoie “NaN” (pas un nombre). La comparaison avec NaN renvoie toujours false :

1 < 'abc'    // false
1 > 'abc'    // false

Mais soyez prudent lorsque vous comparez une valeur numérique avec null, undefined ou des chaînes vides :

1 > ''        // true
1 < ''        // false
1 > null      // true
1 < null      // false
1 > undefined // false
1 < undefined // false

Lorsqu’un opérande est un objet et que l’autre est un nombre, l’objet est converti en nombre avant la comparaison. Ainsi, ’null’ est un cas particulier car ‘Number(null);//0’

new Date(2015)< 1479480185280            // true
null > -1                                //true
({toString:function(){return 123}}) > 122  //true

Inégalité #

L’opérateur != est l’inverse de l’opérateur ==. Retournera “true” si les opérandes ne sont pas égaux. Le moteur javascript essaiera de convertir les deux opérandes en types correspondants s’ils ne sont pas du même type. Remarque : si les deux opérandes ont des références internes différentes en mémoire, alors “false” sera renvoyé.

Goûter:

1 != '1'     // false
1 != 2       // true

Dans l’exemple ci-dessus, 1 != '1' est false car un type de nombre primitif est comparé à une valeur char. Par conséquent, le moteur Javascript ne se soucie pas du type de données de la valeur R.H.S.

Opérateur : !== est l’inverse de l’opérateur ===. Retournera vrai si les opérandes ne sont pas égaux ou si leurs types ne correspondent pas.

Exemple:

1 !== '1'    // true
1 !== 2      // true
1 !== 1      // false

Regroupement de plusieurs instructions logiques #

Vous pouvez regrouper plusieurs instructions logiques booléennes entre parenthèses afin de créer une évaluation logique plus complexe, particulièrement utile dans les instructions if.

if ((age >= 18 && height >= 5.11) || (status === 'royalty' && hasInvitation)) {
  console.log('You can enter our club');
}

Nous pourrions également déplacer la logique groupée vers des variables pour rendre la déclaration un peu plus courte et descriptive :

var isLegal = age >= 18;
var tall = height >= 5.11;
var suitable = isLegal && tall;
var isRoyalty = status === 'royalty';
var specialCase = isRoyalty && hasInvitation;
var canEnterOurBar = suitable || specialCase;

if (canEnterOurBar) console.log('You can enter our club');

Notez que dans cet exemple particulier (et bien d’autres), regrouper les déclarations avec des parenthèses fonctionne de la même manière que si nous les supprimions, suivez simplement une évaluation logique linéaire et vous vous retrouverez avec le même résultat. Je préfère utiliser des parenthèses car cela me permet de comprendre plus clairement ce que je voulais et pourrait éviter les erreurs de logique.

Champs de bits pour optimiser la comparaison des données multi-états #

Un champ de bits est une variable qui contient divers états booléens sous forme de bits individuels. Un peu on représenterait vrai, et off serait faux. Dans le passé, les champs de bits étaient couramment utilisés car ils économisaient de la mémoire et réduisaient la charge de traitement. Bien que la nécessité d’utiliser un champ de bits ne soit plus si importante, ils offrent certains avantages qui peuvent simplifier de nombreuses tâches de traitement.

Par exemple l’entrée de l’utilisateur. Lorsque vous obtenez une entrée à partir des touches de direction d’un clavier haut, bas, gauche, droite, vous pouvez encoder les différentes touches dans une seule variable avec chaque direction affectée d’un bit.

Exemple de clavier de lecture via bitfield

var bitField = 0;  // the value to hold the bits
const KEY_BITS = [4,1,8,2]; // left up right down
const KEY_MASKS = [0b1011,0b1110,0b0111,0b1101]; // left up right down
window.onkeydown = window.onkeyup = function (e) {
    if(e.keyCode >= 37 && e.keyCode <41){
        if(e.type === "keydown"){
            bitField |= KEY_BITS[e.keyCode - 37];
        }else{
            bitField &= KEY_MASKS[e.keyCode - 37];
        }
    }    
}

Exemple de lecture sous forme de tableau

var directionState = [false,false,false,false];
window.onkeydown = window.onkeyup = function (e) {
    if(e.keyCode >= 37 && e.keyCode <41){
        directionState[e.keyCode - 37] = e.type === "keydown";
    }    
}

Pour activer un bit, utilisez ou au niveau du bit | et la valeur correspondant au bit. Donc, si vous souhaitez définir le 2ème bit bitField |= 0b10 l’activera. Si vous souhaitez désactiver un peu, utilisez * et * & au niveau du bit avec une valeur qui a tout par le bit requis activé. Utilisation de 4 bits et désactivation du 2e bit bitfield &= 0b1101;

Vous pouvez dire que l’exemple ci-dessus semble beaucoup plus complexe que d’assigner les différents états de clé à un tableau. Oui C’est un peu plus complexe à paramétrer mais l’avantage vient lors de l’interrogation de l’état.

Si vous voulez tester si toutes les clés sont en place.

// as bit field
if(!bitfield) // no keys are on

// as array test each item in array
if(!(directionState[0] && directionState[1] && directionState[2] && directionState[3])){

Vous pouvez définir des constantes pour faciliter les choses

// postfix U,D,L,R for Up down left right
const KEY_U = 1;
const KEY_D = 2;
const KEY_L = 4;
const KEY_R = 8;
const KEY_UL = KEY_U + KEY_L; // up left
const KEY_UR = KEY_U + KEY_R; // up Right
const KEY_DL = KEY_D + KEY_L; // down left
const KEY_DR = KEY_D + KEY_R; // down right

Vous pouvez ensuite tester rapidement de nombreux états de clavier différents

if ((bitfield & KEY_UL) === KEY_UL) { // is UP and LEFT only down
if (bitfield  & KEY_UL) {             // is Up left down 
if ((bitfield & KEY_U) === KEY_U) {   // is Up only down
if (bitfield & KEY_U) {               // is Up down (any other key may be down)
if (!(bitfield & KEY_U)) {            // is Up up (any other key may be down)
if (!bitfield ) {                     // no keys are down
if (bitfield ) {                      // any one or more keys are down

La saisie au clavier n’est qu’un exemple. Les champs de bits sont utiles lorsque vous avez plusieurs états sur lesquels vous devez agir en combinaison. Javascript peut utiliser jusqu’à 32 bits pour un champ de bits. Leur utilisation peut offrir des augmentations de performances significatives. Ils valent la peine d’être connus.
