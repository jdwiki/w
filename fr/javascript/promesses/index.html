Syntaxe #

  • nouvelle promesse( /* fonction exécuteur : */ fonction(résolution, rejet) { })
  • promise.then(onFulfilled[, onRejected])
  • promesse.catch(onRejected)
  • Promise.resolve (résolution)
  • Promise.reject (raison)
  • Promise.all(itérable)
  • Promise.race (itérable)

Les promesses font partie de la spécification ECMAScript 2015 et [la prise en charge des navigateurs][1] est limitée, 88 % des navigateurs dans le monde la prenant en charge en juillet 2017. Le tableau suivant donne un aperçu des premières versions de navigateur prenant en charge les promesses.

Chrome Bord Firefox Internet Explorer Opéra Opéra Mini safari Safari iOS
32 12 27 x 19 x 7.1 8

Dans les environnements qui ne les prennent pas en charge, Promise peut être polyfill. Les bibliothèques tierces peuvent également fournir des fonctionnalités étendues, telles que la “promisification” automatisée des fonctions de rappel ou des méthodes supplémentaires telles que “progress”, également appelées “notify”.

Le site Web de la norme Promises/A+ fournit une [liste des implémentations conformes aux versions 1.0 et 1.1][4]. Les rappels de promesse basés sur la norme A+ sont toujours exécutés de manière asynchrone en tant que [microtâches dans la boucle d’événements][5].

[1] : http://caniuse.com/#feat=promises [4] : https://promisesaplus.com/implementations [5] : https://html.spec.whatwg.org/multipage/webappapis.html#task-queue

Introduction #

Un objet [Promise][mdn_promise_doc] représente une opération qui a produit ou produira éventuellement une valeur. Les promesses fournissent un moyen robuste d’encapsuler le résultat (éventuellement en attente) d’un travail asynchrone, atténuant le problème des rappels profondément imbriqués (appelés “[callback hell][callback_hell]”).

États et flux de contrôle #

Une promesse peut être dans l’un des trois états suivants :

  • en attente — L’opération sous-jacente n’est pas encore terminée et la promesse est en attente d’exécution.
  • fulfilled — L’opération est terminée et la promesse est fulfilled avec une value. Cela revient à renvoyer une valeur à partir d’une fonction synchrone.
  • rejected — Une erreur s’est produite pendant l’opération et la promesse est rejected avec une raison. Cela revient à lancer une erreur dans une fonction synchrone.

Une promesse est dite réglée (ou résolue) lorsqu’elle est soit tenue soit rejetée. Une fois qu’une promesse est établie, elle devient immuable et son état ne peut pas changer. Les méthodes [then][mdn_then_doc] et [catch][mdn_catch_doc] d’une promesse peuvent être utilisées pour attacher des rappels qui s’exécutent lorsqu’elle est réglée. Ces rappels sont appelés avec la valeur d’exécution et la raison du rejet, respectivement.

[![Organigramme de la promesse][flow_diagram]][flow_diagram]

Exemple #

const promise = new Promise((resolve, reject) => {
    // Perform some work (possibly asynchronous)
    // ...

    if (/* Work has successfully finished and produced "value" */) {
        resolve(value);
    } else {
        // Something went wrong because of "reason"
        // The reason is traditionally an Error object, although
        // this is not required or enforced.
        let reason = new Error(message);
        reject(reason);

        // Throwing an error also rejects the promise.
        throw reason;
    }
});

Les méthodes [then][mdn_then_doc] et [catch][mdn_catch_doc] peuvent être utilisées pour joindre des rappels d’exécution et de rejet :

promise.then(value => {
    // Work has completed successfully,
    // promise has been fulfilled with "value"
}).catch(reason => {
    // Something went wrong,
    // promise has been rejected with "reason"
});

Remarque : Appeler promise.then(...) et promise.catch(...) sur la même promesse peut entraîner une exception non interceptée dans Promise si une erreur se produit, soit lors de l’exécution la promesse ou à l’intérieur de l’un des rappels, la méthode préférée serait donc d’attacher le prochain écouteur à la promesse renvoyée par le précédent then / catch.

Alternativement, les deux rappels peuvent être attachés en un seul appel à [then][mdn_then_doc] :

promise.then(onFulfilled, onRejected);

Attacher des rappels à une promesse qui a déjà été réglée les placera immédiatement dans la [microtask queue][microtask_queue], et ils seront invoqués “dès que possible” (c’est-à-dire immédiatement après le script en cours d’exécution). Il n’est pas nécessaire de vérifier l’état de la promesse avant d’attacher des rappels, contrairement à de nombreuses autres implémentations émettrices d’événements.

Enchaînement des promesses #

La méthode [then][mdn_then_doc] d’une promesse renvoie une nouvelle promesse.

const promise = new Promise(resolve => setTimeout(resolve, 5000));

promise
    // 5 seconds later
    .then(() => 2)
    // returning a value from a then callback will cause
    // the new promise to resolve with this value
    .then(value => { /* value === 2 */ });

Le renvoi d’un [Promise][mdn_promise_doc] à partir d’un rappel [then][mdn_then_doc] l’ajoutera à la chaîne de promesse.

function wait(millis) {
    return new Promise(resolve => setTimeout(resolve, millis));
}

const p = wait(5000).then(() => wait(4000)).then(() => wait(1000));
p.then(() => { /* 10 seconds have passed */ });

Un [catch][mdn_catch_doc] permet à une promesse rejetée de récupérer, de la même manière que catch dans une instruction try/catch fonctionne. Tout [then][mdn_then_doc] chaîné après un [catch][mdn_catch_doc] exécutera son gestionnaire de résolution en utilisant la valeur résolue à partir de [catch][mdn_catch_doc].

const p = new Promise(resolve => {throw 'oh no'});
p.catch(() => 'oh yes').then(console.log.bind(console));  // outputs "oh yes"

S’il n’y a pas de gestionnaire [catch][mdn_catch_doc] ou reject au milieu de la chaîne, un [catch][mdn_catch_doc] à la fin capturera tout rejet dans la chaîne :

p.catch(() => Promise.reject('oh yes'))
  .then(console.log.bind(console))      // won't be called
  .catch(console.error.bind(console));  // outputs "oh yes"

A certaines occasions, vous souhaiterez peut-être “brancher” l’exécution des fonctions. Vous pouvez le faire en renvoyant différentes promesses à partir d’une fonction en fonction de la condition. Plus tard dans le code, vous pouvez fusionner toutes ces branches en une seule pour appeler d’autres fonctions sur celles-ci et/ou pour gérer toutes les erreurs en un seul endroit.

promise
    .then(result => {          
        if (result.condition) {
            return handlerFn1() 
                .then(handlerFn2);
        } else if (result.condition2) {
            return handlerFn3()
                .then(handlerFn4);
        } else {
            throw new Error("Invalid result");
        }
    })
    .then(handlerFn5)
    .catch(err => {
        console.error(err);
    });

Ainsi, l’ordre d’exécution des fonctions ressemble à :

promise --> handlerFn1 -> handlerFn2 --> handlerFn5 ~~> .catch()
         |                            ^
         V                            |
         -> handlerFn3 -> handlerFn4 -^

Le seul catch obtiendra l’erreur sur la branche où elle peut se produire.

[mdn_then_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/then [mdn_catch_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch [mdn_promise_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise

Attente de plusieurs promesses simultanées #

La méthode statique [Promise.all()][mdn_all_doc] accepte un itérable (par exemple, un Array) de promesses et renvoie une nouvelle promesse, qui se résout lorsque toutes les promesses de l’itérable ont été résolues, ou rejette si ** au moins une ** des promesses de l’itérable a été rejetée.

// wait "millis" ms, then resolve with "value"
function resolve(value, milliseconds) {
    return new Promise(resolve => setTimeout(() => resolve(value), milliseconds));
}

// wait "millis" ms, then reject with "reason"
function reject(reason, milliseconds) {
    return new Promise((_, reject) => setTimeout(() => reject(reason), milliseconds));
}

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    resolve(3, 7000)    
]).then(values => console.log(values)); // outputs "[1, 2, 3]" after 7 seconds.

Promise.all([
    resolve(1, 5000),
    reject('Error!', 6000),
    resolve(2, 7000)
]).then(values => console.log(values)) // does not output anything
.catch(reason => console.log(reason)); // outputs "Error!" after 6 seconds.

Les valeurs non promises dans l’itérable sont [“promisified”][promisifying].

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    { hello: 3 }
])
.then(values => console.log(values)); // outputs "[1, 2, { hello: 3 }]" after 6 seconds

La déstructuration de l’affectation peut aider à récupérer les résultats de plusieurs promesses.

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    resolve(3, 7000)
])
.then(([result1, result2, result3]) => {
    console.log(result1);
    console.log(result2);
    console.log(result3);
});

[mdn_all_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/all [promisifiant] : https://www.wikiod.com/fr/javascript/promesses#Des valeurs « prometteuses »

Réduire un tableau en promesses chaînées #

Ce modèle de conception est utile pour générer une séquence d’actions asynchrones à partir d’une liste d’éléments.

Il existe deux variantes :

  • la réduction « alors », qui construit une chaîne qui continue tant que la chaîne connaît le succès.
  • la réduction “catch”, qui construit une chaîne qui continue tant que la chaîne connaît une erreur.

La réduction “alors”

Cette variante du modèle construit une chaîne [.then()][mdn_then_doc] et peut être utilisée pour enchaîner des animations ou créer une séquence de requêtes HTTP dépendantes.

[1, 3, 5, 7, 9].reduce((seq, n) => {
    return seq.then(() => {
        console.log(n);
        return new Promise(res => setTimeout(res, 1000));
    });
}, Promise.resolve()).then(
    () => console.log('done'),
    (e) => console.log(e)
);
// will log 1, 3, 5, 7, 9, 'done' in 1s intervals

Explication:

  1. Nous appelons [.reduce()][mdn_reduce_doc] sur un tableau source et fournissons [Promise.resolve()][mdn_resolve_doc] comme valeur initiale.
  2. Chaque élément réduit ajoutera un [.then()][mdn_then_doc] à la valeur initiale.
  3. Le produit de [reduce()][mdn_reduce_doc] sera Promise.resolve().then(…).then(…).
  4. Nous ajoutons manuellement un [.then(successHandler, errorHandler)][mdn_then_doc] après la réduction, pour exécuter successHandler une fois toutes les étapes précédentes résolues. Si une étape échouait, alors errorHandler s’exécuterait.

Remarque : La réduction “then” est une contrepartie séquentielle de Promise.all().

La réduction “catch”

Cette variante du modèle construit une chaîne [.catch()][mdn_catch_doc] et peut être utilisée pour sonder séquentiellement un ensemble de serveurs Web pour une ressource en miroir jusqu’à ce qu’un serveur fonctionnel soit trouvé.

var working_resource = 5; // one of the values from the source array
[1, 3, 5, 7, 9].reduce((seq, n) => {
    return seq.catch(() => {
        console.log(n);
        if(n === working_resource) { // 5 is working
            return new Promise((resolve, reject) => setTimeout(() => resolve(n), 1000));
        } else { // all other values are not working
            return new Promise((resolve, reject) => setTimeout(reject, 1000));
        }
    });
}, Promise.reject()).then(
    (n) => console.log('success at: ' + n),
    () => console.log('total failure')
);
// will log 1, 3, 5, 'success at 5' at 1s intervals

Explication:

  1. Nous appelons [.reduce()][mdn_reduce_doc] sur un tableau source et fournissons [Promise.reject()][mdn_reject_doc] comme valeur initiale.
  2. Chaque élément réduit ajoutera un [.catch()][mdn_catch_doc] à la valeur initiale.
  3. Le produit de [reduce()][mdn_reduce_doc] sera Promise.reject().catch(...).catch(...).
  4. Nous ajoutons manuellement [.then(successHandler, errorHandler)][mdn_then_doc] après la réduction, pour exécuter successHandler une fois l’une des étapes précédentes résolue. Si toutes les étapes devaient échouer, alors errorHandler s’exécuterait.

Remarque : La réduction “catch” est une contrepartie séquentielle de Promise.any() (telle qu’implémentée dans bluebird.js, mais pas actuellement en ECMAScript natif).

[mdn_reduce_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/Reduce [mdn_resolve_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/resolve [mdn_reject_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/reject [mdn_then_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/then [mdn_catch_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch

Attente de la première de plusieurs promesses simultanées #

La méthode statique [Promise.race()][mdn_race_doc] accepte un itérable de promesses et renvoie une nouvelle promesse qui résout ou rejette dès que la première des promesses de l’itérable a été résolue ou rejetée.

// wait "milliseconds" milliseconds, then resolve with "value"
function resolve(value, milliseconds) {
    return new Promise(resolve => setTimeout(() => resolve(value), milliseconds));
}

// wait "milliseconds" milliseconds, then reject with "reason"
function reject(reason, milliseconds) {
    return new Promise((_, reject) => setTimeout(() => reject(reason), milliseconds));
}

Promise.race([
    resolve(1, 5000),
    resolve(2, 3000),
    resolve(3, 1000)
])
.then(value => console.log(value)); // outputs "3" after 1 second.


Promise.race([
    reject(new Error('bad things!'), 1000),
    resolve(2, 2000)
])
.then(value => console.log(value)) // does not output anything
.catch(error => console.log(error.message)); // outputs "bad things!" after 1 second

[mdn_race_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/race

Fonctions “prometteuses” avec callbacks #

Étant donné une fonction qui accepte un rappel de style Node,

fooFn(options, function callback(err, result) { ... });

vous pouvez le promettre (le convertir en une fonction basée sur des promesses) comme ceci :

function promiseFooFn(options) {
    return new Promise((resolve, reject) =>
        fooFn(options, (err, result) =>
            // If there's an error, reject; otherwise resolve
            err ? reject(err) : resolve(result)
        )
    );
}

Cette fonction peut alors être utilisée comme suit :

promiseFooFn(options).then(result => {
    // success!
}).catch(err => {
    // error!
});

De manière plus générique, voici comment promettre une fonction de style rappel donnée :

function promisify(func) {
    return function(...args) {
        return new Promise((resolve, reject) => {
            func(...args, (err, result) => err ? reject(err) : resolve(result));
        });
    }
}

Cela peut être utilisé comme ceci :

const fs = require('fs');
const promisedStat = promisify(fs.stat.bind(fs));

promisedStat('/foo/bar')
    .then(stat => console.log('STATE', stat))
    .catch(err => console.log('ERROR', err));

La gestion des erreurs #

Les erreurs générées par les promesses sont gérées par le deuxième paramètre (reject) passé à then ou par le gestionnaire passé à catch :

throwErrorAsync()
  .then(null, error => { /* handle error here */ });
// or
throwErrorAsync()
  .catch(error => { /* handle error here */ });

Chaînage #

Si vous avez une chaîne de promesses, une erreur entraînera l’omission des gestionnaires de “résolution” :

throwErrorAsync()
  .then(() => { /* never called */ })
  .catch(error => { /* handle error here */ });

La même chose s’applique à vos fonctions then. Si un gestionnaire resolve lève une exception, le prochain gestionnaire reject sera appelé :

doSomethingAsync()
  .then(result => { throwErrorSync(); })
  .then(() => { /* never called */ })
  .catch(error => { /* handle error from throwErrorSync() */ });

Un gestionnaire d’erreurs renvoie une nouvelle promesse, vous permettant de continuer une chaîne de promesses. La promesse renvoyée par le gestionnaire d’erreurs est résolue avec la valeur renvoyée par le gestionnaire :

throwErrorAsync()
  .catch(error => { /* handle error here */; return result; })
  .then(result => { /* handle result here */ });

Vous pouvez laisser une erreur tomber en cascade dans une chaîne de promesses en lançant à nouveau l’erreur :

throwErrorAsync()
  .catch(error => {
      /* handle error from throwErrorAsync() */
      throw error;
  })
  .then(() => { /* will not be called if there's an error */ })
  .catch(error => { /* will get called with the same error */ });

Il est possible de lever une exception qui n’est pas gérée par la promesse en enveloppant l’instruction throw dans un rappel setTimeout :

new Promise((resolve, reject) => {
  setTimeout(() => { throw new Error(); });
});

Cela fonctionne car les promesses ne peuvent pas gérer les exceptions lancées de manière asynchrone.

Rejets non gérés #

Une erreur sera silencieusement ignorée si une promesse n’a pas de bloc catch ou de gestionnaire reject :

throwErrorAsync()
  .then(() => { /* will not be called */ });
// error silently ignored

Pour éviter cela, utilisez toujours un bloc catch :

throwErrorAsync()
  .then(() => { /* will not be called */ })
  .catch(error => { /* handle error*/ });
// or
throwErrorAsync()
  .then(() => { /* will not be called */ }, error => { /* handle error*/ });

Vous pouvez également vous abonner à l’événement [unhandledrejection][mdn_unhandledrejection_doc] pour intercepter toutes les promesses rejetées non gérées :

window.addEventListener('unhandledrejection', event => {});

Certaines promesses peuvent gérer leur rejet plus tard que leur heure de création. L’événement [rejectionhandled][mdn_rejectionhandled_doc] est déclenché chaque fois qu’une telle promesse est gérée :

window.addEventListener('unhandledrejection', event => console.log('unhandled'));
window.addEventListener('rejectionhandled', event => console.log('handled'));
var p = Promise.reject('test');

setTimeout(() => p.catch(console.log), 1000);

// Will print 'unhandled', and after one second 'test' and 'handled'

L’argument “événement” contient des informations sur le rejet. event.reason est l’objet d’erreur et event.promise est l’objet de promesse qui a provoqué l’événement.

Dans Nodejs, les événements rejectionhandled et unhandledrejection sont appelés [rejectionHandled][node_rejectionhandled_doc] et [unhandledRejection][node_unhandledrejection_doc] sur process, respectivement, et ont une signature différente :

process.on('rejectionHandled', (reason, promise) => {});
process.on('unhandledRejection', (reason, promise) => {});

L’argument reason est l’objet d’erreur et l’argument promise est une référence à l’objet de promesse qui a provoqué le déclenchement de l’événement.

L’utilisation de ces événements unhandledrejection et rejectionhandled doit être envisagée uniquement à des fins de débogage. En règle générale, toutes les promesses doivent gérer leurs rejets.

Remarque : Actuellement, seuls Chrome 49+ et Node.js prennent en charge les événements unhandledrejection et rejectionhandled.

Mises en garde #

Chaînage avec fulfill et reject #

La fonction then(fulfill, rejet) (avec les deux paramètres non null) a un comportement unique et complexe, et ne devrait pas être utilisée à moins que vous ne sachiez exactement comment cela fonctionne.

La fonction fonctionne comme prévu si elle est définie sur “null” pour l’une des entrées :

// the following calls are equivalent
promise.then(fulfill, null) 
promise.then(fulfill)

// the following calls are also equivalent
promise.then(null, reject) 
promise.catch(reject)

Cependant, il adopte un comportement unique lorsque les deux entrées sont données :

// the following calls are not equivalent!
promise.then(fulfill, reject)
promise.then(fulfill).catch(reject)

// the following calls are not equivalent!
promise.then(fulfill, reject)
promise.catch(reject).then(fulfill)

La fonction then(fulfill, rejet) semble être un raccourci pour then(fulfill).catch(reject), mais ce n’est pas le cas et causera des problèmes si elle est utilisée de manière interchangeable. Un de ces problèmes est que le gestionnaire reject ne gère pas les erreurs du gestionnaire fulfill. Voici ce qui va se passer :

Promise.resolve() // previous promise is fulfilled
    .then(() => { throw new Error(); }, // error in the fulfill handler
        error => { /* this is not called! */ });

Le code ci-dessus entraînera une promesse rejetée car l’erreur est propagée. Comparez-le au code suivant, qui se traduit par une promesse tenue :

Promise.resolve() // previous promise is fulfilled
    .then(() => { throw new Error(); }) // error in the fulfill handler
    .catch(error => { /* handle error */ });

Un problème similaire existe lors de l’utilisation de then(fulfill, reflect) de manière interchangeable avec catch(reject).then(fulfill), sauf avec la propagation des promesses remplies au lieu des promesses rejetées.

Lancer de manière synchrone à partir d’une fonction qui devrait renvoyer une promesse #

Imaginez une fonction comme celle-ci :

function foo(arg) {
  if (arg === 'unexepectedValue') {
    throw new Error('UnexpectedValue')
  }

  return new Promise(resolve => 
    setTimeout(() => resolve(arg), 1000)
  )
}

Si une telle fonction est utilisée au milieu d’une chaîne de promesses, il n’y a apparemment aucun problème :

makeSomethingAsync().
  .then(() => foo('unexpectedValue'))
  .catch(err => console.log(err)) // <-- Error: UnexpectedValue will be caught here

Cependant, si la même fonction est appelée en dehors d’une chaîne de promesses, alors l’erreur ne sera pas gérée par elle et sera renvoyée à l’application :

foo('unexpectedValue') // <-- error will be thrown, so the application will crash
  .then(makeSomethingAsync) // <-- will not run
  .catch(err => console.log(err)) // <-- will not catch

Il existe 2 solutions de contournement possibles :

Renvoie une promesse rejetée avec l’erreur #

Au lieu de lancer, procédez comme suit :

function foo(arg) {
  if (arg === 'unexepectedValue') {
    return Promise.reject(new Error('UnexpectedValue'))
  }

  return new Promise(resolve => 
    setTimeout(() => resolve(arg), 1000)
  )
}

Enveloppez votre fonction dans une chaîne de promesses #

Votre instruction “throw” sera correctement interceptée lorsqu’elle se trouve déjà dans une chaîne de promesses :

function foo(arg) {
  return Promise.resolve()
    .then(() => {
      if (arg === 'unexepectedValue') {
        throw new Error('UnexpectedValue')
      }

      return new Promise(resolve => 
        setTimeout(() => resolve(arg), 1000)
      )
    })
}

[mdn_rejectionhandled_doc] : https://developer.mozilla.org/en-US/docs/Web/Events/rejectionhandled [mdn_unhandledrejection_doc] : https://developer.mozilla.org/en-US/docs/Web/Events/unhandledrejection [node_rejectionhandled_doc] : https://nodejs.org/api/process.html#process_event_rejectionhandled [node_unhandledrejection_doc] : https://nodejs.org/api/process.html#process_event_unhandledrejection

Réconcilier les opérations synchrones et asynchrones #

Dans certains cas, vous souhaiterez peut-être encapsuler une opération synchrone dans une promesse pour éviter la répétition dans les branches de code. Prenons cet exemple :

if (result) { // if we already have a result
  processResult(result); // process it
} else {
  fetchResult().then(processResult);
}

Les branches synchrones et asynchrones du code ci-dessus peuvent être réconciliées en enveloppant de manière redondante l’opération synchrone dans une promesse :

var fetch = result
  ? Promise.resolve(result)
  : fetchResult();

fetch.then(processResult);

Lors de la mise en cache du résultat d’un appel asynchrone, il est préférable de mettre en cache la promesse plutôt que le résultat lui-même. Cela garantit qu’une seule opération asynchrone est nécessaire pour résoudre plusieurs demandes parallèles.

Des précautions doivent être prises pour invalider les valeurs mises en cache lorsque des conditions d’erreur sont rencontrées.

// A resource that is not expected to change frequently
var planets = 'http://swapi.co/api/planets/';
// The cached promise, or null
var cachedPromise;

function fetchResult() {
    if (!cachedPromise) {
        cachedPromise = fetch(planets)
            .catch(function (e) {
                // Invalidate the current result to retry on the next fetch
                cachedPromise = null;
                // re-raise the error to propagate it to callers
                throw e;
            });
    }
    return cachedPromise;
}

Retarder l’appel de la fonction #

La méthode [setTimeout()][mdn_settimeout_doc] appelle une fonction ou évalue une expression après un nombre spécifié de millisecondes. C’est aussi un moyen trivial de réaliser une opération asynchrone.

Dans cet exemple, l’appel de la fonction wait résout la promesse après le temps spécifié en premier argument :

function wait(ms) {
    return new Promise(resolve => setTimeout(resolve, ms));  
}

wait(5000).then(() => { 
    console.log('5 seconds have passed...');
});

[mdn_settimeout_doc] : https://developer.mozilla.org/en-US/docs/Web/API/WindowTimers/setTimeout

Valeurs “prometteuses” #

La méthode statique [Promise.resolve][mdn_resolve_doc] peut être utilisée pour encapsuler des valeurs dans des promesses.

let resolved = Promise.resolve(2);
resolved.then(value => {
    // immediately invoked
    // value === 2
});

Si value est déjà une promesse, [Promise.resolve][mdn_resolve_doc] la refond simplement.

let one = new Promise(resolve => setTimeout(() => resolve(2), 1000));
let two = Promise.resolve(one);
two.then(value => {
    // 1 second has passed
    // value === 2
});

En fait, value peut être n’importe quel “thenable” (objet définissant une méthode then qui fonctionne suffisamment comme une promesse conforme aux spécifications). Cela permet à [Promise.resolve][mdn_resolve_doc] de convertir des objets tiers non approuvés en promesses tierces approuvées.

let resolved = Promise.resolve({
    then(onResolved) {
        onResolved(2);
    }
});
resolved.then(value => {
    // immediately invoked
    // value === 2
});

La méthode statique [Promise.reject][mdn_reject_doc] renvoie une promesse qui est immédiatement rejetée avec la raison donnée.

let rejected = Promise.reject("Oops!");
rejected.catch(reason => {
    // immediately invoked
    // reason === "Oops!"
});

[mdn_resolve_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/resolve [mdn_reject_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/reject

Utiliser ES2017 asynchrone/attendre #

Le même exemple ci-dessus, [Image loading][image_loading], peut être écrit en utilisant [async functions][1]. Cela permet également d’utiliser la méthode commune try/catch pour la gestion des exceptions.

Remarque : [en avril 2017, les versions actuelles de tous les navigateurs, sauf qu’Internet Explorer prend en charge les fonctions asynchrones][2].

function loadImage(url) {
    return new Promise((resolve, reject) => {
        const img = new Image();
        img.addEventListener('load', () => resolve(img));
        img.addEventListener('error', () => {
            reject(new Error(`Failed to load ${url}`));
        });
        img.src = url;
    });
}

(async () => {

    // load /image.png and append to #image-holder, otherwise throw error
    try {
        let img = await loadImage('http://example.com/image.png');
        document.getElementById('image-holder').appendChild(img);
    }
    catch (error) {
        console.error(error);
    }

})();

forEach avec des promesses #

Il est possible d’appliquer efficacement une fonction (cb) qui renvoie une promesse à chaque élément d’un tableau, chaque élément attendant d’être traité jusqu’à ce que l’élément précédent soit traité.

function promiseForEach(arr, cb) {
  var i = 0;

  var nextPromise = function () {
    if (i >= arr.length) {
      // Processing finished.
      return;
    }

    // Process next function. Wrap in `Promise.resolve` in case
    // the function does not return a promise
    var newPromise = Promise.resolve(cb(arr[i], i));
    i++;
    // Chain to finish processing.
    return newPromise.then(nextPromise);
  };

  // Kick off the chain.
  return Promise.resolve().then(nextPromise);
};

Cela peut être utile si vous devez traiter efficacement des milliers d’éléments, un à la fois. L’utilisation d’une boucle “for” régulière pour créer les promesses les créera toutes en même temps et occupera une quantité importante de RAM.

Effectuer un nettoyage avec finally() #

Il existe actuellement une [proposition][proposition] (qui ne fait pas encore partie de la norme ECMAScript) pour ajouter un rappel « enfin » aux promesses qui seront exécutées, que la promesse soit remplie ou rejetée. Sémantiquement, ceci est similaire à la clause [finally du bloc try [mdn_finally_doc].

Vous utiliserez généralement cette fonctionnalité pour le nettoyage :

var loadingData = true;

fetch('/data')
    .then(result => processData(result.data))
    .catch(error => console.error(error))
    .finally(() => {
        loadingData = false;
    });

Il est important de noter que le rappel “finally” n’affecte pas l’état de la promesse. Peu importe la valeur renvoyée, la promesse reste dans l’état rempli/rejeté qu’elle avait auparavant. Ainsi, dans l’exemple ci-dessus, la promesse sera résolu avec la valeur de retour de processData(result.data) même si le rappel finally a renvoyé undefined.

Le processus de normalisation étant toujours en cours, la mise en œuvre de vos promesses ne prendra probablement pas en charge les rappels “enfin” prêts à l’emploi. Pour les rappels synchrones, vous pouvez cependant ajouter cette fonctionnalité avec un polyfill :

if (!Promise.prototype.finally) {
    Promise.prototype.finally = function(callback) {
        return this.then(result => {
            callback();
            return result;
        }, error => {
            callback();
            throw error;
        });
    };
}

[proposition] : https://github.com/tc39/proposal-promise-finally [mdn_finally_doc] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/try…catch#The_finally_clause

Requête d’API asynchrone #

Ceci est un exemple d’un simple appel d’API “GET” enveloppé dans une promesse pour tirer parti de sa fonctionnalité asynchrone.

var get = function(path) {
  return new Promise(function(resolve, reject) {
    let request = new XMLHttpRequest();
    request.open('GET', path);
    request.onload = resolve;
    request.onerror = reject;
    request.send();
  });
};

Une gestion des erreurs plus robuste peut être effectuée à l’aide des fonctions [onload][onload] et [onerror][onerror] suivantes.

request.onload = function() {
  if (this.status >= 200 && this.status < 300) {
    if(request.response) {
      // Assuming a successful call returns JSON
      resolve(JSON.parse(request.response));
    } else {
      resolve();
  } else {
    reject({
      'status': this.status,
      'message': request.statusText
    });
  }
};

request.onerror = function() {
  reject({
    'status': this.status,
    'message': request.statusText
  });
};

[onload] : https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequestEventTarget/onload [onerror] : https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequestEventTarget/onerror
