Certains mots - appelés keywords - sont traités spécialement en JavaScript. Il existe une pléthore de différents types de mots-clés, et ils ont changé dans différentes versions du langage.

Identifiants et noms d’identifiants #

En ce qui concerne les mots réservés, il existe une petite distinction entre les “Identifiants” utilisés pour les noms de variables ou de fonctions et les “Noms d’identifiant” autorisés en tant que propriétés des types de données composites.

Par exemple, ce qui suit entraînera une erreur de syntaxe illégale :

var break = true;

Erreur de syntaxe non interceptée : rupture de jeton inattendue

Cependant, le nom est considéré comme valide en tant que propriété d’un objet (à partir d’ECMAScript 5+) :

var obj = {
    break: true
};
console.log(obj.break);

Pour citer [cette réponse](http://stackoverflow.com/questions/40209367/do-reserved-words-need-to-be-quoted-when-set-as-property-names-of-javascript-obj/ 40210179#40210179):

À partir de la Spécification du langage ECMAScript® 5.1 :

Article 7.6

Les noms d’identifiants sont des jetons qui sont interprétés selon la grammaire donnée dans la section « Identifiants » du chapitre 5 de la norme Unicode, avec quelques petites modifications. Un Identifier est un IdentifierName qui n’est pas un ReservedWord (voir 7.6.1) .

Syntaxe

Identifiant :: IdentifierName mais pas ReservedWord

Par spécification, un ReservedWord est :

Rubrique 7.6.1

Un mot réservé est un IdentifierName qui ne peut pas être utilisé comme Identifier.

MotRéservé :: Mot clé FutureReservedWord NullLiteral Littéral booléen

Cela inclut les mots-clés, les futurs mots-clés, “null” et les littéraux booléens. La liste complète des mots-clés se trouve dans les Sections 7.6.1 et les littéraux se trouvent dans la [Section 7.8](http : //www.ecma-international.org/ecma-262/5.1/#sec-7.8).

Ce qui précède (Section 7.6) implique que IdentifierNames peut être ReservedWords, et d’après la spécification pour [object initializers](http://www.ecma-international.org/ecma-262/5.1/#sec- 11.1.5):

Article 11.1.5

Syntaxe

ObjectLiteral : { } { PropertyNameAndValueList } { NomPropriétéEtListeValeur , }

Où PropertyName est, par spécification :

NomPropriété : NomIdentifiant StringLiteral NumericLiteral

Comme vous pouvez le voir, un PropertyName peut être un IdentifierName, permettant ainsi à ReservedWords d’être PropertyNames. Cela nous indique de manière concluante que, par spécification, il est permis d’avoir des ReservedWords tels que class et var comme PropertyNames sans guillemets, tout comme les littéraux de chaîne ou les littéraux numériques.

Pour en savoir plus, voir Section 7.6 - Noms d’identifiants et identifiants.

Remarque : le surligneur de syntaxe dans cet exemple a repéré le mot réservé et l’a toujours mis en surbrillance. Bien que l’exemple soit valide, les développeurs Javascript peuvent se faire prendre par certains outils de compilation / transpilation, linter et minifier qui soutiennent le contraire.
