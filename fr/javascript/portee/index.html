La portée est le contexte dans lequel les variables vivent et sont accessibles par un autre code dans la même portée. Étant donné que JavaScript peut être largement utilisé comme langage de programmation fonctionnel, il est important de connaître la portée des variables et des fonctions, car cela permet d’éviter les bogues et les comportements inattendus lors de l’exécution.

Fermetures #

Lorsqu’une fonction est déclarée, les variables dans le contexte de sa déclaration sont capturées dans sa portée. Par exemple, dans le code ci-dessous, la variable ‘x’ est liée à une valeur dans la portée externe, puis la référence à ‘x’ est capturée dans le contexte de ‘bar’ :

var x = 4; // declaration in outer scope

function bar() {
    console.log(x); // outer scope is captured on declaration
}

bar(); // prints 4 to console

Exemple de sortie : ‘4’

Ce concept de portée de “capture” est intéressant car nous pouvons utiliser et modifier des variables à partir d’une portée externe même après la sortie de la portée externe. Par exemple, considérez ce qui suit :

function foo() {
    var x = 4; // declaration in outer scope

    function bar() {
        console.log(x); // outer scope is captured on declaration
    }

    return bar;
    
    // x goes out of scope after foo returns
}

var barWithX = foo();
barWithX(); // we can still access x

Exemple de sortie : ‘4’

Dans l’exemple ci-dessus, lorsque foo est appelé, son contexte est capturé dans la fonction bar. Ainsi, même après son retour, bar peut toujours accéder et modifier la variable x. La fonction foo, dont le contexte est capturé dans une autre fonction, est dite une fermeture.

Données privées #

Cela nous permet de faire des choses intéressantes, comme définir des variables “privées” qui ne sont visibles que pour une fonction spécifique ou un ensemble de fonctions. Un exemple artificiel (mais populaire):

function makeCounter() {
    var counter = 0;

    return {
        value: function () {
            return counter;
        },
        increment: function () {
            counter++;
        }
    };
}

var a = makeCounter();
var b = makeCounter();

a.increment();

console.log(a.value());
console.log(b.value());

Exemple de sortie : <pré> 1 0

Lorsque makeCounter() est appelé, un instantané du contexte de cette fonction est enregistré. Tout le code à l’intérieur de makeCounter() utilisera cet instantané dans son exécution. Deux appels de makeCounter() créeront donc deux instantanés différents, avec leur propre copie de counter.

Expressions de fonction appelées immédiatement (IIFE) #

Les fermetures sont également utilisées pour empêcher la pollution globale de l’espace de noms, souvent par l’utilisation d’expressions de fonction immédiatement invoquées.

Les expressions de fonction immédiatement appelées (ou, peut-être plus intuitivement, les fonctions anonymes auto-exécutables) sont essentiellement des fermetures qui sont appelées juste après la déclaration. L’idée générale avec IIFE est d’invoquer l’effet secondaire de créer un contexte séparé qui n’est accessible qu’au code dans l’IIFE.

Supposons que nous voulions pouvoir référencer jQuery avec $. Considérez la méthode naïve, sans utiliser d’IIFE :

var $ = jQuery;
// we've just polluted the global namespace by assigning window.$ to jQuery

Dans l’exemple suivant, un IIFE est utilisé pour s’assurer que le $ est lié à jQuery uniquement dans le contexte créé par la fermeture :

(function ($) {
    // $ is assigned to jQuery here
})(jQuery);
// but window.$ binding doesn't exist, so no pollution

Voir la réponse canonique sur Stackoverflow pour plus d’informations sur les fermetures.

Levage #

Qu’est-ce que le levage ? #

Le levage est un mécanisme qui déplace toutes les déclarations de variables et de fonctions vers le haut de leur portée. Cependant, les affectations de variables se produisent toujours là où elles se trouvaient à l’origine.

Par exemple, considérez le code suivant :

console.log(foo);  // → undefined
var foo = 42;
console.log(foo);  // → 42

Le code ci-dessus est le même que :

var foo;             // → Hoisted variable declaration
console.log(foo);    // → undefined
foo = 42;            // → variable assignment remains in the same place
console.log(foo);    // → 42

Notez qu’en raison du levage, le “non défini” ci-dessus n’est pas le même que le “non défini” résultant de l’exécution :

console.log(foo);    // → foo is not defined

Un principe similaire s’applique aux fonctions. Lorsque des fonctions sont affectées à une variable (c’est-à-dire une [expression de fonction] [2]), la déclaration de la variable est hissée tandis que l’affectation reste au même endroit. Les deux extraits de code suivants sont équivalents.

console.log(foo(2, 3));     // → foo is not a function

var foo = function(a, b) {
    return a * b;
}

var foo;
console.log(foo(2, 3));     // → foo is not a function
foo = function(a, b) {
    return a * b;
}

Lors de la déclaration d’[instructions de fonction][3], un scénario différent se produit. Contrairement aux instructions de fonction, les déclarations de fonction sont hissées au sommet de leur portée. Considérez le code suivant :

console.log(foo(2, 3));  // → 6
function foo(a, b) {
    return a * b;
}

Le code ci-dessus est le même que l’extrait de code suivant en raison du levage :

function foo(a, b) {
    return a * b;
}

console.log(foo(2, 3));  // → 6

Voici quelques exemples de ce qui est et de ce qui ne l’est pas :

// Valid code:
foo();

function foo() {}

// Invalid code:
bar();                     // → TypeError: bar is not a function
var bar = function () {};


// Valid code:
foo();
function foo() {
    bar();
}
function bar() {}


// Invalid code:
foo();
function foo() {
    bar();                // → TypeError: bar is not a function
}
var bar = function () {};


// (E) valid:
function foo() {
    bar();
}
var bar = function(){};
foo();

Limites du levage #

L’initialisation d’une variable ne peut pas être Hoisted ou dans de simples déclarations JavaScript Hoists pas d’initialisation.

Par exemple : les scripts ci-dessous donneront différentes sorties.

var x = 2; 
var y = 4; 
alert(x + y);

Cela vous donnera une sortie de 6. Mais ceci…

var x = 2; 
alert(x + y);
var y = 4;

Cela vous donnera une sortie de NaN. Puisque nous initialisons la valeur de y, le levage JavaScript ne se produit pas, donc la valeur y sera indéfinie. Le JavaScript considérera que y n’est pas encore déclaré.

Ainsi, le deuxième exemple est le même que ci-dessous.

var x = 2; 
var y;
alert(x + y);
y = 4;

Cela vous donnera une sortie de NaN.

Différence entre var et let #

(Remarque : tous les exemples utilisant let sont également valables pour const)

var est disponible dans toutes les versions de JavaScript, tandis que let et const font partie d’ECMAScript 6 et uniquement disponible dans certains navigateurs plus récents.

var est limité à la fonction conteneur ou à l’espace global, selon le moment où il est déclaré :

var x = 4; // global scope

function DoThings() {
    var x = 7; // function scope
    console.log(x);
}

console.log(x); // >> 4
DoThings();     // >> 7
console.log(x); // >> 4

Cela signifie qu’il “échappe” aux instructions if et à toutes les constructions de bloc similaires :

var x = 4;
if (true) {
    var x = 7;
}
console.log(x); // >> 7

for (var i = 0; i < 4; i++) {
    var j = 10;
}
console.log(i); // >> 4
console.log(j); // >> 10

Par comparaison, ’let’ est limité au bloc :

let x = 4;

if (true) {
    let x = 7;
    console.log(x); // >> 7
}

console.log(x); // >> 4

for (let i = 0; i < 4; i++) {
    let j = 10;
}
console.log(i); // >> "ReferenceError: i is not defined"
console.log(j); // >> "ReferenceError: j is not defined"

Notez que i et j ne sont déclarés que dans la boucle for et ne sont donc pas déclarés en dehors de celle-ci.

Il existe plusieurs autres différences cruciales :

Déclaration de variable globale #

Dans la portée supérieure (en dehors des fonctions et des blocs), les déclarations var placent un élément dans l’objet global. let ne fait pas :

var x = 4;
let y = 7;

console.log(this.x); // >> 4
console.log(this.y); // >> undefined

Re-déclaration #

Déclarer une variable deux fois en utilisant var ne produit pas d’erreur (même si cela équivaut à la déclarer une fois):

var x = 4;
var x = 7;

Avec ’let’, cela produit une erreur :

let x = 4;
let x = 7;

TypeError : l’identifiant x a déjà été déclaré

Il en va de même lorsque ‘y’ est déclaré avec ‘var’ :

var y = 4;
let y = 7;

TypeError : l’identifiant y a déjà été déclaré

Cependant les variables déclarées avec let peuvent être réutilisées (et non re-déclarées) dans un bloc imbriqué

let i = 5;    
{
   let i = 6;
   console.log(i); // >> 6
}
console.log(i); // >> 5

Dans le bloc, le i externe est accessible, mais si le bloc interne a une déclaration let pour i, le i externe n’est pas accessible et lancera une ReferenceError s’il est utilisé avant le second est déclaré.

let i = 5;
{
    i = 6;  // outer i is unavailable within the Temporal Dead Zone
    let i;
}

ReferenceError : i n’est pas défini

Levage #

Les variables déclarées à la fois avec var et let sont [hissées][1]. La différence est qu’une variable déclarée avec var peut être référencée avant sa propre affectation, puisqu’elle est automatiquement affectée (avec undefined comme valeur), mais let ne peut pas, car elle nécessite spécifiquement que la variable soit déclarée avant d’être invoqué :

console.log(x); // >> undefined
console.log(y); // >> "ReferenceError: `y` is not defined"
//OR >> "ReferenceError: can't access lexical declaration `y` before initialization"
var x = 4;
let y = 7;

La zone entre le début d’un bloc et une déclaration let ou const est connue sous le nom de [Temporal Dead Zone](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/ Statements/let#Temporal_dead_zone_and_errors_with_let), et toute référence à la variable dans cette zone provoquera une ReferenceError. Cela se produit même si la [variable est affectée avant d’être déclarée][2] :

y=7; // >> "ReferenceError: `y` is not defined"
let y;

En mode non-strict, [l’attribution d’une valeur à une variable sans aucune déclaration, déclare automatiquement la variable dans la portée globale][3]. Dans ce cas, au lieu que y soit automatiquement déclaré dans la portée globale, let réserve le nom de la variable (y) et n’autorise aucun accès ou affectation à celle-ci avant la ligne où elle est déclarée/initialisée.

[1] : https://www.wikiod.com/fr/javascript/portee#Levage [2] : http://stackoverflow.com/questions/41451181/does-let-override-a-global-declaration-and-throws-a-referenceerror [3] : https://www.wikiod.com/fr/javascript/mode-strict#Modifications des propriétés globales

Syntaxe et invocation Apply and Call. #

Les méthodes apply et call de chaque fonction lui permettent de fournir une valeur personnalisée pour this.

function print() {
    console.log(this.toPrint);
}

print.apply({ toPrint: "Foo" }); // >> "Foo"
print.call({ toPrint: "Foo" }); // >> "Foo"

Vous remarquerez peut-être que la syntaxe des deux invocations utilisées ci-dessus est la même. c’est-à-dire que la signature est similaire.

Mais il y a une petite différence dans leur utilisation, puisque nous traitons des fonctions et changeons leurs portées, nous devons toujours conserver les arguments d’origine passés à la fonction. apply et call prennent en charge la transmission d’arguments à la fonction cible comme suit :

function speak() {
    var sentences = Array.prototype.slice.call(arguments);
    console.log(this.name+": "+sentences);
}
var person = { name: "Sunny" };
speak.apply(person, ["I", "Code", "Startups"]); // >> "Sunny: I Code Startups"
speak.call(person, "I", "<3", "Javascript"); // >> "Sunny: I <3 Javascript"

Notez que apply vous permet de passer un objet Array ou arguments (de type tableau) comme liste d’arguments, alors que call nécessite que vous passiez chaque argument séparément.

Ces deux méthodes vous donnent la liberté d’être aussi fantaisistes que vous le souhaitez, comme l’implémentation d’une mauvaise version du “bind” natif d’ECMAScript pour créer une fonction qui sera toujours appelée en tant que méthode d’un objet à partir d’une fonction d’origine.

function bind (func, obj) { 
    return function () {
        return func.apply(obj, Array.prototype.slice.call(arguments, 1));
    }
}

var obj = { name: "Foo" };

function print() {
    console.log(this.name);
}

printObj = bind(print, obj);

printObj();

Cela enregistrera

“Fou”

La fonction bind a beaucoup de choses à faire

  1. obj sera utilisé comme valeur de this
  2. transmettre les arguments à la fonction
  3. puis renvoyez la valeur

Invocation de la fonction fléchée #

Lorsque vous utilisez des fonctions fléchées, this prend la valeur de this du contexte d’exécution englobant (c’est-à-dire que this dans les fonctions fléchées a une portée lexicale plutôt que la portée dynamique habituelle). En code global (code qui n’appartient à aucune fonction), ce serait l’objet global. Et cela reste ainsi, même si vous invoquez la fonction déclarée avec la notation fléchée à partir de l’une des autres méthodes décrites ici.

var globalThis = this; //"window" in a browser, or "global" in Node.js

var foo = (() => this);           

console.log(foo() === globalThis);          //true

var obj = { name: "Foo" };
console.log(foo.call(obj) === globalThis);  //true

Voyez comment this hérite du contexte plutôt que de faire référence à l’objet sur lequel la méthode a été appelée.

var globalThis = this;

var obj = {
    withoutArrow: function() {
        return this;
    },
    withArrow: () => this
};

console.log(obj.withoutArrow() === obj);      //true
console.log(obj.withArrow() === globalThis);  //true

var fn = obj.withoutArrow; //no longer calling withoutArrow as a method
var fn2 = obj.withArrow;
console.log(fn() === globalThis);             //true
console.log(fn2() === globalThis);            //true

Invocation liée #

La méthode bind de chaque fonction vous permet de créer une nouvelle version de cette fonction avec le contexte strictement lié à un objet spécifique. Il est particulièrement utile de forcer l’appel d’une fonction en tant que méthode d’un objet.

var obj = { foo: 'bar' };

function foo() {
    return this.foo;
}

fooObj = foo.bind(obj);

fooObj();

Cela enregistrera :

barre

Appel de méthode #

En invoquant une fonction comme méthode d’un objet, la valeur de this sera cet objet.

var obj = {
    name: "Foo",
    print: function () {
        console.log(this.name)
    }
}

Nous pouvons maintenant invoquer print comme méthode de obj. “ceci” sera obj

obj.print();

Cela enregistrera donc :

Foo

Invocation anonyme #

Invoquant une fonction en tant que fonction anonyme, this sera l’objet global (self dans le navigateur).

function func() {
    return this;
}

func() === window; // true

En [mode strict d’ECMAScript 5][1], this sera undefined si la fonction est invoquée de manière anonyme.

(function () {
    "use strict";
    func();
}())

indéfini

Invocation du constructeur #

Lorsqu’une fonction est invoquée en tant que constructeur avec le mot clé new this prend la valeur de l’objet en cours de construction

function Obj(name) {
    this.name = name;
}

var obj = new Obj("Foo");

console.log(obj);

{ nom : “Foo” }

Utilisation de boucles let in au lieu de var (exemple de gestionnaires de clics) #

Disons que nous devons ajouter un bouton pour chaque élément du tableau loadedData (par exemple, chaque bouton doit être un curseur affichant les données ; par souci de simplicité, nous allons simplement alerter un message). On peut essayer quelque chose comme ça :

for(var i = 0; i < loadedData.length; i++)
    jQuery("#container").append("<a class='button'>"+loadedData[i].label+"</a>")
        .children().last() // now let's attach a handler to the button which is a child
        .on("click",function() { alert(loadedData[i].content); });

Mais au lieu d’alerter, chaque bouton provoquera le

TypeError :loadData[i] n’est pas défini

Erreur. En effet, la portée de i est la portée globale (ou une portée de fonction) et après la boucle, i == 3. Ce dont nous avons besoin n’est pas de “se souvenir de l’état de ‘i’”. Cela peut être fait en utilisant let :

for(let i = 0; i < loadedData.length; i++)
    jQuery("#container").append("<a class='button'>"+loadedData[i].label+"</a>")
        .children().last() // now let's attach a handler to the button which is a child
        .on("click",function() { alert(loadedData[i].content); });

Un exemple de loadedData à tester avec ce code :

    var loadedData = [
        { label:"apple",      content:"green and round" },
        { label:"blackberry", content:"small black or blue" },
        { label:"pineapple",  content:"weird stuff.. difficult to explain the shape" }
    ];
