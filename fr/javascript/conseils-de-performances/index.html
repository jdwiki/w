JavaScript, comme tout langage, nous oblige à être judicieux dans l’utilisation de certaines fonctionnalités du langage. La surutilisation de certaines fonctionnalités peut réduire les performances, tandis que certaines techniques peuvent être utilisées pour augmenter les performances.

Rappelez-vous que l’optimisation prématurée est la racine de tous les maux. Écrivez d’abord un code clair et correct, puis si vous rencontrez des problèmes de performances, utilisez un profileur pour rechercher des domaines spécifiques à améliorer. Ne perdez pas de temps à optimiser du code qui n’affecte pas les performances globales de manière significative.

Mesurer, mesurer, mesurer. Les performances peuvent souvent être contre-intuitives et changent avec le temps. Ce qui est plus rapide maintenant ne le sera peut-être pas à l’avenir et peut dépendre de votre cas d’utilisation. Assurez-vous que toutes les optimisations que vous effectuez s’améliorent réellement, ne nuisent pas aux performances, et que le changement en vaut la peine.

Évitez les tentatives/catch dans les fonctions critiques pour les performances #

Certains moteurs JavaScript (par exemple, la version actuelle de Node.js et les anciennes versions de Chrome avant Ignition+turbofan) n’exécutent pas l’optimiseur sur les fonctions qui contiennent un bloc try/catch.

Si vous avez besoin de gérer des exceptions dans du code critique pour les performances, il peut être plus rapide dans certains cas de conserver le try/catch dans une fonction distincte. Par exemple, cette fonction ne sera pas optimisée par certaines implémentations :

function myPerformanceCriticalFunction() {
    try {
        // do complex calculations here
    } catch (e) {
        console.log(e);
    }
}

Cependant, vous pouvez refactoriser pour déplacer le code lent dans une fonction séparée (qui peut être optimisée) et l’appeler depuis l’intérieur du bloc try.

// This function can be optimized
function doCalculations() {
    // do complex calculations here
}

// Still not always optimized, but it's not doing much so the performance doesn't matter
function myPerformanceCriticalFunction() {
    try {
        doCalculations();
    } catch (e) {
        console.log(e);
    }
}

Voici un benchmark jsPerf montrant la différence : https://jsperf.com/try-catch-deoptimization. Dans la version actuelle de la plupart des navigateurs, il ne devrait pas y avoir beaucoup de différence, mais dans les versions moins récentes de Chrome et Firefox, ou IE, la version qui appelle une fonction d’assistance dans le try/catch est susceptible d’être plus rapide.

Notez que des optimisations comme celle-ci doivent être faites avec soin et avec des preuves réelles basées sur le profilage de votre code. Au fur et à mesure que les moteurs JavaScript s’améliorent, cela pourrait finir par nuire aux performances au lieu d’aider, ou ne faire aucune différence (mais compliquer le code sans raison). Que cela aide, blesse ou ne fasse aucune différence peut dépendre de nombreux facteurs, alors mesurez toujours les effets sur votre code. C’est vrai pour toutes les optimisations, mais surtout pour les micro-optimisations comme celle-ci qui dépendent des détails de bas niveau du compilateur/d’exécution.

Limiter les mises à jour DOM #

Une erreur courante observée dans JavaScript lorsqu’il est exécuté dans un environnement de navigateur consiste à mettre à jour le DOM plus souvent que nécessaire.

Le problème ici est que chaque mise à jour de l’interface DOM oblige le navigateur à restituer l’écran. Si une mise à jour modifie la mise en page d’un élément de la page, la mise en page entière doit être recalculée, ce qui est très gourmand en performances, même dans les cas les plus simples. Le processus de redessin d’une page est connu sous le nom de reflow et peut ralentir le navigateur ou même ne plus répondre.

La conséquence d’une mise à jour trop fréquente du document est illustrée par l’exemple suivant d’ajout d’éléments à une liste.

Considérez le document suivant contenant un élément <ul> :

<!DOCTYPE html>
<html>
    <body>
        <ul id="list"></ul>
    </body>
</html>

Nous ajoutons 5000 éléments à la liste en boucle 5000 fois (vous pouvez essayer cela avec un plus grand nombre sur un ordinateur puissant pour augmenter l’effet).

var list = document.getElementById("list");
for(var i = 1; i <= 5000; i++) {             
    list.innerHTML += `<li>item ${i}</li>`;  // update 5000 times
}

Dans ce cas, les performances peuvent être améliorées en regroupant les 5000 modifications dans une seule mise à jour DOM.

var list = document.getElementById("list");
var html = "";
for(var i = 1; i <= 5000; i++) {
    html += `<li>item ${i}</li>`;
}
list.innerHTML = html;     // update once

La fonction [document.createDocumentFragment()][1] peut être utilisée comme conteneur léger pour le code HTML créé par la boucle. Cette méthode est légèrement plus rapide que la modification de la propriété innerHTML de l’élément conteneur (comme illustré ci-dessous).

var list = document.getElementById("list");
var fragment = document.createDocumentFragment();
for(var i = 1; i <= 5000; i++) {
    li = document.createElement("li");
    li.innerHTML = "item " + i;
    fragment.appendChild(li);
    i++;
}
list.appendChild(fragment);

[1] : https://developer.mozilla.org/en-US/docs/Web/API/Document/createDocumentFragment

Analyse comparative de votre code - mesure du temps d’exécution #

La plupart des conseils de performance dépendent beaucoup de l’état actuel des moteurs JS et ne sont censés être pertinents qu’à un moment donné. La loi fondamentale de l’optimisation des performances est que vous devez d’abord mesurer avant d’essayer d’optimiser, et mesurer à nouveau après une optimisation présumée.

Pour mesurer le temps d’exécution du code, vous pouvez utiliser différents outils de mesure du temps comme :

[Performance] [1] interface qui représente les informations de performance liées au temps pour la page donnée (uniquement disponible dans les navigateurs).

[process.hrtime][2] sur Node.js vous donne des informations de temps sous forme de tuples [secondes, nanosecondes]. Appelé sans argument, il renvoie une heure arbitraire mais appelé avec une valeur précédemment renvoyée comme argument, il renvoie la différence entre les deux exécutions.

[Minuteurs de la console][3] console.time("labelName") démarre un minuteur que vous pouvez utiliser pour suivre la durée d’une opération. Vous attribuez à chaque minuterie un nom d’étiquette unique et vous pouvez avoir jusqu’à 10 000 minuteries en cours d’exécution sur une page donnée. Lorsque vous appelez console.timeEnd("labelName") avec le même nom, le navigateur terminera la minuterie pour le nom donné et affichera le temps en millisecondes écoulé depuis le démarrage de la minuterie. Les chaînes passées à time() et timeEnd() doivent correspondre sinon le minuteur ne se terminera pas.

La fonction [Date.now][4] Date.now() renvoie [Timestamp][5] actuel en millisecondes, qui est une représentation [Number][6] du temps depuis le 1er janvier 1970 00:00:00 UTC jusqu’à à présent. La méthode now() est une méthode statique de Date, donc vous l’utilisez toujours comme Date.now().

Exemple 1 utilisant : performance.now()

Dans cet exemple nous allons calculer le temps écoulé pour l’exécution de notre fonction, et nous allons utiliser la méthode [Performance.now()][7] qui renvoie un [DOMHighResTimeStamp][8], mesuré en millisecondes, précis au millième de milliseconde près.

let startTime, endTime;

function myFunction() {
    //Slow code you want to measure
}

//Get the start time
startTime = performance.now();

//Call the time-consuming function
myFunction();

//Get the end time
endTime = performance.now();

//The difference is how many milliseconds it took to call myFunction()
console.debug('Elapsed time:', (endTime - startTime));

Le résultat dans la console ressemblera à ceci :

Elapsed time: 0.10000000009313226

L’utilisation de [performance.now()][7] a la précision la plus élevée dans les navigateurs avec une précision d’un millième de milliseconde, mais la plus faible [compatibilité][9].

Exemple 2 utilisant : Date.now()

Dans cet exemple nous allons calculer le temps écoulé pour l’initialisation d’un grand tableau (1 million de valeurs), et nous allons utiliser la méthode Date.now()

let t0 = Date.now(); //stores current Timestamp in milliseconds since 1 January 1970 00:00:00 UTC
let arr = []; //store empty array
for (let i = 0; i < 1000000; i++) { //1 million iterations
   arr.push(i); //push current i value
}
console.log(Date.now() - t0); //print elapsed time between stored t0 and now

Exemple 3 utilisant : console.time("label") & console.timeEnd("label")

Dans cet exemple, nous effectuons la même tâche que dans l’exemple 2, mais nous allons utiliser les méthodes console.time("label") & console.timeEnd("label")

console.time("t"); //start new timer for label name: "t"
let arr = []; //store empty array
for(let i = 0; i < 1000000; i++) { //1 million iterations
   arr.push(i); //push current i value
}
console.timeEnd("t"); //stop the timer for label name: "t" and print elapsed time

Exemple 4 utilisant process.hrtime()

Dans les programmes Node.js, c’est le moyen le plus précis de mesurer le temps passé.

let start = process.hrtime();

// long execution here, maybe asynchronous

let diff = process.hrtime(start);
// returns for example [ 1, 2325 ]
console.log(`Operation took ${diff[0] * 1e9 + diff[1]} nanoseconds`);
// logs: Operation took 1000002325 nanoseconds

[1] : https://developer.mozilla.org/en-US/docs/Web/API/Performance [2] : https://nodejs.org/api/process.html#process_process_hrtime_time [3] : https://developer.mozilla.org/en-US/docs/Web/API/Console/time [4] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Date/now [5] : http://www.unixtimestamp.com/ [6] : https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Number [7] : https://developer.mozilla.org/en-US/docs/Web/API/Performance/now [8] : https://developer.mozilla.org/en-US/docs/Web/API/DOMHighResTimeStamp [9] : https://developer.mozilla.org/en-US/docs/Web/API/Performance/now#Browser_compatibility

Utilisez un mémoiseur pour les fonctions de calcul intensif #

Si vous construisez une fonction qui peut être lourde sur le processeur (côté client ou côté serveur), vous pouvez envisager un memoizer qui est un cache des exécutions de fonctions précédentes et de leurs valeurs renvoyées. Cela permet de vérifier si les paramètres d’une fonction ont été passés auparavant. N’oubliez pas que les fonctions pures sont celles qui ont donné une entrée, renvoient une sortie unique correspondante et ne provoquent pas d’effets secondaires en dehors de leur portée, vous ne devez donc pas ajouter de mémoires aux fonctions qui sont imprévisibles ou dépendent de ressources externes (comme les appels AJAX ou aléatoirement valeurs renvoyées).

Disons que j’ai une fonction factorielle récursive:

function fact(num) {
  return (num === 0)? 1 : num * fact(num - 1);
}

Si je passe de petites valeurs de 1 à 100 par exemple, il n’y aurait pas de problème, mais une fois que nous commençons à aller plus loin, nous pourrions faire exploser la pile d’appels ou rendre le processus un peu pénible pour le moteur Javascript dans lequel nous faisons cela, surtout si le moteur ne compte pas avec l’optimisation des appels de queue (bien que Douglas Crockford affirme que l’ES6 natif inclut l’optimisation des appels de queue).

Nous pourrions coder en dur notre propre dictionnaire de 1 à dieu sait quel nombre avec leurs factorielles correspondantes mais, je ne sais pas si je le conseille ! Créons un mémoiseur, d’accord ?

var fact = (function() {
  var cache = {}; // Initialise a memory cache object
  
  // Use and return this function to check if val is cached
  function checkCache(val) {
    if (val in cache) {
      console.log('It was in the cache :D');
      return cache[val]; // return cached
    } else {
      cache[val] = factorial(val); // we cache it
      return cache[val]; // and then return it
    }
    
    /* Other alternatives for checking are:
    || cache.hasOwnProperty(val) or !!cache[val]
    || but wouldn't work if the results of those
    || executions were falsy values.
    */
  }

  // We create and name the actual function to be used
  function factorial(num) {
    return (num === 0)? 1 : num * factorial(num - 1);
  } // End of factorial function

  /* We return the function that checks, not the one
  || that computes because  it happens to be recursive,
  || if it weren't you could avoid creating an extra
  || function in this self-invoking closure function.
  */
  return checkCache; 
}());

Maintenant, nous pouvons commencer à l’utiliser :

[![C’est comme ça que ça se passe][1]][1]

Maintenant que je commence à réfléchir à ce que j’ai fait, si je devais incrémenter de 1 au lieu de décrémenter de num, j’aurais pu mettre en cache tous les factoriels de 1 à num dans le cache de manière récursive, mais je vais laisser ça pour toi.

C’est très bien, mais que se passe-t-il si nous avons plusieurs paramètres ? C’est un problème? Pas tout à fait, nous pouvons faire de belles astuces comme utiliser JSON.stringify() sur le tableau d’arguments ou même une liste de valeurs dont dépendra la fonction (pour les approches orientées objet). Ceci est fait pour générer une clé unique avec tous les arguments et dépendances inclus.

Nous pouvons également créer une fonction qui “mémorise” d’autres fonctions, en utilisant le même concept de portée qu’auparavant (renvoyant une nouvelle fonction qui utilise l’original et a accès à l’objet cache) :

ATTENTION : la syntaxe ES6, si vous ne l’aimez pas, remplacez … par rien et utilisez l’astuce var args = Array.prototype.slice.call(null, arguments);; remplacez const et let par var, et les autres choses que vous connaissez déjà.

function memoize(func) {
  let cache = {};

  // You can opt for not naming the function
  function memoized(...args) {
    const argsKey = JSON.stringify(args);
    
    // The same alternatives apply for this example
    if (argsKey in cache) {
      console.log(argsKey + ' was/were in cache :D');
      return cache[argsKey];
    } else {
      cache[argsKey] = func.apply(null, args); // Cache it
      return cache[argsKey]; // And then return it
    }
  }

  return memoized; // Return the memoized function
}

Notez maintenant que cela fonctionnera pour plusieurs arguments mais ne sera pas très utile dans les méthodes orientées objet, je pense, vous aurez peut-être besoin d’un objet supplémentaire pour les dépendances. De plus, func.apply(null, args) peut être remplacé par func(...args) puisque la déstructuration des tableaux les enverra séparément plutôt que sous forme de tableau. De plus, juste pour référence, passer un tableau comme argument à func ne fonctionnera que si vous utilisez Function.prototype.apply comme je l’ai fait.

Pour utiliser la méthode ci-dessus, il vous suffit de :

const newFunction = memoize(oldFunction);

// Assuming new oldFunction just sums/concatenates:
newFunction('meaning of life', 42);
// -> "meaning of life42"

newFunction('meaning of life', 42); // again
// => ["meaning of life",42] was/were in cache :D
// -> "meaning of life42"

Initialisation des propriétés de l’objet avec null #

Tous les compilateurs JavaScript JIT modernes essayant d’optimiser le code en fonction des structures d’objet attendues. Quelques conseils de [mdn][1].

Heureusement, les objets et les propriétés sont souvent “prévisibles”, et dans ces cas, leur structure sous-jacente peut également être prévisible. JIT peut s’appuyer sur cela pour accélérer les accès prévisibles.

La meilleure façon de rendre l’objet prévisible est de définir une structure entière dans un constructeur. Donc, si vous souhaitez ajouter des propriétés supplémentaires après la création de l’objet, définissez-les dans un constructeur avec null. Cela aidera l’optimiseur à prédire le comportement de l’objet pendant tout son cycle de vie. Cependant, tous les compilateurs ont des optimiseurs différents et l’augmentation des performances peut être différente, mais dans l’ensemble, il est recommandé de définir toutes les propriétés dans un constructeur, même lorsque leur valeur n’est pas encore connue.

Le temps de quelques tests. Dans mon test, je crée un grand tableau de certaines instances de classe avec une boucle for. Dans la boucle, j’attribue la même chaîne à la propriété “x” de tous les objets avant l’initialisation du tableau. Si le constructeur initialise la propriété “x” avec null, le tableau traite toujours mieux même s’il fait une instruction supplémentaire.

C’est le code :

function f1() {
    var P = function () {
        this.value = 1
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}

function f2() {
    var P = function () {
        this.value = 1;
        this.x = null;
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}


(function perform(){
    var start = performance.now();
    f1();
    var duration = performance.now() - start;

    console.log('duration of f1  ' + duration);


    start = performance.now();
    f2();
    duration = performance.now() - start;

    console.log('duration of f2 ' + duration);
})()

C’est le résultat pour Chrome et Firefox.

       FireFox     Chrome
 --------------------------
 f1      6,400      11,400
 f2      1,700       9,600

Comme nous pouvons le voir, les améliorations de performances sont très différentes entre les deux.

Réutiliser des objets plutôt que de les recréer #

Exemple A #

var i,a,b,len;
a = {x:0,y:0}
function test(){ // return object created each call
    return {x:0,y:0};
}
function test1(a){ // return object supplied 
    a.x=0;
    a.y=0;
    return a;
}   

for(i = 0; i < 100; i ++){ // Loop A
   b = test();
}

for(i = 0; i < 100; i ++){ // Loop B
   b = test1(a);
}

La boucle B est 4 (400 %) fois plus rapide que la boucle A

Il est très inefficace de créer un nouvel objet dans le code de performance. La boucle A appelle la fonction test() qui renvoie un nouvel objet à chaque appel. L’objet créé est supprimé à chaque itération, la boucle B appelle test1() qui nécessite que les retours d’objet soient fournis. Il utilise ainsi le même objet et évite l’allocation d’un nouvel objet, et les hits GC excessifs. (GC n’a pas été inclus dans le test de performance)

Exemple B #

var i,a,b,len;
a = {x:0,y:0}
function test2(a){
    return {x : a.x * 10,y : a.x * 10};
}   
function test3(a){
    a.x= a.x * 10;
    a.y= a.y * 10;
    return a;
}  
for(i = 0; i < 100; i++){  // Loop A
    b = test2({x : 10, y : 10});
}
for(i = 0; i < 100; i++){ // Loop B
    a.x = 10;
    a.y = 10;
    b = test3(a);                
}

La boucle B est 5 (500%) fois plus rapide que la boucle A

Préférez les variables locales aux variables globales, aux attributs et aux valeurs indexées #

Les moteurs Javascript recherchent d’abord des variables dans la portée locale avant d’étendre leur recherche à des portées plus larges. Si la variable est une valeur indexée dans un tableau ou un attribut dans un tableau associatif, elle recherchera d’abord le tableau parent avant de trouver le contenu.

Cela a des implications lorsque vous travaillez avec du code critique pour les performances. Prenons par exemple une boucle for courante :

var global_variable = 0;
function foo(){
    global_variable = 0;
    for (var i=0; i<items.length; i++) {
        global_variable += items[i];
    }
}

Pour chaque itération dans la boucle for, le moteur recherchera items, recherchera l’attribut length dans les éléments, recherchera à nouveau items, recherchera la valeur à l’index i de items, puis recherchera finalement global_variable, en essayant d’abord la portée locale avant de vérifier la portée globale.

Une réécriture performante de la fonction ci-dessus est :

function foo(){
    var local_variable = 0;
    for (var i=0, li=items.length; i<li; i++) {
        local_variable += items[i];
    }
    return local_variable;
}

Pour chaque itération dans la boucle for réécrite, le moteur recherchera li, recherchera items, recherchera la valeur à l’index i et recherchera local_variable, cette fois n’ayant qu’à vérifier la portée locale.

Soyez cohérent dans l’utilisation des nombres #

Si le moteur est capable de prédire correctement que vous utilisez un petit type spécifique pour vos valeurs, il pourra optimiser le code exécuté.

Dans cet exemple, nous utiliserons cette fonction triviale additionnant les éléments d’un tableau et affichant le temps qu’il a fallu :

// summing properties
var sum = (function(arr){
        var start = process.hrtime();
        var sum = 0;
        for (var i=0; i<arr.length; i++) {
                sum += arr[i];
        }
        var diffSum = process.hrtime(start);
        console.log(`Summing took ${diffSum[0] * 1e9 + diffSum[1]} nanoseconds`);
        return sum;
})(arr);

Faisons un tableau et additionnons les éléments :

var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.random();

Résultat:

Summing took 384416 nanoseconds

Maintenant, faisons la même chose mais avec uniquement des entiers :

var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.round(1000*Math.random());

Résultat:

Summing took 180520 nanoseconds

** La sommation des nombres entiers a pris la moitié du temps ici. **

Les moteurs n’utilisent pas les mêmes types que vous avez en JavaScript. Comme vous le savez probablement, tous les nombres en JavaScript sont des nombres à virgule flottante double précision IEEE754, il n’y a pas de représentation spécifique disponible pour les entiers. Mais les moteurs, lorsqu’ils peuvent prédire que vous n’utilisez que des entiers, peuvent utiliser une représentation plus compacte et plus rapide à utiliser, par exemple des entiers courts.

Ce type d’optimisation est particulièrement important pour les applications gourmandes en calcul ou en données.
