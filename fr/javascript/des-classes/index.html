Syntaxe #

  • classe Foo {}
  • la classe Foo étend la barre {}
  • classe Foo { constructeur() {} }
  • class Foo { maMéthode() {} }
  • class Foo { get myProperty() {} }
  • class Foo { set myProperty(newValue) {} }
  • class Foo { static myStaticMethod() {} }
  • class Foo { static get myStaticProperty() {} }
  • const Foo = classe Foo {} ;
  • const Foo = classe {} ;

La prise en charge de class n’a été ajoutée à JavaScript que dans le cadre de la norme 2015 [tag: es6].

Les classes Javascript sont du sucre syntaxique par rapport à l’héritage basé sur les prototypes déjà existant de JavaScript. Cette nouvelle syntaxe n’introduit pas un nouveau modèle d’héritage orienté objet dans JavaScript, juste une manière plus simple de gérer les objets et l’héritage. Une déclaration de “classe” est essentiellement un raccourci pour définir manuellement une “fonction” de constructeur et ajouter des propriétés au prototype du constructeur. Une différence importante est que les fonctions peuvent être appelées directement (sans le mot clé new), alors qu’une classe appelée directement lèvera une exception.

class someClass {
    constructor () {}
    someMethod () {}
}
 
console.log(typeof someClass);               
console.log(someClass);
console.log(someClass === someClass.prototype.constructor);                         
console.log(someClass.prototype.someMethod);
 
// Output:
// function
// function someClass() { "use strict"; }
// true
// function () { "use strict"; }

Si vous utilisez une version antérieure de JavaScript, vous aurez besoin d’un transpileur comme [tag:babel] ou [tag:google-closure-compiler] afin de compiler le code dans une version que la plate-forme cible pourra comprendre.

Constructeur de classe #

La partie fondamentale de la plupart des classes est son constructeur, qui définit l’état initial de chaque instance et gère tous les paramètres passés lors de l’appel de new.

Il est défini dans un bloc class comme si vous définissiez une méthode nommée constructor, bien qu’il soit en fait traité comme un cas particulier.

class MyClass {
    constructor(option) {
        console.log(`Creating instance using ${option} option.`);
        this.option = option;
    }
}

Exemple d’utilisation :

const foo = new MyClass('speedy'); // logs: "Creating instance using speedy option"

Une petite chose à noter est qu’un constructeur de classe ne peut pas être rendu statique via le mot-clé static, comme décrit ci-dessous pour les autres méthodes.

Héritage de classe #

L’héritage fonctionne exactement comme dans les autres langages orientés objet : les méthodes définies sur la superclasse sont accessibles dans la sous-classe d’extension.

Si la sous-classe déclare son propre constructeur, elle doit invoquer le constructeur parent via super() avant de pouvoir accéder à this.

class SuperClass {

    constructor() {
        this.logger = console.log;
    }

    log() {
        this.logger(`Hello ${this.name}`);
    }

}

class SubClass extends SuperClass {

    constructor() {
        super();
        this.name = 'subclass';
    }

}

const subClass = new SubClass();

subClass.log(); // logs: "Hello subclass"

Méthodes statiques #

Les méthodes et propriétés statiques sont définies sur la classe/le constructeur lui-même, et non sur les objets d’instance. Ceux-ci sont spécifiés dans une définition de classe à l’aide du mot-clé static.

class MyClass {
    static myStaticMethod() {
        return 'Hello';
    }

    static get myStaticProperty() {
        return 'Goodbye';
    }
}

console.log(MyClass.myStaticMethod()); // logs: "Hello"
console.log(MyClass.myStaticProperty); // logs: "Goodbye"

Nous pouvons voir que les propriétés statiques ne sont pas définies sur les instances d’objet :

const myClassInstance = new MyClass();

console.log(myClassInstance.myStaticProperty); // logs: undefined

Cependant, ils sont définis sur des sous-classes :

class MySubClass extends MyClass {};

console.log(MySubClass.myStaticMethod()); // logs: "Hello"
console.log(MySubClass.myStaticProperty); // logs: "Goodbye"

Getters et Setters #

Les getters et les setters vous permettent de définir un comportement personnalisé pour la lecture et l’écriture d’une propriété donnée sur votre classe. Pour l’utilisateur, ils apparaissent comme n’importe quelle propriété typique. Cependant, en interne, une fonction personnalisée que vous fournissez est utilisée pour déterminer la valeur lors de l’accès à la propriété (le getter) et pour effectuer les modifications nécessaires lorsque la propriété est attribuée (le setter).

Dans une définition class, un getter est écrit comme une méthode sans argument préfixée par le mot-clé get. Un setter est similaire, sauf qu’il accepte un argument (la nouvelle valeur étant assignée) et le mot-clé set est utilisé à la place.

Voici un exemple de classe qui fournit un getter et un setter pour sa propriété .name. Chaque fois qu’il est assigné, nous enregistrerons le nouveau nom dans un tableau interne .names_. À chaque accès, nous renverrons le dernier nom.

class MyClass {
    constructor() {
        this.names_ = [];
    }

    set name(value) {
        this.names_.push(value);
    }

    get name() {
        return this.names_[this.names_.length - 1];
    }
}

const myClassInstance = new MyClass();
myClassInstance.name = 'Joe';
myClassInstance.name = 'Bob';

console.log(myClassInstance.name); // logs: "Bob"
console.log(myClassInstance.names_); // logs: ["Joe", "Bob"]

Si vous ne définissez qu’un setter, tenter d’accéder à la propriété renverra toujours undefined.

const classInstance = new class {
    set prop(value) {
        console.log('setting', value);
    }
};

classInstance.prop = 10; // logs: "setting", 10

console.log(classInstance.prop); // logs: undefined

Si vous ne définissez qu’un getter, tenter d’affecter la propriété n’aura aucun effet.

const classInstance = new class {
    get prop() {
        return 5;
    }
};

classInstance.prop = 10;

console.log(classInstance.prop); // logs: 5

Membres privés #

JavaScript ne prend pas techniquement en charge les membres privés en tant que fonctionnalité de langage. La confidentialité - [décrite par Douglas Crockford] [1] - est émulée à la place via des fermetures (portée de la fonction préservée) qui sera générée à chaque appel d’instanciation d’une fonction constructeur.

L’exemple Queue montre comment, avec les fonctions constructeur, l’état local peuvent également être préservés et rendus accessibles via des méthodes privilégiées.

class Queue {

  constructor () {                    // - does generate a closure with each instantiation.

    const list = [];                  // - local state ("private member").

    this.enqueue = function (type) {  // - privileged public method
                                      //   accessing the local state
      list.push(type);                //   "writing" alike.
      return type;
    };
    this.dequeue = function () {      // - privileged public method
                                      //   accessing the local state
      return list.shift();            //   "reading / writing" alike.
    };
  }
}


var q = new Queue;            //
                              //
q.enqueue(9);                 // ... first in ...
q.enqueue(8);                 //
q.enqueue(7);                 //
                              //
console.log(q.dequeue());     // 9 ... first out.
console.log(q.dequeue());     // 8
console.log(q.dequeue());     // 7
console.log(q);               // {}
console.log(Object.keys(q));  // ["enqueue","dequeue"]

À chaque instanciation d’un type Queue, le constructeur génère une fermeture.

Ainsi, les deux méthodes propres à un type Queue enqueue et dequeue (voir Object.keys(q)) ont toujours accès à list qui continue à * vivre * dans sa portée englobante qui, au moment de la construction, a été conservé.

Utiliser ce modèle - imiter les membres privés via le public privilégié méthodes - il faut garder à l’esprit que, avec chaque instance, de la mémoire supplémentaire sera consommé pour chaque méthode own property (car c’est du code qui ne peut pas être partagé/réutilisé). Il en va de même pour la quantité/taille d’état qui va être stockée dans une telle fermeture.

[1] : http://javascript.crockford.com/private.html

Noms de méthodes dynamiques #

Il est également possible d’évaluer des expressions lors de la dénomination de méthodes similaires à la façon dont vous pouvez accéder aux propriétés d’un objet avec []. Cela peut être utile pour avoir des noms de propriété dynamiques, mais est souvent utilisé en conjonction avec des symboles.

let METADATA = Symbol('metadata');

class Car {
  constructor(make, model) {
    this.make = make;
    this.model = model;
  }
  
  // example using symbols
  [METADATA]() {
    return {
      make: this.make,
      model: this.model
    };
  }

  // you can also use any javascript expression

  // this one is just a string, and could also be defined with simply add()
  ["add"](a, b) {
    return a + b;
  }

  // this one is dynamically evaluated
  [1 + 2]() {
    return "three";
  }
}

let MazdaMPV = new Car("Mazda", "MPV");
MazdaMPV.add(4, 5); // 9
MazdaMPV[3](); // "three"
MazdaMPV[METADATA](); // { make: "Mazda", model: "MPV" }

Méthodes #

Les méthodes peuvent être définies dans des classes pour exécuter une fonction et éventuellement renvoyer un résultat. Ils peuvent recevoir des arguments de l’appelant.

class Something {
    constructor(data) {
        this.data = data
    }

    doSomething(text) {
        return {
            data: this.data,
            text
        }
    }
}

var s = new Something({})
s.doSomething("hi") // returns: { data: {}, text: "hi" }

Gérer les données privées avec des classes #

L’un des obstacles les plus courants à l’utilisation des classes est de trouver la bonne approche pour gérer les états privés. Il existe 4 solutions courantes pour gérer les états privés :

Utiliser des symboles #

Les symboles sont un nouveau type primitif introduit dans ES2015, tel que défini à [MDN] [1]

Un symbole est un type de données unique et immuable qui peut être utilisé comme identifiant pour les propriétés d’un objet.

Lorsque vous utilisez un symbole comme clé de propriété, il n’est pas énumérable.

En tant que tels, ils ne seront pas révélés en utilisant for var in ou Object.keys.

Ainsi, nous pouvons utiliser des symboles pour stocker des données privées.

const topSecret = Symbol('topSecret'); // our private key; will only be accessible on the scope of the module file
export class SecretAgent{
    constructor(secret){
        this[topSecret] = secret; // we have access to the symbol key (closure)
        this.coverStory = 'just a simple gardner';
        this.doMission = () => {
            figureWhatToDo(topSecret[topSecret]); // we have access to topSecret
        };
    }
}

Étant donné que les “symboles” sont uniques, nous devons avoir une référence au symbole d’origine pour accéder à la propriété privée.

import {SecretAgent} from 'SecretAgent.js'
const agent = new SecretAgent('steal all the ice cream');
// ok lets try to get the secret out of him!
Object.keys(agent); // ['coverStory'] only cover story is public, our secret is kept.
agent[Symbol('topSecret')]; // undefined, as we said, symbols are always unique, so only the original symbol will help us to get the data.

Mais ce n’est pas 100 % privé ; détruisons cet agent ! Nous pouvons utiliser la méthode Object.getOwnPropertySymbols pour obtenir les symboles d’objet.

const secretKeys = Object.getOwnPropertySymbols(agent);
agent[secretKeys[0]] // 'steal all the ice cream' , we got the secret.

Utilisation de WeakMaps #

WeakMap est un nouveau type d’objet qui a été ajouté pour es6.

Tel que défini sur [MDN][2]

L’objet WeakMap est une collection de paires clé/valeur dans laquelle les clés sont faiblement référencées. Les clés doivent être des objets et les valeurs peuvent être des valeurs arbitraires.

Une autre caractéristique importante de WeakMap est, telle que définie sur [MDN][3].

La clé dans une WeakMap est tenue faiblement. Cela signifie que, s’il n’y a pas d’autres références fortes à la clé, l’entrée entière sera supprimée de la WeakMap par le ramasse-miettes.

L’idée est d’utiliser WeakMap, comme une carte statique pour toute la classe, pour conserver chaque instance comme clé et conserver les données privées comme valeur pour cette clé d’instance.

Ainsi, c’est seulement à l’intérieur de la classe que nous aurons accès à la collection WeakMap.

Essayons notre agent avec WeakMap :

const topSecret = new WeakMap(); // will hold all private data of all instances.
export class SecretAgent{
    constructor(secret){
        topSecret.set(this,secret); // we use this, as the key, to set it on our instance private data
        this.coverStory = 'just a simple gardner';
        this.doMission = () => {
            figureWhatToDo(topSecret.get(this)); // we have access to topSecret
        };
    }
}

Parce que le const topSecret est défini à l’intérieur de notre fermeture de module, et puisque nous ne l’avons pas lié à nos propriétés d’instance, cette approche est totalement privée et nous ne pouvons pas atteindre l’agent topSecret.

Définir toutes les méthodes à l’intérieur du constructeur #

L’idée ici est simplement de définir toutes nos méthodes et tous nos membres à l’intérieur du constructeur et d’utiliser la fermeture pour accéder aux membres privés sans les affecter à “this”.

   export class SecretAgent{
        constructor(secret){
            const topSecret = secret;
            this.coverStory = 'just a simple gardner';
            this.doMission = () => {
                figureWhatToDo(topSecret); // we have access to topSecret
            };
        }
    }

Dans cet exemple également, les données sont 100 % privées et ne peuvent pas être atteintes en dehors de la classe, donc notre agent est en sécurité.

Utilisation des conventions de nommage #

Nous déciderons que toute propriété privée sera préfixée par _.

Notez que pour cette approche, les données ne sont pas vraiment privées.

export class SecretAgent{
    constructor(secret){
        this._topSecret = secret; // it private by convention
        this.coverStory = 'just a simple gardner';
        this.doMission = () => {
            figureWhatToDo(this_topSecret); 
        };
    }
}

Liaison de nom de classe #

Le nom de ClassDeclaration est lié de différentes manières dans différentes portées -

  1. La portée dans laquelle la classe est définie - liaison “let”
  2. La portée de la classe elle-même - entre { et } dans class {} - liaison const
class Foo {
  // Foo inside this block is a const binding
}
// Foo here is a let binding

Par exemple,

class A {
  foo() {
    A = null; // will throw at runtime as A inside the class is a `const` binding
  }
}
A = null; // will NOT throw as A here is a `let` binding

Ce n’est pas la même chose pour une fonction -

function A() {
  A = null; // works
}
A.prototype.foo = function foo() {
  A = null; // works
}
A = null; // works
