Tutoriel en langage C

C est un langage de programmation informatique impératif à usage général, prenant en charge la programmation structurée, la portée des variables lexicales et la récursivité, tandis qu’un système de type statique empêche de nombreuses opérations involontaires. De par sa conception, C fournit des constructions qui correspondent efficacement aux instructions machine typiques, et par conséquent, il a trouvé une utilisation durable dans des applications qui étaient auparavant codées en langage d’assemblage, y compris les systèmes d’exploitation, ainsi que divers logiciels d’application pour ordinateurs allant des superordinateurs aux systèmes embarqués. .

Malgré ses capacités de bas niveau, le langage a été conçu pour encourager la programmation multiplateforme. Un programme C conforme aux normes et écrit de manière portable peut être compilé pour une très grande variété de plates-formes informatiques et de systèmes d’exploitation avec peu de modifications de son code source. Le langage est devenu disponible sur une très large gamme de plates-formes, des microcontrôleurs embarqués aux supercalculateurs.

C a été initialement développé par Dennis Ritchie entre 1969 et 1973 aux Bell Labs et utilisé pour réimplémenter les systèmes d’exploitation [Unix][1]. Il est depuis devenu l’un des langages de programmation les plus utilisés de tous les temps, avec des compilateurs C de divers fournisseurs disponibles pour la majorité des architectures informatiques et des systèmes d’exploitation existants.

Compilateurs communs #

Le processus de compilation d’un programme C diffère entre les compilateurs et les systèmes d’exploitation. La plupart des systèmes d’exploitation sont livrés sans compilateur, vous devrez donc en installer un. Certains choix de compilateurs courants sont :

  • GCC, la collection de compilateurs GNU
  • [clang : un frontal de la famille du langage C pour LLVM] (http://clang.llvm.org/)
  • MSVC, outils de construction Microsoft Visual C/C++

Les documents suivants devraient vous donner un bon aperçu de la façon de commencer à utiliser quelques-uns des compilateurs les plus courants :

  • Démarrer avec Microsoft Visual C
  • Démarrer avec GCC

Prise en charge de la version C du compilateur #

Notez que les compilateurs ont différents niveaux de prise en charge du standard C et que beaucoup ne prennent toujours pas complètement en charge C99. Par exemple, à partir de la version 2015, MSVC prend en charge une grande partie de C99, mais comporte encore quelques exceptions importantes pour la prise en charge du langage lui-même (par exemple, le prétraitement semble non conforme) et pour la bibliothèque C (par exemple, <tgmath.h>) , ni ne documentent nécessairement leurs “choix dépendants de la mise en œuvre”. [Wikipedia a un tableau][2] montrant le support offert par certains compilateurs populaires.

Certains compilateurs (notamment GCC) ont proposé, ou continuent de proposer, des extensions de compilateur qui implémentent des fonctionnalités supplémentaires que les producteurs de compilateurs jugent nécessaires, utiles ou qui, selon eux, pourraient faire partie d’une future version C, mais qui ne font actuellement partie d’aucune version C la norme. Comme ces extensions sont spécifiques au compilateur, elles peuvent être considérées comme non compatibles entre elles et les développeurs du compilateur peuvent les supprimer ou les modifier dans les versions ultérieures du compilateur. L’utilisation de telles extensions peut généralement être contrôlée par des drapeaux de compilateur.

De plus, de nombreux développeurs ont des compilateurs qui ne prennent en charge que des versions spécifiques de C imposées par l’environnement ou la plate-forme qu’ils ciblent.

Si vous sélectionnez un compilateur, il est recommandé de choisir un compilateur qui prend en charge au mieux la dernière version de C autorisée pour l’environnement cible.

Style de code (hors sujet ici): #

Parce que les espaces blancs sont insignifiants en C (c’est-à-dire qu’ils n’affectent pas le fonctionnement du code), les programmeurs utilisent souvent des espaces blancs pour rendre le code plus facile à lire et à comprendre, c’est ce qu’on appelle le style de code. Il s’agit d’un ensemble de règles et de directives utilisées lors de l’écriture du code source. Il couvre des problèmes tels que la façon dont les lignes doivent être indentées, si les espaces ou les tabulations doivent être utilisés, comment les accolades doivent être placées, comment les espaces doivent être utilisés autour des opérateurs et des crochets, comment les variables doivent être nommées, etc.

Le style de code n’est pas couvert par la norme et est principalement basé sur l’opinion (différentes personnes trouvent différents styles plus faciles à lire), en tant que tel, il est généralement considéré comme hors sujet sur SO. Le conseil primordial sur le style dans son propre code est que la cohérence est primordiale - choisissez, ou créez, un style et respectez-le. Il suffit d’expliquer qu’il existe différents styles nommés d’usage courant qui sont souvent choisis par les programmeurs plutôt que de créer leur propre style.

Certains styles d’indentation courants sont : le style K & R, le style Allman, le style GNU, etc. Certains de ces styles ont différentes variantes. Allman, par exemple, est utilisé comme Allman ordinaire ou comme variante populaire, Allman-8. Des informations sur certains des styles populaires peuvent être trouvées sur [Wikipedia][3]. Ces noms de style sont tirés des normes que les auteurs ou les organisations publient souvent pour être utilisées par les nombreuses personnes qui contribuent à leur code, afin que chacun puisse facilement lire le code lorsqu’il connaît le style, comme le [guide de formatage GNU][4] qui fait partie du document [normes de codage GNU][5].

Certaines conventions de dénomination courantes sont : UpperCamelCase, lowerCamelCase, lower_case_with_underscore, ALL_CAPS, etc. Ces styles sont combinés de différentes manières pour être utilisés avec différents objets et types (par exemple, les macros utilisent souvent le style ALL_CAPS)

Le style K & R est généralement recommandé pour une utilisation dans la documentation SO, tandis que les styles plus ésotériques, tels que Pico, sont déconseillés.
