Tutoriel google-cloud-messaging

Google Cloud Messagerie : présentation

Google Cloud Messaging (GCM) est un service gratuit qui permet aux développeurs d’envoyer des messages entre les serveurs et les applications clientes. Cela inclut les messages en aval des serveurs vers les applications clientes et les messages en amont des applications clientes vers les serveurs.

Par exemple, un message léger en aval pourrait informer une application cliente qu’il y a de nouvelles données à extraire du serveur, comme dans le cas d’une notification “nouveau courrier électronique”. Pour les cas d’utilisation tels que la messagerie instantanée, un message GCM peut transférer jusqu’à 4 Ko de charge utile à l’application cliente. Le service GCM gère tous les aspects de la mise en file d’attente des messages et de la livraison vers et depuis l’application cliente cible.

Aperçu architectural

Une implémentation GCM comprend un serveur de connexion Google, un serveur d’applications dans votre environnement qui interagit avec le serveur de connexion via le protocole HTTP ou XMPP, et une application cliente.

Voici comment ces composants interagissent :

  • Google GCM Connection Servers accepte les messages en aval de votre serveur d’applications et les envoie à une application cliente. Le serveur de connexion XMPP peut également accepter les messages envoyés en amont depuis l’application cliente et les transférer vers votre serveur d’applications. Pour plus d’informations, consultez À propos du serveur de connexion GCM.
  • Sur votre App Server, vous implémentez HTTP et/ou [XMPP](https://developers.google. com/cloud-messaging/ccs) pour communiquer avec le(s) serveur(s) de connexion GCM. Les serveurs d’applications envoient des messages en aval à un serveur de connexion GCM ; le serveur de connexion met en file d’attente et stocke le message, puis l’envoie à l’application cliente. Si vous implémentez XMPP, votre serveur d’applications peut recevoir des messages envoyés depuis l’application cliente.
  • L’application client est une application client compatible GCM. Pour recevoir et envoyer des messages GCM, cette application doit s’enregistrer auprès de GCM et obtenir un identifiant unique appelé jeton d’enregistrement. Pour plus d’informations sur la façon d’implémenter l’application cliente, consultez la documentation de votre plate-forme.

Concepts clés

Vous trouverez ci-dessous un résumé des termes et concepts clés impliqués dans GCM. Il est divisé en ces catégories :

  • Composants : les entités qui jouent un rôle principal dans GCM. - Identifiants : identifiants et jetons utilisés dans GCM pour garantir que toutes les parties ont été authentifiées et que le message est acheminé au bon endroit.

Composants et identifiants GCM.

Composants

  • Serveurs de connexion GCM : serveurs Google impliqués dans l’envoi de messages entre le serveur d’application et l’application cliente.
  • Application client : une application client compatible GCM qui communique avec votre serveur d’applications.
  • Serveur d’applications : un serveur d’applications que vous écrivez dans le cadre de la mise en œuvre de GCM. Le serveur d’application envoie des données à une application cliente via le serveur de connexion GCM. Si votre serveur d’applications implémente le protocole XMPP, il peut également recevoir des messages envoyés en amont à partir d’applications clientes.

Identifiants

- Identifiant de l’expéditeur
Une valeur numérique unique créée lorsque vous configurez votre projet d’API. L’ID de l’expéditeur est utilisé dans le processus d’inscription pour identifier un serveur d’applications autorisé à envoyer des messages à l’application cliente.

  • Clé du serveur
    Une clé enregistrée sur le serveur d’applications qui donne au serveur d’applications un accès autorisé aux services Google. En HTTP, la clé du serveur est incluse dans l’en-tête des requêtes POST qui envoient des messages. Dans XMPP, la clé du serveur est utilisée dans la demande d’authentification SASL PLAIN comme mot de passe pour authentifier la connexion. N’incluez la clé du serveur nulle part dans votre code client. Vous obtenez la clé du serveur lorsque vous créez votre projet d’API.

  • ID d’application L’application cliente qui s’enregistre pour recevoir des messages. La façon dont cela est implémenté dépend de la plate-forme :

  • Android : utilisez le nom du package du manifeste de l’application.

  • iOS : utilisez l’identifiant de bundle de l’application.

  • Chrome : utilisez le nom de l’extension Chrome.

  • Jeton d’inscription
    Un identifiant délivré par les serveurs de connexion GCM à l’application cliente qui lui permet de recevoir des messages. Notez que les jetons d’enregistrement doivent être gardés secrets.

Flux du cycle de vie

  • Inscrivez-vous pour activer GCM. Une instance d’une application cliente s’enregistre pour recevoir des messages. Pour en savoir plus, consultez Enregistrement des applications clientes.
  • Envoyer et recevoir des messages en aval.
  • Envoyer un message. Le serveur d’application envoie des messages à l’application client :
    1. The app server sends a message to GCM connection servers.
    2. The GCM connection server enqueues and stores the message if the device is offline.
    3. When the device is online, the GCM connection server sends the message to the device.
    4. On the device, the client app receives the message according to the platform-specific implementation. See your platform-specific documentation for details.
  • Recevoir un message. Une application cliente reçoit un message d’un serveur de connexion GCM. Consultez la documentation spécifique à votre plate-forme pour plus de détails sur la façon dont une application cliente dans cet environnement traite les messages qu’elle reçoit.
  • Envoyer et recevoir des messages en amont. Cette fonctionnalité n’est disponible que si vous utilisez le serveur de connexion XMPP.
  • Envoyer un message. Une application cliente envoie des messages au serveur d’applications :
    1. On the device, the client app sends messages to the XMPP connection server. See your platform-specific documentation for details on how a client app can send a message via XMPP.
    2. The XMPP connection server enqueues and stores the message if the server is disconnected.
    3. When the app server is re-connected, the XMPP connection server sends the message to the app server.
  • Recevoir un message. Un serveur d’applications reçoit un message du serveur de connexion XMPP, puis procède comme suit :
    1. Parses the message header to verify client app sender information.
    2. Sends “ack” to the XMPP connection server to acknowledge receiving the message.
    3. Optionally parses the message payload, as defined by the client app.
