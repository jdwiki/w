Amélioration de la sécurité des types à l’aide de nouveaux types #

Les types d’alias réduisent le passe-partout et améliorent la lisibilité, mais ils ne sont pas plus sûrs que le type alias lui-même. Considérer ce qui suit:

type alias Email = String

type alias Name = String

someEmail = "holmes@private.com"

someName = "Benedict"

sendEmail : Email -> Cmd msg
sendEmail email = ...

En utilisant le code ci-dessus, nous pouvons écrire sendEmail someName, et il compilera, même si ce n’est vraiment pas le cas, car bien que les noms et les e-mails soient tous deux des String, ce sont des choses complètement différentes.

Nous pouvons vraiment distinguer une String d’une autre String au niveau du type en créant un nouveau type. Voici un exemple qui réécrit Email en type plutôt qu’en type alias :

module Email exposing (Email, create, send)

type Email = EmailAddress String

isValid : String -> Bool
isValid email = 
  -- ...validation logic

create : String -> Maybe Email
create email =
    if isValid email then
        Just (EmailAddress email)
    else
        Nothing

send : Email -> Cmd msg
send (EmailAddress email) = ...

Notre fonction isValid fait quelque chose pour déterminer si une chaîne est une adresse e-mail valide. La fonction create vérifie si une String donnée est un e-mail valide, renvoyant un Maybe-wrapped Email pour s’assurer que nous ne renvoyons que des adresses validées. Bien que nous puissions contourner le contrôle de validation en construisant un Email directement en écrivant EmailAddress "somestring", si notre déclaration de module n’expose pas le constructeur EmailAddress, comme indiqué ici

module Email exposing (Email, create, send)

alors aucun autre module n’aura accès au constructeur EmailAddress, bien qu’ils puissent toujours utiliser le type Email dans les annotations. La seule façon de créer un nouvel Email en dehors de ce module est d’utiliser la fonction create qu’il fournit, et cette fonction garantit qu’elle ne renverra que des adresses e-mail valides en premier lieu. Par conséquent, cette API guide automatiquement l’utilisateur vers le bon chemin via sa sécurité de type : send ne fonctionne qu’avec des valeurs construites par create, qui effectue une validation et applique la gestion des e-mails invalides puisqu’elle renvoie un Maybe Email.

Si vous souhaitez exporter le constructeur Email, vous pouvez écrire

module Email exposing (Email(EmailAddress), create, send)

Désormais, tout fichier qui importe Email peut également importer son constructeur. Dans ce cas, cela permettrait aux utilisateurs d’éviter la validation et d’“envoyer” des e-mails non valides, mais vous ne construisez pas toujours une API comme celle-ci, donc l’exportation de constructeurs peut être utile. Avec un type qui a plusieurs constructeurs, vous pouvez également souhaiter n’exporter que certains d’entre eux.

Types de données comparables #

Les types comparables sont des types primitifs qui peuvent être comparés à l’aide des opérateurs de comparaison du module [Basics][1], tels que : (<), (>), (<=), (>=), max, min, comparer

Les types comparables dans Elm sont Int, Float, Time, Char, String et les tuples ou listes de types comparables.

Dans la documentation ou les définitions de type, elles sont désignées comme une variable de type spéciale “comparable”, par exemple. voir la définition de type pour la fonction Basics.max :

max : comparable -> comparable -> comparable

[1] : http://package.elm-lang.org/packages/elm-lang/core/4.0.3/Basics

Construction de types #

La combinaison de mots-clés type alias donne un nouveau nom pour un type, mais le mot-clé type isolément déclare un nouveau type. Examinons l’un des types les plus fondamentaux : Maybe

type Maybe a
    = Just a
    | Nothing

La première chose à noter est que le type Maybe est déclaré avec une [type variable](https://www.wikiod.com/fr/elm/types-variables-de-type-et-constructeurs-de-type /8809/type-variables) de a. La deuxième chose à noter est le caractère pipe, |, qui signifie “ou”. En d’autres termes, quelque chose de type Peut-être un est soit Juste un ou Rien.

Lorsque vous écrivez le code ci-dessus, Just et Nothing entrent dans la portée en tant que constructeurs de valeur, et Maybe entre dans la portée en tant que constructeur de type. Voici leurs signatures :

Just : a -> Maybe a

Nothing : Maybe a

Maybe : a -> Maybe a -- this can only be used in type signatures

En raison de la variable de type a, n’importe quel type peut être “enveloppé à l’intérieur” du type Peut-être. Ainsi, Maybe Int, Maybe (List String) ou Maybe (Maybe (List Html)) sont tous des types valides. Lors de la déstructuration d’une valeur type avec une expression case, vous devez tenir compte de chaque instanciation possible de ce type. Dans le cas d’une valeur de type “Peut-être un”, vous devez prendre en compte à la fois le cas “Juste un” et le cas “Rien”:

thing : Maybe Int
thing = 
    Just 3

blah : Int
blah =
    case thing of
        Just n -> 
            n

        Nothing ->
            42

-- blah = 3

Essayez d’écrire le code ci-dessus sans la clause Nothing dans l’expression case : il ne sera pas compilé. C’est ce qui fait du constructeur de type Maybe un excellent modèle pour exprimer des valeurs qui peuvent ne pas exister, car il vous oblige à gérer la logique du moment où la valeur est Nothing.

Le type Jamais #

Le type Never ne peut pas être construit (le module Basics n’a pas exporté son constructeur de valeur et ne vous a pas non plus donné d’autre fonction renvoyant Never). Il n’y a pas de valeur never : Never ni de fonction createNever : ?? -> Jamais.

Cela a ses avantages : vous pouvez encoder dans un système de type une possibilité qui ne peut pas se produire. Cela peut être vu dans des types comme Task Never Int qui garantit qu’il réussira avec un Int; ou Program Never qui ne prendra aucun paramètre lors de l’initialisation du code Elm à partir de JavaScript.

Taper les signatures #

Dans Elm, les valeurs sont déclarées en écrivant un nom, un signe égal, puis la valeur réelle :

someValue = 42

Les fonctions sont également des valeurs, en plus de prendre une valeur ou des valeurs comme arguments. Ils sont généralement écrits comme suit :

double n = n * 2

Chaque valeur dans Elm a un type. Les types des valeurs ci-dessus seront * déduits * par le compilateur en fonction de la façon dont ils sont utilisés. Mais il est recommandé de toujours déclarer explicitement le type de toute valeur de niveau supérieur, et pour ce faire, vous écrivez une signature de type comme suit :

someValue : Int
someValue = 
    42

someOtherValue : Float
someOtherValue =
    42

Comme nous pouvons le voir, 42 peut être défini comme soit un Int ou un Float. Cela a un sens intuitif, mais consultez Variables de type pour plus d’informations.

Les signatures de type sont particulièrement utiles lorsqu’elles sont utilisées avec des fonctions. Voici la fonction de doublement d’avant :

double : Int -> Int
double n =
    n * 2

Cette fois, la signature a un ->, une flèche, et nous prononçons la signature comme “int to int”, ou “prend un entier et renvoie un entier”. -> indique qu’en donnant à double une valeur Int comme argument, double renverra un Int. Par conséquent, il prend un entier à un entier:

> double
<function> : Int -> Int

> double 3
6 : Int

Types de base #

Dans elm-repl, tapez un morceau de code pour obtenir sa valeur et son type déduit. Essayez ce qui suit pour en savoir plus sur les différents types qui existent :

> 42
42 : number

> 1.987
1.987 : Float

> 42 / 2
21 : Float

> 42 % 2
0 : Int

> 'e'
'e' : Char

> "e"
"e" : String

> "Hello Friend"
"Hello Friend" : String

> ['w', 'o', 'a', 'h']
['w', 'o', 'a', 'h'] : List Char

> ("hey", 42.42, ['n', 'o'])
("hey", 42.42, ['n', 'o']) : ( String, Float, List Char )

> (1, 2.1, 3, 4.3, 'c')
(1,2.1,3,4.3,'c') : ( number, Float, number', Float, Char )

> {}
{} : {}

> { hey = "Hi", someNumber = 43 }
{ hey = "Hi", someNumber = 43 } : { hey : String, someNumber : number }

> ()
() : ()

{} est le type Record vide et () est le type Tuple vide. Ce dernier est souvent utilisé à des fins d’évaluation paresseuse. Voir l’exemple correspondant dans https://www.wikiod.com/fr/elm/fonctions-et-application-partielle

Notez que nombre apparaît sans majuscule. Cela indique qu’il s’agit d’une variable de type, et de plus le mot particulier number fait référence à une variable de type spécial qui peut être soit un Int soit un Float (voir les sections correspondantes pour Suite). Cependant, les types sont toujours en majuscules, tels que Char, Float, List String, etc.

## Variables de type spécial Elm définit les variables de type spécial suivantes qui ont une signification particulière pour le compilateur :

  • comparable : composé de Int, Float, Char, String et leurs tuples. Cela permet l’utilisation des opérateurs < et >.

Exemple : Vous pouvez définir une fonction pour rechercher les éléments les plus petits et les plus grands dans une liste (“extent”). Vous pensez quel type de signature écrire. D’une part, vous pouvez écrire extentInt : List Int -> Maybe (Int, Int) et extentChar : List Char -> Maybe (Char, Char) et un autre pour Float et String. La mise en œuvre de ceux-ci serait la même :

  extentInt list =
    let
      helper x (minimum, maximum) = 
        ((min minimum x), (max maximum x))
    in 
      case list of 
        [] ->
          Nothing
        x :: xs ->
          Just <| List.foldr helper (x, x) xs

 You might be tempted to simply write `extent : List a -> Maybe (a, a)`, but the compiler will not let you do this, because the functions `min` and `max` are not defined for these types (NB: these are just simple wrappers around the `<` operator mentioned above). You can solve this by defining `extent : List comparable -> Maybe (comparable, comparable)`. This allows your solution to be *polymorphic*, which just means that it will work for more than one type.
  • number : composé de Int et Float. Permet l’utilisation d’opérateurs arithmétiques à l’exception de la division. Vous pouvez ensuite définir par exemple sum : List number -> number et le faire fonctionner à la fois pour les entiers et les flottants.
  • appendable : composé de String, List. Autorise l’utilisation de l’opérateur ++.
  • compappend : cela apparaît parfois, mais c’est un détail d’implémentation du compilateur. Actuellement, cela ne peut pas être utilisé dans vos propres programmes, mais est parfois mentionné.

Notez que dans une annotation de type comme celle-ci : number -> number -> number, ils font tous référence au même type, donc passer Int -> Float -> Int serait une erreur de type. Vous pouvez résoudre ce problème en ajoutant un suffixe au nom de la variable de type : number -> number' -> number'' se compilerait alors correctement.

Il n’y a pas de nom officiel pour ceux-ci, ils sont parfois appelés :

  • Variables de type spécial
  • Variables de type typeclass
  • Classes de pseudo-type

En effet, ils fonctionnent comme les [Type Classes] de Haskell (https://www.wikiod.com/fr/haskell/classes-de-types), mais sans la possibilité pour l’utilisateur de les définir.

Variables de type #

Les variables de type sont des noms sans majuscule dans les signatures de type. Contrairement à leurs homologues en majuscules, tels que Int et String, ils ne représentent pas un seul type, mais plutôt n’importe quel type. Ils sont utilisés pour écrire des fonctions génériques qui peuvent fonctionner sur n’importe quel type ou types, et sont particulièrement utiles pour écrire des opérations sur des conteneurs comme List ou Dict. La fonction List.reverse, par exemple, a la signature suivante :

reverse : List a -> List a

Ce qui signifie qu’il peut fonctionner sur une liste de * n’importe quelle valeur de type *, donc List Int, List (List String), ceux-ci et tous les autres peuvent être inversés tout de même. Par conséquent, a est une variable de type qui peut remplacer n’importe quel type.

La fonction reverse aurait pu utiliser n’importe quel nom de variable sans majuscule dans sa signature de type, à l’exception de la poignée de noms de variable de type spécial, tels que number (voir l’exemple correspondant à ce sujet pour plus d’informations):

reverse : List lol -> List lol

reverse : List wakaFlaka -> List wakaFlaka

Les noms des variables de type ne deviennent significatifs que lorsqu’il y a des variables de type différentes dans une même signature, illustrée par la fonction map sur les listes :

map : (a -> b) -> List a -> List b

map prend une fonction de n’importe quel type a vers n’importe quel type b, avec une liste d’éléments d’un certain type a, et renvoie une liste d’éléments d’un certain type b, qu’il obtient en appliquant la fonction donnée à chaque élément de la liste.

Concrétisons la signature pour mieux voir ceci :

plusOne : Int -> Int
plusOne x = 
    x + 1

> List.map plusOne
<function> : List Int -> List Int

Comme nous pouvons le voir, a = Int et b = Int dans ce cas. Mais, si map avait une signature de type comme map : (a -> a) -> List a -> List a, alors cela fonctionnerait uniquement sur les fonctions qui opèrent sur un seul type, et vous ne jamais pouvoir changer le type d’une liste en utilisant la fonction map. Mais comme la signature de type de map a plusieurs variables de type différentes, a et b, nous pouvons utiliser map pour changer le type d’une liste :

isOdd : Int -> Bool
isOdd x =
    x % 2 /= 0

> List.map isOdd
<function> : List Int -> List Bool

Dans ce cas, a = Int et b = Bool. Par conséquent, pour pouvoir utiliser des fonctions qui peuvent prendre et renvoyer des types différents, vous devez utiliser des variables de type différentes.

Tapez les alias #

Parfois, nous voulons donner à un type un nom plus descriptif. Supposons que notre application ait un type de données représentant les utilisateurs :

{ name : String, age : Int, email : String }

Et nos fonctions sur les utilisateurs ont des signatures de type du type :

prettyPrintUser : { name : String, age : Int, email : String } -> String

Cela pourrait devenir assez compliqué avec un type d’enregistrement plus grand pour un utilisateur, alors utilisons un alias de type pour réduire la taille et donner un nom plus significatif à cette structure de données :

type alias User =
    { name: String
    , age : Int
    , email : String
    }


prettyPrintUser : User -> String

Les alias de type facilitent la définition et l’utilisation d’un modèle pour une application :

type alias Model =
    { count : Int
    , lastEditMade : Time
    }

L’utilisation de type alias ne fait qu’aliaser un type avec le nom que vous lui avez donné. L’utilisation du type Model ci-dessus est exactement la même que l’utilisation de { count : Int, lastEditMade : Time }. Voici un exemple montrant comment les alias ne sont pas différents des types sous-jacents :

type alias Bugatti = Int

type alias Fugazi = Int

unstoppableForceImmovableObject : Bugatti -> Fugazi -> Int
unstoppableForceImmovableObject bug fug =
    bug + fug

> unstoppableForceImmovableObject 09 87
96 : Int

Un alias de type pour un type d’enregistrement définit une fonction constructeur avec un argument pour chaque champ dans l’ordre de déclaration.

type alias Point = { x : Int, y : Int }

Point 3 7
{ x = 3, y = 7 } : Point

type alias Person = { last : String, middle : String, first : String }

Person "McNameface" "M" "Namey"
{ last = "McNameface", middle = "M", first = "Namey" } : Person

Chaque alias de type d’enregistrement a son propre ordre de champ, même pour un type compatible.

type alias Person = { last : String, middle : String, first : String }
type alias Person2 = { first : String, last : String, middle : String }

Person2 "Theodore" "Roosevelt" "-"
{ first = "Theodore", last = "Roosevelt", middle = "-" } : Person2

a = [ Person "Last" "Middle" "First", Person2 "First" "Last" "Middle" ]
[{ last = "Last", middle = "Middle", first = "First" },{ first = "First", last = "Last", middle = "Middle" }] : List Person2
