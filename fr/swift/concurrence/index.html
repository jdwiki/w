Syntaxe #

  • Rapide 3.0

  • DispatchQueue.main // Récupère la file d’attente principale

  • DispatchQueue(label : “my-serial-queue”, attributs : [.serial, .qosBackground]) // Créez votre propre file d’attente série privée

  • DispatchQueue.global(attributes: [.qosDefault]) // Accéder à l’une des files d’attente simultanées globales

  • DispatchQueue.main.async { … } // Distribue une tâche de manière asynchrone au thread principal

  • DispatchQueue.main.sync { … } // Distribue une tâche de manière synchrone au thread principal

  • DispatchQueue.main.asyncAfter(deadline: .now() + 3) { … } // Envoie une tâche de manière asynchrone au thread principal à exécuter après x secondes

  • Rapide < 3.0

  • dispatch_get_main_queue() // Récupère la file d’attente principale en cours d’exécution sur le thread principal

  • dispatch_get_global_queue(dispatch_queue_priority_t, 0) // Récupère la file d’attente globale avec la priorité spécifiée dispatch_queue_priority_t

  • dispatch_async(dispatch_queue_t) { () -> Void in … } // Distribue une tâche de manière asynchrone sur le dispatch_queue_t spécifié

  • dispatch_sync(dispatch_queue_t) { () -> Void in … } // Distribue une tâche de manière synchrone sur le dispatch_queue_t spécifié

  • dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(nanosecondes)), dispatch_queue_t, { … }); // Distribue une tâche sur le dispatch_queue_t spécifié après des nanosecondes

Obtention d’une file d’attente Grand Central Dispatch (GCD) #

Grand Central Dispatch fonctionne sur le concept de “files d’attente d’expédition”. Une file d’attente de répartition exécute les tâches que vous désignez dans l’ordre dans lequel elles sont transmises. Il existe trois types de files d’attente de répartition :

  • Les files d’attente de répartition en série (c’est-à-dire les files d’attente de répartition privées) exécutent une tâche à la fois, dans l’ordre. Ils sont fréquemment utilisés pour synchroniser l’accès à une ressource.
  • Les files d’attente de répartition simultanées (c’est-à-dire les files d’attente de répartition globales) exécutent une ou plusieurs tâches simultanément.
  • La Main Dispatch Queue exécute des tâches sur le thread principal.

Pour accéder à la file d’attente principale :

let mainQueue = DispatchQueue.main

let mainQueue = dispatch_get_main_queue()

Le système fournit des files d’attente de répartition globales * simultanées * (globales pour votre application), avec des priorités variables. Vous pouvez accéder à ces files d’attente à l’aide de la classe “DispatchQueue” dans Swift 3 :

let globalConcurrentQueue = DispatchQueue.global(qos: .default)

équivalent à

let globalConcurrentQueue = DispatchQueue.global()

let globalConcurrentQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0)

Dans iOS 8 ou version ultérieure, les valeurs de qualité de service pouvant être transmises sont .userInteractive, .userInitiated, .default, .utility et .background. Celles-ci remplacent les constantes DISPATCH_QUEUE_PRIORITY_.

Vous pouvez également créer vos propres files d’attente avec différentes priorités :

let myConcurrentQueue = DispatchQueue(label: "my-concurrent-queue", qos: .userInitiated, attributes: [.concurrent], autoreleaseFrequency: .workItem, target: nil)
let mySerialQueue = DispatchQueue(label: "my-serial-queue", qos: .background, attributes: [], autoreleaseFrequency: .workItem, target: nil)

let myConcurrentQueue = dispatch_queue_create("my-concurrent-queue", DISPATCH_QUEUE_CONCURRENT)
let mySerialQueue = dispatch_queue_create("my-serial-queue", DISPATCH_QUEUE_SERIAL)

Dans Swift 3, les files d’attente créées avec cet initialiseur sont en série par défaut, et le fait de transmettre .workItem pour la fréquence de publication automatique garantit qu’un pool de publication automatique est créé et vidé pour chaque élément de travail. Il y a aussi .never, ce qui signifie que vous gérerez vous-même vos propres pools de libération automatique, ou .inherit qui hérite du paramètre de l’environnement. Dans la plupart des cas, vous n’utiliserez probablement pas .never sauf en cas de personnalisation extrême.

Boucles simultanées #

GCD fournit un mécanisme pour effectuer une boucle, les boucles se produisant simultanément les unes par rapport aux autres. Ceci est très utile lors de l’exécution d’une série de calculs coûteux en calculs.

Considérez cette boucle :

for index in 0 ..< iterations {
    // Do something computationally expensive here
}

Vous pouvez effectuer ces calculs simultanément en utilisant concurrentPerform (dans Swift 3) ou dispatch_apply (dans Swift 2) :

DispatchQueue.concurrentPerform(iterations: iterations) { index in
    // Do something computationally expensive here
}

dispatch_apply(iterations, queue) { index in
    // Do something computationally expensive here
}

La fermeture de la boucle sera invoquée pour chaque index de 0 à, mais non compris, itérations. Ces itérations seront exécutées simultanément les unes par rapport aux autres, et donc l’ordre dans lequel elles s’exécutent n’est pas garanti. Le nombre réel d’itérations qui se produisent simultanément à un moment donné est généralement dicté par les capacités de l’appareil en question (par exemple, combien de cœurs l’appareil possède-t-il).

Quelques considérations spéciales :

  • Le concurrentPerform/dispatch_apply peut exécuter les boucles simultanément les unes par rapport aux autres, mais tout cela se produit de manière synchrone par rapport au thread à partir duquel vous l’appelez. Donc, n’appelez pas ceci depuis le thread principal, car cela bloquera ce thread jusqu’à ce que la boucle soit terminée.

  • Étant donné que ces boucles se produisent simultanément les unes par rapport aux autres, vous êtes responsable de la sécurité des threads des résultats. Par exemple, si vous mettez à jour un dictionnaire avec les résultats de ces calculs coûteux en calculs, assurez-vous de synchroniser vous-même ces mises à jour.

  • Notez qu’il y a une surcharge associée à l’exécution de boucles simultanées. Ainsi, si les calculs effectués à l’intérieur de la boucle ne sont pas suffisamment intensifs en termes de calcul, vous pouvez constater que toute performance obtenue en utilisant des boucles simultanées peut être diminuée, voire complètement compensée, par la surcharge associée à la synchronisation de tous ces threads simultanés.

    So, you are responsible determining the correct amount of work to be performed in each iteration of the loop. If the calculations are too simple, you may employ “striding” to include more work per loop. For example, rather than doing a concurrent loop with 1 million trivial calculations, you may do 100 iterations in your loop, doing 10,000 calculations per loop. That way there is enough work being performed on each thread, so the overhead associated with managing these concurrent loops becomes less significant.

Exécution de tâches dans une file d’attente Grand Central Dispatch (GCD) #

Pour exécuter des tâches sur une file d’attente de distribution, utilisez les méthodes sync, async et after.

Pour répartir une tâche dans une file d’attente de manière asynchrone :

let queue = DispatchQueue(label: "myQueueName")

queue.async {
    //do something
    
    DispatchQueue.main.async {
        //this will be called in main thread
        //any UI updates should be placed here
    }
}
// ... code here will execute immediately, before the task finished

Pour répartir une tâche dans une file d’attente de manière synchrone :

queue.sync {
    // Do some task
}
// ... code here will not execute until the task is finished

Pour répartir une tâche dans une file d’attente après un certain nombre de secondes :

queue.asyncAfter(deadline: .now() + 3) {
    //this will be executed in a background-thread after 3 seconds
}
// ... code here will execute immediately, before the task finished

REMARQUE : Toute mise à jour de l’interface utilisateur doit être appelée sur le thread principal ! Assurez-vous de mettre le code pour les mises à jour de l’interface utilisateur dans DispatchQueue.main.async { ... }

Types de file d’attente :

let mainQueue = dispatch_get_main_queue()
let highQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_HIGH, 0)
let backgroundQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0)

Pour répartir une tâche dans une file d’attente de manière asynchrone :

dispatch_async(queue) {
    // Your code run run asynchronously. Code is queued and executed 
    // at some point in the future.
}
// Code after the async block will execute immediately

Pour répartir une tâche dans une file d’attente de manière synchrone :

dispatch_sync(queue) {
    // Your sync code
}
// Code after the sync block will wait until the sync task finished

Pour répartir une tâche après un intervalle de temps (utilisez NSEC_PER_SEC pour convertir les secondes en nanosecondes) :

dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(2.5 * Double(NSEC_PER_SEC))), dispatch_get_main_queue()) {
    // Code to be performed in 2.5 seconds here
}

Pour exécuter une tâche de manière asynchrone et mettre à jour l’interface utilisateur :

dispatch_async(queue) {
    // Your time consuming code here
    dispatch_async(dispatch_get_main_queue()) {
        // Update the UI code 
    }
}

REMARQUE : Toute mise à jour de l’interface utilisateur doit être appelée sur le thread principal ! Assurez-vous de mettre le code pour les mises à jour de l’interface utilisateur dans dispatch_async(dispatch_get_main_queue()) { ... }

Exécution de tâches dans une file d’attente d’opérations #

Vous pouvez considérer une OperationQueue comme une ligne de tâches en attente d’exécution. Contrairement aux files d’attente de répartition dans GCD, les files d’attente d’opérations ne sont pas FIFO (premier entré, premier sorti). Au lieu de cela, ils exécutent les tâches dès qu’elles sont prêtes à être exécutées, tant qu’il y a suffisamment de ressources système pour le permettre.

Obtenez la OperationQueue principale :

let mainQueue = OperationQueue.main

Créez une OperationQueue personnalisée :

let queue = OperationQueue()
queue.name = "My Queue"
queue.qualityOfService = .default

La qualité de service spécifie l’importance du travail, ou combien l’utilisateur est susceptible de compter sur les résultats immédiats de la tâche.

Ajoutez une Operation à une OperationQueue :

// An instance of some Operation subclass
let operation = BlockOperation {
    // perform task here
}

queue.addOperation(operation)

Ajoutez un bloc à une OperationQueue :

myQueue.addOperation {
    // some task
}

Ajoutez plusieurs Operations à une OperationQueue :

let operations = [Operation]()
// Fill array with Operations

myQueue.addOperation(operations)

Ajustez le nombre d’opérations pouvant être exécutées simultanément dans la file d’attente :

myQueue.maxConcurrentOperationCount = 3 // 3 operations may execute at once

// Sets number of concurrent operations based on current system conditions
myQueue.maxConcurrentOperationCount = NSOperationQueueDefaultMaxConcurrentOperationCount

La suspension d’une file d’attente l’empêchera de démarrer l’exécution de toute opération existante non démarrée ou de toute nouvelle opération ajoutée à la file d’attente. La façon de reprendre cette file d’attente est de remettre isSuspended à false :

myQueue.isSuspended = true

// Re-enable execution
myQueue.isSuspended = false

La suspension d’une OperationQueue n’arrête pas ou n’annule pas les opérations qui sont déjà en cours d’exécution. Il ne faut tenter de suspendre qu’une file d’attente que vous avez créée, pas les files d’attente globales ou la file d’attente principale.

Création d’opérations de haut niveau #

Le framework Foundation fournit le type “Operation”, qui représente un objet de haut niveau qui encapsule une partie du travail pouvant être exécuté dans une file d’attente. Non seulement la file d’attente coordonne les performances de ces opérations, mais vous pouvez également établir des dépendances entre les opérations, créer des opérations annulables, limiter le degré de simultanéité utilisé par la file d’attente des opérations, etc.

Operation est prêt à être exécuté lorsque toutes ses dépendances ont fini de s’exécuter. La propriété isReady passe alors à true.

Créez une sous-classe “Operation” simple et non concurrente :

class MyOperation: Operation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

class MyOperation: NSOperation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

Ajoutez une opération à une OperationQueue :

myQueue.addOperation(operation)

Cela exécutera l’opération simultanément sur la file d’attente.

Gérer les dépendances sur une Operation.

Les dépendances définissent d’autres “opérations” qui doivent s’exécuter dans une file d’attente avant que cette “opération” soit considérée comme prête à être exécutée.

operation2.addDependency(operation1)

operation2.removeDependency(operation1)

Exécutez une “opération” sans file d’attente :

   operation.start()

Les dépendances seront ignorées. S’il s’agit d’une opération simultanée, la tâche peut toujours être exécutée simultanément si sa méthode “start” décharge le travail dans les files d’attente d’arrière-plan.

Opérations simultanées.

Si la tâche qu’une Operation doit effectuer est elle-même asynchrone (par exemple, une tâche de données URLSession), vous devez implémenter l’Operation en tant qu’opération simultanée. Dans ce cas, votre implémentation isAsynchronous devrait renvoyer true, vous auriez généralement la méthode start qui effectue une configuration, puis appelle sa méthode main qui exécute réellement la tâche.

Lorsque l’implémentation d’une Operation asynchrone commence, vous devez implémenter les méthodes isExecuting, isFinished et KVO. Ainsi, lorsque l’exécution démarre, la propriété isExecuting devient true. Lorsqu’une Operation termine sa tâche, isExecuting est défini sur false et isFinished est défini sur true. Si l’opération est annulée, isCancelled et isFinished deviennent true. Toutes ces propriétés sont observables par valeur-clé.

Annuler une Opération.

Appeler cancel change simplement la propriété isCancelled en true. Pour répondre à une annulation depuis votre propre sous-classe Operation, vous devez vérifier la valeur de isCancelled au moins périodiquement dans main et répondre de manière appropriée.

operation.cancel()
