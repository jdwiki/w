Utilisez une carte de module pour importer des en-têtes C #

Un module map peut simplement “importer monmodule” en le configurant pour lire les fichiers d’en-tête C et les faire apparaître comme Swift les fonctions.

Placez un fichier nommé module.modulemap dans un répertoire nommé mymodule :

Dans le fichier de mappage de module :

// mymodule/module.modulemap
module mymodule {
    header "defs.h"
}

Ensuite, “importez” le module :

// demo.swift
import mymodule
print("Empty color: \(Color())")

Utilisez le drapeau -I répertoire pour indiquer à swiftc où trouver le module :

swiftc -I . demo.swift   # "-I ." means "search for modules in the current directory"

Pour plus d’informations sur la syntaxe des mappages de modules, consultez la documentation Clang sur les mappages de modules.

Utilisation des classes Objective-C à partir du code Swift #

Si MyFramework contient des classes Objective-C dans ses en-têtes publics (et l’en-tête parapluie), alors import MyFramework est tout ce qui est nécessaire pour les utiliser depuis Swift.

En-têtes de pont #

Un en-tête de pont rend les déclarations Objective-C et C supplémentaires visibles pour le code Swift. Lors de l’ajout de fichiers de projet, Xcode peut proposer de créer automatiquement un en-tête de pont :

![boîte de dialogue d’en-tête de pont][1]

Pour en créer un manuellement, modifiez le paramètre de construction Objective-C Bridging Header :

[![entrez la description de l’image ici][2]][2]

Dans l’en-tête de pontage, importez les fichiers nécessaires à utiliser à partir du code :

// MyApp-Bridging-Header.h
#import "MyClass.h"  // allows code in this module to use MyClass

Interface générée #

Cliquez sur le bouton Articles associés (ou appuyez sur ⌃1), puis sélectionnez Interface générée pour voir l’interface Swift qui sera générée à partir d’un en-tête Objective-C.

Spécifiez un en-tête de pontage pour swiftc #

Le drapeau -import-objc-header spécifie un en-tête pour swiftc à importer :

// defs.h
struct Color {
    int red, green, blue;
};

#define MAX_VALUE 255

// demo.swift
extension Color: CustomStringConvertible {  // extension on a C struct
    public var description: String {
        return "Color(red: \(red), green: \(green), blue: \(blue))"
    }
}
print("MAX_VALUE is: \(MAX_VALUE)")  // C macro becomes a constant
let color = Color(red: 0xCA, green: 0xCA, blue: 0xD0)  // C struct initializer
print("The color is \(color)")

$ swiftc demo.swift -import-objc-header defs.h && ./demo
MAX_VALUE est : 255
La couleur est Couleur (rouge : 202, vert : 202, bleu : 208)

Interopérabilité fine entre Objective-C et Swift #

Lorsqu’une API est marquée avec NS_REFINED_FOR_SWIFT, elle sera précédée de deux traits de soulignement (__) lors de son importation dans Swift :

@interface MaClasse : NSObject
- (NSInteger)indexOfObject :(id)obj NS_REFINED_FOR_SWIFT ;
@end

L’[interface générée](https://www.wikiod.com/fr/swift/travailler-avec-c-et-objective-c#Utilisation des classes Objective-C à partir du code Swift 201605140416386495969) ressemble à ceci :

classe publique MaClasse : NSObject {
    public func __indexOfObject(obj: AnyObject) -> Int
}

Vous pouvez maintenant remplacer l’API par une extension plus “Swifty”. Dans ce cas, nous pouvons utiliser une valeur de retour facultatif, filtrant [NSNotFound](https://developer.apple.com /library/mac/documentation/Cocoa/Reference/Foundation/Miscellaneous/Foundation_Constants/#//apple_ref/doc/constant_group/NSNotFound):

extension MaClasse {
    // Rather than returning NSNotFound if the object doesn't exist,
    // this "refined" API returns nil.
    func indexOfObject(obj: AnyObject) -> Int? {
        let idx = __indexOfObject(obj)
        if idx == NSNotFound { return nil }
        return idx
    }
}

// Code Swift, en utilisant "if let" comme il se doit :
soit monobj = MaClass()
si laissez idx = monobj.indexOfObject(quelque chose) {
    // do something with idx
}

Dans la plupart des cas, vous souhaiterez peut-être restreindre le fait qu’un argument d’une fonction Objective-C puisse ou non être “nil”. Cela se fait à l’aide du mot-clé _Nonnull, qui qualifie tout pointeur ou référence de bloc :

annuler
doStuff(const void *const _Données Nonnull, void (^_Complétion Nonnull)())
{
    // complex asynchronous code
}

Avec cela écrit, le compilateur doit émettre une erreur chaque fois que nous essayons de passer “nil” à cette fonction à partir de notre code Swift :

doStuff(
    nil,  // error: nil is not compatible with expected argument type 'UnsafeRawPointer'
    nil)  // error: nil is not compatible with expected argument type '() -> Void'

L’opposé de _Nonnull est _Nullable, ce qui signifie qu’il est acceptable de passer nil dans cet argument. _Nullable est également la valeur par défaut ; cependant, le spécifier explicitement permet un code plus auto-documenté et à l’épreuve du temps.

Pour aider davantage le compilateur à optimiser votre code, vous pouvez également spécifier si le bloc s’échappe :

annuler
callNow(__attribute__((noescape)) void (^_Nonnull f)())
{
    // f is not stored anywhere
}

Avec cet attribut, nous nous engageons à ne pas enregistrer la référence du bloc et à ne pas appeler le bloc une fois l’exécution de la fonction terminée.

Utiliser la bibliothèque standard C #

L’interopérabilité C de Swift vous permet d’utiliser des fonctions et des types de la bibliothèque standard C.

Sous Linux, la bibliothèque standard C est exposée via le module Glibc ; sur les plates-formes Apple, il s’appelle Darwin.

#if os(macOS) || os(iOS) || os(tvOS) || os(watchOS)
import Darwin
#elseif os(Linux)
import Glibc
#endif

// use open(), read(), and other libc features

Utilisation des classes Swift à partir du code Objective-C #

Dans le même module #

Dans un module nommé “MyModule”, Xcode génère un en-tête nommé MyModule-Swift.h qui expose les classes Swift publiques à Objective-C. Importez cet en-tête afin d’utiliser les classes Swift :

// MySwiftClass.swift dans MyApp
Fondation d'importation

// La classe doit être `public` pour être visible, sauf si cette cible a également un en-tête de pontage
Classe publique MySwiftClass : NSObject {
    // ...
}
// MyViewController.m dans MyApp

#import "MaVueContrôleur.h"
#import "MyApp-Swift.h" // importe l'interface générée
#import <MyFramework/MyFramework-Swift.h> // ou utilisez des chevrons pour une cible de cadre

@implementation MyViewController
- (vide) démo {
    [[MySwiftClass alloc] init];  // use the Swift class
}
@end

Paramètres de compilation pertinents :

  • Objective-C Generated Interface Header Name : contrôle le nom de l’en-tête Obj-C généré.
  • Install Objective-C Compatibility Header : si l’en-tête -Swift.h doit être un en-tête public (pour les cibles du framework).

[![Capture d’écran des paramètres de compilation][1]][1]

Dans un autre module #

L’utilisation de @import MyFramework; importe le module entier, y compris les interfaces Obj-C vers les classes Swift (si le paramètre de construction susmentionné est activé).
