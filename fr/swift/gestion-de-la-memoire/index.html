Gestion de la mémoire

  • Cycles de référence et références faibles
  • Gestion manuelle de la mémoire

Sur cette page

  • Cycles de référence et références faibles
  • Gestion manuelle de la mémoire

Cette rubrique décrit comment et quand le runtime Swift doit allouer de la mémoire pour les structures de données d’application, et quand cette mémoire doit être récupérée.

Par défaut, les instances de la classe de sauvegarde de la mémoire sont gérées via le comptage des références. Les structures sont toujours transmises par copie. Pour désactiver le schéma de gestion de la mémoire intégré, on peut utiliser la structure [Unmanaged][1].

[1] : https://developer.apple.com/reference/swift/unmanaged

Quand utiliser le mot-clé faible :

Le mot clé `weak` doit être utilisé si un objet référencé peut être désalloué pendant la durée de vie de l'objet contenant la référence.

Quand utiliser le mot-clé sans propriétaire :

Le mot-clé `unowned` doit être utilisé si un objet référencé ne devrait pas être désalloué pendant la durée de vie de l'objet contenant la référence.

Pièges

Une erreur fréquente est d'oublier de créer des références à des objets, qui doivent continuer à vivre après la fin d'une fonction, comme les gestionnaires d'emplacement, les gestionnaires de mouvement, etc.

Exemple:

class A : CLLocationManagerDelegate
{
    init()
    {
        let locationManager = CLLocationManager()
        locationManager.delegate = self
        locationManager.startLocationUpdates()
    }
}

Cet exemple ne fonctionnera pas correctement, car le gestionnaire d’emplacement est désalloué après le retour de l’initialiseur. La bonne solution consiste à créer une référence forte en tant que variable d’instance :

class A : CLLocationManagerDelegate
{
    let locationManager:CLLocationManager

    init()
    {
        locationManager = CLLocationManager()
        locationManager.delegate = self
        locationManager.startLocationUpdates()
    }
}

Cycles de référence et références faibles #

Le cycle de référence (ou cycle de rétention) est ainsi nommé car il indique un cycle dans le [graphe d’objets](https:/ /en.wikipedia.org/wiki/Cycle_(graph_theory)) /en.wikipedia.org/wiki/Object_graph):

Chaque flèche indique un objet conservant un autre (une référence forte). À moins que le cycle ne soit rompu, la mémoire de ces objets ne sera jamais libérée.

Un cycle de conservation est créé lorsque deux instances de classes se référencent :

class A { var b: B? = nil }
class B { var a: A? = nil }

let a = A()
let b = B()

a.b = b  // a retains b
b.a = a  // b retains a -- a reference cycle

Les deux instances resteront en vie jusqu’à la fin du programme. Il s’agit d’un cycle de rétention.

Références faibles #

Pour éviter les cycles de rétention, utilisez le mot clé weak ou unowned lors de la création de références pour rompre les cycles de rétention.

classe B { faible var a : A ? = néant }

Les références faibles ou sans propriétaire n’augmenteront pas le nombre de références d’une instance. Ces références ne contribuent pas à retenir les cycles. La référence faible devient nil lorsque l’objet auquel elle fait référence est désalloué.

a.b = b  // a retains b
b.a = a  // b holds a weak reference to a -- not a reference cycle

Lorsque vous travaillez avec des fermetures, vous pouvez également utiliser [weak et unowned dans les listes de capture](https://www.wikiod.com/fr/swift/fermetures#Captures, références fortes/faibles et cycles de conservation cycles de conservation#t=201607211834265333132).

Gestion manuelle de la mémoire #

Lors de l’interfaçage avec les API C, on peut souhaiter reculer le compteur de références Swift. Cela est réalisé avec des objets non gérés.

Si vous devez fournir un pointeur typé vers une fonction C, utilisez la méthode toOpaque de la structure Unmanaged pour obtenir un pointeur brut, et fromOpaque pour récupérer l’instance d’origine :

setupDisplayLink() {
  let pointerToSelf: UnsafeRawPointer = Unmanaged.passUnretained(self).toOpaque()
  CVDisplayLinkSetOutputCallback(self.displayLink, self.redraw, pointerToSelf)
}

func redraw(pointerToSelf: UnsafeRawPointer, /* args omitted */) {
  let recoveredSelf = Unmanaged<Self>.fromOpaque(pointerToSelf).takeUnretainedValue()
  recoveredSelf.doRedraw()
}

Notez que, si vous utilisez passUnretained et ses homologues, il est nécessaire de prendre toutes les précautions comme avec les références unowned.

Pour interagir avec les API Objective-C héritées, on peut vouloir affecter manuellement le nombre de références d’un certain objet. Pour cela Unmanaged a les méthodes respectives retain et release. Néanmoins, il est plus souhaitable d’utiliser passRetained et takeRetainedValue, qui effectuent la rétention avant de renvoyer le résultat :

func preferredFilenameExtension(for uti: String) -> String! {
  let result = UTTypeCopyPreferredTagWithClass(uti, kUTTagClassFilenameExtension)
  guard result != nil else { return nil }

  return result!.takeRetainedValue() as String
}

Ces solutions doivent toujours être le dernier recours et les API natives du langage doivent toujours être préférées.
