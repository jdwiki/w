Principes de base de la gestion des erreurs #

Les fonctions de Swift peuvent renvoyer des valeurs, lancer des erreurs, ou les deux :

func reticulateSplines() // pas de valeur de retour et pas d'erreur
func reticulateSplines() -> Int // retourne toujours une valeur
func reticulateSplines() throws // aucune valeur de retour, mais peut générer une erreur
func reticulateSplines() throws -> Int // peut soit retourner une valeur soit lancer une erreur

Toute valeur conforme au protocole ErrorType (y compris les objets NSError) peut être générée comme une erreur. Les énumérations offrent un moyen pratique de définir des erreurs personnalisées :

enum NetworkError: ErrorType {
    case Offline
    case ServerError(String)
}

enum NetworkError: Error {
    // Swift 3 dictates that enum cases should be `lowerCamelCase`
    case offline
    case serverError(String)
}

Une erreur indique un échec non fatal pendant l’exécution du programme et est gérée avec les constructions de flux de contrôle spécialisées do/catch, throw et try.

func fetchResource(ressource : NSURL) lance -> Chaîne {
    if let (statusCode, responseString) = /* ...from elsewhere...*/ {
        if case 500..<600 = statusCode {
            throw NetworkError.serverError(responseString)
        } else {
            return responseString
        }
    } else {
        throw NetworkError.offline
    }
}

Les erreurs peuvent être détectées avec do/catch :

faire {
    let response = try fetchResource(resURL)
    // If fetchResource() didn't throw an error, execution continues here:
    print("Got response: \(response)")
    ...
} attraper {
    // If an error is thrown, we can handle it here.
    print("Whoops, couldn't fetch resource: \(error)")
}

Toute fonction pouvant générer une erreur doit être appelée en utilisant try, try? ou try! :

// erreur : l'appel peut lancer mais n'est pas marqué avec 'try'
let réponse = récupérerResource(resURL)

// "try" fonctionne dans do/catch, ou dans une autre fonction de lancement :
faire {
    let response = try fetchResource(resURL)
} attraper {
    // Handle the error
}

func foo() lance {
    // If an error is thrown, continue passing it up to the caller.
    let response = try fetchResource(resURL)
}

// "essayer?" enveloppe la valeur de retour de la fonction dans un Optional (nil si une erreur a été renvoyée).
si let réponse = essayer ? fetchResource(resURL) {
    // no error was thrown
}

// "essayer!" plante le programme à l'exécution si une erreur se produit.
laissez réponse = essayez ! fetchResource(resURL)

Capture de différents types d’erreurs #

Créons notre propre type d’erreur pour cet exemple.

enum CustomError: ErrorType {
    case SomeError
    case AnotherError
}

func throwing() throws {
    throw CustomError.SomeError
}

enum CustomError: Error {
    case someError
    case anotherError
}

func throwing() throws {
    throw CustomError.someError
}

La syntaxe Do-Catch permet d’attraper une erreur lancée, et crée automatiquement une constante nommée error disponible dans le bloc catch :

do {
    try throwing()
} catch {
    print(error)
}

Vous pouvez également déclarer vous-même une variable :

do {
    try throwing()
} catch let oops {
    print(oops)
}

Il est également possible d’enchaîner différentes instructions catch. C’est pratique si plusieurs types d’erreurs peuvent être générés dans le bloc Do.

Ici, le Do-Catch tentera d’abord de convertir l’erreur en CustomError, puis en NSError si le type personnalisé ne correspond pas.

do {
    try somethingMayThrow()
} catch let custom as CustomError {
    print(custom)
} catch let error as NSError {
    print(error)
}

Dans Swift 3, pas besoin de downcaster explicitement en NSError.

do {
    try somethingMayThrow()
} catch let custom as CustomError {
    print(custom)
} catch {
    print(error)
}

Catch and Switch Pattern pour la gestion explicite des erreurs #

class Plane {
    
    enum Emergency: ErrorType {
        case NoFuel
        case EngineFailure(reason: String)
        case DamagedWing
    }

    var fuelInKilograms: Int

    //... init and other methods not shown

    func fly() throws {
        // ...
        if fuelInKilograms <= 0 {
            // uh oh...
            throw Emergency.NoFuel
        }
    }

}

Dans la classe client :

let airforceOne = Plane()
do {
    try airforceOne.fly()
} catch let emergency as Plane.Emergency {
    switch emergency {
    case .NoFuel:
        // call nearest airport for emergency landing
    case .EngineFailure(let reason):
        print(reason) // let the mechanic know the reason
    case .DamagedWing:
        // Assess the damage and determine if the president can make it
    }
}

Désactivation de la propagation des erreurs #

Les créateurs de Swift ont mis beaucoup d’attention à rendre le langage expressif et la gestion des erreurs est exactement cela, expressive. Si vous essayez d’invoquer une fonction qui peut générer une erreur, l’appel de la fonction doit être précédé du mot-clé try. Le mot-clé try n’est pas magique. Tout ce qu’il fait, c’est rendre le développeur conscient de la capacité de lancement de la fonction.

Par exemple, le code suivant utilise une fonction loadImage(atPath:), qui charge la ressource image à un chemin donné ou génère une erreur si l’image ne peut pas être chargée. Dans ce cas, étant donné que l’image est livrée avec l’application, aucune erreur ne sera générée lors de l’exécution, il convient donc de désactiver la propagation des erreurs.

let photo = try! loadImage(atPath: "./Resources/John Appleseed.jpg")

Créer une erreur personnalisée avec une description localisée #

Créer une “énumération” d’erreurs personnalisées

enum RegistrationError: Error {
    case invalidEmail
    case invalidPassword
    case invalidPhoneNumber
}

Créez une extension de RegistrationError pour gérer la description localisée.

extension RegistrationError: LocalizedError {
    public var errorDescription: String? {
        switch self {
        case .invalidEmail:
            return NSLocalizedString("Description of invalid email address", comment: "Invalid Email")
        case .invalidPassword:
            return NSLocalizedString("Description of invalid password", comment: "Invalid Password")
        case .invalidPhoneNumber:
            return NSLocalizedString("Description of invalid phoneNumber", comment: "Invalid Phone Number")
        }
    }
}

Erreur de gestion :

let error: Error = RegistrationError.invalidEmail
print(error.localizedDescription)
