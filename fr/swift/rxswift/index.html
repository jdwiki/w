Éliminer #

Après la création de l’abonnement, il est important de gérer sa désallocation correcte.

Les docs nous ont dit que

Si une séquence se termine dans un temps fini, ne pas appeler dispose ou ne pas utiliser addDisposableTo(disposeBag) ne causera aucune fuite permanente de ressources. Cependant, ces ressources seront utilisées jusqu’à la fin de la séquence, soit en terminant la production d’éléments, soit en renvoyant une erreur.

Il existe deux manières de désallouer des ressources.

  1. Utilisation de l’opérateur disposeBags et addDisposableTo.
  2. Utilisation de l’opérateur “takeUntil”.

Dans le premier cas, vous transmettez manuellement l’abonnement à l’objet “DisposeBag”, qui efface correctement toute la mémoire utilisée.

let bag = DisposeBag()
Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(bag)

Vous n’avez pas besoin de créer des DisposeBags dans chaque classe que vous créez, jetez simplement un coup d’œil au projet de RxSwift Community nommé [NSObject+Rx][1]. En utilisant le framework, le code ci-dessus peut être réécrit comme suit :

Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(rx_disposeBag)

Dans le second cas, si l’heure de souscription coïncide avec la durée de vie de l’objet self, il est possible d’implémenter la disposition en utilisant takeUntil(rx_deallocated) :

let _ = sequence
    .takeUntil(rx_deallocated)
    .subscribe {
        print($0)
    }

[1] : https://github.com/RxSwiftCommunity/NSObject-Rx

Les bases de RxSwift #

FRP, ou Functional Reactive Programming, a quelques termes de base que vous devez connaître.

Chaque élément de données peut être représenté comme “Observable”, qui est un flux de données asynchrone. La puissance de FRP réside dans la représentation d’événements synchrones et asynchrones sous forme de flux, “Observables”, et en fournissant la même interface pour travailler avec.

Habituellement, Observable contient plusieurs (ou aucun) événements contenant la date - les événements .Next, puis il peut être terminé avec succès (.Success) ou avec une erreur (.Error).

Jetons un coup d’œil au diagramme de marbre suivant :

--(1)--(2)--(3)|-->

Dans cet exemple, il y a un flux de valeurs “Int”. Au fil du temps, trois événements “.Next” se sont produits, puis le flux s’est terminé avec succès.

--X->

Le diagramme ci-dessus montre un cas où aucune donnée n’a été émise et l’événement .Error termine le Observable.

Avant de poursuivre, voici quelques ressources utiles :

  1. [RxSwift][1]. Regardez des exemples, lisez des documents et commencez.
  2. [RxSwift Slack room][2] dispose de quelques canaux pour la résolution de problèmes éducatifs.
  3. Jouez avec [RxMarbles][3] pour savoir ce que fait l’opérateur et lequel est le plus utile dans votre cas.
  4. Jetez un œil [sur cet exemple][4], explorez le code par vous-même.

[1] : https://github.com/ReactiveX/RxSwift [2] : http://rxswift.slack.com [3] : http://rxmarbres.com/ [4] : https://github.com/scotteg/RxSwiftPlayer

Création d’observables #

RxSwift propose de nombreuses façons de créer un Observable, jetons un coup d’œil :

import RxSwift

let intObservale = Observable.just(123) // Observable<Int>
let stringObservale = Observable.just("RxSwift") // Observable<String>
let doubleObservale = Observable.just(3.14) // Observable<Double>

Ainsi, les observables sont créés. Ils ne contiennent qu’une seule valeur et se terminent ensuite avec succès. Néanmoins, rien ne se passe après sa création. Pourquoi?

Il y a deux étapes dans le travail avec Observables : vous observez quelque chose pour créer un flux, puis vous vous abonnez au flux ou le liez à quelque chose pour interagir avec lui.

Observable.just(12).subscribe {
    print($0)
}

La console imprimera :

.Next(12)
.Completed()

Et si je ne m’intéressais qu’à travailler avec des données, qui ont lieu dans des événements .Next, j’utiliserais l’opérateur subscribeNext :

Observable.just(12).subscribeNext {
    print($0) // prints "12" now
}

Si je veux créer un observable de plusieurs valeurs, j’utilise différents opérateurs :

Observable.of(1,2,3,4,5).subscribeNext {
    print($0)
}
// 1
// 2
// 3
// 4
// 5

// I can represent existing data types as Observables also:
[1,2,3,4,5].asObservable().subscribeNext { 
    print($0) 
}
// result is the same as before.

Et enfin, peut-être que je veux un “Observable” qui fonctionne. Par exemple, il est pratique d’encapsuler une opération réseau dans Observable<SomeResultType>. Jetons un coup d’œil à ce que l’on peut faire :

Observable.create { observer in    // create an Observable ...
    MyNetworkService.doSomeWorkWithCompletion { (result, error) in
        if let e = error {
            observer.onError(e)    // ..that either holds an error
        } else {
            observer.onNext(result) // ..or emits the data
            observer.onCompleted()  // ..and terminates successfully.
        }
    }
    return NopDisposable.instance // here you can manually free any resources
                                //in case if this observable is being disposed.
}

Liaisons #

Observable.combineLatest(firstName.rx_text, lastName.rx_text) { $0 + " " + $1 }
.map { "Greetings, \($0)" }
.bindTo(greetingLabel.rx_text)

En utilisant l’opérateur combineLatest chaque fois qu’un élément est émis par l’un des deux Observables, combinez le dernier élément émis par chaque Observable. Ainsi, de cette manière, nous combinons le résultat des deux UITextField en créant un nouveau message avec le texte "Greetings, \($0)" en utilisant l’interpolation de chaîne pour se lier ultérieurement au texte d’un UILabel.

Nous pouvons lier des données à n’importe quel UITableView et UICollectionView de manière très simple :

viewModel
.rows
.bindTo(resultsTableView.rx_itemsWithCellIdentifier("WikipediaSearchCell", cellType: WikipediaSearchCell.self)) { (_, viewModel, cell) in
    cell.title = viewModel.title
    cell.url = viewModel.url
}
.addDisposableTo(disposeBag)

C’est un wrapper Rx autour de la méthode de source de données cellForRowAtIndexPath. Et aussi Rx s’occupe de l’implémentation de numberOfRowsAtIndexPath, qui est une méthode requise au sens traditionnel, mais vous n’avez pas à l’implémenter ici, c’est pris en charge.

RxCocoa et ControlEvents #

RxSwift fournit non seulement les moyens de contrôler vos données, mais également de représenter les actions des utilisateurs de manière réactive.

RxCocoa contient tout ce dont vous avez besoin. Il encapsule la plupart des propriétés des composants de l’interface utilisateur dans Observables, mais pas vraiment. Il existe des Observables mis à jour appelés ControlEvents (qui représentent des événements) et ControlProperties (qui représentent des propriétés, surprise !). Ces éléments contiennent des flux “observables” sous le capot, mais présentent également certaines nuances :

  • Il n’échoue jamais, donc pas d’erreurs.
  • La séquence sera “complète” lorsque le contrôle sera désalloué.
  • Il délivre des événements sur le thread principal (MainScheduler.instance).

En gros, vous pouvez travailler avec eux comme d’habitude :

button.rx_tap.subscribeNext { _ in   // control event
    print("User tapped the button!")
}.addDisposableTo(bag)

textField.rx_text.subscribeNext { text in // control property
    print("The textfield contains: \(text)")
}.addDisposableTo(bag)
// notice that ControlProperty generates .Next event on subscription
// In this case, the log will display 
// "The textfield contains: "
// at the very start of the app.

C’est très important à utiliser : tant que vous utilisez Rx, oubliez les éléments @IBAction, tout ce dont vous avez besoin peut être lié et configuré en même temps. Par exemple, la méthode viewDidLoad de votre contrôleur de vue est un bon candidat pour décrire le fonctionnement des composants de l’interface utilisateur.

Ok, un autre exemple : supposons que nous ayons un champ de texte, un bouton et une étiquette. Nous voulons valider le texte dans le champ de texte lorsque nous appuyons sur le bouton, et afficher les résultats dans l’étiquette. Oui, cela ressemble à une autre tâche de validation d’e-mail, hein ?

Tout d’abord, nous saisissons le ControlEvent button.rx_tap :

----()-----------------------()----->

Ici, les parenthèses vides montrent les taps de l’utilisateur. Ensuite, nous prenons ce qui est écrit dans le textField avec l’opérateur withLatestFrom (regardez-le [ici] [1], imaginez que le flux supérieur représente les taps de l’utilisateur, celui du bas représente le texte dans le champ de texte).

button.rx_tap.withLatestFrom(textField.rx_text)

----("")--------------------("123")--->
//  ^ tap   ^ i wrote 123    ^ tap

Sympa, nous avons un flux de chaînes à valider, émis uniquement lorsque nous avons besoin de valider.

Tout Observable a des opérateurs familiers tels que map ou filter, nous prendrons map pour valider le texte. Créez vous-même la fonction validateEmail, utilisez n’importe quelle regex que vous voulez.

button.rx_tap                                // ControlEvent<Void>
        .withLatestFrom(textField.rx_text)   // Observable<String>
        .map(validateEmail)                  // Observable<Bool>
        .map { (isCorrect) in
            return isCorrect ? "Email is correct" : "Input the correct one, please"
        }                                    // Observable<String>
        .bindTo(label.rx_text)              
        .addDisposableTo(bag)

Fait! Si vous avez besoin de plus de logique personnalisée (comme afficher des vues d’erreur en cas d’erreur, effectuer une transition vers un autre écran en cas de succès…), abonnez-vous simplement au flux final “Bool” et écrivez-le ici.

[1] : http://rxmarbles.com/#withLatestFrom
