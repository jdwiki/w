Le routage est la façon dont ASP.NET MVC fait correspondre un URI à une action. Le module de routage est responsable du mappage des demandes de navigateur entrantes sur des actions de contrôleur MVC particulières.

MVC 5 prend en charge un nouveau type de routage, appelé routage d’attribut. Comme son nom l’indique, le routage d’attributs utilise des attributs pour définir des routes. Le routage des attributs vous donne plus de contrôle sur les URI de votre application Web.

Routage des attributs dans MVC #

Parallèlement à la méthode classique de définition de route, les frameworks MVC WEB API 2, puis MVC 5 ont introduit le “routage d’attributs”:

public class RouteConfig
{
    public static void RegisterRoutes(RouteCollection routes)
    {
        routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
 
        // This enables attribute routing and must go  before other routes are added to the routing table.
        // This makes attribute routes have higher priority
        routes.MapMvcAttributeRoutes();  
    }
}

Pour les routes avec le même préfixe à l’intérieur d’un contrôleur, vous pouvez définir un préfixe commun pour toutes les méthodes d’action à l’intérieur du contrôleur à l’aide de l’attribut RoutePrefix.

[RoutePrefix("Custom")]
public class CustomController : Controller
{
    [Route("Index")]
    public ActionResult Index()
    {
        ...
    }
}

RoutePrefix est facultatif et définit la partie de l’URL qui est préfixée à toutes les actions du contrôleur.

Si vous avez plusieurs itinéraires, vous pouvez définir un itinéraire par défaut en capturant l’action en tant que paramètre, puis l’appliquer à l’ensemble du contrôleur, sauf si un attribut “Route” spécifique est défini sur certaines méthodes d’action qui remplacent l’itinéraire par défaut.

[RoutePrefix("Custom")]
[Route("{action=index}")]
public class CustomController : Controller
{
    public ActionResult Index()
    {
        ...
    }

    public ActionResult Detail()
    {
        ...
    }
}

Itinéraire fourre-tout #

Supposons que nous voulions avoir une route qui autorise un nombre illimité de segments comme ceci :

  • http://example.com/Products/ (voir tous les produits)
  • http://exemple.com/Produits/IT
  • http://example.com/Products/IT/Laptops
  • http://example.com/Products/IT/Laptops/Ultrabook
  • http://example.com/Products/IT/Laptops/Ultrabook/Asus
  • etc.

Nous aurions besoin d’ajouter une route, normalement à la fin de la table de routage car cela intercepterait probablement toutes les requêtes, comme ceci :

routes.MapRoute("Final", "Route/{*segments}",
      new { controller = "Product", action = "View" });

Dans le contrôleur, une action qui pourrait gérer cela pourrait être :

public void ActionResult View(string[] segments /* <- the name of the parameter must match the name of the route parameter */)
{
    // use the segments to obtain information about the product or category and produce data to the user
    // ...
}

Notions de base sur le routage #

Ajout d’une route personnalisée dans Mvc #

L’utilisateur peut ajouter une route personnalisée, mappant une URL à une action spécifique dans un contrôleur. Ceci est utilisé à des fins d’optimisation des moteurs de recherche et rend les URL lisibles.

routes.MapRoute(
  name: "AboutUsAspx", // Route name
  url: "AboutUs.aspx",  // URL with parameters
  defaults: new { controller = "Home", action = "AboutUs", id = UrlParameter.Optional }  // Parameter defaults
);

Route fourre-tout pour activer le routage côté client #

C’est une bonne pratique d’encoder l’état de l’application à page unique (SPA) dans l’url :

my-app.com/admin-spa/users/edit/id123

Cela permet de sauvegarder et de partager l’état de l’application.
Lorsque l’utilisateur met l’URL dans la barre d’adresse du navigateur et qu’il clique sur Entrée, le serveur doit ignorer la partie côté client de l’URL demandée. Si vous servez votre SPA en tant que vue Razor rendue (résultat de l’appel de l’action du contrôleur) plutôt qu’en tant que fichier html statique, vous pouvez utiliser une route fourre-tout :

public class AdminSpaController
{
    [Route("~/admin-spa/{clienSidePart*}")]
    ActionResult AdminSpa()
    {
        ...
    }
}

Dans ce cas, le serveur renvoie uniquement SPA, puis il s’initialise en fonction de la route. Cette approche est plus flexible car elle ne dépend pas du module [url-rewrite][1].

[1] : http://www.iis.net/learn/extensions/url-rewrite-module/creating-rewrite-rules-for-the-url-rewrite-module

Routage des attributs dans les zones #

Pour utiliser le routage d’attributs dans les zones, l’enregistrement des zones et des définitions [RouteArea(...)] est requis.

Dans RouteConfig.cs :

public class RouteConfig
{
    public static void RegisterRoutes(RouteCollection routes)
    {
        routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
        routes.MapMvcAttributeRoutes();
        AreaRegistration.RegisterAllAreas();
    }
}

Dans un exemple de définition de routage d’attribut de contrôleur de zone :

[RouteArea("AreaName", AreaPrefix = "AreaName")]
[RoutePrefix("SampleAreaController")]
public class SampleAreaController : Controller
{
    [Route("Index")]
    public ActionResult Index()
    {
        return View();
    }
}

Pour utiliser les liens Url.Action dans les zones :

@Url.Action("Index", "SampleAreaController", new { area = "AreaName" })

Routage personnalisé #

Le routage personnalisé fournit un besoin spécialisé de routage pour gérer des demandes entrantes spécifiques.

Afin de définir des routes personnalisées, gardez à l’esprit que l’ordre des routes que vous ajoutez à la table de routage est important.

public static void RegisterRoutes(RouteCollection routes)
{
    routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
    
    // this is an advanced custom route
    // you can define custom URL with custom parameter(s) point to certain action method
    routes.MapRoute(
    "CustomEntry", // Route name
    "Custom/{entryId}", // Route pattern
    new { controller = "Custom", action = "Entry" } // Default values for defined parameters above
    );

    // this is a basic custom route
    // any custom routes take place on top before default route
    routes.MapRoute(
    "CustomRoute", // Route name
    "Custom/{controller}/{action}/{id}", // Route pattern
    new { controller = "Custom", action = "Index", id = UrlParameter.Optional } // Default values for defined parameters above
    );

    routes.MapRoute(
    "Default", // Route name
    "{controller}/{action}/{id}", // Route pattern
    new { controller = "Home", action = "Index", id = UrlParameter.Optional } // Default values for defined parameters above
    );
}

Les noms controller et action sont réservés. Par défaut, MVC mappe la partie {controller} de l’URL à la classe <controller>Controller, puis recherche une méthode avec le nom <action> sans ajouter de suffixes.

Bien qu’il puisse être tentant de créer une famille de routes en utilisant le modèle {controller}/{action}/{parameter}, considérez qu’en faisant cela, vous divulguez la structure de votre application et rendez les URL quelque peu fragiles en changeant le nom du contrôleur modifie l’itinéraire et rompt les liens enregistrés par l’utilisateur.

Préférez un paramètre de routage explicite :

routes.MapRoute(
    "CustomRoute", // Route name
    "Custom/Index/{id}", // Route pattern
    new { controller = "Custom", action = nameof(CustomController.Index), id = UrlParameter.Optional }
);

(vous ne pouvez pas utiliser l’opérateur nameof pour le nom du contrôleur car il aura un suffixe supplémentaire Controller) qui doit être omis lors de la définition du nom du contrôleur dans la route.
