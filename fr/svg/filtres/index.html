Syntaxe #

  • Déclaration de filtre : <filter id="filter-id"> …liste des enfants primitives … </filter>
  • Appliquer le filtre via l’attribut SVG : <elementname filter="url(#filter-id)" ... />
  • Appliquer le filtre via CSS propriété : (-prefix-)filter : url("#filter-id");

Paramètres #

Attributs d’élément | Détails | ——————-|—————————— ————————————-| Région de filtre | L’élément de filtre peut éventuellement définir la position, les dimensions, la résolution et les unités pour la sortie d’un effet de filtre. La position et les dimensions d’un filtre peuvent être spécifiées à l’aide des paramètres suivants : x, y, largeur, hauteur. Les valeurs par défaut ne sont pas intuitives et sont : x : -10% y : -10% largeur : 120% hauteur : 120% | Résolution du filtre |L’attribut filterRes est un attribut optionnel dans SVG 1.1 qui peut être utilisé pour spécifier la résolution à laquelle le filtre est traité. Cet attribut avait une prise en charge inégale et est désormais obsolète dans les navigateurs actuels. | Unités de filtre|Par défaut, les unités et le système de coordonnées de la région des effets de filtre (x,y,largeur,hauteur) d’un élément de filtre sont définis par rapport à la boîte englobante de l’élément référençant le filtre. En termes SVG, cela s’appelle “objectBoundingBox”. Lorsque nous écrivons x=“50%”, cela signifie “définir la position x de départ de la région de filtre sur le côté gauche de la boîte englobante de l’élément de référence + 50 % de la largeur de l’élément”. Mais vous pouvez également spécifier explicitement les unités et les coordonnées en définissant la propriété filterUnits. Les deux alternatives sont “objectBoundingBox” (la valeur par défaut que nous venons de décrire) ou “userSpaceOnUse”. userSpaceOnUse définit les unités de filtre et le système de coordonnées sur le canevas de l’élément de référence, ou en termes SVG, le “userSpaceOnUse”. | Unités primitives | En plus du système d’unités pour le filtre lui-même, vous pouvez également spécifier le système d’unités que les primitives de filtre enfants du filtre utiliseront via l’attribut primitiveUnits. Encore une fois, le choix est entre userSpaceOnUse et objectBoundingBox. Celles-ci affectent les coordonnées 0,0 et les valeurs unitaires des primitives de filtre de la même manière que pour filterUnits. | Espace colorimétrique | L’espace colorimétrique par défaut pour les filtres SVG est linearRGB. L’attribut facultatif color-interpolation-filters peut être défini sur sRGB pour modifier l’espace colorimétrique vers l’espace sRGB plus conventionnel. |

La plupart des attributs de filtre peuvent être animés via l’élément <animate>, bien que vous deviez utiliser la bibliothèque “fakeSMIL” sur IE pour obtenir les mêmes résultats. L’animation SMIL (l’élément <animate>) sera obsolète au profit de la nouvelle spécification Web Animations - qui a une prise en charge très limitée à la mi-2016.

Les éléments enfants de l’élément Filter - primitives de filtre - ont deux attributs facultatifs qui spécifient l’espace colorimétrique dans lequel les calculs d’interpolation de couleur sont effectués : color-interpolation et color-interpolation-filters. La valeur par défaut pour le premier est sRGB et la valeur par défaut pour le second est linearRGB. Les manipulations qui inversent l’espace colorimétrique (via feColorMatrix ou feComponentTransfer) peuvent entraîner des résultats non intuitifs - pour ceux qui sont habitués à l’espace colorimétrique CSS sRGB. Par exemple, une inversion de couleur d’une image en niveaux de gris en linearRGB entraînera un décalage prononcé vers des tons plus blancs. Cela peut être corrigé en définissant la valeur de la primitive sur sRGB. Ces attributs peuvent être définis sur les primitives de filtre individuelles ou hérités de l’élément de filtre lui-même.

Si aucune autre entrée n’est spécifiée, mais qu’une est requise, la première primitive de filtre dans un filtre prendra une version pixellisée (bitmap) de l’élément de référence comme entrée. Les primitives de filtre suivantes qui attendent une entrée prendront comme entrée le résultat de la primitive de filtre immédiatement précédente.

Dans les filtres complexes, il peut devenir difficile de suivre (et de déboguer) les entrées et les sorties si elles sont laissées implicites ; et il est recommandé de déclarer explicitement les entrées et les sorties pour chaque primitive.

Les primitives de filtre SVG peuvent être familièrement divisées en entrées, transformations, effets d’éclairage et combinaisons.

Contributions:

feFlood : génère un champ de couleur

feTurbulence : génère une grande variété d’effets sonores

feImage : génère une image à partir d’une référence d’image externe, d’un URI de données ou d’un objet référence (les références d’objet ne sont pas prises en charge dans Firefox à partir de la mi-décembre 2012)

Transformation :

feColorMatrix : transforme les valeurs d’entrée d’un pixel RBGA en valeurs de sortie

feComponentTransfer : ajuste la courbe de couleur d’un canal de couleur individuel

feConvolveMatrix : remplace chaque pixel par un nouveau pixel calculé à partir des valeurs de pixel dans une zone relative au pixel actuel)

feGaussianBlur : remplace le pixel actuel par une moyenne pondérée de pixels dans une zone autour du pixel

feDisplacementMap : déplace chaque pixel de sa position actuelle en fonction des valeurs R, G ou B d’un autre graphique d’entrée.

feMorphology : remplace chaque pixel par un nouveau pixel calculé à partir de la valeur maximale ou minimale de tous les pixels dans une zone rectangulaire autour de ce pixel.

feOffset : déplace l’entrée de sa position actuelle

Effets d’éclairage :

feSpecularLighting : fournit un effet d’éclairage 2D ou pseudo-3D “brillant”

feDiffuseLighting : fournit un effet d’éclairage “mat” 2D ou pseudo-3D

feDistantLight : fournit une source de lumière distante pour un éclairage spéculaire ou diffus

feSpotLight : fournit une source lumineuse à section conique pour un éclairage spéculaire ou diffus

fePointLight : fournit une source lumineuse ponctuelle pour un éclairage spéculaire ou diffus

Combinaisons :

feMerge : crée un simple surcomposite à partir de plusieurs entrées (y compris les entrées de filtre précédentes)

feBlend : mélange plusieurs entrées à l’aide de règles de mélange

feComposite : combine plusieurs entrées à l’aide de règles de combinaison définies, en tenant compte des valeurs alpha.

feTile : saisie de mosaïques pour créer un motif répétitif

Autres notes

Bien que SVG soit une technologie graphique vectorielle, il est important de souligner que les filtres SVG effectuent des opérations au niveau du pixel sur toutes les entrées (y compris les formes SVG) et produisent des sorties pixellisées (bitmap) à un niveau de résolution spécifié. L’application d’une transformation d’échelle 10x (par exemple) sur une courbe SVG simple qui a été filtrée à une résolution d’écran normale produira des bords pixélisés puisque l’anticrénelage du graphique d’origine a été converti en pixels par le filtre et mis à l’échelle. (On ne sait pas si cela est conforme aux spécifications ou simplement une limitation des implémentations actuelles)

N’oubliez pas que SVG est XML lorsque vous écrivez des filtres, donc toutes les balises doivent être fermées et de nombreuses propriétés et attributs doivent être spécifiés explicitement ou le filtre ne s’exécutera pas.

Un élément de filtre n’est jamais rendu directement. Il est uniquement référencé à l’aide de la propriété filter sur l’élément auquel le filtre est appliqué. Notez que la propriété display ne s’applique pas à l’élément de filtre et que les éléments ne sont pas rendus directement même si la propriété display est définie sur une valeur autre que “none”. Inversement, les éléments de filtre sont disponibles pour le référencement même lorsque la propriété display de l’élément de filtre ou de l’un de ses ancêtres est définie sur “aucun”.

Les filtres SVG peuvent être référencés via un filtre CSS, bien qu’à la mi-2016, seul un sous-ensemble de primitives soit pris en charge via ce mécanisme, et ce mécanisme n’est pas pris en charge dans les navigateurs Microsoft.

Filtres d’ombre : ombre portée complexe (contournée, bruyante, en forme) #

<svg width="800px" height="600px">
<defs>
<filter id="complex-shadow" color-interpolation-filters="sRGB" x="-50%" y="-50%" height="200%" width="200%">

<!-- Take source alpha, offset it by angle/distance and blur it by size -->
<feOffset id="offset" in="SourceAlpha" dx="11" dy="6" result="SA-offset"/> 
<feGaussianBlur id="blur" in="SA-offset" stdDeviation="4" result="SA-o-blur"/>

<!-- Apply a contour by using a color curve transform on the alpha and clipping the result to the input -->

<feComponentTransfer in="SA-o-blur" result="SA-o-b-contIN"> 
  <feFuncA id="contour" type="table" tableValues="0 1 .3 .1 0.05 .1 .3 1 "/> 
</feComponentTransfer>

<feComposite operator="in" in="SA-o-blur" in2="SA-o-b-contIN" result="SA-o-b-cont"/>

<!-- Adjust the spread by multiplying alpha by a constant factor --> <feComponentTransfer in="SA-o-b-cont" result="SA-o-b-c-sprd"> 
  <feFuncA id="spread-ctrl" type="linear" slope="2.8"/> 
</feComponentTransfer>

<!-- Adjust color and opacity by adding fixed offsets and an opacity multiplier --> 
<feColorMatrix id="recolor" in="SA-o-b-c-sprd" type="matrix" values="0 0 0 0 0.945 0 0 0 0 0.137 0 0 0 0 0.137 0 0 0 0.49 0" result="SA-o-b-c-s-recolor"/>

<!-- Generate a grainy noise input with baseFrequency between approx .5 to 2.0. And add the noise with k1 and k2 multipliers that sum to 1 -->

              <feFuncA type="discrete" tableValues="0 0 0 0 0 0 0 1"/>
           </feComponentTransfer>
           <feComposite operator="in" in="SourceGraphic" in2="highlightMask" result="highlights"/>
