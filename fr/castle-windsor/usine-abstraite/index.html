Usine de validateur d’adresse - nécessite un paramètre pour sélectionner une implémentation #

Scénario : vous devez sélectionner une implémentation de la validation d’adresse lorsqu’une commande client est soumise, et le validateur est déterminé par le pays vers lequel la commande est expédiée. La fabrique doit inspecter une valeur passée en argument pour sélectionner la bonne implémentation.

Tout d’abord, écrivez une interface pour la fabrique :

public interface IAddressValidatorFactory
{
    IAddressValidator GetAddressValidator(Address address);
    void Release(IAddressValidator created);
}

L’interface indique que l’implémentation de IAddressValidator sera déterminée par Address passé en argument à GetAddressValidator.

Lorsque nous enregistrons plusieurs implémentations de la même interface avec Windsor, nous les nommons généralement. La fabrique devra donc renvoyer le nom d’une implémentation. Dans ce cas, supposons que nous ayons plusieurs implémentations de IAddressValidator enregistrées avec notre conteneur :

container.Register(
    Component.For<IAddressValidator,UnitedStatesAddressValidator>()
        .Named("AddressValidatorFor_USA"),
    Component.For<IAddressValidator, FinlandAddressValidator>()
        .Named("AddressValidatorFor_FIN"),
    Component.For<IAddressValidator, MalawiAddressValidator>()
        .Named("AddressValidatorFor_MWI"),
    Component.For<IAddressValidator, CommonCountryAddressValidator>()
        .Named("FallbackCountryAddressValidator")
        .IsDefault()            
    );

Le travail de notre usine consistera à prendre une “adresse” et à renvoyer le nom de l’implémentation que Windsor résoudra. Cela nécessite un sélecteur de composants.

public class AddressValidatorSelector : DefaultTypedFactoryComponentSelector
{
    public AddressValidatorSelector() 
        : base(fallbackToResolveByTypeIfNameNotFound: true) { }

    protected override string GetComponentName(MethodInfo method, object[] arguments)
    {
        return "AddressValidatorFor_" + ((Address)arguments[0]).CountryCode;
    }
}

Cette classe indique à Windsor de rechercher une implémentation de IAddressValidator nommée en fonction du code pays de la commande. Pour la Finlande, c’est “AddressValidatorFor_Fin”. Et s’il n’y a pas de validateur pour ce pays spécifique, nous pouvons revenir à un validateur par défaut. Nous pourrions également utiliser le même validateur pour plusieurs pays en enregistrant la même implémentation plusieurs fois avec des noms différents.

Nous avons écrit le code du sélecteur de composants, mais nous n’écrivons pas le code de l’usine elle-même. Windsor fournit l’usine, et nous lui disons d’utiliser notre AddressValidatorSelector.

// Add the factory facility once.
container.AddFacility<TypedFactoryFacility>();
container.Register(
    Component.For<IAddressValidatorFactory>()
        .AsFactory(new AddressValidatorSelector()));

Lorsqu’une classe dépend de IAddressValidatorFactory, Windsor injecte sa propre implémentation à l’aide du sélecteur de composants que nous avons fourni.

Usine abstraite simple - l’usine ne prend aucun paramètre #

Scénario : Vous devez résoudre une dépendance lorsqu’une méthode est appelée, pas dans le constructeur.

Solution : injecter une usine abstraite dans le constructeur. Lorsque la méthode est appelée, elle demande la dépendance à la fabrique abstraite, qui à son tour la résout à partir du conteneur. (Votre classe dépend de la fabrique mais n’appelle jamais le conteneur lui-même.)

Déclarez une interface pour votre usine* :

public interface IFooFactory
{
    IFoo CreateFoo();
    void Release(IFoo foo);
}

Ajoutez TypedFactoryFacility à votre conteneur :

container.AddFacility<TypedFactoryFacility>();

Demandez au conteneur d’utiliser TypedFactoryFacility pour résoudre les dépendances sur IFooFactory :

container.Register(
    Component.For<IFooFactory>().AsFactory(),
    Component.For<IFoo, MyFooImplementation>());

Vous n’avez pas besoin de créer une instance de fabrique. Windsor fait ça.

Vous pouvez maintenant injecter la fabrique dans une classe et l’utiliser comme ceci :

public class NeedsFooFactory
{
    private readonly IFooFactory _fooFactory;

    public NeedsFooFactory(IFooFactory fooFactory)
    {
        _fooFactory = fooFactory;
    }

    public void MethodThatNeedsFoo()
    {
        var foo = _fooFactory.CreateFoo();
        foo.DoWhatAFooDoes();
        _fooFactory.Release(foo);
    }
}

L’appel de la méthode Release oblige le conteneur à libérer le composant qu’il a résolu. Sinon, il ne sera pas publié tant que _fooFactory ne sera pas publié (c’est-à-dire chaque fois que NeedsFooFactory sera publié.)

*Windsor déduit quelle méthode est la méthode “créer” et laquelle est la méthode “libérer”. Si une méthode renvoie quelque chose, on suppose que le conteneur doit le résoudre. Si une méthode ne renvoie rien (void) alors c’est la méthode “release”.
