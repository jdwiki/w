Qu’est-ce qu’un comportement indéfini (UB) ? Selon la norme ISO C++ (§1.3.24, N4296), il s’agit d’un “comportement pour lequel cette norme internationale n’impose aucune exigence”.

Cela signifie que lorsqu’un programme rencontre UB, il est autorisé à faire ce qu’il veut. Cela signifie souvent un crash, mais cela peut simplement ne rien faire, faire voler des démons par le nez, ou même apparaître pour travaille correctement!

Inutile de dire que vous devriez éviter d’écrire du code qui invoque UB.

Si un programme contient un comportement indéfini, le standard C++ n’impose aucune contrainte sur son comportement.

  • Il peut sembler fonctionner comme prévu par le développeur, mais il peut également planter ou produire des résultats étranges.
  • Le comportement peut varier entre les exécutions du même programme.
  • Toute partie du programme peut mal fonctionner, y compris les lignes qui précèdent la ligne qui contient un comportement indéfini.
  • L’implémentation n’est pas tenue de documenter le résultat d’un comportement indéfini.

Une implémentation peut documenter le résultat d’une opération qui produit un comportement indéfini selon la norme, mais un programme qui dépend d’un tel comportement documenté n’est pas portable.

** Pourquoi un comportement indéfini existe-t-il ? **

Intuitivement, un comportement indéfini est considéré comme une mauvaise chose car de telles erreurs ne peuvent pas être gérées gracieusement via, par exemple, des gestionnaires d’exceptions.

Mais laisser certains comportements indéfinis fait en fait partie intégrante de la promesse de C++ “vous ne payez pas pour ce que vous n’utilisez pas”. Un comportement indéfini permet à un compilateur de supposer que le développeur sait ce qu’il fait et de ne pas introduire de code pour vérifier les erreurs mises en évidence dans les exemples ci-dessus.

Rechercher et éviter les comportements indéfinis

Certains outils peuvent être utilisés pour découvrir un comportement indéfini pendant le développement :

  • La plupart des compilateurs ont des drapeaux d’avertissement pour avertir de certains cas de comportement indéfini au moment de la compilation.
  • Les nouvelles versions de gcc et de clang incluent un indicateur dit “Undefined Behavior Sanitizer” (-fsanitize=undefined) qui vérifiera le comportement indéfini au moment de l’exécution, au détriment des performances.
  • Les outils de type lint peuvent effectuer une analyse de comportement indéfinie plus approfondie.

Comportement non défini, non spécifié et [défini par la mise en œuvre][1]

De la norme C++14 (ISO/IEC 14882:2014) section 1.9 (Exécution du programme) :

  1. Les descriptions sémantiques de la présente Norme internationale définissent une machine abstraite non déterministe paramétrée. [COUPER]

  2. Certains aspects et opérations de la machine abstraite sont décrits dans la présente Norme internationale comme définis par l’implémentation (par exemple, sizeof(int)). Ceux-ci constituent les paramètres de la machine abstraite. Chaque implémentation doit inclure une documentation décrivant ses caractéristiques et son comportement à ces égards. [COUPER]

  3. Certains autres aspects et opérations de la machine abstraite sont décrits dans la présente Norme internationale comme étant non spécifiés (par exemple, l’évaluation d’expressions dans un new-initializer si la fonction d’allocation échoue à allouer de la mémoire). Dans la mesure du possible, la présente Norme internationale définit un ensemble de comportements admissibles. Celles-ci définissent les aspects non déterministes de la machine abstraite. Une instance de la machine abstraite peut donc avoir plus d’une exécution possible pour un programme donné et une entrée donnée.

  4. Certaines autres opérations sont décrites dans la présente Norme internationale comme indéfinies (ou, par exemple, l’effet d’une tentative de modification d’un objet const). [ Remarque : la présente Norme internationale n’impose aucune exigence sur le comportement des programmes qui contiennent un comportement indéfini. —note de fin ]

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/1363/implementation-defined-behavior

Lecture ou écriture via un pointeur nul #

int *ptr = nullptr;
*ptr = 1; // Undefined behavior

Il s’agit d’un comportement indéfini, car un pointeur nul ne pointe sur aucun objet valide, il n’y a donc pas d’objet à *ptr sur lequel écrire.

Bien que cela provoque le plus souvent un défaut de segmentation, il n’est pas défini et tout peut arriver.

Utilisation d’une variable locale non initialisée #

int a;
std::cout << a; // Undefined behavior!

Il en résulte un comportement indéfini, car a n’est pas initialisé.

On prétend souvent, à tort, que c’est parce que la valeur est “indéterminée” ou “quelle que soit la valeur qui se trouvait auparavant dans cet emplacement mémoire”. Cependant, c’est l’acte d’accéder à la valeur de “a” dans l’exemple ci-dessus qui donne un comportement indéfini. En pratique, l’impression d’une “valeur poubelle” est un symptôme courant dans ce cas, mais ce n’est qu’une forme possible de comportement indéfini.

Bien que très peu probable dans la pratique (puisqu’il dépend d’un support matériel spécifique), le compilateur pourrait également électrocuter le programmeur lors de la compilation de l’exemple de code ci-dessus. Avec un tel compilateur et un tel support matériel, une telle réponse à un comportement indéfini augmenterait considérablement la compréhension du programmeur moyen (vivant) de la véritable signification du comportement indéfini - à savoir que la norme n’impose aucune contrainte sur le comportement résultant.

L’utilisation d’une valeur indéterminée de type unsigned char ne produit pas de comportement indéfini si la valeur est utilisée comme :

  • le deuxième ou le troisième opérande de l’opérateur conditionnel ternaire ;
  • l’opérande droit de l’opérateur virgule intégré ;
  • l’opérande d’une conversion en unsigned char ;
  • l’opérande droit de l’opérateur d’affectation, si l’opérande gauche est également de type unsigned char ;
  • l’initialiseur pour un objet unsigned char ;

ou si la valeur est ignorée. Dans de tels cas, la valeur indéterminée se propage simplement au résultat de l’expression, le cas échéant.

Notez qu’une variable “statique” est toujours initialisée à zéro (si possible) :

static int a;
std::cout << a; // Defined behavior, 'a' is 0

Accéder à un index hors limites #

C’est un comportement indéfini d’accéder à un index qui est hors limites pour un tableau (ou un conteneur de bibliothèque standard d’ailleurs, car ils sont tous implémentés à l’aide d’un tableau raw):

 int array[] = {1, 2, 3, 4, 5};
 array[5] = 0;  // Undefined behavior

Il est autorisé d’avoir un pointeur pointant vers la fin du tableau (dans ce cas tableau + 5), vous ne pouvez pas le déréférencer, car ce n’est pas un élément valide.

 const int *end = array + 5;  // Pointer to one past the last index
 for (int *p = array; p != end; ++p)
   // Do something with `p`

En général, vous n’êtes pas autorisé à créer un pointeur hors limites. Un pointeur doit pointer vers un élément dans le tableau, ou un après la fin.

Suppression d’un objet dérivé via un pointeur vers une classe de base qui n’a pas de destructeur virtuel. #

class base { };
class derived: public base { }; 

int main() {
    base* p = new derived();
    delete p; // The is undefined behavior!
}

Dans la section [expr.delete] §5.3.5/3, la norme indique que si delete est appelé sur un objet dont le type statique n’a pas de destructeur virtual :

si le type statique de l’objet à supprimer est différent de son type dynamique, le type statique doit être une classe de base du type dynamique de l’objet à supprimer et le le type statique doit avoir un destructeur virtuel ou le comportement n’est pas défini.

C’est le cas indépendamment de la question de savoir si la classe dérivée a ajouté des membres de données à la classe de base.

Extension de l’espace de noms std ou posix #

La norme (17.6.4.2.1/1) interdit généralement d’étendre l’espace de noms std :

Le comportement d’un programme C++ est indéfini s’il ajoute des déclarations ou des définitions à l’espace de noms std ou à un espace de noms dans l’espace de noms std, sauf indication contraire.

Il en va de même pour “posix” (17.6.4.2.2/1) :

Le comportement d’un programme C++ est indéfini s’il ajoute des déclarations ou des définitions à l’espace de noms posix ou à un espace de noms dans l’espace de noms posix, sauf indication contraire.

Considérer ce qui suit:

#include <algorithm>

namespace std
{
    int foo(){}
}

Rien dans la norme n’interdit “l’algorithme” (ou l’un des en-têtes qu’il inclut) définissant la même définition, et donc ce code violerait la règle de définition unique.

Donc, en général, c’est interdit. Il existe cependant des exceptions spécifiques autorisées. Peut-être plus utilement, il est permis d’ajouter des spécialisations pour les types définis par l’utilisateur. Ainsi, par exemple, supposons que votre code ait

class foo
{
    // Stuff
};

Alors ce qui suit est bien

namespace std
{
    template<>
    struct hash<foo>
    {
    public:
        size_t operator()(const foo &f) const;
    };
}

Aucune instruction de retour pour une fonction avec un type de retour non vide #

Omettre l’instruction return dans une fonction dont le type de retour n’est pas void est un comportement indéfini.

int function() {  
    // Missing return statement
} 

int main() {
    function(); //Undefined Behavior
}

La plupart des compilateurs modernes émettent un avertissement au moment de la compilation pour ce type de comportement indéfini.

Remarque : “main” est la seule exception à la règle. Si main n’a pas d’instruction return, le compilateur insère automatiquement return 0; pour vous, afin qu’il puisse être omis en toute sécurité.

Division entière par zéro #

int x = 5 / 0;    // Undefined behavior

La division par “0” est mathématiquement indéfinie, et en tant que tel, il est logique qu’il s’agisse d’un comportement indéfini.

Cependant:

float x = 5.0f / 0.0f;   // x is +infinity

La plupart des implémentations implémentent IEEE-754, qui définit la division en virgule flottante par zéro pour renvoyer NaN (si le numérateur est 0.0f), infinity (si le numérateur est positif) ou -infinity (si le numérateur est négatif).

Accéder à une référence pendante #

Il est illégal d’accéder à une référence à un objet qui est sorti de la portée ou qui a été autrement détruit. Une telle référence est dite * pendante * puisqu’elle ne fait plus référence à un objet valide.

#include <iostream>
int& getX() {
    int x = 42;
    return x;
}
int main() {
    int& r = getX();
    std::cout << r << "\n";
}

Dans cet exemple, la variable locale x sort de la portée lorsque getX revient. (Notez que lifetime extension ne peut pas prolonger la durée de vie d’une variable locale au-delà de la portée du bloc dans lequel elle est définie.) Par conséquent, r est une référence pendante. Ce programme a un comportement indéfini, bien qu’il puisse sembler fonctionner et afficher “42” dans certains cas.

Arithmétique de pointeur invalide #

Les utilisations suivantes de l’arithmétique de pointeur provoquent un comportement indéfini :

  • Addition ou soustraction d’un entier, si le résultat n’appartient pas au même objet tableau que l’opérande pointeur. (Ici, l’élément un après la fin est considéré comme appartenant toujours au tableau.)

    int a[10];
int* p1 = &a[5];
int* p2 = p1 + 4; // ok; p2 points to a[9]
int* p3 = p1 + 5; // ok; p2 points to one past the end of a
int* p4 = p1 + 6; // UB
int* p5 = p1 - 5; // ok; p2 points to a[0]
int* p6 = p1 - 6; // UB
int* p7 = p3 - 5; // ok; p7 points to a[5]

  • Soustraction de deux pointeurs s’ils n’appartiennent pas tous les deux au même objet tableau. (Encore une fois, l’élément un après la fin est considéré comme appartenant au tableau.) L’exception est que deux pointeurs nuls peuvent être soustraits, ce qui donne 0.

    int a[10];
int b[10];
int *p1 = &a[8], *p2 = &a[3];
int d1 = p1 - p2; // yields 5
int *p3 = p1 + 2; // ok; p3 points to one past the end of a
int d2 = p3 - p2; // yields 7
int *p4 = &b[0];
int d3 = p4 - p1; // UB

  • Soustraction de deux pointeurs si le résultat dépasse std::ptrdiff_t.

  • Toute arithmétique de pointeur où le type pointé de l’un ou l’autre des opérandes ne correspond pas au type dynamique de l’objet pointé (en ignorant la cv-qualification). Selon la norme, “[en] particulier, un pointeur vers une base class ne peut pas être utilisé pour l’arithmétique de pointeur lorsque le tableau contient des objets d’un type de classe dérivé.”

    struct Base { int x; };
struct Derived : Base { int y; };
Derived a[10];
Base* p1 = &a[1];           // ok
Base* p2 = p1 + 1;          // UB; p1 points to Derived
Base* p3 = p1 - 1;          // likewise
Base* p4 = &a[2];           // ok
auto p5 = p4 - p1;          // UB; p4 and p1 point to Derived
const Derived* p6 = &a[1];
const Derived* p7 = p6 + 1; // ok; cv-qualifiers don't matter

Débordement d’entier signé #

int x = INT_MAX + 1;

// x can be anything -> Undefined behavior

Si lors de l’évaluation d’une expression, le résultat n’est pas défini mathématiquement ou n’est pas dans la plage des valeurs représentables pour son type, le comportement est indéfini.

(C++11 Standard paragraphe 5/4)

C’est l’un des plus désagréables, car il produit généralement un comportement reproductible et sans plantage, de sorte que les développeurs peuvent être tentés de s’appuyer fortement sur le comportement observé.

D’autre part:

unsigned int x = UINT_MAX + 1;

// x is 0

est bien défini puisque :

Les entiers non signés, déclarés non signés, doivent obéir aux lois de l’arithmétique modulo 2^n où n est le nombre de bits dans la représentation de la valeur de cette taille particulière d’entier.

(norme C++11 paragraphe 3.9.1/4)

Parfois, les compilateurs peuvent exploiter un comportement indéfini et optimiser

signed int x ;
if(x > x + 1)
{
    //do something
}

Ici, puisqu’un débordement d’entier signé n’est pas défini, le compilateur est libre de supposer que cela ne se produira peut-être jamais et qu’il peut donc optimiser le bloc “if”

Appariement incorrect de l’allocation et de la désallocation de mémoire #

Un objet ne peut être désalloué par delete que s’il a été alloué par new et n’est pas un tableau. Si l’argument de delete n’a pas été renvoyé par new ou est un tableau, le comportement est indéfini.

Un objet ne peut être désalloué par delete[] que s’il a été alloué par new et qu’il s’agit d’un tableau. Si l’argument de delete[] n’a pas été renvoyé par new ou n’est pas un tableau, le comportement est indéfini.

Si l’argument de free n’a pas été renvoyé par malloc, le comportement est indéfini.

int* p1 = new int;
delete p1;      // correct
// delete[] p1; // undefined
// free(p1);    // undefined

int* p2 = new int[10];
delete[] p2;    // correct
// delete p2;   // undefined
// free(p2);    // undefined

int* p3 = static_cast<int*>(malloc(sizeof(int)));
free(p3);       // correct
// delete p3;   // undefined
// delete[] p3; // undefined

De tels problèmes peuvent être évités en évitant complètement malloc et free dans les programmes C++, en préférant les pointeurs intelligents de la bibliothèque standard à new et delete bruts, et en préférant std::vector et std::string sur raw new et delete[].

Décalage d’un nombre de positions invalide #

Pour l’opérateur de décalage intégré, l’opérande droit doit être non négatif et strictement inférieur à la largeur en bits de l’opérande gauche promu. Sinon, le comportement est indéfini.

const int a = 42;
const int b = a << -1; // UB
const int c = a << 0;  // ok
const int d = a << 32; // UB if int is 32 bits or less
const int e = a >> 32; // also UB if int is 32 bits or less
const signed char f = 'x';
const int g = f << 10; // ok even if signed char is 10 bits or less;
                       // int must be at least 16 bits

Plusieurs définitions non identiques (la règle de définition unique) #

Si une classe, une énumération, une fonction en ligne, un modèle ou un membre d’un modèle a un lien externe et est défini dans plusieurs unités de traduction, toutes les définitions doivent être identiques ou le comportement n’est pas défini selon la [règle de définition unique (ODR)](https ://en.wikipedia.org/wiki/One_Definition_Rule).

foo.h :

class Foo {
  public:
    double x;
  private:
    int y;
};

Foo get_foo();

foo.cpp :

#include "foo.h"
Foo get_foo() { /* implementation */ }

main.cpp :

// I want access to the private member, so I am going to replace Foo with my own type
class Foo {
  public:
    double x;
    int y;
};
Foo get_foo(); // declare this function ourselves since we aren't including foo.h
int main() {
    Foo foo = get_foo();
    // do something with foo.y
}

Le programme ci-dessus présente un comportement indéfini car il contient deux définitions de la classe ::Foo, qui a une liaison externe, dans différentes unités de traduction, mais les deux définitions ne sont pas identiques. Contrairement à la redéfinition d’une classe dans la même unité de traduction, ce problème n’a pas besoin d’être diagnostiqué par le compilateur.

Modification d’un objet const #

Toute tentative de modification d’un objet const entraîne un comportement indéfini. Cela s’applique aux variables const, aux membres des objets const et aux membres de classe déclarés const. (Cependant, un membre mutable d’un objet const n’est pas const.)

Une telle tentative peut être effectuée via const_cast :

const int x = 123;
const_cast<int&>(x) = 456;
std::cout << x << '\n';

Un compilateur intégrera généralement la valeur d’un objet const int, il est donc possible que ce code compile et affiche 123. Les compilateurs peuvent également placer les valeurs des objets const dans la mémoire en lecture seule, de sorte qu’un défaut de segmentation peut se produire. Dans tous les cas, le comportement est indéfini et le programme peut faire n’importe quoi.

Le programme suivant cache une erreur beaucoup plus subtile :

#include <iostream>

class Foo* instance;

class Foo {
  public:
    int get_x() const { return m_x; }
    void set_x(int x) { m_x = x; }
  private:
    Foo(int x, Foo*& this_ref): m_x(x) {
        this_ref = this;
    }
    int m_x;
    friend const Foo& getFoo();
};

const Foo& getFoo() {
    static const Foo foo(123, instance);
    return foo;
}

void do_evil(int x) {
    instance->set_x(x);
}

int main() {
    const Foo& foo = getFoo();
    do_evil(456);
    std::cout << foo.get_x() << '\n';
}

Dans ce code, getFoo crée un singleton de type const Foo et son membre m_x est initialisé à 123. Ensuite, do_evil est appelé et la valeur de foo.m_x est apparemment changée en 456. Qu’est-ce qui n’a pas fonctionné ?

Malgré son nom, do_evil ne fait rien de particulièrement mauvais ; tout ce qu’il fait est d’appeler un setter via un Foo*. Mais ce pointeur pointe vers un objet const Foo même si const_cast n’a pas été utilisé. Ce pointeur a été obtenu via le constructeur de Foo. Un objet const ne devient pas const tant que son initialisation n’est pas terminée, donc this a le type Foo*, et non const Foo*, dans le constructeur.

Par conséquent, un comportement indéfini se produit même s’il n’y a pas de constructions manifestement dangereuses dans ce programme.

Modification d’un littéral de chaîne #

char *str = "hello world";
str[0] = 'H';

"hello world" est un littéral de chaîne, donc sa modification donne un comportement indéfini.

L’initialisation de str dans l’exemple ci-dessus a été formellement obsolète (suppression prévue d’une future version de la norme) en C++03. Un certain nombre de compilateurs antérieurs à 2003 peuvent émettre un avertissement à ce sujet (par exemple, une conversion suspecte). Après 2003, les compilateurs avertissent généralement d’une conversion obsolète.

L’exemple ci-dessus est illégal et entraîne un diagnostic du compilateur, en C++ 11 et versions ultérieures. Un exemple similaire peut être construit pour présenter un comportement indéfini en autorisant explicitement la conversion de type, par exemple :

char *str = const_cast<char *>("hello world");
str[0] = 'H';

Accéder à un objet avec le mauvais type #

Dans la plupart des cas, il est illégal d’accéder à un objet d’un type comme s’il s’agissait d’un type différent (sans tenir compte des cv-qualificatifs). Exemple:

float x = 42;
int y = reinterpret_cast<int&>(x);

Le résultat est un comportement indéfini.

Il existe quelques exceptions à cette règle de crénelage strict :

  • Un objet de type classe est accessible comme s’il était d’un type qui est une classe de base du type de classe réel.
  • N’importe quel type peut être accédé comme un char ou unsigned char, mais l’inverse n’est pas vrai : un tableau char n’est pas accessible comme s’il s’agissait d’un type arbitraire.
  • Un type entier signé est accessible comme le type non signé correspondant et vice versa.

Une règle connexe est que si une fonction membre non statique est appelée sur un objet qui n’a pas réellement le même type que la classe de définition de la fonction, ou une classe dérivée, alors un comportement indéfini se produit. Ceci est vrai même si la fonction n’accède pas à l’objet.

struct Base {
};
struct Derived : Base {
    void f() {}
};
struct Unrelated {};
Unrelated u;
Derived& r1 = reinterpret_cast<Derived&>(u); // ok
r1.f();                                      // UB
Base b;
Derived& r2 = reinterpret_cast<Derived&>(b); // ok
r2.f();                                      // UB

Débordement lors de la conversion vers ou depuis le type à virgule flottante #

Si, lors de la conversion de :

  • un type entier vers un type virgule flottante,
  • un type à virgule flottante en un type entier, ou
  • un type à virgule flottante vers un type à virgule flottante plus court,

la valeur source est en dehors de la plage de valeurs pouvant être représentées dans le type de destination, le résultat est un comportement indéfini. Exemple:

double x = 1e100;
int y = x; // int probably cannot hold numbers that large, so this is UB

Retour d’une fonction [[noreturn]] #

Exemple tiré de la norme, [dcl.attr.noreturn] :

[[ noreturn ]] void f() {
  throw "error"; // OK
}
[[ noreturn ]] void q(int i) { // behavior is undefined if called with an argument <= 0
  if (i > 0)
    throw "positive";
}

Récursion de modèle infinie #

Exemple tiré de la norme, [temp.inst]/17 :

template<class T> class X {
    X<T>* p; // OK
    X<T*> a; // implicit generation of X<T> requires
             // the implicit instantiation of X<T*> which requires
             // the implicit instantiation of X<T**> which ...
};

Débordement en virgule flottante #

Si une opération arithmétique qui produit un type à virgule flottante produit une valeur qui n’est pas dans la plage des valeurs représentables du type de résultat, le comportement n’est pas défini selon la norme C++, mais peut être défini par d’autres normes auxquelles la machine pourrait se conformer, comme IEEE 754.

float x = 1.0;
for (int i = 0; i < 10000; i++) {
    x *= 10.0; // will probably overflow eventually; undefined behavior
}

Appel de membres virtuels (purs) à partir d’un constructeur ou d’un destructeur #

[La norme (10.4)] (http://www.open-std.org/jtc1/sc22/wg21/docs/papers/2015/n4527.pdf) stipule :

Les fonctions membres peuvent être appelées depuis un constructeur (ou destructeur) d’une classe abstraite ; l’effet de faire un appel virtuel (10.3) à une fonction virtuelle pure directement ou indirectement pour l’objet créé (ou détruit) à partir d’un tel constructeur (ou destructeur) est indéfini.

Plus généralement, certaines autorités C++, par ex. Scott Meyers, suggère de ne jamais appeler de fonctions virtuelles (même non pures) à partir de constructeurs et de destructeurs.

Considérez l’exemple suivant, modifié à partir du lien ci-dessus :

class transaction
{
public:
    transaction(){ log_it(); }
    virtual void log_it() const = 0;
};

class sell_transaction : public transaction
{
public:
    virtual void log_it() const { /* Do something */ }
};

Supposons que nous créons un objet sell_transaction :

sell_transaction s;

Cela appelle implicitement le constructeur de sell_transaction, qui appelle d’abord le constructeur de transaction. Lorsque le constructeur de transaction est appelé, l’objet n’est pas encore du type sell_transaction, mais uniquement du type transaction.

Par conséquent, l’appel dans transaction::transaction() à log_it ne fera pas ce qui pourrait sembler être la chose intuitive - à savoir appeler sell_transaction::log_it.

  • Si log_it est purement virtuel, comme dans cet exemple, le comportement est indéfini.

  • Si log_it est virtuel non pur, transaction::log_it sera appelé.

Conversion statique de base à dérivée invalide #

Si static_cast est utilisé pour convertir un pointeur (resp. référence) vers la classe de base en un pointeur (resp. référence) vers la classe dérivée, mais que l’opérande ne pointe pas (resp. réfère) vers un objet du type de classe dérivée, le comportement est indéfini. Voir Conversion de base en dérivé.

Appel de fonction via un type de pointeur de fonction incompatible #

Pour appeler une fonction via un pointeur de fonction, le type du pointeur de fonction doit correspondre exactement au type de la fonction. Sinon, le comportement est indéfini. Exemple:

int f();
void (*p)() = reinterpret_cast<void(*)()>(f);
p(); // undefined

Accès à un membre inexistant via un pointeur vers le membre #

Lors de l’accès à un membre non statique d’un objet via un pointeur sur membre, si l’objet ne contient pas réellement le membre désigné par le pointeur, le comportement est indéfini. (Un tel pointeur vers un membre peut être obtenu via static_cast.)

struct Base { int x; };
struct Derived : Base { int y; };
int Derived::*pdy = &Derived::y;
int Base::*pby = static_cast<int Base::*>(pdy);

Base* b1 = new Derived;
b1->*pby = 42; // ok; sets y in Derived object to 42
Base* b2 = new Base;
b2->*pby = 42; // undefined; there is no y member in Base

Conversion dérivée vers base invalide pour les pointeurs vers les membres #

Lorsque static_cast est utilisé pour convertir T D ::* en T B::*, le membre pointé doit appartenir à une classe qui est une classe de base ou une classe dérivée de B. Sinon, le comportement est indéfini. Voir [Conversion dérivée en base pour les pointeurs vers les membres] (https://www.wikiod.com/fr/docs/c%2b%2b/3090/explicit-type-conversions/18752/derived-to-base-conversion-for-pointers- aux membres)

Détruire un objet qui a déjà été détruit #

Dans cet exemple, un destructeur est invoqué explicitement pour un objet qui sera ensuite automatiquement détruit.

struct S {
    ~S() { std::cout << "destroying S\n"; }
};
int main() {
    S s;
    s.~S();
} // UB: s destroyed a second time here

Un problème similaire se produit lorsqu’un std::unique_ptr<T> pointe vers un T avec une durée de stockage automatique ou statique.

void f(std::unique_ptr<S> p);
int main() {
    S s;
    std::unique_ptr<S> p(&s);
    f(std::move(p)); // s destroyed upon return from f
}                    // UB: s destroyed

Une autre façon de détruire un objet deux fois consiste à faire en sorte que deux shared_ptr gèrent l’objet sans en partager la propriété.

void f(std::shared_ptr<S> p1, std::shared_ptr<S> p2);
int main() {
    S* p = new S;
    // I want to pass the same object twice...
    std::shared_ptr<S> sp1(p);
    std::shared_ptr<S> sp2(p);
    f(sp1, sp2);
} // UB: both sp1 and sp2 will destroy s separately
// NB: this is correct:
// std::shared_ptr<S> sp(p);
// f(sp, sp);
