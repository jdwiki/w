Les développeurs C++ consacrent beaucoup de temps au débogage. Ce sujet est destiné à aider à cette tâche et à donner une inspiration pour les techniques. Ne vous attendez pas à une longue liste de problèmes et de solutions résolus par les outils ou à un manuel sur les outils mentionnés.

Ce sujet n’est pas encore complet, des exemples sur les techniques/outils suivants seraient utiles :

  • Mentionner plus d’outils d’analyse statique
  • Outils d’instrumentation binaire (comme UBSan, TSan, MSan, ESan …)
  • Trempe (CFI…)
  • Fuzzing

Analyse des erreurs de segmentation avec GDB #

Analyse statique #

L’analyse statique est la technique dans laquelle on vérifie le code pour les modèles liés aux bogues connus. L’utilisation de cette technique prend moins de temps qu’une revue de code, cependant, ses vérifications sont limitées à celles programmées dans l’outil.

Les vérifications peuvent inclure le point-virgule incorrect derrière l’instruction if (if (var);) jusqu’aux algorithmes graphiques avancés qui déterminent si une variable n’est pas initialisée.

Avertissements du compilateur #

Activer l’analyse statique est facile, la version la plus simpliste est déjà intégrée dans votre compilateur :

  • clang++ -Wall -Weverything -Werror ...
  • g++ -Wall -Weverything -Werror ...
  • cl.exe /W4 /WX ...

Si vous activez ces options, vous remarquerez que chaque compilateur trouvera des bogues que les autres ne trouveront pas et que vous obtiendrez des erreurs sur des techniques qui pourraient être valides ou valides dans un contexte spécifique. while (staticAtomicBool); peut être acceptable même si while (localBool); ne l’est pas.

Ainsi, contrairement à la revue de code, vous combattez un outil qui comprend votre code, vous indique de nombreux bogues utiles et parfois n’est pas d’accord avec vous. Dans ce dernier cas, vous devrez peut-être supprimer l’avertissement localement.

Comme les options ci-dessus activent tous les avertissements, elles peuvent activer des avertissements dont vous ne voulez pas. (Pourquoi votre code devrait-il être compatible C++98 ?) Si tel est le cas, vous pouvez simplement désactiver cet avertissement spécifique :

  • clang++ -Wall -Weverything -Werror -Wno-errortoaccept ...
  • g++ -Wall -Weverything -Werror -Wno-errortoaccept ...
  • cl.exe /W4 /WX /wd<pas d'avertissement>...

Là où les avertissements du compilateur vous aident pendant le développement, ils ralentissent un peu la compilation. C’est pourquoi vous ne voudrez peut-être pas toujours les activer par défaut. Soit vous les exécutez par défaut, soit vous activez une intégration continue avec les vérifications les plus coûteuses (ou toutes).

Outils externes #

Si vous décidez d’avoir une intégration continue, l’utilisation d’autres outils n’est pas si difficile. Un outil comme clang-tidy a une [liste de vérifications](http://clang.llvm.org/extra/clang-tidy/ checks/list.html) qui couvre un large éventail de problèmes, quelques exemples :

  • Bugs réels
    • Prevention of slicing
    • Asserts with side effects
  • Contrôles de lisibilité
    • Misleading indentation
    • Check identifier naming
  • Contrôles de modernisation
    • Use make_unique()
    • Use nullptr
  • Contrôles de performance
    • Find unneeded copies
    • Find inefficient algorithm calls

La liste n’est peut-être pas si longue, car Clang a déjà beaucoup d’avertissements du compilateur, mais cela vous rapprochera d’une base de code de haute qualité.

Autres outils #

D’autres outils ayant un but similaire existent, comme :

  • [l’analyseur statique Visual Studio] (https://blogs.msdn.microsoft.com/hkamel/2013/10/24/visual-studio-2013-static-code-analysis-in-depth-what-when-and -how/) comme outil externe
  • clazy, un plugin de compilateur Clang pour vérifier le code Qt

Conclusion #

De nombreux outils d’analyse statique existent pour C++, à la fois intégrés au compilateur et outils externes. Les essayer ne prend pas beaucoup de temps pour des configurations faciles et ils trouveront des bogues que vous pourriez manquer lors de la révision du code.

Nettoyer le code #

Le débogage commence par comprendre le code que vous essayez de déboguer.

Mauvais code :

int main() {
    int value;
    std::vector<int> vectorToSort;
    vectorToSort.push_back(42); vectorToSort.push_back(13);
    for (int i = 52; i; i = i - 1)
    {
    vectorToSort.push_back(i *2);
    }
    /// Optimized for sorting small vectors
    if (vectorToSort.size() == 1);
    else
        {
        if (vectorToSort.size() <= 2)
            std::sort(vectorToSort.begin(), std::end(vectorToSort));
        }
    for (value : vectorToSort) std::cout << value << ' ';
return 0; }

Meilleur code :

std::vector<int> createSemiRandomData() {
    std::vector<int> data;
    data.push_back(42);
    data.push_back(13);
    for (int i = 52; i; --i)
        vectorToSort.push_back(i *2);
    return data;
}

/// Optimized for sorting small vectors
void sortVector(std::vector &v) {
    if (vectorToSort.size() == 1)
        return;
    if (vectorToSort.size() > 2)
        return;

    std::sort(vectorToSort.begin(), vectorToSort.end());
}

void printVector(const std::vector<int> &v) {
    for (auto i : v)
        std::cout << i << ' ';
}

int main() {
    auto vectorToSort = createSemiRandomData();
    sortVector(std::ref(vectorToSort));
    printVector(vectorToSort);

    return 0;
 }

Quels que soient les styles de codage que vous préférez et que vous utilisez, un style de codage (et de formatage) cohérent vous aidera à comprendre le code.

En regardant le code ci-dessus, on peut identifier quelques améliorations pour améliorer la lisibilité et la débogabilité :

L’utilisation de fonctions distinctes pour des actions distinctes #

L’utilisation de fonctions séparées vous permet d’ignorer certaines fonctions du débogueur si vous n’êtes pas intéressé par les détails. Dans ce cas précis, vous n’êtes peut-être pas intéressé par la création ou l’impression des données et souhaitez uniquement vous lancer dans le tri.

Un autre avantage est que vous devez lire moins de code (et le mémoriser) tout en parcourant le code. Il vous suffit maintenant de lire 3 lignes de code dans main() pour le comprendre, au lieu de la fonction entière.

Le troisième avantage est que vous avez simplement moins de code à regarder, ce qui aide un œil averti à repérer ce bogue en quelques secondes.

Utiliser des mises en forme/constructions cohérentes #

L’utilisation d’une mise en forme et de constructions cohérentes supprimera l’encombrement du code, ce qui facilitera la concentration sur le code plutôt que sur le texte. De nombreuses discussions ont été alimentées sur le “bon” style de formatage. Indépendamment de ce style, avoir un seul style cohérent dans le code améliorera la familiarité et facilitera la concentration sur le code.

Comme le formatage du code est une tâche chronophage, il est recommandé d’utiliser un outil dédié pour cela. La plupart des IDE ont au moins une sorte de support pour cela et peuvent faire un formatage plus cohérent que les humains.

Vous remarquerez peut-être que le style n’est pas limité aux espaces et aux retours à la ligne car nous ne mélangeons plus le style libre et les fonctions membres pour obtenir le début/la fin du conteneur. (v.begin() vs std::end(v)).

Attirez l’attention sur les parties importantes de votre code. #

Quel que soit le style que vous décidez de choisir, le code ci-dessus contient quelques marqueurs qui pourraient vous donner un indice sur ce qui pourrait être important :

  • Un commentaire indiquant “optimisé”, cela indique des techniques fantaisistes
  • Certains premiers retours dans sortVector() indiquent que nous faisons quelque chose de spécial
  • Le std::ref() indique que quelque chose se passe avec le sortVector()

Conclusion #

Avoir un code propre vous aidera à comprendre le code et réduira le temps dont vous avez besoin pour le déboguer. Dans le deuxième exemple, un réviseur de code peut même repérer le bogue au premier coup d’œil, tandis que le bogue peut être caché dans les détails du premier. (PS : Le bogue est dans la comparaison avec ‘2’.)

Safe-stack (Corruptions de la pile) #

Les corruptions de pile sont des bogues ennuyeux à regarder. Comme la pile est corrompue, le débogueur ne peut souvent pas vous donner une bonne trace de la pile où vous vous trouvez et comment vous y êtes arrivé.

C’est là que le safe-stack entre en jeu. Au lieu d’utiliser une seule pile pour vos threads, il en utilisera deux : une pile sûre et une pile dangereuse. La pile sûre fonctionne exactement comme avant, sauf que certaines pièces sont déplacées vers la pile dangereuse.

Quelles parties de la pile sont déplacées ? #

Chaque partie susceptible de corrompre la pile sera retirée de la pile sécurisée. Dès qu’une variable de la pile est passée par référence ou qu’on prend l’adresse de cette variable, le compilateur décidera de l’allouer sur la deuxième pile au lieu de celle de sécurité.

Par conséquent, toute opération que vous effectuez avec ces pointeurs, toute modification que vous apportez à la mémoire (basée sur ces pointeurs/références) ne peut affecter que la mémoire de la deuxième pile. Comme on n’obtient jamais un pointeur proche de la pile sûre, la pile ne peut pas corrompre la pile et le débogueur peut toujours lire toutes les fonctions sur la pile pour donner une belle trace.

A quoi sert-il réellement ? #

La pile sécurisée n’a pas été inventée pour vous offrir une meilleure expérience de débogage, cependant, c’est un bel effet secondaire pour les bogues désagréables. Son objectif initial est dans le cadre du [projet Code-Pointer Integrity (CPI)] (http://dslab.epfl.ch/proj/cpi/), dans lequel ils essaient d’empêcher le remplacement des adresses de retour pour empêcher l’injection de code. En d’autres termes, ils essaient d’empêcher l’exécution d’un code pirate.

Pour cette raison, la fonctionnalité a été activée sur le chrome et a été rapportée pour avoir un <1 % de surcharge du processeur.

Comment l’activer ? #

À l’heure actuelle, l’option n’est disponible que dans le compilateur clang, où l’on peut transmettre -fsanitize=safe-stack au compilateur. Une proposition a été faite pour implémenter la même fonctionnalité dans GCC.

Conclusion #

Les corruptions de pile peuvent devenir plus faciles à déboguer lorsque la pile sécurisée est activée. En raison d’une faible surcharge de performances, vous pouvez même l’activer par défaut dans votre configuration de construction.
