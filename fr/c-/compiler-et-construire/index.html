Les programmes écrits en C++ doivent être compilés avant de pouvoir être exécutés. Il existe une grande variété de compilateurs disponibles en fonction de votre système d’exploitation.

La plupart des systèmes d’exploitation sont livrés sans compilateur et doivent être installés ultérieurement. Certains choix de compilateurs courants sont :

  • [GCC, la collection de compilateurs GNU][1] [tag:g++]
  • [clang : une interface de la famille du langage C pour LLVM][2] [tag:clang++]
  • [MSVC, Microsoft Visual C++ (inclus dans Visual Studio)][3] [tag:visual-c++]
  • [C++Builder, Embarcadero C++Builder][4] (inclus dans RAD Studio) [tag:c++builder]

Veuillez consulter le manuel du compilateur approprié pour savoir comment compiler un programme C++.

Une autre option pour utiliser un compilateur spécifique avec son propre système de construction spécifique, il est possible de laisser les [systèmes de construction] génériques [5] configurer le projet pour un compilateur spécifique ou pour celui installé par défaut.

[1] : https://gcc.gnu.org/ [2] : http://clang.llvm.org/ [3] : https://www.visualstudio.com/ [4] : https://www.embarcadero.com/products/cbuilder [5] : https://www.wikiod.com/fr/docs/c%2B%2B/8200/build-systems

Compiler avec GCC #

En supposant un seul fichier source nommé main.cpp, la commande pour compiler et lier un exécutable non optimisé est la suivante (Compiler sans optimisation est utile pour le développement initial et le débogage, bien que [-Og][1] soit officiellement recommandé pour les nouvelles versions de GCC).

g++ -o app -Wall main.cpp -O0

Pour produire un exécutable optimisé pour une utilisation en production, utilisez l’une des options [-O][2] (voir : [-O1][3], [-O2][4], [- O3][5], [-Os][6], [-Ofast][7]) :

g++ -o app -Wall -O2 main.cpp

Si l’option -O est omise, -O0, qui signifie aucune optimisation, est utilisé par défaut (spécifier -O sans numéro se résout en -O1).

Vous pouvez également utiliser directement les indicateurs d’optimisation des groupes “O” (ou des optimisations plus expérimentales). L’exemple suivant se construit avec l’optimisation -O2, plus un indicateur du niveau d’optimisation -O3 :

g++ -o app -Wall -O2 -ftree-partial-pre main.cpp

Pour produire un exécutable optimisé spécifique à la plate-forme (pour une utilisation en production sur la machine avec la même architecture), utilisez :

g++ -o app -Wall -O2 -march=native main.cpp

L’un ou l’autre des éléments ci-dessus produira un fichier binaire pouvant être exécuté avec .\app.exe sous Windows et ./app sous Linux, Mac OS, etc.

Le drapeau [-o][8] peut également être ignoré. Dans ce cas, GCC créera l’exécutable de sortie par défaut “a.exe” sous Windows et “a.out” sur les systèmes de type Unix. Pour compiler un fichier sans le lier, utilisez l’option [-c][9] :

g++ -o file.o -Wall -c file.cpp

Cela produit un fichier objet nommé file.o qui peut ensuite être lié à d’autres fichiers pour produire un binaire :

g++ -o app file.o otherfile.o

Vous trouverez plus d’informations sur les options d’optimisation sur gcc.gnu.org. On notera en particulier -Og (optimisation mettant l’accent sur l’expérience de débogage - recommandée pour le cycle standard d’édition-compilation-débogage) et -Ofast (toutes les optimisations, y compris celles qui ne tiennent pas compte de la stricte conformité aux normes).

Le drapeau [-Wall][10] active les avertissements pour de nombreuses erreurs courantes et doit toujours être utilisé. Pour améliorer la qualité du code, il est souvent conseillé d’utiliser également [-Wextra][11] et d’autres drapeaux d’avertissement qui ne sont pas automatiquement activés par -Wall et -Wextra.

Si le code attend une norme C++ spécifique, spécifiez la norme à utiliser en incluant l’indicateur [-std=][12]. Les valeurs prises en charge correspondent à l’année de finalisation de chaque version de la norme ISO C++. Depuis GCC 6.1.0, les valeurs valides pour le drapeau std= sont c++98/c++03, c++11, c++14 et c++ 17/c++1z. Les valeurs séparées par une barre oblique sont équivalentes.

g++ -std=c++11 <file>

GCC inclut certaines extensions spécifiques au compilateur qui sont désactivées lorsqu’elles entrent en conflit avec un standard spécifié par le drapeau -std=. Pour compiler avec toutes les extensions activées, la valeur gnu++XX peut être utilisée, où XX est l’une des années utilisées par les valeurs c++ répertoriées ci-dessus.

La norme par défaut sera utilisée si aucune n’est spécifiée. Pour les versions de GCC antérieures à 6.1.0, la valeur par défaut est -std=gnu++03 ; dans GCC 6.1.0 et supérieur, la valeur par défaut est -std=gnu++14.

Notez qu’en raison de bogues dans GCC, le drapeau -pthread doit être présent lors de la compilation et de la liaison pour que GCC prenne en charge la fonctionnalité de threading standard C++ introduite avec C++11, telle que std :: thread et std :: wait_for. L’omettre lors de l’utilisation des fonctions de threading peut entraîner [pas d’avertissements mais des résultats invalides] [13] sur certaines plates-formes.

Liaison avec les bibliothèques : #

Utilisez l’option -l pour passer le nom de la bibliothèque :

g++ main.cpp -lpcre2-8
#pcre2-8 is the PCRE2 library for 8bit code units (UTF-8)

Si la bibliothèque n’est pas dans le chemin de bibliothèque standard, ajoutez le chemin avec l’option -L :

g++ main.cpp -L/my/custom/path/ -lmylib

Plusieurs bibliothèques peuvent être liées entre elles :

g++ main.cpp -lmylib1 -lmylib2 -lmylib3

Si une bibliothèque dépend d’une autre, placez la bibliothèque dépendante avant la bibliothèque indépendante :

g++ main.cpp -lchild-lib -lbase-lib

Ou laissez l’éditeur de liens déterminer lui-même l’ordre via --start-group et --end-group (remarque : cela a un coût de performance important) :

g++ main.cpp -Wl,--start-group -lbase-lib -lchild-lib -Wl,--end-group

Compilateurs en ligne #

Divers sites Web offrent un accès en ligne aux compilateurs C++. Les fonctionnalités du compilateur en ligne varient considérablement d’un site à l’autre, mais elles permettent généralement d’effectuer les opérations suivantes :

  • Collez votre code dans un formulaire Web dans le navigateur.
  • Sélectionnez certaines options du compilateur et compilez le code.
  • Collecter la sortie du compilateur et/ou du programme.

Le comportement du site Web du compilateur en ligne est généralement assez restrictif car il permet à quiconque d’exécuter des compilateurs et d’exécuter du code arbitraire côté serveur, alors que l’exécution de code arbitraire à distance est généralement considérée comme une vulnérabilité.

Les compilateurs en ligne peuvent être utiles aux fins suivantes :

  • Exécutez un petit extrait de code à partir d’une machine qui n’a pas de compilateur C++ (smartphones, tablettes, etc.).
  • Assurez-vous que le code se compile correctement avec différents compilateurs et s’exécute de la même manière, quel que soit le compilateur avec lequel il a été compilé.
  • Apprendre ou enseigner les bases du C++.
  • Apprenez les fonctionnalités C++ modernes (C++14 et C++17 dans un futur proche) lorsque le compilateur C++ à jour n’est pas disponible sur la machine locale.
  • Repérer un bogue dans votre compilateur par comparaison avec un grand nombre d’autres compilateurs. Vérifiez si un bogue du compilateur a été corrigé dans les futures versions, qui ne sont pas disponibles sur votre machine.
  • Résoudre les problèmes de juge en ligne.

Pour quels compilateurs en ligne ne doivent pas être utilisés :

  • Développer des applications complètes (même petites) en utilisant C++. Habituellement, les compilateurs en ligne ne permettent pas de se lier à des bibliothèques tierces ou de télécharger des artefacts de construction.
  • Effectuer des calculs intensifs. Les ressources informatiques côté serveur sont limitées, de sorte que tout programme fourni par l’utilisateur sera tué après quelques secondes d’exécution. Le temps d’exécution autorisé est généralement suffisant pour les tests et l’apprentissage.
  • Attaquez le serveur du compilateur lui-même ou tout autre hôte tiers sur le net.

Exemples:

Avis de non-responsabilité : les auteurs de la documentation ne sont affiliés à aucune des ressources répertoriées ci-dessous. Les sites Web sont classés par ordre alphabétique.

  • http://codepad.org/ Compilateur en ligne avec partage de code. L’édition de code après la compilation avec un avertissement ou une erreur de code source ne fonctionne pas aussi bien.
  • http://coliru.stacked-crooked.com/ Compilateur en ligne pour lequel vous spécifiez la ligne de commande. Fournit les compilateurs GCC et Clang à utiliser.
  • http://cpp.sh/ - Compilateur en ligne avec prise en charge de C++14. Ne vous permet pas de modifier la ligne de commande du compilateur, mais certaines options sont disponibles via les contrôles de l’interface graphique.
  • https://gcc.godbolt.org/ - Fournit une large liste de versions de compilateur, d’architectures et de sorties de désassemblage. Très utile lorsque vous avez besoin d’inspecter ce que votre code compile dans différents compilateurs. GCC, Clang, MSVC (“CL”), le compilateur Intel (“icc”), ELLCC et Zapcc sont présents, avec un ou plusieurs de ces compilateurs disponibles pour ARM, ARMv8 (comme ARM64), Atmel AVR, MIPS, MIPS64 , MSP430, PowerPC, architectures x86 et x64.   • http://rextester.com/ - Fournit des compilateurs Clang, GCC et Visual Studio pour C et C++ (ainsi que des compilateurs pour d’autres langages), avec la bibliothèque Boost disponible.

Compilation avec Visual C++ (ligne de commande) #

Pour les programmeurs venant de GCC ou Clang vers Visual Studio, ou les programmeurs plus à l’aise avec la ligne de commande en général, vous pouvez utiliser le compilateur Visual C++ depuis la ligne de commande ainsi que l’IDE.

Si vous souhaitez compiler votre code à partir de la ligne de commande dans Visual Studio, vous devez d’abord configurer l’environnement de ligne de commande. Cela peut être fait soit en ouvrant l’invite de commande [Visual Studio Command Prompt/Developer Command Prompt/x86 Native Tools Command Prompt/x64 Native Tools Command Prompt ou similaire][1] (comme fourni par votre version de Visual Studio), ou à l’invite de commande, en accédant au sous-répertoire VC du répertoire d’installation du compilateur (généralement \Program Files (x86)\Microsoft Visual Studio x\VC, où x est le numéro de version ( comme 10.0 pour 2010 ou 14.0 pour 2015) et en exécutant le fichier batch VCVARSALL avec un paramètre de ligne de commande spécifié [ici][2].

Notez que contrairement à GCC, Visual Studio ne fournit pas d’interface pour l’éditeur de liens (link.exe) via le compilateur (cl.exe), mais fournit à la place l’éditeur de liens en tant que programme séparé, que le compilateur appelle comme il sort. cl.exe et link.exe peuvent être utilisés séparément avec différents fichiers et options, ou cl peut être invité à transmettre des fichiers et des options à link si les deux tâches sont effectuées ensemble. Toutes les options de liaison spécifiées pour cl seront traduites en options pour link, et tous les fichiers non traités par cl seront passés directement à link. Comme il s’agit principalement d’un simple guide de compilation avec la ligne de commande de Visual Studio, les arguments de link ne seront pas décrits pour le moment ; si vous avez besoin d’une liste, voir [ici][3].

Notez que les arguments de cl sont sensibles à la casse, alors que les arguments de link ne le sont pas.

[Notez que certains des exemples suivants utilisent la variable “répertoire actuel” du shell Windows, %cd%, lors de la spécification de noms de chemin absolus. Pour ceux qui ne connaissent pas cette variable, elle s’étend au répertoire de travail actuel. À partir de la ligne de commande, ce sera le répertoire dans lequel vous étiez lorsque vous avez exécuté cl, et il est spécifié dans l’invite de commande par défaut (si votre invite de commande est C:\src>, par exemple, alors %cd % est C:\src\).]

En supposant un seul fichier source nommé “main.cpp” dans le dossier actuel, la commande pour compiler et lier un exécutable non optimisé (utile pour le développement initial et le débogage) est (utilisez l’une des options suivantes) :

cl main.cpp
// Generates object file "main.obj".
// Performs linking with "main.obj".
// Generates executable "main.exe".

cl /Od main.cpp
// Same as above.
// "/Od" is the "Optimisation: disabled" option, and is the default when no /O is specified.

En supposant un fichier source supplémentaire “niam.cpp” dans le même répertoire, utilisez ce qui suit :

cl main.cpp niam.cpp
// Generates object files "main.obj" and "niam.obj".
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

Vous pouvez également utiliser des caractères génériques, comme on pourrait s’y attendre :

cl main.cpp src\*.cpp
// Generates object file "main.obj", plus one object file for each ".cpp" file in folder
//  "%cd%\src".
// Performs linking with "main.obj", and every additional object file generated.
// All object files will be in the current folder.
// Generates executable "main.exe".

Pour renommer ou déplacer l’exécutable, utilisez l’une des options suivantes :

cl /o name main.cpp
// Generates executable named "name.exe".

cl /o folder\ main.cpp
// Generates executable named "main.exe", in folder "%cd%\folder".

cl /o folder\name main.cpp
// Generates executable named "name.exe", in folder "%cd%\folder".

cl /Fename main.cpp
// Same as "/o name".

cl /Fefolder\ main.cpp
// Same as "/o folder\".

cl /Fefolder\name main.cpp
// Same as "/o folder\name".

/o et /Fe passent leur paramètre (appelons-le o-param) à link comme /OUT:o-param, en ajoutant l’extension appropriée (généralement .exe ou . dll) pour “nommer” o-param si nécessaire. Alors que /o et /Fe sont à ma connaissance identiques dans leurs fonctionnalités, ce dernier est préféré pour Visual Studio. /o est marqué comme obsolète et semble être principalement fourni aux programmeurs plus familiers avec GCC ou Clang.

Notez que même si l’espace entre /o et le dossier et/ou le nom spécifié est facultatif, il ne peut pas y avoir d’espace entre /Fe et le dossier et/ou le nom spécifié.

De même, pour produire un exécutable optimisé (pour une utilisation en production), utilisez :

cl /O1 main.cpp
// Optimise for executable size.  Produces small programs, at the possible expense of slower
//  execution.

cl /O2 main.cpp
// Optimise for execution speed.  Produces fast programs, at the possible expense of larger
//  file size.

cl /GL main.cpp other.cpp
// Generates special object files used for whole-program optimisation, which allows CL to
//  take every module (translation unit) into consideration during optimisation.
// Passes the option "/LTCG" (Link-Time Code Generation) to LINK, telling it to call CL during
//  the linking phase to perform additional optimisations.  If linking is not performed at this
//  time, the generated object files should be linked with "/LTCG".
// Can be used with other CL optimisation options.

Enfin, pour produire un exécutable optimisé spécifique à la plate-forme (pour une utilisation en production sur la machine avec l’architecture spécifiée), choisissez l’invite de commande appropriée ou le paramètre [VCVARSALL][4] pour la plate-forme cible. link doit détecter la plate-forme souhaitée à partir des fichiers objets ; sinon, utilisez l’option [/MACHINE][5] pour spécifier explicitement la plate-forme cible.

// If compiling for x64, and LINK doesn't automatically detect target platform:
cl main.cpp /link /machine:X64

Tout ce qui précède produira un exécutable avec le nom spécifié par /o ou /Fe, ou si aucun n’est fourni, avec un nom identique au premier fichier source ou objet spécifié au compilateur.

cl a.cpp b.cpp c.cpp
// Generates "a.exe".

cl d.obj a.cpp q.cpp
// Generates "d.exe".

cl y.lib n.cpp o.obj
// Generates "n.exe".

cl /o yo zp.obj pz.cpp
// Generates "yo.exe".

Pour compiler un ou plusieurs fichiers sans lien, utilisez :

cl /c main.cpp
// Generates object file "main.obj".

Cela indique à cl de quitter sans appeler link et produit un fichier objet, qui peut ensuite être lié à d’autres fichiers pour produire un binaire.

cl main.obj niam.cpp
// Generates object file "niam.obj".
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

link main.obj niam.obj
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

Il existe également d’autres paramètres de ligne de commande précieux, qu’il serait très utile pour les utilisateurs de connaître :

cl /EHsc main.cpp
// "/EHsc" specifies that only standard C++ ("synchronous") exceptions will be caught,
//  and `extern "C"` functions will not throw exceptions.
// This is recommended when writing portable, platform-independent code.

cl /clr main.cpp
// "/clr" specifies that the code should be compiled to use the common language runtime,
//  the .NET Framework's virtual machine.
// Enables the use of Microsoft's C++/CLI language in addition to standard ("native") C++,
//  and creates an executable that requires .NET to run.

cl /Za main.cpp
// "/Za" specifies that Microsoft extensions should be disabled, and code should be
//  compiled strictly according to ISO C++ specifications.
// This is recommended for guaranteeing portability.

cl /Zi main.cpp
// "/Zi" generates a program database (PDB) file for use when debugging a program, without
//  affecting optimisation specifications, and passes the option "/DEBUG" to LINK.

cl /LD dll.cpp
// "/LD" tells CL to configure LINK to generate a DLL instead of an executable.
// LINK will output a DLL, in addition to an LIB and EXP file for use when linking.
// To use the DLL in other programs, pass its associated LIB to CL or LINK when compiling those
//  programs.

cl main.cpp /link /LINKER_OPTION
// "/link" passes everything following it directly to LINK, without parsing it in any way.
// Replace "/LINKER_OPTION" with any desired LINK option(s).

Pour tous ceux qui connaissent mieux les systèmes *nix et/ou GCC/Clang, cl, link et d’autres outils de ligne de commande de Visual Studio peuvent accepter les paramètres spécifiés avec un trait d’union (comme -c) au lieu d’une barre oblique ( comme /c). De plus, Windows reconnaît une barre oblique ou une barre oblique inverse comme séparateur de chemin valide, de sorte que les chemins de style * nix peuvent également être utilisés. Cela facilite la conversion des lignes de commande simples du compilateur de g++ ou clang++ en cl, ou vice versa, avec des modifications minimes.

g++ -o app src/main.cpp
cl  -o app src/main.cpp

Bien sûr, lors du portage de lignes de commande qui utilisent des options g++ ou clang++ plus complexes, vous devez rechercher des commandes équivalentes dans les documentations du compilateur applicables et/ou sur les sites de ressources, mais cela facilite le démarrage avec un minimum temps passé à apprendre de nouveaux compilateurs.

Si vous avez besoin de fonctionnalités linguistiques spécifiques pour votre code, une version spécifique de MSVC était requise. À partir de [Visual C++ 2015 Update 3][6], il est possible de choisir la version du standard à compiler via le drapeau /std. Les valeurs possibles sont /std:c++14 et /std:c++latest (/std:c++17 suivra bientôt).

Remarque : dans les anciennes versions de ce compilateur, des indicateurs de fonctionnalité spécifiques étaient disponibles, mais ils étaient principalement utilisés pour les aperçus de nouvelles fonctionnalités.

Compiler avec Clang #

Comme le frontal [Clang][1] est conçu pour être compatible avec GCC, la plupart des programmes pouvant être compilés via [GCC][2] se compileront lorsque vous échangerez g++ par clang++ dans les scripts de construction. Si aucun -std=version n’est donné, gnu11 sera utilisé.

Les utilisateurs Windows habitués à [MSVC][3] peuvent échanger cl.exe avec clang-cl.exe. Par défaut, clang essaie d’être compatible avec la version la plus élevée de MSVC qui a été installée.

Dans le cas d’une compilation avec Visual Studio, clang-cl peut être utilisé en modifiant le Platform toolset dans les propriétés du projet.

Dans les deux cas, clang n’est compatible que via son interface, bien qu’il essaie également de générer des fichiers objets compatibles binaires. Les utilisateurs de clang-cl doivent noter que [la compatibilité avec MSVC n’est pas encore complète][4].

Pour utiliser clang ou clang-cl, on peut utiliser l’installation par défaut sur certaines distributions Linux ou celles fournies avec des IDE (comme XCode sur Mac). Pour les autres versions de ce compilateur ou sur les plates-formes qui ne l’ont pas installé, cela peut être téléchargé depuis la [page de téléchargement officielle][5].

Si vous utilisez CMake pour construire votre code, vous pouvez généralement changer de compilateur en définissant les variables d’environnement CC et CXX comme ceci :

mkdir build
cd build
CC=clang CXX=clang++ cmake ..
cmake --build .

Voir aussi [introduction à Cmake][6].

Le processus de compilation C++ #

Lorsque vous développez un programme C++, l’étape suivante consiste à compiler le programme avant de l’exécuter. La compilation est le processus qui convertit le programme écrit en langage lisible par l’homme comme C, C++, etc. en un code machine, directement compris par l’unité centrale de traitement. Par exemple, si vous avez un fichier de code source C++ nommé prog.cpp et que vous exécutez la commande compile,

   g++ -Wall -ansi -o prog prog.cpp

Il y a 4 étapes principales impliquées dans la création d’un fichier exécutable à partir du fichier source.

  1. Le préprocesseur C++ prend un fichier de code source C++ et traite les en-têtes (#include), les macros (#define) et les autres directives du préprocesseur.

  2. Le fichier de code source C++ étendu produit par le préprocesseur C++ est compiled into the assembly language for the platform.

  3. Le code assembleur généré par le compilateur est assemblé dans le object code for the platform.

  4. Le fichier de code objet produit par l’assembleur est lié with the object code files for any library functions used to produce either a library or an executable file.

Prétraitement

Le préprocesseur gère les directives du préprocesseur, comme #include et #define. Il est indépendant de la syntaxe du C++, c’est pourquoi il doit être utilisé avec précaution.

Il fonctionne sur un fichier source C++ à la fois en remplaçant les directives #include par le contenu des fichiers respectifs (qui ne sont généralement que des déclarations), en remplaçant les macros (#define) et en sélectionnant différentes portions de texte en fonction de #if, Directives #ifdef et #ifndef.

Le préprocesseur fonctionne sur un flux de jetons de prétraitement. La substitution de macro est définie comme le remplacement de jetons par d’autres jetons (l’opérateur ## permet de fusionner deux jetons lorsque cela a du sens).

Après tout cela, le préprocesseur produit une sortie unique qui est un flux de jetons résultant des transformations décrites ci-dessus. Il ajoute également des marqueurs spéciaux qui indiquent au compilateur d’où vient chaque ligne afin qu’il puisse les utiliser pour produire des messages d’erreur sensibles.

Certaines erreurs peuvent être produites à ce stade avec une utilisation intelligente des directives #if et #error.

En utilisant l’indicateur de compilateur ci-dessous, nous pouvons arrêter le processus au stade du prétraitement.

g++ -E prog.cpp

Compilation

L’étape de compilation est effectuée sur chaque sortie du préprocesseur. Le compilateur analyse le code source C++ pur (désormais sans aucune directive de préprocesseur) et le convertit en code assembleur. Invoque ensuite le back-end sous-jacent (assembleur dans la chaîne d’outils) qui assemble ce code en code machine produisant un fichier binaire réel dans un certain format (ELF, COFF, a.out, …). Ce fichier objet contient le code compilé (sous forme binaire) des symboles définis en entrée. Les symboles dans les fichiers objets sont référencés par leur nom.

Les fichiers objets peuvent faire référence à des symboles qui ne sont pas définis. C’est le cas lorsque vous utilisez une déclaration et que vous ne lui fournissez pas de définition. Le compilateur ne s’en soucie pas et produira volontiers le fichier objet tant que le code source est bien formé.

Les compilateurs vous permettent généralement d’arrêter la compilation à ce stade. Ceci est très utile car avec lui, vous pouvez compiler chaque fichier de code source séparément. L’avantage que cela procure est que vous n’avez pas besoin de tout recompiler si vous ne modifiez qu’un seul fichier.

Les fichiers objets produits peuvent être placés dans des archives spéciales appelées bibliothèques statiques, pour une réutilisation ultérieure plus facile.

C’est à ce stade que les erreurs “régulières” du compilateur, telles que les erreurs de syntaxe ou les erreurs de résolution de surcharge échouées, sont signalées.

Afin d’arrêter le processus après l’étape de compilation, nous pouvons utiliser l’option -S :

g++ -Wall -ansi -S prog.cpp

Assemblage

L’assembleur crée du code objet. Sur un système UNIX, vous pouvez voir des fichiers avec un suffixe .o (.OBJ sur MSDOS) pour indiquer des fichiers de code objet. Dans cette phase, l’assembleur convertit ces fichiers objets du code d’assemblage en instructions au niveau de la machine et le fichier créé est un code objet réadressable. Ainsi, la phase de compilation génère le programme objet relocalisable et ce programme peut être utilisé à différents endroits sans avoir à recompiler.

Pour arrêter le processus après l’étape d’assemblage, vous pouvez utiliser l’option -c :

g++ -Wall -ansi -c prog.cpp

Mise en relation

L’éditeur de liens est ce qui produit la sortie finale de la compilation à partir des fichiers objets produits par l’assembleur. Cette sortie peut être soit une bibliothèque partagée (ou dynamique) (et bien que le nom soit similaire, elles n’ont pas grand-chose en commun avec les bibliothèques statiques mentionnées précédemment) ou un exécutable.

Il relie tous les fichiers objets en remplaçant les références aux symboles indéfinis par les adresses correctes. Chacun de ces symboles peut être défini dans d’autres fichiers objets ou dans des bibliothèques. S’ils sont définis dans des bibliothèques autres que la bibliothèque standard, vous devez en informer l’éditeur de liens.

À ce stade, les erreurs les plus courantes sont les définitions manquantes ou les définitions en double. Le premier signifie que soit les définitions n’existent pas (c’est-à-dire qu’elles ne sont pas écrites), soit que les fichiers objets ou les bibliothèques où ils résident n’ont pas été donnés à l’éditeur de liens. Ce dernier est évident : le même symbole a été défini dans deux fichiers objets ou bibliothèques différents.
