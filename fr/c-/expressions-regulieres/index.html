Regular Expressions (parfois appelées regexs ou regexps) sont une syntaxe textuelle qui représente les modèles qui peuvent être mis en correspondance dans les chaînes exploitées.

Les expressions régulières, introduites dans c++11, peuvent éventuellement prendre en charge un tableau de retour de chaînes correspondantes ou une autre syntaxe textuelle définissant comment remplacer les modèles correspondants dans les cordes opérées.

Syntaxe #

  • regex_match // Retourne si la séquence entière de caractères correspond à la regex, en capturant éventuellement dans un objet match
  • regex_search // Retourne si une partie de la séquence de caractères correspond à la regex, capturant éventuellement dans un objet match
  • regex_replace // Renvoie la séquence de caractères d’entrée telle que modifiée par une regex via une chaîne de format de remplacement
  • regex_token_iterator // Initialisé avec une séquence de caractères définie par des itérateurs, une liste d’index de capture à itérer et une regex. Le déréférencement renvoie la correspondance actuellement indexée de la regex. L’incrémentation passe à l’index de capture suivant ou, s’il se trouve actuellement au dernier index, réinitialise l’index et bloque la prochaine occurrence d’une correspondance de regex dans la séquence de caractères
  • regex_iterator // Initialisé avec une séquence de caractères définie par des itérateurs et une regex. Le déréférencement renvoie la partie de la séquence de caractères à laquelle l’expression régulière entière correspond actuellement. L’incrémentation trouve la prochaine occurrence d’une correspondance de regex dans la séquence de caractères

Paramètres #

Signature Description
bool regex_match(Iterator bidirectionnel en premier, Iterator bidirectionnel en dernier, smatch& sm, const regex& re, regex_constraints::match_flag_type flags) BidirectionalIterator est n’importe quel itérateur de caractère qui fournit des opérateurs d’incrémentation et de décrémentation smatch peut être cmatch ou toute autre variante de match_results qui accepte le type de BidirectionalIterator le smatch L’argument peut être omis si les résultats de la regex ne sont pas nécessaires.
bool regex_match(const string& str, smatch& sm, const regex re&, regex_constraints::match_flag_type flags) string peut être soit un const char* soit une L-Value string, les fonctions acceptant une R-Value string sont explicitement supprimées smatch peut être cmatch ou toute autre variante de match_results qui accepte le type de str l’argument smatch peut être omis si les résultats de la regex ne sont pas nécessaires ** Renvoie ** si re correspond à la séquence de caractères entière défini par str

Exemples de base de regex_match et regex_search #

const auto input = "Some people, when confronted with a problem, think \"I know, I'll use regular expressions.\""s;
smatch sm;

cout << input << endl;

// If input ends in a quotation that contains a word that begins with "reg" and another word begining with "ex" then capture the preceeding portion of input
if (regex_match(input, sm, regex("(.*)\".*\\breg.*\\bex.*\"\\s*$"))) {
    const auto capture = sm[1].str();
    
    cout << '\t' << capture << endl; // Outputs: "\tSome people, when confronted with a problem, think\n"
    
    // Search our capture for "a problem" or "# problems"
    if(regex_search(capture, sm, regex("(a|d+)\\s+problems?"))) {
        const auto count = sm[1] == "a"s ? 1 : stoi(sm[1]);
        
        cout << '\t' << count << (count > 1 ? " problems\n" : " problem\n"); // Outputs: "\t1 problem\n"
        cout << "Now they have " << count + 1 << " problems.\n"; // Ouputs: "Now they have 2 problems\n"
    }
}

regex_iterator Exemple #

Lorsque le traitement des captures doit être effectué de manière itérative, un regex_iterator est un bon choix. Déréférencer un regex_iterator renvoie un match_result. C’est idéal pour les captures conditionnelles ou les captures interdépendantes. Disons que nous voulons tokeniser du code C++. Donné:

enum TOKENS {
    NUMBER,
    ADDITION,
    SUBTRACTION,
    MULTIPLICATION,
    DIVISION,
    EQUALITY,
    OPEN_PARENTHESIS,
    CLOSE_PARENTHESIS
};

Nous pouvons tokeniser cette chaîne : const auto input = "42/2 + -8\t=\n(2 + 2) * 2 * 2 -3"s avec un regex_iterator comme ceci :

vector<TOKENS> tokens;
const regex re{ "\\s*(\\(?)\\s*(-?\\s*\\d+)\\s*(\\)?)\\s*(?:(\\+)|(-)|(\\*)|(/)|(=))" };

for_each(sregex_iterator(cbegin(input), cend(input), re), sregex_iterator(), [&](const auto& i) {
    if(i[1].length() > 0) {
        tokens.push_back(OPEN_PARENTHESIS);
    }
    
    tokens.push_back(i[2].str().front() == '-' ? NEGATIVE_NUMBER : NON_NEGATIVE_NUMBER);
    
    if(i[3].length() > 0) {
        tokens.push_back(CLOSE_PARENTHESIS);
    }        
    
    auto it = next(cbegin(i), 4);
    
    for(int result = ADDITION; it != cend(i); ++result, ++it) {
        if (it->length() > 0U) {
            tokens.push_back(static_cast<TOKENS>(result));
            break;
        }
    }
});

match_results<string::const_reverse_iterator> sm;

if(regex_search(crbegin(input), crend(input), sm, regex{ tokens.back() == SUBTRACTION ? "^\\s*\\d+\\s*-\\s*(-?)" : "^\\s*\\d+\\s*(-?)" })) {
    tokens.push_back(sm[1].length() == 0 ? NON_NEGATIVE_NUMBER : NEGATIVE_NUMBER);
}

[Exemple en direct] (http://ideone.com/Rv5WNI)

Un piège notable avec les itérateurs regex est que l’argument regex doit être une valeur L, une valeur R ne fonctionnera pas : http://stackoverflow.com/q/29895747/2642059

Ancres #

C++ ne fournit que 4 ancres :

  • ^ qui affirme le début de la chaîne
  • $ qui affirme la fin de la chaîne
  • \b qui affirme un caractère \W ou le début ou la fin de la chaîne
  • \B qui affirme un caractère \w

Disons par exemple que nous voulons capturer un nombre avec son signe :

auto input = "+1--12*123/+1234"s;
smatch sm;

if(regex_search(input, sm, regex{ "(?:^|\\b\\W)([+-]?\\d+)" })) {

    do {
        cout << sm[1] << endl;
        input = sm.suffix().str();
    } while(regex_search(input, sm, regex{ "(?:^\\W|\\b\\W)([+-]?\\d+)" }));
}

[Exemple en direct] (http://ideone.com/uE4dGr)

Une remarque importante ici est que l’ancre ne consomme aucun caractère.

regex_replace Exemple #

Ce code prend différents styles d’accolades et les convertit en One True Brace Style :

const auto input = "if (KnR)\n\tfoo();\nif (spaces) {\n    foo();\n}\nif (allman)\n{\n\tfoo();\n}\nif (horstmann)\n{\tfoo();\n}\nif (pico)\n{\tfoo(); }\nif (whitesmiths)\n\t{\n\tfoo();\n\t}\n"s;

cout << input << regex_replace(input, regex("(.+?)\\s*\\{?\\s*(.+?;)\\s*\\}?\\s*"), "$1 {\n\t$2\n}\n") << endl;

regex_token_iterator Exemple #

Un [std::regex_token_iterator][1] fournit un outil formidable pour [extraire des éléments d’un fichier de valeurs séparées par des virgules][2]. Outre les avantages de l’itération, cet itérateur est également capable de capturer des virgules échappées là où d’autres méthodes ont du mal :

const auto input = "please split,this,csv, ,line,\\,\n"s;
const regex re{ "((?:[^\\\\,]|\\\\.)+)(?:,|$)" };
const vector<string> m_vecFields{ sregex_token_iterator(cbegin(input), cend(input), re, 1), sregex_token_iterator() };

cout << input << endl;

copy(cbegin(m_vecFields), cend(m_vecFields), ostream_iterator<string>(cout, "\n"));

[Exemple en direct] (http://ideone.com/lySlTJ)

Un piège notable avec les itérateurs regex est que l’argument regex doit être une valeur L. [Une valeur R ne fonctionnera pas] [3].

[1] : http://en.cppreference.com/w/cpp/regex/regex_token_iterator [2] : http://stackoverflow.com/a/28880605/2642059 [3] : http://stackoverflow.com/q/29895747/2642059

Fractionner une chaîne #

std::vector<std::string> split(const std::string &str, std::string regex)
{
    std::regex r{ regex };
    std::sregex_token_iterator start{ str.begin(), str.end(), r, -1 }, end;
    return std::vector<std::string>(start, end);
}

split("Some  string\t with whitespace ", "\\s+"); // "Some", "string", "with", "whitespace"

Quantificateurs #

Disons qu’on nous donne const string input comme numéro de téléphone à valider. Nous pourrions commencer par exiger une entrée numérique avec un zéro ou plusieurs quantificateurs : regex_match(input, regex("\\d*")) ou un un ou plusieurs quantificateurs : regex_match(input , regex("\\d+")) Mais les deux sont vraiment insuffisants si input contient une chaîne numérique invalide comme : “123” Utilisons un n quantificateur ou plus pour nous assurer que nous obtenons au moins 7 chiffres :

regex_match(input, regex("\\d{7,}"))

Cela garantira que nous aurons au moins un numéro de téléphone composé de chiffres, mais input pourrait également contenir une chaîne numérique trop longue comme : “123456789012”. Alors allons-y avec un ** entre n et m quantificateur ** de sorte que l’“entrée” est d’au moins 7 chiffres mais pas plus de 11 :

regex_match(input, regex("\\d{7,11}"));

Cela nous rapproche, mais les chaînes numériques illégales qui sont dans la plage de [7, 11] sont toujours acceptées, comme : “123456789”. Rendons donc le code de pays facultatif avec un quantificateur paresseux :

regex_match(input, regex("\\d?\\d{7,10}"))

Il est important de noter que le quantificateur paresseux correspond à aussi peu de caractères que possible, donc la seule façon dont ce caractère sera mis en correspondance est s’il y a déjà 10 caractères qui ont été mis en correspondance par \d{7,10} . (Pour faire correspondre avidement le premier caractère, nous aurions dû faire : \d{0,1}.) Le quantificateur paresseux peut être ajouté à n’importe quel autre quantificateur.

Maintenant, comment pourrions-nous rendre l’indicatif régional facultatif et n’accepter un indicatif de pays que si l’indicatif régional était présent ?

regex_match(input, regex("(?:\\d{3,4})?\\d{7}"))

Dans cette regex finale, le \d{7} nécessite 7 chiffres. Ces 7 chiffres sont éventuellement précédés de 3 ou 4 chiffres.

Notez que nous n’avons pas ajouté le quantificateur paresseux : \d{3,4}?\d{7}, le \d{3,4}? aurait correspondent à 3 ou 4 caractères, en préférant 3. Au lieu de cela, nous faisons en sorte que le groupe non capturant corresponde au plus une fois, préférant ne pas correspondre. Provoquant une non-concordance si input n’incluait pas l’indicatif régional comme : “1234567”.

En conclusion du sujet sur les quantificateurs, j’aimerais mentionner l’autre quantificateur d’ajout que vous pouvez utiliser, le quantificateur possessif. * Soit * le ** quantificateur paresseux ** ou le ** quantificateur possessif ** peut être ajouté à n’importe quel quantificateur. La seule fonction du quantificateur possessif est d’aider le moteur de regex en lui disant de prendre avidement ces caractères et de ne jamais les abandonner même si cela fait échouer la regex. Ceci par exemple n’a pas beaucoup de sens : regex_match(input, regex("\\d{3,4}+\\d{7})) Parce qu’un input comme : “1234567890” ne serait pas mis en correspondance avec \d{3,4}+ correspondra toujours à 4 caractères même si la correspondance de 3 aurait permis à l’expression régulière de réussir.
Le quantificateur possessif est mieux utilisé lorsque le jeton quantifié limite le nombre de caractères correspondants. Par exemple:

regex_match(input, regex("(?:.*\\d{3,4}+){3}"))

Peut être utilisé pour établir une correspondance si input contenait l’un des éléments suivants :

123 456 7890
123-456-7890
(123)456-7890
(123) 456 - 7890

Mais lorsque cette expression régulière brille vraiment, c’est lorsque input contient une entrée illégale :

12345 - 67890

Sans le quantificateur possessif, le moteur regex doit revenir en arrière et tester chaque combinaison de .* et de 3 ou 4 caractères pour voir s’il peut trouver une combinaison correspondante. Avec le quantificateur possessif, la regex commence là où le 2ème quantificateur possessif s’est arrêté, le caractère “0”, et le moteur de regex essaie d’ajuster le .* à autoriser \d{3,4} à correspondre ; quand ce n’est pas le cas, la regex échoue, aucun retour en arrière n’est effectué pour voir si un ajustement .* antérieur aurait pu permettre une correspondance.
