Syntaxe #

  • ::(opt) new (expression-list)(opt) new-type-id new-initializer(opt)
  • ::(opt) new (expression-list)(opt) (type-id) new-initializer(opt)
  • ::(opt) supprimer cast-expression
  • ::(opt) delete [] cast-expression
  • std::unique_ptr<type-id> var_name(new type-id(opt)); //C++11
  • std ::shared_ptr<type-id> var_name(new type-id(opt)); //C++11
  • std::shared_ptr<type-id> var_name = std::make_shared<type-id>(opt) ; //C++11
  • std::unique_ptr<type-id> var_name = std::make_unique<type-id>(opt) ; //C++14

Un :: de tête force la recherche de l’opérateur new ou delete dans la portée globale, remplaçant tout opérateur new ou delete spécifique à une classe surchargé.

Les arguments facultatifs suivant le mot clé new sont généralement utilisés pour appeler placement new, mais peuvent également être utilisé pour transmettre des informations supplémentaires à l’allocateur, comme une balise demandant que la mémoire soit allouée à partir d’un pool choisi.

Le type alloué est généralement spécifié explicitement, e.g. new Foo, mais peut aussi être écrit comme auto (depuis C++11) ou decltype(auto) (depuis C++14) pour le déduire de l’initialiseur.

L’initialisation de l’objet alloué se fait selon les mêmes règles que l’initialisation des variables locales. En particulier, l’objet sera initialisé par défaut si l’initialiseur est omis, et lors de l’allocation dynamique d’un type scalaire ou d’un tableau de type scalaire, il n’y a aucune garantie que la mémoire sera mise à zéro.

Un objet tableau créé par une new-expression doit être détruit en utilisant delete[], que la new-expression ait été écrite avec [] ou non. Par exemple:

using IA = int[4];
int* pIA = new IA;
delete[] pIA;  // right
// delete pIA;  // wrong

Stockage gratuit (Heap, allocation dynamique …) #

Le terme ‘heap’ est un terme informatique général désignant une zone de mémoire à partir de laquelle des portions peuvent être allouées et désallouées indépendamment de la mémoire fournie par la pile.

En C++, le Standard fait référence à cette zone sous le nom de Free Store, ce qui est considéré comme un terme plus précis.

Les zones de mémoire allouées à partir du Free Store peuvent vivre plus longtemps que la portée d’origine dans laquelle elles ont été allouées. Les données trop volumineuses pour être stockées sur la pile peuvent également être allouées à partir du Free Store.

La mémoire brute peut être allouée et désallouée par les mots clés new et delete.

float *foo = nullptr;
{
    *foo = new float; // Allocates memory for a float
    float bar;              // Stack allocated 
} // End lifetime of bar, while foo still alive

delete foo;          // Deletes the memory for the float at pF, invalidating the pointer
foo = nullptr;       // Setting the pointer to nullptr after delete is often considered good practice

Il est également possible d’allouer des tableaux de taille fixe avec new et delete, avec une syntaxe légèrement différente. L’allocation de tableau n’est pas compatible avec l’allocation non-tableau, et le mélange des deux entraînera une corruption du tas. L’allocation d’un tableau alloue également de la mémoire pour suivre la taille du tableau pour une suppression ultérieure d’une manière définie par l’implémentation.

// Allocates memory for an array of 256 ints
int *foo = new int[256];
// Deletes an array of 256 ints at foo
delete[] foo;

Lorsque vous utilisez new et delete à la place malloc et free, le constructeur et le destructeur seront exécutés (similaire à stack objets basés). C’est pourquoi new et delete sont préférés à malloc et free.

struct ComplexType {
    int a = 0;

    ComplexType() { std::cout << "Ctor" << std::endl; }
    ~ComplexType() { std::cout << "Dtor" << std::endl; }
};

// Allocates memory for a ComplexType, and calls its constructor
ComplexType *foo = new ComplexType();
//Calls the destructor for ComplexType() and deletes memory for a Complextype at pC
delete foo;

À partir de C++11, l’utilisation de pointeurs intelligents est recommandée pour indiquer la propriété.

C++14 a ajouté std::make_unique à la STL, modifiant la recommandation pour favoriser std::make_unique ou std::make_shared au lieu d’utiliser new et delete nus.

Emplacement nouveau #

Il y a des situations où nous ne voulons pas compter sur Free Store pour allouer de la mémoire et nous voulons utiliser des allocations de mémoire personnalisées en utilisant new.

Pour ces situations, nous pouvons utiliser Placement New, où nous pouvons dire à l’opérateur `new’ d’allouer de la mémoire à partir d’un emplacement de mémoire pré-alloué

Par exemple

int a4byteInteger;

char *a4byteChar = new (&a4byteInteger) char[4];

Dans cet exemple, la mémoire pointée par a4byteChar est de 4 octets alloués à ‘stack’ via la variable entière a4byteInteger.

L’avantage de ce type d’allocation de mémoire est le fait que les programmeurs contrôlent l’allocation. Dans l’exemple ci-dessus, puisque a4byteInteger est alloué sur la pile, nous n’avons pas besoin de faire un appel explicite à ‘delete a4byteChar`.

Le même comportement peut également être obtenu pour la mémoire allouée dynamique. Par exemple

int *a8byteDynamicInteger = new int[2];

char *a8byteChar = new (a8byteDynamicInteger) char[8];

Dans ce cas, le pointeur de mémoire par a8byteChar fera référence à la mémoire dynamique allouée par a8byteDynamicInteger. Dans ce cas cependant, nous devons appeler explicitementdelete a8byteDynamicInteger pour libérer la mémoire

Un autre exemple pour la classe C++

struct ComplexType {
    int a;

    ComplexType() : a(0) {}
    ~ComplexType() {}
};

int main() {
    char* dynArray = new char[256];

    //Calls ComplexType's constructor to initialize memory as a ComplexType
    new((void*)dynArray) ComplexType();

    //Clean up memory once we're done
    reinterpret_cast<ComplexType*>(dynArray)->~ComplexType();
    delete[] dynArray;

    //Stack memory can also be used with placement new
    alignas(ComplexType) char localArray[256]; //alignas() available since C++11

    new((void*)localArray) ComplexType();

    //Only need to call the destructor for stack memory
    reinterpret_cast<ComplexType*>(localArray)->~ComplexType();

    return 0;
}

Empiler #

La pile est une petite région de mémoire dans laquelle des valeurs temporaires sont placées pendant l’exécution. L’allocation de données dans la pile est très rapide par rapport à l’allocation de tas, car toute la mémoire a déjà été affectée à cette fin.

int main() {
    int a = 0; //Stored on the stack
    return a;
}

La pile est nommée parce que les chaînes d’appels de fonction auront leur mémoire temporaire “empilée” les unes sur les autres, chacune utilisant une petite section de mémoire distincte.

float bar() {
    //f will be placed on the stack after anything else
    float f = 2;
    return f;
}

double foo() {
    //d will be placed just after anything within main()
    double d = bar();
    return d;
}

int main() {
    //The stack has no user variables stored in it until foo() is called
    return (int)foo();
}

Les données stockées sur la pile ne sont valides que tant que la portée qui a alloué la variable est toujours active.

int* pA = nullptr;

void foo() {
    int b = *pA;
    pA = &b;
}

int main() {
    int a = 5;
    pA = &a;
    foo();
    //Undefined behavior, the value pointed to by pA is no longer in scope
    a = *pA;
}
