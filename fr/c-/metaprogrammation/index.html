En C++, la métaprogrammation fait référence à l’utilisation de macros ou de modèles pour générer du code au moment de la compilation.

En général, les macros sont mal vues dans ce rôle et les modèles sont préférés, bien qu’ils ne soient pas aussi génériques.

La métaprogrammation de modèles utilise souvent des calculs au moment de la compilation, que ce soit via des modèles ou des fonctions constexpr, pour atteindre ses objectifs de génération de code, mais les calculs au moment de la compilation ne sont pas de la métaprogrammation en soi.

La métaprogrammation (ou plus précisément, la métaprogrammation de modèles) est la pratique consistant à utiliser des [modèles] [1] pour créer des constantes, des fonctions ou des structures de données au moment de la compilation. Cela permet aux calculs d’être effectués une seule fois au moment de la compilation plutôt qu’à chaque exécution.

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/460/templates/3999/basic-class-template#t=201607281319383275025

Calcul des factorielles #

Les factorielles peuvent être calculées au moment de la compilation en utilisant des techniques de métaprogrammation de modèles.

#include <iostream>

template<unsigned int n>
struct factorial
{
    enum
    {
        value = n * factorial<n - 1>::value
    };
};

template<>
struct factorial<0>
{
    enum { value = 1 };
};

int main()
{
    std::cout << factorial<7>::value << std::endl;    // prints "5040"
}

factorial est une structure, mais dans la métaprogrammation de modèle, il est traité comme une métafonction de modèle. Par convention, les métafonctions de modèle sont évaluées en vérifiant un membre particulier, soit ::type pour les métafonctions qui génèrent des types, soit ::value pour les métafonctions qui génèrent des valeurs.

Dans le code ci-dessus, nous évaluons la métafonction factorielle en instanciant le modèle avec les paramètres que nous voulons transmettre et en utilisant ::value pour obtenir le résultat de l’évaluation.

La métafonction elle-même repose sur l’instanciation récursive de la même métafonction avec des valeurs plus petites. La spécialisation factorielle<0> représente la condition de terminaison. La métaprogrammation de modèle a la plupart des restrictions d’un [langage de programmation fonctionnel] [1], donc la récursivité est la principale construction de “boucle”.

Étant donné que les métafonctions de modèle s’exécutent au moment de la compilation, leurs résultats peuvent être utilisés dans des contextes qui nécessitent des valeurs au moment de la compilation. Par exemple:

int my_array[factorial<5>::value];

Les tableaux automatiques doivent avoir une taille définie au moment de la compilation. Et le résultat d’une métafonction est une constante de compilation, elle peut donc être utilisée ici.

Limitation : la plupart des compilateurs n’autorisent pas la profondeur de récursivité au-delà d’une limite. Par exemple, le compilateur g++ limite par défaut la profondeur de récursivité à 256 niveaux. Dans le cas de g++, le programmeur peut définir la profondeur de récursivité à l’aide de l’option -ftemplate-depth-X.

Depuis C++11, le modèle std::integral_constant peut être utilisé pour ce type de calcul de modèle :

#include <iostream>
#include <type_traits>

template<long long n>
struct factorial :
  std::integral_constant<long long, n * factorial<n - 1>::value> {};

template<>
struct factorial<0> :
  std::integral_constant<long long, 1> {};

int main()
{
    std::cout << factorial<7>::value << std::endl;    // prints "5040"
}

De plus, les fonctions constexpr deviennent une alternative plus propre.

#include <iostream>

constexpr long long factorial(long long n)
{
  return (n == 0) ? 1 : n * factorial(n - 1);
}

int main()
{
  char test[factorial(3)];
  std::cout << factorial(7) << '\n';
}

Le corps de factorial() est écrit comme une seule instruction car en C++11, les fonctions constexpr ne peuvent utiliser qu’un sous-ensemble assez limité du langage.

Depuis C++14, de nombreuses restrictions pour les fonctions constexpr ont été supprimées et elles peuvent désormais être écrites de manière beaucoup plus pratique :

constexpr long long factorial(long long n)
{
  if (n == 0)
    return 1;
  else
    return n * factorial(n - 1);
}

Ou même:

constexpr long long factorial(int n)
{
  long long result = 1;
  for (int i = 1; i <= n; ++i) {
    result *= i;
  }
  return result;
}

Depuis c++17, on peut utiliser l’expression fold pour calculer la factorielle :

#include <iostream>
#include <utility>

template <class T, T N, class I = std::make_integer_sequence<T, N>>
struct factorial;

template <class T, T N, T... Is>
struct factorial<T,N,std::index_sequence<T, Is...>> {
   static constexpr T value = (static_cast<T>(1) * ... * (Is + 1));
};

int main() {
   std::cout << factorial<int, 5>::value << std::endl;
}

[1] : https://en.wikipedia.org/wiki/Functional_programming

Itérer sur un pack de paramètres #

Souvent, nous devons effectuer une opération sur chaque élément d’un pack de paramètres de modèle variadique. Il existe de nombreuses façons de procéder, et les solutions deviennent plus faciles à lire et à écrire avec C++17. Supposons que nous voulions simplement imprimer chaque élément d’un pack. La solution la plus simple est de répéter :

void print_all(std::ostream& os) {
    // base case
}

template <class T, class... Ts>
void print_all(std::ostream& os, T const& first, Ts const&... rest) {
    os << first;
    
    print_all(os, rest...);
}

Nous pourrions à la place utiliser l’astuce de l’expandeur, pour effectuer tout le streaming en une seule fonction. Cela a l’avantage de ne pas avoir besoin d’une deuxième surcharge, mais a l’inconvénient d’une lisibilité moins que stellaire :

template <class... Ts>
void print_all(std::ostream& os, Ts const&... args) {
    using expander = int[];
    (void)expander{0,
        (void(os << args), 0)...
    };
}

Pour une explication de la façon dont cela fonctionne, voir l’excellente réponse de T.C.

Avec C++17, nous avons deux nouveaux outils puissants dans notre arsenal pour résoudre ce problème. La première est une expression pliée :

template <class... Ts>
void print_all(std::ostream& os, Ts const&... args) {
    ((os << args), ...);
}

Et la seconde est if constexpr, qui nous permet d’écrire notre solution récursive originale dans une seule fonction :

template <class T, class... Ts>
void print_all(std::ostream& os, T const& first, Ts const&... rest) {
    os << first;

    if constexpr (sizeof...(rest) > 0) {        
        // this line will only be instantiated if there are further
        // arguments. if rest... is empty, there will be no call to
        // print_all(os). 
        print_all(os, rest...);
    }
}

Itérer avec std::integer_sequence #

Depuis C++14, la norme fournit le modèle de classe

template <class T, T... Ints>
class integer_sequence;

template <std::size_t... Ints>
using index_sequence = std::integer_sequence<std::size_t, Ints...>;

et une métafonction génératrice pour celle-ci :

template <class T, T N>
using make_integer_sequence = std::integer_sequence<T, /* a sequence 0, 1, 2, ..., N-1 */ >;

template<std::size_t N>
using make_index_sequence = make_integer_sequence<std::size_t, N>;

Bien que cela soit standard en C++14, cela peut être implémenté à l’aide d’outils C++11.

Nous pouvons utiliser cet outil pour appeler une fonction avec un std::tuple d’arguments (normalisé en C++17 comme std::apply) :

namespace detail {
    template <class F, class Tuple, std::size_t... Is>
    decltype(auto) apply_impl(F&& f, Tuple&& tpl, std::index_sequence<Is...> ) {
        return std::forward<F>(f)(std::get<Is>(std::forward<Tuple>(tpl))...);
    }
}

template <class F, class Tuple>
decltype(auto) apply(F&& f, Tuple&& tpl) {
    return detail::apply_impl(std::forward<F>(f),
        std::forward<Tuple>(tpl),
        std::make_index_sequence<std::tuple_size<std::decay_t<Tuple>>::value>{});
}


// this will print 3
int f(int, char, double);

auto some_args = std::make_tuple(42, 'x', 3.14);
int r = apply(f, some_args); // calls f(42, 'x', 3.14)

Envoi de balises #

Un moyen simple de sélectionner entre les fonctions au moment de la compilation consiste à envoyer une fonction à une paire de fonctions surchargées qui prennent une balise comme un (généralement le dernier) argument. Par exemple, pour implémenter std::advance(), nous pouvons dispatcher sur la catégorie iterator :

namespace details {
    template <class RAIter, class Distance>
    void advance(RAIter& it, Distance n, std::random_access_iterator_tag) {
        it += n;
    }

    template <class BidirIter, class Distance>
    void advance(BidirIter& it, Distance n, std::bidirectional_iterator_tag) {
        if (n > 0) {
            while (n--) ++it;
        }
        else {
            while (n++) --it;
        }
    }

    template <class InputIter, class Distance>
    void advance(InputIter& it, Distance n, std::input_iterator_tag) {
        while (n--) {
            ++it;
        }
    }    
}

template <class Iter, class Distance>
void advance(Iter& it, Distance n) {
    details::advance(it, n, 
            typename std::iterator_traits<Iter>::iterator_category{} );
}

Les arguments std::XY_iterator_tag des fonctions details::advance surchargées sont des paramètres de fonction inutilisés. L’implémentation réelle n’a pas d’importance (en fait, elle est complètement vide). Leur seul but est de permettre au compilateur de sélectionner une surcharge en fonction de la classe de balises details::advance qui est appelée.

Dans cet exemple, advance utilise la métafonction iterator_traits<T>::iterator_category qui renvoie l’une des classes iterator_tag, selon le type réel de Iter. Un objet construit par défaut de iterator_category<Iter>::type laisse ensuite le compilateur sélectionner l’une des différentes surcharges de details::advance. (Ce paramètre de fonction est susceptible d’être complètement optimisé, car il s’agit d’un objet construit par défaut d’un struct vide et jamais utilisé.)

La répartition des balises peut vous donner un code beaucoup plus facile à lire que les équivalents utilisant SFINAE et enable_if.

Remarque : bien que if constexpr de C++17 puisse simplifier l’implémentation de advance en particulier, il ne convient pas aux implémentations ouvertes contrairement à la répartition des balises.

Détecter si l’expression est valide #

Il est possible de détecter si un opérateur ou une fonction peut être appelée sur un type. Pour tester si une classe a une surcharge de std::hash, on peut faire ceci :

#include <functional> // for std::hash
#include <type_traits> // for std::false_type and std::true_type
#include <utility> // for std::declval

template<class, class = void>
struct has_hash
    : std::false_type
{};

template<class T>
struct has_hash<T, decltype(std::hash<T>()(std::declval<T>()), void())>
    : std::true_type
{};

Depuis C++17, std::void_t peut être utilisé pour simplifier ce type de construction

#include <functional> // for std::hash
#include <type_traits> // for std::false_type, std::true_type, std::void_t
#include <utility> // for std::declval

template<class, class = std::void_t<> >
struct has_hash
    : std::false_type
{};

template<class T>
struct has_hash<T, std::void_t< decltype(std::hash<T>()(std::declval<T>())) > >
    : std::true_type
{};

où std::void_t est défini comme :

template< class... > using void_t = void;

Pour détecter si un opérateur, tel que operator< est défini, la syntaxe est presque la même :

template<class, class = void>
struct has_less_than
    : std::false_type
{};

template<class T>
struct has_less_than<T, decltype(std::declval<T>() < std::declval<T>(), void())>
    : std::true_type
{};

Ceux-ci peuvent être utilisés pour utiliser un std::unordered_map<T> si T a une surcharge pour std::hash, mais sinon essayez d’utiliser un std::map<T> :

template <class K, class V>
using hash_invariant_map = std::conditional_t<
    has_hash<K>::value,
    std::unordered_map<K, V>,
    std::map<K,V>>;

Si-alors-sinon #

Le type std::conditional dans l’en-tête de la bibliothèque standard <type_traits> peut sélectionner un type ou l’autre, en fonction d’une valeur booléenne au moment de la compilation :

template<typename T>
struct ValueOrPointer
{
    typename std::conditional<(sizeof(T) > sizeof(void*)), T*, T>::type vop;
};

Cette structure contient un pointeur vers T si T est plus grand que la taille d’un pointeur, ou T lui-même s’il est inférieur ou égal à la taille d’un pointeur. Par conséquent, sizeof(ValueOrPointer) sera toujours <= sizeof(void*).

Puissance de calcul avec C++11 (et supérieur) #

Avec C++11 et supérieur, les calculs au moment de la compilation peuvent être beaucoup plus faciles. Par exemple, le calcul de la puissance d’un nombre donné au moment de la compilation sera le suivant :

template <typename T>
constexpr T calculatePower(T value, unsigned power) {
    return power == 0 ? 1 : value * calculatePower(value, power-1);
}

Le mot-clé constexpr est responsable du calcul de la fonction au moment de la compilation, puis et seulement alors, lorsque toutes les exigences pour cela seront remplies (voir plus à la référence du mot-clé constexpr), par exemple tous les arguments doivent être connus au moment de la compilation.

Remarque : dans C++11, la fonction constexpr ne doit composer qu’à partir d’une seule instruction de retour.

Avantages : Comparée à la méthode standard de calcul du temps de compilation, cette méthode est également utile pour les calculs d’exécution. Cela signifie que si les arguments de la fonction ne sont pas connus au moment de la compilation (par exemple, la valeur et la puissance sont données en entrée via l’utilisateur), alors la fonction est exécutée au moment de la compilation, il n’est donc pas nécessaire de dupliquer un code (comme nous serait forcé dans les anciennes normes de C++).

Par exemple.

void useExample() {
    constexpr int compileTimeCalculated = calculatePower(3, 3); // computes at compile time,
                               // as both arguments are known at compilation time
                               // and used for a constant expression.
    int value;
    std::cin >> value;
    int runtimeCalculated = calculatePower(value, 3);  // runtime calculated,
                                    // because value is known only at runtime.
}

Une autre façon de calculer la puissance au moment de la compilation peut utiliser l’expression fold comme suit :

#include <iostream>
#include <utility>

template <class T, T V, T N, class I = std::make_integer_sequence<T, N>>
struct power;

template <class T, T V, T N, T... Is>
struct power<T, V, N, std::integer_sequence<T, Is...>> {
   static constexpr T value = (static_cast<T>(1) * ... * (V * static_cast<bool>(Is + 1)));
};

int main() {
   std::cout << power<int, 4, 2>::value << std::endl;
}

Distinction manuelle des types lorsqu’on leur donne n’importe quel type T #

Lors de la mise en œuvre de SFINAE en utilisant std::enable_if , il est souvent utile d’avoir accès à helper templates qui détermine si un type “T” donné correspond à un ensemble de critères.

Pour nous y aider, la norme fournit déjà deux types analogues à true et false qui sont std::true_type et std::false_type.

L’exemple suivant montre comment détecter si un type T est un pointeur ou non, le modèle is_pointer imite le comportement de l’assistant standard std::is_pointer :

template <typename T>
struct is_pointer_: std::false_type {};

template <typename T>
struct is_pointer_<T*>: std::true_type {};

template <typename T>
struct is_pointer: is_pointer_<typename std::remove_cv<T>::type> { }

Il y a trois étapes dans le code ci-dessus (parfois vous n’en avez besoin que de deux):

  1. La première déclaration de is_pointer_ est le cas par défaut et hérite de std::false_type. Le cas default doit toujours hériter de std::false_type puisqu’il est analogue à une condition “false”.

  2. La deuxième déclaration spécialise le modèle is_pointer_ pour le pointeur T* sans se soucier de ce que T est réellement. Cette version hérite de std::true_type.

  3. La troisième déclaration (la vraie) supprime simplement toutes les informations inutiles de T (dans ce cas, nous supprimons les qualificatifs const et volatile), puis revient à l’une des deux déclarations précédentes.

Étant donné que is_pointer<T> est une classe, pour accéder à sa valeur, vous devez soit :

  • Utilisez ::value, par ex. is_pointer<int>::value – value est un membre de classe statique de type bool hérité de std::true_type ou std::false_type ;
  • Construire un objet de ce type, par ex. is_pointer<int>{} – Cela fonctionne car std::is_pointer hérite de son constructeur par défaut de std::true_type ou std::false_type (qui ont des constructeurs constexpr) et à la fois std::true_type et std::false_type ont des opérateurs de conversion constexpr en bool.

C’est une bonne habitude de fournir des “modèles d’aide d’assistance” qui vous permettent d’accéder directement à la valeur :

template <typename T>
constexpr bool is_pointer_v = is_pointer<T>::value;

En C++ 17 et supérieur, la plupart des modèles d’assistance fournissent déjà une version _v, par exemple :

template< class T > constexpr bool is_pointer_v = is_pointer<T>::value;
template< class T > constexpr bool is_reference_v = is_reference<T>::value;

Générique Min/Max avec nombre d’arguments variable #

Il est possible d’écrire une fonction générique (par exemple min) qui accepte divers types numériques et un nombre arbitraire d’arguments par méta-programmation de modèle. Cette fonction déclare un min pour deux arguments et récursivement pour plus.

template <typename T1, typename T2>
auto min(const T1 &a, const T2 &b) 
-> typename std::common_type<const T1&, const T2&>::type
{
    return a < b ? a : b;
}

template <typename T1, typename T2, typename ... Args>
auto min(const T1 &a, const T2 &b, const Args& ... args)
-> typename std::common_type<const T1&, const T2&, const Args& ...>::type
{
    return min(min(a, b), args...);
}

auto minimum = min(4, 5.8f, 3, 1.8, 3, 1.1, 9);
