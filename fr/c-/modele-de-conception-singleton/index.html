Modèle de conception singleton

Un Singleton est conçu pour s’assurer qu’une classe n’a qu’une seule instance et fournit un point d’accès global à celle-ci. Si vous n’avez besoin que d’une seule instance ou d’un point d’accès global pratique, mais pas des deux, envisagez d’autres options avant de vous tourner vers le singleton.

Les variables globales peuvent compliquer le raisonnement sur le code. Par exemple, si l’une des fonctions appelantes n’est pas satisfaite des données qu’elle reçoit d’un singleton, vous devez maintenant rechercher ce qui donne en premier lieu les mauvaises données au singleton.

Les singletons encouragent également le [couplage] [1], un terme utilisé pour décrire deux composants de code qui sont joints ensemble, réduisant ainsi la propre mesure d’autonomie de chaque composant.

Les singletons ne sont pas compatibles avec la concurrence. Lorsqu’une classe a un point d’accès global, chaque thread a la possibilité d’y accéder, ce qui peut entraîner des blocages et des conditions de concurrence.

Enfin, l’initialisation paresseuse peut entraîner des problèmes de performances si elle est initialisée au mauvais moment. La suppression de l’initialisation paresseuse supprime également certaines des fonctionnalités qui rendent Singleton intéressant en premier lieu, telles que le polymorphisme (voir Sous-classes).

Sources : [Modèles de programmation de jeu][2] par [Robert Nystrom][3]

[1] : https://en.wikipedia.org/wiki/Coupling_(programmation_informatique) [2] : http://gameprogrammingpatterns.com/singleton.html [3] : https://twitter.com/munificentbob

Initialisation paresseuse #

Cet exemple a été extrait de la section “Q & A” ici : http://stackoverflow.com/a/1008289/3807729

Voir cet article pour une conception simple d’un singleton paresseux évalué avec destruction garantie : http://stackoverflow.com/questions/270947/can-any-one-provide-me-a-sample-of-singleton-in-c/271104#271104

** Le singleton paresseux classique évalué et correctement détruit. **

class S
{
    public:
        static S& getInstance()
        {
            static S    instance; // Guaranteed to be destroyed.
                                  // Instantiated on first use.
            return instance;
        }
    private:
        S() {};                   // Constructor? (the {} brackets) are needed here.

        // C++ 03
        // ========
        // Dont forget to declare these two. You want to make sure they
        // are unacceptable otherwise you may accidentally get copies of
        // your singleton appearing.
        S(S const&);              // Don't Implement
        void operator=(S const&); // Don't implement

        // C++ 11
        // =======
        // We can use the better technique of deleting the methods
        // we don't want.
    public:
        S(S const&)               = delete;
        void operator=(S const&)  = delete;

        // Note: Scott Meyers mentions in his Effective Modern
        //       C++ book, that deleted functions should generally
        //       be public as it results in better error messages
        //       due to the compilers behavior to check accessibility
        //       before deleted status
};

Singleton statique sécurisé pour la désinitialisation. #

Il y a des moments avec plusieurs objets statiques où vous devez être en mesure de garantir que le singleton ne sera pas détruit tant que tous les objets statiques qui utilisent le singleton n’en auront plus besoin.

Dans ce cas, std::shared_ptr peut être utilisé pour garder le singleton actif pour tous les utilisateurs même lorsque les destructeurs statiques sont appelés à la fin du programme :

class Singleton
{
public:
    Singleton(Singleton const&) = delete;
    Singleton& operator=(Singleton const&) = delete;

    static std::shared_ptr<Singleton> instance()
    {
        static std::shared_ptr<Singleton> s{new Singleton};
        return s;
    }

private:
    Singleton() {}
};

REMARQUE : Cet exemple apparaît comme réponse dans la section Q&R ici.

Sous-classes #

class API
{
public:
    static API& instance();
    
    virtual ~API() {}
    
    virtual const char* func1() = 0;
    virtual void func2() = 0;
    
protected:
    API() {}
    API(const API&) = delete;
    API& operator=(const API&) = delete;
};

class WindowsAPI : public API
{
public:
    virtual const char* func1()  override { /* Windows code */ }
    virtual void func2() override { /* Windows code */ }    
};

class LinuxAPI : public API
{
public:
    virtual const char* func1() override { /* Linux code */ }
    virtual void func2() override { /* Linux code */ }    
};

API& API::instance() {
#if PLATFORM == WIN32
    static WindowsAPI instance;
#elif PLATFORM = LINUX
    static LinuxAPI instance;
#endif
    return instance;
}

Dans cet exemple, un simple commutateur de compilateur lie la classe “API” à la sous-classe appropriée. De cette manière, API est accessible sans être couplé à un code spécifique à la plate-forme.

Singeton thread-safe #

Les normes C++11 garantissent que l’initialisation des objets de portée de fonction est initialisée de manière synchronisée. Cela peut être utilisé pour implémenter un singleton thread-safe avec [initialisation paresseuse][1].

class Foo
{
public:
    static Foo& instance()
    {
        static Foo inst;
        return inst;
    }        
private:
    Foo() {}
    Foo(const Foo&) = delete;
    Foo& operator =(const Foo&) = delete;
};

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/2713/singleton-design-pattern/9075/lazy-initialization
