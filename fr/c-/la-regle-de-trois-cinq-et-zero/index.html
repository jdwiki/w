Règle du zéro #

Nous pouvons combiner les principes de la règle des cinq et [RAII] [1] pour obtenir une interface beaucoup plus légère : la règle du zéro : toute ressource qui doit être gérée doit être dans son propre type. Ce type devrait suivre la règle des cinq, mais tous les utilisateurs de cette ressource n’ont pas besoin d’écrire aucune des cinq fonctions membres spéciales et peuvent simplement toutes les définir par défaut.

Utilisation de la classe “Person” introduite dans [l’exemple de la règle de trois](https://www.wikiod.com/fr/docs/c%2b%2b/1206/the-rule-of-three-five-and-zero/9867/ rule-of-three#t=201607281743050894887), nous pouvons créer un objet de gestion des ressources pour cstrings :

class cstring {
private:
    char* p;

public:
    ~cstring() { delete [] p; }
    cstring(cstring const& );
    cstring(cstring&& );
    cstring& operator=(cstring const& );
    cstring& operator=(cstring&& );

    /* other members as appropriate */
};

Et une fois que cela est séparé, notre classe Person devient beaucoup plus simple :

class Person {
    cstring name;
    int arg;

public:
    ~Person() = default;
    Person(Person const& ) = default;
    Person(Person&& ) = default;
    Person& operator=(Person const& ) = default;
    Person& operator=(Person&& ) = default;

    /* other members as appropriate */
};

Les membres spéciaux dans Person n’ont même pas besoin d’être déclarés explicitement ; le compilateur les mettra par défaut ou les supprimera de manière appropriée, en fonction du contenu de Person. Par conséquent, ce qui suit est également un exemple de la règle du zéro.

struct Person {
    cstring name;
    int arg;
};

Si cstring devait être un type de déplacement uniquement, avec un constructeur/opérateur d’affectation de copie deleted, alors Person serait également automatiquement de type déplacement uniquement.

Le terme règle du zéro était [introduit par R. Martinho Fernandes] [2]

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/1320/raii-resource-acquisition-is-initialization#t=201607230254049480454 [2] : https://rmf.io/cxx11/rule-of-zero

Règle de cinq #

C++11 introduit deux nouvelles fonctions membres spéciales : le constructeur de déplacement et l’opérateur d’affectation de déplacement. Pour toutes les mêmes raisons que vous voulez suivre la [règle de trois][1] en C++03, vous voulez généralement suivre la règle de cinq en C++11 : Si une classe nécessite UNE des cinq fonctions membres spéciales , et si la sémantique de déplacement est souhaitée, elle nécessite très probablement les CINQ d’entre elles.

Notez cependant que le non-respect de la règle de cinq n’est généralement pas considéré comme une erreur, mais comme une opportunité d’optimisation manquée, tant que la règle de trois est toujours suivie. Si aucun constructeur de déplacement ou opérateur d’affectation de déplacement n’est disponible lorsque le compilateur en utiliserait normalement un, il utilisera à la place la sémantique de copie si possible, ce qui entraînera une opération moins efficace en raison d’opérations de copie inutiles. Si la sémantique de déplacement n’est pas souhaitée pour une classe, elle n’a pas besoin de déclarer un constructeur de déplacement ou un opérateur d’affectation.

Même exemple que pour la règle de trois :

class Person
{
    char* name;
    int age;

public:
    // Destructor 
    ~Person() { delete [] name; }

    // Implement Copy Semantics
    Person(Person const& other)
        : name(new char[std::strlen(other.name) + 1])
        , age(other.age)
    {
        std::strcpy(name, other.name);
    }
    
    Person &operator=(Person const& other) 
    {
        // Use copy and swap idiom to implement assignment.
        Person copy(other);
        swap(*this, copy);
        return *this;
    }

    // Implement Move Semantics
    // Note: It is usually best to mark move operators as noexcept
    //       This allows certain optimizations in the standard library
    //       when the class is used in a container.

    Person(Person&& that) noexcept
        : name(nullptr)               // Set the state so we know it is undefined
        , age(0)
    {
        swap(*this, that);
    }

    Person& operator=(Person&& that) noexcept
    {
        swap(*this, that);
        return *this;
    }

    friend void swap(Person& lhs, Person& rhs) noexcept
    {
        std::swap(lhs.name, rhs.name);
        std::swap(lhs.age, rhs.age);
    }
};

Alternativement, les opérateurs d’affectation de copie et de déplacement peuvent être remplacés par un seul opérateur d’affectation, qui prend une instance par valeur au lieu d’une référence ou d’une référence rvalue pour faciliter l’utilisation de l’idiome de copie et d’échange.

Person& operator=(Person copy)
{
    swap(*this, copy);
    return *this;
}

Passer de la règle de trois à la règle de cinq est important pour des raisons de performance, mais n’est pas strictement nécessaire dans la plupart des cas. L’ajout du constructeur de copie et de l’opérateur d’affectation garantit que le déplacement du type ne provoquera pas de fuite de mémoire (la construction de déplacement reviendra simplement à la copie dans ce cas), mais effectuera des copies que l’appelant n’a probablement pas anticipées.

[1] : https://www.wikiod.com/fr/docs/c%2b%2b/1206/the-rule-of-three-five-and-zero/9867/rule-of-three#t=201607251426414241612

Règle de Trois #

La règle de trois stipule que si un type a besoin d’avoir un constructeur de copie défini par l’utilisateur, un opérateur d’affectation de copie ou un destructeur, alors il doit avoir les trois.

La raison de la règle est qu’une classe qui a besoin de l’un des trois gère certaines ressources (descripteurs de fichiers, mémoire allouée dynamiquement, etc.), et les trois sont nécessaires pour gérer cette ressource de manière cohérente. Les fonctions de copie traitent de la façon dont la ressource est copiée entre les objets, et le destructeur détruirait la ressource, conformément aux [principes RAII][1].

Considérez un type qui gère une ressource de chaîne :

class Person
{
    char* name;
    int age;

public:
    Person(char const* new_name, int new_age)
        : name(new char[std::strlen(new_name) + 1])
        , age(new_age)
    {
       std::strcpy(name, new_name);
    }

    ~Person() {
        delete [] name;
    }
};

Étant donné que name a été alloué dans le constructeur, le destructeur le désalloue pour éviter les fuites de mémoire. Mais que se passe-t-il si un tel objet est copié ?

int main()
{
    Person p1("foo", 11);
    Person p2 = p1;
}

Tout d’abord, ‘p1’ sera construit. Ensuite, p2 sera copié à partir de p1. Cependant, le constructeur de copie généré par C++ copiera chaque composant du type tel quel. Ce qui signifie que p1.name et p2.name pointent tous les deux vers la même chaîne.

Lorsque main se termine, les destructeurs sont appelés. Le premier destructeur de p2 sera appelé ; cela supprimera la chaîne. Ensuite, le destructeur de p1 sera appelé. Cependant, la chaîne est déjà supprimée. L’appel de delete sur la mémoire qui a déjà été supprimée produit un comportement indéfini.

Pour éviter cela, il est nécessaire de fournir un constructeur de copie approprié. Une approche consiste à implémenter un système de comptage de références, dans lequel différentes instances “Person” partagent les mêmes données de chaîne. Chaque fois qu’une copie est effectuée, le nombre de références partagées est incrémenté. Le destructeur décrémente alors le décompte de références, ne libérant la mémoire que si le décompte est égal à zéro.

Ou nous pourrions implémenter [la sémantique des valeurs et le comportement de copie en profondeur][2] :

Person(Person const& other)
    : name(new char[std::strlen(other.name) + 1])
    , age(other.age)
{
    std::strcpy(name, other.name);
}

Person &operator=(Person const& other) 
{
    // Use copy and swap idiom to implement assignment
    Person copy(other);
    swap(copy);            //  assume swap() exchanges contents of *this and copy
    return *this;
}

La mise en œuvre de l’opérateur d’affectation de copie est compliquée par la nécessité de libérer un tampon existant. La technique de copie et d’échange crée un objet temporaire qui contient un nouveau tampon. Échanger le contenu de *this et copy donne la propriété à copy du tampon d’origine. La destruction de copy, au retour de la fonction, libère le tampon précédemment détenu par *this.

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/1320/raii-resource-acquisition-is-initialization#t=201607230301520057618 [2] : https://www.wikiod.com/fr/docs/c%2B%2B/1955/value-and-reference-semantics#t=201607230312543894418

Protection d’auto-assignation #

Lors de l’écriture d’un opérateur d’affectation de copie, il est très important qu’il puisse fonctionner en cas d’auto-affectation. C’est-à-dire qu’il doit permettre ceci:

SomeType t = ...;
t = t;

L’auto-assignation ne se produit généralement pas de manière aussi évidente. Cela se produit généralement via une route détournée à travers divers systèmes de code, où l’emplacement de l’affectation a simplement deux pointeurs ou références “Personne” et n’a aucune idée qu’il s’agit du même objet.

Tout opérateur d’affectation de copie que vous écrivez doit pouvoir en tenir compte.

La façon typique de le faire est d’envelopper toute la logique d’affectation dans une condition comme celle-ci :

SomeType &operator=(const SomeType &other)
{
    if(this != &other)
    {
        //Do assignment logic.
    }
    return *this;
}

Remarque : Il est important de penser à l’auto-attribution et de s’assurer que votre code se comporte correctement lorsqu’il se produit. Cependant, l’auto-assignation est un événement très rare et l’optimisation pour l’empêcher peut en fait pessimiser le cas normal. Étant donné que le cas normal est beaucoup plus courant, pessimiser pour l’auto-assignation peut bien réduire l’efficacité de votre code (faites donc attention à l’utiliser).

A titre d’exemple, la technique normale pour implémenter l’opérateur d’affectation est “l’idiome de copie et d’échange”. L’implémentation normale de cette technique ne prend pas la peine de tester l’auto-assignation (même si l’auto-assignation coûte cher car une copie est faite). La raison en est que la pessimisation du cas normal s’est avérée beaucoup plus coûteuse (comme cela se produit plus souvent).

Les opérateurs d’affectation de déplacement doivent également être protégés contre l’auto-affectation. Cependant, la logique de nombreux opérateurs de ce type est basée sur std::swap, qui peut très bien gérer l’échange depuis/vers la même mémoire. Ainsi, si votre logique d’affectation de déplacement n’est rien de plus qu’une série d’opérations d’échange, vous n’avez pas besoin d’une protection d’auto-affectation.

Si ce n’est pas le cas, vous devez prendre des mesures similaires à celles décrites ci-dessus.
