Le préprocesseur C est un simple analyseur/remplacement de texte exécuté avant la compilation réelle du code. Utilisé pour étendre et faciliter l’utilisation du langage C (et plus tard C++), il peut être utilisé pour :

un. Y compris d’autres fichiers en utilisant #include

b. Définissez une macro de remplacement de texte en utilisant #define

c. Compilation conditionnelle en utilisant #if #ifdef

ré. Logique spécifique à la plate-forme/au compilateur (en tant qu’extension de la compilation conditionnelle)

Les instructions du préprocesseur sont exécutées avant que vos fichiers source ne soient transmis au compilateur. Ils sont capables d’une logique conditionnelle de très bas niveau. Étant donné que les constructions de préprocesseur (par exemple, les macros de type objet) ne sont pas typées comme les fonctions normales (l’étape de prétraitement se produit avant la compilation), le compilateur ne peut pas appliquer les vérifications de type, elles doivent donc être utilisées avec précaution.

Inclure les gardes #

Un fichier d’en-tête peut être inclus par d’autres fichiers d’en-tête. Un fichier source (unité de compilation) qui comprend plusieurs en-têtes peut donc, indirectement, inclure plusieurs en-têtes plus d’une fois. Si un tel fichier d’en-tête qui est inclus plus d’une fois contient des définitions, le compilateur (après le prétraitement) détecte une violation de la règle de définition unique (par exemple §3.2 de la norme C++ 2003) et émet donc un diagnostic et la compilation échoue.

L’inclusion multiple est empêchée à l’aide de “protections d’inclusion”, parfois également appelées protections d’en-tête ou protections de macro. Celles-ci sont implémentées en utilisant les directives du préprocesseur #define, #ifndef, #endif.

par exemple.

// Foo.h
#ifndef FOO_H_INCLUDED 
#define FOO_H_INCLUDED

class Foo    //  a class definition
{
};

#endif

Le principal avantage de l’utilisation des protections d’inclusion est qu’elles fonctionneront avec tous les compilateurs et préprocesseurs conformes aux normes.

Cependant, les protections d’inclusion posent également des problèmes aux développeurs, car il est nécessaire de s’assurer que les macros sont uniques dans tous les en-têtes utilisés dans un projet. Plus précisément, si deux en-têtes (ou plus) utilisent FOO_H_INCLUDED comme garde d’inclusion, le premier de ces en-têtes inclus dans une unité de compilation empêchera effectivement les autres d’être inclus. Des défis particuliers sont introduits si un projet utilise un certain nombre de bibliothèques tierces avec des fichiers d’en-tête qui utilisent des gardes d’inclusion en commun.

Il est également nécessaire de s’assurer que les macros utilisées dans les protections d’inclusion n’entrent pas en conflit avec d’autres macros définies dans les fichiers d’en-tête.

[La plupart des implémentations C++][1] prennent également en charge la directive #pragma once qui garantit que le fichier n’est inclus qu’une seule fois dans une seule compilation. Il s’agit d’une directive [de facto standard][2], mais elle ne fait partie d’aucune norme ISO C++. Par exemple:

// Foo.h
#pragma once

class Foo
{
};

Alors que #pragma once évite certains problèmes associés aux protections d’inclusion, un #pragma - par définition dans les normes - est intrinsèquement un crochet spécifique au compilateur et sera silencieusement ignoré par les compilateurs qui ne le supportent pas. Les projets qui utilisent #pragma once sont plus difficiles à porter vers des compilateurs qui ne le supportent pas.

Un certain nombre de directives de codage et de normes d’assurance pour C++ découragent spécifiquement toute utilisation du préprocesseur autre que pour #include les fichiers d’en-tête ou dans le but de placer des gardes d’inclusion dans les en-têtes.

[1] : https://en.wikipedia.org/wiki/Pragma_once#Portabilité [2] : https://en.wikipedia.org/wiki/De_facto_standard

Logique conditionnelle et gestion multiplateforme #

En un mot, la logique de prétraitement conditionnel consiste à rendre la logique de code disponible ou indisponible pour la compilation à l’aide de définitions de macros.

Les trois principaux cas d’utilisation sont :

  • différents profils d’application (par exemple, débogage, publication, test, optimisé) qui peuvent être des candidats de la même application (par exemple, avec une journalisation supplémentaire).
  • compilation multiplateforme - base de code unique, plusieurs plates-formes de compilation.
  • utilisant une base de code commune pour plusieurs versions d’application (par exemple, les versions Basic, Premium et Pro d’un logiciel) - avec des fonctionnalités légèrement différentes.

Exemple a : Une approche multiplateforme pour la suppression de fichiers (à titre indicatif) :

#ifdef _WIN32
#include <windows.h> // and other windows system files
#endif
#include <cstdio>

bool remove_file(const std::string &path) 
{
#ifdef _WIN32
  return DeleteFile(path.c_str());
#elif defined(_POSIX_VERSION) || defined(__unix__)
  return (0 == remove(path.c_str()));
#elif defined(__APPLE__)
  //TODO: check if NSAPI has a more specific function with permission dialog
  return (0 == remove(path.c_str()));
#else 
#error "This platform is not supported"
#endif
}

Les macros comme _WIN32, __APPLE__ ou __unix__ sont normalement prédéfinies par les implémentations correspondantes.

Exemple b : Activation de la journalisation supplémentaire pour une version de débogage :

void s_PrintAppStateOnUserPrompt()
{
    std::cout << "--------BEGIN-DUMP---------------\n"
              << AppState::Instance()->Settings().ToString() << "\n"
#if ( 1 == TESTING_MODE ) //privacy: we want user details only when testing
              << ListToString(AppState::UndoStack()->GetActionNames())
              << AppState::Instance()->CrntDocument().Name() 
              << AppState::Instance()->CrntDocument().SignatureSHA() << "\n"
#endif
              << "--------END-DUMP---------------\n"
}

Exemple c : Activer une fonctionnalité premium dans une version de produit distincte (remarque : ceci est illustratif. Il est souvent préférable d’autoriser le déverrouillage d’une fonctionnalité sans qu’il soit nécessaire de réinstaller une application)

void MainWindow::OnProcessButtonClick()
{
#ifndef _PREMIUM
    CreatePurchaseDialog("Buy App Premium", "This feature is available for our App Premium users. Click the Buy button to purchase the Premium version at our website");
    return;
#endif
    //...actual feature logic here
}

Quelques astuces courantes :

Définir les symboles au moment de l’invocation :

Le préprocesseur peut être appelé avec des symboles prédéfinis (avec initialisation optionnelle). Par exemple cette commande (gcc -E exécute uniquement le préprocesseur)

gcc -E -DOPTIMISE_FOR_OS_X -DTESTING_MODE=1 Sample.cpp

traite Sample.cpp de la même manière que si #define OPTIMISE_FOR_OS_X et #define TESTING_MODE 1 étaient ajoutés en haut de Sample.cpp.

S’assurer qu’une macro est définie :

Si une macro n’est pas définie et que sa valeur est comparée ou vérifiée, le préprocesseur suppose presque toujours silencieusement que la valeur est “0”. Il y a plusieurs façons de travailler avec cela. Une approche consiste à supposer que les paramètres par défaut sont représentés par 0 et que toute modification (par exemple, au profil de construction de l’application) doit être effectuée explicitement (par exemple, ENABLE_EXTRA_DEBUGGING=0 par défaut, définissez -DENABLE_EXTRA_DEBUGGING=1 pour remplacer). Une autre approche consiste à rendre toutes les définitions et valeurs par défaut explicites. Ceci peut être réalisé en utilisant une combinaison des directives #ifndef et #error :

#ifndef (ENABLE_EXTRA_DEBUGGING)
// please include DefaultDefines.h if not already included.
#    error "ENABLE_EXTRA_DEBUGGING is not defined"
#else
#    if ( 1 == ENABLE_EXTRA_DEBUGGING )
  //code
#    endif
#endif

Macros #

Les macros sont classées en deux groupes principaux : les macros de type objet et les macros de type fonction. Les macros sont traitées comme une substitution de jeton au début du processus de compilation. Cela signifie que de grandes sections (ou répétitives) de code peuvent être abstraites dans une macro de préprocesseur.

// This is an object-like macro
#define    PI         3.14159265358979

// This is a function-like macro.
// Note that we can use previously defined macros
// in other macro definitions (object-like or function-like)
// But watch out, its quite useful if you know what you're doing, but the
// Compiler doesnt know which type to handle, so using inline functions instead
// is quite recommended (But e.g. for Minimum/Maximum functions it is quite useful)
#define    AREA(r)    (PI*(r)*(r))

// They can be used like this:
double pi_macro   = PI;
double area_macro = AREA(4.6);

La bibliothèque Qt utilise cette technique pour créer un système de méta-objets en demandant à l’utilisateur de déclarer la macro Q_OBJECT en tête de la classe définie par l’utilisateur étendant QObject.

Les noms de macro sont généralement écrits en majuscules, pour les différencier plus facilement du code normal. Ce n’est pas une exigence, mais c’est simplement considéré comme un bon style par de nombreux programmeurs.

Lorsqu’une macro de type objet est rencontrée, elle est développée comme une simple opération de copier-coller, le nom de la macro étant remplacé par sa définition. Lorsqu’une macro de type fonction est rencontrée, son nom et ses paramètres sont développés.

double pi_squared = PI * PI;
// Compiler sees:
double pi_squared = 3.14159265358979 * 3.14159265358979;

double area = AREA(5);
// Compiler sees:
double area = (3.14159265358979*(5)*(5))

Pour cette raison, les paramètres de macro de type fonction sont souvent entre parenthèses, comme dans AREA() ci-dessus. Cela permet d’éviter tout bogue pouvant survenir lors de l’expansion de la macro, en particulier les bogues causés par un seul paramètre de macro composé de plusieurs valeurs réelles.

#define BAD_AREA(r) PI * r * r

double bad_area = BAD_AREA(5 + 1.6);
// Compiler sees:
double bad_area = 3.14159265358979 * 5 + 1.6 * 5 + 1.6;

double good_area = AREA(5 + 1.6);
// Compiler sees:
double good_area = (3.14159265358979*(5 + 1.6)*(5 + 1.6));

Notez également qu’en raison de cette extension simple, des précautions doivent être prises avec les paramètres passés aux macros, pour éviter des effets secondaires inattendus. Si le paramètre est modifié lors de l’évaluation, il sera modifié à chaque fois qu’il sera utilisé dans la macro développée, ce qui n’est généralement pas ce que nous souhaitons. Cela est vrai même si la macro met les paramètres entre parenthèses pour empêcher l’expansion de casser quoi que ce soit.

int oops = 5;
double incremental_damage = AREA(oops++);
// Compiler sees:
double incremental_damage = (3.14159265358979*(oops++)*(oops++));

De plus, les macros ne fournissent aucune sécurité de type, ce qui entraîne des erreurs difficiles à comprendre concernant l’incompatibilité de type.

Comme les programmeurs terminent normalement les lignes par un point-virgule, les macros destinées à être utilisées comme lignes autonomes sont souvent conçues pour “avaler” un point-virgule ; cela empêche tout bogue involontaire d’être causé par un point-virgule supplémentaire.

#define IF_BREAKER(Func) Func();

if (some_condition)
    // Oops.
    IF_BREAKER(some_func);
else
    std::cout << "I am accidentally an orphan." << std::endl;

Dans cet exemple, le double point-virgule par inadvertance rompt le bloc if...else, empêchant le compilateur de faire correspondre le else au if. Pour éviter cela, le point-virgule est omis de la définition de la macro, ce qui lui fera “avaler” le point-virgule immédiatement après toute utilisation de celui-ci.

#define IF_FIXER(Func) Func()

if (some_condition)
    IF_FIXER(some_func);
else
    std::cout << "Hooray!  I work again!" << std::endl;

La suppression du point-virgule final permet également d’utiliser la macro sans terminer l’instruction en cours, ce qui peut être bénéfique.

#define DO_SOMETHING(Func, Param) Func(Param, 2)

// ...

some_function(DO_SOMETHING(some_func, 3), DO_SOMETHING(some_func, 42));

Normalement, une définition de macro se termine à la fin de la ligne. Si une macro doit couvrir plusieurs lignes, cependant, une barre oblique inverse peut être utilisée à la fin d’une ligne pour l’indiquer. Cette barre oblique inverse doit être le dernier caractère de la ligne, ce qui indique au préprocesseur que la ligne suivante doit être concaténée sur la ligne courante, en les traitant comme une seule ligne. Cela peut être utilisé plusieurs fois de suite.

#define TEXT "I \
am \
many \
lines."

// ...

std::cout << TEXT << std::endl; // Output:   I am many lines.

Ceci est particulièrement utile dans les macros complexes de type fonction, qui peuvent avoir besoin de couvrir plusieurs lignes.

#define CREATE_OUTPUT_AND_DELETE(Str) \
    std::string* tmp = new std::string(Str); \
    std::cout << *tmp << std::endl; \
    delete tmp;

// ...

CREATE_OUTPUT_AND_DELETE("There's no real need for this to use 'new'.")

Dans le cas de macros de type fonction plus complexes, il peut être utile de leur donner leur propre portée pour éviter d’éventuelles collisions de noms ou pour provoquer la destruction d’objets à la fin de la macro, comme une fonction réelle. Un idiome courant pour cela est do while 0, où la macro est enfermée dans un bloc do-while. Ce bloc n’est généralement pas suivi d’un point-virgule, ce qui lui permet d’avaler un point-virgule.

#define DO_STUFF(Type, Param, ReturnVar) do { \
    Type temp(some_setup_values); \
    ReturnVar = temp.process(Param); \
} while (0)

int x;
DO_STUFF(MyClass, 41153.7, x);

// Compiler sees:

int x;
do {
    MyClass temp(some_setup_values);
    x = temp.process(41153.7);
} while (0);

Il existe également des macros variadiques ; comme pour les fonctions variadiques, celles-ci prennent un nombre variable d’arguments, puis les développent tous à la place d’un paramètre spécial “Varargs”, __VA_ARGS__.

#define VARIADIC(Param, ...) Param(__VA_ARGS__)

VARIADIC(printf, "%d", 8);
// Compiler sees:
printf("%d", 8);

Notez que pendant l’expansion, __VA_ARGS__ peut être placé n’importe où dans la définition, et sera développé correctement.

#define VARIADIC2(POne, PTwo, PThree, ...) POne(PThree, __VA_ARGS__, PTwo)

VARIADIC2(some_func, 3, 8, 6, 9);
// Compiler sees:
some_func(8, 6, 9, 3);

Dans le cas d’un paramètre variadique sans argument, différents compilateurs géreront différemment la virgule finale. Certains compilateurs, tels que Visual Studio, avaleront silencieusement la virgule sans aucune syntaxe spéciale. D’autres compilateurs, tels que GCC, exigent que vous placiez ## immédiatement avant __VA_ARGS__. Pour cette raison, il est sage de définir conditionnellement les macros variadiques lorsque la portabilité est un problème.

// In this example, COMPILER is a user-defined macro specifying the compiler being used.

#if       COMPILER == "VS"
    #define VARIADIC3(Name, Param, ...) Name(Param, __VA_ARGS__)
#elif     COMPILER == "GCC"
    #define VARIADIC3(Name, Param, ...) Name(Param, ##__VA_ARGS__)
#endif /* COMPILER */

X-macros #

Une technique idiomatique pour générer des structures de code répétitives au moment de la compilation.

Une macro X se compose de deux parties : la liste et l’exécution de la liste.

Exemple:

#define LIST \
    X(dog)   \
    X(cat)   \
    X(racoon)

// class Animal {
//  public:
//    void say();
// };

#define X(name) Animal name;
LIST
#undef X

int main() {
#define X(name) name.say();
    LIST
#undef X

    return 0;
}

qui est développé par le préprocesseur comme suit :

Animal dog;
Animal cat;
Animal racoon;

int main() {
    dog.say();
    cat.say();
    racoon.say();

    return 0;
}

Au fur et à mesure que les listes deviennent plus grandes (disons, plus de 100 éléments), cette technique permet d’éviter un copier-coller excessif.

Source : https://en.wikipedia.org/wiki/X_Macro

Voir aussi : https://www.wikiod.com/fr/c/macros-x

Si vous n’aimez pas définir un X apparemment non pertinent avant d’utiliser LIST, vous pouvez également passer un nom de macro en tant qu’argument :

#define LIST(MACRO) \
    MACRO(dog) \
    MACRO(cat) \
    MACRO(racoon)

Maintenant, vous spécifiez explicitement quelle macro doit être utilisée lors de l’expansion de la liste, par ex.

#define FORWARD_DECLARE_ANIMAL(name) Animal name;
LIST(FORWARD_DECLARE_ANIMAL)

Si chaque invocation de la MACRO doit prendre des paramètres supplémentaires - constants par rapport à la liste, des macros variadiques peuvent être utilisées

//a walkaround for Visual studio
#define EXPAND(x) x

#define LIST(MACRO, ...) \
    EXPAND(MACRO(dog, __VA_ARGS__)) \
    EXPAND(MACRO(cat, __VA_ARGS__)) \
    EXPAND(MACRO(racoon, __VA_ARGS__))

Le premier argument est fourni par la LIST, tandis que le reste est fourni par l’utilisateur dans l’invocation de LIST. Par exemple:

#define FORWARD_DECLARE(name, type, prefix) type prefix##name;
LIST(FORWARD_DECLARE,Animal,anim_)
LIST(FORWARD_DECLARE,Object,obj_)

s’étendra à

Animal anim_dog;
Animal anim_cat;
Animal anim_racoon;
Object obj_dog;
Object obj_cat;
Object obj_racoon;

Messages d’erreur du préprocesseur #

Des erreurs de compilation peuvent être générées à l’aide du préprocesseur. Ceci est utile pour un certain nombre de raisons, dont certaines incluent la notification à un utilisateur s’il se trouve sur une plate-forme non prise en charge ou un compilateur non pris en charge.

par exemple. Renvoie une erreur si la version de gcc est 3.0.0 ou antérieure.

#if __GNUC__ < 3
#error "This code requires gcc > 3.0.0"
#endif

par exemple. Renvoie une erreur en cas de compilation sur un ordinateur Apple.

#ifdef __APPLE__
#error "Apple products are not supported in this release"
#endif

Macros prédéfinies #

Les macros prédéfinies sont celles que le compilateur définit (contrairement à celles définies par l’utilisateur dans le fichier source). Ces macros ne doivent pas être redéfinies ou indéfinies par l’utilisateur.

Les macros suivantes sont prédéfinies par le standard C++ :

  • __LINE__ contient le numéro de ligne de la ligne sur laquelle cette macro est utilisée et peut être modifié par la directive #line.
  • __FILE__ contient le nom du fichier dans lequel cette macro est utilisée et peut être modifié par la directive #line.
  • __DATE__ contient la date (au format "Mmm jj aaaa") de la compilation du fichier, où Mmm est formaté comme s’il était obtenu par un appel à std::asctime().
  • __TIME__ contient l’heure (au format "hh:mm:ss") de la compilation du fichier.
  • __cplusplus est défini par les compilateurs C++ (conformes) lors de la compilation des fichiers C++. Sa valeur est la version standard avec laquelle le compilateur est entièrement conforme, c’est-à-dire 199711L pour C++98 et C++03, 201103L pour C++11 et 201402L pour la norme C++14 .
  • __STDC_HOSTED__ est défini sur 1 si l’implémentation est hébergée, ou 0 si elle est autonome.
  • __STDCPP_DEFAULT_NEW_ALIGNMENT__ contient un littéral size_t, qui est l’alignement utilisé pour un appel à operator new non conscient de l’alignement.

De plus, les macros suivantes peuvent être prédéfinies par les implémentations et peuvent être présentes ou non :

  • __STDC__ a une signification dépendante de l’implémentation et est généralement défini uniquement lors de la compilation d’un fichier en tant que C, pour signifier une conformité totale à la norme C. (Ou jamais, si le compilateur décide de ne pas prendre en charge cette macro.)
  • __STDC_VERSION__ a une signification qui dépend de l’implémentation et sa valeur est généralement la version C, de la même manière que __cplusplus est la version C++. (Ou n’est même pas défini, si le compilateur décide de ne pas prendre en charge cette macro.)
  • __STDC_MB_MIGHT_NEQ_WC__ est défini sur 1, si les valeurs du codage étroit du jeu de caractères de base peuvent ne pas être égales aux valeurs de leurs homologues larges (par exemple, si (uintmax_t)'x' != (uintmax_t)L' x')
  • __STDC_ISO_10646__ est défini si wchar_t est encodé en Unicode, et se développe en une constante entière sous la forme yyyymmL, indiquant la dernière révision Unicode prise en charge.
  • __STDCPP_STRICT_POINTER_SAFETY__ est défini sur 1, si l’implémentation a la sécurité stricte du pointeur (sinon elle a la sécurité relâchée du pointeur)
  • __STDCPP_THREADS__ est défini sur 1, si le programme peut avoir plus d’un thread d’exécution (applicable à l’implémentation autonome - les implémentations hébergées peuvent toujours avoir plus d’un thread)

Il convient également de mentionner __func__, qui n’est pas une macro, mais une variable locale de fonction prédéfinie. Il contient le nom de la fonction dans laquelle il est utilisé, sous la forme d’un tableau de caractères statique dans un format défini par l’implémentation.

En plus de ces macros prédéfinies standard, les compilateurs peuvent avoir leur propre ensemble de macros prédéfinies. Il faut se référer à la documentation du compilateur pour les connaître. Par exemple.:

  • [gcc][1]
  • [Microsoft Visual C++][2]
  • [clang][3]
  • [Compilateur Intel C++][4]

Certaines des macros ne servent qu’à interroger la prise en charge de certaines fonctionnalités :

#ifdef __cplusplus // if compiled by C++ compiler
extern "C"{ // C code has to be decorated
   // C library header declarations here
}
#endif

D’autres sont très utiles pour le débogage :

bool success = doSomething( /*some arguments*/ );
if( !success ){
    std::cerr << "ERROR: doSomething() failed on line " << __LINE__ - 2
              << " in function " << __func__ << "()"
              << " in file " << __FILE__
              << std::endl;
}

Et d’autres pour un contrôle de version trivial :

int main( int argc, char *argv[] ){
    if( argc == 2 && std::string( argv[1] ) == "-v" ){
        std::cout << "Hello World program\n"
                  << "v 1.1\n" // I have to remember to update this manually
                  << "compiled: " << __DATE__ << ' ' << __TIME__ // this updates automagically
                  << std::endl;
    }
    else{
        std::cout << "Hello World!\n";
    }
}

#pragmatique une fois #

[La plupart des implémentations C++, mais pas toutes] [1] prennent en charge la directive “#pragma once” qui garantit que le fichier n’est inclus qu’une seule fois dans une seule compilation. Il ne fait partie d’aucune norme ISO C++. Par exemple:

// Foo.h
#pragma once

class Foo
{
};

Alors que #pragma once évite certains problèmes associés à [include guards][2], un #pragma - par définition dans les standards - est intrinsèquement un crochet spécifique au compilateur, et sera silencieusement ignoré par les compilateurs qui ne le font pas soutiens le. Les projets qui utilisent #pragma once doivent être modifiés pour être conformes à la norme.

Avec certains compilateurs - en particulier ceux qui utilisent des [en-têtes précompilés][3] - #pragma once peut entraîner une accélération considérable du processus de compilation. De même, certains préprocesseurs accélèrent la compilation en suivant quels en-têtes ont utilisé des gardes inclus. L’avantage net, lorsque #pragma once et les gardes d’inclusion sont utilisés, dépend de l’implémentation et peut être une augmentation ou une diminution des temps de compilation.

#pragma once combiné avec [include guards][2] était la disposition recommandée pour les fichiers d’en-tête lors de l’écriture d’applications basées sur MFC sous Windows, et a été généré par les assistants add class, add dialog, add windows de Visual Studio . Il est donc très courant de les trouver combinés dans les candidats C++ Windows.

[1] : https://en.wikipedia.org/wiki/Pragma_once#Portabilité [2] : https://www.wikiod.com/fr/docs/c%2b%2b/1098/preprocessor/3525/include-guards [3] : https://en.wikipedia.org/wiki/Precompiled_header

Opérateurs de préprocesseur #

L’opérateur # ou l’opérateur de mise en chaîne est utilisé pour convertir un paramètre Macro en un littéral de chaîne. Il ne peut être utilisé qu’avec les macros ayant des arguments.

// preprocessor will convert the parameter x to the string literal x
#define PRINT(x) printf(#x "\n")

PRINT(This line will be converted to string by preprocessor);
// Compiler sees
printf("This line will be converted to string by preprocessor""\n");

Le compilateur concatène deux chaînes et l’argument final printf() sera une chaîne littérale avec un caractère de nouvelle ligne à sa fin.

Le préprocesseur ignorera les espaces avant ou après l’argument macro. Donc, la déclaration d’impression ci-dessous nous donnera le même résultat.

PRINT(   This line will be converted to string by preprocessor );

Si le paramètre du littéral de chaîne nécessite une séquence d’échappement comme avant un guillemet double (), il sera automatiquement inséré par le préprocesseur.

PRINT(This "line" will be converted to "string" by preprocessor); 
// Compiler sees
printf("This \"line\" will be converted to \"string\" by preprocessor""\n");

L’opérateur ## ou l’opérateur de collage de jeton est utilisé pour concaténer deux paramètres ou jetons d’une macro.

// preprocessor will combine the variable and the x
#define PRINT(x) printf("variable" #x " = %d", variable##x)

int variableY = 15;
PRINT(Y);
//compiler sees
printf("variable""Y"" = %d", variableY);

et la sortie finale sera

variableY = 15
