Différents threads essayant d’accéder au même emplacement mémoire participent à une course aux données si au moins une des opérations est une modification (également appelée opération de stockage). Ces courses de données provoquent un comportement indéfini. Pour les éviter, il faut empêcher ces threads d’exécuter simultanément de telles opérations conflictuelles.

Les primitives de synchronisation (mutex, section critique, etc.) peuvent protéger de tels accès. Le modèle de mémoire introduit dans C++11 définit deux nouvelles méthodes portables pour synchroniser l’accès à la mémoire dans un environnement multithread : les opérations atomiques et les clôtures.

Opérations atomiques #

Il est maintenant possible de lire et d’écrire dans un emplacement mémoire donné en utilisant les opérations atomic load et atomic store. Pour plus de commodité, ils sont encapsulés dans la classe de modèle std::atomic<t>. Cette classe encapsule une valeur de type t mais cette fois loads et stores à l’objet sont atomiques.

Le modèle n’est pas disponible pour tous les types. Les types disponibles sont spécifiques à l’implémentation, mais cela inclut généralement la plupart (ou tous) les types intégraux disponibles ainsi que les types de pointeurs. Ainsi, std::atomic<unsigned> et std::atomic<std::vector<foo> *> devraient être disponibles, tandis que std::atomic<std::pair<bool,char>> ne le sera probablement pas.

Les opérations atomiques ont les propriétés suivantes :

  • Toutes les opérations atomiques peuvent être effectuées simultanément à partir de plusieurs threads sans provoquer de comportement indéfini.
  • Une charge atomique verra soit la valeur initiale avec laquelle l’objet atomique a été construit, soit la valeur qui lui est écrite via une opération de stockage atomique.
  • Les magasins atomiques du même objet atomique sont classés de la même manière dans tous les threads. Si un thread a déjà vu la valeur d’une opération atomic store, les opérations atomic load suivantes verront soit la même valeur, soit la valeur stockée par l’opération suivante atomic store.
  • Les opérations atomiques de lecture-modification-écriture permettent au chargement atomique et au stockage atomique de se produire sans autre stockage atomique entre les deux. Par exemple, on peut incrémenter de manière atomique un compteur à partir de plusieurs threads, et aucun incrément ne sera perdu, quel que soit le conflit entre les threads.
  • Les opérations atomiques reçoivent un paramètre facultatif std::memory_order qui définit les propriétés supplémentaires de l’opération concernant les autres emplacements de mémoire.

std::memory_order | Sens —————————-|——————— —- std::memory_order_relaxed | aucune restriction supplémentaire std::memory_order_release → std::memory_order_acquire | si load-acquire voit la valeur stockée par store-release alors stocke sequenced before le store-release se produit avant les chargements séquencés après load learn std::memory_order_consume | comme memory_order_acquire mais uniquement pour les charges dépendantes std::memory_order_acq_rel | combine load-acquire et store-release std::memory_order_seq_cst | cohérence séquentielle

Ces balises d’ordre de la mémoire permettent trois disciplines d’ordre de la mémoire différentes : cohérence séquentielle, relâché et release-acquire avec son frère release-consume.

Cohérence séquentielle #

Si aucun ordre de mémoire n’est spécifié pour une opération atomique, l’ordre par défaut est cohérence séquentielle. Ce mode peut également être explicitement sélectionné en balisant l’opération avec std::memory_order_seq_cst.

Avec cet ordre aucune opération mémoire ne peut croiser l’opération atomique. Toutes les opérations de mémoire séquencées avant l’opération atomique se produisent avant l’opération atomique et l’opération atomique se produit avant toutes les opérations de mémoire séquencées après elle. Ce mode est probablement le plus facile à raisonner, mais il entraîne également la plus grande dégradation des performances. Il empêche également toutes les optimisations du compilateur qui pourraient autrement tenter de réorganiser les opérations au-delà de l’opération atomique.

### Commande détendue L’opposé de la * cohérence séquentielle * est l’ordre de la mémoire * détendu *. Il est sélectionné avec la balise std::memory_order_relaxed. Le fonctionnement atomique détendu n’imposera aucune restriction sur les autres opérations de mémoire. Le seul effet qui reste, c’est que l’opération est elle-même encore atomique.

Commande de libération-acquisition #

Une opération atomic store peut être étiquetée avec std::memory_order_release et une opération atomic load peut être étiquetée avec std::memory_order_acquire. La première opération est appelée (atomic) store-release tandis que la seconde est appelée (atomic) load-acquire.

Lorsque load-acquire voit la valeur écrite par un store-release, ce qui suit se produit : toutes les opérations de stockage séquencées avant le store-release deviennent visibles pour (arriver avant) les opérations de chargement qui sont séquencées après le load -acquérir.

Les opérations atomiques de lecture-modification-écriture peuvent également recevoir la balise cumulative std::memory_order_acq_rel. Cela fait de la partie charge atomique de l’opération une acquisition de charge atomique tandis que la partie stockage atomique devient libération de stockage atomique.

Le compilateur n’est pas autorisé à déplacer les opérations de stockage après une opération atomic store-release. Il est également interdit de déplacer des opérations de chargement avant atomic load-acquire (ou load-consume).

Notez également qu’il n’y a pas de atomic load-release ou atomic store-acquire. Tenter de créer de telles opérations en fait des opérations détendues.

### Ordonnance de libération-consommation Cette combinaison est similaire à release-acquire, mais cette fois la atomic load est taguée avec std::memory_order_consume et devient l’opération (atomic) load-consume. Ce mode est le même que release-acquire à la seule différence que parmi les opérations de chargement séquencées après le load-consume seules celles qui dépendent de la valeur chargée par le load-consume sont ordonnées.

Clôtures #

Les clôtures permettent également d’ordonner les opérations de mémoire entre les threads. Une clôture est soit une clôture de libération, soit une clôture d’acquisition.

Si une clôture de libération se produit avant une clôture d’acquisition, les magasins séquencés avant la clôture de libération sont visibles pour les chargements séquencés après la clôture d’acquisition. Pour garantir que la clôture de libération se produit avant la clôture d’acquisition, on peut utiliser d’autres primitives de synchronisation, y compris des opérations atomiques relâchées.

Besoin d’un modèle de mémoire #

int x, y;
bool ready = false;

void init()
{
  x = 2;
  y = 3;
  ready = true;
}
void use()
{
  if (ready)
    std::cout << x + y;
}

Un thread appelle la fonction init() tandis qu’un autre thread (ou gestionnaire de signal) appelle la fonction use(). On pourrait s’attendre à ce que la fonction use() imprime 5 ou ne fasse rien. Ce n’est pas toujours le cas pour plusieurs raisons :

  • Le CPU peut réorganiser les écritures qui se produisent dans init() afin que le code qui s’exécute réellement ressemble à :

     void init()
 {
   ready = true;
   x = 2;
   y = 3;
 }

  • Le CPU peut réorganiser les lectures qui se produisent dans use() afin que le code réellement exécuté devienne :

     void use()
 {
   int local_x = x;
   int local_y = y;
   if (ready)
     std::cout << local_x + local_y;
 }

  • Un compilateur C++ optimisant peut décider de réorganiser le programme de la même manière.

Une telle réorganisation ne peut pas changer le comportement d’un programme s’exécutant dans un seul thread car un thread ne peut pas entrelacer les appels à init() et use(). D’un autre côté, dans un environnement multi-thread, un thread peut voir une partie des écritures effectuées par l’autre thread où il peut arriver que use() puisse voir ready==true et des ordures dans x ou y ou les deux.

Le modèle de mémoire C++ permet au programmeur de spécifier quelles opérations de réorganisation sont autorisées et lesquelles ne le sont pas, de sorte qu’un programme multithread puisse également se comporter comme prévu. L’exemple ci-dessus peut être réécrit de manière thread-safe comme ceci :

int x, y;
std::atomic<bool> ready{false};

void init()
{
  x = 2;
  y = 3;
  ready.store(true, std::memory_order_release);
}
void use()
{
  if (ready.load(std::memory_order_acquire))
    std::cout << x + y;
}

Ici, init() effectue l’opération atomic store-release. Cela stocke non seulement la valeur true dans ready, mais indique également au compilateur qu’il ne peut pas déplacer cette opération avant les opérations d’écriture qui sont * séquencées avant *.

La fonction use() effectue une opération atomic load-acquire. Il lit la valeur actuelle de ready et interdit également au compilateur de placer les opérations de lecture qui sont * séquencées après * pour * se produire avant * l’ * acquisition de charge atomique *.

Ces opérations atomiques amènent également le compilateur à placer toutes les instructions matérielles nécessaires pour informer le processeur de s’abstenir des réorganisations indésirables.

Étant donné que atomic store-release se trouve au même emplacement mémoire que atomic load-acquire, le modèle de mémoire stipule que si l’opération load-acquire voit la valeur écrite par l’opération store-release, alors toutes les écritures effectuées par le thread de init() avant cette store-release seront visibles pour les charges que le thread de use() exécute après son load-acquire. C’est-à-dire que si use() voit ready==true, alors il est garanti de voir x==2 et y==3.

Notez que le compilateur et le processeur sont toujours autorisés à écrire dans y avant d’écrire dans x, et de même, les lectures de ces variables dans use() peuvent se produire dans n’importe quel ordre.

Exemple de clôture #

L’exemple ci-dessus peut également être mis en œuvre avec des clôtures et des opérations atomiques relâchées :

int x, y;
std::atomic<bool> ready{false};

void init()
{
  x = 2;
  y = 3;
  atomic_thread_fence(std::memory_order_release);
  ready.store(true, std::memory_order_relaxed);
}
void use()
{
  if (ready.load(std::memory_order_relaxed))
  {
    atomic_thread_fence(std::memory_order_acquire);
    std::cout << x + y;
  }
}

Si l’opération de chargement atomique voit la valeur écrite par le magasin atomique, alors le magasin se produit avant le chargement, ainsi que les clôtures : la clôture de libération se produit avant la clôture d’acquisition, ce qui fait les écritures sur x et y qui précèdent la libération fence pour devenir visible à l’instruction std::cout qui suit la clôture d’acquisition.

Une clôture peut être bénéfique si elle peut réduire le nombre total d’opérations d’acquisition, de libération ou d’autres opérations de synchronisation. Par exemple:

void block_and_use()
{
  while (!ready.load(std::memory_order_relaxed))
    ;
  atomic_thread_fence(std::memory_order_acquire);
  std::cout << x + y;
}

La fonction block_and_use() tourne jusqu’à ce que le drapeau ready soit défini à l’aide d’une charge atomique relâchée. Ensuite, une clôture d’acquisition unique est utilisée pour fournir l’ordre de mémoire nécessaire.
