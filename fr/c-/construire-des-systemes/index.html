C++, comme C, a une histoire longue et variée concernant les workflows de compilation et les processus de construction. Aujourd’hui, C++ possède divers systèmes de construction populaires qui sont utilisés pour compiler des programmes, parfois pour plusieurs plates-formes au sein d’un système de construction. Ici, quelques systèmes de construction seront examinés et analysés.

Actuellement, il n’existe aucun système de construction universel ou dominant pour C++ qui soit à la fois populaire et multiplateforme. Cependant, il existe plusieurs systèmes de construction majeurs qui sont attachés aux principales plates-formes/projets, le plus notable étant GNU Make avec le système d’exploitation GNU/Linux et NMAKE avec le système de projet Visual C++/Visual Studio.

De plus, certains environnements de développement intégrés (IDE) incluent également des systèmes de construction spécialisés à utiliser spécifiquement avec l’IDE natif. Certains générateurs de système de génération peuvent générer ces formats de système/projet de génération IDE natifs, tels que CMake pour Eclipse et Microsoft Visual Studio 2012.

Générer un environnement de construction avec CMake #

[CMake][2] génère des environnements de construction pour presque tous les compilateurs ou IDE à partir d’une seule définition de projet. Les exemples suivants montrent comment ajouter un fichier CMake au [code C++ “Hello World” multiplateforme][1].

Les fichiers CMake sont toujours nommés “CMakeLists.txt” et doivent déjà exister dans le répertoire racine de chaque projet (et éventuellement dans les sous-répertoires également.) Un fichier CMakeLists.txt de base ressemble à :

cmake_minimum_required(VERSION 2.4)

project(HelloWorld)

add_executable(HelloWorld main.cpp)

A voir en direct sur Coliru.

Ce fichier indique à CMake le nom du projet, la version du fichier à attendre et les instructions pour générer un exécutable appelé “HelloWorld” qui nécessite main.cpp.

Générez un environnement de construction pour votre compilateur/IDE installé à partir de la ligne de commande :

> cmake .

Construisez l’application avec :

> cmake --build .

Cela génère l’environnement de construction par défaut pour le système, en fonction du système d’exploitation et des outils installés. Gardez le code source exempt de tout artefact de build grâce à l’utilisation de builds “hors source” :

> mkdir build
> cd build
> cmake ..
> cmake --build .

CMake peut également extraire les commandes de base du shell de la plate-forme de l’exemple précédent :

> cmake -E make_directory build
> cmake -E chdir build cmake .. 
> cmake --build build

CMake inclut [générateurs] [3] pour un certain nombre d’outils de construction et d’IDE courants. Pour générer des makefiles pour [nmake][4] de Visual Studio :

> cmake -G "NMake Makefiles" ..
> nmake

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/206/introduction-to-c/774/hello-world [2] : https://cmake.org/ [3] : https://cmake.org/cmake/help/v3.0/manual/cmake-generators.7.html [4] : https://msdn.microsoft.com/en-us/library/dd9y37ha.aspx

Compiler avec GNU make #

Présentation #

Le GNU Make (style make) est un programme dédié à l’automatisation de l’exécution des commandes shell. GNU Make est un programme spécifique qui appartient à la famille Make. Make reste populaire parmi les systèmes d’exploitation de type Unix et POSIX, y compris ceux dérivés du noyau Linux, Mac OS X et BSD.

GNU Make est particulièrement remarquable pour être attaché au projet GNU, qui est attaché au système d’exploitation populaire GNU/Linux. GNU Make a également des versions compatibles fonctionnant sur différentes versions de Windows et Mac OS X. C’est aussi une version très stable avec une signification historique qui reste populaire. C’est pour ces raisons que GNU Make est souvent enseigné parallèlement au C et au C++.

Règles de base #

Pour compiler avec make, créez un Makefile dans votre répertoire de projet. Votre Makefile pourrait être aussi simple que :

Makefile

# Set some variables to use in our command
# First, we set the compiler to be g++
CXX=g++

# Then, we say that we want to compile with g++'s recommended warnings and some extra ones.
CXXFLAGS=-Wall -Wextra -pedantic

# This will be the output file
EXE=app

SRCS=main.cpp

# When you call `make` at the command line, this "target" is called.
# The $(EXE) at the right says that the `all` target depends on the `$(EXE)` target.
# $(EXE) expands to be the content of the EXE variable
# Note: Because this is the first target, it becomes the default target if `make` is called without target
all: $(EXE)

# This is equivalent to saying
# app: $(SRCS)
# $(SRCS) can be separated, which means that this target would depend on each file.
# Note that this target has a "method body": the part indented by a tab (not four spaces).
# When we build this target, make will execute the command, which is:
# g++ -Wall -Wextra -pedantic -o app main.cpp
# I.E. Compile main.cpp with warnings, and output to the file ./app
$(EXE): $(SRCS)
    @$(CXX) $(CXXFLAGS) -o $@ $(SRCS)

# This target should reverse the `all` target. If you call
# make with an argument, like `make clean`, the corresponding target
# gets called.
clean:
    @rm -f $(EXE)

**REMARQUE : Assurez-vous absolument que les indentations sont avec une tabulation, et non avec quatre espaces. Sinon, vous obtiendrez une erreur de ** Makefile:10: *** séparateur manquant. Arrêtez.

Pour l’exécuter depuis la ligne de commande, procédez comme suit :

$ cd ~/Path/to/project
$ make
$ ls
app  main.cpp  Makefile

$ ./app
Hello World!

$ make clean
$ ls
main.cpp  Makefile

# Builds incrémentiels

Lorsque vous commencez à avoir plus de fichiers, make devient plus utile. Que se passe-t-il si vous modifiez a.cpp mais pas b.cpp ? Recompiler b.cpp prendrait plus de temps.

Avec la structure de répertoire suivante :

.
+-- src
|   +-- a.cpp
|   +-- a.hpp
|   +-- b.cpp
|   +-- b.hpp
+-- Makefile

Ce serait un bon Makefile :

Makefile

CXX=g++
CXXFLAGS=-Wall -Wextra -pedantic
EXE=app

SRCS_GLOB=src/*.cpp
SRCS=$(wildcard $(SRCS_GLOB))
OBJS=$(SRCS:.cpp=.o)

all: $(EXE)

$(EXE): $(OBJS)
    @$(CXX) -o $@ $(OBJS)

depend: .depend

.depend: $(SRCS)
    @-rm -f ./.depend
    @$(CXX) $(CXXFLAGS) -MM $^>>./.depend

clean:
    -rm -f $(EXE)
    -rm $(OBJS)
    -rm *~
    -rm .depend

include .depend

Encore une fois, regardez les onglets. Ce nouveau Makefile garantit que vous ne recompilez que les fichiers modifiés, ce qui réduit le temps de compilation.

Documentation #

Pour en savoir plus sur make, consultez la documentation officielle de la Free Software Foundation, [la documentation stackoverflow](https://www.wikiod.com/fr/docs/ /makefile) et réponse élaborée de dmckee sur stackoverflow.

Construire avec SCons #

Vous pouvez créer le code C++ “Hello World” multiplateforme, en utilisant Scons - Un outil de construction de logiciels en langage Python.

Tout d’abord, créez un fichier appelé SConstruct (notez que SCons recherchera un fichier avec ce nom exact par défaut). Pour l’instant, le fichier doit se trouver dans un répertoire juste à côté de votre hello.cpp. Écrivez dans le nouveau fichier la ligne

 Program('hello.cpp')

Maintenant, depuis le terminal, lancez scons. Vous devriez voir quelque chose comme

$ scons
scons: Reading SConscript files ...
scons: done reading SConscript files.
scons: Building targets ...
g++ -o hello.o -c hello.cpp
g++ -o hello hello.o
scons: done building targets.

(bien que les détails varient en fonction de votre système d’exploitation et du compilateur installé).

Les classes Environment et Glob vous aideront à configurer davantage ce qu’il faut construire. Par exemple, le fichier SConstruct

env=Environment(CPPPATH='/usr/include/boost/',
    CPPDEFINES=[],
    LIBS=[],
    SCONS_CXX_STANDARD="c++11"
    )

env.Program('hello', Glob('src/*.cpp'))

construit l’exécutable hello, en utilisant tous les fichiers cpp dans src. Son CPPPATH est /usr/include/boost et il spécifie la norme C++11.

Outils automatiques (GNU) #

Introduction #

Les Autotools sont un groupe de programmes qui créent un système de construction GNU pour un progiciel donné. Il s’agit d’une suite d’outils qui fonctionnent ensemble pour produire diverses ressources de construction, telles qu’un Makefile (à utiliser avec GNU Make). Ainsi, Autotools peut être considéré comme un générateur de système de construction de facto.

Certains programmes Autotools notables incluent:

  • Autoconf
  • Automake (à ne pas confondre avec make)

En général, Autotools est destiné à générer le script compatible Unix et Makefile pour permettre à la commande suivante de construire (ainsi que d’installer) la plupart des packages (dans le cas simple) :

./configure && make && make install

En tant que tel, Autotools entretient également des relations avec certains gestionnaires de packages, en particulier ceux qui sont attachés à des systèmes d’exploitation conformes au(x) standard(s) POSIX.

## Ninjas

Introduction #

Le système de construction Ninja est décrit par son site Web de projet comme [“un petit système de construction axé sur la vitesse.”] [1] Ninja est conçu pour que ses fichiers soient générés par des générateurs de fichiers système de construction et adopte une approche de bas niveau pour construire des systèmes, contrairement aux gestionnaires de systèmes de construction de niveau supérieur comme CMake ou Meson.

Ninja est principalement écrit en C++ et Python, et a été créé comme une alternative au système de construction SCons pour le projet Chromium.

[1] : https://ninja-build.org/

NMAKE (utilitaire de maintenance des programmes Microsoft) #

Introduction #

NMAKE est un utilitaire de ligne de commande développé par Microsoft pour être utilisé principalement avec Microsoft Visual Studio et/ou les outils de ligne de commande Visual C++.

NMAKE est un système de construction qui appartient à la famille Make des systèmes de construction, mais possède certaines fonctionnalités distinctes qui divergent des programmes Make de type Unix, telles que la prise en charge de la syntaxe de chemin de fichier spécifique à Windows (qui diffère elle-même des chemins de fichier de style Unix).
