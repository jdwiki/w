Syntaxe #

  • vide virtuel f();

  • vide virtuel g() = 0 ;

  • // C++11 ou ultérieur :

  • remplacement du vide virtuel h() ;

  • annuler le remplacement de i() ;

  • vide virtuel j() final ;

  • annuler k() final ;

  • Seules les fonctions membres non statiques et non modèles peuvent être “virtuelles”.

  • Si vous utilisez C++11 ou une version ultérieure, il est recommandé d’utiliser override lors de la redéfinition d’une fonction membre virtuelle à partir d’une classe de base.

  • [Les classes de base polymorphes ont souvent des destructeurs virtuels pour permettre à un objet dérivé d’être supprimé via un pointeur vers la classe de base][1]. Si le destructeur n’était pas virtuel, une telle opération conduit à undefined behavior[expr.delete] §5.3. 5/3.

[1] : https://www.wikiod.com/fr/docs/c%2B%2B/1717/polymorphism/20178/polymorphism-destructors

Utilisation du remplacement avec virtual en C++ 11 et versions ultérieures #

Le spécificateur override a une signification particulière dans C++ 11 et plus, s’il est ajouté à la fin de la signature de la fonction. Cela signifie qu’une fonction est

  • Remplacer la fonction présente dans la classe de base &
  • La fonction de la classe Base est “virtuelle”

Il n’y a pas de signification “d’exécution” de ce spécificateur car il est principalement destiné à servir d’indication pour les compilateurs

L’exemple ci-dessous illustrera le changement de comportement avec our sans utiliser de remplacement.

Sans override :

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // Y::f() will not override X::f() because it has a different signature,
    // but the compiler will accept the code (and silently ignore Y::f()).
    virtual void f(int a) { std::cout << a << "\n"; }
};

Avec override :

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // The compiler will alert you to the fact that Y::f() does not
    // actually override anything.
    virtual void f(int a) override { std::cout << a << "\n"; }
};

Notez que override n’est pas un mot-clé, mais un identifiant spécial qui ne peut apparaître que dans les signatures de fonction. Dans tous les autres contextes, “override” peut toujours être utilisé comme identifiant :

void foo() {
    int override = 1; // OK.
    int virtual = 2;  // Compilation error: keywords can't be used as identifiers.
}

Fonctions membres virtuelles vs non virtuelles #

Avec les fonctions de membre virtuel :

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // Specifying virtual again here is optional
    // because it can be inferred from X::f().
    virtual void f() { std::cout << "Y::f()\n"; } 
};

void call(X& a) {
    a.f();
}

int main() {
    X x;
    Y y;
    call(x); // outputs "X::f()"
    call(y); // outputs "Y::f()"
}

Sans fonctions de membre virtuel :

#include <iostream>

struct X {
   void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
   void f() { std::cout << "Y::f()\n"; }
};

void call(X& a) {
    a.f();
}

int main() {
    X x;
    Y y;
    call(x); // outputs "X::f()"
    call(y); // outputs "X::f()"
}

Fonctions virtuelles finales #

C++11 a introduit le spécificateur “final” qui interdit le remplacement de la méthode s’il apparaît dans la signature de la méthode :

class Base {
public:
    virtual void foo() {
        std::cout << "Base::Foo\n";
    }
};

class Derived1 : public Base {
public:
    // Overriding Base::foo
    void foo() final {
        std::cout << "Derived1::Foo\n";
    }
};

class Derived2 : public Derived1 {
public:
    // Compilation error: cannot override final method
    virtual void foo() {
        std::cout << "Derived2::Foo\n";
    }
};

Le spécificateur “final” ne peut être utilisé qu’avec une fonction membre “virtuelle” et ne peut pas être appliqué à des fonctions membres non virtuelles

Comme final, il existe également un appelant spécificateur ‘override’ qui empêche le remplacement des fonctions virtual dans la classe dérivée.

Les spécificateurs “override” et “final” peuvent être combinés pour obtenir l’effet souhaité :

class Derived1 : public Base {
public:
    void foo() final override {
        std::cout << "Derived1::Foo\n";
    }
};

Comportement des fonctions virtuelles dans les constructeurs et les destructeurs #

Le comportement des fonctions virtuelles dans les constructeurs et les destructeurs est souvent source de confusion lors de leur première rencontre.

#include <iostream>
using namespace std;

class base { 
public:
    base() { f("base constructor"); }
    ~base() { f("base destructor"); }

    virtual const char* v() { return "base::v()"; }

    void f(const char* caller) { 
        cout << "When called from " << caller << ", "  << v() << " gets called.\n"; 
    }        
};

class derived : public base {
public:
    derived() { f("derived constructor"); }
    ~derived() { f("derived destructor"); }

    const char* v() override { return "derived::v()"; }

};

int main() {
     derived d;
}

Production:

Lorsqu’il est appelé depuis le constructeur de base, base::v() est appelé. Lorsqu’elle est appelée à partir du constructeur dérivé, la fonction dérivée ::v() est appelée. Lorsqu’elle est appelée depuis le destructeur dérivé, la fonction dérivée ::v() est appelée. Lorsqu’il est appelé depuis le destructeur de base, base::v() est appelé.

Le raisonnement derrière cela est que la classe dérivée peut définir des membres supplémentaires qui ne sont pas encore initialisés (dans le cas du constructeur) ou déjà détruits (dans le cas du destructeur), et appeler ses fonctions membres ne serait pas sûr. Par conséquent, lors de la construction et de la destruction d’objets C++, le type dynamic de *this est considéré comme la classe du constructeur ou du destructeur et non comme une classe plus dérivée.

Exemple:

#include <iostream>
#include <memory>

using namespace std;
class base {
public:
    base()
    {
        std::cout << "foo is " << foo() << std::endl;
    }
    virtual int foo() { return 42; }
};

class derived : public base {
    unique_ptr<int> ptr_;
public:
    derived(int i) : ptr_(new int(i*i)) { }
    // The following cannot be called before derived::derived due to how C++ behaves, 
    // if it was possible... Kaboom!
    int foo() override   { return *ptr_; } 
};

int main() {
    derived d(4);
}

Fonctions virtuelles pures #

Nous pouvons également spécifier qu’une fonction virtuelle est pure virtuelle (abstraite), en ajoutant = 0 à la déclaration. Les classes avec une ou plusieurs fonctions virtuelles pures sont considérées comme abstraites et ne peuvent pas être instanciées ; seules les classes dérivées qui définissent ou héritent des définitions pour toutes les fonctions virtuelles pures peuvent être instanciées.

struct Abstract {
    virtual void f() = 0;
};

struct Concrete {
    void f() override {}
};

Abstract a; // Error.
Concrete c; // Good.

Même si une fonction est spécifiée comme virtuelle pure, on peut lui attribuer une implémentation par défaut. Malgré cela, la fonction sera toujours considérée comme abstraite et les classes dérivées devront la définir avant de pouvoir être instanciées. Dans ce cas, la version de la classe dérivée de la fonction est même autorisée à appeler la version de la classe de base.

struct DefaultAbstract {
    virtual void f() = 0;
};
void DefaultAbstract::f() {}

struct WhyWouldWeDoThis : DefaultAbstract {
    void f() override { DefaultAbstract::f(); }
};

Il y a plusieurs raisons pour lesquelles nous pourrions vouloir faire cela :

  • Si nous voulons créer une classe qui ne peut pas elle-même être instanciée, mais qui n’empêche pas ses classes dérivées d’être instanciées, nous pouvons déclarer le destructeur comme virtuel pur. Étant le destructeur, il doit être défini de toute façon, si nous voulons pouvoir désallouer l’instance. Et [comme le destructeur est très probablement déjà virtuel pour éviter les fuites de mémoire lors d’une utilisation polymorphe] [1], nous n’encourrons pas de perte de performances inutile en déclarant une autre fonction “virtuelle”. Cela peut être utile lors de la création d’interfaces.

      struct Interface {
      virtual ~Interface() = 0;
  };
  Interface::~Interface() = default;

  struct Implementation : Interface {};
  // ~Implementation() is automatically defined by the compiler if not explicitly
  //  specified, meeting the "must be defined before instantiation" requirement.

  • Si la plupart ou toutes les implémentations de la fonction virtuelle pure contiendront du code en double, ce code peut à la place être déplacé vers la version de classe de base, ce qui facilite la maintenance du code.

      class SharedBase {
      State my_state;
      std::unique_ptr<Helper> my_helper;
      // ...

    public:
      virtual void config(const Context& cont) = 0;
      // ...
  };
  /* virtual */ void SharedBase::config(const Context& cont) {
      my_helper = new Helper(my_state, cont.relevant_field);
      do_this();
      and_that();
  }

  class OneImplementation : public SharedBase {
      int i;
      // ...

    public:
      void config(const Context& cont) override;
      // ...
  };
  void OneImplementation::config(const Context& cont) /* override */ {
      my_state = { cont.some_field, cont.another_field, i };
      SharedBase::config(cont);
      my_unique_setup();
  };

  // And so on, for other classes derived from SharedBase.
