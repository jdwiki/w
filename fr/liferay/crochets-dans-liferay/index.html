Crochet JSP #

Les crochets JSP sont un plugin liferay spécial qui permet de modifier le portlet principal jsp-s, disons que vous voulez modifier le portlet de connexion pour afficher Bienvenue dans ma connexion personnalisée!.

La structure minimale d’un Hook Plugin est la suivante :

[project-name]-hook/
 └── docroot/
     ├── WEB-INF/
     │   ├── src/
     │   ├── lib/
     │   ├── liferay-hook.xml
     │   ├── liferay-plugin-package.properties
     │   └── web.xml
     └── META-INF/
         ├── custom_jsps/
         └── MANIFEST.MF

liferay-hook.xml est le fichier qui distingue le type de crochet que vous utilisez, ici vous définissez à l’intérieur de la balise hook le paramètre approprié pour le crochet, pour le crochet JSP :

<?xml version="1.0"?>
<!DOCTYPE hook PUBLIC "-//Liferay//DTD Hook 6.2.0//EN" "http://www.liferay.com/dtd/liferay-hook_6_2_0.dtd">

<hook>
    <custom-jsp-dir>/custom_jsps</custom-jsp-dir>
</hook>

login.jsp se trouve dans Liferay dans /docroot/html/portlet/login/login.jsp, pour en faire un crochet, nous devons ajouter un jsp avec le même nom et chemin dans notre dossier custom_jsps.

Lorsque le crochet est déployé, Liferay recherche dans liferay-hook.xml la valeur custom-jsp-dir et remplace toutes les pages JSP du portail par celles trouvées dans ce répertoire. Les jsp d’origine sont enregistrés avec le nom <nom d'origine>.portal.jsp pour être restaurés en cas d’annulation du déploiement du hook.

Nous pouvons même appeler les JSP d’origine dans le nouveau JSP modifié si nous voulons conserver le code, ce qui le rend adaptable aux mises à jour ou aux mises à niveau de la version sous-jacente de la plate-forme Liferay. Pour ce faire, dans votre JSP personnalisé, utilisez le modèle suivant :

<liferay-util:buffer var="contentHtml">
    <liferay-util:include page="/html/{ JSP file’s path }" />
</liferay-util:buffer>

où { chemin du fichier JSP } dans ce cas sera portlet/login/login.portal.jsp. Cela s’appelle étendre le jsp d’origine.

Ensuite, nous pouvons y ajouter du contenu avec:

<%
contentHtml = StringUtil.add("Stuff I'm adding BEFORE the original content", contentHtml,"\n");    
contentHtml = StringUtil.add(contentHtml,"Stuff I'm adding AFTER the original content","\n");
%>
<%= contentHtml %>

Crochets d’action Struts #

Ce type de crochet peut être utilisé pour remplacer le portail principal (par exemple, c/portal/login) et les actions de struts de portlet (par exemple, /login/forgot_password), ces actions pour le portail Liferay sont spécifiées dans un struts-config.xml dans son dossier WEB-INF. Pour remplacer une action :

  1. dans le fichier liferay-hook.xml de votre plugin hook sous docroot/WEB-INF, ajoutez un élément struts-action dans l’élément hook.

  2. Dans l’élément struts-action, ajoutez struts-action-path qui spécifie le chemin d’action que vous remplacez et struts-action-impl qui spécifie votre classe d’action personnalisée. Cela ressemble à :

    /login/login com.myhook.action.ExampleStrutsPortletAction

  3. Créez une classe d’action de portlet Struts qui étend BaseStrutsPortletAction. Un exemple de cette classe est :

    public class ExampleStrutsPortletAction extends BaseStrutsPortletAction {

         public void processAction(StrutsPortletAction originalStrutsPortletAction,
             PortletConfig portletConfig, ActionRequest actionRequest,
             ActionResponse actionResponse) throws Exception {

         System.out.println("Custom Struts Action");

         originalStrutsPortletAction.processAction(originalStrutsPortletAction,
                 portletConfig, actionRequest, actionResponse);
     }

 public String render(StrutsPortletAction originalStrutsPortletAction,
         PortletConfig portletConfig, RenderRequest renderRequest,
         RenderResponse renderResponse) throws Exception {

     System.out.println("Custom Struts Action");

     return originalStrutsPortletAction.render(null, portletConfig,
             renderRequest, renderResponse);
 }

    }

L’appel de la méthode remplacée, comme originalStrutsPortletAction.processAction, n’est pas obligatoire mais constitue une bonne pratique pour conserver le comportement de l’action inchangé en ce qui concerne le portail Liferay. Ce type de hook peut également être utilisé pour ajouter de nouvelles actions Struts, c’est la même chose que de modifier une action existante, dans ce cas liferay-hook.xml serait :

<struts-action>
    <struts-action-path>/my/custom/path</struts-action-path>
    <struts-action-impl>
    com.myhook.action.ExampleStrutsAction
    </struts-action-impl>
</struts-action>

Bonjour utilisateur “Nom” avec crochets #

Cet exemple montre comment créer un simple “Hello User [nom]” après la connexion. L’exemple est basé sur [l’exécution d’une action personnalisée à l’aide d’un hook][1]

Depuis votre terminal de ligne de commande, accédez au dossier hooks de votre Plugins SDK. Pour créer un projet hook, vous devez exécuter le script de création. Voici le format à suivre pour exécuter le script :

create.[sh|bat] [nom-projet] “[Hook Display Name]”

Sous Linux et Mac OS X, vous devez saisir une commande similaire à celle de cet exemple :

./create.sh Hello-user “Bonjour utilisateur”

Sous Windows, vous devez saisir une commande similaire à celle de cet exemple :

create.bat Hello-user “My Hook”

L’assistant New Project de Liferay IDE et les scripts de création génèrent des projets hook dans le dossier hooks de votre Plugin SDK. Le SDK Plugins ajoute automatiquement “-hook” au nom de votre projet.

Que vous ayez créé votre projet hook depuis l’IDE Liferay ou depuis la ligne de commande, vous vous retrouvez avec la même structure de projet (voir avant).

  • Déterminez l’événement sur lequel vous souhaitez déclencher votre action personnalisée. Regardez dans la documentation portal.properties pour trouver la correspondance propriété événementielle. Astuce : les propriétés d’événement ont .event dans leur nom. Il existe des propriétés d’événement de session, de démarrage, d’arrêt et de portail dans les sections suivantes de la documentation portal.properties : [Session][2] - [Événements de démarrage][3] - [Événements d’arrêt][4] - [Portail Événements][5]

  • Dans votre projet hook, créez une classe Java qui étend le Classe com.liferay.portal.kernel.events.Action. Remplacez la méthode Action.run(HttpServletRequest, HttpServletResponse).

         import com.liferay.portal.kernel.events.Action;
     import javax.servlet.http.HttpServletRequest;
     import javax.servlet.http.HttpServletResponse;
     import com.liferay.portal.model.User;
     import com.liferay.portal.util.PortalUtil;

     public class HelloUser extends Action {
         public void run(HttpServletRequest req, HttpServletResponse res) {
             User user = PortalUtil.getUser(req);
             System.out.println("Hello User "+user.getScreenName());
         }
     }

Important : Si votre action accède à l’objet HttpServletRequest, prolonger com.liferay.portal.kernel.events.Action ; sinon, prolongez com.liferay.portal.struts.SimpleAction.

  • Créez un fichier de propriétés, portal.properties, à l’intérieur de votre crochet project’s docroot/WEB-INF/src folder. Then add the name of the portal event property that corresponds to the event on which you want to perform your action. Specify your action class’ fully qualified name as the property’s value.

     `login.events.post=HelloUser`

    For example, to perform a class’ action just prior to the portal logging in a user, you’d specify the login.events.pre property with your action class as its value. It could look like this property setting.

Important : étant donné que les propriétés du portail telles que login.events.pre acceptent plusieurs valeurs, vous devez ajouter vos valeurs aux valeurs existantes. Vous pouvez modifier à plusieurs reprises les propriétés à partir de crochets supplémentaires.

Ne modifiez qu’une propriété de portail qui accepte une seule valeur d’un plugin crochet unique. Si vous modifiez la valeur d’une propriété à partir de plusieurs plugins, Liferay ne saura pas quelle valeur utiliser.

  • Modifiez votre fichier docroot/WEB-INF/liferay-hook.xml et ajoutez votre nom du fichier de propriétés du portail de hook comme valeur pour le <portal-properties>...</portal-properties> dans votre l’élément <hook>...</hook> du hook. For example, if your hook’s properties file name is portal.properties, you’d specify this element:

    portal.properties

  • Déployez votre crochet, accédez à votre chemin de crochet et entrez ant clean deployvous verrez le .war dans le dossier dist.

Maintenant, si vous vous connectez à liferay, vous verrez dans le journal du serveur un message du type “Hello user Admin”.

Crochet d’écoute du modèle #

Arrière plan #

Les crochets Model Listener sont un type de plugin Liferay qui écoute les événements pris sur un modèle et exécute le code en réponse. Les crochets d’écoute de modèle sont similaires aux crochets d’action Struts personnalisés en ce sens qu’ils répondent à une action effectuée dans le portail. Cependant, alors que les actions Struts répondent à une action entreprise par un utilisateur, un Model Listener répond (avant ou après) un événement impliquant un modèle Liferay.

Différences #

Voici quelques exemples de Struts Actions vs Model Listeners à titre de comparaison.

  • ** Action de jambes de force **
    • User Login
    • Account Creation
    • Extend Session
    • Move Folder
  • Auditeur modèle
    • After folder is created
    • When user information is updated
    • After bookmark is removed
    • Before a role association is made

La meilleure ressource pour apprendre l’architecture de Liferay est à travers leur code source. Tous leurs fichiers source sont situés sur GitHub et en consultant leurs JavaDocs. Vous pouvez voir tous les modèles de portail principaux [sur les JavaDocs][1] et toutes les actions Struts [sur le GitHub.][2]

Exemple #

Dans ce didacticiel, nous allons développer un écouteur modèle qui envoie un e-mail à un utilisateur après la création de son compte. Pour ce faire, nous allons écrire une classe appelée UserModelListener qui étendra BaseModelListener de Liferay. Nous passerons brièvement en revue la création de crochets et couvrirons les modifications nécessaires aux fichiers de configuration suivants

  • portail.propriétés #

  • liferay-hook.xml #

Commencer #

Pour commencer à développer votre crochet Model Listener, vous devez d’abord lancer votre application Liferay IDE ou Liferay Developer Studio.

L’IDE Liferay et le studio de développement Liferay sont des environnements de développement Eclipse personnalisés. Ils sont étonnamment similaires et un ensemble de directions devrait être suffisant pour les deux environnements.

Dans votre environnement de développement, exécutez les étapes suivantes.

  1. Dans le coin supérieur gauche, cliquez sur Fichier
  2. Passez votre souris sur Nouveau
  3. Cliquez sur Projet de plug-in Liferay

Vous allez faire apparaître cette fenêtre.

[![entrez la description de l’image ici][3]][3]

Veuillez entrer les informations comme indiqué ci-dessus

  • Nom du projet : Utilisateur-Modèle-Écouteur
  • Sélectionnez Utiliser l’emplacement par défaut
  • Type de construction : Fourmi
  • Type de plug-in : Hook

Assurez-vous que votre projet se trouve dans votre répertoire Liferays Plugins SDK Hook. Vous devrez sélectionner votre SDK et votre Runtime en conséquence.

Dans votre perspective Explorateur de packages, vous verrez la structure de répertoires suivante.

Développement de l’auditeur #

Maintenant que vous avez créé votre crochet, vous devez créer votre classe UserModelListener personnalisée. Cette classe étendra la classe BaseModelListener de Liferay.

La classe BaseModelListener de Liferay est une classe abstraite qui implémente l’interface ModelListener. Vous ne souhaitez pas implémenter directement l’interface ModelListener car cela vous obligera à remplacer toutes ses méthodes.

Les méthodes suivantes vous sont fournies par l’interface ModelListener via la classe abstraite BaseModelListener.

  • onAfterAddAssociation
  • surAprèsCréer
  • onAfterRemove
  • onAfterRemoveAssociation
  • après la mise à jour
  • onBeforeAddAssociation
  • onBeforeCreate
  • onBeforeRemove
  • onBeforeRemoveAssociation
  • onBeforeUpdate

Créez votre classe UserModelListener dans le répertoire suivant. Pour créer la classe via l’interface graphique, exécutez simplement les commandes suivantes

  • Cliquez sur Fichier dans le coin supérieur gauche

  • Passez votre souris sur Nouveau

  • Cliquez sur Classe

    docroot/ WEB-INF/ src/

Entrez les informations ci-dessous

[![entrez la description de l’image ici][5]][5]

Dans votre classe UserModelListener, collez le code suivant

package com.example.hook;

import com.liferay.mail.service.MailServiceUtil;
import com.liferay.portal.ModelListenerException;
import com.liferay.portal.kernel.mail.MailMessage;
import com.liferay.portal.kernel.util.Validator;
import com.liferay.portal.model.BaseModelListener;
import com.liferay.portal.model.User;


import javax.mail.internet.InternetAddress;

public class UserModelListener extends BaseModelListener<User> {
    private User user = null;

    @Override
    public void onAfterCreate(User user) throws ModelListenerException {
    this.user = user;
    
    if(isValidEmail()) {
        sendEmail("admin@example.com", user.getEmailAddress(), "Welcome!", "Your account is created!");
    } 
    }

    private boolean isValidEmail() {
    return Validator.isNotNull(user.getEmailAddress()) && Validator.isAddress(user.getEmailAddress());
    }
    
    private void sendEmail(String from, String to, String subject, String body) {
    try {
        MailServiceUtil.sendEmail(new MailMessage(new InternetAddress(from), new InternetAddress(to), subject, body, false));
    } catch (Exception e) {
        System.err.print("E-Mail spawned by User Model Listener failed to " + user.getFullName() + " with message " + e.getMessage());
    }
    }
}

Configuration des propriétés #

Afin de configurer l’association entre notre écouteur personnalisé et notre modèle, nous devons faire quelques derniers ajustements. Tout d’abord, créez un nouveau fichier portal.properties dans le répertoire suivant.

docroot/
     WEB-INF/
           src/
            + portal.properties

Une seule ligne doit être ajoutée au fichier.

value.object.listener.com.liferay.portal.model.User = com.example.hook.UserModelListener

Nous pouvons généraliser cela en disant que pour tout Model Listener que nous créons, la valeur des propriétés associées doit être sous la forme de

value.object.listener.fully.qualified.model.name = fully.qualified.listener.name

En d’autres termes, si nous écrivions une classe CustomerDLFolderModelListener, empaquetée dans com.example.code, pour le modèle DLFolder nous aurions la propriété suivante

value.object.listener.com.liferay.portal.model.DLFolder = com.example.code.CustomerDLFolderModelListener

Enfin, localisez votre fichier liferay-hook.xml. Dans la vue Source, écrivez ce qui suit.

<?xml version="1.0"?>
<!DOCTYPE hook PUBLIC "-//Liferay//DTD Hook 6.2.0//EN" "http://www.liferay.com/dtd/liferay-hook_6_2_0.dtd">

<hook>
    <portal-properties>portal.properties</portal-properties>
</hook>

Explication #

  1. La première ligne est un prologue facultatif qui spécifie la version du document et (dans certains cas) le jeu de caractères.
  2. La ligne 2 est une DocType Definition (DTD) formelle qui définit explicitement quels éléments et attributs sont valides
  3. Les lignes 3 et 5 se composent de l’élément parent **Hook ** (l’un des éléments valides pris en charge par cette DTD)
  4. La ligne 4 remplace et étend le fichier portal.properties dans ${liferay.home}

Pour voir quels autres éléments peuvent être utilisés dans ce fichier XML, vous pouvez référencer l’URL dans la [DocType Definition][6]. Ceci est standard pour tous les fichiers XML et SGML avec une DTD. Un autre exemple de fichier Liferay XML avec une DTD est service.xml (implémentation ORM de Liferay basée sur Hibernate).

Construire et déployer #

Construire et déployer des hooks est un processus simple. Le développement du plugin Liferay prend en charge l’automatisation de la construction et des dépendances avec

  • Fourmi
  • Lierre
  • Maven -Gradle

Dans notre exemple, nous avons utilisé Ant pour l’automatisation de la construction. Le fichier build.xml contient les commandes de génération (appelées targets dans Ant). Pour créer votre hook, exécutez simplement les commandes suivantes.

  1. Localisez votre fichier build.xml
  2. Dans votre IDE, faites glisser le fichier build.xml dans la perspective Ant
  3. Développez le fichier et exécutez la cible all

  3. Une fois en surbrillance, cliquez sur le bouton Ajouter et cliquez sur OK
  4. Cliquez sur le bouton Jouer dans la perspective Serveur
