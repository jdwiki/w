Utilisation du paramètre de rendu public #

Cette approche a été introduite dans JSR 286.

Dans JSR 168, les paramètres de rendu définis dans processAction d’un portlet n’étaient disponibles que dans ce portlet. Avec la fonctionnalité Public Render Parameters, les paramètres de rendu définis dans processAction d’un portlet seront également disponibles dans le rendu des autres portlets. Afin de configurer cela, pour tous les portlets qui le supportent :

Ajoutez la balise <supported-public-render-parameter>, juste avant que la balise de portlet ne se termine par portlet.xml

<security-role-ref>
    <role-name>user</role-name>
</security-role-ref>
<supported-public-render-parameter>{param-name}</supported-public-render-parameter>
</portlet>

Ajoutez la balise <public-render-parameter> juste avant la fin de la balise <portlet-app>

  <public-render-parameter>
    <identifier>{param-name}</identifier>
    <qname xmlns:x="localhost">x:{param-name}</qname>
  </public-render-parameter>
</portlet-app>

Dans la méthode processAction, la valeur du paramètre doit être définie dans la réponse

res.setRenderParameter({param-name},{param-value});

Une fois que nous en avons fini avec la configuration pour tous les portlets requis, après avoir exécuté la phase d’action du portlet concerné, le paramètre doit être disponible en phase de rendu pour tous les portlets de support sur la page, qu’ils fassent partie de la même application ou d’une application différente (war ).

Utilisation de la session Portlet #

Il s’agit d’une approche qui existe depuis JSR 168. Elle nous permet de partager des attributs à l’aide d’une session de portlet. Une session de portlet peut avoir différents types de portée :

  1. Étendue du portlet (attributs disponibles uniquement dans le portlet)
  2. Portée de l’application (attributs disponibles dans toute l’application [guerre])

Pour utiliser cette approche, nous n’avons pas besoin de saisir d’entrées dans la configuration du portlet, car la session de portlet est facilement disponible dans la demande de portlet :

PortletSession session = renderRequest.getPortletSession();
session.setAttribute("attribute-name","attribute-value", PortletSession.APPLICATION_SCOPE);

ou

PortletSession session = renderRequest.getPortletSession();
session.setAttribute("attribute-name","attribute-value", PortletSession.PORTLET_SCOPE);

L’attribut ne peut être récupéré qu’à partir de la portée respective. Comme pour l’attribut défini dans la portée du portlet, nous devons le récupérer à l’aide de

PortletSession session = renderRequest.getPortletSession();
String attributeValue = (String) session.getAttribute("attribute-name", PortletSession.PORTLET_SCOPE);

La principale limitation de cette approche est le manque de partage entre les autres portlets, en dehors de la portée de l’application.

    <private-session-attributes>false</private-session-attributes>
    <header-portlet-css>/css/main.css</header-portlet-css>
    <footer-portlet-javascript>/js/main.js</footer-portlet-javascript>
    <css-class-wrapper>{portlet-name}</css-class-wrapper>
</portlet>

pour tous les portlets, où les attributs sont définis et récupérés.

Utilisation de la fonctionnalité d’événementiel #

Le mécanisme d’événement est une version étendue du paramètre de rendu public, avec une fonctionnalité supplémentaire pour transmettre des objets personnalisés à d’autres portlets, mais avec une surcharge de la phase d’événement.

Pour y parvenir, ce mécanisme consiste à

  1. Portlet de l’éditeur
  2. Portlet de processeur (consommateur), où les deux peuvent faire partie de différentes applications de portlet.

Commencer avec,

Ajoutez la balise <supported-publishing-event> au portlet de l’éditeur dans portlet.xml

    <security-role-ref>
        <role-name>user</role-name>
    </security-role-ref>
    <supported-publishing-event>
         <qname xmlns:x="http:sun.com/events">x:Employee</qname>
    </supported-publishing-event>
  </portlet>

Ajoutez la balise <supported-processing-event> au portlet du processeur dans portlet.xml

<security-role-ref>
        <role-name>user</role-name>
    </security-role-ref>
    <supported-processing-event>
        <qname xmlns:x="http:sun.com/events">x:Employee</qname>
     </supported-processing-event>
</portlet>

Ajoutez la balise <event-definition> aux deux portlets, en définissant le nom de l’événement et en saisissant portlet.xml

<event-definition>   
  <qname xmlns:x="http:sun.com/events">x:Employee</qname>
  <value-type>com.sun.portal.portlet.users.Employee</value-type>
</event-definition>
   </portlet-app>

Ensuite, nous devons créer une classe pour le type d’événement (en cas de type personnalisé)

public class Employee implements Serializable {
  public Employee() {
  }
  private String name;
  private int userId; 

  public String getName() {
    return name;
  }

  public void setName(String name) {
    this.name = name;
  }

  public int getUserId() {
    return userId;
  }
  public void setUserId(int id)
  {
    this.userId = id;
  }

}

Maintenant, dans le portlet de l’éditeur, l’événement doit être publié dans la phase d’action

    QName qname = new QName("http:sun.com/events" , "Employee");
    Employee emp = new Employee();
    emp.setName("Rahul");
    emp.setUserId(4567);
    res.setEvent(qname, emp);

Une fois que nous avons publié l’événement, il doit être traité par le portlet de l’éditeur dans la phase d’événement.

La phase d’événement a été introduite dans JSR 286 et est exécutée avant la phase de rendu du portlet, le cas échéant

@ProcessEvent(qname = "{http:sun.com/events}Employee")
public void processEvent(EventRequest request, EventResponse response) {

    Event event = request.getEvent();
    if(event.getName().equals("Employee")){
      Employee payload = (Employee)event.getValue();
      response.setRenderParameter("EmpName",
      payload.getName());
    }

}

qui peut ensuite être récupéré à partir du paramètre de rendu via une requête de rendu.
