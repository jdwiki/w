Avant d’utiliser OpenCL, il faut configurer son code pour l’utiliser. Cette rubrique se concentre sur la façon de faire fonctionner opencl dans votre projet et d’exécuter un noyau de base. Les exemples sont basés sur le wrapper C# OpenCL.NET mais comme le wrapper n’ajoute aucune abstraction à OpenCL, le code s’exécutera probablement avec très peu de modifications sur C/C++ également.

Les appels en C# peuvent ressembler à ceci : ‘Cl.GetPlatformIDs’. Pour l’API OpenCL de style C, vous appelleriez ‘clGetPlatformIDs’ et pour le style C++ un ‘cl::GetPlatformIDs’

  • NVidia, AMD et Intel ont des implémentations légèrement différentes d’OpenCL mais les différences connues sont (d’après mon expérience) limitées aux exigences de parenthèse et aux conversions implicites. Parfois, NVidia fait planter votre noyau en essayant de déterminer la surcharge correcte pour une méthode. Dans ce cas, il est utile de proposer un casting explicite pour aider le GPU. Le problème a été observé pour les noyaux compilés à l’exécution.

  • Pour avoir plus d’informations sur les appels utilisés dans cette rubrique, il suffit de googler ‘OpenCL’ suivi du nom de la fonction. Le groupe Khronos dispose d’une documentation complète on all parameters and datatypes available on their website.

Initialisation de l’appareil cible #

Les noyaux OpenCL peuvent être exécutés sur le GPU ou sur le CPU. Cela permet des solutions de secours, où le client peut avoir un système très obsolète. Le programmeur peut également choisir de limiter ses fonctionnalités au CPU ou au GPU.

Pour commencer à utiliser OpenCL, vous aurez besoin d’un ‘Contexte’ et d’un ‘Périphérique’. Les deux sont des structures définies par l’API OpenCL (également appelées cl::Context ou clContext & ~Device) et définissent le processeur cible utilisé.

Pour obtenir votre appareil et votre contexte, vous devez interroger une liste de plates-formes disponibles, chacune pouvant héberger plusieurs appareils. Une plate-forme représente votre GPU et votre CPU physiques, tandis qu’un appareil peut distinguer davantage les unités de calcul contenues. Pour les GPU, la plupart des plates-formes n’auront qu’un seul appareil. Mais un CPU peut offrir un GPU intégré supplémentaire en plus de ses capacités CPU.

Le contexte gère la mémoire, les files de commandes, les différents noyaux et programmes. Un contexte peut soit être limité à un seul appareil, mais aussi référencer plusieurs appareils.

Une note rapide sur l’API avant de commencer à coder : presque chaque appel à OpenCL vous donne une valeur d’erreur, soit comme valeur de retour, soit via une valeur de référence (pointeur en C). Commençons maintenant.

ErrorCode err;
var platforms = Cl.GetPlatformIDs(out err);
if(!CheckError(err, "Cl.GetPlatformIDs")) return;
foreach (var platform in platforms) {
    foreach (var device in Cl.GetDeviceIDs(platform, DeviceType.Gpu, out err)) {
        if(!CheckError(err, "Cl.GetDeviceIDs")) continue;
        [...]
    }
}

Cet extrait de code interroge tous les périphériques GPU disponibles sur le système. Vous pouvez maintenant les ajouter à une liste ou commencer votre contexte directement avec le premier match. La fonction ‘CheckError(…)’ est un utilitaire simple, qui vérifie si le code d’erreur a la valeur de succès ou une autre et peut vous proposer une journalisation. Il est recommandé d’utiliser une fonction ou une macro séparée, car vous l’appellerez souvent.

ErrorCode est juste une énumération sur le type de données cl_int pour C#, C/C++ peut comparer la valeur int avec des constantes d’erreur prédéfinies comme indiqué ici : https://www.khronos.org/registry/OpenCL/sdk/1.0/docs/man/xhtml/errors.html

Vous voudrez peut-être également vérifier si l’appareil prend en charge toutes les fonctionnalités nécessaires, sinon vos noyaux pourraient se bloquer lors de l’exécution. Vous pouvez interroger la capacité d’un appareil avec

Cl.GetDeviceInfo(_device, DeviceInfo.ImageSupport, out err)

Cet exemple demande au périphérique s’il peut exécuter des fonctions d’image. Pour la prochaine et dernière étape, nous devons construire notre contexte à partir des appareils collectés.

_context = Cl.CreateContext(null, 1, new[] { _device }, ContextNotify, IntPtr.Zero, out err);

Il se passe des trucs ici. Pour les utilisateurs de C/C++, IntPtr est une adresse de pointeur en C#. Je vais me concentrer sur les parties importantes ici.

  • Le deuxième paramètre définit le nombre d’appareils que vous souhaitez utiliser
  • Le troisième paramètre est un tableau de ces périphériques (ou un pointeur en C/C++)
  • Et le troisième paramètre est un pointeur de fonction pour une fonction de rappel. Cette fonction sera utilisée chaque fois que des erreurs se produiront dans le contexte.

Pour une utilisation ultérieure, vous devrez conserver vos appareils utilisés et le contexte quelque part.

Lorsque vous avez terminé toutes vos interactions OpenCL, vous devrez à nouveau libérer le contexte avec

Cl.ReleaseContext(_context);

Compilation de votre noyau #

Les noyaux peuvent être compilés au moment de l’exécution sur le périphérique cible. Pour ce faire, vous avez besoin

  • le code source du noyau
  • le périphérique cible sur lequel compiler
  • un contexte construit avec l’appareil cible

Une mise à jour rapide de la terminologie : Un programme contient une collection de noyaux. Vous pouvez considérer un programme comme un fichier source C/C++/C# complet, tandis que les noyaux sont les différentes fonctions membres de ce fichier.

Vous devez d’abord créer un programme à partir de votre code source.

var program = Cl.CreateProgramWithSource(_context, 1, new[] { source }, null, out err);

Vous pouvez combiner plusieurs fichiers source en un seul programme et les compiler ensemble, ce qui vous permet d’avoir des noyaux dans différents fichiers et de les compiler ensemble en une seule fois.

À l’étape suivante, vous devrez compiler le programme sur votre appareil cible.

err = Cl.BuildProgram(program, 1, new[] { _device }, string.Empty, null, IntPtr.Zero);

Maintenant, voici une petite mise en garde : le code d’erreur vous indique uniquement si l’appel de la fonction lui-même a réussi, mais pas si votre code s’est réellement compilé. Pour vérifier cela, nous devons demander des informations supplémentaires

BuildStatus status;
status = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Status, out err).CastTo<BuildStatus>();
if (status != BuildStatus.Success) {
    var log = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Log, out err);
}

Les utilisateurs de C/C++ peuvent ignorer le cast à la fin et simplement comparer l’entier renvoyé avec la constante correspondante.

Le premier appel vérifie si notre construction a réellement réussi. Sinon, nous pouvons récupérer un journal et voir exactement où les choses se sont mal passées. Voir les remarques pour certains pièges courants concernant différentes plates-formes.

Une fois le programme construit, vous devez extraire vos différents noyaux du programme compilé. Pour ce faire, vous créez vos noyaux avec

_kernel = Cl.CreateKernel(_program, kernel, out err);

où ‘kernel’ est une chaîne du nom du noyau. Lorsque vous avez terminé avec votre noyau, vous devez le publier avec

Cl.ReleaseKernel(_kernel);

Création d’une file d’attente de commandes #

Pour lancer une opération sur vos appareils, vous aurez besoin d’une file d’attente de commandes pour chaque appareil. La file d’attente garde une trace des différents appels que vous avez faits à l’appareil cible et les maintient dans l’ordre. La plupart des commandes peuvent également être exécutées en mode bloquant ou non bloquant.

La création d’une file d’attente est assez simple :

_queue = Cl.CreateCommandQueue(_context, _device, CommandQueueProperties.None, out err);

L’interaction de base avec votre file d’attente de commandes consiste à mettre en file d’attente différentes opérations que vous souhaitez effectuer, par ex. copier des données vers et depuis votre appareil et lancer un noyau.

Lorsque vous avez fini d’utiliser la file d’attente de commandes, vous devez libérer la file d’attente avec un appel à

Cl.ReleaseCommandQueue(_queue);

Exécution du noyau #

Alors maintenant, passons aux choses sérieuses, en exécutant vos noyaux sur le périphérique parallèle. Veuillez lire les bases du matériel pour bien comprendre la répartition du noyau.

Vous devrez d’abord définir les arguments du noyau avant d’appeler réellement le noyau. Cela se fait via

err = Cl.SetKernelArg(_kernel, $argumentIndex, $argument);

Si vous ne définissez pas tous les arguments avant de lancer le noyau, le noyau échouera.

Avant de lancer réellement notre noyau, nous devons calculer la “taille de travail globale” et la “taille de travail locale”.

la taille de travail globale est le nombre total de threads qui seront lancés sur votre GPU. La taille de travail locale est le nombre de threads à l’intérieur de chaque bloc de threads. La taille de travail locale peut être omise si le noyau n’a pas besoin d’exigences particulières. Mais si la taille de travail locale est donnée, la taille de travail globale doit être un multiple de la taille de travail locale.

Les tailles de travail peuvent être unidimensionnelles, bidimensionnelles ou tridimensionnelles. Le choix du nombre de dimensions que vous souhaitez dépend entièrement de vous et vous pouvez choisir ce qui convient le mieux à votre algorithme.

Maintenant que nous avons décidé de nos tailles de travail, nous pouvons appeler le noyau.

Event clevent;
err = Cl.EnqueueNDRangeKernel(_queue, _kernel, $dimensions, null, $globalWorkSize, $localWorkSize, 0, null, out clevent);

Les $dimensions définissent le nombre de dimensions souhaité, $globalWorkSize est un tableau de taille $dimensions avec la taille de travail globale et la même chose pour $localWorkSize. Le dernier argument vous donne un objet qui représente votre noyau actuellement exécuté.
