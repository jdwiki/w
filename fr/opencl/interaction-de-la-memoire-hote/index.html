Cette rubrique met en évidence différentes manières de placer des données à un endroit où votre appareil peut y accéder.

Lecture d’un tableau #

Pour lire un tableau du périphérique vers l’hôte, on appelle

clEnqueueReadBuffer($queue, $memobj, $blocking, $offset, $size, $target, 0, null, null);

La $queue est la CommandQueue qui a été utilisée pour allouer la mémoire sur le périphérique. Le $memobj contient l’adresse de la mémoire de l’appareil, $offset et $size définissent plus précisément d’où et combien de données sont copiées. Le $target est un pointeur vers la mémoire hôte dans laquelle les données seront stockées. Le $target doit être alloué et avoir une taille appropriée.

Lire une texture #

Lire une image revient presque à lire un tableau. La seule différence étant que la taille et le décalage doivent être tridimensionnels.

clEnqueueReadImage($queue, $memobj, $blocking, $offset, $size, $stride, $slice_pitch, $target, 0, null, null);

Le $stride définit le nombre d’octets d’une ligne. Normalement, il ne s’agit que de largeur * (octets par pixel), mais quelqu’un pourrait vouloir changer cela pour aligner les données sur les banques de mémoire. Il en va de même pour $slice_pitch, sauf que cette valeur est pour la troisième dimension.

Écrire une texture 2D #

Pour copier une texture sur l’appareil, deux étapes sont nécessaires

  1. Allouer la mémoire sur l’appareil

  2. Copiez l’image sur l’appareil

    _mem = clCreateImage2D($context, $mem_flags, $image_format, $width, $height, $stride, $source, &err);

Les $mem_flags définissent comment la mémoire est allouée. Il peut être en lecture seule, en écriture seule ou les deux. De plus, vous pouvez définir où et comment la mémoire est allouée. $width, $height et $stride sont assez explicites.

Si vos mem_flags copient les données, vous avez terminé. Si vous souhaitez le faire manuellement ultérieurement, vous devrez appeler une autre fonction lorsque vous serez prêt.

err = clEnqueueWriteImage($queue, _mem, $blocking, $offset, $size, $stride, $slice_pitch, $source, 0, null, null);

Le $offset et le $size définissent la région de l’image que vous souhaitez copier dans la mémoire cible. Le $stride définit le nombre d’octets d’une ligne. Normalement, il ne s’agit que de largeur * (octets par pixel), mais quelqu’un pourrait vouloir changer cela pour aligner les données sur les banques de mémoire. Il en va de même pour $slice_pitch, sauf que cette valeur est pour la troisième dimension. $stride et $slice_pitch doivent correspondre à vos données d’entrée.

Indicateurs de mémoire #

Lors de l’allocation de mémoire, vous avez la possibilité de choisir entre différents modes :

  • Mémoire en lecture seule
  • Écrire uniquement la mémoire
  • Mémoire de lecture/écriture

La mémoire en lecture seule est allouée dans la région de mémoire __constant, tandis que les deux autres sont allouées dans la région __global normale.

En plus de l’accessibilité, vous pouvez définir où votre mémoire est allouée.

  • Non précisé: Votre mémoire est allouée sur la mémoire de l’appareil comme vous vous en doutez. Le pointeur $source peut être défini sur null.
  • CL_MEM_USE_HOST_PTR : Cela indique à l’appareil que les données sont dans la RAM système et ne doivent pas être déplacées. Au lieu de cela, les données sont manipulées directement dans le bélier.
  • CL_MEM_COPY_HOST_PTR : indique à l’appareil de copier toutes les valeurs à l’adresse donnée dans la mémoire de l’appareil ou, à l’aide de CL_MEM_ALLOC_HOST_PTR, dans une région de mémoire séparée dans la RAM système.
  • CL_MEM_ALLOC_HOST_PTR : indique à l’appareil d’allouer de l’espace sur la RAM système. S’il est utilisé comme seul paramètre, le pointeur $source peut être défini sur null.

En termes de vitesse, l’accès à la mémoire globale de l’appareil est le plus rapide. Mais vous devez également l’appeler deux fois pour copier des données. L’utilisation du pointeur Host est la plus lente, tandis que Alloc_host_ptr offre une vitesse plus élevée.

Lors de l’utilisation de use_host_ptr, l’appareil fait exactement cela : il utilise vos données dans la RAM système, qui est bien sûr paginée par le système d’exploitation. Ainsi, chaque appel mémoire doit passer par le processeur pour gérer les défauts de page potentiels. Lorsque les données sont disponibles, le processeur les copie dans la mémoire épinglée et les transmet au contrôleur DMA en utilisant de précieux cycles d’horloge du processeur. Au contraire, alloc_host_ptr alloue de la mémoire épinglée dans la RAM système. Cette mémoire est placée en dehors du mécanisme de pageswap et a donc une disponibilité garantie. Par conséquent, l’appareil peut ignorer complètement le processeur lors de l’accès à la RAM système et utiliser DMA pour copier rapidement les données sur l’appareil.

Écrire un tableau #

L’écriture d’un tableau comprend deux étapes :

  1. Allocation de la mémoire
  2. Copie des données

Pour allouer la mémoire, un simple appel à

_mem = clCreateBuffer($queue, $mem_flags, $size, $host_ptr, &err);

est assez. Si vous avez décidé de copier le pointeur hôte via les mem_flags, vous avez terminé. Sinon, vous pouvez copier les données quand vous le souhaitez avec

err = clEnqueueWriteBuffer($queue, _mem, $blocking, $offset, $size, $source, 0, null, null);
