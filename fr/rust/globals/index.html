Syntaxe #

  • IDENTIFIANT const : type = constexpr ;

  • identificateur [mut] statique : type = expr ;

  • paresseux_statique ! { IDENTIFIANT de référence statique : type = expr ; }

  • Les valeurs const sont toujours en ligne et n’ont pas d’adresse en mémoire.

  • Les valeurs “statiques” ne sont jamais en ligne et ont une instance avec une adresse fixe.

  • Les valeurs static mut ne sont pas sécurisées en mémoire et ne sont donc accessibles que dans un bloc unsafe.

  • Parfois, l’utilisation de variables mutables statiques globales dans du code multithread peut être dangereuse, alors pensez à utiliser std::sync::Mutex ou d’autres alternatives

  • Les objets lazy_static sont immuables, ne sont initialisés qu’une seule fois, sont partagés entre tous les threads et sont directement accessibles (aucun type de wrapper n’est impliqué). En revanche, les objets thread_local sont censés être modifiables, sont initialisés une fois pour chaque thread et les accès sont indirects (impliquant le type de wrapper LocalKey<T>)

Const. #

Le mot-clé const déclare une liaison constante globale.

const DEADBEEF: u64 = 0xDEADBEEF;

fn main() {
    println("{:X}", DEADBEEF);
}

Cela produit

DEADBEEF

Statique #

Le mot-clé static déclare une liaison statique globale, qui peut être modifiable.

static HELLO_WORLD: &'static str = "Hello, world!";

fn main() {
    println("{}", HELLO_WORLD);
}

Cela produit

Hello, world!

paresseux_statique ! #

Utilisez le crate lazy_static pour créer des variables globales immuables qui sont initialisées au moment de l’exécution. Nous utilisons HashMap comme démonstration.

Dans Cargo.toml :

[dependencies]
lazy_static = "0.1.*"

Dans main.rs :

#[macro_use]
extern crate lazy_static;

lazy_static! {
    static ref HASHMAP: HashMap<u32, &'static str> = {
        let mut m = HashMap::new();
        m.insert(0, "hello");
        m.insert(1, ",");
        m.insert(2, " ");
        m.insert(3, "world");
        m
    };
    static ref COUNT: usize = HASHMAP.len();
}

fn main() {
    // We dereference COUNT because it's type is &usize
    println!("The map has {} entries.", *COUNT);

    // Here we don't dereference with * because of Deref coercions
    println!("The entry for `0` is \"{}\".", HASHMAP.get(&0).unwrap());
}

Objets locaux au thread #

Un objet local de thread est initialisé lors de sa première utilisation dans un thread. Et comme son nom l’indique, chaque thread recevra une nouvelle copie indépendante des autres threads.

use std::cell::RefCell;
use std::thread;

thread_local! {
    static FOO: RefCell<f32> = RefCell::new(1.0);
}

// When this macro expands, `FOO` gets type `thread::LocalKey<RefCell<f32>>`.
//
// Side note: One of its private member is a pointer to a function which is
// responsible for returning the thread-local object. Having all its members
// `Sync` [0], `LocalKey` is also implicitly `Sync`.
//
// [0]: As of writing this, `LocalKey` just has 2 function-pointers as members

fn main() {
    FOO.with(|foo| {
        // `foo` is of type `&RefCell<f64>`
        *foo.borrow_mut() = 3.0;
    });

    thread::spawn(move|| {
        // Note that static objects do not move (`FOO` is the same everywhere),
        // but the `foo` you get inside the closure will of course be different.
        FOO.with(|foo| {
            println!("inner: {}", *foo.borrow());
        });
    }).join().unwrap();

    FOO.with(|foo| {
        println!("main: {}", *foo.borrow());
    });
}

Les sorties:

inner: 1
main: 3

Mut statique sécurisé avec mut_static #

Les éléments globaux mutables (appelés static mut, soulignant la contradiction inhérente à leur utilisation) ne sont pas sûrs car il est difficile pour le compilateur de s’assurer qu’ils sont utilisés de manière appropriée.

Cependant, l’introduction de verrous mutuellement exclusifs autour des données permet des variables globales mutables sécurisées en mémoire. Cela ne signifie PAS qu’ils sont logiquement sûrs !

#[macro_use]
extern crate lazy_static;
extern crate mut_static;

use mut_static::MutStatic;

pub struct MyStruct { value: usize }

impl MyStruct {
    pub fn new(v: usize) -> Self{
        MyStruct { value: v }
    }
    pub fn getvalue(&self) -> usize { self.value }
    pub fn setvalue(&mut self, v: usize) { self.value = v }
}

lazy_static! {
    static ref MY_GLOBAL_STATE: MutStatic<MyStruct> = MutStatic::new();
}

fn main() {
    // Here, I call .set on the MutStatic to put data inside it.
    // This can fail.
    MY_GLOBAL_STATE.set(MyStruct::new(0)).unwrap();
    {
        // Using the global state immutably is easy...
        println!("Before mut: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
    {
         // Using it mutably is too...
         let mut mut_handle = MY_GLOBAL_STATE.write().unwrap();
         mut_handle.setvalue(3);
         println!("Changed value to 3.");
    } 
    {
        // As long as there's a scope change we can get the 
        // immutable version again...
        println!("After mut: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
    {
        // But beware! Anything can change global state!
        foo();
        println!("After foo: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
 
}

// Note that foo takes no parameters
fn foo() {
    let val;
    {
        val = MY_GLOBAL_STATE.read().unwrap().getvalue();
    }
    {
        let mut mut_handle = 
            MY_GLOBAL_STATE.write().unwrap();
        mut_handle.setvalue(val + 1);
    }
}

Ce code produit la sortie :

Before mut: 0
Changed value to 3.
After mut: 3
After foo: 4

Ce n’est pas quelque chose qui devrait arriver normalement dans Rust. foo() n’a pas pris de référence mutable à quoi que ce soit, donc il n’aurait rien dû muter, et pourtant il l’a fait. Cela peut conduire à des erreurs de logique très difficiles à déboguer.

D’un autre côté, c’est parfois exactement ce que vous voulez. Par exemple, de nombreux moteurs de jeu nécessitent un cache global d’images et d’autres ressources qui est chargé paresseusement (ou utilise une autre stratégie de chargement complexe) - MutStatic est parfait à cette fin.
