Serializable et serialVersionUid dans Kotlin #

Pour créer le serialVersionUID pour une classe dans Kotlin, vous disposez de quelques options impliquant toutes l’ajout d’un membre à l’objet compagnon de la classe.

Le bytecode le plus concis provient d’un private const val qui deviendra une variable statique privée sur la classe contenante, dans ce cas MySpecialCase :

class MySpecialCase : Serializable {
    companion object {
        private const val serialVersionUID: Long = 123
    }
}

Vous pouvez également utiliser ces formulaires, chacun ayant pour effet secondaire d’avoir des méthodes getter/setter qui ne sont pas nécessaires pour la sérialisation…

class MySpecialCase : Serializable {
    companion object {
        private val serialVersionUID: Long = 123
    }
}

Cela crée le champ statique mais crée également un getter ainsi getSerialVersionUID sur l’objet compagnon qui n’est pas nécessaire.

class MySpecialCase : Serializable {
    companion object {
        @JvmStatic private val serialVersionUID: Long = 123
    }
}

Cela crée le champ statique mais crée également un getter statique ainsi que “getSerialVersionUID” sur la classe contenante “MySpecialCase”, ce qui n’est pas nécessaire.

Mais tous fonctionnent comme une méthode d’ajout de serialVersionUID à une classe Serializable.

Déléguer à une classe sans le fournir dans le constructeur public #

Supposons que vous souhaitiez [déléguer à une classe][1] mais que vous ne souhaitiez pas fournir la classe déléguée à dans le paramètre constructeur. Au lieu de cela, vous voulez le construire en privé, ce qui rend l’appelant du constructeur inconscient. Au début, cela peut sembler impossible car la délégation de classe permet de déléguer uniquement aux paramètres du constructeur. Cependant, il existe un moyen de le faire, comme indiqué dans [cette réponse][2] :

class MyTable private constructor(table: Table<Int, Int, Int>) : Table<Int, Int, Int> by table {

    constructor() : this(TreeBasedTable.create()) // or a different type of table if desired

}

Avec cela, vous pouvez simplement appeler le constructeur de MyTable comme ceci : MyTable(). La Table<Int, Int, Int> à laquelle les délégués MyTable seront créés en privé. L’appelant du constructeur n’en sait rien.

Cet exemple est basé sur [cette question SO][3].

[1] : https://kotlinlang.org/docs/reference/delegation.html#class-delegation [2] : http://stackoverflow.com/a/37598292/986533 [3] : http://stackoverflow.com/q/37593738/986533

Utilisez let ou également pour simplifier le travail avec des objets nullables #

let dans Kotlin crée une liaison locale à partir de l’objet sur lequel il a été appelé. Exemple:

val str = "foo"
str.let {
    println(it) // it
}

Cela affichera "foo" et renverra Unit.

La différence entre let et also est que vous pouvez renvoyer n’importe quelle valeur d’un bloc let. also d’autre part renverra toujours Unit.

Maintenant, pourquoi est-ce utile, demandez-vous? Parce que si vous appelez une méthode qui peut renvoyer null et que vous voulez exécuter du code uniquement lorsque cette valeur de retour n’est pas null, vous pouvez utiliser let ou also comme ceci :

val str: String? = someFun()
str?.let {
    println(it)
}

Ce morceau de code n’exécutera le bloc let que lorsque str n’est pas null. Notez l’opérateur de sécurité null (?).

Filtrer une liste #

val list = listOf(1,2,3,4,5,6)

//filter out even numbers

val even = list.filter { it % 2 == 0 }

println(even) //returns [2,4]

Méthodes fluides dans Kotlin #

Les méthodes Fluent dans Kotlin peuvent être les mêmes que Java :

fun doSomething() {
   someOtherAction()
   return this
}

Mais vous pouvez aussi les rendre plus fonctionnels en créant une fonction d’extension telle que :

fun <T: Any> T.fluently(func: ()->Unit): T {
    func()
    return this
}

Ce qui permet alors des fonctions plus évidemment fluides :

fun doSomething() {
   return fluently { someOtherAction() }
}

Utilisez apply pour initialiser des objets ou pour réaliser un chaînage de méthodes #

La documentation de “apply” indique ce qui suit :

appelle le bloc fonction spécifié avec la valeur this comme récepteur et renvoie la valeur this.

Bien que le kdoc ne soit pas si utile, “apply” est en effet une fonction utile. En termes simples, apply établit une portée dans laquelle this est lié à l’objet sur lequel vous avez appelé apply. Cela vous permet d’économiser du code lorsque vous devez appeler plusieurs méthodes sur un objet que vous renverrez ensuite. Exemple:

File(dir).apply { mkdirs() }

C’est la même chose que d’écrire ceci :

fun makeDir(String path): File {
    val result = new File(path)
    result.mkdirs()
    return result
}

Création de DTO (POJO/POCO) #

Les classes de données dans kotlin sont des classes créées pour ne rien faire d’autre que contenir des données. Ces classes sont marquées comme data :

data class User(var firstname: String, var lastname: String, var age: Int)

Le code ci-dessus crée une classe “User” avec les éléments suivants générés automatiquement :

  • Getters et Setters pour toutes les propriétés (getters uniquement pour vals)
  • égal()
  • code de hachage()
  • toString()
  • copier()
  • componentN() (où N est la propriété correspondante dans l’ordre de déclaration)

Comme pour une fonction, des valeurs par défaut peuvent également être spécifiées :

data class User(var firstname: String = "Joe", var lastname: String = "Bloggs", var age: Int = 20)

Plus de détails peuvent être trouvés ici [Classes de données][1].

[1] : https://kotlinlang.org/docs/reference/data-classes.html
