L’instruction COBOL CALL permet d’accéder aux routines de bibliothèque compilées.

[![entrez la description de l’image ici][1]][1]

[1] : http://i.stack.imgur.com/USCTk.png

Instruction CALL #

COBOL peut utiliser une liaison statique pour l’instruction suivante. GnuCOBOL utilise la liaison dynamique par défaut pour tous les symboles externes connus au moment de la compilation, même lorsque le symbole est un littéral :

CALL "subprogram" USING a b c *> run a (possibly static linked) sub program
                              *> passing three fields

CALL some-prog USING a b c    *> some-prog is a PIC X item and can be changed
                              *> at run-time to do a dynamic lookup

Cette instruction force la résolution d’édition des liens au moment de la compilation. (Non standard, extension de syntaxe) :

CALL STATIC "subprogram" USING a b c

Les champs en COBOL peuvent être passés BY REFERENCE (la valeur par défaut, jusqu’à ce qu’ils soient remplacés - les remplacements sont sticky dans un ordre de gauche à droite), BY CONTENT (une copie est passée PAR REFERENCE), ou dans certains cas directement `BY VALEUR :

CALL "calculation" USING BY REFERENCE a BY VALUE b BY CONTENT c RETURNING d
    ON EXCEPTION DISPLAY 'No linkage to "calculation"' UPON SYSERR
END-CALL

COBOL est conçu pour être un langage “PAR RÉFÉRENCE”, donc l’utilisation de “PAR VALEUR” peut présenter des problèmes. Par exemple, les nombres littéraux n’ont pas de type explicite et la spécification COBOL n’a pas de règles de promotion de type explicites. Par conséquent, les développeurs doivent se soucier de la configuration du cadre d’appel avec “PAR VALEUR” de littéraux.

Voir http://open-cobol.sourceforge.net/faq/index.html#call pour plus de détails.

HEURE DE DORMIR #

CALL est également un moyen d’étendre les fonctionnalités COBOL et de permettre la réutilisation du code. Il peut également donner accès à la fonctionnalité “système”.

Cet exemple illustre les manières de fournir la fonctionnalité “veille” aux COBOL IBM Mainframe. Gardez à l’esprit que l’obligation de le faire est rare dans la mesure où généralement, lorsque quelqu’un pense qu’il a besoin de “dormir” pour une raison quelconque, c’est la mauvaise chose à faire.

ILBOWAT0 appartient à l’ancienne ère d’exécution spécifique à COBOL sur les mainframes. BXP1SLP et BXP4SLP sont des routines Unix System Services (USS) qui peuvent être utilisées par n’importe quel langage. En fait, ce sont des requêtes Unix “sleep”.

L’IBM Mainframe Runtime (Language Environment (LE)) actuel permet la communication inter-langues, et les services CEE3DLY LE sont illustrés dans un autre exemple, https://www.wikiod.com/fr/cobol/instruction-appel using-z-os-language-environment-thread-delay-service#t=201703112326027373052.

ILBOWAT0 existe depuis très longtemps (peut-être plus de 40 ans), et vous pouvez encore le croiser. Son utilisation doit être remplacée par CEE3DLY ou BXP1SLP, selon ce qui est le plus approprié pour l’exigence particulière.

Parfois, vous devez mettre un programme en veille ou mettre un travail en veille pendant un certain temps (après une étape FTP ou NDM), qui sont généralement exécutés en tant que travaux séparés, et vous devez mettre en veille/boucle à la recherche des ensembles de données résultants.

Voici un petit programme COBOL mignon pour effectuer ladite tâche, appelant les programmes de veille COBOL disponibles dans OS/VS et peut-être d’autres environnements d’exploitation mainframe hérités et actuels.

       IDENTIFICATION DIVISION.
       PROGRAM-ID.  SLEEPYTM.
       ENVIRONMENT DIVISION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  WAIT-PARM.
           05  WAIT-TIME            PIC S9(8) COMP VALUE 90.
           05  WAIT-RESPONSE        PIC S9(8) COMP VALUE 0.
           05  WAIT-PROGRAM-24BIT   PIC  X(8)      VALUE 'ILBOWAT0'.
           05  WAIT-PROGRAM-31BIT   PIC  X(8)      VALUE 'BPX1SLP '.
           05  WAIT-PROGRAM-64BIT   PIC  X(8)      VALUE 'BPX4SLP '.

       PROCEDURE DIVISION.
       GENESIS.
           DISPLAY 'START CALLING WAIT PROGRAM'
           CALL WAIT-PROGRAM-24BIT USING WAIT-TIME WAIT-RESPONSE
           DISPLAY 'END   CALLING WAIT PROGRAM'
           GOBACK
PERIOD     .

façon microfocus #

Pour Microfocus, il utilise l’API “SleepEx”. Par exemple;

environment division.            
special-names.                   
    call-convention 74 is winAPI.
         :
         :
01  wSleep-time              pic 9(8) comp-5.
01  wSleep-ok                pic 9(8) comp-5.
         :
         :
move 10000 to wSleep-time  *>10seconds
call winAPI "SleepEx" using by value wSleep-time
                        by value 0 size 4   
              returning wSleep-ok 
end-call.

Utilisation du service de retard de thread de z/OS Language Environment #

Vous pouvez appeler le service CEE3DLY en mode 24-31- ou 64-bit pour retarder une tâche à la seconde près. Il s’agit d’une sauvegarde CICS et ne fera que retarder le thread.

Un exemple:

    IDENTIFICATION DIVISION.
    PROGRAM-ID.  SLEEPYTM.
    ENVIRONMENT DIVISION.
    DATA DIVISION.
    WORKING-STORAGE SECTION.
    01  WAIT-PARM.
      05  WAIT-SECS            PIC S9(8) COMP VALUE 90.
      05  WAIT-FC              PIC X(12).

    PROCEDURE DIVISION.

      CALL CEE3DLY USING WAIT-SECS WAIT-FC
      
      GOBACK.

Vous pouvez voir plus de détails ici :

[Services appelables d’IBM Language Environment - Veille] [1]
