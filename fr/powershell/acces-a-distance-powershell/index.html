Activation de la communication à distance PowerShell #

La communication à distance PowerShell doit d’abord être activée sur le serveur auquel vous souhaitez vous connecter à distance.

Enable-PSRemoting -Force

Cette commande effectue les opérations suivantes :

  • Exécute l’applet de commande Set-WSManQuickConfig, qui effectue les tâches suivantes :
  • Démarre le service WinRM.
  • Définit le type de démarrage du service WinRM sur Automatique.
  • Crée un écouteur pour accepter les requêtes sur n’importe quelle adresse IP, s’il n’en existe pas déjà.
  • Active une exception de pare-feu pour les communications WS-Management.
  • Enregistre les configurations de session Microsoft.PowerShell et Microsoft.PowerShell.Workflow, si elles ne sont pas déjà enregistrées.
  • Enregistre la configuration de session Microsoft.PowerShell32 sur les ordinateurs 64 bits, si elle n’est pas déjà enregistrée.
  • Active toutes les configurations de session.
  • Modifie le descripteur de sécurité de toutes les configurations de session pour permettre l’accès à distance.
  • Redémarre le service WinRM pour appliquer les modifications précédentes.

Uniquement pour les environnements sans domaine #

Pour les serveurs d’un domaine AD, l’authentification PS Remoting s’effectue via Kerberos (‘Default’) ou NTLM (‘Negotiate’). Si vous souhaitez autoriser l’accès à distance à un serveur n’appartenant pas à un domaine, vous avez deux options.

Configurez la communication WSMan via HTTPS (qui nécessite la génération d’un certificat) ou activez l’authentification de base qui envoie vos informations d’identification sur le fil codé en base64 (c’est fondamentalement la même chose que le texte brut, alors soyez prudent avec cela).

Dans les deux cas, vous devrez ajouter les systèmes distants à votre liste d’hôtes de confiance WSMan.

Activation de l’authentification de base #

Set-Item WSMan:\localhost\Service\AllowUnencrypted $true

Ensuite, sur l’ordinateur à partir duquel vous souhaitez vous connecter, vous devez lui dire de faire confiance à l’ordinateur auquel vous vous connectez au.

Set-Item WSMan:\localhost\Client\TrustedHosts '192.168.1.1,192.168.1.2'

Set-Item WSMan:\localhost\Client\TrustedHosts *.contoso.com

Set-Item WSMan:\localhost\Client\TrustedHosts *

Important : Vous devez dire à votre client de faire confiance à l’ordinateur auquel vous souhaitez vous connecter (par exemple, si vous vous connectez via IP, il doit faire confiance à l’adresse IP et non au nom d’hôte)

Connexion à un serveur distant via PowerShell #

À l’aide des informations d’identification de votre ordinateur local :

Enter-PSSession 192.168.1.1

Demande d’informations d’identification sur l’ordinateur distant

Enter-PSSession 192.168.1.1 -Credential $(Get-Credential)

Exécuter des commandes sur un ordinateur distant #

Une fois la communication à distance Powershell activée (Enable-PSRemoting), vous pouvez exécuter des commandes sur l’ordinateur distant comme ceci :

Invoke-Command -ComputerName "RemoteComputerName" -ScriptBlock {
    Write host "Remote Computer Name: $ENV:ComputerName"
}

La méthode ci-dessus crée une session temporaire et la ferme juste après la fin de la commande ou du bloc de script.

Pour laisser la session ouverte et y exécuter une autre commande ultérieurement, vous devez d’abord créer une session à distance :

$Session = New-PSSession -ComputerName "RemoteComputerName"

Vous pouvez ensuite utiliser cette session chaque fois que vous invoquez des commandes sur l’ordinateur distant :

Invoke-Command -Session $Session -ScriptBlock {
    Write host "Remote Computer Name: $ENV:ComputerName"
}

Invoke-Command -Session $Session -ScriptBlock {
    Get-Date
}

Si vous avez besoin d’utiliser différents Credentials, vous pouvez les ajouter avec le paramètre -Credential :

$Cred = Get-Credential
Invoke-Command -Session $Session -Credential $Cred -ScriptBlock {...}

Utilisation des arguments #

Pour utiliser des arguments comme paramètres pour le bloc de script distant, on peut soit utiliser le paramètre ArgumentList de Invoke-Command, soit utiliser la syntaxe $Using:.

Utilisation de ArgumentList avec des paramètres sans nom (c’est-à-dire dans l’ordre dans lequel ils sont passés au scriptblock):

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ArgumentList $servicesToShow,$fileName -ScriptBlock {
    Write-Host "Calling script block remotely with $($Args.Count)"
    Get-Service -Name $args[0]
    Remove-Item -Path $args[1] -ErrorAction SilentlyContinue -Force
}

Utilisation de ArgumentList avec des paramètres nommés :

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ArgumentList $servicesToShow,$fileName -ScriptBlock {
    Param($serviceToShowInRemoteSession,$fileToDelete)

    Write-Host "Calling script block remotely with $($Args.Count)"
    Get-Service -Name $serviceToShowInRemoteSession
    Remove-Item -Path $fileToDelete -ErrorAction SilentlyContinue -Force
}

Utilisation de la syntaxe $Using: :

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ScriptBlock {
    Get-Service $Using:servicesToShow
    Remove-Item -Path $fileName -ErrorAction SilentlyContinue -Force
}

Une bonne pratique pour nettoyer automatiquement les PSSessions #

Lorsqu’une session distante est créée via l’applet de commande New-PSsession, la PSSession persiste jusqu’à la fin de la session PowerShell en cours. Cela signifie que, par défaut, la PSSession et toutes les ressources associées continueront à être utilisées jusqu’à la fin de la session PowerShell en cours.

Plusieurs PSSessions actives peuvent devenir une pression sur les ressources, en particulier pour les scripts de longue durée ou interconnectés qui créent des centaines de PSSessions dans une seule session PowerShell.

Il est recommandé de supprimer explicitement chaque PSSession une fois qu’il a fini d’être utilisé. [1]

Le modèle de code suivant utilise try-catch-finally afin d’obtenir ce qui précède, en combinant la gestion des erreurs avec un moyen sécurisé de s’assurer que toutes les PSSessions créées sont supprimées lorsqu’elles ont fini d’être utilisées :

try
{
    $session = New-PSsession -Computername "RemoteMachineName"
    Invoke-Command -Session $session -ScriptBlock {write-host "This is running on $ENV:ComputerName"}
}
catch
{
    Write-Output "ERROR: $_"
}
finally
{
    if ($session)
    {
        Remove-PSSession $session
    }
}

Références: [1] https://msdn.microsoft.com/en-us/powershell/reference/5.1/microsoft.powershell.core/new-pssession
