Travailler avec le pipeline PowerShell

PowerShell introduit un modèle de pipeline d’objets, qui vous permet d’envoyer des objets entiers à travers le pipeline pour consommer des commandlets ou (au moins) la sortie. Contrairement au pipeline classique basé sur des chaînes, les informations contenues dans les objets canalisés n’ont pas besoin d’être sur des positions spécifiques. Les commandlets peuvent déclarer interagir avec les objets du pipeline en entrée, tandis que les valeurs de retour sont automatiquement envoyées au pipeline.

Syntaxe #

  • COMMENCER Le premier bloc. Exécuté une fois au début. L’entrée de pipeline ici est $null, car elle n’a pas été définie.
  • PROCESSUS Le deuxième bloc. Exécuté pour chaque élément du pipeline. Le paramètre pipeline est égal à l’élément actuellement traité.
  • FIN Dernier bloc. Exécuté une fois à la fin. Le paramètre pipeline est égal au dernier élément de l’entrée, car il n’a pas été modifié depuis sa définition.

Dans la plupart des cas, l’entrée du pipeline sera un tableau d’objets. Bien que le comportement du bloc PROCESS{} puisse sembler similaire à celui du bloc foreach{}, ignorer un élément du tableau nécessite un processus différent.

Si, comme dans foreach{}, vous utilisiez continue dans le bloc PROCESS{}, cela casserait le pipeline, en sautant toutes les instructions suivantes, y compris le bloc END{}. Au lieu de cela, utilisez return - cela ne fera que terminer le bloc PROCESS{} pour l’élément actuel et passer au suivant.

Dans certains cas, il est nécessaire de sortir le résultat de fonctions avec un codage différent. L’encodage de la sortie des CmdLets est contrôlé par la variable $OutputEncoding. Lorsque la sortie est destinée à être placée dans un pipeline vers des applications natives, il peut être judicieux de corriger l’encodage pour qu’il corresponde à la cible $OutputEncoding = [Console] ::OutputEncoding

Références supplémentaires :

Article de blog avec plus d’informations sur $OutputEncoding https://blogs.msdn.microsoft.com/powershell/2006/12/11/outputencoding-to-the-rescue/

Fonctions d’écriture avec cycle de vie avancé #

Cet exemple montre comment une fonction peut accepter une entrée en pipeline et itérer efficacement.

Notez que les structures begin et end de la fonction sont facultatives lors du pipelining, mais que process est requis lors de l’utilisation de ValueFromPipeline ou ValueFromPipelineByPropertyName.

function Write-FromPipeline{
    [CmdletBinding()]
    param(
        [Parameter(ValueFromPipeline)]
        $myInput
    )
    begin {
        Write-Verbose -Message "Beginning Write-FromPipeline" 
    }
    process {
        Write-Output -InputObject $myInput
    }
    end {
        Write-Verbose -Message "Ending Write-FromPipeline"
    }
}

$foo = 'hello','world',1,2,3

$foo | Write-FromPipeline -Verbose

Production:

VERBOSE: Beginning Write-FromPipeline
hello
world
1
2
3
VERBOSE: Ending Write-FromPipeline

Prise en charge de base du pipeline dans les fonctions #

Ceci est un exemple de fonction avec le support le plus simple possible pour le pipelining. Toute fonction prenant en charge le pipeline doit avoir au moins un paramètre avec le ParameterAttribute ValueFromPipeline ou ValueFromPipelineByPropertyName défini, comme indiqué ci-dessous.

function Write-FromPipeline {
    param(
        [Parameter(ValueFromPipeline)]  # This sets the ParameterAttribute
        [String]$Input
    )
    Write-Host $Input
}

$foo = 'Hello World!'

$foo | Write-FromPipeline

Production:

“Bonjour le monde !”

Remarque : dans PowerShell 3.0 et versions ultérieures, les valeurs par défaut pour ParameterAttributes sont prises en charge. Dans les versions antérieures, vous devez spécifier ValueFromPipeline=$true.

Concept de travail du pipeline #

Dans une série de pipelines, chaque fonction s’exécute parallèlement aux autres, comme des threads parallèles. Le premier objet traité est transmis au pipeline suivant et le traitement suivant est immédiatement exécuté dans un autre thread. Cela explique le gain de vitesse élevé par rapport au standard “ForEach”

@( bigFile_1, bigFile_2, ..., bigFile_n) | Copy-File | Encrypt-File | Get-Md5
  1. étape - copier le premier fichier (dans le fil Copy-file)
  2. étape - copier second file (in Copy-file Thread) and simultaneously Encrypt the first (in Encrypt-File)
  3. étape - copier le troisième fichier (dans Copy-file Thread) et chiffrer simultanément le deuxième fichier (dans Encrypt-File) et simultanément get-Md5 du premier (dans Get-Md5)
