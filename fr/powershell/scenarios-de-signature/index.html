La signature d’un script rendra vos scripts conformes à toutes les politiques d’exécution dans PowerShell et garantira l’intégrité d’un script. Les scripts signés ne pourront pas s’exécuter s’ils ont été modifiés après avoir été signés.

La signature de scripts nécessite un certificat de signature de code. Recommandations :

  • Scripts/tests personnels (non partagés) : certificat d’une autorité de certification de confiance (interne ou tierce) OU un certificat auto-signé.
  • Organisation interne partagée : certificat d’une autorité de certification de confiance (interne ou tierce)
  • Organisation externe partagée : Certificat d’une autorité de certification tierce de confiance

En savoir plus sur [about_Signing @ TechNet][1]

Politiques d’exécution #

PowerShell a des politiques d’exécution configurables qui contrôlent les conditions requises pour l’exécution d’un script ou d’une configuration. Une stratégie d’exécution peut être définie pour plusieurs étendues ; ordinateur, utilisateur actuel et processus actuel. Les politiques d’exécution peuvent facilement être contournées et ne sont pas conçues pour restreindre les utilisateurs, mais plutôt pour les protéger contre la violation involontaire des politiques de signature.

Les politiques disponibles sont :

Paramètre Description
Restreint Aucun script autorisé
AllSigned Tous les scripts doivent être signés
RemoteSigned Tous les scripts locaux sont autorisés ; uniquement les scripts distants signés
Illimité Aucune exigence. Tous les scripts sont autorisés, mais avertiront avant d’exécuter les scripts téléchargés sur Internet
Bypass Tous les scripts sont autorisés et aucun avertissement n’est affiché
Undefined Supprimer la stratégie d’exécution actuelle pour la portée actuelle. Utilise la stratégie parent. Si toutes les politiques ne sont pas définies, restreint sera utilisé.

Vous pouvez modifier les politiques d’exécution actuelles à l’aide de la cmdlet Set-ExecutionPolicy, de la stratégie de groupe ou du paramètre -ExecutionPolicy lors du lancement d’un processus powershell.exe.

En savoir plus sur [about_Execution_Policies @ TechNet][2]

[1] : https://technet.microsoft.com/en-us/library/hh847874(v=wps.640).aspx [2] : https://technet.microsoft.com/en-us/library/hh847748(v=wps.620).aspx

Contournement de la politique d’exécution pour un seul script #

Souvent, vous devrez peut-être exécuter un script non signé qui n’est pas conforme à la stratégie d’exécution actuelle. Un moyen simple de le faire est de contourner la politique d’exécution pour ce processus unique. Exemple:

powershell.exe -ExecutionPolicy Bypass -File C:\MyUnsignedScript.ps1

Ou vous pouvez utiliser le raccourci :

powershell -ep Bypass C:\MyUnsignedScript.ps1

Autres politiques d’exécution : #

Politique La description
Tous signés Seuls les scripts signés par un éditeur de confiance peuvent être exécutés.
Contourner Pas de restrictions; tous les scripts Windows PowerShell peuvent être exécutés.
Par défaut Normalement RemoteSigned, mais est contrôlé via ActiveDirectory
RemoteSigned Les scripts téléchargés doivent être signés par un éditeur de confiance avant de pouvoir être exécutés.
Restreint Aucun script ne peut être exécuté. Windows PowerShell ne peut être utilisé qu’en mode interactif.
Non défini N / A
Illimité* Similaire à “contourner”

Unrestricted* Mise en garde : Si vous exécutez un script non signé qui a été téléchargé depuis Internet, vous êtes invité à fournir une autorisation avant son exécution.

Plus d’informations disponibles ici.

Signer un script #

La signature d’un script s’effectue à l’aide de l’applet de commande Set-AuthenticodeSignature et d’un certificat de signature de code.

#Get the first available personal code-signing certificate for the logged on user
$cert = @(Get-ChildItem -Path Cert:\CurrentUser\My -CodeSigningCert)[0]
    
#Sign script using certificate
Set-AuthenticodeSignature -Certificate $cert -FilePath c:\MyScript.ps1

Vous pouvez également lire un certificat à partir d’un fichier .pfx en utilisant :

$cert = Get-PfxCertificate -FilePath "C:\MyCodeSigningCert.pfx"

Le script sera valide jusqu’à l’expiration du certificat. Si vous utilisez un serveur d’horodatage lors de la signature, le script restera valide après l’expiration du certificat. Il est également utile d’ajouter la chaîne de confiance pour le certificat (y compris l’autorité racine) pour aider la plupart des ordinateurs à faire confiance au certificat utilisé pour signer le script.

Set-AuthenticodeSignature -Certificate $cert -FilePath c:\MyScript.ps1 -IncludeChain All -TimeStampServer "http://timestamp.verisign.com/scripts/timstamp.dll"

Il est recommandé d’utiliser un serveur d’horodatage d’un fournisseur de certificats de confiance comme Verisign, Comodo, Thawte, etc.

Modification de la politique d’exécution à l’aide de Set-ExecutionPolicy #

Pour modifier la stratégie d’exécution pour la portée par défaut (LocalMachine), utilisez :

Set-ExecutionPolicy AllSigned

Pour modifier la stratégie d’un champ d’application spécifique, utilisez :

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy AllSigned

Vous pouvez supprimer les invites en ajoutant le commutateur “-Force”.

Obtenir la politique d’exécution actuelle #

Obtenir la stratégie d’exécution effective pour la session en cours :

PS> Get-ExecutionPolicy
RemoteSigned

Répertoriez toutes les stratégies d’exécution effectives pour la session en cours :

PS> Get-ExecutionPolicy -List

        Scope ExecutionPolicy
        ----- ---------------
MachinePolicy       Undefined
   UserPolicy       Undefined
      Process       Undefined
  CurrentUser       Undefined
 LocalMachine    RemoteSigned

Répertorier la politique d’exécution pour une portée spécifique, ex. traiter:

PS> Get-ExecutionPolicy -Scope Process
Undefined

Obtenir la signature d’un script signé #

Obtenez des informations sur la signature Authenticode à partir d’un script signé à l’aide de l’applet de commande Get-AuthenticodeSignature :

Get-AuthenticodeSignature .\MyScript.ps1 | Format-List *

Création d’un certificat de signature de code auto-signé pour les tests #

Lors de la signature de scripts personnels ou lors du test de signature de code, il peut être utile de créer un certificat de signature de code auto-signé.

À partir de PowerShell 5.0, vous pouvez générer un certificat de signature de code auto-signé à l’aide de l’applet de commande New-SelfSignedCertificate :

New-SelfSignedCertificate -FriendlyName "StackOverflow Example Code Signing" -CertStoreLocation Cert:\CurrentUser\My -Subject "SO User" -Type CodeSigningCert

Dans les versions antérieures, vous pouvez créer un certificat auto-signé à l’aide de l’outil “makecert.exe” disponible dans le SDK .NET Framework et le SDK Windows.

Un certificat auto-signé ne sera approuvé que par les ordinateurs qui ont installé le certificat. Pour les scripts qui seront partagés, un certificat d’une autorité de certification de confiance (interne ou tiers de confiance) est recommandé.
