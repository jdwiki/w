Validation de vos modifications locales dans le référentiel #

Pour publier les modifications que vous avez apportées dans votre copie de travail, exécutez la commande [svn commit][1].

IMPORTANT : Passez en revue vos modifications avant de les valider ! Utilisez [svn status][2] et [svn diff][3] pour vérifier les modifications. Assurez-vous également que vous êtes dans le bon chemin avant d’effectuer un commit. Si vous avez mis à jour de nombreux fichiers dans différents répertoires, vous devez être au niveau approprié pour les inclure tous sous votre emplacement.

Voici un exemple de la commande commit :

svn commit -m "My Descriptive Log Message"

Alternativement, svn ci est le raccourci pour svn commit

Notez l’option -m (--message). De bons messages de commit aident les autres à comprendre pourquoi un commit a été fait. De plus, côté serveur, il est possible de [imposer des messages non vides] [4], et même d’imposer que chaque message de validation mentionne un ticket existant dans votre système de suivi des bogues.

[1] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.commit.html [2] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.status.html [3] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.diff.html [4] : https://tortoisesvn.net/docs/nightly/TortoiseSVN_en/tsvn-howto-minlogmsgsize.html

Examen des journaux #

L’exécution de svn log vous montrera tous les messages de validation, vous ne voudrez probablement revoir que certaines révisions.

  • Afficher les n révisions les plus récentes :

    svn log -n

  • Afficher une révision spécifique :

    svn log -c rXXX

  • Visualiser les chemins concernés :

    svn log -v -c rXXX

Installation et configuration initiale #

Installez le client svn pour commencer à collaborer sur le projet qui utilise Subversion comme système de contrôle de version.

Pour installer Subversion, vous pouvez le créer vous-même à partir d’une version du code source ou télécharger un package binaire pré-construit pour votre système d’exploitation. La liste des sites où vous pouvez obtenir un client Subversion compilé (svn) pour différents systèmes d’exploitation est disponible sur la [page officielle des packages binaires][1]. Si vous avez envie de compiler le logiciel par vous-même, récupérez la source sur la [page du code source][2].

Avec Subversion, vous n’êtes pas limité à l’utilisation du client de ligne de commande standard svn. Il existe quelques clients graphiques Subversion notables pour divers systèmes d’exploitation et la plupart des IDE offrent aujourd’hui une intégration robuste avec SVN dès la sortie de la boîte ou via des plugins. Pour la liste des clients graphiques, consultez la page Wikipedia : https://en.wikipedia.org/wiki/Comparison_of_Subversion_clients.

Juste après avoir installé le client, vous devriez pouvoir l’exécuter en lançant la commande svn. Vous devriez voir ce qui suit :

$ svn
Type 'svn help' for usage.

Tout est presque prêt. Vous devez maintenant créer un espace de travail local appelé copie de travail qui sera connecté au dépôt central distant. En d’autres termes, vous allez vérifier une copie de travail. Vous allez opérer avec les données versionnées à l’aide de la copie de travail et pouvez publier vos modifications (appelées committing dans SVN) afin que d’autres personnes travaillant sur le même projet puissent les voir et bénéficier de vos modifications. Pour récupérer ultérieurement les futures modifications apportées par d’autres à partir du référentiel, vous devez mettre à jour votre copie de travail. Ces opérations de base sont traitées dans d’autres exemples.

[1] : http://subversion.apache.org/packages.html [2] : http://subversion.apache.org/source-code.html [3] : https://tortoisesvn.net/

Création et application de correctifs #

Un correctif est un fichier qui montre les différences entre deux révisions ou entre votre référentiel local et la dernière révision pointée par votre référentiel.

Pour partager ou enregistrer un correctif de vos modifications locales non validées, soit pour examen par les pairs, soit pour postuler ultérieurement, procédez comme suit :

svn diff > new-feature.patch

Pour obtenir un correctif à partir des différences entre deux révisions :

svn diff -r NEWER_REVISION:OLDER_REVISION > feature.patch

Pour appliquer un correctif, exécutez :

svn patch new-feature.patch

Pour appliquer le correctif avec succès, vous devez exécuter la commande à partir du même chemin où le correctif a été créé.

Extraction d’une copie de travail #

Pour commencer à apporter des modifications aux données du projet, vous devez obtenir une copie locale du projet versionné. Utilisez le client svn en ligne de commande ou votre client SVN préféré (TortoiseSVN, par exemple). Votre copie locale du projet est appelée copie de travail dans Subversion et vous l’obtenez en exécutant la commande [svn checkout <URL>][1] où <URL> est une URL de référentiel. par exemple.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk

Alternativement, vous pouvez utiliser svn co <URL> comme raccourci pour extraire une copie locale.

En conséquence, vous obtiendrez une copie de travail du /trunk d’un projet appelé MyProject qui réside dans le référentiel MyRepo. La copie de travail sera située dans un répertoire appelé “trunk” sur votre ordinateur par rapport au répertoire dans lequel vous avez lancé la commande.

Si vous souhaitez avoir un nom différent pour votre copie de travail, vous pouvez l’ajouter en tant que paramètre à la fin de la commande. par exemple.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk MyProjectSource

Cela créera une copie de travail appelée MyProjectSource.

Notez qu’au lieu d’extraire le tronc, vous pouvez extraire une branche, une étagère privée ou une balise (en supposant qu’elles existent déjà dans le référentiel) ; vous pouvez avoir un nombre illimité de copies de travail locales sur votre machine.

Vous pouvez également obtenir la copie de travail de l’ensemble du référentiel MyRepo. Mais vous devriez vous abstenir de le faire. D’une manière générale, vous n’avez pas besoin d’avoir une copie de travail de l’ensemble du référentiel pour votre travail car votre copie de travail peut être instantanément basculée vers une autre branche de développement / tag / peu importe. De plus, le référentiel Subversion peut contenir un certain nombre de projets (non) liés et il est préférable d’avoir une copie de travail dédiée pour chacun d’eux, et non une seule copie de travail pour tous les projets.

[1] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.checkout.html

Exportation des données versionnées (téléchargement simple) #

Si vous souhaitez obtenir les données du projet versionné, mais que vous n’avez besoin d’aucune des fonctionnalités de contrôle de version offertes par Subversion, vous pouvez exécuter la commande [svn export <URL>][1]. Voici un exemple:

$ svn export https://svn.example.com/svn/MyRepo/MyProject/trunk

En conséquence, vous obtiendrez l’exportation des données du projet, mais contrairement à une copie de travail, vous ne pourrez pas exécuter de commandes svn dessus. L’exportation n’est qu’un simple téléchargement des données.

Si, plus tard, vous souhaitez convertir les données téléchargées en une copie de travail entièrement fonctionnelle, exécutez svn checkout <URL> dans le répertoire où vous avez exécuté l’exportation.

[1] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.export.html

Extraction d’une copie de travail à une révision spécifique #

Pour obtenir la version 5394, utilisez :

svn co --revision r5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

Ou la version courte :

svn co -r 5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

Ou en utilisant des révisions indexées :

svn co https://svn.example.com/svn/MyRepo/MyProject/trunk@5394

S’il est déjà extrait, vous pouvez utiliser la commande update pour déplacer a vers une révision particulière, en faisant :

svn up -rXXX

Mise à jour d’une copie de travail #

Vous n’êtes pas la seule personne à travailler sur le projet, n’est-ce pas ? Cela signifie que vos collègues apportent également des modifications aux données du projet. Pour rester à jour et récupérer les modifications validées par d’autres, vous devez exécuter la commande [svn update][1] dans votre copie de travail. En conséquence, votre copie de travail se synchronisera avec le référentiel et téléchargera les modifications apportées par vos collègues.

Le raccourci pour svn update est svn up.

C’est une règle d’exécuter [svn update][1] avant de valider vos modifications.

[1] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.update.html

Apporter des modifications à votre copie de travail locale #

La copie de travail (WC) est votre espace de travail local et privé que vous utilisez pour interagir avec le dépôt central de Subversion. Vous utilisez la copie de travail pour modifier le contenu de votre projet et récupérer les modifications validées par d’autres.

La copie de travail contient les données de votre projet et ressemble et agit comme un répertoire normal sur votre système de fichiers local, mais avec une différence majeure : la copie de travail suit l’état et les modifications des fichiers. et des répertoires à l’intérieur. Vous pouvez considérer la copie de travail comme un répertoire normal avec une saveur de contrôle de version ajoutée par un répertoire de métadonnées .svn caché à sa racine.

La plupart du temps, vous allez apporter des modifications aux données du projet en modifiant le contenu de la copie de travail. Dès que vous êtes satisfait des modifications et que vous les avez soigneusement examinées, vous êtes prêt à les publier dans le référentiel central.

Vous pouvez effectuer n’importe quelle action avec les données de votre projet dans la copie de travail, mais les opérations impliquant la copie, le déplacement, le changement de nom et la suppression doivent être effectuées à l’aide des commandes svn correspondantes :

  • Modification de fichiers existants. Modifiez les fichiers comme vous le faites habituellement en utilisant votre traitement de texte préféré, votre éditeur graphique, votre logiciel d’édition audio, votre IDE, etc. Dès que vous enregistrez les modifications sur le disque, Subversion les reconnaît automatiquement.

  • Ajout de nouveaux fichiers. Mettez les nouveaux fichiers dans la copie de travail et Subversion les reconnaîtra comme non versionnés. Il ne démarrera pas automatiquement le suivi des nouveaux fichiers à moins que vous n’exécutiez la commande [svn add][1] :

    svn add foo.cs

  • Déplacer des fichiers et des répertoires. Déplacez les fichiers et les répertoires à l’aide de la commande [svn move][2] :

    svn move foo.cs bar.cs

  • Renommer des fichiers et des répertoires. Renommez les fichiers et les répertoires à l’aide de la commande [svn rename][2] :

    svn rename foo.cs bar.cs

    NOTE: svn rename command is an alias of svn move command.

  • Copie de fichiers et de répertoires. Copiez les fichiers et les répertoires à l’aide de la commande [svn copy][3] :

    svn copy foo.cs bar.cs

  • Suppression de fichiers et de répertoires. Supprimez les fichiers et les répertoires à l’aide de la commande [svn delete][4] :

    svn delete foo.cs

  • Vérification de l’état des fichiers et des répertoires dans la copie de travail. Passez en revue vos modifications à l’aide de la commande [svn status][5] (ou [svn st][5] en abrégé) :

    svn status

    IMPORTANT: Always review your changes before committing them. This will help you to avoid committing unnecessary or irrelevant changes.

  • Annuler des changements. Annulez vos modifications à l’aide de la commande [svn revert][6] :

    svn revert foo.c

  • Annulation de toutes les modifications : depuis la racine du référentiel :

    svn revert -R .

IMPORTANT : Les modifications non validées annulées seront définitivement perdues. Vous ne pourrez pas récupérer les modifications annulées. Utilisez svn revert avec prudence ! Si vous souhaitez conserver les modifications mais que vous devez les annuler, enregistrez-les dans un patch. Voir l’exemple de création et d’application d’un patch.

[1] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.add.html [2] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.move.html [3] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.copy.html [4] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.delete.html [5] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.status.html [6] : http://svnbook.red-bean.com/en/1.8/svn.ref.svn.c.revert.html

Rétablir ou restaurer un fichier #

Pour restaurer un fichier à la dernière version mise à jour de svn, c’est-à-dire annuler les modifications locales, vous pouvez utiliser revert :

svn revert file

Pour restaurer un fichier vers une version plus ancienne (révision XXX), utilisez update :

svn update -r XXX file

Avertissement : dans les deux cas, vous perdrez toutes les modifications locales apportées au fichier car il sera écrasé.

Pour afficher uniquement l’ancienne version d’un fichier, utilisez cat :

svn cat -r XXX file

Et pour voir les différences avec votre version locale du fichier :

svn diff -r XXX file

Utilisation d’un référentiel protégé par mot de passe #

Un référentiel Subversion peut être configuré de manière à ce que certains contenus ou commandes ne soient accessibles qu’à certains utilisateurs. Pour accéder à ce contenu restreint, vous devrez spécifier un nom d’utilisateur et un mot de passe.

Votre nom d’utilisateur et votre mot de passe peuvent être spécifiés directement dans le cadre de la commande :

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser --password topsecret

Malheureusement, cela fait apparaître votre mot de passe en clair sur la console. Pour éviter ce problème de sécurité possible, spécifiez un nom d’utilisateur mais pas un mot de passe. Cela entraînera l’apparition d’une invite de mot de passe, vous permettant d’entrer votre mot de passe sans l’exposer :

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser
Password for 'JoeUser':

Ne fournir aucune information d’authentification oblige Subversion à vous demander à la fois le nom d’utilisateur et le mot de passe :

$ svn checkout https://svn.example.com/MyRepo/trunk
Username:  JoeUser
Password for 'JoeUser':

Bien que la première méthode soit moins sécurisée, elle est fréquemment rencontrée dans les scripts automatisés car il est difficile pour de nombreux types de scripts de fournir des informations à une invite interactive.

Remarque : les commandes qui ne fonctionnent que sur votre copie de travail (telles que revert ou status) ne nécessiteront jamais de mot de passe, uniquement les commandes nécessitant une communication avec le serveur de référentiel.
