Personnalisation des fonctionnalités de base avec des thèmes

Les thèmes ont des fichiers de mise en page qui, entre autres, peuvent être utilisés pour changer les blocs qui apparaissent sur la page. Le modèle de bloc peut également être modifié et différentes méthodes appelées.

Conceptions au niveau du magasin

Les thèmes structurés hiérarchiques de Magento signifient qu’un thème de base peut être étendu et attribué au niveau du magasin.

Enregistrement de thèmes personnalisés

Les thèmes peuvent être configurés de trois manières :

  1. Par magasin sous Système > Configuration > Design.
  2. Modification de conception avec délais Système > Conception.
  3. Les exceptions thématiques peuvent également être définies au niveau de la catégorie et du produit.

Forfait versus Thème

Un paquet a plusieurs thèmes. Chacun des thèmes d’un package hérite du thème par défaut d’un package.

Conception de repli

La procédure de secours du thème pour localiser les fichiers de modèles est la suivante :

  1. {forfait}/{thème}
  2. {package}/par défaut
  3. base/par défaut

Pour ajouter d’autres répertoires au mécanisme de secours du thème, la méthode Mage_Core_Model_Design_Package :: getFilename doit être réécrite

Pour la zone d’administration, le repli est default/default.

Chemins de modèle et de mise en page

Blocs

Les blocs sont utilisés pour la sortie. Le bloc racine est le parent de tous les blocs et est de type Mage_Page_Block_Html.

Les blocs Mage_Core_Block_Template utilisent des fichiers modèles pour rendre le contenu. Le nom du fichier modèle est défini dans setTemplate() ou addData(’template’) avec des chemins relatifs.

Les modèles ne sont que des morceaux de PHP inclus dans Mage_Core_Block_Template. Par conséquent, $this dans un modèle fait référence au bloc.

Mage_Core_Block_Template utilise un tampon avant d’inclure un modèle pour éviter une sortie prématurée.

La méthode Mage_Core_Model_Layout::createBlock crée des instances de blocs.

La classe Mage_Core_Model_Layout_Update considère quels blocs doivent être créés pour chaque page en regardant les poignées de mise en page.

Tous les blocs de sortie sont rendus, par ex. en appelant toHtml(), qui à son tour peut choisir de rendre ses enfants.

Les blocs Text et Text_List restituent automatiquement leur contenu.

Deux événements sont déclenchés autour du rendu de bloc et peuvent être utilisés pour modifier le bloc avant et après le rendu du HTML :

core_block_abstract_to_html_before core_block_abstract_to_html_after Un bloc enfant ne sera rendu automatiquement que s’il appartient à la classe Mage_Core_Block_Textlist, sinon la méthode getChildHtml doit être appelée.

Les instances de bloc sont accessibles via la mise en page, par ex. Mage::app()->getLayout() et $controller->getLayout(). La sortie du bloc est contrôlée par la fonction _toHtml().

Les modèles sont rendus par les méthodes renderView()/fetchView() à l’intérieur d’un bloc de modèle. La mise en mémoire tampon de sortie peut être désactivée avec $layout->setDirectOutput.

Il est possible d’ajouter un bloc à la mise en page actuelle, mais cela doit être fait avant que la méthode renderLayout() ne soit appelée.

Mise en page XML

<reference>
  -edit a block
<block>
  - define a block 
<action>
  - call method on a block
<update>
  - include nodes from another handle.

Les fichiers de mise en page peuvent être enregistrés dans config.xml :

<config>
    <{area}>
        <layout>
            <updates>
                <{name}>
                    <file>{filepath}</file>
                </{name}>
            </updates>
        </layout>
    </{area}>
</config>

La sortie de la page peut être personnalisée des manières suivantes :

  • Changements de modèle
  • Changements de mise en page
  • Remplacer les blocs
  • Observateurs Les variables sur les blocs peuvent être définies des manières suivantes :
  • Disposition -Through actions or attributes
  • Manette -$this-getLayout()->getBlock()
  • Blocs enfants -$this->getChild()
  • Autre -Mage::app()->getLayout()

Actifs du bloc principal

Les ressources JavaScript et CSS sont gérées dans le bloc Mage_Page_Block_Html_head. Ce bloc gère la fusion des actifs dans un seul fichier pour minimiser les requêtes HTTP. Le fichier fusionné est basé sur l’heure d’édition des fichiers source.

Lors de la fusion de CSS, une fonction de rappel sur Mage_Core_Model_Design_Package est appelée pour mettre à jour toutes les directives @import ou url() avec les URL correctes

Différents mécanismes pour désactiver la sortie du bloc #

  • Si la réponse a déjà été créée et définie sur la réponse objet en dehors du processus de rendu normal (par exemple, dans un observateur), le drapeau ’no-renderLayout’ peut être défini sur le contrôleur d’action à l’aide

    Mage::app()->getFrontController()->getAction()->setFlag(’’,’no-renderLayout’);

  • Cela empêche renderLayout() de traiter les blocs de sortie.

  • La même chose peut être obtenue en appelant setNoRender(true) sur le devant contrôleur : Mage::app()->getFrontController()->setNoRender(true);

  • Définir le drapeau isDispatched() sur l’objet de réponse peut être plus efficace pour obtenir un effet similaire.

Les instances de bloc sont accessibles depuis le contrôleur #

Depuis un contrôleur d’action :

$this->getLayout()->getBlock('head')->getTemplate();

/**
* Get specified tab grid
*/
public function gridOnlyAction()
{
$this->_initProduct();
$this->getResponse()->setBody(
$this->getLayout()->createBlock('adminhtml/catalog_product_edit_tab_' .
$this->getRequest()->gerParam('gridOnlyBlock')
)
->toHtml()
);
}
