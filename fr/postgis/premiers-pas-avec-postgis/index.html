Installation à partir de la source (avec Postgres 9.1 ou supérieur) #

Ce guide est explicitement pour PostgreSQL 9.1 ou supérieur sur les machines Linux. Il utilise la fonctionnalité d’extensions postgres qui améliorera considérablement l’importation d’extensions dans une installation postgres existante. Si vous devez travailler avec une ancienne version de postgres, veuillez vous référer aux [documentations officielles][1].

Résoudre les dépendances

PostGIS est un projet complexe qui a un certain nombre de dépendances. Afin de procéder à la configuration manuelle et à la procédure de construction, vous devrez résoudre ces dépendances et installer les packages suivants manuellement ou via les gestionnaires de packages.

Exigences minimales

  • [PostgreSQL][2] 9.1 ou supérieur. Il est important que vous installiez la base de données, y compris les en-têtes de serveur, qui se trouvent généralement dans les packages dev du gestionnaire de packages de votre référentiel.
  • Le compilateur GNU C [gcc][3].
  • GNU [marque][4]. Pour terminer le processus de construction.
  • [Proj4][5]. Une bibliothèque de projection, pour les transformations de coordonnées.
  • [GEOS][6]. Une bibliothèque de géométrie, qui implémente des descriptions d’entités et des géométries simples. La version 3.5 ou supérieure est recommandée afin d’utiliser des fonctions plus récentes telles que ST_ClipByBox2D et ST_Subdivide.
  • [GDAL][7], version 1.9 ou supérieure. Une bibliothèque qui implémente des formats de données abstraits pour les données géospatiales raster et vectorielles.
  • [LibXML2][8], version 2.5 ou supérieure. Une librairie pour travailler avec XML, XSLT et DTD.
  • [JSON-C][9], version 0.9 ou supérieure. Une bibliothèque pour créer une sortie au format JSON

Exigences facultatives

  • [GTK][10] (nécessite GTK+2.0, 2.8+) pour compiler le shp2pgsql-gui.
  • [SFCGAL][11], version 1.1 (ou supérieure) pourrait être utilisée pour fournir des fonctions d’analyse avancées 2D et 3D supplémentaires à PostGIS.
  • [PCRE][12]. Une bibliothèque pour la correspondance de modèles d’expressions régulières à l’aide de la syntaxe Perl 5. Cette bibliothèque est nécessaire si vous souhaitez travailler avec [Adress Standardizer][13].
  • [UCunit][14]. Un utilitaire de test unitaire, nécessaire pour les tests de régression.
  • [DocBook][15] (xsltproc) est requis pour la construction de la documentation.
  • [DBLatex][16] est nécessaire pour construire la documentation au format PDF.
  • [ImageMagick][17] est requis pour générer les images utilisées dans la documentation.

Obtenir les sources

Afin d’obtenir le code source, téléchargez la dernière archive tar :

wget http://postgis.net/stuff/postgis-2.3.2dev.tar.gz
tar -xvzf postgis-2.3.2dev.tar.gz

ou utilisez le référentiel SVN officiel :

svn checkout http://svn.osgeo.org/postgis/trunk/ postgis-2.3.2dev

Configuration

Si vous avez obtenu les sources via SVN, vous pouvez préparer le script de configuration avec :

./autogen.sh

Afin de configurer le processus de construction pour votre machine spécifique, exécutez dans le dossier du projet :

./configure

Il existe plusieurs paramètres facultatifs pour l’étape de configuration. Veuillez vous référer à la [documentation officielle] [18] pour des instructions détaillées, ceci est généralement facultatif et uniquement pour les serveurs qui utilisent des installations non par défaut.

Construire

Une fois l’étape de configuration terminée avec succès, un makefile sera créé. Pour démarrer l’exécution du processus de génération :

make

La dernière sortie devrait être :

"PostGIS was built successfully. Ready to install."

Depuis la version 1.4.0, toutes les fonctions ont des commentaires générés à partir de la documentation. Si vous souhaitez installer ultérieurement ces commentaires dans vos bases de données spatiales, exécutez la commande qui nécessite docbook.

make comments

Installation

Installez toutes les extensions avec :

make install

Les extensions PostGIS sont construites et installées automatiquement si vous utilisez PostgreSQL 9.1 ou supérieur. Vous pouvez installer les extensions nécessaires manuellement, si vous avez une configuration différente.

Dans le dossier du projet :

cd extensions
cd postgis
make clean
make
make install

cd ..
cd postgis_topology
make clean
make
make install

cd ..
cd postgis_sfcgal
make clean
make
make install

cd ..
cd address_standardizer
make clean
make
make install
make installcheck

cd ..
cd postgis_tiger_geocoder
make clean
make
make install
make installcheck

Si vous souhaitez installer les extensions manuellement sur une autre machine, copiez les fichiers suivants du dossier extensions dans le dossier PostgreSQL/share/extension de la cible. Pour chaque extension :

scp extensions/[EXTENSION]/sql/*.sql user@target:[POSTGIS_PATH]/share/extension

où [EXTENSION] est l’extension sélectionnée (postgis, postgis_topology, postgis_sfcgal, address_standardizer, postgis_tiger_geocoder) et [POSTGIS_PATH] est le chemin d’installation de PostGIS sur votre machine cible.

Vérification de l’installation

Si vous n’avez pas de service de base de données postgres en cours d’exécution, [configurez d’abord votre base de données postgres][19]. Connectez-vous à la base de données en utilisant :

su postgres -c psql

Pour vérifier que les extensions sont accessibles, exécutez les requêtes suivantes dans une session psql :

SELECT name, default_version,installed_version FROM pg_available_extensions WHERE name LIKE 'postgis%' or name LIKE 'address%';

La sortie devrait ressembler à ceci :

             name             | default_version | installed_version
------------------------------+-----------------+-------------------
 address_standardizer         | 2.3.2dev        | 2.3.2dev
 address_standardizer_data_us | 2.3.2dev        | 2.3.2dev
 postgis                      | 2.3.2dev        | 2.3.2dev
 postgis_sfcgal               | 2.3.2dev        |
 postgis_tiger_geocoder       | 2.3.2dev        | 2.3.2dev
 postgis_topology             | 2.3.2dev        |

(6 rows)

Pour effectuer un test post-installation approfondi, exécutez la commande suivante dans votre dossier de projet :

make check

Cela passera par diverses vérifications et tests en utilisant la bibliothèque générée par rapport à une base de données PostgreSQL réelle.

Un “Hello World” géospatial #

Dans cet exemple, nous allons configurer une base de données géospatiales, importer des données de 2 sources différentes et afficher les résultats dans une application appelée QGIS. Ce guide est explicitement écrit pour les machines Linux, si vous travaillez sur une autre plate-forme, certaines commandes ou certains chemins peuvent ne pas fonctionner comme prévu.

Afin de visualiser les données importées, nous utiliserons une application appelée [QGIS][1]. Si vous n’avez pas cette application, [veuillez l’installer d’abord][2], si vous souhaitez utiliser une autre visionneuse ou géo-application (comme ArcGIS), vous pouvez ignorer l’installation de QGIS.

Nos sources seront [New York City State Assembly Districts][3] et la [New York City LION Street Map Database][4]. Veuillez télécharger les fichiers appropriés à partir des emplacements liés. Vous devriez également jeter un œil à la section Métadonnées du matériel, car elle vous donne des informations sur le système de référence de coordonnées utilisé par ces fichiers.

Pour commencer, créez un dossier de travail “nycgis”, copiez les fichiers téléchargés à l’emplacement et décompressez les archives.

mkdir nycgis
cd nycgis
cp ~/Downloads/nyad_16d.zip .
unzip ~/Downloads/nyad_16d.zip
cp ~/Downloads/nylion_16d.zip .
unzip  ~/Downloads/nylion_16d.zip

Dans le dossier “nycgis” vous devriez maintenant avoir 2 dossiers : “nyad_16d”, “lion” avec plusieurs fichiers.

Lorsque vous travaillez avec des données géographiques, il est d’une importance vitale de connaître le système de référence de coordonnées (CRS) de vos données source et de vos données de sortie finales. Dans les sections Métadonnées des emplacements liés ci-dessus ([Métadonnées : Districts de l’Assemblée][5], [Métadonnées : Base de données LION][6]), vous trouverez que le CRS pour les deux fichiers est EPSG : 2263, un système de coordonnées utilisé pour faire référence au nord-est des États-Unis.

Supposons que nous voulions utiliser un CRS différent dans notre base de données. Cela peut avoir différentes raisons, nous pourrions vouloir travailler avec une application géographique basée sur le Web sur la base de données par exemple. Un CRS courant pour ce type d’application est WGS:84 (EPSG:4326).

Pour convertir les systèmes de coordonnées, nous utilisons un outil appelé ogr2ogr qui fait partie du package GDAL. Dans le dossier de travail, nous créons d’abord 2 dossiers représentant les données reprojetées, puis convertissons nos données.

mkdir nyad_16d_proj_4326
ogr2ogr -f "ESRI Shapefile" ./nyad_16d_proj_4326/nyad_4326.shp ./nyad_16d/nyad_16d.shp -s_srs EPSG:2263 -t_srs EPSG:4326

mkdir nylion_16d_proj_4326
ogr2ogr -f "ESRI Shapefile" ./nylion_16d_proj_4326/ny_str_4326.shp ./nylion_16d/lion/lion.gdb/a0000000d.gdbtable -s_srs EPSG:2263 -t_srs EPSG:4326

Notez que nous utilisons uniquement le fichier appelé : “a0000000d.gdbtable” de la base de données LION pour nos besoins. La syntaxe de la commande ogr2ogr est la suivante :

ogr2ogr -f [output-format] [output-file] [input-file] -s_srs [source crs] -t_srs [target crs]

Nous avons maintenant 2 shapefiles, projetés dans le bon CRS. Afin d’utiliser les données de notre base de données, nous devons convertir les shapefiles en sql-statemens. Pour cela, nous utilisons un outil appelé shp2pgsql. Dans le répertoire de travail, exécutez les commandes suivantes :

shp2pgsql ./nyad_16d_proj_4326/nyad_4326.shp > nyad_4326.sql
shp2pgsql ./nylion_16d_proj_4326/ny_str_4326.shp > ny_streets_4326.sql

Les fichiers nyad_4326.sql et ny_streets_4326.sql sont maintenant prêts à être utilisés dans postgis. Pour continuer et importer les données, créez une base de données spatialement activée.

sudo su - postgres
createdb nycgis
psql -d nycgis

Dans la session psql, exécutez :

CREATE EXTENSION postgis;
CREATE EXTENSION postgis_topology;

Quittez les sessions utilisateur psql et postgres avec (CTRL+D). Et importez les fichiers avec :

psql -f nyad_4326.sql -d nycgis
psql -f ny_streets_4326.sql -d nycgis

La base de données nycgis contient maintenant 2 tables dans lesquelles les sources reprojetées ont été importées avec succès.

Pour vérifier cela : ouvrez QGIS

  1. utilisez Couche > Ajouter une couche > PostGIS-Layer
  2. connectez-vous à votre base de données
  3. sélectionnez vos tableaux
  4. (facultatif) définissez le style des calques nouvellement créés.

[![Capture d’écran QGIS][7]][7]

Et voilà : vous avez maintenant une base de données spatialement activée, avec des géodonnées importées et reprojetées.

Installation via le gestionnaire de packages #

Cambre:

Un package pacman officiel est disponible. Installez le package en tant que root, en utilisant :

pacman -S postgis

OpenSuse :

Afin d’utiliser les référentiels openSuse pour les applications géospatiales, activez le géo-référentiel en tant que racine :

zypper ar http://download.opensuse.org/repositories/Application:/Geo/openSUSE_[RELEASE]/ GEO

où [RELEASE] est le numéro de version officiel de votre distribution Suse. Après cela, vous pouvez installer postgis avec :

zypper install postgis

Mise en place d’une base de données géospatiales #

Pour créer une nouvelle base de données vide, exécutez en tant qu’utilisateur postgres :

createdb [yourdatabase]

Connectez-vous à la base de données avec une session psql :

psql -d [yourdatabase]

Dans l’exécution de la session psql :

CREATE EXTENSION postgis;
CREATE EXTENSION postgis_topology;

pour créer les extensions géospatiales nécessaires. Une fois cela fait, la base de données est une base de données géospatiale et elle est prête à être utilisée.
