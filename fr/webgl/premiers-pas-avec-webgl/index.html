Installation ou configuration #

WebGL est une technologie de navigateur, il n’y a donc pas grand chose à configurer autre que d’avoir un navigateur. Vous pouvez démarrer avec WebGL sur JSFiddle ou Codepen ou JSBIn ou n’importe quel nombre d’autres sites qui vous permettent de modifier HTML, CSS et JavaScript en ligne, mais il y aura quelques limitations (voir ci-dessous). Vous pouvez également héberger des fichiers open source sur pages github ou des services similaires.

D’un autre côté, à un moment donné, vous allez probablement travailler localement. Pour ce faire, il est recommandé d’exécuter un simple serveur Web. Il y a beaucoup de choix qui sont simples à utiliser et nécessitent très peu de configuration.

Utiliser node.js comme serveur

  1. installez node.js
  2. Ouvrez une invite de commande de terminal ou de nœud et tapez npm install -g http-server (sur OSX, mettez sudo devant cela.
  3. tapez http-server pour commencer à servir les fichiers dans le dossier actuel OU http-server path-to-folder pour servir un dossier différent
  4. Pointez votre navigateur sur http://localhost:8080/name-of-file pour afficher votre page Web WebGL

Utiliser devd comme serveur

  1. Téléchargez [devd] (https://github.com/cortesi/devd)
  2. Ouvrez un terminal et exécutez devd avec devd . pour les fichiers du serveur à partir du dossier actuel ou devd path-to-folder pour servir un dossier différent
  3. Pointez votre navigateur sur http://localhost:8000/name-of-file pour afficher votre page Web WebGL

Utiliser Servez comme serveur

  1. Download Servez
  2. Installez-le, exécutez-le
  3. Choisissez le dossier à servir
  4. Choisissez “Démarrer”
  5. Allez sur http://localhost:8080 ou choisissez “Lancer le navigateur”

Utilisation de l’extension Chrome “Serveur Web pour Chrome”

  1. Installez le [serveur Web à partir de Chrome] (https://chrome.google.com/webstore/detail/web-server-for-chrome/ofhbbkphhbklhfoeikjpcbhemlocgigb?hl=en)

  2. Lancez-le à partir de l’icône Apps sur une nouvelle page à onglet.

[![comment lancer l’application sur chrome][1]][1]

  1. Définissez le dossier où se trouvent vos fichiers, puis cliquez sur le lien http://127.0.0.1:8787

[![boîte de dialogue des paramètres][2]][2]

Limitation de WebGL sur les services en ligne #

Dans WebGL, il est très courant de charger des images. Dans WebGL, il existe des restrictions sur la façon dont les images peuvent être utilisées. Plus précisément, WebGL ne peut pas utiliser d’images provenant d’autres domaines sans l’autorisation du serveur hébergeant les images. Les services qui autorisent actuellement l’utilisation d’images incluent imgur et flickr. Voir Chargement d’images inter-domaines. Sinon, vous devrez avoir les images sur le même serveur que votre page webgl ou utiliser d’autres solutions créatives comme générer des images avec une balise canvas

Bonjour le monde #

Comme il le mentionne dans la section * remarques *, nous devons fournir deux fonctions. Un vertex shader et un fragment shader

Commençons par un vertex shader

// an attribute will receive data from a buffer
attribute vec4 position;

// all shaders have a main function
void main() {

  // gl_Position is a special variable a vertex shader 
  // is responsible for setting
  gl_Position = position;
}

Si tout était écrit en JavaScript au lieu de GLSL, vous pourriez imaginer qu’il serait utilisé comme ceci

// *** PSUEDO CODE!! ***

var positionBuffer = [
  0, 0, 0, 0,
  0, 0.5, 0, 0,
  0.7, 0, 0, 0,
];
var attributes = {};
var gl_Position;

drawArrays(..., offset, count) {
  for (var i = 0; i < count; ++i) {
     // copy the next 4 values from positionBuffer to the position attribute
     attributes.position = positionBuffer.slice((offset + i) * 4, 4); 
     runVertexShader();
     ...
     doSomethingWith_gl_Position();
}

Ensuite, nous avons besoin d’un fragment shader

// fragment shaders don't have a default precision so we need
// to pick one. mediump, short for medium precision, is a good default.
precision mediump float;

void main() {
  // gl_FragColor is a special variable a fragment shader
  // is responsible for setting
  gl_FragColor = vec4(1, 0, 0.5, 1); // return redish-purple 
}

Ci-dessus, nous définissons gl_FragColor sur 1, 0, 0.5, 1 qui est 1 pour le rouge, 0 pour le vert, 0.5 pour le bleu, 1 pour l’alpha. Les couleurs dans WebGL vont de 0 à 1.

Maintenant que nous avons écrit les 2 fonctions, commençons avec WebGL

Nous avons d’abord besoin d’un élément HTML canvas

 <canvas id="c"></canvas>

Ensuite, en JavaScript, nous pouvons rechercher cela

 var canvas = document.getElementById("c");

Nous pouvons maintenant créer un WebGLRenderingContext

 var gl = canvas.getContext("webgl");
 if (!gl) {
    // no webgl for you!
    ...

Nous devons maintenant compiler ces shaders pour les mettre sur le GPU, nous devons donc d’abord les intégrer dans des chaînes. Vous pouvez obtenir vos chaînes de la manière habituelle. En concaténant, en utilisant AJAX, en les plaçant dans des balises de script non typées JavaScript, ou dans ce cas en utilisant des modèles littéraux multilignes

var vertexShaderSource = `
// an attribute will receive data from a buffer
attribute vec4 position;

// all shaders have a main function
void main() {

  // gl_Position is a special variable a vertex shader 
  // is responsible for setting
  gl_Position = position;
}
`;

var fragmentShaderSource = `
// fragment shaders don't have a default precision so we need
// to pick one. mediump is a good default
precision mediump float;

void main() {
  // gl_FragColor is a special variable a fragment shader
  // is responsible for setting
  gl_FragColor = vec4(1, 0, 0.5, 1); // return redish-purple 
}
`;

Ensuite, nous avons besoin d’une fonction qui va créer un shader, télécharger la source et compiler le shader

function createShader(gl, type, source) {
  var shader = gl.createShader(type);
  gl.shaderSource(shader, source);  
  gl.compileShader(shader);
  var success = gl.getShaderParameter(shader, gl.COMPILE_STATUS);
  if (success) {
    return shader;
  }

  console.log(gl.getShaderInfoLog(shader));
  gl.deleteShader(shader);
}

Nous pouvons maintenant appeler cette fonction pour créer les 2 shaders

var vertexShader = createShader(gl, gl.VERTEX_SHADER, vertexShaderSource);
var fragmentShader = createShader(gl, gl.FRAGMENT_SHADER, fragmentShaderSource);

Nous devons ensuite lier ces 2 shaders dans un programme

function createProgram(gl, vertexShader, fragmentShader) {
  var program = gl.createProgram();
  gl.attachShader(program, vertexShader);
  gl.attachShader(program, fragmentShader);
  gl.linkProgram(program);
  var sucesss = gl.getProgramParameter(program, gl.LINK_STATUS);
  if (success) {
    return program;
  }

  console.log(gl.getProgramInfoLog(program));
  gl.deleteProgram(program);
}

Et appelle-le

var program = createProgram(gl, vertexShader, fragmentShader);

Maintenant que nous avons créé un programme GLSL sur le GPU, nous devons lui fournir des données. La majorité de l’API WebGL consiste à configurer l’état pour fournir des données à nos programmes GLSL. Dans ce cas, notre seule entrée dans notre programme GLSL est position qui est un attribut. La première chose que nous devrions faire est de rechercher l’emplacement de l’attribut pour le programme que nous venons de créer

var positionAttributeLocation = gl.getAttribLocation(program, "position");

Les attributs obtiennent leurs données des tampons, nous devons donc créer un tampon

var positionBuffer = gl.createBuffer();

WebGL nous permet de manipuler de nombreuses ressources WebGL sur des points de liaison globaux. Vous pouvez considérer les points de liaison comme des variables globales internes dans WebGL. Vous définissez d’abord le point de liaison à votre ressource. Ensuite, toutes les autres fonctions font référence à la ressource via le point de liaison. Alors, lions le tampon de position.

gl.bindBuffer(gl.ARRAY_BUFFER, positionBuffer);

Maintenant, nous pouvons mettre des données dans ce tampon en le référençant via le point de liaison

// three 2d points
var positions = [
  0, 0, 
  0, 0.5,
  0.7, 0,
];
gl.bufferData(gl.ARRAY_BUFFER, new Float32Array(positions), gl.STATIC_DRAW);

Il se passe beaucoup de choses ici. La première chose est que nous avons positions qui est un tableau JavaScript. WebGL, d’autre part, a besoin de données fortement typées, de sorte que la ligne new Float32Array(positions) crée un nouveau tableau de nombres à virgule flottante 32 bits et copie les valeurs de positions. gl.bufferData copie ensuite ces données dans le positionBuffer du GPU. Il utilise le tampon de position car nous l’avons lié au point de liaison ARRAY_BUFFER ci-dessus.

Le dernier argument, gl.STATIC_DRAW est un indice pour WebGL sur la façon dont nous allons utiliser les données. Il peut essayer d’utiliser ces informations pour optimiser certaines choses. gl.STATIC_DRAW indique à WebGL que nous ne modifierons probablement pas beaucoup ces données.

Maintenant que nous avons mis les données dans le tampon a, nous devons indiquer à l’attribut comment en extraire les données. Tout d’abord, nous devons activer l’attribut

gl.enableVertexAttribArray(positionAttributeLocation);

Ensuite, nous devons spécifier comment extraire les données

var size = 2;          // 2 components per iteration
var type = gl.FLOAT;   // the data is 32bit floats
var normalize = false; // use the data as is
var stride = 0;        // 0 = move size * sizeof(type) each iteration
var offset = 0;        // start at the beginning of the buffer
gl.vertexAttribPointer(
   positionAttributeLocation, size, type, normalize, stride, offset)

Une partie cachée de gl.vertexAttribPointer est qu’il lie le ARRAY_BUFFER actuel à l’attribut. En d’autres termes, maintenant que cet attribut est lié à positionBuffer, nous sommes libres de lier autre chose au point de liaison ARRAY_BUFFER.

notez que du point de vue de notre vertex shader GLSL l’attribut de position était un vec4

attribute vec4 position;

vec4 est une valeur à 4 flottants. En JavaScript, vous pourriez penser à quelque chose comme position = {x : 0, y : 0, z : 0, w : 0}. Ci-dessus, nous définissons taille = 2. Les attributs par défaut sont 0, 0, 0, 1 donc cet attribut obtiendra ses 2 premières valeurs (x et y) de notre tampon. Le z et le w seront respectivement 0 et 1 par défaut.

Après tout cela, nous pouvons enfin demander à WebGL d’exécuter notre programme GLSL.

var primitiveType = gl.TRIANGLES;
var offset = 0;
var count = 3;
gl.drawArrays(primitiveType, offset, count);

Cela exécutera notre vertex shader 3 fois. La première fois que position.x et position.y dans notre vertex shader seront définis sur les 2 premières valeurs du positionBuffer. La 2ème fois position.xy sera définie sur les 2 2èmes valeurs. La dernière fois, il sera réglé sur les 2 dernières valeurs.

Parce que nous définissons primitiveType sur gl.TRIANGLES, chaque fois que notre vertex shader est exécuté 3 fois, WebGL dessinera un triangle basé sur les 3 valeurs que nous avons définies sur gl_Position. Quelle que soit la taille de notre canevas, ces valeurs sont dans des coordonnées d’espace de clip qui vont de -1 à 1 dans chaque direction.

Parce que notre vertex shader copie simplement nos valeurs de positionBuffer dans gl_Position, le triangle sera dessiné aux coordonnées de l’espace du clip

  0, 0, 
  0, 0.5,
  0.7, 0,

La façon dont ces valeurs se traduisent en pixels dépend du paramètre gl.viewport. gl.viewport utilise par défaut la taille initiale du canevas. Comme nous n’avons pas défini de taille pour notre canevas, c’est la taille par défaut de 300x150. Conversion de l’espace de clip en pixels (souvent appelé espace d’écran dans la littérature WebGL et OpenGL) WebGL va dessiner un triangle à

 clip space      screen space
   0, 0       ->   150, 75
   0, 0.5     ->   150, 112.5
 0.7, 0       ->   255, 75

WebGL va maintenant rendre ce triangle. Pour chaque pixel qu’il est sur le point de dessiner, il appellera notre fragment shader. Notre fragment shader définit simplement gl_FragColor sur 1, 0, 0.5, 1. Étant donné que le canevas est un canevas 8 bits par canal, cela signifie que WebGL va écrire les valeurs [255, 0, 127, 255] dans le canevas.

Il y a 3 choses importantes que nous n’avons toujours pas couvertes à partir des remarques. Textures, variations et uniformes. Chacun de ceux-ci nécessite son propre sujet.
