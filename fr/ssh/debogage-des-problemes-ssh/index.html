Connexion rejetée #

Une erreur “Connexion refusée” se produira si votre client envoie une demande de connexion à un hôte de serveur distant et que l’hôte distant répond en indiquant qu’il refuse d’accepter la demande. L’erreur “Connexion refusée” signifie essentiellement que l’ordinateur n’accepte pas les connexions à l’adresse IP et au port demandés.

“Connexion refusée” peut être causée par un pare-feu qui bloque les demandes de connexion. Un pare-feu configuré pour bloquer les connexions à un point de terminaison particulier peut être configuré pour abandonner les demandes de connexion, auquel cas le client n’obtiendra jamais de réponse et finira par expirer. Ou le pare-feu peut répondre aux demandes de connexion de connexion par une réponse de refus.

Outre les pare-feu, dans le cas de SSH, “connexion refusée” a plusieurs causes possibles :

  • Vous utilisez peut-être le mauvais numéro de port pour vous connecter. Le numéro de port standard pour SSH est 22, mais certaines personnes exécutent le service ssh sur un port différent pour dissuader les tentatives d’accès non autorisées.

  • Vous essayez peut-être de vous connecter au mauvais ordinateur. Vous avez peut-être mal saisi le nom d’hôte ou l’adresse IP. Ou l’ordinateur peut utiliser une adresse attribuée dynamiquement qui a changé.

  • Le processus du serveur ssh n’est peut-être pas en cours d’exécution :

  • Il se peut qu’il n’ait pas encore démarré si le système est en train de démarrer.

  • Il a peut-être été désactivé ; par exemple. lorsque le système est en mode mono-utilisateur.

  • Il a peut-être été mal configuré, ce qui a empêché son démarrage.

  • L’ordinateur n’a peut-être pas de serveur SSH configuré. Les systèmes MS Windows n’incluent généralement pas de serveur SSH. Sur certains systèmes Linux, le serveur SSH peut être un composant facultatif. OS X inclut un serveur SSH, mais il est désactivé par défaut.

  • Le processus du serveur SSH n’écoute peut-être pas les connexions sur l’interface IP spécifique à laquelle vous essayez de vous connecter. La plupart des ordinateurs ont au moins deux interfaces IP, une interface “localhost” et une ou plusieurs interfaces réseau. Chaque interface active aura une adresse IP qui lui est associée. Un serveur SSH est généralement configuré pour accepter les connexions sur n’importe quelle interface IP. Mais il peut être configuré pour accepter les connexions uniquement sur des interfaces particulières. Dans ce cas, l’ordinateur refusera les connexions à une adresse IP que le serveur SSH n’écoute pas, même si la demande de connexion a le bon port.

  • Le serveur peut avoir un arriéré de demandes de connexion au même port. C’est rare et inhabituel, mais si l’hôte reçoit des demandes de connexion plus rapidement qu’elles ne peuvent être traitées, l’hôte finira par rejeter les nouvelles demandes de connexion.

Notez que, mis à part les pare-feu, “connexion refusée” signifie que vous communiquez avec l’ordinateur distant - il n’accepte tout simplement pas votre demande de connexion.

La clé privée n’est pas acceptée (débogage des clients OpenSSH) #

Par défaut, la plupart des informations sont cachées à l’utilisateur. Vous pouvez utiliser les commutateurs -v pour obtenir un journal détaillé de la tentative de connexion, qui identifiera généralement le problème en montrant pourquoi le comportement est différent de ce à quoi vous vous attendiez.

Supposons que vous vous connectez au serveur example.com en utilisant ssh (ou un autre client OpenSSH comme sftp ou scp) et que votre clé privée n’est pas acceptée par le serveur et que le serveur demande le mot de passe (ou rejette la connexion) :

$ ssh example.com
user@example.com's password:

Essayez d’exécuter ssh avec les commutateurs -vvv, qui écriront tous les messages de débogage. Ce sera beaucoup d’informations, mais après un certain temps, il est assez facile de comprendre que :

$ ssh -vvv example.com

Le problème le plus courant est que la clé ne se trouve pas à l’emplacement prévu. Vous pouvez vous attendre à ce que des lignes similaires s’affichent, se plaignant de fichiers manquants. Vérifier que votre fichier a bien été lu est un bon début :

debug1: identity file /home/username/.ssh/id_dsa type -1
debug1: key_load_public: No such file or directory
debug1: identity file /home/username/.ssh/id_dsa-cert type -1

Nous pouvons passer à la partie authentification. La clé peut être proposée, mais rejetée par le serveur, en raison d’un problème de configuration du serveur. Le journal pourrait ressembler à ceci :

debug3: preferred publickey,keyboard-interactive,password
debug3: authmethod_lookup publickey
debug3: remaining preferred: keyboard-interactive,password
debug3: authmethod_is_enabled publickey
debug1: Next authentication method: publickey
debug1: Offering RSA public key: username@localhost
debug3: send_pubkey_test
debug3: send packet: type 50
debug2: we sent a publickey packet, wait for reply
debug3: receive packet: type 51
debug1: Authentications that can continue: publickey,gssapi-keyex,gssapi-with-mic,password

## L’IDENTIFICATION DE L’HÔTE À DISTANCE A CHANGÉ ! L’erreur courante en utilisant ssh est de voir l’erreur comme

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY!
Someone could be eavesdropping on you right now (man-in-the-middle attack)!
It is also possible that a host key has just been changed.
The fingerprint for the RSA key sent by the remote host is
SHA256:L5ri/Xdgpuals893ej1z5F1wlg1n2YNeBf/tsABX+QQ.
Please contact your system administrator.
Add correct host key in /Users/username/.ssh/known_hosts to get rid of this message.
Offending RSA key in /Users/username/.ssh/known_hosts:12
RSA host key for *IP address* has changed and you have requested strict checking.
Host key verification failed.

Cela signifie que vous vous êtes déjà connecté au même serveur et qu’il a été identifié à l’aide de clés d’hôte différentes. Si vous savez que vous avez changé les clés du serveur, réinstallé le serveur ou si l’administrateur du serveur a annoncé des changements, il est généralement acceptable de supprimer l’ancienne clé et de laisser le ssh stocker la nouvelle.

L’ancienne clé peut être supprimée de manière transparente à l’aide de ssh-keygen :

ssh-keygen -R *IP address*

Et la prochaine connexion devrait vous demander de vérifier la nouvelle empreinte :

ssh192.168.0.128
The authenticity of host '192.168.0.128 (192.168.0.128)' can't be established.
ECDSA key fingerprint is SHA256:L5ri/Xdgpuals893ej1z5F1wlg1n2YNeBf/tsABX+QQ.
Are you sure you want to continue connecting (yes/no)?

Si vous n’êtes au courant d’aucun des éléments ci-dessus, le mieux est de contacter votre administrateur de serveur pour vous assurer que tout va bien. Sinon, l’attaquant potentiel pourrait obtenir à la fois vos informations d’authentification et toutes les données transférées !

ssh_exchange_identification : lecture : réinitialisation de la connexion par un pair #

Ce message d’erreur peut être généré par le client OpenSSH ssh. Cela signifie que la connexion TCP entre le client et le serveur a été anormalement fermée par le serveur immédiatement après avoir été acceptée. Les raisons courantes de ce message incluent :

  • Le processus du serveur SSH fonctionne mal - par exemple, il s’est écrasé.
  • Un pare-feu, un routeur ou un autre périphérique réseau entre le client et le serveur interfère avec la connexion SSH.

Les phrases du message d’erreur indiquent précisément ce qui s’est passé :

ssh_exchange_identification : 1. Une fois qu’un client SSH s’est connecté à un serveur SSH, la première étape du démarrage du protocole SSH consiste pour le serveur à envoyer sa version logicielle au client. Une erreur contenant “ssh_exchange_identification” indique que l’erreur s’est produite immédiatement après avoir établi la connexion TCP, alors que le client attendait la version du logiciel du serveur.

Réinitialisation de la connexion : Une réinitialisation de la connexion signifie que la connexion TCP était “anormalement fermée”. Vous pouvez l’obtenir si le processus logiciel contenant l’une des connexions TCP se bloque. Les pare-feux réseau peuvent être configurés pour utiliser les réinitialisations de connexion comme moyen de bloquer les connexions TCP.

by peer signifie que la connexion TCP a été fermée depuis “l’autre extrémité” de la connexion. Dans ce cas, “l’autre extrémité” est le serveur SSH distant.

Notez que cette erreur n’indique aucun type d’échec d’authentification. Le serveur a fermé la connexion TCP immédiatement après l’avoir acceptée. Le client et le serveur n’ont encore échangé aucune donnée. Le serveur n’a pas encore envoyé de clé d’hôte au client et le client n’a pas encore essayé de s’authentifier auprès du serveur.

ssh_exchange_identification : connexion fermée par l’hôte distant #

Ce message d’erreur peut être généré par le client ssh OpenSSH. Cela signifie que la connexion TCP entre le client et le serveur a été fermée par le serveur immédiatement après avoir été acceptée. Ce message indique généralement que le serveur SSH a été configuré pour ne pas accepter les connexions du client pour une raison quelconque :

  • Le serveur peut être configuré pour refuser les connexions à partir de l’adresse IP du client.
  • Le serveur peut être configuré avec une limite sur le nombre de connexions SSH actives.
  • Le client peut s’être connecté à quelque chose qui n’est pas un serveur SSH.

Les phrases du message d’erreur indiquent précisément ce qui s’est passé :

ssh_exchange_identification : 1. Une fois qu’un client SSH s’est connecté à un serveur SSH, la première étape du démarrage du protocole SSH consiste pour le serveur à envoyer sa version logicielle au client. Une erreur contenant “ssh_exchange_identification” indique que l’erreur s’est produite immédiatement après l’établissement de la connexion TCP, alors que le client attendait la version du logiciel du serveur.

Connexion fermée : Cela signifie que la connexion TCP a été fermée normalement. Cela indique que le serveur SSH a délibérément fermé la connexion. Cela contraste avec “Réinitialisation de la connexion”, qui pourrait indiquer que le serveur SSH est tombé en panne ou a mal fonctionné.

par hôte distant signifie que la connexion TCP a été fermée depuis “l’autre extrémité” de la connexion. Dans ce cas, “l’autre extrémité” est le serveur SSH distant.

Notez que cette erreur n’indique aucun type d’échec d’authentification. Le serveur a fermé la connexion TCP immédiatement après l’avoir acceptée. Le client et le serveur n’ont encore échangé aucune donnée. Le serveur n’a pas encore envoyé de clé d’hôte au client et le client n’a pas encore essayé de s’authentifier auprès du serveur.

La connexion a expiré #

Une erreur “Expiration de la connexion” se produit lorsque le système distant ne répond pas à la tentative du client d’ouvrir une connexion TCP/IP. Les causes les plus courantes incluent :

  • Un pare-feu bloque la tentative de connexion sur le port que vous utilisez :
  • Le pare-feu peut être côté client, bloquant les connexions sortantes, côté serveur, bloquant les connexions entrantes, ou ailleurs sur le chemin.
  • Le problème racine pourrait être que vous utilisez le mauvais numéro de port.
  • L’hôte auquel vous tentez de vous connecter est peut-être hors ligne.
  • Une partie du réseau entre le client et le serveur peut être en panne ou interrompue.
  • Il peut y avoir un problème de routage réseau.

Il est difficile de diagnostiquer la cause première des délais d’attente de connexion.
