Les modèles d’expressions régulières sont souvent utilisés avec des modificateurs (également appelés flags) qui redéfinissent le comportement des regex. Les modificateurs Regex peuvent être regular (par exemple /abc/i) et inline (ou embedded) (par exemple (?i)abc). Les modificateurs les plus courants sont les modificateurs globaux, insensibles à la casse, multilignes et dotall. Cependant, les saveurs de regex diffèrent par le nombre de modificateurs de regex pris en charge et leurs types.

Modificateurs PCRE

Modificateur En ligne Descriptif
PCRE_CASELESS (?je) Correspondance insensible à la casse
PCRE_MULTILINE (?m) Correspondance de plusieurs lignes
PCRE_DOTALL (?s) . correspond aux nouvelles lignes
PCRE_ANCHORED (?A) Le méta-caractère ^ correspond uniquement au début
PCRE_EXTENDED (?x) Les espaces blancs sont ignorés
PCRE_DOLLAR_ENDONLY n/a Le méta-caractère $ ne correspond qu’à la fin
PCRE_EXTRA (?X) Analyse d’échappement stricte
PCRE_UTF8 Gère les caractères UTF-8
PCRE_UTF16 Gère les caractères UTF-16
PCRE_UTF32 Gère les caractères UTF-32
PCRE_UNGREEDY (?U) Définit le moteur sur la correspondance paresseuse
PCRE_NO_AUTO_CAPTURE (?:) Désactive les groupes de capture automatique

VERBOSE / COMMENTAIRE / Modificateur IgnorePatternWhitespace #

Le modificateur qui permet d’utiliser des espaces blancs à l’intérieur de certaines parties du motif pour le formater pour une meilleure lisibilité et pour autoriser les commentaires commençant par # :

/(?x)^          # start of string
  (?=\D*\d)     # the string should contain at least 1 digit 
  (?!\d+$)      # the string cannot consist of digits only
  \#            # the string starts with a hash symbol
  [a-zA-Z0-9]+ # the string should have 1 or more alphanumeric symbols
  $             # end of string
/

Exemple de chaîne : #word1here. Notez que le symbole # est échappé pour désigner un # littéral faisant partie d’un motif.

Les espaces blancs non échappés dans le modèle d’expression régulière sont ignorés, échappez-les pour en faire une partie du modèle.

Habituellement, l’espace blanc à l’intérieur des classes de caractères ([...]) est traité comme un espace blanc littéral, sauf en Java.

En outre, il convient de mentionner que dans PCRE, .NET, Python, Ruby Oniguruma, ICU, les saveurs de regex Boost, on peut utiliser des commentaires (?#:...) à l’intérieur du modèle de regex.

Modificateur DOTALL #

Un modèle regex où un modificateur DOTALL (dans la plupart des saveurs regex exprimées avec s) modifie le comportement de . lui permettant de correspondre à un symbole de nouvelle ligne (LF):

/cat (.*?) dog/s

Cette expression régulière de style Perl correspondra à une chaîne telle que "le chat s'est enfui de\na le chien" capturant "s'est enfui de\na" dans le groupe 1.

Une version en ligne : (?s) (par exemple, (?s)cat (.*?) dog)

Remarque : dans Ruby, l’équivalent du modificateur DOTALL est m, [modificateur Regexp::MULTILINE](http://ruby-doc.org/core-2.1.1/Regexp.html#class- Regexp-label-Character+Classes) (par exemple /a.*b/m).

Remarque : JavaScript ne fournit pas de modificateur DOTALL, donc un . ne peut jamais être autorisé à correspondre à un caractère de saut de ligne. Afin d’obtenir le même effet, une solution de contournement est nécessaire, par ex. g. en remplaçant tous les . par une classe de caractères attrape-tout comme [\S\s], ou une classe de caractères pas rien [^] (cependant, cette construction sera traitée comme une erreur par tous d’autres moteurs, et n’est donc pas portable).

Modificateur UNICODE #

Le modificateur UNICODE, généralement exprimé sous la forme u (PHP, Python) ou U (Java), oblige le moteur regex à traiter le modèle et la chaîne d’entrée comme des chaînes et des modèles Unicode, crée des classes abrégées de modèle comme \w , \d, \s, etc. Compatible Unicode.

/\A\p{L}+\z/u

est une expression régulière PHP pour faire correspondre les chaînes composées d’une ou plusieurs lettres Unicode. Voir la démo regex.

Notez que dans [PHP, le modificateur /u][1] permet au moteur PCRE de gérer les chaînes comme des chaînes UTF8 (en activant le verbe PCRE_UTF8) et de rendre les classes de caractères abrégés dans le modèle compatibles Unicode (en activant le verbe PCRE_UCP, voir plus sur pcre.org).

Les chaînes de modèle et de sujet sont traitées comme UTF-8. Ce modificateur est disponible à partir de PHP 4.1.0 ou supérieur sur Unix et à partir de PHP 4.2.3 sur win32. La validité UTF-8 du motif et du sujet est vérifiée depuis PHP 4.3.5. Un sujet invalide fera que la fonction preg_* ne correspondra à rien ; un motif invalide déclenchera une erreur de niveau E_WARNING. Les séquences UTF-8 de cinq et six octets sont considérées comme invalides depuis PHP 5.3.4 (resp. PCRE 7.3 2007-08-28) ; auparavant, ceux-ci étaient considérés comme UTF-8 valides.

En Python 2.x, le [re.UNICODE][2] n’affecte que le motif lui-même : Make \w, \W, \b, \B, \d, \D, \s et \S dépendent de la base de données des propriétés des caractères Unicode.

Une version en ligne : (?u) en Python, (?U) en Java. Par exemple:

print(re.findall(ur"(?u)\w+", u"Dąb")) # [u'D\u0105b']
print(re.findall(r"\w+", u"Dąb"))      # [u'D', u'b']

System.out.println("Dąb".matches("(?U)\\w+")); // true
System.out.println("Dąb".matches("\\w+"));     // false

[1] : http://php.net/manual/en/reference.pcre.pattern.modifiers.php [2] : https://docs.python.org/2/library/re.html#re.UNICODE

Modificateur MULTILIGNE #

Un autre exemple est un modificateur MULTILINE (généralement exprimé avec le drapeau m (pas dans Oniguruma (par exemple Ruby) qui utilise m pour désigner un modificateur DOTALL)) qui fait correspondre les ancres ^ et $ au début / fin de une ligne, pas le début/la fin de toute la chaîne.

/^My Line \d+$/gm

trouvera toutes les lignes qui commencent par “Ma ligne”, puis contiennent un espace et 1+ chiffres jusqu’à la fin de la ligne.

Une version en ligne : (?m) (par exemple, (?m)^Ma ligne \d+$)

REMARQUE : Dans Oniguruma (par exemple dans Ruby), ainsi que dans presque tous les éditeurs de texte prenant en charge les expressions rationnelles, les ancres ^ et $ indiquent les positions de début/fin de ligne par défaut. Vous devez utiliser \A pour définir le début complet du document/chaîne et \z pour indiquer la fin du document/chaîne. La différence entre \Z et \z est que le premier peut correspondre avant le symbole de nouvelle ligne (LF) final à la fin de la chaîne (par exemple, /\Astring\Z/ trouvera une correspondance dans " chaîne\n") (sauf Python, où le comportement \Z est égal à \z et l’ancre \z n’est pas prise en charge).

Modificateur IGNORER CASSE #

Le modificateur commun pour ignorer la casse est i :

/fog/i

correspondra à Fog, foG, etc.

La version en ligne du modificateur ressemble à (?i).

Remarques:

En Java, par défaut, la correspondance [insensible à la casse suppose que seuls les caractères du jeu de caractères US-ASCII sont mis en correspondance. La correspondance insensible à la casse compatible avec Unicode peut être activée en spécifiant l’indicateur UNICODE_CASE conjointement avec cet indicateur (CASE_INSENSITIVE).](http://docs.oracle.com/javase/6/docs/api /java/util/regex/Pattern.html#CASE_INSENSITIVE) (par exemple, Pattern p = Pattern.compile("YOUR_REGEX", Pattern.CASE_INSENSITIVE | Pattern.UNICODE_CASE);). Vous trouverez d’autres informations à ce sujet sur Case-Insensitive Matching in Java RegEx. En outre, [UNICODE_CHARACTER_CLASS][1] peut être utilisé pour rendre la correspondance compatible avec Unicode.

[1] : https://docs.oracle.com/javase/7/docs/api/java/util/regex/Pattern.html#UNICODE_CHARACTER_CLASS

Modificateur de capture explicite #

Il s’agit d’un modificateur spécifique de regex .NET exprimé avec ’n’. Lorsqu’ils sont utilisés, les groupes sans nom (comme (\d+)) ne sont pas capturés. Seules les captures valides sont des groupes explicitement nommés (par exemple, (?<nom> sous-expression)).

(?n)(\d+)-(\w+)-(?<id>\w+)

correspondra à l’ensemble 123-1_abc-00098, mais (\d+) et (\w+) ne créeront pas de groupes dans l’objet match résultant. Le seul groupe sera ${id}. Voir [démo](http://regexstorm.net/tester?p=(%3fn)(%5cd%2b)-(%5cw%2b)-(%3f%3cid%3e%5cw%2b)&i=123 -1_abc-00098&o=ixcs).

modificateur PCRE_DOLLAR_ENDONLY #

Le modificateur PCRE_DOLLAR_ENDONLY conforme à PCRE qui fait correspondre l’ancre $ à la toute fin de la chaîne (à l’exclusion de la position avant la dernière nouvelle ligne dans la chaîne).

/^\d+$/D

est égal à

/^\d+\z/

et correspond à une chaîne entière composée d’un ou plusieurs chiffres et ne correspondra pas à "123\n", mais à "123".

PCRE_UNGREEDY modifier #

L’indicateur PCRE_UNGREEDY conforme à PCRE exprimé avec /U. Il change la gourmandise à l’intérieur d’un motif : /a.*?b/U = /a.*b/ et vice versa.

Modificateur PCRE_INFO_JCHANGED #

Un autre modificateur PCRE qui permet l’utilisation de groupes nommés en double.

REMARQUE : seule la version inline est prise en charge - (?J), et doit être placée au début du motif.

Si tu utilises

/(?J)\w+-(?:new-(?<val>\w+)|\d+-empty-(?<val>[^-]+)-collection)/

les valeurs du groupe “val” ne seront jamais vides (seront toujours définies). Un effet similaire peut cependant être obtenu avec la réinitialisation de la branche.

Modificateur PCRE_EXTRA #

Un modificateur PCRE qui provoque une erreur si une barre oblique inverse dans un modèle est suivie d’une lettre qui n’a pas de signification particulière. Par défaut, une barre oblique inverse suivie d’une lettre sans signification particulière est traitée comme un littéral.

Par exemple.

/big\y/

correspondra à “bigy”, mais

/big\y/X

lèvera une exception.

Version intégrée : (?X)

modificateur PCRE_ANCHORED #

Un autre modificateur conforme à PCRE exprimé avec le modificateur “/A”. Si ce modificateur est défini, le modèle est forcé d’être “ancré”, c’est-à-dire qu’il est contraint de correspondre uniquement au début de la chaîne qui est recherchée (la “chaîne de sujet”). Cet effet peut également être obtenu par des constructions appropriées dans le modèle lui-même, ce qui est le seul moyen de le faire en Perl.

/man/A

est le même que

/^man/
