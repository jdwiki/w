Quels caractères doivent être échappés ? #

L’échappement de caractères est ce qui permet à certains caractères (réservés par le moteur de regex pour manipuler les recherches) d’être littéralement recherchés et trouvés dans la chaîne d’entrée. L’échappement dépend du contexte, donc cet exemple ne couvre pas l’échappement [string][1] ou [delimiter][1].

Barres obliques inverses #

Dire que la barre oblique inverse est le caractère “d’échappement” est un peu trompeur. La barre oblique inverse s’échappe et la barre oblique inverse apporte ; il active ou désactive le métacaractère par rapport au statut littéral du caractère devant lui.

Pour utiliser une barre oblique inverse littérale n’importe où dans une expression régulière, elle doit être échappée par une autre barre oblique inverse.

Échappement (en dehors des classes de personnages) #

Il y a plusieurs caractères qui doivent être échappés pour être pris littéralement (au moins en dehors des classes char):

  • Parenthèses : []
  • Parenthèses : ()
  • Accolades : {}
  • Opérateurs : *, +, ?, |
  • Ancres : ^, $
  • Autres : ., \
  • Pour utiliser un ^ littéral au début ou un $ littéral à la fin d’une regex, le caractère doit être échappé.
  • Certaines variantes utilisent uniquement ^ et $ comme métacaractères lorsqu’ils se trouvent respectivement au début ou à la fin de la regex. Dans ces saveurs, aucune fuite supplémentaire n’est nécessaire. Il est généralement préférable de leur échapper de toute façon.

Échappement dans les classes de personnages #

  • Il est recommandé d’échapper les crochets ([ et ]) lorsqu’ils apparaissent comme des littéraux dans une classe char. Sous certaines conditions, ce n’est [pas obligatoire, selon la saveur] (http://codegolf.stackexchange.com/a/47435), mais cela nuit à la lisibilité.
  • Le caret, ^, est un méta-caractère lorsqu’il est placé comme premier caractère dans une classe char : [^aeiou]. Partout ailleurs dans la classe char, ce n’est qu’un caractère littéral.
  • Le tiret, -, est un méta-caractère, sauf s’il se trouve au début ou à la fin d’une classe de caractères. Si le premier caractère de la classe char est un caret ^, alors ce sera un littéral s’il s’agit du deuxième caractère de la classe char.

Échapper au remplacement #

Il existe également des règles pour s’échapper dans le cadre du remplacement, mais aucune des règles ci-dessus ne s’applique. Les seuls métacaractères sont $ et \, du moins lorsque $ peut être utilisé pour référencer des groupes de capture (comme $1 pour le groupe 1). Pour utiliser un $ littéral, échappez-le : \$5.00. De même \ : C:\\Program Files\\.

Exceptions BRE #

Alors que ERE (expressions régulières étendues) reflète la syntaxe typique de style Perl, BRE (expressions régulières de base) présente des différences significatives en matière d’échappement :

  • Il existe différentes syntaxes abrégées. Tous les \d, \s, \w et ainsi de suite ont disparu. Au lieu de cela, il a sa propre syntaxe (que POSIX appelle de manière confuse “classes de caractères”), comme [:digit:]. Ces constructions doivent appartenir à une classe de caractères.
  • Il y a peu de métacaractères (., *, ^, $) qui peuvent être utilisés normalement. TOUS les autres métacaractères doivent être échappés différemment :

Accolades {}

  • a{1,2} correspond à a{1,2}. Pour faire correspondre a ou aa, utilisez a\{1,2\}

Parenthèses ()

  • (ab)\1 n’est pas valide, car il n’y a pas de groupe de capture 1. Pour le corriger et faire correspondre abab, utilisez \(ab\)\1

Barre oblique inverse

  • À l’intérieur des classes char (appelées expressions entre parenthèses dans POSIX), la barre oblique inverse n’est pas un métacaractère (et n’a pas besoin d’être échappée). [\d] correspond à \ ou d.
  • Partout ailleurs, évadez-vous comme d’habitude.

Autre

  • + et ? sont des littéraux. Si le moteur BRE les prend en charge en tant que métacaractères, ils doivent être échappés sous la forme \? et \+.

[1] : https://www.wikiod.com/fr/regex/sechapper#/Délimiteurs/

Chaînes #

Dans la plupart des langages de programmation, pour avoir une barre oblique inverse dans une chaîne générée à partir d’un littéral de chaîne, chaque barre oblique inverse doit être doublée dans le littéral de chaîne. Sinon, il sera interprété comme un échappement pour le caractère suivant.

Malheureusement, toute barre oblique inverse requise par la regex doit être une barre oblique inverse littérale. C’est pourquoi il devient nécessaire d’avoir des “échappements échappés” (\\) lorsque des regexes sont générés à partir de littéraux de chaîne.

De plus, les guillemets (" ou ') dans le littéral de chaîne peuvent avoir besoin d’être échappés, selon ce qui entoure le littéral de chaîne. Dans certaines langues, il est possible d’utiliser l’un ou l’autre style de guillemets pour une chaîne (choisissez le le plus lisible pour échapper le littéral de chaîne entier).

Dans certains langages (par exemple : Java <=7), les expressions régulières ne peuvent pas être exprimées directement sous forme de littéraux tels que /\w/ ; ils doivent être générés à partir de chaînes, et normalement des littéraux de chaîne sont utilisés - dans ce cas, "\\w". Dans ces cas, les caractères littéraux tels que les guillemets, les barres obliques inverses, etc. doivent être échappés. Le moyen le plus simple d’y parvenir peut être d’utiliser un outil (comme RegexPlanet). Cet outil spécifique est conçu pour Java, mais il fonctionnera pour n’importe quel langage avec une syntaxe de chaîne similaire.

/Délimiteurs/ #

De nombreux langages permettent d’enfermer ou de délimiter les regex entre quelques caractères spécifiques, généralement la barre oblique /.

Les délimiteurs ont un impact sur l’échappement : si le délimiteur est / et que l’expression régulière doit rechercher des littéraux /, alors la barre oblique doit être échappée avant de pouvoir être un littéral (\/).

Une fuite excessive nuit à la lisibilité, il est donc important de considérer les options disponibles :

Javascript est unique car il autorise la barre oblique comme délimiteur, mais rien d’autre (bien qu’il autorise les [regex stringifiées] (http://stackoverflow.com/questions/23438047)).

Perl, par exemple, permet à presque n’importe quoi d’être un délimiteur. Même les caractères arabes :

$str =~ m ش ش

Des règles spécifiques sont mentionnées dans la documentation de Perl.

PCRE autorise deux types de délimiteurs : les délimiteurs correspondants et les délimiteurs de style crochet. Les délimiteurs correspondants utilisent une seule paire de caractères, tandis que les délimiteurs de style crochet utilisent quelques caractères qui représentent une paire d’ouverture et de fermeture.

  • Délimiteurs correspondants : !"#$%&'*+,./:;=?@^_`|~-
  • Délimiteurs de style parenthèse : (), {}, [], <>

Littéraux de chaîne bruts #

Il est préférable pour la lisibilité (et votre santé mentale) d’éviter d’échapper aux fuites. C’est là qu’interviennent les littéraux de chaînes brutes. (Notez que certains langages autorisent les délimiteurs, qui sont généralement préférés aux chaînes. Mais c’est une autre section.)

Ils fonctionnent généralement de la même manière que [cette réponse décrit] (http://stackoverflow.com/a/2081708/6083675):

[Une] barre oblique inverse, \, est considérée comme signifiant “juste une barre oblique inverse” (sauf lorsqu’elle vient juste avant un guillemet qui autrement terminerait le littéral) – pas de “séquences d’échappement” pour représenter les retours à la ligne, les tabulations, les retours arrière, sauts de page, etc.

Toutes les langues n’en ont pas, et celles qui en ont utilisent une syntaxe variable. C# les appelle en fait littéraux de chaîne textuels, mais c’est la même chose.

#Python

pattern = r"regex"

pattern = r'regex'

#C++ (11+)

La syntaxe ici est extrêmement polyvalente. La seule règle est d’utiliser un délimiteur qui n’apparaît nulle part dans la regex. Si vous faites cela, aucun échappement supplémentaire n’est nécessaire pour quoi que ce soit dans la chaîne. Notez que les parenthèses () ne font pas partie de la regex :

pattern = R"delimiter(regex)delimiter";

VB.NET #

Utilisez simplement une chaîne normale. Les barres obliques inverses sont TOUJOURS [littéraux] (http://stackoverflow.com/a/13155483/6083675).

#C#

pattern = @"regex";

Notez que cette syntaxe autorise également "" (deux guillemets doubles) comme forme échappée de ".
