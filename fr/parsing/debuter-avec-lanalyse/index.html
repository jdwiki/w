Un analyseur simple #

La manière la plus simple d’écrire un analyseur est d’utiliser la technique de descente récursive. Cela crée un analyseur descendant (qui peut formellement être décrit comme un LL(1)). Pour commencer l’exemple, nous devons d’abord établir les règles de grammaire de notre langue. Dans cet exemple, nous utiliserons des affectations d’expressions arithmétiques simples pour les expressions qui ne peuvent utiliser que l’opérateur plus :

 Assignment --> Identifier := Expression
 Expression --> Expression + Term | Term
 Term --> Identifier | (Expression)
 Identifier --> x | y | z

Pour chaque règle de la grammaire, nous pouvons écrire une procédure pour reconnaître les jetons entrants dans la règle. Pour les besoins de cet exemple, nous pouvons supposer une routine appelée “NextToken” qui appelle un analyseur lexical pour fournir le jeton, et une routine appelée “error” qui est utilisée pour sortir un message d’erreur. Le langage utilisé est basé sur Pascal.

 var token:char;  (* Updated by NextToken *) 

 procedure identifier;
 begin
    if token in ['x','y','z']
    then
        NextToken
    else
        error('Identifier expected')
 end;

Vous pouvez voir que ce code implémente la règle de reconnaissance d’un Identifier. Nous pouvons ensuite implémenter la règle pour un Term de la même manière :

 procedure expression forward; 

 procedure term;
 begin
     if token = '('
     then
         begin
         nextToken;
         expression;
         if token <> ')'
         then
             error(') expected')
         else NextToken
         end
     else
         identifier
 end;

Lorsque nous arrivons à implémenter la règle pour “Expression”, nous avons un problème ; le premier élément de la règle Expression est lui-même une Expression. Cela nous amènerait à écrire ce qui suit :

procedure expression;
begin
expression;
...
end;

Ceci est directement auto-récursif et bouclerait donc indéfiniment. La grammaire analysée par des algorithmes descendants ne peut pas être récursive à gauche pour cette raison. Un moyen simple de résoudre ce problème consiste à reformuler la récursivité en itération de la manière suivante :

Expression --> Term { + Term}*

Nous pouvons maintenant coder la règle de grammaire comme suit :

 procedure expression;
 begin
     term;
     while token = '+'
         do
         begin
             NextTerm;
             term
         end
 end;

Nous pouvons maintenant terminer l’analyseur avec la règle restante pour l’affectation :

procedure assignment;
begin 
    identifier;
    if token <> '='
    then
        error('= expected')
    else
        begin
        NextToken;
        expression;
        end
 end;

Ce dont vous avez besoin pour l’analyse #

Lors de l’analyse, avant de commencer, la grammaire de la langue doit être spécifiée. Une source de jetons est également nécessaire pour l’analyseur.

L’analyseur peut être un code écrit à la main, ou un outil générateur d’analyseur peut être utilisé. Si un outil générateur d’analyseur est utilisé, cet outil devra être téléchargé et installé s’il n’a pas déjà été inclus dans votre plate-forme.

Définitions grammaticales #

Une grammaire pour un analyseur devrait normalement être écrite dans une forme sans contexte. Une notation telle que BNF (Forme Backus-Naur) ou [EBNF(Extended Back-Naur Form)](https:// en .wikipedia.org/wiki/Extended_Backus%E2%80%93Naur_Form) est souvent utilisé pour cela. D’autres notations couramment utilisées pour décrire les langages de programmation peuvent être [diagrammes ferroviaires] (https://en.wikipedia.org/wiki/Syntax_diagram).

Analyse lexicale #

Les jetons sont normalement fournis à l’analyseur par un analyseur lexical (ou scanner). Plus de détails peuvent être trouvés dans la documentation d’un analyseur lexical (TBC).

Techniques d’analyse #

Pour coder manuellement un analyseur, il faudrait choisir un algorithme approprié qui convienne à la fois au langage analysé et aux moyens de mise en œuvre. Les algorithmes d’analyse sont classés dans les deux types d’analyse descendante et [analyse ascendante](https://en.wikipedia.org/ wiki/Analyse ascendante). Un analyseur descendant (récursif) est plus facile à apprendre pour un débutant lorsqu’il commence à écrire des analyseurs.

Outils de générateur d’analyseur #

La manière la plus courante de créer un analyseur consiste à utiliser un outil générateur d’analyseur. Il existe de nombreux outils de ce type, mais certains des plus couramment utilisés sont :

  • Bison/yacc
  • ANTLR

Exemple d’analyse d’une phrase en anglais #

Par exemple, dans la phrase :

Ce gâteau est extrêmement bon.

Les règles de la langue anglaise feraient de cake un nom, extrêmement un adverbe qui modifie l’adjectif nice, et grâce à cette analyse, le sens pourrait être compris.

Cependant, cette analyse dépend de notre reconnaissance que la séquence de symboles utilisés sont des mots. Si les caractères utilisés ne nous étaient pas familiers, nous ne pourrions pas le faire. Si nous rencontrions une phrase utilisant une notation inconnue, comme le chinois, l’analyse de cette manière pourrait être difficile. Voici un exemple de phrase chinoise :

Je sais écrire un peu de chinois.

Pour quiconque ne lit pas les caractères chinois, il ne serait pas clair quels symboles combinés pour former des mots. La même chose pourrait être vraie pour un algorithme informatique lors du traitement de l’anglais ou du chinois.

Ainsi, l’analyse doit être effectuée par un processus connu sous le nom d’analyse lexicale ou balayage, où les caractères individuels sont regroupés en symboles reconnus, que nous pourrions communément appeler des mots, mais dans les algorithmes d’analyse sont appelés jetons.
