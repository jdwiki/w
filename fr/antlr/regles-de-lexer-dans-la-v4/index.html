Règles de priorité #

Plusieurs règles d’analyseur lexical peuvent correspondre au même texte d’entrée. Dans ce cas, le type de jeton sera choisi comme suit :

  • Tout d’abord, sélectionnez la règle de lexer qui correspond à l’entrée la plus longue
  • Si le texte correspond à un jeton implicitement défini (comme '{'), utilisez la règle implicite
  • Si plusieurs règles de lexer correspondent à la même longueur d’entrée, choisissez la première, en fonction de l’ordre de définition

La grammaire combinée suivante :

grammar LexerPriorityRulesExample;

// Parser rules

randomParserRule: 'foo'; // Implicitly declared token type
    
// Lexer rules
    
BAR: 'bar';
IDENTIFIER: [A-Za-z]+;
BAZ: 'baz';

WS: [ \t\r\n]+ -> skip;

Étant donné l’entrée suivante :

aaa foo bar baz barz

Produira la séquence de jetons suivante à partir du lexer :

IDENTIFIER 'foo' BAR IDENTIFIER IDENTIFIER
  • aaa est de type IDENTIFIER

Seule la règle “IDENTIFIER” peut correspondre à ce jeton, il n’y a pas d’ambiguïté.

  • foo est de type 'foo'

La règle d’analyse randomParserRule introduit le type de jeton implicite 'foo', qui est prioritaire sur la règle IDENTIFIER.

  • bar est de type BAR

Ce texte correspond à la règle BAR, qui est définie avant la règle IDENTIFIER, et a donc la priorité.

  • baz est de type IDENTIFIER

Ce texte correspond à la règle “BAZ”, mais il correspond également à la règle “IDENTIFIER”. Ce dernier est choisi tel qu’il est défini avant BAR.

Compte tenu de la grammaire, BAZ ne pourra jamais correspondre, car la règle IDENTIFIER couvre déjà tout ce que BAZ peut correspondre.

  • barz est de type IDENTIFIER

La règle BAR peut correspondre aux 3 premiers caractères de cette chaîne (bar), mais la règle IDENTIFIER correspondra à 4 caractères. Comme IDENTIFIER correspond à une sous-chaîne plus longue, il est choisi plutôt que BAR.

En règle générale, des règles spécifiques doivent être définies * avant * des règles plus génériques. Si une règle ne peut correspondre qu’à une entrée qui est déjà couverte par une règle précédemment définie, elle ne sera jamais utilisée.

Les règles définies implicitement telles que 'foo' agissent comme si elles étaient définies avant toutes les autres règles de lexer.

Règles de lexer implicites #

Lorsque des jetons comme '{' sont utilisés dans une règle parser, une règle de lexer implicite sera créée pour eux à moins qu’une règle explicite n’existe.

En d’autres termes, si vous avez une règle lexer :

OPEN_BRACE: '{';

Alors ces deux règles d’analyseur sont équivalentes :

parserRule: '{';
parserRule: OPEN_BRACE;

Mais si la règle d’analyseur lexical OPEN_BRACE n’est pas définie, une règle anonyme implicite sera créée. Dans ce cas, la règle implicite sera définie comme si elle était définie avant les autres règles : elle aura une priorité plus élevée que les autres règles.

Actions et prédicats sémantiques #

Une action lexer est un bloc de code arbitraire dans le langage cible entouré de {…}, qui est exécuté lors de la correspondance :

IDENTIFIER: [A-Z]+ { log("matched rule"); };

Un prédicat sémantique est un bloc de code arbitraire dans la langue cible entouré de {…}?, qui s’évalue en une valeur booléenne. Si la valeur renvoyée est fausse, la règle d’analyseur lexical est ignorée.

IDENTIFIER: [A-Z]+ { identifierIsValid() }?;

Les prédicats sémantiques doivent être définis à la fin de la règle chaque fois que possible pour des raisons de performances.

Règles simples #

Les règles Lexer définissent les types de jetons. Leur nom doit commencer par une lettre majuscule pour les distinguer des règles de l’analyseur.

INTEGER: [0-9]+;
IDENTIFIER: [a-zA-Z_] [a-zA-Z_0-9]*;

OPEN_PAREN: '(';
CLOSE_PAREN: ')';

Syntaxe de base :

Syntaxe Sens
A Correspond à la règle de lexer ou au fragment nommé “A”
A B Faites correspondre A suivi de B
(A|B) Faites correspondre A ou B
'texte' Correspondance littérale “texte”
A ? Correspondre A zéro ou une fois
A* Faire correspondre A zéro ou plusieurs fois
A+ Faites correspondre A une ou plusieurs fois
[A-Z0-9] Faites correspondre un caractère dans les plages définies (dans cet exemple entre A-Z ou 0-9)
'a'..'z' Syntaxe alternative pour une plage de caractères
~[A-Z] Négation d’une plage - correspond à n’importe quel caractère unique pas dans la plage
. Faites correspondre n’importe quel caractère unique

Fragments #

Les fragments sont des parties réutilisables des règles de lexer qui ne peuvent pas correspondre par elles-mêmes - elles doivent être référencées à partir d’une règle de lexer.

INTEGER: DIGIT+
       | '0' [Xx] HEX_DIGIT+
       ;

fragment DIGIT: [0-9];
fragment HEX_DIGIT: [0-9A-Fa-f];

Commandes de l’analyseur lexical #

Une règle d’analyseur lexical peut avoir des commandes associées :

WHITESPACE: [ \r\n] -> skip;

Les commandes sont définies après un -> à la fin de la règle.

  • skip : Ignore le texte correspondant, aucun jeton ne sera émis
  • channel(n) : Émet le jeton sur un canal différent
  • type(n) : change le type de jeton émis
  • mode(n), pushMode(n), popMode, more : contrôle les modes de lexer
