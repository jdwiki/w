JOIN est une méthode pour combiner (joindre) les informations de deux tables. Le résultat est un ensemble assemblé de colonnes des deux tables, défini par le type de jointure (INNER/OUTER/CROSS et LEFT/RIGHT/FULL, expliqué ci-dessous) et les critères de jointure (comment les lignes des deux tables sont liées).

Une table peut être jointe à elle-même ou à toute autre table. Si des informations provenant de plus de deux tables doivent être accessibles, plusieurs jointures peuvent être spécifiées dans une clause FROM.

Syntaxe #

  • [ { INTÉRIEUR | { { GAUCHE | DROITE | COMPLET } [ OUTER ] } } ] JOIN

Les jointures, comme leur nom l’indique, sont un moyen d’interroger les données de plusieurs tables de manière conjointe, les lignes affichant des colonnes provenant de plusieurs tables.

Rejoindre soi-même #

Une table peut être jointe à elle-même, avec différentes lignes correspondant les unes aux autres par une condition. Dans ce cas d’utilisation, des alias doivent être utilisés afin de distinguer les deux occurrences de la table.

Dans l’exemple ci-dessous, pour chaque Employé dans la [table d’Employés de la base de données d’exemple][1], un enregistrement est renvoyé contenant le prénom de l’employé ainsi que le prénom correspondant du responsable de l’employé. Comme les managers sont aussi des employés, le tableau est joint à lui-même :

SELECT 
    e.FName AS "Employee", 
    m.FName AS "Manager"
FROM   
    Employees e
JOIN   
    Employees m 
    ON e.ManagerId = m.Id

Cette requête renverra les données suivantes :

Employé Gérant
Jean Jacques
Michel Jacques
Jonathan Jean

Donc comment ça fonctionne? #

La table d’origine contient ces enregistrements :

| identifiant | FName | LName | Numéro de téléphone | ID du gestionnaire | ID de service | Salaire | Date d’embauche | |:—|:———-|:——–|:————|:—— —-| :————| :——-| :———–| | 1 | Jacques | forgeron | 1234567890 | NUL | 1 | 1000 | 01-01-2002 | | 2 | Jean | Johnson | 2468101214 | 1 | 1 | 400 | 23-03-2005 | | 3 | Michel | Williams | 1357911131 | 1 | 2 | 600 | 12-05-2009 | | 4| Jonathan | forgeron | 1212121212 | 2| 1| 500| 24-07-2016 |

La première action consiste à créer un produit cartésien de tous les enregistrements des tables utilisées dans la clause FROM. Dans ce cas, c’est la table Employés deux fois, donc la table intermédiaire ressemblera à ceci (j’ai supprimé tous les champs non utilisés dans cet exemple) :

| e.Id | e.FName | e.ManagerId | m.Id | m.FName | m.ManagerId | |:—–|:———-|:———–|:—–|:——– –| :————| | 1 | Jacques | NUL | 1 | Jacques | NUL | | 1 | Jacques | NUL | 2 | Jean | 1 | | 1 | Jacques | NUL | 3 | Michel | 1 | | 1 | Jacques | NUL | 4 | Johnathon | 2 | | 2 | Jean | 1 | 1 | Jacques | NUL | | 2 | Jean | 1 | 2 | Jean | 1 | | 2 | Jean | 1 | 3 | Michel | 1 | | 2 | Jean | 1| 4 | Jonathan | 2 | | 3 | Michel | 1 | 1 | Jacques | NUL | | 3 | Michel | 1 | 2 | Jean | 1 | | 3 | Michel | 1 | 3 | Michel | 1 | | 3 | Michel | 1 | 4 | Johnathon | 2 | | 4 | Johnathon | 2 | 1 | Jacques | NUL | | 4 | Jonathan | 2 | 2 | Jean | 1| | 4 | Johnathon | 2 | 3 | Michel | 1 | | 4 | Johnathon | 2 | 4 | Johnathon | 2 |

L’action suivante consiste à ne conserver que les enregistrements qui répondent aux critères JOIN, donc tous les enregistrements où la table e aliasée ManagerId est égale à la table m aliasée Id :

e.Id e.FName e.ManagerId m.Id m.FName m.ManagerId
2 Jean 1 1 Jacques NUL
3 Michel 1 1 Jacques NUL
4 Jonathan 2 2 Jean 1

Ensuite, chaque expression utilisée dans la clause SELECT est évaluée pour renvoyer cette table :

e.FName m.FName
Jean Jacques
Michel Jacques
Jonathan Jean

Enfin, les noms de colonne e.FName et m.FName sont remplacés par leurs noms de colonne d’alias, attribués avec l’opérateur [AS][2] :

Employé Gérant
Jean Jacques
Michel Jacques
Jonathan Jean

Différences entre les jointures internes/externes #

SQL a différents types de jointure pour spécifier si des lignes (non-)correspondantes sont incluses dans le résultat : INNER JOIN, LEFT OUTER JOIN, RIGHT OUTER JOIN et FULL OUTER JOIN (les INNER et Les mots-clés OUTER sont facultatifs). La figure ci-dessous souligne les différences entre ces types de jointures : la zone bleue représente les résultats renvoyés par la jointure, et la zone blanche représente les résultats que la jointure ne renverra pas.

[![Diagrammes de Venn représentant les jointures SQL internes/externes][1]][1]

Cross Join SQL Pictorial Presentation ([référence][2]) :

[![entrez la description de l’image ici][3]][2]

Vous trouverez ci-dessous des exemples de [cette] [4] réponse.

Par exemple, il y a deux tables comme ci-dessous :

A    B
-    -
1    3
2    4
3    5
4    6

Notez que (1,2) sont uniques à A, (3,4) sont communs et (5,6) sont uniques à B.

Jointure interne #

Une jointure interne utilisant l’une ou l’autre des requêtes équivalentes donne l’intersection des deux tables, c’est-à-dire les deux lignes qu’elles ont en commun :

select * from a INNER JOIN b on a.a = b.b;
select a.*,b.* from a,b where a.a = b.b;

a | b
--+--
3 | 3
4 | 4

Jointure externe gauche #

Une jointure externe gauche donnera toutes les lignes de A, plus toutes les lignes communes de B :

select * from a LEFT OUTER JOIN b on a.a = b.b;

a |  b
--+-----
1 | null
2 | null
3 |    3
4 |    4

Jointure externe droite #

De même, une jointure externe droite donnera toutes les lignes de B, plus toutes les lignes communes de A :

select * from a RIGHT OUTER JOIN b on a.a = b.b;

a    |  b
-----+----
3    |  3
4    |  4
null |  5
null |  6

Jointure externe complète #

Une jointure externe complète vous donnera l’union de A et B, c’est-à-dire toutes les lignes de A et toutes les lignes de B. Si quelque chose dans A n’a pas de donnée correspondante dans B, alors la partie B est nulle, et vice versa.

select * from a FULL OUTER JOIN b on a.a = b.b;

 a   |  b
-----+-----
   1 | null
   2 | null
   3 |    3
   4 |    4
null |    6
null |    5

JOIN Terminologie : Intérieur, Extérieur, Semi, Anti… #

Supposons que nous ayons deux tables (A et B) et que certaines de leurs lignes correspondent (par rapport à la condition JOIN donnée, quelle qu’elle soit dans le cas particulier) :

[![Join Terminology Overview][1]][1]

Nous pouvons utiliser différents types de jointure pour inclure ou exclure des lignes correspondantes ou non correspondantes de chaque côté, et nommer correctement la jointure en choisissant les termes correspondants dans le diagramme ci-dessus.

Les exemples ci-dessous utilisent les données de test suivantes :

CREATE TABLE A (
    X varchar(255) PRIMARY KEY
);

CREATE TABLE B (
    Y varchar(255) PRIMARY KEY
);

INSERT INTO A VALUES
    ('Amy'),
    ('John'),
    ('Lisa'),
    ('Marco'),
    ('Phil');

INSERT INTO B VALUES
    ('Lisa'),
    ('Marco'),
    ('Phil'),
    ('Tim'),
    ('Vincent');

Jointure interne #

Combine les lignes gauche et droite qui correspondent.

[![Jointure interne][2]][2]

SELECT * FROM A JOIN B ON X = Y;

X      Y
------ -----
Lisa   Lisa
Marco  Marco
Phil   Phil

Jointure externe gauche #

Parfois abrégé en “jointure gauche”. Combine les lignes de gauche et de droite qui correspondent et inclut les lignes de gauche qui ne correspondent pas.

[![Jointure externe gauche][3]][3]

SELECT * FROM A LEFT JOIN B ON X = Y;

X      Y
-----  -----
Amy    NULL
John   NULL
Lisa   Lisa
Marco  Marco
Phil   Phil

Jointure externe droite #

Parfois abrégé en “joindre à droite”. Combine les lignes de gauche et de droite qui correspondent et inclut les lignes de droite qui ne correspondent pas.

[![Jointure externe droite][4]][4]

SELECT * FROM A RIGHT JOIN B ON X = Y;

X      Y
-----  -------
Lisa   Lisa
Marco  Marco
Phil   Phil
NULL   Tim
NULL   Vincent

Jointure externe complète #

Parfois abrégé en “jointure complète”. Union des jointures externes gauche et droite.

[![Jointure externe complète][5]][5]

SELECT * FROM A FULL JOIN B ON X = Y;

X      Y
-----  -------
Amy    NULL
John   NULL
Lisa   Lisa
Marco  Marco
Phil   Phil
NULL   Tim
NULL   Vincent

Semi-jointure gauche #

Inclut les lignes de gauche qui correspondent aux lignes de droite.

[![Semi-jointure gauche][6]][6]

SELECT * FROM A WHERE X IN (SELECT Y FROM B);

X
-----
Lisa
Marco
Phil

Semi-jointure droite #

Inclut les lignes de droite qui correspondent aux lignes de gauche.

[![Right Semi Join][7]][7]

SELECT * FROM B WHERE Y IN (SELECT X FROM A);

Y
-----
Lisa
Marco
Phil

Comme vous pouvez le voir, il n’y a pas de syntaxe IN dédiée pour la semi-jointure gauche ou droite - nous obtenons l’effet simplement en changeant les positions de la table dans le texte SQL.

Anti-semi jointure gauche #

Inclut les lignes de gauche qui ne correspondent pas aux lignes de droite.

[![Gauche Anti Semi Join][8]][8]

SELECT * FROM A WHERE X NOT IN (SELECT Y FROM B);

X
----
Amy
John

AVERTISSEMENT : Soyez prudent si vous utilisez NOT IN sur une colonne compatible NULL ! Plus de détails [ici][9].

Droite Anti Semi Join #

Inclut les lignes de droite qui ne correspondent pas aux lignes de gauche.

[![Right Anti Semi Join][10]][10]

SELECT * FROM B WHERE Y NOT IN (SELECT X FROM A);

Y
-------
Tim
Vincent

Comme vous pouvez le voir, il n’y a pas de syntaxe NOT IN dédiée pour l’anti-semi-jointure gauche vs droite - nous obtenons l’effet simplement en changeant les positions de la table dans le texte SQL.

Jointure croisée #

Un produit cartésien de toutes les lignes de gauche avec toutes les lignes de droite.

SELECT * FROM A CROSS JOIN B;

X      Y
-----  -------
Amy    Lisa
John   Lisa
Lisa   Lisa
Marco  Lisa
Phil   Lisa
Amy    Marco
John   Marco
Lisa   Marco
Marco  Marco
Phil   Marco
Amy    Phil
John   Phil
Lisa   Phil
Marco  Phil
Phil   Phil
Amy    Tim
John   Tim
Lisa   Tim
Marco  Tim
Phil   Tim
Amy    Vincent
John   Vincent
Lisa   Vincent
Marco  Vincent
Phil   Vincent

La jointure croisée est équivalente à une jointure interne avec une condition de jointure qui correspond toujours, donc la requête suivante aurait renvoyé le même résultat :

SELECT * FROM A JOIN B ON 1 = 1;

Auto-adhésion #

Cela dénote simplement une table se joignant à elle-même. Une jointure réflexive peut être l’un des types de jointure décrits ci-dessus. Par exemple, il s’agit d’une auto-jointure interne :

SELECT * FROM A A1 JOIN A A2 ON LEN(A1.X) < LEN(A2.X);

X     X
----  -----
Amy   John
Amy   Lisa
Amy   Marco
John  Marco
Lisa  Marco
Phil  Marco
Amy   Phil

Jointure externe gauche #

Une jointure externe gauche (également appelée jointure gauche ou jointure externe) est une jointure qui garantit que toutes les lignes de la table de gauche sont représentées ; s’il n’existe aucune ligne correspondante dans la table de droite, ses champs correspondants sont NULL.

L’exemple suivant sélectionnera tous les départements et le prénom des employés qui travaillent dans ce département. Les départements sans employés sont toujours renvoyés dans les résultats, mais auront NULL pour le nom de l’employé :

SELECT          Departments.Name, Employees.FName
FROM            Departments 
LEFT OUTER JOIN Employees 
ON              Departments.Id = Employees.DepartmentId

Cela renverrait ce qui suit à partir de la [exemple de base de données][1] :

Départements.Nom Employés.FName
RH Jacques
RH Jean
, RH Johnathon
Ventes Michel
Technologie NUL

Donc comment ça fonctionne? #

Il y a deux tables dans la clause FROM :

| identifiant | FName | LName | Numéro de téléphone | ID du gestionnaire | ID de service | Salaire | Date d’embauche | |:—|:———-|:——–|:————|:—— —-| :————| :——-| :———–| | 1 | Jacques | forgeron | 1234567890 | NUL | 1 | 1000 | 01-01-2002 | | 2 | Jean | Johnson | 2468101214 | 1 | 1 | 400 | 23-03-2005 | | 3 | Michel | Williams | 1357911131 | 1 | 2 | 600 | 12-05-2009 | | 4| Jonathan | forgeron | 1212121212 | 2| 1| 500| 24-07-2016 |

et

identifiant Nom
1 RH
2 Ventes
3 Technologie

Un produit cartésien est d’abord créé à partir des deux tables donnant une table intermédiaire. Les enregistrements qui répondent aux critères de jointure (Departments.Id = Employees.DepartmentId) sont mis en évidence en gras ; ceux-ci sont transmis à l’étape suivante de la requête.

Comme il s’agit d’une JOINTURE EXTERNE GAUCHE, tous les enregistrements sont renvoyés du côté GAUCHE de la jointure (Départements), tandis que tous les enregistrements du côté DROIT reçoivent un marqueur NULL s’ils ne correspondent pas aux critères de jointure. Dans le tableau ci-dessous, cela renverra Tech avec NULL

| identifiant | Nom | identifiant | FName | LName | Numéro de téléphone | ID du gestionnaire | ID de service | Salaire | Date d’embauche | |:——|:———-|:——|:——|:—- ———|:—————|:———-|:———- —|:———|:—————| | 1 | RH | 1 | James | Smith | 1234567890 | NUL | 1 | 1000 | 01-01-2002 | | 1 | RH | 2 | Jean | Johnson | 2468101214 | 1 | 1 | 400 | 23-03-2005 | | 1 | RH | 3 | Michel | Williams | 1357911131 | 1 | 2 | 600 | 12-05-2009 | | 1 | RH | 4 | Johnathon | Smith | 1212121212 | 2 | 1 | 500 | 24-07-2016 | | 2 | Ventes | 1 | Jacques | forgeron | 1234567890 | NUL | 1 | 1000 | 01-01-2002 | | 2 | Ventes | 2 | Jean | Johnson | 2468101214 | 1 | 1 | 400 | 23-03-2005 | | 2 | Ventes | 3 | Michel | Williams | 1357911131 | 1 | 2 | 600 | 12-05-2009 | | 2 | Ventes | 4 | Johnathon | forgeron | 1212121212 | 2 | 1 | 500 | 24-07-2016 | | 3 | Technologie | 1 | Jacques | forgeron | 1234567890 | NUL | 1 | 1000 | 01-01-2002 | | 3 | Technologie | 2 | Jean | Johnson | 2468101214 | 1 | 1 | 400 | 23-03-2005 | | 3 | Technologie | 3 | Michel | Williams | 1357911131 | 1 | 2 | 600 | 12-05-2009 | | 3 | Technologie | 4 | Johnathon | forgeron | 1212121212 | 2 | 1 | 500 | 24-07-2016 |

Enfin, chaque expression utilisée dans la clause SELECT est évaluée pour renvoyer notre table finale :

Départements.Nom Employés.FName
RH Jacques
RH Jean
Ventes Richard
Technologie NUL

Jointure implicite #

Les jointures peuvent également être effectuées en ayant plusieurs tables dans la clause from, séparées par des virgules , et en définissant la relation entre elles dans la clause where. Cette technique s’appelle une jointure implicite (puisqu’elle ne contient pas réellement de clause join).

Tous les SGBDR le supportent, mais la syntaxe est généralement déconseillée. Les raisons pour lesquelles il est déconseillé d’utiliser cette syntaxe sont :

  • Il est possible d’obtenir des jointures croisées accidentelles qui renvoient alors des résultats incorrects, surtout si vous avez beaucoup de jointures dans la requête.
  • Si vous vouliez une jointure croisée, cela n’est pas clair dans la syntaxe (écrivez CROSS JOIN à la place), et quelqu’un est susceptible de le modifier pendant la maintenance.

L’exemple suivant sélectionnera les prénoms des employés et le nom des services pour lesquels ils travaillent :

SELECT e.FName, d.Name
FROM   Employee e, Departments d
WHERE  e.DeptartmentId = d.Id

Cela renverrait ce qui suit à partir de la [exemple de base de données][1] :

e.FName d.Nom
Jacques RH
Jean RH
Richard Ventes

[1] : https://www.wikiod.com/fr/sql/exemples-de-bases-de-donnees-et-de-tables

Jointure interne explicite de base #

Une jointure de base (également appelée “jointure interne”) interroge les données de deux tables, avec leur relation définie dans une clause join.

L’exemple suivant sélectionne les prénoms des employés (FName) dans la table Employees et le nom du service pour lequel ils travaillent (Name) dans la table Departments :

SELECT Employees.FName, Departments.Name
FROM   Employees
JOIN   Departments
ON Employees.DepartmentId = Departments.Id

Cela renverrait ce qui suit à partir de la [exemple de base de données][1] :

Employés.FName Départements.Nom
James RH
Jean RH
Richard Ventes

JOINTURE CROISÉE #

La jointure croisée fait un produit cartésien des deux membres, un produit cartésien signifie que chaque ligne d’une table est combinée avec chaque ligne de la deuxième table dans la jointure. Par exemple, si TABLEA a 20 lignes et TABLEB a 20 lignes, le résultat serait 20*20 = 400 lignes de sortie.

Utilisation de [exemple de base de données][1]

SELECT d.Name, e.FName
FROM   Departments d
CROSS JOIN Employees e;

Qui renvoie :

d.Nom e.FName
RH James
RH John
HR Michael
RH jonathon
Sales James
Sales John
Sales Michael
Ventes Johnathon
Tech James
Tech John
Tech Michael
Technologie Johnathon

Il est recommandé d’écrire un CROSS JOIN explicite si vous voulez faire une jointure cartésienne, pour souligner que c’est ce que vous voulez.

Joindre une sous-requête #

Joindre une sous-requête est souvent utilisé lorsque vous souhaitez obtenir des données agrégées à partir d’une table enfant/détails et les afficher avec les enregistrements de la table parent/en-tête. Par exemple, vous souhaiterez peut-être obtenir le nombre d’enregistrements enfants, la moyenne d’une colonne numérique dans les enregistrements enfants ou la ligne supérieure ou inférieure en fonction d’une date ou d’un champ numérique. Cet exemple utilise des alias, ce qui facilite la lecture des requêtes lorsque plusieurs tables sont impliquées. Voici à quoi ressemble une jointure de sous-requête assez typique. Dans ce cas, nous récupérons toutes les lignes de la table parent Purchase Orders et récupérons uniquement la première ligne pour chaque enregistrement parent de la table enfant PurchaseOrderLineItems.

SELECT po.Id, po.PODate, po.VendorName, po.Status, item.ItemNo, 
  item.Description, item.Cost, item.Price
FROM PurchaseOrders po
LEFT JOIN 
     (
       SELECT l.PurchaseOrderId, l.ItemNo, l.Description, l.Cost, l.Price, Min(l.id) as Id 
       FROM PurchaseOrderLineItems l
       GROUP BY l.PurchaseOrderId, l.ItemNo, l.Description, l.Cost, l.Price
     ) AS item ON item.PurchaseOrderId = po.Id

APPLIQUE CROISÉE ET JOINT LATÉRAL #

Un type très intéressant de JOIN est le LATERAL JOIN (nouveau dans PostgreSQL 9.3+),
qui est également connu sous le nom CROSS APPLY/OUTER APPLY dans SQL-Server & Oracle.

L’idée de base est qu’une fonction table (ou une sous-requête en ligne) est appliquée pour chaque ligne que vous joignez.

Cela permet, par exemple, de joindre uniquement la première entrée correspondante dans une autre table.
La différence entre une jointure normale et une jointure latérale réside dans le fait que vous pouvez utiliser une colonne que vous avez précédemment jointe dans la sous-requête que vous “CROSS APPLY”.

Syntaxe:

PostgreSQL 9.3+

gauche | droite | JOINT intérieur LATERAL

Serveur SQL:

CROIX | EXTÉRIEUR APPLIQUER

INNER JOIN LATERAL est identique à CROSS APPLY
et “LEFT JOIN LATERAL” est identique à “OUTER APPLY”

Exemple d’utilisation (PostgreSQL 9.3+) :

SELECT * FROM T_Contacts 

--LEFT JOIN T_MAP_Contacts_Ref_OrganisationalUnit ON MAP_CTCOU_CT_UID = T_Contacts.CT_UID AND MAP_CTCOU_SoftDeleteStatus = 1 
--WHERE T_MAP_Contacts_Ref_OrganisationalUnit.MAP_CTCOU_UID IS NULL -- 989


LEFT JOIN LATERAL 
(
    SELECT 
         --MAP_CTCOU_UID    
         MAP_CTCOU_CT_UID   
        ,MAP_CTCOU_COU_UID  
        ,MAP_CTCOU_DateFrom 
        ,MAP_CTCOU_DateTo   
   FROM T_MAP_Contacts_Ref_OrganisationalUnit 
   WHERE MAP_CTCOU_SoftDeleteStatus = 1 
   AND MAP_CTCOU_CT_UID = T_Contacts.CT_UID 

    /*  
    AND 
    ( 
        (__in_DateFrom <= T_MAP_Contacts_Ref_OrganisationalUnit.MAP_KTKOE_DateTo) 
        AND 
        (__in_DateTo >= T_MAP_Contacts_Ref_OrganisationalUnit.MAP_KTKOE_DateFrom) 
    ) 
    */
   ORDER BY MAP_CTCOU_DateFrom 
   LIMIT 1 
) AS FirstOE

Et pour SQL-Server

SELECT * FROM T_Contacts 

--LEFT JOIN T_MAP_Contacts_Ref_OrganisationalUnit ON MAP_CTCOU_CT_UID = T_Contacts.CT_UID AND MAP_CTCOU_SoftDeleteStatus = 1 
--WHERE T_MAP_Contacts_Ref_OrganisationalUnit.MAP_CTCOU_UID IS NULL -- 989

-- CROSS APPLY -- = INNER JOIN 
OUTER APPLY    -- = LEFT JOIN 
(
    SELECT TOP 1 
         --MAP_CTCOU_UID    
         MAP_CTCOU_CT_UID   
        ,MAP_CTCOU_COU_UID  
        ,MAP_CTCOU_DateFrom 
        ,MAP_CTCOU_DateTo   
   FROM T_MAP_Contacts_Ref_OrganisationalUnit 
   WHERE MAP_CTCOU_SoftDeleteStatus = 1 
   AND MAP_CTCOU_CT_UID = T_Contacts.CT_UID 

    /*  
    AND 
    ( 
        (@in_DateFrom <= T_MAP_Contacts_Ref_OrganisationalUnit.MAP_KTKOE_DateTo) 
        AND 
        (@in_DateTo >= T_MAP_Contacts_Ref_OrganisationalUnit.MAP_KTKOE_DateFrom) 
    ) 
    */
   ORDER BY MAP_CTCOU_DateFrom 
) AS FirstOE

JOINTURE COMPLÈTE #

Un type de JOIN moins connu est le FULL JOIN.
(Remarque : FULL JOIN n’est pas pris en charge par MySQL depuis 2016)

Une FULL OUTER JOIN renvoie toutes les lignes de la table de gauche et toutes les lignes de la table de droite.

S’il y a des lignes dans le tableau de gauche qui n’ont pas de correspondances dans le tableau de droite, ou s’il y a des lignes dans le tableau de droite qui n’ont pas de correspondances dans le tableau de gauche, ces lignes seront également répertoriées.

Exemple 1 :

SELECT * FROM Table1

FULL JOIN Table2 
     ON 1 = 2

Exemple 2 :

SELECT 
     COALESCE(T_Budget.Year, tYear.Year) AS RPT_BudgetInYear 
    ,COALESCE(T_Budget.Value, 0.0) AS RPT_Value 
FROM T_Budget 

FULL JOIN tfu_RPT_All_CreateYearInterval(@budget_year_from, @budget_year_to) AS tYear 
      ON tYear.Year = T_Budget.Year

Notez que si vous utilisez des suppressions réversibles, vous devrez vérifier à nouveau le statut de suppression réversible dans la clause WHERE (car FULL JOIN se comporte un peu comme un UNION);
Il est facile d’oublier ce petit fait, puisque vous mettez AP_SoftDeleteStatus = 1 dans la clause de jointure.

De plus, si vous faites un FULL JOIN, vous devrez généralement autoriser NULL dans la clause WHERE ; oublier d’autoriser NULL sur une valeur aura les mêmes effets qu’une jointure INNER, ce que vous ne voulez pas si vous faites une FULL JOIN.

Exemple:

SELECT 
     T_AccountPlan.AP_UID
    ,T_AccountPlan.AP_Code
    ,T_AccountPlan.AP_Lang_EN
    ,T_BudgetPositions.BUP_Budget
    ,T_BudgetPositions.BUP_UID 
    ,T_BudgetPositions.BUP_Jahr
FROM T_BudgetPositions    

FULL JOIN T_AccountPlan
    ON T_AccountPlan.AP_UID = T_BudgetPositions.BUP_AP_UID 
    AND T_AccountPlan.AP_SoftDeleteStatus = 1 

WHERE (1=1) 
AND (T_BudgetPositions.BUP_SoftDeleteStatus = 1 OR T_BudgetPositions.BUP_SoftDeleteStatus IS NULL) 
AND (T_AccountPlan.AP_SoftDeleteStatus = 1 OR T_AccountPlan.AP_SoftDeleteStatus IS NULL)

JOIN récursifs #

Les jointures récursives sont souvent utilisées pour obtenir des données parent-enfant. En SQL, ils sont implémentés avec des [expressions de table communes] récursives (https://www.wikiod.com/fr/sql/expressions-de-table-courantes), par exemple :

WITH RECURSIVE MyDescendants AS (
    SELECT Name
    FROM People
    WHERE Name = 'John Doe'

    UNION ALL

    SELECT People.Name
    FROM People
    JOIN MyDescendants ON People.Name = MyDescendants.Parent
)
SELECT * FROM MyDescendants;
