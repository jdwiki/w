Les index sont une structure de données qui contient des pointeurs vers le contenu d’une table disposés dans un ordre spécifique, pour aider la base de données à optimiser les requêtes. Ils sont similaires à l’index de livre, où les pages (lignes du tableau) sont indexées par leur numéro de page.

Plusieurs types d’index existent, et peuvent être créés sur une table. Lorsqu’un index existe sur les colonnes utilisées dans la clause WHERE, la clause JOIN ou la clause ORDER BY d’une requête, il peut améliorer considérablement les performances de la requête.

Les index sont un moyen d’accélérer les requêtes de lecture en triant les lignes d’une table en fonction d’une colonne.

L’effet d’un index n’est pas perceptible pour les petites bases de données comme l’exemple, mais s’il y a un grand nombre de lignes, cela peut grandement améliorer les performances. Au lieu de vérifier chaque ligne de la table, le serveur peut effectuer une recherche binaire sur l’index.

Le compromis pour la création d’un index est la vitesse d’écriture et la taille de la base de données. Le stockage de l’index prend de la place. De plus, chaque fois qu’un INSERT est effectué ou que la colonne est mise à jour, l’index doit être mis à jour. Ce n’est pas une opération aussi coûteuse que l’analyse de la table entière sur une requête SELECT, mais c’est toujours quelque chose à garder à l’esprit.

Création d’un index #

CREATE INDEX ix_cars_employee_id ON Cars (EmployeeId);

Cela créera un index pour la colonne EmployeeId dans la table Cars. Cet index améliorera la vitesse des requêtes demandant au serveur de trier ou de sélectionner par valeurs dans EmployeeId, telles que les suivantes :

SELECT * FROM Cars WHERE EmployeeId = 1

L’index peut contenir plus d’une colonne, comme dans ce qui suit ;

CREATE INDEX ix_cars_e_c_o_ids ON Cars (EmployeeId, CarId, OwnerId);

Dans ce cas, l’index serait utile pour les requêtes demandant de trier ou de sélectionner toutes les colonnes incluses, si l’ensemble de conditions est ordonné de la même manière. Cela signifie que lors de la récupération des données, il peut trouver les lignes à récupérer à l’aide de l’index, au lieu de parcourir toute la table.

Par exemple, le cas suivant utiliserait le deuxième index ;

SELECT * FROM Cars WHERE EmployeeId = 1 Order by CarId DESC

Si l’ordre diffère, cependant, l’index n’a pas les mêmes avantages, comme dans le suivant ;

SELECT * FROM Cars WHERE OwnerId = 17 Order by CarId DESC

L’index n’est pas aussi utile car la base de données doit récupérer l’intégralité de l’index, sur toutes les valeurs de EmployeeId et CarID, afin de trouver les éléments qui ont OwnerId = 17.

(L’index peut toujours être utilisé ; il se peut que l’optimiseur de requête trouve que la récupération de l’index et le filtrage sur le OwnerId, puis la récupération uniquement des lignes nécessaires est plus rapide que la récupération de la table complète, surtout si la table est volumineuse .)

Index groupés, uniques et triés #

Les index peuvent avoir plusieurs caractéristiques qui peuvent être définies soit à la création, soit en modifiant les index existants.

CREATE CLUSTERED INDEX ix_clust_employee_id ON Employees(EmployeeId, Email);

L’instruction SQL ci-dessus crée un nouvel index clusterisé sur Employees. Les index clusterisés sont des index qui dictent la structure réelle de la table ; la table elle-même est triée pour correspondre à la structure de l’index. Cela signifie qu’il peut y avoir au plus un index clusterisé sur une table. Si un index clusterisé existe déjà sur la table, l’instruction ci-dessus échouera. (Les tables sans index cluster sont également appelées tas.)

CREATE UNIQUE INDEX uq_customers_email ON Customers(Email);

Cela créera un index unique pour la colonne Email dans la table Clients. Cet index, en plus d’accélérer les requêtes comme un index normal, forcera également chaque adresse e-mail de cette colonne à être unique. Si une ligne est insérée ou mise à jour avec une valeur Email non unique, l’insertion ou la mise à jour échouera par défaut.

CREATE UNIQUE INDEX ix_eid_desc ON Customers(EmployeeID);

Cela crée un index sur Customers qui crée également une contrainte de table selon laquelle EmployeeID doit être unique. (Cela échouera si la colonne n’est pas actuellement unique - dans ce cas, s’il y a des employés qui partagent un ID.)

CREATE INDEX ix_eid_desc ON Customers(EmployeeID Desc);

Cela crée un index trié par ordre décroissant. Par défaut, les index (sur le serveur MSSQL, au moins) sont croissants, mais cela peut être modifié.

Index trié #

Si vous utilisez un index qui est trié comme vous le feriez pour le récupérer, l’instruction SELECT n’effectuerait pas de tri supplémentaire lors de la récupération.

CREATE INDEX ix_scoreboard_score ON scoreboard (score DESC);

Lorsque vous exécutez la requête

SELECT * FROM scoreboard ORDER BY score DESC;

Le système de base de données n’effectuerait pas de tri supplémentaire, car il peut effectuer une recherche d’index dans cet ordre.

Supprimer un index, ou le désactiver et le reconstruire #

DROP INDEX ix_cars_employee_id ON Cars;

Nous pouvons utiliser la commande DROP pour supprimer notre index. Dans cet exemple, nous allons DROP l’index appelé ix_cars_employee_id sur la table Cars.

Cela supprime entièrement l’index et, si l’index est en cluster, supprimera tout clustering. Il ne peut pas être reconstruit sans recréer l’index, ce qui peut être lent et coûteux en calculs. Comme alternative, l’index peut être désactivé :

ALTER INDEX ix_cars_employee_id ON Cars DISABLE;

Cela permet à la table de conserver la structure, ainsi que les métadonnées sur l’index.

De manière critique, cela conserve les statistiques d’index, de sorte qu’il est possible d’évaluer facilement le changement. Si cela est justifié, l’index peut ensuite être reconstruit ultérieurement, au lieu d’être entièrement recréé ;

ALTER INDEX ix_cars_employee_id ON Cars REBUILD;

Index partiel ou filtré #

SQL Server et SQLite permettent de créer des index qui contiennent non seulement un sous-ensemble de colonnes, mais également un sous-ensemble de lignes.

Considérez une quantité croissante constante de commandes avec order_state_id égal à terminé (2) et une quantité stable de commandes avec order_state_id égal à commencé (1).

Si votre entreprise utilise des requêtes comme celle-ci :

SELECT id, comment
  FROM orders
 WHERE order_state_id =  1
   AND product_id = @some_value;

L’indexation partielle permet de limiter l’index, incluant uniquement les commandes non terminées :

CREATE INDEX Started_Orders
          ON orders(product_id)
       WHERE order_state_id = 1;

Cet index sera plus petit qu’un index non filtré, ce qui économise de l’espace et réduit le coût de mise à jour de l’index.

Insertion avec un index unique #

UPDATE Customers SET Email = "richard0123@example.com" WHERE id = 1;

Cela échouera si un index unique est défini sur la colonne Email de Clients. Cependant, un comportement alternatif peut être défini pour ce cas :

UPDATE Customers SET Email = "richard0123@example.com" WHERE id = 1 ON DUPLICATE KEY;

Reconstruire l’index #

Au fil du temps, les index B-Tree peuvent se fragmenter en raison de la mise à jour/suppression/insertion de données. Dans la terminologie SQLServer, nous pouvons avoir interne (page d’index à moitié vide) et externe (l’ordre logique des pages ne correspond pas à l’ordre physique). La reconstruction de l’index est très similaire à sa suppression et à sa recréation.

On peut reconstruire un index avec

ALTER INDEX index_name REBUILD;

Par défaut, la reconstruction de l’index est une opération hors ligne qui verrouille la table et empêche DML de s’y opposer, mais de nombreux SGBDR autorisent la reconstruction en ligne. En outre, certains fournisseurs de bases de données proposent des alternatives à la reconstruction d’index telles que REORGANIZE (SQLServer) ou COALESCE/SHRINK SPACE (Oracle).

SAP ASE : suppression de l’index #

Cette commande supprimera l’index dans la table. Cela fonctionne sur le serveur SAP ASE.

Syntaxe:

DROP INDEX [table name].[index name]

Exemple:

DROP INDEX Cars.index_1

Index unique qui autorise les valeurs NULL #

CREATE UNIQUE INDEX idx_license_id 
   ON Person(DrivingLicenseID) WHERE DrivingLicenseID IS NOT NULL
GO

Ce schéma permet une relation 0..1 - les gens peuvent avoir zéro ou un permis de conduire et chaque permis ne peut appartenir qu’à une seule personne

Index clusterisé #

Lors de l’utilisation d’un index clusterisé, les lignes de la table sont triées en fonction de la colonne à laquelle l’index clusterisé est appliqué. Par conséquent, il ne peut y avoir qu’un seul index clusterisé sur la table car vous ne pouvez pas ordonner la table par deux colonnes différentes.

En règle générale, il est préférable d’utiliser un index clusterisé lors de la lecture de tables de données volumineuses. L’inconvénient de l’index clusterisé est lors de l’écriture dans la table et les données doivent être réorganisées (retriées).

Un exemple de création d’un index clusterisé sur une table Employees sur la colonne Employee_Surname :

CREATE CLUSTERED INDEX ix_employees_name ON Employees(Employee_Surname);

Index non clusterisé #

Les index non clusterisés sont stockés séparément de la table. Chaque index de cette structure contient un pointeur sur la ligne de la table qu’il représente.

Ces pointeurs sont appelés localisateurs de ligne. La structure du localisateur de ligne varie selon que les pages de données sont stockées dans un tas ou une table en cluster. Pour un tas, un localisateur de ligne est un pointeur vers la ligne. Pour une table en cluster, le localisateur de ligne est la clé d’index en cluster.

Un exemple de création d’un index non clusterisé sur la table Employees et la colonne Employee_Surname :

CREATE NONCLUSTERED INDEX ix_employees_name ON Employees(Employee_Surname);

Il peut y avoir plusieurs index non clusterisés sur la table. Les opérations de lecture sont généralement plus lentes avec des index non clusterisés qu’avec des index clusterisés car il faut d’abord aller à l’index puis à la table. Il n’y a cependant aucune restriction dans les opérations d’écriture.
