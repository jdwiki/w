Algèbre relationnelle

  • Aperçu

Sur cette page

  • Aperçu

Aperçu #

L’algèbre relationnelle n’est pas un langage SQL à part entière, mais plutôt un moyen d’acquérir une compréhension théorique du traitement relationnel. En tant que tel, il ne doit pas faire référence à des entités physiques telles que des tables, des enregistrements et des champs ; il doit faire référence à des constructions abstraites telles que des relations, des tuples et des attributs. Cela dit, je n’utiliserai pas les termes académiques dans ce document et m’en tiendrai aux termes profanes les plus connus - tables, enregistrements et champs.

Quelques règles d’algèbre relationnelle avant de commencer :

  • Les opérateurs utilisés en algèbre relationnelle travaillent sur des tables entières plutôt que sur des enregistrements individuels.
  • Le résultat d’une expression relationnelle sera toujours un tableau (c’est ce qu’on appelle la propriété de fermeture)

Tout au long de ce document, je me référerai aux deux tableaux suivants :

SÉLECTIONNER #

L’opérateur select renvoie un sous-ensemble de la table principale. sélectionner < table > où < condition >

Par exemple, examinez l’expression :

sélectionner Personnes où DepartmentID = 2

Cela peut être écrit comme suit : [![entrez la description de l’image ici][2]][2]

Cela se traduira par une table dont les enregistrements comprennent tous les enregistrements de la table People où la valeur DepartmentID est égale à 2 : [![entrez la description de l’image ici][3]][3]

Des conditions peuvent également être jointes pour restreindre davantage l’expression :

sélectionnez Personnes où StartYear > 2005 et DepartmentID = 2

donnera le tableau suivant : [![entrez la description de l’image ici][4]][4]

PROJET #

L’opérateur projet renverra des valeurs de champ distinctes à partir d’une table. projet < table > sur < liste de champs >

Par exemple, examinez l’expression suivante : projet Personnes plus StartYear

Cela peut être écrit comme suit : [![entrez la description de l’image ici][5]][5]

Cela se traduira par une table comprenant les valeurs distinctes contenues dans le champ StartYear de la table People. [![entrez la description de l’image ici][6]][6]

Les valeurs en double sont supprimées de la table résultante en raison de la propriété de fermeture créant une table relationnelle : tous les enregistrements d’une table relationnelle doivent être distincts.

Si la liste de champs comprend plus d’un seul champ, la table résultante est une version distincte de ces champs. project People over StartYear, DepartmentID renverra : [![entrez la description de l’image ici][7]][7] Un enregistrement est supprimé en raison de la duplication de 2006 StartYear et 1 DepartmentID.

DONNANT #

Les expressions relationnelles peuvent être enchaînées en nommant les expressions individuelles à l’aide du mot clé giving ou en incorporant une expression dans une autre.

< expression d’algèbre relationnelle > donnant < nom d’alias >

Par exemple, considérez les expressions suivantes : select Personnes où DepartmentID = 2 donnant A projet A over PersonName donnant B

Cela se traduira par le tableau B ci-dessous, le tableau A étant le résultat de la première expression. [![entrez la description de l’image ici][8]][8]

La première expression est évaluée et la table résultante reçoit l’alias A. Cette table est ensuite utilisée dans la deuxième expression pour donner la table finale avec un alias de B.

Une autre façon d’écrire cette expression consiste à remplacer le nom d’alias de la table dans la deuxième expression par le texte entier de la première expression entre crochets : project (select People where DepartmentID = 2) over PersonName giving B

C’est ce qu’on appelle une expression imbriquée.

JOINT NATUREL #

Une jointure naturelle colle deux tables ensemble à l’aide d’un champ commun partagé entre les tables.

joindre < table 1 > et < table 2 > où < champ 1 > = < champ 2 > en supposant que < champ 1 > est dans < table 1 > et < champ 2 > est dans < table 2 >.

Par exemple, l’expression de jointure suivante joindra People et Departments en fonction des colonnes DepartmentID et ID dans les tables respectives : rejoindre Personnes et Départements où DepartmentID = ID

[![entrez la description de l’image ici][9]][9]

Notez que seul DepartmentID de la table People est affiché et non ID de la table Department. Un seul des champs comparés doit être affiché, qui est généralement le nom du champ de la première table de l’opération de jointure.

Bien que cela ne soit pas illustré dans cet exemple, il est possible que la jonction de tables aboutisse à deux champs ayant le même en-tête. Par exemple, si j’avais utilisé l’en-tête Name pour identifier les champs PersonName et Dept (c’est-à-dire pour identifier le Person Name et le Department Name). Lorsque cette situation se présente, nous utilisons le nom de la table pour qualifier les noms de champs à l’aide de la notation par points : People.Name et Departments.Name

join combiné avec select et project peuvent être utilisés ensemble pour extraire des informations :

rejoindre Personnes et Départements où DepartmentID = ID donnant A sélectionnez A où StartYear = 2005 et Dept = ‘Production’ donnant B project B over PersonName donnant C

ou sous forme d’expression combinée :

project (select (join People and Departments where DepartmentID = ID) where StartYear = 2005 and Dept = ’ Production’) over PersonName donnant C

Cela donnera ce tableau : [![entrez la description de l’image ici][10]][10]
