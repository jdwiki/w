## Identifier les goulots d’étranglement des performances à l’aide du profileur MATLAB Profiler est un outil de profilage de logiciels) du code MATLAB. À l’aide du profileur, il est possible d’obtenir une représentation visuelle du temps d’exécution et de la consommation de mémoire.

L’exécution du profileur peut être effectuée de deux manières :

  • Cliquer sur le bouton “Run and Time” dans l’interface graphique de MATLAB tout en ayant un fichier .m ouvert dans l’éditeur (ajouté dans ** R2012b **).

[![Bouton dans la barre d’outils][1]][1]

  • Par programmation, en utilisant :

    profile on
<some code we want to test>
profile off

Vous trouverez ci-dessous un exemple de code et le résultat de son profilage :

function docTest

for ind1 = 1:100
  [~] = var(...
            sum(...
                randn(1000)));
end

spy

[![Sortie du profileur][2]][2]

De ce qui précède, nous apprenons que la fonction spy prend environ 25% du temps d’exécution total. Dans le cas du “vrai code”, une fonction qui prend un tel pourcentage de temps d’exécution serait un bon candidat à l’optimisation, par opposition aux fonctions analogues à var et cla dont l’optimisation doit être évitée.

De plus, il est possible de cliquer sur les entrées de la colonne Function Name pour voir une répartition détaillée du temps d’exécution pour cette entrée. Voici l’exemple d’un clic sur “espionner” :

[![Temporisation interne de “l’espion”][3]][3]

Il est également possible de profiler la consommation de mémoire en exécutant profile('-memory') avant d’exécuter le profileur.

Comparaison du temps d’exécution de plusieurs fonctions #

La combinaison largement utilisée de tic et [toc](http:// de.mathworks.com/help/matlab/ref/toc.html) peut donner une idée approximative du temps d’exécution d’une fonction ou d’extraits de code.

Pour comparer plusieurs fonctions, il ne faut pas l’utiliser. Pourquoi ? Il est presque impossible de fournir des * conditions égales * pour tous les extraits de code à comparer dans un script en utilisant la solution ci-dessus. Peut-être que les fonctions partagent le même espace de fonctions et des variables communes, donc appelées plus tard fonctions et extraits de code tirent déjà parti des variables et fonctions précédemment initialisées. De plus, il n’y a aucune idée si le compilateur JIT traiterait de la même manière ces extraits appelés par la suite.

La fonction dédiée aux benchmarks est timeit. L’exemple suivant illustre son utilisation.

Il y a le tableau ‘A’ et la matrice ‘B’. Il convient de déterminer quelle rangée de ‘B’ est la plus similaire à ‘A’ en comptant le nombre d’éléments différents.

function t = bench()
    A = [0 1 1 1 0 0];
    B = perms(A);

    % functions to compare
    fcns = {
        @() compare1(A,B);
        @() compare2(A,B);
        @() compare3(A,B);
        @() compare4(A,B);
    };

    % timeit
    t = cellfun(@timeit, fcns);
end

function Z = compare1(A,B)  
    Z = sum(  bsxfun(@eq,  A,B) , 2);
end
function Z = compare2(A,B)  
    Z = sum(bsxfun(@xor, A, B),2);
end
function Z = compare3(A,B)  
    A = logical(A);
    Z = sum(B(:,~A),2) + sum(~B(:,A),2);
end
function Z = compare4(A,B)  
     Z = pdist2( A, B, 'hamming', 'Smallest', 1 );
end

Cette méthode de référence a été vue pour la première fois dans cette réponse.

C’est normal d’être “célibataire” ! #

Aperçu: #

Le type de données par défaut pour les tableaux numériques dans MATLAB est “double”. double est une [représentation en virgule flottante de nombres][1], et ce format prend 8 octets (ou 64 bits) par valeur. Dans certains cas, où par ex. traitant uniquement des nombres entiers ou lorsque l’instabilité numérique n’est pas un problème imminent, une telle profondeur de bits élevée peut ne pas être nécessaire. Pour cette raison, il est conseillé de considérer les avantages de la “simple” précision (ou d’autres [types][2] appropriés) :

  • Temps d’exécution plus rapide (particulièrement visible sur les GPU).
  • Moitié de la consommation de mémoire : peut réussir là où “double” échoue en raison d’une erreur de mémoire insuffisante ; plus compact lors du stockage sous forme de fichiers.

La conversion d’une variable de n’importe quel type de données pris en charge en “single” s’effectue à l’aide :

sing_var = single(var);

Certaines fonctions couramment utilisées (telles que : zeros, [eye](http://www .mathworks.com/help/matlab/ref/eye.html#btpkgsh-1), ones, [ etc.][3]) qui génèrent des valeurs “doubles” par défaut, permettent de spécifier le type/classe de la sortie.

Conversion de variables dans un script en précision/type/classe non par défaut : #

Depuis juillet 2016, il n’existe aucun moyen documenté de changer le type de données default MATLAB de double.

Dans MATLAB, les nouvelles variables imitent généralement les types de données des variables utilisées lors de leur création. Pour illustrer cela, considérons l’exemple suivant :

A = magic(3);
B = diag(A);
C = 20*B;
>> whos C
  Name      Size            Bytes  Class     Attributes
  C         3x1                24  double 


A = single(magic(3)); % A is converted to "single"
B = diag(A);
C = B*double(20);     % The stricter type, which in this case is "single", prevails
D = single(size(C));  % It is generally advised to cast to the desired type explicitly.
>> whos C
  Name      Size            Bytes  Class     Attributes
  C         3x1                12  single

Ainsi, il peut sembler suffisant de caster/convertir plusieurs variables initiales pour que le changement se répande dans tout le code - mais cela est ** déconseillé ** (voir Caveats & Pitfalls ci-dessous).

Mises en garde et pièges : #

1. Les conversions répétées sont déconseillées en raison de l’introduction de bruit numérique (lors de la diffusion de “simple” à “double”) ou de la perte d’informations (lors de la diffusion de “double” à “single”, ou entre certains entier types), par ex. :

    double(single(1.2)) == double(1.2)   
    ans =
         0
This can be mitigated somewhat using [`typecast`](http://www.mathworks.com/help/matlab/ref/typecast.html).
See also https://www.wikiod.com/fr/matlab/fautes-et-erreurs-courantes#Soyez conscient de l'inexactitude de la virgule flottante

  1. S’appuyer uniquement sur le typage implicite des données (c’est-à-dire ce que MATLAB suppose que le type de sortie d’un calcul devrait être) est ** déconseillé ** en raison de plusieurs effets indésirables qui pourraient survenir :

    • Loss of information: when a double result is expected, but a careless combination of single and double operands yields single precision.

    • Unexpectedly high memory consumption: when a single result is expected but a careless computation results in a double output.

    • Unnecessary overhead when working with GPUs: when mixing gpuArray types (i.e. variables stored in VRAM) with non-gpuArray variables (i.e. those usually stored in RAM) the data will have to be transferred one way or the other before the computation can be performed. This operation takes time, and can be very noticeable in repetative computations.

    • Errors when mixing floating-point types with integer types: functions like mtimes (*) are not defined for mixed inputs of integer and floating point types - and will error. Functions like times (.*) are not defined at all for integer-type inputs - and will again error.

      >> ones(3,3,'int32')*ones(3,3,'int32')
Error using  * 
MTIMES is not fully supported for integer classes. At least one input must be scalar.

>> ones(3,3,'int32').*ones(3,3,'double')
Error using  .* 
Integers can only be combined with integers of the same class, or scalar doubles.

    For better code readability and reduced risk of unwanted types, a defensive approach is advised, where variables are explicitly cast to the desired type.

L’importance de la préallocation #

Les tableaux dans MATLAB sont conservés sous forme de blocs continus en mémoire, alloués et libérés automatiquement par MATLAB. MATLAB cache les opérations de gestion de la mémoire telles que le redimensionnement d’un tableau derrière une syntaxe facile à utiliser :

a = 1:4

a =

     1     2     3     4

a(5) = 10  % or alternatively a = [a, 10]

a =

     1     2     3     4    10

Il est important de comprendre que ce qui précède n’est pas une opération triviale, a (5) = 10 obligera MATLAB à allouer un nouveau bloc de mémoire de taille 5, à copier les 4 premiers nombres et à définir le 5ème sur 10. C’est une opération O(numel(a)), et non O(1).

Considérer ce qui suit:

clear all
n=12345678;
a=0;
tic
for i = 2:n
    a(i) = sqrt(a(i-1)) + i;
end
toc

Elapsed time is 3.004213 seconds.

a est réalloué n fois dans cette boucle (hors quelques optimisations réalisées par MATLAB) ! Notez que MATLAB nous donne un avertissement :

“La variable ‘a’ semble changer de taille à chaque itération de boucle. Envisagez de préallouer pour la vitesse.”

Que se passe-t-il lorsque nous préallouons ?

a=zeros(1,n);
tic
for i = 2:n
    a(i) = sqrt(a(i-1)) + i;
end
toc

Elapsed time is 0.410531 seconds.

Nous pouvons voir que le temps d’exécution est réduit d’un ordre de grandeur.

Méthodes de préallocation :

MATLAB fournit diverses fonctions pour l’allocation de vecteurs et de matrices, en fonction des besoins spécifiques de l’utilisateur. Ceux-ci incluent : [zéros][1], [ones][2], [nan][3], [eye][4], [true][5] etc.

a = zeros(3)       % Allocates a 3-by-3 matrix initialized to 0
a =

     0     0     0
     0     0     0
     0     0     0

a = zeros(3, 2)     % Allocates a 3-by-2 matrix initialized to 0
a =

     0     0
     0     0
     0     0

a = ones(2, 3, 2)      % Allocates a 3 dimensional array (2-by-3-by-2) initialized to 1
a(:,:,1) =

     1     1     1
     1     1     1


a(:,:,2) =

     1     1     1
     1     1     1

a = ones(1, 3) * 7  % Allocates a row vector of length 3 initialized to 7
a =

     7     7     7

Un type de données peut également être spécifié :

a = zeros(2, 1, 'uint8');  % allocates an array of type uint8

Il est également facile de cloner la taille d’un tableau existant :

a = ones(3, 4);       % a is a 3-by-4 matrix of 1's
b = zeros(size(a));  % b is a 3-by-4 matrix of 0's

Et clonez le type :

a = ones(3, 4, 'single');       % a is a 3-by-4 matrix of type single
b = zeros(2, 'like', a);        % b is a 2-by-2 matrix of type single

notez que ’like’ clone également complexity et sparsity.

La préallocation est implicitement réalisée en utilisant n’importe quelle fonction qui renvoie un tableau de la taille finale requise, comme [rand][6], [gallery][7], [kron][8], [bsxfun ][9], [colon][10] et bien d’autres. Par exemple, une manière courante d’allouer des vecteurs avec des éléments variant linéairement consiste à utiliser l’opérateur deux-points (avec la variante à 2 ou 3 opérandes[1][10]) :

a = 1:3 
a =

     1     2     3

a = 2:-3:-4
a =

     2    -1    -4

Les tableaux de cellules peuvent être alloués à l’aide de la fonction cell() de la même manière que zeros().

a = cell(2,3)
a = 

    []    []    []
    []    []    []

Notez que les tableaux de cellules fonctionnent en conservant des pointeurs vers les emplacements en mémoire du contenu des cellules. Ainsi, tous les conseils de pré-allocation s’appliquent également aux éléments individuels du tableau de cellules.

réorganiser un ND-array peut améliorer les performances globales #

Dans certains cas, nous devons appliquer des fonctions à un ensemble de ND-arrays. Regardons cet exemple simple.

A(:,:,1) = [1 2; 4 5];
A(:,:,2) = [11 22; 44 55];
B(:,:,1) = [7 8; 1 2];
B(:,:,2) = [77 88; 11 22];

A =

ans(:,:,1) =

   1   2 
   4   5 

ans(:,:,2) =

   11   22
   44   55

>> B
B =

ans(:,:,1) =

   7   8
   1   2

ans(:,:,2) =

   77   88
   11   22

Les deux matrices sont 3D, disons que nous devons calculer ce qui suit :

result= zeros(2,2);
...
for k = 1:2 
   result(i,j) = result(i,j) + abs( A(i,j,k) - B(i,j,k) );
...

if k is very large, this for-loop can be a bottleneck since MATLAB order the data in a column major fashion. So a better way to compute "result" could be:

% trying to exploit the column major ordering
Aprime = reshape(permute(A,[3,1,2]), [2,4]);
Bprime = reshape(permute(B,[3,1,2]), [2,4]);


>> Aprime
Aprime =

    1    4    2    5
   11   44   22   55

>> Bprime
Bprime =

    7    1    8    2
   77   11   88   22

Maintenant, nous remplaçons la boucle ci-dessus comme suit :

result= zeros(2,2);
....
temp = abs(Aprime - Bprime);
for k = 1:2
    result(i,j) = result(i,j) + temp(k, i+2*(j-1));
...

Nous avons réorganisé les données afin de pouvoir exploiter la mémoire cache. La permutation et le remodelage peuvent être coûteux, mais lorsque vous travaillez avec de grands réseaux ND, le coût de calcul lié à ces opérations est bien inférieur à celui de travailler avec des réseaux non arrangés.
