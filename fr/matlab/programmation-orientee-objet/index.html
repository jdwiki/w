Classes valeur vs poignée #

Les classes dans MATLAB sont divisées en deux grandes catégories : les classes de valeurs et les classes de poignées. La principale différence est que lors de la copie d’une instance d’une classe de valeurs, les données sous-jacentes sont copiées dans la nouvelle instance, tandis que pour les classes de descripteurs, la nouvelle instance pointe vers les données d’origine et la modification des valeurs dans la nouvelle instance les modifie dans l’original. Une classe peut être définie comme un handle en héritant de la classe handle.

classdef valueClass
    properties
        data
    end
end

et

classdef handleClass < handle
    properties
        data
    end
end

alors

>> v1 = valueClass;
>> v1.data = 5;
>> v2 = v1;
>> v2.data = 7;
>> v1.data
ans =
     5

>> h1 = handleClass;
>> h1.data = 5;
>> h2 = h1;
>> h2.data = 7;
>> h1.data
ans =
     7

Définir une classe #

Une classe peut être définie en utilisant classdef dans un fichier .m avec le même nom que la classe. Le fichier peut contenir le bloc classdef…end et des fonctions locales à utiliser dans les méthodes de classe.

La définition de classe MATLAB la plus générale a la structure suivante :

classdef (ClassAttribute = expression, ...) ClassName < ParentClass1 & ParentClass2 & ...

   properties (PropertyAttributes) 
      PropertyName
   end 

   methods (MethodAttributes) 
      function obj = methodName(obj,arg2,...)
         ...
      end
   end

   events (EventAttributes) 
      EventName
   end

   enumeration
      EnumName
   end

end

Documentation MATLAB : Attributs de classe, [Attributs de propriété](http://www.mathworks.com/help/matlab/ matlab_oop/property-attributes.html), Attributs de méthode, [Attributs d’événement](http://www.mathworks. com/help/matlab/matlab_oop/event-attributes.html), restrictions de classe d’énumération.

Exemple de classe : #

Une classe appelée Car peut être définie dans le fichier Car.m comme

classdef Car < handle % handle class so properties persist
    properties
        make
        model
        mileage = 0;
    end

    methods
        function obj = Car(make, model)
            obj.make = make;
            obj.model = model;
        end
        function drive(obj, milesDriven)
            obj.mileage = obj.mileage + milesDriven;
        end
    end
end

Notez que le constructeur est une méthode portant le même nom que la classe. <Un constructeur est une méthode spéciale d’une classe ou d’une structure dans la programmation orientée objet qui initialise un objet de ce type. Un constructeur est une méthode d’instance qui porte généralement le même nom que la classe et peut être utilisée pour définir les valeurs des membres d’un objet, soit par défaut, soit par des valeurs définies par l’utilisateur.>

Une instance de cette classe peut être créée en appelant le constructeur ;

>> myCar = Car('Ford', 'Mustang'); //creating an instance of car class

L’appel de la méthode “drive” augmentera le kilométrage

>> myCar.mileage 
    
    ans = 
            0

>> myCar.drive(450);

>> myCar.mileage
    
   ans = 
            450

Constructeurs #

Un [constructor][ctor] est une méthode spéciale dans une classe qui est appelée lorsqu’une instance d’un objet est créée. Il s’agit d’une fonction MATLAB standard qui accepte les paramètres d’entrée, mais elle doit également suivre certaines [règles][ctor].

Les constructeurs ne sont pas requis car MATLAB en crée un par défaut. En pratique, cependant, c’est un endroit pour définir un état d’un objet. Par exemple, les propriétés peuvent être restreintes en spécifiant [attributs][propattributes]. Ensuite, un constructeur peut [initialiser][defaultvalue] ces propriétés par défaut ou des valeurs définies par l’utilisateur qui peuvent en fait être envoyées par les paramètres d’entrée d’un constructeur.

Appel d’un constructeur d’une classe simple

Il s’agit d’une simple classe “Personne”.

classdef Person
    properties
        name
        surname
        address
    end
    
    methods
        function obj = Person(name,surname,address)
            obj.name = name;
            obj.surname = surname;
            obj.address = address;
        end
    end
end

Le nom d’un constructeur est le même que le nom d’une classe. Par conséquent, les constructeurs sont appelés par le nom de leur classe. Une classe “Personne” peut être créée comme suit :

>> p = Person('John','Smith','London')
p = 
  Person with properties:

       name: 'John'
    surname: 'Smith'
    address: 'London'

Appeler un constructeur d’une classe enfant

Les classes peuvent être héritées des classes parentes si elles partagent des propriétés ou des méthodes communes. Lorsqu’une classe est héritée d’une autre, il est probable qu’un constructeur d’une classe parente doive être appelé.

Une classe Member hérite d’une classe Person car Member utilise les mêmes propriétés que la classe Person mais ajoute également payment à sa définition.

classdef Member < Person
    properties
        payment
    end

    methods
        function obj = Member(name,surname,address,payment)
            obj = obj@Person(name,surname,address);
            obj.payment = payment;
        end
    end
end

Comme pour la classe Person, Member est créé en appelant son constructeur :

>> m = Member('Adam','Woodcock','Manchester',20)
m = 
  Member with properties:

    payment: 20
       name: 'Adam'
    surname: 'Woodcock'
    address: 'Manchester'

Un constructeur de Person nécessite trois paramètres d’entrée. Member doit respecter ce fait et donc appeler un constructeur de la classe Person avec trois paramètres. Il est rempli par la ligne :

obj = obj@Person(name,surname,address);

L’exemple ci-dessus montre le cas où une classe enfant a besoin d’informations pour sa classe parent. C’est pourquoi un constructeur de Member nécessite quatre paramètres : trois pour sa classe parente et un pour lui-même.

[cteur] : http://uk.mathworks.com/help/matlab/matlab_oop/class-constructor-methods.html?requestedDomain=www.mathworks.com [propriété] : http://uk.mathworks.com/help/matlab/matlab_oop/class-components.html#bupsckv [valeur par défaut] : http://uk.mathworks.com/help/matlab/matlab_oop/specifying-properties.html#brqy3km-10 [propattributes] : http://uk.mathworks.com/help/matlab/matlab_oop/property-attributes.html

Héritage des classes et des classes abstraites #

Avis de non-responsabilité : les exemples présentés ici ont uniquement pour but de montrer l’utilisation des classes abstraites et de l’héritage et ne sont pas nécessairement d’une utilité pratique. De plus, il n’y a rien de si polymorphe dans MATLAB et donc l’utilisation de classes abstraites est limitée. Cet exemple est de montrer qui créer une classe, hériter d’une autre classe et appliquer une classe abstraite pour définir une interface commune.

L’utilisation de classes abstraites est plutôt limitée dans MATLAB, mais elle peut toujours s’avérer utile à quelques reprises.

Disons que nous voulons un enregistreur de messages. Nous pourrions créer une classe similaire à celle ci-dessous :

classdef ScreenLogger
    properties(Access=protected)
        scrh;
    end
    
    methods
        function obj = ScreenLogger(screenhandler)
            obj.scrh = screenhandler;
        end
        
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                fprintf(obj.scrh, '%s\n', sprintf(varargin{:}));
            end
        end
    end
end

Propriétés et méthodes

En bref, les propriétés contiennent un état d’un objet tandis que les méthodes sont comme une interface et définissent des actions sur les objets.

La propriété scrh est protégée. C’est pourquoi il doit être initialisé dans un constructeur. Il existe d’autres méthodes (getters) pour accéder à cette propriété mais cela ne correspond pas à cet exemple. Les propriétés et les méthodes sont accessibles via une variable qui contient une référence à un objet en utilisant la notation par points suivie du nom d’une méthode ou d’une propriété :

mylogger = ScreenLogger(1);                         % OK
mylogger.LogMessage('My %s %d message', 'very', 1); % OK
mylogger.scrh = 2;                                  % ERROR!!! Access denied

Les propriétés et les méthodes peuvent être publiques, privées ou protégées. Dans ce cas, protégé signifie que je pourrai accéder à scrh à partir d’une classe héritée mais pas de l’extérieur. Par défaut, toutes les propriétés et méthodes sont publiques. Par conséquent, LogMessage() peut être librement utilisé en dehors de la définition de classe. De plus, LogMessage définit une interface, ce qui signifie que c’est ce que nous devons appeler lorsque nous voulons qu’un objet enregistre nos messages personnalisés.

Application

Disons que j’ai un script où j’utilise mon logger :

clc;
% ... a code
logger = ScreenLogger(1);
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');

Si j’ai plusieurs endroits où j’utilise le même logger et que je veux ensuite le changer pour quelque chose de plus sophistiqué, comme écrire un message dans un fichier, je devrais créer un autre objet :

classdef DeepLogger
    properties(SetAccess=protected)
        FileName
    end
    methods
        function obj = DeepLogger(filename)
            obj.FileName = filename;
        end
        
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end 
end

et changez simplement une ligne de code en ceci:

clc;
% ... a code
logger = DeepLogger('mymessages.log');

La méthode ci-dessus ouvrira simplement un fichier, ajoutera un message à la fin du fichier et le fermera. Pour le moment, pour être cohérent avec mon interface, je dois me rappeler que le nom d’une méthode est LogMessage() mais cela peut aussi être n’importe quoi d’autre. MATLAB peut forcer le développeur à s’en tenir au même nom en utilisant des classes abstraites. Disons que nous définissons une interface commune pour n’importe quel logger :

classdef MessageLogger
    methods(Abstract=true)
        LogMessage(obj, varargin);
    end
end

Maintenant, si ScreenLogger et DeepLogger héritent de cette classe, MATLAB générera une erreur si LogMessage() n’est pas défini. Les classes abstraites aident à construire des classes similaires qui peuvent utiliser la même interface.

Pour les besoins de cet exemple, je vais apporter des modifications légèrement différentes. Je vais supposer que DeepLogger fera à la fois un message de journalisation sur un écran et dans un fichier en même temps. Étant donné que ScreenLogger consigne déjà les messages à l’écran, je vais hériter de DeepLogger du ScreenLoggger pour éviter les répétitions. ScreenLogger ne change pas du tout à part la première ligne :

classdef ScreenLogger < MessageLogger
// the rest of previous code

Cependant, DeepLogger nécessite plus de modifications dans la méthode LogMessage :

classdef DeepLogger < MessageLogger & ScreenLogger
    properties(SetAccess=protected)
        FileName
        Path
    end
    methods
        function obj = DeepLogger(screenhandler, filename)
            [path,filen,ext] = fileparts(filename);
            obj.FileName = [filen ext];
            pbj.Path     = pathn;
            obj = obj@ScreenLogger(screenhandler);
        end
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                LogMessage@ScreenLogger(obj, varargin{:});
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end
end

Tout d’abord, j’initialise simplement les propriétés dans le constructeur. Deuxièmement, parce que cette classe hérite de ScreenLogger, je dois également initialiser cet objet parent. Cette ligne est encore plus importante car le constructeur ScreenLogger nécessite un paramètre pour initialiser son propre objet. Cette ligne:

obj = obj@ScreenLogger(screenhandler);

dit simplement “appelez le constructeur de ScreenLogger et initialisez-le avec un gestionnaire d’écran”. Il convient de noter ici que j’ai défini scrh comme protégé. Par conséquent, je pouvais également accéder à cette propriété à partir de DeepLogger. Si la propriété a été définie comme privée. La seule façon de l’initialiser serait d’utiliser consuctor.

Un autre changement est dans la section “méthodes”. Encore une fois, pour éviter les répétitions, j’appelle LogMessage() à partir d’une classe parente pour consigner un message sur un écran. Si je devais changer quoi que ce soit pour apporter des améliorations à la journalisation d’écran, je dois maintenant le faire en un seul endroit. Le reste du code est le même car il fait partie de DeepLogger.

Étant donné que cette classe hérite également d’une classe abstraite MessageLogger, je devais m’assurer que LogMessage() à l’intérieur de DeepLogger est également défini. L’héritage de MessageLogger est un peu délicat ici. Je pense que la redéfinition de LogMessage est obligatoire - je suppose.

En ce qui concerne le code où un enregistreur est appliqué, grâce à une interface commune dans les classes, je peux être assuré que la modification de cette ligne dans l’ensemble du code ne poserait aucun problème. Les mêmes messages seront affichés à l’écran comme auparavant, mais en plus, le code écrira ces messages dans un fichier.

clc;
% ... a code
logger = DeepLogger(1, 'mylogfile.log');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');

J’espère que ces exemples ont expliqué l’utilisation des classes, l’utilisation de l’héritage et l’utilisation des classes abstraites.

PS. La solution au problème ci-dessus est l’une des nombreuses. Une autre solution, moins complexe, serait de faire de ScreenLoger un composant d’un autre enregistreur comme FileLogger etc. ScreenLogger serait contenu dans l’une des propriétés. Son LogMessage appellerait simplement LogMessage du ScreenLogger et afficherait du texte sur un écran. J’ai choisi une approche plus complexe pour montrer plutôt comment les classes fonctionnent dans MATLAB. L’exemple de code ci-dessous :

classdef DeepLogger < MessageLogger
    properties(SetAccess=protected)
        FileName
        Path
        ScrLogger
    end
    methods
        function obj = DeepLogger(screenhandler, filename)
            [path,filen,ext] = fileparts(filename);
            obj.FileName     = [filen ext];
            obj.Path         = pathn;
            obj.ScrLogger    = ScreenLogger(screenhandler);
        end
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                obj.LogMessage(obj.ScrLogger, varargin{:}); % <-------- thechange here
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end
end
