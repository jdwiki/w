Les ports série sont une interface commune pour communiquer avec des capteurs externes ou des systèmes embarqués tels que Arduinos. Les communications série modernes sont souvent mises en œuvre via des connexions USB à l’aide d’adaptateurs série USB. MATLAB fournit des fonctions intégrées pour les communications série, y compris les protocoles RS-232 et RS-485. Ces fonctions peuvent être utilisées pour les ports série matériels ou les connexions série USB “virtuelles”. Les exemples présentés ici illustrent les communications série dans MATLAB.

Paramètres #

Paramètre de port série ce qu’il fait
Débit en bauds Définit le débit en bauds. Le plus courant aujourd’hui est 57600, mais 4800, 9600 et 115200 sont également fréquemment observés
Taille du tampon d'entrée Le nombre d’octets conservés en mémoire. Matlab a un FIFO, ce qui signifie que les nouveaux octets seront ignorés. La valeur par défaut est de 512 octets, mais elle peut facilement être définie sur 20 Mo sans problème. Il n’y a que quelques cas extrêmes où l’utilisateur voudrait que ce soit petit
Octets disponibles Le nombre d’octets en attente de lecture
ValeursEnvoyées Le nombre d’octets envoyés depuis l’ouverture du port
Valeurs reçues Le nombre d’octets lus depuis l’ouverture du port
BytesAvailableFcn Spécifiez la fonction de rappel à exécuter lorsqu’un nombre spécifié d’octets est disponible dans le tampon d’entrée ou qu’un terminateur est lu
BytesAvailableFcnCount Spécifiez le nombre d’octets qui doivent être disponibles dans le tampon d’entrée pour générer un événement bytes-available
BytesAvailableFcnMode Spécifiez si l’événement bytes-available est généré après qu’un nombre spécifié d’octets est disponible dans le tampon d’entrée ou après la lecture d’un terminateur

Création d’un port série sur Mac/Linux/Windows #

% Define serial port with a baud rate of 115200
rate = 115200;
if ispc
    s = serial('COM1', 'BaudRate',rate);
elseif ismac
    % Note that on OSX the serial device is uniquely enumerated. You will
    % have to look at /dev/tty.* to discover the exact signature of your
    % serial device
    s = serial('/dev/tty.usbserial-A104VFT7', 'BaudRate',rate);
elseif isunix
    s = serial('/dev/ttyusb0', 'BaudRate',rate);
end

% Set the input buffer size to 1,000,000 bytes (default: 512 bytes).
s.InputBufferSize = 1000000;    

% Open serial port    
fopen(s);

Choisir votre mode de communication #

Matlab prend en charge la communication synchrone et asynchrone avec un port série. Il est important de choisir le bon mode de communication. Le choix dépendra de :

  • comment se comporte l’instrument avec lequel vous communiquez.
  • quelles autres fonctions votre programme principal (ou GUI) devra faire en dehors de la gestion du port série.

Je vais définir 3 cas différents pour illustrer, du plus simple au plus exigeant. Pour les 3 exemples, l’instrument auquel je me connecte est un circuit imprimé avec un inclinomètre, qui peut fonctionner dans les 3 modes que je vais décrire ci-dessous.

Mode 1 : Synchrone (Maître/Esclave) #

Ce mode est le plus simple. Cela correspond au cas où le PC est le Maître et l’instrument est l’esclave. L’instrument n’envoie rien au port série de lui-même, il ne fait que répondre à une réponse après avoir reçu une question/commande du maître (le PC, votre programme). Par exemple:

  • Le PC envoie une commande : “Donnez-moi une mesure maintenant”
  • L’instrument reçoit la commande, effectue la mesure puis renvoie la valeur de la mesure sur la ligne série : « La valeur de l’inclinomètre est XXX ».

OU

  • Le PC envoie une commande : “Passage du mode X au mode Y”
  • L’instrument reçoit la commande, l’exécute, puis renvoie un message de confirmation sur la ligne série : “Commande exécutée” (ou “Commande NON exécutée”). Ceci est communément appelé une réponse ACK/NACK (pour “Acknowledge(d)” / “NOT Acknowledged”).

Résumé : dans ce mode, l’instrument (l’Esclave) n’envoie des données qu’à la ligne série immédiatement après avoir été sollicité par le PC (le Maître)

[![Illustration synchrone][1]][1]

Mode 2 : Asynchrone #

Supposons maintenant que j’ai démarré mon instrument, mais c’est plus qu’un simple capteur stupide. Il surveille constamment sa propre inclinaison et tant qu’il est vertical (dans une tolérance, disons +/-15 degrés), il reste silencieux. Si l’appareil est incliné de plus de 15 degrés et se rapproche de l’horizontale, il envoie un message d’alarme à la ligne série, immédiatement suivi d’une lecture de l’inclinaison. Tant que l’inclinaison est supérieure au seuil, il continue d’envoyer une lecture d’inclinaison toutes les 5s.

Si votre programme principal (ou interface graphique) “attend” constamment l’arrivée d’un message sur la ligne série, il peut bien le faire … mais il ne peut rien faire d’autre en attendant. Si le programme principal est une interface graphique, il est très frustrant d’avoir une interface graphique apparemment “gelée” car elle n’acceptera aucune entrée de l’utilisateur. Essentiellement, il est devenu le Slave et l’instrument est le Master. À moins que vous n’ayez un moyen sophistiqué de contrôler votre interface graphique à partir de l’instrument, c’est quelque chose à éviter. Heureusement, le mode de communication asynchrone vous permettra :

  • définir une fonction distincte qui indique à votre programme quoi faire lorsqu’un message est reçu
  • gardez cette fonction dans un coin, elle ne sera appelée et exécutée que lorsqu’un message arrivera sur la ligne série. Le reste du temps, l’interface graphique peut exécuter tout autre code qu’elle doit exécuter.

Résumé : Dans ce mode, l’instrument peut envoyer un message à la ligne série à tout moment (mais pas nécessairement tout le temps). Le PC n’attend pas en permanence qu’un message soit traité. Il est permis d’exécuter n’importe quel autre code. Ce n’est qu’à l’arrivée d’un message qu’il active une fonction qui lira et traitera ensuite ce message.

[![entrez la description de l’image ici][2]][2]

Mode 3 : diffusion en continu (temps réel) #

Libérons maintenant toute la puissance de mon instrument. Je l’ai mis dans un mode où il enverra constamment des mesures à la ligne série. Mon programme souhaite recevoir ces paquets et les afficher sur une courbe ou un affichage numérique. S’il n’envoie qu’une valeur toutes les 5s comme ci-dessus, pas de problème, gardez le mode ci-dessus. Mais mon instrument à plein régime envoie un point de données à la ligne série à 1000 Hz, c’est-à-dire qu’il envoie une nouvelle valeur toutes les millisecondes. Si je reste dans le mode asynchrone décrit ci-dessus, il y a un risque élevé (en fait une certitude garantie) que la fonction spéciale que nous avons définie pour traiter chaque nouveau paquet prenne plus d'1 ms pour s’exécuter (si vous voulez tracer ou afficher la valeur, les fonctions graphiques sont assez lentes, sans même considérer le filtrage ou la FFT du signal). Cela signifie que la fonction commencera à s’exécuter, mais avant qu’elle ne se termine, un nouveau paquet arrivera et déclenchera à nouveau la fonction. La deuxième fonction est placée dans une file d’attente pour exécution et ne démarre que lorsque la première est terminée … mais à ce moment-là, quelques nouveaux paquets sont arrivés et chacun a placé une fonction à exécuter dans la file d’attente. Vous pouvez rapidement prévoir le résultat : Au moment où je trace les 5èmes points, j’en ai déjà des centaines qui attendent d’être tracés aussi… l’interface graphique ralentit, finit par se figer, la pile grossit, les tampons se remplissent, jusqu’à ce que quelque chose cède. Finalement, vous vous retrouvez avec un programme complètement gelé ou simplement en panne.

Pour pallier à cela, nous déconnecterons encore plus le lien de synchronisation entre le PC et l’instrument. On laissera l’instrument envoyer les données à son rythme, sans déclencher immédiatement une fonction à chaque arrivée de paquet. Le tampon du port série accumulera simplement les paquets reçus. Le PC ne collectera les données dans le tampon qu’à un rythme qu’il peut gérer (un intervalle régulier, configuré côté PC), en fera quelque chose (pendant que le tampon est rempli à nouveau par l’instrument), puis collectera un nouveau lot de données de la mémoire tampon … et ainsi de suite.

Résumé : Dans ce mode, l’instrument envoie des données en continu, qui sont collectées par la mémoire tampon du port série. À intervalles réguliers, le PC collecte des données du tampon et en fait quelque chose. Il n’y a pas de lien de synchronisation physique entre le PC et l’instrument. Les deux exécutent leurs tâches à leur propre rythme.

Lecture depuis le port série #

En supposant que vous ayez créé l’objet de port série s comme dans [this](https://www.wikiod.com/fr/matlab/utilisation-des-ports-serie#Création d’un port série sur Mac/Linux/Windows port-on-mac-linux-windows) exemple, puis

% Read one byte
data = fread(s, 1);

% Read all the bytes, version 1
data = fread(s);

% Read all the bytes, version 2
data = fread(s, s.BytesAvailable);

% Close the serial port
fclose(s);

Fermeture d’un port série même s’il est perdu, supprimé ou écrasé #

En supposant que vous ayez créé l’objet de port série s comme dans [this](https://www.wikiod.com/fr/matlab/utilisation-des-ports-serie#Création d’un port série sur Mac/Linux/Windows port-on-mac-linux-windows) exemple, puis de le fermer

fclose(s)

Cependant, vous pouvez parfois perdre accidentellement le port (par exemple, effacer, écraser, modifier la portée, etc.), et fclose(s) ne fonctionnera plus. La solution est facile

fclose(instrfindall)

Plus d’informations sur [instrfindall()][1].

[1] : http://www.mathworks.com/help/matlab/ref/instrfindall.html

Ecriture sur le port série #

En supposant que vous ayez créé l’objet de port série s comme dans [this](https://www.wikiod.com/fr/matlab/utilisation-des-ports-serie#Création d’un port série sur Mac/Linux/Windows port-on-mac-linux-windows) exemple, puis

% Write one byte
fwrite(s, 255);

% Write one 16-bit signed integer
fwrite(s, 32767, 'int16');

% Write an array of unsigned 8-bit integers
fwrite(s,[48 49 50],'uchar');

% Close the serial port
fclose(s);

Traitement automatique des données reçues d’un port série #

Certains appareils connectés via un port série envoient des données à votre programme à un débit constant (données en continu) ou envoient des données à des intervalles imprévisibles. Vous pouvez configurer le port série pour qu’il exécute automatiquement une fonction afin de gérer les données à chaque fois qu’elles arrivent. C’est ce qu’on appelle la “fonction de rappel” pour l’objet port série.

Deux propriétés du port série doivent être définies pour utiliser cette fonctionnalité : le nom de la fonction souhaitée pour le rappel (BytesAvailableFcn) et la condition qui doit déclencher l’exécution de la fonction de rappel (BytesAvailableFcnMode).

Il existe deux manières de déclencher une fonction de rappel :

  1. Lorsqu’un certain nombre d’octets ont été reçus sur le port série (généralement utilisé pour les données binaires)
  2. Lorsqu’un certain caractère est reçu sur le port série (généralement utilisé pour le texte ou les données ASCII)

Les fonctions de rappel ont deux arguments d’entrée requis, appelés obj et event. obj est le port série. Par exemple, si vous souhaitez imprimer les données reçues du port série, définissez une fonction d’impression des données appelée newdata :

function newdata(obj,event)
    [d,c] = fread(obj);  % get the data from the serial port
    % Note: for ASCII data, use fscanf(obj) to return characters instead of binary values
    fprintf(1,'Received %d bytes\n',c);
    disp(d)
end

Par exemple, pour exécuter la fonction “newdata” chaque fois que 64 octets de données sont reçus, configurez le port série comme ceci :

s = serial(port_name);
s.BytesAvailableFcnMode = 'byte';
s.BytesAvailableFcnCount = 64;
s.BytesAvailableFcn = @newdata;

Avec du texte ou des données ASCII, les données sont généralement divisées en lignes avec un “caractère de fin”, tout comme le texte sur une page. Pour exécuter la fonction “newdata” chaque fois que le caractère de retour chariot est reçu, configurez le port série comme ceci :

s = serial(port_name);
s.BytesAvailableFcnMode = 'terminator';
s.Terminator = 'CR';  % the carriage return, ASCII code 13
s.BytesAvailableFcn = @newdata;
