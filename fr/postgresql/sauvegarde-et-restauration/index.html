Sauvegarder le système de fichiers au lieu d’utiliser pg_dumpall et pg_dump #

Il est très important que si vous l’utilisez, vous appeliez la fonction pg_start_backup() avant et la fonction pg_stop_backup() après. Faire des sauvegardes de système de fichiers n’est pas sûr autrement ; même un instantané ZFS ou FreeBSD du système de fichiers sauvegardé sans ces appels de fonction placera la base de données en mode de récupération et peut perdre des transactions.

J’éviterais de faire des sauvegardes de système de fichiers au lieu de sauvegardes Postgres régulières, à la fois pour cette raison et parce que les fichiers de sauvegarde Postgres (en particulier dans le format personnalisé) sont extrêmement polyvalents pour prendre en charge les restaurations alternatives. Puisqu’il s’agit de fichiers uniques, ils sont également moins compliqués à gérer.

Sauvegarde d’une base de données #

pg_dump -Fc -f DATABASE.pgsql DATABASE

Le -Fc sélectionne le “format de sauvegarde personnalisé” qui vous donne plus de puissance que le SQL brut ; voir pg_restore pour plus de détails. Si vous voulez un fichier SQL vanille, vous pouvez faire ceci à la place :

pg_dump -f DATABASE.sql DATABASE

ou même

pg_dump DATABASE > DATABASE.sql

[1] : https://www.postgresql.org/docs/9.5/static/app-pgrestore.html

Restauration des sauvegardes #

psql < backup.sql

Une alternative plus sûre utilise -1 pour encapsuler la restauration dans une transaction. Le -f spécifie le nom du fichier plutôt que d’utiliser la redirection du shell.

psql -1f backup.sql

Les fichiers au format personnalisé doivent être restaurés à l’aide de pg_restore avec l’option -d pour spécifier la base de données :

pg_restore -d DATABASE DATABASE.pgsql

Le format personnalisé peut également être reconverti en SQL :

pg_restore backup.pgsql > backup.sql

L’utilisation du format personnalisé est recommandée car vous pouvez choisir les éléments à restaurer et éventuellement activer le traitement parallèle.

Vous devrez peut-être faire un pg_dump suivi d’un pg_restore si vous mettez à niveau d’une version de postgresql vers une version plus récente.

Sauvegarde de l’ensemble du cluster #

$ pg_dumpall -f backup.sql

Cela fonctionne en arrière-plan en établissant plusieurs connexions au serveur une fois pour chaque base de données et en exécutant pg_dump dessus.

Parfois, vous pourriez être tenté de configurer cela comme une tâche cron, vous voulez donc voir la date à laquelle la sauvegarde a été prise dans le nom du fichier :

$ postgres-backup-$(date +%Y-%m-%d).sql

Cependant, veuillez noter que cela pourrait produire des fichiers volumineux au quotidien. Postgresql a un bien meilleur mécanisme pour les sauvegardes régulières - [archives WAL][1]

La sortie de pg_dumpall est suffisante pour restaurer une instance Postgres configurée de manière identique, mais les fichiers de configuration dans $PGDATA (pg_hba.conf et postgresql.conf) ne font pas partie de la sauvegarde, vous aurez donc pour les sauvegarder séparément.

postgres=# SELECT pg_start_backup('my-backup');
postgres=# SELECT pg_stop_backup();

Pour effectuer une sauvegarde du système de fichiers, vous devez utiliser ces fonctions pour vous assurer que Postgres est dans un état cohérent pendant la préparation de la sauvegarde.

[1] : https://www.postgresql.org/docs/9.2/static/continuous-archiving.html

Utiliser psql pour exporter des données #

Les données peuvent être exportées à l’aide de la commande copy ou en utilisant les options de ligne de commande de la commande psql.

Pour exporter les données csv de l’utilisateur de la table vers le fichier csv :

psql -p \<port> -U \<username> -d \<database> -A -F<delimiter> -c\<sql to execute> \> \<output filename with path>

psql -p 5432 -U postgres -d test_database -A -F, -c "select * from user" > /home/user/user_data.csv

Ici, la combinaison de -A et -F fait l’affaire.

-F est de spécifier le délimiteur

-A or --no-align

Passe en mode de sortie non aligné. (Le mode de sortie par défaut est autrement aligné.)

Utiliser Copier pour importer #

Pour copier des données d’un fichier CSV dans une table #

COPY <tablename> FROM '<filename with path>';

Pour insérer dans la table user à partir d’un fichier nommé user_data.csv placé dans /home/user/ :

COPY user FROM '/home/user/user_data.csv';

Pour copier des données d’un fichier séparé par un tuyau dans une table #

COPY user FROM '/home/user/user_data' WITH DELIMITER '|';

Remarque : En l’absence de l’option avec délimiteur, le délimiteur par défaut est la virgule ,

Pour ignorer la ligne d’en-tête lors de l’importation du fichier #

Utilisez l’option En-tête :

COPY user FROM '/home/user/user_data' WITH DELIMITER '|' HEADER;

Remarque : Si les données sont entre guillemets, les guillemets par défaut sont des guillemets doubles. Si les données sont citées à l’aide d’un autre caractère, utilisez l’option QUOTE ; cependant, cette option n’est autorisée que lors de l’utilisation du format CSV.
