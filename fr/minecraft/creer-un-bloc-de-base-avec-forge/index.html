Créer un bloc de base avec Forge

  • La classe de bloc
  • Le modèle de bloc JSON
  • Bloquer l’enregistrement

Sur cette page

  • La classe de bloc
  • Le modèle de bloc JSON
  • Bloquer l’enregistrement

Créer un bloc décoratif simple avec Forge est l’une des premières tâches qu’un moddeur en herbe devra apprendre. Comment faire cela a changé au fil des différentes versions de Minecraft et est probablement à une difficulté “modérée” après la 1.7.10 en raison du grand nombre d’erreurs faciles à faire.

Si quelque chose ne va pas et que votre bloc personnalisé (qu’il soit placé ou tenu) a une texture (noir/violet) ou un modèle manquant (cube par défaut qui est trop grand lorsqu’il est tenu), consultez le journal. Les problèmes de ce type apparaîtront presque * toujours * dans le journal, à condition que vous ayez correctement enregistré les choses.

La première chose qui apparaîtra est la ligne "Modèle de chargement d'exception pour la variante..." ou similaire, vous indiquant quel bloc ou élément n’a pas pu être chargé correctement. Après une douzaine de lignes commençant par at..., il y aura une autre ligne commençant par Causé par...

Cette ligne “Causé par” est la plus importante, elle vous indiquera quel fichier n’a pas pu être chargé correctement et peut être l’une des nombreuses erreurs, telles que :

  • JSON malformé (votre fichier JSON n’est pas valide et contient une faute de frappe)
  • File Not Found (le fichier recherché par Minecraft n’est pas correctement nommé ou au bon endroit)
  • Exception de variante manquante (votre Blockstate JSON est incomplet)

Vous pouvez même obtenir plus d’une erreur d’échec pour une seule erreur ! Si vous ne savez pas quel bloc pose problème, commentez chaque bloc et élément dont vous savez qu’il fonctionne, réduisez la liste d’erreurs au bloc (ou aux blocs) en question. Vous devrez peut-être le faire plusieurs fois.

De plus, une texture de mélange apparaît aussi différemment qu’une simple liste de toutes les ressources manquantes pour un domaine donné (mod ID).

La classe de bloc #

Nous avons d’abord besoin d’une classe qui représente le bloc

public class CustomBlock extends Block {

    public CustomBlock () {
        super(Material.ROCK);
        setHardness(1.0f);
        setHarvestLevel("pickaxe", 0);
        setResistance(1.0f);
        setCreativeTab(CreativeTabs.DECORATIONS);
        this.setSoundType(SoundType.STONE);
    }
}

Même ici, plusieurs modifications sont disponibles : matériau (qui régit certaines propriétés telles que pouvoir être poussé par des pistons et s’il peut être cassé à la main), dureté (combien de temps il faut pour casser), niveau de récolte (outil et outil appropriés matériel : dans ce cas, pioche en bois), résistance (vs. explosions), l’onglet qu’il apparaît dans le menu créatif et le son qu’il a.

C’est là que tous les blocs de fonctionnalités fantaisistes devront aller, mais pour l’instant, nous créons un bloc qui a l’air sympa, donc nous avons terminé.

Le modèle de bloc JSON #

Ensuite, nous devons dire à Minecraft à quoi nous voulons que notre bloc ressemble.

{
    "parent": "block/cube_all",
    "textures": {
        "all": "example:blocks/decorative"
    }
}

C’est à peu près tout ce qui est nécessaire pour que cela fonctionne une fois le bloc enregistré. La seule chose importante est que le nom de fichier corresponde au nom de registre utilisé pour enregistrer le bloc et doit être en minuscules (les noms de fichiers 1.11+ doivent requis être en minuscules, avant cela, c’est juste sensible aux majuscules et minuscules).

Nommez le fichier modèle JSON my_block.json (correspondant au nom de registre que nous allons lui donner plus tard) et enregistrez-le dans src\main\resources\assets\example\models\block\ (où example est l’ID de mod spécifié dans l’annotation @Mod de votre classe de mod principale).

Le modèle de bloc utilise ici un parent de block/cube_all, ce qui signifie que la seule texture fournie sera utilisée sur toutes les faces. Il existe également d’autres modèles par défaut, tels que :

  • bloc/cube (les six faces sont attribuées indépendamment)
  • block/cube_bottom_top (faces supérieure et inférieure indépendantes des côtés)
  • bloc/orientable (bloc de parement orientable, ex. four)
  • bloc/croix (fleurs, hautes herbes)
  • bloc/culture (blé, carottes)

Notez que chaque modèle spécifie les textures qu’il utilise par un identifiant de nom (par exemple, "all" ou "top"). Regardez le modèle parent pour déterminer quels sont ces noms si vous n’êtes pas sûr. Des textures incorrectement spécifiées peuvent entraîner des problèmes de texture manquante * sans rapport d’erreur *.

Il est également possible de créer un modèle entièrement personnalisé ou de créer un modèle parent personnalisé. Mais pour l’instant, cela suffira.

N’oubliez pas de créer une texture, nommez-la decorative.png (car c’est ce que le fichier JSON a spécifié) et enregistrez-la dans src\main\resources\assets\example\textures\blocks\

Bloquer l’enregistrement #

L’enregistrement des blocs se fait à partir de votre classe mod principale ou d’une méthode de classe ModBlocks invoquée à partir de la classe mod principale pendant preInit.

Block myBlock = new CustomBlock();
string registryname = "my_block";
block.setRegistryName(registryname);
block.setUnlocalizedName(block.getRegistryName().toString());
GameRegistry.register(block);

Il existe une raison importante d’utiliser également block.setUnlocalizedName(block.getRegistryName().toString()); ! Il garantit que les noms non localisés de votre bloc (et de votre élément) contiennent l’ID de mod pour éviter les conflits de fichiers de langue entre les mods.

Oh, vous voulez aussi une version d’objet pour qu’elle puisse également exister dans votre inventaire ? Cela est créé séparément après 1.7.10 et fait comme suit :

ItemBlock ib = new ItemBlock(block);
ib.setRegistryName(registryname);
GameRegistry.register(ib);

Notez que nous définissons le nom de registre de l’ItemBlock sur la même chaîne que notre bloc. C’est ainsi que Forge et associe des blocs à leur homologue ItemBlock et vice versa.

Mais attendez, il y a plus !

Votre bloc peut avoir une forme d’objet, mais cet objet n’a pas encore de modèle ou de texture ! Les modèles sont automatiquement enregistrés pour les blocs, mais pas pour les éléments. Ceci ne peut être appelé qu’à partir du Client Proxy et ne couvre pas les blocs avec des variantes (telles que la laine ou les feuilles).

ModelLoader.setCustomModelResourceLocation(
    ib , 0, new ModelResourceLocation(ib.getRegistryName(),"normal"));

De manière générale, vous n’aurez pas également besoin d’un modèle d’élément JSON car Forge et vanilla se rabattront sur le modèle du bloc à la place, mais ce n’est pas toujours le cas. Si vous constatez que vous avez besoin d’un modèle d’élément JSON, associez-le simplement à votre bloc JSON et enregistrez-le dans src\main\resources\assets\example\models\item\ avec le même nom de fichier que le nom de registre du bloc.

{
    "parent": "example:block/my_block"
}
