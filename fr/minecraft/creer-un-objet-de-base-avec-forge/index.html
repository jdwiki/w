Créer un objet de base avec Forge

  • Classe d’articles
  • Modèle d’article
  • Enregistrement de l’article

Sur cette page

  • Classe d’articles
  • Modèle d’article
  • Enregistrement de l’article

Créer un objet simple avec Forge est l’une des premières tâches qu’un moddeur en herbe devra apprendre. Comment faire cela a changé au fil des différentes versions de Minecraft et est probablement à une difficulté “modérée” après la 1.7.10 en raison du grand nombre d’erreurs faciles à commettre, en particulier pour le rendre correctement.

Si quelque chose ne va pas et que votre élément personnalisé a une texture (noir/violet) ou un modèle manquant (cube par défaut qui est trop grand lorsqu’il est tenu), consultez le journal. Les problèmes de ce type apparaîtront presque * toujours * dans le journal, à condition que vous ayez correctement enregistré les choses.

La première chose qui apparaîtra est la ligne "Modèle de chargement d'exception pour la variante..." ou similaire, vous indiquant quel bloc ou élément n’a pas pu être chargé correctement. Après une douzaine de lignes commençant par at..., il y aura une autre ligne commençant par Causé par...

Cette ligne “Causé par” est la plus importante, elle vous indiquera quel fichier n’a pas pu être chargé correctement et peut être l’une des nombreuses erreurs, telles que :

  • JSON malformé (votre fichier JSON n’est pas valide et contient une faute de frappe)
  • File Not Found (le fichier recherché par Minecraft n’est pas correctement nommé ou au bon endroit)
  • Exception de variante manquante (votre Blockstate JSON est incomplet)

Vous pouvez même obtenir plus d’une erreur d’échec pour une seule erreur ! Si vous ne savez pas quel élément est le problème, commentez chaque bloc et élément dont vous savez qu’il fonctionne, réduisez la liste d’erreurs au bloc (ou aux blocs) en question. Vous devrez peut-être le faire plusieurs fois.

De plus, une texture de mélange apparaît aussi différemment qu’une simple liste de toutes les ressources manquantes pour un domaine donné (mod ID).

Classe d’articles #

Cette partie n’a pas beaucoup changé au fil des versions de Minecraft, bien qu’il y ait eu quelques mutations dans les signatures de méthode exactes ainsi que dans la hiérarchie des classes. Un objet de base (qui n’a aucune fonctionnalité, comme un bâton ou un lingot : c’est vrai, les deux sont des objets qui ne font rien !)

public class CustomItem extends Item {

    public CustomItem () {
        super();
        this.setMaxDamage(0);
        this.setCreativeTab(CreativeTabs.MISC);
    }
}

Pas beaucoup de place pour la personnalisation à ce stade, contrairement aux blocs. À peu près tout ce que nous pouvons faire est de changer si l’élément peut ou non subir des dommages (les outils l’utilisent) et dans quel onglet créatif il existera. Le nom et la texture que nous gérerons lorsque nous enregistrerons l’élément auprès du GameRegistry.

Cependant, c’est tout ce qui est nécessaire pour tenir, transporter, laisser tomber, fabriquer, fondre et autrement utiliser l’objet. Certains objets dans Minecraft (comme les bâtons) n’ont même pas de classe unique et utilisent simplement new Item(). Nous pourrions le faire ici, mais tout élément avec des fonctionnalités supplémentaires aura besoin d’une classe.

Modèle d’article #

Comme pour les blocs, les objets ont également besoin de modèles.

{
    "parent": "item/generated",
    "textures": {
        "layer0": "example:items/basic"
    }
}

C’est à peu près tout ce qui est nécessaire pour que cela fonctionne une fois l’article enregistré. La seule chose importante est que le nom de fichier corresponde au nom de registre utilisé pour enregistrer le bloc et doit être en minuscules (les noms de fichiers 1.11+ doivent être en minuscules, avant cela, il est juste sensible à la casse).

Notez que “layer0” est la seule texture nécessaire et il est très peu probable qu’une autre texture soit spécifiée (bien que certains objets comme les potions et l’armure de cuir aient un “layer1”). Tous les noms sont définis par item/builtin (le modèle parent interne le plus élevé pour les éléments) contrairement aux blocs.

Nommez le fichier JSON modèle my_item.json (correspondant au nom de registre que nous allons lui donner plus tard) et enregistrez-le dans src\main\resources\assets\example\models\item\ (où example est l’ID de mod spécifié dans l’annotation @Mod de votre classe de mod principale).

Créez également une texture pour votre élément, nommez-le basic.png et enregistrez-le dans src\main\resources\assets\example\textures\items\

Le modèle d’objet utilise ici un parent d’item/generated, ce qui signifie que la seule texture fournie sera utilisée (comme avec la plupart des objets non-blocs) et sera tenue dans la main du joueur dans une orientation par défaut. Il existe également un élément / ordinateur de poche qui spécifie différentes orientations d’affichage (pour les outils). Les éléments peuvent également fournir leur propre attribut “affichage”, remplaçant ceux du parent, mais n’est pas nécessaire dans 99,9 % des utilisations.

Enregistrement de l’article #

L’enregistrement des éléments est effectué à partir de votre classe de mod principale ou d’une méthode de classe ModItems invoquée à partir de la classe de mod principale pendant preInit.

Item item = new CustomItem();
string registryname = "my_item";
item.setRegistryName(registryname);
item.setUnlocalizedName(item.getRegistryName().toString());
GameRegistry.register(item);

Il existe une raison importante d’utiliser également item.setUnlocalizedName(item.getRegistryName().toString()); ! Il garantit que le nom non localisé de votre article contient l’ID du mod pour éviter les conflits de fichiers de langue entre les mods.

Maintenant, l’élément a également besoin d’un modèle, et c’est là que les choses se sont compliquées après la version 1.7.10, qui impliquait simplement de dire à Minecraft le nom de la texture à charger et qui pourrait être spécifié dans le constructeur de l’élément.

final ModelResourceLocation fullModelLocation = new ModelResourceLocation(item.getRegistryName().toString(), "inventory");
ModelBakery.registerItemVariants(item, fullModelLocation);
ModelLoader.setCustomMeshDefinition(item, new ItemMeshDefinition()
    {
        public ModelResourceLocation getModelLocation(ItemStack stack)
        {
            return fullModelLocation;
        }
    });

Notez que cette section doit être située côté client uniquement (c’est-à-dire le proxy client) car la plupart des classes référencées n’existent pas sur le serveur dédié.

Enregistrement d’articles avec variantes, par ex. Saplings, doit être fait d’une manière différente, en utilisant ModelLoader.setCustomModelResourceLocation(item, metadata, resourceLocation) bien qu’il nous permette d’utiliser un fichier Blockstate pour spécifier nos variantes (ce qui est beaucoup préféré à l’alternative). Notre article n’utilise pas de variantes, nous avons donc terminé.
