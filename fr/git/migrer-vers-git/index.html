Sous-Git #

[SubGit][1] peut être utilisé pour effectuer une importation unique d’un référentiel SVN vers git.

$ subgit import --non-interactive --svn-url http://svn.my.co/repos/myproject myproject.git

[1] : http://www.subgit.com/remote-book.html#7

Migrer de SVN vers Git à l’aide de l’utilitaire de conversion Atlassian #

Téléchargez l’utilitaire de conversion Atlassian ici. Cet utilitaire nécessite Java, veuillez donc vous assurer que l’environnement d’exécution Java JRE est installé sur la machine sur laquelle vous prévoyez d’effectuer la conversion.

Utilisez la commande java -jar svn-migration-scripts.jar verify pour vérifier s’il manque à votre machine l’un des programmes nécessaires pour terminer la conversion. Plus précisément, cette commande recherche les utilitaires Git, subversion et git-svn. Il vérifie également que vous effectuez la migration sur un système de fichiers sensible à la casse. La migration vers Git doit être effectuée sur un système de fichiers sensible à la casse pour éviter de corrompre le référentiel.

Ensuite, vous devez générer un fichier auteurs. Subversion suit les modifications par le nom d’utilisateur du committer uniquement. Git, cependant, utilise deux informations pour distinguer un utilisateur : un vrai nom et une adresse e-mail. La commande suivante générera un fichier texte mappant les noms d’utilisateur subversion à leurs équivalents Git :

java -jar svn-migration-scripts.jar authors <svn-repo> authors.txt

où <svn-repo> est l’URL du dépôt subversion que vous souhaitez convertir. Après avoir exécuté cette commande, les informations d’identification des contributeurs seront mappées dans authors.txt. Les adresses e-mail seront de la forme <username>@mycompany.com. Dans le fichier des auteurs, vous devrez remplacer manuellement le nom par défaut de chaque personne (qui par défaut est devenu son nom d’utilisateur) par son nom réel. Assurez-vous également de vérifier l’exactitude de toutes les adresses e-mail avant de continuer.

La commande suivante clonera un dépôt svn en tant que dépôt Git :

git svn clone --stdlayout --authors-file=authors.txt <svn-repo> <git-repo-name>

où <svn-repo> est la même URL de référentiel utilisée ci-dessus et <git-repo-name> est le nom du dossier dans le répertoire actuel dans lequel cloner le référentiel. Il y a quelques considérations avant d’utiliser cette commande :

  • Le drapeau --stdlayout ci-dessus indique à Git que vous utilisez une mise en page standard avec les dossiers trunk, branches et tags. Les référentiels Subversion avec des mises en page non standard nécessitent que vous spécifiiez les emplacements du dossier trunk, de tous les dossiers branch et du dossier tags. Cela peut être fait en suivant cet exemple : git svn clone --trunk=/trunk --branches=/branches --branches=/bugfixes --tags=/tags --authors-file=authors.txt <svn-repo > <git-repo-name>.
  • Cette commande peut prendre plusieurs heures en fonction de la taille de votre dépôt.
  • Pour réduire le temps de conversion pour les dépôts volumineux, la conversion peut être exécutée directement sur le serveur hébergeant le dépôt subversion afin d’éliminer la surcharge du réseau.

git svn clone importe les branches subversion (et le tronc) en tant que branches distantes, y compris les balises subversion (branches distantes préfixées par tags/). Pour les convertir en branches et balises réelles, exécutez les commandes suivantes sur une machine Linux dans l’ordre dans lequel elles sont fournies. Après les avoir exécutés, git branch -a devrait afficher les noms de branche corrects et git tag -l devrait afficher les balises du référentiel.

git for-each-ref refs/remotes/origin/tags | cut -d / -f 5- | grep -v @ | while read tagname; do git tag $tagname origin/tags/$tagname; git branch -r -d origin/tags/$tagname; done
git for-each-ref refs/remotes | cut -d / -f 4- | grep -v @ | while read branchname; do git branch "$branchname" "refs/remotes/origin/$branchname"; git branch -r -d "origin/$branchname"; done

La conversion de svn vers Git est maintenant terminée ! “Poussez” simplement votre dépôt local sur un serveur et vous pouvez continuer à contribuer en utilisant Git tout en ayant un historique de version complètement préservé de svn.

Migrer de Team Foundation Version Control (TFVC) vers Git #

Vous pouvez migrer du contrôle de version de la fondation d’équipe vers git en utilisant un outil open source appelé Git-TF. La migration transférera également votre historique existant en convertissant les checkins tfs en commits git.

Pour mettre votre solution dans Git à l’aide de Git-TF, suivez ces étapes :

Télécharger Git-TF

Vous pouvez télécharger (et installer) Git-TF depuis Codeplex : [Git-TF @ Codeplex][1]

Clonez votre solution TFVC

Lancez powershell (win) et tapez la commande

git-tf clone http://my.tfs.server.address:port/tfs/mycollection '$/myproject/mybranch/mysolution' --deep

Le commutateur –deep est le mot clé à noter car il indique à Git-Tf de copier votre historique d’archivage. Vous avez maintenant un référentiel git local dans le dossier à partir duquel vous avez appelé votre commande cloe.

Nettoyer

  • Ajoutez un fichier .gitignore. Si vous utilisez Visual Studio, l’éditeur peut le faire pour vous, sinon vous pouvez le faire manuellement en téléchargeant un fichier complet depuis [github/gitignore][2].
  • Supprimer les liaisons de contrôle de source TFS de la solution (supprimer tous les fichiers *.vssscc). Vous pouvez également modifier votre fichier de solution en supprimant GlobalSection(TeamFoundationVersionControl)……EndClobalSection

Engagez-vous et poussez

Terminez votre conversion en validant et en poussant votre référentiel local vers votre télécommande.

git add .
git commit -a -m "Coverted solution source control from TFVC to Git"

git remote add origin https://my.remote/project/repo.git

git push origin master

[1] : https://gittf.codeplex.com/ [2] : https://github.com/github/gitignore

Migrer Mercurial vers Git #

On peut utiliser les méthodes suivantes pour importer un dépôt Mercurial dans Git :

  1. Utilisation de [exportation rapide] (https://github.com/frej/fast-export) :

    cd git clone git://repo.or.cz/fast-export.git git init git_repo cd git_repo ~/fast-export/hg-fast-export.sh -r /path/to/old/mercurial_repo git checkout HEAD

  2. En utilisant Hg-Git : Une réponse très détaillée ici : https://stackoverflow.com/a/31827990/5283213

  3. Utilisation de [l’importateur de GitHub] (https://help.github.com/articles/about-github-importer/) : Suivez les instructions (détaillées) sur GitHub.

Migrer de SVN vers Git en utilisant svn2git #

svn2git est un wrapper Ruby autour du support SVN natif de git via git-svn, aidant vous avec la migration de projets de Subversion vers Git, en conservant l’historique (y compris l’historique du tronc, des balises et des branches).

Exemples

Pour migrer un référentiel svn avec la disposition standard (c’est-à-dire les branches, les balises et le tronc au niveau racine du référentiel) :

$ svn2git http://svn.example.com/path/to/repo

Pour migrer un dépôt svn qui n’est pas dans la mise en page standard :

$ svn2git http://svn.example.com/path/to/repo --trunk trunk-dir --tags tags-dir --branches branches-dir

Si vous ne souhaitez pas migrer (ou n’avez pas) de branches, de balises ou de tronc, vous pouvez utiliser les options --notrunk, --nobranches et --notags.

Par exemple, $ svn2git http://svn.example.com/path/to/repo --trunk trunk-dir --notags --nobranches ne migrera que l’historique du tronc.

Pour réduire l’espace requis par votre nouveau référentiel, vous souhaiterez peut-être exclure tous les répertoires ou fichiers que vous avez ajoutés alors que vous ne devriez pas en avoir (par exemple, répertoire de construction ou archives):

$ svn2git http://svn.example.com/path/to/repo --exclude build --exclude '.*\.zip$'

Optimisation post-migration

Si vous avez déjà quelques milliers de commits (ou plus) dans votre référentiel git nouvellement créé, vous souhaiterez peut-être réduire l’espace utilisé avant de pousser votre référentiel sur une télécommande. Cela peut être fait en utilisant la commande suivante :

$ git gc --aggressive

Remarque : La commande précédente peut prendre jusqu’à plusieurs heures sur des référentiels volumineux (des dizaines de milliers de commits et/ou des centaines de mégaoctets d’historique).
