Revenir à un commit précédent #

Pour revenir à un commit précédent, recherchez d’abord le hachage du commit à l’aide de [git log][1].

Pour revenir temporairement à ce commit, détachez votre tête avec :

git checkout 789abcd

Cela vous place au commit 789abcd. Vous pouvez maintenant créer de nouveaux commits par-dessus cet ancien commit sans affecter la branche sur laquelle se trouve votre tête. Toute modification peut être apportée à une branche appropriée en utilisant [soit branch ou checkout -b][2].

Pour revenir à un commit précédent tout en conservant les modifications :

git reset --soft 789abcd

Pour annuler le dernier commit :

git reset --soft HEAD~

Pour supprimer définitivement toutes les modifications apportées après un commit spécifique, utilisez :

git reset --hard 789abcd

Pour supprimer définitivement toutes les modifications apportées après le dernier commit :

git reset --hard HEAD~

Attention : Bien que vous puissiez [récupérer les commits supprimés à l’aide de reflog et reset][3], les modifications non validées ne peuvent pas être récupérées. Utilisez [git stash; git reset][4] au lieu de git reset --hard pour être sûr.

[1] : https://www.wikiod.com/fr/git/parcourir-lhistorique [2] : https://www.wikiod.com/fr/git/ramification#Création et vérification de nouvelles branches [3] : https://www.wikiod.com/fr/git/recuperer#Récupération après une réinitialisation [4] : https://www.wikiod.com/fr/git/cachette

Annuler les modifications #

Annulez les modifications apportées à un fichier ou à un répertoire dans la copie de travail.

git checkout -- file.txt

Utilisé sur tous les chemins de fichiers, de manière récursive à partir du répertoire courant, il annulera toutes les modifications dans la copie de travail.

git checkout -- .

Pour annuler uniquement certaines parties des modifications, utilisez --patch. Il vous sera demandé, pour chaque modification, si elle doit être annulée ou non.

git checkout --patch -- dir

Pour annuler les modifications ajoutées à l’index.

git reset --hard

Sans le drapeau --hard, cela effectuera une réinitialisation logicielle.

Avec les commits locaux que vous n’avez pas encore poussés vers une télécommande, vous pouvez également effectuer une réinitialisation logicielle. Vous pouvez ainsi retravailler les fichiers puis les commits.

git reset HEAD~2

L’exemple ci-dessus déroulerait vos deux derniers commits et renverrait les fichiers dans votre copie de travail. Vous pouvez ensuite apporter d’autres modifications et de nouveaux commits.

Attention : Toutes ces opérations, à l’exception des réinitialisations logicielles, supprimeront définitivement vos modifications. Pour une option plus sûre, utilisez respectivement git stash -p ou git stash. Vous pouvez ensuite annuler avec stash pop ou supprimer définitivement avec stash drop.

Annuler les fusions #

Annulation d’une fusion pas encore poussée vers une télécommande

Si vous n’avez pas encore poussé votre fusion vers le référentiel distant, vous pouvez suivre la même procédure que dans [annuler la validation] [1] bien qu’il existe quelques différences subtiles.

Une réinitialisation est l’option la plus simple car elle annulera à la fois le commit de fusion et tous les commits ajoutés à partir de la branche. Cependant, vous devrez savoir sur quel SHA réinitialiser, cela peut être délicat car votre git log affichera désormais les commits des deux branches. Si vous réinitialisez sur le mauvais commit (par exemple un sur l’autre branche) ** cela peut détruire le travail validé. **

> git reset --hard <last commit from the branch you are on>

Ou, en supposant que la fusion était votre commit le plus récent.

> git reset HEAD~

Un retour est plus sûr, dans la mesure où il ne détruira pas le travail validé, mais implique plus de travail car vous devez annuler le retour avant de pouvoir fusionner à nouveau la branche (voir la section suivante).

Annulation d’une fusion poussée vers une télécommande

Supposons que vous fusionnez dans une nouvelle fonctionnalité (add-gremlins)

> git merge feature/add-gremlins
...
   #Resolve any merge conflicts
> git commit #commit the merge
...
> git push
...
   501b75d..17a51fd  master -> master

Ensuite, vous découvrez que la fonctionnalité que vous venez de fusionner a cassé le système pour les autres développeurs, elle doit être annulée immédiatement, et la correction de la fonctionnalité elle-même prendra trop de temps, vous voulez donc simplement annuler la fusion.

> git revert -m 1 17a51fd
...
> git push
...
   17a51fd..e443799  master -> master

À ce stade, les gremlins sont hors du système et vos collègues développeurs ont cessé de vous crier dessus. Cependant, nous n’avons pas encore fini. Une fois que vous avez résolu le problème avec la fonction add-gremlins, vous devrez annuler ce retour avant de pouvoir fusionner.

> git checkout feature/add-gremlins
...
   #Various commits to fix the bug.
> git checkout master
...
> git revert e443799
...
> git merge feature/add-gremlins
...
   #Fix any merge conflicts introduced by the bug fix
> git commit #commit the merge
...
> git push

À ce stade, votre fonctionnalité est maintenant ajoutée avec succès. Cependant, étant donné que les bogues de ce type sont souvent introduits par des conflits de fusion, un workflow légèrement différent est parfois plus utile car il vous permet de résoudre le conflit de fusion sur votre branche.

> git checkout feature/add-gremlins
...
   #Merge in master and revert the revert right away.  This puts your branch in
   #the same broken state that master was in before.
> git merge master
...
> git revert e443799
...
   #Now go ahead and fix the bug (various commits go here)
> git checkout master
...
   #Don't need to revert the revert at this point since it was done earlier
> git merge feature/add-gremlins
...
   #Fix any merge conflicts introduced by the bug fix
> git commit #commit the merge
...
> git push

[1] : https://www.wikiod.com/fr/git/perte

Utiliser le reflog #

Si vous bousillez un rebase, une option pour recommencer est de revenir au commit (pre rebase). Vous pouvez le faire en utilisant reflog (qui a l’historique de tout ce que vous avez fait au cours des 90 derniers jours - cela peut être configuré):

$ git reflog
4a5cbb3 HEAD@{0}: rebase finished: returning to refs/heads/foo
4a5cbb3 HEAD@{1}: rebase: fixed such and such
904f7f0 HEAD@{2}: rebase: checkout upstream/master
3cbe20a HEAD@{3}: commit: fixed such and such
...

Vous pouvez voir le commit avant que le rebase ne soit HEAD@{3} (vous pouvez également vérifier le hachage) :

git checkout HEAD@{3}
  • Maintenant, vous créez une nouvelle branche / supprimez l’ancienne / essayez à nouveau de rebaser. *

Vous pouvez également réinitialiser directement à un point de votre reflog, mais ne le faites que si vous êtes sûr à 100 % que c’est ce que vous voulez faire :

git reset --hard HEAD@{3}

Cela définira votre arbre git actuel pour qu’il corresponde à ce qu’il était à ce moment-là (voir Annuler les modifications).

Cela peut être utilisé si vous constatez temporairement le bon fonctionnement d’une branche lorsqu’elle est rebasée sur une autre branche, mais que vous ne souhaitez pas conserver les résultats.

Rétablir certains commits existants #

Utilisez git revert pour annuler les commits existants, en particulier lorsque ces commits ont été poussés vers un référentiel distant. Il enregistre de nouveaux commits pour inverser l’effet de certains commits antérieurs, que vous pouvez pousser en toute sécurité sans réécrire l’historique.

N’utilisez pas git push --force sauf si vous souhaitez faire tomber l’opprobre de tous les autres utilisateurs de ce référentiel. Ne jamais réécrire l’histoire publique.

Si, par exemple, vous venez de pousser un commit contenant un bogue et que vous devez l’annuler, procédez comme suit :

git revert HEAD~1
git push

Vous êtes maintenant libre d’annuler le commit de retour localement, de corriger votre code et de pousser le bon code :

git revert HEAD~1
work .. work .. work ..
git add -A .
git commit -m "Update error code"
git push

Si le commit que vous souhaitez annuler est déjà plus loin dans l’historique, vous pouvez simplement passer le hachage du commit. Git créera un contre-commit annulant votre commit d’origine, que vous pourrez envoyer à votre télécommande en toute sécurité.

git revert 912aaf0228338d0c8fb8cca0a064b0161a451fdc
git push
