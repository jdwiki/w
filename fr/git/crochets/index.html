Pré-pousser #

Disponible dans Git 1.8.2 et supérieur.

Des crochets de pré-poussée peuvent être utilisés pour empêcher une poussée de passer. Les raisons pour lesquelles cela est utile incluent : le blocage des poussées manuelles accidentelles vers des branches spécifiques ou le blocage des poussées si une vérification établie échoue (tests unitaires, syntaxe).

Un hook pré-push est créé en créant simplement un fichier nommé pre-push sous .git/hooks/, et (gotcha alert), en s’assurant que le fichier est exécutable : chmod +x ./ git/hooks/pre-push.

Voici un exemple de Hannah Wolfe qui bloque un push to master :

#!/bin/bash

protected_branch='master'  
current_branch=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')

if [ $protected_branch = $current_branch ]  
then  
    read -p "You're about to push master, is that what you intended? [y|n] " -n 1 -r < /dev/tty
    echo
    if echo $REPLY | grep -E '^[Yy]$' > /dev/null
    then
        exit 0 # push will execute
    fi
    exit 1 # push will not execute
else  
    exit 0 # push will execute
fi

Voici un exemple de Volkan Unsal qui s’assure Les tests RSpec passent avant d’autoriser le push :

#!/usr/bin/env ruby
require 'pty'
html_path = "rspec_results.html"
begin
  PTY.spawn( "rspec spec --format h > rspec_results.html" ) do |stdin, stdout, pid|
  begin
    stdin.each { |line| print line }
  rescue Errno::EIO
  end
end
rescue PTY::ChildExited
  puts "Child process exit!"
end

# find out if there were any errors  
html = open(html_path).read
examples = html.match(/(\d+) examples/)[0].to_i rescue 0
errors = html.match(/(\d+) errors/)[0].to_i rescue 0
if errors == 0 then
  errors = html.match(/(\d+) failure/)[0].to_i rescue 0
end
pending = html.match(/(\d+) pending/)[0].to_i rescue 0

if errors.zero?
  puts "0 failed! #{examples} run, #{pending} pending"
  # HTML Output when tests ran successfully:
  # puts "View spec results at #{File.expand_path(html_path)}"
  sleep 1
  exit 0
else
  puts "\aCOMMIT FAILED!!"
  puts "View your rspec results at #{File.expand_path(html_path)}"
  puts
  puts "#{errors} failed! #{examples} run, #{pending} pending"
  # Open HTML Ooutput when tests failed
  # `open #{html_path}`
  exit 1
end

Comme vous pouvez le voir, il y a beaucoup de possibilités, mais l’essentiel est de “sortir 0” si de bonnes choses se sont produites et de “sortir 1” si de mauvaises choses se sont produites. Chaque fois que vous exit 1, le push sera empêché et votre code sera dans l’état où il était avant d’exécuter git push....

Lorsque vous utilisez des hooks côté client, gardez à l’esprit que les utilisateurs peuvent ignorer tous les hooks côté client en utilisant l’option “–no-verify” sur un push. Si vous comptez sur le crochet pour appliquer le processus, vous pouvez vous brûler.

Documentation : https://git-scm.com/docs/githooks#_pre_push Échantillon officiel : https://github.com/git/git/blob/87c86dd14abe8db7d00b0df5661ef8cf147a72a3/templates/hooks–pre-push.sample

Vérifiez la construction Maven (ou un autre système de construction) avant de valider #

.git/hooks/pre-commit

#!/bin/sh
if [ -s pom.xml ]; then
    echo "Running mvn verify"
    mvn clean verify
    if [ $? -ne 0 ]; then
        echo "Maven build failed"
        exit 1
    fi
fi

Transférer automatiquement certaines poussées vers d’autres référentiels #

Les hooks post-receive peuvent être utilisés pour transférer automatiquement les pushs entrants vers un autre référentiel.

$ cat .git/hooks/post-receive

#!/bin/bash

IFS=' '
while read local_ref local_sha remote_ref remote_sha
do

  echo "$remote_ref" | egrep '^refs\/heads\/[A-Z]+-[0-9]+$' >/dev/null && {
    ref=`echo $remote_ref | sed -e 's/^refs\/heads\///'`
    echo Forwarding feature branch to other repository: $ref
    git push -q --force other_repos $ref
  }

done

Dans cet exemple, l’expression régulière egrep recherche un format de branche spécifique (ici : JIRA-12345 utilisé pour nommer les problèmes Jira). Vous pouvez laisser cette partie désactivée si vous souhaitez transférer toutes les branches, bien sûr.

Commit-msg #

Ce crochet est similaire au crochet prepare-commit-msg, mais il est appelé après que l’utilisateur a saisi un message de validation plutôt qu’avant. Ceci est généralement utilisé pour avertir les développeurs si leur message de validation est dans un format incorrect.

Le seul argument passé à ce crochet est le nom du fichier qui contient le message. Si vous n’aimez pas le message que l’utilisateur a entré, vous pouvez soit modifier ce fichier sur place (comme prepare-commit-msg) ou vous pouvez abandonner complètement la validation en quittant avec un statut différent de zéro.

L’exemple suivant permet de vérifier si le mot ticket suivi d’un chiffre est présent sur le message de commit

word="ticket [0-9]"
isPresent=$(grep -Eoh "$word" $1)

if [[ -z $isPresent ]]
  then echo "Commit message KO, $word is missing"; exit 1;
  else echo "Commit message OK"; exit 0;
fi

Crochets locaux #

Les crochets locaux n’affectent que les référentiels locaux dans lesquels ils résident. Chaque développeur peut modifier ses propres crochets locaux, de sorte qu’ils ne peuvent pas être utilisés de manière fiable pour appliquer une politique de validation. Ils sont conçus pour permettre aux développeurs de respecter plus facilement certaines directives et d’éviter les problèmes potentiels en cours de route.

Il existe six types de hooks locaux : pré-commit, prepare-commit-msg, commit-msg, post-commit, post-checkout et pre-rebase.

Les quatre premiers crochets concernent les commits et vous permettent d’avoir un certain contrôle sur chaque partie du cycle de vie d’un commit. Les deux derniers vous permettent d’effectuer des actions supplémentaires ou des contrôles de sécurité pour les commandes git checkout et git rebase.

Tous les crochets “pré-” vous permettent de modifier l’action qui est sur le point d’avoir lieu, tandis que les crochets “post-” sont principalement utilisés pour les notifications.

Post-paiement #

Ce crochet fonctionne de la même manière que le crochet post-commit, mais il est appelé chaque fois que vous avez réussi à extraire une référence avec git checkout. Cela pourrait être un outil utile pour vider votre répertoire de travail des fichiers générés automatiquement qui, autrement, causeraient de la confusion.

Ce crochet accepte trois paramètres :

  1. la réf du HEAD précédent,
  2. la réf du nouveau HEAD, et
  3. un indicateur indiquant s’il s’agissait d’une extraction de branche ou d’une extraction de fichier (“1” ou “0”, respectivement).

Son état de sortie n’a aucun effet sur la commande git checkout.

Post-commit #

Ce hook est appelé immédiatement après le hook commit-msg. Il ne peut pas modifier le résultat de l’opération git commit, il est donc principalement utilisé à des fins de notification.

Le script ne prend aucun paramètre et son état de sortie n’affecte en rien la validation.

Post-réception #

Ce hook est appelé après une opération push réussie. Il est généralement utilisé à des fins de notification.

Le script ne prend aucun paramètre, mais reçoit les mêmes informations que pre-receive via l’entrée standard :

<old-value> <new-value> <ref-name>

Pré-commit #

Ce hook est exécuté à chaque fois que vous lancez git commit, pour vérifier ce qui est sur le point d’être validé. Vous pouvez utiliser ce crochet pour inspecter l’instantané qui est sur le point d’être validé.

Ce type de crochet est utile pour exécuter des tests automatisés afin de s’assurer que le commit entrant ne brise pas les fonctionnalités existantes de votre projet. Ce type de crochet peut également vérifier les espaces blancs ou les erreurs EOL.

Aucun argument n’est passé au script de pré-validation et la sortie avec un statut différent de zéro annule la validation entière.

Prepare-commit-msg #

Ce crochet est appelé après le crochet pre-commit pour remplir l’éditeur de texte avec un message de validation. Ceci est généralement utilisé pour modifier les messages de validation générés automatiquement pour les validations écrasées ou fusionnées.

Un à trois arguments sont passés à ce hook :

  • Le nom d’un fichier temporaire qui contient le message.
  • Le type de commit, soit
  • message (option -m ou -F),
  • modèle (option -t),
  • merge (s’il s’agit d’un commit de fusion), ou
  • squash (s’il écrase d’autres commits).
  • Le hachage SHA1 du commit concerné. Ceci n’est donné que si l’option -c, -C ou --amend a été donnée.

Semblable à pre-commit, quitter avec un statut différent de zéro annule la validation.

Pré-rebase #

Ce hook est appelé avant que git rebase ne commence à modifier la structure du code. Ce hook est généralement utilisé pour s’assurer qu’une opération de rebase est appropriée.

Ce hook prend 2 paramètres :

  1. la branche en amont à partir de laquelle la série a été dérivée, et
  2. la branche en cours de rebasage (vide lors du rebasage de la branche courante).

Vous pouvez abandonner l’opération de rebase en quittant avec un état différent de zéro.

Pré-réception #

Ce crochet est exécuté chaque fois que quelqu’un utilise git push pour envoyer des commits au référentiel. Il réside toujours dans le référentiel distant qui est la destination du push et non dans le référentiel d’origine (local).

Le hook s’exécute avant la mise à jour des références. Il est généralement utilisé pour appliquer tout type de politique de développement.

Le script ne prend aucun paramètre, mais chaque ref poussée est transmise au script sur une ligne distincte sur l’entrée standard au format suivant :

<old-value> <new-value> <ref-name>

Mise à jour #

Ce hook est appelé après pre-receive, et il fonctionne de la même manière. Il est appelé avant que quoi que ce soit ne soit réellement mis à jour, mais il est appelé séparément pour chaque référence qui a été poussée plutôt que pour toutes les références à la fois.

Ce hook accepte les 3 arguments suivants :

  • nom de la ref en cours de mise à jour,
  • ancien nom d’objet stocké dans la référence, et
  • nouveau nom d’objet stocké dans la réf.

Il s’agit des mêmes informations transmises à pre-receive, mais puisque update est appelé séparément pour chaque référence, vous pouvez rejeter certaines références tout en en autorisant d’autres.
