De nombreuses commandes Git prennent des paramètres de révision comme arguments. Selon la commande, ils indiquent un commit spécifique ou, pour les commandes qui parcourent le graphe de révision (comme [git-log(1)][git-log]), tous les commits qui peuvent être atteints à partir de ce commit . Ils sont généralement désignés par <commit>, ou <rev>, ou <revision> dans la description de la syntaxe.

La documentation de référence pour la syntaxe des révisions Git est la page de manuel [gitrevisions(7)][gitrevisions].

[git-log] : https://www.kernel.org/pub/software/scm/git/docs/git-log.html [gitrevisions] : https://www.kernel.org/pub/software/scm/git/docs/gitrevisions.html

Toujours absent de cette page :

  • [_] Sortie de git describe, par ex. v1.7.4.2-679-g3bee7fb
  • [_] @ seul comme raccourci pour HEAD
  • [_] @{-<n>}, par ex. @{-1} et - signifiant @{-1}
  • [_] <nom de la branche>@{push}
  • [_] <rev>^@, pour tous les parents de <rev>

Nécessite une documentation distincte :

  • [_] Se référant aux blobs et aux arbres dans le référentiel et dans l’index : <rev>:<path> et :<n>:<path> syntaxe
  • [_] Plages de révision comme A..B, A...B, B ^A, A^1 et limitation de révision comme -<n>, --since

Spécifier la révision par nom d’objet #

$ git show dae86e1950b1277e545cee180551750029cfe735
$ git show dae86e19

Vous pouvez spécifier la révision (ou en vérité n’importe quel objet : balise, arbre, c’est-à-dire le contenu du répertoire, blob, c’est-à-dire le contenu du fichier) en utilisant le nom d’objet SHA-1, soit une chaîne hexadécimale complète de 40 octets, soit une sous-chaîne unique au référentiel.

Noms de références symboliques : branches, balises, branches de suivi à distance #

$ git log master    # specify branch
$ git show v1.0     # specify tag
$ git show HEAD     # specify current branch
$ git show origin   # specify default remote-tracking branch for remote 'origin'

Vous pouvez spécifier la révision à l’aide d’un nom de référence symbolique, qui inclut des branches (par exemple ‘master’, ’next’, ‘maint’), des balises (par exemple ‘v1.0’, ‘v0.6.3-rc2’), remote- suivi des branches (par exemple ‘origin’, ‘origin/master’) et des références spéciales telles que ‘HEAD’ pour la branche actuelle.

Si le nom symbolique de la référence est ambigu, par exemple si vous avez à la fois une branche et une balise nommées “fix” (avoir une branche et une balise avec le même nom n’est pas recommandé), vous devez spécifier le type de référence que vous souhaitez utiliser :

$ git show heads/fix      # or 'refs/heads/fix', to specify branch
$ git show tags/fix       # or 'refs/tags/fix', to specify tag

La révision par défaut : HEAD #

$ git show         # equivalent to 'git show HEAD'

‘HEAD’ nomme le commit sur lequel vous avez basé les changements dans l’arborescence de travail, et est généralement le nom symbolique de la branche actuelle. De nombreuses commandes (mais pas toutes) qui prennent le paramètre de révision par défaut à ‘HEAD’ s’il est manquant.

Références de reflog : @{} #

$ git show @{1}            # uses reflog for current branch
$ git show master@{1}      # uses reflog for branch 'master'
$ git show HEAD@{1}        # uses 'HEAD' reflog

Une ref, généralement une branche ou HEAD, suivie du suffixe @ avec une spécification ordinale entre accolades (par exemple {1}, {15}) spécifie la n-ième valeur précédente de cette ref * dans votre dépôt local*. Vous pouvez vérifier les entrées de reflog récentes avec la commande [git reflog][git-reflog] ou l’option --walk-reflogs / -g de git log.

$ git reflog
08bb350 HEAD@{0}: reset: moving to HEAD^
4ebf58d HEAD@{1}: commit: gitweb(1): Document query parameters
08bb350 HEAD@{2}: pull: Fast-forward
f34be46 HEAD@{3}: checkout: moving from af40944bda352190f05d22b7cb8fe88beb17f3a7 to master
af40944 HEAD@{4}: checkout: moving from master to v2.6.3

$ git reflog gitweb-docs
4ebf58d gitweb-docs@{0}: branch: Created from master

Remarque : l’utilisation de reflogs a pratiquement remplacé l’ancien mécanisme d’utilisation de la référence ORIG_HEAD (à peu près équivalent à HEAD@{1}).

[git-reflog] : https://www.kernel.org/pub/software/scm/git/docs/git-reflog.html

Références de reflog : @{} #

$ git show master@{yesterday}
$ git show HEAD@{5 minutes ago}   # or HEAD@{5.minutes.ago}

Une référence suivie du suffixe @ avec une spécification de date entre accolades (par exemple, {hier}, {il y a 1 mois 2 semaines 3 jours 1 heure 1 seconde} ou {1979-02-26 18 :30:00}) spécifie la valeur de la référence à un moment antérieur (ou au point le plus proche de celui-ci). Notez que cela recherche l’état de votre référence locale à un moment donné ; par exemple, ce qu’il y avait dans votre branche locale ‘master’ la semaine dernière.

Vous pouvez utiliser [git reflog][git-reflog] avec un spécificateur de date pour rechercher l’heure exacte à laquelle vous avez fait quelque chose à une référence donnée dans le référentiel local.

$ git reflog HEAD@{now}
08bb350 HEAD@{Sat Jul 23 19:48:13 2016 +0200}: reset: moving to HEAD^
4ebf58d HEAD@{Sat Jul 23 19:39:20 2016 +0200}: commit: gitweb(1): Document query parameters
08bb350 HEAD@{Sat Jul 23 19:26:43 2016 +0200}: pull: Fast-forward

[git-reflog] : https://www.kernel.org/pub/software/scm/git/docs/git-reflog.html

Branche suivie / en amont : @{upstream} #

$ git log @{upstream}..       # what was done locally and not yet published, current branch
$ git show master@{upstream}  # show upstream of branch 'master'

Le suffixe @{upstream} ajouté à un nom de branche (forme courte <branchname>@{u}) fait référence à la branche sur laquelle la branche spécifiée par branchname est configurée pour se construire (configurée avec branch.<name >.remote et branch.<name>.merge, ou avec git branch --set-upstream-to=<branch>). Un nom de branche manquant est par défaut celui en cours.

Avec la syntaxe des plages de révision, il est très utile de voir les commits de votre branche en amont (commits dans votre référentiel local pas encore présents en amont), et quels commits vous êtes derrière (commits en amont non fusionnés dans la branche locale), ou tous les deux:

$ git log --oneline @{u}..
$ git log --oneline ..@{u}
$ git log --oneline --left-right @{u}...  # same as ...@{u}

Valider la chaîne d’ascendance : ^, ~, etc. #

$ git reset --hard HEAD^             # discard last commit
$ git rebase --interactive HEAD~5    # rebase last 4 commits

Un suffixe ^ à un paramètre de révision signifie le premier parent de cet objet de validation. ^<n> signifie le <n>-ème parent (c’est-à-dire que <rev>^ est équivalent à <rev>^1).

Un suffixe ~<n> à un paramètre de révision désigne l’objet de validation qui est l’ancêtre de <n>-ième génération de l’objet de validation nommé, suivant uniquement les premiers parents. Cela signifie que par exemple <rev>~3 est équivalent à <rev>^^^. En raccourci, <rev>~ signifie <rev>~1, et équivaut à <rev>^1, ou <rev>^ en bref.

Cette syntaxe est composable.

Pour trouver de tels noms symboliques, vous pouvez utiliser la commande [git name-rev][git-name-rev] :

$ git name-rev 33db5f4d9027a10e477ccf054b2c1ab94f74c85a
33db5f4d9027a10e477ccf054b2c1ab94f74c85a tags/v0.99~940

Notez que --pretty=oneline et non --oneline doivent être utilisés dans l’exemple suivant

$ git log --pretty=oneline | git name-rev --stdin --name-only
master Sixth batch of topics for 2.10
master~1 Merge branch 'ls/p4-tmp-refs'
master~2 Merge branch 'js/am-call-theirs-theirs-in-fallback-3way'
[...]
master~14^2 sideband.c: small optimization of strbuf usage
master~16^2 connect: read $GIT_SSH_COMMAND from config file
[...]
master~22^2~1 t7810-grep.sh: fix a whitespace inconsistency
master~22^2~2 t7810-grep.sh: fix duplicated test name

[git-name-rev] : https://www.kernel.org/pub/software/scm/git/docs/git-name-rev.html

Déréférencer les branches et les balises : ^0, ^{} #

Dans certains cas, le comportement d’une commande dépend du fait qu’elle reçoit un nom de branche, un nom de balise ou une révision arbitraire. Vous pouvez utiliser la syntaxe “déréférencement” si vous avez besoin de cette dernière.

Un suffixe ^ suivi d’un nom de type d’objet (tag, commit, tree, blob) entre accolades (par exemple v0.99.8^{commit}) signifie déréférencer l’objet à <rev> récursivement jusqu’à ce qu’un objet de type <type> soit trouvé ou que l’objet ne puisse plus être déréférencé. <rev>^0 est un raccourci pour <rev>^{commit}.

$ git checkout HEAD^0    # equivalent to 'git checkout --detach' in modern Git

Un suffixe ^ suivi d’une paire d’accolades vides (par exemple v0.99.8^{}) signifie déréférencer la balise de manière récursive jusqu’à ce qu’un objet non-balise soit trouvé.

Comparer

$ git show v1.0
$ git cat-file -p v1.0
$ git replace --edit v1.0

avec

$ git show v1.0^{}
$ git cat-file -p v1.0^{}
$ git replace --edit v1.0^{}

Plus jeune commit correspondant : ^{/}, :/ #

$ git show HEAD^{/fix nasty bug}   # find starting from HEAD
$ git show ':/fix nasty bug'       # find starting from any branch

Deux-points (’:’), suivis d’une barre oblique (’/’), suivis d’un texte, nomment un commit dont le message de commit correspond à l’expression régulière spécifiée. Ce nom renvoie le plus jeune commit correspondant accessible depuis toute référence. L’expression régulière peut correspondre à n’importe quelle partie du message de validation. Pour faire correspondre les messages commençant par une chaîne, on peut utiliser par ex. :/^foo. La séquence spéciale :/! est réservée aux modificateurs de ce qui correspond. :/!-foo effectue une correspondance négative, tandis que :/!!foo correspond à un littéral ! caractère, suivi de “foo”.

Un suffixe ^ à un paramètre de révision, suivi d’une paire d’accolades qui contient un texte précédé d’une barre oblique, est identique à la syntaxe :/<texte> ci-dessous, car il renvoie le plus jeune commit correspondant accessible depuis le <rev> avant ^.
