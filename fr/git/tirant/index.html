Contrairement au push avec Git où vos modifications locales sont envoyées au serveur du référentiel central, le pull avec Git prend le code actuel sur le serveur et le “tire” du serveur du référentiel vers votre machine locale. Cette rubrique explique le processus d’extraction de code à partir d’un référentiel à l’aide de Git ainsi que les situations que l’on peut rencontrer lors de l’extraction de code différent dans la copie locale.

Syntaxe #

  • git pull [options [<repository> [<refspec>…]]

Paramètres #

Paramètres Détails
--calme Pas de sortie de texte
-q raccourci pour --quiet
--verbeux sortie de texte verbeux. Passé respectivement aux commandes de récupération et de fusion/rebase.
-v raccourci pour --verbose
--[non-]recurse-submodules[=yes|à la demande|non] Récupérer de nouveaux commits pour les sous-modules ? (Pas que ce n’est pas un pull/checkout)

git pull exécute git fetch avec les paramètres donnés et appelle git merge pour fusionner les têtes de branche récupérées dans la branche actuelle.

Mise à jour avec les modifications locales #

Lorsque des modifications locales sont présentes, la commande git pull annule le rapport :

erreur : vos modifications locales apportées aux fichiers suivants seraient écrasées par fusion

Pour mettre à jour (comme svn update l’a fait avec subversion), vous pouvez lancer :

git stash
git pull --rebase 
git stash pop

Un moyen pratique pourrait être de définir un alias en utilisant :

git config --global alias.up '!git stash && git pull --rebase && git stash pop'

git config --global alias.up 'pull --rebase --autostash'

Ensuite, vous pouvez simplement utiliser :

git up

Extraire, écraser local #

git fetch
git reset --hard origin/master

Attention : Bien que les commits supprimés à l’aide de reset --hard puissent être récupérés à l’aide de reflog et reset, les modifications non validées sont définitivement supprimées.

Remplacez origin et master par la télécommande et la branche vers lesquelles vous souhaitez tirer de force, respectivement, si elles sont nommées différemment.

Extraire le code de la télécommande #

git pull

Conserver l’historique linéaire lors de l’extraction #

Rebasage en tirant #

Si vous récupérez de nouveaux commits à partir du référentiel distant et que vous avez des modifications locales sur la branche actuelle, git fusionnera automatiquement la version distante et votre version. Si vous souhaitez réduire le nombre de fusions sur votre branche, vous pouvez indiquer à git de [rebase][1] vos commits sur la version distante de la branche.

git pull --rebase

En faire le comportement par défaut #

Pour en faire le comportement par défaut des branches nouvellement créées, saisissez la commande suivante :

git config branch.autosetuprebase always

Pour modifier le comportement d’une branche existante, utilisez ceci :

git config branch.BRANCH_NAME.rebase true

Et

git pull --no-rebase

Pour effectuer une extraction de fusion normale.

Vérifier si fast-forwardable #

Pour autoriser uniquement le transfert rapide de la branche locale, vous pouvez utiliser :

git pull --ff-only

Cela affichera une erreur lorsque la branche locale n’est pas à transfert rapide et doit être rebasée ou fusionnée avec l’amont.

[1] : https://www.wikiod.com/fr/git/rebasage

Extraction des modifications vers un référentiel local #

Tirage simple #

Lorsque vous travaillez sur un référentiel distant (par exemple, GitHub) avec quelqu’un d’autre, vous souhaiterez à un moment donné partager vos modifications avec eux. Une fois qu’ils ont [poussé][1] leurs modifications vers un référentiel distant, vous pouvez récupérer ces modifications en extrayant depuis ce référentiel.

git pull

Le fera, dans la majorité des cas.

Tirer d’une télécommande ou d’une branche différente #

Vous pouvez extraire les modifications d’une télécommande ou d’une branche différente en spécifiant leurs noms

git pull origin feature-A

Extraira la branche feature-A form origin dans votre branche locale. Notez que vous pouvez fournir directement une URL au lieu d’un nom distant et un nom d’objet tel qu’un SHA de validation au lieu d’un nom de branche.

Tirage manuel #

Pour imiter le comportement d’un git pull, vous pouvez utiliser git fetch puis git merge

git fetch origin # retrieve objects and update refs from origin
git merge origin/feature-A # actually perform the merge

Cela peut vous donner plus de contrôle et vous permet d’inspecter la branche distante avant de la fusionner. En effet, après la récupération, vous pouvez voir les branches distantes avec git branch -a, et les vérifier avec

git checkout -b local-branch-name origin/feature-A # checkout the remote branch
# inspect the branch, make commits, squash, ammend or whatever
git checkout merging-branches # moving to the destination branch
git merge local-branch-name # performing the merge

Cela peut être très pratique lors du traitement des demandes d’extraction.

[1] : https://www.wikiod.com/fr/git/pousser

Tirer, “autorisation refusée” #

Certains problèmes peuvent survenir si le dossier .git a une mauvaise autorisation. Résoudre ce problème en définissant le propriétaire du dossier .git complet. Parfois, il arrive qu’un autre utilisateur tire et modifie les droits du dossier ou des fichiers .git.

Pour résoudre le problème :

chown -R youruser:yourgroup .git/
