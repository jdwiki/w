Récupération d’un commit perdu #

Si vous êtes revenu à un commit antérieur et avez perdu un commit plus récent, vous pouvez récupérer le commit perdu en exécutant

git reflog

Ensuite, trouvez votre commit perdu et réinitialisez-le en faisant

git reset HEAD --hard <sha1-of-commit>

Restaurer un fichier supprimé après un commit #

Au cas où vous auriez accidentellement commis une suppression sur un fichier et réalisé plus tard que vous en avez besoin.

Recherchez d’abord l’ID de validation du commit qui a supprimé votre fichier.

git log --diff-filter=D --summary

Vous donnera un résumé trié des commits qui ont supprimé des fichiers.

Procédez ensuite à la restauration du fichier en

git checkout 81eeccf~1 <your-lost-file-name>

(Remplacez 81eeccf par votre propre identifiant de validation)

Restaurer le fichier vers une version précédente #

Pour restaurer un fichier vers une version précédente, vous pouvez utiliser reset.

git reset <sha1-of-commit> <file-name>

Si vous avez déjà apporté des modifications locales au fichier (dont vous n’avez pas besoin !), Vous pouvez également utiliser l’option --hard

Récupérer une branche supprimée #

Pour récupérer une branche supprimée, vous devez trouver le commit qui était à la tête de votre branche supprimée en exécutant

git reflog

Vous pouvez ensuite recréer la branche en exécutant

git checkout -b <branch-name> <sha1-of-commit>

Vous ne pourrez pas récupérer les branches supprimées si le [garbage collector][1] de git a supprimé les commits pendants - ceux sans références. Ayez toujours une sauvegarde de votre référentiel, surtout lorsque vous travaillez dans une petite équipe / un projet propriétaire

[1] : https://git-scm.com/docs/git-gc

Récupération après une réinitialisation #

Avec Git, vous pouvez (presque) toujours revenir en arrière #

N’ayez pas peur d’expérimenter avec des commandes qui réécrivent l’historique*. Git ne supprime pas vos commits pendant 90 jours par défaut, et pendant ce temps, vous pouvez facilement les récupérer depuis le reflog :

$ git reset @~3   # go back 3 commits
$ git reflog
c4f708b HEAD@{0}: reset: moving to @~3
2c52489 HEAD@{1}: commit: more changes
4a5246d HEAD@{2}: commit: make important changes
e8571e4 HEAD@{3}: commit: make some changes
... earlier commits ...
$ git reset 2c52489
... and you're back where you started

* Méfiez-vous des options telles que --hard et --force, car elles peuvent supprimer des données. * Évitez également de réécrire l’historique des branches sur lesquelles vous collaborez.

Récupérer de git stash #

Pour obtenir votre stock le plus récent après avoir exécuté git stash, utilisez

git stash apply

Pour voir une liste de vos réserves, utilisez

git stash list

Vous obtiendrez une liste qui ressemble à ceci

stash@{0}: WIP on master: 67a4e01 Merge tests into develop
stash@{1}: WIP on master: 70f0d95 Add user role to localStorage on user login

Choisissez un autre git stash à restaurer avec le numéro qui s’affiche pour le stash que vous voulez

git stash apply stash@{2}

Vous pouvez également choisir ‘git stash pop’, cela fonctionne de la même manière que ‘git stash apply’ comme ..

 git stash pop

ou

 git stash pop stash@{2}

Différence entre git stash apply et git stash pop…

git stash pop : - les données de cachette seront supprimées de la pile de la liste de cachette.

Ex:-

git stash list

Vous obtiendrez une liste qui ressemble à ceci

stash@{0}: WIP on master: 67a4e01 Merge tests into develop
stash@{1}: WIP on master: 70f0d95 Add user role to localStorage on user login

Maintenant, pop les données de cachette en utilisant la commande

git stash pop

Encore une fois, vérifiez la liste de cachette

git stash list

Vous obtiendrez une liste qui ressemble à ceci

 stash@{0}: WIP on master: 70f0d95 Add user role to localStorage on user login

Vous pouvez voir qu’une donnée de réserve est supprimée (apparue) de la liste de réserve et que stash@{1} est devenu stash@{0}.
