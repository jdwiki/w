Il convient de noter que la mise en scène a peu à voir avec les “fichiers” eux-mêmes et tout à voir avec les modifications apportées à chaque fichier donné. Nous préparons les fichiers qui contiennent des modifications, et git suit les modifications en tant que commits (même lorsque les modifications d’un commit sont effectuées sur plusieurs fichiers).

La distinction entre les fichiers et les commits peut sembler mineure, mais comprendre cette différence est fondamentale pour comprendre les fonctions essentielles comme cherry-pick et diff. (Voir la frustration dans [commentaires concernant la complexité d’une réponse acceptée qui propose cherry-pick comme outil de gestion de fichiers](http://stackoverflow.com/questions/449541/how-do-you-merge-selective-files- avec-git-merge).)

Quel est le bon endroit pour expliquer les concepts ? Est-ce dans les remarques ?

Concepts clés:

Un fichier est la métaphore la plus courante des deux dans les technologies de l’information. Les meilleures pratiques dictent qu’un nom de fichier ne change pas lorsque son contenu change (à quelques exceptions près).

Un commit est une métaphore propre à la gestion du code source. Les validations sont des modifications liées à un effort spécifique, comme une correction de bogue. Les commits impliquent souvent plusieurs fichiers. Une seule correction de bogue mineure peut impliquer des ajustements aux modèles et au CSS dans des fichiers uniques. Au fur et à mesure que le changement est décrit, développé, documenté, révisé et déployé, les changements dans les fichiers séparés peuvent être annotés et traités comme une seule unité. L’unité unique dans ce cas est le commit. Tout aussi important, se concentrer uniquement sur le commit lors d’une révision permet d’ignorer en toute sécurité les lignes de code inchangées dans les différents fichiers concernés.

Mise en scène de toutes les modifications apportées aux fichiers #

git add -A

git add .

Dans la version 2.x, git add . organisera toutes les modifications apportées aux fichiers du répertoire courant et de tous ses sous-répertoires. Cependant, dans 1.x, il ne mettra en scène que [les fichiers nouveaux et modifiés, pas les fichiers supprimés] (http://stackoverflow.com/a/26039014/3345375).

Utilisez git add -A, ou sa commande équivalente git add --all, pour mettre en scène toutes les modifications apportées aux fichiers dans n’importe quelle version de git.

Annuler la mise en scène d’un fichier contenant des modifications #

git reset <filePath>

Ajouter des modifications par morceau #

Vous pouvez voir quels “blocs” de travail seraient mis en scène pour la validation à l’aide de l’indicateur de patch :

git add -p

ou

git add --patch

Cela ouvre une invite interactive qui vous permet de consulter les différences et de décider si vous souhaitez les inclure ou non.

Stage this hunk [y,n,q,a,d,/,s,e,?]?
  • y mettre en scène ce morceau pour le prochain commit
  • n ne pas mettre en scène ce morceau pour le prochain commit
  • q quitter ; ne mettez pas en scène ce mec ni aucun des mecs restants
  • un étape ce morceau et tous les morceaux ultérieurs dans le fichier
  • d ne pas mettre en scène ce morceau ou l’un des morceaux ultérieurs dans le fichier
  • g sélectionner un morceau auquel aller
  • / recherche un hunk correspondant à la regex donnée
  • j laisser ce morceau indécis, voir le prochain morceau indécis
  • J laisser ce morceau indécis, voir le morceau suivant
  • k laissez ce morceau indécis, voir le morceau indécis précédent
  • K laissez ce morceau indécis, voir le morceau précédent
  • s divise le morceau actuel en morceaux plus petits
  • e modifier manuellement le morceau actuel
  • ? imprimer l’aide du gros morceau
  • Cela facilite la détection des modifications que vous ne souhaitez pas valider. *

Vous pouvez également l’ouvrir via git add --interactive et en sélectionnant p.

Ajout interactif #

git add -i (ou --interactive) vous donnera une interface interactive où vous pourrez éditer l’index, pour préparer ce que vous voulez avoir dans le prochain commit. Vous pouvez ajouter et supprimer des modifications à des fichiers entiers, ajouter des fichiers non suivis et supprimer des fichiers du suivi, mais également sélectionner une sous-section de modifications à mettre dans l’index, en sélectionnant des parties de modifications à ajouter, en divisant ces parties ou même en modifiant le diff . De nombreux outils graphiques de validation pour Git (comme par exemple git gui) incluent une telle fonctionnalité ; cela pourrait être plus facile à utiliser que la version en ligne de commande.

C’est très utile (1) si vous avez des changements intriqués dans le répertoire de travail que vous voulez mettre dans des commits séparés, et pas tous dans un seul commit (2) si vous êtes au milieu d’un rebase interactif et que vous voulez aussi diviser gros engagement.

$ git add -i
           staged     unstaged path
  1:    unchanged        +4/-4 index.js
  2:        +1/-0      nothing package.json

*** Commands ***
  1: status       2: update       3: revert       4: add untracked
  5: patch        6: diff         7: quit         8: help
What now>

La moitié supérieure de cette sortie montre l’état actuel de l’index divisé en colonnes étagées et non étagées :

  1. index.js a eu 4 lignes ajoutées et 4 lignes supprimées. Il n’est actuellement pas mis en scène, car l’état actuel indique “inchangé”. Lorsque ce fichier devient mis en scène, le bit “+4/-4” sera transféré à la colonne mise en scène et la colonne non mise en scène indiquera “rien”.
  2. package.json a eu une ligne ajoutée et a été mis en scène. Il n’y a pas d’autres changements depuis qu’il a été mis en scène comme indiqué par la ligne “rien” sous la colonne non mis en scène.

La moitié inférieure montre ce que vous pouvez faire. Entrez un nombre (1-8) ou une lettre (s, u, r, a, p, d, q, h).

status affiche une sortie identique à la partie supérieure de la sortie ci-dessus.

update vous permet d’apporter d’autres modifications aux commits par étapes avec une syntaxe supplémentaire.

revert renverra les informations de validation par étapes à HEAD.

add untracked vous permet d’ajouter des chemins de fichiers précédemment non suivis par le contrôle de version.

patch permet de sélectionner un chemin à partir d’une sortie similaire à status pour une analyse plus approfondie.

diff affiche ce qui sera validé.

quit quitte la commande.

help présente une aide supplémentaire sur l’utilisation de cette commande.

Afficher les modifications par étapes #

Pour afficher les hunks mis en scène pour la validation :

git diff --cached

Mise en scène d’un seul fichier #

Pour préparer un fichier pour la validation, exécutez

git add <filename>

Mettre en scène les fichiers supprimés #

git rm filename

Pour supprimer le fichier de git sans le supprimer du disque, utilisez le drapeau --cached

git rm --cached filename
