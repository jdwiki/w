Cloner de très gros référentiels SVN

Si votre historique de dépôt SVN est vraiment très volumineux, cette opération peut prendre des heures, car git-svn doit reconstruire l’historique complet du dépôt SVN. Heureusement, vous n’avez besoin de cloner le dépôt SVN qu’une seule fois ; comme avec tout autre référentiel git, vous pouvez simplement copier le dossier repo vers d’autres collaborateurs. La copie du dossier sur plusieurs ordinateurs sera plus rapide que le simple clonage de gros dépôts SVN à partir de zéro.

À propos des commits et de SHA1

Vos commits git locaux seront réécrits lors de l’utilisation de la commande git svn dcommit. Cette commande ajoutera un texte au message du commit git faisant référence à la révision SVN créée dans le serveur SVN, ce qui est très utile. Cependant, ajouter un nouveau texte nécessite de modifier le message d’un commit existant, ce qui ne peut pas être fait en réalité : les commits git sont immuables. La solution est de créer un nouveau commit avec le même contenu et le nouveau message, mais il s’agit techniquement d’un nouveau commit de toute façon (c’est-à-dire que le SHA1 du commit git changera)

Comme les commits git créés pour git-svn sont locaux, les identifiants SHA1 pour les commits git sont différents entre chaque dépôt git ! Cela signifie que vous ne pouvez pas utiliser un SHA1 pour référencer un commit d’une autre personne car le même commit aura un SHA1 différent dans chaque référentiel git local. Vous devez vous fier au numéro de révision svn ajouté au message de validation lorsque vous poussez vers le serveur SVN si vous souhaitez référencer une validation entre différentes copies du référentiel.

  • Vous pouvez * utiliser le SHA1 pour les opérations locales (afficher/différer un commit spécifique, sélections et réinitialisations, etc.)

Dépannage #

** La commande git svn rebase génère une erreur d’incompatibilité de somme de contrôle **

La commande git svn rebase génère une erreur semblable à celle-ci :

  Checksum mismatch: <path_to_file> <some_kind_of_sha1>
  expected: <checksum_number_1>
    got: <checksum_number_2>

La solution à ce problème est de réinitialiser svn à la révision lorsque le fichier problématique a été modifié pour la dernière fois, et d’effectuer une récupération git svn afin que l’historique SVN soit restauré. Les commandes pour effectuer la réinitialisation SVN sont :

  • git log -1 – <path_to_file> (copiez le numéro de révision SVN qui apparaît dans le message de validation)
  • git svn reset <revision_number>
  • git svn chercher

Vous devriez pouvoir à nouveau envoyer/extraire des données de SVN

Le fichier n’a pas été trouvé dans le commit Lorsque vous essayez de récupérer ou d’extraire de SVN, vous obtenez une erreur similaire à celle-ci

<file_path> was not found in commit <hash>

Cela signifie qu’une révision dans SVN essaie de modifier un fichier qui, pour une raison quelconque, n’existe pas dans votre copie locale. La meilleure façon de se débarrasser de cette erreur est de forcer une récupération en ignorant le chemin de ce fichier et il sera mis à jour à son statut dans la dernière révision SVN :

  • git svn fetch --ignore-paths <file_path>

Clonage du référentiel SVN #

Vous devez créer une nouvelle copie locale du référentiel avec la commande

git svn clone SVN_REPO_ROOT_URL [DEST_FOLDER_PATH] -T TRUNK_REPO_PATH -t TAGS_REPO_PATH -b BRANCHES_REPO_PATH

Si votre référentiel SVN suit la disposition standard (tronc, branches, dossiers de balises), vous pouvez économiser un peu de frappe :

git svn clone -s SVN_REPO_ROOT_URL [DEST_FOLDER_PATH]

git svn clone vérifie chaque révision SVN, une par une, et effectue un commit git dans votre référentiel local afin de recréer l’historique. Si le référentiel SVN a beaucoup de commits, cela prendra un certain temps.

Une fois la commande terminée, vous disposerez d’un référentiel git à part entière avec une branche locale appelée master qui suit la branche tronc dans le référentiel SVN.

Pousser les modifications locales vers SVN #

La commande

git svn dcommit

créera une révision SVN pour chacun de vos commits git locaux. Comme avec SVN, votre historique git local doit être synchronisé avec les dernières modifications du référentiel SVN, donc si la commande échoue, essayez d’abord d’effectuer un git svn rebase.

Travailler localement #

Utilisez simplement votre référentiel git local comme un référentiel git normal, avec les commandes git normales :

  • git add FILE et git checkout -- FILE Pour mettre en scène/désinstaller un fichier
  • git commit Pour enregistrer vos modifications. Ces commits seront locaux et ne seront pas “poussés” vers le référentiel SVN, comme dans un référentiel git normal
  • git stash et git stash pop Permet d’utiliser des caches
  • git reset HEAD --hard Annule toutes vos modifications locales
  • git log Accédez à tout l’historique du référentiel
  • git rebase -i pour que vous puissiez réécrire votre histoire locale librement
  • git branch et git checkout pour créer des branches locales

Comme l’indique la documentation de git-svn “Subversion est un système beaucoup moins sophistiqué que Git”, vous ne pouvez donc pas utiliser toute la puissance de git sans gâcher l’historique du serveur Subversion. Heureusement, les règles sont très simples : ** Gardez l’historique linéaire **

Cela signifie que vous pouvez effectuer presque n’importe quelle opération git : créer des branches, supprimer/réorganiser/écraser des commits, déplacer l’historique, supprimer des commits, etc. Tout sauf fusionner. Si vous avez besoin de réintégrer l’historique des branches locales, utilisez plutôt git rebase.

Lorsque vous effectuez une fusion, un commit de fusion est créé. La particularité des commits de fusion est qu’ils ont deux parents, ce qui rend l’historique non linéaire. L’historique non linéaire confondra SVN dans le cas où vous “pousseriez” un commit de fusion vers le référentiel.

Cependant, ne vous inquiétez pas : vous ne casserez rien si vous “poussez” un commit git merge vers SVN. Si vous le faites, lorsque le commit git merge est envoyé au serveur svn, il contiendra toutes les modifications de tous les commits pour cette fusion, vous perdrez donc l’historique de ces commits, mais pas les modifications de votre code.

Obtenir les dernières modifications de SVN #

L’équivalent de git pull est la commande

git svn rebase

Cela récupère toutes les modifications du référentiel SVN et les applique * en plus * de vos commits locaux dans votre branche actuelle.

Vous pouvez également utiliser la commande

git svn fetch

pour récupérer les modifications du référentiel SVN et les apporter à votre machine locale mais sans les appliquer à votre branche locale.

Gestion des dossiers vides #

git ne reconnaît pas le concept de dossiers, il fonctionne uniquement avec les fichiers et leurs chemins de fichiers. Cela signifie que git ne suit pas les dossiers vides. SVN, cependant, le fait. L’utilisation de git-svn signifie que, par défaut, * toute modification impliquant des dossiers vides avec git ne sera pas propagée à SVN *.

L’utilisation du drapeau --rmdir lors de l’émission d’un commentaire corrige ce problème et supprime un dossier vide dans SVN si vous supprimez localement le dernier fichier qu’il contient :

git svn dcommit --rmdir

Malheureusement ** cela ne supprime pas les dossiers vides existants ** : vous devez le faire manuellement.

Pour éviter d’ajouter le drapeau à chaque fois que vous effectuez un dcommit, ou pour jouer la sécurité si vous utilisez un outil graphique git (comme SourceTree), vous pouvez définir ce comportement par défaut avec la commande :

git config --global svn.rmdir true

Cela modifie votre fichier .gitconfig et ajoute ces lignes :

[svn]
rmdir = true

Pour supprimer tous les fichiers et dossiers non suivis qui doivent rester vides pour SVN, utilisez la commande git :

git clean -fd

Remarque : la commande précédente supprimera tous les fichiers non suivis et les dossiers vides, même ceux qui devraient être suivis par SVN ! Si vous avez besoin de générer à nouveau les dossiers vides suivis par SVN, utilisez la commande

git svn mkdirs

En pratique, cela signifie que si vous souhaitez nettoyer votre espace de travail des fichiers et dossiers non suivis, vous devez toujours utiliser les deux commandes pour recréer les dossiers vides suivis par SVN :

git clean -fd && git svn mkdirs
