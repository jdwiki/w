Clonage d’un dépôt Git ayant des sous-modules #

Lorsque vous clonez un référentiel qui utilise des sous-modules, vous devez les initialiser et les mettre à jour.

$ git clone --recursive https://github.com/username/repo.git

Cela clonera les sous-modules référencés et les placera dans les dossiers appropriés (y compris les sous-modules dans les sous-modules). Cela équivaut à exécuter git submodule update --init --recursive immédiatement après la fin du clonage.

Mise à jour d’un sous-module #

Un sous-module fait référence à un commit spécifique dans un autre référentiel. Pour vérifier l’état exact qui est référencé pour tous les sous-modules, exécutez

git submodule update --recursive

Parfois, au lieu d’utiliser l’état référencé, vous souhaitez mettre à jour votre extraction locale vers le dernier état de ce sous-module sur une télécommande. Pour vérifier tous les sous-modules au dernier état sur la télécommande avec une seule commande, vous pouvez utiliser

git submodule foreach git pull <remote> <branch>

ou utilisez les arguments par défaut git pull

git submodule foreach git pull

Notez que cela ne fera que mettre à jour votre copie de travail locale. L’exécution de git status listera le répertoire du sous-module comme sale s’il a changé à cause de cette commande. Pour mettre à jour votre référentiel afin de référencer le nouvel état à la place, vous devez valider les modifications :

git add <submodule_directory>
git commit

Il peut y avoir des modifications que vous avez qui peuvent avoir un conflit de fusion si vous utilisez git pull afin que vous puissiez utiliser git pull --rebase pour rembobiner vos modifications vers le haut, la plupart du temps cela diminue les risques de conflit. En outre, il tire toutes les branches vers le local.

git submodule foreach git pull --rebase

Pour vérifier le dernier état d’un sous-module spécifique, vous pouvez utiliser :

git submodule update --remote <submodule_directory>

Ajout d’un sous-module #

Vous pouvez inclure un autre référentiel Git en tant que dossier dans votre projet, suivi par Git :

$ git submodule add https://github.com/jquery/jquery.git

Vous devez ajouter et valider le nouveau fichier .gitmodules ; cela indique à Git quels sous-modules doivent être clonés lorsque git submodule update est exécuté.

Définir un sous-module pour suivre une branche #

Un sous-module est toujours extrait à un commit SHA1 spécifique (le “gitlink”, entrée spéciale dans l’index du référentiel parent)

Mais on peut demander de mettre à jour ce sous-module avec le dernier commit d’une branche du référentiel distant du sous-module.

Plutôt que d’aller dans chaque sous-module, en faisant un git checkout abranch --track origin/abranch, git pull, vous pouvez simplement faire (depuis le référentiel parent) a :

git submodule update --remote --recursive

Étant donné que le SHA1 du sous-module changerait, vous auriez toujours besoin de suivre cela avec :

git add .
git commit -m "update submodules"

Cela suppose que les sous-modules étaient :

  • soit ajouté avec une branche à suivre :

      git submodule -b abranch -- /url/of/submodule/repo

  • soit configuré (pour un sous-module existant) pour suivre une branche :

      cd /path/to/parent/repo
  git config -f .gitmodules submodule.asubmodule.branch abranch

Déplacer un sous-module #

Courir:

$ git mv ancien/chemin/vers/module nouveau/chemin/vers/module
  1. Modifiez .gitmodules et changez le chemin du sous-module de manière appropriée, et placez-le dans l’index avec git add .gitmodules.
  2. Si nécessaire, créez le répertoire parent du nouvel emplacement du sous-module (mkdir -p new/path/to).
  3. Déplacez tout le contenu de l’ancien vers le nouveau répertoire (mv -vi old/path/to/module new/path/to/submodule).
  4. Assurez-vous que Git suit ce répertoire (git add new/path/to).
  5. Supprimez l’ancien répertoire avec git rm –cached old/path/to/module.
  6. Déplacez le répertoire .git/modules/old/path/to/module avec tout son contenu vers .git/modules/new/path/to/module</ code>.
  7. Modifiez le fichier .git/modules/new/path/to/config, assurez-vous que l’élément worktree pointe vers les nouveaux emplacements, donc dans cet exemple, il devrait être worktree = ../../../../../ancien/chemin/vers/module. En règle générale, il devrait y avoir deux autres répertoires .. puis dans le chemin direct à cet endroit. . Modifiez le fichier new/path/to/module/.git, assurez-vous que le chemin qu’il contient pointe vers le nouvel emplacement correct dans le dossier principal du projet .git, donc dans cet exemple gitdir : ../../../.git/modules/new/path/to/module.

La sortie de git status ressemble ensuite à ceci :

    # On branch master
    # Changes to be committed:
    #   (use "git reset HEAD <file>..." to unstage)
    #
    #       modified:   .gitmodules
    #       renamed:    old/path/to/submodule -> new/path/to/submodule
    #
  1. Enfin, validez les modifications.

Cet exemple de Stack Overflow, par Axel Beckert

Suppression d’un sous-module #

Vous pouvez supprimer un sous-module (par exemple, the_submodule) en appelant :

$ git submodule deinit the_submodule
$ git rm the_submodule

  • git submodule deinit the_submodule supprime l’entrée the_submodules’ de .git/config. Cela exclut the_submodule des appels git submodule update, git submodule sync et git submodule foreach et supprime son contenu local [(source)](https://git-scm.com/docs/git-submodule#git -submodule-deinit). De plus, cela ne sera pas affiché comme changement dans votre référentiel parent. git submodule init et git submodule update restaureront le sous-module, encore une fois sans modifications validables dans votre référentiel parent.

  • git rm the_submodule supprimera le sous-module de l’arbre de travail. Les fichiers disparaîtront ainsi que l’entrée des sous-modules dans le fichier .gitmodules (source). Si seulement git rm the_submodule (sans git submodule deinit the_submodule préalable est exécuté, cependant, l’entrée des sous-modules dans votre fichier .git/config restera.

Extrait de ici:

  1. Supprimez la section pertinente du fichier .gitmodules.
  2. Étape les changements .gitmodules git add .gitmodules
  3. Supprimez la section pertinente de .git/config.
  4. Exécutez git rm --cached path_to_submodule (pas de barre oblique finale).
  5. Exécutez rm -rf .git/modules/path_to_submodule
  6. Commit git commit -m "Sous-module supprimé <nom>"
  7. Supprimez les fichiers de sous-module désormais non suivis
  8. rm -rf path_to_submodule
