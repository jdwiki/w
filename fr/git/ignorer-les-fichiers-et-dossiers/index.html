Cette rubrique illustre comment éviter d’ajouter des fichiers indésirables (ou des modifications de fichiers) dans un référentiel Git. Il existe plusieurs façons (global ou local .gitignore, .git/exclude, git update-index --assume-unchanged et git update-index --skip-tree), mais gardez à l’esprit Git gère le contenu, ce qui signifie : ignorer ignore en fait un dossier contenu (c’est-à-dire des fichiers). Un dossier vide serait ignoré par défaut, car il ne peut pas être ajouté de toute façon.

Ignorer les fichiers et répertoires avec un fichier .gitignore #

Vous pouvez faire en sorte que Git ignore certains fichiers et répertoires — c’est-à-dire les exclure du suivi par Git — en créant un ou plusieurs fichiers [.gitignore][1] dans votre dépôt.

Dans les projets logiciels, .gitignore contient généralement une liste de fichiers et/ou de répertoires générés pendant le processus de construction ou lors de l’exécution. Les entrées du fichier .gitignore peuvent inclure des noms ou des chemins pointant vers :

  1. ressources temporaires, par ex. caches, fichiers journaux, code compilé, etc.
  2. fichiers de configuration locaux qui ne doivent pas être partagés avec d’autres développeurs
  3. fichiers contenant des informations secrètes, telles que les mots de passe de connexion, les clés et les informations d’identification

Lorsqu’elles sont créées dans le répertoire de niveau supérieur, les règles s’appliqueront de manière récursive à tous les fichiers et sous-répertoires de l’ensemble du référentiel. Lorsqu’elles sont créées dans un sous-répertoire, les règles s’appliqueront à ce répertoire spécifique et à ses sous-répertoires.

Lorsqu’un fichier ou un répertoire est ignoré, il ne sera pas :

  1. suivi par Git
  2. rapporté par des commandes telles que git status ou git diff
  3. mis en scène avec des commandes telles que git add -A

Dans le cas inhabituel où vous devez ignorer les fichiers suivis, des précautions particulières doivent être prises. Voir : [Ignorer les fichiers qui ont déjà été validés dans un référentiel Git][2].

Exemples #

Voici quelques exemples génériques de règles dans un fichier “.gitignore”, basé sur glob file patterns :

# Lines starting with `#` are comments.

# Ignore files called 'file.ext'
file.ext

# Comments can't be on the same line as rules!
# The following line ignores files called 'file.ext # not a comment'
file.ext # not a comment 

# Ignoring files with full path.
# This matches files in the root directory and subdirectories too.
# i.e. otherfile.ext will be ignored anywhere on the tree.
dir/otherdir/file.ext
otherfile.ext

# Ignoring directories
# Both the directory itself and its contents will be ignored.
bin/
gen/

# Glob pattern can also be used here to ignore paths with certain characters.
# For example, the below rule will match both build/ and Build/
[bB]uild/

# Without the trailing slash, the rule will match a file and/or
# a directory, so the following would ignore both a file named `gen`
# and a directory named `gen`, as well as any contents of that directory
bin
gen

# Ignoring files by extension
# All files with these extensions will be ignored in
# this directory and all its sub-directories.
*.apk
*.class

# It's possible to combine both forms to ignore files with certain
# extensions in certain directories. The following rules would be
# redundant with generic rules defined above.
java/*.apk
gen/*.class

# To ignore files only at the top level directory, but not in its
# subdirectories, prefix the rule with a `/`
/*.apk
/*.class

# To ignore any directories named DirectoryA 
# in any depth use ** before DirectoryA
# Do not forget the last /, 
# Otherwise it will ignore all files named DirectoryA, rather than directories
**/DirectoryA/
# This would ignore 
# DirectoryA/
# DirectoryB/DirectoryA/ 
# DirectoryC/DirectoryB/DirectoryA/
# It would not ignore a file named DirectoryA, at any level

# To ignore any directory named DirectoryB within a 
# directory named DirectoryA with any number of 
# directories in between, use ** between the directories
DirectoryA/**/DirectoryB/
# This would ignore 
# DirectoryA/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryW/DirectoryB/

# To ignore a set of files, wildcards can be used, as can be seen above.
# A sole '*' will ignore everything in your folder, including your .gitignore file.
# To exclude specific files when using wildcards, negate them.
# So they are excluded from the ignore list:
!.gitignore 

# Use the backslash as escape character to ignore files with a hash (#)
# (supported since 1.6.2.1)
\#*#

La plupart des fichiers .gitignore sont standard dans différentes langues, donc pour commencer, voici un ensemble de [exemples de fichiers .gitignore][4] répertoriés par langue à partir de laquelle cloner ou copier/modifier dans votre projet. Alternativement, pour un nouveau projet, vous pouvez envisager de générer automatiquement un fichier de démarrage à l’aide d’un [outil en ligne][5].

Autres formes de .gitignore #

Les fichiers .gitignore sont destinés à être validés dans le cadre du référentiel. Si vous souhaitez ignorer certains fichiers sans valider les règles d’ignorance, voici quelques options :

  • Éditez le fichier .git/info/exclude (en utilisant la même syntaxe que .gitignore). Les règles seront globales dans le périmètre du référentiel ;
  • Configurez [un fichier gitignore global] (https://www.wikiod.com/fr/git/ignorer-les-fichiers-et-dossiers#Un fichier global .gitignore) qui appliquera les règles d’ignorance à tous vos dépôts locaux :

De plus, vous pouvez ignorer les modifications locales apportées aux fichiers suivis sans modifier la configuration globale de git avec :

  • git update-index --skip-worktree [<file>...] : pour les modifications locales mineures
  • git update-index --assume-unchanged [<file>...] : pour les fichiers prêts à la production et non changeants en amont

Voir [plus de détails sur les différences entre ces derniers drapeaux][6] et la [documentation de git update-index][7] pour plus d’options.

Nettoyer les fichiers ignorés #

Vous pouvez utiliser git clean -X pour nettoyer les fichiers ignorés :

git clean -Xn #display a list of ignored files
git clean -Xf #remove the previously displayed files

Remarque : -X (majuscules) nettoie uniquement les fichiers ignorés. Utilisez -x (pas de majuscules) pour supprimer également les fichiers non suivis.

Voir la documentation git clean pour plus de détails.

Voir [le manuel de Git][1] pour plus de détails.

Vérifier si un fichier est ignoré #

La commande git check-ignore signale les fichiers ignorés par Git.

Vous pouvez passer des noms de fichiers sur la ligne de commande, et git check-ignore listera les noms de fichiers qui sont ignorés. Par exemple:

$ cat .gitignore
*.o
$ git check-ignore example.o Readme.md
example.o

Ici, seuls les fichiers *.o sont définis dans .gitignore, donc Readme.md n’est pas répertorié dans la sortie de git check-ignore.

Si vous voulez voir la ligne dont .gitignore est responsable pour ignorer un fichier, ajoutez -v à la commande git check-ignore :

$ git check-ignore -v example.o Readme.md
.gitignore:1:*.o        example.o

À partir de Git 1.7.6, vous pouvez également utiliser git status --ignored afin de voir les fichiers ignorés. Vous pouvez trouver plus d’informations à ce sujet dans la [documentation officielle] [1] ou dans [Trouver les fichiers ignorés par .gitignore] [2].

[1] : https://git-scm.com/docs/git-status [2] : https://www.wikiod.com/fr/git/ignorer-les-fichiers-et-dossiers#Recherche de fichiers ignorés par .gitignore

Exceptions dans un fichier .gitignore #

Si vous ignorez des fichiers en utilisant un modèle mais avez des exceptions, préfixez un point d’exclamation (!) à l’exception. Par exemple:

*.txt
!important.txt

L’exemple ci-dessus demande à Git d’ignorer tous les fichiers avec l’extension .txt à l’exception des fichiers nommés important.txt.

Si le fichier se trouve dans un dossier ignoré, vous ne pouvez PAS le réinclure aussi facilement :

folder/
!folder/*.txt

Dans cet exemple, tous les fichiers .txt du dossier resteraient ignorés.

La bonne façon est de réinclure le dossier lui-même sur une ligne séparée, puis d’ignorer tous les fichiers dans folder par *, enfin de réinclure le *.txt dans folder, comme suit :

!folder/
folder/*
!folder/*.txt

Remarque : Pour les noms de fichiers commençant par un point d’exclamation, ajoutez deux points d’exclamation ou échappez-vous avec le caractère \ :

!!includethis
\!excludethis

Un fichier global .gitignore #

Pour que Git ignore certains fichiers dans tous les référentiels, vous pouvez [créer un .gitignore global][1] avec la commande suivante dans votre terminal ou invite de commande :

$ git config --global core.excludesfile <Path_To_Global_gitignore_file>

Git l’utilisera désormais en plus du fichier [.gitignore][2] de chaque référentiel. Les règles pour cela sont :

  • Si le fichier .gitignore local inclut explicitement un fichier alors que le .gitignore global l’ignore, le .gitignore local est prioritaire (le fichier sera inclus)
  • Si le dépôt est cloné sur plusieurs machines, alors le “.gigignore” global doit être chargé sur toutes les machines ou au moins l’inclure, car les fichiers ignorés seront poussés vers le dépôt tandis que le PC avec le “.gitignore” global ne le mettrait pas à jour. C’est pourquoi un dépôt spécifique .gitignore est une meilleure idée qu’un global si le projet est travaillé par une équipe

Ce fichier est un bon endroit pour conserver les éléments ignorés spécifiques à la plate-forme, à la machine ou à l’utilisateur, par ex. OSX .DS_Store, Windows Thumbs.db ou Vim *.ext~ et *.ext.swp ignore si vous ne souhaitez pas les conserver dans le référentiel. Ainsi, un membre de l’équipe travaillant sur OS X peut ajouter tous les .DS_STORE et _MACOSX (ce qui est en fait inutile), tandis qu’un autre membre de l’équipe sous Windows peut ignorer tous les thumbs.bd

[1] : https://help.github.com/articles/ignoring-files/#create-a-global-gitignore [2] : https://git-scm.com/docs/gitignore

Ignorer les fichiers qui ont déjà été validés dans un référentiel Git #

Si vous avez déjà ajouté un fichier à votre référentiel Git et que vous souhaitez maintenant arrêter de le suivre (afin qu’il ne soit plus présent dans les prochains commits), vous pouvez le supprimer de l’index :

git rm --cached <file>

Cela supprimera le fichier du référentiel et empêchera d’autres modifications d’être suivies par Git. L’option --cached s’assurera que le fichier n’est pas physiquement supprimé.

Notez que le contenu précédemment ajouté du fichier sera toujours visible via l’historique Git.

Gardez à l’esprit que si quelqu’un d’autre extrait du référentiel après avoir supprimé le fichier de l’index, sa copie sera physiquement supprimée.

Vous pouvez faire en sorte que Git prétende que la version du répertoire de travail du fichier est à jour et lire la version de l’index à la place (ignorant ainsi les modifications apportées) avec le bit “[skip worktree][]”:

git update-index --skip-worktree <file>

L’écriture n’est pas affectée par ce bit, la sécurité du contenu est toujours la première priorité. Vous ne perdrez jamais vos précieux changements ignorés ; par contre ce bit entre en conflit avec le stash : pour supprimer ce bit, utilisez

git update-index --no-skip-worktree <file>

Il est parfois à tort recommandé de mentir à Git et de lui faire supposer que le fichier est inchangé sans l’examiner. Il semble à première vue ignorer toute modification ultérieure du fichier, sans le supprimer de son index :

git update-index --assume-unchanged <file>

Cela forcera git à ignorer toute modification apportée au fichier (gardez à l’esprit que si vous récupérez des modifications dans ce fichier ou que vous le cachez, vos modifications ignorées seront perdues)

Si vous voulez que git “s’occupe” à nouveau de ce fichier, exécutez la commande suivante :

git update-index --no-assume-unchanged <file>

[ignorer l’arbre de travail] : https://www.kernel.org/pub/software/scm/git/docs/git-update-index.html#_skip_worktree_bit

Ignorer les fichiers localement sans valider les règles d’ignorance #

.gitignore ignore les fichiers localement, mais il est destiné à être validé dans le référentiel et partagé avec d’autres contributeurs et utilisateurs. Vous pouvez définir un “.gitignore” global, mais tous vos référentiels partageront alors ces paramètres.

Si vous souhaitez ignorer localement certains fichiers d’un référentiel et ne pas intégrer le fichier à un référentiel, modifiez .git/info/exclude dans votre référentiel.

Par exemple:

# these files are only ignored on this repo
# these rules are not shared with anyone
# as they are personal                                              
gtk_tests.py
gui/gtk/tests/*
localhost
pushReports.py
server/

Ignorer un fichier dans n’importe quel répertoire #

Pour ignorer un fichier foo.txt dans n’importe quel répertoire, vous devez simplement écrire son nom :

foo.txt # matches all files 'foo.txt' in any directory

Si vous souhaitez ignorer le fichier uniquement dans une partie de l’arborescence, vous pouvez spécifier les sous-répertoires d’un répertoire spécifique avec le motif ** :

bar/**/foo.txt # matches all files 'foo.txt' in 'bar' and all subdirectories

Ou vous pouvez créer un fichier .gitignore dans le répertoire bar/. L’équivalent de l’exemple précédent serait de créer le fichier bar/.gitignore avec ce contenu :

foo.txt # matches all files 'foo.txt' in any directory under bar/

Ignorer les modifications ultérieures apportées à un fichier (sans le supprimer) #

Parfois, vous souhaitez conserver un fichier dans Git mais ignorer les modifications ultérieures.

Dites à Git d’ignorer les modifications apportées à un fichier ou à un répertoire à l’aide de update-index :

git update-index --assume-unchanged my-file.txt

La commande ci-dessus indique à Git de supposer que my-file.txt n’a pas été modifié, et de ne pas vérifier ou signaler les modifications. Le fichier est toujours présent dans le référentiel.

Cela peut être utile pour fournir des valeurs par défaut et autoriser les remplacements d’environnement local, par exemple :

# create a file with some values in
cat <<EOF
MYSQL_USER=app
MYSQL_PASSWORD=FIXME_SECRET_PASSWORD
EOF > .env

# commit to Git
git add .env
git commit -m "Adding .env template"

# ignore future changes to .env
git update-index --assume-unchanged .env

# update your password
vi .env

# no changes!
git status

Ignorer les fichiers dans les sous-dossiers (plusieurs fichiers gitignore) #

Supposons que vous ayez une structure de référentiel comme celle-ci :

examples/
    output.log
src/
    <files not shown>
    output.log
README.md

output.log dans le répertoire des exemples est valide et requis pour que le projet rassemble une compréhension tandis que celui sous src/ est créé lors du débogage et ne doit pas figurer dans l’historique ou faire partie du référentiel.

Il existe deux façons d’ignorer ce fichier. Vous pouvez placer un chemin absolu dans le fichier .gitignore à la racine du répertoire de travail :

# /.gitignore
src/output.log

Alternativement, vous pouvez créer un fichier .gitignore dans le répertoire src/ et ignorer le fichier relatif à ce .gitignore :

# /src/.gitignore
output.log

Créer un dossier vide #

Il n’est pas possible d’ajouter et de valider un dossier vide dans Git car Git gère les fichiers et y attache leur répertoire, ce qui réduit les commits et améliore la vitesse. Pour contourner cela, il existe deux méthodes :

Première méthode : .gitkeep

Un hack pour contourner ce problème consiste à utiliser un fichier .gitkeep pour enregistrer le dossier pour Git. Pour ce faire, créez simplement le répertoire requis et ajoutez un fichier .gitkeep au dossier. Ce fichier est vide et ne sert à rien d’autre qu’à simplement enregistrer le dossier. Pour ce faire sous Windows (qui a des conventions de nommage de fichiers peu pratiques), ouvrez simplement git bash dans le répertoire et exécutez la commande :

$ toucher .gitkeep

Cette commande crée simplement un fichier .gitkeep vide dans le répertoire courant

Deuxième méthode : dummy.txt

Un autre hack pour cela est très similaire à ce qui précède et les mêmes étapes peuvent être suivies, mais au lieu d’un .gitkeep, utilisez simplement un dummy.txt à la place. Cela a l’avantage supplémentaire de pouvoir le créer facilement dans Windows à l’aide du menu contextuel. Et vous pouvez également y laisser des messages amusants. Vous pouvez également utiliser le fichier .gitkeep pour suivre le répertoire vide. .gitkeep est normalement un fichier vide qui est ajouté pour suivre le répertoire vide.

Recherche de fichiers ignorés par .gitignore #

Vous pouvez lister tous les fichiers ignorés par git dans le répertoire courant avec la commande :

git status --ignored

Donc, si nous avons une structure de référentiel comme celle-ci :

.git
.gitignore
./example_1
./dir/example_2
./example_2

…et le fichier .gitignore contenant :

example_2

… que le résultat de la commande sera :

$ git status --ignored

On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

.gitignore
.example_1

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)

dir/
example_2

Si vous souhaitez répertorier les fichiers ignorés de manière récursive dans les répertoires, vous devez utiliser un paramètre supplémentaire - --untracked-files=all

Le résultat ressemblera à ceci :

$ git status --ignored --untracked-files=all
On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

.gitignore
example_1

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)

dir/example_2
example_2

Ignorer seulement une partie d’un fichier [stub] #

Parfois, vous souhaiterez peut-être apporter des modifications locales à un fichier que vous ne souhaitez pas valider ou publier. Idéalement, les paramètres locaux devraient être concentrés dans un fichier séparé qui peut être placé dans .gitignore, mais parfois, comme solution à court terme, il peut être utile d’avoir quelque chose de local dans un fichier archivé.

Vous pouvez faire en sorte que Git “masque” ces lignes en utilisant un filtre propre. Ils n’apparaîtront même pas dans les diffs.

Supposons que voici un extrait du fichier file1.c :

struct settings s;
s.host = "localhost";
s.port = 5653;
s.auth = 1;
s.port = 15653; // NOCOMMIT
s.debug = 1; // NOCOMMIT
s.auth = 0; // NOCOMMIT

Vous ne voulez pas publier de lignes NOCOMMIT n’importe où.

Créez un filtre “nocommit” en ajoutant ceci au fichier de configuration Git comme .git/config :

[filter "nocommit"]
    clean=grep -v NOCOMMIT

Ajoutez (ou créez) ceci à .git/info/attributes ou .gitmodules :

file1.c filter=nocommit

Et vos lignes NOCOMMIT sont cachées de Git.

Mises en garde :

  • L’utilisation d’un filtre propre ralentit le traitement des fichiers, en particulier sous Windows.
  • La ligne ignorée peut disparaître du fichier lorsque Git le met à jour. Il peut être contrecarré avec un filtre anti-taches, mais c’est plus délicat.
  • Non testé sur Windows

Ignorer les modifications dans les fichiers suivis. [bout] #

[.gitignore][1] et .git/info/exclude ne fonctionnent que pour les fichiers non suivis.

Pour définir l’indicateur d’ignorance sur un fichier suivi, utilisez la commande [update-index][2] :

git update-index --skip-worktree myfile.c

Pour revenir en arrière, utilisez :

git update-index --no-skip-worktree myfile.c

Vous pouvez ajouter cet extrait à votre global [git config][3] pour avoir des commandes git hide, git unhide et git hidden plus pratiques :

[alias]
    hide   = update-index --skip-worktree
    unhide = update-index --no-skip-worktree
    hidden  = "!git ls-files -v | grep ^[hsS] | cut -c 3-"

[1] : https://git-scm.com/docs/gitignore [2] : https://git-scm.com/docs/git-update-index [3] : https://git-scm.com/docs/git-config

Vous pouvez également utiliser l’option –assume-unchanged avec la fonction update-index

git update-index --assume-unchanged <file>

Si vous souhaitez revoir ce fichier pour les modifications, utilisez

git update-index --no-assume-unchanged <file>

Lorsque l’indicateur –assume-unchanged est spécifié, l’utilisateur promet de ne pas modifier le fichier et permet à Git de supposer que le fichier de l’arbre de travail correspond à ce qui est enregistré dans l’index. Git échouera s’il doit modifier ce fichier dans l’index. par exemple. lors de la fusion dans un commit ; ainsi, dans le cas où le fichier supposé non suivi est modifié en amont, vous devrez gérer la situation manuellement. Dans ce cas, l’accent est mis sur les performances.

Alors que l’indicateur –skip-worktree est utile lorsque vous demandez à git de ne jamais toucher à un fichier spécifique parce que le fichier va être modifié localement et que vous ne voulez pas accidentellement valider les modifications (c’est-à-dire le fichier de configuration/propriétés configuré pour un fichier particulier environnement). Skip-worktree a priorité sur assume-unchanged lorsque les deux sont définis.

Effacer les fichiers déjà validés, mais inclus dans .gitignore #

Parfois, il arrive qu’un fichier soit suivi par git, mais qu’il ait été ajouté ultérieurement à .gitignore, afin d’arrêter de le suivre. C’est un scénario très courant d’oublier de nettoyer ces fichiers avant leur ajout à .gitignore. Dans ce cas, l’ancien fichier traînera toujours dans le référentiel.

Pour résoudre ce problème, on peut effectuer une suppression “à sec” de tout dans le référentiel, suivie d’un rajout de tous les fichiers. Tant que vous n’avez pas de modifications en attente et que le paramètre --cached est passé, cette commande peut être exécutée en toute sécurité :

# Remove everything from the index (the files will stay in the file system) 
$ git rm -r --cached .

# Re-add everything (they'll be added in the current state, changes included)
$ git add .

# Commit, if anything changed. You should see only deletions
$ git commit -m 'Remove all files that are in the .gitignore'

# Update the remote
$ git push origin master
