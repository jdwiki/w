Dépôt #

Un “référentiel git” est une structure de données sur disque qui stocke les métadonnées d’un ensemble de fichiers et de répertoires.

Il réside dans le dossier .git/ de votre projet. Chaque fois que vous validez des données sur git, elles sont stockées ici. Inversement, .git/ contient chaque commit.

Sa structure de base est la suivante :

.git/
    objects/
    refs/

Objets #

git est fondamentalement un magasin clé-valeur. Lorsque vous ajoutez des données à git, il crée un object et utilise le hachage SHA-1 du contenu de object comme clé.

Par conséquent, tout contenu dans git peut être recherché par son hachage :

git cat-file -p 4bb6f98

Il existe 4 types d’objets :

  • “tache”
  • ‘arbre’
  • “s’engager”
  • ‘balise’

Réf TÊTE #

HEAD est une ref spéciale. Il pointe toujours vers l’objet courant.

Vous pouvez voir où il pointe actuellement en vérifiant le fichier .git/HEAD.

Normalement, HEAD pointe vers une autre ref :

$cat .git/HEAD
ref: refs/heads/mainline

Mais il peut aussi pointer directement vers un object :

$ cat .git/HEAD
4bb6f98a223abc9345a0cef9200562333

C’est ce qu’on appelle une “tête détachée” - parce que HEAD n’est pas attaché à (pointant vers) une ref, mais pointe plutôt directement vers un object.

Réfs #

Un ref est essentiellement un pointeur. C’est un nom qui pointe vers un objet. Par exemple,

"master" --> 1a410e...

Ils sont stockés dans `.git/refs/heads/ dans des fichiers texte brut.

$ cat .git/refs/heads/mainline
4bb6f98a223abc9345a0cef9200562333

C’est communément ce qu’on appelle des “branches”. Cependant, vous remarquerez que dans git il n’y a pas de branch - seulement une ref.

Maintenant, il est possible de naviguer dans git simplement en sautant vers différents objets directement par leurs hachages. Mais ce serait terriblement gênant. Un ref vous donne un nom pratique pour faire référence aux objects. Il est beaucoup plus facile de demander à git d’aller à un endroit spécifique par son nom plutôt que par un hachage.

Objet de validation #

Un commit est probablement le type object le plus familier aux utilisateurs de git, car c’est ce qu’ils ont l’habitude de créer avec les commandes git commit.

Cependant, le commit ne contient pas directement de fichiers ou de données modifiés. Au lieu de cela, il contient principalement des métadonnées et des pointeurs vers d’autres “objets” qui contiennent le contenu réel du “commit”.

Un commit contient quelques éléments :

  • hachage d’un “arbre”
  • hachage d’un parent commit
  • nom/email de l’auteur, nom/email de l’auteur
  • message de validation

Vous pouvez voir le contenu de n’importe quel commit comme ceci :

$ git cat-file commit 5bac93
tree 04d1daef...
parent b7850ef5...
author Geddy Lee <glee@rush.com>
commiter Neil Peart <npeart@rush.com>

First commit!

Arbre #

Une note très importante est que les objets tree stockent TOUS les fichiers de votre projet, et qu’ils stockent des fichiers entiers et non des diffs. Cela signifie que chaque commit contient un instantané de l’ensemble du projet*.

*Techniquement, seuls les fichiers modifiés sont stockés. Mais c’est plus un détail d’implémentation pour l’efficacité. Du point de vue de la conception, un “commit” doit être considéré comme contenant une copie complète du projet.

Parent #

La ligne parent contient un hachage d’un autre objet commit et peut être considérée comme un “pointeur parent” qui pointe vers le “commit précédent”. Cela forme implicitement un graphe de commits connu sous le nom de commit graph. Plus précisément, il s’agit d’un graphe acyclique dirigé (ou DAG).

Objet d’arborescence #

Un “arbre” représente essentiellement un dossier dans un système de fichiers traditionnel : des conteneurs imbriqués pour des fichiers ou d’autres dossiers.

Un “arbre” contient :

  • 0 ou plusieurs objets blob
  • 0 ou plusieurs objets tree

Tout comme vous pouvez utiliser ls ou dir pour lister le contenu d’un dossier, vous pouvez lister le contenu d’un objet tree.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Vous pouvez rechercher les fichiers dans un commit en trouvant d’abord le hachage de tree dans le commit, puis en regardant cet tree :

$ git cat-file commit 4bb6f93a
tree 07b1a631
parent ...
author ...
commiter ... 
 
$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Objet blob #

Un blob contient un contenu de fichier binaire arbitraire. Généralement, il s’agira de texte brut tel que le code source ou un article de blog. Mais cela pourrait tout aussi bien être les octets d’un fichier PNG ou autre chose.

Si vous avez le hachage d’un blob, vous pouvez regarder son contenu.

$ git cat-file -p d429810
package com.example.project

class Foo {
 ...
}
...

Par exemple, vous pouvez parcourir un tree comme ci-dessus, puis regarder l’un des blobs qu’il contient.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
100644 blob cae391ff   Readme.txt

$ git cat-file -p cae391ff
Welcome to my project! This is the readmefile
...

Création de nouveaux commits #

La commande git commit fait plusieurs choses :

  1. Créez des blobs et des trees pour représenter votre répertoire de projet - stockés dans .git/objects
  2. Crée un nouvel objet commit avec vos informations sur l’auteur, le message de validation et la racine tree de l’étape 1 - également stocké dans .git/objects
  3. Met à jour la référence HEAD dans .git/HEAD avec le hachage du commit nouvellement créé

Cela se traduit par l’ajout d’un nouvel instantané de votre projet à git qui est connecté à l’état précédent.

Tête mobile #

Lorsque vous exécutez git checkout sur un commit (spécifié par hash ou ref), vous dites à git de faire en sorte que votre répertoire de travail ressemble à ce qu’il était lorsque l’instantané a été pris.

  1. Mettez à jour les fichiers dans le répertoire de travail pour qu’ils correspondent à “l’arborescence” à l’intérieur du “commit”
  2. Mettez à jour HEAD pour pointer vers le hachage ou la référence spécifié

Déplacer les références #

L’exécution de git reset --hard déplace les références vers le hachage/référence spécifié.

Déplacement de MyBranch vers b8dc53 :

$ git checkout MyBranch      # moves HEAD to MyBranch
$ git reset --hard b8dc53    # makes MyBranch point to b8dc53

Création de nouvelles références #

L’exécution de git checkout -b <refname> créera une nouvelle référence qui pointe vers le commit actuel.

$ cat .git/head
1f324a

$ git checkout -b TestBranch

$ cat .git/refs/heads/TestBranch
1f324a
