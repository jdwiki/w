Qu’est-ce que l’écrasement ? #

Le squashing est le processus consistant à prendre plusieurs commits et à les combiner en un seul commit encapsulant toutes les modifications depuis les commits initiaux.

Écrasement et succursales distantes #

Portez une attention particulière lors de l’écrasement des commits sur une branche qui suit une branche distante ; si vous écrasez un commit qui a déjà été poussé vers une branche distante, les deux branches divergeront et vous devrez utiliser git push -f pour forcer ces modifications sur la branche distante. Sachez que cela peut entraîner des problèmes pour les autres qui suivent cette branche distante, vous devez donc faire preuve de prudence lorsque vous poussez de force des commits écrasés sur des référentiels publics ou partagés.

Si le projet est hébergé sur GitHub, vous pouvez activer la “protection contre la poussée forcée” sur certaines branches, comme master, en l’ajoutant à Settings - Branches - Protected Branches.

Écraser les commits récents sans rebaser #

Si vous souhaitez écraser les commits x précédents en un seul, vous pouvez utiliser les commandes suivantes :

git reset --soft HEAD~x
git commit

Remplacez x par le nombre de commits précédents que vous souhaitez inclure dans le commit écrasé.

N’oubliez pas que cela créera un * nouveau * commit, oubliant essentiellement les informations sur les commits x précédents, y compris leur auteur, leur message et leur date. Vous voudrez probablement d’abord copier-coller un message de validation existant.

Écraser les commits lors d’un rebase #

Les commits peuvent être écrasés lors d’un git rebase. Il est recommandé de comprendre rebasing avant d’essayer d’écraser les commits de cette manière.

  1. Déterminez à partir de quel commit vous souhaitez rebaser et notez son hachage de commit.
  2. Exécutez git rebase -i [commit hash].

Alternativement, vous pouvez taper HEAD~4 au lieu d’un hachage de commit, pour afficher le dernier commit et 4 autres commits avant le dernier. 3. Dans l’éditeur qui s’ouvre lors de l’exécution de cette commande, déterminez les commits que vous souhaitez écraser. Remplacez pick au début de ces lignes par squash pour les écraser dans le commit précédent. 4. Après avoir sélectionné les commits que vous souhaitez écraser, vous serez invité à écrire un message de commit.

Journalisation des commits pour déterminer où rebaser

> git log --oneline
612f2f7 This commit should not be squashed
d84b05d This commit should be squashed
ac60234 Yet another commit
36d15de Rebase from here
17692d1 Did some more stuff
e647334 Another Commit
2e30df6 Initial commit

> git rebase -i 36d15de

À ce stade, votre éditeur de choix apparaît et vous pouvez décrire ce que vous voulez faire avec les commits. Git fournit de l’aide dans les commentaires. Si vous le laissez tel quel, rien ne se passera car chaque commit sera conservé et leur ordre sera le même qu’avant le rebase. Dans cet exemple, nous appliquons les commandes suivantes :

pick ac60234 Yet another commit
squash d84b05d This commit should be squashed
pick 612f2f7 This commit should not be squashed

# Rebase 36d15de..612f2f7 onto 36d15de (3 command(s))
#
# Commands:
# p, pick = use commit
# r, reword = use commit, but edit the commit message
# e, edit = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup = like "squash", but discard this commit's log message
# x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#
# If you remove a line here THAT COMMIT WILL BE LOST.
#
# However, if you remove everything, the rebase will be aborted.
#
# Note that empty commits are commented out

Journal Git après avoir écrit le message de validation

> git log --oneline
77393eb This commit should not be squashed
e090a8c Yet another commit
36d15de Rebase from here
17692d1 Did some more stuff
e647334 Another Commit
2e30df6 Initial commit

Écraser la validation pendant la fusion #

Vous pouvez utiliser git merge --squash pour écraser les changements introduits par une branche dans un seul commit. Aucun commit réel ne sera créé.

git merge --squash <branch>
git commit

C’est plus ou moins équivalent à l’utilisation de git reset, mais c’est plus pratique lorsque les changements incorporés ont un nom symbolique. Comparer:

git checkout <branch>
git reset --soft $(git merge-base master <branch>)
git commit

Autosquash : validation du code que vous souhaitez écraser lors d’un rebase #

Compte tenu de l’historique suivant, imaginez que vous apportez une modification que vous souhaitez insérer dans le commit bbb2222 A second commit :

$ git log --oneline --decorate
ccc3333 (HEAD -> master) A third commit
bbb2222 A second commit
aaa1111 A first commit
9999999 Initial commit

Une fois que vous avez effectué vos modifications, vous pouvez les ajouter à l’index comme d’habitude, puis les valider en utilisant l’argument --fixup avec une référence au commit dans lequel vous souhaitez vous écraser :

$ git add .
$ git commit --fixup bbb2222
[my-feature-branch ddd4444] fixup! A second commit

Cela créera un nouveau commit avec un message de commit que Git pourra reconnaître lors d’un rebase interactif :

$ git log --oneline --decorate
ddd4444 (HEAD -> master) fixup! A second commit
ccc3333 A third commit
bbb2222 A second commit
aaa1111 A first commit
9999999 Initial commit

Ensuite, faites un rebase interactif avec l’argument --autosquash :

$ git rebase --autosquash --interactive HEAD~4

Git vous proposera d’écraser le commit que vous avez fait avec le commit --fixup dans la bonne position :

pick aaa1111 A first commit
pick bbb2222 A second commit
fixup ddd4444 fixup! A second commit
pick ccc3333 A third commit

Pour éviter d’avoir à taper --autosquash à chaque rebase, vous pouvez activer cette option par défaut :

$ git config --global rebase.autosquash true

Autosquashing et corrections #

Lors de la validation des modifications, il est possible de spécifier que le commit sera à l’avenir écrasé par un autre commit et cela peut être fait comme ceci,

git commit --squash=[commit hash du commit vers lequel ce commit sera écrasé]

On peut également utiliser --fixup=[commit hash] alternativement à fixup.

Il est également possible d’utiliser des mots du message de validation au lieu du hachage de validation, comme ceci,

git commit --squash :/choses

où le commit le plus récent avec le mot “choses” serait utilisé.

Le message de ces commits commencerait par 'fixup!' ou 'squash!' suivi du reste du message de commit vers lequel ces commits seront écrasés.

Lors du changement de base, le drapeau --autosquash doit être utilisé pour utiliser la fonction autosquash/fixup.
