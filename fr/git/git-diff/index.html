Syntaxe #

  • git diff [options] [<commit>] [--] [<chemin>…​]
  • git diff [options] --cached [<commit>] [--] [<chemin>…​]
  • git diff [options] <commit> <commit> [--] [<chemin>…​]
  • git diff [options] <blob> <blob>
  • git diff [options] [--no-index] [--] <chemin> <chemin>

Paramètres #

Paramètre Détails
-p, -u, –patch Générer un correctif
-s, –no-patch Supprimer la sortie diff. Utile pour les commandes comme git show qui affichent le patch par défaut, ou pour annuler l’effet de --patch
–raw Générer le diff au format brut
–diff-algorithme= Choisissez un algorithme de diff. Les variantes sont les suivantes : myers, minimal, patience, histogram
–résumé Générer un résumé condensé des informations d’en-tête étendues telles que les créations, les changements de nom et de mode
–nom-seulement Afficher uniquement les noms des fichiers modifiés
–nom-statut Afficher les noms et les statuts des fichiers modifiés Les statuts les plus courants sont M (Modifié), A (Ajouté) et D (Supprimé)
–vérifier Avertir si des modifications introduisent des marqueurs de conflit ou des erreurs d’espacement. Ce qui est considéré comme des erreurs d’espace blanc est contrôlé par la configuration core.whitespace. Par défaut, les espaces blancs de fin (y compris les lignes composées uniquement d’espaces blancs) et un caractère d’espace immédiatement suivi d’un caractère de tabulation à l’intérieur du retrait initial de la ligne sont considérés comme des erreurs d’espace blanc. Quitte avec un état différent de zéro si des problèmes sont détectés. Non compatible avec –exit-code
–index complet Au lieu de la première poignée de caractères, affichez les noms complets des objets blob pré et post-image sur la ligne “index” lors de la génération d’une sortie au format patch
–binary En plus de --full-index, affiche un diff binaire qui peut être appliqué avec git apply
-a, –texte Traitez tous les fichiers comme du texte.
–couleur Définissez le mode couleur ; c’est-à-dire utilisez --color=always si vous souhaitez diriger un diff vers less et conserver la coloration de git

Afficher les différences dans la branche de travail #

git diff

Cela affichera les modifications [unstaged][1] sur la branche actuelle à partir du commit précédent. Il n’affichera que les changements relatifs à l’index, ce qui signifie qu’il montre ce que vous pourriez ajouter au prochain commit, mais que vous ne l’avez pas fait. Pour ajouter (étaper) ces changements, vous pouvez utiliser git add.

Si un fichier est mis en scène, mais a été modifié après avoir été mis en scène, git diff affichera les différences entre le fichier actuel et la version mise en scène.

[1] : https://www.wikiod.com/fr/git/mise-en-scene

Afficher les changements entre deux commits #

git diff 1234abc..6789def    # old   new

Ex. : Afficher les modifications apportées dans les 3 derniers commits :

git diff @~3..@    # HEAD -3   HEAD

Remarque : les deux points (..) sont facultatifs, mais ajoutent de la clarté.

Cela montrera la différence textuelle entre les commits, quel que soit leur emplacement dans l’arborescence.

Afficher les différences pour les fichiers préparés #

git diff --staged

Cela montrera les changements entre le commit précédent et les fichiers actuellement mis en scène.

REMARQUE : Vous pouvez également utiliser les commandes suivantes pour accomplir la même chose :

git diff --cached

Qui n’est qu’un synonyme de “–staged” ou

git status -v

Ce qui déclenchera les paramètres détaillés de la commande status.

Comparer les branches #

Affichez les changements entre la pointe de new et la pointe de original :

git diff original new     # equivalent to original..new

Afficher toutes les modifications sur nouveau depuis qu’il est dérivé de original :

git diff original...new     # equivalent to $(git merge-base original new)..new

En utilisant un seul paramètre tel que

git diff d’origine

est équivalent à

git diff original..HEAD

Afficher les modifications mises en scène et non mises en scène #

Pour afficher toutes les modifications mises en scène et non mises en scène, utilisez :

git diff HEAD

REMARQUE : Vous pouvez également utiliser la commande suivante :

git status -vv

La différence étant que la sortie de ce dernier vous indiquera en fait quelles modifications sont mises en scène pour la validation et lesquelles ne le sont pas.

Afficher les différences pour un fichier ou un répertoire spécifique #

git diff myfile.txt

Affiche les changements entre la validation précédente du fichier spécifié (myfile.txt) et la version modifiée localement qui n’a pas encore été préparée.

Cela fonctionne également pour les répertoires :

git diff documentation

Ce qui précède montre les changements entre la validation précédente de tous les fichiers dans le répertoire spécifié (documentation/) et les versions modifiées localement de ces fichiers, qui n’ont pas encore été mises en scène.

Pour montrer la différence entre une version d’un fichier dans un commit donné et la version locale de HEAD, vous pouvez spécifier le commit avec lequel vous souhaitez comparer :

git diff 27fa75e myfile.txt

Ou si vous souhaitez voir la version entre deux commits distincts :

git diff 27fa75e ada9b57 myfile.txt

Pour afficher la différence entre la version spécifiée par le hachage ada9b57 et le dernier commit sur la branche my_branchname uniquement pour le répertoire relatif appelé my_changed_directory/, vous pouvez faire ceci :

git diff ada9b57 my_branchname my_changed_directory/

Affichage d’un word-diff pour les longues lignes #

git diff [HEAD|--staged...] --word-diff

Plutôt que d’afficher les lignes modifiées, cela affichera les différences au sein des lignes. Par exemple, plutôt que :

-Hello world
+Hello world!

Lorsque toute la ligne est marquée comme modifiée, word-diff modifie la sortie en :

Hello [-world-]{+world!+}

Vous pouvez omettre les marqueurs [-, -], {+, +} en spécifiant --word-diff=color ou --color-words. Cela n’utilisera que le codage couleur pour marquer la différence :

[![Exemple d’utilisation de git diff –color-words][1]][1]

[1] : http://i.stack.imgur.com/1vsUP.png

Utiliser meld pour voir toutes les modifications dans le répertoire de travail #

git difftool -t meld --dir-diff

affichera les modifications apportées au répertoire de travail. Alternativement,

git difftool -t meld --dir-diff  [COMMIT_A] [COMMIT_B]

montrera les différences entre 2 commits spécifiques.

Afficher les différences entre la version actuelle et la dernière version #

git diff HEAD^ HEAD

Cela montrera les changements entre le commit précédent et le commit actuel.

Produire un diff compatible avec les patchs #

Parfois, vous avez juste besoin d’un diff à appliquer à l’aide de patch. Le git --diff normal ne fonctionne pas. Essayez ceci à la place :

git diff --no-prefix > some_file.patch

Ensuite, ailleurs, vous pouvez l’inverser :

patch -p0 < some_file.patch

différence entre deux commit ou branchement #

Pour voir la différence entre deux branches

git diff <branch1>..<branch2>

Pour voir la différence entre deux branches

git diff <commitId1>..<commitId2>

Pour afficher le diff avec la branche actuelle

git diff <branch/commitId>

Pour afficher le résumé des modifications

git diff --stat <branch/commitId>

Pour afficher les fichiers modifiés après un certain commit

git diff --name-only <commitId>

Pour afficher des fichiers différents d’une branche

git diff --name-only <branchName>

Pour afficher les fichiers modifiés dans un dossier après un certain commit

git diff --name-only <commitId> <folder_path>

Texte encodé Diff UTF-16 et fichiers plist binaires #

Vous pouvez différencier les fichiers encodés en UTF-16 (fichier de chaînes de localisation sous iOS et macOS en sont des exemples) en spécifiant comment git doit différencier ces fichiers.

Ajoutez ce qui suit à votre fichier ~/.gitconfig.

[diff "utf16"]
textconv = "iconv -f utf-16 -t utf-8"

iconv est un programme pour [convertir différents encodages] (http://linux.die.net/man/1/iconv).

Ensuite, modifiez ou créez un fichier .gitattributes à la racine du référentiel où vous souhaitez l’utiliser. Ou modifiez simplement ~/.gitattributes.

*.strings diff=utf16

Cela convertira tous les fichiers se terminant par .strings avant git diffs.

Vous pouvez faire des choses similaires pour d’autres fichiers, qui peuvent être convertis en texte.

Pour les fichiers plist binaires, vous modifiez .gitconfig

[diff "plist"]
textconv = plutil -convert xml1 -o -

et .gitattributes

*.plist diff=plist

Affichage d’une fusion à trois voies incluant l’ancêtre commun #

git config --global merge.conflictstyle diff3

Définit le style diff3 par défaut : au lieu du format habituel dans les sections en conflit, affichant les deux fichiers :

<<<<<<< HEAD
left
=======
right
>>>>>>> master

il comprendra une section supplémentaire contenant le texte original (issu de l’ancêtre commun) :

<<<<<<< HEAD
first
second
||||||| 
first
=======
last
>>>>>>> master

Ce format facilite la compréhension du conflit de fusion, c’est-à-dire. dans ce cas, localement ‘second’ a été ajouté, tandis que remote a remplacé ‘first’ par ’last’, en résolvant :

last
second

La même résolution aurait été beaucoup plus difficile en utilisant la valeur par défaut :

<<<<<<< HEAD
first
second
=======
last
>>>>>>> master

git diff branch1..branch2
