Syntaxe #

  • git rebase [-i | --interactive] [options] [--exec <cmd>] [--onto <newbase>] [<upstream>] [<branch>]
  • git rebase [-i | --interactive] [options] [--exec <cmd>] [--onto <newbase>] --root [<branch>]
  • git rebase --continue | --skip | --abort | --edit-all

Paramètres #

Paramètre Détails
–continuer Redémarrez le processus de changement de base après avoir résolu un conflit de fusion.
–abort Abandonnez l’opération de rebase et réinitialisez HEAD sur la branche d’origine. Si la branche a été fournie lorsque l’opération de rebase a été lancée, alors HEAD sera réinitialisé sur la branche. Sinon, HEAD sera réinitialisé à l’endroit où il se trouvait lorsque l’opération de rebasage a été lancée.
–keep-empty Conservez les commits qui ne changent rien de ses parents dans le résultat.
–skip Redémarrez le processus de changement de base en sautant le patch actuel.
-m, –merge Utilisez des stratégies de fusion pour rebaser. Lorsque la stratégie de fusion récursive (par défaut) est utilisée, cela permet à rebase d’être conscient des changements de nom en amont. Notez qu’une fusion de rebase fonctionne en rejouant chaque commit de la branche de travail au-dessus de la branche en amont. Pour cette raison, lorsqu’un conflit de fusion se produit, le côté signalé comme le nôtre est la série rebasée jusqu’à présent, en commençant par l’amont, et le leur est la branche de travail. En d’autres termes, les côtés sont inversés.
–stat Afficher un diffstat de ce qui a changé en amont depuis le dernier rebasage. Le diffstat est également contrôlé par l’option de configuration rebase.stat.
-x, –exec commande Effectuer un rebase interactif, en s’arrêtant entre chaque commit et en exécutant command

Veuillez garder à l’esprit que rebase réécrit efficacement l’historique du référentiel.

Le rebasage des commits qui existent dans le référentiel distant pourrait réécrire les nœuds du référentiel utilisés par d’autres développeurs comme nœud de base pour leurs développements. À moins que vous ne sachiez vraiment ce que vous faites, il est recommandé de rebaser avant de pousser vos modifications.

Rebasage de la succursale locale #

[Rebasing][1] réapplique une série de commits au-dessus d’un autre commit.

Pour « rebaser » une branche, extrayez la branche, puis « rebasez-la » au-dessus d’une autre branche.

git checkout topic
git rebase master  # rebase current branch onto master branch

Cela entraînerait :

      A---B---C topic
     /
D---E---F---G master

Se transformer en:

              A'--B'--C' topic
             /
D---E---F---G master

Ces opérations peuvent être combinées en une seule commande qui extrait la branche et la rebase immédiatement :

git rebase master topic   # rebase topic branch onto master branch

Important : Après le rebase, les commits appliqués auront un hachage différent. Vous ne devez pas rebaser les commits que vous avez déjà poussés vers un hôte distant. Une conséquence peut être une incapacité à git push votre branche locale rebasée vers un hôte distant, laissant votre seule option à git push --force.

[1] : https://git-scm.com/docs/git-rebase

Rebase : le nôtre et le leur, local et distant #

Un rebase change le sens de “le nôtre” et “le leur”:

git checkout topic
git rebase   master    # rebase topic branch on top of master branch

Tout ce que pointe HEAD est “le nôtre”

La première chose qu’un rebase fait est de réinitialiser le HEAD à master; avant de sélectionner les commits de l’ancienne branche “topic” vers une nouvelle (chaque commit de l’ancienne branche “topic” sera réécrit et sera identifié par un hachage différent).

En ce qui concerne les terminologies utilisées par les outils de fusion (à ne pas confondre avec local ref or remote ref)

=> local is master ("ours"),
=> remote is topic ("theirs")

Cela signifie qu’un outil de fusion/diff présentera la branche en amont comme local (master : la branche au-dessus de laquelle vous rebasez) et la branche de travail comme remote (topic : la branche en cours de rebasage)

+-----------------------------------------+
| LOCAL:master |    BASE   | REMOTE:topic |
+-----------------------------------------+
|             MERGED                      |
+-----------------------------------------+

Inversion illustrée #

Sur une fusion : #

c--c--x--x--x(*) <- current branch topic ('*'=HEAD)
    \
     \
      \--y--y--y <- other branch to merge

Nous ne changeons pas la branche actuelle topic, donc ce que nous avons est toujours ce sur quoi nous travaillions (et nous fusionnons à partir d’une autre branche)

c--c--x--x--x---------o(*)  MERGE, still on branch topic
    \       ^        /
     \     ours     /
      \            /
       --y--y--y--/  
               ^
              theirs

Sur un rebase : #

Mais sur un rebase, nous changeons de côté car la première chose que fait un rebase est de vérifier la branche en amont pour rejouer les commits actuels par-dessus !

c--c--x--x--x(*) <- current branch topic ('*'=HEAD)
    \
     \
      \--y--y--y <- upstream branch

Un git rebase en amont définira d’abord HEAD sur la branche en amont, d’où le changement de ‘ours’ et ’theirs’ par rapport à la branche de travail “actuelle” précédente.

c--c--x--x--x <- former "current" branch, new "theirs"
    \
     \
      \--y--y--y(*) <- set HEAD to this commit, to replay x's on it
               ^       this will be the new "ours"
               |
            upstream

Le rebase rejouera alors “leurs” commits sur la nouvelle branche “notre” “topic”:

c--c..x..x..x <- old "theirs" commits, now "ghosts", available through "reflogs"
    \
     \
      \--y--y--y--x'--x'--x'(*) <- topic  once all x's are replayed,
               ^                      point branch topic to this commit
               |
        upstream branch

Rebase interactive #

Cet exemple vise à décrire comment on peut utiliser git rebase en mode interactif. On s’attend à ce que l’on ait une compréhension de base de ce qu’est “git rebase” et de ce qu’il fait.

Le rebase interactif est lancé à l’aide de la commande suivante :

git rebase -i

L’option -i fait référence au mode interactif. À l’aide de la rebase interactive, l’utilisateur peut modifier les messages de validation, ainsi que réorganiser, diviser et/ou écraser (combiner en un seul) les validations.

Supposons que vous souhaitiez réorganiser vos trois derniers commits. Pour ce faire, vous pouvez exécuter :

git rebase -i HEAD~3

Après avoir exécuté l’instruction ci-dessus, un fichier sera ouvert dans votre éditeur de texte où vous pourrez sélectionner la façon dont vos commits seront rebasés. Pour les besoins de cet exemple, modifiez simplement l’ordre de vos commits, enregistrez le fichier et fermez l’éditeur. Cela lancera une nouvelle base avec la commande que vous avez appliquée. Si vous cochez git log, vous verrez vos commits dans le nouvel ordre que vous avez spécifié.

Reformulation des messages de commit #

Maintenant, vous avez décidé que l’un des messages de validation est vague et vous souhaitez qu’il soit plus descriptif. Examinons les trois derniers commits en utilisant la même commande.

git rebase -i HEAD~3

Au lieu de réorganiser l’ordre dans lequel les commits seront rebasés, cette fois nous allons changer pick, la valeur par défaut, en reword sur un commit où vous voudriez changer le message.

Lorsque vous fermez l’éditeur, le rebase démarre et s’arrête au message de validation spécifique que vous vouliez reformuler. Cela vous permettra de changer le message de validation en celui que vous désirez. Après avoir modifié le message, fermez simplement l’éditeur pour continuer.

Changer le contenu d’un commit #

Outre la modification du message de validation, vous pouvez également adapter les modifications apportées par la validation. Pour ce faire, remplacez simplement pick par edit pour un commit. Git s’arrêtera lorsqu’il arrivera à ce commit et fournira les modifications d’origine du commit dans la zone de staging. Vous pouvez maintenant adapter ces modifications en les annulant ou en ajoutant de nouvelles modifications.

Dès que la zone de préparation contient toutes les modifications souhaitées dans ce commit, validez les modifications. L’ancien message de commit sera affiché et peut être adapté pour refléter le nouveau commit.

Diviser un seul commit en plusieurs #

Supposons que vous ayez fait un commit mais que vous ayez décidé ultérieurement que ce commit pourrait être divisé en deux ou plusieurs commits à la place. En utilisant la même commande que précédemment, remplacez pick par edit à la place et appuyez sur Entrée.

Maintenant, git s’arrêtera au commit que vous avez marqué pour l’édition et placera tout son contenu dans la zone de staging. À partir de là, vous pouvez exécuter git reset HEAD^ pour placer le commit dans votre répertoire de travail. Ensuite, vous pouvez ajouter et valider vos fichiers dans un ordre différent - en divisant finalement un seul commit en n commits à la place.

Écraser plusieurs commits en un seul #

Supposons que vous ayez fait du travail et que vous ayez plusieurs commits qui, selon vous, pourraient être un seul commit à la place. Pour cela, vous pouvez exécuter git rebase -i HEAD~3, en remplaçant 3 par un nombre approprié de commits.

Cette fois, remplacez pick par squash à la place. Lors de la rebase, le commit que vous avez demandé d’écraser sera écrasé par-dessus le commit précédent ; les transformant en un seul commit à la place.

Rebase jusqu’au commit initial #

Depuis Git 1.7.12, il est possible de rebaser jusqu’au commit racine. Le commit racine est le premier commit jamais effectué dans un référentiel et ne peut normalement pas être modifié. Utilisez la commande suivante :

git rebase -i --root

Rebasage avant une revue de code #

Sommaire #

Cet objectif est de réorganiser tous vos commits dispersés en commits plus significatifs pour faciliter les revues de code. S’il y a trop de couches de modifications dans trop de fichiers à la fois, il est plus difficile de faire une révision du code. Si vous pouvez réorganiser vos commits créés chronologiquement en commits thématiques, le processus de révision du code est plus facile (et peut-être moins de bogues se glissent dans le processus de révision du code).

Cet exemple trop simplifié n’est pas la seule stratégie d’utilisation de git pour faire de meilleures revues de code. C’est ma façon de faire, et c’est quelque chose qui inspire les autres à réfléchir à la façon de rendre les révisions de code et l’historique de git plus faciles/meilleurs.

Cela démontre aussi pédagogiquement la puissance du rebase en général.

Cet exemple suppose que vous connaissez le rebasage interactif.

En supposant: #

  • vous travaillez sur une branche de fonctionnalité hors du maître
  • votre fonctionnalité comporte trois couches principales : front-end, back-end, DB
  • vous avez fait beaucoup de commits en travaillant sur une branche de fonctionnalité. Chaque commit touche plusieurs couches à la fois
  • vous ne voulez (au final) que trois commits dans votre branche
  • un contenant tous les changements frontaux
  • un contenant tous les changements de back-end
  • un contenant tous les changements de DB

Stratégie: #

  • nous allons changer nos commits chronologiques en commits “topiques”.
  • d’abord, divisez tous les commits en plusieurs commits plus petits - chacun contenant un seul sujet à la fois (dans notre exemple, les sujets sont le front-end, le back-end, les changements de base de données)
  • Ensuite, réorganisez nos commits thématiques ensemble et « écrasez-les » en un seul commit thématique

Exemple: #

$ git log --oneline master..
975430b db adding works: db.sql logic.rb
3702650 trying to allow adding todo items: page.html logic.rb
43b075a first draft: page.html and db.sql
$ git rebase -i master

Cela sera affiché dans l’éditeur de texte :

pick 43b075a first draft: page.html and db.sql
pick 3702650 trying to allow adding todo items: page.html logic.rb
pick 975430b db adding works: db.sql logic.rb

Changez-le en ceci :

e 43b075a first draft: page.html and db.sql
e 3702650 trying to allow adding todo items: page.html logic.rb
e 975430b db adding works: db.sql logic.rb

Ensuite, git appliquera un commit à la fois. Après chaque validation, une invite s’affichera, puis vous pourrez effectuer les opérations suivantes :

Stopped at 43b075a92a952faf999e76c4e4d7fa0f44576579... first draft: page.html and db.sql
You can amend the commit now, with

        git commit --amend

Once you are satisfied with your changes, run

        git rebase --continue

$ git status
rebase in progress; onto 4975ae9
You are currently editing a commit while rebasing branch 'feature' on '4975ae9'.
  (use "git commit --amend" to amend the current commit)
  (use "git rebase --continue" once you are satisfied with your changes)

nothing to commit, working directory clean
$ git reset HEAD^ #This 'uncommits' all the changes in this commit.
$ git status -s
 M db.sql
 M page.html
$ git add db.sql  #now we will create the smaller topical commits
$ git commit -m "first draft: db.sql"
$ git add page.html
$ git commit -m "first draft: page.html"
$ git rebase --continue

Ensuite, vous répéterez ces étapes pour chaque commit. Au final, tu as ça :

$ git log --oneline
0309336 db adding works: logic.rb
06f81c9 db adding works: db.sql
3264de2 adding todo items: page.html
675a02b adding todo items: logic.rb
272c674 first draft: page.html
08c275d first draft: db.sql

Maintenant, nous exécutons rebase une fois de plus pour réorganiser et écraser :

$ git rebase -i master

Cela sera affiché dans l’éditeur de texte :

pick 08c275d first draft: db.sql
pick 272c674 first draft: page.html
pick 675a02b adding todo items: logic.rb
pick 3264de2 adding todo items: page.html
pick 06f81c9 db adding works: db.sql
pick 0309336 db adding works: logic.rb

Changez-le en ceci :

pick 08c275d first draft: db.sql
s 06f81c9 db adding works: db.sql
pick 675a02b adding todo items: logic.rb
s 0309336 db adding works: logic.rb
pick 272c674 first draft: page.html
s  3264de2 adding todo items: page.html

AVIS : assurez-vous que vous indiquez à git rebase d’appliquer/écraser les plus petits commits thématiques dans l’ordre dans lequel ils ont été validés chronologiquement. Sinon, vous pourriez avoir de faux conflits de fusion inutiles à gérer.

Lorsque ce rebase interactif est terminé, vous obtenez ceci :

$ git log --oneline master..
74bdd5f adding todos: GUI layer
e8d8f7e adding todos: business logic layer
121c578 adding todos: DB layer

résumer #

Vous avez maintenant rebasé vos commits chronologiques en commits thématiques. Dans la vraie vie, vous n’aurez peut-être pas besoin de le faire à chaque fois, mais quand vous voulez ou devez le faire, maintenant vous le pouvez. De plus, j’espère que vous en avez appris plus sur git rebase.

Test de tous les commits pendant le rebase #

Avant de faire une pull request, il est utile de s’assurer que la compilation est réussie et que les tests passent pour chaque commit dans la branche. Nous pouvons le faire automatiquement en utilisant le paramètre -x.

Par exemple:

git rebase -i -x make

effectuera le rebase interactif et s’arrêtera après chaque commit pour exécuter make. En cas d’échec de make, git s’arrêtera pour vous donner la possibilité de résoudre les problèmes et de modifier le commit avant de passer au suivant.

Configuration de l’autostash #

Autostash est une option de configuration très utile lors de l’utilisation de rebase pour les modifications locales. Souvent, vous devrez peut-être apporter des commits de la branche en amont, mais vous n’êtes pas encore prêt à vous engager.

Cependant, Git n’autorise pas le démarrage d’un rebase si le répertoire de travail n’est pas propre. Autostash à la rescousse :

git config --global rebase.autostash    # one time configuration
git rebase @{u}                         # example rebase on upstream branch

L’autostash sera appliqué chaque fois que le rebase sera terminé. Peu importe que la rebase se termine avec succès ou qu’elle soit abandonnée. Dans tous les cas, l’autostash sera appliqué. Si le rebase a réussi et que le commit de base a donc changé, il peut y avoir un conflit entre l’autostash et les nouveaux commits. Dans ce cas, vous devrez résoudre les conflits avant de vous engager. Ce n’est pas différent que si vous aviez stocké manuellement, puis appliqué, il n’y a donc aucun inconvénient à le faire automatiquement.

Abandon d’un rebase interactif #

Vous avez démarré un rebase interactif. Dans l’éditeur où vous choisissez vos commits, vous décidez que quelque chose ne va pas (par exemple, un commit est manquant ou vous avez choisi la mauvaise destination de rebase), et vous voulez abandonner le rebase.

Pour ce faire, supprimez simplement tous les commits et actions (c’est-à-dire toutes les lignes ne commençant pas par le signe #) et le rebase sera abandonné !

Le texte d’aide de l’éditeur fournit en fait cet indice :

# Rebase 36d15de..612f2f7 onto 36d15de (3 command(s))
#
# Commands:
# p, pick = use commit
# r, reword = use commit, but edit the commit message
# e, edit = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup = like "squash", but discard this commit's log message
# x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#

Configurez git-pull pour effectuer automatiquement un rebase au lieu d’une fusion #

Si votre équipe suit un flux de travail basé sur le rebase, il peut être avantageux de configurer git afin que chaque branche nouvellement créée effectue une opération de rebase, au lieu d’une opération de fusion, lors d’un git pull.

Pour configurer chaque nouvelle branche pour qu’elle se rebase automatiquement, ajoutez ce qui suit à votre .gitconfig ou .git/config :

[branch]
autosetuprebase = always

Ligne de commande : git config [--global] branch.autosetuprebase always

Alternativement, vous pouvez configurer la commande git pull pour qu’elle se comporte toujours comme si l’option --rebase était passée :

[pull]
rebase = true

Ligne de commande : git config [--global] pull.rebase true

Pousser après un rebase #

Parfois, vous avez besoin de réécrire l’historique avec une rebase, mais git push se plaint de le faire parce que vous avez réécrit l’historique.

Cela peut être résolu avec un git push --force, mais considérez git push --force-with-lease, indiquant que vous voulez que le push échoue si la branche locale de suivi à distance diffère de la branche sur la télécommande , par exemple, quelqu’un d’autre a poussé vers la télécommande après la dernière récupération. Cela évite d’écraser par inadvertance la poussée récente de quelqu’un d’autre.

Remarque : git push --force - et même --force-with-lease d’ailleurs - peut être une commande dangereuse car elle réécrit l’historique de la branche. Si une autre personne a extrait la branche avant le push forcé, son git pull ou git fetch contiendra des erreurs car l’historique local et l’historique distant sont divergents. Cela peut amener la personne à avoir des erreurs inattendues. En consultant suffisamment les reflogs, le travail de l’autre utilisateur peut être récupéré, mais cela peut entraîner une perte de temps considérable. Si vous devez faire un push forcé vers une branche avec d’autres contributeurs, essayez de vous coordonner avec eux afin qu’ils n’aient pas à gérer les erreurs.
