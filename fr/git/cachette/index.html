Syntaxe #

  • liste de cachette git [<options>]
  • git stash show [<stash>]
  • git stash drop [-q|--quiet] [<stash>]
  • git stash ( pop | apply ) [--index] [-q|--quiet] [<stash>]
  • git stash branch <branchname> [<stash>]
  • git stash [save [-p|--patch] [-k|--[no-]keep-index] [-q|--quiet] [-u|--include-untracked] [-a| --all] [<message>]]
  • “git cachette claire”
  • git stash créer [<message>]
  • git stash store [-m|--message <message>] [-q|--quiet] <commit>

Paramètres #

Paramètre Détails
montrer Affiche les modifications enregistrées dans le stash sous la forme d’un diff entre l’état stash et son parent d’origine. Lorsqu’aucun <stash> n’est donné, affiche le dernier.
liste Dressez la liste des réserves que vous possédez actuellement. Chaque stash est répertorié avec son nom (par exemple, stash@{0} est le dernier stash, stash@{1} est celui d’avant, etc.), le nom de la branche qui était en cours lorsque le stash a été créé et un court description du commit sur lequel le stash était basé.
pop Supprime un seul état stocké de la liste de stockage et applique-le au-dessus de l’état actuel de l’arborescence de travail.
apply Comme pop, mais ne supprime pas l’état de la liste de cache.
clair Supprimez tous les états cachés. Notez que ces états seront alors soumis à un élagage et peuvent être impossibles à récupérer.
goutte Supprimer un seul état stocké de la liste de stockage. Lorsqu’aucun <stash> n’est donné, il supprime le dernier. c’est-à-dire stash@{0}, sinon <stash> doit être une référence de journal de cache valide de la forme stash@{<révision>}.
créer Créez une réserve (qui est un objet de validation standard) et renvoyez son nom d’objet, sans le stocker nulle part dans l’espace de noms ref. Ceci est destiné à être utile pour les scripts. Ce n’est probablement pas la commande que vous souhaitez utiliser ; voir “enregistrer” ci-dessus.
magasin Stockez un stash donné créé via git stash create (qui est un commit de fusion suspendu) dans la ref stash, en mettant à jour le reflog stash. Ceci est destiné à être utile pour les scripts. Ce n’est probablement pas la commande que vous souhaitez utiliser ; voir “enregistrer” ci-dessus.

Le stashing nous permet d’avoir un répertoire de travail propre sans perdre aucune information. Ensuite, il est possible de commencer à travailler sur quelque chose de différent et/ou de changer de branche.

Créer une réserve #

Enregistrez l’état actuel du répertoire de travail et de l’index (également appelé zone de staging) dans une pile de caches.

git stash

Pour inclure tous les fichiers non suivis dans la réserve, utilisez les drapeaux --include-untracked ou -u.

git stash --include-untracked

Pour inclure un message avec votre cachette pour la rendre plus facilement identifiable plus tard

git stash save "<whatever message>"

Pour laisser la zone de staging dans son état actuel après le stockage, utilisez les drapeaux --keep-index ou -k.

git stash --keep-index

Appliquer et supprimer la cachette #

Pour appliquer la dernière réserve et la supprimer de la pile, tapez :

git stash pop

Pour appliquer une réserve spécifique et la supprimer de la pile, tapez :

git stash pop stash@{n}

Appliquer la réserve sans la retirer #

Applique la dernière réserve sans la retirer de la pile

git stash apply

Ou une réserve spécifique

git stash apply stash@{n}

Afficher la réserve #

Affiche les modifications enregistrées dans la dernière réserve

git stash show

Ou une réserve spécifique

git stash show stash@{n}

Pour afficher le contenu des modifications enregistrées pour la réserve spécifique

git stash show -p stash@{n}

Qu’est-ce que le Stashing ? #

Lorsque vous travaillez sur un projet, vous êtes peut-être à mi-chemin d’un changement de branche de fonctionnalité lorsqu’un bogue est signalé à master. Vous n’êtes pas prêt à valider votre code, mais vous ne voulez pas non plus perdre vos modifications. C’est là que git stash devient utile.

Exécutez git status sur une branche pour afficher vos modifications non validées :

(master) $ git status
On branch master
Your branch is up-to-date with 'origin/master'.
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Exécutez ensuite git stash pour enregistrer ces modifications dans une pile :

(master) $ git stash
Saved working directory and index state WIP on master: 
2f2a6e1 Merge pull request #1 from test/test-branch
HEAD is now at 2f2a6e1 Merge pull request #1 from test/test-branch

Si vous avez ajouté des fichiers à votre répertoire de travail, ceux-ci peuvent également être stockés. Vous avez juste besoin de les mettre en scène en premier.

(master) $ git stash
Saved working directory and index state WIP on master:
(master) $ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)

        NewPhoto.c

nothing added to commit but untracked files present (use "git add" to track)
(master) $ git stage NewPhoto.c
(master) $ git stash
Saved working directory and index state WIP on master:
(master) $ git status
On branch master
nothing to commit, working tree clean
(master) $

Votre répertoire de travail est maintenant propre de toutes les modifications que vous avez apportées. Vous pouvez le voir en réexécutant git status :

(master) $ git status
On branch master
Your branch is up-to-date with 'origin/master'.
nothing to commit, working directory clean

Pour appliquer le tout dernier stash, exécutez git stash apply (en plus, vous pouvez appliquer et supprimer le dernier stash modifié avec git stash pop):

(master) $ git stash apply
On branch master
Your branch is up-to-date with 'origin/master'.
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Notez cependant que le stockage ne se souvient pas de la branche sur laquelle vous travailliez. Dans les exemples ci-dessus, l’utilisateur stockait sur master. S’ils basculent vers la branche dev, dev, et lancent git stash apply, la dernière réserve est placée sur la branche dev.

(master) $ git checkout -b dev
Switched to a new branch 'dev'
(dev) $ git stash apply
On branch dev
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Cachette partielle #

Si vous souhaitez stocker uniquement certains diffs dans votre ensemble de travail, vous pouvez utiliser un stockage partiel.

git stash -p

Et puis sélectionnez de manière interactive les mecs à cacher.

Depuis la version 2.13.0, vous pouvez également éviter le mode interactif et créer un cache partiel avec un pathspec en utilisant le nouveau mot-clé push.

git stash push -m "My partial stash" -- app.config

## Liste des caches enregistrées git stash list

Cela listera toutes les réserves de la pile dans l’ordre chronologique inverse. Vous obtiendrez une liste qui ressemble à ceci :

stash@{0}: WIP on master: 67a4e01 Merge tests into develop
stash@{1}: WIP on master: 70f0d95 Add user role to localStorage on user login

Vous pouvez faire référence à une réserve spécifique par son nom, par exemple stash@{1}.

Déplacez vos travaux en cours vers une autre succursale #

Si pendant que vous travaillez, vous vous rendez compte que vous êtes sur la mauvaise branche et que vous n’avez pas encore créé de commits, vous pouvez facilement déplacer votre travail vers la bonne branche en utilisant le stockage :

git stash
git checkout correct-branch
git stash pop

N’oubliez pas que git stash pop appliquera le dernier stash et le supprimera de la liste des stash. Pour conserver la réserve dans la liste et ne s’appliquer qu’à certaines branches, vous pouvez utiliser :

git stash apply

Supprimer la réserve #

Supprimer toute la cachette

git stash clear

Supprime la dernière réserve

git stash drop

Ou une réserve spécifique

git stash drop stash@{n}

Appliquer une partie d’une cachette avec paiement #

Vous avez créé une réserve et souhaitez extraire uniquement certains des fichiers de cette réserve.

git checkout stash@{0} -- myfile.txt

Stockage interactif #

Récupérer une réserve abandonnée #

Si vous venez juste de l’ouvrir et que le terminal est toujours ouvert, vous aurez toujours la valeur de hachage imprimée par git stash pop à l’écran :

$ git stash pop
[...]
Dropped refs/stash@{0} (2ca03e22256be97f9e40f08e6d6773c7d41dbfd1)

(Notez que git stash drop produit également la même ligne.)

Sinon, vous pouvez le trouver en utilisant ceci:

git fsck --no-reflog | awk '/dangling commit/ {print $3}'

Cela vous montrera tous les commits aux extrémités de votre graphique de commit qui ne sont plus référencés à partir d’aucune branche ou balise - chaque commit perdu, y compris chaque commit de réserve que vous avez jamais créé, sera quelque part dans ce graphique.

Le moyen le plus simple de trouver le commit de réserve que vous souhaitez est probablement de transmettre cette liste à gitk :

gitk --all $( git fsck --no-reflog | awk '/dangling commit/ {print $3}' )

Cela lancera un navigateur de référentiel vous montrant chaque commit dans le référentiel jamais, qu’il soit accessible ou non.

Vous pouvez remplacer gitk ici par quelque chose comme git log --graph --oneline --decorate si vous préférez un joli graphique sur la console plutôt qu’une application GUI séparée.

Pour repérer les commits stash, recherchez les messages de commit de cette forme :

        WIP sur somebranch : commithash Un vieux message de commit

Une fois que vous connaissez le hachage du commit que vous souhaitez, vous pouvez l’appliquer en tant que réserve :

git stash appliquer $stash_hash

Ou vous pouvez utiliser le menu contextuel de gitk pour créer des branches pour tous les commits inaccessibles qui vous intéressent. Après cela, vous pouvez en faire ce que vous voulez avec tous les outils normaux. Lorsque vous avez terminé, soufflez à nouveau ces branches.
