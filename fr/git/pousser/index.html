Après avoir modifié, mis en scène et validé le code avec Git, le push est nécessaire pour mettre vos modifications à la disposition des autres et transférer vos modifications locales vers le serveur de référentiel. Cette rubrique explique comment pousser correctement le code à l’aide de Git.

Syntaxe #

  • git pousser [-f | –force] [-v | –verbose] [<distant> [<refspec>…]]

Paramètres #

Paramètre Détails
–force Remplace la référence distante pour correspondre à votre référence locale. Peut entraîner la perte de commits par le référentiel distant, utilisez-le donc avec précaution.
–verbeux Exécutez verbalement.
<distant> Le référentiel distant qui est la destination de l’opération push.
<spécification de référence>… Spécifiez quelle référence distante mettre à jour avec quelle référence ou objet local.

Amont Aval #

En termes de contrôle de code source, vous êtes “en aval” lorsque vous copiez (clonez, paiement, etc.) à partir d’un référentiel. L’information circulait « en aval » vers vous.

Lorsque vous apportez des modifications, vous souhaitez généralement les renvoyer “en amont” afin qu’ils le fassent dans ce référentiel afin que tous ceux qui tirent du même source fonctionne avec tous les mêmes changements. Il s’agit surtout d’un enjeu social de savoir comment chacun peut coordonner son travail plutôt qu’un exigence technique de contrôle de source. Vous souhaitez obtenir vos modifications dans le projet principal afin de ne pas suivre des lignes divergentes de développement.

Parfois, vous lirez des informations sur les gestionnaires de packages ou de versions (les personnes, pas l’outil) parlant de soumettre des modifications à “en amont”. Ce signifie généralement qu’ils ont dû ajuster les sources d’origine pour pouvoir créer un package pour leur système. Ils ne veulent pas continuer à faire ces changements, donc s’ils les envoient “en amont” vers la source d’origine, ils ne devraient pas avoir à traiter le même problème dans la prochaine version.

([Source][1])

[1] : http://stackoverflow.com/a/2739476/163024

Pousser un objet spécifique vers une branche distante #

Syntaxe générale #

git push <remotename> <object>:<remotebranchname>

Exemple #

git push origin master:wip-yourname

Poussera votre branche principale vers la branche d’origine wip-yourname (la plupart du temps, le référentiel à partir duquel vous avez cloné).

Supprimer la branche distante #

Supprimer la branche distante revient à y insérer un objet vide.

git push <remotename> :<remotebranchname>

Exemple #

git push origin :wip-yourname

Supprime la branche distante wip-yourname

Au lieu d’utiliser les deux-points, vous pouvez également utiliser l’indicateur –delete, qui est plus lisible dans certains cas.

Exemple #

git push origin --delete wip-yourname

Pousser un seul commit #

Si vous avez un seul commit dans votre branche que vous souhaitez pousser vers une télécommande sans rien pousser d’autre, vous pouvez utiliser ce qui suit

git push <remotename> <commit SHA>:<remotebranchname>

Exemple #

En supposant une histoire de git comme celle-ci

eeb32bc Commit 1 - already pushed
347d700 Commit 2 - want to push
e539af8 Commit 3 - only local
5d339db Commit 4 - only local

pour envoyer uniquement commit 347d700 au master distant, utilisez la commande suivante

git push origin 347d700:master

Pousser #

git push

poussera votre code vers votre amont existant. Selon la configuration push, il poussera soit le code de votre branche actuelle (par défaut dans Git 2.x) soit de toutes les branches (par défaut dans Git 1.x).

Spécifiez le référentiel distant #

Lorsque vous travaillez avec git, il peut être pratique d’avoir plusieurs référentiels distants. Pour spécifier un référentiel distant vers lequel pousser, ajoutez simplement son nom à la commande.

git push origin

Spécifiez la branche #

Pour pousser vers une branche spécifique, dites feature_x :

git push origin feature_x

Définir la branche de suivi à distance #

À moins que la branche sur laquelle vous travaillez ne provienne à l’origine d’un référentiel distant, le simple fait d’utiliser git push ne fonctionnera pas la première fois. Vous devez exécuter la commande suivante pour indiquer à git de pousser la branche actuelle vers une combinaison distante/branche spécifique

git push --set-upstream origin master

Ici, “master” est le nom de la branche sur l’“origine” distante. Vous pouvez utiliser -u comme raccourci pour --set-upstream.

Pousser vers un nouveau dépôt #

Pour pousser vers un dépôt que vous n’avez pas encore créé ou qui est vide :

  1. Créez le référentiel sur GitHub (le cas échéant)
  2. Copiez l’url qui vous est donnée, sous la forme https://github.com/USERNAME/REPO_NAME.git
  3. Accédez à votre référentiel local et exécutez git remote add origin URL
    • To verify it was added, run git remote -v
  4. Exécutez git push origin master

Votre code devrait maintenant être sur GitHub

Pour plus d’informations, consultez [Ajout d’un référentiel distant][1]

Explication #

Le code push signifie que git analysera les différences de vos commits locaux et distants et les enverra pour être écrits en amont. Lorsque le push réussit, votre référentiel local et votre référentiel distant sont synchronisés et les autres utilisateurs peuvent voir vos commits.

Pour plus de détails sur les concepts d’“amont” et d’“aval”, voir [Remarques][2].

[1] : https://www.wikiod.com/fr/git/git-a-distance#Ajouter un référentiel distant [2] : //stackoverflow.com/documentation/git/2600/pushing#remarks

Forcer à pousser #

Parfois, lorsque vous avez des modifications locales incompatibles avec des modifications distantes (c’est-à-dire lorsque vous ne pouvez pas avancer rapidement la branche distante ou que la branche distante n’est pas un ancêtre direct de votre branche locale), la seule façon de pousser vos modifications est une poussée forcée .

git push -f

ou

git push --force

Notes IMPORTANTES #

Cela écrasera toutes les modifications à distance et votre télécommande correspondra à votre local.

Attention : L’utilisation de cette commande peut entraîner la perte de validations par le référentiel distant. De plus, il est fortement déconseillé de faire un push forcé si vous partagez ce référentiel distant avec d’autres, car leur historique conservera chaque commit écrasé, désynchronisant ainsi leur travail avec le référentiel distant.

En règle générale, ne forcez la poussée que lorsque :

  • Personne, sauf vous, n’a retiré les modifications que vous essayez d’écraser
  • Vous pouvez forcer tout le monde à cloner une nouvelle copie après la poussée forcée et faire en sorte que tout le monde y applique ses modifications (les gens peuvent vous détester pour cela).

Étiquettes push #

git push --tags

Pousse tous les tags git dans le référentiel local qui ne sont pas dans le référentiel distant.

Modification du comportement push par défaut #

Current met à jour la branche sur le référentiel distant qui partage un nom avec la branche de travail actuelle.

git config push.default current

Simple pousse vers la branche en amont, mais ne fonctionnera pas si la branche en amont porte un autre nom.

git config push.default simple

Amont pousse vers la branche en amont, quel que soit son nom.

git config push.default upstream

Matching pousse toutes les branches qui correspondent sur le local et le distant git config push.default upstream

Après avoir défini le style préféré, utilisez

git push

pour mettre à jour le référentiel distant.
