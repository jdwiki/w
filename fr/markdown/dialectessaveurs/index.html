Les variations de Markdown avec une syntaxe différente sont appelées “saveurs”. Les saveurs Markdown sont répertoriées sur https://github.com/jgm/CommonMark/wiki/Markdown-Flavors.

Une saveur Markdown peut être implémentée dans divers langages de programmation et applications logicielles. Les bibliothèques qui implémentent un analyseur Markdown sont répertoriées sur https://github.com/markdown/markdown.github.com/wiki/Implementations et https://www.w3.org/community/markdown/wiki/MarkdownImplementations.

Plusieurs versions de Markdown et leurs différences sont documentées sur http://flavoredmarkdown.com.

Pour tester et comparer le code Markdown dans de nombreuses versions différentes et les implémentations de ces versions, vous pouvez utiliser l’outil en ligne Babelmark.

Markdown de débordement de pile #

C’est le type de démarquage utilisé par Stack Overflow et d’autres sites Stack Exchange. Lorsque vous répondez à une question ou ajoutez de la documentation, vous utilisez cette démarque. Cette réponse est faite de SO markdown

Voir [Documentation officielle][1]

Les principaux éléments ajoutés par SO markdown se trouvent sous “Ajouts Stack Exchange” sur cette page Web. En particulier, SO ajoute des tags comme [tag:tag] et des liens de raccourci comme [meta] (pas dans les docs cependant), spoilers :

 ! Ceci est un spoiler

>! This is a spoiler

et dévotions linguistiques personnalisées

<!-- language: java -->
```
This text is formatted as if it were Java code
```

Démarque Doxygen #

Doxygen est un outil de documentation de code couramment utilisé (pour les langages tels que C++, C# et Java) qui prend également en charge l’utilisation de Markdown. En plus de la syntaxe Markdown standard, il existe un certain nombre d’éléments spécifiques à Doxygen.

Les fonctionnalités principales sont l’utilisation des balises @ref pour les références, et les éléments @page, @section/@subsection et @anchor auxquels celles-ci peuvent faire référence.

@ref <nom> [“texte”] #

Cet élément crée un lien (c’est-à-dire une référence) vers une section, une sous-section, une page ou une ancre nommée qui a été définie ailleurs dans la documentation. (voir référence Doxygen)

Le premier paramètre (“nom”) doit correspondre au nom de la section, de la sous-section, de la page ou de l’ancre vers laquelle vous souhaitez créer un lien.

Le deuxième paramètre facultatif ("texte") doit être encapsulé entre guillemets doubles et définira comment le lien de référence apparaîtra sur la page. S’il n’est pas utilisé, le lien apparaîtra comme le titre utilisé dans la référence.

@section <nom-section> (titre de la section) #

Cet élément définit un nom de section. Il est visuellement équivalent à un élément # dans Markdown, mais il définira également une référence qui peut être liée à d’autres sections de votre documentation. (voir référence Doxygen)

Le premier paramètre section-name définit le nom de référence qui peut être utilisé par l’élément @ref. Celui-ci ne peut pas contenir d’espaces.

Le deuxième paramètre “titre de la section” est une chaîne de mots (qui peuvent être séparés par des espaces) qui définit l’apparence du titre de la section sur votre page.

Exemple #

@section Intro Introduction

This is some text in my introduction.

@section Body Body Paragraph

This is some text in my body, where I refer to the @ref Intro.

Production #

Introduction #

Ceci est un texte dans mon introduction.

Corps du paragraphe #

Ceci est un texte dans mon corps, où je me réfère à l’introduction.

NB : Le mot Introduction ci-dessus apparaîtra sous la forme d’un lien qui sautera à l’en-tête Introduction.

Markdown aromatisé GitHub #

[GitHub Flavored Markdown][1] (parfois abrégé en GFM) facilite le travail avec le démarquage sur [GitHub.com][2].

Les fonctionnalités clés de GFM incluent :

  • indentation de code
  • prise en charge de la liste des tâches
  • référencement facile des problèmes GitHub
  • détection automatique du nom d’utilisateur GitHub et SHA
  • détection automatique des URL
  • prise en charge des émoticônes
