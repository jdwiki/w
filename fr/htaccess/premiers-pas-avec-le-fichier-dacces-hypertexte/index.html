Configuration de .htaccess #

Les fichiers .htaccess (ou “fichiers de configuration distribués”) fournissent un moyen pour effectuer des modifications de configuration par répertoire. Un fichier, contenant une ou plusieurs directives de configuration, est placé dans un répertoire de documents particulier, et les directives s’y appliquent répertoire et tous ses sous-répertoires.

Un fichier .htaccess contrôle la manière dont Apache interagit avec votre site. Il est utilisé pour modifier les demandes et modifier le comportement par défaut sans avoir à modifier les fichiers de configuration du serveur principal.

Configurer .htaccess est aussi simple que d’ouvrir un bloc-notes et de l’enregistrer sous .htaccess. Généralement, ce fichier sera placé dans le répertoire “racine” des fichiers de votre site Web, mais vous pouvez l’utiliser dans plusieurs répertoires différents. Ceci est particulièrement utile si vous cherchez à protéger par mot de passe des répertoires spécifiques.

Activer .htaccess #

Parfois même une seule erreur dans votre fichier httpd.conf ou .htacces entraînera un effondrement temporaire du serveur, et les utilisateurs verront 500 - Page Erreur interne du serveur. Donc, assurez-vous de toujours faire une sauvegarde de vos fichiers httpd.conf et .htaccess avant de faire une modification.

<Directory "/var/www">
    AllowOverride All
</Directory>

Les fichiers .htaccess sont normalement activés par défaut. Ceci est contrôlé par la directive AllowOverride dans le fichier httpd.conf. Cette directive ne peut être placée qu’à l’intérieur d’une section <Directory>.

En plus de “Tous”, il existe de nombreuses autres valeurs qui limitent la configuration de certains contextes uniquement. Certains d’entre eux sont:

  • Aucun - Désactivez complètement .htaccess.

  • AuthConfig - Directives d’autorisation telles que celles traitant de l’authentification de base.

  • FileInfo - Directives traitant de la définition des en-têtes, des documents d’erreur, des cookies, de la réécriture d’URL, etc.

  • ** Index ** - Personnalisations de la liste des répertoires par défaut.

  • Limiter : contrôlez l’accès aux pages de différentes manières.

  • Options - Accès similaire aux index mais inclut encore plus de valeurs telles que ExecCGI, FollowSymLinks, Include et plus encore.

    Only allow .htaccess files to override Authorization and Indexes #

    AllowOverride AuthConfig Indexes

Pages d’erreur personnalisées #

.htaccess peut être utilisé pour définir des pages d’erreur personnalisées qui correspondent au thème de votre site Web au lieu de voir une page d’erreur blanche avec un techno-babillage noir lorsque les utilisateurs se retrouvent sur une page avec un code de réponse du serveur d’erreur. La page d’erreur peut être n’importe quel fichier analysable par le navigateur, y compris (mais sans s’y limiter) .html, .php, .asp, .txt, .xml.

Exemples pour presque tous les codes de réponse d’erreur courants :

#Client Errors

ErrorDocument 400 /mycool400page.html    # Bad Request
ErrorDocument 401 /mycool401page.html    # Unauthorized
ErrorDocument 402 /mycool402page.html    # Payment Required
ErrorDocument 403 /mycool403page.html    # Forbidden
ErrorDocument 404 /mycool404page.html    # Page Not Found

#Server Errors

ErrorDocument 500 /mycool500page.html    # Internal Server Error
ErrorDocument 501 /mycool501page.html    # Not Implemented
ErrorDocument 502 /mycool502page.html    # Bad Gateway
ErrorDocument 503 /mycool503page.html    # Service Unavailable
ErrorDocument 504 /mycool504page.html    # Gateway Timeout
ErrorDocument 505 /mycool505page.html    # Internal Server Error

Il est toujours recommandé d’inclure des documents d’erreur pour les réponses d’erreur les plus courantes, 400, 403, 404 et 500, car ces erreurs peuvent se produire sur tous les navigateurs.

l’erreur 500 est l’une des erreurs les plus notoires car elle se produit si quelque chose échoue lors du chargement de la page à envoyer, le plus souvent des échecs de prétraitement html du serveur provenant de choses comme PHP, ASP et d’autres préprocesseurs html. Lors des tests, il est recommandé de définir la page 500 pour afficher l’erreur qui s’est produite, plutôt qu’une page d’erreur 500 non spécifique.

Pour permettre à la page d’erreur 500 d’écrire une erreur spécifique, consultez l’une des options suivantes en fonction du préprocesseur HTML que vous utilisez : [php][1] [asp][2]

[1] : http://stackoverflow.com/questions/2687730/how-can-i-make-php-display-the-error-instead-of-giving-me-500-internal-server-er [2] : http://stackoverflow.com/questions/2640526/detailed-500-error-message-asp-iis-7-5

Définition du fuseau horaire du serveur #

Il existe de nombreux fuseaux horaires dans le monde, il est important de vous assurer que votre serveur est réglé sur le bon. Cela se fait dans .htaccess en utilisant :

SetEnv TZ America/Indianapolis

Quelques exemples d’autres fuseaux horaires possibles :

America/Los_Angeles
America/Los_Angeles - Pacific Time 
Pacific/Honolulu - Hawaii

Assurez-vous simplement d’utiliser SetEnv devant le fuseau horaire que vous avez sélectionné.
