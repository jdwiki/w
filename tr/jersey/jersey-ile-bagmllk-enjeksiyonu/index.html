Jersey’nin HK2’sini kullanarak Temel Bağımlılık Enjeksiyonu #

Jersey (2), bağımlılık ekleme (DI) sistemi olarak HK2 kullanır. Diğer enjeksiyon sistemlerini de kullanabiliriz ancak alt yapısı HK2 ile oluşturulmuş ve bizim uygulamalarımızda da kullanmamıza izin veriyor.

Jersey ile basit bağımlılık enjeksiyonunu ayarlamak sadece birkaç satır kod alır. Örneğin, kaynaklarımıza enjekte etmek istediğimiz bir hizmetimiz olduğunu varsayalım.

public class GreetingService {
    public String getGreeting(String name) {
        return "Hello " + name + "!";
    }
}

Ve bu hizmeti bir Jersey kaynağına enjekte etmek istiyoruz.

@Path("greeting")
public class GreetingResource {

    @Inject
    public GreetingService greetingService;

    @GET
    public String get(@QueryParam("name") String name) {
        return this.greetingService.getGreeting(name);
    }
}

Enjeksiyonun çalışması için tek ihtiyacımız olan basit bir konfigürasyon

@ApplicationPath("/api")
public class AppConfig extends ResourceConfig {
    public AppConfig() {
        register(GreetingResource.class);
        register(new AbstractBinder() {
            @Override
            protected void configure() {
                bindAsContract(GreetingService.class);
            }
        });
    }
}

Burada GreetingServicei enjeksiyon sistemine bağlamak ve aynı sınıf tarafından enjekte edilebilir olarak tanıtmak istediğimizi söylüyoruz. Son ifadenin anlamı, onu yalnızca ‘GreetingService’ olarak enjekte edebileceğimiz ve (muhtemelen açıkçası) başka bir sınıf tarafından değil. Daha sonra göreceğiniz gibi, bunu değiştirmek mümkündür.

Bu kadar. Tüm ihtiyacın olan bu. Bu “ResourceConfig” yapılandırmasına aşina değilseniz (belki web.xml kullanıyorsunuz), lütfen SO Docs’ta [JAX-RS’yi Jersey’de Yapılandırma][2] konusuna bakın.

Not:** Yukarıdaki enjeksiyon, hizmetin kaynak alanına enjekte edildiği alan enjeksiyonudur. Başka bir enjeksiyon türü, hizmetin yapıcıya enjekte edildiği yapıcı enjeksiyonudur.

private final GreetingService greetingService;

@Inject   
public GreetingResource(GreetingService greetingService) {
     this.greetingService = greetingService;
}

Bu, kaynağın birim testini kolaylaştırdığından, muhtemelen saha enjeksiyonunun aksine tercih edilen yoldur. Yapıcı enjeksiyonu herhangi bir farklı konfigürasyon gerektirmez.

Tamam, şimdi diyelim ki bir sınıf yerine ‘GreetingService’ bir arabirimdir ve bunun bir uygulamasına sahibiz (ki bu çok yaygındır). Bunu yapılandırmak için yukarıdaki “configure” yönteminde aşağıdaki sözdizimini kullanırız.

@Override
protected void configure() {
    bind(NiceGreetingService.class).to(GreetingService.class);
}

Bu “NiceGreetingService"i bağla ve “GreetingService” olarak tanıt” olarak okunur. Bu, sözleşmenin reklamını “NiceGreetingService” olarak değil, “GreetingService” olarak tanıttığımız için yukarıdaki “GreetingResource” içinde aynı kodu kullanabileceğimiz anlamına gelir. Ancak asıl uygulama, enjekte edildiğinde ‘NiceGreetingService’ olacaktır.

Şimdi kapsam ne olacak. Herhangi bir enjeksiyon çerçevesi ile çalıştıysanız, hizmetin ömrünü belirleyen kapsam kavramıyla karşılaşmış olacaksınız. Hizmetin yalnızca isteğin süresi boyunca geçerli olduğu bir “Talep Kapsamı"nı duymuş olabilirsiniz. Veya hizmetin yalnızca bir örneğinin bulunduğu bir “Singleton Kapsamı”. Bu kapsamları aşağıdaki sözdizimini kullanarak da yapılandırabiliriz.

@Override
protected void configure() {
    bind(NiceGreetingService.class)
            .to(GreetingService.class)
            .in(RequestScoped.class);
}

Varsayılan kapsam ‘PerLookup’ olup, bu hizmet her istendiğinde yeni bir hizmetin oluşturulacağı anlamına gelir. Yukarıdaki örnekte, ‘RequestScoped’ kullanılarak tek bir istek için yeni bir hizmet oluşturulacaktır. Kaç yere enjekte etmeye çalıştığımıza bağlı olarak, bu “PerLookup” ile aynı olabilir veya olmayabilir. Onu bir filtreye ve bir kaynağa enjekte etmeye çalışıyor olabiliriz. Bu “PerLookup” olsaydı, her istek için iki örnek oluşturulur. Bu durumda sadece bir tane istiyoruz.

Mevcut diğer iki kapsam “Singleton” (yalnızca bir örnek oluşturuldu) ve “Immediate” (“Singleton” gibi) ancak başlangıçta oluşturulur (oysa “Singleton” ile ilk isteğe kadar oluşturulmaz).

Bağlayıcı sınıfların yanı sıra, sadece bir örnek de kullanabiliriz. Bu bize varsayılan bir singleton verir, bu yüzden ‘in’ sözdizimini kullanmamız gerekmez.

@Override
protected void configure() {
    bind(new NiceGreetingService())
            .to(GreetingService.class);
}

Ya bazı karmaşık oluşturma mantığımız varsa veya hizmet için bazı istek bağlam bilgilerine ihtiyacımız varsa. Bu durumda Fabrikalar var. Jersey kaynaklarımıza enjekte edebileceğimiz çoğu şeyi, bir ‘Fabrika’ya da enjekte edebiliriz. örneğin al

public class GreetingServiceFactory implements Factory<GreetingService> {
    
    @Context
    UriInfo uriInfo;
    
    @Override
    public GreetingService provide() {
        return new GreetingService(
                uriInfo.getQueryParameters().getFirst("name"));
    }
    
    @Override
    public void dispose(GreetingService service) {
        /* noop */
    }
}

Burada ‘UriInfo’dan istek bilgisi alan bir fabrikamız var, bu durumda bir sorgu parametreleri ve ondan ‘GreetingService’i oluşturuyoruz. Yapılandırmak için aşağıdaki sözdizimini kullanıyoruz

@Override
protected void configure() {
    bindFactory(GreetingServiceFactory.class)
            .to(GreetingService.class)
            .in(RequestScoped.class);
}

Bu kadar. Bunlar sadece temel bilgiler. HK ve Jersey’nin yapacak daha çok işi var.
