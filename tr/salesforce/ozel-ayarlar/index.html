giriiş #

Kendilerine dayalı kayıtları olan özel nesnelerin aksine, özel ayarlar, kuruluşunuz genelinde özel veri kümelerini kullanmanıza veya özel ölçütlere göre belirli kullanıcıları veya profilleri ayırt etmenize olanak tanır. Bu, örneğin yöneticilerin hiyerarşi özel ayarlarını düzenleyerek tek kullanıcılar veya profiller için İş Akışı / Doğrulama Kurallarını tüm kuruluş için kapatmak zorunda kalmadan devre dışı bırakabilecekleri anlamına gelir (yukarıdaki İş Akışını / Doğrulama Kurallarını Devre Dışı Bırakmak İçin Hiyerarşi Özel Ayarlarını Kullanma örneğine bakın) ).

Doğrulama kurallarının genellikle şu durumlarda geçici olarak devre dışı bırakılması gerekir:

  • Kod, bir doğrulama kuralı etkinleştirilmeden önce en son düzenlenen ve bu nedenle yeni kuralın kriterlerini karşılamayan eski kayıtları güncelliyor.
  • Kod, bir doğrulama kuralının kriterlerinin gerektirdiği değerler olmadan yeni kayıtlar ekliyor.

İş akışı kurallarının genellikle şu durumlarda geçici olarak devre dışı bırakılması gerekir:

  • Kayıtta yaptığınız değişikliklerin üzerine yazacak veya müdahale edecek bir E-posta Uyarısını veya Alan Güncellemesini tetiklerler.

Özel bir ayarın kullanılması, yöneticilere kod üzerinde bazı bildirimsel kontrol sağlar, bu nedenle birçok kullanım durumundan biri, kullanıldığında tetikleyicileri devre dışı bırakmak için kod dağıtmayı gereksiz hale getirebilmeleridir (yukarıdaki Apex Kodunu Devre Dışı Bırakmak için Hiyerarşi Özel Ayarlarını Kullanma örneğine bakın) ).

Geliştiriciler için önemli bir fayda, özel ayarın verilerinin, veritabanına tekrarlanan sorgular maliyeti olmadan verimli erişim sağlayan uygulama önbelleğinde açığa çıkmasıdır. Bu veriler daha sonra formül alanları, doğrulama kuralları, akışlar, Apex ve SOAP API tarafından kullanılabilir - bkz. Salesforce belgeleri.

Özel ayarlar için sınırlar ve hususlar burada belgelenmiştir.

Özel Ayarları Listele #

Liste Özel Ayarlarını oluşturmak da mümkündür, yaygın kullanım örnekleri arasında iki harfli durum kısaltmalarının, uluslararası arama öneklerinin ve ürünler için katalog numaralarının saklanması yer alır. Ancak Salesforce artık Özel Ayarları Listelemek yerine Özel Meta Veri Türlerinin kullanımını teşvik ediyor.

Yeni bir Özel Ayar oluşturmaya gittiğinizde aşağıdaki mesaj görüntülenecektir.

İpucu: Uygulama Yapılandırması için Özel Meta Veri Türlerini Kullan** Liste özel ayarlarını kullanmayı düşünüyorsanız, bunun yerine özel meta veri türlerini kullanmayı düşünün. Liste özel ayarlarından farklı olarak, paketleri veya Meta Veri API araçlarını kullanarak özel meta veri türlerinin kayıtlarını geçirebilirsiniz.

Özel Meta Veri Türlerinin, bu cevap bölümünde açıklandığı gibi Özel Ayarları Listele ile karşılaştırıldığında ek avantajları vardır. Ve CMD’lerin baş geliştiricisine göre “Steroitlerdeki özel ayarlardan çok özel meta veri türleri için planlanmış çok daha fazlası var.”

Özel Ayarlar Oluşturma ve Yönetme #

oluşturma #

Bir Özel Ayar oluşturmak için şuraya gidin:

Klasik Kurulum > Geliştirme > Özel Ayarlar > Yeni

Şimşek Kurulum > Özel Kod > Özel Ayarlar > Yeni

Ayarınızı oluşturun (Hiyerarşi ve Liste özel ayarları arasındaki farklar için bu belgenin ilerleyen kısımlarındaki Açıklamalara bakın). Ayarınızı yönetilen bir pakette dağıtmayı planlamıyorsanız, Görünürlük seçim listesini yok sayabilirsiniz.

Ayar alanlarınızı oluşturmak için Yeni düğmesine tıklayın ve özel alan oluşturmak için olağan süreci izleyin.

Yönetmek #

Alan(lar)ınızı oluşturduktan sonra, Yönet düğmesine tıklayarak ayarı yapılandırmaya başlayabilirsiniz.

Yeni bir görünüm oluşturursanız ve kendinize ayara bir bakışta kapsamlı bir genel bakış sağlamak için oluşturduğunuz tüm alanları dahil ederseniz ayarı yönetmek daha kolaydır. Kurulum Sahibi, ayarın geçerli olduğu kullanıcı veya profildir.

Ayarı kuruluş düzeyinde yönetmek için, Varsayılan Kuruluş Düzeyi Değeri başlığının (aşağıdaki kırmızı kutuda) üzerindeki Yeni düğmesini tıklayın.

Ayarı kullanıcı veya profil düzeyinde yönetmek için aşağıdaki mavi kutudaki Yeni düğmesini tıklayın.

Alan işaretlendiğinde, çalışan kullanıcı veya bu örnekte, profili için doğrulama kuralı devre dışı bırakılır -

Kural, tüm Salesforce kuruluşu için de devre dışı bırakılabilir -

Doğrulama Kuralı #

AND(
  /* the below is the reference to the Val_Rule_Cntrlr__c custom setting's checkbox field All_Opportunity_Disabled__c
  */
  $Setup.Val_Rule_Cntrlr__c.All_Opportunity_Disabled__c = FALSE,

  /* the below is the remainder of the validation rule's formula
  */
  CloseDate < TODAY()
)

Yukarıdaki kuralda, kuralın tetiklenebilmesi için her iki ölçüt parçasının da “DOĞRU” olarak değerlendirilmesi gerekir.

Çalışan kullanıcının profili Sistem Yöneticisi olduğunda ‘All_Opportunity_Disabled__c’ onay kutusu ‘DOĞRU’ olarak değerlendirileceğinden, kural ‘YANLIŞ’ olarak değerlendirilecektir.

Apex Kodunu Devre Dışı Bırakmak İçin Hiyerarşi Özel Ayarlarını Kullanma #

Açıklama #

Bu örnekte, eklenmek veya güncellenmek üzere olan bir Fırsatın Kapanış Tarihini 10 gün ileri bir tarihe değiştirmek için basit bir Tetikleyici oluşturulmuştur.

Apex Controller özel ayarının onay kutusu alanı, kodun kullanıcı / profil / kuruluş düzeyinde devre dışı bırakılmasını sağlar.

Apeks Sınıfı #

trigger CloseDateUpdate on Opportunity (before insert, before update) {
    
    Id userId;
    Apx_Cntrlr__c userApexController;
    Boolean userSetting;
    
    userId = userinfo.getUserId();
    userApexController = Apx_Cntrlr__c.getInstance(userId);
    userSetting = userApexController.Close_Date_Update_Disabled__c;

    if (userSetting == false) {
        for(Opportunity opp : Trigger.new) {
            opp.CloseDate = date.today().addDays(10);
        }
    }
    
}

Ünite testi #

@isTest
public class CloseDateUpdateTest {
    
    @testSetup
    static void dataSetup() {
        
        Profile p = [SELECT Id FROM Profile WHERE Name = 'System Administrator' LIMIT 1];
        
        User u = new User(LastName = 'Test',Alias = 't1',Email = 'example@gmail.com',Username = 'sotest@gmail.com',ProfileId = p.Id,TimeZoneSidKey = 'America/Denver',LocaleSidKey = 'en_US',EmailEncodingKey = 'UTF-8',LanguageLocaleKey = 'en_US');
        insert u;
    }
    
    static testMethod void testCloseDateUpdateEnabled() {
        
        User u = [SELECT Id FROM User WHERE Username = 'sotest@gmail.com'];
        // set the custom setting field to FALSE so that the trigger is not deactivated
        Apx_Cntrlr__c apexController = new Apx_Cntrlr__c(SetupOwnerId = u.Id,Close_Date_Update_Disabled__c = false);
        upsert apexController;
        
        Opportunity[] opportunities1 = new Opportunity[]{};
        
        test.startTest();
        system.runAs(u){
                
                for(integer i = 0; i < 200; i++) {
                    opportunities1.add(new Opportunity(
                        Name          = 'Test Opp ' + i,
                        OwnerId       = u.Id,
                        StageName     = 'Prospecting',
                        CloseDate     = date.today().addDays(1),
                        Amount        = 100));
                }
            insert opportunities1;
        }
        test.stopTest();
        
        List<Opportunity> opportunities2 = [SELECT CloseDate FROM Opportunity];
        
        for(Opportunity o : opportunities2){
            system.assertEquals(date.today().addDays(10), o.closeDate, 'CloseDateUpdate trigger should have changed the Opportunity close date as it was not disabled by the apexController custom setting');
        }
    }
    
    static testMethod void testCloseDateUpdateDisabled() {
        
        User u = [SELECT Id FROM User WHERE Username = 'sotest@gmail.com'];
        // set the custom setting field to TRUE to deactivate the trigger
        Apx_Cntrlr__c apexController = new Apx_Cntrlr__c(SetupOwnerId = u.Id,Close_Date_Update_Disabled__c = true);
        upsert apexController;
        
        Opportunity[] opportunities1 = new Opportunity[]{};
            
        test.startTest();
        system.runAs(u){
            
            for(integer i = 0; i < 200; i++) {
                opportunities1.add(new Opportunity(
                    Name          = 'Test Opp ' + i,
                    OwnerId       = u.Id,
                    StageName     = 'Prospecting',
                    CloseDate     = date.today().addDays(1),
                    Amount        = 100));
            }
            insert opportunities1;
        }
        test.stopTest();
        
        List<Opportunity> opportunities2 = [SELECT CloseDate FROM Opportunity];
        
        for(Opportunity o : opportunities2){
            system.assertEquals(date.today().addDays(1), o.closeDate, 'CloseDateUpdate trigger should not have changed the Opportunity close date as it was disabled by the apexController custom setting');
        }
    }
        
}

Apex Kodunda Hiyerarşi Özel Ayarlarını Güncelleme #

Doğrulama veya iş akışı kurallarını kapatmak için, kodunuzun yürütülmesi sırasında özel ayarlarınızı güncellemek isteyebilirsiniz.

Aşağıdaki kodda, Kapanış Tarihi geçerli tarihten 6 gün veya daha az olan Fırsatların Kapanış Tarihini güncelleyecek ve tarihi 20 gün olarak değiştirecek bir Planlanabilir Apex Sınıfı oluşturdum. gelecek.

Kriterlerime uyan Fırsatları güncellememi engelleyecek tüm doğrulama kurallarını devre dışı bırakmak için Özel Ayar Val_Rule_Cntrlr__c’yi kullanacağım.

global class Scheduled_OppCloseDateUpdate implements Schedulable {

    global void execute(SchedulableContext SC) {
        updOpportunityCloseDates();
    }

    global void updOpportunityCloseDates() {
        
        Id userId;
        Val_Rule_Cntrlr__c setting;
        Boolean validationRulesAlreadyDisabled;
        List<Opportunity> processedOpps = new List<Opportunity>();
        Date d;
        
        // get running user's Id
        userId = userinfo.getUserId();
        // retrieve Custom Setting status, for running user
        setting = Val_Rule_Cntrlr__c.getInstance(userId);
        
        // if the setting field is false, update it to disable validation rules
        if (setting.All_Opportunity_Disabled__c == false) {
            setting.All_Opportunity_Disabled__c = true;
            upsert setting;
        }
        // if the setting field was already true, there's no need to disable it
        // but it shouldn't be switched to false by this class once the process has been completed
        else {
            validationRulesAlreadyDisabled = true;
        }
        
        // execute code to manage business process
   
        d = system.today().addDays(6);
        
        for(Opportunity o : [SELECT Id, CloseDate
                               FROM Opportunity
                              WHERE CloseDate <= :d
                                // class only updates open Opportunities
                                AND Probability > 0 AND Probability < 100])
        {
            o.CloseDate = System.today().addDays(20);
            processedOpps.add(o);
        }
        
        if (processedOpps.size() > 0) {
            update processedOpps;
        }
        
        // reactivate validation rules
        if (validationRulesAlreadyDisabled == false) {
            setting.All_Opportunity_Disabled__c = false;
            upsert setting;
        }
        
    }
    
}

Sınıfımdaki özel ayarlarımda yapılan değişikliklerle doğrulama kurallarımın devre dışı bırakıldığından emin olmak için, Trigger_Validation_Rule__c (kullanıcılar tarafından görülmeyecek veya sayfa düzenlerine eklenmeyecek) bir onay kutusu alanı ve bununla bir doğrulama kuralı oluşturdum. kriterler:

AND(
  $Setup.Val_Rule_Cntrlr__c.All_Opportunity_Disabled__c = FALSE,
  Trigger_Validation_Rule__c = TRUE,

  /* allow the above criteria to be met while inserting the Opportunities, without triggering the rule, in the @testSetup portion of the test */
  NOT(ISNEW())
)

Daha sonra, özel ayar alanı kodum tarafından düzenlenmezse, kural kriterlerinin karşılanması için Fırsatlarımı oluştururken onay kutusu alanını ’true’ olarak ayarladım.

@isTest
private class WE_ScheduledCloseDateUpdateTest {

    @testSetup
    static void dataSetup() {

        Profile p = [SELECT Id FROM Profile WHERE Name = 'System Administrator' LIMIT 1];
        
        User u = new User(LastName = 'Test',Alias = 't1',Email = 'example@gmail.com',Username = 'sotest@gmail.com',ProfileId = p.Id,TimeZoneSidKey = 'America/Denver',LocaleSidKey = 'en_US',EmailEncodingKey = 'UTF-8',LanguageLocaleKey = 'en_US');
        insert u;

        Val_Rule_Cntrlr__c valRuleCntrlr = new Val_Rule_Cntrlr__c(SetupOwnerId = u.Id,All_Opportunity_Disabled__c = false);
        upsert valRuleCntrlr;
            
        List<Opportunity> testOpps = new List<Opportunity>();
            
        // create the Opportunities that will be updated by the class
        for(integer i = 0; i < 200; i++) {
            testOpps.add(new Opportunity(
                Name          = 'Test Opp Update' + i,
                OwnerId       = u.Id,
                StageName     = 'Prospecting',
                CloseDate     = date.today().addDays(1),
                Amount        = 100,
                // set checkbox field to true, to trigger validation rules if they've not been deactivated by class
                Trigger_Validation_Rule__c = true));
        }
        // create the Opportunities that won't be updated by the class
        for(integer i = 0; i < 200; i++) {
            testOpps.add(new Opportunity(
                Name          = 'Test Opp Skip' + i,
                OwnerId       = u.Id,
                StageName     = 'Prospecting',
                CloseDate     = date.today().addDays(15),
                Amount        = 100,
                Trigger_Validation_Rule__c = true));
        }           
        insert testOpps;

    }
    
    // code required to test a scheduled class, see https://developer.salesforce.com/docs/atlas.en-us.apexcode.meta/apexcode/apex_scheduler.htm for more details
    public static String CRON_EXP = '0 0 0 15 3 ? 2022';

    static testmethod void testCloseDateUpdates() {

        // execute scheduled class
        
        Test.startTest();
        
        String jobId = System.schedule('ScheduleApexClassTest',
                                       CRON_EXP,
                                       new Scheduled_OppCloseDateUpdate());
        
        CronTrigger ct = [SELECT Id, CronExpression, TimesTriggered, NextFireTime
                            FROM CronTrigger
                           WHERE id = :jobId];
        
        System.assertEquals(CRON_EXP, ct.CronExpression);
        System.assertEquals(0, ct.TimesTriggered);
        System.assertEquals('2022-03-15 00:00:00', String.valueOf(ct.NextFireTime));
        
        Test.stopTest();

        // test results
        
        Integer updateCount = 0;
        Integer skipCount   = 0;

        List <Opportunity> opportunitys = [SELECT Id, Name, CloseDate FROM Opportunity];

        for(Opportunity o : opportunitys) {
            if (o.Name.contains('Update') &&
                updateCount == 0)
            {
                System.assertEquals(date.today().addDays(20), o.CloseDate, 'Opportunity\'s Close Date should have been updated as it was less than 7 days away');
                updateCount = 1;
            }
            if (o.Name.contains('Skip') &&
                skipCount == 0)
            {
                System.assertEquals(date.today().addDays(15), o.CloseDate, 'Opportunity should not have been updated as it\'s Close Date is more than 7 days away');
                skipCount = 1;
            }
        }
        // check that both lists of Opportunities have been tested
        System.assertEquals(2, updateCount + skipCount, 'Count should be 2 once all assertions have been executed');
    
    }
    
    // check that the class does not change the custom setting's field to false, if it was true before class was executed
    static testmethod void testSettingUpdates() {
        
        User u = [SELECT Id FROM User WHERE UserName = 'sotest@gmail.com'];
        
        // switch the custom setting field to true before the scheduled job executes
        Val_Rule_Cntrlr__c setting;
        setting = Val_Rule_Cntrlr__c.getInstance(u.Id);
        setting.All_Opportunity_Disabled__c = true;
        upsert setting;
        
        System.runAs(u) {
            
            Test.startTest();
            
            String jobId = System.schedule('ScheduleApexClassTest',
                                           CRON_EXP,
                                           new Scheduled_OppCloseDateUpdate());
            
            CronTrigger ct = [SELECT Id, CronExpression, TimesTriggered, NextFireTime
                                FROM CronTrigger
                               WHERE id = :jobId];
            
            System.assertEquals(CRON_EXP, ct.CronExpression);
            System.assertEquals(0, ct.TimesTriggered);
            System.assertEquals('2022-03-15 00:00:00', String.valueOf(ct.NextFireTime));
            
            Test.stopTest();
        }
        setting = Val_Rule_Cntrlr__c.getInstance(u.Id);
        
        // check that the class did not change the All_Opportunity_Disabled__c field to false
        System.assertEquals(true, setting.All_Opportunity_Disabled__c);
    }

}
