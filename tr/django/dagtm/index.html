Basit uzaktan kurulum fabfile.py #

Fabric, uygulama dağıtımı veya sistem yönetimi görevleri için SSH kullanımını kolaylaştıran bir Python (2.5-2.7) kitaplığı ve komut satırı aracıdır. Komut satırı aracılığıyla isteğe bağlı Python işlevlerini yürütmenize olanak tanır.

Kumaşı “pip kurulum kumaşı” ile kurun Kök dizininizde fabfile.py oluşturun:

#myproject/fabfile.py    
from fabric.api import *

@task
def dev():
    # details of development server
    env.user = # your ssh user
    env.password = #your ssh password
    env.hosts = # your ssh hosts (list instance, with comma-separated hosts)
    env.key_filename = # pass to ssh key for github in your local keyfile

@task
def release():
    # details of release server
    env.user = # your ssh user
    env.password = #your ssh password
    env.hosts = # your ssh hosts (list instance, with comma-separated hosts)
    env.key_filename = # pass to ssh key for github in your local keyfile

@task
def run():
    with cd('path/to/your_project/'):
        with prefix('source ../env/bin/activate'): 
        # activate venv, suppose it appear in one level higher
            # pass commands one by one
            run('git pull')
            run('pip install -r requirements.txt')
            run('python manage.py migrate --noinput')
            run('python manage.py collectstatic --noinput')
            run('touch reload.txt')

Dosyayı çalıştırmak için “fab” komutunu kullanmanız yeterlidir:

$ fab dev run  # for release server, `fab release run`

Not: github için ssh anahtarlarını yapılandıramazsınız ve fabfile çalışırken, tuşlarla aynı şekilde oturum açma ve parolayı manuel olarak yazamazsınız.

Gunicorn ile Django uygulamasını çalıştırma #

  1. Gunicorn’u yükleyin

    pip install gunicorn

  2. Django proje klasöründen (manage.py’nin bulunduğu klasör), mevcut Django projesini gunicorn ile çalıştırmak için aşağıdaki komutu çalıştırın.

    gunicorn [projectname].wsgi:application -b 127.0.0.1:[port number]

    You can use the --env option to set the path to load the settings

    gunicorn --env DJANGO_SETTINGS_MODULE=[projectname].settings [projectname].wsgi

    or run as daemon process using -D option

  3. Gunicorn’un başarılı bir şekilde başlatılmasının ardından konsolda aşağıdaki satırlar görünecektir.

    Starting gunicorn 19.5.0

    Listening at: http://127.0.0.1:[port number] ([pid])

    .... (other additional information about gunicorn server)

Heroku Django Başlangıç ​​Şablonunu Kullanma. #

Django web sitenizi Heroku’da barındırmayı planlıyorsanız, projenize Heroku Django Başlangıç ​​Şablonunu kullanarak başlayabilirsiniz:

django-admin.py startproject --template=https://github.com/heroku/heroku-django-template/archive/master.zip --name=Procfile YourProjectName

Statik Dosyalar, Veritabanı Ayarları, Gunicorn, vb. için Üretime hazır yapılandırmaya ve WhiteNoise aracılığıyla Django’nun statik dosya sunma işlevinde Geliştirmelere sahiptir. Bu size zaman kazandıracak, Heroku’da barındırmak için Tamamen Hazır, Sadece web sitenizi bu şablonun üstüne inşa edin

Bu şablonu Heroku’da dağıtmak için:

git init
git add -A
git commit -m "Initial commit"

heroku create
git push heroku master

heroku run python manage.py migrate

Bu kadar!

Django dağıtım talimatları. Linux’ta Nginx + Gunicorn + Süpervizör (Ubuntu) #

Üç temel araç.

  1. nginx - ücretsiz, açık kaynaklı, yüksek performanslı HTTP sunucusu ve yüksek performanslı ters proxy;
  2. gunicorn - ‘Green Unicorn’ UNIX için bir Python WSGI HTTP Sunucusudur (sunucunuzu yönetmek için gereklidir);
  3. süpervizör - kullanıcılarının UNIX benzeri işletim sistemlerinde bir dizi işlemi izlemesine ve kontrol etmesine izin veren bir istemci/sunucu sistemi. Uygulama veya sistem çöktüğünde kullanılır, Django / kereviz / kereviz kameranızı vb. yeniden başlatır;

Bunu basitleştirmek için, uygulamanızın şu dizinde bulunduğunu varsayalım: /home/root/app/src/ ve root kullanıcısını kullanacağız (ancak uygulamanız için ayrı bir kullanıcı oluşturmalısınız). Ayrıca sanal ortamımız /home/root/app/env/ yolunda bulunacaktır.

NGINX #

nginx ile başlayalım. Nginx zaten makinede değilse, sudo apt-get install nginx ile kurun. Daha sonra /etc/nginx/sites-enabled/yourapp.conf nginx dizininizde yeni bir yapılandırma dosyası oluşturmanız gerekir. default.conf adında bir dosya varsa - kaldırın.

Soket dosyası kullanarak hizmetinizi çalıştırmayı deneyecek bir nginx conf dosyasına aşağıdaki kod; Daha sonra bir gunicorn konfigürasyonu olacak. Soket dosyası burada nginx ve gunicorn arasında iletişim kurmak için kullanılır. Portlar kullanılarak da yapılabilir.

# your application name; can be whatever you want
upstream yourappname {
    server        unix:/home/root/app/src/gunicorn.sock fail_timeout=0;
}

server {
    # root folder of your application
    root        /home/root/app/src/;
 
    listen        80;
    # server name, your main domain, all subdomains and specific subdomains
    server_name   yourdomain.com *.yourdomain.com somesubdomain.yourdomain.com
 
    charset       utf-8;

    client_max_body_size                        100m;
 
    # place where logs will be stored;
    # folder and files have to be already located there, nginx will not create
    access_log        /home/root/app/src/logs/nginx-access.log; 
    error_log         /home/root/app/src/logs/nginx-error.log;
    
    # this is where your app is served (gunicorn upstream above)
    location / {
        uwsgi_pass  yourappname;
        include     uwsgi_params; 
    }

    # static files folder, I assume they will be used
    location /static/ {
        alias         /home/root/app/src/static/;
    }
 
    # media files folder
    location /media/ {
        alias         /home/root/app/src/media/;
    }

}

GUNICORN #

Şimdi Django uygulamasını sunucuda çalıştırmaktan sorumlu olacak GUNICORN betiğimiz. İlk iş sanal ortamda pip install gunicorn ile gunicorn kurulumu yapmaktır.

#!/bin/bash

ME="root"
DJANGODIR=/home/root/app/src # django app dir
SOCKFILE=/home/root/app/src/gunicorn.sock # your sock file - do not create it manually
USER=root
GROUP=webapps
NUM_WORKERS=3 
DJANGO_SETTINGS_MODULE=yourapp.yoursettings
DJANGO_WSGI_MODULE=yourapp.wsgi  
echo "Starting $NAME as `whoami`"
 
# Activate the virtual environment
cd $DJANGODIR

source /home/root/app/env/bin/activate
export DJANGO_SETTINGS_MODULE=$DJANGO_SETTINGS_MODULE
export PYTHONPATH=$DJANGODIR:$PYTHONPATH
 
# Create the run directory if it doesn't exist
RUNDIR=$(dirname $SOCKFILE)
test -d $RUNDIR || mkdir -p $RUNDIR
 
# Start your Django Gunicorn
# Programs meant to be run under supervisor should not daemonize themselves (do not use --daemon)
exec /home/root/app/env/bin/gunicorn ${DJANGO_WSGI_MODULE}:application \
  --name root \
  --workers $NUM_WORKERS \
  --user=$USER --group=$GROUP \
  --bind=unix:$SOCKFILE \
  --log-level=debug \
  --log-file=-

gunicorn start betiğini çalıştırabilmek için, yürütme modunun etkinleştirilmiş olması gerekir.

sudo chmod u+x /home/root/app/src/gunicorn_start

şimdi sadece ./gunicorn_start kullanarak gunicorn sunucunuzu başlatabileceksiniz.

SÜPERVİZÖR #

Başta da söylediğimiz gibi, bir süpervizör tarafından başarısız olduğunda uygulamamızın yeniden başlatılmasını istiyoruz. Süpervizör henüz sunucuda değilse, sudo apt-get yükleme süpervizörü ile kurun.

İlk başta, süpervizör yükleyin. Ardından ana dizininizde /etc/supervisor/conf.d/your_conf_file.conf bir .conf dosyası oluşturun.

yapılandırma dosyası içeriği:

[program:yourappname]
command = /home/root/app/src/gunicorn_start
user = root
stdout_logfile = /home/root/app/src/logs/gunicorn_supervisor.log
redirect_stderr = true

Kısa özet, [program:youappname] başlangıçta gereklidir, bu bizim tanımlayıcımız olacaktır. ayrıca ‘stdout_logfile’, hem erişim hem de hatalar için günlüklerin saklanacağı bir dosyadır.

Bunu yaptıktan sonra süpervizörümüze yeni yapılandırma dosyası eklediğimizi söylemeliyiz. Bunu yapmak için farklı Ubuntu sürümleri için farklı işlemler vardır.

‘Ubuntu sürüm 14.04 veya daha düşük’ için şu komutları çalıştırmanız yeterlidir:

sudo controllerctl reread -> süpervizör kataloğundaki tüm yapılandırma dosyalarını yeniden okur bunun yazdırılması gerekir: uygulama adınız: mevcut

sudo controllerctl update -> süpervizörü yeni eklenen yapılandırma dosyalarına günceller; uygulama adınız: eklenen süreç grubu yazdırılmalıdır

“Ubuntu 16.04” Çalıştırması için:

sudo service supervisor restart

ve uygulamanızın doğru çalışıp çalışmadığını kontrol etmek için sadece çalıştırın

sudo controllerctl durumu uygulama adınız

Bu şunu göstermelidir:

uygulama adınız RUNNING pid 18020, çalışma süresi 0:00:50

Bu prosedürün canlı gösterimini almak için bu video‘da gezinin.

Heroku ile dağıtma #

  1. Heroku Toolbelt‘i indirin.
  2. Django uygulamanızın kaynaklarının köküne gidin. tk’ye ihtiyacın olacak
  3. heroku create [app_name] yazın. Bir uygulama adı vermezseniz, Heroku sizin için rastgele bir tane oluşturacaktır. Uygulama URL’niz “http://[uygulama adı].herokuapp.com” olacaktır.
  4. ‘Procfile’ adında bir metin dosyası oluşturun. Sonuna bir uzantı koymayın.

web: <üretim sunucusunu başlatmak için bash komutu>

> Çalışan işleminiz varsa onu da ekleyebilirsiniz. Şu biçimde başka bir satır ekleyin: `işçi-adı: işçiyi başlatmak için <bash komutu>`

5. Bir gereksinim.txt ekleyin.
- Sanal bir ortam kullanıyorsanız, `pip frost > gereksinimleri.txt` komutunu çalıştırın.
- Aksi takdirde, [*sanal bir ortam edinin!*][2]. Ayrıca ihtiyacınız olan Python paketlerini manuel olarak da listeleyebilirsiniz, ancak bu, bu eğitimde ele alınmayacaktır.

6. **Dağıtım zamanı!**
 1. `git push heroku master`
 > Heroku needs a git repository or a dropbox folder to do deploys. You can alternatively set up automatic reloading from a GitHub repository at [`heroku.com`][3], but we won't cover that in this tutorial. <!-- Editors - you can help with this part! -->
 2. `heroku ps:scale web=1`
 > This scales the number of web "dynos" to one. You can learn more about dynos [here.][4]
 3. `heroku open` or navigate to `http://app-name.herokuapp.com`
 > **Tip:** `heroku open` opens the URL to your heroku app in the default browser.

7. **Eklentiler** ekleyin. Django uygulamanızı, Heroku'da "eklentiler" olarak sağlanan veritabanlarına bağlanacak şekilde yapılandırmanız gerekecek. Bu örnek bunu kapsamamaktadır, ancak Heroku'da veritabanlarının dağıtılmasıyla ilgili başka bir örnek boru hattındadır.

## apache/nginx kurmadan yerel olarak dağıtma
Statik içeriği sunmak için Apache/Nginx kullanmak için önerilen üretim dağıtım yöntemi. Bu nedenle, "DEBUG" yanlış statik olduğunda ve medya içeriği yüklenemediğinde. Ancak, uygulamamız için Apache/Nginx sunucusunu aşağıdakileri kullanarak kurmak zorunda kalmadan dağıtımdaki statik içeriği yükleyebiliriz:

 python manage.py runserver --insecure

Bu, yalnızca yerel dağıtım (ör. LAN) için tasarlanmıştır ve asla üretimde kullanılmamalıdır ve yalnızca "staticfiles" uygulaması projenizin "INSTALLED_APPS" ayarındaysa kullanılabilir.
