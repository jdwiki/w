JSON dosyası kullanarak gizli verileri gizleme #

Git veya SVN gibi bir VCS kullanırken, asla sürümlendirilmemesi gereken bazı gizli veriler vardır (depo genel veya özel olsun).

Bu veriler arasında ‘SECRET_KEY’ ayarını ve veritabanı şifresini bulacaksınız.

Bu ayarları sürüm kontrolünden gizlemek için yaygın bir uygulama, projenizin kökünde bir “secrets.json” dosyası oluşturmaktır (fikir için teşekkürler “Two Scoops of Django”):

{
    "SECRET_KEY": "N4HE:AMk:.Ader5354DR453TH8SHTQr",
    "DB_PASSWORD": "v3ry53cr3t"
}

Ve onu yoksayma listenize ekleyin (git için .gitignore):

*.py[co]
*.sw[po]
*~
/secrets.json

Ardından, “ayarlar” modülünüze aşağıdaki işlevi ekleyin:

import json
import os
from django.core.exceptions import ImproperlyConfigured

with open(os.path.join(BASE_DIR, 'secrets.json')) as secrets_file:
    secrets = json.load(secrets_file)

def get_secret(setting, secrets=secrets):
    """Get secret setting or fail with ImproperlyConfigured"""
    try:
        return secrets[setting]
    except KeyError:
        raise ImproperlyConfigured("Set the {} setting".format(setting))

Ardından ayarları şu şekilde doldurun:

SECRET_KEY = get_secret('SECRET_KEY')
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgres',
        'NAME': 'db_name',
        'USER': 'username',
        'PASSWORD': get_secret('DB_PASSWORD'),
    },
}

Kredi: İki Kepçe Django: Django 1.8 için En İyi Uygulamalar, Daniel Roy Greenfeld ve Audrey RoyGreenfeld. Telif Hakkı 2015 Two Scoops Press (ISBN 978-0981467344)

Sunucular arasında Ayarları yönetmek için Ortam değişkenlerini kullanma #

Twelve-Factor App bölümünde belirtildiği gibi, ortam değişkenlerini kullanmak, bir uygulamanın yapılandırmasını ortama bağlı olarak ayarlamanın yaygın olarak kullanılan bir yoludur.

Yapılandırmaların dağıtım ortamları arasında değişmesi muhtemel olduğundan, bu, uygulamanın kaynak kodunu kazmak zorunda kalmadan yapılandırmayı değiştirmenin ve uygulama dosyalarının ve kaynak kod deposunun dışında gizli sırlar tutmanın çok ilginç bir yoludur.

Django’da ana ayarlar projenizin klasöründe settings.py olarak bulunur. Basit bir Python dosyası olduğundan, ortama erişmek için standart kitaplıktan Python’un os modülünü kullanabilirsiniz (ve hatta uygun varsayılanlara sahip olabilirsiniz).

settings.py #

import os

SECRET_KEY = os.environ.get('APP_SECRET_KEY', 'unsafe-secret-key')

DEBUG = bool(os.environ.get('DJANGO_DEBUG', True) == 'False')

ALLOWED_HOSTS = os.environ.get('DJANGO_ALLOWED_HOSTS', '').split()

DATABASES = {
    'default': {
        'ENGINE': os.environ.get('APP_DB_ENGINE', 'django.db.backends.sqlite3'),
        'NAME': os.environ.get('DB_NAME', 'db.sqlite'),    
        'USER': os.environ.get('DB_USER', ''),
        'PASSWORD': os.environ.get('DB_PASSWORD', ''),
        'HOST': os.environ.get('DB_HOST', None),
        'PORT': os.environ.get('DB_PORT', None),
        'CONN_MAX_AGE': 600,
    }
}

Django ile veritabanı teknolojinizi değiştirebilirsiniz, böylece geliştirme makinenizde sqlite3’ü kullanabilirsiniz (ve bu, bir kaynak kontrol sistemine bağlanmak için makul bir varsayılan olmalıdır). Bu mümkün olsa da tavsiye edilmez:

Uygulamanın veritabanı, kuyruk sistemi veya önbellek gibi yedekleme hizmetleri, geliştirme/ürün paritesinin önemli olduğu alanlardan biridir. (On İki Faktör Uygulaması - Geliştirme/ürün paritesi)

Veritabanı bağlantısı için DATABASE_URL parametresi kullanmak için lütfen ilgili örneğe bakın.

2: https://www.wikiod.com/tr/django/ayarlar#Ortamdan bir DATABASE_URL kullanma

Uygulama taşınabilirliğini sağlamak için BASE_DIR kullanma #

Uygulamanızda yolları sabit kodlamak kötü bir fikirdir. Kodunuzun farklı makinelerde sorunsuz bir şekilde çalışabilmesi için her zaman göreli URL’ler kullanılmalıdır. Bunu ayarlamanın en iyi yolu, bunun gibi bir değişken tanımlamaktır.

import os
BASE_DIR = os.path.dirname(os.path.dirname(__file__))

Ardından, diğer tüm ayarlarınızı tanımlamak için bu BASE_DIR değişkenini kullanın.

TEMPLATE_PATH = os.path.join(BASE_DIR, "templates")
STATICFILES_DIRS = [
    os.path.join(BASE_DIR, "static"),

]

Ve benzeri. Bu, kodunuzu herhangi bir endişe duymadan farklı makineler arasında taşıyabilmenizi sağlar.

Ancak, “os.path” biraz ayrıntılıdır. Örneğin, ayar modülünüz project.settings.dev ise, şunu yazmanız gerekecek:

BASE_DIR = os.path.dirname(os.path.dirname(os.path.dirname(__file__)))

Bir alternatif, ‘unipath’ modülünü kullanmaktır (‘pip install unipath’ ile kurabilirsiniz).

from unipath import Path

BASE_DIR = Path(__file__).ancestor(2)  # or ancestor(3) if using a submodule

TEMPLATE_PATH = BASE_DIR.child('templates')
STATICFILES_DIRS = [
    BASE_DIR.child('static'),
]

Birden fazla ayar kullanma #

Django varsayılan proje düzeni, tek bir “settings.py” oluşturur. Bu, genellikle şu şekilde bölmek için yararlıdır:

myprojectroot/
    myproject/
        __init__.py
        settings/
            __init__.py
            base.py
            dev.py
            prod.py
            tests.py

Bu, geliştirme, üretim, test veya başka bir yerde olmanıza göre farklı ayarlarla çalışmanıza olanak tanır.

Varsayılan düzenden bu düzene geçerken, orijinal ‘settings.py’, ‘settings/base.py’ olur. Diğer tüm alt modüller “from .base import *” ile başlayarak “settings/base.py” “alt sınıfını” oluşturacak. Örneğin, “settings/dev.py” şöyle görünebilir:

# -*- coding: utf-8 -*-
from .base import *  # noqa

DEBUG = True
INSTALLED_APPS.extend([
    'debug_toolbar',
])
EMAIL_BACKEND = 'django.core.mail.backends.console.EmailBackend'
INTERNAL_IPS = ['192.168.0.51', '192.168.0.69']

Alternatif #1 #

“Django-admin” komutlarının düzgün çalışması için, “DJANGO_SETTINGS_MODULE” ortam değişkenini (varsayılan olarak “myproject.settings”) ayarlamanız gerekir. Geliştirme aşamasında, bunu “myproject.settings.dev” olarak ayarlayacaksınız. Üretimde, bunu myproject.settings.prod olarak ayarlayacaksınız. Bir sanalenv kullanıyorsanız, onu ‘postactivate’ komut dosyanızda ayarlamak en iyisidir:

#!/bin/sh
export PYTHONPATH="/home/me/django_projects/myproject:$VIRTUAL_ENV/lib/python3.4"
export DJANGO_SETTINGS_MODULE="myproject.settings.dev"

DJANGO_SETTINGS_MODULE tarafından işaret edilmeyen bir ayar modülünü bir kez kullanmak istiyorsanız, django-adminin --settings seçeneğini kullanabilirsiniz:

django-admin test --settings=myproject.settings.tests

Alternatif #2 #

DJANGO_SETTINGS_MODULE varsayılan konfigürasyonunda (myproject.settings) bırakmak istiyorsanız, __init__.py dosyanıza içe aktarmayı yerleştirerek settings modülüne hangi konfigürasyonun yükleneceğini söyleyebilirsiniz.

Yukarıdaki örnekte, bir “init.py” şu şekilde ayarlanarak aynı sonuca ulaşılabilir:

from .dev import *

Birden çok gereksinim dosyası kullanma #

Her gereksinim dosyası, bir ayar dosyasının adıyla eşleşmelidir. Daha fazla bilgi için Birden çok ayarı kullanma bölümünü okuyun.

Yapı #

djangoproject
├── config
│   ├── __init__.py
│   ├── requirements
│   │   ├── base.txt
│   │   ├── dev.txt
│   │   ├── test.txt
│   │   └── prod.txt
│   └── settings
└── manage.py

base.txt dosyasında, tüm ortamlarda kullanılan bağımlılıkları yerleştirin.

# base.txt
Django==1.8.0
psycopg2==2.6.1
jinja2==2.8

Ve diğer tüm dosyalarda, “-r base.txt” ile temel bağımlılıkları dahil edin ve mevcut ortam için gereken belirli bağımlılıkları ekleyin.

# dev.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in dev env
django-queryinspect==0.1.0

# test.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in test env
nose==1.3.7
django-nose==1.4

# prod.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in production env
django-queryinspect==0.1.0
gunicorn==19.3.0
django-storages-redux==1.3
boto==2.38.0

Son olarak, bağımlılıkları yüklemek için. Örnek, dev env’de: pip install -r config/requirements/dev.txt

1: https://www.wikiod.com/tr/django/ayarlar#Birden çok ayar kullanma

Saat dilimini ayarlama #

Django tarafından kullanılacak saat dilimini settings.py dosyasında ayarlayabilirsiniz. Örnekler:

TIME_ZONE = 'UTC'  # use this, whenever possible
TIME_ZONE = 'Europe/Berlin'
TIME_ZONE = 'Etc/GMT+1'

İşte geçerli saat dilimlerinin listesi

Bir Windows ortamında çalışırken bu, sistem saat diliminizle aynı ayarlanmalıdır.

Django’nun saat dilimine duyarlı tarih saatleri kullanmasını istemiyorsanız:

USE_TZ = False

Django en iyi uygulamaları, veri tabanında bilgi depolamak için “UTC” kullanılmasını gerektirir:

Web siteniz yalnızca bir saat diliminde kullanılsa bile, yine de veri tabanınızda UTC’de veri depolamak için iyi uygulama. Asıl sebep Yaz Saati Uygulamasıdır (DST). Birçok ülkede bir DST sistemi vardır, ilkbaharda saatlerin ileri, sonbaharda geri alındığı yer. Eğer yerel saatte çalışıyorsanız, iki kez hatayla karşılaşmanız olasıdır. geçişlerin gerçekleştiği bir yıl.

https://docs.djangoproject.com/en/stable/topics/i18n/timezones/

Ayarlara erişim #

Tüm ayarlarınızı yaptıktan sonra bunları kodunuzda kullanmak isteyeceksiniz. Bunu yapmak için dosyanıza aşağıdaki içe aktarmayı ekleyin:

from django.conf import settings

Ardından ayarlarınıza “ayarlar” modülünün öznitelikleri olarak erişebilirsiniz, örneğin:

if not settings.DEBUG:
    email_user(user, message)

Ortamdan bir DATABASE_URL kullanma #

Heroku gibi PaaS sitelerinde, veritabanı bilgilerini birkaç parametre (host, port, user, password…) yerine tek bir URL ortam değişkeni olarak almak olağandır.

dj_database_url adlı bir modül var ve DATABASE_URL ortam değişkenini Django’da veritabanı ayarlarını enjekte etmeye uygun bir Python sözlüğüne otomatik olarak ayıklıyor.

Kullanım:

import dj_database_url

if os.environ.get('DATABASE_URL'):
    DATABASES['default'] =
        dj_database_url.config(default=os.environ['DATABASE_URL'])
