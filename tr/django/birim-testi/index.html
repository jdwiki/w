Veritabanı ve Test Etme #

Django, test sırasında özel veritabanı ayarları kullanır, böylece testler veritabanını normal şekilde kullanabilir ancak varsayılan olarak boş bir veritabanında çalışabilir. Bir testteki veritabanı değişiklikleri başka bir test tarafından görülmeyecektir. Örneğin, aşağıdaki testlerin her ikisi de başarılı olacaktır:

from django.test import TestCase
from myapp.models import Thing

class MyTest(TestCase):

    def test_1(self):
        self.assertEqual(Thing.objects.count(), 0)
        Thing.objects.create()
        self.assertEqual(Thing.objects.count(), 1)

    def test_2(self):
        self.assertEqual(Thing.objects.count(), 0)
        Thing.objects.create(attr1="value")
        self.assertEqual(Thing.objects.count(), 1)

Armatürler

Birden çok test tarafından kullanılan veritabanı nesnelerine sahip olmak istiyorsanız, bunları test senaryosunun “setUp” yönteminde oluşturun. Ek olarak, Django projenizde fikstür tanımladıysanız, bunlar şu şekilde dahil edilebilir:

class MyTest(TestCase):
    fixtures = ["fixture1.json", "fixture2.json"]

Varsayılan olarak, Django her uygulamadaki “fikstürler” dizininde demirbaşları arar. “FIXTURE_DIRS” ayarı kullanılarak başka dizinler de ayarlanabilir:

# myapp/settings.py
FIXTURE_DIRS = [
    os.path.join(BASE_DIR, 'path', 'to', 'directory'),
]

Aşağıdaki gibi bir model oluşturduğunuzu varsayalım:

# models.py
from django.db import models


class Person(models.Model):
    """A person defined by his/her first- and lastname."""
    firstname = models.CharField(max_length=255)
    lastname = models.CharField(max_length=255)

O zaman .json fikstürleriniz şöyle görünebilir:

# fixture1.json
[
    { "model": "myapp.person",
        "pk": 1,
        "fields": {
            "firstname": "Peter",
            "lastname": "Griffin"
        }
    },
    { "model": "myapp.person",
        "pk": 2,
        "fields": {
            "firstname": "Louis",
            "lastname": "Griffin"
        }
    },
]

Test veritabanını yeniden kullanın

Test çalıştırmalarınızı hızlandırmak için yönetim komutuna test veritabanını yeniden kullanmasını (ve her test çalıştırmasından sonra oluşturulup silinmesini önlemek için) söyleyebilirsiniz. Bu, şu şekilde keepdb (veya stenografi ‘-k’) bayrağı kullanılarak yapılabilir:

# Reuse the test-database (since django version 1.8)
$ python manage.py test --keepdb

Test - eksiksiz bir örnek #

Bu, yeni bir Django projesi başlatmayla ilgili belgeleri okuduğunuzu varsayar. Projenizdeki ana uygulamanın td (test sürüşünün kısaltması) olarak adlandırıldığını varsayalım. İlk testinizi oluşturmak için test_view.py adında bir dosya oluşturun ve aşağıdaki içeriği kopyalayıp içine yapıştırın.

from django.test import Client, TestCase

class ViewTest(TestCase):

    def test_hello(self):
        c = Client()
        resp = c.get('/hello/')
        self.assertEqual(resp.status_code, 200)

Bu testi şu şekilde çalıştırabilirsiniz:

 ./manage.py test

ve doğal olarak başarısız olacak! Aşağıdakine benzer bir hata göreceksiniz.

Traceback (most recent call last):
  File "/home/me/workspace/td/tests_view.py", line 9, in test_hello
    self.assertEqual(resp.status_code, 200)
AssertionError: 200 != 404

Bu neden oluyor? Çünkü bunun için bir görüş tanımlamadık! Öyleyse hadi yapalım. views.py adında bir dosya oluşturun ve içine aşağıdaki kodu yerleştirin.

from django.http import HttpResponse
def hello(request):
    return HttpResponse('hello')

Daha sonra, urls py’yi aşağıdaki gibi düzenleyerek /hello/ ile eşleyin:

from td import views

urlpatterns = [
    url(r'^admin/', include(admin.site.urls)),
    url(r'^hello/', views.hello),
    ....
]

Şimdi testi tekrar ./manage.py test tekrar çalıştırın ve viyola!!

.
----------------------------------------------------------------------
Ran 1 test in 0.004s

OK

Yürütülen test sayısını sınırlayın #

Test çalıştırıcısı tarafından hangi modüllerin keşfedileceğini belirterek “manage.py testi” tarafından yürütülen testleri sınırlamak mümkündür:

# Run only tests for the app names "app1"
$ python manage.py test app1

# If you split the tests file into a module with several tests files for an app
$ python manage.py test app1.tests.test_models

# it's possible to dig down to individual test methods.
$ python manage.py test app1.tests.test_models.MyTestCase.test_something

Bir sürü test yapmak istiyorsanız, bir dosya adı kalıbı iletebilirsiniz. Örneğin, yalnızca modellerinizi içeren testleri çalıştırmak isteyebilirsiniz:

$ python manage.py test -p test_models*
Creating test database for alias 'default'...
.................................................
----------------------------------------------------------------------
Ran 115 tests in 3.869s

OK

Son olarak, --failfast kullanarak test takımını ilk başarısızlıkta durdurmak mümkündür. Bu argüman, süitte karşılaşılan olası hatayı hızlı bir şekilde elde etmeyi sağlar:

$ python manage.py test app1
...F..
----------------------------------------------------------------------
Ran 6 tests in 0.977s

FAILED (failures=1)


$ python manage.py test app1 --failfast
...F
======================================================================
[Traceback of the failing test]
----------------------------------------------------------------------
Ran 4 tests in 0.372s

FAILED (failures=1)

Django Görünümlerinde Erişim Kontrolünü Test Etme #

tl;dr : İki kullanıcı nesnesini tanımlayan bir temel sınıf oluşturun (“user” ve “another_user” deyin). Diğer modellerinizi oluşturun ve üç “Müşteri” örneği tanımlayın.

  • “self.client” : Tarayıcıda oturum açmış “kullanıcı"yı temsil ediyor
  • self.another_client : another_user müşterisini temsil ediyor
  • self.unlogged_client : Günlüğü olmayan kişiyi temsil ediyor

Şimdi bu üç istemci nesnesinden tüm genel ve özel URL’lerinize erişin ve beklediğiniz yanıtı dikte edin. Aşağıda, “özel” (birkaç ayrıcalıklı kullanıcıya ait) veya “genel” (herkes tarafından görülebilen) olabilen bir “Kitap” nesnesi stratejisini sergiledim.

from django.test import TestCase, RequestFactory, Client
from django.core.urlresolvers import reverse

class BaseViewTestCase(TestCase):

    @classmethod
    def setUpClass(cls):
        super(BaseViewTestCase, cls).setUpClass()
        cls.client = Client()
        cls.another_client = Client()
        cls.unlogged_client = Client()
        cls.user = User.objects.create_user(
                'dummy',password='dummy'
                )
        cls.user.save()
        cls.another_user = User.objects.create_user(
                'dummy2', password='dummy2'
                )
        cls.another_user.save()
        cls.first_book = Book.objects.create(
                name='first',
                private = true
        )
        cls.first_book.readers.add(cls.user)
        cls.first_book.save()
        cls.public_book = Template.objects.create(
                name='public',
                private=False
        )
        cls.public_book.save()


    def setUp(self):
        self.client.login(username=self.user.username, password=self.user.username)
        self.another_client.login(username=self.another_user.username, password=self.another_user.username)


"""
   Only cls.user owns the first_book and thus only he should be able to see it.
   Others get 403(Forbidden) error
"""
class PrivateBookAccessTestCase(BaseViewTestCase):
    
    def setUp(self):
        super(PrivateBookAccessTestCase, self).setUp()
        self.url = reverse('view_book',kwargs={'book_id':str(self.first_book.id)})

    def test_user_sees_own_book(self):
        response = self.client.get(self.url)
        self.assertEqual(200, response.status_code)
        self.assertEqual(self.first_book.name,response.context['book'].name)
        self.assertTemplateUsed('myapp/book/view_template.html')

    def test_user_cant_see_others_books(self):
        response = self.another_client.get(self.url)
        self.assertEqual(403, response.status_code)
        
    def test_unlogged_user_cant_see_private_books(self):
        response = self.unlogged_client.get(self.url)
        self.assertEqual(403, response.status_code)

"""
    Since book is public all three clients should be able to see the book
"""
 class PublicBookAccessTestCase(BaseViewTestCase):
    
    def setUp(self):
        super(PublicBookAccessTestCase, self).setUp()
        self.url = reverse('view_book',kwargs={'book_id':str(self.public_book.id)})

    def test_user_sees_book(self):
        response = self.client.get(self.url)
        self.assertEqual(200, response.status_code)
        self.assertEqual(self.public_book.name,response.context['book'].name)
        self.assertTemplateUsed('myapp/book/view_template.html')

    def test_another_user_sees_public_books(self):
        response = self.another_client.get(self.url)
        self.assertEqual(200, response.status_code)
        
    def test_unlogged_user_sees_public_books(self):
        response = self.unlogged_client.get(self.url)
        self.assertEqual(200, response.status_code)

Django Modellerini Etkili Bir Şekilde Test Etme #

Bir sınıf varsayarak

from django.db import models

class Author(models.Model):
   name = models.CharField(max_length=50)
   
    def __str__(self):
        return self.name
    
    def get_absolute_url(self):
        return reverse('view_author', args=[str(self.id)])


class Book(models.Model):
    author = models.ForeignKey(Manufacturer, on_delete=models.CASCADE)
    private = models.BooleanField(default=false)
    publish_date = models.DateField()

    def get_absolute_url(self):
        return reverse('view_book', args=[str(self.id)])
    
    def __str__(self):
        return self.name

Test örnekleri

from django.test import TestCase
from .models import Book, Author

class BaseModelTestCase(TestCase):

    @classmethod
    def setUpClass(cls):
        super(BaseModelTestCase, cls).setUpClass()
        cls.author = Author(name='hawking')
        cls.author.save()
        cls.first_book = Book(author=cls.author, name="short_history_of_time")
        cls.first_book.save()
        cls.second_book = Book(author=cls.author, name="long_history_of_time")
        cls.second_book.save()


class AuthorModelTestCase(BaseModelTestCase):
    def test_created_properly(self):
         self.assertEqual(self.author.name, 'hawking')
         self.assertEqual(True, self.first_book in self.author.book_set.all())
    
    def test_absolute_url(self):
        self.assertEqual(self.author.get_absolute_url(), reverse('view_author', args=[str(self.author.id)]))

        ...
        self.assertEqual(1, len(Book.objects.filter(name__startswith='long'))
    
    def test_absolute_url(self):
        ...

Bazı noktalar

  • ‘created_properly’ testleri, Django modellerinin durum özelliklerini doğrulamak için kullanılır. Varsayılan değerleri, file_upload_paths vb. değiştirdiğimiz durumları yakalamaya yardımcı olurlar.
  • “absolute_url” önemsiz görünebilir ancak url yollarını değiştirirken bazı hataları önlememe yardımcı olduğunu gördüm
  • Benzer şekilde bir model içinde uygulanan tüm yöntemler için test senaryoları yazarım (‘sahte’ nesneler vb. kullanarak)
  • Ortak bir ‘BaseModelTestCase’ tanımlayarak, doğru testi sağlamak için modeller arasında gerekli ilişkileri kurabiliriz.

Son olarak, şüpheye düştüğünüzde bir test yazın. Önemsiz davranış değişiklikleri, ayrıntılara dikkat edilerek yakalanır ve uzun süredir unutulmuş kod parçaları gereksiz sorunlara neden olmaz.
