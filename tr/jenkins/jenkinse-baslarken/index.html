Nginx Proxy’yi Ayarlama #

Doğal olarak, Jenkins 8080 numaralı bağlantı noktasında çalışır. 80 -> 8080 numaralı bağlantı noktasından bir proxy kurabiliriz, böylece Jenkins’e şu şekilde erişilebilir:

http://<url>.com

varsayılan yerine

http://<url>.com:8080

Nginx’i yükleyerek başlayın.

`sudo aptitude -y nginx’i kurun'

Nginx için varsayılan ayarları kaldırın

“cd /etc/nginx/siteler kullanılabilir”

`sudo rm default ../sites-enabled/default'

Yeni yapılandırma dosyasını oluşturun

‘sudo touch jenkins’

Yeni oluşturulan jenkins dosyasına aşağıdaki kodu kopyalayın.

upstream app_server {
  server 127.0.0.1:8080 fail_timeout=0;
}

server {
  listen 80;
  listen [::]:80 default ipv6only=on;
  server_name ;

  location / {
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header Host $http_host;
    proxy_redirect off;

    if (!-f $request_filename) {
        proxy_pass http://app_server;
        break;
    }
  }
}

Kullanılabilir siteler ve etkin siteler arasında sembolik bir bağlantı oluşturun:

sudo ln -s /etc/nginx/sites-available/jenkins /etc/nginx/sites-enabled/

Nginx proxy hizmetini yeniden başlatın

`sudo hizmeti nginx yeniden başlatma'

Jenkins artık 80 numaralı bağlantı noktasından erişilebilir olacak.

Jenkins tek bir yerde tam Giriş #

1. Jenkins :

Jenkins, Java ile yazılmış açık kaynaklı bir sürekli entegrasyon aracıdır. Proje, Oracle ile bir anlaşmazlıktan sonra Hudson’dan ayrıldı.

Özetle, Jenkins önde gelen açık kaynaklı otomasyon sunucusudur. Java ile oluşturulmuş, neredeyse her proje için oluşturma, test etme, dağıtma ve otomasyonu desteklemek için yüzlerce eklenti sağlar.

Özellikler : Jenkins, aşağıdaki ana özellikleri kutunun dışında sunar ve eklentiler aracılığıyla daha pek çoğu eklenebilir:

Kolay kurulum: Java -jar jenkins.war’ı çalıştırın, bir sunucu uygulaması kapsayıcısına dağıtın. Ek kurulum yok, veritabanı yok. Bir yükleyici veya yerel paket mi tercih ediyorsunuz? Bizde de bunlar var. Kolay yapılandırma: Jenkins, kapsamlı anında hata kontrolleri ve satır içi yardım ile tamamen kullanıcı dostu web GUI’sinden yapılandırılabilir. Zengin eklenti ekosistemi: Jenkins, var olan hemen hemen her SCM veya oluşturma aracıyla entegre olur. Eklentileri görüntüleyin. Genişletilebilirlik: Jenkins’in çoğu parçası genişletilebilir ve değiştirilebilir ve yeni Jenkins eklentileri oluşturmak kolaydır. Bu, Jenkins’i ihtiyaçlarınıza göre özelleştirmenizi sağlar. Dağıtılmış yapılar: Jenkins, yapı/test yüklerini farklı işletim sistemlerine sahip birden çok bilgisayara dağıtabilir. OS X, Linux ve Windows için yazılım mı geliştiriyorsunuz? Sorun değil.

Kurulum :

$ wget -q -O - https://jenkins-ci.org/debian/jenkins-ci.org.key | sudo apt-key add -

$ sudo sh -c 'echo deb http://pkg.jenkins-ci.org/debian binary/ > /etc/apt/sources.list.d/jenkins.list'
$ sudo apt-get update
$ sudo apt-get install jenkins

JENKINS_HOME dizini Jenkins, derlemeleri gerçekleştirmek ve arşivleri tutmak için biraz disk alanına ihtiyaç duyar. Bu konumu Jenkins’in yapılandırma ekranından kontrol edebilirsiniz. Varsayılan olarak bu, ~/.jenkins olarak ayarlanmıştır, ancak bunu aşağıdaki yollardan biriyle değiştirebilirsiniz: Sunucu uygulaması kapsayıcısını başlatmadan önce “JENKINS_HOME” ortam değişkenini yeni ana dizine ayarlayın. “JENKINS_HOME” sistem özelliğini sunucu uygulaması kapsayıcısına ayarlayın. JNDI ortam girdisini “JENKINS_HOME” yeni dizine ayarlayın. Kapsayıcınız için bunun nasıl yapılacağı hakkında daha fazla bilgi için kapsayıcıya özel doküman koleksiyonuna bakın. Jenkins’i bir süre kullandıktan sonra da bu konumu değiştirebilirsiniz. Bunu yapmak için Jenkins’i tamamen durdurun, içeriği eski JENKINS_HOME’dan yeni eve taşıyın, yeni JENKINS_HOME’u ayarlayın ve Jenkins’i yeniden başlatın. JENKINS_HOME, aşağıdakine benzeyen oldukça açık bir dizin yapısına sahiptir:

JENKINS_HOME

+- config.xml     (jenkins root configuration)
 +- *.xml          (other site-wide configuration files)
 +- userContent    (files in this directory will be served under your http://server/userContent/)
 +- fingerprints   (stores fingerprint records)
 +- plugins        (stores plugins)
 +- workspace (working directory for the version control system)
     +- [JOBNAME] (sub directory for each job)
 +- jobs
     +- [JOBNAME]      (sub directory for each job)
         +- config.xml     (job configuration file)
         +- latest         (symbolic link to the last successful build)
         +- builds
             +- [BUILD_ID]     (for each build)
                 +- build.xml      (build result summary)
                 +- log            (log file)
                 +- changelog.xml  (change log)

Jenkins Build İşleri :

Jenkins’te yeni bir inşa işi oluşturmak basittir: Jenkins panosundaki “Yeni İş” menü öğesini tıklamanız yeterlidir. Jenkins, yeni bir iş yaratmayı seçtiğinizde size sunulan birkaç farklı inşa işini destekler.

Serbest stil yazılım projesi

Serbest stil yapım işleri, maksimum esneklik sağlayan genel amaçlı yapım işleridir.

Maven projesi “maven2/3 projesi”, Maven projelerine özel olarak uyarlanmış bir yapım işidir. Jenkins, Maven pom dosyalarını ve proje yapılarını anlar ve projenizi kurmak için yapmanız gereken işi azaltmak için pom dosyasından toplanan bilgileri kullanabilir.

İş akışı

Birden çok yapı bağımlı birimine yayılabilen uzun süreli etkinlikleri düzenler. Serbest stil iş türüne kolayca uymayan karmaşık faaliyetler düzenlemek ve/veya boru hatları oluşturmak için uygundur.

Harici bir işi izleyin “Harici bir işi izleyin” oluşturma işi, cron işleri gibi etkileşimli olmayan süreçlere göz atmanızı sağlar.

Çoklu yapılandırma işi “Çoklu konfigürasyon projesi” (“matris projesi” olarak da anılır), aynı inşa işini birçok farklı konfigürasyonda çalıştırmanıza izin verir. Bu güçlü özellik, bir uygulamayı birçok farklı ortamda, farklı veritabanlarıyla ve hatta farklı yapı makinelerinde test etmek için faydalı olabilir.

1. Bir yazılım projesi oluşturma (serbest stil)

Jenkins, sürekli/resmi/gecelik derlemeler yapmak, testler çalıştırmak veya bazı tekrarlayan toplu görevleri gerçekleştirmek gibi tipik derleme sunucusu işlerini gerçekleştirmek için kullanılabilir. Buna Jenkins’te “serbest stil yazılım projesi” denir. Projeyi kurmak Jenkins’in en üst sayfasına gidin, “Yeni İş"i seçin, ardından “Serbest stilde bir yazılım projesi oluşturun"u seçin. Bu iş türü aşağıdaki unsurlardan oluşur: kaynak kodunuzun bulunduğu CVS veya Subversion gibi isteğe bağlı SCM. Jenkins’in derlemeleri ne zaman gerçekleştireceğini kontrol etmek için isteğe bağlı tetikleyiciler. gerçek işin gerçekleştiği yerde derlemeyi (ant, maven, kabuk betiği, toplu iş dosyası, vb.) gerçekleştiren bir tür derleme betiği yapıları arşivleme ve/veya javadoc ve test sonuçlarını kaydetme gibi yapıdan bilgi toplamak için isteğe bağlı adımlar. e-posta, anlık ileti gönderme, sorun izleyiciyi güncelleme vb. gibi derleme sonucuyla diğer kişileri/sistemleri bilgilendirmek için isteğe bağlı adımlar.

Kaynak Dışı Kontrol Projeleri için Derlemeler Bazen yalnızca gösterim amacıyla bir proje oluşturmaya ihtiyaç duyulabilir veya bir SVN/CVS deposuna erişim sağlanamaz. “Kaynak Kodu Yönetimi” altında projeyi “Yok” olarak yapılandırmayı seçerek şunları yapmanız gerekir:

  1. Projeyi en az bir kez oluşturun (başarısız olacaktır), ancak Jenkins jenkins/workspace/PROJECTNAME/ yapısını oluşturacaktır.
  2. Proje dosyalarını jenkins/workspace/PROJECTNAME/ dizinine kopyalayın.
  3. Yeniden oluşturun ve uygun şekilde yapılandırın

Jenkins Ortam Değişkenlerini Ayarladı

Bir Jenkins işi yürütüldüğünde, kabuk betiğinizde, toplu iş komutunuzda, Ant betiğinizde veya Maven POM’da kullanabileceğiniz bazı ortam değişkenlerini ayarlar. ENVIRONMENT_VARIABLE öğesini tıklayarak değişken listesine bakın

Otomatik derlemeleri yapılandırma

Jenkins’deki derlemeler periyodik olarak (bir programa göre, konfigürasyonda belirtilen) veya projede kaynak değişiklikleri tespit edildiğinde veya URL istenerek otomatik olarak tetiklenebilir:

http://SAHİBİNİZ/jenkins/job/PROJECTNAME/build

Bu, Jenkins yapılarını çeşitli kurulumlara bağlamanıza olanak tanır. Daha fazla bilgi için (özellikle bunu güvenlik etkinken yapmak), bkz. Uzaktan erişim API’si.

Kaynak değişikliklerine göre oluşturulur

Jenkins’in değişiklikler için Revizyon Kontrol Sisteminizi yoklamasını sağlayabilirsiniz. Jenkins’in, Unix/Linux’taki crontab ile aynı sözdizimini kullanarak revizyon kontrol sisteminizi ne sıklıkla yoklayacağını belirleyebilirsiniz. Ancak, yoklama süreniz revizyon kontrol sisteminizi yoklamak için gerekenden daha kısaysa, her değişiklik için birden çok derlemeyle karşılaşabilirsiniz. Yoklama sürenizi, revizyon kontrol sisteminizi yoklamak için gereken süreden daha uzun olacak şekilde ayarlamalı veya bir taahhüt sonrası tetikleyici kullanmalısınız. Sisteminizi yoklamanın ne kadar sürdüğünü görmek için her bir derlemenin Yoklama Günlüğünü inceleyebilirsiniz.

Alternatif olarak, sabit bir aralıkta yoklama yapmak yerine, bir URL tetikleyicisi (yukarıda açıklanmıştır) kullanabilirsiniz, ancak URL’nin sonunda /build yerine /oluşturma ile birlikte. Bu, Jenkins’i hemen oluşturmak yerine değişiklikler için SCM’yi yoklatır. Bu, Jenkins’in işle ilgisi olmayan modülleri veya dalları etkileyen taahhütler için ilgili herhangi bir değişiklik olmadan bir yapı çalıştırmasını engeller. /polling kullanılırken, iş yoklama için yapılandırılmalıdır, ancak program boş olabilir.

E-posta ile oluşturulur (sendmail)

Sisteminizin kök hesabına sahipseniz ve sendmail kullanıyorsanız, /etc/aliases’i ayarlamanın en kolay yolunu buldum ve aşağıdaki girişi ekleyin: jenkins-foo: “|/bin/wget -o /dev/null

http://SAHİBİNİZ/jenkins/job/PROJECTNAME/build”

ve sonra sendmail’in değişiklikten haberdar olması için “yeni takma adlar” komutunu çalıştırın. Ne zaman birisi “jenkins-foo@yoursystem” adresine bir e-posta gönderse, bu yeni bir yapıyı tetikleyecektir. Sendmail’i yapılandırma hakkında daha fazla ayrıntı için buna bakın. E-posta ile derlemeler (qmail) qmail ile /var/qmail/alias/.qmail-jenkins’i aşağıdaki gibi yazabilirsiniz: |/bin/wget -o /dev/null http://YOURHOST/jenkins/job/PROJECTNAME/build”

2. Bir Maven projesi inşa etmek

Jenkins, Maven 2/3’e adanmış bir iş türü sağlar. Bu iş türü, Jenkins’i Maven 2/3 ile derinlemesine bütünleştirir ve daha genel serbest stil yazılım projesine kıyasla aşağıdaki avantajları sağlar.

Jenkins, işini yapmak için gereken bilgilerin çoğunu elde etmek için Maven POM’larını ayrıştırır. Sonuç olarak, yapılandırma miktarı büyük ölçüde azalır.

Jenkins, Maven’in infazını dinler ve kendi başına ne yapılması gerektiğini anlar. Örneğin, Maven test aşamasını çalıştırdığında JUnit raporunu otomatik olarak kaydeder. Veya javadoc hedefini çalıştırırsanız, Jenkins otomatik olarak javadoc’u kaydeder.

Jenkins, birbirleri arasında SNAPSHOT bağımlılıkları bildiren projeler arasında otomatik olarak proje bağımlılıkları oluşturur. Aşağıya bakınız. Bu nedenle çoğunlukla SCM bilgilerini ve hangi hedefleri gerçekleştirmek istediğinizi yapılandırmanız yeterlidir ve Jenkins diğer her şeyi çözecektir.

Bu proje türü, aşağıdaki özellikleri otomatik olarak sağlayabilir:

Bir yapı tarafından üretilen arşiv yapıtları

Test sonuçlarını yayınlama

Aşağı akış bağımlılıkları olan projeler için işleri tetikleyin

Yapılarınızı bir Maven deposuna dağıtın

Modüle göre çıkış testi sonuçları

İsteğe bağlı olarak yalnızca değiştirilen modülleri yeniden oluşturarak yapılarınızı hızlandırın

Modül bağımlılıklarından otomatik yapı zincirleme

Jenkins, projenizin bağımlılıklarını POM’unuzdan okur ve bunlar da Jenkins üzerine kuruluysa, tetikleyiciler, bu bağımlılıklardan birinde yeni bir yapı otomatik olarak projenizin yeni bir yapısını başlatacak şekilde ayarlanır. Jenkins, POM’daki her türlü bağımlılığı anlar. Yani, ana POM

<dependencies> section of your project
<plugins> section of your project
<extensions> section of your project
<reporting> section of your project

Bu işlem, sürümleri hesaba katar, böylece aynı Jenkins üzerinde projenizin birden fazla sürümüne/dalına sahip olabilirsiniz ve bu, bağımlılıkları doğru bir şekilde belirleyecektir. Bağımlılık sürümü aralıklarının desteklenmediğini unutmayın, bunun nedeni için https://issues.jenkins-ci.org/browse/JENKINS-2787 bakın.

Bu özellik istendiğinde devre dışı bırakılabilir - Yapılandırma seçeneğine bakın Bir SNAPSHOT bağımlılığı oluşturulduğunda oluştur

Otomatik olarak yükle onay kutusunu işaretlerseniz Jenkins, Maven’in istenen sürümünü sizin için indirip yükleyecek ve bunu Jenkins ana dizinindeki araçlar dizinine yükleyecektir.

Bu nasıl kullanılır

Öncelikle, bir Maven kurulumu yapılandırmanız gerekir (DEV@cloud kullanıyorsanız bu adım atlanabilir). Bu, sistem yapılandırma ekranına gidilerek yapılabilir (Jenkins’i Yönet-> Sistemi Yapılandır). “Maven Kurulumları” bölümünde 1) Ekle butonuna tıklayın, 2) “Maven 3.0.3” gibi bir isim verin ve 3) açılır menüden versiyonu seçin.

Şimdi, Jenkins bu sürümü, ihtiyaç duyulduğunda (örneğin, herhangi bir yeni inşa makinesinde) Apache’den indirerek ve sıkıştırmasını açarak otomatik olarak yükleyecektir.

  4. İşinizi kaydedin

  2. aramak için maven hedefini seçin

  3. Depo URL’sini ve Kimlik Bilgisini ekleyin.

  4. kullanıcı özel maven deposunu kontrol edin:

Bunun için özel yolu da tanımlayabilirsiniz.

  1. Proje Oluştur

Şimdi inşa et’e tıklayarak projenizi oluşturun ve jenkins’in Maven’i kurmasını izlemek, projenizi kontrol etmek ve maven kullanarak inşa etmek için sol taraftaki “Yönetici Durumu Oluştur” ilerleme çubuğuna tıklayın.

Jenkins’i bir bilgisayardan diğerine taşıyın #

Bu benim için Ubuntu 12.04’ten (Jenkins ver. 1.628) Ubuntu 16.04’e (Jenkins ver. 1.651.2) geçmek için çalıştı. İlk önce depolardan Jenkins’i kurdum.

  1. Her iki Jenkins sunucusunu da durdurun

  2. “JENKINS_HOME” dosyasını (örn. /var/lib/jenkins) eski sunucudan yenisine kopyalayın. Yeni sunucudaki bir konsoldan:

    rsync -av username@old-server-IP:/var/lib/jenkins/ /var/lib/jenkins/

  3. [Yeni Jenkins sunucunuzu başlatın2

Buna ihtiyacın olmayabilir, ama benim zorundaydım.

  • “Jenkins’i Yönet” ve “Yapılandırmayı Diskten Yeniden Yükle”.
  • Tüm kölelerin bağlantısını kesin ve tekrar bağlayın.
  • ‘Sistemi Yapılandır > Jenkins Konumu’nda, ‘Jenkins URL’sinin yeni Jenkins sunucusuna doğru şekilde atandığını kontrol edin.

sudo apt-get install jenkins

Red Hat Enterprise Linux (RHEL), CentOS, Fedora veya Scientific Linux gibi RPM Tabanlı dağıtımlar için

Kararlı sürüm için depo dosyasını indirmek için:

sudo wget -O /etc/yum.repos.d/jenkins.repo http://pkg.jenkins-ci.org/redhat-stable/jenkins.repo

Veya en son haftalık sürümleri istiyorsanız:

sudo wget -O /etc/yum.repos.d/jenkins.repo http://pkg.jenkins-ci.org/redhat/jenkins.repo

Genel anahtarı içe aktarın:

sudo rpm --import https://jenkins-ci.org/redhat/jenkins-ci.org.key

Jenkins’i yum kullanarak kurun:

`sudo yum jenkins’i kurun’

Jenkins, çalıştırmak ve yüklemek için Java’ya ihtiyaç duyar:

sudo yum java'yı kurun

Jenkins’i başlatmak/durdurmak/yeniden başlatmak için şunu kullanın:

`sudo service jenkins başlat/durdur/yeniden başlat'

Jenkins’i yükseltme (RPM kurulumları) #

  1. Yedek jenkins ana dizini
  2. Aşağıdaki konumdaki jenkins.war dosyasını yeni WAR dosyasıyla değiştirin. /usr/lib/jenkins/jenkins.war`
  3. Jenkins’i yeniden başlatın
  4. Sabitlenmiş eklentileri kontrol edin ve gerekirse sabitlemeyi kaldırın
  5. Yapılandırmayı Diskten Yeniden Yükleyin

not: Birlikte verilen iskele uygulama sunucusu için Jenkins 2 yükseltmeleri için, /etc/sysconfig/jenkins içinde AJP portunu devre dışı bırakın (JENKINS_AJP_PORT="-1" ayarlayın).

Jenkins’te bir proje yapılandırın #

Burada projemizin kodunun en son kopyasını kontrol edeceğiz, testleri yapacağız ve uygulamayı canlı hale getireceğiz. Bunu başarmak için aşağıdaki adımları izleyin:

  1. Jenkins’i tarayıcıda açın.
  2. Yeni İş bağlantısını tıklayın.
  3. Proje adını girin ve Serbest stilde bir yazılım projesi oluşturun bağlantısını seçin.
  4. Tamam butonuna tıklayın.
  5. Kaynak kodu yönetimi bölümü altında, kaynak kodu yönetimi aracınızın yanındaki radyo kutusunu seçin. Benim durumumda Git‘i seçtim.

Git deposunun URL’sini “git://github.com/example/example.git” gibi sağlayın 6. Oluşturma tetikleyicileri altında, Anket SCM’nin yanındaki radyo kutusunu seçin. 7. Program kutusuna ***** yazın. Bu kutu, yapıyı düzenli aralıklarla tetiklemekten sorumludur. *****, git repo’daki değişiklikler için işin her dakika tetikleneceğini belirtir. 8. Yap bölümünün altında, Yapım Adımı Ekle düğmesini tıklayın ve ardından projeyi oluşturmak istediğiniz seçeneği seçin. Execute Shell‘i seçtim. Komut kutusuna derleme, testleri çalıştırma ve prod’a dağıtma komutunu yazın. 10. Aşağı kaydırın ve Kaydet.

Yukarıda, Jenkins’te git deponuzdaki değişiklik için her dakika yapıyı tetikleyecek temel bir proje yapılandırdık. Not: Karmaşık projeyi kurmak için Jenkins’e bazı eklentiler yüklemeniz gerekebilir.

Jenkins 2 ardışık düzen komut dosyasıyla basit bir derleme projesi yapılandırın #

Burada, aşağıdaki adımları yapmak için Jenkins 2’de bir Groovy boru hattı oluşturacağız:

  • Projemize yeni kod taahhüt edilip edilmediğini her 5 dakikada bir doğrulayın
  • SCM deposundan ödeme kodu
  • Java kodumuzun Maven derlemesi
  • Entegrasyon testlerimizi çalıştırın ve sonuçları yayınlayın

İşte yapacağımız adımlar:

  1. En az 2.0 Jenkins sürümüne sahip olduğumuzdan emin olun (bunu sayfanızın sağ alt köşesinden kontrol edebilirsiniz):

  2. Jenkins ana sayfasında Yeni Öğe‘ye tıklayın

  3. Proje adını girin ve Pipeline‘ı seçin

  4. Oluşturma Tetikleyicileri bölümünde, Anket SCM seçeneğini seçin ve aşağıdaki 5 dakikalık CRON programını ekleyin: */5 * * * *

  5. Pipeline bölümünde, Pipeline Script veya Pipeline Script from SCM‘yi seçin

  6. Önceki adımda **SCM’den Pipeline Script’i seçtiyseniz, şimdi Depo URL’si içinde SCM veri havuzu (Git, Mercurial, Subversion) URL’nizi http://github.com gibi belirtmeniz gerekir. /example/example.git. Ayrıca, example.git deponuzdaki Groovy komut dosyanızın Script Yolu’nu da belirtmeniz gerekir, ör. “boru hatları/example.groovy”

  7. Aşağıdaki Groovy kodunu, daha önce Pipeline Script‘i tıkladıysanız doğrudan Groovy komut dosyası penceresinde veya SCM’den Pipeline Script‘i seçtiyseniz ’example.groovy’nizde kopyalayın.

    node(‘remote’) { // Note : this step is only needed if you’re using direct Groovy scripting stage ‘Checkout Git project’ git url: ‘https://github.com/jglick/simple-maven-project-with-tests.git’ def appVersion = version() if (appVersion) { echo “Building version ${appVersion}” }

     stage 'Build Maven project'
 def mvnHome = tool 'M3'
 sh "${mvnHome}/bin/mvn -B -Dmaven.test.failure.ignore verify"
 step([$class: 'JUnitResultArchiver', testResults: '**/target/surefire-reports/TEST-*.xml'])

    } def version() { def matcher = readFile(‘pom.xml’) =~ ‘(.+)’ matcher ? matcher0 : null }

İşte şimdi, ilk Jenkins projenizi Jenkins 2 Groovy boru hattını kullanarak derleyip test edebilmeniz gerekir.

Eklentiyi harici kaynaktan yükleme #

java -jar [Path to client JAR] -s [Server address] install-plugin [Plugin ID]

JAR istemcisi, Jenkins’in kendisini çalıştıran aynı JAR/WAR değil, CLI JAR dosyası olmalıdır. Benzersiz kimlikler, Jenkins CLI wiki’sindeki (https://wiki.jenkins-ci.org/display/JENKINS/Plugins) eklentilerle ilgili sayfada bulunabilir.
