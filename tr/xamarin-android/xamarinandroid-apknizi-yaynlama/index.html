Bu konu, Xamarin.Android uygulamanızı serbest bırakma moduna nasıl hazırlayacağınıza ve nasıl optimize edeceğinize ilişkin bilgileri gösterir.

APK’nızı Visual Studio’da hazırlama #

Uygulamanızı bitirdiniz, hata ayıklama modunda test ettiniz ve mükemmel çalışıyor. Şimdi, onu Google Play Store’da yayınlamak için hazırlamak istiyorsunuz.

Xamarin belgeleri burada iyi bilgiler sağlar:

https://developer.xamarin.com/guides/android/deployment,_testing,_and_metrics/publishing_an_application/

Android Bildirimi

İlk olarak, Visual Studio’da, Solution Explorer’da Xamarin.Android projenize sağ tıklayın ve Özellikler’i seçin. Ardından, bu ekranı görmek için Android Manifest sekmesine gidin:

Android Studio veya Eclipse’den farklı olarak, AndroidManifest.xml dosyasını yazarak ayarlamanız gerekmez; Xamarin ve Visual Studio bunu sizin için yapar. Etkinlikler, Yayın Alıcıları ve Hizmetler, [sınıflarında belirli nitelikler bildirerek2 Android Manifest’e eklenir.

Bu ekranda seçenekler şunlardır:

  • Uygulama adı: Bu, kullanıcının görebileceği uygulama adıdır.
  • Paket adı: Bu, paket adıdır. Benzersiz olmalıdır, yani Google Play Store’daki diğer uygulamalarla aynı paket adını kullanmamalıdır.
  • Uygulama Simgesi: Bu, Android Studio veya Eclipse projelerinde kullanılan @drawable/ic_launcher’a eşdeğer, kullanıcı tarafından görülebilecek simgedir.
  • Sürüm numarası: Sürüm numarası, sürüm kontrolü için Google Play tarafından kullanılır. Uygulamanızın güncellenmiş bir sürümü için APK yayınlamak istediğinizde, her yeni yükseltme için bu sayıya 1 eklemeniz gerekir.
  • Sürüm adı: Bu, kullanıcıya gösterilecek olan sürüm adıdır.
  • Yükleme konumu: Bu, APK’nizin cihaz deposunda veya SD Kartta nereye kurulacağını belirler.
  • Gerekli izinler: Burada, uygulamanız için hangi izinlerin gerekli olduğunu belirlersiniz.

Android Seçenekleri

Aşağıdaki ekranda derleyici seçeneklerini yapılandırabilirsiniz. Buradaki doğru seçenekleri kullanmak APK boyutunuzu büyük ölçüde azaltabilir ve ayrıca hataları önleyebilir.

  • Yapılandırma: Etkin (Sürüm).
  • Platform: Aktif (Herhangi bir CPU). Bunlar, Google Play Store için APK’nızı oluşturmak için gereklidir. Yapılandırma Hata Ayıklama olarak ayarlanmışsa, Google Play tarafından kabul edilmeyecektir.
  • Paylaşılan Çalışma Zamanını Kullan: yanlış. Bunu true olarak ayarlarsanız APK, yürütmek için Mono Runtime’ı kullanır. Mono Runtime, USB üzerinden hata ayıklama sırasında otomatik olarak yüklenir, ancak Sürüm APK’sına yüklenmez. Cihazda Mono Runtime yüklü değilse ve bu seçenek Sürüm APK’sında true olarak ayarlanırsa uygulama çökecektir.
  • Seçilen ABI başına bir paket (.apk) oluşturun: false. Uyumluluk nedenleriyle APK’nızı mümkün olduğunca çok platform için oluşturun.
  • Multi-Dex’i etkinleştir: true, ancak uygulamanız çok karmaşık değilse (yani 65536’dan az yönteme sahipse buraya bakın yanlış olarak ayarlayabilirsiniz).
  • Proguard’ı etkinleştir: doğru. Bu, uygulamanızda Java kodunu gizleyen Proguard aracını etkinleştirir. .NET kodu için geçerli olmadığını unutmayın; .NET kodunu gizlemek istiyorsanız, Dotfuscator kullanmanız gerekir. Proguard for Xamarin.Android hakkında daha fazla bilgiyi burada bulabilirsiniz.
  • Geliştirici araçlarını etkinleştirin (hata ayıklama ve profil oluşturma): Sürüm APK için false.
  • Bağlama: SDK ve Kullanıcı Derlemeleri. Bu, Xamarin Linker’ın kullanılmayan tüm sınıfları SDK’dan ve kodunuzdan kaldırmasını sağlayarak APK boyutunu azaltır.

Önemli #

Xamarin.Linker, özellikle projenin Çekirdeğinde (PCL kitaplığı) varsa, bazen kodunuz tarafından kullanılmadığı görülen sınıfları kaldırabilir. Bunu önlemek için, Bağlantıyı “Yalnızca Sdk Derlemeleri” olarak ayarlayabilir veya sınıflarınızda Preserve özniteliğini kullanabilirsiniz, örneğin:

PreserveAttribute.cs

namespace My_App_Core.Models
{
    public sealed class PreserveAttribute : System.Attribute
    {
        public bool AllMembers;
        public bool Conditional;
    }
}

Sınıfta:

using System;

namespace My_App_Core.Models
{
    [Preserve(AllMembers = true)]
    public class ServiceException : Exception
    {
        public int errorCode;

        [Preserve(AllMembers = true)]
        public ServiceException() { }

        [Preserve(AllMembers = true)]
        public ServiceException(int errorCode)
        {
            this.errorCode = errorCode;
        }
    }
}
  • Desteklenen mimariler: Uyumluluk nedenleriyle Tümünü seçin.

Her şeyi yapılandırdıktan sonra, başarıyla oluşturulduğundan emin olmak için Projeyi Yeniden Oluşturun.

Xamarin.Android APK’nızda MultiDex’i Etkinleştirme #

MultiDex, Android APK’sında, uygulamanın 65.536’dan fazla yönteme sahip olmasına izin veren bir kitaplıktır.

Android APK’ları, Java kodunuzdan derlenen oluşturulan bayt kodlarını içeren Dalvik Yürütülebilir dosyalarına (.dex) sahiptir. Her .dex dosyası en fazla 65.536 yöntem (2^16) içerebilir.

Android 5.0 Lollipop’tan (API 21) önceki Android işletim sistemi sürümleri, APK başına yalnızca bir .dex dosyasını destekleyen ve APK başına 65.536 yöntemle sınırlayan Dalvik çalışma zamanını kullanır. Android 5.0’dan başlayarak, Android OS, APK başına birden fazla .dex dosyasını destekleyerek sınırdan kaçınan ART çalışma zamanını kullanır.

API 21’in altındaki Android sürümlerinde 65k yöntem sınırını aşmak için geliştiricilerin MultiDex destek kitaplığını kullanması gerekir. MultiDex, classes.dex dosyasında bunlara referans vererek ekstra class.dex dosyaları (classes2.dex, class3.dex, …) oluşturur. Uygulama yüklenmeye başladığında, fazladan .dex dosyalarını yüklemek için bir MultiDexApplication sınıfını kullanır.

Android uygulamanız API 21’e (Android 5.0 Lollipop) eşit veya daha yüksek bir minimum SDK sürümünü hedefliyorsa, işletim sistemi ekstra .dex dosyalarını yerel olarak işlediğinden MultiDex kitaplığını kullanmanız gerekmez. Ancak, uyumluluk nedenleriyle geliştirici eski Android işletim sistemini desteklemek isterse, MultiDex kitaplığını kullanmalıdır.

Xamarin.Android uygulamanızda MultiDex nasıl kullanılır #

İlk olarak, Xamarin.Android uygulamanızda MultiDex’i etkinleştirmek için, aşağıdaki yazdırma ekranında olduğu gibi projenizin Özellikler -> Android Seçenekleri -> Paketleme -> Multi-Dex’i Etkinleştir’e gidin:

Ardından, uygulamanızda bir MultiDexApplication sınıfı oluşturmalısınız. Projenin kökünde yeni bir sınıf oluşturun (Çözüm Gezgini’nde projeye sağ tıklayın, Ekle.. -> Sınıf veya Shift+Alt+C). Yeni sınıf dosyasında, örnek ad alanını Xamarin.Android proje ad alanınızın adıyla değiştirerek aşağıdaki kodu kopyalayın.

using System;
using Android.App;
using Android.Runtime;
using Java.Interop;

namespace Sample
{
    [Register("android/support/multidex/MultiDexApplication", DoNotGenerateAcw = true)]
    public class MultiDexApplication : Application
    {
        internal static readonly JniPeerMembers _members =
        new XAPeerMembers("android/support/multidex/MultiDexApplication", typeof (MultiDexApplication));

        internal static IntPtr java_class_handle;

        private static IntPtr id_ctor;

        [Register(".ctor", "()V", "", DoNotGenerateAcw = true)]
        public MultiDexApplication()
        : base(IntPtr.Zero, JniHandleOwnership.DoNotTransfer)
        {
            if (Handle != IntPtr.Zero)
                return;

            try
            {
                if (GetType() != typeof (MultiDexApplication))
                {
                    SetHandle(
                        JNIEnv.StartCreateInstance(GetType(), "()V"),
                        JniHandleOwnership.TransferLocalRef);
                        JNIEnv.FinishCreateInstance(Handle, "()V");
                    return;
                }

                if (id_ctor == IntPtr.Zero)
                    id_ctor = JNIEnv.GetMethodID(class_ref, "<init>", "()V");
                SetHandle(
                    JNIEnv.StartCreateInstance(class_ref, id_ctor),
                    JniHandleOwnership.TransferLocalRef);
                JNIEnv.FinishCreateInstance(Handle, class_ref, id_ctor);
            }
            finally
            {
            }
        }

        protected MultiDexApplication(IntPtr javaReference, JniHandleOwnership transfer)
            : base(javaReference, transfer)
        {
        }

        internal static IntPtr class_ref
        {
            get { return JNIEnv.FindClass("android/support/multidex/MultiDexApplication", ref java_class_handle); }
        }

        protected override IntPtr ThresholdClass
        {
            get { return class_ref; }
        }

        protected override Type ThresholdType
        {
            get { return typeof (MultiDexApplication); }
        }
    }
}

Kod kaynağı burada.

Windows için Visual Studio’da geliştirme yapıyorsanız, projenizi oluştururken class.dex dosyalarını düzgün bir şekilde oluşturmak için düzeltmeniz gereken Android SDK oluşturma araçlarında da bir hata vardır.

Android SDK klasörünüze gidin, build-tools klasörünü açın ve Android SDK derleyicilerinin numaralarını içeren klasörler olacaktır, örneğin:

C:\android-sdk\build-tools\23.0.3\

C:\android-sdk\build-tools\24.0.1\

C:\android-sdk\build-tools\25.0.2\

Bu klasörlerin her birinin içinde, mainClassesDex.bat adlı bir dosya bulunur; bu, class.dex dosyalarını oluşturmak için kullanılan bir toplu komut dosyasıdır. Her mainClassesDex.bat dosyasını bir metin düzenleyiciyle (Notepad veya Notepad++) açın ve komut dosyasında bloğu bulun ve değiştirin:

if DEFINED output goto redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder "%disableKeepAnnotated%" "%tmpJar%" "%params%"
goto afterClassReferenceListBuilder
:redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder "%disableKeepAnnotated%" "%tmpJar%" "%params%" 1>"%output%"
:afterClassReferenceListBuilder

Blok ile:

SET params=%params:'=%  
if DEFINED output goto redirect  
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder %disableKeepAnnotated% "%tmpJar%" %params%  
goto afterClassReferenceListBuilder  
:redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder %disableKeepAnnotated% "%tmpJar%" %params% 1>"%output%"  
:afterClassReferenceListBuilder

Kaynak burada.

Değişikliklerden sonra her mainClassesDex.bat dosyasını metin düzenleyicisine kaydedin.

Yukarıdaki adımlardan sonra Xamarin.Android uygulamanızı MultiDex ile başarıyla oluşturabilmelisiniz.

Xamarin.Android APK’nızda ProGuard’ı Etkinleştirme #

ProGuard, APK’nizin Java kodunu optimize etmek ve gizlemek ve ayrıca kullanılmayan sınıfları kaldırmak için oluşturma sürecinde kullanılan bir araçtır. ProGuard kullanıldığında ortaya çıkan APK daha küçük bir boyuta sahip olacak ve tersine mühendislik (decompilation) yapmak daha zor olacaktır.

ProGuard, Xamarin.Android uygulamalarında da kullanılabilir ve ayrıca APK dosya boyutunu küçültür ve Java kodunu gizler. Ancak ProGuard karartmasının yalnızca Java kodu için geçerli olduğunu unutmayın. .NET kodunu gizlemek için geliştirici Dotfuscator veya benzeri araçları kullanmalıdır.

Xamarin.Android uygulamanızda ProGuard nasıl kullanılır #

İlk olarak, Xamarin.Android uygulamanızda ProGuard’ı etkinleştirmek için, aşağıdaki yazdırma ekranında olduğu gibi projenizin Özellikler -> Android Seçenekleri -> Paketleme -> ProGuard’ı Etkinleştir’e gidin:

Bu, uygulamanızı oluştururken ProGuard’ı etkinleştirir.

Xamarin.Android, varsayılan olarak, “obj/Debug/proguard” veya “obj/Release/proguard” klasörlerinde, “proguard_project_primary.cfg”, “proguard_project_references.cfg” dosyalarında bulunabilen ProGuard için kendi yapılandırmalarını ayarlar. ’ ve ‘proguard_xamarin.cfg’. Üç dosya, ProGuard için yapılandırmalar olarak birleştirilir ve oluşturma sırasında Xamarin tarafından otomatik olarak oluşturulur.

Geliştirici ProGuard seçeneklerini daha fazla özelleştirmek isterse, projenin kökünde ‘proguard.cfg’ adlı bir dosya oluşturabilir (uzantı .cfg olduğu sürece diğer adlar da geçerlidir) ve Oluşturma Eylemini şu şekilde ayarlayabilir: ProguardConfiguration, aşağıdaki resimdeki gibi:

Dosyaya, “-dontwarn”, “-keep class” ve diğerleri gibi özel ProGuard seçenekleri eklenebilir.

Önemli

Şu anda olduğu gibi (Nisan/2017), genellikle indirilen Android SDK’sı, Java 1.8 kullanılarak uygulama oluşturulurken hatalara neden olabilecek eski bir ProGuard sürümüne sahiptir. İnşa ederken, Hata Listesi aşağıdaki mesajı gösterir:

Error
Can't read [C:\Program Files (x86)\Reference Assemblies\Microsoft\Framework\MonoAndroid\v7.0\mono.android.jar] 
(Can't process class [android/app/ActivityTracker.class]  (Unsupported class version number [52.0] (maximum 51.0, Java 1.7))) [CREATEMULTIDEXMAINDEXCLASSLIST]

Kaynak burada.

Bu sorunu çözmek için ProGuard’ın en son sürümünü (buradan) indirmeli ve .zip dosyasının içeriğini android-sdk\tools\proguard\ klasörüne kopyalamalısınız. Bu, ProGuard’ı güncelleyecektir ve oluşturma süreci sorunsuz çalışmalıdır.

Bundan sonra, Xamarin.Android uygulamanızı ProGuard ile başarıyla oluşturabilmelisiniz.

ProGuard ve Linker ile ilgili “Gizemli” hatalar #

Harika bir uygulama yaptınız ve Debug’da iyi sonuçlarla test ettiniz. Her şey iyi çalışıyordu!

Ancak daha sonra uygulamanızı yayına hazırlamaya karar verdiniz. MultiDex, ProGuard ve Linker’ı kurdunuz ve ardından çalışmayı durdurdu.

Bu eğitim, gizemli hatalara neden olabilecek ProGuard ve Linker ile ilgili yaygın sorunları bulmanıza yardımcı olmayı amaçlamaktadır.

Xamarin.Linker’ı Anlamak #

Xamarin.Linker, kullanılmayan kodu ve sınıfları .NET kodunuzdan (Java kodu değil) kaldıran oluşturma sürecindeki bir araçtır. Projenizin Özellikler -> Android Seçenekleri -> Bağlayıcı’da, seçeneklerle Bağlama bir seçim kutusu olacaktır:

Yok: Hiçbir kod kaldırılmaz.

Yalnızca Sdk Derlemeleri: Bu seçenek, Xamarin.Linker’ın yalnızca Xamarin kitaplıklarında kullanılmayan kodu kontrol etmesini sağlar. Bu seçenek güvenlidir.

Sdk ve Kullanıcı Derlemeleri: Bu seçenek, Xamarin.Linker’ın Xamarin kitaplıklarında ve proje kodunda (PCL’ler, Xamarin bileşenleri ve NuGet paketleri dahil) kullanılmayan kodu denetlemesini sağlar. Bu seçenek her zaman güvenli değildir!

Sdk ve Kullanıcı Derlemeleri seçeneğini kullanırken, Xamarin.Linker, aslında çok fazla kullanıldığında kodun bölümlerinin kullanılmadığını düşünebilir! Bu, bazı kitaplıkların düzgün çalışmayı durdurmasına ve uygulamanızda hatalara neden olabilir.

Xamarin.Linker’ın kodu kaldırmaması için 3 seçenek vardır:

  1. Bağlama seçeneğini Hiçbiri veya Yalnızca Sdk Derlemeleri olarak ayarlama;
  2. Montajları bağlamayı atlayın;
  3. Koru özniteliğini kullanma.

3 için örnek: Koru özelliğini kullanma:

Xamarin.Linker, projenizin çekirdeğindeki model sınıflarından çoğunlukla kod kullanılmayan kod olarak algılar.

Bağlantı işlemi sırasında sınıfın korunmasını sağlamak için Preserve özniteliğini kullanabilirsiniz.

İlk olarak, projenizin çekirdeğinde PreserveAttribute.cs adlı bir sınıf oluşturun, aşağıdaki kodu ekleyin ve ad alanını projenizin ad alanıyla değiştirin:

PreserveAttribute.cs:

namespace My_App_Core.Models
{
    public sealed class PreserveAttribute : System.Attribute
    {
        public bool AllMembers;
        public bool Conditional;
    }
}

Projenizin çekirdeğinin her bir model sınıfına aşağıdaki örnekte olduğu gibi Preserve niteliğini ekleyin:

Ülke.cs:

using System;
using System.Collections.Generic;

namespace My_App_Core.Models
{
    [Preserve(AllMembers = true)]
    public class Country
    {
        public String name { get; set; }
        public String ISOcode { get; set; }

        [Preserve(AllMembers = true)]
        public Country(String name, String ISOCode)
        {
            this.name = name;
            this.ISOCode = ISOCode;
        }
    }
}

Bundan sonra, bağlama işlemi korunan kodu artık kaldırmayacaktır.

ProGuard’ı Anlamak #

ProGuard, kullanılmayan kodu ve sınıfları Java kodunuzdan kaldıran oluşturma sürecindeki bir araçtır. Ayrıca kodu karıştırır ve optimize eder.

Ancak, ProGuard bazen kullanılmamış olarak algıladığı kodu, kullanılmadığı halde kaldırabilir. Bundan kaçınmak için geliştiricinin uygulamada hata ayıklaması (Android Cihaz Monitörü’nde ve Visual Studio Hata Ayıklama’da) ve hangi sınıfın kaldırıldığını algılaması ve ardından sınıfı korumak için ProGuard yapılandırma dosyasını yapılandırması gerekir.

Örnek

Aşağıdaki örnekte, ProGuard, AXML düzen dosyalarında kullanılan, ancak kodda kullanılmadığı algılanan iki sınıfı (Android.Support.V7.Widget.FitWindowsLinearLayout ve Android.Support.Design.Widget.AppBarLayout) kaldırdı. Kaldırma işlemi, etkinlik düzenini oluştururken Java kodunda ClassNotFoundException’a neden oldu:

Bu hatayı düzeltmek için projenin ProGuard yapılandırma dosyasına aşağıdaki satırları eklemek gerekiyordu:

ProGuard Uyarıları

ProGuard, projenizi oluşturduktan sonra bazen Hata Listesinde uyarılar gösterir. Uygulamanızın iyi olup olmadığı konusunda bir soru sorsalar da, özellikle uygulamanız başarıyla oluşturulduysa, uyarılarının tümü sorun olduğunu göstermez.

Buna bir örnek, Picasso kitaplığını kullanırken: ProGuard’ı kullanırken, ‘okio.Okio: başvurulan sınıf bulamıyor (…)’ veya ‘kaynak yazamıyor’ gibi uyarılar gösterebilir. [META-INF/MANIFEST.MF] (Yinelenen zip girişi [okhttp.jar:META-INF/MANIFEST.MF]) (…)`, ancak uygulama kurulur ve kitaplık sorunsuz çalışır.
