Bağımlılık Enjeksiyonu (DI) ve Kontrolün Tersine Çevirilmesi (IoC)

  • Java yapılandırması aracılığıyla bir bağımlılığı otomatik olarak bağlama
  • XML yapılandırması aracılığıyla bir bağımlılığı otomatik kablolama
  • XML yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme
  • Java yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme

Bu sayfada

  • Java yapılandırması aracılığıyla bir bağımlılığı otomatik olarak bağlama
  • XML yapılandırması aracılığıyla bir bağımlılığı otomatik kablolama
  • XML yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme
  • Java yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme

Büyük yazılım uygulamalarının kaynak kodu tipik olarak birden çok birim halinde düzenlenir. Bir birimin tanımı normalde kullanılan programlama diline göre değişir. Örneğin, prosedürel bir programlama dilinde (C gibi) yazılan kod, “fonksiyonlar” veya “prosedürler” olarak düzenlenir. Benzer şekilde, nesne yönelimli bir programlama dilinde (Java, Scala ve C# gibi) kodlar “sınıflar”, “arayüzler” vb. şeklinde düzenlenir. Bu kod organizasyonu birimleri, genel yazılım uygulamasını oluşturan ayrı birimler olarak düşünülebilir.

Uygulamaların birden fazla birimi olduğunda, bir birimin işlevselliğini tamamlamak için diğerlerini kullanması gerektiğinde bu birimler arasındaki karşılıklı bağımlılıklar ortaya çıkar. Bağımlı birimler “tüketiciler” ve bağımlı oldukları birimler de belirli işlevlerin “sağlayıcıları” olarak düşünülebilir.

En kolay programlama yaklaşımı, tüketicilerin uygulamanın genel yürütülmesinde hangi sağlayıcıların somutlaştırılacağına, kullanılacağına ve yok edileceğine karar vererek bir yazılım uygulamasının akışını tam olarak kontrol etmesidir. Tüketicilerin, tüketiciler için “bağımlılık” olan yürütme akışı sırasında sağlayıcılar üzerinde tam kontrole sahip oldukları söylenir. Sağlayıcıların kendi bağımlılıkları olması durumunda, tüketicilerin, sağlayıcıların nasıl başlatılacağı (ve serbest bırakılacağı) konusunda endişelenmesi gerekebilir, bu da yazılımdaki birim sayısı arttıkça kontrol akışını daha da karmaşık hale getirir. Bu yaklaşım aynı zamanda birimler arasındaki bağlantıyı artırarak, yazılımın diğer parçalarını bozma endişesi olmadan birimleri tek tek değiştirmeyi giderek zorlaştırır.

Kontrolün Tersine Çevirilmesi (IoC), birim keşfi, somutlaştırma ve yok etme gibi kontrol akışı faaliyetlerinin tüketicilerden ve sağlayıcılardan bağımsız bir çerçeveye dış kaynak sağlanmasını savunan bir tasarım ilkesidir. . IoC’nin arkasındaki temel ilke, tüketicileri ve sağlayıcıları birbirinden ayırmak, yazılım birimlerini bağımlılıklarını keşfetme, başlatma ve temizleme endişesinden kurtarmak ve birimlerin kendi işlevlerine odaklanmasına izin vermektir. Bu ayrıştırma, yazılımın genişletilebilir ve sürdürülebilir olmasına yardımcı olur.

Bağımlılık enjeksiyonu, bağımlılık örneklerinin (sağlayıcılar) yerine bir yazılım birimine (tüketici) enjekte edildiği kontrol ilkesinin tersine çevrilmesinin uygulanmasına yönelik tekniklerden biridir. tüketicinin bunları bulması ve somutlaştırması gerekiyor.

Spring çerçevesi, merkezinde Spring tarafından yönetilen çekirdeklerin diğer Spring tarafından yönetilen çekirdeklere bağımlılık olarak enjekte edilmesini sağlayan bir bağımlılık enjeksiyon modülü içerir.

Java yapılandırması aracılığıyla bir bağımlılığı otomatik olarak bağlama #

Java yapılandırması aracılığıyla yapıcı enjeksiyonu, aşağıdakiler gibi otomatik kablolamayı da kullanabilir:

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo(Bar bar) { return new Foo(bar); }
}

XML yapılandırması aracılığıyla bir bağımlılığı otomatik kablolama #

Spring çerçevesinin bileşen tarama özelliği kullanılırken bağımlılıklar otomatik olarak bağlanabilir. Otomatik kablolamanın çalışması için aşağıdaki XML yapılandırması yapılmalıdır:

<context:annotation-config/>
<context:component-scan base-package="[base package]"/>

burada, “temel paket”, Spring’in bileşen taraması yapması gereken tam nitelikli Java paketidir.

Yapıcı enjeksiyonu

Bağımlılıklar, sınıf kurucusu aracılığıyla aşağıdaki gibi enjekte edilebilir:

@Component
class Bar { ... }

@Component
class Foo {
  private Bar bar;

  @Autowired
  public Foo(Bar bar) { this.bar = bar; }
}

Burada, “@Autowired”, Yay’a özel bir açıklamadır. Spring ayrıca, diğer Java tabanlı bağımlılık ekleme çerçevelerine uygulama taşınabilirliği sağlamak için JSR-299‘yi de destekler. Bu, “@Autowired” ifadesinin “@Inject” ile şu şekilde değiştirilmesine izin verir:

@Component
class Foo {
  private Bar bar;

  @Inject
  public Foo(Bar bar) { this.bar = bar; }
}

Özellik enjeksiyonu

Bağımlılıklar, aşağıdaki gibi ayarlayıcı yöntemler kullanılarak da enjekte edilebilir:

@Component
class Foo {
  private Bar bar;

  @Autowired
  public void setBar(Bar bar) { this.bar = bar; }
}

Alan enjeksiyonu

Otomatik kablolama ayrıca sınıf örnekleri içindeki alanların doğrudan aşağıdaki gibi başlatılmasına izin verir:

@Component
class Foo {
  @Autowired
  private Bar bar;
}

Spring 4.1+ sürümleri için, isteğe bağlı bağımlılıklar için Optional kullanabilirsiniz.

@Component
class Foo {

    @Autowired
    private Optional<Bar> bar;
}

Aynı yaklaşım, yapıcı DI için de kullanılabilir.

@Component
class Foo {
    private Optional<Bar> bar;

    @Autowired
    Foo(Optional<Bar> bar) {
        this.bar = bar;
    }
}

XML yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme #

Aşağıdaki Java sınıflarını göz önünde bulundurun:

class Foo {
  private Bar bar;

  public void foo() {
    bar.baz();
  }
}

Görüldüğü gibi, ‘foo’ yönteminin başarılı bir şekilde çalışması için ‘Foo’ sınıfının başka bir ‘Bar’ sınıfı örneğinde ‘baz’ yöntemini çağırması gerekir. “Foo”, “Bar” örneği olmadan düzgün çalışamayacağından, “Bar"ın “Foo"ya bağımlılık olduğu söylenir.

Yapıcı enjeksiyonu

Spring tarafından yönetilen fasulyeleri tanımlamak için Spring çerçevesi için XML yapılandırması kullanılırken, “Foo” türünde bir fasulye aşağıdaki gibi yapılandırılabilir:

<bean class="Foo">
  <constructor-arg>
    <bean class="Bar" />
  </constructor-arg>
</bean>

veya alternatif olarak (daha ayrıntılı):

<bean id="bar" class="bar" />

<bean class="Foo">
  <constructor-arg ref="bar" />
</bean>

Her iki durumda da, Spring çerçevesi önce bir ‘Bar’ örneği oluşturur ve bunu bir ‘Foo’ örneğine ’enjekte eder’. Bu örnek, “Foo” sınıfının bir “Bar” örneğini parametre olarak alabilen bir kurucuya sahip olduğunu varsayar, yani:

class Foo {
  private Bar bar;

  public Foo(Bar bar) { this.bar = bar; }
}

Bu stil, yapıcı enjeksiyonu olarak bilinir, çünkü bağımlılık (‘Bar’ örneği) sınıf yapıcısı aracılığıyla enjekte edilir.

Özellik enjeksiyonu

‘Bar’ bağımlılığını ‘Foo’ya eklemek için başka bir seçenek de şudur:

<bean class="Foo">
  <property name="bar">
    <bean class="Bar" />
  </property>
</bean>

veya alternatif olarak (daha ayrıntılı):

<bean id="bar" class="bar" />

<bean class="Foo">
  <property name="bar" ref="bar" />
</bean>

Bu, “Foo” sınıfının aşağıdaki gibi bir “Bar” örneğini kabul eden bir ayarlayıcı yöntemine sahip olmasını gerektirir:

class Foo {
  private Bar bar;

  public void setBar(Bar bar) { this.bar = bar; }
}

Java yapılandırması aracılığıyla manuel olarak bir bağımlılık enjekte etme #

Yukarıda XML konfigürasyonu ile gösterilen aynı örnekler Java konfigürasyonu ile aşağıdaki gibi yeniden yazılabilir.

Yapıcı enjeksiyonu

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() { return new Foo(bar()); }
}

Özellik enjeksiyonu

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() {
    Foo foo = new Foo();
    foo.setBar(bar());

    return foo;
  }
}
