Spring JSR 303 Fasulye Doğrulaması

  • JSR303 Springs örneklerinde açıklama tabanlı doğrulamalar
  • Spring JSR 303 Validation - Hata mesajlarını özelleştirin
  • @İç içe POJO’ları doğrulamak için geçerli kullanım

Bu sayfada

  • JSR303 Springs örneklerinde açıklama tabanlı doğrulamalar
  • Spring JSR 303 Validation - Hata mesajlarını özelleştirin
  • @İç içe POJO’ları doğrulamak için geçerli kullanım

Spring, JSR303 fasulye doğrulama desteğine sahiptir. Bunu, giriş fasulye doğrulaması yapmak için kullanabiliriz. JSR303 kullanarak doğrulama mantığını iş mantığından ayırın.

JSR303 Springs örneklerinde açıklama tabanlı doğrulamalar #

Herhangi bir JSR 303 uygulamasını sınıf yolunuza ekleyin. Kullanılan popüler olan, Hazırda Bekletme’den Hazırda Bekletme doğrulayıcıdır.

<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-validator</artifactId>
    <version>4.2.0.Final</version>
</dependency>

Diyelim ki sistemde kullanıcı oluşturmak için bir dinlenme API’si var.

@RequestMapping(value="/registeruser", method=RequestMethod.POST)
public String registerUser(User user);

Giriş json örneği aşağıdaki gibi görünecektir

{"username" : "abc@abc.com", "password" : "password1", "password2":"password1"}

kullanıcı.java

public class User {

    private String username;
    private String password;
    private String password2;

    getXXX and setXXX

}

JSR 303 doğrulamalarını User Class üzerinde aşağıdaki gibi tanımlayabiliriz.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String username;
    
    @NotEmpty
    private String password;
    
    @NotEmpty
    private String password2;

}

Ayrıca password ve password2(confirm password) gibi bir iş doğrulayıcıya ihtiyacımız olabilir, bunun için aşağıdaki gibi özel bir doğrulayıcı ekleyebiliriz. Veri alanına açıklama eklemek için özel bir açıklama yazın.

@Target({ ElementType.FIELD })
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PasswordValidator.class)
public @interface GoodPassword {
    String message() default "Passwords wont match.";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

Doğrulama mantığını uygulamak için bir Doğrulayıcı sınıfı yazın.

public class PastValidator implements ConstraintValidator<GoodPassword, User> {
    @Override
    public void initialize(GoodPassword annotation) {}
    
    @Override
    public boolean isValid(User user, ConstraintValidatorContext context) {
        return user.getPassword().equals(user.getPassword2());
    }
}

Bu doğrulamayı Kullanıcı Sınıfına ekleme

@GoodPassword
public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String username;
    
    @NotEmpty
    private String password;
    
    @NotEmpty
    private String password2;
}

@Valid, İlkbaharda doğrulamayı tetikler. BindingResult, doğrulamadan sonra hataların listesi olan yay tarafından enjekte edilen bir nesnedir.

public String registerUser(@Valid User user, BindingResult result);

JSR 303 ek açıklaması, özel mesajlar sağlamak için kullanılabilecek mesaj özelliklerine sahiptir.

@GoodPassword
public class User {

    @NotEmpty(message="Username Cant be empty")
    @Size(min=5, message="Username cant be les than 5 chars")
    @Email(message="Should be in email format")
    private String username;
    
    @NotEmpty(message="Password cant be empty")
    private String password;
    
    @NotEmpty(message="Password2 cant be empty")
    private String password2;

}

Spring JSR 303 Validation - Hata mesajlarını özelleştirin #

Doğrulama açıklamaları olan basit bir sınıfımız olduğunu varsayalım.

public class UserDTO {
    @NotEmpty
    private String name;

    @Min(18)
    private int age;

//getters/setters
}

UserDTO geçerliliğini kontrol etmek için bir kontrolör.

@RestController
public class ValidationController {

    @RequestMapping(value = "/validate", method = RequestMethod.POST)
    public ResponseEntity<String> check(@Valid @RequestBody UserDTO userDTO,
           BindingResult bindingResult) {
        return new ResponseEntity<>("ok" , HttpStatus.OK);
    }
}

Ve bir test.

@Test
public void testValid() throws Exception {
    TestRestTemplate template = new TestRestTemplate();
    String url = base + contextPath + "/validate";
    Map<String, Object> params = new HashMap<>();
    params.put("name", "");
    params.put("age", "10");

    MultiValueMap<String, String> headers = new LinkedMultiValueMap<>();
    headers.add("Content-Type", "application/json");

    HttpEntity<Map<String, Object>> request = new HttpEntity<>(params, headers);
    String res = template.postForObject(url, request, String.class);

    assertThat(res, equalTo("ok"));
}

Hem ad hem de yaş geçersiz, bu nedenle BindingResult’da iki doğrulama hatası var. Her birinin kod dizisi vardır.

Min kontrolü için kodlar

0 = "Min.userDTO.age"
1 = "Min.age"
2 = "Min.int"
3 = "Min"

Ve NotEmpty kontrolü için

0 = "NotEmpty.userDTO.name"
1 = "NotEmpty.name"
2 = "NotEmpty.java.lang.String"
3 = "NotEmpty"

Varsayılan mesajları değiştirmek için bir custom.properties dosyası ekleyelim.

@SpringBootApplication
@Configuration
public class DemoApplication {

    @Bean(name = "messageSource")
    public MessageSource messageSource() {
        ReloadableResourceBundleMessageSource bean = new ReloadableResourceBundleMessageSource();
        bean.setBasename("classpath:custom");
        bean.setDefaultEncoding("UTF-8");
        return bean;
    }

    @Bean(name = "validator")
    public LocalValidatorFactoryBean validator() {
        LocalValidatorFactoryBean bean = new LocalValidatorFactoryBean();
        bean.setValidationMessageSource(messageSource());
        return bean;
    }

    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

Custom.properties dosyasına satırı eklersek

NotEmpty=The field must not be empty!

Hata için yeni değer gösterilir. Mesaj doğrulayıcısını çözmek için doğru mesajları bulmak için baştan başlayarak kodları inceler.

Böylece .properties dosyasında @NotEmpty notunun kullanıldığı tüm durumlar için NotEmpty anahtarını tanımladığımızda mesajımız uygulanır.

Bir mesaj tanımlarsak

Min.int=Some custom message here.

Min kontrolünü tamsayı değerlerine uyguladığımız tüm açıklamalar yeni tanımlanan mesajı kullanır.

Doğrulama hata mesajlarını yerelleştirmemiz gerekirse aynı mantık uygulanabilir.

@İç içe POJO’ları doğrulamak için geçerli kullanım #

Doğrulamamız gereken bir POJO sınıfı Kullanıcımız olduğunu varsayalım.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String email;
}

ve kullanıcı örneğini doğrulamak için bir denetleyici yöntemi

public String registerUser(@Valid User user, BindingResult result);

Kullanıcıyı, ayrıca doğrulamamız gereken yuvalanmış bir POJO Adresi ile genişletelim.

public class Address {

    @NotEmpty
    @Size(min=2, max=3)
    private String countryCode;
}

Yuvalanmış POJO’ların doğrulamasını çalıştırmak için adres alanına @Valid notu eklemeniz yeterlidir.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String email;

    @Valid
    private Address address;
}
