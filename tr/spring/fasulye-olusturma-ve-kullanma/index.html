Belirli bir türdeki tüm çekirdeklerin otomatik olarak bağlanması #

Aynı arabirimin birden çok uygulamanız varsa, Spring hepsini bir koleksiyon nesnesine otomatik olarak bağlayabilir. Doğrulayıcı kalıbı kullanan bir örnek kullanacağım1

Foo Sınıfı:

public class Foo {
     private String name;
     private String emailAddress;
     private String errorMessage;
     /** Getters & Setters omitted **/
}

Arayüz:

public interface FooValidator {
    public Foo validate(Foo foo);
}

Ad Doğrulayıcı Sınıfı:

@Component(value="FooNameValidator")
public class FooNameValidator implements FooValidator {
    @Override
    public Foo validate(Foo foo) {
        //Validation logic goes here.
    }
}

E-posta Doğrulayıcı Sınıfı:

@Component(value="FooEmailValidator")
public class FooEmailValidator implements FooValidator {
    @Override
    public Foo validate(Foo foo) {
        //Different validation logic goes here.
    }
}

Artık bu doğrulayıcıları tek tek veya birlikte bir sınıfa otomatik olarak bağlayabilirsiniz.

Arayüz:

public interface FooService {
    public void handleFoo(Foo foo);
}

Sınıf:

@Service
public class FooServiceImpl implements FooService {
    /** Autowire all classes implementing FooValidator interface**/
    @Autowired
    private List<FooValidator> allValidators;

    @Override
    public void handleFoo(Foo foo) {
       /**You can use all instances from the list**/
       for(FooValidator validator : allValidators) {
           foo = validator.validate(foo);               
       }
    }   
}

Spring IoC kapsayıcısında bir arabirimin birden fazla uygulamasına sahipseniz ve `@Qualifier’ ek açıklamasıyla hangisini kullanmak istediğinizi belirtmezseniz, Spring’in başlatmaya çalışırken bir istisna atacağını belirtmekte fayda var. hangi örneği kullanacağını bilemez.

1: Bu tür basit doğrulamaları yapmanın doğru yolu bu değil. Bu, otomatik kablolama hakkında basit bir örnektir. Çok daha kolay bir doğrulama yöntemi hakkında fikir edinmek istiyorsanız, Spring’in ek açıklamalarla doğrulamayı nasıl yaptığına bakın.

Temel açıklama otomatik kablolama #

Arayüz:

public interface FooService {
    public int doSomething();
}

Sınıf:

@Service
public class FooServiceImpl implements FooService {
    @Override
    public int doSomething() {
        //Do some stuff here
        return 0;
    }
}

Spring’in bu sınıfı otomatik olarak bağlayabilmesi için bir sınıfın bir arabirim uygulaması gerektiğine dikkat edilmelidir. Spring’in yükleme süresi dokumasını kullanarak bağımsız sınıfları otomatik olarak bağlamasına izin veren bir yöntem vardır, ancak bu, bu örnek için kapsam dışındadır.

Bu bene, Spring IoC konteyneri tarafından @Autowired açıklamasını kullanarak başlatılan herhangi bir sınıfta erişebilirsiniz.

Kullanım:

@Autowired([required=true])

@Autowired notu önce türe göre otomatik kablolamayı deneyecek ve ardından belirsizlik durumunda fasulye adına geri dönecektir.

Bu açıklama birkaç farklı şekilde uygulanabilir.

Yapıcı enjeksiyonu:

public class BarClass() {
    private FooService fooService         

    @Autowired
    public BarClass(FooService fooService) {
        this.fooService = fooService;
    }
}

Alan enjeksiyonu:

public class BarClass() {
    @Autowired
    private FooService fooService;
}

Setter enjeksiyonu:

public class BarClass() {
    private FooService fooService;

    @Autowired
    public void setFooService(FooService fooService) {
        this.fooService = fooService;
    }
}

Fasulye Bildirme #

Bean bildirmek için, bir yönteme “@Bean” notuyla açıklama ekleyin veya “@Component” notuyla bir sınıfa not ekleyin ("@Service", “@Repository”, “@Controller” ek açıklamaları da kullanılabilir).

JavaConfig böyle bir yöntemle karşılaştığında, bu yöntemi uygulayacak ve dönüş değerini bir BeanFactory içinde bir bean olarak kaydedecektir. Varsayılan olarak, fasulye adı, yöntem adınınki olacaktır.

Fasulyeyi üç yoldan birini kullanarak oluşturabiliriz:

  1. Java Tabanlı Konfigürasyon Kullanımı: Konfigürasyon dosyasında @bean ek açıklamasını kullanarak bean bildirmemiz gerekiyor

     @Configuration
 public class AppConfig {
     @Bean
     public TransferService transferService() {
         return new TransferServiceImpl();
     }
 }

  2. XML tabanlı yapılandırma kullanma: XML tabanlı yapılandırma için, uygulama yapılandırması XML’de beyan bean oluşturmamız gerekir, yani.

     <beans>
     <bean name="transferService" class="com.acme.TransferServiceImpl"/>
 </beans>

  3. Annotation-Driven Component: Annotation-driven component için bean olarak ilan etmek istediğimiz sınıfa @Component notunu eklememiz gerekiyor.

     @Component("transferService")
 public class TransferServiceImpl implements TransferService {
     ...
 }

Artık “transferService” adlı üç çekirdeğin tümü “BeanFactory” veya “ApplicationContext” içinde mevcuttur.

Dinamik fasulye örneklemesi için FactoryBean’i kullanma #

Hangi fasulyelerin enjekte edileceğine dinamik olarak karar vermek için FactoryBeans kullanabiliriz. Bunlar, kap için fasulye örnekleri sağlayan fabrika yöntemi modelini uygulayan sınıflardır. Spring tarafından tanınırlar ve fasulyenin bir fabrikadan geldiğini bilmeye gerek kalmadan şeffaf bir şekilde kullanılabilirler. Örneğin:

public class ExampleFactoryBean extends AbstractFactoryBean<String> {
    // This method determines the type of the bean for autowiring purposes
    @Override
    public Class<?> getObjectType() {
        return String.class;
    }

    // this factory method produces the actual bean
    @Override
    protected String createInstance() throws Exception {
        // The thing you return can be defined dynamically,
        // that is read from a file, database, network or just
        // simply randomly generated if you wish.
        return "Something from factory";
    }
}

Yapılandırma:

@Configuration
public class ExampleConfig {
    @Bean
    public FactoryBean<String> fromFactory() {
        return new ExampleFactoryBean();
    }
}

Fasulyeyi almak:

AbstractApplicationContext context = new AnnotationConfigApplicationContext(ExampleConfig.class);
String exampleString = (String) context.getBean("fromFactory");

Gerçek FactoryBean’i almak için, fasulyenin adından önce ve işareti önekini kullanın:

FactoryBean<String> bean = (FactoryBean<String>) context.getBean("&fromFactory");

Kapsamı “prototip” olarak değiştirmek için “isSingleton” yöntemini geçersiz kılmak için yalnızca “prototip” veya “singleton” kapsamlarını kullanabileceğinizi lütfen unutmayın:

public class ExampleFactoryBean extends AbstractFactoryBean<String> {
    @Override
    public boolean isSingleton() {
        return false;
    }

    // other methods omitted for readability reasons
}

Kapsam belirlemenin, fabrika çekirdeğinin kendisine değil, oluşturulan gerçek örneklere atıfta bulunduğunu unutmayın.

Prototip kapsamlı fasulyeleri singletonlara enjekte edin #

Kap, tek bir çekirdek oluşturur ve işbirlikçileri yalnızca bir kez enjekte eder. Bir singleton bean’in prototip kapsamlı bir ortak çalışanı olduğunda bu istenen davranış değildir, çünkü prototip kapsamlı fasulye erişimci aracılığıyla her erişildiğinde enjekte edilmelidir.

Bu sorunun birkaç çözümü var:

  1. Arama yöntemi enjeksiyonunu kullanın
  2. ‘javax.inject.Provider’ aracılığıyla prototip kapsamlı bir fasulye alın
  3. “org.springframework.beans.factory.ObjectFactory” aracılığıyla prototip kapsamlı bir fasulye alın (#2’ye eşdeğer, ancak Spring’e özgü sınıfla)
  4. ‘ApplicationContextAware’ arayüzünü uygulayarak bir singleton fasulye konteynerini haberdar edin

#3 ve #4 yaklaşımları, bir uygulamayı Spring çerçevesine güçlü bir şekilde bağladıkları için genellikle önerilmez. Bu nedenle, bu örnekte ele alınmamışlardır.

XML yapılandırması ve soyut bir yöntem aracılığıyla arama yöntemi enjeksiyonu

Java Sınıfları

public class Window {
}

public abstract class WindowGenerator {

    public Window generateWindow() {
        Window window = createNewWindow(); // new instance for each call
        ... 
    }

    protected abstract Window createNewWindow(); // lookup method
}

XML

<bean id="window" class="somepackage.Window" scope="prototype" lazy-init="true"/>

<bean id="windowGenerator" class="somepackage.WindowGenerator">
    <lookup-method name="createNewWindow" bean="window"/>
</bean>

Java yapılandırması ve @Component aracılığıyla arama yöntemi enjeksiyonu

Java Sınıfları

public class Window {
}

@Component
public class WindowGenerator {

    public Window generateWindow() {
        Window window = createNewWindow(); // new instance for each call
        ...
    }

    @Lookup
    protected Window createNewWindow() {
        throw new UnsupportedOperationException();
    }
}

Java yapılandırması

@Configuration
@ComponentScan("somepackage") // package where WindowGenerator is located
public class MyConfiguration {

    @Bean
    @Lazy
    @Scope(scopeName = ConfigurableBeanFactory.SCOPE_PROTOTYPE)
    public Window window() {
        return new Window();
    }
}

Java yapılandırması aracılığıyla manuel arama yöntemi enjeksiyonu

Java Sınıfları

public class Window {
}

public abstract class WindowGenerator {

    public Window generateWindow() {
        Window window = createNewWindow(); // new instance for each call
        ...
    }

    protected abstract Window createNewWindow(); // lookup method
}

Java yapılandırması

@Configuration
public class MyConfiguration {

    @Bean
    @Lazy
    @Scope(scopeName = ConfigurableBeanFactory.SCOPE_PROTOTYPE)
    public Window window() {
        return new Window();
    }

    @Bean
    public WindowGenerator windowGenerator(){
        return new WindowGenerator() {
            @Override
            protected Window createNewWindow(){
                return window();
            }
        };
    }
}

Prototip kapsamlı bir fasulyenin javax.inject.Provider aracılığıyla singleton’a enjeksiyonu

Java sınıfları

public class Window {
}

public class WindowGenerator {

    private final Provider<Window> windowProvider;

    public WindowGenerator(final Provider<Window> windowProvider) {
        this.windowProvider = windowProvider;
    }

    public Window generateWindow() {
        Window window = windowProvider.get(); // new instance for each call   
        ...
    }
}

XML

<bean id="window" class="somepackage.Window" scope="prototype" lazy-init="true"/>

<bean id="windowGenerator" class="somepackage.WindowGenerator">
    <constructor-arg>
        <bean class="org.springframework.beans.factory.config.ProviderCreatingFactoryBean">
            <property name="targetBeanName" value="window"/>
        </bean>
    </constructor-arg>
</bean>

Aynı yaklaşımlar diğer kapsamlar için de kullanılabilir (örneğin, istek kapsamındaki bir fasulyeyi singleton’a enjekte etmek için).

@Qualifier ile belirli fasulye örneklerini otomatik kablolama #

Aynı arabirimin birden çok uygulamanız varsa, Spring’in hangisini bir sınıfa otomatik olarak bağlaması gerektiğini bilmesi gerekir. Bu örnekte bir Doğrulayıcı kalıbı kullanacağım.1

Foo Sınıfı:

public class Foo {
     private String name;
     private String emailAddress;
     private String errorMessage;
     /** Getters & Setters omitted **/
}

Arayüz:

public interface FooValidator {
    public Foo validate(Foo foo);
}

Ad Doğrulayıcı Sınıfı:

@Component(value="FooNameValidator")
public class FooNameValidator implements FooValidator {
    @Override
    public Foo validate(Foo foo) {
        //Validation logic goes here.
    }
}

E-posta Doğrulayıcı Sınıfı:

@Component(value="FooEmailValidator")
public class FooEmailValidator implements FooValidator {
    @Override
    public Foo validate(Foo foo) {
        //Different validation logic goes here.
    }
}

Artık bu doğrulayıcıları ayrı ayrı bir sınıfa otomatik olarak bağlayabilirsiniz.

Arayüz:

public interface FooService {
    public void handleFoo(Foo foo);
}

Sınıf:

@Service
public class FooServiceImpl implements FooService {
    /** Autowire validators individually **/
    @Autowired
    /* 
     * Notice how the String value here matches the value 
     * on the @Component annotation? That's how Spring knows which 
     * instance to autowire.
     */
    @Qualifier("FooNameValidator")
    private FooValidator nameValidator;

    @Autowired
    @Qualifier("FooEmailValidator")
    private FooValidator emailValidator;

    @Override
    public void handleFoo(Foo foo) {
        /**You can use just one instance if you need**/
        foo = nameValidator.validate(foo);
    }   
}

Spring IoC kapsayıcısında bir arabirimin birden fazla uygulamasına sahipseniz ve `@Qualifier’ ek açıklamasıyla hangisini kullanmak istediğinizi belirtmezseniz, Spring’in başlatmaya çalışırken bir istisna atacağını belirtmekte fayda var. hangi örneği kullanacağını bilemez.

1: Bu tür basit doğrulamaları yapmanın doğru yolu bu değil. Bu, otomatik kablolama hakkında basit bir örnektir. Çok daha kolay bir doğrulama yöntemi hakkında fikir edinmek istiyorsanız, Spring’in ek açıklamalarla doğrulamayı nasıl yaptığına bakın.

Genel tür parametreleri kullanarak belirli sınıf örneklerini otomatik kablolama #

Genel bir tür parametresine sahip bir arabiriminiz varsa, Spring bunu yalnızca belirttiğiniz bir tür parametresini uygulayan uygulamaları otomatik olarak bağlamak için kullanabilir.

Arayüz:

public interface GenericValidator<T> {
    public T validate(T object);
}

Foo Doğrulayıcı Sınıfı:

@Component
public class FooValidator implements GenericValidator<Foo> {
    @Override
    public Foo validate(Foo foo) {
        //Logic here to validate foo objects.
    }
}

Bar Doğrulayıcı Sınıfı:

@Component
public class BarValidator implements GenericValidator<Bar> {
    @Override
    public Bar validate(Bar bar) {
        //Bar validation logic here
    }
}

Artık, hangi örneğin otomatik olarak bağlanacağına karar vermek için tür parametrelerini kullanarak bu doğrulayıcıları otomatik olarak bağlayabilirsiniz.

Arayüz:

public interface FooService {
    public void handleFoo(Foo foo);
}

Sınıf:

@Service
public class FooServiceImpl implements FooService {
    /** Autowire Foo Validator **/
    @Autowired
    private GenericValidator<Foo> fooValidator;

    @Override
    public void handleFoo(Foo foo) {
        foo = fooValidator.validate(foo);
    }   
}
