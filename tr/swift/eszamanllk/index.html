Sözdizimi #

  • Hızlı 3.0

  • DispatchQueue.main // Ana kuyruğu al

  • DispatchQueue(etiket: “seri kuyruğum”, nitelikler: [.serial, .qosBackground]) // Kendi özel seri kuyruğunuzu oluşturun

  • DispatchQueue.global(öznitelikler: [.qosDefault]) // Global eşzamanlı kuyruklardan birine erişin

  • DispatchQueue.main.async { … } // Ana iş parçacığına eşzamansız olarak bir görev gönder

  • DispatchQueue.main.sync { … } // Ana iş parçacığına eşzamanlı olarak bir görev gönder

  • DispatchQueue.main.asyncAfter(son tarih: .now() + 3) { … } // x saniye sonra yürütülecek ana iş parçacığına eşzamansız olarak bir görev gönderin

  • Hızlı < 3.0

  • sevk_get_main_queue() // Ana iş parçacığında çalışan ana kuyruğu alın -dispatch_get_global_queue(dispatch_queue_priority_t, 0) // Belirtilen önceliğe sahip global sırayı al sevk_queue_priority_t -dispatch_async(dispatch_queue_t) { () -> Void in … } // Belirtilen Dispatch_queue_t üzerinde eşzamansız olarak bir görev gönder

  • sevk_sync(dispatch_queue_t) { () -> Void in … } // Belirtilen sevk_kuyruğunda eşzamanlı olarak bir görev gönder -dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(nanosaniye))),dispatch_queue_t, { … }); // Nanosaniyeden sonra belirtilen sevk_kuyruğu_t üzerinde bir görev gönder

Grand Central Dispatch (GCD) kuyruğu alma #

Grand Central Dispatch, “Sevk Kuyrukları” konsepti üzerinde çalışır. Gönderme kuyruğu, belirlediğiniz görevleri, geçtikleri sırayla yürütür. Üç tür gönderme kuyruğu vardır:

  • Seri Gönderim Kuyrukları (diğer adıyla özel gönderme kuyrukları), sırayla bir seferde bir görevi yürütür. Bir kaynağa erişimi senkronize etmek için sıklıkla kullanılırlar.
  • Eşzamanlı Gönderim Kuyrukları (diğer adıyla genel gönderme kuyrukları), bir veya daha fazla görevi aynı anda yürütür.
  • Ana Gönderim Kuyruğu, görevleri ana iş parçacığında yürütür.

Ana kuyruğa erişmek için:

let mainQueue = DispatchQueue.main
son sürüm 
let mainQueue = dispatch_get_main_queue()
son sürüm

Sistem, değişen önceliklere sahip (uygulamanız için küresel) eşzamanlı küresel gönderme kuyrukları sağlar. Swift 3’te ‘DispatchQueue’ sınıfını kullanarak bu sıralara erişebilirsiniz:

let globalConcurrentQueue = DispatchQueue.global(qos: .default)

eşittir

let globalConcurrentQueue = DispatchQueue.global()
son sürüm 
let globalConcurrentQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0)
son sürüm

iOS 8 veya sonraki sürümlerde, iletilebilecek olası hizmet kalitesi değerleri “.userInteractive”, “.userInitiated”, “.default”, “.utility” ve “.background"dur. Bunlar, “DISPATCH_QUEUE_PRIORITY_” sabitlerinin yerini alır.

Ayrıca değişen önceliklere sahip kendi sıralarınızı da oluşturabilirsiniz:

let myConcurrentQueue = DispatchQueue(label: "my-concurrent-queue", qos: .userInitiated, attributes: [.concurrent], autoreleaseFrequency: .workItem, target: nil)
let mySerialQueue = DispatchQueue(label: "my-serial-queue", qos: .background, attributes: [], autoreleaseFrequency: .workItem, target: nil)
son sürüm 
let myConcurrentQueue = dispatch_queue_create("my-concurrent-queue", DISPATCH_QUEUE_CONCURRENT)
let mySerialQueue = dispatch_queue_create("my-serial-queue", DISPATCH_QUEUE_SERIAL)
son sürüm

Swift 3’te, bu başlatıcı ile oluşturulan kuyruklar varsayılan olarak seridir ve otomatik serbest bırakma sıklığı için “.workItem” iletmek, her iş öğesi için bir otomatik serbest bırakma havuzunun oluşturulmasını ve boşaltılmasını sağlar. Ayrıca “.never” seçeneği de vardır; bu, kendi otomatik yayın havuzlarınızı kendiniz yöneteceğiniz anlamına gelir veya ayarı ortamdan devralan “.inherit”. Çoğu durumda, aşırı özelleştirme durumları dışında muhtemelen “.never” kullanmazsınız.

Eşzamanlı Döngüler #

GCD, bir döngü gerçekleştirmek için mekanizma sağlar, bu sayede döngüler birbirine göre eşzamanlı olarak gerçekleşir. Bu, bir dizi hesaplama açısından pahalı hesaplamalar yaparken çok kullanışlıdır.

Bu döngüyü düşünün:

for index in 0 ..< iterations {
    // Do something computationally expensive here
}

Bu hesaplamaları “concurrentPerform” (Swift 3’te) veya “dispatch_apply” (Swift 2’de) kullanarak aynı anda gerçekleştirebilirsiniz:

DispatchQueue.concurrentPerform(iterations: iterations) { index in
    // Do something computationally expensive here
}
son sürüm 
dispatch_apply(iterations, queue) { index in
    // Do something computationally expensive here
}
son sürüm

Döngü kapatma, “0"dan “yinelemeler"e kadar olan her bir “indeks” için çağrılır. Bu yinelemeler, birbirlerine göre eşzamanlı olarak yürütülür ve bu nedenle, çalıştıkları sıra garanti edilmez. Herhangi bir zamanda eşzamanlı olarak gerçekleşen gerçek yineleme sayısı, genellikle söz konusu cihazın yeteneklerine göre belirlenir (örneğin, cihazın kaç çekirdeği vardır).

Birkaç özel husus:

  • ‘concurrentPerform’/‘dispatch_apply’ döngüleri birbirine göre eşzamanlı olarak çalıştırabilir, ancak bu, onu çağırdığınız iş parçacığına göre eşzamanlı olarak gerçekleşir. Bu nedenle, bunu ana iş parçacığından çağırmayın, çünkü bu, döngü tamamlanana kadar o iş parçacığını engeller.

  • Bu döngüler birbirine göre eşzamanlı olarak gerçekleştiğinden, sonuçların iş parçacığı güvenliğini sağlamaktan siz sorumlusunuz. Örneğin, bazı sözlükleri bu hesaplama açısından pahalı hesaplamaların sonuçlarıyla güncellerseniz, bu güncellemeleri kendiniz senkronize ettiğinizden emin olun.

  • Eşzamanlı döngülerin çalıştırılmasıyla ilişkili bazı ek yükler olduğunu unutmayın. Bu nedenle, döngü içinde gerçekleştirilen hesaplamalar, hesaplama açısından yeterince yoğun değilse, eşzamanlı döngüler kullanılarak elde edilen herhangi bir performansın, bu eşzamanlı iş parçacıklarının tümünün senkronize edilmesiyle ilişkili ek yük tarafından tamamen dengelenmese bile azaltılabileceğini görebilirsiniz.

    So, you are responsible determining the correct amount of work to be performed in each iteration of the loop. If the calculations are too simple, you may employ “striding” to include more work per loop. For example, rather than doing a concurrent loop with 1 million trivial calculations, you may do 100 iterations in your loop, doing 10,000 calculations per loop. That way there is enough work being performed on each thread, so the overhead associated with managing these concurrent loops becomes less significant.

Grand Central Dispatch (GCD) kuyruğunda görevleri çalıştırma #

Görevleri bir gönderme kuyruğunda çalıştırmak için “sync”, “async” ve “after” yöntemlerini kullanın.

Bir görevi eşzamansız olarak bir kuyruğa göndermek için:

let queue = DispatchQueue(label: "myQueueName")

queue.async {
    //do something
    
    DispatchQueue.main.async {
        //this will be called in main thread
        //any UI updates should be placed here
    }
}
// ... code here will execute immediately, before the task finished

Bir görevi bir kuyruğa eşzamanlı olarak göndermek için:

queue.sync {
    // Do some task
}
// ... code here will not execute until the task is finished

Belirli sayıda saniye sonra bir görevi kuyruğa göndermek için:

queue.asyncAfter(deadline: .now() + 3) {
    //this will be executed in a background-thread after 3 seconds
}
// ... code here will execute immediately, before the task finished

NOT:** Kullanıcı arayüzündeki tüm güncellemeler ana başlıktan çağrılmalıdır! UI güncellemelerinin kodunu DispatchQueue.main.async { ... } içine koyduğunuzdan emin olun.

son sürüm

Sıra türleri:

let mainQueue = dispatch_get_main_queue()
let highQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_HIGH, 0)
let backgroundQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0)

Bir görevi eşzamansız olarak bir kuyruğa göndermek için:

dispatch_async(queue) {
    // Your code run run asynchronously. Code is queued and executed 
    // at some point in the future.
}
// Code after the async block will execute immediately

Bir görevi bir kuyruğa eşzamanlı olarak göndermek için:

dispatch_sync(queue) {
    // Your sync code
}
// Code after the sync block will wait until the sync task finished

Bir görevi belirli bir zaman aralığından sonra göndermek için (saniyeyi nanosaniyeye dönüştürmek için NSEC_PER_SEC kullanın):

dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(2.5 * Double(NSEC_PER_SEC))), dispatch_get_main_queue()) {
    // Code to be performed in 2.5 seconds here
}

Bir görevi eşzamansız olarak yürütmek ve daha sonra kullanıcı arayüzünü güncellemek için:

dispatch_async(queue) {
    // Your time consuming code here
    dispatch_async(dispatch_get_main_queue()) {
        // Update the UI code 
    }
}

NOT:** Kullanıcı arayüzündeki tüm güncellemeler ana başlıktan çağrılmalıdır! UI güncellemelerinin kodunu dispatch_async(dispatch_get_main_queue()) { ... } içine koyduğunuzdan emin olun.

son sürüm

Bir OperationQueue’da Görevleri Çalıştırma #

Bir ‘OperationQueue’yu yürütülmeyi bekleyen bir dizi görev olarak düşünebilirsiniz. GCD’deki sevk kuyruklarının aksine, işlem kuyrukları FIFO (ilk giren ilk çıkar) değildir. Bunun yerine, izin vermek için yeterli sistem kaynağı olduğu sürece, yürütülmeye hazır oldukları anda görevleri yürütürler.

Ana ‘OperationQueue’yu alın:

let mainQueue = OperationQueue.main
son sürüm

Özel bir “OperationQueue” oluşturun:

let queue = OperationQueue()
queue.name = "My Queue"
queue.qualityOfService = .default
son sürüm

Hizmet Kalitesi, işin önemini veya kullanıcının görevden elde edilen anlık sonuçlara ne kadar güvenebileceğini belirtir.

Bir “OperationQueue"ya bir “Operation” ekleyin:

// An instance of some Operation subclass
let operation = BlockOperation {
    // perform task here
}

queue.addOperation(operation)
son sürüm

Bir “OperationQueue"ya bir blok ekleyin:

myQueue.addOperation {
    // some task
}
son sürüm

Bir “OperationQueue"ya birden çok “Operation” ekleyin:

let operations = [Operation]()
// Fill array with Operations

myQueue.addOperation(operations)
son sürüm

Kuyruk içinde aynı anda kaç tane “İşlem” yürütülebileceğini ayarlayın:

myQueue.maxConcurrentOperationCount = 3 // 3 operations may execute at once

// Sets number of concurrent operations based on current system conditions
myQueue.maxConcurrentOperationCount = NSOperationQueueDefaultMaxConcurrentOperationCount

Bir kuyruğu askıya almak, kuyruğun mevcut, başlatılmamış herhangi bir işlemin veya sıraya eklenen herhangi bir yeni işlemin yürütülmesini başlatmasını engeller. Bu kuyruğu sürdürmenin yolu, “isSuspended"ı tekrar “false” olarak ayarlamaktır:

myQueue.isSuspended = true

// Re-enable execution
myQueue.isSuspended = false
son sürüm

Bir “OperationQueue"nun askıya alınması, halihazırda yürütülmekte olan işlemleri durdurmaz veya iptal etmez. Genel sıralar veya ana sıra değil, yalnızca oluşturduğunuz bir sıra askıya alınmaya çalışılmalıdır.

Üst Düzey İşlemler Oluşturma #

Foundation çerçevesi, bir kuyrukta yürütülebilecek işin bir bölümünü kapsayan üst düzey bir nesneyi temsil eden ‘İşlem’ türünü sağlar. Kuyruk yalnızca bu işlemlerin performansını koordine etmekle kalmaz, aynı zamanda işlemler arasında bağımlılıklar kurabilir, iptal edilebilir işlemler oluşturabilir, işlem kuyruğu tarafından kullanılan eşzamanlılık derecesini sınırlayabilirsiniz, vb.

İşlem, tüm bağımlılıklarının yürütülmesi bittiğinde yürütülmeye hazır hale gelir. “isReady” özelliği daha sonra “true” olarak değişir.

Eşzamanlı olmayan basit bir “İşlem” alt sınıfı oluşturun:

class MyOperation: Operation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}
son sürüm 
class MyOperation: NSOperation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}
son sürüm

Bir “OperationQueue"ya bir işlem ekleyin:

myQueue.addOperation(operation)
son sürüm

Bu, işlemi sıra üzerinde eşzamanlı olarak yürütecektir.

Bir “İşlem” üzerindeki bağımlılıkları yönetin.

Bağımlılıklar, “İşlem"in yürütülmeye hazır kabul edilmesinden önce bir kuyrukta yürütülmesi gereken diğer “İşlemleri” tanımlar.

operation2.addDependency(operation1)

operation2.removeDependency(operation1)
son sürüm

Kuyruksuz bir “İşlem” çalıştırın:

   operation.start()
son sürüm

Bağımlılıklar yoksayılacak. Bu eşzamanlı bir işlemse, “start” yöntemi işi arka plan kuyruklarına boşaltırsa, görev yine de eşzamanlı olarak yürütülebilir.

Eşzamanlı İşlemler.

Bir “İşlemin” gerçekleştireceği görevin kendisi eşzamansızsa (ör. “URLSession” veri görevi), “İşlemi” eşzamanlı bir işlem olarak uygulamanız gerekir. Bu durumda, “isAsynchronous” uygulamanız “true” döndürmelidir, genellikle bazı kurulumları gerçekleştiren “start” yöntemine sahip olursunuz, ardından görevi gerçekten yürüten “ana” yöntemini çağırırsınız.

Eşzamansız bir “İşlem” başlatıldığında, “isExecuting”, “isFinished” yöntemlerini ve KVO’yu uygulamanız gerekir. Böylece, yürütme başladığında, “isExecuting” özelliği “true” olarak değişir. Bir “İşlem” görevini bitirdiğinde, “isExecuting” “yanlış” olarak ve “isFinished” ise “doğru” olarak ayarlanır. İşlem iptal edilirse, hem ‘isCancelled’ hem de ‘isFinished’, ’true’ olarak değişir. Bu özelliklerin tümü anahtar/değer gözlemlenebilir.

Bir “İşlemi” iptal edin.

“İptal"i çağırmak, “isCancelled” özelliğini “true” olarak değiştirir. İptal işlemine kendi ‘İşlem’ alt sınıfınızdan yanıt vermek için, ‘main’ içinde ‘isCancelled’ değerini en azından periyodik olarak kontrol etmeli ve uygun şekilde yanıt vermelisiniz.
