Yatay kodun yeniden kullanımını kolaylaştıran özellikler #

Diyelim ki günlüğe kaydetme için bir arayüzümüz var:

interface Logger {
    function log($message);
}

Şimdi, “Logger” arabiriminin iki somut uygulamasına sahip olduğumuzu varsayalım: “FileLogger” ve “ConsoleLogger”.

class FileLogger implements Logger {
    public function log($message) {
        // Append log message to some file
    }
}

class ConsoleLogger implements Logger {
    public function log($message) {
        // Log message to the console
    }
}

Şimdi, günlük tutma görevlerini de gerçekleştirmek istediğiniz başka bir ‘Foo’ sınıfı tanımlarsanız, şöyle bir şey yapabilirsiniz:

class Foo implements Logger {
    private $logger;

    public function setLogger(Logger $logger) {
        $this->logger = $logger;
    }

    public function log($message) {
        if ($this->logger) {
            $this->logger->log($message);
        }
    }
}

“Foo” artık bir “Logger"dır, ancak işlevselliği “setLogger()” aracılığıyla kendisine iletilen “Logger” uygulamasına bağlıdır. Eğer şimdi Bar sınıfının da bu kayıt mekanizmasına sahip olmasını istiyorsak, bu mantık parçasını Bar sınıfında çoğaltmamız gerekecek.

Kodu kopyalamak yerine bir özellik tanımlanabilir:

trait LoggableTrait {
    protected $logger;

    public function setLogger(Logger $logger) {
        $this->logger = $logger;
    }

    public function log($message) {
        if ($this->logger) {
            $this->logger->log($message);
        }
    }
}

Artık bir özellikteki mantığı tanımladığımıza göre, mantığı “Foo” ve “Bar” sınıflarına eklemek için bu özelliği kullanabiliriz:

class Foo {
    use LoggableTrait;
}

class Bar {
    use LoggableTrait;
}

Ve örneğin, “Foo” sınıfını şu şekilde kullanabiliriz:

$foo = new Foo();
$foo->setLogger( new FileLogger() );

//note how we use the trait as a 'proxy' to call the Logger's log method on the Foo instance
$foo->log('my beautiful message');

Özellik Nedir? #

PHP yalnızca tek bir kalıtsallığa izin verir. Başka bir deyişle, bir sınıf yalnızca bir başka sınıfı ‘genişletebilir’. Peki ya ana sınıfa ait olmayan bir şeyi eklemeniz gerekirse? PHP 5.4’ten önce yaratıcı olmanız gerekirdi, ancak 5.4’te Özellikler tanıtıldı. Özellikler, bir sınıfın bir bölümünü temel olarak ana sınıfınıza “kopyalayıp yapıştırmanıza” izin verir.

trait Talk {
    /** @var string */
    public $phrase = 'Well Wilbur...';
    public function speak() {
         echo $this->phrase;
    }
}

class MrEd extends Horse {
    use Talk;
    public function __construct() {
         $this->speak();
    }

    public function setPhrase($phrase) {
         $this->phrase = $phrase;
    }
}

Yani burada zaten ‘At’ı genişleten ‘MrEd’ var. Ama tüm atlar “Konuşmaz”, bu yüzden bunun için bir Özelliğimiz var. Bunun ne yaptığını not edelim

İlk olarak, Özelliğimizi tanımlarız. Otomatik yükleme ve Ad Alanları ile kullanabiliriz (ayrıca bkz. <https://www.wikiod.com/tr/php/ad-alanlar#Bir ad alanında bir sınıfa veya işleve başvurma>). Sonra onu use anahtar kelimesi ile MrEd sınıfımıza dahil ediyoruz.

“MrEd"in, “Konuş” işlevlerini ve değişkenlerini tanımlamadan kullanmaya başladığını fark edeceksiniz. kopyala ve yapıştır hakkında söylediklerimizi hatırlıyor musunuz? Bu işlevler ve değişkenlerin tümü, sanki bu sınıf onları tanımlamış gibi, şimdi sınıf içinde tanımlanmıştır.

Özellikler, değişkenleri ve işlevleri tanımlayabilmeniz açısından en çok [Soyut sınıflar](https://www.wikiod.com/tr/php/snflar-ve-nesneler#Soyut Sınıflar) ile ilişkilidir. Ayrıca bir Özelliği doğrudan somutlaştıramazsınız (yani, yeni Özellik()). Özellikler, bir sınıfı, bir Soyut sınıf veya bir Arayüz gibi bir işlevi örtük olarak tanımlamaya zorlayamaz. Özellikler yalnızca açık tanımlar içindir (istediğiniz kadar Arayüz ‘uygulayabileceğiniz’ için, bkz. <https://www.wikiod.com/tr/php/snflar-ve-nesneler#Arayüzler arayüzler>).

Bir Özelliği ne zaman kullanmalıyım? #

Bir Özelliği değerlendirirken yapmanız gereken ilk şey, kendinize şu önemli soruyu sormaktır.

Kodumu yeniden yapılandırarak bir Özelliği kullanmaktan kaçınabilir miyim?

Çoğu zaman cevap Evet olacaktır. Özellikler, tek kalıtımın neden olduğu uç durumlardır. Özellikleri kötüye kullanma veya aşırı kullanma eğilimi yüksek olabilir. Ancak bir Özelliğin kodunuz için başka bir kaynak sunduğunu düşünün, bu da başka bir karmaşıklık katmanı olduğu anlamına gelir. Buradaki örnekte sadece 3 sınıfla ilgileniyoruz. Ancak Özellikler, artık bundan çok daha fazlasıyla uğraşabileceğiniz anlamına gelir. Her Özellik için, sınıfınızla başa çıkmak çok daha zor hale gelir, çünkü şimdi neyi tanımladığını (ve potansiyel olarak bir çarpışmanın nerede olduğunu öğrenmek için her bir Özelliğe başvurmanız gerekir, bkz. <https://www.wikiod.com/tr/docs/php 999/özellikler/7271/çatışma çözümü>). İdeal olarak, kodunuzda mümkün olduğunca az Özellik bulundurmalısınız.

Çatışma çözümü #

Birkaç özelliği tek bir sınıfta kullanmaya çalışmak, çelişkili yöntemler içeren sorunlara neden olabilir. Bu tür çakışmaları manuel olarak çözmeniz gerekir.

Örneğin, bu hiyerarşiyi oluşturalım:

trait MeowTrait {
    public function say() {
        print "Meow \n";
    }
}

trait WoofTrait {
    public function say() {
        print "Woof \n";
    }
}

abstract class UnMuteAnimals {
    abstract function say();
}

class Dog extends UnMuteAnimals {
    use WoofTrait;
}

class Cat extends UnMuteAnimals {
    use MeowTrait;
}

Şimdi aşağıdaki sınıfı oluşturmaya çalışalım:

class TalkingParrot extends UnMuteAnimals {
    use MeowTrait, WoofTrait;
}

php yorumlayıcısı önemli bir hata döndürür:

Önemli hata: TalkingParrot’ta diğer özellik yöntemleriyle çakışmalar olduğu için özellik yöntemi uygulanmadı.

Bu çatışmayı çözmek için şunu yapabiliriz:

  • başka bir özelliğin yöntemi yerine bir özelliğin yöntemini kullanmak için “yerine” anahtar sözcüğünü kullanın

  • WoofTrait::say as sayAsDog; gibi bir yapıya sahip yöntem için bir takma ad oluşturun

    class TalkingParrotV2 extends UnMuteAnimals { use MeowTrait, WoofTrait { MeowTrait::say insteadof WoofTrait; WoofTrait::say as sayAsDog; } }

    $talkingParrot = new TalkingParrotV2(); $talkingParrot->say(); $talkingParrot->sayAsDog();

Bir Singleton’ı Nitelikler Kullanarak Uygulamak #

Sorumluluk reddi: Bu örnek hiçbir şekilde tekillerin kullanımını savunmaz. Singleton’lar çok dikkatli kullanılmalıdır.

PHP’de bir singleton uygulamanın oldukça standart bir yolu vardır:

public class Singleton {
    private $instance;

    private function __construct() { };

    public function getInstance() {
        if (!self::$instance) {
            // new self() is 'basically' equivalent to new Singleton()
            self::$instance = new self();
        }

        return self::$instance;
    }

    // Prevent cloning of the instance
    protected function __clone() { }

    // Prevent serialization of the instance
    protected function __sleep() { }

    // Prevent deserialization of the instance
    protected function __wakeup() { }
}

Kod tekrarını önlemek için bu davranışı bir özelliğe ayıklamak iyi bir fikirdir.

trait SingletonTrait {
    private $instance;

    protected function __construct() { };

    public function getInstance() {
        if (!self::$instance) {
            // new self() will refer to the class that uses the trait
            self::$instance = new self();
        }

        return self::$instance;
    }

    protected function __clone() { }
    protected function __sleep() { }
    protected function __wakeup() { }
}

Artık bir singleton olarak işlev görmek isteyen herhangi bir sınıf, özelliği basitçe kullanabilir:

class MyClass {
    use SingletonTrait;
}

// Error! Constructor is not publicly accessible
$myClass = new MyClass();

$myClass = MyClass::getInstance();

// All calls below will fail due to method visibility
$myClassCopy = clone $myClass; // Error!
$serializedMyClass = serialize($myClass); // Error!
$myClass = deserialize($serializedMyclass); // Error!

Artık bir singleton’u seri hale getirmek imkansız olsa da, seriyi kaldırma yöntemine izin vermemek de yararlıdır.

Çoklu Özellik Kullanımı #

trait Hello {
    public function sayHello() {
        echo 'Hello ';
    }
}

trait World {
    public function sayWorld() {
        echo 'World';
    }
}

class MyHelloWorld {
    use Hello, World;
    public function sayExclamationMark() {
        echo '!';
    }
}

$o = new MyHelloWorld();
$o->sayHello();
$o->sayWorld();
$o->sayExclamationMark();

Yöntem Görünürlüğünü Değiştirme #

trait HelloWorld {
    public function sayHello() {
        echo 'Hello World!';
    }
}

// Change visibility of sayHello
class MyClass1 {
    use HelloWorld { sayHello as protected; }
}

// Alias method with changed visibility
// sayHello visibility not changed
class MyClass2 {
    use HelloWorld { sayHello as private myPrivateHello; }
}

Bu örneği çalıştırırken:

(new MyClass1())->sayHello();
// Fatal error: Uncaught Error: Call to protected method MyClass1::sayHello()

(new MyClass2())->myPrivateHello();
// Fatal error: Uncaught Error: Call to private method MyClass2::myPrivateHello()

(new MyClass2())->sayHello();
// Hello World!

Bu nedenle, ‘MyClass2’deki son örnekte, ‘özellik HelloWorld’den gelen orijinal un-aliased yönteminin olduğu gibi erişilebilir kaldığını unutmayın.

Sınıfları temiz tutma özellikleri #

Zamanla, sınıflarımız giderek daha fazla arayüz uygulayabilir. Bu arayüzler çok sayıda metoda sahip olduğunda sınıfımızdaki metodların toplam sayısı çok fazla olacaktır.

Örneğin, iki arabirimimiz ve bunları uygulayan bir sınıfımız olduğunu varsayalım:

interface Printable {
    public function print();   
    //other interface methods...
}

interface Cacheable {
    //interface methods
}

class Article implements Cachable, Printable {  
    //here we must implement all the interface methods
    public function print(){ {
        /* code to print the article */ 
    }
}

‘Makale’ sınıfının içindeki tüm arabirim yöntemlerini uygulamak yerine, bu arabirimleri uygulamak için ayrı Özellikler kullanabilir, sınıfı daha küçük tutabilir ve arabirim uygulamasının kodunu sınıftan ayırabiliriz.

Örneğin, “Yazdırılabilir” arayüzünü uygulamak için şu özelliği oluşturabiliriz:

trait PrintableArticle {
    //implements here the interface methods
    public function print() {
        /* code to print the article */ 
    }
}

ve sınıfın özelliği kullanmasını sağlayın:

class Article implements Cachable, Printable {
    use PrintableArticle;
    use CacheableArticle; 
}

Birincil faydalar, arabirim uygulama yöntemlerimizin sınıfın geri kalanından ayrılacak ve o belirli nesne türü için arabirimi uygulama sorumluluğuna sahip olan bir özellikte depolanacak olmasıdır.
