Bağımlılık Enjeksiyonu (DI), _“şeyleri iletmek” için süslü bir terimdir. Bunun gerçekten anlamı, bir nesnenin bağımlılıklarını, nesne içinde nesne oluşturma üzerine oluşturmak yerine yapıcı ve/veya ayarlayıcılar aracılığıyla iletmektir.

Bağımlılık Enjeksiyonu, inşaat ve enjeksiyonu otomatikleştiren Bağımlılık Enjeksiyon Konteynerlerine de atıfta bulunabilir.

Yapıcı Enjeksiyonu #

Nesneler genellikle diğer nesnelere bağlı olacaktır. Bağımlılığı yapıcıda oluşturmak yerine, bağımlılık yapıcıya parametre olarak aktarılmalıdır. Bu, nesneler arasında sıkı bir bağlantı olmamasını sağlar ve sınıf somutlaştırmaya olan bağımlılığın değiştirilmesini sağlar. Bunun, bağımlılıkları açık hale getirerek kodun daha kolay okunmasını sağlamanın yanı sıra, bağımlılıklar daha kolay değiştirilebildiği ve daha kolay alay edilebildiği için testi daha basit hale getirme gibi bir dizi avantajı vardır.

Aşağıdaki örnekte, “Bileşen”, “Logger” örneğine bağlı olacaktır, ancak bir tane oluşturmaz. Bunun yerine yapıcıya argüman olarak iletilmesini gerektirir.

interface Logger {
    public function log(string $message);
}

class Component {
    private $logger;

    public function __construct(Logger $logger) {
        $this->logger = $logger;
    }
}

Bağımlılık ekleme olmadan, kod muhtemelen şuna benzer:

class Component {
    private $logger;

    public function __construct() {
        $this->logger = new FooLogger();
    }
}

Yapıcıda yeni nesneler oluşturmak için “new” kullanılması, bağımlılık enjeksiyonunun kullanılmadığını (veya eksik kullanıldığını) ve kodun sıkı bir şekilde birleştiğini gösterir. Ayrıca, kodun eksik test edildiğinin veya program durumu hakkında yanlış varsayımlarda bulunan kırılgan testlere sahip olabileceğinin bir işaretidir.

Bunun yerine bağımlılık enjeksiyonunu kullandığımız yukarıdaki örnekte, gerektiğinde kolayca farklı bir Kaydediciye geçebiliriz. Örneğin, farklı bir konuma günlüğe kaydeden veya farklı bir günlüğe kaydetme biçimi kullanan veya bir dosya yerine veritabanına giriş yapan bir Logger uygulaması kullanabiliriz.

Setter Enjeksiyonu #

Bağımlılıklar ayarlayıcılar tarafından da enjekte edilebilir.

interface Logger {
    public function log($message);
}

class Component {
    private $logger;
    private $databaseConnection;

    public function __construct(DatabaseConnection $databaseConnection) {
        $this->databaseConnection = $databaseConnection;
    }

    public function setLogger(Logger $logger) {
        $this->logger = $logger;
    }

    public function core() {
        $this->logSave();    
        return $this->databaseConnection->save($this);
    }

    public function logSave() {
         if ($this->logger) {
            $this->logger->log('saving');
        }
    }
}

Bu, özellikle sınıfın temel işlevselliği çalışmaya bağımlılığa dayanmadığında ilginçtir.

Burada, tek gerekli bağımlılık ‘DatabaseConnection’dır, yani yapıcıdadır. ‘Logger’ bağımlılığı isteğe bağlıdır ve bu nedenle yapıcının bir parçası olması gerekmez, bu da sınıfın kullanımını kolaylaştırır.

Setter enjeksiyonunu kullanırken, işlevselliği değiştirmek yerine genişletmenin daha iyi olduğunu unutmayın. Bir bağımlılık ayarlarken, bağımlılığın bir noktada değişmeyeceğini doğrulayan hiçbir şey yoktur ve bu da beklenmedik sonuçlara yol açabilir. Örneğin, önce bir ‘FileLogger’ ve ardından bir ‘MailLogger’ ayarlanabilir. Bu, kapsüllemeyi bozar ve günlükleri bulmayı zorlaştırır, çünkü bağımlılığı değiştiriyoruz.

Bunu önlemek için, setter enjeksiyonuna şu şekilde bir bağımlılık eklemeliyiz:

interface Logger {
    public function log($message);
}

class Component {
    private $loggers = array();
    private $databaseConnection;

    public function __construct(DatabaseConnection $databaseConnection) {
        $this->databaseConnection = $databaseConnection;
    }

    public function addLogger(Logger $logger) {
        $this->loggers[] = $logger;
    }

    public function core() {
        $this->logSave();
        return $this->databaseConnection->save($this);
    }

    public function logSave() {
        foreach ($this->loggers as $logger) {
            $logger->log('saving');
        }
    }
}

Bunun gibi, temel işlevi ne zaman kullansak, eklenmiş bir kaydedici bağımlılığı olmasa bile bozulmaz ve eklenen herhangi bir kaydedici, başka bir kaydedici eklenebilse bile kullanılacaktır. İşlevi değiştirmek yerine genişletiyoruz.

Konteyner Enjeksiyon #

Bağımlılık Ekleme Kabı (DIC) kullanımı bağlamında Bağımlılık Ekleme (DI), yapıcı enjeksiyonunun bir üst kümesi olarak görülebilir. Bir DIC, tipik olarak bir sınıf oluşturucunun yazım ipuçlarını analiz eder ve örnek yürütme için gereken bağımlılıkları etkin bir şekilde enjekte ederek ihtiyaçlarını çözer.

Tam uygulama, bu belgenin kapsamının çok ötesine geçer, ancak özünde, bir DIC, bir sınıfın imzasını kullanmaya dayanır…

namespace Documentation;

class Example
{
    private $meaning;

    public function __construct(Meaning $meaning)
    {
        $this->meaning = $meaning;
    }
}

… çoğu zaman bir [otomatik yükleme sistemine] güvenerek otomatik olarak başlatmak için (https://www.wikiod.com/tr/php/snflar-ve-nesneler#Otomatik yükleme).

// older PHP versions
$container->make('Documentation\Example');

// since PHP 5.5
$container->make(\Documentation\Example::class);

En az 5.5 sürümünde PHP kullanıyorsanız ve yukarıda gösterilen şekilde bir sınıf adı almak istiyorsanız, doğru yol ikinci yaklaşımdır. Bu şekilde, potansiyel yeniden düzenleme konusunda size büyük ölçüde yardımcı olacak modern IDE’leri kullanarak sınıfın kullanımlarını hızlı bir şekilde bulabilirsiniz. Normal dizelere güvenmek istemezsiniz.

Bu durumda, “Belgeleme\Örnek”, bir “Anlam"a ihtiyacı olduğunu bilir ve bir DIC, bir “Anlam” türünü başlatır. Somut uygulamanın, tüketen örneğe bağlı olması gerekmez.

Bunun yerine, nesne oluşturmadan önce, gerektiğinde belirli türlerin nasıl başlatılacağını bildiren kapsayıcıda kurallar belirleriz.

Bir DIC’nin yapabileceği gibi bunun bir takım avantajları vardır.

  • Ortak örnekleri paylaşın
  • Bir tür imzasını çözmek için bir fabrika sağlayın
  • Bir arayüz imzasını çözün

Belirli bir türün nasıl yönetilmesi gerektiğine ilişkin kurallar tanımlarsak, hangi türlerin paylaşıldığı, somutlaştırıldığı veya bir fabrikadan oluşturulduğu üzerinde hassas kontrol sağlayabiliriz.
