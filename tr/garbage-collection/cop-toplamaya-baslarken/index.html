Giriiş #

Bir programdaki ana giriş noktaları tarafından artık erişilemiyorsa, nesneler çöp toplama (GC) için uygun hale gelir. GC genellikle kullanıcı tarafından açıkça gerçekleştirilmez, ancak GC’nin bir nesneye artık ihtiyaç duyulmadığını bilmesini sağlamak için bir geliştirici şunları yapabilir:

Dereferans / null ata

someFunction {
     var a = 1;
     var b = 2;
     a = null; // GC can now free the memory used for variable a
     ...
} // local variable b not dereferenced but will be subject to GC when function ends

Zayıf referanslar kullanın

GC’ye sahip dillerin çoğu, bir nesneye GC için referans olarak sayılmayan zayıf referanslar oluşturmanıza izin verir. Bir nesneye yalnızca zayıf referanslar varsa ve güçlü (normal) referanslar yoksa, nesne GC için uygundur.

WeakReference wr = new WeakReference(createSomeObject());

Bu koddan sonra, nesnenin hala var olup olmadığını kontrol etmeden zayıf referansın hedefini kullanmanın tehlikeli olduğunu unutmayın. Acemi programcılar bazen şöyle bir kod kullanma hatasına düşerler:

if wr.target is not null {
    doSomeAction(wr.target);
}

GC, boş denetimden sonra ve doSomeAction yürütülmeden önce çağrılmış olabileceğinden, bu sorunlara neden olabilir. Önce şu şekilde nesneye (geçici) güçlü bir referans oluşturmak daha iyidir:

Object strongRef = wr.target;
if strongRef is not null {
    doSomeAction(strongRef);
}
strongRef = null;

Java’da ayrıntılı gc günlüğe kaydetmeyi etkinleştirme #

Normalde jvm’nin çöp toplaması (gc) kullanıcı (geliştirici/mühendis) için şeffaftır.

Kullanıcı bir bellek sızıntısıyla karşılaşmadıkça veya büyük miktarda bellek gerektiren bir uygulamaya sahip olmadıkça GC ayarı normalde gerekli değildir - her ikisi de sonunda kullanıcıyı soruna bakmaya zorlayan bellek yetersiz bir özel duruma yol açar.

İlk adım tipik olarak belleği artırmaktır (çalışma zamanında yükten mi kaynaklandığına veya uygulamanın kütüphane tabanının büyük olup olmadığına veya sınıf yükleme veya iş parçacığında bir sızıntı olup olmadığına bağlı olarak yığın veya perm-gen/meta-alanı) -işleme mekanizması). Ancak bu mümkün olmadığında, bir sonraki adım neyin yanlış gittiğini anlamaya çalışmaktır.

Belirli bir anda yalnızca anlık görüntü istenirse, jdk’nin parçası olan ‘jstat’ yardımcı programı yeterli olacaktır.

Ancak daha ayrıntılı bir anlayış için, her bir gc olayından önce ve sonra öbek anlık görüntüsünü içeren bir günlüğe sahip olmak yararlıdır. Bunun için kullanıcının, jvm başlatma parametrelerinin bir parçası olarak “-verbose:gc"yi kullanarak ve “-XX:+PrintGCDetails” ve “-XX:+PrintGCTimeStamp” bayraklarını dahil ederek ayrıntılı gc günlüğünü etkinleştirmesi gerekir.

Uygulamalarını proaktif olarak profillemek isteyenler için, jdk’nin bir parçası olan ve uygulama davranışı hakkında fikir edinebilecekleri “jvisualvm” gibi araçlar da vardır.

Aşağıda örnek bir program, gc yapılandırması ve ayrıntılı-gc günlük çıktısı verilmiştir:

package com.example.so.docs.gc.logging;

import java.util.Arrays;
import java.util.Random;

public class HelloWorld {

    public static void main(String[] args) {
        sortTest();
    }
    
    private static void sortTest() {
        System.out.println("HelloWorld");
        
        int count = 3;
        while(count-- > 0) {
            int size = 1024*1024;
            int[] numbers = new int[size];
            Random random = new Random();
            for(int i=0;i<size;i++) {
                numbers[i] = random.nextInt(size);
            }
            
            Arrays.sort(numbers);
        }
        System.out.println("Done");
        
    }
    

}
