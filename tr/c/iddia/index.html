Bir iddia, yazılım tarafından iddiayla karşılaşıldığı anda sunulan koşulun doğru olması gerektiğine dair bir yüklemdir. En yaygın olanı, yürütme zamanında doğrulanan basit iddialardır. Ancak, statik iddialar derleme zamanında kontrol edilir.

Sözdizimi #

  • iddia (ifade)
  • static_assert(ifade, mesaj)
  • _Static_assert(ifade, mesaj)

Parametreler #

parametre Ayrıntılar
ifade skaler tip ifadesi.
mesaj tanılama mesajına dahil edilecek dize değişmezi.

Hem “asset” hem de “static_assert”, “assert.h” içinde tanımlanan makrolardır.

“Assert” tanımı, standart kitaplık tarafından tanımlanmayan “NDEBUG” makrosuna bağlıdır. “NDEBUG” tanımlanmışsa, “assert” işlemsizdir:

#ifdef NDEBUG
#  define assert(condition) ((void) 0)
#else
#  define assert(condition) /* implementation defined */
#endif

Üretim derlemeleri için her zaman “NDEBUG"un kullanılmasının gerekip gerekmediği konusunda görüşler değişir.

  • Pro-camp, ‘assert’in ‘abort’ çağrılarını ve iddia mesajlarının son kullanıcılar için yararlı olmadığını, bu nedenle sonucun kullanıcı için yararlı olmadığını savunuyor. Üretim kodunu kontrol etmek için önemli koşullarınız varsa, normal ‘if/else’ koşullarını ve programı sonlandırmak için ’exit’ veya ‘quick_exit’ kullanmalısınız. ‘abort’un tersine, bunlar programın bazı temizleme işlemleri yapmasına izin verir (‘atexit’ veya ‘at_quick_exit’ ile kayıtlı işlevler aracılığıyla).
  • Con-camp, “iddia” çağrılarının üretim kodunda asla tetiklenmemesi gerektiğini savunuyor, ancak eğer öyleyse, kontrol edilen koşul, önemli ölçüde yanlış bir şey olduğu ve yürütme devam ederse programın daha kötü davranacağı anlamına geliyor. Bu nedenle, iddiaları üretim kodunda aktif hale getirmek daha iyidir çünkü ateşlenirlerse, cehennem zaten kopmuştur.
  • Diğer bir seçenek de, her zaman kontrolü gerçekleştiren ancak geliştirme (“iptal"in uygun olduğu yerlerde) ve üretim (“beklenmeyen bir dahili hata - lütfen Teknik Destek ile iletişime geçin” arasındaki) hataları farklı şekilde ele alan, evde hazırlanmış bir iddia sistemi kullanmaktır. uygun).

“static_assert”, bir anahtar kelime olan “_Static_assert” olarak genişler. Koşul derleme zamanında kontrol edilir, bu nedenle “condition” sabit bir ifade olmalıdır. Bunun geliştirme ve üretim arasında farklı şekilde ele alınmasına gerek yoktur.

Basit İddia #

Bir iddia, o kod satırına ulaşıldığında bir gerçeğin doğru olması gerektiğini iddia etmek için kullanılan bir ifadedir. İddialar, beklenen koşulların karşılanmasını sağlamak için yararlıdır. Bir iddiaya geçirilen koşul doğru olduğunda, herhangi bir işlem yapılmaz. Yanlış koşullardaki davranış, derleyici bayraklarına bağlıdır. Onaylar etkinleştirildiğinde, yanlış bir giriş, programın hemen durmasına neden olur. Devre dışı bırakıldıklarında herhangi bir işlem yapılmaz. Onaylar genellikle sürümde etkinleştirilse de, dahili ve hata ayıklama derlemelerinde onayları etkinleştirmek ve sürüm derlemelerinde bunları devre dışı bırakmak yaygın bir uygulamadır. (Sonlandırmanın hatalardan daha iyi veya daha kötü olup olmadığı programa bağlıdır.) Onaylar yalnızca dahili programlama hatalarını yakalamak için kullanılmalıdır, bu genellikle hatalı parametreler iletilmesi anlamına gelir.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int main(void)
{
    int x = -1;
    assert(x >= 0);

    printf("x = %d\n", x);   
    return 0;
}

‘NDEBUG’ tanımsız olarak olası çıktı:

a.out: main.c:9: main: Assertion `x >= 0' failed.

“NDEBUG” tanımlı olası çıktı:

x = -1

Açık veya kapalı tüm iddialarla kodunuzu kolayca derleyebilmeniz için, ‘NDEBUG’ı global olarak tanımlamak iyi bir uygulamadır. Bunu yapmanın kolay bir yolu, derleyici için bir seçenek olarak “NDEBUG"u tanımlamak veya bunu paylaşılan bir yapılandırma başlığında (örneğin “config.h”) tanımlamaktır.

Statik Onay #

Statik onaylamalar, kod derlendiğinde bir koşulun doğru olup olmadığını kontrol etmek için kullanılır. Değilse, derleyicinin bir hata mesajı vermesi ve derleme işlemini durdurması gerekir.

Statik bir iddia, çalışma zamanında değil, derleme zamanında kontrol edilen bir iddiadır. Koşul sabit bir ifade olmalıdır ve false ise derleyici hatasına neden olur. İlk bağımsız değişken, yani kontrol edilen koşul, sabit bir ifade, ikincisi ise bir dize değişmezi olmalıdır.

Assert’ten farklı olarak, “_Static_assert” bir anahtar kelimedir. Bir kolaylık makrosu “static_assert”, “<assert.h>” içinde tanımlanmıştır.

#include <assert.h>

enum {N = 5};
_Static_assert(N == 5, "N does not equal 5");
static_assert(N > 10, "N is not greater than 10");  /* compiler error */
son sürüm

C11’den önce, statik iddialar için doğrudan bir destek yoktu. Ancak, C99’da, derleme zamanı koşulu yanlışsa derleme hatasını tetikleyecek makrolarla statik onaylar öykünebilir. ‘_Static_assert’ten farklı olarak, ikinci parametrenin onunla bir değişken adı oluşturulabilmesi için uygun bir belirteç adı olması gerekir. Onay başarısız olursa, bu değişken sözdizimsel olarak yanlış bir dizi bildiriminde kullanıldığından, derleyici hatasında değişken adı görülür.

#define STATIC_MSG(msg, l) STATIC_MSG2(msg, l)
#define STATIC_MSG2(msg,l) on_line_##l##__##msg
#define STATIC_ASSERT(x, msg) extern char STATIC_MSG(msg, __LINE__) [(x)?1:-1]
 
enum { N = 5 };
STATIC_ASSERT(N == 5, N_must_equal_5);
STATIC_ASSERT(N > 5, N_must_be_greater_than_5); /* compile error */

C99’dan önce, bir blokta rastgele konumlarda değişkenler bildiremezdiniz, bu nedenle bu makroyu kullanırken son derece dikkatli olmanız ve yalnızca bir değişken bildiriminin geçerli olacağı yerlerde görünmesini sağlamanız gerekir.

son sürüm

Hata Mesajlarını Onaylayın #

Bir iddia ile birlikte bir hata mesajı görüntüleyebilen bir numara var. Normalde, böyle bir kod yazardınız

void f(void *p)
{
    assert(p != NULL);
    /* more code */
}

Onay başarısız olursa, aşağıdakine benzer bir hata mesajı görüntülenir:

Onaylama başarısız: p != NULL, dosya main.c, satır 5

Ancak, bir hata mesajı vermek için mantıksal AND (&&) da kullanabilirsiniz.

void f(void *p)
{
    assert(p != NULL && "function f: p cannot be NULL");
    /* more code */
}

Şimdi, iddia başarısız olursa, bir hata mesajı şöyle bir şey okuyacaktır.

Onaylama başarısız: p != NULL && “fonksiyon f: p NULL olamaz”, dosya main.c, satır 5

Bunun işe yaramasının nedeni, bir dize değişmezinin her zaman sıfırdan farklı (doğru) olarak değerlendirilmesidir. Boole ifadesine “&& 1” eklemenin hiçbir etkisi yoktur. Bu nedenle, && "hata mesajı" eklemenin de hiçbir etkisi yoktur, ancak derleyicinin başarısız olan tüm ifadeyi göstermesi dışında.

Ulaşılamaz Kodun Onaylanması #

Geliştirme sırasında, belirli kod yollarının kontrol akışının erişiminden engellenmesi gerektiğinde, böyle bir koşulun hatalı olduğunu belirtmek için “assert(0)” kullanabilirsiniz:

switch (color) {
    case COLOR_RED:
    case COLOR_GREEN:
    case COLOR_BLUE:
        break;

    default:
        assert(0);
}

“Assert()” makrosunun argümanı false olarak değerlendirdiğinde, makro standart hata akışına tanılama bilgilerini yazar ve ardından programı iptal eder. Bu bilgi, ‘assert()’ ifadesinin dosya ve satır numarasını içerir ve hata ayıklamada çok yardımcı olabilir. Assers, ‘NDEBUG’ makrosu tanımlanarak devre dışı bırakılabilir.

Bir hata oluştuğunda bir programı sonlandırmanın başka bir yolu da standart kitaplık işlevleri “çıkış”, “hızlı_çıkış” veya “iptal"dir. ’exit’ ve ‘quick_exit’ ortamınıza geri iletilebilecek bir argüman alır. ‘abort()’ (ve dolayısıyla ‘iddia’), programınızın gerçekten ciddi bir şekilde sonlandırılmasına neden olabilir ve aksi takdirde yürütmenin sonunda gerçekleştirilecek bazı temizleme işlemleri gerçekleştirilemeyebilir.

Assert()‘in birincil avantajı, hata ayıklama bilgilerini otomatik olarak yazdırmasıdır. Abort()‘u çağırmak, bir onaylama işlemi gibi devre dışı bırakılamaması avantajına sahiptir, ancak herhangi bir hata ayıklama bilgisinin görüntülenmesine neden olmayabilir. Bazı durumlarda, her iki yapıyı birlikte kullanmak faydalı olabilir:

if (color == COLOR_RED || color == COLOR_GREEN) {
   ...
} else if (color == COLOR_BLUE) {
   ...
} else {
   assert(0), abort();
}

Onaylar enabled olduğunda, ‘assert()’ çağrısı hata ayıklama bilgilerini yazdıracak ve programı sonlandıracaktır. Yürütme hiçbir zaman abort() çağrısına ulaşmaz. Onaylar devre dışı olduğunda, ‘assert()’ çağrısı hiçbir şey yapmaz ve ‘abort()’ çağrılır. Bu, programın her zaman bu hata koşulu için sonlandırılmasını sağlar; etkinleştirme ve devre dışı bırakma, yalnızca hata ayıklama çıktısının yazdırılıp yazdırılmadığını etkiler.

Hata ayıklama bilgisi son kullanıcılar için yararlı olmadığından ve “iptal” genellikle “çıkış” veya “hızlı çıkış” için yüklenen temizleme işleyicilerini engelleyen çok ciddi bir sonlandırma olduğundan, üretim kodunda asla böyle bir “onay” bırakmamalısınız. koşmak.

Önkoşul ve Sonkoşul #

İddia için bir kullanım durumu, önkoşul ve sonkoşuldur. Bu, değişmez ve sözleşmeye göre tasarım‘ı korumak için çok yararlı olabilir. Örneğin, bir uzunluk her zaman sıfır veya pozitiftir, bu nedenle bu fonksiyon sıfır veya pozitif bir değer döndürmelidir.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int length2 (int *a, int count)
{
    int i, result = 0;

    /* Precondition: */
    /* NULL is an invalid vector */
    assert (a != NULL);
    /* Number of dimensions can not be negative.*/ 
    assert (count >= 0);

    /* Calculation */
    for (i = 0; i < count; ++i) 
    {
        result = result + (a[i] * a[i]);
    }

    /* Postcondition: */
    /* Resulting length can not be negative. */
    assert (result >= 0);
    return result;
}

#define COUNT 3

int main (void)
{
    int a[COUNT] = {1, 2, 3};
    int *b = NULL;
    int r;
    r = length2 (a, COUNT);
    printf ("r = %i\n", r);
    r = length2 (b, COUNT);
    printf ("r = %i\n", r);
    return 0;
}
