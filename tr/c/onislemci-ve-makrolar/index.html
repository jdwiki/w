Tüm önişlemci komutları, kare (pound) sembolü “#” ile başlar. Bir C makrosu, yalnızca “#define” önişlemci yönergesi kullanılarak tanımlanan bir önişlemci komutudur. Önişleme aşaması sırasında, C önişlemcisi (C derleyicisinin bir parçası), makronun gövdesini adının göründüğü her yerde değiştirir.

Bir derleyici kodda bir makro ile karşılaştığında, basit bir dize değişimi gerçekleştirir, ek bir işlem yapılmaz. Bu nedenle, önişlemci tarafından yapılan değişiklikler C programlarının kapsamına uymaz - örneğin, bir makro tanımı bir blok içinde olmakla sınırlı değildir, bu nedenle bir blok ifadesini sona erdiren bir “}” tarafından etkilenmez.

Önişlemci, tasarım gereği tamamlanmamıştır - yalnızca önişlemci tarafından yapılamayan birkaç tür hesaplama vardır.

Genellikle derleyiciler, ön işleme aşamasından sonra derlemeyi durdurmamıza ve sonucu incelememize izin veren bir komut satırı bayrağına (veya yapılandırma ayarına) sahiptir. POSIX platformlarında bu bayrak “-E"dir. Bu nedenle, gcc’yi bu bayrakla çalıştırmak, genişletilmiş kaynağı stdout’a yazdırır:

$ gcc -E cprog.c

Genellikle önişlemci, derleyici tarafından çağrılan ayrı bir program olarak uygulanır, bu programın ortak adı ‘cpp’dir. Bir dizi önişlemci, hata ayıklama bilgilerini oluşturmak için sonraki derleme aşamaları tarafından kullanılan satır numaraları hakkındaki bilgiler gibi destekleyici bilgiler yayar. Önişlemcinin gcc’ye dayalı olması durumunda, -P seçeneği bu tür bilgileri bastırır.

$ cpp -P cprog.c

#if 0 kod bölümlerini engellemek için #

Kaldırmayı düşündüğünüz veya geçici olarak devre dışı bırakmak istediğiniz kod bölümleri varsa, bunu bir blok yorum ile yorumlayabilirsiniz.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
*/

Ancak bir blok yorum ile çevrelediğiniz kaynak kodun kaynağında blok stili yorumlar varsa, mevcut blok yorumlarının */ bitmesi yeni blok yorumunuzun geçersiz olmasına ve derleme sorunlarına neden olabilir.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;

    /* Return 5 */
    return i;
}
*/

Önceki örnekte, işlevin son iki satırı ve son ‘*/’ derleyici tarafından görülür, bu nedenle hatalarla derlenir. Daha güvenli bir yöntem, engellemek istediğiniz kodun etrafında bir “#if 0” yönergesi kullanmaktır.

#if 0
/* #if 0 evaluates to false, so everything between here and the #endif are
 * removed by the preprocessor. */
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
#endif

Bunun bir avantajı, geri dönüp kodu bulmak istediğinizde, tüm yorumlarınızı aramak yerine “#if 0” için arama yapmanın çok daha kolay olmasıdır.

Bir diğer çok önemli fayda, yorum kodunu #if 0 ile iç içe yerleştirebilmenizdir. Bu yorumlarla yapılamaz.

“#if 0"ı kullanmanın bir alternatifi, “#tanımlı” olmayacak ancak kodun neden bloke edildiğini daha açıklayıcı bir ad kullanmaktır. Örneğin, gereksiz ölü kod gibi görünen bir işlev varsa, başka bir işlev yerine getirildiğinde veya benzeri bir şey olduğunda ihtiyaç duyulan kod için #if tanımlı(POSSIBLE_DEAD_CODE) veya #if tanımlı(FUTURE_CODE_REL_020201) kullanabilirsiniz. Ardından, bu kaynağı kaldırmak veya etkinleştirmek için geri döndüğünüzde, kaynağın bu bölümlerini bulmak kolaydır.

Başlık Korumaları İçerir #

Hemen hemen her başlık dosyası include guard deyimini izlemelidir:

başlık-dosyam.h

#ifndef MY_HEADER_FILE_H
#define MY_HEADER_FILE_H

// Code body for header file

#endif

Bu, birden çok yerde #include "my-header-file.h" yaptığınızda, yinelenen işlevler, değişkenler vb. bildirimleri almamanızı sağlar. Aşağıdaki dosya hiyerarşisini hayal edin:

başlık-1.h

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

başlık-2.h

#include "header-1.h"

int myFunction2(MyStruct *value);

main.c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Bu kodun ciddi bir sorunu var: MyStruct’ın ayrıntılı içeriği iki kez tanımlandı, buna izin verilmedi. Bu, bir başlık dosyası başka bir başlık dosyası içerdiğinden, izlenmesi zor olabilecek bir derleme hatasına neden olur. Bunun yerine başlık korumalarıyla yaptıysanız:

başlık-1.h

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

başlık-2.h

#ifndef HEADER_2_H
#define HEADER_2_H

#include "header-1.h"

int myFunction2(MyStruct *value);

#endif

main.c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Bu daha sonra genişler:

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

#ifndef HEADER_2_H
#define HEADER_2_H

#ifndef HEADER_1_H // Safe, since HEADER_1_H was #define'd before.
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

int myFunction2(MyStruct *value);

#endif

int main() {
    // do something
}

Derleyici header-1.h’nin ikinci eklenmesine ulaştığında, ‘HEADER_1_H’ önceki ekleme tarafından zaten tanımlanmıştı. Ergo, aşağıdakilere kadar kaynar:

#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#define HEADER_2_H

int myFunction2(MyStruct *value);

int main() {
    // do something
}

Ve böylece derleme hatası yoktur.

Not: Başlık korumalarını adlandırmak için birden çok farklı kural vardır. Bazı insanlar onu ‘HEADER_2_H_’ olarak adlandırmayı sever, bazıları ‘MY_PROJECT_HEADER_2_H’ gibi proje adını içerir. Önemli olan, izlediğiniz kuralın, projenizdeki her dosyanın benzersiz bir başlık korumasına sahip olmasını sağlamaktır.

Yapı ayrıntıları başlığa dahil edilmemişse, bildirilen tür eksik veya opaque type olacaktır. Bu tür türler, uygulama ayrıntılarını işlevlerin kullanıcılarından gizleyerek yararlı olabilir. Birçok amaç için, standart C kitaplığındaki “DOSYA” türü opak bir tür olarak kabul edilebilir (ancak genellikle opak değildir, bu nedenle standart G/Ç işlevlerinin makro uygulamaları yapının iç kısımlarını kullanabilir) . Bu durumda, “header-1.h” şunları içerebilir:

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct MyStruct MyStruct;

int myFunction(MyStruct *value);

#endif

Yapının bir etiket adına sahip olması gerektiğine (burada ‘MyStruct’ - bu, etiket ad alanındadır, ‘MyStruct’ typedef adının sıradan tanımlayıcı ad alanından ayrıdır) ve ‘{ … }’ öğesinin atlandığına dikkat edin. Bu, “‘struct MyStruct’ yapı türü var ve bunun için ‘MyStruct’ takma adı var” diyor.

Uygulama dosyasında, türü tamamlamak için yapının ayrıntıları tanımlanabilir:

struct MyStruct {
    …
};

C11 kullanıyorsanız, typedef struct MyStruct MyStruct; bildirimini derleme hatasına neden olmadan tekrarlayabilirsiniz, ancak C’nin önceki sürümleri şikayet ederdi. Sonuç olarak, bu örnekte kodun yalnızca C11’i destekleyen derleyicilerle derlenmiş olması isteğe bağlı olsa da, include guard deyimini kullanmak yine de en iyisidir.

Birçok derleyici, aynı sonuçları veren “#pragma once” yönergesini destekler:

başlık-dosyam.h

#pragma once

// Code for header file

Ancak, “#pragma once” C standardının bir parçası değildir, bu nedenle kodu kullanırsanız daha az taşınabilir olur.

Birkaç başlık, include guard deyimini kullanmaz. Belirli bir örnek, standart “<assert.h>” başlığıdır. Tek bir çeviri birimine birden çok kez dahil edilebilir ve bunu yapmanın etkisi, başlığın her eklendiğinde “NDEBUG” makrosunun tanımlanıp tanımlanmamasına bağlıdır. Ara sıra benzer bir gereksiniminiz olabilir; bu tür vakalar çok az olacaktır. Normalde, başlıklarınız include guard deyimiyle korunmalıdır.

İşlev benzeri makrolar #

İşlev benzeri makrolar “satır içi” işlevlere benzer, bunlar geçici hata ayıklama günlüğü gibi bazı durumlarda kullanışlıdır:

#ifdef DEBUG
# define LOGFILENAME "/tmp/logfile.log"

# define LOG(str) do {                            \
  FILE *fp = fopen(LOGFILENAME, "a");            \
  if (fp) {                                       \
    fprintf(fp, "%s:%d %s\n", __FILE__, __LINE__, \
                 /* don't print null pointer */   \
                 str ?str :"<null>");             \
    fclose(fp);                                   \
  }                                               \
  else {                                          \
    perror("Opening '" LOGFILENAME "' failed");   \
  }                                               \
} while (0)
#else
  /* Make it a NOOP if DEBUG is not defined. */
# define LOG(LINE) (void)0
#endif


#include <stdio.h>

int main(int argc, char* argv[])
{
    if (argc > 1)
        LOG("There are command line arguments");
    else
        LOG("No command line arguments");
    return 0;
}

Burada her iki durumda da (“DEBUG” ile veya değil) çağrı aynı şekilde davranır ‘void’ dönüş tipine sahip bir fonksiyon olarak. Bu, “if/else” koşullarının beklendiği gibi yorumlanmasını sağlar.

“DEBUG” durumunda bu, bir “do { … } aracılığıyla uygulanır. while(0)` yapısı. Diğer durumda, ‘(void)0’, hiçbir yan etkisi olmayan ve göz ardı edilen bir ifadedir.

İkincisi için bir alternatif olabilir

#define LOG(LINE) do { /* empty */ } while (0)

öyle ki her durumda sözdizimsel olarak birinciye eşdeğerdir.

GCC kullanıyorsanız, standart olmayan bir GNU uzantısı olan ifade ifadeleri kullanarak sonuç döndüren işlev benzeri bir makro da uygulayabilirsiniz. Statement-Exprs.html#Statement-Exprs). Örneğin:

#include <stdio.h>

#define POW(X, Y) \
({ \
        int i, r = 1; \
        for (i = 0; i < Y; ++i) \
                r *= X; \
        r; \ // returned value is result of last operation
})

int main(void)
{
        int result;

        result = POW(2, 3); 
        printf("Result: %d\n", result);
}

Kaynak dosya ekleme #

“#include” ön işleme yönergelerinin en yaygın kullanımları aşağıdaki gibidir:

#include <stdio.h>
#include "myheader.h"

“#include”, ifadeyi başvurulan dosyanın içeriğiyle değiştirir. Köşeli parantezler (<>) sistemde yüklü olan başlık dosyalarına atıfta bulunurken, tırnak işaretleri (””) kullanıcı tarafından sağlanan dosyalar içindir.

Bu örnekte gösterildiği gibi, makrolar diğer makroları bir kez genişletebilir:

#if VERSION == 1
    #define INCFILE  "vers1.h"
#elif VERSION == 2
    #define INCFILE  "vers2.h"
    /*  and so on */
#else
    #define INCFILE  "versN.h"
#endif
/* ... */
#include INCFILE

Koşullu dahil etme ve koşullu işlev imzası değişikliği #

Bir kod bloğunu koşullu olarak dahil etmek için, önişlemcinin birkaç yönergesi vardır (örneğin, “#if”, “#ifdef”, “#else”, “#endif”, vb.).

/* Defines a conditional `printf` macro, which only prints if `DEBUG`
 * has been defined
 */
#ifdef DEBUG
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

‘#if’ koşulu için normal C ilişkisel işleçler kullanılabilir

#if __STDC_VERSION__ >= 201112L
/* Do stuff for C11 or higher */
#elif __STDC_VERSION__ >= 199901L
/* Do stuff for C99 */
#else
/* Do stuff for pre C99 */
#endif

‘#if’ yönergeleri, C ‘if’ deyimine benzer şekilde davranır, yalnızca integral sabit ifadeleri içerecek ve yayınlar içermeyecektir. Tanımlayıcı tanımlanmışsa “1”, aksi takdirde “0” döndüren ek bir tekli operatörü, “tanımlı( tanımlayıcı )” destekler.

#if defined(DEBUG) && !defined(QUIET)
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

Belirteç yapıştırma, birinin iki makro argümanını birbirine yapıştırmasını sağlar. Örneğin, “front##back”, “frontback” sonucunu verir. Ünlü bir örnek, Win32’nin <TCHAR.H> başlığıdır. Standart C’de, geniş karakterli bir dize bildirmek için L"string" yazılabilir. Bununla birlikte, Windows API’si geniş karakter dizileri ile dar karakter dizileri arasında basitçe “UNICODE” #define ile dönüştürme yapılmasına izin verir. #ifdef UNICODE

Bir kullanıcı TEXT("merhaba, dünya") yazdığında ve UNICODE tanımlandığında, C önişlemcisi L ve makro argümanını birleştirir. “Merhaba, dünya” ile birleştirilen “L”, “L"merhaba, dünya” verir.

Önceden Tanımlanmış Makrolar #

Önceden tanımlanmış bir makro, programın tanımlamasına gerek kalmadan C ön işlemcisi tarafından zaten anlaşılan bir makrodur. Örnekler şunları içerir:

Zorunlu Önceden Tanımlanmış Makrolar #

  • Geçerli kaynak dosyanın dosya adını veren __FILE__ (bir dize değişmezi),
  • Geçerli satır numarası için __LINE__ (bir tamsayı sabiti),
  • Derleme tarihi için __DATE__ (bir dize değişmezi),
  • Derleme zamanı için __TIME__ (bir dize değişmezi).

Ayrıca bir makro değil olan ilgili önceden tanımlanmış bir tanımlayıcı olan “func” (ISO/IEC 9899:2011 §6.4.2.2) vardır:

__func__ tanımlayıcısı, çevirmen tarafından zımnen, sanki, her işlev tanımının açılış ayracından hemen sonra, bildirim:

statik const char func[] = “işlev-adı”;

göründü, burada işlev-adı sözcüksel olarak çevreleyen işlevin adıdır.

__FILE__, __LINE__ ve __func__ özellikle hata ayıklama amaçları için kullanışlıdır. Örneğin:

fprintf(stderr, "%s: %s: %d: Denominator is 0", __FILE__, __func__, __LINE__);

C99 öncesi derleyiciler, func‘yi destekleyebilir veya desteklemeyebilir veya aynı işlevi gören ve farklı adlandırılmış bir makroya sahip olabilir. Örneğin, gcc, C89 modunda __FUNCTION__ kullandı.

Aşağıdaki makrolar, uygulama hakkında ayrıntılı bilgi istemenizi sağlar:

  • __STDC_VERSION__ Uygulanan C Standardı sürümü. Bu, “yyyymmL” biçimini kullanan sabit bir tamsayıdır (C11 için “201112L” değeri, C99 için “199901L” değeri; C89/C90 için tanımlanmamıştır)
  • ‘STDC_HOSTED’ Barındırılan bir uygulamaysa ‘1’, aksi takdirde ‘0’.
  • __STDC__ 1 ise, uygulama C Standardına uygundur.

Diğer Önceden Tanımlanmış Makrolar (zorunlu değil) #

ISO/IEC 9899:2011 §6.10.9.2 Ortam makroları:

  • __STDC_ISO_10646__ “yyyymmL” biçiminde bir tamsayı sabiti (örneğin, 199712L). Bu sembol tanımlanırsa, Unicode’daki her karakter gerekli küme, “wchar_t” türünde bir nesnede depolandığında, aynı bu karakterin kısa tanımlayıcısı olarak değer. Unicode gerekli kümesi ile birlikte ISO/IEC 10646 tarafından tanımlanan tüm karakterlerden oluşur. belirtilen yıldan itibaren tüm değişiklikler ve teknik düzeltmeler ve ay. Başka bir kodlama kullanılıyorsa, makro tanımlanmaz ve kullanılan gerçek kodlama uygulama tanımlıdır.

  • __STDC_MB_MIGHT_NEQ_WC__ Şunu belirtmeyi amaçlayan tamsayı sabiti 1, temel karakter kümesinin bir üyesi olan ‘wchar_t’ için kodlamaya gerek yoktur. yalnız karakter olarak kullanıldığında değerine eşit bir kod değerine sahip tamsayı karakter sabiti.

  • __STDC_UTF_16__ Tamsayı sabiti 1, bu tür değerleri belirtmek için tasarlanmıştır. “char16_t” UTF-16 kodludur. Başka bir kodlama kullanılırsa, makro tanımlanmaz ve kullanılan gerçek kodlama uygulama tarafından tanımlanır.

  • __STDC_UTF_32__ Tamsayı sabiti 1, bu tür değerleri belirtmek için tasarlanmıştır. “char32_t” UTF−32 kodludur. Başka bir kodlama kullanılırsa, makro tanımlanmaz ve kullanılan gerçek kodlama uygulama tarafından tanımlanır.

ISO/IEC 9899:2011 §6.10.8.3 Koşullu özellik makroları

  • __STDC_ANALYZABLE__ Tamsayı sabiti 1, uygunluğu göstermeyi amaçlar. Ek L’deki özellikler (Analiz edilebilirlik).
  • __STDC_IEC_559__ Tamsayı sabiti 1, aşağıdakilere uygunluğu göstermeyi amaçlamaktadır: Ek F’deki özellikler (IEC 60559 kayan nokta aritmetiği).
  • __STDC_IEC_559_COMPLEX__ Belirtmesi amaçlanan 1 tamsayı sabiti Ek G’deki spesifikasyonlara uygunluk (IEC 60559 uyumlu kompleks aritmetik).
  • __STDC_LIB_EXT1__ Destek belirtmeyi amaçlayan 201112L tamsayı sabiti Ek K’de tanımlanan uzantılar için (Sınır kontrol arayüzleri).
  • __STDC_NO_ATOMICS__ olduğunu belirtmeyi amaçlayan tamsayı sabiti 1, uygulama atom türlerini desteklemiyor (_Atomic dahil tür niteleyici) ve “<stdatomic.h>” başlığı.
  • __STDC_NO_COMPLEX__ olduğunu belirtmeyi amaçlayan tamsayı sabiti 1, uygulama, karmaşık türleri veya “<complex.h>” öğesini desteklemiyor başlık.
  • __STDC_NO_THREADS__ 1 tamsayı sabiti, uygulama “<threads.h>” başlığını desteklemiyor.
  • __STDC_NO_VLA__ olduğunu belirtmeyi amaçlayan tamsayı sabiti 1, uygulama, değişken uzunluklu dizileri veya değişkenleri desteklemez değiştirilmiş tipler.

__cplusplus, C++ ile derlenmiş C++ kodunda C harici öğelerini kullanmak için - ad yönetimi #

Dil farklılıkları nedeniyle derleyicinin C derleyicisi mi yoksa C++ derleyicisi mi olduğuna bağlı olarak bir içerme dosyasının önişlemciden farklı çıktılar üretmesi gerektiği zamanlar vardır.

Örneğin, bir işlev veya başka bir harici, bir C kaynak dosyasında tanımlanır, ancak bir C++ kaynak dosyasında kullanılır. C++, işlev argüman türlerine dayalı benzersiz işlev adları oluşturmak için ad yönetimini (veya ad süslemesini) kullandığından, C++ kaynak dosyasında kullanılan bir C işlev bildirimi bağlantı hatalarına neden olur. C++ derleyicisi, C++ için ad yönetimi kurallarını kullanarak derleyici çıktısı için belirtilen harici adı değiştirir. Sonuç, C++ derleyici çıktısı C derleyici çıktısına bağlandığında, harici öğelerden kaynaklanan bağlantı hatalarıdır.

C derleyicileri ad yönetimi yapmadığı, ancak C++ derleyicileri tarafından C++ derleyicisi tarafından oluşturulan tüm harici etiketler (işlev adları veya değişken adları) için yaptığından, derleyici algılamasına izin vermek için önceden tanımlanmış bir önişlemci makrosu ‘__cplusplus’ tanıtıldı.

C ve C++ arasındaki harici adlar için bu uyumsuz derleyici çıktısı sorununa geçici bir çözüm bulmak için, “__cplusplus” makrosu C++ Önişlemcisinde tanımlanmıştır ve C Önişlemcisinde tanımlanmamıştır. Bu makro adı, bir kaynak kodun veya içerme dosyasının C++ veya C olarak derlenip derlenmediğini söylemek için koşullu önişlemci ‘#ifdef’ yönergesiyle veya ‘defined()’ operatörüyle ‘#if’ ile kullanılabilir.

#ifdef __cplusplus
printf("C++\n");
#else
printf("C\n");
#endif

Ya da kullanabilirsin

#if defined(__cplusplus)
printf("C++\n");
#else
printf("C\n");
#endif

Bir C++ kaynak dosyasında kullanılan C derleyicisi ile derlenmiş bir C kaynak dosyasından bir işlevin doğru işlev adını belirtmek için, “harici “C"ye neden olmak için “__cplusplus” tanımlı sabitini kontrol edebilirsiniz. { /* … */ };başlık dosyası bir C++ kaynak dosyasına dahil edildiğinde C harici öğelerini bildirmek için kullanılacak. Ancak bir C derleyicisi ile derlendiğinde,harici “C” { */ … / };kullanılmaz. Bu koşullu derleme gereklidir çünküextern “C” { / … */ };` C++‘da geçerlidir ama C’de geçerli değildir.

#ifdef __cplusplus
// if we are being compiled with a C++ compiler then declare the
// following functions as C functions to prevent name mangling.
extern "C" {
#endif

// exported C function list.
int foo (void);

#ifdef __cplusplus
// if this is a C++ compiler, we need to close off the extern declaration.
};
#endif

Değişken argümanlar makrosu #

Değişken argümanlara sahip makrolar:

Diyelim ki kodunuzda hata ayıklamak için bir baskı makrosu oluşturmak istiyorsunuz, bu makroyu örnek olarak alalım:

#define debug_print(msg) printf("%s:%d %s", __FILE__, __LINE__, msg)

Bazı kullanım örnekleri:

“somefunc()” işlevi, başarısız olursa -1 ve başarılı olursa 0 döndürür ve kod içinde birçok farklı yerden çağrılır:

int retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

/* some other code */

 retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

‘somefunc()’ uygulaması değişirse ve şimdi farklı olası hata türleriyle eşleşen farklı değerler döndürürse ne olur? Hala hata ayıklama makrosunu kullanmak ve hata değerini yazdırmak istiyorsunuz.

debug_printf(retVal);      /* this would obviously fail */
debug_printf("%d",retVal); /* this would also fail */

Bu sorunu çözmek için __VA_ARGS__ makrosu tanıtıldı. Bu makro, X-makro’nun birden çok parametresine izin verir:

Örnek:

 #define debug_print(msg, ...) printf(msg, __VA_ARGS__) \
                               printf("\nError occurred in file:line (%s:%d)\n", __FILE__, __LINE)

Kullanım:

int retVal = somefunc();

debug_print("retVal of somefunc() is-> %d", retVal);

Bu makro, birden fazla parametre iletmenize ve bunları yazdırmanıza izin verir, ancak şimdi herhangi bir parametre göndermenizi yasaklar.

debug_print("Hey");

Bu, makro en az bir argüman daha beklediğinden ve ön işlemcinin “debug_print()” makrosundaki virgül eksikliğini görmezden gelmeyeceğinden, bazı sözdizimi hatalarına neden olur. Ayrıca debug_print("Hey",); makroya iletilen argümanı boş tutamayacağınız için bir sözdizimi hatasına neden olur.

Bunu çözmek için ##__VA_ARGS__ makrosu tanıtıldı, bu makro değişken argümanları yoksa virgülün ön işlemci tarafından koddan silindiğini belirtir.

Örnek:

 #define debug_print(msg, ...) printf(msg, ##__VA_ARGS__) \
                               printf("\nError occured in file:line (%s:%d)\n", __FILE__, __LINE)

Kullanım:

 debug_print("Ret val of somefunc()?");
 debug_print("%d",somefunc());
son sürüm

Makro Değiştirme #

Makro değiştirmenin en basit biçimi, aşağıdaki gibi bir “açık sabit” tanımlamaktır.

#define ARRSIZE 100
int array[ARRSIZE];

Bu, bir değişkeni “10” ile çarpan ve yeni değeri depolayan işlev benzeri bir makro tanımlar:

#define TIMES10(A) ((A) *= 10)

double b = 34;
int c = 23;

TIMES10(b);   // good: ((b) *= 10);
TIMES10(c);   // good: ((c) *= 10);
TIMES10(5);   // bad:  ((5) *= 10);

Değiştirme, program metninin herhangi bir başka yorumundan önce yapılır. ‘TIMES10’a yapılan ilk çağrıda, tanımdaki ‘A’ adı ‘b’ ile değiştirilir ve böylece genişletilmiş metin çağrının yerine konur. Bu “TIMES10” tanımının şuna eşdeğer olmadığını unutmayın:

#define TIMES10(A) ((A) = (A) * 10)

çünkü bu, istenmeyen yan etkilere neden olabilecek ‘A’nın iki kez değiştirilmesini değerlendirebilir.

Aşağıda, argümanlarının maksimumu olan işlev benzeri bir makro tanımlanır. Herhangi bir uyumlu argüman türü için çalışma ve işlev çağırma ek yükü olmadan satır içi kod oluşturma avantajlarına sahiptir. Argümanlarından birini veya diğerini ikinci kez değerlendirme (yan etkiler dahil) ve birkaç kez çağrıldığında bir işlevden daha fazla kod üretme dezavantajlarına sahiptir.

#define max(a, b) ((a) > (b) ? (a) : (b))

int maxVal = max(11, 43);              /* 43 */
int maxValExpr = max(11 + 36, 51 - 7); /* 47 */

/* Should not be done, due to expression being evaluated twice */
int j = 0, i = 0;
int sideEffect = max(++i, ++j);       /* i == 4 */

Bu nedenle, argümanlarını birden çok kez değerlendiren bu tür makrolardan genellikle üretim kodunda kaçınılır. C11’den beri, bu tür çoklu çağrıları önlemeye izin veren ‘_Generic’ özelliği vardır.

Makro açılımlarında bol miktarda parantez (tanımın sağ tarafı), argümanların ve sonuçtaki ifadenin düzgün bir şekilde bağlanmasını ve makronun çağrıldığı bağlama iyi uymasını sağlar.

FOREACH uygulaması #

Kodun okunmasını ve yazılmasını kolaylaştırmak için makroları da kullanabiliriz. Örneğin, tek ve çift bağlantılı listeler, kuyruklar vb. gibi bazı veri yapıları için C’deki ‘foreach’ yapısını uygulamak için makrolar uygulayabiliriz.

İşte küçük bir örnek.

#include <stdio.h>
#include <stdlib.h>

struct LinkedListNode
{
    int data;
    struct LinkedListNode *next;
};

#define FOREACH_LIST(node, list) \
     for (node=list; node; node=node->next)

/* Usage */
int main(void)
{
    struct LinkedListNode *list, **plist = &list, *node;
    int i;

    for (i=0; i<10; i++)
    {
         *plist = malloc(sizeof(struct LinkedListNode));
         (*plist)->data = i;
         (*plist)->next = NULL;
         plist          = &(*plist)->next;
    }

    /* printing the elements here */
    FOREACH_LIST(node, list)
    {
        printf("%d\n", node->data);
    }
}

Bu tür veri yapıları için standart bir arayüz oluşturabilir ve ‘FOREACH’in genel bir uygulamasını şu şekilde yazabilirsiniz:

#include <stdio.h>
#include <stdlib.h>

typedef struct CollectionItem_
{
    int data;
    struct CollectionItem_ *next;
} CollectionItem;

typedef struct Collection_
{
    /* interface functions */
    void* (*first)(void *coll);
    void* (*last) (void *coll);
    void* (*next) (void *coll, CollectionItem *currItem);

    CollectionItem *collectionHead;
    /* Other fields */
} Collection;

/* must implement */
void *first(void *coll)
{
    return ((Collection*)coll)->collectionHead;
}

/* must implement */
void *last(void *coll)
{
    return NULL;
}

/* must implement */
void *next(void *coll, CollectionItem *curr)
{
    return curr->next;
}

CollectionItem *new_CollectionItem(int data)
{
    CollectionItem *item = malloc(sizeof(CollectionItem));
    item->data = data;
    item->next = NULL;
    return item;
}

void Add_Collection(Collection *coll, int data)
{
    CollectionItem **item = &coll->collectionHead;
    while(*item)
        item = &(*item)->next;
    (*item) = new_CollectionItem(data);
}

Collection *new_Collection()
{
    Collection *nc = malloc(sizeof(Collection));
    nc->first = first;
    nc->last  = last;
    nc->next  = next;
    return nc;
}

/* generic implementation */
#define FOREACH(node, collection)                      \
    for (node  = (collection)->first(collection);      \
         node != (collection)->last(collection);       \
         node  = (collection)->next(collection, node))

int main(void)
{
    Collection *coll = new_Collection();
    CollectionItem *node;
    int i;

    for(i=0; i<10; i++)
    {
         Add_Collection(coll, i);
    }

    /* printing the elements here */
    FOREACH(node, coll)
    {
        printf("%d\n", node->data);
    }
}

Bu genel uygulamayı kullanmak için veri yapınız için bu işlevleri uygulamanız yeterlidir.
