Aliasing ve etkili tip

  • Etkili tip
  • kalifikasyonu kısıtlamak

Bu sayfada

  • Etkili tip
  • kalifikasyonu kısıtlamak
  • Baytları değiştirme

  • Sıkı aliasing kurallarını ihlal etmek

Örtüşme kurallarının ihlali ve bir nesnenin etkin tipinin ihlali iki farklı şeydir ve karıştırılmamalıdır.

  • Aliasing, aynı nesneye, yani “a == b"ye atıfta bulunan iki “a” ve “b” işaretçisinin özelliğidir.

  • Bir veri nesnesinin etkin türü, o nesne üzerinde hangi işlemlerin yapılabileceğini belirlemek için C tarafından kullanılır. Özellikle etkin tip, iki işaretçinin birbirinin yerine geçip geçemeyeceğini belirlemek için kullanılır.

Örtüşme optimizasyon için bir sorun olabilir, çünkü nesneyi bir işaretçi, örneğin “a” aracılığıyla değiştirmek, diğer işaretçi “b” aracılığıyla görünen nesneyi değiştirebilir. C derleyiciniz, türleri ve kökenleri ne olursa olsun, işaretçilerin her zaman birbirinin yerine geçebileceğini varsaymak zorunda kalırsa, birçok optimizasyon fırsatı kaybedilir ve birçok program daha yavaş çalışır.

C’nin katı aliasing kuralları, derleyicideki hangi nesnelerin birbirini alias yaptığını (veya yapmadığını) varsayabileceği durumları ifade eder. Veri işaretçileri için her zaman aklınızda bulundurmanız gereken iki temel kural vardır.

Aksi söylenmedikçe, aynı taban tipine sahip iki işaretçi alias olabilir.

İki türden en az biri bir karakter türü olmadıkça, farklı temel türe sahip iki işaretçi diğer ad olamaz.

Burada temel tür, “const” gibi tür niteliklerini bir kenara koyduğumuz anlamına gelir, ör. “a”, “double*” ve “b”, “const double*” ise, derleyici *a"daki bir değişikliğin “b"yi değiştirebileceğini varsaymalıdır.

İkinci kuralı ihlal etmek feci sonuçlara yol açabilir. Burada katı aliasing kuralını ihlal etmek, derleyiciye gerçekte aynı nesneye işaret eden farklı tipte ‘a’ ve ‘b’ iki işaretçisi sunduğunuz anlamına gelir. Derleyici daha sonra her zaman ikisinin farklı nesnelere işaret ettiğini varsayabilir ve nesneyi ‘*a’ ile değiştirdiyseniz ‘*b’ fikrini güncellemeyecektir.

Bunu yaparsanız, programınızın davranışı tanımsız hale gelir. Bu nedenle, C, böyle bir durumun kazara meydana gelmesini önlemenize yardımcı olmak için işaretçi dönüşümlerine oldukça ciddi kısıtlamalar getirir.

Kaynak veya hedef türü “void” olmadıkça, tüm işaretçi dönüşümleri farklı taban tipine sahip işaretçiler arasında açık olmalıdır.

Veya başka bir deyişle, hedef türe yalnızca “const” gibi bir niteleyici ekleyen bir dönüşüm yapmadığınız sürece bir cast’e ihtiyaçları vardır.

Genel olarak işaretçi dönüşümlerinden ve özellikle yayınlardan kaçınmak, sizi takma sorunlarından korur. Onlara gerçekten ihtiyacınız yoksa ve bu durumlar çok özel değilse, onlardan olabildiğince kaçınmalısınız.

Etkili tip #

Bir veri nesnesinin etkili türü, varsa onunla ilişkilendirilmiş son tür bilgisidir.

// a normal variable, effective type uint32_t, and this type never changes
uint32_t a = 0.0;

// effective type of *pa is uint32_t, too, simply
// because *pa is the object a
uint32_t* pa = &a;

// the object pointed to by q has no effective type, yet
void* q = malloc(sizeof uint32_t);
// the object pointed to by q still has no effective type,
// because nobody has written to it
uint32_t* qb = q;
// *qb now has effective type uint32_t because a uint32_t value was written
*qb = 37;

// the object pointed to by r has no effective type, yet, although
// it is initialized
void* r = calloc(1, sizeof uint32_t);
// the object pointed to by r still has no effective type,
// because nobody has written to or read from it
uint32_t* rc = r;
// *rc now has effective type uint32_t because a value is read
// from it with that type. The read operation is valid because we used calloc.
// Now the object pointed to by r (which is the same as *rc) has
// gained an effective type, although we didn't change its value.
uint32_t c = *rc;

// the object pointed to by s has no effective type, yet.
void* s = malloc(sizeof uint32_t);
// the object pointed to by s now has effective type uint32_t
// because an uint32_t value is copied into it.
memcpy(s, r, sizeof uint32_t);

İkincisi için, bizim bile sahip olmamızın gerekli olmadığını gözlemleyin. o nesneye ‘uint32_t*’ işaretçisi. Bir başkasını kopyaladığımız gerçeği ‘uint32_t’ nesnesi yeterlidir.

kalifikasyonu kısıtlamak #

Aynı türden iki işaretçi argümanımız varsa, derleyici herhangi bir varsayımda bulunamaz ve her zaman “*e"deki değişikliğin “*f"yi değiştirebileceğini varsayması gerekir:

void fun(float* e, float* f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

float fval = 4;
float eval = 77;
 fun(&eval, &fval);

her şey yolunda gidiyor ve bunun gibi bir şey

4 eşittir 4?

yazdırılır. Aynı işaretçiyi geçersek, program yine de doğru olanı yapacak ve yazdıracaktır.

4 eşittir 22?

Eğer “e” ve “f"nin asla aynı veri nesnesine işaret etmeyeceğini bazı dış bilgilerle *bilirsek, bu verimsiz olabilir. İşaretçi parametrelerine “kısıtlama” niteleyicileri ekleyerek bu bilgiyi yansıtabiliriz:

void fan(float*restrict e, float*restrict f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

O zaman derleyici her zaman “e” ve “f"nin farklı nesneleri işaret ettiğini varsayabilir.

Baytları değiştirme #

Bir nesnenin etkin bir türü olduğunda, diğer tür bir karakter türü, “char”, “signed char” veya “unsigned char” olmadığı sürece, onu başka bir türdeki bir işaretçi aracılığıyla değiştirmeye çalışmamalısınız.

#include <inttypes.h>
#include <stdio.h>

int main(void) {
  uint32_t a = 57;
  // conversion from incompatible types needs a cast !
  unsigned char* ap = (unsigned char*)&a;
  for (size_t i = 0; i < sizeof a; ++i) {
    /* set each byte of a to 42 */
    ap[i] = 42;
  }
  printf("a now has value %" PRIu32 "\n", a);
}

Bu, yazdıran geçerli bir programdır.

a şimdi değeri 707406378

Bu işe yarar çünkü:

  • Erişim, ‘unsigned char’ türüyle görülen bireysel baytlara yapılır, böylece her değişiklik iyi tanımlanır.
  • Nesneye iki görünüm, ‘a’ ve ‘*ap’ aracılığıyla, takma ad, ancak ‘ap’ bir karakter türünün işaretçisi olduğundan, katı aliasing kuralı uygulanmaz. Bu nedenle derleyici, “a"nın değerinin “for” döngüsünde değiştirilmiş olabileceğini varsaymalıdır. “a"nın değiştirilmiş değeri, değiştirilen baytlardan oluşturulmalıdır.
  • “a”, “uint32_t” türünde dolgu biti yok. Temsilinin tüm bitleri, burada “707406378” değeri için sayılır ve tuzak gösterimi olamaz.

Karakter türlerine, karakter olmayan türler aracılığıyla erişilemez. #

Bir nesne statik, iş parçacığı veya otomatik saklama süresi ile tanımlanmışsa ve karakter tipine sahipse, karakter tipi, karakter, imzasız karakter veya imzalı karakter, karakter olmayan bir tür tarafından erişilemez. . Aşağıdaki örnekte bir “char” dizisi, “int” türü olarak yeniden yorumlanmıştır ve davranış, “int” işaretçisi “b"nin her başvurusunda tanımsızdır.

int main( void )
{
    char a[100];
    int* b = ( int* )&a;
    *b = 1;      

    static char c[100];
    b = ( int* )&c;
    *b = 2;

    _Thread_local char d[100];
    b = ( int* )&d;
    *b = 3;
}

Bu tanımsızdır, çünkü “etkili tip” kuralını ihlal eder, etkin tipe sahip hiçbir veri nesnesine karakter tipi olmayan başka bir tip üzerinden erişilemez. Buradaki diğer tür “int” olduğundan buna izin verilmez.

Hizalama ve işaretçi boyutlarının sığdığı bilinse bile, bu bu kuraldan muaf olmayacak, davranış yine tanımsız olacaktır.

Bu, özellikle standart C’de rezerve etmenin bir yolu olmadığı anlamına gelir. işaretçiler aracılığıyla kullanılabilen karakter türünde bir arabellek nesnesi tarafından alınan bir arabellek kullanacağınız için farklı türlerle “malloc” veya benzeri bir işlev.

Yukarıdaki örnekte olduğu gibi aynı amaca ulaşmanın doğru bir yolu, bir “birlik” kullanmak.

typedef union bufType bufType;
union bufType {
   char c[sizeof(int[25])];
   int i[25];
};

int main( void )
{
    bufType a = { .c = { 0 } }; // reserve a buffer and initialize
    int* b = a.i;      // no cast necessary
    *b = 1;      

    static bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 2;

    _Thread_local bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 3;
}

Burada, “birlik”, derleyicinin en başından arabelleğin olabileceğini bilmesini sağlar. farklı görünümler aracılığıyla erişilir. Bu, aynı zamanda, arabelleğin, zaten “int” türünde olan bir “a.i” “görünümüne” sahip olması ve işaretçi dönüşümüne gerek olmaması avantajına da sahiptir.

Sıkı aliasing kurallarını ihlal etmek #

Aşağıdaki kodda, basitlik için “float” ve “uint32_t"nin aynı boyuta sahip olduğunu varsayalım.

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    float b = *f;
    print("%g should equal %g\n", a, b);
}

‘u’ ve ‘f’ farklı temel tiplere sahiptir ve bu nedenle derleyici bunların farklı nesnelere işaret ettiğini varsayabilir. “a” ve “b"nin iki başlatması arasında “*f"nin değişmiş olma olasılığı yoktur ve bu nedenle derleyici kodu eşdeğer bir şeye optimize edebilir.

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    print("%g should equal %g\n", a, a);
}

Yani, *f ikinci yükleme işlemi tamamen optimize edilebilir.

Bu fonksiyona “normalde” dersek

 float fval = 4;
 uint32_t uval = 77;
 fun(&uval, &fval);

her şey yolunda gidiyor ve bunun gibi bir şey

4, 4’e eşit olmalıdır

yazdırılır. Ancak, dönüştürdükten sonra hile yapıp aynı işaretçiyi geçersek,

 float fval = 4;
 uint32_t* up = (uint32_t*)&fval;
 fun(up, &fval);

katı takma kuralını ihlal ediyoruz. Sonra davranış tanımsız hale gelir. Derleyici ikinci erişimi veya tamamen farklı bir şeyi optimize etmişse, çıktı yukarıdaki gibi olabilir ve böylece programınız tamamen güvenilmez bir duruma gelir.
