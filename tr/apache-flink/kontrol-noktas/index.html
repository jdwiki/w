(Flink 1.2 ve altında test edilmiştir)

Flink’teki her işlev, kaynak veya operatör durum bilgisi olabilir. Kontrol noktaları, uygulamaya hatasız bir yürütmeyle aynı semantiği vermek için Flink’in akışlardaki durumu ve konumları kurtarmasına izin verir. Bu, hata toleransı ve tam bir kez işleme garantilerinin arkasındaki mekanizmadır.

İçindekileri anlamak için bu makaleyi okuyun.

Denetim noktaları, yalnızca kümede bir hata meydana geldiğinde, örneğin bir görev yöneticisi başarısız olduğunda yararlıdır. İşin kendisi başarısız olduktan veya iptal edildikten sonra devam etmezler.

Başarısızlıktan/iptalden sonra durum bilgisi olan bir işe devam edebilmek için savepoints veya haricileştirilmiş kontrol noktalarına (flink 1.2+) bakın.

Yapılandırma ve kurulum #

Kontrol noktası yapılandırması iki adımda yapılır. İlk olarak, bir backend seçmeniz gerekir. Ardından, uygulama bazında kontrol noktalarının aralığını ve modunu belirleyebilirsiniz.

Arka uçlar #

Kullanılabilir arka uçlar

Kontrol noktalarının depolandığı yer, yapılandırılmış arka uca bağlıdır:

  • MemoryStateBackend: bellek içi durum, JobManager/ZooKeeper belleğine yedekleme. Yalnızca minimum durum (örneğin, Kafka ofsetlerini depolamak için varsayılan olarak maks. 5 MB’dir) veya test ve yerel hata ayıklama için kullanılmalıdır.
  • FsStateBackend: durum, TaskManager’larda bellekte tutulur ve durum anlık görüntüleri (yani kontrol noktaları) bir dosya sisteminde (HDFS, DS3, yerel dosya sistemi, …) saklanır. Bu kurulum, büyük durumlar veya uzun pencereler ve yüksek kullanılabilirlik kurulumları için önerilir.
  • “RocksDBStateBackend”: Uçuş sırasındaki verileri (varsayılan olarak) TaskManager veri dizinlerinde depolanan bir RocksDB veritabanında tutar. Kontrol noktası üzerine, tüm RocksDB veritabanı bir dosyaya yazılır (yukarıdaki gibi). FsStateBackend ile karşılaştırıldığında, daha büyük durumlara izin verir (yalnızca disk alanı ile görev yöneticisi belleğinin boyutu ile sınırlıdır), ancak verim daha düşük olacaktır (veriler her zaman bellekte değildir, diskten yüklenmelidir).

Arka uç ne olursa olsun, meta verilerin (kontrol noktalarının sayısı, yerelleştirme vb.) her zaman iş yöneticisi belleğinde saklandığını ve kontrol noktalarının __ uygulama sonlandırıldıktan/iptal edildikten sonra devam etmeyeceğini unutmayın__.

Arka ucu belirtme

Programınızın “ana” yönteminde arka ucu aşağıdakileri kullanarak belirtirsiniz:

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.setStateBackend(new FsStateBackend("hdfs://namenode:40010/flink/checkpoints"));

Veya varsayılan arka ucu “flink/conf/flink-conf.yaml” içinde ayarlayın:

# Supported backends: 
#  - jobmanager (MemoryStateBackend), 
#  - filesystem (FsStateBackend), 
#  - rocksdb (RocksDBStateBackend), 
#  - <class-name-of-factory>
state.backend: filesystem
    
# Directory for storing checkpoints in a Flink-supported filesystem
# Note: State backend must be accessible from the JobManager and all TaskManagers.
# Use "hdfs://" for HDFS setups, "file://" for UNIX/POSIX-compliant file systems, 
# "S3://" for S3 file system.
state.backend.fs.checkpointdir: file:///tmp/flink-backend/checkpoints

Kontrol noktalarını etkinleştirme #

Her uygulamanın kontrol noktalarını açıkça etkinleştirmesi gerekir:

long checkpointInterval = 5000; // every 5 seconds

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(checkpointInterval);

İsteğe bağlı olarak bir kontrol noktası modu belirleyebilirsiniz. Değilse, varsayılan olarak tam olarak bir kez:

env.enableCheckpointing(checkpointInterval, CheckpointingMode.AT_LEAST_ONCE);

Kontrol noktası modu, arızaların varlığında sistemin hangi tutarlılığı garanti ettiğini tanımlar. Kontrol noktası etkinleştirildiğinde, veri akışları, işlemin kayıp kısımları tekrarlanacak şekilde yeniden oynatılır. ‘EXACTLY_ONCE’ ile sistem, bir kurtarma işleminin operatörler/fonksiyonlar her kaydı “tam olarak bir kez” görmüş gibi davranacağı şekilde kontrol noktaları çizer. “AT_LEAST_ONCE” ile kontrol noktaları, kurtarma sırasında genellikle bazı kopyalarla karşılaşan daha basit bir şekilde çizilir.

Kontrol noktalarını test etme #

Kod #

İşte “Tamsayı” tarafından yönetilen bir duruma sahip durum bilgisi olan bir eşleyici kullanan basit bir göz kırpma uygulaması. Etkilerini görmek için “checkpointEnable”, “checkpointInterval” ve “checkpointMode” değişkenleriyle oynayabilirsiniz:

public class CheckpointExample {

    private static Logger LOG = LoggerFactory.getLogger(CheckpointExample.class);
    private static final String KAFKA_BROKER = "localhost:9092";
    private static final String KAFKA_INPUT_TOPIC = "input-topic";
    private static final String KAFKA_GROUP_ID = "flink-stackoverflow-checkpointer";
    private static final String CLASS_NAME = CheckpointExample.class.getSimpleName();


    public static void main(String[] args) throws Exception {

        // play with them
        boolean checkpointEnable = false;
        long checkpointInterval = 1000;
        CheckpointingMode checkpointMode = CheckpointingMode.EXACTLY_ONCE;

        // ----------------------------------------------------

        LOG.info(CLASS_NAME + ": starting...");
        final StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // kafka source
        // https://ci.apache.org/projects/flink/flink-docs-release-1.2/dev/connectors/kafka.html#kafka-consumer
        Properties prop = new Properties();
        prop.put("bootstrap.servers", KAFKA_BROKER);
        prop.put("group.id", KAFKA_GROUP_ID);
        prop.put("auto.offset.reset", "latest");
        prop.put("enable.auto.commit", "false");

        FlinkKafkaConsumer09<String> source = new FlinkKafkaConsumer09<>(
                KAFKA_INPUT_TOPIC, new SimpleStringSchema(), prop);

        // checkpoints
        // internals: https://ci.apache.org/projects/flink/flink-docs-master/internals/stream_checkpointing.html#checkpointing
        // config: https://ci.apache.org/projects/flink/flink-docs-release-1.3/dev/stream/checkpointing.html
        if (checkpointEnable) env.enableCheckpointing(checkpointInterval, checkpointMode);

        env
                .addSource(source)
                .keyBy((any) -> 1)
                .flatMap(new StatefulMapper())
                .print();

        env.execute(CLASS_NAME);
    }

        /* *****************************************************************
         * Stateful mapper
         * (cf. https://ci.apache.org/projects/flink/flink-docs-release-1.3/dev/stream/state.html)
         * ****************************************************************/

    public static class StatefulMapper extends RichFlatMapFunction<String, String> {
        private transient ValueState<Integer> state;

        @Override
        public void flatMap(String record, Collector<String> collector) throws Exception {
            // access the state value
            Integer currentState = state.value();

            // update the counts
            currentState += 1;
            collector.collect(String.format("%s: (%s,%d)",
                    LocalDateTime.now().format(ISO_LOCAL_DATE_TIME), record, currentState));
            // update the state
            state.update(currentState);
        }

        @Override
        public void open(Configuration parameters) throws Exception {
            ValueStateDescriptor<Integer> descriptor =
                    new ValueStateDescriptor<>("CheckpointExample", TypeInformation.of(Integer.class), 0);
            state = getRuntimeContext().getState(descriptor);
        }
    }
}

Örneği çalıştırma ve başarısızlığı simüle etme #

Kontrol noktalarını kontrol edebilmek için bir “küme” başlatmanız gerekir. En kolay yol, “flink/bin” dizinindeki “start-cluster.sh” betiğini kullanmaktır:

start-cluster.sh
Starting cluster.
[INFO] 1 instance(s) of jobmanager are already running on virusnest.
Starting jobmanager daemon on host virusnest.
Password:
Starting taskmanager daemon on host virusnest.

Şimdi uygulamanızı paketleyin ve yanıp sönmeye gönderin:

mvn clean package
flink run target/flink-checkpoints-test.jar -c CheckpointExample

Bazı veriler oluşturun:

kafka-console-producer --broker-list localhost:9092 --topic input-topic 
a
b
c
^D

Çıktı, “flink/logs/flink--jobmanager-0-.out” içinde mevcut olmalıdır. Örneğin:

tail -f flink/logs/flink-Derlin-jobmanager-0-virusnest.out
2017-03-17T08:21:51.249: (a,1)
2017-03-17T08:21:51.545: (b,2)
2017-03-17T08:21:52.363: (c,3)

Kontrol noktalarını test etmek için, sadece görev yöneticisini öldürün (bu bir arızayı taklit eder), bazı veriler üretin ve yeni bir tane başlatın:

# killing the taskmanager
ps -ef | grep -i taskmanager
kill <taskmanager PID>

# starting a new taskmanager
flink/bin/taskmanager.sh start

Not: yeni bir görev yöneticisi başlatırken, başka bir günlük dosyası kullanır, yani flink/logs/flink-<user>-jobmanager-1-<host>.out (tamsayı artışına dikkat edin).

Ne bekleyebileceğinizi #

  • kontrol noktaları devre dışı: hata sırasında veri üretirseniz, kesinlikle kaybolurlar. Ama şaşırtıcı bir şekilde, sayaçlar doğru olacak!
  • kontrol noktaları etkin: artık veri kaybı yok (ve doğru sayaçlar).
  • en az bir kez modu olan kontrol noktaları: özellikle bir kontrol noktası aralığını yüksek bir sayıya ayarlarsanız ve görev yöneticisini birden çok kez öldürürseniz, yinelenenler görebilirsiniz
