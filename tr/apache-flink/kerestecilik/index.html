Bu konu, Flink uygulamalarında günlüğe kaydetmenin (log4j) nasıl kullanılacağını ve yapılandırılacağını gösterir.

Günlük yapılandırması #

Yerel mod #

Yerel modda, örneğin uygulamanızı bir IDE’den çalıştırırken, “log4j"yi her zamanki gibi, yani sınıf yolunda bir “log4j.properties” hazırlayarak yapılandırabilirsiniz. Maven’de kolay bir yol, “src/main/resources” klasöründe “log4j.properties” oluşturmaktır. İşte bir örnek:

log4j.rootLogger=INFO, console

# patterns:
#  d = date
#  c = class
#  F = file
#  p = priority (INFO, WARN, etc)
#  x = NDC (nested diagnostic context) associated with the thread that generated the logging event
#  m = message


# Log all infos in the console
log4j.appender.console=org.apache.log4j.ConsoleAppender
log4j.appender.console.layout=org.apache.log4j.PatternLayout
log4j.appender.console.layout.ConversionPattern=%d{dd/MM/yyyy HH:mm:ss.SSS} %5p [%-10c] %m%n

# Log all infos in flink-app.log
log4j.appender.file=org.apache.log4j.FileAppender
log4j.appender.file.file=flink-app.log
log4j.appender.file.append=false
log4j.appender.file.layout=org.apache.log4j.PatternLayout
log4j.appender.file.layout.ConversionPattern=%d{dd/MM/yyyy HH:mm:ss.SSS} %5p [%-10c] %m%n

# suppress info messages from flink
log4j.logger.org.apache.flink=WARN

Bağımsız mod #

Bağımsız modda, kullanılan gerçek yapılandırma, ‘jar’ dosyanızdaki yapılandırma değildir. Bunun nedeni, Flink’in kendi yapılandırma dosyalarına sahip olması ve bu dosyaların sizinkinden öncelikli olmasıdır.

Varsayılan dosyalar: Flink, aşağıdaki varsayılan özellikler dosyalarıyla birlikte gelir:

  • “log4j-cli.properties”: Flink komut satırı istemcisi tarafından kullanılır (ör. “flink run”) (kod kümede yürütülmez)
  • log4j-yarn-session.properties: Bir YARN oturumu başlatırken Flink komut satırı istemcisi tarafından kullanılır (yarn-session.sh)
  • “log4j.properties”: JobManager/Taskmanager günlükleri (hem bağımsız hem de YARN)

“${log.file}” dosyasının varsayılan olarak “flink/log” olduğunu unutmayın. ‘flink-conf.yaml’ içinde ’env.log.dir’ ayarlanarak geçersiz kılınabilir,

env.log.dir Flink günlüklerinin kaydedildiği dizini tanımlar. Mutlak bir yol olmalı.

Günlük konumu: günlükler _yereldir, yani JobManager(lar) / Taskmanager(lar) çalıştıran makine(ler)de üretilirler.

Yarn: Flink on Yarn çalıştırırken, Hadoop YARN’ın günlüğe kaydetme yeteneklerine güvenmeniz gerekir. Bunun için en kullanışlı özellik YARN günlük toplama. Etkinleştirmek için, “yarn-site.xml dosyasında” “yarn.log-aggregation-enable” özelliğini “true” olarak ayarlayın. Bu etkinleştirildiğinde, (başarısız) bir YARN oturumunun tüm günlük dosyalarını aşağıdakileri kullanarak alabilirsiniz:

yarn logs -applicationId <application ID>

Ne yazık ki, günlükler yalnızca bir oturumun çalışması durduktan sonra, örneğin bir başarısızlıktan sonra kullanılabilir.

Kodunuzda bir kaydedici kullanma #

“pom.xml” dosyanıza “slf4j” bağımlılığını ekleyin:

<properties>
    <slf4j.version>1.7.21</slf4j.version>
</properties>

<!-- ... --> 

<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-api</artifactId>
    <version>${slf4j.version}</version>
</dependency>
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-log4j12</artifactId>
    <version>${slf4j.version}</version>
</dependency>

Sınıfınızda kullanmak için bir günlükçü nesnesi oluşturun:

private Logger LOGGER = LoggerFactory.getLogger(FlinkApp.class);

“RichMapFunction"ın alt sınıfları gibi serileştirilmesi gereken sınıflarda, “LOGGER"ı “geçici” olarak tanımlamayı unutmayın:

private transient Logger LOG = LoggerFactory.getLogger(MyRichMapper.class);

Kodunuzda her zamanki gibi “LOGGER” kullanın. Nesneleri biçimlendirmek için yer tutucuları ({}) kullanın ve benzeri:

LOGGER.info("my app is starting");
LOGGER.warn("an exception occurred processing {}", record, exception);

Her uygulama için farklı konfigürasyon(lar) kullanma #

Çeşitli uygulamalarınız için farklı ayarlara ihtiyacınız olması durumunda, (Flink 1.2’den itibaren) bunu yapmanın kolay bir yolu yoktur.

Flink’in one-yarn-cluster-per-job modunu kullanırsanız (yani komut dosyalarınızı şu şekilde başlatırsanız: ‘flink run -m yarn-cluster’ …), işte bir geçici çözüm:

  1. projenize yakın bir yerde bir “conf” dizini oluşturun

  2. flink/conf içindeki tüm dosyalar için sembolik bağlantılar oluşturun:

     mkdir conf
 cd conf
 ln -s flink/conf/* .

  3. log4j.properties sembolik bağlantısını (veya değiştirmek istediğiniz başka bir dosyayı) kendi yapılandırmanızla değiştirin

  4. işinizi başlatmadan önce çalıştırın

     export FLINK_CONF_DIR=/path/to/my/conf

Flink sürümünüze bağlı olarak, flink/bin/config.sh dosyasını düzenlemeniz gerekebilir. Koşunuz bu çizgiyi geçerse:

FLINK_CONF_DIR=$FLINK_ROOT_DIR_MANGLED/conf

şununla değiştirin:

if [ -z "$FLINK_CONF_DIR" ]; then 
    FLINK_CONF_DIR=$FLINK_ROOT_DIR_MANGLED/conf; 
fi

Flink-on-Yarn geçici çözümü: rsyslog ile günlükleri gerçek zamanlı olarak alın #

Yarn, varsayılan olarak günlükleri bir uygulama tamamlanmadan toplamaz; bu, sonlandırılmayan akış işlerinde sorun olabilir.

Çözüm, çoğu linux makinesinde bulunan rsyslog kullanmaktır.

İlk olarak, /etc/rsyslog.conf dosyasında aşağıdaki satırların yorumunu kaldırarak gelen udp isteklerine izin verin:

$ModLoad imudp
$UDPServerRun 514

“SyslogAppender"ı kullanmak için “log4j.properties” dosyanızı düzenleyin (bu sayfadaki diğer örneklere bakın):

log4j.rootLogger=INFO, file

# TODO: change package logtest to your package
log4j.logger.logtest=INFO, SYSLOG

# Log all infos in the given file
log4j.appender.file=org.apache.log4j.FileAppender
log4j.appender.file.file=${log.file}
log4j.appender.file.append=false
log4j.appender.file.layout=org.apache.log4j.PatternLayout
log4j.appender.file.layout.ConversionPattern=bbdata: %d{yyyy-MM-dd HH:mm:ss,SSS} %-5p %-60c %x - %m%n

# suppress the irrelevant (wrong) warnings from the netty channel handler
log4j.logger.org.jboss.netty.channel.DefaultChannelPipeline=ERROR, file

# rsyslog
# configure Syslog facility SYSLOG appender
# TODO: replace host and myTag by your own
log4j.appender.SYSLOG=org.apache.log4j.net.SyslogAppender
log4j.appender.SYSLOG.syslogHost=10.10.10.102
log4j.appender.SYSLOG.port=514
#log4j.appender.SYSLOG.appName=bbdata
log4j.appender.SYSLOG.layout=org.apache.log4j.EnhancedPatternLayout
log4j.appender.SYSLOG.layout.conversionPattern=myTag: [%p] %c:%L - %m %throwable %n

Düzen önemlidir, çünkü rsyslog yeni bir satırı yeni bir günlük girişi olarak değerlendirir. Yukarıda, yeni satırlar (örneğin yığın izlerinde) atlanacaktır. Gerçekten çok satırlı/sekmeli günlüklerin “normal” çalışmasını istiyorsanız, rsyslog.conf dosyasını düzenleyin ve şunu ekleyin:

$EscapeControlCharactersOnReceive off

Tüm günlüklerinizi belirli bir dosyaya yeniden yönlendirmek istiyorsanız, “conversionPattern” öğesinin başında “myTag:” öğesinin kullanılması yararlıdır. Bunu yapmak için “rsyslog.conf” dosyasını düzenleyin ve aşağıdaki kuralı ekleyin:

if $programname == 'myTag' then /var/log/my-app.log
& stop
