HTTP API’leri, geniş bir HTTP fiilleri yelpazesi kullanır ve genellikle JSON veya XML yanıtları döndürür.

Kaynak oluştur #

Kaynak yaratmanın anlamsal olarak en doğru yönteminin ne olduğu konusunda herkes hemfikir değildir. Böylece, API’niz “POST” veya “PUT” isteklerini veya her ikisini de kabul edebilir.

Kaynak başarıyla oluşturulduysa, sunucu ‘201 Created’ ile yanıt vermelidir. Değilse en uygun hata kodunu seçin.

Örneğin, çalışan kayıtları oluşturmak için bir API sağlarsanız, istek/yanıt şöyle görünebilir:

POST /employees HTTP/1.1
Host: example.com
Content-Type: application/json

{
    "name": "Charlie Smith",
    "age": 38,
    "job_title": "Software Developer",
    "salary": 54895.00
}

HTTP/1.1 201 Created
Location: /employees/1/charlie-smith
Content-Type: application/json

{
    "employee": {
        "name": "Charlie Smith",
        "age": 38,
        "job_title": "Software Developer",
        "salary": 54895.00
        "links": [
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "self",
                "method": "GET"
            },
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "delete",
                "method": "DELETE"
            },
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "edit",
                "method": "PATCH"
            }
        ]
    },
    "links": [
        {
            "uri": "/employees",
            "rel": "create",
            "method": "POST"
        }
    ]
}

Yanıtta “bağlantılar” JSON alanlarının dahil edilmesi, müşterinin URI’lerini veya yöntemlerini önceden bilmek zorunda kalmadan yeni kaynakla ve bir bütün olarak uygulamayla ilgili kaynağa erişmesini sağlar.

Bir kaynağı düzenleyin #

Bir kaynağı düzenlemek veya güncellemek, API’ler için ortak bir amaçtır. Düzenlemeler, ilgili kaynağa “POST”, “PUT” veya “PATCH” istekleri gönderilerek gerçekleştirilebilir. ‘POST’ bir kaynağın mevcut temsiline veri eklemesine izin verilir olmasına rağmen, daha açık bir anlam ifade ettikleri için ‘PUT’ veya ‘PATCH’ kullanılması önerilir.

Sunucunuz güncelleme yapıldıysa ‘200 OK’, henüz uygulanmadıysa ‘202 Kabul Edildi’ yanıtını vermelidir. Tamamlanamazsa en uygun hata kodunu seçin.

Tam güncellemeler #

‘PUT’, mevcut temsilin isteğe dahil edilen yük ile değiştirilmesi anlamına gelir. Yük, güncellenecek kaynağın mevcut temsiliyle aynı temsil türünde değilse, sunucu hangi yaklaşımın uygulanacağına karar verebilir. RFC7231, sunucunun şunları yapabileceğini tanımlar:

  • Hedef kaynağı yeni medya türünü yansıtacak şekilde yeniden yapılandırın
  • Yeni kaynak durumu olarak kaydetmeden önce PUT gösterimini kaynağınkiyle tutarlı bir biçime dönüştürün
  • Hedef kaynağın belirli bir ortam türüyle (küme) sınırlı olduğunu belirten bir “415 Desteklenmeyen Medya Türü” yanıtıyla isteği reddedin.

Gibi bir JSON HAL temsilini içeren bir temel kaynak …

{
    "name": "Charlie Smith",
    "age": 39,
    "job_title": "Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    }
}

… bunun gibi bir güncelleme talebi alabilir

PUT /users/1234 HTTP/1.1
Host: http://www.acmee.com
Content-Type: "application/json; charset=utf-8"
Content-Length: 85
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer"
}

Sunucu artık kaynağın durumunu verilen istek gövdesiyle değiştirebilir ve ayrıca içerik türünü ‘application/hal+json’dan ‘application/json’a değiştirebilir veya JSON yükünü bir JSON HAL temsiline dönüştürebilir ve ardından kaynağın içeriği dönüştürülmüş olanla veya ‘415 Desteklenmeyen Ortam Türü’ yanıtıyla uygulanamayan bir ortam türü nedeniyle güncelleme isteğini reddeder.

İçeriği doğrudan değiştirmek veya önce temsili tanımlanan temsil modeline dönüştürmek ve ardından mevcut içeriği dönüştürülmüş olanla değiştirmek arasında bir fark vardır. Sonraki bir “GET” isteği, doğrudan değiştirmede aşağıdaki yanıtı döndürür:

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer"
}

dönüşüm ve ardından değiştirme yaklaşımı aşağıdaki temsili döndürür:

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: e0023aa4e

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    }
}

Yan etkiler #

Belirsiz işlem olarak tanımlansa da “PUT"un yan etkilere sahip olmasına izin verildiğini unutmayın! Bu, RFC7231 içinde şu şekilde belgelenmiştir:

Hedef kaynağa uygulanan bir PUT isteğinin diğer kaynaklar üzerinde yan etkileri olabilir. Örneğin, bir makale, her belirli sürümü tanımlayan URI’lerden (bir noktada geçerli sürüm kaynağıyla aynı durumu paylaşan farklı kaynaklar) ayrı olan “geçerli sürümü” (bir kaynak) tanımlamak için bir URI’ye sahip olabilir. “Geçerli sürüm” URI’sinde başarılı bir PUT isteği, bu nedenle, hedef kaynağın durumunu değiştirmeye ek olarak yeni bir sürüm kaynağı oluşturabilir ve ayrıca ilgili kaynaklar arasında bağlantıların eklenmesine neden olabilir.

Ek günlük girdileri üretmek, genellikle bir kaynağın kesinlikle bir durumu olmadığı için genellikle yan etki olarak kabul edilmez.

Kısmi güncellemeler #

RFC7231, kısmi güncellemelerle ilgili olarak bundan bahseder:

Kısmi içerik güncellemeleri, daha büyük kaynağın bir kısmıyla örtüşen durumla ayrı olarak tanımlanmış bir kaynağı hedefleyerek veya kısmi güncellemeler için özel olarak tanımlanmış farklı bir yöntem kullanarak (örneğin, [RFC5789][‘da tanımlanan PATCH yöntemi]) mümkündür. 3]).

Bu nedenle kısmi güncellemeler iki şekilde gerçekleştirilebilir:

  • Bir kaynağa birden çok daha küçük alt kaynak gömmesini sağlayın ve “PUT” aracılığıyla tam kaynak yerine yalnızca ilgili alt kaynağı güncelleyin
  • PATCH kullanarak sunucuya neyin güncelleneceğini söyleyin

Çakışan durumla kısmi güncelleme #

Bir kullanıcının başka bir konuma taşınması nedeniyle bir kullanıcı temsilinin kısmen güncellenmesi gerekiyorsa, kullanıcıyı doğrudan güncellemek yerine, ilgili kaynağın doğrudan güncellenmesi gerekir ki bu da kullanıcı temsilinin kısmen güncellenmesine yansır.

Hareketten önce bir kullanıcı aşağıdaki temsile sahipti

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept: application/hal+json; charset=utf-8
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    },
    "_embedded": {
        "ea:address": {
            "street": "Terrace Drive, Central Park",
            "zip": "NY 10024"
            "city": "New York",
            "country": "United States of America",
            "_links": {
                "self": { "href": "/address/abc" },
                "google_maps": { "href": "http://maps.google.com/?ll=40.7739166,-73.970176" }
            }
        }
    }
}

Kullanıcı yeni bir konuma taşınırken, konum bilgilerini şu şekilde günceller:

PUT /address/abc HTTP/1.1
Host: http://www.acmee.com
Content-Type: "application/json; charset=utf-8"
Content-Length: 109
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)

{
    "street": "Standford Ave",
    "zip": "CA 94306",
    "city": "Pablo Alto",
    "country": "United States of America"
}

Yukarıda açıklandığı gibi, mevcut adres kaynağı ile istekteki adres arasındaki uyumsuz ortam türü için değiştirmeden önce dönüştürme semantiğiyle, adres kaynağı şimdi güncellenir ve bu, kullanıcı kaynağı kullanıcının yeni adresi döndürülür.

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept: application/hal+json; charset=utf-8
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]

    "_embedded": {

Kısmi verileri yamalama #

“PATCH”, RFC5789 içinde tanımlanmıştır ve kendi başına HTTP spesifikasyonunun doğrudan bir parçası değildir. Yaygın bir yanlış inanış, bir ‘PATCH’ isteği içinde yalnızca kısmen güncellenmesi gereken alanları göndermenin yeterlidir olduğudur. Spesifikasyon bu nedenle belirtir

PATCH yöntemi, istek varlığında açıklanan bir dizi değişikliğin İstek-URI’si tarafından tanımlanan kaynağa uygulanmasını talep eder. Değişiklik kümesi, bir ortam türü tarafından tanımlanan “yama belgesi” adı verilen bir biçimde temsil edilir.

Bu, bir istemcinin kaynağı A durumundan B durumuna dönüştürmek için gerekli adımları hesaplaması ve bu talimatları sunucuya göndermesi gerektiği anlamına gelir.

Yama için popüler bir JSON tabanlı medya türü JSON Patch’tir.

Örnek kullanıcımızın yaşı ve iş unvanı değişirse ve kullanıcının gelirini temsil eden ek bir alan eklenmesi gerekiyorsa, JSON Patch kullanılarak PATCH kullanılarak kısmi bir güncelleme şöyle görünebilir:

PATCH /users/1234 HTTP/1.1
Host: http://www.acmee.com
Content-Type: application/json-patch+json; charset=utf-8
Content-Length: 188
Accept: application/json
If-Match: "e0023aa4e"

[
    { "op": "replace", "path": "/age", "value": 40 },
    { "op": "replace", "path": "/job_title", "value": "Senior Software Developer" },
    { "op": "add", "path": "/salery", "value": 63985.00 }
]

“PATCH”, birden fazla kaynağı aynı anda güncelleyebilir ve değişiklikleri atomik olarak uygulamayı gerektirir; bu, tüm değişikliklerin uygulanması gerektiği veya hiç yapılmaması gerektiği anlamına gelir ve bu da API uygulayıcısına işlem yükü getirir.

Başarılı bir güncelleme bunun gibi bir şey döndürebilir

HTTP/1.1 200 OK
Location: /users/1234
Content-Type: application/json
ETag: "df00eb258"

{
    "name": "Charlie Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "salary": 63985.00
}

ancak yalnızca “200 OK” yanıt koduyla sınırlı değildir.

Ara güncellemeleri önlemek için (temsil durumunun önceki getirmesi ile güncelleme arasında yapılan değişiklikler) ‘ETag’, ‘If-Match’ veya ‘If-Unmodified-Since’ başlığı kullanılmalıdır.

Hata yönetimi #

“PATCH” ile ilgili teknik özellik, aşağıdaki hata işlemeyi önerir:

Tip Hata Kodu
Hatalı biçimlendirilmiş yama belgesi 400 Hatalı İstek
Desteklenmeyen yama belgesi 415 Desteklenmeyen Ortam Türü
İşlenemeyen istek, yani kaynak, yama 422 İşlenemeyen Varlık
Kaynak bulunamadı 404 Bulunamadı
Çakışan durum, yani mevcut olmayan bir alanın yeniden adlandırılması (taşınması) 409 Çatışma
Çakışan değişiklik, yani istemci, doğrulamanın başarısız olduğu bir “If-Match” veya “If-Unmodified-Since” başlığı kullanıyorsa. Herhangi bir ön koşul yoksa, sonraki hata kodu döndürülmelidir ‘412 Ön Koşul Başarısız’ veya ‘409 Çakışma’
Eşzamanlı değişiklik, yani başka PATCH istekleri kabul edilmeden önce talebin uygulanması gerekiyorsa 409 Çatışma

Bir kaynağı sil #

HTTP API’lerinin başka bir yaygın kullanımı, mevcut bir kaynağı silmektir. Bu genellikle “DELETE” istekleri kullanılarak yapılmalıdır.

Silme başarılı olduysa, sunucu 200 OK döndürmelidir; değilse uygun bir hata kodu.

Çalışanımız Charlie Smith şirketten ayrıldıysa ve şimdi kayıtlarını silmek istiyorsak, bu şöyle görünebilir:

DELETE /employees/1/charlie-smith HTTP/1.1
Host: example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
    'links': [
        {
            'uri': '/employees',
            'rel': 'create',
            'method': 'POST'
        }
    ]
}

Kaynakları listele #

HTTP API’lerinin son yaygın kullanımı, sunucudaki mevcut kaynakların bir listesini elde etmektir. Bunun gibi listeler, yalnızca verileri aldıkları için ‘GET’ istekleri kullanılarak elde edilmelidir.

Sunucu, listeyi sağlayabiliyorsa 200 OK veya sağlayamıyorsa uygun bir hata kodu vermelidir.

O zaman çalışanlarımızı listelemek şöyle görünebilir:

                },

                    'rel': 'delete',
                    'method': 'DELETE'
                },
                {
                    'uri': '/employees/1/charlie-smith',
                    'rel': 'edit',
                    'method': 'PATCH'
                }
            ]
        },
        {
            'name': 'Donna Prima',
            'age': 30,
            'job_title': 'QA Tester',
            'salary': 77095.00
            'links': [
                {
                    'uri': '/employees/2/donna-prima',
                    'rel': 'self',
                    'method': 'GET'
                },
                {
                    'uri': '/employees/2/donna-prima',
                    'rel': 'delete',
                    'method': 'DELETE'
                },
                {
                    'uri': '/employees/2/donna-prima',
