Uygulamanızı dağıtma #

Rails uygulamanızı içeren dizinde olduğunuzdan emin olun, ardından Heroku’da bir uygulama oluşturun.

$ heroku create example
Creating ⬢ example... done
https://example.herokuapp.com/ | https://git.heroku.com/example.git

Çıktının ilk URL’si olan http://example.herokuapp.com, uygulamanın mevcut olduğu konumdur. İkinci URL, git@heroku.com:example.git, uzak git deposu URL’sidir.

Bu komut yalnızca başlatılmış bir git deposunda kullanılmalıdır. Heroku create komutu, bu URL’yi işaret eden “heroku” adlı bir git uzaktan kumandasını otomatik olarak ekler.

Uygulama adı bağımsız değişkeni (“örnek”) isteğe bağlıdır. Uygulama adı belirtilmezse rastgele bir ad oluşturulur. Heroku uygulama adları global bir ad alanında olduğundan, “blog” veya “wiki” gibi genel adların zaten alınmış olmasını bekleyebilirsiniz. Varsayılan bir adla başlamak ve uygulamayı daha sonra yeniden adlandırmak genellikle daha kolaydır.

Uygulamanızda veritabanını kullanıyorsanız, aşağıdakileri çalıştırarak veritabanını manuel olarak taşımanız gerekir:

$ heroku run rake db:migrate

‘heroku çalıştırması’ndan sonraki herhangi bir komut, bir Heroku dyno’da yürütülecektir. Aşağıdakileri çalıştırarak etkileşimli bir kabuk oturumu elde edebilirsiniz:

$ heroku run bash

Web işlem türünü çalıştıran bir dyno’nuz olduğundan emin olun:

$ heroku ps:scale web=1

heroku ps komutu, uygulamanızın çalışan dinamiklerini listeler:

$ heroku ps
=== web (Standard-1X): bin/rails server -p $PORT -e $RAILS_ENV (1)
web.1: starting 2016/07/08 12:09:06 -0500 (~ 2s ago)

Artık uygulamayı ‘heroku open’ ile tarayıcımızda ziyaret edebilirsiniz.

$ heroku open

Heroku, “herokuapp.com” etki alanında size varsayılan bir web URL’si verir. Üretim için ölçeği büyütmeye hazır olduğunuzda kendi özel alan adınızı ekleyebilirsiniz.

Bir Heroku için Üretim ve hazırlama ortamlarını yönetme #

Her Heroku uygulaması en az iki ortamda çalışır: Heroku’da (buna üretim diyeceğiz) ve yerel makinenizde (geliştirme). Uygulama üzerinde birden fazla kişi çalışıyorsa, genellikle makine başına bir tane olmak üzere birden fazla geliştirme ortamınız vardır. Genellikle her geliştirici, testleri çalıştırmak için bir test ortamına da sahip olur. Ne yazık ki, bu yaklaşım, ortamlar daha az benzer hale geldikçe bozulur. Örneğin Windows ve Mac’ler, Heroku’daki Linux yığınından farklı ortamlar sağlar, bu nedenle yerel geliştirme ortamınızda çalışan kodun üretime dağıttığınızda aynı şekilde çalışacağından her zaman emin olamazsınız.

Çözüm, üretime mümkün olduğunca benzeyen bir hazırlama ortamına sahip olmaktır. Bu, hazırlama uygulamanızı barındıran ikinci bir Heroku uygulaması oluşturarak başarılabilir. Hazırlama ile kodunuzu, gerçek kullanıcılarınızı etkilemeden önce üretim benzeri bir ortamda kontrol edebilirsiniz.

En baştan başlamak

Yerel makinenizde çalışan bir uygulamanız olduğunu ve onu Heroku’ya göndermeye hazır olduğunuzu varsayalım. Hem uzak ortamlar, hem sahneleme hem de üretim oluşturmamız gerekecek. Önce sahnelemeye geçme alışkanlığını kazanmak için şununla başlayacağız:

$ heroku create --remote staging
Creating strong-river-216.... done
http://strong-river-216.heroku.com/ | https://git.heroku.com/strong-river-216.git
Git remote staging added

Varsayılan olarak, heroku CLI, bir heroku git uzaktan kumandasıyla projeler oluşturur. Burada –remote bayrağıyla farklı bir ad belirliyoruz, bu nedenle kodu Heroku’ya göndermek ve uygulamaya karşı komutları çalıştırmak normal git push heroku master’dan biraz farklı görünüyor:

 $ git push staging master
...
 $ heroku ps --remote staging
=== web: `bundle exec puma -C config/puma.rb``
web.1: up for 21s

Hazırlama uygulamanız düzgün şekilde çalışmaya başladığında, üretim uygulamanızı oluşturabilirsiniz:

$ heroku create --remote production
Creating fierce-ice-327.... done
http://fierce-ice-327.heroku.com/ | https://git.heroku.com/fierce-ice-327.git
Git remote production added
$ git push production master
...
$ heroku ps --remote production
=== web: `bundle exec puma -C config/puma.rb
web.1: up for 16s

Ve bununla birlikte, aynı şekilde ayarlanmış biri hazırlama ve biri üretim olmak üzere iki ayrı Heroku uygulaması olarak çalışan aynı kod tabanına sahip olursunuz. Sadece günlük işlerinizde hangi uygulamayı çalıştıracağınızı belirtmeniz gerekeceğini unutmayın. Varsayılan bir uygulama belirtmek için ‘–remote’ bayrağını kullanabilir veya git config’inizi kullanabilirsiniz.
