Rails Uygulamasında Hata Ayıklama #

Bir uygulamada hata ayıklayabilmek, bir uygulamanın mantığını ve veri akışını anlamak için çok önemlidir. Mantıksal hataların çözülmesine yardımcı olur ve programlama deneyimine ve kod kalitesine değer katar. Hata ayıklama için iki popüler mücevher, hata ayıklayıcı (yakut 1.9.2 ve 1.9.3 için) ve byebug (yakut >= 2.x için).

.rb dosyalarında hata ayıklamak için şu adımları izleyin:

  1. “Gemfile"nin “geliştirme” grubuna “hata ayıklayıcı” veya “güle güle” ekleyin
  2. ‘paket yükleme’yi çalıştırın
  3. Kesme noktası olarak “hata ayıklayıcı” veya “güle güle” ekleyin
  4. Kodu çalıştırın veya istekte bulunun
  5. Belirtilen kesme noktasında durdurulan Rails sunucusu günlüğüne bakın
  6. Bu noktada sunucu terminalinizi rails konsolu gibi kullanabilir ve değişken ve params değerlerini kontrol edebilirsiniz.
  7. Bir sonraki talimata geçmek için ’next’ yazın ve ’enter’ tuşuna basın
  8. Dışarı çıkmak için ‘c’ yazın ve ’enter’ tuşuna basın

.html.erb dosyalarında hata ayıklamak istiyorsanız, kesme noktası <% hata ayıklayıcı %> olarak eklenecektir.

Ruby on Rails Hızlıca Hata Ayıklama + Başlangıç ​​tavsiyesi #

İstisnaları yükselterek hata ayıklama, “print” günlük ifadelerine gözlerini kısarak bakmaktan çok daha kolaydır ve çoğu hata için, genellikle “pry” veya “byebug” gibi bir irb hata ayıklayıcısını açmaktan çok daha hızlıdır. Bu araçlar ilk adımınız olmamalıdır.

Ruby/Rails'de Hızlı Hata Ayıklama:

1. Hızlı Yöntem: Sonra bir İstisna oluşturun ve sonucunu .inspect

Ruby (özellikle Rails) kodunda hata ayıklamanın en hızlı yolu, yöntemde veya nesnede (örn. `foo```):

raise foo.inspect

Yukarıdaki kodda, raise, kodunuzun yürütülmesini durduran bir İstisnayı tetikler ve nesne/yöntem hakkında uygun şekilde “.inspect” (yani “foo”) bilgilerini içeren bir hata mesajı döndürür. hata ayıklamaya çalıştığınız satırda.

Bu teknik, bir nesneyi veya yöntemi hızlı bir şekilde incelemek (örneğin, “nil” mi?) ve belirli bir bağlamda bir kod satırının yürütülüp yürütülmediğini hemen doğrulamak için kullanışlıdır.

2. Geri dönüş: byebug veya pry

gibi bir Ruby IRB hata ayıklayıcı kullanın

Yalnızca kod yürütme akışınızın durumu hakkında bilgi sahibi olduktan sonra, yürütme yolunuzdaki nesnelerin durumunu daha derinlemesine inceleyebileceğiniz “pry” veya “byebug” gibi bir Ruby gem irb hata ayıklayıcısına geçmeyi düşünmelisiniz.

Rails’de hata ayıklamak için “byebug” mücevherini kullanmak için:

  1. Gemfile dosyanızdaki development grubunun içine gem 'byebug' ekleyin
  2. ‘paket yükleme’yi çalıştırın
  3. Ardından kullanmak için, incelenmesini istediğiniz kodun yürütme yoluna “byebug” ifadesini ekleyin.

Bu “byebug” değişkeni çalıştırıldığında, kodunuzun bir Ruby IRB oturumunu açarak, kodun yürütüldüğü noktada nesnelerin durumuna doğrudan erişmenizi sağlar.

Byebug gibi IRB hata ayıklayıcıları, yürütülürken kodunuzun durumunu derinlemesine analiz etmek için kullanışlıdır. Ancak, hataları artırmaya kıyasla daha fazla zaman alan prosedürlerdir, bu nedenle çoğu durumda ilk adımınız olmamalıdır.

Genel Başlangıç ​​Tavsiyesi

Bir sorunda hata ayıklamaya çalışırken, her zaman iyi bir tavsiye şudur: ** !@#$ing Hata Mesajını (RTFM) Okuyun**

Bu, hareket etmeden önce hata mesajlarını dikkatle ve tamamen okumak anlamına gelir, böylece size ne anlatmaya çalıştığını anlayabilirsiniz. Hata ayıklarken, bir hata mesajını okurken aşağıdaki zihinsel soruları bu sırayla sorun:

  1. Hata hangi sınıf‘a atıfta bulunuyor? (yani doğru nesne sınıfına sahip miyim yoksa nesnem “nil” mi?)
  2. Hata hangi yönteme başvuruyor? (yani, onların yöntemdeki bir türü; bu yöntemi bu tür/nesne sınıfında çağırabilir miyim?)
  3. Son olarak, son iki sorumdan çıkardığım sonucu kullanarak, hangi kod satırlarını araştırmalıyım? (unutmayın: yığın izlemedeki son kod satırı, sorunun bulunduğu yer değildir.)

Yığın izlemede projenizden gelen kod satırlarına özellikle dikkat edin (örneğin, Rails kullanıyorsanız ‘app/…’ ile başlayan satırlar). Sorunun% 99’u kendi kodunuzla ilgili.

Bu sırayla yorumlamanın neden önemli olduğunu göstermek için…

Örn. birçok yeni başlayanın kafasını karıştıran bir Ruby hata mesajı:

Bir noktada şu şekilde yürütülen kodu yürütürsünüz:

@foo = Foo.new

...

@foo.bar

ve şunu belirten bir hata alırsınız:

```Nil:nilClass```` için tanımsız yöntem “bar”

Yeni başlayanlar bu hatayı görür ve sorunun “bar” yönteminin tanımsız olması olduğunu düşünür. Değil. Bu hatada önemli olan kısım şudur:

for Nil:nilClass

for Nil:nilClass means that @foo is Nil! @foo is not a Foo instance variable! You have an object that is Nil. When you see this error, it’s simply ruby trying to tell you that the method bar doesn’t exist for objects of the class Nil. (well duh! since we are trying to use a method for an object of the class Foo not Nil).

Unfortunately, due to how this error is written (undefined method "bar" for Nil:nilClass) its easy to get tricked into thinking this error has to do with bar being undefined. When not read carefully this error causes beginners to mistakenly go digging into the details of the bar method on Foo, entirely missing the part of the error that hints that the object is of the wrong class (in this case: nil). It’s a mistake that’s easily avoided by reading error messages in their entirety.

Summary:

Always carefully read the entire error message before beginning any debugging. That means: Always check the class type of an object in an error message first, then its methods, before you begin sleuthing into any stacktrace or line of code where you think the error may be occurring. Those 5 seconds can save you 5 hours of frustration.

tl;dr: Don’t squint at print logs: raise exceptions instead. Avoid rabbit holes by reading errors carefully before debugging.

Debugging ruby-on-rails application with pry #

pry is a powerful tool that can be used to debug any ruby application. Setting up a ruby-on-rails application with this gem is very easy and straightforward.

Setup

To start debugging your application with pry

  • Add gem 'pry' to the application’s Gemfile and bundle it

    group :development, :test do gem ‘pry’ end

  • Navigate to the application’s root directory on terminal console and run bundle install. You’re all set to start using it anywhere on your application.

Use

Using pry in your application is just including binding.pry on the breakpoints you want to inspect while debugging. You can add binding.pry breakpoints anywhere in your application that is interpreted by ruby interpreter (any app/controllers, app/models, app/views files)

i) Debugging a Controller

app/controllers/users_controller.rb

class UsersController < ApplicationController
  def show
    use_id = params[:id]
    // breakpoint to inspect if the action is receiving param as expected
    binding.pry
    @user = User.find(user_id)
    respond_to do |format|
      format.html
    end
  end
end

In this example, the rails server pauses with a pry console at the break-point when you try to visit a page routing to show action on UsersController. You can inspect params object and make ActiveRecord query on User model from that breakpoint

ii) Debugging a View

app/views/users/show.html.haml

%table
  %tbody
    %tr
      %td ID
      %td= @user.id
    %tr
      %td email
      %td= @user.email
    %tr
      %td logged in ?          
      %td
        - binding.pry
        - if @user.logged_in?
          %p= "Logged in"
        - else
          %p= "Logged out"

In this example, the break-point pauses with pry console when the users/show page is pre-compiled in the rails server before sending it back to the client’s browser. This break-point allows to debug correctness of @user.logged_in? when it is misbehaving.

ii) Debugging a Model

app/models/user.rb

class User < ActiveRecord::Base    
  def full_name
    binding.pry
    "#{self.first_name} #{self.last_name}"
  end      
end

In this example, the break-point can be used to debug User model’s instance method full_name when this method is called from anywhere in the application.

In conclusion, pry is a powerful debugging tool for rails application with easy setup and straightforward debugging guideline. Give this a try.
