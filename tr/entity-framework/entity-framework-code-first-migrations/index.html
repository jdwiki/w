Taşımayı Etkinleştir #

Varlık çerçevesinde Code First Migrations’ı etkinleştirmek için şu komutu kullanın:

Enable-Migrations

Paket Yöneticisi Konsolu üzerinde.

EF tarafından yönetilen veritabanı nesnelerinizi içeren geçerli bir “DbContext” uygulamasına sahip olmanız gerekir. Bu örnekte, veritabanı bağlamı “BlogPost” ve “Yazar” nesnelerini içerecektir:

internal class DatabaseContext: DbContext
{
    public DbSet<Author> Authors { get; set; }

    public DbSet<BlogPost> BlogPosts { get; set; }
}

Komutu yürüttükten sonra aşağıdaki çıktı görünmelidir:

PM> Enable-Migrations
Checking if the context targets an existing database...
Code First Migrations enabled for project <YourProjectName>.
PM>

Ayrıca, içinde tek bir Configuration.cs dosyası bulunan yeni bir Migrations klasörü görünmelidir:

Sonraki adım, ilk veritabanını oluşturacak ilk veritabanı geçiş komut dosyanızı oluşturmak olacaktır (sonraki örneğe bakın).

İlk geçişinizi ekleyin #

Taşımaları etkinleştirdikten sonra (lütfen [bu örneğe1 bakın) tüm veritabanı tablolarının, dizinlerinin ve bağlantılarının ilk kez oluşturulmasını içeren ilk taşıma işleminizi oluşturabilirsiniz.

Komut kullanılarak bir geçiş oluşturulabilir.

Add-Migration <migration-name>

Bu komut, geçişi uygulamak ve kaldırmak için kullanılan “Yukarı” ve “Aşağı” olmak üzere iki yöntem içeren yeni bir sınıf oluşturacaktır.

Şimdi Initial: adlı bir geçiş oluşturmak için yukarıdaki örneğe göre komutu uygulayın.

PM> Add-Migration Initial
Scaffolding migration 'Initial'.
The Designer Code for this migration file includes a snapshot of your current Code
First model. This snapshot is used to calculate the changes to your model when you
scaffold the next migration. If you make additional changes to your model that you 
want to include in this migration, then you can re-scaffold it by running 
'Add-Migration Initial' again.

Yeni bir timestamp_Initial.cs dosyası oluşturulur (burada yalnızca önemli şeyler gösterilir):

public override void Up()
{
   CreateTable(
       "dbo.Authors",
        c => new
           {
               AuthorId = c.Int(nullable: false, identity: true),
               Name = c.String(maxLength: 128),
           })
        .PrimaryKey(t => t.AuthorId);
        
    CreateTable(
       "dbo.BlogPosts",
       c => new
           {
                Id = c.Int(nullable: false, identity: true),
                Title = c.String(nullable: false, maxLength: 128),
                Message = c.String(),
                Author_AuthorId = c.Int(),
            })
        .PrimaryKey(t => t.Id)
        .ForeignKey("dbo.Authors", t => t.Author_AuthorId)
        .Index(t => t.Author_AuthorId);
}
    
public override void Down()
{
    DropForeignKey("dbo.BlogPosts", "Author_AuthorId", "dbo.Authors");
    DropIndex("dbo.BlogPosts", new[] { "Author_AuthorId" });
    DropTable("dbo.BlogPosts");
    DropTable("dbo.Authors");
}

Görüldüğü gibi Up() metodunda Authors ve BlogPosts adlı iki tablo oluşturulmakta ve alanlar buna göre oluşturulmaktadır. Ayrıca Yazar_YazarKimliği alanı eklenerek iki tablo arasındaki ilişki oluşturulur. Diğer tarafta, Down() yöntemi, geçiş etkinliklerini tersine çevirmeye çalışır.

Geçişinizden eminseniz, aşağıdaki komutu kullanarak geçişi veritabanına uygulayabilirsiniz:

Update-Database

Bekleyen tüm geçişler (bu durumda Initial-migration) veritabanına uygulanır ve ardından tohum yöntemi uygulanır (uygun örnek)

PM> update-database
Specify the '-Verbose' flag to view the SQL statements being applied to the target     
database.
Applying explicit migrations: [201609302203541_Initial].
Applying explicit migration: 201609302203541_Initial.
Running Seed method.

SQL gezgininde etkinliklerin sonuçlarını görebilirsiniz: ![veritabanına genel bakış][2]

get-help Add-Migration

ve

get-help Update-Database

Taşıma sırasında Verileri Tohumlama #

Geçişleri etkinleştirdikten ve oluşturduktan sonra, ilk olarak veritabanınızdaki verileri doldurmanız veya taşımanız gerekebilir. Birkaç olasılık vardır ancak basit geçişler için “enable-migrations” çağrıldıktan sonra oluşturulan Yapılandırma dosyasındaki “Seed()” yöntemini kullanabilirsiniz.

‘Seed()’ işlevi, yalnızca parametre olduğu için bir veritabanı bağlamı alır ve bu işlev içinde EF işlemlerini gerçekleştirebilirsiniz:

protected override void Seed(Model.DatabaseContext context);

Seed() içerisinde her türlü aktiviteyi gerçekleştirebilirsiniz. Herhangi bir arıza durumunda, işlemin tamamı (uygulanan yamalar bile) geri alınır.

Yalnızca bir tablo boşsa veri oluşturan örnek bir işlev şöyle görünebilir:

protected override void Seed(Model.DatabaseContext context)
{
    if (!context.Customers.Any()) {
        Customer c = new Customer{ Id = 1, Name = "Demo" };
        context.Customers.Add(c);
        context.SaveData();
    }
}

EF geliştiricileri tarafından sağlanan güzel bir özellik, AddOrUpdate() uzantı yöntemidir. Bu yöntem, birincil anahtara dayalı olarak verilerin güncellenmesine veya mevcut değilse veri eklenmesine olanak tanır (örnek, Configuration.cs dosyasının oluşturulan kaynak kodundan alınmıştır):

protected override void Seed(Model.DatabaseContext context)
{
    context.People.AddOrUpdate(
      p => p.FullName,
      new Person { FullName = "Andrew Peters" },
      new Person { FullName = "Brice Lambson" },
      new Person { FullName = "Rowan Miller" }
    );
}

Lütfen son yama uygulandıktan sonra Seed() çağrıldığını unutmayın. Yamalar sırasında verileri taşımanız veya tohumlamanız gerekiyorsa, diğer yaklaşımların kullanılması gerekir.

Kodunuz içinde “Update-Database” yapmak #

Geliştirme dışı ortamlarda çalışan uygulamalar genellikle veritabanı güncellemeleri gerektirir. Veritabanı yamalarınızı oluşturmak için Geçiş Ekle komutunu kullandıktan sonra, güncellemeleri diğer ortamlarda ve ardından test ortamında çalıştırmanız gerekir.

Yaygın olarak karşılaşılan zorluklar şunlardır:

  • üretim ortamlarında Visual Studio yüklü değil ve
  • Gerçek hayatta bağlantı/müşteri ortamlarına bağlantı yapılmaz.

Bir geçici çözüm, gerçekleştirilecek güncellemeleri kontrol eden ve bunları sırayla yürüten aşağıdaki kod dizisidir. Hata durumunda hiçbir verinin kaybolmamasını sağlamak için lütfen uygun işlemlerin ve istisnaların ele alınmasını sağlayın.

void UpdateDatabase(MyDbConfiguration configuration) {
   DbMigrator dbMigrator = new DbMigrator( configuration);
   if ( dbMigrator.GetPendingMigrations().Any() )
   {
      // there are pending migrations run the migration job
      dbMigrator.Update(); 
   }
}

“MyDbConfiguration”, kaynaklarınızdaki bir yerde geçiş kurulumunuzdur:

public class MyDbConfiguration : DbMigrationsConfiguration<ApplicationDbContext>

Taşıma işlemleri sırasında Sql() kullanma #

Örneğin: Mevcut bir sütunu gerekli olmayandan gerekli olana taşıyacaksınız. Bu durumda, değiştirilen alanların aslında “NULL” olduğu satırlar için taşıma işleminizde bazı varsayılan değerleri doldurmanız gerekebilir. Varsayılan değerin basit olması durumunda (örneğin “0”), sütun tanımınızda bir “default” veya “defaultSql” özelliğini kullanabilirsiniz. O kadar kolay değilse, taşıma işleminizin ‘Up()’ veya ‘Down()’ üye işlevlerinde ‘Sql()’ işlevini kullanabilirsiniz.

İşte bir örnek. Veri kümesinin bir parçası olarak bir e-posta adresi içeren bir Yazar sınıfının varsayılması. Şimdi e-posta adresini zorunlu alan olarak almaya karar veriyoruz. Mevcut sütunları taşımak için işletmenin, tam adın yazarların boşluksuz tam adı olduğu “tam ad@example.com” gibi sahte e-posta adresleri oluşturma akıllı fikri vardır. “E-posta” alanına “[Zorunlu]” özelliğinin eklenmesi, aşağıdaki taşıma işlemini oluşturur:

public partial class AuthorsEmailRequired : DbMigration
{
    public override void Up()
    {
        AlterColumn("dbo.Authors", "Email", c => c.String(nullable: false, maxLength: 512));
    }
    
    public override void Down()
    {
        AlterColumn("dbo.Authors", "Email", c => c.String(maxLength: 512));
    }
}

Bu, veritabanının içinde bazı NULL alanların olması durumunda başarısız olur:

NULL değeri ‘Email’ sütununa, ‘App.Model.DatabaseContext.dbo.Authors’ tablosuna eklenemiyor; sütun boş değerlere izin vermez. GÜNCELLEME başarısız.

Aşağıdaki gibi before AlterColumn komutu eklemek yardımcı olacaktır:

Sql(@"Update dbo.Authors
    set Email = REPLACE(name, ' ', '') + N'@example.com'
    where Email is null");

‘update-database’ çağrısı başarılı olur ve tablo şöyle görünür (gösterilen örnek veriler):

Diğer Kullanım #

Veritabanınızdaki tüm DML ve DDL etkinlikleri için Sql() fonksiyonunu kullanabilirsiniz. Geçiş işleminin bir parçası olarak yürütülür; SQL başarısız olursa, tam geçiş başarısız olur ve bir geri alma yapılır.

İlk Entity Framework Kodu Adım Adım İlk Geçiş #

  1. Bir konsol uygulaması oluşturun.

  2. “Paket Yöneticisi Konsolu"nda Install-Package EntityFramework çalıştırarak EntityFramework nuget paketini kurun

  3. Bağlantı dizginizi app.config dosyasına ekleyin, bağlantınıza providerName="System.Data.SqlClient" eklemeniz önemlidir.

  4. İstediğiniz gibi bir genel sınıf oluşturun, “Blog” gibi bir şey

  5. DbContext’ten devralan ContextClass’ınızı oluşturun, “BlogContext” gibi bir şey

  6. DbSet türü bağlamında bir özellik tanımlayın, bunun gibi bir şey:

    public class Blog { public int Id { get; set; }

     public string Name { get; set; }

    } public class BlogContext: DbContext { public BlogContext(): base(“name=Your_Connection_Name”) { }

     public virtual DbSet<Blog> Blogs{ get; set; }

    }

  7. Bağlantı adını yapıcıya iletmek önemlidir (burada Your_Connection_Name)

  8. Paket Yöneticisi Konsolunda Enable-Migration komutunu çalıştırın, Bu, projenizde bir geçiş klasörü oluşturacaktır.

  9. Add-Migration Your_Arbitrary_Migraiton_Name komutunu çalıştırın, bu, Up() ve Down() yöntemleriyle geçişler klasöründe bir geçiş sınıfı oluşturacaktır.

  10. Blog tablosu ile bir veritabanı oluşturmak için Update-Database komutunu çalıştırın.
