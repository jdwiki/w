AsNoTracking’i Kullanma #

Kötü örnek:

var location =  dbContext.Location
                     .Where(l => l.Location.ID == location_ID)
                     .SingleOrDefault();

return location;

Yukarıdaki kod, bir varlığı değiştirmeden veya eklemeden basitçe döndürdüğünden, izleme maliyetinden kaçınabiliriz.

İyi örnek:

var location =  dbContext.Location.AsNoTracking()
                     .Where(l => l.Location.ID == location_ID)
                     .SingleOrDefault();

return location;

AsNoTracking() işlevini kullandığımızda, Entity Framework’e varlıkların bağlam tarafından izlenmediğini açıkça söylüyoruz. Bu, özellikle veri deponuzdan büyük miktarda veri alırken faydalı olabilir. Ancak, izlenmeyen varlıklarda değişiklik yapmak istiyorsanız, ‘Değişiklikleri Kaydet’i çağırmadan önce bunları eklemeyi unutmamalısınız.

Yalnızca Gerekli Verileri Yükleme #

Kodda sıklıkla görülen bir sorun, tüm verileri yüklemektir. Bu, sunucudaki yükü büyük ölçüde artıracaktır.

Diyelim ki içinde 10 alan bulunan “konum” adında bir modelim var, ancak aynı anda tüm alanlar gerekli değil. Diyelim ki o modelin sadece ‘LocationName’ parametresini istiyorum.

Kötü örnek

var location =  dbContext.Location.AsNoTracking()
                         .Where(l => l.Location.ID == location_ID)
                         .SingleOrDefault();

return location.Name;

İyi örnek

var location =  dbContext.Location
                          .Where(l => l.Location.ID == location_ID)
                          .Select(l => l.LocationName);
                          .SingleOrDefault();

return location;

“İyi örnek"teki kod yalnızca ‘LocationName’i getirir, başka bir şey getirmez.

Bu örnekte hiçbir varlık somutlaştırılmadığından, AsNoTracking() gerekli değildir. Zaten izlenecek bir şey yok.

Anonim Türlerle daha fazla alan getirme

var location = dbContext.Location
                    .Where(l => l.Location.ID == location_ID)
                    .Select(l => new { Name = l.LocationName, Area = l.LocationArea })
                    .SingleOrDefault();

return location.Name + " has an area of " + location.Area;

Önceki örnekte olduğu gibi, veritabanından sadece ‘LocationName’ ve ‘LocationArea’ alanları alınacaktır, Anonymous Type istediğiniz kadar değeri tutabilir.

Sorguları mümkün olduğunda bellekte değil veritabanında yürütün. #

Teksas’ta kaç tane ilçe olduğunu saymak istediğimizi varsayalım:

var counties = dbContext.States.Single(s => s.Code == "tx").Counties.Count();

Sorgu doğru, ancak verimsiz. States.Single(…) veritabanından bir durum yükler. Ardından, “İlçeler”, 254 ilçenin tümünü, tüm alanlarıyla birlikte ikinci bir sorguda yükler. .Count() daha sonra yüklenen Counties koleksiyonunda bellekte gerçekleştirilir. İhtiyacımız olmayan birçok veri yükledik ve daha iyisini yapabiliriz:

var counties = dbContext.Counties.Count(c => c.State.Code == "tx");

Burada, SQL’de sayım ve birleştirme anlamına gelen yalnızca bir sorgu yapıyoruz. Veritabanından yalnızca sayıyı döndürürüz - dönen satırları, alanları ve nesnelerin oluşturulmasını kaydettik.

Koleksiyon türüne bakarak sorgunun nerede yapıldığını görmek kolaydır: ‘IQueryable’ ve ‘IEnumerable’.

Birden çok sorguyu zaman uyumsuz ve paralel olarak yürütün #

Zaman uyumsuz sorguları kullanırken, aynı anda birden çok sorgu çalıştırabilirsiniz, ancak aynı bağlamda çalıştıramazsınız. Bir sorgunun yürütme süresi 10s ise, kötü örneğin süresi 20s, iyi örneğin süresi ise 10s olacaktır.

Kötü örnek #

IEnumerable<TResult1> result1;
IEnumerable<TResult2> result2;

using(var context = new Context())
{
    result1 = await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
    result2 = await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
}

İyi örnek #

public async Task<IEnumerable<TResult>> GetResult<TResult>()
{
    using(var context = new Context())
    {
        return await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
    }
}



IEnumerable<TResult1> result1;
IEnumerable<TResult2> result2;

var result1Task = GetResult<TResult1>();
var result2Task = GetResult<TResult2>();

await Task.WhenAll(result1Task, result2Task).ConfigureAwait(false);

var result1 = result1Task.Result;
var result2 = result2Task.Result;

Saplama varlıklarıyla çalışma #

Çoktan çoğa ilişkide “Ürün” ve “Kategori"lerimiz olduğunu varsayalım:

public class Product
{
    public Product()
    {
        Categories = new HashSet<Category>();
    }
    public int ProductId { get; set; }
    public string ProductName { get; set; }
    public virtual ICollection<Category> Categories { get; private set; }
}

public class Category
{
    public Category()
    {
        Products = new HashSet<Product>();
    }
    public int CategoryId { get; set; }
    public string CategoryName { get; set; }
    public virtual ICollection<Product> Products { get; set; }
}

Bir “Ürün"e “Kategori” eklemek istiyorsak, ürünü yüklemeli ve kategoriyi “Kategorilerine” eklemeliyiz, örneğin:

Kötü örnek:

var product = db.Products.Find(1);
var category = db.Categories.Find(2);
product.Categories.Add(category);
db.SaveChanges();

(burada “db” bir “DbContext” alt sınıfıdır).

Bu, bağlantı tablosunda “Ürün” ve “Kategori” arasında bir kayıt oluşturur. Ancak bu tablo yalnızca iki “Id” değeri içerir. Küçük bir kayıt oluşturmak için iki tam varlığı yüklemek kaynak israfıdır.

Daha verimli bir yol, saplama varlıklarını, yani bellekte oluşturulan, yalnızca minimum veriyi, genellikle yalnızca bir “Id” değerini içeren varlık nesnelerini kullanmaktır. Şuna benziyor:

İyi örnek:

// Create two stub entities
var product = new Product { ProductId = 1 };
var category = new Category { CategoryId = 2 };

// Attach the stub entities to the context
db.Entry(product).State = System.Data.Entity.EntityState.Unchanged;
db.Entry(category).State = System.Data.Entity.EntityState.Unchanged;

product.Categories.Add(category);
db.SaveChanges();

Sonuç aynıdır, ancak veritabanına iki gidiş dönüşü önler.

Yinelemeleri önleyin

İlişkinin zaten var olup olmadığını kontrol etmek istiyorsanız, ucuz bir sorgu yeterlidir. Örneğin:

var exists = db.Categories.Any(c => c.Id == 1 && c.Products.Any(p => p.Id == 14));

Yine, bu tam varlıkları belleğe yüklemez. Bağlantı tablosunu etkin bir şekilde sorgular ve yalnızca bir boole döndürür.

Değişiklik izlemeyi ve proxy oluşturmayı devre dışı bırakın #

Yalnızca veri almak, ancak hiçbir şeyi değiştirmek istemiyorsanız, değişiklik izlemeyi ve proxy oluşturmayı kapatabilirsiniz. Bu, performansınızı artıracak ve ayrıca tembel yüklemeyi önleyecektir.

Kötü örnek:

using(var context = new Context())
{
    return await context.Set<MyEntity>().ToListAsync().ConfigureAwait(false);
}

İyi örnek:

using(var context = new Context())
{
    context.Configuration.AutoDetectChangesEnabled = false;
    context.Configuration.ProxyCreationEnabled = false;

    return await context.Set<MyEntity>().ToListAsync().ConfigureAwait(false);
}

Bunları, özellikle bunların çözümünüz genelinde ayarlanmasını istiyorsanız, bağlamınızın yapıcısından kapatmak özellikle yaygındır:

public class MyContext : DbContext
{
    public MyContext()
        : base("MyContext")
    {
        Configuration.AutoDetectChangesEnabled = false;
        Configuration.ProxyCreationEnabled = false;
    }

    //snip
}
