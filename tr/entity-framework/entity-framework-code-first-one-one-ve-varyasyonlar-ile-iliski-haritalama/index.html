Entity Framework Code First One-one ve varyasyonları ile ilişki haritalama

Bu konu, Entity Framework kullanarak bire bir tür ilişkilerin nasıl eşleneceğini tartışır.

Bire sıfır veya bir eşleme #

Öyleyse tekrar diyelim ki aşağıdaki modele sahipsin:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
}

public class MyDemoContext : DbContext
{
  public DbSet<Person> People { get; set; }
  public DbSet<Car> Cars { get; set; }
}

Ve şimdi, aşağıdaki spesifikasyonu ifade edebilmeniz için onu ayarlamak istiyorsunuz: bir kişinin bir veya sıfır arabası olabilir ve her araba tam olarak bir kişiye aittir (ilişkiler çift yönlüdür, yani CarA, PersonA’ya aitse, o zaman PersonA ‘sahiptir). ‘CarA).

Öyleyse modeli biraz değiştirelim: navigasyon özelliklerini ve yabancı anahtar özelliklerini ekleyin:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
  public int CarId { get; set; }
  public virtual Car Car { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

Ve yapılandırma:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);                        
  }
}

Bu zamana kadar bu açıklayıcı olmalıdır. Arabada gerekli bir kişi var (HasRequired()), kişinin isteğe bağlı bir arabası var (WithOptional())). Yine, bu ilişkiyi hangi taraftan yapılandırdığınız önemli değil, Has/With ve Required/Optional’ın doğru kombinasyonunu kullanırken dikkatli olun. ‘Kişi’ tarafından, şöyle görünür:

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
     this.HasOptional(p => p.Car).WithOptional(c => c.Person);                        
  }
}

Şimdi db şemasını kontrol edelim:

Yakından bakın: ‘İnsanlar’da ‘Araba’ya atıfta bulunacak bir FK olmadığını görebilirsiniz. Ayrıca, ‘Car’daki FK, ‘PersonId’ değil, ‘CarId’dir. İşte FK için gerçek senaryo:

ALTER TABLE [dbo].[Cars]  WITH CHECK ADD  CONSTRAINT [FK_dbo.Cars_dbo.People_CarId] FOREIGN KEY([CarId])
REFERENCES [dbo].[People] ([PersonId])

Bu, modelde sahip olduğumuz ‘CarId’ ve ‘PersonId’ yabancı anahtar özelliklerinin temelde yok sayıldığı anlamına gelir. Veritabanındalar, ancak beklendiği gibi yabancı anahtarlar değiller. Bunun nedeni, bire bir eşlemelerin FK’yi EF modelinize eklemeyi desteklememesidir. Bunun nedeni, bire bir eşlemelerin ilişkisel bir veritabanında oldukça sorunlu olmasıdır.

Buradaki fikir, her insanın tam olarak bir arabası olabileceği ve o arabanın sadece o kişiye ait olabileceğidir. Veya kendileriyle ilişkili arabaları olmayan kişi kayıtları olabilir.

Peki bu yabancı anahtarlarla nasıl temsil edilebilir? Açıkçası, “Araba"da bir “PersonId” ve “People"da bir “CarId” olabilir. Her kişinin yalnızca bir arabası olabileceğini zorlamak için, “PersonId"nin “Araba"da benzersiz olması gerekir. Ancak, “PersonId”, “People"da benzersizse, “PersonId"nin “NULL” (sahibi olmayan birden fazla araba) olduğu iki veya daha fazla kaydı nasıl ekleyebilirsiniz? Cevap: yapamazsınız (aslında, SQL Server 2008 ve daha yeni sürümlerde filtrelenmiş benzersiz bir dizin oluşturabilirsiniz, ancak bir an için bu tekniği unutalım; diğer RDBMS’den bahsetmiyorum). İlişkinin iki ucunu da belirttiğiniz durumdan bahsetmiyorum bile…

‘People’ ve ‘Car’ tabloları ‘aynı’ birincil anahtara (bağlı kayıtlarda aynı değerler) sahipse, bu kuralı uygulamanın tek gerçek yolu. Ve bunu yapmak için, ‘Car’daki ‘CarId’, Halkın PK’sı için hem bir PK hem de bir FK olmalıdır. Bu da tüm şemayı karmakarışık hale getiriyor. Bunu kullandığımda, PK/FK’yi ‘Car’ ‘PersonId’ olarak adlandırmayı ve buna göre yapılandırmayı tercih ederim:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }        
  public virtual Car Car { get; set; }
}

public class Car
{        
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);
     this.HasKey(c => c.PersonId);
  }
}

İdeal değil, ama belki biraz daha iyi. Yine de, bu çözümü kullanırken dikkatli olmalısınız, çünkü sizi yoldan çıkarabilecek olağan adlandırma kurallarına aykırıdır. İşte bu modelden oluşturulan şema:

Dolayısıyla bu ilişki veritabanı şeması tarafından değil, Entity Framework’ün kendisi tarafından zorlanır. Bu yüzden bunu kullanırken çok dikkatli olmalısınız, kimsenin veritabanına doğrudan müdahale etmesine izin vermemelisiniz.

Bire bir eşleme #

Bire bir eşleme (her iki taraf da gerekli olduğunda) aynı zamanda zor bir iştir.

Bunun yabancı anahtarlarla nasıl temsil edilebileceğini hayal edelim. Yine, “People"da “Car"daki “CarId"e atıfta bulunan bir “CarId” ve Car’da “People"daki “PersonId"e atıfta bulunan bir “PersonId”.

Şimdi bir araba kaydı eklemek isterseniz ne olur? Bunun başarılı olması için, bu araba kaydında belirtilen bir ‘PersonId’ olmalıdır, çünkü gereklidir. Ve bu ‘PersonId’in geçerli olması için ‘People’da karşılık gelen kaydın bulunması gerekir. Tamam, devam edelim ve kişi kaydını ekleyelim. Ancak bunun başarılı olması için kişi kaydında geçerli bir “CarId” olmalıdır — ama o araba henüz takılmadı! Olamaz, çünkü önce sevk edilen kişi kaydını eklememiz gerekiyor. Ancak, araba kaydına atıfta bulunduğundan, yönlendirilen kişi kaydını ekleyemiyoruz, bu yüzden önce girilmesi gerekiyor (yabancı anahtar alımı :)).

Dolayısıyla bu da ‘mantıksal’ şekilde temsil edilemez. Yine yabancı anahtarlardan birini düşürmeniz gerekiyor. Hangisini düşüreceğiniz size kalmış. Yabancı anahtarın bırakıldığı tarafa ‘bağımlı’, yabancı anahtarın bulunmadığı tarafa ise ‘ana’ adı verilir. Ve yine, bağımlıda benzersizliği sağlamak için PK’nin FK olması gerekir, bu nedenle bir FK sütunu eklemek ve bunu modelinize aktarmak desteklenmez.

İşte yapılandırma:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasRequired(c => c.Person).WithRequiredDependent(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Şimdiye kadar gerçekten mantığını öğrenmiş olmalısınız :) Sadece diğer tarafı da seçebileceğinizi unutmayın, sadece WithRequired’in Dependent/Principal sürümlerini kullanmaya dikkat edin (ve yine de PK’yı Arabada yapılandırmanız gerekir).

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
    this.HasRequired(p => p.Car).WithRequiredPrincipal(c => c.Person);
  }
}

DB şemasını kontrol ederseniz, birebir veya sıfır çözümde olduğu gibi tamamen aynı olduğunu göreceksiniz. Bunun nedeni yine, bunun şema tarafından değil, EF’nin kendisi tarafından uygulanmasıdır. Yinede dikkatli ol :)

Bire veya sıfıra bire veya sıfıra eşleme #

Ve bitirmek için, her iki tarafın da isteğe bağlı olduğu duruma kısaca bakalım.

Şimdiye kadar bu örneklerden gerçekten sıkılmış olmalısınız :), bu yüzden ayrıntılara girmeyeceğim ve iki FK-s’ye sahip olma fikri ve olası sorunlarla oynamıyorum ve sizi bu kuralları uygulamamanın tehlikeleri konusunda uyarıyorum. şema ancak sadece EF’nin kendisinde.

İşte uygulamanız gereken yapılandırma:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasOptional(c => c.Person).WithOptionalPrincipal(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Yine diğer taraftan da yapılandırabilirsiniz, sadece doğru yöntemleri kullanmaya dikkat edin :)
