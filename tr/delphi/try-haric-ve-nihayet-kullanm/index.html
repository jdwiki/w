Sözdizimi #

  1. Try-except: [[[on E:ExceptionType do] dışında [ifadeleri] deneyin statement]] [else statement] | [statements] end;

    Try-finally: try [statements] finally [statements] end;

Yeni bir nesnenin istisnasız güvenli dönüşü #

Bir işlev bir nesneyi döndürdüğünde (arayan tarafından iletilen bir nesneyi kullanmaktan farklı olarak), bir istisnanın nesnenin sızmasına neden olmamasına dikkat edin.

function MakeStrings: TStrings;
begin
  // Create a new object before entering the try-block.
  Result := TStringList.Create;
  try
    // Execute code that uses the new object and prepares it for the caller.
    Result.Add('One');
    MightThrow;
  except
    // If execution reaches this point, then an exception has occurred. We cannot
    // know how to handle all possible exceptions, so we merely clean up the resources
    // allocated by this function and then re-raise the exception so the caller can
    // choose what to do with it.
    Result.Free;
    raise;
  end;
  // If execution reaches this point, then no exception has occurred, so the
  // function will return Result normally.
end;

Naif programcılar, tüm istisna türlerini yakalamaya ve böyle bir işlevden ’nil’ döndürmeye çalışabilirler, ancak bu, tüm istisna türlerini ele almadan yakalamanın genel olarak cesareti kırılmış uygulamasının özel bir durumudur.

Try-finally içinde yuvalanmış deneme hariç #

Bir “try”-“except” bloğu, bir “try”-“finally” bloğunun içine yuvalanmış olabilir.

AcquireResource;
try
  UseResource1;
  try
    UseResource2;
  except
    on E: EResourceUsageError do begin
      HandleResourceErrors;
    end;
  end;
  UseResource3;
finally
  ReleaseResource;
end;

“UseResource2” içinde bir “EResourceUsageError” oluşursa, yürütme istisna işleyicisine atlar ve “HandleResourceError"u çağırır. İstisna işlenmiş olarak kabul edilecektir, bu nedenle yürütme “UseResource3” ve ardından “ReleaseResource” ile devam edecektir.

“UseResource2” içinde başka türden bir istisna oluşursa, burada gösterilen istisna işleyici uygulanmaz, bu nedenle yürütme “UseResource3” çağrısının üzerinden atlayacak ve doğrudan “ReleaseResource"un çağrılacağı “finally” bloğuna gidecektir. . Bundan sonra, istisna çağrı yığınını kabarttığı için yürütme bir sonraki uygun istisna işleyicisine atlayacaktır.

Yukarıdaki örnekteki herhangi bir başka çağrıda bir istisna oluşursa, HandleResourceErrors çağrılmaz. Bunun nedeni, istisna işleyicisine karşılık gelen ’try’ bloğu içinde diğer çağrıların hiçbirinin gerçekleşmemesidir.

Basit deneme..son olarak bellek sızıntılarını önlemek için örnek #

Yürütme sırasında bir istisna oluşması durumunda kaynakların (bellek gibi) sızmasını önlemek için try-finally kullanın.

Aşağıdaki prosedür, bir dosyaya bir dize kaydeder ve TStringList sızmasını önler.

procedure SaveStringToFile(const aFilename: TFilename; const aString: string);
var
  SL: TStringList;
begin
  SL := TStringList.Create; // call outside the try 
  try
    SL.Text := aString;
    SL.SaveToFile(aFilename);
  finally
    SL.Free // will be called no matter what happens above
  end;
end;

Dosya kaydedilirken bir istisna oluşup oluşmadığına bakılmaksızın, SL serbest bırakılacaktır. Herhangi bir istisna arayan kişiye gider.

Sonunda 2 veya daha fazla nesneyle deneyin #

Object1 := nil;
Object2 := nil;
try
  Object1 := TMyObject.Create;
  Object2 := TMyObject.Create;
finally
  Object1.Free;
  Object2.Free;
end;

’try-finally’ bloğunun dışında ’nil’ olan nesneleri başlatmazsanız, bunlardan biri oluşturulamazsa, nihayet bloğunda bir AV oluşacaktır, çünkü nesne sıfır olmayacaktır (olduğu gibi). t başlatıldı) ve bir istisnaya neden olur.

“Serbest” yöntemi, nesnenin sıfır olup olmadığını kontrol eder, bu nedenle her iki nesneyi “nil” ile başlatmak, yaratılmamışlarsa serbest bırakırken hataları önler.

Try-nihayet iç içe geçmiş try-hariç #

Bir “try”-“finally” bloğu, bir “try”-“except” bloğunun içine yerleştirilebilir.

try
  AcquireResources;
  try
    UseResource;
  finally
    ReleaseResource;
  end;
except
  on E: EResourceUsageError do begin
    HandleResourceErrors;
  end;
end;

‘UseResource’ içinde bir istisna oluşursa, yürütme ‘ReleaseResource’a atlayacaktır. İstisna bir ‘EResourceUsageError’ ise, yürütme istisna işleyicisine atlayacak ve ‘HandleResourceErrors’u çağıracaktır. Diğer türdeki istisnalar, yukarıdaki istisna işleyiciyi atlayacak ve bir sonraki “dene”-“hariç"e kadar baloncuk oluşturacak ve çağrı yığınını bloke edecektir.

“AcquireResource” veya “ReleaseResource” içindeki istisnalar, ilgili “try” bloğu henüz girilmediği veya “finally” bloğunun zaten olması nedeniyle, yürütmenin istisna işleyicisine gitmesine ve “finally” bloğunun atlanmasına neden olur. girildi.
