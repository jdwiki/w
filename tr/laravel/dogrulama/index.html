Temel Örnek #

“validate” yöntemini (‘ValidatesRequests’ özelliği tarafından sağlanan temel Denetleyicide bulunur) kullanarak istek verilerini doğrulayabilirsiniz.

Kurallar geçerse, kodunuz normal şekilde çalışmaya devam eder; ancak doğrulama başarısız olursa, doğrulama hatalarını içeren bir hata yanıtı otomatik olarak geri gönderilir:

  • tipik HTML form istekleri için, kullanıcı, form gönderilen değerleri koruyarak önceki sayfaya yönlendirilecektir.
  • JSON yanıtı bekleyen istekler için 422 kodlu bir HTTP yanıtı oluşturulur

Örneğin, ‘UserController’ınızda, ‘mağaza’ yönteminde kaydetmeden önce doğrulama gerektiren yeni bir kullanıcı kaydediyor olabilirsiniz.

/**
 * @param  Request  $request
 * @return Response
 */
public function store(Request $request) {
    $this->validate($request, [
        'name' => 'required',
        'email' => 'email|unique:users|max:255'
    ],
    // second array of validation messages can be passed here
    [
        'name.required' => 'Please provide a valid name!',
        'email.required' => 'Please provide a valid email!',
    ]);

    // The validation passed
}

Yukarıdaki örnekte, “ad” alanının boş olmayan bir değerle var olduğunu doğrularız. İkinci olarak, email alanının geçerli bir e-posta formatına sahip olduğunu, “users” veritabanı tablosunda benzersiz olduğunu ve maksimum 255 karakter uzunluğunda olduğunu kontrol ederiz.

| (boru) karakteri, bir alan için farklı doğrulama kurallarını birleştirir.

Bazen, ilk doğrulama hatasından sonra bir öznitelik üzerinde doğrulama kurallarını çalıştırmayı durdurmak isteyebilirsiniz. Bunu yapmak için, özniteliğe “kefalet” kuralını atayın:

$this->validate($request, [
    'name' => 'bail|required',
    'email' => 'email|unique:users|max:255'
]);

Mevcut doğrulama kurallarının tam listesi aşağıdaki parametreler bölümünde bulunabilir.

Diğer Doğrulama Yaklaşımları #

1) Form Talebi Doğrulaması

Uygulamanızda belirli bir istek için yetkilendirme mantığını, doğrulama kurallarını ve hata mesajlarını tutabilen bir “form isteği” oluşturabilirsiniz.

‘make:request’ Artisan CLI komutu sınıfı oluşturur ve onu ‘app/Http/Requests’ dizinine yerleştirir:

php artisan make:request StoreBlogPostRequest

“yetkilendirme” yöntemi, bu istek için yetkilendirme mantığıyla geçersiz kılınabilir:

public function authorize()
{        
    return $this->user()->can('post');
}

“Kurallar” yöntemi, bu istek için belirli kurallarla geçersiz kılınabilir:

public function rules()
{
    return [
        'title' => 'required|unique:posts|max:255',
        'body' => 'required',
    ];
}

“mesajlar” yöntemi, bu istek için belirli mesajlarla geçersiz kılınabilir:

public function messages()
{
    return [
        'title.required' => 'A title is required',
        'title.unique' => 'There is another post with the same title',
        'title.max' => 'The title may not exceed :max characters',
        'body.required' => 'A message is required',
    ];
}

İsteği doğrulamak için, ilgili denetleyici yöntemine belirli istek sınıfını yazmanız yeterlidir. Doğrulama başarısız olursa, bir hata yanıtı geri gönderilir.

public function store(StoreBlogPostRequest $request)
{
    // validation passed
}

2) Doğrulayıcıları Manuel Olarak Oluşturma

Daha fazla esneklik için manuel olarak bir Doğrulayıcı oluşturmak ve başarısız doğrulamayı doğrudan ele almak isteyebilirsiniz:

<?php    
namespace App\Http\Controllers;

use Validator;
use Illuminate\Http\Request;
use App\Http\Controllers\Controller;

class PostController extends Controller
{
    public function store(Request $request)
    {
        $validator = Validator::make($request->all(), [
            'title' => 'required|unique:posts|max:255',
            'body' => 'required',
        ]);

        if ($validator->fails()) {
            return redirect('post/create')
                    ->withErrors($validator)
                    ->withInput();
        }

        // Store the blog post...
    }
}

2) Kuralları akıcı bir şekilde oluşturma

Bazen anında benzersiz kurallar oluşturmanız gerekebilir, bir Servis Sağlayıcı içinde ‘boot()’ yöntemiyle çalışmak aşırı olabilir, Laravel 5.4’ten itibaren ‘Rule’ sınıfını kullanarak akıcı bir şekilde yeni kurallar oluşturabilirsiniz.

Örnek olarak, bir kullanıcı eklemek veya güncellemek istediğinizde UserRequest ile çalışacağız. Şimdilik bir ismin gerekli olmasını ve e-posta adresinin benzersiz olmasını istiyoruz. “Benzersiz” kuralı kullanmanın sorunu, bir kullanıcıyı düzenliyorsanız, aynı e-postayı koruyabilmeleridir, bu nedenle mevcut kullanıcıyı kuralın dışında tutmanız gerekir. Aşağıdaki örnek, yeni ‘Rule’ sınıfını kullanarak bunu nasıl kolayca yapabileceğinizi gösterir.

<?php
namespace App\Http\Requests;
use Illuminate\Foundation\Http\FormRequest;
use Illuminate\Http\Request;
use Illuminate\Validation\Rule;

class UserRequest extends FormRequest
{
    /**
     * Determine if the user is authorized to make this request.
     *
     * @return bool
     */
    public function authorize()
    {
        return true;
    }

    /**
     * Get the validation rules that apply to the request.
     *
     * @return array
     */
    public function rules(Request $request)
    {
        $id = $request->route()->getParameter('user');

        return [
            'name'           =>  'required',
            
            // Notice the value is an array and not a string like usual
            'email'         =>  [
                'required',
                Rule::unique('users')->ignore($id)
            ]
        ];
    }
}

Dizi Doğrulama #

Dizi formu giriş alanlarını doğrulamak çok basittir.

Belirli bir dizideki her adı, e-postayı ve baba adını doğrulamanız gerektiğini varsayalım. Aşağıdakileri yapabilirsiniz:

$validator = \Validator::make($request->all(), [
    'name.*'       => 'required', 
    'email.*'      => 'email|unique:users',
    'fatherName.*' => 'required'
]);

if ($validator->fails()) {
    return back()->withInput()->withErrors($validator->errors());
}

Laravel, doğrulama için varsayılan mesajları görüntüler. Ancak dizi tabanlı alanlar için özel mesajlar istiyorsanız, aşağıdaki kodu ekleyebilirsiniz:

[
    'name.*' => [
        'required' => 'Name field is required',
    ],
    'email.*' => [
        'unique'   => 'Unique Email is required',
    ],
    'fatherName.*' => [
        'required' => 'Father Name required',
    ]
]

Son kodunuz şöyle görünecek:

$validator = \Validator::make($request->all(), [
    'name.*'       => 'required', 
    'email.*'      => 'email|unique:users',
    'fatherName.*' => 'required',
], [
    'name.*'       => 'Name Required',
    'email.*'      => 'Unique Email is required',
    'fatherName.*' => 'Father Name required',
]);

if ($validator->fails()) {
    return back()->withInput()->withErrors($validator->errors());
}

POST, PUT, PATCH için Tek Form İstek Sınıfı #

‘Form Request Validation’ örneğini takiben, aynı İstek Sınıfı ‘POST’, ‘PUT’, ‘PATCH’ için kullanılabilir, böylece aynı/benzer doğrulamaları kullanarak başka bir sınıf oluşturmanız gerekmez. Tablonuzda benzersiz niteliklere sahipseniz bu kullanışlı olur.

/**
 * Get the validation rules that apply to the request.
 *
 * @return array
 */
public function rules() {
    switch($this->method()) {
        case 'GET':
        case 'DELETE':
            return [];
        case 'POST':
            return [
                'name'     => 'required|max:75|unique',
                'category' => 'required',
                'price'    => 'required|between:0,1000',
            ];
        case 'PUT':
        case 'PATCH':
            return [
                'name'     => 'required|max:75|unique:product,name,' . $this->product,
                'category' => 'required',
                'price'    => 'required|between:0,1000',
            ];
        default:break;
    }
}

En baştan başlayarak, switch ifademiz isteğin metod tipine bakacak (GET,DELETE,POST,PUT,PATCH).

Yönteme bağlı olarak tanımlanan kurallar dizisini döndürür. Örnekteki “ad” alanı gibi benzersiz bir alanınız varsa, doğrulamanın yok sayması için belirli bir kimlik belirtmeniz gerekir.

'field_name' => 'unique:table_name,column_name,' . $idToIgnore`

‘id’ dışında bir şeyle etiketlenmiş bir birincil anahtarınız varsa, dördüncü parametre olarak birincil anahtar sütununu belirteceksiniz.

'field_name' => 'unique:table_name,column_name,' . $idToIgnore . ',primary_key_column'

Bu örnekte, PUT kullanıyoruz ve ürün kimliğinin değerini yola (admin/products/{product}) iletiyoruz. Yani “$this->product”, yoksayılacak “id"ye eşit olacaktır.

Artık PUT|PATCH ve POST doğrulama kurallarınızın aynı olması gerekmiyor. Gereksinimlerinize uyan mantığınızı tanımlayın. Bu teknik, özel Form İsteği Sınıfında tanımlamış olabileceğiniz özel iletileri yeniden kullanmanıza olanak tanır.

Hata mesajları #

Hata mesajlarını özelleştirme #

/resources/lang/[lang]/validation.php dosyaları doğrulayıcı tarafından kullanılacak hata mesajlarını içerir. Bunları gerektiği gibi düzenleyebilirsiniz.

Çoğunda, hata mesajı oluşturulurken otomatik olarak değiştirilecek olan yer tutucular bulunur.

Örneğin, 'required' => ':attribute alanı zorunludur.'' bölümünde, ':attribute' yer tutucusu alan adıyla değiştirilecektir (alternatif olarak, Aynı dosyadaki öznitelikler dizisi).

Örnek

mesaj yapılandırması:

'required' => 'Please inform your :attribute.',
//...
'attributes => [
    'email' => 'E-Mail address'
]

tüzük:

`email' => `required`

ortaya çıkan hata mesajı:

“Lütfen E-Posta adresinizi bildirin.”

Bir İstek sınıfı içinde hata mesajlarını özelleştirme #

Request sınıfının, bir dizi döndürmesi gereken bir “messages()” yöntemine erişimi vardır, bu, lang dosyalarına gitmek zorunda kalmadan iletileri geçersiz kılmak için kullanılabilir. Örneğin, özel bir ‘file_exists’ doğrulamamız varsa, aşağıdaki gibi mesajlar gönderebilirsiniz.

class SampleRequest extends Request {

    /**
     * Get the validation rules that apply to the request.
     *
     * @return array
     */
    public function rules()
    {
        return [
            'image' =>  'required|file_exists'
        ];
    }

    /**
     * Determine if the user is authorized to make this request.
     *
     * @return bool
     */
    public function authorize()
    {
        return true;
    }

    public function messages()
    {
        return [
            'image.file_exists' =>  'That file no longer exists or is invalid'
        ];
    }

}

Hata mesajlarını görüntüleme #

Doğrulama hataları oturuma yansıtılır ve ayrıca tüm görünümlerle otomatik olarak paylaşılan “$errors” değişkeninde bulunur.

Bir Blade görünümünde hataları görüntüleme örneği:

@if (count($errors) > 0)
    <div class="alert alert-danger">
        <ul>
            @foreach ($errors->all() as $error)
                <li>{{ $error }}</li>
            @endforeach
        </ul>
    </div>
@endif

Özel Doğrulama Kuralları #

Özel bir doğrulama kuralı oluşturmak istiyorsanız, örneğin Validator cephesi aracılığıyla bir servis sağlayıcının ‘önyükleme’ yönteminde bunu yapabilirsiniz.

<?php
namespace App\Providers;

use Illuminate\Support\ServiceProvider;
use Validator;

class AppServiceProvider extends ServiceProvider
{
    public function boot()
    {
        Validator::extend('starts_with', function($attribute, $value, $parameters, $validator) {
            return \Illuminate\Support\Str::startsWith($value, $parameters[0]);
        });

        Validator::replacer('starts_with', function($message, $attribute, $rule, $parameters) {
            return str_replace(':needle', $parameters[0], $message);
        });
    }
}

“Extend” yöntemi, kuralın adı olacak bir dize ve özniteliğin adını, doğrulanan değeri, kural parametrelerinin bir dizisini ve doğrulayıcı örneğini geçirecek bir işlev alır ve doğrulamanın geçip geçmediğini döndürür. Bu örnekte, değer dizesinin belirli bir alt dizeyle başlayıp başlamadığını kontrol ediyoruz.

Bu özel kural için hata mesajı, /resources/lang/[lang]/validation.php dosyasında her zamanki gibi ayarlanabilir ve örneğin parametre değerleri için yer tutucular içerebilir:

'starts_with' => 'The :attribute must start with :needle.'

‘replacer’ yöntemi, kuralın adı olan bir dize ve sırayla orijinal iletiye (değiştirmeden önce), özniteliğin adını, kuralın adını ve kuralın bir dizisini iletilecek olan bir işlev alır. parametreleri ve gerektiğinde yer tutucuları değiştirdikten sonra mesajı döndürmelidir.

Bu kuralı başka herhangi bir kural olarak kullanın:

$this->validate($request, [
    'phone_number' => 'required|starts_with:+'
]);
