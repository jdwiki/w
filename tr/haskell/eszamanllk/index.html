forkIO İle İplik Oluşturma #

Haskell, birçok eşzamanlılık biçimini destekler ve en belirgin olanı ‘forkIO’ kullanarak bir iş parçacığının çatallanmasıdır.

‘forkIO :: IO () -> IO ThreadId’ işlevi bir ‘IO’ eylemi gerçekleştirir ve ‘ThreadId’ değerini döndürür, bu arada eylem arka planda çalıştırılacaktır.

Bunu ghci’yi kullanarak oldukça özlü bir şekilde gösterebiliriz:

Prelude Control.Concurrent> forkIO $ (print . sum) [1..100000000]
ThreadId 290
Prelude Control.Concurrent> forkIO $ print "hi!"
"hi!"
-- some time later....
Prelude Control.Concurrent> 50000005000000

Her iki eylem de arka planda çalışacak ve ikincisinin sondan önce bitmesi neredeyse garanti!

‘MVar’ ile Konular Arasında Haberleşme #

‘MVar a’ türünü ve ‘Control.Concurrent’ içindeki eşlik eden işlevlerini kullanarak iş parçacıkları arasında bilgi aktarmak çok kolaydır:

  • newEmptyMVar :: IO (MVar a) – yeni bir MVar a oluşturur
  • newMVar :: a -> IO (MVar a) – verilen değerle yeni bir MVar oluşturur
  • takeMVar :: MVar a -> IO a – verilen MVardan değeri alır veya mevcut olana kadar bloklar
  • putMVar :: MVar a -> a -> IO () – verilen değeri MVar veya bloklar boşalana kadar koyar

1’den 100 milyona kadar olan sayıları bir iş parçacığında toplayalım ve sonucu bekleyelim:

import Control.Concurrent
main = do
  m <- newEmptyMVar
  forkIO $ putMVar m $ sum [1..10000000]
  print =<< takeMVar m  -- takeMVar will block 'til m is non-empty!

Daha karmaşık bir gösteri, daha fazla girdi beklerken arka planda kullanıcı girdisi ve toplamı almak olabilir:

main2 = loop
  where 
    loop = do
        m <- newEmptyMVar
        n <- getLine
        putStrLn "Calculating. Please wait"
        -- In another thread, parse the user input and sum
        forkIO $ putMVar m $ sum [1..(read n :: Int)]
        -- In another thread, wait 'til the sum's complete then print it
        forkIO $ print =<< takeMVar m
        loop

Daha önce belirtildiği gibi, ’takeMVar’ı çağırırsanız ve ‘MVar’ boşsa, başka bir iş parçacığı ‘MVar’a bir şey koyana kadar bloke olur, bu da bir Yemek Felsefecileri Problemi ile sonuçlanabilir. Aynı şey “putMVar” için de olur: eğer doluysa, boşalana kadar bloke eder!

Aşağıdaki işlevi alın:

concurrent ma mb = do
  a <- takeMVar ma
  b <- takeMVar mb
  putMVar ma a
  putMVar mb b

İki işlevi bazı ‘MVar’larla çalıştırıyoruz.

concurrent ma mb     -- new thread 1 
concurrent mb ma     -- new thread 2

Ne olabilir ki:

  1. Konu 1’de “ma” yazıyor ve “ma"yı engelliyor
  2. Konu 2 ‘mb’ okur ve böylece ‘mb’yi engeller

Şimdi Konu 1, Thread 2 bloke ettiği için mb' okuyamıyor ve Thread 2, Thread 1 bloke ettiği için ma’ okuyamıyor. Klasik bir kilitlenme!

Yazılım İşlem Belleğine Sahip Atom Blokları #

Haskell’deki bir diğer güçlü ve olgun eşzamanlılık aracı, birden çok iş parçacığının “TVar a” türündeki tek bir değişkene atomik bir şekilde yazmasına izin veren Yazılım İşlem Belleğidir.

TVar a, STM monad ile ilişkili ana tiptir ve işlem değişkeni anlamına gelir. MVar’a çok benzerler ancak aşağıdaki işlevler aracılığıyla ‘STM’ monad içinde kullanılırlar:

atomik olarak :: STM a -> IO a #

Atomik olarak bir dizi STM eylemi gerçekleştirin.

readTVar :: TVar a -> STM a #

“TVar” değerini okuyun, örneğin:

value <- readTVar t

writeTVar :: TVar a -> a -> STM () #

Verilen TVara bir değer yazın.

t <- newTVar Nothing
writeTVar t (Just "Hello")

Bu örnek Haskell Wiki’den alınmıştır:

import Control.Monad
import Control.Concurrent
import Control.Concurrent.STM
 
main = do 
  -- Initialise a new TVar
  shared <- atomically $ newTVar 0
  -- Read the value
  before <- atomRead shared
  putStrLn $ "Before: " ++ show before
  forkIO $ 25 `timesDo` (dispVar shared >> milliSleep 20)
  forkIO $ 10 `timesDo` (appV ((+) 2) shared >> milliSleep 50)
  forkIO $ 20 `timesDo` (appV pred shared >> milliSleep 25)
  milliSleep 800
  after <- atomRead shared
  putStrLn $ "After: " ++ show after
  where timesDo = replicateM_
       milliSleep = threadDelay . (*) 1000

atomRead = atomically . readTVar
dispVar x = atomRead x >>= print
appV fn x = atomically $ readTVar x >>= writeTVar x . fn
