x86 Linux Merhaba Dünya Örneği #

Bu, 32-bit x86 Linux için NASM derlemesinde sistem çağrılarını doğrudan kullanan (herhangi bir libc işlev çağrısı olmadan) temel bir Hello World programıdır. Alınması gereken çok şey var, ancak zamanla anlaşılır hale gelecektir. Noktalı virgül(;) ile başlayan satırlar yorumlardır.

Henüz düşük seviyeli Unix sistem programlamasını bilmiyorsanız, fonksiyonları asm’ye yazmak ve C veya C++ programlarından çağırmak isteyebilirsiniz. Ardından, POSIX sistem çağrısı API’sini ve onu kullanmak için ABI’yi öğrenmeden, kayıtları ve belleği nasıl kullanacağınızı öğrenmekle ilgilenebilirsiniz.

Bu, iki sistem çağrısı yapar: write(2) ve _exit(2) (stdio arabelleklerini temizleyen exit(3) libc sarmalayıcısı değil ve yakında). (Teknik olarak, _exit() sys_exit_group’u çağırır, sys_exit’i değil, ancak bu yalnızca çok iş parçacıklı bir süreçte önemlidir.) Sistem çağrılarıyla ilgili belgeler için ayrıca bkz. genel ve bunları doğrudan yapmak ile libc sarmalayıcı işlevlerini kullanmak arasındaki fark.

Özetle, sistem çağrıları, argümanları uygun kayıtlara ve sistem çağrı numarasını ’eax’a yerleştirerek ve ardından bir ‘int 0x80’ talimatı çalıştırarak yapılır. Asm sistem çağrısı arabiriminin çoğunlukla C sözdizimi ile nasıl belgelendiğine ilişkin daha fazla açıklama için ayrıca bkz. Montajdaki sistem çağrılarının dönüş değerleri nelerdir?.

32-bit ABI için sistem çağrı numaraları /usr/include/i386-linux-gnu/asm/unistd_32.h içindedir (/usr/include/x86_64-linux-gnu/asm/unistd_32.h içindeki aynı içerik). h).

#include <sys/syscall.h> sonunda doğru dosyayı içerecektir, böylece echo '#include <sys/syscall.h>' komutunu çalıştırabilirsiniz | gcc -E - -dM | less makro tanımlarını görmek için (C başlıklarında asm için sabitleri bulma hakkında daha fazla bilgi için bu cevaba bakın)

section .text             ; Executable code goes in the .text section
global _start             ; The linker looks for this symbol to set the process entry point, so execution start here
;;;a name followed by a colon defines a symbol.  The global _start directive modifies it so it's a global symbol, not just one that we can CALL or JMP to from inside the asm.
;;; note that _start isn't really a "function".  You can't return from it, and the kernel passes argc, argv, and env differently than main() would expect.
 _start:
    ;;; write(1, msg, len);
    ; Start by moving the arguments into registers, where the kernel will look for them
    mov     edx,len       ; 3rd arg goes in edx: buffer length
    mov     ecx,msg       ; 2nd arg goes in ecx: pointer to the buffer
    ;Set output to stdout (goes to your terminal, or wherever you redirect or pipe)
    mov     ebx,1         ; 1st arg goes in ebx: Unix file descriptor. 1 = stdout, which is normally connected to the terminal.

    mov     eax,4         ; system call number (from SYS_write / __NR_write from unistd_32.h).
    int     0x80          ; generate an interrupt, activating the kernel's system-call handling code.  64-bit code uses a different instruction, different registers, and different call numbers.
    ;; eax = return value, all other registers unchanged.

    ;;;Second, exit the process.  There's nothing to return to, so we can't use a ret instruction (like we could if this was main() or any function with a caller)
    ;;; If we don't exit, execution continues into whatever bytes are next in the memory page,
    ;;; typically leading to a segmentation fault because the padding 00 00 decodes to  add [eax],al.

    ;;; _exit(0);
    xor     ebx,ebx       ; first arg = exit status = 0.  (will be truncated to 8 bits).  Zeroing registers is a special case on x86, and mov ebx,0 would be less efficient.
                      ;; leaving out the zeroing of ebx would mean we exit(1), i.e. with an error status, since ebx still holds 1 from earlier.
    mov     eax,1         ; put __NR_exit into eax
    int     0x80          ;Execute the Linux function

section     .rodata       ; Section for read-only constants

             ;; msg is a label, and in this context doesn't need to be msg:.  It could be on a separate line.
             ;; db = Data Bytes: assemble some literal bytes into the output file.
msg     db  'Hello, world!',0xa     ; ASCII string constant plus a newline (0x10)

             ;;  No terminating zero byte is needed, because we're using write(), which takes a buffer + length instead of an implicit-length string.
             ;; To make this a C string that we could pass to puts or strlen, we'd need a terminating 0 byte. (e.g. "...", 0x10, 0)

len     equ $ - msg       ; Define an assemble-time constant (not stored by itself in the output file, but will appear as an immediate operand in insns that use it)
                          ; Calculate len = string length.  subtract the address of the start
                          ; of the string from the current position ($)
  ;; equivalently, we could have put a str_end: label after the string and done   len equ str_end - str

Linux’ta bu dosyayı Hello.asm olarak kaydedebilir ve şu komutlarla 32-bit yürütülebilir dosya oluşturabilirsiniz:

nasm -felf32 Hello.asm                  # assemble as 32-bit code.  Add -Worphan-labels -g -Fdwarf  for debug symbols and warnings
gcc -nostdlib -m32 Hello.o -o Hello     # link without CRT startup code or libc, making a static binary

NASM/YASM sözdizimi veya GNU AT&T sözdizimi için GNU as direktifleri için 32 veya 64-bit statik veya dinamik olarak bağlantılı Linux yürütülebilir dosyalarına derleme oluşturma hakkında daha fazla ayrıntı için bu yanıt’e bakın. (Önemli nokta: 64 bitlik bir ana bilgisayarda 32 bitlik kod oluştururken “-m32” veya eşdeğerini kullandığınızdan emin olun, aksi takdirde çalışma zamanında kafa karıştırıcı sorunlar yaşarsınız.)

Yaptığı sistem çağrılarını görmek için yürütmesini ‘strace’ ile takip edebilirsiniz:

$ strace ./Hello 
execve("./Hello", ["./Hello"], [/* 72 vars */]) = 0
[ Process PID=4019 runs in 32 bit mode. ]
write(1, "Hello, world!\n", 14Hello, world!
)         = 14
_exit(0)                                = ?
+++ exited with 0 +++

Stderr’deki izleme ve stdout’taki normal çıktının her ikisi de buradaki terminale gidiyor, bu nedenle ‘write’ sistem çağrısı satırına müdahale ediyorlar. Önemsiyorsanız, bir dosyaya yönlendirin veya izini sürün. Bunun, sistem çağrısı dönüş değerlerini yazdırmak için kod eklemek zorunda kalmadan nasıl kolayca görmemize izin verdiğine ve bunun için normal bir hata ayıklayıcı (gdb gibi) kullanmaktan bile daha kolay olduğuna dikkat edin.

Bu programın x86-64 sürümü, aynı argümanları sadece farklı kayıtlarda aynı sistem çağrılarına ileterek, son derece benzer olacaktır. Ve ‘int 0x80’ yerine ‘syscall’ talimatını kullanarak.

x86 Montaj Dili #

x86 derleme dilleri ailesi, orijinal Intel 8086 mimarisinde onlarca yıllık ilerlemeyi temsil eder. Kullanılan birleştiriciye bağlı olarak birkaç farklı lehçe bulunmasına ek olarak, 1980’lerde kullanılan 16 bitlik derlemeyle geriye dönük olarak uyumlu kalırken, yıllar içinde ek işlemci talimatları, kayıtlar ve diğer özellikler eklenmiştir.

x86 derlemesi ile çalışmanın ilk adımı, amacın ne olduğunu belirlemektir. Örneğin, bir işletim sistemi içinde kod yazmak istiyorsanız, C gibi daha yüksek seviyeli bir dilin tek başına bir derleyici mi yoksa yerleşik satır içi derleme özelliklerini mi kullanmayı seçeceğinizi ayrıca belirlemek isteyeceksiniz. Bir işletim sistemi olmadan “çıplak metal” üzerinde kodlama yapmak istiyorsanız, seçtiğiniz montajcıyı kurmanız ve flash belleğe, önyüklenebilir görüntüye dönüştürülebilen veya başka bir şekilde belleğe yüklenebilen ikili kodun nasıl oluşturulacağını anlamanız yeterlidir. yürütmeye başlamak için uygun yer.

Bir dizi platformda iyi desteklenen çok popüler bir montajcı, http://nasm.us/ adresinden edinilebilen NASM’dir (Netwide Assembler). NASM sitesinde, platformunuz için en son sürüm derlemesini indirmeye devam edebilirsiniz.

Pencereler

NASM’nin hem 32 bit hem de 64 bit sürümleri Windows için mevcuttur. NASM, birleştiriciyi otomatik olarak kurmak için Windows sunucunuzda kullanılabilecek uygun bir yükleyici ile birlikte gelir.

Linux

NASM, Linux sürümünüzde zaten yüklü olabilir. Kontrol etmek için yürütün:

nasm -v

Komut bulunamazsa, bir kurulum gerçekleştirmeniz gerekecektir. Son teknoloji NASM özellikleri gerektiren bir şey yapmıyorsanız, en iyi yol, NASM’yi yüklemek üzere Linux dağıtımınız için yerleşik paket yönetim aracınızı kullanmaktır. Örneğin, Ubuntu ve diğerleri gibi Debian’dan türetilen sistemler altında, bir komut isteminden aşağıdakileri yürütün:

sudo apt-get install nasm

RPM tabanlı sistemler için şunları deneyebilirsiniz:

sudo yum install nasm

Mac OS X

OS X’in son sürümleri (Yosemite ve El Capitan dahil), önceden yüklenmiş eski bir NASM sürümüyle birlikte gelir. Örneğin, El Capitan’ın 0.98.40 sürümü kuruludur. Bu muhtemelen neredeyse tüm normal amaçlar için çalışacak olsa da, aslında oldukça eskidir. Bu yazıda, NASM sürüm 2.11 yayınlandı ve 2.12’nin bir dizi yayın adayı var.

NASM kaynak kodunu yukarıdaki bağlantıdan edinebilirsiniz, ancak kaynaktan yüklemeye özel bir ihtiyacınız yoksa, ikili paketi OS X yayın dizininden indirmek ve sıkıştırmasını açmak çok daha kolaydır.

Sıkıştırmayı açtıktan sonra, NASM’nin sistem tarafından yüklenen sürümünün üzerine yazmamanız kesinlikle önerilir. Bunun yerine, /usr/local içine kurabilirsiniz:

 $ sudo su
 <user's password entered to become root>
 # cd /usr/local/bin
 # cp <path/to/unzipped/nasm/files/nasm> ./
 # exit

Bu noktada, NASM /usr/local/bin içindedir, ancak yolunuzda değildir. Şimdi profilinizin sonuna aşağıdaki satırı eklemelisiniz:

 $ echo 'export PATH=/usr/local/bin:$PATH' >> ~/.bash_profile

Bu, yolunuza /usr/local/bin ekleyecektir. Komut isteminde “nasm -v” yürütüldüğünde artık uygun, daha yeni sürüm görüntülenecektir.
