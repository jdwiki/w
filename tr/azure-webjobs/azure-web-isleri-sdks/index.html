Hatalarla tetiklenir #

Hata işleme son derece önemlidir, tetiklenen işlevlerinizden birinde bir yürütme hatası meydana geldiğinde tetiklenecek işlevleri tanımlayabiliriz:

//Fires when 10 errors occur in the last 30 minutes (sliding)
public static void ErrorMonitor([ErrorTrigger("0:30:00", 10)] TraceFilter filter)
{
        // get the last 5 errors
        filter.GetDetailedMessage(10);
}

Hata işlemeyi merkezileştirmek için özellikle kullanışlıdır.

ErrorTrigger, JobHostConfiguration’da ek bir ayar gerektirir:

config.UseCore();

NuGet paketini Microsoft.Azure.WebJobs.Extensions da yüklemelisiniz.

Ninject kullanarak Bağımlılık Enjeksiyonu #

Aşağıdaki örnek, bir IoC kapsayıcısı olarak Ninject kullanılarak Bağımlılık Enjeksiyonunun nasıl ayarlanacağını gösterir.

Önce WebJob projenize bir CustomModule sınıfı ekleyin ve orada herhangi bir bağımlılık bağlaması ekleyin.

public class CustomModule : NinjectModule
{
    public override void Load()
    {
        Bind<IMyInterface>().To<MyService>();
    }
}

Ardından bir JobActivator sınıfı oluşturun:

class JobActivator : IJobActivator
{
    private readonly IKernel _container;
    public JobActivator(IKernel container)
    {
        _container = container;
    }

    public T CreateInstance<T>()
    {
        return _container.Get<T>();
    }
}

JobHost’u Program sınıfının Ana işlevinde ayarladığınızda, JobActivator’ı JobHostConfiguration’a ekleyin.

public class Program
{
    private static void Main(string[] args)
    {
        //Set up DI
        var module = new CustomModule();
        var kernel = new StandardKernel(module);

        //Configure JobHost
        var storageConnectionString = "connection_string_goes_here";
        var config = new JobHostConfiguration(storageConnectionString) { JobActivator = new JobActivator(kernel) };

        //Pass configuration to JobJost
        var host = new JobHost(config);

        // The following code ensures that the WebJob will be running continuously
        host.RunAndBlock();
    }
}

Son olarak Functions.cs sınıfında hizmetlerinizi enjekte edin.

public class Functions
{
    private readonly IMyInterface _myService;

    public Functions(IMyInterface myService)
    {
        _myService = myService;
    }

    public void ProcessItem([QueueTrigger("queue_name")] string item)
    {
        _myService .Process(item);
    }
}

İşAna Bilgisayarı #

Azure Webjobs SDK, çalıştırılan İşlevleri tanımlamanıza yardımcı olmayı amaçlayan bir Nuget paketi olarak dağıtılan bir çerçevedir Tetikleyiciler tarafından ve diğer Azure hizmetlerine (Azure Storage ve Service Bus gibi) Bağlamalar‘ı bildirimsel bir şekilde kullanın.

SDK, kodlanmış İşlevlerinizi koordine etmek için bir JobHost kullanır. Tipik bir senaryoda, Web İşiniz, JobHost’u şu şekilde başlatan bir Konsol Uygulamasıdır:

class Program
{
    static void Main()
    {
        JobHostConfiguration config = new JobHostConfiguration();
        config.StorageConnectionString = "Your_Azure_Storage_ConnectionString";
        config.DashboardConnectionString = "Your_Azure_Storage_ConnectionString";
        JobHost host = new JobHost(config);
        host.RunAndBlock();
    }
}

JobHostConfiguration, farklı tetikleyiciler için daha fazla ayarı kişiselleştirmenize olanak tanır:

config.Queues.BatchSize = 8;
config.Queues.MaxDequeueCount = 4;
config.Queues.MaxPollingInterval = TimeSpan.FromSeconds(15);
config.JobActivator = new MyCustomJobActivator();

Kuyruklar için Tetikleyiciler #

Bir Kuyruk mesajı tarafından tetiklenen bir İşlevi tanımlayan basit bir örnek:

public static void StringMessage([QueueTrigger("my_queue")] string plainText)
{
    //...
}

Ayrıca [POCO] (https://en.wikipedia.org/wiki/Plain_Old_CLR_Object) serileştirmeyi de destekler:

public static void POCOMessage([QueueTrigger("my_queue")] MyPOCOClass aMessage)
{
    //...
}

config.UseTimers();

Ölçeklendirme #

Azure Web işleri, bir Azure App Service üzerinde çalışır. Uygulama Hizmetimizi yatay olarak ölçeklendirirsek (yeni örnekler eklersek), her örneğin kendi JobHost‘una sahip olacaktır.

Bunun yalnızca Sürekli modunda çalışan Webİşleri için geçerli olduğunu unutmayın. İsteğe bağlı ve zamanlanmış Web İşleri yatay ölçeklendirmeden etkilenmez, her zaman tek bir örnek çalıştırırlar.

Sürekli bir WebJob işleme kuyruğu mesajlarınız varsa ve App Service Plan’ı 3 örneğe ölçeklendirirseniz, çalışan WebJob’un 3 örneğine sahip olursunuz.

Tek bir örnekte çalıştırmak istediğiniz Web İşleri olabilir, çünkü tam olarak bir işleme hattının var olduğundan emin olmanız gerekebilir. Bu Web İşleri için Singleton özniteliğini ekleyebilirsiniz.

[Singleton]
public static void SingletonQueueProcessing([QueueTrigger("my_queue")] MyPOCOClass aMessage)
{
    //...
}

Bu, dağıtılmış kilitleme için Azure Blob Kiralamaları ile sağlanır.

Günlük Yazma #

WebJobs Panosu, günlükleri iki yerde gösterir: WebJob sayfası ve belirli bir WebJob çağrısı sayfası.

Bir işlevde veya Main() yönteminde çağırdığınız Konsol yöntemlerinden çıktılar, belirli bir yöntem çağırma sayfasında değil, Webİşi için Gösterge Tablosu sayfasında görünür. Metod imzanızdaki bir parametreden aldığınız TextWriter nesnesinin çıktısı, bir metod çağırma için Pano sayfasında görünür.

Uygulama izleme günlüklerini yazmak için, ‘Console.Out’ (INFO olarak işaretlenmiş günlükler oluşturur) ve ‘Console.Error’ (ERROR olarak işaretlenmiş günlükler oluşturur) kullanın.

public static void WriteLog([QueueTrigger("logqueue")] string message, TextWriter logger)
{
    Console.WriteLine("Console.Write - " + message);
    Console.Out.WriteLine("Console.Out - " + message);
    Console.Error.WriteLine("Console.Error - " + message);
    logger.WriteLine("TextWriter - " + message);
}

WebJob için Dashboard’da bu mesajlarla sonuçlanacak olan:

[07/28/2016 22:29:18 > 0a1c35: INFO] Console.Write - Hello world!
[07/28/2016 22:29:18 > 0a1c35: INFO] Console.Out - Hello world!
[07/28/2016 22:29:18 > 0a1c35: ERR ] Console.Error - Hello world!

Ve yöntem için Gösterge Tablosu sayfasındaki bu mesaj:

TextWriter - Hello world!
