JavaScript v4 SDK Kanalına Özgü Geri Aramalar #

PubNub JavaScript v3‘de, her kanal için abone olma işlevini çağırdığınız ve bu aboneliğe geri aramayı şu şekilde uyguladığınız sürece, abone olduğunuz her kanal için benzersiz bir geri arama uygulayabilirsiniz:

var pubnub = new PubNub({
    publishKey: "your-pub-key",
    subscribeKey: "your-sub-key"
});

pubnub.subscribe({
    channel: 'ch1',
    message: function (m) {
        console.log(m + " ch1 callback");
    }
});

pubnub.subscribe({
    channel: 'ch2',
    message: function (m) {
        console.log(m + " ch2 callback: ");
    }
});

Yani yukarıdaki kodda, “hello”mesajınıch1`e yayınlarsanız:

publish({channel: "ch1", message: "hello"});

…sonra “merhaba ch1 geri arama” çıktısını alırsınız. Ve tabii ki aynı mesajı ‘ch2’ye yayınlarsanız, ‘merhaba ch2 callback’ çıktısını alırsınız.

Her kanal veya bir kanal grubu için özel bir geri arama sağlama yeteneği, her koşul için yürütülecek uzun bir kanal adları ve kodu “eğer-ise-else” ile monolitik bir geri arama oluşturmaktan daha kullanışlıdır ve genellikle daha arzu edilir. Ayrıca, yukarıdaki basit örnekten daha iyi uygulamalar var, ancak PubNub JavaScript SDK v4 (ve herhangi bir v4 PubNub SDK) tasarımıyla karşılaştırmayı ve kontrast oluşturmayı kolaylaştırmak istedim.

PubNub v4 SDK’ları, tüm “message”, “presence” ve “status” olaylarını almak için tek bir dinleyici kullanır. Bu, bir kanala “abone olduğunuzda”, eşlik eden bir “mesaj geri arama” işlevi yerine yalnızca kanal adlarını verdiğiniz anlamına gelir. JavaScript SDK v4‘de dinleyici şöyle görünür:

pubnub.addListener({
    message: function(m) {
        console.log(JSON.stringify(m));
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(s) {
        console.log(JSON.stringify(s));
    }
});

Birçok geliştiricinin, yukarıda bahsettiğim hiç bitmeyen kanal adı koşullu koduna başvurmadan SDK v3’ten benzersiz “mesaj geri aramaları” ile “abone ol” kodunu bu tür bir tasarımla nasıl taşıyacağını merak edeceğini biliyorum:

pubnub.addListener({
    message: function(m) {
        if (m.subscribedChannel == 'ch1') {
            console.log(m.message + "ch1 callback");
        }
        else if (m.subscribedChannel == 'ch2') {
            console.log(m.message + "ch2 callback");
        }
        else {
            console.log(m.message + "default callback");
        }
    }
    // removed the other two callbacks for brevity purposes
});

Yukarıdaki ileti dinleyicisine iletilen m parametresi, JavaScript SDK v3’ün çoklu parametre tasarımından farklı bir tasarım olan aşağıdaki yapıya sahiptir.

{
    "channel": "ch1",
    "subscription": <undefined>,
    "timetoken": "14721821326909151",
    "message": "hello"
}

Bu ‘zaman belirteci’, gerçek ‘yayın zaman belirteci’dir. Deneyimli PubNub geliştiricileri için, bunun artık aboneye sunulduğunu görmek sizi heyecanlandırmalı, ancak bunun neden önemli ve güçlü olduğuna şu anda girmeyelim.

Artık deneyimli bir JavaScript geliştiricisinin yukarıda gösterildiği gibi kod yazmasını beklemezdim ve birçok ileri düzey geliştirici ne yapacağını zaten biliyor olabilir. Ancak JavaScript ile başlangıç ​​ve orta seviyede olan geliştiriciler için çözüm hemen açık olmayabilir. Ancak, aşağıdaki basit tasarım yaklaşımını gördüğünüzde, JavaScript dilinin sınırsız olanaklarına gözlerinizi açacağını biliyorum - işte başlıyoruz.

Gereksinimi yeniden ifade edelim:

Abone olduğum her kanal için bir bununla ilgili bir mesaj alındığında çağrılacak benzersiz işlev kanal. Ve monolitik koşullu kanal adı yaklaşımından kaçınmak istiyorum.

İlk önce yapmamız gereken bir fonksiyon arama tablosu oluşturmak. Tam olarak bir “hashtable”. Bu tablo, anahtar olarak kanal adlarına ve değerler olarak işlevlere (o kanalda bir mesaj alındığında çağrılacak kod) sahip olacaktır. JavaScript’te biraz yeniyseniz veya bir süredir dille kodlama yapıyorsanız, ancak henüz dil özelliklerine gerçekten girmediyseniz, bu garip ve imkansız gelebilir, ancak JavaScript gerçekten böyle çalışır ve siz bunu baştan beri yapıyorsunuz. ve gerçekten bilmiyordu. Fonksiyon arama tablomuzu tanımlayalım:

ftbl = {};

İşte bu - kanallarınızı ve işlevlerinizi tutacak bir nesneniz var. Oldukça basit, değil mi? Ancak kanalları ve işlevleri nasıl eklersiniz? Tıpkı diğer herhangi bir anahtar/değer gibi.

ftbl.ch1 = function(m){console.log(m.message + " ch1 callback")};
ftbl.ch2 = function(m){console.log(m.message + " ch1 callback")};

…ve tanımlamak istediğiniz her kanal ve fonksiyon ile. Ayrıca tüm kanal/fonksiyon tuşlarınızı kodunuzun bir noktasında oluşturmanız gerekmez. Bir kanala abone olurken her kanalı/fonksiyonu ftbl’ye ekleyebilirsiniz.

ftbl.ch10 = function(m){console.log(m.message + " ch10 callback")};
pubnub.pubnub.subscribe({
    channels: ['ch10'] 
});

Tamam, bu yeterince basit ve bunu nasıl yaptığınızla daha meraklı ve daha gelişmiş olabilirsiniz, ancak sadece temel düzeyde kalabilirsiniz. Ancak bu işlevi bağlı olduğu kanal için nasıl çağırırsınız? Bu nedenle JavaScript çok havalı, güçlü ve kolaydır, özellikle Java gibi katı ve yapılandırılmış bir dilden geliyorsanız - kontrol edin.

pubnub.addListener({
    message: function(m) {
        // use the channel name to get the function
        // from ftbl and invoke it 
        ftbl[m.subscribedChannel](m);
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(statusEvent) {
        console.log(JSON.stringify(s));
    }
});

Hepsi bu kadar. Dinleyicinin ‘message’ geri arama işlevine iletilen kanal adını kullanarak işlevi ftbl’den alın ve sonuna ‘(m)’ ekleyin ve boom, işlevinizi çalıştırır.

Kanal “ch10” ise, “ftblm.subscribedChannel”, “m” parametresini geçen “function(m){console.log(m.message + “ch10 callback”)}” işlevini çağırır. işleviniz gerektiği gibi ayrıştırabilir ve kullanabilir.

Bu nedenle, aşağıdaki ‘yayınla’ işlevini çağırmak:

pubnub.publish(
    {
        channel : "ch10",
        message : "hello"
    }, 
    function(status, response) {
        console.log(status, response);
    }
);

… şu mesajın görüntülenmesine neden olur: hello ch10 callback. Ve fonksiyon arama tablonuzda tanımladığınız diğer kanallara yayınlamak için eşdeğerdir. Bilinmeyen kanallar için bir varsayılan sağlamayı unutmayın.

Dinleyicideki ‘presence’ ve ‘status’ geri aramalarını da unutmayın. Bu, yalnızca iki işlev arama tablosu veya biraz daha karmaşık bir “ftbl” olabilir:

ftbl.message.ch1 = function(m){console.log(m.message + " ch1 message cb")};
ftbl.presence.ch1 = function(m){console.log(m.message + " ch1 presence cb")};
ftbl.status.ch1 = function(m){console.log(m.message + " ch1 status cb")};

veya

ftbl.ch1.message = ...
ftbl.ch1.presence = ...
ftbl.ch1.status = ...

İlkini ikincisinden daha çok seviyorum ama bu gerçekten sana kalmış. Ve muhtemelen yine de bazı genel ‘durum’ olay işleme kodu istiyorsunuz, ancak bu sizin özel gereksinimlerinize bağlı olacaktır.

Ve bu, mesaj yüküne gömdüğünüz bazı ek verilere bağlı olarak çağrılabilen kanal başına isteğe bağlı işlevlerle daha da karmaşık ve sağlam hale gelebilir.

İşte başlıyorsunuz, her kanal için benzersiz geri aramalar. 3x’ten 4x’e geçmek istememek için artık bahane yok. Ancak, taşıma konusunda bazı şüpheleriniz varsa PubNub Destek ile iletişime geçmekten çekinmeyin, ilerlemenizi sağlayalım. PubNub JavaScript SDK v3 to v4 Migration Guide‘ı da incelemeyi unutmayın.
