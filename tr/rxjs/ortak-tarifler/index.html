HTTP yanıtlarını önbelleğe alma #

RxJS için tipik bir kullanım örneği, HTTP istekleri oluşturmak ve sonuçlarını bir süre önbelleğe almaktır. Ayrıca, her zaman bir seferde yalnızca bir istek çalıştırmak ve yanıtını paylaşmak istiyoruz.

Örneğin, aşağıdaki kod 1 öğeyi maks. 1000ms:

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1);

var counter = 1;    
function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100);
}

function requestCachedHttpResult() {
    return updateRequest;
}

‘makeMockHttpRequest()’ işlevi, ‘100ms’ gecikmeyle gelen bir HTTP isteğini simüle eder.

requestCachedHttpResult() işlevi, gerçek veya önbelleğe alınmış yanıt almak için abone olduğumuz yerdir.

“.publishReplay(1, 1000)” ile dahili olarak kullanmak için RxJS çok noktaya yayın kullandık ‘ReplaySubject’ ve ‘1’ öğesini maksimum ‘1000ms’ için tutun. Sonra ‘refCount()’, ‘Observable.defer()’ olan ‘kaynağa’ her zaman yalnızca bir abonelik tutmak için kullanılır. Bu Gözlenebilir, yeni bir istek oluşturmak için kullanılır ve önbelleğe alınmış değerlerin ve yeni aboneliklerin aynı verileri paylaştığını kanıtlamak için “sayacını” artırır.

Güncel verileri almak istediğimizde requestCachedHttpResult() diyoruz. Gözlemcinin veri gönderdikten sonra düzgün bir şekilde tamamlanmasını sağlamak için al(1) operatörünü kullandık.

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

Her istek gecikmeli olarak gönderilir ve aşağıdaki sırayla gerçekleşir:

“0” - “refCount()” öğesinin “kaynağı"na abone olmasını sağlayan ve “mockDataFetch()” çağrısını yapan ilk istek. Yanıtı ‘100ms’ gecikecek. Şu anda, ‘publishReplay()’ operatörünün içindeki ‘ConnectableObservable’ bir Gözlemciye sahiptir.

‘50’ - İkinci istek, ‘ConnectableObservable’a da abone olur. Şu anda ‘publishReplay()’ operatörünün içindeki ‘ConnectableObservable’ iki Observer’a sahiptir. ‘refCount()’ zaten abone olduğu için ‘makeMockHttpRequest()’ ile başka bir istek oluşturmaz.

‘100’ - İlk yanıt hazır. Önce ‘ReplaySubject’ tarafından önbelleğe alınır ve ardından ‘ConnectableObservable’a abone olan iki Gözlemciye yeniden gönderilir. Her iki Gözlemci de al(1) sayesinde tamamlanır ve abonelikten çıkar.

‘200’ - Önbelleğe alınmış değerini hemen yayan ‘ReplaySubject’e abone olur ve bu da ’take(1)’ öğesinin Observer’ı tamamlamasına ve abonelikten hemen çıkmasına neden olur. Hiçbir HTTP isteği yapılmaz ve hiçbir abonelik kalmaz.

“1200” - “0"daki ilk olayla aynı. Bu noktada, “1000ms"den daha eski olduğu için önbelleğe alınan değer atılmıştır.

‘1500’ - ‘200’deki dördüncü olayla aynı.

“3500” - “1200"deki ilk olayla aynı.

Konsoldaki çıktı aşağıdaki gibidir:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: 2
Response 1500: 2
Response 3500: 3

İşleme hataları

Uzak hizmet (‘makeMockHttpRequest’ işlevi) tarafından üretilen hataları işlemek istiyorsak, onları ana Gözlenebilir zincirle birleştirilmeden önce yakalamamız gerekir, çünkü ‘publishReplay()’ içindeki ‘ReplaySubject’ tarafından alınan herhangi bir hata işaretlenir. dahili durumu “durduruldu” (burada daha fazlasını okuyun https://www.wikiod.com/tr/rxjs/ders#Konu ve iç durumu ) ki bu kesinlikle istediğimiz şey değil .

Aşağıdaki örnekte, counter === 2 olduğunda bir hatayı simüle ediyoruz ve onu catch() operatörüyle yakalıyoruz. “hata” bildirimini yalnızca normal bir “sonraki"ne dönüştürmek için “catch()” kullanıyoruz, böylece hatayı gözlemcilerde halledebiliriz:

function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100)
        .map(i => {
            if (i === 2) {
                throw new Error('Invalid URL');
            }
            return i;
        })
        .catch(err => Observable.of(err));
}

Canlı demoyu görün: https://jsbin.com/kavihu/10/edit?js,console

Bu, aşağıdaki çıktıyı konsola etmek için yazdıracaktır. Hataların “sonraki” işleyicilerinde alındığına dikkat edin:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: [object Error] { ... }
Response 1500: [object Error] { ... }
Response 3500: 3

Hataları her gözlemcide normal ‘hata’ bildirimleri olarak ele almak istiyorsak, yukarıda açıklanan nedenlerden dolayı onları ‘publishReplay()’ operatöründen sonra yeniden göndermemiz gerekir.

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1)
    .map(val => {
        if (val instanceof Error) {
            throw val;
        }
        return val;
    });

Birden çok paralel HTTP isteği gönderme #

Web uygulamalarında çok yaygın bir kullanım örneği, birden çok eşzamansız (ör. HTTP) istek gerçekleştirmek ve sonuçları geldiklerinde veya hepsini bir kerede toplamaktır (ör. Angular2’de [HTTP hizmetiyle](https://angular. io/docs/ts/latest/guide/server-communication.html)).

1. Zaman uyumsuz yanıtları geldiklerinde tek tek toplama

Bu genellikle, bir projeksiyon işlevi alan mergeMap() operatörüyle yapılır. bir Gözlenebilir döndürür. Operatör ‘mergeMap()’, önceki Gözlenebilir henüz tamamlanmamış olsa bile, her Gözlenebilir’e dahili olarak abone olur.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .subscribe(val => console.log(val));

Bu, rastgele gecikme nedeniyle konsola yanıtları farklı sırayla yazdırır:

Response from url-3
Response from url-4
Response from url-2
Response from url-1

Canlı demoyu görün: https://jsbin.com/xaqudan/2/edit?js,console

Her yanıt (“sonraki” çağrı yoluyla gönderilen öğe) “mergeMap()” tarafından hemen yeniden gönderilir.

Birden çok HTTP isteği gönderme amacımız için, “mergeMap()” öğesinin toplamda üç bağımsız değişken alabileceğini belirtmekte fayda var:

  1. Bir Gözlenebilir döndürmesi gereken projeksiyon işlevi.
  2. Daha fazla göndermeden önce sonucu değiştirmemizi sağlayan sonuç seçici işlevi.
  3. Eşzamanlı olarak abone olunan Gözlenebilirlerin sayısı.

Paralel isteklerin sayısını kontrol etme

Üçüncü argümanla, kaç tane paralel istek işleyeceğimizi kontrol edebiliriz (her bir Gözlenebilir’in bir HTTP isteği gerçekleştirdiğini “soğuk” olduğunu varsayarak).

Aşağıdaki örnekte aynı anda sadece 2 istek çalıştıracağız.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url), undefined, 2)
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Canlı demoyu görün: https://jsbin.com/sojejal/4/edit?js,console

İlk iki isteğin 1 saniye sonra, diğer ikisinin ise 2 saniye sonra tamamlandığına dikkat edin.

[1004, "Response from url-1"]
[1010, "Response from url-2"]
[2007, "Response from url-3"]
[2012, "Response from url-4"]

İşleme hataları

Kaynak Gözlemlenebilirlerden herhangi biri başarısız olursa (“hata” bildirimi gönderir) “mergeMap()”, hatayı “hata” olarak yeniden gönderir. Her Gözlenebilir’in zarif bir şekilde başarısız olmasını istiyorsak, örneğin catch() kullanmamız gerekir. ) Şebeke.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url).catch(() => Observable.empty()))
  .subscribe(val => console.log(val));

“url-3” yanıtı, “hata” bildirimi olarak gönderilen bir hata veriyor. Bu daha sonra ‘catch()’ operatörü tarafından yakalanır ve sadece ’tam’ bir bildirim olan ‘Observable.empty()’ ile değiştirilir. Bu nedenle bu yanıt dikkate alınmaz.

Bu örneğin çıktısı aşağıdaki gibidir:

Response from url-4
Response from url-1
Response from url-2

Canlı demoya bakın: https://jsbin.com/kuqumud/4/edit?js,console

2. Tüm zaman uyumsuz yanıtları bir kerede toplama

Önceki örnekleri izleyerek tüm yanıtları toArray() operatörüyle toplayabiliriz.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .toArray()
  .subscribe(val => console.log(val));

Ancak, toArray() operatörünü kullanmanın önemli bir sonucu vardır. Abonenin sonuçları alıp almadığı, yalnızca tüm HTTP isteklerinin tamamlanmasıyla değil, aynı zamanda Observable (bizim durumumuzda ‘Observable.from’) kaynağının tamamlanmasıyla da kontrol edilir. Bu, asla tamamlanmayan kaynak Gözlemlenebilirleri kullanamayacağımız anlamına gelir (örn. ‘Observable.fromEvent’).

Aynı sonucu elde etmenin başka bir yolu da Observable.forkJoin() kullanmaktır. Argüman abone olmak istediğimiz ve hepsi en az bir değer yayıp tamamlanana kadar beklemek istediğimiz bir Gözlenebilirler dizisi.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];
var observables = urls.map(url => mockHTTPRequest(url));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Bu, tüm yanıtları tek bir dizi olarak yazdırır:

["Response from url-1", "Response from url-2", "Response from url-3", "Response from url-4"]

Canlı demoyu görün: https://jsbin.com/fomoye/2/edit?js,console

‘Observable.forkJoin()’ ayrıca isteğe bağlı bir argüman olarak, nihai sonucu daha fazla göndermeden önce değiştirmemize izin veren bir sonuç seçici işlevi alır:

Observable.forkJoin(observables, (...results) => {
    return results.length;
  })
  .subscribe(val => console.log(val));

Bu konsola yazdırır:

4

Canlı demoya bakın: https://jsbin.com/muwiqic/1/edit?js,console

Sonuç seçici işlevi için bağımsız değişkenin paketlenmemiş olduğuna dikkat edin.

İşleme hataları

Hata işleme için, catch() operatörü ile önceki örnekteki ile aynı yaklaşımı kullanabiliriz.

Ancak bilinmesi gereken önemli bir şey var. ‘forkJoin()’, her Observable kaynağının en az bir değer yaymasını gerektirir. Daha önce yaptığımız gibi ‘catch(() => Observable.empty())’ kullansaydık, ‘forkJoin()’ asla hiçbir şey yaymazdı çünkü ‘Observable.empty()’ sadece ’tam’ bir bildirimdir. Bu nedenle, örneğin “boş” bir değer olan “Observable.of(null)” ve ardından “complete” bildirimini kullanmamız gerekiyor.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

var observables = urls.map(url => mockHTTPRequest(url).catch(() => Observable.of(null)));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Canlı demoyu görün: https://jsbin.com/yidiked/2/edit?js,console

Bu konsola yazdırır:

["Response from url-1", "Response from url-2", null, "Response from url-4"]

Hatanın “null” ile değiştirildiğine dikkat edin. Sadece ‘Observable.empty()’ kullansaydık, ‘forkJoin()’ asla hiçbir şey yaymazdı.

Yavaş/modası geçmiş dinlenme çağrılarını iptal etme #

Yaygın bir kullanım durumu, belirli kullanıcı girişlerinden sonra artık gerekmeyen belirli dinlenme aramalarını iptal etmektir. En belirgin örnek, bir kullanıcı bir arama işlevini kullandığında, bir istekte bulunduğunda, başka bir istekte bulunduğunda ve herhangi bir nedenle ilk istek ikinci istekten sonra geldiğinde ve uygulama eski isteğin güncel olmayan verilerini görüntülediğinde olabilir.

Bu, “switchMap” operatörü için mükemmel bir kullanım durumudur.

searchInput$
    .switchMap(() => makeRestCall());

Bu durumda, akım dinlenme çağrısına “geçecektir”, ancak yalnızca “searchInput$” üzerinde yeni veriler yayınlanana kadar, daha sonra “switchMap” içindeki akım atılır ve yeni bir dinlenme çağrısı yapılır. Bu nedenle, bir geri kalan sonuç, yalnızca bir sonraki tıklamadan önce bitmişse dikkate alınacaktır.

Ve işte tam teşekküllü bir alaycı örnek:

// some initial data-mocking
const Observable = Rx.Observable;
var counter = 1;
function mockDataFetch() {
    return Observable.of(counter++)
        .delay(500);
}

// the recipe

const searchInput$ = new Rx.Subject();
searchInput$
    .do(searchInput => console.log("Searching for " + searchInput))
    .switchMap(searchInput => mockDataFetch()
              .map(result => ({result, searchInput}))
     )
    .do(data => console.log("Received result for " + data.searchInput + ": " + data.result))
    .subscribe();

// simulating search inputs
searchInput$.next("google");
setTimeout(() => searchInput$.next("froogle"), 600);
setTimeout(() => searchInput$.next("doodle"), 800);
setTimeout(() => searchInput$.next("poodle"), 1000);
setTimeout(() => searchInput$.next("noodle"), 1600);

Canlı demoyu görün: https://jsbin.com/suzakahoro/1/edit?js,console

Birden çok ardışık HTTP isteği gönderme #

Bir dizi HTTP isteği yapmanın iki temel nedeni vardır:

  • İstekler birbirine bağlıdır (ardışık bir istek için bir isteğin sonucu gereklidir).

  • Sunucu yükünü birden çok isteğe yaymak istiyoruz.

1. Birden çok bağımlı istekte bulunmak

Bu, bir yanıtı gerekli parametrelere dönüştürmek için concatMap() operatörü kullanılarak gerçekleştirilebilir. ardışık istek için

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

function timestamp() {
  return (new Date()).getTime() - start;
}

var start = (new Date()).getTime();

Observable.of('url-1') 
  // first request
  .concatMap(url => {
    console.log(timestamp() + ': Sending request to ' + url);
    return mockHTTPRequest(url);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // second request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create new requests url
    return mockHTTPRequest(newUrl);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // third request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create another requests url
    return mockHTTPRequest(newUrl);
  })
  .subscribe(response => console.log(timestamp() + ': ' + response));

‘concatMap()’ operatörü, kendi projeksiyon işlevinden dönen Observable’a dahili olarak abone olur ve tüm değerlerini yeniden gönderirken tamamlanmasını bekler.

Bu örnek, her bir istek ve yanıtı zaman damgası ile belirtir:

3: Sending request to url-1
1010: Response from url-1
1011: Sending request to Response from url-1
2014: Response from url-19
2015: Sending request to Response from url-19
3017: Response from url-20

Canlı demoya bakın: https://jsbin.com/fewidiv/6/edit?js,console

İşleme hataları

HTTP isteklerinden herhangi biri başarısız olursa, aşağıdaki isteği oluşturamadığımız için açıkça devam etmek istemiyoruz.

2. Ardışık isteklerde bulunmak

Önceki HTTP isteğinden gelen yanıtla ilgilenmiyorsak, yalnızca bir dizi URL alabilir ve bunları birbiri ardına yürütebiliriz.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Bu örnek, zaman damgalı yanıtları yazdırır:

[1006, "Response from url-1"]
[2012, "Response from url-2"]
[3014, "Response from url-3"]
[4016, "Response from url-4"]

Canlı demoyu görün: https://jsbin.com/kakede/3/edit?js,console

Ardışık aramaları geciktirme

Ayrıca her istek arasında küçük bir gecikme yapmak isteyebiliriz. Böyle bir durumda, her ‘mockHTTPRequest()’ çağrısından sonra ‘delay()’ eklememiz gerekir.

Observable.from(urls)
  .concatMap(url => {
    return mockHTTPRequest(url)
      .do(response => console.log(((new Date()).getTime() - start) + ': Sending request to ' + url))
      .delay(500);
  })
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Bu, aşağıdaki çıktıyı konsolide etmek için yazdırır:

2024: Sending request to url-1
[2833, "Response from url-1"]
4569: Sending request to url-2
[5897, "Response from url-2"]

[8674, "Response from url-3"]

İşleme hataları

HTTP isteklerinden herhangi biri başarısız olduğunda görmezden gelmek istiyorsak, her bir ‘mockHTTPRequest()’ için ‘catch()’ zincirlememiz gerekir.

function mockHTTPRequest(url) {
  if (url == 'url-3') {
    return Observable.throw(new Error(`Request ${url} failed.`));
  } else {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
  }
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url).catch(obs => Observable.empty()))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Bu, “url-3” çağrısını yok sayar:

[1004, "Response from url-1"]
[2012, "Response from url-2"]
[3016, "Response from url-4"]

Canlı demoyu görün: https://jsbin.com/jowiqo/2/edit?js,console

Eğer ‘catch()’ operatörünü kullanmasaydık ‘url-3’ zincirin bir hata bildirimi göndermesine neden olur ve son ‘url-4’ yürütülmezdi.

Hız sınırlaması #

Uzak hizmetlerle ilgili yaygın bir sorun, hız sınırlamasıdır. Uzak hizmet, zaman periyodu başına yalnızca sınırlı sayıda istek veya miktarda veri göndermemize olanak tanır.

RxJS 5’te bufferTime operatörü tarafından çok benzer bir işlevsellik sağlanır ve özellikle biz ikinci parametreyi belirtilmemiş bırakın (ne sıklıkta yeni bir arabellek oluşturmak istediğimizi tanımlar. Eğer onu tanımsız/boş bırakırsak, mevcut olanı yayınladıktan hemen sonra yeni bir arabellek oluşturur).

“bufferTime"ın tipik bir kullanımı şöyle görünecektir:

bufferTime(1000, null, 5)

Bu, iki koşuldan biri karşılanana kadar öğeleri arabelleğe alacaktır. Sonra arabelleği yayar ve bir tane daha başlatır:

  • operatör “1000ms” için öğeleri topluyor
  • operatör zaten “5” öğe topladı

Gösterim amacıyla, çok hızlı yayılan bir Observable kaynağı oluşturabiliriz, böylece “bufferTime” boyut sınırına (“5”) ulaşır ve her “1000ms"de bir defadan daha sık yayın yapar:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

Sonra onu “bufferTime” ve “concatMap” ile zincirleyeceğiz. “concatMap” operatörü, “1000ms” gecikmesini zorladığımız yerdir:

const startTime = (new Date()).getTime();

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.of(buffer).delay(1000))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Canlı demoyu görün: https://jsbin.com/kotibow/3/edit?js,console

Gecikmenin gerçekten en az ‘1000ms’ olduğundan emin olmak için emisyon sürelerini görmek için ’timestamp()‘ı da ekledik.

Observable.of(buffer) kullanmamız gerekmediğine dikkat edin. Burada, arabelleğe alınan öğelerin sayısının doğru olup olmadığını manuel olarak kontrol etmek için kullanıyoruz.

Konsol çıktısından, iki emisyon arasındaki gecikmenin kabaca “1000ms” olduğunu görebiliriz:

Timestamp { value: [ 1, 2, 3, 4, 5 ], timestamp: 1475 }
Timestamp { value: [ 6, 7, 8, 9, 10 ], timestamp: 2564 }
Timestamp { value: [ 11, 12, 13, 14, 15 ], timestamp: 3567 }
Timestamp { value: [ 16, 17, 18, 19, 20 ], timestamp: 4572 }
Timestamp { value: [ 21, 22, 23, 24, 25 ], timestamp: 5573 }
Timestamp { value: [], timestamp: 6578 }

Şimdi, kaynağın yavaş yaydığı bir durumu da test edebiliriz, böylece “bufferTime” operatörü maksimum aralık koşuluna ulaşır:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(300));

Canlı demoyu görün: https://jsbin.com/tuwowan/2/edit?js,console

Daha sonra çıktı yaklaşık 2 saniye sonra başlamalıdır çünkü “bufferTime” operatörünün yayması “1 saniye” sürdü ve ardından “1 saniye” gecikmesini ekledik;

Timestamp { value: [ 1, 2, 3 ], timestamp: 2017 }
Timestamp { value: [ 4, 5, 6 ], timestamp: 3079 }
Timestamp { value: [ 7, 8, 9, 10 ], timestamp: 4088 }
Timestamp { value: [ 11, 12, 13 ], timestamp: 5093 }
Timestamp { value: [ 14, 15, 16 ], timestamp: 6094 }
Timestamp { value: [ 17, 18, 19, 20 ], timestamp: 7098 }
Timestamp { value: [ 21, 22, 23 ], timestamp: 8103 }
Timestamp { value: [ 24, 25 ], timestamp: 9104 }

Bu yaklaşımı gerçek dünya uygulamasında kullanmak isteseydik, uzak çağrıyı “concatMap” operatörüne koyardık. Bu şekilde, uzak hizmetten gelen istekler veya yanıtlar arasında “1s” gecikmesini zorlamak isteyip istemediğimizi kontrol edebiliriz.

Örneğin, “concatMap” geri çağrısında “forkJoin"i kullanarak istekler arasında minimum “1s” gecikmeyi zorlayabiliriz:

function mockHTTPRequest(buffer) {
  return Observable.of(true).delay(Math.random() * 1500)
}

const startTime = (new Date()).getTime();
const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.forkJoin(
    mockHTTPRequest(buffer),
    Observable.of(buffer).delay(1000)
  ))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Canlı demoyu görün: https://jsbin.com/xijaver/edit?js,console

‘forkJoin’ sayesinde, ‘concatMap’in her iki Gözlenebilir’in de tamamlanmasını beklemesi gerekir.

Öte yandan, yanıtlar arasında “1s” gecikmeyi zorlamak isteseydik, “mockHTTPRequest()“den sonra “delay()” operatörünü eklerdik:

.concatMap(buffer => mockHTTPRequest(buffer).delay(1000))
