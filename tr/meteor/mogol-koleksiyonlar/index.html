Mongo koleksiyonları hakkında düşünmenin yararlı bir yolu Kim, Ne, Ne Zaman, Nerede, Neden ve Nasıldır. Mongo, farklı veri türleri için aşağıdaki optimizasyonlara sahiptir:

Nerede - GeoJSON Ne zaman - ObjectID zaman damgaları Kim - Meteor Hesabı Dizeleri Nasıl - Karar ağaçları için JSON

Bu, varsayılan belgeyi kabaca bir ‘Ne’yi temsil eden Mongo’da bırakır.

Eski Bir Veritabanında Kayıt Oluşturma #

Koleksiyonlarınızı idGeneration alanı ile tanımlayarak normal Mongo formatına öntanımlı olarak getirebilirsiniz.

MyCollection = new Meteor.Collection('mycollection', {idGeneration : 'MONGO'});

Bir belgeye veri ekleme #

Mongo’ya yeni başlayanların çoğu, bir dizi, tarih, boole, oturum değişkeni ve benzeri bir belge kaydına nasıl ekleneceği gibi temel bilgilerle mücadele eder. Bu örnek, temel veri girişleri hakkında bazı rehberlik sağlar.

Todos.insert({
  text: "foo",                        // String
  listId: Session.get('list_id'),     // String
  value: parseInt(2),                 // Number
  done: false,                        // Boolean
  createdAt: new Date(),              // Dimestamp
  timestamp: (new Date()).getTime(),  // Time
  tags: []                            // Array
});

En son oluşturulan belgenin _id’sini alma #

Senkronize olarak alabilirsiniz:

var docId = Todos.insert({text: 'foo'});
console.log(docId);

Veya eşzamansız olarak:

Todos.insert({text: 'foo'}, function(error, docId){
  console.log(docId);
});

Zaman serisi verileri #

MongoDB’yi zaman serisi verileri için kullanmak, resmi teknik incelemeler ve sunumlar ile çok iyi bir belge ve yerleşik kullanım durumudur. Zaman serisi verileri için kendi şemalarınızı icat etmeye çalışmadan önce MongoDB’nin resmi belgelerini okuyun ve izleyin.

Zaman Serisi Verileri için MongoDB

Genel olarak, zaman serisi verileriniz için “kovalar” oluşturmak isteyeceksiniz:

DailyStats.insert({
   "date" : moment().format("MM-DD-YYYY"),
   "dateIncrement" : moment().format("YYYYMMDD"),
   "dailyTotal" : 0,
   'bucketA': 0,
   'bucketB': 0,
   'bucketC': 0
   });

Ardından, uygulamanıza veri girdikçe bu kovaları artırın. Bu artış, bir Meteor Yöntemine, bir koleksiyon gözlemcisine, bir REST API bitiş noktasına ve diğer çeşitli yerlere yerleştirilebilir.

DailyStats.update({_id: doc._id}, {$inc: {bucketA: 1} });

Normal İfadelerle Filtreleme #

Normal ifadeler, reaktif oturum değişkenleri ve deps otomatik çalıştırmalarını kullanarak sunucudaki abonelikleri filtrelemek için basit kalıp.

// create our collection
WordList =  new Meteor.Collection("wordlist");

// and a default session variable to hold the value we're searching for
Session.setDefault('dictionary_search', '');

Meteor.isClient(function(){
    // we create a reactive context that will rerun items when a Session variable gets updated 
    Deps.autorun(function(){
        // and create a subscription that will get re-subscribe to when Session variable gets updated
        Meteor.subscribe('wordlist', Session.get('dictionary_search'));
    });

    Template.dictionaryIndexTemplate.events({
        'keyup #dictionarySearchInput': function(evt,tmpl){
            // we set the Session variable with the value of our input when it changes
            Session.set('dictionary_search', $('#dictionarySearchInput').val());
        },
        'click #dictionarySearchInput':function(){
            // and clear the session variable when we enter the input
            Session.set('dictionary_search', '');
        },
    });
});
Meteor.isServer(function(){
    Meteor.publish('wordlist', function (word_search) {
        // this query gets rerun whenever the client subscribes to this publication
        return WordList.find({
            // and here we do our regex search
            Word: { $regex: word_search, $options: 'i' }
        },{limit: 100});
    });
});

Ve istemcide kullanılan HTML:

<input id="dictionarySearchInput" type="text" placeholder="Filter..." value="hello"></input>

Bu modelin kendisi oldukça basittir, ancak normal ifadeler olmayabilir. Normal ifadelere aşina değilseniz, işte bazı faydalı öğreticiler ve bağlantılar:

Tahsilat Sorgularını Denetleme #

Aşağıdaki örnek, tüm toplama sorgularınızı gerçek zamanlı olarak sunucu konsoluna kaydedecektir.

Meteor.startup(     
  function () {     
    var wrappedFind = Meteor.Collection.prototype.find;     

    // console.log('[startup] wrapping Collection.find')        

    Meteor.Collection.prototype.find = function () {        
      // console.log(this._name + '.find', JSON.stringify(arguments))       
      return wrappedFind.apply(this, arguments);        
    }       
  },        

  function () {     
    var wrappedUpdate = Meteor.Collection.prototype.update;     

    // console.log('[startup] wrapping Collection.find')        

    Meteor.Collection.prototype.update = function () {      
      console.log(this._name + '.update', JSON.stringify(arguments))        
      return wrappedUpdate.apply(this, arguments);      
    }       
  }     
);

Gözlemciler ve Çalışan İşlevleri #

Düğüm olay döngüsü bir bisiklet zinciri gibi davranıyorsa, sunucu tarafı toplama gözlemcisi bir vites değiştirici gibidir. Veriler geldikçe veri toplamaya oturacak bir dişli mekanizmasıdır. Tüm yarış bisikletlerinin vites değiştiricileri olduğu için çok performanslı olabilir. Ama aynı zamanda tüm sistemi kırmak için bir kaynaktır. Bu, sizi havaya uçurabilecek yüksek hızlı reaktif bir işlevdir. Dikkatli olun.

Meteor.startup(function(){
  console.log('starting worker....');

  var dataCursor = Posts.find({viewsCount: {$exists: true}},{limit:20});

  var handle = dataCursor.observeChanges({
    added: function (id, record) {
      if(record.viewsCount > 10){
         // run some statistics
         calculateStatistics();

         // or update a value
         Posts.update({_id: id}, {$set:{
           popular: true
         }});

      }
    },
    removed: function () {
      console.log("Lost one.");
    }
  });
});

20’lik sınırın vites değiştiricinin boyutu olduğuna dikkat edin…. kaç dişi vardır; veya daha spesifik olarak, koleksiyon üzerinde yürürken imleçte kaç öğe olduğu. Bu tür bir işlevde ‘var’ anahtar sözcüğünü kullanırken dikkatli olun. Belleğe mümkün olduğunca az nesne yazın ve eklenen yöntem içinde nesnenin yeniden kullanımına odaklanın. İşlem günlüğü açıldığında ve bu şey tam hızda giderken, bellek yığınına nesneleri Düğüm çöp toplayıcısının temizleyebildiğinden daha hızlı yazıyorsa, kötü bellek sızıntılarını ortaya çıkarmak için başlıca adaydır.

Yukarıdaki çözüm yatay olarak iyi ölçeklenmeyecektir çünkü her Meteor örneği aynı kaydı güncellemeye çalışacaktır. Bu nedenle, bunun yatay olarak ölçeklenmesi için bir tür ortam algılaması gereklidir.

Hizmet çalışanlarını bir kümedeki birden çok makinede senkronize etmenin mükemmel bir örneği için percolatestudios:synced-cron paketine bakın. meteor-synced-cron
