Devralınan Bir Veritabanını Analiz Etme #

Veritabanlarının kara kutu analizi için iki harika yardımcı program vardır. Birincisi, size üst düzey bir genel bakış sağlayacak olan Variety.js. İkincisi, bireysel alanlar hakkında daha fazla ayrıntı için koleksiyonları incelemenize izin verecek olan schema.js’dir. Bir üretim Mongo veritabanını devralırken, bu iki yardımcı program neler olup bittiğini ve koleksiyonların ve belgelerin nasıl yapılandırıldığını anlamanıza yardımcı olabilir.

variety.js

mongo test --eval "var collection = 'users'" variety.js

schema.js

mongo --shell schema.js

*.meteorapp.com Üzerindeki Bir Veritabanına Bağlanma #

–url bayrağını kullanmak zor olabilir. Kimlik doğrulaması için 60 saniyelik bir pencere vardır ve ardından kullanıcı adı/şifre rastgele sıfırlanır. Bu nedenle, komutu çalıştırdığınızda robomongo’nun açık ve yeni bir bağlantı yapılandırmaya hazır olduğundan emin olun.

# get the MONGO_URL string for your app  
meteor mongo --url $METEOR_APP_URL

*.meteor.com’dan bir Veritabanı indirin #

Daha önce olduğu gibi aynı, ancak bilgileri mongodump komutuna kopyalamanız gerekiyor. Aşağıdaki komutları inanılmaz hızlı çalıştırmanız gerekiyor ve bu el/göz koordinasyonu gerektiriyor. Dikkatli olun! Bu inanılmaz bir hile! Ama eğlenceli! Bunu bir video oyunu olarak düşün! :D

# get the MONGO_URL string for your app  
meteor mongo --url $METEOR_APP_URL

# then quickly copy all the info into the following command
mongodump -u username -p password --port 27017 --db meteor_app_url_com --host production-db-b1.meteor.io

Yerel Meteor geliştirme örneğinden Veriler Aktarılsın mı? #

Bu komut bir /dump dizini oluşturacak ve her koleksiyonu ayrı bir BSON blob dosyasında depolayacaktır. Bu, sistemler arasında veritabanlarını yedeklemenin veya aktarmanın en iyi yoludur.

mongodump --db meteor

Bir Dumpfile’dan Verileri Geri Yükleme #

“Meteordump” komutunun analogu “meteorrestore"dur. İçe aktarılacak belirli koleksiyonu seçerek kısmi bir içe aktarma yapabilirsiniz. Bir bırakma komutu çalıştırıldıktan sonra özellikle yararlıdır.

# make sure your app is running
meteor

# then import your data
mongorestore --port 3001 --db meteor /path/to/dump

# a partial import after running > db.comments.drop()
mongorestore --port 3001 --db meteor /path/to/dump -c comments.bson

Koleksiyonu JSON’a Dışa Aktarma #

Meteor çalıştırın, başka bir terminal penceresi açın ve aşağıdaki komutu çalıştırın.

mongoexport --db meteor --collection foo --port 3001 --out foo.json

Bir JSON Dosyasını Meteor’a Aktarın #

Varsayılan bir Meteor örneğine içe aktarmak oldukça kolaydır. json dosyanız başka bir sistemden bir dizi olarak dışa aktarılıyorsa bir –jsonArray seçeneği ekleyebileceğinizi unutmayın.

mongoimport --db meteor --port 3001 --collection foo --file foo.json

Hazırlama ve Yerel Veritabanları Arasında Veri Kopyalama #

Mongo, veritabanından veritabanına kopyalamayı destekler; bu, hazırlama veritabanında yerel bir geliştirme örneğine kopyalamak istediğiniz büyük veritabanlarınız varsa kullanışlıdır.

// run mongod so we can create a staging database
// note that this is a separate instance from the meteor mongo and minimongo instances
mongod

// import the json data into a staging database
// jsonArray is a useful command, particularly if you're migrating from SQL
mongoimport -d staging -c assets < data.json --jsonArray

// navigate to your application
cd myappdir

// run meteor and initiate it's database
meteor

// connect to the meteor mongodb
meteor mongo --port 3002

// copy collections from staging database into meteor database
db.copyDatabase('staging', 'meteor', 'localhost');

Ubuntu Kutusunda Mongo Veritabanını Sıkıştırma #

Ön tahsis. Mongo, boş kaplarda disk alanını bir kenara bırakır, böylece diske bir şeyler yazma zamanı geldiğinde, önce bitleri karıştırmak zorunda kalmaz. Bunu, 2 GB’a ulaşana kadar önceden tahsis edilen disk alanı miktarını iki katına çıkaran bir ikiye katlama algoritması ile yapar; ve bundan sonra her prealloc dosyası 2 GB’dir. Veriler önceden tahsis edildikten sonra, siz özellikle söylemediğiniz sürece tahsisi kaldırmaz. Bu nedenle gözlemlenebilir MongoDB alan kullanımı otomatik olarak yükselme eğilimindedir, ancak azalmaz.

Mongo ön tahsisi hakkında bazı araştırmalar… reduce-mongodb-database-file-size mongo-prealloc-files-take-up-room

// compact the database from within the Mongo shell
db.runCommand( { compact : 'mycollectionname' } )

// repair the database from the command line
mongod --config /usr/local/etc/mongod.conf --repair --repairpath /Volumes/X/mongo_repair --nojournal

// or dump and re-import from the command line
mongodump -d databasename
echo 'db.dropDatabase()' | mongo databasename
mongorestore dump/databasename

Bir Çoğaltma Kümesini Sıfırla #

Yerel veritabanı dosyalarını silin. Sadece Mongo kabuğundan çıkın, /dbpath’a gidin (nerede kurarsanız ayarlayın) ve bu dizindeki dosyaları silin.

*.meteor.com’daki bir Mongo Bulut Sunucusuna Uzaktan Bağlanma #

--url bayrağını biliyor muydunuz? Çok kullanışlı.

meteor mongo --url YOURSITE.meteor.com

Yerel Bir Meteor Eşgörünümünde Mongo Günlük Dosyalarına Erişme #

Kolayca erişilebilir değiller. ‘Meteor paketi’ komutunu çalıştırırsanız, bir tar.gz dosyası oluşturabilir ve ardından uygulamanızı manuel olarak çalıştırabilirsiniz. Bunu yaparak, mongo günlüklerine erişebilmelisiniz… muhtemelen .meteor/db dizininde. Mongodb günlük dosyalarına gerçekten erişmeniz gerekiyorsa, normal bir mongodb örneği kurun ve ardından MONGO_URL ortam değişkenini ayarlayarak Meteor’u harici bir mongo örneğine bağlayın:

MONGO_URL='mongodb://user:password@host:port/databasename'

Bu yapıldıktan sonra, günlüklere her zamanki yerlerden erişebilmelisiniz …

/var/log/mongodb/server1.log

Bir Ubuntu Kutusunda Günlük Dosyalarını Döndürme #

Bu günlük dosyalarını döndürmeniz gerekiyor, yoksa sonunda tüm disk alanınızı tüketecekler. Biraz araştırma ile başlayın… mongodb-log-file-growth döndürme-günlük dosyaları

Log dosyaları aşağıdaki komutla görüntülenebilir…

ls /var/log/mongodb/

Ancak günlük dosyası döndürmeyi ayarlamak için aşağıdakileri yapmanız gerekir…

// put the following in the /etc/logrotate.d/mongod file
/var/log/mongo/*.log {
    daily
    rotate 30
    compress
    dateext
    missingok
    notifempty
    sharedscripts
    copytruncate
    postrotate
        /bin/kill -SIGUSR1 `cat /var/lib/mongo/mongod.lock 2> /dev/null` 2> /dev/null || true
    endscript
}

// to manually initiate a log file rotation, run from the Mongo shell
use admin
db.runCommand( { logRotate : 1 } )
