ASP.NET Web API İçerik Müzakere Temel Bilgileri #

İçerik Pazarlığı, belirli bir kaynak için en iyi temsili seçme süreci olarak tanımlanabilir. Dolayısıyla İçerik anlaşması, istemcinin ve sunucunun, istemcinin gerekli formata göre veri alabilmesi için aralarında pazarlık yapabileceği anlamına gelir.

İnternetin bağlı olduğu üç nokta vardır,

  • Kaynak
  • Kaynağa bir İşaretçi (URL)
  • Kaynağın temsili

Üçüncü nokta diğer ikisinden daha önemlidir, çünkü her şey kaynağı nasıl gördüğümüze göre çalışır. Bir kaynağı iki biçimde temsil edebiliriz.

  1. XML(Genişletilebilir İşaretleme Dili) Formatı
  2. JSON(JavaScript Nesne Gösterimi) Formatı

RESTful hizmetinin standartlarından biri, istemcinin yanıtı JSON veya XML olarak hangi formatta istediğine karar verme yeteneğine sahip olmasıdır. Sunucuya gönderilen istek bir Kabul başlığı içerir. Kabul üstbilgisini kullanarak istemci, yanıtın biçimini belirleyebilir.

Örneğin,

“Kabul et: uygulama/xml”, sonucu XML biçiminde döndürür

‘Kabul et: application/json’, sonucu JSON biçiminde döndürür

İstekteki Kabul Et başlık değerine bağlı olarak sunucu yanıtı gönderir. Buna İçerik Pazarlığı denir.

Belirli bir biçimde veri talep ettiğimizde sahne arkasında ne olur?

ASP.NET Web API denetleyicisi, istemciye göndermek istediğimiz verileri oluşturur ve verileri Web API ardışık düzenine iletir ve ardından istemcinin Kabul başlığını arar. Ardından, verileri biçimlendirmek için uygun bir biçimlendirici seçin.

ASP.NET Web API büyük ölçüde genişletilebilir olduğundan, istek başlığında kabul üstbilgisi için birden çok değer belirtebiliriz.

`Kabul et: uygulama/xml, uygulama/json’

Yukarıdaki durumda, sunucu yanıt verilerini biçimlendirmek için ilk biçimlendiriciyi seçer.

Kabul başlığında kalite faktörünü de belirtebiliriz. Bu durumda sunucu daha yüksek kalite faktörüne sahip bir format seçer.

Kabul et: application/json;q=0.8,application/xml;q=0.5

Herhangi bir Kabul başlığı belirtmezsek, varsayılan sunucu olarak JSON biçimlendiriciyi seçin.

Yanıt, istemciye istenen biçimde gönderilirken, yanıtın ‘Content-Type’ başlığının uygun değere ayarlandığına dikkat edin. Örneğin, istemci ‘application/xml’ talebinde bulunduysa, sunucu verileri XML formatında gönderir ve ayrıca ‘Content-Type=application/xml’ değerini ayarlar.

Biçimlendiriciler, sunucu tarafından hem istek hem de yanıt mesajları için kullanılır. İstemci sunucuya bir istek gönderdiğinde, sunucunun gönderdiğimiz verilerin biçimini bilmesini sağlamak için İçerik Türü başlığını uygun değere ayarlarız.

Örneğin, istemci JSON verisi gönderiyorsa, İçerik Türü başlığı ‘application/json’ olarak ayarlanır. Sunucu, JSON verileriyle uğraştığını bilir, bu nedenle JSON verilerini .NET Türüne dönüştürmek için JSON biçimlendiriciyi kullanır. Benzer şekilde, sunucudan istemciye bir yanıt gönderilirken, Kabul başlık değerine bağlı olarak, .NET türünü JSON, XML vb.‘ye dönüştürmek için uygun biçimlendirici kullanılır.

Farklı yanıt formatı türlerine örnek:

uygulama/json:

{
  "Email": "sample string 1",
  "HasRegistered": true,
  "LoginProvider": "sample string 3"
}

uygulama/xml:

<UserInfoViewModel xmlns:i="http://www.w3.org/2001/XMLSchema-instance" xmlns="http://schemas.datacontract.org/2004/07/WebApiDemo.Models">
  <Email>sample string 1</Email>
  <HasRegistered>true</HasRegistered>
  <LoginProvider>sample string 3</LoginProvider>
</UserInfoViewModel>

Modern web tabanlı uygulamalar, çeşitli dillerde ve formatlarda veri sağlayabilir. Bu nedenle, API’mizi dünyanın her yerindeki küresel kullanıcıları kapsayacak şekilde geliştirirsek İçerik Pazarlığı önemlidir.

Web API’sinde İçerik Pazarlığı #

Kavramı anlamak #

Web API’sinde içerik anlaşmasını anlamak için “Kaynak” terimini anlamak önemlidir.

Web’de erişebildiğimiz herhangi bir bilgi “HTTP kaynağı” olarak adlandırılabilir. Web’de html belgeleri, resimler, video, ses, yürütülebilir dosyalar, elektronik tablolar vb. gibi farklı içerik türlerine sahip muazzam miktarda malzeme var. Kaynak uri’ye http isteği yaparak herhangi bir kaynağı alabiliriz. İsteğin http yanıtı, kaynağı döndürür ve ayrıca “medya türü olarak da bilinen” içerik türünü belirtir.

Herhangi bir kaynağa erişmek için müşteri, belirli kaynak uri ve http fiillerini sağlayarak http isteğinde bulunabilir. Ancak buna ek olarak istemci, kullanıcının aradığı içeriğin formatı olan kabul türünü de belirleyebilir. “Kabul türü”, http istek başlıklarında “kabul” başlığı olarak tanımlanabilir.

Sunucu daha sonra isteklerden “kabul” başlığını kontrol eder ve varsa yanıtı belirtilen biçimde döndürür. Lütfen sunucunun yanıtı yalnızca istenen gösterimde varsa döndürebileceğini unutmayın. İstenen gösterim mevcut değilse, kaynağı varsayılan gösterimde döndürür. İçerik görüşmesi olarak adlandırılmasının nedeni budur.

Pratik bir örnek #

Örnek olarak, http://example.com/customer/1 adresinden 1 kimliğine sahip müşterinin bilgilerini almak için talepte bulunduğunuzu varsayalım. Talepte “kabul” başlığını belirtmezseniz, sunucu bu kaynağın varsayılan temsilini döndürür.

Sunucunun müşteri bilgilerini “json ve xml her ikisi” biçiminde döndürebileceğini varsayalım. Artık istekteki “kabul et” başlığında müşteri bilgilerinin gerekli biçimini belirtmek istemcidedir. “accept” başlığının değeri, json gösterimi için “application/json” veya xml gösterimi için "text/xml" olabilir. Sunucu daha sonra yanıtı istenen formata göre döndürür.

İstenen biçim, bu ana bilgisayar tarafından desteklenmeyen “metin/html” ise (bu örnekte olduğu gibi), o zaman kaynağı yalnızca varsayılan biçimde döndürür. http yanıtı, istemciye kaynağın biçimi hakkında bilgi veren bir “içerik türü” başlığı içerir.

Lütfen, kaynağın istenen temsilinin mevcut olmaması durumunda bile, kaynağın varsayılan gösteriminin yine de döndürüleceğini unutmayın.

Bu yüzden içerik görüşmesi olarak anılır.

Müşteri, yanıtın temsilini müzakere eder, ancak mevcut değil, ardından varsayılanı alır.

Web API’de nasıl yapılandırılır #

Web API’de, içerik anlaşması “WebAPIConfig” sınıfında şu şekilde yapılandırılabilir:

config.Formatters.JsonFormatter.SupportedMediaTypes.Add(new System.Net.Http.Headers.MediaTypeHeaderValue("text/html"));

Ayrıca, IContentNegotiator arabirimini ve onun Negotiate yöntemini uygulayarak Web API’sindeki varsayılan içerik anlaşmasını geçersiz kılabilirsiniz ve ardından bunu Web API istek hattı satırında, WebAPI.config dosyasında aşağıdaki gibi ayarlayabilirsiniz:

GlobalConfiguration.Configuration.Services.Replace(typeof(IContentNegotiator), new CustomContentNegotiator());

Müzakere yönteminin örnek bir uygulaması aşağıdadır.

public class CustomContentNegotiator : DefaultContentNegotiator
    {
        public override ContentNegotiationResult Negotiate(Type type, HttpRequestMessage request, IEnumerable<MediaTypeFormatter> formatters)
        {
            var result = new ContentNegotiationResult(new JsonMediaTypeFormatter(), new MediaTypeHeaderValue("application/json"));
            return result;
        }
