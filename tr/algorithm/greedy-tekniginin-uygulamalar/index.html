Bilet otomatı #

İlk basit Örnek:

1, 2, 5, 10 ve 20 değerinde madeni paralarda değişim veren bir bilet otomatınız var. Değişimin dağıtımı, doğru değer dağıtılana kadar bir dizi madeni para düşüşü olarak görülebilir. Değerine göre jeton sayısı minimum olduğunda dağıtım optimum deriz.

‘[1,50]‘deki ‘M’, ‘T’ biletinin fiyatı ve ‘[1,50]‘deki ‘P’, birinin ‘T’ için ödediği para, ‘P >= M’ olsun. D=P-M olsun. Bir adımın faydasını, bu adımda otomatın dağıttığı ‘c’ ile ‘D’ ve ‘D-c’ arasındaki fark olarak tanımlarız.

Değişim için Açgözlü Tekniği aşağıdaki sözde algoritmik yaklaşımdır:

Adım 1: “D > 20” 20 jeton dağıtırken ve “D = D - 20” olarak ayarlayın
Adım 2: “D > 10” 10 jeton dağıtırken ve “D = D - 10” olarak ayarlayın
Adım 3: “D > 5” 5 jeton dağıtırken ve “D = D - 5” olarak ayarlayın
Adım 4: “D > 2” 2 jeton dağıtırken ve “D = D - 2” olarak ayarlayın
Adım 5: “D > 1” 1 jeton dağıtırken ve “D = D - 1” olarak ayarlayın

Daha sonra tüm madeni paraların toplamı açıkça ‘D’ye eşittir. Bu açgözlü bir algoritma çünkü her adımdan sonra ve adımın her tekrarından sonra fayda en üst düzeye çıkarılır. Daha yüksek faydaya sahip başka bir madeni para dağıtamayız.

Şimdi program olarak bilet otomatı (C++‘da):

#include <iostream>
#include <vector>
#include <string>
#include <algorithm>

using namespace std;

// read some coin values, sort them descending,
// purge copies and guaratee the 1 coin is in it
std::vector<unsigned int> readInCoinValues();

int main()
{
    std::vector<unsigned int> coinValues;   // Array of coin values ascending    
    int ticketPrice;                        // M in example
    int paidMoney;                          // P in example

    // generate coin values
    coinValues = readInCoinValues();
    
    cout << "ticket price: ";
    cin >> ticketPrice;
    
    cout << "money paid: ";
    cin >> paidMoney;
    
    if(paidMoney <= ticketPrice)
    {
        cout << "No exchange money" << endl;
        return 1;
    }
    
    int diffValue = paidMoney - ticketPrice;
    
    // Here starts greedy

    // we save how many coins we have to give out
    std::vector<unsigned int> coinCount;
    
    for(auto coinValue  = coinValues.begin();
             coinValue != coinValues.end(); ++coinValue)
    {
        int countCoins = 0;
        
        while (diffValue >= *coinValue)
        {
            diffValue -= *coinValue;
            countCoins++;
        }
        
        coinCount.push_back(countCoins);
    }
    
    // print out result
    cout << "the difference " << paidMoney - ticketPrice 
         << " is paid with: " << endl;
    
    for(unsigned int i=0; i < coinValues.size(); ++i)
    {
        if(coinCount[i] > 0)
            cout << coinCount[i] << " coins with value " 
                 << coinValues[i] << endl;
    }
    
    return 0;
}

std::vector<unsigned int> readInCoinValues()
{
    // coin values
    std::vector<unsigned int> coinValues;
    
    // make sure 1 is in vectore
    coinValues.push_back(1);

    // read in coin values (attention: error handling is omitted)
    while(true)
    {
        int coinValue;
        
        cout << "Coin value (<1 to stop): ";
        cin >> coinValue;
        
        if(coinValue > 0)
            coinValues.push_back(coinValue);
        
        else
            break;
    }
    
    // sort values
    sort(coinValues.begin(), coinValues.end(), std::greater<int>());
    
    // erase copies of same value
    auto last = std::unique(coinValues.begin(), coinValues.end());
    coinValues.erase(last, coinValues.end());
    
    // print array
    cout << "Coin values: ";
    
    for(auto i : coinValues)
        cout << i << " ";
    
    cout << endl;
    
    return coinValues;
}

Şimdiki madeni para değerlerinde “1” olduğu sürece, algoritma sona erecektir, çünkü:

  • D her adımda kesinlikle azalır
  • “D” asla “>0” değildir ve aynı anda en küçük jeton “1"den daha küçüktür

Ancak algoritmanın iki tuzağı var:

  1. En büyük madeni para değeri C olsun. “D/C” polinom olduğu sürece çalışma zamanı yalnızca polinomdur, çünkü “D"nin gösterimi yalnızca “log D” bitlerini kullanır ve çalışma zamanı “D/C"de en azından doğrusaldır.
  2. Algoritmamız her adımda yerel optimumu seçer. Ancak bu, algoritmanın küresel optimal çözümü bulduğunu söylemek için yeterli değildir (daha fazla bilgiyi buradan veya [Korte ve Vygen’in] Kitabında(http: //www.or.uni-bonn.de/~vygen/co.html)).

Basit bir karşı örnek: madeni paralar “1,3,4” ve “D=6"dır. Optimal çözüm açıkça “3” değerinde iki madeni paradır, ancak açgözlü ilk adımda “4"ü seçer, bu nedenle ikinci ve üçüncü adımda “1"i seçmek zorundadır. Bu yüzden optimal bir çözüm sağlamaz. Bu örnek için olası bir optimal Algoritma dinamik programlamaya dayanmaktadır.

Aralık Planlaması #

Bir dizi işimiz var J={a,b,c,d,e,f,g}. J in J , sj ile başlayıp fj ile biten bir iş olsun. Çakışmıyorlarsa iki iş uyumludur. Örnek olarak bir resim: Amaç, karşılıklı olarak uyumlu işlerin maksimum alt kümesini bulmaktır. Bu sorun için birkaç açgözlü yaklaşım vardır:

  1. En erken başlama zamanı: İşleri artan sj sırasına göre düşünün
  2. En erken bitiş zamanı: İşleri artan “fj” sırasına göre düşünün
  3. En kısa aralık: İşleri artan “fj-sj” sırasına göre düşünün
  4. En az çakışma: Her bir “j” işi için, “cj” çakışan işlerin sayısını sayın

Şimdi soru, hangi yaklaşımın gerçekten başarılı olduğu. Erken başlama zamanı kesinlikle hayır, işte bir karşı örnek En kısa aralık da optimal değildir ve en az çakışma kulağa gerçekten de optimal gelebilir, ancak bu yaklaşım için bir sorun örneği aşağıda verilmiştir: Bu da bizi en erken bitirme zamanı ile baş başa bırakıyor. Sözde kod oldukça basittir:

  1. İşleri bitiş zamanına göre sıralayın, böylece f1<=f2<=...<=fn
  2. ‘A’ boş bir küme olsun
  3. ‘j=1’ ila ’n’ için, eğer ‘j’, ‘A’ kümesi ‘A=A+{j}’ içindeki tüm işlerle uyumluysa
  4. “A”, karşılıklı olarak uyumlu işlerin maksimum alt kümesidir

Veya C++ programı olarak:

#include <iostream>
#include <utility>
#include <tuple>
#include <vector>
#include <algorithm>

const int jobCnt = 10;

// Job start times
const int startTimes[] = { 2, 3, 1, 4, 3, 2, 6, 7, 8, 9};

// Job end times
const int endTimes[]   = { 4, 4, 3, 5, 5, 5, 8, 9, 9, 10};

using namespace std;

int main()
{
    vector<pair<int,int>> jobs;
    
    for(int i=0; i<jobCnt; ++i)
        jobs.push_back(make_pair(startTimes[i], endTimes[i]));
    
    // step 1: sort
    sort(jobs.begin(), jobs.end(),[](pair<int,int> p1, pair<int,int> p2) 
                                     { return p1.second < p2.second; });
    
    // step 2: empty set A
    vector<int> A;
    
    // step 3:
    for(int i=0; i<jobCnt; ++i)
    {
        auto job = jobs[i];
        bool isCompatible = true;
        
        for(auto jobIndex : A)
        {
            // test whether the actual job and the job from A are incompatible
            if(job.second >= jobs[jobIndex].first &&
               job.first  <= jobs[jobIndex].second)
            {
                isCompatible = false;
                break;
            }
        }
    
        if(isCompatible)
            A.push_back(i);
    }
    
    //step 4: print A
    cout << "Compatible: ";
    
    for(auto i : A)
        cout << "(" << jobs[i].first << "," << jobs[i].second << ") ";
    cout << endl;
    
    return 0;
}

Bu örneğin çıktısı: Uyumlu: (1,3) (4,5) (6,8) (9,10)

Algoritmanın uygulanması açıkça Θ(n^2) içindedir. Bir Θ(n log n) uygulaması vardır ve ilgilenen okuyucu aşağıda okumaya devam edebilir (Java Örneği).

Şimdi aralık çizelgeleme problemi için açgözlü bir algoritmamız var, ama bu optimal mi?

Önerme: Açgözlü algoritma en erken bitiş zamanı optimaldir.

Kanıt:(çelişkiyle)

Açgözlülüğün optimal olmadığını ve ‘i1,i2,…,ik’nin açgözlü tarafından seçilen iş kümesini gösterdiğini varsayın. ‘j1,j2,…,jm’, *mümkün olan en büyük için ‘i1=j1,i2=j2,…,ir=jr’ ile bir optimal çözümdeki iş kümesini göstersin. * r değeri.

“i(r+1)” işi var ve “j(r+1)“den (en erken bitirme) önce bitiyor. Ancak j1,j2,...,jr,i(r+1),j(r+2),...,jmden daha optimum bir çözümdür ve içindeki tüm k için [1,(r+1)], jk=ik. bu, “r"nin maksimumluğuyla çelişkidir. Bu, kanıtı tamamlar.

Bu ikinci örnek, genellikle birçok olası açgözlü strateji olduğunu, ancak her durumda en uygun çözümü yalnızca bazılarının veya hatta hiçbirinin bulamayacağını göstermektedir.

Aşağıda Θ(n log n) içinde çalışan bir Java programı verilmiştir.

import java.util.Arrays;
import java.util.Comparator;

class Job
{
    int start, finish, profit;

    Job(int start, int finish, int profit)
    {
        this.start = start;
        this.finish = finish;
        this.profit = profit;
    }
}


class JobComparator implements Comparator<Job>
{
    public int compare(Job a, Job b)
    {
        return a.finish < b.finish ? -1 : a.finish == b.finish ? 0 : 1;
    }
}

public class WeightedIntervalScheduling
{
    static public int binarySearch(Job jobs[], int index)
    {
        int lo = 0, hi = index - 1;

        while (lo <= hi)
        {
            int mid = (lo + hi) / 2;
            if (jobs[mid].finish <= jobs[index].start)
            {
                if (jobs[mid + 1].finish <= jobs[index].start)
                    lo = mid + 1;
                else
                    return mid;
            }
            else
                hi = mid - 1;
        }

        return -1;
    }

    static public int schedule(Job jobs[])
    {
        Arrays.sort(jobs, new JobComparator());

        int n = jobs.length;
        int table[] = new int[n];
        table[0] = jobs[0].profit;

        for (int i=1; i<n; i++)
        {
            int inclProf = jobs[i].profit;
            int l = binarySearch(jobs, i);
            if (l != -1)
                inclProf += table[l];

            table[i] = Math.max(inclProf, table[i-1]);
        }

        return table[n-1];
    }

    public static void main(String[] args)
    {
        Job jobs[] = {new Job(1, 2, 50), new Job(3, 5, 20),
                    new Job(6, 19, 100), new Job(2, 100, 200)};

        System.out.println("Optimal profit is " + schedule(jobs));
    }
}

Gecikmeyi En Aza İndirme #

Gecikmeyi en aza indiren sayısız sorun var, burada bir seferde yalnızca bir işi işleyebilen tek bir kaynağımız var. “j” işi, “tj” işlem süresi birimleri gerektirir ve “dj” zamanında teslim edilir. “j”, “sj” zamanında başlarsa, “fj=sj+tj” zamanında biter. Tüm “j” için gecikme “L=max{0,fj-dh}” tanımlarız. Amaç, maksimum gecikme L’yi en aza indirmektir.

1 2 3 4 5 6
tj 3 2 1 4 3 2
dj 6 8 9 9 10 11

‘L=7’ çözümü açıkça optimal değildir. Bazı açgözlü stratejilere bakalım:

  1. Önce en kısa işlem süresi: işleri artan düzende planlayın og işlem süresi j`
  2. Önce en erken teslim tarihi: İşleri artan son teslim tarihi dj sırasına göre planlayın
  3. En küçük bolluk: işleri artan bolluk dj-tj sırasına göre planlayın

Önce en kısa işlem süresinin optimal olmadığını görmek kolaydır, iyi bir karşı örnek şudur:

1 2
tj 1 5
dj 10 5

en küçük yığın çözümünde benzer sorunlar var

1 2
tj 1 5
dj 3 5

son strateji geçerli görünüyor, bu yüzden bazı sözde kodlarla başlıyoruz:

  1. n işleri, vadesine göre sıralayın, böylece d1<=d2<=...<=dn
  2. t=0 olarak ayarlayın
  3. “j=1” ila “n” için
    • Assign job j to interval [t,t+tj]
    • set sj=t and fj=t+tj
    • set t=t+tj
  4. dönüş aralıkları [s1,f1],[s2,f2],...,[sn,fn]

Ve C++‘da uygulama olarak:

#include <iostream>
#include <utility>
#include <tuple>
#include <vector>
#include <algorithm>

const int jobCnt = 10;

// Job start times
const int processTimes[] = { 2, 3, 1, 4, 3, 2, 3, 5, 2, 1};

// Job end times
const int dueTimes[]     = { 4, 7, 9, 13, 8, 17, 9, 11, 22, 25};

using namespace std;

int main()
{
    vector<pair<int,int>> jobs;
    
    for(int i=0; i<jobCnt; ++i)
        jobs.push_back(make_pair(processTimes[i], dueTimes[i]));
    
    // step 1: sort
    sort(jobs.begin(), jobs.end(),[](pair<int,int> p1, pair<int,int> p2)
                                    { return p1.second < p2.second; });
    
    // step 2: set t=0
    int t = 0;
    
    // step 3:
    vector<pair<int,int>> jobIntervals;
    
    for(int i=0; i<jobCnt; ++i)
    {
        jobIntervals.push_back(make_pair(t,t+jobs[i].first));
        t += jobs[i].first;
    }
            
    //step 4: print intervals
    cout << "Intervals:\n" << endl;
    
    int lateness = 0;
    
    for(int i=0; i<jobCnt; ++i)
    {
        auto pair = jobIntervals[i];
        
        lateness = max(lateness, pair.second-jobs[i].second);

        cout << "(" << pair.first << "," << pair.second << ") "
             << "Lateness: " << pair.second-jobs[i].second << std::endl;
    }
    
    cout << "\nmaximal lateness is " << lateness << endl;
    
    return 0;
}

Algoritmanın çalışma zamanı açıkça Θ(n log n)‘dir, çünkü sıralama bu algoritmanın baskın işlemidir. Şimdi bunun optimal olduğunu göstermemiz gerekiyor. Açıkça optimal bir programın boş zamanı yoktur. önce en erken teslim tarihi programında da boş zaman yoktur.

İşlerin d1<=d2<=...<=dn olacak şekilde numaralandırıldığını varsayalım. Bir çizelgenin ters çevrilmesinin bir çift ‘i’ ve ‘j’ işi olduğunu söylüyoruz, böylece ‘i<j’, ancak j ‘i’den önce programlanmıştır. Tanımı nedeniyle önce en erken son tarih çizelgesinin tersine çevrilmesi yoktur. Tabii ki, bir çizelgenin tersine çevrilmesi varsa, ardışık olarak çizelgelenen bir çift tersine çevrilmiş iş içeren bir çizelgeye sahiptir.

Önerme: İki bitişik, ters çevrilmiş işi değiştirmek, tersine çevirme sayısını bir azaltır ve maksimum gecikmeyi artırmaz.

Kanıt: Takas öncesi gecikme ‘L’ ve sonrasındaki gecikme ‘M’ olsun. İki bitişik işin değiş tokuş edilmesi, diğer işleri konumlarından hareket ettirmediğinden, tüm k != i,j için Lk=Mk olur.

Açıkça, ‘i’ işi daha önce planlandığı için ‘Mi<=Li’dir. ‘j’ işi gecikirse, tanımdan şu anlama gelir:

                         Mj = fi-dj    (definition)
                           <= fi-di    (since i and j are exchanged)
                           <= Li

Bu, takastan sonraki gecikmenin öncekinden daha az veya eşit olduğu anlamına gelir. Bu, kanıtı tamamlar.

Önerme: önce en erken teslim tarihi S planı optimaldir.

Kanıt:(çelişkiyle)

S* mümkün olan en az ters çevirme sayısıyla en uygun program olduğunu varsayalım. “S*“nin boşta kalma süresi olmadığını varsayabiliriz. Eğer S* inversiyonu yoksa, S=S* ve işimiz bitti. Eğer S* bir inversiyona sahipse, bunun yanında bir inversiyona sahiptir. Son Önerme, gecikmeyi artırmadan, tersine çevirme sayısını azaltarak bitişik ters çevirmeyi değiştirebileceğimizi belirtir. Bu, “S*” tanımıyla çelişir.

Minimum zamanlama sorusunun sorulduğu minimum gecikme sorunu ve onunla yakın ilişkili minimum üretim süresi sorunu gerçek dünyada çok sayıda uygulamaya sahiptir. Ancak genellikle yalnızca bir makineniz olmaz, birçok makineniz olur ve aynı görevi farklı oranlarda gerçekleştirirler. Bu problemler NP-tamamını gerçekten hızlı bir şekilde alır.

Tüm görev ve verilerin elimizde olduğu çevrimdışı sorununa değil, görevlerin yürütme sırasında göründüğü çevrimiçi değişkenine bakarsak başka bir ilginç soru ortaya çıkar.

Çevrimdışı Önbelleğe Alma #

Önbelleğe alma sorunu, sonlu alanın sınırlandırılmasından kaynaklanmaktadır. Önbelleğimizde C k sayfaları olduğunu varsayalım. Şimdi, işlenmeden önce önbelleğe yerleştirilmiş olması gereken bir dizi ’m’ öğe isteğini işlemek istiyoruz. Tabii ’m<=k’ ise, o zaman tüm öğeleri önbelleğe koyarız ve çalışır, ancak genellikle m>>k dir.

Öğe zaten önbellekteyken, isteğin bir önbellek isabeti olduğunu söyleriz, aksi takdirde buna önbellek hatası denir. Bu durumda, talep edilen öğeyi önbelleğe almalı ve önbelleğin dolu olduğunu varsayarak başka bir öğeyi çıkarmalıyız. Hedef, tahliye sayısını en aza indiren bir tahliye planıdır.

Bu sorun için çok sayıda açgözlü strateji var, bazılarına bakalım:

  1. İlk giren ilk çıkar (FIFO): En eski sayfa çıkarılır
  2. Son giren ilk çıkar (LIFO): En yeni sayfa çıkarılır
  3. Son çıkış (LRU): En son erişimi en erken olan tahliye sayfası
  4. En az talep edilen(LFU): En az talep edilen tahliye sayfası
  5. En uzun ileri mesafe (LFD): Önbellekteki, en uzak zamana kadar talep edilmeyen sayfayı tahliye edin.

#include <iostream>
#include <memory>

using namespace std;

const int cacheSize     = 3;
const int requestLength = 16;

const char request[]    = {'a','a','d','e','b','b','a','c','f','d','e','a','f','b','e','c'};
char cache[]            = {'a','b','c'};

// for reset
char originalCache[]    = {'a','b','c'};


class Strategy {

public:
    Strategy(std::string name) : strategyName(name) {}
    virtual ~Strategy() = default;

    // calculate which cache place should be used
    virtual int apply(int requestIndex)                                      = 0;

    // updates information the strategy needs
    virtual void update(int cachePlace, int requestIndex, bool cacheMiss)    = 0;

    const std::string strategyName;
};

bool updateCache(int requestIndex, Strategy* strategy)
{
    // calculate where to put request
    int cachePlace = strategy->apply(requestIndex);

    // proof whether its a cache hit or a cache miss
    bool isMiss = request[requestIndex] != cache[cachePlace];

    // update strategy (for example recount distances)
    strategy->update(cachePlace, requestIndex, isMiss);

    // write to cache
    cache[cachePlace] = request[requestIndex];

    return isMiss;
}

int main()
{
    Strategy* selectedStrategy[] = { new FIFO, new LIFO, new LRU, new LFU, new LFD };

    for (int strat=0; strat < 5; ++strat)
    {
        // reset cache
        for (int i=0; i < cacheSize; ++i) cache[i] = originalCache[i];

        cout <<"\nStrategy: " << selectedStrategy[strat]->strategyName << endl;

        cout << "\nCache initial: (";
        for (int i=0; i < cacheSize-1; ++i) cout << cache[i] << ",";
        cout << cache[cacheSize-1] << ")\n\n";

        cout << "Request\t";
        for (int i=0; i < cacheSize; ++i) cout << "cache " << i << "\t";
        cout << "cache miss" << endl;

        int cntMisses = 0;

        for(int i=0; i<requestLength; ++i)
        {
            bool isMiss = updateCache(i, selectedStrategy[strat]);
            if (isMiss) ++cntMisses;

            cout << "  " << request[i] << "\t";
            for (int l=0; l < cacheSize; ++l) cout << "  " << cache[l] << "\t";
            cout << (isMiss ? "x" : "") << endl;
        }

        cout<< "\nTotal cache misses: " << cntMisses << endl;
    }

    for(int i=0; i<5; ++i) delete selectedStrategy[i];
}

FIFO #

class FIFO : public Strategy {
public:
    FIFO() : Strategy("FIFO")
    {
        for (int i=0; i<cacheSize; ++i) age[i] = 0;
    }

    int apply(int requestIndex) override
    {
        int oldest = 0;

        for(int i=0; i<cacheSize; ++i)
        {
            if(cache[i] == request[requestIndex])
                return i;

            else if(age[i] > age[oldest])
                oldest = i;
        }

        return oldest;
    }

    void update(int cachePos, int requestIndex, bool cacheMiss) override
    {
        // nothing changed we dont need to update the ages
        if(!cacheMiss)
            return;

        // all old pages get older, the new one get 0
        for(int i=0; i<cacheSize; ++i)
        {
            if(i != cachePos)
                age[i]++;

            else
                age[i] = 0;
        }
    }

private:
    int age[cacheSize];
};

class LFU : public Strategy {
public:
    LFU() : Strategy("LFU")
    {
        for (int i=0; i<cacheSize; ++i) requestFrequency[i] = 0;
    }

    int apply(int requestIndex) override
    {
        int least = 0;

        for(int i=0; i<cacheSize; ++i)
        {
            if(cache[i] == request[requestIndex])
                return i;

            else if(requestFrequency[i] < requestFrequency[least])
                least = i;
        }

        return least;
    }

    void update(int cachePos, int requestIndex, bool cacheMiss) override
    {
        if(cacheMiss)
            requestFrequency[cachePos] = 1;

        else
            ++requestFrequency[cachePos];
    }

private:

    // how frequently was the page used
    int requestFrequency[cacheSize];
};

LFD #

class LFD : public Strategy {
public:
    LFD() : Strategy("LFD")
    {
        // precalc next usage before starting to fullfill requests
        for (int i=0; i<cacheSize; ++i) nextUse[i] = calcNextUse(-1, cache[i]);
    }

    int apply(int requestIndex) override
    {
        int latest = 0;

        for(int i=0; i<cacheSize; ++i)
        {
            if(cache[i] == request[requestIndex])
                return i;

            else if(nextUse[i] > nextUse[latest])
                latest = i;
        }

        return latest;
    }

    void update(int cachePos, int requestIndex, bool cacheMiss) override
    {
            nextUse[cachePos] = calcNextUse(requestIndex, cache[cachePos]);
    }

private:

    int calcNextUse(int requestPosition, char pageItem)
    {
        for(int i = requestPosition+1; i < requestLength; ++i)
        {
            if (request[i] == pageItem)
                return i;
        }

        return requestLength + 1;
    }

    // next usage of page
    int nextUse[cacheSize];
};

Algoritma ve Gerçeklik #

LFD stratejisi optimaldir, ancak büyük bir sorun vardır. Optimum bir çevrimdışı çözüm. Uygulamada önbelleğe alma genellikle çevrimiçi bir sorundur, bu, stratejinin işe yaramaz olduğu anlamına gelir çünkü bir dahaki sefere belirli bir öğeye ihtiyacımız olduğunda bunu yapamayız. Diğer dört strateji de çevrimiçi stratejilerdir. Çevrimiçi sorunlar için genel olarak farklı bir yaklaşıma ihtiyacımız var.
