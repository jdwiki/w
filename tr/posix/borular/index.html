Borular, tek bir makine kapsamında tek yönlü süreçler arası veya iş parçacıkları arası iletişim için bir mekanizmadır. Mantıksal olarak, bir boru, birine verilerin yazılabileceği ve diğerinden bu verilerin daha sonra okunabileceği iki bağlantılı terminalden oluşur; yazma ve okumaların eşzamanlı olması gerekmeyen bir veri arabelleği ile. Borular, boruların bir uygulaması olan kabuk boru hatlarından ayırt edilmelidir.

Temel oluşturma ve kullanım #

Anonim borular veya basitçe borular, biri okuma terminali ve diğeri yazma terminali için olmak üzere bir çift dosya tanımlayıcısı olarak işlemlere maruz kalan çekirdek tarafından yönetilen nesnelerdir. ‘pipe’(2) işlevi aracılığıyla oluşturulurlar:

int pipefds[2];
int result;

result = pipe(pipefds);

Başarılı olduğunda, ‘pipe()’, sağlanan dizinin 0 dizinindeki borunun okuma ucu tanımlayıcısını ve dizin 1’deki yazma ucu tanımlayıcısını kaydeder; bu dizinler, standart akışlar için geleneksel dosya tanımlayıcı numaralarına benzer.

Bir boru oluşturduktan sonra, yazma ucuna yazmak veya okuma ucundan okumak için POSIX I/O işlevleri kullanılır:

İşlem 1:

ssize_t bytes_written = write(pipefds[1], "Hello, World!", 14);

Süreç 2:

char buffer[256];
ssize_t bytes_read = read(pipefds[0], buffer, sizeof(buffer));

Alternatif olarak, her iki boru ucundan birini C stdio işlevleriyle kullanmak için bir DOSYA yapısında sarmak için fdopen() kullanılabilir:

FILE *write_end = fdopen(pipefds[1]);

if (write_end) {
    fputs("Hello, World!");
}

Boruların sonlu G/Ç arabellekleri vardır ve tam arabelleğe sahip borulara normal yazma işlemleri engellenir. Borular, bu nedenle, bir iş parçacığının kendisiyle iletişim kurması için güvenli bir mekanizma değildir, sanki bir boruya yazılan iş parçacığı aynı zamanda ondan okuyan tek şeymiş gibi, o zaman bir yazma bloke eder etmez iş parçacığı kilitlenir.

Herhangi bir işlem, boru uçlarından herhangi biri için açık bir dosya açıklamasına sahip olduğu sürece bir boru devam eder. ‘close’(2) işlevi, bir dosya tanımlayıcı tarafından temsil edilen bir boru ucunu kapatmak için kullanılabilir ve ‘fclose’(3), bir boru ucunu, etrafına sarılmış bir ‘DOSYA’ aracılığıyla kapatmak için kullanılabilir.

Bir alt süreç için bir boru oluşturma #

Dosya tanımlayıcıları ve ‘FILE’ nesneleri, olağan G/Ç yoluyla süreçler arasında değiş tokuş edilemeyen süreç başına kaynaklardır. Bu nedenle, iki farklı işlemin anonim bir kanal aracılığıyla iletişim kurabilmesi için, katılan işlemlerden birinin veya her ikisinin, boruyu oluşturan süreçten açık bir boru ucunu devralması gerekir, bu nedenle ana işlem veya daha uzak bir ata olmalıdır. En basit durum, bir ana sürecin bir alt süreçle iletişim kurmak istediği durumdur.

Alt işlemin gerekli açık dosya açıklamasını üst öğesinden devralması gerektiğinden, önce kanal oluşturulmalıdır. Ebeveyn daha sonra çatallar. Normalde, her süreç ya tam anlamıyla bir okuyucu ya da kesinlikle bir yazar olacaktır; bu durumda her biri kullanmayı düşünmediği boru ucunu kapatmalıdır.

void demo() {
    int pipefds[2];
    pid_t pid;

    // Create the pipe
    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (pid = fork()) {
        case -1:
            // error - abort ...
            break;
        case 0:   /* child */
            close(pipefds[0]);
            write(pipefds[1], "Goodbye, and thanks for all the fish!", 37);
            exit(0);
        default:  /* parent */
            close(pipefds[1]);

            char buffer[256];
            ssize_t nread = read(pipefds[0], sizeof(buffer) - 1);

            if (nread >= 0) {
                buffer[nread] = '\0';
                printf("My child said '%s'\n", buffer);
            }

            // collect the child
            wait(NULL);

            break;
    }
}

İki alt işlemi bir boru aracılığıyla bağlama #

İki alt işlemin bir boru aracılığıyla bağlanması, iki çocuğun her birinin aynı borunun farklı uçları aracılığıyla üst öğeye bağlanmasıyla gerçekleştirilir. Genellikle ebeveyn, çocuklar arasındaki konuşmaya taraf olmaz, bu nedenle her iki boru ucunun kopyalarını kapatır.

int demo() {
    int pipefds[2];
    pid_t child1, child2;

    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            write(pipefds[1], "Hello, brother!", 15);
            exit(0);
        default:  /* parent */
            // nothing
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 2 */
            char buffer[256];
            ssize_t nread;

            close(pipefds[1]);
            nread = read(pipefds[0], buffer, sizeof(buffer) - 1);
            if (nread < 0) {
                // handle error
            } else {
                buffer[nread] = '\0';
                printf("My brother told me '%s'\n", buffer);
            }
            exit(0);
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    if (child1 >= 0) {
        wait(NULL);
        if (child2 >= 0) {
            wait(NULL);
        }
    }
}

Kabuk tarzı bir ardışık düzen oluşturma #

Kabuk tarzı bir işlem hattı, her birinin standart çıktısı bir sonrakinin standart girdisine bağlı olduğu iki veya daha fazla süreçten oluşur. Çıkıştan girişe bağlantılar borular üzerine kuruludur. Boru hattı benzeri bir süreç kümesi oluşturmak için, başka bir örnekte açıklandığı gibi boru bağlantılı alt süreçler oluşturulur ve ayrıca her bir boru ucunu kendi sürecinin uygun standart dosya tanımlayıcısına kopyalamak için ‘dup2’(2) işlevini kullanır. Özellikle, genellikle olduğu gibi, çocuğun daha sonra farklı bir komut yürütmesi amaçlanıyorsa, orijinal boru ucu dosya tanımlayıcısını daha sonra kapatmak genellikle iyi bir fikirdir.

// Most error handling omitted for brevity, including ensuring that pipe ends are
// always closed and child processes are always collected as needed; see other
// examples for more detail.
int demo() {
    int pipefds[2];
    pid_t child1 = -1, child2 = -1;

    pipe(pipefds);

    switch (child1 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            dup2(pipefds[1], STDOUT_FILENO);
            close(pipefds[1]);
            execl("/bin/cat", "cat", "/etc/motd", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    switch (child2 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 2 */
            close(pipefds[1]);
            dup2(pipefds[0], STDIN_FILENO);
            close(pipefds[0]);
            execl("/bin/grep", "grep", "[Ww]ombat", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    wait(NULL);
    wait(NULL);
}

Bir boru hattı kurarken en yaygın hatalardan biri, kullanmadığı boru uçlarını kapatan bir veya daha fazla işlemin unutulmasıdır. Genel bir kural olarak, nihai sonuç, her bir boru ucunun tam olarak bir prosese ait olması ve sadece o proseste açık olması olmalıdır.

Yukarıdaki demo işlevi gibi bir durumda, ikinci çocuğun veya ebeveynin “pipefds[1]“i kapatamaması, ikinci çocuğun askıda kalmasına neden olur, çünkü “grep” EOF görene kadar girdi beklemeye devam eder ve üst süreç veya ikinci alt sürecin kendisi gibi herhangi bir süreçte borunun yazma ucu açık kaldığı sürece gözlemlenmeyecektir.
