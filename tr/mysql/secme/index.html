‘SEÇ’, bir veya daha fazla tablodan seçilen satırları almak için kullanılır.

Sözdizimi #

  • TabloAdı’ndan [WHERE koşulları] FARKLI [ifadeler] SEÇİN; ///Basit Seçim

  • AYRI SEÇ(a), b …, AYRI SEÇ a, b … ile aynıdır.

  • SEÇ [ TÜMÜ | FARKLI | AYIRT ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ] [ SQL_SMALL_RESULT | SQL_BIG_RESULT ] [ SQL_BUFFER_RESULT ] [ SQL_CACHE | SQL_NO_CACHE ] [ SQL_CALC_FOUND_ROWS ] ifade tablolardan [NEREDE koşulları] [GROUP BY ifadeleri] [HAVING koşulu] [ORDER BY ifadesi [ ASC | AÇIKLAMA ]] [LIMIT [offset_value] number_rows | LIMIT sayı_satır OFFSET ofset_değeri] [PROSEDÜR prosedür_adı] [INTO [ OUTFILE ‘dosya_adı’ seçenekleri | DUMPFILE ‘file_name’ | @variable1, @variable2, … @variable_n] [GÜNCELLEME İÇİN | PAYLAŞIM MODUNDA KİLİT]; ///Tam Seçim Sözdizimi

MySQL’in ‘SELECT’ ifadesi hakkında daha fazla bilgi için MySQL Docs’e bakın.

DISTINCT ile SEÇ #

‘SEÇ’ten sonraki ‘DISTINCT’ yan tümcesi, sonuç kümesinden yinelenen satırları ortadan kaldırır.

CREATE TABLE `car`
(   `car_id` INT UNSIGNED NOT NULL PRIMARY KEY, 
    `name` VARCHAR(20), 
    `price` DECIMAL(8,2)
);

INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (1, 'Audi A1', '20000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (2, 'Audi A1', '15000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (3, 'Audi A2', '40000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (4, 'Audi A2', '40000');

SELECT DISTINCT `name`, `price` FROM CAR;
+---------+----------+
| name    | price    |
+---------+----------+
| Audi A1 | 20000.00 |
| Audi A1 | 15000.00 |
| Audi A2 | 40000.00 |
+---------+----------+

“DISTINCT”, sonuçları sunmak için tek tek sütunlarda değil, tüm sütunlarda çalışır. İkincisi, genellikle yeni SQL geliştiricilerinin bir yanılgısıdır. Kısacası, önemli olan sütun seviyesindeki değil, sonuç kümesinin satır seviyesindeki belirginliktir. Bunu görselleştirmek için yukarıdaki sonuç kümesindeki “Audi A1"e bakın.

MySQL’in sonraki sürümleri için, “DISTINCT”, “ORDER BY” ile birlikte kullanımıyla ilgili etkilere sahiptir. ‘ONLY_FULL_GROUP_BY’ ayarı, MySQL Handling of GROUP BY başlıklı aşağıdaki MySQL Kılavuz Sayfasında görüldüğü gibi devreye girer.

Tüm sütunları SEÇ (*) #

Sorgu

SELECT * FROM stack;

Sonuç

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
|    2 | stack    | stack    |
+------+----------+----------+
2 rows in set (0.00 sec)

Aşağıdakileri yaparak bir birleştirmede bir tablodaki tüm sütunları seçebilirsiniz:

SELECT stack.* FROM stack JOIN Overflow ON stack.id = Overflow.id;

En İyi Uygulama Satır(lar)da hata ayıklama veya ilişkisel dizilere getirme işlemi yapmadığınız sürece * işaretini kullanmayın, aksi takdirde şema değişiklikleri (EKLE/DROP/sütunları yeniden düzenleme) kötü uygulama hatalarına yol açabilir. Ayrıca, sonuç kümenizde ihtiyacınız olan sütunların listesini verirseniz, MySQL’in sorgu planlayıcısı genellikle sorguyu optimize edebilir.

** Artıları:**

  1. Sütun eklediğinizde/kaldırdığınızda, SELECT * kullandığınız yerde değişiklik yapmanız gerekmez.
  2. Yazmak daha kısa
  3. Cevapları da görüyorsunuz, yani SELECT *-kullanımı hiç haklı gösterilebilir mi?

Eksileri:

  1. İhtiyaç duyduğunuzdan daha fazla veri döndürüyorsunuz. Satır başına 200k içeren bir VARBINARY sütunu eklediğinizi varsayalım. Bu verilere tek bir kayıt için yalnızca tek bir yerde ihtiyacınız var - SELECT * kullanarak ihtiyacınız olmayan 10 satır başına 2MB döndürebilirsiniz
  2. Hangi verilerin kullanıldığı konusunda açık olun
  3. Sütunları belirtmek, bir sütun kaldırıldığında bir hata aldığınız anlamına gelir.
  4. Sorgu işlemcisi biraz daha çalışmalıdır - tabloda hangi sütunların bulunduğunu bulmak (teşekkürler @vinodadhikary)
  5. Bir sütunun nerede kullanıldığını daha kolay bulabilirsiniz
  6. SELECT * kullanırsanız tüm sütunları birleşimlerde alırsınız.
  7. Sıralı referanslamayı güvenle kullanamazsınız (sütunlar için sıralı referansları kullanmak başlı başına kötü bir uygulama olsa da)
  8. ‘TEXT’ alanlı karmaşık sorgularda, sorgu, daha az optimum geçici tablo işleme nedeniyle yavaşlayabilir

Sütun adına göre SEÇ #

CREATE TABLE stack(
    id INT,
    username VARCHAR(30) NOT NULL,
    password VARCHAR(30) NOT NULL
);

INSERT INTO stack (`id`, `username`, `password`) VALUES (1, 'Foo', 'hiddenGem');
INSERT INTO stack (`id`, `username`, `password`) VALUES (2, 'Baa', 'verySecret');

Sorgu

SELECT id FROM stack;

Sonuç

+------+
| id   |
+------+
|    1 |
|    2 |
+------+

BEĞEN ile SEÇ (%) #

CREATE TABLE stack
(  id int AUTO_INCREMENT PRIMARY KEY,
   username VARCHAR(100) NOT NULL
);

INSERT stack(username) VALUES 
('admin'),('k admin'),('adm'),('a adm b'),('b XadmY c'), ('adm now'), ('not here');

“adm” her yerde:

SELECT * FROM stack WHERE username LIKE "%adm%";  
+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | k admin   |
|  3 | adm       |
|  4 | a adm b   |
|  5 | b XadmY c |
|  6 | adm now   |
+----+-----------+

“Adm” ile başlar:

SELECT * FROM stack WHERE username LIKE "adm%";
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
|  3 | adm      |
|  6 | adm now  |
+----+----------+

“adm” ile biter:

SELECT * FROM stack WHERE username LIKE "%adm"; 
+----+----------+
| id | username |
+----+----------+
|  3 | adm      |
+----+----------+

Tıpkı bir “LIKE” yan tümcesindeki “%” karakterinin herhangi bir sayıda karakterle eşleşmesi gibi, “_” karakteri de yalnızca bir karakterle eşleşir. Örneğin,

SELECT * FROM stack WHERE username LIKE "adm_n"; 
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
+----+----------+

Performans Notları ‘kullanıcı adı’ üzerinde bir dizin varsa, o zaman

  • LIKE 'adm', `= ‘adm’ ile aynı işlevi görür
  • ‘LIKE ‘adm%’, ‘BETWEEN..AND..‘ye benzer bir “aralıktır”. Sütundaki bir dizini iyi kullanabilir.
  • LIKE '%adm' (veya leading joker karakterli herhangi bir değişken) herhangi bir dizin kullanamaz. Bu nedenle yavaş olacaktır. Çok satırlı tablolarda, o kadar yavaş olması muhtemeldir ki işe yaramaz.
  • “RLIKE” (“REGEXP”), “LIKE"dan daha yavaş olma eğilimindedir, ancak daha fazla özelliğe sahiptir.
  • MySQL, birçok tablo ve sütun türü üzerinde “FULLTEXT” indeksleme sunarken, bu “FULLTEXT” indeksleri “LIKE” kullanılarak sorguları gerçekleştirmek için kullanılmaz.

CASE veya IF ile SEÇ #

Sorgu

SELECT st.name,
       st.percentage, 
       CASE WHEN st.percentage >= 35 THEN 'Pass' ELSE 'Fail' END AS `Remark` 
FROM student AS st ;

Sonuç

+--------------------------------+
|   name   | percentage | Remark |
+--------------------------------+
|   Isha   |     67     |  Pass  |
|   Rucha  |     28     |  Fail  |
|   Het    |     35     |  Pass  |
|   Ansh   |     92     |  Pass  |
+--------------------------------+

Veya EĞER ile

SELECT st.name,
       st.percentage, 
       IF(st.percentage >= 35, 'Pass', 'Fail') AS `Remark` 
FROM student AS st ;

Not

IF(st.percentage >= 35, 'Pass', 'Fail')

Bu şu anlama gelir: Eğer st.percentage >= 35 ise DOĞRU ise o zaman `‘Geçti’’ döndürün ELSE ‘‘Başarısız’’ döndür

Takma Ad (AS) ile SEÇ #

SQL takma adları, bir tabloyu veya sütunu geçici olarak yeniden adlandırmak için kullanılır. Genellikle okunabilirliği artırmak için kullanılırlar.

Sorgu

SELECT username AS val FROM stack; 
SELECT username val FROM stack;

(Not: “AS” sözdizimsel olarak isteğe bağlıdır.)

Sonuç

+-------+
| val   |
+-------+
| admin |
| stack |
+-------+
2 rows in set (0.00 sec)

LIMIT yan tümcesi ile SEÇ #

Sorgu:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 3;

Sonuç:

CustomerID CustomerName ContactName Address City PostalCode Country 1

Alfreds Futterkiste Maria Anders Obere Str. 57 Berlin 12209 Germany 2 Ana Trujillo Emparedados y helados Ana Trujillo Avda. de la Constitución 2222 México D.F. 05021 Mexico 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

En İyi Uygulama ‘LIMIT’ kullanırken her zaman ‘SİPARİŞ BY’ kullanın; aksi takdirde alacağınız satırlar tahmin edilemez olacaktır.

Sorgu:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 2,1;

Açıklama:

Bir “LIMIT” yan tümcesi iki sayı içerdiğinde, “LIMIT offset,count” olarak yorumlanır. Bu nedenle, bu örnekte sorgu iki kaydı atlar ve bir tane döndürür.

Sonuç:

CustomerID CustomerName ContactName Address City PostalCode Country 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Not:

‘LIMIT’ yan tümcelerindeki değerler sabit olmalıdır; bunlar sütun değerleri olmayabilir.

WHERE ile SEÇ #

Sorgu

SELECT * FROM stack WHERE username = "admin" AND password = "admin";

Sonuç

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
+------+----------+----------+
1 row in set (0.00 sec)

WHERE yan tümcesinde yuvalanmış bir SELECT ile sorgu #

‘WHERE’ yan tümcesi, daha karmaşık sorgular yazmak için herhangi bir geçerli ‘SELECT’ ifadesini içerebilir. Bu bir ‘iç içe’ sorgudur

Sorgu

İç içe sorgular genellikle karşılaştırmalar için sorgulardan tek atomik değerler döndürmek için kullanılır.

SELECT title FROM books WHERE author_id = (SELECT id FROM authors WHERE last_name = 'Bar' AND first_name = 'Foo');

E-posta adresi olmayan tüm kullanıcı adlarını seçer

SELECT * FROM stack WHERE username IN (SELECT username FROM signups WHERE email IS NULL);

Sorumluluk Reddi: Tüm bir sonuç kümesini karşılaştırırken performans iyileştirmeleri için birleştirmeler kullanmayı düşünün.

1: http://stackoverflow.com/questions/17946221/sql-join-and- Different-types-of-joins

ARASINDAKİ SEÇ #

BETWEEN yan tümcesini “eşitten büyük VE eşitten küçük” koşullarının bir birleşimini değiştirmek için kullanabilirsiniz.

Veri

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
|  6 | java      |
+----+-----------+

Operatörlerle sorgulama

SELECT * FROM stack WHERE id >= 2 and id <= 5;

BETWEEN ile benzer sorgu

SELECT * FROM stack WHERE id BETWEEN 2 and 5;

Sonuç

+----+-----------+
| id | username  |
+----+-----------+
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
+----+-----------+
4 rows in set (0.00 sec)

Not

BETWEEN >= ve <= kullanır, > ve < değil.

BETWEEN’i kullanma

Negatifi kullanmak istiyorsanız “NOT”u kullanabilirsiniz. Örneğin :

SELECT * FROM stack WHERE id NOT BETWEEN 2 and 5;

Sonuç

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  6 | java      |
+----+-----------+
2 rows in set (0.00 sec)

Not

NOT BETWEEN > ve < kullanır ve >= ve <= kullanılmaz. > 5)`.

‘BETWEEN’ aramasında kullandığınız bir sütunda bir indeksiniz varsa, MySQL bu indeksi bir aralık taraması için kullanabilir.

BEĞEN(_) ile SEÇ #

Bir “LIKE” yan tümcesi desenindeki bir “_” karakteri, tek bir karakterle eşleşir.

Sorgu

SELECT username FROM users WHERE users LIKE 'admin_';

Sonuç

+----------+
| username |  
+----------+
| admin1   |
| admin2   |
| admin-   |
| adminA   |
+----------+

Tarih aralığı ile SEÇ #

SELECT ... WHERE dt >= '2017-02-01'
             AND dt  < '2017-02-01' + INTERVAL 1 MONTH

Elbette, bu “BETWEEN” ve “23:59:59” eklenmesiyle yapılabilir. Ancak, desenin şu faydaları vardır:

  • Bitiş tarihini önceden hesaplamamışsınızdır (genellikle başlangıçtan itibaren tam bir uzunluktur)
  • Her iki uç noktayı da (“BETWEEN"in yaptığı gibi) dahil etmezsiniz veya bundan kaçınmak için ‘23:59:59’ yazmazsınız.
  • “DATE”, “TIMESTAMP”, “DATETIME” ve hatta mikrosaniye dahil “DATETIME(6)” için çalışır.
  • Artık gün, yıl sonu vb. ile ilgilenir.
  • Dizin dostudur (‘BETWEEN’ de öyle).
