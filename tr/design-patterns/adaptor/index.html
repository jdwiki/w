Bağdaştırıcı (Java) #

Mevcut kod tabanınızda şöyle bir MyLogger arayüzü olduğunu varsayalım:

interface MyLogger {
    void logMessage(String message);
    void logException(Throwable exception);
}

“MyFileLogger” ve “MyConsoleLogger” gibi bunların birkaç somut uygulamasını oluşturduğunuzu varsayalım.

Uygulamanızın Bluetooth bağlantısını kontrol etmek için bir çerçeve kullanmak istediğinize karar verdiniz. Bu çerçeve, aşağıdaki kurucuya sahip bir “BluetoothManager” içerir:

class BluetoothManager {
    private FrameworkLogger logger;

    public BluetoothManager(FrameworkLogger logger) {
        this.logger = logger;
    }
}

“BluetoothManager” ayrıca bir kaydediciyi de kabul eder, bu harika! Ancak, arabirimi çerçeve tarafından tanımlanan bir günlükçü bekler ve işlevlerini farklı şekilde adlandırmak yerine yöntem aşırı yüklemesini kullanırlar:

interface FrameworkLogger {
    void log(String message);
    void log(Throwable exception);
}

Halihazırda, yeniden kullanmak istediğiniz bir sürü “MyLogger” uygulamanız var, ancak bunlar “FrameworkLogger"ın arayüzüne uymuyor. Adaptör tasarım modelinin devreye girdiği yer burasıdır:

class FrameworkLoggerAdapter implements FrameworkLogger {
    private MyLogger logger;

    public FrameworkLoggerAdapter(MyLogger logger) {
        this.logger = logger;
    }

    @Override
    public void log(String message) {
        this.logger.logMessage(message);
    }

    @Override
    public void log(Throwable exception) {
        this.logger.logException(exception);
    }
}

“FrameworkLogger” arabirimini uygulayan ve bir “MyLogger” uygulamasını kabul eden bir adaptör sınıfı tanımlanarak, işlevsellik farklı arabirimler arasında eşlenebilir. Artık ‘BluetoothManager’ı aşağıdaki gibi tüm ‘MyLogger’ uygulamalarıyla kullanmak mümkündür:

FrameworkLogger fileLogger = new FrameworkLoggerAdapter(new MyFileLogger());
BluetoothManager manager = new BluetoothManager(fileLogger);

FrameworkLogger consoleLogger = new FrameworkLoggerAdapter(new MyConsoleLogger());
BluetoothManager manager2 = new BluetoothManager(consoleLogger);

Java Örneği #

Bağdaştırıcı modelinin mevcut harika bir örneği SWT MouseListener içinde bulunabilir. %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseListener.html) ve [MouseAdapter](http://help.eclipse.org/kepler/index.jsp?topic=%2Forg.eclipse.platform.doc.isv%2Freference%2Fapi %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseAdapter.html) sınıfları.

MouseListener arayüzü aşağıdaki gibi görünür:

public interface MouseListener extends SWTEventListener {
    public void mouseDoubleClick(MouseEvent e);
    public void mouseDown(MouseEvent e);
    public void mouseUp(MouseEvent e);
}

Şimdi bir UI oluşturduğunuz ve bu dinleyicileri eklediğiniz bir senaryo hayal edin, ancak çoğu zaman bir şeyin tek tıklanması (mouseUp) dışında hiçbir şeyle ilgilenmezsiniz. Sürekli olarak boş uygulamalar oluşturmak istemezsiniz:

obj.addMouseListener(new MouseListener() {

    @Override
    public void mouseDoubleClick(MouseEvent e) {
    }

    @Override
    public void mouseDown(MouseEvent e) {
    }

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Bunun yerine MouseAdapter’ı kullanabiliriz:

public abstract class MouseAdapter implements MouseListener {
    public void mouseDoubleClick(MouseEvent e) { }
    public void mouseDown(MouseEvent e) { }
    public void mouseUp(MouseEvent e) { }
}

Boş, varsayılan uygulamalar sağlayarak, yalnızca bağdaştırıcıdan önemsediğimiz yöntemleri geçersiz kılmakta özgürüz. Yukarıdaki örnekten yola çıkarak:

obj.addMouseListener(new MouseAdapter() {

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Bağdaştırıcı (UML ve örnek durum) #

Adaptör kalıbının kullanımını ve ne zaman uygulanabileceği daha hayal edilebilir hale getirmek için burada küçük, basit ve çok somut bir örnek verilmiştir. Burada kod olmayacak, sadece UML ve örnek durumun ve probleminin bir açıklaması olacak. Kuşkusuz, UML içeriği Java gibi yazılmıştır. (Pekala, ipucu metni “İyi örnekler çoğunlukla koddur” dedi, tasarım kalıplarının da farklı bir şekilde tanıtılacak kadar soyut olduğunu düşünüyorum.)

Genel olarak, bağdaştırıcı modeli, uyumsuz arabirimlere sahip olduğunuz ve bunların hiçbirinin doğrudan yeniden yazılamadığı bir durum için yeterli bir çözümdür.

Güzel, küçük bir pizza dağıtım servisi yönettiğinizi hayal edin. Müşteriler web sitenizden çevrimiçi sipariş verebilir ve pizzalarınızı temsil etmek ve faturaları, vergi raporlarını ve daha fazlasını hesaplamak için bir “Pizza” sınıfı kullanan küçük bir sisteminiz var. Pizzalarınızın fiyatı, fiyatı (seçtiğiniz para biriminin) cent cinsinden temsil eden tek bir tam sayı olarak verilir.

Teslimat hizmetiniz harika çalışıyor, ancak bir noktada artan sayıda müşteriyi artık kendi başınıza idare edemezsiniz, ancak yine de genişlemek istiyorsunuz. Pizzalarınızı büyük bir çevrimiçi meta dağıtım hizmetinin menüsüne eklemeye karar verdiniz. Pek çok farklı yemek sunarlar - sadece pizzalar değil - bu nedenle sistemleri soyutlamayı daha fazla kullanır ve yemekleri temsil eden bir “IMeal” Arayüzüne ve parayı temsil eden bir “MoneyAmount” sınıfına sahiptir.

“MoneyAmount”, biri virgülden önceki miktar (veya bazı rastgele para birimleri) için ve diğeri virgülden sonra 0 ile 99 arasındaki sent miktarı için olmak üzere girdi olarak iki tam sayıdan oluşur;

‘Pizza’nızın fiyatı, toplam fiyatı cent (> 99) olarak temsil eden tek bir tam sayı olduğundan, ‘IMeal’ ile uyumlu değildir. İşte bu noktada adaptör modeli devreye giriyor: Kendi sisteminizi değiştirmek ya da yeni bir sistem oluşturmak çok fazla efor gerektiriyorsa ve uyumsuz bir arayüz uygulamak zorunda kalırsanız, adaptör modelini uygulamak isteyebilirsiniz.

Modeli uygulamanın iki yolu vardır: sınıf bağdaştırıcısı ve nesne bağdaştırıcısı.

Her ikisinin de ortak noktası, bir bağdaştırıcının (‘PizzaAdapter’), yeni arabirim ile uyarlanan (bu örnekte ‘Pizza’) arasında bir tür çevirmen olarak çalışmasıdır. Bağdaştırıcı, yeni arabirimi (IMeal) uygular ve ardından Pizzadan devralır ve kendi fiyatını bir tam sayıdan ikiye (sınıf bağdaştırıcısı) dönüştürür

veya öznitelik olarak “Pizza” türünde bir nesneye sahiptir ve bunun değerlerini dönüştürür (nesne bağdaştırıcısı).

Bağdaştırıcı desenini uygulayarak, uyumsuz arabirimler arasında bir tür “çeviri” yapacaksınız.
