Basit fabrika (Java) #

Bir fabrika, nesne oluşturma kodundan nesneler oluşturması gereken kodlar arasındaki eşleşmeyi azaltır. Nesne oluşturma, bir sınıf kurucusunu çağırarak değil, çağıran adına nesneyi yaratan bir işlevi çağırarak açıkça yapılır. Basit bir Java örneği aşağıdaki gibidir:

interface Car {
}

public class CarFactory{
  static public Car create(String s) {
    switch (s) {
    default:
    case "us":
    case "american": return new Chrysler();
    case "de":
    case "german": return new Mercedes();
    case "jp":
    case "japanese": return new Mazda();
    }
  }
}

class Chrysler implements Car {
  public String toString() { return "Chrysler"; }
}

class Mazda implements Car {
  public String toString() { return "Mazda"; }
}

class Mercedes implements Car {
  public String toString() { return "Mercedes"; }
}

public class CarEx {
  public static void main(String args[]) {
    Car car = CarFactory.create("us");
    System.out.println(car);
  }
}

Bu örnekte, kullanıcı sadece neye ihtiyacı olduğuna dair bir ipucu verir ve fabrika uygun bir şey inşa etmekte özgürdür. Bu bir bağımlılığın tersine çevrilmesidir: ‘Araba’ kavramının uygulayıcısı, inşa edilen somut nesnenin ayrıntılarını bilmeyen kullanıcı tarafından talep edilen uygun bir somut ‘Araba’yı iade etmekte özgürdür.

Bu, fabrikanın nasıl çalıştığına dair basit bir örnek, elbette bu örnekte somut sınıfları somutlaştırmak her zaman mümkündür; ancak bir pakette somut sınıfları gizleyerek bunu önleyebilirsiniz, öyle ki kullanıcı fabrikayı kullanmak zorunda kalır.

.Net Fiddle yukarıdaki örnek için.

Fabrika yöntemini (Java) uygulayarak fabrika örneği #

Niyet:

Bir nesne oluşturmak için bir arabirim tanımlayın, ancak hangi sınıfın başlatılacağına alt sınıfların karar vermesine izin verin. Fabrika Yöntemi, bir sınıfın somutlaştırmayı alt sınıflara ertelemesine izin verir.

UML diyagramı:

Ürün: Fabrika yönteminin oluşturduğu nesnelerin bir arabirimini tanımlar.

ConcreteProduct: Ürün arayüzünü uygular

Yaratıcı: Fabrika yöntemini bildirir

ConcreateCreator: Bir ConcreteProduct örneğini döndürmek için Fabrika yöntemini uygular

Sorun bildirimi: Oyun arabirimini tanımlayan Fabrika Yöntemlerini kullanarak bir Oyun Fabrikası oluşturun.

Kod parçacığı:

import java.util.HashMap;


/* Product interface as per UML diagram */
interface Game{
    /* createGame is a complex method, which executes a sequence of game steps */
    public void createGame();
}

/* ConcreteProduct implementation as per UML diagram */
class Chess implements Game{
    public Chess(){
        createGame();
    }
    public void createGame(){
        System.out.println("---------------------------------------");
        System.out.println("Create Chess game");
        System.out.println("Opponents:2");
        System.out.println("Define 64 blocks");
        System.out.println("Place 16 pieces for White opponent");
        System.out.println("Place 16 pieces for Black opponent");
        System.out.println("Start Chess game");
        System.out.println("---------------------------------------");
    }
}
class Checkers implements Game{
    public Checkers(){
        createGame();
    }
    public void createGame(){
        System.out.println("---------------------------------------");
        System.out.println("Create Checkers game");
        System.out.println("Opponents:2 or 3 or 4 or 6");
        System.out.println("For each opponent, place 10 coins");
        System.out.println("Start Checkers game");
        System.out.println("---------------------------------------");
    }
}
class Ludo implements Game{
    public Ludo(){
        createGame();
    }
    public void createGame(){
        System.out.println("---------------------------------------");
        System.out.println("Create Ludo game");
        System.out.println("Opponents:2 or 3 or 4");
        System.out.println("For each opponent, place 4 coins");
        System.out.println("Create two dices with numbers from 1-6");
        System.out.println("Start Ludo game");
        System.out.println("---------------------------------------");
    }
}

/* Creator interface as per UML diagram */
interface IGameFactory {
    public Game getGame(String gameName);
}

/* ConcreteCreator implementation as per UML diagram */
class GameFactory implements IGameFactory {
        
    HashMap<String,Game> games = new HashMap<String,Game>();
    /*  
        Since Game Creation is complex process, we don't want to create game using new operator every time.
        Instead we create Game only once and store it in Factory. When client request a specific game, 
        Game object is returned from Factory instead of creating new Game on the fly, which is time consuming
    */
    
    public GameFactory(){
        
        games.put(Chess.class.getName(),new Chess());
        games.put(Checkers.class.getName(),new Checkers());
        games.put(Ludo.class.getName(),new Ludo());        
    }
    public Game getGame(String gameName){
        return games.get(gameName);
    }
}

public class NonStaticFactoryDemo{
    public static void main(String args[]){
        if ( args.length < 1){
            System.out.println("Usage: java FactoryDemo gameName");
            return;
        }
     
        GameFactory factory = new GameFactory();
        Game game = factory.getGame(args[0]);
        System.out.println("Game="+game.getClass().getName());
    }
}

çıktı:

java NonStaticFactoryDemo Chess
---------------------------------------
Create Chess game
Opponents:2
Define 64 blocks
Place 16 pieces for White opponent
Place 16 pieces for Black opponent
Start Chess game
---------------------------------------
---------------------------------------
Create Checkers game
Opponents:2 or 3 or 4 or 6
For each opponent, place 10 coins
Start Checkers game
---------------------------------------
---------------------------------------
Create Ludo game
Opponents:2 or 3 or 4
For each opponent, place 4 coins
Create two dices with numbers from 1-6
Start Ludo game
---------------------------------------
Game=Chess

Bu örnek, bir “FactoryMethod” uygulayarak bir “Factory” sınıfını gösterir.

  1. “Oyun”, her tür oyun için arayüzdür. Karmaşık yöntemi tanımlar: createGame()

  2. ‘Satranç, Kızma Birader, Dama’, ‘createGame()’ için uygulama sağlayan oyunların farklı çeşitleridir.

  3. ‘public Game getGame(String gameName)’, ‘IGameFactory’ sınıfında ‘FactoryMethod’dur

  4. “GameFactory”, yapıcıda farklı oyun türlerini önceden oluşturur. IGameFactory fabrika yöntemini uygular.

  5. oyun Adı, ‘NotStaticFactoryDemo’ya komut satırı argümanı olarak iletilir

  6. “GameFactory"deki “getGame” bir oyun adını kabul eder ve ilgili “Game” nesnesini döndürür.

Ne zaman kullanılır:

  1. Fabrika: İstemciye/arayana nesne başlatma mantığını göstermek istemediğinizde
  2. Abstract Factory: İlişkili veya bağımlı nesnelerin ailelerine somut sınıflarını belirtmeden arayüz sağlamak istediğinizde
  3. Fabrika Yöntemi: Bir nesne oluşturmak için bir arayüz tanımlamak, ancak hangi sınıfın başlatılacağına alt sınıfların karar vermesine izin vermek

Diğer yaratılış kalıplarıyla karşılaştırma:

  1. Tasarım, Fabrika Yöntemi (daha az karmaşık, daha özelleştirilebilir, alt sınıflar çoğalır) kullanılarak başlar ve tasarımcı nerede daha fazla esnekliğin olduğunu keşfettikçe Soyut Fabrika, Prototip veya Oluşturucu‘ya (daha esnek, daha karmaşık) doğru gelişir. gereklidir

  2. Soyut Fabrika sınıfları genellikle Fabrika Yöntemleri ile uygulanır, ancak Prototip kullanılarak da uygulanabilirler.

Daha fazla okuma için referanslar: Sourcemaking design-patterns

Soyut Bir Fabrika #

Soyut fabrika (C++) #

Soyut fabrika deseni, fabrika işlevleri koleksiyonu aracılığıyla tutarlı bir nesne koleksiyonu elde etmenin bir yolunu sağlar. Her modelde olduğu gibi, bir dizi nesnenin oluşturulma şekli soyutlanarak kuplaj azaltılır, böylece kullanıcı kodu ihtiyaç duyduğu nesnelerin birçok detayından habersiz olur.

Aşağıdaki C++ örneği, aynı (varsayımsal) GUI ailesinden farklı türde nesnelerin nasıl elde edileceğini gösterir:

#include <iostream>

/* Abstract definitions */
class GUIComponent {
public:
  virtual ~GUIComponent() = default;
  virtual void draw() const = 0;
};
class Frame  : public GUIComponent {};
class Button : public GUIComponent {};
class Label  : public GUIComponent {};

class GUIFactory {
public:
  virtual ~GUIFactory() = default;
  virtual std::unique_ptr<Frame> createFrame() = 0;
  virtual std::unique_ptr<Button> createButton() = 0;
  virtual std::unique_ptr<Label> createLabel() = 0;
  static std::unique_ptr<GUIFactory> create(const std::string& type);
};

/* Windows support */
class WindowsFactory : public GUIFactory {
private:
    class WindowsFrame : public Frame {
    public:
      void draw() const override { std::cout << "I'm a Windows-like frame" << std::endl; }
    };
    class WindowsButton : public Button {
    public:
      void draw() const override { std::cout << "I'm a Windows-like button" << std::endl; }
    };
    class WindowsLabel : public Label {
    public:
      void draw() const override { std::cout << "I'm a Windows-like label" << std::endl; }
    };
public:
  std::unique_ptr<Frame> createFrame() override { return std::make_unique<WindowsFrame>(); }
  std::unique_ptr<Button> createButton() override { return std::make_unique<WindowsButton>(); }
  std::unique_ptr<Label> createLabel() override { return std::make_unique<WindowsLabel>(); }
};

/* Linux support */
class LinuxFactory : public GUIFactory {
private:
    class LinuxFrame : public Frame {
    public:
      void draw() const override { std::cout << "I'm a Linux-like frame" << std::endl; }
    };
    class LinuxButton : public Button {
    public:
      void draw() const override { std::cout << "I'm a Linux-like button" << std::endl; }
    };
    class LinuxLabel : public Label {
    public:
      void draw() const override { std::cout << "I'm a Linux-like label" << std::endl; }
    };
public:
  std::unique_ptr<Frame> createFrame() override { return std::make_unique<LinuxFrame>(); }
  std::unique_ptr<Button> createButton() override { return std::make_unique<LinuxButton>(); }
  std::unique_ptr<Label> createLabel() override { return std::make_unique<LinuxLabel>(); }
};

std::unique_ptr<GUIFactory> GUIFactory::create(const string& type) {
  if (type == "windows") return std::make_unique<WindowsFactory>();
  return std::make_unique<LinuxFactory>();
}

/* User code */
void buildInterface(GUIFactory& factory) {
  auto frame = factory.createFrame();
  auto button = factory.createButton();
  auto label = factory.createLabel();

  frame->draw();
  button->draw();
  label->draw();
}

int main(int argc, char *argv[]) {
  if (argc < 2) return 1;
  auto guiFactory = GUIFactory::create(argv[1]);
  buildInterface(*guiFactory);
}

Oluşturulan yürütülebilir dosya “abstractfactory” olarak adlandırılmışsa, çıktı şunları verebilir:

$ ./abstractfactory windows
I'm a Windows-like frame
I'm a Windows-like button
I'm a Windows-like label
$ ./abstractfactory linux  
I'm a Linux-like frame
I'm a Linux-like button
I'm a Linux-like label

IoC (C#) kullanan basit Fabrika örneği #

Fabrikalar, Inversion of Control (IoC) kitaplıkları ile birlikte de kullanılabilir.

  • Böyle bir fabrika için tipik kullanım durumu, çalışma zamanına kadar bilinmeyen parametrelere (geçerli Kullanıcı gibi) dayalı bir nesne oluşturmak istediğimiz zamandır.
  • Bu durumlarda, bu tür çalışma zamanı bağlamsal bilgilerini işlemek için IoC kitaplığını tek başına yapılandırmak bazen zor olabilir (imkansız değilse de), böylece onu bir fabrikaya sarabiliriz.

Örnek

  • Diyelim ki, özellikleri (ID, Güvenlik yetki seviyesi, vb.) çalışma zamanına kadar bilinmeyen bir ‘Kullanıcı’ sınıfımız var (çünkü mevcut kullanıcı uygulamayı kullanan herhangi biri olabilir).
  • Mevcut Kullanıcıyı almamız ve onlar için kullanıcının belirli eylemleri gerçekleştirmesine izin verilip verilmediğini kontrol etmek için kullanılabilecek bir ‘ISecurityToken’ almamız gerekiyor.
  • ISecurityToken’ın uygulanması, Kullanıcının düzeyine bağlı olarak değişecektir - diğer bir deyişle, ISecurityToken polimorfizm kullanır.

Bu durumda, onları IoC kitaplığında tanımlamayı kolaylaştırmak için Marker Interfaces kullanan iki uygulamamız var; bu durumda IoC kitaplığı, “ICContainer” soyutlaması ile oluşturulur ve tanımlanır.

Ayrıca birçok modern IoC fabrikasının, fabrikaların otomatik olarak oluşturulmasına izin veren ve ayrıca aşağıda gösterildiği gibi işaretleyici arayüzlerine olan ihtiyacı ortadan kaldıran yerel yeteneklere veya eklentilere sahip olduğunu unutmayın; ancak hepsinde olmadığı için, bu örnek basit, en düşük ortak işlevsellik konseptine hitap etmektedir.

//describes the ability to allow or deny an action based on PerformAction.SecurityLevel
public interface ISecurityToken
{
    public bool IsAllowedTo(PerformAction action);
}

//Marker interface for Basic permissions
public interface IBasicToken:ISecurityToken{};
//Marker interface for super permissions
public interface ISuperToken:ISecurityToken{};

//since IBasictoken inherits ISecurityToken, BasicToken can be treated as an ISecurityToken
public class BasicToken:IBasicToken
{
     public bool IsAllowedTo(PerformAction action)
     {
         //Basic users can only perform basic actions
         if(action.SecurityLevel!=SecurityLevel.Basic) return false;
         return true;
     }
}

public class SuperToken:ISuperToken
{
     public bool IsAllowedTo(PerformAction action)
     {
         //Super users can perform all actions         
         return true;
     }
}

Daha sonra, ‘ICContainer’ımızı bağımlılık olarak alacak bir ‘SecurityToken’ fabrikası oluşturacağız.

public class SecurityTokenFactory
{
   readonly IContainer _container;
   public SecurityTokenFactory(IContainer container)
   {
      if(container==null) throw new ArgumentNullException("container");
   }

   public ISecurityToken GetToken(User user)
   {
      if (user==null) throw new ArgumentNullException("user);
      //depending on the user security level, we return a different type; however all types implement ISecurityToken so the factory can produce them.
      switch user.SecurityLevel
      {
          case Basic:
           return _container.GetInstance<BasicSecurityToken>();
          case SuperUser:
           return _container.GetInstance<SuperUserToken>();
      }
   }
}

Bunları ‘ICContainer’a kaydettikten sonra:

IContainer.For<SecurityTokenFactory>().Use<SecurityTokenFactory>().Singleton(); //we only need a single instance per app
IContainer.For<IBasicToken>().Use<BasicToken>().PerRequest(); //we need an instance per-request
IContainer.For<ISuperToken>().Use<SuperToken>().PerRequest();//we need an instance per-request

tüketen kod, çalışma zamanında doğru belirteci almak için bunu kullanabilir:

readonly SecurityTokenFactory _tokenFactory;
...
...
public void LogIn(User user)
{
    var token = _tokenFactory.GetToken(user);
    user.SetSecurityToken(token);
}

Bu şekilde fabrika tarafından sağlanan kapsüllemeden ve ayrıca IoC kitaplığı tarafından sağlanan yaşam döngüsü yönetiminden faydalanırız.

Uçan Ağırlık Fabrikası (C#) #

Basit kelimelerle:

Belirli, zaten bilinen bir anahtar için bir Flyweight fabrikası, yanıt olarak her zaman aynı nesneyi verecektir. Yeni anahtarlar için örneği oluşturacak ve döndürecektir.

Fabrikayı kullanma:

ISomeFactory<string, object> factory = new FlyweightFactory<string, object>();

var result1 = factory.GetSomeItem("string 1");
var result2 = factory.GetSomeItem("string 2");
var result3 = factory.GetSomeItem("string 1");

//Objects from different keys
bool shouldBeFalse = result1.Equals(result2);

//Objects from same key
bool shouldBeTrue = result1.Equals(result3);

Uygulama:

public interface ISomeFactory<TKey,TResult> where TResult : new()
{
    TResult GetSomeItem(TKey key);
}

public class FlyweightFactory<TKey, TResult> : ISomeFactory<TKey, TResult> where TResult : new()
{
    public TResult GetSomeItem(TKey key)
    {
        TResult result;
        if(!Mapping.TryGetValue(key, out result))
        {
            result = new TResult();
            Mapping.Add(key, result);
        }
        return result;
    }

    public Dictionary<TKey, TResult> Mapping { get; set; } = new Dictionary<TKey, TResult>();
}

Ekstra notlar

Bu çözüme kendi yeni örneklerinizi oluşturmak yerine bir ‘IoC Kapsayıcı’ (burada farklı bir örnekte açıklandığı gibi) kullanımını eklemenizi tavsiye ederim. Bunu, kapsayıcıya “TResult” için yeni bir kayıt ekleyerek ve sonra ondan çözümleyerek (örnekteki “sözlük” yerine) yapabilirsiniz.

Fabrika yöntemi #

Fabrika yöntemi kalıbı, istemci kodunu ondan ayırmak için bir nesnenin örnekleme mantığını soyutlayan yaratıcı bir kalıptır.

Bir fabrika yöntemi, Abstract fabrika gibi başka bir fabrika modelinin uygulaması olan bir sınıfa ait olduğunda, genellikle Factory yöntemi modeli yerine o sınıf tarafından uygulanan modele başvurmak daha uygundur.

Fabrika yöntemi kalıbına, öncelikle fabrika olmayan bir sınıfa ait bir fabrika yöntemi açıklanırken daha sık başvurulur.

Örneğin, bir etki alanı kavramını temsil eden bir nesneye bir fabrika yöntemi yerleştirmek, eğer bu nesne başka bir nesnenin oluşturma sürecini basitleştirecek bir durumu kapsıyorsa avantajlı olabilir. Bir fabrika yöntemi, belirli bir bağlamın Her Yerde Bulunan Dili ile daha uyumlu bir tasarıma da yol açabilir.

İşte bir kod örneği:

//Without a factory method
Comment comment = new Comment(authorId, postId, "This is a comment");

//With a factory method
Comment comment = post.comment(authorId, "This is a comment");
