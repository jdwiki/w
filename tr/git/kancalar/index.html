Verilen git komutundaki tüm yerel kancaları atlamak için “–no-verify” veya “-n”. Örn: git commit -n

Bu sayfadaki bilgiler resmi Git dokümanlarından ve Atlassian’dan toplanmıştır.

Ön itme #

Git 1.8.2 ve üzeri sürümlerde mevcuttur.

Ön itme kancaları, bir itmenin yine de gitmesini önlemek için kullanılabilir. Bunun yararlı olmasının nedenleri arasında şunlar yer alır: belirli dallara yanlışlıkla yapılan manuel göndermeleri engelleme veya yerleşik bir denetim başarısız olursa göndermeleri engelleme (birim testleri, sözdizimi).

“.git/hooks/” altında “pre-push” ve (gotcha alert) adlı bir dosya oluşturularak ve dosyanın yürütülebilir olduğundan emin olarak bir ön-itme kancası oluşturulur: ‘chmod +x ./ git/hook/pre-push`.

Hannah Wolfe‘dan master’a geçişi engelleyen bir örnek:

#!/bin/bash

protected_branch='master'  
current_branch=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')

if [ $protected_branch = $current_branch ]  
then  
    read -p "You're about to push master, is that what you intended? [y|n] " -n 1 -r < /dev/tty
    echo
    if echo $REPLY | grep -E '^[Yy]$' > /dev/null
    then
        exit 0 # push will execute
    fi
    exit 1 # push will not execute
else  
    exit 0 # push will execute
fi

İşte Volkan Ünsal‘dan bir örnek. İletmeye izin vermeden önce RSpec testleri geçer:

#!/usr/bin/env ruby
require 'pty'
html_path = "rspec_results.html"
begin
  PTY.spawn( "rspec spec --format h > rspec_results.html" ) do |stdin, stdout, pid|
  begin
    stdin.each { |line| print line }
  rescue Errno::EIO
  end
end
rescue PTY::ChildExited
  puts "Child process exit!"
end

# find out if there were any errors  
html = open(html_path).read
examples = html.match(/(\d+) examples/)[0].to_i rescue 0
errors = html.match(/(\d+) errors/)[0].to_i rescue 0
if errors == 0 then
  errors = html.match(/(\d+) failure/)[0].to_i rescue 0
end
pending = html.match(/(\d+) pending/)[0].to_i rescue 0

if errors.zero?
  puts "0 failed! #{examples} run, #{pending} pending"
  # HTML Output when tests ran successfully:
  # puts "View spec results at #{File.expand_path(html_path)}"
  sleep 1
  exit 0
else
  puts "\aCOMMIT FAILED!!"
  puts "View your rspec results at #{File.expand_path(html_path)}"
  puts
  puts "#{errors} failed! #{examples} run, #{pending} pending"
  # Open HTML Ooutput when tests failed
  # `open #{html_path}`
  exit 1
end

Gördüğünüz gibi, birçok olasılık var, ancak temel parça, iyi şeyler olduysa ‘0’dan çıkmak’ ve kötü şeyler olduysa ‘1’den çıkmaktır. 1’den her çıktığınızda, push işlemi engellenecek ve kodunuz git push... çalıştırmadan önceki durumunda olacaktır.

İstemci tarafı kancalarını kullanırken, kullanıcıların bir anında “–no-verify” seçeneğini kullanarak tüm istemci tarafı kancalarını atlayabileceklerini unutmayın. Süreci zorlamak için kancaya güveniyorsanız, yanabilirsiniz.

Belgeler: https://git-scm.com/docs/githooks#_pre_push Resmi Örnek: https://github.com/git/git/blob/87c86dd14abe8db7d00b0df5661ef8cf147a72a3/templates/hooks–pre-push.sample

Taahhüt etmeden önce Maven derlemesini (veya başka bir derleme sistemini) doğrulayın #

.git/hooks/pre-commit

#!/bin/sh
if [ -s pom.xml ]; then
    echo "Running mvn verify"
    mvn clean verify
    if [ $? -ne 0 ]; then
        echo "Maven build failed"
        exit 1
    fi
fi

Belirli gönderimleri diğer depolara otomatik olarak iletin #

‘alma sonrası’ kancalar, gelen bildirimleri otomatik olarak başka bir havuza iletmek için kullanılabilir.

$ cat .git/hooks/post-receive

#!/bin/bash

IFS=' '
while read local_ref local_sha remote_ref remote_sha
do

  echo "$remote_ref" | egrep '^refs\/heads\/[A-Z]+-[0-9]+$' >/dev/null && {
    ref=`echo $remote_ref | sed -e 's/^refs\/heads\///'`
    echo Forwarding feature branch to other repository: $ref
    git push -q --force other_repos $ref
  }

done

Bu örnekte, “egrep” normal ifadesi belirli bir dal biçimini arar (burada: Jira sorunlarını adlandırmak için kullanılan JIRA-12345). Elbette tüm şubeleri yönlendirmek istiyorsanız bu kısmı kapalı bırakabilirsiniz.

Gönder-msg #

Bu kanca, ‘prepare-commit-msg’ kancasına benzer, ancak daha önce değil, kullanıcı bir taahhüt mesajı girdikten sonra çağrılır. Bu genellikle geliştiricileri, taahhüt mesajlarının yanlış bir biçimde olması durumunda uyarmak için kullanılır.

Bu kancaya iletilen tek argüman, mesajı içeren dosyanın adıdır. Kullanıcının girdiği mesajı beğenmediyseniz, bu dosyayı yerinde değiştirebilir (“prepare-commit-msg” ile aynı) veya sıfır olmayan bir durumla çıkarak işlemi tamamen iptal edebilirsiniz.

Aşağıdaki örnek, taahhüt mesajında ​​bir sayı ile takip edilen kelime biletinin olup olmadığını kontrol etmek için kullanılır.

word="ticket [0-9]"
isPresent=$(grep -Eoh "$word" $1)

if [[ -z $isPresent ]]
  then echo "Commit message KO, $word is missing"; exit 1;
  else echo "Commit message OK"; exit 0;
fi

Yerel kancalar #

Yerel kancalar yalnızca içinde bulundukları yerel depoları etkiler. Her geliştirici kendi yerel kancalarını değiştirebilir, böylece bir taahhüt politikasını zorlamanın bir yolu olarak güvenilir bir şekilde kullanılamazlar. Geliştiricilerin belirli yönergelere uymasını ve yoldaki olası sorunlardan kaçınmasını kolaylaştırmak için tasarlanmıştır.

Altı tür yerel kanca vardır: ön-taahhüt, hazır-taahhüt-msg, taahhüt-msg, taahhüt sonrası, ödeme sonrası ve yeniden temellendirme öncesi.

İlk dört kanca, taahhütlerle ilgilidir ve bir taahhütün yaşam döngüsündeki her bir parça üzerinde biraz kontrol sahibi olmanıza izin verir. Son ikisi, git checkout ve git rebase komutları için bazı ekstra eylemler veya güvenlik kontrolleri gerçekleştirmenize izin verir.

“Ön” kancalarının tümü, gerçekleşmek üzere olan eylemi değiştirmenize izin verirken, “post-” kancaları öncelikle bildirimler için kullanılır.

Ödeme sonrası #

Bu kanca, ‘post-commit’ kancasına benzer şekilde çalışır, ancak ‘git checkout’ ile bir referansı başarıyla kontrol ettiğinizde çağrılır. Bu, aksi takdirde kafa karışıklığına neden olacak otomatik oluşturulmuş dosyalardan oluşan çalışma dizininizi temizlemek için yararlı bir araç olabilir.

Bu kanca üç parametreyi kabul eder:

  1. önceki HEAD’in referansı,
  2. yeni HEAD’in referansı ve
  3. Bunun bir şube teslimi mi yoksa bir dosya teslimi mi olduğunu gösteren bir bayrak (sırasıyla ‘1’ veya ‘0’).

Çıkış durumunun “git checkout” komutu üzerinde hiçbir etkisi yoktur.

Post-taahhüt #

Bu kanca, ‘commit-msg’ kancasından hemen sonra çağrılır. ‘git commit’ işleminin sonucunu değiştiremez, bu nedenle öncelikle bildirim amacıyla kullanılır.

Betik hiçbir parametre almaz ve çıkış durumu taahhüdü hiçbir şekilde etkilemez.

Alım sonrası #

Bu kanca, başarılı bir itme işleminden sonra çağrılır. Genellikle bildirim amacıyla kullanılır.

Komut dosyası hiçbir parametre almaz, ancak standart girdi yoluyla “ön-alma” ile aynı bilgileri gönderir:

<old-value> <new-value> <ref-name>

Ön taahhüt #

Bu kanca, taahhüt edilmek üzere olan şeyi doğrulamak için “git commit” komutunu her çalıştırdığınızda yürütülür. İşlenmek üzere olan anlık görüntüyü incelemek için bu kancayı kullanabilirsiniz.

Bu tür kanca, gelen işlemin projenizin mevcut işlevselliğini bozmadığından emin olmak için otomatik testler yapmak için kullanışlıdır. Bu kanca türü, boşluk veya EOL hatalarını da kontrol edebilir.

Ön taahhüt komut dosyasına hiçbir argüman iletilmez ve sıfırdan farklı bir durumla çıkmak tüm işlemi iptal eder.

Mesaj hazırla-taahhüt et #

Bu kanca, metin düzenleyiciyi bir taahhüt mesajı ile doldurmak için ‘ön taahhüt’ kancasından sonra çağrılır. Bu genellikle, ezilmiş veya birleştirilmiş taahhütler için otomatik olarak oluşturulan taahhüt mesajlarını değiştirmek için kullanılır.

Bu kancaya bir ila üç argüman iletilir:

  • Mesajı içeren geçici bir dosyanın adı.
  • Taahhüt türü ya
  • mesaj (-m veya -F seçeneği),
  • şablon (-t seçeneği),
  • birleştirme (bir birleştirme taahhüdü ise) veya
  • squash (eğer diğer taahhütleri eziyorsa).
  • İlgili taahhüdün SHA1 karması. Bu yalnızca “-c”, “-C” veya “–amend” seçeneği verilmişse verilir.

‘Ön taahhüt’e benzer şekilde, sıfır olmayan bir durumla çıkmak, taahhüdü iptal eder.

Yeniden temel alma #

Bu kanca, “git rebase” kod yapısını değiştirmeye başlamadan önce çağrılır. Bu kanca tipik olarak bir yeniden temelleme işleminin uygun olduğundan emin olmak için kullanılır.

Bu kanca 2 parametre alır:

  1. serinin çatallandığı yukarı akış kolu ve
  2. yeniden temellenen dal (mevcut dalı yeniden temellendirirken boş).

Sıfırdan farklı bir durumla çıkış yaparak rebase işlemini iptal edebilirsiniz.

Ön-alma #

Bu kanca, birileri depoya taahhütleri göndermek için ‘git Push’u her kullandığında yürütülür. Kaynak (yerel) depoda değil, her zaman push’un hedefi olan uzak depoda bulunur.

Kanca, herhangi bir referans güncellenmeden önce çalışır. Genellikle her türlü geliştirme politikasını uygulamak için kullanılır.

Komut dosyası hiçbir parametre almaz, ancak iletilen her başvuru, aşağıdaki biçimde standart girdide ayrı bir satırda komut dosyasına iletilir:

<old-value> <new-value> <ref-name>

Güncelleme #

Bu kanca, “ön-alma"dan sonra çağrılır ve aynı şekilde çalışır. Herhangi bir şey gerçekten güncellenmeden önce çağrılır, ancak aynı anda tüm referanslar yerine gönderilen her referans için ayrı olarak çağrılır.

Bu kanca aşağıdaki 3 argümanı kabul eder:

  • güncellenen hakemin adı,
  • başvuruda saklanan eski nesne adı ve
  • başvuruda saklanan yeni nesne adı.

Bu, “ön-alma"ya iletilen bilgilerin aynısıdır, ancak “güncelleme” her başvuru için ayrı olarak çağrıldığından, bazı başvurulara izin verirken bazı başvuruları reddedebilirsiniz.
