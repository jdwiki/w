Gerçekten büyük SVN depolarını klonlama

SVN depo geçmişiniz gerçekten çok büyükse, git-svn’nin SVN deposunun tüm geçmişini yeniden oluşturması gerektiğinden bu işlem saatler sürebilir. Neyse ki SVN deposunu yalnızca bir kez klonlamanız gerekir; diğer herhangi bir git deposunda olduğu gibi, repo klasörünü diğer ortak çalışanlara kopyalayabilirsiniz. Klasörü birden çok bilgisayara kopyalamak, büyük SVN depolarını sıfırdan klonlamaktan daha hızlı olacaktır.

Taahhütler ve SHA1 hakkında

“git svn dcommit” komutunu kullanırken yerel git taahhütleriniz yeniden yazılacaktır. Bu komut, git commit’in SVN sunucusunda oluşturulan SVN revizyonuna atıfta bulunan mesajına bir metin ekleyecektir ve bu çok kullanışlıdır. Ancak, yeni bir metin eklemek, gerçekte yapılamayan mevcut bir taahhüdün mesajını değiştirmeyi gerektirir: git taahhütleri değişmezdir. Çözüm, aynı içeriğe ve yeni mesaja sahip yeni bir taahhüt oluşturmaktır, ancak teknik olarak yine de yeni bir taahhüttür (yani git commit’in SHA1’i değişecektir)

git-svn için oluşturulan git taahhütleri yerel olduğundan, git taahhütleri için SHA1 kimlikleri her git deposu arasında farklıdır! Bu, aynı işlemin her yerel git deposunda farklı bir SHA1’e sahip olacağından, başka bir kişiden gelen bir işleme atıfta bulunmak için bir SHA1 kullanamayacağınız anlamına gelir. Deponun farklı kopyaları arasında bir işleme başvurmak istiyorsanız, SVN sunucusuna gönderdiğinizde taahhüt mesajına eklenen svn revizyon numarasına güvenmeniz gerekir.

SHA1’i yerel işlemler için kullanabilirsiniz (belirli bir taahhüdü göster/farklılaştır, seçimleri ve sıfırlamaları vb.)

Sorun giderme #

git svn rebase komutu bir sağlama toplamı uyuşmazlığı hatası veriyor

git svn rebase komutu şuna benzer bir hata veriyor:

  Checksum mismatch: <path_to_file> <some_kind_of_sha1>
  expected: <checksum_number_1>
    got: <checksum_number_2>

Bu sorunun çözümü, sorunlu dosya son kez değiştirildiğinde svn’yi revizyona sıfırlamak ve SVN geçmişinin geri yüklenmesi için bir git svn getirme işlemi yapmaktır. SVN sıfırlamasını gerçekleştirme komutları şunlardır:

  • git log -1 – <path_to_file> (kabul mesajında ​​görünen SVN revizyon numarasını kopyalayın)
  • git svn reset <revision_number>
  • git svn getirme

SVN’den tekrar veri gönderebilmeli/çekebilmelisiniz

** Dosya, taahhütte bulunamadı** SVN’den almaya veya çekmeye çalıştığınızda buna benzer bir hata alıyorsunuz.

<file_path> was not found in commit <hash>

Bu, SVN’deki bir revizyonun, herhangi bir nedenle yerel kopyanızda bulunmayan bir dosyayı değiştirmeye çalıştığı anlamına gelir. Bu hatadan kurtulmanın en iyi yolu, o dosyanın yolunu yok sayarak bir getirme işlemini zorlamak ve en son SVN revizyonunda durumuna güncellenecektir:

  • git svn fetch --ignore-paths <file_path>

SVN deposunu klonlama #

Komutla havuzun yeni bir yerel kopyasını oluşturmanız gerekir.

git svn klonu SVN_REPO_ROOT_URL [DEST_FOLDER_PATH] -T TRUNK_REPO_PATH -t TAGS_REPO_PATH -b BRANCHES_REPO_PATH

SVN deponuz standart düzeni (gövde, dallar, etiketler klasörleri) takip ediyorsa, bazı yazarak kaydedebilirsiniz:

git svn klonu -s SVN_REPO_ROOT_URL [DEST_FOLDER_PATH]

git svn klonu her SVN revizyonunu tek tek kontrol eder ve geçmişi yeniden oluşturmak için yerel deponuzda bir git commit yapar. SVN deposunda çok fazla taahhüt varsa, bu biraz zaman alacaktır.

Komut bittiğinde, SVN deposundaki ana hat dalını izleyen master adlı yerel bir şubeye sahip tam teşekküllü bir git deposuna sahip olacaksınız.

Yerel değişiklikleri SVN’ye aktarma #

Komuta

git svn dcommit

yerel git taahhütlerinizin her biri için bir SVN revizyonu oluşturacaktır. SVN’de olduğu gibi, yerel git geçmişiniz SVN deposundaki en son değişikliklerle uyumlu olmalıdır, bu nedenle komut başarısız olursa önce bir “git svn rebase” gerçekleştirmeyi deneyin.

Yerel olarak çalışma #

Yerel git deponuzu normal git komutlarıyla normal bir git deposu olarak kullanın:

  • git add FILE ve git checkout -- FILE Bir dosyayı hazırlamak/sahneyi kaldırmak için
  • git commit Değişikliklerinizi kaydetmek için. Bu taahhütler yerel olacak ve normal bir git deposunda olduğu gibi SVN deposuna “itilmeyecek”
  • git stash ve git stash pop Stash kullanımına izin verir
  • git reset HEAD --hard Tüm yerel değişikliklerinizi geri alın
  • git log Depodaki tüm geçmişe erişin
  • git rebase -i yerel geçmişinizi özgürce yeniden yazabilmeniz için
  • yerel şubeler oluşturmak için “git şube” ve “git checkout”

git-svn belgelerinde belirtildiği gibi “Subversion, Git’ten çok daha az karmaşık bir sistemdir”, bu nedenle Subversion sunucusundaki geçmişi bozmadan git’in tüm gücünü kullanamazsınız. Neyse ki kurallar çok basit: Geçmişi doğrusal tutun

Bu, hemen hemen her türlü git işlemini yapabileceğiniz anlamına gelir: dallar oluşturma, silme/yeniden sıralama/taahhütleri ezme, geçmişi hareket ettirme, taahhütleri silme vb. Birleştirme dışında her şey. Yerel şubelerin geçmişini yeniden birleştirmeniz gerekiyorsa, bunun yerine git rebase kullanın.

Bir birleştirme gerçekleştirdiğinizde, bir birleştirme taahhüdü oluşturulur. Birleştirme taahhütleriyle ilgili özel olan şey, iki ebeveyne sahip olmaları ve bu da tarihi doğrusal olmayan hale getiriyor. Doğrusal olmayan geçmiş, depoya bir birleştirme taahhüdünü “itmeniz” durumunda SVN’yi karıştıracaktır.

Ancak endişelenmeyin: ** bir git birleştirme taahhüdünü SVN’ye “ittirirseniz” hiçbir şeyi bozmazsınız**. Bunu yaparsanız, git birleştirme taahhüdü svn sunucusuna gönderildiğinde, bu birleştirme için tüm taahhütlerin tüm değişikliklerini içerecektir, bu nedenle bu taahhütlerin geçmişini kaybedersiniz, ancak kodunuzdaki değişiklikleri kaybetmezsiniz.

SVN’den en son değişiklikleri alma #

‘git pull’ ile eşdeğer komuttur

git svn rebase

Bu, SVN deposundaki tüm değişiklikleri alır ve bunları mevcut şubenizdeki yerel taahhütlerinizin * üstüne * uygular.

komutunu da kullanabilirsiniz.

git svn fetch

değişiklikleri SVN deposundan almak ve yerel makinenize getirmek, ancak bunları yerel şubenize uygulamadan.

Boş klasörleri işleme #

git, klasör kavramını tanımıyor, sadece dosyalar ve onların dosya yollarıyla çalışıyor. Bu, git’in boş klasörleri izlemediği anlamına gelir. Ancak SVN yapar. git-svn kullanmak, varsayılan olarak git ile boş klasörleri içeren yaptığınız herhangi bir değişikliğin SVN’ye yayılmayacağı anlamına gelir.

Bir yorum yayınlarken --rmdir bayrağının kullanılması bu sorunu düzeltir ve içindeki son dosyayı yerel olarak silerseniz SVN’deki boş bir klasörü kaldırır:

git svn dcommit --rmdir

Maalesef mevcut boş klasörleri kaldırmaz: bunu manuel olarak yapmanız gerekir.

Her dcommit yaptığınızda bayrak eklemekten kaçınmak veya bir git GUI aracı (SourceTree gibi) kullanıyorsanız güvenli oynamak için bu davranışı şu komutla varsayılan olarak ayarlayabilirsiniz:

git config --global svn.rmdir true

Bu, .gitconfig dosyanızı değiştirir ve şu satırları ekler:

[svn]
rmdir = true

SVN için boş tutulması gereken tüm izlenmeyen dosya ve klasörleri kaldırmak için git komutunu kullanın:

git temiz -fd

Lütfen dikkat: önceki komut, izlenmeyen tüm dosyaları ve boş klasörleri, hatta SVN tarafından izlenmesi gerekenler bile kaldıracaktır! SVN tarafından izlenen boş klasörleri yeniden oluşturmanız gerekirse, şu komutu kullanın:

git svn mkdirs

Uygulamalarda bu, çalışma alanınızı izlenmeyen dosya ve klasörlerden temizlemek istiyorsanız, SVN tarafından izlenen boş klasörleri yeniden oluşturmak için her zaman her iki komutu da kullanmanız gerektiği anlamına gelir:

git clean -fd && git svn mkdirs
