Git ile yapılan taahhütler, yazarlara koddaki değişiklikleri atfederek hesap verebilirlik sağlar. Git, taahhütlerin özgüllüğü ve güvenliği için birden çok özellik sunar. Bu konu, Git ile taahhütte bulunmaya yönelik uygun uygulamaları ve prosedürleri açıklar ve gösterir.

Sözdizimi #

  • git commit [bayraklar]

Parametreler #

Parametre             Ayrıntılar
–mesaj, -m Taahhüde dahil edilecek mesaj. Bu parametrenin belirtilmesi, Git’in normal düzenleyici açma davranışını atlar.
–değiştir Şu anda sahnelenen değişikliklerin önceki işleme eklenmesi (değiştirilmesi) gerektiğini belirtin. Dikkatli olun, bu tarihi yeniden yazabilir!
–no-edit Seçili taahhüt mesajını bir düzenleyici başlatmadan kullanın. Örneğin, git commit --amend --no-edit, taahhüt mesajını değiştirmeden bir taahhütte değişiklik yapar.
–hepsi, -a Henüz aşamalı olmayan değişiklikler de dahil olmak üzere tüm değişiklikleri gerçekleştirin.
–tarih Taahhütle ilişkilendirilecek tarihi manuel olarak ayarlayın.
–yalnızca Yalnızca belirtilen yolları taahhüt edin. Bu, söylenmediği sürece, şu anda sahnelediğiniz şeyi taahhüt etmeyecektir.
–patch, -p Hangi değişikliklerin uygulanacağını seçmek için etkileşimli yama seçimi arayüzünü kullanın.
–yardım git commit için kılavuz sayfasını görüntüler
-S[keyid], -S –gpg-sign[=keyid], -S –no-gpg-sign İmza tamamlama, GPG-imza oluşturma, karşı komut `commit.gpgSign’ yapılandırma değişkeni
-n, –no-verify Bu seçenek, ön-taahhüt ve taahhüt-msg kancalarını atlar. Ayrıca bkz. https://www.wikiod.com/tr/git/kancalar

İyi taahhüt mesajları #

‘git log’ üzerinden geçen birinin, her bir taahhüdün neyle ilgili olduğunu kolayca anlaması önemlidir. İyi taahhüt mesajları genellikle bir izleyicideki bir dizi görevi veya sorunu ve neyin, neden yapıldığının ve bazen de nasıl yapıldığının kısa bir açıklamasını içerir.

Daha iyi mesajlar şöyle görünebilir:

TASK-123: Implement login through OAuth
TASK-124: Add auto minification of JS/CSS files
TASK-125: Fix minifier error when name > 200 chars

Oysa aşağıdaki mesajlar o kadar yararlı olmayacaktır:

fix                         // What has been fixed?
just a bit of a change      // What has changed?
TASK-371                    // No description at all, reader will need to look at the tracker themselves for an explanation
Implemented IFoo in IBar    // Why it was needed?

Bir taahhüt mesajının doğru ruh halinde yazılıp yazılmadığını test etmenin bir yolu, boşluğu mesajla değiştirmek ve mantıklı olup olmadığına bakmaktır:

Bu taahhüdü eklersem, depoma ___ yapacağım.

Harika bir git taahhüt mesajının yedi kuralı #

  1. Konu satırını gövdeden boş bir satırla ayırın
  2. Konu satırını 50 karakterle sınırlayın
  3. Konu satırını büyük harfle yazın
  4. Konu satırını nokta ile sonlandırmayın
  5. Konu satırında [zorunlu ruh hali] (https://en.wikipedia.org/wiki/Imperative_mood) kullanın
  6. Gövdenin her satırını 72 karakterle manuel olarak sarın
  7. nasıl yerine _neyi ve neden açıklamak için gövdeyi kullanın

Chris Beam’in blogundan 7 kural.

Bir taahhüdü değiştirme #

En son taahhüdünüz henüz yayınlanmadıysa (bir yukarı akış deposuna gönderilmediyse), taahhüdünüzü değiştirebilirsiniz.

git commit --amend

Bu, şu anda sahnelenen değişiklikleri önceki işleme koyacaktır.

Not: Bu, yanlış bir taahhüt mesajını düzenlemek için de kullanılabilir. Varsayılan düzenleyiciyi (genellikle vi / vim / emacs) getirecek ve önceki mesajı değiştirmenize izin verecektir.

Taahhüt mesajını satır içi belirtmek için:

git commit --amend -m "New commit message"

Veya önceki taahhüt mesajını değiştirmeden kullanmak için:

git commit --amend --no-edit

Değişiklik, taahhüt tarihini günceller, ancak yazar tarihine dokunulmaz. Git’e bilgileri yenilemesini söyleyebilirsiniz.

git commit --amend --reset-author

Ayrıca taahhüdün yazarını şu şekilde değiştirebilirsiniz:

git commit --amend --author "New Author <email@address.com>"

Not: En son taahhüdün değiştirilmesinin onu tamamen değiştirdiğini ve önceki taahhüdün şube geçmişinden kaldırıldığını unutmayın. Halka açık depolarla ve diğer ortak çalışanlarla şubelerde çalışırken bu akılda tutulmalıdır.

Bu, daha önceki taahhüdün zaten itilmiş olması durumunda, onu değiştirdikten sonra ‘–force’ zorlamanız gerekeceği anlamına gelir.

Bir düzenleyici açmadan taahhütte bulunma #

Git, “git commit” çalıştırdığınızda genellikle bir düzenleyici (“vim” veya “emacs” gibi) açar. Komut satırından bir mesaj belirtmek için -m seçeneğini iletin:

git commit -m "Commit message here"

Taahhüt mesajınız birden çok satırdan geçebilir:

git commit -m "Commit 'subject line' message here

More detailed description follows here (after a blank line)."

Alternatif olarak, birden çok “-m” bağımsız değişkeni iletebilirsiniz:

git commit -m "Commit summary" -m "More detailed description follows here"

Bkz. Git Taahhüt Mesajı Nasıl Yazılır.

Udacity Git Commit Message Style Guide

Aşama ve taahhüt değişiklikleri #

Temeller #

Kaynak kodunuzda değişiklik yaptıktan sonra, taahhütte bulunmadan önce bu değişiklikleri Git ile aşamalı yapmalısınız.

Örneğin, “README.md” ve “program.py"yi değiştirirseniz:

git add README.md program.py

Bu, git’e, dosyaları bir sonraki yaptığınız işleme eklemek istediğinizi söyler.

Ardından, değişikliklerinizi ile taahhüt edin

git commit

Bunun, genellikle [vim][2] olan bir metin düzenleyici açacağını unutmayın. Vim’e aşina değilseniz, insert moduna geçmek için ‘i’ tuşuna basabileceğinizi, taahhüt mesajınızı yazabileceğinizi, ardından kaydetmek ve çıkmak için ‘Esc’ ve ‘:wq’ tuşlarına basabileceğinizi bilmek isteyebilirsiniz. Metin düzenleyiciyi açmaktan kaçınmak için mesajınıza “-m” işaretini eklemeniz yeterlidir.

git commit -m "Commit message here"

Taahhüt mesajları genellikle bazı belirli biçimlendirme kurallarını takip eder, daha fazla bilgi için [İyi taahhüt mesajları][3] bölümüne bakın.

Kısayollar #

Her birini listelemek yerine dizindeki birçok dosyayı değiştirdiyseniz, şunları kullanabilirsiniz:

git add --all        # equivalent to "git add -a"

Veya tüm değişiklikleri, silinmiş dosyalar hariç, üst düzey dizinden ve alt dizinlerden eklemek için:

git add .

Veya yalnızca şu anda izlenen (“güncelleme”) dosyaları eklemek için:

git add -u

İstenirse, aşamalı değişiklikleri gözden geçirin:

git status           # display a list of changed files
git diff --cached    # shows staged changes inside staged files

Son olarak, değişiklikleri gerçekleştirin:

git commit -m "Commit message here"

Alternatif olarak, yalnızca mevcut dosyaları değiştirdiyseniz veya dosyaları sildiyseniz ve yenilerini oluşturmadıysanız, “git add” ve “git commit” eylemlerini tek bir komutta birleştirebilirsiniz:

git commit -am "Commit message here"

Bunun tüm değiştirilmiş dosyaları git add --all ile aynı şekilde aşamalandıracağını unutmayın.

Hassas veri #

Parolalar ve hatta özel anahtarlar gibi hassas verileri asla taahhüt etmemelisiniz. Bu durum gerçekleşirse ve değişiklikler zaten merkezi bir sunucuya aktarılırsa, hassas verileri tehlikeye atılmış olarak kabul edin. Aksi takdirde, daha sonra bu tür verileri kaldırmak mümkündür. Hızlı ve kolay bir çözüm, “BFG Repo-Cleaner” kullanımıdır: https://rtyley.github.io/bfg-repo-cleaner/.

bfg --replace-text passwords.txt my-repo.git komutu, passwords.txt dosyasındaki şifreleri okur ve bunları ***KALDIRILDI*** ile değiştirir. Bu işlem, tüm deponun önceki tüm taahhütlerini dikkate alır.

1: https://www.wikiod.com/tr/git/yaplandrma#Hangi düzenleyicinin kullanılacağını ayarlama [2]: https://www.wikiod.com/tr/vim [3]: https://www.wikiod.com/tr/git/taahhut#İyi taahhüt mesajları

Değişiklikleri doğrudan taahhüt etme #

Genellikle, “git commit” yapmadan önce dizine değişiklik eklemek için “git add” veya “git rm” kullanmanız gerekir. Kaldırma işlemleri de dahil olmak üzere her değişikliği (izlenen dosyalara) otomatik olarak dizine eklemek için “-a” veya “–all” seçeneğini iletin:

git commit -a

Ayrıca bir taahhüt mesajı eklemek isterseniz, şunları yaparsınız:

git commit -a -m "your commit message goes here"

Ayrıca, iki bayrağa katılabilirsiniz:

git commit -am "your commit message goes here"

Tüm dosyaları bir kerede kaydetmeniz gerekmez. “-a” veya “–all” işaretini atlayın ve doğrudan hangi dosyayı işlemek istediğinizi belirtin:

git commit path/to/a/file -m "your commit message goes here"

Birden fazla belirli dosyayı doğrudan işlemek için bir veya daha fazla dosya, dizin ve kalıp da belirtebilirsiniz:

git commit path/to/a/file path/to/a/folder/* path/to/b/file -m "your commit message goes here"

Boş bir taahhüt oluşturma #

Genel olarak konuşursak, boş taahhütler (veya ebeveyn ile aynı durumda olan taahhütler) bir hatadır.

Bununla birlikte, derleme kancalarını, CI sistemlerini ve bir taahhüdü tetikleyen diğer sistemleri test ederken, sahte bir dosyayı düzenlemek/dokunmak zorunda kalmadan kolayca taahhütler oluşturabilmek kullanışlıdır.

–allow-empty taahhüdü kontrolü atlayacaktır.

git commit -m "Bu boş bir taahhüttür" --allow-empty

Başkası adına taahhütte bulunmak #

Taahhüt ettiğiniz kodu başka biri yazmışsa, --yazar seçeneğiyle ona kredi verebilirsiniz:

git commit -m "msg" --author "John Smith <johnsmith@example.com>"

Git’in önceki yazarları aramak için kullanacağı bir kalıp da sağlayabilirsiniz:

git commit -m "msg" --author "John"

Bu durumda, “John” içeren bir yazarla yapılan en son taahhüdün yazar bilgileri kullanılacaktır.

GitHub’da, yukarıdaki yollardan herhangi biriyle yapılan taahhütler, büyük bir yazarın küçük resmini, taahhüt edenin daha küçük ve önde olacak şekilde gösterecektir:

Taahhüt için hangi satırların sahneleneceğini seçme #

Bir veya daha fazla dosyada birçok değişikliğiniz olduğunu, ancak her dosyadan yalnızca bazı değişiklikleri yapmak istediğinizi varsayalım, aşağıdakileri kullanarak istediğiniz değişiklikleri seçebilirsiniz:

git add -p

veya

git add -p [file]

Değişikliklerinizin her biri ayrı ayrı görüntülenecek ve her değişiklik için aşağıdaki seçeneklerden birini seçmeniz istenecektir:

y - Yes, add this hunk

n - No, don’t add this hunk

d - No, don’t add this hunk, or any other remaining hunks for this file.
    Useful if you’ve already added what you want to, and want to skip over the rest.

s - Split the hunk into smaller hunks, if possible

e - Manually edit the hunk.  This is probably the most powerful option.
    It will open the hunk in a text editor and you can edit it as needed.

Bu, seçtiğiniz dosyaların bölümlerini hazırlayacaktır. Ardından, aşağıdaki gibi tüm aşamalı değişiklikleri gerçekleştirebilirsiniz:

git commit -m 'Commit Message'

Aşamalı olmayan veya yapılmayan değişiklikler çalışma dosyalarınızda görünmeye devam edecek ve gerekirse daha sonra gerçekleştirilebilir. Veya kalan değişiklikler istenmiyorsa, şunlarla birlikte atılabilirler:

git reset --hard

Bu yaklaşım, büyük bir değişikliği daha küçük taahhütlere bölmenin yanı sıra, taahhüt etmek üzere olduğunuz şeyi incelemek için de yararlıdır. Her değişikliği ayrı ayrı onaylayarak, ne yazdığınızı kontrol etme fırsatına sahip olursunuz ve println/logging deyimleri gibi yanlışlıkla istenmeyen kodların hazırlanmasını önleyebilirsiniz.

Belirli dosyalarda değişiklik yapma #

Belirli dosyalarda yapılan değişiklikleri kabul edebilir ve “git add” kullanarak bunları hazırlamayı atlayabilirsiniz:

git commit file1.c file2.h

Veya önce dosyaları aşamalandırabilirsiniz:

git add file1.c file2.h

ve bunları daha sonra taahhüt edin:

git commit

GPG imzalama taahhütleri #

  1. Anahtar kimliğinizi belirleyin

     gpg --list-secret-keys --keyid-format LONG

 /Users/davidcondrey/.gnupg/secring.gpg
 --------------------------------------
 sec   2048R/YOUR-16-DIGIT-KEY-ID YYYY-MM-DD [expires: YYYY-MM-DD]

    Your ID is a alphanumeric 16-digit code following the first forward-slash.

  2. Git yapılandırmanızda anahtar kimliğinizi tanımlayın

     git config --global user.signingkey YOUR-16-DIGIT-KEY-ID

  3. 1.7.9 sürümünden itibaren git commit, taahhütlerinize imza eklemek için -S seçeneğini kabul eder. Bu seçeneği kullanmak, GPG parolanızı isteyecek ve imzanızı kaydetme günlüğüne ekleyecektir.

     git commit -S -m "Your commit message"

Belirli bir tarihte taahhütte bulunma #

git commit -m 'Fix UI bug' --date 2016-07-01

“–date” parametresi yazar tarihini ayarlar. Bu tarih, örneğin “git log"un standart çıktısında görünecektir.

Taahhüt tarihini de zorlamak için:

GIT_COMMITTER_DATE=2016-07-01 git commit -m 'Fix UI bug' --date 2016-07-01

date parametresi, GNU tarihi tarafından desteklenen esnek biçimleri kabul eder, örneğin:

git commit -m 'Fix UI bug' --date yesterday
git commit -m 'Fix UI bug' --date '3 days ago'
git commit -m 'Fix UI bug' --date '3 hours ago'

Tarih saati belirtmediğinde, geçerli saat kullanılacak ve yalnızca tarih geçersiz kılınacaktır.

Bir taahhüdün zamanını değiştirme #

Kullanarak bir taahhüdün zamanını değiştirirsiniz

git commit --amend --date="Thu Jul 28 11:30 2016 -0400"

ya da

git commit --amend --date="now"

Bir taahhüdün yazarını değiştirme #

Yanlış yazar olarak bir taahhütte bulunursanız, onu değiştirebilir ve sonra düzeltebilirsiniz.

git config user.name "Full Name"
git config user.email "email@example.com"

git commit --amend --reset-author
