.gitignore dosyası içeren dosya ve dizinleri yoksayma #

Git’in belirli dosyaları ve dizinleri yok saymasını sağlayabilirsiniz — yani Git — deponuzda bir veya daha fazla .gitignore dosyası oluşturarak.

Yazılım projelerinde, “.gitignore” genellikle oluşturma işlemi sırasında veya çalışma zamanında oluşturulan dosyaların ve/veya dizinlerin bir listesini içerir. “.gitignore” dosyasındaki girişler, aşağıdakilere işaret eden adlar veya yollar içerebilir:

  1. geçici kaynaklar ör. önbellekler, günlük dosyaları, derlenmiş kod vb.
  2. Diğer geliştiricilerle paylaşılmaması gereken yerel yapılandırma dosyaları
  3. Oturum açma parolaları, anahtarlar ve kimlik bilgileri gibi gizli bilgileri içeren dosyalar

Üst düzey dizinde oluşturulduğunda, kurallar tüm depo boyunca tüm dosyalara ve alt dizinlere yinelemeli olarak uygulanacaktır. Bir alt dizinde oluşturulduğunda, kurallar o belirli dizine ve onun alt dizinlerine uygulanacaktır.

Bir dosya veya dizin yoksayıldığında, şunlar olmaz:

  1. Git tarafından izlenir
  2. “git durumu” veya “git diff” gibi komutlarla rapor edilir
  3. git add -A gibi komutlarla sahnelenir

İzlenen dosyaları yok saymanız gereken olağandışı bir durumda, özel dikkat gösterilmelidir. Bkz.: Git deposuna önceden taahhüt edilmiş dosyaları yoksay.

Örnekler #

glob dosya kalıplarına dayalı bir “.gitignore” dosyasındaki bazı genel kural örnekleri:

# Lines starting with `#` are comments.

# Ignore files called 'file.ext'
file.ext

# Comments can't be on the same line as rules!
# The following line ignores files called 'file.ext # not a comment'
file.ext # not a comment 

# Ignoring files with full path.
# This matches files in the root directory and subdirectories too.
# i.e. otherfile.ext will be ignored anywhere on the tree.
dir/otherdir/file.ext
otherfile.ext

# Ignoring directories
# Both the directory itself and its contents will be ignored.
bin/
gen/

# Glob pattern can also be used here to ignore paths with certain characters.
# For example, the below rule will match both build/ and Build/
[bB]uild/

# Without the trailing slash, the rule will match a file and/or
# a directory, so the following would ignore both a file named `gen`
# and a directory named `gen`, as well as any contents of that directory
bin
gen

# Ignoring files by extension
# All files with these extensions will be ignored in
# this directory and all its sub-directories.
*.apk
*.class

# It's possible to combine both forms to ignore files with certain
# extensions in certain directories. The following rules would be
# redundant with generic rules defined above.
java/*.apk
gen/*.class

# To ignore files only at the top level directory, but not in its
# subdirectories, prefix the rule with a `/`
/*.apk
/*.class

# To ignore any directories named DirectoryA 
# in any depth use ** before DirectoryA
# Do not forget the last /, 
# Otherwise it will ignore all files named DirectoryA, rather than directories
**/DirectoryA/
# This would ignore 
# DirectoryA/
# DirectoryB/DirectoryA/ 
# DirectoryC/DirectoryB/DirectoryA/
# It would not ignore a file named DirectoryA, at any level

# To ignore any directory named DirectoryB within a 
# directory named DirectoryA with any number of 
# directories in between, use ** between the directories
DirectoryA/**/DirectoryB/
# This would ignore 
# DirectoryA/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryW/DirectoryB/

# To ignore a set of files, wildcards can be used, as can be seen above.
# A sole '*' will ignore everything in your folder, including your .gitignore file.
# To exclude specific files when using wildcards, negate them.
# So they are excluded from the ignore list:
!.gitignore 

# Use the backslash as escape character to ignore files with a hash (#)
# (supported since 1.6.2.1)
\#*#

Çoğu “.gitignore” dosyası çeşitli dillerde standarttır, bu nedenle başlamak için, projenize klonlama veya kopyalama/değiştirme için dile göre listelenen [örnek “.gitignore” dosyaları][4] buradadır. Alternatif olarak, yeni bir proje için bir [çevrimiçi araç][5] kullanarak otomatik olarak bir başlangıç ​​dosyası oluşturmayı düşünebilirsiniz.

Diğer .gitignore biçimleri #

“.gitignore” dosyalarının havuzun bir parçası olarak işlenmesi amaçlanmıştır. Belirli dosyaları yoksayma kurallarını uygulamadan yoksaymak istiyorsanız, işte bazı seçenekler:

  • “.git/info/exclude” dosyasını düzenleyin (".gitignore" ile aynı sözdizimini kullanarak). Kurallar, depo kapsamında global olacaktır;
  • Yoksayma kurallarını tüm yerel depolar:

Ayrıca, global git yapılandırmasını şu şekilde değiştirmeden izlenen dosyalarda yapılan yerel değişiklikleri yok sayabilirsiniz:

  • git update-index --skip-worktree [<file>...]: küçük yerel değişiklikler için
  • git update-index --assume-unchanged [<file>...]: üretime hazır, değişmeyen dosyalar yukarı akış için

Daha fazla seçenek için [sonraki işaretler] arasındaki farklar hakkında daha fazla ayrıntıya[6] ve [git güncelleme-index belgelerine][7] bakın.

Yok sayılan dosyaları temizleme #

Yok sayılan dosyaları temizlemek için git clean -X kullanabilirsiniz:

git clean -Xn #display a list of ignored files
git clean -Xf #remove the previously displayed files

Not: -X (büyük harfler) only yoksayılan dosyaları temizler. İzlenmeyen dosyaları da kaldırmak için -x (büyük harf yok) kullanın.

Daha fazla ayrıntı için git clean belgelerine bakın.

Bir dosyanın göz ardı edilip edilmediğini kontrol etme #

git check-ignore komutu Git tarafından yok sayılan dosyalar hakkında rapor verir.

Dosya adlarını komut satırından iletebilirsiniz ve “git check-ignore” yok sayılan dosya adlarını listeleyecektir. Örneğin:

$ cat .gitignore
*.o
$ git check-ignore example.o Readme.md
example.o

Burada yalnızca *.o dosyaları .gitignore’da tanımlanır, bu nedenle Readme.md git check-ignore çıktısında listelenmez.

Bir dosyayı yoksaymaktan .gitignore’un sorumlu olduğu satırı görmek istiyorsanız, git check-ignore komutuna -v ekleyin:

$ git check-ignore -v example.o Readme.md
.gitignore:1:*.o        example.o

Git 1.7.6’dan itibaren, yoksayılan dosyaları görmek için git status --ignored kullanabilirsiniz. Bununla ilgili daha fazla bilgiyi resmi belgelerde veya .gitignore tarafından yok sayılan dosyaları bulma içinde bulabilirsiniz.

.gitignore dosyasındaki istisnalar #

Dosyaları bir kalıp kullanarak yok sayarsanız ancak istisnalarınız varsa, istisnaya bir ünlem işareti(!) ekleyin. Örneğin:

*.txt
!important.txt

Yukarıdaki örnek Git’e “important.txt” adlı dosyalar dışında “.txt” uzantılı tüm dosyaları yok saymasını söyler.

Dosya yoksayılan bir klasördeyse, onu bu kadar kolay bir şekilde DEĞİL yeniden ekleyemezsiniz:

folder/
!folder/*.txt

Bu örnekte, klasördeki tüm .txt dosyaları yoksayılacaktır.

Doğru yol, klasörün kendisini ayrı bir satıra yeniden eklemek, ardından ‘klasör’ içindeki tüm dosyaları ‘’ ile yok saymak ve son olarak ‘.txt’ dosyasını ‘klasöre’ aşağıdaki gibi yeniden eklemektir:

!folder/
folder/*
!folder/*.txt

Not: Ünlem işaretiyle başlayan dosya adları için iki ünlem işareti ekleyin veya \ karakteriyle çıkın:

!!includethis
\!excludethis

Global bir .gitignore dosyası #

Git’in tüm depolarda belirli dosyaları yok saymasını sağlamak için, terminalinizde veya komut isteminde aşağıdaki komutu kullanarak bir global .gitignore oluşturabilirsiniz:

$ git config --global core.excludesfile <Path_To_Global_gitignore_file>

Git şimdi bunu her havuzun kendi .gitignore dosyasına ek olarak kullanacak. Bunun için kurallar:

  • Yerel ‘.gitignore’ dosyası açıkça bir dosya içeriyorsa ve global ‘.gitignore’ onu yoksayarsa, yerel ‘.gitignore’ önceliği alır (dosya dahil edilir)
  • Depo birden fazla makinede klonlanmışsa, global “.gigignore” tüm makinelere yüklenmeli veya en azından dahil edilmelidir, çünkü yok sayılan dosyalar global “.gitignore” ile PC depoya gönderilirken güncellemeyecekti. Bu nedenle, proje üzerinde bir ekip tarafından çalışılıyorsa, depoya özgü bir “.gitignore” küresel olandan daha iyi bir fikirdir.

Bu dosya, platform, makine veya kullanıcıya özel yok sayılanları tutmak için iyi bir yerdir, örn. Bunları depoda tutmak istemiyorsanız, OSX .DS_Store, Windows Thumbs.db veya Vim *.ext~ ve *.ext.swp yok sayar. Böylece, OS X üzerinde çalışan bir ekip üyesi tüm “.DS_STORE” ve “_MACOSX” (aslında işe yaramaz) ekleyebilirken, Windows’taki başka bir ekip üyesi tüm “thumbs.bd"yi yok sayabilir.

Git deposuna önceden taahhüt edilmiş dosyaları yoksay #

Git deponuza zaten bir dosya eklediyseniz ve şimdi bu dosyayı izlemeyi durdurmak istiyorsanız (böylece gelecekteki kayıtlarda mevcut olmayacak), onu dizinden kaldırabilirsiniz:

git rm --cached <file>

Bu, dosyayı depodan kaldıracak ve daha fazla değişikliğin Git tarafından izlenmesini önleyecektir. –cached seçeneği, dosyanın fiziksel olarak silinmemesini sağlar.

Dosyanın önceden eklenen içeriğinin Git geçmişi aracılığıyla görünmeye devam edeceğini unutmayın.

Dosyayı dizinden kaldırdıktan sonra başka biri havuzdan çekerse, kopyasının fiziksel olarak silineceğini unutmayın.

Git’in dosyanın çalışma dizini sürümünün güncel olduğunu iddia etmesini sağlayabilir ve bunun yerine dizin sürümünü okuyabilir (böylece içindeki değişiklikleri yok sayarak) “çalışma ağacını atla” biti ile:

git update-index --skip-worktree <file>

Yazma bu bitten etkilenmez, içerik güvenliği hala birinci önceliktir. Göz ardı edilen değerli değişikliklerinizi asla kaybetmeyeceksiniz; diğer yandan bu bit, saklama ile çelişir: bu biti kaldırmak için şunu kullanın:

git update-index --no-skip-worktree <file>

Bazen Git’e yalan söylemek ve dosyayı incelemeden dosyanın değişmediğini varsayması yanlışlıkla önerilir. İlk bakışta, dosyayı dizininden kaldırmadan, dosyada yapılacak başka değişiklikleri yok sayıyormuş gibi görünür:

git update-index --assume-unchanged <file>

Bu, git’i dosyada yapılan herhangi bir değişikliği yok saymaya zorlar (bu dosyada herhangi bir değişiklik yaparsanız veya onu saklarsanız, yok sayılan değişikliklerinizin kaybolacağını unutmayın)

Git’in bu dosyayla tekrar “ilgilenmesini” istiyorsanız, aşağıdaki komutu çalıştırın:

git update-index --no-assume-unchanged <file>

Yoksayma kurallarını taahhüt etmeden dosyaları yerel olarak yoksay #

“.gitignore”, dosyaları yerel olarak yok sayar, ancak depoya bağlı olması ve diğer katkıda bulunanlar ve kullanıcılarla paylaşılması amaçlanmıştır. Genel bir “.gitignore” ayarlayabilirsiniz, ancak daha sonra tüm depolarınız bu ayarları paylaşır.

Bir havuzdaki belirli dosyaları yerel olarak yoksaymak ve dosyayı herhangi bir havuzun parçası yapmak istemiyorsanız, havuzunuzun içindeki .git/info/exclude dosyasını düzenleyin.

Örneğin:

# these files are only ignored on this repo
# these rules are not shared with anyone
# as they are personal                                              
gtk_tests.py
gui/gtk/tests/*
localhost
pushReports.py
server/

Herhangi bir dizindeki bir dosyayı yoksayma #

herhangi bir dizindeki foo.txt dosyasını yoksaymak için adını yazmanız yeterlidir:

foo.txt # matches all files 'foo.txt' in any directory

Dosyayı yalnızca ağacın bir bölümünde yoksaymak istiyorsanız, belirli bir dizinin alt dizinlerini ** kalıbıyla belirtebilirsiniz:

bar/**/foo.txt # matches all files 'foo.txt' in 'bar' and all subdirectories

Veya bar/ dizininde bir .gitignore dosyası oluşturabilirsiniz. Önceki örneğe eşdeğer, şu içeriklerle “bar/.gitignore” dosyası oluşturmak olacaktır:

foo.txt # matches all files 'foo.txt' in any directory under bar/

Bir dosyada sonraki değişiklikleri yok sayma (kaldırmadan) #

Bazen Git’te tutulan bir dosyaya sahip olmak istersiniz, ancak sonraki değişiklikleri görmezden gelirsiniz.

Git’e “update-index” kullanarak bir dosya veya dizindeki değişiklikleri yok saymasını söyleyin:

git update-index --assume-unchanged my-file.txt

Yukarıdaki komut Git’e “dosyam.txt” dosyasının değiştirilmediğini varsaymasını ve değişiklikleri kontrol etmemesini veya bildirmemesini söyler. Dosya hala depoda mevcuttur.

Bu, varsayılanları sağlamak ve yerel ortam geçersiz kılmalarına izin vermek için yararlı olabilir, örneğin:

# create a file with some values in
cat <<EOF
MYSQL_USER=app
MYSQL_PASSWORD=FIXME_SECRET_PASSWORD
EOF > .env

# commit to Git
git add .env
git commit -m "Adding .env template"

# ignore future changes to .env
git update-index --assume-unchanged .env

# update your password
vi .env

# no changes!
git status

Alt klasörlerdeki dosyaları yok sayma (Birden çok gitignore dosyası) #

Bunun gibi bir depo yapınız olduğunu varsayalım:

examples/
    output.log
src/
    <files not shown>
    output.log
README.md

Örnekler dizinindeki output.log, hata ayıklama sırasında src/ altındaki `` oluşturulduğunda projenin bir anlayış toplaması için geçerlidir ve gereklidir ve arşivin geçmişinde veya bir parçası olmamalıdır.

Bu dosyayı yoksaymanın iki yolu vardır. Çalışma dizininin kökündeki “.gitignore” dosyasına mutlak bir yol yerleştirebilirsiniz:

# /.gitignore
src/output.log

Alternatif olarak, “src/” dizininde bir “.gitignore” dosyası oluşturabilir ve bu “.gitignore” ile ilgili dosyayı yok sayabilirsiniz:

# /src/.gitignore
output.log

Önceden Doldurulmuş .gitignore Şablonları #

“.gitignore” dosyanızda hangi kuralları listeleyeceğinizden emin değilseniz veya projenize genel kabul görmüş istisnalar eklemek istiyorsanız, bir “.gitignore” dosyası seçebilir veya oluşturabilirsiniz:

  • https://www.gitignore.io/
  • https://github.com/github/gitignore

GitHub ve BitBucket gibi birçok barındırma hizmeti, kullandığınız programlama dillerine ve IDE’lere dayalı olarak “.gitignore” dosyaları oluşturma olanağı sunar:

Boş Bir Klasör Oluşturun #

Git’in dosyaları yönetmesi ve dizinlerini bunlara eklemesi nedeniyle, Git’te boş bir klasör eklemek ve işlemek mümkün değildir, bu da taahhütleri azaltır ve hızı artırır. Bunu aşmak için iki yöntem vardır:

Birinci yöntem: “.gitkeep”

Bunu aşmanın bir yolu, Git klasörünü kaydetmek için bir .gitkeep dosyası kullanmaktır. Bunu yapmak için, sadece gerekli dizini oluşturun ve klasöre bir .gitkeep dosyası ekleyin. Bu dosya boştur ve sadece klasörü kaydetmekten başka bir amaca hizmet etmez. Bunu Windows’ta (garip dosya adlandırma kurallarına sahip olan) yapmak için dizinde git bash’i açın ve şu komutu çalıştırın:

$ .gitkeep’e dokunun

Bu komut, geçerli dizinde yalnızca boş bir .gitkeep dosyası oluşturur.

İkinci yöntem: “dummy.txt”

Bunun için başka bir hack, yukarıdakine çok benzer ve aynı adımlar izlenebilir, ancak bir “.gitkeep” yerine “dummy.txt” kullanın. Bu, içerik menüsünü kullanarak Windows’ta kolayca oluşturabilme avantajına sahiptir. Ve içinde komik mesajlar da bırakabilirsiniz. Boş dizini izlemek için .gitkeep dosyasını da kullanabilirsiniz. .gitkeep normalde boş dizini izlemek için eklenen boş bir dosyadır.

.gitignore tarafından yok sayılan dosyaları bulma #

Geçerli dizinde git tarafından yok sayılan tüm dosyaları şu komutla listeleyebilirsiniz:

git status --ignored

Öyleyse, bunun gibi bir depo yapımız varsa:

.git
.gitignore
./example_1
./dir/example_2
./example_2

… ve aşağıdakileri içeren .gitignore dosyası:

example_2

… komutun sonucu şöyle olacaktır:

$ git status --ignored

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

.gitignore

  (use "git add -f <file>..." to include in what will be committed)

Bir dosyanın yalnızca bir kısmını yok sayma [stub] #

Bazen, taahhüt etmek veya yayınlamak istemediğiniz bir dosyada yerel değişiklikler yapmak isteyebilirsiniz. İdeal olarak yerel ayarlar, “.gitignore” içine yerleştirilebilecek ayrı bir dosyada yoğunlaştırılmalıdır, ancak bazen kısa vadeli bir çözüm olarak, teslim edilen bir dosyada yerel bir şey olması yararlı olabilir.

Temiz filtre kullanarak Git’in bu satırları “görmemesini” sağlayabilirsiniz. Farklarda bile görünmeyecekler.

Burada “file1.c” dosyasından bir pasaj olduğunu varsayalım:

struct settings s;
s.host = "localhost";
s.port = 5653;
s.auth = 1;
s.port = 15653; // NOCOMMIT
s.debug = 1; // NOCOMMIT
s.auth = 0; // NOCOMMIT

‘NOCOMMIT’ satırlarını hiçbir yerde yayınlamak istemezsiniz.

Bunu Git yapılandırma dosyasına .git/config gibi ekleyerek “nocommit” filtresi oluşturun:

[filter "nocommit"]
    clean=grep -v NOCOMMIT

Bunu “.git/info/attributes” veya “.gitmodules” dosyasına ekleyin (veya oluşturun):

file1.c filter=nocommit

Ve NOCOMMIT satırlarınız Git’ten gizlenir.

Uyarılar:

  • Temiz filtre kullanmak, özellikle Windows’ta dosyaların işlenmesini yavaşlatır.
  • Git güncellediğinde yok sayılan satır dosyadan kaybolabilir. Bir leke filtresi ile önlenebilir, ancak daha zordur.
  • Windows’ta test edilmedi

İzlenen dosyalardaki değişiklikleri yok sayma. [Taslak] #

.gitignore ve .git/info/exclude yalnızca izlenmeyen dosyalar için çalışır.

İzlenen bir dosyada yoksay bayrağını ayarlamak için update-index komutunu kullanın:

git update-index --skip-worktree myfile.c

Bunu geri almak için şunu kullanın:

git update-index --no-skip-worktree myfile.c

Daha uygun “git hide”, “git unhide” ve “git gizli” komutlarına sahip olmak için bu parçacığı global git config dosyanıza ekleyebilirsiniz:

[alias]
    hide   = update-index --skip-worktree
    unhide = update-index --no-skip-worktree
    hidden  = "!git ls-files -v | grep ^[hsS] | cut -c 3-"

Ayrıca, güncelleme dizini işleviyle –varsayım-değiştirilmemiş seçeneğini de kullanabilirsiniz.

git update-index --assume-unchanged <file>

Değişiklikler için bu dosyayı tekrar izlemek isterseniz,

git update-index --no-assume-unchanged <file>

–assume-unchanged bayrağı belirtildiğinde, kullanıcı dosyayı değiştirmeyeceğine söz verir ve Git’in çalışan ağaç dosyasının dizinde kayıtlı olanla eşleştiğini varsaymasına izin verir. Bu dosyayı dizinde değiştirmesi gerektiğinde Git başarısız olur. Örneğin. bir taahhütte birleşirken; bu nedenle, varsayılan izlenmeyen dosyanın yukarı yönde değiştirilmesi durumunda, durumu manuel olarak ele almanız gerekecektir. Bu durumda odak performans üzerindedir.

–skip-worktree bayrağı, git’e, dosya yerel olarak değiştirileceği ve yanlışlıkla değişiklikleri yapmak istemediğiniz için belirli bir dosyaya dokunmama talimatı verdiğinizde yararlıdır (yani, belirli bir dosya için yapılandırılmış yapılandırma/özellikler dosyası). çevre). Her ikisi de ayarlandığında, çalışma ağacını atla, varsayılanın değişmediğini varsaymaya göre önceliklidir.

Önceden taahhüt edilmiş ancak .gitignore’a dahil edilmiş dosyaları temizleyin #

Bazen bir dosya git tarafından izlenir, ancak daha sonraki bir noktada, izlemeyi durdurmak için .gitignore’a eklenir. .gitignore’a eklenmeden önce bu tür dosyaları temizlemeyi unutmak çok yaygın bir senaryodur. Bu durumda, eski dosya depoda dolanmaya devam edecektir.

Bu sorunu çözmek için, depodaki her şeyin “kuru çalıştırma” kaldırılması ve ardından tüm dosyaların yeniden eklenmesi gerçekleştirilebilir. Bekleyen değişiklikleriniz olmadığı ve “–cached” parametresi iletildiği sürece, bu komutu çalıştırmak oldukça güvenlidir:

# Remove everything from the index (the files will stay in the file system) 
$ git rm -r --cached .

# Re-add everything (they'll be added in the current state, changes included)
$ git add .

# Commit, if anything changed. You should see only deletions
$ git commit -m 'Remove all files that are in the .gitignore'

# Update the remote
$ git push origin master
