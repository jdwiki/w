Bir önceki işleme geri dön #

Bir önceki işleme geri dönmek için, önce git log kullanarak işlemin karmasını bulun.

Bu taahhüde geçici olarak geri dönmek için kafanızı şu şekilde ayırın:

git checkout 789abcd

Bu, sizi ‘789abcd’ taahhüdüne yerleştirir. Artık bu eski taahhüdün üzerine, kafanızın bulunduğu dalı etkilemeden yeni taahhütler yapabilirsiniz. Herhangi bir değişiklik, [“dal” veya “checkout -b”][2] kullanılarak uygun bir dalda yapılabilir.

Değişiklikleri korurken bir önceki işleme geri dönmek için:

git reset --soft 789abcd

son taahhüdünü geri almak için:

git reset --soft HEAD~

Belirli bir taahhütten sonra yapılan değişiklikleri kalıcı olarak atmak için şunu kullanın:

git reset --hard 789abcd

son taahhütten sonra yapılan değişiklikleri kalıcı olarak silmek için:

git reset --hard HEAD~

Dikkat: [‘reflog’ ve ‘reset’ kullanarak atılan taahhütleri kurtarabilirsiniz][3], ancak taahhüt edilmemiş değişiklikler kurtarılamaz. [git stash; Güvende olmak için git reset –hard yerine git reset][4] kullanın.

[2]: https://www.wikiod.com/tr/git/dallanma#Yeni şubeler oluşturma ve kontrol etme [3]: https://www.wikiod.com/tr/git/kurtarma#Sıfırlamadan kurtarma [4]: https://www.wikiod.com/tr/git/saklamak

Değişiklikleri geri alma #

Çalışan kopyada bir dosya veya dizinde yapılan değişiklikleri geri alın.

git checkout -- file.txt

Geçerli dizinden özyinelemeli olarak tüm dosya yollarında kullanıldığında, çalışan kopyadaki tüm değişiklikleri geri alır.

git checkout -- .

Değişikliklerin yalnızca bazı kısımlarını geri almak için “–patch” kullanın. Her değişiklik için geri alınması gerekip gerekmediği sorulacaktır.

git checkout --patch -- dir

dizine eklenen değişiklikleri geri almak için.

git reset --hard

`–hard’ bayrağı olmadan bu, yazılımdan sıfırlama yapacaktır.

Henüz bir uzaktan kumandaya basmadığınız yerel taahhütlerle, yazılımdan sıfırlama da yapabilirsiniz. Böylece dosyaları ve ardından taahhütleri yeniden işleyebilirsiniz.

git reset HEAD~2

Yukarıdaki örnek, son iki taahhüdünüzü çözer ve dosyaları çalışan kopyanıza döndürür. Daha sonra daha fazla değişiklik ve yeni taahhütler yapabilirsiniz.

Dikkat: Yazılım sıfırlamaları dışındaki tüm bu işlemler, değişikliklerinizi kalıcı olarak siler. Daha güvenli bir seçenek için sırasıyla “git stash -p” veya “git stash” kullanın. Daha sonra ‘stash pop’ ile geri alabilir veya ‘stash drop’ ile sonsuza kadar silebilirsiniz.

Birleştirmeleri geri alma #

Henüz uzaktan kumandaya aktarılmayan birleştirme işleminin geri alınması

Henüz birleştirme işleminizi uzak depoya göndermediyseniz, bazı küçük farklılıklar olsa da taahhüdü geri al ile aynı prosedürü uygulayabilirsiniz.

Sıfırlama, hem birleştirme taahhüdünü hem de şubeden eklenen tüm taahhütleri geri alacağı için en basit seçenektir. Bununla birlikte, hangi SHA’nın sıfırlanacağını bilmeniz gerekecek, git logunuz artık her iki daldan da taahhütleri göstereceğinden, bu zor olabilir. Yanlış taahhüdü sıfırlarsanız (ör. diğer dalda bir tane) taahhüt edilen işi yok edebilir.

> git reset --hard <last commit from the branch you are on>

Veya birleştirmenin en son taahhüdünüz olduğunu varsayarsak.

> git reset HEAD~

Geri alma daha güvenlidir, çünkü taahhüt edilen işi yok etmez, ancak şubeyi tekrar birleştirmeden önce geri döndürmek zorunda olduğunuz için daha fazla iş gerektirir (bir sonraki bölüme bakın).

Uzaktan kumandaya aktarılan birleştirme işleminin geri alınması

Yeni bir özellikte birleştirdiğinizi varsayalım (eklenti gremlinleri)

> git merge feature/add-gremlins
...
   #Resolve any merge conflicts
> git commit #commit the merge
...
> git push
...
   501b75d..17a51fd  master -> master

Daha sonra, az önce birleştirdiğiniz özelliğin diğer geliştiriciler için sistemi bozduğunu keşfedersiniz, hemen geri alınması gerekir ve özelliğin düzeltilmesi çok uzun sürer, bu nedenle birleştirmeyi geri almak istersiniz.

> git revert -m 1 17a51fd
...
> git push
...
   17a51fd..e443799  master -> master

Bu noktada gremlinler sistemin dışındadır ve geliştirici arkadaşlarınız size bağırmayı bırakmıştır. Ancak, henüz bitirmedik. Eklenti gremlinleri özelliğiyle ilgili sorunu çözdükten sonra, tekrar birleşmeden önce bu geri dönüşü geri almanız gerekecektir.

> git checkout feature/add-gremlins
...
   #Various commits to fix the bug.
> git checkout master
...
> git revert e443799
...
> git merge feature/add-gremlins
...
   #Fix any merge conflicts introduced by the bug fix
> git commit #commit the merge
...
> git push

Bu noktada, özelliğiniz artık başarıyla eklendi. Ancak, bu tür hataların genellikle birleştirme çakışmaları tarafından ortaya çıkarıldığı göz önüne alındığında, biraz farklı bir iş akışı, şubenizdeki birleştirme çakışmasını düzeltmenize izin verdiği için bazen daha yararlıdır.

> git checkout feature/add-gremlins
...
   #Merge in master and revert the revert right away.  This puts your branch in
   #the same broken state that master was in before.
> git merge master
...
> git revert e443799
...
   #Now go ahead and fix the bug (various commits go here)
> git checkout master
...
   #Don't need to revert the revert at this point since it was done earlier
> git merge feature/add-gremlins
...
   #Fix any merge conflicts introduced by the bug fix
> git commit #commit the merge
...
> git push

reflog’u kullanma #

Bir yeniden temeli bozarsanız, yeniden başlamak için bir seçenek, işleme (rebase öncesi) geri dönmektir. Bunu “reflog"u (son 90 gün içinde yaptığınız her şeyin geçmişine sahiptir - bu yapılandırılabilir) kullanarak yapabilirsiniz:

$ git reflog
4a5cbb3 HEAD@{0}: rebase finished: returning to refs/heads/foo
4a5cbb3 HEAD@{1}: rebase: fixed such and such
904f7f0 HEAD@{2}: rebase: checkout upstream/master
3cbe20a HEAD@{3}: commit: fixed such and such
...

Taahhüdü, yeniden temel “HEAD@{3}” olmadan önce görebilirsiniz (karmayı da kontrol edebilirsiniz):

git checkout HEAD@{3}

Şimdi yeni bir dal oluştur / eskisini sil / yeniden temel almayı dene.

Ayrıca doğrudan ‘reflog’unuzdaki bir noktaya sıfırlayabilirsiniz, ancak bunu yalnızca yapmak istediğiniz şeyin bu olduğundan %100 eminseniz yapın:

git reset --hard HEAD@{3}

Bu, mevcut git ağacınızı o noktada nasıl olduğuna uyacak şekilde ayarlayacaktır (Bkz. Değişiklikleri Geri Alma).

Bu, bir dalın başka bir dalda yeniden temellendiğinde ne kadar iyi çalıştığını geçici olarak görüyorsanız, ancak sonuçları saklamak istemiyorsanız kullanılabilir.

Mevcut bazı taahhütleri geri alın #

Özellikle bu taahhütler uzak bir havuza gönderildiğinde, mevcut taahhütleri geri almak için git revert’i kullanın. Geçmişi yeniden yazmadan güvenle itebileceğiniz bazı önceki taahhütlerin etkisini tersine çevirmek için bazı yeni taahhütleri kaydeder.

**Bu havuzun diğer tüm kullanıcılarının hakaretini azaltmak istemiyorsanız ‘git push –force’ kullanmayın. Asla kamu tarihini yeniden yazmayın.

Örneğin, bir hata içeren bir taahhütte bulunduysanız ve onu geri çekmeniz gerekiyorsa, aşağıdakileri yapın:

git revert HEAD~1
git push

Artık yerel olarak geri alma taahhüdünü geri almakta, kodunuzu düzeltmekte ve iyi kodu göndermekte özgürsünüz:

git revert HEAD~1
work .. work .. work ..
git add -A .
git commit -m "Update error code"
git push

Geri almak istediğiniz taahhüt zaten geçmişteyse, taahhüt karmasını iletebilirsiniz. Git, orijinal taahhüdünüzü geri alan ve uzaktan kumandanıza güvenle gönderebileceğiniz bir karşı taahhüt oluşturacaktır.

git revert 912aaf0228338d0c8fb8cca0a064b0161a451fdc
git push
