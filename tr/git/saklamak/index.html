Sözdizimi #

  • git stash list [<options>]
  • git stash show [<stash>]
  • git stash drop [-q|--quiet] [<stash>]
  • git stash ( pop | uygula ) [--index] [-q|--quiet] [<stash>]
  • git stash dalı <branchname> [<stash>]
  • git stash [kaydet [-p|--patch] [-k|--[no-]keep-index] [-q|--quiet] [-u|--include-tracked] [-a| --all] [<mesaj>]]
  • git stash clear
  • git stash create [<message>]
  • git stash store [-m|--message <message>] [-q|--quiet] <commit>

Parametreler #

parametre Ayrıntılar
göster Saklamada kaydedilen değişiklikleri, saklanan durum ile orijinal ebeveyni arasındaki fark olarak gösterin. <stash> verilmediğinde, en sonuncuyu gösterir.
liste Şu anda sahip olduğunuz zulaları listeleyin. Her zula kendi adıyla (ör. zula@{0} en son zuladır, zula@{1} önceki zuladır, vb.), zula oluşturulduğunda geçerli olan şubenin adı ve kısa bir isimle listelenir. zulanın dayandığı taahhüdün açıklaması.
pop Zayıflama listesinden tek bir saklanmış durumu kaldırın ve mevcut çalışan ağaç durumunun üstüne uygulayın.
apply pop gibi, ancak durumu zula listesinden çıkarmayın.
temizle Tüm saklanan durumları kaldırın. Bu durumların daha sonra budamaya tabi olacağını ve kurtarmanın imkansız olabileceğini unutmayın.
damla Saklama listesinden tek bir saklanan durumu kaldırın. <stash> verilmediğinde, en sonuncuyu kaldırır. yani stash@{0}, aksi takdirde <stash>, stash@{<revizyon>} formunun geçerli bir saklama günlüğü referansı olmalıdır.
oluştur Bir stash (normal bir taahhüt nesnesidir) oluşturun ve nesne adını ref ad alanında herhangi bir yere kaydetmeden döndürün. Bunun komut dosyaları için yararlı olması amaçlanmıştır. Muhtemelen kullanmak istediğiniz komut bu değildir; yukarıdaki “kaydet” bölümüne bakın.
mağaza Stash reflog’u güncelleyerek, stash ref’de git stash create (sarkan bir birleştirme taahhüdü olan) aracılığıyla oluşturulan belirli bir zulayı saklayın. Bunun komut dosyaları için yararlı olması amaçlanmıştır. Muhtemelen kullanmak istediğiniz komut bu değildir; yukarıdaki “kaydet” bölümüne bakın.

Stashing, hiçbir bilgiyi kaybetmeden temiz bir çalışma dizinine sahip olmamızı sağlar. Ardından, farklı bir şey üzerinde çalışmaya başlamak ve/veya dalları değiştirmek mümkündür.

Depo oluştur #

Geçerli çalışma dizinini ve dizini (hazırlama alanı olarak da bilinir) bir zula yığınına kaydedin.

git stash

İzlenmeyen tüm dosyaları zulaya dahil etmek için “–include-untracked” veya “-u” işaretlerini kullanın.

git stash --include-untracked

Daha sonra daha kolay tanımlanabilir hale getirmek için zulanıza bir mesaj eklemek için

git stash save "<whatever message>"

Hazırlama alanını zulalamadan sonra mevcut durumda bırakmak için ‘–keep-index’ veya ‘-k’ bayraklarını kullanın.

git stash --keep-index

zulayı uygula ve kaldır #

Son zulayı uygulamak ve yığından kaldırmak için - şunu yazın:

git stash pop

Belirli bir zulayı uygulamak ve yığından kaldırmak için - şunu yazın:

git stash pop stash@{n}

Zulayı kaldırmadan uygula #

Son zulayı yığından çıkarmadan uygular

git stash apply

Veya belirli bir zula

git stash apply stash@{n}

Depoyu göster #

Son zulada kaydedilen değişiklikleri gösterir

git stash show

Veya belirli bir zula

git stash show stash@{n}

Belirli zula için kaydedilen değişikliklerin içeriğini göstermek için

git stash show -p stash@{n}

Saklama Nedir? #

Bir proje üzerinde çalışırken, master’a karşı bir hata oluştuğunda bir özellik dalı değişikliğinin yarısında olabilirsiniz. Kodunuzu taahhüt etmeye hazır değilsiniz, ancak değişikliklerinizi de kaybetmek istemiyorsunuz. “git stash"ın kullanışlı olduğu yer burasıdır.

Taahhüt edilmemiş değişikliklerinizi göstermek için bir dalda “git status” komutunu çalıştırın:

(master) $ git status
On branch master
Your branch is up-to-date with 'origin/master'.
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Ardından, bu değişiklikleri bir yığına kaydetmek için “git stash” komutunu çalıştırın:

(master) $ git stash
Saved working directory and index state WIP on master: 
2f2a6e1 Merge pull request #1 from test/test-branch
HEAD is now at 2f2a6e1 Merge pull request #1 from test/test-branch

Çalışma dizininize dosya eklediyseniz, bunlar da saklanabilir. Sadece önce onları sahnelemelisin.

(master) $ git stash
Saved working directory and index state WIP on master:
(master) $ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)

        NewPhoto.c

nothing added to commit but untracked files present (use "git add" to track)
(master) $ git stage NewPhoto.c
(master) $ git stash
Saved working directory and index state WIP on master:
(master) $ git status
On branch master
nothing to commit, working tree clean
(master) $

Çalışma dizininiz artık yaptığınız değişikliklerden temizlendi. Bunu “git status” komutunu yeniden çalıştırarak görebilirsiniz:

(master) $ git status
On branch master
Your branch is up-to-date with 'origin/master'.
nothing to commit, working directory clean

En son zulayı uygulamak için, git stash application komutunu çalıştırın (ayrıca, git stash pop ile değiştirilen son saklananları * uygulayabilir ve* kaldırabilirsiniz):

(master) $ git stash apply
On branch master
Your branch is up-to-date with 'origin/master'.
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Ancak, saklamanın üzerinde çalıştığınız dalı hatırlamadığını unutmayın. Yukarıdaki örneklerde, kullanıcı master üzerinde saklanıyordu. dev şubesine, dev’e geçerler ve git stash application komutunu çalıştırırlarsa, son zula dev şubesine konur.

(master) $ git checkout -b dev
Switched to a new branch 'dev'
(dev) $ git stash apply
On branch dev
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

    modified:   business/com/test/core/actions/Photo.c

no changes added to commit (use "git add" and/or "git commit -a")

Kısmi saklama #

Çalışma setinizde yalnızca bazı farkları saklamak istiyorsanız, kısmi bir saklama kullanabilirsiniz.

git stash -p

Ardından, hangi parçaları saklayacağınızı etkileşimli olarak seçin.

2.13.0 sürümünden itibaren ayrıca etkileşimli moddan kaçınabilir ve yeni push anahtar sözcüğünü kullanarak bir yol belirtimi ile kısmi bir saklama oluşturabilirsiniz.

git stash push -m "My partial stash" -- app.config

Kaydedilen depoları listele #

git stash list

Bu, yığındaki tüm zulaları ters kronolojik sırada listeleyecektir. Şuna benzeyen bir liste alacaksınız:

stash@{0}: WIP on master: 67a4e01 Merge tests into develop
stash@{1}: WIP on master: 70f0d95 Add user role to localStorage on user login

Belirli bir zula adıyla başvurabilirsiniz, örneğin stash@{1}.

Devam eden çalışmanızı başka bir şubeye taşıyın #

Çalışırken yanlış dalda olduğunuzu fark ederseniz ve henüz herhangi bir taahhüt oluşturmadıysanız, saklamayı kullanarak çalışmanızı doğru şubeye kolayca taşıyabilirsiniz:

git stash
git checkout correct-branch
git stash pop

Unutmayın, git stash pop son zulayı uygulayacak ve onu zula listesinden silecektir. Saklamayı listede tutmak ve yalnızca bazı şubelere başvurmak için şunları kullanabilirsiniz:

git stash apply

Depoyu kaldır #

Tüm zulayı kaldır

git stash clear

Son zulayı kaldırır

git stash drop

Veya belirli bir zula

git stash drop stash@{n}

Kasanın bir kısmını ödeme ile uygula #

Bir zula yaptınız ve o zuladaki dosyaların sadece bir kısmını kontrol etmek istiyorsunuz.

git checkout stash@{0} -- myfile.txt

Depodan önceki değişiklikleri kurtarma #

Etkileşimli Saklama #

Düşen bir zulayı kurtar #

Henüz açtıysanız ve terminal hala açıksa, ekranda git stash pop tarafından basılan hash değerine sahip olacaksınız:

$ git stash pop
[...]
Dropped refs/stash@{0} (2ca03e22256be97f9e40f08e6d6773c7d41dbfd1)

(git stash drop’un da aynı satırı ürettiğini unutmayın.)

Aksi takdirde, bunu kullanarak bulabilirsiniz:

git fsck --no-reflog | awk '/dangling commit/ {print $3}'

Bu, taahhüt grafiğinizin ipuçlarında, artık herhangi bir daldan veya etiketten referans alınmayan tüm taahhütleri gösterecektir - şimdiye kadar oluşturduğunuz her saklama taahhüdü de dahil olmak üzere kaybedilen her taahhüt, o grafikte bir yerde olacaktır.

İstediğiniz saklama taahhüdünü bulmanın en kolay yolu, muhtemelen bu listeyi “gitk” e iletmektir:

gitk --all $( git fsck --no-reflog | awk '/dangling commit/ {print $3}' )

Bu, erişilebilir olup olmadığına bakılmaksızın, size depodaki her bir taahhüdü gösteren bir depo tarayıcısı başlatacaktır.

Konsolda ayrı bir GUI uygulaması yerine güzel bir grafik tercih ediyorsanız, oradaki “gitk"yi “git log –graph –oneline –decorate” gibi bir şeyle değiştirebilirsiniz.

Stash taahhütlerini tespit etmek için bu formun taahhüt mesajlarını arayın:

        bir şubede WIP: Bazı eski taahhüt mesajlarını onaylayın

İstediğiniz taahhüdün karmasını öğrendikten sonra, bunu bir zula olarak uygulayabilirsiniz:

git stash uygula $stash_hash

Veya ilgilendiğiniz ulaşılamayan taahhütler için şubeler oluşturmak için gitk içindeki içerik menüsünü kullanabilirsiniz. Ondan sonra, tüm normal araçlarla onlarla istediğinizi yapabilirsiniz. İşiniz bittiğinde, o dalları tekrar havaya uçurun.
