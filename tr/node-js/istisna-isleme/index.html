Node.Js’de İstisna İşleme #

Node.js’de istisnaları/hataları ele almanın 3 temel yolu vardır:

  1. deneyin-catch bloğu
  2. Bir “geri arama"nın ilk argümanı olarak hata
  3. eventEmitter kullanarak bir hata olayı “yayınlayın”

try-catch, eşzamanlı kod yürütmesinden kaynaklanan istisnaları yakalamak için kullanılır. Arayan (veya arayanın arayanı, …) dene/yakala kullanmışsa, hatayı yakalayabilirler. Arayanların hiçbiri, programın çökmesinden daha fazla denemediyse. Bir async işleminde try-catch kullanılıyorsa ve async yönteminin geri çağrısından istisna atılmışsa, try-catch tarafından yakalanmayacaktır. Zaman uyumsuz işlem geri aramasından bir istisna yakalamak için promises kullanılması tercih edilir. Daha iyi anlamak için örnek

// ** Example - 1  **
function doSomeSynchronousOperation(req, res) {
    if(req.body.username === ''){
        throw new Error('User Name cannot be empty');
    }  
    return true;  
}

// calling the method above
try {
    // synchronous code   
    doSomeSynchronousOperation(req, res)    
catch(e) {
    //exception handled here   
    console.log(e.message);  
} 

// ** Example - 2 **
function doSomeAsynchronousOperation(req, res, cb) {
    // imitating async operation
    return setTimeout(function(){
        cb(null, []);
    },1000);
}
 
try {
    // asynchronous code   
    doSomeAsynchronousOperation(req, res, function(err, rs){
        throw new Error("async operation exception");
    })   
} catch(e) {
     // Exception will not get handled here
     console.log(e.message);  
}
// The exception is unhandled and hence will cause application to break

geri aramalar çoğunlukla Node.js’de kullanılır, çünkü geri arama bir olayı eşzamansız olarak sunar. Kullanıcı size bir işlev (geri arama) iletir ve zaman uyumsuz işlem tamamlandığında bir süre sonra onu çağırırsınız. Genel model, geri aramanın bir geri arama(err, sonuç) olarak çağrılmasıdır; burada, işlemin başarılı veya başarısız olmasına bağlı olarak, err ve sonuçtan yalnızca biri boş değildir.

function doSomeAsynchronousOperation(req, res, callback) {
   setTimeout(function(){
        return callback(new Error('User Name cannot be empty'));    
   }, 1000);  
   return true;
}

doSomeAsynchronousOperation(req, res, function(err, result) {
   if (err) {
       //exception handled here 
       console.log(err.message);
   }
   
   //do some stuff with valid data
});

emit Daha karmaşık durumlarda, bir geri arama kullanmak yerine, işlevin kendisi bir EventEmitter nesnesi döndürebilir ve arayanın vericideki hata olaylarını dinlemesi beklenir.

const EventEmitter = require('events');

function doSomeAsynchronousOperation(req, res) {
    let myEvent = new EventEmitter();

    // runs asynchronously
    setTimeout(function(){
        myEvent.emit('error', new Error('User Name cannot be empty'));
    }, 1000);

    return myEvent;
}

// Invoke the function
let event = doSomeAsynchronousOperation(req, res);

event.on('error', function(err) {
    console.log(err);
});

event.on('done', function(result) {
    console.log(result); // true
});

Elsiz İstisna Yönetimi #

Node.js tek bir işlem üzerinde çalıştığından, yakalanmayan istisnalar, uygulama geliştirirken dikkat edilmesi gereken bir konudur.

İstisnaları Sessizce İşleme #

Çoğu kişi node.js sunucularının hataları sessizce yutmasına izin verir.

  • İstisnayı sessizce işlemek

    process.on(‘uncaughtException’, function (err) { console.log(err); });

Bu kötü, işe yarayacak ama:

  • İstisnaya (Hata) neyin neden olduğunun çözümüne katkıda bulunmayacağından, kök neden bilinmez olarak kalacaktır.

  • Veritabanı bağlantısının ( havuz ) herhangi bir nedenle kapanması durumunda, bu sürekli hataların yayılmasına neden olur, yani sunucu çalışır ancak db’ye yeniden bağlanmaz.

Başlangıç ​​durumuna dönüş #

Bir “uncaughtException” durumunda, sunucuyu yeniden başlatmak ve çalışacağını bildiğimiz başlangıç ​​durumuna döndürmek iyidir. İstisna günlüğe kaydedilir, uygulama sonlandırılır, ancak sunucunun çalıştığından emin olacak bir kapsayıcıda çalışacağı için sunucunun yeniden başlatılmasını sağlayacağız (ilk çalışma durumuna geri döner).

  • Sonsuza kadar (veya düğüm sunucusunun sürekli çalıştığından emin olmak için başka bir CLI aracı) yükleme

    npm install forever -g

  • Sunucuyu sonsuza kadar başlatmak

    forever start app.js

Neden başlatıldığını ve sonsuza kadar kullanmamızın nedeni sunucu kapandıktan sonradır. sonlandırılmış sonsuza kadar işlem sunucuyu yeniden başlatacaktır.

  • Sunucuyu yeniden başlatma

    process.on(‘uncaughtException’, function (err) { console.log(err);

      // some logging mechanisam
  // ....        

  process.exit(1); // terminates process

    });

Bir yan notta, Kümeler ve Etki Alanları ile ilgili istisnaları ele almanın da bir yolu vardı.

Alan adları kullanımdan kaldırıldı burada daha fazla bilgi.

Hatalar ve Sözler #

Sözler, hataları eşzamanlı veya geri arama odaklı koddan farklı şekilde işler.

const p = new Promise(function (resolve, reject) {
    reject(new Error('Oops'));
});

// anything that is `reject`ed inside a promise will be available through catch
// while a promise is rejected, `.then` will not be called
p
    .then(() => {
        console.log("won't be called");
    })
    .catch(e => {
        console.log(e.message); // output: Oops
    })
    // once the error is caught, execution flow resumes
    .then(() => {
        console.log('hello!'); // output: hello!
    });

şu anda, yakalanmayan bir sözde atılan hatalar, yutulmasına neden oluyor ve bu da hatayı izlemeyi zorlaştırabiliyor. Bu, eslint gibi linting araçları kullanılarak veya her zaman ` catch’ deyimi.

Bu davranış, düğüm 8’de düğüm işleminin sonlandırılması lehine kullanımdan kaldırılmıştır.
