Eşyordam Gücü #

kullanım #

İş parçacığı açısından güvenli olmayan Unity API’sine dayanan uzun süredir devam eden bir işleminiz varsa, bunu birden çok çerçeveye bölmek ve uygulamanızın yanıt vermesini sağlamak için Coroutines kullanın.

Coroutines ayrıca her karede bir eylemi çalıştırmak yerine her n’inci karede pahalı eylemler gerçekleştirmeye yardımcı olur.

Uzun Süreli Rutinleri Birden Çok Çerçeve Üzerinde Bölme ================================================= == Eşyordamlar, uygulamanızın kare hızını korumaya yardımcı olmak için uzun süreli işlemleri birden çok kareye dağıtmaya yardımcı olur.

Prosedürel olarak araziyi boyayan veya oluşturan veya gürültü oluşturan rutinler, Coroutine işlemine ihtiyaç duyabilecek örneklerdir.

for (int y = 0; y < heightmap.Height; y++) 
{
    for (int x = 0; x < heightmap.Width; x++)
    {
        // Generate pixel at (x, y)
        // Assign pixel at (x, y)
        
        // Process only 32768 pixels each frame
        if ((y * heightmap.Height + x) % 32 * 1024) == 0)
            yield return null; // Wait for next frame
    }
}

Yukarıdaki kod anlaşılması kolay bir örnektir. Üretim kodunda, ne zaman “geri dönüşü” (belki 2-3 satırda bir) kontrol eden piksel başına denetimden kaçınmak ve “for” döngü uzunluğunu önceden hesaplamak daha iyidir.

Pahalı İşlemleri Daha Az Sıklıkta Gerçekleştirme #

Eşyordamlar, pahalı eylemleri daha az sıklıkla gerçekleştirmenize yardımcı olur, böylece her karede gerçekleştirildiğinde olacağı kadar büyük bir performans artışı olmaz.

Aşağıdaki örneği doğrudan Manual‘den alarak:

private void ProximityCheck() 
{
    for (int i = 0; i < enemies.Length; i++) 
    {
        if (Vector3.Distance(transform.position, enemies[i].transform.position) < dangerDistance)
                return true;
    }
    return false;
}

private IEnumerator ProximityCheckCoroutine() 
{
    while(true) 
    {
        ProximityCheck();
        yield return new WaitForSeconds(.1f);
    }
}

Yakınlık testleri, CullingGroup API kullanılarak daha da optimize edilebilir.

Ortak tuzaklar #

Geliştiricilerin yaptığı yaygın bir hata, eşyordamın dışındaki eşyordamların sonuçlarına veya yan etkilerine erişmektir. Coroutines, bir “verim dönüşü” ifadesi ile karşılaşılır karşılaşmaz kontrolü arayana döndürür ve sonuç veya yan etki henüz gerçekleştirilmemiş olabilir. Sonucu/yan etkiyi eşyordam dışında kullanmak zorunlu olduğunuz sorunları aşmak için bu yanıtı kontrol edin.

Teller #

Unity’de alçakgönüllü diziden daha fazla kaynak domuzu olduğu iddia edilebilir, ancak bu, erkenden düzeltilmesi daha kolay yönlerden biridir.

Dize işlemleri çöp oluşturur #

Çoğu dize işlemi çok küçük miktarlarda çöp oluşturur, ancak bu işlemler tek bir güncelleme boyunca birkaç kez çağrılırsa, yığınlar toplanır. Zamanla, görünür bir CPU artışına neden olabilecek otomatik Çöp Toplama işlemini tetikleyecektir.

Dize işlemlerinizi önbelleğe alın #

Aşağıdaki örneği düşünün.

string[] StringKeys = new string[] {
    "Key0",
    "Key1",
    "Key2"
};

void Update()
{
    for (var i = 0; i < 3; i++)
    {
        // Cached, no garbage generated
        Debug.Log(StringKeys[i]);
    }

    for (var i = 0; i < 3; i++)
    {
        // Not cached, garbage every cycle
        Debug.Log("Key" + i);
    }

    // The most memory-efficient way is to not create a cache at all and use literals or constants.
    // However, it is not necessarily the most readable or beautiful way.
    Debug.Log("Key0");
    Debug.Log("Key1");
    Debug.Log("Key2");
}

Aptalca ve gereksiz görünebilir, ancak Shaders ile çalışıyorsanız, bu gibi durumlarla karşılaşabilirsiniz. Anahtarları önbelleğe almak bir fark yaratacaktır.

Lütfen değişmez değerler ve sabitler dizelerinin, program yığın alanına statik olarak enjekte edildikleri için herhangi bir çöp oluşturmadığını unutmayın. Çalışma zamanında * dizeler oluşturuyorsanız ve yukarıdaki örnekte olduğu gibi her seferinde aynı dizeleri oluşturmanız * garantili*yse, önbelleğe alma kesinlikle yardımcı olacaktır.

Oluşturulan dizenin her seferinde aynı olmadığı diğer durumlarda, bu dizeleri oluşturmanın başka bir alternatifi yoktur. Bu nedenle, bir seferde on binlerce dize oluşturulmadığı sürece, her seferinde manuel olarak dize oluşturan bellek artışı genellikle ihmal edilebilir.

Çoğu dize işlemi Hata Ayıklama mesajlarıdır #

Debug mesajları için string işlemleri yapmak, örn. Debug.Log("Nesne Adı: " + obj.name) gayet iyi ve geliştirme sırasında bundan kaçınılamaz. Bununla birlikte, alakasız hata ayıklama mesajlarının piyasaya sürülen üründe sonlanmamasını sağlamak önemlidir.

Bunun bir yolu, hata ayıklama çağrılarınızda Koşullu özniteliği kullanmaktır. Bu, yalnızca yöntem çağrılarını değil, aynı zamanda içine giren tüm dize işlemlerini de kaldırır.

using UnityEngine;
using System.Collections;

public class ConditionalDebugExample: MonoBehaviour
{
    IEnumerator Start()
    {
        while(true)
        {
            // This message will pop up in Editor but not in builds
            Log("Elapsed: " + Time.timeSinceLevelLoad);
            yield return new WaitForSeconds(1f);
        }
    }

    [System.Diagnostics.Conditional("UNITY_EDITOR")]
    void Log(string Message)
    {
        Debug.Log(Message);
    }
}

Bu basitleştirilmiş bir örnektir. Daha tam teşekküllü bir günlük kaydı rutini tasarlamak için biraz zaman harcamak isteyebilirsiniz.

Dize karşılaştırması #

Bu küçük bir optimizasyon ama bahsetmeye değer. Dizeleri karşılaştırmak, düşündüğünüzden biraz daha karmaşıktır. Sistem varsayılan olarak kültürel farklılıkları dikkate almaya çalışacaktır. Bunun yerine daha hızlı performans gösteren basit bir ikili karşılaştırma kullanmayı tercih edebilirsiniz.

// Faster string comparison
if (strA.Equals(strB, System.StringComparison.Ordinal)) {...}
// Compared to
if (strA == strB) {...}

// Less overhead
if (!string.IsNullOrEmpty(strA)) {...}
// Compared to
if (strA == "") {...}

// Faster lookups
Dictionary<string, int> myDic = new Dictionary<string, int>(System.StringComparer.Ordinal);
// Compared to
Dictionary<string, int> myDictionary = new Dictionary<string, int>();

Hızlı ve Etkili Kontroller #

Özellikle ‘Güncelleme’ gibi saniyede birçok kez çağrılan bir yöntemde, mümkün olan her yerde gereksiz işlemlerden ve yöntem çağrılarından kaçının.

Mesafe/Mesafe Kontrolleri #

Mesafeleri karşılaştırırken “büyüklük” yerine “sqrMagnitude” kullanın. Bu, gereksiz “sqrt” işlemlerini önler. sqrMagnitude kullanılırken sağ tarafın da karesinin alınması gerektiğini unutmayın.

if ((target.position - transform.position).sqrMagnitude < minDistance * minDistance))

Sınır Kontrolleri #

Nesne kesişimleri, “Çarpıştırıcı”/“Renderer” sınırlarının kesişip kesişmediği kontrol edilerek kabaca kontrol edilebilir. ‘Sınır’ yapısı ayrıca, iki sınırın kesişip kesişmediğini belirlemeye yardımcı olan kullanışlı bir ‘Kesişmeler’ yöntemine sahiptir.

“Sınırlar” ayrıca nesneler arasındaki gerçek (yüzeyden yüzeye) mesafenin yakın bir yaklaşık değerini hesaplamamıza yardımcı olur (bkz. Sınırlar.SqrDistance).

uyarılar #

Sınır denetimi, dışbükey nesneler için gerçekten iyi çalışır, ancak içbükey nesneler üzerindeki sınır denetimleri, nesnenin şekline bağlı olarak çok daha yüksek yanlışlıklara neden olabilir.

Yerel alan sınırlarını döndürdüğü için “Mesh.bounds” kullanılması önerilmez. Bunun yerine MeshRenderer.bounds kullanın.

Boş birlik yöntemlerinden kaçının #

Boş birlik yöntemlerinden kaçının. Kötü programlama stilinin yanı sıra, çalışma zamanı komut dosyası oluşturma ile ilgili çok küçük bir ek yük vardır. Birçok durumda bu, performansı artırabilir ve etkileyebilir.

void Update
{
}

void FixedUpdate
{
}

Önbellek referansları #

Özellikle güncelleme işlevinde pahalı çağrıları önlemek için referansları önbelleğe alın. Bu, eğer varsa veya varsa başlangıçta bu referansları önbelleğe alarak ve referansı tekrar almaktan kaçınmak için boş/bool düz olup olmadığını kontrol ederek yapılabilir.

Örnekler:

Önbellek bileşeni referansları

değiştirmek

void Update()
{
    var renderer = GetComponent<Renderer>();
    renderer.material.SetColor("_Color", Color.green);
}

ile

private Renderer myRenderer;
void Start()
{
    myRenderer = GetComponent<Renderer>();
}

void Update()
{
    myRenderer.material.SetColor("_Color", Color.green);
}

Önbellek nesnesi referansları

değiştirmek

void Update()
{
    var enemy = GameObject.Find("enemy");
    enemy.transform.LookAt(new Vector3(0,0,0));
}

ile

private Transform enemy;

void Start()
{
    this.enemy = GameObject.Find("enemy").transform;
}

void Update()
{
    enemy.LookAt(new Vector3(0, 0, 0));
}

Ek olarak, mümkün olduğunda Mathf’a yapılan aramalar gibi pahalı aramaları önbelleğe alın.

Dizeleri kullanarak yöntemleri çağırmaktan kaçının #

Yöntemleri kabul edebilen dizeleri kullanarak yöntemleri çağırmaktan kaçının. Bu yaklaşım, özellikle güncelleme işlevinde kullanıldığında oyununuzu yavaşlatabilecek yansımadan yararlanacaktır.

Örnekler:

    //Avoid StartCoroutine with method name
    this.StartCoroutine("SampleCoroutine");

    //Instead use the method directly
    this.StartCoroutine(this.SampleCoroutine());

    //Avoid send message
    var enemy = GameObject.Find("enemy");
    enemy.SendMessage("Die");

    //Instead make direct call
    var enemy = GameObject.Find("enemy") as Enemy;
    enemy.Die();
