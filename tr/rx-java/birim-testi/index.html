Tüm Zamanlayıcı yöntemleri statik olduğundan, RxJava kancalarını kullanan birim testleri aynı JVM örneğinde paralel olarak çalıştırılamaz. Eğer buradalarsa, bir birim testinin ortasında bir TestScheduler kaldırılacaktır. Bu temelde Schedulers sınıfını kullanmanın dezavantajıdır.

Test Zamanlayıcı #

TestSchedulers, yoğun beklemeler yapmak, iş parçacıklarına katılmak veya sistem zamanını değiştirmek için herhangi bir şey yapmak yerine, Gözlemlenebilirlerin süresini ve yürütülmesini kontrol etmenize olanak tanır. Öngörülebilir, tutarlı ve hızlı birim testleri yazmak istiyorsanız bu ÇOK önemlidir. Zamanı manipüle ettiğiniz için, artık bir iş parçacığının aç kalması, testinizin daha yavaş bir makinede başarısız olması veya yürütme zamanını bir sonucu beklemekle harcamak gibi bir şansınız yok.

TestSchedulers, herhangi bir RxJava işlemi için bir Zamanlayıcı alan aşırı yükleme yoluyla sağlanabilir.

TestScheduler testScheduler = new TestScheduler();
TestSubscriber<Integer> subscriber = TestSubscriber.create();
Observable.just(1,2,3)
          .delay(10, TimeUnit.SECONDS, testScheduler)
          .subscribe(subscriber);

try {
    Thread.sleep(TimeUnit.SECONDS.toMillis(11));
} catch (InterruptedException ignored) { }
subscriber.assertValues(1,2,3); // fails

testScheduler.advanceTimeBy(10, TimeUnit.SECONDS);
subscriber.assertValues(1,2,3); // success

TestScheduler oldukça basit. Sadece üç yöntemden oluşur.

testScheduler.advanceTimeBy(amount, timeUnit);
testScheduler.advanceTimeTo(when, timeUnit);
testScheduler.triggerActions();

Bu, TestScheduler’ın gelecekte bir zamana ait tüm eylemleri ne zaman başlatması gerektiğini değiştirmenizi sağlar.

Zamanlayıcı çalışırken, TestScheduler ne kadar etkisiz olduğu için yaygın olarak kullanılmaz. Zamanlayıcıları sınıflara geçirmek, az kazanç için çok fazla ekstra kod sağlar. Bunun yerine, RxJava’nın Schedulers.io()/computation()/etc dosyasına bağlanabilirsiniz. Bu, RxJava’nın Kancaları ile yapılır. Bu, Zamanlayıcı yöntemlerinden birinden gelen bir çağrıdan ne döndürüleceğini tanımlamanıza olanak tanır.

public final class TestSchedulers {

    public static TestScheduler test() {
        final TestScheduler testScheduler = new TestScheduler();
        RxJavaHooks.reset();
        RxJavaHooks.setOnComputationScheduler((scheduler) -> {
            return testScheduler;
        });
        RxJavaHooks.setOnIOScheduler((scheduler) -> {
            return testScheduler;
        });
        RxJavaHooks.setOnNewThreadScheduler((scheduler) -> {
            return testScheduler;
        });
        return testScheduler;
    }
}

Bu sınıf, kullanıcının, Zamanlayıcılara yapılan tüm çağrılar için bağlanacak olan test zamanlayıcısını almasına olanak tanır. Bir birim testinin bu zamanlayıcıyı kurulumuna alması yeterlidir. TestScheduler’ınız, zamanı ilerlettiğinizde başka bir birim testinden Eylemleri tetiklemeye çalışabileceğinden, herhangi bir düz eski alan olarak değil, kurulumda edinmenizi şiddetle tavsiye ederiz. Şimdi yukarıdaki örneğimiz olur

TestScheduler testScheduler = new TestScheduler();
TestSubscriber<Integer> subscriber = TestSubscriber.create();
Observable.just(1,2,3)
          .delay(10, TimeUnit.SECONDS, testScheduler)
          .subscribe(subscriber);
testScheduler.advanceTimeBy(9, TimeUnit.SECONDS);
subscriber.assertValues(); // success (delay hasn't finished)
testScheduler.advanceTimeBy(10, TimeUnit.SECONDS);
subscriber.assertValues(1,2,3); // success (delay has finished)

Sistem saatini birim testinizden bu şekilde etkili bir şekilde kaldırabilirsiniz (en azından RxJava ile ilgili olarak 😆)

TestAbonesi #

TestSubscribers, kendi Abonenizi oluşturma işinden kaçınmanıza veya belirli değerlerin teslim edildiği yerde, kaç tane olduğunu, Gözlenebilir tamamlandıysa, bir istisna oluşturulduğunu ve çok daha fazlasını doğrulamak için Eylem‘e abone olmanızı sağlar.

Başlarken #

Bu örnek sadece 1,2,3 ve 4 değerlerinin onNext aracılığıyla Gözlemlenebilir’e geçtiği iddiasını göstermektedir.

TestSubscriber<Integer> ts = TestSubscriber.create();
Observable.just(1,2,3,4).subscribe(ts);
ts.assertValues(1,2,3,4); // Success

“assertValues”, sayımın doğru olduğunu iddia eder. Yalnızca bazı değerleri iletirseniz, onaylama başarısız olur.

TestSubscriber<Integer> ts = TestSubscriber.create();
Observable.just(1,2,3,4).subscribe(ts);
ts.assertValues(1,2,3); // Fail

“assertValues”, onaylama yaparken “eşittir” yöntemini kullanır. Bu, veri olarak kabul edilen sınıfları kolayca test etmenizi sağlar.

TestSubscriber<Object> ts = TestSubscriber.create();
Observable.just(new Object(), new Object()).subscribe(ts);
ts.assertValues(new Object(), new Object()); // Fail

Bu örnek, bir eşittir tanımlanmış ve Observable’dan değerleri öne süren bir sınıfı gösterir.

public class Room {

    public String floor;
    public String number;

    @Override
    public boolean equals(Object o) {
        if (o == this) {
            return true;
        }
        if (o instanceof Room) {
            Room that = (Room) o;
            return (this.floor.equals(that.floor))
                    && (this.number.equals(that.number));
        }
        return false;
    }
}

TestSubscriber<Room> ts = TestSubscriber.create();
Observable.just(new Room("1", "10")).subscribe(ts);
ts.assertValue(new Room("1", "10"); // Success

Ayrıca, yalnızca bir öğeyi kontrol etmemiz gerektiğinden daha kısa olan ‘assertValue’ kullandığımızı da unutmayın.

Tüm olayları alma #

Gerekirse tüm etkinlikleri bir liste halinde de sorabilirsiniz.

TestSubscriber<Integer> ts = TestSubscriber.create();
Observable.just(1,2,3,4).subscribe(ts);
List<Integer> onNextEvents = ts.getOnNextEvents();
List<Throwable> onErrorEvents = ts.getOnErrorEvents();
List<Notification<Integer>> onCompletedEvents = ts.getOnCompletedEvents();

Olaylar üzerinde iddiada bulunmak #

Olaylarınız üzerinde daha kapsamlı testler yapmak istiyorsanız, getOnNextEvents (veya getOn*Events) ile favori onaylama kitaplığınızı birleştirebilirsiniz:

Observable<Integer> obs = Observable.just(1,2,3,4)
    .filter( x -> x % 2 == 0);

// note that we instanciate TestSubscriber via the constructor here 
TestSubscriber<Integer> ts = new TestSubscriber();
obs.subscribe(ts);

// Note that we are not using Observable#forEach here
// but java.lang.Iterable#forEach.
// You should never use Observable#forEach unless you know
// exactly what you're doing
ts.getOnNextEvents()
    .forEach( integer -> assertTrue(integer % 2 == 0));

‘Gözlenebilir#hata’ test ediliyor #

Doğru istisna sınıfının yayınlandığından emin olabilirsiniz:

Observable<Integer> obs = Observable.error(new Exception("I am a Teapot"));

TestSubscriber<Integer> ts = new TestSubscriber<>();
obs.subscribe(ts);

ts.assertError(Exception.class);

Ayrıca, tam İstisna’nın atıldığından emin olabilirsiniz:

Exception e = new Exception("I am a Teapot");
Observable<Integer> obs = Observable.error(e);

TestSubscriber<Integer> ts = new TestSubscriber<>();
obs.subscribe(ts);

ts.assertError(e);
