C ön işlemcisi, kodun gerçek derlemesinden önce çalıştırılan basit bir metin ayrıştırıcı/değiştiricidir. C (ve daha sonra C++) dilinin kullanımını genişletmek ve kolaylaştırmak için kullanılır, aşağıdakiler için kullanılabilir:

a. “#include” kullanılarak diğer dosyalar dahil

b. “#define” kullanarak metin değiştirme makrosu tanımlayın

c. Koşullu Derleme#if``#ifdef kullanarak

d. Platforma/Derleyiciye özel mantık (koşullu derlemenin bir uzantısı olarak)

Kaynak dosyalarınız derleyiciye teslim edilmeden önce önişlemci ifadeleri yürütülür. Çok düşük seviyeli koşullu mantık yeteneğine sahiptirler. Önişlemci yapıları (örneğin nesne benzeri makrolar) normal işlevler gibi yazılmadığından (ön işleme adımı derlemeden önce gerçekleşir), derleyici tür denetimlerini uygulayamaz, bu nedenle dikkatli kullanılmaları gerekir.

Muhafızları Dahil Et #

Bir başlık dosyası, diğer başlık dosyaları tarafından dahil edilebilir. Birden çok başlık içeren bir kaynak dosya (derleme birimi) bu nedenle dolaylı olarak bazı başlıkları birden fazla içerebilir. Birden fazla kez dahil edilen böyle bir başlık dosyası tanımlar içeriyorsa, derleyici (ön işlemeden sonra) Tek Tanım Kuralı’nın (ör.

Bazen başlık korumaları veya makro korumaları olarak da bilinen “muhafazaları dahil et” kullanılarak çoklu dahil etme engellenir. Bunlar, önişlemci #define, #ifndef, #endif yönergeleri kullanılarak uygulanır.

Örneğin.

// Foo.h
#ifndef FOO_H_INCLUDED 
#define FOO_H_INCLUDED

class Foo    //  a class definition
{
};

#endif

Dahil korumaları kullanmanın en önemli avantajı, standart uyumlu tüm derleyiciler ve ön işlemcilerle çalışacak olmalarıdır.

Ancak, dahil etme korumaları, bir projede kullanılan tüm başlıklarda makroların benzersiz olmasını sağlamak gerektiğinden geliştiriciler için bazı sorunlara da neden olur. Spesifik olarak, eğer iki (veya daha fazla) başlık, içerme koruması olarak “FOO_H_INCLUDED” kullanıyorsa, bir derleme birimine dahil edilen bu başlıklardan ilki, diğerlerinin dahil edilmesini etkili bir şekilde önleyecektir. Bir proje, ortak korumaları içeren başlık dosyalarına sahip bir dizi üçüncü taraf kitaplığı kullanıyorsa, belirli zorluklar ortaya çıkar.

Ayrıca, içerme korumalarında kullanılan makroların, başlık dosyalarında tanımlanan diğer makrolarla çakışmamasını sağlamak da gereklidir.

Çoğu C++ uygulaması, dosyanın tek bir derlemeye yalnızca bir kez dahil edilmesini sağlayan #pragma once yönergesini de destekler. Bu bir fiili standart yönergesidir, ancak herhangi bir ISO C++ standardının parçası değildir. Örneğin:

// Foo.h
#pragma once

class Foo
{
};

“#pragma once”, koruma korumalarıyla ilgili bazı sorunları önlerken, “#pragma” - standartlardaki tanımı gereği - doğası gereği derleyiciye özel bir kancadır ve onu desteklemeyen derleyiciler tarafından sessizce göz ardı edilir. #pragma once kullanan projelerin, onu desteklemeyen derleyicilere taşınması daha zordur.

C++ için bir dizi kodlama yönergesi ve güvence standardı, önişlemcinin “#include” başlık dosyaları veya başlıklara içerme korumaları yerleştirme amaçları dışında herhangi bir şekilde kullanılmasını özellikle caydırır.

Koşullu mantık ve platformlar arası işleme #

Özetle, koşullu ön işleme mantığı, makro tanımlarını kullanarak kod mantığını derleme için kullanılabilir veya kullanılamaz hale getirmekle ilgilidir.

Öne çıkan üç kullanım durumu şunlardır:

  • aynı uygulamaya aday olabilecek farklı uygulama profilleri (ör. hata ayıklama, yayın, test, optimize edilmiş) (ör. fazladan günlük kaydı ile).
  • platformlar arası derlemeler - tek kod tabanlı, çoklu derleme platformları.
  • birden fazla uygulama sürümü için ortak bir kod tabanı kullanma (örneğin, bir yazılımın Temel, Premium ve Pro sürümleri) - biraz farklı özelliklerle.

Örnek a: Dosyaları kaldırmak için platformlar arası bir yaklaşım (açıklayıcı):

#ifdef _WIN32
#include <windows.h> // and other windows system files
#endif
#include <cstdio>

bool remove_file(const std::string &path) 
{
#ifdef _WIN32
  return DeleteFile(path.c_str());
#elif defined(_POSIX_VERSION) || defined(__unix__)
  return (0 == remove(path.c_str()));
#elif defined(__APPLE__)
  //TODO: check if NSAPI has a more specific function with permission dialog
  return (0 == remove(path.c_str()));
#else 
#error "This platform is not supported"
#endif
}

‘_WIN32’, ‘APPLE’ veya ‘unix’ gibi makrolar normalde karşılık gelen uygulamalar tarafından önceden tanımlanır.

Bir makro tanımlanmamışsa ve değeri karşılaştırılır veya kontrol edilirse, önişlemci neredeyse her zaman sessizce değerin ‘0’ olduğunu varsayar. Bununla çalışmanın birkaç yolu var. Bir yaklaşım, varsayılan ayarların 0 olarak temsil edildiğini ve herhangi bir değişikliğin (ör. uygulama derleme profilinde) açıkça yapılması gerektiğini varsaymaktır (ör. #    error "ENABLE_EXTRA_DEBUGGING is not defined"
#else
#    if ( 1 == ENABLE_EXTRA_DEBUGGING )
  //code
#    endif
#endif

Makrolar #

Makrolar iki ana gruba ayrılır: nesne benzeri makrolar ve işlev benzeri makrolar. Makrolar, derleme işleminin başlarında bir belirteç ikamesi olarak değerlendirilir. Bu, büyük (veya yinelenen) kod bölümlerinin bir önişlemci makrosuna soyutlanabileceği anlamına gelir.

// This is an object-like macro
#define    PI         3.14159265358979

// This is a function-like macro.
// Note that we can use previously defined macros
// in other macro definitions (object-like or function-like)
// But watch out, its quite useful if you know what you're doing, but the
// Compiler doesnt know which type to handle, so using inline functions instead
// is quite recommended (But e.g. for Minimum/Maximum functions it is quite useful)
#define    AREA(r)    (PI*(r)*(r))

// They can be used like this:
double pi_macro   = PI;
double area_macro = AREA(4.6);

Qt kütüphanesi, kullanıcının QObject’i genişleten kullanıcı tanımlı sınıfın başında Q_OBJECT makrosunu bildirmesini sağlayarak bir meta-nesne sistemi oluşturmak için bu tekniği kullanır.

Makro adları, normal koddan ayırt edilmelerini kolaylaştırmak için genellikle tamamı büyük harflerle yazılır. Bu bir gereklilik değildir, ancak birçok programcı tarafından yalnızca iyi bir stil olarak kabul edilir.

Nesne benzeri bir makroyla karşılaşıldığında, makronun adı, tanımıyla değiştirilerek basit bir kopyala-yapıştır işlemi olarak genişletilir. Fonksiyon benzeri bir makro ile karşılaşıldığında hem adı hem de parametreleri genişletilir.

double pi_squared = PI * PI;
// Compiler sees:
double pi_squared = 3.14159265358979 * 3.14159265358979;

double area = AREA(5);
// Compiler sees:
double area = (3.14159265358979*(5)*(5))

Bu nedenle, işlev benzeri makro parametreleri, yukarıdaki ‘AREA()’ örneğinde olduğu gibi genellikle parantez içine alınır. Bu, makro genişletme sırasında oluşabilecek hataları, özellikle de tek bir makro parametresinin birden çok gerçek değerden oluşmasından kaynaklanan hataları önlemek içindir.

#define BAD_AREA(r) PI * r * r

double bad_area = BAD_AREA(5 + 1.6);
// Compiler sees:
double bad_area = 3.14159265358979 * 5 + 1.6 * 5 + 1.6;

double good_area = AREA(5 + 1.6);
// Compiler sees:
double good_area = (3.14159265358979*(5 + 1.6)*(5 + 1.6));

Ayrıca, bu basit genişleme nedeniyle, beklenmeyen yan etkileri önlemek için makrolara iletilen parametrelere dikkat edilmesi gerektiğini unutmayın. Parametre değerlendirme sırasında değiştirilirse, genellikle istediğimiz gibi olmayan genişletilmiş makroda her kullanıldığında değiştirilir. Bu, genişlemenin herhangi bir şeyi bozmasını önlemek için makro parametreleri parantez içine alsa bile geçerlidir.

int oops = 5;
double incremental_damage = AREA(oops++);
// Compiler sees:
double incremental_damage = (3.14159265358979*(oops++)*(oops++));

Ek olarak, makrolar tür güvenliği sağlamaz, bu da tür uyumsuzluğuyla ilgili anlaşılması zor hatalara yol açar.

Programcılar normalde satırları noktalı virgülle sonlandırdıklarından, bağımsız satırlar olarak kullanılması amaçlanan makrolar genellikle noktalı virgülü “yutmak” için tasarlanmıştır; bu, istenmeyen hataların fazladan bir noktalı virgülden kaynaklanmasını önler.

#define IF_BREAKER(Func) Func();

if (some_condition)
    // Oops.
    IF_BREAKER(some_func);
else
    std::cout << "I am accidentally an orphan." << std::endl;

Bu örnekte, yanlışlıkla çift noktalı virgül ‘if…else’ bloğunu kırarak derleyicinin ’else’ı ‘if’ ile eşleştirmesini engeller. Bunu önlemek için, noktalı virgül makro tanımından çıkarılır ve bu noktalı virgülün herhangi bir kullanımından hemen sonra “yutmasına” neden olur.

#define IF_FIXER(Func) Func()

if (some_condition)
    IF_FIXER(some_func);
else
    std::cout << "Hooray!  I work again!" << std::endl;

Sondaki noktalı virgülün bırakılması, makronun geçerli ifadeyi sonlandırmadan kullanılmasına da olanak tanır, bu da faydalı olabilir.

#define DO_SOMETHING(Func, Param) Func(Param, 2)

// ...

some_function(DO_SOMETHING(some_func, 3), DO_SOMETHING(some_func, 42));

Normalde, bir makro tanımı satırın sonunda biter. Ancak bir makronun birden çok satırı kapsaması gerekiyorsa, bunu belirtmek için satırın sonunda ters eğik çizgi kullanılabilir. Bu ters eğik çizgi, önişlemciye aşağıdaki satırın geçerli satırda birleştirilmesi gerektiğini ve bunları tek bir satır olarak ele alması gerektiğini belirten satırdaki son karakter olmalıdır. Bu, arka arkaya birden çok kez kullanılabilir.

#define TEXT "I \
am \
many \
lines."

// ...

std::cout << TEXT << std::endl; // Output:   I am many lines.

Bu, özellikle birden çok satırı kapsaması gerekebilecek karmaşık işlev benzeri makrolarda kullanışlıdır.

#define CREATE_OUTPUT_AND_DELETE(Str) \
    std::string* tmp = new std::string(Str); \
    std::cout << *tmp << std::endl; \
    delete tmp;

// ...

CREATE_OUTPUT_AND_DELETE("There's no real need for this to use 'new'.")

Daha karmaşık işlev benzeri makrolar söz konusu olduğunda, olası ad çakışmalarını önlemek veya gerçek bir işleve benzer şekilde makronun sonunda nesnelerin yok edilmesine neden olmak için onlara kendi kapsamlarını vermek yararlı olabilir. Bunun için yaygın bir deyim do while 0‘dır, burada makro bir do-while bloğu içine alınır. Bu blok genellikle bir noktalı virgül ile takip edilmez, bu da bir noktalı virgül yutmasına izin verir.

#define DO_STUFF(Type, Param, ReturnVar) do { \
    Type temp(some_setup_values); \
    ReturnVar = temp.process(Param); \
} while (0)

int x;
DO_STUFF(MyClass, 41153.7, x);

// Compiler sees:

int x;
do {
    MyClass temp(some_setup_values);
    x = temp.process(41153.7);
} while (0);

Değişken makrolar da vardır; değişken işlevlere benzer şekilde, bunlar değişken sayıda argüman alır ve ardından hepsini özel bir “Varargs” parametresi olan __VA_ARGS__ yerine genişletir.

#define VARIADIC(Param, ...) Param(__VA_ARGS__)

VARIADIC(printf, "%d", 8);
// Compiler sees:
printf("%d", 8);

Genişletme sırasında __VA_ARGS__ tanımın herhangi bir yerine yerleştirilebileceğini ve doğru şekilde genişletileceğini unutmayın.

#define VARIADIC2(POne, PTwo, PThree, ...) POne(PThree, __VA_ARGS__, PTwo)

VARIADIC2(some_func, 3, 8, 6, 9);
// Compiler sees:
some_func(8, 6, 9, 3);

Sıfır bağımsız değişkenli değişken parametre durumunda, farklı derleyiciler sondaki virgülü farklı şekilde işleyecektir. Visual Studio gibi bazı derleyiciler, herhangi bir özel sözdizimi olmadan virgülü sessizce yutar. GCC gibi diğer derleyiciler, “##” karakterini “VA_ARGS“den hemen önce yerleştirmenizi gerektirir. Bu nedenle, taşınabilirlik söz konusu olduğunda değişken makroları koşullu olarak tanımlamak akıllıca olacaktır.

// In this example, COMPILER is a user-defined macro specifying the compiler being used.

#if       COMPILER == "VS"
    #define VARIADIC3(Name, Param, ...) Name(Param, __VA_ARGS__)
#elif     COMPILER == "GCC"
    #define VARIADIC3(Name, Param, ...) Name(Param, ##__VA_ARGS__)
#endif /* COMPILER */

X makroları #

Derleme zamanında yinelenen kod yapıları oluşturmak için deyimsel bir teknik.

Bir X makrosu iki bölümden oluşur: liste ve listenin yürütülmesi.

Örnek:

#define LIST \
    X(dog)   \
    X(cat)   \
    X(racoon)

// class Animal {
//  public:
//    void say();
// };

#define X(name) Animal name;
LIST
#undef X

int main() {
#define X(name) name.say();
    LIST
#undef X

    return 0;
}

önişlemci tarafından aşağıdakilere genişletilir:

Animal dog;
Animal cat;
Animal racoon;

int main() {
    dog.say();
    cat.say();
    racoon.say();

    return 0;
}

Listeler büyüdükçe (diyelim ki 100’den fazla öğe), bu teknik aşırı kopyala-yapıştır yapmaktan kaçınmaya yardımcı olur.

Kaynak: https://en.wikipedia.org/wiki/X_Macro

Ayrıca bakınız: https://www.wikiod.com/tr/c/x-makrolar

“LIST” kullanmadan önce alakasız bir “X” tanımlamak hoşunuza gitmiyorsa, argüman olarak bir makro adı da iletebilirsiniz:

#define LIST(MACRO) \
    MACRO(dog) \
    MACRO(cat) \
    MACRO(racoon)

Şimdi, listeyi genişletirken hangi makronun kullanılması gerektiğini açıkça belirtiyorsunuz, ör.

#define FORWARD_DECLARE_ANIMAL(name) Animal name;
LIST(FORWARD_DECLARE_ANIMAL)

‘MAKRO’nun her bir çağrısının ek parametreler alması gerekiyorsa - listeye göre sabit, değişken makrolar kullanılabilir

//a walkaround for Visual studio
#define EXPAND(x) x

#define LIST(MACRO, ...) \
    EXPAND(MACRO(dog, __VA_ARGS__)) \
    EXPAND(MACRO(cat, __VA_ARGS__)) \
    EXPAND(MACRO(racoon, __VA_ARGS__))

İlk argüman “LIST” tarafından sağlanırken geri kalanı “LIST” çağrısında kullanıcı tarafından sağlanır. Örneğin:

#define FORWARD_DECLARE(name, type, prefix) type prefix##name;
LIST(FORWARD_DECLARE,Animal,anim_)
LIST(FORWARD_DECLARE,Object,obj_)

genişleyecek

Animal anim_dog;
Animal anim_cat;
Animal anim_racoon;
Object obj_dog;
Object obj_cat;
Object obj_racoon;

Önişlemci hata mesajları #

Derleme hataları önişlemci kullanılarak oluşturulabilir. Bu, bazıları desteklenmeyen bir platformda veya desteklenmeyen bir derleyicideyse kullanıcıyı bilgilendirmeyi içeren birkaç nedenden dolayı yararlıdır.

Örneğin. gcc sürümü 3.0.0 veya daha eskiyse Hata Döndürme.

#if __GNUC__ < 3
#error "This code requires gcc > 3.0.0"
#endif

Örneğin. Bir Apple bilgisayarda derleme yapıyorsanız Hata Döndürme.

#ifdef __APPLE__
#error "Apple products are not supported in this release"
#endif

Önceden tanımlanmış makrolar #

Önceden tanımlanmış makrolar, derleyicinin tanımladığı makrolardır (kullanıcının kaynak dosyada tanımladıklarının aksine). Bu makrolar kullanıcı tarafından yeniden tanımlanmamalı veya tanımlanmamalıdır.

Aşağıdaki makrolar, C++ standardı tarafından önceden tanımlanmıştır:

  • ‘LINE’, bu makronun kullanıldığı satırın satır numarasını içerir ve ‘#line’ yönergesi ile değiştirilebilir.
  • __FILE__, bu makronun kullanıldığı dosyanın adını içerir ve #line yönergesi ile değiştirilebilir.
  • ‘DATE’, dosya derlemesinin tarihini (“Mmm dd yyyy”’ biçiminde) içerir; burada Mmm, ‘std::asctime()’ çağrısıyla elde edilmiş gibi biçimlendirilir.
  • “TIME” dosya derlemesinin zamanını (“ss:dd:ss”` biçiminde) içerir.
  • __cplusplus, C++ dosyalarını derlerken (uyumlu) C++ derleyicileri tarafından tanımlanır. Değeri, derleyicinin tamamen uyumlu olduğu standart sürümdür, yani C++98 ve C++03 için “199711L”, C++11 için “201103L” ve C++14 standardı için “201402L” .
  • ‘STDC_HOSTED’, uygulama barındırılıyorsa ‘1’ veya bağımsız ise ‘0’ olarak tanımlanır.
son sürüm
  • “STDCPP_DEFAULT_NEW_ALIGNMENT”, hizalamadan habersiz “yeni operatör” çağrısı için kullanılan hizalama olan bir “size_t” değişmezi içerir.
son sürüm

Ek olarak, aşağıdaki makroların uygulamalar tarafından önceden tanımlanmasına izin verilir ve mevcut olabilir veya olmayabilir:

  • ‘STDC’, uygulamaya bağlı bir anlama sahiptir ve genellikle tam C standardı uyumluluğunu belirtmek için yalnızca bir dosya C olarak derlenirken tanımlanır. (Ya da derleyici bu makroyu desteklememeye karar verirse asla.)
  • “STDC_VERSION”, uygulamaya bağlı bir anlama sahiptir ve değeri, “__cplusplus"ın C++ sürümü olmasına benzer şekilde, genellikle C sürümüdür. (Ya da derleyici bu makroyu desteklememeye karar verirse tanımlı bile değildir.)
  • ‘STDC_MB_MIGHT_NEQ_WC’, temel karakter kümesinin dar kodlamasının değerleri geniş karşılıklarının değerlerine eşit değilse, ‘1’ olarak tanımlanır (örneğin, if `(uintmax_t)‘x’ != (uintmax_t)L’ x’’)
  • “wchar_t” Unicode olarak kodlanmışsa “STDC_ISO_10646” tanımlanır ve desteklenen en son Unicode revizyonunu gösteren “yyyymmL” biçiminde bir tamsayı sabitine genişler.
  • __STDCPP_STRICT_POINTER_SAFETY__, uygulamanın katı işaretçi güvenliği varsa (aksi takdirde gevşemiş işaretçi güvenliği varsa) 1 olarak tanımlanır.
  • ‘STDCPP_THREADS’, programın birden fazla yürütme iş parçacığına sahip olması durumunda ‘1’ olarak tanımlanır (bağımsız uygulama için geçerlidir — barındırılan uygulamalar her zaman birden fazla iş parçacığına sahip olabilir)

Ayrıca bir makro değil, önceden tanımlanmış bir yerel işlev değişkeni olan func‘dan bahsetmeye değer. Uygulama tanımlı bir biçimde statik bir karakter dizisi olarak kullanıldığı işlevin adını içerir.

son sürüm

Bu standart önceden tanımlanmış makrolara ek olarak, derleyiciler kendi önceden tanımlanmış makro setlerine sahip olabilir. Bunları öğrenmek için derleyici belgelerine başvurmak gerekir. Örneğin.:

  • gcc
  • Microsoft Visual C++
  • klan
  • Intel C++ Derleyici

Bazı makrolar yalnızca bazı özelliklerin desteğini sorgulamak içindir:

#ifdef __cplusplus // if compiled by C++ compiler
extern "C"{ // C code has to be decorated
   // C library header declarations here
}
#endif

Diğerleri hata ayıklama için çok kullanışlıdır:

bool success = doSomething( /*some arguments*/ );
if( !success ){
    std::cerr << "ERROR: doSomething() failed on line " << __LINE__ - 2
              << " in function " << __func__ << "()"
              << " in file " << __FILE__
              << std::endl;
}
son sürüm

Ve önemsiz sürüm kontrolü için diğerleri:

int main( int argc, char *argv[] ){
    if( argc == 2 && std::string( argv[1] ) == "-v" ){
        std::cout << "Hello World program\n"
                  << "v 1.1\n" // I have to remember to update this manually
                  << "compiled: " << __DATE__ << ' ' << __TIME__ // this updates automagically
                  << std::endl;
    }
    else{
        std::cout << "Hello World!\n";
    }

#pragma bir kez #

Tümü olmasa da çoğu C++ uygulaması dosyanın tek bir derlemeye yalnızca bir kez dahil edilmesini sağlayan #pragma once yönergesini destekler. Herhangi bir ISO C++ standardının parçası değildir. Örneğin:

// Foo.h
#pragma once

class Foo
{
};

“#pragma once”, include guards ile ilgili bazı sorunları önlerken, bir “#pragma” - standartlardaki tanımı gereği - doğası gereği derleyiciye özel bir kancadır ve bunu yapmayan derleyiciler tarafından sessizce göz ardı edilir. destekleyin. #pragma once kullanan projeler, standartlara uygun olacak şekilde değiştirilmelidir.

Bazı derleyicilerde - özellikle önceden derlenmiş başlıklar kullananlarda - “#pragma once”, derleme işleminin önemli ölçüde hızlanmasına neden olabilir. Benzer şekilde, bazı önişlemciler, hangi başlıkların korumaları içerdiğini izleyerek derlemenin hızlandırılmasını sağlar. Hem “#pragma once” hem de korumaları içerdiğinde net fayda uygulamaya bağlıdır ve derleme sürelerinde bir artış veya azalma olabilir.

include guards ile birlikte “#pragma once”, pencerelerde MFC tabanlı uygulamalar yazarken başlık dosyaları için önerilen düzendi ve Visual Studio’nun “sınıf ekle”, “ileti kutusu ekle”, “pencere ekle” sihirbazları tarafından oluşturuldu. . Bu nedenle, bunları C++ Windows Başvuru Sahiplerinde birleştirilmiş olarak bulmak çok yaygındır.

Önişlemci Operatörleri #

“#” operatörü veya dize oluşturma operatörü, bir Makro parametresini bir dize değişmezine dönüştürmek için kullanılır. Yalnızca argümanları olan Makrolar ile kullanılabilir.

// preprocessor will convert the parameter x to the string literal x
#define PRINT(x) printf(#x "\n")

PRINT(This line will be converted to string by preprocessor);
// Compiler sees
printf("This line will be converted to string by preprocessor""\n");

Derleyici iki dizeyi birleştirir ve son “printf()” argümanı, sonunda yeni satır karakteri olan bir dize değişmezi olur.

Önişlemci, makro bağımsız değişkeninden önceki veya sonraki boşlukları yok sayar. Yani aşağıdaki print ifadesi bize aynı sonucu verecektir.

PRINT(   This line will be converted to string by preprocessor );

Dize değişmezinin parametresi, çift tırnaktan () önceki gibi bir kaçış dizisi gerektiriyorsa, önişlemci tarafından otomatik olarak eklenecektir.

PRINT(This "line" will be converted to "string" by preprocessor); 
// Compiler sees
printf("This \"line\" will be converted to \"string\" by preprocessor""\n");

“##” operatörü veya Token yapıştırma operatörü, bir Makronun iki parametresini veya jetonunu birleştirmek için kullanılır.

// preprocessor will combine the variable and the x
#define PRINT(x) printf("variable" #x " = %d", variable##x)

int variableY = 15;
PRINT(Y);
//compiler sees
printf("variable""Y"" = %d", variableY);

ve nihai çıktı olacak

variableY = 15
