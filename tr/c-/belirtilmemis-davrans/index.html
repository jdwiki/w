Bir yapının davranışı belirtilmemişse, standart davranışa bazı kısıtlamalar getirir, ancak uygulamaya belirli bir durumda ne olduğunu belgelemek için gerekli olmayan bir miktar özgürlük bırakır. Bu, uygulamanın neler olduğunu belgelemek için gerekli olduğu ve tanımsız davranış, içinde her şey olabilir.

Aralık dışı bir numaralandırmanın değeri #

Kapsamlı bir numaralandırma, değerini tutamayacak kadar küçük bir integral türüne dönüştürülürse, elde edilen değer belirtilmez. Örnek:

enum class E {
    X = 1,
    Y = 1000,
};
// assume 1000 does not fit into a char
char c1 = static_cast<char>(E::X); // c1 is 1
char c2 = static_cast<char>(E::Y); // c2 has an unspecified value

Ayrıca, bir tamsayı bir numaralandırmaya dönüştürülürse ve tamsayı değeri, numaralandırma değerleri aralığının dışındaysa, elde edilen değer belirtilmez. Örnek:

enum Color {
    RED = 1,
    GREEN = 2,
    BLUE = 3,
};
Color c = static_cast<Color>(4);

Bununla birlikte, sonraki örnekte, kaynak değer tüm numaralandırıcılara eşit olmamasına rağmen, kaynak değerin aralığı içinde olduğundan, davranış belirtilmemiştir:

enum Scale {
    ONE = 1,
    TWO = 2,
    FOUR = 4,
};
Scale s = static_cast<Scale>(3);

Burada “s” 3 değerine sahip olacak ve “BİR”, “İKİ” ve “DÖRT"e eşit olmayacak.

İşlev argümanlarının değerlendirme sırası #

Bir fonksiyonun birden fazla argümanı varsa, hangi sırayla değerlendirildiği belirtilmemiştir. Aşağıdaki kod ‘x = 1, y = 2’ veya ‘x = 2, y = 1’ yazdırabilir, ancak hangisi olduğu belirtilmemiştir.

int f(int x, int y) {
    printf("x = %d, y = %d\n", x, y);
}
int get_val() {
    static int x = 0;
    return ++x;
}
int main() {
    f(get_val(), get_val());
}

C++17’de, fonksiyon argümanlarının değerlendirme sırası belirtilmemiş olarak kalır.

Ancak, her işlev bağımsız değişkeni tamamen değerlendirilir ve çağıran nesnenin, herhangi bir işlev bağımsız değişkeni değerlendirilmeden önce değerlendirilmesi garanti edilir.

struct from_int {
  from_int(int x) { std::cout << "from_int (" << x << ")\n"; }
};
int make_int(int x){ std::cout << "make_int (" << x << ")\n"; return x; }


void foo(from_int a, from_int b) {
}
void bar(from_int a, from_int b) {
}

auto which_func(bool b){
  std::cout << b?"foo":"bar" << "\n";
  return b?foo:bar;
}

int main(int argc, char const*const* argv) {
  which_func( true )( make_int(1), make_int(2) );
}

bu yazdırılmalıdır:

bar
make_int(1)
from_int(1)
make_int(2)
from_int(2)

veya

bar
make_int(2)
from_int(2)
make_int(1)
from_int(1)

herhangi bir “make” veya “from"dan sonra “çubuk” yazdıramaz ve yazdıramaz:

bar
make_int(2)
make_int(1)
from_int(2)
from_int(1)

veya benzeri. C++ 17’den önce, herhangi bir “from_int” yapmadan önce “make_int"leri yapmak gibi, “make_int"lerden sonra “bar” yazdırmak yasaldı.

Bazı reinterpret_cast dönüşümlerinin sonucu #

Bir işlev işaretçi türünden diğerine veya bir işlev başvuru türünden diğerine “yeniden yorumlama_gönderme"nin sonucu belirtilmemiş. Örnek:

int f();
auto fp = reinterpret_cast<int(*)(int)>(&f); // fp has unspecified value

Bir nesne işaretçi türünden diğerine veya bir nesne başvuru türünden diğerine “yeniden yorumlama_gönderme"nin sonucu belirtilmemiştir. Örnek:

int x = 42;
char* p = reinterpret_cast<char*>(&x); // p has unspecified value

Ancak, çoğu derleyicide bu, “static_cast<char*>(static_cast<void*>(&x))” işlevine eşdeğerdi, dolayısıyla ortaya çıkan “p” işaretçisi “x"in ilk baytını gösteriyordu. Bu, C++ 11’de standart davranış haline getirildi. Daha fazla ayrıntı için tip punning dönüşümü adresine bakın.

Referans tarafından işgal edilen alan #

Bir başvuru bir nesne değildir ve bir nesnenin aksine, bazı bitişik bellek baytlarını işgal etmesi garanti edilmez. Standart, bir referansın herhangi bir depolama gerektirip gerektirmediğini belirtmeden bırakır. Dilin bir dizi özelliği, referansın kaplayabileceği herhangi bir depolamayı taşınabilir olarak incelemeyi imkansız kılmak için bir araya geliyor:

  • Bir referansa “sizeof” uygulanırsa, başvurulan türün boyutunu döndürür, böylece başvurunun herhangi bir depolama alanı kaplayıp kaplamadığı hakkında hiçbir bilgi vermez.
  • Referans dizileri geçersizdir, bu nedenle bir referansın boyutunu belirlemek için varsayımsal bir dizi referansının ardışık iki öğesinin adreslerini incelemek mümkün değildir.
  • Bir referansın adresi alınırsa, sonuç referansın adresi olur, bu nedenle referansın kendisine bir işaretçi alamayız.
  • Bir sınıfın referans üyesi varsa, o üyenin adresini ‘offsetof’ kullanarak çıkarmaya çalışmak, böyle bir sınıf standart yerleşim sınıfı olmadığı için tanımsız davranış verir.
  • Bir sınıfın referans üyesi varsa, sınıf artık standart düzen değildir, bu nedenle tanımsız veya belirtilmemiş davranışta referans sonuçlarını depolamak için kullanılan herhangi bir veriye erişmeye çalışır.

Uygulamada, bazı durumlarda bir referans değişkeni, bir işaretçi değişkenine benzer şekilde uygulanabilir ve bu nedenle bir işaretçi ile aynı miktarda depolamayı kaplar, diğer durumlarda bir referans, optimize edilebileceği için hiç yer kaplamayabilir. Örneğin:

void f() {
    int x;
    int& r = x;
    // do something with r
}

derleyici, “r"yi “x” için bir takma ad olarak ele almakta ve “f” işlevinin geri kalanında “r"nin tüm oluşumlarını “x” ile değiştirmekte ve “r"yi tutmak için herhangi bir depolama ayırmamakta özgürdür.

Çoğu standart kütüphane sınıfının durumundan taşındı #

Tüm standart kitaplık kapları, taşındıktan sonra geçerli ancak belirtilmemiş bir durumda bırakılır. Örneğin, aşağıdaki kodda, “v2”, taşımadan sonra “{1, 2, 3, 4}” içerecektir, ancak “v1"in boş olacağı garanti edilmez.

int main() {
    std::vector<int> v1{1, 2, 3, 4};
    std::vector<int> v2 = std::move(v1);
}

Bazı sınıfların kesin olarak tanımlanmış bir taşınmış durumu vardır. En önemli durum, taşındıktan sonra boş olması garanti edilen std::unique_ptr<T> durumudur.

TU genelinde globallerin başlatma sırası #

Bir Çeviri Birimi içinde, global değişkenlerin başlatma sırası belirtilirken, Çeviri Birimleri arasında başlatma sırası belirtilmez.

Yani aşağıdaki dosyalarla programlayın

  • foo.cpp

    #include <iostream>

int dummyFoo = ((std::cout << "foo"), 0);

  • bar.cpp

    #include <iostream>

int dummyBar = ((std::cout << "bar"), 0);

  • ana.cpp

    int main() {}

çıktı olarak üretebilir:

foobar

veya

barfoo

Bu, Statik Başlatma Siparişi Fiasco’suna yol açabilir.

Sahte boşluk* değerinden statik yayın #

Bir “void*” değeri, nesne türü “T*” için bir işaretçiye dönüştürülür, ancak “T” için uygun şekilde hizalanmazsa, sonuçta ortaya çıkan işaretçi değeri belirtilmez. Örnek:

// Suppose that alignof(int) is 4
int x = 42;
void* p1 = &x;
// Do some pointer arithmetic...
void* p2 = static_cast<char*>(p1) + 2;
int* p3 = static_cast<int*>(p2);

“p3” değeri belirtilmemiş çünkü “p2”, “int” türünde bir nesneye işaret edemez; değeri düzgün hizalanmış bir adres değil.

Bazı işaretçi karşılaştırmalarının sonucu #

İki işaretçi <, >, <= veya >= kullanılarak karşılaştırılırsa, aşağıdaki durumlarda sonuç belirtilmez:

  • İşaretçiler farklı dizilere işaret eder. (Dizi olmayan bir nesne, 1 boyutunda bir dizi olarak kabul edilir.)

    int x;
int y;
const bool b1 = &x < &y;            // unspecified
int a[10];
const bool b2 = &a[0] < &a[1];      // true
const bool b3 = &a[0] < &x;         // unspecified
const bool b4 = (a + 9) < (a + 10); // true
                                    // note: a+10 points past the end of the array

  • İşaretçiler aynı nesneye, ancak farklı erişim denetimine sahip üyelere işaret eder.

    class A {
  public:
    int x;
    int y;
    bool f1() { return &x < &y; } // true; x comes before y
    bool f2() { return &x < &z; } // unspecified
  private:
    int z;
};
