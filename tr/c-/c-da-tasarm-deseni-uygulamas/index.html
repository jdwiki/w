Bu sayfada, tasarım desenlerinin C++‘da nasıl uygulandığına dair örnekler bulabilirsiniz. Bu kalıplarla ilgili ayrıntılar için tasarım kalıpları belgelerine göz atabilirsiniz.

Tasarım deseni, yazılım tasarımında belirli bir bağlamda yaygın olarak ortaya çıkan bir soruna genel olarak yeniden kullanılabilir bir çözümdür.

Adaptör Kalıbı #

Bir sınıfın arabirimini, istemcilerin beklediği başka bir arabirime dönüştürün. Bağdaştırıcı (veya Sarıcı), uyumsuz arabirimler nedeniyle başka türlü yapamayan sınıfların birlikte çalışmasına olanak tanır. Bağdaştırıcı modelinin motivasyonu, arayüzü değiştirebilirsek mevcut yazılımı yeniden kullanabilmemizdir.

  1. Bağdaştırıcı deseni, nesne bileşimine dayanır.

  2. İstemci, Adapter nesnesinde işlemi çağırır.

  3. Adaptör, işlemi gerçekleştirmek için Adaptee’yi çağırır.

  4. STL’de, vektörden uyarlanmış yığın: Yığın push()‘u çalıştırdığında, alttaki vektör vector::push_back() işlevini yapar.

Örnek:

#include <iostream>

// Desired interface (Target)
class Rectangle 
{
  public:
    virtual void draw() = 0;
};

// Legacy component (Adaptee)
class LegacyRectangle 
{
  public:
    LegacyRectangle(int x1, int y1, int x2, int y2) {
        x1_ = x1;
        y1_ = y1;
        x2_ = x2;
        y2_ = y2;
        std::cout << "LegacyRectangle(x1,y1,x2,y2)\n";
    }
    void oldDraw() {
        std::cout << "LegacyRectangle:  oldDraw(). \n";
    }
  private:
    int x1_;
    int y1_;
    int x2_;
    int y2_;
};

// Adapter wrapper
class RectangleAdapter: public Rectangle, private LegacyRectangle 
{
  public:
    RectangleAdapter(int x, int y, int w, int h):
      LegacyRectangle(x, y, x + w, y + h) {
         std::cout << "RectangleAdapter(x,y,x+w,x+h)\n";
      }

    void draw() {
        std::cout << "RectangleAdapter: draw().\n"; 
        oldDraw();
    }
};

int main()
{
  int x = 20, y = 50, w = 300, h = 200;
  Rectangle *r = new RectangleAdapter(x,y,w,h);
  r->draw();
}

//Output:
//LegacyRectangle(x1,y1,x2,y2)
//RectangleAdapter(x,y,x+w,x+h)

Kodun özeti:

  1. Müşteri bir “Dikdörtgen” ile konuştuğunu düşünüyor

  2. Hedef, ‘Dikdörtgen’ sınıfıdır. İstemcinin yöntemi çağırdığı şey budur.

     Rectangle *r = new RectangleAdapter(x,y,w,h);
 r->draw();

  3. Bağdaştırıcı sınıfının çoklu kalıtım kullandığını unutmayın.

     class RectangleAdapter: public Rectangle, private LegacyRectangle {
     ...
 }

  4. Bağdaştırıcı ‘RectangleAdapter’, ‘LegacyRectangle’ın isteğe (‘Dikdörtgen’ üzerinde ‘draw()’) HER İKİ sınıfı devralarak yanıt vermesini sağlar.

  5. ‘LegacyRectangle’ sınıfı, ‘Rectangle’ ile aynı yöntemlere (‘draw()’) sahip değildir, ancak ‘Adapter(RectangleAdapter)’, ‘Rectangle’ yöntem çağrılarını alabilir ve ‘LegacyRectangle’, ‘oldDraw()’ üzerinde geri dönüp yöntemi çağırabilir.

     class RectangleAdapter: public Rectangle, private LegacyRectangle {
   public:
     RectangleAdapter(int x, int y, int w, int h):
       LegacyRectangle(x, y, x + w, y + h) {
         std::cout << "RectangleAdapter(x,y,x+w,x+h)\n";
       }

     void draw() {
         std::cout << "RectangleAdapter: draw().\n"; 
         oldDraw();
     }
 };

Adaptör tasarım deseni, bir sınıfın arayüzünü uyumlu ancak farklı bir arayüze dönüştürür. Dolayısıyla bu, tek bileşenli bir sarmalayıcı olması bakımından proxy modeline benzer. Ancak bağdaştırıcı sınıfı ile orijinal sınıfın arayüzü farklı olabilir.

Yukarıdaki örnekte gördüğümüz gibi, bu bağdaştırıcı modeli, diğer kodlarla çalışmasına izin vermek için mevcut bir API için farklı bir arabirim ortaya çıkarmak için kullanışlıdır. Ayrıca, bağdaştırıcı desenini kullanarak, heterojen arabirimleri alabilir ve tutarlı API sağlamak için bunları dönüştürebiliriz.

Köprü deseni, nesne adaptörüne benzer bir yapıya sahiptir, ancak Bridge’in farklı bir amacı vardır: Kolay ve bağımsız olarak değiştirilebilmesi için bir arayüzü uygulamasından ayırması amaçlanmıştır. Bir bağdaştırıcı, mevcut bir nesnenin arayüzünü değiştirmek içindir.

Gözlemci modeli #

Gözlemci Modelinin amacı, nesneler arasında bire çok bağımlılık tanımlamaktır, böylece bir nesne durum değiştirdiğinde, tüm bağımlıları otomatik olarak bildirilir ve güncellenir.

Özne ve gözlemciler bire çok ilişkisini tanımlar. Gözlemciler özneye öyle bağımlıdırlar ki öznenin durumu değiştiğinde gözlemciler haberdar edilir. Bildirime bağlı olarak, gözlemciler de yeni değerlerle güncellenebilir.

İşte Gamma’nın “Tasarım Modelleri” kitabından bir örnek.

#include <iostream>
#include <vector>

class Subject; 

class Observer 
{ 
public:
    virtual ~Observer() = default;
    virtual void Update(Subject&) = 0;
};

class Subject 
{ 
public: 
     virtual ~Subject() = default;
     void Attach(Observer& o) { observers.push_back(&o); }
     void Detach(Observer& o)
     {
         observers.erase(std::remove(observers.begin(), observers.end(), &o));
     }
     void Notify()
     {
         for (auto* o : observers) {
             o->Update(*this);
         }
     }
private:
     std::vector<Observer*> observers; 
};

class ClockTimer : public Subject 
{ 
public:

    void SetTime(int hour, int minute, int second)
    {
        this->hour = hour; 
        this->minute = minute;
        this->second = second;

        Notify(); 
    }

    int GetHour() const { return hour; }
    int GetMinute() const { return minute; }
    int GetSecond() const { return second; }

private: 
    int hour;
    int minute;
    int second;
}; 

class DigitalClock: public Observer 
{ 
public: 
     explicit DigitalClock(ClockTimer& s) : subject(s) { subject.Attach(*this); }
     ~DigitalClock() { subject.Detach(*this); }
     void Update(Subject& theChangedSubject) override
     {
         if (&theChangedSubject == &subject) {
             Draw();
         }
     }

     void Draw()
     {
         int hour = subject.GetHour(); 
         int minute = subject.GetMinute(); 
         int second = subject.GetSecond(); 

         std::cout << "Digital time is " << hour << ":" 
                   << minute << ":" 
                   << second << std::endl;           
     }

private:
     ClockTimer& subject;
};

class AnalogClock: public Observer 
{ 
public: 
     explicit AnalogClock(ClockTimer& s) : subject(s) { subject.Attach(*this); }
     ~AnalogClock() { subject.Detach(*this); }
     void Update(Subject& theChangedSubject) override
     {
         if (&theChangedSubject == &subject) {
             Draw();
         }
     }
     void Draw()
     {
         int hour = subject.GetHour(); 
         int minute = subject.GetMinute(); 
         int second = subject.GetSecond(); 

         std::cout << "Analog time is " << hour << ":" 
                   << minute << ":" 
                   << second << std::endl; 
     }
private:
     ClockTimer& subject;
};

int main()
{ 
    ClockTimer timer; 

    DigitalClock digitalClock(timer); 
    AnalogClock analogClock(timer); 

    timer.SetTime(14, 41, 36);
}

Çıktı:

Digital time is 14:41:36
Analog time is 14:41:36

İşte kalıbın özeti:

  1. Nesneler (“DigitalClock” veya “AnalogClock” nesnesi), ya gözlemci olarak abone olmak (kaydolmak) ya da kendilerini gözlemci olmaktan çıkarmak (kaldırmak) için Konu arabirimlerini (‘Attach()’ veya ‘Detach()’) kullanır. .Attach(*bu);,subject.Detach(*bu); `.

  2. Her konunun birçok gözlemcisi olabilir ( vector<Observer*> gözlemci;).

  3. Tüm gözlemcilerin, Gözlemci arayüzünü uygulaması gerekir. Bu arabirimde, Konunun durumu değiştiğinde çağrılan “Update()” adlı bir yöntem vardır (‘Update(Subject &)’)

  4. ‘Attach()’ ve ‘Detach()’ yöntemlerine ek olarak, somut özne, durum değiştiğinde tüm mevcut gözlemcileri güncellemek için kullanılan bir ‘Notify()’ yöntemini uygular. Ancak bu durumda, bunların tümü, Subject' (Subject::Attach (Observer&) , void Subject::Detach(Observer&)vevoid Subject::Notify()` üst sınıfında yapılır.

  5. Somut nesne, durumunu ayarlamak ve almak için yöntemlere de sahip olabilir.

  6. Somut gözlemciler, Observer arabirimini uygulayan herhangi bir sınıf olabilir. Her gözlemci, güncelleme almak için somut bir konu ile abone olur (kaydeder) (subject.Attach(*this); ).

  7. Gözlemci Modelinin iki nesnesi gevşek bir şekilde bağlanmıştır, etkileşime girebilirler ancak birbirleri hakkında çok az bilgi sahibidirler.

Varyasyon:

Sinyal ve Yuvalar

Sinyaller ve yuvalar, Qt’de tanıtılan ve ortak koddan kaçınırken Gözlemci modelini uygulamayı kolaylaştıran bir dil yapısıdır. Konsept, kontrollerin (widget olarak da bilinir), yuvalar olarak bilinen özel işlevleri kullanarak diğer kontroller tarafından alınabilen olay bilgilerini içeren sinyaller gönderebilmesidir. Qt’deki yuva, bu şekilde bildirilen bir sınıf üyesi olmalıdır. Sinyal/slot sistemi, Grafiksel Kullanıcı Arayüzlerinin tasarlanma şekline çok uygundur. Benzer şekilde, sinyal/yuva sistemi asenkron G/Ç (soketler, borular, seri cihazlar vb. dahil) olay bildirimi veya zaman aşımı olaylarını uygun nesne örnekleri ve yöntemleri veya işlevleriyle ilişkilendirmek için kullanılabilir. Qt’nin Meta Nesne Derleyicisi (MOC) gerekli altyapıyı otomatik olarak oluşturduğundan, kayıt/kayıt silme/çağırma kodunun yazılmasına gerek yoktur.

C# dili de benzer bir yapıyı destekler, ancak farklı bir terminoloji ve sözdizimi vardır: olaylar sinyal rolünü oynar ve delegeler yuvalardır. Ek olarak, bir temsilci, bir işlev işaretçisi gibi yerel bir değişken olabilirken, Qt’deki bir yuva, bu şekilde bildirilen bir sınıf üyesi olmalıdır.

Fabrika Deseni #

Fabrika modeli, nesne oluşturmayı ayırır ve ortak bir arabirim kullanarak ada göre oluşturmaya olanak tanır:

class Animal{
public:
    virtual std::shared_ptr<Animal> clone() const = 0;
    virtual std::string  getname() const = 0;
};

class Bear: public Animal{
public:
    virtual std::shared_ptr<Animal> clone() const override
    {
        return std::make_shared<Bear>(*this);
    }
    virtual std::string getname() const override
    {
        return "bear";
    }
};


class Cat: public Animal{
public:
    virtual std::shared_ptr<Animal> clone() const override
    {
        return std::make_shared<Cat>(*this);
    }
    virtual std::string  getname() const override
    {
        return "cat";
    }
};

class AnimalFactory{
public:
    static std::shared_ptr<Animal> getAnimal( const std::string&   name )
    {
      if ( name == "bear" )
        return std::make_shared<Bear>();
      if ( name == "cat" )
        return std::shared_ptr<Cat>();
     
    return nullptr;
    }


};

Akıcı API ile Oluşturucu Modeli #

Oluşturucu Modeli, nesnenin oluşturulmasını nesnenin kendisinden ayırır. Bunun arkasındaki ana fikir, bir nesnenin kendi yaratılışından sorumlu olması gerekmediğidir. Karmaşık bir nesnenin doğru ve geçerli montajı kendi içinde karmaşık bir görev olabilir, bu nedenle bu görev başka bir sınıfa devredilebilir.

[C#’ta E-posta Oluşturucu](https://www.wikiod.com/tr/design-patterns/olusturucu-modeli#Oluşturucu Modeli / C# / Akıcı Arayüz)‘den esinlenerek, I Burada bir C++ sürümü yapmaya karar verdim. Bir E-posta nesnesi mutlaka çok karmaşık bir nesne değildir, ancak kalıbı gösterebilir.

#include <iostream>
#include <sstream>
#include <string>

using namespace std;

// Forward declaring the builder
class EmailBuilder;

class Email
{
  public:
    friend class EmailBuilder;  // the builder can access Email's privates
    
    static EmailBuilder make();
    
    string to_string() const {
        stringstream stream;
        stream << "from: " << m_from
               << "\nto: " << m_to
               << "\nsubject: " << m_subject
               << "\nbody: " << m_body;
        return stream.str();
    }
    
  private:
    Email() = default; // restrict construction to builder
    
    string m_from;
    string m_to;
    string m_subject;
    string m_body;
};

class EmailBuilder
{
  public:
    EmailBuilder& from(const string &from) {
        m_email.m_from = from;
        return *this;
    }
    
    EmailBuilder& to(const string &to) {
        m_email.m_to = to;
        return *this;
    }
    
    EmailBuilder& subject(const string &subject) {
        m_email.m_subject = subject;
        return *this;
    }
    
    EmailBuilder& body(const string &body) {
        m_email.m_body = body;
        return *this;
    }
    
    operator Email&&() {
        return std::move(m_email); // notice the move
    }
    
  private:
    Email m_email;
};

EmailBuilder Email::make()
{
    return EmailBuilder();
}

// Bonus example!
std::ostream& operator <<(std::ostream& stream, const Email& email)
{
    stream << email.to_string();
    return stream;
}


int main()
{
    Email mail = Email::make().from("me@mail.com")
                              .to("you@mail.com")
                              .subject("C++ builders")
                              .body("I like this API, don't you?");
                              
    cout << mail << endl;
}

C++‘ın eski sürümleri için, ‘std::move’ işlemi görmezden gelinebilir ve dönüştürme operatöründen && kaldırılabilir (ancak bu geçici bir kopya oluşturacaktır).

Oluşturucu, yerleşik e-postayı operatör E-postası&&() tarafından serbest bıraktığında işini bitirir. Bu örnekte, oluşturucu geçici bir nesnedir ve yok edilmeden önce e-postayı döndürür. Dönüştürme operatörü yerine Email EmailBuilder::build() {...} gibi açık bir işlem de kullanabilirsiniz.

İnşaatçıyı gezdirin #

Oluşturucu Modelinin sağladığı harika bir özellik, bir nesneyi birlikte oluşturmak için birkaç aktörü kullanma yeteneğidir. Bu, oluşturucuyu, her biri yapılı nesneye biraz daha fazla bilgi verecek olan diğer aktörlere geçirerek yapılır. Bu, bir tür sorgu oluştururken, filtreler ve diğer özellikler eklerken özellikle güçlüdür.

void add_addresses(EmailBuilder& builder)
{
    builder.from("me@mail.com")
           .to("you@mail.com");
}

void compose_mail(EmailBuilder& builder)
{
    builder.subject("I know the subject")
           .body("And the body. Someone else knows the addresses.");
}

int main()
{
    EmailBuilder builder;
    add_addresses(builder);
    compose_mail(builder);
    
    Email mail = builder;
    cout << mail << endl;
}

Tasarım varyantı: Değişken nesne #

Bu desenin tasarımını ihtiyaçlarınıza göre değiştirebilirsiniz. Bir varyant vereceğim.

Verilen örnekte Email nesnesi değişmezdir, yani özellikleri değiştirilemez çünkü bunlara erişim yoktur. Bu istenen bir özellikti. Oluşturulduktan sonra nesneyi değiştirmeniz gerekiyorsa, ona bazı ayarlayıcılar sağlamanız gerekir. Bu ayarlayıcılar oluşturucuda çoğaltılacağından, hepsini tek bir sınıfta yapmayı düşünebilirsiniz (artık oluşturucu sınıfına gerek yok). Bununla birlikte, ilk etapta inşa edilmiş nesneyi değiştirilebilir hale getirme ihtiyacını düşünürdüm.
