İfade şablonlarını gösteren temel bir örnek #

Bir ifade şablonu, çoğunlukla bilimsel hesaplamada kullanılan bir derleme zamanı optimizasyon tekniğidir. Ana amacı, gereksiz geçicilerden kaçınmak ve tek bir geçiş kullanarak döngü hesaplamalarını optimize etmektir (tipik olarak sayısal kümeler üzerinde işlemler gerçekleştirirken). İfade şablonları başlangıçta sayısal ‘Dizi’ veya ‘Matris’ türleri uygulanırken naif operatör aşırı yüklemesinin verimsizliklerini ortadan kaldırmak için tasarlandı. İfade şablonları için eşdeğer bir terminoloji, “The C++ Programming Language” adlı kitabının en son sürümünde onları “birleştirilmiş işlemler” olarak adlandıran Bjarne Stroustrup tarafından tanıtıldı.

İfade şablonlarına gerçekten dalmadan önce, onlara neden ihtiyaç duyduğunuzu anlamalısınız. Bunu göstermek için aşağıda verilen çok basit Matrix sınıfını düşünün:

template <typename T, std::size_t COL, std::size_t ROW>
class Matrix {
public:
    using value_type = T;

    Matrix() : values(COL * ROW) {}

    static size_t cols() { return COL; }
    static size_t rows() { return ROW; }

    const T& operator()(size_t x, size_t y) const { return values[y * COL + x]; }
    T& operator()(size_t x, size_t y) { return values[y * COL + x]; }

private:
    std::vector<T> values;
};

template <typename T, std::size_t COL, std::size_t ROW>
Matrix<T, COL, ROW>
operator+(const Matrix<T, COL, ROW>& lhs, const Matrix<T, COL, ROW>& rhs)
{
    Matrix<T, COL, ROW> result;

    for (size_t y = 0; y != lhs.rows(); ++y) {
        for (size_t x = 0; x != lhs.cols(); ++x) {
            result(x, y) = lhs(x, y) + rhs(x, y);
        }
    }
    return result;
}

Önceki sınıf tanımı göz önüne alındığında, artık aşağıdaki gibi Matrix ifadeleri yazabilirsiniz:

const std::size_t cols = 2000;
const std::size_t rows = 1000;

Matrix<double, cols, rows> a, b, c;

// initialize a, b & c
for (std::size_t y = 0; y != rows; ++y) {
    for (std::size_t x = 0; x != cols; ++x) {
        a(x, y) = 1.0;
        b(x, y) = 2.0;
        c(x, y) = 3.0;
    }
}  

Matrix<double, cols, rows> d = a + b + c;  // d(x, y) = 6

Yukarıda gösterildiği gibi, “operatör+()“yi aşırı yükleyebilmek, matrisler için doğal matematiksel gösterimi taklit eden bir gösterim sağlar.

Ne yazık ki, önceki uygulama, eşdeğer bir “el yapımı” versiyona kıyasla oldukça verimsizdir.

Nedenini anlamak için ‘Matris d = a + b + c’ gibi bir ifade yazdığınızda ne olduğunu düşünmelisiniz. Bu aslında ((a + b) + c) veya operatör+(operatör+(a, b), c) şeklinde genişler. Başka bir deyişle, ‘operator+()’ içindeki döngü iki kez yürütülürken, tek geçişte kolayca gerçekleştirilebilirdi. Bu aynı zamanda performansı daha da düşüren 2 geçicinin oluşturulmasına neden olur. Özünde, matematiksel karşılığına yakın bir gösterim kullanma esnekliği ekleyerek, “Matrix” sınıfını da oldukça verimsiz hale getirdiniz.

Örneğin, operatör aşırı yüklemesi olmadan, tek bir geçiş kullanarak çok daha verimli bir Matrix toplamı uygulayabilirsiniz:

template<typename T, std::size_t COL, std::size_t ROW>
Matrix<T, COL, ROW> add3(const Matrix<T, COL, ROW>& a,
                         const Matrix<T, COL, ROW>& b,
                         const Matrix<T, COL, ROW>& c)
{
    Matrix<T, COL, ROW> result;
    for (size_t y = 0; y != ROW; ++y) {
        for (size_t x = 0; x != COL; ++x) {
            result(x, y) = a(x, y) + b(x, y) + c(x, y);
        }
    }
    return result;
}

Ancak önceki örneğin kendi dezavantajları vardır çünkü Matrix sınıfı için çok daha dolambaçlı bir arayüz oluşturur (Matrix::add2(), Matrix::AddMultiply() ve benzeri yöntemleri göz önünde bulundurmanız gerekir) .

Bunun yerine bir adım geriye gidelim ve operatör aşırı yüklemesini daha verimli bir şekilde performans gösterecek şekilde nasıl uyarlayabileceğimizi görelim.

Sorun, ‘Matrix d = a + b + c’ ifadesinin, daha siz tüm ifade ağacını oluşturma fırsatı bulamadan çok “hevesle” değerlendirilmesinden kaynaklanmaktadır. Başka bir deyişle, gerçekten elde etmek istediğiniz şey, “a + b + c"yi tek geçişte değerlendirmek ve yalnızca bir kez ortaya çıkan ifadeyi “d"ye atamanız gerekir.

İfade şablonlarının arkasındaki temel fikir budur: iki Matrix örneğinin eklenmesinin sonucunu hemen değerlendirmek için “operatör+()” yerine, tüm ifade ağacı oluşturulduktan sonra gelecekteki değerlendirme için bir “ifade şablonu” döndürür.

Örneğin, 2 türün toplamına karşılık gelen bir ifade şablonu için olası bir uygulama:

template <typename LHS, typename RHS>
class MatrixSum
{
public:
    using value_type = typename LHS::value_type;

    MatrixSum(const LHS& lhs, const RHS& rhs) : rhs(rhs), lhs(lhs) {}
    
    value_type operator() (int x, int y) const  {
        return lhs(x, y) + rhs(x, y);
    }
private:
    const LHS& lhs;
    const RHS& rhs;
};

Ve işte operatör+()nın güncellenmiş versiyonu

template <typename LHS, typename RHS>
MatrixSum<LHS, RHS> operator+(const LHS& lhs, const LHS& rhs) {
    return MatrixSum<LHS, RHS>(lhs, rhs);
}

Gördüğünüz gibi, ‘operator+()’ artık 2 Matrix örneği eklemenin sonucunun (başka bir Matrix örneği olurdu) sonucunun bir “istekli değerlendirmesini” değil, bunun yerine toplama işlemini temsil eden bir ifade şablonunu döndürür. Unutulmaması gereken en önemli nokta, ifadenin henüz değerlendirilmemiş olmasıdır. Yalnızca işlenenlerine referanslar tutar.

Aslında, hiçbir şey sizi “MatrixSum<>” ifade şablonunu aşağıdaki gibi somutlaştırmaktan alıkoyamaz:

MatrixSum<Matrix<double>, Matrix<double> > SumAB(a, b);

Ancak daha sonraki bir aşamada, toplamanın sonucuna gerçekten ihtiyacınız olduğunda, ’d = a + b’ ifadesini aşağıdaki gibi değerlendirebilirsiniz:

for (std::size_t y = 0; y != a.rows(); ++y) {
    for (std::size_t x = 0; x != a.cols(); ++x) {
        d(x, y) = SumAB(x, y);
    }
}

Gördüğünüz gibi, bir ifade şablonu kullanmanın bir başka yararı da, temelde “a” ve “b"nin toplamını değerlendirmeyi başarmış ve bunu tek geçişte “d"ye atamış olmanızdır.

Ayrıca, birden fazla ifade şablonunu birleştirmenizi hiçbir şey engelleyemez. Örneğin, “a + b + c” aşağıdaki ifade şablonuyla sonuçlanır:

MatrixSum<MatrixSum<Matrix<double>, Matrix<double> >, Matrix<double> > SumABC(SumAB, c);

Ve burada yine tek bir geçiş kullanarak nihai sonucu değerlendirebilirsiniz:

for (std::size_t y = 0; y != a.rows(); ++y) {
    for (std::size_t x = 0; x != a.cols(); ++x) {
        d(x, y) = SumABC(x, y);
    }
}

Son olarak, bulmacanın son parçası, ifade şablonunuzu gerçekten ‘Matrix’ sınıfına eklemektir. Bu, esasen, daha önce “manuel” yaptığınız gibi, ifade şablonunu bir argüman olarak alan ve tek geçişte değerlendiren ‘Matrix::operator=()’ için bir uygulama sağlayarak elde edilir:

template <typename T, std::size_t COL, std::size_t ROW>
class Matrix {
public:
    using value_type = T;

    Matrix() : values(COL * ROW) {}

    static size_t cols() { return COL; }
    static size_t rows() { return ROW; }

    const T& operator()(size_t x, size_t y) const { return values[y * COL + x]; }
    T& operator()(size_t x, size_t y) { return values[y * COL + x]; }

    template <typename E>
    Matrix<T, COL, ROW>& operator=(const E& expression) {
        for (std::size_t y = 0; y != rows(); ++y) {
            for (std::size_t x = 0; x != cols(); ++x) {
                values[y * COL + x] = expression(x, y);
            }
        }
        return *this;
    }

private:
    std::vector<T> values;
};

Öğe bazında cebirsel ifadelerde temel ifade şablonları #

Giriş ve motivasyon

İfade şablonları (aşağıda ET olarak gösterilir), bazen oldukça pahalı ifadelerin hesaplamalarını hızlandırmak için kullanılan güçlü bir şablon meta programlama tekniğidir. Örneğin lineer cebir kitaplıklarının uygulanması gibi farklı alanlarda yaygın olarak kullanılmaktadır.

Bu örnek için, lineer cebirsel hesaplamaların bağlamını düşünün. Daha spesifik olarak, yalnızca öğe bazında işlemleri içeren hesaplamalar. Bu tür hesaplamalar ET’lerin en temel uygulamalarıdır ve ET’lerin dahili olarak nasıl çalıştığına dair iyi bir giriş görevi görürler.

Motive edici bir örneğe bakalım. Aşağıdaki ifadenin hesaplanmasını düşünün:

Vector vec_1, vec_2, vec_3;

// Initializing vec_1, vec_2 and vec_3.

Vector result = vec_1 + vec_2*vec_3;

Burada basitlik adına, “Vektör” sınıfının ve işlem + (vektör artı: eleman bazında artı işlem) ve işlem * (burada vektör iç çarpımı anlamına gelir: ayrıca eleman bazında işlem anlamına gelir) her ikisinin de doğru şekilde uygulandığını varsayacağım. , nasıl olması gerektiği gibi, matematiksel olarak.

ET’ler (veya diğer benzer teknikler) kullanılmayan geleneksel bir uygulamada, nihai “sonucu” elde etmek için “Vektör” örneklerinin en az beş yapısı yer alır:

  1. “vec_1”, “vec_2” ve “vec_3” ile ilgili üç örnek.
  2. “_tmp = vec_2*vec_3;” sonucunu temsil eden geçici bir “Vector” örneği “_tmp”.
  3. Son olarak, dönüş değeri optimizasyonunun doğru kullanımıyla, ‘sonuç = vec_1 + _tmp;’ içindeki nihai ‘sonucun’ oluşturulması.

ET’ler kullanılarak uygulama, 2’de geçici ‘Vector _tmp’ oluşturulmasını **ortadan kaldırabilir, böylece ‘Vector’ örneklerinin yalnızca dört yapısı kalır. Daha da ilginç olanı, daha karmaşık olan aşağıdaki ifadeyi düşünün:

Vector result = vec_1 + (vec_2*vec3 + vec_1)*(vec_2 + vec_3*vec_1);

Ayrıca toplamda ‘Vector’ örneklerinin dört yapısı olacaktır: ‘vec_1, vec_2, vec_3’ ve ‘result’. Başka bir deyişle, bu örnekte, yalnızca eleman bazında işlemlerin dahil olduğu, ara hesaplamalardan hiçbir geçici nesne oluşturulmayacağı garanti edilir.

ET’ler nasıl çalışır

Temel olarak konuşursak, herhangi bir cebirsel hesaplama için ET’ler iki yapı taşından oluşur:

  1. Saf cebirsel ifadeler (PAE): cebirsel ifadelerin vekilleri / soyutlamalarıdır. Saf bir cebir gerçek hesaplamalar yapmaz, bunlar sadece hesaplama iş akışının soyutlamaları/modelleridir. Bir PAE, herhangi bir cebirsel ifadenin girdisi veya çıktısı için bir model olabilir. PAE örneklerinin kopyalanması genellikle ucuz olarak kabul edilir.
  2. Tembel değerlendirmeler: gerçek hesaplamaların uygulanmasıdır. Aşağıdaki örnekte, yalnızca öğe bazında işlemleri içeren ifadeler için, tembel değerlendirmelerin, dizinlenmiş erişim işlemi içindeki gerçek hesaplamaları nihai sonuç üzerinde uygulayabildiğini, böylece isteğe bağlı bir değerlendirme şeması oluşturduğunu göreceğiz: bir hesaplama yapılmaz sadece nihai sonuca ulaşılırsa / istenirse.

Peki, özellikle bu örnekte ET’leri nasıl uygularız? Şimdi üzerinden geçelim.

Her zaman aşağıdaki kod parçacığını göz önünde bulundurun:

Vector vec_1, vec_2, vec_3;

// Initializing vec_1, vec_2 and vec_3.

Vector result = vec_1 + vec_2*vec_3;

Sonuç hesaplanacak ifade, iki alt ifadeye daha da ayrıştırılabilir:

  1. Bir vektör artı ifadesi (plus_ifade olarak gösterilir)
  2. Bir vektör iç çarpım ifadesi (innerprod_expr olarak gösterilir).

ET’ler şunları yapar:

  • Her bir alt ifadeyi hemen hesaplamak yerine, ET’ler önce tüm ifadeyi bir grafik yapı kullanarak modeller. Grafikteki her düğüm bir PAE‘yi temsil eder. Düğümlerin uç bağlantısı, gerçek hesaplama akışını temsil eder. Böylece yukarıdaki ifade için aşağıdaki grafiği elde ederiz:

            result = plus_expr( vec_1, innerprod_expr(vec_2, vec_3) )
           /   \
          /     \
         /       \
        /   innerprod_expr( vec_2, vec_3 )
       /         /  \
      /         /    \
     /         /      \
  vec_1     vec_2    vec_3

  • Son hesaplama grafik hiyerarşisine bakılarak yapılır: burada yalnızca eleman bazında işlemlerle uğraştığımız için, sonuç'taki indekslenmiş her değerin hesaplanması **bağımsız olarak yapılabilir* *: "sonucun" nihai değerlendirmesi, "sonucun" her bir öğesinin **eleman bazında değerlendirilmesine** tembelce ertelenebilir. Başka bir deyişle, bir "sonuç" öğesinin hesaplanması, "elem_res", "vec_1" ('elem_1'), 'vec_2' ('elem_2') ve 'vec_3' ('elem_3') içindeki karşılık gelen öğeler kullanılarak ifade edilebilir. ) olarak:

     elem_res = elem_1 + elem_2*elem_3;

bu nedenle, ara iç çarpım sonucunu saklamak için geçici bir ‘Vektör’ oluşturmaya gerek yoktur: bir öğe için tüm hesaplama birlikte yapılabilir ve dizinlenmiş erişim işlemi içinde kodlanabilir.

#ifndef EXPR_VEC
# define EXPR_VEC

# include <vector>
# include <cassert>
# include <utility>
# include <iostream>
# include <algorithm>
# include <functional>

///
/// This is a wrapper for std::vector. It's only purpose is to print out a log when a
/// vector constructions in called.
/// It wraps the indexed access operator [] and the size() method, which are 
/// important for later ETs implementation.
///

// std::vector wrapper.
template<typename ScalarType> class Vector
{
public:
  explicit Vector() { std::cout << "ctor called.\n"; };
  explicit Vector(int size): _vec(size) { std::cout << "ctor called.\n"; };
  explicit Vector(const std::vector<ScalarType> &vec): _vec(vec)
  { std::cout << "ctor called.\n"; };
  
  Vector(const Vector<ScalarType> & vec): _vec{vec()}
  { std::cout << "copy ctor called.\n"; };
  Vector(Vector<ScalarType> && vec): _vec(std::move(vec()))
  { std::cout << "move ctor called.\n"; };

  Vector<ScalarType> & operator=(const Vector<ScalarType> &) = default;
  Vector<ScalarType> & operator=(Vector<ScalarType> &&) = default;

  decltype(auto) operator[](int indx) { return _vec[indx]; }
  decltype(auto) operator[](int indx) const { return _vec[indx]; }

  decltype(auto) operator()() & { return (_vec); };        
  decltype(auto) operator()() const & { return (_vec); };  
  Vector<ScalarType> && operator()() && { return std::move(*this); }

  int size() const { return _vec.size(); }
  
private:
  std::vector<ScalarType> _vec;
};

///
/// These are conventional overloads of operator + (the vector plus operation)
/// and operator * (the vector inner product operation) without using the expression
/// templates. They are later used for bench-marking purpose.
///

// + (vector plus) operator.
template<typename ScalarType>
auto operator+(const Vector<ScalarType> &lhs, const Vector<ScalarType> &rhs)
{
  assert(lhs().size() == rhs().size() &&
         "error: ops plus -> lhs and rhs size mismatch.");
  
  std::vector<ScalarType> _vec;
  _vec.resize(lhs().size());
  std::transform(std::cbegin(lhs()), std::cend(lhs()),
                 std::cbegin(rhs()), std::begin(_vec),
                 std::plus<>());
  return Vector<ScalarType>(std::move(_vec));
}

// * (vector inner product) operator.
template<typename ScalarType>
auto operator*(const Vector<ScalarType> &lhs, const Vector<ScalarType> &rhs)
{
  assert(lhs().size() == rhs().size() &&
         "error: ops multiplies -> lhs and rhs size mismatch.");
  
  std::vector<ScalarType> _vec;
  _vec.resize(lhs().size());
  std::transform(std::cbegin(lhs()), std::cend(lhs()),
                 std::cbegin(rhs()), std::begin(_vec),
                 std::multiplies<>());
  return Vector<ScalarType>(std::move(_vec));
}

#endif //!EXPR_VEC

Dosya ifade.hh : eleman bazında işlemler için ifade şablonlarının uygulanması (vektör artı ve vektör iç çarpımı) #

Bölümlere ayıralım.

  1. Bölüm 1, tüm ifadeler için bir temel sınıf uygular. Merakla Tekrarlanan Şablon Kalıbını (CRTP) kullanır.

  2. Bölüm 2, ilk PAE‘yi uygular: hesaplama için gerçek girdi değeri içeren bir girdi veri yapısının yalnızca bir sarmalayıcısı (sabit referansı) olan bir terminal.

  3. Bölüm 3, daha sonra vector_plus ve vector_innerprod için kullanılan bir sınıf şablonu olan ikinci PAE: ikili_operasyon‘u uygular. İşlem türü, sol taraf PAE ve sağ taraf PAE tarafından parametrelendirilir. Gerçek hesaplama, dizinlenmiş erişim operatöründe kodlanmıştır.

  4. Bölüm 4, vector_plus ve vector_innerprod işlemlerini eleman bazında işlem olarak tanımlar. Ayrıca PAE’ler için + ve * operatörünü de aşırı yükler: öyle ki bu iki işlem de PAE döndürür.

    #ifndef EXPR_EXPR

    define EXPR_EXPR #

    /// Fwd declaration. template class Vector;

    namespace expr {

    /// —————————————– /// /// Section 1. /// /// The first section is a base class template for all kinds of expression. It
    /// employs the Curiously Recurring Template Pattern, which enables its instantiation /// to any kind of expression structure inheriting from it. /// /// —————————————–

    /// Base class for all expressions. template class expr_base { public: const Expr& self() const { return static_cast<const Expr&>(*this); } Expr& self() { return static_cast<Expr&>(*this); }

    protected: explicit expr_base() {}; int size() const { return self().size_impl(); } auto operator[](int indx) const { return self().at_impl(indx); } auto operator()() const { return self()(); }; };

    /// —————————————– /// /// The following section 2 & 3 are abstractions of pure algebraic expressions (PAE). /// Any PAE can be converted to a real object instance using operator(): it is in /// this conversion process, where the real computations are done.

    /// /// Section 2. Terminal /// /// A terminal is an abstraction wrapping a const reference to the Vector data /// structure. It inherits from expr_base, therefore providing a unified interface /// wrapping a Vector into a PAE. /// /// It provides the size() method, indexed access through at_impl() and a conversion /// to referenced object through () operator. /// /// It might no be necessary for user defined data structures to have a terminal /// wrapper, since user defined structure can inherit expr_base, therefore eliminates /// the need to provide such terminal wrapper. /// /// —————————————–

    /// Generic wrapper for underlying data structure. template class terminal: expr_base<terminal> { public: using base_type = expr_base<terminal>; using base_type::size; using base_type::operator[]; friend base_type;

     explicit terminal(const DataType &val): _val(val) {}
 int size_impl() const { return _val.size(); };
 auto at_impl(int indx) const { return _val[indx]; };
 decltype(auto) operator()() const { return (_val); }

    private: const DataType &_val; };

    /// —————————————– /// /// Section 3. Binary operation expression. /// /// This is a PAE abstraction of any binary expression. Similarly it inherits from /// expr_base. /// /// It provides the size() method, indexed access through at_impl() and a conversion /// to referenced object through () operator. Each call to the at_impl() method is /// a element wise computation. /// /// —————————————–

    /// Generic wrapper for binary operations (that are element-wise). template<typename Ops, typename lExpr, typename rExpr> class binary_ops: public expr_base<binary_ops<Ops,lExpr,rExpr» { public: using base_type = expr_base<binary_ops<Ops,lExpr,rExpr»; using base_type::size; using base_type::operator[]; friend base_type;

     explicit binary_ops(const Ops &ops, const lExpr &lxpr, const rExpr &rxpr)
   : _ops(ops), _lxpr(lxpr), _rxpr(rxpr) {};
 int size_impl() const { return _lxpr.size(); };

 /// This does the element-wise computation for index indx.
 auto at_impl(int indx) const { return _ops(_lxpr[indx], _rxpr[indx]); };

 /// Conversion from arbitrary expr to concrete data type. It evaluates
 /// element-wise computations for all indices.
 template<typename DataType> operator DataType()
 {
   DataType _vec(size());
   for(int _ind = 0; _ind < _vec.size(); ++_ind)
     _vec[_ind] = (*this)[_ind];
   return _vec;
 }

    private: /// Ops and expr are assumed cheap to copy. Ops _ops; lExpr _lxpr; rExpr _rxpr; };

    /// —————————————– /// Section 4. /// /// The following two structs defines algebraic operations on PAEs: here only vector /// plus and vector inner product are implemented. /// /// First, some element-wise operations are defined : in other words, vec_plus and /// vec_prod acts on elements in Vectors, but not whole Vectors. /// /// Then, operator + & * are overloaded on PAEs, such that: + & * operations on PAEs
    /// also return PAEs. /// /// —————————————–

    /// Element-wise plus operation. struct vec_plus_t { constexpr explicit vec_plus_t() = default; template<typename LType, typename RType> auto operator()(const LType &lhs, const RType &rhs) const { return lhs+rhs; } };

    /// Element-wise inner product operation. struct vec_prod_t { constexpr explicit vec_prod_t() = default; template<typename LType, typename RType> auto operator()(const LType &lhs, const RType &rhs) const { return lhs*rhs; } };

    /// Constant plus and inner product operator objects. constexpr vec_plus_t vec_plus{}; constexpr vec_prod_t vec_prod{};

    /// Plus operator overload on expressions: return binary expression. template<typename lExpr, typename rExpr> auto operator+(const lExpr &lhs, const rExpr &rhs) { return binary_ops<vec_plus_t,lExpr,rExpr>(vec_plus,lhs,rhs); }

    /// Inner prod operator overload on expressions: return binary expression. template<typename lExpr, typename rExpr> auto operator*(const lExpr &lhs, const rExpr &rhs) { return binary_ops<vec_prod_t,lExpr,rExpr>(vec_prod,lhs,rhs); }

    } //!expr

    #endif //!EXPR_EXPR

Dezavantajlar ve uyarılar

  • ET’lerin bariz bir dezavantajı, öğrenme eğrisi, uygulamanın karmaşıklığı ve kod bakım zorluğudur. Yalnızca eleman bazında işlemlerin dikkate alındığı yukarıdaki örnekte, uygulama, her hesaplamada daha karmaşık cebirsel ifadelerin meydana geldiği ve eleman bazında bağımsızlığın artık geçerli olmadığı (örneğin matris çarpımı) gerçek dünyada, zaten çok büyük miktarda kazan levhası içermektedir. ), zorluk üstel olacaktır.

  • ET’leri kullanmanın bir başka uyarısı, “auto” anahtar kelimesiyle iyi oynamalarıdır. Yukarıda bahsedildiği gibi, PAE’ler esasen vekillerdir: ve vekiller temelde “otomatik” ile iyi oynamazlar. Aşağıdaki örneği göz önünde bulundurun:

      auto result = ...;                // Some expensive expression: 
                                    // auto returns the expr graph, 
                                    // NOT the computed value.
  for(auto i = 0; i < 100; ++i)
      ScalrType value = result* ... // Some other expensive computations using result.

Burada for döngüsünün her yinelemesinde, hesaplanan değer yerine ifade grafiği for döngüsüne iletildiği için sonuç yeniden değerlendirilecektir.

ET’leri uygulayan mevcut kütüphaneler

  • boost::proto, kendi ifadeleriniz için kendi kurallarınızı ve gramerlerinizi tanımlamanıza ve ET’leri kullanarak yürütmenize izin veren güçlü bir kitaplıktır.
  • Eigen, ET’leri kullanarak çeşitli cebirsel hesaplamaları verimli bir şekilde uygulayan lineer cebir için bir kitaplıktır.
