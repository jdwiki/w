Aşırı yük çözünürlüğü birkaç farklı durumda gerçekleşir

  • Adlandırılmış aşırı yüklenmiş işlevlere çağrılar. Adaylar, isim aramasıyla bulunan tüm fonksiyonlardır.
  • Sınıf nesnesine çağrılar. Adaylar genellikle sınıfın tüm aşırı yüklenmiş işlev çağrısı operatörleridir.
  • Operatör kullanımı. Adaylar, ad alanı kapsamındaki aşırı yüklenmiş işleç işlevleri, sol sınıf nesnesindeki (varsa) aşırı yüklenmiş işleç işlevleri ve yerleşik işleçlerdir.
  • Bir başlatma için çağrılacak doğru dönüştürme işleci işlevini veya oluşturucuyu bulmak için aşırı yük çözünürlüğü
  • Liste dışı doğrudan başlatma için (Sınıf c(değer)), adaylar Sınıfın kurucularıdır.
  • Liste dışı kopya başlatma (‘Sınıf c = değer’) ve kullanıcı tanımlı bir dönüştürme sırasında çağrılacak kullanıcı tanımlı dönüştürme işlevini bulmak için. Adaylar, “Sınıf"ın kurucularıdır ve eğer kaynak bir sınıf nesnesiyse, dönüştürme operatörü çalışır.
  • Sınıf olmayan bir nesnenin bir sınıf nesnesinden başlatılması için (Sınıf dışı c = classObject). Adaylar, başlatıcı nesnesinin dönüştürme işleci işlevleridir.
  • Sınıf, doğrudan “r"ye bağlanabilen değerler veren dönüştürme işleci işlevlerine sahip olduğunda, bir sınıf nesnesiyle (R &r = classObject) bir referans başlatmak için. Adaylar, bu tür dönüştürme operatörü işlevleridir.
  • Toplu olmayan bir sınıf nesnesinin (Sınıf c{1, 2, 3}) liste başlatması için adaylar, aşırı yük çözümlemesinden ilk geçiş için başlatıcı listesi oluşturucularıdır. Bu uygun bir aday bulamazsa, ‘Sınıf’ yapıcıları aday olarak, aşırı yük çözümlemesinden ikinci bir geçiş yapılır.

Argümanın parametre maliyetine göre sınıflandırılması #

Aşırı yük çözünürlüğü, bir parametreye bir argüman iletme maliyetini “diziler” adı verilen dört farklı kategoriden birine böler. Her dizi sıfır, bir veya birkaç dönüşüm içerebilir

  • Standart dönüştürme sırası

    void f(int a); f(42);

  • Kullanıcı tanımlı dönüştürme sırası

    void f(std::string s); f("hello");

  • Üç nokta dönüştürme sırası

    void f(...); f(42);

  • Başlatma sırasını listele

    void f(std::vector<int> v); f({1, 2, 3});

Genel ilke, Standart dönüştürme dizilerinin en ucuzu olması, ardından kullanıcı tanımlı dönüştürme dizilerinin ve ardından üç noktalı dönüştürme dizilerinin gelmesidir.

Özel bir durum, bir dönüştürme oluşturmayan liste başlatma dizisidir (bir başlatıcı listesi, tür içeren bir ifade değildir). Maliyeti, parametre tipine ve başlatıcı listesinin biçimine bağlı olarak diğer üç dönüşüm dizisinden birine eşdeğer olarak tanımlanarak belirlenir.

Tam eşleşme #

Parametre türleri için dönüştürme gerektirmeyen bir aşırı yükleme veya yalnızca tam eşleşme olarak kabul edilen türler arasında gereken dönüştürmeler, çağırmak için başka dönüştürmeler gerektiren bir aşırı yüklemeye tercih edilir.

void f(int x);
void f(double x);
f(42); // calls f(int)

Bir bağımsız değişken aynı türdeki bir başvuruya bağlandığında, başvuru daha fazla özgeçmiş nitelikli olsa bile eşleşmenin dönüştürme gerektirmediği kabul edilir.

void f(int& x);
void f(double x);
int x = 42;
f(x); // argument type is int; exact match with int&

void g(const int& x);
void g(int x);
g(x); // ambiguous; both overloads give exact match

Aşırı yük çözümlemesi amacıyla, “T” dizisi tipinin “T” işaretçisi türüyle tam olarak eşleştiği kabul edilir ve “T” işlev türünün, işlev işaretçi türüyle tam olarak eşleştiği kabul edilir. T*, her ikisi de dönüşüm gerektirse de.

void f(int* p);
void f(void* p);

void g(int* p);
void g(int (&p)[100]);

int a[100];
f(a); // calls f(int*); exact match with array-to-pointer conversion
g(a); // ambiguous; both overloads give exact match

İsim arama ve erişim kontrolü #

Aşırı yük çözünürlüğü, ad aramasından sonra oluşur. Bu, ad aramasını kaybederse, aşırı yük çözünürlüğü tarafından daha iyi eşleşen bir işlevin seçilmeyeceği anlamına gelir:

void f(int x);
struct S {
    void f(double x);
    void g() { f(42); } // calls S::f because global f is not visible here,
                        // even though it would be a better match
};

Aşırı yük çözünürlüğü erişim kontrolünden önce gerçekleşir. Erişilemeyen bir işlev, erişilebilir bir işlevden daha iyi bir eşleşme ise aşırı yük çözünürlüğü ile seçilebilir.

class C {
  public:
    static void f(double x);
  private:
    static void f(int x);
};
C::f(42); // Error! Calls private C::f(int) even though public C::f(double) is viable.

Benzer şekilde, aşırı yük çözünürlüğü, ortaya çıkan aramanın “açık” ile ilgili olarak iyi biçimlendirilmiş olup olmadığını kontrol etmeden gerçekleşir:

struct X {
    explicit X(int );
    X(char );
};

void foo(X );
foo({4}); // X(int) is better much, but expression is 
          // ill-formed because selected constructor is explicit

Yönlendirme Referansında Aşırı Yükleme #

Çok iyi eşleşebileceğinden bir yönlendirme referansı aşırı yüklemesi sağlarken çok dikkatli olmalısınız:

struct A {
    A() = default;           // #1
    A(A const& ) = default;  // #2

    template <class T>
    A(T&& );                 // #3
};

Buradaki amaç, “A"nın kopyalanabilir olması ve başka bir üyeyi başlatabilecek bu diğer kurucuya sahip olmamızdı. Yine de:

A a;     // calls #1
A b(a);  // calls #3!

İnşaat çağrısı için iki uygun eşleşme var:

A(A const& ); // #2
A(A& );       // #3, with T = A&

Her ikisi de Tam Eşleşmedir, ancak “#3”, “#2"den daha az cv nitelikli bir nesneye referans alır, bu nedenle daha iyi standart dönüştürme sırasına sahiptir ve en uygun işlevdir.

Buradaki çözüm, bu kurucuları her zaman kısıtlamaktır (örneğin, SFINAE kullanarak):

template <class T,
    class = std::enable_if_t<!std::is_convertible<std::decay_t<T>*, A*>::value>
    >
A(T&& );

Buradaki tip özelliği, herhangi bir ‘A’ veya ‘A’dan açıkça türetilmiş herhangi bir ‘A’ veya sınıfı dikkate almamaktır; bu, bu kurucuyu daha önce açıklanan örnekte kötü biçimlendirecek (ve dolayısıyla aşırı yük kümesinden çıkarılacaktır). Sonuç olarak, kopya oluşturucu çağrılır - bu bizim istediğimiz şeydi.

Aritmetik promosyonlar ve dönüşümler #

Bir tamsayı türünü karşılık gelen yükseltilmiş türe dönüştürmek, onu başka bir tamsayı türüne dönüştürmekten daha iyidir.

void f(int x);
void f(short x);
signed char c = 42;
f(c); // calls f(int); promotion to int is better than conversion to short
short s = 42;
f(s); // calls f(short); exact match is better than promotion to int

Bir “kayan nokta"yı “çift"e yükseltmek, onu başka bir kayan nokta türüne dönüştürmekten daha iyidir.

void f(double x);
void f(long double x);
f(3.14f); // calls f(double); promotion to double is better than conversion to long double

Promosyonlar dışındaki aritmetik dönüşümler birbirinden ne daha iyi ne de daha kötüdür.

void f(float x);
void f(long double x);
f(3.14); // ambiguous

void g(long x);
void g(long double x);
g(42); // ambiguous
g(3.14); // ambiguous

Bu nedenle, herhangi bir standart türdeki integral veya kayan nokta argümanlarıyla bir ‘f’ işlevi çağrılırken belirsizlik olmamasını sağlamak için, toplam sekiz aşırı yüklemeye ihtiyaç vardır, böylece her olası argüman türü için ya bir aşırı yükleme tam olarak eşleşir veya yükseltilmiş argüman türüyle benzersiz aşırı yükleme seçilir.

void f(int x);
void f(unsigned int x);
void f(long x);
void f(unsigned long x);
void f(long long x);
void f(unsigned long long x);
void f(double x);
void f(long double x);

Sabitlik ve değişkenlik üzerine aşırı yükleme #

Mümkünse, bir “T*” parametresine bir işaretçi argümanı iletmek, onu bir “const T*” parametresine geçirmekten daha iyidir.

struct Base {};
struct Derived : Base {};
void f(Base* pb);
void f(const Base* pb);
void f(const Derived* pd);
void f(bool b);

Base b;
f(&b); // f(Base*) is better than f(const Base*)
Derived d;
f(&d); // f(const Derived*) is better than f(Base*) though;
       // constness is only a "tie-breaker" rule

Benzer şekilde, mümkünse bir “T&” parametresine bir argüman iletmek, her ikisinin de tam eşleşme sıralamasına sahip olsa bile onu bir “const T&” parametresine iletmekten daha iyidir.

void f(int& r);
void f(const int& r);
int x;
f(x); // both overloads match exactly, but f(int&) is still better
const int y = 42;
f(y); // only f(const int&) is viable

Bu kural, const olmayan nesnelere değişken erişime ve const nesnelere değişmez erişime izin vermenin önemli olduğu, const nitelikli üye işlevleri için de geçerlidir.

class IntVector {
  public:
    // ...
    int* data() { return m_data; }
    const int* data() const { return m_data; }
  private:
    // ...
    int* m_data;
};
IntVector v1;
int* data1 = v1.data();       // Vector::data() is better than Vector::data() const;
                              // data1 can be used to modify the vector's data
const IntVector v2;
const int* data2 = v2.data(); // only Vector::data() const is viable;
                              // data2 can't be used to modify the vector's data

Aynı şekilde, geçici bir aşırı yükleme, geçici olmayan bir aşırı yüklemeden daha az tercih edilecektir.

class AtomicInt {
  public:
    // ...
    int load();
    int load() volatile;
  private:
    // ...
};
AtomicInt a1;
a1.load(); // non-volatile overload preferred; no side effect
volatile AtomicInt a2;
a2.load(); // only volatile overload is viable; side effect
static_cast<volatile AtomicInt&>(a1).load(); // force volatile semantics for a1

Aşırı Yük Çözme Adımları #

Aşırı yük çözümleme adımları şunlardır:

  1. Ad arama yoluyla aday işlevleri bulun. Niteliksiz aramalar hem normal niteliksiz arama hem de bağımsız değişkene bağlı arama (varsa) gerçekleştirir.

  2. Aday işlevler kümesini bir dizi uygulanabilir işleve göre filtreleyin. İşlevin çağrıldığı bağımsız değişkenler ile işlevin aldığı parametreler arasında örtük bir dönüştürme dizisi bulunan geçerli bir işlev.

     void f(char);          // (1)
 void f(int ) = delete; // (2)
 void f();              // (3)
 void f(int& );         // (4)

 f(4); // 1,2 are viable (even though 2 is deleted!) 
       // 3 is not viable because the argument lists don't match
       // 4 is not viable because we cannot bind a temporary to 
       //     a non-const lvalue reference

  3. En uygun adayı seçin. Geçerli bir “F1” işlevi, “F1"deki her bağımsız değişken için örtük dönüştürme dizisi “F2"deki karşılık gelen örtük dönüştürme dizisinden daha kötü değilse, başka bir geçerli “F2” işlevinden daha iyi bir işlevdir ve…:

    3.1. For some argument, the implicit conversion sequence for that argument in F1 is a better conversion sequence than for that argument in F2, or

     void f(int );  // (1)
 void f(char ); // (2)

 f(4);  // call (1), better conversion sequence

    3.2. In a user-defined conversion, the standard conversion sequence from the return of F1 to the destination type is a better conversion sequence than that of the return type of F2, or

     struct A 
 {
     operator int();
     operator double();
 } a;

 int i = a; // a.operator int() is better than a.operator double() and a conversion
 float f = a; // ambiguous

    3.3. In a direct reference binding, F1 has the same kind of reference by F2 is not, or

     struct A 
 {
     operator X&();  // #1
     operator X&&(); // #2
 };
 A a;
 X& lx = a;  // calls #1
 X&& rx = a; // calls #2

    3.4. F1 is not a function template specialization, but F2 is, or

     template <class T> void f(T ); // #1
 void f(int );                  // #2

 f(42); // calls #2, the non-template

    3.5. F1 and F2 are both function template specializations, but F1 is more specialized than F2.

     template <class T> void f(T );  // #1
 template <class T> void f(T* ); // #2

 int* p;
 f(p); // calls #2, more specialized

Buradaki sıralama önemlidir. Daha iyi dönüşüm sırası kontrolü, şablona karşı şablon olmayan kontrolden önce gerçekleşir. Bu, yönlendirme referansında aşırı yükleme ile ilgili yaygın bir hataya yol açar:

struct A {
    A(A const& ); // #1
    
    template <class T>
    A(T&& );      // #2, not constrained
};

A a;
A b(a); // calls #2!
        // #1 is not a template but #2 resolves to
        // A(A& ), which is a less cv-qualified reference than #1
        // which makes it a better implicit conversion sequence

Sonunda en uygun tek bir aday yoksa, çağrı belirsizdir:

void f(double ) { }
void f(float ) { }

f(42); // error: ambiguous

Bir sınıf hiyerarşisi içinde aşırı yükleme #

Aşağıdaki örnekler bu sınıf hiyerarşisini kullanacaktır:

struct A { int m; };
struct B : A {};
struct C : B {};

Türetilmiş sınıf türünden temel sınıf türüne dönüştürme, kullanıcı tanımlı dönüştürmelere tercih edilir. Bu, değere veya referansa göre geçerken ve işaretçiden türetilene işaretçiden tabana dönüştürülürken geçerlidir.

struct Unrelated {
    Unrelated(B b);
};
void f(A a);
void f(Unrelated u);
B b;
f(b); // calls f(A)

Türetilmiş sınıftan temel sınıfa bir işaretçi dönüşümü de “void*“e dönüştürmekten daha iyidir.

void f(A* p);
void f(void* p);
B b;
f(&b); // calls f(A*)

Aynı kalıtım zinciri içinde birden fazla aşırı yükleme varsa, en çok türetilmiş temel sınıf aşırı yüklemesi tercih edilir. Bu, sanal gönderme ile benzer bir ilkeye dayanır: “en uzmanlaşmış” uygulama seçilir. Bununla birlikte, aşırı yük çözünürlüğü her zaman derleme zamanında gerçekleşir ve hiçbir zaman dolaylı olarak aşağı atılmaz.

void f(const A& a);
void f(const B& b);
C c;
f(c); // calls f(const B&)
B b;
A& r = b;
f(r); // calls f(const A&); the f(const B&) overload is not viable

Sınıfa göre aykırı olan üyelere işaretçiler için, benzer bir kural ters yönde uygulanır: en az türetilmiş sınıf tercih edilir.

void f(int B::*p);
void f(int C::*p);
int A::*p = &A::m;
f(p); // calls f(int B::*)
