Sözdizimi #

  • // Arama:

  • değişken.member_function();

  • değişken_işaretçi->üye_fonksiyonu();

  • // Tanım:

  • ret_type class_name::member_function() cv niteleyicileri {

    • body;
    • }

  • // Prototip:

  • sınıf sınıf_adı {

    • virt-specifier ret_type member_function() cv-qualifiers virt-specifier-seq;
    • // virt-specifier: “virtual”, if applicable.
    • // cv-qualifiers: “const” and/or “volatile”, if applicable.
    • // virt-specifier-seq: “override” and/or “final”, if applicable.
    • }

Statik olmayan bir üye işlevi, belirli bir örnekte çağrılan ve söz konusu örnek üzerinde çalışan bir ‘sınıf’/‘yapı’/‘birlik’ üye işlevidir. ‘Statik’ üye işlevlerinden farklı olarak, bir örnek belirtmeden çağrılamaz.

Sınıflar, yapılar ve birleşimler hakkında bilgi için lütfen ana konu konusuna bakın.

Statik Olmayan Üye İşlevleri #

Bir “sınıf” veya “yapı”, üye değişkenlerin yanı sıra üye işlevlere de sahip olabilir. Bu işlevlerin sözdizimi çoğunlukla bağımsız işlevlere benzerdir ve sınıf tanımının içinde veya dışında tanımlanabilir; sınıf tanımının dışında tanımlanmışsa, işlevin adının önüne sınıf adı ve kapsam (::) operatörü eklenir.

class CL {
  public:
    void  definedInside() {}
    void definedOutside();
};
void CL::definedOutside() {}

Bu işlevler, nokta (.) operatörüyle sınıfın bir örneğinde (veya bir örneğe referansta) veya ok (->) operatörüyle bir örneğe işaretçide çağrılır ve her çağrı işlevin çağrıldığı örnek; bir örnekte bir üye işlev çağrıldığında, bu örneğin tüm alanlarına (`bu’ işaretçisi aracılığıyla) erişebilir, ancak yalnızca bu örnekler parametre olarak sağlanmışsa diğer örneklerin alanlarına erişebilir.

struct ST {
    ST(const std::string& ss = "Wolf", int ii = 359) : s(ss), i(ii) { }

    int get_i() const { return i; }
    bool compare_i(const ST& other) const { return (i == other.i); }

  private:
    std::string s;
    int i;
};
ST st1;
ST st2("Species", 8472);

int  i = st1.get_i(); // Can access st1.i, but not st2.i.
bool b = st1.compare_i(st2); // Can access st1 & st2.

Bu işlevlerin, değişken veya işlevin erişim değiştiricilerinden bağımsız olarak üye değişkenlere ve/veya diğer üye işlevlere erişmesine izin verilir. Ayrıca, derleyici bir sınıfı derlemeye başlamadan önce tüm sınıf tanımının ayrıştırılması gerektiğinden, üye değişkenlere erişerek ve/veya onlardan önce bildirilen üye işlevleri çağırarak sıra dışı olarak da yazılabilirler.

class Access {
  public:
    Access(int i_ = 8088, int j_ = 8086, int k_ = 6502) : i(i_), j(j_), k(k_) {}

    int i;
    int get_k() const { return k; }
    bool private_no_more() const { return i_be_private(); }
  protected:
    int j;
    int get_i() const { return i; }
  private:
    int k;
    int get_j() const { return j; }
    bool i_be_private() const { return ((i > j) && (k < j)); }
};

Kapsülleme #

Üye işlevlerinin yaygın bir kullanımı, alanlara doğrudan erişmek yerine bir accessor (genellikle alıcı olarak bilinir) ve bir mutator (genellikle ayarlayıcı olarak bilinir) kullanarak kapsülleme içindir.

class Encapsulator {
    int encapsulated;

  public:
    int  get_encapsulated() const { return encapsulated; }
    void set_encapsulated(int e)  { encapsulated = e; }

    void some_func() {
        do_something_with(encapsulated);
    }
};

Sınıf içinde, ’encapsulated’, statik olmayan herhangi bir üye işlev tarafından serbestçe erişilebilir; sınıfın dışında, ona erişim, onu okumak için get_encapsulated() ve değiştirmek için set_encapsulated() kullanılarak üye işlevler tarafından düzenlenir. Bu, değişkeni okumak ve yazmak için ayrı işlevler kullanıldığından, değişkende kasıtsız değişiklikleri önler. [Her iki iddia için de iyi argümanlarla, alıcıların ve ayarlayıcıların kapsüllemeyi sağlayıp sağlamadığı veya kırdığı konusunda birçok tartışma var; böyle hararetli tartışmalar bu örneğin kapsamı dışındadır.]

Ad Gizleme ve İçe Aktarma #

Bir temel sınıf, bir dizi aşırı yüklenmiş işlev sağladığında ve türetilmiş bir sınıf, kümeye başka bir aşırı yük eklediğinde, bu, temel sınıf tarafından sağlanan tüm aşırı yüklemeleri gizler.

struct HiddenBase {
    void f(int) { std::cout << "int" << std::endl; }
    void f(bool) { std::cout << "bool" << std::endl; }
    void f(std::string) { std::cout << "std::string" << std::endl; }
};

struct HidingDerived : HiddenBase {
    void f(float) { std::cout << "float" << std::endl; }
};

// ...

HiddenBase hb;
HidingDerived hd;
std::string s;

hb.f(1);    // Output:  int
hb.f(true); // Output:  bool
hb.f(s);    // Output:  std::string;

hd.f(1.f);  // Output:  float
hd.f(3);    // Output:  float
hd.f(true); // Output:  float
hd.f(s);    // Error: Can't convert from std::string to float.

Bu, ad çözümleme kurallarından kaynaklanmaktadır: Ad araması sırasında, doğru ad bulunduğunda, bu ada sahip varlığın doğru sürümünü açıkça bulamamış olsak bile (örneğin hd.f(s ile) aramayı bırakırız. )); bu nedenle, türetilmiş sınıftaki işlevin aşırı yüklenmesi, ad aramasının temel sınıftaki aşırı yüklemeleri keşfetmesini engeller. Bundan kaçınmak için, adları temel sınıftan türetilmiş sınıfa “içe aktarmak” için bir using bildirimi kullanılabilir, böylece ad arama sırasında kullanılabilir olurlar.

struct HidingDerived : HiddenBase {
     // All members named HiddenBase::f shall be considered members of HidingDerived for lookup.
    using HiddenBase::f;

    void f(float) { std::cout << "float" << std::endl; }
};

// ...

HidingDerived hd;

hd.f(1.f);  // Output:  float
hd.f(3);    // Output:  int
hd.f(true); // Output:  bool
hd.f(s);    // Output:  std::string

Türetilmiş bir sınıf, bir using bildirimi ile adları içe aktarır, ancak aynı zamanda temel sınıftaki işlevlerle aynı imzaya sahip işlevleri bildirirse, temel sınıf işlevleri sessizce geçersiz kılınır veya gizlenir.

struct NamesHidden {
    virtual void hide_me()      {}
    virtual void hide_me(float) {}
    void hide_me(int)           {}
    void hide_me(bool)          {}
};

struct NameHider : NamesHidden {
    using NamesHidden::hide_me;

    void hide_me()    {} // Overrides NamesHidden::hide_me().
    void hide_me(int) {} // Hides NamesHidden::hide_me(int).
};

İçe aktarılan varlığın temel sınıfta “genel” veya “korumalı” olması koşuluyla, erişim değiştiricilerini değiştirmek için bir using bildirimi de kullanılabilir.

struct ProMem {
  protected:
    void func() {}
};

struct BecomesPub : ProMem {
    using ProMem::func;
};

// ...

ProMem pm;
BecomesPub bp;

pm.func(); // Error: protected.
bp.func(); // Good.

Benzer şekilde, kalıtım hiyerarşisindeki belirli bir sınıftan açıkça bir üye işlevi çağırmak istiyorsak, işlevi çağırırken o sınıfı adıyla belirterek işlev adını niteleyebiliriz.

struct One {
    virtual void f() { std::cout << "One." << std::endl; }
};

struct Two : One {
    void f() override {
        One::f(); // this->One::f();
        std::cout << "Two." << std::endl;
    }
};

struct Three : Two {
    void f() override {
        Two::f(); // this->Two::f();
        std::cout << "Three." << std::endl;
    }
};

// ...

Three t;

t.f();      // Normal syntax.
t.Two::f(); // Calls version of f() defined in Two.
t.One::f(); // Calls version of f() defined in One.

Sanal Üye İşlevleri #

Üye işlevler ayrıca ‘sanal’ olarak da bildirilebilir. Bu durumda, bir işaretçi veya bir örneğe referansla çağrılırsa, bunlara doğrudan erişilmez; bunun yerine, işlevi sanal işlev tablosunda ararlar (daha yaygın olarak “vtable” veya “vftable” olarak bilinen sanal işlevler için işaretçiler-üye işlevlerinin bir listesi) ve bunu uygun sürümü çağırmak için kullanırlar. örneğin dinamik (gerçek) türü için. İşlev doğrudan bir sınıfın değişkeninden çağrılırsa, arama yapılmaz.

struct Base {
    virtual void func() { std::cout << "In Base." << std::endl; }
};

struct Derived : Base {
    void func() override { std::cout << "In Derived." << std::endl; }
};

void slicer(Base x) { x.func(); }

// ...

Base b;
Derived d;

Base *pb = &b, *pd = &d; // Pointers.
Base &rb = b, &rd = d;   // References.

b.func();   // Output:  In Base.
d.func();   // Output:  In Derived.

pb->func(); // Output:  In Base.
pd->func(); // Output:  In Derived.

rb.func();  // Output:  In Base.
rd.func();  // Output:  In Derived.

slicer(b);  // Output:  In Base.
slicer(d);  // Output:  In Base.

“pd"nin “Temel*” ve “rd"nin bir “Temel” olmasına rağmen, “Temel::func(” yerine “Derived::func()” çağrılarından herhangi birinde “func()” çağrıldığını unutmayın. )`; bunun nedeni, “Derived” için “vtable” öğesinin “Base::func()” girişini bunun yerine “Derived::func()” öğesini gösterecek şekilde güncellemesidir. Tersine, bir örneği ‘slicer()’ öğesine geçirmenin, geçirilen örnek bir ‘Türetilen’ olsa bile, her zaman ‘Base::func()’ çağrılmasına neden olduğuna dikkat edin; bunun nedeni, veri dilimleme olarak bilinen bir şeydir; burada bir “Türev” örneğinin bir “Temel” parametresine değere göre iletilmesi, “Türev” örneğinin bir “Temel” örnek olmayan kısmını erişilemez hale getirir.

Bir üye işlevi sanal olarak tanımlandığında, geçersiz kılma işlevinin “sanal” olarak belirtilip belirtilmediğine bakılmaksızın, aynı imzaya sahip tüm türetilmiş sınıf üye işlevleri onu geçersiz kılar. Bu, türetilmiş sınıfları programcılar için ayrıştırmayı zorlaştırabilir, ancak hangi işlevin/işlevlerin ‘sanal’ olduğuna dair bir gösterge olmadığından.

struct B {
    virtual void f() {}
};

struct D : B {
    void f() {} // Implicitly virtual, overrides B::f.
                //  You'd have to check B to know that, though.
};

Ancak, türetilmiş bir işlevin yalnızca imzaları eşleşirse bir temel işlevi geçersiz kıldığına dikkat edin; türetilmiş bir işlev açıkça “sanal” olarak bildirilse bile, imzalar eşleşmezse bunun yerine yeni bir sanal işlev yaratacaktır.

struct BadB {
    virtual void f() {}
};

struct BadD : BadB {
    virtual void f(int i) {} // Does NOT override BadB::f.
};

C++ 11’den itibaren, geçersiz kılma amacı, bağlama duyarlı ‘geçersiz kılma’ anahtar sözcüğüyle açık hale getirilebilir. Bu, derleyiciye, programcının bir temel sınıf işlevini geçersiz kılmasını beklediğini söyler; bu da, herhangi bir şeyi geçersiz kılmazsa, derleyicinin bir hatayı atlamasına neden olur.

struct CPP11B {
    virtual void f() {}
};

struct CPP11D : CPP11B {
    void f() override {}
    void f(int i) override {} // Error: Doesn't actually override anything.
};

Bu aynı zamanda programcılara işlevin hem sanal olduğunu hem de karmaşık sınıfların ayrıştırılmasını kolaylaştırabilecek en az bir temel sınıfta bildirildiğini söyleme avantajına sahiptir.

son sürüm

Bir işlev “sanal” olarak bildirildiğinde ve sınıf tanımının dışında tanımlandığında, “sanal” belirteci işlev bildirimine dahil edilmeli ve tanımda tekrarlanmamalıdır.

Bu aynı zamanda ‘geçersiz kılma’ için de geçerlidir.

son sürüm 
struct VB {
    virtual void f(); // "virtual" goes here.
    void g();
};
/* virtual */ void VB::f() {} // Not here.
virtual void VB::g() {} // Error.

Bir temel sınıf bir “sanal” işlevi aşırı yüklerse, yalnızca “sanal” olarak açıkça belirtilen aşırı yüklemeler sanal olacaktır.

struct BOverload {
    virtual void func() {}
    void func(int) {}
};

struct DOverload : BOverload {
    void func() override {}
    void func(int) {}
};

// ...

BOverload* bo = new DOverload;
bo->func(); // Calls DOverload::func().
bo->func(1); // Calls BOverload::func(int).

Daha fazla bilgi için ilgili konuya bakın.

Sabit Doğruluk #

“Bu” özgeçmiş niteleyicilerinin birincil kullanımlarından biri ‘const’ doğruluğu‘dur. Bu, yalnızca bir nesneyi değiştirmek için gerekli olan erişimin, nesneyi değiştirmek için _gerektiğini ve bir nesneyi değiştirmesi gerekmeyen herhangi bir (üye veya üye olmayan) işlevin buna yazma erişimine sahip olmadığını garanti etme uygulamasıdır. nesne (doğrudan veya dolaylı olarak). Bu, kasıtsız değişiklikleri önleyerek kodu daha az hataya açık hale getirir. Ayrıca, durumu değiştirmesi gerekmeyen herhangi bir işlevin, işlevi yeniden yazmaya veya aşırı yüklemeye gerek kalmadan “const” veya “const” olmayan bir nesne alabilmesine olanak tanır.

‘const’ doğruluğu, doğası gereği, aşağıdan yukarıya başlar: Durumunu değiştirmesi gerekmeyen herhangi bir sınıf üyesi işlevi ‘const’ olarak bildirilir, böylece ‘const’ üzerinde çağrılabilir. örnekler. Bu da, değiştirilmeleri gerekmediğinde referansa göre geçirilen parametrelerin “const” olarak bildirilmesine izin verir, bu da işlevlerin şikayet etmeden “const” veya “const” olmayan nesneleri almasına ve “const” öğelerini almasına izin verir. `-ness bu şekilde dışa doğru yayılabilir. Bu nedenle, mantıksal durumu değiştirmesi gerekmeyen diğer işlevler gibi alıcılar da sıklıkla “const” olur.

class ConstIncorrect {
    Field fld;

  public:
    ConstIncorrect(const Field& f) : fld(f) {}     // Modifies.

    const Field& get_field()       { return fld; } // Doesn't modify; should be const.
    void set_field(const Field& f) { fld = f; }    // Modifies.

    void do_something(int i) {                     // Modifies.
        fld.insert_value(i);
    }
    void do_nothing()        { }                   // Doesn't modify; should be const.
};

class ConstCorrect {
    Field fld;

  public:
    ConstCorrect(const Field& f) : fld(f) {}       // Not const: Modifies.

    const Field& get_field() const { return fld; } // const: Doesn't modify.
    void set_field(const Field& f) { fld = f; }    // Not const: Modifies.

    void do_something(int i) {                     // Not const: Modifies.
        fld.insert_value(i);
    }
    void do_nothing() const  { }                   // const: Doesn't modify.
};

// ...

const ConstIncorrect i_cant_do_anything(make_me_a_field());
// Now, let's read it...
Field f = i_cant_do_anything.get_field();
  // Error: Loses cv-qualifiers, get_field() isn't const.
i_cant_do_anything.do_nothing();
  // Error: Same as above.
// Oops.

const ConstCorrect but_i_can(make_me_a_field());
// Now, let's read it...
Field f = but_i_can.get_field(); // Good.
but_i_can.do_nothing();          // Good.

‘ConstIncorrect’ ve ‘ConstCorrect’ hakkındaki yorumlarda gösterildiği gibi, uygun şekilde cv niteleme işlevleri de belgeleme işlevi görür. Bir sınıf “const” doğruysa, “const” olmayan herhangi bir işlevin güvenli bir şekilde durumunu değiştirdiği varsayılabilir ve “const” olan herhangi bir işlevin durumu değiştirmediği varsayılabilir.
