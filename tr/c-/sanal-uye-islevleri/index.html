Sözdizimi #

  • sanal boşluk f();

  • sanal boşluk g() = 0;

  • // C++11 veya üstü:

  • sanal boşluk h() geçersiz kılma;

  • geçersiz i() geçersiz kılma;

  • sanal boşluk j() final;

  • geçersiz k() son;

  • Yalnızca statik olmayan, şablon olmayan üye işlevler ‘sanal’ olabilir.

  • C++11 veya sonraki bir sürümünü kullanıyorsanız, bir temel sınıftan bir sanal üye işlevini geçersiz kılarken ‘geçersiz kıl’ kullanılması önerilir.

  • [Polimorfik temel sınıflar genellikle, türetilmiş bir nesnenin temel sınıfa bir işaretçi aracılığıyla silinmesine izin vermek için sanal yıkıcılara sahiptir1. Yıkıcı sanal değilse, böyle bir işlem tanımsız davranışa[expr.delete] §5.3 yol açar. 5/3 .

C++11 ve sonraki sürümlerde geçersiz kılmayı sanal ile kullanma #

“geçersiz kılma” belirteci, işlev imzasının sonuna eklenirse, C++ 11’den itibaren özel bir anlama sahiptir. Bu, bir fonksiyonun

  • Temel sınıfta bulunan işlevi geçersiz kılmak &
  • Temel sınıf işlevi ‘sanal’

Bu belirticinin bir “çalışma zamanı” önemi yoktur, çünkü esas olarak derleyiciler için bir göstergedir.

Aşağıdaki örnek, geçersiz kılma kullanmadan bizim davranışımızdaki değişikliği gösterecektir.

“Geçersiz kıl” olmadan:

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // Y::f() will not override X::f() because it has a different signature,
    // but the compiler will accept the code (and silently ignore Y::f()).
    virtual void f(int a) { std::cout << a << "\n"; }
};

“Geçersiz kıl” ile:

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // The compiler will alert you to the fact that Y::f() does not
    // actually override anything.
    virtual void f(int a) override { std::cout << a << "\n"; }
};

“geçersiz kılma"nın bir anahtar kelime olmadığını, yalnızca işlev imzalarında görünebilen özel bir tanımlayıcı olduğunu unutmayın. Diğer tüm bağlamlarda ‘geçersiz kılma’ yine de bir tanımlayıcı olarak kullanılabilir:

void foo() {
    int override = 1; // OK.
    int virtual = 2;  // Compilation error: keywords can't be used as identifiers.
}

Sanal ve sanal olmayan üye işlevleri #

Sanal üye işlevleriyle:

#include <iostream>

struct X {
    virtual void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
    // Specifying virtual again here is optional
    // because it can be inferred from X::f().
    virtual void f() { std::cout << "Y::f()\n"; } 
};

void call(X& a) {
    a.f();
}

int main() {
    X x;
    Y y;
    call(x); // outputs "X::f()"
    call(y); // outputs "Y::f()"
}

Sanal üye işlevleri olmadan:

#include <iostream>

struct X {
   void f() { std::cout << "X::f()\n"; }
};

struct Y : X {
   void f() { std::cout << "Y::f()\n"; }
};

void call(X& a) {
    a.f();
}

int main() {
    X x;
    Y y;
    call(x); // outputs "X::f()"
    call(y); // outputs "X::f()"
}

Son sanal işlevler #

C++11, yöntem imzasında görünüyorsa yöntemi geçersiz kılmayı yasaklayan “son” belirteci tanıttı:

class Base {
public:
    virtual void foo() {
        std::cout << "Base::Foo\n";
    }
};

class Derived1 : public Base {
public:
    // Overriding Base::foo
    void foo() final {
        std::cout << "Derived1::Foo\n";
    }
};

class Derived2 : public Derived1 {
public:
    // Compilation error: cannot override final method
    virtual void foo() {
        std::cout << "Derived2::Foo\n";
    }
};

‘final’ belirteci yalnızca ‘sanal’ üye işleviyle kullanılabilir ve sanal olmayan üye işlevlere uygulanamaz

‘final’ gibi, türetilmiş sınıfta ‘sanal’ işlevlerin geçersiz kılınmasını önleyen ‘geçersiz kılma’ belirteci de vardır.

“geçersiz kılma” ve “son” belirteçleri, istenen etkiye sahip olmak için birleştirilebilir:

class Derived1 : public Base {
public:
    void foo() final override {
        std::cout << "Derived1::Foo\n";
    }
};

Oluşturucularda ve yıkıcılarda sanal işlevlerin davranışı #

Yapıcılar ve yıkıcılardaki sanal işlevlerin davranışı, ilk karşılaşıldığında genellikle kafa karıştırıcıdır.

#include <iostream>
using namespace std;

class base { 
public:
    base() { f("base constructor"); }
    ~base() { f("base destructor"); }

    virtual const char* v() { return "base::v()"; }

    void f(const char* caller) { 
        cout << "When called from " << caller << ", "  << v() << " gets called.\n"; 
    }        
};

class derived : public base {
public:
    derived() { f("derived constructor"); }
    ~derived() { f("derived destructor"); }

    const char* v() override { return "derived::v()"; }

};

int main() {
     derived d;
}

Çıktı:

Temel kurucudan çağrıldığında, base::v() çağrılır. Türetilmiş yapıcıdan çağrıldığında türetilmiş::v() çağrılır. Türetilmiş yıkıcıdan çağrıldığında türetilmiş::v() çağrılır. Base yıkıcıdan çağrıldığında base::v() çağrılır.

Bunun arkasındaki mantık, türetilmiş sınıfın henüz başlatılmamış (kurucu durumunda) veya zaten yok edilmiş (yıkıcı durumunda) ek üyeler tanımlayabilmesidir ve üye işlevlerini çağırmak güvenli olmayacaktır. Bu nedenle, C++ nesnelerinin inşası ve imhası sırasında, *thisin dinamik türü, daha türetilmiş bir sınıf değil, yapıcının veya yıkıcının sınıfı olarak kabul edilir.

Örnek:

#include <iostream>
#include <memory>

using namespace std;
class base {
public:
    base()
    {
        std::cout << "foo is " << foo() << std::endl;
    }
    virtual int foo() { return 42; }
};

class derived : public base {
    unique_ptr<int> ptr_;
public:
    derived(int i) : ptr_(new int(i*i)) { }
    // The following cannot be called before derived::derived due to how C++ behaves, 
    // if it was possible... Kaboom!
    int foo() override   { return *ptr_; } 
};

int main() {
    derived d(4);
}

Saf sanal işlevler #

Ayrıca, bildirime ‘= 0’ ekleyerek bir ‘sanal’ işlevin saf sanal (soyut) olduğunu belirtebiliriz. Bir veya daha fazla saf sanal işleve sahip sınıflar soyut olarak kabul edilir ve somutlaştırılamaz; yalnızca tüm saf sanal işlevleri tanımlayan veya tanımlarını devralan türetilmiş sınıflar somutlaştırılabilir.

struct Abstract {
    virtual void f() = 0;
};

struct Concrete {
    void f() override {}
};

Abstract a; // Error.
Concrete c; // Good.

Bir işlev saf sanal olarak belirtilse bile, ona varsayılan bir uygulama verilebilir. Buna rağmen, işlev yine de soyut olarak kabul edilecek ve türetilmiş sınıflar, somutlaştırılmadan önce onu tanımlamak zorunda kalacaklar. Bu durumda, işlevin türetilmiş sınıfının sürümünün, temel sınıfın sürümünü çağırmasına bile izin verilir.

struct DefaultAbstract {
    virtual void f() = 0;
};
void DefaultAbstract::f() {}

struct WhyWouldWeDoThis : DefaultAbstract {
    void f() override { DefaultAbstract::f(); }
};

Bunu yapmak istememizin birkaç nedeni var:

  • Kendisi somutlaştırılamayan, ancak türetilmiş sınıflarının somutlaştırılmasını engellemeyen bir sınıf oluşturmak istiyorsak, yıkıcıyı saf sanal olarak ilan edebiliriz. Örneği serbest bırakabilmek istiyorsak, yıkıcı olarak, yine de tanımlanmalıdır. Ve yıkıcı, polimorfik kullanım sırasında bellek sızıntılarını önlemek için büyük olasılıkla zaten sanal olduğundan, başka bir işlevi ‘sanal’ olarak bildirmekten gereksiz bir performans darbesi almayacağız. Bu, arayüzler oluştururken faydalı olabilir.

      struct Interface {
      virtual ~Interface() = 0;
  };
  Interface::~Interface() = default;

  struct Implementation : Interface {};
  // ~Implementation() is automatically defined by the compiler if not explicitly
  //  specified, meeting the "must be defined before instantiation" requirement.

  • Saf sanal işlevin çoğu veya tüm uygulamaları yinelenen kod içeriyorsa, bu kod bunun yerine temel sınıf sürümüne taşınabilir, bu da kodun bakımını kolaylaştırır.

      class SharedBase {
      State my_state;
      std::unique_ptr<Helper> my_helper;
      // ...

    public:
      virtual void config(const Context& cont) = 0;
      // ...
  };
  /* virtual */ void SharedBase::config(const Context& cont) {
      my_helper = new Helper(my_state, cont.relevant_field);
      do_this();
      and_that();
  }

  class OneImplementation : public SharedBase {
      int i;
      // ...

    public:
      void config(const Context& cont) override;
      // ...
  };
  void OneImplementation::config(const Context& cont) /* override */ {
      my_state = { cont.some_field, cont.another_field, i };
      SharedBase::config(cont);
      my_unique_setup();
  };

  // And so on, for other classes derived from SharedBase.
