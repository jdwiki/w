SFINAE nedir #

SFINAE **İkame **İkame HataHata Is Not An Error anlamına gelir. Bir işlev şablonunu veya bir sınıf şablonunu somutlaştırmak için türlerin (veya değerlerin) değiştirilmesinden kaynaklanan kötü biçimli kod, sabit bir derleme hatası değildir, yalnızca bir kesinti hatası olarak kabul edilir.

İşlev şablonlarının veya sınıf şablonu uzmanlıklarının somutlaştırılmasındaki kesinti başarısızlıkları, o adayı değerlendirme kümesinden çıkarır - sanki bu başarısız aday başlangıçta yokmuş gibi.

template <class T>
auto begin(T& c) -> decltype(c.begin()) { return c.begin(); }

template <class T, size_t N>
T* begin(T (&arr)[N]) { return arr; }

int vals[10];
begin(vals); // OK. The first function template substitution fails because
             // vals.begin() is ill-formed. This is not an error! That function
             // is just removed from consideration as a viable overload candidate,
             // leaving us with the array overload.

Yalnızca acil bağlamda yer alan ikame hataları kesinti hataları olarak kabul edilir, diğerlerinin tümü sabit hatalar olarak kabul edilir.

template <class T>
void add_one(T& val) { val += 1; }

int i = 4;
add_one(i); // ok

std::string msg = "Hello";
add_one(msg); // error. msg += 1 is ill-formed for std::string, but this
              // failure is NOT in the immediate context of substituting T

void_t #

“void_t”, herhangi bir (sayı) türü “void” yazmak üzere eşleyen bir meta işlevdir. “void_t"nin birincil amacı, tip özelliklerinin yazılmasını kolaylaştırmaktır.

std::void_t C++17’nin bir parçası olacak, ancak o zamana kadar uygulanması son derece basittir:

template <class...> using void_t = void;

Bazı derleyiciler biraz farklı bir uygulama gerektirir:

template <class...>
struct make_void { using type = void; };

template <typename... T>
using void_t = typename make_void<T...>::type;

‘void_t’nin birincil uygulaması, bir ifadenin geçerliliğini kontrol eden tip özellikleri yazmaktır. Örneğin, bir türün hiçbir argüman almayan ‘foo()’ üye işlevine sahip olup olmadığını kontrol edelim:

template <class T, class=void>
struct has_foo : std::false_type {};

template <class T>
struct has_foo<T, void_t<decltype(std::declval<T&>().foo())>> : std::true_type {};

Bu nasıl çalışıyor? “has_foo::value” örneğini başlatmaya çalıştığımda, bu, derleyicinin “has_foo<T, void>” için en iyi uzmanlığı aramaya çalışmasına neden olur. İki seçeneğimiz var: birincil ve bu temel ifadeyi somutlaştırmayı gerektiren bu ikincil seçenek:

  • Eğer ‘T’ **, ‘foo()’ üye işlevine sahipse, döndürülen tür ne olursa olsun ‘void’e dönüştürülür ve şablon kısmi sıralama kurallarına dayalı olarak uzmanlık birincil olana tercih edilir. Yani ‘has_foo::value’, ’true’ olacak
  • Eğer T böyle bir üye işlevine sahip değilse (veya birden fazla argüman gerektiriyorsa), bu durumda uzmanlık için ikame başarısız olur ve yalnızca yedeklenecek birincil şablona sahibiz. Bu nedenle, “has_foo::value”, “yanlış"tır.

Daha basit bir durum:

template<class T, class=void>
struct can_reference : std::false_type {};

template<class T>
struct can_reference<T, std::void_t<T&>> : std::true_type {};

bu, “std::declval” veya “decltype” kullanmaz.

Bir geçersiz argümanın ortak bir modelini fark edebilirsiniz. Bunu hesaba katabiliriz:

struct details {
  template<template<class...>class Z, class=void, class...Ts>
  struct can_apply:
    std::false_type
  {};
  template<template<class...>class Z, class...Ts>
  struct can_apply<Z, std::void_t<Z<Ts...>>, Ts...>:
    std::true_type
  {};
};

template<template<class...>class Z, class...Ts>
using can_apply = details::can_apply<Z, void, Ts...>;

bu, ‘std::void_t’ kullanımını gizler ve ‘can_apply’ öğesinin, ilk şablon bağımsız değişkeni olarak sağlanan türün, diğer türlerle değiştirildikten sonra iyi biçimlendirilip biçimlenmediğinin bir göstergesi gibi davranmasını sağlar. Önceki örnekler şimdi “can_apply” kullanılarak şu şekilde yeniden yazılabilir:

template<class T>
using ref_t = T&;

template<class T>
using can_reference = can_apply<ref_t, T>;    // Is T& well formed for T?

ve:

template<class T>
using dot_foo_r = decltype(std::declval<T&>().foo());

template<class T>
using can_dot_foo = can_apply< dot_foo_r, T >;    // Is T.foo() well formed for T?

orijinal sürümlerden daha basit görünüyor.

‘can_apply’ye benzer ‘std’ özellikleri için C++17 sonrası teklifler vardır.

“void_t"nin faydası Walter Brown tarafından keşfedildi. CppCon 2016’da bununla ilgili harika bir sunu yaptı.

son sürüm

enable_if #

std::enable_if, SFINAE’yi tetiklemek için boole koşullarını kullanmak için uygun bir yardımcı programdır. Şu şekilde tanımlanır:

template <bool Cond, typename Result=void>
struct enable_if { };

template <typename Result>
struct enable_if<true, Result> {
    using type = Result;
};

Yani, “enable_if<true, R>::type”, “R” için bir takma addır, oysa “enable_if<false, T>::type”, “enable_if” uzmanlığının bir " type` üye türü.

‘std::enable_if’ şablonları kısıtlamak için kullanılabilir:

int negate(int i) { return -i; }

template <class F>
auto negate(F f) { return -f(); }

Burada, “olumsuz (1)” çağrısı, belirsizlik nedeniyle başarısız olur. Ancak ikinci aşırı yükleme, integral türleri için kullanılmak üzere tasarlanmamıştır, bu nedenle şunları ekleyebiliriz:

int negate(int i) { return -i; }

template <class F, class = typename std::enable_if<!std::is_arithmetic<F>::value>::type>
auto negate(F f) { return -f(); }

Şimdi, “olumsuz” örneğini başlatmak, “!std::is_arithmetic::değer” “yanlış” olduğundan, bir ikame hatasına neden olur. SFINAE nedeniyle, bu zor bir hata değildir, bu aday basitçe aşırı yük kümesinden çıkarılır. Sonuç olarak, “olumsuz (1)” yalnızca tek bir geçerli adaya sahiptir ve bu aday daha sonra çağrılır.

Ne zaman kullanılır #

‘std::enable_if’ öğesinin SFINAE’nin üstünde bir yardımcı olduğunu akılda tutmakta fayda var, ancak SFINAE’yi en başta çalıştıran şey bu değil. ‘std::size’ işlevine benzer bir işlevsellik uygulamak için bu iki alternatifi, yani bir kap veya dizinin boyutunu üreten bir aşırı yükleme kümesi ‘size(arg)’ düşünelim:

// for containers
template<typename Cont>
auto size1(Cont const& cont) -> decltype( cont.size() );

// for arrays
template<typename Elt, std::size_t Size>
std::size_t size1(Elt const(&arr)[Size]);

// implementation omitted
template<typename Cont>
struct is_sizeable;

// for containers
template<typename Cont, std::enable_if_t<std::is_sizeable<Cont>::value, int> = 0>
auto size2(Cont const& cont);

// for arrays
template<typename Elt, std::size_t Size>
std::size_t size2(Elt const(&arr)[Size]);

‘is_sizeable’ın uygun şekilde yazıldığını varsayarsak, bu iki bildirim SFINAE’ye göre tam olarak eşdeğer olmalıdır. Hangisini yazması en kolay, hangisi bir bakışta gözden geçirmesi ve anlaması en kolayı?

Şimdi, sarma veya modüler davranış lehine imzalı tamsayı taşmasını önleyen aritmetik yardımcıları nasıl uygulamak isteyebileceğimizi düşünelim. Yani örn. “incr(i, 3)”, “i”, “INT_MAX” değerine sahip bir “int” olsa bile sonucun her zaman tanımlanacağı gerçeği dışında, “i += 3” ile aynı olacaktır. Bunlar iki olası alternatiftir:

// handle signed types
template<typename Int>
auto incr1(Int& target, Int amount)
-> std::void_t<int[static_cast<Int>(-1) < static_cast<Int>(0)]>;

// handle unsigned types by just doing target += amount
// since unsigned arithmetic already behaves as intended
template<typename Int>
auto incr1(Int& target, Int amount)
-> std::void_t<int[static_cast<Int>(0) < static_cast<Int>(-1)]>;
 
template<typename Int, std::enable_if_t<std::is_signed<Int>::value, int> = 0>
void incr2(Int& target, Int amount);
 
template<typename Int, std::enable_if_t<std::is_unsigned<Int>::value, int> = 0>
void incr2(Int& target, Int amount);

Bir kez daha hangisi yazması en kolay, hangisi bir bakışta gözden geçirmesi ve anlaması en kolay?

‘std::enable_if’ özelliğinin gücü, yeniden düzenleme ve API tasarımı ile nasıl oynadığıdır. “is_sizeable::value”, “cont.size()” öğesinin geçerli olup olmadığını yansıtmayı amaçlıyorsa, o zaman ifadeyi “size1” için göründüğü gibi kullanmak daha kısa olabilir, ancak bu “is_sizeable” olup olmamasına bağlı olabilir. birkaç yerde kullanılır veya kullanılmaz. Bunu, uygulamasının “incr1” bildirimine sızdığında olduğundan çok daha açık bir şekilde yansıtan “std::is_signed” ile karşılaştırın.

Tespit edildi #

type_trait oluşturmayı genelleştirmek için:SFINAE’ye dayalı “tespit edildi_veya”, “tespit edildi_t”, “tespit edildi” gibi deneysel özellikler vardır.

typename Default, template <typename...> Op ve typename ... Args şablon parametreleriyle:

  • “is_detected”: “Op<Args…>” öğesinin geçerliliğine bağlı olarak “std::true_type” veya “std::false_type” takma adı
  • “algılandı_t”: “Op<Args…>” öğesinin geçerliliğine bağlı olarak “Op<Args…>” veya “nonesuch"un diğer adı.
  • “algılanan_veya”: “değer_t” olan ve “algılanan” bir yapının diğer adı ve “Op<Args…>” veya “Op<Args…>‘in geçerliliğine bağlı olarak “Op<Args…>” veya “Varsayılan” olan “type” `

SFINAE için std::void_t kullanılarak aşağıdaki gibi uygulanabilir:

namespace detail {
    template <class Default, class AlwaysVoid,
              template<class...> class Op, class... Args>
    struct detector
    {
        using value_t = std::false_type;
        using type = Default;
    };

    template <class Default, template<class...> class Op, class... Args>
    struct detector<Default, std::void_t<Op<Args...>>, Op, Args...>
    {
        using value_t = std::true_type;
        using type = Op<Args...>;
    };

} // namespace detail

// special type to indicate detection failure
struct nonesuch {
    nonesuch() = delete;
    ~nonesuch() = delete;
    nonesuch(nonesuch const&) = delete;
    void operator=(nonesuch const&) = delete;
};

template <template<class...> class Op, class... Args>
using is_detected =
    typename detail::detector<nonesuch, void, Op, Args...>::value_t;

template <template<class...> class Op, class... Args>
using detected_t = typename detail::detector<nonesuch, void, Op, Args...>::type;

template <class Default, template<class...> class Op, class... Args>
using detected_or = detail::detector<Default, void, Op, Args...>;
son sürüm

Yöntemin varlığını tespit etmeye yönelik özellikler daha sonra basitçe uygulanabilir:

typename <typename T, typename ...Ts>
using foo_type = decltype(std::declval<T>().foo(std::declval<Ts>()...));

struct C1 {};

struct C2 {
    int foo(char) const;
};

template <typename T>
using has_foo_char = is_detected<foo_type, T, char>;

static_assert(!has_foo_char<C1>::value, "Unexpected");
static_assert(has_foo_char<C2>::value, "Unexpected");

static_assert(std::is_same<int, detected_t<foo_type, C2, char>>::value,
              "Unexpected");

static_assert(std::is_same<void, // Default
                           detected_or<void, foo_type, C1, char>>::value,
              "Unexpected");
static_assert(std::is_same<int, detected_or<void, foo_type, C2, char>>::value,
              "Unexpected");

Çok sayıda seçenekle aşırı yük çözünürlüğü #

Birkaç seçenek arasından seçim yapmanız gerekirse, ’enable_if<>’ aracılığıyla yalnızca birini etkinleştirmek oldukça zahmetli olabilir, çünkü birkaç koşulun da olumsuzlanması gerekir.

Aşırı yükler arasındaki sıralama bunun yerine kullanılarak seçilebilir. kalıtım, yani etiket gönderimi.

İyi şekillendirilmesi gereken şeyi test etmek yerine ve ayrıca diğer tüm sürüm koşullarının olumsuzluğunu test etmek yerine test ediyoruz sadece ihtiyacımız olan şey için, tercihen bir geri dönüşte bir ‘decltype’ içinde.
Bu, birkaç seçeneği iyi biçimlendirilmiş bırakabilir, bunlar arasında ayrım yaparız. yineleyici özellik etiketlerine benzer şekilde ’etiketler’ kullanma (‘random_access_tag’ ve diğerleri). Bu işe yarar çünkü doğrudan bir eşleşme, bir temel sınıftan daha iyidir, ki bu daha iyidir bir temel sınıfın temel sınıfı, vb.

#include <algorithm>
#include <iterator>

namespace detail
{
    // this gives us infinite types, that inherit from each other
    template<std::size_t N>
    struct pick : pick<N-1> {};
    template<>
    struct pick<0> {};

    // the overload we want to be preferred have a higher N in pick<N>
    // this is the first helper template function
    template<typename T>
    auto stable_sort(T& t, pick<2>)
        -> decltype( t.stable_sort(), void() )
    {
        // if the container have a member stable_sort, use that
        t.stable_sort();
    }

    // this helper will be second best match
    template<typename T>
    auto stable_sort(T& t, pick<1>)
        -> decltype( t.sort(), void() )
    {
        // if the container have a member sort, but no member stable_sort
        // it's customary that the sort member is stable
        t.sort();
    }

    // this helper will be picked last
    template<typename T>
    auto stable_sort(T& t, pick<0>)
        -> decltype( std::stable_sort(std::begin(t), std::end(t)), void() )
    {
        // the container have neither a member sort, nor member stable_sort
        std::stable_sort(std::begin(t), std::end(t));
    }

}

// this is the function the user calls. it will dispatch the call
// to the correct implementation with the help of 'tags'.
template<typename T>
void stable_sort(T& t)
{
    // use an N that is higher that any used above.
    // this will pick the highest overload that is well formed.
    detail::stable_sort(t, detail::pick<10>{});
}

Aşırı yüklemeleri ayırt etmek için yaygın olarak kullanılan başka yöntemler de vardır. tam eşlemenin dönüşümden daha iyi olması, üç noktadan daha iyi olması gibi.

Bununla birlikte, etiket gönderimi herhangi bir sayıda seçeneğe kadar uzanabilir ve amaç açısından biraz daha açıktır.

işlev şablonlarında sondaki dekltype #

Kısıtlayıcı işlevlerden biri, dönüş türünü belirtmek için sondaki ‘decltype’ kullanmaktır:

namespace details {
   using std::to_string;

   // this one is constrained on being able to call to_string(T)
   template <class T>
   auto convert_to_string(T const& val, int )
       -> decltype(to_string(val))
   {
       return to_string(val);
   }

   // this one is unconstrained, but less preferred due to the ellipsis argument
   template <class T>
   std::string convert_to_string(T const& val, ... )
   {
       std::ostringstream oss;
       oss << val;
       return oss.str();
   }
}

template <class T>
std::string convert_to_string(T const& val)
{
    return details::convert_to_string(val, 0);
}

’to_string()‘i çağırabileceğim bir argümanla ‘convert_to_string()’ işlevini çağırırsam, ‘details::convert_to_string()’ için iki geçerli işlevim olur. Birincisi tercih edilir, çünkü “0"dan “int"e dönüştürme, “0"dan “…“e dönüştürmeden daha iyi bir örtük dönüştürme sırasıdır.

‘convert_to_string()’ işlevini, ’to_string()’ işlevini çağıramadığım bir bağımsız değişkenle çağırırsam, ilk işlev şablonu somutlaştırması ikame hatasına yol açar (‘decltype(to_string(val))’ yoktur). Sonuç olarak, bu aday aşırı yük kümesinden çıkarılır. İkinci fonksiyon şablonu kısıtsızdır, bu yüzden seçilir ve bunun yerine operator<<(std::ostream&, T) üzerinden gideriz. Bu tanımsızsa, o zaman ‘oss « val’ satırında bir şablon yığınıyla ilgili bir sabit derleme hatası var demektir.

son sürüm

enable_if_all / enable_if_any #

Motivasyon örneği

Aşağıdaki kod parçacığında olduğu gibi, şablon parametreleri listesinde değişken bir şablon paketiniz olduğunda:

template<typename ...Args> void func(Args &&...args) { //... };

Standart kitaplık (C++17’den önce), “Args” içindeki tüm parametrelere veya **parametrelerden herhangi birine SFINAE kısıtlamaları uygulamak için enable_if yazmanın doğrudan bir yolunu sunmaz. ** “Args” içinde. C++17, bu sorunu çözen std::conjunction ve std::disjunction seçeneklerini sunar. Örneğin:

/// C++17: SFINAE constraints on all of the parameters in Args.
template<typename ...Args,
         std::enable_if_t<std::conjunction_v<custom_conditions_v<Args>...>>* = nullptr>
void func(Args &&...args) { //... };

/// C++17: SFINAE constraints on any of the parameters in Args.
template<typename ...Args,
         std::enable_if_t<std::disjunction_v<custom_conditions_v<Args>...>>* = nullptr>
void func(Args &&...args) { //... };

Elinizde C++17 yoksa, bunları başarmak için birkaç çözüm vardır. Bunlardan biri, bu sorunun yanıtlarında gösterildiği gibi, temel durum sınıfı ve kısmi uzmanlıklar kullanmaktır.

Alternatif olarak, ‘std::conjunction’ ve ‘std::disjunction’ davranışını oldukça basit bir şekilde elle de uygulayabilirsiniz. Aşağıdaki örnekte, uygulamaları göstereceğim ve anlamsal olarak tam olarak ne yapmaları gerekiyorsa onu yapan iki takma ad oluşturmak için bunları ‘std::enable_if’ ile birleştireceğim: ’enable_if_all’ ve ’enable_if_any’. Bu, daha ölçeklenebilir bir çözüm sağlayabilir.

**** “enable_if_all” ve “enable_if_any"nin uygulanması

Önce, sırasıyla özelleştirilmiş ‘seq_and’ ve ‘seq_or’ kullanarak ‘std::conjunction’ ve ‘std::disjunction’ı taklit edelim:

/// Helper for prior to C++14.
template<bool B, class T, class F >
using conditional_t = typename std::conditional<B,T,F>::type;

/// Emulate C++17 std::conjunction.
template<bool...> struct seq_or: std::false_type {};
template<bool...> struct seq_and: std::true_type {};

template<bool B1, bool... Bs>
struct seq_or<B1,Bs...>: 
  conditional_t<B1,std::true_type,seq_or<Bs...>> {};

template<bool B1, bool... Bs>
struct seq_and<B1,Bs...>:
  conditional_t<B1,seq_and<Bs...>,std::false_type> {};

O zaman uygulama oldukça basittir:

template<bool... Bs>
using enable_if_any = std::enable_if<seq_or<Bs...>::value>;

template<bool... Bs>
using enable_if_all = std::enable_if<seq_and<Bs...>::value>;

Sonunda bazı yardımcılar:

template<bool... Bs>
using enable_if_any_t = typename enable_if_any<Bs...>::type;

template<bool... Bs>
using enable_if_all_t = typename enable_if_all<Bs...>::type;

Kullanım

Kullanım da düz ileri:

    /// SFINAE constraints on all of the parameters in Args.
    template<typename ...Args,
             enable_if_all_t<custom_conditions_v<Args>...>* = nullptr>
    void func(Args &&...args) { //... };

    /// SFINAE constraints on any of the parameters in Args.
    template<typename ...Args,
             enable_if_any_t<custom_conditions_v<Args>...>* = nullptr>
    void func(Args &&...args) { //... };
