Sıfır Kuralı #

Daha yalın bir arayüz elde etmek için Beş Kuralı ve RAII ilkelerini birleştirebiliriz: Sıfır Kuralı: yönetilmesi gereken herhangi bir kaynak kendi türünde olmalıdır. Bu türün Beş Kuralı izlemesi gerekir, ancak bu kaynağın tüm kullanıcılarının beş özel üye işlevinden hiçbirini yazması gerekmez ve hepsini “varsayılan” olarak yapabilir.

[Üç Kural örneğinde] tanıtılan ‘Person’ sınıfını kullanma(https://www.wikiod.com/tr/docs/c%2b%2b/1206/the-rule-of- Three-five-and-zero/9867/ üç kural#t=201607281743050894887), “cstrings” için kaynak yöneten bir nesne oluşturabiliriz:

class cstring {
private:
    char* p;

public:
    ~cstring() { delete [] p; }
    cstring(cstring const& );
    cstring(cstring&& );
    cstring& operator=(cstring const& );
    cstring& operator=(cstring&& );

    /* other members as appropriate */
};

Ve bu ayrı olduğunda, ‘Person’ sınıfımız çok daha basit hale gelir:

class Person {
    cstring name;
    int arg;

public:
    ~Person() = default;
    Person(Person const& ) = default;
    Person(Person&& ) = default;
    Person& operator=(Person const& ) = default;
    Person& operator=(Person&& ) = default;

    /* other members as appropriate */
};

‘Person’daki özel üyelerin açıkça belirtilmesine bile gerek yoktur; derleyici, ‘Kişi’ içeriğine bağlı olarak bunları uygun şekilde varsayılan yapacak veya silecektir. Bu nedenle, aşağıdaki de sıfır kuralına bir örnektir.

struct Person {
    cstring name;
    int arg;
};

“cstring”, “delete” kopya oluşturucu/atama işleciyle birlikte yalnızca hareket eden bir tür olsaydı, “Kişi” de otomatik olarak yalnızca hareket ederdi.

Sıfır kuralı terimi R. Martinho Fernandes tarafından tanıtıldı

Beş Kuralı #

C++11 iki yeni özel üye işlevi sunar: taşıma yapıcısı ve taşıma atama operatörü. C++03’teki Üç Kuralı‘i takip etmek istemenizle aynı nedenlerle, genellikle C++11’deki Beş Kuralı’nı takip etmek istersiniz: Bir sınıf beş özel üye işlevinden BİRİNİ gerektiriyorsa ve hareket semantiği isteniyorsa, büyük olasılıkla BEŞİNİN TÜMÜNÜ gerektirir.

Bununla birlikte, Beş Kuralı takip etmemenin genellikle bir hata olarak kabul edilmediğini, ancak Üç Kuralı takip edildiği sürece kaçırılmış bir optimizasyon fırsatı olarak kabul edildiğini unutmayın. Derleyici normalde bir tane kullandığında hiçbir hareket oluşturucu veya taşıma atama operatörü mevcut değilse, bunun yerine mümkünse kopyalama semantiğini kullanacak ve gereksiz kopyalama işlemleri nedeniyle daha az verimli bir işlemle sonuçlanacaktır. Bir sınıf için hareket semantiği istenmiyorsa, bir hareket oluşturucu veya atama operatörü bildirmeye gerek yoktur.

Üç Kuralı ile aynı örnek:

class Person
{
    char* name;
    int age;

public:
    // Destructor 
    ~Person() { delete [] name; }

    // Implement Copy Semantics
    Person(Person const& other)
        : name(new char[std::strlen(other.name) + 1])
        , age(other.age)
    {
        std::strcpy(name, other.name);
    }
    
    Person &operator=(Person const& other) 
    {
        // Use copy and swap idiom to implement assignment.
        Person copy(other);
        swap(*this, copy);
        return *this;
    }

    // Implement Move Semantics
    // Note: It is usually best to mark move operators as noexcept
    //       This allows certain optimizations in the standard library
    //       when the class is used in a container.

    Person(Person&& that) noexcept
        : name(nullptr)               // Set the state so we know it is undefined
        , age(0)
    {
        swap(*this, that);
    }

    Person& operator=(Person&& that) noexcept
    {
        swap(*this, that);
        return *this;
    }

    friend void swap(Person& lhs, Person& rhs) noexcept
    {
        std::swap(lhs.name, rhs.name);
        std::swap(lhs.age, rhs.age);
    }
};

Alternatif olarak, hem kopyalama hem de taşıma atama operatörü, kopyala ve değiştir deyiminin kullanımını kolaylaştırmak için referans veya değer referansı yerine değere göre bir örnek alan tek bir atama operatörü ile değiştirilebilir.

Person& operator=(Person copy)
{
    swap(*this, copy);
    return *this;
}

Üçlü Kuraldan Beşli Kurala geçmek performans nedenleriyle önemlidir, ancak çoğu durumda kesinlikle gerekli değildir. Kopya oluşturucu ve atama operatörünün eklenmesi, türün taşınmasının bellek sızdırmamasını sağlar (bu durumda taşıma oluşturma işlemi yalnızca kopyalamaya geri döner), ancak arayan kişinin muhtemelen beklemediği kopyaları gerçekleştirecektir.

Üçlü Kural #

Üç Kuralı, bir türün kullanıcı tanımlı bir kopya oluşturucuya, kopya atama operatörüne veya yıkıcıya ihtiyacı varsa, o zaman üçünün de olması gerektiğini belirtir.

Kuralın nedeni, üçünden herhangi birine ihtiyaç duyan bir sınıfın bazı kaynakları (dosya tanıtıcıları, dinamik olarak ayrılmış bellek, vb.) Kopyalama işlevleri, kaynağın nesneler arasında nasıl kopyalandığıyla ilgilenir ve yıkıcı RAII ilkeleri uyarınca kaynağı yok eder.

Bir dize kaynağını yöneten bir tür düşünün:

class Person
{
    char* name;
    int age;

public:
    Person(char const* new_name, int new_age)
        : name(new char[std::strlen(new_name) + 1])
        , age(new_age)
    {
       std::strcpy(name, new_name);
    }

    ~Person() {
        delete [] name;
    }
};

Yapıcıda ’name’ tahsis edildiğinden, yıkıcı bellek sızıntısını önlemek için onu serbest bırakır. Ancak böyle bir nesne kopyalanırsa ne olur?

int main()
{
    Person p1("foo", 11);
    Person p2 = p1;
}

İlk olarak, ‘p1’ inşa edilecektir. Sonra ‘p2’, ‘p1’den kopyalanacaktır. Ancak, C++ tarafından oluşturulan kopya oluşturucu, türün her bileşenini olduğu gibi kopyalayacaktır. Bu, “p1.name” ve “p2.name"nin her ikisinin de aynı dizeye işaret ettiği anlamına gelir.

‘Main’ sona erdiğinde, yıkıcılar çağrılacak. İlk p2 yıkıcısı çağrılır; dizeyi silecektir. Sonra p1 yıkıcısı çağrılacak. Ancak, dize zaten silinmiş. Zaten silinmiş olan bellekte “delete” çağrısı yapmak tanımsız davranışa neden olur.

Bundan kaçınmak için uygun bir kopya oluşturucu sağlamak gerekir. Bir yaklaşım, farklı ‘Kişi’ örneklerinin aynı dize verilerini paylaştığı bir referans sayılan sistemi uygulamaktır. Her kopyalama yapıldığında, paylaşılan referans sayısı artırılır. Yıkıcı daha sonra referans sayısını azaltır, yalnızca sayım sıfırsa belleği serbest bırakır.

Veya değer semantiği ve derin kopyalama davranışı uygulayabiliriz:

Person(Person const& other)
    : name(new char[std::strlen(other.name) + 1])
    , age(other.age)
{
    std::strcpy(name, other.name);
}

Person &operator=(Person const& other) 
{
    // Use copy and swap idiom to implement assignment
    Person copy(other);
    swap(copy);            //  assume swap() exchanges contents of *this and copy
    return *this;
}

Kopya atama operatörünün uygulanması, mevcut bir arabelleği serbest bırakma ihtiyacı nedeniyle karmaşıktır. Kopyala ve değiştir tekniği, yeni bir arabellek tutan geçici bir nesne oluşturur. ‘*bu’ ve ‘kopya’ içeriğinin değiştirilmesi, orijinal arabelleğin ‘kopyasının’ sahipliğini verir. İşlev dönerken “kopya"nın yok edilmesi, daha önce “*bu"nun sahip olduğu arabelleği serbest bırakır.

Kendi Kendine Atama Koruması #

Bir kopya atama operatörü yazarken, kendi kendine atama durumunda çalışabilmesi çok önemlidir. Yani, buna izin vermelidir:

SomeType t = ...;
t = t;

Kendi kendine atama genellikle bu kadar açık bir şekilde gerçekleşmez. Tipik olarak, atamanın konumunun yalnızca iki ‘Kişi’ işaretçisine veya referansına sahip olduğu ve bunların aynı nesne oldukları hakkında hiçbir fikrinin olmadığı, çeşitli kod sistemlerinden geçen dolambaçlı bir yol aracılığıyla gerçekleşir.

Yazdığınız herhangi bir kopya atama operatörü bunu hesaba katabilmelidir.

Bunu yapmanın tipik yolu, tüm atama mantığını aşağıdaki gibi bir koşula sarmaktır:

SomeType &operator=(const SomeType &other)
{
    if(this != &other)
    {
        //Do assignment logic.
    }
    return *this;
}

Not: Kendi kendine atama hakkında düşünmek ve bu gerçekleştiğinde kodunuzun doğru şekilde davrandığından emin olmak önemlidir. Bununla birlikte, kendi kendine atama çok nadir görülen bir durumdur ve bunu önlemek için optimizasyon yapmak normal durumu gerçekten kötüleştirebilir. Normal durum çok daha yaygın olduğu için, kendi kendine atama için kötümserlik yapmak kod verimliliğinizi azaltabilir (bu yüzden onu kullanırken dikkatli olun).

Örnek olarak, atama operatörünü uygulamak için normal teknik “kopyala ve değiştir deyimi"dir. Bu tekniğin normal uygulaması, kendi kendine atamayı test etme zahmetine girmez (bir kopya yapıldığı için kendi kendine atama pahalı olsa da). Bunun nedeni, normal durumun karamsarlığının çok daha maliyetli olduğunun gösterilmesidir (daha sık olduğu gibi).

Taşı atama operatörleri de kendi kendine atamaya karşı korunmalıdır. Bununla birlikte, bu tür operatörlerin çoğu için mantık, aynı bellekten/aynı belleğe geçişi gayet iyi idare edebilen ‘std::swap’a dayanmaktadır. Dolayısıyla, hareket atama mantığınız bir dizi takas işleminden başka bir şey değilse, kendi kendine atama korumasına ihtiyacınız yoktur.

Durum böyle değilse, yukarıdakine benzer önlemler almalısınız.
