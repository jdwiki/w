Çağrılabilir ve Gelecek #

‘Runnable’, yürütülecek kodu farklı bir iş parçacığına sarmak için bir araç sağlarken, yürütmeden bir sonuç döndüremeyeceği için bir sınırlaması vardır. Bir “Çalıştırılabilir"in yürütülmesinden bir miktar dönüş değeri elde etmenin tek yolu, sonucu “Çalıştırılabilir” dışında bir kapsamda erişilebilir bir değişkene atamaktır.

“Callable”, Java 5’te “Runnable” ile eş olarak tanıtıldı. “Çağrı”, “çalıştır” yerine “çağrı” yöntemine sahip olması dışında temelde aynıdır. “call” yönteminin ek bir sonuç döndürme yeteneği vardır ve ayrıca işaretli istisnalar atmasına da izin verilir.

Çağırılabilir görev gönderiminden elde edilen sonuç, bir Gelecek aracılığıyla izlenebilir

“Gelecek”, “Çağrılabilir” hesaplamanın sonucunu barındıran bir tür kapsayıcı olarak düşünülebilir. Çağrılabilir olanın hesaplanması başka bir iş parçacığında devam edebilir ve bir ‘Geleceğin’ sonucuna dokunmaya yönelik herhangi bir girişim engellenir ve yalnızca uygun olduğunda sonucu döndürür.

Çağrılabilir Arayüz

public interface Callable<V> {
    V call() throws Exception;
}

Gelecek

interface Future<V> {
    V get();
    V get(long timeout, TimeUnit unit);
    boolean cancel(boolean mayInterruptIfRunning);
    boolean isCancelled();
    boolean isDone();
}

Callable ve Future örneğini kullanma:

public static void main(String[] args) throws Exception {
    ExecutorService es = Executors.newSingleThreadExecutor();
          
    System.out.println("Time At Task Submission : " + new Date());
    Future<String> result = es.submit(new ComplexCalculator());
    // the call to Future.get() blocks until the result is available.So we are in for about a 10 sec wait now
    System.out.println("Result of Complex Calculation is : " + result.get());
    System.out.println("Time At the Point of Printing the Result : " + new Date());
}

Uzun bir hesaplama yapan Callable’ımız

public class ComplexCalculator implements Callable<String> {

    @Override
    public String call() throws Exception {
        // just sleep for 10 secs to simulate a lengthy computation
        Thread.sleep(10000);            
        System.out.println("Result after a lengthy 10sec calculation");
        return "Complex Result"; // the result 
    }
}

Get() ‘Geleceğin sağladığı gerçek sonucu çıkarma yöntemi iken

  • get(long timeout, TimeUnit unit) geçerli iş parçacığı sırasında bir sonuç bekleyeceği maksimum süreyi tanımlar;
  • Görev çağrısını iptal etmek için cancel(mayInterruptIfRunning). ‘mayInterrupt’ bayrağı, görev başlatıldıysa ve şu anda çalışıyorsa, görevin kesilmesi gerektiğini belirtir;

  • Uzun görevin iptal edilip edilmediğini kontrol etmek için isCancelled().

Temel Çoklu Kullanım #

Yürütülecek çok sayıda göreviniz varsa ve tüm bu görevler öncekilerin sonucuna bağlı değilse, bilgisayarınızın tüm bu görevleri aynı anda daha fazla işlemci kullanarak yapması için Multithreading‘i kullanabilirsiniz. . Bu, bazı büyük bağımsız görevleriniz varsa, program yürütmenizi daha hızlı hale getirebilir.

class CountAndPrint implements Runnable {

    private final String name;

    CountAndPrint(String name) {
        this.name = name;
    }

    /** This is what a CountAndPrint will do */
    @Override
    public void run() {
        for (int i = 0; i < 10000; i++) {
            System.out.println(this.name + ": " + i);
        }
    }

    public static void main(String[] args) {
        // Launching 4 parallel threads
        for (int i = 1; i <= 4; i++) {
            // `start` method will call the `run` method 
            // of CountAndPrint in another thread
            new Thread(new CountAndPrint("Instance " + i)).start();
        }

        // Doing some others tasks in the main Thread
        for (int i = 0; i < 10000; i++) {
            System.out.println("Main: " + i);
        }
    }
}

Geri Sayım Mandalı #

CountDownLatch

Bir veya daha fazla iş parçacığının, diğer iş parçacıklarında gerçekleştirilen bir dizi işlem tamamlanana kadar beklemesine izin veren bir senkronizasyon yardımı.

  1. Belirli bir sayı ile bir “CountDownLatch” başlatılır.
  2. ‘countDown()’ yönteminin çağrıları nedeniyle geçerli sayım sıfıra ulaşana kadar bekleme yöntemleri bloğu, ardından tüm bekleyen diziler serbest bırakılır ve sonraki tüm bekleme çağrıları hemen geri döner.
  3. Bu tek seferlik bir olgudur—sayı sıfırlanamaz. Sayıyı sıfırlayan bir sürüme ihtiyacınız varsa, bir “CyclicBarrier” kullanmayı düşünün.

Anahtar Yöntemler:

public void await() throws InterruptedException

İplik kesilmediği sürece, mevcut iş parçacığının mandal sıfıra geri sayana kadar beklemesine neden olur.

public void countDown()

Mandalın sayısını azaltır, sayım sıfıra ulaşırsa bekleyen tüm dizileri serbest bırakır.

Örnek:

import java.util.concurrent.*;

class DoSomethingInAThread implements Runnable {
    CountDownLatch latch;
    public DoSomethingInAThread(CountDownLatch latch) {
        this.latch = latch;
    } 
    public void run() {
        try {
            System.out.println("Do some thing");
            latch.countDown();
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

public class CountDownLatchDemo {
    public static void main(String[] args) {
        try {
            int numberOfThreads = 5;
            if (args.length < 1) {
                System.out.println("Usage: java CountDownLatchDemo numberOfThreads");
                return;
            }
            try {
                numberOfThreads = Integer.parseInt(args[0]);
            } catch(NumberFormatException ne) {
            
            }
            CountDownLatch latch = new CountDownLatch(numberOfThreads);
            for (int n = 0; n < numberOfThreads; n++) {
                Thread t = new Thread(new DoSomethingInAThread(latch));
                t.start();
            }
            latch.await();
            System.out.println("In Main thread after completion of " + numberOfThreads + " threads");
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

Do some thing
Do some thing
Do some thing
In Main thread after completion of 5 threads

  1. “CountDownLatch”, Ana iş parçacığında 5’lik bir sayaçla başlatılır
  2. Ana iş parçacığı await() yöntemini kullanarak bekliyor.
  3. Beş DoSomethingInAThread örneği oluşturuldu. Her örnek, sayacı countDown() yöntemiyle azalttı.
  4. Sayaç sıfır olduğunda, Ana iş parçacığı devam edecektir.

Senkronizasyon yardımcıları olarak kilitler #

Java 5’in eşzamanlı paket tanıtımından önce iş parçacığı oluşturma daha düşük seviyeydi. Bu paketin tanıtımı birkaç yüksek seviyeli eşzamanlı programlama yardımcıları/yapıları sağladı.

Kilitler, esasen senkronize bloklar veya anahtar kelimelerle aynı amaca hizmet eden iş parçacığı senkronizasyon mekanizmalarıdır.

İçsel Kilitleme

int count = 0; // shared among multiple threads

public void doSomething() {
    synchronized(this) {
        ++count; // a non-atomic operation
    }
}

Kilitleri kullanarak senkronizasyon

int count = 0; // shared among multiple threads

Lock lockObj = new ReentrantLock();
public void doSomething() {
    try {
        lockObj.lock();
        ++count; // a non-atomic operation
    } finally {    
        lockObj.unlock(); // sure to release the lock without fail
    }
}

Kilitler ayrıca, kilitleme ancak kesintiye yanıt verme veya kilitlemeye çalışma ve yapılamadığında engellememe gibi içsel kilitlemenin sunmadığı işlevlere sahiptir.

Kilitleme, kesintiye duyarlı

class Locky {
    int count = 0; // shared among multiple threads

    Lock lockObj = new ReentrantLock();

    public void doSomething() {
        try {
            try {
                lockObj.lockInterruptibly();
                ++count; // a non-atomic operation
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // stopping
            }
        } finally {
            if (!Thread.currentThread().isInterrupted()) {
                lockObj.unlock(); // sure to release the lock without fail
            }
        }
    }
}

Yalnızca kilitleyebildiğiniz zaman bir şeyler yapın

public class Locky2 {
    int count = 0; // shared among multiple threads

    Lock lockObj = new ReentrantLock();

    public void doSomething() {
        boolean locked = lockObj.tryLock(); // returns true upon successful lock
        if (locked) {
            try {
                ++count; // a non-atomic operation
            } finally {
                lockObj.unlock(); // sure to release the lock without fail
            }
        }
    }
}

Birkaç kilit çeşidi mevcuttur. Daha fazla ayrıntı için buradan api belgelerine bakın.

Senkronizasyon #

Java’da, dil düzeyinde yerleşik bir kilitleme mekanizması vardır: herhangi bir Java nesnesini içsel bir kilit olarak kullanabilen “eşzamanlı” blok (yani, her Java nesnesinin kendisiyle ilişkilendirilmiş bir monitörü olabilir).

İçsel kilitler, ifade gruplarına atomiklik sağlar. Bunun bizim için ne anlama geldiğini anlamak için, “senkronize"nin yararlı olduğu bir örneğe bakalım:

private static int t = 0;
private static Object mutex = new Object();

public static void main(String[] args) {
    ExecutorService executorService = Executors.newFixedThreadPool(400); // The high thread count is for demonstration purposes.
    for (int i = 0; i < 100; i++) {
        executorService.execute(() -> {
            synchronized (mutex) {
                t++;
                System.out.println(MessageFormat.format("t: {0}", t));
            }
        });
    }
    executorService.shutdown();
}

Bu durumda, “senkronize” blok olmasaydı, birden fazla eşzamanlılık sorunu söz konusu olurdu. İlki, artış sonrası operatörüyle (kendi içinde atomik değildir) ve ikincisi, t’nin değerini, rastgele miktarda başka iş parçacığı onu değiştirme şansına sahip olduktan sonra gözlemliyor olmamız olacaktır. Ancak, içsel bir kilit elde ettiğimiz için burada hiçbir yarış koşulu olmayacak ve çıktı, normal sıralarında 1’den 100’e kadar sayıları içerecektir.

Java’daki dahili kilitler mutekslerdir (yani karşılıklı yürütme kilitleri). Karşılıklı yürütme, eğer bir iş parçacığı kilidi ele geçirirse, ikincisinin, kilidi kendi başına alabilmesi için ilkinin onu serbest bırakmasını beklemek zorunda kalacağı anlamına gelir. Not: İş parçacığını bekleme (uyku) durumuna getirebilecek bir işleme engelleme işlemi denir. Bu nedenle, bir kilit elde etmek bir engelleme işlemidir.

Java’daki dahili kilitler geçicidir. Bu, bir iş parçacığı zaten sahip olduğu bir kilidi almaya çalışırsa, engellemeyeceği ve başarılı bir şekilde alacağı anlamına gelir. Örneğin, aşağıdaki kod çağrıldığında engellenmez:

public void bar(){
    synchronized(this){
        ...
    }
}
public void foo(){
    synchronized(this){
        bar();
    }
}

‘Senkronize’ blokların yanı sıra ‘senkronize’ yöntemler de vardır.

Aşağıdaki kod blokları pratik olarak eşdeğerdir (bayt kodu farklı görünse de):

  1. “bu” üzerinde “senkronize” blok:

    public void foo() {
    synchronized(this) {
        doStuff();
    }
}

  2. “senkronize” yöntem:

     public synchronized void foo() {
     doStuff();
 }

Aynı şekilde “statik” yöntemler için de bu:

class MyClass {
    ...
    public static void bar() {
        synchronized(MyClass.class) {
            doSomeOtherStuff();
        }
    }
}

bununla aynı etkiye sahiptir:

class MyClass {
    ...
    public static synchronized void bar() {
        doSomeOtherStuff();
    }
}

class DeadLock {
    
    Object mLock1 = new Object();
    Object mLock2 = new Object();
    

    public void methodA() {
        System.out.println("methodA wait for mLock1  " + Thread.currentThread().getName());
        synchronized (mLock1) {
            System.out.println("methodA mLock1 acquired   " + Thread.currentThread().getName());

        }

Çalıştırılabilir Nesne #

“Runnable” arabirimi, iş parçacığında yürütülen kodu içermesi amaçlanan “run()” adlı tek bir yöntemi tanımlar.

‘Runnable’ nesnesi, ‘Thread’ yapıcısına iletilir. Ve Thread’in start() metodu çağrılır.

Örnek

public class HelloRunnable implements Runnable {

    @Override
    public void run() {
        System.out.println("Hello from a thread");
    }

    public static void main(String[] args) {
        new Thread(new HelloRunnable()).start();
    }
}

Java8’deki örnek:

public static void main(String[] args) {
    Runnable r = () -> System.out.println("Hello world");
    new Thread(r).start();
}

Çalıştırılabilir vs Thread alt sınıfı

Bir “Çalıştırılabilir” nesne istihdamı daha geneldir, çünkü “Çalıştırılabilir” nesne, “İş parçacığı” dışında bir sınıfı alt sınıflara ayırabilir.

‘İş parçacığı’ alt sınıflamasının basit uygulamalarda kullanımı daha kolaydır, ancak görev sınıfınızın ‘İş parçacığı’nın alt sınıfı olması gerektiği gerçeğiyle sınırlıdır.

Bir “Runnable” nesnesi, üst düzey iş parçacığı yönetimi API’lerine uygulanabilir.

Semafor #

Semafor, iş parçacıkları tarafından alınabilen ve bırakılabilen bir dizi izin tutan üst düzey bir eşzamanlayıcıdır. Bir Semafor, bir iş parçacığı edindiğinde azaltılacak ve bir iş parçacığı serbest bırakıldığında artırılacak bir izinler sayacı olarak hayal edilebilir. Bir iş parçacığı almaya çalıştığında izin miktarı “0” ise, bir izin sağlanana kadar (veya iş parçacığı kesilene kadar) iş parçacığı engellenir.

Bir semafor şu şekilde başlatılır:

`Semafor semaforu = yeni Semafor(1); // izinlerin sayısı olan int değeri’

Semaphore oluşturucu, adalet için ek bir boole parametresini kabul eder. false olarak ayarlandığında, bu sınıf, iş parçacıklarının izinleri alma sırası hakkında hiçbir garanti vermez. Adalet doğru ayarlandığında, semafor, edinme yöntemlerinden herhangi birini çağıran iş parçacıklarının, bu yöntemleri çağırmalarının işlendiği sırayla izin almak için seçilmesini garanti eder. Aşağıdaki şekilde ilan edilir:

Semafor semaforu = yeni Semafor(1, doğru);

Şimdi javadocs’tan bir öğe havuzuna erişimi kontrol etmek için Semaphore’un kullanıldığı bir örneğe bakalım. Bu örnekte, getItem() çağrıldığında elde edilecek öğelerin her zaman olmasını sağlamak için engelleme işlevi sağlamak için bir Semafor kullanılmıştır.

Java.lang.Thread örneği oluşturma #

Java’da bir iş parçacığı oluşturmanın iki ana yaklaşımı vardır. Özünde, bir iş parçacığı oluşturmak, içinde yürütülecek kodu yazmak kadar kolaydır. İki yaklaşım, bu kodu tanımladığınız yere göre farklılık gösterir.

Java’da bir iş parçacığı bir nesneyle temsil edilir - java.lang.Thread örneği veya onun alt sınıfı. Dolayısıyla ilk yaklaşım, bu alt sınıfı oluşturmak ve run() yöntemini geçersiz kılmaktır.

Not: Thread‘ı java.lang.Thread sınıfına atıfta bulunmak için ve thread‘ı evrelerin mantıksal konseptine atıfta bulunmak için kullanacağım.

class MyThread extends Thread {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Thread running!");
        }
    }
}

Yürütülecek kodu zaten tanımladığımız için, iş parçacığı basitçe şu şekilde oluşturulabilir:

MyThread t = new MyThread();

Thread sınıfı ayrıca, iş parçacığının adı olarak kullanılacak bir dize kabul eden bir kurucu içerir. Bu, özellikle çok iş parçacıklı bir programda hata ayıklarken faydalı olabilir.

class MyThread extends Thread {
    public MyThread(String name) {
        super(name);
    }
    
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Thread running! ");
        }
    }
}

MyThread t = new MyThread("Greeting Producer");

İkinci yaklaşım, kodu java.lang.Runnable ve onun tek yöntemi olan run() kullanarak tanımlamaktır. Thread sınıfı daha sonra bu yöntemi ayrı bir iş parçacığında yürütmenize izin verir. Bunu başarmak için, Runnable arabiriminin bir örneğini kabul eden bir kurucu kullanarak iş parçacığı oluşturun.

Thread t = new Thread(aRunnable);

Bu, lambdalar veya yöntem referansları ile birleştirildiğinde çok güçlü olabilir (yalnızca Java 8):

Thread t = new Thread(operator::hardWork);

Konunun adını da belirtebilirsiniz.

Thread t = new Thread(operator::hardWork, "Pi operator");

Pratik olarak konuşursak, her iki yaklaşımı da endişelenmeden kullanabilirsiniz. Ancak genel bilgelik ikincisini kullanmayı söylüyor.

Bahsedilen dört kurucunun her biri için, ilk parametre olarak bir Java.lang.ThreadGroup örneğini kabul eden bir alternatif de vardır.

ThreadGroup tg = new ThreadGroup("Operators");
Thread t = new Thread(tg, operator::hardWork, "PI operator");

ThreadGroup bir dizi iş parçacığını temsil eder. Bir Thread‘i yalnızca bir Thread yapıcısını kullanarak bir ThreadGroup’e ekleyebilirsiniz. ThreadGroup daha sonra tüm Thread’lerini birlikte yönetmek için kullanılabilir ve Thread, ThreadGroup’ten bilgi alabilir.

Özetlemek gerekirse, Thread şu genel kuruculardan biri ile oluşturulabilir:

Thread()
Thread(String name)
Thread(Runnable target)
Thread(Runnable target, String name)
Thread(ThreadGroup group, String name)
Thread(ThreadGroup group, Runnable target)
Thread(ThreadGroup group, Runnable target, String name)
Thread(ThreadGroup group, Runnable target, String name, long stackSize)

Sonuncusu, yeni iş parçacığı için istenen yığın boyutunu tanımlamamızı sağlar.

Aynı özelliklere veya aynı modele sahip birçok İş Parçacığı oluştururken ve yapılandırırken genellikle kod okunabilirliği zarar görür. İşte o zaman java.util.concurrent.ThreadFactory kullanılabilir. Bu arabirim, iş parçacığı oluşturma prosedürünü fabrika modeli ve onun tek yöntemi newThread(Runnable) aracılığıyla kapsüllemenize olanak tanır.

class WorkerFactory implements ThreadFactory {
    private int id = 0;

    @Override
    public Thread newThread(Runnable r) {
        return new Thread(r, "Worker " + id++);
    }
}

Özel yazma / Eşzamanlı okuma erişimi #

Bazen bir işlemin aynı “veri"yi aynı anda yazması ve okuması gerekir.

‘ReadWriteLock’ arabirimi ve ‘ReentrantReadWriteLock’ uygulaması, aşağıdaki gibi tanımlanabilecek bir erişim modeline izin verir:

  1. Verilerin herhangi bir sayıda eşzamanlı okuyucusu olabilir. En az bir okuyucu erişimi verilmişse, yazar erişimi mümkün değildir.
  2. Verilerde en fazla bir tek yazar olabilir. Verilen bir yazar erişimi varsa, hiçbir okuyucu verilere erişemez.

Bir uygulama şöyle görünebilir:

import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;
public class Sample {

// Our lock. The constructor allows a "fairness" setting, which guarantees the chronology of lock attributions.
protected static final ReadWriteLock RW_LOCK = new ReentrantReadWriteLock();

// This is a typical data that needs to be protected for concurrent access
protected static int data = 0;

/** This will write to the data, in an exclusive access */
public static void writeToData() {
    RW_LOCK.writeLock().lock();
    try {
        data++;
    } finally {
        RW_LOCK.writeLock().unlock();
    }
}

public static int readData() {
    RW_LOCK.readLock().lock();
    try {
        return data;
    } finally {
        RW_LOCK.readLock().unlock();
    }
}

}

NOT 1 : Bu kesin kullanım örneğinin ‘AtomicInteger’ kullanan daha temiz bir çözümü vardır, ancak burada açıklanan, buradaki verilerin bir Atomik varyant olarak bir tamsayı olduğu gerçeğinden bağımsız olarak çalışan bir erişim modelidir.

NOT 2 : Sıradan bir okuyucu için öyle görünmese de, okuma kısmındaki kilit gerçekten gereklidir. Gerçekten de, okuyucu tarafına kilitlenmezseniz, aşağıdakiler de dahil olmak üzere birçok şey ters gidebilir:

  1. İlkel değerlerin yazılmasının tüm JVM’lerde atomik olduğu garanti edilmez, bu nedenle okuyucu ör. “veri” 64 bit uzunluğunda bir türse, 64 bitin yalnızca 32 biti yazılır
  2. Bunu gerçekleştirmeyen bir iş parçacığından yazmanın görünürlüğü, yalnızca yazmalar ve okumalar arasında Önceden Oldu ilişkisi kurarsak JVM tarafından garanti edilir. Bu ilişki, hem okuyucular hem de yazarlar kendi kilitlerini kullandıklarında kurulur, ancak başka türlü olmaz.

Daha yüksek performansın gerekli olduğu durumlarda, belirli kullanım türlerinin altında, diğer şeylerin yanı sıra iyimser bir kilit modu uygulayan ‘Damgalı Kilit’ adı verilen daha hızlı bir kilit türü mevcuttur. Bu kilit, ‘ReadWriteLock’tan çok farklı çalışır ve bu örnek yer değiştirilemez.

Üretici-Tüketici #

Üretici-tüketici problem çözümüne basit bir örnek. JDK sınıflarının (AtomicBoolean ve BlockingQueue) senkronizasyon için kullanıldığına dikkat edin, bu da geçersiz bir çözüm oluşturma şansını azaltır. Çeşitli [BlockingQueue] türleri için Javadoc’a danışın(https://docs.Oracle.com/javase/8/docs/api/java/util/concurrent/BlockingQueue.html); farklı bir uygulama seçmek, bu örneğin davranışını büyük ölçüde değiştirebilir (DelayQueue veya [Priority Queue] gibi) (https://docs.Oracle.com/javase/8/docs/api/java/util/concurrent/PriorityBlockingQueue.html)).

public class Producer implements Runnable {

    private final BlockingQueue<ProducedData> queue;

    public Producer(BlockingQueue<ProducedData> queue) {
        this.queue = queue;
    }

    public void run() {
        int producedCount = 0;
        try {
            while (true) {
                producedCount++;
                //put throws an InterruptedException when the thread is interrupted
                queue.put(new ProducedData());
            }
        } catch (InterruptedException e) {
            // the thread has been interrupted: cleanup and exit
            producedCount--;
            //re-interrupt the thread in case the interrupt flag is needeed higher up
            Thread.currentThread().interrupt();
        }
        System.out.println("Produced " + producedCount + " objects");
    }
}

public class Consumer implements Runnable {

    private final BlockingQueue<ProducedData> queue;

    public Consumer(BlockingQueue<ProducedData> queue) {
        this.queue = queue;
    }

    public void run() {
        int consumedCount = 0;
        try {
            while (true) {
                //put throws an InterruptedException when the thread is interrupted
                ProducedData data = queue.poll(10, TimeUnit.MILLISECONDS);
                // process data
                consumedCount++;
            }
        } catch (InterruptedException e) {
            // the thread has been interrupted: cleanup and exit
            consumedCount--;
            //re-interrupt the thread in case the interrupt flag is needeed higher up
            Thread.currentThread().interrupt();
        }
        System.out.println("Consumed " + consumedCount + " objects");
    }
}


public class ProducerConsumerExample {
    static class ProducedData {    
        // empty data object
    }

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<ProducedData> queue = new ArrayBlockingQueue<ProducedData>(1000);
        // choice of queue determines the actual behavior: see various BlockingQueue implementations

        Thread producer = new Thread(new Producer(queue));
        Thread consumer = new Thread(new Consumer(queue));

        producer.start();
        consumer.start();

        Thread.sleep(1000);
        producer.interrupt();
        Thread.sleep(10);
        consumer.interrupt();
    }
}

ThreadLocal’ı Kullanma #

Java Eşzamanlılığı’nda kullanışlı bir araç ‘ThreadLocal’dır - bu, belirli bir iş parçacığı için benzersiz olacak bir değişkene sahip olmanızı sağlar. Bu nedenle, aynı kod farklı iş parçacıklarında çalışıyorsa, bu yürütmeler değeri paylaşmaz, bunun yerine her iş parçacığının, iş parçacığı için _yerel olan kendi değişkeni vardır.

Örneğin, bu, bir sunucu uygulamasında bir isteğin ele alınmasının bağlamını (yetki bilgileri gibi) oluşturmak için sıklıkla kullanılır. Bunun gibi bir şey yapabilirsiniz:

private static final ThreadLocal<MyUserContext> contexts = new ThreadLocal<>();

public static MyUserContext getContext() {
    return contexts.get(); // get returns the variable unique to this thread
}

public void doGet(...) {
    MyUserContext context = magicGetContextFromRequest(request); 
    contexts.put(context); // save that context to our thread-local - other threads
                           // making this call don't overwrite ours
    try {
        // business logic
    } finally {
        contexts.remove(); // 'ensure' removal of thread-local variable
    }
}

Şimdi, her bir metoda MyUserContext geçirmek yerine, ihtiyacınız olan yerde MyServlet.getContext() kullanabilirsiniz. Tabii ki, bu, belgelenmesi gereken bir değişken getiriyor, ancak iş parçacığı için güvenli, bu kadar yüksek kapsamlı bir değişken kullanmanın birçok dezavantajını ortadan kaldırıyor.

Buradaki en önemli avantaj, her iş parçacığının, o “bağlamlar” kapsayıcısında kendi iş parçacığı yerel değişkenine sahip olmasıdır. Tanımlanmış bir giriş noktasından kullandığınız sürece (her sunucu uygulamasının bağlamını korumasını talep etmek veya belki bir sunucu uygulaması filtresi ekleyerek) ihtiyacınız olduğunda bu bağlamın orada olduğuna güvenebilirsiniz.

Senkronize/geçici kullanırken okuma/yazma engellerini görselleştirme #

Bildiğimiz gibi, bir yöntemin veya bloğun yürütülmesini özel kılmak için synchronized anahtar sözcüğünü kullanmamız gerektiğini biliyoruz. Ancak çok azımız ‘senkronize’ ve ‘uçucu’ anahtar kelime kullanmanın bir önemli yönünün farkında olmayabiliriz: kod birimini atomik hale getirmenin yanı sıra, okuma/yazma engeli de sağlar. Bu okuma/yazma engeli nedir? Bunu bir örnek kullanarak tartışalım:

class Counter {

  private Integer count = 10;

  public synchronized void incrementCount() {
    count++;
  }

  public Integer getCount() {
    return count;
  }
}

Diyelim ki bir A dizisi önce “incrementCount()“u çağırıyor, sonra başka bir B dizisi “getCount()“u çağırıyor. Bu senaryoda, B’nin “count"un güncellenmiş değerini göreceğinin garantisi yoktur. Yine de ‘count’u ‘10’ olarak görebilir, hatta ‘count’un güncellenmiş değerini hiçbir zaman görmemesi de mümkündür.

Bu davranışı anlamak için Java bellek modelinin donanım mimarisiyle nasıl bütünleştiğini anlamamız gerekir. Java’da her iş parçacığının kendi iş parçacığı yığını vardır. Bu yığın şunları içerir: yöntem çağrı yığını ve bu iş parçacığında oluşturulan yerel değişken. Çok çekirdekli bir sistemde, iki iş parçacığının aynı anda ayrı çekirdeklerde çalışması oldukça olasıdır. Böyle bir senaryoda, bir iş parçacığı yığınının bir kısmının bir çekirdeğin kayıt/önbellek içinde olması mümkündür. Bir iş parçacığının içindeyse, bir nesneye ‘synchronized’ (veya ‘volatile’) anahtar sözcüğü kullanılarak erişilir, iş parçacığının o değişkenin yerel kopyasını ana bellekle eşitleyen ‘synchronized’ bloğundan sonra. Bu, bir okuma/yazma engeli oluşturur ve iş parçacığının o nesnenin en son değerini görmesini sağlar.

Ama bizim durumumuzda, B parçacığı “sayma” için senkronize erişim kullanmadığından, kayıtta saklanan “sayma” değerine atıfta bulunuyor olabilir ve A dizisinden asla güncellemeleri görmeyebilir. B’nin en son sayım değerini gördüğünden emin olmak için getCount()u da senkronize etmeniz gerekiyor.

public synchronized Integer getCount() {
  return count;
}

Şimdi, A iş parçacığı ‘count’ güncellemesi ile bittiğinde, ‘Counter’ örneğinin kilidini açar, aynı zamanda yazma engeli oluşturur ve bu blok içinde yapılan tüm değişiklikleri ana belleğe temizler. Benzer şekilde, B thread’i aynı ‘Counter’ örneğinde kilit aldığında, okuma bariyerine girer ve ana bellekten ‘count’ değerini okur ve tüm güncellemeleri görür.

Aynı görünürlük etkisi “geçici” okuma/yazma işlemleri için de geçerlidir. ‘volatile’ye yazmadan önce güncellenen tüm değişkenler ana belleğe aktarılacak ve ‘volatile’ değişken okumasından sonraki tüm okumalar ana bellekten olacaktır.

Paylaşılan global sıraya sahip çoklu üretici/tüketici örneği #

Aşağıdaki kod, birden fazla Üretici/Tüketici programını gösterir. Hem Üretici hem de Tüketici iş parçacıkları aynı küresel kuyruğu paylaşır.

import java.util.concurrent.*;
import java.util.Random;

public class ProducerConsumerWithES {
    public static void main(String args[]) {
        BlockingQueue<Integer> sharedQueue = new LinkedBlockingQueue<Integer>();
         
        ExecutorService pes = Executors.newFixedThreadPool(2);
        ExecutorService ces = Executors.newFixedThreadPool(2);
          
        pes.submit(new Producer(sharedQueue, 1));
        pes.submit(new Producer(sharedQueue, 2));
        ces.submit(new Consumer(sharedQueue, 1));
        ces.submit(new Consumer(sharedQueue, 2));
         
        pes.shutdown();
        ces.shutdown();
    }
}

/* Different producers produces a stream of integers continuously to a shared queue, 
which is shared between all Producers and consumers */

class Producer implements Runnable {
    private final BlockingQueue<Integer> sharedQueue;
    private int threadNo;
    private Random random = new Random();
    public Producer(BlockingQueue<Integer> sharedQueue,int threadNo) {
        this.threadNo = threadNo;
        this.sharedQueue = sharedQueue;
    }
    @Override
    public void run() {
        // Producer produces a continuous stream of numbers for every 200 milli seconds
        while (true) {
            try {
                int number = random.nextInt(1000);
                System.out.println("Produced:" + number + ":by thread:"+ threadNo);
                sharedQueue.put(number);
                Thread.sleep(200);
            } catch (Exception err) {
                err.printStackTrace();
            }
        }
    }
}
/* Different consumers consume data from shared queue, which is shared by both producer and consumer threads */
class Consumer implements Runnable {
    private final BlockingQueue<Integer> sharedQueue;
    private int threadNo;
    public Consumer (BlockingQueue<Integer> sharedQueue,int threadNo) {
        this.sharedQueue = sharedQueue;
        this.threadNo = threadNo;
    }
    @Override
    public void run() {
        // Consumer consumes numbers generated from Producer threads continuously
        while(true){
            try {
                int num = sharedQueue.take();
                System.out.println("Consumed: "+ num + ":by thread:"+threadNo);
            } catch (Exception err) {
               err.printStackTrace();
            }
        }
    }   
}

Kod parçacığı:

import java.util.Set;

public class ThreadStatus {
    public static void main(String args[]) throws Exception {
        for (int i = 0; i < 5; i++){
            Thread t = new Thread(new MyThread());
            t.setName("MyThread:" + i);
            t.start();
        }
        int threadCount = 0;
        Set<Thread> threadSet = Thread.getAllStackTraces().keySet();
        for (Thread t : threadSet) {
            if (t.getThreadGroup() == Thread.currentThread().getThreadGroup()) {
                System.out.println("Thread :" + t + ":" + "state:" + t.getState());
                ++threadCount;
            }
        }
        System.out.println("Thread count started by Main thread:" + threadCount);
    }
}

class MyThread implements Runnable {
    public void run() {
        try {
            Thread.sleep(2000);
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

Yürütmeyi Duraklatma #

‘Thread.sleep’, geçerli iş parçacığının belirli bir süre boyunca yürütmeyi askıya almasına neden olur. Bu, bir uygulamanın veya bir bilgisayar sisteminde çalışıyor olabilecek diğer uygulamaların diğer iş parçacıkları için işlemci zamanını kullanılabilir hale getirmenin etkili bir yoludur. Thread sınıfında aşırı yüklenmiş iki “uyku” yöntemi vardır.

Uyku süresini milisaniye olarak belirten biri

public static void sleep(long millis) throws InterruptedException

Nanosaniye için uyku süresini belirten bir

public static void sleep(long millis, int nanos)

Yürütmeyi 1 saniye duraklatma

Thread.sleep(1000);

Bunun işletim sisteminin çekirdeğinin zamanlayıcısı için bir ipucu olduğuna dikkat etmek önemlidir. Bu mutlaka kesin olmayabilir ve bazı uygulamalar nanosaniye parametresini bile dikkate almaz (muhtemelen en yakın milisaniyeye yuvarlama).

Try/catch’e ‘Thread.sleep’ çağrısını dahil etmeniz ve ‘InterruptedException’ı yakalamanız önerilir.

Threadpool kullanarak iki `int’ dizisi ekleyin #

Bir Threadpool, her biri bu Threadlerden birinde yürütülecek olan bir Görev Kuyruğuna sahiptir.

Aşağıdaki örnek, bir Threadpool kullanarak iki “int” dizisinin nasıl ekleneceğini gösterir.

int[] firstArray = { 2, 4, 6, 8 };
int[] secondArray = { 1, 3, 5, 7 };
int[] result = { 0, 0, 0, 0 };

ExecutorService pool = Executors.newCachedThreadPool(); 

// Setup the ThreadPool:
// for each element in the array, submit a worker to the pool that adds elements
for (int i = 0; i < result.length; i++) {
    final int worker = i;
    pool.submit(() -> result[worker] = firstArray[worker] + secondArray[worker] );
}

// Wait for all Workers to finish:
try {
    // execute all submitted tasks
    pool.shutdown();
    // waits until all workers finish, or the timeout ends
    pool.awaitTermination(12, TimeUnit.SECONDS);
}
catch (InterruptedException e) {
    pool.shutdownNow(); //kill thread
}

System.out.println(Arrays.toString(result));

  1. Bu örnek tamamen açıklayıcıdır. Uygulamada, bu kadar küçük bir görev için iş parçacığı kullanarak herhangi bir hızlanma olmayacak. Görev oluşturma ve zamanlamanın genel giderleri, bir görevi çalıştırmak için geçen süreyi dolduracağından, bir yavaşlama olasıdır.

  2. Java 7 ve önceki sürümlerini kullanıyor olsaydınız, görevleri uygulamak için lambdalar yerine anonim sınıfları kullanırdınız.

Thread Kesintisi / Threadlerin Durdurulması #

Her Java İş Parçacığı, başlangıçta yanlış olan bir kesme bayrağına sahiptir. Bir iş parçacığını kesintiye uğratmak, aslında o bayrağı true olarak ayarlamaktan başka bir şey değildir. Bu iş parçacığında çalışan kod, bazen bayrağı kontrol edebilir ve buna göre hareket edebilir. Kod ayrıca tamamen yok sayabilir. Ama neden her Thread’in böyle bir bayrağı olsun ki? Sonuçta, bir iplik üzerinde bir boole bayrağına sahip olmak, ihtiyacımız olduğunda ve gerektiğinde kendimizi organize edebileceğimiz bir şeydir. Üzerinde çalıştıkları iş parçacığı kesintiye uğradığında özel bir şekilde davranan yöntemler vardır. Bu yöntemlere engelleme yöntemleri denir. Bunlar, iş parçacığını WAITING veya TIMED_WAITING durumuna sokan yöntemlerdir. Bir iş parçacığı bu durumdayken, onu kesintiye uğratmak, kesme bayrağının doğru olarak ayarlanması yerine, kesilen iş parçacığında bir InterruptedException oluşmasına neden olur ve iş parçacığı yeniden ÇALIŞTIRILABİLİR. Bir engelleme yöntemini çağıran kod, denetlenen bir istisna olduğu için InterruptedException ile uğraşmak zorunda kalır. Dolayısıyla ve dolayısıyla adı, bir kesme, bir WAIT’i kesintiye uğratma ve onu etkin bir şekilde bitirme etkisine sahip olabilir. Bir şekilde bekleyen tüm yöntemlerin (örn. Son olarak, kesme bayrağı ayarlanmış olan, bir engelleme yöntemine giren (yani, bekleme durumuna girmeye çalışan) bir iş parçacığı, hemen bir InterruptedException oluşturacak ve kesme bayrağı temizlenecektir.

Bu mekanikler dışında Java, kesintiye herhangi bir özel anlamsal anlam yüklemez. Kod, bir kesmeyi istediği şekilde yorumlamakta serbesttir. Ancak çoğu zaman kesinti, bir iş parçacığına en kısa zamanda çalışmayı durdurması gerektiğini bildirmek için kullanılır. Ancak, yukarıdan da anlaşılacağı gibi, çalışmayı durdurmak için bu kesintiye uygun şekilde tepki vermek o iş parçacığındaki koda bağlıdır. Bir iş parçacığını durdurmak bir işbirliğidir. Bir iş parçacığı kesintiye uğradığında, çalışan kodu yığın izlemenin derinliklerinde birkaç seviye olabilir. Çoğu kod bir engelleme yöntemi çağırmaz ve iş parçacığının durdurulmasını gereksiz yere geciktirmeyecek kadar zamanında tamamlanır. Çoğunlukla kesintiye yanıt vermekle ilgilenmesi gereken kod, hiçbir şey kalmayıncaya veya bu döngüyü durdurmak için bir bayrak ayarlanana kadar döngü işleme görevlerinde olan koddur. Muhtemelen sonsuz görevleri işleyen döngüler (yani prensipte çalışmaya devam ederler), döngüden çıkmak için kesme bayrağını kontrol etmelidir. Sonlu döngüler için anlambilim, tüm görevlerin sona ermeden önce tamamlanması gerektiğini veya bazı görevlerin işlenmeden bırakılması uygun olabilir. Engelleme yöntemlerini çağıran kod, InterruptedException ile uğraşmak zorunda kalacak. Anlamsal olarak mümkünse, sadece InterruptedException’ı yayabilir ve onu fırlatacağını bildirebilir. Bu nedenle, arayanlar açısından bir engelleme yöntemi haline gelir. İstisnayı yayamıyorsa, en azından kesinti bayrağını ayarlamalıdır, böylece yığının yukarısındaki arayanlar da iş parçacığının kesintiye uğradığını bilir. Bazı durumlarda, yöntemin InterruptedException’dan bağımsız olarak beklemeye devam etmesi gerekir; bu durumda, kesintiye uğrayan bayrağın ayarlanmasını bekleme bitene kadar geciktirmesi gerekir; bu, yöntemden çıkmadan önce kontrol edilmesi gereken yerel bir değişkenin ayarlanmasını içerebilir. sonra ipliğini kes.

Kesinti üzerine görevleri işlemeyi durduran kod örneği

class TaskHandler implements Runnable {
    
    private final BlockingQueue<Task> queue;

    TaskHandler(BlockingQueue<Task> queue) {
        this.queue = queue;
    }

    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) { // check for interrupt flag, exit loop when interrupted
            try {
                Task task = queue.take(); // blocking call, responsive to interruption
                handle(task);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // cannot throw InterruptedException (due to Runnable interface restriction) so indicating interruption by setting the flag
            }
        }
    }
    
    private void handle(Task task) {
        // actual handling
    }
}

class MustFinishHandler implements Runnable {

    private final BlockingQueue<Task> queue;

    MustFinishHandler(BlockingQueue<Task> queue) {
        this.queue = queue;
    }

    @Override
    public void run() {
        boolean shouldInterrupt = false;
        
        while (true) {
            try {
                Task task = queue.take();
                if (task.isEndOfTasks()) {
                    if (shouldInterrupt) {
                        Thread.currentThread().interrupt();
                    }
                    return;
                }
                handle(task);
            } catch (InterruptedException e) {
                shouldInterrupt = true; // must finish, remember to set interrupt flag when we're done
            }
        }
    }

    private void handle(Task task) {
        // actual handling
    }
}
