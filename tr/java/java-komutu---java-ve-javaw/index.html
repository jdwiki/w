Sözdizimi #

  • java [ <opt> ... ] <sınıf-adı> [ <argüman> ... ]

  • java [ <opt> ... ] -jar <jar-file-pathname> [ <argument> ... ]

Java uygulamasını komut satırından çalıştırmak için java komutu kullanılır. Herhangi bir Java SE JRE veya JDK’nın bir parçası olarak mevcuttur.

Windows sistemlerinde “java” komutunun iki çeşidi vardır:

  • “Java” çeşidi, uygulamayı yeni bir konsol penceresinde başlatır.
  • ‘javaw’ varyantı, yeni bir konsol penceresi oluşturmadan uygulamayı başlatır.

Diğer sistemlerde (örn. Linux, Mac OSX, UNIX) yalnızca “java” komutu sağlanır ve yeni bir konsol penceresi başlatmaz.

Sözdizimindeki “” sembolü, “java” komut satırındaki bir seçeneği belirtir. “Java Seçenekleri” ve “Yığın ve yığın boyutlandırma seçenekleri” konuları, en sık kullanılan seçenekleri kapsar. Diğerleri https://www.wikiod.com/tr/Java/jvm-bayraklar konusunda ele alınmıştır.

Giriş noktası sınıfları #

Bir Java giriş noktası sınıfı, aşağıdaki imza ve değiştiricilere sahip bir “main” yöntemine sahiptir:

public static void main(String[] args)

Sidenote: dizilerin çalışma şekli nedeniyle (String args[]) da olabilir

‘Java’ komutu sanal makineyi başlattığında, belirtilen giriş noktası sınıflarını yükler ve ‘main’i bulmaya çalışır. Başarılı olursa, komut satırındaki argümanlar Java String nesnelerine dönüştürülür ve bir dizide birleştirilir. Eğer ‘main’ bu şekilde çağrılırsa, dizi ’null’ değildir ve herhangi bir ’null’ girdi içermeyecektir.

Geçerli bir giriş noktası sınıf yöntemi aşağıdakileri yapmalıdır:

  • “main” olarak adlandırılın (büyük/küçük harfe duyarlı)
  • “Genel” ve “statik” olun
  • ‘void’ dönüş tipine sahip olun
  • String[] dizisiyle tek bir argümana sahip olun. Argüman mevcut olmalıdır ve birden fazla argümana izin verilmez.
  • Genel olun: tür parametrelerine izin verilmez.
  • Genel olmayan, üst düzey (iç içe veya iç içe olmayan) bir sınıfa sahip olmak

Sınıfı ‘public’ olarak ilan etmek gelenekseldir, ancak bu kesinlikle gerekli değildir. Java 5’ten itibaren, “ana” yöntemin argüman türü, bir dize dizisi yerine bir “Dize” değişkenleri olabilir. ‘main’ isteğe bağlı olarak istisnalar atabilir ve parametresi herhangi bir şey olarak adlandırılabilir, ancak geleneksel olarak ‘args’ olur.

JavaFX giriş noktaları #

Java 8’den itibaren ‘java’ komutu doğrudan bir JavaFX uygulamasını da başlatabilir. JavaFX, JavaFX etiketinde belgelenmiştir, ancak bir JavaFX giriş noktası aşağıdakileri yapmalıdır:

  • “javafx.application.Application” öğesini genişletin
  • “Soyut” değil, “genel” olun
  • Genel veya iç içe olmamalı
  • Açık veya örtük bir “genel” tartışmasız kurucuya sahip olun

Yürütülebilir bir JAR dosyasını çalıştırma #

Yürütülebilir JAR dosyaları, Java kodunu yürütülebilecek tek bir dosyada birleştirmenin en basit yoludur. *(Editör Notu: JAR dosyalarının oluşturulması ayrı bir Konu başlığında ele alınmalıdır.) *

“” yol adına sahip yürütülebilir bir JAR dosyanız olduğunu varsayarsak, onu aşağıdaki gibi çalıştırabilmelisiniz:

java -jar <jar-path>

Komut, komut satırı bağımsız değişkenleri gerektiriyorsa, bunları ““den sonra ekleyin. Örneğin:

java -jar <jar-path> arg1 arg2 arg3

“Java” komut satırında ek JVM seçenekleri sağlamanız gerekiyorsa, “-jar” seçeneğinden önce gitmeleri gerekir. “-jar” kullanırsanız “-cp” / “-classpath” seçeneğinin yok sayılacağını unutmayın. Uygulamanın sınıf yolu, JAR dosya bildirimi tarafından belirlenir.

Bir “ana” sınıf aracılığıyla Java uygulamalarını çalıştırma #

Bir uygulama yürütülebilir bir JAR olarak paketlenmediğinde, ‘java’ komut satırında bir giriş noktası sınıfı adını sağlamanız gerekir.

HelloWorld sınıfını çalıştırma #

“HelloWorld” örneği https://www.wikiod.com/tr/java/java-dilini-kullanmaya-baslama#İlk Java Programınızı Oluşturma içinde açıklanmıştır. Bir giriş noktası için gereksinimleri karşılayan ‘HelloWorld’ adlı tek bir sınıftan oluşur.

(Derlenmiş) “HelloWorld.class” dosyasının geçerli dizinde olduğunu varsayarsak, aşağıdaki gibi başlatılabilir:

java HelloWorld

Dikkat edilmesi gereken bazı önemli noktalar şunlardır:

  • Sınıfın adını sağlamalıyız: “.class” dosyasının veya “.java” dosyasının yol adını değil.
  • Sınıf bir paket içinde bildirilmişse (çoğu Java sınıfında olduğu gibi), o zaman java komutuna verdiğimiz sınıf adı tam sınıf adı olmalıdır. Örneğin, ‘com.example’ paketinde ‘SomeClass’ bildirilmişse, tam sınıf adı ‘com.example.SomeClass’ olacaktır.

Bir sınıf yolu belirtme #

‘Java -jar’ komut sözdiziminde kullanmadığımız sürece, ‘java’ komutu, sınıf yolunu arayarak yüklenecek sınıfı arar; https://www.wikiod.com/tr/java/snf-yolu adresine bakın. Yukarıdaki komut, geçerli dizin olan (veya dahil olan) varsayılan sınıf yoluna dayanmaktadır. -cp seçeneğini kullanarak kullanılacak sınıf yolunu belirterek bu konuda daha açık olabiliriz.

java -cp . HelloWorld

Bu, geçerli dizini (”.” neyi ifade eder) sınıf yolundaki tek giriş yapmak için söylüyor.

“-cp”, “java” komutu tarafından işlenen bir seçenektir. ‘Java’ komutuna yönelik tüm seçenekler, sınıf adından önce olmalıdır. Sınıftan sonraki her şey Java uygulaması için bir komut satırı argümanı olarak ele alınacak ve ‘main’ yöntemine iletilen ‘String[]’ içindeki uygulamaya geçirilecektir.

("-cp" seçeneği sağlanmazsa, “java”, “CLASSPATH” ortam değişkeni tarafından verilen sınıf yolunu kullanır. Bu değişken ayarlanmamış veya boşsa, “java” varsayılan sınıf yolu olarak “.” kullanır. )

‘Java’ komutunda sorun giderme #

Bu örnek, ‘java’ komutunun kullanılmasıyla ilgili yaygın hataları kapsar.

“Komut bulunamadı” #

Aşağıdaki gibi bir hata mesajı alırsanız:

java: command not found

‘java’ komutunu çalıştırmaya çalıştığınızda, bu, kabuğunuzun komut arama yolunda ‘java’ komutunun olmadığı anlamına gelir. Nedeni şunlar olabilir:

  • yüklü bir Java JRE veya JDK’nız yok,
  • kabuk başlatma dosyanızdaki PATH ortam değişkenini (doğru) güncellemediyseniz veya
  • geçerli kabuktaki ilgili başlatma dosyasını “kaynak yapmadınız”.

Atmanız gereken adımlar için “Java’yı Yükleme” bölümüne bakın.

“Ana sınıf bulunamadı veya yüklenemedi” #

Bu hata mesajı, belirttiğiniz giriş noktası sınıfını bulamadıysa / yükleyemediyse java komutu tarafından verilir. Genel anlamda, bunun gerçekleşmesinin üç geniş nedeni vardır:

  • Var olmayan bir giriş noktası sınıfı belirlediniz.
  • Sınıf var, ancak yanlış belirttiniz.
  • Sınıf var ve onu doğru bir şekilde belirttiniz, ancak sınıf yolu yanlış olduğu için Java onu bulamıyor.

Sorunu teşhis etmek ve çözmek için bir prosedür:

  1. Giriş noktası sınıfının tam adını öğrenin.

    • If you have source code for a class, then the full name consists of the package name and the simple class name. The instance the “Main” class is declared in the package “com.example.myapp” then its full name is “com.example.myapp.Main”.
    • If you have a compiled class file, you can find the class name by running javap on it.
    • If the class file is in a directory, you can infer the full class name from the directory names.
    • If the class file is in a JAR or ZIP file, you can infer the full class name from the file path in the JAR or ZIP file.

  2. java komutundan gelen hata mesajına bakın. Mesaj, ‘java’nın kullanmaya çalıştığı tam sınıf adıyla bitmelidir.

    • Check that it exactly matches the full classname for the entry-point class.
    • It should not end with “.java” or “.class”.
    • It should not contain slashes or any other character that is not legal in a Java identifier1.
    • The casing of the name should exactly match the full class name.

  3. Doğru sınıf adını kullanıyorsanız, sınıfın gerçekten sınıf yolunda olduğundan emin olun:

    • Work out the pathname that the classname maps to; see https://www.wikiod.com/tr/java/snf-yolu/19816/mapping-classnames-to-pathnames#t=201609101442436616455
    • Work out what the classpath is; see this example: https://www.wikiod.com/tr/java/snf-yolu/12852/different-ways-to-specify-the-classpath
    • Look at each of the JAR and ZIP files on the classpath to see if they contain a class with the required pathname.
    • Look at each directory to see if the pathname resolves to a file within the directory.

Sınıf yolunu elle kontrol etmek sorunu bulamadıysa, “-Xdiag” ve “-XshowSettings” seçeneklerini ekleyebilirsiniz. İlki, yüklenen tüm sınıfları listeler ve ikincisi, JVM için etkin sınıf yolunu içeren ayarları yazdırır.

Son olarak, bu sorunun bazı belirsiz nedenleri vardır:

  • Var olmayan bir sınıfı belirten bir “Ana-Sınıf” özniteliğine sahip yürütülebilir bir JAR dosyası.
  • Yanlış bir “Sınıf-Yolu” özniteliğine sahip yürütülebilir bir JAR dosyası.
  • Sınıf adından önceki seçenekleri2 karıştırırsanız, java komutu bunlardan birini sınıf adı olarak yorumlamaya çalışabilir.
  • Birisi Java stil kurallarını göz ardı ettiyse ve yalnızca büyük/küçük harf bakımından farklılık gösteren paket veya sınıf tanımlayıcıları kullandıysa ve dosya adlarındaki büyük/küçük harf durumunu önemsiz olarak değerlendiren bir platformda çalışıyorsanız.
  • Koddaki veya komut satırındaki sınıf adlarındaki homogliflerle ilgili sorunlar.

“Ana yöntem <name>” sınıfında bulunamadı #

Bu sorun, ‘java’ komutu aday gösterdiğiniz sınıfı bulup yükleyebildiğinde, ancak daha sonra bir giriş noktası yöntemi bulamadığında ortaya çıkar.

Üç olası açıklama vardır:

  • Yürütülebilir bir JAR dosyası çalıştırmaya çalışıyorsanız, JAR bildiriminde, geçerli bir giriş noktası sınıfı olmayan bir sınıfı belirten yanlış bir “Ana Sınıf” özniteliği vardır.
  • Java komutuna giriş noktası sınıfı olmayan bir sınıf söylediniz.
  • Giriş noktası sınıfı yanlış; daha fazla bilgi için https://www.wikiod.com/tr/java/java-komutu—java-ve-javaw#Giriş noktası sınıfları adresine bakın.

Java uygulamasını kitaplık bağımlılıklarıyla çalıştırma #

Bu, “ana sınıf” ve “yürütülebilir JAR” örneklerinin devamıdır.

Tipik Java uygulamaları, uygulamaya özel bir koddan ve uyguladığınız veya üçüncü taraflarca uygulanmış çeşitli yeniden kullanılabilir kitaplık kodlarından oluşur. İkincisi genellikle kitaplık bağımlılıkları olarak adlandırılır ve genellikle JAR dosyaları olarak paketlenir.

Java dinamik olarak bağlı bir dildir. Kitaplık bağımlılıkları olan bir Java uygulaması çalıştırdığınızda, JVM’nin sınıfları gerektiği gibi yükleyebilmesi için bağımlılıkların nerede olduğunu bilmesi gerekir. Genel olarak konuşursak, bununla başa çıkmanın iki yolu vardır:

  • Uygulama ve bağımlılıkları, gerekli tüm sınıfları ve kaynakları içeren tek bir JAR dosyasında yeniden paketlenebilir.

  • JVM’ye, çalışma zamanı sınıf yolu aracılığıyla bağımlı JAR dosyalarını nerede bulacağı söylenebilir.

Yürütülebilir bir JAR dosyası için, çalışma zamanı sınıf yolu, “Sınıf-Yolu” bildirim özniteliği tarafından belirtilir. (Editoryal Not: Bu, ‘jar’ komutunda ayrı bir Konuda açıklanmalıdır.) Aksi takdirde, çalışma zamanı sınıf yolunun ‘-cp’ seçeneği veya ‘CLASSPATH’ ortam değişkeni kullanılarak sağlanması gerekir.

Örneğin, “myApp.jar” dosyasında giriş noktası sınıfı “com.example.MyApp” olan bir Java uygulamamız olduğunu varsayalım. Ayrıca uygulamanın “lib/library1.jar” ve “lib/library2.jar” kitaplık JAR dosyalarına bağlı olduğunu varsayalım. Komut satırında aşağıdaki gibi java komutunu kullanarak uygulamayı başlatabiliriz:

$ # Alternative 1 (preferred)
$ java -cp myApp.jar:lib/library1.jar:lib/library2.jar com.example.MyApp

$ # Alternative 2
$ export CLASSPATH=myApp.jar:lib/library1.jar:lib/library2.jar
$ java com.example.MyApp

(Windows’ta, sınıf yolu ayırıcısı olarak “:” yerine “;” kullanırsınız ve (yerel) “CLASSPATH” değişkenini “dışa aktarma” yerine “set” kullanarak ayarlarsınız.)

Bir Java geliştiricisi bu konuda rahat olsa da, “kullanıcı dostu” değildir. Bu nedenle, kullanıcının bilmesi gerekmeyen ayrıntıları gizlemek için basit bir kabuk komut dosyası (veya Windows toplu iş dosyası) yazmak yaygın bir uygulamadır. Örneğin, aşağıdaki kabuk betiğini “myApp” adlı bir dosyaya koyarsanız, onu yürütülebilir hale getirir ve komut arama yolundaki bir dizine koyarsanız:

#!/bin/bash
# The 'myApp' wrapper script

export DIR=/usr/libexec/myApp
export CLASSPATH=$DIR/myApp.jar:$DIR/lib/library1.jar:$DIR/lib/library2.jar
java com.example.MyApp

sonra aşağıdaki gibi çalıştırabilirsiniz:

$ myApp arg1 arg2 ...

Komut satırındaki herhangi bir argüman, Java uygulamasına "$@" genişletmesi yoluyla iletilecektir. (Sözdizimi farklı olsa da, bir Windows toplu iş dosyasıyla benzer bir şey yapabilirsiniz.)

Bağımsız değişkenlerdeki boşluklar ve diğer özel karakterler #

Her şeyden önce, argümanlardaki boşlukları işleme sorunu aslında bir Java sorunu DEĞİLDİR. Bunun yerine, bir Java programını çalıştırırken kullandığınız komut kabuğu tarafından ele alınması gereken bir sorundur.

Örnek olarak, bir dosyanın boyutunu yazdıran aşağıdaki basit programa sahip olduğumuzu varsayalım:

import java.io.File;

public class PrintFileSizes {
    
    public static void main(String[] args) {
        for (String name: args) {
            File file = new File(name);
            System.out.println("Size of '" + file + "' is " + file.size());
        }
    }
}

Şimdi, yol adında boşluk bulunan bir dosyanın boyutunu yazdırmak istediğimizi varsayalım; Örneğin. /home/steve/Test Dosyası.txt. Komutu şu şekilde çalıştırırsak:

$ java PrintFileSizes /home/steve/Test File.txt

kabuk, /home/steve/Test File.txt dosyasının aslında bir yol adı olduğunu bilmeyecek. Bunun yerine, Java uygulamasına, ilgili dosya boyutlarını bulmaya çalışacak ve bu yollara sahip dosyalar (muhtemelen) bulunmadığından başarısız olacak 2 farklı argüman iletecektir.

POSIX kabuğu kullanan çözümler #

POSIX kabukları, “sh” ile “bash” ve “ksh” gibi türevleri içerir. Bu kabuklardan birini kullanıyorsanız, argümanı alıntılayarak sorunu çözebilirsiniz.

$ java PrintFileSizes "/home/steve/Test File.txt"

Yol adının etrafındaki çift tırnak, kabuğa bunun tek bir argüman olarak iletilmesi gerektiğini söyler. Bu olduğunda alıntılar kaldırılacaktır. Bunu yapmanın birkaç yolu daha var:

$ java PrintFileSizes '/home/steve/Test File.txt'

Tek (düz) tırnaklar, argüman içindeki çeşitli açılımları da bastırmaları dışında çift tırnak gibi değerlendirilir.

$ java PrintFileSizes /home/steve/Test\ File.txt

Bir ters eğik çizgi, aşağıdaki boşluktan kaçar ve bunun bir bağımsız değişken ayırıcısı olarak yorumlanmamasına neden olur.

Bağımsız değişkenlerdeki diğer özel karakterlerle nasıl başa çıkılacağına ilişkin açıklamalar da dahil olmak üzere daha kapsamlı belgeler için lütfen Bash belgelerindeki alıntılama konusuna bakın.

Windows için Çözüm #

Windows için temel sorun, işletim sistemi düzeyinde, bağımsız değişkenlerin bir alt sürece tek bir dize (kaynak) olarak iletilmesidir. Bu, komut satırının ayrıştırılmasının (veya yeniden ayrıştırılmasının) nihai sorumluluğunun programa veya çalışma zamanı kitaplıklarına ait olduğu anlamına gelir. Bir sürü tutarsızlık var.

Java durumunda, uzun bir hikayeyi kısa kesmek için:

  • Bir ‘java’ komutunda bir argümanın etrafına çift tırnak işareti koyabilirsiniz ve bu, içinde boşluk olan argümanları iletmenize izin verecektir.

  • Görünüşe göre, ‘java’ komutunun kendisi komut dizesini ayrıştırıyor ve aşağı yukarı doğru yapıyor

  • Bununla birlikte, bunu bir toplu iş dosyasında ‘SET’ kullanımı ve değişken ikamesi ile birleştirmeye çalıştığınızda, çift tırnakların kaldırılıp kaldırılmayacağı gerçekten karmaşık hale gelir.

  • “cmd.exe” kabuğunun görünüşe göre başka kaçış mekanizmaları var; Örneğin. çift ​​tırnakları iki katına çıkarmak ve ^ çıkışlarını kullanmak.

Java Seçenekleri #

“Java” komutu çok çeşitli seçenekleri destekler:

  • Tüm seçenekler tek bir kısa çizgi veya eksi işaretiyle (-) başlar: GNU/Linux’un “–” kullanma kuralı “uzun” seçenekler için desteklenmez.

  • Seçenekler, tanınmak için <classname> veya -jar <jarfile> argümanından önce görünmelidir. Onlardan sonraki tüm argümanlar, çalıştırılmakta olan Java uygulamasına iletilecek argümanlar olarak ele alınacaktır.

  • “-X” veya “-XX” ile başlamayan seçenekler standart seçeneklerdir. Herhangi bir standart seçeneği desteklemek için tüm Java uygulamalarına1 güvenebilirsiniz.

  • “-X” ile başlayan seçenekler standart olmayan seçeneklerdir ve bir Java sürümünden diğerine geri çekilebilir.

  • ‘-XX’ ile başlayan seçenekler gelişmiş seçeneklerdir ve geri çekilebilir.

-D ile sistem özelliklerini ayarlama #

‘-D=’ seçeneği, sistem ‘Properties’ nesnesindeki bir özelliği ayarlamak için kullanılır. Bu parametre, farklı özellikler ayarlamak için tekrar edilebilir.

Bellek, Yığın ve Çöp Toplayıcı seçenekleri #

Yığın ve yığın boyutlarını kontrol etmek için ana seçenekler https://www.wikiod.com/tr/java/java-bellek-yonetimi#Yığın, PermGen ve Yığın boyutlarını ayarlama içinde belgelenmiştir. (Editör notu: Çöp Toplayıcı seçenekleri aynı başlıkta açıklanmalıdır.)

Onayları etkinleştirme ve devre dışı bırakma #

“-ea” ve “-da” seçenekleri sırasıyla Java “assert” denetimini etkinleştirir ve devre dışı bırakır:

  • Tüm onay denetimi varsayılan olarak devre dışıdır.
  • -ea seçeneği tüm iddiaların kontrol edilmesini sağlar
  • -ea:<packagename>... bir paketteki ve tüm alt paketlerdeki iddiaların kontrol edilmesini sağlar.
  • -ea:<classname>..., bir sınıftaki iddiaların kontrol edilmesini sağlar.
  • -da seçeneği tüm iddiaların kontrol edilmesini devre dışı bırakır
  • -da:<paketadı>... bir paketteki ve tüm alt paketlerdeki iddiaların kontrol edilmesini devre dışı bırakır.
  • -da:<classname>..., bir sınıftaki iddiaların kontrol edilmesini devre dışı bırakır.
  • -esa seçeneği tüm sistem sınıflarının kontrol edilmesini sağlar.
  • -dsa seçeneği tüm sistem sınıflarını kontrol etmeyi devre dışı bırakır.

Seçenekler birleştirilebilir. Örneğin.

$ # Enable all assertion checking in non-system classes 
$ java -ea -dsa MyApp

$ # Enable assertions for all classes in a package except for one.
$ java -ea:com.wombat.fruitbat... -da:com.wombat.fruitbat.Brickbat MyApp

Onay denetiminin etkinleştirilmesinin bir Java programlamasının davranışını değiştirebileceğini unutmayın.

  • Genel olarak uygulamayı yavaşlatmakla yükümlüdür.
  • Çok iş parçacıklı bir uygulamada iş parçacıklarının zamanlamasını değiştirebilecek belirli yöntemlerin çalışmasının daha uzun sürmesine neden olabilir.
  • Hafıza anormalliklerinin kaybolmasına neden olabilecek tesadüfi önce-olan ilişkileri ortaya çıkarabilir.
  • Yanlış uygulanmış bir “iddia” ifadesi istenmeyen yan etkilere neden olabilir.

VM türünü seçme #

“-client” ve “-server” seçenekleri, HotSpot VM’nin iki farklı biçimi arasında seçim yapmanızı sağlar:

  • “İstemci” formu, kullanıcı uygulamalarına göre ayarlanmıştır ve daha hızlı başlatma sunar.
  • “Sunucu” formu, uzun süre çalışan uygulamalar için ayarlanmıştır. JVM “ısınması” sırasında istatistiklerin yakalanması daha uzun sürer, bu da JIT derleyicisinin yerel kodu optimize etme işini daha iyi yapmasına olanak tanır.

Varsayılan olarak JVM, platformun özelliklerine bağlı olarak mümkünse 64 bit modunda çalışacaktır. “-d32” ve “-d64” seçenekleri, modu açıkça seçmenize olanak tanır.

1 - “Java” komutu için resmi kılavuza bakın. Bazen standart bir seçenek “değişebilir” olarak tanımlanır.
