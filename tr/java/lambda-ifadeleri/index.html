Lambda ifadeleri, bir ifade kullanarak tek yöntemli bir arabirim uygulamanın açık ve özlü bir yolunu sağlar. Oluşturmanız ve sürdürmeniz gereken kod miktarını azaltmanıza izin verirler. Anonim sınıflara benzer olmakla birlikte, kendi başlarına hiçbir tür bilgisi yoktur. Tür çıkarımının gerçekleşmesi gerekir.

Yöntem referansları, ifadeler yerine mevcut yöntemleri kullanarak işlevsel arayüzleri uygular. Onlar da lambda ailesine aittir.

Sözdizimi #

  • () -> { dönüş ifadesi; } // Bir değer döndürmek için fonksiyon gövdesi ile sıfır-arity.
  • () -> ifade // Yukarıdaki bildirimin kısaltması; ifadeler için noktalı virgül yoktur.
  • () -> { function-body } // İşlemleri gerçekleştirmek için lambda ifadesinde yan etki.
  • parametreAdı -> ifade // Bir-arity lambda ifadesi. Tek argümanlı lambda ifadelerinde parantez kaldırılabilir.
  • (ParametreAdı yazın, ikinciParametreAdı yazın, …) -> ifade // solda listelenen parametrelerle bir ifadeyi değerlendiren lambda
  • (parameterName, secondParameterName, …) -> ifade // Parametre adları için parametre türlerini kaldıran kısayol. Yalnızca, verilen parametre listesi boyutunun beklenen işlevsel arabirimlerin boyutundan biriyle (ve yalnızca biriyle) eşleştiği derleyici tarafından çıkarılabilen bağlamlarda kullanılabilir.

Java lambdalarına giriş #

Fonksiyonel Arayüzler #

Lambda’lar yalnızca, yalnızca bir soyut yöntemle bir arabirim olan işlevsel bir arabirimde çalışabilir. İşlevsel arabirimler, herhangi bir sayıda varsayılan veya statik yönteme sahip olabilir. (Bu nedenle bazen Tek Soyut Yöntem Arayüzleri veya SAM Arayüzleri olarak anılırlar).

interface Foo1 {
    void bar();
}

interface Foo2 {
    int bar(boolean baz);
}

interface Foo3 {
    String bar(Object baz, int mink);
}

interface Foo4 {
    default String bar() { // default so not counted
        return "baz";
    }
    void quux();
}

İşlevsel bir arayüz bildirirken @FunctionalInterface ek açıklaması eklenebilir. Bunun özel bir etkisi yoktur, ancak bu açıklama işlevsel olmayan bir arabirime uygulanırsa bir derleyici hatası oluşturulur, bu nedenle arabirimin değiştirilmemesi gerektiğini hatırlatır.

@FunctionalInterface
interface Foo5 {
    void bar();
}

@FunctionalInterface
interface BlankFoo1 extends Foo3 { // inherits abstract method from Foo3
}

@FunctionalInterface
interface Foo6 {
    void bar();
    boolean equals(Object obj); // overrides one of Object's method so not counted
}

Tersine, bu, birden fazla soyut yönteme sahip olduğundan, işlevsel bir arabirim değildir:

interface BadFoo {
    void bar();
    void quux(); // <-- Second method prevents lambda: which one should 
                 // be considered as lambda?
}

Bu ayrıca işlevsel bir arabirim değildir, çünkü herhangi bir yöntemi yoktur:

interface BlankFoo2 { }

Aşağıdakilere dikkat edin. Diyelim ki sahipsin

arayüz Ebeveyn { public int parentMethod(); }

ve

arayüz Çocuk, Ebeveyn'i genişletir { public int ChildMethod(); }

O zaman Child işlevsel bir arayüz olamaz çünkü iki belirtilen yöntemler.

Java 8 ayrıca java.util.function paketinde bir dizi genel şablonlu işlevsel arabirim sağlar. Örneğin, “Predicate” yerleşik arabirimi, “T” türünde bir değer giren ve bir “boolean” döndüren tek bir yöntemi sarar.

Lambda İfadeleri #

Lambda ifadesinin temel yapısı şöyledir:

‘fi’ daha sonra, ‘FunctionalInterface’ uygulayan ve bir yöntemin tanımının ‘{ System.out.println(“Hello”); }`. Başka bir deyişle, yukarıdakiler çoğunlukla şuna eşdeğerdir:

FunctionalInterface fi = new FunctionalInterface() {
    @Override
    public void theOneMethod() {
        System.out.println("Hello");
    }
};

Lambda, anonim sınıfa yalnızca “çoğunlukla eşdeğerdir” çünkü bir lambdada “this”, “super” veya “toString()” gibi ifadelerin anlamı, yeni oluşturulan nesneye değil, atamanın gerçekleştiği sınıfa başvurur. .

Bir lambda kullanırken yöntemin adını belirtemezsiniz - ancak buna gerek duymamalısınız, çünkü işlevsel bir arabirimin yalnızca bir soyut yöntemi olması gerekir, bu nedenle Java bunu geçersiz kılar.

Lambda türünün kesin olmadığı durumlarda (örneğin aşırı yüklenmiş yöntemler), derleyiciye türünün ne olması gerektiğini söylemek için lambdaya bir döküm ekleyebilirsiniz, şöyle:

Object fooHolder = (Foo1) () -> System.out.println("Hello");
System.out.println(fooHolder instanceof Foo1); // returns true

İşlevsel arayüzün tek yöntemi parametreler alıyorsa, bunların yerel biçimsel adları lambda’nın parantezleri arasında görünmelidir. Arayüzden alındığı için parametrenin tipini bildirmeye veya döndürmeye gerek yoktur (her ne kadar isterseniz parametre tiplerini bildirmek bir hata değildir). Böylece, bu iki örnek eşdeğerdir:

Foo2 longFoo = new Foo2() {
    @Override
    public int bar(boolean baz) {
        return baz ? 1 : 0;
    }
};
Foo2 shortFoo = (x) -> { return x ? 1 : 0; };

İşlevin yalnızca bir bağımsız değişkeni varsa, bağımsız değişkenin etrafındaki parantezler atlanabilir:

Foo2 np = x -> { return x ? 1 : 0; }; // okay
Foo3 np2 = x, y -> x.toString() + y // not okay

Örtük Getiriler #

Bir lambda içine yerleştirilen kod, bir ifade yerine bir Java _ifadesiyse, ifadenin değerini döndüren bir yöntem olarak kabul edilir. Böylece, aşağıdaki ikisi eşdeğerdir:

IntUnaryOperator addOneShort = (x) -> (x + 1);
IntUnaryOperator addOneLong = (x) -> { return (x + 1); };

Yerel Değişkenlere Erişim (değer kapanışları) #

Lambdalar anonim sınıflar için sözdizimsel kestirme olduklarından, çevreleyen kapsamdaki yerel değişkenlere erişmek için aynı kuralları takip ederler; değişkenler ‘son’ olarak ele alınmalı ve lambda içinde değiştirilmemelidir.

IntUnaryOperator makeAdder(int amount) {
    return (x) -> (x + amount); // Legal even though amount will go out of scope
                                // because amount is not modified
}

IntUnaryOperator makeAccumulator(int value) {
    return (x) -> { value += x; return value; }; // Will not compile
}

Değişen bir değişkeni bu şekilde sarmak gerekirse, değişkenin bir kopyasını tutan normal bir nesne kullanılmalıdır. https://www.wikiod.com/tr/java/lambda-ifadeleri#Lambda ifadeleriyle Java Kapanışları. adresinde daha fazlasını okuyun

Lambdaları Kabul Etme #

Bir lambda bir arabirimin uygulaması olduğundan, bir yöntemin bir lambda kabul etmesi için özel bir şey yapılmasına gerek yoktur: işlevsel bir arabirim alan herhangi bir işlev bir lambda da kabul edebilir.

public void passMeALambda(Foo1 f) {
    f.bar();
}
passMeALambda(() -> System.out.println("Lambda called"));

Lambda İfadesinin Türü #

Bir lambda ifadesinin kendi başına belirli bir türü yoktur. Bir dönüş değerinin türü ile birlikte parametre türlerinin ve sayısının bazı tür bilgilerini iletebileceği doğru olsa da, bu tür bilgiler yalnızca hangi türlere atanabileceğini sınırlayacaktır. Lambda, aşağıdaki yollardan biriyle işlevsel bir arabirim türüne atandığında bir tür alır:

  • Bir fonksiyonel tipe doğrudan atama, örn. myPredicate = s -> s.isEmpty()
  • İşlevsel bir türe sahip bir parametre olarak iletmek, ör. stream.filter(s -> s.isEmpty())
  • İşlevsel bir tür döndüren bir işlevden döndürme, ör. dönüş s -> s.isEmpty()
  • İşlevsel bir türe dökme, ör. (Yüklem<String>) s -> s.isEmpty()

İşlevsel bir türe böyle bir atama yapılıncaya kadar lambda’nın kesin bir türü yoktur. Örneklemek için, o -> o.isEmpty() lambda ifadesini düşünün. Aynı lambda ifadesi birçok farklı işlevsel türe atanabilir:

Predicate<String> javaStringPred = o -> o.isEmpty();
Function<String, Boolean> javaFunc = o -> o.isEmpty();
Predicate<List> javaListPred = o -> o.isEmpty();
Consumer<String> javaStringConsumer = o -> o.isEmpty(); // return value is ignored!
com.google.common.base.Predicate<String> guavaPredicate = o -> o.isEmpty();

Artık atandıklarına göre, lambda ifadeleri aynı görünse de gösterilen örnekler tamamen farklı türlerdedir ve birbirlerine atanamazlar.

Koleksiyonu Sıralamak için Lambda İfadelerini Kullanma #

Listeleri sıralama #

Java 8’den önce, bir listeyi sıralarken, java.util.Comparator arabirimini anonim (veya adlandırılmış) bir sınıfla uygulamak gerekliydi1:

List<Person> people = ...
Collections.sort(
    people,
    new Comparator<Person>() {
        public int compare(Person p1, Person p2){
            return p1.getFirstName().compareTo(p2.getFirstName());
        }
    }
);
son sürüm

Java 8 ile başlayarak, anonim sınıf bir lambda ifadesi ile değiştirilebilir. Derleyici bunları otomatik olarak çıkaracağından, “p1” ve “p2” parametrelerinin türlerinin dışarıda bırakılabileceğine dikkat edin:

Collections.sort(
    people, 
    (p1, p2) -> p1.getFirstName().compareTo(p2.getFirstName())
);

Örnek, :: (çift iki nokta üst üste) sembolü kullanılarak ifade edilen Comparator.comparing ve yöntem referansları kullanılarak basitleştirilebilir.

Collections.sort(
    people,
    Comparator.comparing(Person::getFirstName)
);

Statik bir içe aktarma, bunu daha kısa bir şekilde ifade etmemize izin verir, ancak bunun genel okunabilirliği iyileştirip iyileştirmediği tartışmalıdır:

import static java.util.Collections.sort;
import static java.util.Comparator.comparing;
//...
sort(people, comparing(Person::getFirstName));

Bu şekilde inşa edilen karşılaştırıcılar da birbirine zincirlenebilir. Örneğin, kişileri adlarına göre karşılaştırdıktan sonra, aynı ada sahip kişiler varsa, ’thenComparing’ yöntemi de soyadına göre karşılaştır:

sort(people, comparing(Person::getFirstName).thenComparing(Person::getLastName));

1 - Collections.sort(…) öğesinin yalnızca “Liste"nin alt türleri olan koleksiyonlarda çalıştığını unutmayın. “Set” ve “Collection” API’leri, öğelerin sıralandığı anlamına gelmez.

Haritaları sıralama #

Bir “HashMap"in girişlerini benzer bir şekilde değere göre sıralayabilirsiniz. (Hedef olarak bir ‘LinkedHashMap’ kullanılması gerektiğini unutmayın. Sıradan bir ‘HashMap’ içindeki anahtarlar sırasızdır.)

Map<String, Integer> map = new HashMap();  // ... or any other Map class
// populate the map
map = map.entrySet()
    .stream()
    .sorted(Map.Entry.<String, Integer>comparingByValue())
    .collect(Collectors.toMap(k -> k.getKey(), v -> v.getValue(),
                              (k, v) -> k, LinkedHashMap::new));

3: https://www.wikiod.com/tr/java/lambda-ifadeleri#Yöntem Referansları

Yöntem Referansları #

Yöntem referansları, uyumlu bir işlevsel arabirime bağlı olan önceden tanımlanmış statik veya örnek yöntemlerin, anonim bir lambda ifadesi yerine bağımsız değişkenler olarak geçirilmesine izin verir.

Bir modelimiz olduğunu varsayalım:

class Person {
    private final String name;
    private final String surname;

    public Person(String name, String surname){
        this.name = name;
        this.surname = surname;
    }

    public String getName(){ return name; }
    public String getSurname(){ return surname; }
}

List<Person> people = getSomePeople();

Örnek yöntemi referansı (rastgele bir örneğe) #

people.stream().map(Person::getName)

eşdeğer lambda:

people.stream().map(person -> person.getName())

Bu örnekte, “Person” türündeki “getName()” örnek yöntemine bir yöntem başvurusu iletilmektedir. Koleksiyon türünde olduğu bilindiğinden, örnekteki yöntem (daha sonra bilinir) çağrılır.

Örnek yöntemi referansı (belirli bir örneğe) #

people.forEach(System.out::println);

“System.out”, “PrintStream"in bir örneği olduğundan, bu özel örneğe bir yöntem referansı argüman olarak iletilir.

eşdeğer lambda:

people.forEach(person -> System.out.println(person));

Statik yöntem referansı #

Ayrıca akışları dönüştürmek için statik yöntemlere referanslar uygulayabiliriz:

List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5, 6);
numbers.stream().map(String::valueOf)

Bu örnek, “String” türündeki statik “valueOf()” yöntemine bir başvuru iletir. Bu nedenle, koleksiyondaki örnek nesnesi, ‘valueOf()’ öğesine bir argüman olarak iletilir.

eşdeğer lambda:

 numbers.stream().map(num -> String.valueOf(num))

Bir kurucuya referans #

List<String> strings = Arrays.asList("1", "2", "3");
strings.stream().map(Integer::new)

Öğelerin koleksiyona nasıl toplanacağını görmek için Akış Öğelerini Koleksiyonda Topla bölümünü okuyun.

Burada, argüman olarak sağlanan dize verilen bir tamsayı oluşturmak için “Tamsayı” türündeki tek Dize argüman oluşturucusu kullanılmaktadır. Bu durumda, dize bir sayıyı temsil ettiği sürece, akış Tamsayılarla eşlenecektir. eşdeğer lambda:

strings.stream().map(s -> new Integer(s));

Kopya kağıdı #

Yöntem Referans Formatı Kod Eşdeğer Lambda
Statik yöntem TürAdı::yöntem (args) -> TypeName.method(args)
Statik olmayan yöntem (örnekte*) “örnek::yöntem” (args) -> örnek.method(args)
Statik olmayan yöntem (örnek yok) TürAdı::yöntem (örnek, argümanlar) -> örnek.method(args)
Yapıcı** TürAdı::yeni (args) -> yeni TypeName(args)
Dizi oluşturucu TürAdı[]::yeni (int size) -> yeni TypeName[size]

* “örnek”, bir örneğe referans olarak değerlendirilen herhangi bir ifade olabilir, ör. “getInstance()::yöntem”, “bu::yöntem”

** “TypeName” statik olmayan bir iç sınıfsa, yapıcı başvurusu yalnızca bir dış sınıf örneğinin kapsamında geçerlidir

Birden çok arayüz uygulama #

Bazen birden fazla arabirim uygulayan bir lambda ifadesine sahip olmak isteyebilirsiniz. Bu, çoğunlukla işaretleyici arabirimlerinde (java.io.Serializable gibi) kullanışlıdır çünkü soyut yöntemler ekleyin.

Örneğin, özel bir “Karşılaştırıcı” ile bir TreeSet oluşturmak ve ardından onu seri hale getirmek ve ağ üzerinden göndermek istiyorsunuz. Önemsiz yaklaşım:

TreeSet<Long> ts = new TreeSet<>((x, y) -> Long.compare(y, x));

karşılaştırıcının lambdası `` Seri hale getirilebilir ’’ uygulamadığından çalışmaz. Bunu, kesişim türlerini kullanarak ve bu lambda’nın seri hale getirilebilir olması gerektiğini açıkça belirterek düzeltebilirsiniz:

TreeSet<Long> ts = new TreeSet<>(
    (Comparator<Long> & Serializable) (x, y) -> Long.compare(y, x));

Sık sık kesişim türleri kullanıyorsanız (örneğin, neredeyse her şeyin seri hale getirilebilir olması gereken Apache Spark gibi bir çerçeve kullanıyorsanız), boş oluşturabilirsiniz. arayüzler ve bunları kodunuzda kullanın:

public interface SerializableComparator extends Comparator<Long>, Serializable {}

public class CustomTreeSet {
  public CustomTreeSet(SerializableComparator comparator) {}
}

Bu şekilde, geçen karşılaştırıcının seri hale getirilebileceği garanti edilir.

Lambda - Dinleyici Örneği #

Anonim sınıf dinleyicisi

Java 8’den önce, aşağıdaki kodda gösterildiği gibi, bir JButton’un click olayını işlemek için anonim bir sınıfın kullanılması çok yaygındır. Bu örnek, “btn.addActionListener” kapsamında anonim bir dinleyicinin nasıl uygulanacağını gösterir.

JButton btn = new JButton("My Button");
btn.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("Button was pressed");
    }
});

Lambda dinleyicisi

“ActionListener” arabirimi yalnızca bir “actionPerformed()” yöntemini tanımladığından, bu işlevsel bir arabirimdir, yani standart kodu değiştirmek için Lambda ifadelerini kullanmak için bir yer vardır. Yukarıdaki örnek, Lambda ifadeleri kullanılarak aşağıdaki gibi yeniden yazılabilir:

JButton btn = new JButton("My Button");
btn.addActionListener(e -> {
    System.out.println("Button was pressed");
});

Lambda ifadeleriyle Java Kapanışları. #

Bir lambda ifadesi, çevreleyen bir kapsamın (genel veya yerel) değişkenlerine başvurduğunda bir lambda kapanışı oluşturulur. Bunu yapmanın kuralları, satır içi yöntemler ve anonim sınıflar için olanlarla aynıdır.

Bir lambda içinde kullanılan, bir kapalı kapsamdan yerel değişkenler ‘final’ olmalıdır. Java 8 (lambdaları destekleyen en eski sürüm) ile, bunların dış bağlamda son olarak ilan edilmeleri gerekmez, ancak bu şekilde ele alınmaları gerekir. Örneğin:

int n = 0; // With Java 8 there is no need to explicit final
Runnable r = () -> { // Using lambda
    int i = n;
    // do something
};

’n’ değişkeninin değeri değişmediği sürece bu yasaldır. Değişkeni lambda içinde veya dışında değiştirmeye çalışırsanız, aşağıdaki derleme hatasını alırsınız:

“lambda ifadesinden başvurulan yerel değişkenler final veya etkili bir şekilde final olmalıdır”.

Örneğin:

int n = 0;
Runnable r = () -> { // Using lambda
    int i = n;
    // do something
};
n++; // Will generate an error.

Bir lambda içinde değişen bir değişken kullanmak gerekirse, normal yaklaşım, değişkenin ‘son’ bir kopyasını bildirmek ve kopyayı kullanmaktır. Örneğin

int n = 0;
final int k = n; // With Java 8 there is no need to explicit final
Runnable r = () -> { // Using lambda
    int i = k;
    // do something
};
n++;      // Now will not generate an error
r.run();  // Will run with i = 0 because k was 0 when the lambda was created

Doğal olarak, lambdanın gövdesi orijinal değişkendeki değişiklikleri görmez.

Java’nın gerçek kapatmaları desteklemediğini unutmayın. Bir Java lambda, somutlaştırıldığı ortamdaki değişiklikleri görmesine izin verecek şekilde oluşturulamaz. Çevresini gözlemleyen veya üzerinde değişiklik yapan bir kapatma uygulamak istiyorsanız, onu normal bir sınıf kullanarak simüle etmelisiniz. Örneğin:

// Does not compile ...
public IntUnaryOperator createAccumulator() {
    int value = 0;
    IntUnaryOperator accumulate = (x) -> { value += x; return value; };
    return accumulate;
}

Yukarıdaki örnek, daha önce tartışılan nedenlerle derlenmeyecektir. Derleme hatasını aşağıdaki gibi çözebiliriz:

// Compiles, but is incorrect ...
public class AccumulatorGenerator {
    private int value = 0;

    public IntUnaryOperator createAccumulator() {
        IntUnaryOperator accumulate = (x) -> { value += x; return value; };
        return accumulate;
    }
}

Sorun şu ki, bunun örneklerin işlevsel ve durumsuz olması gerektiğini belirten IntUnaryOperator arayüzü için tasarım sözleşmesini bozuyor. Böyle bir kapatma, işlevsel nesneleri kabul eden yerleşik işlevlere geçirilirse, çökmelere veya hatalı davranışlara neden olabilir. Değişebilir durumu kapsayan kapanışlar, normal sınıflar olarak uygulanmalıdır. Örneğin.

// Correct ...
public class Accumulator {
   private int value = 0;

   public int accumulate(int x) {
      value += x;
      return value;
   }
}

Kendi işlevsel arayüzünüzle lambda ifadesini kullanma #

Lambda’lar, tekli yöntem arabirimleri için satır içi uygulama kodu ve normal değişkenlerle yaptığımız gibi bunları aktarma yeteneği sağlamak içindir. Biz onlara Fonksiyonel Arayüz diyoruz.

Örneğin, anonim sınıfta bir Runnable yazmak ve bir Thread başlatmak şuna benzer:

//Old way
new Thread(
        new Runnable(){
            public void run(){
                System.out.println("run logic...");
            }
        }
).start();

//lambdas, from Java 8
new Thread(
        ()-> System.out.println("run logic...")
).start();

Şimdi, yukarıdakiyle uyumlu olarak, bazı özel arayüzünüz olduğunu varsayalım:

interface TwoArgInterface {
    int operate(int a, int b);
}

Bu arayüzün kodunuzda uygulanmasını sağlamak için lambda’yı nasıl kullanıyorsunuz? Yukarıda gösterilen Çalıştırılabilir örnekle aynı. Aşağıdaki sürücü programına bakın:

public class CustomLambda {
    public static void main(String[] args) {

        TwoArgInterface plusOperation = (a, b) -> a + b;
        TwoArgInterface divideOperation = (a,b)->{
            if (b==0) throw new IllegalArgumentException("Divisor can not be 0");
            return a/b;
        };

        System.out.println("Plus operation of 3 and 5 is: " + plusOperation.operate(3, 5));
        System.out.println("Divide operation 50 by 25 is: " + divideOperation.operate(50, 25));

    }
}

Lambdalar ve bellek kullanımı #

Java lambdaları kapanış olduğundan, çevreleyen sözlük kapsamındaki değişkenlerin değerlerini “yakalayabilirler”. Tüm lambdalar hiçbir şey yakalamasa da – s -> s.length() gibi basit lambdalar hiçbir şey yakalamazlar ve durumsuz olarak adlandırılırlar – lambdaları yakalamak, yakalanan değişkenleri tutmak için geçici bir nesne gerektirir. Bu kod parçacığında, lambda () -> j bir yakalama lambdasıdır ve değerlendirildiğinde bir nesnenin tahsis edilmesine neden olabilir:

public static void main(String[] args) throws Exception {
    for (int i = 0; i < 1000000000; i++) {
        int j = i;
        doSomethingWithLambda(() -> j);
    }
}

’new’ anahtar sözcüğü snippet’in hiçbir yerinde görünmediğinden hemen belli olmasa da, bu kod, ‘() -> j’ lambda ifadesinin örneklerini temsil etmek için 1.000.000.000 ayrı nesne oluşturmakla yükümlüdür. Bununla birlikte, Java1‘ın gelecekteki sürümlerinin, çalışma zamanında lambda örneklerinin yeniden kullanılması veya başka bir şekilde temsil edilmesi için bunu optimize edebileceğine de dikkat edilmelidir.

1 - Örneğin, Java 9, Java derleme dizisine bunun gibi global optimizasyonlar yapma fırsatı sağlayacak isteğe bağlı bir “bağlantı” aşaması sunar.

Lambda’lar ve Yürütme Çevresinde Desen #

Basit senaryolarda bir FunctionalInterface olarak lambda kullanmanın birkaç iyi örneği vardır. Lambdalar tarafından geliştirilebilecek oldukça yaygın bir kullanım durumu, Yürüt-Etrafında Model olarak adlandırılan şeydir. Bu modelde, kullanım durumuna özel kodu çevreleyen birden çok senaryo için gerekli olan bir dizi standart kurulum/yırtılma kodunuz vardır. Bunun birkaç yaygın örneği dosya io, veritabanı io, try/catch bloklarıdır.

interface DataProcessor {
    void process( Connection connection ) throws SQLException;;
}

public void doProcessing( DataProcessor processor ) throws SQLException{
    try (Connection connection = DBUtil.getDatabaseConnection();) {
        processor.process(connection);
        connection.commit();
    } 
}

Sonra bu yöntemi bir lambda ile çağırmak şöyle görünebilir:

public static void updateMyDAO(MyVO vo) throws DatabaseException {
    doProcessing((Connection conn) -> MyDAO.update(conn, ObjectMapper.map(vo)));
}

Bu, G/Ç işlemleriyle sınırlı değildir. Küçük değişikliklerle benzer kurulum/yıkım görevlerinin uygulanabilir olduğu herhangi bir senaryoya uygulanabilir. Bu Modelin ana yararı, kodun yeniden kullanılması ve DRY’nin (Kendinizi Tekrar Etmeyin) zorlamasıdır.

“dönüş”, dış yöntemden değil, yalnızca lambdadan döner #

‘return’ yöntemi, dış yöntemden değil, yalnızca lambda’dan döner.

Bunun Scala ve Kotlin’den farklı olduğuna dikkat edin!

void threeTimes(IntConsumer r) {
  for (int i = 0; i < 3; i++) {
    r.accept(i);
  }
}

void demo() {
  threeTimes(i -> {
    System.out.println(i);
    return; // Return from lambda to threeTimes only!
  });
}

Bu, “for” döngüleri “return” gibi yerleşik yapılarda farklı davrandığı için, kendi dil yapılarını yazmaya çalışırken beklenmeyen davranışlara yol açabilir:

void demo2() {
  for (int i = 0; i < 3; i++) {
    System.out.println(i);
    return; // Return from 'demo2' entirely
  }
}

Scala ve Kotlin’de “demo” ve “demo2” her ikisi de yalnızca “0” yazdırır. Ancak bu daha tutarlı *değildir. Java yaklaşımı, yeniden düzenleme ve sınıfların kullanımı ile tutarlıdır - en üstteki koddaki “dönüş” ve aşağıdaki kod aynı şekilde davranır:

void demo3() {
  threeTimes(new MyIntConsumer());
}

class MyIntConsumer implements IntConsumer {
  public void accept(int i) {
    System.out.println(i);
    return;
  }
}

Bu nedenle, Java ‘dönüşü’, sınıf yöntemleri ve yeniden düzenleme ile daha tutarlıdır, ancak ‘for’ ve ‘while’ yerleşikleriyle daha az tutarlıdır, bunlar özel kalır.

Bu nedenle, aşağıdaki ikisi Java’da eşdeğerdir:

IntStream.range(1, 4)
    .map(x -> x * x)
    .forEach(System.out::println);
IntStream.range(1, 4)
    .map(x -> { return x * x; })
    .forEach(System.out::println);

Ayrıca, kaynakları denemenin kullanımı Java’da güvenlidir:

class Resource implements AutoCloseable {
  public void close() { System.out.println("close()"); }
}

void executeAround(Consumer<Resource> f) {
  try (Resource r = new Resource()) {
    System.out.print("before ");
    f.accept(r);
    System.out.print("after ");
  }
}

void demo4() {
  executeAround(r -> {
    System.out.print("accept() ");
    return; // Does not return from demo4, but frees the resource.
  });
}

``kapat()dan sonra kabul()den önce yazacaktır. Scala ve Kotlin semantiğinde, kaynaklarla deneme kapatılmaz, ancak yalnızca “before kabul()” yazdırılır.

Bir listeden belirli bir değer(ler) almak için lambda ifadeleri ve yüklemlerini kullanma #

Java 8 ile başlayarak, lambda ifadeleri ve yüklemlerini kullanabilirsiniz.

Örnek: Bir listeden belirli bir değer elde etmek için bir lambda ifadeleri ve bir yüklem kullanın. Bu örnekte, her kişinin 18 yaşında ve daha büyük olup olmadığı bilgisi yazdırılacaktır.

Kişi Sınıfı:

public class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public int getAge() { return age; }
    public String getName() { return name; }
}

Java.util.function.Predicate paketlerindeki yerleşik arabirim Predicate , ‘boolean test(T t)’ yöntemine sahip işlevsel bir arabirimdir.

Örnek Kullanım:

import java.util.ArrayList;
import java.util.List;
import java.util.function.Predicate;

public class LambdaExample {
    public static void main(String[] args) {
        List<Person> personList = new ArrayList<Person>();
        personList.add(new Person("Jeroen", 20));
        personList.add(new Person("Jack", 5));
        personList.add(new Person("Lisa", 19));

        print(personList, p -> p.getAge() >= 18);
    }

    private static void print(List<Person> personList, Predicate<Person> checker) {
        for (Person person : personList) {
            if (checker.test(person)) {
                System.out.print(person + " matches your expression.");
            } else {
                System.out.println(person  + " doesn't match your expression.");
            }
        }
    }
}

print(personList, p -> p.getAge() >= 18); yöntemi, gerekli ifadeyi tanımlayabileceğiniz bir lambda ifadesi alır (çünkü Predicate bir parametre kullanılır). Denetleyicinin test yöntemi, bu ifadenin doğru olup olmadığını kontrol eder: checker.test(person).

Bunu kolayca başka bir şeye değiştirebilirsiniz, örneğin print(personList, p -> p.getName().startsWith("J"));. Bu, kişinin adının “J” ile başlayıp başlamadığını kontrol edecektir.
