Java 8’de tanıtılan Varsayılan Yöntem, geliştiricilerin bu arabirimin mevcut uygulamalarını bozmadan bir arabirime yeni yöntemler eklemesine olanak tanır. Bu arabirimi uygulayan bir sınıf, bu yöntemin bir uygulamasını sağlayamadığında, arabirimin varsayılan olarak kullanılacak bir uygulamayı tanımlamasına izin vermek için esneklik sağlar.

Sözdizimi #

  • genel varsayılan geçersiz yöntemAdı() {/* yöntem gövdesi */}

Varsayılan yöntemler #

  • Mevcut alt sınıfları uygulamaya zorlamadan bir davranışı tanıtmak için bir arayüz içinde kullanılabilir.
  • Alt sınıflar veya bir alt arayüz tarafından geçersiz kılınabilir.
  • Java.lang.Object sınıfındaki yöntemleri geçersiz kılmasına izin verilmez.
  • Birden fazla arabirim uygulayan bir sınıf, arabirimlerin her birinden aynı yöntem imzalarına sahip varsayılan yöntemleri miras alırsa, bu, varsayılan yöntemler değilmiş gibi (çoklu kalıtımı çözmenin bir parçası olarak) geçersiz kılmalı ve kendi arabirimini sağlamalıdır.
  • Mevcut uygulamaları bozmadan bir davranış getirmesi amaçlanmış olsa da, yeni tanıtılan varsayılan yöntemle aynı yöntem imzasına sahip statik bir yöntemle mevcut alt sınıflar yine de bozulacaktır. Ancak bu, bir üst sınıfta bir örnek yöntemi tanıtılması durumunda bile geçerlidir.



Statik yöntemler #

  • Can be used within an interface, primarily intended to be used as a utility method for default methods.
  • Cannot be overridden by subclasses or by a sub-interface (is hidden to them). However as is the case with static methods even now, each class or interface can have its own.
  • Are not allowed to override instance methods in java.lang.Object class (as is presently the case for subclasses as well).



Alt sınıf ve süper sınıf arasındaki etkileşimi özetleyen bir tablo aşağıdadır.

- SÜPER_SINIF-ÖRNEK-YÖNTEM SÜPER_SINIF-STATİK-YÖNTEM

ALT_SINIF-STATİK-YÖNTEM oluşturur-derleme-hatası gizler



Aşağıda arayüz ve uygulama sınıfı arasındaki etkileşimi özetleyen bir tablo bulunmaktadır.

- ARAYÜZ-VARSAYILAN-YÖNTEM ARAYÜZ-STATİK-YÖNTEM

Varsayılan yöntemlerin temel kullanımı #

/**
 * Interface with default method
 */
public interface Printable {
    default void printString() {
        System.out.println( "default implementation" );
    }
}

/**
 * Class which falls back to default implementation of {@link #printString()}
 */
public class WithDefault
    implements Printable
{
}

/**
 * Custom implementation of {@link #printString()}
 */
public class OverrideDefault
    implements Printable {
    @Override
    public void printString() {
        System.out.println( "overridden implementation" );
    }
}

Aşağıdaki ifadeler

    new WithDefault().printString();
    new OverrideDefault().printString();

“varsayılan uygulama” ‘geçersiz kılınan uygulama’

Sınıf uygulamadan geçersiz kılınan varsayılan yöntemlere erişme #

Sınıflarda, super.foo() sadece üst sınıflara bakacaktır. Bir süper arabirimden varsayılan bir uygulama çağırmak istiyorsanız, “süper"i, arayüz adıyla “Fooable.super.foo()” olarak nitelendirmeniz gerekir.

public interface Fooable {
    default int foo() {return 3;}
}

public class A extends Object implements Fooable {
    @Override
    public int foo() {
        //return super.foo() + 1; //error: no method foo() in java.lang.Object
        return Fooable.super.foo() + 1; //okay, returns 4
    }
}

Neden Varsayılan Yöntemler Kullanılır? #

Basit cevap, mevcut uygulamaları bozmadan mevcut bir arayüzü geliştirmenize izin vermesidir.

Örneğin 20 yıl önce yayınladığınız Swim arayüzünüz var.

public interface Swim {
    void backStroke();
}

Harika bir iş çıkardık, arayüzümüz çok popüler, dünya çapında bununla ilgili birçok uygulama var ve kaynak kodları üzerinde kontrolünüz yok.

public class FooSwimmer implements Swim {
    public void backStroke() {
         System.out.println("Do backstroke");
    }
}

20 yıl sonra arayüze yeni işlevler eklemeye karar verdiniz, ancak görünüşe göre arayüzümüz mevcut uygulamaları bozacağı için donmuş durumda.

Neyse ki Java 8, Varsayılan yöntem. adlı yepyeni bir özellik sunuyor.

Artık Swim arayüzüne yeni metod ekleyebiliriz.

public interface Swim {
    void backStroke();
    default void sideStroke() {
        System.out.println("Default sidestroke implementation. Can be overridden");
    }
}

Artık arayüzümüzün mevcut tüm uygulamaları çalışmaya devam edebilir. Ama en önemlisi yeni eklenen yöntemi kendi zamanlarında uygulayabiliyorlar.

Bu değişikliğin en büyük nedenlerinden biri ve en büyük kullanımlarından biri Java Koleksiyonları çerçevesindedir. Oracle, Iterable’ı uygulayan tüm mevcut kodları kırmadan mevcut Yinelenebilir arabirime bir “foreach” yöntemini ekleyemedi. Varsayılan yöntemler eklendiğinde, mevcut Yinelenebilir uygulama, varsayılan uygulamayı devralır.

Varsayılan yöntem içinde diğer arabirim yöntemlerine erişme #

Varsayılan yönteminizden diğer arabirim yöntemlerine de erişebilirsiniz.

public interface Summable {
    int getA();

    int getB();

    default int calculateSum() {
        return getA() + getB();
    }
}

public class Sum implements Summable {
    @Override
    public int getA() {
        return 1;
    }

    @Override
    public int getB() {
        return 2;
    }
}

Aşağıdaki ifade 3: yazdırılacaktır:

System.out.println(new Sum().calculateSum());

Varsayılan yöntemler, arabirim statik yöntemleriyle birlikte kullanılabilir:

public interface Summable {
    static int getA() {
        return 1;
    }

    static int getB() {
        return 2;
    }

    default int calculateSum() {
        return getA() + getB();
    }
}

public class Sum implements Summable {}

Aşağıdaki ifade de 3 yazdıracaktır:

System.out.println(new Sum().calculateSum());

Varsayılan yöntem çoklu kalıtım çakışması #

Bir sonraki örneği düşünün:

public interface A {
    default void foo() { System.out.println("A.foo"); }
}

public interface B {
    default void foo() { System.out.println("B.foo"); }
}

Aynı imzayla “varsayılan” yöntemi “foo” olarak bildiren iki arabirim aşağıda verilmiştir.

Yeni arayüzde bu iki arayüzü “genişletmeye” çalışacaksanız, ikisinden birini seçmeniz gerekir, çünkü Java sizi bu çarpışmayı açıkça çözmeye zorlar.

Önce, “A” ve “B” davranışını geçersiz kılacak olan “abstract” ile aynı imzaya sahip “foo” yöntemini bildirebilirsiniz.

public interface ABExtendsAbstract extends A, B {
    @Override
    void foo();
}

Ve ‘sınıf’ içinde ‘ABExtendsAbstract’ı ‘uyguladığınızda’, ‘foo’ uygulamasını sağlamanız gerekecek:

public class ABExtendsAbstractImpl implements ABExtendsAbstract {
    @Override
    public void foo() { System.out.println("ABImpl.foo"); }
}

Veya ikinci, tamamen yeni bir “varsayılan” uygulama sağlayabilirsiniz. Ayrıca, https://www.wikiod.com/tr/java/varsaylan-yontemler#Uygulama sınıfından geçersiz kılınan varsayılan yöntemlere erişme ile “A” ve “B” “foo” yöntemlerinin kodunu yeniden kullanabilirsiniz. sınıf.

public interface ABExtends extends A, B {
    @Override
    default void foo() { System.out.println("ABExtends.foo"); }
}

Ve ‘sınıf’ içinde ‘ABExtends’i ‘uyguladığınızda’, ‘foo’ uygulamasını sağlamanız gerekmeyecek:

public class ABExtendsImpl implements ABExtends {}

Sınıf, Soyut sınıf ve Arayüz yöntemi önceliği #

Soyut bildirimler de dahil olmak üzere sınıflardaki uygulamalar, tüm arabirim varsayılanlarına göre önceliklidir.

  • Soyut sınıf yöntemi, Arayüz Varsayılan Yöntemi‘den önceliklidir.

    public interface Swim { default void backStroke() { System.out.println(“Swim.backStroke”); } }

    public abstract class AbstractSwimmer implements Swim { public void backStroke() { System.out.println(“AbstractSwimmer.backStroke”); } }

    public class FooSwimmer extends AbstractSwimmer { }

Aşağıdaki ifade

new FooSwimmer().backStroke();

Üretecek

AbstractSwimmer.backStroke

  • Sınıf yöntemi, Arayüz Varsayılan Yöntemi‘den önceliklidir

    public interface Swim { default void backStroke() { System.out.println(“Swim.backStroke”); } }

    public abstract class AbstractSwimmer implements Swim { }

    public class FooSwimmer extends AbstractSwimmer { public void backStroke() { System.out.println(“FooSwimmer.backStroke”); } }

Aşağıdaki ifade

new FooSwimmer().backStroke();

Üretecek

FooSwimmer.backStroke
