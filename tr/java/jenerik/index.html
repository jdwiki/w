Generics, Java’nın tür sistemini, bir türün veya yöntemin derleme zamanı tür güvenliği sağlarken çeşitli türlerdeki nesneler üzerinde çalışmasına izin verecek şekilde genişleten bir genel programlama tesisidir. Özellikle, Java koleksiyonları çerçevesi, bir koleksiyon örneğinde depolanan nesnelerin türünü belirtmek için jenerikleri destekler.

Sözdizimi #

  • sınıf ArrayList<E> {} // E tipi parametreli genel bir sınıf
  • sınıf HashMap<K, V> {} // K ve V olmak üzere iki tip parametreye sahip genel bir sınıf
  • <E> void print(E element) {} // E tipi parametreli genel bir metot
  • ArrayList<String> isimler; // genel bir sınıfın bildirimi
  • ArrayList<?> nesneler; // tür parametresi bilinmeyen bir genel sınıfın bildirimi
  • yeni ArrayList<String>() // genel bir sınıfın somutlaştırılması
  • yeni ArrayList<>() // “diamond” tür çıkarımıyla örnekleme (Java 7 veya üstü)

Jenerikler, Java’da Type silme yoluyla uygulanır; bu, çalışma zamanı sırasında genel bir sınıfın örneğinde belirtilen Type bilgisinin mevcut olmadığı anlamına gelir. Örneğin, List<String>names = new ArrayList<>(); ifadesi, çalışma zamanında String öğe türünün kurtarılamadığı bir liste nesnesi üretir. Ancak, liste List<String> türünde bir alanda depolanıyorsa veya bu aynı türdeki bir yöntem/yapıcı parametresine iletilirse veya bu dönüş türündeki bir yöntemden döndürülürse, tam tür bilgisi can Java Reflection API aracılığıyla çalışma zamanında kurtarılabilir.

Bu aynı zamanda genel bir türe yayın yaparken (örneğin: (List<String>) list), yayının kontrol edilmemiş bir yayın olduğu anlamına gelir. ``parametresi silindiği için, JVM birListeden bir List```e yapılan bir yayının doğru olup olmadığını kontrol edemez; JVM, çalışma zamanında yalnızca “List” için “List” için bir yayın görür.

Genel Sınıf Oluşturma #

Generics, diğer sınıfları ve arabirimleri tür parametreleri olarak almak için sınıfları, arabirimleri ve yöntemleri etkinleştirir.

Bu örnek, açılı ayraçlarla (<>) sınırlandırılmış tek bir tür parametresi T almak için genel Param sınıfını kullanır:

public class Param<T> {
    private T value;

    public T getValue() {
        return value;
    }

    public void setValue(T value) {
        this.value = value;
    }
}

Bu sınıfı somutlaştırmak için “T” yerine bir tür argümanı sağlayın. Örneğin, “Tamsayı”:

Param<Integer> integerParam = new Param<Integer>();

Tür bağımsız değişkeni, diziler ve diğer genel türler dahil olmak üzere herhangi bir başvuru türü olabilir:

Param<String[]> stringArrayParam;
Param<int[][]> int2dArrayParam;
Param<Param<Object>> objectNestedParam;

Java SE 7 ve sonraki sürümlerinde, tür bağımsız değişkeni, diamond olarak adlandırılan boş bir tür bağımsız değişkenleri (<>) ile değiştirilebilir:

Param<Integer> integerParam = new Param<>();

Diğer tanımlayıcılardan farklı olarak, tür parametrelerinin adlandırma kısıtlaması yoktur. Ancak isimleri genellikle büyük harflerle amaçlarının ilk harfidir. (Bu, resmi JavaDocs’ta bile geçerlidir.) Örnekler arasında “tür” için T, “element” için “E” sayılabilir. ://docs.oracle.com/javase/8/docs/api/java/util/List.html) ve “anahtar”/“değer” için [K/V](https://docs. oracle.com/javase/8/docs/api/java/util/Map.html).

Genel bir sınıfı genişletme #

public abstract class AbstractParam<T> {
    private T value;

    public T getValue() {
        return value;
    }

    public void setValue(T value) {
        this.value = value;
    }
}

‘AbstractParam’, ‘T’ tür parametresiyle bildirilen bir [soyut sınıf][3]. Bu sınıfı genişletirken, bu tür parametresi <> içine yazılmış bir tür argümanı ile değiştirilebilir veya tür parametresi değişmeden kalabilir. Aşağıdaki birinci ve ikinci örneklerde, “Dize” ve “Tamsayı”, type parametresinin yerini alır. Üçüncü örnekte, type parametresi değişmeden kalır. Dördüncü örnek, jenerikleri hiç kullanmaz, bu nedenle sınıfın bir “Object” parametresine sahip olmasına benzer. Derleyici, “AbstractParam"ın ham bir tür olduğu konusunda uyaracak, ancak “ObjectParam” sınıfını derleyecektir. Beşinci örnek, üst sınıfa geçirilen tür parametresi olarak ikinci parametreyi seçerek 2 tür parametresine sahiptir (aşağıdaki “çoklu tür parametreleri"ne bakın).

public class Email extends AbstractParam<String> {
    // ...
}

public class Age extends AbstractParam<Integer> {
    // ...
}

public class Height<T> extends AbstractParam<T> {
    // ...
}

public class ObjectParam extends AbstractParam {
    // ...
}

public class MultiParam<T, E> extends AbstractParam<E> {
    // ...
}

Aşağıdaki kullanımdır:

Email email = new Email();
email.setValue("test@example.com");
String retrievedEmail = email.getValue();

Age age = new Age();
age.setValue(25);
Integer retrievedAge = age.getValue();
int autounboxedAge = age.getValue();

Height<Integer> heightInInt = new Height<>();
heightInInt.setValue(125);

Height<Float> heightInFloat = new Height<>();
heightInFloat.setValue(120.3f);

MultiParam<String, Double> multiParam = new MultiParam<>();
multiParam.setValue(3.3);

“Email” sınıfında, “T getValue()” yönteminin, “String getValue()” imzasına sahipmiş gibi davrandığına ve “void setValue(T)” yönteminin, “void” olarak bildirilmiş gibi davrandığına dikkat edin. setValue(Dize)`.

Boş küme parantezleri ({}) ile anonim iç sınıfla örnekleme yapmak da mümkündür:

AbstractParam<Double> height = new AbstractParam<Double>(){};
height.setValue(198.6);

[anonim iç sınıflarla elmasın kullanılmasına izin verilmediğini] unutmayın.][4]

Çoklu tip parametreleri #

Java, genel bir sınıf veya arabirimde birden fazla tür parametresi kullanma yeteneği sağlar. Açılı ayraçlar arasına virgülle ayrılmış bir tür listesi yerleştirerek bir sınıfta veya arabirimde birden çok tür parametresi kullanılabilir. Örnek:

public class MultiGenericParam<T, S> {
    private T firstParam;
    private S secondParam;
   
    public MultiGenericParam(T firstParam, S secondParam) {
        this.firstParam = firstParam;
        this.secondParam = secondParam;
    }
    
    public T getFirstParam() {
        return firstParam;
    }
    
    public void setFirstParam(T firstParam) {
        this.firstParam = firstParam;
    }
    
    public S getSecondParam() {
        return secondParam;
    }
    
    public void setSecondParam(S secondParam) {
        this.secondParam = secondParam;
    }
}

Kullanım aşağıdaki gibi yapılabilir:

MultiGenericParam<String, String> aParam = new MultiGenericParam<String, String>("value1", "value2");
MultiGenericParam<Integer, Double> dayOfWeekDegrees = new MultiGenericParam<Integer, Double>(1, 2.6);

T, `` arasında karar vermek? süper Tve? T’yi uzatır #

Java jeneriklerinin sınırlı joker karakterlerinin sözdizimi, bilinmeyen türü “?” ile temsil eder:

  • ? extensions T, üst sınırlı bir joker karakteri temsil eder. Bilinmeyen tür, T’nin bir alt türü olması gereken bir türü veya T türünün kendisini temsil eder.

  • ? super T, daha düşük sınırlı bir joker karakteri temsil eder. Bilinmeyen tür, T’nin bir üst türü olması gereken bir türü temsil eder veya T türünün kendisidir.

Genel bir kural olarak, kullanmalısınız

  • ? yalnızca "okuma" erişimine ("giriş") ihtiyacınız varsa T öğesini genişletir
  • ? super T “yazma” erişimine (“çıktı”) ihtiyacınız varsa
  • İkisine de ihtiyacınız varsa T (“değiştir”)

‘Extends’ veya ‘super’ kullanmak genellikle daha iyidir çünkü aşağıda göreceğiniz gibi kodunuzu daha esnek hale getirir (alt türlerin ve üst türlerin kullanımına izin verilmesi gibi).

class Shoe {}
class IPhone {}
interface Fruit {}
class Apple implements Fruit {}
class Banana implements Fruit {}
class GrannySmith extends Apple {}

   public class FruitHelper {

        public void eatAll(Collection<? extends Fruit> fruits) {}

        public void addApple(Collection<? super Apple> apples) {}
}

Derleyici artık bazı kötü kullanımları tespit edebilecektir:

 public class GenericsTest {
      public static void main(String[] args){
  FruitHelper fruitHelper = new FruitHelper() ;
    List<Fruit> fruits = new ArrayList<Fruit>();
    fruits.add(new Apple()); // Allowed, as Apple is a Fruit
    fruits.add(new Banana()); // Allowed, as Banana is a Fruit
    fruitHelper.addApple(fruits); // Allowed, as "Fruit super Apple"
    fruitHelper.eatAll(fruits); // Allowed

    Collection<Banana> bananas = new ArrayList<>();
    bananas.add(new Banana()); // Allowed
    //fruitHelper.addApple(bananas); // Compile error: may only contain Bananas!
    fruitHelper.eatAll(bananas); // Allowed, as all Bananas are Fruits

    Collection<Apple> apples = new ArrayList<>();
    fruitHelper.addApple(apples); // Allowed
    apples.add(new GrannySmith()); // Allowed, as this is an Apple
    fruitHelper.eatAll(apples); // Allowed, as all Apples are Fruits.
    
    Collection<GrannySmith> grannySmithApples = new ArrayList<>();
    fruitHelper.addApple(grannySmithApples); //Compile error: Not allowed.
                                   // GrannySmith is not a supertype of Apple
    apples.add(new GrannySmith()); //Still allowed, GrannySmith is an Apple
    fruitHelper.eatAll(grannySmithApples);//Still allowed, GrannySmith is a Fruit

    Collection<Object> objects = new ArrayList<>();
    fruitHelper.addApple(objects); // Allowed, as Object super Apple
    objects.add(new Shoe()); // Not a fruit
    objects.add(new IPhone()); // Not a fruit
    //fruitHelper.eatAll(objects); // Compile error: may contain a Shoe, too!
}

Doğru ‘T’yi seçmek, ‘? süper T’ veya ‘? extension T`, alt türlerle kullanıma izin vermek için gerekli. Derleyici daha sonra tip güvenliğini sağlayabilir; düzgün kullanırsanız, yayınlamanız gerekmez (ki bu güvenli değildir ve programlama hatalarına neden olabilir).

Bir sınırlama, örneklemede type parametresini yine de sağlamanız gereken Anonim Sınıflar içindir:

// This will compile:

Comparator<String> caseInsensitiveComparator = new Comparator<String>() {

// But this will not:

Comparator<String> caseInsensitiveComparator = new Comparator<>() {
    @Override
    public int compare(String s1, String s2) {
        return s1.compareToIgnoreCase(s2);
    }

Genel Yöntem Bildirme #

Metotlar ayrıca generic tipi parametrelere sahip olabilir.

public class Example {

    // The type parameter T is scoped to the method
    // and is independent of type parameters of other methods.
    public <T> List<T> makeList(T t1, T t2) {
        List<T> result = new ArrayList<T>();
        result.add(t1);
        result.add(t2);
        return result;
    }

    public void usage() {
        List<String> listString = makeList("Jeff", "Atwood");
        List<Integer> listInteger = makeList(1, 2);
    }
}

Genel bir yönteme gerçek bir tür argümanı iletmemiz gerekmediğine dikkat edin. Derleyici, hedef türe (örneğin sonucu atadığımız değişkene) veya gerçek argümanların türlerine göre tür argümanını bizim için çıkarır. Genellikle, çağrı tipini doğru yapacak en spesifik tip argümanını çıkarır.

Bazen, nadiren de olsa, bu tür çıkarımını açık tür bağımsız değişkenleriyle geçersiz kılmak gerekebilir:

void usage() {
    consumeObjects(this.<Object>makeList("Jeff", "Atwood").stream());
}

void consumeObjects(Stream<Object> stream) { ... }

Bu örnekte gereklidir, çünkü derleyici, “akım()” çağrıldıktan sonra “T” için “Nesne"nin istendiğini göremez ve aksi takdirde “makeList” argümanlarına dayalı olarak “Dize” sonucunu çıkarır. Java dilinin, tür bağımsız değişkenleri açıkça sağlandığında, yöntemin çağrıldığı sınıfın veya nesnenin (yukarıdaki örnekte ’this’) atlanmasını desteklemediğini unutmayın.

Birden çok üst sınır gerektirmesi (“A ve B’yi genişletir”) #

Birden çok üst sınırı genişletmek için genel bir tür gerektirebilirsiniz.

Örnek: Bir sayı listesini sıralamak istiyoruz ama ‘Sayı’, ‘Karşılaştırılabilir’i uygulamıyor.

public <T extends Number & Comparable<T>> void sortNumbers( List<T> n ) {
  Collections.sort( n );
}

Bu örnekte, “T”, “Sayı"yı genişletmelidir ve “Tamsayı” veya “BigDecimal” gibi tüm “normal” yerleşik sayı uygulamalarına uyması gereken ancak “gibi daha egzotik olanlara uymayan” Karşılaştırılabilir‘yi uygular. Çizgili64`.

Çoklu mirasa izin verilmediğinden, en fazla bir sınıfı sınır olarak kullanabilirsiniz ve ilk listelenen olmalıdır. Örneğin, Karşılaştırılabilir bir arabirim olduğu ve bir sınıf olmadığı için “<T Extends Comparable & Number>” öğesine izin verilmez.

Derleme zamanında daha güçlü tip kontrolleri #

Bir Java derleyicisi, genel koda güçlü tür denetimi uygular ve kod tür güvenliğini ihlal ederse hatalar verir. Derleme zamanı hatalarını düzeltmek, bulunması zor olabilecek çalışma zamanı hatalarını düzeltmekten daha kolaydır.


Alımların ortadan kaldırılması #

Jenerik içermeyen aşağıdaki kod parçacığı, yayınlamayı gerektirir:

List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);

jenerik kullanmak için yeniden yazıldığında, kod döküm gerektirmez:

List<String> list = new ArrayList<>();

Programcıların genel algoritmaları uygulamasını sağlama #

Programcılar, jenerik kullanarak, farklı türlerdeki koleksiyonlar üzerinde çalışan, özelleştirilebilen, güvenli ve okunması kolay genel algoritmalar uygulayabilir.

Genel bir tür örneği oluşturma #

Tür silme nedeniyle aşağıdakiler çalışmayacaktır:

public <T> void genericMethod() {
    T t = new T(); // Can not instantiate the type T.
}

T tipi silinir. Çalışma zamanında, JVM ‘T’nin orijinal olarak ne olduğunu bilmediğinden, hangi kurucuyu arayacağını bilemez.

Geçici çözümler #

  1. ‘genericMethod’ çağrılırken ‘T’ sınıfını geçmek:

    public <T> void genericMethod(Class<T> cls) {
    try {
        T t = cls.newInstance();
    } catch (InstantiationException | IllegalAccessException e) {
         System.err.println("Could not instantiate: " + cls.getName());
    }
}

    genericMethod(String.class);

    Which throws exceptions, since there is no way to know if the passed class has an accessible default constructor.

  1. “T"nin yapıcısına bir [referans](https://www.wikiod.com/tr/java/lambda-ifadeleri#Yöntem Referansları) iletmek:

    public <T> void genericMethod(Supplier<T> cons) {
    T t = cons.get();
}

    genericMethod(String::new);
son sürüm

Çalışma zamanında genel parametreyi karşılayan sınıfı elde edin #

Statik bir yöntemde kullanılanlar gibi birçok ilişkisiz genel parametre çalışma zamanında kurtarılamaz (bkz. Silme üzerinde Diğer Konular). Ancak, çalışma zamanında bir sınıf üzerinde genel bir parametreyi karşılayan türe erişmek için kullanılan ortak bir strateji vardır. Bu, her aramada tür bilgilerini işlemek zorunda kalmadan * türe erişime bağlı genel koda izin verir.

Arka fon

Bir sınıf üzerindeki genel parametreleştirme, anonim bir iç sınıf oluşturularak incelenebilir. Bu sınıf, tür bilgilerini yakalayacaktır. Genel olarak bu mekanizma, Neal Gafter’in blog gönderisinde ayrıntılı olarak açıklanan süper tip belirteçler olarak adlandırılır.

Uygulamalar

Java’daki üç yaygın uygulama şunlardır:

  • Guava’nın TypeToken’ı
  • Spring’s ParameterizedTypeReference
  • Jackson’ın TypeReference

Örnek kullanım

public class DataService<MODEL_TYPE> {
     private final DataDao dataDao = new DataDao();
     private final Class<MODEL_TYPE> type = (Class<MODEL_TYPE>) new TypeToken<MODEL_TYPE>
                                                                (getClass()){}.getRawType();
     public List<MODEL_TYPE> getAll() {
         return dataDao.getAllOfType(type);
    }
}

// the subclass definitively binds the parameterization to User
// for all instances of this class, so that information can be 
// recovered at runtime
public class UserService extends DataService<User> {}

public class Main {
    public static void main(String[] args) {
          UserService service = new UserService();
          List<User> users = service.getAll();
    }
}

Sınırlı Genel Sınıf Oluşturma #

Bir genel sınıfta kullanılan geçerli türleri, bu türü sınıf tanımında sınırlayarak kısıtlayabilirsiniz. Aşağıdaki basit tür hiyerarşisi göz önüne alındığında:

public abstract class Animal {
    public abstract String getSound();
}

public class Cat extends Animal {
    public String getSound() {
        return "Meow";
    }
}

public class Dog extends Animal {
    public String getSound() {
        return "Woof";
    }
}

sınırlı jenerikler olmadan, hem genel hem de her öğenin bir hayvan olduğunu bilen bir kap sınıfı yapamayız:

public class AnimalContainer<T> {

    private Collection<T> col;

    public AnimalContainer() {
        col = new ArrayList<T>();
    }

    public void add(T t) {
        col.add(t);
    }

    public void printAllSounds() {
        for (T t : col) {
            // Illegal, type T doesn't have makeSound()
            // it is used as an java.lang.Object here
            System.out.println(t.makeSound()); 
        }
    }
}

Sınıf tanımında jenerik sınır ile bu artık mümkün.

public class BoundedAnimalContainer<T extends Animal> { // Note bound here.

    private Collection<T> col;

    public BoundedAnimalContainer() {
        col = new ArrayList<T>();
    }

    public void add(T t) {
        col.add(t);
    }

    public void printAllSounds() {
        for (T t : col) {
            // Now works because T is extending Animal
            System.out.println(t.makeSound()); 
        }
    }
}

Bu, genel türün geçerli örneklemelerini de kısıtlar:

// Legal
AnimalContainer<Cat> a = new AnimalContainer<Cat>();

// Legal
AnimalContainer<String> a = new AnimalContainer<String>();

// Legal because Cat extends Animal
BoundedAnimalContainer<Cat> b = new BoundedAnimalContainer<Cat>();

// Illegal because String doesn't extends Animal
BoundedAnimalContainer<String> b = new BoundedAnimalContainer<String>();

Kendi beyanı içinde beyan edilen jenerik tipe atıfta bulunur #

Bildirilen genel türün kendisinde bir yöntem bildiriminde (muhtemelen daha ileri) miras alınan bir genel türün bir örneğini nasıl kullanırsınız? Bu, jeneriklere biraz daha derine indiğinizde karşılaşacağınız sorunlardan biridir, ancak yine de oldukça yaygın bir sorundur.

“T” tipi değerleri içeren genel bir veri serisini tanımlayan bir “DataSeries” tipine (arayüz burada) sahip olduğumuzu varsayalım. Örneğin; double değerleri, bu nedenle `DoubleSeries, DataSeries‘ı genişletir. Şimdi, orijinal “DataSeries” türünün, aynı uzunlukta başka bir dizi ekleyen ve yeni bir tane döndüren bir “add(values)” yöntemi olduğunu varsayalım. Türetilmiş sınıfımızda “değerler” türünü ve “DataSeries” yerine “DoubleSeries” dönüş türünü nasıl zorunlu kılarız?

Sorun, bildirilmekte olan türe atıfta bulunan genel bir tür parametresi eklenerek ve bildirilmekte olan türü genişleterek çözülebilir (burada bir arabirime uygulanır, ancak aynısı sınıflar için geçerlidir):

public interface DataSeries<T, DS extends DataSeries<T, DS>> {
    DS add(DS values);
    List<T> data();
}

Burada “T”, serinin sahip olduğu veri türünü temsil eder, ör. ‘Double’ ve ‘DS’ serisinin kendisi. Miras alınan bir tür (veya türler), yukarıda belirtilen parametrenin karşılık gelen bir türetilmiş türle değiştirilmesiyle artık kolayca uygulanabilir, böylece formun somut bir “Çift” tabanlı tanımı elde edilir:

public interface DoubleSeries extends DataSeries<Double, DoubleSeries> {
    static DoubleSeries instance(Collection<Double> data) {
        return new DoubleSeriesImpl(data);
    }
}

Şu anda, bir IDE bile yukarıdaki arayüzü doğru türlerle birlikte uygulayacaktır; bu, biraz içerik doldurma işleminden sonra şöyle görünebilir:

class DoubleSeriesImpl implements DoubleSeries {
    private final List<Double> data;

    DoubleSeriesImpl(Collection<Double> data) {
        this.data = new ArrayList<>(data);
    }

    @Override
    public DoubleSeries add(DoubleSeries values) {
        List<Double> incoming = values != null ? values.data() : null;
        if (incoming == null || incoming.size() != data.size()) {
            throw new IllegalArgumentException("bad series");
        }
        List<Double> newdata = new ArrayList<>(data.size());
        for (int i = 0; i < data.size(); i++) {
            newdata.add(this.data.get(i) + incoming.get(i)); // beware autoboxing
        }
        return DoubleSeries.instance(newdata);
    }

    @Override
    public List<Double> data() {
        return Collections.unmodifiableList(data);
    }
}

Gördüğünüz gibi add metodu DoubleSeries add(DoubleSeries değerleri) olarak bildirilmiş ve derleyici mutludur.

Gerekirse desen daha da iç içe olabilir.

Genel parametreyi 1’den fazla türe bağlama #

Genel parametreler ayrıca T, Type1 & Type2 & ... sözdizimini genişleterek birden fazla tipe bağlanabilir.

Diyelim ki Generic tipi hem Flushable hem de Closeable uygulayacak bir sınıf oluşturmak istiyorsunuz, yazabilirsiniz.

class ExampleClass<T extends Flushable & Closeable> {
}

Şimdi, ‘ÖrnekSınıf’ yalnızca genel parametreler olarak kabul eder, hem ‘Yıkanabilir’ ** hem de ** ‘Kapatılabilir’i uygulayan türleri.

ExampleClass<BufferedWriter> arg1; // Works because BufferedWriter implements both Flushable and Closeable

ExampleClass<Console> arg4; // Does NOT work because Console only implements Flushable
ExampleClass<ZipFile> arg5; // Does NOT work because ZipFile only implements Closeable

ExampleClass<Flushable> arg2; // Does NOT work because Closeable bound is not satisfied.
ExampleClass<Closeable> arg3; // Does NOT work because Flushable bound is not satisfied.

Sınıf yöntemleri, genel tür bağımsız değişkenlerini “Kapatılabilir” veya “Yıkılabilir” olarak çıkarmayı seçebilir.

class ExampleClass<T extends Flushable & Closeable> {
    /* Assign it to a valid type as you want. */
    public void test (T param) {
        Flushable arg1 = param; // Works
        Closeable arg2 = param; // Works too.
    }

    /* You can even invoke the methods of any valid type directly. */
    public void test2 (T param) {
        param.flush(); // Method of Flushable called on T and works fine.
        param.close(); // Method of Closeable called on T and works fine too.
    }
}

Not: #

Genel parametreyi, OR (|) yan tümcesini kullanarak herhangi bir türe bağlayamazsınız. Yalnızca AND (&) yan tümcesi desteklenir. Genel tür, yalnızca bir sınıfı ve birçok arabirimi genişletebilir. sınıf olmalıdır listenin başına yerleştirilir.

Otomatik yayınlamak için Jenerikleri kullanma #

Jenerik ilaçlarla, arayanın beklediği her şeyi geri vermek mümkündür:

private Map<String, Object> data;
public <T> T get(String key) {
    return (T) data.get(key);
}

Yöntem bir uyarı ile derlenecektir. Kod aslında göründüğünden daha güvenli çünkü Java çalışma zamanı onu kullandığınızda bir döküm yapacak:

Bar bar = foo.get("bar");

Genel türleri kullandığınızda daha az güvenlidir:

List<Bar> bars = foo.get("bars");

Burada, döndürülen tür herhangi bir ‘Liste’ olduğunda çalışır (yani, ‘List’ döndürmek bir ‘ClassCastException’ı tetiklemez; sonunda onu listeden öğeleri çıkarırken alırsınız).

Bu soruna geçici bir çözüm bulmak için, yazılan anahtarları kullanan bir API oluşturabilirsiniz:

public final static Key<List<Bar>> BARS = new Key<>("BARS");

bu “put()” yöntemiyle birlikte:

public <T> T put(Key<T> key, T value);

Bu yaklaşımla, haritaya yanlış türü koyamazsınız, bu nedenle sonuç her zaman doğru olacaktır (yanlışlıkla aynı ada ancak farklı türlere sahip iki anahtar oluşturmadığınız sürece).

Instanceof’un Generics ile kullanımı #

Instanceof içindeki türü tanımlamak için jeneriklerin kullanılması

“” biçimsel parametresiyle bildirilen aşağıdaki genel “Örnek” sınıfını göz önünde bulundurun:

class Example<T> {
    public boolean isTypeAString(String s) {
        return s instanceof T; // Compilation error, cannot use T as class type here
    }
}

Bu her zaman bir Derleme hatası verecektir çünkü derleyici Java kaynağını Java bayt koduna derler derlemez, tüm genel kodu genel olmayan koda dönüştüren ve ayırt etmeyi imkansız hale getiren tür silme olarak bilinen bir işlem uygular. çalışma zamanında T türleri arasında. ‘instanceof’ ile kullanılan tür yeniden dönüştürülebilir olmalıdır; bu, türle ilgili tüm bilgilerin çalışma zamanında mevcut olması gerektiği anlamına gelir ve bu genellikle genel türler için geçerli değildir.

Aşağıdaki sınıf, ‘Example’, ‘Example’ ve ‘Example’ adlı iki farklı sınıfın, type erasure: tarafından jenerikler çıkarıldıktan sonra nasıl göründüğünü temsil eder:

class Example { // formal parameter is gone
    public boolean isTypeAString(String s) {
        return s instanceof Object; // Both <String> and <Number> are now Object
    }
}

Türler ortadan kalktığından, JVM’nin hangi türün “T” olduğunu bilmesi mümkün değildir.

Önceki kuralın istisnası

“instanceof” içinde bir tür belirtmek için her zaman sınırsız joker karakter (?) kullanabilirsiniz:

    public boolean isAList(Object obj) {
        return obj instanceof List<?>;
    }

Bu, bir “obj” örneğinin bir “Liste” olup olmadığını değerlendirmek için yararlı olabilir:

System.out.println(isAList("foo")); // prints false
System.out.println(isAList(new ArrayList<String>()); // prints true
System.out.println(isAList(new ArrayList<Float>()); // prints true

Aslında, sınırsız joker karakter, yeniden yapılandırılabilir bir tür olarak kabul edilir.

** instanceof ile genel bir örnek kullanma**

Madalyonun diğer yüzü, aşağıdaki örnekte gösterildiği gibi, “instanceof” ile “T"nin bir “t” örneğinin kullanılmasının yasal olmasıdır:

class Example<T> {
    public boolean isTypeAString(T t) {
        return t instanceof String; // No compilation error this time
    }
}

çünkü tip silme işleminden sonra sınıf aşağıdaki gibi görünecektir:

class Example { // formal parameter is gone
    public boolean isTypeAString(Object t) {
        return t instanceof String; // No compilation error this time
    }
}

Tür silme işlemi yine de gerçekleşse bile, JVM, aşağıdaki kod parçasının gösterdiği gibi aynı başvuru türünü (“Nesne”) kullansalar bile bellekteki farklı türleri ayırt edebilir:

Object obj1 = new String("foo"); // reference type Object, object type String
Object obj2 = new Integer(11); // reference type Object, object type Integer
System.out.println(obj1 instanceof String); // true
System.out.println(obj2 instanceof String); // false, it's an Integer, not a String

Genel Arabirimi uygulamanın (veya Genel Sınıfı genişletmenin) farklı yolları #

Aşağıdaki genel arabirimin bildirildiğini varsayalım:

public interface MyGenericInterface<T> {
    public void foo(T t);
}

Aşağıda, uygulamanın olası yolları listelenmiştir.

Belirli bir türe sahip genel olmayan sınıf uygulaması

“MyGenericClass"ın “” biçimsel tür parametresini değiştirmek için belirli bir tür seçin ve aşağıdaki örnekte olduğu gibi uygulayın:

public class NonGenericClass implements MyGenericInterface<String> {
    public void foo(String t) { } // type T has been replaced by String
}

Bu sınıf yalnızca “String” ile ilgilenir ve bu, aşağıdaki snippet’in gösterdiği gibi, “MyGenericInterface"in farklı parametrelerle (ör. “Integer”, “Object” vb.) kullanılmasının derlenmeyeceği anlamına gelir:

NonGenericClass myClass = new NonGenericClass();
myClass.foo("foo_string"); // OK, legal
myClass.foo(11); // NOT OK, does not compile
myClass.foo(new Object()); // NOT OK, does not compile

Genel sınıf uygulaması

“MyGenericInterface"i uygulayan resmi tür parametresi “” ile başka bir genel arabirimi aşağıdaki gibi tanımlayın:

public class MyGenericSubclass<T> implements MyGenericInterface<T> {
    public void foo(T t) { } // type T is still the same
    // other methods...
}

Aşağıdaki gibi farklı bir resmi tip parametresinin kullanılmış olabileceğini unutmayın:

public class MyGenericSubclass<U> implements MyGenericInterface<U> { // equivalent to the previous declaration
    public void foo(U t) { }
    // other methods...
}

Ham tip sınıf uygulaması

MyGenericInteface’i ham tür olarak uygulayan genel olmayan bir sınıfı (hiç jenerik kullanmayan) aşağıdaki gibi bildirin:

public class MyGenericSubclass implements MyGenericInterface {
    public void foo(Object t) { } // type T has been replaced by Object
    // other possible methods
}

Bu yol önerilmez, çünkü raw type (alt sınıfın) ile generics (arayüzün) ile karıştırıldığı için çalışma zamanında %100 güvenli değildir ve aynı zamanda kafa karıştırıcıdır. Modern Java derleyicileri bu tür bir uygulama ile bir uyarı verir, ancak kod - daha eski JVM (1.4 veya öncesi) ile uyumluluk nedenleriyle - derlenir.

Genel bir arabirim yerine bir üst tür olarak genel bir sınıf kullanıldığında, yukarıda listelenen tüm yollara da izin verilir.
