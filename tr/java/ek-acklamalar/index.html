Java’da bir annotation, Java kaynak koduna eklenebilen bir sözdizimsel meta veri biçimidir. Programın kendisinin parçası olmayan bir program hakkında veri sağlar. Ek açıklamaların, açıkladıkları kodun çalışması üzerinde doğrudan bir etkisi yoktur. Sınıflar, yöntemler, değişkenler, parametreler ve paketlere açıklama eklenmesine izin verilir.

Sözdizimi #

  • @AnnotationName // ‘İşaret notu’ (parametre yok)
  • @AnnotationName(someValue) // parametreyi ‘değer’ adıyla ayarlar
  • @AnnotationName(param1 = value1) // adlandırılmış parametre
  • @AnnotationName(param1 = değer1, param2 = değer2) // birden çok adlandırılmış parametre
  • @AnnotationName(param1 = {1, 2, 3}) // adlı dizi parametresi
  • @AnnotationName({value1}) // ‘değer’ adında parametre olarak tek elemanlı dizi

Parametre türleri #

Bu türlerin dizilerinin yanı sıra parametreler için yalnızca aşağıdaki türlerin sabit ifadelerine izin verilir:

  • “Dize”
  • “Sınıf”
  • ilkel türler
  • Numaralandırma türleri
  • Ek Açıklama Türleri

Ek açıklama türlerini tanımlama #

Açıklama türleri @interface ile tanımlanır. Parametreler, normal bir arabirimin yöntemlerine benzer şekilde tanımlanır.

@interface MyAnnotation {
    String param1();
    boolean param2();
    int[] param3();  // array parameter 
}

Varsayılan değerler #

@interface MyAnnotation {
    String param1() default "someValue";
    boolean param2() default true;
    int[] param3() default {};
}

Meta-Ek Açıklamalar #

Meta açıklamalar, açıklama türlerine uygulanabilen açıklamalardır. Özel önceden tanımlanmış meta açıklama, açıklama türlerinin nasıl kullanılabileceğini tanımlar.

@Hedef #

“@Target” meta açıklaması, ek açıklamanın uygulanabileceği türleri kısıtlar.

@Target(ElementType.METHOD)
@interface MyAnnotation {
    // this annotation can only be applied to methods
}

Dizi gösterimi kullanılarak birden çok değer eklenebilir, ör. @Target({ElementType.FIELD, ElementType.TYPE})

@Tutulma #

@Retention meta açıklaması, uygulama derleme işlemi veya yürütme sırasında açıklama görünürlüğünü tanımlar. Varsayılan olarak, açıklamalar “.class” dosyalarına dahil edilir, ancak çalışma zamanında görünmez. Bir notu çalışma zamanında erişilebilir kılmak için, o notta RetentionPolicy.RUNTIME ayarlanmalıdır.

@Retention(RetentionPolicy.RUNTIME)
@interface MyAnnotation {
    // this annotation can be accessed with reflections at runtime
}

Mevcut değerler #

Tutma Politikası Etkisi
SINIF Açıklama .class dosyasında mevcuttur, ancak çalışma zamanında mevcut değildir
ÇALIŞMA SÜRESİ Açıklama, çalışma zamanında mevcuttur ve yansıma yoluyla erişilebilir
KAYNAK Açıklama derleme zamanında mevcuttur, ancak ‘.class’ dosyalarına eklenmez. Açıklama örneğin kullanılabilir. açıklama işlemcisi tarafından

@Belgelenmiş #

@Documented meta açıklaması, kullanımı javadoc gibi API dokümantasyon oluşturucuları tarafından belgelenmesi gereken açıklamaları işaretlemek için kullanılır. Değerleri yoktur. @Documented ile, açıklamayı kullanan tüm sınıflar, onu oluşturulan dokümantasyon sayfasında listeleyecektir. @Documented olmadan, belgelerde ek açıklamayı hangi sınıfların kullandığını görmek mümkün değildir.

@Miras #

“@Inherited” meta açıklaması, sınıflara uygulanan ek açıklamalarla ilgilidir. Değerleri yoktur. Bir ek açıklamayı “@Inherited” olarak işaretlemek, açıklama sorgulamanın çalışma şeklini değiştirir.

  • Devralınmayan bir ek açıklama için sorgu yalnızca incelenmekte olan sınıfı inceler.
  • Devralınan bir ek açıklama için sorgu, ek açıklamanın bir örneği bulunana kadar süper sınıf zincirini (yinelemeli olarak) da kontrol eder.

Yalnızca süper sınıfların sorgulandığını unutmayın: sınıflar hiyerarşisindeki arabirimlere eklenen tüm açıklamalar yoksayılır.

@Tekrarlanabilir #

@Repeatable meta açıklaması Java 8’e eklendi. Bu, açıklamanın birden çok örneğinin açıklamanın hedefine eklenebileceğini gösterir. Bu meta açıklamanın değeri yok.

Yansıma yoluyla çalışma zamanı açıklama kontrolleri #

Java’nın Reflection API’si, programcının çalışma zamanı sırasında sınıf alanları, yöntemler ve açıklamalar üzerinde çeşitli kontroller ve işlemler gerçekleştirmesini sağlar. Ancak, bir ek açıklamanın çalışma zamanında tamamen görünür olması için, aşağıdaki örnekte gösterildiği gibi “RetentionPolicy"nin “RUNTIME” olarak değiştirilmesi gerekir:

@interface MyDefaultAnnotation {

}

@Retention(RetentionPolicy.RUNTIME)
@interface MyRuntimeVisibleAnnotation {

}

public class AnnotationAtRuntimeTest {

    @MyDefaultAnnotation
    static class RuntimeCheck1 {
    }
    
    @MyRuntimeVisibleAnnotation
    static class RuntimeCheck2 {
    }
    
    public static void main(String[] args) {
        Annotation[] annotationsByType = RuntimeCheck1.class.getAnnotations();
        Annotation[] annotationsByType2 = RuntimeCheck2.class.getAnnotations();

        System.out.println("default retention: " + Arrays.toString(annotationsByType));
        System.out.println("runtime retention: " + Arrays.toString(annotationsByType2));
    }
}

Yerleşik ek açıklamalar #

Java’nın Standart Sürümü, önceden tanımlanmış bazı ek açıklamalarla birlikte gelir. Bunları kendiniz tanımlamanız gerekmez ve hemen kullanabilirsiniz. Derleyicinin yöntemler, sınıflar ve kodun bazı temel kontrollerini etkinleştirmesine izin verirler.

@Geçersiz kıl

Bu açıklama bir yöntem için geçerlidir ve bu yöntemin bir üst sınıf yöntemini geçersiz kılması veya soyut bir üst sınıf yöntem tanımını uygulaması gerektiğini söyler. Bu açıklama başka bir yöntemle kullanılırsa, derleyici bir hata verecektir.

beton üst sınıf

public class Vehicle {
   public void drive() {
        System.out.println("I am driving");
   }
}

class Car extends Vehicle {
    // Fine
    @Override
    public void drive() {
        System.out.prinln("Brrrm, brrm");
    }
}

soyut sınıf

abstract class Animal  {
   public abstract void makeNoise(); 
}

class Dog extends Animal {
    // Fine
    @Override
    public void makeNoise() {
        System.out.prinln("Woof");
    }
}

Çalışmıyor

class Logger1 {
    public void log(String logString) {
        System.out.prinln(logString);
    }
}

class Logger2 {
    // This will throw compile-time error. Logger2 is not a subclass of Logger1. 
    // log method is not overriding anything
    @Override
    public void log(String logString) {
        System.out.println("Log 2" + logString);
    }
}

Asıl amaç, bir yöntemi geçersiz kıldığınızı düşündüğünüz, ancak aslında yeni bir tane tanımladığınız yanlış yazmayı yakalamaktır.

class Vehicle {
   public void drive() {
        System.out.println("I am driving");
   }
}

class Car extends Vehicle {
    // Compiler error. "dirve" is not the correct method name to override.
    @Override
    public void dirve() {
        System.out.prinln("Brrrm, brrm");
    }
}

“@Override"ın anlamının zaman içinde değiştiğini unutmayın:

  • Java 5’te, açıklamalı yöntemin üst sınıf zincirinde bildirilen soyut olmayan bir yöntemi geçersiz kılması gerektiği anlamına geliyordu.
  • Java 6’dan itibaren, açıklamalı yöntemin sınıflar üst sınıfı/arayüz hiyerarşisinde bildirilen soyut bir yöntemi uygulaması durumunda ayrıca yerine getirilir.

(Bu, kodu Java 5’e geri taşırken zaman zaman sorunlara neden olabilir.)

@Kullanımdan kaldırıldı

Bu, yöntemi kullanımdan kaldırılmış olarak işaretler. Bunun birkaç nedeni olabilir:

  • API kusurlu ve düzeltilmesi pratik değil,

  • API kullanımının hatalara yol açması muhtemeldir,

  • API’nin yerine başka bir API geçmişse,

  • API eski,

  • API deneyseldir ve uyumsuz değişikliklere tabidir,

  • veya yukarıdakilerin herhangi bir kombinasyonu.

Kullanımdan kaldırmanın özel nedeni genellikle API belgelerinde bulunabilir.

Açıklama, kullanırsanız derleyicinin bir hata vermesine neden olur. IDE’ler ayrıca bu yöntemi bir şekilde kullanımdan kaldırılmış olarak vurgulayabilir.

class ComplexAlgorithm {
    @Deprecated
    public void oldSlowUnthreadSafeMethod() {
        // stuff here
    }
    
    public void quickThreadSafeMethod() {
        // client code should use this instead
    }
}

@SuppressUyarılar

Hemen hemen her durumda, derleyici bir uyarı verdiğinde, en uygun eylem nedeni düzeltmektir. Bazı durumlarda (örneğin, tür güvenli olmayan ön jenerik kod kullanan Jenerik kod) bu, mümkün olabilir ve beklenmeyen uyarıları daha net görebilmeniz için beklediğiniz ve düzeltemeyeceğiniz uyarıları bastırmak daha iyidir.

Bu açıklama tüm sınıfa, yönteme veya satıra uygulanabilir. Parametre olarak uyarı kategorisini alır.

@SuppressWarnings("deprecation")
public class RiddledWithWarnings {
    // several methods calling deprecated code here
}

@SuppressWarning("finally")
public boolean checkData() {
    // method calling return from within finally block
}

Beklenmeyen uyarıların da gizlenmesini önlemek için ek açıklamanın kapsamını mümkün olduğunca sınırlamak daha iyidir. Örneğin, ek açıklamanın kapsamını tek bir satırla sınırlamak:

ComplexAlgorithm algorithm = new ComplexAlgorithm();
@SuppressWarnings("deprecation") algoritm.slowUnthreadSafeMethod(); 
// we marked this method deprecated in an example above

@SuppressWarnings("unsafe") List<Integer> list = getUntypeSafeList(); 
// old library returns, non-generic List containing only integers

Bu ek açıklama tarafından desteklenen uyarılar derleyiciden derleyiciye değişiklik gösterebilir. JLS’de yalnızca “işaretlenmemiş” ve “kullanımdan kaldırma” uyarılarından özellikle bahsedilmiştir. Tanınmayan uyarı türleri yoksayılır.

@SafeVarargs

Tür silme nedeniyle, ‘void method(T… t)’, ‘void method(Object[] t)‘ye dönüştürülecektir, bu da derleyicinin varargs kullanımının tip açısından güvenli olduğunu her zaman doğrulayamayacağı anlamına gelir. Örneğin:

private static <T> void generatesVarargsWarning(T... lists) {

Kullanımın güvenli olduğu durumlar vardır, bu durumda uyarıyı bastırmak için SafeVarargs notuyla yönteme açıklama ekleyebilirsiniz. Bu, kullanımınız da güvenli değilse, uyarıyı açıkça gizler.

@FonksiyonelArayüz

Bu, bir FunctionalInterface’i işaretlemek için kullanılan isteğe bağlı bir açıklamadır. FunctionalInterface spesifikasyonuna uymuyorsa derleyicinin şikayet etmesine neden olur (tek bir soyut yöntemi vardır)

@FunctionalInterface
public interface ITrade {
  public boolean check(Trade t);
}

@FunctionalInterface
public interface Predicate<T> {
  boolean test(T t);
}

Ek Açıklamaların arkasındaki fikir #

Java Dil Belirtimi, Ek Açıklamaları şu şekilde açıklar:

Açıklama, bilgileri bir program yapısıyla ilişkilendiren, ancak çalışma zamanında hiçbir etkisi olmayan bir işaretçidir.

Ek açıklamalar, türlerden veya bildirimlerden önce görünebilir. Hem bir türe hem de bir bildirime başvurabilecekleri bir yerde ortaya çıkmaları mümkündür. Bir ek açıklamanın tam olarak ne için geçerli olduğu “meta açıklama” @Target tarafından yönetilir. Daha fazla bilgi için [“Ek açıklama türlerini tanımlama”2 bölümüne bakın.

Ek açıklamalar çok sayıda amaç için kullanılır. Spring ve Spring-MVC gibi çerçeveler, Bağımlılıkların nereye enjekte edileceğini veya isteklerin nereye yönlendirileceğini tanımlamak için ek açıklamalardan yararlanır.

Diğer çerçeveler, kod oluşturma için ek açıklamalar kullanır. Lombok ve JPA, Java (ve SQL) kodu oluşturmak için ek açıklamalar kullanan başlıca örneklerdir.

Açıklama işlemcisini kullanarak zaman işlemeyi derleyin #

Bu örnek, açıklamalı bir öğenin derleme zamanı denetiminin nasıl yapıldığını gösterir.

açıklama #

@Setter notu, yöntemlere uygulanabilecek bir işaretçidir. Ek açıklama, derleme sırasında atılacak, daha sonra mevcut olmayacaktır.

package annotation;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

@Retention(RetentionPolicy.SOURCE)
@Target(ElementType.METHOD)
public @interface Setter {
}

Açıklama işlemcisi #

SetterProcessor sınıfı, derleyici tarafından ek açıklamaları işlemek için kullanılır. @Setter notu ile açıklamalı metotların public olup olmadığını kontrol eder, ismi set ile başlayan ve 4. harf büyük harf olan statik olmayan yöntemlerdir. Bu koşullardan biri sağlanmazsa, ‘Mesajcı’ya bir hata yazılır. Derleyici bunu stderr’e yazar, ancak diğer araçlar bu bilgiyi farklı şekilde kullanabilir. Örneğin. NetBeans IDE, kullanıcının düzenleyicide hata mesajlarını görüntülemek için kullanılan açıklama işlemcilerini belirlemesine izin verir.

package annotation.processor;

import annotation.Setter;
import java.util.Set;
import javax.annotation.processing.AbstractProcessor;
import javax.annotation.processing.Messager;
import javax.annotation.processing.ProcessingEnvironment;
import javax.annotation.processing.RoundEnvironment;
import javax.annotation.processing.SupportedAnnotationTypes;
import javax.annotation.processing.SupportedSourceVersion;
import javax.lang.model.SourceVersion;
import javax.lang.model.element.Element;
import javax.lang.model.element.ElementKind;
import javax.lang.model.element.ExecutableElement;
import javax.lang.model.element.Modifier;
import javax.lang.model.element.TypeElement;
import javax.tools.Diagnostic;

@SupportedAnnotationTypes({"annotation.Setter"})
@SupportedSourceVersion(SourceVersion.RELEASE_8)
public class SetterProcessor extends AbstractProcessor {

    private Messager messager;

    @Override
    public boolean process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv) {
        // get elements annotated with the @Setter annotation
        Set<? extends Element> annotatedElements = roundEnv.getElementsAnnotatedWith(Setter.class);

        for (Element element : annotatedElements) {
            if (element.getKind() == ElementKind.METHOD) {
                // only handle methods as targets
                checkMethod((ExecutableElement) element);
            }
        }

        // don't claim annotations to allow other processors to process them
        return false;
    }

    private void checkMethod(ExecutableElement method) {
        // check for valid name
        String name = method.getSimpleName().toString();
        if (!name.startsWith("set")) {
            printError(method, "setter name must start with \"set\"");
        } else if (name.length() == 3) {
            printError(method, "the method name must contain more than just \"set\"");
        } else if (Character.isLowerCase(name.charAt(3))) {
            if (method.getParameters().size() != 1) {
                printError(method, "character following \"set\" must be upper case");
            }
        }

        // check, if setter is public
        if (!method.getModifiers().contains(Modifier.PUBLIC)) {
            printError(method, "setter must be public");
        }

        // check, if method is static
        if (method.getModifiers().contains(Modifier.STATIC)) {
            printError(method, "setter must not be static");
        }
    }

    private void printError(Element element, String message) {
        messager.printMessage(Diagnostic.Kind.ERROR, message, element);
    }

    @Override
    public void init(ProcessingEnvironment processingEnvironment) {
        super.init(processingEnvironment);

        // get messager for printing errors
        messager = processingEnvironment.getMessager();
    }

}

Ambalajlama #

Derleyici tarafından uygulanabilmesi için, açıklama işlemcisinin SPI’ye sunulması gerekir (bkz. ServiceLoader).

Bunu yapmak için, diğer dosyalara ek olarak açıklama işlemcisini ve açıklamayı içeren jar dosyasına META-INF/services/javax.annotation.processing.Processor metin dosyasının eklenmesi gerekir. Dosya, açıklama işlemcisinin tam adını içermelidir, yani şöyle görünmelidir

annotation.processor.SetterProcessor

jar dosyasının aşağıda “AnnotationProcessor.jar” olarak adlandırıldığını varsayacağız.

Yinelenen Ek Açıklamalar #

Java 8’e kadar, aynı açıklamanın iki örneği tek bir öğeye uygulanamazdı. Standart geçici çözüm, bir dizi başka açıklama içeren bir kapsayıcı açıklama kullanmaktı:

// Author.java
@Retention(RetentionPolicy.RUNTIME)
public @interface Author {
    String value();
}

// Authors.java
@Retention(RetentionPolicy.RUNTIME)
public @interface Authors {
    Author[] value();
}

// Test.java
@Authors({
    @Author("Mary"),
    @Author("Sam")
})
public class Test {
    public static void main(String[] args) {
        Author[] authors = Test.class.getAnnotation(Authors.class).value();
        for (Author author : authors) {
            System.out.println(author.value());
            // Output:
            // Mary
            // Sam
        }
    }
}

Java 8, “@Repeatable” ek açıklamasını kullanarak kapsayıcı açıklamalarını kullanmanın daha temiz ve şeffaf bir yolunu sağlar. İlk önce bunu ‘Author’ sınıfına ekliyoruz:

@Repeatable(Authors.class)

Bu, Java’ya birden çok “@Yazar” ek açıklamasını “@Yazarlar” kapsayıcısıyla çevrelenmiş gibi işlemesini söyler. “@Author” dizisine kapsayıcısı yerine kendi sınıfına göre erişmek için “Class.getAnnotationsByType()” öğesini de kullanabiliriz:

@Author("Mary")
@Author("Sam")
public class Test {
    public static void main(String[] args) {
        Author[] authors = Test.class.getAnnotationsByType(Author.class);
        for (Author author : authors) {
            System.out.println(author.value());
            // Output:
            // Mary
            // Sam
        }
    }
}
son sürüm

Devralınan Ek Açıklamalar #

Varsayılan olarak sınıf açıklamaları, onları genişleten türler için geçerli değildir. Bu, açıklama tanımına “@Inherited” notu eklenerek değiştirilebilir.

Örnek #

Aşağıdaki 2 Ek Açıklamayı göz önünde bulundurun:

@Inherited
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface InheritedAnnotationType {
}

ve

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface UninheritedAnnotationType {
}

Üç sınıf şöyle açıklanırsa:

@UninheritedAnnotationType
class A {
}

@InheritedAnnotationType
class B extends A {
}

class C extends B {
}

bu kodu çalıştırıyor

System.out.println(new A().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println(new B().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println(new C().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println("_________________________________");
System.out.println(new A().getClass().getAnnotation(UninheritedAnnotationType.class));
System.out.println(new B().getClass().getAnnotation(UninheritedAnnotationType.class));
System.out.println(new C().getClass().getAnnotation(UninheritedAnnotationType.class));

buna benzer bir sonuç yazdıracaktır (açıklamanın paketlerine bağlı olarak):

null
@InheritedAnnotationType()
@InheritedAnnotationType()
_________________________________
@UninheritedAnnotationType()
null
null

Ek açıklamaların arabirimlerden değil, yalnızca sınıflardan devralınabileceğini unutmayın.

Çalışma zamanında Açıklama değerlerini alma #

Ek Açıklamanın geçerli özelliklerini, kendisine bir Açıklama uygulanmış olan Yöntemi veya Alanı veya Sınıfı getirmek için Yansıma kullanarak ve ardından istenen özellikleri getirerek getirebilirsiniz.

@Retention(RetentionPolicy.RUNTIME)
@interface MyAnnotation {
    String key() default "foo";
    String value() default "bar";
}


class AnnotationExample {
    // Put the Annotation on the method, but leave the defaults
    @MyAnnotation
    public void testDefaults() throws Exception {
        // Using reflection, get the public method "testDefaults", which is this method with no args
        Method method = AnnotationExample.class.getMethod("testDefaults", null);

        // Fetch the Annotation that is of type MyAnnotation from the Method
        MyAnnotation annotation = (MyAnnotation)method.getAnnotation(MyAnnotation.class);

        // Print out the settings of the Annotation
        print(annotation);
    }

    //Put the Annotation on the method, but override the settings
    @MyAnnotation(key="baz", value="buzz")
    public void testValues() throws Exception {
        // Using reflection, get the public method "testValues", which is this method with no args
        Method method = AnnotationExample.class.getMethod("testValues", null);

        // Fetch the Annotation that is of type MyAnnotation from the Method
        MyAnnotation annotation = (MyAnnotation)method.getAnnotation(MyAnnotation.class);

        // Print out the settings of the Annotation
        print(annotation);
    }

    public void print(MyAnnotation annotation) {
        // Fetch the MyAnnotation 'key' & 'value' properties, and print them out 
        System.out.println(annotation.key() + " = " + annotation.value());
    }

    public static void main(String[] args) {
        AnnotationExample example = new AnnotationExample();
        try {
            example.testDefaults();
            example.testValues();
        } catch( Exception e ) {
            // Shouldn't throw any Exceptions
            System.err.println("Exception [" + e.getClass().getName() + "] - " + e.getMessage());
            e.printStackTrace(System.err);
        }
    }
}

çıktı olacak

foo = bar
baz = buzz

’this’ ve alıcı parametreleri için açıklamalar #

Java ek açıklamaları ilk kez tanıtıldığında, bir örnek yöntemin hedefine veya bir iç sınıf yapıcısı için gizli yapıcı parametresine açıklama eklemek için herhangi bir hüküm yoktu. Bu, Java 8’de alıcı parametresi bildirimlerinin eklenmesiyle giderildi; JLS 8.4.1’e bakın.

Alıcı parametresi, bir örnek yöntemi veya bir iç sınıfın oluşturucusu için isteğe bağlı bir sözdizimsel aygıttır. Bir örnek yöntemi için alıcı parametresi, yöntemin çağrıldığı nesneyi temsil eder. Bir iç sınıfın oluşturucusu için alıcı parametresi, yeni oluşturulan nesnenin hemen çevreleyen örneğini temsil eder. Her iki durumda da, alıcı parametresi yalnızca, temsil edilen nesnenin türünün kaynak kodunda belirtilmesine izin vermek için mevcuttur, böylece türe açıklama eklenebilir. Alıcı parametresi resmi bir parametre değildir; daha doğrusu, herhangi bir değişkenin bildirimi değildir (§4.12.3), hiçbir zaman bir yöntem çağırma ifadesinde veya nitelikli sınıf örneği oluşturma ifadesinde argüman olarak iletilen herhangi bir değere bağlı değildir ve hiçbir şekilde etkisi yoktur. Çalışma süresi.

Aşağıdaki örnek, her iki tür alıcı parametresi için sözdizimini gösterir:

public class Outer {
    public class Inner {
        public Inner (Outer this) {
           // ...
        }
        public void doIt(Inner this) {
           // ...
        }
    }
}

Alıcı parametrelerinin tek amacı, ek açıklamalar eklemenize izin vermektir. Örneğin, amacı bir yöntem çağrıldığında bir “Kapatılabilir” nesnenin kapatılmadığını belirtmek olan özel bir “@IsOpen” ek açıklamanız olabilir. Örneğin:

public class MyResource extends Closeable {
    public void update(@IsOpen MyResource this, int value) {
        // ...
    }

    public void close() {
        // ...
    }
}

Bir düzeyde, ’this’ üzerindeki ‘@IsOpen’ ek açıklaması sadece belge işlevi görebilir. Ancak, potansiyel olarak daha fazlasını yapabiliriz. Örneğin:

  • Bir açıklama işlemcisi, “güncelleme” çağrıldığında “bu"nun kapalı durumda olmadığına dair bir çalışma zamanı denetimi ekleyebilir.
  • Bir kod denetleyicisi, “güncelleme” çağrıldığında “bu"nun kapatılabileceği durumları bulmak için statik bir kod analizi yapabilir.

Birden çok açıklama değeri ekleyin #

Bir Annotation parametresi, bir dizi olarak tanımlanmışsa, birden çok değeri kabul edebilir. Örneğin, @SuppressWarnings standart ek açıklaması şu şekilde tanımlanır:

public @interface SuppressWarnings {
    String[] value();
}

“değer” parametresi, bir Dize dizisidir. Dizi başlatıcılara benzer bir gösterim kullanarak birden çok değer ayarlayabilirsiniz:

@SuppressWarnings({"unused"})
@SuppressWarnings({"unused", "javadoc"})

Yalnızca tek bir değer ayarlamanız gerekiyorsa, parantezler atlanabilir:

@SuppressWarnings("unused")
