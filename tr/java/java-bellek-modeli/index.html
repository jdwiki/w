Java Bellek Modeli, bir iş parçacığının başka bir iş parçacığı tarafından yapılan bellek yazmalarının etkilerini görmesinin garanti edildiği koşulları belirten JLS bölümüdür. Son sürümlerdeki ilgili bölüm “JLS 17.4 Bellek Modeli"dir (Java 8, Java 7, Java 6’te)

Java 5’te Java Bellek Modelinde (diğer şeylerin yanı sıra) “geçici” çalışma şeklini değiştiren büyük bir revizyon yapıldı. O zamandan beri, bellek modeli esasen değişmedi.

Hafıza Modeli için Motivasyon #

Aşağıdaki örneği göz önünde bulundurun:

public class Example {
    public int a, b, c, d;
    
    public void doIt() {
       a = b + 1;
       c = d + 1;
    }
}

Bu sınıf tek iş parçacıklı bir uygulama ise, gözlemlenebilir davranış tam olarak beklediğiniz gibi olacaktır. Örneğin:

public class SingleThreaded {
    public static void main(String[] args) {
        Example eg = new Example();
        System.out.println(eg.a + ", " + eg.c);
        eg.doIt();
        System.out.println(eg.a + ", " + eg.c);
    }
}

çıktı alacak:

0, 0
1, 1

“main” iş parçacığının anlayabileceği kadar, ‘main()’ yöntemindeki ve ‘doIt()’ yöntemindeki ifadeler kaynak kodda yazıldığı sırayla yürütülür. Bu, Java Dil Belirtiminin (JLS) açık bir gereksinimidir.

Şimdi çok iş parçacıklı bir uygulamada kullanılan aynı sınıfı düşünün.

public class MultiThreaded {
    public static void main(String[] args) {
        final Example eg = new Example();
        new Thread(new Runnable() {
            public void run() {
                while (true) {
                    eg.doIt();
                }
            }
        }).start();
        while (true) {
            System.out.println(eg.a + ", " + eg.c);
        }
    }
}

Bu ne yazdıracak?

Aslında, JLS’ye göre bunun basılacağını tahmin etmek mümkün değil:

  • Başlamak için muhtemelen birkaç ‘0, 0’ satırı göreceksiniz.
  • O zaman muhtemelen N, N veya N, N + 1 gibi satırlar görürsünüz.
  • N + 1, N gibi satırlar görebilirsiniz.
  • Teoride, “0, 0” satırlarının sonsuza kadar devam ettiğini bile görebilirsiniz1.

1 - Pratikte ‘println’ ifadelerinin varlığı, tesadüfi bir senkronizasyona ve bellek önbelleğinin boşaltılmasına neden olabilir. Bunun, yukarıdaki davranışa neden olacak bazı etkileri gizlemesi muhtemeldir.

Peki bunları nasıl açıklayabiliriz?

Görevlerin yeniden sıralanması #

Beklenmeyen sonuçlar için olası bir açıklama, JIT derleyicisinin “doIt()” yöntemindeki atamaların sırasını değiştirmiş olmasıdır. JLS, ifadelerin geçerli iş parçacığının perspektifinden *sırayla görünüyor yürütülmesini gerektirir. Bu durumda, “doIt()” yönteminin kodundaki hiçbir şey, bu iki ifadenin (varsayımsal) yeniden sıralanmasının etkisini gözlemleyemez. Bu, JIT derleyicisinin bunu yapmasına izin verileceği anlamına gelir.

Bunu neden yapsın?

Tipik modern donanımda, makine talimatları, bir talimat dizisinin farklı aşamalarda olmasına izin veren bir talimat boru hattı kullanılarak yürütülür. Komut yürütmenin bazı aşamaları diğerlerinden daha uzun sürer ve bellek işlemleri daha uzun sürer. Akıllı bir derleyici, çakışma miktarını en üst düzeye çıkarmak için talimatları sipariş ederek işlem hattının talimat verimini optimize edebilir. Bu, ifadelerin bölümlerinin sıra dışı yürütülmesine neden olabilir. JLS, geçerli iş parçacığının perspektifinden hesaplamanın sonucunu etkilememesi koşuluyla buna izin verir.

Bellek önbelleklerinin etkileri #

İkinci bir olası açıklama, bellek önbelleğe almanın etkisidir. Klasik bir bilgisayar mimarisinde, her işlemcinin küçük bir kayıt seti ve daha büyük miktarda belleği vardır. Kayıtlara erişim, ana belleğe erişimden çok daha hızlıdır. Modern mimarilerde, yazmaçlardan daha yavaş, ancak ana bellekten daha hızlı olan bellek önbellekleri vardır.

Bir derleyici, değişkenlerin kopyalarını kayıtlarda veya bellek önbelleklerinde tutmaya çalışarak bundan yararlanacaktır. Bir değişkenin ana belleğe boşaltılması gerekmiyorsa veya bellekten okunması *gerekmiyorsa, bunu yapmamanın önemli performans avantajları vardır. JLS’nin bellek işlemlerinin başka bir iş parçacığı tarafından görülmesini gerektirmediği durumlarda, Java JIT derleyicisi, ana bellek okumalarını ve yazmalarını zorlayacak “okuma engeli” ve “yazma engeli” talimatlarını eklemeyecektir. Bir kez daha, bunu yapmanın performans faydaları önemlidir.

Uygun senkronizasyon #

Şimdiye kadar JLS’nin JIT derleyicisinin bellek işlemlerini yeniden düzenleyerek veya bunlardan kaçınarak tek iş parçacıklı kodu daha hızlı yapan kod oluşturmasına izin verdiğini gördük. Ancak diğer iş parçacıkları ana bellekteki (paylaşılan) değişkenlerin durumunu gözlemleyebildiğinde ne olur?

Cevap, diğer iş parçacıklarının Java ifadelerinin kod sırasına göre imkansız gibi görünen değişken durumları gözlemlemekle yükümlü olduğudur. Bunun çözümü uygun senkronizasyon kullanmaktır. Üç ana yaklaşım şunlardır:

  • İlkel muteksler ve ‘senkronize’ yapıları kullanma.
  • ‘uçucu’ değişkenleri kullanma.
  • Daha yüksek düzeyde eşzamanlılık desteği kullanma; Örneğin. ‘java.util.concurrent’ paketlerindeki sınıflar.

Ancak bununla bile, senkronizasyonun nerede gerekli olduğunu ve hangi etkilere güvenebileceğinizi anlamak önemlidir. Java Bellek Modeli burada devreye giriyor.

Bellek Modeli #

Java Bellek Modeli, bir iş parçacığının başka bir iş parçacığı tarafından yapılan bellek yazmalarının etkilerini görmesinin garanti edildiği koşulları belirten JLS bölümüdür. Bellek Modeli, adil bir biçimsel titizlik derecesi ile belirtilir ve (sonuç olarak) anlamak için ayrıntılı ve dikkatli bir okuma gerektirir. Ancak temel ilke, belirli yapıların bir değişkenin bir iş parçacığı tarafından yazılması ve aynı değişkenin başka bir iş parçacığı tarafından daha sonra okunması arasında bir “önceki olay” ilişkisi yaratmasıdır. “Önceden olur” ilişkisi varsa, JIT derleyicisi, okuma işleminin yazma tarafından yazılan değeri görmesini sağlayacak kodu oluşturmakla zorunludur.

Bununla donanmış olarak, bir Java programında bellek tutarlılığı hakkında akıl yürütmek ve bunun tüm yürütme platformları için öngörülebilir ve tutarlı olup olmayacağına karar vermek mümkündür.

ilişkilerden önce olur #

(Aşağıda Java Dil Belirtiminin söylediklerinin basitleştirilmiş bir versiyonu bulunmaktadır. Daha derin bir anlayış için belirtimin kendisini okumanız gerekir.)

Önce-olur ilişkileri, Bellek Modelinin, belleğin görünürlüğünü anlamamızı ve bunun hakkında muhakeme yapmamızı sağlayan bir parçasıdır. JLS’nin dediği gibi (JLS 17.4.5):

“İki eylem bir önce-olur ilişkisiyle sıralanabilir. Bir eylem diğerinden önce olursa, ilki ikinciden önce görünür ve sıralanır.”

Ne anlama geliyor?

Hareketler #

Yukarıdaki alıntının atıfta bulunduğu eylemler JLS 17.4.2‘de belirtilmiştir. Spesifikasyon tarafından tanımlanan 5 tür eylem listelenmiştir:

  • Oku: Uçucu olmayan bir değişkeni okuma.

  • Yaz: Uçucu olmayan bir değişken yazmak.

  • Senkronizasyon işlemleri:

    • Volatile read: Reading a volatile variable.

    • Volatile write: Writing a volatile variable.

    • Lock. Locking a monitor

    • Unlock. Unlocking a monitor.

    • The (synthetic) first and last actions of a thread.

    • Actions that start a thread or detect that a thread has terminated.

  • Dış Eylemler. Programın içinde bulunduğu ortama bağlı olarak sonucu olan bir eylem.

  • İplik sapma eylemleri. Bunlar, belirli türdeki sonsuz döngülerin davranışını modeller.

Program Sırası ve Senkronizasyon Sırası #

Bu iki sıralama ( JLS 17.4.3 ve JLS 17.4.4 ) bir Java’daki ifadelerin yürütülmesini yönetir.

Program sırası, tek bir iş parçacığı içinde deyim yürütme sırasını tanımlar.

Senkronizasyon sırası, bir senkronizasyon ile bağlanan iki ifade için ifade yürütme sırasını tanımlar:

  • Monitördeki kilit açma eylemi, o monitördeki sonraki tüm kilitleme eylemleriyle eşzamanlanır.

  • Uçucu bir değişkene yazma, aynı değişkenin herhangi bir iş parçacığı tarafından sonraki tüm okumalarıyla eşzamanlanır.

  • Bir diziyi başlatan bir eylem (yani “Thread.start()” çağrısı) başlattığı dizideki ilk eylemle eşzamanlanır (yani dizinin “run()” yöntemine yapılan çağrı).

  • Alanların varsayılan başlatması, her iş parçacığındaki ilk eylemle eşzamanlanır. (Bunun açıklaması için JLS’ye bakın.)

  • Bir iş parçacığındaki son işlem, sonlandırmayı algılayan başka bir iş parçacığındaki herhangi bir işlemle eşzamanlanır; Örneğin. “true” döndüren bir “join()” çağrısının veya “isTerminated()” çağrısının dönüşü.

  • Bir iş parçacığı başka bir iş parçacığını keserse, ilk iş parçacığındaki kesme çağrısı, başka bir iş parçacığının iş parçacığının kesildiğini algıladığı noktayla senkronize olur.

Sipariş Öncesi #

Bu sıralama ( JLS 17.4.5 ), bir belleğe yazma işleminin sonraki bir bellek okumasında görünür olmasının garanti edilip edilmediğini belirleyen şeydir.

Daha spesifik olarak, bir “v” değişkeninin okunmasının, ancak ve ancak “yazma(v)” önce-olursa “okuma(v)” olması ve " v`. Araya giren yazma işlemleri varsa, o zaman ‘read(v)’ öncekinden ziyade bunların sonuçlarını görebilir.

Önceden olur sıralamasını tanımlayan kurallar aşağıdaki gibidir:

  • Kural #1’den Önce Olur - x ve y aynı iş parçacığının eylemleriyse ve x program sırasında y’den önce geliyorsa, o zaman x y’den önce olur.

  • Önce Olur Kural #2 - Bir nesnenin yapıcısının sonundan o nesne için bir sonlandırıcının başlangıcına kadar bir önce-olur kenarı vardır.

  • Önce-Önce-Kural #3 - Eğer bir x eylemi, sonraki bir y eylemiyle eşzamanlanırsa, o zaman x *y’den-önce-olur.

  • Önce-Önce-Kural #4 - Eğer x y’den-önce-olursa ve y z’den önce-olursa o zaman x z’den-önce-olur.

Ayrıca, Java standart kitaplıklarındaki çeşitli sınıflar, önce-olur ilişkilerini tanımlayacak şekilde belirtilir. Bunu, garantinin nasıl karşılanacağını tam olarak bilmenize gerek kalmadan * bir şekilde * gerçekleştiği şeklinde yorumlayabilirsiniz.

Bazı örneklere uygulanan öncesi akıl yürütme #

Yazmaların sonraki okumalarda görünür olup olmadığını kontrol etmek için önce-olur mantığının nasıl uygulanacağını göstermek için bazı örnekler sunacağız.

Tek iş parçacıklı kod #

Beklediğiniz gibi, yazma işlemleri tek iş parçacıklı bir programda sonraki okumalarda her zaman görülebilir.

public class SingleThreadExample {
    public int a, b;
    
    public int add() {
       a = 1;         // write(a)
       b = 2;         // write(b)
       return a + b;  // read(a) followed by read(b)
    }
}

Olmadan Önce Kural 1’e göre:

  1. ‘yaz(a)’ eylemi, ‘yaz(b)’ eyleminden önce olur.
  2. ‘yaz(b)’ eylemi, ‘oku(a)’ eyleminden önce olur.
  3. ‘oku(a)’ eylemi, ‘oku(a)’ eyleminden önce olur.

Kural 4: Olmadan Önce:

  1. ‘yazma(a)’ önce olur ‘yazma(b)’ VE ‘yazma(b)’ önce olur ‘okuma(a)’ GEREKTİRİR ‘yazma(a)’ önce olur ’ (a)` okuyun.
  2. ‘yazma(b)’ önce-olur ‘oku(a)’ VE ‘oku(a)’ önce-olur ‘oku(b)’ ANLAMINA GELİR ‘yaz(b)’ önce-olur ’ (b)`yi okuyun.

Özetliyor:

  1. ‘write(a)’ daha önce gerçekleşir ‘okuma(a)’ ilişkisi, ‘a + b’ ifadesinin ‘a’nın doğru değerini görmesinin garanti edildiği anlamına gelir.
  2. ‘write(b)’ daha önce gerçekleşir ‘okuma(b)’ ilişkisi, ‘a + b’ ifadesinin ‘b’nin doğru değerini görmesinin garanti edildiği anlamına gelir.

2 iş parçacığına sahip bir örnekte ‘uçucu’ davranışı

Aşağıdaki örnek kodu, “uçucu” için Bellek Modelinin bazı etkilerini araştırmak için kullanacağız.

public class VolatileExample {
    private volatile int a;
    private int b;         // NOT volatile
    
    public void update(int first, int second) {
       b = first;         // write(b)
       a = second;         // write-volatile(a)
    }

    public int observe() {
       return a + b;       // read-volatile(a) followed by read(b)
    }
}

İlk olarak, 2 iş parçacığı içeren aşağıdaki ifade dizisini göz önünde bulundurun:

  1. Tek bir “VolatileExample” örneği oluşturulur; ona ’ve’ deyin,
  2. ve.update(1, 2) tek bir iş parçacığında çağrılır ve
  3. ve.observe() başka bir iş parçacığında çağrılır.

Olmadan Önce Kural 1’e göre:

  1. ‘(a) yaz’ eylemi, ‘geçici-yaz(a)’ eyleminden önce olur.
  2. ‘geçici-oku(a)’ eylemi, ‘oku(b)’ eyleminden önce olur.

Kural 2’den Önce Olursa:

  1. İlk dizideki ‘uçucu-yazma(a)’ eylemi, ikinci iş parçacığındaki ‘geçici-okuma(a)’ eyleminden önce gerçekleşir.

Kural 4: Olmadan Önce:

  1. İlk dizideki ‘yazma(b)’ eylemi, ikinci dizideki ‘oku(b)’ eyleminden önce gerçekleşir.

Başka bir deyişle, bu belirli dizi için, 2. iş parçacığının, ilk iş parçacığı tarafından yapılan geçici olmayan değişken ‘b’ için yapılan güncellemeyi göreceğini garanti ediyoruz. Bununla birlikte, “update” yöntemindeki atamalar tam tersi olsaydı veya “observe()” yöntemi “a"dan önce “b” değişkenini okursa, o zaman önce olur zincir kırılacaktı. İkinci iş parçacığındaki “uçucu-okuma(a)”, ilk iş parçacığındaki “uçucu-yazma(a)“dan sonra değilse zincir de kırılırdı.

Zincir kırıldığında, “gözlem()“in “b"nin doğru değerini göreceğine dair bir garanti yoktur.

Üç iplikli uçucu #

Önceki örneğe üçüncü bir iş parçacığı eklediğimizi varsayalım:

  1. Tek bir “VolatileExample” örneği oluşturulur; ona ’ve’ deyin,
  2. İki ileti dizisi “güncelleme” adını verir:
    • ve.update(1, 2) is called in one thread,
    • ve.update(3, 4) is called in the second thread,
  3. ve.observe() daha sonra üçüncü bir iş parçacığında çağrılır.

Bunu tamamen analiz etmek için, birinci ve ikinci iş parçacığındaki ifadelerin olası tüm serpiştirmelerini göz önünde bulundurmamız gerekir. Bunun yerine, sadece ikisini ele alacağız.

Senaryo 1 - “güncelleme(1, 2)“nin “güncelleme(3,4)“den önce geldiğini varsayalım, şu sırayı alıyoruz:

write(b, 1), write-volatile(a, 2)     // first thread
write(b, 3), write-volatile(a, 4)     // second thread
read-volatile(a), read(b)             // third thread

Bu durumda, ‘write(b, 3)‘den ‘read(b)‘ye kadar kesintisiz bir önce olur zinciri olduğunu görmek kolaydır. Ayrıca, “b"ye araya giren bir yazma yoktur. Dolayısıyla, bu senaryo için, üçüncü iş parçacığının “b"yi “3” değerine sahip olarak görmesi garanti edilir.

Senaryo #2 - “güncelleme(1, 2)” ve “güncelleme(3,4)” öğelerinin üst üste bindiğini ve katyonların aşağıdaki gibi serpiştirildiğini varsayalım:

write(b, 3)                           // second thread
write(b, 1)                           // first thread
write-volatile(a, 2)                  // first thread
write-volatile(a, 4)                  // second thread
read-volatile(a), read(b)             // third thread

Şimdi, ‘write(b, 3)‘den ‘read(b)‘ye bir önce-olur zinciri varken, diğer iş parçacığı tarafından gerçekleştirilen araya giren bir ‘write(b, 1)’ eylemi var. Bu, read(b)nin hangi değeri göreceğinden emin olamayacağımız anlamına gelir.

(Bir kenara: Bu, çok sınırlı durumlar dışında, geçici olmayan değişkenlerin görünürlüğünü sağlamak için “uçucu"ya güvenemeyeceğimizi gösterir.)

Bellek Modelini anlama ihtiyacından nasıl kaçınılır #

Bellek Modelini anlamak ve uygulamak zordur. Çok kanallı kodun doğruluğu hakkında akıl yürütmeniz gerekiyorsa, ancak yazdığınız her çok kanallı uygulama için bu akıl yürütmeyi yapmak istemiyorsanız yararlıdır.

Java’da eşzamanlı kod yazarken aşağıdaki ilkeleri benimserseniz, önceki olaylar mantığına başvurma ihtiyacından büyük ölçüde kurtulabilirsiniz.

  • Mümkünse değişmez veri yapıları kullanın. Düzgün uygulanmış değişmez bir sınıf iş parçacığı için güvenli olacak ve onu diğer sınıflarla birlikte kullandığınızda iş parçacığı güvenliği sorunları ortaya çıkarmayacaktır.

  • “Güvenli olmayan yayınları” anlayın ve kaçının.

  • İş parçacığı için güvenli olması gereken değişken nesnelerde duruma erişimi senkronize etmek için ilkel muteksler veya ‘Kilitle’ nesneleri kullanın1.

  • Doğrudan yönetim dizileri oluşturmaya çalışmak yerine ‘Executor’ / ‘ExecutorService’ veya çatal birleştirme çerçevesini kullanın.

  • Doğrudan wait/notify/notifyAll kullanmak yerine gelişmiş kilitler, semaforlar, mandallar ve bariyerler sağlayan `java.util.concurrent sınıflarını kullanın.

  • Eşzamanlı olmayan koleksiyonların harici senkronizasyonu yerine haritaların, kümelerin, listelerin, kuyrukların ve deque’lerin ‘java.util.concurrent’ sürümlerini kullanın.

Genel ilke, “kendi eşzamanlılığınızı yuvarlamak” yerine Java’nın yerleşik eşzamanlılık kitaplıklarını kullanmaya çalışmaktır. Düzgün kullanırsanız, çalıştıklarına güvenebilirsiniz.

1 - Tüm nesnelerin iş parçacığı için güvenli olması gerekmez. Örneğin, bir nesne veya nesneler iş parçacığıyla sınırlıysa (yani yalnızca bir iş parçacığına erişilebiliyorsa), iş parçacığı güvenliği önemli değildir.
