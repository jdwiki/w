Java’da nesneler yığında tahsis edilir ve yığın belleği otomatik çöp toplama ile geri alınır. Bir uygulama programı, bir Java nesnesini açıkça silemez.

Java çöp toplamanın temel ilkeleri, [Çöp toplama][1] örneğinde açıklanmıştır. Diğer örnekler, sonlandırmayı, çöp toplayıcının elle nasıl tetikleneceğini ve depolama sızıntıları sorununu açıklar.

[1]: https://www.wikiod.com/tr/java/java-bellek-yonetimi#Çöp toplama

Yığın, PermGen ve Yığın boyutlarını ayarlama #

Bir Java sanal makinesi başladığında, Yığının ne kadar büyük olacağını ve iş parçacığı yığınları için varsayılan boyutu bilmesi gerekir. Bunlar, ‘java’ komutundaki komut satırı seçenekleri kullanılarak belirtilebilir. Java 8’den önceki Java sürümleri için, Heap’in PermGen bölgesinin boyutunu da belirtebilirsiniz.

PermGen’in Java 8’de kaldırıldığını ve PermGen boyutunu ayarlamaya çalışırsanız seçeneğin yok sayılacağını (bir uyarı mesajıyla birlikte) unutmayın.

Yığın ve Yığın boyutlarını açıkça belirtmezseniz JVM, sürüme ve platforma özel bir şekilde hesaplanan varsayılanları kullanır. Bu, uygulamanızın çok az veya çok fazla bellek kullanmasına neden olabilir. Bu genellikle iş parçacığı yığınları için uygundur, ancak çok fazla bellek kullanan bir program için sorunlu olabilir.

Yığın, PermGen ve varsayılan Yığın boyutlarını ayarlama:

Aşağıdaki JVM seçenekleri yığın boyutunu ayarlar:

  • -Xms<size> - ilk yığın boyutunu ayarlar
  • -Xmx<size> - maksimum yığın boyutunu ayarlar
  • -XX:PermSize<size> - ilk PermGen boyutunu ayarlar
  • -XX:MaxPermSize<size> - maksimum PermGen boyutunu ayarlar
  • -Xss<size> - varsayılan iş parçacığı yığın boyutunu ayarlar

“” parametresi bir bayt sayısı olabilir veya “k”, “m” veya “g” son ekine sahip olabilir. İkincisi, boyutu sırasıyla kilobayt, megabayt ve gigabayt olarak belirtir.

Örnekler:

$ java -Xms512m -Xmx1024m JavaApp
$ java -XX:PermSize=64m -XX:MaxPermSize=128m JavaApp
$ java -Xss512k JavaApp

Varsayılan boyutları bulma:

-XX:+printFlagsFinal seçeneği, JVM’yi başlatmadan önce tüm bayrakların değerlerini yazdırmak için kullanılabilir. Bu, yığın ve yığın boyutu ayarları için varsayılanları aşağıdaki gibi yazdırmak için kullanılabilir:

  • Linux, Unix, Solaris ve Mac OSX için

    $ java -XX:+PrintFlagsFinal -version | grep -iE ‘HeapSize|PermSize|ThreadStackSize’

  • Pencereler için:

    java -XX:+PrintFlagsFinal -version | findstr /i “HeapSize PermSize ThreadStackSize”

Yukarıdaki komutların çıktısı aşağıdakine benzer olacaktır:

uintx InitialHeapSize                          := 20655360        {product}
uintx MaxHeapSize                              := 331350016       {product}
uintx PermSize                                  = 21757952        {pd product}
uintx MaxPermSize                               = 85983232        {pd product}
 intx ThreadStackSize                           = 1024            {pd product}

Boyutlar bayt olarak verilmiştir.

C++ yaklaşımı - yeni ve sil #

C++ gibi bir dilde, uygulama programı, dinamik olarak ayrılmış bellek tarafından kullanılan belleği yönetmekten sorumludur. C++ yığınında “yeni” operatörü kullanılarak bir nesne oluşturulduğunda, nesneyi elden çıkarmak için “delete” operatörünün karşılık gelen bir kullanımı olmalıdır:

  • Program bir nesneyi ‘silmeyi’ unutursa ve onu “unutursa”, ilgili bellek uygulamaya kaybolur. Bu durum için kullanılan terim bir bellek sızıntısıdır ve çok fazla bellek sızıntısı olması, bir uygulamanın giderek daha fazla bellek kullanmasına ve sonunda çökmesine neden olur.

  • Öte yandan, bir uygulama aynı nesneyi iki kez ‘silmeye’ çalışırsa veya silindikten sonra bir nesneyi kullanırsa, bellek bozulmasıyla ilgili sorunlar nedeniyle uygulama çökebilir.

Karmaşık bir C++ programında, “new” ve “delete” kullanarak bellek yönetimi uygulamak zaman alıcı olabilir. Gerçekten de, bellek yönetimi yaygın bir hata kaynağıdır.

Java yaklaşımı - çöp toplama #

Java farklı bir yaklaşım benimsiyor. Java, açık bir “sil” operatörü yerine, artık ihtiyaç duyulmayan nesneler tarafından kullanılan belleği geri kazanmak için çöp toplama olarak bilinen otomatik bir mekanizma sağlar. Java çalışma zamanı sistemi, atılacak nesneleri bulma sorumluluğunu üstlenir. Bu görev, çöp toplayıcı veya kısaca GC adı verilen bir bileşen tarafından gerçekleştirilir.

Bir Java programının yürütülmesi sırasında herhangi bir zamanda, mevcut tüm nesneler kümesini iki ayrı alt kümeye bölebiliriz1:

  • Ulaşılabilir nesneler JLS tarafından şu şekilde tanımlanır:

    A reachable object is any object that can be accessed in any potential continuing computation from any live thread.

    In practice, this means that there is a chain of references starting from an in-scope local variable or a static variable by which some code might be able to reach the object.

  • Ulaşılamayan nesneler, yukarıdaki gibi erişilemeyen nesnelerdir.

Erişilemeyen nesneler çöp toplama için uygundur. Bu, çöplerin toplanacağı olacakları anlamına gelmez. Aslında:

  • Ulaşılamayan bir nesne, ulaşılamaz hale geldiğinde hemen toplanmaz1.
  • Erişilemeyen bir nesne hiçbir zaman çöp toplanmayabilir.

Java dili Belirtimi, erişilemeyen nesnelerin ne zaman toplanacağına karar vermek için bir JVM uygulamasına çok fazla serbestlik verir. Ayrıca (pratikte), bir JVM uygulamasının erişilemeyen nesneleri nasıl algıladığı konusunda muhafazakar olmasına izin verir.

JLS’nin garanti ettiği tek şey, erişilebilir hiçbir nesnenin asla çöp toplanmayacağıdır.

Bir nesne ulaşılamaz hale geldiğinde ne olur? #

Her şeyden önce, bir nesne ulaşılamaz hale geldiğinde* özel olarak hiçbir şey olmaz. Olaylar yalnızca çöp toplayıcı çalıştığında ve nesneye ulaşılamadığını algıladığında olur. Ayrıca, bir GC çalışmasının erişilemeyen tüm nesneleri algılamaması yaygın bir durumdur.

GC ulaşılamaz bir nesne algıladığında aşağıdaki olaylar meydana gelebilir.

  1. Nesneye gönderme yapan herhangi bir ‘Referans’ nesnesi varsa, nesne silinmeden önce bu referanslar temizlenecektir.

  2. Eğer nesne sonlandırılabilir ise, o zaman sonlandırılacaktır. Bu, nesne silinmeden önce olur.

  3. Nesne silinebilir ve kapladığı bellek geri alınabilir.

Yukarıdaki olayların oluşabileceği açık bir sıra olduğuna dikkat edin, ancak hiçbir şey çöp toplayıcının herhangi bir belirli nesnenin son silmesini herhangi bir belirli zaman diliminde gerçekleştirmesini gerektirmez.

Ulaşılabilir ve ulaşılamaz nesnelere örnekler #

Aşağıdaki örnek sınıfları göz önünde bulundurun:

// A node in simple "open" linked-list.
public class Node {
    private static int counter = 0;

    public int nodeNumber = ++counter;
    public Node next;
}

public class ListTest {
    public static void main(String[] args) {
        test();                    // M1
        System.out.prinln("Done"); // M2
    }
    
    private static void test() {
        Node n1 = new Node();      // T1
        Node n2 = new Node();      // T2
        Node n3 = new Node();      // T3
        n1.next = n2;              // T4
        n2 = null;                 // T5
        n3 = null;                 // T6
    }
}

test() çağrıldığında ne olduğunu inceleyelim. T1, T2 ve T3 ifadeleri “Düğüm” nesneleri oluşturur ve nesnelerin tümüne sırasıyla “n1”, “n2” ve “n3” değişkenleri aracılığıyla erişilebilir. T4 ifadesi, 2. “Düğüm” nesnesine başvuruyu birincinin “sonraki” alanına atar. Bu yapıldığında, 2. “Düğüme” iki yoldan erişilebilir:

 n2 -> Node2
 n1 -> Node1, Node1.next -> Node2

T5 deyiminde, n2’ye ’null’ atadık. Bu, “Düğüm2” için erişilebilirlik zincirlerinden ilkini kırar, ancak ikincisi kesintisiz kalır, bu nedenle “Düğüm2"ye hala erişilebilir.

T6 deyiminde, n3’e ’null’ atadık. Bu, “Düğüm3"ü erişilemez hale getiren “Düğüm3"ün tek erişilebilirlik zincirini kırar. Ancak, “Düğüm1” ve “Düğüm2"nin her ikisine de “n1” değişkeni aracılığıyla erişilebilir.

Son olarak, “test()” yöntemi döndüğünde, “n1”, “n2” ve “n3” yerel değişkenleri kapsam dışına çıkar ve bu nedenle hiçbir şey tarafından erişilemez. Bu, “Düğüm1” ve “Düğüm2” için kalan erişilebilirlik zincirlerini kırar ve tüm “Düğüm” nesneleri erişilemez ve çöp toplama için uygun değildir.

1 - Bu, sonuçlandırmayı ve “Referans” sınıflarını yok sayan bir sadeleştirmedir. 2 - Varsayımsal olarak, bir Java uygulaması bunu yapabilir, ancak bunu yapmanın performans maliyeti onu pratik yapmaz.

Java’da bellek sızıntısı #

[Çöp toplama][1] örneğinde, Java’nın bellek sızıntısı sorununu çözdüğünü ima etmiştik. Bu aslında doğru değil. Sızıntıların nedenleri oldukça farklı olsa da, bir Java programı bellek sızdırabilir.

Ulaşılabilir nesneler sızıntı yapabilir #

Aşağıdaki saf yığın uygulamasını düşünün.

public class NaiveStack {
    private Object[] stack = new Object[100];
    private int top = 0;

    public void push(Object obj) {
        if (top >= stack.length) {
            throw new StackException("stack overflow");
        }
        stack[top++] = obj;
    }

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        return stack[--top];
    }

    public boolean isEmpty() {
        return top == 0;
    }
}

Bir nesneyi ‘ittiğinizde’ ve ardından hemen ‘patladığınızda’, yine de ‘yığın’ dizisindeki nesneye bir referans olacaktır.

Yığın uygulamasının mantığı, bu referansın API’nin bir istemcisine döndürülemeyeceği anlamına gelir. Bir nesne açılırsa, o zaman * “herhangi bir canlı iş parçacığından herhangi bir potansiyel devam eden hesaplamada erişilemediğini”* kanıtlayabiliriz. Sorun şu ki, mevcut nesil bir JVM bunu kanıtlayamaz. Mevcut nesil JVM’ler, referanslara ulaşılabilir olup olmadığını belirlerken programın mantığını dikkate almaz. (Başlangıç ​​için pratik değildir.)

Ancak erişilebilirliğin gerçekte ne anlama geldiği konusunu bir kenara bırakırsak, burada ‘NaiveStack’ uygulamasının geri alınması gereken nesnelere “dayandığı” bir durumla karşı karşıyayız. Bu bir bellek sızıntısı.

Bu durumda, çözüm basittir:

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        Object popped = stack[--top];
        stack[top] = null;              // Overwrite popped reference with null.
        return popped;
    }

Önbellekler bellek sızıntıları olabilir #

Hizmet performansını iyileştirmeye yönelik yaygın bir strateji, sonuçları önbelleğe almaktır. Buradaki fikir, önbellek olarak bilinen bir bellek içi veri yapısında ortak isteklerin ve bunların sonuçlarının kaydını tutmanızdır. Ardından, her istek yapıldığında, isteği önbellekte ararsınız. Arama başarılı olursa, ilgili kayıtlı sonuçları döndürürsünüz.

Bu strateji, doğru bir şekilde uygulandığı takdirde çok etkili olabilir. Ancak, yanlış uygulanırsa önbellek bir bellek sızıntısı olabilir. Aşağıdaki örneği göz önünde bulundurun:

public class RequestHandler {
    private Map<Task, Result> cache = new HashMap<>();

    public Result doRequest(Task task) {
        Result result = cache.get(task);
        if (result == null) {
            result == doRequestProcessing(task);
            cache.put(task, result);
        }
        return result;
    }
}

Bu kodla ilgili sorun, herhangi bir “doRequest” çağrısının önbelleğe yeni bir giriş ekleyebilmesine rağmen, bunları kaldıracak hiçbir şey olmamasıdır. Hizmet sürekli olarak farklı görevler alıyorsa, önbellek sonunda tüm kullanılabilir belleği tüketecektir. Bu bir bellek sızıntısı şeklidir.

Bunu çözmeye yönelik bir yaklaşım, maksimum boyutta bir önbellek kullanmak ve önbellek maksimumu aştığında eski girişleri atmak. (En son kullanılan girdiyi atmak iyi bir stratejidir.) Diğer bir yaklaşım, yığın çok dolmaya başlarsa JVM’nin önbellek girdilerini çıkarabilmesi için ‘WeakHashMap’ kullanarak önbelleği oluşturmaktır.

Sonlandırma #

Bir Java nesnesi bir “finalize” yöntemi bildirebilir. Bu yöntem, Java nesne için belleği serbest bırakmadan hemen önce çağrılır. Genellikle şöyle görünecektir:

public class MyClass {
  
    //Methods for the class

    @Override
    protected void finalize() throws Throwable {
        // Cleanup code
    }
}

Ancak, Java’nın sonlandırılmasının davranışı hakkında bazı önemli uyarılar vardır.

  • Java, bir “finalize()” yönteminin ne zaman çağrılacağına dair hiçbir garanti vermez.
  • Java, çalışan uygulamanın ömrü boyunca bir “finalize()” yönteminin çağrılacağını bile garanti etmez.
  • Garanti edilen tek şey, nesne silinmeden önce yöntemin çağrılacağıdır … nesne silinirse.

Yukarıdaki uyarılar, zamanında gerçekleştirilmesi gereken temizleme (veya diğer) eylemleri gerçekleştirmek için “sonlandırma” yöntemine güvenmenin kötü bir fikir olduğu anlamına gelir. Sonlandırmaya aşırı güvenmek, depolama sızıntılarına, bellek sızıntılarına ve diğer sorunlara yol açabilir.

Kısacası, sonuçlandırmanın gerçekten iyi bir çözüm olduğu çok az durum vardır.

Sonlandırıcılar yalnızca bir kez çalışır #

Normalde, bir nesne sonlandırıldıktan sonra silinir. Ancak, bu her zaman olmaz. Aşağıdaki örneği inceleyin1:

public class CaptainJack {
    public static CaptainJack notDeadYet = null;

    protected void finalize() {
        // Resurrection!
        notDeadYet = this;
    }
}

Bir “CaptainJack” örneğine ulaşılamaz hale geldiğinde ve çöp toplayıcı onu geri almaya çalıştığında, “finalize()” yöntemi, örneğe “notDeadYet” değişkenine bir başvuru atar. Bu, örneği bir kez daha erişilebilir hale getirecek ve çöp toplayıcı onu silmez.

Soru: Kaptan Jack ölümsüz mü?

Cevap: Hayır.

Yakalama, JVM’nin ömrü boyunca bir nesne üzerinde yalnızca bir sonlandırıcı çalıştıracağıdır. Yeniden dirilen bir örneğe bir kez daha ulaşılamamasına neden olacak şekilde ’notDeadYet’ öğesine ’null’ atarsanız, çöp toplayıcı nesne üzerinde ‘finalize()’ öğesini çağırmaz.
