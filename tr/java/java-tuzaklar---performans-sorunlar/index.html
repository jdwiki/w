Bu konu, Java uygulama performansıyla ilgili bir dizi “tuzak"ı (yani, acemi Java programcılarının yaptığı hatalar) açıklamaktadır.

Bu konu, verimsiz olan bazı “mikro” Java kodlama uygulamalarını açıklar. Çoğu durumda, verimsizlikler nispeten küçüktür, ancak yine de onlardan kaçınmaya değer.

Pitfall - Bir döngüdeki dize bitiştirme ölçeklenmiyor #

Aşağıdaki kodu örnek olarak düşünün:

public String joinWords(List<String> words) {
    String message = "";
    for (String word : words) {
        message = message + " " + word;
    }
    return message;
}

Ne yazık ki, “kelimeler” listesi uzunsa bu kod verimsizdir. Sorunun kökü şu ifadedir:

message = message + " " + word;

Her döngü yinelemesi için, bu ifade, orijinal “mesaj” dizesindeki tüm karakterlerin kendisine eklenmiş ekstra karakterlerle bir kopyasını içeren yeni bir “mesaj” dizesi oluşturur. Bu, çok sayıda geçici dize oluşturur ve çok fazla kopyalama yapar.

joinWordsü analiz ettiğimizde, ortalama uzunluğu M olan N kelime olduğunu varsayarak, O(N) geçici karakter dizilerinin oluşturulduğunu ve O(M.N2) karakterlerinin dizine kopyalanacağını görüyoruz. işlem. N2 bileşeni özellikle rahatsız edici.

Bu tür bir sorun için önerilen yaklaşım1, dize birleştirme yerine aşağıdaki gibi bir “StringBuilder” kullanmaktır:

public String joinWords2(List<String> words) {
    StringBuilder message = new StringBuilder();
    for (String word : words) {
        message.append(" ").append(word);
    }
    return message.toString();
}

“joinWords2” analizinin, oluşturucunun karakterlerini tutan “StringBuilder” destek dizisini “büyütmenin” genel giderlerini hesaba katması gerekir. Ancak, oluşturulan yeni nesnelerin sayısının O(logN) olduğu ve kopyalanan karakter sayısının O(M.N) karakter olduğu ortaya çıktı. Sonuncusu, son toString() çağrısında kopyalanan karakterleri içerir.

(Başlamak için doğru kapasiteye sahip ‘StringBuilder’ oluşturarak bunu daha da ayarlamak mümkün olabilir. Ancak, genel karmaşıklık aynı kalır.)

Orijinal ‘joinWords’ yöntemine geri dönersek, kritik ifadenin tipik bir Java derleyicisi tarafından aşağıdaki gibi optimize edileceği ortaya çıkıyor:

  StringBuilder tmp = new StringBuilder();
  tmp.append(message).append(" ").append(word);
  message = tmp.toString();

Ancak, Java derleyicisi, “joinWords2” kodunda elle yaptığımız gibi, “StringBuilder"ı döngüden “kaldırmaz”.

  • “Bir döngüde Java’nın String ‘+’ operatörü yavaş mı?”

1 - Java 8 ve sonraki sürümlerde, bu sorunu çözmek için ‘Joiner’ sınıfı kullanılabilir. Ancak, bu örneğin aslında olması gereken bu değildir.

Tuzak - Bir koleksiyonun boş olup olmadığını test etmek için size() kullanmak verimsizdir. #

Java Koleksiyonları Çerçevesi, tüm “Koleksiyon” nesneleri için ilgili iki yöntem sağlar:

  • size() bir ‘Koleksiyon’daki giriş sayısını döndürür, ve
  • isEmpty() yöntemi true değerini döndürürse (ve yalnızca) Koleksiyon boş.

Her iki yöntem de koleksiyon boşluğunu test etmek için kullanılabilir. Örneğin:

Collection<String> strings = new ArrayList<>();
boolean isEmpty_wrong = strings.size() == 0; // Avoid this
boolean isEmpty = strings.isEmpty();         // Best

Bu yaklaşımlar aynı görünse de bazı koleksiyon uygulamaları boyutu saklamaz. Böyle bir koleksiyon için, size() uygulamasının her çağrıldığında boyutu hesaplaması gerekir. Örneğin:

  • Basit bir bağlantılı liste sınıfının (ancak java.util.LinkedList değil) öğeleri saymak için listeyi geçmesi gerekebilir.
  • ConcurrentHashMap sınıfının, haritanın tüm “segmentlerindeki” girdileri toplaması gerekir.
  • Bir koleksiyonun tembel bir uygulamasının, öğeleri saymak için tüm koleksiyonu bellekte gerçekleştirmesi gerekebilir.

Buna karşılık, bir “isEmpty()” yönteminin yalnızca koleksiyonda en az bir öğe olup olmadığını test etmesi gerekir. Bu, elemanların sayılmasını gerektirmez.

“size() == 0” her zaman “isEmpty()“den daha az verimli olmasa da, uygun şekilde uygulanan bir “isEmpty()“nin “size() == 0"dan daha az verimli olması düşünülemez. Bu nedenle isEmpty() tercih edilir.

Tuzak - İlkel sarmalayıcı örnekleri oluşturmak için ‘yeni’ kullanmak verimsiz #

Java dili, “Tamsayı”, “Boolean” vb. örnekler oluşturmak için “yeni” kullanmanıza izin verir, ancak bu genellikle kötü bir fikirdir. Otomatik kutulama (Java 5 ve üstü) veya “valueOf” yöntemini kullanmak daha iyidir.

 Integer i1 = new Integer(1);      // BAD
 Integer i2 = 2;                   // BEST (autoboxing)
 Integer i3 = Integer.valueOf(3);  // OK

Açıkça “new Integer(int)” kullanmanın kötü bir fikir olmasının nedeni, yeni bir nesne oluşturmasıdır (JIT derleyicisi tarafından optimize edilmediği sürece). Buna karşılık, otomatik kutulama veya açık bir “valueOf” çağrısı kullanıldığında, Java çalışma zamanı, önceden var olan nesnelerin önbelleğinden bir “Tamsayı” nesnesini yeniden kullanmayı deneyecektir. Çalışma zamanının bir önbellek “isabetine” sahip olduğu her zaman, bir nesne oluşturmaktan kaçınır. Bu aynı zamanda yığın belleğinden tasarruf sağlar ve nesne dalgalanmasının neden olduğu GC ek yüklerini azaltır.

Notlar:

  1. Son Java uygulamalarında, autoboxing ‘valueOf’ çağrılarak uygulanır ve ‘Boolean’, ‘Byte’, ‘Short’, ‘Integer’, ‘Long’ ve ‘Character’ için önbellekler vardır.
  2. İntegral türleri için önbelleğe alma davranışı, Java Dil Belirtimi tarafından zorunlu kılınmıştır.

Pitfall - == kullanabilmeniz için dizeleri dahil etmek kötü bir fikirdir #

Bazı programcılar bu tavsiyeyi gördüğünde:

“== kullanarak dizeleri test etmek yanlış (dizeler enterne edilmedikçe)”

ilk tepkileri, == kullanabilmeleri için dizgileri intern etmektir. (Sonuçta ==, String.equals(...) çağrısından daha hızlıdır, değil mi?)

Bu, birkaç açıdan yanlış bir yaklaşımdır:

kırılganlık #

Her şeyden önce, == yalnızca test ettiğiniz String nesnelerinin tümünün enterne edildiğini biliyorsanız güvenle kullanabilirsiniz. JLS, kaynak kodunuzdaki String değişmezlerinin enterne edilmiş olacağını garanti eder. Ancak, standart Java SE API’lerinin hiçbiri, “String.intern(String)” dışında, interned dizeleri döndürmeyi garanti etmez. Eğer enterne edilmemiş ‘String’ nesnelerinin yalnızca bir kaynağını kaçırırsanız, uygulamanız güvenilmez olacaktır. Bu güvenilmezlik, tespit edilmesini zorlaştırabilecek istisnalar yerine yanlış negatifler olarak kendini gösterecektir.

‘intern()’ kullanmanın maliyeti #

Kaputun altında, interning, daha önce interne edilmiş ‘String’ nesnelerini içeren bir hash tablosunu koruyarak çalışır. Bir çeşit zayıf referans mekanizması kullanılır, böylece interning hash tablosu bir depolama sızıntısı olmaz. Hash tablosu yerel kodda uygulanmış olsa da (“HashMap”, “HashTable” ve benzerlerinden farklı olarak), “intern” çağrıları kullanılan CPU ve bellek açısından hala nispeten maliyetlidir.

Bu maliyet, ’eşittir’ yerine ‘==’ kullanarak elde edeceğimiz tasarrufla karşılaştırılmalıdır. Aslında, interne edilen her dize diğer dizelerle “birkaç” kez karşılaştırılmadıkça bile kırılmayacağız.

(Bir kenara: interning’in faydalı olduğu birkaç durum, aynı dizelerin birçok kez yinelendiği bir uygulamanın bellek ayak izini azaltmakla ilgili olma eğilimindedir, ve bu dizelerin uzun bir kullanım ömrü vardır.)

Çöp toplama üzerindeki etkisi #

Yukarıda açıklanan doğrudan CPU ve bellek maliyetlerine ek olarak, dahili String’ler çöp toplayıcı performansını etkiler.

Java 7’den önceki Java sürümleri için, nadiren toplanan “PermGen” alanında iç dizeler tutulur. PermGen’in toplanması gerekiyorsa, bu (genellikle) tam bir çöp toplama işlemini tetikler. PermGen alanı tamamen dolarsa, normal yığın alanlarında boş alan olsa bile JVM çöker.

Java 7’de, dize havuzu “PermGen"den normal yığına taşındı. Ancak, hash tablosu hala uzun ömürlü bir veri yapısı olacak ve bu da herhangi bir interned string’in uzun ömürlü olmasına neden olacak. (İçerdeki dize nesneleri Eden alanına tahsis edilmiş olsa bile, büyük olasılıkla toplanmadan önce terfi ettirilirler.)

Bu nedenle, her durumda, bir dizeyi enterne etmek, sıradan bir dizeye göre ömrünü uzatacaktır. Bu, JVM’nin ömrü boyunca çöp toplama giderlerini artıracaktır.

İkinci sorun, karma tablonun, dize interning sızan belleği önlemek için bir tür zayıf referans mekanizması kullanması gerektiğidir. Ancak böyle bir mekanizma çöp toplayıcı için daha fazla iş.

Bu çöp toplama giderlerini ölçmek zordur, ancak var olduklarına dair çok az şüphe vardır. Eğer “intern” kelimesini yoğun bir şekilde kullanırsanız, bunlar önemli olabilir.

Dize havuzu karma tablo boyutu #

Bu kaynağa göre, Java 6’dan itibaren, dize havuzu, aynı kovaya hash olan dizelerle başa çıkmak için zincirlerle sabit boyutlu karma tablo olarak uygulanır. Java 6’nın ilk sürümlerinde, karma tablosunun (kablolu) sabit bir boyutu vardı. Java 6’ya bir orta yaş güncellemesi olarak bir ayar parametresi (-XX:StringTableSize) eklendi. Ardından Java 7’ye yapılan bir orta ömür güncellemesinde, havuzun varsayılan boyutu “1009” iken “60013” olarak değiştirildi. .

Sonuç olarak, kodunuzda yoğun olarak ‘intern’ kullanmayı düşünüyorsanız, karma tablo boyutunun ayarlanabildiği bir Java sürümü seçmeniz ve boyutu uygun şekilde ayarladığınızdan emin olmanız tavsiye edilir. Aksi takdirde, havuz büyüdükçe ‘intern’ performansı düşebilir.

Potansiyel bir hizmet reddi vektörü olarak staj #

Dizeler için karma kod algoritması iyi bilinmektedir. Kötü niyetli kullanıcılar veya uygulamalar tarafından sağlanan dizeleri stajyerseniz, bu hizmet reddi (DoS) saldırısının bir parçası olarak kullanılabilir. Kötü niyetli aracı, sağladığı tüm dizelerin aynı karma koduna sahip olduğunu düzenlerse, bu dengesiz bir karma tabloya ve “intern” için “O(N)” performansına yol açabilir… burada “N” çarpışanların sayısıdır. Teller.

(Bir hizmete karşı DoS saldırısı başlatmanın daha basit/etkili yolları vardır. Ancak bu vektör, DoS saldırısının amacı güvenliği kırmak veya birinci basamak DoS savunmalarından kaçmaksa kullanılabilir.)

Pitfall - Düzenli ifadelerle ilgili verimlilik endişeleri #

Düzenli ifade eşleştirme (Java’da ve diğer bağlamlarda) güçlü bir araçtır, ancak bazı dezavantajları vardır. Bunlardan biri, normal ifadeler oldukça pahalı olma eğilimindedir.

Kalıp ve Eşleştirici örnekleri yeniden kullanılmalıdır #

Aşağıdaki örneği göz önünde bulundurun:

/**
 * Test if all strings in a list consist of English letters and numbers.
 * @param strings the list to be checked
 * @return 'true' if an only if all strings satisfy the criteria
 * @throws NullPointerException if 'strings' is 'null' or a 'null' element.
 */
public boolean allAlphanumeric(List<String> strings) {
    for (String s : strings) {
        if (!s.matches("[A-Za-z0-9]*")) {
            return false;
        }  
    }
    return true;
}

Bu kod doğrudur, ancak verimsizdir. Sorun ’eşleşmeler(…)’ çağrısında. Başlık altında, s.matches("[A-Za-z0-9]*") şuna eşdeğerdir:

Pattern.matches(s, "[A-Za-z0-9]*")

hangi sırayla eşdeğerdir

Pattern.compile("[A-Za-z0-9]*").matcher(s).matches()

Pattern.compile("[A-Za-z0-9]*") çağrısı, normal ifadeyi ayrıştırır, analiz eder ve normal ifade motoru tarafından kullanılacak veri yapısını tutan bir “Pattern” nesnesi oluşturur. Bu önemsiz olmayan bir hesaplamadır. Ardından, ’s’ argümanını sarmak için bir ‘Eşleştirici’ nesnesi oluşturulur. Son olarak, gerçek desen eşleştirmesini yapmak için ‘match()’ diyoruz.

Sorun, bu çalışmanın her döngü yinelemesi için tekrarlanmasıdır. Çözüm, kodu aşağıdaki gibi yeniden yapılandırmaktır:

private static Pattern ALPHA_NUMERIC = Pattern.compile("[A-Za-z0-9]*");

public boolean allAlphanumeric(List<String> strings) {
    Matcher matcher = ALPHA_NUMERIC.matcher("");
    for (String s : strings) {
        matcher.reset(s);
        if (!matcher.matches()) {
            return false;
        }  
    }
    return true;
}

“Pattern” için javadoc şunu belirtir:

Bu sınıfın örnekleri değişmezdir ve birden çok eşzamanlı iş parçacığı tarafından kullanım için güvenlidir. “Matcher” sınıfının örnekleri bu tür kullanım için güvenli değildir.

find() kullanmanız gerektiğinde match() kullanmayın #

Bir s dizesinin arka arkaya üç veya daha fazla rakam içerip içermediğini test etmek istediğinizi varsayalım. Bunu aşağıdakiler de dahil olmak üzere çeşitli şekillerde ifade edebilirsiniz:

  if (s.matches(".*[0-9]{3}.*")) {
      System.out.println("matches");
  }

veya

  if (Pattern.compile("[0-9]{3}").matcher(s).find()) {
      System.out.println("matches");
  }

İlki daha kısadır, ancak daha az verimli olması da muhtemeldir. Görünüşte, ilk sürüm tüm dizeyi desenle eşleştirmeye çalışacak. Ayrıca, “.*” “açgözlü” bir kalıp olduğundan, kalıp eşleştirici muhtemelen dizinin sonuna “hevesle” ilerlemeye çalışır ve bir eşleşme bulana kadar geri döner.

Buna karşılık, ikinci versiyon soldan sağa doğru arama yapacak ve arka arkaya 3 rakamı bulduğu anda aramayı bırakacaktır.

Normal ifadeler için daha verimli alternatifler kullanın

Normal ifadeler güçlü bir araçtır, ancak tek aracınız olmamalıdır. Birçok görev başka yollarla daha verimli bir şekilde yapılabilir. Örneğin:

 Pattern.compile("ABC").matcher(s).find()

aynı şeyi yapar:

 s.contains("ABC")

bunun dışında ikincisi çok daha verimlidir. (Düzenli ifadeyi derlemenin maliyetini amorti edebilseniz bile.)

Genellikle, normal ifade olmayan form daha karmaşıktır. Örneğin, önceki ‘allAlplanumeric’ yöntemini çağıran ‘matches()’ tarafından gerçekleştirilen test şu şekilde yeniden yazılabilir:

 public boolean matches(String s) {
     for (char c : s) {
         if ((c >= 'A' && c <= 'Z') ||
             (c >= 'a' && c <= 'z') ||
             (c >= '0' && c <= '9')) {
              return false;
         }
     }
     return true;
 }

Şimdi bu, bir ‘Eşleştirici’ kullanmaktan daha fazla koddur, ancak aynı zamanda önemli ölçüde daha hızlı olacaktır.

Felaket Geri İzleme #

(Bu, normal ifadelerin tüm uygulamalarında potansiyel olarak bir sorundur, ancak ‘Desen’ kullanımı için bir tuzak olduğu için burada bahsedeceğiz.)

Bu (uydurulmuş) örneği düşünün:

Pattern pat = Pattern.compile("(A+)+B");
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB").matches());
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAC").matches());

İlk “println” çağrısı hızlı bir şekilde “true” yazdıracaktır. İkincisi ‘yanlış’ yazdıracaktır. Sonunda. Gerçekten de, yukarıdaki kodu denediğinizde, “C"den önce bir “A” eklediğinizde, sürenin iki katına çıkacağını göreceksiniz.

Bu davranış, felaket bir geri izleme örneğidir. Normal ifade eşleştirmesini uygulayan kalıp eşleştirme motoru, kalıbın * eşleşebileceği tüm olası yolları sonuçsuz bir şekilde deniyor.

(A+)+Bnin gerçekte ne anlama geldiğine bakalım. Yüzeysel olarak, “bir veya daha fazla ‘A’ karakteri ve ardından bir ‘B’ değeri” deniyor gibi görünüyor, ancak gerçekte her biri bir veya daha fazla ‘A’ karakterinden oluşan bir veya daha fazla grup diyor. Yani mesela:

  • ‘AB’ yalnızca tek yönlü eşleşir: ‘(A)B’
  • ‘AAB’ iki şekilde eşleşir: ‘(AA)B’ veya ‘(A)(A)B’
  • ‘AAAB’ dört yolla eşleşir: ‘(AAA)B’ veya ‘(AA)(A)B’ veya ‘(A)(AA)B’ veya ‘(A)(A)(A)B’
  • ve benzeri

Başka bir deyişle, olası eşleşme sayısı 2N‘tur; burada N, “A” karakterlerinin sayısıdır.

Yukarıdaki örnek açıkça uydurmadır, ancak bu tür performans özellikleri sergileyen kalıplar (yani, büyük bir “K” için “O(2^N)” veya “O(N^K)”), iyi düşünülmemiş düzenli ifadeler olduğunda sıklıkla ortaya çıkar. kullanılmış. Bazı standart çözümler vardır:

  • Yinelenen kalıpları diğer yinelenen kalıpların içine yerleştirmekten kaçının.
  • Çok fazla yinelenen desen kullanmaktan kaçının.
  • Uygun olduğu şekilde geri izleme yapmayan tekrarı kullanın.
  • Karmaşık ayrıştırma görevleri için normal ifadeler kullanmayın. (Bunun yerine uygun bir ayrıştırıcı yazın.)

Son olarak, bir kullanıcının veya bir API istemcisinin patolojik özelliklere sahip bir normal ifade dizesi sağlayabileceği durumlara dikkat edin. Bu, yanlışlıkla veya kasıtlı olarak “hizmet reddine” yol açabilir.

Tuzak - Arabelleğe alınmamış akışlarda küçük okumalar/yazmalar verimsizdir #

Bir dosyayı diğerine kopyalamak için aşağıdaki kodu göz önünde bulundurun:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new FileInputStream(args[0]);
             OutputStream os = new FileOutputStream(args[1])) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

(Bu örneğin noktasıyla alakalı olmadıkları için, normal argüman kontrolü, hata raporlaması ve benzerlerini ihmal ettik.)

Yukarıdaki kodu derler ve büyük bir dosyayı kopyalamak için kullanırsanız, çok yavaş olduğunu fark edeceksiniz. Aslında, standart işletim sistemi dosya kopyalama yardımcı programlarından en az birkaç büyüklük sırası daha yavaş olacaktır.

(Gerçek performans ölçümlerini buraya ekleyin!)

Yukarıdaki örneğin yavaş olmasının birincil nedeni (büyük dosya durumunda), arabelleğe alınmamış bayt akışlarında bir baytlık okuma ve bir baytlık yazma gerçekleştirmesidir. Performansı artırmanın basit yolu, akışları arabelleğe alınmış akışlarla sarmaktır. Örneğin:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new BufferedInputStream(
                     new FileInputStream(args[0]));
             OutputStream os = new BufferedOutputStream(
                     new FileOutputStream(args[1]))) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

Bu küçük değişiklikler, platformla ilgili çeşitli faktörlere bağlı olarak veri kopyalama hızını en az birkaç büyüklük mertebesi artıracaktır. Arabelleğe alınmış akış sarmalayıcıları, verilerin daha büyük parçalar halinde okunmasına ve yazılmasına neden olur. Örneklerin her ikisi de bayt dizileri olarak uygulanan arabelleklere sahiptir.

  • is ile, veri dosyadan bir defada birkaç kilobayt arabelleğe okunur. ‘read()’ çağrıldığında, uygulama genellikle arabellekten bir bayt döndürür. Yalnızca arabellek boşaltılmışsa, temel alınan girdi akışından okuyacaktır.

  • ‘os’ davranışı benzerdir. ‘os.write(int)’ çağrıları arabelleğe tek bayt yazar. Veri, yalnızca arabellek dolduğunda veya “os” boşaltıldığında veya kapatıldığında çıktı akışına yazılır.

Karakter tabanlı akışlar ne olacak? #

Bildiğiniz gibi, Java I/O, ikili ve metin verilerini okumak ve yazmak için farklı API’ler sağlar.

  • “InputStream” ve “OutputStream”, akış tabanlı ikili G/Ç için temel API’lerdir
  • “Okuyucu” ve “Yazıcı”, akış tabanlı metin G/Ç için temel API’lerdir.

Metin G/Ç için, “BufferedReader” ve “BufferedWriter”, “BufferedInputStream” ve “BufferedOutputStream” için eşdeğerdir.

Arabelleğe alınan akışlar neden bu kadar fark yaratıyor? #

Arabelleğe alınan akışların performansa yardımcı olmasının gerçek nedeni, bir uygulamanın işletim sistemiyle konuşma biçimiyle ilgilidir:

  • Bir Java uygulamasındaki Java yöntemi veya JVM’nin yerel çalışma zamanı kitaplıklarındaki yerel prosedür çağrıları hızlıdır. Genellikle birkaç makine talimatı alırlar ve minimum performans etkisine sahiptirler.

  • Buna karşılık, işletim sistemine yapılan JVM çalışma zamanı çağrıları hızlı değildir. “Sistem çağrısı” olarak bilinen bir şeyi içerirler. Bir sistem çağrısı için tipik model aşağıdaki gibidir:

    1. Put the syscall arguments into registers.
    2. Execute a SYSENTER trap instruction.
    3. The trap handler switched to privileged state and changes the virtual memory mappings. Then it dispatches to the code to handle the specific syscall.
    4. The syscall handler checks the arguments, taking care that it isn’t being told to access memory that the user process should not see.
    5. The syscall specific work is performed. In the case of a read syscall, this may involve:
      1. checking that there is data to be read at the file descriptor’s current position
      2. calling the file system handler to fetch the required data from disk (or wherever it is stored) into the buffer cache,
      3. copying data from the buffer cache to the JVM-supplied address
      4. adjusting thstream pointerse file descriptor position
    6. Return from the syscall. This entails changing VM mappings again and switching out of privileged state.

Tahmin edebileceğiniz gibi, tek bir sistem çağrısı yapmak binlerce makine talimatı verebilir. Muhafazakar olarak, normal bir yöntem çağrısından en az iki büyüklük sırası daha uzun. (Muhtemelen üç veya daha fazla.)

Bu göz önüne alındığında, arabelleğe alınan akışların büyük bir fark yaratmasının nedeni, sistem çağrılarının sayısını büyük ölçüde azaltmalarıdır. Her ‘read()’ çağrısı için bir sistem çağrısı yapmak yerine, arabelleğe alınan girdi akışı, gerektiği gibi büyük miktarda veriyi arabelleğe okur. Arabelleğe alınmış akıştaki çoğu “read()” çağrısı, bazı basit sınır denetimi yapar ve daha önce okunan bir “bayt” döndürür. Benzer akıl yürütme, çıktı akışı durumunda ve ayrıca karakter akışı durumlarında da geçerlidir.

(Bazı insanlar, arabelleğe alınan G/Ç performansının, okuma isteği boyutu ile disk bloğu boyutu, disk dönme gecikmesi ve benzeri şeyler arasındaki uyumsuzluktan kaynaklandığını düşünür. Aslında, modern bir işletim sistemi, Uygulamanın tipik olarak diski beklemesine gerek yoktur. Gerçek açıklama bu değildir.)

Arabelleğe alınan akışlar her zaman bir kazanç mıdır? #

Her zaman değil. Uygulamanız çok sayıda “küçük” okuma veya yazma yapacaksa, arabelleğe alınmış akışlar kesinlikle bir kazançtır. Ancak, uygulamanızın yalnızca büyük bir “bayt[]” veya “karakter[]“den büyük okumalar veya yazma işlemleri yapması gerekiyorsa, arabelleğe alınmış akışlar size gerçek bir fayda sağlamayacaktır. Hatta (küçük) bir performans cezası bile olabilir.

Java’da bir dosyayı kopyalamanın en hızlı yolu bu mu? #

Hayır değil. Bir dosyayı kopyalamak için Java’nın akış tabanlı API’lerini kullandığınızda, verilerin en az bir fazladan bellekten belleğe kopyalanması maliyetine maruz kalırsınız. NIO ByteBuffer ve Channel API’lerini kullanıyorsanız bunu önlemek mümkündür. (Buraya ayrı bir örneğe bağlantı ekleyin.)

Tuzak - Günlük mesajları oluşturmanın genel giderleri #

‘TRACE’ ve ‘DEBUG’ günlük seviyeleri, verilen kodun çalışma zamanında çalışması hakkında yüksek ayrıntıyı iletebilmek için vardır. Günlük düzeyinin bunların üzerinde ayarlanması genellikle önerilir, ancak bu ifadelerin “kapalı” göründüğünde bile performansı etkilememesi için biraz özen gösterilmelidir.

Bu günlük ifadesini göz önünde bulundurun:

// Processing a request of some kind, logging the parameters
LOG.debug("Request coming from " + myInetAddress.toString() 
          + " parameters: " + Arrays.toString(veryLongParamArray));

Günlük düzeyi ‘INFO’ olarak ayarlansa bile, ‘debug()’ işlevine iletilen argümanlar satırın her yürütülmesinde değerlendirilecektir. Bu, birkaç açıdan gereksiz yere tüketmesini sağlar:

  • “Dize” birleştirme: birden çok “Dize” örneği oluşturulacak
  • InetAddress bir DNS araması bile yapabilir.
  • ‘veryLongParamArray’ çok uzun olabilir - bunun dışında bir String oluşturmak hafıza tüketir, zaman alır

Çözüm #

Çoğu günlük kaydı çerçevesi, düzeltme dizeleri ve nesne referansları kullanarak günlük mesajları oluşturma araçları sağlar. Günlük iletisi, yalnızca ileti gerçekten günlüğe kaydedilirse değerlendirilir. Örnek:

// No toString() evaluation, no string concatenation if debug is disabled
LOG.debug("Request coming from {} parameters: {}", myInetAddress, parameters));

Bu, tüm parametreler String.valueOf(Object) kullanılarak dizelere dönüştürülebildiği sürece çok iyi çalışır. Günlük mesajı hesaplaması daha karmaşıksa, günlüğe kaydetmeden önce günlük düzeyi kontrol edilebilir:

if (LOG.isDebugEnabled()) {
    // Argument expression evaluated only when DEBUG is enabled
    LOG.debug("Request coming from {}, parameters: {}", myInetAddress,
              Arrays.toString(veryLongParamArray);
}

Burada, maliyetli “Arrays.toString(Obect[])” hesaplamasıyla “LOG.debug()” yalnızca “DEBUG” gerçekten etkinleştirildiğinde işlenir.

Pitfall - ’new String(String)’ çağrısı verimsiz #

Bir dizgiyi çoğaltmak için new String(String) kullanmak verimsizdir ve neredeyse her zaman gereksizdir.

  • Dize nesneleri değişmezdir, bu nedenle değişikliklere karşı korunmak için onları kopyalamaya gerek yoktur.
  • Java’nın bazı eski sürümlerinde, ‘String’ nesneleri, diğer ‘String’ nesneleriyle destek dizilerini paylaşabilir. Bu sürümlerde, (büyük) bir dizenin (küçük) bir alt dizesini oluşturarak ve onu koruyarak bellek sızdırmak mümkündür. Ancak, Java 7’den itibaren, ‘String’ destek dizileri paylaşılmaz.

Herhangi bir somut yararın yokluğunda, “new String(String)“i çağırmak sadece israftır:

  • Kopyanın yapılması CPU zamanını alır.
  • Kopya, uygulamanın bellek kapladığı alanı artıran ve/veya GC genel giderlerini artıran daha fazla bellek kullanır.
  • ’equals(Object)’ ve ‘hashCode()’ gibi işlemler, String nesneleri kopyalanırsa daha yavaş olabilir.

Pitfall - System.gc() işlevinin çağrılması verimsiz #

System.gc() çağırmak (neredeyse her zaman) kötü bir fikirdir.

“gc()” yönteminin javadoc’u aşağıdakileri belirtir:

“‘gc’ yöntemini çağırmak, Java Sanal Makinesinin, halihazırda işgal ettikleri belleği hızlı yeniden kullanım için kullanılabilir hale getirmek için kullanılmayan nesneleri geri dönüştürmek için çaba sarf ettiğini gösterir. Kontrol, yöntem çağrısından döndüğünde, Java Sanal Makinesi en iyisini yapmıştır. atılan tüm nesnelerden alanı geri kazanma çabası.”

Bundan çıkarılabilecek birkaç önemli nokta var:

  1. “Anlatıyor” yerine “öneriyor” kelimesinin kullanılması, JVM’nin öneriyi görmezden gelmekte özgür olduğu anlamına gelir. Varsayılan JVM davranışı (son sürümler) öneriyi takip etmektir, ancak bu, JVM başlatılırken “-XX:+DisableExplicitGC” ayarlanarak geçersiz kılınabilir.

  2. “Atılan tüm nesnelerden alan kazanmak için en iyi çaba” ifadesi, “gc"yi çağırmanın “dolu” bir çöp toplama işlemini tetikleyeceğini ima eder.

Peki, System.gc()‘yi çağırmak neden kötü bir fikir?

İlk olarak, tam bir çöp toplama işlemi yürütmek pahalıdır. Tam bir GC, hala erişilebilir olan her nesneyi ziyaret etmeyi ve “işaretlemeyi” içerir; yani çöp olmayan her nesne. Toplanacak çok fazla çöp olmadığında bunu tetiklerseniz, GC nispeten az fayda için çok iş yapar.

İkincisi, tam bir çöp toplama, toplanmayan nesnelerin “yerellik” özelliklerini bozmakla yükümlüdür. Aynı iş parçacığı tarafından kabaca aynı zamanda ayrılan nesneler, bellekte birbirine yakın olarak ayrılma eğilimindedir. Bu iyi. Aynı anda tahsis edilen nesnelerin birbiriyle ilişkili olması muhtemeldir; yani birbirlerine referans. Uygulamanız bu referansları kullanıyorsa, çeşitli bellek ve sayfa önbelleğe alma etkileri nedeniyle bellek erişiminin daha hızlı olması ihtimali vardır. Ne yazık ki, tam bir çöp toplama, bir zamanlar yakın olan nesneler şimdi daha uzakta olacak şekilde nesneleri hareket ettirme eğilimindedir.

Üçüncüsü, tam bir çöp toplama çalıştırmak, toplama tamamlanana kadar uygulamanızı duraklatmakla yükümlüdür. Bu olurken, uygulamanız yanıt vermeyecektir.

Aslında en iyi strateji, GC’nin ne zaman çalıştırılacağına ve ne tür bir koleksiyonun çalıştırılacağına JVM’nin karar vermesine izin vermektir. Müdahale etmezseniz JVM, verimi optimize eden veya GC duraklama sürelerini en aza indiren bir zaman ve koleksiyon türü seçecektir.

Başlangıçta “… (neredeyse her zaman) kötü bir fikir…” dedik. Aslında bunun * iyi bir fikir olabileceği * birkaç senaryo vardır:

  1. Çöp toplamaya duyarlı bazı kodlar için bir birim testi uyguluyorsanız (örneğin, sonlandırıcıları veya zayıf / yumuşak / hayali referansları içeren bir şey), o zaman System.gc() çağırmak gerekli olabilir.

  2. Bazı etkileşimli uygulamalarda, kullanıcının çöp toplama duraklaması olup olmadığını umursamayacağı belirli noktalar olabilir. Bir örnek, “oyun"da doğal duraklamaların olduğu bir oyundur; Örneğin. yeni bir seviye yüklerken.

Tuzak - İlkel sarmalayıcı türlerinin aşırı kullanımı verimsizdir #

Bu iki kod parçasını göz önünde bulundurun:

int a = 1000;
int b = a + 1;

ve

Integer a = 1000;
Integer b = a + 1;

Soru: Hangi sürüm daha verimli?

Cevap: İki versiyon neredeyse aynı görünüyor, ancak ilk versiyon ikincisinden çok daha verimli.

İkinci sürüm, daha fazla alan kullanan sayılar için bir temsil kullanıyor ve sahne arkasında otomatik kutulamaya ve otomatik kutudan çıkarmaya güveniyor. Aslında ikinci sürüm doğrudan aşağıdaki koda eşdeğerdir:

Integer a = Integer.valueOf(1000);               // box 1000
Integer b = Integer.valueOf(a.intValue() + 1);   // unbox 1000, add 1, box 1001

Bunu ‘int’ kullanan diğer sürümle karşılaştırıldığında, ‘Integer’ kullanıldığında açıkça üç ekstra yöntem çağrısı vardır. “valueOf” durumunda, çağrıların her biri yeni bir “Integer” nesnesi oluşturacak ve başlatacaktır. Tüm bu ekstra kutulama ve kutudan çıkarma çalışmalarının ikinci versiyonu birincisinden daha yavaş hale getirmesi muhtemel.

Buna ek olarak, ikinci sürüm, her bir ‘valueOf’ çağrısında öbek üzerindeki nesneleri tahsis ediyor. Alan kullanımı platforma özel olsa da, her ‘Tamsayı’ nesnesi için 16 baytlık bölgede olması muhtemeldir. Buna karşılık, “a” ve “b"nin yerel değişkenler olduğunu varsayarsak, “int” sürümü sıfır ekstra yığın alanına ihtiyaç duyar.

İlkellerin kutulu eşdeğerlerinden daha hızlı olmasının bir başka büyük nedeni de ilgili dizi türlerinin bellekte nasıl düzenlendiğidir.

Örnek olarak ‘int[]’ ve ‘Tamsayı[]’ alırsanız, ‘int[]’ durumunda ‘int’ değerleri bitişik olarak bellekte düzenlenir. Ancak bir “Tamsayı[]” durumunda, ortaya konan değerler değil, gerçek “int” değerlerini içeren “Tamsayı” nesnelerine yapılan referanslardır (işaretçiler).

Fazladan bir dolaylı seviye olmasının yanı sıra, bu değerler üzerinde yineleme yapıldığında önbellek konumu söz konusu olduğunda büyük bir tank olabilir. Bir “int[]” olması durumunda, CPU dizideki tüm değerleri bir kerede önbelleğine getirebilir, çünkü bunlar bellekte bitişiktir. Ancak bir “Tamsayı[]” durumunda, dizi yalnızca gerçek değerlere referanslar içerdiğinden, CPU’nun potansiyel olarak her öğe için ek bir bellek getirmesi yapması gerekir.

Kısacası, ilkel sarmalayıcı türlerini kullanmak hem CPU hem de bellek kaynakları açısından nispeten pahalıdır. Bunları gereksiz yere kullanmak verimlidir.

Pitfall - Bir Haritanın anahtarlarını yinelemek verimsiz olabilir #

Aşağıdaki örnek kod, olması gerekenden daha yavaştır:

Map<String, String> map = new HashMap<>(); 
for (String key : map.keySet()) {
    String value = map.get(key);
    // Do something with key and value
}

Bunun nedeni, haritadaki her anahtar için bir harita araması (get() yöntemi) gerektirmesidir. Bu arama verimli olmayabilir (bir HashMap’te, anahtar üzerinde “hashCode” çağrılmasını, ardından dahili veri yapılarında doğru kovanın aranmasını ve hatta bazen “eşittir” çağrısını gerektirir). Büyük bir haritada, bu önemsiz bir ek yük olmayabilir.
