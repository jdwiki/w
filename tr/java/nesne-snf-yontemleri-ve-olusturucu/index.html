  • genel son geçersiz bekleme (uzun zaman aşımı, int nanos) InterruptedException’ı atar
  • genel yerel int hashCode()
  • public boole eşittir(Object obj)
  • genel String toString()
  • korumalı yerel Object klon(), CloneNotSupportedException’ı atar
  • korumalı void finalize() Fırlatılabilir fırlatır

hashCode() yöntemi #

Bir Java sınıfı “equals” yöntemini geçersiz kıldığında, “hashCode” yöntemini de geçersiz kılmalıdır. Yöntem sözleşmesinde tanımlandığı gibi:

  • Bir Java uygulamasının yürütülmesi sırasında aynı nesne üzerinde bir kereden fazla çağrıldığında, “hashCode” yöntemi, nesne üzerindeki eşittir karşılaştırmalarında kullanılan hiçbir bilginin değiştirilmemesi koşuluyla, tutarlı bir şekilde aynı tamsayıyı döndürmelidir. Bu tamsayının, bir uygulamanın bir yürütmesinden aynı uygulamanın başka bir yürütmesine kadar tutarlı kalması gerekmez.
  • “equals(Object)” yöntemine göre iki nesne eşitse, o zaman iki nesnenin her birinde “hashCode” yöntemini çağırmak aynı tamsayı sonucunu üretmelidir.
  • “equals(Object)” yöntemine göre iki nesne eşit değilse, o zaman iki nesnenin her biri üzerinde “hashCode” yöntemini çağırmanın farklı tamsayı sonuçları üretmesi gerekmez. Ancak programcı, eşit olmayan nesneler için farklı tamsayı sonuçları üretmenin karma tabloların performansını iyileştirebileceğini bilmelidir.

Hash kodları, “HashMap”, “HashTable” ve “HashSet” gibi karma uygulamalarda kullanılır. ‘hashCode’ işlevinin sonucu, bir nesnenin yerleştirileceği kovayı belirler. Sağlanan “hashCode” uygulaması iyiyse, bu karma uygulamalar daha verimlidir. İyi bir “hashCode” uygulamasının önemli bir özelliği, “hashCode” değerlerinin dağılımının tek tip olmasıdır. Başka bir deyişle, aynı kovada çok sayıda örneğin depolanacağı küçük bir olasılıktır.

Bir karma kod değerini hesaplamak için bir algoritma aşağıdakine benzer olabilir:

public class Foo {
    private int field1, field2;
    private String field3;

    public Foo(int field1, int field2, String field3) {
        this.field1 = field1;
        this.field2 = field2;
        this.field3 = field3;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) {
            return true;
        }
        if (obj == null || getClass() != obj.getClass()) {
            return false;
        }

        Foo f = (Foo) obj;
        return field1 == f.field1 &&
               field2 == f.field2 &&
               (field3 == null ? f.field3 == null : field3.equals(f.field3);
    }

    @Override
    public int hashCode() {
        int hash = 1;
        hash = 31 * hash + field1;
        hash = 31 * hash + field2;
        hash = 31 * hash + (field3 == null ? 0 : field3.hashCode());
        return hash;
    }
}

Kısa yol olarak Arrays.hashCode() kullanma #

Java 1.2 ve üzeri sürümlerde, bir karma kodu hesaplamak için bir algoritma geliştirmek yerine, alan değerlerini içeren bir Object veya primitives dizisi sağlayarak “java.util.Arrays#hashCode” kullanılarak bir algoritma oluşturulabilir:

@Override
public int hashCode() {
    return Arrays.hashCode(new Object[] {field1, field2, field3});
}
son sürüm

Java 1.7, kendisine sağlanan nesnelerin değerlerine dayalı bir karma kod hesaplayan bir kolaylık yöntemi olan “hash(Object…nesneler)” sağlayan “java.util.Objects” sınıfını tanıttı. Bu yöntem tıpkı java.util.Arrays#hashCode gibi çalışır.

@Override
public int hashCode() {
    return Objects.hash(field1, field2, field3);
}
son sürüm

Not: Bu yaklaşım verimsizdir ve özel “hashCode()” yönteminiz her çağrıldığında çöp nesneler üretir:

  • Geçici bir ‘Nesne[]’ oluşturulur. (Objects.hash() sürümünde dizi, “varargs” mekanizması tarafından oluşturulur.)
  • Alanlardan herhangi biri ilkel türlerse, kutulanmaları gerekir ve bu daha geçici nesneler oluşturabilir.
  • Dizi doldurulmalıdır.
  • Dizi, “Arrays.hashCode” veya “Objects.hash” yöntemiyle yinelenmelidir.
  • “Arrays.hashCode” veya “Objects.hash” tarafından yapılması gereken (muhtemelen) “Object.hashCode()” çağrıları satır içine alınamaz.

Hash kodlarının dahili önbelleğe alınması #

Bir nesnenin karma kodunun hesaplanması pahalı olabileceğinden, karma kod değerini ilk hesaplandığında nesne içinde önbelleğe almak çekici olabilir. Örneğin

public final class ImmutableArray {
    private int[] array;
    private volatile int hash = 0;

    public ImmutableArray(int[] initial) {
        array = initial.clone();
    }

    // Other methods

    @Override
    public boolean equals(Object obj) {
         // ...
    }

    @Override
    public int hashCode() {
        int h = hash;
        if (h == 0) {
            h = Arrays.hashCode(array);
            hash = h;
        }
        return h;
    }
}

Bu yaklaşım, hash kodunu önbelleğe almak için fazladan bir alanın ek yüküne karşı hash kodunu hesaplamanın (tekrar tekrar) maliyetini ortadan kaldırır. Bunun bir performans optimizasyonu olarak işe yarayıp yaramayacağı, belirli bir nesnenin ne sıklıkta hash işlemine (arama) ve diğer faktörlere bağlı olacaktır.

Ayrıca, bir ‘ImmutableArray’in gerçek karma kodunun sıfır olması durumunda (232‘ta bir şans), önbelleğin etkisiz olduğunu fark edeceksiniz.

Son olarak, hash ettiğimiz nesne değişkense, bu yaklaşımın doğru bir şekilde uygulanması çok daha zordur. Ancak, karma kodların değişmesi durumunda daha büyük endişeler vardır; yukarıdaki sözleşmeye bakın.

toString() yöntemi #

toString() yöntemi, nesnenin içeriğini kullanarak bir nesnenin bir String temsilini oluşturmak için kullanılır. Sınıfınızı yazarken bu yöntem geçersiz kılınmalıdır. Bir nesne “merhaba” + anObject’de olduğu gibi bir dizeyle birleştirildiğinde, “toString()” örtük olarak çağrılır.

Aşağıdakileri göz önünde bulundur:

public class User {
    private String firstName;
    private String lastName;
    
    public User(String firstName, String lastName) {
        this.firstName = firstName;
        this.lastName = lastName;
    }
    
    @Override
    public String toString() {
        return firstName + " " + lastName;
    }
    
    public static void main(String[] args) {
        User user = new User("John", "Doe");
        System.out.println(user.toString()); // Prints "John Doe"
    }   
}

Burada, “Object” sınıfından “toString()”, yazdırırken nesneyle ilgili anlamlı veriler sağlamak için “User” sınıfında geçersiz kılınır.

println() kullanılırken, nesnenin toString() yöntemi örtük olarak çağrılır. Bu nedenle, bu ifadeler aynı şeyi yapar:

System.out.println(user); // toString() is implicitly called on `user`
System.out.println(user.toString());

Eğer ’toString()’ yukarıda bahsedilen ‘User’ sınıfında geçersiz kılınmazsa, ‘System.out.println(user)’, ‘User@659e0bfd’ veya sınıf adı dışında neredeyse hiçbir yararlı bilgi içermeyen benzer bir Dize döndürebilir. Bunun nedeni, çağrının, “Kullanıcı” sınıfının yapısı veya iş kuralları hakkında hiçbir şey bilmeyen temel Java “Object” sınıfının “toString()” uygulamasını kullanması olacaktır. Bu işlevi sınıfınızda değiştirmek istiyorsanız, yöntemi geçersiz kılmanız yeterlidir.

eşittir() yöntemi #

TL;DR

== referans eşitliği için testler (bunların aynı nesne olup olmadıkları)

.equals() değer eşitliğini test eder (bunların mantıksal olarak “eşit” olup olmadıkları)

’equals()’, iki nesneyi eşitlik için karşılaştırmak için kullanılan bir yöntemdir. “Object” sınıfındaki “equals()” yönteminin varsayılan uygulaması, yalnızca ve ancak her iki referans da aynı örneğe işaret ediyorsa “true” değerini döndürür. Bu nedenle, == ile karşılaştırma ile aynı şekilde davranır.

public class Foo {
    int field1, field2;
    String field3;

    public Foo(int i, int j, String k) {
        field1 = i;
        field2 = j;
        field3 = k;
    }

    public static void main(String[] args) {
        Foo foo1 = new Foo(0, 0, "bar");
        Foo foo2 = new Foo(0, 0, "bar");

        System.out.println(foo1.equals(foo2)); // prints false
    }
}

‘foo1’ ve ‘foo2’ aynı alanlarla oluşturulmuş olsa da, bellekte iki farklı nesneye işaret ediyorlar. Varsayılan “eşittir()” uygulaması bu nedenle “yanlış” olarak değerlendirilir.

Bir nesnenin içeriğini eşitlik açısından karşılaştırmak için “equals()” geçersiz kılınmalıdır.

public class Foo {
    int field1, field2;
    String field3;

    public Foo(int i, int j, String k) {
        field1 = i;
        field2 = j;
        field3 = k;
    }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) {
            return true;
        }
        if (obj == null || getClass() != obj.getClass()) {
            return false;
        }

        Foo f = (Foo) obj;
        return field1 == f.field1 &&
               field2 == f.field2 &&
               (field3 == null ? f.field3 == null : field3.equals(f.field3));
    }

    @Override
    public int hashCode() {
        int hash = 1;
        hash = 31 * hash + this.field1;
        hash = 31 * hash + this.field2;
        hash = 31 * hash + (field3 == null ? 0 : field3.hashCode());
        return hash;
    }

    public static void main(String[] args) {
        Foo foo1 = new Foo(0, 0, "bar");
        Foo foo2 = new Foo(0, 0, "bar");

        System.out.println(foo1.equals(foo2)); // prints true
    }
}

Burada geçersiz kılınan “equals()” yöntemi, alanları aynıysa nesnelerin eşit olduğuna karar verir.

hashCode() yönteminin de üzerine yazıldığına dikkat edin. Bu yöntemin sözleşmesi, iki nesne eşit olduğunda, karma değerlerinin de aynı olması gerektiğini belirtir. Bu nedenle, hemen hemen her zaman ‘hashCode()’ ve ’equals()’ birlikte geçersiz kılınmalıdır.

“Equals” yönteminin argüman türüne özellikle dikkat edin. Bu, “Foo obj” değil, “Object obj"dir. İkincisini yönteminize koyarsanız, bu “eşittir” yönteminin geçersiz kılınması değildir.

Kendi sınıfınızı yazarken, ’equals()’ ve ‘hashCode()’ öğelerini geçersiz kılarken benzer bir mantık yazmanız gerekecektir. Çoğu IDE bunu sizin için otomatik olarak oluşturabilir.

Bir “equals()” uygulamasının bir örneği, çekirdek Java API’sinin bir parçası olan “String” sınıfında bulunabilir. İşaretçileri karşılaştırmak yerine, “String” sınıfı “String"in içeriğini karşılaştırır.

Java 1.7, iki potansiyel olarak “boş” referansı karşılaştıran bir kolaylık yöntemi olan “equals” sağlayan “java.util.Objects” sınıfını tanıttı, böylece “equals” yönteminin uygulamalarını basitleştirmek için kullanılabilir.

@Override
public boolean equals(Object obj) {
    if (this == obj) {
        return true;
    }
    if (obj == null || getClass() != obj.getClass()) {
        return false;
    }

    Foo f = (Foo) obj;
    return field1 == f.field1 && field2 == f.field2 && Objects.equals(field3, f.field3);
}
son sürüm

Sınıf Karşılaştırması #

Equals yöntemi herhangi bir nesneye karşı çalışabileceğinden, yöntemin sıklıkla yaptığı ilk şeylerden biri (“boş” olup olmadığını kontrol ettikten sonra), karşılaştırılan nesnenin sınıfının mevcut sınıfla eşleşip eşleşmediğini kontrol etmektir.

@Override
public boolean equals(Object obj) {
    //...check for null
    if (getClass() != obj.getClass()) {
        return false;
    }
    //...compare fields
}

Bu genellikle sınıf nesnelerini karşılaştırarak yukarıdaki gibi yapılır. Ancak, bu, açık olmayabilecek birkaç özel durumda başarısız olabilir. Örneğin, bazı çerçeveler sınıfların dinamik vekillerini üretir ve bu dinamik vekiller aslında farklı bir sınıftır. İşte JPA kullanan bir örnek.

Foo detachedInstance = ...
Foo mergedInstance = entityManager.merge(detachedInstance);
if (mergedInstance.equals(detachedInstance)) {
    //Can never get here if equality is tested with getClass()
    //as mergedInstance is a proxy (subclass) of Foo
}

Bu sınırlamayı aşmanın bir mekanizması, “instanceof” kullanarak sınıfları karşılaştırmaktır.

@Override
public final boolean equals(Object obj) {
    if (!(obj instanceof Foo)) {
        return false;
    }
    //...compare fields
}

Ancak, “instanceof” kullanılırken kaçınılması gereken birkaç tuzak vardır. Foo potansiyel olarak başka alt sınıflara sahip olabileceğinden ve bu alt sınıflar “equals()” öğesini geçersiz kılabileceğinden, bir “Foo"nun bir “FooSubclass"a eşit olduğu ancak “FooSubclass"ın “Foo"ya eşit olmadığı bir duruma girebilirsiniz.

Foo foo = new Foo(7);
FooSubclass fooSubclass = new FooSubclass(7, false);
foo.equals(fooSubclass) //true
fooSubclass.equals(foo) //false

Bu, simetri ve geçişlilik özelliklerini ihlal eder ve dolayısıyla “equals()” yönteminin geçersiz bir uygulamasıdır. Sonuç olarak, “instanceof” kullanılırken “equals()” yöntemini “final” yapmak iyi bir uygulamadır (yukarıdaki örnekte olduğu gibi). Bu, hiçbir alt sınıfın “equals()” öğesini geçersiz kılmamasını ve temel varsayımları ihlal etmemesini sağlayacaktır.

wait() ve notify() yöntemleri #

wait() ve notify() birlikte çalışır – bir iş parçacığı bir nesne üzerinde ‘wait()‘i çağırdığında, o iş parçacığı aynı nesne üzerinde başka bir iş parçacığı ’notify()’ veya ’notifyAll()’ çağırana kadar engellenir.

(Ayrıca Bakınız: https://www.wikiod.com/tr/java)

package com.example.examples.object;

import java.util.concurrent.atomic.AtomicBoolean;

public class WaitAndNotify {

    public static void main(String[] args) throws InterruptedException {
        final Object obj = new Object();
        AtomicBoolean aHasFinishedWaiting = new AtomicBoolean(false);
    
        Thread threadA = new Thread("Thread A") {
            public void run() {
                System.out.println("A1: Could print before or after B1");
                System.out.println("A2: Thread A is about to start waiting...");
                try {
                    synchronized (obj) { // wait() must be in a synchronized block
                        // execution of thread A stops until obj.notify() is called
                        obj.wait();
                    }
                    System.out.println("A3: Thread A has finished waiting. "
                            + "Guaranteed to happen after B3");
                } catch (InterruptedException e) {
                    System.out.println("Thread A was interrupted while waiting");
                } finally {
                    aHasFinishedWaiting.set(true);
                }
            }
        };
    
        Thread threadB = new Thread("Thread B") {
            public void run() {
                System.out.println("B1: Could print before or after A1");

                System.out.println("B2: Thread B is about to wait for 10 seconds");
                for (int i = 0; i < 10; i++) {
                    try {                        
                        Thread.sleep(1000); // sleep for 1 second 
                    } catch (InterruptedException e) {
                        System.err.println("Thread B was interrupted from waiting");
                    }
                }
            
                System.out.println("B3: Will ALWAYS print before A3 since "
                        + "A3 can only happen after obj.notify() is called.");
            
                while (!aHasFinishedWaiting.get()) {
                    synchronized (obj) {
                        // notify ONE thread which has called obj.wait()
                        obj.notify();
                    }
                }
            }
        };
    
        threadA.start();
        threadB.start();
    
        threadA.join();
        threadB.join();
    
        System.out.println("Finished!");
    }
}

A3: Thread A has finished waiting. Guaranteed to happen after B3
Finished!

A1: Could print before or after B1

getClass() yöntemi #

getClass() yöntemi, bir nesnenin çalışma zamanı sınıf türünü bulmak için kullanılabilir. Aşağıdaki örneğe bakın:

public class User {
   
    private long userID;
    private String name;

    public User(long userID, String name) {
        this.userID = userID;
        this.name = name;
    }
}

public class SpecificUser extends User {
    private String specificUserID;

    public SpecificUser(String specificUserID, long userID, String name) {
        super(userID, name);
        this.specificUserID = specificUserID;
    }
}

public static void main(String[] args){
    User user = new User(879745, "John");
    SpecificUser specificUser = new SpecificUser("1AAAA", 877777, "Jim");
    User anotherSpecificUser = new SpecificUser("1BBBB", 812345, "Jenny");

    System.out.println(user.getClass()); //Prints "class User"
    System.out.println(specificUser.getClass()); //Prints "class SpecificUser"
    System.out.println(anotherSpecificUser.getClass()); //Prints "class SpecificUser"
}

“getClass()” yöntemi en spesifik sınıf türünü döndürür, bu nedenle “anotherSpecificUser” üzerinde “getClass()” çağrıldığında, dönüş değeri “class SpecificUser” olur çünkü bu, kalıtım ağacında “User"dan daha aşağıdadır. `.

‘getClass’ yönteminin şu şekilde bildirilmesi dikkat çekicidir:

public final native Class<?> getClass();

‘getClass’ çağrısı tarafından döndürülen gerçek statik tür, ‘Class<? T>‘yi genişletir, burada ‘T’, üzerinde ‘getClass’ın çağrıldığı nesnenin statik türüdür.

yani, aşağıdakiler derlenecektir:

Class<? extends String> cls = "".getClass();

klon() yöntemi #

clone() yöntemi, bir nesnenin bir kopyasını oluşturmak ve döndürmek için kullanılır. Bu tartışmalı yöntem sorunlu olduğu için kaçınılmalıdır ve bir kopya oluşturucu veya kopyalama için başka bir yaklaşım ‘klon()’ lehine kullanılmalıdır.

Yöntemin kullanılabilmesi için, yöntemi çağıran tüm sınıfların ‘Klonlanabilir’ arabirimini uygulaması gerekir.

“Klonlanabilir” arabirimin kendisi, çağıran nesneler sınıfının “Klonlanabilir” olup olmadığını kontrol eden “yerel” “klon()” yönteminin davranışını değiştirmek için kullanılan bir etiket arabirimidir. Arayan bu arabirimi uygulamazsa, bir ‘CloneNotSupportedException’ atılır.

“Object” sınıfının kendisi bu arabirimi uygulamaz, bu nedenle çağıran nesne “Object” sınıfındaysa bir “CloneNotSupportedException” oluşturulur.

Bir klonun doğru olması için klonlandığı nesneden bağımsız olması gerekir, bu nedenle nesneyi döndürülmeden önce değiştirmek gerekebilir. Bu, klonlanan nesnenin iç yapısını oluşturan değişebilir nesnelerden herhangi birini de kopyalayarak esasen bir “derin kopya” oluşturmak anlamına gelir. Bu doğru bir şekilde uygulanmazsa, klonlanan nesne bağımsız olmayacak ve klonlandığı nesne ile değiştirilebilir nesnelere aynı referanslara sahip olmayacaktır. Birinde yapılacak herhangi bir değişiklik diğerini etkileyeceğinden, bu tutarsız davranışlarla sonuçlanacaktır.

class Foo implements Cloneable {
    int w;
    String x;
    float[] y;
    Date z;
    
    public Foo clone() {
        try {
            Foo result = new Foo();
            // copy primitives by value
            result.w = this.w;
            // immutable objects like String can be copied by reference
            result.x = this.x;
            
            // The fields y and z refer to a mutable objects; clone them recursively.
            if (this.y != null) {
              result.y = this.y.clone();
            }
            if (this.z != null) {
              result.z = this.z.clone();
            }
            
            // Done, return the new object
            return result;
            
        } catch (CloneNotSupportedException e) {
            // in case any of the cloned mutable fields do not implement Cloneable
            throw new AssertionError(e);
        }
    }
}

finalize() yöntemi #

Bu, “Object” sınıfının korumalı ve statik olmayan bir yöntemidir. Bu yöntem, bellekten silinmeden önce bir nesne üzerinde bazı son işlemleri gerçekleştirmek veya işlemleri temizlemek için kullanılır.

Dokümana göre, çöp toplama nesneye daha fazla referans olmadığını belirlediğinde, bu yöntem çöp toplayıcı tarafından bir nesne üzerinde çağrılır.

Ancak, nesne hala erişilebilir durumdaysa veya nesne uygun hale geldiğinde hiçbir Çöp Toplayıcı çalışmadığında “finalize()” yönteminin çağrılacağının garantisi yoktur. Bu yüzden bu yönteme güvenmemek daha iyidir.

Java çekirdek kitaplıklarında bazı kullanım örnekleri bulunabilir, örneğin “FileInputStream.java"da:

protected void finalize() throws IOException {
    if ((fd != null) &&  (fd != FileDescriptor.in)) {
        /* if fd is shared, the references in FileDescriptor
         * will ensure that finalizer is only called when
         * safe to do so. All references using the fd have
         * become unreachable. We can call close()
         */
        close();
    }
}

Bu durumda, o kaynak daha önce kapatılmamışsa, kaynağı kapatmak için son şanstır.

Genel olarak, her türlü uygulamada ‘finalize()’ yöntemini kullanmak kötü bir uygulama olarak kabul edilir ve bundan kaçınılmalıdır.

Sonlandırıcılar, kaynakları serbest bırakmak (örneğin, dosyaları kapatmak) için değildir. Sistem yığın alanı azaldığında (eğer!) Çöp toplayıcı çağrılır. Sistemin dosya tanıtıcıları azaldığında veya başka herhangi bir nedenle çağrılacağına güvenemezsiniz.

Sonlandırıcılar için amaçlanan kullanım durumu, yaklaşan kıyamet hakkında başka bir nesneyi bilgilendirmek üzere geri kazanılmak üzere olan bir nesne içindir. Artık bu amaç için daha iyi bir mekanizma var — java.lang.ref.WeakReference<T> sınıfı. Bir ‘finalize()’ yöntemi yazmanız gerektiğini düşünüyorsanız, aynı sorunu bunun yerine ‘WeakReference’ kullanarak çözüp çözemeyeceğinize bakmalısınız. Bu, sorununuzu çözmezse, tasarımınızı daha derin bir düzeyde yeniden düşünmeniz gerekebilir.

Daha fazla okumak için burada Joshua Bloch’un “Etkili Java” kitabından ‘finalize()’ yöntemiyle ilgili bir Öğe yer almaktadır.

Nesne oluşturucu #

Java’daki tüm kurucular, “Nesne” yapıcısına bir çağrı yapmalıdır. Bu, ‘super()’ çağrısı ile yapılır. Bu, bir kurucudaki ilk satır olmalıdır. Bunun nedeni, nesnenin herhangi bir ek başlatma gerçekleştirilmeden önce öbek üzerinde gerçekten oluşturulabilmesidir.

Bir kurucuda ‘super()’ çağrısını belirtmezseniz, derleyici sizin için onu yerleştirecektir.

Yani bu örneklerin üçü de işlevsel olarak aynıdır.

‘super()’ yapıcısına açık çağrı ile

public class MyClass {

    public MyClass() {
        super();
    }
}

“super()” yapıcısına örtülü çağrı ile

public class MyClass {

    public MyClass() {
        // empty
    }
}

örtük kurucu ile

public class MyClass {

}

Yapıcı Zincirleme ne olacak?

Bir kurucunun ilk talimatı olarak diğer kurucuları çağırmak mümkündür. Hem bir süper kurucuya yapılan açık çağrı hem de başka bir kurucuya yapılan çağrının her ikisinin de ilk talimat olması gerektiğinden, bunlar birbirini dışlar.

public class MyClass {

    public MyClass(int size) {

        doSomethingWith(size);

    }

    public MyClass(Collection<?> initialValues) {

        this(initialValues.size());
        addInitialValues(initialValues);
    }
}

Yeni MyClass(Arrays.asList("a", "b", "c")) çağrısı, ikinci kurucuyu List argümanıyla çağırır, bu da sırayla ilk kurucuya yetki verir (bu da dolaylı olarak super()) ve ardından listenin ikinci boyutuyla addInitialValues(int size) öğesini çağırın. Bu, birden çok kurucunun aynı işi yapması gerektiğinde kod tekrarını azaltmak için kullanılır.

Belirli bir kurucuyu nasıl çağırırım?

Yukarıdaki örnekte, new MyClass("argument") veya new MyClass("argument", 0) çağrılabilir. Başka bir deyişle, yöntem aşırı yüklemesi gibi, yalnızca kurucuyu seçtiğiniz kurucu için gerekli olan parametrelerle çağırırsınız.

Object sınıfı yapıcısında ne olacak?

Varsayılan boş bir kurucuya sahip bir alt sınıfta olacağından başka bir şey olmaz (eksi super() çağrısı).

Varsayılan boş kurucu açıkça tanımlanabilir, ancak değilse, başka bir kurucu tanımlanmadığı sürece, derleyici onu sizin için koyacaktır.

Object içindeki yapıcıdan bir Object nasıl oluşturulur?

Nesnelerin gerçek yaratılması JVM’ye bağlıdır. Java’daki her kurucu, örneğin başlatılmasından sorumlu olan “” adlı özel bir yöntem olarak görünür. Bu “” yöntemi derleyici tarafından sağlanır ve “” Java’da geçerli bir tanımlayıcı olmadığı için doğrudan dilde kullanılamaz.

JVM bu <init> yöntemini nasıl çağırır?

JVM, “invokespecial” talimatını kullanarak “” yöntemini çağırır ve yalnızca başlatılmamış sınıf örneklerinde çağrılabilir.
