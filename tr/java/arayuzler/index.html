Bir interface, “interface” anahtar sözcüğü kullanılarak bildirilebilen, sınıfa benzer bir başvuru türüdür.

Arabirimler yalnızca sabitler, yöntem imzaları, varsayılan yöntemler, statik yöntemler ve iç içe türler içerebilir. Yöntem gövdeleri yalnızca varsayılan yöntemler ve statik yöntemler için mevcuttur. Soyut sınıflar gibi, Arayüzler de somutlaştırılamaz; yalnızca sınıflar tarafından uygulanabilir veya diğer arayüzler tarafından genişletilebilirler.

Arayüz, Java’da tam soyutlama elde etmenin yaygın bir yoludur.

Sözdizimi #

  • genel arayüz Foo { void foo(); /* diğer yöntemler */ }
  • Foo1 genel arabirimi Foo’yu genişletir { void bar(); /* diğer yöntemler */ }
  • public class Foo2, Foo, Foo1 { /* Foo ve Foo1’in uygulanması */ } uygular

Birden çok arayüz uygulama #

Bir Java sınıfı birden çok arabirim uygulayabilir.

public interface NoiseMaker {
    String noise = "Making Noise"; // interface variables are public static final by default

    String makeNoise(); //interface methods are public abstract by default
}

public interface FoodEater {
    void eat(Food food);
}

public class Cat implements NoiseMaker, FoodEater { 
    @Override
    public String makeNoise() {
        return "meow";
    }

    @Override
    public void eat(Food food) {
        System.out.println("meows appreciatively");
    }
}

“Cat” sınıfının her iki arabirimde de devralınan “soyut” yöntemleri nasıl uygulaması gerektiğine dikkat edin. Ayrıca, bir sınıfın gerektiği kadar çok arabirimi pratik olarak nasıl uygulayabileceğine dikkat edin (JVM Sınırlaması nedeniyle 65,535 bir sınır vardır).

NoiseMaker noiseMaker = new Cat(); // Valid
FoodEater foodEater = new Cat(); // Valid
Cat cat = new Cat(); // valid

Cat invalid1 = new NoiseMaker(); // Invalid
Cat invalid2 = new FoodEater(); // Invalid

Not:

  1. Bir arabirimde bildirilen tüm değişkenler ‘public static final’dir
  2. Bir arabirim yöntemlerinde bildirilen tüm yöntemler ‘genel özet’tir (Bu ifade yalnızca Java 7 aracılığıyla geçerlidir. Java 8’den itibaren, bir arabirimde soyut olması gerekmeyen yöntemlere sahip olmanıza izin verilir; bu tür yöntemler [ olarak bilinir]. varsayılan yöntemler2)
  3. Arayüzler “son” olarak bildirilemez
  4. Birden fazla arabirim aynı imzaya sahip bir yöntem bildirirse, etkin bir şekilde yalnızca bir yöntem olarak kabul edilir ve hangi arabirim yönteminin uygulandığını ayırt edemezsiniz.
  5. Derlemenin ardından her arabirim için karşılık gelen bir InterfaceName.class dosyası oluşturulur.

Bir Arayüz Bildirme ve Uygulama #

“interface” anahtar sözcüğünü kullanarak bir arayüzün bildirilmesi:

public interface Animal {
    String getSound(); // Interface methods are public by default
}

Ek Açıklamayı Geçersiz Kıl

@Override
public String getSound() {
    // Code goes here...
}

Bu, derleyiciyi geçersiz kıldığımızı kontrol etmeye zorlar ve programın yeni bir yöntem tanımlamasını veya yöntem imzasını bozmasını engeller.

Arayüzler “uygulamalar” anahtar kelimesi kullanılarak uygulanır.

public class Cat implements Animal {

    @Override 
    public String getSound() {
        return "meow";
    }
}

public class Dog implements Animal {

    @Override
    public String getSound() {
        return "woof";
    }
}

Örnekte, “Cat” ve “Dog” sınıfları, bir arabirimin yöntemleri doğal olarak soyut olduğundan (varsayılan yöntemler dışında) “getSound()” yöntemini tanımlamalıdır**.

Arayüzleri kullanma

Animal cat = new Cat();
Animal dog = new Dog();

System.out.println(cat.getSound()); // prints "meow"
System.out.println(dog.getSound()); // prints "woof"

Bir arayüzü genişletme #

Bir arabirim, ’extends’ anahtar sözcüğü aracılığıyla başka bir arabirimi genişletebilir.

public interface BasicResourceService {
    Resource getResource();
}

public interface ExtendedResourceService extends BasicResourceService {
    void updateResource(Resource resource);
}

Şimdi ‘ExtendedResourceService’ uygulayan bir sınıfın hem ‘getResource()’ hem de ‘updateResource()’ uygulamasını uygulaması gerekecek.

Birden çok arayüzü genişletme

Sınıflardan farklı olarak, “extends” anahtar sözcüğü, birden çok arabirimi (virgülle ayrılmış) genişletmek için kullanılabilir ve arabirim kombinasyonlarının yeni bir arabirime dönüştürülmesine olanak tanır.

public interface BasicResourceService {
    Resource getResource();
}

public interface AlternateResourceService {
    Resource getAlternateResource();
}

public interface ExtendedResourceService extends BasicResourceService, AlternateResourceService {
    Resource updateResource(Resource resource);
}

Bu durumda, ‘ExtendedResourceService’ uygulayan bir sınıfın ‘getResource()’, ‘getAlternateResource()’ ve ‘updateResource()’ uygulamalarını gerçekleştirmesi gerekecektir.

Arayüzlerin kullanışlılığı #

Arayüzler birçok durumda son derece yardımcı olabilir. Örneğin, bir hayvan listeniz olduğunu ve listede dolaşmak istediğinizi ve her birinin çıkardıkları sesi yazdırdığını varsayalım.

{cat, dog, bird}

Bunu yapmanın bir yolu arayüzleri kullanmaktır. Bu, aynı yöntemin tüm sınıflarda çağrılmasına izin verir.

public interface Animal {
    public String getSound();
}

“Hayvan” uygulayan herhangi bir sınıfın içinde bir “getSound()” yöntemi de olmalıdır, ancak hepsinin farklı uygulamaları olabilir.

public class Dog implements Animal {
    public String getSound() {
        return "Woof";
    }
}

public class Cat implements Animal {
    public String getSound() {
        return "Meow";
    }
}

public class Bird implements Animal{
    public String getSound() {
        return "Chirp";
    }
}

Artık her biri bir getSound() yöntemine sahip üç farklı sınıfımız var. Bu sınıfların tümü, “getSound()” yöntemini bildiren “Animal” arabirimini “uyguladığı” için, herhangi bir “Animal” örneğinin üzerinde “getSound()” çağrılabilir.

Animal dog = new Dog();
Animal cat = new Cat();
Animal bird = new Bird();

dog.getSound(); // "Woof"
cat.getSound(); // "Meow"
bird.getSound(); // "Chirp"

Bunların her biri bir ‘Hayvan’ olduğundan, hayvanları bir listeye koyabilir, aralarında dolaşabilir ve seslerini yazdırabiliriz.

Animal[] animals = { new Dog(), new Cat(), new Bird() };
for (Animal animal : animals) {
    System.out.println(animal.getSound());
}

Dizinin sırası Dog, Cat ve ardından Bird olduğundan, “Woof Miow Chirp” konsola yazdırılacaktır.

Arayüzler, işlevler için dönüş değeri olarak da kullanılabilir. Örneğin, giriş “dog” ise bir “Köpek”, giriş “kedi” ise “Kedi” ve “kuş” ise “Kuş” döndürülür ve ardından bu hayvan kullanılarak yapılabilir

public Animal getAnimalByName(String name) {
    switch(name.toLowerCase()) {
        case "dog":
            return new Dog();
        case "cat":
            return new Cat();
        case "bird":
            return new Bird();
        default:
            return null;
    }
}

public String getAnimalSoundByName(String name){
    Animal animal = getAnimalByName(name);
    if (animal == null) {
        return null;
    } else { 
        return animal.getSound();
    }
}

String dogSound = getAnimalSoundByName("dog"); // "Woof"
String catSound = getAnimalSoundByName("cat"); // "Meow"
String birdSound = getAnimalSoundByName("bird"); // "Chirp"
String lightbulbSound = getAnimalSoundByName("lightbulb"); // null

Arayüzler, genişletilebilirlik için de kullanışlıdır, çünkü yeni bir ‘Hayvan’ türü eklemek isterseniz, üzerlerinde yaptığınız işlemlerde herhangi bir değişiklik yapmanız gerekmez.

Varsayılan yöntemler #

Java 8’de tanıtılan varsayılan yöntemler, bir arabirim içindeki bir uygulamayı belirtmenin bir yoludur. Bu, bir arabirimin kısmi uygulamasını sağlayarak ve alt sınıf hiyerarşisini kısıtlayarak tipik “Temel” veya “Özet” sınıftan kaçınmak için kullanılabilir.

Gözlemci desen uygulaması #

Örneğin, Gözlemci-Dinleyici modelini doğrudan arabirime uygulamak mümkündür, bu da uygulama sınıflarına daha fazla esneklik sağlar.

interface Observer {
    void onAction(String a);
}

interface Observable{
    public abstract List<Observer> getObservers();

    public default void addObserver(Observer o){
        getObservers().add(o);
    }

    public default void notify(String something ){
        for( Observer l : getObservers() ){
            l.onAction(something);
        }
    }
}

Artık, herhangi bir sınıf, yalnızca Gözlemlenebilir arabirimi uygulanarak “Gözlenebilir” hale getirilebilirken, farklı bir sınıf hiyerarşisinin parçası olmakta özgürdür.

abstract class Worker{
    public abstract void work();
}

public class MyWorker extends Worker implements Observable {

    private List<Observer> myObservers = new ArrayList<Observer>();
    
    @Override
    public List<Observer> getObservers() {
        return myObservers;
    }

    @Override
    public void work(){
        notify("Started work");

        // Code goes here...

        notify("Completed work");
    }
    
    public static void main(String[] args) {    
        MyWorker w = new MyWorker();
       
        w.addListener(new Observer() {
            @Override
            public void onAction(String a) {
                System.out.println(a + " (" + new Date() + ")");
            }
        });
        
        w.work();
    }
}

Elmas sorunu #

Java 8’deki derleyici, bir sınıf aynı imzaya sahip bir yöntem içeren arabirimler uygularken ortaya çıkan elmas sorununun farkındadır.

Bunu çözmek için, bir uygulayıcı sınıfın paylaşılan yöntemi geçersiz kılması ve kendi uygulamasını sağlaması gerekir.

interface InterfaceA {
    public default String getName(){
        return "a";
    }
}

interface InterfaceB {
    public default String getName(){
        return "b";
    }
}

public class ImpClass implements InterfaceA, InterfaceB {

    @Override
    public String getName() {    
        //Must provide its own implementation
        return InterfaceA.super.getName() + InterfaceB.super.getName();
    }
    
    public static void main(String[] args) {    
        ImpClass c = new ImpClass();
        
        System.out.println( c.getName() );                   // Prints "ab"
        System.out.println( ((InterfaceA)c).getName() );     // Prints "ab"
        System.out.println( ((InterfaceB)c).getName() );     // Prints "ab"
    }
}

Aynı ada sahip yöntemlere ve derlenmeyecek farklı dönüş türlerine sahip parametrelere sahip olma sorunu hala var.

Uyumluluk sorunlarını çözmek için varsayılan yöntemleri kullanın #

Varsayılan yöntem uygulamaları, arabirimlerin birkaç sınıf tarafından kullanıldığı mevcut bir sistemde bir arabirime bir yöntem eklenirse çok kullanışlı olur.

Tüm sistemi parçalamaktan kaçınmak için, bir arabirime bir yöntem eklediğinizde varsayılan bir yöntem uygulaması sağlayabilirsiniz. Bu şekilde sistem hala derlenecek ve gerçek uygulamalar adım adım yapılabilecektir.

Daha fazla bilgi için Varsayılan Yöntemler konusuna bakın.

Arayüzlerdeki Değiştiriciler #

Oracle Java Stil Kılavuzu şunları belirtir:

Değiştiriciler örtük olduklarında yazılmamalıdır.

(Bkz. Oracle Resmi Kod Standardı. com/documentation/Java/2697/oracle-official-code-standard#t=201705210148056874377) bağlam ve gerçek Oracle belgesine bağlantı için.)

Bu stil kılavuzu özellikle arayüzler için geçerlidir. Aşağıdaki kod parçasını ele alalım:

interface I {
    public static final int VARIABLE = 0;

    public abstract void method();

    public static void staticMethod() { ... }
    public default void defaultMethod() { ... }
}

Değişkenler #

Tüm arabirim değişkenleri örtük olarak sabittir ve örtük “genel” (herkes için erişilebilir), “statik” (arayüz adıyla erişilebilir) ve “final” (bildirim sırasında başlatılmalıdır) değiştiricileri vardır:

genel statik final int DEĞİŞKEN = 0;

Yöntemler #

  1. Uygulama sağlamayan tüm yöntemler örtük olarak ‘genel’ ve ‘soyut’tur.
genel özet void method();
  1. “Statik” veya “varsayılan” değiştiricisi * olan tüm yöntemler uygulama* sağlamalıdır ve dolaylı olarak “genel"dir.
public static void staticMethod() { ... }
son sürüm

Yukarıdaki değişikliklerin tümü uygulandıktan sonra aşağıdakileri alacağız:

interface I {
    int VARIABLE = 0;
    
    void method();

    static void staticMethod() { ... }
    default void defaultMethod() { ... }
}

Arayüzleri Jeneriklerle Kullanma #

Farklı kanal türlerine (örneğin AMQP, JMS, vb.) veri yayınlamaya/tüketmeye izin veren bir arayüz tanımlamak istediğinizi, ancak uygulama ayrıntılarını değiştirebilmek istediğinizi varsayalım…

Birden çok uygulamada yeniden kullanılabilecek temel bir IO arabirimi tanımlayalım:

public interface IO<IncomingType, OutgoingType> {

    void publish(OutgoingType data);
    IncomingType consume();
    IncomingType RPCSubmit(OutgoingType data);

}

Şimdi bu arayüzü somutlaştırabilirim, ancak bu yöntemler için varsayılan uygulamalarımız olmadığından, somutlaştırdığımızda bir uygulamaya ihtiyacı olacak:

    IO<String, String> mockIO = new IO<String, String>() {

        private String channel = "somechannel";

        @Override
        public void publish(String data) {
            System.out.println("Publishing " + data + " to " + channel);
        }

        @Override
        public String consume() {
            System.out.println("Consuming from " + channel);
            return "some useful data";
        }

        @Override
        public String RPCSubmit(String data) {
            return "received " + data + " just now ";
        }

    };

    mockIO.consume(); // prints: Consuming from somechannel
    mockIO.publish("TestData"); // Publishing TestData to somechannel
    System.out.println(mockIO.RPCSubmit("TestData")); // received TestData just now

Bu arayüzle daha faydalı bir şey de yapabiliriz, diyelim ki onu bazı temel RabbitMQ fonksiyonlarını sarmak için kullanmak istiyoruz:

public class RabbitMQ implements IO<String, String> {

    private String exchange;
    private String queue;

    public RabbitMQ(String exchange, String queue){
        this.exchange = exchange;
        this.queue = queue;
    }

    @Override
    public void publish(String data) {
        rabbit.basicPublish(exchange, queue, data.getBytes());
    }

    @Override
    public String consume() {
        return rabbit.basicConsume(exchange, queue);
    }

    @Override
    public String RPCSubmit(String data) {
        return rabbit.rpcPublish(exchange, queue, data);
    }

}

Diyelim ki bu IO arayüzünü, son sistemimden bu yana web siteme yapılan ziyaretleri saymanın ve ardından toplam ziyaret sayısını gösterebilmenin bir yolu olarak kullanmak istiyorum - şöyle bir şey yapabilirsiniz:

import java.util.concurrent.atomic.AtomicLong;

public class VisitCounter implements IO<Long, Integer> {

    private static AtomicLong websiteCounter = new AtomicLong(0);
    
    @Override
    public void publish(Integer count) {
        websiteCounter.addAndGet(count);
    }

    @Override
    public Long consume() {
        return websiteCounter.get();
    }

    @Override
    public Long RPCSubmit(Integer count) {
        return websiteCounter.addAndGet(count);
    }
    
}

Şimdi VisitCounter’ı kullanalım:

    VisitCounter counter = new VisitCounter();

    // just had 4 visits, yay
    counter.publish(4);
    // just had another visit, yay
    counter.publish(1);

    // get data for stats counter
    System.out.println(counter.consume()); // prints 5

    // show data for stats counter page, but include that as a page view
    System.out.println(counter.RPCSubmit(1)); // prints 6

Birden çok arabirim uygularken, aynı arabirimi iki kez uygulayamazsınız. Bu aynı zamanda genel arayüzler için de geçerlidir. Bu nedenle, aşağıdaki kod geçersizdir ve bir derleme hatasıyla sonuçlanacaktır:

interface Printer<T> {
    void print(T value);
}

// Invalid!
class SystemPrinter implements Printer<Double>, Printer<Integer> {
    @Override public void print(Double d){ System.out.println("Decimal: " + d); }
    @Override public void print(Integer i){ System.out.println("Discrete: " + i); }
}

Sınırlı tür parametrelerini güçlendirin #

Sınırlı tür parametreleri, genel tür bağımsız değişkenleri üzerinde kısıtlamalar belirlemenize olanak tanır:

class SomeClass {

}

class Demo<T extends SomeClass> {

}

Ancak bir tür parametresi yalnızca tek bir sınıf türüne bağlanabilir.

Bir arabirim türü, zaten bir bağlamaya sahip olan bir türe bağlanabilir. Bu, & sembolü kullanılarak elde edilir:

interface SomeInterface {

}

class GenericClass<T extends SomeClass & SomeInterface> {

}

Bu, bağlamayı güçlendirir ve potansiyel olarak birden çok türden türetilmesi için tür bağımsız değişkenleri gerektirir.

Birden çok arabirim türü, bir tür parametresine bağlanabilir:

class Demo<T extends SomeClass & FirstInterface & SecondInterface> {

}

Ama dikkatli kullanılmalıdır. Çoklu arabirim bağlamaları genellikle bir kod kokusunun bir işaretidir, bu da diğer türler için bağdaştırıcı görevi gören yeni bir türün oluşturulması gerektiğini gösterir:

interface NewInterface extends FirstInterface, SecondInterface {

}

class Demo<T extends SomeClass & NewInterface> {

}

Soyut bir sınıfta arayüzleri uygulama #

Bir “arayüz"de tanımlanan bir yöntem, varsayılan olarak “genel özet"tir. Bir “soyut sınıf” bir “arayüz” uyguladığında, “arayüz"de tanımlanan herhangi bir yöntemin “soyut sınıf” tarafından uygulanması gerekmez. Bunun nedeni, “soyut” olarak bildirilen bir “sınıf"ın soyut yöntem bildirimlerini içerebilmesidir. Bu nedenle, herhangi bir arabirimden ve/veya ‘soyut sınıftan’ miras alınan herhangi bir “soyut” yöntemi uygulamak ilk somut alt sınıfın sorumluluğundadır.

public interface NoiseMaker {
    void makeNoise();
}

public abstract class Animal implements NoiseMaker {
    //Does not need to declare or implement makeNoise()
    public abstract void eat();
}

//Because Dog is concrete, it must define both makeNoise() and eat()
public class Dog extends Animal {
    @Override
    public void makeNoise() {
        System.out.println("Borf borf");
    }

    @Override
    public void eat() {
        System.out.println("Dog eats some kibble.");
    }
}

Java 8’den itibaren, bir “arayüz"ün yöntemlerin “varsayılan” uygulamalarını bildirmesi mümkündür, bu da yöntemin “soyut” olmayacağı anlamına gelir, bu nedenle herhangi bir somut alt sınıf, yöntemi uygulamaya zorlanmaz, ancak yöntemi miras alır. geçersiz kılınmadıkça “varsayılan” uygulama.
