Java Programlama Dili için Oracle resmi stil kılavuzu, Oracle’daki geliştiriciler tarafından takip edilen ve diğer herhangi bir Java geliştiricisi tarafından takip edilmesi önerilen bir standarttır. Dosya adlarını, dosya organizasyonunu, girintiyi, yorumları, bildirimleri, ifadeleri, boşlukları, adlandırma kurallarını, programlama uygulamalarını kapsar ve bir kod örneği içerir.

  • Yukarıdaki örnekler kesinlikle Oracle’ın yeni resmi stil rehberini takip etmektedir. Başka bir deyişle, bunlar bu sayfanın yazarları tarafından öznel olarak oluşturulmuş değildir.

  • Resmi stil kılavuzu, orijinal stil kılavuzu ve kodun çoğuyla geriye dönük uyumlu olacak şekilde dikkatlice yazılmıştır. vahşi doğada.

  • Resmi stil kılavuzu peer incelendi /pipermail/discuss/2015-August/003771.html), Brian Goetz (Java Dili Mimarı) ve Mark Reinhold (Java Platformunun Baş Mimarı) tarafından yazılmıştır.

  • Örnekler normatif değildir; Kodu biçimlendirmenin doğru yolunu göstermeyi amaçlasalar da, kodu doğru biçimde biçimlendirmenin başka yolları da olabilir. Bu genel bir ilkedir: Kodu biçimlendirmenin, tümü resmi yönergelere bağlı kalarak birkaç yolu olabilir.

Adlandırma Kuralları #

Paket adları #

  • Paket adları, alt çizgi veya diğer özel karakterler olmadan küçük harf olmalıdır.
  • Paket adları, geliştirici firmanın web adresinin ters yetki kısmı ile başlar. Bu kısım proje/program yapısına bağlı paket altyapısı ile takip edilebilir.
  • Çoğul biçim kullanmayın. Örneğin, “java.lang.annotations” yerine “java.lang.annotation” kullanan standart API’nin kurallarına uyun.
  • Örnekler: “com.sirketiniz.widget.button”, “com.sirketiniz.core.api”

Sınıf, Arayüz ve Enum Adları #

  • Sınıf ve numaralandırma adları genellikle isimler olmalıdır.
  • Arayüz adları tipik olarak …able ile biten isimler veya sıfatlar olmalıdır.
  • Her kelimenin ilk harfi büyük olacak şekilde karışık harf kullanın (yani CamelCase).
  • ^[A-Z][a-zA-Z0-9]*$ normal ifadesini eşleştirin.
  • Kısaltma uzun biçimden daha yaygın olarak kullanılmadıkça, tam sözcükler kullanın ve kısaltma kullanmaktan kaçının.
  • Daha uzun bir sınıf adının parçasıysa bir kısaltmayı kelime olarak biçimlendirin.
  • Örnekler: “ArrayList”, “BigInteger”, “ArrayIndexOutOfBoundsException”, “Yinelenebilir”.

Yöntem Adları #

Yöntem adları tipik olarak fiiller veya eylemlerin diğer açıklamaları olmalıdır.

  • ^[a-z][a-zA-Z0-9]*$ normal ifadesiyle eşleşmelidirler.
  • İlk harf küçük harfle karışık olarak kullanın.
  • Örnekler: “toString”, “hashCode”

Değişkenler #

Değişken isimleri ilk harf küçük harfle karışık yazılmalıdır.

  • ^[a-z][a-zA-Z0-9]*$ normal ifadesini eşleştirin
  • Daha fazla öneri: Değişkenler
  • Örnekler: “elemanlar”, “currentIndex”

Tür Değişkenleri #

Birkaç tür değişkenin dahil olduğu basit durumlar için tek bir büyük harf kullanın.

  • ^[A-Z][0-9]?$ normal ifadesini eşleştirin
  • Bir harf diğerinden daha açıklayıcıysa (haritalardaki anahtarlar ve değerler için K ve V veya bir işlev dönüş türü için R gibi) bunu kullanın, aksi takdirde T kullanın.
  • Tek harfli değişkenlerin kafa karıştırıcı hale geldiği karmaşık durumlar için, tamamı büyük harflerle yazılmış daha uzun isimler kullanın ve kelimeleri ayırmak için alt çizgi (_) kullanın.
  • Örnekler: “T”, “V”, “SRC_VERTEX”

sabitler #

Sabitler (dil kuralları veya sözleşme gereği içeriği değişmez olan ‘statik son’ alanlar) sözcükleri ayırmak için tamamı büyük harflerle ve alt çizgi (’_’) ile adlandırılmalıdır.

  • ^[A-Z][A-Z0-9]*(_[A-Z0-9]+)*$ normal ifadesini eşleştirin
  • Örnekler: “BUFFER_SIZE”, “MAX_LEVEL”

Adlandırmayla ilgili diğer yönergeler #

  • Dış kapsamlarda yöntemleri, değişkenleri ve değişkenleri gizleme/gölgelemeden kaçının.
  • Adın ayrıntı düzeyinin kapsamın boyutuyla ilişkili olmasına izin verin. (Örneğin, büyük sınıfların alanları için tanımlayıcı adlar ve yerel kısa ömürlü değişkenler için kısa adlar kullanın.)
  • Genel statik üyeleri adlandırırken, statik olarak içe aktarılacağını düşünüyorsanız, tanımlayıcının kendi kendini açıklayıcı olmasına izin verin.
  • Daha fazla okuma: Adlandırma Bölümü (resmi Java Stil Kılavuzunda)

Kaynak: Oracle’dan Java Stil Yönergeleri

Sınıf Yapısı #

Sınıf üyelerinin sırası #

Sınıf üyeleri aşağıdaki gibi sıralanmalıdır:

  1. Alanlar (genel, korumalı ve özel sırasına göre)
  2. Yapıcılar
  3. Fabrika yöntemleri
  4. Diğer Yöntemler (genel, korumalı ve özel sırasına göre)

Alanları ve yöntemleri öncelikle erişim değiştiricilerine veya tanımlayıcılarına göre sıralamak gerekli değildir.

İşte bu siparişin bir örneği:

class Example {

    private int i;

    Example(int i) {
        this.i = i;
    }

    static Example getExample(int i) {
        return new Example(i);
    }

    @Override
    public String toString() {
        return "An example [" + i + "]";
    }

}

Sınıf üyelerinin gruplandırılması #

  • İlgili alanlar birlikte gruplandırılmalıdır.
  • Yuvalanmış bir tür, ilk kullanımından hemen önce bildirilebilir; aksi takdirde alanlardan önce bildirilmelidir.
  • Yapıcılar ve aşırı yüklenmiş yöntemler, işlevselliğe göre birlikte gruplanmalı ve artan arite ile sıralanmalıdır. Bu, bu yapılar arasındaki yetkilendirmenin kodda aşağı doğru aktığı anlamına gelir.
  • Yapıcılar, aralarında başka üyeler olmadan birlikte gruplandırılmalıdır.
  • Bir yöntemin aşırı yüklenmiş varyantları, aralarında diğer üyeler olmadan birlikte gruplandırılmalıdır.

Ek Açıklamalar #

Beyanname açıklamaları, açıklamalı beyandan ayrı bir satıra konmalıdır.

@SuppressWarnings("unchecked")
public T[] toArray(T[] typeHolder) {
    ...
}

Ancak, okunabilirliği artırıyorsa, tek satırlı bir yöntemi açıklayan az sayıda veya kısa açıklama, yöntemle aynı satıra konabilir. Örneğin, biri şunu yazabilir:

@Nullable String getName() { return name; }

Tutarlılık ve okunabilirlik açısından ya tüm açıklamalar aynı satıra konulmalı ya da her açıklama ayrı bir satıra konulmalıdır.

// Bad.
@Deprecated @SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Even worse.
@Deprecated @SafeVarargs
@CustomAnnotation public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated
@SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated @SafeVarargs @CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

  • …ikincil olarak aşağıdaki sıraya göre paket menşeine göre

Joker karakter içe aktarma #

  • Joker karakter içe aktarmaları genel olarak kullanılmamalıdır.
  • Çok sayıda yakından ilişkili sınıfı içe aktarırken (düzinelerce farklı “düğüm” sınıfı olan bir ağaç üzerinde bir ziyaretçinin uygulanması gibi), bir joker karakter içe aktarma kullanılabilir.
  • Her durumda, dosya başına birden fazla joker karakter içe aktarma kullanılmamalıdır.

Gereksiz Parantezler #

return flag ? "yes" : "no";

String cmp = (flag1 != flag2) ? "not equal" : "equal";

// Don't do this
return (flag ? "yes" : "no");
  • Okunabilirliği artırıyorlarsa, gereksiz gruplama parantezleri (yani değerlendirmeyi etkilemeyen parantezler) kullanılabilir.
  • Fazladan gruplama parantezleri, genellikle ortak işleçleri içeren daha kısa ifadelerde dışarıda bırakılmalı, ancak daha uzun ifadelerde veya parantezler olmadan önceliği ve ilişkilendirilebilirliği belirsiz olan işleçleri içeren ifadelerde dahil edilmelidir. Önemsiz koşullara sahip üçlü ifadeler ikincisine aittir.
  • Bir “return” anahtar sözcüğünü izleyen ifadenin tamamı parantez içine alınmamalıdır.

Diş telleri #

class Example {
    void method(boolean error) {
        if (error) {
            Log.error("Error occurred!");
            System.out.println("Error!");
        } else { // Use braces since the other block uses braces.
            System.out.println("No error");
        }
    }
}

  • Açma parantezleri tek başına bir satıra değil, mevcut satırın sonuna yerleştirilmelidir.

  • Blok boş olmadığı sürece kapatma parantezinin önünde yeni bir satır olmalıdır (aşağıdaki Kısa Formlara bakın)

  • Parantezler, tek satırlı if ve döngü gövdeleri gibi, dilin onları isteğe bağlı kıldığı durumlarda bile önerilir.

  • Bir blok birden fazla satıra yayılıyorsa (yorumlar dahil) parantez içermelidir.

  • ‘if’ / ’else’ ifadesindeki bloklardan biri parantez içeriyorsa, diğer blok da parantez içermelidir.

  • Blok, bir çevreleyen blokta en son geliyorsa, parantezleri olmalıdır.

  • ‘do…while’ döngülerindeki ’else’, ‘catch’ ve ‘while’ anahtar sözcükleri, önceki bloğun kapanış ayracı ile aynı satırda gider.

Kısa formlar #

enum Response { YES, NO, MAYBE }
public boolean isReference() { return true; }

Yukarıdaki tavsiyeler, tekdüzeliği iyileştirmeyi (ve böylece aşinalığı / okunabilirliği artırmayı) amaçlamaktadır. Bazı durumlarda, yukarıdaki yönergelerden sapan “kısa formlar” da aynı şekilde okunabilirdir ve bunun yerine kullanılabilir. Bu durumlar, örneğin basit numaralandırma bildirimlerini ve önemsiz yöntemleri ve lambda ifadelerini içerir.

Girinti #

  • Girinti düzeyi dört boşluktur.

  • Girinti için sadece boşluk karakterleri kullanılabilir. Sekme yok.

  • Boş satırlar girintili olmamalıdır. (Bu, takip eden beyaz boşluk kuralıyla ima edilir.)

  • ‘case’ satırları dört boşlukla girintili olmalı ve case içindeki ifadeler dört boşlukla daha girintili olmalıdır.

    switch (var) { case TWO: setChoice(“two”); break; case THREE: setChoice(“three”); break; default: throw new IllegalArgumentException(); }

Devam satırlarının nasıl girintileneceğine ilişkin yönergeler için https://www.wikiod.com/tr/java/oracle-resmi-kod-standard#sarma ifadeleri adresine bakın.

  • Diğer tüm sayısal ön ekler, ekler ve son ekler küçük harf kullanmalıdır.

Paket beyanı #

package com.example.my.package;

Paket bildirimi, önerilen maksimum satır uzunluğunu aşıp aşmadığına bakılmaksızın satıra sarılmamalıdır.

Lambda İfadeleri #

Runnable r = () -> System.out.println("Hello World");

Supplier<String> c = () -> "Hello World";

// Collection::contains is a simple unary method and its behavior is
// clear from the context. A method reference is preferred here.
appendFilter(goodStrings::contains);

// A lambda expression is easier to understand than just tempMap::put in this case
trackTemperature((time, temp) -> tempMap.put(time, temp));
  • Tek satırlı blok lambdalar yerine ifade lambdaları tercih edilir.
  • Yöntem referansları genellikle lambda ifadelerine tercih edilmelidir.
  • Bağlı örnek yöntemi referansları veya birden fazla aritesi olan yöntemler için bir lambda ifadesinin anlaşılması daha kolay olabilir ve bu nedenle tercih edilebilir. Özellikle yöntemin davranışı bağlamdan net değilse.
  • Parametre türleri, okunabilirliği artırmadıkça kullanılmamalıdır.
  • Bir lambda ifadesi birkaç satırdan fazla uzanıyorsa, bir yöntem oluşturmayı düşünün.

Java Kaynak Dosyaları #

  • Tüm satırlar, örneğin CR veya CR+LF değil, satır besleme karakteri (LF, ASCII değeri 10) ile sonlandırılmalıdır.

  • Bir satırın sonunda boşluk bırakmayabilir.

  • Bir kaynak dosyanın adı, yalnızca bir paket özel sınıfı içeren dosyalar için bile, içerdiği sınıfın adına ve ardından `.java’ uzantısına eşit olmalıdır. Bu, “package-info.java” gibi herhangi bir sınıf bildirimi içermeyen dosyalar için geçerli değildir.

İfadeleri sarma #

  • Kaynak kodu ve yorumlar genellikle satır başına 80 karakteri geçmemeli ve nadiren satır başı girinti dahil 100 karakteri geçmemelidir.

Karakter sınırı duruma göre değerlendirilmelidir. Asıl önemli olan satırın anlamsal “yoğunluğu” ve okunabilirliğidir. Satırları gereksiz uzun yapmak onları okumayı zorlaştırır; benzer şekilde, onları 80 sütuna sığdırmak için “kahramanca girişimler” yapmak da onları okumayı zorlaştırabilir. Burada özetlenen esneklik, monitör gayrimenkulünün kullanımını en üst düzeye çıkarmak için değil, geliştiricilerin bu aşırı uçlardan kaçınmasını sağlamayı amaçlamaktadır.

  • URL’ler veya örnek komutlar kaydırılmamalıdır.

    // Ok even though it might exceed max line width when indented. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable(targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext(targetContainerType));

    // Wrapping preferable String pretty = Stream.of(args) .map(Argument::prettyPrint) .collectors(joining(", “));

    // Too strict interpretation of max line width. Readability suffers. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable( targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext( targetContainerType);

    // Should be wrapped even though it fits within the character limit String pretty = Stream.of(args).map(Argument::prettyPrint).collectors(joining(”, “));

  • Daha yüksek bir sözdizimsel düzeyde sarmalama, daha düşük bir sözdizimsel düzeyde sarmaya tercih edilir.

  • Her satırda en fazla 1 ifade olmalıdır.

  • Devam satırı aşağıdaki dört yoldan biriyle girintili olmalıdır

  • Varyant 1: Önceki satırın girintisine göre 8 ekstra boşluk ile.

  • Varyant 2: Sarılmış ifadenin başlangıç ​​sütununa göre fazladan 8 boşluk ile.

  • Varyant 3: Önceki kardeş ifadesi ile hizalanmıştır (devam satırı olduğu açık olduğu sürece)

  • Varyant 4: Zincirleme bir ifadede önceki yöntem çağrısı ile hizalanmıştır.

Özel karakterler #

  • LF dışında izin verilen tek boşluk karakteri Boşluktur (ASCII değeri 32). Bunun, diğer boşluk karakterlerinin (örneğin, dize ve karakter değişmezlerinde) çıkışlı biçimde yazılması gerektiği anlamına geldiğini unutmayın.

  • \', \", \\, \t, \b, \r, \f ve \n karşılık gelen sekizliğe tercih edilmelidir (ör. \047) veya Unicode (ör. \u0027) çıkış karakterleri.

  • Test amacıyla yukarıdaki kurallara karşı çıkmak gerekirse, test gerekli girdiyi programlı olarak oluşturmalıdır.

  • Dizilerin köşeli parantezleri (String[] args) türünde olmalı ve (String args[]) değişkeninde olmamalıdır.
  • İlk kullanılmadan hemen önce yerel bir değişken bildirin ve bunu bildirime mümkün olduğunca yakın bir şekilde başlatın.
