Değişmez Nesneler

  • Savunma amaçlı kopyalamayı kullanarak bir türün değişmez bir sürümünü oluşturma.
  • Değişmez bir sınıfın tarifi
  • Bir sınıfın değişmez olmasını engelleyen tipik tasarım kusurları

Bu sayfada

  • Savunma amaçlı kopyalamayı kullanarak bir türün değişmez bir sürümünü oluşturma.
  • Değişmez bir sınıfın tarifi
  • Bir sınıfın değişmez olmasını engelleyen tipik tasarım kusurları

Değişmez nesnelerin sabit durumları vardır (ayarlayıcı yoktur), bu nedenle tüm durum nesne oluşturma zamanında bilinmelidir.

Teknik olarak gerekli olmasa da, tüm alanları “final” yapmak en iyi uygulamadır. Bu, değişmez sınıfı iş parçacığı için güvenli hale getirecektir (bkz. Java Concurrency in Practice, 3.4.1).

Örnekler, bunu başarmaya yardımcı olabilecek çeşitli kalıpları göstermektedir.

Savunma amaçlı kopyalamayı kullanarak bir türün değişmez bir sürümünü oluşturma. #

Java’daki bazı temel türler ve sınıflar temelde değiştirilebilir. Örneğin, tüm dizi türleri değişkendir ve “java.util.Data” gibi sınıflar da değişkendir. Bu, değişmez bir türün zorunlu olduğu durumlarda garip olabilir.

Bununla başa çıkmanın bir yolu, değiştirilebilir tür için değişmez bir sarmalayıcı oluşturmaktır. İşte bir dizi tamsayı için basit bir sarmalayıcı

public class ImmutableIntArray {
    private final int[] array;
    
    public ImmutableIntArray(int[] array) {
        this.array = array.clone();
    }

    public int[] getValue() {
        return this.clone();
    }
}

Bu sınıf, değişebilir durumu (int[]) onu değiştirebilecek herhangi bir koddan izole etmek için savunmacı kopyalama kullanarak çalışır:

  • Yapıcı, parametre dizisinin ayrı bir kopyasını oluşturmak için clone() kullanır. Oluşturucunun çağırıcısı daha sonra parametre dizisini değiştirirse, bu ‘ImmutableIntArray’ durumunu etkilemez.

  • “getValue()” yöntemi, döndürülen diziyi oluşturmak için “clone()” yöntemini de kullanır. Arayan, sonuç dizisini değiştirecek olsaydı, ‘ImmutableIntArray’ durumunu etkilemezdi.

Sarılmış dizide salt okunur işlemleri gerçekleştirmek için ImmutableIntArraye yöntemler de ekleyebiliriz; Örneğin. uzunluğunu alın, belirli bir dizindeki değeri alın, vb.

Bu şekilde uygulanan değişmez bir sarmalayıcı türünün orijinal türle uyumlu olmadığını unutmayın. Birincisini ikincisinin yerine basitçe koyamazsınız.

Değişmez bir sınıfın tarifi #

Değişmez bir nesne, durumu değiştirilemeyen bir nesnedir. Değişmez sınıf, örnekleri tasarım ve uygulama açısından değişmez olan bir sınıftır. Değişmezliğin bir örneği olarak en yaygın olarak sunulan Java sınıfı java.lang.String‘dir.

Aşağıdaki stereotipik bir örnektir:

public final class Person {
    private final String name;
    private final String ssn;     // (SSN == social security number)

    public Person(String name, String ssn) {
        this.name = name;
        this.ssn = ssn;
    }

    public String getName() {
        return name;
    }
   
    public String getSSN() {
        return ssn;
    }
}

Bunun bir varyasyonu, yapıcıyı “özel” olarak bildirmek ve bunun yerine bir “genel statik” fabrika yöntemi sağlamaktır.

Değişmez bir sınıf için standart tarif aşağıdaki gibidir:

  • Tüm özellikler yapıcı(lar)da veya fabrika yöntem(ler)inde ayarlanmalıdır.
  • Hiçbir ayarlayıcı olmamalıdır.
  • Arayüz uyumluluğu nedeniyle ayarlayıcıların dahil edilmesi gerekiyorsa, ya hiçbir şey yapmamalı ya da bir istisna atmalıdır.
  • Tüm özellikler ‘özel’ ve ‘son’ olarak bildirilmelidir.
  • Değiştirilebilir türlere referans olan tüm özellikler için:
  • özellik, yapıcı aracılığıyla iletilen değerin derin bir kopyasıyla başlatılmalıdır ve
  • mülkün alıcısı, mülk değerinin derin bir kopyasını getirmelidir.
  • Birinin değişmez bir sınıfın değişken bir alt sınıfını oluşturmasını önlemek için sınıf ‘final’ olarak bildirilmelidir.

Dikkat edilmesi gereken birkaç şey daha:

  • Değişmezlik, nesnenin null yapılabilir olmasını engellemez; Örneğin. “null”, bir “Dize” değişkenine atanabilir.
  • Bir değişmez sınıf özellikleri “final” olarak bildirilirse, örnekler doğal olarak iş parçacığı için güvenlidir. Bu, değişmez sınıfları çok iş parçacıklı uygulamaları uygulamak için iyi bir yapı taşı yapar.

Bir sınıfın değişmez olmasını engelleyen tipik tasarım kusurları #

Yapıcı(lar)da gerekli tüm özellikleri ayarlamadan bazı ayarlayıcıları kullanma

public final class Person { // example of a bad immutability
    private final String name;
    private final String surname;
    public Person(String name) {
        this.name = name;
      }
    public String getName() { return name;}
    public String getSurname() { return surname;}
    public void setSurname(String surname) { this.surname = surname); }
}

‘Person’ sınıfının değişmez olmadığını göstermek kolaydır:

Person person = new Person("Joe");
person.setSurname("Average"); // NOT OK, change surname field after creation

Bunu düzeltmek için setSurname()‘yi silin ve yapıcıyı aşağıdaki gibi yeniden düzenleyin:

public Person(String name, String surname) {
    this.name = name;
    this.surname = surname;
  }

Örnek değişkenleri özel ve nihai olarak işaretlememek

Aşağıdaki sınıfa bir göz atın:

public final class Person {
    public String name;
    public Person(String name) {
        this.name = name;
     }
    public String getName() {
        return name;
    }
    
}

Aşağıdaki snippet, yukarıdaki sınıfın değişmez olmadığını gösterir:

Person person = new Person("Average Joe");
person.name = "Magic Mike"; // not OK, new name for person after creation

Bunu düzeltmek için name özelliğini “private” ve “final” olarak işaretlemeniz yeterlidir.

Bir alıcıda sınıfın değişken bir nesnesini gösterme

Aşağıdaki sınıfa bir göz atın:

import java.util.List;
import java.util.ArrayList;
public final class Names {
    private final List<String> names;
    public Names(List<String> names) {
        this.names = new ArrayList<String>(names);
    }
    public List<String> getNames() {
        return names;
    }
    public int size() {
        return names.size();
    }
}

‘Names’ sınıfı ilk bakışta değişmez görünüyor, ancak aşağıdaki kodun gösterdiği gibi değil:

List<String> namesList = new ArrayList<String>();
namesList.add("Average Joe");
Names names = new Names(namesList);
System.out.println(names.size()); // 1, only containing "Average Joe"
namesList = names.getNames();
namesList.add("Magic Mike");
System.out.println(names.size()); // 2, NOT OK, now names also contains "Magic Mike"

Bunun nedeni, “getNames()” tarafından döndürülen referans Listesinde yapılan bir değişikliğin, “Adlar"ın gerçek listesini değiştirebilmesidir.

Bunu düzeltmek için, aşağıdaki gibi savunma kopyaları oluşturarak sınıfın değişken nesnelerine ya da başvuran referansları döndürmekten kaçının:

public List<String> getNames() {
   return new ArrayList<String>(this.names); // copies elements
}

veya alıcıları yalnızca diğer değişmez nesnelerin ve ilkellerin döndürüleceği şekilde aşağıdaki gibi tasarlayarak:

public String getName(int index) {
    return names.get(index);
}
public int size() {
    return names.size();
}

Yapıcıyı değişmez sınıfın dışında değiştirilebilen nesne(ler) ile enjekte etme

Bu, önceki kusurun bir varyasyonudur. Aşağıdaki sınıfa bir göz atın:

import java.util.List;
public final class NewNames {
    private final List<String> names;
    public Names(List<String> names) {
        this.names = names;
    }
    public String getName(int index) {
        return names.get(index);
    }
    public int size() {
        return names.size();
    }
}

Daha önce ‘Names’ sınıfı gibi, ‘NewNames’ sınıfı da ilk bakışta değişmez görünüyor, ancak öyle değil, aslında aşağıdaki pasaj bunun aksini kanıtlıyor:

List<String> namesList = new ArrayList<String>();
namesList.add("Average Joe");
NewNames names = new NewNames(namesList);
System.out.println(names.size()); // 1, only containing "Average Joe"
namesList.add("Magic Mike");
System.out.println(names.size()); // 2, NOT OK, now names also contains "Magic Mike"

Bunu düzeltmek için, önceki kusurda olduğu gibi, nesneyi doğrudan değişmez sınıfa atamadan savunma kopyalarını yapın, yani yapıcı aşağıdaki gibi değiştirilebilir:

    public Names(List<String> names) {
        this.names = new ArrayList<String>(names);
    }

Sınıfın yöntemlerinin geçersiz kılınmasına izin verilmesi

Aşağıdaki sınıfa bir göz atın:

public class Person {
    private final String name;
    public Person(String name) {
        this.name = name;
      }
    public String getName() { return name;}
}

‘Person’ sınıfı ilk bakışta değişmez görünüyor, ancak yeni bir ‘Person’ alt sınıfının tanımlandığını varsayalım:

public class MutablePerson extends Person {
    private String newName;
    public MutablePerson(String name) {
        super(name);            
    }
    @Override
    public String getName() {
        return newName;
    }
    public void setName(String name) {
        newName = name;
    }
}

şimdi ‘Kişi’ (im)değişebilirliği, yeni alt sınıf kullanılarak polimorfizm yoluyla kullanılabilir:

Person person = new MutablePerson("Average Joe");
System.out.println(person.getName()); prints Average Joe
person.setName("Magic Mike"); // NOT OK, person has now a new name!
System.out.println(person.getName()); // prints Magic Mike

Bunu düzeltmek için ya sınıfı genişletilemeyecek şekilde “son” olarak işaretleyin ya da tüm oluşturucu(lar)ını “özel” olarak ilan edin.
