Java kodu için belgeler genellikle javadoc kullanılarak oluşturulur. Javadoc, Sun Microsystems tarafından Java kaynak kodundan HTML biçiminde API belgeleri oluşturmak amacıyla oluşturulmuştur. HTML biçimini kullanmak, ilgili belgeleri birbirine köprüleme kolaylığı sağlar.

Sözdizimi #

  • /** – bir sınıf, alan, yöntem veya paket üzerinde JavaDoc’un başlangıcı
  • @author // Sınıfın, arayüzün veya enum’un yazarını adlandırmak için. Bu gereklidir.
  • @version // Bu sınıfın, arabirimin veya enum’un sürümü. Bu gereklidir. Ödeme sırasında doldurmak için kaynak kontrol yazılımınız için %I% veya %G% gibi makrolar kullanabilirsiniz.
  • @param // Bir yöntemin veya yapıcının argümanlarını (parametrelerini) göstermek için. Her parametre için bir @param etiketi belirtin.
  • @return // Void olmayan yöntemlerin dönüş türlerini göstermek için.
  • @exception // Yöntemden veya yapıcıdan hangi istisnaların atılabileceğini gösterir. ZORUNLU olan istisnalar burada listelenmelidir. İsterseniz, ArrayIndexOutOfBoundsException gibi yakalanması gerekmeyenleri de dahil edebilirsiniz. Atılabilecek her istisna için bir @istisna belirtin.
  • @throws // @exception ile aynı.
  • @see // Bir yönteme, alana, sınıfa veya pakete bağlantılar. package.Class#something şeklinde kullanın.
  • @since // Bu yöntem, alan veya sınıf eklendiğinde. Örneğin, java.util.Optional<T> gibi bir sınıf için JDK-8.
  • @serial, @serialField, @serialData // serialVersionUID’yi göstermek için kullanılır.
  • @deprecated // Bir sınıfı, yöntemi veya alanı kullanım dışı olarak işaretlemek için. Örneğin, biri java.io.StringBufferInputStream olacaktır. Mevcut kullanımdan kaldırılan sınıfların tam listesine buradan bakın.
  • {@link} // @see’ye benzer, ancak özel metinle kullanılabilir: {@link #setDefaultCloseOperation(int closeOperation) daha fazla bilgi için bkz. JFrame#setDefaultCloseOperation}.
  • {@linkplain} // {@link}’e benzer, ancak kod yazı tipi içermez.
  • {@code} // HTML etiketleri gibi değişmez kod için. Örneğin: {@code <html></html>}. Ancak, bu tek aralıklı bir yazı tipi kullanacaktır. Aynı sonucu tek aralıklı yazı tipi olmadan elde etmek için {@literal} kullanın.
  • {@literal} // {@code} ile aynı, ancak tek aralıklı yazı tipi yok.
  • {@value} // Statik bir alanın değerini gösterir: JFrame#EXIT_ON_CLOSE değeri {@value}‘dur. Veya belirli bir alana bağlanabilirsiniz: {@value AppConstants#APP_NAME} uygulama adını kullanır.
  • {@docRoot} // Geçerli dosyaya göre JavaDoc HTML’nin kök klasörü. Örnek: <a href="{@docRoot}/credits.html">Krediler</a>.
  • HTML’ye izin verilir: <code>“Merhaba çerezler”.substring(3)</code>.
  • */ – JavaDoc bildiriminin sonu

Javadoc, kod içi yorumların bir HTML belgelerine dönüştürülmesine izin veren JDK’da bulunan bir araçtır. Java API Spesifikasyonu Javadoc kullanılarak oluşturulmuştur. Aynısı, 3. taraf kitaplıkların belgelerinin çoğu için de geçerlidir.

Sınıf Belgeleri #

Tüm Javadoc yorumları bir blok yorumuyla başlar ve ardından bir yıldız işareti (/**) gelir ve blok yorumu yaptığında (*/) biter. İsteğe bağlı olarak, her satır isteğe bağlı bir boşluk ve tek bir yıldızla başlayabilir; dokümantasyon dosyaları oluşturulduğunda bunlar yok sayılır.

/**
 * Brief summary of this class, ending with a period.
 *
 * It is common to leave a blank line between the summary and further details.
 * The summary (everything before the first period) is used in the class or package 
 * overview section.
 *
 * The following inline tags can be used (not an exhaustive list): 
 * {@link some.other.class.Documentation} for linking to other docs or symbols
 * {@link some.other.class.Documentation Some Display Name} the link's appearance can be 
 * customized by adding a display name after the doc or symbol locator
 * {@code code goes here} for formatting as code
 * {@literal <>[]()foo} for interpreting literal text without converting to HTML markup 
 * or other tags.
 *
 * Optionally, the following tags may be used at the end of class documentation 
 * (not an exhaustive list):
 *
 * @author John Doe
 * @version 1.0
 * @since 5/10/15
 * @see some.other.class.Documentation
 * @deprecated This class has been replaced by some.other.package.BetterFileReader
 * 
 * You can also have custom tags for displaying additional information.
 * Using the @custom.<NAME> tag and the -tag custom.<NAME>:htmltag:"context"
 * command line option, you can create a custom tag.
 *
 * Example custom tag and generation:
 * @custom.updated 2.0    
 * Javadoc flag: -tag custom.updated🅰️"Updated in version:"
 * The above flag will display the value of @custom.updated under "Updated in version:"
 *
 */
public class FileReader {
}

“Sınıflar” için kullanılan aynı etiketler ve biçim, “Sayılar” ve “Arayüzler” için de kullanılabilir.

Yöntem Belgeleri #

Tüm Javadoc yorumları bir blok yorumuyla başlar ve ardından bir yıldız işareti (/**) gelir ve blok yorumu yaptığında (*/) biter. İsteğe bağlı olarak, her satır isteğe bağlı bir boşluk ve tek bir yıldızla başlayabilir; dokümantasyon dosyaları oluşturulduğunda bunlar yok sayılır.

/**
 * Brief summary of method, ending with a period.
 *
 * Further description of method and what it does, including as much detail as is 
 * appropriate.  Inline tags such as
 * {@code code here}, {@link some.other.Docs}, and {@literal text here} can be used.
 *
 * If a method overrides a superclass method, {@inheritDoc} can be used to copy the 
 * documentation
 * from the superclass method
 *
 * @param stream Describe this parameter.  Include as much detail as is appropriate
 *               Parameter docs are commonly aligned as here, but this is optional.
 *               As with other docs, the documentation before the first period is 
 *               used as a summary.
 *
 * @return Describe the return values.  Include as much detail as is appropriate
 *         Return type docs are commonly aligned as here, but this is optional.
 *         As with other docs, the documentation before the first period is used as a 
 *         summary.
 *
 * @throws IOException Describe when and why this exception can be thrown.
 *                     Exception docs are commonly aligned as here, but this is 
 *                     optional.
 *                     As with other docs, the documentation before the first period 
 *                     is used as a summary.
 *                     Instead of @throws, @exception can also be used.
 *
 * @since 2.1.0
 * @see some.other.class.Documentation
 * @deprecated  Describe why this method is outdated. A replacement can also be specified.
 */
public String[] read(InputStream stream) throws IOException {

}

Paket Belgeleri #

Javadocs’ta “package-info.java” adlı bir dosya kullanarak paket düzeyinde belgeler oluşturmak mümkündür. Bu dosya aşağıdaki gibi formatlanmalıdır. Baştaki boşluk ve yıldız işaretleri isteğe bağlıdır, genellikle biçimlendirme nedeniyle her satırda bulunur

/**
 * Package documentation goes here; any documentation before the first period will 
 * be used as a summary.
 *
 * It is common practice to leave a blank line between the summary and the rest
 * of the documentation; use this space to describe the package in as much detail
 * as is appropriate.
 * 
 * Inline tags such as {@code code here}, {@link reference.to.other.Documentation},
 * and {@literal text here} can be used in this documentation.
 */
package com.example.foo;

// The rest of the file must be empty.

Yukarıdaki durumda, bu package-info.java dosyasını Java paketinin com.example.foo klasörüne koymalısınız.

Komut Satırından Javadocs Oluşturma #

Birçok IDE, Javadocs’tan otomatik olarak HTML oluşturmak için destek sağlar; bazı derleme araçları (örneğin Maven ve Gradle), HTML oluşturmayı işleyebilen eklentilere de sahiptir.

Ancak, bu araçlar Javadoc HTML’sini oluşturmak için gerekli değildir; bu, komut satırı ‘javadoc’ aracı kullanılarak yapılabilir.

Aracın en temel kullanımı:

javadoc JavaFile.java

Bu, JavaFile.java içindeki Javadoc yorumlarından HTML üretecektir.

Komut satırı aracının daha pratik bir kullanımı, “[kaynak-dizin]” içindeki tüm Java dosyalarını tekrar tekrar okuyacak, “[paket.adı]” ve tüm alt paketler için belgeler oluşturacak ve oluşturulan HTML’yi " [docs-dizini]`:

javadoc -d [docs-directory] -subpackages -sourcepath [source-directory] [package.name]

Bağlantılar #

Diğer Javadoc’lara bağlantı @link etiketi ile yapılır:

/**
 * You can link to the javadoc of an already imported class using {@link ClassName}.
 *
 * You can also use the fully-qualified name, if the class is not already imported: 
 *  {@link some.other.ClassName}
 *
 * You can link to members (fields or methods) of a class like so:
 *  {@link ClassName#someMethod()}
 *  {@link ClassName#someMethodWithParameters(int, String)}
 *  {@link ClassName#someField}
 *  {@link #someMethodInThisClass()} - used to link to members in the current class
 *  
 * You can add a label to a linked javadoc like so:
 *  {@link ClassName#someMethod() link text} 
 */

@see etiketiyle Ayrıca bkz. bölümüne öğeler ekleyebilirsiniz. @param veya @return gibi göründükleri yer alakalı değildir. Spesifikasyon, “@return"den sonra yazmanız gerektiğini söylüyor.

/**
 * This method has a nice explanation but you might found further
 * information at the bottom.
 *
 * @see ClassName#someMethod()
 */

Dış kaynaklara bağlantılar eklemek istiyorsanız, HTML <a> etiketini kullanabilirsiniz. Bunu satır içinde herhangi bir yerde veya hem “@link” hem de “@see” etiketlerinin içinde kullanabilirsiniz.

/**
 * Wondering how this works? You might want
 * to check this <a href="http://stackoverflow.com/">great service</a>.
 *
 * @see <a href="http://stackoverflow.com/">Stack Overflow</a>
 */

Belgelerin içindeki kod parçacıkları #

Belgelerin içine kod yazmanın kurallı yolu {@code } yapısıdır. Eğer çok satırlı kodunuz varsa, <pre></pre> içine sarın.

/**
 * The Class TestUtils.
 * <p>
 * This is an {@code inline("code example")}.
 * <p>
 * You should wrap it in pre tags when writing multiline code.
 * <pre>{@code
 *  Example example1 = new FirstLineExample();
 *  example1.butYouCanHaveMoreThanOneLine();
 * }</pre>
 * <p>
 * Thanks for reading.
 */
class TestUtils {

Bazen javadoc yorumunun içine bazı karmaşık kodlar koymanız gerekebilir. @ işareti özellikle sorunludur. “{@literal }” yapısının yanında eski “” etiketinin kullanılması sorunu çözer.

/**
 * Usage:
 * <pre><code>
 * class SomethingTest {
 * {@literal @}Rule
 *  public SingleTestRule singleTestRule = new SingleTestRule("test1");
 *
 * {@literal @}Test
 *  public void test1() {
 *      // only this test will be executed
 *  }
 *
 *  ...
 * }
 * </code></pre>
 */
class SingleTestRule implements TestRule { }

Satır İçi Kod Belgeleri #

Javadoc belgelendirme kodu dışında satır içi belgelenebilir.

Tek Satırlı yorumlar // ile başlatılır ve aynı satırdaki bir ifadeden sonra konumlandırılabilir, ancak daha önce değil.

public void method() {
  
  //single line comment
  someMethodCall(); //single line comment after statement
  
}

Çok Satırlı yorumlar /* ve */ arasında tanımlanır. Birden çok satıra yayılabilirler ve hatta ifadeler arasına yerleştirilebilirler.

public void method(Object object) {
  
  /*
    multi 
    line 
    comment
  */
  object/*inner-line-comment*/.method(); 
}

JavaDocs, /** ile başlayan çok satırlı yorumların özel bir şeklidir.

Çok fazla satır içi yorum kodun okunabilirliğini azaltabileceğinden, kodun yeterince açıklayıcı olmaması veya tasarım kararının açık olmaması durumunda seyrek olarak kullanılmalıdır.

Tek satırlı yorumlar için ek bir kullanım durumu, kısa, kurala dayalı anahtar kelimeler olan TAG’lerin kullanılmasıdır. Bazı geliştirme ortamları, bu tür tek yorumlar için belirli kuralları tanır. Yaygın örnekler

  • //TÜMÜ
  • //DÜZELTME
