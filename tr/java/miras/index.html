Kalıtım, bir sınıfın “extends” anahtar sözcüğünü kullanarak başka bir sınıfın özelliklerini edindiği ve genişlettiği temel bir nesne yönelimli özelliktir. Arayüzler ve “uygular” anahtar sözcüğü için, bkz. arayüzler.

Sözdizimi #

  • ClassB sınıfı ClassA’yı genişletir {…}
  • ClassB sınıfı InterfaceA’yı uygular {…}
  • InterfaceB arayüzü InterfaceA’yı genişletir {…}
  • ClassB sınıfı, ClassA’yı genişletir, InterfaceC, InterfaceD {…}
  • soyut sınıf AbstractClassB, ClassA’yı {…} genişletir
  • soyut sınıf AbstractClassB, AbstractClassA’yı {…} genişletir
  • soyut sınıf AbstractClassB, ClassA’yı genişletir InterfaceC, InterfaceD {…}

Kalıtım genellikle jeneriklerle birleştirilir, böylece temel sınıf bir veya daha fazla tür parametresine sahip olur. Bkz. Genel Sınıf Oluşturma.

Soyut Sınıflar #

Soyut bir sınıf, “abstract” anahtar sözcüğüyle işaretlenmiş bir sınıftır. Soyut olmayan sınıfın aksine, soyut - uygulamasız - yöntemler içerebilir. Ancak, soyut yöntemler olmadan soyut bir sınıf oluşturmak geçerlidir.

Soyut bir sınıf somutlaştırılamaz. Alt sınıf da soyut olduğu veya süper sınıflar tarafından soyut olarak işaretlenen tüm yöntemleri uyguladığı sürece alt sınıflanabilir (genişletilebilir).

Soyut bir sınıf örneği:

public abstract class Component {
    private int x, y;
    
    public setPosition(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public abstract void render();
}

Sınıf, en az bir soyut yöntemi olduğunda, soyut olarak işaretlenmelidir. Soyut bir yöntem, uygulaması olmayan bir yöntemdir. Diğer yöntemler, herhangi bir alt sınıf için ortak kod sağlamak amacıyla, uygulaması olan bir soyut sınıf içinde bildirilebilir.

Bu sınıfı somutlaştırmaya çalışmak bir derleme hatası sağlayacaktır:

//error: Component is abstract; cannot be instantiated   
Component myComponent = new Component();

Bununla birlikte, ‘Component’i genişleten ve tüm soyut yöntemleri için bir uygulama sağlayan ve somutlaştırılabilen bir sınıf.

public class Button extends Component {

    @Override
    public void render() {
        //render a button
    }
}

public class TextBox extends Component {

    @Override
    public void render() {
        //render a textbox
    }
}

Miras alan sınıfların örnekleri de üst sınıf (normal kalıtım) olarak kullanılabilir ve soyut yöntem çağrıldığında polimorfik bir etki sağlarlar.

Component myButton = new Button();
Component myTextBox = new TextBox();

myButton.render(); //renders a button
myTextBox.render(); //renders a text box

Soyut sınıflar ve Arayüzler

Soyut sınıflar ve arabirimler, uygulamayı sağlamak için genişletme/uygulama sınıfı gerektirirken yöntem imzalarını tanımlamanın bir yolunu sağlar.

Soyut sınıflar ve arayüzler arasında iki temel fark vardır:

  • Bir sınıf yalnızca tek bir sınıfı genişletebilir, ancak birçok arabirim uygulayabilir.
  • Soyut bir sınıf, örnek (“statik” olmayan) alanlar içerebilir, ancak arabirimler yalnızca “statik” alanlar içerebilir.

Arabirimlerde bildirilen yöntemler uygulamaları içeremezdi, bu nedenle uygulamaların soyut yöntemler olarak adlandırdığı ek yöntemler sağlamanın yararlı olduğu durumlarda soyut sınıflar kullanıldı.

Soyut Sınıfların anonim alt sınıfları

Kolaylık sağlamak için Java, yeni nesnenin yaratılması üzerine soyut yöntemler için uygulamalar sağlayan soyut sınıfların alt sınıflarının anonim örneklerinin somutlaştırılmasına izin verir. Yukarıdaki örneği kullanarak bu şöyle görünebilir:

Component myAnonymousComponent = new Component() {
    @Override
    public void render() {
        // render a quick 1-time use component
    }
}

Devralmayı ve geçersiz kılmayı kısıtlamak için ‘final’ kullanma #

Son sınıflar #

Bir “sınıf” bildiriminde kullanıldığında, “final” değiştiricisi, sınıfı “genişleten” diğer sınıfların bildirilmesini engeller. Bir “son” sınıf, kalıtım sınıfı hiyerarşisinde bir “yaprak” sınıftır.

// This declares a final class
final class MyFinalClass {
    /* some code */
}

// Compilation error: cannot inherit from final MyFinalClass
class MySubClass extends MyFinalClass {
    /* more code */
}

Final sınıfları için kullanım örnekleri #

Son sınıflar, bir sınıfın somutlaştırılmasını kontrol etmek veya önlemek için bir “özel” kurucu ile birleştirilebilir. Bu, yalnızca statik üyeleri tanımlayan bir “yardımcı sınıf” oluşturmak için kullanılabilir; yani sabitler ve statik yöntemler.

public final class UtilityClass {

    // Private constructor to replace the default visible constructor
    private UtilityClass() {}

    // Static members can still be used as usual
    public static int doSomethingCool() {
        return 123;
    }

}

Değişmez sınıflar da ‘final’ olarak bildirilmelidir. (Bir değişmez sınıf, örnekleri oluşturulduktan sonra değiştirilemeyen sınıftır; bkz. Immutable Objects konusu. ) Bunu yaparak, değişmez bir sınıfın değiştirilebilir bir altsınıfını yaratmayı imkansız hale getirirsiniz. Bu, bir alt türün üst türlerinin “davranışsal sözleşmesine” uymasını gerektiren Liskov Değiştirme İlkesi’ni ihlal ederdi.

Pratik bir bakış açısından, değişmez bir sınıfın ‘final’ olarak ilan edilmesi, program davranışı hakkında akıl yürütmeyi kolaylaştırır. Ayrıca, bir güvenlik sanal alanında güvenilmeyen kodun yürütüldüğü senaryodaki güvenlik endişelerini de giderir. (Örneğin, “Dize” “final” olarak bildirildiğinden, güvenilen bir sınıfın, değiştirilebilen alt sınıfı kabul etmesi için kandırılabileceğinden endişelenmesi gerekmez; bu, güvenilmeyen arayan tarafından gizlice değiştirilebilir.)

‘Son’ sınıfların bir dezavantajı, Mockito gibi bazı alaycı çerçevelerle çalışmamalarıdır. Güncelleme: Mockito sürüm 2 artık son sınıfların alay edilmesini destekliyor.

Son yöntemler #

‘final’ değiştirici, alt sınıflarda geçersiz kılınmalarını önlemek için yöntemlere de uygulanabilir:

public class MyClassWithFinalMethod {

    public final void someMethod() {
    }
}

public class MySubClass extends MyClassWithFinalMethod {

    @Override
    public void someMethod() { // Compiler error (overridden method is final)
    }
}

Son yöntemler, genellikle, alt sınıfları tamamen yasaklamadan bir alt sınıfın bir sınıfta neleri değiştirebileceğini kısıtlamak istediğinizde kullanılır.

“final” değiştiricisi değişkenlere de uygulanabilir, ancak değişkenler için “final"in anlamı kalıtımla ilgili değildir.

Miras #

Sınıflar arasında ’extends’ anahtar sözcüğünün kullanılmasıyla, üst sınıfın tüm özellikleri (Üst Sınıf veya Temel Sınıf olarak da bilinir) alt sınıfta bulunur (Alt Sınıf veya Türetilen olarak da bilinir). Sınıf)

public class BaseClass {

    public void baseMethod(){
        System.out.println("Doing base class stuff");
    }
}

public class SubClass extends BaseClass {

}

“SubClass” örnekleri, “baseMethod()” yöntemini devralmıştır:

SubClass s = new SubClass();
s.baseMethod();  //Valid, prints "Doing base class stuff"

Bir alt sınıfa ek içerik eklenebilir. Bunu yapmak, temel sınıfta veya aynı temel sınıftan diğer alt sınıflarda herhangi bir değişiklik olmaksızın alt sınıfta ek işlevsellik sağlar:

public class Subclass2 extends BaseClass {

    public void anotherMethod() {
        System.out.println("Doing subclass2 stuff");
    }
}

Subclass2 s2 = new Subclass2();
s2.baseMethod(); //Still valid , prints "Doing base class stuff"
s2.anotherMethod(); //Also valid, prints "Doing subclass2 stuff"

Alanlar da miras alınır:

public class BaseClassWithField {

    public int x;

}

public class SubClassWithField extends BaseClassWithField {

    public SubClassWithField(int x) {
        this.x = x; //Can access fields
    }
}

‘özel’ alanlar ve yöntemler alt sınıf içinde hala mevcuttur, ancak erişilebilir değildir:

public class BaseClassWithPrivateField {

    private int x = 5;

    public int getX() {
        return x;
    }
}

public class SubClassInheritsPrivateField extends BaseClassWithPrivateField {

    public void printX() {
        System.out.println(x); //Illegal, can't access private field x
        System.out.println(getX()); //Legal, prints 5
    }
}

SubClassInheritsPrivateField s = new SubClassInheritsPrivateField();
int x = s.getX(); //x will have a value of 5.

Java’da her sınıf en fazla bir başka sınıfı genişletebilir.

public class A{}
public class B{}
public class ExtendsTwoClasses extends A, B {} //Illegal

Bu, çoklu kalıtım olarak bilinir ve bazı dillerde yasal olsa da Java, sınıflarda buna izin vermez.

Bunun bir sonucu olarak, her sınıf, tüm sınıfların türediği ‘Nesne’ye giden dallanmayan atasal bir sınıf zincirine sahiptir.

Liskov İkame Prensibi #

İkame edilebilirlik, Barbara Liskov tarafından 1987’deki bir konferans açılış konuşmasında tanıtılan nesne yönelimli programlamada bir ilkedir ve “B” sınıfı “A” sınıfının bir alt sınıfıysa, “A” nerede olursa olsun. bekleniyor, bunun yerine ``` B kullanılabilir:

class A {...}
class B extends A {...}

public void method(A obj) {...}

A a = new B(); // Assignment OK
method(new B()); // Passing as parameter OK

Bu aynı zamanda, tür, nesneler arasında herhangi bir hiyerarşik ilişkinin gerekmediği bir arabirim olduğunda da geçerlidir:

interface Foo {
    void bar();
}

class A implements Foo {
    void bar() {...}
}

class B implements Foo {
    void bar() {...}
}

List<Foo> foos = new ArrayList<>();
foos.add(new A()); // OK
foos.add(new B()); // OK

Şimdi liste aynı sınıf hiyerarşisinden olmayan nesneleri içeriyor.

Statik Kalıtım #

Statik yöntem, normal yöntemlere benzer şekilde miras alınabilir, ancak normal yöntemlerden farklı olarak, statik yöntemi geçersiz kılmaya zorlamak için “abstract” yöntemleri oluşturmak imkansızdır. Bir süper sınıfta statik bir yöntemle aynı imzaya sahip bir yöntem yazmak, bir tür geçersiz kılma gibi görünüyor, ancak bu gerçekten yeni bir işlev yaratır, diğerini gizler.

public class BaseClass {
    
    public static int num = 5;

    public static void sayHello() {
        System.out.println("Hello");
    }

    public static void main(String[] args) {
        BaseClass.sayHello();
        System.out.println("BaseClass's num: " + BaseClass.num);
            
        SubClass.sayHello();
        //This will be different than the above statement's output, since it runs
        //A different method
        SubClass.sayHello(true);
        
        StaticOverride.sayHello();
        System.out.println("StaticOverride's num: " + StaticOverride.num);
    }
}

public  class SubClass extends BaseClass {
    
    //Inherits the sayHello function, but does not override it   
    public static void sayHello(boolean test) {
        System.out.println("Hey");
    }
}

public static class StaticOverride extends BaseClass {

    //Hides the num field from BaseClass
    //You can even change the type, since this doesn't affect the signature
    public static String num = "test";
        
    //Cannot use @Override annotation, since this is static
    //This overrides the sayHello method from BaseClass
    public static void sayHello() {
        System.out.println("Static says Hi");
    }

}

Bu sınıflardan herhangi birini çalıştırmak çıktıyı üretir:

Hello
BaseClass's num: 5
Hello
Hey
Static says Hi
StaticOverride's num: test

Normal kalıtımdan farklı olarak, statik kalıtım yöntemlerinde gizli olmadığını unutmayın. Temel “sayHello” yöntemini her zaman “BaseClass.sayHello()” kullanarak çağırabilirsiniz. Ancak sınıflar statik yöntemleri devralır _eğer alt sınıfta aynı imzaya sahip hiçbir yöntem bulunmazsa. İki yöntemin imzaları farklıysa, ad aynı olsa bile her iki yöntem de alt sınıftan çalıştırılabilir.

Statik alanlar da benzer şekilde birbirini gizler.

Soyut sınıf ve Arayüz kullanımı: “Is-a” ilişkisi ve “Has-a” özelliği #

Soyut sınıflar ne zaman kullanılır: Birden çok ilişkili nesne arasında aynı veya farklı davranışı uygulamak için

Arayüzler ne zaman kullanılır: birden çok ilgisiz nesne tarafından bir sözleşme uygulamak için

Soyut sınıflar “is a” ilişkileri oluştururken, interface’ler “sahiptir” yeteneği sağlar.

Bu, aşağıdaki kodda görülebilir:

public class InterfaceAndAbstractClassDemo{
    public static void main(String args[]){
        
        Dog dog = new Dog("Jack",16);
        Cat cat = new Cat("Joe",20);
            
        System.out.println("Dog:"+dog);
        System.out.println("Cat:"+cat);
        
        dog.remember();
        dog.protectOwner();
        Learn dl = dog;
        dl.learn();
                
        cat.remember();
        cat.protectOwner();
        
        Climb c = cat;
        c.climb();
        
        Man man = new Man("Ravindra",40);
        System.out.println(man);
        
        Climb cm = man;
        cm.climb();
        Think t = man;
        t.think();
        Learn l = man;
        l.learn();
        Apply a = man;
        a.apply();
    }
}

abstract class Animal{
    String name;
    int lifeExpentency;
    public Animal(String name,int lifeExpentency ){
        this.name = name;
        this.lifeExpentency=lifeExpentency;
    }
    public abstract void remember();
    public abstract void protectOwner();
    
    public String toString(){
        return this.getClass().getSimpleName()+":"+name+":"+lifeExpentency;
    }
}
class Dog extends Animal implements Learn{
    
    public Dog(String name,int age){
        super(name,age);
    }
    public void remember(){
        System.out.println(this.getClass().getSimpleName()+" can remember for 5 minutes");
    }
    public void protectOwner(){
        System.out.println(this.getClass().getSimpleName()+ " will protect owner");
    }
    public void learn(){
        System.out.println(this.getClass().getSimpleName()+ " can learn:");
    }
}
class Cat extends Animal implements Climb {
    public Cat(String name,int age){
        super(name,age);
    }
    public void remember(){
        System.out.println(this.getClass().getSimpleName() + " can remember for 16 hours");
    }
    public void protectOwner(){
        System.out.println(this.getClass().getSimpleName()+ " won't protect owner");
    }
    public void climb(){
        System.out.println(this.getClass().getSimpleName()+ " can climb");
    }
}
interface Climb{
    void climb();
}
interface Think {
    void think();
}

interface Learn {
    void learn();
}
interface Apply{
    void apply();
}

class Man implements Think,Learn,Apply,Climb{
    String name;
    int age;

    public Man(String name,int age){
        this.name = name;
        this.age = age;
    }
    public void think(){
        System.out.println("I can think:"+this.getClass().getSimpleName());
    }
    public void learn(){
        System.out.println("I can learn:"+this.getClass().getSimpleName());
    }
    public void apply(){
        System.out.println("I can apply:"+this.getClass().getSimpleName());
    }
    public void climb(){
        System.out.println("I can climb:"+this.getClass().getSimpleName());
    }
    public String toString(){
        return "Man :"+name+":Age:"+age;
    }
}

Önemli notlar:

  1. “Animal”, paylaşılan öznitelikleri olan soyut bir sınıftır: “name” ve “lifeExpectancy” ve soyut yöntemler: “remember()” ve “protectOwner()”. “Köpek” ve “Kedi”, “remember()” ve “protectOwner()” yöntemlerini uygulayan “Hayvanlar"dır.

  2. ‘Kedi’ ’tırmanabilir ()’ ancak ‘Köpek’ yapamaz. “Köpek” “düşünebilir ()” ama “Kedi” yapamaz. Bu özel yetenekler, uygulama ile “Kedi” ve “Köpek"e eklenir.

  3. “İnsan” bir “Hayvan” değildir, ancak “Düşünebilir”, “Öğrenebilir”, “Uygulayabilir” ve “Tırmanabilir”.

  4. ‘Kedi’ bir ‘Adam’ değildir ama ‘Tırmanabilir’.

  5. ‘Köpek’ bir ‘Adam’ değil, ‘Öğrenebilir’

  6. “İnsan” ne bir “Kedi” ne de bir “Köpek"tir, ancak “Hayvan”, “Kedi” veya “Köpek"i genişletmeden son ikisinin bazı yeteneklerine sahip olabilir. Bu, Arayüzler ile yapılır.

  7. ‘Animal’ soyut bir sınıf olmasına rağmen, arayüzün aksine bir yapıcısı vardır.

TL; DR:

İlgili olmayan sınıflar, arayüzler aracılığıyla yeteneklere sahip olabilir, ancak ilgili sınıflar, temel sınıfların genişletilmesi yoluyla davranışı değiştirir.

Belirli bir kullanım durumunda hangisinin kullanılacağını anlamak için Java belgelerine sayfa bakın.

Soyut sınıfları kullanmayı düşünün eğer…

  1. Birbiriyle yakından ilişkili birkaç sınıf arasında kod paylaşmak istiyorsunuz.
  2. Soyut sınıfınızı genişleten sınıfların birçok ortak yöntem veya alana sahip olmasını veya genel (korumalı ve özel gibi) dışında erişim değiştiricileri gerektirmesini bekliyorsunuz.
  3. Statik olmayan veya nihai olmayan alanlar bildirmek istiyorsunuz.

Arayüzleri kullanmayı düşünün eğer…

  1. İlişkisiz sınıfların arayüzünüzü gerçekleştirmesini bekliyorsunuz. Örneğin, pek çok ilişkisiz nesne ‘Serializable’ arabirimini uygulayabilir.
  2. Belirli bir veri türünün davranışını belirtmek istiyorsunuz, ancak davranışını kimin uyguladığıyla ilgilenmiyorsunuz.
  3. Birden çok tür mirasından yararlanmak istiyorsunuz.

Kalıtım ve Statik Yöntemler #

Java’da ebeveyn ve alt sınıfın her ikisi de aynı ada sahip statik yöntemlere sahip olabilir. Ancak bu gibi durumlarda, çocukta statik yöntemin uygulanması gizleniyor ana sınıf’ uygulama, yöntemi geçersiz kılma değildir. Örneğin:

class StaticMethodTest {

  // static method and inheritance
  public static void main(String[] args) {
    Parent p = new Child();
    p.staticMethod(); // prints Inside Parent
    ((Child) p).staticMethod(); // prints Inside Child
  }

  static class Parent {
    public static void staticMethod() {
      System.out.println("Inside Parent");
    }
  }

  static class Child extends Parent {
    public static void staticMethod() {
      System.out.println("Inside Child");
    }
  }
}

Statik yöntemler bir örneğe değil bir sınıfa bağlanır ve bu yöntem bağlaması derleme zamanında gerçekleşir. İlk ‘staticMethod()’ çağrısında, ‘p’ üst sınıf referansı kullanıldığından, ‘Parent’in ‘staticMethod()’ versiyonu çağrılır. İkinci durumda, ‘Child’ sınıfına ‘p’ attık, ‘Child’in ‘staticMethod()‘u çalıştırıldı.

Değişken gölgeleme #

Değişkenler SHADOWED ve yöntemler OVERRIDDEN. Hangi değişkenin kullanılacağı, değişkenin bildirildiği sınıfa bağlıdır. Hangi yöntemin kullanılacağı, değişken tarafından başvurulan nesnenin gerçek sınıfına bağlıdır.

class Car {
    public int gearRatio = 8;

    public String accelerate() {
        return "Accelerate : Car";
    }
}

class SportsCar extends Car {
    public int gearRatio = 9;

    public String accelerate() {
        return "Accelerate : SportsCar";
    }

    public void test() {

    }


    public static void main(String[] args) {

        Car car = new SportsCar();
        System.out.println(car.gearRatio + "  " + car.accelerate());
        // will print out 8  Accelerate : SportsCar
    }
}

Nesne referanslarının daraltılması ve genişletilmesi #

Temel sınıfın bir örneğini ‘b = (B) a;’ biçiminde olduğu gibi bir alt sınıfa yayınlamaya daraltma denir (temel sınıf nesnesini daha belirli bir sınıf nesnesine daraltmaya çalıştığınız için) ve açık bir türe ihtiyaç duyar -oyuncu kadrosu.

Bir alt sınıfın örneğini bir temel sınıfa şu şekilde yayınlamak: A a = b;, genişletme olarak adlandırılır ve bir tip-kayıt gerektirmez.

Örneklemek için, aşağıdaki sınıf bildirimlerini göz önünde bulundurun ve kodu test edin:

class Vehicle {
}

class Car extends Vehicle {
}

class Truck extends Vehicle {
}

class MotorCycle extends Vehicle {
}

class Test {

    public static void main(String[] args) {
    
        Vehicle vehicle = new Car();
        Car car = new Car();        
    
        vehicle = car; // is valid, no cast needed

        Car c = vehicle // not valid
        Car c = (Car) vehicle; //valid
    }
}

Vehicle Vehicle = new Car(); ifadesi geçerli bir Java ifadesidir. “Araba"nın her örneği aynı zamanda bir “Araç"tır. Bu nedenle, atama, açık bir tip dökümüne ihtiyaç duymadan yasaldır.

Öte yandan, Car c = araç; geçerli değildir. “Vehicle” değişkeninin statik türü “Vehicle"dır; bu, “Car”, Truck”, “MotorCycle” örneğine veya “Vehicle"ın herhangi bir mevcut veya gelecekteki alt sınıfına atıfta bulunabileceği anlamına gelir. (Ya da aslında, onu bir “soyut” sınıf olarak bildirmediğimiz için “Araç"ın kendisinin bir örneği.) Bu atamaya izin verilemez, çünkü bu, bir “Kamyon” örneğine atıfta bulunan “araba"ya yol açabilir.

Bu durumu önlemek için açık bir type-cast eklememiz gerekiyor:

Car c = (Car) vehicle;

Tip dökümü derleyiciye, “araç” değerinin bir “Araba” veya “Araba"nın bir alt sınıfı olmasını beklediğimizi söyler. Gerekirse, derleyici bir çalışma zamanı türü denetimi gerçekleştirmek için kod ekler. Kontrol başarısız olursa, kod yürütüldüğünde bir ClassCastException atılır.

Tüm tip tahminlerinin geçerli olmadığını unutmayın. Örneğin:

String s = (String) vehicle;  // not valid

Java derleyicisi, ‘Vehicle’ ile tip uyumlu bir örneğin asla ‘String’ ile uyumlu olamayacağını bilir. Tip dökümü asla başarılı olamaz ve JLS, bunun bir derleme hatası vermesini zorunlu kılar.

Bir arayüze programlama #

Bir arabirime programlamanın ardındaki fikir, kodu öncelikle arabirimlere dayandırmak ve somutlaştırma sırasında yalnızca somut sınıfları kullanmaktır. Bu bağlamda, örn. Java koleksiyonları şöyle görünecek (yöntemin kendisinin herhangi bir kullanımı olduğundan değil, sadece örnekleme):

public <T> Set<T> toSet(Collection<T> collection) {
  return Sets.newHashSet(collection);
}

kötü kod şöyle görünebilir:

public <T> HashSet<T> toSet(ArrayList<T> collection) {
  return Sets.newHashSet(collection);
}

Yalnızca birincisi daha geniş bir argüman seçimine uygulanamaz, sonuçları genellikle bir arayüze programlama kavramına bağlı olan diğer geliştiriciler tarafından sağlanan kodla daha uyumlu olacaktır. Ancak, ilkini kullanmanın en önemli nedenleri şunlardır:

  • Çoğu zaman sonucun kullanıldığı bağlam, somut uygulamanın sağladığı kadar çok ayrıntıya ihtiyaç duymaz ve gerekmemelidir;
  • bir arabirime bağlı kalmak, daha temiz kodu zorlar ve belirli bir senaryoya hizmet eden bir sınıfa başka bir genel yöntem gibi daha az saldırı eklenir;
  • arayüzler kolayca taklit edilebilir olduğundan kod daha test edilebilir;
  • son olarak, konsept yalnızca bir uygulama beklense bile yardımcı olur (en azından test edilebilirlik için).

Öyleyse, belirli bir uygulamayı akılda tutarak yeni kod yazarken programlama kavramını bir arayüze nasıl kolayca uygulayabilirsiniz? Yaygın olarak kullandığımız seçeneklerden biri, aşağıdaki kalıpların birleşimidir:

  • bir arayüze programlama
  • fabrika
  • inşaatçı

Bu ilkelere dayanan aşağıdaki örnek, bir dizi farklı protokol için yazılmış bir RPC uygulamasının basitleştirilmiş ve kısaltılmış bir versiyonudur:

public interface RemoteInvoker {
  <RQ, RS> CompletableFuture<RS> invoke(RQ request, Class<RS> responseClass);
}

Yukarıdaki arayüzün doğrudan bir fabrika aracılığıyla somutlaştırılması gerekmiyor, bunun yerine daha somut arayüzler türetiyoruz, biri HTTP çağırma için ve diğeri AMQP için, her biri daha sonra örnekleri oluşturmak için bir fabrikaya ve bir oluşturucuya sahip, bu da sırasıyla örnekleridir. yukarıdaki arayüz:

public interface AmqpInvoker extends RemoteInvoker {
  static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
    return new AmqpInvokerBuilder(instanceId, factory);
  }
}

AMQP ile kullanım için “RemoteInvoker” örnekleri artık şu kadar kolay (veya oluşturucuya bağlı olarak daha fazla dahil) oluşturulabilir:

RemoteInvoker invoker = AmqpInvoker.with(instanceId, factory)
  .requestRouter(router)
  .build();

Ve bir isteğin çağrılması şu kadar kolaydır:

Response res = invoker.invoke(new Request(data), Response.class).get();

Java 8’in statik yöntemlerin doğrudan arayüzlere yerleştirilmesine izin vermesi nedeniyle, ara fabrika, AmqpInvoker.with() ile değiştirilen yukarıdaki kodda örtük hale geldi. Sürüm 8’den önceki Java’da, aynı etki bir iç “Factory” sınıfıyla elde edilebilir:

public interface AmqpInvoker extends RemoteInvoker {
  class Factory {
    public static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
      return new AmqpInvokerBuilder(instanceId, factory);
    }
  }
}

Karşılık gelen örnekleme daha sonra şuna dönüşür:

RemoteInvoker invoker = AmqpInvoker.Factory.with(instanceId, factory)
  .requestRouter(router)
  .build();

Yukarıda kullanılan oluşturucu şöyle görünebilir (gerçek olanın varsayılanlardan sapan 15 adede kadar parametrenin tanımlanmasına izin verdiği için bu bir basitleştirme olmasına rağmen). Yapının herkese açık olmadığını unutmayın, bu nedenle yalnızca yukarıdaki “AmqpInvoker” arabiriminden özellikle kullanılabilir:

public class AmqpInvokerBuilder {
  ...
  AmqpInvokerBuilder(String instanceId, ConnectionFactory factory) {
    this.instanceId = instanceId;
    this.factory = factory;
  }

  public AmqpInvokerBuilder requestRouter(RequestRouter requestRouter) {
    this.requestRouter = requestRouter;
    return this;
  }

  public AmqpInvoker build() throws TimeoutException, IOException {
    return new AmqpInvokerImpl(instanceId, factory, requestRouter);
  }
}

Genel olarak, bir oluşturucu, FreeBuilder gibi bir araç kullanılarak da oluşturulabilir.

Son olarak, bu arabirimin standart (ve tek beklenen) uygulaması, arabirimin, fabrikanın ve oluşturucunun kullanımını zorunlu kılmak için yerel paket sınıfı olarak tanımlanır:

class AmqpInvokerImpl implements AmqpInvoker {
  AmqpInvokerImpl(String instanceId, ConnectionFactory factory, RequestRouter requestRouter) {
    ...
  }

  @Override
  public <RQ, RS> CompletableFuture<RS> invoke(final RQ request, final Class<RS> respClass) {
    ...
  }
}

Bu arada, bu model, işlevsellik ne kadar basit veya karmaşık olursa olsun, tüm yeni kodlarımızı geliştirmede çok verimli olduğunu kanıtladı.

Kalıtımda Geçersiz Kılma #

Kalıtımda geçersiz kılma, bir alt sınıftaki bir süper sınıftan önceden tanımlanmış bir yöntemi kullandığınızda, ancak yöntemin süper sınıfta orijinal olarak tasarlanmasından farklı bir şekilde kullandığınızda kullanılır. Geçersiz kılma, kullanıcının mevcut materyali kullanarak ve kullanıcının ihtiyaçlarına daha iyi uyacak şekilde değiştirerek kodu yeniden kullanmasına olanak tanır.

Aşağıdaki örnek, “SınıfB"nin, yazdırma yöntemiyle gönderilenleri değiştirerek “SınıfA"nın işlevselliğini nasıl geçersiz kıldığını gösterir:

Örnek:

public static void main(String[] args) {
    ClassA a = new ClassA();
    ClassA b = new ClassB();
    a.printing();
    b.printing();
}

class ClassA {
    public void printing() {        
        System.out.println("A");
    }
}

class ClassB extends ClassA {
    public void printing() {
         System.out.println("B");
    }
}
