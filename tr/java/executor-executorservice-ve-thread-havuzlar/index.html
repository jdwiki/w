Java’daki Executor arabirimi, iş parçacığı kullanımı, zamanlama vb. ayrıntıları da dahil olmak üzere, her bir görevin nasıl çalıştırılacağının mekaniklerinden görev gönderimini ayırmanın bir yolunu sağlar. Normalde, açıkça iş parçacıkları oluşturmak yerine bir Yürütücü kullanılır. Yürütücüler ile geliştiricilerin, programlarının görev yürütme politikasını kolayca ayarlayabilmeleri için kodlarını önemli ölçüde yeniden yazmaları gerekmeyecek.

Tuzaklar

  • Bir görevi tekrarlanan yürütme için zamanladığınızda, kullanılan ScheduledExecutorService’e bağlı olarak, görevinizin yürütülmesi işlenmeyen bir özel duruma neden olursa, göreviniz daha fazla yürütmeden askıya alınabilir. Bkz. Mother F**k the ScheduledExecutorService!

ThreadPoolExecutor #

Kullanılan yaygın bir Yürütücü, İplik işleme ile ilgilenen ‘ThreadPoolExecutor’dur. Yapacak fazla bir şey olmadığında (buna çekirdek boyutu denir) yürütücünün her zaman sürdürmesi gereken minimum İş parçacığı miktarını ve yapılacak daha fazla iş varsa Havuzun büyüyebileceği maksimum İş parçacığı boyutunu yapılandırabilirsiniz. İş yükü azaldığında, Havuz, minimum boyuta ulaşana kadar İplik sayısını tekrar yavaş yavaş azaltır.

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    1,                                     // keep at least one thread ready, 
                                           // even if no Runnables are executed
    5,                                     // at most five Runnables/Threads
                                           // executed in parallel
    1, TimeUnit.MINUTES,                   // idle Threads terminated after one
                                           // minute, when min Pool size exceeded
    new ArrayBlockingQueue<Runnable>(10)); // outstanding Runnables are kept here

pool.execute(new Runnable() {
    @Override public void run() {
        //code to run
    }
});

Not ‘ThreadPoolExecutor’ı bir unbounded kuyruğu ile yapılandırırsanız, yeni iş parçacıkları yalnızca sıra doluysa oluşturulduğundan, iş parçacığı sayısı ‘corePoolSize’ değerini aşmaz:

Tüm parametrelerle ThreadPoolExecutor:

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

JavaDoc‘den

corePoolSize’den fazla ancak maksimumPoolSize’den daha az iş parçacığı çalışıyorsa, yalnızca sıra doluysa yeni bir iş parçacığı oluşturulur.

Avantajlar:

  1. BlockingQueue boyutu kontrol edilebilir ve yetersiz bellek senaryoları önlenebilir. Sınırlı sınırlı kuyruk boyutu ile uygulama performansı düşmeyecektir.

  2. Mevcut veya yeni Reddetme İşleyicisi ilkeleri oluşturabilirsiniz.

  3. Varsayılan ThreadPoolExecutor.AbortPolicy’de, işleyici reddedildikten sonra bir RejectedExecutionException çalışma zamanı atar.

  4. ThreadPoolExecutor.CallerRunsPolicy içinde, çağıran iş parçacığı görevi kendisi yürütür. Bu, yeni görevlerin gönderilme hızını yavaşlatacak basit bir geri bildirim kontrol mekanizması sağlar.

  5. ThreadPoolExecutor.DiscardPolicy içinde, yürütülemeyen bir görev basitçe bırakılır.

  6. ‘ThreadPoolExecutor.DiscardOldestPolicy’ içinde, yürütücü kapatılmazsa, iş kuyruğunun başındaki görev bırakılır ve ardından yürütme yeniden denenir (bu, tekrar başarısız olabilir ve bunun tekrarlanmasına neden olabilir.)

  7. Özel ‘ThreadFactory’ yapılandırılabilir, bu da yararlıdır:

  8. Daha açıklayıcı bir iş parçacığı adı ayarlamak için

  9. İş parçacığı arka plan programı durumunu ayarlamak için

  10. İplik önceliğini ayarlamak için

Hesaplamadan değer alma - Çağrılabilir #

Hesaplamanız daha sonra gerekli olan bir dönüş değeri üretiyorsa, basit bir Çalıştırılabilir görev yeterli değildir. Bu gibi durumlarda, yürütme tamamlandıktan sonra bir değer döndüren ExecutorService.submit([Callable][callable-and-future]<T>) kullanabilirsiniz.

Hizmet, görev yürütmenin sonucunu almak için kullanabileceğiniz bir [‘Gelecek’][çağrılabilir-ve-gelecek] döndürür.

// Submit a callable for execution
ExecutorService pool = anExecutorService;
Future<Integer> future = pool.submit(new Callable<Integer>() {
    @Override public Integer call() {
        //do some computation
        return new Random().nextInt();
    }
});    
// ... perform other tasks while future is executed in a different thread

Geleceğin sonucunu almanız gerektiğinde, future.get() çağırın.

  • Geleceğin bir sonuçla bitmesini süresiz olarak bekleyin.

      try {
      // Blocks current thread until future is completed
      Integer result = future.get(); 
  catch (InterruptedException || ExecutionException e) {
      // handle appropriately
  }

  • Geleceğin bitmesini bekleyin, ancak belirtilen süreden fazla değil.

      try {
      // Blocks current thread for a maximum of 500 milliseconds.
      // If the future finishes before that, result is returned,
      // otherwise TimeoutException is thrown.
      Integer result = future.get(500, TimeUnit.MILLISECONDS); 
  catch (InterruptedException || ExecutionException || TimeoutException e) {
      // handle appropriately
  }

Zamanlanmış veya çalışan bir görevin sonucu artık gerekli değilse, onu iptal etmek için Future.cancel(boolean) öğesini arayabilirsiniz.

  • ‘cancel(false)’ çağrısı, görevi çalıştırılacak görevler sırasından kaldırır.
  • ‘cancel(true)’ çağrısı, o anda çalışıyorsa görevi ayrıca kesecektir.

[çağrılabilir-ve-gelecek]: https://www.wikiod.com/tr/java/eszamanl-programlama-is-parcacg#Çağrılabilir ve Gelecek

gönder() ve yürütme() istisna işleme farklılıkları #

Genellikle yürüt ve unut çağrıları için (sonucu analiz etmeye gerek kalmadan) execute() komutu, Future nesnesinin sonucunu analiz etmek için ise gönder() komutu kullanılır.

Bu iki komut arasındaki İstisna İşleme mekanizmalarının temel farkının farkında olmalıyız.

Gönderme() işlevinden gelen istisnalar, onları yakalamadıysanız çerçeve tarafından yutulur.

Farkı anlamak için kod örneği:

Durum 1: İstisnayı bildiren Runnable with execute() komutunu gönderin.

import java.util.concurrent.*;
import java.util.*;

public class ExecuteSubmitDemo {
    public ExecuteSubmitDemo() {
        System.out.println("creating service");
        ExecutorService service = Executors.newFixedThreadPool(2);
        //ExtendedExecutor service = new ExtendedExecutor();
        for (int i = 0; i < 2; i++){
            service.execute(new Runnable(){
                 public void run(){
                    int a = 4, b = 0;
                    System.out.println("a and b=" + a + ":" + b);
                    System.out.println("a/b:" + (a / b));
                    System.out.println("Thread Name in Runnable after divide by zero:"+Thread.currentThread().getName());
                 }
            });
        }
        service.shutdown();
    }
    public static void main(String args[]){
        ExecuteSubmitDemo demo = new ExecuteSubmitDemo();
    }
}

class ExtendedExecutor extends ThreadPoolExecutor {

   public ExtendedExecutor() { 
       super(1, 1, 60, TimeUnit.SECONDS, new ArrayBlockingQueue<Runnable>(100));
   }
   // ...
   protected void afterExecute(Runnable r, Throwable t) {
     super.afterExecute(r, t);
     if (t == null && r instanceof Future<?>) {
       try {
         Object result = ((Future<?>) r).get();
       } catch (CancellationException ce) {
           t = ce;
       } catch (ExecutionException ee) {
           t = ee.getCause();
       } catch (InterruptedException ie) {
           Thread.currentThread().interrupt(); // ignore/reset
       }
     }
     if (t != null)
       System.out.println(t);
   }
 }

Ateş Et ve Unut - Çalıştırılabilir Görevler #

Yürütücüler, başka bir Thread’de çalıştırılmak üzere (potansiyel olarak hesaplamalı veya başka bir şekilde uzun süreli veya ağır) kod içeren bir “java.lang.Runnable” kodunu kabul eder.

Kullanım şöyle olacaktır:

Executor exec = anExecutor;
exec.execute(new Runnable() {
    @Override public void run() {
        //offloaded work, no need to get result back
    }
});

Bu yürütücü ile herhangi bir hesaplanmış değeri geri alma imkanınız olmadığını unutmayın. Java 8 ile kod örneğini kısaltmak için lambdalar kullanılabilir.

Executor exec = anExecutor;
exec.execute(() -> {
    //offloaded work, no need to get result back
});

Eğer

  1. Görevleri bir kapatma Yürütücüsüne göndermeye çalışıyorsunuz veya
  2. sıra doygun (sadece sınırlı olanlarla mümkündür) ve maksimum İş parçacığı sayısına ulaşıldı,

RejectedExecutionHandler.rejectedExecution(Runnable, ThreadPoolExecutor) çağrılacak.

Varsayılan davranış, arayana atılan bir RejectedExecutionException almanızdır. Ancak daha önceden tanımlanmış davranışlar mevcuttur:

  • ThreadPoolExecutor.AbortPolicy (varsayılan, REE’yi atar)
  • ThreadPoolExecutor.CallerRunsPolicy (görevi arayanın iş parçacığında yürütür - blocking it)
  • ThreadPoolExecutor.DiscardPolicy (görevi sessizce iptal et)
  • ThreadPoolExecutor.DiscardOldestPolicy (kuyruktaki en eski görevi sessizce atın ve yeni görevin yürütülmesini yeniden deneyin)

Bunları ThreadPool constructors‘den birini kullanarak ayarlayabilirsiniz:

Farklı eşzamanlılık yapıları için vakaları kullanın #

  1. ExecutorService

ExecutorService yürütücüsü = Executors.newFixedThreadPool(50);

Basit ve kullanımı kolaydır. ThreadPoolExecutorun düşük seviye detaylarını gizler.

‘Çağrılabilir/Çalıştırılabilir’ görevlerin sayısının az olduğu ve görevlerin sınırsız kuyrukta yığılmasının belleği artırmadığı ve sistemin performansını düşürmediği durumlarda bunu tercih ederim. “CPU/Memory” kısıtlamalarınız varsa, görevlerin reddini işlemek için kapasite kısıtlamalarıyla “ThreadPoolExecutor” ve “RejectedExecutionHandler” kullanmayı tercih ederim.

  1. CountDownLatch

CountDownLatch belirli bir sayı ile başlatılacaktır. Bu sayı, countDown() yöntemine yapılan çağrılarla azaltılır. Bu sayının sıfıra ulaşmasını bekleyen iş parçacıkları ‘await()’ yöntemlerinden birini çağırabilir. ‘await()’ çağrısı, sayım sıfıra ulaşana kadar iş parçacığını engeller. Bu sınıf, bir java iş parçacığının diğer iş parçacığı kümesi görevlerini tamamlayana kadar beklemesini sağlar.

Kullanım durumları:

  1. Maksimum Paralellik Elde Etme: Bazen maksimum paralellik elde etmek için aynı anda birkaç iş parçacığı başlatmak isteriz.

  2. Yürütmeye başlamadan önce N iş parçacığının tamamlanmasını bekleyin

  3. Kilitlenme algılama.

  4. ThreadPoolExecutor : Daha fazla kontrol sağlar. Uygulama bekleyen Çalıştırılabilir/Çağrılabilir görevlerin sayısıyla sınırlıysa, maksimum kapasiteyi ayarlayarak sınırlı kuyruğu kullanabilirsiniz. Kuyruk maksimum kapasiteye ulaştığında RejectionHandler’ı tanımlayabilirsiniz. Java, dört tür “RejectedExecutionHandler” politikaları sağlar.

      1. `ThreadPoolExecutor.AbortPolicy`, the handler throws a runtime RejectedExecutionException upon rejection.

    1. ThreadPoolExecutor.CallerRunsPolicy`, the thread that invokes execute itself runs the task. This provides a simple feedback control mechanism that will slow down the rate that new tasks are submitted.

    2. In ThreadPoolExecutor.DiscardPolicy, a task that cannot be executed is simply dropped.

    3. ThreadPoolExecutor.DiscardOldestPolicy, if the executor is not shut down, the task at the head of the work queue is dropped, and then execution is retried (which can fail again, causing this to be repeated.)

‘CountDownLatch’ davranışını simüle etmek istiyorsanız, ‘invokeAll()’ yöntemini kullanabilirsiniz.

  1. Alıntı yapmadığınız bir mekanizma daha ForkJoinPool

“ForkJoinPool” Java 7’de Java’ya eklendi. “ForkJoinPool” şuna benzer: Java ExecutorService ancak bir farkla. ‘ForkJoinPool’ yapar görevlerin işlerini daha sonra daha küçük görevlere bölmeleri kolaydır. ‘ForkJoinPool’a da gönderildi. Serbest çalışan iş parçacıkları meşgul çalışan iş parçacığı kuyruğundan görevleri çaldığında ‘ForkJoinPool’da görev çalma gerçekleşir.

Java 8, iş çalma havuzu oluşturmak için ExecutorService içinde bir API daha tanıttı. ‘RecursiveTask’ ve ‘RecursiveAction’ oluşturmanız gerekmez, ancak yine de ‘ForkJoinPool’u kullanabilirsiniz.

   public static ExecutorService newWorkStealingPool()

Hedef paralellik düzeyi olarak mevcut tüm işlemcileri kullanarak iş çalan bir iş parçacığı havuzu oluşturur.

Varsayılan olarak, parametre olarak CPU çekirdeği sayısını alacaktır.

Bu dört mekanizma birbirinin tamamlayıcısıdır. Kontrol etmek istediğiniz ayrıntı düzeyine bağlı olarak, doğru olanları seçmelisiniz.

ExecutorService’deki tüm görevlerin tamamlanmasını bekleyin #

Executor’e gönderilen görevlerin tamamlanmasını beklemek için çeşitli seçeneklere bir göz atalım.

  1. ExecutorService invokeAll()

Verilen görevleri yürütür, her şey tamamlandığında durumlarını ve sonuçlarını tutan bir Futures listesi döndürür.

Örnek:

import java.util.concurrent.*;
import java.util.*;

public class InvokeAllDemo{
    public InvokeAllDemo(){
        System.out.println("creating service");
        ExecutorService service = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
        
        List<MyCallable> futureList = new ArrayList<MyCallable>();
        for (int i = 0; i < 10; i++){
            MyCallable myCallable = new MyCallable((long)i);
            futureList.add(myCallable);
        }
        System.out.println("Start");
        try{
            List<Future<Long>> futures = service.invokeAll(futureList);  
        } catch(Exception err){
            err.printStackTrace();
        }
        System.out.println("Completed");
        service.shutdown();
    }
    public static void main(String args[]){
        InvokeAllDemo demo = new InvokeAllDemo();
    }
    class MyCallable implements Callable<Long>{
        Long id = 0L;
        public MyCallable(Long val){
            this.id = val;
        }
        public Long call(){
            // Add your business logic
            return id;
        }
    }
}

  1. CountDownLatch

    A synchronization aid that allows one or more threads to wait until a set of operations being performed in other threads completes.

    A CountDownLatch is initialized with a given count. The await methods block until the current count reaches zero due to invocations of the countDown() method, after which all waiting threads are released and any subsequent invocations of await return immediately. This is a one-shot phenomenon – the count cannot be reset. If you need a version that resets the count, consider using a CyclicBarrier.

  2. Executors içindeki ForkJoinPool veya newWorkStealingPool()

  3. ‘ExecutorService’e gönderdikten sonra oluşturulan tüm ‘Future’ nesnelerini yineleyin

  4. ExecutorService’nın Oracle dokümantasyon sayfasından önerilen kapatma yöntemi:

    void shutdownAndAwaitTermination(ExecutorService pool) {
    pool.shutdown(); // Disable new tasks from being submitted
    try {
      // Wait a while for existing tasks to terminate
      if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
        pool.shutdownNow(); // Cancel currently executing tasks
        // Wait a while for tasks to respond to being cancelled
        if (!pool.awaitTermination(60, TimeUnit.SECONDS))
            System.err.println("Pool did not terminate");
      }
    } catch (InterruptedException ie) {
      // (Re-)Cancel if current thread also interrupted
      pool.shutdownNow();
      // Preserve interrupt status
      Thread.currentThread().interrupt();
    }

    shutdown(): Initiates an orderly shutdown in which previously submitted tasks are executed, but no new tasks will be accepted.

shutdownNow():Aktif olarak yürütülen tüm görevleri durdurmaya çalışır, bekleyen görevlerin işlenmesini durdurur ve yürütülmeyi bekleyen görevlerin bir listesini döndürür.

Yukarıdaki örnekte, görevlerinizin tamamlanması daha fazla zaman alıyorsa, if koşulunu while koşulu olarak değiştirebilirsiniz.

Yer değiştirmek

   if (!pool.awaitTermination(60, TimeUnit.SECONDS))

ile birlikte

   while(!pool.awaitTermination(60, TimeUnit.SECONDS)) {
     Thread.sleep(60000);
 }

Farklı ExecutorService türleri için kullanım örnekleri #

Executors, belirli bir ihtiyaca yönelik farklı türde ThreadPools catering döndürür.

  1. genel statik ExecutorService newSingleThreadExecutor()

Sınırsız bir kuyrukta çalışan tek bir çalışan iş parçacığı kullanan bir Yürütücü oluşturur

Java belgesinin ikincisi için söylediği gibi, “newFixedThreadPool(1)” ve “newSingleThreadExecutor()” arasında bir fark vardır:

> Unlike the otherwise equivalent newFixedThreadPool(1) the returned executor is guaranteed not to be reconfigurable to use additional threads.

Which means that a `newFixedThreadPool` can be reconfigured later in the program by: `((ThreadPoolExecutor) fixedThreadPool).setMaximumPoolSize(10)`

’newSingleThreadExecutor’ için bu mümkün değil

Kullanım durumları:

1. You want to execute the submitted tasks in a sequence.
2. You need only one Thread to handle all your request

Eksileri:

1. Unbounded queue is harmful
  1. genel statik ExecutorService newFixedThreadPool(int nThreads)

Paylaşılan bir sınırsız kuyrukta çalışan sabit sayıda iş parçacığını yeniden kullanan bir iş parçacığı havuzu oluşturur. Herhangi bir noktada, çoğu nThreads iş parçacığı etkin işleme görevleri olacaktır. Tüm iş parçacıkları aktifken ek görevler gönderilirse, bir iş parçacığı uygun olana kadar kuyrukta beklerler.

Kullanım durumları:

  1. Mevcut çekirdeklerin etkin kullanımı. “nThreads” öğesini “Runtime.getRuntime().availableProcessors()” olarak yapılandırın
  2. İş parçacığı sayısının, iş parçacığı havuzundaki bir sayıyı geçmemesi gerektiğine karar verdiğinizde

Eksileri:

  1. Sınırsız kuyruk zararlıdır.

  2. genel statik ExecutorService newCachedThreadPool()

Gerektiğinde yeni diziler oluşturan ancak daha önce oluşturulmuş dizileri kullanılabilir olduklarında yeniden kullanacak olan bir ileti dizisi havuzu oluşturur

Kullanım durumları:

  1. Kısa ömürlü asenkron görevler için

Eksileri:

  1. Sınırsız kuyruk zararlıdır.

  2. Mevcut tüm iş parçacıkları meşgulse, her yeni görev yeni bir iş parçacığı oluşturacaktır. Görev uzun sürüyorsa, sistemin performansını düşürecek daha fazla sayıda iş parçacığı oluşturulacaktır. Bu durumda alternatif: “newFixedThreadPool”

  3. genel statik ScheduledExecutorService newScheduledThreadPool(int corePoolSize)

Belirli bir gecikmeden sonra çalışacak veya periyodik olarak yürütülecek komutları programlayabilen bir iş parçacığı havuzu oluşturur.

Kullanım durumları:

  1. Gelecekte belirli zaman aralıklarında meydana gelecek gecikmeli yinelenen olayları ele almak

Eksileri:

  1. Sınırsız kuyruk zararlıdır.

5.genel statik ExecutorService newWorkStealingPool()

Hedef paralellik düzeyi olarak mevcut tüm işlemcileri kullanarak iş çalan bir iş parçacığı havuzu oluşturur

Kullanım durumları:

  1. Böl ve yönet tipi problemler için.
  2. Boşta kalan dişlerin etkin kullanımı. Boşta olan iş parçacıkları, meşgul iş parçacıklarından görevleri çalar.

Eksileri:

  1. Sınırsız kuyruk boyutu zararlıdır.

Tüm bu ExecutorService’de ortak bir dezavantaj görebilirsiniz: sınırsız kuyruk. Bu, ThreadPoolExecutor ile ele alınacaktır.

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

ThreadPoolExecutor ile şunları yapabilirsiniz:

  1. İplik havuzu boyutunu dinamik olarak kontrol edin
  2. BlockingQueue kapasitesini ayarlayın
  3. Sıra dolduğunda ‘RejectionExecutionHander’ı tanımlayın
  4. Thread oluşturma sırasında bazı ek işlevler eklemek için “CustomThreadFactory” (genel Thread newThread(Runnable r)"

Görevleri belirli bir zamanda, gecikmeden sonra veya tekrar tekrar çalışacak şekilde zamanlama #

‘ScheduledExecutorService’ sınıfı, tek veya tekrarlanan görevleri çeşitli şekillerde zamanlamak için bir yöntem sağlar. Aşağıdaki kod örneği, “havuzun” bildirildiğini ve aşağıdaki gibi başlatıldığını varsayar:

ScheduledExecutorService pool = Executors.newScheduledThreadPool(2);

Normal “ExecutorService” yöntemlerine ek olarak, “ScheduledExecutorService” API’si, görevleri zamanlayan ve “ScheduledFuture” nesnelerini döndüren 4 yöntem ekler. İkincisi, sonuçları almak (bazı durumlarda) ve görevleri iptal etmek için kullanılabilir.

Sabit bir gecikmeden sonra bir göreve başlama #

Aşağıdaki örnek, bir görevi on dakika sonra başlayacak şekilde zamanlar.

ScheduledFuture<Integer> future = pool.schedule(new Callable<>() {
        @Override public Integer call() {
            // do something
            return 42;
        }
    }, 
    10, TimeUnit.MINUTES);

Görevleri sabit bir oranda başlatma #

Aşağıdaki örnek, bir görevi on dakika sonra başlayacak ve ardından dakikada bir olmak üzere art arda başlayacak şekilde zamanlar.

ScheduledFuture<?> future = pool.scheduleAtFixedRate(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

Görev yürütme, “havuz” kapatılana, “gelecek” iptal edilene veya görevlerden biri bir istisna ile karşılaşana kadar programa göre devam eder.

Belirli bir “scheduledAtFixedRate” çağrısı tarafından zamanlanan görevlerin zamanla çakışmayacağı garanti edilir. Bir görev öngörülen süreden daha uzun sürerse, bir sonraki ve sonraki görev yürütmeleri geç başlayabilir.

Görevleri sabit bir gecikmeyle başlatma #

Aşağıdaki örnek, bir görevi on dakika sonra başlayacak ve ardından bir görevin bitmesi ile bir sonrakinin başlaması arasında bir dakikalık bir gecikmeyle art arda başlayacak şekilde zamanlar.

ScheduledFuture<?> future = pool.scheduleWithFixedDelay(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

Görev yürütme, “havuz” kapatılana, “gelecek” iptal edilene veya görevlerden biri bir istisna ile karşılaşana kadar programa göre devam eder.
