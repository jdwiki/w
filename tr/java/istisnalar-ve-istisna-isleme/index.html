  • denemek {

    }

    }

  • en sonunda {

try-catch ile bir istisna yakalama #

try...catch ifadesi kullanılarak bir istisna yakalanabilir ve işlenebilir. (Aslında try ifadeleri, try...catch...finally ve try-with-resources ile ilgili diğer örneklerde açıklandığı gibi başka biçimler alır.)

Tek bir yakalama bloğu ile try-catch #

En basit form şöyle görünür:

try {
    doSomething();
} catch (SomeException e) {
    handle(e);
}
// next statement

Basit bir “try…catch” davranışı aşağıdaki gibidir:

  • try bloğundaki ifadeler yürütülür.
  • try bloğundaki ifadeler tarafından herhangi bir istisna atılmazsa, kontrol try...catch den sonraki deyime geçer.
  • try bloğu içinde bir istisna atılırsa.
    • The exception object is tested to see if it is an instance of SomeException or a subtype.
    • If it is, then the catch block will catch the exception:
      • The variable e is bound to the exception object.
      • The code within the catch block is executed.
      • If that code throws an exception, then the newly thrown exception is propagated in place of the original one.
      • Otherwise, control passes to the next statement after the try...catch.
    • If it is not, the original exception continues to propagate.

Birden fazla yakalama ile deneyin-yakala #

Bir “try…catch” birden çok “catch” bloğuna da sahip olabilir. Örneğin:

try {
    doSomething();
} catch (SomeException e) {
    handleOneWay(e)
} catch (SomeOtherException e) {
    handleAnotherWay(e);
}
// next statement

Birden fazla “catch” bloğu varsa, istisna için bir eşleşme bulunana kadar ilkinden başlayarak teker teker denenirler. Karşılık gelen işleyici yürütülür (yukarıdaki gibi) ve ardından kontrol, “try…catch” ifadesinden sonraki bir sonraki ifadeye geçirilir. Eşleşenden sonraki “catch” blokları her zaman atlanır, işleyici kodu bir istisna atsa bile.

“Yukarıdan aşağıya” eşleştirme stratejisinin, “catch” bloklarındaki istisnaların ayrık olmadığı durumlarda sonuçları vardır. Örneğin:

try {
    throw new RuntimeException("test");
} catch (Exception e) {
    System.out.println("Exception");
} catch (RuntimeException e) {
    System.out.println("RuntimeException");
}

Bu kod parçacığı, “RuntimeException” yerine “Exception” çıktısını verir. “RuntimeException”, “İstisna"nın bir alt türü olduğundan, ilk (daha genel) “yakalama” eşleştirilecektir. İkinci (daha spesifik) “catch” asla yürütülmez.

Bundan alınacak ders, (istisna türleri açısından) en spesifik “yakalama” bloklarının önce görünmesi ve en genel olanların en son olması gerektiğidir. (Bazı Java derleyicileri, bir “yakalama” asla gerçekleştirilemezse sizi uyarır, ancak bu bir derleme hatası değildir.)

Çok özel durum yakalama blokları #

Java SE 7’den başlayarak, tek bir “catch” bloğu, ilgisiz istisnaların bir listesini işleyebilir. İstisna türü, dikey bir çubuk (|) simgesiyle ayrılmış olarak listelenir. Örneğin:

try {
    doSomething();
} catch (SomeException | SomeOtherException e) {
    handleSomeException(e);
}

Çok özel durum yakalama davranışı, tek özel durum durumunun basit bir uzantısıdır. Atılan istisna (en azından) listelenen istisnalardan biriyle eşleşirse, “catch” eşleşir.

Spesifikasyonda bazı ek incelikler var. “e” türü, listedeki istisna türlerinin sentetik bir birleşimidir. ’e’ değeri kullanıldığında, statik türü, tür birleşiminin en az yaygın olan üst türüdür. Ancak, “catch” bloğu içinde “e” yeniden oluşturulursa, atılan istisna türleri birleşimdeki türlerdir. Örneğin:

public void method() throws IOException, SQLException
    try {
        doSomething();
    } catch (IOException | SQLException e) {
        report(e);
        throw e;
    }

Yukarıda, ‘IOException’ ve ‘SQLException’, en az yaygın üst türü ‘İstisna’ olan kontrol edilen istisnalardır. Bu, “rapor” yönteminin “rapor(İstisna)” ile eşleşmesi gerektiği anlamına gelir. Ancak derleyici, “throw"un yalnızca “IOException” veya “SQLException” atabileceğini bilir. Bu nedenle, “method”, “throws Exception” yerine “throws IOException, SQLException” olarak bildirilebilir. (Bu iyi bir şey: bkz. Pitfall - Fırlatma Fırlatma, İstisna, Hata veya RuntimeException.)

Kaynaklarla deneyin ifadesi #

try-catch-final deyimi örneğinin gösterdiği gibi, bir ‘finally’ yan tümcesi kullanılarak kaynak temizleme, uç durumları doğru bir şekilde uygulamak için önemli miktarda “kazan plakası” kodu gerektirir. Java 7, kaynaklarla dene ifadesi biçiminde bu sorunla başa çıkmanın çok daha basit bir yolunu sunar.

Kaynak nedir? #

Java 7, sınıfların * kaynaklarla dene* ifadesi kullanılarak yönetilmesine izin vermek için “java.lang.AutoCloseable” arabirimini tanıttı. “AutoCloseable” uygulayan sınıf örneklerine kaynaklar denir. Bunların tipik olarak, çöp toplayıcının imha etmesine güvenmek yerine zamanında imha edilmesi gerekir.

“AutoCloseable” arabirimi tek bir yöntem tanımlar:

public void close() throws Exception

Bir ‘close()’ yöntemi, kaynağı uygun bir şekilde imha etmelidir. Belirtim, daha önce elden çıkarılmış bir kaynak üzerinde yöntemi çağırmanın güvenli olması gerektiğini belirtir. Ek olarak, “Otomatik Kapatılabilir"i uygulayan sınıflar, “İstisna"dan daha özel bir istisna oluşturmak için “kapat()” yöntemini veya hiç istisna olmadığını bildirmek için şiddetle teşvik edilir.

Çok çeşitli standart Java sınıfları ve arabirimleri ‘AutoCloseable’ı uygular. Bunlar şunları içerir:

  • “InputStream”, “OutputStream” ve bunların alt sınıfları
  • ‘Okuyucu’, ‘Yazar’ ve bunların alt sınıfları
  • ‘Socket’ ve ‘ServerSocket’ ve bunların alt sınıfları
  • “Kanal” ve alt sınıfları ve
  • JDBC ‘Connection’, ‘Statement’ ve ‘ResultSet’ ve bunların alt sınıflarını arayüzler.

Uygulama ve üçüncü taraf sınıfları da bunu yapabilir.

Temel kaynakla deneme ifadesi #

kaynaklarla dene sözdizimi klasik try-catch, try-finally ve try-catch-finally formlarına dayanır. İşte bir “temel” form örneği; yani “yakalama” veya “nihayet” içermeyen form.

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

Yönetilecek kaynaklar, try deyiminden sonra (...) bölümünde değişkenler olarak bildirilir. Yukarıdaki örnekte, bir “akım” kaynak değişkeni tanımladık ve onu yeni oluşturulan bir “PrintStream” ile başlattık.

Kaynak değişkenleri başlatıldığında, try bloğu yürütülür. Bu tamamlandığında, kaynağın sızmamasını sağlamak için otomatik olarak stream.close() çağrılır. Blok nasıl tamamlanırsa tamamlansın close() çağrısının gerçekleştiğine dikkat edin.

Geliştirilmiş kaynakla deneme ifadeleri #

try-with-resources ifadesi, Java 7 öncesi try-catch-finally sözdiziminde olduğu gibi “catch” ve “finally” bloklarıyla geliştirilebilir. Aşağıdaki kod parçacığı, “PrintStream” yapıcısının atabileceği “FileNotFoundException” ile başa çıkmak için öncekine bir “catch” bloğu ekler:

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
} catch (FileNotFoundException ex) {
    System.err.println("Cannot open the file");
} finally {
    System.err.println("All done");
}

Kaynak başlatma veya try bloğu istisna atarsa, “catch” bloğu yürütülür. “finally” bloğu, geleneksel bir try-catch-finally ifadesinde olduğu gibi her zaman yürütülür.

Yine de dikkat edilmesi gereken birkaç şey var:

  • Kaynak değişkeni “catch” ve “finally” bloklarında kapsam dışı.
  • Kaynak temizleme, ifade “catch” bloğuyla eşleşmeye çalışmadan önce gerçekleşir.
  • Otomatik kaynak temizleme bir istisna oluşturduysa, bu *catch’ bloklarından birinde yakalanabilir.

Birden çok kaynağı yönetme #

Yukarıdaki kod parçacıkları, yönetilen tek bir kaynağı gösterir. Aslında, kaynaklarla dene, tek bir ifadede birden çok kaynağı yönetebilir. Örneğin:

try (InputStream is = new FileInputStream(file1);
     OutputStream os = new FileOutputStream(file2)) {
    // Copy 'is' to 'os'
}

Bu, beklediğiniz gibi davranır. Hem “is” hem de “os”, “try” bloğunun sonunda otomatik olarak kapatılır. Dikkat edilmesi gereken birkaç nokta var:

  • Başlatmalar kod sırasında gerçekleşir ve daha sonra kaynak değişkeni başlatıcıları öncekilerin değerlerini kullanabilir.
  • Başarıyla başlatılan tüm kaynak değişkenleri temizlenecek.
  • Kaynak değişkenleri, bildirimlerinin tersine göre temizlenir.

Bu nedenle, yukarıdaki örnekte, “is”, “os"dan önce başlatılır ve ondan sonra temizlenir ve “os” başlatılırken bir istisna varsa “is” temizlenir.

Kaynakla deneme ve klasik deneme-yakala-nihai olarak eşdeğerliği

Java Dil Belirtimi, kaynakla dene formlarının davranışını klasik try-catch-finally deyimi açısından belirtir. (Tüm ayrıntılar için lütfen JLS’ye bakın.)

Örneğin, bu temel kaynakla dene :

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

bu try-catch-finally: ile eşdeğer olarak tanımlanmıştır:

// Note that the constructor is not part of the try-catch statement
PrintStream stream = new PrintStream("hello.txt");

// This variable is used to keep track of the primary exception thrown
// in the try statement. If an exception is thrown in the try block,
// any exception thrown by AutoCloseable.close() will be suppressed.
Throwable primaryException = null;

// The actual try block
try {
    stream.println("Hello world!");
} catch (Throwable t) {
    // If an exception is thrown, remember it for the finally block
    primaryException = t;
    throw t;
} finally {
    if (primaryException == null) {
        // If no exception was thrown so far, exceptions thrown in close() will
        // not be caught and therefore be passed on to the enclosing code.
        stream.close();
    } else {
        // If an exception has already been thrown, any exception thrown in
        // close() will be suppressed as it is likely to be related to the
        // previous exception. The suppressed exception can be retrieved
        // using primaryException.getSuppressed().
        try {
            stream.close();
        } catch (Throwable suppressedException) {
            primaryException.addSuppressed(suppressedException);
        }
    }
}

(JLS, gerçek ’t’ ve ‘primaryException’ değişkenlerinin normal Java koduna görünmez olacağını belirtir.)

Kaynaklarla denemenin geliştirilmiş biçimi, temel biçimle eşdeğer olarak belirtilir. Örneğin:

try (PrintStream stream = new PrintStream(fileName)) {
    stream.println("Hello world!");
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

şuna eşittir:

try {
    try (PrintStream stream = new PrintStream(fileName)) {
        stream.println("Hello world!");
    }
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

Özel İstisnalar #

Çoğu durumda, istisnalar atarken mevcut genel `Exception`` sınıflarını kullanmak kod tasarımı açısından daha basittir. Bu, özellikle yalnızca basit bir hata mesajı taşımak için istisnaya ihtiyacınız varsa geçerlidir. Bu durumda, kontrol edilen bir İstisna olmadığından RuntimeException genellikle tercih edilir. Yaygın hata sınıfları için başka istisna sınıfları mevcuttur:

  • UnsupportedOperationException - belirli bir işlem desteklenmez
  • IllegalArgumentException - bir yönteme geçersiz bir parametre değeri iletildi
  • IllegalStateException - API’niz dahili olarak asla olmaması gereken veya bunun sonucunda oluşan bir duruma ulaştı API’nizi geçersiz bir şekilde kullanmaktan

Özel bir istisna sınıfı kullanmak istediğiniz durumlar aşağıdakileri içerir:

  • Başkaları tarafından kullanılmak üzere bir API veya kitaplık yazıyorsunuz ve API’nizin kullanıcılarının API’nizdeki istisnaları özel olarak yakalamasına ve işlemesine ve bu istisnaları diğer, daha genel istisnalardan ayırt edebilmesine izin vermek istiyorsunuz .
  • Programınızın bir bölümünde yakalamak ve işlemek istediğiniz belirli bir hata türü için istisnalar atıyorsunuz ve bu hataları diğerlerinden ayırt edebilmek istiyorsunuz, daha fazlası genel hatalar.

Kontrol edilmeyen bir istisna için RuntimeException öğesini genişleterek kendi özel istisnalarınızı veya RuntimeException_ öğesinin alt sınıfı olmayan herhangi bir Exception _ işaretli istisnayı genişleterek kendi özel istisnalarınızı oluşturabilirsiniz, çünkü:

RuntimeException’ın da alt sınıfları olmayan İstisnanın alt sınıfları, kontrol edilen istisnalardır

public class StringTooLongException extends RuntimeException {
    // Exceptions can have methods and fields like other classes
    // those can be useful to communicate information to pieces of code catching
    // such an exception
    public final String value;
    public final int maximumLength;

    public StringTooLongException(String value, int maximumLength){
        super(String.format("String exceeds maximum Length of %s: %s", maximumLength, value));
        this.value = value;
        this.maximumLength = maximumLength;
    }
}

Bunlar önceden tanımlanmış istisnalar olarak kullanılabilir:

void validateString(String value){
    if (value.length() > 30){
        throw new StringTooLongException(value, 30);
    }
}

Ve alanlar, istisnanın yakalandığı ve işlendiği yerlerde kullanılabilir:

void anotherMethod(String value){
    try {
        validateString(value);
    } catch(StringTooLongException e){
        System.out.println("The string '" + e.value + 
                "' was longer than the max of " + e.maximumLength );
    }
}

Oracle’ın Java Belgelerine göre şunları unutmayın:

[…] Bir istemcinin bir istisnadan makul bir şekilde kurtulması bekleniyorsa, bunu kontrol edilen bir istisna yapın. İstemci istisnadan kurtarmak için hiçbir şey yapamıyorsa, bunu denetlenmeyen bir istisna yapın.

InterruptedException’ı İşleme #

InterruptedException kafa karıştırıcı bir canavardır - Thread.sleep() gibi görünüşte zararsız yöntemlerde ortaya çıkar, ancak yanlış kullanılması, eşzamanlı ortamlar

En temelde, bir ‘InterruptedException’ yakalanırsa bu, kodunuzun şu anda çalıştığı iş parçacığında Thread.interrupt() adında bir yerde biri olduğu anlamına gelir. “Bu benim” deme eğiliminde olabilirsiniz. kod! Onu asla bölmeyeceğim!” ve bu nedenle böyle bir şey yapın:

// Bad. Don't do this.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  // disregard
}

Ancak bu, meydana gelen “imkansız” bir olayı ele almanın tam olarak yanlış yoludur. Uygulamanızın asla bir `InterruptedException’ ile karşılaşmayacağını biliyorsanız, böyle bir olayı programınızın varsayımlarının ciddi bir ihlali olarak görmeli ve mümkün olduğunca çabuk çıkmalısınız.

“İmkansız” bir kesintiyi işlemenin doğru yolu şöyledir:

// When nothing will interrupt your code
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  throw new AssertionError(e);
}

Bu iki şey yapar; önce iş parçacığının kesme durumunu geri yükler (sanki ‘InterruptedException’ ilk etapta atılmamış gibi) ve ardından uygulamanızın temel değişmezlerinin ihlal edildiğini belirten bir ‘AssertionError’ atar. Eğer ipi asla kesmeyeceğinizi kesin olarak biliyorsanız, bu kodun içinde çalıştığı güvenlidir çünkü catch bloğuna asla ulaşılamamalıdır.

Guava’nın Uninterruptibles sınıfını kullanmak bu kalıbı basitleştirmeye yardımcı olur; Uninterruptibles.sleepUninterruptably() öğesinin çağrılması, uyku süresi sona erene kadar bir iş parçacığının kesintiye uğramış durumunu dikkate almaz (bu noktada, daha sonraki çağrıların kendi ‘InterruptedException’larını incelemesi ve oluşturması için geri yüklenir). Bu tür bir kodu asla kesmeyeceğinizi biliyorsanız, bu, uyku çağrılarınızı bir try-catch bloğuna sarmanıza gerek duymaz.

Ancak daha sık olarak, iş parçacığınızın asla kesintiye uğramayacağını garanti edemezsiniz. Özellikle, bir [Yönetici][6] veya başka bir iş parçacığı yönetimi tarafından yürütülecek bir kod yazıyorsanız, kodunuzun kesintilere anında yanıt vermesi çok önemlidir, aksi takdirde uygulamanız durur ve hatta kilitlenir.

Bu gibi durumlarda yapılacak en iyi şey genellikle her yönteme sırayla bir “throws InterruptedException” ekleyerek “InterruptedException"ın çağrı yığınını yaymasına izin vermektir. Bu kulağa hoş gelmeyebilir ama aslında istenen bir özelliktir - yönteminizin imzaları şimdi arayanlara kesintilere anında yanıt vereceğini gösterir.

// Let the caller determine how to handle the interrupt if you're unsure
public void myLongRunningMethod() throws InterruptedException {
  ...
}

Sınırlı durumlarda (örneğin, kontrol edilen istisnaları “atmayan” bir yöntemi geçersiz kılarken), kesintiyi işlemek için hangi kodun yürütüleceğini bekleyerek, bir istisna oluşturmadan kesintiye uğramış durumu sıfırlayabilirsiniz. Bu, kesintiyi ele almayı geciktirir ancak tamamen bastırmaz.

// Suppresses the exception but resets the interrupted state letting later code
// detect the interrupt and handle it properly.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  return ...; // your expectations are still broken at this point - try not to do more work.
}

try catch bloğundaki ifadeleri döndür #

Kötü bir uygulama olmasına rağmen, bir istisna işleme bloğuna birden çok iade ifadesi eklemek mümkündür:

 public static int returnTest(int number){
    try{
        if(number%2 == 0) throw new Exception("Exception thrown");
        else return x;
    }
    catch(Exception e){
        return 3;
    }
    finally{
        return 7;
    }
}

Try/catch bloğuyla ilişkili nihayet bloğu herhangi bir şey döndürülmeden önce yürütüldüğünden, bu yöntem her zaman 7 değerini döndürür. Şimdi, nihayet return 7; olduğu için, bu değer, try/catch dönüş değerlerinin yerine geçer.

Eğer catch bloğu ilkel bir değer döndürürse ve bu ilkel değer daha sonra nihayet bloğunda değiştirilirse, catch bloğunda döndürülen değer döndürülecek ve final bloğundan yapılan değişiklikler yok sayılacaktır.

Aşağıdaki örnekte “1” değil “0” yazılacaktır.

public class FinallyExample {

    public static void main(String[] args) {
        int n = returnTest(4);
        
        System.out.println(n);
    }

    public static int returnTest(int number) {
        
        int returnNumber = 0; 
        
        try {
            if (number % 2 == 0)
                throw new Exception("Exception thrown");
            else
                return returnNumber;
        } catch (Exception e) {
            return returnNumber;
        } finally {
            returnNumber = 1;
        }
    }
}

Java İstisna Hiyerarşisi - Kontrol Edilmeyen ve Kontrol Edilen İstisnalar #

Tüm Java istisnaları, İstisna sınıfı hiyerarşisindeki sınıfların örnekleridir. Bu, aşağıdaki gibi temsil edilebilir:

  • java.lang.Throwable - Bu, tüm istisna sınıfları için temel sınıftır. Yöntemleri ve oluşturucuları, tüm istisnalar için ortak olan bir dizi işlevsellik uygular.
    • java.lang.Exception - This is the superclass of all normal exceptions.
      • various standard and custom exception classes.
      • java.lang.RuntimeException - This the superclass of all normal exceptions that are unchecked exceptions.
        • various standard and custom runtime exception classes.
    • java.lang.Error - This is the superclass of all “fatal error” exceptions.

Notlar:

  1. İşaretli ve işaretsiz istisnalar arasındaki fark aşağıda açıklanmıştır.
  2. ‘Throwable’, ‘Exception’ ve ‘RuntimeException’ sınıfı ‘soyut’ olarak ele alınmalıdır; https://www.wikiod.com/tr/Java/java-tuzaklar—istisna-kullanm#Pitfall - Fırlatma Atılabilir, İstisna, Hata veya RuntimeException adresine bakın.
  3. ‘Hata’ istisnaları, bir uygulamanın kurtarmaya çalışmasının güvenli olmadığı veya akıllıca olmayacağı durumlarda JVM tarafından oluşturulur.
  4. “Atılabilir"in özel alt türlerini belirtmek akıllıca olmaz. Java araçları ve kitaplıkları, “Hata” ve “İstisna"nın “Atılabilir"in yalnızca doğrudan alt türleri olduğunu varsayabilir ve bu varsayım yanlışsa hatalı davranabilir.

Kontrol Edilen ve Kontrol Edilmeyen İstisnalar #

Bazı programlama dillerinde istisna desteğinin eleştirilerinden biri, belirli bir yöntem veya prosedürün hangi istisnaları atabileceğini bilmenin zor olmasıdır. İşlenmeyen bir özel durumun bir programın çökmesine neden olabileceği göz önüne alındığında, bu, istisnaları bir kırılganlık kaynağı haline getirebilir.

Java dili, kontrol edilen istisna mekanizmasıyla bu sorunu giderir. İlk olarak Java, istisnaları iki kategoriye ayırır:

  • Kontrol edilen istisnalar, tipik olarak, bir uygulamanın başa çıkabilmesi gereken beklenen olayları temsil eder. Örneğin, “IOException” ve alt türleri, G/Ç işlemlerinde oluşabilecek hata koşullarını temsil eder. Örnekler arasında, bir dosya veya dizin mevcut olmadığı için dosyanın açılmaması, ağ bağlantısının kesilmesi nedeniyle ağ okuma ve yazma işlemlerinin başarısız olması vb.

  • Denetlenmeyen istisnalar, genellikle bir uygulamanın başa çıkamayacağı beklenmeyen olayları temsil eder. Bunlar genellikle uygulamadaki bir hatanın sonucudur.

(Aşağıda, “atılan”, açıkça (bir “atma” ifadesi tarafından) veya örtük olarak (başarısız bir başvuruda, tür atma vb.) bu, iç içe bir çağrıya atıldı ve bu çağrıya dahil edilmedi. Aşağıdaki örnek kod bunu gösterecektir.)

Kontrol edilen istisna mekanizmasının ikinci kısmı, kontrol edilen bir istisnanın meydana gelebileceği yöntemlerde kısıtlamalar olmasıdır:

  • Bir yöntemde işaretli bir istisna atıldığında veya yayıldığında, ya yöntem tarafından yakalanmalı ya da yöntemin ’throws’ yan tümcesinde listelenmelidir. (“Throws” deyiminin önemi bu örnekte açıklanmıştır.)
  • Bir başlatıcı bloğunda işaretli bir istisna atıldığında veya yayıldığında, blokta yakalanması gerekir.
  • İşaretli bir istisna, bir alan başlatma ifadesinde bir yöntem çağrısı tarafından yayılamaz. (Böyle bir istisnayı yakalamanın bir yolu yoktur.)

Kısacası, kontrol edilen bir istisna ya işlenmeli ya da bildirilmelidir.

Bu kısıtlamalar, denetlenmeyen istisnalar için geçerli değildir. Bu, bir istisnanın örtük olarak atıldığı tüm durumları içerir, çünkü tüm bu tür durumlar denetlenmeyen istisnalar atar.

Kontrol edilen istisna örnekleri #

Bu kod parçacıkları, denetlenen istisna kısıtlamalarını göstermeyi amaçlamaktadır. Her durumda, kodun derleme hatası olan bir sürümünü ve hatanın düzeltilmiş ikinci bir sürümünü gösteririz.

// This declares a custom checked exception.
public class MyException extends Exception {
    // constructors omitted.
}

// This declares a custom unchecked exception.
public class MyException2 extends RuntimeException {
    // constructors omitted.
}

İlk örnek, açıkça atılan kontrol edilen istisnaların, eğer yöntemde ele alınmamaları gerekiyorsa, nasıl “atıldı” olarak bildirilebileceğini gösterir.

// INCORRECT
public void methodThrowingCheckedException(boolean flag) {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

// CORRECTED
public void methodThrowingCheckedException(boolean flag) throws MyException {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

// INCORRECT

    ...
}

    ...
}

// CORRECTED (Version B)

  1. try bloğundaki kodu çalıştırmaya başlayın.
  2. Try bloğunda bir istisna oluşursa, hemen iptal edin ve bu istisnanın ‘catch’ bloğu tarafından yakalanıp yakalanmadığını kontrol edin (bu durumda, İstisna bir ‘ArithmeticException’ örneği olduğunda).
  3. İstisna yakalanmış ise, ’e’ değişkenine atanır ve ‘catch’ bloğu yürütülür.
  4. “try” veya “catch” bloklarından biri tamamlandıysa (yani, kod yürütme sırasında yakalanmamış istisnalar oluşmuyorsa), “try-catch” bloğunun altındaki kodu yürütmeye devam edin.

Yığın izleri oluşturma ve okuma #

Bir istisna nesnesi oluşturulduğunda (yani onu “yeni” yaptığınızda), “Throwable” yapıcısı, istisnanın oluşturulduğu bağlam hakkında bilgi yakalar. Daha sonra, bu bilgiler, ilk başta istisnaya neden olan sorunu teşhis etmeye yardımcı olmak için kullanılabilecek bir yığın izleme biçiminde çıktılanabilir.

Yığın izi yazdırma #

Bir yığın izini yazdırmak, basitçe “printStackTrace()” yöntemini çağırma meselesidir. Örneğin:

try {
    int a = 0;
    int b = 0;
    int c = a / b;
} catch (ArithmeticException ex) {
    // This prints the stacktrace to standard output
    ex.printStackTrace();
}

Bağımsız değişkenler olmadan printStackTrace() yöntemi, uygulamanın standart çıktısına yazdırılacaktır; yani mevcut ‘System.out’. Ayrıca belirtilen bir “Akış” veya “Yazıcı"ya yazdıran “printStackTrace(PrintStream)” ve “printStackTrace(PrintWriter)” aşırı yüklemeleri de vardır.

Notlar:

  1. Yığın izleme, özel durumun ayrıntılarını içermez. Bu ayrıntıları almak için toString() yöntemini kullanabilirsiniz; Örneğin.

         // Print exception and stacktrace
     System.out.println(ex);
     ex.printStackTrace();

  2. Stacktrace yazdırma dikkatli kullanılmalıdır; https://www.wikiod.com/tr/Java/java-tuzaklar—istisna-kullanm#Pitfall - Aşırı veya uygunsuz yığın izleri adresine bakın. Bir günlük kaydı çerçevesi kullanmak ve günlüğe kaydedilecek istisna nesnesini iletmek genellikle daha iyidir.

İstisna zincirleme ve iç içe yığın izleri #

Bir String olarak bir stacktrace yakalama #

Bazen, bir uygulamanın başka amaçlar için kullanılabilmesi için bir Java “Dizesi” olarak bir yığın izlemesi yakalaması gerekir. Bunu yapmak için genel yaklaşım, bir bellek içi arabelleğe yazan ve bunu “printStackTrace(…)“e ileten geçici bir “OutputStream” veya “Writer” oluşturmaktır.

Apache Commons ve Guava kitaplıkları, bir yığın izini bir Dize olarak yakalamak için yardımcı yöntemler sağlar:

org.apache.commons.lang.exception.ExceptionUtils.getStackTrace(Throwable)

com.google.common.base.Throwables.getStackTraceAsString(Throwable)

Kod tabanınızda üçüncü taraf kitaplıkları kullanamıyorsanız, görevi aşağıdaki yöntemle yapın:

   /**
     * Returns the string representation of the stack trace.
     *
     * @param throwable the throwable
     * @return the string.
     */
    public static String stackTraceToString(Throwable throwable) {
        StringWriter stringWriter = new StringWriter();
        throwable.printStackTrace(new PrintWriter(stringWriter));
        return stringWriter.toString();
    }

Amacınız yığın izini analiz etmekse, bir yığın izini ayrıştırmaya çalışmaktansa “getStackTrace()” ve “getCause()” kullanmanın daha kolay olduğunu unutmayın.

Bir istisna atma #

Aşağıdaki örnek, bir istisna oluşturmanın temellerini gösterir:

public void checkNumber(int number) throws IllegalArgumentException {
    if (number < 0) {
        throw new IllegalArgumentException("Number must be positive: " + number);
    }
}

İstisna 3. satırda atılır. Bu ifade iki kısma ayrılabilir:

  • “new IllegalArgumentException(…)”, istisnanın bildirdiği hatayı açıklayan bir mesajla “IllegalArgumentException” sınıfının bir örneğini yaratıyor.

  • throw ... daha sonra istisna nesnesini atıyor.

İstisna atıldığında, ekteki ifadelerin istisna *işlenene kadar * anormal bir şekilde * sonlandırılmasına neden olur. Bu, diğer örneklerde açıklanmıştır.

Yukarıda gösterildiği gibi, istisna nesnesini tek bir ifadede hem oluşturmak hem de atmak iyi bir uygulamadır. Programcının sorunun nedenini anlamasına yardımcı olmak için istisnaya anlamlı bir hata mesajı eklemek de iyi bir uygulamadır. Ancak, bu mutlaka son kullanıcıya göstermeniz gereken mesaj değildir. (Başlangıç ​​olarak, Java’nın istisna mesajlarını uluslararası hale getirmek için doğrudan bir desteği yoktur.)

Yapılması gereken birkaç nokta daha var:

  • ‘checkNumber’ı ’throws IllegalArgumentException’ olarak ilan ettik. ‘IllegalArgumentException’ işaretli bir istisna olduğundan, bu kesinlikle gerekli değildi; https://www.wikiod.com/tr/Java/istisnalar-ve-istisna-isleme#Java İstisna Hiyerarşisi - Kontrol Edilmeyen ve Kontrol Edilen İstisnalar adresine bakın. Ancak, bunu yapmak ve ayrıca bir yöntemin javadoc yorumlarına atılan istisnaları dahil etmek iyi bir uygulamadır.

  • Bir “throw” ifadesinin hemen ardından gelen kod ulaşılamaz. Dolayısıyla şunu yazarsak:

       throw new IllegalArgumentException("it is bad");
   return;

    the compiler would report a compilation error for the return statement.

İstisna zincirleme #

Birçok standart istisna, geleneksel “message” argümanına ek olarak ikinci bir “neden” argümanına sahip bir kurucuya sahiptir. ’neden’ istisnaları zincirlemenize izin verir. İşte bir örnek.

İlk önce, uygulamamızın kurtarılamaz bir hatayla karşılaştığında atacağı denetlenmeyen bir istisna tanımlıyoruz. Bir “neden” argümanını kabul eden bir kurucu eklediğimizi unutmayın.

    public class AppErrorException extends RuntimeException {
        public AppErrorException() {
            super();
        }

        public AppErrorException(String message) {
            super(message);
        }

        public AppErrorException(String message, Throwable cause) {
            super(message, cause);
        }
    }

Sırada, istisna zincirlemeyi gösteren bazı kodlar var.

    public String readFirstLine(String file) throws AppErrorException {
        try (Reader r = new BufferedReader(new FileReader(file))) {
            String line = r.readLine();
            if (line != null) {
                return line;
            } else {
                throw new AppErrorException("File is empty: " + file);
            }
        } catch (IOException ex) {
            throw new AppErrorException("Cannot read file: " + file, ex);
        }
    }

’try’ bloğu içindeki ’throw’ bir sorunu tespit eder ve bunu bir istisna aracılığıyla basit bir mesajla bildirir. Buna karşılık, “catch” bloğu içindeki “throw”, “IOException"ı yeni (işaretli) bir istisnaya sararak ele alıyor. Ancak, orijinal istisnayı atmıyor. ‘IOException’ı ’neden’ olarak ileterek, https://www.wikiod.com/tr/Java/istisnalar-ve-istisna-isleme açıklandığı gibi stacktrace’de yazdırılabilmesi için onu kaydederiz. 1815/yaratma-ve-okuma-yığın izleri#t=20161028160848944857.

İstisnaların gelişmiş özellikleri #

Bu örnek, bazı gelişmiş özellikleri ve İstisnalar için kullanım durumlarını kapsar.

Çağrı yığınını programlı olarak inceleme #

İstisna yığın izlerinin birincil kullanımı, programcının sorunu tanılayıp düzeltebilmesi için bir uygulama hatası ve bağlamı hakkında bilgi sağlamaktır. Bazen başka şeyler için de kullanılabilir. Örneğin, bir “SecurityManager” sınıfının, çağrı yapan kodun güvenilir olup olmadığına karar vermek için çağrı yığınını incelemesi gerekebilir.

Çağrı yığınını programlı olarak aşağıdaki gibi incelemek için istisnaları kullanabilirsiniz:

    Exception ex = new Exception();   // this captures the call stack
    StackTraceElement[] frames = ex.getStackTrace();
    System.out.println("This method is " + frames[0].getMethodName());
    System.out.println("Called from method " + frames[1].getMethodName());

Bu konuda bazı önemli uyarılar var:

  1. Bir “StackTraceElement” içinde bulunan bilgiler sınırlıdır. ‘printStackTrace’ tarafından görüntülenenden daha fazla bilgi yok. (Çerçevedeki yerel değişkenlerin değerleri mevcut değildir.)

  2. getStackTrace() için javadocs, bir JVM’nin çerçeveleri dışarıda bırakmasına izin verildiğini belirtir:

    Some virtual machines may, under some circumstances, omit one or more stack frames from the stack trace. In the extreme case, a virtual machine that has no stack trace information concerning this throwable is permitted to return a zero-length array from this method.

İstisna yapısını optimize etme #

Başka bir yerde bahsedildiği gibi, bir istisna oluşturmak oldukça pahalıdır çünkü mevcut iş parçacığındaki tüm yığın çerçeveleri hakkında bilgi yakalamayı ve kaydetmeyi gerektirir. Bazen, bu bilgilerin belirli bir istisna için asla kullanılmayacağını biliyoruz; Örneğin. stacktrace asla yazdırılmayacak. Bu durumda, bilgilerin yakalanmamasına neden olmak için özel bir istisnada kullanabileceğimiz bir uygulama hilesi vardır.

Yığın izlemeleri için gereken yığın çerçevesi bilgisi, ‘Throwable’ oluşturucuları ‘Throwable.fillInStackTrace()’ yöntemini çağırdığında yakalanır. Bu yöntem ‘genel’dir, yani bir alt sınıf onu geçersiz kılabilir. İşin püf noktası, ‘Throwable’dan miras alınan yöntemi hiçbir şey yapmayan bir yöntemle geçersiz kılmaktır; Örneğin.

  public class MyException extends Exception {
      // constructors

      @Override 
      public void fillInStackTrace() {
          // do nothing
      }
  }

Bu yaklaşımla ilgili sorun, “fillInStackTrace()” işlevini geçersiz kılan bir istisnanın yığın izini asla yakalayamaması ve buna ihtiyaç duyduğunuz senaryolarda işe yaramaz olmasıdır.

Yığın izini silme veya değiştirme #

Bazı durumlarda, normal şekilde oluşturulan bir istisnanın yığın izlemesi, ya yanlış bilgiler ya da geliştiricinin kullanıcıya açıklamak istemediği bilgiler içerir. Bu senaryolar için, bilgileri tutan “StackTraceElement” nesneleri dizisini değiştirmek için “Throwable.setStackTrace” kullanılabilir.

Örneğin, bir istisnanın yığın bilgilerini atmak için aşağıdakiler kullanılabilir:

 exception.setStackTrace(new StackTraceElement[0]);

Bastırılmış istisnalar #

Java 7, kaynaklarla dene yapısını ve ilgili istisna bastırma kavramını tanıttı. Aşağıdaki parçacığı göz önünde bulundurun:

try (Writer w = new BufferedWriter(new FileWriter(someFilename))) {
    // do stuff
    int temp = 0 / 0;    // throws an ArithmeticException
}

İstisna atıldığında, ’try’, ‘w’ üzerinde ‘close()’ öğesini çağıracak ve bu, arabelleğe alınmış herhangi bir çıktıyı temizleyecek ve ardından ‘FileWriter’ı kapatacaktır. Ancak çıktı temizlenirken bir ‘IOException’ atılırsa ne olur?

Olan şudur ki, bir kaynağı temizlerken ortaya çıkan herhangi bir istisna bastırılır. İstisna yakalanır ve birincil istisnanın bastırılmış istisna listesine eklenir. Daha sonra kaynaklarla deneme, diğer kaynakların temizlenmesiyle devam edecektir. Son olarak, birincil istisna yeniden atılacaktır.

Kaynak başlatma sırasında bir istisna atılırsa veya “try” bloğu normal şekilde tamamlanırsa benzer bir model oluşur. Atılan ilk istisna, birincil istisna olur ve temizlemeden kaynaklanan sonraki istisnalar bastırılır.

Bastırılmış istisnalar, “getSuppressedExceptions” çağrılarak birincil istisna nesnesinden alınabilir.

try-finally ve try-catch-finally ifadeleri #

try...catch...finally ifadesi, istisna işlemeyi temizleme koduyla birleştirir. ‘finally’ bloğu, her koşulda yürütülecek kodu içerir. Bu, onları kaynak yönetimi ve diğer temizleme türleri için uygun hale getirir.

sonunda dene #

İşte daha basit (try...nihally) formun bir örneği:

try {
    doSomething();  
} finally {
    cleanUp();
}

’try…finally’ davranışı aşağıdaki gibidir:

  • try bloğundaki kod yürütülür.
  • “try” bloğunda herhangi bir istisna atılmadıysa:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, control passes to the next statement after the try...finally.
  • try bloğunda bir istisna atıldıysa:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, the original exception continues to propagate.

‘finally’ bloğundaki kod her zaman yürütülür. (Yalnızca ‘System.exit(int)’ çağrılırsa veya JVM paniklerse istisnadır.) Bu nedenle, bir ‘finally’ bloğu her zaman yürütülmesi gereken doğru yer kodudur; Örneğin. dosyaları ve diğer kaynakları kapatma veya kilitleri serbest bırakma.

dene-yakala-nihayet #

İkinci örneğimiz “catch” ve “nially"nin birlikte nasıl kullanılabileceğini gösteriyor. Ayrıca kaynakları temizlemenin kolay olmadığını da gösteriyor.

// This code snippet writes the first line of a file to a string
String result = null;
Reader reader = null;
try {
    reader = new BufferedReader(new FileReader(fileName));
    result = reader.readLine();
} catch (IOException ex) {
    Logger.getLogger.warn("Unexpected IO error", ex);  // logging the exception
} finally {
    if (reader != null) {
        try {
            reader.close();
        } catch (IOException ex) {
            // ignore / discard this exception
        }
    }
}

Bu örnekte “dene…yakala…sonunda"nın (varsayımsal) davranışlarının tamamı burada açıklanamayacak kadar karmaşıktır. Basit versiyon, “finally” bloğundaki kodun her zaman çalıştırılacağıdır.

Buna kaynak yönetimi açısından bakıldığında:

  • “try” bloğundan önce “resource” (yani “reader” değişkeni) ilan ediyoruz, böylece “finally” bloğu kapsamında olacak.
  • ‘yeni DosyaOkuyucu(…)’ koyarak, ‘catch’, dosya açılırken atılan herhangi bir ‘IOError’ istisnasını işleyebilir.
  • ‘finally’ bloğunda bir ‘reader.close()‘a ihtiyacımız var, çünkü ’try’ bloğunda veya ‘catch’ bloğunda engelleyemediğimiz bazı istisna yolları var.
  • Bununla birlikte, “okuyucu” başlatılmadan önce olabilir bir istisna oluşturulduğundan, açık bir “boş” teste de ihtiyacımız var.
  • Son olarak, reader.close() çağrısı (varsayımsal olarak) bir istisna verebilir. Bununla ilgilenmiyoruz, ancak istisnayı kaynağında yakalayamazsak, çağrı yığınının daha yukarısında bununla ilgilenmemiz gerekir.

Java 7 ve sonraki sürümleri, kaynak temizlemeyi önemli ölçüde basitleştiren alternatif bir kaynaklarla deneyin sözdizimi sağlar.

Bir yöntem bildirimindeki ’throws’ yan tümcesi #

Java’nın kontrollü istisna mekanizması, programcının belirli yöntemlerin belirtilen kontrol edilmiş istisnaları atabileceğini beyan etmesini gerektirir. Bu, ’throws’ yan tümcesi kullanılarak yapılır. Örneğin:

public class OddNumberException extends Exception { // a checked exception
}

public void checkEven(int number) throws OddNumberException {
    if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

“throws OddNumberException”, “checkEven” * için yapılan bir çağrının “OddNumberException” türünde bir istisna oluşturabileceğini bildirir.

Bir “throws” yan tümcesi bir tür listesi bildirebilir ve denetlenmeyen istisnaların yanı sıra denetlenen istisnaları da içerebilir.

public void checkEven(Double number) 
        throws OddNumberException, ArithmeticException {
    if (!Double.isFinite(number)) {
        throw new ArithmeticException("INF or NaN");
    } else if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

Kontrol edilmeyen istisnaları atılmış olarak bildirmenin amacı nedir?

Bir yöntem bildirimindeki “throws” maddesi iki amaca hizmet eder:

  1. Derleyiciye hangi istisnaların atıldığını söyler, böylece derleyici yakalanmamış (kontrol edilmiş) istisnaları hata olarak bildirebilir.

  2. Yöntemi çağıran kodu yazan bir programcıya hangi istisnaların bekleneceğini söyler. Bu amaçla, genellikle bir “atar” listesine denetlenmeyen istisnaları dahil etmek mantıklıdır.

Not: “atma” listesinin, API belgeleri oluşturulurken javadoc aracı tarafından ve tipik bir IDE’nin “vurgulu metin” yöntemi ipuçları tarafından da kullanıldığını unutmayın.

Atar ve yöntemi geçersiz kılma #

‘Throws’ yan tümcesi, yöntemi geçersiz kılmak amacıyla bir yöntemin imzasının bir parçasını oluşturur. Bir geçersiz kılma yöntemi, geçersiz kılınan yöntem tarafından atılanla aynı denetlenen özel durumlar kümesiyle veya bir alt kümeyle bildirilebilir. Ancak geçersiz kılma yöntemi, fazladan kontrol edilen istisnalar ekleyemez. Örneğin:

@Override
public void checkEven(int number) throws NullPointerException // OK—NullPointerException is an unchecked exception
    ...

@Override
public void checkEven(Double number) throws OddNumberException // OK—identical to the superclass
    ...

class PrimeNumberException extends OddNumberException {}
class NonEvenNumberException extends OddNumberException {}

@Override
public void checkEven(int number) throws PrimeNumberException, NonEvenNumberException // OK—these are both subclasses

@Override
public void checkEven(Double number) throws IOExcepion         // ERROR

Bu kuralın nedeni, geçersiz kılınan bir yöntem, geçersiz kılınan yöntemin atamayacağı bir kontrol edilmiş istisna atabilirse, bunun tür ikame edilebilirliğini bozmasıdır.
