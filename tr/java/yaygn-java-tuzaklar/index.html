Tuzak: Tamsayı gibi ilkel sarmalayıcı nesnelerini karşılaştırmak için == kullanma #

(Bu tuzak, tüm ilkel sarmalayıcı türleri için eşit olarak geçerlidir, ancak bunu ‘Tamsayı’ ve ‘int’ için göstereceğiz.)

‘Tamsayı’ nesnelerle çalışırken, değerleri karşılaştırmak için ‘==’ kullanmak cezbedicidir, çünkü ‘int’ değerleriyle yapacağınız şey budur. Ve bazı durumlarda bu işe yarayacak gibi görünecektir:

Integer int1_1 = Integer.valueOf("1");
Integer int1_2 = Integer.valueOf(1);

System.out.println("int1_1 == int1_2: " + (int1_1 == int1_2));          // true
System.out.println("int1_1 equals int1_2: " + int1_1.equals(int1_2));   // true

Burada ‘1’ değerine sahip iki ‘Integer’ nesnesi oluşturduk ve bunları karşılaştırdık (Bu durumda bir ‘String’ ve bir ‘int’ değişmezinden yarattık. Başka alternatifler var). Ayrıca, iki karşılaştırma yönteminin de (’==’ ve ’eşittir’) her ikisinin de ’true’ verdiğini gözlemliyoruz.

Bu davranış, farklı değerler seçtiğimizde değişir:

Integer int2_1 = Integer.valueOf("1000");
Integer int2_2 = Integer.valueOf(1000);

System.out.println("int2_1 == int2_2: " + (int2_1 == int2_2));          // false
System.out.println("int2_1 equals int2_2: " + int2_1.equals(int2_2));   // true

Bu durumda, yalnızca “eşittir” karşılaştırması doğru sonucu verir.

Davranıştaki bu farklılığın nedeni, JVM’nin -128 ila 127 aralığı için bir “Tamsayı” nesnelerinin önbelleğini tutmasıdır. (Üst değer, “Java.lang.Integer.IntegerCache.high” sistem özelliğiyle geçersiz kılınabilir. veya JVM bağımsız değişkeni “-XX:AutoBoxCacheMax=size”). Bu aralıktaki değerler için, Integer.valueOf() yeni bir tane oluşturmak yerine önbelleğe alınan değeri döndürür.

Bu nedenle, ilk örnekte “Integer.valueOf(1)” ve “Integer.valueOf(“1”)” çağrıları aynı önbelleğe alınmış “Integer” örneğini döndürdü. Buna karşılık, ikinci örnekte “Integer.valueOf(1000)” ve “Integer.valueOf(“1000”)” hem yeni “Tamsayı” nesneleri yarattı hem de döndürdü.

Referans türleri için == operatörü, referans eşitliğini (yani aynı nesneyi) test eder. Bu nedenle, ilk örnekte int1_1 == int1_2, referanslar aynı olduğu için true'dur. İkinci örnekte referanslar farklı olduğu için int2_1 == int2_2` yanlıştır.

Tuzak: dizeleri karşılaştırmak için == kullanma #

Java’ya yeni başlayanlar için yaygın bir hata, iki dizenin eşit olup olmadığını test etmek için == operatörünü kullanmaktır. Örneğin:

public class Hello {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0] == "hello") {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

Yukarıdaki programın ilk komut satırı argümanını test etmesi ve “merhaba” kelimesi olmadığında farklı mesajlar yazdırması gerekiyor. Ama sorun şu ki, işe yaramayacak. Bu program “Bugün huysuz mu hissediyorsun?” ilk komut satırı argümanı ne olursa olsun.

Bu özel durumda, “Dize” “merhaba” dize havuzuna yerleştirilirken, “Dize” args[0] öbek üzerinde bulunur. Bu, her biri referansıyla aynı değişmezi temsil eden iki nesne olduğu anlamına gelir. == referanslar için testler gerçek eşitlik değil olduğundan, karşılaştırma çoğu zaman yanlış verir. Bu her zaman böyle olacağı anlamına gelmez.

Dizeleri test etmek için == kullandığınızda, aslında test ettiğiniz şey, iki String nesnesinin aynı Java nesnesi olup olmadığıdır. Ne yazık ki, Java’da dize eşitliği bu anlama gelmez. Aslında, dizeleri test etmenin doğru yolu “equals(Object)” yöntemini kullanmaktır. Bir çift dizge için, genellikle aynı karakterlerden aynı sırayla oluşup oluşmadıklarını test etmek isteriz.

public class Hello2 {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0].equals("hello")) {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

Ama aslında daha da kötüleşiyor. Sorun şu ki, == * bazı durumlarda beklenen cevabı * verecektir. Örneğin

public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hello";
        if (s1 == s2) {
            System.out.println("same");
        } else {
            System.out.println("different");
        }
    }
}

İlginç bir şekilde, dizeleri yanlış şekilde test etmemize rağmen, bu “aynı” yazdıracaktır. Nedenmiş? Çünkü Java Dil Belirtimi (Bölüm 3.10.5: String Literals), aynı karakterlerden oluşan herhangi iki string »literals«‘ın aslında aynı Java nesnesi tarafından temsil edileceğini şart koşar. Dolayısıyla, == testi eşit değişmezler için true değerini verecektir. (Dize değişmezleri “interne edilir” ve kodunuz yüklendiğinde paylaşılan bir “dize havuzuna” eklenir, ancak bu aslında bir uygulama ayrıntısıdır.)

Karışıklığa ek olarak, Java Dil Belirtimi ayrıca, iki dize değişmezini birleştiren bir derleme zamanı sabit ifadeniz olduğunda, bunun tek bir değişmeze eşdeğer olduğunu şart koşar. Böylece:

    public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hel" + "lo";
        String s3 = " mum";
        if (s1 == s2) {
            System.out.println("1. same");
        } else {
            System.out.println("1. different");
        }
        if (s1 + s3 == "hello mum") {
            System.out.println("2. same");
        } else {
            System.out.println("2. different");
        }
    }
}

Bu, “1. aynı” ve “2. farklı” çıktısını verecektir. İlk durumda, + ifadesi derleme zamanında değerlendirilir ve bir String nesnesini kendisiyle karşılaştırırız. İkinci durumda, çalışma zamanında değerlendirilir ve iki farklı String nesnesini karşılaştırırız.

Özetle, Java’da dizeleri test etmek için == kullanmak neredeyse her zaman yanlıştır, ancak yanlış cevap vereceği garanti edilmez.

Tuzak: kaynakları serbest bırakmayı unutmak #

Bir program dosya veya ağ bağlantısı gibi bir kaynağı her açtığında, onu kullanmayı bitirdikten sonra kaynağı serbest bırakmak önemlidir. Bu tür kaynaklarla ilgili işlemler sırasında herhangi bir istisna atılacaksa da benzer önlemler alınmalıdır. FileInputStream öğesinin bir çöp toplama olayında close() yöntemini çağıran bir finalizer’e sahip olduğu iddia edilebilir; ancak, bir çöp toplama döngüsünün ne zaman başlayacağından emin olamadığımız için, girdi akışı süresiz olarak bilgisayar kaynaklarını tüketebilir. Kaynak, bir try-catch bloğunun “nihai” bölümünde kapatılmalıdır:

private static void printFileJava6() throws IOException {
    FileInputStream input;
    try {
        input = new FileInputStream("file.txt");
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    } finally {
        if (input != null) {
            input.close();
        }
    }
}
son sürüm

Java 7’den beri, Java 7’de özellikle bu durum için tanıtılan, kaynaklarla deneme adı verilen gerçekten kullanışlı ve temiz bir ifade vardır:

private static void printFileJava7() throws IOException {
    try (FileInputStream input = new FileInputStream("file.txt")) {
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    }
}
son sürüm

try-with-resources ifadesi, “Kapatılabilir” veya “Otomatik Kapatılabilir” arabirimini uygulayan herhangi bir nesneyle kullanılabilir. Her kaynağın ifadenin sonuna kadar kapatılmasını sağlar. İki arabirim arasındaki fark, ‘Closeable’ın ‘close()’ yönteminin bir şekilde ele alınması gereken bir ‘IOException’ atmasıdır.

Kaynağın zaten açıldığı ancak kullanımdan sonra güvenli bir şekilde kapatılması gerektiği durumlarda, kaynaklarla deneme içindeki yerel bir değişkene atanabilir.

private static void printFileJava7(InputStream extResource) throws IOException {
    try (InputStream input = extResource) {
        ... //access resource
    }
}
son sürüm

Kaynaklarla deneme yapıcısında oluşturulan yerel kaynak değişkeni etkin bir şekilde nihaidir.

Tuzak: bir dosyayı açmaya çalışmadan önce test etme. #

Bazı kişiler, daha iyi tanılama sağlamak veya istisnalarla uğraşmaktan kaçınmak için bir dosyayı açmaya çalışmadan önce çeşitli testler uygulamanızı önerir. Örneğin, bu yöntem “yol"un okunabilir bir dosyaya karşılık gelip gelmediğini kontrol etmeye çalışır:

public static File getValidatedFile(String path) throws IOException {
    File f = new File(path);
    if (!f.exists()) throw new IOException("Error: not found: " + path);
    if (!f.isFile()) throw new IOException("Error: Is a directory: " + path);
    if (!f.canRead()) throw new IOException("Error: cannot read file: " + path);
    return f;
}

Yukarıdaki yöntemi şu şekilde kullanabilirsiniz:

File f = null;
try {
    f = getValidatedFile("somefile");
} catch (IOException ex) {
    System.err.println(ex.getMessage());
    return;
}
try (InputStream is = new FileInputStream(file)) {
    // Read data etc.
}

İlk sorun ‘FileInputStream(File)’ imzasındadır çünkü derleyici burada veya yığının daha ilerisinde ‘IOException’ yakalamamız konusunda ısrar edecektir.

İkinci sorun, ‘getValidatedFile’ tarafından gerçekleştirilen kontrollerin ‘FileInputStream’in başarılı olacağını garanti etmemesidir.

  • Yarış koşulları: başka bir iş parçacığı veya ayrı bir işlem, dosyayı yeniden adlandırabilir, dosyayı silebilir veya “getValidatedFile” geri döndükten sonra okuma erişimini kaldırabilir. Bu, özel mesaj olmadan “düz” bir “IOException"a yol açar.

  • Bu testlerin kapsamadığı uç durumlar var. Örneğin, SELinux’un “zorlama” modunda olduğu bir sistemde, bir dosyayı okuma denemesi, “canRead()“in “true” döndürmesine rağmen başarısız olabilir.

Üçüncü sorun, testlerin verimsiz olmasıdır. Örneğin, “exists”, “isFile” ve “canRead” çağrılarının her biri, gerekli kontrolü gerçekleştirmek için bir sistem çağrısı yapacaktır. Daha sonra dosyayı açmak için başka bir sistem çağrısı yapılır ve bu aynı kontrolleri sahne arkasında tekrarlar.

Kısacası, “getValidatedFile” gibi yöntemler yanlış yönlendirilmiştir. Dosyayı açmaya ve istisnayı işlemeye çalışmak daha iyidir:

try (InputStream is = new FileInputStream("somefile")) {
    // Read data etc.
} catch (IOException ex) {
    System.err.println("IO Error processing 'somefile': " + ex.getMessage());
    return;
}

Açma ve okuma sırasında atılan IO hatalarını ayırt etmek istiyorsanız, iç içe bir try/catch kullanabilirsiniz. Açık hatalar için daha iyi tanılama üretmek istiyorsanız, işleyicide “exists”, “isFile” ve “canRead” kontrollerini gerçekleştirebilirsiniz.

Tuzak: değişkenleri nesneler olarak düşünmek #

Hiçbir Java değişkeni bir nesneyi temsil etmez.

String foo;   // NOT AN OBJECT

Herhangi bir Java dizisi de nesneler içermez.

String bar[] = new String[100];  // No member is an object.

Değişkenleri yanlışlıkla nesneler olarak düşünürseniz, Java dilinin gerçek davranışı sizi şaşırtacaktır.

  • İlkel bir türü olan (‘int’ veya ‘float’ gibi) Java değişkenleri için değişken, değerin bir kopyasını tutar. İlkel bir değerin tüm kopyaları ayırt edilemez; yani, bir numara için yalnızca bir “int” değeri vardır. İlkel değerler nesne değildir ve nesneler gibi davranmazlar.

  • Referans tipine (sınıf veya dizi tipi) sahip Java değişkenleri için değişken bir referans içerir. Bir referansın tüm kopyaları ayırt edilemez. Referanslar nesnelere işaret edebilir veya hiçbir nesneye işaret etmedikleri anlamına gelen “null” olabilir. Ancak bunlar nesne değillerdir ve nesneler gibi davranmazlar.

Değişkenler her iki durumda da nesne değildir ve her iki durumda da nesne içermezler. nesnelere referanslar içerebilirler, ancak bu farklı bir şey söylüyor.

Örnek sınıf #

Aşağıdaki örnekler, 2B uzayda bir noktayı temsil eden bu sınıfı kullanır.

public final class MutableLocation {
   public int x;
   public int y;

   public MutableLocation(int x, int y) {
       this.x = x;
       this.y = y;
   }

   public boolean equals(Object other) {
       if (!(other instanceof MutableLocation) {
           return false;
       }
       MutableLocation that = (MutableLocation) other;
       return this.x == that.x && this.y == that.y;
   }
}

Bu sınıfın bir örneği, “int” tipine sahip iki “x” ve “y” alanına sahip bir nesnedir.

MutableLocation sınıfının birçok örneğine sahip olabiliriz. Bazıları 2B uzayda aynı yerleri temsil edecek; yani ilgili “x” ve “y” değerleri eşleşecektir. Diğerleri farklı yerleri temsil edecek.

Birden çok değişken aynı nesneye işaret edebilir #

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

Yukarıda, “MutableLocation” nesnelerine referansları tutabilen “burada”, “orada” ve “başka yerde” olmak üzere üç değişken tanımladık.

Eğer (yanlış bir şekilde) bu değişkenleri nesneler olarak düşünürseniz, o zaman ifadeleri şu şekilde yanlış okumanız muhtemeldir:

  1. “[1, 2]” konumunu “buraya” kopyalayın
  2. “[1, 2]” konumunu “oraya” kopyalayın
  3. “[1, 2]” konumunu “başka bir yere” kopyalayın

Bundan, üç değişkende üç bağımsız nesnemiz olduğu sonucunu çıkarmanız muhtemeldir. Aslında, yukarıdakiler tarafından yalnızca iki nesne yaratılmıştır. “burada” ve “orada” değişkenleri aslında aynı nesneye atıfta bulunur.

Bunu gösterebiliriz. Değişken bildirimlerini yukarıdaki gibi varsayarsak:

System.out.println("BEFORE: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);
here.x = 42;
System.out.println("AFTER: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);

Bu, aşağıdaki çıktıyı verecektir:

BEFORE: here.x is 1, there.x is 1, elsewhere.x is 1
AFTER: here.x is 42, there.x is 42, elsewhere.x is 1

‘here.x’e yeni bir değer atadık ve ‘orada.x’ ile gördüğümüz değeri değiştirdi. Aynı nesneye atıfta bulunuyorlar. Ancak ’elsewhere.x’ aracılığıyla gördüğümüz değer değişmedi, bu nedenle ’elsewhere’ farklı bir nesneye gönderme yapmalıdır.

Bir değişken bir nesneyse, “burada.x = 42” ataması “orada.x” değerini değiştirmez.

Eşitlik operatörü, iki nesnenin eşit olduğunu test ETMEZ

Referans değerlerine eşitlik (==) operatörünün uygulanması, değerlerin aynı nesneye atıfta bulunup bulunmadığını test eder. İki (farklı) nesnenin sezgisel anlamda “eşit” olup olmadığını test etmez.

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

 if (here == there) {
     System.out.println("here is there");
 }
 if (here == elsewhere) {
     System.out.println("here is elsewhere");
 }

Bu, “burası orada” yazdıracak, ancak “burası başka bir yerde” yazmayacak. (“burada” ve “başka yerde” referansları iki farklı nesne içindir.)

Buna karşılık, yukarıda uyguladığımız ’equals(Object)’ yöntemini çağırırsak, iki ‘MutableLocation’ örneğinin eşit bir konuma sahip olup olmadığını test edeceğiz.

 if (here.equals(there)) {
     System.out.println("here equals there");
 }
 if (here.equals(elsewhere)) {
     System.out.println("here equals elsewhere");
 }

Bu, her iki mesajı da yazdıracaktır. Özellikle, ‘here.equals(elsewhere)’, ’true’ değerini döndürür, çünkü iki ‘MutableLocation’ nesnesinin eşitliği için seçtiğimiz anlamsal ölçütler karşılanmıştır.

Yöntem çağrıları nesneleri hiç iletmez #

Java yöntemi çağrıları, argümanları iletmek ve bir sonuç döndürmek için pass by value1 kullanır.

Bir yönteme bir başvuru değeri ilettiğinizde, aslında bir nesneye değere göre bir başvuru iletmiş olursunuz, bu da nesne başvurusunun bir kopyasını oluşturduğu anlamına gelir.

Her iki nesne referansı da aynı nesneye işaret ettiği sürece, o nesneyi herhangi bir referanstan değiştirebilirsiniz ve bu, bazıları için kafa karışıklığına neden olur.

Ancak, 2 referansıyla bir nesneyi geçirmiyorsunuz. Aradaki fark, nesne referans kopyası başka bir nesneye işaret edecek şekilde değiştirilirse, orijinal nesne referansının hala orijinal nesneyi göstermesidir.

void f(MutableLocation foo) {  
    foo = new MutableLocation(3, 4);   // Point local foo at a different object.
}

void g() {
    MutableLocation foo = MutableLocation(1, 2);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 1".
}

Nesnenin bir kopyasını da geçmiyorsunuz.

void f(MutableLocation foo) {  
    foo.x = 42;
}

void g() {
    MutableLocation foo = new MutableLocation(0, 0);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 42"
}

1 - Python ve Ruby gibi dillerde, bir nesnenin / referansın “değerine göre geçiş” için “paylaşarak ilet” terimi tercih edilir.

2 - “Referansla geçiş” veya “referansla arama” terimlerinin programlama dili terminolojisinde çok özel bir anlamı vardır. Aslında bu, bir değişkenin veya bir dizi öğesinin* adresini * ilettiğiniz anlamına gelir, böylece çağrılan yöntem biçimsel bağımsız değişkene yeni bir değer atadığında, orijinal değişkendeki değeri değiştirir. Java bunu desteklemiyor. Parametreleri iletmek için farklı mekanizmaların daha kapsamlı bir açıklaması için lütfen https://en.wikipedia.org/wiki/Evaluation_strategy adresine bakın.

Tuzak: bellek sızıntıları #

Java belleği otomatik olarak yönetir. Belleği manuel olarak boşaltmanız gerekmez. Bir nesnenin öbek üzerindeki belleği, nesne artık canlı bir iş parçacığı tarafından erişilebilir olmadığında bir çöp toplayıcı tarafından serbest bırakılabilir.

Ancak, artık ihtiyaç duyulmayan nesnelerin erişilebilir olmasına izin vererek belleğin serbest kalmasını önleyebilirsiniz. Buna ister bellek sızıntısı, ister bellek paketleme deyin, sonuç aynıdır - ayrılan bellekte gereksiz bir artış.

Java’da bellek sızıntıları çeşitli şekillerde olabilir, ancak en yaygın neden sonsuz nesne referanslarıdır, çünkü çöp toplayıcı hala referanslar varken nesneleri yığından kaldıramaz.

Statik alanlar

Böyle bir referans, bazı nesne koleksiyonlarını içeren bir “statik” alanla sınıf tanımlayarak ve koleksiyona artık ihtiyaç duyulmadığında bu “statik” alanı “boş” olarak ayarlamayı unutarak oluşturulabilir. “Statik” alanlar GC kökleri olarak kabul edilir ve asla toplanmaz. Başka bir sorun, JNI kullanıldığında yığın olmayan bellekteki sızıntılardır.

Sınıf yükleyici sızıntısı

Şimdiye kadar, en sinsi bellek sızıntısı türü [classloader sızıntısı][]‘dır. Bir sınıf yükleyici yüklediği her sınıfa bir referans tutar ve her sınıf kendi sınıf yükleyicisine bir referans tutar. Her nesne kendi sınıfına da bir referans içerir. Bu nedenle, bir sınıf yükleyici tarafından yüklenen bir sınıfın tek bir nesnesi bile çöp değilse, o sınıf yükleyicinin yüklediği tek bir sınıf toplanamaz. Her sınıf aynı zamanda kendi statik alanlarına da atıfta bulunduğundan, bunlar da toplanamaz.

Birikim sızıntısı Birikim sızıntısı örneği aşağıdaki gibi görünebilir:

final ScheduledExecutorService scheduledExecutorService = Executors.newScheduledThreadPool(1);
final Deque<BigDecimal> numbers = new LinkedBlockingDeque<>();
final BigDecimal divisor = new BigDecimal(51);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    BigDecimal number = numbers.peekLast();
    if (number != null && number.remainder(divisor).byteValue() == 0) {
        System.out.println("Number: " + number);
        System.out.println("Deque size: " + numbers.size());
    }
}, 10, 10, TimeUnit.MILLISECONDS);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    numbers.add(new BigDecimal(System.currentTimeMillis()));
}, 10, 10, TimeUnit.MILLISECONDS);

try {
    scheduledExecutorService.awaitTermination(1, TimeUnit.DAYS);
} catch (InterruptedException e) {
    e.printStackTrace();
}

Bu örnek, iki zamanlanmış görev oluşturur. İlk görev, ‘sayılar’ adı verilen bir deque’den son sayıyı alır ve sayı 51’e bölünebiliyorsa sayıyı ve deque’in boyutunu yazdırır. İkinci görev, sayıları deque içine koyar. Her iki görev de sabit bir oranda zamanlanır ve her 10 ms’de bir çalışır.

Kod yürütülürse, deque boyutunun kalıcı olarak arttığını göreceksiniz. Bu, sonunda deque’nin tüm kullanılabilir yığın belleğini tüketen nesnelerle doldurulmasına neden olur.

Bu programın semantiğini korurken bunu önlemek için deque’den sayı almak için farklı bir yöntem kullanabiliriz: pollLast. ‘peekLast’ yönteminin aksine, ‘pollLast’ öğeyi döndürür ve deque’den kaldırırken, ‘peekLast’ yalnızca son öğeyi döndürür.

Tuzak: String’in değişmez bir sınıf olduğunu anlamamak #

Yeni Java programcıları genellikle Java String sınıfının değişmez olduğunu unuturlar veya tam olarak anlayamazlar. Bu, aşağıdaki örnekte olduğu gibi sorunlara yol açar:

public class Shout {
    public static void main(String[] args) {
        for (String s : args) {
            s.toUpperCase();
            System.out.print(s);
            System.out.print(" ");
        }
        System.out.println();
    }
}

Yukarıdaki kodun komut satırı argümanlarını büyük harfle yazdırması gerekiyor. Ne yazık ki çalışmıyor, argümanların durumu değişmiyor. Sorun şu ifadedir:

s.toUpperCase();

’toUpperCase()’ çağrısının ’s’yi büyük harfli bir dizgeye dönüştüreceğini düşünebilirsiniz. Öyle değil. Yapamaz! ‘String’ nesneleri değişmezdir. Değiştirilemezler.

Gerçekte, “toUpperCase()” yöntemi, çağırdığınız “Dize"nin büyük harfli versiyonu olan bir “Dize” nesnesini döndürür. Bu muhtemelen yeni bir ‘String’ nesnesi olacaktır, ancak ’s’ zaten büyük harf ise, sonuç mevcut dize olabilir.

Dolayısıyla bu metodu etkili bir şekilde kullanmak için metot çağrısının döndürdüğü nesneyi kullanmanız gerekir; örneğin:

s = s.toUpperCase();

Aslında, “dizeler asla değişmez” kuralı tüm “String” yöntemleri için geçerlidir. Bunu hatırlarsanız, yeni başlayanların hatalarından oluşan bir kategoriden kaçınabilirsiniz.

Tuzak: atama ve yan etkileri birleştirme #

Ara sıra, bunun gibi bir şey soran StackOverflow Java soruları (ve C veya C++ soruları) görürüz:

i += a[i++] + b[i--];

‘i’, ‘a’ ve ‘b’ bazı bilinen başlangıç ​​durumları için … olarak değerlendirir.

Genel konuşma:

  • Java için yanıt her zaman belirtilir1, ancak açık değildir ve genellikle anlaşılması zordur
  • C ve C++ için cevap genellikle belirsizdir.

Bu tür örnekler genellikle sınavlarda veya iş görüşmelerinde öğrencinin veya görüşülen kişinin Java programlama dilinde ifade değerlendirmesinin gerçekten nasıl çalıştığını anlayıp anlamadığını görmek için kullanılır. Bu bir “bilgi testi” olarak tartışmalı bir şekilde meşrudur, ancak bu, bunu gerçek bir programda yapmanız gerektiği anlamına gelmez.

Örneklemek gerekirse, aşağıdaki görünüşte basit olan örnek StackOverflow sorularında birkaç kez göründü ([bu bir 1] gibi). Bazı durumlarda, birinin kodunda gerçek bir hata olarak görünür.

int a = 1;
a = a++;
System.out.println(a);    // What does this print.

Bu ifadeleri hızlıca okuyan çoğu programcı (Java uzmanları dahil) çıktının ‘2’ olduğunu söyler. Aslında, ‘1’ çıktısını verir. Nedeniyle ilgili ayrıntılı bir açıklama için lütfen bu Yanıtı okuyun.

Bununla birlikte, bu ve benzeri örneklerden gerçek çıkarım, aynı değişkenin ve yan etkilerine her ikisinin atadığı herhangi bir Java ifadesinin en iyi anlaşılması zor ve en kötü düpedüz yanıltıcı olacağıdır. . Bunun gibi kod yazmaktan kaçınmalısınız.

1 - değişkenler veya nesneler diğer iş parçacıkları tarafından görülebiliyorsa Java Bellek Modeli ile ilgili modulo potansiyel sorunları.
