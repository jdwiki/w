AspNetCoreRateLimit, istemcilerin IP adresine veya istemci kimliğine dayalı olarak bir Web API’sine veya MVC uygulamasına yapabileceği isteklerin oranını kontrol etmek için tasarlanmış açık kaynaklı bir ASP.NET Core hız sınırlama çözümüdür.

Müşteri kimliğine göre hız sınırlaması #

ClientRateLimit ara yazılımı ile, bir İstemcinin saniye, 15 dakika vb. gibi bir zaman aralığında maksimum sayıda çağrı yapmasına izin vermek gibi farklı senaryolar için birden fazla sınır belirleyebilirsiniz. Bu sınırları, bir API’ye veya size yapılan tüm istekleri karşılamak için tanımlayabilirsiniz. her bir URL yolunun veya HTTP fiilinin ve yolunun sınırlarını kapsayabilir.

Kurmak #

NuGet kurulumu:

Kurulum Paketi AspNetCoreRateLimit

Startup.cs kodu:

public void ConfigureServices(IServiceCollection services)
{
    // needed to load configuration from appsettings.json
    services.AddOptions();

    // needed to store rate limit counters and ip rules
    services.AddMemoryCache();

    //load general configuration from appsettings.json
    services.Configure<ClientRateLimitOptions>(Configuration.GetSection("ClientRateLimiting"));

    //load client rules from appsettings.json
    services.Configure<ClientRateLimitPolicies>(Configuration.GetSection("ClientRateLimitPolicies"));

    // inject counter and rules stores
    services.AddSingleton<IClientPolicyStore, MemoryCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore, MemoryCacheRateLimitCounterStore>();

    // Add framework services.
    services.AddMvc();
}

public void Configure(IApplicationBuilder app, IHostingEnvironment env, ILoggerFactory loggerFactory)
{
    loggerFactory.AddConsole(Configuration.GetSection("Logging"));
    loggerFactory.AddDebug();

    app.UseClientRateLimiting();

    app.UseMvc();
}

Ara yazılımı, loggerFactory dışındaki tüm diğer bileşenlerden önce kaydetmelisiniz.

uygulamanızı yüklerseniz, tüm kerkenez örneklerinin aynı oran limiti deposuna sahip olması için Redis veya SQLServer ile IDistributedCache kullanmanız gerekir. Bellek içi depolar yerine, dağıtılmış depoları şu şekilde enjekte etmelisiniz:

    // inject counter and rules distributed cache stores
    services.AddSingleton<IClientPolicyStore, DistributedCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore,DistributedCacheRateLimitCounterStore>();

Yapılandırma ve genel kurallar appsettings.json:

  "ClientRateLimiting": {
    "EnableEndpointRateLimiting": false,
    "StackBlockedRequests": false,
    "ClientIdHeader": "X-ClientId",
    "HttpStatusCode": 429,
    "EndpointWhitelist": [ "get:/api/license", "*:/api/status" ],
    "ClientWhitelist": [ "dev-id-1", "dev-id-2" ],
    "GeneralRules": [
      {
        "Endpoint": "*",
        "Period": "1s",
        "Limit": 2
      },
      {
        "Endpoint": "*",
        "Period": "15m",
        "Limit": 100
      },
      {
        "Endpoint": "*",
        "Period": "12h",
        "Limit": 1000
      },
      {
        "Endpoint": "*",
        "Period": "7d",
        "Limit": 10000
      }
    ]
  }

“EnableEndpointRateLimiting”, “false” olarak ayarlanırsa, sınırlar genel olarak uygulanır ve yalnızca bitiş noktası “*” olan kurallar uygulanır. Örneğin, saniyede 5 çağrı sınırı belirlerseniz, herhangi bir uç noktaya yapılan herhangi bir HTTP çağrısı bu sınıra dahil edilir.

“EnableEndpointRateLimiting”, “true” olarak ayarlanırsa, sınırlar her uç nokta için “{HTTP_Verb}{PATH}” içindeki gibi geçerli olacaktır. Örneğin, ‘*:/api/values’ için saniyede 5 çağrı sınırı belirlerseniz, bir istemci ‘GET /api/values’u saniyede 5 kez ve ayrıca 5 kez ‘PUT /api/values’ çağrısı yapabilir.

“StackBlockedRequests” “false” olarak ayarlanırsa, reddedilen çağrılar kısma sayacına eklenmez. Bir müşteri saniyede 3 istekte bulunursa ve saniyede bir arama sınırı belirlediyseniz, dakika veya gün başına sayaçlar gibi diğer sınırlar yalnızca engellenmeyen ilk aramayı kaydeder. Reddedilen isteklerin diğer sınırlara dahil edilmesini istiyorsanız, “StackBlockedRequests"i “true” olarak ayarlamanız gerekir.

‘ClientIdHeader’, müşteri kimliğini çıkarmak için kullanılır, bu başlıkta bir müşteri kimliği varsa ve ClientWhitelist’te belirtilen bir değerle eşleşirse, herhangi bir hız sınırı uygulanmaz.

Hız sınırı kurallarını tanımlama #

Bir kural, bir bitiş noktası, bir nokta ve bir sınırdan oluşur.

Bitiş noktası biçimi {HTTP_Verb}:{PATH} şeklindedir, asterix sembolünü kullanarak herhangi bir HTTP fiilini hedefleyebilirsiniz.

Dönem biçimi {INT}{PERIOD_TYPE}, şu dönem türlerinden birini kullanabilirsiniz: s, m, h, d.

Sınır biçimi “{LONG}” şeklindedir.

Örnekler:

Tüm uç noktaları saniyede 2 çağrı ile sınırlayın:

{
 "Endpoint": "*",
 "Period": "1s",
 "Limit": 2
}

Aynı saniyede, bir istemci api/değerlere 3 GET çağrısı yaparsa, son çağrı engellenir. Ancak aynı anda PUT api/değerlerini de çağırırsa, istek farklı bir uç nokta olduğu için gerçekleşecektir. Uç nokta hız sınırlaması etkinleştirildiğinde, her çağrının hızı “{HTTP_Verb}{PATH}” temelinde sınırlandırılır.

Herhangi bir HTTP Fiili ile /api/values ile 15 dakikada 5 çağrıya kadar hız sınırı çağrıları:

{
 "Endpoint": "*:/api/values",
 "Period": "15m",
 "Limit": 5
}

/api/values’a GET çağrısının hız sınırı saatte 5 çağrıya kadar:

{
 "Endpoint": "get:/api/values",
 "Period": "1h",
 "Limit": 5
}

Bir saat içinde, bir istemci api/değerlere 6 GET çağrısı yaparsa, son çağrı engellenir. Ancak aynı saatte GET api/values/1’i de çağırırsa, istek farklı bir uç nokta olduğu için gerçekleşecektir.

Davranış #

Bir istemci bir HTTP çağrısı yaptığında ClientRateLimitMiddleware aşağıdakileri yapar:

  • İstemci kimliğini, HTTP fiilini ve URL’yi istek nesnesinden çıkarır, kendi çıkarma mantığınızı uygulamak istiyorsanız ‘ClientRateLimitMiddleware.SetIdentity’yi geçersiz kılabilirsiniz.
  • beyaz listelerde Müşteri kimliğini ve URL’yi arar, herhangi bir eşleşme varsa herhangi bir işlem yapılmaz
  • bir eşleşme için Müşteri kurallarında arama yapar, geçerli tüm kurallar döneme göre gruplandırılır, her dönem için en kısıtlayıcı kural kullanılır
  • bir eşleşme için Genel kurallarda arama yapar, eşleşen genel bir kuralın Müşteri kurallarında mevcut olmayan tanımlanmış bir periyodu varsa, bu genel kural da kullanılır
  • her eşleşen kural için hız sınırı sayacı artırılır, sayaç değeri kural sınırından büyükse istek engellenir

İstek engellenirse, müşteri şöyle bir metin yanıtı alır:

Status Code: 429
Retry-After: 58
Content: API calls quota exceeded! maximum admitted 2 per 1m.

Bu “HttpStatusCode” ve “QuotaExceededMessage” seçeneklerini değiştirerek yanıtı özelleştirebilirsiniz, kendi yanıtınızı uygulamak istiyorsanız “ClientRateLimitMiddleware.ReturnQuotaExceededResponse” öğesini geçersiz kılabilirsiniz. ‘Retry-After’ başlık değeri saniye cinsinden ifade edilir.

İstek hız sınırlaması almazsa, X-Rate-Limit başlıklarını oluşturmak için eşleştirme kurallarında tanımlanan en uzun süre kullanılır, bu başlıklar yanıta eklenir:

X-Rate-Limit-Limit: the rate limit period (eg. 1m, 12h, 1d)
X-Rate-Limit-Remaining: number of request remaining 
X-Rate-Limit-Reset: UTC date time when the limits resets

Varsayılan olarak engellenen istekler “Microsoft.Extensions.Logging.ILogger” kullanılarak günlüğe kaydedilir, eğer kendi günlük kaydınızı uygulamak istiyorsanız “ClientRateLimitMiddleware.LogBlockedRequest"i geçersiz kılabilirsiniz. Varsayılan günlükçü, bir istek hız sınırlı olduğunda aşağıdaki bilgileri yayar:

info: AspNetCoreRateLimit.ClientRateLimitMiddleware[0]
      Request get:/api/values from ClientId client-id-1 has been blocked, quota 2/1m exceeded by 3. Blocked by rule *:/api/value, TraceIdentifier 0HKTLISQQVV9D.

Çalışma zamanında hız sınırlarını güncelleyin #

Uygulama başlangıcında, “appsettings.json” içinde tanımlanan istemci hızı sınırı kuralları, kullandığınız önbellek sağlayıcısının türüne bağlı olarak “MemoryCacheClientPolicyStore” veya “DistributedCacheClientPolicyStore” tarafından önbelleğe yüklenir. Bir denetleyici içindeki istemci ilke deposuna erişebilir ve aşağıdaki gibi kuralları değiştirebilirsiniz:

public class ClientRateLimitController : Controller
{
    private readonly ClientRateLimitOptions _options;
    private readonly IClientPolicyStore _clientPolicyStore;

    public ClientRateLimitController(IOptions<ClientRateLimitOptions> optionsAccessor, IClientPolicyStore clientPolicyStore)
    {
        _options = optionsAccessor.Value;
        _clientPolicyStore = clientPolicyStore;
    }

    [HttpGet]
    public ClientRateLimitPolicy Get()
    {
        return _clientPolicyStore.Get($"{_options.ClientPolicyPrefix}_cl-key-1");
    }

    [HttpPost]
    public void Post()
    {
        var id = $"{_options.ClientPolicyPrefix}_cl-key-1";
        var clPolicy = _clientPolicyStore.Get(id);
        clPolicy.Rules.Add(new RateLimitRule
        {
            Endpoint = "*/api/testpolicyupdate",
            Period = "1h",
            Limit = 100
        });
        _clientPolicyStore.Set(id, clPolicy);
    }
}

Bu şekilde, istemci hızı sınırlarını bir veritabanında saklayabilir ve her uygulama başladıktan sonra bunları önbelleğe aktarabilirsiniz.

Ancak aynı saniyede PUT api/değerlerini de çağırırsanız, istek farklı bir uç nokta olduğu için gerçekleşecektir. Uç nokta hız sınırlaması etkinleştirildiğinde, her çağrının hızı “{HTTP_Verb}{PATH}” temelinde sınırlandırılır.

Herhangi bir HTTP Fiili ile /api/values ile 15 dakikada 5 çağrıya kadar hız sınırı çağrıları:

{
 "Endpoint": "*:/api/values",
 "Period": "15m",
 "Limit": 5
}

/api/values’a GET çağrısının hız sınırı saatte 5 çağrıya kadar:

{
 "Endpoint": "get:/api/values",
 "Period": "1h",
 "Limit": 5
}

Aynı IP’den bir saat içinde api/değerlere 6 GET araması yaparsanız, son arama engellenir. Ancak aynı saatte GET api/values/1 öğesini de çağırırsanız, bu farklı bir uç nokta olduğu için istek gerçekleştirilecektir.

Davranış #

Bir istemci bir HTTP çağrısı yaptığında, IpRateLimitMiddleware aşağıdakileri yapar:

  • istek nesnesinden IP’yi, İstemci kimliğini, HTTP fiilini ve URL’yi çıkarır, eğer kendi çıkarma mantığınızı uygulamak istiyorsanız ‘IpRateLimitMiddleware.SetIdentity’yi geçersiz kılabilirsiniz.
  • beyaz listelerde IP, İstemci kimliği ve URL’yi arar, herhangi bir eşleşme varsa herhangi bir işlem yapılmaz
  • bir eşleşme için IP kurallarında arama yapar, geçerli tüm kurallar döneme göre gruplandırılır, her dönem için en kısıtlayıcı kural kullanılır
  • Bir eşleşme için Genel kurallarda arama yapar, eşleşen genel bir kuralın IP kurallarında bulunmayan tanımlanmış bir periyodu varsa, bu genel kural da kullanılır
  • her eşleşen kural için hız sınırı sayacı artırılır, sayaç değeri kural sınırından büyükse istek engellenir

İstek engellenirse, müşteri şöyle bir metin yanıtı alır:

Status Code: 429
Retry-After: 58
Content: API calls quota exceeded! maximum admitted 2 per 1m.

Bu HttpStatusCode ve QuotaExceededMessage seçeneklerini değiştirerek yanıtı özelleştirebilirsiniz, eğer kendi yanıtınızı uygulamak istiyorsanız, IpRateLimitMiddleware.ReturnQuotaExceededResponse öğesini geçersiz kılabilirsiniz. ‘Retry-After’ başlık değeri saniye cinsinden ifade edilir.

İstek hız sınırlaması almazsa, X-Rate-Limit başlıklarını oluşturmak için eşleştirme kurallarında tanımlanan en uzun süre kullanılır, bu başlıklar yanıta eklenir:

X-Rate-Limit-Limit: the rate limit period (eg. 1m, 12h, 1d)
X-Rate-Limit-Remaining: number of request remaining 
X-Rate-Limit-Reset: UTC date time when the limits resets

Varsayılan olarak, engellenen istek ‘Microsoft.Extensions.Logging.ILogger’ kullanılarak günlüğe kaydedilir, eğer kendi günlük kaydınızı uygulamak istiyorsanız ‘IpRateLimitMiddleware.LogBlockedRequest’i geçersiz kılabilirsiniz. Varsayılan günlükçü, bir istek hız sınırlı olduğunda aşağıdaki bilgileri yayar:

info: AspNetCoreRateLimit.IpRateLimitMiddleware[0]
      Request get:/api/values from IP 84.247.85.224 has been blocked, quota 2/1m exceeded by 3. Blocked by rule *:/api/value, TraceIdentifier 0HKTLISQQVV9D.

Çalışma zamanında hız sınırlarını güncelleyin #

Uygulama başlangıcında, “appsettings.json” içinde tanımlanan IP hızı sınırı kuralları, kullandığınız önbellek sağlayıcısının türüne bağlı olarak “MemoryCacheClientPolicyStore” veya “DistributedCacheIpPolicyStore” tarafından önbelleğe yüklenir. Bir denetleyici içindeki IP ilke deposuna erişebilir ve IP kurallarını şu şekilde değiştirebilirsiniz:

public class IpRateLimitController : Controller
{
    private readonly IpRateLimitOptions _options;
    private readonly IIpPolicyStore _ipPolicyStore;

    public IpRateLimitController(IOptions<IpRateLimitOptions> optionsAccessor, IIpPolicyStore ipPolicyStore)
    {
        _options = optionsAccessor.Value;
        _ipPolicyStore = ipPolicyStore;
    }

    [HttpGet]
    public IpRateLimitPolicies Get()
    {
        return _ipPolicyStore.Get(_options.IpPolicyPrefix);
    }

    [HttpPost]
    public void Post()
    {
        var pol = _ipPolicyStore.Get(_options.IpPolicyPrefix);

        pol.IpRules.Add(new IpRateLimitPolicy
        {
            Ip = "8.8.4.4",
            Rules = new List<RateLimitRule>(new RateLimitRule[] {
                new RateLimitRule {
                    Endpoint = "*:/api/testupdate",
                    Limit = 100,
                    Period = "1d" }
            })
        });

        _ipPolicyStore.Set(_options.IpPolicyPrefix, pol);
    }
}

Bu şekilde, IP hızı sınırlarını bir veritabanında saklayabilir ve her uygulama başladıktan sonra bunları önbelleğe aktarabilirsiniz.
