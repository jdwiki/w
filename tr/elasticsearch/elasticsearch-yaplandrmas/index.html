Elasticsearch, geliştirme için kullanıma hazır iyi bir deneyim sağlayan bir dizi varsayılanla birlikte gelir. Buradaki örtük ifade, kendi ihtiyaçlarınıza göre uyarlanması gereken ve bu nedenle tahmin edilemeyen üretim için mutlaka harika olmadığıdır.

Varsayılan ayarlar, herhangi bir yapılandırma değişikliği olmadan aynı makinede birden çok düğümün indirilmesini ve çalıştırılmasını kolaylaştırır.

Ayarlar nerede? #

Her Elasticsearch kurulumunun içinde bir config/elasticsearch.yml bulunur. Bu, aşağıdaki ayarların bulunduğu yerdir:

  • “küme.adı”
    • The name of the cluster that the node is joining. All nodes in the same cluster must share the same name.
    • Currently defaults to elasticsearch.
  • düğüm.*
    • node.name
      • If not supplied, a random name will be generated each time the node starts. This can be fun, but it is not good for production environments.
      • Names do not have to be unique, but they should be unique.
    • node.master
      • A boolean setting. When true, it means that the node is an eligible master node and it can be the elected master node.
      • Defaults to true, meaning every node is an eligible master node.
    • node.data
      • A boolean setting. When true, it means that the node stores data and handles search activity.
      • Defaults to true.
  • yol.*
    • path.data
      • The location that files are written for the node. All nodes use this directory to store metadata, but data nodes will also use it to store/index documents.
      • Defaults to ./data.
        • This means that data will be created for you as a peer directory to config inside of the Elasticsearch directory.
    • path.logs
      • The location that log files are written.
      • Defaults to ./logs.
  • ağ.*

    • network.host

      • Defaults to _local_, which is effectively localhost.
        • This means that, by default, nodes cannot be communicated with from outside of the current machine!

    • network.bind_host

      • Potentially an array, this tells Elasticsearch what addresses of the current machine to bind sockets too.
        • It is this list that enables machines from outside of the machine (e.g., other nodes in the cluster) to talk to this node.
      • Defaults to network.host.

    • network.publish_host

      • A singular host that is used to advertise to other nodes how to best communicate with this node.
        • When supplying an array to network.bind_host, this should be the one host that is intended to be used for inter-node communication.
      • Defaults to network.host`.
  • keşif.zen.*
    • discovery.zen.minimum_master_nodes
      • Defines quorum for master election. This must be set using this equation: (M / 2) + 1 where M is the number of eligible master nodes (nodes using node.master: true implicitly or explicitly).
      • Defaults to 1, which only is valid for a single node cluster!
    • discovery.zen.ping.unicast.hosts
      • The mechanism for joining this node to the rest of a cluster.
      • This should list eligible master nodes so that a node can find the rest of the cluster.
      • The value that should be used here is the network.publish_host of those other nodes.
      • Defaults to localhost, which means it only looks on the local machine for a cluster to join.

Ne tür ayarlar var? #

Elasticsearch, üç farklı ayar türü sağlar:

  • Küme çapında ayarlar
    • These are settings that apply to everything in the cluster, such as all nodes or all indices.
  • Düğüm ayarları
    • These are settings that apply to just the current node.
  • Dizin ayarları
    • These are settings that apply to just the index.

Ayara bağlı olarak şunlar olabilir:

  • Çalışma zamanında dinamik olarak değiştirildi
  • Dizinin yeniden başlatılması (kapat / aç) sonrasında değişti
    • Some index-level settings do not require the index to be closed and reopened, but might require the index to be forceably re-merged for the setting to apply.
      • The compression level of an index is an example of this type of setting. It can be changed dynamically, but only new segments take advantage of the change. So if an index will not change, then it never takes advantage of the change unless you force the index to recreate its segments.
  • Düğümün yeniden başlatılmasının ardından değişti
  • Kümenin yeniden başlatılmasının ardından değişti
  • Asla değişmedi

Bir ayarla ne yapıp ne yapamayacağınızı öğrenmek için her zaman Elasticsearch sürümünüzün belgelerine bakın.

Ayarları nasıl uygulayabilirim? #

Ayarları, bazıları önerilmeyen birkaç şekilde ayarlayabilirsiniz:

  • Komut Satırı Argümanları

Elasticsearch 1.x ve 2.x’te, çoğu ayarı es. ön eki olan Java Sistem Özellikleri olarak gönderebilirsiniz:

$ bin/elasticsearch -Des.cluster.name=my_cluster -Des.node.name=`hostname`

Elasticsearch 5.x’te bu, Java Sistem Özellikleri’ni kullanmaktan kaçınmak için, bunun yerine “-E"nin “-Des.“nin yerini aldığı özel bir argüman türü kullanmaktan kaçınmak için değişir:

$ bin/elasticsearch -Ecluster.name=my_cluster -Enode.name=`hostname`

Ayarları uygulamaya yönelik bu yaklaşım, kümeyi başlatmak ve durdurmak için Puppet, Chef veya Ansible gibi araçlar kullanıldığında harika çalışır. Ancak manuel olarak yapıldığında çok kötü çalışıyor.

-YAML ayarları - Shown in examples

  • Dinamik ayarlar
    • Shown in examples

Ayarların uygulandığı sıra, en dinamik sırasına göredir:

  1. Geçici ayarlar
  2. Kalıcı ayarlar
  3. Komut satırı ayarları
  4. YAML (statik) ayarları

Ayar, bir kez bu düzeylerden birinde olmak üzere iki kez ayarlanırsa, en yüksek düzey geçerli olur.

Aynı Anda Birden Fazla Endeks için Dinamik İndeks Ayarları #

“Dizin Ayarları” örneğinde gösterilen değişikliğin aynısını, tek bir istekle, hatta bunların bir alt kümesiyle tüm mevcut dizinlere uygulayabilirsiniz:

PUT /*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

veya

PUT /_all/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

veya

PUT /_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Bunu da daha seçici bir şekilde yapmayı tercih ederseniz, hepsini sağlamadan birden çok seçim yapabilirsiniz:

PUT /logstash-*,my_other_index,some-other-*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Statik Elasticsearch Ayarları #

Elasticsearch, varsayılan Elasticsearch dizininde bulunabilen bir YAML (Yine Başka Bir İşaretleme Dili) yapılandırma dosyası kullanır ([RPM ve DEB yüklemeleri, diğer şeylerin yanı sıra bu konumu değiştirir](https://www.elastic.co/guide/en/elasticsearch /referans/current/setup-dir-layout.html)).

config/elasticsearch.yml içinde temel ayarları yapabilirsiniz:

# Change the cluster name. All nodes in the same cluster must use the same name!
cluster.name: my_cluster_name

# Set the node's name using the hostname, which is an environment variable!
# This is a convenient way to uniquely set it per machine without having to make
#  a unique configuration file per node.
node.name: ${HOSTNAME}

# ALL nodes should set this setting, regardless of node type
path.data: /path/to/store/data

# This is a both a master and data node (defaults)
node.master: true
node.data: true

# This tells Elasticsearch to bind all sockets to only be available
#  at localhost (default)
network.host: _local_

Kalıcı Dinamik Küme Ayarları #

Bir ayarı küme başladıktan sonra dinamik olarak uygulamanız gerekiyorsa ve bu gerçekten dinamik olarak ayarlanabiliyorsa, bunu _cluster/settings API’sini kullanarak ayarlayabilirsiniz.

Kalıcı ayarlar, uygulanabilecek iki tür küme çapında ayardan biridir. Kalıcı bir ayar, tam bir kümenin yeniden başlatılmasından sağ kalır.

Not: Tüm ayarlar dinamik olarak uygulanamaz. Örneğin, kümenin adı dinamik olarak yeniden adlandırılamaz. Düğüm düzeyindeki ayarların çoğu dinamik olarak da ayarlanamaz (çünkü tek tek hedeflenemezler).

Bu, dizin düzeyinde ayarları belirlemek için kullanılacak API değildir. index. ile başlaması gerektiğinden, ayarın bir indeks seviyesi ayarı olduğunu söyleyebilirsiniz. Adı `indeksler’ biçiminde olan ayarlar. Tüm indeksler için geçerli oldukları için küme çapında ayarlardır.

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}

Uyarı: Elasticsearch 1.x ve 2.x’te kalıcı bir ayarı ayarlayamazsınız.

Neyse ki, bu, Elasticsearch 5.x’te geliştirildi ve artık bir ayarı “boş” olarak ayarlayarak kaldırabilirsiniz:

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}

Ayarlanmamış bir ayar, varsayılan değerine veya daha düşük bir öncelik düzeyinde tanımlanan herhangi bir değere (ör. komut satırı ayarları) geri döner.

Geçici Dinamik Küme Ayarları #

Bir ayarı küme başladıktan sonra dinamik olarak uygulamanız gerekiyorsa ve bu gerçekten dinamik olarak ayarlanabiliyorsa, bunu _cluster/settings API’sini kullanarak ayarlayabilirsiniz.

Geçici ayarlar, uygulanabilecek iki tür küme çapında ayardan biridir. Geçici bir ayar, tam bir kümenin yeniden başlatılmasından geçemez.

Not: Tüm ayarlar dinamik olarak uygulanamaz. Örneğin, kümenin adı dinamik olarak yeniden adlandırılamaz. Düğüm düzeyindeki ayarların çoğu dinamik olarak da ayarlanamaz (çünkü tek tek hedeflenemezler).

Bu, dizin düzeyinde ayarları belirlemek için kullanılacak API değildir. index. ile başlaması gerektiğinden, ayarın bir indeks seviyesi ayarı olduğunu söyleyebilirsiniz. Adı `indeksler’ biçiminde olan ayarlar. Tüm indeksler için geçerli oldukları için küme çapında ayarlardır.

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "none"
  }
}

Uyarı: Elasticsearch 1.x ve 2.x’te, tam küme yeniden başlatması olmadan geçici ayarları geri alamazsınız.

Neyse ki bu, Elasticsearch 5.x’te geliştirildi ve artık bir ayarı null olarak ayarlayarak kaldırabilirsiniz:

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": null
  }
}

Ayarlanmamış bir ayar, varsayılan değerine veya daha düşük bir öncelik düzeyinde tanımlanan herhangi bir değere (örneğin, “kalıcı” ayarlar) geri döner.

Dizin Ayarları #

Dizin ayarları, tek bir dizine uygulanan ayarlardır. Bu tür ayarlar index. ile başlayacaktır. Bu kuralın istisnası, aynı zamanda “index.shards_of_” ve “index.number_of_replikas” biçiminde de bulunan “parça_sayısı” ve “kopya_sayısı"dır.

Adından da anlaşılacağı gibi, dizin düzeyindeki ayarlar tek bir dizine uygulanır. Dizin için birincil parça sayısını kontrol eden “index.number_of_shards” ayarı gibi bazı ayarlar dinamik olarak değiştirilemediğinden oluşturma sırasında uygulanmalıdır.

PUT /my_index
{
  "settings": {
    "index.number_of_shards": 1,
    "index.number_of_replicas": 1
  }
}

veya daha kısa bir biçimde, anahtar önekleri her . harfinde birleştirebilirsiniz:

PUT /my_index
{
  "settings": {
    "index": {
      "number_of_shards": 1,
      "number_of_replicas": 1
    }
  }
}

Yukarıdaki örnekler, sağlanan ayarlarla bir dizin oluşturacaktır. “_settings” dizini uç noktasını kullanarak dizin başına ayarları dinamik olarak değiştirebilirsiniz. Örneğin, burada slowlog ayarlarını uyarı düzeyi için yalnızca dinamik olarak değiştiririz:

PUT /my_index/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Uyarı: Elasticsearch 1.x ve 2.x, dizin düzeyinde ayar adlarını kesin olarak doğrulamamıştır. Bir yazım hatası yaptıysanız veya basitçe bir ayar yaptıysanız, körü körüne kabul eder, aksi takdirde görmezden gelirdi. Elasticsearch 5.x, ayar adlarını kesinlikle doğrular ve bilinmeyen ayar(lar)la (yazım hatası veya eksik eklenti nedeniyle) dizin ayarlarını uygulama girişimlerini reddeder. Her iki ifade de dinamik olarak değişen dizin ayarları için ve oluşturma sırasında geçerlidir.
