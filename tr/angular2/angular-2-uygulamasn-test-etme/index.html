Jasmine test çerçevesini yükleme #

Angular 2 uygulamalarını test etmenin en yaygın yolu Jasmine test çerçevesidir. Jasmine, kodunuzu tarayıcıda test etmenizi sağlar.

Düzenlemek #

Başlamak için tek ihtiyacınız olan “yasemin çekirdeği” paketidir (“yasemin” değil).

npm install jasmine-core --save-dev --save-exact

Doğrulayın #

Jasmine’in düzgün kurulduğunu doğrulamak için aşağıdaki içerikle ./src/unit-tests.html dosyasını oluşturun ve tarayıcıda açın.

<!DOCTYPE html>
<html>
<head>
  <meta http-equiv="content-type" content="text/html;charset=utf-8">
  <title>Ng App Unit Tests</title>
  <link rel="stylesheet" href="../node_modules/jasmine-core/lib/jasmine-core/jasmine.css">
  <script src="../node_modules/jasmine-core/lib/jasmine-core/jasmine.js"></script>
  <script src="../node_modules/jasmine-core/lib/jasmine-core/jasmine-html.js"></script>
  <script src="../node_modules/jasmine-core/lib/jasmine-core/boot.js"></script>
</head>
<body>
   <!-- Unit Testing Chapter #1: Proof of life.  -->
   <script>
     it('true is true', function () {
       expect(true).toEqual(true);
     });
   </script>
</body>
</html>

Gulp, Webpack, Karma ve Jasmine ile test kurulumu #

İhtiyacımız olan ilk şey, webpack motoru için ayarladığımız bir konfigürasyon altında, karma’ya testlerimizi okumak için Webpack’i kullanmasını söylemek. Burada, kodumu ES6’da yazdığım için babel kullanıyorum, bunu Typescript gibi diğer lezzetler için değiştirebilirsiniz. Veya Pug (eski adıyla Jade) şablonları kullanıyorum, buna gerek yok.

Yine de strateji aynı kalıyor.

Yani, bu bir web paketi yapılandırmasıdır:

const webpack = require("webpack");
let packConfig = {
    entry: {},
    output: {},
    plugins:[
        new webpack.DefinePlugin({
            ENVIRONMENT: JSON.stringify('test')
        })
    ],
    module: {
       loaders: [
        {
            test: /\.js$/,
            exclude:/(node_modules|bower_components)/,
            loader: "babel",
            query:{
                presets:["es2015", "angular2"]
            }
        },
        {
            test: /\.woff2?$|\.ttf$|\.eot$|\.svg$/,
            loader: "file"
        },
        {
            test: /\.scss$/,
            loaders: ["style", "css", "sass"]
        },
        {
            test: /\.pug$/,
            loader: 'pug-html-loader'
        },
        ]
    },
    devtool : 'inline-cheap-source-map'
};
module.exports = packConfig;

Ardından, bu web paketi yapılandırmasını kullanmak için bir karma.config.js dosyasına ihtiyacımız var:

const packConfig = require("./webpack.config.js");
module.exports = function (config) {
    config.set({
    basePath: '',
    frameworks: ['jasmine'],
    exclude:[],
    files: [
        {pattern: './karma.shim.js', watched: false}
    ],

    preprocessors: {
        "./karma.shim.js":["webpack"]
    },
    webpack: packConfig,

    webpackServer: {noInfo: true},

    port: 9876,

    colors: true,

    logLevel: config.LOG_INFO,

    browsers: ['PhantomJS'],

    concurrency: Infinity,

    autoWatch: false,
    singleRun: true
});
};

Şimdiye kadar Karma’ya web paketini kullanmasını söyledik ve ona karma.shim.js adlı bir dosyadan başlamasını söyledik. bu dosya, web paketi için başlangıç ​​noktası görevi görecektir. web paketi bu dosyayı okuyacak ve tüm bağımlılıklarımızı toplamak ve testlerimizi çalıştırmak için import ve require ifadelerini kullanacak.

Şimdi karma.shim.js dosyasına bakalım:

// Start of ES6 Specific stuff
import "es6-shim";
import "es6-promise";
import "reflect-metadata";
// End of ES6 Specific stuff

import "zone.js/dist/zone";

import "zone.js/dist/fake-async-test";

import 'rxjs/add/operator/map';
import 'rxjs/add/observable/of';

Error.stackTraceLimit = Infinity;

import {TestBed} from "@angular/core/testing";
import { BrowserDynamicTestingModule, platformBrowserDynamicTesting} from "@angular/platform-browser-dynamic/testing";

TestBed.initTestEnvironment(
     BrowserDynamicTestingModule,
     platformBrowserDynamicTesting());

let testContext = require.context('../src/app', true, /\.spec\.js/);
testContext.keys().forEach(testContext);

Özünde, tüm testlerimiz için yalnızca bir kez başlatılması gerektiğinden, açısal çekirdek testinden TestBed‘i içe aktarıyoruz ve ortamı başlatıyoruz. Ardından, src/app dizinini yinelemeli olarak inceliyoruz ve .spec.js ile biten her dosyayı okuyoruz ve çalışsınlar diye onları testContext’e besliyoruz.

Genelde testlerimi sınıfla aynı yere koymaya çalışırım. Personat tadı, bağımlılıkları ve sınıflarla yeniden düzenleme testlerini içe aktarmamı kolaylaştırıyor. Ancak testlerinizi örneğin src/test dizini altında olduğu gibi başka bir yere koymak istiyorsanız, işte şansınız. karma.shim.js dosyasında sondan önceki satırı değiştirin.

Mükemmel. ne kaldı? ah, yukarıda yaptığımız karma.config.js dosyasını kullanan yudum görevi:

gulp.task("karmaTests",function(done){
    var Server = require("karma").Server;
    new Server({
        configFile : "./karma.config.js",
        singleRun: true,
        autoWatch: false
    }, function(result){
        return result ? done(new Error(`Karma failed with error code ${result}`)):done();
    }).start();
});

Şimdi oluşturduğumuz yapılandırma dosyasıyla sunucuyu başlatıyorum, bir kez çalışmasını ve değişiklikleri izlememesini söylüyorum. Bunu bana daha uygun buluyorum, çünkü testler yalnızca ben çalışmaya hazırsam çalışır, ancak elbette farklı istiyorsanız, nerede değiştireceğinizi bilirsiniz.

Ve son kod örneğim olarak, burada Angular 2 öğreticisi “Tour of Heroes” için bir dizi test var.

import {
    TestBed,
    ComponentFixture,
    async
} from "@angular/core/testing";

import {AppComponent} from "./app.component";
import {AppModule} from "./app.module";
import Hero from "./hero/hero";

describe("App Component", function () {

    beforeEach(()=> {
        TestBed.configureTestingModule({
            imports: [AppModule]
        });
    
        this.fixture = TestBed.createComponent(AppComponent);
        this.fixture.detectChanges();
    });

    it("Should have a title", async(()=> {
        this.fixture.whenStable().then(()=> {
            expect(this.fixture.componentInstance.title).toEqual("Tour of Heros");
        });
    }));

    it("Should have a hero", async(()=> {
        this.fixture.whenStable().then(()=> {
            expect(this.fixture.componentInstance.selectedHero).toBeNull();
        });
    }));

    it("Should have an array of heros", async(()=>
        this.fixture.whenStable().then(()=> {
            const cmp = this.fixture.componentInstance;
            expect(cmp.heroes).toBeDefined("component should have a list of heroes");
            expect(cmp.heroes.length).toEqual(10, "heroes list should have 10 members");
            cmp.heroes.map((h, i)=> {
                expect(h instanceof Hero).toBeTruthy(`member ${i} is not a Hero instance. ${h}`)
            });
        })));

        it("Should have one list item per hero", async(()=>
        this.fixture.whenStable().then(()=> {
            const ul = this.fixture.nativeElement.querySelector("ul.heroes");
            const li = Array.prototype.slice.call(
                this.fixture.nativeElement.querySelectorAll("ul.heroes>li"));
            const cmp = this.fixture.componentInstance;
            expect(ul).toBeTruthy("There should be an unnumbered list for heroes");
            expect(li.length).toEqual(cmp.heroes.length, "there should be one li for each hero");
            li.forEach((li, i)=> {
                expect(li.querySelector("span.badge"))
                    .toBeTruthy(`hero ${i} has to have a span for id`);
                expect(li.querySelector("span.badge").textContent.trim())
                    .toEqual(cmp.heroes[i].id.toString(), `hero ${i} had wrong id displayed`);
                expect(li.textContent)
                    .toMatch(cmp.heroes[i].name, `hero ${i} has wrong name displayed`);
            });
        })));

    it("should have correct styling of hero items", async(()=>
        this.fixture.whenStable().then(()=> {
            const hero = this.fixture.nativeElement.querySelector("ul.heroes>li");
            const win = hero.ownerDocument.defaultView ||hero.ownerDocument.parentWindow;
            const styles = win.getComputedStyle(hero);
            expect(styles["cursor"]).toEqual("pointer", "cursor should be pointer on hero");
            expect(styles["borderRadius"]).toEqual("4px", "borderRadius should be 4px");
        })));

    it("should have a click handler for hero items",async(()=>
        this.fixture.whenStable().then(()=>{
            const cmp = this.fixture.componentInstance;
            expect(cmp.onSelect)
                .toBeDefined("should have a click handler for heros");
            expect(this.fixture.nativeElement.querySelector("input.heroName"))
                .toBeNull("should not show the hero details when no hero has been selected");
            expect(this.fixture.nativeElement.querySelector("ul.heroes li.selected"))
                .toBeNull("Should not have any selected heroes at start");

            spyOn(cmp,"onSelect").and.callThrough();
            this.fixture.nativeElement.querySelectorAll("ul.heroes li")[5].click();

            expect(cmp.onSelect)
                .toHaveBeenCalledWith(cmp.heroes[5]);
            expect(cmp.selectedHero)
                .toEqual(cmp.heroes[5], "click on hero should change hero");
        })
    ));
});

Burada dikkate değer olan şey, beforeEach() bir test modülünü nasıl yapılandırdığımız ve testte bileşeni nasıl oluşturduğumuz ve açısalın gerçekten çift bağlamadan geçmesi için detectChanges() olarak nasıl adlandırdığımızdır.

Her testin async() için bir çağrı olduğuna ve fikstürü incelemeden önce her zaman WhenStable sözünün çözülmesini beklediğine dikkat edin. Daha sonra componentInstance aracılığıyla bileşene ve nativeElement aracılığıyla öğeye erişimi olur.

Doğru stili kontrol eden bir test var. Eğiticinin bir parçası olarak, Angular ekibi bileşenlerin içindeki stillerin kullanımını gösterir. Testimizde, stillerin belirttiğimiz yerden geldiğini kontrol etmek için getComputedStyle() kullanıyoruz, ancak bunun için Window nesnesine ihtiyacımız var ve testte gördüğünüz gibi onu elementten alıyoruz.

Http Hizmetini Test Etme #

Genellikle, hizmetler veri almak/göndermek için uzak Api’yi arar. Ancak birim testleri ağ aramaları yapmamalıdır. Angular, http isteklerini yapmak için dahili olarak XHRBackend sınıfını kullanır. Kullanıcı, davranışı değiştirmek için bunu geçersiz kılabilir. Açısal test modülü, http isteklerini test etmek ve onaylamak için kullanılabilecek “MockBackend” ve “MockConnection” sınıfları sağlar.

“posts.service.ts” Bu hizmet, gönderilerin listesini almak için bir API uç noktasına ulaşır.

import { Http } from '@angular/http';
import { Injectable } from '@angular/core';
import { Observable }     from 'rxjs/rx';

import 'rxjs/add/operator/map';

export interface IPost {
    userId: number;
    id: number;
    title: string;
    body: string;
}

@Injectable()
export class PostsService {
    posts: IPost[];

    private postsUri = 'http://jsonplaceholder.typicode.com/posts';

    constructor(private http: Http) {
    }

    get(): Observable<IPost[]> {
        return this.http.get(this.postsUri)
                .map((response) => response.json());
    }
}

“posts.service.spec.ts” Burada, http api çağrılarını taklit ederek yukarıdaki hizmeti test edeceğiz.

import { TestBed, inject, fakeAsync } from '@angular/core/testing';
import {
    HttpModule,
    XHRBackend,
    ResponseOptions,
    Response,
    RequestMethod
} from '@angular/http';
import {
    MockBackend,
    MockConnection
} from '@angular/http/testing';

import { PostsService } from './posts.service';

describe('PostsService', () => {
    // Mock http response
    const mockResponse = [
        {
            'userId': 1,

Açısal Bileşenleri Test Etme - Temel #

Bileşen kodu aşağıdaki gibi verilmiştir.

import { Component } from '@angular/core';

@Component({

Yukarıdaki örnekte, bileşen varlığı için test durumunu açıklayan yalnızca bir test durumu vardır. Yukarıdaki örnekte “TestBed” ve “ComponentFixture” gibi açısal test araçları kullanılmıştır.

Açısal test modülünü oluşturmak için TestBed kullanılır ve test etmek istediğimiz sınıf için modül ortamını üretmek için bu modülü configureTestingModule metodu ile yapılandırırız. Her test senaryosunun yürütülmesinden önce yapılandırılacak test modülü, bu nedenle test modülünü ‘öncesi’ işlevinde yapılandırıyoruz.

Test edilen bileşenin örneğini oluşturmak için “TestBed"in “createComponent” yöntemi kullanılır. ‘createComponent’, ‘ComponentFixture’ı döndürür. Fikstür, bileşen örneğinin kendisine erişim sağlar.
