Yazılım yazmak için alınabilecek bir yaklaşım, gerektiğinde bağımlılıklar oluşturmaktır. Bu, bir program yazmanın oldukça sezgisel bir yoludur ve kısmen takip edilmesi kolay olduğu için çoğu insanın öğretilme eğiliminde olduğu yoldur. Bu yaklaşımla ilgili sorunlardan biri, test etmenin zor olabileceğidir. Geçerli tarihe göre bazı işlemler yapan bir yöntem düşünün. Yöntem, aşağıdaki gibi bazı kodlar içerebilir:

if (DateTime.Now.Date > processDate)
{
    // Do some processing
}

Kodun geçerli tarihe doğrudan bağımlılığı vardır. Bu yöntemi test etmek zor olabilir çünkü mevcut tarih kolayca değiştirilemez. Kodu daha test edilebilir hale getirmeye yönelik bir yaklaşım, geçerli tarihe doğrudan referansı kaldırmak ve bunun yerine geçerli tarihi işlemeyi yapan yönteme sağlamak (veya enjekte etmektir). Bu bağımlılık enjeksiyonu, birim testinin kurulum adımını basitleştirmek için doubles kullanarak kodun özelliklerini test etmeyi çok daha kolaylaştırabilir.

IOC sistemleri

Dikkate alınması gereken bir diğer husus, bağımlılıkların ömrüdür; sınıfın kendi bağımlılıklarını (değişmezler olarak da bilinir) oluşturması durumunda, bunların ortadan kaldırılmasından sorumludur. Bağımlılık Enjeksiyonu bunu tersine çevirir (ve bu nedenle bir enjeksiyon kitaplığına genellikle “Kontrolün Tersine Çevirilmesi” sistemi olarak atıfta bulunuruz) ve bağımlılıklarını oluşturmaktan, yönetmekten ve temizlemekten sorumlu olan sınıf yerine, harici bir ajanın (bunda) olduğu anlamına gelir. durumda, IoC sistemi) bunun yerine yapar.

Bu, aynı sınıfın örnekleri arasında paylaşılan bağımlılıklara sahip olmayı çok daha basit hale getirir; örneğin, bir sınıfın tüketmesi için bir HTTP uç noktasından veri getiren bir hizmeti düşünün. Bu hizmet durumsuz olduğundan (yani herhangi bir dahili durumu olmadığından), bu nedenle uygulamamız boyunca gerçekten bu hizmetin yalnızca tek bir örneğine ihtiyacımız var. Bunu manuel olarak yapmak (örneğin, statik bir sınıf kullanarak) mümkün olsa da, sınıfı oluşturmak ve IoC sistemine bunun bir Singleton olarak oluşturulacağını söylemek çok daha kolaydır, bu sayede her bir sınıfın yalnızca bir örneği var.

Başka bir örnek, bir web uygulamasında kullanılan veritabanı bağlamları olabilir; burada, bir denetleyici örneği başına değil, istek (veya iş parçacığı) başına yeni bir Bağlam gerekir; bu, bağlamın o iş parçacığı tarafından yürütülen her katmana manuel olarak iletilmesine gerek kalmadan enjekte edilmesini sağlar.

Bu, tüketen sınıfları bağımlılıkları yönetmekten kurtarır.

Yapıcı Enjeksiyonu #

Yapıcı enjeksiyonu, tüm sınıfın bağlı olduğu bağımlılıkları enjekte etmenin en güvenli yoludur. Bu tür bağımlılıklar genellikle değişmezler olarak adlandırılır, çünkü sınıfın bir örneği, bunlar sağlanmadan oluşturulamaz. Bağımlılığın inşaatta enjekte edilmesini gerektirerek, bir nesnenin tutarsız bir durumda oluşturulamayacağı garanti edilir.

Hata koşullarında bir günlük dosyasına yazması gereken bir sınıf düşünün. Gerektiğinde enjekte edilebilen ve kullanılabilen bir “ILogger"a bağımlıdır.

public class RecordProcessor
{
    readonly private ILogger _logger;

    public RecordProcessor(ILogger logger)
    {
        _logger = logger;
    }

    public void DoSomeProcessing() {
        // ...
        _logger.Log("Complete");
    }
}

Bazen testler yazarken, yapıcının test edilen bir durum için gerçekte ihtiyaç duyulandan daha fazla bağımlılık gerektirdiğini not edebilirsiniz. Bu tür testler ne kadar çok olursa, sınıfınızın Tek Sorumluluk İlkesini (SRP) ihlal etme olasılığı o kadar artar. Bu nedenle, potansiyel uyarı sinyalini maskeleyebileceğinden, test sınıfı başlatma aşamasında enjekte edilen tüm bağımlılıklar için varsayılan davranışı tanımlamak çok iyi bir uygulama değildir.

Bunun için en iyi birim aşağıdaki gibi görünecektir:

[Test]
public void RecordProcessor_DependencyInjectionExample()
{
    ILogger logger = new FakeLoggerImpl(); //or create a mock by a mocking Framework
    
    var sut = new RecordProcessor(logger); //initialize with fake impl in testcode

    Assert.IsTrue(logger.HasCalledExpectedMethod());
}

Özellik Enjeksiyonu #

Özellik ekleme, oluşturulduktan sonra bir sınıf bağımlılıklarının güncellenmesine izin verir. Bu, nesne oluşturmayı basitleştirmek, ancak yine de test çiftleriyle yapılan testleriniz tarafından bağımlılıkların geçersiz kılınmasına izin vermek istiyorsanız yararlı olabilir.

Hata durumunda bir günlük dosyasına yazması gereken bir sınıf düşünün. Sınıf, varsayılan bir ‘Logger’ın nasıl oluşturulacağını bilir, ancak özellik enjeksiyonu yoluyla geçersiz kılınmasına izin verir. Bununla birlikte, özellik enjeksiyonunu bu şekilde kullanarak, bu verilen örnekte ‘ConcreteLogger’ olan ‘ILogger’ın tam bir uygulamasıyla bu sınıfı sıkı bir şekilde bağladığınızı belirtmekte fayda var. Olası bir geçici çözüm, gerekli ILogger uygulamasını döndüren bir fabrika olabilir.

public class RecordProcessor
{
    public RecordProcessor()
    {
        Logger = new ConcreteLogger();
    }

    public ILogger Logger { get; set; }

    public void DoSomeProcessing()
    {
        // ...
        _logger.Log("Complete");
    }
}

Çoğu durumda Yapıcı Enjeksiyonu, Nesnenin yapımından hemen sonra durumu hakkında daha iyi garantiler sağladığı için Özellik Enjeksiyonuna tercih edilir.

Yöntem Enjeksiyonu #

Yöntem enjeksiyonu, bağımlılıkları işlemeye enjekte etmenin ince taneli bir yoludur. Geçerli tarihe göre bazı işlemler yapan bir yöntem düşünün. Geçerli tarihi bir testten değiştirmek zordur, bu nedenle test etmek istediğiniz yönteme bir tarih iletmek çok daha kolaydır.

public void ProcessRecords(DateTime currentDate)
{
    foreach(var record in _records) 
    {
        if (currentDate.Date > record.ProcessDate)
        {
            // Do some processing
        }
    }
}

Konteynerler / DI Çerçeveleri #

Bağımlılıkları enjekte edilebilmesi için kodunuzdan çıkarmak, kodunuzu test etmeyi kolaylaştırırken, sorunu hiyerarşide daha yukarıya iter ve ayrıca oluşturulması zor nesnelerle sonuçlanabilir. Bu sorunun üstesinden gelmeye yardımcı olmak için çeşitli bağımlılık ekleme çerçeveleri / Kontrol Kapsayıcılarının Tersine Çevirilmesi yazılmıştır. Bunlar, tip eşlemelerinin kaydedilmesine izin verir. Bu kayıtlar daha sonra kapsayıcıdan bir nesne oluşturması istendiğinde bağımlılıkları çözmek için kullanılır.

Bu sınıfları göz önünde bulundurun:

public interface ILogger {
    void Log(string message);
}

public class ConcreteLogger : ILogger
{
    public ConcreteLogger()
    {
        // ...
    }
    public void Log(string message)
    {
        // ...
    }
}
public class SimpleClass
{
    public SimpleClass()
    {
        // ...
    }
}

public class SomeProcessor
{
    public SomeProcessor(ILogger logger, SimpleClass simpleClass)
    {
        // ...
    }
}

“SomeProcessor” oluşturmak için hem “ILogger” hem de “SimpleClass” örneği gereklidir. Unity gibi bir kapsayıcı bu işlemi otomatikleştirmeye yardımcı olur.

Önce kapsayıcının inşa edilmesi gerekiyor ve ardından eşlemeler onunla kaydediliyor. Bu genellikle bir uygulama içinde yalnızca bir kez yapılır. Bunun meydana geldiği sistem alanı genellikle Bileşim Kökü olarak bilinir.

// Register the container
var container = new UnityContainer();

// Register a type mapping.  This allows a `SimpleClass` instance
// to be constructed whenever it is required.
container.RegisterType<SimpleClass, SimpleClass>();

// Register an instance.  This will use this instance of `ConcreteLogger` 
// Whenever an `ILogger` is required.
container.RegisterInstance<ILogger>(new ConcreteLogger());

Kap yapılandırıldıktan sonra, gerektiğinde bağımlılıkları otomatik olarak çözerek nesneler oluşturmak için kullanılabilir:

var processor = container.Resolve<SomeProcessor>();
