Alaylı nesneye davranış ekleyin #

Mockito.when(mock.returnSomething()).thenReturn("my val");

mock.returnSomething(); // returns "my val"
mock.returnSomething(); // returns "my val" again
mock.returnSomething(); // returns "my val" again and again and again...

İkinci çağrıda farklı bir değer istiyorsanız, thenReturn yöntemine istenen dönüş argümanını ekleyebilirsiniz:

Mockito.when(mock.returnSomething()).thenReturn("my val", "other val");

mock.returnSomething(); // returns "my val"
mock.returnSomething(); // returns "other val"
mock.returnSomething(); // returns "other val" again

Alay etmek için davranış eklemeden yöntemi çağırırsanız, null döndürür:

barMock.mock.returnSomethingElse(); // returns null

Alaylı yöntemin parametreleri olması durumunda, değerleri de bildirmelisiniz:

Mockito.when(mock.returnSomething("param 1")).thenReturn("my val 1");
Mockito.when(mock.returnSomething("param 2")).thenReturn("my val 2");

mock.returnSomething("param 1"); // returns "my val 1"
mock.returnSomething("param 2"); // returns "my val 2"
mock.returnSomething("param 3"); // returns null

Param değerini umursamıyorsanız Matchers.any() kullanabilirsiniz:

Mockito.when(mock.returnSomething(Matchers.any())).thenReturn("p1");

mock.returnSomething("param 1"); // returns "p1"
mock.returnSomething("param other"); // returns "p1"

İstisna atmak için thenThrow yöntemini kullanın:

Mockito.when(mock.returnSomething()).thenThrow(new Exception());

mock.returnSomething(); // throws Exception

Mockito tarafından alay edilen nesneler oluşturun #

Mockito tarafından alay edilen nesne oluşturmanın iki yolu vardır:

  • açıklama yoluyla
  • sahte işlev aracılığıyla

Ek açıklama yoluyla:

JUnit test çalıştırıcısı ile:

@RunWith(MockitoJUnitRunner.class)
public class FooTest {
    @Mock
    private Bar barMock;

    // ...
}

Mockito’nun “MockitoJUnitRunner” ile aynı işlevselliği sağlayan ancak bir “@RunWith” test çalıştırıcısına ihtiyaç duymayan JUnit “@Rule” özelliğini de kullanabilirsiniz:

public class FooTest {
    @Rule
    public MockitoRule mockito = MockitoJUnit.rule();        

    @Mock
    private Bar barMock;

    // ...
}

‘@RunWith’ veya ‘@Rule’ ek açıklamasını kullanamıyorsanız, “el başına” alayları da başlatabilirsiniz:

public class FooTest {
    @Mock
    private Bar barMock;

    @Before
    public void setUp() {
        MockitoAnnotations.initMocks(this);
    }

    // ...
}

Sahte işlev aracılığıyla:

public class FooTest {
    private Bar barMock = Mockito.mock(Bar.class);

    // ...
}

Tür silme nedeniyle, genel bir sınıfla yukarıdaki gibi alay edemezsiniz. Temel sınıfla alay etmeli ve açıkça doğru genel türe aktarmalısınız:

public class FooTest {
    private Bar<String> genericBarMock = (Bar<String>) Mockito.mock(Bar.class);

    // ...
}

Sahte nesnede yöntem çağrılarını doğrulayın #

Sahte bir nesnede bir yöntemin çağrıldığını kontrol etmek için “Mockito.verify” yöntemini kullanabilirsiniz:

Mockito.verify(someMock).bla();

Bu örnekte, “someMock” sahte nesnesinde “bla” yönteminin çağrıldığını iddia ediyoruz.

Belirli parametrelerle bir yöntemin çağrıldığını da kontrol edebilirsiniz:

Mockito.verify(someMock).bla("param 1");

Bir yöntemin çağrılmadığını kontrol etmek isterseniz, “doğrulama"ya ek bir “DoğrulamaModu” parametresi iletebilirsiniz:

Mockito.verify(someMock, Mockito.times(0)).bla();

Bu aynı zamanda, bu yöntemin birden fazla kez çağrıldığını kontrol etmek isterseniz de işe yarar (bu durumda, bla yönteminin 23 kez çağrıldığını kontrol ederiz):

Mockito.verify(someMock, Mockito.times(23)).bla();

Bunlar, bir yöntemin çağrılma sayısı üzerinde daha fazla kontrol sağlayan “VerificationMode” parametresi için daha fazla örnektir:

Mockito.verify(someMock, Mockito.never()).bla(); // same as Mockito.times(0)

Mockito.verify(someMock, Mockito.atLeast(3)).bla(); // min 3 calls

Mockito.verify(someMock, Mockito.atLeastOnce()).bla(); // same as Mockito.atLeast(1)

Mockito.verify(someMock, Mockito.atMost(3)).bla(); // max 3 calls

Mockito kullanarak basit birim testi #

Test edeceğimiz sınıf:

public class Service {

    private Collaborator collaborator;

    public Service(Collaborator collaborator) {
        this.collaborator = collaborator;
    }
    
    public String performService(String input) {
        return collaborator.transformString(input);
    }
}

İşbirlikçisi:

public class Collaborator {

    public String transformString(String input) {
        return doStuff();
    }

    private String doStuff() {
        // This method may be full of bugs
        . . .
        return someString;
    }

}

Testimizde, ‘Collaborator’ ve hatalarından olan bağımlılığı kırmak istiyoruz, bu yüzden ‘Collaborator’ ile alay edeceğiz:

import static org.junit.Assert.*;
import static org.mockito.Mockito.*;

import org.junit.Test;

public class ServiceTest {
    @Test
    public void testPerformService() throws Exception {
        // Configure mock
        Collaborator collaboratorMock = mock(Collaborator.class);
        doReturn("output").when(collaboratorMock).transformString("input");

        // Perform the test
        Service service = new Service(collaboratorMock);
        String actual = service.performService("input");
        
        // Junit asserts
        String expected = "output";
        assertEquals(expected, actual);
    }  
}

Mockito ek açıklamalarını kullanma #

Test edeceğimiz sınıf:

public class Service{

    private Collaborator collaborator;

    public Service(Collaborator collaborator){
        this.collaborator = collaborator;
    }
    
    
    public String performService(String input){
        return collaborator.transformString(input);
    }
}

İşbirlikçisi:

public class Collaborator {

    public String transformString(String input){
        return doStuff();
    }

    private String doStuff()
    {
        // This method may be full of bugs
        . . .
        return someString;
    }

}

Testimizde, ‘Collaborator’ ve hatalarından olan bağımlılığı kırmak istiyoruz, bu yüzden ‘Collaborator’ ile alay edeceğiz. “@Mock” ek açıklamasını kullanmak, her test için farklı deneme örnekleri oluşturmanın uygun bir yoludur:

import static org.junit.Assert.*;
import static org.mockito.Mockito.*;

import org.junit.Test;
import org.junit.runner.RunWith;
import org.mockito.Mock;
import org.mockito.InjectMocks;
import org.mockito.runners.MockitoJUnitRunner;

@RunWith(MockitoJUnitRunner.class)
public class ServiceTest {

    @Mock
    private Collaborator collaboratorMock;

    @InjectMocks
    private Service service;
    
    @Test
    public void testPerformService() throws Exception {
        // Configure mock
        doReturn("output").when(collaboratorMock).transformString("input");            

        // Perform the test
        String actual = service.performService("input");
        
        // Junit asserts
        String expected = "output";
        assertEquals(expected, actual);
    }
    
    
    @Test(expected=Exception.class)
    public void testPerformServiceShouldFail() throws Exception {
        // Configure mock
        doThrow(new Exception()).when(collaboratorMock).transformString("input");

        // Perform the test
        service.performService("input");
    }
}

Mockito, bağımlılık enjeksiyonunu aşağıdaki sırayla çözmeye çalışacaktır:

  1. Yapıcı tabanlı enjeksiyon - çoğu argümanla yapıcıya sahteler enjekte edilir (bazı argümanlar bulunamazsa, boş değerler iletilir). Yapıcı aracılığıyla bir nesne başarıyla oluşturulduysa, başka hiçbir strateji uygulanmaz.
  2. Setter tabanlı enjeksiyon - sahteler türe göre enjekte edilir. Aynı türden birkaç özellik varsa, özellik adları ve sahte adlar eşleştirilecektir.
  3. Doğrudan alan enjeksiyonu - setter bazlı enjeksiyon ile aynı.

Yukarıda belirtilen stratejilerden herhangi birinin başarısız olması durumunda herhangi bir başarısızlık rapor edilmediğini unutmayın.

Mockito’nun en son sürümündeki bu mekanizma hakkında daha ayrıntılı bilgi için lütfen en son @InjectMocks adresine bakın.

Kurulum ve kurulum #

Kurulum #

Mockito’yu kurmanın tercih edilen yolu, tercih edilen bir yapı sistemi ile “mockito-core” bağımlılığını bildirmektir. 22 Temmuz 2016 itibariyle, beta olmayan en son sürüm 1.10.19’dur, ancak 2.x’in ’e taşınması zaten teşvik edilmektedir.

Uzman

<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <version>1.10.19</version>
    <scope>test</scope>
</dependency>

Kepçe

repositories { jcenter() }
dependencies { testCompile "org.mockito:mockito-core:1.+" }

Mockito’nun yanı sıra Hamcrest ve Objenesis içeren ‘mockito-all’ da var. Özellikle karınca kullanıcıları için Maven aracılığıyla teslim edilir, ancak Mockito 2.x’te dağıtım durdurulmuştur.

İçe aktarmak #

Mockito tesislerinin çoğu, ‘org.mockito.Mockito’nun statik yöntemleridir. Böylece, Mockito statik olarak bir sınıfa şu şekilde aktarılabilir:

import static org.mockito.Mockito.*;

Dokümantasyon giriş noktası bu sınıfın javadoc içinde bulunur.

Bir nesne üzerinde bazı yöntemlerle alay edin #

Basit minimal Mockito Testi #

Bu örnek, alaylı bir “ArrayList” kullanan minimal bir Mockito testini gösterir:

import static org.mockito.Mockito.*;
import static org.junit.Assert.*;

import java.util.ArrayList;

import org.junit.Test;
import org.junit.runner.RunWith;
import org.mockito.Mock;
import org.mockito.runners.MockitoJUnitRunner;

@RunWith(MockitoJUnitRunner.class)
public class MockitoTest
{
    @Mock
    ArrayList<String> listMock;

    @Test
    public void testAppend() {
        // configure the mock to return "foobar" whenever "get()" 
        // is called on "listMock" with an int value as parameter
        doReturn("foobar").when(listMock).get(anyInt());            
        String result = listMock.get(0);
        
        assertEquals("foobar", result);
    }
}

ArgumentCaptor ile argümanları doğrulama #

Sahte olarak çağrılan yöntemlere ilişkin argümanları doğrulamak için ‘ArgumentCaptor’ sınıfını kullanın. Bu, argümanları test yönteminize çıkarmanıza ve bunlar üzerinde iddialar gerçekleştirmenize olanak tanır.

Bu örnek, belirli bir kimliğe sahip bir kullanıcının adını güncelleyen bir yöntemi test eder. Yöntem kullanıcıyı yükler, name niteliğini verilen değerle günceller ve daha sonra kaydeder. Test, “save” yöntemine iletilen argümanın, doğru kimlik ve ada sahip bir “Kullanıcı” nesnesi olduğunu doğrulamak ister.

// This is mocked in the test
interface UserDao {
    void save(User user);
}

@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    UserDao userDao;

    @Test
    public void testSetNameForUser() {
        UserService serviceUnderTest = new UserService(userDao);
        
        serviceUnderTest.setNameForUser(1L, "John");

        ArgumentCaptor<User> userArgumentCaptor = ArgumentCaptor.forClass(User.class);
        
        verify(userDao).save(userArgumentCaptor.capture());
        User savedUser = userArgumentCaptor.getValue();
        assertTrue(savedUser.getId() == 1);
        assertTrue(savedUser.getName().equals("John"));
    }
}

ArgumentMatcher ile Argümanları Doğrulama #

Mockito, bağımsız değişkenleri doğrulamak için soyut bir “ArgumentMatcher” sınıfıyla birlikte bir “Matcher” arabirimi sağlar. Aynı kullanım durumu için “ArgumentCaptor"dan farklı bir yaklaşım kullanır. Ek olarak, ArgumentMatcher alay etmede de kullanılabilir. Her iki kullanım durumu da makul ölçüde okunabilir bir test kodu sağlayan Mockito.argThat() yöntemini kullanır.

verify(someMock).someMethod(Mockito.argThat(new ArgumentMatcher<String>() {
       
    @Override
    public boolean matches(Object o) {
        return o instanceof String && !((String)o).isEmpty();
    }

});

ArgumentMatcher’ın JavaDocs’ından:

Uyarı: Testi daha az okunabilir hale getirebileceğinden, özellikle özel bağımsız değişken eşleştiriciler olmak üzere karmaşık bağımsız değişken eşleştirmeyi kullanırken makul olun. Bazen, taklitlere iletilen argümanlar için equals() uygulamak daha iyidir (Mockito, argüman eşleştirme için doğal olarak equals() kullanır). Bu, testi daha temiz hale getirebilir.
