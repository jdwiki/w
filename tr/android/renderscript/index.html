RenderScript, yüksek performanslı grafik işleme ve ham hesaplama kodu yazmanıza izin veren bir betik dilidir. Sistemin daha sonra üzerinde çalışabileceği işlemci için yerel koda derlediği performans açısından kritik kod yazma aracı sağlar. Bu, CPU, çok çekirdekli bir CPU veya hatta GPU olabilir. Sonunda çalıştığı şey, geliştiricinin kolayca erişemeyeceği birçok faktöre bağlıdır, ancak aynı zamanda dahili platform derleyicisinin hangi mimariyi desteklediğine de bağlıdır.

Başlarken #

RenderScript, Android’de yüksek performanslı paralel hesaplamaya izin veren bir çerçevedir. Yazdığınız komut dosyaları, mevcut tüm işlemcilerde (örn.

Komut dosyaları C99 tabanlı bir dilde yazılır (C99, C programlama dili standardının eski bir sürümüdür). Her Komut Dosyası için, Java kodunuzdaki RenderScript ile kolayca etkileşim kurmanıza olanak tanıyan bir Java sınıfı oluşturulur.

Projenizi kurmak #

Android Framework kitaplıkları veya Destek Kitaplığı ile uygulamanızda RenderScript’e erişmenin iki farklı yolu vardır. API seviyesi 11’den önce cihazları hedeflemek istemeseniz bile, birçok farklı cihaz arasında cihaz uyumluluğu sağladığı için her zaman Destek Kitaplığı uygulamasını kullanmalısınız. Destek kitaplığı uygulamasını kullanmak için en azından ‘18.1.0’ sürüm oluşturma araçları kullanmanız gerekir!

Şimdi uygulamanızın build.gradle dosyasını kuralım:

android {
    compileSdkVersion 24
    buildToolsVersion '24.0.1'

    defaultConfig {
        minSdkVersion 8
        targetSdkVersion 24

        renderscriptTargetApi 18
        renderscriptSupportModeEnabled true
    }
}
  • renderscriptTargetApi: Bu, ihtiyacınız olan tüm RenderScript işlevlerini sağlayan en eski API düzeyine ayarlanmalıdır.
  • renderscriptSupportModeEnabled : Bu, Destek Kitaplığı RenderScript uygulamasının kullanılmasını sağlar.

RenderScript nasıl çalışır? #

Tipik bir RenderScript iki şeyden oluşur: Çekirdekler ve İşlevler. Bir fonksiyon kulağa nasıl geliyorsa öyledir - bir girdiyi kabul eder, o girdiyle bir şeyler yapar ve bir çıktı döndürür. Çekirdek, RenderScript’in gerçek gücünün geldiği yerdir.

Çekirdek, bir “Ayırma” içindeki her öğeye karşı yürütülen bir işlevdir. Bir ‘Ayırma’, ‘Bitmap’ veya ‘byte’ dizisi gibi verileri bir ‘RenderScript’e geçirmek için kullanılabilir ve ayrıca bir Çekirdekten sonuç almak için kullanılırlar. Çekirdekler ya bir “Ayırma"yı girdi, diğerini çıktı olarak alabilir ya da sadece bir “Ayırma” içindeki verileri değiştirebilirler.

Bir Çekirdeklerinizi yazabilirsiniz, ancak Gauss Görüntü Bulanıklığı gibi yaygın işlemleri gerçekleştirmek için kullanabileceğiniz önceden tanımlanmış birçok Çekirdek de vardır.

Her RenderScript dosyası için daha önce belirtildiği gibi, onunla etkileşim kurmak için bir sınıf oluşturulur. Bu sınıflar her zaman ‘ScriptC_’ önekiyle başlar ve ardından RenderScript dosyasının adı gelir. Örneğin, RenderScript dosyanıza “example” adı verilirse, oluşturulan Java sınıfı “ScriptC_example” olarak adlandırılır. Tüm önceden tanımlı Komut Dosyaları yalnızca ‘Komut Dosyası’ önekiyle başlar - örneğin Gauss Görüntü Bulanıklığı Komut Dosyası ‘ScriptIntrinsicBlur’ olarak adlandırılır.

RenderScript Tanımlama Tablosu #

RenderScript dosyaları projenizdeki src/main/rs klasöründe bulunur. Her dosya “.rs” dosya uzantısına sahiptir ve en üstte iki “#pragma” ifadesi içermelidir:

#pragma version(1)
#pragma rs java_package_name(your.package.name)

  • #pragma version(1): Bu, kullandığınız RenderScript sürümünü ayarlamak için kullanılabilir. Şu anda sadece sürüm 1 var.

  • #pragma rs java_package_name(your.package.name): Bu, bu belirli RenderScript ile etkileşim kurmak için oluşturulan Java sınıfının paket adını ayarlamak için kullanılabilir.

Genellikle RenderScript dosyalarınızın her birinde ayarlamanız gereken başka bir “#pragma” vardır ve kayan nokta hassasiyetini ayarlamak için kullanılır. Kayan nokta hassasiyetini üç farklı düzeye ayarlayabilirsiniz:

  • #pragma rs_fp_full: Bu, en yüksek hassasiyete sahip en katı ayardır ve ayrıca hiçbir şey belirtmezseniz varsayılan değerdir. Yüksek kayan nokta hassasiyetine ihtiyacınız varsa bunu kullanmalısınız.
  • #pragma rs_fp_relaxed: Bu, o kadar yüksek kayan nokta hassasiyeti sağlamaz, ancak bazı mimarilerde komut dosyalarınızın daha hızlı çalışmasına neden olabilecek bir dizi optimizasyon sağlar.
  • #pragma rs_fp_imprecise: Bu, daha da az hassasiyet sağlar ve kayan nokta hassasiyeti komut dosyanız için gerçekten önemli değilse kullanılmalıdır.

Çoğu komut dosyası, gerçekten yüksek kayan nokta hassasiyetine ihtiyacınız yoksa #pragma rs_fp_relaxed kullanabilir.

Genel Değişkenler #

Şimdi tıpkı C kodunda olduğu gibi global değişkenleri veya sabitleri tanımlayabilirsiniz:

const static float3 gMonoMult = {0.299f, 0.587f, 0.114f};

float saturationLevel = 0.0f;

“gMonoMult” değişkeni “float3” türündedir. Bu, 3 kayan sayıdan oluşan bir vektör olduğu anlamına gelir. ‘saturationValue’ olarak adlandırılan diğer ‘float’ değişkeni sabit değildir, bu nedenle çalışma zamanında istediğiniz bir değere ayarlayabilirsiniz. Bunun gibi değişkenleri Çekirdeklerinizde veya işlevlerinizde kullanabilirsiniz ve bu nedenle bunlar, RenderScript’lerinize girdi vermenin veya onlardan çıktı almanın başka bir yoludur. Her sabit olmayan değişken için, ilişkili Java sınıfında bir alıcı ve ayarlayıcı yöntemi oluşturulacaktır.

Çekirdekler #

Ama şimdi Çekirdeği uygulamaya başlayalım. Bu örneğin amaçları doğrultusunda, görüntünün doygunluğunu değiştirmek için Çekirdekte kullanılan matematiği açıklamayacağım, bunun yerine bir Çekirdeğin nasıl uygulanacağına ve nasıl kullanılacağına odaklanacağım. Bu bölümün sonunda, bu Çekirdekteki kodun gerçekte ne yaptığını hızlıca açıklayacağım.

Genel olarak çekirdekler #

Önce kaynak koduna bir göz atalım:

uchar4 __attribute__((kernel)) saturation(uchar4 in) {
    float4 f4 = rsUnpackColor8888(in);
    float3 dotVector = dot(f4.rgb, gMonoMult);
    float3 newColor = mix(dotVector, f4.rgb, saturationLevel);
    return rsPackColorTo8888(newColor);
}

Gördüğünüz gibi, bir istisna dışında normal bir C işlevi gibi görünüyor: Dönüş türü ile yöntem adı arasındaki __attribute__((çekirdek)). RenderScript’e bu yöntemin bir Çekirdek olduğunu söyleyen şey budur. Fark edebileceğiniz başka bir şey de, bu yöntemin bir “uchar4” parametresini kabul etmesi ve başka bir “uchar4” değeri döndürmesidir. “uchar4” - önceki bölümde tartıştığımız “float3” değişkeni gibi - bir vektördür. 0 ile 255 aralığında yalnızca bayt değerleri olan 4 “uchar” değeri içerir.

Bu bireysel değerlere birçok farklı yoldan erişebilirsiniz, örneğin “in.r” bir pikselin kırmızı kanalına karşılık gelen baytı döndürür. Her piksel kırmızı için ‘r’, yeşil için ‘g’, mavi için ‘b’ ve alfa için ‘a’ olmak üzere 4 değerden oluştuğu için bir ‘uchar4’ kullanıyoruz ve bunlara bu kısayol ile erişebilirsiniz. RenderScript ayrıca bir vektörden istediğiniz sayıda değer almanıza ve bunlarla başka bir vektör oluşturmanıza olanak tanır. Örneğin, “in.rgb”, alfa değeri olmadan pikselin yalnızca kırmızı, yeşil ve mavi kısımlarını içeren bir “uchar3” değeri döndürür.

Çalışma zamanında RenderScript, bir görüntünün her pikseli için bu Kernel yöntemini çağırır, bu nedenle dönüş değeri ve parametresi yalnızca bir “uchar4” değeridir. RenderScript, bu çağrıların çoğunu mevcut tüm işlemcilerde paralel olarak çalıştıracaktır, bu nedenle RenderScript bu kadar güçlüdür. Bu aynı zamanda, iş parçacığı oluşturma veya iş parçacığı güvenliği konusunda endişelenmenize gerek olmadığı, her piksele yapmak istediğinizi uygulayabileceğiniz ve gerisini RenderScript’in halledeceği anlamına gelir.

Java’da bir Çekirdeği çağırırken, biri girdi verilerini içeren ve diğeri çıktıyı alacak olan iki ‘Ayırma’ değişkeni sağlarsınız. Kernel yönteminiz ‘Ayırma’ girişindeki her değer için çağrılacak ve sonucu ‘Ayırma’ çıkışına yazacaktır.

RenderScript Runtime API yöntemleri #

Yukarıdaki Çekirdekte, kutunun dışında sağlanan birkaç yöntem kullanılır. RenderScript bu tür birçok yöntem sağlar ve bunlar RenderScript ile yapacağınız hemen hemen her şey için hayati öneme sahiptir. Bunlar arasında sin() gibi matematik işlemleri yapma yöntemleri ve iki değeri başka değerlere göre karıştıran ‘mix()’ gibi yardımcı yöntemler vardır. Ancak vektörler, kuaterniyonlar ve matrislerle uğraşırken daha karmaşık işlemler için yöntemler de vardır.

Resmi RenderScript Runtime API Reference, belirli bir konu hakkında daha fazla bilgi edinmek istiyorsanız bulabileceğiniz en iyi kaynaktır. veya bir matrisin nokta çarpımını hesaplamak gibi ortak bir işlemi gerçekleştiren belirli bir yöntem arıyorlar. Bu belgeleri burada bulabilirsiniz.

Çekirdek Uygulaması #

Şimdi bu Çekirdeğin ne yaptığının özelliklerine bir göz atalım. İşte Kernel’deki ilk satır:

float4 f4 = rsUnpackColor8888(in);

İlk satır, ` uchar4’ değerini bir “float4” değerlerine dönüştürün. Her renk kanalı ayrıca “0.0f - 1.0f” aralığına dönüştürülür; burada “0.0f”, “0” ve “1.0f” ila “255” bayt değerine karşılık gelir. Bunun temel amacı, bu Çekirdekteki tüm matematiği çok daha basit hale getirmektir.

float3 dotVector = dot(f4.rgb, gMonoMult);

Bu sonraki satır, noktayı hesaplamak için yerleşik dot() yöntemini kullanır. iki vektörün çarpımı. “gMonoMult”, yukarıda birkaç bölüm tanımladığımız sabit bir değerdir. Nokta çarpımını hesaplamak için her iki vektörün de aynı uzunlukta olması gerektiğinden ve ayrıca bir pikselin alfa kanalını değil sadece renk kanallarını etkilemek istediğimizden, yeni bir “float3” vektörü elde etmek için “.rgb” kısayolunu kullanırız. sadece kırmızı, yeşil ve mavi renk kanallarını içerir. Nokta çarpımının nasıl çalıştığını okuldan hala hatırlayanlarımız, nokta çarpımının bir vektör değil sadece bir değer döndürmesi gerektiğini hemen fark edeceklerdir. Yine de yukarıdaki kodda sonucu bir ‘float3’ vektörüne atadık. Bu yine RenderScript’in bir özelliğidir. Bir vektöre tek boyutlu bir sayı atadığınızda, vektördeki tüm öğeler bu değere ayarlanacaktır. Örneğin, aşağıdaki snippet, “float3” vektöründeki üç değerin her birine “2.0f” değerini atayacaktır:

float3 example = 2.0f;

Bu nedenle, yukarıdaki nokta çarpımının sonucu, yukarıdaki ‘float3’ vektöründeki her bir öğeye atanır.

Şimdi, görüntünün doygunluğunu değiştirmek için global saurationLevel değişkenini kullandığımız kısım geliyor:

float3 newColor = mix(dotVector, f4.rgb, saturationLevel);

Bu, orijinal rengi bir araya getirmek için yerleşik mix() yöntemini kullanır. yukarıda oluşturduğumuz nokta çarpım vektörü ile. Bunların nasıl karıştırıldığı, global “saurationLevel” değişkeni tarafından belirlenir. Dolayısıyla, “0.0f” değerinde bir “doyma Düzeyi”, ortaya çıkan rengin orijinal renk değerlerinin hiçbir parçası olmamasına neden olur ve yalnızca “dotVector” içindeki değerlerden oluşur, bu da siyah beyaz veya grileşmiş bir görüntüyle sonuçlanır. ‘1.0f’ değeri, ortaya çıkan rengin tamamen orijinal renk değerlerinden oluşmasına neden olur ve ‘1.0f’ üzerindeki değerler orijinal renkleri çarparak onları daha parlak ve yoğun hale getirir.

return rsPackColorTo8888(newColor);

Bu, Kernel’deki son bölümdür. rsPackColorTo8888() “float3” vektörünü tekrar “uchar4"e dönüştürür daha sonra döndürülen değer. Ortaya çıkan bayt değerleri 0 ile 255 arasında bir aralığa sıkıştırılır, bu nedenle “1.0f"den yüksek kayan değerler 255 bayt değeriyle ve “0.0"dan düşük değerler bayt değeri “0” ile sonuçlanır.

Ve tüm Çekirdek uygulaması budur. Artık sadece bir kısım kaldı: Java’da Çekirdek nasıl çağırılır.

Java’da RenderScript’i çağırma #

Temel Bilgiler #

Her RenderScript dosyası için yukarıda belirtildiği gibi, betiklerinizle etkileşime girmenizi sağlayan bir Java sınıfı oluşturulur. Bu dosyalar ScriptC_ ön ekine ve ardından RenderScript dosyasının adına sahiptir. Bu sınıfların bir örneğini oluşturmak için öncelikle ‘RenderScript’ sınıfının bir örneğine ihtiyacınız var:

final RenderScript renderScript = RenderScript.create(context);

Statik yöntem “create()”, bir “Bağlam"dan bir “RenderScript” örneği oluşturmak için kullanılabilir. Ardından, betiğiniz için oluşturulan Java sınıfını başlatabilirsiniz. RenderScript dosyasını “saturation.rs” olarak adlandırdıysanız, sınıf “ScriptC_saturation” olarak adlandırılır:

final ScriptC_saturation script = new ScriptC_saturation(renderScript);

Bu sınıfta artık doygunluk seviyesini ayarlayabilir ve Çekirdeği arayabilirsiniz. ‘saturationLevel’ değişkeni için oluşturulan ayarlayıcı, ‘set_’ önekini ve ardından değişkenin adını alacaktır:

script.set_saturationLevel(1.0f);

Ayrıca, halihazırda ayarlanmış olan doygunluk seviyesini elde etmenize izin veren ‘get_’ ön ekine sahip bir alıcı vardır:

float saturationLevel = script.get_saturationLevel();

RenderScript’inizde tanımladığınız çekirdeklerin önüne ‘forEach_’ ve ardından Kernel yönteminin adı gelir. Yazdığımız Çekirdek, parametreleri olarak bir “Ayırma” girdisi ve bir “Ayırma” çıktısı bekliyor:

script.forEach_saturation(inputAllocation, outputAllocation);

“Ayırma” girdisinin girdi görüntüsünü içermesi gerekir ve “forEach_sauration” yöntemi bittikten sonra çıktı tahsisi değiştirilmiş görüntü verilerini içerecektir.

Bir “Ayırma” örneğine sahip olduğunuzda, “copyFrom()” ve “copyTo()” yöntemlerini kullanarak bu “Ayırmalar"dan veri kopyalayabilirsiniz. Örneğin, `Ayırma girişinize yeni bir resim kopyalayabilirsiniz:

inputAllocation.copyFrom(inputBitmap);

Aynı şekilde, “Allocation” çıktısında “copyTo()” öğesini çağırarak sonuç görüntüsünü alabilirsiniz:

outputAllocation.copyTo(outputBitmap);

Tahsis örnekleri oluşturma #

Bir “Ayırma” oluşturmanın birçok yolu vardır. Bir “Ayırma” örneğine sahip olduğunuzda, yukarıda açıklandığı gibi “copyTo()” ve “copyFrom()” ile bu “Ayırmalar"dan yeni verileri kopyalayabilirsiniz, ancak bunları oluşturmak için başlangıçta ne tür verilerle bilmeniz gerekir. tam olarak birlikte çalışıyorsun. “Ayırma” girişiyle başlayalım:

Bir “Bitmap"ten “Ayırma” girdimizi hızlı bir şekilde oluşturmak için “createFromBitmap()” statik yöntemini kullanabiliriz:

final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, image);

Bu örnekte giriş görüntüsü asla değişmez, bu nedenle Ayırma girişini bir daha asla değiştirmemiz gerekmez. Yeni bir “Bitmap” çıktısı oluşturmak için “doyma Düzeyi” her değiştiğinde onu yeniden kullanabiliriz.

“Ayırma” çıktısını oluşturmak biraz daha karmaşıktır. İlk önce ‘Tür’ denilen şeyi yaratmamız gerekiyor. Bir “Tür”, bir “Ayırma"ya ne tür verilerle uğraştığını söylemek için kullanılır. Genellikle uygun bir ‘Type’ hızlı bir şekilde oluşturmak için ‘Type.Builder’ sınıfını kullanır. Önce koda bir göz atalım:

final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

4 renk kanalı ile piksel başına Bitmap normal 32 bit (ya da diğer bir deyişle 4 bayt) ile çalışıyoruz. Bu nedenle ‘Type’ oluşturmak için ‘Element.RGBA_8888’i seçiyoruz. Ardından, çıktı görüntüsünün genişliğini ve yüksekliğini giriş görüntüsüyle aynı boyuta ayarlamak için setX() ve setY() yöntemlerini kullanırız. create() yöntemi daha sonra belirttiğimiz parametrelerle Typeı oluşturur.

Doğru ‘Tür’e sahip olduğumuzda, ‘createTyped()’ statik yöntemiyle ‘Ayırma’ çıktısını oluşturabiliriz:

final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

Şimdi neredeyse bitirdik. Ayrıca, “Ayırma” çıktısından verileri kopyalayabileceğimiz bir “Bitmap” çıktısına da ihtiyacımız var. Bunu yapmak için, “Bitmap” girişiyle aynı boyut ve konfigürasyona sahip yeni bir boş “Bitmap” oluşturmak için “createBitmap()” statik yöntemini kullanıyoruz.

final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

Ve bununla RenderScript’imizi yürütmek için tüm yapboz parçalarına sahibiz.

Tam örnek #

Şimdi tüm bunları tek bir örnekte toplayalım:

// Create the RenderScript instance
final RenderScript renderScript = RenderScript.create(context);

// Create the input Allocation 
final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, inputBitmap);

// Create the output Type.
final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

// And use the Type to create am output Allocation
final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

// Create an empty output Bitmap from the input Bitmap
final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

// Create an instance of our script
final ScriptC_saturation script = new ScriptC_saturation(renderScript);

// Set the saturation level
script.set_saturationLevel(2.0f);

// Execute the Kernel
script.forEach_saturation(inputAllocation, outputAllocation);

// Copy the result data to the output Bitmap
outputAllocation.copyTo(outputBitmap);

// Display the result Bitmap somewhere
someImageView.setImageBitmap(outputBitmap);

Çözüm #

Bu tanıtımla, basit görüntü işleme için kendi RenderScript Çekirdeklerinizi yazmaya hazır olmalısınız. Ancak aklınızda bulundurmanız gereken birkaç şey var:

  • RenderScript yalnızca Uygulama projelerinde çalışır: Şu anda RenderScript dosyaları bir kitaplık projesinin parçası olamaz.
  • Hafızaya dikkat edin: RenderScript çok hızlıdır, ancak aynı zamanda yoğun hafızaya sahip olabilir. Herhangi bir zamanda asla birden fazla “RenderScript” örneği olmamalıdır. Ayrıca mümkün olduğunca tekrar kullanmalısınız. Normalde, “Ayırma” örneklerinizi bir kez oluşturmanız yeterlidir ve bunları gelecekte yeniden kullanabilirsiniz. Aynı şey “Bitmap” çıktısı veya komut dosyası örnekleriniz için de geçerlidir. Mümkün olduğunca tekrar kullanın.
  • İşinizi arka planda yapın: Yine RenderScript çok hızlıdır, ancak hiçbir şekilde anında değildir. Herhangi bir Çekirdek, özellikle karmaşık olanlar, bir “AsyncTask” veya benzeri bir şekilde UI iş parçacığından yürütülmelidir. Ancak çoğunlukla bellek sızıntıları konusunda endişelenmenize gerek yok. RenderScript ile ilgili tüm sınıflar yalnızca “Context” uygulamasını kullanır ve bu nedenle bellek sızıntılarına neden olmaz. Ancak yine de “Görünüm”, “Etkinlik” veya kendiniz kullandığınız herhangi bir “Bağlam” örneğini sızdırmak gibi olağan şeyler için endişelenmeniz gerekiyor!
  • Yerleşik öğeleri kullanın: Görüntü bulanıklaştırma, karıştırma, dönüştürme, yeniden boyutlandırma gibi görevleri gerçekleştiren önceden tanımlanmış birçok komut dosyası vardır. Ve çekirdeklerinizi uygulamanıza yardımcı olacak daha birçok yerleşik yöntem vardır. Şansınız, bir şey yapmak istiyorsanız, yapmaya çalıştığınız şeyi zaten yapan bir komut dosyası veya yöntem vardır. Tekerleği yeniden icat etmeyin.

Hızlı bir şekilde başlamak ve gerçek kodla oynamak istiyorsanız, bu eğitimde bahsedilen örneği tam olarak uygulayan örnek GitHub projesine bir göz atmanızı tavsiye ederim. Projeyi burada bulabilirsiniz. RenderScript ile iyi eğlenceler!

BlurBitmapTask.java #

public class BlurBitmapTask extends AsyncTask<Bitmap, Void, Bitmap> {
    private final WeakReference<ImageView> imageViewReference;
    private final RenderScript renderScript;

    private boolean shouldRecycleSource = false;

    public BlurBitmapTask(@NonNull Context context, @NonNull ImageView imageView) {
        // Use a WeakReference to ensure
        // the ImageView can be garbage collected
        imageViewReference = new WeakReference<>(imageView);
        renderScript = RenderScript.create(context);
    }

    // Decode image in background.
    @Override
    protected Bitmap doInBackground(Bitmap... params) {
        Bitmap bitmap = params[0];
        return blurBitmap(bitmap);
    }

    // Once complete, see if ImageView is still around and set bitmap.
    @Override
    protected void onPostExecute(Bitmap bitmap) {
        if (bitmap == null || isCancelled()) {
            return;
        }

        final ImageView imageView = imageViewReference.get();
        if (imageView == null) {
            return;
        }

        imageView.setImageBitmap(bitmap);
    }

    public Bitmap blurBitmap(Bitmap bitmap) {
        // https://plus.google.com/+MarioViviani/posts/fhuzYkji9zz

        //Let's create an empty bitmap with the same size of the bitmap we want to blur
        Bitmap outBitmap = Bitmap.createBitmap(bitmap.getWidth(), bitmap.getHeight(),
                Bitmap.Config.ARGB_8888);

        //Instantiate a new Renderscript


        //Create an Intrinsic Blur Script using the Renderscript
        ScriptIntrinsicBlur blurScript = ScriptIntrinsicBlur.create(renderScript, Element.U8_4(renderScript));

        //Create the in/out Allocations with the Renderscript and the in/out bitmaps
        Allocation allIn = Allocation.createFromBitmap(renderScript, bitmap);
        Allocation allOut = Allocation.createFromBitmap(renderScript, outBitmap);

        //Set the radius of the blur
        blurScript.setRadius(25.f);

        //Perform the Renderscript
        blurScript.setInput(allIn);
        blurScript.forEach(allOut);

        //Copy the final bitmap created by the out Allocation to the outBitmap
        allOut.copyTo(outBitmap);

        // recycle the original bitmap
        // nope, we are using the original bitmap as well :/
        if (shouldRecycleSource) {
            bitmap.recycle();
        }

        //After finishing everything, we destroy the Renderscript.
        renderScript.destroy();

        return outBitmap;
    }

    public boolean isShouldRecycleSource() {
        return shouldRecycleSource;
    }

    public void setShouldRecycleSource(boolean shouldRecycleSource) {
        this.shouldRecycleSource = shouldRecycleSource;
    }
}
