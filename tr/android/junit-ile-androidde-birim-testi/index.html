Yerel birim testleri oluşturma #

Test sınıflarınızı buraya yerleştirin: /src/test/<pkg_name>/

Örnek test sınıfı #

public class ExampleUnitTest {
    @Test
    public void addition_isCorrect() throws Exception {
        int a=4, b=5, c;
        c = a + b;
        assertEquals(9, c); // This test passes
        assertEquals(10, c); //Test fails
    }
}

Bozulmak #

public class ExampleUnitTest {
  ...
}

Test sınıfı, birkaç test sınıfı oluşturabilir ve bunları test paketinin içine yerleştirebilirsiniz.

@Test
public void addition_isCorrect() {
   ...
}

Test yöntemi, bir test sınıfı içinde birkaç test yöntemi oluşturulabilir.

@Test notuna dikkat edin.

Test notu, JUnit’e public void yönteminin hangi ekli bir test durumu olarak çalıştırılabilir.

@Önce, @Sonra vb. gibi birkaç faydalı ek açıklama daha vardır. Bu sayfa başlamak için iyi bir yer olacaktır.

assertEquals(9, c); // This test passes
assertEquals(10, c); //Test fails

Bu yöntemler Assert sınıfının üyesidir. Diğer bazı yararlı yöntemler ‘assertFalse()’, ‘assertNotNull()’, ‘assertTrue’ vb.‘dir. İşte ayrıntılı bir Açıklama.

JUnit Testi için Açıklama Bilgileri:

@Test: Test notu, JUnit’e eklendiği public void yönteminin bir test durumu olarak çalıştırılabileceğini söyler. Yöntemi çalıştırmak için JUnit önce sınıfın yeni bir örneğini oluşturur, ardından açıklamalı yöntemi çağırır.

@Önce: Testler yazarken, birkaç testin çalıştırılmadan önce benzer nesnelere ihtiyaç duyması yaygın bir durumdur. Bir public void yöntemine “@Before” ile açıklama eklemek, bu yöntemin Test yönteminden önce çalıştırılmasına neden olur.

@Sonra: Harici kaynakları Before yöntemine tahsis ederseniz, bunları test çalıştırmalarından sonra serbest bırakmanız gerekir. Bir public void yöntemine “@After” ile açıklama eklemek, bu yöntemin Test yönteminden sonra çalıştırılmasına neden olur. Tüm “@After” yöntemlerinin, Before veya Test yöntemi bir istisna oluştursa bile çalışması garanti edilir

İş Mantığını Android Bileşenlerinden Çıkarma #

Yerel JVM birim testlerinden elde edilen değerin çoğu, uygulamanızı tasarlama şeklinizden gelir. İş mantığınızı Android Bileşenlerinizden ayırabileceğiniz şekilde tasarlamanız gerekir. İşte Model-View-Presenter pattern kullanılarak böyle bir yol örneği. Sadece bir kullanıcı adı ve şifre alan temel bir kayıt ekranı uygulayarak bunu pratik yapalım. Android uygulamamız, kullanıcının sağladığı kullanıcı adının boş olmadığını, şifrenin en az sekiz karakter uzunluğunda ve en az bir rakam içerdiğini doğrulamaktan sorumludur. Kullanıcı adı/şifre geçerliyse kayıt api çağrımızı gerçekleştiririz, aksi takdirde bir hata mesajı görüntüleriz.

İş mantığının Android Bileşeni ile yüksek oranda eşleştiği örnek.

public class LoginActivity extends Activity{
    ...
    private void onSubmitButtonClicked(){
        String username = findViewById(R.id.username).getText().toString();
        String password = findViewById(R.id.password).getText().toString();
        boolean isUsernameValid = username != null && username.trim().length() != 0;
        boolean isPasswordValid = password != null && password.trim().length() >= 8 && password.matches(".*\\d+.*");
        if(isUsernameValid && isPasswordValid){
            performSignUpApiCall(username, password);
        } else {
            displayInvalidCredentialsErrorMessage();
        }
    }
}

İş mantığının Android Bileşeninden ayrıldığı örnek.

Burada, çeşitli sınıflarımız arasındaki çeşitli etkileşimleri barındıracak olan LoginContract adlı tek bir sınıfta tanımlıyoruz.

public interface LoginContract {
    public interface View {
        performSignUpApiCall(String username, String password);
        displayInvalidCredentialsErrorMessage();
    }
    public interface Presenter {
        void validateUserCredentials(String username, String password);
    }
}

LoginActivity’miz, bir kullanıcının kayıt formunu (iş mantığımız) nasıl doğrulayacağımızı bilme sorumluluğunu ortadan kaldırmamız dışında çoğunlukla aynıdır. LoginActivity artık doğrulama yapmak için yeni LoginPresenter’ımıza güvenecek.

public class LoginActivity extends Activity implements LoginContract.View{
    private LoginContract.Presenter presenter;

    protected void onCreate(Bundle savedInstanceState) {
            super.onCreate(savedInstanceState);
            presenter = new LoginPresenter(this);
            ....
        }
        ...

        private void onSubmitButtonClicked(){
            String username = findViewById(R.id.username).getText().toString();
            String password = findViewById(R.id.password).getText().toString();
            presenter.validateUserCredentials(username, password);
    }
    ...
}

Artık iş mantığınız yeni LoginPresenter sınıfınızda yer alacaktır.

public class LoginPresenter implements LoginContract.Presenter{
    private LoginContract.View view;

    public LoginPresenter(LoginContract.View view){
        this.view = view;
    }

    public void validateUserCredentials(String username, String password){
        boolean isUsernameValid = username != null && username.trim().length() != 0;
        boolean isPasswordValid = password != null && password.trim().length() >= 8 && password.matches(".*\\d+.*");
        if(isUsernameValid && isPasswordValid){
            view.performSignUpApiCall(username, password);
        } else {
            view.displayInvalidCredentialsErrorMessage();
        }
    }
}

Ve şimdi yeni LoginPresenter sınıfınıza karşı yerel JVM birim testleri oluşturabiliriz.

public class LoginPresenterTest {

    @Mock
    LoginContract.View view;

    private LoginPresenter presenter;

    @Before
    public void setUp() throws Exception {
        MockitoAnnotations.initMocks(this);
        presenter = new LoginPresenter(view);
    }

    @Test
    public void test_validateUserCredentials_userDidNotEnterUsername_displayErrorMessage() throws Exception {
        String username = "";
        String password = "kingslayer1";
        presenter.validateUserCredentials(username, password);
        Mockito.verify(view). displayInvalidCredentialsErrorMessage();
    }

    @Test
    public void test_validateUserCredentials_userEnteredFourLettersAndOneDigitPassword_displayErrorMessage() throws Exception {
        String username = "Jaime Lanninster";
        String password = "king1";
        presenter.validateUserCredentials(username, password);
        Mockito.verify(view). displayInvalidCredentialsErrorMessage();
    }

    @Test
    public void test_validateUserCredentials_userEnteredNineLettersButNoDigitsPassword_displayErrorMessage() throws Exception {
        String username = "Jaime Lanninster";
        String password = "kingslayer";
        presenter.validateUserCredentials(username, password);
        Mockito.verify(view). displayInvalidCredentialsErrorMessage();
    }

    @Test
    public void test_validateUserCredentials_userEnteredNineLettersButOneDigitPassword_performApiCallToSignUpUser() throws Exception {
        String username = "Jaime Lanninster";
        String password = "kingslayer1";
        presenter.validateUserCredentials(username, password);
        Mockito.verify(view).performSignUpApiCall(username, password);
    }
}

Gördüğünüz gibi LoginActivity’den iş mantığımızı çıkarıp LoginPresenter’a POJO yerleştirdiğimizde. Artık iş mantığımıza göre yerel JVM birim testleri oluşturabiliriz.

Her sınıfın tek bir sorumluluğa sahip olmasına, ek sınıflara vb. bağlı kalmaya yakın olduğumuz gibi, mimarideki değişikliğimizden başka çeşitli çıkarımlar olduğu da belirtilmelidir. Bunlar, bunu yapmak için seçtiğim yolun yan etkileridir. MVP stili ile ayrıştırma. MVP, bunu yapmanın sadece bir yoludur, ancak MVVM gibi bakmak isteyebileceğiniz başka alternatifler de vardır. Size en uygun sistemi seçmeniz yeterli.

JUnit’i kullanmaya başlama #

Kurmak #

Android projenizi JUnit kullanarak birim testine başlamak için projenize JUnit bağımlılığını eklemeniz ve birim testleri için kaynak kodunu içerecek bir test kaynak seti oluşturmanız gerekir. Android Studio ile oluşturulan projeler genellikle JUnit bağımlılığını ve test kaynak kümesini içerir.

Closure bağımlılıkları içindeki build.gradle modülünüzün dosyasına aşağıdaki satırı ekleyin:

testCompile 'junit:junit:4.12'

JUnit test sınıfları, “test” adlı özel bir kaynak kümesinde bulunur. Bu kaynak kümesi yoksa, kendiniz yeni bir klasör oluşturmanız gerekir. Varsayılan bir Android Studio (Gradle tabanlı) projesinin klasör yapısı şöyle görünür:

<project-root-folder>
    /app (module root folder)
        /build
        /libs
        /src
            /main (source code)
            /test (unit test source code)
            /androidTest (instrumentation test source code)
        build.gradle (module gradle file)
    /build
    /gradle
    build.gradle (project gradle file)
    gradle.properties
    gradlew
    gradlew.bat
    local.properties
    settings.gradle (gradle settings)

Projenizde /app/src/test klasörü yoksa onu kendiniz oluşturmanız gerekir. test klasörü içinde ayrıca bir java klasörüne ihtiyacınız var (eğer yoksa oluşturun). “test” kaynak kümesindeki Java klasörü, “ana” kaynak kümenizle aynı paket yapısını içermelidir.

Not: ‘androidTest’ kaynak kümesine sahip olmanız gerekmez, bu kaynak kümesi genellikle Android Studio tarafından oluşturulan projelerde bulunur ve buraya referans olarak dahil edilmiştir.

Test yazma #

  1. **test' kaynak kümesi içinde yeni bir sınıf oluşturun.** Proje görünümünde test kaynağı kümesine sağ tıklayın 'Yeni' > 'Java sınıfı'nı seçin. En çok kullanılan adlandırma kalıbı, test edeceğiniz sınıfın adını Test’ eklenmiş olarak kullanmaktır. Böylece ‘StringUtilities’, ‘StringUtilitiesTest’ olur.

  2. @RunWith ek açıklamasını ekleyin JUnit’in test sınıfımızda tanımlayacağımız testleri çalıştırabilmesi için @RunWith notuna ihtiyaç vardır. Varsayılan JUnit koşucusu (JUnit 4 için) ‘BlockJUnit4ClassRunner’dır, ancak bu koşuyu doğrudan kullanmak yerine, varsayılan JUnit koşucusu için bir kısayol olan ‘JUnit4’ takma adını kullanmak daha uygundur.

     @RunWith(JUnit4.class)
 public class StringUtilitiesTest {

 }

  3. Bir test oluşturun Bir birim testi, esasen, çoğu durumda çalıştırıldığında başarısız olmaması gereken bir yöntemdir. Başka bir deyişle, bir istisna atmamalıdır. Bir test yönteminin içinde, hemen hemen her zaman belirli koşulların karşılanıp karşılanmadığını kontrol eden iddialar bulacaksınız. Bir iddia başarısız olursa, yöntemin/testin başarısız olmasına neden olan bir istisna atar. Bir test yöntemine her zaman “@Test” notu eklenir. Bu açıklama olmadan JUnit testi otomatik olarak çalıştırmaz.

     @RunWith(JUnit4.class)
 public class StringUtilitiesTest {

     @Test
     public void addition_isCorrect() throws Exception {
         assertEquals("Hello JUnit", "Hello" + " " + "JUnit");
     }
 }

    Note: unlike the standard Java method naming convention unit test method names do often contain underscores.

İstisnalar #

JUnit, bir yöntemin belirli bir girdi için belirli bir istisna atıp atmadığını test etmek için de kullanılabilir.

Bu örnekte, Boole biçimi (giriş) tanınmadıysa/bilinmiyorsa, aşağıdaki yöntemin gerçekten bir istisna atıp atmadığını test edeceğiz:

public static boolean parseBoolean(@NonNull String raw) throws IllegalArgumentException{
    raw = raw.toLowerCase().trim();
    switch (raw) {
        case "t": case "yes": case "1": case "true":
            return true;
        case "f": case "no": case "0": case "false":
            return false;
        default:
            throw new IllegalArgumentException("Unknown boolean format: " + raw);
    }
}

‘@Test’ ek açıklamasına ‘beklenen’ parametresini ekleyerek, hangi istisnanın atılmasının beklendiği tanımlanabilir. Bu istisna oluşmazsa birim testi başarısız olur ve istisna gerçekten atılırsa başarılı olur:

@Test(expected = IllegalArgumentException.class)
public void parseBoolean_parsesInvalidFormat_throwsException(){
    StringUtilities.parseBoolean("Hello JUnit");
}

Bu iyi çalışır, ancak sizi yöntem içinde yalnızca tek bir test durumuyla sınırlar. Bazen birden fazla durumu tek bir yöntemle test etmek isteyebilirsiniz. Bu sınırlamanın üstesinden gelmek için sıklıkla kullanılan bir teknik, “try-catch” bloklarını ve “Assert.fail()” yöntemini kullanmaktır:

@Test
public void parseBoolean_parsesInvalidFormats_throwsException(){
    try {
        StringUtilities.parseBoolean("Hello!");
        fail("Expected IllegalArgumentException");
    } catch(IllegalArgumentException e){
    }

    try {
        StringUtilities.parseBoolean("JUnit!");
        fail("Expected IllegalArgumentException");
    } catch(IllegalArgumentException e){
    }
}

Not: Bazı insanlar, bir birim testi içinde tek bir vakadan fazlasını test etmenin kötü bir uygulama olduğunu düşünür.

Statik içe aktarma #

JUnit, her ilkel tür için en az bir tane ve Objects için bir tane olmak üzere oldukça fazla ‘assertEquals’ yöntemi tanımlar. Bu yöntemler varsayılan olarak doğrudan çağrılamaya uygun değildir ve şu şekilde çağrılmalıdır: Assert.assertEquals. Ancak bu yöntemler çok sık kullanıldığından, insanlar neredeyse her zaman statik bir içe aktarma kullanır, böylece yöntem doğrudan sınıfın bir parçasıymış gibi kullanılabilir.

“assetEquals” yöntemine statik bir içe aktarma eklemek için aşağıdaki içe aktarma ifadesini kullanın:

import static org.junit.Assert.assertEquals;

Ayrıca, aşağıdaki statik içe aktarmayı kullanarak “assertArrayEquals”, “assertNotNull” ve “assertFalse” vb. dahil tüm assert yöntemlerini statik olarak içe aktarabilirsiniz:

import static org.junit.Assert.*;

Statik içe aktarma olmadan:

@Test
public void addition_isCorrect(){
    Assert.assertEquals(4 , 2 + 2);
}

Statik içe aktarma ile:

@Test
public void addition_isCorrect(){
    assertEquals(4 , 2 + 2);
}
