AsyncTask ile Aktiviteleri sızdırmaktan kaçının #

Bir uyarı: AsyncTask, burada açıklanan bellek sızıntısı dışında birçok soruna sahiptir. Bu nedenle, bu API’ye dikkat edin veya sonuçları tam olarak anlamadıysanız, bundan tamamen kaçının. Birçok alternatif var (Thread, EventBus, RxAndroid, vb.).

“AsyncTask” ile ilgili yaygın bir hata, ana bilgisayar “Activity” (veya “Fragment”) için güçlü bir referans yakalamaktır:

class MyActivity extends Activity {
  private AsyncTask<Void, Void, Void> myTask = new AsyncTask<Void, Void, Void>() {
    // Don't do this! Inner classes implicitly keep a pointer to their
    // parent, which in this case is the Activity!
  }
}

Bu bir sorundur çünkü “AsyncTask”, örneğin görev çalışırken bir yapılandırma değişikliği olursa, üst “Etkinlik"ten kolayca daha uzun yaşayabilir.

Bunu yapmanın doğru yolu, görevinizi üst öğeyi yakalamayan bir “statik” sınıf yapmak ve ana bilgisayar “Activity"ye bir zayıf referans tutmaktır:

class MyActivity extends Activity {
  static class MyTask extends AsyncTask<Void, Void, Void> {
    // Weak references will still allow the Activity to be garbage-collected
    private final WeakReference<MyActivity> weakActivity;

    MyTask(MyActivity myActivity) {
      this.weakActivity = new WeakReference<>(myActivity);
    }

    @Override
    public Void doInBackground(Void... params) {
      // do async stuff here
    }

    @Override
    public void onPostExecute(Void result) {
      // Re-acquire a strong reference to the activity, and verify
      // that it still exists and is active.
      MyActivity activity = weakActivity.get();
      if (activity == null
          || activity.isFinishing()
          || activity.isDestroyed()) {
        // activity is no longer valid, don't do anything!
        return;
      }

      // The activity is still valid, do main-thread stuff here
    }
  }
}

Yaygın bellek sızıntıları ve bunların nasıl düzeltileceği #

1. Bağlamlarınızı düzeltin: #

Uygun bağlamı kullanmayı deneyin: Örneğin, bir Tost yalnızca bir etkinlikte değil, birçok etkinlikte görülebildiğinden, tostlar için “getApplicationContext()” öğesini kullanın ve bir etkinlik sona ermiş olsa bile hizmetler çalışmaya devam edebildiğinden, bir hizmeti şununla başlatın:

Intent myService = new Intent(getApplicationContext(), MyService.class);

Hangi bağlamın uygun olduğuna dair hızlı bir kılavuz olarak bu tabloyu kullanın:

Orijinal buradaki bağlamla ilgili makale.

2. Bağlama statik referans #

Ciddi bir bellek sızıntısı hatası, ‘Görünüm’ için statik bir referans tutmaktır. Her “Görünüm"ün “Bağlam"a bir iç referansı vardır. Bu, tüm görünüm hiyerarşisine sahip eski bir Aktivitenin, uygulama sonlandırılıncaya kadar çöp toplanmayacağı anlamına gelir. Ekranı döndürürken uygulamanız iki kez hafızanızda olacak.

Görünüm, Bağlam veya bunların soyundan gelenlerden herhangi birine kesinlikle statik referans olmadığından emin olun.

3. Hizmetlerinizi gerçekten bitirip bitirmediğinizi kontrol edin. #

Örneğin, Google konum hizmeti API’sini kullanan bir amaç Hizmetim var. Ve googleApiClient.disconnect(); aramayı unuttum:

//Disconnect from API onDestroy()
if (googleApiClient.isConnected()) {
    LocationServices.FusedLocationApi.removeLocationUpdates(googleApiClient, GoogleLocationService.this);
    googleApiClient.disconnect();
}

4. Görüntü ve bitmap kullanımını kontrol edin: #

Square’in kütüphanesi Picasso kullanıyorsanız, .fit() kullanmayarak bellek sızdırdığımı fark ettim, bu da bellek ayak izimi ortalama 50MB’den 19MB’nin altına düşürdü:

Picasso.with(ActivityExample.this)                   //Activity context
                .load(object.getImageUrl())           
                .fit()                                //This avoided the OutOfMemoryError
                .centerCrop()                         //makes image to not stretch
                .into(imageView);

5. Yayın alıcıları kullanıyorsanız, bunların kaydını kaldırın. #

6. Eğer java.util.Observer (Gözlemci kalıbı) kullanıyorsanız: #

deleteObserver(gözlemci); kullandığınızdan emin olun.

LeakCanary kitaplığı ile bellek sızıntılarını tespit edin #

LeakCanary, hata ayıklama yapılarınızdaki bellek sızıntılarını tespit etmek için bir Açık Kaynak Java kitaplığıdır.

Sadece “build.gradle” içindeki bağımlılıkları ekleyin:

 dependencies {
   debugCompile 'com.squareup.leakcanary:leakcanary-android:1.5.1'
   releaseCompile 'com.squareup.leakcanary:leakcanary-android-no-op:1.5.1'
   testCompile 'com.squareup.leakcanary:leakcanary-android-no-op:1.5.1'
 }

Ardından, “Uygulama” sınıfınızda:

public class ExampleApplication extends Application {

  @Override public void onCreate() {
    super.onCreate();
    
    if (LeakCanary.isInAnalyzerProcess(this)) {
      // This process is dedicated to LeakCanary for heap analysis.
      // You should not init your app in this process.
      return;
    }

    LeakCanary.install(this);
  }
}

Artık LeakCanary, hata ayıklama yapınızda bir etkinlik belleği sızıntısı algılandığında otomatik olarak bir bildirim gösterecektir.

NOT: Yayın kodu, ’leakcanary-android-no-op’ bağımlılığında bulunan iki boş sınıf dışında LeakCanary’ye referans içermeyecektir.

Aktivitelerde anonim geri arama #

Anonim bir sınıf oluşturduğunuzda, üst sınıfına örtük bir referansı korur. Yani yazarken:

public class LeakyActivity extends Activity
{

...

    foo.registerCallback(new BarCallback() 
    {            
        @Override
        public void onBar() 
        {
            // do something                
        }            
    });
}

Aslında, foo’ya LeakyActivity örneğinize bir referans gönderiyorsunuz. Kullanıcı LeakyActivity’nizden uzaklaştığında bu referans, LeakyActivity örneğinin çöp toplanmasını engelleyebilir. Bu ciddi bir sızıntıdır, çünkü etkinlikler tüm görünüm hiyerarşisine referansta bulunur ve bu nedenle bellekte oldukça büyük nesnelerdir.

Bu sızıntı nasıl önlenir:

Elbette etkinliklerde anonim geri aramaları kullanmaktan tamamen kaçınabilirsiniz. Ayrıca aktivite yaşam döngüsüne göre tüm geri aramalarınızın kaydını silebilirsiniz. şöyle:

public class NonLeakyActivity extends Activity
{
    private final BarCallback mBarCallback = new BarCallback() 
    {            
        @Override
        public void onBar() 
        {
            // do something                
        }            
    });

    @Override
    protected void onResume()
    {
        super.onResume();
        foo.registerCallback(mBarCallback);
    }

    @Override
    protected void onPause()
    {
        super.onPause();
        foo.unregisterCallback(mBarCallback);
    }
}

Statik sınıflarda Etkinlik Bağlamı #

Dinleyicilerle Etkinlikleri sızdırmaktan kaçının #

Bir Activity içinde bir dinleyici uygular veya oluşturursanız, her zaman dinleyicinin kayıtlı olduğu nesnenin yaşam döngüsüne dikkat edin.

Bir kullanıcının ne zaman oturum açtığı veya kapattığı ile ilgilenen birkaç farklı etkinliğin/parçanın olduğu bir uygulama düşünün. Bunu yapmanın bir yolu, kullanıcının durumu değiştiğinde bildirim almak için abone olunabilecek tek bir “UserController” örneğine sahip olmaktır:

public class UserController {
    private static UserController instance;
    private List<StateListener> listeners;

    public static synchronized UserController getInstance() {
        if (instance == null) {
            instance = new UserController();
        }
        return instance;
    }

    private UserController() {
        // Init
    }

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(listener);
    }

    public void logout() {
        for (StateListener listener : listeners) {
            listener.userLoggedOut();
        }
    }

    public void login() {
        for (StateListener listener : listeners) {
            listener.userLoggedIn();
        }
    }

    public interface StateListener {
        void userLoggedIn();
        void userLoggedOut();
    }
}

Sonra iki etkinlik vardır, “SignInActivity”:

public class SignInActivity extends Activity implements UserController.StateListener{

    UserController userController;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        this.userController = UserController.getInstance();
        this.userController.registerUserStateChangeListener(this);
    }

    @Override
    public void userLoggedIn() {
        startMainActivity();
    }

    @Override
    public void userLoggedOut() {
        showLoginForm();
    }

    ...

    public void onLoginClicked(View v) {
        userController.login();
    }
}

Ve “MainActivity”:

public class MainActivity extends Activity implements UserController.StateListener{
    UserController userController;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        this.userController = UserController.getInstance();
        this.userController.registerUserStateChangeListener(this);
    }

    @Override
    public void userLoggedIn() {
        showUserAccount();
    }

    @Override
    public void userLoggedOut() {
        finish();
    }

    ...

    public void onLogoutClicked(View v) {
        userController.logout();
    }
}

Bu örnekte olan şey, kullanıcının her oturum açıp ardından tekrar çıkış yaptığında, bir “MainActivity” örneğinin sızdırılmasıdır. Sızıntı, “UserController#listeners” içindeki etkinliğe bir başvuru olduğu için oluşur.

Lütfen dikkat: Dinleyici olarak anonim bir iç sınıf kullansak bile, aktivite yine de sızdırılır:

...
this.userController.registerUserStateChangeListener(new UserController.StateListener() {
    @Override
    public void userLoggedIn() {
        showUserAccount();
    }

    @Override
    public void userLoggedOut() {
        finish();
    }
});    
...

Aktivite yine de sızacaktır, çünkü anonim iç sınıf, dış sınıfa (bu durumda aktivite) örtük bir referansa sahiptir. Bu nedenle, dış sınıftaki örnek yöntemleri iç sınıftan çağırmak mümkündür. Aslında, dış sınıfa referansı olmayan tek iç sınıf türü statik iç sınıflardır.

Kısacası, statik olmayan iç sınıfların tüm örnekleri, onları yaratan dış sınıfın örneğine örtük bir referans taşır.

Bunu çözmenin iki ana yaklaşımı vardır; ya bir dinleyiciyi “UserController#listeners"dan kaldırmak için bir yöntem ekleyerek ya da dinleyicilerin referansını tutmak için bir “WeakReference” kullanarak.

Alternatif 1: Dinleyicileri kaldırma #

Yeni bir removeUserStateChangeListener(StateListener listener) metodu oluşturarak başlayalım:

public class UserController {

    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(listener);
    }

    public void removeUserStateChangeListener(StateListener listener) {
        listeners.remove(listener);
    }

    ...
}

O zaman bu metodu aktivitenin onDestroy metodunda çağıralım:

public class MainActivity extends Activity implements UserController.StateListener{
    ...

    @Override
    protected void onDestroy() {
        super.onDestroy();
        userController.removeUserStateChangeListener(this);
    }
}

Bu değişiklikle, kullanıcı oturum açıp kapattığında “MainActivity” örnekleri artık sızdırılmıyor. Bununla birlikte, belgeler net değilse, “UserController"ı kullanmaya başlayan bir sonraki geliştirici, etkinlik yok edildiğinde dinleyicinin kaydını silmenin gerekliliğini gözden kaçırabilir, bu da bizi bu türlerden kaçınmanın ikinci yöntemine götürür. sızıntılar.

Alternatif 2: Zayıf referansları kullanma #

Öncelikle zayıf referansın ne olduğunu açıklayarak başlayalım. Adından da anlaşılacağı gibi zayıf bir referans, bir nesneye zayıf bir referans içerir. Güçlü bir referans olan normal bir örnek alanıyla karşılaştırıldığında, zayıf bir referans, çöp toplayıcı GC’nin nesneleri kaldırmasını engellemez. Yukarıdaki örnekte bu, “UserController”, dinleyicilere referans vermek için “WeakReference” kullanmışsa, yok edildikten sonra “MainActivity"nin çöp olarak toplanmasına izin verir.

Kısacası, zayıf bir referans, GC’ye, bu nesneye başka kimsenin güçlü bir referansı yoksa, devam edin ve onu kaldırın.

Dinleyicilerini takip etmek için bir ‘WeakReference’ listesini kullanmak için ‘UserController’ı değiştirelim:

public class UserController {

    ...
    private List<WeakReference<StateListener>> listeners;
    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(new WeakReference<>(listener));
    }

    public void removeUserStateChangeListener(StateListener listenerToRemove) {
        WeakReference referencesToRemove = null;
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null && listener == listenerToRemove) {
                referencesToRemove = listenerRef;
                break;
            }
        }
        listeners.remove(referencesToRemove);
    }

    public void logout() {
        List referencesToRemove = new LinkedList();
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null) {
                listener.userLoggedOut();
            } else {
                referencesToRemove.add(listenerRef);
            }
        }
    }

    public void login() {
        List referencesToRemove = new LinkedList();
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null) {
                listener.userLoggedIn();
            } else {
                referencesToRemove.add(listenerRef);
            }
        }
    }
    ...    
}

Bu değişiklikle, UserController herhangi bir dinleyiciye güçlü referanslar içermediğinden, dinleyicilerin kaldırılıp kaldırılmamasının bir önemi yoktur. Ancak, bu ortak kod kodunu her seferinde yazmak zahmetlidir. Bu nedenle, WeakCollection adında bir genel sınıf oluşturalım:

public class WeakCollection<T> {
    private LinkedList<WeakReference<T>> list;

    public WeakCollection() {
        this.list = new LinkedList<>();
    }
    public void put(T item){
        //Make sure that we don't re add an item if we already have the reference.
        List<T> currentList = get();
        for(T oldItem : currentList){
            if(item == oldItem){
                return;
            }
        }
        list.add(new WeakReference<T>(item));
    }

    public List<T> get() {
        List<T> ret = new ArrayList<>(list.size());
        List<WeakReference<T>> itemsToRemove = new LinkedList<>();
        for (WeakReference<T> ref : list) {
            T item = ref.get();
            if (item == null) {
                itemsToRemove.add(ref);
            } else {
                ret.add(item);
            }
        }
        for (WeakReference ref : itemsToRemove) {
            this.list.remove(ref);
        }
        return ret;
    }

    public void remove(T listener) {
        WeakReference<T> refToRemove = null;
        for (WeakReference<T> ref : list) {
            T item = ref.get();
            if (item == listener) {
                refToRemove = ref;
            }
        }
        if(refToRemove != null){
            list.remove(refToRemove);
        }
    }
}

Şimdi bunun yerine ‘WeakCollection’ kullanmak için ‘UserController’ı yeniden yazalım:

public class UserController {
    ...
    private WeakCollection<StateListener> listenerRefs;
    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listenerRefs.put(listener);
    }

    public void removeUserStateChangeListener(StateListener listenerToRemove) {
        listenerRefs.remove(listenerToRemove);
    }

    public void logout() {
        for (StateListener listener : listenerRefs.get()) {
            listener.userLoggedOut();
        }
    }

    public void login() {
        for (StateListener listener : listenerRefs.get()) {
            listener.userLoggedIn();
        }
    }

    ...
}

Yukarıdaki kod örneğinde gösterildiği gibi, WeakCollection<T>, normal bir liste yerine WeakReference kullanmak için gereken tüm ortak kod kodunu kaldırır. Hepsini kapatmak için: UserController#removeUserStateChangeListener(StateListener) çağrısı kaçırılırsa, dinleyici ve referans verdiği tüm nesneler sızıntı yapmaz.

Anonim Sınıf, İşleyici, Zamanlayıcı Görevi, İş Parçacığı ile bellek sızıntılarını önleyin #


Android’de her geliştirici bir projede en az bir kez ‘Anonim Sınıf’ (Çalıştırılabilir) kullanır. Herhangi bir “Anonim Sınıf”, ebeveynine (aktivitesine) bir referansa sahiptir. Uzun süren bir görev gerçekleştirirsek, görev sona erene kadar ana etkinlik yok edilmeyecektir.
Örnek, işleyici ve Anonymous ‘Runnable’ sınıfını kullanır. Runnable bitmeden aktiviteden çıktığımızda hafıza sızdırılacaktır.

  new Handler().postDelayed(new Runnable() {
        @Override
        public void run() {
            // do abc long 5s or so
        }
    }, 10000); // run "do abc" after 10s. It same as timer, thread...

Nasıl çözeriz?

  1. ‘Anonim Sınıf’ ile uzun süre işlem yapmayın yoksa bunun için bir ‘Statik sınıf’a ihtiyacımız var ve bunun içine ‘WeakReference’ geçirin (aktivite, görünüm gibi). ‘İş parçacığı’, ‘Anonim Sınıf’ ile aynıdır.
  2. Faaliyet yok edildiğinde ‘İşleyici’, ‘Zamanlayıcı’yı iptal edin.
