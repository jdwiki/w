JavaScript, herhangi bir dil gibi, belirli dil özelliklerinin kullanımında sağduyulu olmamızı gerektirir. Bazı özelliklerin aşırı kullanımı performansı düşürebilirken, bazı teknikler performansı artırmak için kullanılabilir.

Erken optimizasyonun tüm kötülüklerin kökü olduğunu unutmayın. Önce açık ve doğru kodu yazın, ardından performans sorunlarınız varsa, iyileştirilecek belirli alanları aramak için bir profil oluşturucu kullanın. Genel performansı anlamlı bir şekilde etkilemeyen kodu optimize ederek zaman kaybetmeyin.

Ölçün, ölçün, ölçün. Performans çoğu zaman mantığa aykırı olabilir ve zamanla değişir. Şimdi daha hızlı olan gelecekte olmayabilir ve kullanım durumunuza bağlı olabilir. Yaptığınız tüm optimizasyonların gerçekten iyileştiğinden, performansa zarar vermediğinden ve değişikliğin faydalı olduğundan emin olun.

Performans açısından kritik işlevlerde denemek/yakalamaktan kaçının #

Bazı JavaScript motorları (örneğin, Node.js’nin mevcut sürümü ve Chrome’un Ignition+turbofan’dan önceki eski sürümleri), bir try/catch bloğu içeren işlevlerde optimize ediciyi çalıştırmaz.

Performans açısından kritik koddaki istisnaları işlemeniz gerekiyorsa, bazı durumlarda try/catch’i ayrı bir işlevde tutmak daha hızlı olabilir. Örneğin, bu işlev bazı uygulamalar tarafından optimize edilmeyecektir:

function myPerformanceCriticalFunction() {
    try {
        // do complex calculations here
    } catch (e) {
        console.log(e);
    }
}

Bununla birlikte, yavaş kodu ayrı bir işleve (bu optimize edilebilir) taşımak için yeniden düzenleme yapabilir ve onu try bloğunun içinden çağırabilirsiniz.

// This function can be optimized
function doCalculations() {
    // do complex calculations here
}

// Still not always optimized, but it's not doing much so the performance doesn't matter
function myPerformanceCriticalFunction() {
    try {
        doCalculations();
    } catch (e) {
        console.log(e);
    }
}

İşte farkı gösteren bir jsPerf karşılaştırması: https://jsperf.com/try-catch-deoptimization. Çoğu tarayıcının mevcut sürümünde, varsa çok fazla fark olmamalıdır, ancak Chrome ve Firefox’un veya IE’nin daha az yeni sürümlerinde, dene/yakala içinde yardımcı işlevi çağıran sürüm muhtemelen daha hızlı olacaktır.

Bunun gibi optimizasyonların dikkatli bir şekilde ve kodunuzun profilini çıkarmaya dayalı gerçek kanıtlarla yapılması gerektiğini unutmayın. JavaScript motorları daha iyi hale geldikçe, yardımcı olmak yerine performansa zarar verebilir veya hiçbir fark yaratmayabilir (ancak kodu sebepsiz yere karmaşık hale getirebilir). Faydası, zararı veya hiçbir fark yaratmaması birçok faktöre bağlı olabilir, bu nedenle her zaman kodunuz üzerindeki etkileri ölçün. Bu, tüm optimizasyonlar için geçerlidir, ancak özellikle derleyici/çalışma zamanının düşük seviyeli ayrıntılarına bağlı olan bunun gibi mikro optimizasyonlar.

DOM Güncellemelerini Sınırla #

Bir tarayıcı ortamında çalıştırıldığında JavaScript’te görülen yaygın bir hata, DOM’yi gerekenden daha sık güncellemektir.

Buradaki sorun, DOM arayüzündeki her güncellemenin tarayıcının ekranı yeniden oluşturmasına neden olmasıdır. Bir güncelleme, sayfadaki bir öğenin düzenini değiştirirse, tüm sayfa düzeninin yeniden hesaplanması gerekir ve bu, en basit durumlarda bile çok performans açısından ağırdır. Bir sayfayı yeniden çizme işlemi reflow olarak bilinir ve tarayıcının yavaş çalışmasına ve hatta yanıt vermemesine neden olabilir.

Belgeyi çok sık güncellemenin sonucu, bir listeye öğe eklemeye ilişkin aşağıdaki örnekle gösterilmiştir.

Bir “

    ” öğesi içeren aşağıdaki belgeyi göz önünde bulundurun:

    <!DOCTYPE html>
<html>
    <body>
        <ul id="list"></ul>
    </body>
</html>

    5000 kez döngü yaparak listeye ‘5000’ öğeleri ekliyoruz (etkiyi artırmak için bunu güçlü bir bilgisayarda daha büyük bir sayıyla deneyebilirsiniz).

    var list = document.getElementById("list");
for(var i = 1; i <= 5000; i++) {             
    list.innerHTML += `<li>item ${i}</li>`;  // update 5000 times
}

    Bu durumda, 5000 değişikliğin tümünü tek bir DOM güncellemesinde gruplayarak performans artırılabilir.

    var list = document.getElementById("list");
var html = "";
for(var i = 1; i <= 5000; i++) {
    html += `<li>item ${i}</li>`;
}
list.innerHTML = html;     // update once

    document.createDocumentFragment() işlevi, döngü tarafından oluşturulan HTML için hafif bir kapsayıcı olarak kullanılabilir. Bu yöntem, kapsayıcı öğesinin “innerHTML” özelliğini (aşağıda gösterildiği gibi) değiştirmekten biraz daha hızlıdır.

    var list = document.getElementById("list");
var fragment = document.createDocumentFragment();
for(var i = 1; i <= 5000; i++) {
    li = document.createElement("li");
    li.innerHTML = "item " + i;
    fragment.appendChild(li);
    i++;
}
list.appendChild(fragment);

    Kodunuzu karşılaştırma - yürütme süresini ölçme #

    Performans ipuçlarının çoğu, JS motorlarının mevcut durumuna çok bağlıdır ve yalnızca belirli bir zamanda alakalı olmaları beklenir. Performans optimizasyonunun temel yasası, optimize etmeye çalışmadan önce ölçmeniz ve varsayılan bir optimizasyondan sonra tekrar ölçmeniz gerektiğidir.

    Kod yürütme süresini ölçmek için aşağıdakiler gibi farklı zaman ölçüm araçlarını kullanabilirsiniz:

    Verilen sayfa için zamanlamayla ilgili performans bilgilerini temsil eden Performans arabirimi (yalnızca tarayıcılarda bulunur).

    Node.js’deki process.hrtime, zamanlama bilgilerini [saniye, nanosaniye] demetleri olarak verir. Argümansız çağrıldığında keyfi bir zaman döndürür, ancak argüman olarak daha önce döndürülen bir değerle çağrılır, iki yürütme arasındaki farkı döndürür.

    Konsol zamanlayıcıları console.time("labelName"), bir işlemin ne kadar sürdüğünü izlemek için kullanabileceğiniz bir zamanlayıcı başlatır. Her zamanlayıcıya benzersiz bir etiket adı verirsiniz ve belirli bir sayfada 10.000’e kadar zamanlayıcı çalışıyor olabilir. Aynı adla console.timeEnd("labelName") öğesini çağırdığınızda, tarayıcı verilen ad için zamanlayıcıyı tamamlar ve zamanlayıcı başladığından beri geçen süreyi milisaniye cinsinden verir. time() ve timeEnd() öğelerine iletilen dizeler eşleşmelidir, aksi takdirde zamanlayıcı tamamlanmayacaktır.

    Tarih.now işlevi “Date.now()”, 1 Ocak 1970 00:00:00 UTC’ye kadar 1 Ocak 1970’den bu yana zamanın Sayı temsili olan geçerli Zaman Damgası milisaniye cinsinden döndürür. şimdi. Now() yöntemi, Date’in statik bir yöntemidir, bu nedenle onu her zaman Date.now() olarak kullanırsınız.

    Örnek 1 şunun kullanıldığı: performance.now()

    Bu örnekte, fonksiyonumuzun yürütülmesi için geçen süreyi hesaplayacağız ve milisaniye cinsinden ölçülen bir DOMHighResTimeStamp döndüren Performance.now() yöntemini kullanacağız, milisaniyenin binde birine kadar doğrudur.

    let startTime, endTime;

function myFunction() {
    //Slow code you want to measure
}

//Get the start time
startTime = performance.now();

//Call the time-consuming function
myFunction();

//Get the end time
endTime = performance.now();

//The difference is how many milliseconds it took to call myFunction()
console.debug('Elapsed time:', (endTime - startTime));

    Konsoldaki sonuç şöyle görünecek:

    Elapsed time: 0.10000000009313226

    performance.now() kullanımı, tarayıcılarda milisaniyenin binde birine kadar doğrulukla en yüksek hassasiyete sahiptir, ancak en düşük uyumluluk.

    Örnek 2 şunun kullanıldığı: Date.now()

    Bu örnekte büyük bir dizinin (1 milyon değer) başlatılması için geçen süreyi hesaplayacağız ve Date.now() yöntemini kullanacağız.

    let t0 = Date.now(); //stores current Timestamp in milliseconds since 1 January 1970 00:00:00 UTC
let arr = []; //store empty array
for (let i = 0; i < 1000000; i++) { //1 million iterations
   arr.push(i); //push current i value
}
console.log(Date.now() - t0); //print elapsed time between stored t0 and now

    Örnek 3 şunun kullanıldığı: console.time("label") & console.timeEnd("label")

    Yoğun bilgi işlem işlevleri için bir memoizer kullanın #

    İşlemci üzerinde ağır olabilecek bir işlev oluşturuyorsanız (istemci tarafı veya sunucu tarafı), *önceki işlev yürütmelerinin ve bunların döndürülen değerlerinin önbelleği olan bir memoizer‘ı düşünmek isteyebilirsiniz. Bu, bir fonksiyonun parametrelerinin daha önce geçirilip geçirilmediğini kontrol etmenizi sağlar. Unutmayın, saf işlevler bir girdi veren, karşılık gelen benzersiz bir çıktı döndüren ve kapsamları dışında yan etkilere neden olmayan işlevlerdir, bu nedenle, öngörülemeyen veya harici kaynaklara bağlı işlevlere (AJAX çağrıları veya rastgele gibi) hatırlatıcılar eklememelisiniz. döndürülen değerler).

    Diyelim ki özyinelemeli bir faktöriyel işlevim var:

    function fact(num) {
  return (num === 0)? 1 : num * fact(num - 1);
}

    Örneğin 1’den 100’e kadar küçük değerleri iletirsem sorun olmazdı, ancak daha derine inmeye başladığımızda, çağrı yığınını patlatabilir veya bunu yaptığımız Javascript motoru için süreci biraz acı verici hale getirebiliriz, özellikle motor kuyruk çağrısı optimizasyonu ile sayılmazsa (Douglas Crockford, yerel ES6’nın kuyruk çağrısı optimizasyonunun dahil olduğunu söylese de).

    Kendi sözlüğümüzü 1’den tanrı bilir-hangi sayıya karşılık gelen faktöriyelleriyle sabit kodlayabiliriz, ancak bunu tavsiye edip etmediğimden emin değilim! Bir memoizer oluşturalım, olur mu?

    var fact = (function() {
  var cache = {}; // Initialise a memory cache object
  
  // Use and return this function to check if val is cached
  function checkCache(val) {
    if (val in cache) {
      console.log('It was in the cache :D');
      return cache[val]; // return cached
    } else {
      cache[val] = factorial(val); // we cache it
      return cache[val]; // and then return it
    }
    
    /* Other alternatives for checking are:
    || cache.hasOwnProperty(val) or !!cache[val]
    || but wouldn't work if the results of those
    || executions were falsy values.
    */
  }

  // We create and name the actual function to be used
  function factorial(num) {
    return (num === 0)? 1 : num * factorial(num - 1);
  } // End of factorial function

  /* We return the function that checks, not the one
  || that computes because  it happens to be recursive,
  || if it weren't you could avoid creating an extra
  || function in this self-invoking closure function.
  */
  return checkCache; 
}());

    Artık kullanmaya başlayabiliriz:

    Şimdi ne yaptığımı düşünmeye başladığıma göre, num‘dan azaltmak yerine 1’den artıracak olsaydım, 1’den num‘a kadar olan tüm faktöriyelleri tekrar tekrar önbellekte önbelleğe alabilirdim, ama bunu bırakacağım. senin için.

    Bu harika ama ya birden çok parametremiz varsa? Bu bir problem? Tam olarak değil, argümanlar dizisinde JSON.stringify() kullanmak veya hatta işlevin bağlı olacağı bir değerler listesi (nesne yönelimli yaklaşımlar için) kullanmak gibi bazı güzel numaralar yapabiliriz. Bu, dahil edilen tüm bağımsız değişkenler ve bağımlılıklar ile benzersiz bir anahtar oluşturmak için yapılır.

    Ayrıca, öncekiyle aynı kapsam kavramını kullanarak (orijinali kullanan ve önbellek nesnesine erişimi olan yeni bir işlev döndürerek) diğer işlevleri “hatırlayan” bir işlev de oluşturabiliriz:

    UYARI: ES6 sözdizimi, beğenmediyseniz, … öğesini hiçbir şeyle değiştirin ve `var args = Array.prototype.slice.call(null, argümanlar);’ hilesini kullanın; const ve let’i var ile değiştirin ve zaten bildiğiniz diğer şeyler.

    function memoize(func) {
  let cache = {};

  // You can opt for not naming the function
  function memoized(...args) {
    const argsKey = JSON.stringify(args);
    
    // The same alternatives apply for this example
    if (argsKey in cache) {
      console.log(argsKey + ' was/were in cache :D');
      return cache[argsKey];
    } else {
      cache[argsKey] = func.apply(null, args); // Cache it
      return cache[argsKey]; // And then return it
    }
  }

  return memoized; // Return the memoized function
}

    Şimdi bunun birden fazla argüman için işe yarayacağını, ancak nesne yönelimli yöntemlerde fazla kullanılmayacağını düşünüyorum, bağımlılıklar için fazladan bir nesneye ihtiyacınız olabilir. Ayrıca, “func.apply(null, args)”, “func(…args)” ile değiştirilebilir, çünkü dizi yok etme onları bir dizi formu yerine ayrı olarak gönderir. Ayrıca, sadece referans olarak, benim yaptığım gibi Function.prototype.apply kullanmadığınız sürece bir diziyi func’a argüman olarak iletmek işe yaramaz.

    Yukarıdaki yöntemi kullanmak için sadece:

    const newFunction = memoize(oldFunction);

// Assuming new oldFunction just sums/concatenates:
newFunction('meaning of life', 42);
// -> "meaning of life42"

newFunction('meaning of life', 42); // again
// => ["meaning of life",42] was/were in cache :D
// -> "meaning of life42"

    Nesne özelliklerini null ile başlatma #

    Beklenen nesne yapılarına dayalı olarak kodu optimize etmeye çalışan tüm modern JavaScript JIT derleyicileri. mdn‘den bir ipucu.

    Neyse ki, nesneler ve özellikler genellikle “öngörülebilir” ve bu tür vakaların altında yatan yapı da tahmin edilebilir. JIT’ler öngörülebilir erişimleri daha hızlı hale getirmek için buna güvenebilir.

    Nesneyi öngörülebilir hale getirmenin en iyi yolu, bir yapıcıda bütün bir yapı tanımlamaktır. Bu nedenle, nesne oluşturulduktan sonra bazı ekstra özellikler ekleyecekseniz, bunları bir kurucuda ’null’ ile tanımlayın. Bu, optimize edicinin tüm yaşam döngüsü boyunca nesne davranışını tahmin etmesine yardımcı olacaktır. Bununla birlikte, tüm derleyicilerin farklı optimize edicileri vardır ve performans artışı farklı olabilir, ancak genel olarak, değerleri henüz bilinmese bile bir kurucudaki tüm özellikleri tanımlamak iyi bir uygulamadır.

    Biraz test zamanı. Testimde, bazı sınıf örneklerinden oluşan büyük bir dizi oluşturuyorum. bir for döngüsü. Döngü içinde, dizi başlatmadan önce tüm nesnelerin “x” özelliğine aynı dizeyi atıyorum. Yapıcı “x” özelliğini null ile başlatırsa, fazladan ifade yapsa bile dizi her zaman daha iyi işler.

    Bu kod:

    function f1() {
    var P = function () {
        this.value = 1
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}

function f2() {
    var P = function () {
        this.value = 1;
        this.x = null;
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}


(function perform(){
    var start = performance.now();

    Örnek A #

    var i,a,b,len;
a = {x:0,y:0}
function test(){ // return object created each call
    return {x:0,y:0};
}
function test1(a){ // return object supplied 
    a.x=0;
    a.y=0;
    return a;
}   

for(i = 0; i < 100; i ++){ // Loop A
   b = test();
}

for(i = 0; i < 100; i ++){ // Loop B
   b = test1(a);
}

    Performans kodunda yeni bir nesne oluşturmak çok verimsizdir. Döngü A, her çağrıda yeni bir nesne döndüren ’test()’ işlevini çağırır. Oluşturulan nesne her yinelemede atılır, Döngü B, nesne dönüşlerinin sağlanmasını gerektiren “test1()“i çağırır. Böylece aynı nesneyi kullanır ve yeni bir nesnenin tahsis edilmesini ve aşırı GC isabetlerini önler. (GC, performans testine dahil edilmemiştir)

    Yerel değişkenleri globallere, niteliklere ve dizine alınmış değerlere tercih edin #

    Javascript motorları, aramalarını daha geniş kapsamlara genişletmeden önce yerel kapsamdaki değişkenleri arar. Değişken, bir dizideki dizinlenmiş bir değer veya bir ilişkisel dizideki bir öznitelik ise, içeriği bulmadan önce ilk olarak üst diziyi arayacaktır.

    Bunun, performans açısından kritik kodla çalışırken etkileri vardır. Örneğin, ortak bir “for” döngüsü alın:

    var global_variable = 0;
function foo(){
    global_variable = 0;
    for (var i=0; i<items.length; i++) {
        global_variable += items[i];
    }
}

    “for” döngüsündeki her yineleme için, motor “öğeleri” arayacak, öğelerin içindeki “uzunluk” niteliğini arayacak, tekrar “öğeleri” arayacak, “öğelerin” “i” dizinindeki değeri arayacak ve son olarak “öğeler” arayacaktır. global_variable`, global kapsamı kontrol etmeden önce yerel kapsamı deneyin.

    Yukarıdaki işlevin bir performans yeniden yazılması:

    function foo(){
    var local_variable = 0;
    for (var i=0, li=items.length; i<li; i++) {
        local_variable += items[i];
    }
    return local_variable;
}

    Yeniden yazılan “for” döngüsündeki her yineleme için, motor “li"yi arar, “öğeleri” arar, “i” dizinindeki değeri arar ve “yerel_değişkeni” arar, bu sefer yalnızca yerel kapsamı kontrol etmesi gerekir.

    Sayıların kullanımında tutarlı olun #

    Motor, değerleriniz için belirli bir küçük tür kullandığınızı doğru bir şekilde tahmin edebiliyorsa, yürütülen kodu optimize edebilecektir.

    Bu örnekte, bir dizinin öğelerini toplayan ve bunun için geçen süreyi çıkaran bu önemsiz işlevi kullanacağız:

    // summing properties
var sum = (function(arr){
        var start = process.hrtime();
        var sum = 0;
        for (var i=0; i<arr.length; i++) {
                sum += arr[i];
        }
        var diffSum = process.hrtime(start);
        console.log(`Summing took ${diffSum[0] * 1e9 + diffSum[1]} nanoseconds`);
        return sum;
})(arr);

    Bir dizi oluşturalım ve öğeleri toplayalım:

    var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.random();

    Sonuç:

    Summing took 384416 nanoseconds

    Şimdi aynısını sadece tam sayılarla yapalım:

    var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.round(1000*Math.random());

    Sonuç:

    Summing took 180520 nanoseconds

    Tam sayıları toplamak buradaki zamanın yarısını aldı.

    Motorlar, JavaScript’te sahip olduğunuz türlerin aynısını kullanmaz. Muhtemelen bildiğiniz gibi, JavaScript’teki tüm sayılar IEEE754 çift duyarlıklı kayan noktalı sayılardır, tamsayılar için belirli bir gösterim yoktur. Ancak motorlar, yalnızca tamsayıları kullandığınızı tahmin edebildiklerinde, örneğin kısa tamsayılar gibi temsili kullanmak için daha kompakt ve daha hızlı kullanabilirler.

    Bu tür bir optimizasyon, özellikle hesaplama veya veri yoğun uygulamalar için önemlidir.
