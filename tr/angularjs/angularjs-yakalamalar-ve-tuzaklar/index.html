html5Mode kullanırken yapılması gerekenler #

“html5Mode([mode])” kullanılırken şunlar gereklidir:

  1. Uygulamanın temel URL’sini, “index.html"nizin başında bir “” ile belirtirsiniz.

  2. ‘base’ etiketinin, url istekleri olan herhangi bir etiketten önce gelmesi önemlidir. Aksi takdirde, bu şu hataya neden olabilir - "Kaynak stil sayfası olarak yorumlandı ancak MIME tipi metin/html ile aktarıldı". Örneğin:

     <head>
     <meta charset="utf-8">
     <title>Job Seeker</title>

     <base href="/">

     <link rel="stylesheet" href="bower_components/bootstrap/dist/css/bootstrap.css" />
     <link rel="stylesheet" href="/styles/main.css">
 </head>

  3. Bir “temel” etiketi belirtmek istemiyorsanız, “requireBase:false” içeren bir tanım nesnesini “$locationProvider.html5Mode()” öğesine bu şekilde ileterek “$locationProvider” öğesini bir “base” etiketi gerektirmeyecek şekilde yapılandırın. :

     $locationProvider.html5Mode({
     enabled: true,
     requireBase: false
 });

  4. HTML5 URL’lerinin doğrudan yüklenmesini desteklemek için, sunucu tarafı URL yeniden yazmayı etkinleştirmeniz gerekir. AngularJS / Geliştirici Kılavuzu / $konumu Kullanmak‘dan

Bu modu kullanmak, sunucu tarafında URL’nin yeniden yazılmasını gerektirir, temel olarak tüm bağlantılarınızı uygulamanızın giriş noktasına (ör. `index.html’) yeniden yazmanız gerekir. Angular’ın url’nin uygulama tabanı olan kısmı ile uygulama tarafından işlenmesi gereken yol arasında ayrım yapmasına izin verdiğinden, bu durumda bir “” etiketinin gerekli olması da önemlidir.

Çeşitli HTTP sunucusu uygulamaları için istek yeniden yazma örnekleri için mükemmel bir kaynak ui-router SSS - Nasıl Yapılır: Sunucunuzu html5Mode ile çalışacak şekilde yapılandırın bölümünde bulunabilir. Örneğin, Apaçi

    RewriteEngine on

    # Don't rewrite files or directories
    RewriteCond %{REQUEST_FILENAME} -f [OR]
    RewriteCond %{REQUEST_FILENAME} -d
    RewriteRule ^ - [L]

    # Rewrite everything else to index.html to allow html5 state links
    RewriteRule ^ index.html [L]

nginx

    server {
        server_name my-app;

        root /path/to/app;

        location / {
            try_files $uri $uri/ /index.html;
        }
    }

İfade etmek

    var express = require('express');
    var app = express();

    app.use('/js', express.static(__dirname + '/js'));
    app.use('/dist', express.static(__dirname + '/../dist'));
    app.use('/css', express.static(__dirname + '/css'));
    app.use('/partials', express.static(__dirname + '/partials'));

    app.all('/*', function(req, res, next) {
        // Just send the index.html for other files to support HTML5Mode
        res.sendFile('index.html', { root: __dirname });
    });

    app.listen(3006); //the port you want to use

İki yönlü veri bağlama çalışmayı durdurur #

Şunu akılda tutmak gerekir:

  1. Angular’ın veri bağlaması JavaScript’in prototip mirasına dayanır, bu nedenle değişken gölgelemeye tabidir.
  2. Bir alt kapsam normalde prototipik olarak kendi üst kapsamından miras alır. Bu kuralın bir istisnası, prototipik olarak devralmadığı için yalıtılmış bir kapsamı olan bir yönergedir.
  3. Yeni bir çocuk kapsamı oluşturan bazı yönergeler vardır: ’ng-repeat’, ’ng-switch’, ’ng-view’, ’ng-if’, ’ng-controller’, ’ng-include’, vb. .

Bu, bazı verileri bir alt kapsamın içindeki bir ilkele iki yönlü bağlamaya çalıştığınızda (veya tam tersi), işlerin beklendiği gibi çalışmayabileceği anlamına gelir. Burada AngularJS’yi “kırmanın” ne kadar kolay olduğuna dair bir örnek.

Bu sorun, aşağıdaki adımları izleyerek kolayca önlenebilir:

  1. Bir “.” bazı verileri bağladığınızda HTML şablonunuzun içinde
  2. “noktalı” bir nesneye bağlama kullanımını teşvik ettiği için “controllerAs” sözdizimini kullanın
  3. $parent, alt kapsam yerine ana “kapsam” değişkenlerine erişmek için kullanılabilir. ng-if içinde olduğu gibi ng-model="$parent.foo" kullanabiliriz..

Yukarıdakilere bir alternatif, argümanlarla çağrıldığında modelin önbelleğe alınmış sürümünü güncelleyecek veya argüman olmadan çağrıldığında onu döndürecek olan bir alıcı/ayarlayıcı işlevine ngModel’i bağlamaktır. Bir alıcı/ayarlayıcı işlevini kullanmak için, ’ngModal’ özniteliğine sahip öğeye ’ng-model-options=”{ getterSetter: true }"’ eklemeniz ve görüntülemek istiyorsanız alıcı işlevini çağırmanız gerekir. ifadedeki değeri (Çalışma örneği).

Örnek #

Görüş:

<div ng-app="myApp" ng-controller="MainCtrl">
    <input type="text" ng-model="foo" ng-model-options="{ getterSetter: true }">
    <div ng-if="truthyValue">
        <!-- I'm a child scope (inside ng-if), but i'm synced with changes from the outside scope -->
        <input type="text" ng-model="foo">
    </div>
    <div>$scope.foo: {{ foo() }}</div>
</div>

Denetleyici:

angular.module('myApp', []).controller('MainCtrl', ['$scope', function($scope) {
    $scope.truthyValue = true;
      
    var _foo = 'hello'; // this will be used to cache/represent the value of the 'foo' model 
      
    $scope.foo = function(val) {
        // the function return the the internal '_foo' varibale when called with zero arguments,
        // and update the internal `_foo` when called with an argument
        return arguments.length ? (_foo = val) : _foo;
    };
}]);

En İyi Uygulama: Alıcıları hızlı tutmak en iyisidir çünkü Angular onları kodunuzun diğer bölümlerinden daha sık çağırır (referans )).

AngularJS’nin 7 Ölümcül Günahı #

Aşağıda, geliştiricilerin AngularJS işlevlerinin kullanımı sırasında sıklıkla yaptığı bazı hataların listesi, öğrenilen bazı dersler ve bunlara yönelik çözümler yer almaktadır.

1. DOM’yi denetleyici aracılığıyla değiştirme

Yasaldır, ancak kaçınılmalıdır. Denetleyiciler, bağımlılıklarınızı tanımladığınız, verilerinizi görünüme bağladığınız ve daha fazla iş mantığı oluşturduğunuz yerlerdir. DOM’yi bir denetleyicide teknik olarak değiştirebilirsiniz, ancak uygulamanızın başka bir bölümünde aynı veya benzer bir işleme ihtiyaç duyduğunuzda, başka bir denetleyiciye ihtiyaç duyulacaktır. Bu nedenle, bu yaklaşımın en iyi uygulaması, tüm manipülasyonları içeren bir yönerge oluşturmak ve yönergeyi uygulamanız boyunca kullanmaktır. Bu nedenle, denetleyici görünümü olduğu gibi bırakır ve işini yapar. Bir yönergede, bağlantı işlevi, DOM’yi işlemek için en iyi yerdir. Kapsam ve öğeye tam erişime sahiptir, bu nedenle bir yönerge kullanarak yeniden kullanılabilirlikten de yararlanabilirsiniz.

link: function($scope, element, attrs) {
    //The best place to manipulate DOM
}

DOM öğelerine, “element” parametresi, “angular.element()” yöntemi veya salt Javascript gibi çeşitli yollarla bağlantı işlevinde erişebilirsiniz.

2. Dönüştürmede veri bağlama

AngularJS, iki yönlü veri bağlamasıyla ünlüdür. Ancak bazen verilerinizin direktifler içinde sadece tek yönlü olduğu ile karşılaşabilirsiniz. Orada durun, AngularJS yanlış değil ama muhtemelen sizsiniz. Alt kapsamlar ve yalıtılmış kapsamlar söz konusu olduğundan, yönergeler biraz tehlikeli yerlerdir. Bir transclusion ile aşağıdaki yönergeye sahip olduğunuzu varsayalım.

<my-dir>
    <my-transclusion>
    </my-transclusion>
</my-dir>

Ve benim transklüzyonumun içinde, dış kapsamdaki verilere bağlı bazı öğeleriniz var.

<my-dir>
    <my-transclusion>
        <input ng-model="name">
    </my-transclusion>
</my-dir>

Yukarıdaki kod düzgün çalışmayacaktır. Burada, transclusion bir alt kapsam yaratır ve isim değişkenini alabilirsiniz, doğru, ancak bu değişkende yaptığınız her değişiklik orada kalacaktır. Böylece, bu değişkene gerçekten $parent.name olarak erişebilirsiniz. Ancak, bu kullanım en iyi uygulama olmayabilir. Değişkenleri bir nesnenin içine sarmak daha iyi bir yaklaşım olacaktır. Örneğin, denetleyicide şunları oluşturabilirsiniz:

$scope.data = {
    name: 'someName'
}

Ardından, dönüştürmede, bu değişkene ‘data’ nesnesi aracılığıyla erişebilir ve iki yönlü bağlamanın mükemmel çalıştığını görebilirsiniz!

<input ng-model="data.name">

Yalnızca dönüştürmelerde değil, uygulama genelinde noktalı gösterimi kullanmak iyi bir fikirdir.

3. Birden çok yönerge bir arada

Kurala uyduğunuz sürece, aynı öğe içinde iki yönergeyi birlikte kullanmak aslında yasaldır: aynı öğe üzerinde iki ayrı kapsam bulunamaz. Genel olarak konuşursak, yeni bir özel yönerge oluştururken, kolay parametre geçişi için yalıtılmış bir kapsam tahsis edersiniz. myDirA ve myDirB yönergelerinin yalıtılmış kapsamlara sahip olduğunu ve myDirC’nin olmadığını varsayarsak, aşağıdaki öğe geçerli olacaktır:

<input my-dir-a my-dirc>

aşağıdaki öğe konsol hatasına neden olurken:

<input my-dir-a my-dir-b>

Bu nedenle direktifler, kapsamları dikkate alınarak akıllıca kullanılmalıdır.

4. $emit‘in yanlış kullanımı

$emit, $yayın ve $on, bunlar gönderici-alıcı prensibiyle çalışır. Başka bir deyişle, denetleyiciler arasında bir iletişim aracıdır. Örneğin, aşağıdaki satır, ilgili denetleyici B tarafından yakalanması için denetleyici A’dan ‘someEvent’ yayar.

$scope.$emit('someEvent', args);

Ve aşağıdaki satır ‘someEvent’i yakalar

$scope.$on('someEvent', function(){});

Buraya kadar her şey mükemmel görünüyor. Ancak, B denetleyicisi henüz çağrılmazsa, olayın yakalanmayacağını unutmayın; bu, bu çalışmayı sağlamak için hem verici hem de alıcı denetleyicilerinin çağrılması gerektiği anlamına gelir. Yine, kesinlikle $emit kullanmanız gerektiğinden emin değilseniz, bir hizmet oluşturmak daha iyi bir yol gibi görünüyor.

5. $scope.$watch‘un yanlış kullanımı

$scope.$watch değişken değişikliğini izlemek için kullanılır. Bir değişken değiştiğinde, bu yöntem çağrılır. Ancak, yapılan yaygın bir hata, $scope.$watch içindeki değişkeni değiştirmektir. Bu, bir noktada tutarsızlığa ve sonsuz $digest döngüsüne neden olacaktır.

$scope.$watch('myCtrl.myVariable', function(newVal) {
    this.myVariable++;
});

Bu nedenle, yukarıdaki işlevde, myVariable ve newVal üzerinde herhangi bir işlem yapmadığınızdan emin olun.

6. Görünümlere bağlama yöntemleri

Bu en ölümcül günahlardan biridir. AngularJS’nin iki yönlü bağlaması vardır ve bir şey değiştiğinde, görünümler birçok kez güncellenir. Bu nedenle, bir yöntemi bir görünümün özniteliğine bağlarsanız, bu yöntem potansiyel olarak yüz kez çağrılabilir ve bu da hata ayıklama sırasında sizi çıldırtabilir. Ancak, yöntemleri parametre olarak bekleyen ng-click, ng-blur, ng-on-change, vb. gibi yöntem bağlama için oluşturulmuş yalnızca bazı öznitelikler vardır. Örneğin, işaretlemenizde aşağıdaki görünüme sahip olduğunuzu varsayalım:

<input ng-disabled="myCtrl.isDisabled()" ng-model="myCtrl.name">

Burada görünümün devre dışı durumunu isDisabled yöntemiyle kontrol edersiniz. myCtrl denetleyicisinde şunlara sahipsiniz:

vm.isDisabled = function(){
    if(someCondition)
        return true;
    else
        return false;
}

Teorik olarak doğru görünebilir, ancak teknik olarak bu, yöntem sayısız kez çalışacağı için aşırı yüklenmeye neden olacaktır. Bunu çözmek için bir değişkeni bağlamanız gerekir. Kontrol cihazınızda aşağıdaki değişken bulunmalıdır:

vm.isDisabled

Denetleyicinin aktivasyonunda bu değişkeni tekrar başlatabilirsiniz.

if(someCondition)
    vm.isDisabled = true
else
    vm.isDisabled = false

Durum stabil değilse, bunu başka bir olaya bağlayabilirsiniz. O zaman bu değişkeni görünüme bağlamanız gerekir:

<input ng-disabled="myCtrl.isDisabled" ng-model="myCtrl.name">

Şimdi, görünümün tüm öznitelikleri beklediklerine sahiptir ve yöntemler yalnızca gerektiğinde çalışacaktır.

7. Angular’ın işlevlerini kullanmamak

AngularJS, yalnızca kodunuzu basitleştirmekle kalmayıp aynı zamanda daha verimli hale getirerek bazı işlevleriyle büyük kolaylık sağlar. Bu özelliklerden bazıları aşağıda listelenmiştir:

  1. angular.forEach for the loops (Caution, you can't "break;" it, you can only prevent getting into the body, so consider performance here.)
  2. angular.element for DOM selectors
  3. angular.copy: Use this when you should not modify the main object
  4. Form validations are already awesome. Use dirty, pristine, touched, valid, required and so on.
  5. Besides Chrome debugger, use remote debugging for mobile development too.
  6. And make sure you use Batarang. It's a free Chrome extension where you can easily inspect scopes
