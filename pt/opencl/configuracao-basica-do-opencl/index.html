Antes de utilizar o OpenCL, é preciso configurar seu código para usá-lo. Este tópico se concentra em como colocar o opencl em execução em seu projeto e executar um kernel básico. Os exemplos são baseados no wrapper C# OpenCL.NET, mas como o wrapper não adiciona nenhuma abstração ao OpenCL, o código provavelmente será executado com poucas alterações também no C/C++.

As chamadas em C# podem ter a seguinte aparência: ‘Cl.GetPlatformIDs’. Para o C-Style OpenCL Api você chamaria ‘clGetPlatformIDs’ e para o estilo C++ um ‘cl::GetPlatformIDs’

  • NVidia, AMD e Intel têm implementações ligeiramente diferentes do OpenCL, mas as diferenças conhecidas são (para minha experiência) limitadas a requisitos de parênteses e conversões implícitas. Às vezes, a NVidia trava seu kernel enquanto tenta descobrir a sobrecarga correta para um método. Nesse caso, é útil oferecer um cast explícito para auxiliar a GPU. O problema foi observado para kernels compilados em tempo de execução.

  • Para obter mais informações sobre as chamadas utilizadas neste tópico, basta pesquisar no Google ‘OpenCL’ seguido do nome da função. O grupo Khronos possui uma documentação completa on all parameters and datatypes available on their website.

Inicializando o dispositivo de destino #

Kernels OpenCL podem ser executados na GPU ou na CPU. Isso permite soluções de fallback, onde o cliente pode ter um sistema muito desatualizado. O programador também pode optar por limitar sua funcionalidade à CPU ou GPU.

Para começar a usar o OpenCL, você precisará de um ‘Contexto’ e um ‘Dispositivo’. Ambas são estruturas definidas pela API OpenCL (também conhecidas como cl::Context ou clContext & ~Device) e definem o processador de destino usado.

Para obter seu dispositivo e contexto, você precisa consultar uma lista de plataformas disponíveis, cada uma podendo hospedar vários dispositivos. Uma plataforma representa sua GPU física e CPU, enquanto um dispositivo pode distinguir ainda mais as unidades de computação contidas. Para GPUs, a maioria das plataformas terá apenas um dispositivo. Mas uma CPU pode oferecer uma GPU integrada adicional além de seus recursos de CPU.

O contexto gerencia a memória, as filas de comandos, os diferentes kernels e programas. Um contexto pode ser limitado a um único dispositivo, mas também pode fazer referência a vários dispositivos.

Uma nota rápida sobre a API antes de começarmos a codificar: Quase todas as chamadas para OpenCL fornecem um valor de erro, seja como valor de retorno ou por meio de um valor de referência (ponteiro em C). Agora vamos começar.

ErrorCode err;
var platforms = Cl.GetPlatformIDs(out err);
if(!CheckError(err, "Cl.GetPlatformIDs")) return;
foreach (var platform in platforms) {
    foreach (var device in Cl.GetDeviceIDs(platform, DeviceType.Gpu, out err)) {
        if(!CheckError(err, "Cl.GetDeviceIDs")) continue;
        [...]
    }
}

Este snippet de código consulta todos os dispositivos GPU disponíveis no sistema. Agora você pode adicioná-los a uma lista ou iniciar seu contexto diretamente com a primeira correspondência. A função ‘CheckError(…)’ é um utilitário simples, que verifica se o código de erro tem o valor de sucesso ou um diferente e pode oferecer algum log. Recomenda-se usar uma função ou macro separada, porque você vai chamar muito isso.

ErrorCode é apenas uma enumeração no tipo de dados cl_int para C#, C/C++ pode comparar o valor int com constantes de erro predefinidas, conforme listado aqui: https://www.khronos.org/registry/OpenCL/sdk/1.0/docs/man/xhtml/errors.html

Você também pode querer verificar se o dispositivo suporta todos os recursos necessários, caso contrário, seus kernels podem travar em tempo de execução. Você pode consultar um recurso de dispositivo com

Cl.GetDeviceInfo(_device, DeviceInfo.ImageSupport, out err)

Este exemplo pergunta ao dispositivo se ele pode executar funções de imagem. Para a próxima e última etapa, precisamos construir nosso contexto a partir dos dispositivos coletados.

_context = Cl.CreateContext(null, 1, new[] { _device }, ContextNotify, IntPtr.Zero, out err);

Algumas coisas estão acontecendo aqui. Para o pessoal de C/C++, IntPtr é um endereço de ponteiro em C#. Vou me concentrar nas partes importantes aqui.

  • O segundo parâmetro define o número de dispositivos que você deseja usar
  • O terceiro parâmetro é um array desses dispositivos (ou um ponteiro em C/C++)
  • E o terceiro parâmetro é um ponteiro de função para uma função de retorno de chamada. Esta função será utilizada sempre que ocorrerem erros dentro do contexto.

Para uso posterior, você precisará preservar seus dispositivos usados ​​e o contexto em algum lugar.

Quando você terminar toda a sua interação OpenCL, você precisará liberar o contexto novamente com

Cl.ReleaseContext(_context);

Compilando seu Kernel #

Os kernels podem ser compilados em tempo de execução no dispositivo de destino. Para isso, você precisa

  • o código-fonte do kernel
  • o dispositivo de destino no qual compilar
  • um contexto construído com o dispositivo de destino

Uma rápida atualização de terminologia: um programa contém uma coleção de kernels. Você pode pensar em um programa como um arquivo de origem C/C++/C# completo, enquanto os kernels são os diferentes membros de função desse arquivo.

Primeiro você precisará criar um programa a partir de seu código-fonte.

var program = Cl.CreateProgramWithSource(_context, 1, new[] { source }, null, out err);

Você pode combinar vários arquivos de origem em um programa e compilá-los juntos, o que permite ter kernels em arquivos diferentes e compilá-los juntos de uma só vez.

Na próxima etapa, você precisará compilar o programa em seu dispositivo de destino.

err = Cl.BuildProgram(program, 1, new[] { _device }, string.Empty, null, IntPtr.Zero);

Agora vem uma pequena advertência: O código de erro apenas informa se a chamada da função em si foi bem-sucedida, mas não se seu código realmente compilou. Para verificar isso, temos que consultar algumas informações adicionais

BuildStatus status;
status = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Status, out err).CastTo<BuildStatus>();
if (status != BuildStatus.Success) {
    var log = Cl.GetProgramBuildInfo(program, _device, ProgramBuildInfo.Log, out err);
}

Pessoas de C/C++ podem ignorar a conversão no final e apenas comparar o inteiro retornado com a constante correspondente.

A primeira chamada verifica se nossa compilação foi realmente bem-sucedida. Caso contrário, podemos recuperar um log e ver exatamente onde as coisas deram errado. Veja as observações para algumas armadilhas comuns em relação a diferentes plataformas.

Uma vez que o programa é compilado, você precisa extrair seus diferentes kernels do programa compilado. Para fazer isso, você cria seus kernels com

_kernel = Cl.CreateKernel(_program, kernel, out err);

onde ‘kernel’ é uma string do nome do kernel. Quando você terminar com seu kernel, você precisa liberá-lo com

Cl.ReleaseKernel(_kernel);

Criando uma fila de comandos #

Para iniciar qualquer operação em seus dispositivos, você precisará de uma fila de comandos para cada dispositivo. A Fila acompanha as diferentes chamadas que você fez para o dispositivo de destino e as mantém em ordem. A maioria dos comandos também pode ser executada no modo de bloqueio ou não.

Criar uma fila é bastante simples:

_queue = Cl.CreateCommandQueue(_context, _device, CommandQueueProperties.None, out err);

A interação básica com sua fila de comandos é enfileirar diferentes operações que você deseja executar, por exemplo, copie dados de e para o seu dispositivo e inicie um kernel.

Quando você terminar de usar a fila de comandos, você precisa liberar a fila com uma chamada para

Cl.ReleaseCommandQueue(_queue);

Executando o Kernel #

Então agora vamos ao que interessa, executando seus kernels no dispositivo paralelo. Por favor, leia sobre o básico de hardware para entender completamente o despacho do kernel.

Primeiro você precisará definir os argumentos do kernel antes de realmente chamar o kernel. Isso é feito através

err = Cl.SetKernelArg(_kernel, $argumentIndex, $argument);

Se você não definir todos os argumentos antes de iniciar o kernel, o kernel falhará.

Antes de realmente lançarmos nosso kernel, precisamos calcular o ’tamanho do trabalho global’ e o ’tamanho do trabalho local’.

o tamanho do trabalho global é o número total de threads que serão lançados em sua GPU. O tamanho do trabalho local é o número de threads dentro de cada bloco de thread. O tamanho do trabalho local pode ser omitido se o kernel não precisar de nenhum requisito especial. Mas se o tamanho do trabalho local for fornecido, o tamanho do trabalho global deve ser um múltiplo do tamanho do trabalho local.

Os tamanhos de trabalho podem ser unidimensionais, bidimensionais ou tridimensionais. A escolha de quantas dimensões você deseja é inteiramente sua e você pode escolher o que melhor se adequar ao seu algoritmo.

Agora que decidimos o tamanho do nosso trabalho, podemos chamar o kernel.

Event clevent;
err = Cl.EnqueueNDRangeKernel(_queue, _kernel, $dimensions, null, $globalWorkSize, $localWorkSize, 0, null, out clevent);

As $dimensions definem nosso número desejado de dimensões, $globalWorkSize é uma matriz de tamanho $dimensions com o tamanho global de trabalho e o mesmo para $localWorkSize. O último argumento fornece um objeto que representa seu kernel atualmente executado.
