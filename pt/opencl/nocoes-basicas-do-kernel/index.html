Este tópico tem como objetivo explicar os fundamentos da escrita de kernels para opencl

Kernel em tons de cinza #

Vamos construir um kernel para gerar uma imagem em tons de cinza. Usaremos dados de imagem que são definidos usando uints para cada componente e com a ordem RGBA.

__constant sampler_t sampler = CLK_NORMALIZED_COORDS_FALSE |
                                CLK_ADDRESS_CLAMP_TO_EDGE |
                                CLK_FILTER_LINEAR;

__kernel void Grayscale(__read_only image2d_t input, __write_only image2d_t output) {
    int2 gid = (int2)(get_global_id(0), get_global_id(1));
    int2 size = get_image_dim(input);

    if(all(gid < size)){
        uint4 pixel = read_imageui(input, sampler, gid);
        float4 color = convert_float4(pixel) / 255;
        color.xyz = 0.2126*color.x + 0.7152*color.y + 0.0722*color.z;
        pixel = convert_uint4_rte(color * 255);
        write_imageui(output, gid, pixel);
    }
}

Agora vamos percorrer esse código passo a passo. A primeira linha cria uma variável na região de memória __constant do tipo sampler_t. Este amostrador é usado para especificar ainda mais o acesso aos nossos dados de imagem. Consulte o Khronos Docs para obter uma documentação completa.

Nós alocamos a entrada como read_only e a saída como write_only antes de chamarmos nosso kernel, então adicionamos esses modificadores aqui.

image2d e image3d são sempre alocados na memória global, portanto, podemos omitir o modificador __global aqui.

Em seguida, obtemos nosso ID de thread que define o pixel que vamos converter em escala de cinza. Também consultamos o tamanho para garantir que nosso encadeamento não esteja acessando memória não alocada. Isso definitivamente travará seu kernel se você esquecer isso.

Depois de nos certificarmos de que somos um thread legítimo, lemos nosso pixel de nossa imagem de entrada. Em seguida, convertemos para float para evitar perda de casas decimais, fazemos alguns cálculos, convertemos de volta e escrevemos na saída.

Esqueleto do Kernel #

Vamos percorrer o kernel mais simples que existe e algumas variações dele

__kernel void myKernel() {
}

Um kernel que pode ser iniciado a partir do seu código principal é identificado pela palavra-chave __kernel. Uma função Kernel só pode ter o tipo de retorno void.

__kernel void myKernel(float a, uint b, byte c) {

}

Claro que você pode criar mais funções que não são expostas como kernels. Nesse caso, você pode simplesmente omitir o modificador __kernel.

Uma função pode expor variáveis ​​como qualquer outra função C/C++ faria. A única diferença é quando você deseja fazer referência à memória. Isso se aplica a todos os ponteiros, sejam eles argumentos ou usados ​​no código.

float*  ptr;

é um ponteiro para uma região de memória à qual somente o thread em execução tem acesso. Na verdade é o mesmo que

__private float* ptr;

Existem quatro modificadores de região de memória diferentes disponíveis. Dentro do kernel você normalmente não precisa se preocupar com isso, mas para argumentos isso é essencial.

  • __global: Este modificador refere-se a um ponteiro que é colocado na memória global
  • __constant: refere-se a um ponteiro de memória constante
  • __local: refere-se a um ponteiro de memória compartilhada
  • __private: refere-se a um ponteiro de memória local

Além disso, podemos definir como queremos acessar a memória

  • sem modificador: ler e escrever
  • __somente leitura
  • __write_only

Esses sinalizadores precisam corresponder à maneira como alocamos o buffer de memória em nosso host.

ID do kernel #

Para trabalhar adequadamente com os dados, cada encadeamento precisa saber sua posição no bloco de encadeamentos/conjunto de encadeamentos global. Isso pode ser arquivado com

get_local_id($dim);
get_global_id($dim);

Essas duas funções retornam a posição do thread em relação ao threadblock ou a todos os threads.

get_working_dim();

Obtém o número total de dimensões com as quais o kernel foi iniciado.

get_local_size($dim);
get_global_size($dim);

Obtém o número total de threads no bloco de threads ou no total para uma determinada dimensão.

Advertência: certifique-se sempre de que seu encadeamento não está excedendo o tamanho dos dados. Isso é muito provável de acontecer e deve sempre ser verificado.

Vetores em OpenCL #

Cada tipo opencl fundamental tem uma versão vetorial. Você pode usar o tipo vetorial anexando o número de componentes desejados após o tipo. O número de componentes suportados é 2,3,4,8 e 16. O OpenCL 1.0 não oferece três componentes.

Você pode inicializar qualquer vetor usando duas maneiras:

  • Fornecer um único escalar

  • Satisfazer todos os componentes

    float4 a = (float4)(1); //a = (1, 1, 1, 1) ou

    float4 b = (float4)(1, 2, 3, 4); float4 c = (float4)(1, (float3)(2));

ou qualquer outra combinação de vetores que satisfaça o número de componentes. Para acessar os elementos de um vetor, você pode usar diferentes métodos. Você pode usar a indexação:

a[0] = 2;

ou use literais. A vantagem dos literais é que você pode combiná-los da maneira que quiser, faça isso em um momento. Você pode acessar todos os componentes vetoriais com

a.s0 = 2; // same as a[0] = 2

você também pode combinar vários componentes em um novo vetor

a.s02 = (float2)(0, 0); // same as  a[0] = 0; a[2] = 0; or even a.s20 = (float2)(0, 0)

você pode alterar a ordem dos componentes da maneira que desejar.

a.s1423 = a.s4132; // flip the vector

mas você não pode fazer algo como

a.s11 = ... // twice the same component is not possible

Existem alguns atalhos convenientes para acessar componentes vetoriais. As seguintes abreviações aplicam-se apenas aos tamanhos 2, 4, 8 e 16

a.hi //=a.s23 for vectors of size 4, a.4567 for size 8 and so on.
a.lo //=a.s01
a.even //=a.s02
a.odd //=a.13

Para tamanhos de vetor 2,3 e 4, existem algumas abreviações adicionais

a.x //=a.s0
a.y //=a.s1
a.z //=a.s2
a.w //=a.s3

Kernel de correção de gama #

Vamos olhar para um kernel de correção de gama

__constant sampler_t sampler = CLK_NORMALIZED_COORDS_FALSE |
                                CLK_ADDRESS_CLAMP_TO_EDGE |
                                CLK_FILTER_LINEAR;

__kernel void Gamma(__read_only image2d_t input, __write_only image2d_t output, __constant float gamma) {
    int2 gid = (int2)(get_global_id(0), get_global_id(1));
    int2 size = get_image_dim(input);

    if(all(gid < size)){
        uint4 pixel = read_imageui(input, sampler, gid);
        float4 color = convert_float4(pixel) / 255;
        color = pow(color, (float4)(gamma));
        pixel = convert_uint4_rte(color * 255);
        write_imageui(output, gid, pixel);
    }
}

Agora vamos percorrer esse código passo a passo. A primeira linha cria uma variável na região de memória __constant do tipo sampler_t. Este amostrador é usado para especificar ainda mais o acesso aos nossos dados de imagem. Consulte o Khronos Docs para obter uma documentação completa.

Nós alocamos a entrada como read_only e a saída como write_only antes de chamarmos nosso kernel, então adicionamos esses modificadores aqui.

image2d e image3d são sempre alocados na memória global, portanto, podemos omitir o modificador __global aqui. Nosso valor gama está localizado na memória __constant, então especificamos isso também.

Em seguida, obtemos nosso id de thread que define o pixel que vamos corrigir com a gama. Também consultamos o tamanho para garantir que nosso encadeamento não esteja acessando memória não alocada. Isso definitivamente travará seu kernel se você esquecer isso.

Depois de nos certificarmos de que somos um thread legítimo, lemos nosso pixel de nossa imagem de entrada. Em seguida, convertemos para float para evitar perda de casas decimais, fazemos alguns cálculos, convertemos de volta e escrevemos na saída.
