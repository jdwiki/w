Instalação da fonte (com Postgres 9.1 ou superior) #

Este guia é explicitamente para PostgreSQL 9.1 ou superior em máquinas Linux. Ele usa o recurso de extensões do postgres que melhorará muito a importação de extensões para uma instalação do postgres existente. Se você tiver que trabalhar com uma versão mais antiga do postgres, consulte as documentações oficiais.

Resolver dependências

PostGIS é um projeto complexo que possui várias dependências. Para prosseguir com a configuração manual e o procedimento de compilação, você terá que resolver essas dependências e instalar os seguintes pacotes manualmente ou por meio dos gerenciadores de pacotes.

Requerimentos mínimos

  • PostgreSQL 9.1 ou superior. É importante que você instale o banco de dados incluindo os cabeçalhos do servidor, normalmente encontrados nos pacotes dev do gerenciador de pacotes do seu repositório.
  • O compilador GNU C gcc.
  • GNU fazer. Para concluir o processo de construção.
  • Proj4. Uma biblioteca de projeção, para transformações de coordenadas.
  • GEOS. Uma biblioteca de geometria, que implementa descrições de recursos e geometrias simples. A versão 3.5 ou superior é recomendada para usar funções mais recentes, como ST_ClipByBox2D e ST_Subdivide.
  • GDAL, versão 1.9 ou superior. Uma biblioteca que implementa formatos de dados abstratos para dados geoespaciais baseados em raster e vetor.
  • [LibXML2][8], versão 2.5 ou superior. Um libraray para trabalhar com XML, XSLT e DTD.
  • [JSON-C][9], versão 0.9 ou superior. Uma biblioteca para criar saída no formato JSON

Requisitos Opcionais

  • [GTK][10] (requer GTK+2.0, 2.8+) para compilar o shp2pgsql-gui.
  • [SFCGAL][11], versão 1.1 (ou superior) pode ser usado para fornecer funções de análise avançada 2D e 3D adicionais ao PostGIS.
  • [PCRE][12]. Uma biblioteca para correspondência de padrões de expressão regular usando a sintaxe Perl 5. Esta biblioteca é necessária se você quiser trabalhar com o [Padronizador de Endereços][13].
  • [CUnit][14]. Um utilitário de teste de unidade, necessário para testes de regressão.
  • [DocBook][15] (xsltproc) é necessário para construir a documentação.
  • [DBLatex][16] é necessário para construir a documentação em formato PDF.
  • [ImageMagick][17] é necessário para gerar as imagens usadas na documentação.

Obtenha as Fontes

Para obter o código-fonte, baixe o tarball mais recente:

wget http://postgis.net/stuff/postgis-2.3.2dev.tar.gz
tar -xvzf postgis-2.3.2dev.tar.gz

ou use o repositório SVN oficial:

svn checkout http://svn.osgeo.org/postgis/trunk/ postgis-2.3.2dev

Configuração

Se você obteve as fontes via SVN, você pode preparar o script de configuração com:

./autogen.sh

Para configurar o processo de compilação para sua máquina específica, execute na pasta do projeto:

./configure

Existem vários parâmetros opcionais para a etapa de configuração. Consulte a [documentação oficial][18] para obter instruções detalhadas, isso geralmente é opcional e apenas para servidores que usam instalações não padrão.

Construir

Assim que a etapa de configuração for concluída com sucesso, um makefile será criado. Para iniciar o processo de compilação, execute:

make

A última saída deve ser:

"PostGIS was built successfully. Ready to install."

A partir da versão 1.4.0, todas as funções possuem comentários gerados a partir da documentação. Se você deseja instalar esses comentários em seus bancos de dados espaciais posteriormente, execute o comando que requer docbook.

make comments

Instalação

Instale todas as extensões com:

make install

As extensões do PostGIS são criadas e instaladas automaticamente se você estiver usando o PostgreSQL 9.1 ou superior. Você pode instalar as extensões necessárias manualmente, se tiver uma configuração diferente.

Na pasta do projeto:

cd extensions
cd postgis
make clean
make
make install

cd ..
cd postgis_topology
make clean
make
make install

cd ..
cd postgis_sfcgal
make clean
make
make install

cd ..
cd address_standardizer
make clean
make
make install
make installcheck

cd ..
cd postgis_tiger_geocoder
make clean
make
make install
make installcheck

Se você quiser instalar as extensões manualmente em uma máquina diferente, copie os seguintes arquivos da pasta extensions para a pasta PostgreSQL/share/extension do destino. Para cada extensão:

scp extensions/[EXTENSION]/sql/*.sql user@target:[POSTGIS_PATH]/share/extension

onde [EXTENSION] é a extensão selecionada (postgis, postgis_topology, postgis_sfcgal, address_standardizer, postgis_tiger_geocoder) e [POSTGIS_PATH] é o caminho de instalação do PostGIS em sua máquina de destino.

Verificando a instalação

Se você não tiver um serviço de banco de dados postgres em execução, [configure seu banco de dados postgres primeiro][19]. Conecte-se ao banco de dados, usando:

su postgres -c psql

Para verificar se as extensões estão acessíveis, execute as seguintes consultas em uma sessão psql:

SELECT name, default_version,installed_version FROM pg_available_extensions WHERE name LIKE 'postgis%' or name LIKE 'address%';

A saída deve ficar assim:

             name             | default_version | installed_version
------------------------------+-----------------+-------------------
 address_standardizer         | 2.3.2dev        | 2.3.2dev
 address_standardizer_data_us | 2.3.2dev        | 2.3.2dev
 postgis                      | 2.3.2dev        | 2.3.2dev
 postgis_sfcgal               | 2.3.2dev        |
 postgis_tiger_geocoder       | 2.3.2dev        | 2.3.2dev
 postgis_topology             | 2.3.2dev        |

(6 rows)

Para realizar um teste de pós-instalação aprofundado, execute o seguinte comando na pasta do projeto:

make check

Isso será executado por meio de várias verificações e testes usando a biblioteca gerada em um banco de dados PostgreSQL real.

Um “Hello World” geoespacial #

Neste exemplo vamos configurar um banco de dados geoespacial, importar dados de 2 fontes diferentes e visualizar os resultados em um aplicativo chamado QGIS. Este guia foi escrito explicitamente para máquinas linux, se você operar em outra plataforma, alguns comandos ou caminhos podem não funcionar como esperado.

Para visualizar os dados importados, usaremos um aplicativo chamado QGIS. Se você não tem este aplicativo, por favor, instale-o primeiro, se você gosta de usar outro visualizador ou aplicativo geográfico (como ArcGIS), você pode pular a instalação do QGIS.

Nossas fontes serão os Distritos da Assembleia Estadual da Cidade de Nova York e o Banco de Dados de Mapas de Ruas LION da Cidade de Nova York. Faça o download dos arquivos apropriados dos locais vinculados. Você também deve dar uma olhada na seção Metadados do material, pois ela fornece informações sobre qual sistema de referência de coordenadas esses arquivos usam.

Para começar, crie uma pasta de trabalho “nycgis”, copie os arquivos baixados para o local e descompacte os arquivos.

mkdir nycgis
cd nycgis
cp ~/Downloads/nyad_16d.zip .
unzip ~/Downloads/nyad_16d.zip
cp ~/Downloads/nylion_16d.zip .
unzip  ~/Downloads/nylion_16d.zip

Na pasta “nycgis” agora você deve ter 2 pastas: “nyad_16d”, “lion” com vários arquivos.

Ao trabalhar com dados geográficos, é de vital importância conhecer o sistema de referência de coordenadas (CRS) de seus dados de origem e de seus dados de saída finais. Nas seções de metadados dos locais vinculados acima (Metadados: Assembly Districts, Metadata: LION Database), você descobrirá que o CRS para ambos os arquivos é EPSG:2263, um sistema de coordenadas usado para fazer referência ao nordeste dos EUA.

Vamos supor que queremos usar um CRS diferente em nosso banco de dados. Isso pode ter diferentes razões, podemos querer trabalhar com um aplicativo geográfico baseado na web no banco de dados, por exemplo. Um SRC comum para este tipo de aplicação é o WGS:84 (EPSG:4326).

Para converter os sistemas de coordenadas usamos uma ferramenta chamada ogr2ogr que faz parte do pacote GDAL. Na pasta de trabalho, primeiro criamos 2 pastas representando os dados reprojetados e depois convertemos nossos dados.

mkdir nyad_16d_proj_4326
ogr2ogr -f "ESRI Shapefile" ./nyad_16d_proj_4326/nyad_4326.shp ./nyad_16d/nyad_16d.shp -s_srs EPSG:2263 -t_srs EPSG:4326

mkdir nylion_16d_proj_4326
ogr2ogr -f "ESRI Shapefile" ./nylion_16d_proj_4326/ny_str_4326.shp ./nylion_16d/lion/lion.gdb/a0000000d.gdbtable -s_srs EPSG:2263 -t_srs EPSG:4326

Observe que usamos apenas o arquivo chamado: “a0000000d.gdbtable” do banco de dados LION para nossos propósitos. A sintaxe do comando ogr2ogr é a seguinte:

ogr2ogr -f [output-format] [output-file] [input-file] -s_srs [source crs] -t_srs [target crs]

Agora temos 2 shapefiles, projetados no CRS correto. Para usar os dados em nosso banco de dados, devemos converter os shapefiles para sql-statemens. Para isso usamos uma ferramenta chamada shp2pgsql. No diretório de trabalho execute os seguintes comandos:

shp2pgsql ./nyad_16d_proj_4326/nyad_4326.shp > nyad_4326.sql
shp2pgsql ./nylion_16d_proj_4326/ny_str_4326.shp > ny_streets_4326.sql

Os arquivos nyad_4326.sql e ny_streets_4326.sql estão agora prontos para uso no postgis. Para prosseguir e importar os dados, crie um banco de dados habilitado espacialmente.

sudo su - postgres
createdb nycgis
psql -d nycgis

Na sessão psql, execute:

CREATE EXTENSION postgis;
CREATE EXTENSION postgis_topology;

Saia das sessões de usuário psql e postgres com (CTRL+D). E importe os arquivos com:

psql -f nyad_4326.sql -d nycgis
psql -f ny_streets_4326.sql -d nycgis

O banco de dados nycgis possui agora 2 tabelas nas quais as fontes reprojetadas foram importadas com sucesso.

Para verificar isso: abra o QGIS

  1. use Camada > Adicionar Camada > PostGIS-Layer
  2. conecte-se ao seu banco de dados
  3. selecione suas mesas
  4. (opcional) defina o estilo das camadas recém-criadas.

Et voilà: Agora você tem um banco de dados habilitado espacialmente, com geodados importados e reprojetados.

Instalação via Gerenciador de Pacotes #

Arco:

Um pacote pacman oficial está disponível. Instale o pacote como root, usando:

pacman -S postgis

OpenSuse:

Para usar os repositórios do openSuse para aplicativos geoespaciais, habilite o Geo-repository como root:

zypper ar http://download.opensuse.org/repositories/Application:/Geo/openSUSE_[RELEASE]/ GEO

onde [RELEASE] é o número oficial de lançamento da sua distribuição Suse. Depois disso, você pode instalar o postgis com:

zypper install postgis

Configurando um banco de dados geoespacial #

Para criar um novo banco de dados vazio, execute como postgres-user:

createdb [yourdatabase]

Conecte-se ao banco de dados com uma sessão psql:

psql -d [yourdatabase]

Na sessão psql execute:

CREATE EXTENSION postgis;
CREATE EXTENSION postgis_topology;

para criar as extensões geoespaciais necessárias. Feito isso, o banco de dados é um banco de dados habilitado geoespacialmente e está pronto para uso.
