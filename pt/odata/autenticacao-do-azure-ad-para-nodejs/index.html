1.5. Visão geral

Este documento fornece uma visão geral técnica dos seguintes componentes. • Módulo Node.JS • API da Web personalizada • Biblioteca de EDUCAÇÃO • Azure Active Directory • Documento DB 2. Representação Arquitetônica (Visão Lógica) A solução é dividida em dois componentes. um é o aplicativo do lado do cliente que seria executado no componente móvel e do lado do servidor em execução no ambiente do Azure.

Aplicativo para dispositivos móveis (cliente Node.js) Node.js is light weight responsive user interface. A mobile application would simply be the, written in the native language or using a hybrid solution like Cordova (aka PhoneGap). This user interface would get data and perform business logic via calls to a server side API. Se o aplicativo móvel exigir partes da lógica de negócios, ele deverá ser reescrito em Objective-C para iOS e Java para Android. Embora o Cordova ofereça uma solução para alguma reutilização de código, ele não pode aproveitar o Node ou seu ecossistema diretamente.

Componente do lado do servidor Servidor Node.js Node é um runtime de servidor web JavaScript. Este atua como servidor Web para aplicativo móvel. Ele usa JSON como formato de troca de dados, JSON, pode ser analisado nativamente por ambas as extremidades. O Node difere da maioria dos tempos de execução de aplicativos da Web na maneira como lida com a simultaneidade. Em vez de usar threading para realizar a simultaneidade, o Node depende de um loop orientado a eventos executado em um único processo. Node suporta um modelo assíncrono (sem bloqueio). API da Web Esta é a solução de API REST em execução no Azure como API Web. Está sendo executado em cima do Document DB e atua como camada protetora para o Document DB. Ferramentas ASP.NET e middleware de segurança dos componentes Microsoft Open Web Interface for .NET (OWIN) que simplificam a proteção da API da Web. O ponto de extremidade é protegido usando tokens de acesso do Azure AD OAuth Bearer 2.0.   3. Processo Técnico

3.1. Autenticação A autenticação se resume a pedir ao chamador, ao enviar uma mensagem para um servidor, que inclua algum tipo de credencial que possa verificar sua identidade ou recuperar seus atributos. O servidor então usa essas informações para autorização – determinando se o acesso deve ser concedido e em que termos. Os recursos geralmente transferem a maioria das funções de autenticação para um provedor de serviços externo, comumente conhecido como autoridade ou provedor de identidade. Esses provedores cuidam de tarefas pesadas, como integração de usuários, atribuição de credenciais, tratamento de fluxos de ciclo de vida (como recuperação de senha), fornecimento de uma interface do usuário para autenticação de usuários, verificação de credenciais em vários protocolos, gerenciamento de vários fatores de autenticação, detecção de fraudes e muito mais. Com essas funções fora do caminho, a única tarefa de autenticação restante é verificar se a autenticação foi bem-sucedida na autoridade escolhida. Isso normalmente envolve examinar um token de segurança, um fragmento de dados emitido por uma autoridade para um chamador após uma autenticação bem-sucedida. Os tokens de segurança geralmente são criados de acordo com um formato específico, assinado digitalmente por uma chave que identificará inequivocamente a autoridade emissora e contém alguns dados que vinculam exclusivamente o token ao recurso de destino. Quando o recurso recebe uma solicitação, ele procura um token de acompanhamento. Se encontrar um que atenda aos atributos de validação necessários, o chamador será autenticado. Nesse nível de detalhe, o padrão é tão genérico que pode descrever muitas abordagens diferentes de autenticação. Vou aplicá-lo a este cenário atribuindo as funções de alto nível a entidades concretas. O Recurso O recurso será o projeto ASP.NET Web API 2 que preciso proteger. Você pode aplicar requisitos de autenticação com granularidade mais fina.

3.2. API da Web personalizada e ponto final seguro A Autoridade configurará a API Web para descarregar suas necessidades de autenticação para o Windows Azure AD, uma Plataforma como Serviço (PaaS) disponível para todos os assinantes do Windows Azure. O Windows Azure AD foi projetado especificamente para dar suporte a cargas de trabalho de aplicativos baseados em nuvem. O serviço armazena informações sobre usuários (atributos e credenciais) e estrutura organizacional. Ele pode sincronizar dados com o Windows Server Active Directory, desde que conecte exclusivamente na nuvem. 3.3. Formato do token OAuth O processo de validação OAuth 2.0 não exige nenhum formato de token específico, mas o formato JSON Web Token (JWT) (bit.ly/14EhlE8) para cenários REST tornou-se o padrão de fato. Os componentes Windows Azure AD e Microsoft OWIN oferecem suporte ao formato JWT em fluxos OAuth 2.0. A mecânica de aquisição e validação de JWT é toda cuidada pelo middleware, e o formato do token é transparente para o código do aplicativo. 3.4. Módulo Node.js com Biblioteca ADAL Os módulos Node.js são uma interface móvel que contém componentes de interface do usuário e módulo de autenticação para obter o token de segurança do Active Directory usando a biblioteca ADAL. A biblioteca ADAL para node.js facilita a autenticação de aplicativos node.js no Azure AD para acessar recursos da Web protegidos do Azure AD.

4.1. Segurança

Para poder autenticar usuários, você deve registrar o aplicativo Node.js no Azure Active Directory (AAD). Isso é feito em duas etapas. Primeiro, você deve registrar seu serviço móvel e expor permissões nele. Segundo, você deve registrar seu aplicativo iOS e conceder acesso a essas permissões 4.2. Persistência

A persistência de dados será abordada usando o banco de dados Azure Document DB.

4.3. Confiabilidade/Disponibilidade

4.4. atuação

Não há restrições específicas relacionadas ao desempenho do sistema. Prevê-se que o sistema deve responder a qualquer solicitação bem sob os tempos limite padrão do banco de dados e do script do servidor da Web (20 segundos), além disso, o desempenho do sistema pode depender do hardware disponível, da rede PSU e dos recursos de conexão com a Internet. Além disso, os tempos de upload/download podem depender do tamanho dos dados, que por sua vez depende da entrada do usuário. Portanto, o desempenho real pode ser determinado somente após a implantação e teste do sistema.
