Aplicativo Laravel 5 em Hospedagem Compartilhada em Servidor Linux #

Por padrão, a pasta public do projeto Laravel expõe o conteúdo do aplicativo que pode ser solicitado de qualquer lugar por qualquer pessoa, o restante do código do aplicativo é invisível ou inacessível para qualquer pessoa sem as devidas permissões.

Depois de desenvolver o aplicativo em sua máquina de desenvolvimento, ele precisa ser enviado para um servidor de produção para que possa ser acessado pela Internet de qualquer lugar - certo?

Para a maioria dos aplicativos/sites, a primeira escolha é usar o pacote de hospedagem compartilhada de provedores de serviços de hospedagem como GoDaddy, HostGator etc. principalmente devido ao baixo custo.

nota: você pode pedir ao seu provedor para alterar manualmente document_root, então tudo o que você precisa fazer é enviar seu aplicativo Laravel para o servidor (via FTP), solicitar a alteração do root para {app}/ public e você deve ser bom.

Esses pacotes de hospedagem compartilhada, no entanto, têm limitações em termos de acesso ao terminal e permissões de arquivo. Por padrão, deve-se enviar seu aplicativo/código para a pasta public_html em sua conta de hospedagem compartilhada.

Então, se você quiser fazer upload de um projeto Laravel para uma conta de hospedagem compartilhada, como você faria? Você deve enviar o aplicativo inteiro (pasta) para a pasta public_html em sua conta de hospedagem compartilhada? - Certamente não

Porque tudo na pasta public_html é acessível “publicamente, ou seja, por qualquer pessoa”, o que seria um grande risco de segurança.

Etapas para fazer upload de um projeto para uma conta de hospedagem compartilhada - o caminho Laravel

Passo 1 Crie uma pasta chamada laravel (ou qualquer coisa que você goste) no mesmo nível da pasta public_html.

Eg:  
/
|--var  
    |---www
        |----laravel       //create this folder in your shared hosting account
        |----public_html  
        |----log

Passo 2 Copie tudo, exceto a pasta public do seu projeto laravel (na máquina de desenvolvimento) na pasta laravel (no host do servidor - conta de hospedagem compartilhada). Você pode usar:

  • C-panel: qual seria a opção mais lenta
  • Cliente FTP: como FileZilla para se conectar à sua conta de hospedagem compartilhada e transferir seus arquivos e pastas através do upload por FTP
  • Mapear unidade de rede: você também pode criar uma unidade de rede mapeada em sua máquina de desenvolvimento para se conectar à pasta raiz da sua conta de hospedagem compartilhada usando “ftp://nome-do-domínio” como endereço de rede.

Etapa 3 Abra a pasta public do seu projeto laravel (na máquina de desenvolvimento), copie tudo e cole na pasta public_html (no host do servidor - conta de hospedagem compartilhada). Passo 4 Agora abra o arquivo index.php na pasta public_html na conta de hospedagem compartilhada (no editor cpanel ou qualquer outro editor conectado) e:

Mudar:

require __DIR__.'/../bootstrap/autoload.php';

Para:

require __DIR__.'/../laravel/bootstrap/autoload.php';

E Mudança:

$app = require_once __DIR__.'/../bootstrap/app.php';

Para:

$app = require_once __DIR__.'/../laravel/bootstrap/app.php';

Salvar e fechar.

Passo 5 Agora vá para a pasta laravel (na conta de hospedagem compartilhada -server) e abra o arquivo server.php Mudar

require_once __DIR__.'/public/index.php';

Para:

require_once __DIR__.'../public_html/index.php';

Salvar e fechar.

Passo 6 Defina as permissões de arquivo para a pasta laravel/storage (recursivamente) e todos os arquivos, subpastas e arquivos dentro deles na conta de hospedagem compartilhada - servidor para 777. Nota: Cuidado com as permissões de arquivos no linux, elas são como uma faca de dois gumes, se não forem usadas corretamente, podem tornar seu aplicativo vulnerável a ataques. Para entender as permissões de arquivo do Linux, você pode ler https://www.linux.com/learn/tutorials/309527-understanding-linux-file-permissions

Passo 7

Como o arquivo .env do servidor local/de desenvolvimento é ignorado pelo git e deve ser ignorado, pois possui todas as variáveis ​​de ambiente, incluindo o APP_KEY, e não deve ser exposto ao público empurrando-o para os repositórios’. Você também pode ver que o arquivo .gitignore tem .env mencionado, portanto, ele não será enviado para os repositórios.

Após seguir todos os passos acima crie um arquivo .env na pasta laravel e adicione todas as variáveis ​​de ambiente que você usou do arquivo .env do servidor local/de desenvolvimento para o arquivo .env do servidor de produção.

Mesmo existem arquivos de configuração como app.php, database.php na pasta de configuração do aplicativo laravel que define essas variáveis ​​como por padrão no segundo parâmetro de env() mas não codifique os valores nestes arquivos, pois isso afetará os arquivos de configuração dos usuários que puxam seu repositório. Portanto, é recomendado criar o arquivo .env manualmente!

O laravel também fornece o arquivo .env-example que você pode usar como referência.

É isso.

Agora, quando você visita o URL que você configurou como o domínio com seu servidor, seu aplicativo laravel deve funcionar exatamente como funcionou em seu localhost - máquina de desenvolvimento, enquanto o código do aplicativo é seguro e não pode ser acessado por ninguém sem permissões de arquivo adequadas.
