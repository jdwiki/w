Roteamento Básico #

O roteamento define um mapa entre métodos HTTP e URIs de um lado e ações do outro. As rotas são normalmente escritas no arquivo app/Http/routes.php.

Em sua forma mais simples, uma rota é definida chamando o método HTTP correspondente na fachada Route, passando como parâmetros uma string que corresponde ao URI (relativo à raiz do aplicativo) e um retorno de chamada.

Por exemplo: uma rota para o URI raiz do site que retorna uma visualização home se parece com isso:

Route::get('/', function() {
    return view('home');
});

Uma rota para uma solicitação de postagem que simplesmente ecoa as variáveis ​​de postagem:

Route::post('submit', function() {
   return Input::all();
});

//or

Route::post('submit', function(\Illuminate\Http\Request $request) {
   return $request->all();
});

Rotas apontando para um método Controller #

Em vez de definir o retorno de chamada embutido, a rota pode se referir a um método do controlador na sintaxe [NomeClasseDoControlador@Método]:

Route::get('login', 'LoginController@index');

Uma rota para vários verbos #

O método match pode ser usado para corresponder a uma matriz de métodos HTTP para uma determinada rota:

Route::match(['GET', 'POST'], '/', 'LoginController@index');

Além disso, você pode usar all para corresponder a qualquer método HTTP para uma determinada rota:

Route::all('login', 'LoginController@index');

Grupos de rotas #

As rotas podem ser agrupadas para evitar repetição de código.

Digamos que todos os URIs com o prefixo /admin usem um certo middleware chamado admin e todos eles vivam no namespace App\Http\Controllers\Admin.

Uma maneira limpa de representar isso usando Grupos de Rotas é a seguinte:

Route::group([
    'namespace' => 'Admin',
    'middleware' => 'admin',
    'prefix' => 'admin'
    ], function () {
        
        // something.dev/admin
        // 'App\Http\Controllers\Admin\IndexController'
        // Uses admin middleware
        Route::get('/', ['uses' => 'IndexController@index']);    
        
        // something.dev/admin/logs
        // 'App\Http\Controllers\Admin\LogsController'
        // Uses admin middleware
        Route::get('/logs', ['uses' => 'LogsController@index']);    
});

As rotas são correspondidas na ordem em que são declaradas #

Esta é uma pegadinha comum com as rotas do Laravel. As rotas são correspondidas na ordem em que são declaradas. A primeira rota correspondente é a que é usada.

Este exemplo funcionará como esperado:

Route::get('/posts/{postId}/comments/{commentId}', 'CommentController@show');
Route::get('/posts/{postId}', 'PostController@show');

Uma solicitação get para /posts/1/comments/1 invocará CommentController@show. Uma solicitação get para /posts/1 invocará PostController@show.

No entanto, este exemplo não funcionará da mesma maneira:

Route::get('/posts/{postId}', 'PostController@show');
Route::get('/posts/{postId}/comments/{commentId}', 'CommentController@show');

Uma solicitação get para /posts/1/comments/1 invocará PostController@show. Uma solicitação get para /posts/1 invocará PostController@show.

Como o Laravel usa a primeira rota correspondente, a solicitação para /posts/1/comments/1 corresponde a Route::get('/posts/{postId}', 'PostController@show'); e atribui a variável $postId para o valor 1/comments/1. Isso significa que CommentController@show nunca será invocado.

Rota nomeada #

As rotas nomeadas são usadas para gerar uma URL ou redirecionar para uma rota específica. A vantagem de usar uma rota nomeada é que, se alterarmos o URI de uma rota no futuro, não precisaremos alterar a URL ou redirecionamentos apontando para essa rota se estivermos usando uma rota nomeada. Mas se os links foram gerados usando a url [ ex. url('/admin/login')], então teríamos que mudar em todos os lugares onde ele é usado.

As rotas nomeadas são criadas usando a chave de matriz as

Route::get('login', ['as' => 'loginPage', 'uses' => 'LoginController@index']);

ou usando o método name

Route::get('login', 'LoginController@index')->name('loginPage');

Gera URL usando rota nomeada #

Para gerar um URL usando o nome da rota

$url = route('loginPage');

Se você estiver usando o nome da rota para redirecionamento

$redirect = Redirect::route('loginPage');

Parâmetros de rota #

Você pode usar parâmetros de rota para obter a parte do segmento URI. Você pode definir parâmetros de rota opcionais ou obrigatórios ao criar uma rota. Parâmetros opcionais têm um ? anexado ao final do nome do parâmetro. Este nome está entre chaves {}

Parâmetro opcional #

Route::get('profile/{id?}', ['as' => 'viewProfile', 'uses' => 'ProfileController@view']);

Esta rota pode ser acessada por domain.com/profile/23 onde 23 é o parâmetro id. Neste exemplo o id é passado como parâmetro no método view do ProfileController. Como este é um parâmetro opcional, o acesso a domain.com/profile funciona muito bem.

Parâmetro obrigatório #

Route::get('profile/{id}', ['as' => 'viewProfile', 'uses' => 'ProfileController@view']);

Observe que o nome do parâmetro obrigatório não tem um ? no final do nome do parâmetro.

Acessando o parâmetro no controller #

Em seu controlador, seu método de visualização recebe um parâmetro com o mesmo nome do routes.php e pode ser usado como uma variável normal. O Laravel se encarrega de injetar o valor:

public function view($id){
    echo $id;
}

Pegue todas as rotas #

Se você quiser pegar todas as rotas, você pode usar uma expressão regular como mostrado:

Route::any('{catchall}', 'CatchAllController@handle')->where('catchall', '.*');

Importante: Se você tiver outras rotas e não quiser que o catch-all interfira, coloque-o no final. Por exemplo:

Capturando todas as rotas exceto as já definidas #

Route::get('login',  'AuthController@login');
Route::get('logout', 'AuthController@logout');
Route::get('home',   'HomeController@home');

// The catch-all will match anything except the previous defined routes.
Route::any('{catchall}', 'CatchAllController@handle')->where('catchall', '.*');

Rotas que não diferenciam maiúsculas de minúsculas #

As rotas no Laravel diferenciam maiúsculas de minúsculas. Isso significa que uma rota como

Route::get('login', ...);

corresponderá a uma solicitação GET a /login, mas não a uma solicitação GET a /Login.

Para tornar suas rotas insensíveis a maiúsculas e minúsculas, você precisa criar uma nova classe validadora que corresponda às URLs solicitadas em relação às rotas definidas. A única diferença entre o novo validador e o existente é que ele anexará o modificador i no final da expressão regular para a rota compilada para habilitar a correspondência que não diferencia maiúsculas de minúsculas.

<?php namespace Some\Namespace;

use Illuminate\Http\Request;
use Illuminate\Routing\Route;
use Illuminate\Routing\Matching\ValidatorInterface;

class CaseInsensitiveUriValidator implements ValidatorInterface
{
  public function matches(Route $route, Request $request)
  {
    $path = $request->path() == '/' ? '/' : '/'.$request->path();
    return preg_match(preg_replace('/$/','i', $route->getCompiled()->getRegex()), rawurldecode($path));
  }
}

Para que o Laravel use seu novo validador, você precisa atualizar a lista de matchers que são usados ​​para corresponder a URL a uma rota e substituir o UriValidator original pelo seu.

Para fazer isso, adicione o seguinte no topo do seu arquivo routes.php:

<?php
use Illuminate\Routing\Route as IlluminateRoute;
use Your\Namespace\CaseInsensitiveUriValidator;
use Illuminate\Routing\Matching\UriValidator;

$validators = IlluminateRoute::getValidators();
$validators[] = new CaseInsensitiveUriValidator;
IlluminateRoute::$validators = array_filter($validators, function($validator) { 
  return get_class($validator) != UriValidator::class;
});

Isso removerá o validador original e adicionará o seu à lista de validadores.
