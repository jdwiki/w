Parâmetros #

Parâmetro Detalhes
necessário O campo é obrigatório
às vezes Execute verificações de validação em um campo somente se esse campo estiver presente na matriz de entrada
e-mail A entrada é um email válido
max:valor O valor de entrada deve estar abaixo do valor máximo
exclusivo:db_table_name O valor de entrada deve ser exclusivo no nome da tabela de banco de dados fornecida
aceito Sim / Ligado / 1 verdadeiro, útil para verificar TOS
active_url Deve ser um URL válido de acordo com checkdnsrr
depois de :data O campo em validação deve fornecer um valor após a data indicada
alfa O campo sob validação deve ser inteiramente de caracteres alfabéticos.
alpha_dash O campo em validação pode ter caracteres alfanuméricos, bem como travessões e sublinhados.
alfa_num O campo sob validação deve ser inteiramente de caracteres alfanuméricos.
matriz Deve ser um PHP array
antes de :data O campo deve ser um valor na data especificada
entre:min,max O valor de entrada deve estar entre o valor mínimo (min) e máximo (max)
booleano O campo em validação deve poder ser convertido como um booleano. As entradas aceitas são true, false, 1, 0, "1" e "0".
confirmado O campo em validação deve ter um campo correspondente de foo_confirmation. Por exemplo, se o campo sob validação for password, um campo correspondente password_confirmation deve estar presente na entrada.
data O campo em validação deve ser uma data válida de acordo com a função strtotime PHP.
inteiro O campo em validação deve ser um integer
seqüência O campo em validação deve ser do tipo string.

Exemplo básico #

Você pode validar os dados da solicitação usando o método validate (disponível no Controller base, fornecido pelo atributo ValidatesRequests).

Se as regras forem aprovadas, seu código continuará executando normalmente; no entanto, se a validação falhar, uma resposta de erro contendo os erros de validação será automaticamente enviada de volta:

  • para solicitações típicas de formulários HTML, o usuário será redirecionado para a página anterior, com o formulário mantendo os valores enviados
  • para solicitações que esperam uma resposta JSON, uma resposta HTTP com o código 422 será gerada

Por exemplo, em seu UserController, você pode estar salvando um novo usuário no método store, que precisaria de validação antes de salvar.

/**
 * @param  Request  $request
 * @return Response
 */
public function store(Request $request) {
    $this->validate($request, [
        'name' => 'required',
        'email' => 'email|unique:users|max:255'
    ],
    // second array of validation messages can be passed here
    [
        'name.required' => 'Please provide a valid name!',
        'email.required' => 'Please provide a valid email!',
    ]);

    // The validation passed
}

No exemplo acima, validamos que o campo name existe com valor não vazio. Em segundo lugar, verificamos se o campo email possui um formato de e-mail válido, é único na tabela do banco de dados “usuários” e possui comprimento máximo de 255 caracteres.

O caractere | (pipe) combina diferentes regras de validação para um campo.

Às vezes, você pode querer parar de executar as regras de validação em um atributo após a primeira falha de validação. Para fazer isso, atribua a regra bail ao atributo:

$this->validate($request, [
    'name' => 'bail|required',
    'email' => 'email|unique:users|max:255'
]);

A lista completa de regras de validação disponíveis pode ser encontrada na seção de parâmetros abaixo.

Outras abordagens de validação #

1) Validação de solicitação de formulário

Você pode criar uma “solicitação de formulário” que pode conter a lógica de autorização, regras de validação e mensagens de erro para uma solicitação específica em seu aplicativo.

O comando make:request Artisan CLI gera a classe e a coloca no diretório app/Http/Requests:

php artisan make:request StoreBlogPostRequest

O método authorize pode ser substituído pela lógica de autorização para esta solicitação:

public function authorize()
{        
    return $this->user()->can('post');
}

O método rules pode ser substituído pelas regras específicas para esta solicitação:

public function rules()
{
    return [
        'title' => 'required|unique:posts|max:255',
        'body' => 'required',
    ];
}

O método messages pode ser substituído pelas mensagens específicas para esta solicitação:

public function messages()
{
    return [
        'title.required' => 'A title is required',
        'title.unique' => 'There is another post with the same title',
        'title.max' => 'The title may not exceed :max characters',
        'body.required' => 'A message is required',
    ];
}

Para validar a solicitação, basta digitar a classe de solicitação específica no método do controlador correspondente. Se a validação falhar, uma resposta de erro será enviada de volta.

public function store(StoreBlogPostRequest $request)
{
    // validation passed
}

2) Como criar validadores manualmente

Para maior flexibilidade, você pode criar um Validador manualmente e lidar com a validação com falha diretamente:

<?php    
namespace App\Http\Controllers;

use Validator;
use Illuminate\Http\Request;
use App\Http\Controllers\Controller;

class PostController extends Controller
{
    public function store(Request $request)
    {
        $validator = Validator::make($request->all(), [
            'title' => 'required|unique:posts|max:255',
            'body' => 'required',
        ]);

        if ($validator->fails()) {
            return redirect('post/create')
                    ->withErrors($validator)
                    ->withInput();
        }

        // Store the blog post...
    }
}

2) Criar regras com fluência

Ocasionalmente você pode precisar criar regras únicas rapidamente, trabalhar com o método boot() dentro de um Provedor de Serviço pode ser exagerado, a partir do Laravel 5.4 você pode criar novas regras fluentemente usando a classe Rule.

Como exemplo vamos trabalhar com o UserRequest para quando você deseja inserir ou atualizar um usuário. Por enquanto, queremos que um nome seja obrigatório e o endereço de e-mail seja exclusivo. O problema de usar a regra unique é que se você estiver editando um usuário, ele pode manter o mesmo e-mail, então você precisa excluir o usuário atual da regra. O exemplo a seguir mostra como você pode fazer isso facilmente utilizando a nova classe Rule.

<?php
namespace App\Http\Requests;
use Illuminate\Foundation\Http\FormRequest;
use Illuminate\Http\Request;
use Illuminate\Validation\Rule;

class UserRequest extends FormRequest
{
    /**
     * Determine if the user is authorized to make this request.
     *
     * @return bool
     */
    public function authorize()
    {
        return true;
    }

    /**
     * Get the validation rules that apply to the request.
     *
     * @return array
     */
    public function rules(Request $request)
    {
        $id = $request->route()->getParameter('user');

        return [
            'name'           =>  'required',
            
            // Notice the value is an array and not a string like usual
            'email'         =>  [
                'required',
                Rule::unique('users')->ignore($id)
            ]
        ];
    }
}

Validação de matriz #

Validar campos de entrada de formulário de array é muito simples.

Suponha que você tenha que validar cada nome, e-mail e nome do pai em um determinado array. Você poderia fazer o seguinte:

$validator = \Validator::make($request->all(), [
    'name.*'       => 'required', 
    'email.*'      => 'email|unique:users',
    'fatherName.*' => 'required'
]);

if ($validator->fails()) {
    return back()->withInput()->withErrors($validator->errors());
}

Laravel exibe mensagens padrão para validação. No entanto, se você quiser mensagens personalizadas para campos baseados em matriz, poderá adicionar o seguinte código:

[
    'name.*' => [
        'required' => 'Name field is required',
    ],
    'email.*' => [
        'unique'   => 'Unique Email is required',
    ],
    'fatherName.*' => [
        'required' => 'Father Name required',
    ]
]

Seu código final ficará assim:

$validator = \Validator::make($request->all(), [
    'name.*'       => 'required', 
    'email.*'      => 'email|unique:users',
    'fatherName.*' => 'required',
], [
    'name.*'       => 'Name Required',
    'email.*'      => 'Unique Email is required',
    'fatherName.*' => 'Father Name required',
]);

if ($validator->fails()) {
    return back()->withInput()->withErrors($validator->errors());
}

Classe de solicitação de formulário único para POST, PUT, PATCH #

Seguindo o exemplo ‘Form Request Validation’, a mesma classe de solicitação pode ser usada para POST, PUT, PATCH para que você não precise criar outra classe usando validações iguais/semelhantes. Isso é útil se você tiver atributos exclusivos em sua tabela.

/**
 * Get the validation rules that apply to the request.
 *
 * @return array
 */
public function rules() {
    switch($this->method()) {
        case 'GET':
        case 'DELETE':
            return [];
        case 'POST':
            return [
                'name'     => 'required|max:75|unique',
                'category' => 'required',
                'price'    => 'required|between:0,1000',
            ];
        case 'PUT':
        case 'PATCH':
            return [
                'name'     => 'required|max:75|unique:product,name,' . $this->product,
                'category' => 'required',
                'price'    => 'required|between:0,1000',
            ];
        default:break;
    }
}

Começando do início, nossa instrução switch vai olhar para o tipo de método da requisição (GET, DELETE, POST, PUT, PATCH).

Dependendo do método retornará o array de regras definidas. Se você tiver um campo exclusivo, como o campo name no exemplo, precisará especificar um id específico para a validação ignorar.

'field_name' => 'unique:table_name,column_name,' . $idToIgnore`

Se você tiver uma chave primária rotulada como algo diferente de id, você especificará a coluna da chave primária como o quarto parâmetro.

'field_name' => 'unique:table_name,column_name,' . $idToIgnore . ',primary_key_column'

Neste exemplo, estamos usando PUT e passando para a rota (admin/products/{product}) o valor do id do produto. Então $this->product será igual ao id a ser ignorado.

Agora suas regras de validação PUT|PATCH e POST não precisam ser as mesmas. Defina sua lógica que se adapta aos seus requisitos. Essa técnica permite que você reutilize as mensagens personalizadas que você definiu na classe de solicitação de formulário personalizada.

Mensagens de erro #

Personalizando mensagens de erro #

Os arquivos /resources/lang/[lang]/validation.php contêm as mensagens de erro a serem usadas pelo validador. Você pode editá-los conforme necessário.

A maioria deles possui espaços reservados que serão substituídos automaticamente ao gerar a mensagem de erro.

Por exemplo, em 'required' => 'O campo :attribute é obrigatório.', o espaço reservado :attribute será substituído pelo nome do campo (alternativamente, você também pode personalizar o valor de exibição de cada campo no array attribute no mesmo arquivo).

Exemplo

configuração da mensagem:

'required' => 'Please inform your :attribute.',
//...
'attributes => [
    'email' => 'E-Mail address'
]

as regras:

`email' => `required`

mensagem de erro resultante:

“Informe seu endereço de e-mail.”

Personalizando mensagens de erro em uma classe Request #

A classe Request tem acesso a um método messages() que deve retornar um array, este pode ser usado para sobrescrever mensagens sem ter que entrar nos arquivos lang. Por exemplo, se tivermos uma validação file_exists personalizada, você pode enviar mensagens como abaixo.

class SampleRequest extends Request {

    /**
     * Get the validation rules that apply to the request.
     *
     * @return array
     */
    public function rules()
    {
        return [
            'image' =>  'required|file_exists'
        ];
    }

    /**
     * Determine if the user is authorized to make this request.
     *
     * @return bool
     */
    public function authorize()
    {
        return true;
    }

    public function messages()
    {
        return [
            'image.file_exists' =>  'That file no longer exists or is invalid'
        ];
    }

}

Exibindo mensagens de erro #

Os erros de validação são exibidos na sessão e também estão disponíveis na variável $errors, que é compartilhada automaticamente com todas as visualizações.

Exemplo de exibição dos erros em uma visualização Blade:

@if (count($errors) > 0)
    <div class="alert alert-danger">
        <ul>
            @foreach ($errors->all() as $error)
                <li>{{ $error }}</li>
            @endforeach
        </ul>
    </div>
@endif

Regras de validação personalizadas #

Se você deseja criar uma regra de validação personalizada, pode fazê-lo, por exemplo, no método boot de um provedor de serviços, por meio da fachada Validator.

<?php
namespace App\Providers;

use Illuminate\Support\ServiceProvider;
use Validator;

class AppServiceProvider extends ServiceProvider
{
    public function boot()
    {
        Validator::extend('starts_with', function($attribute, $value, $parameters, $validator) {
            return \Illuminate\Support\Str::startsWith($value, $parameters[0]);
        });

        Validator::replacer('starts_with', function($message, $attribute, $rule, $parameters) {
            return str_replace(':needle', $parameters[0], $message);
        });
    }
}

O método extend recebe uma string que será o nome da regra e uma função que por sua vez receberá o nome do atributo, o valor que está sendo validado, um array dos parâmetros da regra e a instância do validador, devendo retorne se a validação for aprovada. Neste exemplo, estamos verificando se a string de valor começa com uma determinada substring.

A mensagem de erro para esta regra personalizada pode ser definida normalmente no arquivo /resources/lang/[lang]/validation.php e pode conter espaços reservados, por exemplo, para valores de parâmetros:

'starts_with' => 'The :attribute must start with :needle.'

O método replacer pega uma string que é o nome da regra e uma função que por sua vez receberá a mensagem original (antes de substituir), o nome do atributo, o nome da regra e um array da regra parâmetros e deve retornar a mensagem após substituir os espaços reservados conforme necessário.

Use esta regra como qualquer outra:

$this->validate($request, [
    'phone_number' => 'required|starts_with:+'
]);
