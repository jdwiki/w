Inserindo modelos relacionados #

Suponha que você tenha um modelo Post com um relacionamento hasMany com Comment. Você pode inserir um objeto Comment relacionado a uma postagem fazendo o seguinte:

$post = Post::find(1);

$commentToAdd = new Comment(['message' => 'This is a comment.']);

$post->comments()->save($commentToAdd);

Você pode salvar vários modelos de uma vez usando a função saveMany:

$post = Post::find(1);

$post->comments()->saveMany([
    new Comment(['message' => 'This a new comment']),
    new Comment(['message' => 'Me too!']),
    new Comment(['message' => 'Eloquent is awesome!'])
]);

Alternativamente, há também um método create que aceita um array PHP simples em vez de uma instância do modelo Eloquent.

$post = Post::find(1);

$post->comments()->create([
    'message' => 'This is a new comment message'
]);

Tipos de relacionamento #

Um para muitos #

Digamos que cada Post pode ter um ou vários comentários e cada comentário pertence a apenas um único Post.

então a tabela de comentários terá post_id. Neste caso, as relações serão as seguintes.

Post Modelo

public function comments()
{
   return $this->belongsTo(Post::class);
}

Se a chave estrangeira for diferente de post_id, por exemplo, a chave estrangeira é example_post_id.

public function comments()
{
   return $this->belongsTo(Post::class, 'example_post_id');
}

e mais, se a chave local for diferente de id, por exemplo, a chave local é other_id

public function comments()
{
   return $this->belongsTo(Post::class, 'example_post_id', 'other_id');
}

Modelo de comentários

definindo o inverso de um para muitos

public function post()
{
   return $this->hasMany(Comment::class);
}

Um a um #

Como associar entre dois modelos (exemplo: modelo User e Phone) #

Aplicativo\Usuário

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class User extends Model
{
    /**
     * Get the phone record associated with the user.
     */
    public function phone()
    {
        return $this->hasOne('Phone::class', 'foreign_key', 'local_key');
    }
}

Aplicativo\Telefone

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Phone extends Model
{
    /**
     * Get the user that owns the phone.
     */
    public function user()
    {
        return $this->belongsTo('User::class', 'foreign_key', 'local_key');
    }
}

foreign_key : Por padrão, o Eloquent assumirá este valor como other_model_name_id (neste caso user_id e phone_id), altere-o se não for o caso.

local_key : Por padrão, o Eloquent assumirá este valor como id (chave primária do modelo atual), altere-o se não for o caso.

Se o nome do arquivo do banco de dados estiver de acordo com o padrão laravel, você não precisará fornecer a chave estrangeira e a chave local na declaração de relacionamento

Explicação #

Muitos para muitos #

Vamos dizer que há funções e permissões. Cada função pode pertencer a muitas permissões e cada permissão pode pertencer a muitas funções. então serão 3 mesas. dois modelos e uma mesa dinâmica. uma tabela roles, users e permission_role.

Modelo

public function permissions()
{
   return $this->belongsToMany(Permission::class);
}

Modelo de permissão

public function roles()
{
   return $this->belongsToMany(Roles::class);
}

Nota 1

considere seguir ao usar um nome de tabela diferente para a tabela dinâmica.

Suponha que você queira usar role_permission em vez de permission_role, pois o eloquent usa ordem alfabética para construir os nomes das chaves de pivô. você precisará passar o nome da tabela dinâmica como segundo parâmetro da seguinte maneira.

Modelo

public function permissions()
{
   return $this->belongsToMany(Permission::class, 'role_permission');
}

Modelo de permissão

public function roles()
{
   return $this->belongsToMany(Roles::class, 'role_permission');
}

Nota 2

considere seguir ao usar nomes de chave diferentes na tabela dinâmica.

O Eloquent assume que se nenhuma chave for passada como terceiro e quarto parâmetros, serão os nomes das tabelas singulares com _id. então assume que o pivô terá os campos role_id e permission_id. Se chaves diferentes dessas forem usadas, elas devem ser passadas como terceiro e quarto parâmetros.

Vamos dizer se other_role_id em vez de role_id e other_permission_id em vez de permission_id deve ser usado. Então seria o seguinte.

Modelo

public function permissions()
{
   return $this->belongsToMany(Permission::class, 'role_permission', 'other_role_id', 'other_permission_id');
}

Modelo de permissão

public function roles()
{
   return $this->belongsToMany(Roles::class, 'role_permission', 'other_permission_id', 'other_role_id');
}

Polimórfico #

As relações polimórficas permitem que um modelo pertença a mais de um outro modelo em uma única associação. Um bom exemplo seriam as imagens, tanto um usuário quanto um produto podem ter uma imagem. A estrutura da tabela pode ter a seguinte aparência:

user
    id - integer
    name - string
    email - string

product
    id - integer
    title - string
    SKU - string

image
    id - integer
    url - string
    imageable_id - integer
    imageable_type - string

As colunas importantes a serem observadas estão na tabela de imagens. A coluna imageable_id conterá o valor do ID do usuário ou produto, enquanto a coluna imageable_type conterá o nome da classe do modelo proprietário. Em seus modelos, você configura as relações da seguinte forma:

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Image extends Model
{
    /**
     * Get all of the owning imageable models.
     */
    public function imageable()
    {
        return $this->morphTo();
    }
}

class User extends Model
{
    /**
     * Get all of the user's images.
     */
    public function images()
    {
        return $this->morphMany('Image::class', 'imageable');
    }
}

class Product extends Model
{
    /**
     * Get all of the product's images.
     */
    public function images()
    {
        return $this->morphMany('Image::class', 'imageable');
    }
}

Você também pode recuperar o proprietário de uma relação polimórfica do modelo polimórfico acessando o nome do método que executa a chamada a morphTo. No nosso caso, esse é o método imageable no modelo Image. Então, vamos acessar esse método como uma propriedade dinâmica

$image = App\Image::find(1);

$imageable = $image->imageable;

Este imageable retornará um usuário ou um produto.

Consultando relacionamentos #

O Eloquent também permite consultar relacionamentos definidos, conforme mostrado abaixo:

User::whereHas('articles', function (Builder $query) {
    $query->where('published', '!=', true);
})->get();

Isso requer que o nome do seu método de relacionamento seja articles neste caso. O argumento passado para o encerramento é o Query Builder para o modelo relacionado, portanto, você pode usar qualquer consulta aqui que puder em outro lugar.

Carregamento ansioso

Suponha que o modelo de usuário tenha um relacionamento com o modelo de artigo e você queira carregar rapidamente os artigos relacionados. Isso significa que os artigos do usuário serão carregados durante a recuperação do usuário.

articles é o nome do relacionamento (método) no modelo User.

User::with('articles')->get();

se você tem relacionamento múltiplo. por exemplo, artigos e postagens.

User::with('articles','posts')->get();

e para selecionar relacionamentos aninhados

User::with('posts.comments')->get();

Chamar mais de um relacionamento aninhado

User::with('posts.comments.likes')->get()

public function permissions()

public function roles()

public function permissions()

public function roles()

Modelo de permissão

public function roles()

   return $this->belongsToMany(Roles::class, 'role_permission', 'other_permission_id', 'other_role_id');
}

Acessando a tabela intermediária usando withPivot()

Suponha que você tenha uma terceira coluna ‘permission_assigned_date’ na tabela dinâmica . Por padrão, apenas as chaves do modelo estarão presentes no objeto pivô. Agora, para obter esta coluna no resultado da consulta, você precisa adicionar o nome na função withPivot().

   public function permissions()
        {
           return $this->belongsToMany(Permission::class, 'role_permission', 'other_role_id', 'other_permission_id')->withPivot('permission_assigned_date');
        }

Anexando/Desanexando

O Eloquent também fornece alguns métodos auxiliares adicionais para tornar o trabalho com modelos relacionados mais conveniente. Por exemplo, vamos imaginar que um usuário pode ter muitas funções e uma função pode ter muitas permissões. Para anexar uma função a uma permissão inserindo um registro na tabela intermediária que une os modelos, use o método attach:

$role= App\Role::find(1);    
$role->permissions()->attach($permissionId);

Ao anexar um relacionamento a um modelo, você também pode passar um array de dados adicionais para serem inseridos na tabela intermediária:

$rol->roles()->attach($permissionId, ['permission_assigned_date' => $date]);

Da mesma forma, para remover uma permissão específica em relação a uma função, use a função detach

$role= App\Role::find(1);
//will remove permission 1,2,3 against role 1
$role->permissions()->detach([1, 2, 3]);

Sincronizando associações

Você também pode usar o método de sincronização para construir associações muitos-para-muitos. O método de sincronização aceita uma matriz de IDs para colocar na tabela intermediária. Quaisquer IDs que não estejam na matriz fornecida serão removidos da tabela intermediária. Portanto, após a conclusão desta operação, apenas os IDs no array fornecido existirão na tabela intermediária:

//will keep permission id's 1,2,3 against Role id 1

$role= App\Role::find(1)
$role->permissions()->sync([1, 2, 3]);

Introdução #

Os relacionamentos do Eloquent são definidos como funções em suas classes de modelo do Eloquent. Como, como os próprios modelos do Eloquent, os relacionamentos também servem como poderosos construtores de consultas, a definição de relacionamentos como funções fornece recursos poderosos de encadeamento e consulta de métodos. Por exemplo, podemos encadear restrições adicionais neste relacionamento de postagens:

$user->posts()->where('active', 1)->get();
