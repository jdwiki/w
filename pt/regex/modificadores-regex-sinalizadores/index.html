Padrões de expressão regular são frequentemente usados ​​com modificadores (também chamados de flags) que redefinem o comportamento regex. Os modificadores Regex podem ser regulares (por exemplo, /abc/i) e inline (ou incorporados) (por exemplo, (?i)abc). Os modificadores mais comuns são modificadores globais, não diferenciam maiúsculas de minúsculas, multilinhas e dotall. No entanto, os tipos de regex diferem no número de modificadores de regex suportados e seus tipos.

Modificadores de PCRE

Modificador Em linha Descrição
PCRE_CASELESS (?i) Correspondência sem distinção entre maiúsculas e minúsculas
PCRE_MULTILINE (?m) Correspondência de várias linhas
PCRE_DOTALL (?s) . corresponde a novas linhas
PCRE_ANCHORED (?A) Meta-caractere ^ corresponde apenas no início
PCRE_EXTENDED (?x) Os espaços em branco são ignorados
PCRE_DOLLAR_ENDONLY s/d Meta-caractere $ corresponde apenas no final
PCRE_EXTRA (?X) Análise de escape estrita
PCRE_UTF8 Manipula caracteres UTF-8
PCRE_UTF16 Manipula caracteres UTF-16
PCRE_UTF32 Manipula caracteres UTF-32
PCRE_UNGREEDY (?U) Define o mecanismo para correspondência lenta
PCRE_NO_AUTO_CAPTURE (?:) Desativa grupos de captura automática

VERBOSE / COMMENT / IgnorePatternWhitespace modificador #

O modificador que permite usar espaços em branco dentro de algumas partes do padrão para formatá-lo para melhor legibilidade e permitir comentários começando com #:

/(?x)^          # start of string
  (?=\D*\d)     # the string should contain at least 1 digit 
  (?!\d+$)      # the string cannot consist of digits only
  \#            # the string starts with a hash symbol
  [a-zA-Z0-9]+ # the string should have 1 or more alphanumeric symbols
  $             # end of string
/

Exemplo de uma string: #word1here. Observe que o símbolo # é escapado para denotar um literal # que faz parte de um padrão.

O espaço em branco sem escape no padrão de expressão regular é ignorado, faça escape para torná-lo parte do padrão.

Normalmente, o espaço em branco dentro de classes de caracteres ([...]) é tratado como um espaço em branco literal, exceto em Java.

Além disso, vale a pena mencionar que nos sabores PCRE, .NET, Python, Ruby Oniguruma, ICU, Boost regex pode-se usar comentários (?#:...) dentro do padrão regex.

modificador DOTALL #

Um padrão de regex em que um modificador DOTALL (na maioria dos sabores de regex expressos com s) altera o comportamento de . permitindo que ele corresponda a um símbolo de nova linha (LF):

/cat (.*?) dog/s

Esta regex no estilo Perl corresponderá a uma string como "gato fugiu de\na cachorro" capturando "fugiu de\na" no Grupo 1.

Uma versão inline: (?s) (por exemplo, (?s)gato (.*?) cachorro)

Observação: Em Ruby, o equivalente do modificador DOTALL é m, [modificador Regexp::MULTILINE](http://ruby-doc.org/core-2.1.1/Regexp.html#class- Regexp-label-Character+Classes) (por exemplo, /a.*b/m).

Observação: JavaScript não fornece um modificador DOTALL, portanto, um . nunca pode corresponder a um caractere de nova linha. Para obter o mesmo efeito, é necessária uma solução alternativa, por exemplo. g. substituindo todos os .s por uma classe de caracteres abrangente como [\S\s], ou uma classe de caracteres não nada [^] (no entanto, esta construção será tratada como um erro por todos outros motores e, portanto, não é portátil).

Modificador UNICODE #

O modificador UNICODE, geralmente expresso como u (PHP, Python) ou U (Java), faz com que o mecanismo regex trate o padrão e a string de entrada como strings e padrões Unicode, torne o padrão classes abreviadas como \w , \d, \s, etc. Compatível com Unicode.

/\A\p{L}+\z/u

é um regex PHP para combinar strings que consistem em 1 ou mais letras Unicode. Consulte a demonstração de regex.

Observe que em PHP, o modificador /u permite que o mecanismo PCRE manipule strings como strings UTF8 (ativando o verbo PCRE_UTF8) e torne as classes de caracteres abreviadas no padrão Unicode cientes (habilitando o verbo PCRE_UCP, veja mais em pcre.org).

As strings de padrão e assunto são tratadas como UTF-8. Este modificador está disponível no PHP 4.1.0 ou superior no Unix e no PHP 4.2.3 no win32. A validade UTF-8 do padrão e o assunto são verificados desde o PHP 4.3.5. Um assunto inválido fará com que a função preg_* não corresponda a nada; um padrão inválido acionará um erro de nível E_WARNING. Cinco e seis sequências UTF-8 de octetos são consideradas inválidas desde o PHP 5.3.4 (resp. PCRE 7.3 2007-08-28); anteriormente, esses eram considerados como UTF-8 válidos.

No Python 2.x, o re.UNICODE afeta apenas o próprio padrão: Make \w, \W, \b, \B, \d, \D, \s e \S dependentes do banco de dados de propriedades de caracteres Unicode.

Uma versão inline: (?u) em Python, (?U) em Java. Por exemplo:

print(re.findall(ur"(?u)\w+", u"Dąb")) # [u'D\u0105b']
print(re.findall(r"\w+", u"Dąb"))      # [u'D', u'b']

System.out.println("Dąb".matches("(?U)\\w+")); // true
System.out.println("Dąb".matches("\\w+"));     // false

Modificador MULTILINE #

Outro exemplo é um modificador MULTILINE (geralmente expresso com o sinalizador m (não em Oniguruma (por exemplo, Ruby) que usa m para denotar um modificador DOTALL)) que faz com que as âncoras ^ e $ correspondam ao início/fim de uma linha, não o início/fim de toda a string.

/^My Line \d+$/gm

encontrará todas as linhas que começam com ‘Minha Linha’ e contêm um espaço e 1+ dígitos até o final da linha.

Uma versão inline: (?m) (por exemplo, (?m)^My Line \d+$)

NOTA: No Oniguruma (por exemplo, em Ruby), e também em quase todos os editores de texto que suportam regexps, as âncoras ^ e $ denotam linha posições inicial/final por padrão. Você precisa usar \A para definir todo o início do documento/string e \z para denotar o final do documento/string. A diferença entre \Z e \z é que o primeiro pode corresponder antes do símbolo de nova linha (LF) no final da string (por exemplo, /\Astring\Z/ encontrará uma correspondência em " string\n") (exceto Python, onde o comportamento \Z é igual a \z e a âncora \z não é suportada).

Modificador IGNORE CASE #

O modificador comum para ignorar maiúsculas e minúsculas é i:

/fog/i

corresponderá a Fog, foG, etc.

A versão inline do modificador se parece com (?i).

Notas:

Em Java, por padrão, a correspondência [não diferencia maiúsculas de minúsculas assume que apenas os caracteres no conjunto de caracteres US-ASCII estão sendo correspondidos. A correspondência sem distinção entre maiúsculas e minúsculas com reconhecimento de Unicode pode ser habilitada especificando o sinalizador UNICODE_CASE em conjunto com este sinalizador (CASE_INSENSITIVE).](http://docs.oracle.com/javase/6/docs/api /java/util/regex/Pattern.html#CASE_INSENSITIVE) (por exemplo, Pattern p = Pattern.compile("YOUR_REGEX", Pattern.CASE_INSENSITIVE | Pattern.UNICODE_CASE);). Um pouco mais sobre isso pode ser encontrado em Case-Insensitive Matching in Java RegEx. Além disso, UNICODE_CHARACTER_CLASS pode ser usado para reconhecer Unicode correspondente.

Modificador de captura explícita #

Este é um modificador específico de regex .NET expresso com n. Quando usado, grupos sem nome (como (\d+)) não são capturados. Somente capturas válidas são grupos explicitamente nomeados (por exemplo, (?<nome> subexpressão)).

(?n)(\d+)-(\w+)-(?<id>\w+)

corresponderá ao 123-1_abc-00098 inteiro, mas (\d+) e (\w+) não criarão grupos no objeto de correspondência resultante. O único grupo será ${id}. Veja [demonstração](http://regexstorm.net/tester?p=(%3fn)(%5cd%2b)-(%5cw%2b)-(%3f%3cid%3e%5cw%2b)&i=123 -1_abc-00098&o=ixcs).

modificador PCRE_DOLLAR_ENDONLY #

O modificador PCRE_DOLLAR_ENDONLY compatível com PCRE que faz a âncora $ corresponder no final da string (excluindo a posição antes da nova linha final na string).

/^\d+$/D

é igual a

/^\d+\z/

e corresponde a uma string inteira que consiste em 1 ou mais dígitos e não corresponde a "123\n", mas corresponde a "123".

Edição PCRE_UNGREEDY #

O sinalizador PCRE_UNGREEDY compatível com PCRE expresso com /U. Ele alterna a ganância dentro de um padrão: /a.*?b/U = /a.*b/ e vice-versa.

Modificador PCRE_INFO_JCHANGED #

Mais um modificador PCRE que permite o uso de grupos nomeados duplicados.

NOTA: somente a versão inline é suportada - (?J), e deve ser colocada no início do padrão.

Se você usar

/(?J)\w+-(?:new-(?<val>\w+)|\d+-empty-(?<val>[^-]+)-collection)/

os valores do grupo “val” nunca estarão vazios (serão sempre definidos). Um efeito semelhante pode ser alcançado com a reinicialização do branch.

Modificador PCRE_EXTRA #

Um modificador PCRE que causa um erro se qualquer barra invertida em um padrão for seguida por uma letra sem significado especial. Por padrão, uma barra invertida seguida por uma letra sem significado especial é tratada como literal.

Por exemplo.

/big\y/

corresponderá a grande, mas

/big\y/X

lançará uma exceção.

Versão inline: (?X)

Modificador PCRE_ANCHORED #

Outro modificador compatível com PCRE expresso com o modificador /A. Se este modificador estiver definido, o padrão é forçado a ser “ancorado”, ou seja, é restrito a corresponder apenas no início da string que está sendo pesquisada (a “string de assunto”). Este efeito também pode ser alcançado por construções apropriadas no próprio padrão, que é a única maneira de fazer isso em Perl.

/man/A

é o mesmo que

/^man/
