Sintaxe #

  • CREATE ROLE nome [ [ WITH ] opção [ ... ] ]

  • CREATE USER name [ [ WITH ] opção [ ... ] ]

  • onde a opção pode ser: SUPERUSER | NOSUPERUSER | CREATEDB | NOCREATEDB | CREATEROLE | NOCREATEROLE | CREATEUSER | NOCREATEUSER | INHERIT | NOINHERIT | LOGIN | NOLOGIN | CONNECTION LIMIT connlimit | [ ENCRYPTED | UNENCRYPTED ] PASSWORD 'password' | VALID UNTIL 'timestamp' | IN ROLE role_name [, ...] | IN GROUP role_name [, ...] | ROLE role_name [, ...] | ADMIN role_name [, ...] | USER role_name [, ...] | SYSID uid

Crie um usuário com uma senha #

Geralmente você deve evitar usar a função de banco de dados padrão (geralmente postgres) em seu aplicativo. Em vez disso, você deve criar um usuário com níveis mais baixos de privilégios. Aqui nós fazemos um chamado niceusername e damos a ele uma senha very-strong-password

CREATE ROLE niceusername with PASSWORD 'very-strong-password' LOGIN;

O problema com isso é que as consultas digitadas no console psql são salvas em um arquivo de histórico .psql_history no diretório home do usuário e também podem ser registradas no log do servidor de banco de dados PostgreSQL, expondo assim a senha.

Para evitar isso, use o comando \password para definir a senha do usuário. Se o usuário que emite o comando for um superusuário, a senha atual não será solicitada. (Deve ser superusuário para alterar as senhas dos superusuários)

CREATE ROLE niceusername with LOGIN;
\password niceusername

Criar função e banco de dados correspondente #

Para oferecer suporte a um determinado aplicativo, você geralmente cria uma nova função e banco de dados para corresponder.

Os comandos do shell a serem executados seriam estes:

$ createuser -P blogger
Enter password for the new role: ********
Enter it again: ********

$ createdb -O blogger blogger

Isso assume que o pg_hba.conf foi configurado corretamente, o que provavelmente se parece com isso:

# TYPE  DATABASE        USER            ADDRESS                 METHOD
host    sameuser        all             localhost               md5
local   sameuser        all                                     md5

Conceder e revogar privilégios. #

Suponha que temos três usuários:

  1. O Administrador do banco de dados > admin

  2. O aplicativo com acesso total para seus dados > read_write

  3. O acesso somente leitura > read_only

    –ACCESS DB REVOKE CONNECT ON DATABASE nova FROM PUBLIC; GRANT CONNECT ON DATABASE nova TO user;

Com as consultas acima, usuários não confiáveis ​​não podem mais se conectar ao banco de dados.

--ACCESS SCHEMA
REVOKE ALL     ON SCHEMA public FROM PUBLIC;
GRANT  USAGE   ON SCHEMA public  TO user;

O próximo conjunto de consultas revoga todos os privilégios de usuários não autenticados e fornece um conjunto limitado de privilégios para o usuário read_write.

--ACCESS TABLES
REVOKE ALL ON ALL TABLES IN SCHEMA public FROM PUBLIC ;
GRANT SELECT                         ON ALL TABLES IN SCHEMA public TO read_only ;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO read_write ;
GRANT ALL                            ON ALL TABLES IN SCHEMA public TO admin ;

--ACCESS SEQUENCES
REVOKE ALL   ON ALL SEQUENCES IN SCHEMA public FROM PUBLIC;
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO read_only; -- allows the use of CURRVAL
GRANT UPDATE ON ALL SEQUENCES IN SCHEMA public TO read_write; -- allows the use of NEXTVAL and SETVAL
GRANT USAGE  ON ALL SEQUENCES IN SCHEMA public TO read_write; -- allows the use of CURRVAL and NEXTVAL
GRANT ALL    ON ALL SEQUENCES IN SCHEMA public TO admin;

Alterar search_path padrão do usuário #

Com os comandos abaixo, o search_path padrão do usuário pode ser definido.

  1. Verifique o caminho de pesquisa antes de definir o esquema padrão.

    postgres=# \c postgres user1 You are now connected to database “postgres” as user “user1”. postgres=> show search_path; search_path #

    “$user”,public (1 row)

  2. Defina search_path com o comando alter user para anexar um novo esquema my_schema

    postgres=> \c postgres postgres You are now connected to database “postgres” as user “postgres”. postgres=# alter user user1 set search_path=‘my_schema, “$user”, public’; ALTER ROLE

  3. Verifique o resultado após a execução.

    postgres=# \c postgres user1 Password for user user1: You are now connected to database “postgres” as user “user1”. postgres=> show search_path; search_path #

    my_schema, “$user”, public (1 row)

Alternativo:

postgres=# set role user1;
postgres=# show search_path;
 search_path 
-------------
 my_schema, "$user", public
(1 row)

Concede privilégios de acesso em objetos criados no futuro. #

Suponha que temos três usuários:

  1. O Administrador do banco de dados > admin
  2. O aplicativo com acesso total para seus dados > read_write
  3. O acesso somente leitura > read_only

Com as consultas abaixo, você pode definir privilégios de acesso em objetos criados no futuro no esquema especificado.

ALTER DEFAULT PRIVILEGES IN SCHEMA myschema GRANT SELECT                      ON TABLES TO read_only;
ALTER DEFAULT PRIVILEGES IN SCHEMA myschema GRANT SELECT,INSERT,DELETE,UPDATE ON TABLES TO read_write;
ALTER DEFAULT PRIVILEGES IN SCHEMA myschema GRANT ALL                         ON TABLES TO admin;

Ou você pode definir privilégios de acesso em objetos criados no futuro pelo usuário especificado.

ALTER DEFAULT PRIVILEGES FOR ROLE admin GRANT SELECT  ON TABLES TO read_only;

Criar usuário somente leitura #

CREATE USER readonly WITH ENCRYPTED PASSWORD 'yourpassword';
GRANT CONNECT ON DATABASE <database_name> to readonly;

GRANT USAGE ON SCHEMA public to readonly;
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
