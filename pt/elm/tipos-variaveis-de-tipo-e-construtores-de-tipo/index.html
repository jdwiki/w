Melhorando a segurança de tipos usando novos tipos #

Os tipos de alias reduzem o clichê e aumentam a legibilidade, mas não são mais seguros do que o próprio tipo de alias. Considere o seguinte:

type alias Email = String

type alias Name = String

someEmail = "holmes@private.com"

someName = "Benedict"

sendEmail : Email -> Cmd msg
sendEmail email = ...

Usando o código acima, podemos escrever sendEmail someName, e ele irá compilar, mesmo que não devesse, porque apesar de nomes e emails serem Strings, eles são coisas completamente diferentes.

Podemos realmente distinguir uma String de outra String no nível do tipo criando um novo tipo. Aqui está um exemplo que reescreve Email como um type em vez de um type alias:

module Email exposing (Email, create, send)

type Email = EmailAddress String

isValid : String -> Bool
isValid email = 
  -- ...validation logic

create : String -> Maybe Email
create email =
    if isValid email then
        Just (EmailAddress email)
    else
        Nothing

send : Email -> Cmd msg
send (EmailAddress email) = ...

Nossa função isValid faz algo para determinar se uma string é um endereço de e-mail válido. A função create verifica se uma determinada String é um e-mail válido, retornando um Email encapsulado em Maybe para garantir que retornamos apenas endereços validados. Embora possamos contornar a verificação de validação construindo um Email diretamente escrevendo EmailAddress "somestring", se nossa declaração de módulo não expor o construtor EmailAddress, como mostrado aqui

module Email exposing (Email, create, send)

então nenhum outro módulo terá acesso ao construtor EmailAddress, embora eles ainda possam usar o tipo Email nas anotações. A única maneira de construir um novo Email fora deste módulo é usando a função create que ele fornece, e essa função garante que ela retornará apenas endereços de e-mail válidos em primeiro lugar. Portanto, esta API guia automaticamente o usuário pelo caminho correto por meio de sua segurança de tipo: send só funciona com valores construídos por create, que executa uma validação e impõe o tratamento de emails inválidos, pois retorna um Maybe Email.

Se você quiser exportar o construtor Email, você pode escrever

module Email exposing (Email(EmailAddress), create, send)

Agora qualquer arquivo que importe Email também pode importar seu construtor. Nesse caso, isso permitiria que os usuários evitassem a validação e ’enviassem’ e-mails inválidos, mas você nem sempre está construindo uma API como essa, então exportar construtores pode ser útil. Com um tipo que possui vários construtores, você também pode querer exportar apenas alguns deles.

Tipos de dados comparáveis #

Tipos comparáveis ​​são tipos primitivos que podem ser comparados usando operadores de comparação do módulo Basics, como: (<), (>), (<=), (>=), max, min, comparar

Tipos comparáveis ​​no Elm são Int, Float, Time, Char, String e tuplas ou listas de tipos comparáveis.

Na documentação ou definições de tipo eles são referidos como uma variável de tipo especial comparável, por exemplo. veja a definição de tipo para a função Basics.max:

max : comparable -> comparable -> comparable

Tipos de construção #

A combinação de palavras-chave type alias fornece um novo nome para um tipo, mas a palavra-chave type isoladamente declara um novo tipo. Vamos examinar um dos mais fundamentais desses tipos: Maybe

type Maybe a
    = Just a
    | Nothing

A primeira coisa a notar é que o tipo Maybe é declarado com uma [type variable](https://www.wikiod.com/pt/elm/tipos-variaveis-de-tipo-e-construtores-de-tipo /8809/type-variables) de a. A segunda coisa a notar é o caractere pipe, |, que significa “ou”. Em outras palavras, algo do tipo ‘Talvez um’ é ou ‘Apenas um’ ou ‘Nada’.

Quando você escreve o código acima, Just e Nothing entram no escopo como construtores de valor, e Maybe entra no escopo como um construtor de tipo. Estas são as suas assinaturas:

Just : a -> Maybe a

Nothing : Maybe a

Maybe : a -> Maybe a -- this can only be used in type signatures

Por causa da variável type a, qualquer tipo pode ser “empacotado dentro” do tipo Maybe. Portanto, Maybe Int, Maybe (List String) ou Maybe (Maybe (List Html)), são todos tipos válidos. Ao desestruturar qualquer valor type com uma expressão case, você deve considerar cada possível instanciação desse tipo. No caso de um valor do tipo Talvez a, você deve considerar tanto o caso Apenas um quanto o caso Nada:

thing : Maybe Int
thing = 
    Just 3

blah : Int
blah =
    case thing of
        Just n -> 
            n

        Nothing ->
            42

-- blah = 3

Tente escrever o código acima sem a cláusula Nothing na expressão case: ele não compilará. Isso é o que torna o construtor de tipos Maybe um ótimo padrão para expressar valores que podem não existir, pois força você a lidar com a lógica de quando o valor é Nothing.

O tipo Nunca #

O tipo Never não pode ser construído (o módulo Basics não exportou seu construtor de valor e também não forneceu nenhuma outra função que retorne Never). Não há valor never : Never ou uma função createNever : ?? -> Nunca.

Isso tem seus benefícios: você pode codificar em um sistema de tipos uma possibilidade que não pode acontecer. Isso pode ser visto em tipos como Task Never Int que garante que será bem sucedido com um Int; ou Program Never que não aceitará nenhum parâmetro ao inicializar o código Elm do JavaScript.

Tipo de assinaturas #

No Elm, os valores são declarados escrevendo um nome, um sinal de igual e, em seguida, o valor real:

someValue = 42

As funções também são valores, com a adição de receber um valor ou valores como argumentos. Geralmente são escritos da seguinte forma:

double n = n * 2

Cada valor em Elm tem um tipo. Os tipos dos valores acima serão inferidos pelo compilador dependendo de como são usados. Mas é uma prática recomendada sempre declarar explicitamente o tipo de qualquer valor de nível superior e, para isso, você escreve uma assinatura de tipo da seguinte forma:

someValue : Int
someValue = 
    42

someOtherValue : Float
someOtherValue =
    42

Como podemos ver, 42 pode ser definido como ou um Int ou um Float. Isso faz sentido intuitivo, mas consulte Variáveis ​​de tipo para obter mais informações.

As assinaturas de tipo são particularmente valiosas quando usadas com funções. Aqui está a função de duplicação de antes:

double : Int -> Int
double n =
    n * 2

Desta vez, a assinatura tem um ->, uma seta, e pronunciamos a assinatura como “int para int”, ou “pega um inteiro e retorna um inteiro”. -> indica que dando double um valor Int como argumento, double retornará um Int. Portanto, leva um inteiro para um inteiro:

> double
<function> : Int -> Int

> double 3
6 : Int

Tipos básicos #

Em elm-repl, digite um pedaço de código para obter seu valor e tipo inferido. Tente o seguinte para aprender sobre os vários tipos que existem:

> 42
42 : number

> 1.987
1.987 : Float

> 42 / 2
21 : Float

> 42 % 2
0 : Int

> 'e'
'e' : Char

> "e"
"e" : String

> "Hello Friend"
"Hello Friend" : String

> ['w', 'o', 'a', 'h']
['w', 'o', 'a', 'h'] : List Char

> ("hey", 42.42, ['n', 'o'])
("hey", 42.42, ['n', 'o']) : ( String, Float, List Char )

> (1, 2.1, 3, 4.3, 'c')
(1,2.1,3,4.3,'c') : ( number, Float, number', Float, Char )

> {}
{} : {}

> { hey = "Hi", someNumber = 43 }
{ hey = "Hi", someNumber = 43 } : { hey : String, someNumber : number }

> ()
() : ()

{} é o tipo de registro vazio e () é o tipo de tupla vazio. Este último é frequentemente usado para fins de avaliação preguiçosa. Consulte o exemplo correspondente em https://www.wikiod.com/pt/elm/funcoes-e-aplicacao-parcial

Observe como number aparece sem maiúsculas. Isso indica que é uma Variável de tipo e, além disso, a palavra específica number se refere a uma Variável de tipo especial que pode ser uma Int ou uma Float (consulte as seções correspondentes para mais). Os tipos são sempre em maiúsculas, como Char, Float, List String, etc.

Variáveis ​​de tipo especial #

Elm define as seguintes variáveis ​​de tipo especial que têm um significado particular para o compilador:

  • comparável: Composto por Int, Float, Char, String e suas tuplas. Isso permite o uso dos operadores < e >.

Exemplo: Você pode definir uma função para encontrar os elementos menores e maiores em uma lista (extensão). Você pensa que tipo de assinatura escrever. Por um lado, você poderia escrever extentInt : List Int -> Maybe (Int, Int) e extentChar : List Char -> Maybe (Char, Char) e outro para Float e String. A implementação destes seria a mesma:

  extentInt list =
    let
      helper x (minimum, maximum) = 
        ((min minimum x), (max maximum x))
    in 
      case list of 
        [] ->
          Nothing
        x :: xs ->
          Just <| List.foldr helper (x, x) xs

 You might be tempted to simply write `extent : List a -> Maybe (a, a)`, but the compiler will not let you do this, because the functions `min` and `max` are not defined for these types (NB: these are just simple wrappers around the `<` operator mentioned above). You can solve this by defining `extent : List comparable -> Maybe (comparable, comparable)`. This allows your solution to be *polymorphic*, which just means that it will work for more than one type.
  • number: Composto por Int e Float. Permite o uso de operadores aritméticos, exceto divisão. Você pode então definir, por exemplo, sum : List number -> number e fazer com que funcione para ints e floats.
  • appendable: Composto por String, List. Permite o uso do operador ++.
  • compappend: Isso às vezes aparece, mas é um detalhe de implementação do compilador. Atualmente isso não pode ser usado em seus próprios programas, mas às vezes é mencionado.

Observe que em uma anotação de tipo como esta: number -> number -> number todos se referem ao mesmo tipo, então passar Int -> Float -> Int seria um erro de tipo. Você pode resolver isso adicionando um sufixo ao nome da variável de tipo: number -> number' -> number'' compilaria bem.

Não há nome oficial para estes, às vezes são chamados:

  • Variáveis ​​de Tipo Especial
  • Variáveis ​​de tipo tipo classe
  • Classes pseudo-tipo

Isso ocorre porque eles funcionam como as [Type Classes] de Haskell (https://www.wikiod.com/pt/haskell/classes-de-tipo), mas sem a capacidade de o usuário defini-las.

Variáveis ​​de tipo #

Variáveis ​​de tipo são nomes sem maiúsculas em assinaturas de tipo. Ao contrário de suas contrapartes em maiúsculas, como Int e String, eles não representam um único tipo, mas sim qualquer tipo. Eles são usados ​​para escrever funções genéricas que podem operar em qualquer tipo ou tipos, e são particularmente úteis para escrever operações em contêineres como List ou Dict. A função List.reverse, por exemplo, tem a seguinte assinatura:

reverse : List a -> List a

O que significa que pode funcionar em uma lista de qualquer valor de tipo, então List Int, List (List String), ambos e quaisquer outros podem ser revertidos da mesma forma. Portanto, a é uma variável de tipo que pode substituir qualquer tipo.

A função reverse poderia ter usado qualquer nome de variável sem letras maiúsculas em sua assinatura de tipo, exceto pelo punhado de nomes de variáveis ​​de tipo especial, como number (veja o exemplo correspondente para mais informações):

reverse : List lol -> List lol

reverse : List wakaFlaka -> List wakaFlaka

Os nomes das variáveis ​​de tipo tornam-se significativos apenas quando existem variáveis ​​de tipo diferentes dentro de uma única assinatura, exemplificado pela função map nas listas:

map : (a -> b) -> List a -> List b

map pega alguma função de qualquer tipo a para qualquer tipo b, junto com uma lista com elementos de algum tipo a, e retorna uma lista de elementos de algum tipo b, que obtém aplicando a função dada para cada elemento da lista.

Vamos tornar a assinatura concreta para ver melhor isso:

plusOne : Int -> Int
plusOne x = 
    x + 1

> List.map plusOne
<function> : List Int -> List Int

Como podemos ver, tanto a = Int quanto b = Int neste caso. Mas, se map tivesse uma assinatura de tipo como map : (a -> a) -> List a -> List a, então ele somente funcionaria em funções que operam em um único tipo, e você nunca poderá alterar o tipo de uma lista usando a função map. Mas como a assinatura de tipo de map tem várias variáveis ​​de tipo diferentes, a e b, podemos usar map para alterar o tipo de uma lista:

isOdd : Int -> Bool
isOdd x =
    x % 2 /= 0

> List.map isOdd
<function> : List Int -> List Bool

Neste caso, a = Int e b = Bool. Portanto, para poder usar funções que podem receber e retornar tipos diferentes, você deve usar variáveis ​​de tipo diferentes.

Alias ​​de tipo #

Às vezes, queremos dar a um tipo um nome mais descritivo. Digamos que nosso aplicativo tenha um tipo de dados representando usuários:

{ name : String, age : Int, email : String }

E nossas funções nos usuários têm assinaturas de tipo ao longo das linhas de:

prettyPrintUser : { name : String, age : Int, email : String } -> String

Isso pode se tornar bastante complicado com um tipo de registro maior para um usuário, então vamos usar um alias de tipo para reduzir o tamanho e dar um nome mais significativo a essa estrutura de dados:

type alias User =
    { name: String
    , age : Int
    , email : String
    }


prettyPrintUser : User -> String

Os aliases de tipo tornam muito mais fácil definir e usar um modelo para um aplicativo:

type alias Model =
    { count : Int
    , lastEditMade : Time
    }

Usar type alias literalmente apenas aliase um tipo com o nome que você dá a ele. Usar o tipo Model acima é exatamente o mesmo que usar { count : Int, lastEditMade : Time }. Aqui está um exemplo que mostra como os aliases não são diferentes dos tipos subjacentes:

type alias Bugatti = Int

type alias Fugazi = Int

unstoppableForceImmovableObject : Bugatti -> Fugazi -> Int
unstoppableForceImmovableObject bug fug =
    bug + fug

> unstoppableForceImmovableObject 09 87
96 : Int

Um alias de tipo para um tipo de registro define uma função construtora com um argumento para cada campo na ordem de declaração.

type alias Point = { x : Int, y : Int }

Point 3 7
{ x = 3, y = 7 } : Point

type alias Person = { last : String, middle : String, first : String }

Person "McNameface" "M" "Namey"
{ last = "McNameface", middle = "M", first = "Namey" } : Person

Cada alias de tipo de registro tem sua própria ordem de campo, mesmo para um tipo compatível.

type alias Person = { last : String, middle : String, first : String }
type alias Person2 = { first : String, last : String, middle : String }

Person2 "Theodore" "Roosevelt" "-"
{ first = "Theodore", last = "Roosevelt", middle = "-" } : Person2

a = [ Person "Last" "Middle" "First", Person2 "First" "Last" "Middle" ]
[{ last = "Last", middle = "Middle", first = "First" },{ first = "First", last = "Last", middle = "Middle" }] : List Person2
