Introdução #

Verilog é uma linguagem de descrição de hardware (HDL) usada para modelar sistemas eletrônicos. Ele descreve mais comumente um sistema eletrônico no nível de transferência de registro (RTL) de abstração. Também é usado na verificação de circuitos analógicos e circuitos de sinais mistos. Sua estrutura e principais princípios (conforme descritos abaixo) são projetados para descrever e implementar com sucesso um sistema eletrônico.

  • Rigidez Um circuito eletrônico é uma entidade física que possui uma estrutura fixa e o Verilog é adaptado para isso. Módulos (módulo), Portas (entrada/saída/saída) , conexões (fios), blocos (@sempre), registradores (reg) são todos fixos em tempo de compilação. O número de entidades e interconexões não muda dinamicamente. Há sempre um “módulo” no nível superior representando a estrutura do chip (para síntese) e um no nível do sistema para verificação.

  • Paralelismo As operações simultâneas inerentes no chip físico são imitadas na linguagem por blocos sempre (mais comuns), iniciais e fork/join.

    module top();
reg r1,r2,r3,r4; // 1-bit registers
  initial
  begin
    r1 <= 0 ;
  end
  initial
  begin
    fork
       r2 <= 0 ;
       r3 <= 0 ;
    join
  end
   always @(r4)
      r4 <= 0 ;

    endmodule

Todas as instruções acima são executadas em paralelo dentro da mesma unidade de tempo.

  • Tempo e Sincronização Verilog suporta várias construções para descrever a natureza temporal dos circuitos. Tempos e atrasos em circuitos podem ser implementados em Verilog, por exemplo, por construções #delay. Da mesma forma, o Verilog também acomoda circuitos e componentes síncronos e assíncronos como flops, latches e lógica combinatória usando várias construções, por exemplo, blocos “sempre”. Um conjunto de blocos também pode ser sincronizado por meio de um sinal de clock comum ou um bloco pode ser acionado com base em um conjunto específico de entradas.

     #10   ;                 // delay for 10 time units
 always @(posedge clk )  // synchronous 
 always @(sig1 or sig2 ) // combinatorial logic  
 @(posedge event1)       // wait for post edge transition of event1
 wait (signal  == 1)     // wait for signal to be 1

  • Incerteza Verilog suporta algumas das incertezas inerentes aos circuitos eletrônicos. “X” é usado para representar o estado desconhecido do circuito. “Z” é usado para representar o estado não acionado do circuito.

     reg1 = 1'bx;
 reg2 = 1'bz;

  • Abstração Verilog suporta o design em diferentes níveis de abstração. O nível mais alto de abstração para um projeto é o nível de transferência de resistência (RTL), sendo o próximo o nível de porta e o mais baixo o nível de célula ( Primitivos definidos pelo usuário ), sendo a abstração RTL a mais comumente usada. O Verilog também suporta o nível comportamental de abstração sem levar em conta a realização estrutural do projeto, usado principalmente para verificação.

    // Example of a D flip flop at RTL abstraction module dff ( clk , // Clock Input reset , // Reset input d , // Data Input q // Q output ); //———–Input Ports————— input d, clk, reset ;

    //———–Output Ports————— output q;

    reg q;

    always @ ( posedge clk) if (~reset) begin q <= 1’b0; end else begin q <= d; end

    endmodule

    // And gate model based at Gate level abstraction module and(input x,input y,output o);

    wire w; // Two instantiations of the module NAND nand U1(w,x, y); nand U2(o, w, w);

    endmodule

    // Gate modeled at Cell-level Abstraction primitive udp_and( a, // declare three ports b, c ); output a; // Outputs input b,c; // Inputs

    // UDP function code here // A = B & C; table // B C : A 1 1 : 1; 0 1 : 0; 1 0 : 0; 0 0 : 0; endtable

    endprimitive

Existem três casos de uso principais para o Verilog. Eles determinam a estrutura do código e sua interpretação e também determinam os conjuntos de ferramentas usados. Todos os três aplicativos são necessários para a implementação bem-sucedida de qualquer projeto Verilog.

  1. Projeto Físico / Back-end Aqui o Verilog é usado principalmente para visualizar o design como uma matriz de portas de interconexão que implementam o design lógico. O RTL/lógica/Design passa por várias etapas, desde a síntese -> colocação -> construção da árvore do relógio -> roteamento -> DRC -> LVS -> para tapeout. As etapas e sequências precisas variam com base na natureza exata da implementação.
  2. Simulação Neste caso de uso, o objetivo principal é gerar vetores de teste para validar o projeto de acordo com a especificação. O código escrito neste caso de uso não precisa ser sintetizado e permanece dentro da esfera de verificação. O código aqui se assemelha mais a estruturas de software genéricas como loops for/while/do etc.
  3. Projeto O projeto envolve a implementação da especificação de um circuito geralmente no nível de abstração RTL. O código Verilog é fornecido para verificação e o código totalmente verificado é fornecido para implementação física. O código é escrito usando apenas as construções sintetizáveis ​​do Verilog. Certo estilo de codificação RTL pode causar incompatibilidade entre simulação e síntese e deve-se tomar cuidado para evitá-los.

Existem dois fluxos de implementação principais. Eles também afetarão a forma como o código Verilog é escrito e implementado. Certos estilos de codificação e certas estruturas são mais adequados em um fluxo sobre o outro.

  • ASIC Flow (circuito integrado específico da aplicação)
  • FPGA Flow (matriz de portas programáveis ​​em campo) - inclui FPGA e CPLD’s

Olá Mundo #

Este exemplo usa o compilador icarus verilog.

Etapa 1: crie um arquivo chamado hello.v

module myModule();

initial
  begin
    $display("Hello World!");   // This will display a message
    $finish ; // This causes the simulation to end.  Without, it would go on..and on.
  end

endmodule

Etapa 2. Compilamos o arquivo .v usando icarus:

>iverilog -o hello.vvp hello.v

A opção -o atribui um nome ao arquivo de objeto de saída. Sem essa opção, o arquivo de saída seria chamado de a.out. O hello.v indica o arquivo fonte a ser compilado. Praticamente não deve haver saída ao compilar esse código-fonte, a menos que haja erros.

Etapa 3. Você está pronto para simular este programa de verilog Hello World. Para fazer isso, invoque como tal:

>vvp hello.vvp 
Hello World!
>

Instalação do Icarus Verilog Compiler para Mac OSX Sierra #

  1. Instale o Xcode da App Store.

  2. Instale as ferramentas de desenvolvedor do Xcode

    xcode-select –install

Isso fornecerá ferramentas básicas de linha de comando, como gcc e make

  1. Instale as portas Mac https://www.macports.org/install.php

O pacote de instalação do OSX Sierra fornecerá um método de código aberto de instalação e atualização de pacotes de software adicionais na plataforma Mac. Pense em yum ou apt-get para o Mac.

  1. Instale o icarus usando portas Mac

    sudo port install iverilog

  2. Verifique a instalação na linha de comando

    $ iverilog iverilog: no source files.

    Usage: iverilog [-ESvV] [-B base] [-c cmdfile|-f cmdfile] [-g1995|-g2001|-g2005] [-g] [-D macro[=defn]] [-I includedir] [-M depfile] [-m module] [-N file] [-o filename] [-p flag=value] [-s topmodule] [-t target] [-T min|typ|max] [-W class] [-y dir] [-Y suf] source_file(s)

    See the man page for details. $ Agora você está pronto para compilar e simular seu primeiro arquivo Verilog no Mac.

Instale o GTKWave para exibição gráfica de dados de simulação no Mac OSx Sierra #

GTKWave é um pacote de visualização gráfica completo que suporta vários padrões de armazenamento de dados gráficos, mas também suporta VCD, que é o formato que o vvp produzirá. Então, para pegar o GTKWave, você tem algumas opções

  1. Vá para http://gtkwave.sourceforge.net/gtkwave.zip e faça o download. Esta versão é normalmente a mais recente.
  2. Se você instalou o MacPorts (https://www.macports.org/), simplesmente execute sudo port install gtkwave. Isso provavelmente vai querer instalar nas dependências para. Observe que esse método geralmente fornecerá uma versão mais antiga. Se você não tiver o MacPorts instalado, há um exemplo de configuração de instalação para fazer isso nesta página. Sim! Você precisará de todas as ferramentas do desenvolvedor xcode, pois esses métodos “criarão” um GTKWave a partir da fonte.

Quando a instalação estiver concluída, você pode ser solicitado a selecionar uma versão do python. Eu já tinha o 2.7.10 instalado, então nunca “selecionei” um novo.

Neste ponto você pode iniciar o gtkwave a partir da linha de comando com gtkwave. Quando ele é iniciado, você pode ser solicitado a instalar ou atualizar o XQuarts. Faça isso. No meu caso, o XQuarts 2.7.11 está instalado.

Nota: Eu realmente precisei reiniciar para obter o XQuarts corretamente, então digitei gtkwave novamente e o aplicativo apareceu.

No próximo exemplo, criarei dois arquivos independentes, um testbench e um módulo para testar, e usaremos o gtkwave para visualizar o design.

Usando Icarus Verilog e GTKWaves para simular e visualizar um projeto graficamente #

Este exemplo usa Icarus e GTKWave. As instruções de instalação para essas ferramentas no OSx são fornecidas em outras partes desta página.

Vamos começar com o design do módulo. Este módulo é um display BCD para 7 segmentos. Eu codifiquei o design de uma maneira obtusa simplesmente para nos dar algo que é facilmente quebrado e podemos gastar algum tempo corrigindo graficamente. Então temos um clock, reset, uma entrada de 4 dados representando um valor BCD e uma saída de 7 bits que representa o display de sete segmentos. Crie um arquivo chamado bcd_to_7seg.v e coloque a fonte abaixo nele.

module bcd_to_7seg (
   input clk,
   input reset,
   input [3:0] bcd,
   output [6:0] seven_seg_display

);
   parameter TP = 1;
   reg seg_a;
   reg seg_b;
   reg seg_c;
   reg seg_d;
   reg seg_e;
   reg seg_f;
   reg seg_g;

   
   always @ (posedge clk or posedge reset)
      begin
      if (reset)
         begin
            seg_a <= #TP 1'b0;
            seg_b <= #TP 1'b0;
            seg_c <= #TP 1'b0;
            seg_d <= #TP 1'b0;
            seg_e <= #TP 1'b0;
            seg_f <= #TP 1'b0;
            seg_g <= #TP 1'b0;
         end
      else
         begin
            seg_a <= #TP  ~(bcd == 4'h1 || bcd == 4'h4);
            seg_b <= #TP  bcd < 4'h5 || bcd > 6;
            seg_c <= #TP   bcd != 2;
            seg_d <= #TP   bcd == 0 || bcd[3:1] == 3'b001 || bcd == 5 || bcd == 6 || bcd == 8;
            seg_e <= #TP  bcd == 0 || bcd == 2 || bcd == 6 || bcd == 8;
            seg_f <= #TP  bcd == 0 || bcd == 4 || bcd == 5 || bcd == 6 || bcd > 7;
            seg_g <= #TP  (bcd > 1 && bcd < 7) || (bcd > 7);
         end
    end
 
    assign seven_seg_display = {seg_g,seg_f,seg_e,seg_d,seg_c,seg_b,seg_a};
endmodule

Em seguida, precisamos de um teste para verificar se este módulo está funcionando corretamente. A instrução case no testbench é realmente mais fácil de ler na minha opinião e mais clara quanto ao que ela faz. Mas eu não queria colocar a mesma declaração de caso no design E no teste. Isso é má prática. Em vez disso, dois projetos independentes estão sendo usados ​​para validar um ao outro.

Dentro do código abaixo, você notará duas linhas $dumpfile("testbench.vcd"); e $dumpvars(0,testbench);. Essas linhas são o que cria o arquivo de saída VCD que será usado para realizar a análise gráfica do projeto. Se você deixá-los de fora, não será gerado um arquivo VCD. Crie um arquivo chamado testbench.v e coloque a fonte abaixo nele.

`timescale 1ns/100ps
module testbench;
reg clk;
reg reset;
reg [31:0] ii;
reg [31:0] error_count;
reg [3:0] bcd;
wire [6:0] seven_seg_display; 
parameter TP = 1;
parameter CLK_HALF_PERIOD = 5;
 
   // assign clk = #CLK_HALF_PERIOD ~clk;  // Create a clock with a period of ten ns
   initial
   begin
     clk = 0;
     #5;
     forever clk = #( CLK_HALF_PERIOD )  ~clk;
   end

   initial
     begin
       $dumpfile("testbench.vcd");
       $dumpvars(0,testbench);
       // clk  = #CLK_HALF_PERIOD ~clk; 
       $display("%0t, Reseting system", $time);
       error_count = 0;
       bcd  = 4'h0;
       reset = #TP 1'b1;
       repeat (30) @ (posedge clk);
       reset  = #TP 1'b0;
       repeat (30) @ (posedge clk);
       $display("%0t, Begin BCD test", $time); // This displays a message


       for (ii = 0; ii < 10; ii = ii + 1)
          begin
          repeat (1) @ (posedge clk);
          bcd  = ii[3:0];
          repeat (1) @ (posedge clk); 
          if (seven_seg_display !== seven_seg_prediction(bcd)) 
             begin
                $display("%0t, ERROR: For BCD %d, module output 0b%07b does not match prediction logic value of 0b%07b.",$time,bcd, seven_seg_display,seven_seg_prediction(bcd));
                error_count = error_count + 1;
             end
          end
       $display("%0t, Test Complete with %d errors", $time, error_count);
       $display("%0t, Test %s", $time, ~|error_count ? "pass." : "fail.");
       $finish ; // This causes the simulation to end.
     end


parameter SEG_A = 7'b0000001;
parameter SEG_B = 7'b0000010;
parameter SEG_C = 7'b0000100;
parameter SEG_D = 7'b0001000;
parameter SEG_E = 7'b0010000;
parameter SEG_F = 7'b0100000;
parameter SEG_G = 7'b1000000;

function [6:0] seven_seg_prediction;
   input [3:0] bcd_in;

   //    +--- A ---+
   //    |         |
   //    F         B
   //    |         |
   //    +--- G ---+
   //    |         |
   //    E         C
   //    |         |
   //    +--- D ---+

   begin
      case (bcd_in)
         4'h0: seven_seg_prediction = SEG_A | SEG_B | SEG_C | SEG_D | SEG_E | SEG_F;
         4'h1: seven_seg_prediction = SEG_B | SEG_C;
         4'h2: seven_seg_prediction = SEG_A | SEG_B | SEG_G | SEG_E | SEG_D;
         4'h3: seven_seg_prediction = SEG_A | SEG_B | SEG_G | SEG_C | SEG_D;
         4'h4: seven_seg_prediction = SEG_F | SEG_G | SEG_B | SEG_C;
         4'h5: seven_seg_prediction = SEG_A | SEG_F | SEG_G | SEG_C | SEG_D;
         4'h6: seven_seg_prediction = SEG_A | SEG_F | SEG_G | SEG_E | SEG_C | SEG_D;
         4'h7: seven_seg_prediction = SEG_A | SEG_B | SEG_C;
         4'h8: seven_seg_prediction = SEG_A | SEG_B | SEG_C | SEG_D | SEG_E | SEG_F | SEG_G;
         4'h9: seven_seg_prediction = SEG_A | SEG_F | SEG_G | SEG_B | SEG_C;
         default: seven_seg_prediction = 7'h0;
      endcase
   end
endfunction


bcd_to_7seg u0_bcd_to_7seg (
.clk               (clk),
.reset             (reset),
.bcd               (bcd),
.seven_seg_display (seven_seg_display)
);


endmodule

Agora que temos dois arquivos, um testbench.v e bcd_to_7seg.v, precisamos compilar, elaborar usando Icarus. Para fazer isso:

$ iverilog -o testbench.vvp testbench.v bcd_to_7seg.v

Em seguida, precisamos simular

$ vvp testbench.vvp 
LXT2 info: dumpfile testbench.vcd opened for output.
0, Reseting system
6000, Begin BCD test
8000, Test Complete with          0 errors
8000, Test pass.

Neste ponto, se você quiser validar se o arquivo está realmente sendo testado, vá para o arquivo bcd_2_7seg.v e mova um pouco da lógica e repita essas duas primeiras etapas.

Como exemplo eu mudo a linha seg_c <= #TP bcd != 2; para seg_c <= #TP bcd != 4;. Recompilar e simular faz o seguinte:

$ iverilog -o testbench.vvp testbench.v bcd_to_7seg.v
$ vvp testbench.vvp 
LXT2 info: dumpfile testbench.vcd opened for output.
0, Reseting system
6000, Begin BCD test
6600, ERROR: For BCD  2, module output 0b1011111 does not match prediction logic value of 0b1011011.
7000, ERROR: For BCD  4, module output 0b1100010 does not match prediction logic value of 0b1100110.
8000, Test Complete with          2 errors
8000, Test fail.
$

Então agora, vamos ver a simulação usando GTKWave. Na linha de comando, emita um

gtkwave testbench.vcd &

Quando a janela GTKWave aparecer, você verá na caixa superior esquerda, o nome do módulo testbench. Clique. Isso revelará os submódulos, tarefas e funções associadas a esse arquivo. Fios e registros também aparecerão na caixa inferior esquerda.

Agora arraste, clk, bcd, error_count e seven_seg_display para a caixa de sinal ao lado da janela da forma de onda. Os sinais serão agora plotados. Error_count mostrará qual entrada BCD específica gerou a saída seven_seg_display errada.

Agora você está pronto para solucionar um bug Verilog graficamente.
