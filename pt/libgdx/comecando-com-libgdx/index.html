Instalação ou Configuração #

LibGDX tem uma configuração bastante simples, com a ajuda de um programa Java simples. Você pode encontrar o download aqui. Quando você inicializar o aplicativo, ele ficará assim:
Nota: Esta captura de tela foi feita no Linux e mostra o caminho que difere de uma instalação do Windows. No entanto, o formato é o mesmo em qualquer SO compatível com este aplicativo de configuração

Caixas de entrada #

Na caixa de entrada “Nome”, está o nome do jogo para seu projeto. A caixa de entrada “Pacote” é o pacote para sua classe principal. A caixa de entrada “Game Class” é a classe principal que é chamada quando o jogo é executado. A caixa de entrada “Destino” é o destino para onde seu projeto será gerado. A caixa de entrada “Andriod SDK”, o caminho para onde seu SDK Android está. Essa caixa de entrada é totalmente opcional, portanto, se você não deseja implantar seu aplicativo no Android, não precisa se preocupar com isso.

Subprojetos #

Os subprojetos são apenas as plataformas nas quais serão implantados. Essa parte é bem autoexplicativa. Se você não quiser implantá-lo em HTML, por exemplo, simplesmente desmarque a caixa de seleção.

Geração #

Depois de ter todas as suas configurações corretas, você pode pressionar o botão “Gerar”. Isso pode levar alguns segundos, mas gerará os arquivos básicos e o Gradle necessários para o seu projeto. Uma vez feito isso, é hora de importar o projeto para o seu IDE.

LibGDX Olá Mundo #

Fundamentos

Os projetos gerados contêm uma aplicação básica do tipo Hello World já implementada.

O projeto principal é o projeto principal, que contém todo o código independente de plataforma. Isso é obrigatório, mas com base nas configurações de geração, você pode ter vários outros projetos para cada plataforma selecionada.

O exemplo

Abra com.mygdx.game.MyGdxGame.java no projeto core. Você verá o seguinte código:

public class MyGdxGame extends ApplicationAdapter {
    SpriteBatch batch;
    Texture img;
    
    @Override
    public void create () {
        batch = new SpriteBatch();
        img = new Texture("badlogic.jpg");
    }

    @Override
    public void render () {
        Gdx.gl.glClearColor(1, 0, 0, 1);
        Gdx.gl.glClear(GL20.GL_COLOR_BUFFER_BIT);
        batch.begin();
        batch.draw(img, 0, 0);
        batch.end();
    }
    
    @Override
    public void dispose () {
        batch.dispose();
        img.dispose();
    }
}

Embora este seja seu projeto principal, você não irá executá-lo diretamente, você sempre tem que executar o Launcher específico da plataforma, para desktop ele é chamado com.mygdx.game.desktop.DesktopLauncher.java no projeto desktop.

public class DesktopLauncher {
    public static void main (String[] arg) {
        LwjglApplicationConfiguration config = new LwjglApplicationConfiguration();
        new LwjglApplication(new MyGdxGame(), config);
    }
}

Nesta aula, você pode definir configurações específicas da plataforma.

O que está acontecendo na classe MyGdxGame? #

Primeiro é chamado o método create, que irá inicializar o lote que desenha na tela. Em seguida, o método carrega o badlogic.jpg na memória.

Depois disso, o método render é chamado repetidamente até que o aplicativo seja interrompido. Este método irá redefinir a cor de fundo para vermelho e desenhará a imagem na tela. Como você pode ver, você sempre tem que começar e terminar o desenho do lote. Por fim, quando a aplicação está prestes a parar, o método dispose é chamado, que liberará o espaço de memória usado pela textura e pelo lote.

(É bom saber que o descarte pode acontecer em tempo de execução também em outras plataformas, por exemplo, Android, porque o Android pode liberar espaço de memória quando os aplicativos estão em segundo plano, mas esse é um tópico mais avançado)

Visão geral do LibGDX #

LibGDX é uma biblioteca de desenvolvimento de jogos gratuita e de código aberto desenvolvida em Java. Seus objetivos são permitir que os usuários desenvolvam jogos multiplataforma que rodam em desktop, Android, iOS e navegadores da web. Escreva o código uma vez, implante-o em qualquer uma das principais plataformas.

Adicionando suporte para código específico da plataforma #

O LibGDX foi projetado de forma que você possa escrever o mesmo código e implantá-lo em várias plataformas diferentes. No entanto, há momentos em que você deseja obter acesso ao código específico da plataforma. Por exemplo, se você tiver placares e conquistas em seu jogo, convém usar ferramentas específicas da plataforma (como o Google Play Games), além de armazená-las localmente. Ou você quer usar um banco de dados, ou algo completamente diferente.

Você não pode adicionar esse tipo de código ao módulo principal. Então o primeiro passo é criar uma Interface. Crie-o no módulo principal. Este primeiro é um utilitário para gerenciar outras interfaces:

public interface PlatformWrapper{
    //here you can also add other things you need that are platform specific.
    //If you want to create your own file saver for an instance, a JSON based database,
    //achievements, leaderboards, in app purchases and anything else you need platform specific code for.
    SQLWrapper getSQLSaver();//This one will be used in this example
    AchievementWrapper getAchievementHandler();//this line is here as an example
}

Em seguida, precisamos criar uma segunda interface, a SQLWrapper. Este também vai no módulo principal.

public interface SQLWrapper{
    void init(String DATABASE);
    void saveSerializable(int id, Object o);
    Object loadSerializable(int id, Object o);
    void saveString(int id, String s);
    //.... and other methods you need here. This is something that varies
    //with usage. You may not need the serializable methods, but really need a custom method for saving 
    //an entire game world. This part is entirely up to you to find what methods you need
   
    //With these three methods, always assume it is the active database in question. Unless 
    //otherwise specified, it most likely is
    String getActiveDatabase();
    void deleteDatabase();
    void deleteTable(String table);//delete the active table
}

Agora, você precisa entrar em cada projeto e criar uma classe para implementar PlatformWrapper e outra para implementar SQLWrapper. Em cada projeto você adiciona o código necessário, como instâncias, construtores e assim por diante.

Depois de substituir todas as interfaces que você criou, certifique-se de que todas tenham uma instância na classe que implementa PlatformWrapper (e que haja um getter). Finalmente, você altera o construtor na classe principal. A classe principal é a classe que você faz referência no inicializador. Ele estende ApplicationAdapter, implementa ApplicationListener ou estende Game. Edite o construtor e adicione PlatformWrapper como argumento. Dentro do wrapper da plataforma você tem alguns utilitários (se você adicionou algum) além de todos os outros wrappers (sql, achievements ou qualquer outro que você adicionou).

Agora, se tudo estiver configurado corretamente, você pode fazer uma chamada para o PlatformWrapper e obter qualquer uma das interfaces de plataforma cruzada. Chame qualquer método e (supondo que o código executado esteja correto) você verá em qualquer plataforma, ele faz o que deveria fazer com o código específico da plataforma

Gerente de ativos #

O AssetManager é uma classe que ajuda você a gerenciar seus ativos.

Primeiro, você precisa criar uma instância:

AssetManager am = new AssetManager();

Depois que isso for inicializado e antes de renderizar qualquer coisa, você deseja obter os recursos:

am.load("badlogic.jpg", Texture.class);//Texture.class is the class this asset is of. If it is a 
//sound asset, it doesn't go under Texture. if it is a 3D model, it doesn't go under Texture.class
//Which class added depends on the asset you load

//... other loading here ...//

//when finished, call finishLoading:
am.finishLoading();

Agora, onde quer que você queira obter badlogic.jpg:

Texture texture = am.get("badlogic.jpg");
//Ready to render! The rendering itself is in the normal way

O uso do AssetManager permite carregá-los uma vez na memória do AssetManager e, em seguida, obtê-los quantas vezes quiser.

Por que usar o AssetManager? (do wiki):

AssetManager (código) ajuda você a carregar e gerenciar seus ativos. É a maneira recomendada de carregar seus ativos, devido aos seguintes comportamentos:

  • O carregamento da maioria dos recursos é feito de forma assíncrona, para que você possa exibir uma tela de carregamento reativa enquanto as coisas são carregadas
  • Os ativos são contados por referência. Se dois ativos A e B dependem de outro ativo C, C não será alienado até que A e B tenham sido alienados. Isso também significa que, se você carregar um ativo várias vezes, ele será compartilhado e só ocupará memória uma vez!
  • Um único lugar para armazenar todos os seus ativos.
  • Permite implementar de forma transparente coisas como caches (veja FileHandleResolver abaixo)

O gráfico de cena 2D #

Quando você começa a usar Java ou Android, aprende rapidamente que (0,0) está no canto superior esquerdo. No LibGDX, no entanto, (0,0) está por padrão no canto inferior esquerdo.

Usando uma câmera ortográfica, você pode obter (0, 0) no canto superior esquerdo. Embora por padrão, (0, 0) esteja no canto inferior esquerdo. Isso é algo que é importante saber, pois também muda qual canto das texturas tem as coordenadas X e Y.
