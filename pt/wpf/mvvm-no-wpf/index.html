Modelos e modelos de visualização

A definição de um modelo é frequentemente debatida acaloradamente, e a linha entre um modelo e um modelo de visualização pode ser borrada. Alguns preferem não “poluir” seus modelos com a interface INotifyPropertyChanged e, em vez disso, duplicar as propriedades do modelo no modelo de exibição, que implementa essa interface. Como muitas coisas no desenvolvimento de software, não há resposta certa ou errada. Seja pragmático e faça o que parecer certo.

Ver Separação

A intenção do MVVM é separar essas três áreas distintas - Model, view-model e View. Embora seja aceitável que a exibição acesse o modelo de exibição (VM) e (indiretamente) o modelo, a regra mais importante com o MVVM é que a VM não deve ter acesso à exibição ou seus controles. A VM deve expor tudo o que a visualização precisa, por meio de propriedades públicas. A VM não deve expor ou manipular diretamente controles de interface do usuário, como TextBox, Button, etc.

Em alguns casos, essa separação estrita pode ser difícil de trabalhar, especialmente se você precisar colocar algumas funcionalidades complexas de interface do usuário em funcionamento. Aqui, é perfeitamente aceitável recorrer ao uso de eventos e manipuladores de eventos no arquivo “code-behind” da visualização. Se for puramente uma funcionalidade de interface do usuário, use eventos na exibição. Também é aceitável que esses manipuladores de eventos chamem métodos públicos na instância da VM - apenas não passe referências a controles de interface do usuário ou algo assim.

Comando de Relé

Infelizmente, a classe RelayCommand usada neste exemplo não faz parte da estrutura do WPF (deveria ser!), mas você a encontrará na caixa de ferramentas de quase todos os desenvolvedores do WPF. Uma pesquisa rápida online revelará muitos trechos de código que você pode levantar para criar o seu próprio.

Uma alternativa útil ao RelayCommand é o ActionCommand que é fornecido como parte do Microsoft.Expression.Interactivity.Core que fornece funcionalidade comparável.

O Modelo #

O modelo é o primeiro “M” em MVVM. O modelo geralmente é uma classe que contém os dados que você deseja expor por meio de algum tipo de interface do usuário.

Aqui está uma classe de modelo muito simples, expondo algumas propriedades: -

public class Customer : INotifyPropertyChanged
{
    private string _forename;
    private string _surname;
    private bool _isValid;

    public event PropertyChangedEventHandler PropertyChanged;

    /// <summary>
    /// Customer forename.
    /// </summary>
    public string Forename
    {
        get
        {
            return _forename;
        }
        set
        {
            if (_forename != value)
            {
                _forename = value;
                OnPropertyChanged();
                SetIsValid();
            }
        }
    }

    /// <summary>
    /// Customer surname.
    /// </summary>
    public string Surname
    {
        get
        {
            return _surname;
        }
        set
        {
            if (_surname != value)
            {
                _surname = value;
                OnPropertyChanged();
                SetIsValid();
            }
        }
    }

    /// <summary>
    /// Indicates whether the model is in a valid state or not.
    /// </summary>
    public bool IsValid
    {
        get
        {
            return _isValid;
        }
        set
        {
            if (_isValid != value)
            {
                _isValid = value;
                OnPropertyChanged();
            }
        }
    }

    /// <summary>
    /// Sets the value of the IsValid property.
    /// </summary>
    private void SetIsValid()
    {
        IsValid = !string.IsNullOrEmpty(Forename) && !string.IsNullOrEmpty(Surname);
    }

    /// <summary>
    /// Raises the PropertyChanged event.
    /// </summary>
    /// <param name="propertyName">Name of the property.</param>
    private void OnPropertyChanged([CallerMemberName] string propertyName = "")
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}

Esta classe implementa a interface INotifyPropertyChanged que expõe um evento PropertyChanged. Este evento deve ser gerado sempre que um dos valores da propriedade for alterado - você pode ver isso em ação no código acima. O evento PropertyChanged é uma peça chave nos mecanismos de vinculação de dados do WPF, pois sem ele, a interface do usuário não seria capaz de refletir as alterações feitas no valor de uma propriedade.

O modelo também contém uma rotina de validação muito simples que é chamada pelos setters de propriedades. Ele define uma propriedade pública indicando se o modelo está ou não em um estado válido. Incluí essa funcionalidade para demonstrar um recurso “especial” dos comandos do WPF, que você verá em breve. A estrutura do WPF fornece várias abordagens mais sofisticadas para validação, mas elas estão fora do escopo deste artigo.

A vista #

A View é o “V” em MVVM. Esta é a sua interface de usuário. Você pode usar o designer de arrastar e soltar do Visual Studio, mas a maioria dos desenvolvedores acaba codificando o XAML bruto - uma experiência semelhante à escrita de HTML.

Aqui está o XAML de uma visualização simples para permitir a edição de um modelo Cliente. Em vez de criar uma nova visualização, isso pode ser colado no arquivo MainWindow.xaml de um projeto WPF, entre as tags <Window ...> e </Window>:-

<StackPanel Orientation="Vertical"
            VerticalAlignment="Top"
            Margin="20">
    <Label Content="Forename"/>
    <TextBox Text="{Binding CustomerToEdit.Forename}"/>

    <Label Content="Surname"/>
    <TextBox Text="{Binding CustomerToEdit.Surname}"/>

    <Button Content="Apply Changes"
            Command="{Binding ApplyChangesCommand}" />
</StackPanel>

Este código cria um formulário de entrada de dados simples que consiste em dois TextBoxes - um para o nome próprio do cliente e outro para o sobrenome. Há um ‘Label’ acima de cada ‘TextBox’ e um ‘Aplicar’ ‘Botão’ na parte inferior do formulário.

Localize o primeiro TextBox e observe sua propriedade Text:

Text="{Binding CustomerToEdit.Forename}"

Em vez de definir o texto do TextBox para um valor fixo, essa sintaxe especial de chaves está vinculando o texto ao “caminho” CustomerToEdit.Forename. A que esse caminho é relativo? É o “contexto de dados” da visão - neste caso, nosso modelo de visão. O caminho de ligação, como você pode descobrir, é a propriedade CustomerToEdit do view-model, que é do tipo Customer que por sua vez expõe uma propriedade chamada Forename - daí a notação de caminho “pontilhado”.

Da mesma forma, se você observar o XAML do Button, ele tem um Command que está vinculado à propriedade ApplyChangesCommand do modelo de exibição. Isso é tudo o que é necessário para conectar um botão ao comando da VM.

O contexto de dados

Então, como você define o modelo de exibição para ser o contexto de dados da exibição? Uma maneira é defini-lo no “code-behind” da exibição. Pressione F7 para ver este arquivo de código e adicione uma linha ao construtor existente para criar uma instância do modelo de exibição e atribuí-lo à propriedade DataContext da janela. Deve ficar assim:

    public MainWindow()
    {
        InitializeComponent();

        // Our new line:-
        DataContext = new CustomerEditViewModel();
    }

Em sistemas do mundo real, outras abordagens são frequentemente usadas para criar o modelo de exibição, como injeção de dependência ou estruturas MVVM.

O modelo de visualização #

O modelo de visualização é a “VM” em MVVM. Esta é uma classe que atua como um intermediário, expõe o(s) modelo(s) à interface do usuário (visualização) e lida com solicitações da visualização, como comandos gerados por cliques de botão. Aqui está um modelo de visualização básico:

public class CustomerEditViewModel
{
    /// <summary>
    /// The customer to edit.
    /// </summary>
    public Customer CustomerToEdit { get; set; }

    /// <summary>
    /// The "apply changes" command
    /// </summary>
    public ICommand ApplyChangesCommand { get; private set; }

    /// <summary>
    /// Constructor
    /// </summary>
    public CustomerEditViewModel()
    {
        CustomerToEdit = new Customer
                         {
                             Forename = "John",
                             Surname = "Smith"
                         };

        ApplyChangesCommand = new RelayCommand(
            o => ExecuteApplyChangesCommand(), 
            o => CustomerToEdit.IsValid);
    }

    /// <summary>
    /// Executes the "apply changes" command.
    /// </summary>
    private void ExecuteApplyChangesCommand()
    {
        // E.g. save your customer to database
    }
}

O construtor cria um objeto de modelo Customer e o atribui à propriedade CustomerToEdit, para que fique visível para a visualização.

O construtor também cria um objeto RelayCommand e o atribui à propriedade ApplyChangesCommand, tornando-o novamente visível para a visão. Os comandos do WPF são usados ​​para lidar com solicitações da exibição, como cliques de botão ou item de menu.

O RelayCommand recebe dois parâmetros - o primeiro é o delegado que é chamado quando o comando é executado (por exemplo, em resposta a um clique de botão). O segundo parâmetro é um delegado que retorna um valor booleano indicando se o comando pode ser executado; neste exemplo, ele está conectado à propriedade IsValid do objeto cliente. Quando retorna false, desativa o botão ou item de menu vinculado a esse comando (outros controles podem se comportar de maneira diferente). Este é um recurso simples, mas eficaz, evitando a necessidade de escrever código para habilitar ou desabilitar controles com base em diferentes condições.

Se você colocar este exemplo em funcionamento, tente esvaziar um dos TextBoxes (para colocar o modelo Customer em um estado inválido). Ao sair do TextBox, você deve descobrir que o botão “Aplicar” fica desabilitado.

Observação sobre a criação do cliente

O modelo de exibição não implementa INotifyPropertyChanged (INPC). Isso significa que se um objeto Customer diferente fosse atribuído à propriedade CustomerToEdit, os controles da visualização não mudariam para refletir o novo objeto - o TextBoxes ainda conteria o nome e sobrenome do cliente anterior .

O código de exemplo funciona porque o Customer é criado no construtor do view-model, antes de ser atribuído ao DataContext da view (no ponto em que as ligações são conectadas). Em um aplicativo do mundo real, você pode estar recuperando clientes de um banco de dados em métodos diferentes do construtor. Para suportar isso, a VM deve implementar o INPC, e a propriedade CustomerToEdit deve ser alterada para usar o padrão getter e setter “extended” que você vê no código do modelo de exemplo, gerando o evento PropertyChanged no setter.

O ApplyChangesCommand do view-model não precisa implementar o INPC, pois é muito improvável que o comando mude. Você precisaria implementar este padrão se estivesse criando o comando em algum lugar diferente do construtor, por exemplo, algum tipo de método Initialize().

A regra geral é: implemente INPC se a propriedade estiver vinculada a qualquer controle de visualização e o valor da propriedade puder ser alterado em qualquer lugar que não seja no construtor. Você não precisa implementar o INPC se o valor da propriedade só for atribuído no construtor (e você economizará digitando no processo).

Exemplo básico de MVVM usando WPF e C# #

Este é um exemplo básico para usar o modelo MVVM em um aplicativo de área de trabalho do Windows, usando WPF e C#. O código de exemplo implementa uma caixa de diálogo simples de “informações do usuário”.

O XAML

<Grid>
    <Grid.ColumnDefinitions>
        <ColumnDefinition Width="Auto"/>
        <ColumnDefinition Width="*"/>
    </Grid.ColumnDefinitions>
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto"/>
        <RowDefinition Height="Auto"/>
         <RowDefinition Height="Auto"/>
         <RowDefinition Height="Auto"/>
     </Grid.RowDefinitions>

    <!-- UpdateSourceTrigger=PropertyChanged makes sure that changes in the TextBoxes are immediately applied to the model. -->
    <TextBox Grid.Column="1" Grid.Row="1" Margin="4" Text="{Binding FirstName, UpdateSourceTrigger=PropertyChanged}" HorizontalAlignment="Left" Width="200"/>

    <Label Grid.Column="0" Grid.Row="2" Margin="4" Content="Last Name:" HorizontalAlignment="Right"/>
    <TextBox Grid.Column="1" Grid.Row="2" Margin="4" Text="{Binding LastName, UpdateSourceTrigger=PropertyChanged}" HorizontalAlignment="Left"  Width="200"/>

    <Label Grid.Column="0" Grid.Row="3" Margin="4" Content="Age:" HorizontalAlignment="Right"/>
    <TextBlock Grid.Column="1" Grid.Row="3" Margin="4" Text="{Binding Age}" HorizontalAlignment="Left"/>

public partial class MainWindow : Window
{
    private readonly MyViewModel _viewModel;

    public MainWindow() {
        InitializeComponent();
        _viewModel = new MyViewModel();
        // The DataContext serves as the starting point of Binding Paths
        DataContext = _viewModel;

O modelo de visualização

// INotifyPropertyChanged notifies the View of property changes, so that Bindings are updated.
sealed class MyViewModel : INotifyPropertyChanged
{
    private User user;
    
    public string FirstName { 
        get {return user.FirstName;} 
        set {
            if(user.FirstName != value) {
                user.FirstName = value;
                OnPropertyChange("FirstName");
                // If the first name has changed, the FullName property needs to be udpated as well.
                OnPropertyChange("FullName");
            }
        }
    }

    public string LastName {
        get { return user.LastName; }
        set {
            if (user.LastName != value) {
                user.LastName = value;
                OnPropertyChange("LastName");
                // If the first name has changed, the FullName property needs to be udpated as well.
                OnPropertyChange("FullName");
            }
        }
    }
    
    // This property is an example of how model properties can be presented differently to the View.
    // In this case, we transform the birth date to the user's age, which is read only.
    public int Age { 
        get {
            DateTime today = DateTime.Today;
            int age = today.Year - user.BirthDate.Year;
            if (user.BirthDate > today.AddYears(-age)) age--;
            return age;
        }
    }

    // This property is just for display purposes and is a composition of existing data.
    public string FullName {
        get { return FirstName + " " + LastName; }
    }

    public MyViewModel() {
        user = new User {
            FirstName = "John",
            LastName = "Doe",
            BirthDate = DateTime.Now.AddYears(-30)
        };
    }

    public event PropertyChangedEventHandler PropertyChanged;

    protected void OnPropertyChange(string propertyName) {
        if(PropertyChanged != null) {
            PropertyChanged(this, new PropertyChangedEventArgs(propertyName));
        }
    }
}

Comandando no MVVM #

Os comandos são usados ​​para lidar com Events no WPF, respeitando o padrão MVVM.

Um EventHandler normal ficaria assim (localizado em Code-Behind):

public MainWindow()
{
    _dataGrid.CollectionChanged += DataGrid_CollectionChanged;
}

private void DataGrid_CollectionChanged(object sender, System.Collections.Specialized.NotifyCollectionChangedEventArgs e)
{
        //Do what ever
}

Não para fazer o mesmo no MVVM usamos Commands:

 <Button Command="{Binding Path=CmdStartExecution}" Content="Start" />

Eu recomendo usar algum tipo de prefixo (Cmd) para suas propriedades de comando, porque você precisará delas principalmente em xaml - assim elas são mais fáceis de reconhecer.

Como é MVVM, você deseja manipular esse comando (para Button “eq” Button_Click) em seu ViewModel.

Para isso precisamos basicamente de duas coisas:

  1. Sistema.Windows.Input.ICommand
  2. RelayCommand (por exemplo, retirado de aqui.

Um exemplo simples pode ser assim:

private RelayCommand _commandStart;
public ICommand CmdStartExecution
{
   get
   {
      if(_commandStart == null)
      {
          _commandStart = new RelayCommand(param => Start(), param => CanStart());
      }
      return _commandStart;
   }
}

public void Start()
{
   //Do what ever
}

public bool CanStart()
{
    return (DateTime.Now.DayOfWeek == DayOfWeek.Monday); //Can only click that button on mondays.
}

Então, o que isso está fazendo em detalhes:

O ICommand é o que o Control em xaml está vinculado. O RelayCommand roteará seu comando para uma Ação (ou seja, chamar um Método). O Null-Check apenas garante que cada Command seja inicializado apenas uma vez (devido a problemas de desempenho). Se você leu o link para o RelayCommand acima, deve ter notado que o RelayCommand tem duas sobrecargas para seu construtor. (Action<object> execute) e (Action<object> execute, Predicate<object> canExecute).

Isso significa que você pode (adicionalmente) adicionar um segundo Method retornando um bool para dizer que o Control se o “Evento” pode disparar ou não.

Uma coisa boa para isso é que Buttons por exemplo será Enabled="false" se o Method retornar false

CommandParameters

<DataGrid x:Name="TicketsDataGrid">
    <DataGrid.InputBindings>
        <MouseBinding Gesture="LeftDoubleClick" 
                      Command="{Binding CmdTicketClick}" 
                      CommandParameter="{Binding ElementName=TicketsDataGrid, 
                                                 Path=SelectedItem}" />
    </DataGrid.InputBindings>
<DataGrid />

Neste exemplo eu quero passar o DataGrid.SelectedItem para o Click_Command no meu ViewModel.

Seu método deve ficar assim enquanto a implementação do ICommand permanece como acima.

private RelayCommand _commandTicketClick;

public ICommand CmdTicketClick
{
   get
   {
       if(_commandTicketClick == null)
       {
           _commandTicketClick = new RelayCommand(param => HandleUserClick(param));
       }
       return _commandTicketClick;
   }
}

private void HandleUserClick(object item)
{
    MyModelClass selectedItem = item as MyModelClass;
    if (selectedItem != null)
    {
        //Do sth. with that item
    }
}
