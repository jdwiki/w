Sintaxe #

  • {Binding PropertyName} é equivalente a {Binding Path=PropertyName}
  • {Binding Path=SomeProperty.SomeOtherProperty.YetAnotherProperty}
  • {Binding Path=SomeListProperty1}

Parâmetros #

Parâmetro Detalhes
Caminho Especifica o caminho para vincular. Se não especificado, vincula-se ao próprio DataContext.
UpdateSourceTrigger Especifica quando a origem da ligação tem seu valor atualizado. O padrão é LostFocus. O valor mais usado é PropertyChanged.
Modo Normalmente OneWay ou TwoWay. Se não for especificado pela vinculação, o padrão será OneWay, a menos que o destino da vinculação solicite que seja TwoWay. Ocorre um erro quando TwoWay é usado para vincular a uma propriedade somente leitura, por exemplo. OneWay deve ser definido explicitamente ao vincular uma propriedade de string somente leitura a TextBox.Text.
Fonte Permite usar um StaticResource como fonte de ligação em vez do DataContext atual.
RelativeSource Permite usar outro elemento XAML como fonte de associação em vez do DataContext atual.
ElementName Permite usar um elemento XAML nomeado como uma fonte de associação em vez do DataContext atual.
FallbackValue Se a associação falhar, esse valor será fornecido ao destino da associação.
TargetNullValue Se o valor de origem da vinculação for null, esse valor será fornecido ao destino da vinculação.
Conversor Especifica o conversor StaticResource que é usado para converter o valor da associação, por exemplo, converter um booleano em um item de enumeração Visibility.
ConverterParameter Especifica um parâmetro opcional a ser fornecido ao conversor. Esse valor deve ser estático e não pode ser vinculado.
StringFormat Especifica uma string de formato a ser usada ao exibir o valor vinculado.
Atraso (WPF 4.5+) Especifica um atraso em milissegundos para a vinculação atualizar o BindingSource no ViewModel. Isso deve ser usado com Mode=TwoWay e UpdateSourceTrigger=PropertyChanged para ter efeito.

UpdateSourceTrigger #

Por padrão, o WPF atualiza a fonte de associação quando o controle perde o foco. No entanto, se houver apenas um controle que pode obter foco - algo comum em exemplos - você precisará especificar UpdateSourceTrigger=PropertyChanged para que as atualizações funcionem.

Você vai querer usar PropertyChanged como o gatilho em muitas ligações bidirecionais, a menos que atualizar a fonte de ligação em cada pressionamento de tecla seja caro ou a validação de dados ao vivo seja indesejável.

Usar LostFocus tem um efeito colateral infeliz: pressionar enter para enviar um formulário usando um botão marcado como IsDefault não atualiza a propriedade que suporta sua ligação, efetivamente desfazendo suas alterações. Felizmente, existem algumas soluções alternativas.

Observe também que, ao contrário do UWP, o WPF (4.5+) também possui a propriedade Delay nas associações, o que pode ser suficiente para algumas associações com configurações de inteligência apenas locais ou simples, como algumas validações de TextBox.

Converta um valor booleano em valor de visibilidade #

Este exemplo oculta a caixa vermelha (borda) se a caixa de seleção não estiver marcada usando um IValueConverter.

Observação: O BooleanToVisibilityConverter usado no exemplo abaixo é um conversor de valor interno, localizado no namespace System.Windows.Controls.

XAML:

<Window x:Class="StackOverflowDataBindingExample.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="MainWindow" Height="350" Width="525">
    <Window.Resources>
        <BooleanToVisibilityConverter x:Key="VisibleIfTrueConverter" />
    </Window.Resources>
    <StackPanel>
        <CheckBox x:Name="MyCheckBox"
                  IsChecked="True" />
        <Border Background="Red" Width="20" Height="20"
                Visibility="{Binding Path=IsChecked,ElementName=MyCheckBox, Converter={StaticResource VisibleIfTrueConverter}}" />
    </StackPanel>
</Window>

Implementando INotifyPropertyChanged #

INotifyPropertyChanged é uma interface usada por fontes de associação (ou seja, o DataContext) para permitir que a interface do usuário ou outros componentes saibam que uma propriedade foi alterada. O WPF atualiza automaticamente a interface do usuário para você quando vê o evento PropertyChanged gerado. É desejável ter esta interface implementada em uma classe base da qual todos os seus viewmodels possam herdar.

Em C# 6, isso é tudo que você precisa:

public abstract class ViewModelBase : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler PropertyChanged;

    protected void NotifyPropertyChanged([CallerMemberName] string name = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(name));
    }
}

Isso permite que você invoque NotifyPropertyChanged de duas maneiras diferentes:

  1. NotifyPropertyChanged(), que acionará o evento para o setter que o invocar, graças ao atributo CallerMemberName.
  2. NotifyPropertyChanged(nameof(SomeOtherProperty)), que acionará o evento para SomeOtherProperty.

Para .NET 4.5 e superior usando C# 5.0, isso pode ser usado:

public abstract class ViewModelBase : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler PropertyChanged;

    protected void NotifyPropertyChanged([CallerMemberName] string name = null)
    {
        var handler = PropertyChanged;
        if (handler != null)
        {
            handler(this, new PropertyChangedEventArgs(name));
        }
    }
}

Nas versões do .NET anteriores à 4.5, você precisa se contentar com nomes de propriedades como constantes de string ou uma solução usando expressões.

Observação: É possível vincular a uma propriedade de um “objeto C# simples e antigo” (POCO) que não implementa INotifyPropertyChanged e observar que as vinculações funcionam melhor do que o esperado. Este é um recurso oculto no .NET e provavelmente deve ser evitado. Especialmente porque causará vazamentos de memória quando o Mode da ligação não for OneTime (veja aqui).

Vincular à propriedade de um ancestral #

Você pode vincular a uma propriedade de um ancestral na árvore visual usando uma vinculação RelativeSource. O controle mais próximo na árvore visual que tem o mesmo tipo ou é derivado do tipo que você especificar será usado como fonte da associação:

<Grid Background="Blue">
    <Grid Background="Gray" Margin="10">
        <Border Background="Red" Margin="20">
            <StackPanel Background="White" Margin="20">
                <Button Margin="10" Content="Button1" Background="{Binding Background, RelativeSource={RelativeSource Mode=FindAncestor, AncestorType={x:Type Grid}}}" />
                <Button Margin="10" Content="Button2" Background="{Binding Background, RelativeSource={RelativeSource Mode=FindAncestor, AncestorType={x:Type FrameworkElement}}}" />
            </StackPanel>
        </Border>
    </Grid>
</Grid>

Neste exemplo, Button1 tem um fundo cinza porque o ancestral Grid mais próximo tem um fundo cinza. Button2 tem um fundo branco porque o ancestral mais próximo derivado de FrameworkElement é o StackPanel branco.

Vincular à propriedade de outro elemento nomeado #

Você pode associar a uma propriedade em um elemento nomeado, mas o elemento nomeado deve estar no escopo.

<StackPanel>
    <CheckBox x:Name="MyCheckBox" IsChecked="True" />
    <TextBlock Text="{Binding IsChecked, ElementName=MyCheckBox}" />
</StackPanel>

Definindo o DataContext #

Para trabalhar com ligações no WPF, você precisa definir um DataContext. O DataContext informa às associações de onde obter seus dados por padrão.

<Window x:Class="StackOverflowDataBindingExample.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:d="http://schemas.microsoft.com/expression/blend/2008"
        xmlns:mc="http://schemas.openxmlformats.org/markup-compatibility/2006"
        xmlns:local="clr-namespace:StackOverflowDataBindingExample"
        xmlns:vm="clr-namespace:StackOverflowDataBindingExample.ViewModels"
        mc:Ignorable="d"
        Title="MainWindow" Height="350" Width="525">
    <Window.DataContext>
        <vm:HelloWorldViewModel />
    </Window.DataContext>
    ...
</Window>

Você também pode definir o DataContext por meio de code-behind, mas vale a pena observar que o XAML IntelliSense é um pouco exigente: um DataContext fortemente tipado deve ser definido em XAML para que o IntelliSense sugira propriedades disponíveis para associação.

/// <summary>
/// Interaction logic for MainWindow.xaml
/// </summary>
public partial class MainWindow : Window
{
    public MainWindow()
    {
        InitializeComponent();
        DataContext = new HelloWorldViewModel();
    }
}

Embora existam estruturas para ajudá-lo a definir seu DataContext de maneira mais flexível (por exemplo, MVVM Light possui um localizador de viewmodel que usa inversão de controle), usamos o método rápido e sujo para os fins deste tutorial.

Você pode definir um DataContext para praticamente qualquer elemento visual no WPF. O DataContext geralmente é herdado de ancestrais na árvore visual, a menos que tenha sido substituído explicitamente, por exemplo. dentro de um ContentPresenter.

Vinculando vários valores com um MultiBinding #

O MultiBinding permite vincular vários valores à mesma propriedade. No exemplo a seguir, vários valores são associados à propriedade Text de uma Textbox e formatados usando a propriedade StringFormat.

<TextBlock>
    <TextBlock.Text>
        <MultiBinding StringFormat="{}{0} {1}">
            <Binding Path="User.Forename"/>
            <Binding Path="User.Surname"/>
        </MultiBinding>
    </TextBlock.Text>
</TextBlock>

Além de StringFormat, um IMultiValueConverter também pode ser usado para converter os valores de Bindings em um valor para o destino do MultiBinding.

No entanto, MultiBindings não pode ser aninhado.
