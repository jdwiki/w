Olá Recursos #

O WPF apresenta um conceito muito útil: a capacidade de armazenar dados como um recurso, seja localmente para um controle, localmente para toda a janela ou globalmente para todo o aplicativo. Os dados podem ser praticamente o que você quiser, desde informações reais até uma hierarquia de controles WPF. Isso permite que você coloque os dados em um lugar e depois os use de ou vários outros lugares, o que é muito útil. O conceito é muito usado para estilos e modelos.

<Window x:Class="WPFApplication.ResourceSample"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:sys="clr-namespace:System;assembly=mscorlib"
        Title="ResourceSample" Height="150" Width="350">
    <Window.Resources>
        <sys:String x:Key="strHelloWorld">Hello, world!</sys:String>
    </Window.Resources>
    <StackPanel Margin="10">
        <TextBlock Text="{StaticResource strHelloWorld}" FontSize="56" />
        <TextBlock>Just another "<TextBlock Text="{StaticResource strHelloWorld}" />" example, but with resources!</TextBlock>
    </StackPanel>
</Window>

Os recursos recebem uma chave, usando o atributo x:Key, que permite referenciá-lo de outras partes do aplicativo usando essa chave, em combinação com a extensão de marcação StaticResource. Neste exemplo, apenas armazeno uma string simples, que uso de dois controles TextBlock diferentes.

Tipos de recursos #

Compartilhar uma string simples foi fácil, mas você pode fazer muito mais. Neste exemplo, também armazenarei uma matriz completa de strings, juntamente com um pincel de gradiente a ser usado para o plano de fundo. Isso deve lhe dar uma boa ideia de quanto você pode fazer com os recursos:

<Window x:Class="WPFApplication.ExtendedResourceSample"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:sys="clr-namespace:System;assembly=mscorlib"
        Title="ExtendedResourceSample" Height="160" Width="300"
        Background="{DynamicResource WindowBackgroundBrush}">
    <Window.Resources>
        <sys:String x:Key="ComboBoxTitle">Items:</sys:String>

        <x:Array x:Key="ComboBoxItems" Type="sys:String">
            <sys:String>Item #1</sys:String>
            <sys:String>Item #2</sys:String>
            <sys:String>Item #3</sys:String>
        </x:Array>

        <LinearGradientBrush x:Key="WindowBackgroundBrush">
            <GradientStop Offset="0" Color="Silver"/>
            <GradientStop Offset="1" Color="Gray"/>
        </LinearGradientBrush>
    </Window.Resources>
    <StackPanel Margin="10">
        <Label Content="{StaticResource ComboBoxTitle}" />
        <ComboBox ItemsSource="{StaticResource ComboBoxItems}" />
    </StackPanel>
</Window>

Desta vez, adicionamos alguns recursos extras, para que nossa janela agora contenha uma string simples, uma matriz de strings e um LinearGradientBrush. A string é usada para o rótulo, a matriz de strings é usada como itens para o controle ComboBox e o pincel de gradiente é usado como plano de fundo para toda a janela. Então, como você pode ver, praticamente qualquer coisa pode ser armazenada como um recurso.

Recursos locais e amplos de aplicativos #

Se você precisar apenas de um determinado recurso para um controle específico, poderá torná-lo mais local adicionando-o a esse controle específico, em vez da janela. Funciona exatamente da mesma maneira, a única diferença é que agora você só pode acessar de dentro do escopo do controle onde o colocou:

<StackPanel Margin="10">
    <StackPanel.Resources>
        <sys:String x:Key="ComboBoxTitle">Items:</sys:String>
    </StackPanel.Resources>
    <Label Content="{StaticResource ComboBoxTitle}" />
</StackPanel>

Nesse caso, adicionamos o recurso ao StackPanel e o usamos a partir de seu controle filho, o Label. Outros controles dentro do StackPanel também poderiam tê-lo usado, assim como os filhos desses controles filho poderiam acessá-lo. Os controles fora deste StackPanel específico não teriam acesso a ele.

Se você precisar acessar o recurso de várias janelas, isso também é possível. O arquivo App.xaml pode conter recursos como a janela e qualquer tipo de controle WPF e, quando você os armazena em App.xaml, eles ficam globalmente acessíveis em todas as janelas e controles de usuário do projeto. Funciona exatamente da mesma maneira que ao armazenar e usar a partir de uma janela:

<Application x:Class="WpfSamples.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:sys="clr-namespace:System;assembly=mscorlib"
             StartupUri="WPFApplication/ExtendedResourceSample.xaml">
    <Application.Resources>
        <sys:String x:Key="ComboBoxTitle">Items:</sys:String>
    </Application.Resources>
</Application>

Usá-lo também é o mesmo - o WPF subirá automaticamente o escopo, do controle local para a janela e depois para App.xaml, para encontrar um determinado recurso:

<Label Content="{StaticResource ComboBoxTitle}" />

Recursos do Code-behind #

Neste exemplo, acessaremos três recursos diferentes do Code-behind, cada um armazenado em um escopo diferente

App.xaml:

<Application x:Class="WpfSamples.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:sys="clr-namespace:System;assembly=mscorlib"
             StartupUri="WPFApplication/ResourcesFromCodeBehindSample.xaml">
    <Application.Resources>
        <sys:String x:Key="strApp">Hello, Application world!</sys:String>
    </Application.Resources>
</Application>

Janela:

<Window x:Class="WpfSamples.WPFApplication.ResourcesFromCodeBehindSample"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:sys="clr-namespace:System;assembly=mscorlib"
        Title="ResourcesFromCodeBehindSample" Height="175" Width="250">
    <Window.Resources>
        <sys:String x:Key="strWindow">Hello, Window world!</sys:String>
    </Window.Resources>
    <DockPanel Margin="10" Name="pnlMain">
        <DockPanel.Resources>
            <sys:String x:Key="strPanel">Hello, Panel world!</sys:String>
        </DockPanel.Resources>

        <WrapPanel DockPanel.Dock="Top" HorizontalAlignment="Center" Margin="10">
            <Button Name="btnClickMe" Click="btnClickMe_Click">Click me!</Button>
        </WrapPanel>

        <ListBox Name="lbResult" />
    </DockPanel>
</Window>

Código por trás:

using System;
using System.Windows;

namespace WpfSamples.WPFApplication
{
        public partial class ResourcesFromCodeBehindSample : Window
        {
                public ResourcesFromCodeBehindSample()
                {
                        InitializeComponent();
                }

                private void btnClickMe_Click(object sender, RoutedEventArgs e)
                {
                        lbResult.Items.Add(pnlMain.FindResource("strPanel").ToString());
                        lbResult.Items.Add(this.FindResource("strWindow").ToString());
                        lbResult.Items.Add(Application.Current.FindResource("strApp").ToString());
                }
        }
}

Então, como você pode ver, nós armazenamos três diferentes “Hello, world!” mensagens: Uma em App.xaml, uma dentro da janela e uma localmente para o painel principal. A interface consiste em um botão e um ListBox.

No Code-behind, tratamos do evento click do botão, no qual adicionamos cada uma das strings de texto ao ListBox, conforme visto na captura de tela. Usamos o método FindResource(), que retornará o recurso como um objeto (se encontrado), e então o transformamos na string que sabemos que é usando o método ToString().

Observe como usamos o método FindResource() em diferentes escopos - primeiro no painel, depois na janela e depois no objeto Application atual. Faz sentido procurar o recurso onde sabemos que ele está, mas, como já mencionado, se um recurso não for encontrado, a pesquisa avança na hierarquia, portanto, em princípio, poderíamos ter usado o método FindResource() no painel em todos os três casos, pois teria continuado até a janela e, posteriormente, até o nível do aplicativo, se não fosse encontrado.

O mesmo não acontece ao contrário - a pesquisa não navega para baixo na árvore, portanto, você não pode começar a procurar um recurso no nível do aplicativo, se tiver sido definido localmente para o controle ou para a janela.
