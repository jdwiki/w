Ambiente de desenvolvimento e produção usando diferentes fontes de dados #

Após a configuração bem-sucedida do aplicativo Spring-Boot, toda a configuração é tratada em um arquivo application.properties. Você encontrará o arquivo em src/main/resources/.

Normalmente é necessário ter um banco de dados por trás do aplicativo. Para desenvolvimento é bom ter uma configuração de ambientes dev e prod. Usando vários arquivos application.properties você pode dizer ao Spring-Boot com qual ambiente o aplicativo deve iniciar.

Um bom exemplo é configurar dois bancos de dados. Um para dev e outro para productive.

Para o ambiente dev você pode usar um banco de dados na memória como H2. Crie um novo arquivo no diretório src/main/resources/ chamado application-dev.properties. Dentro do arquivo está a configuração do banco de dados in-memory:

spring.datasource.url=jdbc:h2:mem:test
spring.datasource.driverClassName=org.h2.Driver
spring.datasource.username=sa
spring.datasource.password=

Para o ambiente prod nos conectaremos a um banco de dados “real” por exemplo postgreSQL. Crie um novo arquivo no diretório src/main/resources/ chamado application-prod.properties. Dentro do arquivo está a configuração do banco de dados postgreSQL:

spring.datasource.url= jdbc:postgresql://localhost:5432/yourDB
spring.datasource.username=postgres
spring.datasource.password=secret

Em seu arquivo application.properties padrão, agora você pode definir qual perfil é ativado e usado pelo Spring-Boot. Basta definir um atributo dentro:

spring.profiles.active=dev

ou

spring.profiles.active=prod

Importante é que a parte após - em application-dev.properties é o identificador do arquivo.

Agora você pode iniciar o aplicativo Spring-Boot no modo de desenvolvimento ou produção apenas alterando o identificador. Um banco de dados na memória será inicializado ou a conexão com um banco de dados “real”. Claro que também há muito mais casos de uso para ter vários arquivos de propriedades.

Defina o perfil de mola certo construindo o aplicativo automaticamente (maven) #

Ao criar vários arquivos de propriedades para os diferentes ambientes ou casos de uso, às vezes é difícil alterar manualmente o valor de active.profile para o correto. Mas existe uma maneira de definir o active.profile no arquivo application.properties enquanto cria o aplicativo usando maven-profiles.

Digamos que existam três arquivos de propriedades de ambientes em nosso aplicativo:

application-dev.properties:

spring.profiles.active=dev
server.port=8081

application-test.properties:

spring.profiles.active=test
server.port=8082

application-prod.properties.

spring.profiles.active=prod
server.port=8083

Esses três arquivos diferem apenas na porta e no nome do perfil ativo.

No arquivo principal application.properties definimos nosso perfil de mola usando uma variável maven:

aplicativo.propriedades.

spring.profiles.active=@profileActive@

Depois disso, basta adicionar os perfis do maven em nosso pom.xml. Vamos definir os perfis para os três ambientes:

 <profiles>
        <profile>
            <id>dev</id>
            <activation>
                <activeByDefault>true</activeByDefault>
            </activation>
            <properties>
                <build.profile.id>dev</build.profile.id>
                <profileActive>dev</profileActive>
            </properties>
        </profile>
        <profile>
            <id>test</id>
            <properties>
                <build.profile.id>test</build.profile.id>
                <profileActive>test</profileActive>
            </properties>
        </profile>
        <profile>
            <id>prod</id>
            <properties>
                <build.profile.id>prod</build.profile.id>
                <profileActive>prod</profileActive>
            </properties>
        </profile>
    </profiles>

Agora você pode construir o aplicativo com maven. Se você não definir nenhum perfil maven, está construindo o padrão (neste exemplo é dev). Para especificar um, você deve usar uma palavra-chave maven. A palavra-chave para definir um perfil no maven é -P seguida diretamente pelo nome do perfil: mvn clean install -Ptest.

Agora, você também pode criar compilações personalizadas e salvá-las em seu IDE para compilações mais rápidas.

Exemplos:
