Como testar um aplicativo Spring Boot simples #

Temos um aplicativo Spring boot de amostra que armazena dados do usuário no MongoDB e estamos usando serviços Rest para recuperar dados

Primeiro, há uma classe de domínio, ou seja, POJO

@Document
public class User{
    @Id
    private String id;

    private String name;

}

Um repositório correspondente baseado no Spring Data MongoDB

public interface UserRepository extends MongoRepository<User, String> {
}

Então nosso controlador de usuário

@RestController
class UserController {
 
    @Autowired
    private UserRepository repository;
 
    @RequestMapping("/users")
    List<User> users() {
        return repository.findAll();
    }
 
    @RequestMapping(value = "/Users/{id}", method = RequestMethod.DELETE)
    @ResponseStatus(HttpStatus.NO_CONTENT)
    void delete(@PathVariable("id") String id) {
        repository.delete(id);
    }
 
    // more controller methods
}

E finalmente nosso aplicativo Spring Boot

@SpringBootApplication
public class Application {
    public static void main(String args[]){
     SpringApplication.run(Application.class, args);
    }
}

Se, digamos que John Cena, The Rock e TripleHHH fossem os únicos três usuários no banco de dados, uma solicitação para /users daria a seguinte resposta:

$ curl localhost:8080/users
[{"name":"John Cena","id":"1"},{"name":"The Rock","id":"2"},{"name":"TripleHHH","id":"3"}]

Agora para testar o código vamos verificar se o aplicativo funciona

@RunWith(SpringJUnit4ClassRunner.class)   // 1
@SpringApplicationConfiguration(classes = Application.class)   // 2
@WebAppConfiguration   // 3
@IntegrationTest("server.port:0")   // 4
public class UserControllerTest {

    @Autowired   // 5
    UserRepository repository;

    User cena;
    User rock;
    User tripleHHH;

    @Value("${local.server.port}")   // 6
    int port;

    @Before
    public void setUp() {
        // 7
        cena = new User("John Cena");
        rock = new User("The Rock");
        tripleHHH = new User("TripleHH");

        // 8
        repository.deleteAll();
        repository.save(Arrays.asList(cena, rock, tripleHHH));

        // 9
        RestAssured.port = port;
    }

    // 10
    @Test
    public void testFetchCena() {
        String cenaId = cena.getId();

        when().
                get("/Users/{id}", cenaId).
        then().
                statusCode(HttpStatus.SC_OK).
                body("name", Matchers.is("John Cena")).
                body("id", Matchers.is(cenaId));
    }

    @Test
    public void testFetchAll() {
        when().
                get("/users").
        then().
                statusCode(HttpStatus.SC_OK).
                body("name", Matchers.hasItems("John Cena", "The Rock", "TripleHHH"));
    }

    @Test
    public void testDeletetripleHHH() {
        String tripleHHHId = tripleHHH.getId();

        when()
                .delete("/Users/{id}", tripleHHHId).
        then().
                statusCode(HttpStatus.SC_NO_CONTENT);
    }
}

Explicação

  1. Como qualquer outro teste baseado em Spring, precisamos do SpringJUnit4ClassRunner para que um contexto de aplicação seja criado.
  2. A anotação @SpringApplicationConfiguration é semelhante à anotação @ContextConfiguration, pois é usada para especificar quais contextos de aplicativo devem ser usados ​​no teste. Além disso, ele acionará a lógica para ler configurações, propriedades e assim por diante específicas do Spring Boot.
  3. @WebAppConfiguration deve estar presente para informar ao Spring que um WebApplicationContext deve ser carregado para o teste. Ele também fornece um atributo para especificar o caminho para a raiz do aplicativo da web.
  4. @IntegrationTest é usado para informar ao Spring Boot que o servidor da Web incorporado deve ser iniciado. Ao fornecer pares nome-valor separados por dois pontos ou iguais, qualquer variável de ambiente pode ser substituída. Neste exemplo, o "server.port:0" substituirá a configuração de porta padrão do servidor. Normalmente, o servidor começaria a usar o número de porta especificado, mas o valor 0 tem um significado especial. Quando especificado como 0, ele informa ao Spring Boot para verificar as portas no ambiente do host e iniciar o servidor em uma porta disponível aleatória. Isso é útil se tivermos diferentes serviços ocupando portas diferentes nas máquinas de desenvolvimento e o servidor de compilação que poderia colidir com a porta do aplicativo, caso em que o aplicativo não será iniciado. Em segundo lugar, se criarmos vários testes de integração com diferentes contextos de aplicação, eles também podem colidir se os testes estiverem sendo executados simultaneamente.
  5. Temos acesso ao contexto da aplicação e podemos usar autowiring para injetar qualquer bean Spring.
  6. O @Value("${local.server.port}") será resolvido para o número de porta real que é usado.
  7. Criamos algumas entidades que podemos usar para validação.
  8. O banco de dados MongoDB é limpo e reinicializado para cada teste para que sempre validemos em relação a um estado conhecido. Como a ordem dos testes não está definida, é provável que o teste testFetchAll() falhe se for executado após o teste testDeletetripleHHH().
  9. Instruímos Fique tranquilo para usar a porta correta. É um projeto de código aberto que fornece um Java DSL para testar serviços restful
  10. Os testes são implementados usando Rest Assured. podemos implementar os testes usando o TestRestTemplate ou qualquer outro cliente http, mas eu uso Rest Assured porque podemos escrever uma documentação concisa usando RestDocs

Carregar um arquivo yaml [ou propriedades] diferente ou substituir algumas propriedades #

Quando usamos @SpringApplicationConfiguration ele usará a configuração de application.yml [propriedades] que em certas situações não é apropriada. Então, para substituir as propriedades, podemos usar a anotação @TestPropertySource.

@TestPropertySource(
        properties = {
                "spring.jpa.hibernate.ddl-auto=create-drop",
                "liquibase.enabled=false"
        }
)
@RunWith(SpringJUnit4ClassRunner.class)
@SpringApplicationConfiguration(Application.class)
public class ApplicationTest{

    // ...

}

Podemos usar o atributo properties de @TestPropertySource para substituir as propriedades específicas que queremos. No exemplo acima estamos substituindo a propriedade spring.jpa.hibernate.ddl-auto para create-drop. E liquibase.enabled para false.

Carregando arquivo yml diferente #

Se você quiser carregar um arquivo yml totalmente diferente para teste, você pode usar o atributo locations em @TestPropertySource.

@TestPropertySource(locations="classpath:test.yml")
@RunWith(SpringJUnit4ClassRunner.class)
@SpringApplicationConfiguration(Application.class)
public class ApplicationTest{

    // ...

}

Opções alternativas #

Opção 1:

Você também pode carregar um arquivo yml diferente colocando um arquivo yml no diretório test > resource

Opção 2:

Usando a anotação @ActiveProfiles

@RunWith(SpringJUnit4ClassRunner.class)
@SpringApplicationConfiguration(classes = Application.class)
@ActiveProfiles("somename")
public class MyIntTest{
}

Você pode ver que estamos usando a anotação @ActiveProfiles e estamos passando o somename como o valor.

Crie um arquivo chamado application-somename.yml e o teste carregará este arquivo.
