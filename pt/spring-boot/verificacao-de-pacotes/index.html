Neste tópico, apresentarei uma visão geral da verificação do pacote de inicialização da primavera.

Você pode encontrar algumas informações básicas nos documentos de inicialização de primavera no link a seguir ([using-boot-structuring-your-code](http://docs.spring.io/spring-boot/docs/current/reference/html/using -boot-structuring-your-code.html)), mas tentarei fornecer informações mais detalhadas.

Spring boot, e spring em geral, fornecem um recurso para varrer automaticamente os pacotes para certas anotações para criar beans e configuration.

Parâmetros #

Anotação Detalhes
@SpringBootApplication Anotação do aplicativo de inicialização da mola principal. usado uma vez no aplicativo, contém um método principal e atua como pacote principal para verificação de pacotes
@SpringBootConfiguration Indica que uma classe fornece o aplicativo Spring Boot. Deve ser declarado apenas uma vez na aplicação, geralmente automaticamente definindo @SpringBootApplication
@EnableAutoConfiguration Habilite a configuração automática do Spring Application Context. Deve ser declarado apenas uma vez na aplicação, geralmente automaticamente definindo @SpringBootApplication
@ComponentScan Usado para acionar a verificação automática de pacotes em um determinado pacote e seus filhos ou para definir a verificação personalizada de pacotes
@Configuração Usado para declarar um ou mais métodos @Bean. Pode ser escolhido por varredura automática de pacotes para declarar um ou mais métodos @Bean em vez da configuração xml tradicional
@Bean Indica que um método produz um bean a ser gerenciado pelo contêiner Spring. Normalmente os métodos anotados @Bean serão colocados em classes anotadas @Configuration que serão escolhidas pela varredura de pacotes para criar beans baseados em configuração java.
@Componente Ao declarar uma classe como @Component, ela se torna candidata à detecção automática ao usar configuração baseada em anotação e varredura de caminho de classe. Normalmente uma classe anotada com @Component se tornará um bean na aplicação
@Repositório Originalmente definido por Domain-Driven Design (Evans, 2003) como “um mecanismo para encapsular armazenamento. Geralmente é usado para indicar um ‘Repositório’ para ‘dados de mola’
@Serviço Muito semelhante na prática ao @Component. originalmente definido pelo Domain-Driven Design (Evans, 2003) como “uma operação oferecida como uma interface que fica sozinha no modelo, sem estado encapsulado”.
@Controller Indica que uma classe anotada é um “Controller” (por exemplo, um controlador da web).
@RestController Uma anotação de conveniência que é anotada com @Controller e @ResponseBody. Será selecionado automaticamente por padrão porque contém a anotação @Controller que é selecionada por padrão.

@ComponentScan #

Você pode usar @ComponentScan para configurar uma varredura de pacotes mais complexa. Há também @ComponentScans que atua como uma anotação de contêiner que agrega várias anotações @ComponentScan.

Exemplos básicos de código

@ComponentScan
public class DemoAutoConfiguration {
}

@ComponentScans({@ComponentScan("com.example1"), @ComponentScan("com.example2")})
public class DemoAutoConfiguration {
}

Declarar @ComponentScan sem configuração age como @SpringBootApplication e verifica todos os pacotes sob a classe anotada com esta anotação.

Neste exemplo vou indicar alguns dos atributos úteis de @ComponentScan:

  1. basePackages - pode ser usado para indicar pacotes específicos a serem verificados.
  2. useDefaultFilters - configurando este atributo para false (o padrão é true) você pode ter certeza que o Spring não escaneia @Component, @Repository, @Service ou @Controller automaticamente.
  3. includeFilters - pode ser usado para incluir padrões de regex/anotações de mola específicos para incluir na verificação de pacotes.
  4. excludeFilters - pode ser usado para excluir anotações de mola/padrões regex específicos para incluir na verificação de pacotes.

Existem muitos outros atributos, mas esses são os mais usados ​​para personalizar a verificação de pacotes.

Criando sua própria configuração automática #

Spring boot é baseado em muitos projetos pai de configuração automática pré-fabricados. Você já deve estar familiarizado com os projetos de inicialização por mola.

Você pode criar facilmente seu próprio projeto inicial executando as seguintes etapas fáceis:

  1. Crie algumas classes @Configuration para definir beans padrão. Você deve usar propriedades externas o máximo possível para permitir a personalização e tentar usar anotações auxiliares de configuração automática como @AutoConfigureBefore, @AutoConfigureAfter, @ConditionalOnBean, @ConditionalOnMissingBean etc. Você pode encontrar informações mais detalhadas em cada anotação na documentação oficial [Anotações de condição](http://docs.spring.io/spring-boot/docs/current/reference/html/boot-features-developing-auto-configuration.html#boot-features-condition -anotações)

  2. Coloque um arquivo/arquivos de autoconfiguração que agregue todas as classes @Configuration.

  3. Crie um arquivo chamado spring.factories e coloque-o em src/main/resources/META-INF.

  4. Em spring.factories, defina a propriedade org.springframework.boot.autoconfigure.EnableAutoConfiguration com valores separados por vírgula de suas classes @Configuration:

    org.springframework.boot.autoconfigure.EnableAutoConfiguration=
    com.mycorp.libx.autoconfigure.LibXAutoConfiguration,
    com.mycorp.libx.autoconfigure.LibXWebAutoConfiguration Usando este método, você pode criar suas próprias classes de configuração automática que serão escolhidas pelo spring-boot. O Spring-boot verifica automaticamente todas as dependências do maven/gradle em busca de um arquivo spring.factories, se encontrar um, ele adiciona todas as classes @Configuration especificadas nele ao seu processo de autoconfiguração.

Certifique-se de que seu projeto inicial auto-configuration não contém spring boot maven plugin porque ele irá empacotar o projeto como um JAR executável e não será carregado pelo classpath como pretendido - o spring boot não poderá encontrar seu spring.factories e não carregará sua configuração

@SpringBootApplication #

A maneira mais básica de estruturar seu código usando spring boot para uma boa varredura automática de pacotes é usando a anotação @SpringBootApplication. Esta anotação fornece em si 3 outras anotações que ajudam na verificação automática: @SpringBootConfiguration, @EnableAutoConfiguration, @ComponentScan (mais informações sobre cada anotação na seção Parameters).

@SpringBootApplication normalmente será colocado no pacote principal e todos os outros componentes serão colocados em pacotes sob este arquivo:

com
 +- example
     +- myproject
         +- Application.java (annotated with @SpringBootApplication)
         |
         +- domain
         |   +- Customer.java
         |   +- CustomerRepository.java
         |
         +- service
         |   +- CustomerService.java
         |
         +- web
             +- CustomerController.java

A menos que mencionado de outra forma, o spring boot detecta as anotações @Configuration, @Component, @Repository, @Service, @Controller, @RestController automaticamente nos pacotes verificados (@Configuration e @ RestController estão sendo escolhidos porque são anotados por @Component e @Controller de acordo).

Exemplo de código básico:

@SpringBootApplication
public class Application {

    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
    
}

Definindo pacotes/classes explicitamente

Desde a versão 1.3 você também pode dizer ao spring boot para escanear pacotes específicos configurando scanBasePackages ou scanBasePackageClasses em @SpringBootApplication em vez de especificar @ComponentScan.

  1. @SpringBootApplication(scanBasePackages = "com.example.myproject") - define com.example.myproject como o pacote base a ser verificado.
  2. @SpringBootApplication(scanBasePackageClasses = CustomerController.class) - alternativa segura de tipo para scanBasePackages define o pacote de CustomerController.java, com.example.myproject.web, como o pacote base a ser verificado.

Excluindo configuração automática

Outro recurso importante é a capacidade de excluir classes de configuração automática específicas usando exclude ou excludeName (excludeName existe desde a versão 1.3).

  1. @SpringBootApplication(exclude = DemoConfiguration.class) - excluirá DemoConfiguration da verificação automática de pacotes.
  2. @SpringBootApplication(excludeName = "DemoConfiguration") - fará o mesmo usando o nome totalmente classificado da classe.
