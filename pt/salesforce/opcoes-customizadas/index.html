Introdução #

Ao contrário dos objetos personalizados que têm registros baseados neles, as configurações personalizadas permitem que você utilize conjuntos de dados personalizados em sua organização ou diferencie usuários ou perfis específicos com base em critérios personalizados. Isso significa, por exemplo, que os administradores podem editar configurações personalizadas de hierarquia para desativar as regras de fluxo de trabalho/validação para usuários ou perfis únicos, sem precisar desativá-las para toda a organização (consulte o exemplo Usando configurações personalizadas de hierarquia para desabilitar o fluxo de trabalho/regras de validação acima ).

As regras de validação geralmente precisam ser desativadas temporariamente quando:

  • O código está atualizando registros antigos, que foram editados pela última vez antes de uma regra de validação ser ativada e, portanto, não atendem aos critérios da regra mais recente.
  • O código está inserindo novos registros sem os valores exigidos pelos critérios de uma regra de validação.

As regras de fluxo de trabalho geralmente precisam ser desativadas temporariamente quando:

  • Eles acionariam um alerta de e-mail ou atualização de campo que substituiria ou interferiria nas alterações que você está fazendo no registro.

O uso de uma configuração personalizada concede aos administradores algum controle declarativo sobre o código, de modo que um dos muitos casos de uso é que, quando utilizado, pode tornar desnecessária a implantação de código para desabilitar acionadores (consulte o exemplo Usando configurações personalizadas de hierarquia para desabilitar o Apex Code acima ).

Um benefício importante para os desenvolvedores é que os dados da configuração personalizada são expostos no cache do aplicativo, o que permite acesso eficiente sem o custo de consultas repetidas ao banco de dados. Esses dados podem ser usados ​​por campos de fórmula, regras de validação, fluxos, Apex e API SOAP - consulte a documentação do Salesforce.

Os limites e considerações para configurações personalizadas estão documentados aqui.

Listar configurações personalizadas #

Também é possível criar configurações personalizadas de lista, casos de uso comuns incluem armazenamento de abreviações de estado de duas letras, prefixos de discagem internacional e números de catálogo para produtos. No entanto, o Salesforce agora está promovendo o uso de tipos de metadados personalizados, em vez de listar configurações personalizadas.

Quando você for criar uma nova configuração personalizada, a seguinte mensagem será exibida

Dica: use tipos de metadados personalizados para configuração do aplicativo Se você estiver pensando em usar configurações personalizadas de lista, considere usar tipos de metadados personalizados. Ao contrário das configurações personalizadas de lista, você pode migrar os registros de tipos de metadados personalizados usando pacotes ou ferramentas de API de metadados.

Os tipos de metadados personalizados têm benefícios adicionais em relação às configurações personalizadas da lista, conforme descrito nesta resposta. E de acordo com o desenvolvedor líder de CMDs “Há muito mais planejado para tipos de metadados personalizados do que configurações personalizadas em esteróides.”

Criando e gerenciando configurações personalizadas #

Criação #

Para criar uma configuração personalizada, acesse:

Clássico Configuração > Desenvolver > Configurações personalizadas > Novo

Relâmpago Configuração > Código personalizado > Configurações personalizadas > Novo

Crie sua configuração (consulte os Comentários posteriormente neste documento para as diferenças entre as configurações personalizadas de Hierarquia e Lista). Você pode ignorar a lista de opções Visibilidade, a menos que planeje implantar sua configuração em um pacote gerenciado.

Para criar seus campos de configuração, clique no botão Novo e siga o processo usual para criar um campo personalizado.

Gestão #

Depois de criar seu(s) campo(s), você pode começar a configurar a configuração clicando no botão Gerenciar.

É mais fácil gerenciar a configuração se você criar uma nova exibição e incluir todos os campos que criou para obter uma visão geral abrangente da configuração rapidamente. O Proprietário da Configuração é o usuário ou perfil ao qual a configuração se aplica.

Para gerenciar a configuração no nível da organização, clique no botão Novo acima do cabeçalho Valor do nível da organização padrão (na caixa vermelha abaixo).

Para gerenciar a configuração no nível do usuário ou perfil, clique no botão Novo na caixa azul abaixo.

A regra também pode ser desativada para uma organização inteira do Salesforce -

Regra de validação #

AND(
  /* the below is the reference to the Val_Rule_Cntrlr__c custom setting's checkbox field All_Opportunity_Disabled__c
  */
  $Setup.Val_Rule_Cntrlr__c.All_Opportunity_Disabled__c = FALSE,

  /* the below is the remainder of the validation rule's formula
  */
  CloseDate < TODAY()
)

Na regra acima, ambos os critérios devem ser avaliados como “TRUE” para que a regra seja acionada.

Como a caixa de seleção All_Opportunity_Disabled__c será avaliada como TRUE quando o perfil do usuário em execução for Administrador do Sistema, a regra será avaliada como FALSE.

Regras de fluxo de trabalho #

A mesma abordagem pode ser aplicada para desativar as regras de fluxo de trabalho.

Usando configurações personalizadas de hierarquia para desativar o Apex Code #

Explicação #

Neste exemplo, um simples Trigger foi criado para alterar a data de fechamento de uma oportunidade, que está prestes a ser inserida ou atualizada, para uma data 10 dias à frente.

O campo de caixa de seleção da configuração personalizada do Apex Controller permite que o código seja desabilitado no nível de usuário/perfil/organização.

Classe Apex #

trigger CloseDateUpdate on Opportunity (before insert, before update) {
    
    Id userId;
    Apx_Cntrlr__c userApexController;
    Boolean userSetting;
    
    userId = userinfo.getUserId();
    userApexController = Apx_Cntrlr__c.getInstance(userId);
    userSetting = userApexController.Close_Date_Update_Disabled__c;

    if (userSetting == false) {
        for(Opportunity opp : Trigger.new) {
            opp.CloseDate = date.today().addDays(10);
        }
    }
    
}

Teste de unidade #

@isTest
public class CloseDateUpdateTest {
    
    @testSetup
    static void dataSetup() {
        
        Profile p = [SELECT Id FROM Profile WHERE Name = 'System Administrator' LIMIT 1];
        
        User u = new User(LastName = 'Test',Alias = 't1',Email = 'example@gmail.com',Username = 'sotest@gmail.com',ProfileId = p.Id,TimeZoneSidKey = 'America/Denver',LocaleSidKey = 'en_US',EmailEncodingKey = 'UTF-8',LanguageLocaleKey = 'en_US');
        insert u;
    }
    
    static testMethod void testCloseDateUpdateEnabled() {
        
        User u = [SELECT Id FROM User WHERE Username = 'sotest@gmail.com'];
        // set the custom setting field to FALSE so that the trigger is not deactivated
        Apx_Cntrlr__c apexController = new Apx_Cntrlr__c(SetupOwnerId = u.Id,Close_Date_Update_Disabled__c = false);
        upsert apexController;
        
        Opportunity[] opportunities1 = new Opportunity[]{};
        
        test.startTest();
        system.runAs(u){
                
                for(integer i = 0; i < 200; i++) {
                    opportunities1.add(new Opportunity(
                        Name          = 'Test Opp ' + i,
                        OwnerId       = u.Id,
                        StageName     = 'Prospecting',
                        CloseDate     = date.today().addDays(1),
                        Amount        = 100));
                }
            insert opportunities1;
        }
        test.stopTest();
        
        List<Opportunity> opportunities2 = [SELECT CloseDate FROM Opportunity];
        
        for(Opportunity o : opportunities2){
            system.assertEquals(date.today().addDays(10), o.closeDate, 'CloseDateUpdate trigger should have changed the Opportunity close date as it was not disabled by the apexController custom setting');
        }
    }
    
    static testMethod void testCloseDateUpdateDisabled() {
        
        User u = [SELECT Id FROM User WHERE Username = 'sotest@gmail.com'];
        // set the custom setting field to TRUE to deactivate the trigger
        Apx_Cntrlr__c apexController = new Apx_Cntrlr__c(SetupOwnerId = u.Id,Close_Date_Update_Disabled__c = true);
        upsert apexController;
        
        Opportunity[] opportunities1 = new Opportunity[]{};
            
        test.startTest();
        system.runAs(u){
            
            for(integer i = 0; i < 200; i++) {
                opportunities1.add(new Opportunity(
                    Name          = 'Test Opp ' + i,
                    OwnerId       = u.Id,
                    StageName     = 'Prospecting',
                    CloseDate     = date.today().addDays(1),
                    Amount        = 100));
            }
            insert opportunities1;
        }
        test.stopTest();
        
        List<Opportunity> opportunities2 = [SELECT CloseDate FROM Opportunity];
        
        for(Opportunity o : opportunities2){
            system.assertEquals(date.today().addDays(1), o.closeDate, 'CloseDateUpdate trigger should not have changed the Opportunity close date as it was disabled by the apexController custom setting');
        }
    }
        
}

Atualizando configurações personalizadas de hierarquia no Apex Code #

Você pode querer atualizar suas configurações personalizadas durante a execução do seu código, para desativar a validação ou as regras de fluxo de trabalho.

No código abaixo, criei uma Classe Apex Programável que atualizará a Data de fechamento de quaisquer Oportunidades cuja Data de fechamento seja menor ou igual a 6 dias da data atual, alterando a data para 20 dias no futuro.

Usarei minha configuração personalizada Val_Rule_Cntrlr__c para desativar quaisquer regras de validação que me impeçam de atualizar as oportunidades que atendem aos meus critérios.

global class Scheduled_OppCloseDateUpdate implements Schedulable {

    global void execute(SchedulableContext SC) {
        updOpportunityCloseDates();
    }

    global void updOpportunityCloseDates() {
        
        Id userId;
        Val_Rule_Cntrlr__c setting;
        Boolean validationRulesAlreadyDisabled;
        List<Opportunity> processedOpps = new List<Opportunity>();
        Date d;
        
        // get running user's Id
        userId = userinfo.getUserId();
        // retrieve Custom Setting status, for running user
        setting = Val_Rule_Cntrlr__c.getInstance(userId);
        
        // if the setting field is false, update it to disable validation rules
        if (setting.All_Opportunity_Disabled__c == false) {
            setting.All_Opportunity_Disabled__c = true;
            upsert setting;
        }
        // if the setting field was already true, there's no need to disable it
        // but it shouldn't be switched to false by this class once the process has been completed
        else {
            validationRulesAlreadyDisabled = true;
        }
        
        // execute code to manage business process
   
        d = system.today().addDays(6);
        
        for(Opportunity o : [SELECT Id, CloseDate
                               FROM Opportunity
                              WHERE CloseDate <= :d
                                // class only updates open Opportunities
                                AND Probability > 0 AND Probability < 100])
        {
            o.CloseDate = System.today().addDays(20);
            processedOpps.add(o);
        }
        
        if (processedOpps.size() > 0) {
            update processedOpps;
        }
        
        // reactivate validation rules
        if (validationRulesAlreadyDisabled == false) {
            setting.All_Opportunity_Disabled__c = false;
            upsert setting;
        }
        
    }
    
}

Para ter certeza de que minhas regras de validação estão sendo desativadas pelas alterações na minha configuração personalizada na minha classe, criei um campo de caixa de seleção Trigger_Validation_Rule__c (que não seria visível para os usuários ou adicionado aos layouts de página) e uma regra de validação com isso critério:

AND(
  $Setup.Val_Rule_Cntrlr__c.All_Opportunity_Disabled__c = FALSE,
  Trigger_Validation_Rule__c = TRUE,

  /* allow the above criteria to be met while inserting the Opportunities, without triggering the rule, in the @testSetup portion of the test */
  NOT(ISNEW())
)

Em seguida, defino o campo da caixa de seleção como true ao criar minhas oportunidades para que os critérios das regras sejam atendidos, se o campo de configuração personalizada não for editado pelo meu código.

@isTest
private class WE_ScheduledCloseDateUpdateTest {

    @testSetup
    static void dataSetup() {

        Profile p = [SELECT Id FROM Profile WHERE Name = 'System Administrator' LIMIT 1];
        
        User u = new User(LastName = 'Test',Alias = 't1',Email = 'example@gmail.com',Username = 'sotest@gmail.com',ProfileId = p.Id,TimeZoneSidKey = 'America/Denver',LocaleSidKey = 'en_US',EmailEncodingKey = 'UTF-8',LanguageLocaleKey = 'en_US');
        insert u;

        Val_Rule_Cntrlr__c valRuleCntrlr = new Val_Rule_Cntrlr__c(SetupOwnerId = u.Id,All_Opportunity_Disabled__c = false);
        upsert valRuleCntrlr;
            
        List<Opportunity> testOpps = new List<Opportunity>();
            
        // create the Opportunities that will be updated by the class
        for(integer i = 0; i < 200; i++) {
            testOpps.add(new Opportunity(
                Name          = 'Test Opp Update' + i,
                OwnerId       = u.Id,
                StageName     = 'Prospecting',
                CloseDate     = date.today().addDays(1),
                Amount        = 100,
                // set checkbox field to true, to trigger validation rules if they've not been deactivated by class
                Trigger_Validation_Rule__c = true));
        }
        // create the Opportunities that won't be updated by the class
        for(integer i = 0; i < 200; i++) {
            testOpps.add(new Opportunity(
                Name          = 'Test Opp Skip' + i,
                OwnerId       = u.Id,
                StageName     = 'Prospecting',
                CloseDate     = date.today().addDays(15),
                Amount        = 100,
                Trigger_Validation_Rule__c = true));
        }           
        insert testOpps;

    }
    
    // code required to test a scheduled class, see https://developer.salesforce.com/docs/atlas.en-us.apexcode.meta/apexcode/apex_scheduler.htm for more details
    public static String CRON_EXP = '0 0 0 15 3 ? 2022';

    static testmethod void testCloseDateUpdates() {

        // execute scheduled class
        
        Test.startTest();
        
        String jobId = System.schedule('ScheduleApexClassTest',
                                       CRON_EXP,
                                       new Scheduled_OppCloseDateUpdate());
        
        CronTrigger ct = [SELECT Id, CronExpression, TimesTriggered, NextFireTime
                            FROM CronTrigger
                           WHERE id = :jobId];
        
        System.assertEquals(CRON_EXP, ct.CronExpression);
        System.assertEquals(0, ct.TimesTriggered);
        System.assertEquals('2022-03-15 00:00:00', String.valueOf(ct.NextFireTime));
        
        Test.stopTest();

        // test results
        
        Integer updateCount = 0;
        Integer skipCount   = 0;

        List <Opportunity> opportunitys = [SELECT Id, Name, CloseDate FROM Opportunity];

        for(Opportunity o : opportunitys) {
            if (o.Name.contains('Update') &&
                updateCount == 0)
            {
                System.assertEquals(date.today().addDays(20), o.CloseDate, 'Opportunity\'s Close Date should have been updated as it was less than 7 days away');
                updateCount = 1;
            }
            if (o.Name.contains('Skip') &&
                skipCount == 0)
            {
                System.assertEquals(date.today().addDays(15), o.CloseDate, 'Opportunity should not have been updated as it\'s Close Date is more than 7 days away');
                skipCount = 1;
            }
        }
        // check that both lists of Opportunities have been tested
        System.assertEquals(2, updateCount + skipCount, 'Count should be 2 once all assertions have been executed');
    
    }
    
    // check that the class does not change the custom setting's field to false, if it was true before class was executed
    static testmethod void testSettingUpdates() {
        
        User u = [SELECT Id FROM User WHERE UserName = 'sotest@gmail.com'];
        
        // switch the custom setting field to true before the scheduled job executes
        Val_Rule_Cntrlr__c setting;
        setting = Val_Rule_Cntrlr__c.getInstance(u.Id);
        setting.All_Opportunity_Disabled__c = true;
        upsert setting;
        
        System.runAs(u) {
            
            Test.startTest();
            
            String jobId = System.schedule('ScheduleApexClassTest',
                                           CRON_EXP,
                                           new Scheduled_OppCloseDateUpdate());
            
            CronTrigger ct = [SELECT Id, CronExpression, TimesTriggered, NextFireTime
                                FROM CronTrigger
                               WHERE id = :jobId];
            
            System.assertEquals(CRON_EXP, ct.CronExpression);
            System.assertEquals(0, ct.TimesTriggered);
            System.assertEquals('2022-03-15 00:00:00', String.valueOf(ct.NextFireTime));
            
            Test.stopTest();
        }
        setting = Val_Rule_Cntrlr__c.getInstance(u.Id);
        
        // check that the class did not change the All_Opportunity_Disabled__c field to false
        System.assertEquals(true, setting.All_Opportunity_Disabled__c);
    }

}
