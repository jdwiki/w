O ciclo de vida do aplicativo Xamarin.Android é o mesmo do aplicativo Android normal. Ao falar sobre ciclo de vida, precisamos falar sobre: ​​Ciclo de vida do aplicativo, Ciclo de vida da atividade e Ciclo de vida do fragmento.

A seguir, tentarei fornecer uma boa descrição e maneira de usá-los. Eu obtive esta documentação da documentação oficial do Android e do Xamarin e muitos recursos úteis da Web fornecidos na seção de comentários abaixo.

Links interessantes para ampliar seu conhecimento sobre o ciclo de vida do aplicativo Android:

Ciclo de vida do aplicativo #

Antes de tudo, você deve saber que pode estender a classe Android.Application para acessar dois métodos importantes relacionados ao ciclo de vida do aplicativo:

  • OnCreate - Chamado quando o aplicativo está iniciando, antes que qualquer outro objeto de aplicativo tenha sido criado (como MainActivity).

  • Em Terminar - Este método é para uso em ambientes de processo emulados. Ele nunca será chamado em um dispositivo Android de produção, onde os processos são removidos simplesmente matando-os; Nenhum código de usuário (incluindo este retorno de chamada) é executado ao fazer isso. Da documentação: https://developer.android.com/reference/android/app/Application.html#onTerminate()

No aplicativo Xamarin.Android, você pode estender a classe Application da maneira apresentada abaixo. Adicione uma nova classe chamada “MyApplication.cs” ao seu projeto:

[Application]
public class MyApplication : Application
{
    public MyApplication(IntPtr handle, JniHandleOwnership ownerShip) : base(handle, ownerShip)
    {
    }

    public override void OnCreate()
    {
        base.OnCreate();
    }

    public override void OnTerminate()
    {
        base.OnTerminate();
    }
}

Como você escreveu acima, você pode usar o método OnCreate. Você pode, por exemplo, inicializar o banco de dados local aqui ou configurar alguma configuração adicional.

Há também mais métodos que podem ser substituídos como: OnConfigurationChanged ou OnLowMemory.

Ciclo de vida da atividade #

O ciclo de vida da atividade é bem mais complexo. Como você sabe, Activity é uma página única no aplicativo Android, onde o usuário pode interagir com ela.

No diagrama abaixo, você pode ver como é o ciclo de vida da atividade do Android:

Como você pode ver, há um fluxo específico do ciclo de vida da atividade. No aplicativo móvel, você tem métodos claros em cada classe Activity que lidam com fragmentos de ciclo de vida específicos:

[Activity(Label = "LifecycleApp", MainLauncher = true, Icon = "@mipmap/icon")]
public class MainActivity : Activity
{
    protected override void OnCreate(Bundle savedInstanceState)
    {
        base.OnCreate(savedInstanceState);
        Log.Debug("OnCreate", "OnCreate called, Activity components are being created");

        // Set our view from the "main" layout resource
        SetContentView(Resource.Layout.MainActivity);
    }

    protected override void OnStart()
    {
        Log.Debug("OnStart", "OnStart called, App is Active");
        base.OnStart();
    }

    protected override void OnResume()
    {
        Log.Debug("OnResume", "OnResume called, app is ready to interact with the user");
        base.OnResume();
    }

    protected override void OnPause()
    {
        Log.Debug("OnPause", "OnPause called, App is moving to background");
        base.OnPause();
    }

    protected override void OnStop()
    {
        Log.Debug("OnStop", "OnStop called, App is in the background");
        base.OnStop();
    }

    protected override void OnDestroy()
    {
        base.OnDestroy();
        Log.Debug("OnDestroy", "OnDestroy called, App is Terminating");
    }
}

Há uma boa descrição na documentação oficial do Android:

  • Todo o tempo de vida de uma atividade acontece entre a primeira chamada para onCreate(Bundle) até uma única chamada final para onDestroy(). Uma atividade fará toda a configuração do estado “global” em onCreate() e liberará todos os recursos restantes em onDestroy(). Por exemplo, se tiver um thread rodando em segundo plano para baixar dados da rede, ele pode criar esse thread em onCreate() e então parar o thread em onDestroy().

  • O tempo de vida visível de uma atividade acontece entre uma chamada para onStart() até uma chamada correspondente para onStop(). Durante esse tempo, o usuário pode ver a atividade na tela, embora não esteja em primeiro plano e interagindo com o usuário. Entre esses dois métodos você pode manter os recursos necessários para mostrar a atividade ao usuário. Por exemplo, você pode registrar um BroadcastReceiver em onStart() para monitorar as alterações que afetam sua interface do usuário e cancelar o registro em onStop() quando o usuário não vir mais o que você está exibindo. Os métodos onStart() e onStop() podem ser chamados várias vezes, à medida que a atividade se torna visível e oculta para o usuário.

  • O tempo de vida em primeiro plano de uma atividade acontece entre uma chamada para onResume() até uma chamada correspondente para onPause(). Durante esse tempo, a atividade fica à frente de todas as outras atividades e interage com o usuário. Uma atividade pode frequentemente ficar entre os estados retomado e pausado - por exemplo, quando o dispositivo entra em suspensão, quando um resultado de atividade é entregue, quando uma nova intenção é entregue - portanto, o código nesses métodos deve ser bastante leve.

Ciclo de vida do fragmento #

Como você sabe, você pode ter uma atividade, mas diferentes fragmentos embutidos nela. É por isso que o ciclo de vida do fragmento também é importante para os desenvolvedores.

No diagrama abaixo, você pode ver como é o ciclo de vida do fragmento do Android:

Conforme descrito na documentação oficial do Android, você deve implementar pelo menos três métodos abaixo:

  • OnCreate - o sistema chama isso ao criar o fragmento. Em sua implementação, você deve inicializar os componentes essenciais do fragmento que deseja reter quando o fragmento for pausado ou interrompido e, em seguida, retomado.

  • OnCreateView - O sistema chama isso quando é hora do fragmento desenhar sua interface de usuário pela primeira vez. Para desenhar uma UI para seu fragmento, você deve retornar uma View deste método que é a raiz do layout do seu fragmento. Você pode retornar null se o fragmento não fornecer uma interface do usuário.

  • OnPause - O sistema chama este método como a primeira indicação de que o usuário está deixando o fragmento (embora isso nem sempre signifique que o fragmento está sendo destruído). Geralmente é aqui que você deve confirmar as alterações que devem ser mantidas além da sessão do usuário atual (porque o usuário pode não voltar).

Aqui está a implementação de exemplo no Xamarin.Android:

public class MainFragment : Fragment
{
    public override void OnCreate(Bundle savedInstanceState)
    {
        base.OnCreate(savedInstanceState);

        // Create your fragment here
        // You should initialize essential components of the fragment
        // that you want to retain when the fragment is paused or stopped, then resumed.
    }

    public override View OnCreateView(LayoutInflater inflater, ViewGroup container, Bundle savedInstanceState)
    {
        // Use this to return your custom view for this Fragment
        // The system calls this when it's time for the fragment to draw its user interface for the first time.

        var mainView = inflater.Inflate(Resource.Layout.MainFragment, container, false);
        return mainView;
    }

    public override void OnPause()
    {
        // The system calls this method as the first indication that the user is leaving the fragment 

        base.OnPause();
    }
}

Claro que você pode adicionar métodos adicionais aqui se quiser lidar com diferentes estados.
