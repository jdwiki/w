Este tópico mostra informações sobre como preparar seu aplicativo Xamarin.Android para o modo de lançamento e como otimizá-lo.

Preparando seu APK no Visual Studio #

Você terminou seu aplicativo, testou no modo de depuração e está funcionando perfeitamente. Agora, você deseja prepará-lo para publicar na Google Play Store.

A documentação do Xamarin fornece boas informações aqui:

https://developer.xamarin.com/guides/android/deployment,_testing,_and_metrics/publishing_an_application/

Manifesto Android

Primeiro, no Visual Studio, clique com o botão direito do mouse no projeto Xamarin.Android no Solution Explorer e selecione Properties. Em seguida, vá para a guia Android Manifest, para ver esta tela:

Ao contrário do Android Studio ou Eclipse, você não precisa definir o arquivo AndroidManifest.xml escrevendo; O Xamarin e o Visual Studio fazem isso para você. Atividades, BroadcastReceivers e Services são inseridos no Android Manifest declarando atributos específicos em suas classes.

Nesta tela, as opções são:

  • Nome do aplicativo: Este é o nome do aplicativo que ficará visível para o usuário.
  • Nome do pacote: Este é o nome do pacote. Ele deve ser exclusivo, o que significa que não deve usar o mesmo nome de pacote de outros aplicativos na Google Play Store.
  • Ícone do Aplicativo: Este é o ícone que ficará visível para o usuário, equivalente ao @drawable/ic_launcher usado em projetos Android Studio ou Eclipse.
  • Número da versão: O número da versão é usado pelo Google Play para controle de versão. Quando você quiser publicar um APK para uma versão atualizada do seu aplicativo, adicione 1 a esse número para cada nova atualização.
  • Nome da versão: Este é o nome da versão que será exibido ao usuário.
  • Local de instalação: determina onde seu APK será instalado, no armazenamento do dispositivo ou no cartão SD.
  • Permissões necessárias: aqui você determina quais permissões são necessárias para seu aplicativo.

Opções do Android

Na tela abaixo, você pode configurar as opções do compilador. Usar as opções certas aqui pode reduzir muito o tamanho do seu APK e também evitar erros.

  • Configuração: Ativa (versão).
  • Plataforma: Ativa (qualquer CPU). Eles são necessários para criar seu APK para a Google Play Store. Se a Configuração estiver definida como Depurar, ela não será aceita pelo Google Play.
  • Usar tempo de execução compartilhado: falso. Se você definir como verdadeiro, o APK usará o Mono Runtime para executar. O Mono Runtime é instalado automaticamente ao depurar através de USB, mas não no Release APK. Se o Mono Runtime não estiver instalado no dispositivo e esta opção estiver definida como verdadeira no Release APK, o aplicativo travará.
  • Gerar um pacote (.apk) por ABI selecionada: false. Crie seu APK para o maior número de plataformas possível, por motivos de compatibilidade.
  • Ativar Multi-Dex: true, mas você pode defini-lo como false se seu aplicativo não for muito complexo (ou seja, tiver menos de 65.536 métodos, veja aqui).
  • Ativar Proguard: true. Isso habilita a ferramenta Proguard que ofusca o código Java em seu aplicativo. Observe que ele não se aplica ao código .NET; se você quiser ofuscar o código .NET, você deve usar Dotfuscator. Mais informações sobre o Proguard para Xamarin.Android podem ser encontradas aqui.
  • Ativar instrumentação do desenvolvedor (depuração e criação de perfil): false para o APK de lançamento.
  • Vinculação: SDK e assemblies de usuário. Isso fará com que o Xamarin Linker remova todas as classes não utilizadas do SDK e do seu código, reduzindo o tamanho do APK.

Importante #

O Xamarin.Linker às vezes pode remover classes que não parecem ser usadas pelo seu código, especialmente se estiverem no Core do projeto (biblioteca PCL). Para evitar isso, você pode definir o Linking para “Sdk Assemblies Only” ou usar o atributo Preserve em suas classes, exemplo:

PreserveAttribute.cs

namespace My_App_Core.Models
{
    public sealed class PreserveAttribute : System.Attribute
    {
        public bool AllMembers;
        public bool Conditional;
    }
}

Em uma aula:

using System;

namespace My_App_Core.Models
{
    [Preserve(AllMembers = true)]
    public class ServiceException : Exception
    {
        public int errorCode;

        [Preserve(AllMembers = true)]
        public ServiceException() { }

        [Preserve(AllMembers = true)]
        public ServiceException(int errorCode)
        {
            this.errorCode = errorCode;
        }
    }
}
  • Arquiteturas compatíveis: Selecione tudo, por motivos de compatibilidade.

Depois de configurar tudo, reconstrua o projeto para garantir que ele seja compilado com êxito.

Criando o APK para o modo de lançamento

Você terminou de configurar seu projeto Android para lançamento. O tutorial abaixo mostra como gerar o APK no Visual Studio. Um tutorial completo da documentação do Xamarin pode ser encontrado aqui:

https://developer.xamarin.com/guides/android/deployment,_testing,and_metrics/publishing_an_application/part_2-_signing_the_android_application_package/

Para criar o arquivo APK, clique com o botão direito do mouse no projeto Xamarin.Android no Solution Explorer e selecione Archive…

Isso abrirá o gerenciador de arquivos e começará a arquivar o projeto, preparando-se para criar o arquivo APK.

Quando terminar de arquivar o projeto, clique em Distribuir… para prosseguir.

A tela Distribuir apresentará duas opções: Ad-hoc e Google Play. O primeiro irá criar um APK e salvá-lo em seu computador. O segundo publicará diretamente o aplicativo no Google Play.

A escolha do primeiro é recomendada, para que você possa testar o APK em outros dispositivos, se desejar.

Na tela a seguir, é necessário um Android Key Store para assinar o APK. Se você já possui um, pode usá-lo clicando em Importar…; caso contrário, você pode criar uma nova Android Key Store clicando em +.

Criando uma nova tela do Android Key Store:

Para criar o APK, clique em Salvar como. Você pode ser solicitado a digitar a senha do armazenamento de chaves.

Quando terminar, você pode clicar em Abrir pasta na tela Arquivos para ver o arquivo APK gerado.

Habilitando o MultiDex em seu APK Xamarin.Android #

MultiDex é uma biblioteca no APK do Android que permite que o app tenha mais de 65.536 métodos.

Os APKs do Android possuem arquivos executáveis ​​Dalvik (.dex) que contêm os bytecodes gerados compilados a partir do seu código Java. Cada arquivo .dex pode conter até 65.536 métodos (2^16).

As versões do SO Android anteriores ao Android 5.0 Lollipop (API 21) usam o tempo de execução Dalvik, que suporta apenas um arquivo .dex por APK, limitando a 65.536 métodos por APK. A partir do Android 5.0, o sistema operacional Android usa o tempo de execução ART, que pode oferecer suporte a mais de um arquivo .dex por APK, evitando o limite.

Para ultrapassar o limite de 65k métodos nas versões do Android abaixo da API 21, os desenvolvedores devem usar a biblioteca de suporte MultiDex. O MultiDex cria arquivos extra classes.dex (classes2.dex, classes3.dex, …) referenciando-os no arquivo classes.dex. Quando o aplicativo começa a carregar, ele usa uma classe MultiDexApplication para carregar os arquivos .dex extras.

Se o seu aplicativo Android visa uma versão mínima do SDK superior ou igual à API 21 (Android 5.0 Lollipop), não é necessário usar a biblioteca MultiDex, pois o SO trata de forma nativa os arquivos .dex extras. No entanto, se por motivos de compatibilidade o desenvolvedor quiser oferecer suporte ao sistema operacional Android mais antigo, ele deverá usar a biblioteca MultiDex.

Como usar o MultiDex em seu aplicativo Xamarin.Android #

Primeiro, para habilitar o MultiDex em seu aplicativo Xamarin.Android, vá em Propriedades do seu projeto -> Opções do Android -> Embalagem -> Habilitar Multi-Dex, conforme a tela de impressão abaixo:

Em seguida, você deve criar uma classe MultiDexApplication em seu aplicativo. Na raiz do projeto, crie uma nova classe (no Solution Explorer, clique com o botão direito do mouse no projeto, Add.. -> Class ou Shift+Alt+C). No novo arquivo de classe, copie o código a seguir, substituindo o namespace Sample pelo nome do namespace do projeto Xamarin.Android.

using System;
using Android.App;
using Android.Runtime;
using Java.Interop;

namespace Sample
{
    [Register("android/support/multidex/MultiDexApplication", DoNotGenerateAcw = true)]
    public class MultiDexApplication : Application
    {
        internal static readonly JniPeerMembers _members =
        new XAPeerMembers("android/support/multidex/MultiDexApplication", typeof (MultiDexApplication));

        internal static IntPtr java_class_handle;

        private static IntPtr id_ctor;

        [Register(".ctor", "()V", "", DoNotGenerateAcw = true)]
        public MultiDexApplication()
        : base(IntPtr.Zero, JniHandleOwnership.DoNotTransfer)
        {
            if (Handle != IntPtr.Zero)
                return;

            try
            {
                if (GetType() != typeof (MultiDexApplication))
                {
                    SetHandle(
                        JNIEnv.StartCreateInstance(GetType(), "()V"),
                        JniHandleOwnership.TransferLocalRef);
                        JNIEnv.FinishCreateInstance(Handle, "()V");
                    return;
                }

                if (id_ctor == IntPtr.Zero)
                    id_ctor = JNIEnv.GetMethodID(class_ref, "<init>", "()V");
                SetHandle(
                    JNIEnv.StartCreateInstance(class_ref, id_ctor),
                    JniHandleOwnership.TransferLocalRef);
                JNIEnv.FinishCreateInstance(Handle, class_ref, id_ctor);
            }
            finally
            {
            }
        }

        protected MultiDexApplication(IntPtr javaReference, JniHandleOwnership transfer)
            : base(javaReference, transfer)
        {
        }

        internal static IntPtr class_ref
        {
            get { return JNIEnv.FindClass("android/support/multidex/MultiDexApplication", ref java_class_handle); }
        }

        protected override IntPtr ThresholdClass
        {
            get { return class_ref; }
        }

        protected override Type ThresholdType
        {
            get { return typeof (MultiDexApplication); }
        }
    }
}

Fonte do código aqui.

Se você estiver desenvolvendo no Visual Studio para Windows, também há um bug nas ferramentas de compilação do Android SDK que você precisa corrigir para criar corretamente os arquivos classes.dex ao compilar seu projeto.

Vá para a pasta do Android SDK, abra a pasta build-tools e haverá pastas com os números dos compiladores do Android SDK, como:

C:\android-sdk\build-tools\23.0.3\

C:\android-sdk\build-tools\24.0.1\

C:\android-sdk\build-tools\25.0.2\

Dentro de cada uma dessas pastas, há um arquivo chamado mainClassesDex.bat, um script em lote usado para criar os arquivos classes.dex. Abra cada arquivo mainClassesDex.bat com um editor de texto (Notepad ou Notepad++) e em seu script, localize e substitua o bloco:

if DEFINED output goto redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder "%disableKeepAnnotated%" "%tmpJar%" "%params%"
goto afterClassReferenceListBuilder
:redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder "%disableKeepAnnotated%" "%tmpJar%" "%params%" 1>"%output%"
:afterClassReferenceListBuilder

Com o bloco:

SET params=%params:'=%  
if DEFINED output goto redirect  
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder %disableKeepAnnotated% "%tmpJar%" %params%  
goto afterClassReferenceListBuilder  
:redirect
call "%java_exe%" -Djava.ext.dirs="%frameworkdir%" com.android.multidex.MainDexListBuilder %disableKeepAnnotated% "%tmpJar%" %params% 1>"%output%"  
:afterClassReferenceListBuilder

Salve cada mainClassesDex.bat no editor de texto após as alterações.

Após as etapas acima, você poderá criar com êxito seu aplicativo Xamarin.Android com o MultiDex.

Habilitando o ProGuard em seu APK Xamarin.Android #

ProGuard é uma ferramenta usada no processo de construção para otimizar e ofuscar o código Java do seu APK e também remover classes não utilizadas. O APK resultante ao usar o ProGuard terá um tamanho menor e será mais difícil de engenharia reversa (descompilação).

O ProGuard também pode ser usado em aplicativos Xamarin.Android e também reduzirá o tamanho do arquivo APK e ofuscará o código Java. Esteja ciente, no entanto, de que a ofuscação do ProGuard se aplica apenas ao código Java. Para ofuscar o código .NET, o desenvolvedor deve usar o Dotfuscator ou ferramentas semelhantes.

Como usar o ProGuard em seu aplicativo Xamarin.Android #

Primeiro, para habilitar o ProGuard em seu aplicativo Xamarin.Android, vá para Propriedades do seu projeto -> Opções do Android -> Embalagem -> Habilitar o ProGuard, conforme a tela de impressão abaixo:

Isso habilita o ProGuard ao criar seu aplicativo.

O Xamarin.Android, por padrão, define suas próprias configurações para o ProGuard, que podem ser encontradas dentro das pastas obj/Debug/proguard ou obj/Release/proguard, nos arquivos proguard_project_primary.cfg, proguard_project_references.cfg e proguard_xamarin.cfg. Os três arquivos são combinados como configurações para o ProGuard e são criados automaticamente pelo Xamarin ao compilar.

Se o desenvolvedor quiser personalizar ainda mais as opções do ProGuard, ele pode criar um arquivo na raiz do projeto chamado proguard.cfg (outros nomes também são válidos, desde que a extensão seja .cfg) e definir sua ação de compilação como ProguardConfiguration, conforme a figura abaixo:

No arquivo, opções personalizadas do ProGuard podem ser inseridas, como -dontwarn, -keep class e others.

Bugs “misteriosos” relacionados ao ProGuard e Linker #

Você fez um ótimo aplicativo e o testou no Debug, com bons resultados. Tudo estava funcionando bem!

Mas então, você decidiu preparar seu aplicativo para lançamento. Você configurou o MultiDex, ProGuard e Linker e, em seguida, parou de funcionar.

Este tutorial tem como objetivo ajudá-lo a descobrir problemas comuns relacionados ao ProGuard e ao Linker que podem causar bugs misteriosos.

Compreendendo o Xamarin.Linker #

Xamarin.Linker é uma ferramenta no processo de construção que remove código não utilizado e classes do seu código .NET (não código Java). Nas Propriedades do seu projeto -> Opções do Android -> Linker, haverá uma caixa de seleção Vinculando com as opções:

Nenhum: nenhum código é removido.

Sdk Assemblies Only: Esta opção faz com que o Xamarin.Linker verifique o código não utilizado apenas nas bibliotecas Xamarin. Esta opção é segura.

Sdk e assemblies de usuário: essa opção faz com que o Xamarin.Linker verifique o código não utilizado nas bibliotecas Xamarin e no código do projeto (incluindo PCLs, componentes Xamarin e pacotes NuGet). Esta opção nem sempre é segura!

Ao usar a opção SDK e User Assemblies, o Xamarin.Linker pode pensar que partes do código não são usadas quando na verdade são muito usadas! Isso pode fazer com que algumas bibliotecas parem de funcionar corretamente e causem bugs em seu aplicativo.

Para que o Xamarin.Linker não remova o código, existem 3 opções:

  1. Configurando a opção Linking para None ou Sdk Assemblies Only;
  2. Pule as montagens de ligação;
  3. Usando o atributo Preservar.

Exemplo para 3. Usando o atributo Preservar:

O Xamarin.Linker percebe como código não utilizado principalmente código de classes de modelo no núcleo do seu projeto.

Para tornar a classe preservada durante o processo de vinculação, você pode usar o atributo Preserve.

Primeiro, crie no núcleo do seu projeto uma classe chamada PreserveAttribute.cs, insira o seguinte código e substitua o namespace pelo namespace do seu projeto:

PreserveAttribute.cs:

namespace My_App_Core.Models
{
    public sealed class PreserveAttribute : System.Attribute
    {
        public bool AllMembers;
        public bool Conditional;
    }
}

Em cada classe de modelo do núcleo do seu projeto, insira o atributo Preserve como no exemplo abaixo:

País.cs:

using System;
using System.Collections.Generic;

namespace My_App_Core.Models
{
    [Preserve(AllMembers = true)]
    public class Country
    {
        public String name { get; set; }
        public String ISOcode { get; set; }

        [Preserve(AllMembers = true)]
        public Country(String name, String ISOCode)
        {
            this.name = name;
            this.ISOCode = ISOCode;
        }
    }
}

Depois disso, o processo de vinculação não removerá mais o código preservado.

Compreendendo o ProGuard #

O ProGuard é uma ferramenta no processo de construção que remove código e classes não utilizados do seu código Java. Ele também ofusca e otimiza o código.

No entanto, o ProGuard às vezes pode remover o código que percebe como não utilizado, quando não é. Para evitar isso, o desenvolvedor deve depurar o aplicativo (no Android Device Monitor e no Visual Studio Debug) e detectar qual classe foi removida, para então configurar o arquivo de configuração do ProGuard para manter a classe.

Exemplo

No exemplo abaixo, o ProGuard removeu duas classes (Android.Support.V7.Widget.FitWindowsLinearLayout e Android.Support.Design.Widget.AppBarLayout) usadas em arquivos de layout AXML, mas que foram percebidas como não utilizadas no código. A remoção causou ClassNotFoundException no código Java ao renderizar o layout da atividade:

Avisos do ProGuard

O ProGuard às vezes mostra avisos na Lista de Erros depois de compilar seu projeto. Embora eles levantem a questão de saber se seu aplicativo está OK ou não, nem todos os avisos indicam problemas, especialmente se o aplicativo for compilado com êxito.

Um exemplo disso é ao usar a biblioteca Picasso: ao usar o ProGuard, isso pode mostrar avisos como okio.Okio: can't find referenced class (...) ou ‘can’t write resource [META-INF/MANIFEST.MF] (Entrada zip duplicada [okhttp.jar:META-INF/MANIFEST.MF]) (…)`, mas o aplicativo é compilado e a biblioteca funciona sem problemas.
