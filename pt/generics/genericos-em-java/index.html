Sintaxe #

  • class MyClass<T1, T2 estende CharSequence> implementa Comparable<MyClass> //...
  • interface MyListInterface<T extends Serializable> extends List<T> //...
  • public <T1, T2 estende Instant> T1 provideClone(T1 toClone, T2 instant> //...
  • public static List<CharSequence> safe(Collection<? extends CharSequence> l) { return new ArrayList<>(l);}
  • Set<String> strings = Collections.singleton("Hello world");
  • List<CharSequence> chsList = safe(strings);

O apagamento de tipo limita a reflexão, embora isso não seja específico da JVM, por exemplo Ceylon [usa genéricos reificados](http://ceylon-lang.org/documentation/1.2/spec /html/execution.html#reification).

O suporte de tipo existencial não é necessariamente suportado por outras linguagens neste formato: Kotlin [suporta-o por meio de projeções de tipo](https://kotlinlang.org/docs/reference/java-interop. html#java-generics-in-kotlin).

Introdução #

Generics foi introduzido em Java em sua versão (1.)5. Eles são apagados durante a compilação, portanto, a reflexão em tempo de execução não é possível para eles. Os genéricos geram novos tipos parametrizados por outros tipos. Por exemplo, não precisamos criar novas classes para usar a coleção de tipo seguro de Strings e Numbers, ArrayList<T> genérico pode ser usado em todos os casos, como: `new ArrayList ()».

Exemplo:

List<String> variable = new ArrayList<String>();

No Java 7, algum açúcar sintático foi introduzido para facilitar a construção (<> também conhecido como diamante):

List<String> variable = new ArrayList<>();

Curiosamente também era possível (a partir do Java 5) usar inferência de tipo, quando um método estático tinha como valor de retorno (muitas vezes usado no Google Guava por exemplo):

List<String> singleton = Collections.singletonList();//Note the missing `<>` or `<String>`!

Em Java os tipos existenciais eram usados ​​para fornecer polimorfismo para os tipos, pois os tipos genéricos são invariáveis ​​(por exemplo: List<String> não é um subtipo, nem um supertipo de List<CharSequence>, embora em Java String [] é um subtipo de CharSequence[]; nota: String implementa a interface CharSequence). Os tipos genéricos existenciais podem ser expressos como:

List<? extends CharSequence> list = new ArrayList<String>();
Comparable<? super ChronoLocalDate> ccld = LocalDate.now();
ChronoLocalDate cld = JapaneseDate.now(); //ChronoLocalDate extends Comparable<ChronoLocalDate>
ccld.compareTo(cld);
//cld.compareTo(ccld);//fails to compile because ccld is not a `ChronoLocalDate` (compile time)

Ambas as instâncias podem ser usadas em uma lista parametrizada pelo Comparable correspondente:

List<Comparable<? super ChronoLocalDate>> list2 = new ArrayList<>();
list2.add(cld);
list2.add(ccld);

Métodos genéricos #

Parâmetros de tipo genéricos são comumente definidos no nível de classe ou interface, mas métodos e (raramente) construtores também suportam a declaração de parâmetros de tipo vinculados ao escopo de uma única chamada de método.

class Utility // no generics at the class level
{
    @SafeVarargs
    public static <T> T randomOf(T first, T... rest) {
        int choice = new java.util.Random().nextInt(rest.length + 1);
        return choice == rest.length ? first : rest[choice];
    }

    public static <T extends Comparable<T>> T max(T t1, T t2) {
        return t1.compareTo(t2) < 0 ? t2 : t1;
    }
}

Observe que as declarações de parâmetro de tipo, T e <T extends Comparable<T>> respectivamente, aparecem após os modificadores de método e antes do tipo de retorno. Isso permite que o parâmetro de tipo T seja usado dentro do escopo de tais métodos, atuando como:

  • tipos de argumentos
  • tipo de retorno
  • tipos de variáveis ​​locais

Embora ambos os métodos acima usem o mesmo nome de parâmetro de tipo T, no nível do método eles são completamente independentes um do outro. O compilador inferirá o tipo real com base nos argumentos passados ​​para o método em cada site de chamada que invoca o método. Como o método max declara que T estende Comparable<T>, o compilador também impõe que os tipos inferidos sejam implementações compatíveis da interface Comparable.

Integer num1 = 1;
Integer num2 = 2;
String str1 = "abc";
String str2 = "xyz";

Integer bigger = Utility.max(num1, num2);
assert bigger == num2;

String later = Utility.max(str2, str1);
assert later == str2;

Utility.max(num1, str1); // compiler error: num1 and str1 are incompatible types

Utility.max(new Object(), new Object()); // compiler error: Object does not implement Comparable

O Java 8 melhorou significativamente a capacidade do compilador de inferir corretamente os tipos genéricos em sites de chamadas. Se o compilador não inferir o tipo adequado, os desenvolvedores podem declarar explicitamente o tipo como parte da chamada:

Object obj = Utility.<Object>randomOf(str1, new Object(), num1);
