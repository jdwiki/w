O arquivo ec2.py é um script python que executa e retorna seus recursos da AWS com base no ec2.ini que é o arquivo de configuração que você precisa personalizar se quiser limitar o escopo do seu projeto a algumas regiões específicas , tags específicas etc…

Como configurar corretamente o Ansible para se conectar ao Amazon Web Services #

O gerenciamento de recursos da AWS que são dimensionados para cima e para baixo atinge os limites do arquivo host de inventário estático, é por isso que precisamos de algo dinâmico. E é para isso que servem os inventários dinâmicos. Vamos começar:

Baixe estes arquivos ec2.ini e ec2.py para a pasta do seu projeto:

cd my_ansible_project 
wget https://raw.githubusercontent.com/ansible/ansible/devel/contrib/inventory/ec2.py    
wget https://raw.githubusercontent.com/ansible/ansible/devel/contrib/inventory/ec2.ini

Uma vez feito, torne o arquivo ec2.py executável:

chmod +x ec2.py

Agora, exporte sua chave secreta e de acesso da AWS como variáveis ​​de ambiente:

export AWS_ACCESS_KEY_ID='ABCDEFGHIJKLM'
export AWS_SECRET_ACCESS_KEY='NOPQRSTUVWXYZ'

Para usar o script ec2.py precisamos do Python AWS SDK, boto então você precisa instalá-lo:

sudo pip install boto

Para testar se está tudo certo, tente executar o ec2.py listando seus recursos:

./ec2.py --list

você deve ver algo semelhante a:

{
  "_meta": {
    "hostvars": {}
  }
}

Agora queremos usar o inventário dinâmico junto com nosso arquivo de hosts estáticos. Primeiro, crie uma pasta chamada inventory, adicione ec2.py, ec2.ini e nosso arquivo hosts e diga ao Ansible para usar essa pasta como um arquivo de inventário:

mkdir inventory 
mv ec2.py inventory/ec2.py
mv ec2.ini inventory/ec2.ini
mv hosts inventory/hosts

Em seguida, devemos definir a configuração de nível de projeto para o Ansible criando um arquivo de configuração do Ansible na pasta do projeto chamado ansible.cfg e adicionando isto:

[defaults]
hostfile = inventory
[ssh_connection]
pipelining = False
ssh_args = -o ControlMaster=auto -o ControlPersist=30m -o StrictHostKeyChecking=no

Em seguida, precisamos configurar o Ansible para usar uma chave SSH para autenticar o acesso às nossas instâncias do EC2. Usar um agente SSH é a melhor maneira de autenticar com recursos, pois isso facilita o gerenciamento de chaves:

ssh-agent bash 
ssh-add ~/.ssh/keypair.pem

É isso! Se você seguiu isso, você pode testá-lo usando o ping module e então, você verá suas instâncias em execução que foram configuradas para usar sua chave respondendo com pong:

ansible -m ping all
11.22.33.44 | success >> {
    "changed": false, 
    "ping": "pong"
}

Como iniciar a instância do EC2 a partir de AMIs oficiais da Amazon, modificá-la e armazená-la como nova AMI #

Esse é um fluxo de trabalho muito comum ao usar o Ansible para provisionar uma instância do AWS EC2. Esta postagem pressupõe uma compreensão básica do Ansible e, mais importante, pressupõe que você o configurou corretamente para se conectar à AWS.

Como a documentação oficial do Ansible insiste, usaremos quatro funções:

1- ami_find para obter o id da ami com base no qual iniciaremos nossa instância do EC2.

2- ec2_ami_creation para iniciar efetivamente a instância do EC2.

3- code_deploy para modificação da instância; isso pode ser qualquer coisa, então simplesmente transferiremos um arquivo para a máquina de destino.

4- build_ami para construir nossa nova imagem com base na instância ec2 em execução. Este post assume que você está no nível superior do seu projeto Ansible: my_ansible_project

A primeira função: ami_find

cd my_ansible_project/roles && ansible-galaxy init ami_find

Nesta função, usaremos o módulo ec2_ami_find e, como exemplo, procuraremos uma máquina Ubuntu e obteremos seu ami_id ( ami-xxxxxxxx ). Agora edite o arquivo my_ansible_project/roles/ami_find/tasks/main.yml:

---
- ec2_ami_find:
    name: "ubuntu/images/hvm-ssd/ubuntu-trusty-14.04-amd64-server-*"
    sort: name
    sort_order: descending
    sort_end: 1
    region: "{{ aws_region }}"
  register: ami_find
- set_fact: ami_ubuntu="{{ ami_find.results[0].ami_id }}"

A segunda função: ec2_ami_creation

Aqui, usaremos o ami_id que obtivemos do primeiro papel e, em seguida, iniciaremos nossa nova instância com base nele:

cd my_ansible_project/roles && ansible-galaxy init ec2_ami_creation

Nesta função, usaremos mais importante o ec2_module para iniciar nossa instância. Agora edite o arquivo my_ansible_project/roles/ec2_ami_creation/tasks/main.yml:

---
- ec2_vpc_subnet_facts:
    region: "{{aws_region}}"
  register: vpc
- name: creation of security group of the ec2 instance
  ec2_group:
    name: example
    description: an example EC2 group
    region: "{{ aws_region }}"
    rules:
      - proto: tcp
        from_port: 22
        to_port: 22
        cidr_ip: 0.0.0.0/0
    state: present
  register: ec2_sg

- name: create instance using Ansible
  ec2:
    key_name: "{{ ansible_key }}"
    group: example
    vpc_subnet_id: "{{vpc.subnets[0].id}}"
    instance_type: "{{ instance_type }}"
    ec2_region: "{{ aws_region }}"
    image: "{{ base_image }}"
    assign_public_ip: yes
    wait: yes
  register: ec2

- set_fact: id={{ec2.instances[0].id}}

- name: adding the newly created instance to a temporary group in order to access it later from another play
  add_host: name={{ item.public_ip }} groups=just_created
  with_items: ec2.instances

- name: Wait for SSH to come up
  wait_for: host={{ item.public_dns_name }} port=22 delay=10 timeout=640 state=started
  with_items: ec2.instances

A terceira função: code_deploy

Aqui, provisionaremos esta instância, que foi adicionada a um grupo chamado just_created

cd my_ansible_project/roles && ansible-galaxy init code_deploy

Nesta função, vamos usar o template_module para transferir um arquivo e escrever o nome do host da máquina nele. Agora edite o arquivo my_ansible_project/roles/code_deploy/tasks/main.yml:

---
- template: src=my_file.txt.j2 dest=/etc/my_file.txt

em seguida, mova para a pasta de modelos dentro de sua função:

cd my_ansible_project/roles/templates e adicione um arquivo chamado my_file.txt.j2 contendo:

my name is {{ ansible_hostname }}`

A quarta função: build_ami

Agora vamos criar uma imagem da instância em execução usando o módulo ec2_ami. Mova para a pasta do seu projeto e:

 cd my_ansible_project/roles && ansible-galaxy init build_ami

Agora edite o arquivo my_ansible_project/roles/build_ami/tasks/main.yml:

---
- ec2_ami:
    instance_id: "{{ instance_id }}"
    wait: yes
    name: Base_Image

Agora, acho que você está se perguntando como orquestrar todos esses papéis. Estou certo? Se sim, continue lendo.

Vamos escrever um manual, composto por três peças: a primeira peça aplicável em localhost chamará nossos dois primeiros papéis, a segunda peça aplicável em nosso grupo just_created. última função será aplicável em localhost. Por que localhost? Quando queremos gerenciar alguns recursos da AWS, usamos nossa máquina local, simples assim. Em seguida, usaremos um arquivo vars no qual colocaremos nossas variáveis: ansible_key, aws_region, etc…

crie uma pasta de infraestrutura no topo do seu projeto e adicione um arquivo dentro dela chamado aws.yml:

---
aws_region: ap-southeast-2
ansible_key: ansible
instance_type: t2.small

Então, no topo do seu projeto, crie build_base_image.yml e adicione isto:

---
   - hosts: localhost
     connection: local
     gather_facts: False
     vars_files:
       - infrastructure/aws.yml
     roles:
       - ami_find
       - { role: ec2_creation, base_image: "{{ ami_ubuntu }}"}

   - hosts: just_created
     connection: ssh
     gather_facts: True
     become: yes
     become_method: sudo
     roles:
       - code_deploy

   - hosts: localhost
     connection: local
     gather_facts: False
     vars_files:
       - infrastructure/aws.yml
     roles:
       - { role: new_image, instance_id: "{{ id }}"}

É isso, não se esqueça de excluir seus recursos depois de testar isso, ou por que não criar uma função para excluir a instância em execução :-)
