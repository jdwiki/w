Parâmetros #

Parâmetro Explicação
ansible_connection Tipo de conexão com o host. Este pode ser o nome de qualquer um dos plugins de conexão do ansible. Os tipos de protocolo SSH são smart, ssh ou paramiko. O padrão é inteligente. Os tipos não baseados em SSH são descritos na próxima seção.
ansible_host O nome do host ao qual se conectar, se for diferente do alias que você deseja dar a ele.
ansible_port O número da porta ssh, se não 22
ansible_user O nome de usuário ssh padrão a ser usado.
ansible_ssh_pass A senha ssh a ser usada (isso é inseguro, recomendamos usar chaves --ask-pass ou SSH)
ansible_ssh_private_key_file Arquivo de chave privada usado por ssh. Útil se estiver usando várias chaves e você não quiser usar o agente SSH.
ansible_ssh_common_args Essa configuração é sempre anexada à linha de comando padrão para sftp, scp e ssh. Útil para configurar um ProxyCommand para um determinado host (ou grupo).
ansible_sftp_extra_args Essa configuração é sempre anexada à linha de comando padrão sftp.
ansible_scp_extra_args Essa configuração é sempre anexada à linha de comando padrão scp.
ansible_ssh_extra_args Essa configuração é sempre anexada à linha de comando padrão ssh.
ansible_ssh_pipelining Determina se deve ou não usar o pipeline SSH. Isso pode substituir a configuração pipelining em ansible.cfg.
ansible_become Equivalente a ansible_sudo ou ansible_su, permite forçar o escalonamento de privilégios
ansible_become_method Permite definir o método de escalonamento de privilégios
ansible_become_user Equivalente a ansible_sudo_user ou ansible_su_user, permite definir o usuário que você se torna através do escalonamento de privilégios
ansible_become_pass Equivalente a ansible_sudo_pass ou ansible_su_pass, permite definir a senha de escalonamento de privilégios
ansible_shell_type O tipo de shell do sistema de destino. Você não deve usar esta configuração a menos que tenha configurado o ansible_shell_executable para um shell não compatível com Bourne (sh). Por padrão, os comandos são formatados usando a sintaxe no estilo sh. Definir isso como csh ou fish fará com que os comandos executados nos sistemas de destino sigam a sintaxe desses shell.
ansible_python_interpreter O caminho do python do host de destino. Isso é útil para sistemas com mais de um Python ou não localizados em /usr/bin/python, como BSD, ou onde /usr/bin/python não é um Python da série 2.X. Não usamos o mecanismo /usr/bin/env, pois isso exige que o caminho do usuário remoto seja definido corretamente e também assume que o executável python é denominado python, onde o executável pode ser nomeado como * python2.6.
ansible_*_interpreter Funciona para qualquer coisa como ruby ​​ou perl e funciona exatamente como ansible_python_interpreter. Isso substitui o shebang de módulos que serão executados nesse host.
ansible_shell_executable Isso define o shell que o controlador ansible usará na máquina de destino, substitui o executable em ansible.cfg cujo padrão é /bin/sh. Você só deve alterá-lo se não for possível usar /bin/sh (ou seja, /bin/sh não está instalado na máquina de destino ou não pode ser executado a partir do sudo.). Novo na versão 2.1.

Inventário com chave privada personalizada #

[targethost]
192.168.1.1 ansible_user=mrtuovinen ssh_private_key_file=~/.ssh/custom_key

Inventário, Vars de Grupo e Você #

estrutura do projeto (melhor prática ansible).

project/
  group_vars/
     development
  inventory.development
  playbook.yaml

tudo começa com o inventário.desenvolvimento

[development]
dev.fakename.io

[development:vars]
ansible_host: 192.168.0.1
ansible_user: dev
ansible_pass: pass
ansible_port: 2232

[api:children]
development

que permite vincular a group_vars. Mantenha dados ’específicos’ para esse ambiente …

---
app_name: NewApp_Dev
app_url: https://dev.fakename.io
app_key: f2390f23f01233f23f

que permite executar o seguinte manual CONTRA o arquivo de inventário:

---
- name: Install api.
  hosts: api
  gather_facts: true
  sudo: true
  tags:
    - api
  roles:
    - { role: api,         tags: ["api"]         }

com a seguinte linha de execução:

ansible-playbook playbook.yaml -i  inventory.development

Inventário com nome de usuário e senha #

O inventário é a maneira Ansible de rastrear todos os sistemas em sua infraestrutura. Aqui está um arquivo de inventário simples contendo um único sistema e as credenciais de login do Ansible.

[targethost]
192.168.1.1 ansible_user=mrtuovinen ansible_ssh_pass=PassW0rd

Inventário com porta SSH personalizada #

[targethost]
192.168.1.1 ansible_user=mrtuovinen ansible_port=2222

Passe o inventário estático para o ansible-playbook #

ansible-playbook -i path/to/static-inventory-file -l myhost myplaybook.yml

Passe o inventário dinâmico para o ansible-playbook #

ansible-playbook -i path/to/dynamic-inventory-script.py -l myhost myplaybook.yml

Consulte inventário dinâmico para mais detalhes.

Arquivo de hosts #

O arquivo host é usado para armazenar conexões para playbooks Anisble. Existem opções para definir parâmetros de conexão:

ansible_host é o nome do host ou endereço IP

ansible_port é a porta que a máquina usa para SSH

ansible_user é o usuário remoto para se conectar como

ansible_ssh_pass se estiver usando uma senha para SSH

ansible_ssh_private_key_file se você precisar usar várias chaves específicas para hosts

Estas são as opções mais utilizadas. Mais informações podem ser encontradas na documentação oficial do Ansible.

Aqui está um exemplo de arquivo hosts:

# Consolidation of all groups
[hosts:children]
web-servers
offsite
onsite
backup-servers

[web-servers]
server1 ansible_host=192.168.0.1 ansible_port=1600
server2 ansible_host=192.168.0.2 ansible_port=1800

[offsite]
server3 ansible_host=10.160.40.1 ansible_port=22 ansible_user=root
server4 ansible_host=10.160.40.2 ansible_port=4300 ansible_user=root

# You can make groups of groups
[offsite:children]
backup-servers

[onsite]
server5 ansible_host=10.150.70.1 ansible_ssh_pass=password

[backup-servers]
server6 ansible_host=10.160.40.3 ansible_port=77
