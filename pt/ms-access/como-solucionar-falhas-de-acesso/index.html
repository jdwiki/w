Como solucionar falhas de acesso

Quando você recebe um erro: “O Microsoft Access encontrou um problema e precisa ser fechado”, geralmente não há muitas informações para ajudá-lo a identificar a causa do erro. Abaixo estão uma série de etapas que você pode seguir para solucionar a causa dos erros.

Certifique-se de remover outras variáveis ​​da equação durante o teste

Corrupção de rede

Não carregue o cliente fora de uma rede. Coloque-o na unidade local e execute-o a partir daí.

Construções corporativas

Se você estiver em um ambiente corporativo que está usando “compilações de computador” e não teve sucesso com descompilação, teste de memória ou remoção de dados binários - recuse-se a fazer mais testes até que a equipe de TI possa fornecer ao usuário uma máquina de teste que tenha apenas Windows, Office e Service Packs instalados.

Todos os softwares e atualizações devem ser instalados manualmente sem usar instalações autônomas. Não instale antivírus nesta máquina para teste.

Entenda que muitos departamentos de TI simplesmente tentam fazer uma abordagem de tamanho único com as compilações, e suas compilações são todas baseadas umas nas outras. Com o tempo, os conflitos de software podem fazer com que o Access falhe diretamente ou aja de forma estranha.

Energia ruim

Conforme mencionado no exemplo de memória - as flutuações de energia podem causar erros no computador. Se o banco de dados estiver em um prédio industrial, tente colocar as mãos em um condicionador de energia ou um no-break que forneça energia limpa (fora da bateria, não da fonte principal passando por um varistor de óxido de metal)

Além disso, verifique o cabo de alimentação que está conectado à barra de alimentação ou tomada. Certifique-se de que as especificações do medidor e da tensão sejam suficientes. Os departamentos de TI geralmente deixam os cabos de alimentação conectados na estação e apenas removem a máquina. Depois de muitos anos, eles estão usando fontes de alimentação mais robustas, mas não trocaram o cabo. Faz diferença. Na dúvida, leve um cabo novo e mais grosso.

Descompilar banco de dados #

Esta deve ser sempre a sua correção inicial. Uma boa política é descompilar o banco de dados antes de cada lançamento.

  1. Crie um atalho de descompilação. Isso carrega o banco de dados com uma opção “/decompile”.

  2. Clique com o botão direito do mouse em seu arquivo de banco de dados. Selecione Copiar

  3. Clique com o botão direito na janela do explorer e selecione “Colar atalho”

  4. Clique com o botão direito do mouse no atalho e selecione “Propriedades”

  5. Na caixa Destino, vá até o final da linha e adicione /decompile

  6. Clique em Ok para fechar o atalho

  7. Abra o banco de dados com Shift. Mantenha pressionada a tecla shift enquanto clica duas vezes neste atalho. Isso impede que qualquer execução automática seja executada no banco de dados. Você deve ir direto para a janela de navegação.

  8. Compactar e reparar o banco de dados. Depois que o banco de dados estiver carregado, você precisará clicar no botão Compactar e Reparar.

  9. Localize o botão Compact and Repair Database na faixa Tools.

  10. Mantenha pressionada a tecla Shift. Mantenha-o pressionado enquanto clica no botão Compactar e Reparar.

  11. Recompilar o banco de dados

  12. Vá para a janela VBA (Control + G)

  13. Selecione Depurar -> Compilar no menu

Este é o processo de descompilação completo. Geralmente deve corrigir 99% de todas as falhas do Access ou comportamento de formulário estranho.

Teste a memória do computador #

Se suas falhas forem aleatórias ou esporádicas, execute esta etapa. Se suas falhas ocorrerem toda vez que você executar o banco de dados, essa etapa não corrigirá o problema (embora a memória ruim possa ser a razão pela qual a corrupção ocorreu em primeiro lugar).

Use um testador de memória que inicialize fora do sistema operacional e execute várias passagens. Duas opções populares são MemTest86 (Comercial) e MemTest86+ (Open Source)

Inicie o teste e deixe-o funcionar durante o horário de trabalho. A razão para isso é porque outros fatores no prédio, como ruídos nos circuitos de energia, podem causar erros de memória, então você deve tentar manter as variáveis ​​iguais.

Se você tiver erros de memória, precisará identificar se é devido à memória ruim no computador ou a algum outro fator. No entanto, isso está além do escopo deste documento.

Remover dados binários do formulário #

Às vezes, as falhas ocorrem constantemente em um único formulário ou relatório, ou ocorrem apenas durante a impressão. É possível que os dados binários no formulário/relatório estejam corrompidos.

Salve o objeto Formulário/Relatório como texto Existem duas funções não documentadas. Application.SaveAsText e Application.LoadFromText. Você pode usar essas funções para exportar as definições de formulário/relatório, limpar a definição e importá-la novamente.

  1. Faça um backup do seu banco de dados antes de continuar
  2. Vá para a janela imediata do VBA (Control + G)
  3. Digite Application.SaveAsText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt" (Substitua MeuForm pelo nome do Formulário/Relatório. Use acReport se for um relatório corrompido que você está corrigindo)
  4. Renomeie o item de formulário original (por exemplo, renomeie para MyForm.Bak) na janela do banco de dados

Limpe o arquivo de definições de formulário/relatório

  1. Abra o arquivo exportado (por exemplo, MyForm.txt) no bloco de notas

  2. Exclua a linha “Checksum=” (deve estar na linha 3)

  3. Limpe os dados binários

  4. Identifique os blocos de dados binários. Examine o arquivo e você verá linhas que começam com “Parameter = Begin”. Seguindo essas linhas, você terá linhas de dados binários codificados. Finalmente, o bloco binário terminará com uma linha composta apenas por “End”. O bloco Binary Data inclui a primeira linha (com a instrução Begin) e todas as linhas até e incluindo a última linha (com a instrução End).

    Note: All of these blocks should appear BEFORE your form control definitions

  5. Exclua os blocos de dados binários para os seguintes parâmetros:

    • NameMap
    • PrtMip
    • PrtDevMode
    • PrtDevNames
    • PrtDevModeW
    • PrtDevNamesW

  6. Procure outros problemas. Com o arquivo aberto, percorra o restante do arquivo e procure por qualquer coisa que chame sua atenção, especialmente no código do módulo VBA na parte inferior. Você estará procurando por qualquer coisa que se destaque do resto e possa ser corrupção.

  7. Salve o arquivo.

Carregue o formulário/relatório de volta no Access and Test

  1. Carregue o formulário de volta no Access.
    • In Access, go to the immediate window (Control + G)
    • Type Application.LoadFromText acForm, "MyForm", CurrentProject.Path & "\MyForm.txt"
    • Decompile / Compact Repair / Recompile (See the other example within the documentation)
    • Open the form / report to test. Hopefully everything is working now.
    • Delete the old corrupt form (e.g. MyForm.bak)

Evite essa corrupção no futuro

A causa mais comum de dados binários corrompidos em um relatório/formulário é quando vários computadores/usuários usam o mesmo arquivo de cliente de banco de dados em vez de ter sua própria cópia separada. É por isso que cada usuário deve ter seu próprio arquivo de cliente em sua área de trabalho que eles executam.

Remova os campos “Objeto OLE” #

Se você tiver imagens ou outros dados armazenados no próprio Access como objetos OLE, deverá encontrar uma abordagem melhor. Quando os dados OLE são armazenados, eles são armazenados de acordo com o software (e a versão do software) no computador que os armazena. Quando outro computador vai exibir os dados do objeto OLE no formulário, mas não tem o software/versão exato instalado - muitas vezes isso resulta em uma falha do aplicativo.

Se você estiver armazenando dados de imagem, uma abordagem melhor é armazenar o nome do arquivo e, em vez disso, salvar as imagens em um local padrão. Versões mais recentes de acesso têm os controles nativos para tornar isso o caminho a seguir.

Reconstruir todo o banco de dados #

Isso é muito trabalho, então faça isso como último recurso depois de esgotar todas as outras opções. Você só precisa fazer isso se o problema estiver ocorrendo para usuários diferentes, em máquinas diferentes. Se não estiver ocorrendo para todos os usuários, provavelmente não é um contêiner de banco de dados corrompido.

Semelhante às etapas na remoção de dados binários, você reconstruirá seu banco de dados do zero. Este processo é um pouco ritualístico, mas se feito meticulosamente com cuidado para não “preservar” qualquer possível corrupção, então o processo é altamente eficaz.

Crie um novo contêiner de banco de dados de acesso.

  • No Access, na aba Arquivo, você pode selecionar “Novo”. Crie um novo banco de dados vazio no formato ACCDB.

Mova todos os objetos para o novo contêiner

Não use as funções Importar/Exportar no Access para mover os objetos e não basta clicar e arrastar. Isso pode copiar os itens corrompidos para o novo contêiner.

Tabelas:

  • Para cada tabela no contêiner de acesso antigo, crie uma nova tabela no novo contêiner.
  • Na visualização do projeto, copie/cole as definições de campo.
  • Verifique as propriedades da tabela para garantir que elas correspondam em ambos os bancos de dados
  • Mova quaisquer Macros de Dados também (veja a seção Macros para saber como fazer isso)
  • Para mover os dados, exporte os dados antigos para XML ou CSV e depois importe desse formato.

Dúvidas:

  • Carregue cada consulta na visualização SQL.
  • Copie / Cole o texto SQL.
  • Cole no novo banco de dados.
  • Compare as propriedades da consulta para garantir que elas correspondam.

Formulários/Relatórios:

  • Para cada Formulário/Relatório, utilize a função Application.SaveAsText para exportar os formulários/relatórios para um arquivo de texto.
  • Remova os dados binários (consulte a documentação Remover dados binários do formulário para se familiarizar com este processo)
  • Use a função Application.LoadFromText para reimportar os objetos para o novo banco de dados

Macros

Você tem três métodos de mover as Macros.

  1. Recrie cada macro manualmente no novo contêiner de banco de dados.
  2. Use o método Application.SaveAsText / Application.LoadFromText com o parâmetro acMacro.
  3. Copiar/colar definições de macro para cada macro
  • Selecione Todos (Control + A) para selecionar todos os elementos de macro. Em seguida, Copie (Control + C).
  • Abra um documento em branco do Bloco de Notas e Cole (Control + V) a Macro XML.
  • Crie uma nova macro em branco no novo contêiner de banco de dados.
  • No bloco de notas, selecione todo o texto (Control + A). Em seguida, copie (Control + C)
  • Na macro em branco, Cole (Control + V). A Macro deve aparecer. Salve isso.

Módulos

  • Para cada módulo, selecione todo o código (Control + A) e cole (Control + V) no novo container do banco de dados.
  • Certifique-se de verificar as propriedades do banco de dados (na janela VBA, vá em Ferramentas -> Propriedades do cliente)

Macros de dados

Para cada Macro de Dados, use os métodos SaveAsText / LoadFromText.

  1. Vá para a janela imediata do VBA (Control + G)
  2. Digite Application.SaveAsText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt" (substitua MyTableName pelo nome da tabela que contém as macros de dados)
  3. Revise o arquivo em busca de sinais de corrupção
  4. No novo contêiner de banco de dados, carregue a definição usando Application.LoadFromText acTableDataMacro, "MyTableName", CurrentProject.Path & "\MyTableName.txt"

Como mencionado anteriormente, este é um MUITO trabalho, mas tem resultados. Esse método também deve ser usado ao migrar um banco de dados do Access 97 para 2000 ou um banco de dados do Access 2000 para 2003.
