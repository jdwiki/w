Injeção de dependência com Guice - Play 2.4, 2.5 #

Guice é a estrutura de injeção de dependência padrão (mais DI) do Play. Outros frameworks também podem ser usados, mas usar o Guice facilita os esforços de desenvolvimento, já que o Play cuida das coisas sob o véu.

Injeção de APIs do Google Play #

A partir do Play 2.5, várias APIs, que eram estáticas nas versões anteriores, devem ser criadas com DI. São, por exemplo, Configuration, JPAApi, CacheApi, etc.

O método de injeção de API-s do Play é diferente para uma classe, que é injetada automaticamente pelo Play e para uma classe personalizada. A injeção em uma classe injetada automaticamente é tão simples quanto colocar a anotação @Inject apropriada em qualquer campo ou construtor. Por exemplo, para injetar Configuration em um controller com injeção de propriedade:

@Inject 
private Configuration configuration;

ou com injeção de construtor:

private Configuration configuration;
@Inject
public MyController(Configuration configuration) {
    this.configuration = configuration;
}

A injeção em uma classe personalizada, que é registrada para DI, deve ser feita da mesma forma que é feita para uma classe injetada automaticamente - com a anotação @Inject.

A injeção de uma classe custom, que não está vinculada a DI, deve ser feita por uma chamada explícita para um injetor com Play.current().injector(). Por exemplo, para injetar Configuration em uma classe personalizada, defina um membro de dados de configuração como este:

private Configuration configuration = Play.current().injector().instanceOf(Configuration.class);

Vinculação de injeção personalizada #

A vinculação de injeção personalizada pode ser feita com a anotação @ImplementedBy ou de forma programática com o módulo Guice.

Injeção com anotação @ImplementedBy #

A injeção com a anotação @ImplementedBy é a maneira mais simples. O exemplo abaixo mostra um serviço que fornece uma fachada para cache.

  1. O serviço é definido por uma interface CacheProvider da seguinte forma:

    @ImplementedBy(RunTimeCacheProvider.class)
public interface CacheProvider {  
   CacheApi getCache();  
}

  2. O serviço é implementado por uma classe RunTimeCacheProvider:

    public class RunTimeCacheProvider implements CacheProvider {  
   @Inject  
   private CacheApi appCache;  
   @Override  
   public public CacheApi getCache() {  
     return appCache;  
   }  
}

Observação: o membro de dados appCache é injetado na criação de uma instância RunTimeCacheProvider. 3. O inspetor de cache é definido como um membro de um controlador com anotação @Inject e é chamado do controlador:

public class HomeController extends Controller {
  @Inject
  private CacheProvider cacheProvider;
  ...      
  public Result getCacheData() {
        Object cacheData = cacheProvider.getCache().get("DEMO-KEY");
        return ok(String.format("Cache content:%s", cacheData));  
  }

A injeção com a anotação @ImplementedBy cria a ligação fixa: CacheProvider no exemplo acima é sempre instanciado com RunTimeCacheProvider. Tal método serve apenas para um caso, quando há uma interface com uma única implementação. Não pode ajudar uma interface com várias implementações ou uma classe implementada como singleton sem interface abstrata. Honestamente falando, @ImplementedBy será usado em casos raros, se for tudo. É mais provável que use ligação programática com o módulo Guice.

Ligação de injeção com um módulo Play padrão #

O módulo Play padrão é uma classe chamada Module no diretório raiz do projeto definido assim:

import com.google.inject.AbstractModule;  
public class Module extends AbstractModule {  
  @Override  
  protected void configure() {  
      // bindings are here           
  }  
}

Observação: O trecho acima mostra a vinculação dentro do configure, mas é claro que qualquer outro método de vinculação será respeitado.

Para vinculação programática de CacheProvider a RunTimeCacheProvider:

  1. Remova a anotação @ImplementedBy da definição de CacheProvider:

    public interface CacheProvider {
    CacheApi getCache();
    }

  2. Implemente o módulo configure da seguinte forma:

    public class Module extends AbstractModule {
    @Override
    protected void configure() {
    bind(CacheProvider.class).to(RunTimeCacheProvider.class); }
    }

Ligação de injeção flexível com um módulo Play padrão

RunTimeCacheProvider não funciona bem em testes JUnit com aplicação falsa (veja o tópico de testes unitários). Assim, a implementação diferente de CacheProvider é exigida para testes de unidade. A ligação por injeção deve ser feita de acordo com o ambiente.

Vamos ver um exemplo.

  1. A classe FakeCache fornece uma implementação stub de CacheApi para ser usada durante a execução de testes (sua implementação não é tão interessante - é apenas um mapa).

  2. A classe FakeCacheProvider implementa CacheProvider para ser usado durante a execução de testes:

    public class FakeCacheProvider implements CacheProvider {
    private final CacheApi fakeCache = new FakeCache(); @Override
    public CacheApi getCache() {
    return fakeCache; }
    }

  3. O módulo é implementado da seguinte forma:

    public class Module extends AbstractModule { private final Environment environment; public Module(Environment environment, Configuration configuration) { this.environment = environment; } @Override
    protected void configure() {
    if (environment.isTest() ) { bind(CacheProvider.class).to(FakeCacheProvider.class); } else { bind(CacheProvider.class).to(RuntimeCacheProvider.class);
    } }
    }
    O exemplo é bom apenas para fins educacionais. A vinculação para testes dentro do módulo não é a melhor prática, pois isso acopla aplicativo e testes. A vinculação para testes deve ser feita pelos próprios testes e o módulo não deve estar ciente da implementação específica do teste. Veja como fazer isso melhor em … .

Ligação de injeção com um módulo personalizado #

Um módulo personalizado é muito semelhante ao módulo Play padrão. A diferença é que pode ter qualquer nome e pertencer a qualquer pacote. Por exemplo, um módulo OnStartupModule pertence aos módulos do pacote.

package modules;  
import com.google.inject.AbstractModule;  
public class OnStartupModule extends AbstractModule {  
   @Override  
   protected void configure() {  
       ...            
   }  
}

Um módulo personalizado deve ser explicitamente habilitado para invocação pelo Play. Para o módulo OnStartupModule o seguinte deve ser adicionado em application.conf:

play.modules.enabled += "modules.OnStartupModule"
