O núcleo do Aspnet é construído com injeção de dependência como um de seus principais conceitos principais. Ele apresenta uma abstração de contêiner em conformidade para que você possa substituir o interno por um contêiner de terceiros de sua escolha.

Sintaxe #

  • IServiceCollection.Add(ServiceDescriptor item);
  • IServiceCollection.AddScoped(Type serviceType);
  • IServiceCollection.AddScoped(Tipo tipo de serviço, Tipo de implementação);
  • IServiceCollection.AddScoped(Type serviceType, Func<IServiceProvider, object> deploymentFactory);
  • IServiceCollection.AddScoped<TService>()
  • IServiceCollection.AddScoped<TService>(Func<IServiceProvider, TService> deploymentFactory)
  • IServiceCollection.AddScoped<TService, TImplementation>()
  • IServiceCollection.AddScoped<TService, TImplementation>(Func<IServiceProvider, TImplementation> deploymentFactory)
  • IServiceCollection.AddSingleton(Tipo tipo de serviço);
  • IServiceCollection.AddSingleton(Tipo serviceType, Func<IServiceProvider, object> deploymentFactory);
  • IServiceCollection.AddSingleton(Tipo tipo de serviço, Tipo de implementação);
  • IServiceCollection.AddSingleton(Tipo tipo de serviço, implementação de objetoInstance);
  • IServiceCollection.AddSingleton<TService>()
  • IServiceCollection.AddSingleton<TService>(Func<IServiceProvider, TService> deploymentFactory)
  • IServiceCollection.AddSingleton<TService>(TService ImplementationInstance)
  • IServiceCollection.AddSingleton<TService, TImplementation>()
  • IServiceCollection.AddSingleton<TService, TImplementation>(Func<IServiceProvider, TImplementation> deploymentFactory)
  • IServiceCollection.AddTransient(Type serviceType);
  • IServiceCollection.AddTransient(Tipo serviceType, Func<IServiceProvider, object> deploymentFactory);
  • IServiceCollection.AddTransient(Tipo tipo de serviço, Tipo de implementação);
  • IServiceCollection.AddTransient<TService>()
  • IServiceCollection.AddTransient<TService>(Func<IServiceProvider, TService> deploymentFactory)
  • IServiceCollection.AddTransient<TService, TImplementation>()
  • IServiceCollection.AddTransient<TService, TImplementation>(Func<IServiceProvider, TImplementation> deploymentFactory)
  • IServiceProvider.GetService(Type serviceType)
  • IServiceProvider.GetService<T>()
  • IServiceProvider.GetServices(Type serviceType)
  • IServiceProvider.GetServices<T>()

Para usar variantes genéricas dos métodos IServiceProvider, você deve incluir o seguinte namespace:

using Microsoft.Extensions.DependencyInjection;

Registrar dependências #

O contêiner interno vem com um conjunto de recursos internos:

Controle vitalício #

public void ConfigureServices(IServiceCollection services)
    {
        // ...
    
        services.AddTransient<ITestService, TestService>();
        // or
        services.AddScoped<ITestService, TestService>();
        // or
        services.AddSingleton<ITestService, TestService>();
        // or
        services.AddSingleton<ITestService>(new TestService());
    }
  • AddTransient: Criado sempre que é resolvido
  • AddScoped: criado uma vez por solicitação
  • AddSingleton: preguiçosamente criado uma vez por aplicativo
  • AddSingleton (instance): fornece uma instância criada anteriormente por aplicativo

Dependências enumeráveis #

Também é possível registrar dependências enumeráveis:

 services.TryAddEnumerable(ServiceDescriptor.Transient<ITestService, TestServiceImpl1>());
 services.TryAddEnumerable(ServiceDescriptor.Transient<ITestService, TestServiceImpl2>());

Você pode consumi-los da seguinte forma:

public class HomeController : Controller
{
    public HomeController(IEnumerable<ITestService> services)
    {
        // do something with services.
    }
}

Dependências genéricas #

Você também pode registrar dependências genéricas:

services.Add(ServiceDescriptor.Singleton(typeof(IKeyValueStore<>), typeof(KeyValueStore<>)));

E, em seguida, consumi-lo da seguinte forma:

public class HomeController : Controller
{
    public HomeController(IKeyValueStore<UserSettings> userSettings)
    {
        // do something with services.
    }
}

Registre-se e resolva manualmente #

A maneira preferida de descrever dependências é usando a injeção de construtor que segue o Princípio de Dependências Explícitas:

ITestService.cs

public interface ITestService
{
    int GenerateRandom();
}

TestService.cs

public class TestService : ITestService
{
    public int GenerateRandom()
    {
        return 4;
    }
}

Startup.cs (ConfigureServices)

public void ConfigureServices(IServiceCollection services)
{
    // ...

    services.AddTransient<ITestService, TestService>();
}

HomeController.cs

using Microsoft.Extensions.DependencyInjection;

namespace Core.Controllers
{
    public class HomeController : Controller
    {
        public HomeController(ITestService service)
        {
            int rnd = service.GenerateRandom();
        }
    }
}

Injetando uma dependência em uma ação do controlador #

Um recurso interno menos conhecido é a injeção de ação do controlador usando o FromServicesAttribute.

[HttpGet]
public async Task<IActionResult> GetAllAsync([FromServices]IProductService products)
{
     return Ok(await products.GetAllAsync());
}

Uma observação importante é que o [FromServices] não pode ser usado como mecanismo geral de “Injeção de propriedade” ou “injeção de método”! Ele só pode ser usado em parâmetros de método de uma ação de controlador ou construtor de controlador (no construtor é obsoleto, pois o sistema ASP.NET Core DI já usa injeção de construtor e não há marcadores extras necessários).

Não pode ser usado em qualquer lugar fora de controladores, ação do controlador. Também é muito específico para ASP.NET Core MVC e reside no assembly Microsoft.AspNetCore.Mvc.Core.

Citação original do problema do ASP.NET Core MVC GitHub (Limit [FromServices] to apply only to parameters) em relação a este atributo:

@rynowak:

@Eilon:

O problema com propriedades é que parece para muitas pessoas que ela pode ser aplicada a qualquer propriedade de qualquer objeto.

De acordo, tivemos vários problemas postados por usuários com confusão sobre como esse recurso deve ser usado. Houve realmente uma quantidade bastante grande de comentários dos dois tipos “[FromServices] é estranho e eu não gosto disso” e “[FromServices] me confundiu”. Parece uma armadilha e algo sobre o qual a equipe ainda estaria respondendo a perguntas daqui a alguns anos.

Nós sentimos que o cenário mais valioso para [FromServices] está no parâmetro do método para uma ação para um serviço que você só precisa naquele lugar.

/cc @danroth27 - alterações de documentos

Para qualquer um apaixonado pelo atual [FromServices] , eu recomendo fortemente procurar um sistema de DI que possa fazer injeção de propriedades (Autofac, por exemplo).

Notas:

  • Qualquer serviço registrado com o sistema .NET Core Dependency Injection pode ser injetado dentro de uma ação do controlador usando o atributo [FromServices].
  • O caso mais relevante é quando você precisa de um serviço apenas em um único método de ação e não quer sobrecarregar o construtor do seu controlador com outra dependência, que será usada apenas uma vez.
  • Não pode ser usado fora do ASP.NET Core MVC (ou seja, aplicativos de console .NET Framework ou .NET Core puros), porque reside no assembly Microsoft.AspNetCore.Mvc.Core.
  • Para injeção de propriedade ou método, você deve usar um dos contêineres IoC de terceiros disponíveis (Autofac, Unity, etc.).

Usando serviços com escopo durante a inicialização do aplicativo / Semeadura de banco de dados #

A resolução de serviços com escopo durante a inicialização do aplicativo pode ser difícil, pois não há solicitação e, portanto, nenhum serviço com escopo definido.

Resolver um serviço com escopo durante a inicialização do aplicativo por meio de app.ApplicationServices.GetService<AppDbContext>() pode causar problemas, porque ele será criado no escopo do contêiner global, tornando-o efetivamente um singleton com o tempo de vida do aplicativo, que pode levar a exceções como Não é possível acessar um objeto descartado no ASP.NET Core ao injetar DbContext.

O padrão a seguir resolve o problema criando primeiro um novo escopo e, em seguida, resolvendo os serviços com escopo a partir dele e, depois que o trabalho estiver concluído, descartando o contêiner com escopo definido.

public Configure(IApplicationBuilder app)
{
    // serviceProvider is app.ApplicationServices from Configure(IApplicationBuilder app) method
    using (var serviceScope = app.ApplicationServices.GetRequiredService<IServiceScopeFactory>().CreateScope())
    {
        var db = serviceScope.ServiceProvider.GetService<AppDbContext>();

        if (await db.Database.EnsureCreatedAsync())
        {
            await SeedDatabase(db);
        }
    }
}

Essa é uma maneira semi-oficial da equipe principal do Entity Framework para propagar dados durante a inicialização do aplicativo e é refletida no aplicativo MusicStore sample.

Recuperar dependências em um Controller #

Uma vez registrada, uma dependência pode ser recuperada adicionando parâmetros no construtor Controller.

// ...
using System;
using Microsoft.Extensions.DependencyInjection;

namespace Core.Controllers
{
    public class HomeController : Controller
    {
        public HomeController(ITestService service)
        {
            int rnd = service.GenerateRandom();
        }
    }
}

O padrão Options / Injetando opções em serviços #

Com o ASP.NET Core, a equipe da Microsoft também introduziu o padrão Options, que permite ter opções digitadas fortes e, uma vez configurada, a capacidade de injetar as opções em seus serviços.

Primeiro começamos com uma classe de tipagem forte, que manterá nossa configuração.

public class MySettings 
{
    public string Value1 { get; set; }
    public string Value2 { get; set; }
}

E uma entrada no appsettings.json.

{
  "mysettings" : {
    "value1": "Hello",
    "value2": "World"
  }
}

Em seguida, inicializamos na classe Startup. Existem duas maneiras de fazer isso

  1. Carregue-o diretamente da seção appsettings.json “mysettings”

    services.Configure<MySettings>(Configuration.GetSection("mysettings"));

  2. Faça manualmente

    services.Configure<MySettings>(new MySettings 
{
    Value1 = "Hello",
    Value2 = Configuration["mysettings:value2"]
});

Cada nível de hierarquia do appsettings.json é separado por um :. Como value2 é uma propriedade do objeto mysettings, nós o acessamos via mysettings:value2.

Finalmente podemos injetar as opções em nossos serviços, usando a interface IOptions<T>

public class MyService : IMyService
{
    private readonly MySettings settings;

    public MyService(IOptions<MySettings> mysettings) 
    {
        this.settings = mySettings.Value;
    }
}

Observações #

Se o IOptions<T> não for configurado durante a inicialização, injetar IOptions<T> injetará a instância padrão da classe T.

Resolver controladores, ViewComponents e TagHelpers via injeção de dependência #

Por padrão, os Controllers, ViewComponents e TagHelpers não são registrados e resolvidos por meio do contêiner de injeção de dependência. Isso resulta na incapacidade de fazer, ou seja, injeção de propriedade ao usar um contêiner de Inversão de Controle (IoC) de terceiros como o AutoFac.

Para fazer com que o ASP.NET Core MVC também resolva esses tipos via IoC, é necessário adicionar os seguintes registros no Startup.cs (retirado do [amostra ControllersFromService] 1 oficial no GitHub)

public void ConfigureServices(IServiceCollection services)
{
    var builder = services
        .AddMvc()
        .ConfigureApplicationPartManager(manager => manager.ApplicationParts.Clear())
        .AddApplicationPart(typeof(TimeScheduleController).GetTypeInfo().Assembly)
        .ConfigureApplicationPartManager(manager =>
        {
            manager.ApplicationParts.Add(new TypesPart(
              typeof(AnotherController),
              typeof(ComponentFromServicesViewComponent),
              typeof(InServicesTagHelper)));

            manager.FeatureProviders.Add(new AssemblyMetadataReferenceFeatureProvider());
        })
        .AddControllersAsServices()
        .AddViewComponentsAsServices()
        .AddTagHelpersAsServices();

    services.AddTransient<QueryValueService>();
    services.AddTransient<ValueService>();
    services.AddSingleton<IHttpContextAccessor, HttpContextAccessor>();
}

Exemplo de injeção de dependência simples (sem Startup.cs) #

Isso mostra como usar o pacote nuget Microsoft.Extensions.DependencyInjection sem o uso do WebHostBuilder do kestrel (por exemplo, quando você quiser para construir outra coisa do que um webApp):

internal class Program
{
    public static void Main(string[] args)
    {
        var services = new ServiceCollection(); //Creates the service registry
        services.AddTransient<IMyInterface, MyClass>(); //Add registration of IMyInterface (should create an new instance of MyClass every time)
        var serviceProvider = services.BuildServiceProvider(); //Build dependencies into an IOC container
        var implementation = serviceProvider.GetService<IMyInterface>(); //Gets a dependency

        //serviceProvider.GetService<ServiceDependingOnIMyInterface>(); //Would throw an error since ServiceDependingOnIMyInterface is not registered
        var manualyInstaniate = new ServiceDependingOnIMyInterface(implementation); 

        services.AddTransient<ServiceDependingOnIMyInterface>();
        var spWithService = services.BuildServiceProvider(); //Generaly its bad practise to rebuild the container because its heavey and promotes use of anti-pattern.
        spWithService.GetService<ServiceDependingOnIMyInterface>(); //only now i can resolve
    }
}

interface IMyInterface
{
}

class MyClass : IMyInterface
{
}

class ServiceDependingOnIMyInterface
{
    private readonly IMyInterface _dependency;

    public ServiceDependingOnIMyInterface(IMyInterface dependency)
    {
        _dependency = dependency;
    }
}

Funcionamento interno de Microsoft.Extensions.DependencyInjection #

IServiceCollection #

Para começar a construir um contêiner IOC com o pacote nuget DI da Microsoft, você começa criando um IServiceCollection. Você pode usar a coleção já fornecida: ServiceCollection:

var services = new ServiceCollection();

Este IServiceCollection nada mais é do que uma implementação de: IList<ServiceDescriptor>, ICollection<ServiceDescriptor>, IEnumerable<ServiceDescriptor>, IEnumerable

Todos os métodos a seguir são apenas métodos de extensão para adicionar instâncias ServiceDescriptor à lista:

services.AddTransient<Class>(); //add registration that is always recreated
services.AddSingleton<Class>(); // add registration that is only created once and then re-used
services.AddTransient<Abstract, Implementation>(); //specify implementation for interface
services.AddTransient<Interface>(serviceProvider=> new Class(serviceProvider.GetService<IDependency>())); //specify your own resolve function/ factory method.
services.AddMvc(); //extension method by the MVC nuget package, to add a whole bunch of registrations.
// etc..

//when not using an extension method:
services.Add(new ServiceDescriptor(typeof(Interface), typeof(Class)));

IServiceProvider #

O serviceprovider é aquele que ‘Compila’ todos os registros para que possam ser usados ​​rapidamente, isso pode ser feito com services.BuildServiceProvider() que é basicamente um método de extensão para:

var provider = new ServiceProvider( services, false); //false is if it should validate scopes

Nos bastidores, cada ServiceDescriptor no IServiceCollection é compilado para um método de fábrica Func<ServiceProvider, object> onde o objeto é o tipo de retorno e é: a instância criada do tipo Implementação, o Singleton ou sua própria fábrica definida método.

Esses registros são adicionados ao ServiceTable que é basicamente um ConcurrentDictionary com a chave sendo o ServiceType e o valor o método Factory definido acima.

Resultado #

Agora temos um ConcurrentDictionary<Type, Func<ServiceProvider, object>> que podemos usar simultaneamente para solicitar a criação de Serviços para nós. Para mostrar um exemplo básico de como isso poderia ter sido.

  var serviceProvider = new ConcurrentDictionary<Type, Func<ServiceProvider, object>>();
  var factoryMethod = serviceProvider[typeof(MyService)];
  var myServiceInstance = factoryMethod(serviceProvider)

Não é assim que funciona!

Este ConcurrentDictionary é uma propriedade do ServiceTable que é uma propriedade do ServiceProvider
