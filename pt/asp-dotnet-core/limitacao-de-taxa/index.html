AspNetCoreRateLimit é uma solução de limitação de taxa ASP.NET Core de código aberto projetada para controlar a taxa de solicitações que os clientes podem fazer a uma API Web ou aplicativo MVC com base no endereço IP ou ID do cliente.

Limitação de taxa com base no ID do cliente #

Com o middleware ClientRateLimit, você pode definir vários limites para diferentes cenários, como permitir que um cliente faça um número máximo de chamadas em um intervalo de tempo como por segundo, 15 minutos, etc. Você pode definir esses limites para atender a todas as solicitações feitas a uma API ou pode definir os limites para cada caminho de URL ou verbo e caminho HTTP.

Configurar #

Instalação do NuGet:

Instalar-Pacote AspNetCoreRateLimit

Código Startup.cs:

public void ConfigureServices(IServiceCollection services)
{
    // needed to load configuration from appsettings.json
    services.AddOptions();

    // needed to store rate limit counters and ip rules
    services.AddMemoryCache();

    //load general configuration from appsettings.json
    services.Configure<ClientRateLimitOptions>(Configuration.GetSection("ClientRateLimiting"));

    //load client rules from appsettings.json
    services.Configure<ClientRateLimitPolicies>(Configuration.GetSection("ClientRateLimitPolicies"));

    // inject counter and rules stores
    services.AddSingleton<IClientPolicyStore, MemoryCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore, MemoryCacheRateLimitCounterStore>();

    // Add framework services.
    services.AddMvc();
}

public void Configure(IApplicationBuilder app, IHostingEnvironment env, ILoggerFactory loggerFactory)
{
    loggerFactory.AddConsole(Configuration.GetSection("Logging"));
    loggerFactory.AddDebug();

    app.UseClientRateLimiting();

    app.UseMvc();
}

Você deve registrar o middleware antes de qualquer outro componente, exceto loggerFactory.

se você balancear a carga do seu aplicativo, precisará usar IDistributedCache com Redis ou SQLServer para que todas as instâncias do Kestrel tenham o mesmo armazenamento de limite de taxa. Em vez dos armazenamentos na memória, você deve injetar os armazenamentos distribuídos assim:

    // inject counter and rules distributed cache stores
    services.AddSingleton<IClientPolicyStore, DistributedCacheClientPolicyStore>();
    services.AddSingleton<IRateLimitCounterStore,DistributedCacheRateLimitCounterStore>();

Configuração e regras gerais appsettings.json:

  "ClientRateLimiting": {
    "EnableEndpointRateLimiting": false,
    "StackBlockedRequests": false,
    "ClientIdHeader": "X-ClientId",
    "HttpStatusCode": 429,
    "EndpointWhitelist": [ "get:/api/license", "*:/api/status" ],
    "ClientWhitelist": [ "dev-id-1", "dev-id-2" ],
    "GeneralRules": [
      {
        "Endpoint": "*",
        "Period": "1s",
        "Limit": 2
      },
      {
        "Endpoint": "*",
        "Period": "15m",
        "Limit": 100
      },
      {
        "Endpoint": "*",
        "Period": "12h",
        "Limit": 1000
      },
      {
        "Endpoint": "*",
        "Period": "7d",
        "Limit": 10000
      }
    ]
  }

Se EnableEndpointRateLimiting for definido como false, os limites serão aplicados globalmente e somente as regras que tiverem como endpoint * serão aplicadas. Por exemplo, se você definir um limite de 5 chamadas por segundo, qualquer chamada HTTP para qualquer endpoint contará para esse limite.

Se EnableEndpointRateLimiting for definido como true, os limites serão aplicados para cada endpoint como em {HTTP_Verb}{PATH}. Por exemplo, se você definir um limite de 5 chamadas por segundo para *:/api/values um cliente pode chamar GET /api/values 5 vezes por segundo, mas também 5 vezes PUT /api/values.

Se StackBlockedRequests for definido como false, as chamadas rejeitadas não serão adicionadas ao contador de aceleração. Se um cliente fizer 3 solicitações por segundo e você tiver definido um limite de uma chamada por segundo, outros limites como contadores por minuto ou por dia registrarão apenas a primeira chamada, aquela que não foi bloqueada. Se você quiser que as solicitações rejeitadas sejam contabilizadas para os outros limites, você terá que definir StackBlockedRequests como true.

O ClientIdHeader é usado para extrair o ID do cliente, se um ID do cliente estiver presente neste cabeçalho e corresponder a um valor especificado em ClientWhitelist, nenhum limite de taxa será aplicado.

Substituir regras gerais para clientes específicos appsettings.json:

 "ClientRateLimitPolicies": {
    "ClientRules": [
      {
        "ClientId": "client-id-1",
        "Rules": [
          {
            "Endpoint": "*",
            "Period": "1s",
            "Limit": 10
          },
          {
            "Endpoint": "*",
            "Period": "15m",
            "Limit": 200
          }
        ]
      },
      {
        "Client": "client-id-2",
        "Rules": [
          {
            "Endpoint": "*",
            "Period": "1s",
            "Limit": 5
          },
          {
            "Endpoint": "*",
            "Period": "15m",
            "Limit": 150
          },
          {
            "Endpoint": "*",
            "Period": "12h",
            "Limit": 500
          }
        ]
      }
    ]
  }

Definindo regras de limite de taxa #

Uma regra é composta por um endpoint, um período e um limite.

O formato do endpoint é {HTTP_Verb}:{PATH}, você pode direcionar qualquer verbo HTTP usando o símbolo de asterisco.

O formato do período é {INT}{PERIOD_TYPE}, você pode usar um dos seguintes tipos de período: s, m, h, d.

O formato do limite é {LONG}.

Exemplos:

Limite de taxa todos os endpoints a 2 chamadas por segundo:

{
 "Endpoint": "*",
 "Period": "1s",
 "Limit": 2
}

Se no mesmo segundo, um cliente fizer 3 chamadas GET para api/values, a última chamada será bloqueada. Mas se no mesmo segundo ele chamar PUT api/values ​​também, a requisição passará porque é um endpoint diferente. Quando a limitação de taxa de endpoint está habilitada, cada chamada é limitada por taxa com base em {HTTP_Verb}{PATH}.

Taxa de limite de chamadas com qualquer verbo HTTP para /api/values para 5 chamadas por 15 minutos:

{
 "Endpoint": "*:/api/values",
 "Period": "15m",
 "Limit": 5
}

Limite de taxa de chamada GET para /api/values para 5 chamadas por hora:

{
 "Endpoint": "get:/api/values",
 "Period": "1h",
 "Limit": 5
}

Se em uma hora, um cliente fizer 6 chamadas GET para api/values, a última chamada será bloqueada. Mas se na mesma hora ele chamar GET api/values/1 também, a requisição passará porque é um endpoint diferente.

Comportamento #

Quando um cliente faz uma chamada HTTP, o ClientRateLimitMiddleware faz o seguinte:

  • extrai o ID do cliente, verbo HTTP e URL do objeto de solicitação, se você quiser implementar sua própria lógica de extração, você pode substituir o ClientRateLimitMiddleware.SetIdentity
  • procura o ID do cliente e o URL nas listas brancas, se houver correspondência, nenhuma ação será tomada
  • pesquisa nas regras do Cliente por uma partida, todas as regras que se aplicam são agrupadas por período, para cada período é usada a regra mais restritiva
  • pesquisa nas Regras Gerais por uma partida, se uma regra geral que corresponde tem um período definido que não está presente nas regras do Cliente então esta regra geral também é usada
  • para cada regra correspondente, o contador de limite de taxa é incrementado, se o valor do contador for maior que o limite de regra, a solicitação será bloqueada

Se a solicitação for bloqueada, o cliente receberá uma resposta de texto como esta:

Status Code: 429
Retry-After: 58
Content: API calls quota exceeded! maximum admitted 2 per 1m.

Você pode personalizar a resposta alterando essas opções HttpStatusCode e QuotaExceededMessage, se você quiser implementar sua própria resposta você pode substituir o ClientRateLimitMiddleware.ReturnQuotaExceededResponse. O valor do cabeçalho Retry-After é expresso em segundos.

Se a solicitação não for limitada à taxa, o período mais longo definido nas regras de correspondência é usado para compor os cabeçalhos X-Rate-Limit, esses cabeçalhos são injetados na resposta:

X-Rate-Limit-Limit: the rate limit period (eg. 1m, 12h, 1d)
X-Rate-Limit-Remaining: number of request remaining 
X-Rate-Limit-Reset: UTC date time when the limits resets

Por padrão, solicitações bloqueadas são registradas usando Microsoft.Extensions.Logging.ILogger, se você quiser implementar seu próprio registro, você pode substituir o ClientRateLimitMiddleware.LogBlockedRequest. O registrador padrão emite as seguintes informações quando uma solicitação é limitada por taxa:

info: AspNetCoreRateLimit.ClientRateLimitMiddleware[0]
      Request get:/api/values from ClientId client-id-1 has been blocked, quota 2/1m exceeded by 3. Blocked by rule *:/api/value, TraceIdentifier 0HKTLISQQVV9D.

Atualizar limites de taxa em tempo de execução #

Na inicialização do aplicativo, as regras de limite de taxa do cliente definidas em appsettings.json são carregadas no cache por MemoryCacheClientPolicyStore ou DistributedCacheClientPolicyStore dependendo do tipo de provedor de cache que você está usando. Você pode acessar o armazenamento de política do cliente dentro de um controlador e modificar as regras da seguinte forma:

public class ClientRateLimitController : Controller
{
    private readonly ClientRateLimitOptions _options;
    private readonly IClientPolicyStore _clientPolicyStore;

    public ClientRateLimitController(IOptions<ClientRateLimitOptions> optionsAccessor, IClientPolicyStore clientPolicyStore)
    {
        _options = optionsAccessor.Value;
        _clientPolicyStore = clientPolicyStore;
    }

    [HttpGet]
    public ClientRateLimitPolicy Get()
    {
        return _clientPolicyStore.Get($"{_options.ClientPolicyPrefix}_cl-key-1");
    }

    [HttpPost]
    public void Post()
    {
        var id = $"{_options.ClientPolicyPrefix}_cl-key-1";
        var clPolicy = _clientPolicyStore.Get(id);
        clPolicy.Rules.Add(new RateLimitRule
        {
            Endpoint = "*/api/testpolicyupdate",
            Period = "1h",
            Limit = 100
        });
        _clientPolicyStore.Set(id, clPolicy);
    }
}

Dessa forma, você pode armazenar os limites de taxa do cliente em um banco de dados e enviá-los para o cache após o início de cada aplicativo.

Atualizar limites de taxa em tempo de execução #

Na inicialização do aplicativo, as regras de limite de taxa de IP definidas em appsettings.json são carregadas no cache por MemoryCacheClientPolicyStore ou DistributedCacheIpPolicyStore dependendo do tipo de provedor de cache que você está usando. Você pode acessar o armazenamento de política de IP dentro de um controlador e modificar as regras de IP da seguinte forma:

public class IpRateLimitController : Controller
{
    private readonly IpRateLimitOptions _options;
    private readonly IIpPolicyStore _ipPolicyStore;

    public IpRateLimitController(IOptions<IpRateLimitOptions> optionsAccessor, IIpPolicyStore ipPolicyStore)
    {
        _options = optionsAccessor.Value;
        _ipPolicyStore = ipPolicyStore;
    }

    [HttpGet]
    public IpRateLimitPolicies Get()
    {
        return _ipPolicyStore.Get(_options.IpPolicyPrefix);
    }

    [HttpPost]
    public void Post()
    {
        var pol = _ipPolicyStore.Get(_options.IpPolicyPrefix);

        pol.IpRules.Add(new IpRateLimitPolicy
        {
            Ip = "8.8.4.4",
            Rules = new List<RateLimitRule>(new RateLimitRule[] {
                new RateLimitRule {
                    Endpoint = "*:/api/testupdate",
                    Limit = 100,
                    Period = "1d" }
            })
        });

        _ipPolicyStore.Set(_options.IpPolicyPrefix, pol);
    }
}

Dessa forma, você pode armazenar os limites de taxa de IP em um banco de dados e enviá-los para o cache após o início de cada aplicativo.
