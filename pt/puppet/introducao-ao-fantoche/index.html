O que é fantoche e por que devo me importar? #

Puppet é uma solução de gerenciamento de configuração. Os usuários descrevem o estado desejado de um servidor ou software e o gerenciamento de configuração atinge esse estado. Isso traz as seguintes vantagens:

  • As configurações podem ser reproduzidas exatamente da mesma forma todas as vezes, quantas vezes forem necessárias
  • As configurações de todos os softwares e servidores são armazenadas em um local central. Isso torna o backup e o controle de versão das configurações facilmente alcançáveis
  • As alterações em todos os servidores se propagam por toda a infraestrutura em alguns minutos, sem precisar fazer login em nenhuma máquina diretamente
  • Tudo é descrito na mesma linguagem, facilitando a configuração de novos softwares
  • Os módulos são semelhantes às bibliotecas e permitem que as configurações sejam consolidadas. Módulos para todos os principais pacotes de software já existem, tornando a instalação extremamente fácil
  • Os servidores podem compartilhar informações entre si, influenciando na configuração de outros servidores. Por exemplo, um novo servidor pode se registrar automaticamente no balanceador de carga e na solução de monitoramento

Puppet usa Ruby para descrever o estado desejado de um servidor, chamado de nó. Ele faz isso com o uso de primitivos chamados tipos de recursos. Por padrão, a cada 30 minutos, o agente do fantoche se autentica no servidor do fantoche. Em seguida, ele envia uma lista de propriedades de si mesmo chamada fatos. O servidor analisa os fatos e os arquivos de configuração chamados manifestos e compila o estado desejado para o nó. Em seguida, ele envia essa configuração de volta ao nó, onde o agente a aplica.

Para dar uma ideia de quão poderoso isso pode ser, aqui estão alguns exemplos de complexidade crescente mostrando o que o fantoche pode fazer por você.

Exemplo: usuário

Este exemplo cria o usuário username no nó myserver e o adiciona ao grupo wheel.

node 'myserver' {
    user { 'username':
      ensure => 'present',
      groups => ['wheel'],
    }
}

Este arquivo que seria armazenado no servidor da marionete é o manifesto. O tipo de recurso neste exemplo é usuário. Cada tipo de recurso tem propriedades opcionais e obrigatórias. Neste exemplo, garantir é obrigatório e grupos é opcional. Essa configuração específica só seria aplicada a myserver. Você pode aplicar configurações a todos os nós colocando-os fora de uma definição de nó.

É possível pegar algumas definições de recursos e armazená-las como módulos. Um módulo é semelhante a uma biblioteca. Esses módulos podem ser compartilhados on-line e você geralmente encontra um para cada pacote de software principal. A maneira oficial de compartilhar módulos é através da forja de marionetes: https://forge.puppet.com/

Exemplo: Postgres

Este exemplo instala um servidor postgres no nó myserver e cria um banco de dados db, de propriedade de username, identificado por password. Ele faz isso usando o módulo postgresql.

node 'myserver' {
    class { 'postgresql::server': }
    
    postgresql::server::db { 'db':
        user     => 'username',
        password => 'password',
    }
}

Neste caso postgresql é um módulo. O próprio módulo se encarrega de identificar o sistema operacional, baixar e instalar o programa e depois configurá-lo de acordo com o manifesto. Este é um exemplo básico, mas o módulo permite uma grande personalização.

Observe que não é necessário saber SQL ou como instalar um servidor postgres para fazê-lo. Os módulos oficiais são bem mantidos e fornecem uma configuração de base sã e segura.

Também é possível usar fatos em manifestos. Os fatos agem como variáveis.

Exemplo: Condições usando fatos

Este exemplo usa o módulo rsyslog para configurar o rsyslog em todas as máquinas que não são Windows.

if $osfamily != 'windows' {
  class { 'rsyslog::client': }
}

$osfamily é um fato. Esses fatos são coletados toda vez que o agente fantoche é executado. Observe que, como essa definição está fora de uma definição de nó, ela é aplicada a todos os nós. No entanto, rsyslog::client só será executado em nós que não executam janelas.

Como o puppet usa Ruby, elementos programáticos como fluxos de controle e variáveis ​​podem ser usados ​​em manifestos.

Com a adição do PuppetDB, você pode compartilhar informações entre vários nós. Isso permite que um nó influencie a configuração em um nó diferente. Exemplos clássicos incluem balanceadores de carga ou soluções de monitoramento.

Exemplo: registrar um host com monitoramento usando recursos exportados

Este exemplo cria um recurso exportado em um nó e, em seguida, importa esse recurso no servidor de monitoramento, incluindo o host no monitoramento. Ele está usando o módulo de marionetes Icinga2.

@@icinga2::object::host { $::fqdn:
  display_name     => $::fqdn,
  ipv4_address     => $::ipaddress_eth0,
}

node 'icinga2' {
    Icinga2::Object::Host <<| |>> { }
}

@@icinga2::object::host cria um objeto de definição de host. Isso é criado por cada nó que executa esse código. O @@ o marca como um recurso exportado. Normalmente, os nós não compartilham informações no fantoche. Os recursos exportados permitem fazer isso.

Observe que todos os valores de propriedade na definição do host são fatos. Isso significa que eles serão diferentes para cada nó que o executa.

Finalmente, o recurso exportado é importado pelo nó icinga2. O módulo Icinga2 é responsável por garantir que os arquivos de configuração corretos sejam criados e recarregados.

É para você? #

Se você faz implantações, configura seus aplicativos em vários servidores e precisa fazer login em seus servidores e fazer algumas alterações na infraestrutura, aplicativos, pré-requisitos, etc., o puppet pode definitivamente ajudá-lo.

Exceto tudo isso, se você lida com uma grande infraestrutura e deseja um gerenciamento centralizado, também pode dar uma olhada.

Antes de iniciar #

Antes de decidir trabalhar no fantoche, há algumas coisas que você precisa saber.

  1. trabalho de marionete tanto na arquitetura cliente-servidor (amplamente usada) quanto single machine (specially for testing purpose)

  2. Puppet master só pode ser configurado em uma máquina linux (master machine/node), windows can be used only as client (managed machine/node)

  3. se estiver configurando o master, você deve estar ciente do uso da máquina linux e dos comandos básicos

  4. Puppet fornece sua própria linguagem de configuração que se parece com json

Instalação #

Requisitos de sistema #

No entanto, o serviço Puppet master consome bastante recursos e deve ser instalado em um servidor dedicado robusto.

  • No mínimo, seu servidor Puppet master deve ter dois núcleos de processador e pelo menos 1 GB de RAM.
  • Para atender confortavelmente a pelo menos 1.000 nós, ele deve ter de 2 a 4 núcleos de processador e pelo menos 4 GB de RAM.

Verifique sua configuração de rede: #

Em uma implantação de agente/mestre, você deve preparar sua rede para o tráfego do Puppet.

  • Firewalls: o servidor Puppet master deve permitir conexões de entrada na porta 8140, e os nós do agente devem poder se conectar ao master nessa porta.
  • Resolução de nome: cada nó deve ter um nome de host exclusivo. O DNS direto e reverso devem ser configurados corretamente.

Observação: O nome de host padrão do Puppet master é fantoche. Seus nós de agente podem estar prontos mais cedo se este nome de host for resolvido para seu Puppet mestre.

A hora deve ser definida com precisão no servidor Puppet master que atuar como autoridade de certificação. Você provavelmente deve usar NTP.

Instalando o servidor Puppet #

O Puppet fornece pacotes oficiais que instalam o Puppet Server 2.4 e todos os seus pré-requisitos nas seguintes plataformas.

Red Hat Enterprise Linux

  • Enterprise Linux 6
  • Linux empresarial 7

Debian

  • Debian 7 (Wheezy)
  • Debian 8 (Jessie)

Ubuntu

  • Ubuntu 12.04 (Preciso)
  • Ubuntu 14.04 (Confiável)
  • Ubuntu 15.10 (Wily) -Ubuntu 16.04 (Xenial)

Habilitar os repositórios de pacotes Puppet #

Linux empresarial 7

sudo rpm -Uvh https://yum.puppetlabs.com/puppetlabs-release-pc1-el-7.noarch.rpm

Para outras versões veja aqui

Instalando o mestre de marionetes

yum install puppetserver

ou

apt-get install puppetserver

O Puppet Server está configurado para usar 2 GB de RAM por padrão. Para alterar veja aqui

Inicie o serviço Puppet Server:

systemctl start puppetserver

ou

service puppetserver start
