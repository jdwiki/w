Qualquer #

Diferentes tipos de dados para erro/sucesso

def getPersonFromWebService(url: String): Either[String, Person] = {
    
    val response = webServiceClient.get(url)

    response.webService.status match {
        case 200 => {
            val person = parsePerson(response)
            if(!isValid(person)) Left("Validation failed")
            else Right(person)
        }

        case _ => Left(s"Request failed with error code $response.status")
    }
}

Correspondência de padrões em qualquer valor

getPersonFromWebService("http://some-webservice.com/person") match {
    case Left(errorMessage) => println(errorMessage)
    case Right(person) => println(person.surname)
}

Converter qualquer valor em opção

val maybePerson: Option[Person] = getPersonFromWebService("http://some-webservice.com/person").right.toOption

Tentar #

Usando Try com map, getOrElse e ​​flatMap:

import scala.util.Try

val i = Try("123".toInt)     // Success(123)
i.map(_ + 1).getOrElse(321)  // 124

val j = Try("abc".toInt)     // Failure(java.lang.NumberFormatException)
j.map(_ + 1).getOrElse(321)  // 321

Try("123".toInt) flatMap { i =>
  Try("234".toInt)
    .map(_ + i)
}                            // Success(357)

Usando Try com correspondência de padrões:

Try(parsePerson("John Doe")) match {
    case Success(person) => println(person.surname)
    case Failure(ex) => // Handle error ...
}

Opção #

O uso de valores null é fortemente desencorajado, a menos que interaja com código Java legado que espera null. Em vez disso, Option deve ser usado quando o resultado de uma função pode ser algo (Some) ou nada (None).

Um bloco try-catch é mais apropriado para tratamento de erros, mas se a função pode legitimamente não retornar nada, Option é apropriado para uso e simples.

Uma Option[T] pode ser Some(value) (contém um valor do tipo T) ou None:

def findPerson(name: String): Option[Person]

Se nenhuma pessoa for encontrada, None pode ser retornado. Caso contrário, um objeto do tipo Some contendo um objeto Person é retornado. O que segue são maneiras de lidar com um objeto do tipo Option.

Correspondência de padrões #

findPerson(personName) match {
    case Some(person) => println(person.surname)
    case None => println(s"No person found with name $personName")
}

Usando map e getOrElse #

val name = findPerson(personName).map(_.firstName).getOrElse("Unknown")
println(name) // Prints either the name of the found person or "Unknown"

Usando fold #

val name = findPerson(personName).fold("Unknown")(_.firstName)
// equivalent to the map getOrElse example above.

Convertendo para Java #

Se você precisar converter um tipo Option em um tipo Java com capacidade nula para interoperabilidade:

val s: Option[String] = Option("hello")
s.orNull           // "hello": String
s.getOrElse(null)  // "hello": String

val n: Option[Int] = Option(42)
n.orNull           // compilation failure (Cannot prove that Null <:< Int.)
n.getOrElse(null)  // 42

Lidando com Erros Originados em Futuros #

Quando uma exceção é lançada de dentro de um Futuro, você pode (deve) usar recover para lidar com isso.

Por exemplo,

def runFuture: Future = Future { throw new FairlyStupidException }

val itWillBeAwesome: Future = runFuture

…irá lançar uma Exception de dentro do Future. Mas visto que podemos prever que uma Exception do tipo FairlyStupidException com uma alta probabilidade, podemos lidar especificamente com este caso de uma maneira elegante:

val itWillBeAwesomeOrIllRecover = runFuture recover { 
    case stupid: FairlyStupidException => 
         BadRequest("Another stupid exception!") 
}

Como você pode ver, o método dado para recover é um PartialFunction sobre o domínio de todos os Throwable, então você pode lidar com apenas alguns tipos e depois deixar o resto ir para o éter do tratamento de exceções em níveis mais altos em a pilha Futuro.

Observe que isso é semelhante a executar o código a seguir em um contexto não Future:

def runNotFuture: Unit = throw new FairlyStupidException

try {
    runNotFuture
} catch {
    case e: FairlyStupidException => BadRequest("Another stupid exception!")
}

É muito importante lidar com exceções geradas dentro do Future porque na maioria das vezes elas são mais traiçoeiras. Eles geralmente não aparecem na sua cara, porque são executados em um contexto de execução e thread diferente e, portanto, não solicitam que você os corrija quando eles acontecem, especialmente se você não notar nada nos logs ou no comportamento do inscrição.

Usando cláusulas try-catch #

Além de construções funcionais como Try, Option e Either para tratamento de erros, Scala também suporta uma sintaxe semelhante à de Java, usando uma cláusula try-catch (com um possível bloco finally também). A cláusula catch é uma correspondência de padrão:

try { 
  // ... might throw exception
} catch {
  case ioe: IOException => ... // more specific cases first
  case e: Exception => ...
  // uncaught types will be thrown
} finally {
  // ...
}

Converter Exceções em Tipos de Opção ou #

Para converter exceções em tipos Either ou Option, você pode usar métodos fornecidos em scala.util.control.Exception

import scala.util.control.Exception._

val plain = "71a"
val optionInt: Option[Int] = catching(classOf[java.lang.NumberFormatException]) opt { plain.toInt }
val eitherInt = Either[Throwable, Int] = catching(classOf[java.lang.NumberFormatException]) either { plain.toInt }
