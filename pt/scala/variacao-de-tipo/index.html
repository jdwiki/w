Covariância #

O símbolo + marca um parâmetro de tipo como covariante - aqui dizemos que “Producer é covariante em A”:

trait Producer[+A] {
  def produce: A
}

Um parâmetro de tipo covariante pode ser considerado um tipo de “saída”. Marcar A como covariante afirma que Producer[X] <: Producer[Y] desde que X <: Y. Por exemplo, um Produtor[Gato] é um Produtor[Animal] válido, pois todos os gatos produzidos também são animais válidos.

Um parâmetro de tipo covariante não pode aparecer na posição contravariante (entrada). O exemplo a seguir não será compilado, pois estamos afirmando que Co[Cat] <: Co[Animal], mas Co[Cat] tem def handle(a: Cat): Unit que não pode lidar com nenhum Animal conforme exigido por Co[Animal]!

trait Co[+A] {
  def produce: A
  def handle(a: A): Unit
}

Uma abordagem para lidar com essa restrição é usar parâmetros de tipo limitados pelo parâmetro de tipo covariante. No exemplo a seguir, sabemos que B é um supertipo de A. Portanto, dado Option[X] <: Option[Y] para X <: Y, sabemos que Option[X] é def getOrElse[B >: X](b: => B): B pode aceitar qualquer supertipo de X - que inclui os supertipos de Y conforme exigido por Option[Y]:

trait Option[+A] {
  def getOrElse[B >: A](b: => B): B
}

Contravariância #

O símbolo - marca um parâmetro de tipo como contravariante - aqui dizemos que “Handler é contravariante em A”:

trait Handler[-A] {
  def handle(a: A): Unit
}

Um parâmetro de tipo contravariante pode ser considerado um tipo de “entrada”. Marcar A como contravariante afirma que Handler[X] <: Handler[Y] desde que X >: Y. Por exemplo, um Handler[Animal] é um Handler[Cat] válido, pois um Handler[Animal] também deve lidar com gatos.

Um parâmetro de tipo contravariante não pode aparecer na posição covariante (saída). O exemplo a seguir não será compilado, pois estamos afirmando que um Contra[Animal] <: Contra[Cat], no entanto, um Contra[Animal] tem def Produce: Animal que não é garantido para produzir gatos conforme exigido por Contra[Gato]!

trait Contra[-A] {
   def handle(a: A): Unit
   def produce: A
}

Cuidado, no entanto: para fins de resolução de sobrecarga, a contravariância também inverte contra-intuitivamente a especificidade de um tipo no parâmetro de tipo contravariante - Handler[Animal] é considerado “mais específico” do que Handler[Cat].

Como não é possível sobrecarregar métodos em parâmetros de tipo, esse comportamento geralmente só se torna problemático ao resolver argumentos implícitos. No exemplo a seguir, ofCat nunca será usado, pois o tipo de retorno de ofAnimal é mais específico:

implicit def ofAnimal: Handler[Animal] = ???
implicit def ofCat: Handler[Cat] = ???

implicitly[Handler[Cat]].handle(new Cat)

Este comportamento está atualmente programado para change in dotty, e é por isso que (como um exemplo) scala.math.Ordering é invariável em seu parâmetro de tipo T. Uma solução é tornar sua typeclass invariante e parametrizar a definição implícita no caso de você querer que ela seja aplicada a subclasses de um determinado tipo:

trait Person
object Person {
  implicit def ordering[A <: Person]: Ordering[A] = ???
}

Invariância #

Por padrão, todos os parâmetros de tipo são invariáveis ​​- dado o trait A[B], dizemos que “A é invariável em B”. Isso significa que, dadas duas parametrizações A[Cat] e A[Animal], não afirmamos nenhum relacionamento de sub/superclasse entre esses dois tipos - não sustenta que A[Cat] <: A[Animal] nem que A[Cat] >: A[Animal] independentemente da relação entre Cat e Animal.

As anotações de variação nos fornecem um meio de declarar tal relacionamento e impõe regras sobre o uso de parâmetros de tipo para que o relacionamento permaneça válido.

Covariância em um traço invariável #

Há também uma maneira de fazer com que um único método aceite um argumento covariante, em vez de ter toda a covariante do traço. Isso pode ser necessário porque você gostaria de usar T em uma posição contravariante, mas ainda tem covariante.

trait LocalVariance[T]{
  /// ??? throws a NotImplementedError
  def produce: T = ???
  // the implicit evidence provided by the compiler confirms that S is a
  // subtype of T.
  def handle[S](s: S)(implicit evidence: S <:< T) = {
    // and we can use the evidence to convert s into t.
    val t: T = evidence(s)
    ???
  }
}

trait A {}
trait B extends A {}

object Test {
  val lv = new LocalVariance[A] {}

  // now we can pass a B instead of an A.
  lv.handle(new B {})
}
