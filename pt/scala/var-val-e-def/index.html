Como val são semanticamente estáticos, eles são inicializados “no local” onde quer que apareçam no código. Isso pode produzir um comportamento surpreendente e indesejável quando usado em classes e características abstratas.

Por exemplo, digamos que gostaríamos de fazer uma característica chamada PlusOne que define uma operação de incremento em um Int encapsulado. Como Ints são imutáveis, o valor mais um é conhecido na inicialização e nunca será alterado depois, então semanticamente é um val. No entanto, defini-lo dessa maneira produzirá um resultado inesperado.

trait PlusOne {
    val i:Int

    val incr = i + 1
}

class IntWrapper(val i: Int) extends PlusOne

Não importa com que valor i você construa IntWrapper, chamar .incr no objeto retornado sempre retornará 1. Isso ocorre porque o val incr é inicializado no trait, antes da classe de extensão, e nesse momento i só tem o valor padrão de 0. (Em outras condições, pode ser preenchido com Nil, null ou um padrão semelhante.)

A regra geral, então, é evitar usar val em qualquer valor que dependa de um campo abstrato. Em vez disso, use lazy val, que não avalia até que seja necessário, ou def, que avalia toda vez que é chamado. Observe, no entanto, que se o lazy val for forçado a ser avaliado por um val antes da conclusão da inicialização, o mesmo erro ocorrerá.

Um violino (escrito em Scala-Js, mas o mesmo comportamento se aplica) pode ser encontrado aqui.

Var, Val e Def #

foi #

Um var é uma variável de referência, semelhante a variáveis ​​em linguagens como Java. Diferentes objetos podem ser atribuídos livremente a um var, desde que o objeto dado tenha o mesmo tipo que o var foi declarado com:

scala> var x = 1
x: Int = 1

scala> x = 2
x: Int = 2

scala> x = "foo bar"
<console>:12: error: type mismatch;
 found   : String("foo bar")
 required: Int
       x = "foo bar"
       ^

Observe no exemplo acima que o tipo de var foi inferido pelo compilador dado a primeira atribuição de valor.

valor #

Um val é uma referência constante. Assim, um novo objeto não pode ser atribuído a um val que já foi atribuído.

scala> val y = 1
y: Int = 1

scala> y = 2
<console>:12: error: reassignment to val
       y = 2
         ^

No entanto, o objeto para o qual um val aponta é não constante. Esse objeto pode ser modificado:

scala> val arr = new Array[Int](2)
arr: Array[Int] = Array(0, 0)

scala> arr(0) = 1

scala> arr
res1: Array[Int] = Array(1, 0)

def #

Um def define um método. Um método não pode ser reatribuído.

scala> def z = 1
z: Int

scala> z = 2
<console>:12: error: value z_= is not a member of object $iw
       z = 2
       ^

Nos exemplos acima, val y e def z retornam o mesmo valor. No entanto, um def é avaliado quando é chamado, enquanto um val ou var é avaliado quando é atribuído. Isso pode resultar em comportamento diferente quando a definição tem efeitos colaterais:

scala> val a = {println("Hi"); 1}
Hi
a: Int = 1

scala> def b = {println("Hi"); 1}
b: Int

scala> a + 1
res2: Int = 2

scala> b + 1
Hi
res3: Int = 2

Funções #

Como as funções são valores, elas podem ser atribuídas a val/var/defs. Todo o resto funciona da mesma maneira que acima:

scala> val x = (x: Int) => s"value=$x"
x: Int => String = <function1>

scala> var y = (x: Int) => s"value=$x"
y: Int => String = <function1>

scala> def z = (x: Int) => s"value=$x"
z: Int => String

scala> x(1)
res0: String = value=1

scala> y(2)
res1: String = value=2

scala> z(3)
res2: String = value=3

Valor preguiçoso #

lazy val é um recurso de linguagem onde a inicialização de um val é atrasada até que seja acessado pela primeira vez. Após esse ponto, ele age como um val regular.

Para usá-lo, adicione a palavra-chave lazy antes de val. Por exemplo, usando o REPL:

scala> lazy val foo = {
     |   println("Initializing")
     |   "my foo value"
     | }
foo: String = <lazy>

scala> val bar = {
     |   println("Initializing bar")
     |   "my bar value"
     | }
Initializing bar
bar: String = my bar value

scala> foo
Initializing
res3: String = my foo value

scala> bar
res4: String = my bar value

scala> foo
res5: String = my foo value

Este exemplo demonstra a ordem de execução. Quando o lazy val é declarado, tudo o que é salvo no valor foo é uma chamada de função lenta que ainda não foi avaliada. Quando o val regular é definido, vemos a chamada println ser executada e o valor é atribuído a bar. Quando avaliamos foo na primeira vez, vemos println ser executado - mas não quando é avaliado pela segunda vez. Da mesma forma, quando bar é avaliado, não vemos println sendo executado - apenas quando é declarado.

Quando usar ‘preguiçoso’ #

  1. A inicialização é computacionalmente cara e o uso de val é raro.

    lazy val tiresomeValue = {(1 to 1000000).filter(x => x % 113 == 0).sum}
if (scala.util.Random.nextInt > 1000) {
  println(tiresomeValue)
}

tiresomeValue leva muito tempo para calcular e nem sempre é usado. Torná-lo um vale preguiçoso economiza computação desnecessária.

  1. Resolvendo dependências cíclicas

Vejamos um exemplo com dois objetos que precisam ser declarados ao mesmo tempo durante a instanciação:

   object comicBook {
     def main(args:Array[String]): Unit = {
       gotham.hero.talk()
       gotham.villain.talk()
     }
   }

   class Superhero(val name: String) {
     lazy val toLockUp = gotham.villain
     def talk(): Unit = {
       println(s"I won't let you win ${toLockUp.name}!")
     }
   }

   class Supervillain(val name: String) {
     lazy val toKill = gotham.hero
     def talk(): Unit = {
       println(s"Let me loosen up Gotham a little bit ${toKill.name}!")
     }
   }

   object gotham {
     val hero: Superhero = new Superhero("Batman")
     val villain: Supervillain = new Supervillain("Joker")
   }

Sem a palavra-chave lazy, os respectivos objetos não podem ser membros de um objeto. A execução de tal programa resultaria em um java.lang.NullPointerException. Usando lazy, a referência pode ser atribuída antes de ser inicializada, sem medo de ter um valor não inicializado.

Def de Sobrecarga #

Você pode sobrecarregar um def se a assinatura for diferente:

def printValue(x: Int) {
  println(s"My value is an integer equal to $x")
}

def printValue(x: String) {
  println(s"My value is a string equal to '$x'")
}

printValue(1)  // prints "My value is an integer equal to 1"
printValue("1") // prints "My value is a string equal to '1'"

Isso funciona da mesma forma dentro de classes, características, objetos ou não.

Parâmetros nomeados #

Ao invocar um def, os parâmetros podem ser atribuídos explicitamente pelo nome. Isso significa que eles não precisam ser ordenados corretamente. Por exemplo, defina printUs() como:

// print out the three arguments in order.
def printUs(one: String, two: String, three: String) = 
   println(s"$one, $two, $three")

Agora pode ser chamado destas formas (entre outras):

printUs("one", "two", "three") 
printUs(one="one", two="two", three="three")
printUs("one", two="two", three="three")
printUs(three="three", one="one", two="two")

Isso resulta em um, dois, três sendo impresso em todos os casos.

Se nem todos os argumentos forem nomeados, os primeiros argumentos serão correspondidos por ordem. Nenhum argumento posicional (sem nome) pode seguir um argumento nomeado:

printUs("one", two="two", three="three") // prints 'one, two, three'
printUs(two="two", three="three", "one") // fails to compile: 'positional after named argument'
