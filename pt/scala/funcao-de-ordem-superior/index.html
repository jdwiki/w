Função de ordem superior

  • Argumentos avaliação preguiçosa

  • Usando métodos como valores de função
  • Funções de alta ordem (função como parâmetro)
  • Argumentos avaliação preguiçosa

Scala não mede esforços para tratar métodos e funções como sintaticamente idênticos. Mas sob o capô, são conceitos distintos.

Um método é um código executável e não tem representação de valor.

Uma função é uma instância de objeto real do tipo Function1 (ou um tipo similar de outra aridade). Seu código está contido em seu método apply. Efetivamente, ele simplesmente atua como um valor que pode ser repassado.

Aliás, a capacidade de tratar funções como valores é exatamente o que se entende por uma linguagem com suporte para funções de ordem superior. As instâncias de função são a abordagem do Scala para implementar esse recurso.

Uma função real de ordem superior é uma função que recebe um valor de função como argumento ou retorna um valor de função. Mas em Scala, como todas as operações são métodos, é mais geral pensar em métodos que recebem ou retornam parâmetros de funções. Então map, como definido em Seq, pode ser pensado como uma “função de ordem superior” devido ao seu parâmetro ser uma função, mas não é literalmente uma função; é um método.

Usando métodos como valores de função #

O compilador Scala converterá automaticamente métodos em valores de função com o objetivo de passá-los para funções de ordem superior.

object MyObject {
  def mapMethod(input: Int): String = {
    int.toString
  }
}

Seq(1, 2, 3).map(MyObject.mapMethod) // Seq("1", "2", "3")

No exemplo acima, MyObject.mapMethod não é uma chamada de função, mas é passado para map como um valor. De fato, map requer um valor de função passado para ele, como pode ser visto em sua assinatura. A assinatura para o map de uma List[A] (uma lista de objetos do tipo A) é:

def map[B](f: (A) ⇒ B): List[B]

A parte f: (A) => B indica que o parâmetro para esta chamada de método é alguma função que recebe um objeto do tipo A e retorna um objeto do tipo B. A e B são definidos arbitrariamente. Voltando ao primeiro exemplo, podemos ver que mapMethod recebe um Int (que corresponde a A) e retorna uma String (que corresponde a B). Assim mapMethod é um valor de função válido para passar para map. Poderíamos reescrever o mesmo código assim:

Seq(1, 2, 3).map(x:Int => int.toString)

Isso inline o valor da função, o que pode adicionar clareza para funções simples.

Funções de alta ordem (função como parâmetro) #

Uma função de ordem superior, em oposição a uma função de primeira ordem, pode ter uma das três formas:

  • Um ou mais de seus parâmetros é uma função e ela retorna algum valor.

  • Retorna uma função, mas nenhum de seus parâmetros é uma função.

  • Ambos os itens acima: Um ou mais de seus parâmetros é uma função e retorna uma função.

      object HOF {
      def main(args: Array[String]) {
      val list = List(("Srini","E"),("Subash","R"),("Ranjith","RK"),("Vicky","s"),("Sudhar","s"))
      //HOF
       val fullNameList= list.map(n => getFullName(n._1, n._2))

       }

      def getFullName(firstName: String, lastName: String): String = firstName + "." + lastName
      }

Aqui a função map recebe uma função getFullName(n._1,n._2) como parâmetro. Isso é chamado de HOF (função de ordem superior).

Argumentos avaliação preguiçosa #

Scala suporta avaliação lenta para argumentos de função usando a notação: def func(arg: => String). Tal argumento de função pode receber um objeto String regular ou uma função de ordem superior com o tipo de retorno String. No segundo caso, o argumento da função seria avaliado no acesso ao valor.

Por favor, veja o exemplo:

def calculateData: String = {
  print("Calculating expensive data! ")
  "some expensive data"
}

def dumbMediator(preconditions: Boolean, data: String): Option[String] = {
  print("Applying mediator")
  preconditions match {
    case true => Some(data)
    case false => None
  }
}

def smartMediator(preconditions: Boolean, data: => String): Option[String] = {
  print("Applying mediator")
  preconditions match {
    case true => Some(data)
    case false => None
  }
}

smartMediator(preconditions = false, calculateData)

dumbMediator(preconditions = false, calculateData)

A chamada smartMediator retornaria o valor None e imprimiria a mensagem "Aplicando mediador".

A chamada dumbMediator retornaria o valor None e imprimiria a mensagem "Calculando dados caros! Aplicando mediador".

A avaliação preguiçosa pode ser extremamente útil quando você deseja otimizar uma sobrecarga de cálculo de argumentos caros.
