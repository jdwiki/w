Os nomes de valores e variáveis ​​devem estar em minúsculas camel case #

Os nomes das constantes devem estar em maiúsculas. Ou seja, se o membro é final, imutável e pertence a um objeto de pacote ou a um objeto, pode ser considerado uma constante

Nomes de métodos, valores e variáveis ​​devem estar em minúsculas camel case

Fonte: http://docs.scala-lang.org/style/naming-conventions.html

Esta compilação:

val (a,b) = (1,2)
// a: Int = 1
// b: Int = 2

mas isso não:

val (A,B) = (1,2)
// error: not found: value A
// error: not found: value B

Não é apenas val vs. var #

val e var #

scala> val a = 123
a: Int = 123

scala> a = 456
<console>:8: error: reassignment to val
       a = 456

scala> var b = 123
b: Int = 123

scala> b = 321
b: Int = 321
  • Referências val são imutáveis: como uma variável final em Java, uma vez inicializada, você não pode alterá-la
  • Referências var são reatribuíveis como uma simples declaração de variável em Java

Coleções imutáveis ​​e mutáveis #

  val mut = scala.collection.mutable.Map.empty[String, Int]
  mut += ("123" -> 123)
  mut += ("456" -> 456)
  mut += ("789" -> 789)

  val imm = scala.collection.immutable.Map.empty[String, Int]
  imm + ("123" -> 123)
  imm + ("456" -> 456)
  imm + ("789" -> 789)

  scala> mut
    Map(123 -> 123, 456 -> 456, 789 -> 789)

  scala> imm
    Map()

scala> imm + ("123" -> 123) + ("456" -> 456) + ("789" -> 789)
    Map(123 -> 123, 456 -> 456, 789 -> 789)

A biblioteca padrão Scala oferece estruturas de dados imutáveis ​​e mutáveis, não a referência a ela. Cada vez que uma estrutura de dados imutável é “modificada”, uma nova instância é produzida em vez de modificar a coleção original no local. Cada instância da coleção pode compartilhar uma estrutura significativa com outra instância.

Coleção Mutável e Imutável (Documentação Oficial Scala)

Mas não posso usar imutabilidade neste caso! #

Vamos pegar como exemplo uma função que recebe 2 Map e retorna um Map contendo todos os elementos em ma e mb:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int]

Uma primeira tentativa pode ser iterar pelos elementos de um dos mapas usando for ((k, v) <-map) e de alguma forma retornar o mapa mesclado.

def merge2Maps(ma: ..., mb: ...): Map[String, Int] = {

  for ((k, v) <- mb) {
    ???
  }

}

Este primeiro movimento adiciona imediatamente uma restrição: uma mutação fora desse for agora é necessária. Isso fica mais claro ao remover o açúcar do for:

// this:
for ((k, v) <- map) { ??? }

// is equivalent to:
map.foreach { case (k, v) => ??? }

“Por que temos que sofrer mutação?” #

foreach depende de efeitos colaterais. Toda vez que queremos que algo aconteça dentro de um foreach, precisamos “efeito colateral em algo”, neste caso, podemos alterar uma variável var result ou podemos usar uma estrutura de dados mutável.

Criando e preenchendo o mapa result #

Vamos supor que ma e mb sejam scala.collection.immutable.Map, poderíamos criar o mapa result a partir de ma:

val result = mutable.Map() ++ ma

Em seguida, itere através de mb adicionando seus elementos e se a chave do elemento atual em ma já existir, vamos substituí-la pela mb.

mb.foreach { case (k, v) => result += (k -> v) }

Implementação mutável #

Até aí tudo bem, “tivemos que usar coleções mutáveis” e uma implementação correta poderia ser:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  val result = scala.collection.mutable.Map() ++ ma
  mb.foreach { case (k, v) => result += (k -> v) }
  result.toMap // to get back an immutable Map
}

Como esperado:

scala> merge2Maps(Map("a" -> 11, "b" -> 12), Map("b" -> 22, "c" -> 23))
  Map(a -> 11, b -> 22, c -> 23)

Dobrando para o resgate #

Como podemos nos livrar de foreach neste cenário? Se tudo o que fazemos é basicamente iterar sobre os elementos da coleção e aplicar uma função enquanto acumulamos o resultado na opção, poderíamos usar .foldLeft:

def merge2Maps(ma: Map[String, Int], mb: Map[String, Int]): Map[String, Int] = {
  mb.foldLeft(ma) { case (result, (k, v)) => result + (k -> v) }
  // or more concisely mb.foldLeft(ma) { _ + _ }
}

Neste caso nosso “resultado” é o valor acumulado a partir de ma, o zero do .foldLeft.

Resultado intermediário #

Obviamente, esta solução imutável está produzindo e destruindo muitas instâncias do Map durante o fold, mas vale a pena mencionar que essas instâncias não são um clone completo do Map acumulado, mas sim estão compartilhando uma estrutura (dados) significativa com a instância existente.

Razoabilidade mais fácil #

É mais fácil raciocinar sobre a semântica se for mais declarativa como a abordagem .foldLeft. O uso de estruturas de dados imutáveis ​​pode ajudar a tornar nossa implementação mais fácil de raciocinar.
