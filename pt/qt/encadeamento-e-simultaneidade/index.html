Algumas notas que já são mencionadas nos documentos oficiais aqui e [aqui](http://doc.qt. io/qt-5/qobject.html#moveToThread):

  • Se um objeto tem um pai, ele deve estar no mesmo thread que o pai, ou seja, ele não pode ser movido para um novo encadeamento, nem você pode definir um pai para um objeto se o pai e o objeto estiverem em encadeamentos diferentes
  • Quando um objeto é movido para um novo thread, todos os seus filhos também são movidos para o novo thread
  • Você só pode enviar objetos para um novo segmento. Você não pode puxá-los para um novo thread, ou seja, você só pode chamar moveToThread do thread em que o objeto está atualmente vivendo

Uso básico do QThread #

QThread é um handle para um thread de plataforma. Ele permite que você gerencie o encadeamento monitorando seu tempo de vida e solicitando que ele termine seu trabalho.

Na maioria dos casos, a herança da classe não é recomendada. O método padrão run inicia um loop de eventos que pode despachar eventos para objetos que vivem na classe. Conexões de slot de sinal entre threads são implementadas despachando um QMetaCallEvent para o objeto de destino.

Uma instância QObject pode ser movida para uma thread, onde processará seus eventos, como eventos de timer ou chamadas de slot/método.

Para trabalhar em um thread, primeiro crie sua própria classe de trabalho que deriva de QObject. Em seguida, mova-o para o fio. O objeto pode executar seu próprio código automaticamente, por exemplo usando QMetaObject::invokeMethod().

#include <QObject>

class MyWorker : public QObject
{
    Q_OBJECT
public:
    Q_SLOT void doWork() {
        qDebug() << "doWork()" << QThread::currentThread();
        // and do some long operation here
    }
    MyWorker(QObject * parent = nullptr) : QObject{parent} {}
};

class MyController : public QObject
{
    Q_OBJECT
    Worker worker;
    QThread workerThread;
public:
    MyController()  {
        worker.moveToThread(&workerThread);
        // provide meaningful debug output
        workerThread.setObjectName("workerThread");
        workerThread.start();
        // the thread starts the event loop and blocks waiting for events
    }
    ~MyController()  {
        workerThread.quit();
        workerThread.wait();
    }
    void operate() {
        // Qt::QueuedConnection ensures that the slot is invoked in its own thread
        QMetaObject::invokeMethod(&worker, "doWork", Qt::QueuedConnection);
    }
};

Se seu trabalhador deve ser efêmero e existir apenas enquanto seu trabalho está sendo feito, é melhor enviar um functor ou um método thread-safe para execução no pool de threads via QtConcurrent::run.

Execução simultânea de Qt #

Se você achar o gerenciamento de QThreads e primitivos de baixo nível, como mutexes ou semáforos, muito complexo, o namespace Qt Concurrent é o que você está procurando. Inclui classes que permitem mais gerenciamento de threads de alto nível.

Vejamos a execução simultânea. QtConcurrent::run() permite executar a função em uma nova thread. Quando você gostaria de usá-lo? Quando você tem alguma operação longa e não quer criar thread manualmente.

Agora o código:

#include <qtconcurrentrun.h>

void longOperationFunction(string parameter)
{
    // we are already in another thread
    // long stuff here
} 

void mainThreadFunction()
{
    QFuture<void> f = run(longOperationFunction, "argToPass");
    f.waitForFinished();
}

Então as coisas são simples: quando precisamos executar outra função em outra thread, basta chamar QtConcurrent::run, passar a função e seus parâmetros e pronto!

QFuture apresenta o resultado de nossa computação assíncrona. No caso de QtConcurrent::run não podemos cancelar a execução da função.

Invocando slots de outros threads #

Quando um loop de eventos Qt é usado para executar operações e um usuário não experiente em Qt precisa interagir com esse loop de eventos, escrever o slot para lidar com invocações regulares de outro thread pode simplificar as coisas para outros usuários.

main.cpp:

#include "OperationExecutioner.h"
#include <QCoreApplication>
#include <QThread>

int main(int argc, char** argv)
{
    QCoreApplication app(argc, argv);

    QThread thrd;
    thrd.setObjectName("thrd");
    thrd.start();
    while(!thrd.isRunning())
        QThread::msleep(10);

    OperationExecutioner* oe = new OperationExecutioner;
    oe->moveToThread(&thrd);
    oe->doIt1(123,'A');
    oe->deleteLater();
    thrd.quit();
    while(!thrd.isFinished())
        QThread::msleep(10);

    return 0;
}

OperationExecutioner.h:

#ifndef OPERATION_EXECUTIONER_H
#define OPERATION_EXECUTIONER_H

#include <QObject>

class OperationExecutioner : public QObject
{
    Q_OBJECT
public slots:
    void doIt1(int argi, char argc);
};

#endif  // OPERATION_EXECUTIONER_H

OperationExecutioner.cpp:

#include "OperationExecutioner.h"
#include <QMetaObject>
#include <QThread>
#include <QDebug>

void OperationExecutioner::doIt1(int argi, char argc)
{
    if (QThread::currentThread() != thread()) {
        qInfo() << "Called from thread" << QThread::currentThread();
        QMetaObject::invokeMethod(this, "doIt1", Qt::QueuedConnection,
                                  Q_ARG(int,argi), Q_ARG(char,argc));
        return;
    }

    qInfo() << "Called from thread" << QThread::currentThread()
            << "with args" << argi << argc;
}

OperationExecutioner.pro:

HEADERS += OperationExecutioner.h
SOURCES += main.cpp OperationExecutioner.cpp
QT -= gui
