Vinculando bibliotecas Swift

  • Vinculando uma biblioteca Swift no Xamarin.iOS

Nesta página

  • Vinculando uma biblioteca Swift no Xamarin.iOS

Um guia fácil de seguir que o guiará pelo processo de vinculação de arquivos Swift .framework para uso em um projeto Xamarin.

  1. Ao construir uma biblioteca no Xcode tem a opção de incluir as bibliotecas swift. Não! Eles serão incluídos em seu aplicativo final como NAME.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib, mas devem ser incluídos como NAME.app/Frameworks/libswift*.dylib

  2. Você pode encontrar essa informação em outro lugar, mas vale a pena mencionar: Não inclua Bitcode na biblioteca. A partir de agora, o Xamarin não inclui Bitcode para iOS e a Apple exige que todas as bibliotecas suportem as mesmas arquiteturas.

Vinculando uma biblioteca Swift no Xamarin.iOS #

A vinculação de uma biblioteca Swift no Xamarin.iOS segue o mesmo processo para Objective-C, conforme mostrado em https://developer.xamarin.com/guides/ios/advanced_topics/binding_objective-c/, mas com algumas ressalvas.

  1. Uma classe swift deve herdar de NSObject para ser vinculada.
  2. O compilador Swift traduzirá nomes de classes e protocolos em outra coisa, a menos que você use a anotação @objc (por exemplo, @objc(MyClass)) em suas classes swift para especificar o nome c objetivo explícito.
  3. Em tempo de execução, seu APP deve incluir algumas bibliotecas de núcleo Swift junto com seu framework vinculado em uma pasta chamada Frameworks;
  4. Quando o aplicativo é enviado para a AppStore, ele deve incluir uma pasta SwiftSupport ao lado de sua pasta Payload. Esses estão dentro do arquivo IPA.

Aqui você pode encontrar uma ligação de amostra simples: https://github.com/Flash3001/Xamarin.BindingSwiftLibrarySample

E uma amostra de ligação completa: https://github.com/Flash3001/iOSCharts.Xamarin

Por favor, encontre as etapas abaixo:

1.1 Prepare as classes Swift que deseja exportar #

Para qualquer classe Swift que você deseja usar, você precisa herdar de NSObject e tornar o nome Objective-C explícito usando a anotação objc. Caso contrário, o compilador Swift gerará nomes diferentes. Abaixo está um código de exemplo de como uma classe Swift pode se parecer. Observe que não importa qual classe ela herda, desde que a classe raiz herde de NSObject.

//Add this to specify explicit objective c name
@objc(MyClass)
open class MyClass: NSObject {
    open func getValue() -> String
    {
        return "Value came from MyClass.swift!";
    }
}

1.2 Construir a estrutura #

Desabilite o código de bits. *

Build para lançamento para Dispositivo e Simulador. *

  • Não relacionado apenas à vinculação Swift.

2. Crie uma biblioteca de gordura #

Um framework contém vários arquivos, o que precisa comer um pouco é NAME.framework/NAME (sem extensão).

  • Copie Release-iphoneos/NAME.framework para NAME.framework
  • Crie a biblioteca FAT usando:
    • lipo -create Release-iphonesimulator/NAME.framework/NAME Release-iphoneos/NAME.framework/NAME -saída NAME.framework/NAME
  • Copie os arquivos em Release-iphonesimulator/NAME.framework/Modules/NAME.swiftmodule para NAME.framework/Modules/NAME.swiftmodule (até agora só continha arquivos dos iphoneos)

3. Importe a biblioteca #

Presumo que você já tenha criado o projeto Binding em File -> New -> iOS -> Binding Library.

Xamarin oferece suporte à importação de .frameworks. Basta clicar com o botão direito do mouse em ‘Referências nativas’ e clicar em ‘Adicionar referência nativa’. Encontre a estrutura de gordura recém-criada e adicione-a.

4. Crie a ApiDefinition baseada no arquivo LIBRARY-Swift.h dentro dos cabeçalhos. #

Você pode fazer isso manualmente, mas não será legal. Você pode usar a Sharpie Objetiva. A ferramenta que o Xamarin usa para vincular suas próprias bibliotecas.

Como usá-lo em https://developer.xamarin.com/guides/cross-platform/macios/binding/objective-sharpie/

O comando básico será algo como: sharpie bind -sdk iphoneos9.3 NAME-Swift.h

Se você obtiver uma System.Reflection.TargetInvocationException provavelmente é porque você tem uma versão diferente do SDK instalada. Execute o seguinte comando para verificar com o iPhone OS SDK que você instalou:

sharpie xcode -sdks

O arquivo NAME-Swift.h está localizado em NAME.framework/Headers/NAME-Swift.h

Nota: As classes swift devem herdar de “NSObject”, caso contrário NAME-Swift.h não importará suas classes e a Sharpie Objetiva não converterá nada.

Substitua o conteúdo do seu projeto de ligação ApiDefinition.cs pelo recém-criado.

5. Altere todos os [Protocol] e [BaseType] para incluir o nome da classe no tempo de execução do Objective-C. #

Caso a classe ou protocolo original do Swift não inclua a anotação @objc(MyClass) conforme especificado na etapa 1.1, eles terão seus nomes internos de Objective-C alterados, então você precisa mapeá-lo para o correto.

Todos os nomes estão disponíveis no arquivo NAME-Swift.h no seguinte formato:

SWIFT_CLASS("_TtC11SwiftSample7MyClass")
@interface MyClass : NSObject

E

SWIFT_PROTOCOL("_TtP6Charts17ChartDataProvider_")
@protocol ChartDataProvider

Para definir o nome, você usa BaseTypeAttribute.Name https://developer.xamarin.com/guides/cross-platform/macios/binding/binding-types-reference/#BaseType.Name propriedade para classes e ProcotolAttribute.Name https:// developer.xamarin.com/api/property/MonoTouch.Foundation.ProtocolAttribute.Name/ para protocolos.

[BaseType(typeof(NSObject), Name = "_TtC11SwiftSample7MyClass")]
interface MyClass

Fazer manualmente não é legal. Você pode usar esta ferramenta https://github.com/Flash3001/SwiftClassify para inserir todos os nomes. (É um trabalho em andamento. Mas é bem simples, apenas olhando o código você verá como ele funciona).

6.1 Inclua todas as dependências do Swift para serem executadas. #

Se você tentar consumir a biblioteca em um aplicativo e tentar executá-lo agora, ele falhará. O erro é devido à falta de libswiftCore.dylib

Algo assim:

Dyld Error Message:
  Library not loaded: @rpath/libswiftCore.dylib
  Referenced from: /Users/USER/Library/Developer/CoreSimulator/Devices/AC440891-C819-4050-8CAB-CE15AB4B3830/data/Containers/Bundle/Application/27D2EC87-5042-4FA7-9B80-A24A8971FB48/SampleUsing.app/Frameworks/SwiftSample.framework/SwiftSample
  Reason: image not found

O Xamarin.iOS não oferece suporte oficial para vincular uma biblioteca Swift. Portanto, você deve incluir manualmente as bibliotecas swift core nas pastas Frameworks e SwiftSupport. Os arquivos da pasta Frameworks são diferentes para Simulador e Dispositivo. Eles podem ser encontrados em /Applications/Xcode.app/Contents/Developer//XcodeDefault.xctoolchain/usr/lib/swift.Toolchains

Em vez de copiar manualmente os arquivos dentro da pasta Framework, você pode usar esta biblioteca https://github.com/Flash3001/Xamarin.Swift3.Support . Ele inclui todas as dependências que o Swift 3.1 precisa, cada uma em um único pacote NuGet.

Como você pode ver, o pacote Nuget está incluído no aplicativo do consumidor, não na vinculação em si. Se você tentar incluí-lo na ligação, receberá erros de compilação.

Se você estiver criando um pacote Nuget, poderá instruir o Nuget a incluí-lo como uma dependência.

6.2. Descobrir quais dependências do Swift incluir. #

Uma coisa importante a fazer é descobrir cada pacote que você precisa incluir em seu projeto. Uma ligação simples geralmente precisará de:

     libswiftCore.dylib
     libswiftCoreGraphics.dylib
     libswiftCoreImage.dylib
     libswiftDarwin.dylib
     libswiftDispatch.dylib
     libswiftFoundation.dylib
     libswiftObjectiveC.dylib
     libswiftQuartzCore.dylib
     libswiftUIKit.dylib

Para listar cada dependência, você pode executar o seguinte comando dentro do seu LibraryName.framework

otool -l -arch armv7 LibraryName | grep libswift

Não inclua todos os pacotes disponíveis no NuGet para Swift3, pois eles podem aumentar o tamanho do seu aplicativo.

7. Inclua o SwiftSupport para enviar o aplicativo para a AppStore. #

A Apple exige que seu aplicativo seja enviado com uma pasta SwiftSupport ao lado da pasta Payload. Ambos estão dentro do seu pacote IPA.

Você pode usar este script https://github.com/bq/ipa-packager para fazer esse trabalho para você.

Este processo é o único que o consumidor da biblioteca terá que fazer manualmente. Toda vez que ele tenta empurrar o aplicativo para a AppStore.

Clique em ‘Assinar e Distribuir’ e Salvar no Disco

Descompacte seu .IPA

Crie o novo IPA usando o script antes mencionado

Se você descompactar o arquivo sabe, ele conterá a pasta SwiftSupport.

Observações #

Ao construir uma biblioteca no Xcode, há a opção de incluir as bibliotecas swift. Não! Eles serão incluídos em seu aplicativo final como NAME.app/Frameworks/LIBRARY.framework/Frameworks/libswift*.dylib, mas devem ser incluídos como NAME.app/Frameworks/libswift*.dylib

Você pode encontrar essas informações em outro lugar, mas vale a pena mencionar: Não inclua Bitcode na biblioteca. A partir de agora, o Xamarin não inclui Bitcode para iOS e a Apple exige que todas as bibliotecas suportem as mesmas arquiteturas.
