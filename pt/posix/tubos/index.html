Pipes são um mecanismo para comunicação unidirecional entre processos ou entre threads dentro do escopo de uma única máquina. Logicamente, um pipe consiste em dois terminais conectados, um no qual os dados podem ser gravados e outro a partir do qual esses dados podem ser lidos posteriormente, com um buffer de dados entre os quais as gravações e leituras não precisam ser síncronas. Os tubos devem ser diferenciados dos pipelines de casca, que são uma aplicação de tubos.

Criação e uso básico #

Pipes anônimos, ou simplesmente pipes, são objetos gerenciados pelo kernel expostos a processos como um par de descritores de arquivo, um para o terminal de leitura e outro para o terminal de gravação. Eles são criados através da função pipe(2):

int pipefds[2];
int result;

result = pipe(pipefds);

Em caso de sucesso, pipe() grava o descritor para o final de leitura do pipe no índice 0 do array fornecido e o descritor para o final de gravação no índice 1; esses índices são análogos aos números do descritor de arquivo convencional para os fluxos padrão.

Tendo criado um pipe, usa-se as funções POSIX I/O para escrever no final de gravação ou ler no final de leitura:

Processo 1:

ssize_t bytes_written = write(pipefds[1], "Hello, World!", 14);

Processo 2:

char buffer[256];
ssize_t bytes_read = read(pipefds[0], buffer, sizeof(buffer));

Alternativamente, pode-se usar fdopen() para envolver uma das duas extremidades do pipe em uma estrutura FILE para uso com funções C stdio:

FILE *write_end = fdopen(pipefds[1]);

if (write_end) {
    fputs("Hello, World!");
}

Os pipes têm buffers de E/S finitos e as gravações comuns em pipes com buffers completos serão bloqueadas. Os pipes não são, portanto, um mecanismo seguro para um thread se comunicar consigo mesmo, como se o thread que escreve em um pipe também fosse o único que o lê, assim que uma gravação bloqueia esse thread é bloqueado.

Um pipe persiste enquanto qualquer processo tiver uma descrição de arquivo aberta para qualquer uma das extremidades do pipe. A função close(2) pode ser usada para fechar um final de pipe representado por um descritor de arquivo, e fclose(3) pode ser usado para fechar um final de pipe através de um FILE envolvido nele.

Estabelecendo um pipe para um processo filho #

Descritores de arquivo e objetos FILE são recursos por processo que não podem ser trocados entre processos via E/S comum. Portanto, para que dois processos distintos se comuniquem por meio de um pipe anônimo, um ou ambos os processos participantes devem herdar uma extremidade de pipe aberta do processo que criou o pipe, que deve, portanto, ser o processo pai ou um ancestral mais distante. O caso mais simples é aquele em que um processo pai deseja se comunicar com um processo filho.

Como o processo filho deve herdar a descrição de arquivo aberta necessária de seu pai, o pipe deve ser criado primeiro. O pai então bifurca. Normalmente, cada processo será estritamente um leitor ou estritamente um escritor; nesse caso, cada um deve fechar a ponta do tubo que não pretende utilizar.

void demo() {
    int pipefds[2];
    pid_t pid;

    // Create the pipe
    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (pid = fork()) {
        case -1:
            // error - abort ...
            break;
        case 0:   /* child */
            close(pipefds[0]);
            write(pipefds[1], "Goodbye, and thanks for all the fish!", 37);
            exit(0);
        default:  /* parent */
            close(pipefds[1]);

            char buffer[256];
            ssize_t nread = read(pipefds[0], sizeof(buffer) - 1);

            if (nread >= 0) {
                buffer[nread] = '\0';
                printf("My child said '%s'\n", buffer);
            }

            // collect the child
            wait(NULL);

            break;
    }
}

Conectando dois processos filhos por meio de um pipe #

A conexão de dois processos filho por meio de um pipe é realizada conectando cada um dos dois filhos ao pai por meio de diferentes extremidades do mesmo pipe. Normalmente, o pai não fará parte da conversa entre os filhos, então ele fecha suas cópias de ambas as extremidades do pipe.

int demo() {
    int pipefds[2];
    pid_t child1, child2;

    if (pipe(pipefds)) {
        // error - abort ...
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            write(pipefds[1], "Hello, brother!", 15);
            exit(0);
        default:  /* parent */
            // nothing
    }

    switch (child1 = fork()) {
        case -1:
            // error - abort
            break;
        case 0:   /* child 2 */
            char buffer[256];
            ssize_t nread;

            close(pipefds[1]);
            nread = read(pipefds[0], buffer, sizeof(buffer) - 1);
            if (nread < 0) {
                // handle error
            } else {
                buffer[nread] = '\0';
                printf("My brother told me '%s'\n", buffer);
            }
            exit(0);
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    if (child1 >= 0) {
        wait(NULL);
        if (child2 >= 0) {
            wait(NULL);
        }
    }
}

Criando um pipeline no estilo shell #

Um pipeline no estilo shell consiste em dois ou mais processos, cada um com sua saída padrão conectada à entrada padrão do próximo. As conexões de saída para entrada são construídas em tubos. Para estabelecer um conjunto de processos do tipo pipeline, cria-se processos filhos conectados a pipes conforme descrito em outro exemplo e, além disso, usa-se a função dup2(2) para duplicar cada extremidade de pipe no descritor de arquivo padrão apropriado de seu processo. Geralmente, é uma boa ideia fechar o descritor de arquivo final do pipe original, especialmente se, como geralmente é o caso, pretende-se que o filho execute posteriormente um comando diferente.

// Most error handling omitted for brevity, including ensuring that pipe ends are
// always closed and child processes are always collected as needed; see other
// examples for more detail.
int demo() {
    int pipefds[2];
    pid_t child1 = -1, child2 = -1;

    pipe(pipefds);

    switch (child1 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 1 */
            close(pipefds[0]);
            dup2(pipefds[1], STDOUT_FILENO);
            close(pipefds[1]);
            execl("/bin/cat", "cat", "/etc/motd", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    switch (child2 = fork()) {
        case -1:
            // handle error ...
            break;
        case 0:   /* child 2 */
            close(pipefds[1]);
            dup2(pipefds[0], STDIN_FILENO);
            close(pipefds[0]);
            execl("/bin/grep", "grep", "[Ww]ombat", NULL);
            exit(1);  // execl() returns only on error
        default:  /* parent */
            // nothing
    }

    // Only the parent reaches this point
    close(pipefds[0]);
    close(pipefds[1]);
    wait(NULL);
    wait(NULL);
}

Um dos erros mais comuns na configuração de um pipeline é esquecer de fazer com que um ou mais dos processos envolvidos fechem as extremidades do tubo que não estão sendo usadas. Como regra geral, o resultado final deve ser que cada extremidade do tubo seja de propriedade de exatamente um processo e esteja aberta apenas nesse processo.

Em um caso como o da função demo acima, a falha do segundo filho ou do pai em fechar pipefds[1] resultará na suspensão do segundo filho, pois o grep continuará esperando pela entrada até ver EOF, e que não será observado enquanto a extremidade de gravação do pipe permanecer aberta em qualquer processo, como o processo pai ou o próprio segundo filho.
