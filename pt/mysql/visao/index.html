Sintaxe #

  • CREATE VIEW view_name AS SELECT column_name(s) FROM table_name WHERE condição; ///Sintaxe de visualização de criação simples

  • CRIO [OR REPLACE] [ALGORITHM = {UNDEFINED | MERGE | TEMPTABLE}] [DEFINER = { user | CURRENT_USER }] [SQL SECURITY { DEFINER | INVOKER }] VIEW view_name [(column_list)] AS select_statement [WITH [CASCADED | LOCAL] CHECK OPTION]; /// Full Create view syntax

  • DROP VIEW [SE EXISTE] [db_name.]view_name; ///Sintaxe de visualização suspensa

Parâmetros #

Parâmetros Detalhes
view_name Nome da Vista
instrução SELECT Instruções SQL a serem empacotadas nas visualizações. Pode ser uma instrução SELECT para buscar dados de uma ou mais tabelas.

As visualizações são tabelas virtuais e não contêm os dados retornados. Eles podem evitar que você escreva consultas complexas repetidas vezes.

  • Antes de uma visualização ser feita sua especificação consiste inteiramente em uma instrução SELECT. A instrução SELECT não pode conter uma subconsulta na cláusula FROM.
  • Uma vez que uma visão é feita ela é usada em grande parte como uma tabela e pode ser ‘SELECIONADA’ a partir de uma tabela.

Você precisa criar visualizações, quando quiser restringir algumas colunas da sua tabela, do outro usuário.

  • Por exemplo: Em sua organização, você deseja que seus gerentes visualizem poucas informações de uma tabela chamada “Vendas”, mas não deseja que seus engenheiros de software possam visualizar todos os campos da tabela “Vendas”. Aqui, você pode criar duas visualizações diferentes para seus gerentes e engenheiros de software.

Atuação. VIEWs são açúcar sintático. No entanto, o desempenho pode ou não ser pior do que a consulta equivalente com a seleção da visualização dobrada. O Optimizer tenta fazer isso “dobrar” para você, mas nem sempre é bem-sucedido. O MySQL 5.7.6 fornece mais algumas melhorias no Optimizer. Mas, independentemente, usar um VIEW não irá gerar uma consulta faster.

Criar uma visualização #

Privilégios

A instrução CREATE VIEW requer o privilégio CREATE VIEW para a exibição e algum privilégio para cada coluna selecionada pela instrução SELECT. Para colunas usadas em outro lugar na instrução SELECT, você deve ter o privilégio SELECT. Se a cláusula OR REPLACE estiver presente, você também deverá ter o privilégio DROP para a visualização. CREATE VIEW também pode exigir o privilégio SUPER, dependendo do valor DEFINER, conforme descrito posteriormente nesta seção.

Quando uma visualização é referenciada, ocorre a verificação de privilégios.

Uma exibição pertence a um banco de dados. Por padrão, uma nova exibição é criada no banco de dados padrão. Para criar a exibição explicitamente em um determinado banco de dados, use um nome totalmente qualificado

Por exemplo:

db_name.view_name

mysql> CREATE VIEW test.v AS SELECT * FROM t;

Observação - Em um banco de dados, as tabelas base e as visualizações compartilham o mesmo namespace, portanto, uma tabela base e uma visualização não podem ter o mesmo nome.

Uma VIEW pode:

  • ser criado a partir de vários tipos de instruções SELECT
  • consulte as tabelas base ou outras visualizações
  • use junções, UNION e subconsultas
  • SELECT não precisa nem se referir a nenhuma tabela

Outro exemplo

O exemplo a seguir define uma exibição que seleciona duas colunas de outra tabela, bem como uma expressão calculada a partir dessas colunas:

mysql> CREATE TABLE t (qty INT, price INT);
mysql> INSERT INTO t VALUES(3, 50);
mysql> CREATE VIEW v AS SELECT qty, price, qty*price AS value FROM t;
mysql> SELECT * FROM v;
+------+-------+-------+
| qty  | price | value |
+------+-------+-------+
|    3 |    50 |   150 |
+------+-------+-------+

Restrições

  • Antes do MySQL 5.7.7, a instrução SELECT não pode conter uma subconsulta em a cláusula FROM.
  • A instrução SELECT não pode se referir a variáveis ​​do sistema ou definidas pelo usuário variáveis.
  • Dentro de um programa armazenado, a instrução SELECT não pode se referir ao programa parâmetros ou variáveis ​​locais.
  • A instrução SELECT não pode fazer referência aos parâmetros da instrução preparada.
  • Qualquer tabela ou visualização referida na definição deve existir. Depois de foi criada, é possível remover uma tabela ou visão que a definição se refere. Nesse caso, o uso da visualização resulta em uma erro. Para verificar uma definição de visão para problemas desse tipo, use o Instrução CHECK TABLE.
  • A definição não pode se referir a uma tabela TEMPORÁRIA, e você não pode crie uma visualização TEMPORÁRIA.
  • Você não pode associar um gatilho a uma visualização.
  • Os aliases para nomes de colunas na instrução SELECT são verificados em relação o comprimento máximo da coluna de 64 caracteres (não o alias máximo comprimento de 256 caracteres).
  • Um VIEW pode ou não otimizar, assim como o equivalente SELECT. É improvável que otimize melhor.

Uma visão de duas tabelas #

Uma visualização é mais útil quando pode ser usada para extrair dados de mais de uma tabela.

CREATE VIEW myview AS
SELECT a.*, b.extra_data FROM main_table a 
LEFT OUTER JOIN other_table b 
ON a.id = b.id

Em visualizações mysql não são materializadas. Se você agora executar a consulta simples SELECT * FROM myview, o mysql irá realmente executar o LEFT JOIN nos bastidores.

Uma visualização, uma vez criada, pode ser unida a outras visualizações ou tabelas

Atualizando uma tabela via VIEW #

Um VIEW funciona como uma tabela. Embora você possa UPDATE uma tabela, você pode ou não ser capaz de atualizar uma visão nessa tabela. Em geral, se o SELECT na visualização for complexo o suficiente para exigir uma tabela temporária, então UPDATE não é permitido.

Coisas como GROUP BY, UNION, HAVING, DISTINCT e algumas subconsultas impedem que a visualização seja atualizável. Detalhes em reference manual.

REDUZINDO UMA VISUALIZAÇÃO #

– Cria e solta uma visualização no banco de dados atual.

CREATE VIEW few_rows_from_t1 AS SELECT * FROM t1 LIMIT 10;
DROP VIEW few_rows_from_t1;

– Cria e descarta uma exibição referenciando uma tabela em um banco de dados diferente.

CREATE VIEW table_from_other_db AS SELECT x FROM db1.foo WHERE x IS NOT NULL;
DROP VIEW table_from_other_db;
