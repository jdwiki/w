Sintaxe #

  • Não use DISTINCT e GROUP BY no mesmo SELECT.

  • Não paginar via OFFSET, “lembre-se de onde parou”.

  • WHERE (a,b) = (22,33) não otimiza nada.

  • Diga explicitamente ALL ou DISTINCT após UNION – isso lembra que você pode escolher entre o ALL mais rápido ou o DISTINCT mais lento.

  • Não use SELECT *, especialmente se você tiver colunas TEXT ou BLOB que você não precisa. Há sobrecarga nas tabelas tmp e na transmissão.

  • É mais rápido quando GROUP BY e ORDER BY podem ter exatamente a mesma lista.

  • Não use FORCE INDEX; pode ajudar hoje, mas provavelmente vai doer amanhã.

Veja também discussões sobre ORDER BY, LIKE, REGEXP, etc. Nota: isso precisa ser editado com links e mais tópicos.

Livro de receitas sobre como construir índices ideais.

Adicione o índice correto #

Este é um tópico enorme, mas também é a questão de “desempenho” mais importante.

A principal lição para um iniciante é aprender sobre índices “compostos”. Aqui está um exemplo rápido:

INDEX(last_name, first_name)

é excelente para estes:

WHERE last_name = '...'
WHERE first_name = '...' AND last_name = '...'   -- (order in WHERE does not matter)

mas não para

WHERE first_name = '...'   -- order in INDEX _does_ matter
WHERE last_name = '...' OR first_name = '...'   -- "OR" is a killer

Não se esconda na função #

Um erro comum é ocultar uma coluna indexada dentro de uma chamada de função. Por exemplo, isso não pode ser ajudado por um índice:

WHERE DATE(dt) = '2000-01-01'

Em vez disso, dado INDEX(dt), eles podem usar o índice:

WHERE dt = '2000-01-01'  -- if `dt` is datatype `DATE`

Isso funciona para DATE, DATETIME, TIMESTAMP e até mesmo DATETIME(6) (microssegundos):

WHERE dt >= '2000-01-01'
  AND dt  < '2000-01-01' + INTERVAL 1 DAY

OU #

Defina o cache corretamente #

innodb_buffer_pool_size deve ser cerca de 70% da RAM disponível.

Negativos #

Aqui estão algumas coisas que provavelmente não ajudarão no desempenho. Eles decorrem de informações desatualizadas e/ou ingenuidade.

  • O InnoDB melhorou até o ponto em que é improvável que o MyISAM seja melhor.
  • PARTITIONing raramente oferece benefícios de desempenho; pode até prejudicar o desempenho.
  • Definir query_cache_size maior que 100M geralmente prejudicará o desempenho.
  • Aumentar muitos valores em my.cnf pode levar a ’troca’, que é um sério problema de desempenho.
  • “Índices de prefixo” (como INDEX(foo(20))) são geralmente inúteis.
  • OPTIMIZE TABLE é quase sempre inútil. (E envolve bloquear a mesa.)

INSCREVA-SE + GRUPO POR #

Evite construções ineficientes #

x IN ( SELECT ... )

transformar em um ‘JOIN’

Quando possível, evite OR.

Não ’esconda’ uma coluna indexada em uma função, como WHERE DATE(x) = ...; reformule como WHERE x = ...

Você geralmente pode evitar WHERE LCASE(name1) = LCASE(name2) tendo um agrupamento adequado.

Não use OFFSET para “paginação”, em vez disso, ’lembre-se de onde você parou'.

Evite SELECT * ... (a menos que depuração).
