SELECT é usado para recuperar linhas selecionadas de uma ou mais tabelas.

Sintaxe #

  • SELECT DISTINCT [expressões] FROM TableName [WHERE condições]; ///Seleção simples

  • SELECT DISTINCT(a), b … é o mesmo que SELECT DISTINCT a, b …

  • SELECIONAR [ TODOS | DISTINTO | DISTINTROW ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ] [ SQL_SMALL_RESULT | SQL_BIG_RESULT ] [ SQL_BUFFER_RESULT ] [ SQL_CACHE | SQL_NO_CACHE ] [ SQL_CALC_FOUND_ROWS ] expressões DAS tabelas [ONDE condições] [GROUP BY expressões] [TER condição] [Expressão ORDER BY [ ASC | DESC ]] [LIMIT [valor_deslocamento] número_linhas | LIMIT number_rows OFFSET offset_value] [PROCEDIMENTO nome_do_procedimento] [INTO [ OUTFILE ’nome_do_arquivo’ opções | DUMPFILE ‘file_name’ | @variable1, @variable2, … @variable_n] [PARA ATUALIZAÇÃO | BLOQUEIO EM MODO COMPARTILHAMENTO]; ///Sintaxe de seleção completa

Para mais informações sobre a instrução SELECT do MySQL, consulte MySQL Docs.

SELECT com DISTINCT #

A cláusula DISTINCT após SELECT elimina linhas duplicadas do conjunto de resultados.

CREATE TABLE `car`
(   `car_id` INT UNSIGNED NOT NULL PRIMARY KEY, 
    `name` VARCHAR(20), 
    `price` DECIMAL(8,2)
);

INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (1, 'Audi A1', '20000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (2, 'Audi A1', '15000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (3, 'Audi A2', '40000');
INSERT INTO CAR (`car_id`, `name`, `price`) VALUES (4, 'Audi A2', '40000');

SELECT DISTINCT `name`, `price` FROM CAR;
+---------+----------+
| name    | price    |
+---------+----------+
| Audi A1 | 20000.00 |
| Audi A1 | 15000.00 |
| Audi A2 | 40000.00 |
+---------+----------+

DISTINCT funciona em todas as colunas para fornecer os resultados, não em colunas individuais. Este último é muitas vezes um equívoco de novos desenvolvedores de SQL. Em resumo, é a distinção no nível da linha do conjunto de resultados que importa, não a distinção no nível da coluna. Para visualizar isso, veja “Audi A1” no conjunto de resultados acima.

Para versões posteriores do MySQL, DISTINCT tem implicações com seu uso junto com ORDER BY. A configuração para ONLY_FULL_GROUP_BY entra em jogo como visto na seguinte página de manual do MySQL intitulada MySQL Handling of GROUP BY.

SELECIONE todas as colunas (*) #

Consulta

SELECT * FROM stack;

Resultado

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
|    2 | stack    | stack    |
+------+----------+----------+
2 rows in set (0.00 sec)

Você pode selecionar todas as colunas de uma tabela em uma junção fazendo:

SELECT stack.* FROM stack JOIN Overflow ON stack.id = Overflow.id;

Prática recomendada Não use * a menos que você esteja depurando ou buscando a(s) linha(s) em arrays associativos, caso contrário, alterações de esquema (ADD/DROP/reorganizar colunas) podem levar a erros desagradáveis ​​de aplicativo. Além disso, se você fornecer a lista de colunas necessárias em seu conjunto de resultados, o planejador de consultas do MySQL geralmente pode otimizar a consulta.

Prós:

  1. Ao adicionar/remover colunas, você não precisa fazer alterações onde usou SELECT *
  2. É mais curto escrever
  3. Você também vê as respostas, então o uso de SELECT * pode ser justificado?

Contras:

  1. Você está retornando mais dados do que precisa. Digamos que você adicione uma coluna VARBINARY que contém 200k por linha. Você só precisa desses dados em um lugar para um único registro - usando SELECT * você pode acabar retornando 2 MB por 10 linhas que você não precisa
  2. Explícito sobre quais dados são usados
  3. Especificar colunas significa que você recebe um erro quando uma coluna é removida
  4. O processador de consultas precisa trabalhar um pouco mais - descobrir quais colunas existem na tabela (obrigado @vinodadhikary)
  5. Você pode encontrar onde uma coluna é usada com mais facilidade
  6. Você obtém todas as colunas em junções se usar SELECT *
  7. Você não pode usar referências ordinais com segurança (embora usar referências ordinais para colunas seja uma prática ruim em si)
  8. Em consultas complexas com campos TEXT, a consulta pode ser retardada pelo processamento de tabela temporária menos ideal

INSERT INTO stack (`id`, `username`, `password`) VALUES (1, 'Foo', 'hiddenGem');

SELECIONE com LIKE (%) #

CREATE TABLE stack
(  id int AUTO_INCREMENT PRIMARY KEY,
   username VARCHAR(100) NOT NULL
);

INSERT stack(username) VALUES 
('admin'),('k admin'),('adm'),('a adm b'),('b XadmY c'), ('adm now'), ('not here');

“adm” em qualquer lugar:

SELECT * FROM stack WHERE username LIKE "%adm%";  
+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | k admin   |
|  3 | adm       |
|  4 | a adm b   |
|  5 | b XadmY c |
|  6 | adm now   |
+----+-----------+

Começa com “adm”:

SELECT * FROM stack WHERE username LIKE "adm%";
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
|  3 | adm      |
|  6 | adm now  |
+----+----------+

Termina com “adm”:

SELECT * FROM stack WHERE username LIKE "%adm"; 
+----+----------+
| id | username |
+----+----------+
|  3 | adm      |
+----+----------+

Assim como o caractere % em uma cláusula LIKE corresponde a qualquer número de caracteres, o caractere _ corresponde a apenas um caractere. Por exemplo,

SELECT * FROM stack WHERE username LIKE "adm_n"; 
+----+----------+
| id | username |
+----+----------+
|  1 | admin    |
+----+----------+

Notas de desempenho Se houver um índice em username, então

  • LIKE 'adm' executa o mesmo que `= ‘adm’
  • LIKE 'adm% é um “intervalo”, semelhante a BETWEEN..AND.. Pode fazer bom uso de um índice na coluna.
  • LIKE '%adm' (ou qualquer variante com um curinga leading) não pode usar nenhum índice. Portanto, será lento. Em tabelas com muitas linhas, é provável que seja tão lento que seja inútil.
  • RLIKE (REGEXP) tende a ser mais lento que LIKE, mas tem mais recursos.
  • Enquanto o MySQL oferece indexação FULLTEXT em muitos tipos de tabela e coluna, esses índices FULLTEXT não são usados ​​para preencher consultas usando LIKE.

SELECT com CASE ou IF #

Consulta

SELECT st.name,
       st.percentage, 
       CASE WHEN st.percentage >= 35 THEN 'Pass' ELSE 'Fail' END AS `Remark` 
FROM student AS st ;

Resultado

+--------------------------------+
|   name   | percentage | Remark |
+--------------------------------+
|   Isha   |     67     |  Pass  |
|   Rucha  |     28     |  Fail  |
|   Het    |     35     |  Pass  |
|   Ansh   |     92     |  Pass  |
+--------------------------------+

Ou com SE

SELECT st.name,
       st.percentage, 
       IF(st.percentage >= 35, 'Pass', 'Fail') AS `Remark` 
FROM student AS st ;

N.B

IF(st.percentage >= 35, 'Pass', 'Fail')

Isso significa: IF st.percentage >= 35 é TRUE então retorne 'Pass' ELSE return 'Falha'

SELECT com Alias ​​(AS) #

Os aliases SQL são usados ​​para renomear temporariamente uma tabela ou coluna. Eles geralmente são usados ​​para melhorar a legibilidade.

Consulta

SELECT username AS val FROM stack; 
SELECT username val FROM stack;

(Nota: AS é sintaticamente opcional.)

Resultado

+-------+
| val   |
+-------+
| admin |
| stack |
+-------+
2 rows in set (0.00 sec)

SELECT com uma cláusula LIMIT #

Consulta:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 3;

Resultado:

CustomerID CustomerName ContactName Address City PostalCode Country 1

Alfreds Futterkiste Maria Anders Obere Str. 57 Berlin 12209 Germany 2 Ana Trujillo Emparedados y helados Ana Trujillo Avda. de la Constitución 2222 México D.F. 05021 Mexico 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Prática recomendada Sempre use ORDER BY ao usar LIMIT; caso contrário, as linhas que você obterá serão imprevisíveis.

Consulta:

 SELECT *
   FROM Customers
  ORDER BY CustomerID 
  LIMIT 2,1;

Explicação:

Quando uma cláusula LIMIT contém dois números, ela é interpretada como LIMIT offset,count. Portanto, neste exemplo, a consulta pula dois registros e retorna um.

Resultado:

CustomerID CustomerName ContactName Address City PostalCode Country 3 Antonio Moreno Taquería Antonio Moreno Mataderos 2312 México D.F. 05023 Mexico

Observação:

Os valores nas cláusulas LIMIT devem ser constantes; eles podem não ser valores de coluna.

Consulta

SELECT * FROM stack WHERE username = "admin" AND password = "admin";

Resultado

+------+----------+----------+
| id   | username | password |
+------+----------+----------+
|    1 | admin    | admin    |
+------+----------+----------+
1 row in set (0.00 sec)

Consulta com um SELECT aninhado na cláusula WHERE #

A cláusula WHERE pode conter qualquer instrução SELECT válida para escrever consultas mais complexas. Esta é uma consulta ‘aninhada’

Consulta

As consultas aninhadas geralmente são usadas para retornar valores atômicos únicos de consultas para comparações.

SELECT title FROM books WHERE author_id = (SELECT id FROM authors WHERE last_name = 'Bar' AND first_name = 'Foo');

Seleciona todos os nomes de usuário sem endereço de e-mail

SELECT * FROM stack WHERE username IN (SELECT username FROM signups WHERE email IS NULL);

Isenção de responsabilidade: considere usar joins para melhorias de desempenho ao comparar um conjunto de resultados completo.

SELECT com BETWEEN #

Você pode usar a cláusula BETWEEN para substituir uma combinação de condições “maior que igual E menor que igual”.

Dados

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
|  6 | java      |
+----+-----------+

Consulta com operadores

SELECT * FROM stack WHERE id >= 2 and id <= 5;

Consulta semelhante com BETWEEN

SELECT * FROM stack WHERE id BETWEEN 2 and 5;

Resultado

+----+-----------+
| id | username  |
+----+-----------+
|  2 | root      |
|  3 | toor      |
|  4 | mysql     |
|  5 | thanks    |
+----+-----------+
4 rows in set (0.00 sec)

Observação

BETWEEN usa >= e <=, não > e <.

Usando NÃO ENTRE

Se você quiser usar o negativo, você pode usar NOT. Por exemplo :

SELECT * FROM stack WHERE id NOT BETWEEN 2 and 5;

Resultado

+----+-----------+
| id | username  |
+----+-----------+
|  1 | admin     |
|  6 | java      |
+----+-----------+
2 rows in set (0.00 sec)

Observação

NOT BETWEEN usa > e < e não >= e <= Ou seja, WHERE id NOT BETWEEN 2 and 5 é o mesmo que `WHERE (id < 2 OR id > 5)».

Se você tiver um índice em uma coluna que você usa em uma pesquisa BETWEEN, o MySQL pode usar esse índice para uma varredura de intervalo.

SELECIONE com LIKE(_) #

Um caractere _ em um padrão de cláusula LIKE corresponde a um único caractere.

Consulta

SELECT username FROM users WHERE users LIKE 'admin_';

Resultado

+----------+
| username |  
+----------+
| admin1   |
| admin2   |
| admin-   |
| adminA   |
+----------+

SELECT com intervalo de datas #

SELECT ... WHERE dt >= '2017-02-01'
             AND dt  < '2017-02-01' + INTERVAL 1 MONTH

Claro, isso pode ser feito com BETWEEN e inclusão de 23:59:59. Mas, o padrão tem esses benefícios:

  • Você não precisa pré-calcular a data de término (que geralmente é uma duração exata desde o início)
  • Você não inclui os dois endpoints (como BETWEEN), nem digite ‘23:59:59’ para evitá-lo.
  • Funciona para DATE, TIMESTAMP, DATETIME, e até mesmo para o DATETIME(6) incluído em microssegundos.
  • Cuida de dias bissextos, fim de ano, etc.
  • É amigável ao índice (assim como BETWEEN).
