Código de erro 1064: erro de sintaxe #

select LastName, FirstName,
from Person

Mensagem de retorno:

Código de erro: 1064. Você tem um erro em sua sintaxe SQL; verifique o manual que corresponde à versão do seu servidor MySQL para a sintaxe correta para usar perto de ‘from Person’ na linha 2.

Receber uma mensagem “1064 error” do MySQL significa que a consulta não pode ser analisada sem erros de sintaxe. Em outras palavras, não pode dar sentido à consulta.

A citação na mensagem de erro começa com o primeiro caractere da consulta que o MySQL não consegue descobrir como analisar. Neste exemplo, o MySQL não pode fazer sentido, no contexto, de from Person. Neste caso, há uma vírgula extra imediatamente antes de from Person. A vírgula diz ao MySQL para esperar outra descrição de coluna na cláusula SELECT

Um erro de sintaxe sempre diz ... near '...'. A coisa no início das aspas está muito perto de onde está o erro. Para localizar um erro, observe o primeiro token nas aspas e o último token antes das aspas.

Às vezes você vai chegar ... perto de ''; ou seja, nada nas aspas. Isso significa que o primeiro caractere que o MySQL não consegue descobrir está bem no final ou no início da instrução. Isso sugere que a consulta contém aspas não balanceadas (' ou ") ou parênteses não balanceados ou que você não encerrou a instrução antes corretamente.

No caso de uma rotina armazenada, você pode ter esquecido de usar corretamente DELIMITER.

Portanto, quando você receber o erro 1064, observe o texto da consulta e encontre o ponto mencionado na mensagem de erro. Inspecione visualmente o texto da consulta em torno desse ponto.

Se você pedir ajuda a alguém para solucionar o erro 1064, é melhor fornecer o texto de toda a consulta e o texto da mensagem de erro.

Código de erro 1175: Atualização segura #

Este erro aparece ao tentar atualizar ou excluir registros sem incluir a cláusula WHERE que usa a coluna KEY.

Para executar a exclusão ou atualização de qualquer maneira - digite:

SET SQL_SAFE_UPDATES = 0;

Para ativar o modo de segurança novamente - digite:

SET SQL_SAFE_UPDATES = 1;

Código de erro 1215: Não é possível adicionar restrição de chave estrangeira #

Este erro ocorre quando as tabelas não estão estruturadas adequadamente para lidar com a verificação de pesquisa rápida de requisitos de chave estrangeira (FK) que o desenvolvedor está exigindo.

CREATE TABLE `gtType` (
  `type` char(2) NOT NULL,
  `description` varchar(1000) NOT NULL,
  PRIMARY KEY (`type`)
) ENGINE=InnoDB;

CREATE TABLE `getTogethers` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `type` char(2) NOT NULL,
  `eventDT` datetime NOT NULL,
  `location` varchar(1000) NOT NULL,
  PRIMARY KEY (`id`),
  KEY `fk_gt2type` (`type`), -- see Note1 below 
  CONSTRAINT `gettogethers_ibfk_1` FOREIGN KEY (`type`) REFERENCES `gtType` (`type`)
) ENGINE=InnoDB;

Nota1: uma KEY como esta será criada automaticamente se necessário devido à definição FK na linha que o segue. O desenvolvedor pode ignorá-lo e a KEY (também conhecida como índice) será adicionada, se necessário. Um exemplo dele sendo ignorado pelo desenvolvedor é mostrado abaixo em someOther.

Até aí tudo bem, até a chamada abaixo.

CREATE TABLE `someOther` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `someDT` datetime NOT NULL,
  PRIMARY KEY (`id`),
  CONSTRAINT `someOther_dt` FOREIGN KEY (`someDT`) REFERENCES `getTogethers` (`eventDT`)
) ENGINE=InnoDB;

Código de erro: 1215. Não é possível adicionar restrição de chave estrangeira

Neste caso, ele falha devido à falta de um índice na tabela referenciada getTogethers para lidar com a pesquisa rápida de um eventDT. A ser resolvido na próxima declaração.

CREATE INDEX `gt_eventdt` ON getTogethers (`eventDT`);

A tabela getTogethers foi modificada, e agora a criação de someOther será bem-sucedida.

Da página de manual do MySQL Usando restrições FOREIGN KEY:

MySQL requer índices em chaves estrangeiras e chaves referenciadas para que verificações de chave estrangeira podem ser rápidas e não requerem uma verificação de tabela. No tabela de referência, deve haver um índice onde a chave estrangeira colunas são listadas como as primeiras colunas na mesma ordem. Tal o índice é criado na tabela de referência automaticamente se não existem.

As colunas correspondentes na chave estrangeira e a chave referenciada devem têm tipos de dados semelhantes. O tamanho e o sinal dos tipos inteiros devem ser o mesmo. O comprimento dos tipos de string não precisa ser o mesmo. Por colunas de string não binárias (caracteres), o conjunto de caracteres e o agrupamento deve ser o mesmo.

O InnoDB permite que uma chave estrangeira faça referência a qualquer coluna de índice ou grupo de colunas. No entanto, na tabela referenciada, deve haver um índice onde as colunas referenciadas são listadas como as primeiras colunas no mesma ordem.

Observe que o último ponto acima sobre as primeiras colunas (mais à esquerda) e a falta de um requisito de chave primária (embora altamente recomendado).

Após a criação bem-sucedida de uma tabela referenciada (filho), todas as chaves que foram criadas automaticamente para você ficam visíveis com um comando como o seguinte:

SHOW CREATE TABLE someOther;

Outros casos comuns de ocorrência desse erro incluem, conforme mencionado acima nos documentos, mas devem ser destacados:

  • Diferenças aparentemente triviais em INT que é assinado, apontando em direção a INT UNSIGNED.

  • Desenvolvedores com problemas para entender as KEYS de várias colunas (compostas) e os primeiros requisitos de pedido (mais à esquerda).

1067, 1292, 1366, 1411 - Valor incorreto para número, data, padrão, etc. #

1067 Isso provavelmente está relacionado aos padrões de TIMESTAMP, que mudaram ao longo do tempo. Consulte Padrões de TIMESTAMP na página Datas e Horas. (que ainda não existe)

1292/1366 DOUBLE/Integer Verifique se há letras ou outros erros de sintaxe. Verifique se as colunas estão alinhadas; talvez você pense que está colocando em um VARCHAR mas está alinhado com uma coluna numérica.

1292 DATETIME Verifique se há muito tempo no passado ou no futuro. Verifique entre 2h e 3h de uma manhã em que o horário de verão mudou. Verifique se há sintaxe incorreta, como coisas de fuso horário +00.

1292 VARIABLE Verifique os valores permitidos para a VARIABLE que você está tentando SET.

1292 LOAD DATA Observe a linha que está ‘ruim’. Verifique os símbolos de escape, etc. Observe os tipos de dados.

1411 STR_TO_DATE Data formatada incorretamente?

1045 Acesso negado #

Veja as discussões em “GRANT” e “Recuperando a senha do root”.

1236 “posição impossível” na replicação #

Normalmente isso significa que o Master travou e que o sync_binlog estava DESLIGADO. A solução é CHANGE MASTER para POS=0 do próximo arquivo binlog (veja o Master) no Slave.

A causa: O Mestre envia itens de replicação para o Escravo antes de liberar para seu log binário (quando sync_binlog=OFF). Se o Master travar antes do flush, o Slave já passou logicamente do final do arquivo no log binário. Quando o mestre é inicializado novamente, ele inicia um novo log binário, portanto, MUDAR para o início desse log binário é a melhor solução disponível.

Uma solução de longo prazo é sync_binlog=ON, se você puder pagar a E/S extra que ela causa.

2002, 2003 Não é possível conectar #

Verifique se há um problema de firewall bloqueando a porta 3306.

Alguns diagnósticos e/ou soluções possíveis

  • O servidor está realmente funcionando?
  • “service firewalld stop” e “systemctl disable firewalld”
  • telnet mestre 3306
  • Verifique o endereço de ligação
  • verifique skip-name-resolve
  • verifique o soquete.

126, 127, 134, 144, 145 #

Ao tentar acessar os registros do banco de dados MySQL, você pode receber essas mensagens de erro. Essas mensagens de erro ocorreram devido à corrupção no banco de dados MySQL. Seguem os tipos

MySQL error code 126 = Index file is crashed
MySQL error code 127 = Record-file is crashed
MySQL error code 134 = Record was already deleted (or record file crashed)
MySQL error code 144 = Table is crashed and last repair failed
MySQL error code 145 = Table was marked as crashed and should be repaired

Bug do MySQL, ataque de vírus, travamento do servidor, desligamento inadequado, tabela danificada são a razão por trás dessa corrupção. Quando é corrompido, torna-se inacessível e você não pode mais acessá-los. Para obter acessibilidade, a melhor maneira de recuperar dados de um backup atualizado. No entanto, se você não tiver atualizado ou nenhum backup válido, poderá optar pelo MySQL Repair.

Se o tipo de mecanismo de tabela for MyISAM, aplique CHECK TABLE e depois REPAIR TABLE a ele.

Então pense seriamente em converter para InnoDB, para que esse erro não aconteça novamente.

Sintaxe

CHECK TABLE <table name> ////To check the extent of database corruption
REPAIR TABLE <table name> ////To repair table

139 #

O erro 139 pode significar que o número e o tamanho dos campos na definição da tabela excedem algum limite. Soluções alternativas:

  • Repense o esquema
  • Normalizar alguns campos
  • Divida a mesa verticalmente

1366 #

Isso geralmente significa que a manipulação do conjunto de caracteres não foi consistente entre cliente e servidor. Consulte … para obter mais assistência.

24 Não é possível abrir o arquivo (muitos arquivos abertos)

open_files_limit vem de uma configuração do SO. table_open_cache precisa ser menor que isso.

Estes podem causar esse erro:

  • Falha ao DEALLOCATE PREPARE em um procedimento armazenado.

  • Tabela(s) PARTICIONADA(s) com um grande número de partições e innodb_file_per_table = ON. Recomenda-se não ter mais de 50 partições em uma determinada tabela (por vários motivos). (Quando as “Partições Nativas” estiverem disponíveis, este aviso poderá mudar.)

A solução óbvia é definir aumentar o limite do SO: Para permitir mais arquivos, altere ulimit ou /etc/security/limits.conf ou em sysctl.conf (kern.maxfiles & kern.maxfilesperproc) ou qualquer outra coisa (dependente do SO). Então aumente open_files_limit e table_open_cache.

A partir do 5.6.8, open_files_limit é dimensionado automaticamente com base em max_connections, mas não há problema em alterá-lo do padrão.

1062 - Entrada Duplicada

Este erro ocorre principalmente devido aos dois motivos a seguir

  1. Valor duplicado - Código de erro: 1062. Entrada duplicada '12' para chave ‘PRIMARY’

    The primary key column is unique and it will not accept the duplicate entry. So when you are trying to insert a new row which is already present in you table will produce this error.

Para resolver isso, defina a coluna de chave primária como AUTO_INCREMENT. E ao tentar inserir uma nova linha, ignore a chave primária coluna ou insira o valor NULL na chave primária.

CREATE TABLE userDetails(
  userId INT(10) NOT NULL AUTO_INCREMENT,
  firstName VARCHAR(50),
  lastName VARCHAR(50),
  isActive INT(1) DEFAULT 0,
  PRIMARY KEY (userId) );

--->and now while inserting 
INSERT INTO userDetails VALUES (NULL ,'John', 'Doe', 1);

  1. Campo de dados exclusivo - Código de erro: 1062. Entrada duplicada 'A' para chave ‘code’

    You may assigned a column as unique and trying to insert a new row with already existing value for that column will produce this error.

Para superar este erro, use INSERT IGNORE em vez do INSERT normal. Se a nova linha que você está tentando inserir não duplicar um registro existente, o MySQL a insere normalmente. Se o registro for duplicado, a palavra-chave IGNORE o descarta sem gerar nenhum erro.

INSERT IGNORE INTO userDetails VALUES (NULL ,'John', 'Doe', 1);
