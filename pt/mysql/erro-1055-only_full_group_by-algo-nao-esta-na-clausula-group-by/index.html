Erro 1055 ONLY_FULL_GROUP_BY algo não está na cláusula GROUP BY ...

Recentemente, novas versões de servidores MySQL começaram a gerar erros 1055 para consultas que costumavam funcionar. Este tópico explica esses erros.

A equipe do MySQL tem trabalhado para aposentar a extensão não padrão para GROUP BY, ou pelo menos para tornar mais difícil para os desenvolvedores que escrevem consultas serem queimados por ela.

Por muito tempo, o MySQL contém uma notória extensão não padrão para GROUP BY, que permite um comportamento excêntrico em nome da eficiência. Essa extensão permitiu que inúmeros desenvolvedores ao redor do mundo usassem GROUP BY no código de produção sem entender completamente o que estavam fazendo.

Em particular, é uma má idéia usar SELECT * em uma consulta GROUP BY, porque uma cláusula padrão GROUP BY requer a enumeração das colunas. Muitos desenvolvedores, infelizmente, fizeram isso.

Leia isso. https://dev.mysql.com/doc/refman/5.7/en/group-by-handling.html

A equipe do MySQL está tentando corrigir esse erro sem estragar o código de produção. Eles adicionaram um sinalizador sql_mode no 5.7.5 chamado ONLY_FULL_GROUP_BY para obrigar o comportamento padrão. Em uma versão recente, eles ativaram esse sinalizador por padrão. Quando você atualizou seu MySQL local para 5.7.14, o sinalizador foi ativado e seu código de produção, dependente da extensão antiga, parou de funcionar.

Se você começou recentemente a receber erros 1055, quais são suas escolhas?

  1. corrija as consultas SQL ofensivas ou faça com que seus autores façam isso.
  2. reverter para uma versão pronta para uso do MySQL compatível com o software aplicativo que você usa.
  3. mude o sql_mode do seu servidor para se livrar do modo ONLY_FULL_GROUP_BY recém-definido.

Você pode mudar o modo fazendo um comando SET.

SET  sql_mode = 'STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'

deve fazer o truque se você fizer isso logo após seu aplicativo se conectar ao MySQL.

Ou você pode encontrar o arquivo init em sua instalação do MySQL, localizar a linha sql_mode= e alterá-la para omitir ONLY_FULL_GROUP_BY e reinicie seu servidor.

Uso indevido de GROUP BY para retornar resultados imprevisíveis: Lei de Murphy #

SELECT item.item_id, uses.category,   /* nonstandard */ 
       COUNT(*) number_of_uses 
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

mostrará as linhas em uma tabela chamada item e mostrará a contagem de linhas relacionadas em uma tabela chamada use. Também mostrará o valor de uma coluna chamada uses.category.

Esta consulta funciona no MySQL (antes do sinalizador ONLY_FULL_GROUP_BY aparecer). Ele usa extensão não padrão do MySQL para GROUP BY.

Mas a consulta tem um problema: se várias linhas na tabela uses corresponderem à condição ON na cláusula JOIN, o MySQL retornará a coluna category de apenas uma dessas linhas. Qual linha? O escritor da consulta e o usuário do aplicativo não sabem disso com antecedência. Formalmente falando, é imprevisível: o MySQL pode retornar qualquer valor que desejar.

Imprevisível é como aleatório, com uma diferença significativa. Pode-se esperar que uma escolha aleatória mude de tempos em tempos. Portanto, se uma escolha for aleatória, você poderá detectá-la durante a depuração ou teste. O resultado imprevisível é pior: o MySQL retorna o mesmo resultado cada vez que você usa a consulta, até que isso não aconteça. Às vezes é uma nova versão do servidor MySQL que causa um resultado diferente. Às vezes, é uma tabela crescente que está causando o problema. O que pode dar errado, dará errado, e quando você não espera. Isso se chama Lei de Murphy.

A equipe do MySQL tem trabalhado para tornar mais difícil para os desenvolvedores cometerem esse erro. Versões mais recentes do MySQL na sequência 5.7 têm um sinalizador sql_mode chamado ONLY_FULL_GROUP_BY. Quando esse sinalizador é definido, o servidor MySQL retorna o erro 1055 e se recusa a executar esse tipo de consulta.

Uso incorreto de GROUP BY com SELECT * e como corrigi-lo. #

Às vezes, uma consulta se parece com isso, com um * na cláusula SELECT.

 SELECT item.*,     /* nonstandard */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

Tal consulta precisa ser refatorada para cumprir o padrão ONLY_FULL_GROUP_BY.

Para fazer isso, precisamos de uma subconsulta que use GROUP BY corretamente para retornar o valor number_of_uses para cada item_id. Esta subconsulta é curta e agradável, porque ela só precisa olhar para a tabela uses.

                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id

Então, podemos juntar essa subconsulta com a tabela item.

 SELECT item.*, usecount.number_of_uses
   FROM item
   JOIN (
                              SELECT item_id, COUNT(*) number_of_uses
                                FROM  uses 
                               GROUP BY item_id
        ) usecount ON item.item_id = usecount.item_id

Isso permite que a cláusula GROUP BY seja simples e correta, e também nos permite usar o especificador *.

Nota: no entanto, desenvolvedores sábios evitam usar o especificador * em qualquer caso. Geralmente, é melhor listar as colunas que você deseja em uma consulta.

Usando e usando mal GROUP BY #

 SELECT item.item_id, item.name,     /* not SQL-92 */ 
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

irá mostrar as linhas em uma tabela chamada item, e mostrará a contagem de linhas relacionadas em uma tabela chamada uses. Isso funciona bem, mas infelizmente não é o SQL-92 padrão.

Por que não? porque a cláusula SELECT (e a cláusula ORDER BY) em consultas GROUP BY devem conter colunas que são

  1. mencionado na cláusula GROUP BY, ou
  2. funções agregadas como COUNT(), MIN() e similares.

A cláusula SELECT deste exemplo menciona item.name, uma coluna que não atende a nenhum desses critérios. MySQL 5.6 e anteriores irão rejeitar esta consulta se o modo SQL contiver ONLY_FULL_GROUP_BY.

Esta consulta de exemplo pode ser feita para estar em conformidade com o padrão SQL-92 alterando a cláusula GROUP BY, assim.

SELECT item.item_id, item.name, 
       COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id, item.name

O padrão SQL-99 posterior permite que uma instrução SELECT omita colunas não agregadas da chave do grupo se o SGBD puder provar uma dependência funcional entre elas e as colunas da chave do grupo. Como item.name é funcionalmente dependente de item.item_id, o exemplo inicial é SQL-99 válido. O MySQL ganhou um provador de dependência funcional na versão 5.7. O exemplo original funciona em ONLY_FULL_GROUP_BY.

QUALQUER VALOR() #

 SELECT item.item_id, ANY_VALUE(uses.tag) tag,   
        COUNT(*) number_of_uses
  FROM item 
  JOIN uses ON item.item_id, uses.item_id
 GROUP BY item.item_id

mostra as linhas em uma tabela chamada item, a contagem de linhas relacionadas e um dos valores na tabela relacionada chamada usos.

Você pode pensar nesta função ANY_VALUE() como um tipo estranho de função agregada. Em vez de retornar uma contagem, soma ou máximo, ele instrui o servidor MySQL a escolher, arbitrariamente, um valor do grupo em questão. É uma maneira de contornar o erro 1055.

Tenha cuidado ao usar ANY_VALUE() em consultas em aplicativos de produção.

Ele realmente deveria ser chamado de SURPRISE_ME(). Ele retorna o valor de alguma linha no grupo GROUP BY. Qual linha ele retorna é indeterminado. Isso significa que depende inteiramente do servidor MySQL. Formalmente, ele retorna um valor imprevisível.

O servidor não escolhe um valor aleatório, é pior que isso. Ele retorna o mesmo valor toda vez que você executa a consulta, até que isso não aconteça. Ele pode mudar, ou não, quando uma tabela cresce ou diminui, ou quando o servidor tem mais ou menos RAM, ou quando a versão do servidor muda, ou quando Marte está retrógrado (o que quer que isso signifique), ou sem motivo algum.

Você foi avisado.
