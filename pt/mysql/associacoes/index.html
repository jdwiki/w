Sintaxe #

  • INNER e OUTER são ignorados.

  • FULL não é implementado no MySQL.

  • “commajoin” (FROM a,b WHERE a.x=b.y) é desaprovado; use FROM a JOIN b ON a.x=b.y.

  • A partir de um JOIN b ON a.x=b.y inclui linhas que correspondem em ambas as tabelas.

  • A partir de um LEFT JOIN b ON a.x=b.y inclui todas as linhas de a, mais dados correspondentes de b ou NULLs se não houver nenhuma linha correspondente.

Junções visualizadas #

Se você é uma pessoa visualmente orientada, este diagrama de Venn pode ajudá-lo a entender os diferentes tipos de ‘JOIN’s que existem no MySQL.

JOIN com subconsulta (tabela “derivada”) #

SELECT x, ...
    FROM ( SELECT y, ... FROM ... ) AS a
    JOIN tbl  ON tbl.x = a.y
    WHERE ...

Isso avaliará a subconsulta em uma tabela temporária e, em seguida, ‘JOIN’ para ’tbl’.

Antes do 5.6, não podia haver um índice na tabela temporária. Então, isso era potencialmente muito ineficiente:

SELECT ...
    FROM ( SELECT y, ... FROM ... ) AS a
    JOIN ( SELECT x, ... FROM ... ) AS b  ON b.x = a.y
    WHERE ...

Com 5.6, o otimizador descobre o melhor índice e o cria na hora. (Isso tem alguma sobrecarga, então ainda não é ‘perfeito’.)

Outro paradigma comum é ter uma subconsulta para inicializar algo:

SELECT 
        @n := @n + 1,
        ...
    FROM ( SELECT @n := 0 ) AS initialize
    JOIN the_real_table
    ORDER BY ...

(Nota: este é tecnicamente um CROSS JOIN (produto cartesiano), conforme indicado pela falta de ON. No entanto, é eficiente porque a subconsulta retorna apenas uma linha que deve corresponder às n linhas em the_real_table .)

O que queremos ver?

Queremos obter uma lista, na qual vemos quem possui quais ferramentas e quais ferramentas podem não ter um proprietário.

As consultas

Para fazer isso, podemos combinar duas consultas usando UNION. Nesta primeira consulta estamos juntando as ferramentas nos proprietários usando um LEFT JOIN. Isso adicionará todos os nossos proprietários ao nosso conjunto de resultados, não importa se eles realmente possuem ferramentas.

Na segunda consulta estamos usando um RIGHT JOIN para unir as ferramentas aos proprietários. Dessa forma, conseguimos obter todas as ferramentas em nosso conjunto de resultados, se elas não forem de propriedade de ninguém, sua coluna de proprietário conterá simplesmente NULL. Ao adicionar uma cláusula WHERE que está filtrando por owners.owner_id IS NULL estamos definindo o resultado como aqueles conjuntos de dados, que ainda não foram retornados pela primeira consulta, pois estamos apenas procurando os dados à direita mesa unida.

Como estamos usando UNION ALL, o conjunto de resultados da segunda consulta será anexado ao conjunto de resultados da primeira consulta.

SELECT `owners`.`owner`, tools.tool
FROM `owners`
LEFT JOIN `tools` ON `owners`.`owner_id` = `tools`.`owner_id`
UNION ALL
SELECT `owners`.`owner`, tools.tool
FROM `owners`
RIGHT JOIN `tools` ON `owners`.`owner_id` = `tools`.`owner_id`
WHERE `owners`.`owner_id` IS NULL;

Exemplos de junção #

Consulta para criar tabela no db

CREATE TABLE `user` (
`id` smallint(5) unsigned NOT NULL AUTO_INCREMENT,
`name` varchar(30) NOT NULL,
`course` smallint(5) unsigned DEFAULT NULL,
PRIMARY KEY (`id`)
) ENGINE=InnoDB;

CREATE TABLE `course` (
`id` smallint(5) unsigned NOT NULL AUTO_INCREMENT,
`name` varchar(50) NOT NULL,
PRIMARY KEY (`id`)
) ENGINE=InnoDB;

Como estamos usando tabelas InnoDB e sabemos que user.course e course.id estão relacionados, podemos especificar um relacionamento de chave estrangeira:

ALTER TABLE `user`
ADD CONSTRAINT `FK_course`
FOREIGN KEY (`course`) REFERENCES `course` (`id`)
ON UPDATE CASCADE;

Consulta de junção (junção interna)

SELECT user.name, course.name
FROM `user`
INNER JOIN `course` on user.course = course.id;

Recupere clientes com pedidos – variações de um tema #

Isso obterá todos os pedidos para todos os clientes:

SELECT c.CustomerName, o.OrderID
    FROM Customers AS c
    INNER JOIN Orders AS o
        ON c.CustomerID = o.CustomerID
    ORDER BY c.CustomerName, o.OrderID;

Isso contará o número de pedidos para cada cliente:

SELECT c.CustomerName, COUNT(*) AS 'Order Count'
    FROM Customers AS c
    INNER JOIN Orders AS o
        ON c.CustomerID = o.CustomerID
    GROUP BY c.CustomerID;
    ORDER BY c.CustomerName;

Além disso, conta, mas provavelmente mais rápido:

SELECT  c.CustomerName,
        ( SELECT COUNT(*) FROM Orders WHERE CustomerID = c.CustomerID ) AS 'Order Count'
    FROM Customers AS c
    ORDER BY c.CustomerName;

Liste apenas o cliente com pedidos.

SELECT  c.CustomerName,
    FROM Customers AS c
    WHERE EXISTS ( SELECT * FROM Orders WHERE CustomerID = c.CustomerID )
    ORDER BY c.CustomerName;

Inner-join para 3 mesas #

vamos supor que temos três tabelas que podem ser usadas para um site simples com Tags.

  • Mesa de punho é para Postes.
  • Segundo para Tags
  • Terceiro para Tags e relação Post

mesa de punho “videogame”

identificação título data_reg Conteúdo
1 BioShock Infinito 2016-08-08 ….

tabela “etiquetas”

identificação nome
1 yennefer
2 elizabeth

tabela “tags_meta”

post_id tag_id
1 2 
SELECT videogame.id,
    videogame.title,
    videogame.reg_date,
    tags.name,
    tags_meta.post_id
FROM tags_meta
INNER JOIN videogame ON videogame.id = tags_meta.post_id
INNER JOIN tags ON tags.id = tags_meta.tag_id
WHERE tags.name = "elizabeth"
ORDER BY videogame.reg_date

este código pode retornar todas as postagens relacionadas a essa tag “#elizabeth”
