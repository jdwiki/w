Log de consulta lenta #

O log de consultas lentas consiste em eventos de log para consultas que levam até long_query_time segundos para serem concluídas. Por exemplo, até 10 segundos para concluir. Para ver o limite de tempo atualmente definido, emita o seguinte:

SELECT @@long_query_time;
+-------------------+
| @@long_query_time |
+-------------------+
|         10.000000 |
+-------------------+

Pode ser definida como uma variável GLOBAL, no arquivo my.cnf ou my.ini. Ou pode ser definido pela conexão, embora isso seja incomum. O valor pode ser definido entre 0 a 10 (segundos). Qual valor usar?

  • 10 é tão alto que é quase inútil;
  • 2 é um compromisso;
  • 0,5 e outras frações são possíveis;
  • 0 captura tudo; isso pode encher o disco perigosamente rápido, mas pode ser muito útil.

A captura de consultas lentas é ativada ou desativada. E o arquivo registrado também é especificado. O abaixo captura esses conceitos:

SELECT @@slow_query_log; -- Is capture currently active? (1=On, 0=Off)
SELECT @@slow_query_log_file; -- filename for capture. Resides in datadir
SELECT @@datadir; -- to see current value of the location for capture file

SET GLOBAL slow_query_log=0; -- Turn Off
-- make a backup of the Slow Query Log capture file. Then delete it.
SET GLOBAL slow_query_log=1; -- Turn it back On (new empty file is created)

Para obter mais informações, consulte a página do manual do MySQL The Slow Query Log

Nota: As informações acima sobre ligar/desligar o slowlog foram alteradas em 5.6(?); versão mais antiga tinha outro mecanismo.

A “melhor” maneira de ver o que está deixando seu sistema lento:

long_query_time=...
turn on the slowlog
run for a few hours
turn off the slowlog (or raise the cutoff)
run pt-query-digest to find the 'worst' couple of queries.  Or mysqldumpslow -s t

Uma lista #

  • Log geral - todas as consultas - veja VARIABLE general_log
  • Log lento - consultas mais lentas que long_query_time - slow_query_log_file
  • Binlog - para replicação e backup - log_bin_basename
  • Log de retransmissão - também para replicação
  • erros gerais - mysqld.err
  • start/stop - mysql.log (não muito interessante) - log_error
  • Redo log do InnoDB - iblog*

Veja as variáveis ​​basedir e datadir para localização padrão para muitos logs

Alguns logs são ativados/desativados por outras VARIÁVEIS. Alguns são gravados em um arquivo ou em uma tabela.

(Nota aos revisores: isso precisa de mais detalhes e mais explicações.)

Documenters: inclua o local e o nome padrão para cada tipo de log, tanto para Windows quanto para *nix. (Ou pelo menos o quanto você puder.)

Log de consulta geral #

O Log de Consulta Geral contém uma lista de informações gerais de conexões, desconexões e consultas do cliente. É inestimável para depuração, mas representa um obstáculo ao desempenho (citação?).

Um exemplo de visualização de um Log de consulta geral é visto abaixo:

Para determinar se o Log geral está sendo capturado no momento:

SELECT @@general_log; -- 1 = Capture is active; 0 = It is not.

Para determinar o nome do arquivo de captura:

SELECT @@general_log_file; -- Full path to capture file

Se o caminho completo para o arquivo não for mostrado, o arquivo existe no datadir.

Exemplo do Windows:

+----------------------------------------------------------+
| @@general_log_file                                       |
+----------------------------------------------------------+
| C:\ProgramData\MySQL\MySQL Server 5.7\Data\GuySmiley.log |
+----------------------------------------------------------+

Linux:

+-----------------------------------+
| @@general_log_file                |
+-----------------------------------+
| /var/lib/mysql/ip-ww-xx-yy-zz.log |
+-----------------------------------+

Quando alterações são feitas na variável GLOBAL general_log_file, o novo log é salvo no datadir. No entanto, o caminho completo pode não ser mais refletido examinando a variável.

No caso de não haver entrada para general_log_file no arquivo de configuração, o padrão será @@hostname.log no datadir.

As melhores práticas são DESLIGAR a captura. Salve o arquivo de log em um diretório de backup com um nome de arquivo que reflita a data e hora de início/término da captura. Excluindo o arquivo anterior se um movimento do sistema de arquivos não ocorreu desse arquivo. Estabeleça um novo nome de arquivo para o arquivo de log e ative a captura (todos mostrados abaixo). As melhores práticas também incluem uma determinação cuidadosa se você deseja capturar no momento. Normalmente, a captura está ATIVADA apenas para fins de depuração.

Um nome de arquivo de sistema de arquivos típico para um log de backup pode ser:

/LogBackup/GeneralLog_20160802_1520_to_20160802_1815.log

onde a data e a hora fazem parte do nome do arquivo como um intervalo.

Para Windows, observe a sequência a seguir com as alterações de configuração.

SELECT @@general_log; -- 0. Not being captured
SELECT @@general_log_file; -- C:\ProgramData\MySQL\MySQL Server 5.6\Data\GuySmiley.log
SELECT @@datadir; -- C:\ProgramData\MySQL\MySQL Server 5.7\Data\
SET GLOBAL general_log_file='GeneralLogBegin_20160803_1420.log'; -- datetime clue
SET GLOBAL general_log=1; -- Turns on actual log capture. File is created under `datadir`
SET GLOBAL general_log=0; -- Turn logging off

Linux é semelhante. Isso representaria mudanças dinâmicas. Qualquer reinicialização do servidor pegaria as configurações do arquivo de configuração.

Quanto ao arquivo de configuração, considere as seguintes configurações de variáveis ​​relevantes:

[mysqld]
general_log_file = /path/to/currentquery.log
general_log      = 1

Além disso, a variável log_output pode ser configurada para saída TABLE, não apenas FILE. Para isso, consulte Destinos.

Por favor, veja a Página do Manual do MySQL The General Query Log.

Registro de erros #

O Log de Erros é preenchido com informações de início e parada e eventos críticos encontrados pelo servidor.

Segue um exemplo de seu conteúdo:

A variável log_error contém o caminho para o arquivo de log para registro de erros.

Na ausência de uma entrada de arquivo de configuração para log_error, o sistema irá padronizar seus valores para @@hostname.err no datadir. Observe que log_error não é uma variável dinâmica. Como tal, as alterações são feitas por meio de alterações no arquivo cnf ou ini e uma reinicialização do servidor (ou vendo “Descarregando e renomeando o arquivo de log de erros” no link da página de manual na parte inferior aqui).

O log não pode ser desabilitado para erros. Eles são importantes para a integridade do sistema ao solucionar problemas. Além disso, as entradas não são frequentes em comparação com o Log de Consulta Geral.

A variável GLOBAL log_warnings define o nível de detalhamento que varia de acordo com a versão do servidor. O trecho a seguir ilustra:

SELECT @@log_warnings; -- make a note of your prior setting
SET GLOBAL log_warnings=2; -- setting above 1 increases output (see server version)

log_warnings como visto acima é uma variável dinâmica.

As alterações do arquivo de configuração nos arquivos cnf e ini podem ter a seguinte aparência.

[mysqld]
log_error        = /path/to/CurrentError.log
log_warnings     = 2

O MySQL 5.7.2 expandiu a verbosidade do nível de aviso para 3 e adicionou o GLOBAL log_error_verbosity. Novamente, foi introduzido em 5.7.2. Ele pode ser definido dinamicamente e verificado como uma variável ou definido através das configurações do arquivo de configuração cnf ou ini.

A partir do MySQL 5.7.2:

[mysqld]
log_error           = /path/to/CurrentError.log
log_warnings        = 2
log_error_verbosity = 3

Por favor, veja a página do manual do MySQL intitulada The Error Log especialmente para liberar e renomear o arquivo de log de erros, e sua seção Error Log Verbosity com versões relacionadas a log_warnings e error_log_verbosity.
