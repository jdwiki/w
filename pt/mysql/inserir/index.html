Sintaxe #

  1. INSERIR [LOW_PRIORITY | ATRASO | HIGH_PRIORITY] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] {VALUES | VALUE} ({expr | DEFAULT},…),(…),… [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  2. INSERIR [LOW_PRIORITY | ATRASO | HIGH_PRIORITY] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] SET col_name={expr | DEFAULT}, … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  3. INSERIR [LOW_PRIORITY | HIGH_PRIORITY] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr [, col_name=expr] … ]

  4. Uma expressão expr pode se referir a qualquer coluna que foi definida anteriormente em uma lista de valores. Por exemplo, você pode fazer isso porque o valor de col2 se refere a col1, que foi atribuído anteriormente: INSERT INTO nome_tabela (coluna1,coluna2) VALUES(15,coluna1*2);

  5. As instruções INSERT que usam a sintaxe VALUES podem inserir várias linhas. Para fazer isso, inclua várias listas de valores de coluna, cada uma entre parênteses e separadas por vírgulas. Exemplo: INSERT INTO nome_tabela (a,b,c) VALUES(1,2,3),(4,5,6),(7,8,9);

  6. A lista de valores para cada linha deve ser colocada entre parênteses. A instrução a seguir é ilegal porque o número de valores na lista não corresponde ao número de nomes de coluna: INSERT INTO nome_tabela (a,b,c) VALUES(1,2,3,4,5,6,7,8,9);

  7. INSERIR … SELECIONAR Sintaxe INSERIR [LOW_PRIORITY | HIGH_PRIORITY] [IGNORAR] [INTO] tbl_name [PARTITION (partition_name,…)] [(col_name,…)] SELECT … [ ON DUPLICATE KEY UPDATE col_name=expr, … ]

  8. Com INSERT … SELECT, você pode inserir rapidamente várias linhas em uma tabela de uma ou várias tabelas. Por exemplo: INSERT INTO tbl_temp2 (fld_id) SELECIONE tbl_temp1.fld_order_id FROM tbl_temp1 WHERE tbl_temp1.fld_order_id > 100;

Sintaxe oficial INSERT

INSERIR, NA ATUALIZAÇÃO DE CHAVE DUPLICADA #

INSERT INTO `table_name` 
    (`index_field`, `other_field_1`, `other_field_2`) 
    VALUES 
    ('index_value', 'insert_value', 'other_value') 
    ON DUPLICATE KEY UPDATE 
        `other_field_1` = 'update_value',
        `other_field_2` = VALUES(`other_field_2`);

Isto irá INSERIR em table_name os valores especificados, mas se a chave exclusiva já existir, ela irá atualizar o other_field_1 para ter um novo valor. Às vezes, ao atualizar uma chave duplicada, é útil usar VALUES() para acessar o valor original que foi passado para o INSERT em vez de definir o valor diretamente. Desta forma, você pode definir valores diferentes usando INSERT e UPDATE. Veja o exemplo acima onde other_field_1 é definido como insert_value em INSERT ou update_value em UPDATE enquanto other_field_2 é sempre definido como other_value.

Crucial para que o Insert on Duplicate Key Update (IODKU) funcione é o esquema contendo uma chave exclusiva que sinalizará um conflito duplicado. Essa chave exclusiva pode ser uma Chave Primária ou não. Pode ser uma chave exclusiva em uma única coluna ou uma multicoluna (chave composta).

Inserindo várias linhas #

INSERT INTO `my_table` (`field_1`, `field_2`) VALUES 
    ('data_1', 'data_2'),
    ('data_1', 'data_3'),
    ('data_4', 'data_5');

Esta é uma maneira fácil de adicionar várias linhas de uma vez com uma instrução INSERT.

Esse tipo de inserção ’lote’ é muito mais rápido do que inserir linhas uma a uma. Normalmente, inserir 100 linhas em uma única inserção de lote dessa maneira é 10 vezes mais rápido do que inseri-las individualmente.

Ignorando linhas existentes #

Ao importar grandes conjuntos de dados, pode ser preferível, em determinadas circunstâncias, pular linhas que normalmente fariam com que a consulta falhasse devido a uma restrição de coluna, por exemplo, chaves primárias duplicadas. Isso pode ser feito usando INSERT IGNORE.

Considere o seguinte exemplo de banco de dados:

SELECT * FROM `people`;
--- Produces:
+----+------+
| id | name |
+----+------+
|  1 | john |
|  2 | anna |
+----+------+


INSERT IGNORE INTO `people` (`id`, `name`) VALUES
    ('2', 'anna'), --- Without the IGNORE keyword, this record would produce an error
    ('3', 'mike');

SELECT * FROM `people`;
--- Produces:
+----+--------+
| id |  name  |
+----+--------+
|  1 |  john  |
|  2 |  anna  |
|  3 |  mike  |
+----+--------+

O importante a lembrar é que INSERT IGNORE também ignorará outros erros silenciosamente, aqui está o que as documentações oficiais do Mysql dizem:

Conversões de dados que acionariam erros cancelam a instrução se IGNORE não for > especificado. Com IGNORE, os valores inválidos são ajustados para os valores mais próximos e >inseridos; avisos são produzidos, mas a instrução não é abortada.

Observação:- A seção abaixo é adicionada para fins de completude, mas não é considerada uma prática recomendada (isso falharia, por exemplo, se outra coluna fosse adicionada à tabela).

Se você especificar o valor da coluna correspondente para todas as colunas da tabela, poderá ignorar a lista de colunas na instrução INSERT da seguinte forma:

INSERT INTO `my_table` VALUES 
        ('data_1', 'data_2'),
        ('data_1', 'data_3'),
        ('data_4', 'data_5');

Inserção básica #

INSERT INTO `table_name` (`field_one`, `field_two`) VALUES ('value_one', 'value_two');

Neste exemplo trivial, table_name é onde os dados devem ser adicionados, field_one e field_two são campos para definir os dados, e value_one e value_two são os dados a serem feitos em relação a field_one e field_two respectivamente.

É uma boa prática listar os campos nos quais você está inserindo dados em seu código, pois se a tabela for alterada e novas colunas forem adicionadas, sua inserção será interrompida caso elas não estejam lá

INSERT com AUTO_INCREMENT + LAST_INSERT_ID() #

Quando uma tabela tem um AUTO_INCREMENT PRIMARY KEY, normalmente não se insere nessa coluna. Em vez disso, especifique todas as outras colunas e pergunte qual era o novo id.

CREATE TABLE t (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    this ...,
    that ...,
    PRIMARY KEY(id) );

INSERT INTO t (this, that) VALUES (..., ...);
SELECT LAST_INSERT_ID() INTO @id;
INSERT INTO another_table (..., t_id, ...) VALUES (..., @id, ...);

Observe que LAST_INSERT_ID() está vinculado à sessão, portanto, mesmo que várias conexões sejam inseridas na mesma tabela, cada uma terá seu próprio id.

A API do seu cliente provavelmente tem uma maneira alternativa de obter o LAST_INSERT_ID() sem realmente executar um SELECT e devolver o valor ao cliente em vez de deixá-lo em um @variable dentro do MySQL. Tal é geralmente preferível.

Exemplo mais longo e detalhado

O uso “normal” do IODKU é acionar “chave duplicada” com base em alguma chave UNIQUE, não na AUTO_INCREMENT PRIMARY KEY. O seguinte demonstra tal. Observe que ele não fornece o id no INSERT.

Configuração para exemplos a seguir:

CREATE TABLE iodku (
    id INT AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    misc INT NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
) ENGINE=InnoDB;

INSERT INTO iodku (name, misc)
    VALUES
    ('Leslie', 123),
    ('Sally', 456);
Query OK, 2 rows affected (0.00 sec)
Records: 2  Duplicates: 0  Warnings: 0
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  |  456 |
+----+--------+------+

O caso de IODKU realizando uma “atualização” e LAST_INSERT_ID() recuperando o id relevante:

INSERT INTO iodku (name, misc)
    VALUES
    ('Sally', 3333)            -- should update
    ON DUPLICATE KEY UPDATE    -- `name` will trigger "duplicate key"
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();       -- picking up existing value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                2 |
+------------------+

O caso em que IODKU executa uma “inserção” e LAST_INSERT_ID() recupera o novo id:

INSERT INTO iodku (name, misc)
    VALUES
    ('Dana', 789)          -- Should insert
    ON DUPLICATE KEY UPDATE
    id = LAST_INSERT_ID(id),
    misc = VALUES(misc);
SELECT LAST_INSERT_ID();   -- picking up new value
+------------------+
| LAST_INSERT_ID() |
+------------------+
|                3 |
+------------------+

Conteúdo da tabela resultante:

SELECT * FROM iodku;
+----+--------+------+
| id | name   | misc |
+----+--------+------+
|  1 | Leslie |  123 |
|  2 | Sally  | 3333 |  -- IODKU changed this
|  3 | Dana   |  789 |  -- IODKU added this
+----+--------+------+

INSERT SELECT (Inserindo dados de outra Tabela) #

Esta é a maneira básica de inserir dados de outra tabela com a instrução SELECT.

INSERT INTO `tableA` (`field_one`, `field_two`) 
   SELECT `tableB`.`field_one`, `tableB`.`field_two` 
   FROM `tableB` 
   WHERE `tableB`.clmn <> 'someValue'
   ORDER BY `tableB`.`sorting_clmn`;

Você pode SELECT * FROM, mas então tableA e tableB devem ter uma contagem de colunas e tipos de dados correspondentes.

Colunas com AUTO_INCREMENT são tratadas como na cláusula INSERT com VALUES.

Essa sintaxe facilita o preenchimento (temporário) de tabelas com dados de outras tabelas, ainda mais quando os dados devem ser filtrados na inserção.

IDs AUTO_INCREMENT perdidos #

Várias funções de ‘inserção’ podem “gravar” ids. Aqui está um exemplo, usando InnoDB (outros Engines podem funcionar de forma diferente):

CREATE TABLE Burn (
    id SMALLINT UNSIGNED AUTO_INCREMENT NOT NULL,
    name VARCHAR(99) NOT NULL,
    PRIMARY KEY(id),
    UNIQUE(name)
        ) ENGINE=InnoDB;

INSERT IGNORE INTO Burn (name) VALUES ('first'), ('second');
SELECT LAST_INSERT_ID();          -- 1
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('second');  -- dup 'IGNOREd', but id=3 is burned
SELECT LAST_INSERT_ID();          -- Still "1" -- can't trust in this situation
SELECT * FROM Burn ORDER BY id;
  +----+--------+
  |  1 | first  |
  |  2 | second |
  +----+--------+

INSERT IGNORE INTO Burn (name) VALUES ('third');
SELECT LAST_INSERT_ID();           -- now "4"
SELECT * FROM Burn ORDER BY id;    -- note that id=3 was skipped over
  +----+--------+
  |  1 | first  |
  |  2 | second |
  |  4 | third  |    -- notice that id=3 has been 'burned'
  +----+--------+

Pense nisso (aproximadamente) desta forma: primeiro a inserção olha para ver quantas linhas poderiam ser inseridas. Em seguida, pegue tantos valores do auto_increment para essa tabela. Por fim, insira as linhas, usando ids conforme necessário e queimando as sobras.

A única vez que as sobras são recuperáveis ​​é se o sistema for desligado e reiniciado. Na reinicialização, efetivamente MAX(id) é executado. Isso pode reutilizar ids que foram queimados ou que foram liberados por DELETEs dos ids mais altos.

Essencialmente, qualquer sabor de INSERT (incluindo REPLACE, que é DELETE + INSERT) pode gravar ids. No InnoDB, a variável global (não de sessão!) [innodb_autoinc_lock_mode] 1 pode ser usada para controlar parte do que está acontecendo.

Ao “normalizar” strings longas em um ID de INCREMENTO AUTOMÁTICO, a gravação pode acontecer facilmente. Isso poderia levar ao estouro do tamanho do INT que você escolheu.
