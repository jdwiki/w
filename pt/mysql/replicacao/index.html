A replicação é usada para copiar dados de [Backup] de um servidor de banco de dados MySQL para um ou mais servidores de banco de dados MySQL.

Master – O servidor de banco de dados MySQL, que está servindo os dados a serem copiados

Slave – O servidor de banco de dados MySQL, copia dados que são servidos pelo Master

Com o MySQL, a replicação é assíncrona por padrão. Isso significa que os escravos não precisam estar conectados permanentemente para receber atualizações do mestre. Por exemplo, se o seu escravo estiver DESLIGADO ou não estiver conectado com o mestre e você estiver LIGANDO o escravo ou se conectar com o Mestre posteriormente, ele será sincronizado automaticamente com o Mestre.

Dependendo da configuração, você pode replicar todos os bancos de dados, bancos de dados selecionados ou até mesmo tabelas selecionadas em um banco de dados.

Formatos de replicação

Existem dois tipos principais de formatos de replicação

Replicação Baseada em Instruções (SBR) – que replica instruções SQL inteiras. Neste, o mestre grava instruções SQL no log binário. A replicação do mestre para o escravo funciona executando essas instruções SQL no escravo.

Row Based Replication (RBR) – que replica apenas as linhas alteradas. Neste, o mestre grava eventos no log binário que indicam como as linhas individuais da tabela são alteradas. A replicação do mestre para o escravo funciona copiando os eventos que representam as alterações nas linhas da tabela para o escravo.

Você também pode usar uma terceira variedade, Mixed Based Replication (MBR). Neste, tanto o log baseado em instrução quanto o baseado em linha são usados. O log será criado dependendo do que for mais apropriado para a alteração.

O formato baseado em instrução era o padrão nas versões do MySQL anteriores a 5.7.7. No MySQL 5.7.7 e posterior, o formato baseado em linha é o padrão.

Configuração de replicação mestre - escravo #

Considere 2 servidores MySQL para configuração de replicação, um é um mestre e o outro é um escravo.

Vamos configurar o Master para que ele mantenha um registro de todas as ações realizadas nele. Vamos configurar o servidor Slave para que ele veja o log no Master e sempre que ocorrer alguma alteração no log do Master, ele deverá fazer a mesma coisa.

Configuração Mestre

Antes de tudo, precisamos criar um usuário no Master. Este usuário será utilizado pelo Slave para criar uma conexão com o Master.

CREATE USER 'user_name'@'%' IDENTIFIED BY 'user_password';
GRANT REPLICATION SLAVE ON *.* TO 'user_name'@'%';
FLUSH PRIVILEGES;

Altere user_name e user_password de acordo com seu nome de usuário e senha.

Agora o arquivo my.inf (my.cnf no Linux) deve ser editado. Inclua as seguintes linhas na seção [mysqld].

server-id = 1
log-bin = mysql-bin.log
binlog-do-db = your_database

A primeira linha é usada para atribuir um ID a este servidor MySQL.

A segunda linha diz ao MySQL para começar a escrever um log no arquivo de log especificado. No Linux isso pode ser configurado como log-bin = /home/mysql/logs/mysql-bin.log. Se você estiver iniciando a replicação em um servidor MySQL no qual a replicação já foi usada, certifique-se de que este diretório esteja vazio de todos os logs de replicação.

A terceira linha é usada para configurar o banco de dados para o qual vamos escrever o log. Você deve substituir your_database pelo nome do seu banco de dados.

Certifique-se de que skip-networking não foi habilitado e reinicie o servidor MySQL (Master)

Configuração do escravo

O arquivo my.inf deve ser editado no Slave também. Inclua as seguintes linhas na seção [mysqld].

server-id = 2
master-host = master_ip_address
master-connect-retry = 60

master-user = user_name
master-password = user_password
replicate-do-db = your_database

relay-log = slave-relay.log
relay-log-index = slave-relay-log.index

A primeira linha é usada para atribuir um ID a este servidor MySQL. Esse ID deve ser exclusivo.

A segunda linha é o endereço I.P do servidor Master. Altere isso de acordo com o I.P do seu sistema Master.

A terceira linha é usada para definir um limite de repetição em segundos.

As próximas duas linhas informam o nome de usuário e senha para o Escravo, através dos quais ele conecta o Mestre.

A próxima linha define o banco de dados que ele precisa replicar.

As duas últimas linhas usadas para atribuir nomes de arquivo relay-log e relay-log-index.

Certifique-se de que skip-networking não foi habilitado e reinicie o servidor MySQL (Slave)

Copiar dados para o escravo

Se os dados estiverem sendo adicionados constantemente ao Master, teremos que impedir todo o acesso ao banco de dados no Master para que nada possa ser adicionado. Isso pode ser feito executando a seguinte instrução em Master.

FLUSH TABLES WITH READ LOCK;

Se nenhum dado estiver sendo adicionado ao servidor, você pode pular a etapa acima.

Vamos fazer backup de dados do Master usando mysqldump

mysqldump your_database -u root -p > D://Backup/backup.sql;

Altere your_database e ​​diretório de backup de acordo com sua configuração. Você terá agora um arquivo chamado backup.sql no local fornecido.

Se seu banco de dados não existir em seu Slave, crie-o executando o seguinte

CREATE DATABASE `your_database`;

Agora temos que importar o backup para o servidor Slave MySQL.

mysql -u root -p your_database  <D://Backup/backup.sql
--->Change `your_database` and backup directory according to your setup

Iniciar Replicação

Para iniciar a replicação, precisamos encontrar o nome do arquivo de log e a posição do log no Master. Então, execute o seguinte no Master

SHOW MASTER STATUS;

Isso lhe dará uma saída como abaixo

+---------------------+----------+-------------------------------+------------------+
| File                | Position | Binlog_Do_DB                  | Binlog_Ignore_DB |
+---------------------+----------+-------------------------------+------------------+
| mysql-bin.000001    | 130      | your_database                 |                  |
+---------------------+----------+-------------------------------+------------------+

Em seguida, execute o seguinte no Slave

SLAVE STOP;
CHANGE MASTER TO MASTER_HOST='master_ip_address', MASTER_USER='user_name', 
   MASTER_PASSWORD='user_password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS=130;
SLAVE START;

Primeiro paramos o Escravo. Em seguida, informamos exatamente onde procurar no arquivo de log mestre. Para o nome MASTER_LOG_FILE e MASTER_LOG_POS, use os valores que obtivemos executando o comando SHOW MASTER STATUS no Master.

Você deve alterar o I.P do Master em MASTER_HOST e alterar o usuário e a senha de acordo.

O Escravo agora estará esperando. O status do Slave pode ser visualizado executando o seguinte

SHOW SLAVE STATUS;

Se você executou anteriormente FLUSH TABLES WITH READ LOCK no Master, libere as tabelas do bloqueio executando o seguinte

UNLOCK TABLES;

Agora o Master mantém um log para cada ação realizada nele e o servidor Slave olha o log no Master. Sempre que ocorrem alterações no log do Master, o Slave replica isso.

Erros de replicação #

Sempre que houver um erro durante a execução de uma consulta no escravo, o MySQL interrompe a replicação automaticamente para identificar o problema e corrigi-lo. Isso principalmente porque um evento causou uma chave duplicada ou uma linha não foi encontrada e não pode ser atualizada ou excluída. Você pode pular esses erros, mesmo que isso não seja recomendado

Para pular apenas uma consulta que está pendurando o escravo, use a seguinte sintaxe

SET GLOBAL sql_slave_skip_counter = N;

Esta instrução ignora os próximos N eventos do mestre. Esta instrução é válida apenas quando os threads escravos não estão em execução. Caso contrário, produz um erro.

STOP SLAVE;
SET GLOBAL sql_slave_skip_counter=1;
START SLAVE;

Em alguns casos isso é bom. Mas se a instrução for parte de uma transação com várias instruções, ela se tornará mais complexa, porque pular a instrução que produz o erro fará com que toda a transação seja ignorada.

Se você quiser pular mais consultas que produzem o mesmo código de erro e se tiver certeza de que pular esses erros não tornará seu escravo inconsistente e quiser pular todas elas, você adicionaria uma linha para pular esse código de erro em seu arquivo my. cnf.

Por exemplo, você pode querer pular todos os erros duplicados que pode estar recebendo

1062 | Error 'Duplicate entry 'xyz' for key 1' on query

Em seguida, adicione o seguinte ao seu my.cnf

slave-skip-errors = 1062

Você também pode pular outros tipos de erros ou todos os códigos de erro, mas certifique-se de que pular esses erros não tornará seu escravo inconsistente. A seguir estão a sintaxe e os exemplos

slave-skip-errors=[err_code1,err_code2,...|all]

slave-skip-errors=1062,1053
slave-skip-errors=all
slave-skip-errors=ddl_exist_errors
