Conjuração implícita/automática #

VARCHAR(255) – ou não #

Lentura máxima sugerida

Primeiro, mencionarei algumas strings comuns que são sempre hexadecimais ou limitadas a ASCII. Para estes, você deve especificar CHARACTER SET ascii (latin1 está ok) para que não perca espaço:

UUID CHAR(36) CHARACTER SET ascii -- or pack into BINARY(16)
country_code CHAR(2) CHARACTER SET ascii
ip_address CHAR(39) CHARACTER SET ascii -- or pack into BINARY(16)
phone VARCHAR(20) CHARACTER SET ascii -- probably enough to handle extension
postal_code VARCHAR(20) CHARACTER SET ascii -- (not 'zip_code') (don't know the max

city VARCHAR(100) -- This Russian town needs 91:
    Poselok Uchebnogo Khozyaystva Srednego Professionalno-Tekhnicheskoye Uchilishche Nomer Odin
country VARCHAR(50) -- probably enough
name VARCHAR(64) -- probably adequate; more than some government agencies allow

Por que não simplesmente 255? Há duas razões para evitar a prática comum de usar (255) para tudo.

  • Quando um SELECT complexo precisa criar uma tabela temporária (para uma subconsulta, UNION, GROUP BY, etc), a escolha preferida é usar o mecanismo MEMORY, que coloca os dados na RAM. Mas VARCHARs são transformados em CHAR no processo. Isso faz com que VARCHAR(255) CHARACTER SET utf8mb4 leve 1020 bytes. Isso pode levar à necessidade de derramar no disco, que é mais lento.
  • Em certas situações, o InnoDB irá olhar para o tamanho potencial das colunas em uma tabela e decidir que será muito grande, abortando um CREATE TABLE.

VARCHAR versus TEXT

Dicas de uso para *TEXT, CHAR e VARCHAR, além de algumas práticas recomendadas:

  • Nunca use TINYTEXT.
  • Quase nunca use CHAR – é de tamanho fixo; cada caractere é o comprimento máximo do CHARACTER SET (por exemplo, 4 bytes/caractere para utf8mb4).
  • Com CHAR, use CHARACTER SET ascii a menos que você saiba o contrário.
  • VARCHAR(n) irá truncar em n caracteres; TEXT irá truncar em algum número de bytes. (Mas, você quer truncamento?)
  • *TEXT may torna mais lento os SELECTs complexos devido à forma como as tabelas temporárias são tratadas.

INT como AUTO_INCREMENT #

Qualquer tamanho de INT pode ser usado para AUTO_INCREMENT. UNSIGNED é sempre apropriado.

Tenha em mente que certas operações “gravam” ids AUTO_INCREMENT. Isso pode levar a uma lacuna inesperada. Exemplos: INSERT IGNORE e REPLACE. Eles podem pré-alocar um id antes percebendo que não será necessário. Esse é o comportamento esperado e por design no mecanismo InnoDB e não deve desencorajar seu uso.

Tipos de ponto fixo #

Os tipos DECIMAL e NUMERIC do MySQL armazenam valores de dados numéricos exatos. Recomenda-se usar esses tipos para preservar a precisão exata, como para dinheiro.

Decimal #

Esses valores são armazenados em formato binário. Em uma declaração de coluna, a precisão e a escala devem ser especificadas

Precisão representa o número de dígitos significativos que são armazenados para valores.

A escala representa o número de dígitos armazenados depois do decimal

salary DECIMAL(5,2)

5 representa a precisão e 2 representa a escala. Para este exemplo, o intervalo de valores que podem ser armazenados nesta coluna é -999,99 a 999,99

Se o parâmetro de escala for omitido, o padrão é 0

Este tipo de dados pode armazenar até 65 dígitos.

O número de bytes ocupados por DECIMAL(M,N) é aproximadamente M/2.

Tipos de ponto flutuante #

FLOAT e DOUBLE representam tipos de dados aproximados.

Tipo Armazenamento Precisão Gama
FLUTUAR 4 bytes 23 bits significativos / ~7 dígitos decimais 10^+/-38
DUPLO 8 bytes 53 bits significativos / ~16 dígitos decimais 10^+/-308

REAL é sinônimo de FLOAT. DOUBLE PRECISION é sinônimo de DOUBLE.

Embora o MySQL também permita o qualificador (M,D), não o use. (M,D) significa que os valores podem ser armazenados com até M dígitos totais, onde D pode estar após o decimal. Números serão arredondados duas vezes ou truncados; isso causará mais problemas do que benefícios.

Como os valores de ponto flutuante são aproximados e não armazenados como valores exatos, as tentativas de tratá-los como exatos em comparações podem causar problemas. Observe em particular que um valor FLOAT raramente é igual a um valor DOUBLE.

Tipo de valor de bits #

O tipo BIT é útil para armazenar valores de campos de bits. BIT(M) permite o armazenamento de valores de até M bits onde M está no intervalo de 1 a 64

Você também pode especificar valores com a notação bit value.

b'111'      -> 7
b'10000000' -> 128

Às vezes é útil usar ‘shift’ para construir um valor de bit único, por exemplo (1 << 7) para 128.

O tamanho máximo combinado de todas as colunas BIT em uma tabela NDB é 4096.

CHAR(n) #

CHAR(n) é uma string de comprimento fixed de n characters. Se for CHARACTER SET utf8mb4, significa que ocupa exatamente 4*n bytes, independentemente do texto que está nele.

A maioria dos casos de uso para CHAR(n) envolve strings que contêm caracteres em inglês, portanto, deve ser CHARACTER SET ascii. (latin1 fará o mesmo.)

country_code CHAR(2) CHARACTER SET ascii,
postal_code  CHAR(6) CHARACTER SET ascii,
uuid    CHAR(39) CHARACTER SET ascii,  -- more discussion elsewhere

DATA, DATETIME, TIMESTAMP, YEAR e TIME #

O tipo de dados DATE compreende a data, mas não o componente de hora. Seu formato é 'AAAA-MM-DD' com um intervalo de ‘1000-01-01’ a ‘9999-12-31’.

O tipo DATETIME inclui a hora com um formato de ‘AAAA-MM-DD HH:MM:SS’. Tem um intervalo de ‘1000-01-01 00:00:00’ a ‘9999-12-31 23:59:59’.

O tipo TIMESTAMP é um tipo inteiro que compreende data e hora com um intervalo efetivo de ‘1970-01-01 00:00:01’ UTC a ‘2038-01-19 03:14:07’ UTC.

O tipo YEAR representa um ano e varia de 1901 a 2155.

O tipo TIME representa uma hora com formato ‘HH:MM:SS’ e contém um intervalo de ‘-838:59:59’ a ‘838:59:59’.

Requisitos de armazenamento:

|-----------|--------------------|----------------------------------------|
| Data Type | Before MySQL 5.6.4 | as of MySQL 5.6.4                      |
|-----------|--------------------|----------------------------------------|
| YEAR      |      1 byte        |  1 byte                                |
| DATE      |      3 bytes       |  3 bytes                               |
| TIME      |      3 bytes       |  3 bytes + fractional seconds storage  |
| DATETIME  |      8 bytes       |  5 bytes + fractional seconds storage  |
| TIMESTAMP |      4 bytes       |  4 bytes + fractional seconds storage  |
|-----------|--------------------|----------------------------------------|

Segundos fracionários (a partir da versão 5.6.4):

|------------------------------|------------------|
| Fractional Seconds Precision | Storage Required |
|------------------------------|------------------|
|              0               |      0 bytes     |
|              1,2             |      1 byte      |
|              3,4             |      2 byte      |
|              5,6             |      3 byte      |
|------------------------------|------------------|
