Usando uma estrutura de registro de erros personalizada #

Às vezes, é útil integrar uma estrutura de log de erros personalizada para garantir que todas as exceções sejam registradas.

[ServiceContract]
[ErrorHandler]
public interface IMyService
{
       
}

[AttributeUsage(AttributeTargets.Interface)]
public class CustomErrorHandler : Attribute, IContractBehavior, IErrorHandler
{
    public bool HandleError(Exception error)
    {
        return false;
    }

    public void ProvideFault(Exception error, MessageVersion version, ref Message fault)
    {
        if (error == null)
        {
            return;
        }

        //my custom logging framework
    }

    public void ApplyDispatchBehavior(ContractDescription contractDescription, ServiceEndpoint endpoint, DispatchRuntime dispatchRuntime)
    {
        dispatchRuntime.ChannelDispatcher.ErrorHandlers.Add(this);
    }

    public void ApplyClientBehavior(ContractDescription contractDescription, ServiceEndpoint endpoint,
        ClientRuntime clientRuntime)
    {

    }

    public void AddBindingParameters(ContractDescription contractDescription, ServiceEndpoint endpoint,
        BindingParameterCollection bindingParameters)
    {

    }

    public void Validate(ContractDescription contractDescription, ServiceEndpoint endpoint)
    {

    }
}

Mostrando informações adicionais quando ocorre uma exceção #

É importante lidar com exceções em seu serviço. Ao desenvolver o serviço, você pode configurar o WCF para fornecer informações mais detalhadas, adicionando esta tag ao arquivo de configuração, geralmente Web.config:

 <serviceDebug includeExceptionDetailInFaults="true"/>

Essa tag deve ser colocada dentro da tag serviceBehavior, geralmente assim:

<system.serviceModel>
    <behaviors>
      <serviceBehaviors>
        <behavior>
          <serviceDebug includeExceptionDetailInFaults="true"/>
        </behavior>
      </serviceBehaviors>
    </behaviors>
</system.serviceModel>

Exemplo de informações detalhadas:

Rastreamento de pilha do servidor: em System.ServiceModel.Channels.ServiceChannel.HandleReply(ProxyOperationRuntime operação, ProxyRpc& rpc) em System.ServiceModel.Channels.ServiceChannel.Call(String ação, Boolean oneway, operação ProxyOperationRuntime, Object[] ins, Object[] outs, TimeSpan timeout) em System.ServiceModel.Channels.ServiceChannelProxy.InvokeService(IMethodCallMessage methodCall, operação ProxyOperationRuntime) em System.ServiceModel.Channels.ServiceChannelProxy.Invoke(IMessage mensagem)

Exceção relançada em [0]: em System.Runtime.Remoting.Proxies.RealProxy.HandleReturnMessage(IMessage reqMsg, IMessage retMsg) em System.Runtime.Remoting.Proxies.RealProxy.PrivateInvoke(MessageData& msgData, tipo Int32) em IMyService.GetDataOperation(RequestObterBeneficiario solicitar) em MyServiceClient.GetDataOpration(RequestData pedido)

Isso retornará ao cliente informações detalhadas. Durante o desenvolvimento isso pode ajudar, mas quando seu serviço entrar em produção, você não o manterá mais porque seu serviço pode enviar dados confidenciais, como o nome ou a configuração do banco de dados.

Lançando uma mensagem amigável com FaultException #

Você pode lidar com exceções e lançar uma mensagem mais amigável como ‘Serviço indisponível’ lançando um FaultException:

try
{
    // your service logic here
}
catch (Exception ex)
{
    // You can do something here, like log the original exception
    throw new FaultException("There was a problem processing your request");
}

No seu cliente, você pode facilmente receber a mensagem:

try
{
    // call the service
}
catch (FaultException faultEx)
{
   var errorMessage = faultEx.Message;
   // do something with error message
}

Você pode distinguir a exceção tratada de outras exceções, como um erro de rede, adicionando outras capturas ao seu código:

try
{
    // call the service
}
catch (FaultException faultEx)
{
   var errorMessage = faultEx.Message;
   // do something with error message
}
catch (CommunicationException commProblem)
{
    // Handle the communication error, like trying another endpoint service or logging
}

Lançando uma FaultException com um código de falha #

A FaultException também pode incluir um FaultCode, que é uma string de dados que você pode usar para passar informações adicionais, para que o cliente possa distinguir diferentes exceções:

try
{
    // your service logic here
}
catch (Exception ex)
{
   throw new FaultException("There was a problem processing your request",
      new FaultCode(("01"));
}

Obtendo o código de falha:

try
{
    // call the service
}
catch (FaultException faultEx)
{
   switch (faultEx.Code.Name)
   {
    case "01":
       // do something
       break;
    case "02":
        // do another something
        break
   }
}

Desacoplamento do registro de ErrorHandlerAttribute da implementação do serviço #

Para desacoplar e reutilizar o mesmo código de registro de erros para todos seus serviços, você precisa de duas classes padrão e guardá-las em uma biblioteca em algum lugar.

ErrorhandlerAttribute implementando IServiceBehavior. FaultErrorhandler implementando IErrorhandler que registra todas as exceções.

[AttributeUsage(AttributeTargets.Class)]
public class ErrorHandlerAttribute : Attribute, IServiceBehavior
{
    Type mErrorType;
    public ErrorHandlerAttribute(Type t)
    {
        if (!typeof(IErrorHandler).IsAssignableFrom(t))
            throw new ArgumentException("Type passed to ErrorHandlerAttribute constructor must inherit from IErrorHandler");
        mErrorType = t;
    }

    public void AddBindingParameters(ServiceDescription serviceDescription, ServiceHostBase serviceHostBase, Collection<ServiceEndpoint> endpoints, BindingParameterCollection bindingParameters)
    {
    }

    public void ApplyDispatchBehavior(ServiceDescription serviceDescription, ServiceHostBase serviceHostBase)
    {
        foreach (ChannelDispatcher dispatcher in serviceHostBase.ChannelDispatchers)
        {
            dispatcher.ErrorHandlers.Add((IErrorHandler)Activator.CreateInstance(mErrorType));
        }
    }

    public void Validate(ServiceDescription serviceDescription, ServiceHostBase serviceHostBase)
    {
    }
}


class FaultErrorHandler : IErrorHandler
{
    public bool HandleError(Exception error)
    {
        // LOG ERROR
        return false; // false so the session gets aborted
    }

    public void ProvideFault(Exception error, MessageVersion version, ref Message fault)
    {
    }
}

Em seguida, em sua implementação de serviço, adicione o atributo ErrorHandler passando na instância Type de FaultErrorHandler. ErrorHandler construirá uma instância desse tipo no qual HandleError é chamado.

[ServiceBehavior]
[ErrorHandler(typeof(FaultErrorHandler))]
public class MyService : IMyService
{
}
