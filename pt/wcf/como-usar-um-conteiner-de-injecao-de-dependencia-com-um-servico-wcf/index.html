Como configurar um serviço WCF para usar um contêiner de injeção de dependência (Castle Windsor) #

Este exemplo tem duas partes - algumas etapas padrão para adicionar o Castle Windsor ao seu serviço WCF e, em seguida, um exemplo simples e concreto para mostrar como configuramos e usamos o contêiner do Windsor.

Isso torna o exemplo um pouco longo. Se você já entende o uso de um contêiner DI, provavelmente só se importa com as etapas do clichê. Se o uso de um contêiner DI não for familiar, será necessário um pouco mais - ver a coisa toda funcionar de ponta a ponta - antes que faça sentido.

Etapas da Caldeira

  1. Adicione o pacote Nuget Castle Windsor WCF Integration ao seu aplicativo de serviço WCF. Isso adicionará referências ao Castle Windsor, bem como alguns componentes específicos para serviços WCF.

  2. Adicione uma Classe de Aplicativo Global (global.asax) ao seu projeto: Adicionar > Novo Item > Visual C# > Web > Classe de Aplicativo Global. O código que configura seu container deve ser chamado a partir do método Application_Start. Para mantê-lo organizado, podemos colocar toda a nossa configuração de contêiner em uma classe separada. Não precisamos. Não importa. Você verá isso feito de forma diferente em diferentes exemplos. O que importa é que ele é chamado a partir do método Application_Start porque essa é sua raiz de composição - onde o aplicativo é iniciado. A ideia é configurar seu container quando o aplicativo iniciar e nunca mais tocar diretamente nele. Ele apenas fica em segundo plano fazendo seu trabalho.

  3. Crie uma classe para configurar o contêiner. Isso faz duas coisas:

    • ContainerInstance.AddFacility<WcfFacility>() tells Windsor’s WCF code to use this particular container when creating instances of your WCF services.
    • ContainerInstance.Install(FromAssembly.This()) tells Windsor to scan This assembly (in other words, your WCF project) looking for classes that implement IWindsorInstaller. Those classes will provide instructions to tell your container how to resolve dependencies. (We’ll create one a few steps later.)

    public static class Container { private static readonly IWindsorContainer ContainerInstance = new WindsorContainer();

     public static void Configure()
 {
     ContainerInstance.AddFacility<WcfFacility>();
     ContainerInstance.Install(FromAssembly.This());
 }

    }

  4. Chame este método de Application_Start em seu global.asax:

    public class Global : System.Web.HttpApplication { protected void Application_Start(object sender, EventArgs e) { Container.Configure(); } }

  5. Crie um instalador. Esta é apenas uma classe que implementa IWindsorInstaller. Este está vazio. Não faz nada. Adicionaremos a esta classe em algumas etapas. Quando chamamos ContainerInstance.Install(FromAssembly.This()), ContainerInstance é passado para o método Install para que possamos registrar dependências com o container.

    public class WindsorInstaller : IWindsorInstaller { public void Install(IWindsorContainer container, IConfigurationStore store) { // Nothing here yet! } }

  6. Na marcação de qualquer serviço WCF que você criar, inclua esta diretiva. Isso indica que o aplicativo usará o recurso WCF do Windsor para criar instâncias do serviço, o que, por sua vez, significa que pode injetar dependências quando uma instância é criada.

    Factory=“Castle.Facilities.WcfIntegration.DefaultServiceHostFactory, Castle.Facilities.WcfIntegration”

Isso encerra as etapas padrão para configurar o serviço para usar um contêiner de injeção de dependência do Castle Windsor.

Mas o exemplo não está completo a menos que configuremos pelo menos um serviço WCF para injeção de dependência. O resto não é clichê, apenas um exemplo simples e concreto.

  1. Crie um novo serviço WCF chamado GreetingService.svc.

  2. Edite a marcação. Deve ficar assim:

    <%@ ServiceHost Language=“C#” Debug=“true” Service=“WcfWindsorDocumentation.GreetingService” CodeBehind=“GreetingService.svc.cs”
    Factory=“Castle.Facilities.WcfIntegration.DefaultServiceHostFactory, Castle.Facilities.WcfIntegration” %>

  3. Substitua IGreetingService (o contrato de serviço) por este:

    [ServiceContract] public interface IGreetingService { [OperationContract] string GetGreeting(); }

  4. Substitua GreetingService (em GreetingService.svc.cs) por este código. Observe que o construtor requer uma instância de IGreetingProvider que precisaremos de nosso contêiner para injetar.

    public class GreetingService : IGreetingService { private readonly IGreetingProvider _greetingProvider;

     public GreetingService(IGreetingProvider greetingProvider)
 {
     _greetingProvider = greetingProvider;
 }

 public string GetGreeting()
 {
     return _greetingProvider.GetGreeting();
 }

    }

  5. Adicione esta implementação de IGreetingProvider. Ele também tem algumas dependências próprias que precisaremos do contêiner para fornecer. As especificidades dessas classes não são muito importantes. Eles são apenas para criar algo fácil de seguir.

    public interface IGreetingProvider { string GetGreeting(); }

    public interface IComputerNameProvider { string GetComputerName(); }

    public class ComputerNameGreetingProvider : IGreetingProvider { private readonly IComputerNameProvider _computerNameProvider;

     public ComputerNameGreetingProvider(IComputerNameProvider computerNameProvider)
 {
     _computerNameProvider = computerNameProvider;
 }

 public string GetGreeting()
 {
     return string.Concat("Hello from ", _computerNameProvider.GetComputerName());
 }

    }

    public class EnvironmentComputerNameProvider : IComputerNameProvider { public string GetComputerName() { return System.Environment.MachineName; } }

  6. Agora temos todas as classes que precisamos. Tudo o que resta é registrar as dependências com nosso container. Em outras palavras, vamos dizer ao container quais classes ele precisa criar para que ele saiba como “construir” uma instância de GreetingService. Este código vai em nossa implementação de IWindsorInstaller (etapa 5 do código clichê.)

    public class WindsorInstaller : IWindsorInstaller { public void Install(IWindsorContainer container, IConfigurationStore store) { container.Register( Component.For<IGreetingService, GreetingService>(), Component.For<IGreetingProvider, ComputerNameGreetingProvider>(), Component.For<IComputerNameProvider, EnvironmentComputerNameProvider>()); } }

Isso informa ao contêiner:

  • É responsável por criar GreetingService quando necessário.
  • Quando tenta criar GreetingService vai precisar de um IGreetingProvider. Quando for necessário, deve-se criar um ComputerNameGreetingProvider.
  • Quando tenta criar ComputerNameGreetingProvider, essa classe requer um IComputerNameProvider. Ele deve criar uma instância de EnvironmentComputerNameProvider para atender a essa necessidade.

Se, em algum lugar no processo de criação do GreetingService ou de uma de suas dependências, encontrar um requisito para uma dependência que não registramos, ele nos informará com uma exceção útil, como esta.

Dependência ausente. O componente WcfWindsorDocumentation.ComputerNameGreetingProvider tem uma dependência em WcfWindsorDocumentation.IComputerNameProvider, que não pôde ser resolvido. Certifique-se de que a dependência esteja registrada corretamente no container como um serviço ou fornecida como argumento embutido.

Isso significa que algo dependia de IComputerNameProvider, mas não havia nada registrado para atender a essa dependência.

Isso só dá o pontapé inicial. Há muito mais para configurar e usar corretamente um contêiner de injeção de dependência para cenários do mundo real. Este exemplo cobre apenas o que é específico para adicionar Windsor a um aplicativo de serviço WCF. Se você usar um contêiner diferente, como Autofac ou Unity, descobrirá que, embora a sintaxe e os detalhes variem, em princípio eles fazem as mesmas coisas e você identificará facilmente as semelhanças.
