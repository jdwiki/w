A linguagem C é tradicionalmente uma linguagem compilada (em oposição a interpretada). O padrão C define fases de tradução, e o produto de sua aplicação é uma imagem de programa (ou programa compilado). Em [tag:c11], as fases estão listadas em §5.1.1.2.

Extensão de nome de arquivo Descrição
.c Arquivo fonte. Geralmente contém definições e código.
.h Arquivo de cabeçalho. Geralmente contém declarações.
.o arquivo objeto. Código compilado em linguagem de máquina.
.obj Extensão alternativa para arquivos de objetos.
.a Arquivo da biblioteca. Pacote de arquivos de objetos.
.dll Biblioteca de vínculo dinâmico no Windows.
.so Objeto compartilhado (biblioteca) em muitos sistemas do tipo Unix.
.dylib Biblioteca Dynamic-Link no OSX (variante Unix).
.exe, .com Arquivo executável do Windows. Formado pela vinculação de arquivos de objetos e arquivos de biblioteca. Em sistemas do tipo Unix, não há extensão de nome de arquivo especial para arquivo executável.
sinalizadores do compilador POSIX c99 Descrição
-o nome do arquivo Nome do arquivo de saída, por exemplo. (bin/program.exe, program)
-I diretório procure por cabeçalhos em diretório.
-D nome definir macro nome
-L diretório procure por bibliotecas em diretório.
-l nome biblioteca de links libname.

Compiladores em plataformas POSIX (Linux, mainframes, Mac) geralmente aceitam essas opções, mesmo que não sejam chamadas de c99.

  • Veja também c99 - compilar programas C padrão
Sinalizadores do GCC (GNU Compiler Collection) Descrição
-Parede Permite que todas as mensagens de aviso que são comumente aceitas sejam úteis.
-Wextra Ativa mais mensagens de aviso, pode ser muito barulhento.
-pedante Forçar avisos onde o código viola o padrão escolhido.
-Wconversão Habilite avisos na conversão implícita, use com cuidado.
-c Compila arquivos de origem sem vinculação.
-v Imprime informações de compilação.
  • gcc aceita os sinalizadores POSIX e muitos outros.
  • Muitos outros compiladores em plataformas POSIX (clang, compiladores específicos de fornecedores) também usam os sinalizadores listados acima.
  • Consulte também Invocando o GCC para muitas outras opções.
Sinalizadores TCC (Tiny C Compiler) Descrição
-Wimplicit-function-declaration Avisar sobre a declaração de função implícita.
-Wunsupported Avise sobre recursos do GCC não suportados que são ignorados pelo TCC.
-Wwrite-strings Faça constantes de string do tipo const char * em vez de char *.
-Erro Interrompa a compilação se forem emitidos avisos.
-Parede Ative todos os avisos, exceto -Werror, -Wunusupported e -Wwrite strings.

Tipos de arquivo #

A compilação de programas C requer que você trabalhe com cinco tipos de arquivos:

  1. Arquivos de origem: Esses arquivos contêm definições de funções e têm nomes que terminam em .c por convenção. Nota: .cc e .cpp são arquivos C++; não arquivos C. por exemplo, foo.c

  2. Arquivos de cabeçalho: Esses arquivos contêm protótipos de funções e várias instruções de pré-processador (veja abaixo). Eles são usados ​​para permitir que arquivos de código-fonte acessem funções definidas externamente. Os arquivos de cabeçalho terminam em .h por convenção. por exemplo, foo.h

  3. Arquivos de objetos: Esses arquivos são produzidos como saída do compilador. Eles consistem em definições de funções em formato binário, mas não são executáveis ​​por si mesmos. Arquivos de objeto terminam em .o por convenção, embora em alguns sistemas operacionais (por exemplo, Windows, MS-DOS), eles geralmente terminam em .obj. por exemplo, foo.o foo.obj

  4. Executivos binários: Estes são produzidos como saída de um programa chamado “linker”. O vinculador vincula vários arquivos de objeto para produzir um arquivo binário que pode ser executado diretamente. Os executáveis ​​binários não têm sufixo especial em sistemas operacionais Unix, embora geralmente terminem em .exe no Windows. por exemplo, foo foo.exe

  5. Bibliotecas: Uma biblioteca é um binário compilado, mas não é em si um executável (ou seja, não há função main() em uma biblioteca). Uma biblioteca contém funções que podem ser usadas por mais de um programa. Uma biblioteca deve ser fornecida com arquivos de cabeçalho que contenham protótipos para todas as funções da biblioteca; esses arquivos de cabeçalho devem ser referenciados (por exemplo, #include <library.h>) em qualquer arquivo de origem que use a biblioteca. O vinculador precisa então ser referido à biblioteca para que o programa possa ser compilado com êxito. Existem dois tipos de bibliotecas: estáticas e dinâmicas.

  • Biblioteca estática: Uma biblioteca estática (arquivos .a para sistemas POSIX e arquivos .lib para Windows — não deve ser confundido com [arquivos de biblioteca de importação DLL](https://msdn.microsoft.com /en-us/library/windows/desktop/ms682592(v=vs.85).aspx), que também usa a extensão .lib) está estaticamente embutido no programa . Bibliotecas estáticas têm a vantagem de que o programa sabe exatamente qual versão de uma biblioteca é usada. Por outro lado, os tamanhos dos executáveis ​​são maiores, pois todas as funções de biblioteca usadas estão incluídas. por exemplo, libfoo.a foo.lib
  • Biblioteca dinâmica: Uma biblioteca dinâmica (arquivos .so para a maioria dos sistemas POSIX, arquivos .dylib para OSX e arquivos .dll para Windows) é vinculado dinamicamente em tempo de execução pelo programa. Às vezes, também são chamadas de bibliotecas compartilhadas porque uma imagem de biblioteca pode ser compartilhada por muitos programas. As bibliotecas dinâmicas têm a vantagem de ocupar menos espaço em disco se mais de um aplicativo estiver usando a biblioteca. Além disso, eles permitem atualizações de bibliotecas (correções de bugs) sem ter que reconstruir executáveis. por exemplo, foo.so foo.dylib foo.dll

O Linker #

O trabalho do vinculador é vincular vários arquivos de objeto (arquivos .o) em um executável binário. O processo de vinculação envolve principalmente resolver endereços simbólicos em endereços numéricos. O resultado do processo de link é normalmente um programa executável.

Durante o processo de link, o linker pegará todos os módulos de objeto especificados na linha de comando, adicionará algum código de inicialização específico do sistema na frente e tentará resolver todas as referências externas no módulo de objeto com definições externas em outros arquivos de objeto (os arquivos de objeto podem ser especificados diretamente na linha de comando ou podem ser adicionados implicitamente por meio de bibliotecas). Em seguida, ele atribuirá endereços de carregamento para os arquivos de objeto, ou seja, especificará onde o código e os dados terminarão no espaço de endereço do programa finalizado. Depois de obter os endereços de carregamento, ele pode substituir todos os endereços simbólicos no código do objeto por endereços numéricos “reais” no espaço de endereço do destino. O programa está pronto para ser executado agora.

Isso inclui os arquivos de objeto que o compilador criou a partir de seus arquivos de código-fonte, bem como os arquivos de objeto que foram pré-compilados para você e coletados em arquivos de biblioteca. Esses arquivos têm nomes que terminam em .a ou .so, e você normalmente não precisa saber sobre eles, pois o linker sabe onde a maioria deles está localizada e os vinculará automaticamente conforme necessário.

Invocação implícita do linker #

Assim como o pré-processador, o linker é um programa separado, freqüentemente chamado de ld (mas o Linux usa collect2, por exemplo). Também como o pré-processador, o vinculador é invocado automaticamente para você quando você usa o compilador. Assim, a maneira normal de usar o linker é a seguinte:

% gcc foo.o bar.o baz.o -o myprog

Esta linha diz ao compilador para vincular três arquivos objeto (foo.o, bar.o e baz.o) em um arquivo executável binário chamado myprog. Agora você tem um arquivo chamado myprog que você pode executar e que esperamos fazer algo legal e/ou útil.

Invocação explícita do vinculador #

É possível invocar o vinculador diretamente, mas isso raramente é aconselhável e geralmente é muito específico da plataforma. Ou seja, as opções que funcionam no Linux não funcionarão necessariamente no Solaris, AIX, macOS, Windows e similarmente em qualquer outra plataforma. Se você trabalha com GCC, pode usar gcc -v para ver o que é executado em seu nome.

Opções para o vinculador #

O vinculador também recebe alguns argumentos para modificar seu comportamento. O comando a seguir diria ao gcc para vincular foo.o e bar.o, mas também incluir a biblioteca ncurses. 

% gcc foo.o bar.o -o foo -lncurses

Isso é na verdade (mais ou menos) equivalente a

% gcc foo.o bar.o /usr/lib/libncurses.so -o foo

(embora libncurses.so possa ser libncurses.a, que é apenas um arquivo criado com ar). Observe que você deve listar as bibliotecas (por nome de caminho ou via opções -lname) após os arquivos de objeto. Com bibliotecas estáticas, a ordem em que são especificadas é importante; muitas vezes, com bibliotecas compartilhadas, a ordem não importa.

Observe que em muitos sistemas, se você estiver usando funções matemáticas (de <math.h>), precisará especificar -lm para carregar a biblioteca de matemática — mas o Mac OS X e o macOS Sierra não exigem isso. Existem outras bibliotecas que são bibliotecas separadas no Linux e em outros sistemas Unix, mas não no macOS — threads POSIX e POSIX em tempo real e bibliotecas de rede são exemplos. Consequentemente, o processo de vinculação varia entre as plataformas.

Outras opções de compilação #

Isso é tudo que você precisa saber para começar a compilar seus próprios programas em C. Geralmente, também recomendamos que você use a opção de linha de comando -Wall:

% gcc -Wall -c foo.cc

A opção -Wall faz com que o compilador o avise sobre construções de código legais, mas duvidosas, e o ajudará a detectar muitos bugs muito cedo.

Se você quiser que o compilador lance mais avisos para você (incluindo variáveis ​​que são declaradas mas não usadas, esquecendo de retornar um valor etc.), você pode usar este conjunto de opções, pois -Wall, apesar do nome, não t ligue todos os avisos possíveis:

% gcc -Wall -Wextra -Wfloat-equal -Wundef -Wcast-align -Wwrite-strings -Wlogical-op \
>     -Wmissing-declarations -Wredundant-decls -Wshadow …

Note que clang tem uma opção -Weverything que realmente ativa todos os avisos em clang.

O compilador #

Depois que o pré-processador C incluiu todos os arquivos de cabeçalho e expandiu todas as macros, o compilador pode compilar o programa. Ele faz isso transformando o código-fonte C em um arquivo de código-objeto, que é um arquivo que termina em .o que contém a versão binária do código-fonte. O código objeto não é executável diretamente, no entanto. Para tornar um executável, você também tem que adicionar código para todas as funções de biblioteca que foram #included no arquivo (isso não é o mesmo que incluir as declarações, que é o que #include faz). Este é o trabalho do [linker](https://www.wikiod.com/pt/c/compilacao#O vinculador).

Em geral, a sequência exata de como invocar um compilador C depende muito do sistema que você está usando. Aqui estamos usando o compilador GCC, embora deva-se notar que existem muitos outros compiladores:

% gcc -Wall -c foo.c

% é o prompt de comando do SO. Isso diz ao compilador para executar o pré-processador no arquivo foo.c e então compilá-lo no arquivo de código objeto foo.o. A opção -c significa compilar o arquivo de código fonte em um arquivo objeto, mas não invocar o vinculador. Esta opção -c está disponível em sistemas POSIX, como Linux ou macOS; outros sistemas podem usar sintaxe diferente.

Se todo o seu programa estiver em um arquivo de código-fonte, você poderá fazer isso:

% gcc -Wall foo.c -o foo

Isso diz ao compilador para executar o pré-processador em foo.c, compilá-lo e vinculá-lo para criar um executável chamado foo. A opção -o indica que a próxima palavra na linha é o nome do arquivo executável binário (programa). Se você não especificar o -o, (se você apenas digitar gcc foo.c), o executável será nomeado a.out por razões históricas.

Em geral, o compilador executa quatro etapas ao converter um arquivo .c em um executável:

  1. pré-processamento - expande textualmente diretivas #include e macros #define em seu arquivo .c
  2. compilação - converte o programa em assembly (você pode parar o compilador nesta etapa adicionando a opção -S)
  3. montagem - converte a montagem em código de máquina
  4. linkage - vincula o código do objeto a bibliotecas externas para criar um executável

Observe também que o nome do compilador que estamos usando é GCC, que significa “compilador GNU C” e “coleção de compiladores GNU”, dependendo do contexto. Existem outros compiladores C. Para sistemas operacionais do tipo Unix, muitos deles têm o nome cc, para “compilador C”, que geralmente é um link simbólico para algum outro compilador. Em sistemas Linux, cc é frequentemente um alias para GCC. No macOS ou OS-X, aponta para clang.

Os padrões POSIX atualmente exigem c99 como o nome de um compilador C — ele suporta o padrão C99 por padrão. Versões anteriores do POSIX mandavam c89 como o compilador. O POSIX também exige que este compilador entenda as opções -c e -o que usamos acima.

Nota: A opção -Wall presente em ambos os exemplos gcc diz ao compilador para imprimir avisos sobre construções questionáveis, o que é fortemente recomendado. Também é uma boa ideia adicionar outras opções de aviso, por exemplo, -Wextra.

O pré-processador #

Antes que o compilador C comece a compilar um arquivo de código-fonte, o arquivo é processado em uma fase de pré-processamento. Esta fase pode ser feita por um programa separado ou totalmente integrada em um executável. Em qualquer caso, ele é invocado automaticamente pelo compilador antes do início da compilação propriamente dita. A fase de pré-processamento converte seu código-fonte em outro código-fonte ou unidade de tradução aplicando substituições textuais. Você pode pensar nisso como um código-fonte “modificado” ou “expandido”. Essa origem expandida pode existir como um arquivo real no sistema de arquivos ou pode ser armazenada na memória apenas por um curto período de tempo antes de ser processada.

Os comandos do pré-processador começam com o sinal de sustenido ("#"). Existem vários comandos de pré-processador; dois dos mais importantes são:

  1. Definições:

    #define is mainly used to define constants. For instance,

     #define BIGNUM 1000000
 int a = BIGNUM;

    becomes

     int a = 1000000;

`#define` is used in this way so as to avoid having to explicitly write out some constant value in many different places in a source code file. This is important in case you need to change the constant value later on; it's much less bug-prone to change it once, in the `#define`, than to have to change it in multiple places scattered all over the code.

    Because #define just does advanced search and replace, you can also declare macros. For instance:

     #define ISTRUE(stm) do{stm = stm ? 1 : 0;}while(0)
 // in the function:
 a = x;
 ISTRUE(a);

    becomes:

     // in the function:
 a = x;
 do {
     a = a ? 1 : 0;
 } while(0);

    At first approximation, this effect is roughly the same as with inline functions, but the preprocessor doesn’t provide type checking for #define macros. This is well known to be error-prone and their use necessitates great caution.

    Also note here, that the preprocessor would also replace comments with a blanks as explained below.

  2. Inclui:

#include é usado para acessar definições de funções definidas fora de um arquivo de código-fonte. Por exemplo:

    #include <stdio.h>

faz com que o pré-processador cole o conteúdo de <stdio.h> no arquivo de código fonte no local da instrução #include antes de ser compilado. #include é quase sempre usado para incluir arquivos de cabeçalho, que são arquivos que contêm principalmente declarações de função e instruções #define. Neste caso, usamos #include para podermos usar funções como printf e scanf, cujas declarações estão localizadas no arquivo stdio.h. Os compiladores C não permitem que você use uma função a menos que ela tenha sido declarada ou definida anteriormente nesse arquivo; As instruções #include são, portanto, a maneira de reutilizar o código escrito anteriormente em seus programas C.

  1. Operações lógicas:

    #if defined A || defined B
variable = another_variable + 1;
#else
variable = another_variable * 2;
#endif

    will be changed to:

    variable = another_variable + 1;

se A ou B foram definidos em algum lugar no projeto antes. Se este não for o caso, é claro que o pré-processador fará isso:

   variable = another_variable * 2;

Isso geralmente é usado para código, que é executado em diferentes sistemas ou compila em diferentes compiladores. Como existem definições globais, que são específicas do compilador/sistema, você pode testar essas definições e sempre deixar o compilador usar o código que ele compilará com certeza.

  1. Comentários

    The Preprocessor replaces all comments in the source file by single spaces. Comments are indicated by // up to the end of the line, or a combination of opening /* and closing */ comment brackets.

As Fases de Tradução #

A partir do padrão C 2011, listado em §5.1.1.2 Fases de tradução, a tradução do código-fonte para a imagem do programa (por exemplo, o executável) é listada para ocorrer em 8 etapas ordenadas.

  1. A entrada do arquivo de origem é mapeada para o conjunto de caracteres de origem (se necessário). As trigrafias são substituídas nesta etapa.
  2. As linhas de continuação (linhas que terminam com \) são unidas com a próxima linha.
  3. O código-fonte é analisado em espaços em branco e tokens de pré-processamento.
  4. Aplica-se o pré-processador, que executa diretivas, expande macros e aplica pragmas. Cada arquivo de origem obtido por #include passa pelas fases de tradução de 1 a 4 (recursivamente, se necessário). Todas as diretivas relacionadas ao pré-processador são então excluídas.
  5. Os valores do conjunto de caracteres de origem em constantes de caracteres e literais de string são mapeados para o conjunto de caracteres de execução.
  6. Os literais de string adjacentes são concatenados.
  7. O código-fonte é analisado em tokens, que compõem a unidade de tradução.
  8. As referências externas são resolvidas e a imagem do programa é formada.

Uma implementação de um compilador C pode combinar várias etapas, mas a imagem resultante ainda deve se comportar como se as etapas acima tivessem ocorrido separadamente na ordem listada acima.
