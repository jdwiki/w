Todos os comandos do pré-processador começam com o símbolo de hash (libra) #. Uma macro C é apenas um comando de pré-processador que é definido usando a diretiva de pré-processador #define. Durante o estágio de pré-processamento, o pré-processador C (uma parte do compilador C) simplesmente substitui o corpo da macro onde quer que seu nome apareça.

Quando um compilador encontra uma macro no código, ele executa uma simples substituição de string, nenhuma operação adicional é executada. Por causa disso, as alterações feitas pelo pré-processador não respeitam o escopo dos programas C - por exemplo, uma definição de macro não se limita a estar dentro de um bloco, portanto não é afetada por um '}' que encerra uma instrução de bloco.

O pré-processador não é, por design, completo - existem vários tipos de computação que não podem ser feitos apenas pelo pré-processador.

Normalmente, os compiladores têm um sinalizador de linha de comando (ou configuração) que nos permite interromper a compilação após a fase de pré-processamento e inspecionar o resultado. Em plataformas POSIX este sinalizador é -E. Então, executar gcc com este sinalizador imprime a fonte expandida para stdout:

$ gcc -E cprog.c

Muitas vezes o pré-processador é implementado como um programa separado, que é invocado pelo compilador, o nome comum para esse programa é cpp. Vários pré-processadores emitem informações de suporte, como informações sobre números de linha - que são usadas pelas fases subsequentes de compilação para gerar informações de depuração. Caso o pré-processador seja baseado em gcc, a opção -P suprime tal informação.

$ cpp -P cprog.c

#if 0 para bloquear seções de código #

Se houver seções de código que você está pensando em remover ou deseja desabilitar temporariamente, você pode comentar com um comentário de bloco.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
*/

No entanto, se o código-fonte que você cercou com um comentário de bloco tiver comentários de estilo de bloco na fonte, o final */ dos comentários de bloco existentes pode fazer com que seu novo comentário de bloco seja inválido e cause problemas de compilação.

/* Block comment around whole function to keep it from getting used.
 * What's even the purpose of this function?
int myUnusedFunction(void)
{
    int i = 5;

    /* Return 5 */
    return i;
}
*/

No exemplo anterior, as duas últimas linhas da função e o último ‘*/’ são vistos pelo compilador, então ele compilaria com erros. Um método mais seguro é usar uma diretiva #if 0 em torno do código que você deseja bloquear.

#if 0
/* #if 0 evaluates to false, so everything between here and the #endif are
 * removed by the preprocessor. */
int myUnusedFunction(void)
{
    int i = 5;
    return i;
}
#endif

Uma vantagem disso é que quando você quiser voltar e encontrar o código, é muito mais fácil fazer uma busca por “#if 0” do que pesquisar todos os seus comentários.

Outro benefício muito importante é que você pode aninhar o código de comentários com #if 0. Isso não pode ser feito com comentários.

Uma alternativa para usar #if 0 é usar um nome que não será #definido, mas é mais descritivo do motivo pelo qual o código está sendo bloqueado. Por exemplo, se houver uma função que parece ser um código morto inútil, você pode usar #ifdefined(POSSIBLE_DEAD_CODE) ou #ifdefined(FUTURE_CODE_REL_020201) para o código necessário quando outra funcionalidade estiver em vigor ou algo semelhante. Então, ao voltar para remover ou ativar essa fonte, essas seções da fonte são fáceis de encontrar.

Cabeçalho Inclui Guardas #

Praticamente todos os arquivos de cabeçalho devem seguir o idioma include guard:

meu-arquivo-cabeçalho.h

#ifndef MY_HEADER_FILE_H
#define MY_HEADER_FILE_H

// Code body for header file

#endif

Isso garante que quando você #include "my-header-file.h" em vários lugares, você não obtém declarações duplicadas de funções, variáveis, etc. Imagine a seguinte hierarquia de arquivos:

cabeçalho-1.h

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

cabeçalho-2.h

#include "header-1.h"

int myFunction2(MyStruct *value);

main.c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Este código tem um problema sério: o conteúdo detalhado de MyStruct é definido duas vezes, o que não é permitido. Isso resultaria em um erro de compilação que pode ser difícil de rastrear, pois um arquivo de cabeçalho inclui outro. Se você fez isso com guardas de cabeçalho:

cabeçalho-1.h

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

cabeçalho-2.h

#ifndef HEADER_2_H
#define HEADER_2_H

#include "header-1.h"

int myFunction2(MyStruct *value);

#endif

main.c

#include "header-1.h"
#include "header-2.h"

int main() {
    // do something
}

Isso se expandiria para:

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

#ifndef HEADER_2_H
#define HEADER_2_H

#ifndef HEADER_1_H // Safe, since HEADER_1_H was #define'd before.
#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#endif

int myFunction2(MyStruct *value);

#endif

int main() {
    // do something
}

Quando o compilador alcança a segunda inclusão de header-1.h, HEADER_1_H já estava definido pela inclusão anterior. Logo, resume-se ao seguinte:

#define HEADER_1_H

typedef struct {
    …
} MyStruct;

int myFunction(MyStruct *value);

#define HEADER_2_H

int myFunction2(MyStruct *value);

int main() {
    // do something
}

E, portanto, não há erro de compilação.

Nota: Existem várias convenções diferentes para nomear os protetores de cabeçalho. Algumas pessoas gostam de chamá-lo de HEADER_2_H_, alguns incluem o nome do projeto como MY_PROJECT_HEADER_2_H. O importante é garantir que a convenção que você segue faça com que cada arquivo em seu projeto tenha uma proteção de cabeçalho exclusiva.

Se os detalhes da estrutura não fossem incluídos no cabeçalho, o tipo declarado seria incompleto ou um tipo opaco. Esses tipos podem ser úteis, ocultando detalhes de implementação dos usuários das funções. Para muitos propósitos, o tipo FILE na biblioteca C padrão pode ser considerado um tipo opaco (embora geralmente não seja opaco, de modo que as implementações de macro das funções de E/S padrão possam usar os componentes internos da estrutura) . Nesse caso, o header-1.h poderia conter:

#ifndef HEADER_1_H
#define HEADER_1_H

typedef struct MyStruct MyStruct;

int myFunction(MyStruct *value);

#endif

Observe que a estrutura deve ter um nome de tag (aqui MyStruct — que está no namespace de tags, separado do namespace de identificadores comuns do nome typedef MyStruct), e que o { … } é omitido. Isso diz que “há um tipo de estrutura struct MyStruct e há um alias para ele MyStruct”.

No arquivo de implementação, os detalhes da estrutura podem ser definidos para tornar o tipo completo:

struct MyStruct {
    …
};

Se você estiver usando C11, você pode repetir a declaração typedef struct MyStruct MyStruct; sem causar um erro de compilação, mas versões anteriores de C reclamariam. Conseqüentemente, ainda é melhor usar o idioma include guard, embora neste exemplo, seria opcional se o código fosse compilado apenas com compiladores que suportassem C11.

Muitos compiladores suportam a diretiva #pragma once, que tem os mesmos resultados:

meu-arquivo-cabeçalho.h

#pragma once

// Code for header file

No entanto, #pragma once não faz parte do padrão C, então o código é menos portátil se você o usar.

Alguns cabeçalhos não usam o idioma include guard. Um exemplo específico é o cabeçalho padrão <assert.h>. Ele pode ser incluído várias vezes em uma única unidade de tradução, e o efeito de fazer isso depende se a macro NDEBUG é definida cada vez que o cabeçalho é incluído. Você pode ocasionalmente ter um requisito análogo; tais casos serão poucos e distantes entre si. Normalmente, seus cabeçalhos devem ser protegidos pelo idioma include guard.

Macros semelhantes a funções #

Macros de função são semelhantes a funções inline, elas são úteis em alguns casos, como log de depuração temporário:

#ifdef DEBUG
# define LOGFILENAME "/tmp/logfile.log"

# define LOG(str) do {                            \
  FILE *fp = fopen(LOGFILENAME, "a");            \
  if (fp) {                                       \
    fprintf(fp, "%s:%d %s\n", __FILE__, __LINE__, \
                 /* don't print null pointer */   \
                 str ?str :"<null>");             \
    fclose(fp);                                   \
  }                                               \
  else {                                          \
    perror("Opening '" LOGFILENAME "' failed");   \
  }                                               \
} while (0)
#else
  /* Make it a NOOP if DEBUG is not defined. */
# define LOG(LINE) (void)0
#endif


#include <stdio.h>

int main(int argc, char* argv[])
{
    if (argc > 1)
        LOG("There are command line arguments");
    else
        LOG("No command line arguments");
    return 0;
}

Aqui em ambos os casos (com DEBUG ou não) a chamada se comporta da mesma maneira como uma função com tipo de retorno void. Isso garante que as condicionais if/else sejam interpretadas conforme o esperado.

No caso DEBUG isto é implementado através de um do { ... } while(0) construção. No outro caso, (void)0 é uma declaração sem efeito colateral que é simplesmente ignorada.

Uma alternativa para este último seria

#define LOG(LINE) do { /* empty */ } while (0)

tal que é em todos os casos sintaticamente equivalente ao primeiro.

Se você usa o GCC, também pode implementar uma macro funcional que retorna o resultado usando uma extensão GNU não padrão — [expressões de instrução](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/ Statement-Exprs.html#Statement-Exprs). Por exemplo:

#include <stdio.h>

#define POW(X, Y) \
({ \
        int i, r = 1; \
        for (i = 0; i < Y; ++i) \
                r *= X; \
        r; \ // returned value is result of last operation
})

int main(void)
{
        int result;

        result = POW(2, 3); 
        printf("Result: %d\n", result);
}

Inclusão do arquivo de origem #

Os usos mais comuns das diretivas de pré-processamento #include são os seguintes:

#include <stdio.h>
#include "myheader.h"

#include substitui a instrução pelo conteúdo do arquivo referido. Os colchetes angulares (<>) referem-se aos arquivos de cabeçalho instalados no sistema, enquanto as aspas ("") são para arquivos fornecidos pelo usuário.

As próprias macros podem expandir outras macros uma vez, como este exemplo ilustra:

#if VERSION == 1
    #define INCFILE  "vers1.h"
#elif VERSION == 2
    #define INCFILE  "vers2.h"
    /*  and so on */
#else
    #define INCFILE  "versN.h"
#endif
/* ... */
#include INCFILE

Inclusão condicional e modificação de assinatura de função condicional #

Para incluir condicionalmente um bloco de código, o pré-processador tem várias diretivas (por exemplo, #if, #ifdef, #else, #endif, etc).

/* Defines a conditional `printf` macro, which only prints if `DEBUG`
 * has been defined
 */
#ifdef DEBUG
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

Operadores relacionais normais C podem ser usados ​​para a condição #if

#if __STDC_VERSION__ >= 201112L
/* Do stuff for C11 or higher */
#elif __STDC_VERSION__ >= 199901L
/* Do stuff for C99 */
#else
/* Do stuff for pre C99 */
#endif

As diretivas #if se comportam de forma semelhante à instrução C if, ela deve conter apenas expressões de constantes integrais e sem casts. Ele suporta um operador unário adicional, defined(identificador), que retorna 1 se o identificador for definido e 0 caso contrário.

#if defined(DEBUG) && !defined(QUIET)
#define DLOG(x) (printf(x))
#else
#define DLOG(x)
#endif

Modificação de assinatura de função condicional

Na maioria dos casos, espera-se que uma compilação de lançamento de um aplicativo tenha a menor sobrecarga possível. No entanto, durante o teste de uma compilação provisória, logs adicionais e informações sobre problemas encontrados podem ser úteis.

Por exemplo, suponha que exista alguma função SHORT SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd) que ao fazer uma compilação de teste se deseja gerar um log sobre seu uso. Porém esta função é utilizada em vários lugares e deseja-se que ao gerar o log, parte da informação seja saber de onde está a função que está sendo chamada.

Portanto, usando a compilação condicional, você pode ter algo como o seguinte no arquivo de inclusão declarando a função. Isso substitui a versão padrão da função por uma versão de depuração da função. O pré-processador é usado para substituir chamadas à função SerOpPluAllRead() por chamadas à função SerOpPluAllRead_Debug() com dois argumentos adicionais, o nome do arquivo e o número da linha onde a função é usada.

A compilação condicional é usada para escolher se deseja substituir a função padrão por uma versão de depuração ou não.

#if 0
// function declaration and prototype for our debug version of the function.
SHORT   SerOpPluAllRead_Debug(PLUIF *pPif, USHORT usLockHnd, char *aszFilePath, int nLineNo);

// macro definition to replace function call using old name with debug function with additional arguments.
#define SerOpPluAllRead(pPif,usLock) SerOpPluAllRead_Debug(pPif,usLock,__FILE__,__LINE__)
#else
// standard function declaration that is normally used with builds.
SHORT   SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd);
#endif

Isso permite que você substitua a versão padrão da função SerOpPluAllRead() por uma versão que fornecerá o nome do arquivo e o número da linha no arquivo de onde a função é chamada.

Há uma consideração importante: qualquer arquivo usando esta função deve incluir o arquivo de cabeçalho onde esta abordagem é usada para que o pré-processador modifique a função. Caso contrário, você verá um erro de vinculador.

A definição da função seria algo como o seguinte. O que esta fonte faz é solicitar que o pré-processador renomeie a função SerOpPluAllRead() para SerOpPluAllRead_Debug() e modifique a lista de argumentos para incluir dois argumentos adicionais, um ponteiro para o nome do arquivo onde a função foi chamada e o número da linha no arquivo no qual a função é usada.

#if defined(SerOpPluAllRead)
// forward declare the replacement function which we will call once we create our log.
SHORT    SerOpPluAllRead_Special(PLUIF *pPif, USHORT usLockHnd);

SHORT    SerOpPluAllRead_Debug(PLUIF *pPif, USHORT usLockHnd, char *aszFilePath, int nLineNo)
{
    int iLen = 0;
    char  xBuffer[256];

    // only print the last 30 characters of the file name to shorten the logs.
    iLen = strlen (aszFilePath);
    if (iLen > 30) {
        iLen = iLen - 30;
    }
    else {
        iLen = 0;
    }

    sprintf (xBuffer, "SerOpPluAllRead_Debug(): husHandle = %d, File %s, lineno = %d", pPif->husHandle, aszFilePath + iLen, nLineNo);
    IssueDebugLog(xBuffer);

    // now that we have issued the log, continue with standard processing.
    return SerOpPluAllRead_Special(pPif, usLockHnd);
}

// our special replacement function name for when we are generating logs.
SHORT    SerOpPluAllRead_Special(PLUIF *pPif, USHORT usLockHnd)
#else
// standard, normal function name (signature) that is replaced with our debug version.
SHORT   SerOpPluAllRead(PLUIF *pPif, USHORT usLockHnd)
#endif
{
    if (STUB_SELF == SstReadAsMaster()) {
        return OpPluAllRead(pPif, usLockHnd);
    } 
    return OP_NOT_MASTER;
}

Colagem de token #

A colagem de token permite colar dois argumentos de macro. Por exemplo, front##back produz frontback. Um exemplo famoso é o cabeçalho <TCHAR.H> do Win32. No padrão C, pode-se escrever L"string" para declarar uma cadeia de caracteres larga. No entanto, a API do Windows permite converter entre cadeias de caracteres largas e cadeias de caracteres estreitas simplesmente #defineing UNICODE. Para implementar os literais de string, TCHAR.H usa este

#ifdef UNICODE
#define TEXT(x) L##x
#endif

Sempre que um usuário escreve TEXT("hello, world"), e UNICODE é definido, o pré-processador C concatena L e o argumento macro. L concatenado com "hello, world" dá L"hello, world".

Macros predefinidas #

Uma macro predefinida é uma macro que já é compreendida pelo pré-processador C sem que o programa precise defini-la. Exemplos incluem

Macros pré-definidas obrigatórias #

  • __FILE__, que dá o nome do arquivo de origem atual (uma string literal),
  • __LINE__ para o número da linha atual (uma constante inteira),
  • __DATE__ para a data de compilação (uma string literal),
  • __TIME__ para o tempo de compilação (uma string literal).

Há também um identificador predefinido relacionado, __func__ (ISO/IEC 9899:2011 §6.4.2.2), que não é uma macro:

O identificador __func__ deve ser declarado implicitamente pelo tradutor como se, imediatamente após a chave de abertura de cada definição de função, a declaração:

static const char func[] = “function-name”;

apareceu, onde function-name é o nome da função que encerra lexicalmente.

__FILE__, __LINE__ e __func__ são especialmente úteis para fins de depuração. Por exemplo:

fprintf(stderr, "%s: %s: %d: Denominator is 0", __FILE__, __func__, __LINE__);

Compiladores pré-C99, podem ou não suportar __func__ ou podem ter uma macro que age da mesma forma que tem um nome diferente. Por exemplo, o gcc usou __FUNCTION__ no modo C89.

As macros abaixo permitem solicitar detalhes sobre a implementação:

  • __STDC_VERSION__ A versão do C Standard implementada. Este é um inteiro constante usando o formato yyyymmL (o valor 201112L para C11, o valor 199901L para C99; não foi definido para C89/C90)
  • __STDC_HOSTED__ 1 se for uma implementação hospedada, senão 0.
  • __STDC__ Se 1, a implementação está em conformidade com o C Standard.

Outras macros pré-definidas (não obrigatórias) #

ISO/IEC 9899:2011 §6.10.9.2 Macros de ambiente:

  • __STDC_ISO_10646__ Uma constante inteira no formato yyyymmL (por exemplo, 199712L). Se este símbolo for definido, então cada caractere no Unicode conjunto requerido, quando armazenado em um objeto do tipo wchar_t, tem o mesmo value como o identificador curto desse caractere. O conjunto necessário de Unicode consiste em todos os caracteres que são definidos pela ISO/IEC 10646, juntamente com todas as alterações e correções técnicas, a partir do ano especificado e mês. Se alguma outra codificação for usada, a macro não deve ser definida e a codificação real usada é definida pela implementação.

  • __STDC_MB_MIGHT_NEQ_WC__ A constante inteira 1, destinada a indicar que, em a codificação para wchar_t, um membro do conjunto básico de caracteres não precisa tem um valor de código igual ao seu valor quando usado como o caractere solitário em um constante de caractere inteiro.

  • __STDC_UTF_16__ A constante inteira 1, destinada a indicar que os valores do tipo char16_t são codificados em UTF-16. Se alguma outra codificação for usada, o macro não deve ser definida e a codificação real usada é definida pela implementação.

  • __STDC_UTF_32__ A constante inteira 1, destinada a indicar que os valores do tipo char32_t são codificados em UTF-32. Se alguma outra codificação for usada, o macro não deve ser definida e a codificação real usada é definida pela implementação.

ISO/IEC 9899:2011 §6.10.8.3 Macros de recursos condicionais

  • __STDC_ANALYZABLE__ A constante inteira 1, destinada a indicar conformidade com as especificações no anexo L (Analisabilidade).
  • __STDC_IEC_559__ A constante inteira 1, destinada a indicar conformidade com o especificações no anexo F (aritmética de ponto flutuante IEC 60559).
  • __STDC_IEC_559_COMPLEX__ A constante inteira 1, destinada a indicar aderência às especificações no anexo G (complexo compatível com IEC 60559 aritmética).
  • __STDC_LIB_EXT1__ A constante inteira 201112L, destinada a indicar suporte para as extensões definidas no anexo K (interfaces de verificação de limites).
  • __STDC_NO_ATOMICS__ A constante inteira 1, destinada a indicar que o implementação não suporta tipos atômicos (incluindo o _Atomic qualificador de tipo) e o cabeçalho <stdatomic.h>.
  • __STDC_NO_COMPLEX__ A constante inteira 1, destinada a indicar que o implementação não suporta tipos complexos ou o <complex.h> cabeçalho.
  • __STDC_NO_THREADS__ A constante inteira 1, destinada a indicar que o implementação não suporta o cabeçalho <threads.h>.
  • __STDC_NO_VLA__ A constante inteira 1, destinada a indicar que o implementação não suporta arrays de comprimento variável ou variável tipos modificados.

__cplusplus para usar C externals em código C++ compilado com C++ - name mangling #

Há momentos em que um arquivo de inclusão precisa gerar uma saída diferente do pré-processador, dependendo se o compilador é um compilador C ou um compilador C++ devido a diferenças de idioma.

Por exemplo, uma função ou outro externo é definido em um arquivo de origem C, mas é usado em um arquivo de origem C++. Como o C++ usa o desmembramento de nome (ou decoração de nome) para gerar nomes de função exclusivos com base nos tipos de argumento de função, uma declaração de função C usada em um arquivo de origem C++ causará erros de link. O compilador C++ modificará o nome externo especificado para a saída do compilador usando as regras de desmembramento de nomes para C++. O resultado são erros de link devido a elementos externos não encontrados quando a saída do compilador C++ está vinculada à saída do compilador C.

Uma vez que os compiladores C não fazem o desmembramento de nomes, mas os compiladores C++ fazem para todos os rótulos externos (nomes de funções ou nomes de variáveis) gerados pelo compilador C++, uma macro de pré-processador predefinida, __cplusplus, foi introduzida para permitir a detecção do compilador.

Para contornar este problema de saída do compilador incompatível para nomes externos entre C e C++, a macro __cplusplus é definida no pré-processador C++ e não é definida no pré-processador C. Este nome de macro pode ser usado com a diretiva do pré-processador condicional #ifdef ou #if com o operador defined() para informar se um código-fonte ou arquivo de inclusão está sendo compilado como C++ ou C.

#ifdef __cplusplus
printf("C++\n");
#else
printf("C\n");
#endif

Ou você poderia usar

#if defined(__cplusplus)
printf("C++\n");
#else
printf("C\n");
#endif

Para especificar o nome correto de uma função de um arquivo fonte C compilado com o compilador C que está sendo usado em um arquivo fonte C++, você pode verificar a constante definida __cplusplus para causar o extern "C" { /* ... */ }; a ser usado para declarar C externals quando o arquivo de cabeçalho for incluído em um arquivo de origem C++. No entanto, quando compilado com um compilador C, o extern "C" { */ ... */ }; não é usado. Essa compilação condicional é necessária porque extern "C" { /* ... */ }; é válido em C++, mas não em C.

#ifdef __cplusplus
// if we are being compiled with a C++ compiler then declare the
// following functions as C functions to prevent name mangling.
extern "C" {
#endif

// exported C function list.
int foo (void);

#ifdef __cplusplus
// if this is a C++ compiler, we need to close off the extern declaration.
};
#endif

Macro de argumentos variáveis #

Macros com argumentos variádicos:

Digamos que você queira criar alguma macro de impressão para depurar seu código, vamos usar esta macro como exemplo:

#define debug_print(msg) printf("%s:%d %s", __FILE__, __LINE__, msg)

Alguns exemplos de uso:

A função somefunc() retorna -1 se falhar e 0 se for bem sucedida, e é chamada de vários lugares diferentes dentro do código:

int retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

/* some other code */

 retVal = somefunc();

if(retVal == -1)
{
    debug_printf("somefunc() has failed");
}

O que acontece se a implementação de somefunc() for alterada e agora retornar valores diferentes correspondendo a diferentes tipos de erro possíveis? Você ainda deseja usar a macro de depuração e imprimir o valor do erro.

debug_printf(retVal);      /* this would obviously fail */
debug_printf("%d",retVal); /* this would also fail */

Para resolver este problema foi introduzida a macro __VA_ARGS__. Esta macro permite vários parâmetros X-macro’s:

Exemplo:

 #define debug_print(msg, ...) printf(msg, __VA_ARGS__) \
                               printf("\nError occurred in file:line (%s:%d)\n", __FILE__, __LINE)

Uso:

int retVal = somefunc();

debug_print("retVal of somefunc() is-> %d", retVal);

Essa macro permite que você passe vários parâmetros e os imprima, mas agora proíbe você de enviar qualquer parâmetro.

debug_print("Hey");

Isso geraria algum erro de sintaxe, pois a macro espera pelo menos mais um argumento e o pré-processador não ignoraria a falta de vírgula na macro debug_print(). Também debug_print("Hey",); geraria um erro de sintaxe, pois você não pode manter o argumento passado para a macro vazio.

Para resolver isso, foi introduzida a macro ##__VA_ARGS__, esta macro afirma que se não existir nenhum argumento de variável, a vírgula é excluída pelo pré-processador do código.

Exemplo:

 #define debug_print(msg, ...) printf(msg, ##__VA_ARGS__) \
                               printf("\nError occured in file:line (%s:%d)\n", __FILE__, __LINE)

Uso:

 debug_print("Ret val of somefunc()?");
 debug_print("%d",somefunc());

Substituição de macro #

A forma mais simples de substituição de macro é definir uma ‘constante de manifesto’, como em

#define ARRSIZE 100
int array[ARRSIZE];

Isso define uma macro semelhante a uma função que multiplica uma variável por 10 e armazena o novo valor:

#define TIMES10(A) ((A) *= 10)

double b = 34;
int c = 23;

TIMES10(b);   // good: ((b) *= 10);
TIMES10(c);   // good: ((c) *= 10);
TIMES10(5);   // bad:  ((5) *= 10);

A substituição é feita antes de qualquer outra interpretação do texto do programa. Na primeira chamada para TIMES10 o nome A da definição é substituído por b e o texto assim expandido é então colocado no lugar da chamada. Observe que esta definição de TIMES10 não é equivalente a

#define TIMES10(A) ((A) = (A) * 10)

porque isso poderia avaliar a substituição de A, duas vezes, o que pode ter efeitos colaterais indesejados.

O seguinte define uma macro do tipo função cujo valor é o máximo de seus argumentos. Ele tem as vantagens de trabalhar para qualquer tipo compatível de argumentos e de gerar código in-line sem a sobrecarga da chamada de função. Tem as desvantagens de avaliar um ou outro de seus argumentos uma segunda vez (incluindo efeitos colaterais) e de gerar mais código do que uma função se invocada várias vezes.

#define max(a, b) ((a) > (b) ? (a) : (b))

int maxVal = max(11, 43);              /* 43 */
int maxValExpr = max(11 + 36, 51 - 7); /* 47 */

/* Should not be done, due to expression being evaluated twice */
int j = 0, i = 0;
int sideEffect = max(++i, ++j);       /* i == 4 */

Por causa disso, essas macros que avaliam seus argumentos várias vezes geralmente são evitadas no código de produção. Desde o C11 existe o recurso _Generic que permite evitar tais invocações múltiplas.

Os parênteses abundantes nas expansões da macro (lado direito da definição) garantem que os argumentos e a expressão resultante sejam vinculados adequadamente e se encaixem bem no contexto em que a macro é chamada.

Diretiva de erro #

Se o pré-processador encontrar uma diretiva #error, a compilação será interrompida e a mensagem de diagnóstico incluída será impressa.

#define DEBUG

#ifdef DEBUG
#error "Debug Builds Not Supported"
#endif

int main(void) {
    return 0;
}

Saída possível:

$ gcc error.c
error.c: error: #error "Debug Builds Not Supported"

Implementação do FOREACH #

Também podemos usar macros para tornar o código mais fácil de ler e escrever. Por exemplo, podemos implementar macros para implementar a construção foreach em C para algumas estruturas de dados como listas, filas, etc.

Aqui está um pequeno exemplo.

#include <stdio.h>
#include <stdlib.h>

struct LinkedListNode
{
    int data;
    struct LinkedListNode *next;
};

#define FOREACH_LIST(node, list) \
     for (node=list; node; node=node->next)

/* Usage */
int main(void)
{
    struct LinkedListNode *list, **plist = &list, *node;
    int i;

    for (i=0; i<10; i++)
    {
         *plist = malloc(sizeof(struct LinkedListNode));
         (*plist)->data = i;
         (*plist)->next = NULL;
         plist          = &(*plist)->next;
    }

    /* printing the elements here */
    FOREACH_LIST(node, list)
    {
        printf("%d\n", node->data);
    }
}

Você pode fazer uma interface padrão para tais estruturas de dados e escrever uma implementação genérica de FOREACH como:

#include <stdio.h>
#include <stdlib.h>

typedef struct CollectionItem_
{
    int data;
    struct CollectionItem_ *next;
} CollectionItem;

typedef struct Collection_
{
    /* interface functions */
    void* (*first)(void *coll);
    void* (*last) (void *coll);
    void* (*next) (void *coll, CollectionItem *currItem);

    CollectionItem *collectionHead;
    /* Other fields */
} Collection;

/* must implement */
void *first(void *coll)
{
    return ((Collection*)coll)->collectionHead;
}

/* must implement */
void *last(void *coll)
{
    return NULL;
}

/* must implement */
void *next(void *coll, CollectionItem *curr)
{
    return curr->next;
}

CollectionItem *new_CollectionItem(int data)
{
    CollectionItem *item = malloc(sizeof(CollectionItem));
    item->data = data;
    item->next = NULL;
    return item;
}

void Add_Collection(Collection *coll, int data)
{
    CollectionItem **item = &coll->collectionHead;
    while(*item)
        item = &(*item)->next;
    (*item) = new_CollectionItem(data);
}

Collection *new_Collection()
{
    Collection *nc = malloc(sizeof(Collection));
    nc->first = first;
    nc->last  = last;
    nc->next  = next;
    return nc;
}

/* generic implementation */
#define FOREACH(node, collection)                      \
    for (node  = (collection)->first(collection);      \
         node != (collection)->last(collection);       \
         node  = (collection)->next(collection, node))

int main(void)
{
    Collection *coll = new_Collection();
    CollectionItem *node;
    int i;

    for(i=0; i<10; i++)
    {
         Add_Collection(coll, i);
    }

    /* printing the elements here */
    FOREACH(node, coll)
    {
        printf("%d\n", node->data);
    }
}

Para usar essa implementação genérica, basta implementar essas funções para sua estrutura de dados.

1.  void* (*first)(void *coll);
2.  void* (*last) (void *coll);
3.  void* (*next) (void *coll, CollectionItem *currItem);
