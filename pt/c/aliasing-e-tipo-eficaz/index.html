Aliasing e tipo eficaz

Violações de regras de alias e violação do tipo efetivo de um objeto são duas coisas diferentes e não devem ser confundidas.

  • Aliasing é a propriedade de dois ponteiros a e b que se referem ao mesmo objeto, ou seja, a == b.

  • O tipo efetivo de um objeto de dados é usado por C para determinar quais operações podem ser feitas nesse objeto. Em particular, o tipo efetivo é usado para determinar se dois ponteiros podem ser alias um do outro.

Aliasing pode ser um problema para otimização, porque mudar o objeto através de um ponteiro, a digamos, pode mudar o objeto que é visível através do outro ponteiro, b. Se o seu compilador C tivesse que assumir que os ponteiros sempre poderiam ser alias uns dos outros, independentemente de seu tipo e proveniência, muitas oportunidades de otimização seriam perdidas e muitos programas seriam executados mais lentamente.

As regras estritas de alias de C se referem a casos no compilador podem assumir quais objetos fazem (ou não) alias uns aos outros. Existem duas regras básicas que você sempre deve ter em mente para ponteiros de dados.

Salvo disposição em contrário, dois ponteiros com o mesmo tipo de base podem ser alias.

Dois ponteiros com tipo base diferente não podem ser alias, a menos que pelo menos um dos dois tipos seja um tipo de caractere.

Aqui tipo base significa que deixamos de lado qualificações de tipo como const, por exemplo Se a for double* e b for const double*, o compilador deve geralmente assumir que uma alteração de *a pode alterar *b.

Violar a segunda regra pode ter resultados catastróficos. Aqui, violar a regra de aliasing estrita significa que você apresenta dois ponteiros a e b de tipo diferente para o compilador que na realidade apontam para o mesmo objeto. O compilador pode sempre assumir que os dois apontam para objetos diferentes e não atualizará sua ideia de *b se você alterar o objeto por meio de *a.

Se você fizer isso, o comportamento do seu programa se tornará indefinido. Portanto, C coloca restrições bastante severas nas conversões de ponteiro para ajudá-lo a evitar que tal situação ocorra acidentalmente.

A menos que o tipo de origem ou destino seja void, todas as conversões de ponteiro entre ponteiros com tipo base diferente deve ser explícito.

Ou, em outras palavras, eles precisam de um cast, a menos que você faça uma conversão que apenas adicione um qualificador como const ao tipo de destino.

Evitar conversões de ponteiro em geral e casts em particular protege você de problemas de alias. A menos que você realmente precise deles, e esses casos são muito especiais, você deve evitá-los ao máximo.

Tipo efetivo #

O tipo efetivo de um objeto de dados é a última informação de tipo que foi associada a ele, se houver.

// a normal variable, effective type uint32_t, and this type never changes
uint32_t a = 0.0;

// effective type of *pa is uint32_t, too, simply
// because *pa is the object a
uint32_t* pa = &a;

// the object pointed to by q has no effective type, yet
void* q = malloc(sizeof uint32_t);
// the object pointed to by q still has no effective type,
// because nobody has written to it
uint32_t* qb = q;
// *qb now has effective type uint32_t because a uint32_t value was written
*qb = 37;

// the object pointed to by r has no effective type, yet, although
// it is initialized
void* r = calloc(1, sizeof uint32_t);
// the object pointed to by r still has no effective type,
// because nobody has written to or read from it
uint32_t* rc = r;
// *rc now has effective type uint32_t because a value is read
// from it with that type. The read operation is valid because we used calloc.
// Now the object pointed to by r (which is the same as *rc) has
// gained an effective type, although we didn't change its value.
uint32_t c = *rc;

// the object pointed to by s has no effective type, yet.
void* s = malloc(sizeof uint32_t);
// the object pointed to by s now has effective type uint32_t
// because an uint32_t value is copied into it.
memcpy(s, r, sizeof uint32_t);

Observe que para este último, não era necessário que tivéssemos sequer um ponteiro uint32_t* para esse objeto. O fato de termos copiado outro objeto uint32_t é suficiente.

restrição de qualificação #

Se tivermos dois argumentos de ponteiro do mesmo tipo, o compilador não pode fazer nenhuma suposição e sempre terá que assumir que a mudança para *e pode mudar para *f:

void fun(float* e, float* f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

float fval = 4;
float eval = 77;
 fun(&eval, &fval);

tudo vai bem e algo como

4 é igual a 4?

é impresso. Se passarmos o mesmo ponteiro, o programa ainda fará a coisa certa e imprimirá

4 é igual a 22?

Isso pode se tornar ineficiente, se sabemos por alguma informação externa que e e f nunca apontarão para o mesmo objeto de dados. Podemos refletir esse conhecimento adicionando qualificadores restrict aos parâmetros do ponteiro:

void fan(float*restrict e, float*restrict f) {
    float a = *f
    *e = 22;
    float b = *f;
    print("is %g equal to %g?\n", a, b);
}

Então o compilador pode sempre supor que e e f apontam para objetos diferentes.

Alterando bytes #

Uma vez que um objeto tenha um tipo efetivo, você não deve tentar modificá-lo através de um ponteiro de outro tipo, a menos que esse outro tipo seja um tipo de caractere, char, signed char ou unsigned char.

#include <inttypes.h>
#include <stdio.h>

int main(void) {
  uint32_t a = 57;
  // conversion from incompatible types needs a cast !
  unsigned char* ap = (unsigned char*)&a;
  for (size_t i = 0; i < sizeof a; ++i) {
    /* set each byte of a to 42 */
    ap[i] = 42;
  }
  printf("a now has value %" PRIu32 "\n", a);
}

Este é um programa válido que imprime

a agora tem valor 707406378

Isso funciona porque:

  • O acesso é feito aos bytes individuais vistos com o tipo unsigned char para que cada modificação seja bem definida.
  • As duas visualizações para o objeto, por meio de a e por meio de *ap, alias, mas como ap é um ponteiro para um tipo de caractere, a regra de alias estrita não se aplica. Assim, o compilador deve assumir que o valor de a pode ter sido alterado no loop for. O valor modificado de a deve ser construído a partir dos bytes que foram alterados.
  • O tipo de a, uint32_t não possui bits de preenchimento. Todos os seus bits da representação contam para o valor, aqui 707406378, e não pode haver representação de trap.

Tipos de caracteres não podem ser acessados ​​por meio de tipos que não sejam de caracteres. #

Se um objeto for definido com duração de armazenamento estática, thread ou automática e tiver um tipo de caractere: char, unsigned char ou signed char, ele não poderá ser acessado por um tipo que não seja de caractere . No exemplo abaixo, um array char é reinterpretado como o tipo int, e o comportamento é indefinido em cada desreferência do ponteiro int b.

int main( void )
{
    char a[100];
    int* b = ( int* )&a;
    *b = 1;      

    static char c[100];
    b = ( int* )&c;
    *b = 2;

    _Thread_local char d[100];
    b = ( int* )&d;
    *b = 3;
}

Isso é indefinido porque viola a regra do “tipo efetivo”, nenhum objeto de dados que tenha um tipo efetivo pode ser acessado por meio de outro tipo que não seja um tipo de caractere. Como o outro tipo aqui é int, isso não é permitido.

Mesmo se o alinhamento e os tamanhos de ponteiro fossem adequados, isso não estaria isento dessa regra, o comportamento ainda seria indefinido.

Isso significa, em particular, que não há como no padrão C reservar um objeto buffer do tipo caractere que pode ser usado por meio de ponteiros com tipos diferentes, como você usaria um buffer que foi recebido por malloc ou função semelhante.

Uma maneira correta de atingir o mesmo objetivo do exemplo acima seria ser usar uma união.

typedef union bufType bufType;
union bufType {
   char c[sizeof(int[25])];
   int i[25];
};

int main( void )
{
    bufType a = { .c = { 0 } }; // reserve a buffer and initialize
    int* b = a.i;      // no cast necessary
    *b = 1;      

    static bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 2;

    _Thread_local bufType a = { .c = { 0 } };
    int* b = a.i;
    *b = 3;
}

Aqui, o union garante que o compilador saiba desde o início que o buffer pode ser acessados ​​por diferentes visões. Isso também tem a vantagem de que agora o buffer tem uma “view” a.i que já é do tipo int e nenhuma conversão de ponteiro é necessária.

Violação das regras estritas de alias #

No código a seguir vamos supor para simplificar que float e uint32_t tenham o mesmo tamanho.

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    float b = *f;
    print("%g should equal %g\n", a, b);
}

u e f têm diferentes tipos de base e, portanto, o compilador pode assumir que eles apontam para objetos diferentes. Não há possibilidade de que *f possa ter mudado entre as duas inicializações de a e b, e assim o compilador pode otimizar o código para algo equivalente a

void fun(uint32_t* u, float* f) {
    float a = *f
    *u = 22;
    print("%g should equal %g\n", a, a);
}

Ou seja, a segunda operação de carregamento de *f pode ser otimizada completamente.

Se chamarmos esta função “normalmente”

 float fval = 4;
 uint32_t uval = 77;
 fun(&uval, &fval);

tudo vai bem e algo como

4 deve ser igual a 4

é impresso. Mas se trapacearmos e passarmos o mesmo ponteiro, depois de convertê-lo,

 float fval = 4;
 uint32_t* up = (uint32_t*)&fval;
 fun(up, &fval);

violamos a regra de aliasing estrita. Então o comportamento se torna indefinido. A saída poderia ser como acima, se o compilador tivesse otimizado o segundo acesso, ou algo completamente diferente, e assim seu programa acaba em um estado completamente não confiável.
