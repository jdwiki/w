Uma asserção é um predicado de que a condição apresentada deve ser verdadeira no momento em que a asserção é encontrada pelo software. As mais comuns são as asserções simples, que são validadas em tempo de execução. No entanto, asserções estáticas são verificadas em tempo de compilação.

Sintaxe #

  • assert(expressão)
  • static_assert(expressão, mensagem)
  • _Static_assert(expressão, mensagem)

Parâmetros #

Parâmetro Detalhes
expressão expressão do tipo escalar.
mensagem literal de string a ser incluído na mensagem de diagnóstico.

Ambos assert e static_assert são macros definidas em assert.h.

A definição de assert depende da macro NDEBUG que não é definida pela biblioteca padrão. Se NDEBUG for definido, assert não será operacional:

#ifdef NDEBUG
#  define assert(condition) ((void) 0)
#else
#  define assert(condition) /* implementation defined */
#endif

A opinião varia sobre se o NDEBUG deve sempre ser usado para compilações de produção.

  • O pró-camp argumenta que assert chama abort e mensagens de assertion não são úteis para usuários finais, então o resultado não é útil para o usuário. Se você tiver condições fatais para verificar no código de produção, você deve usar as condições normais if/else e ​​exit ou quick_exit para encerrar o programa. Em contraste com abort, estes permitem que o programa faça alguma limpeza (através de funções registradas com atexit ou at_quick_exit).
  • O con-camp argumenta que as chamadas assert nunca devem ser acionadas no código de produção, mas se o fizerem, a condição verificada significa que há algo dramaticamente errado e o programa se comportará mal se a execução continuar. Portanto, é melhor ter as asserções ativas no código de produção, porque se elas dispararem, o inferno já está solto.
  • Outra opção é usar um sistema de asserções caseiro que sempre executa a verificação, mas trata os erros de maneira diferente entre desenvolvimento (onde abort é apropriado) e produção (onde um ’erro interno inesperado - entre em contato com o suporte técnico’ pode ser mais apropriado).

static_assert se expande para _Static_assert que é uma palavra-chave. A condição é verificada em tempo de compilação, portanto, condition deve ser uma expressão constante. Não há necessidade de que isso seja tratado de forma diferente entre desenvolvimento e produção.

Asserção Simples #

Uma asserção é uma declaração usada para afirmar que um fato deve ser verdadeiro quando essa linha de código é alcançada. Asserções são úteis para garantir que as condições esperadas sejam atendidas. Quando a condição passada para uma asserção é verdadeira, não há ação. O comportamento em condições falsas depende dos sinalizadores do compilador. Quando as asserções estão habilitadas, uma entrada falsa causa uma parada imediata do programa. Quando eles estão desabilitados, nenhuma ação é tomada. É uma prática comum habilitar asserções em compilações internas e de depuração e desativá-las em compilações de versão, embora as asserções sejam frequentemente habilitadas na versão. (Se a terminação é melhor ou pior do que os erros depende do programa.) Asserções devem ser usadas apenas para detectar erros internos de programação, o que geralmente significa passar parâmetros incorretos.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int main(void)
{
    int x = -1;
    assert(x >= 0);

    printf("x = %d\n", x);   
    return 0;
}

Saída possível com NDEBUG indefinido:

a.out: main.c:9: main: Assertion `x >= 0' failed.

Saída possível com NDEBUG definido:

x = -1

É uma boa prática definir NDEBUG globalmente, para que você possa compilar facilmente seu código com todas as asserções ativadas ou desativadas. Uma maneira fácil de fazer isso é definir NDEBUG como uma opção para o compilador, ou defini-lo em um cabeçalho de configuração compartilhado (por exemplo, config.h).

Asserção estática #

Asserções estáticas são usadas para verificar se uma condição é verdadeira quando o código é compilado. Se não for, o compilador deverá emitir uma mensagem de erro e interromper o processo de compilação.

Uma asserção estática é aquela que é verificada em tempo de compilação, não em tempo de execução. A condição deve ser uma expressão constante e, se for falsa, resultará em um erro do compilador. O primeiro argumento, a condição que é verificada, deve ser uma expressão constante e o segundo, uma string literal.

Ao contrário de assert, _Static_assert é uma palavra-chave. Uma macro de conveniência static_assert é definida em <assert.h>.

#include <assert.h>

enum {N = 5};
_Static_assert(N == 5, "N does not equal 5");
static_assert(N > 10, "N is not greater than 10");  /* compiler error */

Antes do C11, não havia suporte direto para asserções estáticas. No entanto, em C99, asserções estáticas podem ser emuladas com macros que acionariam uma falha de compilação se a condição de tempo de compilação fosse falsa. Ao contrário de _Static_assert, o segundo parâmetro precisa ser um nome de token adequado para que um nome de variável possa ser criado com ele. Se a asserção falhar, o nome da variável é visto no erro do compilador, pois essa variável foi usada em uma declaração de array sintaticamente incorreta.

#define STATIC_MSG(msg, l) STATIC_MSG2(msg, l)
#define STATIC_MSG2(msg,l) on_line_##l##__##msg
#define STATIC_ASSERT(x, msg) extern char STATIC_MSG(msg, __LINE__) [(x)?1:-1]
 
enum { N = 5 };
STATIC_ASSERT(N == 5, N_must_equal_5);
STATIC_ASSERT(N > 5, N_must_be_greater_than_5); /* compile error */

Antes de C99, você não podia declarar variáveis ​​em locais arbitrários em um bloco, então você teria que ser extremamente cauteloso ao usar essa macro, garantindo que ela aparecesse apenas onde uma declaração de variável fosse válida.

Afirmar mensagens de erro #

Existe um truque que pode exibir uma mensagem de erro junto com uma asserção. Normalmente, você escreveria código assim

void f(void *p)
{
    assert(p != NULL);
    /* more code */
}

Se a asserção falhou, uma mensagem de erro seria semelhante

Falha na declaração: p != NULL, arquivo main.c, linha 5

No entanto, você pode usar o AND lógico (&&) para fornecer uma mensagem de erro também

void f(void *p)
{
    assert(p != NULL && "function f: p cannot be NULL");
    /* more code */
}

Agora, se a asserção falhar, uma mensagem de erro lerá algo assim

Falha na asserção: p != NULL && “função f: p não pode ser NULL”, arquivo main.c, linha 5

A razão pela qual isso funciona é que um literal de string sempre é avaliado como diferente de zero (verdadeiro). Adicionar && 1 a uma expressão booleana não tem efeito. Assim, adicionar && "mensagem de erro" também não tem efeito, exceto que o compilador exibirá toda a expressão que falhou.

Declaração de código inacessível #

Durante o desenvolvimento, quando certos caminhos de código devem ser impedidos do alcance do fluxo de controle, você pode usar assert(0) para indicar que tal condição está errada:

switch (color) {
    case COLOR_RED:
    case COLOR_GREEN:
    case COLOR_BLUE:
        break;

    default:
        assert(0);
}

Sempre que o argumento da macro assert() for avaliado como falso, a macro gravará informações de diagnóstico no fluxo de erro padrão e, em seguida, abortará o programa. Esta informação inclui o arquivo e o número da linha da instrução assert() e pode ser muito útil na depuração. Asserts podem ser desabilitadas definindo a macro NDEBUG.

Outra maneira de encerrar um programa quando ocorre um erro é com as funções de biblioteca padrão exit, quick_exit ou abort. exit e quick_exit recebem um argumento que pode ser passado de volta para o seu ambiente. abort() (e, portanto, assert) pode ser uma terminação muito severa do seu programa, e certas limpezas que de outra forma seriam realizadas no final da execução, podem não ser executadas.

A principal vantagem do assert() é que ele imprime automaticamente as informações de depuração. Chamar abort() tem a vantagem de não poder ser desabilitado como um assert, mas pode não fazer com que nenhuma informação de depuração seja exibida. Em algumas situações, usar as duas construções juntas pode ser benéfico:

if (color == COLOR_RED || color == COLOR_GREEN) {
   ...
} else if (color == COLOR_BLUE) {
   ...
} else {
   assert(0), abort();
}

Quando as assertivas estiverem ativadas, a chamada assert() imprimirá informações de depuração e encerrará o programa. A execução nunca atinge a chamada abort(). Quando as assertivas estão desativadas, a chamada assert() não faz nada e abort() é chamada. Isso garante que o programa sempre termine para esta condição de erro; habilitar e desabilitar asserções só afeta se a saída de depuração é impressa ou não.

Você nunca deve deixar tal assert no código de produção, porque as informações de depuração não são úteis para usuários finais e porque abort geralmente é uma terminação muito severa que inibe os manipuladores de limpeza instalados para exit ou quick_exit para correr.

Pré-condição e Pós-condição #

Um caso de uso para asserção é a pré-condição e a pós-condição. Isso pode ser muito útil para manter invariante e design por contrato. Por exemplo, um comprimento é sempre zero ou positivo, portanto, esta função deve retornar um valor zero ou positivo.

#include <stdio.h>
/* Uncomment to disable `assert()` */
/* #define NDEBUG */
#include <assert.h>

int length2 (int *a, int count)
{
    int i, result = 0;

    /* Precondition: */
    /* NULL is an invalid vector */
    assert (a != NULL);
    /* Number of dimensions can not be negative.*/ 
    assert (count >= 0);

    /* Calculation */
    for (i = 0; i < count; ++i) 
    {
        result = result + (a[i] * a[i]);
    }

    /* Postcondition: */
    /* Resulting length can not be negative. */
    assert (result >= 0);
    return result;
}

#define COUNT 3

int main (void)
{
    int a[COUNT] = {1, 2, 3};
    int *b = NULL;
    int r;
    r = length2 (a, COUNT);
    printf ("r = %i\n", r);
    r = length2 (b, COUNT);
    printf ("r = %i\n", r);
    return 0;
}
