Para gerenciar memória alocada dinamicamente, a biblioteca C padrão fornece as funções malloc(), calloc(), realloc() e free(). No C99 e posterior, também existe o aligned_alloc(). Alguns sistemas também fornecem alloca().

Sintaxe #

  • void *aligned_alloc(alinhamento tamanho_t, tamanho_t tamanho); /* Somente desde C11 */
  • void *calloc(size_t elementos, size_t size);
  • void free(void *ptr);
  • void *malloc(tamanho_t tamanho);
  • void *realloc(void *ptr, tamanho_t tamanho);
  • void *alloca(tamanho_t tamanho); /* de alloca.h, não padrão, não portátil, perigoso. */

Parâmetros #

nome descrição
size (malloc, realloc e aligned_alloc) tamanho total da memória em bytes. Para aligned_alloc o tamanho deve ser um múltiplo inteiro do alinhamento.
tamanho (calloc) tamanho de cada elemento
elementos número de elementos
ptr ponteiro para memória alocada anteriormente retornado por malloc, calloc, realloc ou aligned_alloc
alinhamento alinhamento da memória alocada

Observe que aligned_alloc() só é definido para C11 ou posterior.

Sistemas como os baseados em POSIX fornecem outras maneiras de alocar memória alinhada (por exemplo, posix_memalign()) , e também tem outras opções de gerenciamento de memória (por exemplo, mmap()).

Liberando memória #

É possível liberar memória alocada dinamicamente chamando free().

int *p = malloc(10 * sizeof *p); /* allocation of memory */
if (p == NULL) 
{
    perror("malloc failed");
    return -1;
}

free(p); /* release of memory */
/* note that after free(p), even using the *value* of the pointer p
   has undefined behavior, until a new value is stored into it. */

/* reusing/re-purposing the pointer itself */
int i = 42;
p = &i; /* This is valid, has defined behaviour */

A memória apontada por p é recuperada (seja pela implementação da libc ou pelo sistema operacional subjacente) após a chamada para free(), portanto, acessar esse bloco de memória liberado via p levará a comportamento indefinido. Ponteiros que fazem referência a elementos de memória que foram liberados são comumente chamados de ponteiros pendentes e apresentam um risco de segurança. Além disso, o padrão C afirma que mesmo [acessando o valor](https://www.wikiod.com/pt/c/comportamento-indefinido#Valor de leitura do ponteiro que foi liberado 201608050732042665317) de um ponteiro pendente tem comportamento indefinido. Observe que o próprio ponteiro p pode ser redefinido como mostrado acima.

Observe que você só pode chamar free() em ponteiros que foram retornados diretamente das funções malloc(), calloc(), realloc() e aligned_alloc(), ou onde a documentação diz você a memória foi alocada dessa forma (funções como strdup() são exemplos notáveis). Liberando um ponteiro que é,

  • obtido usando o operador & em uma variável, ou
  • no meio de um bloco alocado,

é proibido. Esse erro geralmente não será diagnosticado pelo seu compilador, mas levará a execução do programa a um estado indefinido.

Existem duas estratégias comuns para evitar tais instâncias de comportamento indefinido.

A primeira e preferível é simples - faça com que o próprio p deixe de existir quando não for mais necessário, por exemplo:

if (something_is_needed())
{

    int *p = malloc(10 * sizeof *p);
    if (p == NULL) 
    {
        perror("malloc failed");
        return -1;
    }

    /* do whatever is needed with p */

    free(p);
}

Ao chamar free() diretamente antes do final do bloco que o contém (ou seja, o }), o próprio p deixa de existir. O compilador dará um erro de compilação em qualquer tentativa de usar p depois disso.

Uma segunda abordagem é também invalidar o próprio ponteiro após liberar a memória para a qual ele aponta:

free(p);
p = NULL;     // you may also use 0 instead of NULL

Argumentos para esta abordagem:

  • Em muitas plataformas, uma tentativa de desreferenciar um ponteiro nulo causará uma falha instantânea: Falha de segmentação. Aqui, obtemos pelo menos um rastreamento de pilha apontando para a variável que foi usada após ser liberada.

Sem definir ponteiro para NULL temos ponteiro pendente. O programa provavelmente ainda travará, mas mais tarde, porque a memória para a qual o ponteiro aponta será corrompida silenciosamente. Esses bugs são difíceis de rastrear porque podem resultar em uma pilha de chamadas completamente não relacionada ao problema inicial.

Esta abordagem, portanto, segue o [conceito de falha rápida] 3.

  • É seguro liberar um ponteiro nulo. O C Standard especifica que free(NULL) não tem efeito:

A função free faz com que o espaço apontado por ptr seja desalocado, ou seja, disponibilizado para posterior alocação. Se ptr for um ponteiro nulo, nenhuma ação ocorrerá. Caso contrário, se o argumento não corresponder a um ponteiro retornado anteriormente pela função calloc, malloc ou realloc, ou se o espaço foi desalocado por uma chamada para free ou realloc, o comportamento é Indefinido.

  • Às vezes, a primeira abordagem não pode ser usada (por exemplo, a memória é alocada em uma função e desalocada muito mais tarde em uma função completamente diferente)

5: https://www.wikiod.com/pt/c/ponteiros#Erros comuns

Alocando memória #

Alocação padrão #

As funções de alocação de memória dinâmica C são definidas no cabeçalho <stdlib.h>. Se alguém deseja alocar espaço de memória para um objeto dinamicamente, o seguinte código pode ser usado:

int *p = malloc(10 * sizeof *p);
if (p == NULL) 
{
    perror("malloc() failed");
    return -1;
}

Isso calcula o número de bytes que dez ints ocupam na memória, então solicita esses muitos bytes de malloc e atribui o resultado (ou seja, o endereço inicial do bloco de memória que acabou de ser criado usando malloc) a um ponteiro chamado p.

É uma boa prática usar sizeof para calcular a quantidade de memória a ser solicitada, pois o resultado de sizeof é definido pela implementação (exceto para tipos de caracteres, que são char, signed char e unsigned char , para o qual sizeof é definido para sempre fornecer 1).

Como malloc pode não ser capaz de atender a solicitação, ele pode retornar um ponteiro nulo. É importante verificar isso para evitar tentativas posteriores de desreferenciar o ponteiro nulo.

A memória alocada dinamicamente usando malloc() pode ser redimensionada usando realloc() ou, quando não for mais necessária, liberada usando free().

Alternativamente, declarar int array[10]; alocaria a mesma quantidade de memória. No entanto, se ele for declarado dentro de uma função sem a palavra-chave static, ele só será utilizável dentro da função em que foi declarado e das funções que ele chama (porque o array será alocado na pilha e o espaço será liberado para reutilizar quando a função retornar). Alternativamente, se for definido com static dentro de uma função, ou se for definido fora de qualquer função, então seu tempo de vida é o tempo de vida do programa. Ponteiros também podem ser retornados de uma função, porém uma função em C não pode retornar um array.

Memória zerada #

A memória retornada por malloc pode não ser inicializada com um valor razoável, e deve-se tomar cuidado para zerar a memória com memset ou copiar imediatamente um valor adequado para ela. Alternativamente, calloc retorna um bloco do tamanho desejado onde todos os bits são inicializados em 0. Isso não precisa ser o mesmo que a representação de zero de ponto flutuante ou uma constante de ponteiro nulo.

int *p = calloc(10, sizeof *p);
if (p == NULL) 
{
    perror("calloc() failed");
    return -1;
}

Uma nota sobre calloc: A maioria das implementações (comumente usadas) otimizará calloc() para desempenho, então será mais rápido do que chamar malloc(), então memset( ), mesmo que o efeito líquido seja idêntico.

Memória Alinhada #

C11 introduziu uma nova função aligned_alloc() que aloca espaço com o alinhamento fornecido. Ele pode ser usado se a memória a ser alocada precisar ser alinhada em certos limites que não podem ser atendidos por malloc() ou calloc(). As funções malloc() e calloc() alocam memória adequadamente alinhada para qualquer tipo de objeto (ou seja, o alinhamento é alignof(max_align_t)). Mas com aligned_alloc() alinhamentos maiores podem ser solicitados.

/* Allocates 1024 bytes with 256 bytes alignment. */
char *ptr = aligned_alloc(256, 1024);
if (ptr) {
    perror("aligned_alloc()");
    return -1;
}
free(ptr);

O padrão C11 impõe duas restrições: 1) o size (segundo argumento) solicitado deve ser um múltiplo inteiro do alignment (primeiro argumento) e 2) o valor de alignment deve ser um alinhamento válido suportado pela implementação . A falha em atender a qualquer um deles resulta em comportamento indefinido.

Realocação de memória #

Pode ser necessário expandir ou reduzir o espaço de armazenamento do ponteiro após ter alocado memória para ele. A função void *realloc(void *ptr, size_t size) desaloca o objeto antigo apontado por ptr e retorna um ponteiro para um objeto que tem o tamanho especificado por size. ptr é o ponteiro para um bloco de memória previamente alocado com malloc, calloc ou realloc (ou um ponteiro nulo) a ser realocado. O conteúdo máximo possível da memória original é preservado. Se o novo tamanho for maior, qualquer memória adicional além do tamanho antigo não será inicializada. Se o novo tamanho for menor, o conteúdo da parte encolhida será perdido. Se ptr for NULL, um novo bloco é alocado e um ponteiro para ele é retornado pela função.

#include <stdio.h>
#include <stdlib.h>
 
int main(void)
{
    int *p = malloc(10 * sizeof *p);
    if (NULL == p) 
    {
        perror("malloc() failed");
        return EXIT_FAILURE;
    }
 
    p[0] = 42;
    p[9] = 15;

    /* Reallocate array to a larger size, storing the result into a
     * temporary pointer in case realloc() fails. */
    {
        int *temporary = realloc(p, 1000000 * sizeof *temporary);

        /* realloc() failed, the original allocation was not free'd yet. */
        if (NULL == temporary)
        {
            perror("realloc() failed");
            free(p); /* Clean up. */
            return EXIT_FAILURE;
        }      

        p = temporary;
    }

    /* From here on, array can be used with the new size it was 
     * realloc'ed to, until it is free'd. */

    /* The values of p[0] to p[9] are preserved, so this will print:
       42 15
    */
    printf("%d %d\n", p[0], p[9]);

    free(p);

    return EXIT_SUCCESS;
}

O objeto realocado pode ou não ter o mesmo endereço que *p. Portanto, é importante capturar o valor de retorno de realloc que contém o novo endereço se a chamada for bem-sucedida.

Certifique-se de atribuir o valor de retorno de realloc a um temporary em vez do p original. realloc retornará null em caso de qualquer falha, o que substituiria o ponteiro. Isso perderia seus dados e criaria um vazamento de memória.

realloc(ptr, 0) não é equivalente a free(ptr) #

realloc é conceitualmente equivalente a malloc + memcpy + free no outro ponteiro.

Se o tamanho do espaço solicitado for zero, o comportamento de realloc é definido pela implementação. Isso é semelhante para todas as funções de alocação de memória que recebem um parâmetro size de valor 0. Tais funções podem de fato retornar um ponteiro não nulo, mas isso nunca deve ser desreferenciado.

Assim, realloc(ptr,0) não é equivalente a free(ptr). Pode

  • seja uma implementação “preguiçosa” e apenas retorne ptr
  • free(ptr), aloque um elemento fictício e retorne esse
  • free(ptr) e retorne 0
  • apenas retorne 0 para falha e não faça mais nada.

Assim, em particular, os dois últimos casos são indistinguíveis pelo código do aplicativo.

Isso significa que realloc(ptr,0) não pode realmente liberar/desalocar a memória e, portanto, nunca deve ser usado como substituto para free.

Arrays multidimensionais de tamanho variável #

Desde C99, C tem arrays de comprimento variável, VLA, que modelam arrays com limites que são conhecidos apenas no momento da inicialização. Embora você tenha que ter cuidado para não alocar VLA muito grande (eles podem esmagar sua pilha), usar ponteiros para VLA e usá-los em expressões sizeof é bom.

double sumAll(size_t n, size_t m, double A[n][m]) {
    double ret = 0.0;
    for (size_t i = 0; i < n; ++i)
       for (size_t j = 0; j < m; ++j)
          ret += A[i][j]
    return ret;
}

int main(int argc, char *argv[argc+1]) {
   size_t n = argc*10;
   size_t m = argc*8;
   double (*matrix)[m] = malloc(sizeof(double[n][m]));
   // initialize matrix somehow
   double res = sumAll(n, m, matrix);
   printf("result is %g\n", res);
   free(matrix);
}

Aqui matrix é um ponteiro para elementos do tipo double[m], e a expressão sizeof com double[n][m] garante que ela contenha espaço para n tais elementos.

Todo este espaço é alocado de forma contígua e pode assim ser desalocado por uma única chamada para free.

A presença de VLA na linguagem também afeta as possíveis declarações de arrays e ponteiros em cabeçalhos de função. Agora, uma expressão inteira geral é permitida dentro do [] dos parâmetros do array. Para ambas as funções, as expressões em [] usam parâmetros que foram declarados antes na lista de parâmetros. Para sumAll estes são os comprimentos que o código do usuário espera para a matriz. Como para todos os parâmetros de função de matriz em C, a dimensão mais interna é reescrita para um tipo de ponteiro, portanto, isso é equivalente à declaração

  double sumAll(size_t n, size_t m, double (*A)[m]);

Ou seja, n não é realmente parte da interface da função, mas a informação pode ser útil para documentação e também pode ser usada por compiladores de verificação de limites para alertar sobre acesso fora dos limites.

Da mesma forma, para main, a expressão argc+1 é o comprimento mínimo que o padrão C prescreve para o argumento argv.

Observe que oficialmente o suporte a VLA é opcional em C11, mas não conhecemos nenhum compilador que implemente C11 e que não os tenha. Você pode testar com a macro __STDC_NO_VLA__ se precisar.

alloca: aloca memória na pilha #

Aviso: alloca só é mencionado aqui para ser completo. É totalmente não portátil (não coberto por nenhum dos padrões comuns) e possui vários recursos potencialmente perigosos que o tornam inseguro para quem não sabe. O código C moderno deve substituí-lo por Variable Length Arrays (VLA).

Página do manual

#include <alloca.h>
// glibc version of stdlib.h include alloca.h by default

void foo(int size) {
    char *data = alloca(size);
    /*
      function body;
    */
    // data is automatically freed
}

Aloque memória no quadro de pilha do chamador, o espaço referenciado pelo ponteiro retornado é automaticamente livre’d quando a função do chamador termina.

Embora esta função seja conveniente para gerenciamento automático de memória, esteja ciente de que solicitar uma alocação grande pode causar um estouro de pilha e que você não pode usar free com memória alocada com alloca (o que pode causar mais problemas com estouro de pilha).

Por esta razão não é recomendado usar alloca dentro de um loop nem de uma função recursiva.

E porque a memória é free’d no retorno da função, você não pode retornar o ponteiro como resultado da função (o comportamento seria indefinido).

Resumo #

  • chamada idêntica a malloc
  • automaticamente liberado após o retorno da função
  • incompatível com funções free,realloc (undefined behavior)
  • ponteiro não pode ser retornado como resultado de uma função (comportamento indefinido)
  • tamanho de alocação limitado pelo espaço de pilha, que (na maioria das máquinas) é muito menor que o espaço de heap disponível para uso por malloc()
  • evite usar alloca() e VLAs (matrizes de comprimento variável) em uma única função
  • alloca() não é tão portátil quanto malloc() et al

Recomendação #

  • Não use alloca() no novo código

Alternativa moderna.

void foo(int size) {
    char data[size];
    /*
      function body;
    */
    // data is automatically freed
}

Isso funciona onde alloca() funciona, e funciona em lugares onde alloca() não funciona (loops internos, por exemplo). Ele assume uma implementação C99 ou uma implementação C11 que não define __STDC_NO_VLA__.

Gerenciamento de memória definido pelo usuário #

malloc() geralmente chama funções subjacentes do sistema operacional para obter páginas de memória. Mas não há nada de especial sobre a função e ela pode ser implementada diretamente em C declarando uma grande matriz estática e alocando a partir dela (há uma pequena dificuldade em garantir o alinhamento correto, na prática alinhar a 8 bytes é quase sempre adequado).

Para implementar um esquema simples, um bloco de controle é armazenado na região da memória imediatamente antes do ponteiro a ser retornado da chamada. Isso significa que free() pode ser implementado subtraindo do ponteiro retornado e lendo as informações de controle, que normalmente são o tamanho do bloco mais algumas informações que permitem que ele seja colocado de volta na lista livre - uma lista vinculada de não alocados blocos.

Quando o usuário solicita uma alocação, a lista livre é pesquisada até que seja encontrado um bloco de tamanho idêntico ou maior ao valor solicitado e, se necessário, é dividido. Isso pode levar à fragmentação da memória se o usuário estiver continuamente fazendo muitas alocações e liberações de tamanho imprevisível e em intervalos imprevisíveis (nem todos os programas reais se comportam assim, o esquema simples geralmente é adequado para programas pequenos).

/* typical control block */
struct block
{
   size_t size;         /* size of block */
   struct block *next;  /* next block in free list */ 
   struct block *prev;  /* back pointer to previous block in memory */
   void *padding;       /* need 16 bytes to make multiple of 8 */
}

static struct block arena[10000]; /* allocate from here */
static struct block *firstfree;

Muitos programas requerem um grande número de alocações de pequenos objetos do mesmo tamanho. Isso é muito fácil de implementar. Basta usar um bloco com um próximo ponteiro. Portanto, se for necessário um bloco de 32 bytes:

union block
{
   union block * next;
   unsigned char payload[32];
}  

static union block arena[100];
static union block * head; 
void init(void)
{
    int i;
    for (i = 0; i < 100 - 1; i++)
        arena[i].next = &arena[i + 1];
    arena[i].next = 0; /* last one, null */
    head = &block[0];
}
 
void *block_alloc()
{
    void *answer = head;
    if (answer)
        head = head->next;
    return answer;
}

void block_free(void *ptr)
{
    union block *block = ptr;
    block->next = head;
    head - block;
}

Este esquema é extremamente rápido e eficiente, podendo ser tornado genérico com certa perda de clareza.
