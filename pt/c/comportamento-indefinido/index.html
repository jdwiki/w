Em C, algumas expressões geram comportamento indefinido. O padrão opta explicitamente por não definir como um compilador deve se comportar se encontrar tal expressão. Como resultado, um compilador é livre para fazer o que achar melhor e pode produzir resultados úteis, resultados inesperados ou até mesmo travar.

O código que invoca UB pode funcionar como pretendido em um sistema específico com um compilador específico, mas provavelmente não funcionará em outro sistema ou com um compilador, versão do compilador ou configurações do compilador diferentes.

O que é comportamento indefinido (UB)?

Comportamento indefinido é um termo usado no padrão C. A norma C11 (ISO/IEC 9899:2011) define o termo comportamento indefinido como

comportamento, mediante o uso de uma construção de programa não portátil ou errônea ou de dados errôneos, para os quais esta Norma não impõe requisitos

O que acontece se houver UB no meu código?

Estes são os resultados que podem ocorrer devido ao comportamento indefinido de acordo com o padrão:

NOTA Possíveis comportamentos indefinidos vão desde ignorar a situação completamente com resultados imprevisíveis, para se comportar durante a tradução ou execução do programa de forma documentada característica do ambiente (com ou sem emissão de mensagem de diagnóstico), para encerrar uma tradução ou execução (com a emissão de um mensagem de diagnóstico).

A citação a seguir é frequentemente usada para descrever (embora menos formalmente) os resultados que ocorrem a partir de um comportamento indefinido:

“Quando o compilador encontra [uma determinada construção indefinida] é legal fazer demônios sair do seu nariz” (a implicação é que o compilador pode escolher qualquer maneira arbitrariamente bizarra de interpretar o código sem violar o padrão ANSI C)

Por que o UB existe?

Se é tão ruim, por que eles simplesmente não o definiram ou o definiram para implementação?

Comportamento indefinido permite mais oportunidades de otimização; O compilador pode, justificadamente, assumir que qualquer código não contém comportamento indefinido, o que pode permitir que ele evite verificações em tempo de execução e realize otimizações cuja validade seria dispendiosa ou impossível de provar o contrário.

Por que é difícil rastrear UB?

Há pelo menos duas razões pelas quais o comportamento indefinido cria bugs difíceis de detectar:

  • O compilador não é obrigado a - e geralmente não pode de forma confiável - avisá-lo sobre comportamento indefinido. Na verdade, exigir que o faça iria diretamente contra a razão da existência de comportamento indefinido.
  • Os resultados imprevisíveis podem não começar a se desdobrar no ponto exato da operação onde ocorre o construto cujo comportamento é indefinido; O comportamento indefinido mancha toda a execução e seus efeitos podem ocorrer a qualquer momento: durante, depois ou mesmo antes da construção indefinida.

Considere a desreferência de ponteiro nulo: o compilador não precisa diagnosticar a desreferência de ponteiro nulo e nem poderia, pois em tempo de execução qualquer ponteiro passado para uma função ou em uma variável global pode ser nulo. E quando ocorre a desreferência do ponteiro nulo, o padrão não exige que o programa precise travar. Em vez disso, o programa pode travar mais cedo, mais tarde ou não travar de forma alguma; ele pode até se comportar como se o ponteiro nulo apontasse para um objeto válido e se comportar de maneira completamente normal, apenas para travar em outras circunstâncias.

No caso de desreferência de ponteiro nulo, a linguagem C difere de linguagens gerenciadas como Java ou C#, onde o comportamento da desreferência de ponteiro nulo é definido: uma exceção é lançada, no momento exato (NullPointerException em Java, NullReferenceException em C#), assim aqueles vindos de Java ou C# podem incorretamente acreditar que em tal caso, um programa C deve travar, com ou sem a emissão de uma mensagem de diagnóstico.

Informação adicional

Existem várias dessas situações que devem ser claramente distinguidas:

  • Comportamento explicitamente indefinido, é onde o padrão C diz explicitamente que você está fora dos limites.
  • Comportamento implicitamente indefinido, onde simplesmente não há texto no padrão que preveja um comportamento para a situação em que você trouxe seu programa.

Também tenha em mente que em muitos lugares o comportamento de certas construções é deliberadamente indefinido pelo padrão C para deixar espaço para que compiladores e implementadores de bibliotecas criem suas próprias definições. Um bom exemplo são os sinais e manipuladores de sinais, onde extensões para C, como o padrão do sistema operacional POSIX, definem regras muito mais elaboradas. Nesses casos, basta verificar a documentação da sua plataforma; o padrão C não pode lhe dizer nada.

Observe também que, se o comportamento indefinido ocorrer no programa, isso não significa que apenas o ponto em que o comportamento indefinido ocorreu é problemático, mas o programa inteiro se torna sem sentido.

Por causa de tais preocupações, é importante (especialmente porque os compiladores nem sempre nos avisam sobre UB) que a pessoa que programa em C esteja pelo menos familiarizada com o tipo de coisas que desencadeiam comportamento indefinido.

Deve-se notar que existem algumas ferramentas (por exemplo, ferramentas de análise estática, como PC-Lint) que auxiliam na detecção de comportamento indefinido, mas, novamente, elas não podem detectar todas as ocorrências de comportamento indefinido.

Desreferenciando um ponteiro para variável além de seu tempo de vida #

int* foo(int bar)
{
    int baz = 6;
    baz += bar;
    return &baz; /* (&baz) copied to new memory location outside of foo. */
} /* (1) The lifetime of baz and bar end here as they have automatic storage   
   * duration (local variables), thus the returned pointer is not valid! */

int main (void)
{
    int* p;

    p = foo(5);  /* (2) this expression's behavior is undefined */
    *p = *p - 6; /* (3) Undefined behaviour here */

    return 0;
}

Alguns compiladores apontam isso de maneira útil. Por exemplo, gcc avisa com:

warning: function returns address of local variable [-Wreturn-local-addr]

e clang avisa com:

warning: address of stack memory associated with local variable 'baz' returned 
[-Wreturn-stack-address]

para o código acima. Mas os compiladores podem não ser capazes de ajudar em códigos complexos.

(1) Retornar referência à variável declarada como static é um comportamento definido, pois a variável não é destruída após deixar o escopo atual.

(2) De acordo com a ISO/IEC 9899:2011 6.2.4 §2, “O valor de um ponteiro se torna indeterminado quando o objeto para o qual aponta atinge o fim de sua vida útil.”

(3) Desreferenciar o ponteiro retornado pela função foo é um comportamento indefinido, pois a memória que ela referencia possui um valor indeterminado.

Copiando memória sobreposta #

Uma grande variedade de funções de biblioteca padrão tem entre seus efeitos copiar seqüências de bytes de uma região de memória para outra. A maioria dessas funções tem comportamento indefinido quando as regiões de origem e destino se sobrepõem.

Por exemplo, este…

#include <string.h> /* for memcpy() */

char str[19] = "This is an example";
memcpy(str + 7, str, 10);

… tenta copiar 10 bytes onde as áreas de memória de origem e destino se sobrepõem em três bytes. Vizualizar:

               overlapping area
               |
               _ _
              |   |
              v   v
T h i s   i s   a n   e x a m p l e \0
^             ^
|             |
|             destination
|
source

Devido à sobreposição, o comportamento resultante é indefinido.

Entre as funções de biblioteca padrão com uma limitação desse tipo estão memcpy(), strcpy(), strcat(), sprintf() e sscanf(). A norma diz sobre essas e várias outras funções:

Se a cópia ocorrer entre objetos que se sobrepõem, o comportamento é indefinido.

A função memmove() é a principal exceção a esta regra. Sua definição especifica que a função se comporta como se os dados de origem fossem copiados primeiro em um buffer temporário e depois gravados no endereço de destino. Não há exceção para regiões de origem e destino sobrepostas, nem qualquer necessidade de uma, então memmove() tem um comportamento bem definido nesses casos.

A distinção reflete uma eficiência vs. troca de generalidade. Copiar como essas funções executam geralmente ocorre entre regiões disjuntas da memória e, muitas vezes, é possível saber no momento do desenvolvimento se uma instância específica de cópia de memória estará nessa categoria. Assumir a não sobreposição fornece implementações comparativamente mais eficientes que não produzem resultados corretos de forma confiável quando a suposição não se sustenta. A maioria das funções da biblioteca C permite implementações mais eficientes, e memmove() preenche as lacunas, atendendo aos casos em que a origem e o destino podem ou se sobrepõem. Para produzir o efeito correto em todos os casos, no entanto, deve realizar testes adicionais e/ou empregar uma implementação comparativamente menos eficiente.

Estouro de inteiro assinado #

De acordo com o parágrafo 6.5/5 de C99 e C11, a avaliação de uma expressão produz um comportamento indefinido se o resultado não for um valor representável do tipo da expressão. Para tipos aritméticos, isso é chamado de estouro. A aritmética de inteiros sem sinal não transborda porque o parágrafo 6.2.5/9 se aplica, fazendo com que qualquer resultado sem sinal que de outra forma estaria fora do intervalo seja reduzido para um valor dentro do intervalo. No entanto, não há provisão análoga para tipos inteiros assinados; estes podem e transbordam, produzindo um comportamento indefinido. Por exemplo,

#include <limits.h>      /* to get INT_MAX */

int main(void) {
    int i = INT_MAX + 1; /* Overflow happens here */
    return 0;
}

A maioria dos casos desse tipo de comportamento indefinido é mais difícil de reconhecer ou prever. O estouro pode, em princípio, surgir de qualquer operação de adição, subtração ou multiplicação em inteiros com sinal (sujeito às conversões aritméticas usuais) onde não há limites efetivos ou uma relação entre os operandos para evitá-lo. Por exemplo, esta função:

int square(int x) {
    return x * x;  /* overflows for some values of x */
}

é razoável e faz a coisa certa para valores de argumento pequenos o suficiente, mas seu comportamento é indefinido para valores de argumento maiores. Você não pode julgar apenas pela função se os programas que a chamam exibem um comportamento indefinido como resultado. Depende de quais argumentos eles passam para ele.

Por outro lado, considere este exemplo trivial de aritmética de inteiros com sinal seguro contra estouro:

int zero(int x) {
    return x - x;  /* Cannot overflow */
}

A relação entre os operandos do operador de subtração garante que a subtração nunca transborde. Ou considere este exemplo um pouco mais prático:

int sizeDelta(FILE *f1, FILE *f2) {
    int count1 = 0;
    int count2 = 0;
    while (fgetc(f1) != EOF) count1++;  /* might overflow */
    while (fgetc(f2) != EOF) count2++;  /* might overflow */

    return count1 - count2; /* provided no UB to this point, will not overflow */
}

Desde que os contadores não transbordem individualmente, os operandos da subtração final serão ambos não negativos. Todas as diferenças entre quaisquer dois desses valores são representáveis ​​como int.

Uso de uma variável não inicializada #

int a; 
printf("%d", a);

A variável a é um int com duração de armazenamento automática. O código de exemplo acima está tentando imprimir o valor de uma variável não inicializada (a nunca foi inicializado). Variáveis ​​automáticas que não são inicializadas possuem valores indeterminados; acessá-los pode levar a um comportamento indefinido.

Observação: variáveis ​​com armazenamento local estático ou de encadeamento, incluindo variáveis ​​globais sem a palavra-chave static, são inicializadas com zero ou com seu valor inicializado. Portanto, o seguinte é legal.

static int b;
printf("%d", b);

Um erro muito comum é não inicializar as variáveis ​​que servem como contadores com 0. Você adiciona valores a elas, mas como o valor inicial é lixo, você invocará Comportamento Indefinido, como na questão Compilação no terminal emite aviso de ponteiro e símbolos estranhos.

Exemplo:

#include <stdio.h>

int main(void) {
    int i, counter;
    for(i = 0; i < 10; ++i)
        counter += i;
    printf("%d\n", counter);
    return 0;
}

Resultado:

C02QT2UBFVH6-lm:~ gsamaras$ gcc main.c -Wall -o main
main.c:6:9: warning: variable 'counter' is uninitialized when used here [-Wuninitialized]
        counter += i;
        ^~~~~~~
main.c:4:19: note: initialize the variable 'counter' to silence this warning
    int i, counter;
                  ^
                   = 0
1 warning generated.
C02QT2UBFVH6-lm:~ gsamaras$ ./main
32812

As regras acima são aplicáveis ​​para ponteiros também. Por exemplo, o seguinte resulta em comportamento indefinido

int main(void)
{
    int *p;
    p++; // Trying to increment an uninitialized pointer.
}

Observe que o código acima por si só pode não causar um erro ou falha de segmentação, mas tentar desreferenciar esse ponteiro posteriormente causaria o comportamento indefinido.

Corrida de dados #

O C11 introduziu suporte para vários threads de execução, o que permite a possibilidade de corridas de dados. Um programa contém uma corrida de dados se um objeto nele for acessado1 por dois encadeamentos diferentes, onde pelo menos um dos acessos não é atômico, pelo menos um modifica o objeto e a semântica do programa falha em certifique-se de que os dois acessos não possam se sobrepor temporalmente.2 Observe bem que a concorrência real dos acessos envolvidos não é uma condição para uma corrida de dados; as corridas de dados cobrem uma classe mais ampla de problemas decorrentes de inconsistências (permitidas) nas visões de memória de diferentes threads.

Considere este exemplo:

#include <threads.h>

int a = 0;

int Function( void* ignore )
{
    a = 1;

    return 0;
}

int main( void )
{
    thrd_t id;
    thrd_create( &id , Function , NULL );

    int b = a;

    thrd_join( id , NULL );
}

A thread principal chama thrd_create para iniciar uma nova thread executando a função Function. O segundo thread modifica a, e o thread principal lê a. Nenhum desses acessos é atômico e os dois threads não fazem nada individualmente ou em conjunto para garantir que não se sobreponham, portanto, há uma corrida de dados.

Entre as maneiras pelas quais este programa poderia evitar a corrida de dados estão

  • a thread principal poderia realizar sua leitura de a antes de iniciar a outra thread;
  • a thread principal pode realizar sua leitura de a depois de garantir via thrd_join que a outra terminou;
  • as threads podem sincronizar seus acessos através de um mutex, cada um bloqueando esse mutex antes de acessar a e desbloqueando-o depois.

Como a opção mutex demonstra, evitar uma corrida de dados não requer garantir uma ordem específica de operações, como o thread filho modificando a antes que o thread principal o leia; é suficiente (para evitar uma corrida de dados) garantir que, para uma determinada execução, um acesso aconteça antes do outro.

1 Modificando ou lendo um objeto.

2 (citado de ISO:IEC 9889:201x, seção 5.1.2.4 “Execuções multithread e corridas de dados”) A execução de um programa contém uma corrida de dados se ele contém duas ações conflitantes em threads diferentes, pelo menos uma das quais não é atômica e nenhuma acontece antes da outra. Qualquer corrida de dados resulta em comportamento indefinido.

Ler o valor do ponteiro que foi liberado #

Mesmo apenas lendo o valor de um ponteiro que foi liberado (ou seja, sem tentar desreferenciar o ponteiro) é um comportamento indefinido (UB), por exemplo

char *p = malloc(5);
free(p);
if (p == NULL) /* NOTE: even without dereferencing, this may have UB */
{

}

Citando ISO/IEC 9899:2011, seção 6.2.4 §2:

[…] O valor de um ponteiro se torna indeterminado quando o objeto que ele aponta para (ou apenas passado) chega ao fim de sua vida útil.

O uso de memória indeterminada para qualquer coisa, incluindo comparação aparentemente inofensiva ou aritmética, pode ter um comportamento indefinido se o valor puder ser uma representação de trap para o tipo.

Modificar literal de string #

Neste exemplo de código, o ponteiro char p é inicializado para o endereço de uma string literal. A tentativa de modificar o literal de string tem um comportamento indefinido.

char *p = "hello world";
p[0] = 'H'; // Undefined behavior

No entanto, modificar um array mutável de char diretamente, ou através de um ponteiro, naturalmente não é um comportamento indefinido, mesmo que seu inicializador seja uma string literal. O seguinte é bom:

char a[] = "hello, world";
char *p = a;

a[0] = 'H';
p[7] = 'W';

Isso porque o literal de string é efetivamente copiado para o array toda vez que o array é inicializado (uma vez para variáveis ​​com duração estática, cada vez que o array é criado para variáveis ​​com duração automática ou de encadeamento — variáveis ​​com duração alocada não são inicializadas) e não há problema em modificar o conteúdo do array.

Usando especificador de formato incorreto em printf #

Usar um especificador de formato incorreto no primeiro argumento para printf invoca um comportamento indefinido. Por exemplo, o código abaixo invoca um comportamento indefinido:

long z = 'B';
printf("%c\n", z);

Aqui está outro exemplo

printf("%f\n",0);

A linha de código acima é um comportamento indefinido. %f espera o dobro. No entanto, 0 é do tipo int.

Observe que seu compilador geralmente pode ajudá-lo a evitar casos como esses, se você ativar os sinalizadores apropriados durante a compilação (-Wformat em clang e gcc). Do último exemplo:

warning: format specifies type 'double' but the argument has type
      'int' [-Wformat]
    printf("%f\n",0);
            ~~    ^
            %d

Modificando qualquer objeto mais de uma vez entre dois pontos de sequência #

int i = 42;
i = i++; /* Assignment changes variable, post-increment as well */
int a = i++ + i--;

Códigos como esse geralmente levam a especulações sobre o “valor resultante” de i. Em vez de especificar um resultado, no entanto, os padrões C especificam que avaliar tal expressão produz comportamento indefinido. Antes do C2011, a norma formalizou essas regras em termos dos chamados pontos de sequência:

Entre o ponto de sequência anterior e o seguinte um objeto escalar deve ter seu valor armazenado modificado no máximo uma vez pela avaliação de uma expressão. Além disso, o valor anterior deve ser lido apenas para determinar o valor a ser armazenado.

(Padrão C99, seção 6.5, parágrafo 2)

Esse esquema provou ser um pouco grosseiro, resultando em algumas expressões exibindo um comportamento indefinido em relação ao C99 que plausivelmente não deveria fazer. O C2011 retém pontos de sequência, mas apresenta uma abordagem mais diferenciada para essa área com base em sequencing e um relacionamento que ele chama de “sequenced before”:

Se um efeito colateral em um objeto escalar não for sequenciado em relação a um efeito colateral diferente no mesmo objeto escalar ou a um cálculo de valor usando o valor do mesmo objeto escalar, o comportamento é indefinido. Se houver várias ordenações permitidas das subexpressões de uma expressão, o comportamento será indefinido se ocorrer um efeito colateral não sequenciado em qualquer uma das ordenações.

(Norma C2011, seção 6.5, parágrafo 2)

Os detalhes completos da relação “sequenciado antes” são longos demais para serem descritos aqui, mas eles complementam os pontos de sequência em vez de suplantá-los, de modo que têm o efeito de definir o comportamento para algumas avaliações cujo comportamento anteriormente era indefinido. Em particular, se houver um ponto de sequência entre duas avaliações, aquele antes do ponto de sequência será “sequenciado antes” do seguinte.

O exemplo a seguir tem um comportamento bem definido:

int i = 42;
i = (i++, i+42); /* The comma-operator creates a sequence point */

O exemplo a seguir tem comportamento indefinido:

int i = 42;
printf("%d %d\n", i++, i++); /* commas as separator of function arguments are not comma-operators */

Como acontece com qualquer forma de comportamento indefinido, observar o comportamento real de avaliar expressões que violam as regras de sequenciamento não é informativo, exceto em um sentido retrospectivo. O padrão da linguagem não fornece base para esperar que tais observações sejam preditivas até mesmo do comportamento futuro do mesmo programa.

Passando um ponteiro nulo para a conversão de printf %s #

A conversão %s de printf afirma que o argumento correspondente um ponteiro para o elemento inicial de um array do tipo caractere. Um ponteiro nulo não aponta para o elemento inicial de nenhuma matriz de tipo de caractere e, portanto, o comportamento do seguinte é indefinido:

char *foo = NULL;
printf("%s", foo); /* undefined behavior */

No entanto, o comportamento indefinido nem sempre significa que o programa trava — alguns sistemas tomam medidas para evitar o travamento que normalmente acontece quando um ponteiro nulo é desreferenciado. Por exemplo, Glibc é conhecido por imprimir

(null)

para o código acima. No entanto, adicione (apenas) uma nova linha à string de formato e você terá uma falha:

char *foo = 0;
printf("%s\n", foo); /* undefined behavior */

Neste caso, isso acontece porque o GCC tem uma otimização que transforma printf("%s\n", argument); em uma chamada para puts com puts(argument), e puts na Glibc não lidar com ponteiros nulos. Todo esse comportamento está em conformidade com o padrão.

Observe que o ponteiro nulo é diferente de uma string vazia. Portanto, o seguinte é válido e não tem comportamento indefinido. Ele apenas imprimirá uma nova linha:

char *foo = "";
printf("%s\n", foo);

Liberando memória duas vezes #

Liberar memória duas vezes é um comportamento indefinido, por exemplo.

int * x = malloc(sizeof(int));
*x = 9;
free(x);
free(x);

Citação do padrão (7.20.3.2. A função livre de C99 ):

Caso contrário, se o argumento não corresponder a um ponteiro retornado anteriormente pela função calloc, malloc ou realloc, ou se o espaço foi desalocado por uma chamada para free ou realloc, o comportamento é indefinido.

Deslocamento de bits usando contagens negativas ou além da largura do tipo #

Se o valor de shift count for um valor negativo, as operações de shift para a esquerda e deslocamento para a direita serão indefinidas1:

int x = 5 << -3; /* undefined */
int x = 5 >> -3; /* undefined */

Se shift para a esquerda for executado em um valor negativo, ele será indefinido:

int x = -5 << 3; /* undefined */

Se deslocamento para a esquerda for executado em um valor positivo e o resultado do valor matemático não puder ser representado no tipo, ele será indefinido1:

/* Assuming an int is 32-bits wide, the value '5 * 2^72' doesn't fit 
 * in an int. So, this is undefined. */
       
int x = 5 << 72;

Observe que deslocamento para a direita em um valor negativo (por exemplo, -5 >> 3) não é indefinido, mas definido pela implementação.

1 Citando ISO/IEC 9899:201x, seção 6.5.7:

Se o valor do operando direito for negativo ou for maior ou igual à largura do operando esquerdo promovido, o comportamento é indefinido.

Acessando a memória além do pedaço alocado #

Um ponteiro para um pedaço de memória contendo elementos n só pode ser desreferenciado se estiver no intervalo memory e memory + (n - 1). Desreferenciar um ponteiro fora desse intervalo resulta em comportamento indefinido. Como exemplo, considere o seguinte código:

int array[3];
int *beyond_array = array + 3;
*beyond_array = 0; /* Accesses memory that has not been allocated. */

A terceira linha acessa o 4º elemento em uma matriz com apenas 3 elementos, levando a um comportamento indefinido. Da mesma forma, o comportamento da segunda linha no fragmento de código a seguir também não está bem definido:

int array[3];
array[3] = 0;

Observe que apontar para além do último elemento de um array não é um comportamento indefinido (beyond_array = array + 3 está bem definido aqui), mas desreferenciar é (*beyond_array é um comportamento indefinido). Essa regra também vale para memória alocada dinamicamente (como buffers criados por meio de malloc).

Retornando de uma função declarada com o especificador de função _Noreturn ou noreturn #

O especificador de função _Noreturn foi introduzido em C11. O cabeçalho <stdnoreturn.h> fornece uma macro noreturn que se expande para _Noreturn. Portanto, usar _Noreturn ou noreturn de <stdnoreturn.h> é bom e equivalente.

Uma função declarada com _Noreturn (ou noreturn) não tem permissão para retornar ao seu chamador. Se tal função retornar ao seu chamador, o comportamento é indefinido.

No exemplo a seguir, func() é declarado com o especificador noreturn mas retorna ao seu chamador.

#include <stdio.h>
#include <stdlib.h>
#include <stdnoreturn.h>

noreturn void func(void);

void func(void)
{
    printf("In func()...\n");
} /* Undefined behavior as func() returns */

int main(void)
{
    func();
    return 0;
}

gcc e clang produzem avisos para o programa acima:

$ gcc test.c
test.c: In function ‘func’:
test.c:9:1: warning: ‘noreturn’ function does return
 }
 ^
$ clang test.c
test.c:9:1: warning: function declared 'noreturn' should not return [-Winvalid-noreturn]
}
^

Um exemplo usando noreturn que tem comportamento bem definido:

#include <stdio.h>
#include <stdlib.h>
#include <stdnoreturn.h>

noreturn void my_exit(void);

/* calls exit() and doesn't return to its caller. */
void my_exit(void)
{
    printf("Exiting...\n");
    exit(0);
}

int main(void)
{
    my_exit();
    return 0;
}

Desreferenciando um ponteiro nulo #

Este é um exemplo de desreferenciação de um ponteiro NULL, causando um comportamento indefinido.

int * pointer = NULL;
int value = *pointer; /* Dereferencing happens here */

Um ponteiro NULL é garantido pelo padrão C para comparar desigual a qualquer ponteiro para um objeto válido, e desreferenciar ele invoca um comportamento indefinido.

Lendo um objeto não inicializado que não é suportado pela memória #

Ler um objeto causará um comportamento indefinido, se o objeto for1:

  • não inicializado
  • definido com duração de armazenamento automática
  • seu endereço nunca é tomado

A variável a no exemplo abaixo satisfaz todas essas condições:

void Function( void )
{
    int a;
    int b = a;
}

1 (citado de: ISO:IEC 9899:201X 6.3.2.1 Lvalores, matrizes e designadores de função 2) Se o lvalue designar um objeto de duração de armazenamento automática que poderia ter sido declarado com a classe de armazenamento de registradores (nunca teve seu endereço usado), e esse objeto não for inicializado (não declarado com um inicializador e nenhuma atribuição a ele foi executada antes do uso ), o comportamento é indefinido.

Adição ou subtração de ponteiro não delimitado corretamente #

O código a seguir tem comportamento indefinido:

char buffer[6] = "hello";
char *ptr1 = buffer - 1;  /* undefined behavior */
char *ptr2 = buffer + 5;  /* OK, pointing to the '\0' inside the array */
char *ptr3 = buffer + 6;  /* OK, pointing to just beyond */
char *ptr4 = buffer + 7;  /* undefined behavior */

De acordo com C11, se a adição ou subtração de um ponteiro em, ou um pouco além, um objeto de matriz e um tipo inteiro produzir um resultado que não aponte para, ou um pouco além, do mesmo objeto de matriz, o comportamento é indefinido (6.5.6 ).

Além disso, é um comportamento naturalmente indefinido desreferenciar um ponteiro que aponta para um pouco além da matriz:

char buffer[6] = "hello";
char *ptr3 = buffer + 6;  /* OK, pointing to just beyond */
char value = *ptr3;       /* undefined behavior */

Modificando uma variável const usando um ponteiro #

int main (void)
{
    const int foo_readonly = 10;
    int *foo_ptr;

    foo_ptr = (int *)&foo_readonly; /* (1) This casts away the const qualifier */
    *foo_ptr = 20; /* This is undefined behavior */

    return 0;
}

Citando ISO/IEC 9899:201x, seção 6.7.3 §2:

Se for feita uma tentativa de modificar um objeto definido com um tipo const-qualified através do uso de um lvalue com tipo não const-qualified, o comportamento é indefinido. […]

(1) No GCC, isso pode gerar o seguinte aviso: warning: atribuição descarta o qualificador 'const' do tipo de destino do ponteiro [-Wdiscarded-qualifiers]

Usando fflush em um fluxo de entrada #

Os padrões POSIX e C declaram explicitamente que usar fflush em um fluxo de entrada é um comportamento indefinido. O fflush é definido apenas para fluxos de saída.

#include <stdio.h>

int main()
{
    int i;
    char input[4096];

    scanf("%i", &i);
    fflush(stdin); // <-- undefined behavior
    gets(input);

    return 0;
}

Não há uma maneira padrão de descartar caracteres não lidos de um fluxo de entrada. Por outro lado, algumas implementações usam fflush para limpar o buffer stdin. A Microsoft define o comportamento de fflush em um fluxo de entrada: Se o fluxo estiver aberto para entrada, fflush limpa o conteúdo do buffer. De acordo com POSIX.1-2008, o comportamento de fflush é indefinido a menos que o arquivo de entrada seja pesquisável.

Veja Usando fflush(stdin) para mais detalhes.

Instrução de retorno ausente na função de retorno de valor #

int foo(void) {
  /* do stuff */
  /* no return here */
}

int main(void) {
  /* Trying to use the (not) returned value causes UB */
  int value = foo();
  return 0;
}

Quando uma função é declarada para retornar um valor, ela deve fazê-lo em todos os caminhos de código possíveis através dela. O comportamento indefinido ocorre assim que o chamador (que está esperando um valor de retorno) tenta usar o valor de retorno1.

Observe que o comportamento indefinido ocorre somente se o chamador tentar usar/acessar o valor da função. Por exemplo,

int foo(void) {
  /* do stuff */
  /* no return here */
}

int main(void) {
  /* The value (not) returned from foo() is unused. So, this program
   * doesn't cause *undefined behaviour*. */
  foo();
  return 0;
}

A função main() é uma exceção a esta regra, pois é possível que ela seja encerrada sem uma instrução de retorno porque um valor de retorno assumido de 0 será usado automaticamente neste caso2.

1 (ISO/IEC 9899:201x, 6.9.1/12)

Se o } que termina uma função for alcançado e o valor da chamada da função for usado pelo chamador, o comportamento será indefinido.

2 (ISO/IEC 9899:201x, 5.1.2.2.3/1)

atingindo o } que finaliza a função main retorna um valor de 0.

Divisão por zero #

int x = 0;
int y = 5 / x;  /* integer division */

ou

double x = 0.0;
double y = 5.0 / x;  /* floating point division */

ou

int x = 0;
int y = 5 % x;  /* modulo operation */

Para a segunda linha em cada exemplo, onde o valor do segundo operando (x) é zero, o comportamento é indefinido.

Observe que a maioria das implementações da matemática de ponto flutuante segue um padrão (por exemplo, IEEE 754), caso em que operações como divisão por zero terão resultados consistentes (por exemplo, INFINITY) mesmo embora o padrão C diga que a operação é indefinida.

Ligação inconsistente de identificadores #

extern int var;
static int var; /* Undefined behaviour */

C11, §6.2.2, 7 diz:

Se, dentro de uma unidade de tradução, o mesmo identificador aparecer com ambos ligação interna e externa, o comportamento é indefinido.

Observe que, se uma declaração anterior de um identificador estiver visível, ela terá a ligação da declaração anterior. C11, §6.2.2, 4 permite:

Para um identificador declarado com o especificador de classe de armazenamento externo em um escopo em que uma declaração prévia desse identificador é visível,31) se a declaração prévia especifica a ligação interna ou externa, a ligação do identificador na declaração posterior é a mesma que a ligação especificada na declaração anterior. Se nenhuma declaração prévia for visível, ou se a declaração anterior não especifica nenhuma ligação, então o identificador tem ligação externa.

/* 1. This is NOT undefined */
static int var;
extern int var; 


/* 2. This is NOT undefined */
static int var;
static int var; 

/* 3. This is NOT undefined */
extern int var;
extern int var;

A conversão entre os tipos de ponteiro produz um resultado alinhado incorretamente #

O seguinte pode ter um comportamento indefinido devido ao alinhamento incorreto do ponteiro:

 char *memory_block = calloc(sizeof(uint32_t) + 1, 1);
 uint32_t *intptr = (uint32_t*)(memory_block + 1);  /* possible undefined behavior */
 uint32_t mvalue = *intptr;

O comportamento indefinido acontece à medida que o ponteiro é convertido. De acordo com C11, se uma conversão entre dois tipos de ponteiro produzir um resultado alinhado incorretamente (6.3.2.3), o comportamento será indefinido. Aqui um uint32_t pode requerer alinhamento de 2 ou 4 por exemplo.

calloc, por outro lado, é necessário para retornar um ponteiro que esteja alinhado adequadamente para qualquer tipo de objeto; assim memory_block está devidamente alinhado para conter um uint32_t em sua parte inicial. Então, em um sistema onde uint32_t exigiu alinhamento de 2 ou 4, memory_block + 1 será um endereço ímpar e, portanto, não alinhado corretamente.

Observe que o padrão C solicita que já a operação de conversão seja indefinida. Isso é imposto porque em plataformas onde os endereços são segmentados, o endereço de byte memory_block + 1 pode nem ter uma representação adequada como um ponteiro inteiro.

A conversão de char * para ponteiros para outros tipos sem nenhuma preocupação com os requisitos de alinhamento às vezes é usada incorretamente para decodificar estruturas compactadas, como cabeçalhos de arquivos ou pacotes de rede.

Você pode evitar o comportamento indefinido decorrente da conversão desalinhada do ponteiro usando memcpy:

memcpy(&mvalue, memory_block + 1, sizeof mvalue);

Aqui não ocorre nenhuma conversão de ponteiro para uint32_t* e os bytes são copiados um por um.

Esta operação de cópia para nosso exemplo leva apenas ao valor válido de mvalue porque:

  • Usamos calloc, para que os bytes sejam inicializados corretamente. No nosso caso, todos os bytes têm valor ‘0’, mas qualquer outra inicialização adequada serviria.
  • uint32_t é um tipo de largura exata e não possui bits de preenchimento
  • Qualquer padrão de bits arbitrário é uma representação válida para qualquer tipo sem sinal.

Modificando a string retornada pelas funções getenv, strerror e setlocale #

A modificação das strings retornadas pelas funções padrão getenv(), strerror() e setlocale() é indefinida. Portanto, as implementações podem usar armazenamento estático para essas strings.

A função getenv(), C11, §7.22.4.7, 4, diz:

A função getenv retorna um ponteiro para uma string associada ao membro da lista correspondente. A string apontada não deve ser modificada pelo programa, mas pode ser substituído por uma chamada subsequente ao getenv função.

A função strerror(), C11, §7.23.6.3, 4 diz:

A função strerror retorna um ponteiro para a string, o conteúdo de que são locais específicos. A matriz apontada não deve ser modificada por o programa, mas pode ser substituído por uma chamada subsequente ao função strerror.

A função setlocale(), C11, §7.11.1.1, 8 diz:

O ponteiro para string retornado pela função setlocale é tal que uma chamada subsequente com esse valor de string e sua categoria associada irá restaurar essa parte da localidade do programa. A corda apontada para não deve ser modificado pelo programa, mas pode ser substituído por um chamada subsequente para a função setlocale.

Da mesma forma, a função localeconv() retorna um ponteiro para struct lconv que não deve ser modificado.

A função localeconv(), C11, §7.11.2.1, 8 diz:

A função localeconv retorna um ponteiro para o objeto preenchido. o estrutura apontada pelo valor de retorno não deve ser modificada pelo programa, mas pode ser substituído por uma chamada subsequente para o localeconv função.
