Misturando inteiros com e sem sinal em operações aritméticas #

Geralmente não é uma boa idéia misturar inteiros signed e unsigned em operações aritméticas. Por exemplo, qual será a saída do exemplo a seguir?

#include <stdio.h>

int main(void)
{ 
    unsigned int a = 1000;
    signed int b = -1;

    if (a > b) puts("a is more than b");
    else puts("a is less or equal than b"); 

    return 0;
}

Como 1000 é maior que -1, você esperaria que a saída fosse ‘a é maior que b’, mas esse não será o caso.

As operações aritméticas entre diferentes tipos integrais são executadas dentro de um tipo comum definido pelas chamadas conversões aritméticas usuais (veja a especificação da linguagem, 6.3.1.8).

Nesse caso, o “tipo comum” é unsigned int, porque, conforme indicado em Conversões aritméticas usuais,

714  Caso contrário, se o operando que possui o tipo inteiro sem sinal tiver rank maior ou igual ao rank do tipo do outro operando, então o operando com tipo inteiro assinado é convertido para o tipo de o operando com tipo inteiro sem sinal.

Isso significa que int operando b será convertido em unsigned int antes da comparação.

Quando -1 é convertido em um unsigned int o resultado é o valor máximo possível de unsigned int, que é maior que 1000, significando que a > b é falso.

Macros são simples substituições de strings #

Macros são substituições de strings simples. (Estritamente falando, eles trabalham com tokens de pré-processamento, não com strings arbitrárias.)

#include <stdio.h>

#define SQUARE(x) x*x

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Você pode esperar que este código imprima 9 (3*3), mas na verdade 5 será impresso porque a macro será expandida para 1+2*1+2.

Você deve envolver os argumentos e toda a expressão da macro entre parênteses para evitar esse problema.

#include <stdio.h>

#define SQUARE(x) ((x)*(x))

int main(void) {
    printf("%d\n", SQUARE(1+2));
    return 0;
}

Outro problema é que não é garantido que os argumentos de uma macro sejam avaliados uma vez; eles podem não ser avaliados ou podem ser avaliados várias vezes.

#include <stdio.h>

#define MIN(x, y) ((x) <= (y) ? (x) : (y))

int main(void) {
    int a = 0;
    printf("%d\n", MIN(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

Neste código, a macro será expandida para ((a++) <= (10) ? (a++): (10)). Como a++ (0) é menor que 10, a++ será avaliado duas vezes e fará com que o valor de a e o que é retornado de MIN sejam diferentes do esperado.

Isso pode ser evitado usando funções, mas observe que os tipos serão corrigidos pela definição da função, enquanto as macros podem ser (muito) flexíveis com os tipos.

#include <stdio.h>

int min(int x, int y) {
    return x <= y ? x : y;
}

int main(void) {
    int a = 0;
    printf("%d\n", min(a++, 10));
    printf("a = %d\n", a);
    return 0;
}

Agora o problema da avaliação dupla foi corrigido, mas esta função min não pode lidar com dados double sem truncar, por exemplo.

As diretivas de macro podem ser de dois tipos:

#define OBJECT_LIKE_MACRO     followed by a "replacement list" of preprocessor tokens
#define FUNCTION_LIKE_MACRO(with, arguments) followed by a replacement list

O que distingue esses dois tipos de macros é o caractere que segue o identificador após #define: se for um lparen, é uma macro do tipo função; caso contrário, é uma macro semelhante a um objeto. Se a intenção é escrever uma macro do tipo função, não deve haver nenhum espaço em branco entre o final do nome da macro e (. Verifique this para uma explicação detalhada.

No C99 ou posterior, você pode usar static inline int min(int x, int y) { … }.

Em C11, você poderia escrever uma expressão ’tipo-genérica’ para ‘min’.

#include <stdio.h>

#define min(x, y) _Generic((x), \
                        long double: min_ld, \
                        unsigned long long: min_ull, \
                        default: min_i \
                        )(x, y)

#define gen_min(suffix, type) \
    static inline type min_##suffix(type x, type y) { return (x < y) ? x : y; }

gen_min(ld, long double)
gen_min(ull, unsigned long long)
gen_min(i, int)

int main(void)
{
    unsigned long long ull1 = 50ULL;
    unsigned long long ull2 = 37ULL;
    printf("min(%llu, %llu) = %llu\n", ull1, ull2, min(ull1, ull2));
    long double ld1 = 3.141592653L;
    long double ld2 = 3.141592652L;
    printf("min(%.10Lf, %.10Lf) = %.10Lf\n", ld1, ld2, min(ld1, ld2));
    int i1 = 3141653;
    int i2 = 3141652;
    printf("min(%d, %d) = %d\n", i1, i2, min(i1, i2));
    return 0;
}

A expressão genérica pode ser estendida com mais tipos como double, float, long long, unsigned long, long, unsigned — e chamadas de macro gen_min apropriadas escritas.

Esquecer de copiar o valor de retorno de realloc em um temporário #

Se realloc falhar, ele retornará NULL. Se você atribuir o valor do buffer original ao valor de retorno de realloc e se ele retornar NULL, o buffer original (o ponteiro antigo) será perdido, resultando em um vazamento de memória . A solução é copiar para um ponteiro temporário e, se esse temporário não for NULL, então copie para o buffer real.

char *buf, *tmp;

buf = malloc(...);
...

/* WRONG */
if ((buf = realloc(buf, 16)) == NULL)
    perror("realloc");

/* RIGHT */
if ((tmp = realloc(buf, 16)) != NULL)
    buf = tmp;
else
    perror("realloc");

Esquecer de alocar um byte extra para \0 #

Quando você estiver copiando uma string em um buffer malloc, lembre-se sempre de adicionar 1 a strlen.

char *dest = malloc(strlen(src)); /* WRONG */
char *dest = malloc(strlen(src) + 1); /* RIGHT */

strcpy(dest, src);

Isso ocorre porque o strlen não inclui o \0 à direita no comprimento. Se você usar a abordagem WRONG (como mostrado acima), ao chamar strcpy, seu programa invocará um comportamento indefinido.

Também se aplica a situações em que você está lendo uma string de comprimento máximo conhecido de stdin ou alguma outra fonte. Por exemplo

#define MAX_INPUT_LEN 42

char buffer[MAX_INPUT_LEN]; /* WRONG */
char buffer[MAX_INPUT_LEN + 1]; /* RIGHT */

scanf("%42s", buffer);  /* Ensure that the buffer is not overflowed */

Decaimento da matriz mal-entendido #

Um problema comum em código que usa arrays multidimensionais, arrays de ponteiros, etc., é o fato de que Type** e Type[M][N] são tipos fundamentalmente diferentes:

#include <stdio.h>

void print_strings(char **strings, size_t n)
{
    size_t i;
    for (i = 0; i < n; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(s, 4);
    return 0;
}

Exemplo de saída do compilador:

file1.c: In function 'main':
file1.c:13:23: error: passing argument 1 of 'print_strings' from incompatible pointer type [-Wincompatible-pointer-types]
         print_strings(strings, 4);
                       ^
file1.c:3:10: note: expected 'char **' but argument is of type 'char (*)[20]'
     void print_strings(char **strings, size_t n)

O erro informa que o array s na função main é passado para a função print_strings, que espera um tipo de ponteiro diferente do recebido. Ele também inclui uma nota expressando o tipo que é esperado por print_strings e o tipo que foi passado para ele de main.

O problema é devido a algo chamado array decay. O que acontece quando s com seu tipo char[4][20] (array de 4 arrays de 20 caracteres) é passado para a função é que ele se transforma em um ponteiro para seu primeiro elemento como se você tivesse escrito &s[ 0], que tem o tipo char (*)[20] (ponteiro para 1 array de 20 caracteres). Isso ocorre para qualquer matriz, incluindo uma matriz de ponteiros, uma matriz de matrizes de matrizes (matrizes 3-D) e uma matriz de ponteiros para uma matriz. Abaixo está uma tabela que ilustra o que acontece quando uma matriz decai. As alterações na descrição do tipo são destacadas para ilustrar o que acontece:

Antes da Decadência   Após Decadência  
char [20] matriz de (20 caracteres) char * ponteiro para (1 caractere)
char [4][20] matriz de (4 matrizes de 20 caracteres) char (*)[20] ponteiro para (1 array de 20 caracteres)
char *[4] matriz de (4 ponteiros para 1 caractere) char ** ponteiro para (1 ponteiro para 1 caractere)
char [3][4][20] matriz de (3 matrizes de 4 matrizes de 20 caracteres) char (*)[4][20] ponteiro para (1 array de 4 arrays de 20 caracteres)
char (*[4])[20] matriz de (4 ponteiros para 1 matriz de 20 caracteres) char (**)[20] ponteiro para (1 ponteiro para 1 array de 20 caracteres)

Se uma matriz pode decair para um ponteiro, pode-se dizer que um ponteiro pode ser considerado uma matriz de pelo menos 1 elemento. Uma exceção a isso é um ponteiro nulo, que não aponta para nada e, consequentemente, não é um array.

O decaimento do array ocorre apenas uma vez. Se uma matriz decaiu para um ponteiro, agora é um ponteiro, não uma matriz. Mesmo se você tiver um ponteiro para um array, lembre-se de que o ponteiro pode ser considerado um array de pelo menos um elemento, então o decaimento do array já ocorreu.

Em outras palavras, um ponteiro para um array (char (*)[20]) nunca se tornará um ponteiro para um ponteiro (char **). Para corrigir a função print_strings, simplesmente faça com que ela receba o tipo correto:

void print_strings(char (*strings)[20], size_t n)
/* OR */
void print_strings(char strings[][20], size_t n)

Um problema surge quando você quer que a função print_strings seja genérica para qualquer array de caracteres: e se houver 30 caracteres em vez de 20? Ou 50? A resposta é adicionar outro parâmetro antes do parâmetro array:

#include <stdio.h>

/*
 * Note the rearranged parameters and the change in the parameter name
 * from the previous definitions:
 *      n (number of strings)
 *   => scount (string count)
 *
 * Of course, you could also use one of the following highly recommended forms
 * for the `strings` parameter instead:
 *
 *    char strings[scount][ccount]
 *    char strings[][ccount]
 */
void print_strings(size_t scount, size_t ccount, char (*strings)[ccount])
{
    size_t i;
    for (i = 0; i < scount; i++)
        puts(strings[i]);
}

int main(void)
{
    char s[4][20] = {"Example 1", "Example 2", "Example 3", "Example 4"};
    print_strings(4, 20, s);
    return 0;
}

Compilá-lo não produz erros e resulta na saída esperada:

Example 1
Example 2
Example 3
Example 4

Esquecer de liberar memória (vazamentos de memória) #

Uma prática recomendada de programação é liberar qualquer memória que tenha sido alocada diretamente pelo seu próprio código ou implicitamente chamando uma função interna ou externa, como uma API de biblioteca como strdup(). A falha ao liberar memória pode introduzir um vazamento de memória, que pode se acumular em uma quantidade substancial de memória desperdiçada que não está disponível para seu programa (ou sistema), possivelmente levando a travamentos ou comportamento indefinido. É mais provável que ocorram problemas se o vazamento ocorrer repetidamente em um loop ou função recursiva. O risco de falha do programa aumenta quanto mais tempo um programa com vazamento é executado. Às vezes, os problemas aparecem instantaneamente; outras vezes os problemas não serão vistos por horas ou mesmo anos de operação constante. As falhas de esgotamento de memória podem ser catastróficas, dependendo das circunstâncias.

O loop infinito a seguir é um exemplo de um vazamento que eventualmente esgotará o vazamento de memória disponível chamando getline(), uma função que aloca implicitamente nova memória, sem liberar essa memória.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    /* The loop below leaks memory as fast as it can */

    for(;;) { 
        getline(&line, &size, stdin); /* New memory implicitly allocated */

        /* <do whatever> */

        line = NULL;
    }

    return 0;
}

Em contrapartida, o código abaixo também utiliza a função getline(), mas desta vez, a memória alocada é liberada corretamente, evitando um vazamento.

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char *line = NULL;
    size_t size = 0;

    for(;;) {
        if (getline(&line, &size, stdin) < 0) {
            free(line);
            line = NULL;

            /* Handle failure such as setting flag, breaking out of loop and/or exiting */
        }

        /* <do whatever> */

        free(line);
        line = NULL;

    }

    return 0;
}

O vazamento de memória nem sempre tem consequências tangíveis e não é necessariamente um problema funcional. Embora a “prática recomendada” dite a liberação rigorosa de memória em pontos e condições estratégicos, para reduzir o volume de memória e diminuir o risco de esgotamento da memória, pode haver exceções. Por exemplo, se um programa é limitado em duração e escopo, o risco de falha de alocação pode ser considerado muito pequeno para se preocupar. Nesse caso, ignorar a desalocação explícita pode ser considerado aceitável. Por exemplo, a maioria dos sistemas operacionais modernos libera automaticamente toda a memória consumida por um programa quando ele termina, seja devido a uma falha do programa, uma chamada do sistema para exit(), finalização do processo ou finalização de main(). Liberar memória explicitamente no ponto de encerramento iminente do programa pode realmente ser redundante ou introduzir uma penalidade de desempenho.

A alocação pode falhar se houver memória insuficiente disponível e as falhas de manipulação devem ser contabilizadas nos níveis apropriados da pilha de chamadas. getline(), mostrado acima é um caso de uso interessante porque é uma função de biblioteca que não apenas aloca memória que deixa para o chamador liberar, mas pode falhar por vários motivos, todos os quais devem ser levados em consideração . Portanto, é essencial ao usar uma API C, ler a documentação (man page) e prestar atenção especial às condições de erro e uso de memória, e estar ciente de qual camada de software carrega o fardo de liberar memória retornada .

Outra prática comum de manipulação de memória é definir consistentemente os ponteiros de memória para NULL imediatamente após a memória referenciada por esses ponteiros ser liberada, para que esses ponteiros possam ser testados quanto à validade a qualquer momento (por exemplo, verificados para NULL / não NULL), porque o acesso à memória liberada pode levar a problemas graves, como obter dados inúteis (operação de leitura) ou corrupção de dados (operação de gravação) e/ou travamento do programa. Na maioria dos sistemas operacionais modernos, liberar o local de memória 0 (NULL) é um NOP (por exemplo, é inofensivo), conforme exigido pelo padrão C - portanto, definindo um ponteiro para NULL, não há risco de liberar memória dupla se o ponteiro é passado para free(). Lembre-se de que a liberação dupla de memória pode levar a falhas muito demoradas, confusas e difíceis de diagnosticar.

Copiando demais #

char buf[8]; /* tiny buffer, easy to overflow */

printf("What is your name?\n");
scanf("%s", buf); /* WRONG */
scanf("%7s", buf); /* RIGHT */

Se o usuário digitar uma string com mais de 7 caracteres (-1 para o terminador nulo), a memória atrás do buffer buf será substituída. Isso resulta em comportamento indefinido. Hackers mal-intencionados geralmente exploram isso para substituir o endereço de retorno e alterá-lo para o endereço do código mal-intencionado do hacker.

Escrevendo erroneamente = em vez de == ao comparar #

O operador = é usado para atribuição.

O operador == é usado para comparação.

Deve-se ter cuidado para não misturar os dois. Às vezes, escreve-se erroneamente

/* assign y to x */
if (x = y) {
     /* logic */
}

quando o que realmente se queria é:

/* compare if x is equal to y */
if (x == y) {
    /* logic */
}

O primeiro atribui valor de y a x e verifica se esse valor é diferente de zero, em vez de fazer comparação, o que equivale a:

if ((x = y) != 0) {
    /* logic */
}

Há momentos em que testar o resultado de uma atribuição é intencional e é comumente usado, porque evita ter que duplicar o código e ter que tratar a primeira vez de forma especial. Comparar

while ((c = getopt_long(argc, argv, short_options, long_options, &option_index)) != -1) {
        switch (c) {
        ...
        }
}

contra

c = getopt_long(argc, argv, short_options, long_options, &option_index);
while (c != -1) {
        switch (c) {
        ...
        }
        c = getopt_long(argc, argv, short_options, long_options, &option_index);
}

Compiladores modernos reconhecerão esse padrão e não avisarão quando a atribuição estiver entre parênteses como acima, mas poderão avisar para outros usos. Por exemplo:

if (x = y)         /* warning */

if ((x = y))       /* no warning */
if ((x = y) != 0)  /* no warning; explicit */

Alguns programadores usam a estratégia de colocar a constante à esquerda do operador (comumente chamada de condições Yoda). Como as constantes são rvalues, esse estilo de condição fará com que o compilador lance um erro se o operador errado for usado.

if (5 = y) /* Error */

if (5 == y) /* No error */

No entanto, isso reduz severamente a legibilidade do código e não é considerado necessário se o programador seguir boas práticas de codificação C e não ajuda na comparação de duas variáveis, portanto, não é uma solução universal. Além disso, muitos compiladores modernos podem dar avisos quando o código é escrito com condições Yoda.

Uso imprudente de ponto e vírgula #

Cuidado com o ponto e vírgula. Exemplo a seguir

if (x > a);
   a = x;

na verdade significa:

if (x > a) {}
a = x;

o que significa que x será atribuído a a em qualquer caso, o que pode não ser o que você queria originalmente.

Às vezes, a falta de um ponto e vírgula também causará um problema imperceptível:

    return
day = date[0];
hour = date[1];
minute = date[2];

O ponto e vírgula atrás do retorno foi perdido, então dia=data[0] será retornado.

Uma técnica para evitar este e outros problemas semelhantes é sempre usar chaves em condicionais e loops de várias linhas. Por exemplo:

O caractere de nova linha não é consumido em uma chamada típica de scanf() #

Quando este programa

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char str[128], *lf;

    scanf("%d", &num);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

é executado com esta entrada

42
life

a saída será 42 "" em vez do esperado 42 "vida".

Isso ocorre porque um caractere de nova linha após 42 não é consumido na chamada de scanf() e é consumido por fgets() antes de ler life. Então, fgets() pare de ler antes de ler life.

Para evitar este problema, uma maneira que é útil quando o comprimento máximo de uma linha é conhecido – ao resolver problemas no sistema de juiz online, por exemplo – é evitar usar scanf() diretamente e ler todas as linhas via fgets( ). Você pode usar sscanf() para analisar as linhas lidas.

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char line_buffer[128] = "", str[128], *lf;

    fgets(line_buffer, sizeof(line_buffer), stdin);
    sscanf(line_buffer, "%d", &num);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

Outra maneira é ler até encontrar um caractere de nova linha depois de usar scanf() e antes de usar fgets().

#include <stdio.h>
#include <string.h>

int main(void) {
    int num = 0;
    char str[128], *lf;
    int c;

    scanf("%d", &num);
    while ((c = getchar()) != '\n' && c != EOF);
    fgets(str, sizeof(str), stdin);

    if ((lf = strchr(str, '\n')) != NULL) *lf = '\0';
    printf("%d \"%s\"\n", num, str);
    return 0;
}

Adicionando um ponto e vírgula a um #define #

É fácil ficar confuso no pré-processador C e tratá-lo como parte do próprio C, mas isso é um erro porque o pré-processador é apenas um mecanismo de substituição de texto. Por exemplo, se você escrever

/* WRONG */

int arr[MAX];

o código se expande para

que é um erro de sintaxe. A solução é remover o ponto e vírgula da linha #define. É quase invariavelmente um erro terminar um #define com um ponto e vírgula.

Erros de referência indefinidos ao vincular #

Um dos erros mais comuns na compilação acontece durante o estágio de vinculação. O erro é semelhante a este:

$ gcc undefined_reference.c 
/tmp/ccoXhwF0.o: In function `main':
undefined_reference.c:(.text+0x15): undefined reference to `foo'
collect2: error: ld returned 1 exit status
$

Então, vamos ver o código que gerou esse erro:

int foo(void);

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Vemos aqui uma declaração de foo (int foo();) mas nenhuma definição dele (função real). Portanto, fornecemos ao compilador o cabeçalho da função, mas não havia tal função definida em nenhum lugar, então o estágio de compilação passa, mas o vinculador sai com um erro Referência indefinida. Para corrigir esse erro em nosso pequeno programa, teríamos apenas que adicionar uma definição para foo:

/* Declaration of foo */
int foo(void);

/* Definition of foo */
int foo(void)
{
    return 5;
}

int main(int argc, char **argv)
{
    int foo_val;
    foo_val = foo();
    return foo_val;
}

Agora este código irá compilar. Uma situação alternativa surge onde a fonte para foo() está em um arquivo fonte separado foo.c (e há um cabeçalho foo.h para declarar foo() que está incluído tanto em foo.c ereferência_indefinida.c). Então a correção é vincular o arquivo objeto de foo.ceundefined_reference.c`, ou compilar ambos os arquivos de origem:

$ gcc -c undefined_reference.c 
$ gcc -c foo.c
$ gcc -o working_program undefined_reference.o foo.o
$

Ou:

$ gcc -o working_program undefined_reference.c foo.c
$

Um caso mais complexo é onde as bibliotecas estão envolvidas, como no código:

#include <stdio.h>
#include <stdlib.h>
#include <math.h>

int main(int argc, char **argv)
{
    double first;
    double second;
    double power;

    if (argc != 3)
    {
        fprintf(stderr, "Usage: %s <denom> <nom>\n", argv[0]);
        return EXIT_FAILURE;
    }

    /* Translate user input to numbers, extra error checking
     * should be done here. */
    first = strtod(argv[1], NULL);
    second = strtod(argv[2], NULL);

    /* Use function pow() from libm - this will cause a linkage 
     * error unless this code is compiled against libm! */
    power = pow(first, second);

    printf("%f to the power of %f = %f\n", first, second, power);

    return EXIT_SUCCESS;
}

O código está sintaticamente correto, a declaração para pow() existe em #include <math.h>, então tentamos compilar e vincular, mas recebemos um erro como este:

$ gcc no_library_in_link.c -o no_library_in_link
/tmp/ccduQQqA.o: In function `main':
no_library_in_link.c:(.text+0x8b): undefined reference to `pow'
collect2: error: ld returned 1 exit status
$

Isso acontece porque a definição para pow() não foi encontrada durante o estágio de vinculação. Para corrigir isso, temos que especificar que queremos vincular com a biblioteca matemática chamada libm especificando o sinalizador -lm. (Observe que existem plataformas como o macOS em que -lm não é necessário, mas quando você obtém a referência indefinida, a biblioteca é necessária.)

Então, voltamos a executar o estágio de compilação, desta vez especificando a biblioteca (após os arquivos fonte ou objeto):

$ gcc no_library_in_link.c -lm -o library_in_link_cmd
$ ./library_in_link_cmd 2 4
2.000000 to the power of 4.000000 = 16.000000
$

E funciona!

Verificando a expressão lógica contra ’true’ #

O padrão C original não tinha tipo booleano intrínseco, então bool, true e false não tinham significado inerente e eram frequentemente definidos pelos programadores. Normalmente, true seria definido como 1 e false seria definido como 0.

C99 adiciona o tipo interno _Bool e o cabeçalho <stdbool.h> que define bool (expandindo para _Bool), false e ​​true. Ele também permite redefinir bool, true e false, mas observa que esse é um recurso obsoleto.

Mais importante, as expressões lógicas tratam qualquer avaliação que seja zero como falsa e qualquer avaliação diferente de zero como verdadeira. Por exemplo:

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == true)  /* Comparison only succeeds if true is 0x80 and bitField has that bit set */
    {
        return true;
    }
    else
    {
        return false;
    }
}

No exemplo acima, a função está tentando verificar se o bit superior está definido e retornar true se estiver. No entanto, verificando explicitamente em relação a true, a instrução if só terá sucesso se (bitfield & 0x80) for avaliado como true como definido, que normalmente é 1 e muito raramente 0x80. Verifique explicitamente em relação ao caso que você espera:

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    if ((bitField & 0x80) == 0x80) /* Explicitly test for the case we expect */
    {
        return true;
    }
    else
    {
        return false;
    }
}

Ou avalie qualquer valor diferente de zero como verdadeiro.

/* Return 'true' if the most significant bit is set */
bool isUpperBitSet(uint8_t bitField)
{
    /* If upper bit is set, result is 0x80 which the if will evaluate as true */
    if (bitField & 0x80)
    {
        return true;
    }
    else
    {
        return false;
    }
}

Fazendo escala extra na aritmética de ponteiro #

Na aritmética de ponteiro, o inteiro a ser adicionado ou subtraído ao ponteiro é interpretado não como mudança de endereço, mas como número de elementos a serem movidos.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + sizeof(int) * 2; /* wrong */
    printf("%d %d\n", *ptr, *ptr2);
    return 0;
}

Este código faz escala extra no cálculo do ponteiro atribuído a ptr2. Se sizeof(int) for 4, o que é típico em ambientes modernos de 32 bits, a expressão significa “8 elementos após array[0]”, que está fora do intervalo, e invoca *comportamento indefinido *.

Para ter ptr2 apontando para o que são 2 elementos após array[0], você deve simplesmente adicionar 2.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = ptr + 2;
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

A aritmética explícita de ponteiros usando operadores aditivos pode ser confusa, portanto, usar subscritos de matriz pode ser melhor.

#include <stdio.h>

int main(void) {
    int array[] = {1, 2, 3, 4, 5};
    int *ptr = &array[0];
    int *ptr2 = &ptr[2];
    printf("%d %d\n", *ptr, *ptr2); /* "1 3" will be printed */
    return 0;
}

E1[E2] é idêntico a (*((E1)+(E2))) ([N1570](http://chimera.roma1.infn.it/SP/COMMON/iso-iec-9899- 1990.pdf) 6.5.2.1, parágrafo 2), e &(E1[E2]) é equivalente a ((E1)+(E2)) (N1570 6.5.3.2, nota de rodapé 102).

Alternativamente, se a aritmética de ponteiro for preferida, lançar o ponteiro para endereçar um tipo de dados diferente pode permitir o endereçamento de byte. No entanto, tenha cuidado: endianness pode se tornar um problema, e lançar para tipos diferentes de ‘pointer to character’ leva a estritos problemas de aliasing.

#include <stdio.h>

int main(void) {
    int array[3] = {1,2,3};  // 4 bytes * 3 allocated
    unsigned char *ptr = (unsigned char *) array;  // unsigned chars only take 1 byte
    /*
     * Now any pointer arithmetic on ptr will match
     * bytes in memory.  ptr can be treated like it
     * was declared as: unsigned char ptr[12];
     */

    return 0;
}

Comparando números de ponto flutuante #

Tipos de ponto flutuante (float, double e long double) não podem representar com precisão alguns números porque eles têm precisão finita e representam os valores em formato binário. Assim como temos dízimas periódicas na base 10 para frações como 1/3, existem frações que também não podem ser representadas finitamente em binário (como 1/3, mas também, mais importante, 1/10). Não compare diretamente valores de ponto flutuante; use um delta em vez disso.

#include <float.h> // for DBL_EPSILON and FLT_EPSILON
#include <math.h>  // for fabs()

int main(void)
{
    double a = 0.1; // imprecise: (binary) 0.000110...

    // may be false or true
    if (a + a + a + a + a + a + a + a + a + a == 1.0) {
        printf("10 * 0.1 is indeed 1.0. This is not guaranteed in the general case.\n");
    }

    // Using a small delta value.
    if (fabs(a + a + a + a + a + a + a + a + a + a - 1.0) < 0.000001) {
        // C99 5.2.4.2.2p8 guarantees at least 10 decimal digits
        // of precision for the double type.
        printf("10 * 0.1 is almost 1.0.\n");
    }

    return 0;
}

Ignorando valores de retorno de funções de biblioteca #

Quase todas as funções na biblioteca padrão C retornam algo em caso de sucesso e outra em caso de erro. Por exemplo, malloc retornará um ponteiro para o bloco de memória alocado pela função em caso de sucesso e, se a função falhou em alocar o bloco de memória solicitado, um ponteiro nulo. Portanto, você deve sempre verificar o valor de retorno para facilitar a depuração.

Isto é mau:

char* x = malloc(100000000000UL * sizeof *x);
/* more code */
scanf("%s", x); /* This might invoke undefined behaviour and if lucky causes a segmentation violation, unless your system has a lot of memory */

Isso é bom:

#include <stdlib.h>
#include <stdio.h>

int main(void)
{
    char* x = malloc(100000000000UL * sizeof *x);
    if (x == NULL) {
        perror("malloc() failed");
        exit(EXIT_FAILURE);
    }

    if (scanf("%s", x) != 1) {
        fprintf(stderr, "could not read string\n");
        free(x);
        exit(EXIT_FAILURE);
    }

    /* Do stuff with x. */

    /* Clean up. */
    free(x);

    return EXIT_SUCCESS;
}

Dessa forma, você sabe imediatamente a causa do erro, caso contrário, pode passar horas procurando um bug em um lugar completamente errado.

Literais de ponto flutuante são do tipo double por padrão #

Deve-se tomar cuidado ao inicializar variáveis ​​do tipo float para valores literais ou compará-las com valores literais, porque literais regulares de ponto flutuante como 0.1 são do tipo double. Isso pode levar a surpresas:

#include <stdio.h>
int main() {
    float  n;
    n = 0.1;
    if (n > 0.1) printf("Wierd\n");
    return 0;
}
// Prints "Wierd" when n is float

Aqui, n é inicializado e arredondado para precisão simples, resultando no valor 0,10000000149011612. Em seguida, n é convertido novamente para precisão dupla para ser comparado com o literal 0.1 (que equivale a 0,10000000000000001), resultando em uma incompatibilidade.

Além de erros de arredondamento, misturar variáveis ​​float com literais double resultará em desempenho ruim em plataformas que não possuem suporte de hardware para precisão dupla.

Usando constantes de caracteres em vez de literais de string e vice-versa #

Em C, constantes de caracteres e literais de string são coisas diferentes.

Um caractere entre aspas simples como 'a' é uma constante de caractere. Uma constante de caractere é um inteiro cujo valor é o código de caractere que representa o caractere. Como interpretar constantes de caracteres com vários caracteres como 'abc' é definido pela implementação.

Zero ou mais caracteres entre aspas duplas como "abc" é uma string literal. Um literal de string é um array não modificável cujos elementos são do tipo char. A string entre aspas duplas mais o caractere nulo final é o conteúdo, então "abc" tem 4 elementos ({'a', 'b', 'c', '\0'})

Neste exemplo, uma constante de caractere é usada onde um literal de string deve ser usado. Essa constante de caractere será convertida em um ponteiro de maneira definida pela implementação e há pouca chance de o ponteiro convertido ser válido, portanto, este exemplo invocará comportamento indefinido.

#include <stdio.h>

int main(void) {
    const char *hello = 'hello, world'; /* bad */
    puts(hello);
    return 0;
}

Neste exemplo, um literal de string é usado onde uma constante de caractere deve ser usada. O ponteiro convertido do literal de string será convertido em um inteiro de uma maneira definida pela implementação e será convertido em char de uma maneira definida pela implementação. (Como converter um inteiro para um tipo com sinal que não pode representar o valor a ser convertido é definido pela implementação, e se char é assinado também é definido pela implementação.) A saída será alguma coisa sem sentido.

#include <stdio.h>

int main(void) {
    char c = "a"; /* bad */
    printf("%c\n", c);
    return 0;
}

Em quase todos os casos, o compilador reclamará dessas confusões. Se isso não acontecer, você precisa usar mais opções de aviso do compilador ou é recomendável usar um compilador melhor.

Função recursiva — perdendo a condição base #

Calcular o fatorial de um número é um exemplo clássico de uma função recursiva.

Falta a condição básica:

#include <stdio.h>

int factorial(int n)
{
       return n * factorial(n - 1);
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Saída típica: Falha de segmentação: 11

O problema com essa função é que ela faria um loop infinito, causando uma falha de segmentação — ela precisa de uma condição básica para interromper a recursão.

Condição Base Declarada:

#include <stdio.h>

int factorial(int n)
{
    if (n == 1) // Base Condition, very crucial in designing the recursive functions.
    {
       return 1;
    }
    else
    {
       return n * factorial(n - 1);
    }
}

int main()
{
    printf("Factorial %d = %d\n", 3, factorial(3));
    return 0;
}

Saída de amostra

Factorial 3 = 6

Esta função terminará assim que atingir a condição n igual a 1 (desde que o valor inicial de n seja pequeno o suficiente — o limite superior é 12 quando int for uma quantidade de 32 bits).

Regras a serem seguidas:

  1. Inicialize o algoritmo. Programas recursivos geralmente precisam de um valor inicial para começar. Isso é feito usando um parâmetro passado para a função ou fornecendo uma função de gateway não recursiva, mas que configura os valores iniciais para o cálculo recursivo.
  2. Verifique se os valores atuais que estão sendo processados ​​correspondem ao caso base. Em caso afirmativo, processe e devolva o valor.
  3. Redefina a resposta em termos de um subproblema ou subproblemas menores ou mais simples.
  4. Execute o algoritmo no subproblema.
  5. Combine os resultados na formulação da resposta.
  6. Retorne os resultados.

Passando arrays não adjacentes para funções esperando arrays multidimensionais “reais” #

Ao alocar arrays multidimensionais com malloc, calloc e realloc, um padrão comum é alocar os arrays internos com várias chamadas (mesmo que a chamada apareça apenas uma vez, pode ser em um loop):

/* Could also be `int **` with malloc used to allocate outer array. */
int *array[4];
int i;

/* Allocate 4 arrays of 16 ints. */
for (i = 0; i < 4; i++)
    array[i] = malloc(16 * sizeof(*array[i]));

A diferença em bytes entre o último elemento de um dos arrays internos e o primeiro elemento do próximo array interno pode não ser 0 como seria com um array multidimensional “real” (por exemplo, int array[4][16]; ):

/* 0x40003c, 0x402000 */
printf("%p, %p\n", (void *)(array[0] + 15), (void *)array[1]);

Levando em conta o tamanho de int, você obtém uma diferença de 8128 bytes (8132-4), que são 2032 elementos de array do tamanho de int, e esse é o problema: um array multidimensional “real” não tem intervalos entre elementos.

Se você precisar usar um array alocado dinamicamente com uma função esperando um array multidimensional “real”, você deve alocar um objeto do tipo int * e usar aritmética para realizar os cálculos:

void func(int M, int N, int *array);
...

/* Equivalent to declaring `int array[M][N] = {{0}};` and assigning to array4_16[i][j]. */
int *array;
int M = 4, N = 16;
array = calloc(M, N * sizeof(*array));
array[i * N + j] = 1;
func(M, N, array);

Se N for uma macro ou um literal inteiro em vez de uma variável, o código pode simplesmente usar a notação de array 2-D mais natural depois de alocar um ponteiro para um array:

void func(int M, int N, int *array);
#define N 16
void func_N(int M, int (*array)[N]);
...

int M = 4;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;

/* Cast to `int *` works here because `array` is a single block of M*N ints with no gaps,
   just like `int array2[M * N];` and `int array3[M][N];` would be. */
func(M, N, (int *)array);
func_N(M, array);

Se N não for uma macro ou um literal inteiro, então array apontará para um array de comprimento variável (VLA). Isso ainda pode ser usado com func lançando para int * e uma nova função func_vla substituiria func_N:

void func(int M, int N, int *array);
void func_vla(int M, int N, int array[M][N]);
...

int M = 4, N = 16;
int (*array)[N];
array = calloc(M, sizeof(*array));
array[i][j] = 1;
func(M, N, (int *)array);
func_vla(M, N, array);

Observação: VLAs são opcionais a partir de C11. Se sua implementação suporta C11 e define a macro __STDC_NO_VLA__ como 1, você está preso aos métodos pré-C99.

Ultrapassando os limites do array #

As matrizes são baseadas em zero, ou seja, o índice sempre começa em 0 e termina com o comprimento da matriz de índice menos 1. Assim, o código a seguir não produzirá o primeiro elemento da matriz e produzirá lixo para o valor final impresso.

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 1; x <= 5; x++) //Looping from 1 till 5.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Saída: 2 3 4 5 GarbageValue

O seguinte demonstra a maneira correta de obter a saída desejada:

#include <stdio.h>

int main(void)
{
    int x = 0;
    int myArray[5] = {1, 2, 3, 4, 5}; //Declaring 5 elements

    for(x = 0; x < 5; x++) //Looping from 0 till 4.
       printf("%d\t", myArray[x]);

    printf("\n");
    return 0;
}

Saída: 1 2 3 4 5

É importante saber o comprimento de um array antes de trabalhar com ele, caso contrário você pode corromper o buffer ou causar uma falha de segmentação acessando locais de memória que estão fora dos limites.
