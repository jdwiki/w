Criar e incluir arquivos de cabeçalho

No C moderno, os arquivos de cabeçalho são ferramentas cruciais que devem ser projetadas e usadas corretamente. Eles permitem que o compilador faça uma verificação cruzada de partes compiladas independentemente de um programa.

Os cabeçalhos declaram tipos, funções, macros etc. que são necessários para os consumidores de um conjunto de instalações. Todo o código que usa qualquer uma dessas facilidades inclui o cabeçalho. Todo o código que define essas facilidades inclui o cabeçalho. Isso permite que o compilador verifique se os usos e as definições correspondem.

Introdução #

Há uma série de diretrizes a serem seguidas ao criar e usar arquivos de cabeçalho em um projeto C:

  • Idemopotência

Se um arquivo de cabeçalho for incluído várias vezes em uma unidade de tradução (TU), não deve quebrar compilações.

  • Autocontenção

Se você precisar das facilidades declaradas em um arquivo de cabeçalho, você não deve tem que incluir quaisquer outros cabeçalhos explicitamente.

  • Minimalidade

Você não poderá remover nenhuma informação de um cabeçalho sem fazendo com que as compilações falhem.

  • Incluir o que você usa (IWYU)

De maior preocupação para C++ do que para C, mas ainda assim importante em C também. Se o código em uma TU (chame-o de code.c) usa diretamente os recursos declarado por um cabeçalho (chame-o de "headerA.h"), então code.c deve #include "headerA.h" diretamente, mesmo que a TU inclua outro header (chame-o de "headerB.h") que acontece, no momento, incluir "headerA.h".

Ocasionalmente, pode haver razões suficientes para quebrar um ou mais dos essas diretrizes, mas vocês dois devem estar cientes de que estão quebrando o regra e esteja ciente das consequências de fazê-lo antes de quebrá-lo.

Autocontenção #

Cabeçalhos modernos devem ser autocontidos, o que significa que um programa que precisa usar os recursos definidos por header.h pode incluir esse cabeçalho (#include "header.h") e não se preocupar se outros cabeçalhos precisam ser incluído primeiro.

Recomendação: Os arquivos de cabeçalho devem ser autocontidos. #

Regras históricas #

Historicamente, este tem sido um assunto levemente controverso.

Era uma vez em outro milênio, o AT&T Indian Hill C Style and Coding Padrões declarou:

Os arquivos de cabeçalho não devem ser aninhados. O prólogo de um arquivo de cabeçalho deve, portanto, descrever quais outros os cabeçalhos precisam ser #included para que o cabeçalho seja funcional. Em casos extremos, onde um grande número de arquivos de cabeçalho deve ser incluído em vários arquivos de origem diferentes, é aceitável colocar todos os #includes comuns em um arquivo de inclusão.

Esta é a antítese da autocontenção.

Regras modernas #

No entanto, desde então, a opinião tendeu na direção oposta. Se um arquivo de origem precisar usar as facilidades declaradas por um cabeçalho header.h, o programador deve ser capaz de escrever:

#include "header.h"

e (sujeito apenas a ter os caminhos de pesquisa corretos definidos no comando linha), quaisquer cabeçalhos de pré-requisito necessários serão incluídos por header.h sem precisar de mais cabeçalhos adicionados ao arquivo de origem.

Isso fornece melhor modularidade para o código-fonte. Ele também protege a fonte do enigma “adivinhe por que este cabeçalho foi adicionado” que surge depois que o código foi modificado e hackeado por uma década ou duas.

Os padrões de codificação NASA Goddard Space Flight Center (GSFC) para C é um dos padrões mais modernos — mas agora é um pouco difícil de rastrear. Ele afirma que os cabeçalhos devem ser independentes. Ele também fornece uma maneira simples de garantir que os cabeçalhos sejam autocontidos: o arquivo de implementação para o cabeçalho deve incluir o cabeçalho como o primeiro cabeçalho. Se não for autocontido, esse código não será compilado.

A justificativa dada pelo GSFC inclui:

§2.1.1 Cabeçalho inclui justificativa

Este padrão requer que o cabeçalho de uma unidade contenha #include instruções para todos os outros cabeçalhos exigidos pelo cabeçalho da unidade. Colocar #include para o cabeçalho da unidade primeiro no corpo da unidade permite o compilador para verificar se o cabeçalho contém todos os instruções #include.

Um projeto alternativo, não permitido por esta norma, não permite instruções #include nos cabeçalhos; todos os #includes são feitos no corpo arquivos. Os arquivos de cabeçalho de unidade devem conter instruções #ifdef que verificam que os cabeçalhos necessários são incluídos na ordem correta.

Uma vantagem do design alternativo é que a lista #include em o arquivo do corpo é exatamente a lista de dependências necessária em um makefile e esta lista é verificada pelo compilador. Com o projeto padrão, uma ferramenta deve ser usada para gerar o lista de dependências. No entanto, todos os ambientes de desenvolvimento recomendados pela ramificação fornecer tal ferramenta.

Uma grande desvantagem do design alternativo é que, se o alterações na lista de cabeçalho necessárias, cada arquivo que usa essa unidade deve ser editado para atualizar a lista de instruções #include. Além disso, a lista de cabeçalhos necessária para uma unidade de biblioteca do compilador pode ser diferentes em diferentes alvos.

Outra desvantagem do design alternativo é que a biblioteca do compilador arquivos de cabeçalho e outros arquivos de terceiros devem ser modificados para adicionar o instruções #ifdef necessárias.

Assim, autocontenção significa que:

  • Se um cabeçalho header.h precisar de um novo cabeçalho aninhado extra.h, você não precisa tem que verificar cada arquivo fonte que usa header.h para ver se você precisa adicionar extra.h.
  • Se um cabeçalho header.h não precisar mais incluir um cabeçalho específico notneeded.h, você não precisa verificar todos os arquivos de origem que usam header.h para ver se você pode remover com segurança notneeded.h (mas veja [Incluir o que você usa](https://www.wikiod.com/pt/c/criar-e-incluir-arquivos-de-cabecalho /21641/incluir-o-que-você-usa-iwyu).
  • Você não precisa estabelecer a sequência correta para incluir o cabeçalhos de pré-requisito (o que requer uma classificação topológica para fazer o trabalho devidamente).

Verificando a autocontenção #

Consulte Vinculando com um estático biblioteca para um script chkhdr que pode ser usado para testar idempotência e autocontenção de um arquivo de cabeçalho.

Minimalidade #

Os cabeçalhos são um mecanismo crucial de verificação de consistência, mas devem ser o menor possível. Em particular, isso significa que um cabeçalho não deve incluir outros cabeçalhos apenas porque o arquivo de implementação precisará dos outros cabeçalhos. Um cabeçalho deve conter apenas os cabeçalhos necessários para um consumidor de os serviços descritos.

Por exemplo, um cabeçalho de projeto não deve incluir <stdio.h> a menos que um das interfaces de função usa o tipo FILE * (ou um dos outros tipos definidos exclusivamente em <stdio.h>). Se uma interface usa size_t, o menor cabeçalho que é suficiente é <stddef.h>. Obviamente, se outro cabeçalho que define size_t for incluído, há não é necessário incluir <stddef.h> também.

Se os cabeçalhos forem mínimos, ele manterá o tempo de compilação em um mínimo também.

É possível conceber cabeçalhos cujo único propósito é incluir muito de outros cabeçalhos. Isso raramente se torna uma boa ideia a longo prazo, porque poucos os arquivos de origem realmente precisarão de todas as facilidades descritas por todos os cabeçalhos. Por exemplo, um <standard-c.h> pode ser concebido que inclui todos os cabeçalhos C padrão — com cuidado, pois alguns cabeçalhos nem sempre são presente. No entanto, muito poucos programas realmente usam os recursos de <locale.h> ou <tgmath.h>.

Notação e Miscelânea #

O padrão C diz que há muito pouca diferença entre o notações #include <header.h> e #include "header.h".

[#include <header.h>] pesquisa uma sequência de lugares para um cabeçalho identificado exclusivamente pela sequência especificada entre os delimitadores < e >, e provoca a substituição daquele por todo o conteúdo do cabeçalho. Como os lugares são especificados ou o cabeçalho identificado é definido pela implementação.

[#include "header.h"] causa a substituição dessa diretiva por todo o conteúdo do arquivo de origem identificado pelo especificado seqüência entre os delimitadores "…". O arquivo de origem nomeado é pesquisado em um arquivo definido pela implementação maneira. Se esta pesquisa não for suportada, ou se a pesquisa falhar, a diretiva é reprocessado como se fosse [#include <header.h>] …

Assim, a forma de aspas duplas pode aparecer em mais lugares do que o forma de colchetes angulares. A norma especifica por exemplo que os cabeçalhos padrão devem ser incluído em colchetes angulares, mesmo que a compilação funcione se você usar aspas duplas em vez disso. Da mesma forma, padrões como POSIX usam o formato de colchetes angulares — e você também deveria. Reserve cabeçalhos com aspas duplas para cabeçalhos definidos pelo projeto. Para cabeçalhos definidos externamente (incluindo cabeçalhos de outros projetos seu projeto depende), a notação de colchetes angulares é mais apropriada.

Observe que deve haver um espaço entre #include e o cabeçalho, mesmo que os compiladores não aceitem nenhum espaço lá. Os espaços são baratos.

Vários projetos usam uma notação como:

#include <openssl/ssl.h>
#include <sys/stat.h>
#include <linux/kernel.h>

Você deve considerar se deve usar esse controle de namespace em seu projeto (provavelmente é uma boa idéia). Você deve evitar os nomes usados ​​pelos projetos existentes (em particular, tanto sys quanto linux seriam más escolhas).

Se você usar isso, seu código deve ser cuidadoso e consistente no uso da notação.

Não use a notação #include "../include/header.h".

Arquivos de cabeçalho raramente devem definir variáveis. Embora você mantenha as variáveis ​​globais no mínimo, se você precisar de um variável global, você a declarará em um cabeçalho e a definirá em um arquivo de origem adequado, e esse arquivo de origem incluirá o cabeçalho para verifique a declaração e definição, e todos os arquivos de origem que use a variável usará o cabeçalho para declará-la.

Corolário: você não declarará variáveis ​​globais em um arquivo de origem — um arquivo de origem conterá apenas definições.

Arquivos de cabeçalho raramente devem declarar funções static, com a notável exceção de funções static inline que serão definidas nos cabeçalhos se a função for necessária em mais de um arquivo de origem.

  • Os arquivos de origem definem variáveis ​​globais e funções globais.
  • Os arquivos fonte não declaram a existência de variáveis ​​ou funções globais; eles incluem o cabeçalho que declara a variável ou função.
  • Arquivos de cabeçalho declaram variáveis ​​e funções globais (e tipos e outros materiais de apoio).
  • Arquivos de cabeçalho não definem variáveis ​​ou quaisquer funções exceto (static) funções inline.

Idempotência #

Se um arquivo de cabeçalho específico for incluído mais de uma vez em uma unidade de tradução (TU), não deve haver problemas de compilação. Isso é chamado de ‘idempotência’; seus cabeçalhos devem ser idempotentes. Pense em como a vida seria difícil se você tivesse que garantir que #include <stdio.h> fosse incluído apenas uma vez.

Existem duas maneiras de alcançar a idempotência: guardas de cabeçalho e a diretiva #pragma once.

Protetores de cabeçalho #

Os protetores de cabeçalho são simples e confiáveis ​​e estão em conformidade com o padrão C. As primeiras linhas sem comentários em um arquivo de cabeçalho devem ter o formato:

#ifndef UNIQUE_ID_FOR_HEADER
#define UNIQUE_ID_FOR_HEADER

A última linha sem comentários deve ser #endif, opcionalmente com um comentário após:

#endif /* UNIQUE_ID_FOR_HEADER */

Todo o código operacional, incluindo outras diretivas #include, deve estar entre essas linhas.

Cada nome deve ser único. Freqüentemente, um esquema de nomes como HEADER_H_INCLUDED é usado. Alguns códigos antigos usam um símbolo definido como a proteção do cabeçalho (por exemplo, #ifndef BUFSIZ em <stdio.h>), mas não é tão confiável quanto um nome exclusivo.

Uma opção seria usar um hash MD5 (ou outro) gerado para o nome do protetor de cabeçalho. Você deve evitar emular os esquemas usados ​​pelos cabeçalhos do sistema que freqüentemente usam nomes reservados para a implementação — nomes que começam com um sublinhado seguido por outro sublinhado ou uma letra maiúscula.

A diretiva #pragma once #

Alternativamente, alguns compiladores suportam a diretiva #pragma once que tem o mesmo efeito que as três linhas mostradas para guardas de cabeçalho.

#pragma once

Os compiladores que suportam #pragma once incluem MS Visual Studio e GCC e Clang. No entanto, se a portabilidade for uma preocupação, é melhor usar protetores de cabeçalho ou usar ambos. Compiladores modernos (aqueles que suportam C89 ou posterior) são obrigados a ignorar, sem comentários, pragmas que eles não reconhecem (“Qualquer pragma que não seja reconhecido pela implementação é ignorado”), mas as versões antigas do GCC não eram tão indulgentes.

Incluir o que você usa (IWYU) #

O projeto Include What You Use do Google, ou IWYU, garante que os arquivos de origem incluam todos os cabeçalhos usados ​​no código.

Suponha que um arquivo fonte source.c inclua um cabeçalho arbitrary.h que por sua vez coincidentemente inclui freeloader.h, mas o arquivo fonte também usa explicitamente e independentemente as facilidades de freeloader.h. Tudo está bem para começar. Então um dia o arbitrary.h é alterado para que seus clientes não precisem mais das facilidades do freeloader.h. De repente, source.c para de compilar — porque não atendeu aos critérios da IWYU. Porque o código em source.c explicitamente usou as facilidades de freeloader.h, ele deveria ter incluído o que ele usa — deveria haver um #include "freeloader.h" explícito na fonte também. (Idempotência garantiria que não houvesse um problema.)

A filosofia IWYU maximiza a probabilidade de que o código continue a compilar mesmo com alterações razoáveis ​​feitas nas interfaces. Claramente, se o seu código chama uma função que é posteriormente removida da interface publicada, nenhuma preparação pode impedir que as alterações se tornem necessárias. É por isso que as alterações nas APIs são evitadas quando possível e por que existem ciclos de descontinuação em várias versões etc.

Este é um problema específico em C++ porque os cabeçalhos padrão podem incluir uns aos outros. O arquivo fonte file.cpp pode incluir um cabeçalho header1.h que em uma plataforma inclui outro cabeçalho header2.h. file.cpp pode vir a usar as facilidades de header2.h também. Isso não seria um problema inicialmente - o código seria compilado porque header1.h inclui header2.h. Em outra plataforma, ou uma atualização da plataforma atual, o header1.h pode ser revisado para que não inclua mais o header2.h, e então o file.cpp pararia de compilar como resultado.

A IWYU identificaria o problema e recomendaria que o header2.h fosse incluído diretamente no file.cpp. Isso garantiria que ele continuasse a compilar. Considerações análogas também se aplicam ao código C.
