Estofamento e embalagem da estrutura

  • Estruturas de embalagem
  • Estrutura de preenchimento

Nesta página

  • Estruturas de embalagem
  • Estrutura de preenchimento

Por padrão, os compiladores C estabelecem estruturas para que cada membro possa ser acessado rapidamente, sem incorrer em penalidades por ‘acesso desalinhado, um problema com máquinas RISC como o DEC Alpha e algumas CPUs ARM.

Dependendo da arquitetura da CPU e do compilador, uma estrutura pode ocupar mais espaço na memória do que a soma dos tamanhos de seus membros componentes. O compilador pode adicionar preenchimento entre membros ou no final da estrutura, mas não no início.

A embalagem substitui o preenchimento padrão.

Eric Raymond tem um artigo sobre [The Lost Art of C Structure Packing] (http://www.catb.org/esr/structure-packing/) que é uma leitura útil.

Estruturas de embalagem #

Por padrão, as estruturas são preenchidas em C. Se você quiser evitar esse comportamento, precisará solicitá-lo explicitamente. Sob GCC é __attribute__((__packed__)). Considere este exemplo em uma máquina de 64 bits:

struct foo {
    char *p;  /* 8 bytes */
    char c;   /* 1 byte  */
    long x;   /* 8 bytes */
};

A estrutura será automaticamente preenchida para ter alinhamento de 8 bytes e ficará assim:

struct foo {
    char *p;     /* 8 bytes */
    char c;      /* 1 byte  */

    char pad[7]; /* 7 bytes added by compiler */

    long x;      /* 8 bytes */
};

Então sizeof(struct foo) nos dará 24 em vez de 17. Isso aconteceu por causa de um compilador de 64 bits ler/escrever de/para a memória em 8 bytes de palavra em cada etapa e óbvio quando tentar escrever char c; um byte na memória um total de 8 bytes (ou seja, palavra) buscado e consumido apenas o primeiro byte dele e seus sete bytes sucessivos permanecem vazios e inacessíveis para qualquer operação de leitura e gravação para preenchimento de estrutura.

Embalagem da estrutura #

Mas se você adicionar o atributo packed, o compilador não adicionará preenchimento:

struct __attribute__((__packed__)) foo {
    char *p;  /* 8 bytes */
    char c;   /* 1 byte  */
    long x;   /* 8 bytes */
};

Agora sizeof(struct foo) retornará 17.

Geralmente são usadas estruturas compactadas:

  • Para economizar espaço.
  • Formatar uma estrutura de dados para transmitir pela rede sem depender do alinhamento de cada arquitetura de cada nó da rede.

Deve-se levar em consideração que alguns processadores como o ARM Cortex-M0 não permitem acesso desalinhado à memória; nesses casos, o empacotamento da estrutura pode levar a comportamento indefinido e pode travar a CPU.

Estrutura de preenchimento #

Suponha que este struct seja definido e compilado com um compilador de 32 bits:

struct test_32 {
    int a;      // 4 byte
    short b;    // 2 byte
    int c;      // 4 byte    
} str_32;

Podemos esperar que este struct ocupe apenas 10 bytes de memória, mas imprimindo sizeof(str_32) vemos que ele usa 12 bytes.

Isso aconteceu porque o compilador alinha variáveis ​​para acesso rápido. Um padrão comum é que quando o tipo base ocupa N bytes (onde N é uma potência de 2 como 1, 2, 4, 8, 16 — e raramente maior), a variável deve ser alinhada em um limite de N bytes ( um múltiplo de N bytes).

Para a estrutura mostrada com sizeof(int) == 4 e sizeof(short) == 2, um layout comum é:

  • int a; armazenado no deslocamento 0; tamanho 4.
  • short b; armazenado no deslocamento 4; tamanho 2.
  • preenchimento sem nome no deslocamento 6; tamanho 2.
  • int c; armazenado no deslocamento 8; tamanho 4.

Assim struct test_32 ocupa 12 bytes de memória. Neste exemplo, não há preenchimento à direita.

O compilador garantirá que quaisquer variáveis ​​struct test_32 sejam armazenadas começando em um limite de 4 bytes, para que os membros dentro da estrutura sejam alinhados adequadamente para acesso rápido. Funções de alocação de memória como malloc(), calloc() e realloc() são necessárias para garantir que o ponteiro retornado esteja suficientemente bem alinhado para uso com qualquer tipo de dados, portanto, estruturas alocadas dinamicamente também serão alinhadas adequadamente .

Você pode acabar em situações estranhas, como em um processador Intel x86_64 de 64 bits (por exemplo, Intel Core i7 — um Mac executando macOS Sierra ou Mac OS X), onde ao compilar no modo de 32 bits, os compiladores colocam double alinhados em um limite de 4 bytes; mas, no mesmo hardware, ao compilar em modo de 64 bits, os compiladores colocam double alinhado em um limite de 8 bytes.
