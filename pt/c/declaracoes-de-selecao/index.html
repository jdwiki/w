Aninhado if()…else VS if()..else Ladder #

Instruções aninhadas if()...else levam mais tempo de execução (são mais lentas) em comparação com uma escada if()...else porque as instruções aninhadas if()...else verificam todos os instruções condicionais internas uma vez que a instrução condicional externa if() é satisfeita, enquanto a escada if()..else parará o teste de condição uma vez que qualquer uma das instruções condicionais if() ou else if() são verdadeiros.

Uma escada if()...else:

#include <stdio.h>

int main(int argc, char *argv[])
{
  int a, b, c;
  printf("\nEnter Three numbers = ");
  scanf("%d%d%d", &a, &b, &c);
  if ((a < b) && (a < c))
  {
    printf("\na = %d is the smallest.", a);
  }
  else if ((b < a) && (b < c))
  {
    printf("\nb = %d is the smallest.", b);
  }
  else if ((c < a) && (c < b))
  {
    printf("\nc = %d is the smallest.", c);
  }
  else
  {
    printf("\nImprove your coding logic");
  }
  return 0;
}

É, no caso geral, considerado melhor que o equivalente aninhado if()...else:

#include <stdio.h>

int main(int argc, char *argv[])
{
  int a, b, c;
  printf("\nEnter Three numbers = ");
  scanf("%d%d%d", &a, &b, &c);
  if (a < b)
  {
    if (a < c)
      {
        printf("\na = %d is the smallest.", a);
      }
    else
      {
        printf("\nc = %d is the smallest.", c);
      }
  }
  else
  {
    if(b < c)
    {
      printf("\nb = %d is the smallest.", b);
    }
    else
    {
      printf("\nc = %d is the smallest.", c);
    }
  }
  return 0;  
}

if () Declarações #

Uma das maneiras mais simples de controlar o fluxo do programa é usando instruções de seleção if. Se um bloco de código deve ser executado ou não pode ser decidido por esta instrução.

A sintaxe para a instrução de seleção if em C pode ser a seguinte:

if(cond) 
{
  statement(s);  /*to be executed, on condition being true*/
}

Por exemplo,

if (a > 1) {
    puts("a is larger than 1");
}

Onde a > 1 é uma condição que deve ser avaliada como true para executar as instruções dentro do bloco if. Neste exemplo, “a é maior que 1” é impresso apenas se a > 1 for verdadeiro.

As instruções de seleção if podem omitir as chaves { e } se houver apenas uma instrução dentro do bloco. O exemplo acima pode ser reescrito para

if (a > 1)
    puts("a is larger than 1");

No entanto, para executar várias instruções dentro do bloco, as chaves devem ser usadas.

A condição para if pode incluir várias expressões. if só executará a ação se o resultado final da expressão for verdadeiro.

Por exemplo

if ((a > 1) && (b > 1)) {
    puts("a is larger than 1");
    a++;
}

só executará printf e a++ se ambos a e b forem maiores que 1.

switch () Declarações #

As instruções switch são úteis quando você deseja que seu programa faça muitas coisas diferentes de acordo com o valor de uma variável de teste específica.

Um exemplo de uso da instrução switch é assim:

int a = 1;

switch (a) {
case 1:
    puts("a is 1");
    break;
case 2:
    puts("a is 2");
    break;
default:
    puts("a is neither 1 nor 2");
    break;
}

Este exemplo é equivalente a

int a = 1;

if (a == 1) {
    puts("a is 1");
} else if (a == 2) {
    puts("a is 2");
} else {
    puts("a is neither 1 nor 2");
}

Se o valor de a for 1 quando a instrução switch for usada, a é 1 será impresso. Se o valor de a for 2, então, a é 2 será impresso. Caso contrário, a não é 1 nem 2 será impresso.

case n: é usado para descrever onde o fluxo de execução irá saltar quando o valor passado para a instrução switch for n. n deve ser constante em tempo de compilação e o mesmo n pode existir no máximo uma vez em uma instrução switch.

default: é usado para descrever quando o valor não corresponde a nenhuma das opções para case n:. É uma boa prática incluir um caso default em cada instrução switch para detectar comportamentos inesperados.

Uma instrução break; é necessária para sair do bloco switch.

Observação: Se você acidentalmente esquecer de adicionar um break após o final de um case, o compilador assumirá que você pretende “fall through” e todas as instruções case subsequentes, se any, será executado (a menos que uma instrução break seja encontrada em qualquer um dos casos subsequentes), independentemente de as instruções case subsequentes corresponderem ou não. Esta propriedade particular é usada para implementar o [Dispositivo de Duff][3]. Esse comportamento é frequentemente considerado uma falha na especificação da linguagem C.

Abaixo está um exemplo que mostra os efeitos da ausência de break;:

int a = 1;

switch (a) {
case 1:
case 2:
    puts("a is 1 or 2");
case 3:
    puts("a is 1, 2 or 3");
    break;
default:
    puts("a is neither 1, 2 nor 3");
    break;
}

Quando o valor de a for 1 ou 2, a é 1 ou 2 e a é 1, 2 ou 3 serão ambos impressos. Quando a é 3, apenas a é 1, 2 ou 3 será impresso. Caso contrário, a não é 1, 2 nem 3 será impresso.

Observe que o caso default não é necessário, especialmente quando o conjunto de valores que você obtém no switch é finalizado e conhecido em tempo de compilação.

O melhor exemplo é usar um switch em um enum.

enum msg_type { ACK, PING, ERROR };
void f(enum msg_type t)
{
  switch (t) {
  case ACK:
    // do nothing
    break;
  case PING:
    // do something
    break;
  case ERROR:
    // do something else
    break;
  }
}

Existem várias vantagens em fazer isso:

  • a maioria dos compiladores reportará um aviso se você não manipular um valor (isso não seria reportado se um caso default estivesse presente)
  • pelo mesmo motivo, se você adicionar um novo valor ao enum, você será notificado de todos os lugares onde você esqueceu de lidar com o novo valor (com um caso default, você precisaria explorar manualmente seu código procurando por tais casos)
  • O leitor não precisa descobrir “o que está oculto pelo default:”, se existem outros valores de enum ou se é uma proteção para “just in case”. E se houver outros valores enum, o codificador usou intencionalmente o caso default para eles ou existe um bug que foi introduzido quando ele adicionou o valor?
  • manipular cada valor enum torna o código autoexplicativo, pois você não pode se esconder atrás de um curinga, você deve lidar explicitamente com cada um deles.

No entanto, você não pode impedir que alguém escreva códigos malignos como:

enum msg_type t = (enum msg_type)666; // I'm evil

Assim, você pode adicionar uma verificação extra antes do switch para detectá-lo, se realmente precisar.

void f(enum msg_type t)
{
   if (!is_msg_type_valid(t)) {
      // Handle this unlikely error
   }

   switch(t) { 
    // Same code than before
   }
}

[3]: https://www.wikiod.com/pt/c/instrucoesloops-de-iteracao-para-while-do-while#Desenrolar Loop e Dispositivo de Duff

if () … else instruções e sintaxe #

Enquanto if executa uma ação somente quando sua condição é avaliada como true, if / else permite que você especifique as diferentes ações quando a condição for true e quando a condição for false.

Exemplo:

if (a > 1)
    puts("a is larger than 1");
else 
    puts("a is not larger than 1");

Assim como a instrução if, quando o bloco dentro de if ou else consiste em apenas uma instrução, as chaves podem ser omitidas (mas isso não é recomendado, pois pode facilmente introduzir problemas involuntariamente). No entanto, se houver mais de uma instrução dentro do bloco if ou else, as chaves devem ser usadas nesse bloco específico.

if (a > 1) 
{
    puts("a is larger than 1");
    a--;
}
else 
{
    puts("a is not larger than 1");
    a++;
}

if()…else Ladder Encadeando duas ou mais instruções if() … else #

Enquanto a instrução if ()... else permite definir apenas um comportamento (padrão) que ocorre quando a condição dentro do if () não é atendida, encadeando dois ou mais if () ... else instruções permitem definir mais alguns comportamentos antes de ir para a última ramificação else agindo como um “padrão”, se houver.

Exemplo:

int a = ... /* initialise to some value. */

if (a >= 1) 
{
    printf("a is greater than or equals 1.\n");
} 
else if (a == 0) //we already know that a is smaller than 1
{
    printf("a equals 0.\n");
}
else /* a is smaller than 1 and not equals 0, hence: */
{ 
    printf("a is negative.\n");
}
