Comparando literal e variável #

Suponha que você esteja comparando valor com alguma variável

if ( i  == 2) //Bad-way
{
    doSomething;
}

Agora suponha que você tenha confundido == com =. Então levará seu tempo doce para descobrir isso.

if( 2 == i) //Good-way
{
    doSomething;
}

Então, se um sinal de igual for acidentalmente omitido, o compilador reclamará sobre uma “tentativa de atribuição ao literal”. Isso não vai te proteger ao comparar duas variáveis, mas cada pouquinho ajuda.

Veja aqui para mais informações.

Não deixe a lista de parâmetros de uma função em branco — use void #

Suponha que você esteja criando uma função que não requer argumentos quando é chamada e se depara com o dilema de como definir a lista de parâmetros no protótipo da função e na definição da função.

  • Você tem a opção de manter a lista de parâmetros vazia tanto para protótipo quanto para definição. Assim, eles se parecem com a instrução de chamada de função que você precisará.

  • Você leu em algum lugar que um dos usos da palavra-chave void (existem apenas alguns deles), é definir a lista de parâmetros de funções que não aceitam nenhum argumento em sua chamada. Então, isso também é uma escolha.

Então, qual é a escolha correta?

RESPOSTA: usando a palavra-chave void

CONSELHO GERAL: se um idioma fornecer determinado recurso para um propósito especial, é melhor usá-lo em seu código. Por exemplo, usando enums em vez de macros #define (isso é para outro exemplo).

C11 seção 6.7.6.3 “Declaradores de função”, parágrafo 10, afirma:

O caso especial de um parâmetro sem nome do tipo void como o único item na lista especifica que a função não possui parâmetros.

O parágrafo 14 dessa mesma seção mostra a única diferença:

… Uma lista vazia em um declarador de função que faz parte de uma definição dessa função especifica que a função não possui parâmetros. A lista vazia em um declarador de função que não faz parte de uma definição dessa função especifica que nenhuma informação sobre o número ou tipos dos parâmetros é fornecida.

Uma explicação simplificada fornecida por K&R (pgs-72-73) para as coisas acima:

Além disso, se uma declaração de função não incluir argumentos, como em duplo atof();, isso também significa que nada deve ser assumido sobre os argumentos de atof; toda a verificação de parâmetros está desativada. Este significado especial da lista de argumentos vazia destina-se a permitir que programas C mais antigos compilem com novos compiladores. Mas é uma má ideia usá-lo com novos programas. Se a função receber argumentos, declare-os; se não receber argumentos, use void.

Então é assim que seu protótipo de função deve ficar:

int foo(void);

E é assim que a definição da função deve ser:

int foo(void)
{
    ...
    <statements>
    ...
    return 1;
}

Uma vantagem de usar o acima, sobre o tipo de declaração int foo() (ou seja, sem usar a palavra-chave void), é que o compilador pode detectar o erro se você chamar sua função usando uma instrução errônea como foo(42). Esse tipo de instrução de chamada de função não causaria nenhum erro se você deixar a lista de parâmetros em branco. O erro passaria silenciosamente, sem ser detectado e o código ainda seria executado.

Isso também significa que você deve definir a função main() assim:

int main(void)
{
    ...
    <statements>
    ...
    return 0;
}

Observe que, embora uma função definida com uma lista de parâmetros vazia não receba argumentos, ela não fornece um protótipo para a função, portanto, o compilador não reclamará se a função for chamada posteriormente com argumentos. Por exemplo:

#include <stdio.h>

static void parameterless()
{
    printf("%s called\n", __func__);
}

int main(void)
{
    parameterless(3, "arguments", "provided");
    return 0;
}

Se esse código for salvo no arquivo proto79.c, ele pode ser compilado no Unix com GCC (versão 7.1.0 no macOS Sierra 10.12.5 usado para demonstração) assim:

$ gcc -O3 -g -std=c11 -Wall -Wextra -Werror -Wmissing-prototypes -pedantic proto79.c -o proto79
$

Se você compilar com opções mais rigorosas, receberá erros:

$ gcc -O3 -g -std=c11 -Wall -Wextra -Werror -Wmissing-prototypes -Wstrict-prototypes -Wold-style-definition -pedantic proto79.c -o proto79 
proto79.c:3:13: error: function declaration isn’t a prototype [-Werror=strict-prototypes]
 static void parameterless()
             ^~~~~~~~~~~~~
proto79.c: In function ‘parameterless’:
proto79.c:3:13: error: old-style function definition [-Werror=old-style-definition]
cc1: all warnings being treated as errors
$

Se você der à função o protótipo formal static void parameterless(void), a compilação dará erros:

$ gcc -O3 -g -std=c11 -Wall -Wextra -Werror -Wmissing-prototypes -Wstrict-prototypes -Wold-style-definition -pedantic proto79.c -o proto79 
proto79.c: In function ‘main’:
proto79.c:10:5: error: too many arguments to function ‘parameterless’
     parameterless(3, "arguments", "provided");
     ^~~~~~~~~~~~~
proto79.c:3:13: note: declared here
 static void parameterless(void)
             ^~~~~~~~~~~~~
$

Moral – sempre certifique-se de ter protótipos e certifique-se de que seu compilador lhe diga quando você não está obedecendo as regras.
