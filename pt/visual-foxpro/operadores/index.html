No VFP, os operadores são agrupados naqueles:

  • Operadores Numéricos
  • Operadores lógicos
  • Operadores de personagens

Também existem operadores, implementados como funções (como operações bit a bit, comparação de objetos …).

Vamos olhar para cada um por exemplo.

Operadores Numéricos #

Os operadores numéricos são os mais fáceis e quase os mesmos que em outras linguagens.

  • +, -, * e /. Operadores de adição, subtração, multiplicação e divisão (no VFP não há divisão inteira, você pode converter um resultado para inteiro com as funções INT(), CEILING() e FLOOR()).
  • Operador % Módulo. *^ e **. Poder do(s) operador(es). Ambos fazem a mesma coisa.
  • (). Operadores de agrupamento.
  • Os operadores têm precedência. A ordem é:

( ) ^ (ou **) / e *

  • e +
    ? 10 / 5 + 2 && Outputs 4
    ? 2 + 10 / 5 && Outputs 4 as well. Division has precedence.
     
    * Both multiplication and division have same precedence
    * They would be interpreted from left to right.
    ? 4 * 5 / 2 + 5 && Outputs 15
    * Use parentheses whenever you are in doubt or want to be explicit
    ? ( (4 * 5) / 2 ) + 5 && Outputs 15. Explicit grouping of operations

    ? 4 * 5^2 && ^ has precedence, this is same as 4 * (5^2) = 100.
    ? (4 + 5)^2 && Using parentheses we say add 5 to 4 (9) and then square = 81.

Operadores lógicos #

Os operadores lógicos no VFP, em sua ordem de precedência, são:

Operador Descrição
( ) Parênteses, expressões de grupos
NÃO, ! Negue logicamente a expressão. NÃO ou ! não tem diferença.
E Logicamente E as expressões
OU Logicamente OR as expressões
<>, !=, # Verifique a desigualdade. Assim mesmo como lógico exclusivo OR - XOR

Historicamente, NOT, AND, OR são escritos como .NOT., .AND., .OR. Você ainda pode usá-los se quiser, mas AND, OR, NOT é mais simples e limpo.

Para false e true, você deve usar .F. e T. literais respectivamente. Você não pode optar por usar F e T em vez disso.

* Some logical variables
local llOld, llEmail  && any variable declaration implicitly initializes the variable as .F. - false
? m.llOld, m.llEmail && Prints .F. .F.

llOld   = .T.
llEmail = .F.

if ( m.llOld AND m.llEmail )
   ? 'Old AND should be emailed to'
endif
if ( m.llOld OR m.llEmail )
   ? 'Old OR should be emailed to'
endif
if ( m.llOld AND !m.llEmail ) && Same as (m.llOld AND NOT m.llEmail)
   ? 'Old BUT should NOT be emailed to'
endif

* Above code outputs
Old OR should be emailed to
Old BUT should NOT be emailed to

No VFP, as expressões lógicas são avaliadas de forma abreviada. Ou seja, se a primeira parte da verificação satisfizer todo o resultado, o restante da expressão nem será interpretado. Segue uma amostra:

? 1 = '2' && An obvious error. It would complain operator/operand type mismatch.

* However we could use such an expression in an if and get no error
* because it is not interpreted at all 
* (VFP is dynamic and there is no compile time check)

local llProcess
llProcess = .T.

if (m.llProcess OR (1='2'))
   ? 'Should do processing'
endif

* Would output

Should do processing

* without any error because m.llProcess true means
* the whole expression would be true, thus the expression after OR 
* is not interpreted at all.

Uma armadilha que pega os novatos é que, às vezes, você pode precisar de várias verificações, digamos em uma consulta SQL, que estão conectadas com os operadores AND, OR. Quando há muitos deles, pode-se ignorar o fato de que os operadores têm precedência (na ordem (), NOT, AND, OR) e pensar que a interpretação seria feita da esquerda para a direita em uma cadeia. Considere uma amostra:

selecione * de myTable onde !isCliente AND débito > 5000 OU desconto > 5

qual é a intenção desta consulta? Se tornarmos explícito usando parênteses de agrupamento, ele diz:

((NÃO éCliente) E débito > 5000) OU desconto > 5 simplificado parece “firstExpression” OU (desconto > 5). Qualquer que fosse a intenção, por causa disso OU selecionaria: todas as linhas que possuem (desconto > 5) - e também aquelas em que é cliente com débito acima de 5000.

Provavelmente a intenção era “dar-me aqueles em que NÃO é cliente E (débito superior a 5000 OU desconto superior a 5)”. Ficaria claro desde o início se usássemos parênteses:

selecione * from myTable onde !isCustomer AND (débito > 5000 OR desconto > 5) Você pode usar, mas não vale a pena ter parênteses para o operador NOT inicial, quando seu operando é uma única expressão é legível o suficiente com sua precedência - !isCustomer é claramente lido como (NÃO isCustomer).

Operadores de caracteres #

Existem apenas 4 operadores de caracteres, em sua ordem de precedência:

Operador Descrição
() Parênteses para agrupamento. Nota: a documentação do VFP, que eu tenho, perde esta. Sem isso, o operador - é quase sempre inútil.
+ Concatena (junta) strings lado a lado.
- Concatena strings movendo os espaços à direita da string esquerda para o final da string direita.
$ Verifica se a primeira string está contida na segunda.

+ é o mais simples e também usado para concatenar strings em muitas outras linguagens.

local firstName, lastName
firstName = "John"
lastName  = "Smith"

? m.firstName + " " + m.lastName

Saídas: John Smith

- é um pouco complicado e não é amplamente conhecido. Ele pega espaços à direita da string da esquerda, acrescenta esses espaços à string da direita. Suponha que você tenha uma tabela com nomes e sobrenomes e cada um deles tenha 20 caracteres. Queremos concatenar nomes e sobrenomes para criar um nome completo, e também queremos que o tamanho resultante seja fixo (neste caso 20 + 20 + 1 espaço = 41). Vamos fazer com que você também tenha uma coluna de nome do meio e queremos que o nome completo se pareça com “lastName, firstName middleName_______”. É mais fácil fazer isso usando o operador -, mas você deve observar o truque de usar parênteses aqui para agrupar, para obter exatamente o que queremos:

* Create a cursor for our sample and fill in a few names
Create Cursor Names (firstName c(20), midName c(20), lastName c(20))

Insert Into Names (firstName, midName, lastName) Values ('Cetin','', 'Basoz')
Insert Into Names (firstName, midName, lastName) Values ('John', 'M', 'Smith')
Insert Into Names (firstName, midName, lastName) Values ('John', 'F', 'Kennedy')
Insert Into Names (firstName, midName, lastName) Values ('Tom', '', 'Hanks')

* Select with tricky - operator
Select *, ;
    lastName - (', '+firstName-(' '+midName)) As FullName ;
    from Names ;
    INTO Cursor crsNames ;
    nofilter

Browse

E a saída é assim:

firstName midName sobrenome nomecompleto
Cetina Por floresta Por floresta, Cetin
João M Smith Smith, John M
João F Kennedy Kennedy, John F
Tom Hanks Hanks, Tom

Na coluna fullName, todos os espaços à direita são empurrados bem para o final. Se você verificar a estrutura, a coluna fullName tem 63 caracteres de largura (3 * 20 + 3 caracteres que adicionamos).

Observe a importância de agrupar os parênteses (tente remover os parênteses ou organizando de uma maneira diferente).

Embora o operador - possa ser tentador em tais casos, há o outro lado da moeda. Esse operador é específico do VFP e, portanto, o SQL não é portátil. Você pode obter o mesmo resultado por este SQL compatível com ANSI:

Select *, ;
    CAST(RTRIM(lastName) +', '+ RTRIM(firstName) +' '+ midName as char(63)) As FullName ;
    from Names ;
    INTO Cursor crsNames ;
    nofilter

O último operador é $. Ele simplesmente verifica se a string esquerda faz parte da string direita.

local upcased, digits, hexDigits
upcased = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ'
digits  = '0123456789'
hexDigits  = m.digits + 'ABCDEF'

? 'A' $ m.upcased && .T.
? 'a' $ m.upcased && .F.
? '1' $ m.digits && .T.
? 'F' $ m.digits && .F.
? 'F' $ m.hexDigits && .T.

Importante: no VFP, embora você possa escrever seu código em qualquer caso (maior, inferior ou misto), as strings sempre diferenciam maiúsculas de minúsculas. Por exemplo: “Smith” e “smith” são dois valores distintos. Ou em sua tabela, se houver uma coluna de país, você não encontrará ‘EUA’ se pesquisar com ‘usa’. O mesmo acontece com o operador $, “GE” $ “Alemanha” é falso.

Nota pessoal: Embora você possa gostar de $ por sua simplicidade e possa encontrá-lo frequentemente usado em códigos-fonte da Microsoft, IMHO é de muito pouco valor. Pensando em muitos milhares de linhas que escrevi em minha operadora, acho que encontraria muito poucas ocorrências disso em meu próprio código. Quase sempre há uma alternativa melhor (especialmente quando o operando esquerdo não é um único caractere e\ou a distinção entre maiúsculas e minúsculas).

Operadores de data e hora #

Existem basicamente dois operadores para valores de data e hora. + e - estão sobrecarregados (provavelmente um termo C) para fazer cálculos de data/data e hora:

Operador Descrição
+ Adiciona dias (data) ou segundos (datahora) a um valor de data/datahora.
- Obtém a diferença de dois valores de data/datahora. Subtrai dias (data) ou segundos (datahora) dos valores de data e hora.

+ é o mais fácil. Ele tem dois operandos, um é um valor de data ou data e hora e o outro é um numérico (embora você possa usar qualquer numérico, é um número inteiro para todos os propósitos práticos).

Quando um dos operandos é uma data, então o operando numérico é tomado como “dia”:

? Date() + 10 && Get the date 10 days later
* VFP is leap year aware when doing date math
? Date(2016, 2, 28) + 1 && Add 1 day to Feb 28, 2016. Returns Feb 29, 2016.
? Date(2017, 2, 28) + 1 && Add 1 day to Feb 28, 2017. Returns Mar 1, 2017.

Quando um dos operandos é um datetime, então o operando numérico é tomado como “segundo”:

Há 24 * 60 * 60 = 86.400 segundos em um dia

? Datetime() + 86400 && Add 1 day to current datetime.

Adicionar 4 horas e 15 minutos a 1º de janeiro de 2016 14h20

? Datetime(2016, 1, 1, 14, 20, 0) + (4 * 3600 + 15 * 60)

Saídas sexta-feira, 1º de janeiro de 2016, 18h35.

Com uma impressão simples usando ?, o que você vê na tela depende de suas configurações de data. Por exemplo, se você não alterou nada, suas configurações de data são de estilo americano (MDY), formato de 12 horas (AM/PM) e o século é mostrado apenas com os últimos 2 dígitos.

Existe um símbolo especial ^ para data e data/hora forçando uma string a ser interpretada ’estritamente’ como formato aaaa/MM/dd [HH:mm:ss|hh:mm:ss tt]. Portanto, ^ também pode ser considerado como operador de data/data/hora. Por exemplo, considere que alguns dados são recebidos de uma fonte em um formato como 201610082230 (aaaaMMddHHmm). Para obter esse valor como um Datetime válido:

Local cSample, tSample
cSample = '201610082230'
tSample = Ctot(Transform(m.cSample, '@R ^9999/99/99 99:99'))
? Transform(m.tSample, '@YL')

Saídas (dependendo da configuração de data longa do seu sistema):

Sábado, 8 de outubro de 2016, 22h30

- é usado para subtração. Seus operandos são ambos valores de data/data e hora OU um é data/data e hora e o outro é numérico.

Vamos começar com a data/datahora e os operandos numéricos mais simples (como com o operador +):

Quando um dos operandos é uma data, então o operando numérico é tomado como “dia”:

? Date() - 10 && What was the date 10 days ago?
? Date(2016, 3, 1) - 1 && Returns Feb 29, 2016.
? Date(2017, 3, 1) - 1 && Returns Feb 28, 2017.

Quando um dos operandos é um datetime, então o operando numérico é tomado como “segundo”:

? Datetime() - 86400 && Go back exactly one day

Obter 1 hora e 30 minutos atrás de “agora”:

? Datetime() - (1 * 3600 + 30 * 60)

A segunda forma é obter a diferença entre dois valores de data/data e hora. Operandos são tanto de data quanto de data e hora, você não pode usar data e data e hora ao mesmo tempo (faça a conversão de tipo conforme necessário, o VFP não faz isso por você). As regras são como em + e -, operandos são data, então a diferença é em dias, operandos são data e hora, então a diferença é em segundos.

Quantos dias para a véspera de ano novo (para o ano de 2016)?

? Date(2016, 12, 31) - Date()

Quantos segundos faltam para a meia-noite?

? Dtot(Date()+1) - Datetime()

No último exemplo usamos uma função Date/Datetime, DTOT - DateToTime para obter o valor da meia-noite de amanhã. Existem muitas funções úteis de data/data e hora disponíveis, nós ignoramos todas elas, pois tecnicamente não são consideradas operadores (embora operem em data/hora:) O mesmo vale para outros operadores também.

A subtração de data/data/hora é assinada. Ou seja, se você usar a data/datahora menor como o primeiro operando, o resultado será negativo. Você pode usar a função abs() se precisar obter um resultado positivo, independentemente da ordem de data/data/hora.

Operadores relacionais #

De todos os operadores, os relacionais são os mais complexos, por isso os deixamos para o final.

Os operadores relacionais também são conhecidos como operadores de comparação, são usados ​​para comparar coisas.

O resultado da comparação é booleano falso ou verdadeiro.

Curiosamente, porém, se você verificar no VFP, você verá apenas um pequeno operações de lista e mais algumas linhas como se fosse tudo sobre isso operadores.

Bem, a complexidade vem do fato de que eles operam em qualquer tipo seja um numérico, data, data e hora, lógico ou uma string, e até mesmo em objetos. Além disso, o comportamento pode parecer estranho, você não obtém o que espera a menos que saiba o que afeta os resultados.

Vamos começar com uma lista de operadores relacionais:

Operador Descrição MOST Amostra básica
> Maior que ? 1 > 2 && .F.
< Menos de ? 1 < 2 && .T.
>= Maior ou igual a ? 1 >= 2 && .F.
<= Menor ou igual a ? 1 <= 2 && .T.
= Igual a ? 1 = 1 && .T.
== É exatamente igual a (faz sentido para strings) ? ‘1’ = ‘1’ && .T.
!=, #, <> Diferente de (todos os 3 operadores agem da mesma forma, escolha o seu favorito) ? 1 != 1 && .F.

Embora você possa usá-los com todos os tipos de dados, deve haver uma compatibilidade de tipo entre os operandos. Por exemplo, você obteria um erro se tentasse comparar uma data com um inteiro.

Date e Datetime podem ser comparados, embora sejam tipos diferentes, o VFP faz a conversão implicitamente para você.

? Date() > DateTime() && .F. 
? Date() <= DateTime() && .T. 
? Date() < DateTime() && .T. if it is not midnight

Quando os operandos são numéricos, todos esses operadores são simples e diretos, funcionam como na expressão matemática.

Com os operandos lógicos, .F. é considerado inferior a .T.

Com objetos o que estamos comparando é a referência do objeto na memória. Assim, a comparação mais utilizada é determinar se duas variáveis ​​de objeto estão apontando para o mesmo objeto. ou seja:

local o1, o2
o1 = createobject('Label')
o2 = createobject('Label')
? m.o1 = m.o2 && is o1 and o2 the same object?
? m.o1 > m.o2 && this would work too but likely you would never use

* remember we are comparing their references in memory
* 
* They are different objects, but do they have any difference in their properties?
? CompObj(m.o1, m.o2) && .T. They are identical properties wise

Comparação de tipo de dados de caracteres, também conhecida como comparação de strings, é a mais confusa no VFP. Não funciona como em outras linguagens e/ou bancos de dados e é exclusivo do VFP (e talvez de alguma outra linguagem xBase).

Muitos anos atrás, eu vi alguns membros realmente avançados na comunidade que ainda não sabiam como esses operadores trabalham no VFP. Portanto, é bastante compreensível que pequenas nuances possam confundir os novatos facilmente.

A comparação é basicamente sobre ser igual ou não. Se não forem iguais, podemos pensar nos operadores >, <, >=, <=, certo? Com strings é confuso quando duas strings são consideradas iguais.

Importante: as strings VFP diferenciam maiúsculas de minúsculas. ‘A’ e ‘a’ são duas strings distintas. Este não é o caso de muitos bancos de dados em que o padrão é usar uma ordenação que não diferencia maiúsculas de minúsculas. Por exemplo, em postgreSQL ou MS SQL Server em uma tabela criada com agrupamento sem distinção entre maiúsculas e minúsculas (CI):

select * from myTable where Country = 'TURKEY'

select * from myTable where Country = 'Turkey'

daria o mesmo resultado. No VFP, porém, você só obtém aqueles em que o revestimento corresponde. No entanto, o VFP tem algum suporte de agrupamento e faz uma comparação que não diferencia maiúsculas de minúsculas. (Não confie, veja abaixo)

  • Se duas strings não forem iguais, até agora tudo bem, desde que você não alterou nenhum padrão, elas serão comparadas com base em seus valores ASCII.

    ? 'Basoz' < 'Cetin' && is true.
? 'basoz' < 'Cetin' && is false.
? 'Cetin' < 'David' && is true.
? 'Çetin' < 'David' && is false.

O padrão para agrupamento é ‘máquina’ e é isso que você obtém então. Quando você altera o agrupamento para outra coisa, obtém a comparação com base na ordem de classificação desse agrupamento. Com a configuração de agrupamento diferente da máquina padrão, você também está sugerindo uma diferenciação entre maiúsculas e minúsculas na comparação (NÃO confie nisso para igualdade):

  set collate to 'GENERAL'
  ? 'Basoz' < 'Cetin'
  ? 'basoz' < 'Cetin'
  ? 'Cetin' < 'David'
  ? 'Çetin' < 'David'

Agora todas essas expressões são TRUE.

Conselho Pessoal: Os agrupamentos no VFP nunca foram confiáveis ​​o suficiente. Eu sugiro que você não use collations e fique com o padrão ‘MACHINE’. Se você usar agrupamentos, lembre-se de verificar primeiro quando ocorrer algo muito inesperado em relação aos dados de caracteres. Eu vi e demonstrei que ele falha em muitos casos, mas depois parei de tentar usá-lo muito antes da versão VFP9, pode ser consistente agora, realmente não sei.

Considerando que cobrimos casos de desigualdade com strings, o complicado é o caso de igualdade. No VFP basicamente duas configurações efetuam a comparação:

  1. SET EXACT (O padrão é OFF e efetua comparações regulares - exceto SQL)
  2. SET ANSI (O padrão é OFF e efetua comparações somente em SQL. SET EXACT não tem efeito em comparações feitas em consultas SQL.

Com SET EXACT OFF, leia a comparação como “a string à direita começa com a string à esquerda”? Eles são comparados até o comprimento da corda certa.

? "Bobby" = "B" && Bobby starts with B, so TRUE
? "Bobby" = "Bob" && Bobby starts with Bob, so TRUE
? "Bobby" = "Bob " && Bobby starts with Bob but there is a trailing space there, FALSE
? "Bobby" = "bob" && would be true with collation set to GENERAL

Observe que com comparação regular, “Bobby” = “B” é VERDADEIRO, mas “B” = “Bobby” é FALSO. Em outras palavras, o lugar dos operandos é importante.

Com SET EXACT ON, as strings devem corresponder totalmente, mas seus espaços à direita são ignorados (estamos ignorando set collate aqui, o que também não diferencia maiúsculas de minúsculas):

? "BOBBY" = "BOB" && FALSE 
? "BOBBY" = "BOBBY" && TRUE
? "BOBBY" = "BOBBY     " && TRUE 
? "BOBBY     " = "BOBBY" && TRUE

Agora, com comandos SQL, SET EXACT não tem efeito e se comportaria como SET EXACT OFF.

Select * from Customers where Country = 'U'

Selecionaria os clientes dos EUA, Reino Unido, qualquer país começando com ‘U’.

No SQL, no entanto, por definição, alterar a ordem dos operandos deve produzir o mesmo resultado. Desta forma:

Select * from Customers where 'U' = Country

também funcionaria da mesma maneira (observe a diferença dos comandos não SQL).

Quando você deseja implicar correspondências exatas, uma opção é ativar o ANSI:

SET ANSI ON
Select * from Customers where Country = 'USA'

retorna todos esses clientes dos EUA. Observe que os espaços à direita no campo de país OR na expressão à direita são ignorados. Não importa quantos atrás de cada lado você tem. Você obtém a comparação como se fosse feita assim: RTRIM(Country) = RTRIM(‘USA’).

Embora não seja mencionado em Operadores no VFP, um operador SQL é LIKE. Quando você usa LIKE, você obtém uma comparação de correspondência exata, independentemente da configuração SET ANSI (usando forças LIKE e caso ANSI ON implícito - afinal, é um operador ANSI). No entanto, cuidado, há uma pequena diferença no comportamento. Ele não ignoraria espaços à direita, a menos que o tamanho total com trailers seja igual ou menor que o tamanho do campo. Por exemplo, se o campo Country for C(10), então Country = ‘USA’ ou Country = ‘USA__’ funcionaria, mas Country = ‘USA___________’ falharia (sublinhados denotam um espaço e o último tem mais de 7 espaços à direita ).

Finalmente chegamos ao último operador, ==. Isso significa exatamente igual e serve para usar com strings. Uma vantagem é que, usando == você sempre quer dizer que deseja uma correspondência exata independente de configurações SET EXACT ou SET ANSI. No entanto, tome cuidado novamente, seu comportamento é diferente quando é um comando SQL ou um comando regular não SQL.

Com SQL:

Select * from Customers where Country == 'USA'

quaisquer que sejam as configurações ANSI e EXACT, queremos todos os clientes apenas dos EUA. Espaços à direita em ambos os lados são ignorados.

Com não SQL:

? m.lcString1 == m.lcString2

seria verdadeiro somente se eles fossem exatamente os mesmos, em relação ao seu corpo e comprimento (espaços à direita NÃO são ignorados). Não é afetado pelas configurações SET ANSI, EXACT ou COLLATE.
