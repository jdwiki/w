Classes e objetos são usados ​​para tornar seu código mais eficiente e menos repetitivo agrupando tarefas semelhantes.

Uma classe é usada para definir as ações e a estrutura de dados usada para construir objetos. Os objetos são então construídos usando essa estrutura predefinida.

Sintaxe #

  • class <ClassName> [ estende <ParentClassName> ] [ implementa <Interface1> [, <Interface2>, ... ] { } // Declaração de classe
  • interface <InterfaceName> [ estende <ParentInterface1> [, <ParentInterface2>, ...] ] { } // Declaração de interface
  • use <Trait1> [, <Trait2>, ...]; // Usa traços
  • [ público | protegido | private ] [ static ] $<varName>; // Declaração de atributo
  • const <CONST_NAME>; // Declaração constante
  • [ público | protegido | private ] [ static ] function <methodName>([args...]) { } // Declaração do método

Classes e componentes de interface #

As classes podem ter propriedades, constantes e métodos.

  • Propriedades mantêm variáveis ​​no escopo do objeto. Eles podem ser inicializados na declaração, mas somente se contiverem um valor primitivo.

  • Constantes devem ser inicializadas na declaração e podem conter apenas um valor primitivo. Os valores constantes são fixos em tempo de compilação e não podem ser atribuídos em tempo de execução.

  • Métodos devem ter um corpo, mesmo vazio, a menos que o método seja declarado abstrato.

    class Foo { private $foo = ‘foo’; // OK private $baz = array(); // OK private $bar = new Bar(); // Error! }

Interfaces não podem ter propriedades, mas podem ter constantes e métodos.

  • A interface constantes deve ser inicializada na declaração e pode conter apenas um valor primitivo. Os valores constantes são fixos em tempo de compilação e não podem ser atribuídos em tempo de execução.

  • Interface métodos não tem corpo.

    interface FooBar { const FOO_VALUE = ‘bla’; public function doAnything(); }

Constantes de classe #

As constantes de classe fornecem um mecanismo para manter valores fixos em um programa. Ou seja, eles fornecem uma maneira de dar um nome (e verificação de tempo de compilação associada) a um valor como 3.14 ou "Apple". Constantes de classe só podem ser definidas com a palavra-chave const - a função define não pode ser usada neste contexto.

Como exemplo, pode ser conveniente ter uma representação abreviada para o valor de π ao longo de um programa. Uma classe com valores const fornece uma maneira simples de armazenar esses valores.

class MathValues {
    const PI = M_PI;
    const PHI = 1.61803;
}

$area = MathValues::PI * $radius * $radius;

As constantes de classe podem ser acessadas usando o operador de dois pontos (chamado operador de resolução de escopo) em uma classe, assim como as variáveis ​​estáticas. Ao contrário das variáveis ​​estáticas, no entanto, as constantes de classe têm seus valores fixados em tempo de compilação e não podem ser reatribuídos (por exemplo, MathValues::PI = 7 produziria um erro fatal).

As constantes de classe também são úteis para definir coisas internas a uma classe que podem precisar ser alteradas posteriormente (mas não mudam com frequência suficiente para garantir o armazenamento, digamos, em um banco de dados). Podemos fazer referência a isso internamente usando o resolutor de escopo self (que funciona em implementações instanciadas e estáticas)

class Labor {
    /** How long, in hours, does it take to build the item? */
    const LABOR_UNITS = 0.26;
    /** How much are we paying employees per hour? */
    const LABOR_COST = 12.75;

    public function getLaborCost($number_units) {
         return (self::LABOR_UNITS * self::LABOR_COST) * $number_units;
    }
}

Constantes de classe só podem conter valores escalares em versões < 5.6

A partir do PHP 5.6 podemos usar expressões com constantes, significando que declarações matemáticas e strings com concatenação são constantes aceitáveis

class Labor {
    /** How much are we paying employees per hour? Hourly wages * hours taken to make */
    const LABOR_COSTS = 12.75 * 0.26;

    public function getLaborCost($number_units) {
         return self::LABOR_COSTS * $number_units;
    }
}

A partir do PHP 7.0, constantes declaradas com define agora podem conter arrays.

define("BAZ", array('baz'));

As constantes de classe são úteis para mais do que apenas armazenar conceitos matemáticos. Por exemplo, ao preparar uma torta, pode ser conveniente ter uma única classe Pie capaz de levar diferentes tipos de frutas.

class Pie {
    protected $fruit;

    public function __construct($fruit) {
        $this->fruit = $fruit;
    }
}

Podemos então usar a classe Pie assim

$pie = new Pie("strawberry");

O problema que surge aqui é que, ao instanciar a classe Pie, nenhuma orientação é fornecida quanto aos valores aceitáveis. Por exemplo, ao fazer uma torta “boysenberry”, pode ser escrito “boisenberry” incorretamente. Além disso, podemos não apoiar uma torta de ameixa. Em vez disso, seria útil ter uma lista de tipos de frutas aceitáveis ​​já definidos em algum lugar em que faria sentido procurá-los. Diga uma classe chamada Fruit:

class Fruit {
    const APPLE = "apple";
    const STRAWBERRY = "strawberry";
    const BOYSENBERRY = "boysenberry";
}

$pie = new Pie(Fruit::STRAWBERRY);

Listar os valores aceitáveis ​​como constantes de classe fornece uma dica valiosa sobre os valores aceitáveis ​​que um método aceita. Ele também garante que erros ortográficos não possam passar pelo compilador. Enquanto new Pie('aple') e new Pie('apple') são aceitáveis ​​para o compilador, new Pie(Fruit::APLE) produzirá um erro do compilador.

Finalmente, usar constantes de classe significa que o valor real da constante pode ser modificado em um único lugar, e qualquer código que use a constante automaticamente tem os efeitos da modificação.

Embora o método mais comum para acessar uma constante de classe seja MyClass::CONSTANT_NAME, ela também pode ser acessada por:

echo MyClass::CONSTANT;

$classname = "MyClass";
echo $classname::CONSTANT; // As of PHP 5.3.0

As constantes de classe em PHP são convencionalmente nomeadas todas em maiúsculas com sublinhados como separadores de palavras, embora qualquer nome de rótulo válido possa ser usado como um nome de constante de classe.

A partir do PHP 7.1, as constantes de classe agora podem ser definidas com diferentes visibilidades do escopo público padrão. Isso significa que as constantes protegidas e privadas agora podem ser definidas para evitar que constantes de classe vazem desnecessariamente no escopo público (consulte https://www.wikiod.com/pt/php/classes-e-objetos#Método e Visibilidade da Propriedade -propriedade-visibilidade). Por exemplo:

class Something {
    const PUBLIC_CONST_A = 1;
    public const PUBLIC_CONST_B = 2;
    protected const PROTECTED_CONST = 3;
    private const PRIVATE_CONST = 4;
}

define vs constantes de classe #

Embora esta seja uma construção válida:

function bar() { return 2; };

define('BAR', bar());

Se você tentar fazer o mesmo com constantes de classe, receberá um erro:

function bar() { return 2; };

class Foo {
    const BAR = bar(); // Error: Constant expression contains invalid operations
}

Mas você pode fazer:

function bar() { return 2; };

define('BAR', bar());

class Foo {
    const BAR = BAR; // OK
}

Para obter mais informações, consulte constantes no manual.

Usando ::class para recuperar o nome da classe #

O PHP 5.5 introduziu a sintaxe ::class para recuperar o nome completo da classe, levando em consideração o escopo do namespace e as instruções use.

namespace foo;
use bar\Bar;
echo json_encode(Bar::class); // "bar\\Bar"
echo json_encode(Foo::class); // "foo\\Foo"
echo json_encode(\Foo::class); // "Foo"

O acima funciona mesmo que as classes não estejam definidas (ou seja, este trecho de código funciona sozinho).

Essa sintaxe é útil para funções que exigem um nome de classe. Por exemplo, pode ser usado com class_exists para verificar a existência de uma classe. Nenhum erro será gerado independentemente do valor de retorno neste snippet:

class_exists(ThisClass\Will\NeverBe\Loaded::class, false);

Aulas de resumo #

Uma classe abstrata é uma classe que não pode ser instanciada. Classes abstratas podem definir métodos abstratos, que são métodos sem nenhum corpo, apenas uma definição:

abstract class MyAbstractClass {
    abstract public function doSomething($a, $b);
}

As classes abstratas devem ser estendidas por uma classe filha que pode fornecer a implementação desses métodos abstratos.

O principal objetivo de uma classe como essa é fornecer um tipo de modelo que permita que classes filhas herdem, “forçando” uma estrutura a aderir. Vamos detalhar isso com um exemplo:

Neste exemplo estaremos implementando uma interface Worker. Primeiro definimos a interface:

interface Worker {
    public function run();
}

Para facilitar o desenvolvimento de outras implementações de Worker, criaremos uma classe de trabalhador abstrata que já fornece o método run() da interface, mas especifica alguns métodos abstratos que precisam ser preenchidos por qualquer classe filha:

abstract class AbstractWorker implements Worker {
    protected $pdo;
    protected $logger;

    public function __construct(PDO $pdo, Logger $logger) {
        $this->pdo = $pdo;
        $this->logger = $logger;
    }

    public function run() {
        try {
            $this->setMemoryLimit($this->getMemoryLimit());
            $this->logger->log("Preparing main");
            $this->prepareMain();
            $this->logger->log("Executing main");
            $this->main();
        } catch (Throwable $e) {
            // Catch and rethrow all errors so they can be logged by the worker
            $this->logger->log("Worker failed with exception: {$e->getMessage()}");
            throw $e;
        }
    }

    private function setMemoryLimit($memoryLimit) {
        ini_set('memory_limit', $memoryLimit);
        $this->logger->log("Set memory limit to $memoryLimit");
    }

    abstract protected function getMemoryLimit();

    abstract protected function prepareMain();

    abstract protected function main();
}

Em primeiro lugar, fornecemos um método abstrato getMemoryLimit(). Qualquer classe que se estende de AbstractWorker precisa fornecer este método e retornar seu limite de memória. O AbstractWorker então define o limite de memória e o registra.

Em segundo lugar, o AbstractWorker chama os métodos prepareMain() e main(), após registrar que eles foram chamados.

Finalmente, todas essas chamadas de métodos foram agrupadas em um bloco try-catch. Portanto, se qualquer um dos métodos abstratos definidos pela classe filha lançar uma exceção, nós a pegaremos, a registraremos e a lançaremos novamente. Isso evita que todas as classes filhas tenham que implementar isso sozinhas.

Agora vamos definir uma classe filha que se estende do AbstractWorker:

class TranscactionProcessorWorker extends AbstractWorker {
    private $transactions;

    protected function getMemoryLimit() {
        return "512M";
    }

    protected function prepareMain() {
        $stmt = $this->pdo->query("SELECT * FROM transactions WHERE processed = 0 LIMIT 500");
        $stmt->execute();
        $this->transactions = $stmt->fetchAll();
    }

    protected function main() {
        foreach ($this->transactions as $transaction) {
            // Could throw some PDO or MYSQL exception, but that is handled by the AbstractWorker
            $stmt = $this->pdo->query("UPDATE transactions SET processed = 1 WHERE id = {$transaction['id']} LIMIT 1");
            $stmt->execute();
        }
    }
}

Como você pode ver, o TransactionProcessorWorker foi bastante fácil de implementar, pois só tivemos que especificar o limite de memória e nos preocupar com as ações reais que ele precisava executar. Nenhum tratamento de erro é necessário no TransactionProcessorWorker porque isso é tratado no AbsractWorker.

Nota importante #

Ao herdar de uma classe abstrata, todos os métodos marcados como abstratos em a declaração de classe do pai deve ser definida pelo filho (ou pelo próprio filho também deve ser marcado resumo); além disso, esses métodos devem ser definidos com o mesmo (ou menos restrita) visibilidade. Por exemplo, se o método abstrato for definido como protegido, a implementação da função deve ser definida como protegido ou público, mas não privado.

Extraído da Documentação do PHP para abstração de classe.

Se você não definir os métodos das classes abstratas pai dentro da classe filha, você receberá um Fatal PHP Error como o seguinte.

Erro fatal: A classe X contém 1 método abstrato e, portanto, deve ser declarado abstrato ou implementar os métodos restantes (X::x) em

Ligação estática tardia #

No PHP 5.3+ e acima você pode utilizar late static binding para controlar de qual classe uma propriedade ou método estático é chamado. Ele foi adicionado para superar o problema inerente ao resolutor de escopo self::. Pegue o seguinte código

class Horse {
    public static function whatToSay() {
         echo 'Neigh!';
    }

    public static function speak() {
         self::whatToSay();
    }
}

class MrEd extends Horse {
    public static function whatToSay() {
         echo 'Hello Wilbur!';
    }
}

Você esperaria que a classe MrEd substituiria a função pai whatToSay(). Mas quando executamos isso, obtemos algo inesperado

Horse::speak(); // Neigh!
MrEd::speak(); // Neigh!

O problema é que self::whatToSay(); só pode se referir à classe Horse, significando que não obedece MrEd. Se mudarmos para o resolutor de escopo static::, não teremos esse problema. Este método mais novo diz à classe para obedecer à instância que o chama. Assim, obtemos a herança que esperamos

class Horse {
    public static function whatToSay() {
         echo 'Neigh!';
    }

    public static function speak() {
         static::whatToSay(); // Late Static Binding
    }
}

Horse::speak(); // Neigh!
MrEd::speak(); // Hello Wilbur!

Espaçamento de nomes e carregamento automático #

Tecnicamente, o carregamento automático funciona executando um retorno de chamada quando uma classe PHP é necessária, mas não encontrada. Esses retornos de chamada geralmente tentam carregar essas classes.

Geralmente, o carregamento automático pode ser entendido como a tentativa de carregar arquivos PHP (especialmente arquivos de classe PHP, onde um arquivo de origem PHP é dedicado a uma classe específica) de caminhos apropriados de acordo com o nome totalmente qualificado da classe (FQN) quando uma classe é necessária .

Suponha que temos estas classes:

Arquivo de classe para application\controllers\Base:

<?php
namespace application\controllers { class Base {...} }

Arquivo de classe para application\controllers\Control:

<?php
namespace application\controllers { class Control {...} }

Arquivo de classe para application\models\Page:

<?php
namespace application\models { class Page {...} }

Na pasta de origem, essas classes devem ser colocadas nos caminhos como seus FQNs respectivamente:

  • Pasta de origem
  • aplicativos
    • controllers
      • Base.php
      • Control.php
    • models
      • Page.php

Essa abordagem possibilita resolver programaticamente o caminho do arquivo de classe de acordo com o FQN, usando esta função:

function getClassPath(string $sourceFolder, string $className, string $extension = ".php") {
    return $sourceFolder . "/" . str_replace("\\", "/", $className) . $extension; // note that "/" works as a directory separator even on Windows
}

A função spl_autoload_register nos permite carregar uma classe quando necessário usando uma função definida pelo usuário:

const SOURCE_FOLDER = __DIR__ . "/src";
spl_autoload_register(function (string $className) {
    $file = getClassPath(SOURCE_FOLDER, $className);
    if (is_readable($file)) require_once $file;
});

Esta função pode ser estendida para usar métodos de fallback de carregamento:

const SOURCE_FOLDERS = [__DIR__ . "/src", "/root/src"]);
spl_autoload_register(function (string $className) {
    foreach(SOURCE_FOLDERS as $folder) {
        $extensions = [
            // do we have src/Foo/Bar.php5_int64?
            ".php" . PHP_MAJOR_VERSION . "_int" . (PHP_INT_SIZE * 8),
            // do we have src/Foo/Bar.php7?
            ".php" . PHP_MAJOR_VERSION,
            // do we have src/Foo/Bar.php_int64?
            ".php" . "_int" . (PHP_INT_SIZE * 8),
            // do we have src/Foo/Bar.phps?
            ".phps"
            // do we have src/Foo/Bar.php?
            ".php"
        ];
        foreach($extensions as $ext) {
            $path = getClassPath($folder, $className, $extension);
            if(is_readable($path)) return $path;
        }
    }
});

Observe que o PHP não tenta carregar as classes sempre que um arquivo que usa essa classe é carregado. Pode ser carregado no meio de um script, ou até mesmo em funções de desligamento . Esta é uma das razões pelas quais os desenvolvedores, especialmente aqueles que usam autoloading, devem evitar a substituição de arquivos de origem em execução no tempo de execução, especialmente em arquivos phar.

Interfaces #

Introdução #

Interfaces são definições das classes de APIs públicas que devem ser implementadas para satisfazer a interface. Eles funcionam como “contratos”, especificando o que um conjunto de subclasses faz, mas não como o fazem.

A definição de interface é muito parecida com a definição de classe, alterando a palavra-chave class para interface:

interface Foo {

}

As interfaces podem conter métodos e/ou constantes, mas nenhum atributo. As constantes de interface têm as mesmas restrições que as constantes de classe. Os métodos de interface são implicitamente abstratos:

interface Foo {
    const BAR = 'BAR';

    public function doSomething($param1, $param2);
}

Observação: interfaces não devem declarar construtores ou destruidores, pois são detalhes de implementação no nível de classe.

Realização #

Qualquer classe que precise implementar uma interface deve fazê-lo usando a palavra-chave implements. Para isso, a classe precisa fornecer uma implementação para cada método declarado na interface, respeitando a mesma assinatura.

Uma única classe pode implementar mais de uma interface por vez.

interface Foo {
    public function doSomething($param1, $param2);
}

interface Bar {
    public function doAnotherThing($param1);
}


class Baz implements Foo, Bar {
    public function doSomething($param1, $param2) {
        // ...
    }

    public function doAnotherThing($param1) {
        // ...
    }
}

Quando classes abstratas implementam interfaces, elas não precisam implementar todos os métodos. Qualquer método não implementado na classe base deve então ser implementado pela classe concreta que o estende:

abstract class AbstractBaz implements Foo, Bar {
    // Partial implementation of the required interface...
    public function doSomething($param1, $param2) {
        // ...
    }
}

class Baz extends AbstractBaz {
    public function doAnotherThing($param1) {
        // ...
    }
}

Observe que a realização da interface é uma característica herdada. Ao estender uma classe que implementa uma interface, você não precisa redeclará-la na classe concreta, pois ela é implícita.

Observação: Antes do PHP 5.3.9, uma classe não podia implementar duas interfaces que especificavam um método com o mesmo nome, pois causaria ambiguidade. Versões mais recentes do PHP permitem isso desde que os métodos duplicados tenham a mesma assinatura[1].

Herança #

Assim como as classes, é possível estabelecer um relacionamento de herança entre interfaces, usando a mesma palavra-chave extends. A principal diferença é que a herança múltipla é permitida para interfaces:

interface Foo {

}

interface Bar {

}

interface Baz extends Foo, Bar {

}

Exemplos #

No exemplo abaixo temos uma interface de exemplo simples para um veículo. Os veículos podem ir para frente e para trás.

interface VehicleInterface {
    public function forward();

    public function reverse();

    ...
}

class Bike implements VehicleInterface {
    public function forward() {
        $this->pedal();
    }

    public function reverse() {
        $this->backwardSteps();
    }

    protected function pedal() {
        ...
    }

    protected function backwardSteps() {
        ...
    }

    ...
}

class Car implements VehicleInterface {
    protected $gear = 'N';

    public function forward() {
        $this->setGear(1);
        $this->pushPedal();
    }

    public function reverse() {
        $this->setGear('R');
        $this->pushPedal();
    }

    protected function setGear($gear) {
        $this->gear = $gear;
    }

    protected function pushPedal() {
        ...
    }

    ...
}

Em seguida, criamos duas classes que implementam a interface: Bike e Car. Bike e Car internamente são muito diferentes, mas ambos são veículos e devem implementar os mesmos métodos públicos que o VehicleInterface fornece.

Typehinting permite que métodos e funções solicitem interfaces. Vamos supor que temos uma classe de garagem, que contém veículos de todos os tipos.

class ParkingGarage {
    protected $vehicles = [];

    public function addVehicle(VehicleInterface $vehicle) {
        $this->vehicles[] = $vehicle;
    }
}

Como addVehicle requer um $vehicle do tipo VehicleInterface—não uma implementação concreta—podemos inserir bicicletas e carros, que o ParkingGarage pode manipular e usar.

Método e Visibilidade da Propriedade #

Existem três tipos de visibilidade que você pode aplicar a métodos (funções de classe/objeto) e propriedades (variáveis ​​de classe/objeto) dentro de uma classe, que fornecem controle de acesso para o método ou propriedade à qual são aplicados.

Você pode ler extensivamente sobre isso na Documentação PHP para Visibilidade OOP.

Público #

Declarar um método ou propriedade como público permite que o método ou propriedade seja acessado por:

  • A classe que o declarou.
  • As classes que estendem a classe declarada.
  • Quaisquer objetos, classes ou códigos externos fora da hierarquia de classes.

Um exemplo desse acesso público seria:

class MyClass {
    // Property
    public $myProperty = 'test';

    // Method
    public function myMethod() {
        return $this->myProperty;
    }
}

$obj = new MyClass();
echo $obj->myMethod();
// Out: test

echo $obj->myProperty;
// Out: test

Protegido #

Declarar um método ou propriedade como “protegido” permite que o método ou propriedade seja acessado por:

  • A classe que o declarou.
  • As classes que estendem a classe declarada.

Isso não permite que objetos externos, classes ou códigos fora da hierarquia de classes acessem esses métodos ou propriedades. Se algo usando este método/propriedade não tiver acesso a ele, ele não estará disponível e um erro será lançado. Somente instâncias do self declarado (ou subclasses dele) têm acesso a ele.

Um exemplo desse acesso protegido seria:

class MyClass {
    protected $myProperty = 'test';

    protected function myMethod() {
        return $this->myProperty;
    }
}

class MySubClass extends MyClass {
    public function run() {
        echo $this->myMethod();
    }
}

$obj = new MySubClass();
$obj->run(); // This will call MyClass::myMethod();
// Out: test

$obj->myMethod(); // This will fail.
// Out: Fatal error: Call to protected method MyClass::myMethod() from context ''

O exemplo acima observa que você só pode acessar os elementos protected dentro de seu próprio escopo. Essencialmente: “O que está na casa só pode ser acessado de dentro da casa.”

Privado #

Declarar um método ou propriedade como privado permite que o método ou propriedade seja acessado por:

  • A classe que o declarou Somente (não subclasses).

Um método ou propriedade private só é visível e acessível dentro da classe que o criou.

Observe que objetos do mesmo tipo terão acesso uns aos outros membros privados e protegidos, mesmo que não sejam as mesmas instâncias.

class MyClass {
    private $myProperty = 'test';

    private function myPrivateMethod() {
        return $this->myProperty;
    }

    public function myPublicMethod() {
        return $this->myPrivateMethod();
    }

    public function modifyPrivatePropertyOf(MyClass $anotherInstance) {
        $anotherInstance->myProperty = "new value";
    }
}

class MySubClass extends MyClass {
    public function run() {
        echo $this->myPublicMethod();
    }

    public function runWithPrivate() {
        echo $this->myPrivateMethod();
    }
}

$obj = new MySubClass();
$newObj = new MySubClass();

// This will call MyClass::myPublicMethod(), which will then call
// MyClass::myPrivateMethod();
$obj->run(); 
// Out: test

    
$obj->modifyPrivatePropertyOf($newObj);

$newObj->run();
// Out: new value

echo $obj->myPrivateMethod(); // This will fail.
// Out: Fatal error: Call to private method MyClass::myPrivateMethod() from context ''

echo $obj->runWithPrivate(); // This will also fail.
// Out: Fatal error: Call to private method MyClass::myPrivateMethod() from context 'MySubClass'

Como observado, você só pode acessar o método/propriedade private de dentro de sua classe definida.

Palavra-chave final #

Def: Final A palavra-chave impede que classes filhas substituam um método prefixando a definição com final. Se a própria classe estiver sendo definida como final, ela não poderá ser estendida

Método final

class BaseClass {
   public function test() {
       echo "BaseClass::test() called\n";
   }
   
   final public function moreTesting() {
       echo "BaseClass::moreTesting() called\n";
   }
}

class ChildClass extends BaseClass {
   public function moreTesting() {
       echo "ChildClass::moreTesting() called\n";
   }
}
// Results in Fatal error: Cannot override final method BaseClass::moreTesting()

Aula Final:

final class BaseClass {
   public function test() {
       echo "BaseClass::test() called\n";
   }

   // Here it doesn't matter if you specify the function as final or not
   final public function moreTesting() {
       echo "BaseClass::moreTesting() called\n";
   }
}

class ChildClass extends BaseClass {
}
// Results in Fatal error: Class ChildClass may not inherit from final class (BaseClass)

Constantes finais: Ao contrário de Java, a palavra-chave final não é usada para constantes de classe em PHP. Use a palavra-chave const.

Por que tenho que usar final?

  1. Evitando a cadeia de herança maciça da desgraça
  2. Composição encorajadora
  3. Forçar o desenvolvedor a pensar na API pública do usuário
  4. Forçar o desenvolvedor a reduzir a API pública de um objeto
  5. Uma classe ‘final’ sempre pode ser extensível
  6. estende quebra o encapsulamento
  7. Você não precisa dessa flexibilidade
  8. Você é livre para alterar o código

Quando evitar final: As aulas finais só funcionam efetivamente sob os seguintes pressupostos:

  1. Existe uma abstração (interface) que a classe final implementa
  2. Toda a API pública da classe final faz parte dessa interface

Carregamento automático #

Ninguém quer requerer ou incluir toda vez que uma classe ou herança é usada. Porque pode ser doloroso e fácil de esquecer, o PHP está oferecendo o chamado carregamento automático. Se você já estiver usando o Composer, leia sobre autoloading using Composer.

O que exatamente é o carregamento automático?

O nome basicamente diz tudo. Você não precisa obter o arquivo onde a classe solicitada está armazenada, mas o PHP automaticamente carregas ele.

Como posso fazer isso em PHP básico sem código de terceiros?

Existe a função [__autoload][2], mas considera-se melhor prática usar [spl_autoload_register][3]. Essas funções serão consideradas pelo PHP toda vez que uma classe não for definida dentro do espaço fornecido. Portanto, adicionar autoload a um projeto existente não é problema, pois as classes definidas (via require, ou seja, funcionarão como antes. Por uma questão de precisão, os exemplos a seguir usarão funções anônimas, se você usar PHP < 5.3, você pode definir a função e passar seu nome como argumento para spl_autoload_register.

Exemplos

spl_autoload_register(function ($className) {
    $path = sprintf('%s.php', $className);
    if (file_exists($path)) {
        include $path;
    } else {
        // file not found
    }
});

O código acima simplesmente tenta incluir um nome de arquivo com o nome da classe e a extensão anexada “.php” usando [sprintf][4]. Se o FooBar precisa ser carregado, ele verifica se o FooBar.php existe e, em caso afirmativo, o inclui.

É claro que isso pode ser estendido para atender às necessidades individuais do projeto. Se _ dentro de um nome de classe for usado para agrupar, por exemplo User_Post e User_Image se referem a User, ambas as classes podem ser mantidas em uma pasta chamada “User” assim:

spl_autoload_register(function ($className) {
    //                        replace _ by / or \ (depending on OS)
    $path = sprintf('%s.php', str_replace('_', DIRECTORY_SEPARATOR, $className) );
    if (file_exists($path)) {
        include $path;
    } else {
        // file not found
    }
});

A classe User_Post agora será carregada de “User/Post.php”, etc.

spl_autoload_register pode ser adaptado para várias necessidades. Todos os seus arquivos com classes são chamados de “class.CLASSNAME.php”? Sem problemas. Vários aninhamentos (User_Post_Content => “User/Post/Content.php”)? Nenhum problema também.

Se você deseja um mecanismo de carregamento automático mais elaborado - e ainda não deseja incluir o Composer - você pode trabalhar sem adicionar bibliotecas de terceiros.

spl_autoload_register(function ($className) {
    $path = sprintf('%1$s%2$s%3$s.php',
        // %1$s: get absolute path
        realpath(dirname(__FILE__)),
        // %2$s: / or \ (depending on OS)
        DIRECTORY_SEPARATOR,
        // %3$s: don't wory about caps or not when creating the files
        strtolower(
            // replace _ by / or \ (depending on OS)
            str_replace('_', DIRECTORY_SEPARATOR, $className)
        )
    );

    if (file_exists($path)) {
        include $path;
    } else {
        throw new Exception(
            sprintf('Class with name %1$s not found. Looked in %2$s.',
                $className,
                $path
            )
        );
    }
});

Usando autoloaders como este, você pode escrever códigos como este:

require_once './autoload.php'; // where spl_autoload_register is defined

$foo = new Foo_Bar(new Hello_World());

Usando classes:

class Foo_Bar extends Foo {}

class Hello_World implements Demo_Classes {}

Esses exemplos incluirão classes de foo/bar.php, foo.php, hello/world.php e demo/classes.php.

1: https://www.wikiod.com/pt/php/gerenciador-de-dependencias-do-compositor#Carregamento automático com o Composer [2]: https://secure.php.net/manual/function.autoload.php [3]: https://secure.php.net/manual/function.spl-autoload-register.php [4]: https://secure.php.net/sprintf

Vinculação dinâmica #

A vinculação dinâmica, também chamada de substituição de método, é um exemplo de polimorfismo em tempo de execução que ocorre quando várias classes contêm diferentes implementações do mesmo método, mas o objeto no qual o método será chamado é desconhecido até tempo de execução.

Isso é útil se uma determinada condição ditar qual classe será usada para executar uma ação, onde a ação tem o mesmo nome em ambas as classes.

interface Animal {
    public function makeNoise();
}

class Cat implements Animal {
    public function makeNoise
    {
        $this->meow();
    }
    ...
}

class Dog implements Animal {
    public function makeNoise {
        $this->bark();
    }
    ...
}

class Person {
    const CAT = 'cat';
    const DOG = 'dog';

    private $petPreference;
    private $pet;

    public function isCatLover(): bool {
        return $this->petPreference == self::CAT;
    }

    public function isDogLover(): bool {
        return $this->petPreference == self::DOG;
    }

    public function setPet(Animal $pet) {
        $this->pet = $pet;
    }

    public function getPet(): Animal {
        return $this->pet;
    }
}

if($person->isCatLover()) {
    $person->setPet(new Cat());
} else if($person->isDogLover()) {
    $person->setPet(new Dog());
}

$person->getPet()->makeNoise();

No exemplo acima, a classe Animal (Dog|Cat) que irá makeNoise é desconhecida até o tempo de execução, dependendo da propriedade dentro da classe User.

Chamando um construtor pai ao instanciar um filho #

Uma armadilha comum das classes filhas é que, se seu pai e filho contiverem um método constructor(__construct()), somente o construtor da classe filha será executado. Pode haver ocasiões em que você precise executar o método pai __construct() a partir de seu filho. Se você precisar fazer isso, precisará usar o resolutor de escopo parent:::

parent::__construct();

Agora, aproveitar isso em uma situação do mundo real seria algo como:

class Foo {

    function __construct($args) { 
        echo 'parent'; 
    }

}

class Bar extends Foo {

    function __construct($args) {
        parent::__construct($args);
    }
}

O acima irá executar o pai __construct() resultando na execução do echo.

$this, self e static mais o singleton #

Use $this para se referir ao objeto atual. Use self para se referir ao classe atual. Em outras palavras, use $this->member para não estático membros, use self::$member para membros estáticos.

No exemplo abaixo, sayHello() e sayGoodbye() estão usando self e a diferença $this pode ser observada aqui.

class Person {
    private $name;

    public function __construct($name) {
        $this->name = $name;
    }

    public function getName() {
        return $this->name;
    }

    public function getTitle() {
        return $this->getName()." the person";
    }

    public function sayHello() {
        echo "Hello, I'm ".$this->getTitle()."<br/>";
    }

    public function sayGoodbye() {
        echo "Goodbye from ".self::getTitle()."<br/>";
    }
}

class Geek extends Person {
    public function __construct($name) {
        parent::__construct($name);
    }

    public function getTitle() {
        return $this->getName()." the geek";
    }
}

$geekObj = new Geek("Ludwig");
$geekObj->sayHello();
$geekObj->sayGoodbye();

static refere-se a qualquer classe na hierarquia em que você chamou o método. Ele permite uma melhor reutilização de propriedades de classes estáticas quando as classes são herdadas.

Considere o seguinte código:

class Car {
    protected static $brand = 'unknown';
    
    public static function brand() {
         return self::$brand."\n";
    }
}

class Mercedes extends Car {
    protected static $brand = 'Mercedes';
}

class BMW extends Car {
    protected static $brand = 'BMW';
}

echo (new Car)->brand();
echo (new BMW)->brand();
echo (new Mercedes)->brand();

Isso não produz o resultado desejado:

desconhecido desconhecido desconhecido

Isso porque self se refere à classe Car sempre que o método brand() é chamado.

Para se referir à classe correta, você precisa usar static:

class Car {
    protected static $brand = 'unknown';
    
    public static function brand() {
         return static::$brand."\n";
    }
}

class Mercedes extends Car {
    protected static $brand = 'Mercedes';
}

class BMW extends Car {
    protected static $brand = 'BMW';
}

echo (new Car)->brand();
echo (new BMW)->brand();
echo (new Mercedes)->brand();

Isso produz a saída desejada:

desconhecido BMW Mercedes

Consulte também https://www.wikiod.com/pt/php/classes-e-objetos#Ligação estática tardia

O singleton #

Se você tem um objeto que é caro para criar ou representa uma conexão com algum recurso externo que você deseja reutilizar, ou seja, uma conexão de banco de dados onde não há pool de conexão ou um soquete para algum outro sistema, você pode usar o static e self palavras-chave em uma classe para torná-la um singleton. Existem opiniões fortes sobre se o padrão singleton deve ou não ser usado, mas ele tem seus usos.

class Singleton {
    private static $instance = null;

    public static function getInstance(){
        if(!isset(self::$instance)){
            self::$instance = new self();
        }
        
        return self::$instance;
    }
    
    private function __construct() {
        // Do constructor stuff
    }
}

Como você pode ver no código de exemplo, estamos definindo uma propriedade estática privada $instance para manter a referência do objeto. Como isso é estático, essa referência é compartilhada em TODOS os objetos desse tipo.

O método getInstance() usa um método conhecido como instanciação lenta para atrasar a criação do objeto até o último momento possível, pois você não quer ter objetos não utilizados na memória que nunca foram usados. Também economiza tempo e CPU no carregamento da página não tendo que carregar mais objetos do que o necessário. O método está verificando se o objeto está definido, criando-o se não estiver e retornando-o. Isso garante que apenas um objeto desse tipo seja criado.

Também estamos configurando o construtor como privado para garantir que ninguém o crie com a palavra-chave new de fora. Se você precisar herdar desta classe, basta alterar as palavras-chave private para protected.

Para usar este objeto, basta escrever o seguinte:

$singleton = Singleton::getInstance();

Agora eu imploro que você use injeção de dependência onde puder e mire em objetos fracamente acoplados, mas às vezes isso não é razoável e o padrão singleton pode ser útil.

Definindo uma classe básica #

Um objeto em PHP contém variáveis ​​e funções. Objetos normalmente pertencem a uma classe, que define as variáveis ​​e funções que todos os objetos desta classe irão conter.

A sintaxe para definir uma classe é:

class Shape {
    public $sides = 0;
    
    public function description() {
        return "A shape with $this->sides sides.";
    }
}

Depois que uma classe é definida, você pode criar uma instância usando:

$myShape = new Shape();

Variáveis ​​e funções no objeto são acessadas assim:

$myShape = new Shape();
$myShape->sides = 6;

print $myShape->description(); // "A shape with 6 sides"

Construtor #

As classes podem definir um método especial __construct(), que é executado como parte da criação do objeto. Isso é frequentemente usado para especifique os valores iniciais de um objeto:

class Shape {
    public $sides = 0;
    
    public function __construct($sides) {
        $this->sides = $sides;
    }
    
    public function description() {
        return "A shape with $this->sides sides.";
    }
}

$myShape = new Shape(6);

print $myShape->description(); // A shape with 6 sides

Estendendo outra classe #

As definições de classe podem estender as definições de classe existentes, adicionando novas variáveis ​​e funções, bem como modificando aquelas definidas na classe pai.

Aqui está uma classe que estende o exemplo anterior:

class Square extends Shape {
    public $sideLength = 0;
    
    public function __construct($sideLength) {
       parent::__construct(4);
       
       $this->sideLength = $sideLength;
    }
    
    public function perimeter() {
        return $this->sides * $this->sideLength;
    }

    public function area() {
        return $this->sideLength * $this->sideLength;
    }
}

A classe Square contém variáveis ​​e comportamento tanto para a classe Shape quanto para a classe Square:

$mySquare = new Square(10);

print $mySquare->description()/ // A shape with 4 sides

print $mySquare->perimeter() // 40

print $mySquare->area() // 100

Aulas anônimas #

Classes anônimas foram introduzidas no PHP 7 para permitir a criação rápida de objetos únicos. Eles podem receber argumentos do construtor, estender outras classes, implementar interfaces e usar características da mesma forma que as classes normais.

Em sua forma mais básica, uma classe anônima se parece com o seguinte:

new class("constructor argument") {
    public function __construct($param) {
        var_dump($param);
    }
}; // string(20) "constructor argument"

Aninhar uma classe anônima dentro de outra classe não lhe dá acesso a métodos privados ou protegidos ou propriedades dessa classe externa. O acesso a métodos protegidos e propriedades da classe externa pode ser obtido estendendo a classe externa da classe anônima. O acesso às propriedades privadas da classe externa pode ser obtido passando-as para o construtor da classe anônima.

Por exemplo:

class Outer {
    private $prop = 1;
    protected $prop2 = 2;

    protected function func1() {
        return 3;
    }

    public function func2() {
        // passing through the private $this->prop property
        return new class($this->prop) extends Outer {
            private $prop3;

            public function __construct($prop) {
                $this->prop3 = $prop;
            }

            public function func3() {
                // accessing the protected property Outer::$prop2
                // accessing the protected method Outer::func1()
                // accessing the local property self::$prop3 that was private from Outer::$prop
                return $this->prop2 + $this->func1() + $this->prop3;
            }
        };
    }
}

echo (new Outer)->func2()->func3(); // 6
