Características para facilitar a reutilização de código horizontal #

Digamos que temos uma interface para registro:

interface Logger {
    function log($message);
}

Agora digamos que temos duas implementações concretas da interface Logger: o FileLogger e o ConsoleLogger.

class FileLogger implements Logger {
    public function log($message) {
        // Append log message to some file
    }
}

class ConsoleLogger implements Logger {
    public function log($message) {
        // Log message to the console
    }
}

Agora, se você definir alguma outra classe Foo que você também deseja executar tarefas de log, você pode fazer algo assim:

class Foo implements Logger {
    private $logger;

    public function setLogger(Logger $logger) {
        $this->logger = $logger;
    }

    public function log($message) {
        if ($this->logger) {
            $this->logger->log($message);
        }
    }
}

Foo agora também é um Logger, mas sua funcionalidade depende da implementação do Logger passada para ele via setLogger(). Se agora quisermos que a classe Bar também tenha esse mecanismo de registro, teríamos que duplicar essa parte da lógica na classe Bar.

Em vez de duplicar o código, um traço pode ser definido:

trait LoggableTrait {
    protected $logger;

    public function setLogger(Logger $logger) {
        $this->logger = $logger;
    }

    public function log($message) {
        if ($this->logger) {
            $this->logger->log($message);
        }
    }
}

Agora que definimos a lógica em um trait, podemos usar o trait para adicionar a lógica às classes Foo e Bar:

class Foo {
    use LoggableTrait;
}

class Bar {
    use LoggableTrait;
}

E, por exemplo, podemos usar a classe Foo assim:

$foo = new Foo();
$foo->setLogger( new FileLogger() );

//note how we use the trait as a 'proxy' to call the Logger's log method on the Foo instance
$foo->log('my beautiful message');

O que é um traço? #

PHP só permite herança simples. Em outras palavras, uma classe só pode ’estender’ uma outra classe. Mas e se você precisar incluir algo que não pertence à classe pai? Antes do PHP 5.4 você teria que ser criativo, mas no 5.4 Traits foram introduzidos. As características permitem que você basicamente “copie e cole” uma parte de uma classe em sua classe principal

trait Talk {
    /** @var string */
    public $phrase = 'Well Wilbur...';
    public function speak() {
         echo $this->phrase;
    }
}

class MrEd extends Horse {
    use Talk;
    public function __construct() {
         $this->speak();
    }

    public function setPhrase($phrase) {
         $this->phrase = $phrase;
    }
}

Então aqui temos MrEd, que já está estendendo Horse. Mas nem todos os cavalos ‘Falam’, então temos uma Característica para isso. Vamos observar o que isso está fazendo

Primeiro, definimos nossa Característica. Podemos usá-lo com carregamento automático e Namespaces (consulte também <https://www.wikiod.com/pt/php/namespaces#Fazendo referência a uma classe ou função em um namespace>). Em seguida, incluímos em nossa classe MrEd com a palavra-chave use.

Você notará que MrEd usa as funções e variáveis ​​Talk sem defini-las. Lembra do que dissemos sobre copiar e colar? Essas funções e variáveis ​​estão todas definidas dentro da classe agora, como se essa classe as tivesse definido.

As características estão mais intimamente relacionadas às [classes abstratas](https://www.wikiod.com/pt/php/classes-e-objetos#Aulas abstratas) em que você pode definir variáveis ​​e funções. Você também não pode instanciar um Trait diretamente (ou seja, new Trait()). Os traços não podem forçar uma classe a definir implicitamente uma função como uma classe abstrata ou uma interface pode. Os traços são apenas para definições explícitas (já que você pode implementar quantas Interfaces quiser, veja <https://www.wikiod.com/pt/php/classes-e-objetos#Interfaces interfaces>).

Quando devo usar uma Característica? #

A primeira coisa que você deve fazer, ao considerar uma Característica, é fazer a si mesmo essa importante pergunta

Posso evitar o uso de um Trait reestruturando meu código?

Na maioria das vezes, a resposta será Sim. Traços são casos extremos causados ​​por herança simples. A tentação de abusar ou abusar das Características pode ser alta. Mas considere que um Trait introduz outra fonte para seu código, o que significa que há outra camada de complexidade. No exemplo aqui, estamos lidando apenas com 3 classes. Mas Traços significam que agora você pode estar lidando com muito mais do que isso. Para cada Característica, sua classe se torna muito mais difícil de lidar, pois agora você deve fazer referência a cada Característica para descobrir o que ela define (e potencialmente onde ocorreu uma colisão, consulte <https://www.wikiod.com/pt/docs/php 999/traits/7271/resolução de conflitos>). Idealmente, você deve manter o mínimo possível de Traits em seu código.

Resolução de conflitos #

Tentar usar várias características em uma classe pode resultar em problemas envolvendo métodos conflitantes. Você precisa resolver esses conflitos manualmente.

Por exemplo, vamos criar esta hierarquia:

trait MeowTrait {
    public function say() {
        print "Meow \n";
    }
}

trait WoofTrait {
    public function say() {
        print "Woof \n";
    }
}

abstract class UnMuteAnimals {
    abstract function say();
}

class Dog extends UnMuteAnimals {
    use WoofTrait;
}

class Cat extends UnMuteAnimals {
    use MeowTrait;
}

Agora, vamos tentar criar a seguinte classe:

class TalkingParrot extends UnMuteAnimals {
    use MeowTrait, WoofTrait;
}

O interpretador php retornará um erro fatal:

Erro fatal: O método Trait diz que não foi aplicado, porque existem colisões com outros métodos de trait no TalkingParrot

Para resolver esse conflito, poderíamos fazer isso:

  • use a palavra-chave insteadof para usar o método de uma característica em vez do método de outra característica

  • crie um alias para o método com uma construção como WoofTrait::say as sayAsDog;

    class TalkingParrotV2 extends UnMuteAnimals { use MeowTrait, WoofTrait { MeowTrait::say insteadof WoofTrait; WoofTrait::say as sayAsDog; } }

    $talkingParrot = new TalkingParrotV2(); $talkingParrot->say(); $talkingParrot->sayAsDog();

Este código produzirá a seguinte saída:

Miau Woof

Implementando um Singleton usando Traits #

Isenção de responsabilidade: De forma alguma este exemplo defende o uso de singletons. Singletons devem ser usados ​​com muito cuidado.

Em PHP existe uma maneira bastante padrão de implementar um singleton:

public class Singleton {
    private $instance;

    private function __construct() { };

    public function getInstance() {
        if (!self::$instance) {
            // new self() is 'basically' equivalent to new Singleton()
            self::$instance = new self();
        }

        return self::$instance;
    }

    // Prevent cloning of the instance
    protected function __clone() { }

    // Prevent serialization of the instance
    protected function __sleep() { }

    // Prevent deserialization of the instance
    protected function __wakeup() { }
}

Para evitar a duplicação de código, é uma boa ideia extrair esse comportamento em uma característica.

trait SingletonTrait {
    private $instance;

    protected function __construct() { };

    public function getInstance() {
        if (!self::$instance) {
            // new self() will refer to the class that uses the trait
            self::$instance = new self();
        }

        return self::$instance;
    }

    protected function __clone() { }
    protected function __sleep() { }
    protected function __wakeup() { }
}

Agora, qualquer classe que queira funcionar como singleton pode simplesmente usar o trait:

class MyClass {
    use SingletonTrait;
}

// Error! Constructor is not publicly accessible
$myClass = new MyClass();

$myClass = MyClass::getInstance();

// All calls below will fail due to method visibility
$myClassCopy = clone $myClass; // Error!
$serializedMyClass = serialize($myClass); // Error!
$myClass = deserialize($serializedMyclass); // Error!

Mesmo que agora seja impossível serializar um singleton, ainda é útil não permitir o método desserialize.

Uso de vários traços #

trait Hello {
    public function sayHello() {
        echo 'Hello ';
    }
}

trait World {
    public function sayWorld() {
        echo 'World';
    }
}

class MyHelloWorld {
    use Hello, World;
    public function sayExclamationMark() {
        echo '!';
    }
}

$o = new MyHelloWorld();
$o->sayHello();
$o->sayWorld();
$o->sayExclamationMark();

O exemplo acima produzirá:

Hello World!

Alterando a visibilidade do método #

trait HelloWorld {
    public function sayHello() {
        echo 'Hello World!';
    }
}

// Change visibility of sayHello
class MyClass1 {
    use HelloWorld { sayHello as protected; }
}

// Alias method with changed visibility
// sayHello visibility not changed
class MyClass2 {
    use HelloWorld { sayHello as private myPrivateHello; }
}

Executando este exemplo:

(new MyClass1())->sayHello();
// Fatal error: Uncaught Error: Call to protected method MyClass1::sayHello()

(new MyClass2())->myPrivateHello();
// Fatal error: Uncaught Error: Call to private method MyClass2::myPrivateHello()

(new MyClass2())->sayHello();
// Hello World!

Portanto, esteja ciente de que no último exemplo em MyClass2 o método original sem alias de trait HelloWorld permanece acessível como está.

Traços para manter as aulas limpas #

Com o tempo, nossas classes podem implementar mais e mais interfaces. Quando essas interfaces têm muitos métodos, o número total de métodos em nossa classe se tornará muito grande.

Por exemplo, vamos supor que temos duas interfaces e uma classe implementando-as:

interface Printable {
    public function print();   
    //other interface methods...
}

interface Cacheable {
    //interface methods
}

class Article implements Cachable, Printable {  
    //here we must implement all the interface methods
    public function print(){ {
        /* code to print the article */ 
    }
}

Em vez de implementar todos os métodos de interface dentro da classe Article, poderíamos usar Traits separados para implementar essas interfaces, mantendo a classe menor e separando o código da implementação da interface da classe.

Por exemplo, para implementar a interface Printable, poderíamos criar esta característica:

trait PrintableArticle {
    //implements here the interface methods
    public function print() {
        /* code to print the article */ 
    }
}

e faça a classe usar a característica:

class Article implements Cachable, Printable {
    use PrintableArticle;
    use CacheableArticle; 
}

Os principais benefícios seriam que nossos métodos de implementação de interface serão separados do resto da classe e armazenados em uma característica que tem a responsabilidade exclusiva de implementar a interface para esse tipo específico de objeto.
