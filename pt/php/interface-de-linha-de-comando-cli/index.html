Manipulando Opções do Programa #

As opções do programa podem ser tratadas com a função getopt(). Ele opera com uma sintaxe semelhante ao comando POSIX getopt, com suporte adicional para opções longas no estilo GNU.

#!/usr/bin/php

// a single colon indicates the option takes a value
// a double colon indicates the value may be omitted
$shortopts = "hf✌️:d";
// GNU-style long options are not required
$longopts = ["help", "version"];
$opts = getopt($shortopts, $longopts);

// options without values are assigned a value of boolean false
// you must check their existence, not their truthiness
if (isset($opts["h"]) || isset($opts["help"])) {
    fprintf(STDERR, "Here is some help!\n");
    exit;
}

// long options are called with two hyphens: "--version"
if (isset($opts["version"])) {
    fprintf(STDERR, "%s Version 223.45" . PHP_EOL, $argv[0]);
    exit;
}

// options with values can be called like "-f foo", "-ffoo", or "-f=foo"
$file = "";
if (isset($opts["f"])) {
    $file = $opts["f"];
}
if (empty($file)) {
    fprintf(STDERR, "We wanted a file!" . PHP_EOL);
    exit(1);
}
fprintf(STDOUT, "File is %s" . PHP_EOL, $file);

// options with optional values must be called like "-v5" or "-v=5"
$verbosity = 0;
if (isset($opts["v"])) {
    $verbosity = ($opts["v"] === false) ? 1 : (int)$opts["v"];
}
fprintf(STDOUT, "Verbosity is %d" . PHP_EOL, $verbosity);

// options called multiple times are passed as an array
$debug = 0;
if (isset($opts["d"])) {
    $debug = is_array($opts["d"]) ? count($opts["d"]) : 1;
}
fprintf(STDOUT, "Debug is %d" . PHP_EOL, $debug);

// there is no automated way for getopt to handle unexpected options

Este script pode ser testado assim:

./test.php --help
./test.php --version
./test.php -f foo -ddd
./test.php -v -d -ffoo
./test.php -v5 -f=foo
./test.php -f foo -v 5 -d

Observe que o último método não funcionará porque -v 5 não é válido.

Nota: A partir do PHP 5.3.0, o getopt é independente do SO, funcionando também no Windows.

Manipulação de argumentos #

Os argumentos são passados ​​para o programa de maneira semelhante à maioria das linguagens de estilo C. $argc é um inteiro contendo o número de argumentos incluindo o nome do programa, e $argv é um array contendo argumentos para o programa. O primeiro elemento de $argv é o nome do programa.

#!/usr/bin/php

printf("You called the program %s with %d arguments\n", $argv[0], $argc - 1);
unset($argv[0]);
foreach ($argv as $i => $arg) {
    printf("Argument %d is %s\n", $i, $arg);
}

Chamar o aplicativo acima com php example.php foo bar (onde example.php contém o código acima) resultará na seguinte saída:

Você chamou o programa example.php com 2 argumentos O argumento 1 é foo O argumento 2 é barra

Observe que $argc e $argv são variáveis ​​globais, não variáveis ​​superglobais. Eles devem ser importados para o escopo local usando a palavra-chave global se forem necessários em uma função.

Este exemplo mostra como os argumentos são agrupados quando são usados ​​escapes como "" ou \.

Exemplo de roteiro

var_dump($argc, $argv);

Linha de comando

  string(1) "-"

Manipulação de entrada e saída #

Quando executado a partir da CLI, as constantes STDIN, STDOUT e STDERR são predefinidas. Essas constantes são identificadores de arquivo e podem ser consideradas equivalentes aos resultados da execução dos seguintes comandos:

STDIN = fopen("php://stdin", "r");
STDOUT = fopen("php://stdout", "w");
STDERR = fopen("php://stderr", "w");

As constantes podem ser usadas em qualquer lugar que um identificador de arquivo padrão seria:

#!/usr/bin/php

while ($line = fgets(STDIN)) {
    $line = strtolower(trim($line));
    switch ($line) {
        case "bad":
            fprintf(STDERR, "%s is bad" . PHP_EOL, $line);
            break;
        case "quit":
            exit;
        default:
            fprintf(STDOUT, "%s is good" . PHP_EOL, $line);
            break;
    }
}

Os endereços de stream integrados mencionados anteriormente (php://stdin, php://stdout e php://stderr) podem ser usados ​​no lugar de nomes de arquivos na maioria dos contextos:

file_put_contents('php://stdout', 'This is stdout content');
file_put_contents('php://stderr', 'This is stderr content');

// Open handle and write multiple times.
$stdout = fopen('php://stdout', 'w');

fwrite($stdout, 'Hello world from stdout' . PHP_EOL);
fwrite($stdout, 'Hello again');

fclose($stdout);

Como alternativa, você também pode usar readline() para entrada e também pode usar echo ou *print * ou qualquer outra função de impressão de string para saída.

$name = readline("Please enter your name:");
print "Hello, {$name}.";

Códigos de retorno #

A construção exit pode ser usada para passar um código de retorno para o ambiente de execução.

#!/usr/bin/php

if ($argv[1] === "bad") {
    exit(1);
} else {
    exit(0);
}

Por padrão, um código de saída 0 será retornado se nenhum for fornecido, ou seja, exit é o mesmo que exit(0). Como exit não é uma função, os parênteses não são necessários se nenhum código de retorno estiver sendo passado.

Os códigos de retorno devem estar no intervalo de 0 a 254 (255 é reservado pelo PHP e não deve ser usado). Por convenção, sair com um código de retorno de 0 informa ao programa de chamada que o script PHP foi executado com sucesso. Use um código de retorno diferente de zero para informar ao programa chamador que ocorreu uma condição de erro específica.

Diferenças comportamentais na linha de comando #

Ao executar a partir da CLI, o PHP exibe alguns comportamentos diferentes do que quando executado a partir de um servidor web. Essas diferenças devem ser mantidas em mente, especialmente no caso em que o mesmo script pode ser executado em ambos os ambientes.

  • Sem alteração de diretório Ao executar um script de um servidor web, o diretório de trabalho atual é sempre o do próprio script. O código require("./stuff.inc"); assume que o arquivo está no mesmo diretório que o script. Na linha de comando, o diretório de trabalho atual é o diretório em que você está quando chama o script. Os scripts que serão chamados a partir da linha de comando devem sempre usar caminhos absolutos. (Observe que as constantes mágicas __DIR__ e __FILE__ continuam funcionando conforme o esperado e retornam a localização do script.)
  • Sem buffer de saída As diretivas php.ini output_buffering e implicit_flush padronizam para false e ​​true, respectivamente. O buffer ainda está disponível, mas deve ser habilitado explicitamente, caso contrário, a saída sempre será exibida em tempo real.
  • Sem limite de tempo A diretiva php.ini max_execution_time é definida como zero, então os scripts não expirarão por padrão.
  • Sem erros de HTML Caso você tenha habilitado a diretiva php.ini html_errors, ela será ignorada na linha de comando.
  • Diferentes php.ini podem ser carregados. Quando você está usando php do cli, ele pode usar php.ini diferente do servidor web. Você pode saber qual arquivo está usando executando php --ini.

Executando seu script #

No Linux/UNIX ou no Windows, um script pode ser passado como argumento para o executável do PHP, com as opções e os argumentos desse script a seguir:

php ~/example.php foo bar
c:\php\php.exe c:\example.php foo bar

Isso passa foo e bar como argumentos para example.php.

No Linux/UNIX, o método preferido de execução de scripts é usar um shebang (por exemplo, #!/usr/bin/env php) como a primeira linha de um arquivo e definir o bit executável no Arquivo. Supondo que o script esteja em seu caminho, você pode chamá-lo diretamente:

example.php foo bar

Usar /usr/bin/env php torna o executável do PHP para ser encontrado usando o PATH. Seguindo como o PHP é instalado, ele pode não estar localizado no mesmo lugar (como /usr/bin/php ou /usr/local/bin/php), ao contrário de env que é comumente disponível em / usr/bin/env.

No Windows, você pode ter o mesmo resultado adicionando o diretório do PHP e seu script ao PATH e editando o PATHEXT para permitir que .php seja detectado usando o PATH. Outra possibilidade é adicionar um arquivo chamado example.bat ou example.cmd no mesmo diretório do seu script PHP e escrever esta linha nele:

c:\php\php.exe "%~dp0example.php" %*

Ou, se você adicionou o diretório do PHP no PATH, para uso conveniente:

php "%~dp0example.php" %*

Casos extremos de getopt() #

Este exemplo mostra o comportamento de getopt quando a entrada do usuário é incomum:

getopt.php #

var_dump(
    getopt("ab:c::", ["delta", "epsilon:", "zeta::"])
);

Linha de comando do Shell #

$ php getopt.php -a -a -bbeta -b beta -cgamma --delta --epsilon --zeta --zeta=f  -c gamma
array(6) {
  ["a"]=>
  array(2) {
    [0]=>
    bool(false)
    [1]=>
    bool(false)
  }
  ["b"]=>
  array(2) {
    [0]=>
    string(4) "beta"
    [1]=>
    string(4) "beta"
  }
  ["c"]=>
  array(2) {
    [0]=>
    string(5) "gamma"
    [1]=>
    bool(false)
  }
  ["delta"]=>
  bool(false)
  ["epsilon"]=>
  string(6) "--zeta"
  ["zeta"]=>
  string(1) "f"
}

A partir deste exemplo, pode-se ver que:

  • Opções individuais (sem dois pontos) sempre carregam um valor booleano de false se habilitadas.
  • Se uma opção for repetida, o respectivo valor na saída de getopt se tornará um array.
  • As opções de argumento obrigatórias (um dois-pontos) aceitam um espaço ou nenhum espaço (como opções de argumento opcionais) como separador
  • Após um argumento que não pode ser mapeado em nenhuma opção, as opções por trás também não serão mapeadas.
