Simulação simples #

Introdução #

O Manual do PHPUnit descreve o mocking como tal:

A prática de substituir um objeto por um duplo de teste que verifica expectativas, por exemplo, afirmar que um método foi chamado, é referido como zombaria.

Então, em vez de apagar o código, é criado um observador que não apenas substitui o código que precisa ser silenciado, mas observa que uma atividade específica teria acontecido no mundo real.

Configurar #

Vamos começar com uma classe de logger simples que, por uma questão de clareza, simplesmente exibe o texto enviado para o parâmetro (normalmente faria algo que pode ser problemático para testes de unidade, como atualizar um banco de dados):

class Logger {
    public function log($text) {
        echo $text;
    }
}

Agora, vamos criar uma classe Application. Ele aceita um objeto Logger como parâmetro para o método run, que por sua vez invoca o método log do Logger para capturar que o aplicativo foi iniciado.

class Application {
  public function run(Logger $logger) {
    // some code that starts up the application

    // send out a log that the application has started
    $logger->log('Application has started');
  }
}

Se o código a seguir foi executado conforme escrito:

$logger = new Logger();
$app = new Application();
$app->run($logger);

Em seguida, o texto “Application has started” seria exibido de acordo com o método log dentro do Logger.

Teste de unidade com simulação #

O teste de unidade da classe Application não precisa verificar o que acontece dentro do método Logger log, ele só precisa verificar se ele foi chamado.

No teste PHPUnit, um observador é criado para substituir a classe Logger. Esse observador é configurado para garantir que o método log seja invocado apenas uma vez, com o valor do parâmetro “Application has started”.

Em seguida, o observador é enviado para o método run, que verifica se de fato o método log foi chamado apenas uma vez e o caso de teste passa, mas nenhum texto foi exibido.

class ApplicationTest extends \PHPUnit_Framework_TestCase {

  public function testThatRunLogsApplicationStart() {

    // create the observer
    $mock = $this->createMock(Logger::class);
    $mock->expects($this->once())
        ->method('log')
        ->with('Application has started');
    
    // run the application with the observer which ensures the log method was called
    $app = new Application();
    $app->run($mock);
  
  }
}

stubbing simples #

Às vezes, há seções de código difíceis de testar, como acessar um banco de dados ou interagir com o usuário. Você pode eliminar essas seções de código, permitindo que o restante do código seja testado.

Vamos começar com uma classe que solicita ao usuário. Para simplificar, ele tem apenas dois métodos, um que realmente solicita ao usuário (que seria usado por todos os outros métodos) e aquele que vamos testar, que solicita e filtra apenas respostas sim e não. Observe que este código é excessivamente simplista para fins de demonstração.

class Answer
{
    // prompt the user and check if the answer is yes or no, anything else, return null
    public function getYesNoAnswer($prompt) {

        $answer = $this->readUserInput($prompt);

        $answer = strtolower($answer);
        if (($answer === "yes") || ($answer === "no")) {
            return $answer;
        } else {
            return null;
        }

    }

    // Simply prompt the user and return the answer
    public function readUserInput($prompt) {
        return readline($prompt);
    }

}

Para testar getYesNoAnswer, o readUserInput precisa ser stub para imitar as respostas de um usuário.

class AnswerTest extends PHPUnit_Framework_TestCase
{

    public function test_yes_no_answer() {

        $stub = $this->getMockBuilder(Answer::class)
                ->setMethods(["readUserInput"])
                ->getMock();

        $stub->method('readUserInput')
            ->will($this->onConsecutiveCalls("yes","junk"));

        // stub will return "yes"
        $answer = $stub->getYesNoAnswer("Student? (yes/no)");
        $this->assertSame("yes",$answer);

        // stub will return "junk"
        $answer = $stub->getYesNoAnswer("Student? (yes/no)");
        $this->assertNull($answer);


    }

}

A primeira linha de código cria o stub e usa getMockBuilder em vez de createMock. createMock é um atalho para chamar getMockBuilder com padrões. Um desses padrões é stub fora de todos os métodos. Para este exemplo, queremos testar getYesNoAnswer, para que não possa ser apagado. O getMockBuilder invoca setMethods para solicitar que apenas readUserInput seja removido.

A segunda linha de código cria o comportamento de stub. Ele apaga o método readUserInput e configura dois valores de retorno nas chamadas subsequentes, “yes” seguido por “junk”.

A terceira e quarta linhas de código testam getYesNoAnswer. Na primeira vez, a pessoa falsa responde com “sim” e o código testado retorna corretamente “sim”, pois é uma seleção válida. Na segunda vez, a pessoa falsa responde com “lixo” e o código testado retorna corretamente nulo.
