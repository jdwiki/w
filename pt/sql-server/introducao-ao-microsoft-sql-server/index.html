INSERT / SELECT / UPDATE / DELETE: o básico da linguagem de manipulação de dados #

Data Manipulation Language (DML para abreviar) inclui operações como INSERT, UPDATE e DELETE:

-- Create a table HelloWorld

CREATE TABLE HelloWorld (
    Id INT IDENTITY,
    Description VARCHAR(1000)
)


-- DML Operation INSERT, inserting a row into the table
INSERT INTO HelloWorld (Description) VALUES ('Hello World')


-- DML Operation SELECT, displaying the table 
SELECT * FROM HelloWorld  


-- Select a specific column from table
SELECT Description FROM HelloWorld


-- Display number of records in the table
SELECT Count(*) FROM HelloWorld


-- DML Operation UPDATE, updating a specific row in the table
UPDATE HelloWorld SET Description = 'Hello, World!' WHERE Id = 1


-- Selecting rows from the table (see how the Description has changed after the update?)
SELECT * FROM HelloWorld


-- DML Operation - DELETE, deleting a row from the table
DELETE FROM HelloWorld WHERE Id = 1


-- Selecting the table. See table content after DELETE operation 
SELECT * FROM HelloWorld

Neste script, estamos criando uma tabela para demonstrar algumas consultas básicas.

Os exemplos a seguir mostram como consultar tabelas:

USE Northwind;
GO
SELECT TOP 10 * FROM Customers 
ORDER BY CompanyName

irá selecionar os primeiros 10 registros da tabela Customer, ordenados pela coluna CompanyName do banco de dados Northwind (que é um dos bancos de dados de exemplo da Microsoft, pode ser baixado em [aqui](https:// technet.microsoft.com/en-us/library/ms143221(v=sql.105).aspx)):

Observe que Use Northwind; altera o banco de dados padrão para todas as consultas subsequentes. Você ainda pode fazer referência ao banco de dados usando a sintaxe totalmente qualificada na forma de [Database].[Schema].[Table]:

SELECT TOP 10 * FROM Northwind.dbo.Customers 
ORDER BY CompanyName

SELECT TOP 10 * FROM Pubs.dbo.Authors
ORDER BY City

Isso é útil se você estiver consultando dados de bancos de dados diferentes. Observe que dbo, especificado “entre” é chamado de esquema e precisa ser especificado ao usar a sintaxe totalmente qualificada. Você pode pensar nisso como uma pasta dentro do seu banco de dados. dbo é o esquema padrão. O esquema padrão pode ser omitido. Todos os outros esquemas definidos pelo usuário precisam ser especificados.

Se a tabela do banco de dados contiver colunas nomeadas como palavras reservadas, por exemplo, Data, você precisa colocar o nome da coluna entre colchetes, assim:

-- descending order
SELECT TOP 10 [Date] FROM dbo.MyLogTable
ORDER BY [Date] DESC

O mesmo se aplica se o nome da coluna contiver espaços em seu nome (o que não é recomendado). Uma sintaxe alternativa é usar aspas duplas em vez de colchetes, por exemplo:

-- descending order
SELECT top 10 "Date" from dbo.MyLogTable
order by "Date" desc

é equivalente, mas não tão comumente usado. Observe a diferença entre aspas duplas e aspas simples: Aspas simples são usadas para strings, ou seja,

-- descending order
SELECT top 10 "Date" from dbo.MyLogTable
where UserId='johndoe'
order by "Date" desc

é uma sintaxe válida. Observe que o T-SQL tem um prefixo N para os tipos de dados NChar e NVarchar, por exemplo.

SELECT TOP 10 * FROM Northwind.dbo.Customers 
WHERE CompanyName LIKE N'AL%'
ORDER BY CompanyName

retorna todas as empresas com um nome de empresa começando com AL (% é um curinga, use-o como você usaria o asterisco em uma linha de comando do DOS, por exemplo, DIR AL*). Para LIKE, há alguns curingas disponíveis, veja aqui para saber mais detalhes.

Ingressa #

As junções são úteis se você deseja consultar campos que não existem em uma única tabela, mas em várias tabelas. Por exemplo: Você deseja consultar todas as colunas da tabela Region no banco de dados Northwind. Mas você percebe que precisa também da RegionDescription, que é armazenada em uma tabela diferente, Region. No entanto, existe uma chave comum, RgionID que você pode usar para combinar essas informações em uma única consulta da seguinte forma (Top 5 apenas retorna as 5 primeiras linhas, omita-o para obter todas as linhas):

SELECT TOP 5 Territories.*, 
    Regions.RegionDescription 
FROM Territories 
INNER JOIN Region 
    ON Territories.RegionID=Region.RegionID
ORDER BY TerritoryDescription

irá mostrar todas as colunas de Territórios mais a coluna RegionDescription de Region. O resultado é ordenado por TerritoryDescription.

Aliases de tabela #

Quando sua consulta requer uma referência a duas ou mais tabelas, pode ser útil usar um Alias ​​de Tabela. Os aliases de tabela são referências abreviadas para tabelas que podem ser usadas no lugar de um nome de tabela completo e podem reduzir a digitação e a edição. A sintaxe para usar um alias é:

<TableName> [as] <alias>

Onde as é uma palavra-chave opcional. Por exemplo, a consulta anterior pode ser reescrita como:

SELECT TOP 5 t.*, 
    r.RegionDescription 
FROM Territories t
INNER JOIN Region r 
    ON t.RegionID = r.RegionID
ORDER BY TerritoryDescription

Os aliases devem ser exclusivos para todas as tabelas em uma consulta, mesmo se você usar a mesma tabela duas vezes. Por exemplo, se sua tabela Employee incluiu um campo SupervisorId, você pode usar esta consulta para retornar um funcionário e o nome de seu supervisor:

SELECT e.*, 
    s.Name as SupervisorName -- Rename the field for output
FROM Employee e
INNER JOIN Employee s
    ON e.SupervisorId = s.EmployeeId
WHERE e.EmployeeId = 111

Sindicatos #

Como vimos antes, um Join adiciona colunas de diferentes origens de tabela. Mas e se você quiser combinar linhas de diferentes origens? Neste caso, você pode usar uma UNION. Suponha que você esteja planejando uma festa e queira convidar não apenas os funcionários, mas também os clientes. Então você pode executar esta consulta para fazer isso:

SELECT FirstName+' '+LastName as ContactName, Address, City FROM Employees
UNION
SELECT ContactName, Address, City FROM Customers

Ele retornará nomes, endereços e cidades dos funcionários e clientes em uma única tabela. Observe que as linhas duplicadas (se houver) são eliminadas automaticamente (se você não quiser isso, use um UNION ALL). O número da coluna, os nomes das colunas, a ordem e o tipo de dados devem corresponder em todas as instruções select que fazem parte da união - é por isso que o primeiro SELECT combina FirstName e LastName de Employee em ContactName.

Variáveis ​​da Tabela #

Pode ser útil, se você precisar lidar com dados temporários (especialmente em um procedimento armazenado), usar variáveis ​​de tabela: A diferença entre uma tabela “real” e uma variável de tabela é que ela existe apenas na memória para processamento temporário.

Exemplo:

DECLARE @Region TABLE
(
  RegionID int, 
  RegionDescription NChar(50)
)

cria uma tabela na memória. Neste caso o prefixo @ é obrigatório porque é uma variável. Você pode executar todas as operações DML mencionadas acima para inserir, excluir e selecionar linhas, por exemplo,

INSERT INTO @Region values(3,'Northern')
INSERT INTO @Region values(4,'Southern')

Mas normalmente, você o preencheria com base em uma tabela real como

INSERT INTO @Region
SELECT * FROM dbo.Region WHERE RegionID>2;

que leria os valores filtrados da tabela real dbo.Region e os inseriria na tabela de memória @Region - onde pode ser usado para processamento adicional. Por exemplo, você pode usá-lo em uma junção como

SELECT * FROM Territories t
JOIN @Region r on t.RegionID=r.RegionID

que neste caso retornaria todos os territórios Norte e Sul. Informações mais detalhadas podem ser encontradas aqui. Tabelas temporárias são discutidas aqui, se você estiver interessado em ler mais sobre esse tópico .

OBSERVAÇÃO: a Microsoft só recomenda o uso de variáveis ​​de tabela se o número de linhas de dados na variável de tabela for menor que 100. Se você estiver trabalhando com grandes quantidades de dados, use uma tabela temporária, ou tabela temporária, em vez disso.

SELECIONE todas as linhas e colunas de uma tabela #

Sintaxe:

SELECT *
FROM table_name

Usar o operador asterisco * serve como um atalho para selecionar todas as colunas da tabela. Todas as linhas também serão selecionadas porque esta instrução SELECT não possui uma cláusula WHERE, para especificar qualquer critério de filtragem.

Isso também funcionaria da mesma maneira se você adicionasse um alias à tabela, por exemplo e neste caso:

SELECT *
FROM Employees AS e

Ou se você quiser selecionar tudo de uma tabela específica, você pode usar o alias + “.*”:

SELECT e.*, d.DepartmentName
FROM Employees AS e
    INNER JOIN Department AS d 
        ON e.DepartmentID = d.DepartmentID

Objetos de banco de dados também podem ser acessados ​​usando nomes totalmente qualificados:

SELECT * FROM [server_name].[database_name].[schema_name].[table_name]

Isso não é necessariamente recomendado, pois alterar os nomes do servidor e/ou do banco de dados faria com que as consultas usando nomes totalmente qualificados não fossem mais executadas devido a nomes de objetos inválidos.

Observe que os campos antes de table_name podem ser omitidos em muitos casos se as consultas forem executadas em um único servidor, banco de dados e esquema, respectivamente. No entanto, é comum que um banco de dados tenha vários esquemas e, nesses casos, o nome do esquema não deve ser omitido quando possível.

Aviso: usar SELECT * no código de produção ou procedimentos armazenados pode levar a problemas mais tarde (à medida que novas colunas são adicionadas à tabela ou se as colunas são reorganizadas na tabela), especialmente se seu código faz suposições simples sobre a ordem das colunas ou o número de colunas retornadas. Portanto, é mais seguro sempre especificar explicitamente os nomes das colunas nas instruções SELECT para o código de produção.

SELECT col1, col2, col3
FROM table_name

UPDATE linha específica #

UPDATE HelloWorlds
SET HelloWorld = 'HELLO WORLD!!!'
WHERE Id = 5

O código acima atualiza o valor do campo “HelloWorld” com “HELLO WORLD!!!” para o registro onde “Id = 5” na tabela HelloWorlds.

Observação: em uma instrução de atualização, é recomendável usar uma cláusula “where” para evitar atualizar a tabela inteira, a menos que e até que seu requisito seja diferente.

Comentários no código #

O Transact-SQL oferece suporte a duas formas de escrita de comentários. Os comentários são ignorados pelo mecanismo de banco de dados e devem ser lidos pelas pessoas.

Comentários são precedidos por -- e são ignorados até que uma nova linha seja encontrada:

-- This is a comment
SELECT *
FROM MyTable -- This is another comment
WHERE Id = 1;

Comentários com asteriscos começam com /* e terminam com */. Todo o texto entre esses delimitadores é considerado um bloco de comentários.

/* This is
a multi-line
comment block. */
SELECT Id = 1, [Message] = 'First row'
UNION ALL
SELECT 2, 'Second row'
/* This is a one liner */
SELECT 'More';

Os comentários de estrela de barra têm a vantagem de manter o comentário utilizável se a instrução SQL perder novos caracteres de linha. Isso pode acontecer quando o SQL é capturado durante a solução de problemas.

Comentários de estrela barra podem ser aninhados e um /* inicial dentro de um comentário de estrela barra precisa ser finalizado com um */ para ser válido. O código a seguir resultará em um erro

/*
SELECT *
FROM CommentTable
WHERE Comment = '/*'
*/

A estrela barra, embora dentro da citação, é considerada como o início de um comentário. Por isso, precisa ser finalizado com outra barra de estrela de fechamento. A forma correta seria

/*
SELECT *
FROM CommentTable
WHERE Comment = '/*'
*/  */

DELETE todas as linhas #

DELETE
FROM Helloworlds

Isso excluirá todos os dados da tabela. A tabela não conterá linhas depois que você executar esse código. Ao contrário de DROP TABLE, isso preserva a própria tabela e sua estrutura e você pode continuar a inserir novas linhas nessa tabela.

Outra maneira de excluir todas as linhas da tabela é truncá-la, conforme a seguir:

TRUNCATE TABLE HelloWords

As diferenças com a operação DELETE são várias:

  1. A operação de truncar não é armazenada no arquivo de log de transações
  2. Se existir o campo IDENTITY, este será redefinido
  3. TRUNCATE pode ser aplicado em toda a tabela e não em parte dela (ao invés do comando DELETE você pode associar uma cláusula WHERE)

Restrições de TRUNCATE

  1. Não é possível TRUNCATE uma tabela se houver uma referência FOREIGN KEY
  2. Se a mesa for participada em uma VISUALIZAÇÃO INDEXADA
  3. Se a tabela for publicada usando TRANSACTIONAL REPLICATION ou MERGE REPLICATION
  4. Não irá disparar nenhum TRIGGER definido na tabela

[sic]

IMPRIMIR #

Exiba uma mensagem no console de saída. Usando o SQL Server Management Studio, isso será exibido na guia de mensagens, em vez da guia de resultados:

PRINT 'Hello World!';

Selecione as linhas que correspondem a uma condição #

Geralmente, a sintaxe é:

SELECT <column names>
FROM <table name>
WHERE <condition>

Por exemplo:

SELECT FirstName, Age
FROM Users
WHERE LastName = 'Smith'

As condições podem ser complexas:

SELECT FirstName, Age
FROM Users
WHERE LastName = 'Smith' AND (City = 'New York' OR City = 'Los Angeles')

ATUALIZAR todas as linhas #

Uma forma simples de atualização é incrementar todos os valores em um determinado campo da tabela. Para isso, precisamos definir o campo e o valor do incremento

Veja a seguir um exemplo que incrementa o campo Score em 1 (em todas as linhas):

UPDATE Scores
SET score = score + 1

Isso pode ser perigoso, pois você pode corromper seus dados se fizer um UPDATE acidentalmente para uma linha específica com um UPDATE para Todas as linhas na tabela.

Recuperar informações básicas do servidor #

SELECT @@VERSION

Retorna a versão do MS SQL Server em execução na instância.

SELECT @@SERVERNAME

Retorna o nome da instância do MS SQL Server.

SELECT @@SERVICENAME

Retorna o nome do serviço do Windows como o MS SQL Server está sendo executado.

SELECT serverproperty('ComputerNamePhysicalNetBIOS');

Retorna o nome físico da máquina em que o SQL Server está sendo executado. Útil para identificar o nó em um cluster de failover.

SELECT * FROM fn_virtualservernodes();

Em um cluster de failover retorna todos os nós em que o SQL Server pode ser executado. Ele não retorna nada se não um cluster.

TRUNCATE TABLE #

TRUNCATE TABLE Helloworlds

Este código excluirá todos os dados da tabela Helloworlds. Truncar tabela é quase semelhante ao código Excluir da tabela. A diferença é que você não pode usar cláusulas where com Truncar. Truncar a tabela é considerado melhor do que excluir porque usa menos espaços de log de transações.

Observe que, se existir uma coluna de identidade, ela será redefinida para o valor de semente inicial (por exemplo, o ID incrementado automaticamente será reiniciado a partir de 1). Isso pode levar à inconsistência se as colunas de identidade forem usadas como chave estrangeira em outra tabela.

Cria uma nova tabela e insere registros da tabela antiga #

SELECT * INTO NewTable FROM OldTable

Cria uma nova tabela com a estrutura da tabela antiga e insere todas as linhas na nova tabela.

Algumas Restrições

  1. Você não pode especificar uma variável de tabela ou parâmetro com valor de tabela como a nova tabela.
  2. Você não pode usar SELECT…INTO para criar uma tabela particionada, mesmo quando a tabela de origem estiver particionada. SELECT…INTO não usa o esquema de partição da tabela de origem; em vez disso, a nova tabela é criado no grupo de arquivos padrão. Para inserir linhas em um particionado table, você deve primeiro criar a tabela particionada e depois usar o Instrução INSERT INTO…SELECT FROM.
  3. Índices, restrições e gatilhos definidos na tabela de origem não são transferidos para a nova tabela, nem podem ser especificados no Instrução SELECT…INTO. Se esses objetos forem necessários, você pode criar depois de executar a instrução SELECT…INTO.
  4. A especificação de uma cláusula ORDER BY não garante que as linhas sejam inseridas na ordem especificada. Quando uma coluna esparsa é incluída em na lista de seleção, a propriedade da coluna esparsa não é transferida para a coluna na nova tabela. Se esta propriedade for exigida no novo tabela, altere a definição da coluna após executar o SELECT…INTO declaração para incluir esta propriedade.
  5. Quando uma coluna computada é incluída na lista de seleção, a coluna correspondente na nova tabela não é uma coluna computada. o os valores na nova coluna são os valores que foram calculados no momento SELECT…INTO foi executado.

[sic]

Usando transações para alterar dados com segurança #

Sempre que você altera dados, em um comando Data Manipulation Language (DML), você pode agrupar suas alterações em uma transação. DML inclui UPDATE, TRUNCATE, INSERT e DELETE. Uma das maneiras de garantir que você está alterando os dados corretos seria usar uma transação.

As alterações de DML bloquearão as linhas afetadas. Ao iniciar uma transação, você deve encerrar a transação ou todos os objetos que estão sendo alterados na DML permanecerão bloqueados por quem iniciou a transação. Você pode encerrar sua transação com ROLLBACK ou COMMIT. ROLLBACK retorna tudo dentro da transação ao seu estado original. COMMIT coloca os dados em um estado final onde você não pode desfazer suas alterações sem outra instrução DML.

Exemplo:

--Create a test table

USE [your database]
GO
CREATE TABLE test_transaction (column_1 varchar(10))
GO

INSERT INTO 
 dbo.test_transaction
        ( column_1 )
VALUES
        ( 'a' )

BEGIN TRANSACTION --This is the beginning of your transaction

UPDATE dbo.test_transaction
SET column_1 = 'B'
OUTPUT INSERTED.*
WHERE column_1 = 'A'
  

ROLLBACK TRANSACTION  --Rollback will undo your changes
           --Alternatively, use COMMIT to save your results

SELECT * FROM dbo.test_transaction   --View the table after your changes have been run

DROP TABLE dbo.test_transaction

Notas:

  • Este é um exemplo simplificado que não inclui tratamento de erros. Mas qualquer operação de banco de dados pode falhar e, portanto, lançar uma exceção. Aqui está um exemplo como esse tratamento de erros obrigatório pode parecer. Você nunca deve usar transações sem um gerenciador de erros, caso contrário você pode deixar a transação em um estado desconhecido.
  • Dependendo do nível de isolamento, as transações estão colocando bloqueios nos dados que estão sendo consultados ou alterado. Você precisa garantir que as transações não estejam em execução por um longo tempo, porque elas bloquearão registros em um banco de dados e podem levar a [deadlocks](https://www.simple-talk.com/sql/performance/sql- server-deadlocks-by-example/) com outras transações paralelas. Mantenha as operações encapsuladas em transações o mais curtas possível e minimize o impacto com a quantidade de dados que você está bloqueando.

Obtendo a contagem de linhas da tabela #

O exemplo a seguir pode ser usado para encontrar a contagem total de linhas para uma tabela específica em um banco de dados se table_name for substituído pela tabela que você deseja consultar:

SELECT COUNT(*) AS [TotalRowCount] FROM table_name;

Também é possível obter a contagem de linhas para todas as tabelas juntando-se novamente à partição da tabela com base no HEAP das tabelas (index_id = 0) ou no índice clusterizado do cluster (index_id = 1) usando o seguinte script:

SELECT  [Tables].name                AS [TableName],
        SUM( [Partitions].[rows] )    AS [TotalRowCount]
FROM    sys.tables AS [Tables]
JOIN    sys.partitions AS [Partitions]
    ON  [Tables].[object_id]    =    [Partitions].[object_id]
    AND [Partitions].index_id IN ( 0, 1 )
--WHERE    [Tables].name = N'table name' /* uncomment to look for a specific table */
GROUP BY    [Tables].name;

Isso é possível porque cada tabela é essencialmente uma única tabela de partição, a menos que partições extras sejam adicionadas a ela. Este script também tem a vantagem de não interferir nas operações de leitura/gravação nas linhas das tabelas'.
