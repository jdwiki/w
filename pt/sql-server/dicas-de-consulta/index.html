Dicas de PARTICIPAÇÃO #

Quando você associa duas tabelas, o otimizador de consultas (QO) do SQL Server pode escolher diferentes tipos de associações que serão usadas na consulta:

  • Junção de HASH
  • Junte-se ao LOOP
  • MERGE juntar

O QO explorará os planos e escolherá o operador ideal para unir as tabelas. No entanto, se você tiver certeza de que sabe qual seria o operador de junção ideal, poderá especificar que tipo de JOIN deve ser usado. A junção de LOOP interno forçará o QO a escolher Junção de loop aninhado ao unir duas tabelas:

select top 100 *
from Sales.Orders o 
    inner loop join Sales.OrderLines ol 
    on o.OrderID = ol.OrderID

A junção de mesclagem interna forçará o operador de junção de MERGE:

select top 100 *
from Sales.Orders o 
    inner merge join Sales.OrderLines ol 
    on o.OrderID = ol.OrderID

A junção de hash interna forçará o operador de junção de HASH:

select top 100 *
from Sales.Orders o 
    inner hash join Sales.OrderLines ol 
    on o.OrderID = ol.OrderID

Agrupar por dicas #

Quando você usa a cláusula GROUP BY, o otimizador de consulta (QO) do SQL Server pode escolher diferentes tipos de operadores de agrupamento:

  • HASH Aggregate que cria hash-map para agrupar entradas
  • Stream Aggregate que funciona bem com entradas pré-encomendadas

Você pode exigir explicitamente que o QO escolha um ou outro operador agregado se souber qual seria o ideal. Com OPTION (ORDER GROUP), o QO sempre escolherá o agregado Stream e adicionará o operador Sort na frente do agregado Stream se a entrada não for classificada:

select OrderID, AVG(Quantity)
from Sales.OrderLines
group by OrderID
OPTION (ORDER GROUP)

Com OPTION (HASH GROUP), QO sempre escolherá Hash agregado:

select OrderID, AVG(Quantity)
from Sales.OrderLines
group by OrderID
OPTION (HASH GROUP)

Dica de linhas FAST #

Especifica que a consulta é otimizada para recuperação rápida das primeiras number_rows. Este é um número inteiro não negativo. Depois que as primeiras number_rows são retornadas, a consulta continua a execução e produz seu conjunto de resultados completo.

select OrderID, AVG(Quantity)
from Sales.OrderLines
group by OrderID
OPTION (FAST 20)

dicas de UNIÃO #

Quando você usa o operador UNION em dois resultados de consulta, o otimizador de consulta (QO) pode usar os seguintes operadores para criar uma união de dois conjuntos de resultados:

  • Fusão (União)
  • Concat (União)
  • Correspondência de Hash (União)

Você pode especificar explicitamente qual operador deve ser usado usando a dica OPTION():

select OrderID, OrderDate, ExpectedDeliveryDate, Comments
from Sales.Orders
where OrderDate > DATEADD(day, -1, getdate())
UNION
select PurchaseOrderID as OrderID, OrderDate, ExpectedDeliveryDate, Comments
from Purchasing.PurchaseOrders
where OrderDate > DATEADD(day, -1, getdate())
OPTION(HASH UNION)
-- or OPTION(CONCAT UNION)
-- or OPTION(MERGE UNION)

Opção MAXDOP #

Especifica o grau máximo de paralelismo para a consulta que especifica esta opção.

SELECT OrderID,
    AVG(Quantity)
FROM Sales.OrderLines
GROUP BY OrderID
OPTION (MAXDOP 2);

Essa opção substitui a opção de configuração MAXDOP de sp_configure e Resource Governor. Se MAXDOP for definido como zero, o servidor escolherá o grau máximo de paralelismo.

Dicas de ÍNDICE #

As dicas de índice são usadas para forçar uma consulta a usar um índice específico, em vez de permitir que o Otimizador de Consultas do SQL Server escolha o que considera o melhor índice. Em alguns casos, você pode obter benefícios especificando o índice que uma consulta deve usar. Normalmente, o Query Optimizer do SQL Server escolhe o índice mais adequado para a consulta, mas devido a estatísticas ausentes/desatualizadas ou necessidades específicas, você pode forçá-lo.

SELECT *
FROM mytable WITH (INDEX (ix_date))
WHERE field1 > 0
    AND CreationDate > '20170101'
