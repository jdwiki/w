Sintaxe #

  • DEFINIR O NÍVEL DE ISOLAMENTO DA TRANSAÇÃO { LEIA SEM COMPROMISSO | LEIA COMPROMETIDA | LEITURA REPETÍVEL | INSTANTÂNEO | SERIALIZÁVEL } [ ; ]

Referência do MSDN: SET TRANSACTION ISOLATION LEVEL

Ler não confirmado #

SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED

Este é o nível de isolamento mais permissivo, pois não causa nenhum bloqueio. Ele especifica que as instruções podem ler todas as linhas, incluindo linhas que foram escritas em transações, mas ainda não confirmadas (ou seja, elas ainda estão em transação). Esse nível de isolamento pode estar sujeito a “leituras sujas”.

Leitura confirmada #

SET TRANSACTION ISOLATION LEVEL READ COMMITTED

Este nível de isolamento é o 2º mais permissivo. Evita leituras sujas. O comportamento de READ COMMITTED depende da configuração do READ_COMMITTED_SNAPSHOT:

  • Se definido como OFF (a configuração padrão), a transação usa bloqueios compartilhados para impedir que outras transações modifiquem linhas usadas pela transação atual, bem como bloquear a transação atual de ler linhas modificadas por outras transações.

  • Se definido como ON, a dica de tabela READCOMMITTEDLOCK pode ser usada para solicitar bloqueio compartilhado em vez de controle de versão de linha para transações executadas no modo READ COMMITTED.

Nota: READ COMMITTED é o comportamento padrão do SQL Server.

O que são “leituras sujas”? #

Leituras sujas (ou leituras não confirmadas) são leituras de linhas que estão sendo modificadas por uma transação aberta.

Esse comportamento pode ser replicado usando 2 consultas separadas: uma para abrir uma transação e gravar alguns dados em uma tabela sem confirmar, a outra para selecionar os dados a serem gravados (mas ainda não confirmados) com esse nível de isolamento.

Consulta 1 - Preparar uma transação, mas não finalizá-la:

CREATE TABLE dbo.demo (
    col1 INT,
    col2 VARCHAR(255)
);
GO
--This row will get committed normally:
BEGIN TRANSACTION;
    INSERT INTO dbo.demo(col1, col2) 
    VALUES (99, 'Normal transaction');
COMMIT TRANSACTION;
--This row will be "stuck" in an open transaction, causing a dirty read
BEGIN TRANSACTION;
    INSERT INTO dbo.demo(col1, col2) 
    VALUES (42, 'Dirty read');
--Do not COMMIT TRANSACTION or ROLLBACK TRANSACTION here

Consulta 2 - Leia as linhas, incluindo a transação aberta:

SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT * FROM dbo.demo;

Devoluções:

col1 col2

99 Transação normal 42 Leitura suja

P.S.: Não se esqueça de limpar esses dados de demonstração:

COMMIT TRANSACTION;
DROP TABLE dbo.demo;
GO

Leitura repetível #

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ

Este nível de isolamento de transação é um pouco menos permissivo do que READ COMMITTED, pois bloqueios compartilhados são colocados em todos os dados lidos por cada instrução na transação e são mantidos até que a transação seja concluída, em vez de serem liberados após cada instrução .

Nota: Use esta opção apenas quando necessário, pois é mais provável que cause degradação do desempenho do banco de dados, bem como deadlocks, do que READ COMMITTED.

Instantâneo #

SET TRANSACTION ISOLATION LEVEL SNAPSHOT

Especifica que os dados lidos por qualquer instrução em uma transação serão a versão transacionalmente consistente dos dados que existiam no início da transação, ou seja, apenas lerá os dados que foram confirmados antes do início da transação.

As transações SNAPSHOT não solicitam ou causam bloqueios nos dados que estão sendo lidos, pois estão apenas lendo a versão (ou instantâneo) dos dados que existiam no momento em que a transação começou.

Uma transação em execução no nível de isolamento SNAPSHOT lê apenas suas próprias alterações de dados enquanto está em execução. Por exemplo, uma transação pode atualizar algumas linhas e depois ler as linhas atualizadas, mas essa alteração só será visível para a transação atual até que ela seja confirmada.

Nota: A opção de banco de dados ALLOW_SNAPSHOT_ISOLATION deve ser definida como ON antes que o nível de isolamento SNAPSHOT possa ser usado.

Serializavel #

SET TRANSACTION ISOLATION LEVEL SERIALIZEABLE

Este nível de isolamento é o mais restritivo. Ele solicita bloqueios de intervalo o intervalo de valores de chave que são lidos por cada instrução na transação. Isso também significa que as instruções INSERT de outras transações serão bloqueadas se as linhas a serem inseridas estiverem no intervalo bloqueado pela transação atual.

Esta opção tem o mesmo efeito que definir HOLDLOCK em todas as tabelas em todas as instruções SELECT em uma transação.

Nota: Este isolamento de transação tem a menor simultaneidade e só deve ser usado quando necessário.
