O SQL Server 2016 apresenta suporte para tabelas temporais com versão do sistema como um recurso de banco de dados que oferece suporte interno para fornecer informações sobre dados armazenados na tabela a qualquer momento, em vez de apenas os dados corretos no momento atual.

Uma tabela temporal com versão do sistema é um novo tipo de tabela de usuário no SQL Server 2016, projetada para manter um histórico completo de alterações de dados e permitir uma fácil análise pontual no tempo. Esse tipo de tabela temporal é chamado de tabela temporal com versão do sistema porque o período de validade de cada linha é gerenciado pelo sistema (ou seja, mecanismo de banco de dados). Cada tabela temporal tem duas colunas definidas explicitamente, cada uma com um tipo de dados datetime2. Essas colunas são chamadas de colunas de período. Essas colunas de período são usadas exclusivamente pelo sistema para registrar o período de validade de cada linha sempre que uma linha for modificada.

CRIAR Tabelas Temporais #

CREATE TABLE dbo.Employee   
(    
  [EmployeeID] int NOT NULL PRIMARY KEY CLUSTERED   
  , [Name] nvarchar(100) NOT NULL  
  , [Position] varchar(100) NOT NULL   
  , [Department] varchar(100) NOT NULL  
  , [Address] nvarchar(1024) NOT NULL  
  , [AnnualSalary] decimal (10,2) NOT NULL  
  , [ValidFrom] datetime2 (2) GENERATED ALWAYS AS ROW START  
  , [ValidTo] datetime2 (2) GENERATED ALWAYS AS ROW END  
  , PERIOD FOR SYSTEM_TIME (ValidFrom, ValidTo)  
 )    
 WITH (SYSTEM_VERSIONING = ON (HISTORY_TABLE = dbo.EmployeeHistory));

INSERTS: em um INSERT, o sistema define o valor da coluna ValidFrom para a hora de início da transação atual (no fuso horário UTC) com base no relógio do sistema e atribui o valor da coluna ValidTo para o valor máximo de 9999-12-31. Isso marca a linha como aberta.

UPDATES: em um UPDATE, o sistema armazena o valor anterior da linha na tabela de histórico e define o valor da coluna ValidTo para a hora de início da transação atual (no fuso horário UTC) com base no relógio do sistema. Isso marca a linha como fechada, com um período registrado para o qual a linha era válida. Na tabela atual, a linha é atualizada com seu novo valor e o sistema define o valor da coluna ValidFrom para a hora de início da transação (no fuso horário UTC) com base no relógio do sistema. O valor da linha atualizada na tabela atual para a coluna ValidTo permanece o valor máximo de 9999-12-31.

DELETES: em um DELETE, o sistema armazena o valor anterior da linha na tabela de histórico e define o valor da coluna ValidTo para a hora de início da transação atual (no fuso horário UTC) com base no relógio do sistema. Isso marca a linha como fechada, com um período registrado para o qual a linha anterior era válida. Na tabela atual, a linha é removida. As consultas da tabela atual não retornarão esta linha. Somente consultas que lidam com dados de histórico retornam dados para os quais uma linha está fechada.

MERGE: em um MERGE, a operação se comporta exatamente como se até três instruções (um INSERT, um UPDATE e/ou um DELETE) fossem executados , dependendo do que é especificado como ações na instrução MERGE.

Dica: as horas registradas nas colunas datetime2 do sistema são baseadas na hora de início da própria transação. Por exemplo, todas as linhas inseridas em uma única transação terão o mesmo horário UTC registrado na coluna correspondente ao início do período SYSTEM_TIME.

PARA SYSTEM_TIME TODOS #

Retorna a união das linhas que pertencem à tabela atual e ao histórico.

SELECT * FROM Employee
    FOR SYSTEM_TIME ALL

Criando uma tabela temporal com versão do sistema otimizada para memória e limpando a tabela de histórico do SQL Server #

Criar uma tabela temporal com uma tabela de histórico padrão é uma opção conveniente quando você deseja controlar a nomenclatura e ainda contar com o sistema para criar uma tabela de histórico com a configuração padrão. No exemplo abaixo, uma nova tabela temporal com otimização de memória com versão do sistema vinculada a uma nova tabela de histórico baseada em disco.

CREATE SCHEMA History  
GO  
CREATE TABLE dbo.Department   
(  
    DepartmentNumber char(10) NOT NULL PRIMARY KEY NONCLUSTERED,   
    DepartmentName varchar(50) NOT NULL,   
    ManagerID int  NULL,   
    ParentDepartmentNumber char(10) NULL,   
    SysStartTime datetime2 GENERATED ALWAYS AS ROW START HIDDEN NOT NULL,   
    SysEndTime datetime2 GENERATED ALWAYS AS ROW END HIDDEN NOT NULL,     
    PERIOD FOR SYSTEM_TIME (SysStartTime,SysEndTime)     
)  
WITH   
    (  
        MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA,  
        SYSTEM_VERSIONING = ON ( HISTORY_TABLE = History.DepartmentHistory )   
    );

Limpando a tabela de histórico do SQL Server Com o tempo, a tabela de histórico pode crescer significativamente. Como não é permitido inserir, atualizar ou excluir dados da tabela de histórico, a única maneira de limpar a tabela de histórico é primeiro desabilitar o controle de versão do sistema:

ALTER TABLE dbo.Employee

SET (SYSTEM_VERSIONING = OFF); VAI

Exclua dados desnecessários da tabela de histórico:

    DELETE FROM dbo.EmployeeHistory

WHERE EndTime <= ‘2017-01-26 14:00:29’;

e, em seguida, reative o controle de versão do sistema:

ALTER TABLE dbo.Employee

SET (SYSTEM_VERSIONING = ON (HISTORY_TABLE = [dbo].[EmployeeHistory], DATA_CONSISTENCY_CHECK = ON));

Limpar a tabela de histórico nos Bancos de Dados SQL do Azure é um pouco diferente, pois os bancos de dados SQL do Azure têm suporte interno para limpeza da tabela de histórico. Primeiro, a limpeza de retenção de histórico temporal precisa ser habilitada em um nível de banco de dados:

ALTER DATABASE CURRENT

DEFINIR TEMPORAL_HISTORY_RETENTION ATIVADO VAI

Em seguida, defina o período de retenção por tabela:

ALTER TABLE dbo.Employee

SET (SYSTEM_VERSIONING = ON (HISTORY_RETENTION_PERIOD = 90 DIAS));

Isso excluirá todos os dados da tabela de histórico com mais de 90 dias. Os bancos de dados locais do SQL Server 2016 não dão suporte a TEMPORAL_HISTORY_RETENTION e HISTORY_RETENTION_PERIOD e qualquer uma das duas consultas acima é executada nos bancos de dados locais do SQL Server 2016, os seguintes erros ocorrerão.

Para o erro TEMPORAL_HISTORY_RETENTION será:

Msg 102, Level 15, State 6, Line 34

Sintaxe incorreta perto de ‘TEMPORAL_HISTORY_RETENTION’.

Para o erro HISTORY_RETENTION_PERIOD será:

Msg 102, Level 15, State 1, Line 39

Sintaxe incorreta perto de ‘HISTORY_RETENTION_PERIOD’.

Como consultar dados temporais? #

SELECT * FROM Employee   
    FOR SYSTEM_TIME    
        BETWEEN '2014-01-01 00:00:00.0000000' AND '2015-01-01 00:00:00.0000000'   
            WHERE EmployeeID = 1000 ORDER BY ValidFrom;

Retorna o valor real especificado no momento (FOR SYSTEM_TIME AS OF <date_time>) #

Retorna uma tabela com linhas contendo os valores que eram reais (atuais) no momento especificado no passado.

SELECT * FROM Employee   
    FOR SYSTEM_TIME AS  OF '2016-08-06 08:32:37.91'

PARA SYSTEM_TIME ENTRE <start_date_time> AND <end_date_time> #

O mesmo que acima na descrição FOR SYSTEM_TIME FROM <start_date_time>TO <end_date_time>, exceto que a tabela de linhas retornadas inclui linhas que se tornaram ativas no limite superior definido pelo ponto de extremidade <end_date_time>.

SELECT * FROM Employee   
    FOR SYSTEM_TIME BETWEEN  '2015-01-01' AND '2015-12-31'

PARA SYSTEM_TIME FROM <start_date_time> TO <end_date_time> #

Retorna uma tabela com os valores de todas as versões de linha que estavam ativas dentro do intervalo de tempo especificado, independentemente de terem começado a ser ativas antes do valor do parâmetro <start_date_time> para o argumento FROM ou deixaram de ser ativas após o valor do parâmetro <end_date_time> para o PARA argumento. Internamente, uma união é realizada entre a tabela temporal e sua tabela de histórico e os resultados são filtrados para retornar os valores de todas as versões de linha que estavam ativas a qualquer momento durante o intervalo de tempo especificado. As linhas que se tornaram ativas exatamente no limite inferior definido pelo terminal FROM são incluídas e os registros que se tornaram ativos exatamente no limite superior definido pelo terminal TO não são incluídos.

SELECT * FROM Employee   
    FOR SYSTEM_TIME FROM '2015-01-01' TO '2015-12-31'

FOR SYSTEM_TIME CONTAINED IN (<start_date_time> , <end_date_time>) #

Retorna uma tabela com os valores de todas as versões de linha que foram abertas e fechadas dentro do intervalo de tempo especificado definido pelos dois valores de data e hora para o argumento CONTAINED IN. As linhas que se tornaram ativas exatamente no limite inferior ou deixaram de ser ativas exatamente no limite superior são incluídas.

SELECT * FROM Employee
    FOR SYSTEM_TIME CONTAINED IN ('2015-04-01', '2015-09-25')
