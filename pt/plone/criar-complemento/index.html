Um conto sobre como criar um add-on para o Plone #

Preâmbulo #

TL;TR: Para fins de ilustração, as linhas deste documento começam com ‘TL;TR:’ são seguidos pela linha de comando referindo-se aos scripts Plone-helper do ovo ‘adi.devgen’. Se você executar o comando dado após ‘TL;TR:’, ele crie todos os arquivos explicados no capítulo seguinte. Existem muitas ferramentas de script auxiliares alternativas, no entanto, caso você queira use esta ferramenta, você pode instalá-lo rapidamente assim da linha de comando:

pip install adi.devgen

Estrutura de um Plone-Add-on mínimo #

TL;TR: devgen addBase mynamespace.myaddon

Assim como o Plone consiste em mais de 200 ovos Python, um Plone-Add-On também é um Python-egg e um egg é um módulo Python com namespace. Qualquer ovo estará disponível para o interpretador Python do ZOPE. Para fazer um egg instalável em um site Plone, ele também precisa de um chamado ‘perfil’. A seguir irá explicar esses termos.

Módulo Python #

Um módulo Python é um diretório que contém pelo menos um arquivo que deve ser chamado __init__.py. O nome é um sinalizador para o interpretador Python reconhecer este diretório como um módulo, significando que seu caminho pode ser importado de outros Scripts Python, por exemplo assumindo que temos um diretório chamado ‘myaddon’, que contém um arquivo chamado ‘myfile’ e esse arquivo contém uma definição chamada ‘myDefinition’, você pode importar a definição de outro arquivo Python como este:

from myaddon.myfile import myDefinition

Python-ovo #

Um ovo Python é um módulo Python com namespace, um namespace pode ser qualquer nome, mas é melhor não ser tomado já, caso você queira compartilhar seu complemento com o mundo.

A razão para o namespace é principalmente excluir a possibilidade de que dois ou mais módulos podem ter o mesmo nome (neste exemplo chamado ‘myaddon’), pois resultaria em conflitos quando um script Python tenta importar o caminho, sendo não é mais único.

Aqui está uma visão geral da estrutura de diretórios e arquivos envolvidos, antes de continuar a explicar seus propósitos em detalhes mais adiante:

mynamespace.myaddon
    setup.py
    mynamespace
        __init__.py
        myaddon
            __init__.py
            myfile.py

Para namespace um add-on primeiro crie um diretório chamado ‘mynamespace.myaddon’.

Este diretório deve conter o arquivo ‘setup.py’, torna o ovo registrável para o interpretador Python da instância ZOPE, isso é explicado em mais detalhes no próximo capítulo[TODO].

Em seguida, nesse diretório, crie outro diretório com o nome do seu namespace: ‘mynamespace’ Agora, tornamos este diretório um caminho de módulo importável, adicionando o arquivo com a palavra-chave __init__.py nele. Também deve conter os seguintes linhas, que fazem a mágica do namespace acontecer, explicada no último parágrafo:

[TODO: namespace-magic-lines]

Ufa quase pronto, por último neste novo diretório colocamos o módulo do anterior capítulo: O diretório ‘myaddon’, contendo o ‘init.py’ e ‘myfile.py’ com o ‘myDefinition’.

Agora temos um ovo registrável para a instância ZOPE e podemos referenciar seu métodos de qualquer outro ovo registrado, usando esta notação de caminho:

from mynamespace.myaddon.myfile import myDefinition

Você pode ver a mágica, os dois primeiros diretórios são omitidos, não:

from mynamespace.myaddon.mynamespace.myaddon.myfile import myDefinition

Fazendo um addon instalável dentro de um site Plone #

Um ‘perfil’ fará com que o complemento apareça no painel de controle de complementos de um site, para que um administrador possa (des)ativá-lo lá para o site.

Além disso, como um site Plone é sempre um filho de uma instância ZOPE e um A instância ZOPE pode conter vários sites, talvez não queiramos ter componentes involuntários de nosso add-on instalados em outros sites Plone, portanto, nós os vinculamos a um ‘perfil’.

TL;TR: devgen addProfile mynamespace.myaddon

Isso nos leva às partes realmente interessantes, a ZOPE-Component-Architecture, controlado pelos arquivos que terminam com ‘.zcml’, que significa ‘ZOPE-Component- Linguagem de marcação’. Com ele você pode cadastrar um perfil, vincular views a interfaces e muito mais. Na verdade, ele merece um capítulo principal próprio e um subcapítulo para cada um dos diretórios chamados “perfil”, “skins”, “navegador” e “conteúdo”.

Visão geral da estrutura do arquivo #

Esqueleto mínimo #

mynamespace.myaddon     | The container-directory of the add-on.

    setup.py            | Register this directory to the Python-interpreter of
                        | the ZOPE-instance.

    mynamespace         | The namespace-directory of the add-on.

        __init__.py     | Makes this directory a Python-module and contains
                        | the lines for the namespace-magic.

        myaddon         | The base-directory of the add-on.

            __init__.py | Empty file, make this directory a Python-module.

            myfile.py   | Example-file: If it contained a method called
                        | 'myMethod', it would be importable of any other
                        | registered egg like this:
                        |    from mynamespace.myaddon.myfile import myMethod

Componentes opcionais #

O diretório ‘mynamespace.myaddon/mynamespace/myaddon’ pode conter:

configure.zcml  | Register all following files and directories, here.

setuphandlers.py | Make an addon available to ZOPE-instance.

profiles        | Make an addon installable to Plone-site and generally
                | hold GenericSetup-xml-files.

skins           | The "old-school"-way, where resources like templates,
                | stylesheets, etc., live.

browser         | The "new-school"-way, where resources live.

content         | Holds the Python-files for defining custom content-types.

subscriber.py   | Subscribers listen to events, describe here what to do
                | when an event happens, e.g.: "An item has been added to the
                | site, notify admin about it via e-mail-notifica."
