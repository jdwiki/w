Iniciando e Parando #

Para iniciar ao ar livre:

  1. Mude para o usuário ao ar livre
  2. Mude para o diretório $ALFRESCO_HOME
  3. Execute ./alfresco.sh start

Para parar ao ar livre:

  1. Mude para o usuário ao ar livre
  2. Mude para o diretório $ALFRESCO_HOME
  3. Execute ./alfresco.sh start

Exploração madeireira #

Os logs ao ar livre ficam em $ALFRESCO_HOME/tomcat/logs/catalina.out.

Cópias de segurança #

Há muitas maneiras de fazer backup de um sistema Alfresco. É importante fazer backup do banco de dados, bem como do armazenamento de conteúdo. Você também pode querer fazer backup dos índices Solr.

Supondo que você instalou usando o instalador binário e tudo está em $ALRESCO_HOME, você pode fazer backup do banco de dados assim:

  1. Pare ao ar livre
  2. Mude para o usuário ao ar livre
  3. Mude para o diretório $ALFRESCO_HOME/postgresql/bin
  4. Faça o dump do banco de dados com ./pg_dump alfresco --user alfresco > $ALFRESCO_HOME/alf_data/db-backup.sql. Você pode ser solicitado a fornecer uma senha. Deve ser a mesma coisa que a senha de administrador que você forneceu durante a instalação. Caso contrário, verifique $ALFRESCO_HOME/tomcat/shared/classes/alfresco-global.properties para obter a senha do banco de dados.

Agora você tem seu banco de dados de backup. É importante fazer isso primeiro. A próxima etapa é fazer backup do armazenamento de conteúdo.

  1. Altere para o diretório $ALFRESCO_HOME/alf_data/contentstore.deleted.
  2. Apague tudo aqui. Não há motivo para manter esses arquivos por perto e não há motivo para fazer backup deles.
  3. Altere para o diretório $ALFRESCO_HOME/alf_data.
  4. Tar up a coisa toda, que também incluirá o backup do banco de dados que você criou na etapa anterior, assumindo que você o colocou no diretório alf_data. Execute tar czvf ~/alfresco-backup.tar.gz ..

O conteúdo desse arquivo TAR agora tem tudo o que você precisa para restaurar seu sistema de trabalho.

Auditoria #

A auditoria é um recurso Alfresco que permite ao usuário rastrear e registrar alguns eventos específicos durante o uso da plataforma ECM.

Ativar auditoria

Para habilitar a auditoria você deve adicionar algumas linhas de configuração ao arquivo alfresco-global.properties, que reside em tomcat/shared/classes/

audit.enabled = true
audit.alfresco-access.enabled=true

Você tem que salvar as alterações no arquivo alfresco-global.properties e reiniciar o servidor Alfresco, para habilitar a auditoria.

Auditoria da configuração padrão

Aqui a lista completa de propriedades de configuração que podem ser substituídas modificando o arquivo alfresco-global.properties:

# Audit configuration                                                                                                                                                                       
audit.enabled=true
audit.tagging.enabled=true
audit.alfresco-access.enabled=false
audit.alfresco-access.sub-actions.enabled=false
audit.cmischangelog.enabled=false
audit.dod5015.enabled=false

# Setting this flag to true will force startup failure when invalid audit configurations are detected                                                                                       
audit.config.strict=false

# Audit map filter for AccessAuditor - restricts recorded events to user driven events. In this case it neglect events issued by a System or a null user, the content or folder path is under /sys:archivedItem or under /ver: and the node type is not cm:folder, cm:content or st:site                                                                                                     
audit.filter.alfresco-access.default.enabled=false
audit.filter.alfresco-access.transaction.user=~System;~null;.*
audit.filter.alfresco-access.transaction.type=cm:folder;cm:content;st:site
audit.filter.alfresco-access.transaction.path=~/sys:archivedItem;~/ver:;.*

#The default to preserve all cm:auditable data on a node when the process is not directly driven by a user action                                                                                        
system.auditableData.preserve=${system.preserve.modificationData}

#Specific control of how the FileFolderService treats cm:auditable data when performing moves                                                                                                            
system.auditableData.FileFolderService=${system.auditableData.preserve}

#Specific control of whether ACL changes on a node trigger the cm:auditable aspect                                                                                                                       
system.auditableData.ACLs=${system.auditableData.preserve}

Como de costume, você deve salvar as alterações no arquivo alfresco-global.properties e reiniciar o servidor Alfresco, para habilitar essas modificações.

Filtros de auditoria

Filtros de auditoria são propriedades que especificam a estratégia usada para filtrar eventos de auditoria usando uma expressão regular específica para incluir ou excluir eventos. Os filtros de auditoria padrão e personalizado podem ser adicionados como substituições no arquivo de configuração alfresco-global.properties.

A anatomia de uma propriedade de filtro de auditoria é a seguinte:

audit.filter.<data_producer>.<path>

onde <data-producer> é um dos produtores de dados internos do Alfresco:

  1. alfresco-access: um amplo grupo de eventos de alto nível, como logins (tanto bem-sucedidos quanto com falha), atualizações de propriedades, CRUD em nós, leituras/atualizações de conteúdo, adição e remoção de aspectos, controle de versão, check-in/check- fora de operações
  2. nó ao ar livre
  3. alfresco-api: eventos emitidos pela chamada de métodos e serviços de API de baixo nível. Por exemplo, pode ser usado para listar os parâmetros da lista de pesquisa SearchServices, listagem de propriedades usando PropertyServices, operações em nós usando NodeServices e assim por diante.

e path é o valor do caminho real para filtrar.

Os nomes de propriedade têm um prefixo audit.filter.* e usam ‘.’ como um separador onde, como componentes de rootPath e chaves no mapa de auditoria, use ‘/’.

As listas são avaliadas da esquerda para a direita e se nenhuma correspondência for feita até o final da lista, o valor será rejeitado. Se não houver uma propriedade para um determinado valor ou uma lista vazia for definida, qualquer valor será aceito.

Cada expressão regular na lista é separada por um ponto e vírgula (’;’). Expressões que incluem um ponto e vírgula podem ser escapadas usando um ‘'.

Observe que, se o sinalizador audit.config.strict estiver definido como verdadeiro, a inicialização ao ar livre falhará em caso de detecção de configurações de auditoria inválidas.

Uma expressão que começa com um '~' indica que qualquer valor correspondente deve ser rejeitado. Se o primeiro caractere de uma expressão precisa ser um '~', ele pode ser escapado com um '\'.

Adicionar .* no final de um filtro incluirá todos os valores que não foram especificamente excluídos

Os filtros podem ser um dos seguintes:

transaction.user - especifica quais ações de usuário(s) serão/não serão auditadas. Por exemplo: As ações de todos os usuários, exceto ‘Sistema’, serão auditadas

transaction.type - as ações executadas em relação ao tipo de documento especificado serão auditadas.

default.path - ações que ocorrem em documentos dentro do caminho especificado serão auditadas

transaction.action - especifica quais ações serão e não serão auditadas. Alguns dos eventos de auditoria que podem ser habilitados ou desabilitados usando esta propriedade são: READ, MOVE, COPY, CHECK IN, CHECK OUT, CANCEL CHECK OUT, CREATE VERSION, readContent, addNodeAspect, deleteNodeAspect, updateNodeProperties.

Para obter mais informações sobre filtros de auditoria:

https://github.com/tsgrp/OpenContent/wiki/Alfresco-Audit-Configuration

http://docs.alfresco.com/5.1/concepts/audit-example-filter.html
