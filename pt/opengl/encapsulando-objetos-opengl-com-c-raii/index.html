Encapsulando objetos OpenGL com C++ RAII

  • Em C++11 e posterior
  • Em C++98/03

Nesta página

  • Em C++11 e posterior
  • Em C++98/03

Exemplos de várias maneiras de fazer com que objetos OpenGL funcionem com C++ RAII.

O encapsulamento RAII de objetos OpenGL tem perigos. O mais inevitável é que os objetos OpenGL estão associados ao contexto OpenGL que os criou. Portanto, a destruição de um objeto C++ RAII deve ser feita em um contexto OpenGL que compartilha a propriedade do objeto OpenGL gerenciado por esse objeto C++.

Isso também significa que se todos os contextos que possuem o objeto forem destruídos, então quaisquer objetos OpenGL encapsulados em RAII existentes tentarão destruir objetos que não existem mais.

Você deve seguir etapas manuais para lidar com problemas de contexto como esse.

Em C++11 e posterior #

O C++11 oferece ferramentas que aprimoram a funcionalidade de objetos OpenGL encapsulados em RAII. Sem recursos do C++ 11, como semântica de movimentação, esses objetos teriam que ser alocados dinamicamente se você quiser passá-los, pois eles não podem ser copiados. O suporte a movimento permite que eles sejam passados ​​para frente e para trás como valores normais, embora não copiando:

class BufferObject
{
public:
    BufferObject(GLenum target, GLsizeiptr size, const void *data, GLenum usage)
    {
        glGenBuffers(1, &object_);
        glBindBuffer(target, object_);
        glBufferData(target, size, data, usage);
        glBindBuffer(target, 0);
    }
    
    //Cannot be copied.
    BufferObject(const BufferObject &) = delete;
    BufferObject &operator=(const BufferObject &) = delete;
    
    //Can be moved
    BufferObject(BufferObject &&other) noexcept : object_(other.Release())
    {}
    
    //Self-assignment is OK with this implementation.
    BufferObject &operator=(BufferObject &&other) noexcept
    {
        Reset(other.Release());
    }
    
    //Destroys the old buffer and claims ownership of a new buffer object.
    //It's OK to call glDeleteBuffers on buffer object 0.
    GLuint Reset(GLuint object = 0)
    {
        glDeleteBuffers(1, &object_);
        object_ = object;
    }
    
    //Relinquishes ownership of the object without destroying it
    GLuint Release()
    {
        GLuint ret = object_;
        object_ = 0;
        return ret;
    }    
    
    ~BufferObject()
    {
        Reset();
    }
    
    //Accessors and manipulators
    void Bind(GLenum target) const {glBindBuffer(target, object_);}
    GLuint GetObject() const {return object_;}

private:
    GLuint object_;
};

Esse tipo pode ser retornado por uma função:

BufferObject CreateStaticBuffer(GLsizeiptr byteSize) {return BufferObject(GL_ARRAY_BUFFER, byteSize, nullptr, GL_STATIC_DRAW);}

O que permite armazená-los em seus próprios tipos (implicitamente apenas para movimentação):

struct Mesh
{
public:
private:
    //Default member initializer.
    BufferObject buff_ = CreateStaticBuffer(someSize);
};

Uma classe de fichário com escopo também pode ter semântica de movimento, permitindo assim que o fichário seja retornado de funções e armazenado em contêineres de biblioteca padrão C++:

class BindBuffer
{
public:
    BindBuffer(GLenum target, const BufferObject &buff) : target_(target)
    {
        buff.Bind(target_);
    }

    //Non-copyable.
    BindBuffer(const BindBuffer &) = delete;
    BindBuffer &operator=(const BindBuffer &) = delete;
    
    //Move-constructible.
    BindBuffer(BindBuffer &&other) noexcept : target_(other.target_)
    {
        other.target_ = 0;
    }
    
    //Not move-assignable.
    BindBuffer &operator=(BindBuffer &&) = delete;
    
    ~BindBuffer()
    {
        //Only unbind if not moved from.
        if(target_)
            glBindBuffer(target_, 0);
    }
    
private:
    GLenum target_;
};

Observe que o objeto é construtível por movimento, mas não pode ser atribuído por movimento. A ideia com isso é evitar a religação de uma ligação de buffer com escopo definido. Uma vez que é definido, a única coisa que pode desarmá-lo está sendo movido.

Em C++98/03 #

Encapsular um objeto OpenGL em C++98/03 requer obedecer à regra C++ de 3. Isso significa adicionar um construtor de cópia, um operador de atribuição de cópia e um destruidor.

No entanto, os construtores de cópia devem copiar logicamente o objeto. E copiar um objeto OpenGL é uma tarefa não trivial. Igualmente importante, é quase certamente algo que o usuário não deseja fazer.

Então, em vez disso, tornaremos o objeto não copiável:

class BufferObject
{
public:
    BufferObject(GLenum target, GLsizeiptr size, const void *data, GLenum usage)
    {
        glGenBuffers(1, &object_);
        glBindBuffer(target, object_);
        glBufferData(target, size, data, usage);
        glBindBuffer(target, 0);
    }
    
    ~BufferObject()
    {
        glDeleteBuffers(1, &object_);
    }
    
    //Accessors and manipulators
    void Bind(GLenum target) const {glBindBuffer(target, object_);}
    GLuint GetObject() const {return object_;}

private:
    GLuint object_;
    
    //Prototypes, but no implementation.
    BufferObject(const BufferObject &);
    BufferObject &operator=(const BufferObject &);
};

O construtor criará o objeto e inicializará os dados do objeto buffer. O destruidor destruirá o objeto. Ao declarar o construtor/atribuição de cópia sem defini-los, o vinculador dará um erro se algum código tentar chamá-los. E ao declará-los privados, apenas os membros do BufferObject poderão chamá-los.

Observe que BufferObject não retém o target passado para o construtor. Isso porque um objeto de buffer OpenGL pode ser usado com qualquer destino, não apenas aquele com o qual foi criado inicialmente. Isso é diferente de objetos de textura, que devem sempre estar vinculados ao destino com o qual foram criados inicialmente.

Como o OpenGL é muito dependente da vinculação de objetos ao contexto para vários propósitos, geralmente é útil ter também a vinculação de objetos com escopo no estilo RAII. Como objetos diferentes têm necessidades de vinculação diferentes (alguns têm alvos, outros não), temos que implementar um para cada objeto individualmente.

class BindBuffer
{
public:
    BindBuffer(GLenum target, const BufferObject &buff) : target_(target)
    {
        buff.Bind(target_);
    }
    
    ~BindBuffer()
    {
        glBindBuffer(target_, 0);
    }
    
private:
    GLenum target_;

    //Also non-copyable.
    BindBuffer(const BindBuffer &);
    BindBuffer &operator=(const BindBuffer &);
};

BindBuffer não é copiável, pois copiá-lo não faz sentido. Observe que ele não retém o acesso ao BufferObject que vincula. Isso porque é desnecessário.
