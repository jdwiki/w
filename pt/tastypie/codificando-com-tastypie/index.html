Tastypie é um framework de API de webservice para Django. Ele fornece uma abstração conveniente, mas poderosa e altamente personalizável para criar interfaces no estilo REST. O Tastypie facilita a exposição de seus modelos, mas oferece controle total sobre o que você expõe, permitindo que você abstraia o banco de dados o quanto for necessário. O Tastypie também facilita a integração com fontes de dados não ORM. Portanto, o Tastypie pode ser usado com bancos de dados ORM e não ORM.

Começo rápido #

  1. Adicione saborosa torta a INSTALLED_APPS.

  2. Crie um diretório api em seu aplicativo com um init.py simples.

  3. Crie um arquivo <my_app>/api/resources.py e coloque o seguinte nele:

    from tastypie.resources import ModelResource from my_app.models import MyModel

    class MyModelResource(ModelResource): class Meta: queryset = MyModel.objects.all() allowed_methods = [‘get’]

Em seu URLconf raiz, adicione o seguinte código (em torno de onde o código de administrador pode estar):

from django.conf.urls import url, include
from tastypie.api import Api
from my_app.api.resources import MyModelResource

v1_api = Api(api_name='v1')
v1_api.register(MyModelResource())

urlpatterns = [
  # ...more URLconf bits here...
  # Then add:
  url(r'^api/', include(v1_api.urls)),
]

Instalação do Tastypie #

O Tastypie pode ser instalado com gerenciamento de pacotes python, ou seja, pip ou podemos verificar diretamente o código do Github

  1. pip instala o django-tastypie
  2. Finalizar compra no Github

Por que Tastypie? #

Existem outros frameworks de API para Django. Você precisa avaliar as opções disponíveis e decidir por si mesmo. Dito isto, aqui estão algumas razões comuns para a saborosa torta.

  • Você precisa de uma API que seja RESTful e use HTTP bem.
  • Você quer apoiar relações profundas.
  • Você NÃO quer ter que escrever seu próprio serializador para fazer o direito de saída.
  • Você quer um framework de API que tenha pouca mágica, muito flexível e mapeia bem para o domínio do problema.
  • Você quer/precisa de serialização XML que é tratada igualmente para JSON (e YAML está lá também).

Para desenvolver API com Django, temos muitas opções. As principais opções são Tastypie e Django Rest Framework(DRF).

O que faz um framework de API decente?

Estas características:

  1. paginação
  2. lançamento de dados com validação
  3. Publicação de metadados junto com conjuntos de consultas
  4. Descoberta de API
  5. tratamento adequado da resposta HTTP
  6. cache
  7. serialização
  8. estrangulamento
  9. permissões
  10. autenticação

As estruturas de API adequadas também precisam de:

  1. Cobertura de teste muito boa de seu código

  2. Desempenho decente

  3. Documentação

  4. Uma comunidade ativa para avançar e apoiar a estrutura

Se você considerar esses fatores, neste momento existem apenas duas estruturas de API que valem a pena usar (depende das visualizações do usuário), django-tastypie e django-rest-framework.

Qual é o melhor? django-tastypie ou django-rest-framework?

Digo que são iguais. Você simplesmente não pode dar errado com qualquer um. Os autores e as comunidades por trás de ambos são ativos, o código é sólido e testado. E aqui estão meus pensamentos específicos sobre ambos:

Gostoso:

Vantagens:

  • Fácil de começar e fornecer funcionalidades básicas OOB (out da caixa)
  • Na maioria das vezes você não estará lidando com conceitos avançados do Django como CBVs, Formulários etc
  • Código mais legível e menos magia!
  • Se seus modelos são NÃO-ORM, vá em frente.

Desvantagens:

  • Não segue estritamente o Django idiomático (lembre-se bem de python e as filosofias do django são bem diferentes)
  • Provavelmente um pouco difícil de personalizar APIs quando você for grande
  • Sem O-Auth

DRF:

Vantagens:

  • Siga django idiomático. (Se você conhece o django de dentro para fora, e muito confortável com CBV, Forms etc, sem dúvida vá em frente)

  • Fornece funcionalidade REST pronta para uso usando ModelViewSets. No ao mesmo tempo, proporciona maior controle para customização usando CustomSerializer, APIView, GenericViews etc.

  • Melhor autenticação.

  • Mais fácil de escrever classes de permissão personalizadas.

  • Funciona muito bem e, principalmente, muito fácil de fazer funcionar com o 3º bibliotecas de terceiros e OAuth.

  • DJANGO-REST-AUTH vale a pena mencionar LIBRARY para Autenticação/Autenticação Social/Registro.

Desvantagens:

  • Se você não conhece muito bem o Django, não faça isso.
  • Magia! Algum tempo muito difícil de entender a magia.
  • Porque foi escrito em cima do CBV do django que por sua vez é bastante complexo por natureza.
  • Tem curva de aprendizado íngreme.

Introdução ao Tastypie #

O Tastypie é um aplicativo reutilizável (ou seja, depende apenas de seu próprio código e se concentra em fornecer apenas uma API estilo REST) ​​e é adequado para fornecer uma API para qualquer aplicativo sem precisar modificar as fontes desse aplicativo.

Nem todas as necessidades são as mesmas, então o Tastypie se esforça para fornecer muitos ganchos para substituir ou estender como funciona.

Para fins de exemplo, adicionaremos uma API a um blog simples aplicação.

A única configuração obrigatória é adicionar ‘tastypie’ ao seu INSTALLED_APPS. Isso não é estritamente necessário, pois o Tastypie tem apenas dois modelos não obrigatórios, mas pode facilitar o uso.

Aqui está myapp/models.py:

from tastypie.utils.timezone import now
from django.contrib.auth.models import User
from django.db import models
from django.utils.text import slugify


class Entry(models.Model):
    user = models.ForeignKey(User)
    pub_date = models.DateTimeField(default=now)
    title = models.CharField(max_length=200)
    slug = models.SlugField(null=True, blank=True)
    body = models.TextField()

Criando recursos

A arquitetura no estilo REST fala sobre recursos, então, sem surpresa, a integração com o Tastypie envolve a criação de classes de recursos. Para nosso aplicativo simples, criaremos um arquivo para eles em myapp/api.py, embora possam residir em qualquer lugar em seu aplicativo:

meuaplicativo/api.py #

from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'

No REST tudo é um recurso, significa que os objetos são descritos como recursos. Portanto, para criar uma API REST Tastypie envolve a criação de uma classe de recurso. Para criar um recurso, precisamos subclassificar a classe ModelResource. Essa classe EntryResource verificará todos os campos não relacionais no modelo Entry e criará seus próprios campos de API. Isso funciona exatamente como o ModelForm do Django Forms.

Conectando o recurso: Após criar o Resource, precisamos informar ao Django que um recurso é criado conectando nosso EntryResource à URL.

#urls.py

from django.conf.urls import url, include
from myapp.api import EntryResource

entry_resource = EntryResource()

urlpatterns = [
    url(r'^blog/', include('myapp.urls')),
    url(r'^api/', include(entry_resource.urls)),
]

Agora, se verificarmos localhost:8000/api/entry/, obteremos a resposta da API.

(O resource_name dentro da classe Meta é opcional. Se não for fornecido, ele será gerado automaticamente a partir do nome da classe, removendo quaisquer instâncias de Resource e colocando a string em minúsculas. Portanto, EntryResource se tornaria apenas entry. É por isso que no URL especificamos ../entry/ )

Autorização Tastypie #

Até agora tentamos obter request. (Se você precisar tentar outra solicitação HTTP, use algumas ferramentas de teste de API, como curl ou carteiro)

Se você tentar enviar um POST/PUT/DELETE para o recurso, você receberá erros “401 Unauthorized”. Por segurança, o Tastypie é fornecido com a classe de autorização definida como ReadOnlyAuthorization. Isso torna seguro expor na web, mas nos impede de fazer POST/PUT/DELETE. Vamos habilitá-los.

meuaplicativo/api.py #

from tastypie.authorization import Authorization
from tastypie.resources import ModelResource
from myapp.models import Entry


class EntryResource(ModelResource):
    class Meta:
        queryset = Entry.objects.all()
        resource_name = 'entry'
        authorization = Authorization()
