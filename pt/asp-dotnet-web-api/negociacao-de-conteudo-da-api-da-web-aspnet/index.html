Informações básicas de negociação de conteúdo da API Web ASP.NET #

A Negociação de Conteúdo pode ser definida como o processo de seleção da melhor representação para um determinado recurso. Portanto, a negociação de conteúdo significa que o cliente e o servidor podem negociar entre eles para que o cliente possa obter dados de acordo com o formato necessário.

Há três pontos dos quais a internet depende,

  • O Recurso
  • Um ponteiro para recurso (URL)
  • Representação do recurso

O terceiro ponto é mais importante que os outros dois, pois tudo funciona com base em como podemos ver o recurso. Podemos representar um recurso em dois formatos.

  1. Formato XML (Extensible Markup Language)
  2. Formato JSON (JavaScript Object Notation)

Um dos padrões do serviço RESTful é que o cliente deve ter a capacidade de decidir em qual formato deseja a resposta seja em JSON ou XML. Uma solicitação enviada ao servidor inclui um cabeçalho Accept. Usando o cabeçalho Accept, o cliente pode especificar o formato da resposta.

Por exemplo,

Accept: application/xml retorna o resultado no formato XML

Accept: application/json retorna o resultado no formato JSON

Dependendo do valor do cabeçalho Accept na solicitação, o servidor envia a resposta. Isso é chamado de Negociação de Conteúdo.

O que acontece nos bastidores quando solicitamos dados em um formato específico?

O controlador ASP.NET Web API gera os dados que queremos enviar para o cliente e entrega os dados ao pipeline da Web API que, em seguida, procura o cabeçalho Accept do cliente. Em seguida, escolha um formatador apropriado para formatar os dados.

Como a API da Web do ASP.NET é bastante extensível, também podemos especificar vários valores para aceitar o cabeçalho no cabeçalho da solicitação.

Aceitar: application/xml,application/json

No caso acima, o servidor escolhe o primeiro formatador para formatar os dados de resposta.

Também podemos especificar o fator de qualidade no cabeçalho de aceitação. Neste caso, o servidor escolhe um formato que tenha maior fator de qualidade.

Aceitar: application/json;q=0.8,application/xml;q=0.5

Se não especificarmos nenhum cabeçalho Accept, por padrão o servidor escolherá o formatador JSON.

Quando a resposta estiver sendo enviada ao cliente no formato solicitado, observe que o cabeçalho Content-Type da resposta está definido com o valor apropriado. Por exemplo, se o cliente solicitou application/xml, o servidor envia os dados no formato XML e também define o Content-Type=application/xml.

Os formatadores são usados ​​pelo servidor para mensagens de solicitação e resposta. Quando o cliente envia uma solicitação ao servidor, definimos o cabeçalho Content-Type com o valor apropriado para informar ao servidor o formato dos dados que estamos enviando.

Por exemplo, se o cliente estiver enviando dados JSON, o cabeçalho Content-Type será definido como application/json. O servidor sabe que está lidando com dados JSON, então ele usa o formatador JSON para converter dados JSON em tipo .NET. Da mesma forma, quando uma resposta está sendo enviada do servidor para o cliente, dependendo do valor do cabeçalho Accept, o formatador apropriado é usado para converter o tipo .NET em JSON, XML etc.

Exemplo de diferentes tipos de formato de resposta:

aplicativo/json:

{
  "Email": "sample string 1",
  "HasRegistered": true,
  "LoginProvider": "sample string 3"
}

aplicativo/xml:

<UserInfoViewModel xmlns:i="http://www.w3.org/2001/XMLSchema-instance" xmlns="http://schemas.datacontract.org/2004/07/WebApiDemo.Models">
  <Email>sample string 1</Email>
  <HasRegistered>true</HasRegistered>
  <LoginProvider>sample string 3</LoginProvider>
</UserInfoViewModel>

Os aplicativos modernos baseados na web podem fornecer dados em vários idiomas e formatos. Portanto, se desenvolvermos nossa API para cobrir usuários globais em todo o mundo, a Negociação de Conteúdo será relevante.

Negociação de conteúdo na API da Web #

Entendendo o conceito #

Para entender a negociação de conteúdo na API da Web, é importante entender o termo Recurso.

Na web, qualquer informação que possamos acessar pode ser chamada de recurso HTTP. Há uma quantidade enorme de material para visualizar na web que possui diferentes tipos de conteúdo, como documentos html, imagens, vídeo, áudio, arquivos executáveis, planilhas etc. Podemos obter qualquer recurso fazendo uma solicitação http ao recurso uri. A resposta http para a solicitação retorna o recurso e também especifica o tipo de conteúdo, que é também conhecido como tipo de mídia.

Para acessar qualquer recurso, o cliente pode fazer uma solicitação http fornecendo o recurso específico uri e os verbos http. No entanto, além disso, o cliente também pode especificar o tipo de aceitação que é o formato do conteúdo que o usuário está procurando. O “accept-type” pode ser definido nos cabeçalhos da solicitação http como o cabeçalho “accept”.

O servidor então verifica o cabeçalho “aceitar” das solicitações e retorna a resposta no formato especificado, se disponível. Observe que o servidor só pode retornar a resposta na representação solicitada se estiver disponível. Se a representação solicitada não estiver disponível, ele retornará o recurso na representação padrão. Essa é a razão pela qual é chamado de negociação de conteúdo.

Um exemplo prático #

Por exemplo, suponha que você esteja fazendo uma solicitação para http://example.com/customer/1 para obter as informações do cliente com o id 1. Se você não especificar o cabeçalho “aceitar” na solicitação, o server retornará a representação padrão deste recurso.

Suponha que o servidor possa retornar as informações do cliente em json e xml ambos. Agora, cabe ao cliente especificar o formato necessário das informações do cliente no cabeçalho “aceitar” na solicitação. O valor do cabeçalho “accept” pode ser “application/json” para representação json, ou “text/xml” para representação xml. O servidor retornará a resposta conforme o formato solicitado.

Se o formato solicitado for “text/html” que não é suportado por este host (como neste exemplo), ele simplesmente retornará o recurso no formato padrão. A resposta http contém um cabeçalho “content-type” que informa ao cliente sobre o formato do recurso.

Observe que, mesmo no caso em que a representação solicitada do recurso não estiver disponível, a representação padrão do recurso ainda será retornada.

É por isso que é referido como negociação de conteúdo.

O cliente negocia a representação da resposta, porém, se não está disponível, então obtém o padrão.

Como configurar na API da Web #

Na API da Web, a negociação de conteúdo pode ser configurada na classe WebAPIConfig como

config.Formatters.JsonFormatter.SupportedMediaTypes.Add(new System.Net.Http.Headers.MediaTypeHeaderValue("text/html"));

Você também pode substituir a negociação de conteúdo padrão na API da Web implementando a interface IContentNegotiator e seu método Negotiate e, em seguida, configurar isso no pipeline de solicitação da API da Web, no arquivo WebAPI.config conforme abaixo:

GlobalConfiguration.Configuration.Services.Replace(typeof(IContentNegotiator), new CustomContentNegotiator());

A seguir está um exemplo de implementação do método Negotiate.

public class CustomContentNegotiator : DefaultContentNegotiator
    {
        public override ContentNegotiationResult Negotiate(Type type, HttpRequestMessage request, IEnumerable<MediaTypeFormatter> formatters)
        {
            var result = new ContentNegotiationResult(new JsonMediaTypeFormatter(), new MediaTypeHeaderValue("application/json"));
            return result;
        }
