Ativar streaming do frasco #

O Flask tem esse recurso que permite transmitir dados de uma visualização usando geradores.

Vamos alterar o arquivo app.py

  • adicione from flask import Response

  • adicione from datetime import datetime

  • adicione from time import sleep

  • crie uma nova visualização:

    @app.route("/time/") def time(): def streamer(): while True: yield “

    {}

    ".format(datetime.now()) sleep(1)

      return Response(streamer())

Agora abra seu navegador em localhost/time/. O site será carregado para sempre porque o nginx espera até que a resposta seja concluída. Nesse caso, a resposta nunca será completa porque enviará a data e hora atuais para sempre.

Para evitar que o nginx espere, precisamos adicionar uma nova linha à configuração.

Edite /etc/nginx/sites-available/flaskconfig

server {
    listen 80;
    server_name localhost;

    location / {
        include uwsgi_params;
        uwsgi_pass unix:///tmp/flask.sock;
        uwsgi_buffering off;  # <-- this line is new
    }
}

A linha uwsgi_buffering off; diz ao nginx para não esperar até que uma resposta seja concluída.

Reinicie o nginx: sudo service nginx restart e veja localhost/time/ novamente.

Agora você verá que a cada segundo uma nova linha aparece.

Configure o Flask Application, uWGSI, Nginx - modelo de caldeira de configurações do servidor (padrão, proxy e cache) #

Esta é uma portabilidade da configuração originada do tutorial da DigitalOcean de How To Serve Flask Applications with uWSGI and Nginx on Ubuntu 14.04

e alguns recursos git úteis para servidores nginx.

Aplicação de Frasco

Este tutorial pressupõe que você use o Ubuntu.

  1. localize a pasta var/www/.
  2. Crie a pasta do seu aplicativo web mkdir myexample
  3. cd meu exemplo

opcional Você pode querer configurar o ambiente virtual para implantar aplicativos da web no servidor de produção.

sudo pip install virtualenv

para instalar o ambiente virtual.

virtualenv myexample

para configurar o ambiente virtual para seu aplicativo.

source myprojectenv/bin/activate

para ativar seu ambiente. Aqui você instalará todos os pacotes python.

final opcional, mas recomendado

Configurar frasco e gateway uWSGI

Instale o frasco e o gateway uSWGI:

pip install uwsgi flask

Exemplo de aplicativo de frasco em myexample.py:

from flask import Flask
application = Flask(__name__)

@application.route("/")
def hello():
    return "<h1>Hello World</h1>"

if __name__ == "__main__":
    application.run(host='0.0.0.0')

Crie um arquivo para se comunicar entre seu aplicativo da web e o servidor da web: interface de gateway [https://en.wikipedia.org/wiki/Web_Server_Gateway_Interface]

nano wsgi.py

em seguida, importe seu módulo webapp e execute-o a partir do ponto de entrada do gateway.

from myexample import application

if __name__ == "__main__":
    application.run()

Para testar o uWSGI:

uwsgi --socket 0.0.0.0:8000 --protocol=http -w wsgi

Para configurar o uWSGI:

  1. Crie um arquivo de configuração .ini

    nano myexample.ini

  2. Configuração básica para gateway uWSGI

    include header for using uwsgi #

    [uwsgi]

    point it to your python module wsgi.py #

    module = wsgi

    tell uWSGI to start a master node to serve requests #

    master = true

    spawn number of processes handling requests #

    processes = 5

    use a Unix socket to communicate with Nginx. Nginx will pass connections to uWSGI through a socket, instead of using ports. This is preferable because Nginx and uWSGI stays on the same machine. #

    socket = myexample.sock

    ensure file permission on socket to be readable and writable #

    chmod-socket = 660

    clean the socket when processes stop #

    vacuum = true

    use die-on-term to communicate with Ubuntu versions using Upstart initialisations: see: #

    http://uwsgi-docs.readthedocs.io/en/latest/Upstart.html?highlight=die%20on%20term #

    die-on-term = true

opcional se você estiver usando env virtual Você pode desativar seu ambiente virtual.

Configuração do Nginx Vamos usar o nginx como:

  1. servidor padrão para passar a solicitação ao soquete, usando o protocolo uwsgi
  2. servidor proxy na frente do servidor padrão
  3. servidor de cache para armazenar em cache solicitações bem-sucedidas (por exemplo, você pode querer armazenar em cache solicitações GET se seu aplicativo da web)

Localize seu diretório sites-available e crie um arquivo de configuração para seu aplicativo:

sudo nano /etc/nginx/sites-available/myexample

Adicione o seguinte bloco, nos comentários o que ele faz:

server {
   

    # setting up default server listening to port 80
    listen 8000 default_server;
    server_name myexample.com; #you can also use your IP 
    
    # specify charset encoding, optional
    charset utf-8;

    # specify root of your folder directory
    root /var/www/myexample;

    # specify locations for your web apps.
    # here using /api endpoint as example
    location /api {
        # include parameters of wsgi.py and pass them to socket
        include uwsgi_params;
        uwsgi_pass unix:/var/www/myexample/myexample.sock;
    }

}

# Here you will specify caching zones that will be used by your virtual server
# Cache will be stored in /tmp/nginx folder
# ensure nginx have permissions to write and read there!
# See also:
# http://nginx.org/en/docs/http/ngx_http_proxy_module.html

proxy_cache_path /tmp/nginx levels=1:2 keys_zone=my_zone:10m inactive=60m;
proxy_cache_key "$scheme$request_method$host$request_uri";

# set up the virtual host!
server {
    listen   80  default_server;
    
    # Now www.example.com will listen to port 80 and pass request to http://example.com
    server_name www.example.com;

    # Why not caching responses

    location /api {
        # set up headers for caching
        add_header X-Proxy-Cache $upstream_cache_status;

        # use zone specified above
        proxy_cache my_zone;
        proxy_cache_use_stale updating;
        proxy_cache_lock on;
        
        # cache all responses ?
        # proxy_cache_valid 30d;

        # better cache only 200 responses :)
        proxy_cache_valid 200 30d;

        # ignore headers to make cache expire
        proxy_ignore_headers X-Accel-Expires Expires Cache-Control;

        # pass requests to default server on port 8000
        proxy_pass http://example.com:8000/api;
    }
}

Finalmente, vincule o arquivo ao diretório sites-enabled. Para obter uma explicação dos sites disponíveis e ativados, consulte a resposta: [http://serverfault.com/a/527644]

sudo ln -s /etc/nginx/sites-available/myproject /etc/nginx/sites-enabled

Você terminou agora com o nginx. No entanto, você pode querer verificar este modelo de caldeira muito precioso: [https://github.com/h5bp/server-configs-nginx]

Muito útil para ajustes finos.

Agora teste o Nginx:

sudo nginx -t

Inicie o Nginx:

sudo service nginx restart

Automatize o Ubuntu para iniciar o uWSGI A última coisa é fazer com que o Ubuntu inicie o gateway wsgi se comunicando com seu aplicativo, caso contrário, você deve fazê-lo manualmente.

  1. Localize o diretório para scripts de inicialização no Ubuntu e crie um novo script:

sudo nano /etc/init/myexample.conf

  1. Adicione o bloco a seguir, comentários na linha para explicar o que ele faz

    # description for the purpose of this script
description "uWSGI server instance configured to serve myproject"

# Tell to start on system runtime 2, 3, 4, 5. Stop at any other level (0,1,6). 
# Linux run levels: [http://www.debianadmin.com/debian-and-ubuntu-linux-run-levels.html]
start on runlevel [2345]
stop on runlevel [!2345]

# Set up permissions! "User" will be the username of your user account on ubuntu.
setuid user
# Allow www-data group to read and write from the socket file. 
# www-data is normally the group Nginx and your web applications belong to.
# you may have all web application projects under /var/www/ that belongs to www-data group
setgid www-data

# tell Ubunutu which environment to use.
# This is the path of your virtual environment: python will be in this path if you installed virtualenv. Otherwise, use path of your python installation
env PATH=/var/www/myexample/myexample/bin
# then tell to Ubuntu to change and locate your web application directory
chdir /var/www/myexample
# finally execute initialisation script, that load your web app myexample.py
exec uwsgi --ini myexample.ini

Agora você pode ativar seu script: sudo start myexample

Usando uWSGI para executar um aplicativo de frasco #

O servidor werkzeug integrado certamente não é adequado para executar servidores de produção. A razão mais óbvia é o fato de que o servidor werkzeug é single-thread e, portanto, só pode lidar com uma solicitação por vez.

Por isso, queremos usar o servidor uWSGI para servir nosso aplicativo. Neste exemplo, instalaremos o uWSGI e executaremos um aplicativo de teste simples com ele.

Instalando o uWSGI:

pip install uwsgi

É tão simples quanto isso. Se você não tiver certeza sobre a versão do python que seu pip usa, torne-o explícito:

python3 -m pip install uwsgi  # for python3
python2 -m pip install uwsgi  # for python2

Agora vamos criar um aplicativo de teste simples:

app.py

from flask import Flask
from sys import version

app = Flask(__name__)

@app.route("/")
def index():
    return "Hello uWSGI from python version: <br>" + version

application = app

No frasco o nome convencional para o aplicativo é app mas o uWSGI procura por application por padrão. É por isso que criamos um alias para nosso aplicativo na última linha.

Agora é hora de executar o aplicativo:

uwsgi --wsgi-file app.py --http :5000

Você deve ver a mensagem “Hello uWSGI …” apontando seu navegador para localhost:5000

Para não digitar o comando completo toda vez, criaremos um arquivo uwsgi.ini para armazenar essa configuração:

uwsgi.ini

[uwsgi]
http = :9090
wsgi-file = app.py
single-interpreter = true
enable-threads = true
master = true

As opções http e wsgi-file são as mesmas do comando manual. Mas há mais três opções:

  • single-interpreter: Recomenda-se ativar isso porque pode interferir na próxima opção

  • enable-threads: Isso precisa ser ativado se você estiver usando threads adicionais em seu aplicativo. Nós não os usamos agora, mas agora não precisamos nos preocupar com isso.

  • master: o modo mestre deve ser ativado por vários motivos

Agora podemos executar o aplicativo com este comando:

uwsgi --ini uwsgi.ini

Instalando o nginx e configurando-o para uWSGI #

Agora queremos instalar o nginx para servir nosso aplicativo.

sudo apt-get install nginx  # on debian/ubuntu

Em seguida, criamos uma configuração para o nosso site

cd /etc/nginx/site-available  # go to the configuration for available sites
# create a file flaskconfig with your favourite editor

flaskconfig

server {
    listen 80;
    server_name localhost;

    location / {
        include uwsgi_params;
        uwsgi_pass unix:///tmp/flask.sock;
    }
}

Isso diz ao nginx para escutar na porta 80 (padrão para http) e servir algo no caminho raiz (/). Lá nós dizemos ao nginx para simplesmente agir como um proxy e passar todas as requisições para um socket chamado flask.sock localizado em /tmp/.

Vamos habilitar o site:

cd /etc/nginx/sites-enabled
sudo ln -s ../sites-available/flaskconfig .

Você pode querer remover a configuração padrão se estiver habilitada:

# inside /etc/sites-enabled
sudo rm default

Em seguida, reinicie o nginx:

sudo service nginx restart

Aponte seu navegador para localhost e você verá um erro: 502 Bad Gateway.

Isso significa que o nginx está ativo e funcionando, mas o soquete está ausente. Então vamos criar isso.

Volte ao seu arquivo uwsgi.ini e abra-o. Em seguida, acrescente estas linhas:

socket = /tmp/flask.sock
chmod-socket = 666

A primeira linha diz ao uwsgi para criar um soquete no local fornecido. O soquete será usado para receber solicitações e enviar de volta as respostas. Na última linha, permitimos que outros usuários (incluindo o nginx) possam ler e escrever nesse soquete.

Inicie o uwsgi novamente com uwsgi --ini uwsgi.ini. Agora aponte seu navegador novamente para localhost e você verá a saudação “Hello uWSGI” novamente.

Observe que você ainda pode ver a resposta em localhost:5000 porque o uWSGI agora serve o aplicativo via http e o soquete. Então vamos desabilitar a opção http no arquivo ini

http = :5000  # <-- remove this line and restart uwsgi

Agora, o aplicativo só pode ser acessado a partir do nginx (ou lendo esse soquete diretamente :) ).
