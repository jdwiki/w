Capturar a exceção esperada #

É possível capturar facilmente a exceção sem nenhum bloco try catch.

public class ListTest {
  private final List<Object> list = new ArrayList<>();

  @Test(expected = IndexOutOfBoundsException.class)
  public void testIndexOutOfBoundsException() {
    list.get(0);
  }
}

O exemplo acima deve ser suficiente para casos mais simples, quando você não quer/precisa verificar a mensagem transportada pela exceção lançada.

Se você quiser verificar as informações sobre a exceção, você pode usar o bloco try/catch:

@Test
public void testIndexOutOfBoundsException() {
    try {
        list.get(0);
        Assert.fail("Should throw IndexOutOfBoundException");
    } catch (IndexOutOfBoundsException ex) {
        Assert.assertEquals("Index: 0, Size: 0", ex.getMessage());
    }
}

Para este exemplo, você deve estar ciente de sempre adicionar Assert.fail() para garantir que o teste falhará quando nenhuma exceção for lançada.

Para casos mais elaborados, JUnit possui a ExpectedException @Rule, que também pode testar esta informação e é utilizada da seguinte forma:

public class SimpleExpectedExceptionTest {
     @Rule
     public ExpectedException expectedException = ExpectedException.none();

     @Test
     public void throwsNothing() {
         // no exception expected, none thrown: passes.
     }

     @Test
     public void throwsExceptionWithSpecificType() {
         expectedException.expect(NullPointerException.class);

         throw new NullPointerException();
     }

     @Test
     public void throwsExceptionWithSpecificTypeAndMessage() {
         expectedException.expect(IllegalArgumentException.class);
         expectedException.expectMessage("Wanted a donut.");

         throw new IllegalArgumentException("Wanted a donut.");
     }
}

Testando exceções no JUnit5 #

Para conseguir o mesmo no JUnit 5, você usa um mecanismo completamente novo:

O método testado #

public class Calculator {
    public double divide(double a, double b) {
        if (b == 0.0) {
            throw new IllegalArgumentException("Divider must not be 0");
        }
        return a/b;
    }
}

O método teste #

public class CalculatorTest {
    @Test
    void triangularMinus5() { // The test method does not have to be public in JUnit5
        Calculator calc = new Calculator();

        IllegalArgumentException thrown = assertThrows(
            IllegalArgumentException.class, 
            () -> calculator.divide(42.0, 0.0));
        // If the exception has not been thrown, the above test has failed.

        // And now you may further inspect the returned exception...
        // ...e.g. like this:
        assertEquals("Divider must not be 0", thrown.getMessage());
}

Ignorando testes #

Para ignorar um teste, basta adicionar a anotação @Ignore ao teste e, opcionalmente, fornecer um parâmetro para a anotação com o motivo.

@Ignore("Calculator add not implemented yet.")
@Test
public void testPlus() {
    assertEquals(5, calculator.add(2,3));
}

Comparado a comentar o teste ou remover a anotação @Test, o executor de teste ainda relatará este teste e observará que ele foi ignorado.

Também é possível ignorar um caso de teste condicionalmente usando JUnit suposições. Um exemplo de caso de uso seria executar o caso de teste somente após um determinado bug ser corrigido por um desenvolvedor. Exemplo:

import org.junit.Assume;
import org.junit.Assert;
...

@Test 
public void testForBug1234() {

    Assume.assumeTrue(isBugFixed(1234));//will not run this test until the bug 1234 is fixed

    Assert.assertEquals(5, calculator.add(2,3));
}

O executor padrão trata os testes com suposições com falha como ignorados. É possível que outros corredores possam se comportar de maneira diferente, por exemplo. tratá-los como passado.

JUnit – Exemplos básicos de anotação #

Aqui estão algumas anotações básicas do JUnit que você deve entender: ================================================== =========

@BeforeClass – Run once before any of the test methods in the class, public static void 
@AfterClass – Run once after all the tests in the class has been run, public static void
@Before – Run before @Test, public void
@After – Run after @Test, public void
@Test – This is the test method to run, public void

Instalação ou Configuração #

Como JUnit é uma biblioteca Java, tudo o que você precisa fazer para instalá-la é adicionar alguns arquivos JAR no caminho de classe do seu projeto Java e você está pronto para começar.

Você pode baixar esses dois arquivos JAR manualmente: junit.jar & hamcrest-core.jar.

Se você estiver usando o Maven, você pode simplesmente adicionar uma dependência ao seu pom.xml:

<dependency>
  <groupId>junit</groupId>
  <artifactId>junit</artifactId>
  <version>4.12</version>
  <scope>test</scope>
</dependency>

Ou se você estiver usando Gradle, adicione uma dependência em seu build.gradle:

apply plugin: 'java'

dependencies {
    testCompile 'junit:junit:4.12'
}

Depois disso, você pode criar sua primeira classe de teste:

import static org.junit.Assert.assertEquals;

import org.junit.Test;

public class MyTest {
    @Test
    public void onePlusOneShouldBeTwo() {
        int sum = 1 + 1;
        assertEquals(2, sum);
    }
}

e execute-o a partir da linha de comando:

  • Windows java -cp .;junit-X.YY.jar;hamcrest-core-X.Y.jar org.junit.runner.JUnitCore MyTest
  • Linux ou OsX java -cp .:junit-X.YY.jar:hamcrest-core-X.Y.jar org.junit.runner.JUnitCore MyTest

ou com Maven: mvn test

Exemplo de teste de unidade básico #

Este exemplo é uma configuração básica para teste de unidade do StringBuilder.toString() usando junit.

import static org.junit.Assert.assertEquals;

import org.junit.Test;

public class StringBuilderTest {

    @Test
    public void stringBuilderAppendShouldConcatinate()  {
        StringBuilder stringBuilder = new StringBuilder();
        stringBuilder.append("String");
        stringBuilder.append("Builder");
        stringBuilder.append("Test");

        assertEquals("StringBuilderTest", stringBuilder.toString());
    }

}

@Antes Depois #

Um método anotado com @Before será executado antes de cada execução dos métodos @Test. Análogo, um método anotado @After é executado após cada método @Test. Isso pode ser usado para configurar repetidamente uma configuração de teste e limpar após cada teste. Portanto, os testes são independentes e o código de preparação não é copiado dentro do método @Test.

Exemplo:

import static org.junit.Assert.assertEquals;

import java.util.ArrayList;
import java.util.List;

import org.junit.After;
import org.junit.Before;
import org.junit.Test;

public class DemoTest {

    private List<Integer> list;

    @Before
    public void setUp() {
        list = new ArrayList<>();
        list.add(3);
        list.add(1);
        list.add(4);
        list.add(1);
        list.add(5);
        list.add(9);
    }

    @After
    public void tearDown() {
        list.clear();
    }

    @Test
    public void shouldBeOkToAlterTestData() {
        list.remove(0); // Remove first element of list.
        assertEquals(5, list.size()); // Size is down to five
    }

    @Test
    public void shouldBeIndependentOfOtherTests() {
        assertEquals(6, list.size());
    }
}

Métodos anotados com @Before ou @After devem ser public void e com zero argumentos.
