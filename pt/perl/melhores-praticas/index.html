Usando Perl::Critic #

Se você deseja começar a implementar as práticas recomendadas para você ou sua equipe, Perl::Critic é o melhor lugar para começar. O módulo é baseado no livro Perl Best Practices de Damien Conway e faz um bom trabalho implementando as sugestões feitas nele.

Nota: Devo mencionar (e o próprio Conway diz no livro) que estas são sugestões. Descobri que o livro fornece um raciocínio sólido na maioria dos casos, embora certamente não concorde com todos eles. O importante a lembrar é que, quaisquer que sejam as práticas que você decida adotar, você permanece consistente. Quanto mais previsível for o seu código, mais fácil será a manutenção.

Você também pode experimentar o Perl::Critic através do seu navegador em perlcritic.com.

Instalação #

cpan Perl::Critic

Isso instalará o conjunto de regras básico e um script perlcritic que pode ser chamado a partir da linha de comando.

Uso básico #

O documento CPAN para perlcritic contém a documentação completa, portanto, vou abordar apenas os casos de uso mais comuns para você começar. O uso básico é simplesmente chamar perlcritic no arquivo:

 perlcritic -1 /path/to/script.pl

perlcritic funciona tanto em scripts quanto em módulos. O -1 refere-se ao nível de gravidade das regras que você deseja executar no script. Existem cinco níveis que correspondem a quanto Perl::Critic irá separar seu código.

-5 é o mais suave e só avisará sobre problemas potencialmente perigosos que podem causar resultados inesperados. -1 é o mais brutal e reclamará de coisas tão pequenas quanto seu código estar arrumado ou não. Na minha experiência, manter o código compatível com o nível 3 é bom o suficiente para se manter fora de perigo sem ficar muito meticuloso.

Por padrão, qualquer falha listará o motivo e a gravidade em que a regra é acionada:

perlcritic -3 --verbose 8 /path/to/script.pl

Debugging module loaded at line 16, column 1.  You've loaded Data::Dumper, which probably shouln't be loaded in production.  (Severity: 4)
Private subroutine/method '_sub_name' declared but not used at line 58, column 1.  Eliminate dead code.  (Severity: 3)
Backtick operator used at line 230, column 37.  Use IPC::Open3 instead.  (Severity: 3)
Backtick operator used at line 327, column 22.  Use IPC::Open3 instead.  (Severity: 3)

Ignorando código #

Haverá momentos em que você não poderá cumprir uma política Perl::Critic. Nesses casos, você pode incluir comentários especiais, “## use critic()” e “## no critic”, em torno de seu código para fazer com que Perl::Critic os ignore. Basta adicionar as regras que você deseja ignorar entre parênteses (os múltiplos podem ser separados por uma vírgula).

##no critic qw(InputOutput::ProhibitBacktickOperator)
my $filesystem_diff = join q{}, `diff $trunk_checkout $staging_checkout`;
## use critic

Certifique-se de envolver todo o bloco de código ou o Critic pode não reconhecer a instrução ignore.

## no critic (Subroutines::ProhibitExcessComplexity)
sub no_time_to_refactor_this {
    ...
}
## use critic

Observe que existem certas políticas que são executadas no nível do documento e não podem ser isentas dessa maneira. No entanto, eles podem ser desligados …

Criando Exceções Permanentes #

Usar ## no critical() é bom, mas quando você começar a adotar padrões de codificação, provavelmente desejará fazer exceções permanentes a certas regras. Você pode fazer isso criando um arquivo de configuração .perlcriticrc.

Esse arquivo permitirá que você personalize não apenas quais políticas são executadas, mas como elas são executadas. Usá-lo é tão simples quanto colocar o arquivo em seu diretório pessoal (no Linux, não tenha certeza se é o mesmo local no Windows). Ou você pode especificar o arquivo de configuração ao executar o comando usando a opção –profile:

perlcritic -1 --profile=/path/to/.perlcriticrc /path/to/script.pl

Novamente, a página perlcritic CPAN tem uma lista completa dessas opções. Vou listar alguns exemplos do meu próprio arquivo de configuração:

Aplicar configurações básicas:

#very very harsh
severity = 1
color-severity-medium = bold yellow
color-severity-low = yellow
color-severity-lowest = bold blue

Desative uma regra (observe o traço na frente do nome da política):

# do not require version control numbers
[-Miscellanea::RequireRcsKeywords]

# pod spelling is too over-zealous, disabling
[-Documentation::PodSpelling]

Modificando uma regra:

# do not require checking for print failure ( false positives for printing to stdout, not filehandle )
[InputOutput::RequireCheckedSyscalls]
    functions = open close

# Allow specific unused subroutines for moose builders
[Subroutines::ProhibitUnusedPrivateSubroutines]
private_name_regex = _(?!build_)\w+

Conclusão #

Devidamente utilizado, o Perl::Critic pode ser uma ferramenta inestimável para ajudar as equipes a manter sua codificação consistente e de fácil manutenção, independentemente das políticas de práticas recomendadas que você emprega.
