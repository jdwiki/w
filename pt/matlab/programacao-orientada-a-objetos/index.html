Classes Value vs Handle #

As classes no MATLAB são divididas em duas categorias principais: classes de valor e classes de manipulação. A principal diferença é que ao copiar uma instância de uma classe de valor, os dados subjacentes são copiados para a nova instância, enquanto para as classes handle a nova instância aponta para os dados originais e a alteração dos valores na nova instância os altera no original. Uma classe pode ser definida como um handle herdando da classe handle.

classdef valueClass
    properties
        data
    end
end

e

classdef handleClass < handle
    properties
        data
    end
end

então

>> v1 = valueClass;
>> v1.data = 5;
>> v2 = v1;
>> v2.data = 7;
>> v1.data
ans =
     5

>> h1 = handleClass;
>> h1.data = 5;
>> h2 = h1;
>> h2.data = 7;
>> h1.data
ans =
     7

Definindo uma classe #

Uma classe pode ser definida usando classdef em um arquivo .m com o mesmo nome da classe. O arquivo pode conter o bloco classdef…end e funções locais para uso dentro de métodos de classe.

A definição de classe MATLAB mais geral tem a seguinte estrutura:

classdef (ClassAttribute = expression, ...) ClassName < ParentClass1 & ParentClass2 & ...

   properties (PropertyAttributes) 
      PropertyName
   end 

   methods (MethodAttributes) 
      function obj = methodName(obj,arg2,...)
         ...
      end
   end

   events (EventAttributes) 
      EventName
   end

   enumeration
      EnumName
   end

end

Documentação do MATLAB: Atributos de classe, [Atributos de propriedade](http://www.mathworks.com/help/matlab/ matlab_oop/property-attributes.html), Atributos do método, [Atributos do evento](http://www.mathworks. com/help/matlab/matlab_oop/event-attributes.html), Restrições de classe de enumeração.

Classe de exemplo: #

Uma classe chamada Car pode ser definida no arquivo Car.m como

classdef Car < handle % handle class so properties persist
    properties
        make
        model
        mileage = 0;
    end

    methods
        function obj = Car(make, model)
            obj.make = make;
            obj.model = model;
        end
        function drive(obj, milesDriven)
            obj.mileage = obj.mileage + milesDriven;
        end
    end
end

Observe que o construtor é um método com o mesmo nome da classe. <Um construtor é um método especial de uma classe ou estrutura em programação orientada a objetos que inicializa um objeto desse tipo. Um construtor é um método de instância que geralmente tem o mesmo nome da classe e pode ser usado para definir os valores dos membros de um objeto, seja para valores padrão ou definidos pelo usuário.>

Uma instância dessa classe pode ser criada chamando o construtor;

>> myCar = Car('Ford', 'Mustang'); //creating an instance of car class

Chamar o método drive aumentará a milhagem

>> myCar.mileage 
    
    ans = 
            0

>> myCar.drive(450);

>> myCar.mileage
    
   ans = 
            450

Construtores #

Um construtor é um método especial em uma classe que é chamado quando uma instância de um objeto é criada. É uma função normal do MATLAB que aceita parâmetros de entrada, mas também deve seguir certas regras.

Construtores não são necessários, pois o MATLAB cria um padrão. Na prática, no entanto, este é um lugar para definir um estado de um objeto. Por exemplo, as propriedades podem ser restritas especificando attributes. Então, um construtor pode [inicializar][defaultvalue] tais propriedades por padrão ou valores definidos pelo usuário que de fato podem ser enviados por parâmetros de entrada de um construtor.

Chamando um construtor de uma classe simples

Esta é uma classe simples Person.

classdef Person
    properties
        name
        surname
        address
    end
    
    methods
        function obj = Person(name,surname,address)
            obj.name = name;
            obj.surname = surname;
            obj.address = address;
        end
    end
end

O nome de um construtor é o mesmo nome de uma classe. Consequentemente, os construtores são chamados pelo nome de sua classe. Uma classe Person pode ser criada da seguinte forma:

>> p = Person('John','Smith','London')
p = 
  Person with properties:

       name: 'John'
    surname: 'Smith'
    address: 'London'

Chamando um construtor de uma classe filha

As classes podem ser herdadas de classes pai se compartilharem propriedades ou métodos comuns. Quando uma classe é herdada de outra, é provável que um construtor de uma classe pai precise ser chamado.

Uma classe Member herda de uma classe Person porque Member usa as mesmas propriedades que a classe Person, mas também adiciona payment à sua definição.

classdef Member < Person
    properties
        payment
    end

    methods
        function obj = Member(name,surname,address,payment)
            obj = obj@Person(name,surname,address);
            obj.payment = payment;
        end
    end
end

Da mesma forma que a classe Person, Member é criado chamando seu construtor:

>> m = Member('Adam','Woodcock','Manchester',20)
m = 
  Member with properties:

    payment: 20
       name: 'Adam'
    surname: 'Woodcock'
    address: 'Manchester'

Um construtor de Person requer três parâmetros de entrada. Member deve respeitar este fato e, portanto, chamar um construtor da classe Person com três parâmetros. É preenchido pela linha:

obj = obj@Person(name,surname,address);

O exemplo acima mostra o caso em que uma classe filha precisa de informações para sua classe pai. É por isso que um construtor de Member requer quatro parâmetros: três para sua classe pai e um para si mesmo.

Herdando de classes e classes abstratas #

Isenção de responsabilidade: os exemplos apresentados aqui são apenas para fins de mostrar o uso de classes abstratas e herança e podem não ser necessariamente de uso prático. Além disso, não existe algo tão polimórfico no MATLAB e, portanto, o uso de classes abstratas é limitado. Este exemplo é para mostrar quem deve criar uma classe, herdar de outra classe e aplicar uma classe abstrata para definir uma interface comum.

O uso de classes abstratas é bastante limitado no MATLAB, mas ainda pode ser útil em algumas ocasiões.

Digamos que queremos um registrador de mensagens. Podemos criar uma classe semelhante à abaixo:

classdef ScreenLogger
    properties(Access=protected)
        scrh;
    end
    
    methods
        function obj = ScreenLogger(screenhandler)
            obj.scrh = screenhandler;
        end
        
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                fprintf(obj.scrh, '%s\n', sprintf(varargin{:}));
            end
        end
    end
end

Propriedades e métodos

Em suma, as propriedades mantêm um estado de um objeto enquanto os métodos são como interfaces e definem ações em objetos.

A propriedade scrh está protegida. É por isso que ele deve ser inicializado em um construtor. Existem outros métodos (getters) para acessar essa propriedade, mas está fora do contexto deste exemplo. Propriedades e métodos podem ser acessados ​​por meio de uma variável que contém uma referência a um objeto usando a notação de ponto seguida por um nome de um método ou uma propriedade:

mylogger = ScreenLogger(1);                         % OK
mylogger.LogMessage('My %s %d message', 'very', 1); % OK
mylogger.scrh = 2;                                  % ERROR!!! Access denied

Propriedades e métodos podem ser públicos, privados ou protegidos. Neste caso, protegido significa que poderei acessar scrh de uma classe herdada, mas não de fora. Por padrão, todas as propriedades e métodos são públicos. Portanto, LogMessage() pode ser usado livremente fora da definição da classe. Também LogMessage define uma interface que significa que isso é o que devemos chamar quando queremos que um objeto registre nossas mensagens personalizadas.

Inscrição

Digamos que eu tenha um script em que utilizo meu logger:

clc;
% ... a code
logger = ScreenLogger(1);
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');

Se eu tiver vários lugares onde uso o mesmo logger e depois quiser alterá-lo para algo mais sofisticado, como escrever uma mensagem em um arquivo, teria que criar outro objeto:

classdef DeepLogger
    properties(SetAccess=protected)
        FileName
    end
    methods
        function obj = DeepLogger(filename)
            obj.FileName = filename;
        end
        
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end 
end

e apenas mude uma linha de um código para isso:

clc;
% ... a code
logger = DeepLogger('mymessages.log');

O método acima simplesmente abrirá um arquivo, anexará uma mensagem no final do arquivo e o fechará. No momento, para ser consistente com minha interface, preciso lembrar que o nome de um método é LogMessage(), mas também pode ser qualquer outra coisa. O MATLAB pode forçar o desenvolvedor a manter o mesmo nome usando classes abstratas. Digamos que definimos uma interface comum para qualquer registrador:

classdef MessageLogger
    methods(Abstract=true)
        LogMessage(obj, varargin);
    end
end

Agora, se ScreenLogger e DeepLogger herdarem desta classe, o MATLAB irá gerar um erro se LogMessage() não estiver definido. Classes abstratas ajudam a construir classes semelhantes que podem usar a mesma interface.

Por causa deste exemplo, farei uma mudança um pouco diferente. Vou assumir que o DeepLogger fará tanto a mensagem de log em uma tela quanto em um arquivo ao mesmo tempo. Como o ScreenLogger já registra mensagens na tela, vou herdar o DeepLogger do ScreenLoggger para evitar repetições. ScreenLogger não muda nada além da primeira linha:

classdef ScreenLogger < MessageLogger
// the rest of previous code

No entanto, o DeepLogger precisa de mais alterações no método LogMessage:

classdef DeepLogger < MessageLogger & ScreenLogger
    properties(SetAccess=protected)
        FileName
        Path
    end
    methods
        function obj = DeepLogger(screenhandler, filename)
            [path,filen,ext] = fileparts(filename);
            obj.FileName = [filen ext];
            pbj.Path     = pathn;
            obj = obj@ScreenLogger(screenhandler);
        end
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                LogMessage@ScreenLogger(obj, varargin{:});
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end
end

Em primeiro lugar, eu simplesmente inicializo as propriedades no construtor. Em segundo lugar, porque esta classe herda de ScreenLogger eu tenho que inicializar este objeto pai também. Esta linha é ainda mais importante porque o construtor ScreenLogger requer um parâmetro para inicializar seu próprio objeto. Está linha:

obj = obj@ScreenLogger(screenhandler);

simplesmente diz “chame o construtor do ScreenLogger e inicialize-o com um manipulador de tela”. Vale a pena notar aqui que eu defini scrh como protegido. Portanto, eu poderia acessar esta propriedade igualmente de DeepLogger. Se a propriedade foi definida como privada. A única maneira de inicializá-lo seria usando consuctor.

Outra mudança está na seção métodos. Novamente para evitar repetição, eu chamo LogMessage() de uma classe pai para registrar uma mensagem em uma tela. Se eu tivesse que mudar alguma coisa para fazer melhorias no log de tela, agora tenho que fazer isso em um só lugar. O código restante é o mesmo que faz parte do DeepLogger.

Como essa classe também herda de uma classe abstrata MessageLogger, tive que me certificar de que LogMessage() dentro de DeepLogger também estivesse definido. Herdar de MessageLogger é um pouco complicado aqui. Eu acho que a redefinição de LogMessage é obrigatória - meu palpite.

Em termos do código onde o logger é aplicado, graças a uma interface comum nas classes, posso ter certeza de que mudar essa linha em todo o código não causaria nenhum problema. As mesmas mensagens serão registradas na tela como antes, mas adicionalmente o código gravará essas mensagens em um arquivo.

clc;
% ... a code
logger = DeepLogger(1, 'mylogfile.log');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');
% ... a code
logger.LogMessage('something');

Espero que esses exemplos expliquem o uso de classes, o uso de herança e o uso de classes abstratas.

PS. A solução para o problema acima é uma das muitas. Outra solução, menos complexa, seria fazer com que o ScreenLoger fosse um componente de outro logger como FileLogger etc. ScreenLogger seria mantido em uma das propriedades. Seu LogMessage simplesmente chamaria LogMessage do ScreenLogger e mostraria o texto em uma tela. Eu escolhi uma abordagem mais complexa para mostrar como as classes funcionam no MATLAB. O código de exemplo abaixo:

classdef DeepLogger < MessageLogger
    properties(SetAccess=protected)
        FileName
        Path
        ScrLogger
    end
    methods
        function obj = DeepLogger(screenhandler, filename)
            [path,filen,ext] = fileparts(filename);
            obj.FileName     = [filen ext];
            obj.Path         = pathn;
            obj.ScrLogger    = ScreenLogger(screenhandler);
        end
        function LogMessage(obj, varargin)
            if ~isempty(varargin)
                varargin{1} = num2str(varargin{1});
                obj.LogMessage(obj.ScrLogger, varargin{:}); % <-------- thechange here
                fid = fopen(obj.fullfname, 'a+t');
                fprintf(fid, '%s\n', sprintf(varargin{:}));
                fclose(fid);
            end
        end
    end
end
