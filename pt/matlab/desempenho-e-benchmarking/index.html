Identificando gargalos de desempenho usando o Profiler #

O MATLAB Profiler é uma ferramenta para software profiling) do código MATLAB. Usando o Profiler, é possível obter uma representação visual do tempo de execução e do consumo de memória.

A execução do Profiler pode ser feita de duas maneiras:

  • Clicar no botão “Run and Time” na GUI do MATLAB enquanto tem algum arquivo .m aberto no editor (adicionado em R2012b).

  • Programaticamente, usando:

    profile on
<some code we want to test>
profile off

Abaixo está algum código de exemplo e o resultado de sua criação de perfil:

function docTest

for ind1 = 1:100
  [~] = var(...
            sum(...
                randn(1000)));
end

spy

Do exposto acima, aprendemos que a função spy leva cerca de 25% do tempo total de execução. No caso de “código real”, uma função que leva uma porcentagem tão grande de tempo de execução seria uma boa candidata para otimização, ao contrário de funções análogas a var e cla cuja otimização deve ser evitada.

Comparando o tempo de execução de várias funções #

A combinação amplamente usada de tic e [toc](http:// de.mathworks.com/help/matlab/ref/toc.html) pode fornecer uma ideia aproximada do tempo de execução de uma função ou trechos de código.

Para comparar várias funções não deve ser usado. Por quê? É quase impossível fornecer condições iguais para que todos os trechos de código sejam comparados em um script usando a solução acima. Talvez as funções compartilhem o mesmo espaço de função e variáveis ​​comuns, então funções chamadas posteriormente e trechos de código já aproveitam as variáveis ​​e funções inicializadas anteriormente. Além disso, não há uma visão se o compilador JIT lidaria com esses snippets subsequentemente chamados igualmente.

A função dedicada para benchmarks é timeit. O exemplo a seguir ilustra seu uso.

Existem a matriz ‘A’ e a matriz ‘B’. Deve-se determinar qual linha de B é a mais semelhante a A contando o número de elementos diferentes.

function t = bench()
    A = [0 1 1 1 0 0];
    B = perms(A);

    % functions to compare
    fcns = {
        @() compare1(A,B);
        @() compare2(A,B);
        @() compare3(A,B);
        @() compare4(A,B);
    };

    % timeit
    t = cellfun(@timeit, fcns);
end

function Z = compare1(A,B)  
    Z = sum(  bsxfun(@eq,  A,B) , 2);
end
function Z = compare2(A,B)  
    Z = sum(bsxfun(@xor, A, B),2);
end
function Z = compare3(A,B)  
    A = logical(A);
    Z = sum(B(:,~A),2) + sum(~B(:,A),2);
end
function Z = compare4(A,B)  
     Z = pdist2( A, B, 'hamming', 'Smallest', 1 );
end

Essa forma de benchmark foi vista pela primeira vez em esta resposta.

Tudo bem ser solteiro! #

Visão geral: #

O tipo de dados padrão para arrays numéricos no MATLAB é double. double é uma representação de números em ponto flutuante, e esse formato leva 8 bytes (ou 64 bits) por valor. Em alguns casos, onde, por exemplo, lidando apenas com números inteiros ou quando a instabilidade numérica não é um problema iminente, essa profundidade de bits alta pode não ser necessária. Por esse motivo, é aconselhável considerar os benefícios da precisão single (ou outros types apropriados):

  • Tempo de execução mais rápido (especialmente perceptível em GPUs).
  • Metade do consumo de memória: pode ter sucesso onde double falhar devido a um erro de falta de memória; mais compacto ao armazenar como arquivos.

A conversão de uma variável de qualquer tipo de dados suportado para single é feita usando:

sing_var = single(var);

Algumas funções comumente usadas (como: zeros, [eye](http://www .mathworks.com/help/matlab/ref/eye.html#btpkgsh-1), ones, etc.) que geram valores double por padrão, permitem especificar o tipo/classe da saída.

Convertendo variáveis ​​em um script para uma precisão/tipo/classe não padrão: #

A partir de julho de 2016, não existe nenhuma maneira documentada de alterar o tipo de dados default MATLAB de double.

No MATLAB, novas variáveis ​​geralmente imitam os tipos de dados das variáveis ​​usadas ao criá-las. Para ilustrar isso, considere o seguinte exemplo:

A = magic(3);
B = diag(A);
C = 20*B;
>> whos C
  Name      Size            Bytes  Class     Attributes
  C         3x1                24  double 


A = single(magic(3)); % A is converted to "single"
B = diag(A);
C = B*double(20);     % The stricter type, which in this case is "single", prevails
D = single(size(C));  % It is generally advised to cast to the desired type explicitly.
>> whos C
  Name      Size            Bytes  Class     Attributes
  C         3x1                12  single

Assim, pode parecer suficiente converter/converter várias variáveis ​​iniciais para que a mudança permeie todo o código - no entanto, isso é desencorajado (veja Avisos e Armadilhas abaixo).

Advertências e Armadilhas: #

  1. Conversões repetidas são desencorajadas devido à introdução de ruído numérico (ao converter de single para double) ou à perda de informações (ao converter de double para single, ou entre determinados integer tipos), ex. :

     double(single(1.2)) == double(1.2)   
 ans =
      0

    This can be mitigated somewhat using typecast. See also https://www.wikiod.com/pt/matlab/erros-e-erros-comuns#Esteja ciente da imprecisão do ponto flutuante

  2. Confiar apenas na digitação de dados implícita (ou seja, o que o MATLAB supõe que o tipo de saída de uma computação deve ser) é desencorajado devido a vários efeitos indesejados que podem surgir:

    • Loss of information: when a double result is expected, but a careless combination of single and double operands yields single precision.

    • Unexpectedly high memory consumption: when a single result is expected but a careless computation results in a double output.

    • Unnecessary overhead when working with GPUs: when mixing gpuArray types (i.e. variables stored in VRAM) with non-gpuArray variables (i.e. those usually stored in RAM) the data will have to be transferred one way or the other before the computation can be performed. This operation takes time, and can be very noticeable in repetative computations.

    • Errors when mixing floating-point types with integer types: functions like mtimes (*) are not defined for mixed inputs of integer and floating point types - and will error. Functions like times (.*) are not defined at all for integer-type inputs - and will again error.

      >> ones(3,3,'int32')*ones(3,3,'int32')
Error using  * 
MTIMES is not fully supported for integer classes. At least one input must be scalar.

>> ones(3,3,'int32').*ones(3,3,'double')
Error using  .* 
Integers can only be combined with integers of the same class, or scalar doubles.

    For better code readability and reduced risk of unwanted types, a defensive approach is advised, where variables are explicitly cast to the desired type.

A importância da pré-alocação #

Arrays no MATLAB são mantidos como blocos contínuos na memória, alocados e liberados automaticamente pelo MATLAB. O MATLAB oculta as operações de gerenciamento de memória, como o redimensionamento de uma matriz, por trás de uma sintaxe fácil de usar:

a = 1:4

a =

     1     2     3     4

a(5) = 10  % or alternatively a = [a, 10]

a =

     1     2     3     4    10

É importante entender que a operação acima não é uma operação trivial, a(5) = 10 fará com que o MATLAB aloque um novo bloco de memória de tamanho 5, copie os primeiros 4 números e defina o 5º como 10. Essa é uma operação O(numel(a)), e não O(1).

Considere o seguinte:

clear all
n=12345678;
a=0;
tic
for i = 2:n
    a(i) = sqrt(a(i-1)) + i;
end
toc

Elapsed time is 3.004213 seconds.

a é realocado n vezes neste loop (excluindo algumas otimizações realizadas pelo MATLAB)! Observe que o MATLAB nos dá um aviso:

“A variável ‘a’ parece mudar de tamanho em cada iteração de loop. Considere pré-alocação para velocidade.”

O que acontece quando pré-alocamos?

a=zeros(1,n);
tic
for i = 2:n
    a(i) = sqrt(a(i-1)) + i;
end
toc

Elapsed time is 0.410531 seconds.

Podemos ver que o tempo de execução é reduzido em uma ordem de magnitude.

Métodos para pré-alocação:

O MATLAB fornece várias funções para alocação de vetores e matrizes, dependendo das necessidades específicas do usuário. Estes incluem: zeros, ones, nan, eye, true etc.

a = zeros(3)       % Allocates a 3-by-3 matrix initialized to 0
a =

     0     0     0
     0     0     0
     0     0     0

a = zeros(3, 2)     % Allocates a 3-by-2 matrix initialized to 0
a =

     0     0
     0     0
     0     0

a = ones(2, 3, 2)      % Allocates a 3 dimensional array (2-by-3-by-2) initialized to 1
a(:,:,1) =

     1     1     1
     1     1     1


a(:,:,2) =

     1     1     1
     1     1     1

a = ones(1, 3) * 7  % Allocates a row vector of length 3 initialized to 7
a =

     7     7     7

Um tipo de dados também pode ser especificado:

a = zeros(2, 1, 'uint8');  % allocates an array of type uint8

Também é fácil clonar o tamanho de um array existente:

a = ones(3, 4);       % a is a 3-by-4 matrix of 1's
b = zeros(size(a));  % b is a 3-by-4 matrix of 0's

E clone o tipo:

a = ones(3, 4, 'single');       % a is a 3-by-4 matrix of type single
b = zeros(2, 'like', a);        % b is a 2-by-2 matrix of type single

note que ’like’ também clona complexity e sparsity.

A pré-alocação é obtida implicitamente usando qualquer função que retorne um array do tamanho final necessário, como rand, gallery, kron, bsxfun , colon e muitos outros. Por exemplo, uma maneira comum de alocar vetores com elementos de variação linear é usando o operador de dois pontos (com a variante de 2 ou 3 operandos1):

a = 1:3 
a =

     1     2     3

a = 2:-3:-4
a =

     2    -1    -4

Arrays de células podem ser alocados usando a função cell() da mesma forma que zeros().

a = cell(2,3)
a = 

    []    []    []
    []    []    []

Observe que as matrizes de células funcionam mantendo ponteiros para os locais na memória do conteúdo da célula. Portanto, todas as dicas de pré-alocação também se aplicam aos elementos individuais da matriz de células.

reorganizar um ND-array pode melhorar o desempenho geral #

Em alguns casos, precisamos aplicar funções a um conjunto de matrizes ND. Vejamos este exemplo simples.

A(:,:,1) = [1 2; 4 5];
A(:,:,2) = [11 22; 44 55];
B(:,:,1) = [7 8; 1 2];
B(:,:,2) = [77 88; 11 22];

A =

ans(:,:,1) =

   1   2 
   4   5 

ans(:,:,2) =

   11   22
   44   55

>> B
B =

ans(:,:,1) =

   7   8
   1   2

ans(:,:,2) =

   77   88
   11   22

Ambas as matrizes são 3D, digamos que temos que calcular o seguinte:

result= zeros(2,2);
...
for k = 1:2 
   result(i,j) = result(i,j) + abs( A(i,j,k) - B(i,j,k) );
...

if k is very large, this for-loop can be a bottleneck since MATLAB order the data in a column major fashion. So a better way to compute "result" could be:

% trying to exploit the column major ordering
Aprime = reshape(permute(A,[3,1,2]), [2,4]);
Bprime = reshape(permute(B,[3,1,2]), [2,4]);


>> Aprime
Aprime =

    1    4    2    5
   11   44   22   55

>> Bprime
Bprime =

    7    1    8    2
   77   11   88   22

Agora substituímos o loop acima pelo seguinte:

result= zeros(2,2);
....
temp = abs(Aprime - Bprime);
for k = 1:2
    result(i,j) = result(i,j) + temp(k, i+2*(j-1));
...

Reorganizamos os dados para que possamos explorar a memória cache. A permutação e a remodelação podem ser caras, mas ao trabalhar com grandes matrizes ND, o custo computacional relacionado a essas operações é muito menor do que trabalhar com matrizes não organizadas.
