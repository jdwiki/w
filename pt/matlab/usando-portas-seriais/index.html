As portas seriais são uma interface comum para comunicação com sensores externos ou sistemas embarcados, como Arduinos. As comunicações seriais modernas são frequentemente implementadas em conexões USB usando adaptadores seriais USB. O MATLAB fornece funções integradas para comunicações seriais, incluindo protocolos RS-232 e RS-485. Essas funções podem ser usadas para portas seriais de hardware ou conexões seriais USB “virtuais”. Os exemplos aqui ilustram as comunicações seriais no MATLAB.

Parâmetros #

Parâmetro da porta serial o que faz
BaudRate Define a taxa de transmissão. O mais comum hoje é 57600, mas 4800, 9600 e 115200 também são vistos com frequência
InputBufferSize O número de bytes mantidos na memória. O Matlab possui um FIFO, o que significa que novos bytes serão descartados. O padrão é 512 bytes, mas pode ser facilmente definido para 20 MB sem problemas. Existem apenas alguns casos extremos em que o usuário gostaria que isso fosse pequeno
BytesDisponível O número de bytes esperando para serem lidos
Valores Enviados O número de bytes enviados desde que a porta foi aberta
Valores Recebidos O número de bytes lidos desde que a porta foi aberta
BytesAvailableFcn Especifique a função de retorno de chamada a ser executada quando um número especificado de bytes estiver disponível no buffer de entrada ou um terminador for lido
BytesAvailableFcnCount Especifique o número de bytes que devem estar disponíveis no buffer de entrada para gerar um evento bytes-available
BytesAvailableFcnMode Especifique se o evento bytes-available é gerado após um número especificado de bytes estar disponível no buffer de entrada ou após a leitura de um terminador

Criando uma porta serial no Mac/Linux/Windows #

% Define serial port with a baud rate of 115200
rate = 115200;
if ispc
    s = serial('COM1', 'BaudRate',rate);
elseif ismac
    % Note that on OSX the serial device is uniquely enumerated. You will
    % have to look at /dev/tty.* to discover the exact signature of your
    % serial device
    s = serial('/dev/tty.usbserial-A104VFT7', 'BaudRate',rate);
elseif isunix
    s = serial('/dev/ttyusb0', 'BaudRate',rate);
end

% Set the input buffer size to 1,000,000 bytes (default: 512 bytes).
s.InputBufferSize = 1000000;    

% Open serial port    
fopen(s);

Escolhendo seu modo de comunicação #

Matlab suporta comunicação síncrona e assíncrona com uma porta serial. É importante escolher o modo de comunicação correto. A escolha dependerá:

  • como o instrumento com o qual você está se comunicando se comporta.
  • quais outras funções seu programa principal (ou GUI) terá que fazer além de gerenciar a porta serial.

Vou definir 3 casos diferentes para ilustrar, do mais simples ao mais exigente. Para os 3 exemplos, o instrumento ao qual estou conectando é uma placa de circuito com um inclinômetro, que pode funcionar nos 3 modos que descreverei abaixo.

Modo 1: Síncrono (Mestre/Escravo) #

Este modo é o mais simples. Corresponde ao caso em que o PC é o Master e o instrumento é o slave. O instrumento não envia nada para a porta serial por conta própria, apenas responde uma resposta após ser feita uma pergunta/comando do Mestre (o PC, seu programa). Por exemplo:

  • O PC envia um comando: “Me dê uma medida agora”
  • O instrumento recebe o comando, faz a medição e envia de volta o valor da medição para a linha serial: “O valor do inclinômetro é XXX”.

OU

  • O PC envia um comando: “Mudar do modo X para o modo Y”
  • O instrumento recebe o comando, executa-o e envia uma mensagem de confirmação de volta para a linha serial: “Comando executado” (ou “Comando NÃO executado”). Isso é comumente chamado de resposta ACK/NACK (para “Acknowledge(d)” / “NOT Acknowledged”).

Resumo: neste modo, o instrumento (o Slave) apenas envia dados para a linha serial imediatamente após ser solicitado pelo PC (o Master)

Modo 2: Assíncrono #

Agora suponha que eu iniciei meu instrumento, mas ele é mais do que apenas um sensor burro. Ele monitora constantemente sua própria inclinação e enquanto estiver na vertical (dentro de uma tolerância, digamos +/-15 graus), permanece silencioso. Caso o dispositivo seja inclinado em mais de 15 graus e se aproxime da horizontal, ele envia uma mensagem de alarme para a linha serial, seguida imediatamente de uma leitura da inclinação. Enquanto a inclinação estiver acima do limite, ele continua a enviar uma leitura de inclinação a cada 5s.

Se o seu programa principal (ou GUI) está constantemente “aguardando” a chegada de mensagens na linha serial, ele pode fazer isso bem… mas não pode fazer mais nada nesse meio tempo. Se o programa principal for uma GUI, é altamente frustrante ter uma GUI aparentemente “congelada” porque não aceitará nenhuma entrada do usuário. Essencialmente, tornou-se o Slave e o instrumento é o Master. A menos que você tenha uma maneira sofisticada de controlar sua GUI a partir do instrumento, isso é algo a ser evitado. Felizmente, o modo de comunicação assíncrono permitirá:

  • defina uma função separada que diz ao seu programa o que fazer quando uma mensagem é recebida
  • mantenha esta função em um canto, ela só será chamada e executada quando chegar uma mensagem na linha serial. O resto do tempo, a GUI pode executar qualquer outro código que precise ser executado.

Resumo: Neste modo, o instrumento pode enviar mensagem para a linha serial a qualquer momento (mas não necessariamente todo o tempo). O PC não espera permanentemente pelo processamento de uma mensagem. É permitido executar qualquer outro código. Somente quando uma mensagem chega, ele ativa uma função que lerá e processará essa mensagem.

Modo 3: Transmissão (Tempo real) #

Agora vamos liberar todo o poder do meu instrumento. Eu o coloco em um modo em que ele enviará constantemente medições para a linha serial. Meu programa deseja receber esses pacotes e exibi-los em uma curva ou display digital. Se ele enviar apenas um valor a cada 5s como acima, não há problema, mantenha o modo acima. Mas meu instrumento a todo vapor envia um ponto de dados para a linha serial em 1000Hz, ou seja, envia um novo valor a cada milissegundo. Se eu permanecer no modo assíncrono descrito acima, há um alto risco (na verdade, uma certeza garantida) de que a função especial que definimos para processar cada novo pacote leve mais de 1ms para ser executada (se você quiser plotar ou exibir o valor, funções gráficas são bastante lentas, nem mesmo considerando a filtragem ou FFT do sinal). Isso significa que a função começará a ser executada, mas antes de terminar, um novo pacote chegará e acionará a função novamente. A segunda função é colocada em uma fila para execução, e só será iniciada quando a primeira estiver pronta… mas a essa altura alguns novos pacotes chegaram e cada um colocou uma função para executar na fila. Você pode prever rapidamente o resultado: No momento em que estou plotando os 5 pontos, já tenho centenas esperando para serem plotados também… o gui fica mais lento, eventualmente congela, a pilha cresce, os buffers se enchem, até que algo dá. Eventualmente, você fica com um programa completamente congelado ou simplesmente travado.

Para superar isso, desconectaremos ainda mais o link de sincronização entre o PC e o instrumento. Vamos deixar o instrumento enviar dados em seu próprio ritmo, sem acionar imediatamente uma função a cada chegada de pacote. O buffer da porta serial apenas acumulará os pacotes recebidos. O PC só coletará dados no buffer em um ritmo que possa gerenciar (um intervalo regular, configurado no lado do PC), faça algo com ele (enquanto o buffer estiver sendo recarregado pelo instrumento) e, em seguida, colete um novo lote de dados do buffer… e assim por diante.

Resumo: Neste modo, o instrumento envia dados continuamente, que são coletados pelo buffer da porta serial. Em intervalos regulares, o PC coleta dados do buffer e faz algo com eles. Não há link de sincronização rígido entre o PC e o instrumento. Ambos executam suas tarefas em seu próprio tempo.

Leitura da porta serial #

Supondo que você criou o objeto de porta serial s como em [este](https://www.wikiod.com/pt/matlab/usando-portas-seriais#Criando uma porta serial no Mac/Linux/Windows port-on-mac-linux-windows), então

% Read one byte
data = fread(s, 1);

% Read all the bytes, version 1
data = fread(s);

% Read all the bytes, version 2
data = fread(s, s.BytesAvailable);

% Close the serial port
fclose(s);

Fechando uma porta serial mesmo se perdida, excluída ou substituída #

Supondo que você criou o objeto de porta serial s como em [este](https://www.wikiod.com/pt/matlab/usando-portas-seriais#Criando uma porta serial no Mac/Linux/Windows port-on-mac-linux-windows) e, em seguida, para fechá-lo

fclose(s)

No entanto, às vezes você pode acidentalmente perder a porta (por exemplo, limpar, substituir, alterar escopo, etc…), e fclose(s) não funcionará mais. A solução é fácil

fclose(instrfindall)

Mais informações em instrfindall().

Gravando na porta serial #

Supondo que você criou o objeto de porta serial s como em [este](https://www.wikiod.com/pt/matlab/usando-portas-seriais#Criando uma porta serial no Mac/Linux/Windows port-on-mac-linux-windows), então

% Write one byte
fwrite(s, 255);

% Write one 16-bit signed integer
fwrite(s, 32767, 'int16');

% Write an array of unsigned 8-bit integers
fwrite(s,[48 49 50],'uchar');

% Close the serial port
fclose(s);

Processando automaticamente os dados recebidos de uma porta serial #

Alguns dispositivos conectados através de uma porta serial enviam dados para seu programa em uma taxa constante (streaming de dados) ou enviam dados em intervalos imprevisíveis. Você pode configurar a porta serial para executar uma função automaticamente para lidar com dados sempre que eles chegarem. Isso é chamado de “função de retorno de chamada” para o objeto de porta serial.

Existem duas propriedades da porta serial que devem ser configuradas para usar este recurso: o nome da função que você deseja para o retorno de chamada (BytesAvailableFcn), e a condição que deve acionar a execução da função de retorno de chamada (BytesAvailableFcnMode).

Há duas maneiras de acionar uma função de retorno de chamada:

  1. Quando um certo número de bytes foi recebido na porta serial (normalmente usado para dados binários)
  2. Quando um determinado caractere é recebido na porta serial (normalmente usado para texto ou dados ASCII)

As funções de retorno de chamada têm dois argumentos de entrada obrigatórios, chamados obj e event. obj é a porta serial. Por exemplo, se você deseja imprimir os dados recebidos da porta serial, defina uma função para imprimir os dados chamada newdata:

function newdata(obj,event)
    [d,c] = fread(obj);  % get the data from the serial port
    % Note: for ASCII data, use fscanf(obj) to return characters instead of binary values
    fprintf(1,'Received %d bytes\n',c);
    disp(d)
end

Por exemplo, para executar a função newdata sempre que 64 bytes de dados forem recebidos, configure a porta serial assim:

s = serial(port_name);
s.BytesAvailableFcnMode = 'byte';
s.BytesAvailableFcnCount = 64;
s.BytesAvailableFcn = @newdata;

Com dados de texto ou ASCII, os dados são normalmente divididos em linhas com um “caractere terminador”, assim como o texto em uma página. Para executar a função newdata sempre que o caractere de retorno de carro for recebido, configure a porta serial assim:

s = serial(port_name);
s.BytesAvailableFcnMode = 'terminator';
s.Terminator = 'CR';  % the carriage return, ASCII code 13
s.BytesAvailableFcn = @newdata;
