Ativar migrações #

Para habilitar as Migrações do Code First na estrutura de entidade, use o comando

Enable-Migrations

no Console do Gerenciador de Pacotes.

Você precisa ter uma implementação válida de DbContext contendo seus objetos de banco de dados gerenciados pelo EF. Neste exemplo, o contexto do banco de dados conterá os objetos BlogPost e Author:

internal class DatabaseContext: DbContext
{
    public DbSet<Author> Authors { get; set; }

    public DbSet<BlogPost> BlogPosts { get; set; }
}

Após executar o comando, a seguinte saída deve aparecer:

PM> Enable-Migrations
Checking if the context targets an existing database...
Code First Migrations enabled for project <YourProjectName>.
PM>

Além disso, uma nova pasta Migrations deve aparecer com um único arquivo Configuration.cs dentro:

O próximo passo seria criar seu primeiro script de migração de banco de dados que criará o banco de dados inicial (veja o próximo exemplo).

Adicione sua primeira migração #

Depois de habilitar as migrações (consulte este exemplo), agora você pode criar sua primeira migração contendo uma criação inicial de todas as tabelas, índices e conexões do banco de dados.

Uma migração pode ser criada usando o comando

Add-Migration <migration-name>

Este comando criará uma nova classe contendo dois métodos Up e Down que são usados ​​para aplicar e remover a migração.

Agora aplique o comando baseado no exemplo acima para criar uma migração chamada Initial:

PM> Add-Migration Initial
Scaffolding migration 'Initial'.
The Designer Code for this migration file includes a snapshot of your current Code
First model. This snapshot is used to calculate the changes to your model when you
scaffold the next migration. If you make additional changes to your model that you 
want to include in this migration, then you can re-scaffold it by running 
'Add-Migration Initial' again.

Um novo arquivo timestamp_Initial.cs é criado (apenas as coisas importantes são mostradas aqui):

public override void Up()
{
   CreateTable(
       "dbo.Authors",
        c => new
           {
               AuthorId = c.Int(nullable: false, identity: true),
               Name = c.String(maxLength: 128),
           })
        .PrimaryKey(t => t.AuthorId);
        
    CreateTable(
       "dbo.BlogPosts",
       c => new
           {
                Id = c.Int(nullable: false, identity: true),
                Title = c.String(nullable: false, maxLength: 128),
                Message = c.String(),
                Author_AuthorId = c.Int(),
            })
        .PrimaryKey(t => t.Id)
        .ForeignKey("dbo.Authors", t => t.Author_AuthorId)
        .Index(t => t.Author_AuthorId);
}
    
public override void Down()
{
    DropForeignKey("dbo.BlogPosts", "Author_AuthorId", "dbo.Authors");
    DropIndex("dbo.BlogPosts", new[] { "Author_AuthorId" });
    DropTable("dbo.BlogPosts");
    DropTable("dbo.Authors");
}

Como você pode ver, no método Up() duas tabelas Authors e BlogPosts são criadas e os campos são criados de acordo. Além disso, a relação entre as duas tabelas é criada adicionando o campo Author_AuthorId. Por outro lado, o método Down() tenta reverter as atividades de migração.

Se você se sentir confiante com sua migração, poderá aplicar a migração ao banco de dados usando o comando:

Update-Database

Todas as migrações pendentes (neste caso a migração Initial) são aplicadas ao banco de dados e depois o método seed é aplicado (o exemplo apropriado)

PM> update-database
Specify the '-Verbose' flag to view the SQL statements being applied to the target     
database.
Applying explicit migrations: [201609302203541_Initial].
Applying explicit migration: 201609302203541_Initial.
Running Seed method.

Você pode ver os resultados das atividades no SQL Explorer: ![visão geral do banco de dados][2]

Para os comandos Add-Migration e Update-Database estão disponíveis várias opções que podem ser usadas para ajustar as atividades. Para ver todas as opções, use

get-help Add-Migration

e

get-help Update-Database

Semeando dados durante as migrações #

Depois de habilitar e criar migrações, pode ser necessário preencher ou migrar inicialmente dados em seu banco de dados. Existem várias possibilidades, mas para migrações simples você pode usar o método ‘Seed()’ no arquivo Configuration criado após chamar enable-migrations.

A função Seed() recupera um contexto de banco de dados como seu único parâmetro e você pode executar operações EF dentro desta função:

protected override void Seed(Model.DatabaseContext context);

Você pode realizar todos os tipos de atividades dentro de Seed(). Em caso de falha, a transação completa (mesmo os patches aplicados) está sendo revertida.

Uma função de exemplo que cria dados somente se uma tabela estiver vazia pode ter esta aparência:

protected override void Seed(Model.DatabaseContext context)
{
    if (!context.Customers.Any()) {
        Customer c = new Customer{ Id = 1, Name = "Demo" };
        context.Customers.Add(c);
        context.SaveData();
    }
}

Um bom recurso fornecido pelos desenvolvedores do EF é o método de extensão AddOrUpdate(). Este método permite atualizar dados com base na chave primária ou inserir dados se ainda não existir (o exemplo é retirado do código-fonte gerado de Configuration.cs):

protected override void Seed(Model.DatabaseContext context)
{
    context.People.AddOrUpdate(
      p => p.FullName,
      new Person { FullName = "Andrew Peters" },
      new Person { FullName = "Brice Lambson" },
      new Person { FullName = "Rowan Miller" }
    );
}

Por favor, esteja ciente de que Seed() é chamado após a aplicação do último patch. Se você precisar migrar ou propagar dados durante os patches, outras abordagens precisam ser usadas.

Fazendo “Update-Database” dentro do seu código #

Os aplicativos executados em ambientes que não são de desenvolvimento geralmente exigem atualizações de banco de dados. Após usar o comando Add-Migration para criar seus patches de banco de dados, há a necessidade de executar as atualizações em outros ambientes e, em seguida, no ambiente de teste também.

Os desafios comumente enfrentados são:

  • nenhum Visual Studio instalado em ambientes de produção e
  • nenhuma conexão permitida para ambientes de conexão/cliente na vida real.

Uma solução alternativa é a seguinte sequência de código que verifica as atualizações a serem executadas e as executa em ordem. Certifique-se de transações adequadas e tratamento de exceções para garantir que nenhum dado seja perdido em caso de erros.

void UpdateDatabase(MyDbConfiguration configuration) {
   DbMigrator dbMigrator = new DbMigrator( configuration);
   if ( dbMigrator.GetPendingMigrations().Any() )
   {
      // there are pending migrations run the migration job
      dbMigrator.Update(); 
   }
}

onde MyDbConfiguration é sua configuração de migração em algum lugar em suas fontes:

public class MyDbConfiguration : DbMigrationsConfiguration<ApplicationDbContext>

Usando Sql() durante migrações #

Por exemplo: Você vai migrar uma coluna existente de não obrigatória para obrigatória. Neste caso, você pode precisar preencher alguns valores padrão em sua migração para linhas onde os campos alterados são realmente NULL. Caso o valor padrão seja simples (por exemplo, “0”), você pode usar uma propriedade default ou defaultSql em sua definição de coluna. Caso não seja tão fácil, você pode usar a função Sql() nas funções membro Up() ou Down() de suas migrações.

Aqui está um exemplo. Assumindo uma classe Author que contém um endereço de e-mail como parte do conjunto de dados. Agora decidimos ter o endereço de e-mail como um campo obrigatório. Para migrar colunas existentes, a empresa tem a ideia inteligente de criar endereços de e-mail fictícios como fullname@example.com, onde o nome completo é o nome completo do autor sem espaços. Adicionar o atributo [Required] ao campo Email criaria a seguinte migração:

public partial class AuthorsEmailRequired : DbMigration
{
    public override void Up()
    {
        AlterColumn("dbo.Authors", "Email", c => c.String(nullable: false, maxLength: 512));
    }
    
    public override void Down()
    {
        AlterColumn("dbo.Authors", "Email", c => c.String(maxLength: 512));
    }
}

Isso falharia caso alguns campos NULL estivessem dentro do banco de dados:

Não é possível inserir o valor NULL na coluna ‘Email’, tabela ‘App.Model.DatabaseContext.dbo.Authors’; coluna não permite nulos. A ATUALIZAÇÃO falha.

Adicionar o seguinte como antes do comando AlterColumn ajudará:

Sql(@"Update dbo.Authors
    set Email = REPLACE(name, ' ', '') + N'@example.com'
    where Email is null");

Outro uso #

Você pode usar a função Sql() para todos os tipos de atividades DML e DDL em seu banco de dados. Ele é executado como parte da transação de migração; Se o SQL falhar, a migração completa falha e uma reversão é feita.

Primeira migração de código de estrutura de entidade inicial passo a passo #

  1. Crie um aplicativo de console.

  2. Instale o pacote nuget EntityFramework executando Install-Package EntityFramework no “Console do Gerenciador de Pacotes”

  3. Adicione sua string de conexão no arquivo app.config , é importante incluir providerName="System.Data.SqlClient" em sua conexão.

  4. Crie uma classe pública como desejar, algo como “Blog”

  5. Crie sua ContextClass que herda de DbContext , algo como “BlogContext”

  6. Defina uma propriedade em seu contexto do tipo DbSet, algo assim:

    public class Blog { public int Id { get; set; }

     public string Name { get; set; }

    } public class BlogContext: DbContext { public BlogContext(): base(“name=Your_Connection_Name”) { }

     public virtual DbSet<Blog> Blogs{ get; set; }

    }

  7. É importante passar o nome da conexão no construtor (aqui Your_Connection_Name)

  8. No Console do Gerenciador de Pacotes, execute o comando Enable-Migration, Isso criará uma pasta de migração em seu projeto

  9. Execute o comando Add-Migration Your_Arbitrary_Migraiton_Name, isso criará uma classe de migração na pasta migrations com dois métodos Up() e Down()

  10. Execute o comando Update-Database para criar um banco de dados com uma tabela de blog
