Convenção é um conjunto de regras padrão para configurar automaticamente um modelo conceitual baseado em definições de classe de domínio ao trabalhar com Code-First. As convenções Code-First são definidas no namespace System.Data.Entity.ModelConfiguration.Conventions (EF 5 & EF 6).

Removendo convenções #

Você pode remover qualquer uma das convenções definidas no namespace System.Data.Entity.ModelConfiguration.Conventions, substituindo o método OnModelCreating.

O exemplo a seguir remove PluralizingTableNameConvention.

public class EshopContext : DbContext 
{ 
    public DbSet<Product> Products { set; get; }
    . . . 
 
    protected override void OnModelCreating(DbModelBuilder modelBuilder) 
    { 
        modelBuilder.Conventions.Remove<PluralizingTableNameConvention>(); 
    } 
}

Por padrão, o EF criará uma tabela de banco de dados com o nome da classe de entidade com o sufixo ’s’. Neste exemplo, Code First está configurado para ignorar a convenção PluralizingTableName para que, em vez da tabela dbo.Products, a tabela dbo.Product seja criada.

Convenção de chave primária #

Por padrão, uma propriedade é uma chave primária se uma propriedade em uma classe for denominada “ID” (não diferencia maiúsculas de minúsculas) ou o nome da classe seguido por “ID”. Se o tipo da propriedade de chave primária for numérico ou GUID, ela será configurada como uma coluna de identidade. Exemplo Simples:

public class Room
{ 
    // Primary key 
    public int RoomId{ get; set; } 
    ... 
}

Descoberta de tipos #

Por padrão, o Code First inclui no modelo

  1. Tipos definidos como propriedade DbSet na classe de contexto.
  2. Tipos de referência incluídos em tipos de entidade, mesmo que sejam definidos em different assembly.
  3. Classes derivadas mesmo que apenas a classe base seja definida como DbSet property

Aqui está um exemplo, que estamos adicionando apenas Company como DbSet<Company> em nossa classe de contexto:

public class Company
{
    public int Id { set; get; }
    public string Name { set; get; }
    public virtual ICollection<Department> Departments { set; get; }
}

public class Department
{
    public int Id { set; get; }
    public string Name { set; get; }
    public virtual ICollection<Person> Staff { set; get; }
}

[Table("Staff")]
public class Person
{
    public int Id { set; get; }
    public string Name { set; get; }
    public decimal Salary { set; get; }
}

public class ProjectManager : Person
{
   public string ProjectManagerProperty { set; get; }
}

public class Developer : Person
{
    public string DeveloperProperty { set; get; }
}

public class Tester : Person
{
    public string TesterProperty { set; get; }
}    

public class ApplicationDbContext : DbContext
{
    public DbSet<Company> Companies { set; get; }
}

Podemos ver que todas as classes estão incluídas no modelo

DecimalPropertyConvention #

Por padrão, o Entity Framework mapeia propriedades decimais para colunas decimais(18,2) em tabelas de banco de dados.

public class Box
{
    public int Id { set; get; }
    public decimal Length { set; get; }
    public decimal Width { set; get; }
    public decimal Height { set; get; }
}

Podemos alterar a precisão das propriedades decimais:

  1. Use a API Fluente:

    protected override void OnModelCreating(DbModelBuilder modelBuilder) { modelBuilder.Entity().Property(b => b.Width).HasPrecision(20, 4); }

Somente a propriedade “Width” é mapeada para decimal(20, 4).

  1. Substitua a convenção:

    protected override void OnModelCreating(DbModelBuilder modelBuilder) { modelBuilder.Conventions.Remove(); modelBuilder.Conventions.Add(new DecimalPropertyConvention(10, 4)); }

Cada propriedade decimal é mapeada para colunas decimais (10,4).

Convenção de Relacionamento #

Code First inferir o relacionamento entre as duas entidades usando a propriedade de navegação. Essa propriedade de navegação pode ser um tipo de referência simples ou um tipo de coleção. Por exemplo, definimos a propriedade de navegação Standard na classe Student e a propriedade de navegação ICollection na classe Standard. Assim, o Code First criou automaticamente um relacionamento um-para-muitos entre a tabela Standards e Students DB inserindo a coluna de chave estrangeira Standard_StandardId na tabela Students.

public class Student
{
    
    public int StudentID { get; set; }
    public string StudentName { get; set; }
    public DateTime DateOfBirth { get; set; }      
        
    //Navigation property
    public Standard Standard { get; set; }
}

public class Standard
{
   
    public int StandardId { get; set; }
    public string StandardName { get; set; }
    
    //Collection navigation property
    public IList<Student> Students { get; set; }
   
}

As entidades acima criaram o seguinte relacionamento usando a chave estrangeira Standard_StandardId.

Convenção de chave estrangeira #

Se a classe A estiver em relação com a classe B e a classe B tiver uma propriedade com o mesmo nome e tipo que a chave primária de A, o EF automaticamente assume que a propriedade é uma chave estrangeira.

public class Department
{
    public int DepartmentId { set; get; }
    public string Name { set; get; }
    public virtual ICollection<Person> Staff { set; get; }
}

public class Person
{
    public int Id { set; get; }
    public string Name { set; get; }
    public decimal Salary { set; get; }
    public int DepartmentId { set; get; }
    public virtual Department Department { set; get; }
}

Nesse caso, DepartmentId é uma chave estrangeira sem especificação explícita.
