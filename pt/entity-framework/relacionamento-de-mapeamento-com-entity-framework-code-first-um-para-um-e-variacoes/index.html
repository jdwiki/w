Este tópico discute como mapear relacionamentos de tipo um para um usando o Entity Framework.

Mapeamento de um a zero ou um #

Então, digamos novamente que você tem o seguinte modelo:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
}

public class MyDemoContext : DbContext
{
  public DbSet<Person> People { get; set; }
  public DbSet<Car> Cars { get; set; }
}

E agora você quer configurá-lo para que possa expressar a seguinte especificação: uma pessoa pode ter um ou zero carro, e cada carro pertence a uma pessoa exatamente (os relacionamentos são bidirecionais, então se CarA pertence a PersonA, então PersonA ‘possui ‘CarA).

Então vamos modificar um pouco o modelo: adicione as propriedades de navegação e as propriedades de chave estrangeira:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
  public int CarId { get; set; }
  public virtual Car Car { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

E a configuração:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);                        
  }
}

A essa altura, isso deve ser autoexplicativo. O carro tem uma pessoa obrigatória (HasRequired()), com a pessoa tendo um carro opcional (WithOptional()). Novamente, não importa de que lado você configura esse relacionamento, apenas tome cuidado ao usar a combinação certa de Has/With e Required/Optional. Do lado Person, ficaria assim:

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
     this.HasOptional(p => p.Car).WithOptional(c => c.Person);                        
  }
}

Agora vamos verificar o esquema db:

Olhe atentamente: você pode ver que não há FK em ‘Pessoas’ para se referir a ‘Carro’. Além disso, o FK em Car não é o PersonId, mas o CarId. Aqui está o script real para o FK:

ALTER TABLE [dbo].[Cars]  WITH CHECK ADD  CONSTRAINT [FK_dbo.Cars_dbo.People_CarId] FOREIGN KEY([CarId])
REFERENCES [dbo].[People] ([PersonId])

Portanto, isso significa que as propriedades de chave estrangeira CarId e PersonId que temos no modelo são basicamente ignoradas. Eles estão no banco de dados, mas não são chaves estrangeiras, como seria de esperar. Isso ocorre porque os mapeamentos um para um não dão suporte à adição do FK ao seu modelo EF. E isso porque os mapeamentos um para um são bastante problemáticos em um banco de dados relacional.

A ideia é que cada pessoa pode ter exatamente um carro, e esse carro só pode pertencer a essa pessoa. Ou pode haver registros de pessoas, que não têm carros associados a eles.

Então, como isso poderia ser representado com chaves estrangeiras? Obviamente, pode haver um PersonId em Car e um CarId em People. Para garantir que cada pessoa possa ter apenas um carro, PersonId teria que ser único em Car. Mas se PersonId for único em People, então como você pode adicionar dois ou mais registros onde PersonId é NULL(mais de um carro que não tem proprietários)? Resposta: você não pode (bem, na verdade, você pode criar um índice exclusivo filtrado no SQL Server 2008 e mais recente, mas vamos esquecer esse detalhe técnico por um momento; para não mencionar outros RDBMS). Sem falar no caso em que você especifica as duas pontas do relacionamento…

A única maneira real de aplicar esta regra se as tabelas People e Car tiverem a ‘mesma’ chave primária (mesmos valores nos registros conectados). E para fazer isso, CarId em Car deve ser tanto um PK quanto um FK para o PK de People. E isso torna todo o esquema uma bagunça. Quando uso isso, prefiro nomear o PK/FK em Car PersonId e configurá-lo de acordo:

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }        
  public virtual Car Car { get; set; }
}

public class Car
{        
  public string LicensePlate { get; set; }
  public int PersonId { get; set; }
  public virtual Person Person { get; set; }
}

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Person).WithOptional(p => p.Car);
     this.HasKey(c => c.PersonId);
  }
}

Não é o ideal, mas talvez um pouco melhor. Ainda assim, você deve estar alerta ao usar esta solução, pois ela vai contra as convenções de nomenclatura usuais, o que pode levá-lo ao erro. Aqui está o esquema gerado a partir deste modelo:

Portanto, esse relacionamento não é imposto pelo esquema do banco de dados, mas pelo próprio Entity Framework. É por isso que você tem que ter muito cuidado ao usar isso, para não deixar ninguém temperar diretamente com o banco de dados.

Mapeamento um para um #

Mapear um para um (quando ambos os lados são necessários) também é uma coisa complicada.

Vamos imaginar como isso poderia ser representado com chaves estrangeiras. Novamente, um CarId em People que se refere a CarId em Car, e um PersonId em Car que se refere ao PersonId em People.

Agora, o que acontece se você quiser inserir um registro de carro? Para que isso seja bem-sucedido, deve haver um PersonId especificado neste registro de carro, porque é obrigatório. E para que este PersonId seja válido, o registro correspondente em People deve existir. OK, então vamos em frente e inserir o registro de pessoa. Mas para que isso seja bem-sucedido, um CarId válido deve estar no registro de pessoa — mas esse carro ainda não está inserido! Não pode ser, pois temos que inserir primeiro o registro da pessoa referida. Mas não podemos inserir o registro da pessoa referida, pois ele se refere ao registro do carro, de modo que deve ser inserido primeiro (ceção de chave estrangeira :) ).

Portanto, isso também não pode ser representado da maneira ’lógica’. Novamente, você deve descartar uma das chaves estrangeiras. Qual você solta é com você. O lado que fica com uma chave estrangeira é chamado de ‘dependente’, o lado que fica sem uma chave estrangeira é chamado de ‘principal’. E, novamente, para garantir a exclusividade no dependente, o PK deve ser o FK, portanto, adicionar uma coluna FK e importá-la para o seu modelo não é suportado.

Então aqui está a configuração:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasRequired(c => c.Person).WithRequiredDependent(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Até agora você realmente deve ter entendido a lógica disso :) Apenas lembre-se que você pode escolher o outro lado também, apenas tome cuidado para usar as versões Dependente/Principal do WithRequired (e você ainda tem que configurar o PK no carro).

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
    this.HasRequired(p => p.Car).WithRequiredPrincipal(c => c.Person);
  }
}

Se você verificar o esquema de banco de dados, verá que é exatamente o mesmo que era no caso da solução um para um ou zero. Isso porque, novamente, isso não é imposto pelo esquema, mas pelo próprio EF. Então, mais uma vez, tenha cuidado :)

Mapeando um ou zero-para-um ou zero #

E para finalizar, vejamos brevemente o caso em que ambos os lados são opcionais.

Até agora você deve estar realmente entediado com esses exemplos :), então não vou entrar em detalhes e brincar com a ideia de ter dois FK-s e os possíveis problemas e alertá-lo sobre os perigos de não aplicar essas regras no esquema, mas apenas no próprio EF.

Aqui está a configuração que você precisa aplicar:

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
    this.HasOptional(c => c.Person).WithOptionalPrincipal(p => p.Car);
    this.HasKey(c => c.PersonId);
  }
}

Novamente, você também pode configurar do outro lado, apenas tome cuidado para usar os métodos corretos :)
