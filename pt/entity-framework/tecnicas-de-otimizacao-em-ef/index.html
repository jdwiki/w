Usando AsNoTracking #

Mau exemplo:

var location =  dbContext.Location
                     .Where(l => l.Location.ID == location_ID)
                     .SingleOrDefault();

return location;

Como o código acima está simplesmente retornando uma entidade sem modificá-la ou adicioná-la, podemos evitar o custo de rastreamento.

Bom exemplo:

var location =  dbContext.Location.AsNoTracking()
                     .Where(l => l.Location.ID == location_ID)
                     .SingleOrDefault();

return location;

Quando usamos a função AsNoTracking() estamos dizendo explicitamente ao Entity Framework que as entidades não são rastreadas pelo contexto. Isso pode ser especialmente útil ao recuperar grandes quantidades de dados de seu armazenamento de dados. No entanto, se você quiser fazer alterações em entidades não rastreadas, lembre-se de anexá-las antes de chamar SaveChanges.

Carregando apenas os dados necessários #

Um problema frequentemente visto no código é carregar todos os dados. Isso aumentará muito a carga no servidor.

Digamos que eu tenha um modelo chamado “local” que tenha 10 campos, mas nem todos os campos são obrigatórios ao mesmo tempo. Digamos que eu queira apenas o parâmetro ‘LocationName’ desse modelo.

Mau exemplo

var location =  dbContext.Location.AsNoTracking()
                         .Where(l => l.Location.ID == location_ID)
                         .SingleOrDefault();

return location.Name;

Bom exemplo

var location =  dbContext.Location
                          .Where(l => l.Location.ID == location_ID)
                          .Select(l => l.LocationName);
                          .SingleOrDefault();

return location;

O código no “bom exemplo” buscará apenas ‘LocationName’ e nada mais.

Observe que, como nenhuma entidade é materializada neste exemplo, AsNoTracking() não é necessário. Não há nada para ser rastreado de qualquer maneira.

Buscando mais campos com tipos anônimos

var location = dbContext.Location
                    .Where(l => l.Location.ID == location_ID)
                    .Select(l => new { Name = l.LocationName, Area = l.LocationArea })
                    .SingleOrDefault();

return location.Name + " has an area of " + location.Area;

Igual ao exemplo anterior, apenas os campos ‘LocationName’ e ‘LocationArea’ serão recuperados do banco de dados, o Tipo Anônimo pode conter quantos valores você desejar.

Execute consultas no banco de dados quando possível, não na memória. #

Suponha que queremos contar quantos condados existem no Texas:

var counties = dbContext.States.Single(s => s.Code == "tx").Counties.Count();

A consulta está correta, mas ineficiente. States.Single(…) carrega um estado do banco de dados. Em seguida, Counties carrega todos os 254 condados com todos os seus campos em uma segunda consulta. .Count() é então executado na memória na coleção Counties carregada. Carregamos muitos dados de que não precisamos e podemos fazer melhor:

var counties = dbContext.Counties.Count(c => c.State.Code == "tx");

Aqui fazemos apenas uma consulta, que em SQL se traduz em uma contagem e uma junção. Retornamos apenas a contagem do banco de dados - salvamos o retorno de linhas, campos e criação de objetos.

É fácil ver onde a consulta é feita observando o tipo de coleção: IQueryable<T> vs. IEnumerable<T>.

Executar várias consultas assíncronas e em paralelo #

Ao usar consultas assíncronas, você pode executar várias consultas ao mesmo tempo, mas não no mesmo contexto. Se o tempo de execução de uma consulta for 10s, o tempo para o mau exemplo será 20s, enquanto o tempo para o bom exemplo será 10s.

Mau exemplo #

IEnumerable<TResult1> result1;
IEnumerable<TResult2> result2;

using(var context = new Context())
{
    result1 = await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
    result2 = await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
}

Bom exemplo #

public async Task<IEnumerable<TResult>> GetResult<TResult>()
{
    using(var context = new Context())
    {
        return await context.Set<TResult1>().ToListAsync().ConfigureAwait(false);
    }
}



IEnumerable<TResult1> result1;
IEnumerable<TResult2> result2;

var result1Task = GetResult<TResult1>();
var result2Task = GetResult<TResult2>();

await Task.WhenAll(result1Task, result2Task).ConfigureAwait(false);

var result1 = result1Task.Result;
var result2 = result2Task.Result;

Trabalhando com entidades de stub #

Digamos que temos Produtos e Categorias em um relacionamento muitos-para-muitos:

public class Product
{
    public Product()
    {
        Categories = new HashSet<Category>();
    }
    public int ProductId { get; set; }
    public string ProductName { get; set; }
    public virtual ICollection<Category> Categories { get; private set; }
}

public class Category
{
    public Category()
    {
        Products = new HashSet<Product>();
    }
    public int CategoryId { get; set; }
    public string CategoryName { get; set; }
    public virtual ICollection<Product> Products { get; set; }
}

Se quisermos adicionar uma Categoria a um Produto, temos que carregar o produto e adicionar a categoria às suas Categorias, por exemplo:

Mau exemplo:

var product = db.Products.Find(1);
var category = db.Categories.Find(2);
product.Categories.Add(category);
db.SaveChanges();

(onde db é uma subclasse DbContext).

Isso cria um registro na tabela de junção entre Product e Category. No entanto, esta tabela contém apenas dois valores Id. É um desperdício de recursos carregar duas entidades completas para criar um pequeno registro.

Uma maneira mais eficiente é usar entidades de stub, ou seja, objetos de entidade, criados na memória, contendo apenas o mínimo de dados, geralmente apenas um valor ‘Id’. Isto é o que parece:

Bom exemplo:

// Create two stub entities
var product = new Product { ProductId = 1 };
var category = new Category { CategoryId = 2 };

// Attach the stub entities to the context
db.Entry(product).State = System.Data.Entity.EntityState.Unchanged;
db.Entry(category).State = System.Data.Entity.EntityState.Unchanged;

product.Categories.Add(category);
db.SaveChanges();

O resultado final é o mesmo, mas evita duas idas e voltas ao banco de dados.

Evitar duplicatas

Se você quiser verificar se a associação já existe, basta uma consulta barata. Por exemplo:

var exists = db.Categories.Any(c => c.Id == 1 && c.Products.Any(p => p.Id == 14));

Novamente, isso não carregará entidades completas na memória. Ele efetivamente consulta a tabela de junção e retorna apenas um booleano.

Desative o rastreamento de alterações e a geração de proxy #

Se você deseja apenas obter dados, mas não modificar nada, pode desativar o rastreamento de alterações e a criação de proxy. Isso melhorará seu desempenho e também evitará o carregamento lento.

Mau exemplo:

using(var context = new Context())
{
    return await context.Set<MyEntity>().ToListAsync().ConfigureAwait(false);
}

Bom exemplo:

using(var context = new Context())
{
    context.Configuration.AutoDetectChangesEnabled = false;
    context.Configuration.ProxyCreationEnabled = false;

    return await context.Set<MyEntity>().ToListAsync().ConfigureAwait(false);
}

É particularmente comum desativá-los de dentro do construtor do seu contexto, especialmente se você deseja que eles sejam definidos em sua solução:

public class MyContext : DbContext
{
    public MyContext()
        : base("MyContext")
    {
        Configuration.AutoDetectChangesEnabled = false;
        Configuration.ProxyCreationEnabled = false;
    }

    //snip
}
