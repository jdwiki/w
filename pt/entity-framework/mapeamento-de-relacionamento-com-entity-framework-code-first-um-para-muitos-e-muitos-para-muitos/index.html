O tópico discute como você pode mapear relacionamentos um para muitos e muitos para muitos usando o Entity Framework Code First.

Mapeamento de um para muitos #

Então, digamos que você tenha duas entidades diferentes, algo assim:

public class Person
{
   public int PersonId { get; set; }
   public string Name { get; set; }
}

public class Car
{
   public int CarId { get; set; }
   public string LicensePlate { get; set; }
}

public class MyDemoContext : DbContext
{
   public DbSet<Person> People { get; set; }
   public DbSet<Car> Cars { get; set; }
}

E você quer configurar um relacionamento um-para-muitos entre eles, ou seja, uma pessoa pode ter zero, um ou mais carros, e um carro pertence exatamente a uma pessoa. Todo relacionamento é bidirecional, então se uma pessoa tem um carro, o carro pertence a essa pessoa.

Para fazer isso, basta modificar suas classes de modelo:

public class Person
{
   public int PersonId { get; set; }
   public string Name { get; set; }
   public virtual ICollection<Car> Cars { get; set; } // don't forget to initialize (use HashSet)
}

public class Car
{
   public int CarId { get; set; }
   public string LicensePlate { get; set; }
   public int PersonId { get; set; }
   public virtual Person Person { get; set; }
}

E é isso :) Você já tem seu relacionamento configurado. No banco de dados, isso é representado com chaves estrangeiras, é claro.

Mapeamento de um para muitos: contra a convenção #

No último exemplo, você pode ver que o EF descobre qual coluna é a chave estrangeira e para onde ela deve apontar. Como? Usando convenções. Ter uma propriedade do tipo Person chamada Person com uma propriedade PersonId leva o EF a concluir que PersonId é uma chave estrangeira e aponta para a chave primária da tabela representada pelo tipo Person .

Mas e se você mudar PersonId para OwnerId e Person para Owner no tipo Car?

public class Car
{
   public int CarId { get; set; }
   public string LicensePlate { get; set; }
   public int OwnerId { get; set; }
   public virtual Person Owner { get; set; }
}

Bem, infelizmente neste caso, as convenções não são suficientes para produzir o esquema de banco de dados correto:

Sem problemas; você pode ajudar a EF com algumas dicas sobre seus relacionamentos e chaves no modelo. Basta configurar seu tipo Car para usar a propriedade OwnerId como o FK. Crie uma configuração de tipo de entidade e aplique-a em seu OnModelCreating():

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasRequired(c => c.Owner).WithMany(p => p.Cars).HasForeignKey(c => c.OwnerId);
  }
}

Isso basicamente diz que Car tem uma propriedade necessária, Owner (HasRequired() ) e no tipo de Owner, a propriedade Cars é usada para fazer referência às entidades de carro ([WithMany()](https://msdn.microsoft.com/en-us/library/gg696499( v=vs.113).aspx)). E, finalmente, a propriedade que representa a chave estrangeira é especificada (HasForeignKey()). Isso nos dá o esquema que queremos:

Você também pode configurar o relacionamento do lado Person:

public class PersonEntityTypeConfiguration : EntityTypeConfiguration<Person>
{
  public PersonEntityTypeConfiguration()
  {
    this.HasMany(p => p.Cars).WithRequired(c => c.Owner).HasForeignKey(c => c.OwnerId);
  }
}

A ideia é a mesma, apenas os lados são diferentes (observe como você pode ler a coisa toda: ’essa pessoa tem muitos carros, cada carro com um dono obrigatório’). Não importa se você configura o relacionamento do lado Pessoa ou do lado Carro. Você pode até incluir os dois, mas neste caso tenha o cuidado de especificar a mesma relação em ambos os lados!

Mapeamento zero ou um para muitos #

Nos exemplos anteriores, um carro não pode existir sem uma pessoa. E se você quisesse que a pessoa fosse opcional do lado do carro? Bem, é meio fácil, saber fazer um-para-muitos. Basta alterar o PersonId em Car para ser anulável:

public class Car
{
    public int CarId { get; set; }
    public string LicensePlate { get; set; }
    public int? PersonId { get; set; }
    public virtual Person Person { get; set; }
}

E, em seguida, use o HasOptional() (ou [WithOptional()](https:/ /msdn.microsoft.com/en-us/library/gg696231(v=vs.113).aspx), dependendo de qual lado você faz a configuração):

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
  public CarEntityTypeConfiguration()
  {
     this.HasOptional(c => c.Owner).WithMany(p => p.Cars).HasForeignKey(c => c.OwnerId);
  }
}

Muitos para muitos #

Vamos passar para o outro cenário, onde cada pessoa pode ter vários carros e cada carro pode ter vários proprietários (mas, novamente, a relação é bidirecional). Esta é uma relação de muitos para muitos. A maneira mais fácil é deixar o EF fazer sua mágica usando convenções.

Basta alterar o modelo assim:

 public class Person
{
   public int PersonId { get; set; }
   public string Name { get; set; }
   public virtual ICollection<Car> Cars { get; set; }
}

public class Car
{
   public int CarId { get; set; }
   public string LicensePlate { get; set; }        
   public virtual ICollection<Person> Owners { get; set; }
}

Quase perfeito. Como você pode ver, a EF reconheceu a necessidade de uma tabela de junção, onde você pode acompanhar os pares pessoa-carro.

Muitos para muitos: personalizando a tabela de junção #

Você pode querer renomear os campos na tabela de junção para serem um pouco mais amigáveis. Você pode fazer isso usando os métodos de configuração usuais (novamente, não importa de que lado você faz a configuração):

public class CarEntityTypeConfiguration : EntityTypeConfiguration<Car>
{
   public CarEntityTypeConfiguration()
   {
      this.HasMany(c => c.Owners).WithMany(p => p.Cars)
          .Map(m =>
              {
                 m.MapLeftKey("OwnerId");
                 m.MapRightKey("CarId");
                 m.ToTable("PersonCars");
              }
        );
  }
}

Muito fácil de ler mesmo: este carro tem muitos proprietários (HasMany()), com cada proprietário com muitos carros (WithMany()). Mapeie isso para mapear a chave esquerda para OwnerId ([MapLeftKey()](https://msdn.microsoft.com/en-us/library/system.data.entity.modelconfiguration.configuration.manytomanyassociationmappingconfiguration.mapleftkey(v =vs.113).aspx)), a chave certa para CarId ([MapRightKey()](https://msdn.microsoft.com/en-us/library/system.data.entity.modelconfiguration.configuration.manytomanyassociationmappingconfiguration .maprightkey(v=vs.113).aspx)) e tudo para a tabela PersonCars ([ToTable()](https://msdn.microsoft.com/en-us/library/gg679488(v=vs .113).aspx)). Muitos para muitos: entidade de junção personalizada #

Eu tenho que admitir, não sou muito fã de deixar o EF inferir a tabela de junção sem uma entidade de junção. Você não pode rastrear informações extras para uma associação pessoa-carro (digamos, a data a partir da qual ela é válida), porque você não pode modificar a tabela.

Além disso, o CarId na tabela de junção faz parte da chave primária, portanto, se a família comprar um carro novo, você deve primeiro excluir as associações antigas e adicionar novas. O EF oculta isso de você, mas isso significa que você precisa fazer essas duas operações em vez de uma simples atualização (sem mencionar que inserções/exclusões frequentes podem levar à fragmentação do índice — coisa boa [há uma correção fácil](http: //www.dotnetfalcon.com/azure-automation-job-for-index-maintenance/) para isso).

Nesse caso, o que você pode fazer é criar uma entidade de junção que tenha uma referência a um carro específico e a uma pessoa específica. Basicamente, você vê sua associação de muitos para muitos como uma combinação de duas associações de um para muitos:

public class PersonToCar
{
   public int PersonToCarId { get; set; }
   public int CarId { get; set; }
   public virtual Car Car { get; set; }
   public int PersonId { get; set; }
   public virtual Person Person { get; set; }
   public DateTime ValidFrom { get; set; }
}

public class Person
{
  public int PersonId { get; set; }
  public string Name { get; set; }
  public virtual ICollection<PersonToCar> CarOwnerShips { get; set; }
}

public class Car
{
  public int CarId { get; set; }
  public string LicensePlate { get; set; }        
  public virtual ICollection<PersonToCar> Ownerships { get; set; }
}

public class MyDemoContext : DbContext
{
  public DbSet<Person> People { get; set; }
  public DbSet<Car> Cars { get; set; }
  public DbSet<PersonToCar> PersonToCars { get; set; }
}

Isso me dá muito mais controle e é muito mais flexível. Agora posso adicionar dados personalizados à associação e cada associação tem sua própria chave primária, para que eu possa atualizar o carro ou a referência do proprietário nelas.

Observe que isso realmente é apenas uma combinação de dois relacionamentos um-para-muitos, portanto, você pode usar todas as opções de configuração discutidas nos exemplos anteriores.
