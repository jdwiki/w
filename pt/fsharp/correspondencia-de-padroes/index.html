A correspondência de padrões é um recurso poderoso de muitas linguagens funcionais, pois geralmente permite que a ramificação seja tratada de forma muito sucinta em comparação com o uso de várias instruções de estilo if/else if/else. No entanto, com opções suficientes e “quando” guardas, a correspondência de padrões também pode se tornar detalhada e difícil de entender de relance.

Quando isso acontece, os Padrões Ativos do F# podem ser uma ótima maneira de dar nomes significativos à lógica de correspondência, o que simplifica o código e também permite a reutilização.

A correspondência de padrões verifica se todo o domínio está coberto #

let x = true
match x with
| true -> printfn "x is true"

gera um aviso #

C:\Program Files (x86)\Microsoft VS Code\Untitled-1(2,7): aviso FS0025: Correspondências de padrão incompletas nesta expressão. Por exemplo, o valor ‘false’ pode indicar um caso não coberto pelo(s) padrão(ões).

Isso ocorre porque nem todos os valores booleanos possíveis foram cobertos.

bools podem ser listados explicitamente, mas os ints são mais difíceis de listar #

let x = 5
match x with
| 1 -> printfn "x is 1"
| 2 -> printfn "x is 2"
| _ -> printfn "x is something else"

aqui usamos o caractere especial _. O _ corresponde a todos os outros casos possíveis.

O _ pode causar problemas #

considere um tipo que nós mesmos criamos, ele se parece com isso

type Sobriety = 
    | Sober
    | Tipsy
    | Drunk

Podemos escrever uma correspondência com expressão que se pareça com isso

match sobriety with
| Sober -> printfn "drive home"
| _ -> printfn "call an uber"

O código acima faz sentido. Estamos assumindo que, se você não estiver sóbrio, deve chamar um uber, então usamos o _ para denotar que

Mais tarde, refatoramos nosso código para isso

type Sobriety = 
    | Sober
    | Tipsy
    | Drunk
    | Unconscious

O compilador F# deve nos dar um aviso e solicitar que refatoremos nossa expressão de correspondência para que a pessoa procure atendimento médico. Em vez disso, a expressão do jogo silenciosamente trata a pessoa inconsciente como se ela estivesse apenas embriagada. O ponto é que você deve optar por listar explicitamente os casos quando possível para evitar erros de lógica.

Quando os guardas permitem adicionar condicionais arbitrárias #

type Person = {
    Age : int
    PassedDriversTest : bool }

let someone = { Age = 19; PassedDriversTest = true }

match someone.PassedDriversTest with
| true when someone.Age >= 16 -> printfn "congrats"
| true -> printfn "wait until you are 16"
| false -> printfn "you need to pass the test"

Os casos são avaliados de cima para baixo e a primeira correspondência é usada #

Uso incorreto:

No snippet a seguir, a última correspondência nunca será usada:

let x = 4
match x with
| 1 -> printfn "x is 1"
| _ -> printfn "x is anything that wasn't listed above"
| 4 -> printfn "x is 4"

impressões

x é qualquer coisa que não foi listada acima

Modo de usar correto:

Aqui, tanto x = 1 quanto x = 4 atingirão seus casos específicos, enquanto todo o resto cairá no caso padrão _:

let x = 4
match x with
| 1 -> printfn "x is 1"
| 4 -> printfn "x is 4"
| _ -> printfn "x is anything that wasn't listed above"

impressões

x é 4

Opções de correspondência #

A correspondência de padrões pode ser útil para lidar com Opções:

let result = Some("Hello World")
match result with
| Some(message) -> printfn message
| None -> printfn "Not feeling talkative huh?"
