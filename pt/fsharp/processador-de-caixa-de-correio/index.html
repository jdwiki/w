Processador de caixa de correio

O MailboxProcessor mantém uma fila de mensagens interna, onde vários produtores podem postar mensagens usando várias variantes do método Post. Essas mensagens são então recuperadas e processadas por um único consumidor (a menos que você implemente de outra forma) usando as variantes Retrieve e Scan. Por padrão, tanto a produção quanto o consumo das mensagens são thread-safe.

Por padrão, não há tratamento de erros fornecido. Se uma exceção não capturada for lançada dentro do corpo do processador, a função do corpo será encerrada, todas as mensagens na fila serão perdidas, nenhuma mensagem poderá ser postada e o canal de resposta (se disponível) receberá uma exceção em vez de uma resposta. Você deve fornecer todo o tratamento de erros caso esse comportamento não seja adequado ao seu caso de uso.

Básico Olá Mundo #

Vamos primeiro criar um simples “Hello world!” MailboxProcessor que processa um tipo de mensagem e imprime saudações.

Você precisará do tipo de mensagem. Pode ser qualquer coisa, mas Sindicatos Discriminados são uma escolha natural aqui, pois listam todos os casos possíveis em um só lugar e você pode facilmente usar a correspondência de padrões ao processá-los.

// In this example there is only a single case, it could technically be just a string
type GreeterMessage = SayHelloTo of string

Agora defina o próprio processador. Isso pode ser feito com o método estático MailboxProcessor<'message>.Start que retorna um processador iniciado pronto para fazer seu trabalho. Você também pode usar o construtor, mas precisa ter certeza de iniciar o processador mais tarde.

let processor = MailboxProcessor<GreeterMessage>.Start(fun inbox ->
    let rec innerLoop () = async {
        // This way you retrieve message from the mailbox queue
        // or await them in case the queue empty.
        // You can think of the `inbox` parameter as a reference to self.
        let! message = inbox.Receive()
        // Now you can process the retrieved message.
        match message with
        | SayHelloTo name ->
            printfn "Hi, %s! This is mailbox processor's inner loop!" name
        // After that's done, don't forget to recurse so you can process the next messages!
        innerLoop()
    }
    innerLoop ())

O parâmetro para Start é uma função que faz referência ao próprio MailboxProcessor (que ainda não existe enquanto você o está criando, mas estará disponível assim que a função for executada). Isso lhe dá acesso aos vários métodos Receive e Scan para acessar as mensagens da caixa de correio. Dentro desta função, você pode fazer o processamento que precisar, mas uma abordagem usual é um loop infinito que lê as mensagens uma a uma e chama a si mesmo após cada uma.

Agora o processador está pronto, mas não adianta nada! Por quê? Você precisa enviar uma mensagem para processar. Isso é feito com as variantes do método Post - vamos usar o mais básico, disparar e esquecer.

processor.Post(SayHelloTo "Alice")

Isso coloca uma mensagem na fila interna do processador, a caixa de correio, e retorna imediatamente para que o código de chamada possa continuar. Depois que o processador recuperar a mensagem, ele a processará, mas isso será feito de forma assíncrona para publicá-la e provavelmente será feito em um encadeamento separado.

Logo em seguida você deverá ver a mensagem "Oi, Alice! Este é o loop interno do processador de caixa de correio!" impressa na saída e você estará pronto para amostras mais complicadas.

Gerenciamento de estado mutável #

Os processadores de caixa de correio podem ser usados ​​para gerenciar o estado mutável de maneira transparente e segura para threads. Vamos construir um contador simples.

// Increment or decrement by one.
type CounterMessage =
    | Increment
    | Decrement

let createProcessor initialState =
    MailboxProcessor<CounterMessage>.Start(fun inbox ->
        // You can represent the processor's internal mutable state
        // as an immutable parameter to the inner loop function
        let rec innerLoop state = async {
            printfn "Waiting for message, the current state is: %i" state
            let! message = inbox.Receive()
            // In each call you use the current state to produce a new
            // value, which will be passed to the next call, so that
            // next message sees only the new value as its local state
            match message with
            | Increment ->
                let state' = state + 1
                printfn "Counter incremented, the new state is: %i" state'
                innerLoop state'
            | Decrement ->
                let state' = state - 1
                printfn "Counter decremented, the new state is: %i" state'
                innerLoop state'
        }
        // We pass the initialState to the first call to innerLoop
        innerLoop initialState)

// Let's pick an initial value and create the processor
let processor = createProcessor 10

Agora vamos gerar algumas operações

processor.Post(Increment)
processor.Post(Increment)
processor.Post(Decrement)
processor.Post(Increment)

E você verá o seguinte log

Waiting for message, the current state is: 10
Counter incremented, the new state is: 11
Waiting for message, the current state is: 11
Counter incremented, the new state is: 12
Waiting for message, the current state is: 12
Counter decremented, the new state is: 11
Waiting for message, the current state is: 11
Counter incremented, the new state is: 12
Waiting for message, the current state is: 12

Simultaneidade #

Como o processador da caixa de correio processa as mensagens uma a uma e não há intercalação, você também pode produzir as mensagens de vários segmentos e não verá os problemas típicos de operações perdidas ou duplicadas. Não há como uma mensagem usar o estado antigo de outras mensagens, a menos que você implemente especificamente o processador.

let processor = createProcessor 0

[ async { processor.Post(Increment) }
  async { processor.Post(Increment) }
  async { processor.Post(Decrement) }
  async { processor.Post(Decrement) } ]
|> Async.Parallel
|> Async.RunSynchronously

Todas as mensagens são postadas de tópicos diferentes. A ordem na qual as mensagens são postadas na caixa de correio não é determinística, então a ordem de processamento não é determinística, mas como o número total de incrementos e decrementos é balanceado, você verá o estado final sendo 0, não importa em qual ordem e de quais threads as mensagens foram enviadas.

Verdadeiro estado mutável #

No exemplo anterior, apenas simulamos o estado mutável passando o parâmetro de loop recursivo, mas o processador de caixa de correio tem todas essas propriedades mesmo para um estado verdadeiramente mutável. Isso é importante quando você mantém um estado grande e a imutabilidade é impraticável por motivos de desempenho.

Podemos reescrever nosso contador para a seguinte implementação

let createProcessor initialState =
    MailboxProcessor<CounterMessage>.Start(fun inbox ->
        // In this case we represent the state as a mutable binding
        // local to this function. innerLoop will close over it and
        // change its value in each iteration instead of passing it around
        let mutable state = initialState

        let rec innerLoop () = async {
            printfn "Waiting for message, the current state is: %i" state
            let! message = inbox.Receive()
            match message with
            | Increment ->
                let state <- state + 1
                printfn "Counter incremented, the new state is: %i" state'
                innerLoop ()
            | Decrement ->
                let state <- state - 1
                printfn "Counter decremented, the new state is: %i" state'
                innerLoop ()
        }
        innerLoop ())

Mesmo que isso definitivamente não fosse seguro para threads se o estado do contador fosse modificado diretamente de vários threads, você pode ver usando a mensagem paralela Posts da seção anterior que o processador da caixa de correio processa as mensagens uma após a outra sem intercalação, de modo que cada mensagem usa o valor mais atual.

Valores de retorno #

Você pode retornar de forma assíncrona um valor para cada mensagem processada se enviar um AsyncReplyChannel<'a> como parte da mensagem.

type MessageWithResponse = Message of InputData * AsyncReplyChannel<OutputData>

Em seguida, o processador da caixa postal pode usar esse canal ao processar a mensagem para enviar um valor de volta ao chamador.

let! message = inbox.Receive()
match message with
| MessageWithResponse (data, r) ->
    // ...process the data
    let output = ...
    r.Reply(output)

Agora, para criar uma mensagem, você precisa do AsyncReplyChannel<'a> - o que é e como você cria uma instância de trabalho? A melhor maneira é deixar MailboxProcessor fornecer isso para você e extrair a resposta para um Async<'a> mais comum. Isso pode ser feito usando, por exemplo, o método PostAndAsynReply, onde você não publica a mensagem completa, mas sim uma função do tipo (no nosso caso) AsyncReplyChannel<OutputData> -> MessageWithResponse:

let! output = processor.PostAndAsyncReply(r -> MessageWithResponse(input, r))

Isso postará a mensagem em uma fila e aguardará a resposta, que chegará assim que o processador chegar a essa mensagem e responder usando o canal.

Há também uma variante síncrona PostAndReply que bloqueia a thread de chamada até que o processador responda.

Processamento de mensagens fora de ordem #

Você pode usar os métodos Scan ou TryScan para procurar mensagens específicas na fila e processá-las independentemente de quantas mensagens estejam antes delas. Ambos os métodos examinam as mensagens na fila na ordem em que chegaram e procuram uma mensagem especificada (até o tempo limite opcional). Caso não haja tal mensagem, TryScan retornará Nenhum, enquanto Scan continuará esperando até que tal mensagem chegue ou a operação expire.

Vamos ver na prática. Queremos que o processador processe RegularOperations quando puder, mas sempre que houver uma PriorityOperation, ela deve ser processada o mais rápido possível, não importa quantas outras RegularOperations estejam na fila.

type Message =
    | RegularOperation of string
    | PriorityOperation of string

let processor = MailboxProcessor<Message>.Start(fun inbox ->
    let rec innerLoop () = async {
        let! priorityData = inbox.TryScan(fun msg ->
            // If there is a PriorityOperation, retrieve its data.
            match msg with
            | PriorityOperation data -> Some data
            | _ -> None)

        match priorityData with
        | Some data ->
            // Process the data from PriorityOperation.
        | None ->
            // No PriorityOperation was in the queue at the time, so
            // let's fall back to processing all possible messages
            let! message = inbox.Receive()
            match message with
            | RegularOperation data ->
                // We have free time, let's process the RegularOperation.
            | PriorityOperation data ->
                // We did scan the queue, but it might have been empty
                // so it is possible that in the meantime a producer
                // posted a new message and it is a PriorityOperation.
        // And never forget to process next messages.
        innerLoop ()
    }
    innerLoop())
