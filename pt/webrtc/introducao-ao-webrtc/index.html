Configurando um sistema de comunicação baseado em WebRTC #

Para configurar um sistema de comunicação baseado em WebRTC, você precisa de três componentes principais:

  1. Um servidor de sinalização WebRTC

    To establish a WebRTC connections, peers need to contact a signaling server, which then provides the address information the peers require to set up a peer-to-peer connection. Signaling servers are for example:

    • signalmaster: Lightweight, JavaScript-based signaling server
    • NextRTC: Java-based signaling server
    • Kurento: Comprehensive WebRTC framework
    • Janus: General purpose WebRTC Gateway

  2. Um aplicativo cliente WebRTC

    The client accesses either a browser’s WebRTC implementation through a JavaScript API or uses a WebRTC library (i.e. as part of a desktop or mobile app). To establish the connection to a peer, the client first needs to connect to the signaling server. Examples for WebRTC clients are:

    • Several Kurento projects

    • OpenWebRTC, a cross-platform client with mobile focus

    • Peer.js A browser-based client (Peer.js also provides a light-weight server)

    • Janus Demo examples

  3. Um servidor STUN/TURN

    Session Traversal Utilities for NAT (STUN) enables peers to exchange address information even if they are behind routers employing Network Adress Translation (NAT). If network restrictions prevent peers from communication directly at all, the traffic is routed via a Traversal Using Relays around NAT (TURN) server. You find a detailed and graphical explanation of STUN and TURN at http://www.avaya.com/blogs/archives/2014/08/understanding-webrtc-media-connections-ice-stun-and-turn.html. Examples for WebRTC STUN/TURN servers are:

    • coturn combines STUN and TURN and is typically part of a fully-fledged WebRTC infrastructure.
    • Janus WebRTC Gateway comes with an integrated STUN/TURN server.

Introdução ao WebRTC #

WebRTC é uma estrutura aberta para a web que permite comunicações em tempo real no navegador. Ele inclui os blocos de construção fundamentais para comunicações de alta qualidade na Web, como componentes de rede, áudio e vídeo usados ​​em aplicativos de bate-papo por voz e vídeo.

Esses componentes, quando implementados em um navegador, podem ser acessados ​​por meio de uma API JavaScript, permitindo que os desenvolvedores implementem facilmente seu próprio aplicativo web RTC.

O esforço do WebRTC está sendo padronizado em um nível de API no W3C e no nível de protocolo no [IETF](https://tools.ietf. org/wg/rtcweb/).

  • Um fator chave para o sucesso da web é que suas principais tecnologias – como HTML, HTTP e TCP/IP – são abertos e implementáveis ​​livremente. Atualmente, não existe uma solução completa, gratuita e de alta qualidade disponível que permite a comunicação no navegador. O WebRTC permite isto.
  • Já integrado com os melhores motores de voz e vídeo que foram implantados em milhões de endpoints nos últimos 8 anos. O Google não cobra royalties pelo WebRTC.
  • Inclui e abstrai a chave NAT e a tecnologia de passagem de firewall, usando STUN, ICE, TURN, RTP-over-TCP e suporte para proxies.
  • Baseia-se na força do navegador da web: sinalização de resumos WebRTC oferecendo uma máquina de estado de sinalização que mapeia diretamente para PeerConnection. Os desenvolvedores da Web podem, portanto, escolher o protocolo de escolha para seu cenário de uso (por exemplo, mas não limitado a, SIP, XMPP/Jingle, etc).

Tenha acesso ao seu áudio e vídeo usando a API getUserMedia(), Olá WebRTC! #

navigator.mediaDevices é o método comum adaptado no Chrome e FF para getUserMedia a partir de agora.

Um retorno de chamada prometido que retorna o fluxo local em caso de sucesso

navigator.mediaDevices.getUserMedia({ audio: true, video: true })
    .then(stream => {
        // attach this stream to window object so you can reuse it later
        window.localStream = stream;
        // Your code to use the stream
    })
    .catch((err) =>{
        console.log(err);
    });

Você pode passar restrições de áudio e vídeo para getUserMedia para controlar as configurações de captura, como resolução, taxa de quadros, preferência de dispositivo e muito mais.

Anexar o fluxo a um elemento de vídeo

// Set the video element to autoplay to ensure the video appears live
videoElement.autoplay = true;
// Mute the local video so the audio output doesn't feedback into the input
videoElement.muted = true;
// attach the stream to the video element

parar o vídeo e o áudio

localStream.getTracks().forEach((track) => {
    track.stop();
});

parar apenas áudio

localStream.getAudioTracks()[0].stop();

parar apenas vídeo

localStream.getVideoTracks()[0].stop();
