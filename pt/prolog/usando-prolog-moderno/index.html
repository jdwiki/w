CLP(FD) para aritmética inteira #

Tradicionalmente, o Prolog realizava operações aritméticas usando os operadores is e =:=. No entanto, vários Prologs atuais oferecem CLP(FD) (Constraint Logic Programming over Finite Domains) como uma alternativa mais limpa para aritmética inteira. O CLP(FD) é baseado no armazenamento das restrições que se aplicam a um valor inteiro e na combinação delas na memória.

CLP(FD) é uma extensão na maioria dos Prologs que o suportam, então deve ser carregado explicitamente. Uma vez carregado, a sintaxe #= pode substituir tanto is quanto =:=. Por exemplo, no SWI-Prolog:

?- X is 2+2.
X = 4.

?- use_module(library(clpfd)).
?- X #= 2+2.
X = 4.

Ao contrário de is, #= é capaz de resolver equações simples e unificar em ambas as direções:

?- 4 is 2+X.
ERROR: is/2: Arguments are not sufficiently instantiated

?- 4 #= 2+X.
X = 2.

O CLP(FD) fornece sua própria sintaxe de gerador.

?- between(1,100,X).
X = 1;
X = 2;
X = 3...

?- X in 1..100.
X in 1..100.

Observe que o gerador não é executado: apenas a restrição de intervalo é armazenada, pronta para restrições posteriores serem combinadas com ela. O gerador pode ser forçado a rodar (e restrições de força bruta) usando o predicado label:

?- X in 1..100, label([X]).
X = 1;
X = 2;
X = 3..

O uso do CLP pode permitir alguma redução inteligente de casos de força bruta. Por exemplo, usando aritmética inteira de estilo antigo:

?- trace.
?- between(1,10,X), Y is X+5, Y>10.
...
Exit: (8) 6 is 1+5 ? creep
Call: (8) 6 > 10 ? creep
...
X = 6, Y = 11; ...

O Prolog ainda percorre os valores de 1 a 5, embora seja matematicamente demonstrável a partir das condições dadas, que esses valores não podem ser úteis. Usando CLP(FD):

?- X in 1..10, Y #= X+5, Y #> 10.
X is 6..10,
X+5 #= Y,
Y is 11..15.

O CLP(FD) imediatamente faz as contas e calcula os intervalos disponíveis. Adicionar label([Y]) fará com que X percorra apenas os valores úteis 6..10. Neste exemplo de brinquedo, isso não aumenta o desempenho porque com um intervalo tão pequeno como 1-10, o processamento de álgebra demora tanto quanto o loop levaria; mas quando um intervalo maior de números está sendo processado, isso pode reduzir significativamente o tempo de computação.

O suporte para CLP(FD) é variável entre Prologs. O melhor desenvolvimento reconhecido do CLP(FD) está no SICStus Prolog, que é comercial e caro. SWI-Prolog e outros Prologs abertos geralmente têm alguma implementação. Visual Prolog não inclui CLP(FD) em sua biblioteca padrão, embora bibliotecas de extensão para ele estejam disponíveis.

Introdução #

Muitos sistemas Prolog modernos estão em desenvolvimento contínuo e adicionaram novos recursos para resolver as deficiências clássicas da linguagem. Infelizmente, muitos livros didáticos de Prolog e até mesmo cursos de ensino ainda apresentam apenas o prólogo desatualizado. Este tópico destina-se a ilustrar como o Prolog moderno superou alguns dos problemas e a sintaxe bastante complicada que aparece no Prolog mais antigo e pode ainda estar sendo introduzida.

Forall em vez de loops orientados a falhas #

Alguns livros de Prolog “clássicos” ainda usam a sintaxe de loop orientada a falhas confusa e propensa a erros, onde uma construção ‘fail’ é usada para forçar o retrocesso para aplicar um objetivo a cada valor de um gerador. Por exemplo, para imprimir todos os números até um determinado limite:

fdl(X) :- between(1,X,Y), print(Y), fail.
fdl(_).

A grande maioria dos Prólogos Modernos não requer mais essa sintaxe, em vez disso, fornece um predicado de ordem superior para resolver isso.

nicer(X) :- forall(between(1,X,Y), print(Y)).

Não apenas isso é muito mais fácil de ler, mas se uma meta que poderia falhar fosse usada no lugar de print, sua falha seria detectada e transmitida corretamente - enquanto as falhas das metas em um loop orientado a falhas são confundidas com o falha forçada que aciona o loop.

O Visual Prolog possui um açúcar sintático personalizado para esses loops, combinado com predicados de função (veja abaixo):

vploop(X) :- foreach Y = std::fromTo(1,X) do
                 console::write(X)
             end foreach.

Embora pareça um loop imperativo for, ele ainda segue as regras do Prolog: em particular, cada iteração do foreach é seu próprio escopo.

Predicados de estilo de função #

Tradicionalmente no Prolog, “funções” (com uma saída e entradas vinculadas) eram escritas como predicados regulares:

mangle(X,Y) :- Y is (X*5)+2.

Isso pode criar a dificuldade de que, se um predicado de estilo de função for chamado várias vezes, será necessário “encadear” variáveis ​​temporárias.

multimangle(X,Y) :- mangle(X,A), mangle(A,B), mangle(B,Y).

Na maioria dos Prologs, é possível evitar isso escrevendo um operador infixo alternativo para usar no lugar de is que expande expressões incluindo a função alternativa.

% Define the new infix operator
:- op(900, xfy, <-).

% Define our function in terms of the infix operator - note the cut to avoid
% the choice falling through
R <- mangle(X) :- R is (X*5)+2, !.

% To make the new operator compatible with is..
R <- X :-
    compound(X),            % If the input is a compound/function
    X =.. [OP, X2, X3],     % Deconstruct it
    R2 <- X2,               % Recurse to evaluate the arguments
    R3 <- X3,
    Expr =.. [OP, R2, R3],  % Rebuild a compound with the evaluated arguments
    R is Expr,              % And send it to is
    !.
R <- X :- R is X, !.        % If it's not a compound, just use is directly

Podemos agora escrever:

multimangle(X,Y) :- X <- mangle(mangle(mangle(Y))).

No entanto, alguns Prologs modernos vão além e oferecem uma sintaxe personalizada para esse tipo de predicado. Por exemplo, no Visual Prolog:

mangle(X) = Y :- Y = ((X*5)+2).
multimangle(X,Y) :- Y = mangle(mangle(mangle(X))).

Observe que o operador <- e o predicado de estilo funcional acima ainda se comportam como relações - é legal que eles tenham pontos de escolha e realizem unificação múltipla. No primeiro exemplo, evitamos isso usando cortes. No Visual Prolog, é normal usar a sintaxe funcional para que as relações e os pontos de escolha sejam criados da maneira normal - por exemplo, o objetivo X = (std::fromTo(1,10))*10 é bem-sucedido com ligações X =10, X=20, X=30, X=40, etc.

Declarações de fluxo/modo #

Ao programar em Prolog nem sempre é possível, ou desejável, criar predicados que unifiquem para cada combinação possível de parâmetros. Por exemplo, o predicado between(X,Y,Z) que expressa que Z está numericamente entre X e Y. É facilmente implementado nos casos em que X, Y e Z estão todos vinculados (ou Z está entre X e Y ou não é), ou onde X e Y são vinculados e Z é livre (Z unifica com todos os números entre X e Y, ou o predicado falha se Y<X); mas em outros casos, como onde X e Z são limitados e Y é livre, há potencialmente um número infinito de unificações. Embora isso possa ser implementado, geralmente não seria.

Declaração de fluxo ou declarações de modo permitem uma descrição explícita de como os predicados se comportam quando chamados com diferentes combinações de parâmetros vinculados. No caso de between, a declaração seria:

%! between(+X,+Y,+Z) is semidet.
%! between(+X,+Y,-Z) is nondet.

Cada linha especifica um padrão de chamada em potencial para o predicado. Cada argumento é decorado com + para indicar casos em que está vinculado, ou - para indicar casos em que não está (há também outras decorações disponíveis para tipos mais complexos, como tuplas ou listas que podem ser parcialmente vinculadas). A palavra-chave após is indica o comportamento do predicado nesse caso e pode ser uma destas:

  • det se o predicado sempre for bem sucedido sem nenhum ponto de escolha. Por exemplo, add(+X,+Y,-Z) é det porque adicionar dois números X e Y sempre terá exatamente uma resposta.
  • semidet se o predicado for bem sucedido ou falhar, sem nenhum ponto de escolha. Como acima, between(+X,+Y,+Z) é semidet porque Z está entre X e Y ou não é.
  • multi se o predicado sempre for bem sucedido, mas pode ter pontos de escolha (mas também não pode). Por exemplo, factor(+X,-Y) seria multi porque um número sempre tem pelo menos um fator - ele mesmo - mas pode ter mais.
  • nondet se o predicado pode ter sucesso com pontos de escolha ou falhar. Por exemplo, between(+X,+Y,-Z) é nondet porque pode haver várias unificações possíveis de Z para números entre X e Y, ou se Y<X então não há números entre eles e o predicado falha.

Declarações de fluxo/modo também podem ser combinadas com rotulagem de argumentos para esclarecer o significado dos termos ou com a digitação. Por exemplo, between(+From:Int, +To:Int, +Mid:Int) é semidet.

Em Prologs puros, as declarações de fluxo e modo são opcionais e usadas apenas para geração de documentação, mas podem ser extremamente úteis para ajudar os programadores a identificar a causa dos erros de instanciação.

No Mercury, as declarações (e tipos) de fluxo e modo são obrigatórias e são validadas pelo compilador. A sintaxe usada é como acima.

No Visual Prolog, as declarações e tipos de fluxo e modo também são obrigatórios e a sintaxe é diferente. A declaração acima seria escrita como:

between : (int From, int To, int Mid) determ (i,i,i) nondeterm (i,i,o).

O significado é o mesmo que acima, mas com as diferenças que:

  • As declarações de fluxo/modo são separadas das declarações de tipo (já que é assumido que fluxo/modo para um único predicado não varia com a sobrecarga de tipo);
  • i e o são usados ​​para + e - e são combinados com os parâmetros baseados na ordenação;
  • Os termos utilizados são diferentes. det torna-se procedure, semidet torna-se determ, e nondet torna-se nondeterm (multi ainda é multi).
