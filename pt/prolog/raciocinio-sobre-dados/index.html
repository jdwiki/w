Uma nova seção chamada Estruturas de dados ganhou vida, onde são fornecidas explicações de certas estruturas e alguns exemplos simples de criação. Para manter seu conteúdo conciso e organizado, ele não deve conter nenhuma documentação sobre manipulação de dados.

Portanto, esta seção foi renomeada para “Raciocínio sobre dados” com o objetivo de generalizar o raciocínio sobre dados em Prolog. Isso pode incluir tópicos que variam de ‘inferência de cima para baixo’ a ‘percurso de listas’, bem como muitos outros. Por causa de sua ampla generalização, subseções claras devem ser feitas!

Recursão #

O Prolog não tem iteração, mas toda iteração pode ser reescrita usando recursão. A recursão aparece quando um predicado contém um objetivo que se refere a si mesmo. Ao escrever tais predicados em Prolog, um padrão recursivo padrão sempre tem pelo menos duas partes:

  • Cláusula de base (não recursiva): Normalmente, a(s) regra(s) do caso base representa(m) o(s) menor(es) exemplo(s) possível(is) do problema que você está tentando resolver - uma lista sem membros ou apenas um membro, ou se você estiver trabalhando com uma estrutura de árvore, pode lidar com uma árvore vazia, ou uma árvore com apenas um nó nela, etc. Descreve de forma não recursiva a base do processo recursivo.

  • cláusula recursiva (contínua): contém qualquer lógica necessária, incluindo uma chamada para si mesma, recursão contínua.

Como exemplo, definiremos o conhecido predicado append/3. Visto declarativamente, append(L1,L2,L3) é válido quando a lista L3 é o resultado de anexar listas L1 e L2. Quando tentamos descobrir o significado declarativo de um predicado, tentamos descrever soluções para as quais o predicado é válido. A dificuldade aqui está em tentar evitar detalhes recorrentes passo a passo, mantendo em mente o comportamento processual que o predicado deve exibir.

% Base case
append([],L,L).

% Recursive clause
append([X|L1],L2,[X|L3]) :- append(L1,L2,L3).

O caso base declara declarativamente “qualquer L anexado à lista vazia é L”, observe que isso não diz nada sobre L estar vazio - ou mesmo ser uma lista (lembre-se, em Prolog tudo se resume a termos):

?- append(X,some_term(a,b),Z).
X = [],
Z = some_term(a, b).

Para descrever a regra recursiva, embora o Prolog execute as regras da esquerda para a direita, omitimos a cabeça por um segundo e olhamos primeiro para o corpo – lendo a regra da direita para a esquerda:

<pré> append([X|L1],L2,[X|L3]) :- append(L1,L2,L3).

Agora dizemos que se o corpo for válido: “assumindo que append(L1,L2,L3) seja válido”

<pré> append([X|L1],L2,[X|L3]) :- append(L1,L2,L3).

Então o mesmo acontece com a cabeça: “então também append([X|L1],L2,[X|L3])”

Em inglês simples, isso simplesmente se traduz em:

Assumindo que L3 é a concatenação de L1 e L2, então [X seguido de L3] é também a concatenação de [X seguido de L1] e L2.

Em um exemplo prático:

“Assumindo que [1,2,3] é a concatenação de [1] e [2,3], então [a,1,2,3] é também a concatenação de [a,1] and [2,3].”

Agora vamos a algumas consultas:

É sempre uma boa ideia testar inicialmente seu predicado com a consulta mais geral em vez de fornecer um caso de teste de cenário específico. Pense nisso: por causa da unificação do Prolog, não somos obrigados a fornecer dados de teste, apenas entregamos variáveis ​​livres!

?- append(L1,L2,L3).
L1 = [],
L2 = L3 ;                                   % Answer #1
L1 = [_G1162],
L3 = [_G1162|L2] ;                          % Answer #2
L1 = [_G1162, _G1168],
L3 = [_G1162, _G1168|L2] ;                  % Answer #3
L1 = [_G1162, _G1168, _G1174],
L3 = [_G1162, _G1168, _G1174|L2] ;          % Answer #4
...

Vamos substituir a notação de variável livre _G1162 por letras alfabéticas para obter uma visão geral melhor:

?- append(L1,L2,L3).
L1 = [],
L2 = L3 ;                                   % Answer #1
L1 = [_A],
L3 = [_A|L2] ;                              % Answer #2
L1 = [_A, _B],
L3 = [_A, _B|L2] ;                          % Answer #3
L1 = [_A, _B, _C],
L3 = [_A, _B, _C|L2] ;                      % Answer #4
...

Na primeira resposta, o caso base foi correspondido ao padrão e o Prolog instancia L1 para a lista vazia e unificou L2 e L3 provando que L3 é a concatenação da lista vazia e L2.

Na resposta #2, por meio de retrocesso cronológico, a cláusula recursiva entra em jogo e Prolog tenta provar que algum elemento no cabeçalho de L1 concatenado com L2 é L3 com esse mesmo elemento no cabeçalho da lista. Para fazer isso, uma nova variável livre _A é unificada com a cabeça de L1 e L3 é agora [_A|L2].

Uma nova chamada recursiva é feita, agora com L1 = [_A]. Mais uma vez, Prolog tenta provar que algum elemento colocado na cabeça de L1, concatenado com L2 é L3 com esse mesmo elemento em sua cabeça. Observe que _A já é o cabeça de L1, que combina perfeitamente com a regra, então agora, através de recursão, Prolog coloca _A na frente de uma nova variável livre e obtemos L1 = [_A,_B] e L3 = [_A,_B|L2]

Vemos claramente o padrão recursivo se repetindo e podemos ver facilmente que, por exemplo, o resultado do 100º passo na recursão ficaria assim:

L1 = [X1,X2,..,X99],
L3 = [X1,X2,..,X99|L2]

Nota: como é típico para um bom código Prolog, a definição recursiva de append/3 nos fornece não apenas a possibilidade de verificar se uma lista é a concatenação de duas outras listas, mas também gerar todas as respostas possíveis que satisfazem as relações lógicas com listas total ou parcialmente instanciadas.

Acessando listas #

Membro

member/2 tem assinatura member(?Elem, ?List) e denota true se Elem for um membro de List. Esse predicado pode ser usado para acessar variáveis ​​em uma lista, onde diferentes soluções são recuperadas por meio de retrocesso.

Exemplos de consultas:

?- member(X, [1,2,3]).
X = 1 ;
X = 2 ;
X = 3.

?- member(X,[Y]).
X = Y.

?- member(X,Y).
Y = [X|_G969] ;
Y = [_G968, X|_G972] ;
Y = [_G968, _G971, X|_G975] ;
Y = [_G968, _G971, _G974, X|_G978]
...

Correspondência de padrões

Quando os índices que você precisa acessar são pequenos, a correspondência de padrões pode ser uma boa solução, por exemplo:

third([_,_,X|_], X).
fourth([_,_,_,X|_], X).
