JavaScript, como qualquer linguagem, exige que sejamos criteriosos no uso de certos recursos de linguagem. O uso excessivo de alguns recursos pode diminuir o desempenho, enquanto algumas técnicas podem ser usadas para aumentar o desempenho.

Lembre-se de que a otimização prematura é a raiz de todo mal. Escreva primeiro um código claro e correto e, em seguida, se tiver problemas de desempenho, use um criador de perfil para procurar áreas específicas a serem aprimoradas. Não perca tempo otimizando o código que não está afetando o desempenho geral de maneira significativa.

Meça, meça, meça. O desempenho muitas vezes pode ser contra-intuitivo e muda com o tempo. O que é mais rápido agora pode não ser no futuro e pode depender do seu caso de uso. Certifique-se de que todas as otimizações que você fizer estão realmente melhorando, não prejudicando o desempenho, e que a mudança vale a pena.

Evite try/catch em funções críticas de desempenho #

Alguns mecanismos JavaScript (por exemplo, a versão atual do Node.js e versões mais antigas do Chrome antes do Ignition+turbofan) não executam o otimizador em funções que contêm um bloco try/catch.

Se você precisar lidar com exceções no código de desempenho crítico, pode ser mais rápido em alguns casos manter o try/catch em uma função separada. Por exemplo, esta função não será otimizada por algumas implementações:

function myPerformanceCriticalFunction() {
    try {
        // do complex calculations here
    } catch (e) {
        console.log(e);
    }
}

No entanto, você pode refatorar para mover o código lento para uma função separada (que pode ser otimizada) e chamá-la de dentro do bloco try.

// This function can be optimized
function doCalculations() {
    // do complex calculations here
}

// Still not always optimized, but it's not doing much so the performance doesn't matter
function myPerformanceCriticalFunction() {
    try {
        doCalculations();
    } catch (e) {
        console.log(e);
    }
}

Aqui está um benchmark jsPerf mostrando a diferença: https://jsperf.com/try-catch-deoptimization. Na versão atual da maioria dos navegadores, não deve haver muita diferença, se houver, mas em versões menos recentes do Chrome e Firefox, ou IE, a versão que chama uma função auxiliar dentro do try/catch provavelmente será mais rápida.

Observe que otimizações como essa devem ser feitas com cuidado e com evidências reais baseadas na criação de perfil do seu código. À medida que os mecanismos JavaScript ficam melhores, isso pode acabar prejudicando o desempenho em vez de ajudar, ou não fazendo diferença alguma (mas complicando o código sem motivo). Se isso ajuda, prejudica ou não faz diferença, pode depender de muitos fatores, portanto, sempre meça os efeitos em seu código. Isso vale para todas as otimizações, mas especialmente para micro-otimizações como essa que dependem de detalhes de baixo nível do compilador/tempo de execução.

Limitar atualizações do DOM #

Um erro comum visto em JavaScript quando executado em um ambiente de navegador é atualizar o DOM com mais frequência do que o necessário.

O problema aqui é que cada atualização na interface DOM faz com que o navegador renderize novamente a tela. Se uma atualização altera o layout de um elemento na página, todo o layout da página precisa ser recalculado, e isso é muito pesado para o desempenho, mesmo nos casos mais simples. O processo de redesenhar uma página é conhecido como reflow e pode fazer com que um navegador seja executado lentamente ou até mesmo pare de responder.

A consequência de atualizar o documento com muita frequência é ilustrada com o seguinte exemplo de adição de itens a uma lista.

Considere o seguinte documento contendo um elemento <ul>:

<!DOCTYPE html>
<html>
    <body>
        <ul id="list"></ul>
    </body>
</html>

Adicionamos itens 5000 à lista fazendo um loop 5000 vezes (você pode tentar isso com um número maior em um computador poderoso para aumentar o efeito).

var list = document.getElementById("list");
for(var i = 1; i <= 5000; i++) {             
    list.innerHTML += `<li>item ${i}</li>`;  // update 5000 times
}

Nesse caso, o desempenho pode ser aprimorado agrupando todas as 5.000 alterações em uma única atualização do DOM.

var list = document.getElementById("list");
var html = "";
for(var i = 1; i <= 5000; i++) {
    html += `<li>item ${i}</li>`;
}
list.innerHTML = html;     // update once

A função document.createDocumentFragment() pode ser usada como um contêiner leve para o HTML criado pelo loop. Este método é um pouco mais rápido do que modificar a propriedade innerHTML do elemento container (como mostrado abaixo).

var list = document.getElementById("list");
var fragment = document.createDocumentFragment();
for(var i = 1; i <= 5000; i++) {
    li = document.createElement("li");
    li.innerHTML = "item " + i;
    fragment.appendChild(li);
    i++;
}
list.appendChild(fragment);

Comparando seu código - medindo o tempo de execução #

A maioria das dicas de desempenho depende muito do estado atual dos mecanismos JS e espera-se que sejam relevantes apenas em um determinado momento. A lei fundamental da otimização de desempenho é que você deve primeiro medir antes de tentar otimizar e medir novamente após uma otimização presumida.

Para medir o tempo de execução do código, você pode usar diferentes ferramentas de medição de tempo, como:

Desempenho interface que representa informações de desempenho relacionadas ao tempo para a página em questão (disponível apenas em navegadores).

process.hrtime no Node.js fornece informações de tempo como tuplas de [segundos, nanossegundos]. Chamado sem argumento, ele retorna um tempo arbitrário, mas chamado com um valor retornado anteriormente como argumento, ele retorna a diferença entre as duas execuções.

Temporizadores de console console.time("labelName") inicia um cronômetro que você pode usar para rastrear quanto tempo uma operação leva. Você dá a cada cronômetro um nome de rótulo exclusivo e pode ter até 10.000 cronômetros em execução em uma determinada página. Quando você chama console.timeEnd("labelName") com o mesmo nome, o navegador terminará o cronômetro para determinado nome e exibirá o tempo em milissegundos decorrido desde que o cronômetro foi iniciado. As strings passadas para time() e timeEnd() devem corresponder, caso contrário o cronômetro não terminará.

A função Date.now Date.now() retorna Timestamp atual em milissegundos, que é uma representação Number do tempo desde 1º de janeiro de 1970 00:00:00 UTC até agora. O método now() é um método estático de Date, portanto você sempre o utiliza como Date.now().

Exemplo 1 usando: performance.now()

Neste exemplo vamos calcular o tempo decorrido para a execução de nossa função, e vamos usar o método Performance.now() que retorna um DOMHighResTimeStamp, medido em milissegundos, precisão de um milésimo de milissegundo.

let startTime, endTime;

function myFunction() {
    //Slow code you want to measure
}

//Get the start time
startTime = performance.now();

//Call the time-consuming function
myFunction();

//Get the end time
endTime = performance.now();

//The difference is how many milliseconds it took to call myFunction()
console.debug('Elapsed time:', (endTime - startTime));

O resultado no console será mais ou menos assim:

Elapsed time: 0.10000000009313226

O uso de performance.now() tem a mais alta precisão em navegadores com precisão de um milésimo de milissegundo, mas a menor compatibilidade.

Exemplo 2 usando: Date.now()

Neste exemplo vamos calcular o tempo decorrido para a inicialização de um grande array (1 milhão de valores), e vamos usar o método Date.now()

let t0 = Date.now(); //stores current Timestamp in milliseconds since 1 January 1970 00:00:00 UTC
let arr = []; //store empty array
for (let i = 0; i < 1000000; i++) { //1 million iterations
   arr.push(i); //push current i value
}
console.log(Date.now() - t0); //print elapsed time between stored t0 and now

Exemplo 3 usando: console.time("label") & console.timeEnd("label")

Neste exemplo estamos fazendo a mesma tarefa do Exemplo 2, mas vamos usar os métodos console.time("label") & console.timeEnd("label")

console.time("t"); //start new timer for label name: "t"
let arr = []; //store empty array
for(let i = 0; i < 1000000; i++) { //1 million iterations
   arr.push(i); //push current i value
}
console.timeEnd("t"); //stop the timer for label name: "t" and print elapsed time

Exemplo 4 usando process.hrtime()

Nos programas Node.js, essa é a maneira mais precisa de medir o tempo gasto.

let start = process.hrtime();

// long execution here, maybe asynchronous

let diff = process.hrtime(start);
// returns for example [ 1, 2325 ]
console.log(`Operation took ${diff[0] * 1e9 + diff[1]} nanoseconds`);
// logs: Operation took 1000002325 nanoseconds

Use um memoizer para funções de computação pesada #

Se você está construindo uma função que pode ser pesada no processador (seja do lado do cliente ou do lado do servidor), você pode querer considerar um memoizer que é um cache de execuções de funções anteriores e seus valores retornados. Isso permite verificar se os parâmetros de uma função foram passados ​​antes. Lembre-se, funções puras são aquelas que fornecem uma entrada, retornam uma saída única correspondente e não causam efeitos colaterais fora de seu escopo, portanto, você não deve adicionar memoizers a funções que são imprevisíveis ou dependem de recursos externos (como chamadas AJAX ou valores retornados).

Digamos que eu tenha uma função fatorial recursiva:

function fact(num) {
  return (num === 0)? 1 : num * fact(num - 1);
}

Se eu passar pequenos valores de 1 a 100, por exemplo, não haverá problema, mas quando começarmos a ir mais fundo, podemos explodir a pilha de chamadas ou tornar o processo um pouco doloroso para o mecanismo Javascript em que estamos fazendo isso, especialmente se o motor não contar com otimização de chamada de cauda (embora Douglas Crockford diga que o ES6 nativo tem otimização de chamada de cauda incluída).

Poderíamos codificar nosso próprio dicionário de 1 a Deus sabe-que número com seus fatoriais correspondentes, mas não tenho certeza se aconselho isso! Vamos criar um memoizer, certo?

var fact = (function() {
  var cache = {}; // Initialise a memory cache object
  
  // Use and return this function to check if val is cached
  function checkCache(val) {
    if (val in cache) {
      console.log('It was in the cache :D');
      return cache[val]; // return cached
    } else {
      cache[val] = factorial(val); // we cache it
      return cache[val]; // and then return it
    }
    
    /* Other alternatives for checking are:
    || cache.hasOwnProperty(val) or !!cache[val]
    || but wouldn't work if the results of those
    || executions were falsy values.
    */
  }

  // We create and name the actual function to be used
  function factorial(num) {
    return (num === 0)? 1 : num * factorial(num - 1);
  } // End of factorial function

  /* We return the function that checks, not the one
  || that computes because  it happens to be recursive,
  || if it weren't you could avoid creating an extra
  || function in this self-invoking closure function.
  */
  return checkCache; 
}());

Agora podemos começar a usar:

Agora que começo a refletir sobre o que fiz, se eu fosse incrementar de 1 em vez de decrementar de num, eu poderia ter armazenado em cache todos os fatoriais de 1 a num no cache recursivamente, mas vou deixar isso para voce.

Isso é ótimo, mas e se tivermos vários parâmetros? Isto é um problema? Não exatamente, podemos fazer alguns truques legais como usar JSON.stringify() no array de argumentos ou até mesmo uma lista de valores dos quais a função dependerá (para abordagens orientadas a objetos). Isso é feito para gerar uma chave exclusiva com todos os argumentos e dependências incluídos.

Também podemos criar uma função que “memoiza” outras funções, usando o mesmo conceito de escopo de antes (retornando uma nova função que usa a original e tem acesso ao objeto cache):

AVISO: Sintaxe ES6, se você não gostar, substitua … por nada e use o truque var args = Array.prototype.slice.call(null, arguments);; substitua const e let por var, e as outras coisas que você já sabe.

function memoize(func) {
  let cache = {};

  // You can opt for not naming the function
  function memoized(...args) {
    const argsKey = JSON.stringify(args);
    
    // The same alternatives apply for this example
    if (argsKey in cache) {
      console.log(argsKey + ' was/were in cache :D');
      return cache[argsKey];
    } else {
      cache[argsKey] = func.apply(null, args); // Cache it
      return cache[argsKey]; // And then return it
    }
  }

  return memoized; // Return the memoized function
}

Agora observe que isso funcionará para vários argumentos, mas não será muito útil em métodos orientados a objetos, acho que você pode precisar de um objeto extra para dependências. Além disso, func.apply(null, args) pode ser substituído por func(...args) já que a desestruturação de array irá enviá-los separadamente ao invés de como um array. Além disso, apenas para referência, passar um array como argumento para func não funcionará a menos que você use Function.prototype.apply como eu fiz.

Para usar o método acima, basta:

const newFunction = memoize(oldFunction);

// Assuming new oldFunction just sums/concatenates:
newFunction('meaning of life', 42);
// -> "meaning of life42"

newFunction('meaning of life', 42); // again
// => ["meaning of life",42] was/were in cache :D

Inicializando as propriedades do objeto com null #

Todos os compiladores JIT JavaScript modernos tentam otimizar o código com base nas estruturas de objetos esperadas. Alguma dica de mdn.

Felizmente, os objetos e propriedades são muitas vezes “previsíveis”, e em nesses casos, sua estrutura subjacente também pode ser previsível. JITs pode contar com isso para tornar os acessos previsíveis mais rápidos.

A melhor maneira de tornar o objeto previsível é definir uma estrutura inteira em um construtor. Então, se você for adicionar algumas propriedades extras após a criação do objeto, defina-as em um construtor com null. Isso ajudará o otimizador a prever o comportamento do objeto durante todo o seu ciclo de vida. No entanto, todos os compiladores têm otimizadores diferentes, e o aumento de desempenho pode ser diferente, mas no geral é uma boa prática definir todas as propriedades em um construtor, mesmo quando seu valor ainda não é conhecido.

Tempo para alguns testes. No meu teste, estou criando uma grande matriz de algumas instâncias de classe com um laço for. Dentro do loop, estou atribuindo a mesma string à propriedade “x” de todos os objetos antes da inicialização do array. Se o construtor inicializar a propriedade “x” com null, o array sempre processa melhor, mesmo que esteja fazendo uma instrução extra.

Este é o código:

function f1() {
    var P = function () {
        this.value = 1
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}

function f2() {
    var P = function () {
        this.value = 1;
        this.x = null;
    };
    var big_array = new Array(10000000).fill(1).map((x, index)=> {
        p = new P();
        if (index > 5000000) {
            p.x = "some_string";
        }

        return p;
    });
    big_array.reduce((sum, p)=> sum + p.value, 0);
}


(function perform(){
    var start = performance.now();
    f1();
    var duration = performance.now() - start;

    console.log('duration of f1  ' + duration);


    start = performance.now();
    f2();
    duration = performance.now() - start;

    console.log('duration of f2 ' + duration);
})()

Este é o resultado para Chrome e Firefox.

       FireFox     Chrome
 --------------------------
 f1      6,400      11,400
 f2      1,700       9,600

Como podemos ver, as melhorias de desempenho são muito diferentes entre os dois.

Reutilize objetos em vez de recriar #

Exemplo A #

var i,a,b,len;
a = {x:0,y:0}
function test(){ // return object created each call
    return {x:0,y:0};
}
function test1(a){ // return object supplied 
    a.x=0;
    a.y=0;
    return a;
}   

for(i = 0; i < 100; i ++){ // Loop A
   b = test();
}

for(i = 0; i < 100; i ++){ // Loop B
   b = test1(a);
}

O Loop B é 4 (400%) vezes mais rápido que o Loop A

É muito ineficiente criar um novo objeto no código de desempenho. Loop A chama a função test() que retorna um novo objeto a cada chamada. O objeto criado é descartado a cada iteração, o Loop B chama test1() que requer que os retornos do objeto sejam fornecidos. Assim, ele usa o mesmo objeto e evita a alocação de um novo objeto e acertos excessivos de GC. (GC não foram incluídos no teste de desempenho)

Exemplo B #

var i,a,b,len;
a = {x:0,y:0}
function test2(a){
    return {x : a.x * 10,y : a.x * 10};
}   
function test3(a){
    a.x= a.x * 10;
    a.y= a.y * 10;
    return a;
}  
for(i = 0; i < 100; i++){  // Loop A
    b = test2({x : 10, y : 10});
}
for(i = 0; i < 100; i++){ // Loop B
    a.x = 10;
    a.y = 10;
    b = test3(a);                
}

O loop B é 5 (500%) vezes mais rápido que o loop A

Prefira variáveis ​​locais a globais, atributos e valores indexados #

Os mecanismos Javascript primeiro procuram variáveis ​​no escopo local antes de estender sua pesquisa para escopos maiores. Se a variável for um valor indexado em uma matriz ou um atributo em uma matriz associativa, ela primeiro procurará a matriz pai antes de encontrar o conteúdo.

Isso tem implicações ao trabalhar com código de desempenho crítico. Tomemos por exemplo um loop for comum:

var global_variable = 0;
function foo(){
    global_variable = 0;
    for (var i=0; i<items.length; i++) {
        global_variable += items[i];
    }
}

Para cada iteração no loop for, o mecanismo procurará items, procurará o atributo length dentro dos itens, procurará items novamente, procurará o valor no índice i de items e, finalmente, procurará global_variable, primeiro tentando o escopo local antes de verificar o escopo global.

Uma reescrita performática da função acima é:

function foo(){
    var local_variable = 0;
    for (var i=0, li=items.length; i<li; i++) {
        local_variable += items[i];
    }
    return local_variable;
}

Para cada iteração no loop for reescrito, a engine irá procurar li, procurar items, procurar o valor no índice i, e procurar local_variable, desta vez só precisando verificar o escopo local.

Seja consistente no uso de números #

Se o mecanismo for capaz de prever corretamente que você está usando um tipo pequeno específico para seus valores, ele poderá otimizar o código executado.

Neste exemplo, usaremos esta função trivial somando os elementos de um array e gerando o tempo que levou:

// summing properties
var sum = (function(arr){
        var start = process.hrtime();
        var sum = 0;
        for (var i=0; i<arr.length; i++) {
                sum += arr[i];
        }
        var diffSum = process.hrtime(start);
        console.log(`Summing took ${diffSum[0] * 1e9 + diffSum[1]} nanoseconds`);
        return sum;
})(arr);

Vamos fazer um array e somar os elementos:

var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.random();

Resultado:

Summing took 384416 nanoseconds

Agora, vamos fazer o mesmo, mas apenas com números inteiros:

var     N = 12345,
        arr = [];
for (var i=0; i<N; i++) arr[i] = Math.round(1000*Math.random());

Resultado:

Summing took 180520 nanoseconds

A soma de números inteiros levou metade do tempo aqui.

Os mecanismos não usam os mesmos tipos que você tem em JavaScript. Como você provavelmente sabe, todos os números em JavaScript são números de ponto flutuante de precisão dupla IEEE754, não há representação específica disponível para números inteiros. Mas os motores, quando eles podem prever que você usa apenas inteiros, podem usar uma representação mais compacta e mais rápida para usar, por exemplo, inteiros curtos.

Esse tipo de otimização é especialmente importante para computação ou aplicativos intensivos de dados.
