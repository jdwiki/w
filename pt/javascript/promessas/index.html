Sintaxe #

  • new Promise( /* função executora: */ function(resolver, rejeitar) { })
  • promessa.then(onFulfilled[, onRejected])
  • promessa.catch(onRejected)
  • Promessa.resolver(resolução)
  • Promise.reject(motivo)
  • Promise.all(iterável)
  • Promise.race(iterável)

As promessas fazem parte da especificação ECMAScript 2015 e o suporte ao navegador é limitado, com 88% dos navegadores em todo o mundo suportando-o em julho de 2017. A tabela a seguir fornece uma visão geral das versões mais antigas do navegador que oferecem suporte para promessas.

Chrome Borda Firefox Explorador de Internet Ópera Ópera Mini Safári iOS Safari
32 12 27 x 19 x 7.1 8

Em ambientes que não os suportam, Promise pode ser polipreenchido. Bibliotecas de terceiros também podem fornecer funcionalidades estendidas, como “promisificação” automatizada de funções de retorno de chamada ou métodos adicionais como progress—também conhecido como notify.

O site padrão Promises/A+ fornece uma lista de implementações compatíveis com 1.0 e 1.1. Os retornos de chamada de promessa baseados no padrão A+ são sempre executados de forma assíncrona como microtarefas no loop de eventos.

Introdução #

Um objeto Promise representa uma operação que produziu ou eventualmente produzirá um valor. As promessas fornecem uma maneira robusta de encapsular o resultado (possivelmente pendente) do trabalho assíncrono, mitigando o problema de retornos de chamada profundamente aninhados (conhecidos como “callback hell”).

Estados e fluxo de controle #

Uma promessa pode estar em um dos três estados:

  • pendente — A operação subjacente ainda não foi concluída e a promessa está pendente de cumprimento.
  • fulfilled — A operação foi concluída e a promessa é fulfilled com um value. Isso é análogo a retornar um valor de uma função síncrona.
  • rejeitado — Ocorreu um erro durante a operação e a promessa é rejeitada com um motivo. Isso é análogo a lançar um erro em uma função síncrona.

Uma promessa é considerada resolvida (ou resolvida) quando é cumprida ou rejeitada. Uma vez que uma promessa é estabelecida, ela se torna imutável e seu estado não pode mudar. Os métodos then e catch de uma promessa podem ser usados ​​para anexar callbacks que são executados quando ela é liquidada. Esses retornos de chamada são invocados com o valor de atendimento e o motivo da rejeição, respectivamente.

[![Diagrama de fluxo da promessa][diagrama_fluxo]][diagrama_fluxo]

Exemplo #

const promise = new Promise((resolve, reject) => {
    // Perform some work (possibly asynchronous)
    // ...

    if (/* Work has successfully finished and produced "value" */) {
        resolve(value);
    } else {
        // Something went wrong because of "reason"
        // The reason is traditionally an Error object, although
        // this is not required or enforced.
        let reason = new Error(message);
        reject(reason);

        // Throwing an error also rejects the promise.
        throw reason;
    }
});

Os métodos then e catch podem ser usados ​​para anexar callbacks de fulfillment e rejeição:

promise.then(value => {
    // Work has completed successfully,
    // promise has been fulfilled with "value"
}).catch(reason => {
    // Something went wrong,
    // promise has been rejected with "reason"
});

Observação: Chamar promise.then(...) e promise.catch(...) na mesma promessa pode resultar em uma exceção não capturada na Promise se ocorrer um erro, seja durante a execução a promessa ou dentro de um dos retornos de chamada, então a maneira preferida seria anexar o próximo ouvinte na promessa retornada pelo then / catch anterior.

Alternativamente, ambos os callbacks podem ser anexados em uma única chamada para then:

promise.then(onFulfilled, onRejected);

Anexar retornos de chamada a uma promessa que já foi liquidada os colocará imediatamente na [fila de microtarefas][microtarefa_fila], e eles serão invocados “assim que possível” (ou seja, imediatamente após o script em execução no momento). Não é necessário verificar o estado da promessa antes de anexar retornos de chamada, ao contrário de muitas outras implementações de emissão de eventos.

Demonstração ao vivo

Encadeamento de promessas #

O método then de uma promessa retorna uma nova promessa.

const promise = new Promise(resolve => setTimeout(resolve, 5000));

promise
    // 5 seconds later
    .then(() => 2)
    // returning a value from a then callback will cause
    // the new promise to resolve with this value
    .then(value => { /* value === 2 */ });

Retornar uma Promise de um callback then irá anexá-la à cadeia de promessas.

function wait(millis) {
    return new Promise(resolve => setTimeout(resolve, millis));
}

const p = wait(5000).then(() => wait(4000)).then(() => wait(1000));
p.then(() => { /* 10 seconds have passed */ });

Um catch permite que uma promessa rejeitada seja recuperada, semelhante a como catch em uma instrução try/catch funciona. Qualquer then encadeado após um catch executará seu manipulador de resolução usando o valor resolvido do catch.

const p = new Promise(resolve => {throw 'oh no'});
p.catch(() => 'oh yes').then(console.log.bind(console));  // outputs "oh yes"

Se não houver manipuladores catch ou reject no meio da cadeia, um catch no final capturará qualquer rejeição na cadeia:

p.catch(() => Promise.reject('oh yes'))
  .then(console.log.bind(console))      // won't be called
  .catch(console.error.bind(console));  // outputs "oh yes"

Em certas ocasiões, você pode querer “ramificar” a execução das funções. Você pode fazer isso retornando promessas diferentes de uma função dependendo da condição. Mais adiante no código, você pode mesclar todas essas ramificações em uma para chamar outras funções nelas e/ou lidar com todos os erros em um só lugar.

promise
    .then(result => {          
        if (result.condition) {
            return handlerFn1() 
                .then(handlerFn2);
        } else if (result.condition2) {
            return handlerFn3()
                .then(handlerFn4);
        } else {
            throw new Error("Invalid result");
        }
    })
    .then(handlerFn5)
    .catch(err => {
        console.error(err);
    });

Assim, a ordem de execução das funções fica assim:

promise --> handlerFn1 -> handlerFn2 --> handlerFn5 ~~> .catch()
         |                            ^
         V                            |
         -> handlerFn3 -> handlerFn4 -^

O único catch receberá o erro em qualquer ramo que possa ocorrer.

Aguardando várias promessas simultâneas #

O método estático Promise.all() aceita um iterável (por exemplo, um Array) de promessas e retorna uma nova promessa, que é resolvida quando todas as promessas no iterável são resolvidas ou rejeitadas se pelo menos uma das promessas no iterável for rejeitada.

// wait "millis" ms, then resolve with "value"
function resolve(value, milliseconds) {
    return new Promise(resolve => setTimeout(() => resolve(value), milliseconds));
}

// wait "millis" ms, then reject with "reason"
function reject(reason, milliseconds) {
    return new Promise((_, reject) => setTimeout(() => reject(reason), milliseconds));
}

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    resolve(3, 7000)    
]).then(values => console.log(values)); // outputs "[1, 2, 3]" after 7 seconds.

Promise.all([
    resolve(1, 5000),
    reject('Error!', 6000),
    resolve(2, 7000)
]).then(values => console.log(values)) // does not output anything
.catch(reason => console.log(reason)); // outputs "Error!" after 6 seconds.

Valores não promissores no iterável são [“promisified”][promisifying].

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    { hello: 3 }
])
.then(values => console.log(values)); // outputs "[1, 2, { hello: 3 }]" after 6 seconds

A atribuição de desestruturação pode ajudar a recuperar resultados de várias promessas.

Promise.all([
    resolve(1, 5000),
    resolve(2, 6000),
    resolve(3, 7000)
])
.then(([result1, result2, result3]) => {
    console.log(result1);
    console.log(result2);
    console.log(result3);
});

Reduza um array para promessas encadeadas #

Esse padrão de design é útil para gerar uma sequência de ações assíncronas de uma lista de elementos.

Existem duas variantes:

  • a redução “depois”, que constrói uma cadeia que continua enquanto a cadeia tiver sucesso.
  • a redução “catch”, que constrói uma cadeia que continua enquanto a cadeia apresentar erros.

A redução “então”

Essa variante do padrão cria uma cadeia .then() e pode ser usada para encadear animações ou fazer uma sequência de solicitações HTTP dependentes.

[1, 3, 5, 7, 9].reduce((seq, n) => {
    return seq.then(() => {
        console.log(n);
        return new Promise(res => setTimeout(res, 1000));
    });
}, Promise.resolve()).then(
    () => console.log('done'),
    (e) => console.log(e)
);
// will log 1, 3, 5, 7, 9, 'done' in 1s intervals

Explicação:

  1. Chamamos .reduce() em um array de origem e fornecemos Promise.resolve() como um valor inicial.
  2. Cada elemento reduzido adicionará um .then() ao valor inicial.
  3. O produto de reduce() será Promise.resolve().then(…).then(…).
  4. Anexamos manualmente um .then(successHandler, errorHandler) após a redução, para executar o successHandler uma vez que todas as etapas anteriores tenham sido resolvidas. Se alguma etapa falhar, o errorHandler será executado.

Nota: A redução “then” é uma contrapartida sequencial de Promise.all().

A redução de “pega”

Esta variante do padrão constrói uma cadeia .catch() e pode ser usada para sondar sequencialmente um conjunto de servidores web para algum recurso espelhado até que um servidor funcional seja encontrado.

var working_resource = 5; // one of the values from the source array
[1, 3, 5, 7, 9].reduce((seq, n) => {
    return seq.catch(() => {
        console.log(n);
        if(n === working_resource) { // 5 is working
            return new Promise((resolve, reject) => setTimeout(() => resolve(n), 1000));
        } else { // all other values are not working
            return new Promise((resolve, reject) => setTimeout(reject, 1000));
        }
    });
}, Promise.reject()).then(
    (n) => console.log('success at: ' + n),
    () => console.log('total failure')
);
// will log 1, 3, 5, 'success at 5' at 1s intervals

Explicação:

  1. Chamamos .reduce() em um array de origem e fornecemos Promise.reject() como um valor inicial.
  2. Cada elemento reduzido adicionará um .catch() ao valor inicial.
  3. O produto de reduce() será Promise.reject().catch(...).catch(...).
  4. Anexamos manualmente .then(successHandler, errorHandler) após a redução, para executar o successHandler uma vez que qualquer uma das etapas anteriores tenha sido resolvida. Se todas as etapas falharem, o errorHandler será executado.

Nota: A redução “catch” é uma contrapartida sequencial de Promise.any() (conforme implementado em bluebird.js, mas não atualmente em ECMAScript nativo).

Aguardando a primeira de várias promessas simultâneas #

O método estático Promise.race() aceita um iterável de Promises e retorna um novo Promise que resolve ou rejeita assim que a primeira promessa no iterável for resolvida ou rejeitada.

// wait "milliseconds" milliseconds, then resolve with "value"
function resolve(value, milliseconds) {
    return new Promise(resolve => setTimeout(() => resolve(value), milliseconds));
}

// wait "milliseconds" milliseconds, then reject with "reason"
function reject(reason, milliseconds) {
    return new Promise((_, reject) => setTimeout(() => reject(reason), milliseconds));
}

Promise.race([
    resolve(1, 5000),
    resolve(2, 3000),
    resolve(3, 1000)
])
.then(value => console.log(value)); // outputs "3" after 1 second.


Promise.race([
    reject(new Error('bad things!'), 1000),
    resolve(2, 2000)
])
.then(value => console.log(value)) // does not output anything
.catch(error => console.log(error.message)); // outputs "bad things!" after 1 second

Funções “promissoras” com callbacks #

Dada uma função que aceita um retorno de chamada no estilo Node,

fooFn(options, function callback(err, result) { ... });

você pode prometer (converter para uma função baseada em promessa) assim:

function promiseFooFn(options) {
    return new Promise((resolve, reject) =>
        fooFn(options, (err, result) =>
            // If there's an error, reject; otherwise resolve
            err ? reject(err) : resolve(result)
        )
    );
}

Esta função pode então ser usada da seguinte forma:

promiseFooFn(options).then(result => {
    // success!
}).catch(err => {
    // error!
});

De uma maneira mais genérica, veja como prometer qualquer função de estilo de retorno de chamada:

function promisify(func) {
    return function(...args) {
        return new Promise((resolve, reject) => {
            func(...args, (err, result) => err ? reject(err) : resolve(result));
        });
    }
}

Isso pode ser usado assim:

const fs = require('fs');
const promisedStat = promisify(fs.stat.bind(fs));

promisedStat('/foo/bar')
    .then(stat => console.log('STATE', stat))
    .catch(err => console.log('ERROR', err));

Manipulação de erros #

Erros lançados de promessas são tratados pelo segundo parâmetro (reject) passado para then ou pelo manipulador passado para catch:

throwErrorAsync()
  .then(null, error => { /* handle error here */ });
// or
throwErrorAsync()
  .catch(error => { /* handle error here */ });

Encadeamento #

Se você tiver uma cadeia de promessas, um erro fará com que os manipuladores resolve sejam ignorados:

throwErrorAsync()
  .then(() => { /* never called */ })
  .catch(error => { /* handle error here */ });

O mesmo se aplica às suas funções then. Se um manipulador resolve lançar uma exceção, o próximo manipulador reject será invocado:

doSomethingAsync()
  .then(result => { throwErrorSync(); })
  .then(() => { /* never called */ })
  .catch(error => { /* handle error from throwErrorSync() */ });

Um manipulador de erros retorna uma nova promessa, permitindo que você continue uma cadeia de promessas. A promessa retornada pelo manipulador de erros é resolvida com o valor retornado pelo manipulador:

throwErrorAsync()
  .catch(error => { /* handle error here */; return result; })
  .then(result => { /* handle result here */ });

Você pode deixar um erro cair em cascata em uma cadeia de promessas lançando novamente o erro:

throwErrorAsync()
  .catch(error => {
      /* handle error from throwErrorAsync() */
      throw error;
  })
  .then(() => { /* will not be called if there's an error */ })
  .catch(error => { /* will get called with the same error */ });

É possível lançar uma exceção que não é tratada pela promessa envolvendo a instrução throw dentro de um retorno de chamada setTimeout:

new Promise((resolve, reject) => {
  setTimeout(() => { throw new Error(); });
});

Isso funciona porque as promessas não podem manipular exceções lançadas de forma assíncrona.

Rejeições não tratadas #

Um erro será ignorado silenciosamente se uma promessa não tiver um bloco catch ou um manipulador reject:

throwErrorAsync()
  .then(() => { /* will not be called */ });
// error silently ignored

Para evitar isso, sempre use um bloco catch:

throwErrorAsync()
  .then(() => { /* will not be called */ })
  .catch(error => { /* handle error*/ });
// or
throwErrorAsync()
  .then(() => { /* will not be called */ }, error => { /* handle error*/ });

Alternativamente, assine o evento unhandledrejection para capturar quaisquer promessas rejeitadas não tratadas:

window.addEventListener('unhandledrejection', event => {});

Algumas promessas podem lidar com sua rejeição mais tarde do que o tempo de criação. O evento rejectionhandled é acionado sempre que tal promessa é tratada:

window.addEventListener('unhandledrejection', event => console.log('unhandled'));
window.addEventListener('rejectionhandled', event => console.log('handled'));
var p = Promise.reject('test');

setTimeout(() => p.catch(console.log), 1000);

// Will print 'unhandled', and after one second 'test' and 'handled'

O argumento evento contém informações sobre a rejeição. event.reason é o objeto de erro e event.promise é o objeto de promessa que causou o evento.

No Nodejs os eventos rejectionhandled e unhandledrejection são chamados de rejectionHandled e unhandledRejection no process, respectivamente, e têm uma assinatura diferente:

process.on('rejectionHandled', (reason, promise) => {});
process.on('unhandledRejection', (reason, promise) => {});

O argumento reason é o objeto de erro e o argumento promise é uma referência ao objeto de promessa que causou o disparo do evento.

O uso desses eventos unhandledrejection e rejectionhandled deve ser considerado apenas para fins de depuração. Normalmente, todas as promessas devem lidar com suas rejeições.

Observação: atualmente, apenas o Chrome 49+ e o Node.js são compatíveis com eventos unhandledrejection e rejectionhandled.

Ressalvas #

Encadeamento com fulfill e reject #

A função then(fulfill, reject) (com ambos os parâmetros não null) tem um comportamento único e complexo, e não deve ser usada a menos que você saiba exatamente como ela funciona.

A função funciona como esperado se for fornecido null para uma das entradas:

// the following calls are equivalent
promise.then(fulfill, null) 
promise.then(fulfill)

// the following calls are also equivalent
promise.then(null, reject) 
promise.catch(reject)

No entanto, ele adota um comportamento único quando ambas as entradas são fornecidas:

// the following calls are not equivalent!
promise.then(fulfill, reject)
promise.then(fulfill).catch(reject)

// the following calls are not equivalent!
promise.then(fulfill, reject)
promise.catch(reject).then(fulfill)

A função then(fulfill, reject) parece ser um atalho para then(fulfill).catch(reject), mas não é, e causará problemas se usado de forma intercambiável. Um desses problemas é que o manipulador reject não trata os erros do manipulador fulfill. Aqui está o que vai acontecer:

Promise.resolve() // previous promise is fulfilled
    .then(() => { throw new Error(); }, // error in the fulfill handler
        error => { /* this is not called! */ });

O código acima resultará em uma promessa rejeitada porque o erro é propagado. Compare-o com o código a seguir, que resulta em uma promessa cumprida:

Promise.resolve() // previous promise is fulfilled
    .then(() => { throw new Error(); }) // error in the fulfill handler
    .catch(error => { /* handle error */ });

Um problema semelhante existe ao usar then(fulfill, reject) de forma intercambiável com catch(reject).then(fulfill), exceto com a propagação de promessas cumpridas em vez de promessas rejeitadas.

Lançamento síncrono da função que deve retornar uma promessa #

Imagine uma função assim:

function foo(arg) {
  if (arg === 'unexepectedValue') {
    throw new Error('UnexpectedValue')
  }

  return new Promise(resolve => 
    setTimeout(() => resolve(arg), 1000)
  )
}

Se essa função for usada no meio de uma cadeia de promessas, aparentemente não há problema:

makeSomethingAsync().
  .then(() => foo('unexpectedValue'))
  .catch(err => console.log(err)) // <-- Error: UnexpectedValue will be caught here

No entanto, se a mesma função for chamada fora de uma cadeia de promessas, o erro não será tratado por ela e será lançado para o aplicativo:

foo('unexpectedValue') // <-- error will be thrown, so the application will crash
  .then(makeSomethingAsync) // <-- will not run
  .catch(err => console.log(err)) // <-- will not catch

Existem 2 soluções possíveis:

Retorna uma promessa rejeitada com o erro #

Em vez de jogar, faça o seguinte:

function foo(arg) {
  if (arg === 'unexepectedValue') {
    return Promise.reject(new Error('UnexpectedValue'))
  }

  return new Promise(resolve => 
    setTimeout(() => resolve(arg), 1000)
  )
}

Envolva sua função em uma cadeia de promessas #

Sua instrução throw será capturada corretamente quando já estiver dentro de uma cadeia de promessas:

function foo(arg) {
  return Promise.resolve()
    .then(() => {
      if (arg === 'unexepectedValue') {
        throw new Error('UnexpectedValue')
      }

      return new Promise(resolve => 
        setTimeout(() => resolve(arg), 1000)
      )
    })
}

Reconciliando operações síncronas e assíncronas #

Em alguns casos, você pode querer envolver uma operação síncrona dentro de uma promessa para evitar a repetição em ramificações de código. Tome este exemplo:

if (result) { // if we already have a result
  processResult(result); // process it
} else {
  fetchResult().then(processResult);
}

As ramificações síncronas e assíncronas do código acima podem ser reconciliadas envolvendo de forma redundante a operação síncrona dentro de uma promessa:

var fetch = result
  ? Promise.resolve(result)
  : fetchResult();

fetch.then(processResult);

Ao armazenar em cache o resultado de uma chamada assíncrona, é preferível armazenar em cache a promessa em vez do resultado em si. Isso garante que apenas uma operação assíncrona seja necessária para resolver várias solicitações paralelas.

Deve-se tomar cuidado para invalidar valores armazenados em cache quando forem encontradas condições de erro.

// A resource that is not expected to change frequently
var planets = 'http://swapi.co/api/planets/';
// The cached promise, or null
var cachedPromise;

function fetchResult() {
    if (!cachedPromise) {
        cachedPromise = fetch(planets)
            .catch(function (e) {
                // Invalidate the current result to retry on the next fetch
                cachedPromise = null;
                // re-raise the error to propagate it to callers
                throw e;
            });
    }
    return cachedPromise;
}

Chamada de função de atraso #

O método setTimeout() chama uma função ou avalia uma expressão após um número especificado de milissegundos. É também uma maneira trivial de obter uma operação assíncrona.

Neste exemplo, chamar a função wait resolve a promessa após o tempo especificado como primeiro argumento:

function wait(ms) {
    return new Promise(resolve => setTimeout(resolve, ms));  
}

wait(5000).then(() => { 
    console.log('5 seconds have passed...');
});

Valores “promissores” #

O método estático Promise.resolve pode ser usado para envolver valores em promessas.

let resolved = Promise.resolve(2);
resolved.then(value => {
    // immediately invoked
    // value === 2
});

Se value já for uma promessa, Promise.resolve simplesmente a reformula.

let one = new Promise(resolve => setTimeout(() => resolve(2), 1000));
let two = Promise.resolve(one);
two.then(value => {
    // 1 second has passed
    // value === 2
});

Na verdade, value pode ser qualquer “thenable” (objeto que define um método then que funciona suficientemente como uma promessa compatível com especificações). Isso permite que Promise.resolve converta objetos de terceiros não confiáveis ​​em promessas de terceiros confiáveis.

let resolved = Promise.resolve({
    then(onResolved) {
        onResolved(2);
    }
});
resolved.then(value => {
    // immediately invoked
    // value === 2
});

O método estático Promise.reject retorna uma promessa que imediatamente rejeita com o motivo fornecido.

let rejected = Promise.reject("Oops!");
rejected.catch(reason => {
    // immediately invoked
    // reason === "Oops!"
});

Usando ES2017 assíncrono/aguardando #

O mesmo exemplo acima, Image loading, pode ser escrito usando funções assíncronas. Isso também permite usar o método comum try/catch para tratamento de exceções.

Observação: a partir de abril de 2017, as versões atuais de todos os navegadores, exceto o Internet Explorer, oferecem suporte a funções assíncronas.

function loadImage(url) {
    return new Promise((resolve, reject) => {
        const img = new Image();
        img.addEventListener('load', () => resolve(img));
        img.addEventListener('error', () => {
            reject(new Error(`Failed to load ${url}`));
        });
        img.src = url;
    });
}

(async () => {

    // load /image.png and append to #image-holder, otherwise throw error
    try {
        let img = await loadImage('http://example.com/image.png');
        document.getElementById('image-holder').appendChild(img);
    }
    catch (error) {
        console.error(error);
    }

})();

forEach com promessas #

É possível aplicar efetivamente uma função (cb) que retorna uma promessa para cada elemento de um array, com cada elemento esperando para ser processado até que o elemento anterior seja processado.

function promiseForEach(arr, cb) {
  var i = 0;

  var nextPromise = function () {
    if (i >= arr.length) {
      // Processing finished.
      return;
    }

    // Process next function. Wrap in `Promise.resolve` in case
    // the function does not return a promise
    var newPromise = Promise.resolve(cb(arr[i], i));
    i++;
    // Chain to finish processing.
    return newPromise.then(nextPromise);
  };

  // Kick off the chain.
  return Promise.resolve().then(nextPromise);
};

Isso pode ser útil se você precisar processar milhares de itens com eficiência, um de cada vez. Usar um loop for regular para criar as promessas as criará de uma só vez e ocupará uma quantidade significativa de RAM.

Executando limpeza com finally() #

Existe atualmente uma proposta (ainda não faz parte do padrão ECMAScript) para adicionar um retorno de chamada finally às promessas que serão executadas independentemente de a promessa ser cumprida ou rejeitada. Semanticamente, isso é semelhante à cláusula finally do bloco try.

Você normalmente usaria esta funcionalidade para limpeza:

var loadingData = true;

fetch('/data')
    .then(result => processData(result.data))
    .catch(error => console.error(error))
    .finally(() => {
        loadingData = false;
    });

É importante notar que o callback finally não afeta o estado da promessa. Não importa qual valor ela retorne, a promessa permanece no estado cumprido/rejeitado que tinha antes. Então, no exemplo acima, a promessa será resolvido com o valor de retorno de processData(result.data) mesmo que o retorno de chamada finally tenha retornado undefined.

Com o processo de padronização ainda em andamento, sua implementação de promessas provavelmente não suportará retornos de chamada finally prontos para uso. Para retornos de chamada síncronos, você pode adicionar essa funcionalidade com um polyfill, no entanto:

if (!Promise.prototype.finally) {
    Promise.prototype.finally = function(callback) {
        return this.then(result => {
            callback();
            return result;
        }, error => {
            callback();
            throw error;
        });
    };
}

Solicitação de API assíncrona #

Este é um exemplo de uma simples chamada de API GET envolta em uma promessa para aproveitar sua funcionalidade assíncrona.

var get = function(path) {
  return new Promise(function(resolve, reject) {
    let request = new XMLHttpRequest();
    request.open('GET', path);
    request.onload = resolve;
    request.onerror = reject;
    request.send();
  });
};

Um tratamento de erros mais robusto pode ser feito usando as seguintes funções onload e onerror.

request.onload = function() {
  if (this.status >= 200 && this.status < 300) {
    if(request.response) {
      // Assuming a successful call returns JSON
      resolve(JSON.parse(request.response));
    } else {
      resolve();
  } else {
    reject({
      'status': this.status,
      'message': request.statusText
    });
  }
};

request.onerror = function() {
  reject({
    'status': this.status,
    'message': request.statusText
  });
};
