Escopo é o contexto no qual as variáveis ​​vivem e podem ser acessadas por outro código no mesmo escopo. Como o JavaScript pode ser amplamente usado como uma linguagem de programação funcional, conhecer o escopo de variáveis ​​e funções é importante, pois ajuda a evitar bugs e comportamentos inesperados em tempo de execução.

Fechamentos #

Quando uma função é declarada, as variáveis ​​no contexto de sua declaração são capturadas em seu escopo. Por exemplo, no código abaixo, a variável x é vinculada a um valor no escopo externo e, em seguida, a referência a x é capturada no contexto de bar:

var x = 4; // declaration in outer scope

function bar() {
    console.log(x); // outer scope is captured on declaration
}

bar(); // prints 4 to console

Saída de amostra: 4

Esse conceito de “capturar” escopo é interessante porque podemos usar e modificar variáveis ​​de um escopo externo mesmo após a saída do escopo externo. Por exemplo, considere o seguinte:

function foo() {
    var x = 4; // declaration in outer scope

    function bar() {
        console.log(x); // outer scope is captured on declaration
    }

    return bar;
    
    // x goes out of scope after foo returns
}

var barWithX = foo();
barWithX(); // we can still access x

Saída de amostra: 4

No exemplo acima, quando foo é chamado, seu contexto é capturado na função bar. Então, mesmo depois de retornar, bar ainda pode acessar e modificar a variável x. A função foo, cujo contexto é capturado em outra função, é chamada de fechamento.

Dados privados #

Isso nos permite fazer algumas coisas interessantes, como definir variáveis ​​"privadas" que são visíveis apenas para uma função específica ou conjunto de funções. Um exemplo artificial (mas popular):

function makeCounter() {
    var counter = 0;

    return {
        value: function () {
            return counter;
        },
        increment: function () {
            counter++;
        }
    };
}

var a = makeCounter();
var b = makeCounter();

a.increment();

console.log(a.value());
console.log(b.value());

Saída de amostra: <pré> 1 0

Quando makeCounter() é chamado, um instantâneo do contexto dessa função é salvo. Todo o código dentro de makeCounter() usará esse instantâneo em sua execução. Duas chamadas de makeCounter() criarão dois snapshots diferentes, com sua própria cópia de counter.

Expressões de função invocadas imediatamente (IIFE) #

Os fechamentos também são usados ​​para evitar a poluição do namespace global, geralmente por meio do uso de expressões de função invocadas imediatamente.

Expressões de função imediatamente invocadas (ou, talvez mais intuitivamente, funções anônimas auto-executáveis) são essencialmente closures que são chamadas logo após a declaração. A ideia geral dos IIFEs é invocar o efeito colateral de criar um contexto separado que seja acessível apenas ao código dentro do IIFE.

Suponha que queremos poder referenciar jQuery com $. Considere o método ingênuo, sem usar um IIFE:

var $ = jQuery;
// we've just polluted the global namespace by assigning window.$ to jQuery

No exemplo a seguir, um IIFE é usado para garantir que o $ seja vinculado ao jQuery apenas no contexto criado pelo encerramento:

(function ($) {
    // $ is assigned to jQuery here
})(jQuery);
// but window.$ binding doesn't exist, so no pollution

Consulte a resposta canônica no Stackoverflow para obter mais informações sobre fechamentos.

Elevação #

O que é içamento? #

Içamento é um mecanismo que move todas as declarações de variáveis ​​e funções para o topo de seu escopo. No entanto, as atribuições de variáveis ​​ainda acontecem onde estavam originalmente.

Por exemplo, considere o seguinte código:

console.log(foo);  // → undefined
var foo = 42;
console.log(foo);  // → 42

O código acima é o mesmo que:

var foo;             // → Hoisted variable declaration
console.log(foo);    // → undefined
foo = 42;            // → variable assignment remains in the same place
console.log(foo);    // → 42

Observe que, devido ao içamento, o undefined acima não é o mesmo que o notdefined resultante da execução:

console.log(foo);    // → foo is not defined

Um princípio semelhante se aplica às funções. Quando as funções são atribuídas a uma variável (ou seja, uma expressão de função), a declaração da variável é elevada enquanto a atribuição permanece no mesmo local. Os dois trechos de código a seguir são equivalentes.

console.log(foo(2, 3));     // → foo is not a function

var foo = function(a, b) {
    return a * b;
}

var foo;
console.log(foo(2, 3));     // → foo is not a function
foo = function(a, b) {
    return a * b;
}

Ao declarar instruções de função, ocorre um cenário diferente. Ao contrário das declarações de função, as declarações de função são elevadas ao topo de seu escopo. Considere o seguinte código:

console.log(foo(2, 3));  // → 6
function foo(a, b) {
    return a * b;
}

O código acima é o mesmo que o próximo trecho de código devido ao içamento:

function foo(a, b) {
    return a * b;
}

console.log(foo(2, 3));  // → 6

Aqui estão alguns exemplos do que é e do que não é içamento:

// Valid code:
foo();

function foo() {}

// Invalid code:
bar();                     // → TypeError: bar is not a function
var bar = function () {};


// Valid code:
foo();
function foo() {
    bar();
}
function bar() {}


// Invalid code:
foo();
function foo() {
    bar();                // → TypeError: bar is not a function
}
var bar = function () {};


// (E) valid:
function foo() {
    bar();
}
var bar = function(){};
foo();

Limitações de Içamento #

A inicialização de uma variável não pode ser Hoisted ou Em simples declarações de Hoists em JavaScript não pode ser inicializada.

Por exemplo: Os scripts abaixo fornecerão diferentes saídas.

var x = 2; 
var y = 4; 
alert(x + y);

Isso lhe dará uma saída de 6. Mas isso…

var x = 2; 
alert(x + y);
var y = 4;

Isso lhe dará uma saída de NaN. Como estamos inicializando o valor de y, o JavaScript Hoisting não está acontecendo, então o valor de y ficará indefinido. O JavaScript considerará que y ainda não foi declarado.

Portanto, o segundo exemplo é o mesmo abaixo.

var x = 2; 
var y;
alert(x + y);
y = 4;

Isso lhe dará uma saída de NaN.

Diferença entre var e let #

(Nota: Todos os exemplos usando let também são válidos para const)

var está disponível em todas as versões do JavaScript, enquanto let e const fazem parte do ECMAScript 6 e disponível apenas em alguns navegadores mais recentes.

var tem como escopo a função que o contém ou o espaço global, dependendo de quando é declarado:

var x = 4; // global scope

function DoThings() {
    var x = 7; // function scope
    console.log(x);
}

console.log(x); // >> 4
DoThings();     // >> 7
console.log(x); // >> 4

Isso significa que “escapa” instruções if e todas as construções de bloco semelhantes:

var x = 4;
if (true) {
    var x = 7;
}
console.log(x); // >> 7

for (var i = 0; i < 4; i++) {
    var j = 10;
}
console.log(i); // >> 4
console.log(j); // >> 10

Por comparação, let tem escopo de bloco:

let x = 4;

if (true) {
    let x = 7;
    console.log(x); // >> 7
}

console.log(x); // >> 4

for (let i = 0; i < 4; i++) {
    let j = 10;
}
console.log(i); // >> "ReferenceError: i is not defined"
console.log(j); // >> "ReferenceError: j is not defined"

Observe que i e j são declarados apenas no loop for e, portanto, não são declarados fora dele.

Existem várias outras diferenças cruciais:

Declaração de variável global #

No escopo superior (fora de quaisquer funções e blocos), as declarações var colocam um elemento no objeto global. let não:

var x = 4;
let y = 7;

console.log(this.x); // >> 4
console.log(this.y); // >> undefined

Re-declaração #

Declarar uma variável duas vezes usando var não produz um erro (mesmo que seja equivalente a declará-la uma vez):

var x = 4;
var x = 7;

Com let, isso produz um erro:

let x = 4;
let x = 7;

TypeError: Identificador x já foi declarado

O mesmo acontece quando y é declarado com var:

var y = 4;
let y = 7;

TypeError: Identificador y já foi declarado

No entanto, as variáveis ​​declaradas com let podem ser reutilizadas (não declaradas novamente) em um bloco aninhado

let i = 5;    
{
   let i = 6;
   console.log(i); // >> 6
}
console.log(i); // >> 5

Dentro do bloco, o i externo pode ser acessado, mas se o bloco interno tiver uma declaração let para i, o i externo não poderá ser acessado e lançará um ReferenceError se usado antes do segundo ser declarado.

let i = 5;
{
    i = 6;  // outer i is unavailable within the Temporal Dead Zone
    let i;
}

ReferenceError: i não está definido

Elevação #

Variáveis ​​declaradas com var e let são hoisted. A diferença é que uma variável declarada com var pode ser referenciada antes de sua própria atribuição, uma vez que ela é automaticamente atribuída (com undefined como seu valor), mas let não pode e requer especificamente que a variável seja declarada antes de ser invocado:

console.log(x); // >> undefined
console.log(y); // >> "ReferenceError: `y` is not defined"
//OR >> "ReferenceError: can't access lexical declaration `y` before initialization"
var x = 4;
let y = 7;

A área entre o início de um bloco e uma declaração let ou const é conhecida como [Temporal Dead Zone](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/ Statements/let#Temporal_dead_zone_and_errors_with_let), e quaisquer referências à variável nesta área causarão um ReferenceError. Isso acontece mesmo se a variável for atribuída antes de ser declarada:

y=7; // >> "ReferenceError: `y` is not defined"
let y;

No modo não estrito, atribuir um valor a uma variável sem qualquer declaração, declara automaticamente a variável no escopo global. Neste caso, ao invés de y ser declarado automaticamente no escopo global, let reserva o nome da variável (y) e não permite nenhum acesso ou atribuição a ela antes da linha onde ela é declarada/inicializada.

Sintaxe e invocação de aplicação e chamada. #

Os métodos apply e call em cada função permitem que ela forneça um valor personalizado para this.

function print() {
    console.log(this.toPrint);
}

print.apply({ toPrint: "Foo" }); // >> "Foo"
print.call({ toPrint: "Foo" }); // >> "Foo"

Você pode notar que a sintaxe para ambas as invocações usadas acima é a mesma. ou seja, a assinatura é semelhante.

Mas há uma pequena diferença em seu uso, já que estamos lidando com funções e alterando seus escopos, ainda precisamos manter os argumentos originais passados ​​para a função. Tanto apply quanto call suportam passar argumentos para a função de destino da seguinte forma:

function speak() {
    var sentences = Array.prototype.slice.call(arguments);
    console.log(this.name+": "+sentences);
}
var person = { name: "Sunny" };
speak.apply(person, ["I", "Code", "Startups"]); // >> "Sunny: I Code Startups"
speak.call(person, "I", "<3", "Javascript"); // >> "Sunny: I <3 Javascript"

Observe que apply permite que você passe um objeto Array ou arguments (tipo array) como a lista de argumentos, enquanto call precisa que você passe cada argumento separadamente.

Esses dois métodos dão a você a liberdade de ser tão sofisticado quanto quiser, como implementar uma versão ruim do bind nativo do ECMAScript para criar uma função que sempre será chamada como um método de um objeto de uma função original.

function bind (func, obj) { 
    return function () {
        return func.apply(obj, Array.prototype.slice.call(arguments, 1));
    }
}

var obj = { name: "Foo" };

function print() {
    console.log(this.name);
}

printObj = bind(print, obj);

printObj();

Isso irá registrar

“Foo”

A função bind tem muita coisa acontecendo

  1. obj será usado como o valor de this
  2. encaminhe os argumentos para a função
  3. e depois retorne o valor

Invocação da função de seta #

Ao usar funções de seta, this pega o valor do this do contexto de execução envolvente (ou seja, this nas funções de seta tem escopo léxico em vez do escopo dinâmico usual). Em código global (código que não pertence a nenhuma função) seria o objeto global. E continua assim, mesmo se você invocar a função declarada com a notação de seta de qualquer um dos outros métodos aqui descritos.

var globalThis = this; //"window" in a browser, or "global" in Node.js

var foo = (() => this);           

console.log(foo() === globalThis);          //true

var obj = { name: "Foo" };
console.log(foo.call(obj) === globalThis);  //true

Veja como this herda o contexto ao invés de se referir ao objeto no qual o método foi chamado.

var globalThis = this;

var obj = {
    withoutArrow: function() {
        return this;
    },
    withArrow: () => this
};

console.log(obj.withoutArrow() === obj);      //true
console.log(obj.withArrow() === globalThis);  //true

var fn = obj.withoutArrow; //no longer calling withoutArrow as a method
var fn2 = obj.withArrow;
console.log(fn() === globalThis);             //true
console.log(fn2() === globalThis);            //true

Invocação vinculada #

O método bind de cada função permite que você crie uma nova versão dessa função com o contexto estritamente vinculado a um objeto específico. É especialmente útil forçar uma função a ser chamada como um método de um objeto.

var obj = { foo: 'bar' };

function foo() {
    return this.foo;
}

fooObj = foo.bind(obj);

fooObj();

Isso registrará:

barra

Invocação do método #

Invocando uma função como método de um objeto, o valor de this será aquele objeto.

var obj = {
    name: "Foo",
    print: function () {
        console.log(this.name)
    }
}

Agora podemos invocar print como um método de obj. this será obj

obj.print();

Isso registrará assim:

Foo

Invocação anônima #

Invocando uma função como uma função anônima, this será o objeto global (self no navegador).

function func() {
    return this;
}

func() === window; // true

No modo estrito do ECMAScript 5, this será undefined se a função for invocada anonimamente.

(function () {
    "use strict";
    func();
}())

Isso vai sair

indefinido

Invocação do construtor #

Quando uma função é invocada como construtor com a palavra-chave new this pega o valor do objeto que está sendo construído

function Obj(name) {
    this.name = name;
}

var obj = new Obj("Foo");

console.log(obj);

Isso irá registrar

{ nome: “Foo” }

Usando let in loops em vez de var (exemplo de manipuladores de clique) #

Digamos que precisamos adicionar um botão para cada parte do array loadedData (por exemplo, cada botão deve ser um controle deslizante mostrando os dados; para simplificar, apenas alertaremos uma mensagem). Pode-se tentar algo assim:

for(var i = 0; i < loadedData.length; i++)
    jQuery("#container").append("<a class='button'>"+loadedData[i].label+"</a>")
        .children().last() // now let's attach a handler to the button which is a child
        .on("click",function() { alert(loadedData[i].content); });

Mas em vez de alertar, cada botão fará com que o

TypeError: loadData[i] é indefinido

erro. Isso ocorre porque o escopo de i é o escopo global (ou um escopo de função) e após o loop, i == 3. O que precisamos não é “lembrar o estado de i”. Isso pode ser feito usando let:

for(let i = 0; i < loadedData.length; i++)
    jQuery("#container").append("<a class='button'>"+loadedData[i].label+"</a>")
        .children().last() // now let's attach a handler to the button which is a child
        .on("click",function() { alert(loadedData[i].content); });

Um exemplo de loadedData para ser testado com este código:

    var loadedData = [
        { label:"apple",      content:"green and round" },
        { label:"blackberry", content:"small black or blue" },
        { label:"pineapple",  content:"weird stuff.. difficult to explain the shape" }
    ];
