Introdução OOP #

#Introdução A Programação Orientada a Objetos (principalmente conhecida como OOP) é ​​um paradigma de programação para resolver problemas.
A beleza de um programa OO (orientado a objetos) é que pensamos no programa como um monte de objetos se comunicando entre si, em vez de um script sequencial seguindo ordens específicas.

Existem muitas linguagens de programação que suportam OOP, algumas das mais populares são:

-Java

  • C++
  • c#

O Python também é conhecido por oferecer suporte a OOP, mas não possui algumas propriedades.

Terminologia OOP #

O termo mais básico em POO é uma classe.
Uma classe é basicamente um objeto, que possui um estado e funciona de acordo com seu estado.

Outro termo importante é uma instância.
Pense em uma classe como um modelo usado para criar instâncias de si mesmo. A classe é um template e a(s) instância(s) são os objetos concretos.

Uma instância criada a partir da classe A geralmente é chamada de ’tipo A’, exatamente como o tipo de 5 é int e o tipo de “abcd” é uma string.

Um exemplo de criação de uma instância chamada insance1 do tipo (class) ClassA:

###Java

ClasseA instância1 = new ClasseA();

C++ #

instância1 de classe A; ou

ClasseA * instância1 = new ClasseA(); # Na pilha

Python #

instância1 = ClasseA()

Como você pode ver no exemplo acima, em todos os casos o nome da classe foi mencionado e depois dele havia parênteses vazios (exceto para o C++ onde se estiverem vazios os parênteses podem ser eliminados). Nesses parênteses podemos passar argumentos para o construtor de nossa classe.

Um construtor é um método de uma classe que é chamado toda vez que uma instância é criada. Pode receber argumentos ou não. Se o programador não especificar nenhum construtor para uma classe que ele constrói, um construtor vazio será criado (um construtor que não faz nada).
Na maioria das linguagens o construtor é definido como um método sem definir seu tipo de retorno e com o mesmo nome da classe (exemplo em algumas seções).

Um exemplo de criação de uma instância chamada b1 do tipo (classe) ClassB. O construtor de ClassB recebe um argumento do tipo int:

###Java

ClasseA instância1 = new ClasseA(5); ou

int i = 5; ClasseA instância1 = new ClasseA(i);

C++ #

instância ClasseA1(5);

Python #

instância1 = ClasseA(5) Como você pode ver, o processo de criação de uma instância é muito semelhante ao processo de chamada de uma função.

Funções vs Métodos #

Ambas as funções e métodos são muito semelhantes, mas no Design Orientado a Objetos (OOD) cada um tem seu próprio significado.
Um método é uma operação executada em uma instância de uma classe. O próprio método geralmente usa o estado da instância para operar.
Enquanto isso, uma função pertence a uma classe e não a uma instância específica. Isso significa que ele não usa o estado da classe ou quaisquer dados armazenados em uma instância.

A partir de agora vamos mostrar nossos exemplos apenas em Java já que a POO é muito clara nesta linguagem, mas os mesmos princípios funcionam em qualquer outra linguagem POO.

Em Java, uma função tem a palavra static em sua definição, assim:

// File's name is ClassA
public static int add(int a, int b) {
    return a + b;
}

Isso significa que você pode chamá-lo de qualquer lugar no script.

// From the same file
System.out.println(add(3, 5));

// From another file in the same package (or after imported)
System.out.println(ClassA.add(3, 5));

Quando chamamos a função de outro arquivo usamos o nome da classe (em Java este também é o nome do arquivo) ao qual ela pertence, isso dá a intuição de que a função pertence à classe e não a nenhuma de suas instâncias.

Em contraste, podemos definir um método em ClassA assim:

// File's name is ClassA
public int subtract(int a, int b){
    return a - b;
}

Após esta declaração podemos chamar este método assim:

ClassA a = new ClassA();
System.out.println(a.subtract(3, 5));

Aqui precisávamos criar uma instância de ClassA para chamar seu método subtrair. Observe que NÃO PODEMOS fazer o seguinte:

System.out.println(ClassA.subtract(3, 5));

Esta linha produzirá um erro de compilação reclamando que chamamos este método não estático sem uma instância.

Usando o estado de uma classe #

Vamos supor que queremos implementar nosso método subtract novamente, mas desta vez queremos sempre subtrair o mesmo número (para cada instância). Podemos criar a seguinte classe:

class ClassB {

    private int sub_amount;

    public ClassB(int sub_amount) {
        this.sub_amount = sub_amount;
    }

    public int subtract(int a) {
        return a - sub_amount;
    }

    public static void main(String[] args) {
        ClassB b = new ClassB(5);
        System.out.println(b.subtract(3)); // Ouput is -2
    }
}

Quando executamos este código, uma nova instância chamada b da classe ClassB é criada e seu construtor é alimentado com o valor 5.
O construtor agora pega o sub_amount fornecido e o armazena como seu próprio campo privado, também chamado de sub_amount (esta convenção é muito conhecida em Java, para nomear os argumentos da mesma forma que os campos).
Depois disso, imprimimos no console o resultado da chamada do método subtract em b com o valor de 3.

Observe que na implementação de subtract não usamos this. como no construtor.
Em Java, this só precisa ser escrito quando houver outra variável com o mesmo nome definida nesse escopo. O mesmo funciona com o self do Python.
Então, quando usamos sub_amount em subtrair, fazemos referência ao campo privado que é diferente para cada classe.

Outro exemplo para enfatizar.
Vamos apenas alterar a função main no código acima para o seguinte:

ClassB b1 = new ClassB(1);
ClassB b2 = new ClassB(2);

System.out.println(b1.subtract(10)); // Output is 9
System.out.println(b2.subtract(10)); // Output is 8

Como podemos ver, b1 e b2 são independentes e cada um tem seu próprio estado.

Interfaces e Herança #

Uma interface é um contrato, ele define quais métodos uma classe terá e, portanto, suas capacidades. Uma interface não tem uma implementação, ela apenas define o que precisa ser feito.
Um exemplo em Java seria:

interface Printalbe {
    public void print();
}

A interface Printalbe define um método chamado print mas não fornece sua implementação (bem estranho para Java). Toda classe que se declara ‘implementando’ essa interface deve fornecer uma implementação para o método draw. Por exemplo:

class Person implements Printalbe {

    private String name;

    public Person(String name) {
        this.name = name;
    }

    public void print() {
        System.out.println(name);
    }
}

Se Person se declarasse implementando Drawable mas não fornecesse uma implementação para print, haveria um erro de compilação e o programa não compilaria.

Herança é um termo que aponta para uma classe estendendo outra classe. Por exemplo, digamos que agora temos uma pessoa que tem uma idade. Uma maneira de implementar uma pessoa como essa seria copiar a classe Person e escrever uma nova classe chamada AgedPerson que tem os mesmos campos e métodos, mas tem outra propriedade -age.
Isso seria terrível, pois duplicamos nosso código inteiro apenas para adicionar um recurso simples à nossa classe.
Podemos usar herança para herdar de Person e, assim, obter todos os seus recursos e aprimorá-los com nosso novo recurso, assim:

class AgedPerson extends Person {

    private int age;

    public AgedPerson(String name, int age) {
        super(name);
        this.age = age;
    }

    public void print() {
        System.out.println("Name: " + name + ", age:" + age);
    }
}

Tem algumas novidades acontecendo:

  • Usamos a palavra salva extends para indicar que estamos herdando de Person (e também sua implementação para Printable, então não precisamos declarar implementing Printable novamente).
  • Usamos a palavra de salvamento super para chamar o construtor de Person.
  • Substituímos o método print de Person por um novo.

Isso está ficando muito técnico em Java, então não vou me aprofundar neste tópico. Mas vou mencionar que existem muitos casos extremos que devem ser aprendidos sobre herança e interfaces antes de começar a usá-los. Por exemplo, quais métodos e funções são herdados? O que acontece com os campos privados/públicos/protegidos ao herdar de uma classe? e assim por diante.

Aula de Resumos #

Uma classe abstrata é um termo bastante avançado em OOP que descreve uma combinação de interfaces e herança. Ele permite que você escreva uma classe que tenha métodos/funções implementados e não implementados. Em Java isso é feito usando a palavra-chave abstract e não vou explicar mais do que um exemplo rápido:

abstract class AbstractIntStack {

    abstract public void push(int element);

    abstract public void pop();

    abstract public int top();

    final public void replaceTop(int element) {
        pop();
        push(element);
    }
}

Nota: a palavra-chave final afirma que você não pode substituir este método quando herdar desta classe. Se uma classe é declarada final, então nenhuma classe pode herdar dela.

Introdução #

OOP - Object Oriented Programming é um paradigma de programação amplamente utilizado nos dias de hoje. Em OOP, modelamos problemas do mundo real usando Objetos e seus comportamentos, para resolvê-los de forma programática.

Existem quatro principais OOP Concepts

  1. Herança
  2. Polimorfismo
  3. Abstração
  4. Encapsulamento

Esses quatro conceitos juntos são usados ​​para desenvolver programas em OOP.
