Introdução #

Complementos: #

Os complementos do Firefox geralmente são agrupados em Extensões e, em seguida, em “outros tipos” de complementos do Firefox.

Extensões #

As extensões permitem que o Firefox seja personalizado adicionando ou modificando a funcionalidade do Firefox. Alguns dos tipos de coisas que podem ser feitas com extensões incluem:

  • Altere como sites específicos aparecem, seu conteúdo ou como eles interagem.
  • Personalize a interface de usuário do Firefox
  • Adicione recursos adicionais ao Firefox
  • Altere como os recursos existentes do Firefox funcionam

As extensões do Firefox são, principalmente, escritas em JavaScript com a adição de algumas APIs JavaScript.

Descontinuação e remoção de todos os tipos de extensões que não sejam WebExtensions

Os complementos do Firefox, principalmente as extensões, estão em um estado de fluxo no momento. A Mozilla anunciou e confirmou que eles descontinuaram todos os tipos de extensões do Firefox, exceto WebExtensions, e que todas as extensões não baseadas em WebExtensions serão desativadas no Firefox 57, que está programado para 2017-11-14.

Tipos de extensões #

O Firefox tem quatro tipos de extensões (todos os quais são comumente chamados de complementos):

  • WebExtensions: No futuro, as WebExtensions são o único tipo de extensão do Firefox que será suportada. Esses complementos são descritos por um arquivo manifest.json. Essa API é semelhante ao que é usado para extensões do Google Chrome. Esses complementos usam HTML e CSS além de Javascript. Embora a Mozilla tenha declarado que esta API é o futuro das extensões do Firefox, esta API ainda está em desenvolvimento. Por enquanto, é melhor desenvolver e testar seu complemento WebExtension com o Firefox Developer Edition ou Firefox Nightly. Você também deve anotar cuidadosamente qual versão do Firefox é necessária para a funcionalidade que deseja usar. Essas informações estão contidas na seção “Compatibilidade do navegador” das páginas de documentação do MDN.

WebExtensions usam uma API significativamente diferente dos outros três tipos de extensões. Não há, intencionalmente, capacidade de usar as interfaces fornecidas por qualquer um dos outros tipos de complemento.

  • SDK Add-on: [obsoleto; agendado para remoção] Esses complementos são descritos por um arquivo package.json que é gerado inicialmente executando jpm init. Essas extensões geralmente usam require() para carregar APIs High-Level ou Low-Level para fazer interface com o Firefox. Esses complementos usam HTML e CSS além de Javascript. Atualmente, esses complementos são agrupados em uma extensão bootstrap quando são carregados para teste pelo jpm run ou consolidados em um arquivo .xpi pelo jpm xpi para distribuição (ou seja, upload para AMO/Mozilla). Em outras palavras, eles são extensões bootstrap com um wrapper SDK.

A Mozilla parece estar comprometida em continuar a oferecer suporte a extensões baseadas em SDKs complementares, desde que a extensão não use require("chrome") ou dependa de XUL, XPCOM, e XBL.

A maioria das coisas que podem ser feitas em uma extensão bootstrap podem ser feitas em uma extensão baseada em Add-on SDK. No entanto, muitas dessas coisas ignoram o SDK, o que perde uma parte significativa dos benefícios de usar o SDK Add-on.

  • Bootstrapped: [obsoleto; agendada para remoção] Essas extensões também são comumente chamadas de “restartless” porque foram o primeiro tipo de extensão do Mozilla que não exigia que o aplicativo fosse reiniciado para carregar/descarregar o complemento. No entanto, restartless é um descritor de como eles funcionam. Usar “restartless” como o nome para esse tipo de complemento é confuso porque os complementos SDK de complemento e WebExtension também não exigem que o aplicativo seja reiniciado ao carregar ou descarregar o complemento. Por esse motivo, há uma tendência de não usar mais “restartless” como nome para esse tipo de complemento.

Esses complementos usam HTML e CSS além de Javascript. Muitos também interagem com o Firefox usando XUL.

Esses complementos têm um arquivo JavaScript chamado bootstrap.js que deve conter pontos de entrada (funções) que são chamados para o complemento startup(), shutdown(), install() e desinstalar().

Esses complementos contêm um arquivo install.rdf que descreve o complemento. Eles geralmente, mas nem sempre, também contêm um arquivo chrome.manifest que descreve como os arquivos e diretórios na extensão se relacionam com o aplicativo Mozilla (por exemplo, Firefox).

A maioria, mas não todas as coisas que podem ser feitas em extensões overlay/XUL/Legacy podem ser realizadas em complementos bootstrap. Tudo o que pode ser feito no SDK Add-on pode ser feito em uma extensão bootstrap (as extensões do SDK Add-on são complementos bootstrap com algumas camadas de API baseadas em JavaScript).

A Mozilla afirmou que eles planejam descontinuar “complementos que dependem de XUL, XPCOM e XBL”. Embora nem todos os complementos de inicialização dependam dessas tecnologias, há uma tendência de os complementos de inicialização operarem em um nível mais baixo do que os complementos SDK de complemento e WebExtension. Assim, eles são mais propensos a usar essas tecnologias. Embora existam alguns que estão dizendo que todos os complementos de bootstrap estão planejados para serem preteridos, não está claro se esse é o caso. Afinal, as extensões do SDK Add-on não estão sendo preteridas (a menos que usem require("chrome") ou dependam de XUL, XPCOM ou XBL) e todas as extensões SDK Add-on são extensões bootstrap, apenas com um Envoltório do SDK.

  • Sobreposição/XUL/Legado: [obsoleto; agendado para remoção] Esses complementos contêm um arquivo install.rdf que descreve o complemento e um arquivo chrome.manifest para descrever como os arquivos do complemento se relacionam (por exemplo, sobreposição) aos arquivos do aplicativo. Como o complemento funciona com o aplicativo depende completamente dos relacionamentos descritos no arquivo chrome.manifest. As únicas exceções a isso são algumas coisas como ícones para a extensão e o arquivo que descreve as opções da extensão que são indicadas no arquivo install.rdf. Essas extensões interagem com o aplicativo (por exemplo, Firefox) em um nível muito baixo. Isso tende a torná-los mais propensos a quebrar quando são feitas alterações no aplicativo.

Esses complementos usam XUL, HTML e CSS além de Javascript. Alguns também usam XPCOM e XBL.

Todas as extensões Overlay/XUL/Legacy estão planejadas para serem descontinuadas.

Outros tipos de complementos do Firefox #

Quando a maioria das pessoas pensa nos complementos do Firefox, eles estão pensando nas extensões descritas acima. No entanto, existem alguns tipos adicionais de complementos do Firefox:

  • Temas [parcialmente obsoletos] permitem a personalização da aparência do Firefox. Principalmente fornecendo diferentes regras CSS para todas as partes do navegador. “Temas completos” estão obsoletos e planejados para serem parcialmente substituídos por uma interface atualmente inexistente.
  • Complementos móveis são para Firefox para Android. Atualmente, todos os tipos de extensão do Firefox sem reinicialização são suportados.
  • Plugins de mecanismo de pesquisa são usados ​​para adicionar mecanismos de pesquisa adicionais à barra de pesquisa do navegador.
  • Dicionários do usuário permitem o uso de verificação ortográfica em idiomas adicionais.
  • Pacotes de idiomas adicionam idiomas adicionais à interface de usuário do Firefox.
  • Plugins são bibliotecas compartilhadas para exibir conteúdo que o próprio aplicativo não pode exibir nativamente.

Instalando um complemento temporário #

Para testar um complemento que você está desenvolvendo, você provavelmente desejará instalá-lo temporariamente no Firefox. Você pode fazer isso carregando-o como um Temporary Add-on. Para fazer isso:

  1. Vá para sobre:depuração
  2. Clique em “Carregar Complemento Temporário”
  3. No seletor de arquivos, navegue até o diretório que contém os arquivos complementares
  4. Selecione qualquer arquivo na pasta
  5. Clique em “Abrir”

A animação a seguir mostra o carregamento de um complemento chamado “aaaaaaaaaaaaaaaaaa - demo add-on” de about:debugging e que o complemento aparece em about:addons:

Como um complemento temporário, você pode carregar um complemento descompactado (um diretório que contém todos os arquivos para o complemento que você empacotaria em um arquivo .xpi) ou um complemento que é empacotado em um arquivo .xpi. Complementos temporários não precisam ser assinados. O complemento temporário permanece instalado até que seja desinstalado manualmente ou o Firefox seja reiniciado.

Extensões da Web #

WebExtensions podem ser carregadas como Complementos Temporários. Isso pode ser feito com os arquivos complementares descompactados ou empacotados em um arquivo .xpi.

SDK de complemento do Firefox #

Você não pode carregar uma extensão SDK Add-on do Firefox como um add-on temporário sem primeiro empacotá-la em um arquivo .xpi com jpm xpi. Em geral, você usará jpm run para testar sua extensão SDK Add-on do Firefox.

Os arquivos que normalmente são editados para uma extensão SDK Add-on não fazem uma extensão completa sem algumas funções de encapsulamento adicionais e o arquivo package.json sendo traduzido em um install.rdf e, possivelmente, um arquivo chrome.manifest. Este processo envolve a extensão SDK Add-on em um add-on Bootstrap/Restartless, que é entendido pelo Firefox. Sem esse processo, o Firefox não poderá carregar o complemento. Este processo é executado pelo jpm xpi resultando em um arquivo .xpi compactado. A execução do jpm run também executa esse processo, mas armazena os arquivos resultantes em um local temporário e invoca o Firefox com o complemento instalado.

Bootstrap/Restartless #

Complementos Bootstrap/Restartless podem ser carregados como complementos temporários. Isso pode ser feito com os arquivos complementares descompactados ou empacotados em um arquivo .xpi.

Legado/Sobreposição/XUL #

Complementos Legacy/Overlay/XUL não podem ser carregados como complementos temporários.

Instalando complementos não assinados #

Para instalar uma extensão como um complemento normal nas versões Release ou Beta do Firefox superiores ou iguais à versão 48, a extensão deve ser assinada pela Mozilla. Uma extensão é assinada por submetendo-a à AMO. Uma vez assinada, a extensão pode ser instalada em qualquer versão do Firefox compatível. Para versões do Firefox anteriores à versão 43, as extensões não precisavam ser assinadas pela Mozilla. Tipos de complementos que não sejam extensões não precisam ser assinados pela Mozilla.

Você pode instalar extensões não assinadas como complementos normais em outras versões do Firefox (por exemplo, Firefox Developer Edition, [Firefox Nightly](https: //nightly.mozilla.org/), Unbranded Beta ou Unbranded Release) configurando xpinstall.signatures.required como false em about:config. A configuração dessa opção também foi efetiva nas versões 43–48 do Firefox Release e Beta. Definir esta opção não é eficaz nas versões Release e Beta do Firefox a partir da versão 48.

A necessidade de poder instalar complementos não assinados durante o desenvolvimento de complementos foi bastante reduzida pela disponibilidade de instalação de extensões como Complementos temporários. Complementos temporários não precisam ser assinados e podem ser carregados em qualquer versão atual do Firefox. Como o nome indica, a principal desvantagem dos Complementos Temporários é que eles são temporários. Eles devem ser reinstalados sempre que o Firefox for reiniciado. No entanto, há casos de uso em que é preferível instalar um complemento como um complemento normal, mas não assinado, em vez de um complemento temporário. Um complemento deve ser instalado como um complemento não assinado se for necessário que o complemento permaneça instalado mesmo depois que o Firefox for reiniciado. Isso pode ser desejável por vários motivos, incluindo: teste de uso a longo prazo ou para testar como o complemento funciona quando o Firefox é inicializado.

Instalando complementos para desenvolvimento #

Os complementos podem ser instalados como:

  1. Complementos normais, que são instalados até serem desinstalados
  2. Complementos temporários (somente extensões): são instalados apenas até que o Firefox seja reiniciado ou podem ser desinstalados manualmente antes.
  3. Usando jpm run (somente Add-on SDK): Executa automaticamente o Firefox usando um perfil temporário com seu complemento carregado.
  4. Usando web-ext run (somente WebExtensions): Executa automaticamente o Firefox usando um perfil temporário com seu complemento carregado como um complemento temporário. Por padrão, monitora seus arquivos de extensão para alterações e recarrega automaticamente sua extensão quando os arquivos são alterados.

Complementos normais #

Instalar extensões empacotadas (ou seja, o arquivo .xpi) pode ser uma simples questão de arrastá-lo e soltá-lo em uma janela do Firefox executando o perfil no qual você deseja instalá-lo. Eles também podem ser instalados diretamente baixando a extensão de AMO. Dependendo do seu objetivo (um perfil, todos os perfis, todos os usuários, qual SO, etc.), existem outras opções de como instalar extensões.

Essas outras opções incluem vários diretórios fora do diretório do perfil nos quais você pode colocar o arquivo .xpi para que seja instalado para todos os usuários de uma versão específica do Firefox ou todos os perfis de um usuário específico. No Windows, você também pode instalar uma extensão adicionando uma chave ao Registro do Windows. Em geral, esses outros diretórios não são aqueles nos quais você instalaria um complemento no qual está escrevendo no momento. No entanto, eles podem ser usados ​​para garantir que você tenha complementos usados ​​para suportar seus testes/desenvolvimento carregados em qualquer perfil do Firefox ou versão do Firefox que você usa. Por exemplo, colocando um arquivo .xpi em <diretório de instalação do Firefox>/browser/extensions você pode ter uma extensão disponível mesmo no perfil temporário criado por jpm run (usado para testar o Firefox Add-on SDK baseado extensões).

Para desenvolvimento/teste, você pode ter a extensão em qualquer diretório em sua unidade local usando um arquivo proxy de extensão Firefox (crie um arquivo chamado como <em:id> da extensão (em install.rdf para Bootstrap/Restartless e Overlay/Legacy) no diretório extensions do perfil contendo uma linha com o caminho completo para o diretório que contém os arquivos da extensão). Extensões instaladas dessa maneira quase sempre não serão assinadas (veja abaixo). Assim, este método não é muito útil se você deseja instalar a extensão em uma versão Release ou Beta do Firefox.

Complementos temporários (somente extensões) #

Extensões Web, extensões baseadas no SDK Add-on do Firefox e extensões Restartless/Bootstrap podem ser instaladas como Complementos temporários em about:debugging. As extensões podem ser carregadas como complementos temporários em qualquer versão atual do Firefox. Complementos temporários não precisam ser assinados e podem ser carregados descompactados (um diretório com arquivos) ou compactados (por exemplo, um arquivo .xpi).

Para obter informações detalhadas sobre a instalação temporária de extensões, consulte Instalando um complemento temporário.
