Regras de prioridade #

Várias regras de lexer podem corresponder ao mesmo texto de entrada. Nesse caso, o tipo de token será escolhido da seguinte forma:

  • Primeiro, selecione a regra lexer que corresponde à entrada mais longa
  • Se o texto corresponder a um token definido implicitamente (como '{'), use a regra implícita
  • Se várias regras de lexer corresponderem ao mesmo comprimento de entrada, escolha a primeira, com base na ordem de definição

A seguinte gramática combinada:

grammar LexerPriorityRulesExample;

// Parser rules

randomParserRule: 'foo'; // Implicitly declared token type
    
// Lexer rules
    
BAR: 'bar';
IDENTIFIER: [A-Za-z]+;
BAZ: 'baz';

WS: [ \t\r\n]+ -> skip;

Dada a seguinte entrada:

aaa foo bar baz barz

Produzirá a seguinte sequência de token do lexer:

IDENTIFIER 'foo' BAR IDENTIFIER IDENTIFIER
  • aaa é do tipo IDENTIFICADOR

Apenas a regra IDENTIFIER pode corresponder a este token, não há ambiguidade.

  • foo é do tipo 'foo'

A regra do analisador randomParserRule introduz o tipo de token implícito 'foo', que é prioritário sobre a regra IDENTIFIER.

  • bar é do tipo BAR

Este texto corresponde à regra BAR, que é definida antes da regra IDENTIFIER e, portanto, tem precedência.

  • baz é do tipo IDENTIFICADOR

Este texto corresponde à regra BAZ, mas também corresponde à regra IDENTIFIER. O último é escolhido como é definido antes de BAR.

Dada a gramática, BAZ nunca será capaz de corresponder, pois a regra IDENTIFIER já cobre tudo que BAZ pode corresponder.

  • barz é do tipo IDENTIFICADOR

A regra BAR pode corresponder aos 3 primeiros caracteres desta string (bar), mas a regra IDENTIFIER corresponde a 4 caracteres. Como IDENTIFIER corresponde a uma substring mais longa, ela é escolhida em vez de BAR.

Como regra geral, regras específicas devem ser definidas antes de regras mais genéricas. Se uma regra puder corresponder apenas a uma entrada que já esteja coberta por uma regra definida anteriormente, ela nunca será usada.

Regras definidas implicitamente como 'foo' agem como se fossem definidas antes de todas as outras regras do lexer.

Regras de lexer implícitas #

Quando tokens como '{' são usados ​​em uma regra parser, uma regra lexer implícita será criada para eles, a menos que exista uma regra explícita.

Em outras palavras, se você tiver uma regra lexer:

OPEN_BRACE: '{';

Então, essas duas regras do analisador são equivalentes:

parserRule: '{';
parserRule: OPEN_BRACE;

Mas se a regra lexer OPEN_BRACE não estiver definida, uma regra anônima implícita será criada. Nesse caso, a regra implícita será definida como se tivesse sido definida antes das outras regras: ela terá uma prioridade maior que as outras regras.

Ações e predicados semânticos #

Uma ação lexer é um bloco de código arbitrário no idioma de destino cercado por {…}, que é executado durante a correspondência:

IDENTIFIER: [A-Z]+ { log("matched rule"); };

Um predicado semântico é um bloco de código arbitrário no idioma de destino cercado por {…}?, que resulta em um valor booleano. Se o valor retornado for false, a regra do lexer será ignorada.

IDENTIFIER: [A-Z]+ { identifierIsValid() }?;

Predicados semânticos devem ser definidos no final da regra sempre que possível por motivos de desempenho.

Regras simples #

As regras do Lexer definem os tipos de token. Seu nome deve começar com uma letra maiúscula para distingui-los das regras do analisador.

INTEGER: [0-9]+;
IDENTIFIER: [a-zA-Z_] [a-zA-Z_0-9]*;

OPEN_PAREN: '(';
CLOSE_PAREN: ')';

Sintaxe básica:

Sintaxe Significado
A Corresponder à regra ou fragmento do lexer chamado “A”
A B Corresponder a A seguido por B
(A|B) Corresponde a A ou B
'texto' Corresponde ao literal “texto”
A? Corresponder a “A” zero ou uma vez
A* Corresponder a “A” zero ou mais vezes
A+ Corresponder a “A” uma ou mais vezes
[A-Z0-9] Corresponder a um caractere nos intervalos definidos (neste exemplo entre A-Z ou 0-9)
'a'..'z' Sintaxe alternativa para um intervalo de caracteres
~[A-Z] Negação de um intervalo - corresponde a qualquer caractere único não no intervalo
. Corresponder a qualquer caractere único

Fragmentos #

Fragmentos são partes reutilizáveis ​​de regras lexer que não podem corresponder por conta própria - eles precisam ser referenciados a partir de uma regra lexer.

INTEGER: DIGIT+
       | '0' [Xx] HEX_DIGIT+
       ;

fragment DIGIT: [0-9];
fragment HEX_DIGIT: [0-9A-Fa-f];

Comandos Lexer #

Uma regra lexer pode ter comandos associados:

WHITESPACE: [ \r\n] -> skip;

Os comandos são definidos após um -> no final da regra.

  • skip: Ignora o texto correspondente, nenhum token será emitido
  • channel(n): Emite o token em um canal diferente
  • type(n): Altera o tipo de token emitido
  • mode(n), pushMode(n), popMode, more: Controla os modos lexer
