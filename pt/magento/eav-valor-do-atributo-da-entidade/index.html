EAV (Valor do Atributo da Entidade)

  • implemente a interface de modelos de front-end, fonte e back-end de atributos

Nesta página

  • implemente a interface de modelos de front-end, fonte e back-end de atributos

Entidade

Armazena informações sobre o tipo de dados que estão sendo armazenados. No caso do Magento isso é cliente, produto, categoria, etc.

Atributo

As propriedades individuais de cada uma das entidades, e. nome, peso, endereço de e-mail etc.

Valor

O valor de uma determinada entidade e atributo. Por exemplo, podemos especificar a entidade do cliente e o atributo de email e, em seguida, atribuir a ele o valor hello@example.com.

Esquema de banco de dados

eav_entity

A tabela de entidades.

eav_entity_attribute

A tabela de atributos.

eav_entity_{type}

As tabelas de valores. Os tipos são datetime, decimals, int, text e varchar.

Importante observar que a tabela eav_entity_varchar tem o tipo varchar para valores, mesmo que date ou integer se ajuste melhor ao valor.

Modelos versus modelos de recursos

Todos os modelos dentro de Mage/Eav/Model/Resource são Mysql4 e são modelos de recursos.

Além disso, Entity/Abstract.php e Entity/Setup.php.

Plano versus EAV

Os modelos EAV são mais complexos, fornecendo lógica para salvar e carregar de várias tabelas, enquanto os modelos simples ou padrão são relativamente diretos (tradicionais).

Os modelos padrão gerenciam principalmente suas propriedades com setters e getters de dados trabalhando com uma única tabela. Os modelos EAV gerenciam principalmente seus modelos de atributos. Os modelos padrão apenas salvam seus dados em uma tabela e carregam a partir dela. Os modelos EAV carregam todos os atributos (ou um conjunto específico) após carregar os dados de base e salvam os atributos após salvar os dados (incluindo inserir, atualizar e excluir atributos).

Exemplos de modelo de recursos EAV

Para encontrar entidades usando o esquema de armazenamento EAV, a pesquisa da string extende Mage_Eav_Model_Entity_Abstract pode ser usada. Isso deve revelar todos os modelos de recursos baseados na estrutura EAV. No entanto, a lista resultante incluirá muitas classes obsoletas do módulo Mage_Sales que não estão mais sendo usadas.

Os únicos módulos que contêm entidades usando o esquema de armazenamento EAV são Mage_Catalog (categorias e produtos) e Mage_Customer (clientes e endereços).

Grupos de clientes usam o esquema de armazenamento de tabela simples. Todas as entidades de vendas foram convertidas em entidades de tabela plana com o lançamento do Magento 1.4.

O motivo pelo qual o EAV é usado é para que as entidades possam ter um número indeterminado de propriedades e, portanto, permanecerem flexíveis. Por exemplo, quando você adiciona um novo atributo a uma entidade Cliente (que é uma entidade EAV), a tabela do banco de dados não precisa ser alterada para que esse novo atributo seja adicionado.

Vantagens

Flexibilidade O esquema do banco de dados não precisa ser alterado com o modelo Rápido de implementar

Desvantagens

  • Ineficiente

Uma consulta retornando 20 colunas normalmente consistiria em 20 autojunções no EAV. No entanto, o módulo Mage_Eav geralmente não usa junções para carregar os dados do valor do atributo. Em vez disso, as seleções de união são usadas. As junções são usadas apenas para filtrar coleções de EAV.

  • Nenhum mecanismo para relacionamentos entre subtipos
  • Nenhum agrupamento de subtipos de entidade

Escopos do site e da loja

Para lidar com os valores de atributo de escopo do site e da loja no EAV, existe um valor store_id na entidade do catálogo para mostrar o escopo que vincula de volta ao core_store. Junto com as lojas normais (vistas da loja) há também uma loja ‘0’ que é o valor global. Quando estiver em uma loja específica, o sistema verificará primeiro um valor de entidade na loja atual e, em seguida, retornará à entidade global. As entidades Mage_Customer EAV não têm uma coluna de escopo store_id.

Inserir, atualizar e excluir

Para determinar se uma inserção, atualização ou exclusão precisa ser executada em um atributo, é feita uma comparação com o objeto original. O objeto original é basicamente uma duplicata do objeto de dados quando a entidade foi recuperada do banco de dados.

  • Se o atributo existe originalmente e seu novo valor não está vazio; ele atualiza.
  • Se o atributo existe originalmente, mas seu novo valor está definido como vazio; ele exclui. - Se o atributo não existir originalmente e seu novo valor não estiver vazio; ele insere.

Gerenciamento de Atributos

Modelos de atributos

Modelo de atributo Representa o atributo no formulário do banco de dados, sua lógica é padrão em todos os atributos e é difícil de alterar.

Modelo Frontal

A interface do atributo para o frontend e fornece qualquer lógica que o atributo requer no frontend, por exemplo, o método getUrl() em imagens.

Modelo de back-end

Eles executam a validação no atributo antes que ele seja salvo no banco de dados. Por exemplo, o modelo de back-end de senha converte a senha em um hash antes de ser salva. Ele também verifica se a senha e a confirmação da senha correspondem antes de salvar.

Modelos de origem

Usado para preencher as opções disponíveis para um atributo, por exemplo, catalog/product_status foi ativado e desativado.

Métodos obrigatórios

Um modelo de origem requer:

<?php
    public function getAllOptions();
    public function getOptionText($value);
?>

Normalmente, apenas getAllOptions() precisa ser implementado, pois já existe uma implementação para getOptionText() no modelo de origem abstrata Mage_Eav_Model_Entity_Attribute_Source_Abstract.

Um modelo frontend não requer o método getValue().

Um modelo de back-end requer:

<?php
    public function getTable();
    public function isStatic();
    public function getType();
    public function getEntityIdField();
    public function setValueId($valueId);
    public function getValueId();
    public function afterLoad($object);
    public function beforeSave($object);
    public function afterSave($object);
    public function beforeDelete($object);
    public function afterDelete($object);
    public function getEntityValueId($entity);
    public function setEntityValidId($entity, $valueId);
?>

Todos esses métodos são implementados no modelo abstrato de backend Mage_Eav_Model_Entity_Attribute_Backend_Abstract. Para modelos de back-end personalizados, apenas os métodos que exigem personalização precisam ser substituídos.

Modelos de origem de configuração do sistema

Não pode ser usado para atributos EAV. Os modelos de origem EAV implementam o método getAllOptions enquanto os modelos de origem adminhtml implementam o método toOptionArray().

Os modelos de origem de configuração do sistema padrão podem ser encontrados em Mage/Adminhtml/Model/System/Config/Source/.

Modelos de origem de atributos

A finalidade dos Modelos de Origem de Atributos é fornecer a lista de opções e valores para atributos de seleção e seleção múltipla. Eles também fornecem as informações da coluna para o indexador de tabela simples do catálogo, se necessário.

Para obter uma lista de todas as opções de um atributo, faça o seguinte:

<?php
    $options = $attribute->getSource()->getAllOptions(false);

    // or for admin
    $options = $_attribute->getSource()->getAllOptions(true, true);
?>

Modelos de atributo padrão

Se nenhuma classe for especificada como frontend, backend ou - para atributos de seleção ou multisseleção - modelos de origem, uma classe padrão será usada.

O modelo de front-end de atributo padrão é Mage_Eav_Model_Entity_Attribute_Frontend_Default.

O modelo de backend de atributo padrão depende do código de atributo e é determinado no método Mage_Eav_Model_Entity_Attribute::_getDefaultBackendModel().

<?php
    protected function _getDefaultBackendModel()
    {
        switch ($this->getAttributeCode()) {
            case 'created_at':
                return 'eav/entity_attribute_backend_time_created';

            case 'updated_at':
                return 'eav/entity_attribute_backend_time_updated';

            case 'store_id':
                return 'eav/entity_attribute_backend_store';

            case 'increment_id':
                return 'eav/entity_attribute_backend_increment';
        }

        return parent::_getDefaultBackendModel();
    }
?>

Se o método for até a última linha, Mage_Eav_Model_Entity_Attribute_Backend_Default será usado.

O modelo de origem padrão é definido em Mage_Eav_Model_Entity_Attribute_Source_Table. Isso é definido no modelo de atributo de módulos de catálogo. O modelo de origem de configuração padrão especificado no módulo eav nunca é usado.

Adicionar atributo

Para adicionar atributos EAV, use Mage_Eav_Model_Entity_Setup estendendo na classe de configuração.

adicionarAtributo() Cria os atributos, adiciona-os a grupos e conjuntos (incluindo padrão) ou atualiza se já existir atualizaçãoAtributo() Atualiza apenas os dados do atributo. As classes de configuração personalizadas podem ser usadas para estender esses métodos, adicionando dados adicionais ou simplificando os argumentos necessários.

Mesas planas

Os atributos de catálogo simples são gerenciados por indexadores:

Mage_Catalog_Model_Resource_Product_Flat_Indexer::updateAttribute() Mage_Catalog_Model_Resource_Category_Flat::synchronise() Os atributos do produto são adicionados à tabela plana se forem (consulte Mage_Catalog_Model_Resource_Product_Flat_Indexer::getAttributeCodes()):

Estático (tipo de back-end) Filtrável Usado na lista de produtos Usado para regras promocionais Usado para classificar por Atributos do sistema Há uma tabela plana diferente para cada loja, cada uma contém um valor de atributo de entidade com escopo de loja diferente. Os valores multilíngues são gerenciados com armazenamentos diferentes para cada idioma.

Interface Frontal

/**
* Entity attribute frontend interface
*
* Frontend is providing the user interface for the attribute
*
*/
interface Mage_Eav_Model_Entity_Attribute_Frontend_Interface
{

}

Interface de origem

/**
* Entity attribute select source interface
*
* Source is providing the selection options for user interface
*
*/
interface Mage_Eav_Model_Entity_Attribute_Source_Interface
{
/**
* Retrieve All options
*
* @return array
*/
public function getAllOptions();

/**
* Retrieve Option value text
*
* @param string $value
* @return mixed
*/
public function getOptionText($value);
}

Interface de back-end

/**
* Entity attribute backend interface
*
* Backend is responsible for saving the values of the attribute
* and performing pre and post actions
*
*/
interface Mage_Eav_Model_Entity_Attribute_Backend_Interface
{
public function getTable();
public function isStatic();
public function getType();
public function getEntityIdField();
public function setValueId($valueId);
public function getValueId();
public function afterLoad($object);
public function beforeSave($object);
public function afterSave($object);
public function beforeDelete($object);
public function afterDelete($object);

/**
* Get entity value id
*
* @param Varien_Object $entity
*/
public function getEntityValueId($entity);

/**
* Set entity value id
*
* @param Varien_Object $entity
* @param int $valueId
*/
public function setEntityValueId($entity, $valueId);
}
