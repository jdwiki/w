Esquemas de desserialização integrados #

SimpleStringSchema: SimpleStringSchema desserializa a mensagem como uma string. Caso suas mensagens tenham chaves, estas serão ignoradas.

new FlinkKafkaConsumer09<>(kafkaInputTopic, new SimpleStringSchema(), prop);

JSONDeserializationSchema

JSONDeserializationSchema desserializa mensagens formatadas em json usando jackson e retorna um fluxo de objetos com.fasterxml.jackson.databind.node.ObjectNode. Você pode então usar o método .get("property") para acessar os campos. Mais uma vez, as chaves são ignoradas.

new FlinkKafkaConsumer09<>(kafkaInputTopic, new JSONDeserializationSchema(), prop);

JSONKeyValueDeserializationSchema

JSONKeyValueDeserializationSchema é muito semelhante ao anterior, mas lida com mensagens com chaves e valores json codificados.

boolean fetchMetadata = true;
new FlinkKafkaConsumer09<>(kafkaInputTopic, new JSONKeyValueDeserializationSchema(fetchMetadata), properties);

O ObjectNode retornado contém os seguintes campos:

  • key: todos os campos presentes na chave
  • value: todos os campos da mensagem
  • (opcional) metadata: expõe o offset, partition e topic da mensagem (passa true para o construtor para buscar metadados também).

Por exemplo:

kafka-console-producer --broker-list localhost:9092 --topic json-topic \
    --property parse.key=true \
    --property key.separator=|
{"keyField1": 1, "keyField2": 2} | {"valueField1": 1, "valueField2" : {"foo": "bar"}}
^C

Será decodificado como:

{
    "key":{"keyField1":1,"keyField2":2},
    "value":{"valueField1":1,"valueField2":{"foo":"bar"}},
    "metadata":{
        "offset":43,
        "topic":"json-topic",
        "partition":0
    }
}

Partições Kafka e paralelismo Flink #

No kafka, cada consumidor do mesmo grupo de consumidores recebe uma ou mais partições. Observe que não é possível que dois consumidores consumam da mesma partição. O número de consumidores de flink depende do paralelismo de flink (o padrão é 1).

Existem três casos possíveis:

  1. partições kafka == paralelismo flink: este caso é ideal, pois cada consumidor cuida de uma partição. Se suas mensagens estiverem balanceadas entre as partições, o trabalho será distribuído uniformemente pelos operadores flink;

  2. partições kafka < flink parallelism: algumas instâncias do flink não receberão nenhuma mensagem. Para evitar isso, você precisa chamar rebalance em seu fluxo de entrada antes de qualquer operação, o que faz com que os dados sejam particionados novamente:

    inputStream = env.addSource(new FlinkKafkaConsumer10("topic", new SimpleStringSchema(), properties));

inputStream
    .rebalance()
    .map(s -> "message" + s)
    .print();
  1. partições kafka > paralelismo flink: neste caso, algumas instâncias irão lidar com múltiplas partições. Mais uma vez, você pode usar rebalance para distribuir mensagens uniformemente entre os trabalhadores.

Exemplo KafkaConsumer #

FlinkKafkaConsumer permite que você consuma dados de um ou mais tópicos kafka.

versões #

O consumidor a ser usado depende da sua distribuição kafka.

  • FlinkKafkaConsumer08: usa a antiga API SimpleConsumer do Kafka. As compensações são tratadas pelo Flink e enviadas ao tratador.
  • FlinkKafkaConsumer09: usa a nova API Consumer do Kafka, que lida com deslocamentos e reequilíbrio automaticamente.
  • FlinkKafkaProducer010: este conector suporta mensagens Kafka com timestamps tanto para produção quanto para consumo (útil para operações de janela).

uso #

Os binários não fazem parte do flink core, então você precisa importá-los:

<dependency>
  <groupId>org.apache.flink</groupId>
  <artifactId>flink-connector-kafka-0.${kafka.version}_2.10</artifactId>
  <version>RELEASE</version>
</dependency>

O construtor recebe três argumentos:

  • um ou mais tópicos para ler
  • um esquema de desserialização informando ao Flink como interpretar/decodificar as mensagens
  • propriedades de configuração do consumidor kafka. Esses são os mesmos que um consumidor “regular” de kafka. O mínimo exigido são:
    • bootstrap.servers: a comma separated list of Kafka brokers in the form ip:port. For version 8, use zookeeper.connect (list of zookeeper servers) instead
    • group.id: the id of the consumer group (see kafka documentation for more details)

Em Java:

Properties properties = new Properties();
properties.put("group.id", "flink-kafka-example");
properties.put("bootstrap.servers", "localhost:9092");

DataStream<String> inputStream = env.addSource( 
        new FlinkKafkaConsumer09<>(
            kafkaInputTopic, new SimpleStringSchema(), properties));

Em escala:

val properties = new Properties();
properties.setProperty("bootstrap.servers", "localhost:9092");
properties.setProperty("group.id", "test");

inputStream = env.addSource(
        new FlinkKafkaConsumer08[String](
            "topic", new SimpleStringSchema(), properties))

Durante o desenvolvimento, você pode usar as propriedades do kafka enable.auto.commit=false e ​​auto.offset.reset=earliest para consumir novamente os mesmos dados toda vez que iniciar seu programa.

Tolerância ao erro #

Conforme explicado nos documentos,

Com o checkpointing do Flink habilitado, o Flink Kafka Consumer consumirá registros de um tópico e verificará periodicamente todos os seus offsets Kafka, juntamente com o estado de outras operações, de maneira consistente. Em caso de falha no trabalho, o Flink restaurará o programa de streaming para o estado do último ponto de verificação e consumirá novamente os registros do Kafka, começando pelos deslocamentos armazenados no ponto de verificação.

O intervalo de pontos de verificação de desenho, portanto, define quanto o programa pode ter que voltar no máximo, em caso de falha.

Para usar consumidores Kafka tolerantes a falhas, você precisa habilitar o checkpointing no ambiente de execução usando o método enableCheckpointing:

final StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(5000); // checkpoint every 5 seconds
