Este tópico mostra como usar e configurar o log (log4j) em aplicativos Flink.

Configuração de registro #

Modo local #

No modo local, por exemplo, ao executar seu aplicativo a partir de um IDE, você pode configurar o log4j como de costume, ou seja, disponibilizando um log4j.properties no classpath. Uma maneira fácil no maven é criar log4j.properties na pasta src/main/resources. Aqui está um exemplo:

log4j.rootLogger=INFO, console

# patterns:
#  d = date
#  c = class
#  F = file
#  p = priority (INFO, WARN, etc)
#  x = NDC (nested diagnostic context) associated with the thread that generated the logging event
#  m = message


# Log all infos in the console
log4j.appender.console=org.apache.log4j.ConsoleAppender
log4j.appender.console.layout=org.apache.log4j.PatternLayout
log4j.appender.console.layout.ConversionPattern=%d{dd/MM/yyyy HH:mm:ss.SSS} %5p [%-10c] %m%n

# Log all infos in flink-app.log
log4j.appender.file=org.apache.log4j.FileAppender
log4j.appender.file.file=flink-app.log
log4j.appender.file.append=false
log4j.appender.file.layout=org.apache.log4j.PatternLayout
log4j.appender.file.layout.ConversionPattern=%d{dd/MM/yyyy HH:mm:ss.SSS} %5p [%-10c] %m%n

# suppress info messages from flink
log4j.logger.org.apache.flink=WARN

Modo autônomo #

No modo autônomo, a configuração real usada não é aquela em seu arquivo jar. Isso ocorre porque o Flink possui seus próprios arquivos de configuração, que têm precedência sobre os seus.

Arquivos padrão: O Flink é fornecido com os seguintes arquivos de propriedades padrão:

  • log4j-cli.properties: Usado pelo cliente de linha de comando Flink (por exemplo, flink run) (não é código executado no cluster)
  • log4j-yarn-session.properties: Usado pelo cliente de linha de comando Flink ao iniciar uma sessão YARN (yarn-session.sh)
  • log4j.properties: logs do JobManager/Taskmanager (tanto autônomo quanto YARN)

Observe que o padrão ${log.file} é flink/log. Ele pode ser substituído em flink-conf.yaml, definindo env.log.dir,

env.log.dir define o diretório onde os logs do Flink são salvos. Tem que ser um caminho absoluto.

Log location: os logs são local, ou seja, são produzidos na(s) máquina(s) que executam o(s) JobManager(s) / Taskmanager(es).

Yarn: ao executar o Flink no Yarn, você precisa confiar nos recursos de log do Hadoop YARN. O recurso mais útil para isso é a agregação de log YARN. Para habilitá-lo, defina a propriedade yarn.log-aggregation-enable como true no arquivo yarn-site.xml. Uma vez ativado, você pode recuperar todos os arquivos de log de uma sessão YARN (com falha) usando:

yarn logs -applicationId <application ID>

Infelizmente, os logs estão disponíveis somente depois que uma sessão parou de ser executada, por exemplo, após uma falha.

Usando um logger em seu código #

Adicione a dependência slf4j ao seu pom.xml:

<properties>
    <slf4j.version>1.7.21</slf4j.version>
</properties>

<!-- ... --> 

<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-api</artifactId>
    <version>${slf4j.version}</version>
</dependency>
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-log4j12</artifactId>
    <version>${slf4j.version}</version>
</dependency>

Crie um objeto logger para uso em sua classe:

private Logger LOGGER = LoggerFactory.getLogger(FlinkApp.class);

Em classes que precisam ser serializadas, como subclasses de RichMapFunction, não se esqueça de declarar LOGGER como transient:

private transient Logger LOG = LoggerFactory.getLogger(MyRichMapper.class);

Em seu código, use LOGGER como de costume. Use espaços reservados ({}) para formatar objetos e outros:

LOGGER.info("my app is starting");
LOGGER.warn("an exception occurred processing {}", record, exception);

Usando configurações diferentes para cada aplicativo #

Caso você precise de configurações diferentes para seus vários aplicativos, não há (a partir do Flink 1.2) uma maneira fácil de fazer isso.

Se você usar o modo one-yarn-cluster-per-job do flink (ou seja, você inicia seus scripts com: flink run -m yarn-cluster …), aqui está uma solução alternativa:

  1. crie um diretório conf em algum lugar próximo ao seu projeto

  2. crie links simbólicos para todos os arquivos em flink/conf:

     mkdir conf
 cd conf
 ln -s flink/conf/* .

  3. substitua o link simbólico log4j.properties (ou qualquer outro arquivo que você queira alterar) por sua própria configuração

  4. antes de iniciar seu trabalho, execute

     export FLINK_CONF_DIR=/path/to/my/conf

Dependendo da sua versão do flink, você pode precisar editar o arquivo flink/bin/config.sh. Se você correr por esta linha:

FLINK_CONF_DIR=$FLINK_ROOT_DIR_MANGLED/conf

altere com:

if [ -z "$FLINK_CONF_DIR" ]; then 
    FLINK_CONF_DIR=$FLINK_ROOT_DIR_MANGLED/conf; 
fi

Solução alternativa do Flink-on-Yarn: obtenha logs em tempo real com rsyslog #

Por padrão, o Yarn não agrega logs antes que um aplicativo seja concluído, o que pode ser problemático com trabalhos de streaming que nem terminam.

Uma solução é usar rsyslog, que está disponível na maioria das máquinas Linux.

Primeiro, permita solicitações udp recebidas descomentando as seguintes linhas em /etc/rsyslog.conf:

$ModLoad imudp
$UDPServerRun 514

Edite seu log4j.properties (veja os outros exemplos nesta página) para usar SyslogAppender:

log4j.rootLogger=INFO, file

# TODO: change package logtest to your package
log4j.logger.logtest=INFO, SYSLOG

# Log all infos in the given file
log4j.appender.file=org.apache.log4j.FileAppender
log4j.appender.file.file=${log.file}
log4j.appender.file.append=false
log4j.appender.file.layout=org.apache.log4j.PatternLayout
log4j.appender.file.layout.ConversionPattern=bbdata: %d{yyyy-MM-dd HH:mm:ss,SSS} %-5p %-60c %x - %m%n

# suppress the irrelevant (wrong) warnings from the netty channel handler
log4j.logger.org.jboss.netty.channel.DefaultChannelPipeline=ERROR, file

# rsyslog
# configure Syslog facility SYSLOG appender
# TODO: replace host and myTag by your own
log4j.appender.SYSLOG=org.apache.log4j.net.SyslogAppender
log4j.appender.SYSLOG.syslogHost=10.10.10.102
log4j.appender.SYSLOG.port=514
#log4j.appender.SYSLOG.appName=bbdata
log4j.appender.SYSLOG.layout=org.apache.log4j.EnhancedPatternLayout
log4j.appender.SYSLOG.layout.conversionPattern=myTag: [%p] %c:%L - %m %throwable %n

O layout é importante porque o rsyslog trata uma nova linha como uma nova entrada de log. Acima, as novas linhas (em stacktraces, por exemplo) serão ignoradas. Se você realmente quer que logs multilinhas/com abas funcionem “normalmente”, edite rsyslog.conf e adicione:

$EscapeControlCharactersOnReceive off

O uso de myTag: no início do conversionPattern é útil se você deseja redirecionar todos os seus logs para um arquivo específico. Para fazer isso, edite rsyslog.conf e adicione a seguinte regra:

if $programname == 'myTag' then /var/log/my-app.log
& stop
