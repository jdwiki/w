(testado no Flink 1.2 e abaixo)

Cada função, fonte ou operador no Flink pode ser stateful. Os pontos de verificação permitem que o Flink recupere o estado e as posições nos fluxos para fornecer ao aplicativo a mesma semântica de uma execução sem falhas. É o mecanismo por trás das garantias de tolerância a falhas e processamento exatamente uma vez.

Leia este artigo para entender os detalhes internos.

Os pontos de verificação são úteis apenas quando ocorre uma falha no cluster, por exemplo, quando um gerenciador de tarefas falha. Eles não persistem depois que o próprio trabalho falhou ou foi cancelado.

Para poder retomar um trabalho com estado após falha/cancelamento, dê uma olhada em savepoints ou pontos de verificação externos (flink 1.2+).

Configuração e configuração #

A configuração do ponto de verificação é feita em duas etapas. Primeiro, você precisa escolher um backend. Em seguida, você pode especificar o intervalo e o modo dos pontos de verificação por aplicativo.

back-ends #

Back-ends disponíveis

Onde os pontos de verificação são armazenados depende do back-end configurado:

  • MemoryStateBackend: estado na memória, backup na memória do JobManager/ZooKeeper. Deve ser usado apenas para estado mínimo (padrão para máx. 5 MB, para armazenar deslocamentos Kafka, por exemplo) ou teste e depuração local.
  • FsStateBackend: o estado é mantido na memória nos Gerenciadores de Tarefas e os instantâneos de estado (ou seja, pontos de verificação) são armazenados em um sistema de arquivos (HDFS, DS3, sistema de arquivos local, …). Essa configuração é recomendada para estados grandes ou janelas longas e para configurações de alta disponibilidade.
  • RocksDBStateBackend: mantém dados em vôo em um banco de dados RocksDB que é (por padrão) armazenado nos diretórios de dados do TaskManager. Após o checkpoint, todo o banco de dados RocksDB é gravado em um arquivo (como acima). Comparado ao FsStateBackend, ele permite estados maiores (limitados apenas pelo espaço em disco versus o tamanho da memória do gerenciador de tarefas), mas a taxa de transferência será menor (os dados nem sempre estão na memória, devem ser carregados do disco).

Observe que qualquer que seja o backend, os metadados (número de pontos de verificação, localização, etc.) são sempre armazenados na memória do gerenciador de tarefas e os pontos de verificação não persistirão após o término/cancelamento do aplicativo.

Especificando o back-end

Você especifica o backend no método main do seu programa usando:

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.setStateBackend(new FsStateBackend("hdfs://namenode:40010/flink/checkpoints"));

Ou defina o backend padrão em flink/conf/flink-conf.yaml:

# Supported backends: 
#  - jobmanager (MemoryStateBackend), 
#  - filesystem (FsStateBackend), 
#  - rocksdb (RocksDBStateBackend), 
#  - <class-name-of-factory>
state.backend: filesystem
    
# Directory for storing checkpoints in a Flink-supported filesystem
# Note: State backend must be accessible from the JobManager and all TaskManagers.
# Use "hdfs://" for HDFS setups, "file://" for UNIX/POSIX-compliant file systems, 
# "S3://" for S3 file system.
state.backend.fs.checkpointdir: file:///tmp/flink-backend/checkpoints

Ativando checkpoints #

Todo aplicativo precisa habilitar explicitamente os pontos de verificação:

long checkpointInterval = 5000; // every 5 seconds

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.enableCheckpointing(checkpointInterval);

Opcionalmente, você pode especificar um modo de ponto de verificação. Caso contrário, o padrão é exatamente uma vez:

env.enableCheckpointing(checkpointInterval, CheckpointingMode.AT_LEAST_ONCE);

O modo checkpointing define quais garantias de consistência o sistema oferece na presença de falhas. Quando o ponto de verificação é ativado, os fluxos de dados são reproduzidos de forma que partes perdidas do processamento sejam repetidas. Com EXACTLY_ONCE, o sistema desenha pontos de verificação de forma que uma recuperação se comporte como se os operadores/funções vissem cada registro “exatamente uma vez”. Com AT_LEAST_ONCE, os pontos de verificação são desenhados de uma forma mais simples que normalmente encontra algumas duplicatas na recuperação.

Pontos de verificação de teste #

O código #

Aqui está um aplicativo flink simples usando um mapeador de estado com um estado gerenciado Integer. Você pode brincar com as variáveis ​​checkpointEnable, checkpointInterval e checkpointMode para ver seu efeito:

public class CheckpointExample {

    private static Logger LOG = LoggerFactory.getLogger(CheckpointExample.class);
    private static final String KAFKA_BROKER = "localhost:9092";
    private static final String KAFKA_INPUT_TOPIC = "input-topic";
    private static final String KAFKA_GROUP_ID = "flink-stackoverflow-checkpointer";
    private static final String CLASS_NAME = CheckpointExample.class.getSimpleName();


    public static void main(String[] args) throws Exception {

        // play with them
        boolean checkpointEnable = false;
        long checkpointInterval = 1000;
        CheckpointingMode checkpointMode = CheckpointingMode.EXACTLY_ONCE;

        // ----------------------------------------------------

        LOG.info(CLASS_NAME + ": starting...");
        final StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // kafka source
        // https://ci.apache.org/projects/flink/flink-docs-release-1.2/dev/connectors/kafka.html#kafka-consumer
        Properties prop = new Properties();
        prop.put("bootstrap.servers", KAFKA_BROKER);
        prop.put("group.id", KAFKA_GROUP_ID);
        prop.put("auto.offset.reset", "latest");
        prop.put("enable.auto.commit", "false");

        FlinkKafkaConsumer09<String> source = new FlinkKafkaConsumer09<>(
                KAFKA_INPUT_TOPIC, new SimpleStringSchema(), prop);

        // checkpoints
        // internals: https://ci.apache.org/projects/flink/flink-docs-master/internals/stream_checkpointing.html#checkpointing
        // config: https://ci.apache.org/projects/flink/flink-docs-release-1.3/dev/stream/checkpointing.html
        if (checkpointEnable) env.enableCheckpointing(checkpointInterval, checkpointMode);

        env
                .addSource(source)
                .keyBy((any) -> 1)
                .flatMap(new StatefulMapper())
                .print();

        env.execute(CLASS_NAME);
    }

        /* *****************************************************************
         * Stateful mapper
         * (cf. https://ci.apache.org/projects/flink/flink-docs-release-1.3/dev/stream/state.html)
         * ****************************************************************/

    public static class StatefulMapper extends RichFlatMapFunction<String, String> {
        private transient ValueState<Integer> state;

        @Override
        public void flatMap(String record, Collector<String> collector) throws Exception {
            // access the state value
            Integer currentState = state.value();

            // update the counts
            currentState += 1;
            collector.collect(String.format("%s: (%s,%d)",
                    LocalDateTime.now().format(ISO_LOCAL_DATE_TIME), record, currentState));
            // update the state
            state.update(currentState);
        }

        @Override
        public void open(Configuration parameters) throws Exception {
            ValueStateDescriptor<Integer> descriptor =
                    new ValueStateDescriptor<>("CheckpointExample", TypeInformation.of(Integer.class), 0);
            state = getRuntimeContext().getState(descriptor);
        }
    }
}

Executando o exemplo e simulando a falha #

Para poder verificar os pontos de verificação, você precisa iniciar um cluster. A maneira mais fácil é usar o script start-cluster.sh no diretório flink/bin:

start-cluster.sh
Starting cluster.
[INFO] 1 instance(s) of jobmanager are already running on virusnest.
Starting jobmanager daemon on host virusnest.
Password:
Starting taskmanager daemon on host virusnest.

Agora, empacote seu aplicativo e envie-o para flink:

mvn clean package
flink run target/flink-checkpoints-test.jar -c CheckpointExample

Crie alguns dados:

kafka-console-producer --broker-list localhost:9092 --topic input-topic 
a
b
c
^D

A saída deve estar disponível em flink/logs/flink-<user>-jobmanager-0-<host>.out. Por exemplo:

tail -f flink/logs/flink-Derlin-jobmanager-0-virusnest.out
2017-03-17T08:21:51.249: (a,1)
2017-03-17T08:21:51.545: (b,2)
2017-03-17T08:21:52.363: (c,3)

Para testar os checkpoints, basta matar o taskmanager (isso irá emular uma falha), produzir alguns dados e iniciar um novo:

# killing the taskmanager
ps -ef | grep -i taskmanager
kill <taskmanager PID>

# starting a new taskmanager
flink/bin/taskmanager.sh start

Observação: ao iniciar um novo gerenciador de tarefas, ele usará outro arquivo de log, chamado flink/logs/flink-<usuário>-jobmanager-1-<host>.out (observe o incremento de inteiro).

O que esperar #

  • checkpoints desativados: se você produzir dados durante a falha, eles serão definitivamente perdidos. Mas surpreendentemente, os contadores estarão certos!
  • checkpoints habilitados: não há mais perda de dados (e contadores corretos).
  • pontos de verificação com modo pelo menos uma vez: você pode ver duplicatas, especialmente se definir um intervalo de ponto de verificação para um número alto e matar o gerenciador de tarefas várias vezes
