Ativando a comunicação remota do PowerShell #

A comunicação remota do PowerShell deve primeiro ser habilitada no servidor ao qual você deseja se conectar remotamente.

Enable-PSRemoting -Force

Este comando faz o seguinte:

  • Executa o cmdlet Set-WSManQuickConfig, que executa as seguintes tarefas:
  • Inicia o serviço WinRM.
  • Define o tipo de inicialização no serviço WinRM como Automático.
  • Cria um listener para aceitar solicitações em qualquer endereço IP, caso ainda não exista.
  • Ativa uma exceção de firewall para comunicações WS-Management.
  • Registra as configurações de sessão Microsoft.PowerShell e Microsoft.PowerShell.Workflow, caso ainda não estejam registradas.
  • Registra a configuração da sessão Microsoft.PowerShell32 em computadores de 64 bits, caso ainda não esteja registrado.
  • Habilita todas as configurações de sessão.
  • Altera o descritor de segurança de todas as configurações de sessão para permitir acesso remoto.
  • Reinicia o serviço WinRM para efetivar as alterações anteriores.

Apenas para ambientes sem domínio #

Para servidores em um domínio AD, a autenticação remota PS é feita por meio de Kerberos (‘Padrão’) ou NTLM (‘Negociar’). Se você deseja permitir a comunicação remota para um servidor que não seja de domínio, você tem duas opções.

Configure a comunicação WSMan sobre HTTPS (que requer geração de certificado) ou habilite a autenticação básica que envia suas credenciais através do fio codificado em base64 (que é basicamente o mesmo que texto simples, portanto, tenha cuidado com isso).

Em ambos os casos, você terá que adicionar os sistemas remotos à sua lista de hosts confiáveis ​​do WSMan.

Ativando a autenticação básica #

Set-Item WSMan:\localhost\Service\AllowUnencrypted $true

Então, no computador do qual você deseja se conectar de, você deve dizer a ele para confiar no computador ao qual você está se conectando ao.

Set-Item WSMan:\localhost\Client\TrustedHosts '192.168.1.1,192.168.1.2'

Set-Item WSMan:\localhost\Client\TrustedHosts *.contoso.com

Set-Item WSMan:\localhost\Client\TrustedHosts *

Importante: Você deve dizer ao seu cliente para confiar no computador endereçado da maneira que você deseja se conectar (por exemplo, se você se conectar via IP, ele deve confiar no IP e não no nome do host)

Conectando-se a um servidor remoto via PowerShell #

Usando credenciais do seu computador local:

Enter-PSSession 192.168.1.1

Solicitando credenciais no computador remoto

Enter-PSSession 192.168.1.1 -Credential $(Get-Credential)

Executar comandos em um computador remoto #

Depois que a comunicação remota do Powershell estiver habilitada (Enable-PSRemoting), você pode executar comandos no computador remoto como este:

Invoke-Command -ComputerName "RemoteComputerName" -ScriptBlock {
    Write host "Remote Computer Name: $ENV:ComputerName"
}

O método acima cria uma sessão temporária e a fecha logo após o término do comando ou scriptblock.

Para deixar a sessão aberta e executar outro comando nela posteriormente, você precisa primeiro criar uma sessão remota:

$Session = New-PSSession -ComputerName "RemoteComputerName"

Em seguida, você pode usar esta sessão sempre que invocar comandos no computador remoto:

Invoke-Command -Session $Session -ScriptBlock {
    Write host "Remote Computer Name: $ENV:ComputerName"
}

Invoke-Command -Session $Session -ScriptBlock {
    Get-Date
}

Se você precisar usar credenciais diferentes, poderá adicioná-las com o parâmetro -Credential:

$Cred = Get-Credential
Invoke-Command -Session $Session -Credential $Cred -ScriptBlock {...}

Uso do argumento #

Para usar argumentos como parâmetros para o bloco de script remoto, pode-se usar o parâmetro ArgumentList de Invoke-Command ou usar a sintaxe $Using:.

Usando ArgumentList com parâmetros sem nome (ou seja, na ordem em que são passados ​​para o scriptblock):

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ArgumentList $servicesToShow,$fileName -ScriptBlock {
    Write-Host "Calling script block remotely with $($Args.Count)"
    Get-Service -Name $args[0]
    Remove-Item -Path $args[1] -ErrorAction SilentlyContinue -Force
}

Usando ArgumentList com parâmetros nomeados:

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ArgumentList $servicesToShow,$fileName -ScriptBlock {
    Param($serviceToShowInRemoteSession,$fileToDelete)

    Write-Host "Calling script block remotely with $($Args.Count)"
    Get-Service -Name $serviceToShowInRemoteSession
    Remove-Item -Path $fileToDelete -ErrorAction SilentlyContinue -Force
}

Usando a sintaxe $Using::

$servicesToShow = "service1"
$fileName = "C:\temp\servicestatus.csv"
Invoke-Command -Session $session -ScriptBlock {
    Get-Service $Using:servicesToShow
    Remove-Item -Path $fileName -ErrorAction SilentlyContinue -Force
}

Uma prática recomendada para limpeza automática de PSSessions #

Quando uma sessão remota é criada por meio do cmdlet New-PSsession, a PSSession persiste até que a sessão atual do PowerShell termine. Isso significa que, por padrão, a PSSession e todos os recursos associados continuarão sendo usados ​​até que a sessão atual do PowerShell termine.

Várias PSSessions ativas podem se tornar uma sobrecarga nos recursos, principalmente para scripts de longa execução ou interligados que criam centenas de PSSessions em uma única sessão do PowerShell.

É uma prática recomendada remover explicitamente cada PSSession depois que ela terminar de ser usada. [1]

O modelo de código a seguir utiliza try-catch-finally para alcançar o acima, combinando o tratamento de erros com uma maneira segura de garantir que todas as PSSessions criadas sejam removidas quando terminarem de ser usadas:

try
{
    $session = New-PSsession -Computername "RemoteMachineName"
    Invoke-Command -Session $session -ScriptBlock {write-host "This is running on $ENV:ComputerName"}
}
catch
{
    Write-Output "ERROR: $_"
}
finally
{
    if ($session)
    {
        Remove-PSSession $session
    }
}

Referências: [1] https://msdn.microsoft.com/en-us/powershell/reference/5.1/microsoft.powershell.core/new-pssession
