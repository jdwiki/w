Sintaxe #

  • docker run [OPÇÕES] IMAGEM [COMANDO] [ARG…]

Excluir automaticamente um contêiner após executá-lo #

Normalmente, um contêiner do Docker persiste após ser encerrado. Isso permite que você execute o contêiner novamente, inspecione seu sistema de arquivos e assim por diante. No entanto, às vezes você deseja executar um contêiner e excluí-lo imediatamente após sua saída. Por exemplo, para executar um comando ou mostrar um arquivo do sistema de arquivos. O Docker fornece a opção de linha de comando --rm para esta finalidade:

docker run --rm ubuntu cat /etc/hosts

Isso criará um contêiner a partir da imagem “ubuntu”, mostrará o conteúdo do arquivo /etc/hosts e, em seguida, excluirá o contêiner imediatamente após ele sair. Isso ajuda a evitar a necessidade de limpar os contêineres depois que você terminar de experimentar.

Observação: O sinalizador --rm não funciona em conjunto com o sinalizador -d (--detach) na janela de encaixe < 1.13.0.

Quando o sinalizador --rm é definido, o Docker também remove os volumes associados ao contêiner quando o contêiner é removido. Isso é semelhante a executar docker rm -v my-container. Somente os volumes especificados sem nome são removidos.

Por exemplo, com docker run -it --rm -v /etc -v logs:/var/log centos /bin/produce_some_logs, o volume de /etc será removido, mas o volume de /var/log não. Volumes herdados via –volumes-from serão removidos com a mesma lógica – se o volume original foi especificado com um nome, ele não será removido.

Política de reinicialização do contêiner (iniciando um contêiner na inicialização) #

docker run --restart=always -d <container>

Por padrão, o Docker não reiniciará os contêineres quando o daemon do Docker for reiniciado, por exemplo, após a reinicialização do sistema host. O Docker fornece uma política de reinicialização para seus contêineres fornecendo a opção de linha de comando --restart. Fornecer --restart=always sempre fará com que um contêiner seja reiniciado após o daemon do Docker ser reiniciado. No entanto quando esse contêiner é interrompido manualmente (por exemplo, com docker stop <container>), a política de reinicialização não será aplicada ao contêiner.

Várias opções podem ser especificadas para a opção --restart, com base no requisito (--restart=[policy]). Essas opções também afetam como o contêiner é iniciado na inicialização.

Política Resultado
não O valor padrão. Não reiniciará o contêiner automaticamente quando o contêiner for interrompido.
on-failure[:max-retries] Reiniciar apenas se o container sair com uma falha (status de saída diferente de zero). Para evitar reiniciá-lo indefinidamente (em caso de algum problema), pode-se limitar o número de tentativas de reinicialização do daemon do Docker.
sempre Sempre reinicie o contêiner, independentemente do status de saída. Quando você especifica always, o daemon do Docker tentará reiniciar o container indefinidamente. O contêiner também sempre iniciará na inicialização do daemon, independentemente do estado atual do contêiner.
unless-stopped Sempre reinicie o contêiner, independentemente de seu status de saída, mas não o inicie na inicialização do daemon se o contêiner tiver sido colocado em um estado parado antes.

Especificando um nome #

Por padrão, os contêineres criados com docker run recebem um nome aleatório como small_roentgen ou modest_dubinsky. Esses nomes não são particularmente úteis para identificar a finalidade de um contêiner. É possível fornecer um nome para o container passando a opção de linha de comando --name:

docker run --name my-ubuntu ubuntu:14.04

Os nomes devem ser exclusivos; se você passar um nome que outro container já esteja usando, o Docker imprimirá um erro e nenhum novo container será criado.

A especificação de um nome será útil ao fazer referência ao contêiner em uma rede do Docker. Isso funciona para contêineres do Docker em segundo plano e em primeiro plano.

Os contêineres na rede de ponte padrão devem ser vinculados para se comunicarem por nome.

Execute um contêiner em segundo plano #

Para manter um container rodando em segundo plano, forneça a opção de linha de comando -d durante a inicialização do container:

docker run -d busybox top

A opção -d executa o container em modo desanexado. Também é equivalente a -d=true.

Um container no modo desanexado não pode ser removido automaticamente quando para, isso significa que não se pode usar a opção –rm em combinação com a opção -d.

Atribuir um volume a um container #

Um volume do Docker é um arquivo ou diretório que persiste além do tempo de vida do contêiner. É possível montar um arquivo ou diretório de host em um contêiner como um volume (ignorando o UnionFS).

Adicione um volume com a opção de linha de comando -v:

docker run -d -v "/data" awesome/app bootstrap.sh

Isso criará um volume e o montará no caminho /data dentro do contêiner.

  • Nota: Você pode usar o sinalizador --rm para remover automaticamente o volume quando o contêiner for removido.

Montando diretórios de host

Para montar um arquivo ou diretório de host em um contêiner:

docker run -d -v "/home/foo/data:/data" awesome/app bootstrap.sh
  • Ao especificar um diretório de host, um caminho absoluto deve ser fornecido.

Isto irá montar o diretório do host /home/foo/data em /data dentro do container. Este volume “diretório de host montado em ligação” é a mesma coisa que um mount --bind do Linux e, portanto, monta temporariamente o diretório de host no caminho do contêiner especificado durante a vida útil do contêiner. As alterações no volume do host ou do contêiner são refletidas imediatamente no outro, pois são o mesmo destino no disco.

Exemplo UNIX montando uma pasta relativa

docker run -d -v $(pwd)/data:/data awesome/app bootstrap.sh

Nomeando volumes

Um volume pode ser nomeado fornecendo uma string em vez de um caminho de diretório de host, o docker criará um volume usando esse nome.

docker run -d -v "my-volume:/data" awesome/app bootstrap.sh

Depois de criar um volume nomeado, o volume pode ser compartilhado com outros contêineres usando esse nome.

Obtendo um shell em um contêiner em execução (desanexado) #

Faça login em um contêiner em execução #

Um usuário pode inserir um contêiner em execução em um novo shell bash interativo com o comando exec.

Digamos que um contêiner seja chamado jovial_morse, então você pode obter um shell bash pseudo-TTY interativo executando:

docker exec -it jovial_morse bash

Faça login em um container em execução com um usuário específico #

Se você deseja inserir um contêiner como um usuário específico, pode defini-lo com o parâmetro -u ou --user. O nome de usuário deve existir no contêiner.

-u, --user Nome de usuário ou UID (formato: <nome|uid>[:<grupo|gid>])

Este comando fará login em jovial_morse com o usuário dockeruser

docker exec -it -u dockeruser jovial_morse bash

Faça login em um container em execução como root #

Se você quiser logar como root, simplesmente use o parâmetro -u root. O usuário root sempre existe.

docker exec -it -u root jovial_morse bash

Entrar em uma imagem #

Você também pode fazer login em uma imagem com o comando run, mas isso requer um nome de imagem em vez de um nome de contêiner.

docker run -it dockerimage bash

Execute um contêiner interativamente #

Para executar um container interativamente, passe as opções -it:

$ docker run -it ubuntu:14.04 bash
root@8ef2356d919a:/# echo hi
hi
root@8ef2356d919a:/#

-i mantém o STDIN aberto, enquanto -t aloca um pseudo-TTY.

Desconectando de um contêiner #

Enquanto estiver conectado a um container em execução com um pty atribuído (docker run -it ...), você pode pressionar ControlP - Control Q para desanexar.

Vinculando uma porta de contêiner ao host #

docker run -p "8080:8080" myApp
docker run -p "192.168.1.12:80:80" nginx
docker run -P myApp

Para usar portas no host que foram expostas em uma imagem (através da diretiva EXPOSE Dockerfile, ou opção de linha de comando --expose para docker run), essas portas precisam ser vinculadas ao host usando o opções de linha de comando -p ou -P. Usar -p requer que a porta específica (e interface de host opcional) seja especificada. Usar a opção de linha de comando maiúsculo -P forçará o Docker a vincular todas as portas expostas na imagem de um contêiner ao host.

Configurando variáveis ​​de ambiente #

$ docker run -e "ENV_VAR=foo" ubuntu /bin/bash

Ambos -e e --env podem ser usados ​​para definir variáveis ​​de ambiente dentro de um container. É possível fornecer muitas variáveis ​​de ambiente usando um arquivo de texto:

$ docker run --env-file ./env.list ubuntu /bin/bash

Exemplo de arquivo de variável de ambiente:

# This is a comment
TEST_HOST=10.10.0.127

O sinalizador --env-file recebe um nome de arquivo como argumento e espera que cada linha esteja no formato VARIABLE=VALUE, imitando o argumento passado para --env. As linhas de comentário só precisam ser prefixadas com #.

Independentemente da ordem desses três sinalizadores, o --env-file é processado primeiro e, em seguida, os sinalizadores -e/--env. Dessa forma, quaisquer variáveis ​​de ambiente fornecidas individualmente com -e ou --env substituirão as variáveis ​​fornecidas no arquivo de texto --env-var.

Passando stdin para o container #

Em casos como restaurar um dump de banco de dados, ou de outra forma desejar enviar algumas informações através de um pipe do host, você pode usar o sinalizador -i como um argumento para docker run ou docker exec.

Por exemplo, supondo que você queira colocar em um cliente mariadb containerizado um dump de banco de dados que você tem no host, em um arquivo dump.sql local, você pode executar o seguinte comando:

docker exec -i mariadb bash -c 'mariadb "-p$MARIADB_PASSWORD" ' < dump.sql

No geral,

docker exec -i container command < file.stdin

Ou

docker exec -i container command <<EOF
inline-document-from-host-shell-HEREDOC-syntax
EOF

Substituindo a diretiva do ponto de entrada da imagem #

docker run --name="test-app" --entrypoint="/bin/bash" example-app

Este comando substituirá a diretiva ENTRYPOINT da imagem example-app quando o contêiner test-app for criado. A diretiva CMD da imagem permanecerá inalterada, a menos que especificado de outra forma:

docker run --name="test-app" --entrypoint="/bin/bash" example-app /app/test.sh

No exemplo acima, tanto o ENTRYPOINT quanto o CMD da imagem foram substituídos. Este processo de contêiner se torna /bin/bash /app/test.sh.

Especificando um nome de host #

Por padrão, os contêineres criados com o docker run recebem um nome de host aleatório. Você pode dar ao contêiner um nome de host diferente passando o sinalizador –hostname:

docker run --hostname redbox -d ubuntu:14.04

Executando um contêiner #

docker run hello-world

Isso buscará a imagem hello-world mais recente do Docker Hub (se você ainda não a tiver), criará um novo contêiner e o executará. Você deverá ver uma mensagem informando que sua instalação parece estar funcionando corretamente.

Executando um comando diferente no container #

docker run docker/whalesay cowsay 'Hello, StackExchange!'

Este comando diz ao Docker para criar um container a partir da imagem docker/whalesay e executar o comando cowsay 'Hello, StackExchange!' nele. Ele deve imprimir uma imagem de uma baleia dizendo ‘Olá, StackExchange!’ no seu terminal.

Se o ponto de entrada na imagem for o padrão, você poderá executar qualquer comando disponível na imagem:

docker run docker/whalesay ls /

Se tiver sido alterado durante a criação da imagem, você precisa revertê-lo para o padrão

docker run --entrypoint=/bin/bash docker/whalesay -c ls /

Impede que o container pare quando nenhum comando estiver em execução #

Um contêiner será interrompido se nenhum comando estiver sendo executado em primeiro plano. Usar a opção -t impedirá que o container pare, mesmo quando desanexado com a opção -d.

docker run -t -d debian bash

Parando um contêiner #

docker stop mynginx

Além disso, o ID do contêiner também pode ser usado para interromper o contêiner em vez de seu nome.

Isso interromperá um contêiner em execução enviando o sinal SIGTERM e, em seguida, o sinal SIGKILL, se necessário.

Além disso, o comando kill pode ser usado para enviar imediatamente um SIGKILL ou qualquer outro sinal especificado usando a opção -s.

docker kill mynginx

Sinal especificado:

docker kill -s SIGINT mynginx

Parar um contêiner não o exclui. Use docker ps -a para ver seu container parado.

Contêiner em execução com limites de memória/troca #

Defina o limite de memória e desative o limite de troca

docker run -it -m 300M --memory-swap -1 ubuntu:14.04 /bin/bash

Defina o limite de memória e de troca. Nesse caso, o contêiner pode usar 300M de memória e 700M de swap.

docker run -it -m 300M --memory-swap 1G ubuntu:14.04 /bin/bash

Adicionar entrada de host ao contêiner #

docker run --add-host="app-backend:10.15.1.24" awesome-app

Este comando adiciona uma entrada ao arquivo /etc/hosts do contêiner, que segue o formato --add-host <nome>:<endereço>. Neste exemplo, o nome app-backend será resolvido para 10.15.1.24. Isso é particularmente útil para vincular componentes de aplicativos diferentes programaticamente.

Execute outro comando em um container em execução #

Quando necessário, você pode dizer ao Docker para executar comandos adicionais em um contêiner já em execução usando o comando exec. Você precisa do ID do contêiner, que pode ser obtido com docker ps.

docker exec 294fbc4c24b3 echo "Hello World"

Você pode anexar um shell interativo se usar a opção -it.

docker exec -it 294fbc4c24b3 bash

Executando aplicativos GUI em um contêiner Linux #

Por padrão, um contêiner do Docker não poderá executar um aplicativo GUI.

Antes disso, o socket X11 deve ser encaminhado primeiro para o container, para que possa ser utilizado diretamente. A variável de ambiente DISPLAY também deve ser encaminhada:

docker run -v /tmp/.X11-unix:/tmp/.X11-unix -e DISPLAY=unix$DISPLAY <image-name>

Isso falhará no início, pois não definimos as permissões para o host do servidor X:

cannot connect to X server unix:0

A maneira mais rápida (mas não mais segura) é permitir o acesso diretamente com:

xhost +local:root

Depois de terminar com o container, podemos voltar ao estado original com:

xhost -local:root

Outra maneira (mais segura) é preparar um Dockerfile que construirá uma nova imagem que usará nossas credenciais de usuário para acessar o servidor X:

FROM <iamge-name>
MAINTAINER <you>

# Arguments picked from the command line!
ARG user
ARG uid
ARG gid

#Add new user with our credentials
ENV USERNAME ${user}
RUN useradd -m $USERNAME && \
        echo "$USERNAME:$USERNAME" | chpasswd && \
        usermod --shell /bin/bash $USERNAME && \
        usermod  --uid ${uid} $USERNAME && \
        groupmod --gid ${gid} $USERNAME

USER ${user}

WORKDIR /home/${user}

Ao invocar o docker build a partir da linha de comando, temos que passar as variáveis ​​ARG que aparecem no Dockerfile:

docker build --build-arg user=$USER --build-arg uid=$(id -u) --build-arg gid=$(id -g) -t <new-image-with-X11-enabled-name> -f <Dockerfile-for-X11> .

Agora, antes de gerar um novo container, temos que criar um arquivo xauth com permissão de acesso:

xauth nlist $DISPLAY | sed -e 's/^..../ffff/' | xauth -f /tmp/.docker.xauth nmerge -

Este arquivo deve ser montado no contêiner ao criá-lo/executá-lo:

docker run -e DISPLAY=unix$DISPLAY -v /tmp/.X11-unix:/tmp/.X11-unix -v /tmp/.docker.xauth:/tmp/.docker.xauth:rw -e XAUTHORITY=/tmp/.docker.xauth
