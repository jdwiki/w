Como configurar uma réplica do Mongo de três nós usando a imagem do Docker e provisionada usando o Chef

  • Etapa de construção

Nesta página

  • Etapa de construção

Esta documentação descreve como criar um conjunto de réplicas do Mongo de três nós usando o Docker Image e provisionado automaticamente usando o Chef.

Etapa de construção #

Passos:

  1. Gere um arquivo de chave Base 64 para autenticação do nó Mongo. Coloque este arquivo em chef data_bags

  2. Vá ao supermercado do chef e baixe o livro de receitas docker. Gere um livro de receitas personalizado (por exemplo, custom_mongo) e adicione o ‘docker’, ‘~> 2.0’ ao arquivo metadata.rb do seu livro de receitas

  3. Crie um atributo e uma receita em seu livro de receitas personalizado

  4. Inicialize o Mongo para formar o cluster Rep Set

Etapa 1: criar arquivo de chave

crie data_bag chamado mongo-keyfile e item chamado keyfile. Isso estará no diretório data_bags no chef. O conteúdo do item será como abaixo

openssl rand -base64 756 > <path-to-keyfile>

conteúdo do item do arquivo-chave

{
  "id": "keyfile",
  "comment": "Mongo Repset keyfile",
  "key-file": "generated base 64 key above"
}

Etapa 2: Baixe o livro de receitas docker do mercado de jantar do chef e crie o livro de receitas custom_mongo

knife cookbook site download docker 
knife cookbook create custom_mongo

em metadat.rb de custom_mongo adicione

depends          'docker', '~> 2.0'

Etapa 3: criar atributo e receita

Atributos

default['custom_mongo']['mongo_keyfile'] = '/data/keyfile' 
default['custom_mongo']['mongo_datadir'] = '/data/db'
default['custom_mongo']['mongo_datapath'] = '/data'
default['custom_mongo']['keyfilename'] = 'mongodb-keyfile'

Receita

#
# Cookbook Name:: custom_mongo
# Recipe:: default
#
# Copyright 2017, Innocent Anigbo
#
# All rights reserved - Do Not Redistribute
#

data_path = "#{node['custom_mongo']['mongo_datapath']}"
data_dir = "#{node['custom_mongo']['mongo_datadir']}"
key_dir = "#{node['custom_mongo']['mongo_keyfile']}"
keyfile_content = data_bag_item('mongo-keyfile', 'keyfile')
keyfile_name = "#{node['custom_mongo']['keyfilename']}"

#chown of keyfile to docker user
execute 'assign-user' do
 command "chown 999 #{key_dir}/#{keyfile_name}"
 action :nothing
end

#Declaration to create Mongo data DIR and Keyfile DIR
%W[ #{data_path} #{data_dir} #{key_dir} ].each do |path|
directory path do
  mode '0755'
  end
end

#declaration to copy keyfile from data_bag to keyfile DIR on your mongo server
file "#{key_dir}/#{keyfile_name}" do
  content keyfile_content['key-file']
  group 'root'
  mode '0400'
  notifies :run, 'execute[assign-user]', :immediately
end

#Install docker
docker_service 'default' do
  action [:create, :start]
end

#Install mongo 3.4.2
docker_image 'mongo' do
  tag '3.4.2'
  action :pull
end

Crie uma função chamada mongo-role no diretório de funções

{
  "name": "mongo-role",
  "description": "mongo DB Role",
  "run_list": [
    "recipe[custom_mongo]"
  ]
}

Adicione a função acima à lista de execução de três nós do mongo

knife node run_list add FQDN_of_node_01 'role[mongo-role]'
knife node run_list add FQDN_of_node_02 'role[mongo-role]'
knife node run_list add FQDN_of_node_03 'role[mongo-role]'

Etapa 4: Inicialize o Mongo de três nós para formar o repset

Estou assumindo que a função acima já foi aplicada em todos os três nós do Mongo. Somente no nó 01, inicie o Mongo com –auth para habilitar a autenticação

docker run --name mongo -v /data/db:/data/db -v /data/keyfile:/opt/keyfile --hostname="mongo-01.example.com" -p 27017:27017 -d mongo:3.4.2 --keyFile /opt/keyfile/mongodb-keyfile --auth

Acesse o shell interativo do contêiner docker em execução no nó 01 e crie um usuário administrador

docker exec -it mongo /bin/sh
    mongo
    use admin
    db.createUser( {
         user: "admin-user",
         pwd: "password",
         roles: [ { role: "userAdminAnyDatabase", db: "admin" } ]
       });

Criar usuário raiz

db.createUser( {
         user: "RootAdmin",
         pwd: "password",
         roles: [ { role: "root", db: "admin" } ]
       });

Pare e exclua o contêiner do Docker criado acima no nó 01. Isso não afetará os dados e o arquivo-chave no DIR do host. Após a exclusão, inicie o Mongo novamente no nó 01, mas desta vez com o sinalizador repset

docker rm -fv mongo
docker run --name mongo-uat -v /data/db:/data/db -v /data/keyfile:/opt/keyfile --hostname="mongo-01.example.com" -p 27017:27017 -d mongo:3.4.2 --keyFile /opt/keyfile/mongodb-keyfile --replSet "rs0"

agora inicie o mongo no nó 02 e 03 com o sinalizador de conjunto de repetições

docker run --name mongo -v /data/db:/data/db -v /data/keyfile:/opt/keyfile --hostname="mongo-02.example.com" -p 27017:27017 -d mongo:3.4.2 --keyFile /opt/keyfile/mongodb-keyfile --replSet "rs0"
docker run --name mongo -v /data/db:/data/db -v /data/keyfile:/opt/keyfile --hostname="mongo-03.example.com" -p 27017:27017 -d mongo:3.4.2 --keyFile /opt/keyfile/mongodb-keyfile --replSet "rs0"

Autentique-se com o usuário root no nó 01 e inicie o conjunto de réplicas

use admin
db.auth("RootAdmin", "password");
rs.initiate()

No nó 01, adicione os nós 2 e 3 ao conjunto de réplicas para formar o cluster repset0

rs.add("mongo-02.example.com")
rs.add("mongo-03.example.com")

Teste

No primário execute db.printSlaveReplicationInfo() e observe o SyncedTo e Behind o horário primário. O mais tarde deve ser de 0 seg conforme abaixo

Resultado

 rs0:PRIMARY> db.printSlaveReplicationInfo()
    source: mongo-02.example.com:27017
            syncedTo: Mon Mar 27 2017 15:01:04 GMT+0000 (UTC)
            0 secs (0 hrs) behind the primary
    source: mongo-03.example.com:27017
            syncedTo: Mon Mar 27 2017 15:01:04 GMT+0000 (UTC)
            0 secs (0 hrs) behind the primary
