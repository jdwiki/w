Construindo uma imagem de um Dockerfile #

Uma vez que você tenha um Dockerfile, você pode construir uma imagem dele usando docker build. A forma básica deste comando é:

compilação do docker -t nome da imagem caminho

Se o seu Dockerfile não for chamado de Dockerfile, você pode usar o sinalizador -f para dar o nome do Dockerfile a ser compilado.

docker build -t image-name -f Dockerfile2 .

Por exemplo, para construir uma imagem chamada dockerbuild-example:1.0.0 a partir de um Dockerfile no diretório de trabalho atual:

$ ls
Dockerfile Dockerfile2

$ docker build -t dockerbuild-example:1.0.0 .

$ docker build -t dockerbuild-example-2:1.0.0 -f Dockerfile2 .

Veja a documentação de uso do docker build para mais opções e configurações.

Um erro comum é criar um Dockerfile no diretório inicial do usuário (~). Esta é uma má idéia porque durante o docker build -t mytag . esta mensagem aparecerá por um longo tempo:

Carregando contexto

A causa é o daemon do docker tentando copiar todos os arquivos do usuário (tanto o diretório inicial quanto seus subdiretórios). Evite isso sempre especificando um diretório para o Dockerfile.

Adicionar um arquivo .dockerignore ao diretório de compilação é uma boa prática. Sua sintaxe é semelhante aos arquivos .gitignore e garantirá que apenas os arquivos e diretórios desejados sejam carregados como contexto da compilação.

Diferença entre ENTRYPOINT e CMD #

Existem duas diretivas Dockerfile para especificar qual comando executar por padrão nas imagens criadas. Se você especificar apenas CMD, o docker executará esse comando usando o ENTRYPOINT padrão, que é /bin/sh -c. Você pode substituir um ou ambos o ponto de entrada e/ou o comando ao iniciar a imagem criada. Se você especificar ambos, então o ENTRYPOINT especifica o executável do seu processo de contêiner, e o CMD será fornecido como os parâmetros desse executável.

Por exemplo, se o seu Dockerfile contiver

FROM ubuntu:16.04
CMD ["/bin/date"]

Então você está usando a diretiva padrão ENTRYPOINT de /bin/sh -c, e executando /bin/date com esse ponto de entrada padrão. O comando do seu processo de contêiner será /bin/sh -c /bin/date. Depois de executar esta imagem, por padrão, ela imprimirá a data atual

$ docker build -t test .
$ docker run test
Tue Jul 19 10:37:43 UTC 2016

Você pode substituir o CMD na linha de comando, nesse caso ele executará o comando que você especificou.

$ docker run test /bin/hostname
bf0274ec8820

Se você especificar uma diretiva ENTRYPOINT, o Docker usará esse executável e a diretiva CMD especifica os parâmetros padrão do comando. Portanto, se o seu Dockerfile contiver:

FROM ubuntu:16.04
ENTRYPOINT ["/bin/echo"]
CMD ["Hello"]

Em seguida, executá-lo produzirá

$ docker build -t test .
$ docker run test
Hello

Você pode fornecer parâmetros diferentes se quiser, mas todos eles executarão /bin/echo

$ docker run test Hi
Hi

Se você deseja substituir o ponto de entrada listado em seu Dockerfile (ou seja, se deseja executar um comando diferente de echo neste contêiner), você precisa especificar o parâmetro --entrypoint na linha de comando:

$ docker run --entrypoint=/bin/hostname test
b2c70e74df18

Geralmente você usa a diretiva ENTRYPOINT para apontar para o aplicativo principal que deseja executar e CMD para os parâmetros padrão.

Um simples Dockerfile #

FROM node:5

A diretiva FROM especifica uma imagem para começar. Qualquer [referência de imagem] válida pode ser usada.

WORKDIR /usr/src/app

A diretiva WORKDIR define o diretório de trabalho atual dentro do contêiner, equivalente a executar o cd dentro do contêiner. (Nota: RUN cd não mudará o diretório de trabalho atual.)

RUN npm install cowsay knock-knock-jokes

RUN executa o comando dado dentro do container.

COPY cowsay-knockknock.js ./

COPY copia o arquivo ou diretório especificado no primeiro argumento do contexto de compilação (o path passado para docker build path< /code>) para o local no contêiner especificado pelo segundo argumento.

CMD node cowsay-knockknock.js

CMD especifica um comando a ser executado quando a imagem é [executada] e nenhum comando é fornecido. Ele pode ser substituído [passando um comando para docker run].

Existem muitas outras instruções e opções; consulte a [referência do Dockerfile] para obter uma lista completa.

[referência de imagem]: https://www.wikiod.com/pt/docker/gerenciando-imagens#Imagens de referência [executar]: https://www.wikiod.com/pt/docker/conteineres-em-execucao#Executando um contêiner [passando um comando para docker run]: https://www.wikiod.com/pt/docker/conteineres-em-execucao#Executando um comando diferente no contêiner [Referência do Dockerfile]: https://docs.docker.com/engine/reference/builder/

Empurrando e puxando uma imagem para o Docker Hub ou outro registro #

As imagens criadas localmente podem ser enviadas para o Docker Hub ou qualquer outro host de repositório do Docker, conhecido como registro. Use docker login para entrar em uma conta existente do docker hub.

docker login

Login with your Docker ID to push and pull images from Docker Hub.
If you don't have a Docker ID, head over to https://hub.docker.com to create one.

Username: cjsimon
Password:
Login Succeeded

Um registro docker diferente pode ser usado especificando um nome de servidor. Isso também funciona para registros privados ou auto-hospedados. Além disso, é possível usar um armazenamento de credenciais externo para segurança.

docker login quay.io

Você pode então marcar e enviar imagens para o registro no qual você está conectado. Seu repositório deve ser especificado como server/username/reponame:tag. A omissão do servidor atualmente é o padrão do Docker Hub. (O registro padrão não pode ser alterado para outro provedor e não há nenhum plano para implementar esse recurso.)

docker tag mynginx quay.io/cjsimon/mynginx:latest

Diferentes tags podem ser usadas para representar diferentes versões, ou ramificações, da mesma imagem. Uma imagem com várias tags diferentes exibirá cada tag no mesmo repositório.

Use docker images para ver uma lista de imagens instaladas instaladas em sua máquina local, incluindo sua imagem recém-marcada. Em seguida, use push para carregá-lo no registro e puxe para baixar a imagem.

docker push quay.io/cjsimon/mynginx:latest

Todas as tags de uma imagem podem ser puxadas especificando a opção -a

docker pull quay.io/cjsimon/mynginx:latest

Expondo uma porta no Dockerfile #

EXPOSE <port> [<port>...]

Da documentação do Docker:

A instrução EXPOSE informa ao Docker que o contêiner escuta nas portas de rede especificadas em tempo de execução. EXPOSE não torna as portas do container acessíveis ao host. Para fazer isso, você deve usar o sinalizador -p para publicar um intervalo de portas ou o sinalizador -P para publicar todas as portas expostas. Você pode expor um número de porta e publicá-lo externamente em outro número.

Exemplo: #

Dentro do seu Dockerfile:

EXPOSE 8765

Para acessar esta porta da máquina host, inclua este argumento em seu comando docker run:

-p 8765:8765

ENTRYPOINT e CMD vistos como verbo e parâmetro #

Suponha que você tenha um Dockerfile terminando com

ENTRYPOINT ["nethogs"] CMD ["wlan0"]

se você construir esta imagem com um

docker built -t inspector .

inicie a imagem criada com esse Dockerfile com um comando como

docker run -it --net=host --rm inspector

, os nethogs irão monitorar a interface chamada wlan0

Agora se você quiser monitorar a interface eth0 (ou wlan1, ou ra1…), você fará algo como

docker run -it --net=host --rm inspector eth0

ou

docker run -it --net=host --rm inspector wlan1

Construindo usando um proxy #

Muitas vezes, ao construir uma imagem do Docker, o Dockerfile contém instruções que executam programas para buscar recursos da Internet (wget por exemplo para puxar um programa binário compilado no GitHub, por exemplo).

É possível instruir o Docker a passar variáveis ​​de ambiente set set para que esses programas executem essas buscas por meio de um proxy:

$ docker build --build-arg http_proxy=http://myproxy.example.com:3128 \
               --build-arg https_proxy=http://myproxy.example.com:3128 \
               --build-arg no_proxy=internal.example.com \
               -t test .

build-arg são variáveis ​​de ambiente que estão disponíveis apenas em tempo de compilação.
