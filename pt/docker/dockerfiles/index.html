Dockerfiles são arquivos usados ​​para construir imagens do Docker programaticamente. Eles permitem que você crie uma imagem do Docker de forma rápida e reproduzível e, portanto, são úteis para colaboração. Os Dockerfiles contêm instruções para criar uma imagem do Docker. Cada instrução é escrita em uma linha e é dada na forma <INSTRUCTION><argument(s)>. Os Dockerfiles são usados ​​para construir imagens do Docker usando o comando docker build.

Os Dockerfiles têm o formato:

# This is a comment
INSTRUCTION arguments
  • Comentários começam com um #
  • As instruções são apenas em maiúsculas
  • A primeira instrução de um Dockerfile deve ser FROM para especificar a imagem base

Ao construir um Dockerfile, o cliente Docker enviará um “contexto de compilação” para o daemon do Docker. O contexto de compilação inclui todos os arquivos e pastas no mesmo diretório que o Dockerfile. As operações COPY e ADD só podem usar arquivos deste contexto.

Alguns arquivos do Docker podem começar com:

# escape=`

Isso é usado para instruir o analisador do Docker a usar ` como um caractere de escape em vez de \. Isso é útil principalmente para arquivos do Windows Docker.

Práticas recomendadas do Dockerfiles #

Operações comuns do grupo

O Docker cria imagens como uma coleção de camadas. Cada camada só pode adicionar dados, mesmo que esses dados digam que um arquivo foi excluído. Cada instrução cria uma nova camada. Por exemplo:

RUN apt-get -qq update
RUN apt-get -qq install some-package

Tem algumas desvantagens:

  • Criará duas camadas, produzindo uma imagem maior.
  • Usar o apt-get update sozinho em uma instrução RUN causa problemas de cache e, posteriormente, as instruções do apt-get install podem falhar. Suponha que você modifique posteriormente o apt-get install adicionando pacotes extras, então o docker interpreta as instruções iniciais e modificadas como idênticas e reutiliza o cache das etapas anteriores. Como resultado, o comando apt-get update não é executado porque sua versão em cache é usada durante a compilação.

Em vez disso, use:

RUN apt-get -qq update && \
    apt-get -qq install some-package

pois isso produz apenas uma camada.

Mencione o mantenedor

Esta é geralmente a segunda linha do Dockerfile. Ele diz quem está no comando e será capaz de ajudar.

LABEL maintainer John Doe <john.doe@example.com>

Se você pular, não quebrará sua imagem. Mas também não ajudará seus usuários.

Ser conciso

Mantenha seu Dockerfile curto. Se uma configuração complexa for necessária, considere usar um script dedicado ou configurar imagens de base.

Instrução EXPOSE #

EXPOSE <port> [<port>...]

A instrução EXPOSE informa ao Docker que o contêiner escuta nas portas de rede especificadas em tempo de execução. EXPOSE NÃO torna as portas do container acessíveis ao host. Para fazer isso, você deve usar o sinalizador -p para publicar um intervalo de portas ou o sinalizador -P para publicar todas as portas expostas. Esses sinalizadores são usados ​​no docker run [OPTIONS] IMAGE [COMMAND][ARG...] para expor a porta ao host. Você pode expor um número de porta e publicá-lo externamente em outro número.

docker run -p 2500:80 <image name>

Este comando criará um container com o nome <image> e vinculará a porta 80 do container à porta 2500 da máquina host.

Para configurar o redirecionamento de porta no sistema host, consulte usando o sinalizador -P. O recurso de rede do Docker oferece suporte à criação de redes sem a necessidade de expor portas dentro da rede, para obter informações detalhadas, consulte a visão geral desse recurso).

Instrução CMD #

A instrução CMD tem três formas:

CMD ["executable","param1","param2"] (exec form, this is the preferred form)
CMD ["param1","param2"] (as default parameters to ENTRYPOINT)
CMD command param1 param2 (shell form)

Só pode haver uma instrução CMD em um Dockerfile. Se você listar mais de um CMD, somente o último CMD terá efeito.

O principal objetivo de um CMD é fornecer padrões para um contêiner em execução. Esses padrões podem incluir um executável, ou podem omitir o executável, neste caso você deve especificar uma instrução ENTRYPOINT também.

Nota: Se CMD for usado para fornecer argumentos padrão para a instrução ENTRYPOINT, ambas as instruções CMD e ENTRYPOINT devem ser especificadas com o formato de matriz JSON.

Nota: O formulário exec é analisado como uma matriz JSON, o que significa que você deve usar aspas duplas (“) em torno de palavras e não aspas simples (‘).

Nota: Ao contrário do formulário shell, o formulário exec não invoca um shell de comando. Isso significa que o processamento normal do shell não acontece. Por exemplo, CMD [ "echo", "$HOME" ] não fará a substituição de variável em $HOME. Se você deseja processamento de shell, use o formulário de shell ou execute um shell diretamente, por exemplo: CMD [ "sh", "-c", "echo $HOME" ].

Quando usada nos formatos shell ou exec, a instrução CMD define o comando a ser executado ao executar a imagem.

Se você usar o formulário shell do CMD, o comando será executado em /bin/sh -c:

FROM ubuntu
CMD echo "This is a test." | wc -

Se você deseja executar seu comando sem um shell, deve expressar o comando como uma matriz JSON e fornecer o caminho completo para o executável. Este formato de matriz é o formato preferido de CMD. Quaisquer parâmetros adicionais devem ser expressos individualmente como strings na matriz:

FROM ubuntu
CMD ["/usr/bin/wc","--help"]

Se você quiser que seu container rode sempre o mesmo executável, então você deve considerar usar ENTRYPOINT em combinação com CMD. Consulte PONTO DE ENTRADA.

Se o usuário especificar argumentos para executar o docker, eles substituirão o padrão especificado em CMD.

Nota: não confunda RUN com CMD. RUN realmente executa um comando no momento da construção da imagem e confirma o resultado; CMD não executa nada em tempo de compilação, mas especifica o comando pretendido para a imagem.

A Instrução ENV e ARG #

ENV #

ENV <key> <value>
ENV <key>=<value> ...

A instrução ENV define a variável de ambiente <key> para o valor . Esse valor estará no ambiente de todos os comandos “descendentes” do Dockerfile e também pode ser substituído inline em muitos.

A instrução ENV tem duas formas. A primeira forma, ENV <chave> <valor>, definirá uma única variável para um valor. A string inteira após o primeiro espaço será tratada como o <valor> - incluindo caracteres como espaços e aspas.

A segunda forma, ENV <chave>=<valor> ..., permite que várias variáveis ​​sejam definidas ao mesmo tempo. Observe que o segundo formulário usa o sinal de igual (=) na sintaxe, enquanto o primeiro formulário não. Assim como a análise de linha de comando, aspas e barras invertidas podem ser usadas para incluir espaços nos valores.

Por exemplo:

ENV myName="John Doe" myDog=Rex\ The\ Dog \
    myCat=fluffy

e

ENV myName John Doe
ENV myDog Rex The Dog
ENV myCat fluffy

produzirá os mesmos resultados líquidos no contêiner final, mas a primeira forma é preferida porque produz uma única camada de cache.

As variáveis ​​de ambiente definidas usando ENV persistirão quando um contêiner for executado a partir da imagem resultante. Você pode visualizar os valores usando docker inspect e alterá-los usando docker run --env <key>=<value>.

ARG #

Se você não deseja manter a configuração, use ARG. ARG definirá ambientes apenas durante a compilação. Por exemplo, definir

ENV DEBIAN_FRONTEND noninteractive

pode confundir usuários do apt-get em uma imagem baseada em Debian quando eles entram no container em um contexto interativo via docker exec -it the-container bash.

Em vez disso, use:

ARG DEBIAN_FRONTEND noninteractive

Como alternativa, você também pode definir um valor para um único comando usando:

RUN <key>=<value> <command>

##HelloWorld Dockerfile Um Dockerfile mínimo se parece com isso:

FROM alpine
CMD ["echo", "Hello StackOverflow!"]

Isso instruirá o Docker a construir uma imagem baseada em Alpine (FROM), uma distribuição mínima para contêineres, e a executar um comando específico (CMD) ao executar a imagem resultante.

Construa e execute:

docker build -t hello .
docker run --rm hello

Isto produzirá:

Hello StackOverflow!

Instrução do USUÁRIO #

USER daemon

A instrução USER define o nome de usuário ou UID a ser usado ao executar a imagem e para quaisquer instruções RUN, CMD e ENTRYPOINT que o seguem no Dockerfile.

Instrução WORKDIR #

WORKDIR /path/to/workdir

A instrução WORKDIR define o diretório de trabalho para quaisquer instruções RUN, CMD, ENTRYPOINT, COPY e ADD que seguem no Dockerfile. Se o WORKDIR não existir, ele será criado mesmo que não seja usado em nenhuma instrução Dockerfile subsequente.

Ele pode ser usado várias vezes em um Dockerfile. Se um caminho relativo for fornecido, ele será relativo ao caminho da instrução WORKDIR anterior. Por exemplo:

WORKDIR /a
WORKDIR b
WORKDIR c
RUN pwd

A saída do comando final pwd neste Dockerfile seria /a/b/c.

A instrução WORKDIR pode resolver variáveis ​​de ambiente previamente definidas usando ENV. Você só pode usar variáveis ​​de ambiente definidas explicitamente no Dockerfile. Por exemplo:

ENV DIRPATH /path
WORKDIR $DIRPATH/$DIRNAME
RUN pwd

A saída do comando final pwd neste Dockerfile seria /path/$DIRNAME

Instrução VOLUME #

VOLUME ["/data"]

A instrução VOLUME cria um ponto de montagem com o nome especificado e o marca como contendo volumes montados externamente do host nativo ou de outros contêineres. O valor pode ser uma matriz JSON, VOLUME ["/var/log/"] ou uma string simples com vários argumentos, como VOLUME /var/log ou VOLUME /var/log /var/db . Para obter mais informações/exemplos e instruções de montagem por meio do cliente Docker, consulte a documentação Compartilhar diretórios por meio de volumes.

O comando docker run inicializa o volume recém-criado com quaisquer dados que existam no local especificado na imagem base. Por exemplo, considere o seguinte trecho do Dockerfile:

FROM ubuntu
RUN mkdir /myvol
RUN echo "hello world" > /myvol/greeting
VOLUME /myvol

Esse Dockerfile resulta em uma imagem que faz com que o docker run crie um novo ponto de montagem em /myvol e copie o arquivo de saudação no volume recém-criado.

Observação: se alguma etapa de compilação alterar os dados no volume após sua declaração, essas alterações serão descartadas.

Nota: A lista é analisada como uma matriz JSON, o que significa que você deve usar aspas duplas (“) em torno de palavras e não aspas simples (‘).

Instrução COPIAR #

COPY tem duas formas:

COPY <src>... <dest>
COPY ["<src>",... "<dest>"] (this form is required for paths containing whitespace)

A instrução COPY copia novos arquivos ou diretórios de <src> e os adiciona ao sistema de arquivos do container no caminho <dest>.

Vários recursos <src> podem ser especificados, mas devem ser relativos ao diretório de origem que está sendo compilado (o contexto da compilação).

Cada <src> pode conter curingas e a correspondência será feita usando as regras filepath.Match do Go. Por exemplo:

COPY hom* /mydir/        # adds all files starting with "hom"
COPY hom?.txt /mydir/    # ? is replaced with any single character, e.g., "home.txt"

O <dest> é um caminho absoluto ou um caminho relativo a [WORKDIR](https://www.wikiod.com/pt/docker/dockerfiles#Instrução WORKDIR), em qual a origem será copiada dentro do container de destino.

COPY test relativeDir/   # adds "test" to `WORKDIR`/relativeDir/
COPY test /absoluteDir/  # adds "test" to /absoluteDir/

Todos os novos arquivos e diretórios são criados com um UID e GID de 0.

Nota: Se você compilar usando stdin (docker build - < somefile), não há contexto de compilação, então COPY não pode ser usado.

COPY obedece as seguintes regras:

  • O caminho <src> deve estar dentro do contexto da compilação; você não pode COPY ../something /something, porque o primeiro passo de uma compilação do docker é enviar o diretório de contexto (e subdiretórios) para o daemon do docker.

  • Se <src> for um diretório, todo o conteúdo do diretório será copiado, incluindo metadados do sistema de arquivos. Nota: O diretório em si não é copiado, apenas seu conteúdo.

  • Se <src> for qualquer outro tipo de arquivo, ele é copiado individualmente junto com seus metadados. Neste caso, se <dest> terminar com uma barra /, será considerado um diretório e o conteúdo de <src> será escrito em <dest>/base(<src>).

  • Se vários recursos <src> forem especificados, diretamente ou devido ao uso de um curinga, então <dest> deve ser um diretório e deve terminar com uma barra /.

  • Se <dest> não terminar com uma barra final, ele será considerado um arquivo regular e o conteúdo de <src> será escrito em <dest>.

  • Se <dest> não existir, ele será criado junto com todos os diretórios ausentes em seu caminho.

Instrução LABEL #

LABEL <key>=<value> <key>=<value> <key>=<value> ...

A instrução LABEL adiciona metadados a uma imagem. Um LABEL é um par chave-valor. Para incluir espaços em um valor LABEL, use aspas e barras invertidas como faria na análise de linha de comando. Alguns exemplos de uso:

LABEL "com.example.vendor"="ACME Incorporated"
LABEL com.example.label-with-value="foo"
LABEL version="1.0"
LABEL description="This text illustrates \
that label-values can span multiple lines."

Uma imagem pode ter mais de um rótulo. Para especificar vários rótulos, o Docker recomenda combinar rótulos em uma única instrução LABEL sempre que possível. Cada instrução LABEL produz uma nova camada que pode resultar em uma imagem ineficiente se você usar muitos rótulos. Este exemplo resulta em uma única camada de imagem.

LABEL multi.label1="value1" multi.label2="value2" other="value3"

O acima também pode ser escrito como:

LABEL multi.label1="value1" \
      multi.label2="value2" \
      other="value3"

Os rótulos são aditivos, incluindo LABELs em imagens FROM. Se o Docker encontrar um rótulo/chave que já existe, o novo valor substituirá todos os rótulos anteriores por chaves idênticas.

Para visualizar os rótulos de uma imagem, use o comando docker inspect.

"Labels": {
    "com.example.vendor": "ACME Incorporated"
    "com.example.label-with-value": "foo",
    "version": "1.0",
    "description": "This text illustrates that label-values can span multiple lines.",
    "multi.label1": "value1",
    "multi.label2": "value2",
    "other": "value3"
},

Instrução MAINTAINER #

MAINTAINER <name>

A instrução MAINTAINER permite definir o campo Autor das imagens geradas.

NÃO USE A DIRETORIA DE MANTENEDOR

De acordo com a Documentação oficial do Docker, a instrução MAINTAINER está obsoleta. Em vez disso, deve-se usar a instrução LABEL para definir o autor das imagens geradas. A instrução LABEL é mais flexível, permite definir metadados e pode ser facilmente visualizada com o comando docker inspect.

LABEL maintainer="someone@something.com"

DE Instrução #

FROM <image>

Ou

FROM <image>:<tag>

Ou

FROM <image>@<digest>

A instrução FROM define a imagem base para instruções subsequentes. Como tal, um Dockerfile válido deve ter FROM como sua primeira instrução. A imagem pode ser qualquer imagem válida – é especialmente fácil começar puxando uma imagem dos Repositórios Públicos.

FROM deve ser a primeira instrução sem comentários no Dockerfile.

FROM pode aparecer várias vezes em um único Dockerfile para criar várias imagens. Simplesmente anote a última saída de ID de imagem pelo commit antes de cada novo comando FROM.

Os valores de tag ou resumo são opcionais. Se você omitir qualquer um deles, o construtor assumirá o mais recente por padrão. O construtor retorna um erro se não corresponder ao valor da tag.

Instrução EXECUTAR #

RUN tem 2 formas:

RUN <command> (shell form, the command is run in a shell, which by default is /bin/sh -c on Linux or cmd /S /C on Windows)
RUN ["executable", "param1", "param2"] (exec form)

A instrução RUN executará qualquer comando em uma nova camada em cima da imagem atual e confirmará os resultados. A imagem confirmada resultante será usada para a próxima etapa no Dockerfile.

A camada de instruções RUN e a geração de commits estão em conformidade com os conceitos centrais do Docker, onde os commits são baratos e os contêineres podem ser criados a partir de qualquer ponto no histórico de uma imagem, assim como o controle de origem.

O formulário exec torna possível evitar o uso de strings de shell e comandos RUN usando uma imagem base que não contém o executável shell especificado.

O shell padrão para o formulário shell pode ser alterado usando o comando SHELL.

Na forma de shell você pode usar um \ (barra invertida) para continuar uma única instrução RUN na próxima linha. Por exemplo, considere estas duas linhas:

RUN /bin/bash -c 'source $HOME/.bashrc ;\
echo $HOME'

Juntos, eles são equivalentes a esta única linha:

RUN /bin/bash -c 'source $HOME/.bashrc ; echo $HOME'

Nota: Para usar um shell diferente de ‘/bin/sh’, use o formulário exec passando o shell desejado. Por exemplo, RUN ["/bin/bash", "-c", "echo hello"]

Nota: O formulário exec é analisado como um array JSON, o que significa que você deve usar aspas duplas (“) em torno de palavras e não aspas simples (‘).

Nota: Ao contrário do formulário shell, o formulário exec não invoca um shell de comando. Isso significa que o processamento normal do shell não acontece. Por exemplo, RUN [ "echo", "$HOME" ] não fará a substituição de variável em $HOME. Se você deseja processamento de shell, use o formulário de shell ou execute um shell diretamente, por exemplo: RUN [ "sh", "-c", "echo $HOME" ].

Observação: no formulário JSON, é necessário escapar das barras invertidas. Isso é particularmente relevante no Windows, onde a barra invertida é o separador de caminho. A linha a seguir seria tratada como um formulário de shell por não ser um JSON válido e falharia de maneira inesperada: RUN ["c:\windows\system32\tasklist.exe"]

A sintaxe correta para este exemplo é: RUN ["c:\\windows\\system32\\tasklist.exe"]

O cache para instruções RUN não é invalidado automaticamente durante a próxima compilação. O cache para uma instrução como RUN apt-get dist-upgrade -y será reutilizado durante a próxima compilação. O cache para instruções RUN pode ser invalidado usando o sinalizador –no-cache, por exemplo, docker build –no-cache.

Consulte o guia de práticas recomendadas do Dockerfile para obter mais informações.

O cache para instruções RUN pode ser invalidado por instruções ADD. Veja abaixo para detalhes.

Instrução ONBULD #

ONBUILD [INSTRUCTION]

A instrução ONBUILD adiciona à imagem uma instrução de gatilho para ser executada posteriormente, quando a imagem for usada como base para outra compilação. O trigger será executado no contexto da compilação downstream, como se tivesse sido inserido imediatamente após a instrução FROM no Dockerfile downstream.

Qualquer instrução de construção pode ser registrada como um gatilho.

Isso é útil se você estiver construindo uma imagem que será usada como base para construir outras imagens, por exemplo, um ambiente de construção de aplicativo ou um daemon que pode ser customizado com configuração específica do usuário.

Por exemplo, se sua imagem for um construtor de aplicativos Python reutilizável, ela exigirá que o código-fonte do aplicativo seja incluído em um diretório específico e poderá exigir que um script de compilação seja chamado depois disso. Você não pode simplesmente chamar ADD e RUN agora, porque você ainda não tem acesso ao código-fonte do aplicativo e será diferente para cada compilação do aplicativo. Você poderia simplesmente fornecer aos desenvolvedores de aplicativos um Dockerfile padrão para copiar e colar em seu aplicativo, mas isso é ineficiente, propenso a erros e difícil de atualizar porque se mistura com o código específico do aplicativo.

A solução é usar o ONBUILD para registrar instruções avançadas para serem executadas posteriormente, durante o próximo estágio de construção.

Veja como funciona:

Quando encontra uma instrução ONBUILD, o construtor adiciona um gatilho aos metadados da imagem que está sendo construída. A instrução não afeta a compilação atual.

No final da compilação, uma lista de todos os gatilhos é armazenada no manifesto da imagem, sob a chave OnBuild. Eles podem ser inspecionados com o comando docker inspect. Mais tarde a imagem pode ser usada como base para uma nova compilação, usando a instrução FROM. Como parte do processamento da instrução FROM, o construtor downstream procura por gatilhos ONBUILD e os executa na mesma ordem em que foram registrados. Se algum dos gatilhos falhar, a instrução FROM será abortada, o que, por sua vez, fará com que a compilação falhe. Se todos os gatilhos forem bem-sucedidos, a instrução FROM será concluída e a compilação continuará normalmente.

Os gatilhos são apagados da imagem final após serem executados. Em outras palavras, eles não são herdados por construções de “netos”.

Por exemplo, você pode adicionar algo assim:

[...]
ONBUILD ADD . /app/src
ONBUILD RUN /usr/local/bin/python-build --dir /app/src
[...]

Aviso: O encadeamento de instruções ONBUILD usando ONBUILD ONBUILD não é permitido.

Atenção: A instrução ONBUILD pode não acionar as instruções FROM ou MAINTAINER.

Instrução STOPSIGNAL #

STOPSIGNAL signal

A instrução STOPSIGNAL configura o sinal de chamada do sistema que será enviado ao container para sair. Este sinal pode ser um número não assinado válido que corresponde a uma posição na tabela syscall do kernel, por exemplo 9, ou um nome de sinal no formato SIGNAME, por exemplo SIGKILL.

Instruções do HEALTHCHECK #

A instrução HEALTHCHECK tem duas formas:

HEALTHCHECK [OPTIONS] CMD command (check container health by running a command inside the container)
HEALTHCHECK NONE (disable any healthcheck inherited from the base image)

A instrução HEALTHCHECK informa ao Docker como testar um contêiner para verificar se ele ainda está funcionando. Isso pode detectar casos como um servidor da Web que está preso em um loop infinito e incapaz de lidar com novas conexões, mesmo que o processo do servidor ainda esteja em execução.

Quando um contêiner tem uma verificação de integridade especificada, ele possui um status de integridade além de seu status normal. Este status está começando inicialmente. Sempre que uma verificação de integridade é aprovada, ela se torna íntegra (independentemente do estado em que estava anteriormente). Após um certo número de falhas consecutivas, torna-se insalubre.

As opções que podem aparecer antes de CMD são:

--interval=DURATION (default: 30s)
--timeout=DURATION (default: 30s)
--retries=N (default: 3)

A verificação de integridade executará primeiro o intervalo de segundos após o contêiner ser iniciado e, novamente, o intervalo de segundos após a conclusão de cada verificação anterior.

Se uma única execução da verificação demorar mais do que segundos de tempo limite, a verificação será considerada com falha.

São necessárias novas tentativas de falhas consecutivas da verificação de integridade para que o contêiner seja considerado não íntegro.

Só pode haver uma instrução HEALTHCHECK em um Dockerfile. Se você listar mais de um, somente o último HEALTHCHECK terá efeito.

O comando após a palavra-chave CMD pode ser um comando shell (por exemplo, HEALTHCHECK CMD /bin/check-running) ou um array exec (como com outros comandos do Dockerfile; veja, por exemplo, ENTRYPOINT para detalhes).

O status de saída do comando indica o status de integridade do contêiner. Os valores possíveis são:

  • 0: success - o container está íntegro e pronto para uso
  • 1: insalubre - o contêiner não está funcionando corretamente
  • 2: start - o container ainda não está pronto para uso, mas está funcionando corretamente

Se o probe retornar 2 (“iniciando”) quando o contêiner já tiver saído do estado “inicial”, ele será tratado como “não saudável”.

Por exemplo, para verificar a cada cinco minutos mais ou menos se um servidor web é capaz de servir a página principal do site em três segundos:

HEALTHCHECK --interval=5m --timeout=3s \
  CMD curl -f http://localhost/ || exit 1

Para ajudar a depurar testes com falha, qualquer texto de saída (codificado em UTF-8) que o comando grava em stdout ou stderr será armazenado no status de integridade e pode ser consultado com docker inspect. Essa saída deve ser mantida curta (apenas os primeiros 4096 bytes são armazenados atualmente).

Quando o status de integridade de um contêiner muda, um evento health_status é gerado com o novo status.

O recurso HEALTHCHECK foi adicionado no Docker 1.12.

Instrução SHELL #

SHELL ["executable", "parameters"]

A instrução SHELL permite que o shell padrão usado para a forma shell dos comandos seja substituído. O shell padrão no Linux é ["/bin/sh", "-c"] e no Windows é ["cmd", "/S", "/C"]. A instrução SHELL deve ser escrita no formato JSON em um Dockerfile.

A instrução SHELL é particularmente útil no Windows onde existem dois shells nativos comumente usados ​​e bastante diferentes: cmd e powershell, bem como shells alternativos disponíveis, incluindo sh.

A instrução SHELL pode aparecer várias vezes. Cada instrução SHELL substitui todas as instruções SHELL anteriores e afeta todas as instruções subsequentes. Por exemplo:

FROM windowsservercore

# Executed as cmd /S /C echo default
RUN echo default

# Executed as cmd /S /C powershell -command Write-Host default
RUN powershell -command Write-Host default

# Executed as powershell -command Write-Host hello
SHELL ["powershell", "-command"]
RUN Write-Host hello

# Executed as cmd /S /C echo hello
SHELL ["cmd", "/S"", "/C"]
RUN echo hello

As instruções a seguir podem ser afetadas pela instrução SHELL quando a forma de shell delas é usada em um Dockerfile: RUN, CMD e ENTRYPOINT.

O exemplo a seguir é um padrão comum encontrado no Windows que pode ser simplificado usando a instrução SHELL:

...
RUN powershell -command Execute-MyCmdlet -param1 "c:\foo.txt"
...

O comando invocado pelo docker será:

cmd /S /C powershell -command Execute-MyCmdlet -param1 "c:\foo.txt"

Isso é ineficiente por dois motivos. Primeiro, há um processador de comando cmd.exe desnecessário (também conhecido como shell) sendo invocado. Em segundo lugar, cada instrução RUN no formulário shell requer um comando powershell extra como prefixo do comando.

Para tornar isso mais eficiente, um dos dois mecanismos pode ser empregado. Uma é usar a forma JSON do comando RUN como:

...
RUN ["powershell", "-command", "Execute-MyCmdlet", "-param1 \"c:\\foo.txt\""]
...

Embora o formulário JSON não seja ambíguo e não use o cmd.exe desnecessário, ele exige mais detalhamento por meio de aspas duplas e escape. O mecanismo alternativo é usar a instrução SHELL e o formulário shell, tornando uma sintaxe mais natural para usuários do Windows, especialmente quando combinada com a diretiva do analisador de escape:

# escape=`

FROM windowsservercore
SHELL ["powershell","-command"]
RUN New-Item -ItemType Directory C:\Example
ADD Execute-MyCmdlet.ps1 c:\example\
RUN c:\example\Execute-MyCmdlet -sample 'hello world'

A instrução SHELL também pode ser usada para modificar a maneira como um shell opera. Por exemplo, usando SHELL cmd /S /C /V:ON|OFF no Windows, a semântica de expansão de variável de ambiente atrasada pode ser modificada.

A instrução SHELL também pode ser usada no Linux caso um shell alternativo seja necessário, como zsh, csh, tcsh e outros.

O recurso SHELL foi adicionado no Docker 1.12.

Copiando arquivos #

Para copiar arquivos do contexto de compilação em uma imagem do Docker, use a instrução COPY:

COPY localfile.txt containerfile.txt

Se o nome do arquivo contiver espaços, use a sintaxe alternativa:

COPY ["local file", "container file"]

O comando COPY suporta curingas. Ele pode ser usado, por exemplo, para copiar todas as imagens para o diretório images/:

COPY *.jpg images/

Nota: neste exemplo, images/ pode não existir. Nesse caso, o Docker o criará automaticamente.

Expondo uma porta #

Para declarar portas expostas de um Dockerfile use a instrução EXPOSE:

EXPOSE 8080 8082

A configuração de portas expostas pode ser substituída na linha de comando do Docker, mas é uma boa prática defini-las explicitamente no Dockerfile, pois isso ajuda a entender o que um aplicativo faz.

Instalando pacotes Debian/Ubuntu #

Execute a instalação em um único comando de execução para mesclar a atualização e a instalação. Se você adicionar mais pacotes posteriormente, isso executará a atualização novamente e instalará todos os pacotes necessários. Se a atualização for executada separadamente, ela será armazenada em cache e as instalações de pacotes poderão falhar. Definir o frontend como não interativo e passar o -y para instalar é necessário para instalações com script. A limpeza e a purga no final da instalação minimizam o tamanho da camada.

FROM debian

RUN apt-get update \
 && DEBIAN_FRONTEND=noninteractive apt-get install -y \
    git \
    openssh-client \
    sudo \
    vim \
    wget \
 && apt-get clean \
 && rm -rf /var/lib/apt/lists/*
