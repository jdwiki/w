Um swarm é uma série de Docker Engines (ou nós) que implantam serviços coletivamente. O Swarm é usado para distribuir o processamento em muitas máquinas físicas, virtuais ou em nuvem.

Sintaxe #

  • Inicializar um enxame: docker swarm init [OPÇÕES]

  • Junte-se a um enxame como um nó e/ou gerente: docker swarm join [OPÇÕES] HOST:PORT

  • Criar um novo serviço: docker service create [OPTIONS] IMAGE [COMMAND] [ARG…]

  • Exibir informações detalhadas sobre um ou mais serviços: docker service inspect [OPTIONS] SERVICE [SERVICE…]

  • List services: docker service ls [OPTIONS]

  • Remover um ou mais serviços: docker service rm SERVICE [SERVICE…]

  • Escale um ou vários serviços replicados: escala de serviço docker SERVICE=REPLICAS [SERVICE=REPLICAS…]

  • Listar as tarefas de um ou mais serviços: docker service ps [OPTIONS] SERVICE [SERVICE…]

  • Atualizar um serviço: atualização do serviço docker [OPTIONS] SERVICE

O modo Swarm implementa os seguintes recursos:

  • Gerenciamento de cluster integrado ao Docker Engine
  • Design descentralizado
  • Modelo de serviço declarativo
  • Dimensionamento
  • Reconciliação do estado desejado
  • Rede multi-host
  • Descoberta de serviço
  • Balanceamento de carga
  • Design seguro por padrão
  • Atualizações contínuas

Para obter mais documentação oficial do Docker sobre o Swarm, visite: Visão geral do modo Swarm

Crie um enxame no Linux usando docker-machine e VirtualBox #

# Create the nodes
# In a real world scenario we would use at least 3 managers to cover the fail of one manager.
docker-machine create -d virtualbox manager
docker-machine create -d virtualbox worker1

# Create the swarm
# It is possible to define a port for the *advertise-addr* and *listen-addr*, if none is defined the default port 2377 will be used.
docker-machine ssh manager \
    docker swarm init \
    --advertise-addr $(docker-machine ip manager)
    --listen-addr $(docker-machine ip manager)

# Extract the Tokens for joining the Swarm
# There are 2 different Tokens for joining the swarm.
MANAGER_TOKEN=$(docker-machine ssh manager docker swarm join-token manager --quiet)
WORKER_TOKEN=$(docker-machine ssh manager docker swarm join-token worker --quiet)


# Join a worker node with the worker token
docker-machine ssh worker1 \
    docker swarm join \
    --token $WORKER_TOKEN \
    --listen-addr $(docker-machine ip worker1) \
    $(docker-machine ip manager):2377

Disponibilidade do nó #

Disponibilidade do nó do modo enxame:

  • Ativo significa que o agendador pode atribuir tarefas a um nó.
  • Pausa significa que o agendador não atribui novas tarefas ao nó, mas as tarefas existentes permanecem em execução.
  • Drenar significa que o agendador não atribui novas tarefas ao nó. o agendador encerra todas as tarefas existentes e as agenda em um nó disponível.

Para alterar a disponibilidade do modo:

#Following commands can be used on swarm manager(s)
docker node update --availability drain node-1
#to verify:
docker node ls

Promover ou rebaixar nós de enxame #

Para promover um nó ou conjunto de nós, execute docker node promote a partir de um nó do gerenciador:

docker node promote node-3 node-2

Node node-3 promoted to a manager in the swarm.
Node node-2 promoted to a manager in the swarm.

Para rebaixar um nó ou um conjunto de nós, execute docker node demote em um nó do gerenciador:

docker node demote node-3 node-2

Manager node-3 demoted in the swarm.
Manager node-2 demoted in the swarm.

Descubra o token de ingresso do trabalhador e do gerente #

Ao automatizar o provisionamento de novos nós para um swarm, você precisa saber qual é o token de junção correto para o swarm, bem como o endereço anunciado do gerente. Você pode descobrir isso executando os seguintes comandos em qualquer um dos nós gerenciadores existentes:

# grab the ipaddress:port of the manager (second last line minus the whitespace)
export MANAGER_ADDRESS=$(docker swarm join-token worker | tail -n 2 |  tr -d '[[:space:]]')

# grab the manager and worker token
export MANAGER_TOKEN=$(docker swarm join-token manager -q)
export WORKER_TOKEN=$(docker swarm join-token worker -q)

A opção -q gera apenas o token. Sem esta opção, você obtém o comando completo para se registrar em um enxame.

Em seguida, em nós recém-provisionados, você pode ingressar no enxame usando.

docker swarm join --token $WORKER_TOKEN $MANAGER_ADDRESS

Aplicativo Olá mundo #

Normalmente, você deseja criar uma pilha de serviços para formar um aplicativo replicado e orquestrado.

Um aplicativo Web moderno típico consiste em um banco de dados, uma API, um front-end e um proxy reverso.

Persistência

O banco de dados precisa de persistência, então precisamos de algum sistema de arquivos que seja compartilhado por todos os nós em um enxame. Pode ser NAS, servidor NFS, GFS2 ou qualquer outra coisa. Configurá-lo está fora do escopo aqui. Atualmente o Docker não contém e não gerencia a persistência em um swarm. Este exemplo assume que há um local compartilhado /nfs/ montado em todos os nós.

Rede

Para poder se comunicar uns com os outros, os serviços em um enxame precisam estar na mesma rede.

Escolha um intervalo de IP (aqui 10.0.9.0/24) e nome de rede (hello-network) e execute um comando:

docker network create \
  --driver overlay \
  --subnet 10.0.9.0/24 \
  --opt encrypted \
  hello-network

Base de dados

O primeiro serviço que precisamos é um banco de dados. Vamos usar o postgresql como exemplo. Crie uma pasta para um banco de dados em nfs/postgres e execute isto:

docker service create --replicas 1 --name hello-db \
       --network hello-network -e PGDATA=/var/lib/postgresql/data \
       --mount type=bind,src=/nfs/postgres,dst=/var/lib/postgresql/data \
       kiasaki/alpine-postgres:9.5

Observe que usamos as opções --network hello-network e --mount.

API

A criação de API está fora do escopo deste exemplo, então vamos fingir que você tem uma imagem de API em username/hello-api.

docker service create --replicas 1 --name hello-api \
       --network hello-network \
       -e NODE_ENV=production -e PORT=80 -e POSTGRESQL_HOST=hello-db \
       username/hello-api

Observe que passamos um nome do nosso serviço de banco de dados. O Docker Swarm possui um servidor DNS round-robin incorporado, portanto, a API poderá se conectar ao banco de dados usando seu nome DNS.

Procurado reverso

Vamos criar o serviço nginx para servir nossa API para um mundo externo. Crie arquivos de configuração nginx em um local compartilhado e execute isto:

docker service create --replicas 1 --name hello-load-balancer \
       --network hello-network \
       --mount type=bind,src=/nfs/nginx/nginx.conf,dst=/etc/nginx/nginx.conf \
       -p 80:80 \
       nginx:1.10-alpine

Observe que usamos a opção -p para publicar um port. Esta porta estaria disponível para qualquer nó em um enxame.

Deixando o enxame #

Nó do trabalhador:

#Run the following on the worker node to leave the swarm.

docker swarm leave
Node left the swarm.

Se o nó tiver a função Gerente, você receberá um aviso sobre a manutenção do quorum de gerentes. Você pode usar –force para deixar no nó do gerenciador:

#Manager Node

docker swarm leave --force
Node left the swarm.

Os nós que deixaram o Swarm ainda aparecerão na saída do docker node ls.

Para remover nós da lista:

docker node rm node-2
