Instalação e configuração #

Instalação #

Instalando o Codename One no NetBeans #

Estas instruções pressupõem que você tenha baixado uma versão recente do NetBeans (no momento 8.x), instalado e iniciado.

  • Selecione a opção de menu Ferramentas->Plugins

  • Selecione a guia Plug-ins Disponíveis

  • Verifique o plugin CodenameOne

  • clique no botão instalar abaixo. Siga as instruções do assistente para instalar o plug-in

Instalando o Codename One no Eclipse #

Inicie o Eclipse e clique em Help->Install New Software. Você deve obter esta caixa de diálogo

Cole https://www.codenameone.com/files/eclipse/site.xml para o local para trabalhar e pressione Enter.

Selecione as entradas e siga o assistente para instalar

Instalando o Codename One no IntelliJ IDEA #

Baixe e instale o IntelliJ/IDEA. Observe que o Android Studio não funcionará.

Instale o plug-in usando o Centro de plug-ins

Use a funcionalidade de pesquisa no centro de plugins para encontrar e instalar o plugin Codename One.

O que é o Codename One e como funciona? #

Codename One é um conjunto de ferramentas para desenvolvimento de aplicativos móveis que derivam grande parte de sua arquitetura de Java.

A declaração de missão do Codename One é:

Unifique a tarefa complexa e fragmentada de programação de dispositivos móveis em um único conjunto de ferramentas, APIs e serviços. Como resultado, crie um ambiente mais abordagem gerenciável para o desenvolvimento de aplicativos móveis sem sacrificando o poder/controle dado aos desenvolvedores.

Isso significa efetivamente trazer o velho mantra Java “Write Once Run Anywhere” (WORA) para dispositivos móveis sem “emburrecer” o menor denominador comum.

Como o Codename One funciona? #

Codename One unifica várias tecnologias e conceitos em uma única fachada:

  • API - abstrai as diferenças entre os vários dispositivos.
  • Plugin - o único software instalado nas máquinas clientes, inclui os seguintes recursos:
    • IDE integration - preferences, completion, the ability to send a native build
    • Simulator - native device simulator that runs locally and allows debugging the application
    • Designer/GUI Builder - high level tools
  • Servidores de compilação - Os servidores de compilação aceitam compilações de dispositivos nativos enviadas pelo plug-in e convertem os binários (JARs, não fontes) em aplicativos nativos conforme explicado abaixo.
  • Servidores em nuvem - Os servidores em nuvem fornecem recursos como notificação por push, registro em nuvem etc.

Por que construir servidores? #

Os servidores de compilação permitem criar aplicativos iOS nativos sem um Mac e aplicativos Windows nativos sem uma máquina Windows. Eles eliminam a necessidade de instalar/atualizar cadeias de ferramentas complexas e simplificam o processo de criação de um aplicativo nativo com um clique com o botão direito.

Por exemplo: Como construir aplicativos iOS nativos requer uma máquina Mac OS X com uma versão recente do xcode Codename One mantém essas máquinas na nuvem. Quando os desenvolvedores enviam uma compilação do iOS, esse Mac será usado para gerar o código-fonte C usando ParparVM e então compilará o código-fonte C usando xcode e assinará o binário resultante usando xcode. Você pode instalar o binário em seu dispositivo ou criar um binário de distribuição para a appstore. Como o código C é gerado, isso também significa que seu aplicativo será “à prova de futuro” em caso de alterações da Apple. Você também pode injetar código nativo do Objective-C no aplicativo, mantendo-o 100% portátil graças ao recurso de “interfaces nativas” do Codename One.

Os assinantes podem receber o código-fonte C de volta usando o recurso de inclusão de fontes do Codename One e usar essas fontes para benchmarking, depuração em dispositivos etc.

O mesmo vale para a maioria das outras plataformas. Para o Android, J2ME e Blackberry, o código Java padrão é executado como está.

A sintaxe do Java 8 é suportada através do retrolambda instalado nos servidores Codename One. Isso é usado para converter bytecode perfeitamente para níveis de sintaxe Java 5. A sintaxe do Java 5 é traduzida para o subconjunto JDK 1.3 cldc no J2ME/Blackberry para fornecer esses recursos de linguagem e APIs em todos os dispositivos. Isso é feito usando um processador de bytecode baseado em servidor baseado em retroweaver e uma grande quantidade de código personalizado. Observe que essa arquitetura é transparente para os desenvolvedores, pois os servidores de compilação abstraem a maioria das diferenças dolorosas entre os dispositivos.

Por que ParparVM #

No iOS, Codename One usa ParparVM que traduz bytecode Java para código C e possui um GC sem bloqueio, bem como suporte a 64 bits/bitcode. Esta VM é totalmente open source no Codename One git repository. No passado, o Codename One usava XMLVM para gerar código nativo de maneira muito semelhante, mas a solução XMLVM era muito genérica para as necessidades do Codename One. ParparVM possui uma arquitetura única de tradução de código para C (semelhante ao XMLVM), por isso Codename One é a única solução desse tipo que pode garantir compatibilidade futura com iOS, já que a cadeia de ferramentas iOS oficialmente suportada é sempre usado em vez de comportamentos não documentados.

OBSERVAÇÃO: XMLVM poderia garantir isso em teoria, mas não é mais mantido.

As principais vantagens do ParparVM sobre outras abordagens são:

  • Verdadeiramente nativo - como o código é traduzido para C em vez de diretamente para o código ARM ou LLVM, o aplicativo é “mais nativo”. Ele usa as ferramentas e abordagens oficiais da Apple e pode se beneficiar de seus avanços, por exemplo. os mais recentes recursos de bitcode ou criação de perfil.

  • Biblioteca de classes menor - ParparVM inclui um segmento muito pequeno das APIs Java completas, resultando em binários finais menores que as alternativas em ordens de magnitude. Isso mapeia diretamente para desempenho e sobrecarga de memória.

  • Simples e extensível - para trabalhar com ParparVM, você precisa de um conhecimento básico de C. Isso é crucial para o mundo em rápida evolução do desenvolvimento móvel, pois a Apple muda as coisas para a esquerda e para a direita, precisamos de uma VM mais ágil.

Windows Phone/UWP #

Codename One tem 2 portas principais de VM do Windows e 3 ou 4 pipelines de renderização nessas portas.

A antiga porta do Windows Phone usava XMLVM para traduzir o bytecode Java para C#. Observe que o back-end XMLVM que traduz para C# é muito diferente daquele que era usado no passado para traduzir código para iOS.

O Codename One agora tem como alvo o UWP, aproveitando uma versão modificada do iKVM para criar aplicativos universais do Windows nativos.

Porta JavaScript #

A porta JavaScript do Codename One é baseada no incrível trabalho do projeto TeaVM. A equipe por trás do TeaVM construiu efetivamente uma JVM que traduz bytecode Java em código-fonte JavaScript, mantendo a semântica de encadeamento usando uma abordagem muito imaginativa.

A porta JavaScript permite que aplicativos Codename One não modificados sejam executados em um navegador de desktop ou móvel. A porta em si é baseada na API HTML5 Canvas para fornecer uma implementação perfeita em pixels das APIs Codename One.

OBSERVAÇÃO: A porta JavaScript está disponível apenas para assinantes de nível Enterprise do Codename One.

Desktop, Android, RIM e J2ME #

As outras portas do Codename One usam as VMs disponíveis nas máquinas/ambientes host para executar o runtime. https://github.com/orfjackal/retrolambda[Retrolambda] é usado para fornecer recursos da linguagem Java 8 de forma portátil, para dispositivos mais antigos o retroweaver é usado para trazer recursos do Java 5.

A porta Android usa as ferramentas nativas do Android, incluindo o ambiente de compilação gradle nas versões mais recentes.

A porta desktop cria um aplicativo JavaSE padrão que é empacotado com o JRE e um instalador.

OBSERVAÇÃO: A porta Desktop está disponível apenas para assinantes de nível profissional do Codename One.

Componentes leves #

O que faz o Codename One se destacar é a abordagem adotada para a interface do usuário, onde ele usa uma “arquitetura leve”, permitindo que a interface do usuário funcione perfeitamente em todas as plataformas. Como resultado, a maior parte da interface do usuário é desenvolvida em Java e, portanto, é notavelmente portátil e depurável. A arquitetura leve ainda inclui a capacidade de incorporar widgets “pesados” no lugar entre os “leves”.

Origem da Arquitetura Leve #

Componentes leves remontam aos frameworks Smalltalk, essa noção foi popularizada no mundo Java pelo Swing. Swing foi a principal fonte de inspiração para o antecessor de Codename One, LWUIT. Muitos frameworks adotaram essa abordagem ao longo dos anos, incluindo JavaFX e, mais recentemente, Ionic no mundo JavaScript.

No Codinome Um #

Um componente Lightweight é um componente escrito inteiramente em Java, desenha sua própria interface e manipula seus próprios eventos/estados. Isso tem enormes vantagens de portabilidade, pois o mesmo código é executado em todas as plataformas, mas traz muitas vantagens adicionais.

Componentes leves são infinitamente personalizáveis ​​usando herança padrão e substituindo a manipulação de pintura/evento. Como um componente leve é ​​escrito inteiramente em Java, os desenvolvedores podem visualizar o aplicativo com precisão nos simuladores e construtor de GUI. Isso evita muitas armadilhas comuns de outras soluções WORA em que o comportamento específico da plataforma frustrou qualquer esforço economizado. Portanto, todo o esforço economizado na codificação foi perdido na depuração de dispositivos esotéricos apenas esquisitices.

O Codename One alcança um desempenho rápido desenhando usando as APIs de jogos nativas da maioria das plataformas, por exemplo OpenGL ES no iOS.

Versões em Codename One #

Uma das coisas confusas sobre Codename One são as versões. Como o Codename One é um produto SaaS, o versionamento não é tão simples quanto um moniker 2.x ou 3.x. No entanto, para estar em conformidade com esta convenção, o Codename One faz versões com versões que contribuem para a confusão geral.

Quando uma versão do Codename One é lançada, o número da versão se refere às bibliotecas no momento do lançamento. Essas bibliotecas são congeladas e disponibilizadas para desenvolvedores que usam o recurso [Versioned Builds][7]. O plug-in, que inclui o designer, bem como todo o desenvolvimento não relacionado a versões com versões, continua com suas atualizações regulares imediatamente após o lançamento. O mesmo vale para os servidores de compilação que se movem diretamente para seu ciclo de atualização padrão.

[7]: https://www.codenameone.com/how-do-i—get-repeatable-builds-build-against-a-consistent-version-of-codename-one-use-the-versioning- recurso.html
