Uma interface é um tipo de referência, semelhante a uma classe, que pode ser declarada usando a palavra-chave interface.

As interfaces podem conter apenas constantes, assinaturas de métodos, métodos padrão, métodos estáticos e tipos aninhados. Corpos de método existem apenas para métodos padrão e métodos estáticos. Assim como as classes abstratas, as Interfaces não podem ser instanciadas – elas só podem ser implementadas por classes ou estendidas por outras interfaces.

Interface é uma maneira comum de obter abstração completa em Java.

Sintaxe #

  • public interface Foo { void foo(); /* quaisquer outros métodos */ }
  • interface pública Foo1 estende Foo { void bar(); /* quaisquer outros métodos */ }
  • public class Foo2 implementa Foo, Foo1 { /* implementação de Foo e Foo1 */ }

Implementando várias interfaces #

Uma classe Java pode implementar várias interfaces.

public interface NoiseMaker {
    String noise = "Making Noise"; // interface variables are public static final by default

    String makeNoise(); //interface methods are public abstract by default
}

public interface FoodEater {
    void eat(Food food);
}

public class Cat implements NoiseMaker, FoodEater { 
    @Override
    public String makeNoise() {
        return "meow";
    }

    @Override
    public void eat(Food food) {
        System.out.println("meows appreciatively");
    }
}

Observe como a classe Cat deve implementar os métodos abstract herdados em ambas as interfaces. Além disso, observe como uma classe pode implementar praticamente quantas interfaces forem necessárias (há um limite de 65.535 devido a JVM Limitation).

NoiseMaker noiseMaker = new Cat(); // Valid
FoodEater foodEater = new Cat(); // Valid
Cat cat = new Cat(); // valid

Cat invalid1 = new NoiseMaker(); // Invalid
Cat invalid2 = new FoodEater(); // Invalid

Observação:

  1. Todas as variáveis ​​declaradas em uma interface são ‘public static final’
  2. Todos os métodos declarados em métodos de interface são public abstract (Esta declaração é válida apenas através do Java 7. A partir do Java 8, você tem permissão para ter métodos em uma interface, que não precisam ser abstratos; tais métodos são conhecidos como métodos padrão)
  3. Interfaces não podem ser declaradas como final
  4. Se mais de uma interface declarar um método com assinatura idêntica, ele será efetivamente tratado como apenas um método e você não poderá distinguir de qual método de interface está implementado
  5. Um arquivo InterfaceName.class correspondente seria gerado para cada interface, após a compilação

Declarando e Implementando uma Interface #

Declaração de uma interface usando a palavra-chave interface:

public interface Animal {
    String getSound(); // Interface methods are public by default
}

Substituir anotação

@Override
public String getSound() {
    // Code goes here...
}

Isso força o compilador a verificar se estamos substituindo e evita que o programa defina um novo método ou estrague a assinatura do método.

**As interfaces são implementadas usando a palavra-chave ** implements.

public class Cat implements Animal {

    @Override 
    public String getSound() {
        return "meow";
    }
}

public class Dog implements Animal {

    @Override
    public String getSound() {
        return "woof";
    }
}

No exemplo, as classes Cat e Dog devem definir o método getSound() já que os métodos de uma interface são inerentemente abstratos (com exceção dos métodos padrão).

Usando as interfaces

Animal cat = new Cat();
Animal dog = new Dog();

System.out.println(cat.getSound()); // prints "meow"
System.out.println(dog.getSound()); // prints "woof"

Estendendo uma interface #

Uma interface pode estender outra interface através da palavra-chave extends.

public interface BasicResourceService {
    Resource getResource();
}

public interface ExtendedResourceService extends BasicResourceService {
    void updateResource(Resource resource);
}

Agora, uma classe que implementa ExtendedResourceService precisará implementar getResource() e updateResource().

Estendendo várias interfaces

Ao contrário das classes, a palavra-chave extends pode ser usada para estender várias interfaces (separadas por vírgulas) permitindo combinações de interfaces em uma nova interface

public interface BasicResourceService {
    Resource getResource();
}

public interface AlternateResourceService {
    Resource getAlternateResource();
}

public interface ExtendedResourceService extends BasicResourceService, AlternateResourceService {
    Resource updateResource(Resource resource);
}

Neste caso, uma classe implementando ExtendedResourceService precisará implementar getResource(), getAlternateResource() e updateResource().

Utilidade das interfaces #

As interfaces podem ser extremamente úteis em muitos casos. Por exemplo, digamos que você tenha uma lista de animais e queira percorrer a lista, cada um imprimindo o som que eles fazem.

{cat, dog, bird}

Uma maneira de fazer isso seria usar interfaces. Isso permitiria que o mesmo método fosse chamado em todas as classes

public interface Animal {
    public String getSound();
}

Qualquer classe que implementa Animal também deve ter um método getSound(), mas todas podem ter implementações diferentes

public class Dog implements Animal {
    public String getSound() {
        return "Woof";
    }
}

public class Cat implements Animal {
    public String getSound() {
        return "Meow";
    }
}

public class Bird implements Animal{
    public String getSound() {
        return "Chirp";
    }
}

Agora temos três classes diferentes, cada uma com um método getSound(). Como todas essas classes implementam a interface Animal, que declara o método getSound(), qualquer instância de um Animal pode ter getSound() chamado nele

Animal dog = new Dog();
Animal cat = new Cat();
Animal bird = new Bird();

dog.getSound(); // "Woof"
cat.getSound(); // "Meow"
bird.getSound(); // "Chirp"

Como cada um deles é um ‘Animal’, podemos até colocar os animais em uma lista, percorrê-los e imprimir seus sons

Animal[] animals = { new Dog(), new Cat(), new Bird() };
for (Animal animal : animals) {
    System.out.println(animal.getSound());
}

Como a ordem do array é Dog, Cat e depois Bird, “Woof Meow Chirp” será impresso no console.

Interfaces também podem ser usadas como valor de retorno para funções. Por exemplo, retornando um Dog se a entrada for “dog”, Cat se a entrada for “cat” e Bird se for “bird” e, em seguida, imprimir o som de esse animal poderia ser feito usando

public Animal getAnimalByName(String name) {
    switch(name.toLowerCase()) {
        case "dog":
            return new Dog();
        case "cat":
            return new Cat();
        case "bird":
            return new Bird();
        default:
            return null;
    }
}

public String getAnimalSoundByName(String name){
    Animal animal = getAnimalByName(name);
    if (animal == null) {
        return null;
    } else { 
        return animal.getSound();
    }
}

String dogSound = getAnimalSoundByName("dog"); // "Woof"
String catSound = getAnimalSoundByName("cat"); // "Meow"
String birdSound = getAnimalSoundByName("bird"); // "Chirp"
String lightbulbSound = getAnimalSoundByName("lightbulb"); // null

Interfaces também são úteis para extensibilidade, porque se você quiser adicionar um novo tipo de ‘Animal’, você não precisaria alterar nada com as operações que você executa neles.

Métodos padrão #

Introduzidos no Java 8, os métodos padrão são uma maneira de especificar uma implementação dentro de uma interface. Isso pode ser usado para evitar a típica classe “Base” ou “Abstract”, fornecendo uma implementação parcial de uma interface e restringindo a hierarquia de subclasses.

Implementação do padrão Observer #

Por exemplo, é possível implementar o padrão Observer-Listener diretamente na interface, proporcionando mais flexibilidade às classes implementadoras.

interface Observer {
    void onAction(String a);
}

interface Observable{
    public abstract List<Observer> getObservers();

    public default void addObserver(Observer o){
        getObservers().add(o);
    }

    public default void notify(String something ){
        for( Observer l : getObservers() ){
            l.onAction(something);
        }
    }
}

Agora, qualquer classe pode se tornar “Observável” apenas implementando a interface Observable, sendo livre para fazer parte de uma hierarquia de classes diferente.

abstract class Worker{
    public abstract void work();
}

public class MyWorker extends Worker implements Observable {

    private List<Observer> myObservers = new ArrayList<Observer>();
    
    @Override
    public List<Observer> getObservers() {
        return myObservers;
    }

    @Override
    public void work(){
        notify("Started work");

        // Code goes here...

        notify("Completed work");
    }
    
    public static void main(String[] args) {    
        MyWorker w = new MyWorker();
       
        w.addListener(new Observer() {
            @Override
            public void onAction(String a) {
                System.out.println(a + " (" + new Date() + ")");
            }
        });
        
        w.work();
    }
}

Problema do diamante #

O compilador em Java 8 está ciente do problema do diamante que é causado quando uma classe está implementando interfaces contendo um método com a mesma assinatura.

Para resolvê-lo, uma classe de implementação deve substituir o método compartilhado e fornecer sua própria implementação.

interface InterfaceA {
    public default String getName(){
        return "a";
    }
}

interface InterfaceB {
    public default String getName(){
        return "b";
    }
}

public class ImpClass implements InterfaceA, InterfaceB {

    @Override
    public String getName() {    
        //Must provide its own implementation
        return InterfaceA.super.getName() + InterfaceB.super.getName();
    }
    
    public static void main(String[] args) {    
        ImpClass c = new ImpClass();
        
        System.out.println( c.getName() );                   // Prints "ab"
        System.out.println( ((InterfaceA)c).getName() );     // Prints "ab"
        System.out.println( ((InterfaceB)c).getName() );     // Prints "ab"
    }
}

Ainda há o problema de ter métodos com o mesmo nome e parâmetros com tipos de retorno diferentes, que não compilam.

Use métodos padrão para resolver problemas de compatibilidade #

As implementações de métodos padrão são muito úteis se um método for adicionado a uma interface em um sistema existente onde as interfaces são usadas por várias classes.

Para evitar a quebra de todo o sistema, você pode fornecer uma implementação de método padrão ao adicionar um método a uma interface. Desta forma, o sistema ainda irá compilar e as implementações reais podem ser feitas passo a passo.

Para obter mais informações, consulte o tópico Métodos padrão.

Modificadores em Interfaces #

O Oracle Java Style Guide declara:

Modificadores não devem ser escritos quando estão implícitos.

(Consulte Modifiers em [Oracle Official Code Standard](http://stackoverflow. com/documentation/java/2697/oracle-official-code-standard#t=201705210148056874377) para o contexto e um link para o documento Oracle real.)

Essa orientação de estilo se aplica particularmente a interfaces. Vamos considerar o seguinte trecho de código:

interface I {
    public static final int VARIABLE = 0;

    public abstract void method();

    public static void staticMethod() { ... }
    public default void defaultMethod() { ... }
}

Variáveis ​​## #

Todas as variáveis ​​de interface são implicitamente constantes com modificadores implícitos public (acessíveis para todos), static (são acessíveis pelo nome da interface) e final (deve ser inicializado durante a declaração):

final estático público int VARIABLE = 0;

Métodos #

  1. Todos os métodos que não fornecem implementação são implicitamente public e abstract.
resumo público método void();
  1. Todos os métodos com modificador static ou default devem fornecer implementação e são implicitamente públicos.
public static void staticMethod() { ... }

Depois que todas as alterações acima forem aplicadas, obteremos o seguinte:

interface I {
    int VARIABLE = 0;
    
    void method();

    static void staticMethod() { ... }
    default void defaultMethod() { ... }
}

Usando interfaces com genéricos #

Digamos que você queira definir uma interface que permita publicar/consumir dados de e para diferentes tipos de canais (por exemplo, AMQP, JMS, etc), mas deseja poder alternar os detalhes de implementação …

Vamos definir uma interface de E/S básica que pode ser reutilizada em várias implementações:

public interface IO<IncomingType, OutgoingType> {

    void publish(OutgoingType data);
    IncomingType consume();
    IncomingType RPCSubmit(OutgoingType data);

}

Agora posso instanciar essa interface, mas como não temos implementações padrão para esses métodos, ela precisará de uma implementação quando a instanciarmos:

    IO<String, String> mockIO = new IO<String, String>() {

        private String channel = "somechannel";

        @Override
        public void publish(String data) {
            System.out.println("Publishing " + data + " to " + channel);
        }

        @Override
        public String consume() {
            System.out.println("Consuming from " + channel);
            return "some useful data";
        }

        @Override
        public String RPCSubmit(String data) {
            return "received " + data + " just now ";
        }

    };

    mockIO.consume(); // prints: Consuming from somechannel
    mockIO.publish("TestData"); // Publishing TestData to somechannel
    System.out.println(mockIO.RPCSubmit("TestData")); // received TestData just now

Também podemos fazer algo mais útil com essa interface, digamos que queremos usá-la para envolver algumas funções básicas do RabbitMQ:

public class RabbitMQ implements IO<String, String> {

    private String exchange;
    private String queue;

    public RabbitMQ(String exchange, String queue){
        this.exchange = exchange;
        this.queue = queue;
    }

    @Override
    public void publish(String data) {
        rabbit.basicPublish(exchange, queue, data.getBytes());
    }

    @Override
    public String consume() {
        return rabbit.basicConsume(exchange, queue);
    }

    @Override
    public String RPCSubmit(String data) {
        return rabbit.rpcPublish(exchange, queue, data);
    }

}

Digamos que eu queira usar essa interface de E/S agora como uma maneira de contar as visitas ao meu site desde a última reinicialização do sistema e poder exibir o número total de visitas - você pode fazer algo assim:

import java.util.concurrent.atomic.AtomicLong;

public class VisitCounter implements IO<Long, Integer> {

    private static AtomicLong websiteCounter = new AtomicLong(0);
    
    @Override
    public void publish(Integer count) {
        websiteCounter.addAndGet(count);
    }

    @Override
    public Long consume() {
        return websiteCounter.get();
    }

    @Override
    public Long RPCSubmit(Integer count) {
        return websiteCounter.addAndGet(count);
    }
    
}

Agora vamos usar o VisitCounter:

    VisitCounter counter = new VisitCounter();

    // just had 4 visits, yay
    counter.publish(4);
    // just had another visit, yay
    counter.publish(1);

    // get data for stats counter
    System.out.println(counter.consume()); // prints 5

    // show data for stats counter page, but include that as a page view
    System.out.println(counter.RPCSubmit(1)); // prints 6

Ao implementar várias interfaces, você não pode implementar a mesma interface duas vezes. Isso também se aplica a interfaces genéricas. Assim, o código a seguir é inválido e resultará em um erro de compilação:

interface Printer<T> {
    void print(T value);
}

// Invalid!
class SystemPrinter implements Printer<Double>, Printer<Integer> {
    @Override public void print(Double d){ System.out.println("Decimal: " + d); }
    @Override public void print(Integer i){ System.out.println("Discrete: " + i); }
}

Reforçar parâmetros de tipo limitado #

Parâmetros de tipo limitado permitem definir restrições em argumentos de tipo genérico:

class SomeClass {

}

class Demo<T extends SomeClass> {

}

Mas um parâmetro de tipo só pode ser vinculado a um único tipo de classe.

Um tipo de interface pode ser associado a um tipo que já tinha uma associação. Isso é feito usando o símbolo &:

interface SomeInterface {

}

class GenericClass<T extends SomeClass & SomeInterface> {

}

Isso fortalece a ligação, potencialmente exigindo que os argumentos de tipo sejam derivados de vários tipos.

Vários tipos de interface podem ser vinculados a um parâmetro de tipo:

class Demo<T extends SomeClass & FirstInterface & SecondInterface> {

}

Mas deve ser usado com cautela. Múltiplas ligações de interface geralmente são um sinal de code smell, sugerindo que um novo tipo deve ser criado que atue como um adaptador para os outros tipos:

interface NewInterface extends FirstInterface, SecondInterface {

}

class Demo<T extends SomeClass & NewInterface> {

}

Implementando interfaces em uma classe abstrata #

Um método definido em uma interface é por padrão public abstract. Quando uma classe abstrata implementa uma interface, quaisquer métodos definidos na interface não precisam ser implementados pela classe abstrata. Isso ocorre porque uma class que é declarada como abstract pode conter declarações de métodos abstratos. Portanto, é responsabilidade da primeira subclasse concreta implementar quaisquer métodos abstract herdados de qualquer interface e/ou da classe abstrata.

public interface NoiseMaker {
    void makeNoise();
}

public abstract class Animal implements NoiseMaker {
    //Does not need to declare or implement makeNoise()
    public abstract void eat();
}

//Because Dog is concrete, it must define both makeNoise() and eat()
public class Dog extends Animal {
    @Override
    public void makeNoise() {
        System.out.println("Borf borf");
    }

    @Override
    public void eat() {
        System.out.println("Dog eats some kibble.");
    }
}

A partir do Java 8 é possível para uma interface declarar implementações default de métodos, o que significa que o método não será abstrato, portanto, quaisquer subclasses concretas não serão forçadas a implementar o método, mas herdarão a implementação default, a menos que substituída.
