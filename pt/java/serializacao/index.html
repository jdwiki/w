Java fornece um mecanismo, chamado serialização de objetos, onde um objeto pode ser representado como uma sequência de bytes que inclui os dados do objeto, bem como informações sobre o tipo do objeto e os tipos de dados armazenados no objeto.

Depois que um objeto serializado é gravado em um arquivo, ele pode ser lido do arquivo e desserializado, ou seja, as informações de tipo e os bytes que representam o objeto e seus dados podem ser usados ​​para recriar o objeto na memória.

Serialização básica em Java #

O que é serialização

A serialização é o processo de conversão do estado de um objeto (incluindo suas referências) em uma sequência de bytes, bem como o processo de reconstrução desses bytes em um objeto ativo em algum momento futuro. A serialização é usada quando você deseja persistir o objeto. Ele também é usado pelo Java RMI para passar objetos entre JVMs, como argumentos em uma chamada de método de um cliente para um servidor ou como valores de retorno de uma chamada de método ou como exceções lançadas por métodos remotos. Em geral, a serialização é usada quando queremos que o objeto exista além do tempo de vida da JVM.

java.io.Serializable é uma interface de marcador (não tem corpo). Ele é usado apenas para “marcar” classes Java como serializáveis.

O runtime de serialização associa a cada classe serializável um número de versão, chamado serialVersionUID, que é usado durante a des-serialização para verificar se o remetente e o destinatário de um objeto serializado carregaram classes para esse objeto que são compatíveis com relação a serialização. Se o receptor carregou uma classe para o objeto que tem um serialVersionUID diferente da classe do remetente correspondente, a desserialização resultará em uma InvalidClassException. Uma classe serializável pode declarar seu próprio serialVersionUID explicitamente declarando um campo chamado serialVersionUID que deve ser static, final e do tipo long:

ANY-ACCESS-MODIFIER static final long serialVersionUID = 1L;

Como tornar uma classe qualificada para serialização

Para persistir um objeto a respectiva classe deve implementar a interface java.io.Serializable.

import java.io.Serializable;

public class SerialClass implements Serializable {

    private static final long serialVersionUID = 1L;  
    private Date currentTime;
    
    public SerialClass() {
        currentTime = Calendar.getInstance().getTime();
    }

    public Date getCurrentTime() {
        return currentTime;
    }
}

Como escrever um objeto em um arquivo

Agora precisamos escrever este objeto em um sistema de arquivos. Usamos java.io.ObjectOutputStream para este propósito.

import java.io.FileOutputStream;
import java.io.ObjectOutputStream;
import java.io.IOException;

public class PersistSerialClass {

    public static void main(String [] args) {
        String filename = "time.ser";            
        SerialClass time = new SerialClass(); //We will write this object to file system.
        try {
            ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream(filename));
            out.writeObject(time); //Write byte stream to file system.
            out.close();
        } catch(IOException ex){
            ex.printStackTrace();
        }
     }
 }

Como recriar um objeto de seu estado serializado

O objeto armazenado pode ser lido do sistema de arquivos posteriormente usando java.io.ObjectInputStream como mostrado abaixo:

import java.io.FileInputStream;
import java.io.ObjectInputStream;
import java.io.IOException;
import java.io.java.lang.ClassNotFoundException;

 public class ReadSerialClass {

    public static void main(String [] args) {
        String filename = "time.ser";    
        SerialClass time = null;

        try {
            ObjectInputStream in = new ObjectInputStream(new FileInputStream(filename));
            time = (SerialClass)in.readObject();
            in.close();
        } catch(IOException ex){
            ex.printStackTrace();
        } catch(ClassNotFoundException cnfe){
            cnfe.printStackTrace();
        }
        // print out restored time
        System.out.println("Restored time: " + time.getTime());
     }
 }

A classe serializada está em formato binário. A desserialização pode ser problemática se a definição de classe for alterada: consulte o [capítulo sobre versão de objetos serializados da especificação de serialização Java](https://docs.oracle.com/javase/8/docs/platform/serialization/spec/version. html) para detalhes.

A serialização de um objeto serializa todo o gráfico do objeto do qual ele é a raiz e opera corretamente na presença de gráficos cíclicos. Um método reset() é fornecido para forçar o ObjectOutputStream a esquecer objetos que já foram serializados.

Campos transitórios - Serialização

Serialização personalizada #

Neste exemplo, queremos criar uma classe que irá gerar e enviar para o console, um número aleatório entre um intervalo de dois inteiros que são passados ​​como argumentos durante a inicialização.

    public class SimpleRangeRandom implements Runnable {
    private int min;
    private int max;

    private Thread thread;

    public SimpleRangeRandom(int min, int max){
        this.min = min;
        this.max = max;
        thread = new Thread(this);
        thread.start();
    }

    @Override
 private void WriteObject(ObjectOutputStreamout) throws IO Exception;
    private void ReadObject(ObjectInputStream in) throws IOException, ClassNotFoundException;
    public void run() {
        while(true) {
            Random rand = new Random();
            System.out.println("Thread: " + thread.getId() + " Random:" + rand.nextInt(max - min));
            try {
                Thread.sleep(10000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }
    }

Agora, se quisermos tornar essa classe Serializable, haverá alguns problemas. O Thread é uma das certas classes de nível de sistema que não são Serializable. Portanto, precisamos declarar o encadeamento como transitório. Ao fazer isso, poderemos serializar os objetos dessa classe, mas ainda teremos um problema. Como você pode ver no construtor definimos os valores min e max do nosso randomizer e depois disso iniciamos a thread que é responsável por gerar e imprimir o valor aleatório. Assim, ao restaurar o objeto persistido chamando o readObject() o construtor não será executado novamente, pois não há criação de um novo objeto. Nesse caso, precisamos desenvolver uma Serialização Personalizada fornecendo dois métodos dentro da classe. Esses métodos são:

private void writeObject(ObjectOutputStream out) throws IOException;
private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException;

Assim, adicionando nossa implementação no readObject(), podemos iniciar e iniciar nosso thread:

class RangeRandom implements Serializable, Runnable {

private int min;
private int max;

private transient Thread thread;
//transient should be any field that either cannot be serialized e.g Thread or any field you do not want serialized

public RangeRandom(int min, int max){
    this.min = min;
    this.max = max;
    thread = new Thread(this);
    thread.start();
}

@Override
public void run() {
    while(true) {
        Random rand = new Random();
        System.out.println("Thread: " + thread.getId() + " Random:" + rand.nextInt(max - min));
        try {
            Thread.sleep(10000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}

private void writeObject(ObjectOutputStream oos) throws IOException {
    oos.defaultWriteObject();
}

private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
    in.defaultReadObject();
    thread = new Thread(this);
    thread.start();
}
}

Controle de versão e serialVersionUID #

Quando você implementa a interface java.io.Serializable para tornar uma classe serializável, o compilador procura por um campo static final chamado serialVersionUID do tipo long. Se a classe não tiver este campo declarado explicitamente então o compilador irá criar um tal campo e atribuí-lo com um valor que vem de uma computação dependente de implementação de serialVersionUID. Esse cálculo depende de vários aspectos da classe e segue as Especificações de serialização de objetos fornecidas pela Sun. Mas não é garantido que o valor seja o mesmo em todas as implementações do compilador.

Este valor é usado para verificar a compatibilidade das classes com relação à serialização e isso é feito durante a desserialização de um objeto salvo. O Serialization Runtime verifica se serialVersionUID lido dos dados desserializados e o serialVersionUID declarado na classe são exatamente os mesmos. Se não for esse o caso, ele lança um InvalidClassException.

É altamente recomendável que você declare e inicialize explicitamente o campo final estático do tipo long e chamado ‘serialVersionUID’ em todas as suas classes que você deseja tornar Serializable em vez de confiar no cálculo padrão do valor para este campo, mesmo que você não vá usar versionamento. ** A computação ‘serialVersionUID’ é extremamente sensível e pode variar de uma implementação de compilador para outra e, portanto, você pode obter o ‘InvalidClassException’ mesmo para a mesma classe apenas porque você usou diferentes implementações de compilador nas extremidades do remetente e do receptor do processo de serialização.**

public class Example implements Serializable {          
    static final long serialVersionUID = 1L /*or some other value*/;
    //...
}

Contanto que serialVersionUID seja o mesmo, a Serialização Java pode lidar com diferentes versões de uma classe. Alterações compatíveis e incompatíveis são;

Alterações compatíveis #

  • Adicionando campos: Quando a classe que está sendo reconstituída possui um campo que não ocorre no fluxo, esse campo no objeto será inicializado com o valor padrão para seu tipo. Se a inicialização específica da classe for necessária, a classe pode fornecer um método readObject que pode inicializar o campo para valores não padrão.
  • Adicionando classes: O stream conterá a hierarquia de tipos de cada objeto no stream. A comparação dessa hierarquia no fluxo com a classe atual pode detectar classes adicionais. Como não há informações no fluxo para inicializar o objeto, os campos da classe serão inicializados com os valores padrão.
  • Removendo classes : comparar a hierarquia de classes no stream com a da classe atual pode detectar que uma classe foi excluída. Nesse caso, os campos e objetos correspondentes a essa classe são lidos do fluxo. Campos primitivos são descartados, mas os objetos referenciados pela classe excluída são criados, pois podem ser referenciados posteriormente no fluxo. Eles serão coletados como lixo quando o fluxo for coletado como lixo ou redefinido.
  • Adicionando métodos writeObject/readObject : Se a versão que está lendo o fluxo tiver esses métodos, espera-se que readObject, como de costume, leia os dados necessários gravados no fluxo pela serialização padrão. Ele deve chamar defaultReadObject antes de ler quaisquer dados opcionais. O método writeObject é esperado como de costume para chamar defaultWriteObject para gravar os dados necessários e, em seguida, pode gravar dados opcionais.
  • Adicionando java.io.Serializable : Isso é equivalente a adicionar tipos. Não haverá valores no fluxo para esta classe, portanto, seus campos serão inicializados com valores padrão. O suporte para subclasses de classes não serializáveis ​​requer que o supertipo da classe tenha um construtor no-arg e que a própria classe seja inicializada com valores padrão. Se o construtor no-arg não estiver disponível, a InvalidClassException será lançada.
  • Alterando o acesso a um campo: Os modificadores de acesso public, package, protected e private não têm efeito sobre a capacidade de serialização de atribuir valores aos campos.
  • Mudando um campo de estático para não estático ou transitório para não transitório : Ao contar com a serialização padrão para calcular os campos serializáveis, essa alteração é equivalente a adicionar um campo à classe. O novo campo será gravado no fluxo, mas as classes anteriores ignorarão o valor, pois a serialização não atribuirá valores a campos estáticos ou transitórios.

Alterações incompatíveis #

  • Excluindo campos: Se um campo for deletado em uma classe, o fluxo escrito não conterá seu valor. Quando o fluxo é lido por uma classe anterior, o valor do campo será definido como o valor padrão porque nenhum valor está disponível no fluxo. No entanto, esse valor padrão pode prejudicar a capacidade da versão anterior de cumprir seu contrato.
  • Movendo classes para cima ou para baixo na hierarquia: isso não pode ser permitido, pois os dados no fluxo aparecem na sequência errada.
  • Mudando um campo não estático para estático ou um campo não transitório para transitório: Ao contar com a serialização padrão, essa alteração equivale a excluir um campo da classe. Esta versão da classe não gravará esses dados no fluxo, portanto, não estará disponível para ser lido por versões anteriores da classe. Assim como ao excluir um campo, o campo da versão anterior será inicializado com o valor padrão, o que pode fazer com que a classe falhe de maneiras inesperadas.
  • Mudando o tipo declarado de um campo primitivo: Cada versão da classe grava os dados com seu tipo declarado. Versões anteriores da classe tentando ler o campo falharão porque o tipo dos dados no fluxo não corresponde ao tipo do campo.
  • Alterando o método writeObject ou readObject para que ele não escreva ou leia mais os dados do campo padrão ou altere-o para que ele tente escrevê-lo ou lê-lo quando a versão anterior não o fez. Os dados de campo padrão devem aparecer consistentemente ou não no fluxo.
  • Alterar uma classe de Serializable para Externalizable ou vice-versa é uma alteração incompatível, pois o fluxo conterá dados incompatíveis com a implementação da classe disponível.
  • Alterar uma classe de um tipo não-enum para um tipo enum ou vice-versa, pois o fluxo conterá dados incompatíveis com a implementação da classe disponível.
  • A remoção de Serializable ou Externalizable é uma alteração incompatível, pois, quando escrita, não fornecerá mais os campos necessários para versões mais antigas da classe.
  • Adicionar o método writeReplace ou readResolve a uma classe é incompatível se o comportamento produzir um objeto incompatível com qualquer versão mais antiga da classe.
