Em Java, uma anotação é uma forma de metadados sintáticos que podem ser adicionados ao código-fonte Java. Fornece dados sobre um programa que não faz parte do próprio programa. As anotações não têm efeito direto sobre a operação do código que anotam. Classes, métodos, variáveis, parâmetros e pacotes podem ser anotados.

Sintaxe #

  • @AnnotationName // ‘Anotação do marcador’ (sem parâmetros)
  • @AnnotationName(someValue) // define o parâmetro com o nome ‘value’
  • @AnnotationName(param1 = value1) // parâmetro nomeado
  • @AnnotationName(param1 = value1, param2 = value2) // vários parâmetros nomeados
  • @AnnotationName(param1 = {1, 2, 3}) // parâmetro de array nomeado
  • @AnnotationName({value1}) // array com elemento único como parâmetro com o nome ‘value’

tipos de parâmetros #

Apenas expressões constantes dos seguintes tipos são permitidas para parâmetros, bem como matrizes desses tipos:

  • String
  • Classe
  • tipos primitivos
  • Tipos de enumeração
  • Tipos de anotação

Definindo tipos de anotação #

Os tipos de anotação são definidos com @interface. Os parâmetros são definidos de forma semelhante aos métodos de uma interface regular.

@interface MyAnnotation {
    String param1();
    boolean param2();
    int[] param3();  // array parameter 
}

Valores padrão #

@interface MyAnnotation {
    String param1() default "someValue";
    boolean param2() default true;
    int[] param3() default {};
}

Meta-Anotações #

Meta-anotações são anotações que podem ser aplicadas a tipos de anotação. A meta-anotação especial predefinida define como os tipos de anotação podem ser usados.

@Alvo #

A meta-anotação @Target restringe os tipos aos quais a anotação pode ser aplicada.

@Target(ElementType.METHOD)
@interface MyAnnotation {
    // this annotation can only be applied to methods
}

Vários valores podem ser adicionados usando a notação de matriz, por exemplo. @Target({ElementType.FIELD, ElementType.TYPE})

@Retenção #

A meta-anotação @Retention define a visibilidade da anotação durante o processo de compilação ou execução de aplicativos. Por padrão, as anotações são incluídas em arquivos .class, mas não são visíveis em tempo de execução. Para tornar uma anotação acessível em tempo de execução, RetentionPolicy.RUNTIME deve ser definido nessa anotação.

@Retention(RetentionPolicy.RUNTIME)
@interface MyAnnotation {
    // this annotation can be accessed with reflections at runtime
}

Valores disponíveis #

Política de retenção Efeito
CLASSE A anotação está disponível no arquivo .class, mas não em tempo de execução
TEMPO DE EXECUÇÃO A anotação está disponível em tempo de execução e pode ser acessada via reflexão
FONTE A anotação está disponível em tempo de compilação, mas não é adicionada aos arquivos .class. A anotação pode ser usada e. por um processador de anotações.

@Documentado #

A meta-anotação @Documented é usada para marcar anotações cujo uso deve ser documentado por geradores de documentação da API como javadoc. Não tem valores. Com @Documented, todas as classes que usam a anotação irão listá-la em sua página de documentação gerada. Sem @Documented, não é possível ver quais classes usam a anotação na documentação.

@Herdado #

A meta-anotação @Inherited é relevante para anotações que são aplicadas a classes. Não tem valores. Marcar uma anotação como @Inherited altera a maneira como a consulta de anotações funciona.

  • Para uma anotação não herdada, a consulta examina apenas a classe que está sendo examinada.
  • Para uma anotação herdada, a consulta também verificará a cadeia de superclasses (recursivamente) até que uma instância da anotação seja encontrada.

Observe que apenas as superclasses são consultadas: quaisquer anotações anexadas a interfaces na hierarquia de classes serão ignoradas.

@Repetivel #

A meta-anotação @Repeatable foi adicionada no Java 8. Ela indica que várias instâncias da anotação podem ser anexadas ao destino da anotação. Esta meta-anotação não tem valores.

Verificações de anotação de tempo de execução via reflexão #

A API de reflexão do Java permite que o programador execute várias verificações e operações em campos de classe, métodos e anotações durante o tempo de execução. No entanto, para que uma anotação fique visível em tempo de execução, o RetentionPolicy deve ser alterado para RUNTIME, conforme demonstrado no exemplo abaixo:

@interface MyDefaultAnnotation {

}

@Retention(RetentionPolicy.RUNTIME)
@interface MyRuntimeVisibleAnnotation {

}

public class AnnotationAtRuntimeTest {

    @MyDefaultAnnotation
    static class RuntimeCheck1 {
    }
    
    @MyRuntimeVisibleAnnotation
    static class RuntimeCheck2 {
    }
    
    public static void main(String[] args) {
        Annotation[] annotationsByType = RuntimeCheck1.class.getAnnotations();
        Annotation[] annotationsByType2 = RuntimeCheck2.class.getAnnotations();

        System.out.println("default retention: " + Arrays.toString(annotationsByType));
        System.out.println("runtime retention: " + Arrays.toString(annotationsByType2));
    }
}

Anotações incorporadas #

A Standard Edition do Java vem com algumas anotações predefinidas. Você não precisa defini-los sozinho e pode usá-los imediatamente. Eles permitem que o compilador habilite algumas verificações fundamentais de métodos, classes e código.

@Sobrepor

Esta anotação se aplica a um método e diz que este método deve substituir o método de uma superclasse ou implementar uma definição de método de uma superclasse abstrata. Se esta anotação for usada com qualquer outro tipo de método, o compilador lançará um erro.

Superclasse de concreto

public class Vehicle {
   public void drive() {
        System.out.println("I am driving");
   }
}

class Car extends Vehicle {
    // Fine
    @Override
    public void drive() {
        System.out.prinln("Brrrm, brrm");
    }
}

Aula abstrata

abstract class Animal  {
   public abstract void makeNoise(); 
}

class Dog extends Animal {
    // Fine
    @Override
    public void makeNoise() {
        System.out.prinln("Woof");
    }
}

Não funciona

class Logger1 {
    public void log(String logString) {
        System.out.prinln(logString);
    }
}

class Logger2 {
    // This will throw compile-time error. Logger2 is not a subclass of Logger1. 
    // log method is not overriding anything
    @Override
    public void log(String logString) {
        System.out.println("Log 2" + logString);
    }
}

O objetivo principal é detectar erros de digitação, onde você acha que está substituindo um método, mas na verdade está definindo um novo.

class Vehicle {
   public void drive() {
        System.out.println("I am driving");
   }
}

class Car extends Vehicle {
    // Compiler error. "dirve" is not the correct method name to override.
    @Override
    public void dirve() {
        System.out.prinln("Brrrm, brrm");
    }
}

Observe que o significado de @Override mudou ao longo do tempo:

  • No Java 5, isso significava que o método anotado tinha que substituir um método não abstrato declarado na cadeia de superclasses.
  • A partir do Java 6 em diante, é também satisfeito se o método anotado implementa um método abstrato declarado na hierarquia de superclasse/interface de classes.

(Isso pode ocasionalmente causar problemas ao fazer a portabilidade de código para Java 5.)

@Descontinuada

Isso marca o método como obsoleto. Pode haver várias razões para isso:

  • a API tem falhas e é impraticável para corrigir,

  • o uso da API pode levar a erros,

  • a API foi substituída por outra API,

  • a API está obsoleta,

  • a API é experimental e está sujeita a alterações incompatíveis,

  • ou qualquer combinação dos itens acima.

O motivo específico da descontinuação geralmente pode ser encontrado na documentação da API.

A anotação fará com que o compilador emita um erro se você usá-la. IDEs também podem destacar esse método de alguma forma como obsoleto

class ComplexAlgorithm {
    @Deprecated
    public void oldSlowUnthreadSafeMethod() {
        // stuff here
    }
    
    public void quickThreadSafeMethod() {
        // client code should use this instead
    }
}

@Suprimir avisos

Em quase todos os casos, quando o compilador emite um aviso, a ação mais adequada é corrigir a causa. Em alguns casos (código genérico usando código pré-genérico sem tipo seguro, por exemplo) isso pode não ser possível e é melhor suprimir os avisos que você espera e não pode corrigir, para que você possa ver mais claramente os avisos inesperados.

Esta anotação pode ser aplicada a toda uma classe, método ou linha. Toma a categoria de aviso como parâmetro.

@SuppressWarnings("deprecation")
public class RiddledWithWarnings {
    // several methods calling deprecated code here
}

@SuppressWarning("finally")
public boolean checkData() {
    // method calling return from within finally block
}

É melhor limitar o escopo da anotação o máximo possível, para evitar que avisos inesperados também sejam suprimidos. Por exemplo, limitando o escopo da anotação a uma única linha:

ComplexAlgorithm algorithm = new ComplexAlgorithm();
@SuppressWarnings("deprecation") algoritm.slowUnthreadSafeMethod(); 
// we marked this method deprecated in an example above

@SuppressWarnings("unsafe") List<Integer> list = getUntypeSafeList(); 
// old library returns, non-generic List containing only integers

Os avisos suportados por esta anotação podem variar de compilador para compilador. Apenas os avisos unchecked e deprecation são especificamente mencionados no JLS. Os tipos de aviso não reconhecidos serão ignorados.

@SafeVarargs

Por causa do apagamento de tipo, void method(T... t) será convertido em void method(Object[] t) significando que o compilador nem sempre é capaz de verificar se o uso de varargs é de tipo seguro. Por exemplo:

private static <T> void generatesVarargsWarning(T... lists) {

Existem casos em que o uso é seguro, caso em que você pode anotar o método com a anotação SafeVarargs para suprimir o aviso. Isso obviamente oculta o aviso se o seu uso também for inseguro.

@Interface Funcional

Esta é uma anotação opcional usada para marcar uma FunctionalInterface. Isso fará com que o compilador reclame se não estiver em conformidade com a especificação FunctionalInterface (possui um único método abstrato)

@FunctionalInterface
public interface ITrade {
  public boolean check(Trade t);
}

@FunctionalInterface
public interface Predicate<T> {
  boolean test(T t);
}

A ideia por trás das anotações #

A Especificação da linguagem Java descreve as anotações da seguinte forma:

Uma anotação é um marcador que associa informações a uma construção de programa, mas não tem efeito em tempo de execução.

As anotações podem aparecer antes de tipos ou declarações. É possível que eles apareçam em um local onde possam se aplicar a um tipo ou a uma declaração. O que exatamente uma anotação se aplica é governado pela “meta-anotação” @Target. Consulte “Definindo tipos de anotação” para obter mais informações.

As anotações são usadas para uma infinidade de propósitos. Frameworks como Spring e Spring-MVC fazem uso de anotações para definir onde as dependências devem ser injetadas ou onde as solicitações devem ser roteadas.

Outros frameworks usam anotações para geração de código. Lombok e JPA são os principais exemplos, que usam anotações para gerar código Java (e SQL).

Este tópico tem como objetivo fornecer uma visão abrangente de:

  • Como definir suas próprias anotações?
  • Quais anotações a linguagem Java oferece?
  • Como as anotações são usadas na prática?

Processamento de tempo de compilação usando processador de anotação #

Este exemplo demonstra como fazer a verificação de tempo de compilação de um elemento anotado.

A anotação #

A anotação @Setter é um marcador que pode ser aplicado a métodos. A anotação será descartada durante a compilação e não estará disponível posteriormente.

package annotation;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

@Retention(RetentionPolicy.SOURCE)
@Target(ElementType.METHOD)
public @interface Setter {
}

O processador de anotações #

A classe SetterProcessor é usada pelo compilador para processar as anotações. Ele verifica se os métodos anotados com a anotação @Setter são métodos públicos, não estático com um nome começando com set e tendo uma letra maiúscula como 4ª letra. Se uma dessas condições não for atendida, um erro será gravado no Messager. O compilador grava isso no stderr, mas outras ferramentas podem usar essas informações de maneira diferente. Por exemplo. o NetBeans IDE permite que o usuário especifique processadores de anotação que são usados ​​para exibir mensagens de erro no editor.

package annotation.processor;

import annotation.Setter;
import java.util.Set;
import javax.annotation.processing.AbstractProcessor;
import javax.annotation.processing.Messager;
import javax.annotation.processing.ProcessingEnvironment;
import javax.annotation.processing.RoundEnvironment;
import javax.annotation.processing.SupportedAnnotationTypes;
import javax.annotation.processing.SupportedSourceVersion;
import javax.lang.model.SourceVersion;
import javax.lang.model.element.Element;
import javax.lang.model.element.ElementKind;
import javax.lang.model.element.ExecutableElement;
import javax.lang.model.element.Modifier;
import javax.lang.model.element.TypeElement;
import javax.tools.Diagnostic;

@SupportedAnnotationTypes({"annotation.Setter"})
@SupportedSourceVersion(SourceVersion.RELEASE_8)
public class SetterProcessor extends AbstractProcessor {

    private Messager messager;

    @Override
    public boolean process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv) {
        // get elements annotated with the @Setter annotation
        Set<? extends Element> annotatedElements = roundEnv.getElementsAnnotatedWith(Setter.class);

        for (Element element : annotatedElements) {
            if (element.getKind() == ElementKind.METHOD) {
                // only handle methods as targets
                checkMethod((ExecutableElement) element);
            }
        }

        // don't claim annotations to allow other processors to process them
        return false;
    }

    private void checkMethod(ExecutableElement method) {
        // check for valid name
        String name = method.getSimpleName().toString();
        if (!name.startsWith("set")) {
            printError(method, "setter name must start with \"set\"");
        } else if (name.length() == 3) {
            printError(method, "the method name must contain more than just \"set\"");
        } else if (Character.isLowerCase(name.charAt(3))) {
            if (method.getParameters().size() != 1) {
                printError(method, "character following \"set\" must be upper case");
            }
        }

        // check, if setter is public
        if (!method.getModifiers().contains(Modifier.PUBLIC)) {
            printError(method, "setter must be public");
        }

        // check, if method is static
        if (method.getModifiers().contains(Modifier.STATIC)) {
            printError(method, "setter must not be static");
        }
    }

    private void printError(Element element, String message) {
        messager.printMessage(Diagnostic.Kind.ERROR, message, element);
    }

    @Override
    public void init(ProcessingEnvironment processingEnvironment) {
        super.init(processingEnvironment);

        // get messager for printing errors
        messager = processingEnvironment.getMessager();
    }

}

Embalagem #

Para ser aplicado pelo compilador, o processador de anotação precisa ser disponibilizado ao SPI (consulte ServiceLoader).

Para fazer isso, um arquivo de texto META-INF/services/javax.annotation.processing.Processor precisa ser adicionado ao arquivo jar contendo o processador de anotação e a anotação, além dos outros arquivos. O arquivo precisa incluir o nome totalmente qualificado do processador de anotação, ou seja, deve ficar assim

annotation.processor.SetterProcessor

Vamos supor que o arquivo jar é chamado de AnnotationProcessor.jar abaixo.

Exemplo de classe anotada #

A classe a seguir é uma classe de exemplo no pacote padrão com as anotações sendo aplicadas aos elementos corretos de acordo com a política de retenção. No entanto, apenas o processador de anotação considera apenas o segundo método como um destino de anotação válido.

import annotation.Setter;

public class AnnotationProcessorTest {
    
    @Setter
    private void setValue(String value) {}

    @Setter
    public void setString(String value) {}
    
    @Setter
    public static void main(String[] args) {}
    
}

Usando o processador de anotações com javac #

Se o processador de anotação for descoberto usando o SPI, ele será usado automaticamente para processar elementos anotados. Por exemplo. compilando a classe AnnotationProcessorTest usando

javac -cp AnnotationProcessor.jar AnnotationProcessorTest.java

produz a seguinte saída

                 ^
AnnotationProcessorTest.java:12: error: setter name must start with "set"
    public static void main(String[] args) {}
                       ^
2 errors

em vez de compilar normalmente. Nenhum arquivo .class é criado.

Isso pode ser evitado especificando a opção -proc:none para javac. Você também pode renunciar à compilação usual especificando -proc:only.

Repetindo Anotações #

Até o Java 8, duas instâncias da mesma anotação não podiam ser aplicadas a um único elemento. A solução padrão era usar uma anotação de contêiner contendo uma matriz de alguma outra anotação:

// Author.java
@Retention(RetentionPolicy.RUNTIME)
public @interface Author {
    String value();
}

// Authors.java
@Retention(RetentionPolicy.RUNTIME)
public @interface Authors {
    Author[] value();
}

// Test.java
@Authors({
    @Author("Mary"),
    @Author("Sam")
})
public class Test {
    public static void main(String[] args) {
        Author[] authors = Test.class.getAnnotation(Authors.class).value();
        for (Author author : authors) {
            System.out.println(author.value());
            // Output:
            // Mary
            // Sam
        }
    }
}

O Java 8 fornece uma maneira mais limpa e transparente de usar anotações de contêiner, usando a anotação @Repeatable. Primeiro, adicionamos isso à classe Autor:

@Repeatable(Authors.class)

Isso diz ao Java para tratar várias anotações @Author como se estivessem cercadas pelo contêiner @Authors. Também podemos usar Class.getAnnotationsByType() para acessar o array @Author por sua própria classe, ao invés de através de seu container:

@Author("Mary")
@Author("Sam")
public class Test {
    public static void main(String[] args) {
        Author[] authors = Test.class.getAnnotationsByType(Author.class);
        for (Author author : authors) {
            System.out.println(author.value());
            // Output:
            // Mary
            // Sam
        }
    }
}

Anotações herdadas #

Por padrão, as anotações de classe não se aplicam a tipos que as estendem. Isso pode ser alterado adicionando a anotação @Inherited à definição da anotação

Exemplo #

Considere as 2 anotações a seguir:

@Inherited
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface InheritedAnnotationType {
}

e

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface UninheritedAnnotationType {
}

Se três classes forem anotadas assim:

@UninheritedAnnotationType
class A {
}

@InheritedAnnotationType
class B extends A {
}

class C extends B {
}

executando este código

System.out.println(new A().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println(new B().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println(new C().getClass().getAnnotation(InheritedAnnotationType.class));
System.out.println("_________________________________");
System.out.println(new A().getClass().getAnnotation(UninheritedAnnotationType.class));
System.out.println(new B().getClass().getAnnotation(UninheritedAnnotationType.class));
System.out.println(new C().getClass().getAnnotation(UninheritedAnnotationType.class));

imprimirá um resultado semelhante a este (dependendo dos pacotes da anotação):

null
@InheritedAnnotationType()
@InheritedAnnotationType()
_________________________________
@UninheritedAnnotationType()
null
null

Observe que as anotações só podem ser herdadas de classes, não de interfaces.

Obtendo valores de anotação em tempo de execução #

Você pode buscar as propriedades atuais da Anotação usando Reflexão para buscar o Método ou Campo ou Classe que tem uma Anotação aplicada a ele, e então buscar as propriedades desejadas.

@Retention(RetentionPolicy.RUNTIME)
@interface MyAnnotation {
    String key() default "foo";
    String value() default "bar";
}


class AnnotationExample {
    // Put the Annotation on the method, but leave the defaults
    @MyAnnotation
    public void testDefaults() throws Exception {
        // Using reflection, get the public method "testDefaults", which is this method with no args
        Method method = AnnotationExample.class.getMethod("testDefaults", null);

        // Fetch the Annotation that is of type MyAnnotation from the Method
        MyAnnotation annotation = (MyAnnotation)method.getAnnotation(MyAnnotation.class);

        // Print out the settings of the Annotation
        print(annotation);
    }

    //Put the Annotation on the method, but override the settings
    @MyAnnotation(key="baz", value="buzz")
    public void testValues() throws Exception {
        // Using reflection, get the public method "testValues", which is this method with no args
        Method method = AnnotationExample.class.getMethod("testValues", null);

        // Fetch the Annotation that is of type MyAnnotation from the Method
        MyAnnotation annotation = (MyAnnotation)method.getAnnotation(MyAnnotation.class);

        // Print out the settings of the Annotation
        print(annotation);
    }

    public void print(MyAnnotation annotation) {
        // Fetch the MyAnnotation 'key' & 'value' properties, and print them out 
        System.out.println(annotation.key() + " = " + annotation.value());
    }

    public static void main(String[] args) {
        AnnotationExample example = new AnnotationExample();
        try {
            example.testDefaults();
            example.testValues();
        } catch( Exception e ) {
            // Shouldn't throw any Exceptions
            System.err.println("Exception [" + e.getClass().getName() + "] - " + e.getMessage());
            e.printStackTrace(System.err);
        }
    }
}

A saída será

foo = bar
baz = buzz

Anotações para ’this’ e parâmetros do receptor #

Quando as anotações Java foram introduzidas pela primeira vez, não havia provisão para anotar o destino de um método de instância ou o parâmetro construtor oculto para um construtor de classes internas. Isso foi corrigido no Java 8 com a adição de declarações de parâmetro do receptor; veja JLS 8.4.1.

O parâmetro receiver é um dispositivo sintático opcional para um método de instância ou construtor de uma classe interna. Para um método de instância, o parâmetro receiver representa o objeto para o qual o método é invocado. Para o construtor de uma classe interna, o parâmetro receiver representa a instância imediatamente delimitadora do objeto recém-construído. De qualquer forma, o parâmetro receiver existe apenas para permitir que o tipo do objeto representado seja indicado no código-fonte, para que o tipo possa ser anotado. O parâmetro receiver não é um parâmetro formal; mais precisamente, não é uma declaração de qualquer tipo de variável (§4.12.3), ela nunca está vinculada a nenhum valor passado como um argumento em uma expressão de invocação de método ou expressão de criação de instância de classe qualificada, e não tem nenhum efeito em tempo de execução.

O exemplo a seguir ilustra a sintaxe para os dois tipos de parâmetro do receptor:

public class Outer {
    public class Inner {
        public Inner (Outer this) {
           // ...
        }
        public void doIt(Inner this) {
           // ...
        }
    }
}

O único propósito dos parâmetros do receptor é permitir que você adicione anotações. Por exemplo, você pode ter uma anotação personalizada @IsOpen cujo objetivo é declarar que um objeto Closeable não foi fechado quando um método é chamado. Por exemplo:

public class MyResource extends Closeable {
    public void update(@IsOpen MyResource this, int value) {
        // ...
    }

    public void close() {
        // ...
    }
}

Em um nível, a anotação @IsOpen em this poderia simplesmente servir como documentação. No entanto, poderíamos potencialmente fazer mais. Por exemplo:

  • Um processador de anotação pode inserir uma verificação em tempo de execução de que this não está no estado fechado quando update é chamado.
  • Um verificador de código pode realizar uma análise estática de código para encontrar casos em que this pode ser fechado quando update é chamado.

Adicione vários valores de anotação #

Um parâmetro de anotação pode aceitar vários valores se for definido como uma matriz. Por exemplo, a anotação padrão @SuppressWarnings é definida assim:

public @interface SuppressWarnings {
    String[] value();
}

O parâmetro value é um array de Strings. Você pode definir vários valores usando uma notação semelhante aos inicializadores de matriz:

@SuppressWarnings({"unused"})
@SuppressWarnings({"unused", "javadoc"})

Se você precisar definir apenas um único valor, os colchetes podem ser omitidos:

@SuppressWarnings("unused")
