Expressões em Java são a construção primária para fazer cálculos.

Para obter uma referência sobre os operadores que podem ser usados ​​em expressões, consulte Operators.

Operador precedente #

Quando uma expressão contém vários operadores, ela pode ser lida de maneiras diferentes. Por exemplo, a expressão matemática 1 + 2 x 3 pode ser lida de duas maneiras:

  1. Adicione 1 e 2 e multiplique o resultado por 3. Isso dá a resposta 9. Se adicionarmos parênteses, isso ficaria como ( 1 + 2 ) x 3.
  2. Adicione 1 ao resultado da multiplicação de 2 e 3. Isso dá a resposta 7. Se adicionarmos parênteses, isso ficaria como 1 + ( 2 x 3 ).

Em matemática, a convenção é ler a expressão da segunda maneira. A regra geral é que a multiplicação e a divisão são feitas antes da adição e subtração. Quando uma notação matemática mais avançada é usada, ou o significado é “auto-evidente” (para um matemático treinado!), ou parênteses são adicionados para desambiguar. Em ambos os casos, a eficácia da notação para transmitir significado depende da inteligência e do conhecimento compartilhado dos matemáticos.

Java tem as mesmas regras claras sobre como ler uma expressão, com base na precedência dos operadores usados.

Em geral, a cada operador é atribuído um valor de precedência; veja a tabela abaixo.

Por exemplo:

  1 + 2 * 3

A precedência de + é menor que a precedência de *, então o resultado da expressão é 7, não 9.

Ordem de avaliação de expressão #

As expressões Java são avaliadas seguindo as seguintes regras:

  • Os operandos são avaliados da esquerda para a direita.
  • Os operandos de um operador são avaliados antes do operador.
  • Os operadores são avaliados de acordo com a precedência do operador
  • As listas de argumentos são avaliadas da esquerda para a direita.

Exemplo Simples #

No exemplo a seguir:

int i = method1() + method2();

a ordem de avaliação é:

  1. O operando esquerdo do operador = é avaliado para o endereço de i.
  2. O operando esquerdo do operador + (method1()) é avaliado.
  3. O operando direito do operador + (method2()) é avaliado.
  4. A operação + é avaliada.
  5. A operação = é avaliada, atribuindo o resultado da adição a i.

Observe que se os efeitos das chamadas forem observáveis, você poderá observar que a chamada para method1 ocorre antes da chamada para method2.

Exemplo com um operador que tem um efeito colateral #

No exemplo a seguir:

int i = 1;
intArray[i] = ++i + 1;

a ordem de avaliação é:

  1. O operando esquerdo do operador = é avaliado. Isso fornece o endereço de intArray[1].
  2. O pré-incremento é avaliado. Isso adiciona 1 a i e resulta em 2.
  3. O operando à direita do + é avaliado.
  4. A operação + é avaliada como: 2 + 1 -> 3.
  5. A operação = é avaliada, atribuindo 3 a intArray[1].

Observe que, como o operando à esquerda do = é avaliado primeiro, ele não é influenciado pelo efeito colateral da subexpressão ++i.

Referência:

  • JLS 15.7 - Ordem de Avaliação

Noções básicas de expressão #

Expressões em Java são a construção primária para fazer cálculos. aqui estão alguns exemplos:

1                 // A simple literal is an expression
1 + 2             // A simple expression that adds two numbers
(i + j) / k       // An expression with multiple operations
(flag) ? c : d    // An expression using the "conditional" operator
(String) s        // A type-cast is an expression
obj.test()        // A method call is an expression
new Object()      // Creation of an object is an expression
new int[]         // Creation of an object is an expression

Em geral, uma expressão consiste nas seguintes formas:

  • Nomes de expressões que consistem em:
    • Simple identifiers; e.g. someIdentifier
    • Qualified identifiers; e.g. MyClass.someField
  • Primários que consistem em:
    • Literals; e.g. 1, 1.0, 'X', "hello", false and null
    • Class literal expressions; e.g. MyClass.class
    • this and <TypeName> . this
    • Parenthesized expressions; e.g. ( a + b )
    • Class instance creation expressions; e.g. new MyClass(1, 2, 3)
    • Array instance creation expressions; e.g. new int[3]
    • Field access expressions; e.g. obj.someField or this.someField
    • Array access expressions; e.g. vector[21]
    • Method invocations; e.g. obj.doIt(1, 2, 3)
    • Method references (Java 8 and later); e.g. MyClass::doIt
  • Expressões de operadores unários; por exemplo. !a ou i++
  • Expressões de operadores binários; por exemplo. a + b ou obj == null
  • Expressões de operadores ternários; por exemplo. (obj == null) ? 1: obj.getCount()
  • Expressões lambda (Java 8 e posterior); por exemplo. obj -> obj.getCount()

Os detalhes das diferentes formas de expressão podem ser encontrados em outros Tópicos.

  • O tópico Operadores abrange expressões de operadores unários, binários e ternários.
  • O tópico Expressões Lambda abrange expressões lambda e expressões de referência de método.
  • O tópico Classes and Objects abrange expressões de criação de instância de classe.
  • O tópico Arrays abrange expressões de acesso a array e expressões de criação de instância de array.
  • O tópico Literais cobre os diferentes tipos de expressões literais.

O tipo de expressão #

Na maioria dos casos, uma expressão tem um tipo estático que pode ser determinado em tempo de compilação examinando e suas subexpressões. Elas são chamadas de expressões independentes.

No entanto, (no Java 8 e posterior) os seguintes tipos de expressões podem ser expressões poli:

  • Expressões entre parênteses
  • Expressões de criação de instância de classe
  • Expressões de invocação de método
  • Expressões de referência de método
  • Expressões condicionais
  • Expressões lambda

Quando uma expressão é uma expressão poli, seu tipo pode ser influenciado pelo tipo de destino da expressão; ou seja, para que ele está sendo usado.

O valor de uma expressão #

O valor de uma expressão é uma atribuição compatível com seu tipo. A exceção a isso é quando ocorre poluição de heap; por exemplo. porque os avisos de “conversão não segura” foram (inapropriadamente) suprimidos ou ignorados.

Declarações de Expressão #

Ao contrário de muitas outras linguagens, Java geralmente não permite que expressões sejam usadas como instruções. Por exemplo:

public void compute(int i, int j) {
    i + j;   // ERROR
}

Como o resultado da avaliação de uma expressão como não pode ser usado, e como não pode afetar a execução do programa de nenhuma outra forma, os projetistas de Java assumiram a posição de que tal uso é um erro ou mal orientado.

No entanto, isso não se aplica a todas as expressões. Um subconjunto de expressões são (de fato) legais como declarações. O conjunto é composto por:

  • Expressão de atribuição, incluindo atribuições de operação e transformação.
  • Expressões pré e pós incremento e decremento.
  • Chamadas de métodos (void ou não-void).
  • Expressões de criação de instância de classe.

Expressões constantes #

Uma expressão constante é uma expressão que produz um tipo primitivo ou uma String e cujo valor pode ser avaliado em tempo de compilação para um literal. A expressão deve ser avaliada sem lançar uma exceção e deve ser composta apenas pelo seguinte:

  • Literais primitivos e String.

  • Conversão de tipos para tipos primitivos ou String.

  • Os seguintes operadores unários: +, -, ~ e !.

  • Os seguintes operadores binários: *, /, %, +, -, <<, >>, >>>, <, <= , >, >=, ==, !=, &, ^, |, && e ||.

  • O operador condicional ternário ? :.

  • Expressões constantes entre parênteses.

  • Nomes simples que se referem a variáveis ​​constantes. (Uma variável constante é uma variável declarada como ‘final’ onde a expressão inicializadora é ela mesma uma expressão constante.)

  • Nomes qualificados no formato <TypeName> . <Identifier> que se referem a variáveis ​​constantes.

Observe que a lista acima exclui ++ e --, os operadores de atribuição, class e instanceof, chamadas de métodos e referências a variáveis ​​ou campos gerais.

Expressões constantes do tipo String resultam em uma String “interna” e as operações de ponto flutuante em expressões constantes são avaliadas com semântica restrita a FP.

Usos para Expressões Constantes #

Expressões constantes podem ser usadas (quase) em qualquer lugar em que uma expressão normal possa ser usada. No entanto, eles têm um significado especial nos seguintes contextos.

Expressões constantes são necessárias para rótulos de caso em instruções switch. Por exemplo:

switch (someValue) {
case 1 + 1:            // OK
case Math.min(2, 3):   // Error - not a constant expression
    doSomething();
}

Quando a expressão no lado direito de uma atribuição é uma expressão constante, a atribuição pode executar uma conversão de restrição primitiva. Isso é permitido desde que o valor da expressão constante esteja dentro do intervalo do tipo no lado esquerdo. (Consulte JLS 5.1.3 e 5.2) Por exemplo:

byte b1 = 1 + 1;             // OK - primitive narrowing conversion.
byte b2 = 127 + 1;           // Error - out of range
byte b3 = b1 + 1;            // Error - not a constant expession
byte b4 = (byte) (b1 + 1);   // OK

Quando uma expressão constante é usada como condição em um do, while ou for, isso afeta a análise de legibilidade. Por exemplo:

while (false) {
    doSomething();           // Error - statenent not reachable
}
boolean flag = false;
while (flag) {
    doSomething();           // OK
}

(Observe que isso não se aplica a instruções if. O compilador Java permite que o bloco then ou else de uma instrução if seja inacessível. Este é o análogo Java da compilação condicional em C e C++.)

Finalmente, os campos static final em uma classe ou interface com inicializadores de expressão constante são inicializados avidamente. Assim, é garantido que essas constantes serão observadas no estado inicializado, mesmo quando houver um ciclo no grafo de dependência de inicialização da classe.

Para obter mais informações, consulte JLS 15.28. Expressões Constantes.
