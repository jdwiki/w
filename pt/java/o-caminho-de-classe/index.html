O classpath lista os locais onde o Java Runtime deve procurar classes e recursos. O classpath também é usado pelo compilador Java para localizar dependências externas e compiladas anteriormente.

Carregamento de classe Java

A JVM (Java Virtual Machine) carregará classes como e quando as classes forem necessárias (isso é chamado de carregamento lento). As localizações das classes a serem usadas são especificadas em três lugares: -

  1. Aqueles requeridos pela Plataforma Java são carregados primeiro, como aqueles na Biblioteca de Classes Java e suas dependências.
  2. As classes de extensão são carregadas em seguida (ou seja, aquelas em jre/lib/ext/)
  3. As classes definidas pelo usuário através do classpath são então carregadas

As classes são carregadas usando classes que são subtipos de java.lang.ClassLoader. Isso é descrito com mais detalhes neste tópico: https://www.wikiod.com/pt/java/carregadores-de-classe

Caminho de classe

O caminho de classe é um parâmetro usado pela JVM ou compilador que especifica os locais de classes e pacotes definidos pelo usuário. Isso pode ser definido na linha de comando como na maioria desses exemplos ou por meio de uma variável de ambiente (CLASSPATH)

Diferentes maneiras de especificar o caminho de classe #

Existem três maneiras de definir o caminho de classe.

  1. Pode ser definido usando a variável de ambiente CLASSPATH:

     set CLASSPATH=...         # Windows and csh
 export CLASSPATH=...      # Unix ksh/bash

  2. Pode ser definido na linha de comando da seguinte forma

     java -classpath ...
 javac -classpath ...

Observe que a opção -classpath (ou -cp) tem precedência sobre a variável de ambiente CLASSPATH.

  1. O caminho de classe para um arquivo JAR executável é especificado usando o elemento Class-Path em MANIFEST.MF:

     Class-Path: jar1-name jar2-name directory-name/jar3-name

Observe que isso só se aplica quando o arquivo JAR é executado assim:

    java -jar some.jar ...

Neste modo de execução, a opção -classpath e a variável de ambiente CLASSPATH serão ignoradas, mesmo que o arquivo JAR não possua o elemento Class-Path.

Se nenhum caminho de classe for especificado, o caminho de classe padrão será o arquivo JAR selecionado ao usar java -jar, ou o diretório atual caso contrário.

Relacionado:

  • https://docs.oracle.com/javase/tutorial/deployment/jar/downman.html
  • http://docs.oracle.com/javase/7/docs/technotes/tools/windows/classpath.html

Adicionando todos os JARs em um diretório ao classpath #

Se você deseja incluir todos os JARs no diretório no caminho de classe, você pode fazer isso de forma concisa usando a sintaxe curinga do caminho de classe; por exemplo:

 someFolder/*

Isso diz à JVM para adicionar todos os arquivos JAR e ZIP no diretório someFolder ao caminho de classe. Esta sintaxe pode ser usada em um argumento -cp, uma variável de ambiente CLASSPATH ou um atributo Class-Path no arquivo de manifesto de um arquivo JAR executável.Consulte [Definindo o caminho da classe: curingas do caminho da classe] 1 para exemplos e advertências.

Notas:

  1. Os curingas do caminho de classe foram introduzidos pela primeira vez no Java 6. As versões anteriores do Java não tratam “*” como um curinga.
  2. Você não pode colocar outros caracteres antes ou depois do “”; por exemplo. “someFolder/.jar” não é um curinga.
  3. Um curinga corresponde apenas a arquivos com o sufixo “.jar” ou “.JAR”. Os arquivos ZIP são ignorados, assim como os arquivos JAR com sufixos diferentes.
  4. Um curinga corresponde apenas a arquivos JAR no próprio diretório, não em seus subdiretórios.
  5. Quando um grupo de arquivos JAR é correspondido por uma entrada curinga, sua ordem relativa no caminho de classe não é especificada.

Carrega um recurso do classpath #

Pode ser útil carregar um recurso (imagem, arquivo de texto, propriedades, KeyStore, …) que está empacotado dentro de um JAR. Para isso, podemos usar as classes Class e ClassLoaders.

Suponha que temos a seguinte estrutura de projeto:

program.jar
|
\-com
  \-project
    |
    |-file.txt
    \-Test.class

E queremos acessar o conteúdo de file.txt da classe Test. Podemos fazer isso perguntando ao carregador de classe:

InputStream is = Test.class.getClassLoader().getResourceAsStream("com/project/file.txt");

Ao usar o carregador de classe, precisamos especificar o caminho totalmente qualificado do nosso recurso (cada pacote).

Ou, alternativamente, podemos perguntar diretamente ao objeto da classe Test

InputStream is = Test.class.getResourceAsStream("file.txt");

Usando o objeto de classe, o caminho é relativo à própria classe. Nosso Test.class estando no pacote com.project, o mesmo que file.txt, não precisamos especificar nenhum caminho.

Podemos, no entanto, usar caminhos absolutos do objeto de classe, assim:

 is = Test.class.getResourceAsStream("/com/project/file.txt");

Sintaxe do caminho do classpath #

O caminho de classe é uma sequência de entradas que são nomes de caminho de diretório, nomes de caminho de arquivo JAR ou ZIP ou especificações curinga JAR/ZIP.

  • Para um caminho de classe especificado na linha de comando (por exemplo, -classpath) ou como uma variável de ambiente, as entradas devem ser separadas por caracteres ; (ponto e vírgula) no Windows ou caracteres : (dois pontos) em outras plataformas ( Linux, UNIX, MacOSX e assim por diante).

  • Para o elemento Class-Path no MANIFEST.MF de um arquivo JAR, use um único espaço para separar as entradas.

Às vezes é necessário incorporar um espaço em uma entrada de caminho de classe

  • Quando o classpath é especificado na linha de comando, é simplesmente uma questão de usar a citação de shell apropriada. Por exemplo:

     export CLASSPATH="/home/user/My JAR Files/foo.jar:second.jar"

(Os detalhes podem depender do shell de comando que você usa.)

  • Quando o classpath é especificado em um arquivo JAR como um arquivo “MANIFEST.MF”, a codificação de URL deve ser usada.

      Class-Path: /home/user/My%20JAR%20Files/foo.jar second.jar

Caminho de classe dinâmico #

Às vezes, apenas adicionar todos os JARs de uma pasta não é suficiente, por exemplo, quando você tem código nativo e precisa selecionar um subconjunto de JARs. Neste caso, você precisa de dois métodos main(). O primeiro constrói um carregador de classe e então usa este carregador de classe para chamar o segundo main().

Aqui está um exemplo que seleciona o JAR nativo do SWT correto para sua plataforma, adiciona todos os JARs do seu aplicativo e, em seguida, invoca o método main() real: http://stackoverflow.com/questions/2706222/create-cross-platform- java-swt-application/3204032#3204032

Mapeando nomes de classe para nomes de caminho #

A cadeia de ferramentas Java padrão (e ferramentas de terceiros projetadas para interoperar com elas) possuem regras específicas para mapear os nomes das classes para os nomes de caminho dos arquivos e outros recursos que as representam.

Os mapeamentos são os seguintes

  • Para classes no pacote padrão, os nomes de caminho são nomes de arquivo simples.
  • Para classes em um pacote nomeado, os componentes do nome do pacote são mapeados para diretórios.
  • Para classes aninhadas e internas nomeadas, o componente filename é formado juntando os nomes das classes com um caractere $.
  • Para classes internas anônimas, os números são usados ​​no lugar dos nomes.

Isso é ilustrado na tabela a seguir:

Nome da classe Caminho de origem Caminho do arquivo de classe
AlgumClass SomeClass.java SomeClass.class
com.example.SomeClass com/example/SomeClass.java com/example/SomeClass.class
SomeClass.Inner (em SomeClass.java) SomeClass$Inner.class
SomeClass em classes internas (em SomeClass.java) SomeClass$1.class, SomeClass$2.class, etc

O que significa o classpath: como as pesquisas funcionam #

A finalidade do classpath é informar a uma JVM onde encontrar classes e outros recursos. O significado do classpath e o processo de busca estão interligados.

O classpath é uma forma de caminho de pesquisa que especifica uma sequência de locais para procurar recursos. Em um caminho de classe padrão, esses locais são um diretório no sistema de arquivos do host, um arquivo JAR ou um arquivo ZIP. Em cada caso, o local é a raiz de um namespace que será pesquisado.

O procedimento padrão para procurar uma classe no classpath é o seguinte:

  1. Mapeie o nome da classe para um nome de caminho de arquivo de classe relativo RP. O mapeamento de nomes de classe para nomes de arquivo de classe é descrito em outro lugar.

  2. Para cada entrada E no caminho de classe:

  • Se a entrada for um diretório do sistema de arquivos: - Resolve RP relative to E to give an absolute pathname AP. - Test if AP is a path for an existing file. - If yes, load the class from that file
  • Se a entrada for um arquivo JAR ou ZIP: - Lookup RP in the JAR / ZIP file index. - If the corresponding JAR / ZIP file entry exists, load the class from that entry.

O procedimento para procurar um recurso no caminho de classe depende se o caminho do recurso é absoluto ou relativo. Para um caminho de recurso absoluto, o procedimento é como acima. Para um caminho de recurso relativo resolvido usando Class.getResource ou Class.getResourceAsStream, o caminho para o pacote de classes é prefixado antes da pesquisa.

(Observe que estes são os procedimentos implementados pelos carregadores de classe Java padrão. Um carregador de classe personalizado pode realizar a pesquisa de forma diferente.)

O caminho de classe do bootstrap #

Os carregadores de classe Java normais procuram classes primeiro no caminho de classe de bootstrap, antes de verificar as extensões e o caminho de classe do aplicativo. Por padrão, o classpath de bootstrap consiste no arquivo “rt.jar” e alguns outros arquivos JAR importantes que são fornecidos pela instalação do JRE. Eles fornecem todas as classes na biblioteca de classes padrão do Java SE, juntamente com várias classes de implementação “internas”.

Em circunstâncias normais, você não precisa se preocupar com isso. Por padrão, comandos como java, javac e assim por diante usarão as versões apropriadas das bibliotecas de tempo de execução.

Muito ocasionalmente, é necessário substituir o comportamento normal do Java Runtime usando uma versão alternativa de uma classe nas bibliotecas padrão. Por exemplo, você pode encontrar um bug “show stopper” nas bibliotecas de tempo de execução que não pode ser contornado por meios normais. Em tal situação, é possível criar um arquivo JAR contendo a classe alterada e adicioná-lo ao classpath de bootstrap que inicia a JVM.

O comando java fornece as seguintes opções -X para modificar o classpath do bootstrap:

  • -Xbootclasspath:<path> substitui o classpath de inicialização atual pelo caminho fornecido.
  • -Xbootclasspath/a:<path> anexa o caminho fornecido ao caminho de classe de inicialização atual.
  • -Xbootclasspath/p:<path> precede o caminho fornecido para o caminho de classe de inicialização atual.

Observe que ao usar as opções bootclasspath para substituir ou substituir uma classe Java (etc.), você está modificando tecnicamente o Java. Pode haver implicações de licenciamento se você distribuir seu código. (Consulte os termos e condições da Licença Binária Java … e consulte um advogado.)
