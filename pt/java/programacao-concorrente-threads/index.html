A computação concorrente é uma forma de computação na qual vários cálculos são executados simultaneamente em vez de sequencialmente. A linguagem Java foi projetada para suportar programação simultânea por meio do uso de threads. Objetos e recursos podem ser acessados ​​por vários threads; cada encadeamento pode acessar potencialmente qualquer objeto no programa e o programador deve garantir que o acesso de leitura e gravação aos objetos seja sincronizado adequadamente entre os encadeamentos.

Tópicos relacionados no StackOverflow:

  • Tipos atômicos
  • Executor, ExecutorService e Thread pools
  • [Estender Thread versus implementar Runnable](https://www.wikiod.com/pt/java/armadilhas-do-java—threads-e-simultaneidade#Armadilha - Estendendo ‘java.lang.Thread’)

Exigível e Futuro #

Embora Runnable forneça um meio de encapsular o código a ser executado em um thread diferente, ele tem uma limitação, pois não pode retornar um resultado da execução. A única maneira de obter algum valor de retorno da execução de um Runnable é atribuir o resultado a uma variável acessível em um escopo fora do Runnable.

Callable foi introduzido no Java 5 como um par para Runnable. Callable é essencialmente o mesmo, exceto que tem um método call em vez de run. O método call tem a capacidade adicional de retornar um resultado e também tem permissão para lançar exceções verificadas.

O resultado de um Envio de tarefa que pode ser chamado está disponível para ser acessado por meio de um Futuro

Future pode ser considerado um tipo de contêiner que abriga o resultado da computação Callable. A computação do callable pode continuar em outra thread, e qualquer tentativa de tocar no resultado de um Future bloqueará e só retornará o resultado quando estiver disponível.

Interface de chamada

public interface Callable<V> {
    V call() throws Exception;
}

Futuro

interface Future<V> {
    V get();
    V get(long timeout, TimeUnit unit);
    boolean cancel(boolean mayInterruptIfRunning);
    boolean isCancelled();
    boolean isDone();
}

Usando o exemplo Callable e Future:

public static void main(String[] args) throws Exception {
    ExecutorService es = Executors.newSingleThreadExecutor();
          
    System.out.println("Time At Task Submission : " + new Date());
    Future<String> result = es.submit(new ComplexCalculator());
    // the call to Future.get() blocks until the result is available.So we are in for about a 10 sec wait now
    System.out.println("Result of Complex Calculation is : " + result.get());
    System.out.println("Time At the Point of Printing the Result : " + new Date());
}

Nosso Callable que faz um cálculo demorado

public class ComplexCalculator implements Callable<String> {

    @Override
    public String call() throws Exception {
        // just sleep for 10 secs to simulate a lengthy computation
        Thread.sleep(10000);            
        System.out.println("Result after a lengthy 10sec calculation");
        return "Complex Result"; // the result 
    }
}

Resultado

Time At Task Submission : Thu Aug 04 15:05:15 EDT 2016
Result after a lengthy 10sec calculation
Result of Complex Calculation is : Complex Result
Time At the Point of Printing the Result : Thu Aug 04 15:05:25 EDT 2016

Outras operações permitidas no Futuro

Enquanto get() é o método para extrair o resultado real Future tem provisão

  • get(long timeout, TimeUnit unit) define o período de tempo máximo durante o thread atual aguardará um resultado;
  • Para cancelar a tarefa chame cancel(mayInterruptIfRunning). A flag mayInterrupt indica que a tarefa deve ser interrompida se foi iniciada e está sendo executada agora;
  • Para verificar se a tarefa foi concluída/terminada chamando isDone();
  • Para verificar se a tarefa longa foi cancelada isCancelled().

Multithreading básico #

Se você tem muitas tarefas para executar, e todas essas tarefas não são dependentes do resultado das anteriores, você pode usar Multithreading para seu computador fazer todas essas tarefas ao mesmo tempo usando mais processadores se o seu computador puder . Isso pode tornar a execução do seu programa mais rápida se você tiver algumas grandes tarefas independentes.

class CountAndPrint implements Runnable {

    private final String name;

    CountAndPrint(String name) {
        this.name = name;
    }

    /** This is what a CountAndPrint will do */
    @Override
    public void run() {
        for (int i = 0; i < 10000; i++) {
            System.out.println(this.name + ": " + i);
        }
    }

    public static void main(String[] args) {
        // Launching 4 parallel threads
        for (int i = 1; i <= 4; i++) {
            // `start` method will call the `run` method 
            // of CountAndPrint in another thread
            new Thread(new CountAndPrint("Instance " + i)).start();
        }

        // Doing some others tasks in the main Thread
        for (int i = 0; i < 10000; i++) {
            System.out.println("Main: " + i);
        }
    }
}

Contagem Regressiva #

[Contagem Regressiva] 1

Um auxílio de sincronização que permite que um ou mais encadeamentos esperem até que um conjunto de operações sendo executadas em outros encadeamentos seja concluído.

  1. Um CountDownLatch é inicializado com uma determinada contagem.
  2. Os métodos await bloqueiam até que a contagem atual atinja zero devido a invocações do método countDown(), após o qual todas as threads em espera são liberadas e quaisquer invocações subsequentes de await retornam imediatamente.
  3. Este é um fenômeno único—a contagem não pode ser reiniciada. Se você precisar de uma versão que redefina a contagem, considere usar um CyclicBarrier.

Principais métodos:

public void await() throws InterruptedException

Faz com que o encadeamento atual espere até que a trava tenha uma contagem regressiva até zero, a menos que o encadeamento seja interrompido.

public void countDown()

Diminui a contagem do latch, liberando todas as threads em espera se a contagem chegar a zero.

Exemplo:

import java.util.concurrent.*;

class DoSomethingInAThread implements Runnable {
    CountDownLatch latch;
    public DoSomethingInAThread(CountDownLatch latch) {
        this.latch = latch;
    } 
    public void run() {
        try {
            System.out.println("Do some thing");
            latch.countDown();
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

public class CountDownLatchDemo {
    public static void main(String[] args) {
        try {
            int numberOfThreads = 5;
            if (args.length < 1) {
                System.out.println("Usage: java CountDownLatchDemo numberOfThreads");
                return;
            }
            try {
                numberOfThreads = Integer.parseInt(args[0]);
            } catch(NumberFormatException ne) {
            
            }
            CountDownLatch latch = new CountDownLatch(numberOfThreads);
            for (int n = 0; n < numberOfThreads; n++) {
                Thread t = new Thread(new DoSomethingInAThread(latch));
                t.start();
            }
            latch.await();
            System.out.println("In Main thread after completion of " + numberOfThreads + " threads");
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

resultado:

java CountDownLatchDemo 5
Do some thing
Do some thing
Do some thing
Do some thing
Do some thing
In Main thread after completion of 5 threads

Explicação:

  1. CountDownLatch é inicializado com um contador de 5 na thread principal
  2. A thread principal está esperando usando o método await().
  3. Cinco instâncias de DoSomethingInAThread foram criadas. Cada instância diminuiu o contador com o método countDown().
  4. Assim que o contador se tornar zero, o thread principal será retomado

Bloqueios como auxiliares de sincronização #

Antes da introdução do pacote concorrente do Java 5, o encadeamento era de nível mais baixo. A introdução deste pacote forneceu vários auxílios/construções de programação concorrente de nível superior.

Bloqueios são mecanismos de sincronização de threads que servem essencialmente ao mesmo propósito que blocos sincronizados ou palavras-chave.

Bloqueio Intrínseco

int count = 0; // shared among multiple threads

public void doSomething() {
    synchronized(this) {
        ++count; // a non-atomic operation
    }
}

Sincronização usando bloqueios

int count = 0; // shared among multiple threads

Lock lockObj = new ReentrantLock();
public void doSomething() {
    try {
        lockObj.lock();
        ++count; // a non-atomic operation
    } finally {    
        lockObj.unlock(); // sure to release the lock without fail
    }
}

Os bloqueios também têm funcionalidades disponíveis que o bloqueio intrínseco não oferece, como bloquear, mas permanecer responsivo à interrupção ou tentar bloquear e não bloquear quando não conseguir.

Bloqueio, responsivo à interrupção

class Locky {
    int count = 0; // shared among multiple threads

    Lock lockObj = new ReentrantLock();

    public void doSomething() {
        try {
            try {
                lockObj.lockInterruptibly();
                ++count; // a non-atomic operation
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // stopping
            }
        } finally {
            if (!Thread.currentThread().isInterrupted()) {
                lockObj.unlock(); // sure to release the lock without fail
            }
        }
    }
}

Só faça algo quando puder bloquear

public class Locky2 {
    int count = 0; // shared among multiple threads

    Lock lockObj = new ReentrantLock();

    public void doSomething() {
        boolean locked = lockObj.tryLock(); // returns true upon successful lock
        if (locked) {
            try {
                ++count; // a non-atomic operation
            } finally {
                lockObj.unlock(); // sure to release the lock without fail
            }
        }
    }
}

Sincronização #

Em Java, existe um mecanismo de bloqueio em nível de linguagem embutido: o bloco synchronized, que pode usar qualquer objeto Java como um bloqueio intrínseco (ou seja, todo objeto Java pode ter um monitor associado a ele).

Os bloqueios intrínsecos fornecem atomicidade a grupos de instruções. Para entender o que isso significa para nós, vamos dar uma olhada em um exemplo onde synchronized é útil:

private static int t = 0;
private static Object mutex = new Object();

public static void main(String[] args) {
    ExecutorService executorService = Executors.newFixedThreadPool(400); // The high thread count is for demonstration purposes.
    for (int i = 0; i < 100; i++) {
        executorService.execute(() -> {
            synchronized (mutex) {
                t++;
                System.out.println(MessageFormat.format("t: {0}", t));
            }
        });
    }
    executorService.shutdown();
}

Neste caso, se não fosse pelo bloco synchronized, haveria vários problemas de simultaneidade envolvidos. O primeiro seria com o operador pós-incremento (não é atômico em si), e o segundo seria que estaríamos observando o valor de t após uma quantidade arbitrária de outras threads terem a chance de modificá-lo. No entanto, como adquirimos um bloqueio intrínseco, não haverá condições de corrida aqui e a saída conterá números de 1 a 100 em sua ordem normal.

Bloqueios intrínsecos em Java são mutexes (ou seja, bloqueios de execução mútua). A execução mútua significa que, se um thread adquiriu o bloqueio, o segundo será forçado a esperar que o primeiro o libere antes que possa adquirir o bloqueio por si mesmo. Nota: Uma operação que pode colocar a thread no estado de espera (suspensão) é chamada de operação de bloqueio. Assim, adquirir um bloqueio é uma operação de bloqueio.

Bloqueios intrínsecos em Java são reentrantes. Isso significa que, se um thread tentar adquirir um bloqueio que já possui, ele não o bloqueará e o adquirirá com sucesso. Por exemplo, o código a seguir não bloqueará quando chamado:

public void bar(){
    synchronized(this){
        ...
    }
}
public void foo(){
    synchronized(this){
        bar();
    }
}

Além dos blocos synchronized, existem também métodos synchronized.

Os seguintes blocos de código são praticamente equivalentes (mesmo que o bytecode pareça ser diferente):

  1. Bloco synchronized em this:

    public void foo() {
    synchronized(this) {
        doStuff();
    }
}

  2. Método “sincronizado”:

     public synchronized void foo() {
     doStuff();
 }

Da mesma forma para métodos static, isto:

class MyClass {
    ...
    public static void bar() {
        synchronized(MyClass.class) {
            doSomeOtherStuff();
        }
    }
}

tem o mesmo efeito que este:

class MyClass {
    ...
    public static synchronized void bar() {
        doSomeOtherStuff();
    }
}

Criando um sistema básico de deadlock #

Um impasse ocorre quando duas ações concorrentes esperam que a outra termine e, portanto, nenhuma delas termina. Em java existe um bloqueio associado a cada objeto. Para evitar modificações simultâneas feitas por vários threads em um único objeto, podemos usar synchronized palavra-chave, mas tudo tem um custo. Usar a palavra-chave synchronized incorretamente pode levar a sistemas travados chamados de sistema em deadlock.

Considere que existem 2 threads trabalhando em 1 instância, Vamos chamar threads como First e Second, e digamos que temos 2 recursos R1 e R2. Primeiro adquire R1 e também precisa de R2 para sua conclusão enquanto o Segundo adquire R2 e precisa de R1 para conclusão.

digamos que no tempo t = 0,

O primeiro tem R1 e o segundo tem R2. agora o primeiro está esperando por R2 enquanto o segundo está esperando por R1. essa espera é indefinida e isso leva ao impasse.

public class Example2 {
    
    public static void main(String[] args) throws InterruptedException {
        final DeadLock dl = new DeadLock();
        Thread t1 = new Thread(new Runnable() {
    
            @Override
            public void run() {
                // TODO Auto-generated method stub
                dl.methodA();
            }
        });
   
        Thread t2 = new Thread(new Runnable() {
    
            @Override
            public void run() {
                // TODO Auto-generated method stub
                try {
                    dl.method2();
                } catch (InterruptedException e) {
                    // TODO Auto-generated catch block
                    e.printStackTrace();
                }
            }
        });
        t1.setName("First");
        t2.setName("Second");
        t1.start();
        t2.start();
    }
}

class DeadLock {
    
    Object mLock1 = new Object();
    Object mLock2 = new Object();
    

    public void methodA() {
        System.out.println("methodA wait for mLock1  " + Thread.currentThread().getName());
        synchronized (mLock1) {
            System.out.println("methodA mLock1 acquired   " + Thread.currentThread().getName());
            try {
                Thread.sleep(100);
                method2();
            } catch (InterruptedException e) {
                // TODO Auto-generated catch block
                e.printStackTrace();
            }
        }
    }
    public void method2() throws InterruptedException {
        System.out.println("method2 wait for mLock2  " + Thread.currentThread().getName());
        synchronized (mLock2) {
            System.out.println("method2  mLock2 acquired   " + Thread.currentThread().getName());
            Thread.sleep(100);
            method3();
        }
    }
    public void method3() throws InterruptedException {
        System.out.println("method3  mLock1  "+ Thread.currentThread().getName());
        synchronized (mLock1) {
            System.out.println("method3   mLock1 acquired  " + Thread.currentThread().getName());
        }
    }
}

Objeto executável #

A interface Runnable define um único método, run(), destinado a conter o código executado na thread.

O objeto Runnable é passado para o construtor Thread. E o método start() do Thread é chamado.

Exemplo

public class HelloRunnable implements Runnable {

    @Override
    public void run() {
        System.out.println("Hello from a thread");
    }

    public static void main(String[] args) {
        new Thread(new HelloRunnable()).start();
    }
}

Exemplo em Java8:

public static void main(String[] args) {
    Runnable r = () -> System.out.println("Hello world");
    new Thread(r).start();
}

Subclasse executável vs thread

Um emprego de objeto Runnable é mais geral, porque o objeto Runnable pode subclassificar uma classe diferente de Thread.

A subclasse Thread é mais fácil de usar em aplicações simples, mas é limitada pelo fato de que sua classe de tarefa deve ser descendente de Thread.

Um objeto Runnable é aplicável às APIs de gerenciamento de threads de alto nível.

Semáforo #

Um Semáforo é um sincronizador de alto nível que mantém um conjunto de permissões que podem ser adquiridas e liberadas por threads. Um Semáforo pode ser imaginado como um contador de permissões que será decrementado quando um thread adquirir e incrementado quando um thread for liberado. Se a quantidade de permissões for ‘0’ quando um encadeamento tentar adquirir, o encadeamento será bloqueado até que uma permissão seja disponibilizada (ou até que o encadeamento seja interrompido).

Um semáforo é inicializado como:

Semáforo semáforo = new Semáforo(1); // O valor int é o número de permissões

O construtor Semaphore aceita um parâmetro booleano adicional para imparcialidade. Quando definida como false, essa classe não oferece garantias sobre a ordem na qual os encadeamentos adquirem permissões. Quando a imparcialidade é definida como verdadeira, o semáforo garante que os encadeamentos que invocam qualquer um dos métodos de aquisição sejam selecionados para obter permissões na ordem em que a invocação desses métodos foi processada. É declarado da seguinte forma:

Semáforo semáforo = new Semáforo(1, true);

Agora vamos ver um exemplo de javadocs, onde Semáforo é usado para controlar o acesso a um conjunto de itens. Um semáforo é usado neste exemplo para fornecer funcionalidade de bloqueio para garantir que sempre haja itens a serem obtidos quando getItem() for chamado.

class Pool {

        if (markAsUnused(x))
            available.release();
    }

    }

Operações atômicas #

Criando uma instância java.lang.Thread #

Existem duas abordagens principais para criar uma thread em Java. Em essência, criar uma thread é tão fácil quanto escrever o código que será executado nela. As duas abordagens diferem em onde você define esse código.

Em Java, uma thread é representada por um objeto - uma instância de java.lang.Thread ou sua subclasse. Portanto, a primeira abordagem é criar essa subclasse e substituir o método run().

Observação: usarei Thread para fazer referência à classe java.lang.Thread e thread para fazer referência ao conceito lógico de threads.

class MyThread extends Thread {
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Thread running!");
        }
    }
}

Agora como já definimos o código a ser executado, a thread pode ser criada simplesmente como:

MyThread t = new MyThread();

A classe Thread também contém um construtor que aceita uma string, que será usada como nome da thread. Isso pode ser particularmente útil ao depurar um programa multi-thread.

class MyThread extends Thread {
    public MyThread(String name) {
        super(name);
    }
    
    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            System.out.println("Thread running! ");
        }
    }
}

MyThread t = new MyThread("Greeting Producer");

A segunda abordagem é definir o código usando java.lang.Runnable e seu único método run(). A classe Thread permite que você execute esse método em um thread separado. Para conseguir isso, crie o thread usando um construtor que aceita uma instância da interface Runnable.

Thread t = new Thread(aRunnable);

Isso pode ser muito poderoso quando combinado com referências lambdas ou métodos (somente Java 8):

Thread t = new Thread(operator::hardWork);

Você também pode especificar o nome do thread.

Thread t = new Thread(operator::hardWork, "Pi operator");

Praticamente falando, você pode usar as duas abordagens sem preocupações. No entanto, a sabedoria geral diz para usar o último.

Para cada um dos quatro construtores mencionados, há também uma alternativa que aceita uma instância de java.lang.ThreadGroup como o primeiro parâmetro.

ThreadGroup tg = new ThreadGroup("Operators");
Thread t = new Thread(tg, operator::hardWork, "PI operator");

O ThreadGroup representa um conjunto de threads. Você só pode adicionar um Thread a um ThreadGroup usando o construtor de um Thread. O ThreadGroup pode então ser usado para gerenciar todos os seus Threads juntos, assim como o Thread pode obter informações de seu ThreadGroup.

Então, para resumir, o Thread pode ser criado com um desses construtores públicos:

Thread()
Thread(String name)
Thread(Runnable target)
Thread(Runnable target, String name)
Thread(ThreadGroup group, String name)
Thread(ThreadGroup group, Runnable target)
Thread(ThreadGroup group, Runnable target, String name)
Thread(ThreadGroup group, Runnable target, String name, long stackSize)

A última permite definir o tamanho de pilha desejado para a nova thread.

Muitas vezes a legibilidade do código sofre ao criar e configurar muitos Threads com as mesmas propriedades ou do mesmo padrão. É quando java.util.concurrent.ThreadFactory pode ser usado. Esta interface permite encapsular o procedimento de criação do thread através do padrão de fábrica e seu único método newThread(Runnable).

class WorkerFactory implements ThreadFactory {
    private int id = 0;

    @Override
    public Thread newThread(Runnable r) {
        return new Thread(r, "Worker " + id++);
    }
}

Acesso exclusivo de gravação/leitura simultânea #

Às vezes, é necessário que um processo grave e leia simultaneamente os mesmos “dados”.

A interface ReadWriteLock e sua implementação ReentrantReadWriteLock permitem um padrão de acesso que pode ser descrito da seguinte forma:

  1. Pode haver qualquer número de leitores simultâneos dos dados. Se houver pelo menos um acesso de leitor concedido, nenhum acesso de gravador será possível.
  2. Pode haver no máximo um único gravador para os dados. Se houver um acesso de gravador concedido, nenhum leitor poderá acessar os dados.

Uma implementação poderia se parecer com:

import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;
public class Sample {

// Our lock. The constructor allows a "fairness" setting, which guarantees the chronology of lock attributions.
protected static final ReadWriteLock RW_LOCK = new ReentrantReadWriteLock();

// This is a typical data that needs to be protected for concurrent access
protected static int data = 0;

/** This will write to the data, in an exclusive access */
public static void writeToData() {
    RW_LOCK.writeLock().lock();
    try {
        data++;
    } finally {
        RW_LOCK.writeLock().unlock();
    }
}

public static int readData() {
    RW_LOCK.readLock().lock();
    try {
        return data;
    } finally {
        RW_LOCK.readLock().unlock();
    }
}

}

NOTA 1 : Este caso de uso preciso tem uma solução mais limpa usando AtomicInteger, mas o que é descrito aqui é um padrão de acesso, que funciona independentemente do fato de que os dados aqui são um inteiro que é uma variante Atomic.

NOTA 2: O bloqueio na parte de leitura é realmente necessário, embora possa não parecer assim para o leitor casual. De fato, se você não travar o lado do leitor, várias coisas podem dar errado, entre as quais:

  1. Não é garantido que as gravações de valores primitivos sejam atômicas em todas as JVMs, portanto, o leitor pode ver, por exemplo, apenas 32 bits de uma gravação de 64 bits se data fosse um tipo de 64 bits
  2. A visibilidade da escrita de um encadeamento que não a executou é garantida pela JVM somente se estabelecermos o relacionamento Happen Before entre as gravações e as leituras. Essa relação é estabelecida quando leitores e escritores usam seus respectivos bloqueios, mas não de outra forma.

Caso seja necessário maior desempenho, sob certos tipos de uso, existe um tipo de bloqueio mais rápido disponível, chamado StampedLock, que entre outras coisas implementa um modo de bloqueio otimista. Este bloqueio funciona de forma muito diferente do ReadWriteLock, e este exemplo não é transponível.

Produtor-Consumidor #

Um exemplo simples de solução de problema produtor-consumidor. Observe que as classes JDK (AtomicBoolean e BlockingQueue) são usadas para sincronização, o que reduz a chance de criar uma solução inválida. Consulte Javadoc para vários tipos de BlockingQueue; escolher uma implementação diferente pode alterar drasticamente o comportamento deste exemplo (como DelayQueue ou [Priority Queue] (https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/PriorityBlockingQueue.html)).

public class Producer implements Runnable {

    private final BlockingQueue<ProducedData> queue;

    public Producer(BlockingQueue<ProducedData> queue) {
        this.queue = queue;
    }

    public void run() {
        int producedCount = 0;
        try {
            while (true) {
                producedCount++;
                //put throws an InterruptedException when the thread is interrupted
                queue.put(new ProducedData());
            }
        } catch (InterruptedException e) {
            // the thread has been interrupted: cleanup and exit
            producedCount--;
            //re-interrupt the thread in case the interrupt flag is needeed higher up
            Thread.currentThread().interrupt();
        }
        System.out.println("Produced " + producedCount + " objects");
    }
}

public class Consumer implements Runnable {

    private final BlockingQueue<ProducedData> queue;

    public Consumer(BlockingQueue<ProducedData> queue) {
        this.queue = queue;
    }

    public void run() {
        int consumedCount = 0;
        try {
            while (true) {
                //put throws an InterruptedException when the thread is interrupted
                ProducedData data = queue.poll(10, TimeUnit.MILLISECONDS);
                // process data
                consumedCount++;
            }
        } catch (InterruptedException e) {
            // the thread has been interrupted: cleanup and exit
            consumedCount--;
            //re-interrupt the thread in case the interrupt flag is needeed higher up
            Thread.currentThread().interrupt();
        }
        System.out.println("Consumed " + consumedCount + " objects");
    }
}


public class ProducerConsumerExample {
    static class ProducedData {    
        // empty data object
    }

    public static void main(String[] args) throws InterruptedException {
        BlockingQueue<ProducedData> queue = new ArrayBlockingQueue<ProducedData>(1000);
        // choice of queue determines the actual behavior: see various BlockingQueue implementations

        Thread producer = new Thread(new Producer(queue));
        Thread consumer = new Thread(new Consumer(queue));

        producer.start();
        consumer.start();

        Thread.sleep(1000);
        producer.interrupt();
        Thread.sleep(10);
        consumer.interrupt();
    }
}

Usando ThreadLocal #

Uma ferramenta útil em Java Concurrency é ThreadLocal – isso permite que você tenha uma variável que será exclusiva para um determinado thread. Assim, se o mesmo código for executado em threads diferentes, essas execuções não compartilharão o valor, mas cada thread terá sua própria variável que é local para a thread.

Por exemplo, isso é frequentemente usado para estabelecer o contexto (como informações de autorização) de manipulação de um pedido em um servlet. Você pode fazer algo assim:

private static final ThreadLocal<MyUserContext> contexts = new ThreadLocal<>();

public static MyUserContext getContext() {
    return contexts.get(); // get returns the variable unique to this thread
}

public void doGet(...) {
    MyUserContext context = magicGetContextFromRequest(request); 
    contexts.put(context); // save that context to our thread-local - other threads
                           // making this call don't overwrite ours
    try {
        // business logic
    } finally {
        contexts.remove(); // 'ensure' removal of thread-local variable
    }
}

Agora, em vez de passar MyUserContext para cada método, você pode usar MyServlet.getContext() onde precisar. Agora, é claro, isso introduz uma variável que precisa ser documentada, mas é thread-safe, o que elimina muitas das desvantagens de usar uma variável com escopo tão alto.

A principal vantagem aqui é que cada thread tem sua própria variável local de thread nesse contêiner contexts. Contanto que você o use a partir de um ponto de entrada definido (como exigir que cada servlet mantenha seu contexto, ou talvez adicionando um filtro de servlet), você pode confiar que esse contexto estará lá quando você precisar.

Visualizando barreiras de leitura/gravação ao usar sincronizado/volátil #

Como sabemos, devemos usar a palavra-chave synchronized para tornar a execução de um método ou bloco exclusivo. Mas poucos de nós podem não estar cientes de mais um aspecto importante de usar as palavras-chave synchronized e volatile: além de tornar uma unidade de código atômica, ela também fornece barreira de leitura/gravação. O que é essa barreira de leitura/gravação? Vamos discutir isso usando um exemplo:

class Counter {

  private Integer count = 10;

  public synchronized void incrementCount() {
    count++;
  }

  public Integer getCount() {
    return count;
  }
}

Vamos supor que um thread A chame incrementCount() primeiro e depois outro thread B chame getCount(). Neste cenário não há garantia de que B verá o valor atualizado de count. Ele ainda pode ver count como 10, embora também seja possível que nunca veja o valor atualizado de count.

Para entender esse comportamento, precisamos entender como o modelo de memória Java se integra à arquitetura de hardware. Em Java, cada thread tem sua própria pilha de threads. Esta pilha contém: pilha de chamada de método e variável local criada nesse thread. Em um sistema com vários núcleos, é bem possível que dois threads estejam sendo executados simultaneamente em núcleos separados. Nesse cenário é possível que parte da pilha de uma thread esteja dentro do registrador/cache de um núcleo. Se dentro de uma thread, um objeto é acessado usando a palavra-chave synchronized (ou volatile), após o bloco synchronized essa thread sincroniza sua cópia local dessa variável com a memória principal. Isso cria uma barreira de leitura/gravação e garante que o thread esteja vendo o valor mais recente desse objeto.

Mas no nosso caso, como a thread B não usou o acesso sincronizado a count, ela pode estar se referindo ao valor de count armazenado no registro e pode nunca ver atualizações da thread A. Para garantir que B veja o valor mais recente de count, nós precisa fazer getCount() sincronizado também.

public synchronized Integer getCount() {
  return count;
}

Agora, quando a thread A terminar de atualizar o count, ela desbloqueia a instância Counter, ao mesmo tempo cria uma barreira de gravação e libera todas as alterações feitas dentro desse bloco para a memória principal. Da mesma forma, quando a thread B adquire o bloqueio na mesma instância de Counter, ela entra na barreira de leitura e lê o valor de count da memória principal e vê todas as atualizações.

O mesmo efeito de visibilidade vale para leitura/gravação volátil também. Todas as variáveis ​​atualizadas antes da escrita para volatile serão descarregadas na memória principal e todas as leituras após a leitura da variável volatile serão da memória principal.

Exemplo de vários produtores/consumidores com fila global compartilhada #

O código abaixo mostra vários programas de Produtor/Consumidor. Os encadeamentos Produtor e Consumidor compartilham a mesma fila global.

import java.util.concurrent.*;
import java.util.Random;

public class ProducerConsumerWithES {
    public static void main(String args[]) {
        BlockingQueue<Integer> sharedQueue = new LinkedBlockingQueue<Integer>();
         
        ExecutorService pes = Executors.newFixedThreadPool(2);
        ExecutorService ces = Executors.newFixedThreadPool(2);
          
        pes.submit(new Producer(sharedQueue, 1));
        pes.submit(new Producer(sharedQueue, 2));
        ces.submit(new Consumer(sharedQueue, 1));
        ces.submit(new Consumer(sharedQueue, 2));
         
        pes.shutdown();
        ces.shutdown();
    }
}

/* Different producers produces a stream of integers continuously to a shared queue, 
which is shared between all Producers and consumers */

class Producer implements Runnable {
    private final BlockingQueue<Integer> sharedQueue;
    private int threadNo;
    private Random random = new Random();
    public Producer(BlockingQueue<Integer> sharedQueue,int threadNo) {
        this.threadNo = threadNo;
        this.sharedQueue = sharedQueue;
    }
    @Override
    public void run() {
        // Producer produces a continuous stream of numbers for every 200 milli seconds
        while (true) {
            try {
                int number = random.nextInt(1000);
                System.out.println("Produced:" + number + ":by thread:"+ threadNo);
                sharedQueue.put(number);
                Thread.sleep(200);
            } catch (Exception err) {
                err.printStackTrace();
            }
        }
    }
}
/* Different consumers consume data from shared queue, which is shared by both producer and consumer threads */
class Consumer implements Runnable {
    private final BlockingQueue<Integer> sharedQueue;
    private int threadNo;
    public Consumer (BlockingQueue<Integer> sharedQueue,int threadNo) {
        this.sharedQueue = sharedQueue;
        this.threadNo = threadNo;
    }
    @Override
    public void run() {
        // Consumer consumes numbers generated from Producer threads continuously
        while(true){
            try {
                int num = sharedQueue.take();
                System.out.println("Consumed: "+ num + ":by thread:"+threadNo);
            } catch (Exception err) {
               err.printStackTrace();
            }
        }
    }   
}

resultado:

Produced:69:by thread:2
Produced:553:by thread:1
Consumed: 69:by thread:1
Consumed: 553:by thread:2
Produced:41:by thread:2
Produced:796:by thread:1
Consumed: 41:by thread:1
Consumed: 796:by thread:2
Produced:728:by thread:2
Consumed: 728:by thread:1

e assim por diante …………….

Explicação:

  1. O sharedQueue, que é um LinkedBlockingQueue, é compartilhado entre todas as threads Produtoras e Consumidoras.
  2. Os encadeamentos do produtor produzem um inteiro a cada 200 milissegundos continuamente e o anexam a sharedQueue
  3. O segmento Consumer consome inteiros de sharedQueue continuamente.
  4. Este programa é implementado sem construções explícitas synchronized ou Lock. BlockingQueue é a chave para alcançá-lo.

As implementações BlockingQueue são projetadas para serem usadas principalmente para filas produtor-consumidor.

As implementações BlockingQueue são thread-safe. Todos os métodos de enfileiramento atingem seus efeitos atomicamente usando travas internas ou outras formas de controle de concorrência.

Obtenha o status de todos os encadeamentos iniciados pelo seu programa, excluindo os encadeamentos do sistema #

Fragmento de código:

import java.util.Set;

public class ThreadStatus {
    public static void main(String args[]) throws Exception {
        for (int i = 0; i < 5; i++){
            Thread t = new Thread(new MyThread());
            t.setName("MyThread:" + i);
            t.start();
        }
        int threadCount = 0;
        Set<Thread> threadSet = Thread.getAllStackTraces().keySet();
        for (Thread t : threadSet) {
            if (t.getThreadGroup() == Thread.currentThread().getThreadGroup()) {
                System.out.println("Thread :" + t + ":" + "state:" + t.getState());
                ++threadCount;
            }
        }
        System.out.println("Thread count started by Main thread:" + threadCount);
    }
}

class MyThread implements Runnable {
    public void run() {
        try {
            Thread.sleep(2000);
        } catch(Exception err) {
            err.printStackTrace();
        }
    }
}

Resultado:

Explicação:

Thread.getAllStackTraces().keySet() retorna todos os Threads incluindo threads de aplicativos e threads do sistema. Se você estiver interessado apenas no status das Threads, iniciadas pelo seu aplicativo, itere o conjunto Thread marcando o Grupo de Threads de uma thread específica em relação à thread principal do programa.

Na ausência da condição ThreadGroup acima, o programa retorna o status abaixo de System Threads:

Signal Dispatcher
Attach Listener
Finalizer

Pausando a execução #

Thread.sleep faz com que o thread atual suspenda a execução por um período especificado. Esse é um meio eficiente de disponibilizar o tempo do processador para outros threads de um aplicativo ou outros aplicativos que possam estar sendo executados em um sistema de computador. Existem dois métodos sleep sobrecarregados na classe Thread.

Um que especifica o tempo de suspensão para o milissegundo

public static void sleep(long millis) throws InterruptedException

Um que especifica o tempo de suspensão para o nanossegundo

public static void sleep(long millis, int nanos)

Pausando a execução por 1 segundo

Thread.sleep(1000);

É importante notar que esta é uma dica para o agendador do kernel do sistema operacional. Isso pode não ser necessariamente preciso e algumas implementações nem consideram o parâmetro de nanossegundos (possivelmente arredondando para o milissegundo mais próximo).

É recomendado incluir uma chamada para Thread.sleep em try/catch e catch InterruptedException.

Adicione dois arrays int usando um Threadpool #

Um Threadpool possui uma Fila de tarefas, sendo que cada uma delas será executada em uma dessas Threads.

O exemplo a seguir mostra como adicionar dois arrays int usando um Threadpool.

int[] firstArray = { 2, 4, 6, 8 };
int[] secondArray = { 1, 3, 5, 7 };
int[] result = { 0, 0, 0, 0 };

ExecutorService pool = Executors.newCachedThreadPool(); 

// Setup the ThreadPool:
// for each element in the array, submit a worker to the pool that adds elements
for (int i = 0; i < result.length; i++) {
    final int worker = i;
    pool.submit(() -> result[worker] = firstArray[worker] + secondArray[worker] );
}

// Wait for all Workers to finish:
try {
    // execute all submitted tasks
    pool.shutdown();
    // waits until all workers finish, or the timeout ends
    pool.awaitTermination(12, TimeUnit.SECONDS);
}
catch (InterruptedException e) {
    pool.shutdownNow(); //kill thread
}

System.out.println(Arrays.toString(result));

Notas:

  1. Este exemplo é meramente ilustrativo. Na prática, não haverá aceleração usando threads para uma tarefa tão pequena. É provável que ocorra uma desaceleração, pois as despesas gerais de criação e agendamento de tarefas sobrecarregarão o tempo necessário para executar uma tarefa.

  2. Se você estivesse usando Java 7 e versões anteriores, usaria classes anônimas em vez de lambdas para implementar as tarefas.

Interrupção de Threads / Parando Threads #

Cada Java Thread tem um sinalizador de interrupção, que é inicialmente falso. Interromper um encadeamento é essencialmente nada mais do que definir esse sinalizador como verdadeiro. O código em execução nesse encadeamento pode verificar o sinalizador ocasionalmente e agir sobre ele. O código também pode ignorá-lo completamente. Mas por que cada Thread teria esse sinalizador? Afinal, ter um sinalizador booleano em um thread é algo que podemos nos organizar, se e quando precisarmos. Bem, existem métodos que se comportam de uma maneira especial quando o thread em que estão sendo executados é interrompido. Esses métodos são chamados de métodos de bloqueio. Esses são métodos que colocam a thread no estado WAITING ou TIMED_WAITING. Quando um encadeamento está nesse estado, interrompê-lo fará com que uma InterruptedException seja lançada no encadeamento interrompido, em vez de o sinalizador de interrupção ser definido como verdadeiro, e o encadeamento se tornará RUNNABLE novamente. O código que invoca um método de bloqueio é forçado a lidar com a InterruptedException, pois é uma exceção verificada. Assim, e daí o seu nome, uma interrupção pode ter o efeito de interromper um WAIT, encerrando-o efetivamente. Observe que nem todos os métodos que estão esperando de alguma forma (por exemplo, bloqueando IO) respondem à interrupção dessa maneira, pois não colocam o encadeamento em estado de espera. Por fim, um thread que tem seu sinalizador de interrupção definido, que entra em um método de bloqueio (ou seja, tenta entrar em um estado de espera), lançará imediatamente uma InterruptedException e o sinalizador de interrupção será limpo.

Além dessas mecânicas, Java não atribui nenhum significado semântico especial à interrupção. O código é livre para interpretar uma interrupção da maneira que quiser. Mas, na maioria das vezes, a interrupção é usada para sinalizar a um thread que ele deve parar de ser executado o mais rápido possível. Mas, como deve ficar claro acima, cabe ao código nesse encadeamento reagir adequadamente a essa interrupção para interromper a execução. Parar um thread é uma colaboração. Quando um thread é interrompido, seu código em execução pode estar em vários níveis no stacktrace. A maioria dos códigos não chama um método de bloqueio e termina em tempo hábil o suficiente para não atrasar indevidamente a interrupção do encadeamento. O código que deve se preocupar principalmente em responder à interrupção é o código que está em um loop manipulando tarefas até que não haja mais nenhum, ou até que um sinalizador seja definido sinalizando para interromper esse loop. Loops que lidam com tarefas possivelmente infinitas (ou seja, eles continuam em execução em princípio) devem verificar o sinalizador de interrupção para sair do loop. Para loops finitos, a semântica pode ditar que todas as tarefas devem ser concluídas antes de terminar, ou pode ser apropriado deixar algumas tarefas sem tratamento. O código que chama métodos de bloqueio será forçado a lidar com a InterruptedException. Se for semanticamente possível, ele pode simplesmente propagar a InterruptedException e declarar lançá-la. Como tal, ele se torna um método de bloqueio em relação a seus chamadores. Se ele não puder propagar a exceção, ele deve, no mínimo, definir o sinalizador interrompido, para que os chamadores mais acima na pilha também saibam que o encadeamento foi interrompido. Em alguns casos, o método precisa continuar esperando independentemente da InterruptedException, nesse caso ele deve atrasar a configuração do sinalizador interrompido até depois de terminar a espera, isso pode envolver a configuração de uma variável local, que deve ser verificada antes de sair do método para em seguida, interrompa seu encadeamento.

Exemplos :

Exemplo de código que para de processar tarefas após interrupção

class TaskHandler implements Runnable {
    
    private final BlockingQueue<Task> queue;

    TaskHandler(BlockingQueue<Task> queue) {
        this.queue = queue;
    }

    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) { // check for interrupt flag, exit loop when interrupted
            try {
                Task task = queue.take(); // blocking call, responsive to interruption
                handle(task);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // cannot throw InterruptedException (due to Runnable interface restriction) so indicating interruption by setting the flag
            }
        }
    }
    
    private void handle(Task task) {
        // actual handling
    }
}

Exemplo de código que atrasa a configuração do sinalizador de interrupção até a conclusão completa :

class MustFinishHandler implements Runnable {

    private final BlockingQueue<Task> queue;

    MustFinishHandler(BlockingQueue<Task> queue) {
        this.queue = queue;
    }

    @Override
    public void run() {
        boolean shouldInterrupt = false;
        
        while (true) {
            try {
                Task task = queue.take();
                if (task.isEndOfTasks()) {
                    if (shouldInterrupt) {
                        Thread.currentThread().interrupt();
                    }
                    return;
                }
                handle(task);
            } catch (InterruptedException e) {
                shouldInterrupt = true; // must finish, remember to set interrupt flag when we're done
            }
        }
    }

    private void handle(Task task) {
        // actual handling
    }
}

Exemplo de código que tem uma lista fixa de tarefas, mas pode encerrar cedo quando interrompido

class GetAsFarAsPossible implements Runnable {

    private final List<Task> tasks = new ArrayList<>();

    @Override
    public void run() {
        for (Task task : tasks) {
            if (Thread.currentThread().isInterrupted()) {
                return;
            }
            handle(task);
        }
    }

    private void handle(Task task) {
        // actual handling
    }
}
