Vários usos incorretos da linguagem de programação Java podem conduzir um programa para gerar resultados incorretos apesar de serem compilados corretamente. O objetivo principal deste tópico é listar armadilhas comuns relacionadas ao tratamento de exceções e propor a maneira correta de evitar tais armadilhas.

Armadilha - Capturando Throwable, Exception, Error ou RuntimeException #

Um padrão de pensamento comum para programadores Java inexperientes é que as exceções são “um problema” ou “um fardo” e a melhor maneira de lidar com isso é pegá-las todas1 o mais rápido possível. Isso leva a um código como este:

....
try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (Exception ex) {
    System.out.println("Could not open file " + fileName);
}

O código acima tem uma falha significativa. O catch na verdade vai pegar mais exceções do que o programador está esperando. Suponha que o valor de fileName seja null, devido a um bug em outro lugar no aplicativo. Isso fará com que o construtor FileInputStream lance um NullPointerException. O manipulador pegará isso e informará ao usuário:

    Could not open file null

o que é inútil e confuso. Pior ainda, suponha que foi o código “process the input” que lançou a exceção inesperada (marcada ou desmarcada!). Agora, o usuário receberá a mensagem enganosa de um problema que não ocorreu ao abrir o arquivo e pode não estar relacionado à E/S.

A raiz do problema é que o programador codificou um manipulador para Exception. Isso é quase sempre um erro:

  • Capturar Exception irá capturar todas as exceções marcadas, e a maioria das exceções não verificadas também.
  • Capturar RuntimeException capturará a maioria das exceções não verificadas.
  • Capturar Error capturará exceções não verificadas que sinalizam erros internos da JVM. Esses erros geralmente não são recuperáveis ​​e não devem ser detectados.
  • Capturar Throwable capturará todas as exceções possíveis.

O problema de capturar um conjunto muito amplo de exceções é que o manipulador normalmente não consegue lidar com todas elas adequadamente. No caso da Exception e assim por diante, é difícil para o programador prever o que poderia ser capturado; ou seja, o que esperar.

Em geral, a solução correta é lidar com as exceções que são lançadas. Por exemplo, você pode pegá-los e manipulá-los in situ:

try {
    InputStream is = new FileInputStream(fileName);
    // process the input
} catch (FileNotFoundException ex) {
    System.out.println("Could not open file " + fileName);
}

ou você pode declará-los como thrown pelo método anexo.

Existem muito poucas situações em que capturar ‘Exception’ é apropriado. O único que surge comumente é algo assim:

public static void main(String[] args) {
    try {
        // do stuff
    } catch (Exception ex) {
        System.err.println("Unfortunately an error has occurred. " +
                           "Please report this to X Y Z");
        // Write stacktrace to a log file.
        System.exit(1);
    }
}

Aqui nós realmente queremos lidar com todas as exceções, então pegar Exception (ou mesmo Throwable) está correto.

1 - Também conhecido como Pokemon Exception Handling.

Armadilha - Ignorando ou esmagando exceções #

Este exemplo é sobre ignorar ou “esmagar” exceções deliberadamente. Ou, para ser mais preciso, trata-se de como capturar e tratar uma exceção de uma maneira que a ignore. No entanto, antes de descrevermos como fazer isso, devemos primeiro salientar que esmagar exceções geralmente não é a maneira correta de lidar com elas.

As exceções geralmente são lançadas (por algo) para notificar outras partes do programa que ocorreu algum evento significativo (ou seja, “excepcional”). Geralmente (embora nem sempre) uma exceção significa que algo deu errado. Se você codificar seu programa para eliminar a exceção, há uma boa chance de que o problema reapareça de outra forma. Para piorar as coisas, quando você esmaga a exceção, está jogando fora as informações no objeto de exceção e seu rastreamento de pilha associado. Isso provavelmente tornará mais difícil descobrir qual era a fonte original do problema.

Na prática, o esmagamento de exceções ocorre frequentemente quando você usa o recurso de correção automática de um IDE para “corrigir” um erro de compilação causado por uma exceção não tratada. Por exemplo, você pode ver um código como este:

try {
    inputStream = new FileInputStream("someFile");
} catch (IOException e) {
    /* add exception handling code here */
}

Claramente, o programador aceitou a sugestão do IDE de eliminar o erro de compilação, mas a sugestão era inadequada. (Se o arquivo aberto falhou, o programa provavelmente deve fazer algo sobre isso. Com a “correção” acima, o programa pode falhar mais tarde; por exemplo, com um NullPointerException porque inputStream agora é null.)

Dito isto, aqui está um exemplo de deliberadamente esmagar uma exceção. (Para fins de argumento, suponha que determinamos que uma interrupção ao mostrar a selfie é inofensiva.) O comentário diz ao leitor que esmagamos a exceção deliberadamente e por que fizemos isso.

try {
    selfie.show();
} catch (InterruptedException e) {
    // It doesn't matter if showing the selfie is interrupted.
}

Outra maneira convencional de destacar que estamos deliberadamente esmagando uma exceção sem dizer por que é indicar isso com o nome da variável de exceção, assim:

try { 
    selfie.show(); 
} catch (InterruptedException ignored) {  }

Alguns IDEs (como o IntelliJ IDEA) não exibirão um aviso sobre o bloco catch vazio se o nome da variável estiver definido como ‘ignorado’.

Armadilha - Lançando Throwable, Exception, Error ou RuntimeException #

Embora capturar as exceções Throwable, Exception, Error e RuntimeException seja ruim, lançá-las é ainda pior.

O problema básico é que quando seu aplicativo precisa lidar com exceções, a presença das exceções de nível superior dificulta a discriminação entre diferentes condições de erro. Por exemplo

try {
    InputStream is = new FileInputStream(someFile);  // could throw IOException
    ...
    if (somethingBad) {
        throw new Exception();  // WRONG
    }
} catch (IOException ex) {
    System.err.println("cannot open ...");
} catch (Exception ex) {
    System.err.println("something bad happened");  // WRONG
}

O problema é que porque lançamos uma instância Exception, somos forçados a capturá-la. No entanto, conforme descrito em outro exemplo, capturar Exception é ruim. Nesta situação, torna-se difícil discriminar entre o caso “esperado” de uma Exception que é lançada se somethingBad for true, e o caso inesperado em que realmente capturamos uma exceção não verificada como NullPointerException.

Se a exceção de nível superior puder se propagar, encontraremos outros problemas:

  • Agora temos que lembrar de todas as diferentes razões pelas quais jogamos o nível superior e discriminá-las / lidar com elas.
  • No caso de Exception e Throwable também precisamos adicionar essas exceções à cláusula throws dos métodos se quisermos que a exceção se propague. Isso é problemático, conforme descrito abaixo.

Em suma, não lance essas exceções. Lance uma exceção mais específica que descreva mais de perto o “evento excepcional” que aconteceu. Se precisar, defina e use uma classe de exceção personalizada.

Declarar Throwable ou Exception nos “lançamentos” de um método é problemático.

É tentador substituir uma longa lista de exceções lançadas na cláusula throws de um método por Exception ou mesmo `Throwable. Esta é uma má ideia:

  1. Força o chamador a manipular (ou propagar) Exception.
  2. Não podemos mais confiar no compilador para nos informar sobre exceções verificadas específicas que precisam ser tratadas.
  3. É difícil lidar com ‘Exception’ corretamente. É difícil saber quais exceções reais podem ser capturadas e, se você não sabe o que pode ser capturado, é difícil saber qual estratégia de recuperação é apropriada.
  4. Manipular o Throwable é ainda mais difícil, já que agora você também tem que lidar com possíveis falhas das quais nunca devem ser recuperadas.

Este conselho significa que alguns outros padrões devem ser evitados. Por exemplo:

try {
    doSomething();
} catch (Exception ex) {
    report(ex);
    throw ex;
}

O acima tenta registrar todas as exceções à medida que elas passam, sem tratá-las definitivamente. Infelizmente, antes do Java 7, a instrução throw ex; fazia com que o compilador pensasse que qualquer Exception poderia ser lançada. Isso pode forçá-lo a declarar o método envolvente como throws Exception. A partir do Java 7 em diante, o compilador sabe que o conjunto de exceções que poderiam ser (relançadas) ali é menor.

Armadilha - Usando exceções para controle de fluxo normal #

Existe um mantra que alguns especialistas em Java costumam recitar:

“Exceções só devem ser usadas em casos excepcionais.”

(Por exemplo: http://programmers.stackexchange.com/questions/184654 )

A essência disso é que é uma má ideia (em Java) usar exceções e manipulação de exceções para implementar o controle de fluxo normal. Por exemplo, compare essas duas maneiras de lidar com um parâmetro que pode ser nulo.

public String truncateWordOrNull(String word, int maxLength) {
    if (word == null) {
        return "";
    } else {
        return word.substring(0, Math.min(word.length(), maxLength));
    }
}

public String truncateWordOrNull(String word, int maxLength) {
    try {
        return word.substring(0, Math.min(word.length(), maxLength));
    } catch (NullPointerException ex) {
        return "";
    }
}

Neste exemplo, estamos (por design) tratando o caso em que word é null como se fosse uma palavra vazia. As duas versões lidam com null usando o convencional if … else e ou try … catch. Como devemos decidir qual versão é melhor?

O primeiro critério é a legibilidade. Embora a legibilidade seja difícil de quantificar objetivamente, a maioria dos programadores concorda que o significado essencial da primeira versão é mais fácil de discernir. De fato, para realmente entender a segunda forma, você precisa entender que uma NullPointerException não pode ser lançada pelos métodos Math.min ou String.substring.

O segundo critério é a eficiência. Nas versões do Java anteriores ao Java 8, a segunda versão é significativamente (ordens de magnitude) mais lenta do que a primeira versão. Em particular, a construção de um objeto de exceção envolve a captura e gravação dos stackframes, caso o stacktrace seja necessário.

Por outro lado, existem muitas situações em que usar exceções é mais legível, mais eficiente e (às vezes) mais correto do que usar código condicional para lidar com eventos “excepcionais”. De fato, são raras as situações em que é necessário usá-los para eventos “não excepcionais”; ou seja, eventos que ocorrem com relativa frequência. Para este último, vale a pena procurar maneiras de reduzir as despesas gerais da criação de objetos de exceção.

Armadilha - Capturando a exceção interrompida #

Como já apontado em outras armadilhas, pegar todas as exceções usando

try {
    // Some code
} catch (Exception) {
    // Some error handling
}

Vem com muitos problemas diferentes. Mas um problema específico é que isso pode levar a deadlocks, pois quebra o sistema de interrupção ao escrever aplicativos multi-thread.

Se você iniciar um thread, geralmente também precisará interrompê-lo abruptamente por vários motivos.

Thread t = new Thread(new Runnable() {
    public void run() {
         while (true) {
             //Do something indefinetely
         }
    }
}

t.start();

//Do something else

// The thread should be canceld if it is still active. 
// A Better way to solve this is with a shared variable that is tested 
// regularily by the thread for a clean exit, but for this example we try to 
// forcibly interrupt this thread.
if (t.isAlive()) {
   t.interrupt();
   t.join();
}

//Continue with program

O t.interrupt() irá gerar uma InterruptedException nesse thread, que tem a intenção de encerrar o thread. Mas e se o Thread precisar limpar alguns recursos antes de parar completamente? Para isso, ele pode capturar o InterruptedException e fazer alguma limpeza.

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                //Do something indefinetely
            }
        } catch (InterruptedException ex) {
            //Do some quick cleanup

            // In this case a simple return would do. 
            // But if you are not 100% sure that the thread ends after 
            // catching the InterruptedException you will need to raise another 
            // one for the layers surrounding this code.                
            Thread.currentThread().interrupt(); 
        }
    }
}

Mas se você tiver uma expressão abrangente em seu código, a InterruptedException também será capturada por ela e a interrupção não continuará. O que neste caso pode levar a um impasse já que o thread pai espera indefinidamente que este thead pare com t.join().

 Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                }
                catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Dead code as the interrupt exception was already caught in
            // the inner try-catch           
            Thread.currentThread().interrupt(); 
        }
    }
}

Portanto, é melhor capturar Exceptions individualmente, mas se você insistir em usar um catch-all, pelo menos capture a InterruptedException individualmente antes.

Thread t = new Thread(new Runnable() {
    public void run() {
        try {
            while (true) {
                try {
                    //Do something indefinetely
                } catch (InterruptedException ex) {
                    throw ex; //Send it up in the chain
                } catch (Exception ex) {
                    ex.printStackTrace();
                }
            }
        } catch (InterruptedException ex) {
            // Some quick cleanup code 
    
            Thread.currentThread().interrupt(); 
        }
    }
}

Armadilha - Traços de pilha excessivos ou inadequados #

Uma das coisas mais irritantes que os programadores podem fazer é espalhar chamadas para printStackTrace() por todo o código.

O problema é que o printStackTrace() vai escrever o stacktrace na saída padrão.

  • Para um aplicativo destinado a usuários finais que não são programadores Java, um stacktrace é, na melhor das hipóteses, pouco informativo e, na pior, alarmante.

  • Para um aplicativo do lado do servidor, as chances são de que ninguém veja a saída padrão.

Uma idéia melhor é não chamar printStackTrace diretamente, ou se você chamar, faça isso de uma forma que o rastreamento de pilha seja gravado em um arquivo de log ou arquivo de erro em vez de no console do usuário final.

Uma maneira de fazer isso é usar uma estrutura de log e passar o objeto de exceção como parâmetro do evento de log. No entanto, mesmo registrar a exceção pode ser prejudicial se feito de forma imprudente. Considere o seguinte:

public void method1() throws SomeException {
    try {
        method2();
        // Do something
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method1", ex);
        throw ex;
    }
}

public void method2() throws SomeException {
    try {
        // Do something else
    } catch (SomeException ex) {
        Logger.getLogger().warn("Something bad in method2", ex);
        throw ex;
    }
}

Se a exceção for lançada em method2, você provavelmente verá duas cópias do mesmo stacktrace no arquivo de log, correspondendo à mesma falha.

Em resumo, registre a exceção ou jogue-a novamente (possivelmente embrulhada com outra exceção). Não faça os dois.

Armadilha - Subclassificando diretamente Throwable #

Throwable tem duas subclasses diretas, Exception e Error. Embora seja possível criar uma nova classe que estenda Throwable diretamente, isso é desaconselhável, pois muitos aplicativos assumem que existem apenas Exception e Error.

Mais ao ponto, não há benefício prático em subclassificar diretamente Throwable, pois a classe resultante é, na verdade, simplesmente uma exceção verificada. Subclassificar Exception resultará no mesmo comportamento, mas transmitirá mais claramente sua intenção.
