Vários usos incorretos da linguagem de programação Java podem conduzir um programa para gerar resultados incorretos apesar de serem compilados corretamente. O objetivo principal deste tópico é listar armadilhas comuns com suas causas, e propor a forma correta de evitar cair em tais problemas.

Este tópico trata de aspectos específicos da sintaxe da linguagem Java que são propensos a erros ou que não devem ser usados ​​de determinadas maneiras.

Armadilha - Falta uma ‘pausa’ em um caso de ‘interruptor’ #

Esses problemas de Java podem ser muito embaraçosos e, às vezes, permanecem desconhecidos até serem executados em produção. O comportamento de fallthrough em instruções switch geralmente é útil; no entanto, perder uma palavra-chave “break” quando tal comportamento não é desejado pode levar a resultados desastrosos. Se você esqueceu de colocar um “break” no “case 0” no exemplo de código abaixo, o programa escreverá “Zero” seguido de “One”, pois o fluxo de controle dentro daqui passará por toda a instrução “switch” até chega a uma “pausa”. Por exemplo:

public static void switchCasePrimer() {
        int caseIndex = 0;
        switch (caseIndex) {
            case 0:
                System.out.println("Zero");
            case 1:
                System.out.println("One");
                break;
            case 2:
                System.out.println("Two");
                break;
            default:
                System.out.println("Default");
        }
}

Na maioria dos casos, a solução mais limpa seria usar interfaces e mover código com comportamento específico em implementações separadas (composição sobre herança)

Se uma instrução switch for inevitável, é recomendável documentar falhas “esperadas” se ocorrerem. Dessa forma, você mostra aos colegas desenvolvedores que está ciente da pausa ausente e que esse é o comportamento esperado.

switch(caseIndex) {
    [...]
    case 2:
        System.out.println("Two");
        // fallthrough
    default:
        System.out.println("Default");

Armadilha - Deixando os suspensórios: os problemas do “se pendente” e do “outro pendente” #

A versão mais recente do guia de estilo Oracle Java exige que as instruções “then” e “else” em uma instrução if sejam sempre colocadas entre “colchetes” ou “colchetes”. Regras semelhantes se aplicam aos corpos de várias instruções de loop.

if (a) {           // <- open brace
    doSomething();
    doSomeMore();
}                  // <- close brace

Na verdade, isso não é exigido pela sintaxe da linguagem Java. De fato, se a parte “then” de uma instrução if for uma única instrução, é legal deixar de fora os colchetes

if (a)
    doSomething();

ou mesmo

if (a) doSomething();

No entanto, há perigos em ignorar as regras de estilo Java e deixar de fora as chaves. Especificamente, você aumenta significativamente o risco de que o código com recuo defeituoso seja mal interpretado.

O problema do “se pendente”:

Considere o código de exemplo acima, reescrito sem chaves.

if (a)
   doSomething();
   doSomeMore();

Este código parece dizer que as chamadas para doSomething e doSomeMore ocorrerão se e somente se a for true. Na verdade, o código está recuado incorretamente. A especificação da linguagem Java que a chamada doSomeMore() é uma instrução separada após a instrução if. A indentação correta é a seguinte:

if (a)
   doSomething();
doSomeMore();

O problema do “mais pendente”

Um segundo problema aparece quando adicionamos else à mistura. Considere o exemplo a seguir com chaves ausentes.

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
else
   doZ();

O código acima parece dizer que doZ será chamado quando a for false. Na verdade, o recuo está incorreto mais uma vez. A indentação correta para o código é:

if (a)
   if (b)
      doX();
   else if (c)
      doY(); 
   else
      doZ();

Se o código fosse escrito de acordo com as regras de estilo Java, na verdade ficaria assim:

if (a) {
   if (b) {
      doX();
   } else if (c) {
      doY(); 
   } else {
      doZ();
   }
}

Para ilustrar por que isso é melhor, suponha que você tenha acidentalmente digitado incorretamente o código. Você pode acabar com algo assim:

if (a) {                         if (a) {
   if (b) {                          if (b) {
      doX();                            doX();
   } else if (c) {                   } else if (c) {
      doY();                            doY();
} else {                         } else {
   doZ();                            doZ();
}                                    }
}                                }

Mas em ambos os casos, o código mal indentado “parece errado” aos olhos de um programador Java experiente.

Armadilha - Declarando classes com os mesmos nomes das classes padrão #

Às vezes, os programadores que são novos em Java cometem o erro de definir uma classe com um nome que é o mesmo de uma classe amplamente usada. Por exemplo:

package com.example;

/**
 * My string utilities
 */
public class String {
    ....
}

Então eles se perguntam por que recebem erros inesperados. Por exemplo:

package com.example;

public class Test {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

Se você compilar e tentar executar as classes acima, receberá um erro:

$ javac com/example/*.java
$ java com.example.Test
Error: Main method not found in class test.Test, please define the main method as:
   public static void main(String[] args)
or a JavaFX application class must extend javafx.application.Application

Alguém olhando para o código para a classe Test veria a declaração de main e olharia para sua assinatura e se perguntaria do que o comando java está reclamando. Mas, na verdade, o comando java está dizendo a verdade.

Quando declaramos uma versão de String no mesmo pacote que Test, esta versão tem precedência sobre a importação automática de java.lang.String. Assim, a assinatura do método Test.main é na verdade

void main(com.example.String[] args)

ao invés de

void main(java.lang.String[] args)

e o comando java não reconhecerá that como um método de ponto de entrada.

Lição: Não defina classes que tenham o mesmo nome de classes existentes em java.lang, ou outras classes comumente usadas na biblioteca Java SE. Se você fizer isso, estará se abrindo para todos os tipos de erros obscuros.

Armadilha - Ignorando a visibilidade do método #

Mesmo desenvolvedores Java experientes tendem a pensar que Java possui apenas três modificadores de proteção. A linguagem na verdade tem quatro! O nível de visibilidade do pacote privado (também conhecido como padrão) é muitas vezes esquecido.

Você deve prestar atenção em quais métodos você torna públicos. Os métodos públicos em um aplicativo são a API visível do aplicativo. Isso deve ser o mais pequeno e compacto possível, especialmente se você estiver escrevendo uma biblioteca reutilizável (consulte também o princípio SOLID). É importante considerar da mesma forma a visibilidade de todos os métodos e usar apenas acesso protegido ou privado de pacote quando apropriado.

Ao declarar métodos que devem ser privados como públicos, você expõe os detalhes de implementação interna da classe.

Um corolário disso é que você apenas testes unitários os métodos públicos de sua classe - na verdade, você pode apenas testar métodos públicos. É uma prática ruim aumentar a visibilidade de métodos privados apenas para poder executar testes de unidade nesses métodos. Testar métodos públicos que chamam os métodos com visibilidade mais restritiva deve ser suficiente para testar uma API inteira. Você deve nunca expandir sua API com mais métodos públicos apenas para permitir testes de unidade.

Armadilha - Literais octais #

Considere o seguinte trecho de código:

// Print the sum of the numbers 1 to 10
int count = 0;
for (int i = 1; i < 010; i++) {    // Mistake here ....
    count = count + i;
}
System.out.println("The sum of 1 to 10 is " + count);

Um iniciante em Java pode se surpreender ao saber que o programa acima imprime a resposta errada. Na verdade, ele imprime a soma dos números de 1 a 8.

A razão é que um literal inteiro que começa com o dígito zero (‘0’) é interpretado pelo compilador Java como um literal octal, não um literal decimal como você poderia esperar. Assim, 010 é o número octal 10, que é 8 em decimal.

Armadilha - Usando ‘==’ para testar um booleano #

Às vezes, um novo programador Java escreverá código como este:

public void check(boolean ok) {
    if (ok == true) {           // Note 'ok == true'
        System.out.println("It is OK");
    }
}

Um programador experiente identificaria isso como desajeitado e desejaria reescrevê-lo como:

public void check(boolean ok) {
    if (ok) {
       System.out.println("It is OK");
    }
}

No entanto, há mais errado com ok == true do que simples falta de jeito. Considere esta variação:

public void check(boolean ok) {
    if (ok = true) {           // Oooops!
        System.out.println("It is OK");
    }
}

Aqui o programador digitou errado == como = … e agora o código tem um bug sutil. A expressão x = true atribui incondicionalmente true a x e então é avaliada como true. Em outras palavras, o método check agora imprimirá “It is OK” não importa qual seja o parâmetro.

A lição aqui é perder o hábito de usar == false e ​​== true. Além de serem detalhados, eles tornam sua codificação mais propensa a erros.

Nota: Uma possível alternativa para ok == true que evita a armadilha é usar condições Yoda; ou seja, coloque o literal no lado esquerdo do operador relacional, como em true == ok. Isso funciona, mas a maioria dos programadores provavelmente concordaria que as condições do Yoda parecem estranhas. Certamente ok (ou !ok) é mais conciso e mais natural.

Pitfall - Pontos e vírgulas mal colocados e chaves ausentes #

Este é um erro que causa confusão real para iniciantes em Java, pelo menos na primeira vez que o fazem. Em vez de escrever isso:

if (feeling == HAPPY)
    System.out.println("Smile");
else
    System.out.println("Frown");

eles acidentalmente escrevem isso:

if (feeling == HAPPY);
    System.out.println("Smile");
else
    System.out.println("Frown");

e ficam confusos quando o compilador Java diz a eles que o else está fora do lugar. O compilador Java interpreta o acima da seguinte forma:

if (feeling == HAPPY)
    /*empty statement*/ ;
System.out.println("Smile");   // This is unconditional
else                           // This is misplaced.  A statement cannot
                               // start with 'else'
System.out.println("Frown");

Em outros casos, não haverá erros de compilação, mas o código não fará o que o programador pretende. Por exemplo:

for (int i = 0; i < 5; i++);
    System.out.println("Hello");

imprime “Olá” apenas uma vez. Mais uma vez, o ponto e vírgula falso significa que o corpo do loop for é uma instrução vazia. Isso significa que a chamada println que segue é incondicional.

Outra variação:

for (int i = 0; i < 5; i++);
    System.out.println("The number is " + i);

Isso dará um erro “Não é possível encontrar o símbolo” para i. A presença do ponto e vírgula falso significa que a chamada println está tentando usar i fora de seu escopo.

Nesses exemplos, há uma solução direta: simplesmente exclua o ponto e vírgula falso. No entanto, há algumas lições mais profundas a serem extraídas desses exemplos:

  1. O ponto e vírgula em Java não é “ruído sintático”. A presença ou ausência de um ponto e vírgula pode alterar o significado do seu programa. Não basta adicioná-los no final de cada linha.

  2. Não confie na indentação do seu código. Na linguagem Java, o espaço em branco extra no início de uma linha é ignorado pelo compilador.

  3. Use um indentador automático. Todos os IDEs e muitos editores de texto simples entendem como recuar corretamente o código Java.

  4. Esta é a lição mais importante. Siga as diretrizes de estilo Java mais recentes e coloque chaves em torno das instruções “then” e “else” e da instrução do corpo de um loop. A chave aberta ({) não deve estar em uma nova linha.

Se o programador seguiu as regras de estilo, o exemplo if com um ponto e vírgula mal colocado ficaria assim:

if (feeling == HAPPY); {
    System.out.println("Smile");
} else {
    System.out.println("Frown");
}

Isso parece estranho para um olho experiente. Se você recuasse automaticamente esse código, provavelmente ficaria assim:

if (feeling == HAPPY); {
                           System.out.println("Smile");
                       } else {
                           System.out.println("Frown");
                       }

que deve se destacar como errado até mesmo para um iniciante.

Armadilha - Importações curinga podem tornar seu código frágil #

Considere o seguinte exemplo parcial:

import com.example.somelib.*;
import com.acme.otherlib.*;

public class Test {
    private Context x = new Context();   // from com.example.somelib
    ...
}

Suponha que quando você desenvolveu o código pela primeira vez contra a versão 1.0 de somelib e a versão 1.0 de otherlib. Então, em algum momento posterior, você precisa atualizar suas dependências para versões posteriores e decide usar otherlib versão 2.0. Suponha também que uma das mudanças que eles fizeram no otherlib entre 1.0 e 2.0 foi adicionar uma classe Context.

Agora, quando você recompilar Test, você receberá um erro de compilação informando que Context é uma importação ambígua.

Se você estiver familiarizado com a base de código, isso provavelmente é apenas um pequeno inconveniente. Se não, então você tem algum trabalho a fazer para resolver esse problema, aqui e potencialmente em outros lugares.

O problema aqui são as importações curinga. Por um lado, usar curingas pode tornar suas classes algumas linhas mais curtas. Por outro lado:

  • Alterações compatíveis para cima em outras partes de sua base de código, em bibliotecas padrão Java ou em bibliotecas de terceiros podem levar a erros de compilação.

  • A legibilidade sofre. A menos que você esteja usando um IDE, descobrir qual das importações curinga está puxando uma classe nomeada pode ser difícil.

A lição é que é uma má ideia usar importações curinga em código que precisa ser duradouro. Importações específicas (não curinga) não exigem muito esforço para manter se você usar um IDE, e o esforço vale a pena.

Armadilha: Usando ‘assert’ para argumento ou validação de entrada do usuário #

Uma questão que ocasionalmente no StackOverflow é se é apropriado usar assert para validar argumentos fornecidos a um método, ou mesmo entradas fornecidas pelo usuário.

A resposta simples é que não é apropriado.

As melhores alternativas incluem:

  • Lançando um IllegalArgumentException usando código personalizado.
  • Usando os métodos Preconditions disponíveis na biblioteca Google Guava.
  • Usando os métodos Validate disponíveis na biblioteca Apache Commons Lang3.

Isto é o que a Java Language Specification (JLS 14.10, for Java 8) aconselha sobre este assunto:

Normalmente, a verificação de asserção é habilitada durante o desenvolvimento e teste do programa e desabilitada para implantação, para melhorar o desempenho.

Como as asserções podem ser desabilitadas, os programas não devem assumir que as expressões contidas nas asserções serão avaliadas. Assim, essas expressões booleanas geralmente devem estar livres de efeitos colaterais. A avaliação dessa expressão booleana não deve afetar nenhum estado visível após a conclusão da avaliação. Não é ilegal que uma expressão booleana contida em uma declaração tenha um efeito colateral, mas geralmente é inapropriado, pois pode fazer com que o comportamento do programa varie dependendo se as declarações foram habilitadas ou desabilitadas.

Diante disso, asserções não devem ser usadas para verificação de argumentos em métodos públicos. A verificação de argumentos normalmente faz parte do contrato de um método e esse contrato deve ser mantido independentemente de as asserções estarem habilitadas ou desabilitadas.

Um problema secundário com o uso de asserções para verificação de argumentos é que argumentos errôneos devem resultar em uma exceção de tempo de execução apropriada (como IllegalArgumentException, ArrayIndexOutOfBoundsException ou NullPointerException). Uma falha de asserção não lançará uma exceção apropriada. Novamente, não é ilegal usar asserções para verificação de argumentos em métodos públicos, mas geralmente é inapropriado. Pretende-se que AssertionError nunca seja capturado, mas é possível fazê-lo, portanto, as regras para instruções try devem tratar asserções que aparecem em um bloco try de forma semelhante ao tratamento atual de instruções throw.

Armadilha - Sobrecarregar em vez de substituir #

Considere o seguinte exemplo:

public final class Person {
    private final String firstName;
    private final String lastName;
   
    public Person(String firstName, String lastName) {
        this.firstName = (firstName == null) ? "" : firstName;
        this.lastName = (lastName == null) ? "" : lastName;
    }

    public boolean equals(String other) {
        if (!(other instanceof Person)) {
            return false;
        }
        Person p = (Person) other;
        return firstName.equals(p.firstName) &&
                lastName.equals(p.lastName);
    }

    public int hashcode() {
        return firstName.hashCode() + 31 * lastName.hashCode();
    }
}

Este código não vai se comportar como esperado. O problema é que os métodos equals e hashcode para Person não substituem os métodos padrão definidos por Object.

  • O método equals tem a assinatura errada. Deve ser declarado como equals(Object) e não como equals(String).
  • O método hashcode tem o nome errado. Deve ser hashCode() (observe a letra maiúscula C).

Esses erros significam que declaramos sobrecargas acidentais, e elas não serão usadas se Person for usado em um contexto polimórfico.

No entanto, existe uma maneira simples de lidar com isso (a partir do Java 5). Use a anotação @Override sempre que você pretende que seu método seja uma substituição:

public final class Person {
    ...

    @Override
    public boolean equals(String other) {
        ....
    }

    @Override
    public hashcode() {
        ....
    }
}

Quando adicionamos uma anotação @Override a uma declaração de método, o compilador verificará se o método não substitui (ou implementa) um método declarado em uma superclasse ou interface. Portanto, no exemplo acima, o compilador nos dará dois erros de compilação, o que deve ser suficiente para nos alertar sobre o erro.

Armadilha do Auto-Unboxing de Objetos Nulos em Primitivos #

public class Foobar {
    public static void main(String[] args) {

        // example: 
        Boolean ignore = null;
        if (ignore == false) {
            System.out.println("Do not ignore!");
        }
    }
}

A armadilha aqui é que null é comparado a false. Já que estamos comparando um boolean primitivo com um Boolean, Java tenta desempacotar o Boolean Object em um equivalente primitivo, pronto para comparação. No entanto, como esse valor é null, uma NullPointerException é lançada.

Java é incapaz de comparar tipos primitivos com valores null, o que causa uma NullPointerException em tempo de execução. Considere o caso primitivo da condição false == null; isso geraria um erro de tempo de compilação tipos incomparáveis: int e <null>.
