A documentação para código java geralmente é gerada usando javadoc. O Javadoc foi criado pela Sun Microsystems com o propósito de gerar a documentação da API no formato HTML a partir do código-fonte Java. O uso do formato HTML oferece a conveniência de poder vincular documentos relacionados.

Sintaxe #

  • /** – início do JavaDoc em uma classe, campo, método ou pacote
  • @author // Para nomear o autor da classe, interface ou enum. É necessário.
  • @version // A versão dessa classe, interface ou enum. É necessário. Você pode usar macros como %I% ou %G% para seu software de controle de origem para preencher no checkout.
  • @param // Para mostrar os argumentos (parâmetros) de um método ou construtor. Especifique uma tag @param para cada parâmetro.
  • @return // Para mostrar os tipos de retorno para métodos não void.
  • @exception // Mostra quais exceções podem ser lançadas do método ou construtor. Exceções que ** DEVEM** ser detectadas devem ser listadas aqui. Se desejar, você também pode incluir aqueles que não precisam ser capturados, como ArrayIndexOutOfBoundsException. Especifique uma @exception para cada exceção que pode ser lançada.
  • @throws // Igual a @exception.
  • @see // Links para um método, campo, classe ou pacote. Use na forma de package.Class#something.
  • @since // Quando este método, campo ou classe foi adicionado. Por exemplo, JDK-8 para uma classe como java.util.Optional<T>.
  • @serial, @serialField, @serialData // Usado para mostrar o serialVersionUID.
  • @deprecated // Para marcar uma classe, método ou campo como obsoleto. Por exemplo, um seria java.io.StringBufferInputStream. Veja uma lista completa de classes obsoletas existentes aqui.
  • {@link} // Semelhante a @see, mas pode ser usado com texto personalizado: {@link #setDefaultCloseOperation(int closeOperation) consulte JFrame#setDefaultCloseOperation para obter mais informações}.
  • {@linkplain} // Semelhante a {@link}, mas sem a fonte do código.
  • {@code} // Para código literal, como tags HTML. Por exemplo: {@code <html></html>}. No entanto, isso usará uma fonte monoespaçada. Para obter o mesmo resultado sem a fonte monoespaçada, use {@literal}.
  • {@literal} // Igual a {@code}, mas sem a fonte monoespaçada.
  • {@value} // Mostra o valor de um campo estático: o valor de JFrame#EXIT_ON_CLOSE é {@value}. Ou você pode vincular a um determinado campo: usa o nome do aplicativo {@value AppConstants#APP_NAME}.
  • {@docRoot} // A pasta raiz do HTML JavaDoc em relação ao arquivo atual. Exemplo: <a href="{@docRoot}/credits.html">Créditos</a>.
  • HTML é permitido: <code>“Olá cookies”.substring(3)</code>.
  • */ – fim da declaração JavaDoc

Javadoc é uma ferramenta incluída no JDK que permite que comentários no código sejam convertidos em uma documentação HTML. A Especificação da API Java foi gerada usando Javadoc. O mesmo vale para grande parte da documentação de bibliotecas de terceiros.

Documentação da aula #

Todos os comentários Javadoc começam com um comentário de bloco seguido por um asterisco (/**) e terminam quando o comentário de bloco termina (*/). Opcionalmente, cada linha pode começar com um espaço em branco arbitrário e um único asterisco; estes são ignorados quando os arquivos de documentação são gerados.

/**
 * Brief summary of this class, ending with a period.
 *
 * It is common to leave a blank line between the summary and further details.
 * The summary (everything before the first period) is used in the class or package 
 * overview section.
 *
 * The following inline tags can be used (not an exhaustive list): 
 * {@link some.other.class.Documentation} for linking to other docs or symbols
 * {@link some.other.class.Documentation Some Display Name} the link's appearance can be 
 * customized by adding a display name after the doc or symbol locator
 * {@code code goes here} for formatting as code
 * {@literal <>[]()foo} for interpreting literal text without converting to HTML markup 
 * or other tags.
 *
 * Optionally, the following tags may be used at the end of class documentation 
 * (not an exhaustive list):
 *
 * @author John Doe
 * @version 1.0
 * @since 5/10/15
 * @see some.other.class.Documentation
 * @deprecated This class has been replaced by some.other.package.BetterFileReader
 * 
 * You can also have custom tags for displaying additional information.
 * Using the @custom.<NAME> tag and the -tag custom.<NAME>:htmltag:"context"
 * command line option, you can create a custom tag.
 *
 * Example custom tag and generation:
 * @custom.updated 2.0    
 * Javadoc flag: -tag custom.updated🅰️"Updated in version:"
 * The above flag will display the value of @custom.updated under "Updated in version:"
 *
 */
public class FileReader {
}

As mesmas tags e formatos usados ​​para Classes também podem ser usados ​​para Enums e Interfaces.

Documentação do método #

Todos os comentários Javadoc começam com um comentário de bloco seguido por um asterisco (/**) e terminam quando o comentário de bloco termina (*/). Opcionalmente, cada linha pode começar com um espaço em branco arbitrário e um único asterisco; estes são ignorados quando os arquivos de documentação são gerados.

/**
 * Brief summary of method, ending with a period.
 *
 * Further description of method and what it does, including as much detail as is 
 * appropriate.  Inline tags such as
 * {@code code here}, {@link some.other.Docs}, and {@literal text here} can be used.
 *
 * If a method overrides a superclass method, {@inheritDoc} can be used to copy the 
 * documentation
 * from the superclass method
 *
 * @param stream Describe this parameter.  Include as much detail as is appropriate
 *               Parameter docs are commonly aligned as here, but this is optional.
 *               As with other docs, the documentation before the first period is 
 *               used as a summary.
 *
 * @return Describe the return values.  Include as much detail as is appropriate
 *         Return type docs are commonly aligned as here, but this is optional.
 *         As with other docs, the documentation before the first period is used as a 
 *         summary.
 *
 * @throws IOException Describe when and why this exception can be thrown.
 *                     Exception docs are commonly aligned as here, but this is 
 *                     optional.
 *                     As with other docs, the documentation before the first period 
 *                     is used as a summary.
 *                     Instead of @throws, @exception can also be used.
 *
 * @since 2.1.0
 * @see some.other.class.Documentation
 * @deprecated  Describe why this method is outdated. A replacement can also be specified.
 */
public String[] read(InputStream stream) throws IOException {
    return null;
}

Documentação do pacote #

É possível criar documentação em nível de pacote em Javadocs usando um arquivo chamado package-info.java. Este arquivo deve ser formatado conforme abaixo. Espaço em branco à esquerda e asteriscos opcionais, normalmente presentes em cada linha por motivo de formatação

/**
 * Package documentation goes here; any documentation before the first period will 
 * be used as a summary.
 *
 * It is common practice to leave a blank line between the summary and the rest
 * of the documentation; use this space to describe the package in as much detail
 * as is appropriate.
 * 
 * Inline tags such as {@code code here}, {@link reference.to.other.Documentation},
 * and {@literal text here} can be used in this documentation.
 */
package com.example.foo;

// The rest of the file must be empty.

No caso acima, você deve colocar este arquivo package-info.java dentro da pasta do pacote Java com.example.foo.

Construindo Javadocs a partir da linha de comando #

Muitos IDEs fornecem suporte para gerar HTML a partir de Javadocs automaticamente; algumas ferramentas de compilação (Maven e Gradle, por exemplo) também possuem plugins que podem lidar com a criação de HTML.

No entanto, essas ferramentas não são necessárias para gerar o HTML Javadoc; isso pode ser feito usando a ferramenta de linha de comando javadoc.

O uso mais básico da ferramenta é:

javadoc JavaFile.java

Que irá gerar HTML a partir dos comentários Javadoc em JavaFile.java.

Um uso mais prático da ferramenta de linha de comando, que lerá recursivamente todos os arquivos java em [source-directory], criará documentação para [package.name] e todos os sub-pacotes, e colocará o HTML gerado no arquivo [docs-directory] é:

javadoc -d [docs-directory] -subpackages -sourcepath [source-directory] [package.name]

Links #

A vinculação a outros Javadocs é feita com a tag @link:

/**
 * You can link to the javadoc of an already imported class using {@link ClassName}.
 *
 * You can also use the fully-qualified name, if the class is not already imported: 
 *  {@link some.other.ClassName}
 *
 * You can link to members (fields or methods) of a class like so:
 *  {@link ClassName#someMethod()}
 *  {@link ClassName#someMethodWithParameters(int, String)}
 *  {@link ClassName#someField}
 *  {@link #someMethodInThisClass()} - used to link to members in the current class
 *  
 * You can add a label to a linked javadoc like so:
 *  {@link ClassName#someMethod() link text} 
 */

Com a tag @see você pode adicionar elementos à seção Veja também. Como @param ou @return o local onde eles aparecem não é relevante. A especificação diz que você deve escrevê-lo após @return.

/**
 * This method has a nice explanation but you might found further
 * information at the bottom.
 *
 * @see ClassName#someMethod()
 */

Se você quiser adicionar links para recursos externos, basta usar a tag HTML <a>. Você pode usá-lo inline em qualquer lugar ou dentro das tags @link e @see.

/**
 * Wondering how this works? You might want
 * to check this <a href="http://stackoverflow.com/">great service</a>.
 *
 * @see <a href="http://stackoverflow.com/">Stack Overflow</a>
 */

Documentação de campo #

Todos os comentários Javadoc começam com um comentário de bloco seguido por um asterisco (/**) e terminam quando o comentário de bloco termina (*/). Opcionalmente, cada linha pode começar com um espaço em branco arbitrário e um único asterisco; estes são ignorados quando os arquivos de documentação são gerados.

/**
 * Fields can be documented as well.
 * 
 * As with other javadocs, the documentation before the first period is used as a
 * summary, and is usually separated from the rest of the documentation by a blank 
 * line.
 * 
 * Documentation for fields can use inline tags, such as:
 * {@code code here}
 * {@literal text here}
 * {@link other.docs.Here}
 * 
 * Field documentation can also make use of the following tags:
 * 
 * @since 2.1.0
 * @see some.other.class.Documentation
 * @deprecated Describe why this field is outdated
 */
public static final String CONSTANT_STRING = "foo";

Trechos de código dentro da documentação #

A maneira canônica de escrever código dentro da documentação é com a construção {@code }. Se você tiver quebra de código de várias linhas dentro de <pre></pre>.

/**
 * The Class TestUtils.
 * <p>
 * This is an {@code inline("code example")}.
 * <p>
 * You should wrap it in pre tags when writing multiline code.
 * <pre>{@code
 *  Example example1 = new FirstLineExample();
 *  example1.butYouCanHaveMoreThanOneLine();
 * }</pre>
 * <p>
 * Thanks for reading.
 */
class TestUtils {

Às vezes você pode precisar colocar algum código complexo dentro do comentário javadoc. O sinal @ é especialmente problemático. O uso da antiga tag <code> junto com a construção {@literal } resolve o problema.

/**
 * Usage:
 * <pre><code>
 * class SomethingTest {
 * {@literal @}Rule
 *  public SingleTestRule singleTestRule = new SingleTestRule("test1");
 *
 * {@literal @}Test
 *  public void test1() {
 *      // only this test will be executed
 *  }
 *
 *  ...
 * }
 * </code></pre>
 */
class SingleTestRule implements TestRule { }

Documentação de código embutido #

Além do código de documentação Javadoc pode ser documentado inline.

Comentários de linha única são iniciados por // e podem ser posicionados após uma instrução na mesma linha, mas não antes.

public void method() {
  
  //single line comment
  someMethodCall(); //single line comment after statement
  
}

Comentários de várias linhas são definidos entre /* e */. Eles podem abranger várias linhas e podem até ser posicionados entre instruções.

public void method(Object object) {
  
  /*
    multi 
    line 
    comment
  */
  object/*inner-line-comment*/.method(); 
}

JavaDocs são uma forma especial de comentários de várias linhas, começando com /**.

Como muitos comentários embutidos podem diminuir a legibilidade do código, eles devem ser usados ​​de forma esparsa caso o código não seja autoexplicativo o suficiente ou a decisão de design não seja óbvia.

Um caso de uso adicional para comentários de linha única é o uso de TAGs, que são palavras-chave curtas e orientadas por convenções. Alguns ambientes de desenvolvimento reconhecem certas convenções para esses comentários únicos. Exemplos comuns são

  • //TODO
  • //FIXME

Ou emita referências, ou seja, para Jira

  • //PRJ-1234
