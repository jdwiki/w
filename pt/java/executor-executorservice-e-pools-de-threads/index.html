A interface Executor em Java fornece uma maneira de desacoplar o envio de tarefas da mecânica de como cada tarefa será executada, incluindo detalhes de uso de encadeamentos, agendamento, etc. Um Executor é normalmente usado em vez de criar encadeamentos explicitamente. Com Executors, os desenvolvedores não terão que reescrever significativamente seu código para poder ajustar facilmente a política de execução de tarefas de seu programa.

Armadilhas

  • Quando você agenda uma tarefa para execução repetida, dependendo do ScheduledExecutorService usado, sua tarefa pode ser suspensa de qualquer execução posterior, se uma execução de sua tarefa causar uma exceção que não é tratada. Veja Mãe F**k the ScheduledExecutorService!

ThreadPoolExecutor #

Um Executor comum usado é o ThreadPoolExecutor, que cuida do tratamento de Threads. Você pode configurar a quantidade mínima de Threads que o executor sempre precisa manter quando não há muito o que fazer (é chamado de tamanho do núcleo) e um tamanho máximo de Thread para o qual o Pool pode crescer, se houver mais trabalho a fazer. Depois que a carga de trabalho diminui, o Pool reduz lentamente a contagem de threads novamente até atingir o tamanho mínimo.

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    1,                                     // keep at least one thread ready, 
                                           // even if no Runnables are executed
    5,                                     // at most five Runnables/Threads
                                           // executed in parallel
    1, TimeUnit.MINUTES,                   // idle Threads terminated after one
                                           // minute, when min Pool size exceeded
    new ArrayBlockingQueue<Runnable>(10)); // outstanding Runnables are kept here

pool.execute(new Runnable() {
    @Override public void run() {
        //code to run
    }
});

Observação Se você configurar o ThreadPoolExecutor com uma fila unbounded, a contagem de threads não excederá corePoolSize já que novos threads são criados apenas se a fila estiver cheia:

ThreadPoolExecutor com todos os parâmetros:

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

de JavaDoc

Se houver mais de corePoolSize, mas menos de maximumPoolSize threads em execução, um novo thread será criado somente se a fila estiver cheia.

Vantagens:

  1. O tamanho do BlockingQueue pode ser controlado e os cenários de falta de memória podem ser evitados. O desempenho do aplicativo não será prejudicado com tamanho de fila limitado limitado.

  2. Você pode usar políticas existentes ou criar novas políticas de Manipulador de Rejeição.

  3. No ThreadPoolExecutor.AbortPolicy padrão, o manipulador lança um RejectedExecutionException de tempo de execução após a rejeição.

  4. Em ThreadPoolExecutor.CallerRunsPolicy, a thread que invoca a execução executa a tarefa. Isso fornece um mecanismo de controle de feedback simples que diminuirá a taxa de envio de novas tarefas.

  5. Em ThreadPoolExecutor.DiscardPolicy, uma tarefa que não pode ser executada é simplesmente descartada.

  6. Em ThreadPoolExecutor.DiscardOldestPolicy, se o executor não for encerrado, a tarefa no início da fila de trabalho é descartada e, em seguida, a execução é repetida (o que pode falhar novamente, fazendo com que isso seja repetido).

  7. O ThreadFactory personalizado pode ser configurado, o que é útil:

  8. Para definir um nome de thread mais descritivo

  9. Para definir o status do daemon de encadeamento

  10. Para definir a prioridade do encadeamento

Recuperando o valor da computação - Callable #

Se o seu cálculo produzir algum valor de retorno que posteriormente será necessário, uma tarefa simples Runnable não será suficiente. Para tais casos, você pode usar ExecutorService.submit([Callable][callable-and-future]<T>) que retorna um valor após a conclusão da execução.

O Serviço retornará um [Future][callable-and-future] que você pode usar para recuperar o resultado da execução da tarefa.

// Submit a callable for execution
ExecutorService pool = anExecutorService;
Future<Integer> future = pool.submit(new Callable<Integer>() {
    @Override public Integer call() {
        //do some computation
        return new Random().nextInt();
    }
});    
// ... perform other tasks while future is executed in a different thread

Quando você precisar obter o resultado do futuro, chame future.get()

  • Aguarde indefinidamente para o futuro terminar com um resultado.

      try {
      // Blocks current thread until future is completed
      Integer result = future.get(); 
  catch (InterruptedException || ExecutionException e) {
      // handle appropriately
  }

  • Espere o futuro terminar, mas não mais do que o tempo especificado.

      try {
      // Blocks current thread for a maximum of 500 milliseconds.
      // If the future finishes before that, result is returned,
      // otherwise TimeoutException is thrown.
      Integer result = future.get(500, TimeUnit.MILLISECONDS); 
  catch (InterruptedException || ExecutionException || TimeoutException e) {
      // handle appropriately
  }

Se o resultado de uma tarefa agendada ou em execução não for mais necessário, você pode chamar Future.cancel(boolean) para cancelá-lo.

  • Chamar cancel(false) apenas removerá a tarefa da fila de tarefas a serem executadas.
  • Chamar cancel(true) também interromperá a tarefa se ela estiver em execução.

Geralmente o comando execute() é usado para chamadas de fogo e esquecimento (sem necessidade de analisar o resultado) e o comando submit() é usado para analisar o resultado do objeto Future.

Dispare e esqueça - Tarefas executáveis #

Os executores aceitam um java.lang.Runnable que contém código (potencialmente computacionalmente ou de longa duração ou pesado) para ser executado em outro Thread.

O uso seria:

Executor exec = anExecutor;
exec.execute(new Runnable() {
    @Override public void run() {
        //offloaded work, no need to get result back
    }
});

Observe que, com este executor, você não tem meios de recuperar nenhum valor calculado. Com o Java 8, pode-se utilizar lambdas para encurtar o exemplo de código.

Executor exec = anExecutor;
exec.execute(() -> {
    //offloaded work, no need to get result back
});

Lidar com execução rejeitada #

Se

  1. você tenta enviar tarefas para um Executor de desligamento ou
  2. a fila está saturada (somente possível com limitadas) e o número máximo de Threads foi atingido,

RejectedExecutionHandler.rejectedExecution(Runnable, ThreadPoolExecutor) será chamado.

O comportamento padrão é que você receberá um RejectedExecutionException lançado no chamador. Mas há mais comportamentos predefinidos disponíveis:

  • ThreadPoolExecutor.AbortPolicy (padrão, lançará REE)
  • ThreadPoolExecutor.CallerRunsPolicy (executa a tarefa no thread do chamador - bloqueando-o)
  • ThreadPoolExecutor.DiscardPolicy (descartar tarefa silenciosamente)
  • ThreadPoolExecutor.DiscardOldestPolicy (descartar silenciosamente a tarefa mais antiga na fila e repetir a execução da nova tarefa)

Você pode configurá-los usando um dos ThreadPool construtores:

public ThreadPoolExecutor(int corePoolSize,
                      int maximumPoolSize,
                      long keepAliveTime,
                      TimeUnit unit,
                      BlockingQueue<Runnable> workQueue,
                      RejectedExecutionHandler handler) // <--

public ThreadPoolExecutor(int corePoolSize,
                      int maximumPoolSize,
                      long keepAliveTime,
                      TimeUnit unit,
                      BlockingQueue<Runnable> workQueue,
                      ThreadFactory threadFactory,
                      RejectedExecutionHandler handler) // <--

Você também pode implementar seu próprio comportamento estendendo a interface RejectedExecutionHandler:

void rejectedExecution(Runnable r, ThreadPoolExecutor executor)

Casos de uso para diferentes tipos de construções de simultaneidade #

  1. Serviço de Executor

ExecutorService executor = Executors.newFixedThreadPool(50);

É simples e fácil de usar. Ele oculta detalhes de baixo nível de ThreadPoolExecutor.

Eu prefiro este quando o número de tarefas ‘Callable/Runnable’ é pequeno em número e empilhamento de tarefas em fila ilimitada não aumenta a memória e degrada o desempenho do sistema. Se você tem restrições CPU/Memory, eu prefiro usar ThreadPoolExecutor com restrições de capacidade e RejectedExecutionHandler para lidar com a rejeição de tarefas.

  1. Trava de contagem regressiva

CountDownLatch será inicializado com uma determinada contagem. Esta contagem é diminuída por chamadas ao método countDown(). Threads esperando que essa contagem chegue a zero podem chamar um dos métodos await(). Chamar await() bloqueia o thread até que a contagem chegue a zero. Esta classe permite que um encadeamento java aguarde até que outro conjunto de encadeamentos conclua suas tarefas.

Casos de uso:

  1. Alcançando o Paralelismo Máximo: Às vezes queremos iniciar um número de threads ao mesmo tempo para alcançar o paralelismo máximo

  2. Aguarde a conclusão de N threads antes de iniciar a execução

  3. Detecção de impasse.

  4. ThreadPoolExecutor : Fornece mais controle. Se o aplicativo for restringido pelo número de tarefas executáveis/chamáveis ​​pendentes, você poderá usar a fila limitada definindo a capacidade máxima. Quando a fila atingir a capacidade máxima, você poderá definir RejectionHandler. Java fornece quatro tipos de RejectedExecutionHandler políticas.

      1. `ThreadPoolExecutor.AbortPolicy`, the handler throws a runtime RejectedExecutionException upon rejection.

    1. ThreadPoolExecutor.CallerRunsPolicy`, the thread that invokes execute itself runs the task. This provides a simple feedback control mechanism that will slow down the rate that new tasks are submitted.

    2. In ThreadPoolExecutor.DiscardPolicy, a task that cannot be executed is simply dropped.

    3. ThreadPoolExecutor.DiscardOldestPolicy, if the executor is not shut down, the task at the head of the work queue is dropped, and then execution is retried (which can fail again, causing this to be repeated.)

Se você quiser simular o comportamento CountDownLatch, você pode usar o método invokeAll().

  1. Mais um mecanismo que você não citou é ForkJoinPool

O ForkJoinPool foi adicionado ao Java no Java 7. O ForkJoinPool é semelhante ao o Java ExecutorService, mas com uma diferença. O ForkJoinPool torna fácil para as tarefas dividirem seu trabalho em tarefas menores que são então submetido ao ForkJoinPool também. O roubo de tarefas acontece em ForkJoinPool quando threads de trabalho livres roubam tarefas da fila de threads de trabalho ocupada.

O Java 8 introduziu mais uma API em ExecutorService para criar um pool de roubo de trabalho. Você não precisa criar RecursiveTask e RecursiveAction, mas ainda pode usar ForkJoinPool.

   public static ExecutorService newWorkStealingPool()

Cria um pool de threads que rouba o trabalho usando todos os processadores disponíveis como seu nível de paralelismo de destino.

Por padrão, levará o número de núcleos da CPU como parâmetro.

Todos esses quatro mecanismos são complementares entre si. Dependendo do nível de granularidade que você deseja controlar, você deve escolher os corretos.

Aguarde a conclusão de todas as tarefas no ExecutorService #

Vamos dar uma olhada em várias opções para aguardar a conclusão das tarefas enviadas para Executor

  1. ExecutorService invokeAll()

Executa as tarefas dadas, retornando uma lista de Futuros com seus status e resultados quando tudo estiver concluído.

Exemplo:

import java.util.concurrent.*;
import java.util.*;

public class InvokeAllDemo{
    public InvokeAllDemo(){
        System.out.println("creating service");
        ExecutorService service = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());
        
        List<MyCallable> futureList = new ArrayList<MyCallable>();
        for (int i = 0; i < 10; i++){
            MyCallable myCallable = new MyCallable((long)i);
            futureList.add(myCallable);
        }
        System.out.println("Start");
        try{
            List<Future<Long>> futures = service.invokeAll(futureList);  
        } catch(Exception err){
            err.printStackTrace();
        }
        System.out.println("Completed");
        service.shutdown();
    }
    public static void main(String args[]){
        InvokeAllDemo demo = new InvokeAllDemo();
    }
    class MyCallable implements Callable<Long>{
        Long id = 0L;
        public MyCallable(Long val){
            this.id = val;
        }
        public Long call(){
            // Add your business logic
            return id;
        }
    }
}

  1. Trava de contagem regressiva

    A synchronization aid that allows one or more threads to wait until a set of operations being performed in other threads completes.

    A CountDownLatch is initialized with a given count. The await methods block until the current count reaches zero due to invocations of the countDown() method, after which all waiting threads are released and any subsequent invocations of await return immediately. This is a one-shot phenomenon – the count cannot be reset. If you need a version that resets the count, consider using a CyclicBarrier.

  2. ForkJoinPool ou newWorkStealingPool() em Executors

  3. Itere todos os objetos Future criados após o envio para ExecutorService

  4. Maneira recomendada de desligamento da página de documentação do Oracle de ExecutorService:

    void shutdownAndAwaitTermination(ExecutorService pool) {
    pool.shutdown(); // Disable new tasks from being submitted
    try {
      // Wait a while for existing tasks to terminate
      if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
        pool.shutdownNow(); // Cancel currently executing tasks
        // Wait a while for tasks to respond to being cancelled
        if (!pool.awaitTermination(60, TimeUnit.SECONDS))
            System.err.println("Pool did not terminate");
      }
    } catch (InterruptedException ie) {
      // (Re-)Cancel if current thread also interrupted
      pool.shutdownNow();
      // Preserve interrupt status
      Thread.currentThread().interrupt();
    }

    shutdown(): Initiates an orderly shutdown in which previously submitted tasks are executed, but no new tasks will be accepted.

shutdownNow():Tenta parar todas as tarefas em execução ativa, interrompe o processamento das tarefas em espera e retorna uma lista das tarefas que estavam aguardando execução.

No exemplo acima, se suas tarefas estão demorando mais para serem concluídas, você pode alterar if condition para while condition

Substituir

   if (!pool.awaitTermination(60, TimeUnit.SECONDS))

com

   while(!pool.awaitTermination(60, TimeUnit.SECONDS)) {
     Thread.sleep(60000);
 }

Casos de uso para diferentes tipos de ExecutorService #

Executors retorna diferentes tipos de ThreadPools atendendo a necessidades específicas.

  1. Serviço de Executor estático público newSingleThreadExecutor()

Cria um Executor que usa um único thread de trabalho operando em uma fila ilimitada

Há uma diferença entre newFixedThreadPool(1) e newSingleThreadExecutor() como o documento java diz para o último:

> Unlike the otherwise equivalent newFixedThreadPool(1) the returned executor is guaranteed not to be reconfigurable to use additional threads.

Which means that a `newFixedThreadPool` can be reconfigured later in the program by: `((ThreadPoolExecutor) fixedThreadPool).setMaximumPoolSize(10)`

Isso não é possível para newSingleThreadExecutor

Casos de uso:

1. You want to execute the submitted tasks in a sequence.
2. You need only one Thread to handle all your request

Contras:

1. Unbounded queue is harmful
  1. public static ExecutorService newFixedThreadPool(int nThreads)

Cria um pool de encadeamentos que reutiliza um número fixo de encadeamentos operando em uma fila ilimitada compartilhada. Em qualquer ponto, no máximo nThreads threads serão tarefas de processamento ativas. Se tarefas adicionais forem enviadas quando todos os encadeamentos estiverem ativos, eles aguardarão na fila até que um encadeamento esteja disponível

Casos de uso:

  1. Uso efetivo dos núcleos disponíveis. Configure nThreads como Runtime.getRuntime().availableProcessors()
  2. Quando você decidir que o número de threads não deve exceder um número no pool de threads

Contras:

  1. A fila ilimitada é prejudicial.

  2. public static ExecutorService newCachedThreadPool()

Cria um pool de threads que cria novos threads conforme necessário, mas reutilizará threads construídos anteriormente quando estiverem disponíveis

Casos de uso:

  1. Para tarefas assíncronas de curta duração

Contras:

  1. A fila ilimitada é prejudicial.

  2. Cada nova tarefa criará um novo encadeamento se todos os encadeamentos existentes estiverem ocupados. Se a tarefa estiver demorando muito, mais número de threads será criado, o que prejudicará o desempenho do sistema. Alternativa neste caso: newFixedThreadPool

  3. public static ScheduledExecutorService newScheduledThreadPool(int corePoolSize)

Cria um pool de threads que pode agendar comandos para serem executados após um determinado atraso ou para serem executados periodicamente.

Casos de uso:

  1. Lidar com eventos recorrentes com atrasos, que acontecerão no futuro em determinado intervalo de tempo

Contras:

  1. A fila ilimitada é prejudicial.

5.public static ExecutorService newWorkStealingPool()

Cria um pool de threads que rouba o trabalho usando todos os processadores disponíveis como seu nível de paralelismo de destino

Casos de uso:

  1. Para problemas do tipo dividir e conquistar.
  2. Uso eficaz de threads ociosos. Threads ociosos roubam tarefas de threads ocupados.

Contras:

  1. O tamanho ilimitado da fila é prejudicial.

Você pode ver uma desvantagem comum em todos esses ExecutorService: fila ilimitada. Isso será resolvido com ThreadPoolExecutor

ThreadPoolExecutor(int corePoolSize, int maximumPoolSize, long keepAliveTime, 
TimeUnit unit, BlockingQueue<Runnable> workQueue, ThreadFactory threadFactory,
RejectedExecutionHandler handler)

Com ThreadPoolExecutor, você pode

  1. Controle o tamanho do pool de threads dinamicamente
  2. Defina a capacidade para BlockingQueue
  3. Defina RejectionExecutionHander quando a fila estiver cheia
  4. CustomThreadFactory para adicionar algumas funcionalidades adicionais durante a criação do Thread (public Thread newThread(Runnable r)

Agendamento de tarefas para execução em um horário fixo, após um atraso ou repetidamente #

A classe ScheduledExecutorService fornece métodos para agendar tarefas simples ou repetidas de várias maneiras. O exemplo de código a seguir pressupõe que pool foi declarado e inicializado da seguinte forma:

ScheduledExecutorService pool = Executors.newScheduledThreadPool(2);

Além dos métodos normais ExecutorService, a API ScheduledExecutorService adiciona 4 métodos que agendam tarefas e retornam objetos ScheduledFuture. Este último pode ser usado para recuperar resultados (em alguns casos) e cancelar tarefas.

Iniciar uma tarefa após um atraso fixo #

O exemplo a seguir agenda uma tarefa para iniciar após dez minutos.

ScheduledFuture<Integer> future = pool.schedule(new Callable<>() {
        @Override public Integer call() {
            // do something
            return 42;
        }
    }, 
    10, TimeUnit.MINUTES);

Iniciar tarefas a uma taxa fixa #

O exemplo a seguir agenda uma tarefa para iniciar após dez minutos e, em seguida, repetidamente a uma taxa de uma vez a cada minuto.

ScheduledFuture<?> future = pool.scheduleAtFixedRate(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

A execução da tarefa continuará de acordo com o cronograma até que o pool seja encerrado, o futuro seja cancelado ou uma das tarefas encontre uma exceção.

É garantido que as tarefas agendadas por uma determinada chamada scheduledAtFixedRate não se sobreponham no tempo. Se uma tarefa demorar mais do que o período prescrito, as execuções de tarefas seguintes e posteriores podem começar com atraso.

Iniciar tarefas com um atraso fixo #

O exemplo a seguir agenda uma tarefa para iniciar após dez minutos e, em seguida, repetidamente com um atraso de um minuto entre o término de uma tarefa e o início da próxima.

ScheduledFuture<?> future = pool.scheduleWithFixedDelay(new Runnable() {
        @Override public void run() {
            // do something
        }
    }, 
    10, 1, TimeUnit.MINUTES);

A execução da tarefa continuará de acordo com o cronograma até que o pool seja encerrado, o futuro seja cancelado ou uma das tarefas encontre uma exceção.
