Armadilhas do Java - Nulos e NullPointerException

O valor null é o valor padrão para um valor não inicializado de um campo cujo tipo é um tipo de referência.

NullPointerException (ou NPE) é a exceção que é lançada quando você tenta realizar uma operação inadequada na referência de objeto null. Tais operações incluem:

  • chamando um método de instância em um objeto de destino null,
  • acessando um campo de um objeto de destino null,
  • tentando indexar um objeto de array null ou acessar seu comprimento,
  • usando uma referência de objeto null como o mutex em um bloco synchronized,
  • lançando uma referência de objeto null,
  • unboxing de uma referência de objeto null, e
  • lançando uma referência de objeto null.

As causas mais comuns para NPEs:

  • esquecer de inicializar um campo com um tipo de referência,
  • esquecer de inicializar elementos de um array de um tipo de referência, ou
  • não testar os resultados de determinados métodos de API que são especificados como retornando null em determinadas circunstâncias.

Exemplos de métodos comumente usados ​​que retornam null incluem:

  • O método get(key) na API Map retornará um null se você chamá-lo com uma chave que não tenha um mapeamento.
  • Os métodos getResource(path) e getResourceAsStream(path) nas APIs ClassLoader e Class retornarão null se o recurso não puder ser encontrado.
  • O método get() na API Reference retornará null se o coletor de lixo tiver limpado a referência.
  • Vários métodos getXxxx nas APIs de servlet Java EE retornarão null se você tentar buscar um parâmetro de solicitação inexistente, sessão ou atributo de sessão e assim por diante.

Existem estratégias para evitar NPEs indesejados, como testar explicitamente para null ou usar “Yoda Notation”, mas essas estratégias geralmente têm o resultado indesejável de esconder problemas em seu código que realmente deveriam ser corrigidos.

Armadilha - Nulos inesperados “Fazendo bem” #

No StackOverflow, muitas vezes vemos código como este em Respostas:

public String joinStrings(String a, String b) {
    if (a == null) {
        a = "";
    }
    if (b == null) {
        b = "";
    }
    return a + ": " + b;
}

Muitas vezes, isso é acompanhado de uma declaração que é a “melhor prática” para testar null assim para evitar NullPointerException.

É a melhor prática? Resumindo: Não.

Existem algumas suposições subjacentes que precisam ser questionadas antes que possamos dizer se é uma boa ideia fazer isso em nosso joinStrings:

O que significa “a” ou “b” ser nulo? #

Um valor String pode ter zero ou mais caracteres, então já temos uma forma de representar uma string vazia. null significa algo diferente de ""? Se não, então é problemático ter duas maneiras de representar uma string vazia.

O nulo veio de uma variável não inicializada? #

Um null pode vir de um campo não inicializado ou de um elemento de matriz não inicializado. O valor pode ser não inicializado por design ou por acidente. Se foi por acidente, então isso é um bug.

O nulo representa um “não sei” ou “valor ausente”?

Às vezes, um null pode ter um significado genuíno; por exemplo. que o valor real de uma variável é desconhecido ou indisponível ou “opcional”. No Java 8, a classe Optional fornece uma maneira melhor de expressar isso.

Se isso for um bug (ou um erro de design), devemos “fazer o bem”?

Uma interpretação do código é que estamos “remediando” um ’null’ inesperado usando uma string vazia em seu lugar. É a estratégia correta? Seria melhor deixar o NullPointerException acontecer e, em seguida, capturar a exceção mais acima na pilha e registrá-la como um bug?

O problema de “fazer o bem” é que pode esconder o problema ou torná-lo mais difícil de diagnosticar.

Isso é eficiente/bom para a qualidade do código? #

Se a abordagem “fazer bom” for usada consistentemente, seu código conterá muitos testes nulos “defensivos”. Isso vai torná-lo mais longo e mais difícil de ler. Além disso, todos esses testes e “reparos” podem afetar o desempenho do seu aplicativo.

Resumindo #

Se null for um valor significativo, então testar o caso null é a abordagem correta. O corolário é que se um valor null for significativo, então isso deve ser claramente documentado nos javadocs de qualquer método que aceite o valor null ou o retorne.

Caso contrário, é melhor tratar um null inesperado como um erro de programação, e deixar o NullPointerException acontecer para que o desenvolvedor saiba que há um problema no código.

Armadilha - Usando null para representar uma matriz ou coleção vazia #

Alguns programadores pensam que é uma boa ideia economizar espaço usando um null para representar uma matriz ou coleção vazia. Embora seja verdade que você pode economizar uma pequena quantidade de espaço, o outro lado é que isso torna seu código mais complicado e mais frágil. Compare estas duas versões de um método para somar uma matriz:

A primeira versão é como você normalmente codificaria o método:

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed
 * @return the sum
 **/
public int sum(int[] values) {
    int sum = 0;
    for (int value : values) {
        sum += value;
    }
    return sum;
}

A segunda versão é como você precisa codificar o método se tiver o hábito de usar null para representar um array vazio.

/**
 * Sum the values in an array of integers.
 * @arg values the array to be summed, or null.
 * @return the sum, or zero if the array is null.
 **/
public int sum(int[] values) {
    int sum = 0;
    if (values != null) {
        for (int value : values) {
            sum += value;
        }
    }
    return sum;
}

Como você pode ver, o código é um pouco mais complicado. Isso é diretamente atribuível à decisão de usar null dessa maneira.

Agora considere se este array que pode ser um null é usado em muitos lugares. Em cada lugar onde você o usa, você precisa considerar se você precisa testar para null. Se você perder um teste null que precisa estar lá, você corre o risco de um NullPointerException. Portanto, a estratégia de usar null dessa forma torna sua aplicação mais frágil; ou seja, mais vulneráveis ​​às consequências dos erros do programador.

A lição aqui é usar arrays vazios e listas vazias quando é isso que você quer dizer.

int[] values = new int[0];                     // always empty
List<Integer> list = new ArrayList();          // initially empty
List<Integer> list = Collections.emptyList();  // always empty

A sobrecarga de espaço é pequena e existem outras maneiras de minimizá-la, se isso valer a pena.

Pitfall - Retornando null em vez de lançar uma exceção #

Alguns programadores Java têm uma aversão geral a lançar ou propagar exceções. Isso leva a um código como o seguinte:

public Reader getReader(String pathname) {
    try {
        return new BufferedReader(FileReader(pathname));
    } catch (IOException ex) {
        System.out.println("Open failed: " + ex.getMessage());
        return null;
    }

}

Então qual é o problema com isso?

O problema é que o getReader está retornando um null como um valor especial para indicar que o Reader não pôde ser aberto. Agora o valor retornado precisa ser testado para ver se é null antes de ser usado. Se o teste for deixado de fora, o resultado será um NullPointerException.

Na verdade, existem três problemas aqui:

  1. O IOException foi detectado cedo demais.
  2. A estrutura deste código significa que existe o risco de vazamento de um recurso.
  3. Um null foi usado e então retornado porque nenhum Reader “real” estava disponível para retornar.

Na verdade, supondo que a exceção precisasse ser capturada cedo assim, havia algumas alternativas para retornar null:

  1. Seria possível implementar uma classe NullReader; por exemplo. uma onde as operações da API se comportam como se o leitor já estivesse na posição “final do arquivo”.
  2. Com Java 8, seria possível declarar getReader como retornando um Optional<Reader>.

Armadilha - Não verificar se um fluxo de E/S não foi inicializado ao fechá-lo #

Para evitar vazamentos de memória, não se deve esquecer de fechar um fluxo de entrada ou um fluxo de saída cujo trabalho esteja concluído. Isso geralmente é feito com uma instrução try-catch-finally sem a parte catch:

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        out.close();
    }
}

Embora o código acima possa parecer inocente, ele tem uma falha que pode impossibilitar a depuração. Se a linha onde out for inicializado (out = new FileOutputStream(filename)) lançar uma exceção, então out será null quando out.close() for executado, resultando em uma desagradável NullPointerException !

Para evitar isso, simplesmente certifique-se de que o stream não seja null antes de tentar fechá-lo.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    FileOutputStream out = null;
    try {
        out = new FileOutputStream(filename);
        for(; count > 0; count--)
            out.write(0);
    } finally {
        if (out != null)
            out.close();
    }
}

Uma abordagem ainda melhor é try-with-resources, já que fechará automaticamente o fluxo com uma probabilidade de 0 para lançar um NPE sem a necessidade de um bloco finally.

void writeNullBytesToAFile(int count, String filename) throws IOException {
    try (FileOutputStream out = new FileOutputStream(filename)) {
        for(; count > 0; count--)
            out.write(0);
    }
}

Armadilha - O uso desnecessário de Primitive Wrappers pode levar a NullPointerExceptions #

Às vezes, os programadores que são novos em Java usarão tipos primitivos e wrappers de forma intercambiável. Isso pode levar a problemas. Considere este exemplo:

public class MyRecord {
    public int a, b;
    public Integer c, d;
}

...
MyRecord record = new MyRecord();
record.a = 1;               // OK
record.b = record.b + 1;    // OK
record.c = 1;               // OK
record.d = record.d + 1;    // throws a NullPointerException

Nossa classe MyRecord1 depende da inicialização padrão para inicializar os valores em seus campos. Assim, quando new um registro, os campos a e b serão definidos como zero, e os campos c e d serão definidos como null.

Quando tentamos usar os campos inicializados padrão, vemos que os campos int funcionam o tempo todo, mas os campos Integer funcionam em alguns casos e não em outros. Especificamente, no caso de falha (com d), o que acontece é que a expressão do lado direito tenta desempacotar uma referência null, e é isso que faz com que a NullPointerException seja lançada.

Existem algumas maneiras de ver isso:

  • Se os campos c e d precisam ser wrappers primitivos, então não devemos confiar na inicialização padrão, ou devemos testar para null. Para o primeiro é a abordagem correta a menos que haja um significado definido para os campos no estado null.

  • Se os campos não precisam ser wrappers primitivos, então é um erro torná-los wrappers primitivos. Além desse problema, os wrappers primitivos têm sobrecargas extras em relação aos tipos primitivos.

A lição aqui é não usar tipos de wrapper primitivos, a menos que você realmente precise.

1 - Esta classe não é um exemplo de boas práticas de codificação. Por exemplo, uma classe bem projetada não teria campos públicos. No entanto, esse não é o objetivo deste exemplo.

Armadilha - Usando “notação Yoda” para evitar NullPointerException #

Muitos códigos de exemplo postados no StackOverflow incluem trechos como este:

if ("A".equals(someString)) {
    // do something
}

Isso “previne” ou “evita” uma possível NullPointerException no caso de someString ser null. Além disso, é discutível que

    "A".equals(someString)

é melhor que:

    someString != null && someString.equals("A")

(É mais conciso e, em algumas circunstâncias, pode ser mais eficiente. No entanto, como argumentamos abaixo, a concisão pode ser negativa.)

No entanto, a verdadeira armadilha é usar o teste Yoda para evitar NullPointerExceptions como uma questão de hábito.

Quando você escreve "A".equals(someString) você está na verdade “fazendo bem” o caso onde someString passa a ser null. Mas, como outro exemplo (https://www.wikiod.com/pt/java/armadilhas-do-java—nulos-e-nullpointerexception#Armadilha - “Fazer o bem” nulos inesperados ) explica, “tornar bom” null valores podem ser prejudiciais por várias razões.

Isso significa que as condições do Yoda não são “práticas recomendadas”1. A menos que o null seja esperado, é melhor deixar o NullPointerException acontecer para que você possa obter uma falha no teste de unidade (ou um relatório de bug). Isso permite que você encontre e corrija o bug que causou o aparecimento do null inesperado/indesejado.

As condições Yoda só devem ser usadas nos casos em que o null é esperado porque o objeto que você está testando veio de uma API que está documentada como retornando um null. E, sem dúvida, poderia ser melhor usar uma das maneiras menos bonitas de expressar o teste porque isso ajuda a destacar o teste null para alguém que está revisando seu código.

1 - De acordo com Wikipedia: “As melhores práticas de codificação são um conjunto de regras informais que a comunidade de desenvolvimento de software aprendeu ao longo do tempo que pode ajudar a melhorar a qualidade do software.”. Usar a notação Yoda não alcança isso. Em muitas situações, isso piora o código.
