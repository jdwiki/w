  • TreeSet de um tipo Java simples
  • TreeMap/TreeSet de um tipo Java personalizado

TreeMap e TreeSet são coleções básicas de Java adicionadas no Java 1.2. TreeMap é uma implementação mutável, ordenada, Map. Da mesma forma, TreeSet é uma implementação mutável, ordenada de Set.

TreeMap é implementado como uma árvore Red-Black, que fornece tempos de acesso O(log n). TreeSet é implementado usando um TreeMap com valores fictícios.

Ambas as coleções não são thread-safe.

TreeMap de um tipo Java simples #

Primeiro, criamos um mapa vazio e inserimos alguns elementos nele:

TreeMap<Integer, String> treeMap = new TreeMap<>();

TreeMap<Integer, String> treeMap = new TreeMap<Integer, String>();

treeMap.put(10, "ten");
treeMap.put(4, "four");
treeMap.put(1, "one");
treeSet.put(12, "twelve");

Assim que tivermos alguns elementos no mapa, podemos realizar algumas operações:

System.out.println(treeMap.firstEntry()); // Prints 1=one
System.out.println(treeMap.lastEntry()); // Prints 12=twelve
System.out.println(treeMap.size()); // Prints 4, since there are 4 elemens in the map
System.out.println(treeMap.get(12)); // Prints twelve
System.out.println(treeMap.get(15)); // Prints null, since the key is not found in the map

Também podemos iterar sobre os elementos do mapa usando um Iterator ou um loop foreach. Observe que as entradas são impressas de acordo com sua ordenação natural, não a pedido de inserção:

for (Entry<Integer, String> entry : treeMap.entrySet()) {
    System.out.print(entry + " "); //prints 1=one 4=four 10=ten 12=twelve 
}

Iterator<Entry<Integer, String>> iter = treeMap.entrySet().iterator();
while (iter.hasNext()) {
    System.out.print(iter.next() + " "); //prints 1=one 4=four 10=ten 12=twelve 
}

TreeSet de um tipo Java simples #

Primeiro, criamos um conjunto vazio e inserimos alguns elementos nele:

TreeSet<Integer> treeSet = new TreeSet<>();

TreeSet<Integer> treeSet = new TreeSet<Integer>();

treeSet.add(10);
treeSet.add(4);
treeSet.add(1);
treeSet.add(12);

Uma vez que temos alguns elementos no conjunto, podemos realizar algumas operações:

System.out.println(treeSet.first()); // Prints 1
System.out.println(treeSet.last()); // Prints 12
System.out.println(treeSet.size()); // Prints 4, since there are 4 elemens in the set
System.out.println(treeSet.contains(12)); // Prints true
System.out.println(treeSet.contains(15)); // Prints false

Também podemos iterar sobre os elementos do mapa usando um Iterator ou um loop foreach. Observe que as entradas são impressas de acordo com sua ordenação natural, não com a ordem de inserção:

for (Integer i : treeSet) {
    System.out.print(i + " "); //prints 1 4 10 12
}

Iterator<Integer> iter = treeSet.iterator();
while (iter.hasNext()) {
    System.out.print(iter.next() + " "); //prints 1 4 10 12
}

TreeMap/TreeSet de um tipo Java personalizado #

Já que TreeMaps e TreeSets mantêm chaves/elementos de acordo com sua ordenação natural. Portanto, as chaves TreeMap e os elementos TreeSet devem ser comparáveis ​​entre si.

Digamos que tenhamos uma classe Person personalizada:

public class Person  {

    private int id;
    private String firstName, lastName;
    private Date birthday;

    //... Constuctors, getters, setters and various methods
}

Se o armazenarmos como está em um TreeSet (ou uma chave em um TreeMap):

TreeSet<Person2> set = ...      
set.add(new Person(1,"first","last",Date.from(Instant.now())));

Em seguida, encontraríamos uma exceção como esta:

Exception in thread "main" java.lang.ClassCastException: Person cannot be cast to java.lang.Comparable
    at java.util.TreeMap.compare(TreeMap.java:1294)
    at java.util.TreeMap.put(TreeMap.java:538)
    at java.util.TreeSet.add(TreeSet.java:255)

Para corrigir isso, vamos supor que queremos ordenar instâncias Person com base na ordem de seus ids (private int id). Poderíamos fazê-lo de duas maneiras:

  1. Uma solução é modificar Person para que ele implemente a Interface Comparável:

     public class Person implements Comparable<Person> {
     private int id;
     private String firstName, lastName;
     private Date birthday;

     //... Constuctors, getters, setters and various methods

     @Override
     public int compareTo(Person o) {
         return Integer.compare(this.id, o.id); //Compare by id
     }
 }

  2. Outra solução é fornecer ao TreeSet um Comparator:

TreeSet<Person> treeSet = new TreeSet<>((personA, personB) -> Integer.compare(personA.getId(), personB.getId()));

TreeSet<Person> treeSet = new TreeSet<>(new Comparator<Person>(){
    @Override
    public int compare(Person personA, Person personB) {
        return Integer.compare(personA.getId(), personB.getId());
    }
});

No entanto, há duas ressalvas para ambas as abordagens:

  1. É muito importante não modificar nenhum campo usado para ordenar uma vez que uma instância tenha sido inserida em um TreeSet/TreeMap. No exemplo acima, se alterarmos o id de uma pessoa que já está inserida na coleção, podemos nos deparar com um comportamento inesperado.

  2. É importante implementar a comparação de forma adequada e consistente. De acordo com o Javadoc:

O implementador deve garantir sgn(x.compareTo(y)) == -sgn(y.compareTo(x)) para todos os x e y. (Isto implica que x.compareTo(y) deve lançar uma exceção se y.compareTo(x) lançar uma exceção.)

O implementador também deve garantir que a relação seja transitiva: (x.compareTo(y)>0 && y.compareTo(z)>0) implica x.compareTo(z)>0.

Finalmente, o implementador deve garantir que x.compareTo(y)==0 implica que sgn(x.compareTo(z)) == sgn(y.compareTo(z)), para todos os z.

TreeMap e TreeSet Thread Safety #

TreeMap e TreeSet não são coleções thread-safe, portanto, cuidado deve ser tomado para garantir quando usado em programas multithread.

Tanto o TreeMap quanto o TreeSet são seguros quando lidos, mesmo simultaneamente, por vários threads. Portanto, se eles foram criados e preenchidos por um único thread (digamos, no início do programa) e somente então lidos, mas não modificados por vários threads, não há motivo para sincronização ou bloqueio.

No entanto, se lido e modificado simultaneamente, ou modificado simultaneamente por mais de um thread, a coleção pode gerar uma ConcurrentModificationException ou se comportar de forma inesperada. Nesses casos, é imperativo sincronizar/bloquear o acesso à coleção usando uma das seguintes abordagens:

  1. Usando Collections.synchronizedSorted..:

    SortedSet<Integer> set = Collections.synchronizedSortedSet(new TreeSet<Integer>());
SortedMap<Integer,String> map = Collections.synchronizedSortedMap(new TreeMap<Integer,String>());

Isso fornecerá uma implementação SortedSet/SortedMap apoiada pela coleção real e sincronizada em algum objeto mutex. Observe que isso sincronizará todo o acesso de leitura e gravação à coleção em um único bloqueio, portanto, mesmo leituras simultâneas não seriam possíveis.

  1. Sincronizando manualmente em algum objeto, como a própria coleção:

     TreeSet<Integer> set = new TreeSet<>();

    …

    //Thread 1
synchronized (set) {
    set.add(4);
}

    …

    //Thread 2
synchronized (set) {
    set.remove(5);
}

  2. Usando um bloqueio, como um ReentrantReadWriteLock:

     TreeSet<Integer> set = new TreeSet<>(); 
 ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

    …

     //Thread 1
 lock.writeLock().lock();
 set.add(4);
 lock.writeLock().unlock();

    …

     //Thread 2
 lock.readLock().lock();
 set.contains(5);
 lock.readLock().unlock();

Ao contrário dos métodos de sincronização anteriores, usar um ReadWriteLock permite que vários threads leiam do mapa simultaneamente.
