Este tópico descreve uma série de “armadilhas” (ou seja, erros que programadores java iniciantes cometem) relacionadas ao desempenho do aplicativo Java.

Este tópico descreve algumas práticas de codificação Java “micro” que são ineficientes. Na maioria dos casos, as ineficiências são relativamente pequenas, mas ainda vale a pena evitá-las se possível.

Armadilha - A concatenação de strings em um loop não é dimensionada #

Considere o código a seguir como uma ilustração:

public String joinWords(List<String> words) {
    String message = "";
    for (String word : words) {
        message = message + " " + word;
    }
    return message;
}

Infelizmente este código é ineficiente se a lista de palavras for longa. A raiz do problema é esta afirmação:

message = message + " " + word;

Para cada iteração de loop, esta instrução cria uma nova string message contendo uma cópia de todos os caracteres da string message original com caracteres extras anexados a ela. Isso gera muitas strings temporárias e faz muitas cópias.

Quando analisamos joinWords, assumindo que existem N palavras com um comprimento médio de M, descobrimos que O(N) strings temporárias são criadas e O(M.N2) caracteres serão copiados no processo. O componente N2 é particularmente preocupante.

A abordagem recomendada para esse tipo de problema1 é usar um StringBuilder em vez da concatenação de strings da seguinte forma:

public String joinWords2(List<String> words) {
    StringBuilder message = new StringBuilder();
    for (String word : words) {
        message.append(" ").append(word);
    }
    return message.toString();
}

A análise de joinWords2 precisa levar em conta as despesas gerais de “aumentar” o array de apoio StringBuilder que contém os caracteres do construtor. No entanto, verifica-se que o número de novos objetos criados é O(logN) e que o número de caracteres copiados é O(M.N) caracteres. O último inclui caracteres copiados na chamada final toString().

(Pode ser possível ajustar isso ainda mais, criando o StringBuilder com a capacidade correta para começar. No entanto, a complexidade geral permanece a mesma.)

Voltando ao método joinWords original, verifica-se que a declaração crítica será otimizada por um compilador Java típico para algo assim:

  StringBuilder tmp = new StringBuilder();
  tmp.append(message).append(" ").append(word);
  message = tmp.toString();

No entanto, o compilador Java não “irá” o StringBuilder para fora do loop, como fizemos manualmente no código para joinWords2.

Referência:

  • “O operador String ‘+’ do Java está em um loop lento?”

1 - No Java 8 e posterior, a classe Joiner pode ser usada para resolver esse problema específico. No entanto, não é sobre isso que este exemplo realmente deveria tratar.

Armadilha - Usar size() para testar se uma coleção está vazia é ineficiente. #

O Java Collections Framework fornece dois métodos relacionados para todos os objetos Collection:

  • size() retorna o número de entradas em uma Collection, e
  • O método isEmpty() retorna true se (e somente se) o Coleção está vazio.

Ambos os métodos podem ser usados ​​para testar a vacuidade da coleção. Por exemplo:

Collection<String> strings = new ArrayList<>();
boolean isEmpty_wrong = strings.size() == 0; // Avoid this
boolean isEmpty = strings.isEmpty();         // Best

Embora essas abordagens pareçam iguais, algumas implementações de coleção não armazenam o tamanho. Para tal coleção, a implementação de size() precisa calcular o tamanho cada vez que é chamada. Por exemplo:

  • Uma simples classe de lista vinculada (mas não a java.util.LinkedList) pode precisar percorrer a lista para contar os elementos.
  • A classe ConcurrentHashMap precisa somar as entradas em todos os “segmentos” do mapa.
  • Uma implementação lenta de uma coleção pode precisar realizar a coleção inteira na memória para contar os elementos.

Por outro lado, um método isEmpty() só precisa testar se há pelo menos um elemento na coleção. Isso não implica contar os elementos.

Embora size() == 0 nem sempre seja menos eficiente que isEmpty(), é inconcebível que um isEmpty() implementado corretamente seja menos eficiente que size() == 0. Portanto, isEmpty() é preferido.

Armadilha - Usar ’novo’ para criar instâncias de wrapper primitivas é ineficiente #

A linguagem Java permite que você use new para criar instâncias Integer, Boolean e assim por diante, mas geralmente é uma má idéia. É melhor usar o autoboxing (Java 5 e posterior) ou o método valueOf.

 Integer i1 = new Integer(1);      // BAD
 Integer i2 = 2;                   // BEST (autoboxing)
 Integer i3 = Integer.valueOf(3);  // OK

A razão pela qual usar new Integer(int) explicitamente é uma má ideia é que ele cria um novo objeto (a menos que otimizado pelo compilador JIT). Por outro lado, quando o autoboxing ou uma chamada valueOf explícita são usados, o Java runtime tentará reutilizar um objeto Integer de um cache de objetos pré-existentes. Cada vez que o runtime tem um “hit” de cache, ele evita a criação de um objeto. Isso também economiza memória heap e reduz as sobrecargas de GC causadas pela rotatividade de objetos.

Notas:

  1. Em implementações Java recentes, o autoboxing é implementado chamando valueOf, e existem caches para Boolean, Byte, Short, Integer, Long e Character.
  2. O comportamento de armazenamento em cache para os tipos integrais é exigido pela Java Language Specification.

Pitfall - Internar strings para que você possa usar == é uma má ideia #

Quando alguns programadores veem este conselho:

“Testar strings usando == está incorreto (a menos que as strings sejam internas)”

sua reação inicial é internar strings para que possam usar ==. (Afinal, == é mais rápido do que chamar String.equals(...), não é.)

Esta é a abordagem errada, de várias perspectivas:

Fragilidade #

Em primeiro lugar, você só pode usar == com segurança se souber que todos os objetos String que você está testando foram internados. O JLS garante que os literais String em seu código-fonte serão internados. No entanto, nenhuma das APIs padrão do Java SE garante o retorno de strings internas, além do próprio String.intern(String). Se você perder apenas uma fonte de objetos String que não foram internados, seu aplicativo não será confiável. Essa falta de confiabilidade se manifestará como falsos negativos em vez de exceções, o que pode dificultar a detecção.

Custos de usar ‘intern()’ #

Sob o capô, o internamento funciona mantendo uma tabela de hash que contém objetos String previamente internados. Algum tipo de mecanismo de referência fraco é usado para que a tabela de hash interna não se torne um vazamento de armazenamento. Enquanto a tabela de hash é implementada em código nativo (ao contrário de HashMap, HashTable e assim por diante), as chamadas intern ainda são relativamente caras em termos de CPU e memória usada.

Este custo deve ser comparado com a economia que obteremos usando == em vez de equals. Na verdade, não vamos atingir o ponto de equilíbrio a menos que cada string interna seja comparada com outras strings “algumas” vezes.

(Além disso: as poucas situações em que o internamento vale a pena tendem a reduzir a pegada de memória de um aplicativo onde as mesmas strings se repetem muitas vezes, e essas strings têm uma longa vida útil.)

O impacto na coleta de lixo #

Além dos custos diretos de CPU e memória descritos acima, as Strings internas impactam no desempenho do coletor de lixo.

Para versões do Java anteriores ao Java 7, as strings internas são mantidas no espaço “PermGen”, que é coletado com pouca frequência. Se PermGen precisar ser coletado, isso (normalmente) acionará uma coleta de lixo completa. Se o espaço PermGen for totalmente preenchido, a JVM travará, mesmo se houver espaço livre nos espaços de heap regulares.

No Java 7, o pool de strings foi movido de “PermGen” para o heap normal. No entanto, a tabela de hash ainda será uma estrutura de dados de longa duração, o que fará com que qualquer string interna tenha vida longa. (Mesmo que os objetos de string internos fossem alocados no espaço do Éden, eles provavelmente seriam promovidos antes de serem coletados.)

Assim, em todos os casos, internar uma string prolongará sua vida útil em relação a uma string comum. Isso aumentará as sobrecargas de coleta de lixo durante a vida útil da JVM.

O segundo problema é que a tabela de hash precisa usar um mecanismo de referência fraco de algum tipo para evitar que a memória interna da cadeia de caracteres vaze. Mas esse mecanismo é mais trabalhoso para o coletor de lixo.

Essas sobrecargas de coleta de lixo são difíceis de quantificar, mas há pouca dúvida de que elas existem. Se você usar intern extensivamente, eles podem ser significativos.

O tamanho da tabela de hash do pool de strings #

De acordo com esta fonte, a partir do Java 6, o pool de strings é implementado como uma tabela de hash de tamanho fixo com chains para lidar com strings que fazem hash para o mesmo bucket. Nas primeiras versões do Java 6, a tabela de hash tinha um tamanho constante (com fio). Um parâmetro de ajuste (-XX:StringTableSize) foi adicionado como uma atualização de meia-vida para Java 6. Então, em uma atualização de meia-vida para Java 7, o tamanho padrão do pool foi alterado de 1009 para 60013 .

A linha inferior é que se você pretende usar intern intensivamente em seu código, é aconselhável escolher uma versão do Java onde o tamanho da tabela de hash seja ajustável e certifique-se de ajustar o tamanho adequadamente. Caso contrário, o desempenho do intern pode degradar à medida que o pool aumenta.

Internação como um potencial vetor de negação de serviço #

O algoritmo hashcode para strings é bem conhecido. Se você internar strings fornecidas por usuários ou aplicativos mal-intencionados, isso pode ser usado como parte de um ataque de negação de serviço (DoS). Se o agente malicioso fizer com que todas as strings que ele fornece tenham o mesmo código de hash, isso pode levar a uma tabela de hash desbalanceada e a um desempenho O(N) para intern … onde N é o número de colisões cordas.

(Existem maneiras mais simples/eficazes de lançar um ataque DoS contra um serviço. No entanto, esse vetor pode ser usado se o objetivo do ataque DoS for quebrar a segurança ou fugir das defesas DoS de primeira linha.)

Armadilha - Preocupações de eficiência com expressões regulares #

A correspondência de expressões regulares é uma ferramenta poderosa (em Java e em outros contextos), mas tem algumas desvantagens. Uma delas é que as expressões regulares tendem a ser bastante caras.

As instâncias de padrão e correspondência devem ser reutilizadas #

Considere o seguinte exemplo:

/**
 * Test if all strings in a list consist of English letters and numbers.
 * @param strings the list to be checked
 * @return 'true' if an only if all strings satisfy the criteria
 * @throws NullPointerException if 'strings' is 'null' or a 'null' element.
 */
public boolean allAlphanumeric(List<String> strings) {
    for (String s : strings) {
        if (!s.matches("[A-Za-z0-9]*")) {
            return false;
        }  
    }
    return true;
}

Este código está correto, mas é ineficiente. O problema está na chamada matches(...). Nos bastidores, s.matches("[A-Za-z0-9]*") é equivalente a isso:

Pattern.matches(s, "[A-Za-z0-9]*")

que por sua vez é equivalente a

Pattern.compile("[A-Za-z0-9]*").matcher(s).matches()

A chamada Pattern.compile("[A-Za-z0-9]*") analisa a expressão regular, analisa-a e constrói um objeto Pattern que contém a estrutura de dados que será usada pelo mecanismo regex. Este é um cálculo não trivial. Em seguida, um objeto Matcher é criado para envolver o argumento s. Finalmente, chamamos match() para fazer a correspondência de padrões real.

O problema é que esse trabalho é todo repetido para cada iteração do loop. A solução é reestruturar o código da seguinte forma:

private static Pattern ALPHA_NUMERIC = Pattern.compile("[A-Za-z0-9]*");

public boolean allAlphanumeric(List<String> strings) {
    Matcher matcher = ALPHA_NUMERIC.matcher("");
    for (String s : strings) {
        matcher.reset(s);
        if (!matcher.matches()) {
            return false;
        }  
    }
    return true;
}

Observe que o javadoc para Pattern afirma:

As instâncias desta classe são imutáveis ​​e seguras para uso por vários threads simultâneos. Instâncias da classe Matcher não são seguras para tal uso.

Não use match() quando você deveria usar find() #

Suponha que você queira testar se uma string s contém três ou mais dígitos em uma linha. Você pode expressar isso de várias maneiras, incluindo:

  if (s.matches(".*[0-9]{3}.*")) {
      System.out.println("matches");
  }

ou

  if (Pattern.compile("[0-9]{3}").matcher(s).find()) {
      System.out.println("matches");
  }

O primeiro é mais conciso, mas também provavelmente menos eficiente. À primeira vista, a primeira versão tentará combinar a string inteira com o padrão. Além disso, como “.*” é um padrão “ganancioso”, o correspondente de padrão provavelmente avançará “avidamente” até o final da string e retrocederá até encontrar uma correspondência.

Por outro lado, a segunda versão pesquisará da esquerda para a direita e parará de pesquisar assim que encontrar os 3 dígitos seguidos.

Use alternativas mais eficientes para expressões regulares

As expressões regulares são uma ferramenta poderosa, mas não devem ser sua única ferramenta. Muitas tarefas podem ser feitas com mais eficiência de outras maneiras. Por exemplo:

 Pattern.compile("ABC").matcher(s).find()

faz a mesma coisa que:

 s.contains("ABC")

exceto que o último é muito mais eficiente. (Mesmo que você possa amortizar o custo de compilar a expressão regular.)

Muitas vezes, a forma não regex é mais complicada. Por exemplo, o teste realizado pela chamada matches() do método anterior allAlplanumeric pode ser reescrito como:

 public boolean matches(String s) {
     for (char c : s) {
         if ((c >= 'A' && c <= 'Z') ||
             (c >= 'a' && c <= 'z') ||
             (c >= '0' && c <= '9')) {
              return false;
         }
     }
     return true;
 }

Agora isso é mais código do que usar um Matcher, mas também será significativamente mais rápido.

Retrocesso catastrófico #

(Isso é potencialmente um problema com todas as implementações de expressões regulares, mas vamos mencioná-lo aqui porque é uma armadilha para o uso de Padrão.)

Considere este exemplo (artificial):

Pattern pat = Pattern.compile("(A+)+B");
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB").matches());
System.out.println(pat.matcher("AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAC").matches());

A primeira chamada println imprimirá rapidamente true. O segundo imprimirá false. Eventualmente. De fato, se você experimentar o código acima, verá que cada vez que adicionar um ‘A’ antes do ‘C’, o tempo gasto dobrará.

Este comportamento é um exemplo de retrocesso catastrófico. O mecanismo de correspondência de padrões que implementa a correspondência de regex está tentando infrutíferamente todas as maneiras possíveis que o padrão pode corresponder.

Vejamos o que (A+)+B realmente significa. Superficialmente, parece dizer “um ou mais caracteres ‘A’ seguidos por um valor ‘B’”, mas na realidade diz um ou mais grupos, cada um dos quais consiste em um ou mais caracteres ‘A’. Assim, por exemplo:

  • ‘AB’ corresponde apenas de uma maneira: ‘(A)B’
  • ‘AAB’ corresponde de duas maneiras: ‘(AA)B’ ou ‘(A)(A)B`
  • ‘AAAB’ corresponde a quatro formas: ‘(AAA)B’ ou ‘(AA)(A)B ou '(A)(AA)B ou ‘(A)(A)(A)B`
  • e assim por diante

Em outras palavras, o número de correspondências possíveis é 2N onde N é o número de caracteres A.

O exemplo acima é claramente planejado, mas os padrões que exibem esse tipo de características de desempenho (ou seja, O(2^N) ou O(N^K) para um K grande) surgem frequentemente quando expressões regulares mal consideradas são usados. Existem alguns remédios padrão:

  • Evite aninhar padrões de repetição dentro de outros padrões de repetição.
  • Evite usar muitos padrões de repetição.
  • Use a repetição sem retrocesso conforme apropriado.
  • Não use regexes para tarefas de análise complicadas. (Escreva um analisador adequado em vez disso.)

Por fim, tome cuidado com situações em que um usuário ou um cliente de API possa fornecer uma string regex com características patológicas. Isso pode levar a uma “negação de serviço” acidental ou deliberada.

  • A tag Expressões regulares, especialmente https://www.wikiod.com/pt/regex/retrocedendo e https://www.wikiod.com/pt/regex/quando-voce-nao-deve-usar-expressoes-regulares deve-não-usar-expressões-regulares#t=201610010339593564913
  • “Desempenho Regex” por Jeff Atwood.
  • “Como matar Java com uma expressão regular” por Andreas Haufler.

Armadilha - Pequenas leituras/gravações em fluxos sem buffer são ineficientes #

Considere o seguinte código para copiar um arquivo para outro:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new FileInputStream(args[0]);
             OutputStream os = new FileOutputStream(args[1])) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

(Deliberamos omitir a verificação normal de argumentos, relatórios de erros e assim por diante porque eles não são relevantes para o ponto deste exemplo.)

Se você compilar o código acima e usá-lo para copiar um arquivo enorme, notará que ele é muito lento. Na verdade, será pelo menos algumas ordens de magnitude mais lenta do que os utilitários padrão de cópia de arquivos do sistema operacional.

(Adicione medições de desempenho reais aqui!)

A principal razão pela qual o exemplo acima é lento (no caso de arquivo grande) é que ele está executando leituras de um byte e gravações de um byte em fluxos de bytes sem buffer. A maneira simples de melhorar o desempenho é agrupar os fluxos com fluxos em buffer. Por exemplo:

import java.io.*;

public class FileCopy {

    public static void main(String[] args) throws Exception {
        try (InputStream is = new BufferedInputStream(
                     new FileInputStream(args[0]));
             OutputStream os = new BufferedOutputStream(
                     new FileOutputStream(args[1]))) {
           int octet;
           while ((octet = is.read()) != -1) {
               os.write(octet);
           }
        }
    }
}

Essas pequenas alterações melhorarão a taxa de cópia de dados em pelo menos algumas ordens de magnitude, dependendo de vários fatores relacionados à plataforma. Os wrappers de fluxo em buffer fazem com que os dados sejam lidos e gravados em partes maiores. As instâncias têm buffers implementados como arrays de bytes.

  • Com is, os dados são lidos do arquivo no buffer alguns kilobytes de cada vez. Quando read() é chamado, a implementação normalmente retornará um byte do buffer. Ele só lerá do fluxo de entrada subjacente se o buffer tiver sido esvaziado.

  • O comportamento para os é análogo. Chamadas para os.write(int) gravam bytes únicos no buffer. Os dados só são gravados no fluxo de saída quando o buffer está cheio ou quando os é liberado ou fechado.

E quanto aos fluxos baseados em personagens? #

Como você deve estar ciente, Java I/O fornece diferentes APIs para leitura e gravação de dados binários e de texto.

  • InputStream e OutputStream são as APIs básicas para E/S binária baseada em fluxo
  • Reader e Writer são as APIs básicas para E/S de texto baseado em fluxo.

Para E/S de texto, BufferedReader e BufferedWriter são os equivalentes para BufferedInputStream e BufferedOutputStream.

Por que os fluxos em buffer fazem tanta diferença? #

A verdadeira razão pela qual os fluxos em buffer ajudam no desempenho tem a ver com a maneira como um aplicativo se comunica com o sistema operacional:

  • O método Java em um aplicativo Java ou as chamadas de procedimento nativo nas bibliotecas de tempo de execução nativas da JVM são rápidas. Eles normalmente recebem algumas instruções de máquina e têm um impacto mínimo no desempenho.

  • Por outro lado, as chamadas de tempo de execução da JVM para o sistema operacional não são rápidas. Eles envolvem algo conhecido como “syscall”. O padrão típico para uma syscall é o seguinte:

    1. Put the syscall arguments into registers.
    2. Execute a SYSENTER trap instruction.
    3. The trap handler switched to privileged state and changes the virtual memory mappings. Then it dispatches to the code to handle the specific syscall.
    4. The syscall handler checks the arguments, taking care that it isn’t being told to access memory that the user process should not see.
    5. The syscall specific work is performed. In the case of a read syscall, this may involve:
      1. checking that there is data to be read at the file descriptor’s current position
      2. calling the file system handler to fetch the required data from disk (or wherever it is stored) into the buffer cache,
      3. copying data from the buffer cache to the JVM-supplied address
      4. adjusting thstream pointerse file descriptor position
    6. Return from the syscall. This entails changing VM mappings again and switching out of privileged state.

Como você pode imaginar, executar um único syscall pode conter milhares de instruções de máquina. Conservadoramente, pelo menos duas ordens de grandeza a mais do que uma chamada de método regular. (Provavelmente três ou mais.)

Diante disso, a razão pela qual os fluxos em buffer fazem uma grande diferença é que eles reduzem drasticamente o número de syscalls. Em vez de fazer uma syscall para cada chamada read(), o fluxo de entrada em buffer lê uma grande quantidade de dados em um buffer conforme necessário. A maioria das chamadas read() no fluxo em buffer fazem algumas verificações simples de limites e retornam um byte que foi lido anteriormente. Raciocínio semelhante se aplica no caso do fluxo de saída e também nos casos do fluxo de caracteres.

(Algumas pessoas pensam que o desempenho de E/S em buffer vem da incompatibilidade entre o tamanho da solicitação de leitura e o tamanho de um bloco de disco, latência rotacional de disco e coisas assim. Na verdade, um sistema operacional moderno usa várias estratégias para garantir que o aplicativo normalmente não precisa esperar pelo disco. Esta não é a explicação real.)

Os fluxos em buffer são sempre uma vitória? #

Nem sempre. Streams em buffer são definitivamente uma vitória se o seu aplicativo for fazer muitas leituras ou gravações “pequenas”. No entanto, se o seu aplicativo precisar apenas realizar grandes leituras ou gravações em/de um grande byte[] ou char[], os fluxos em buffer não lhe trarão benefícios reais. Na verdade, pode até haver uma (pequena) penalidade de desempenho.

Esta é a maneira mais rápida de copiar um arquivo em Java? #

Não, não é. Ao usar as APIs baseadas em fluxo do Java para copiar um arquivo, você incorre no custo de pelo menos uma cópia extra de memória para memória dos dados. É possível evitar isso se você usar as APIs ByteBuffer e Channel do NIO. (Adicione um link para um exemplo separado aqui.)

Armadilha - As despesas gerais da criação de mensagens de log #

Os níveis de log TRACE e DEBUG existem para transmitir detalhes altos sobre a operação do código fornecido em tempo de execução. Definir o nível de log acima desses geralmente é recomendado, no entanto, alguns cuidados devem ser tomados para que essas instruções não afetem o desempenho, mesmo quando aparentemente “desativadas”.

Considere esta declaração de log:

// Processing a request of some kind, logging the parameters
LOG.debug("Request coming from " + myInetAddress.toString() 
          + " parameters: " + Arrays.toString(veryLongParamArray));

Mesmo quando o nível de log é definido como INFO, os argumentos passados ​​para debug() serão avaliados em cada execução da linha. Isso faz com que consuma desnecessariamente em vários aspectos:

  • Concatenação String: múltiplas instâncias String serão criadas
  • InetAddress pode até fazer uma pesquisa de DNS.
  • o veryLongParamArray pode ser muito longo - criar uma String a partir dele consome memória, leva tempo

Solução #

A maioria das estruturas de log fornece meios para criar mensagens de log usando strings de correção e referências de objeto. A mensagem de log será avaliada somente se a mensagem estiver realmente registrada. Exemplo:

// No toString() evaluation, no string concatenation if debug is disabled
LOG.debug("Request coming from {} parameters: {}", myInetAddress, parameters));

Isso funciona muito bem desde que todos os parâmetros possam ser convertidos em strings usando String.valueOf(Object). Se o cálculo da mensagem de log for mais complexo, o nível de log pode ser verificado antes do log:

if (LOG.isDebugEnabled()) {
    // Argument expression evaluated only when DEBUG is enabled
    LOG.debug("Request coming from {}, parameters: {}", myInetAddress,
              Arrays.toString(veryLongParamArray);
}

Aqui, LOG.debug() com o caro cálculo Arrays.toString(Obect[]) é processado somente quando DEBUG está realmente habilitado.

Armadilha - Chamar ’new String(String)’ é ineficiente #

Usar new String(String) para duplicar uma string é ineficiente e quase sempre desnecessário.

  • Os objetos String são imutáveis, portanto, não há necessidade de copiá-los para proteger contra alterações.
  • Em algumas versões mais antigas do Java, objetos String podem compartilhar arrays de apoio com outros objetos String. Nessas versões, é possível vazar memória criando uma substring (pequena) de uma string (grande) e retendo-a. No entanto, do Java 7 em diante, os arrays de apoio String não são compartilhados.

Na ausência de qualquer benefício tangível, chamar new String(String) é simplesmente um desperdício:

  • Fazer a cópia leva tempo de CPU.
  • A cópia usa mais memória, o que aumenta o espaço de memória do aplicativo e/ou aumenta as despesas gerais do GC.
  • Operações como equals(Object) e hashCode() podem ser mais lentas se objetos String forem copiados.

Armadilha - Chamar System.gc() é ineficiente #

É (quase sempre) uma má idéia chamar System.gc().

O javadoc para o método gc() especifica o seguinte:

“Chamar o método gc sugere que a Java Virtual Machine despenda esforço para reciclar objetos não utilizados para tornar a memória que ocupam atualmente disponível para reutilização rápida. Quando o controle retorna da chamada do método, a Java Virtual Machine fez uma melhor esforço para recuperar o espaço de todos os objetos descartados.”

Há alguns pontos importantes que podem ser extraídos disso:

  1. O uso da palavra “sugerir” em vez de (digamos) “informar” significa que a JVM está livre para ignorar a sugestão. O comportamento padrão da JVM (lançamentos recentes) é seguir a sugestão, mas isso pode ser substituído configurando -XX:+DisableExplicitGC ao iniciar a JVM.

  2. A frase “um melhor esforço para recuperar espaço de todos os objetos descartados” implica que chamar gc acionará uma coleta de lixo “completa”.

Então, por que chamar System.gc() é uma má ideia?

Primeiro, executar uma coleta de lixo completa é caro. Um GC completo envolve visitar e “marcar” cada objeto que ainda pode ser alcançado; ou seja, todo objeto que não é lixo. Se você acionar isso quando não houver muito lixo a ser coletado, o GC fará muito trabalho por relativamente pouco benefício.

Em segundo lugar, uma coleta de lixo completa pode perturbar as propriedades de “localidade” dos objetos que não são coletados. Objetos que são alocados pelo mesmo thread aproximadamente ao mesmo tempo tendem a ser alocados juntos na memória. Isso é bom. Objetos que são alocados ao mesmo tempo provavelmente estão relacionados; ou seja, referenciar um ao outro. Se seu aplicativo usa essas referências, é provável que o acesso à memória seja mais rápido devido a vários efeitos de cache de memória e página. Infelizmente, uma coleta de lixo completa tende a mover objetos de modo que os objetos que antes estavam próximos agora estão mais distantes.

Terceiro, a execução de uma coleta de lixo completa pode fazer seu aplicativo pausar até que a coleta seja concluída. Enquanto isso estiver acontecendo, seu aplicativo não responderá.

Na verdade, a melhor estratégia é deixar a JVM decidir quando executar o GC e que tipo de coleção executar. Se você não interferir, a JVM escolherá um horário e um tipo de coleta que otimize o rendimento ou minimize os tempos de pausa do GC.

No início dissemos “… (quase sempre) uma má ideia…”. Na verdade, existem alguns cenários em que pode ser uma boa ideia:

  1. Se você estiver implementando um teste de unidade para algum código que é sensível à coleta de lixo (por exemplo, algo envolvendo finalizadores ou referências fracas/soft/fantasmas), então chamar System.gc() pode ser necessário.

  2. Em alguns aplicativos interativos, pode haver pontos específicos no tempo em que o usuário não se importará se houver uma pausa na coleta de lixo. Um exemplo é um jogo onde há pausas naturais no “jogo”; por exemplo. ao carregar um novo nível.

Armadilha - O uso excessivo de tipos de wrapper primitivos é ineficiente #

Considere estes dois pedaços de código:

int a = 1000;
int b = a + 1;

e

Integer a = 1000;
Integer b = a + 1;

Pergunta: Qual versão é mais eficiente?

Resposta: As duas versões parecem quase idênticas, mas a primeira versão é muito mais eficiente que a segunda.

A segunda versão está usando uma representação para os números que usa mais espaço e conta com auto-boxing e auto-unboxing nos bastidores. Na verdade, a segunda versão é diretamente equivalente ao seguinte código:

Integer a = Integer.valueOf(1000);               // box 1000
Integer b = Integer.valueOf(a.intValue() + 1);   // unbox 1000, add 1, box 1001

Comparando isso com a outra versão que usa int, existem claramente três chamadas de métodos extras quando Integer é usado. No caso de valueOf, as chamadas vão criar e inicializar um novo objeto Integer. Todo esse trabalho extra de boxing e unboxing provavelmente tornará a segunda versão uma ordem de magnitude mais lenta que a primeira.

Além disso, a segunda versão está alocando objetos no heap em cada chamada valueOf. Embora a utilização do espaço seja específica da plataforma, é provável que esteja na região de 16 bytes para cada objeto ‘Integer’. Por outro lado, a versão int precisa de zero espaço de heap extra, assumindo que a e b são variáveis ​​locais.

Outra grande razão pela qual os primitivos são mais rápidos que seus equivalentes em caixa é como seus respectivos tipos de array são dispostos na memória.

Se você tomar int[] e Integer[] como exemplo, no caso de um int[] os values int são dispostos de forma contígua na memória. Mas no caso de um Integer[] não são os valores que são definidos, mas referências (ponteiros) para objetos Integer, que por sua vez contêm os valores int reais.

Além de ser um nível extra de indireção, isso pode ser um grande tanque quando se trata de localidade de cache ao iterar sobre os valores. No caso de um int[] a CPU poderia buscar todos os valores no array, em seu cache de uma vez, porque eles são contíguos na memória. Mas no caso de um Integer[] a CPU potencialmente tem que fazer uma busca de memória adicional para cada elemento, já que o array contém apenas referências aos valores reais.

Em resumo, usar tipos de wrapper primitivos é relativamente caro em recursos de CPU e memória. Usá-los desnecessariamente é eficiente.

Armadilha - Iterar as chaves de um mapa pode ser ineficiente #

O código de exemplo a seguir é mais lento do que precisa:

Map<String, String> map = new HashMap<>(); 
for (String key : map.keySet()) {
    String value = map.get(key);
    // Do something with key and value
}

Isso porque requer uma pesquisa de mapa (o método get()) para cada chave no mapa. Essa pesquisa pode não ser eficiente (em um HashMap, ela envolve chamar hashCode na chave, depois procurar o bucket correto nas estruturas de dados internas e às vezes até chamar equals). Em um mapa grande, isso pode não ser uma sobrecarga trivial.

A maneira correta de evitar isso é iterar nas entradas do mapa, que é detalhado no [tópico de coleções](https://www.wikiod.com/pt/java/colecoes#Iterando sobre coleções 201608260922476000177)
