Herança é um recurso básico orientado a objetos no qual uma classe adquire e estende as propriedades de outra classe, usando a palavra-chave extends. Para Interfaces e a palavra-chave implements, veja interfaces.

Sintaxe #

  • classe ClassB estende ClassA {…}
  • classe ClassB implementa InterfaceA {…}
  • interface InterfaceB estende InterfaceA {…}
  • classe ClassB estende ClassA implementa InterfaceC, InterfaceD {…}
  • classe abstrata AbstractClassB estende ClassA {…}
  • classe abstrata AbstractClassB estende AbstractClassA {…}
  • classe abstrata AbstractClassB estende ClassA implementa InterfaceC, InterfaceD {…}

A herança geralmente é combinada com genéricos para que a classe base tenha um ou mais parâmetros de tipo. Consulte Criando uma Classe Genérica.

1: https://www.wikiod.com/pt/java/genericos#Criando uma classe genérica

Aulas de resumo #

Uma classe abstrata é uma classe marcada com a palavra-chave abstract. Ela, ao contrário da classe não-abstrata, pode conter métodos abstratos - sem implementação -. No entanto, é válido criar uma classe abstrata sem métodos abstratos.

Uma classe abstrata não pode ser instanciada. Ele pode ser subclassificado (estendido) desde que a subclasse também seja abstrata ou implemente todos os métodos marcados como abstratos por superclasses.

Um exemplo de uma classe abstrata:

public abstract class Component {
    private int x, y;
    
    public setPosition(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public abstract void render();
}

A classe deve ser marcada como abstrata, quando possui pelo menos um método abstrato. Um método abstrato é um método que não tem implementação. Outros métodos podem ser declarados dentro de uma classe abstrata que tenha implementação para fornecer código comum para qualquer subclasse.

A tentativa de instanciar esta classe fornecerá um erro de compilação:

//error: Component is abstract; cannot be instantiated   
Component myComponent = new Component();

No entanto, uma classe que estende Component e fornece uma implementação para todos os seus métodos abstratos e pode ser instanciada.

public class Button extends Component {

    @Override
    public void render() {
        //render a button
    }
}

public class TextBox extends Component {

    @Override
    public void render() {
        //render a textbox
    }
}

Instâncias de classes herdadas também podem ser convertidas como a classe pai (herança normal) e fornecem um efeito polimórfico quando o método abstrato é chamado.

Component myButton = new Button();
Component myTextBox = new TextBox();

myButton.render(); //renders a button
myTextBox.render(); //renders a text box

Classes abstratas x Interfaces

As classes e interfaces abstratas fornecem uma maneira de definir assinaturas de método enquanto exigem que a classe de extensão/implementação forneça a implementação.

Existem duas diferenças principais entre classes abstratas e interfaces:

  • Uma classe só pode estender uma única classe, mas pode implementar muitas interfaces.
  • Uma classe abstrata pode conter campos de instância (não static), mas interfaces podem conter apenas campos static.

Métodos declarados em interfaces não podiam conter implementações, então classes abstratas eram usadas quando era útil fornecer métodos adicionais cujas implementações chamavam de métodos abstratos.

O Java 8 permite que as interfaces contenham métodos padrão, geralmente [implementados usando os outros métodos da interface](https:// stackoverflow.com/documentation/java/113/default-methods/456/accessing-other-interface-methods-within-default-method#t=201607191734172336331), tornando interfaces e classes abstratas igualmente poderosas a esse respeito.

Subclasses anônimas de classes abstratas

Como conveniência, java permite a instanciação de instâncias anônimas de subclasses de classes abstratas, que fornecem implementações para os métodos abstratos ao criar o novo objeto. Usando o exemplo acima, isso pode ficar assim:

Component myAnonymousComponent = new Component() {
    @Override
    public void render() {
        // render a quick 1-time use component
    }
}

Usando ‘final’ para restringir herança e substituição #

Aulas finais #

Quando usado em uma declaração class, o modificador final impede que outras classes sejam declaradas que estendem a classe. Uma classe final é uma classe “folha” na hierarquia de classes de herança.

// This declares a final class
final class MyFinalClass {
    /* some code */
}

// Compilation error: cannot inherit from final MyFinalClass
class MySubClass extends MyFinalClass {
    /* more code */
}

Casos de uso para classes finais #

As classes finais podem ser combinadas com um construtor private para controlar ou impedir a instanciação de uma classe. Isso pode ser usado para criar uma chamada “classe de utilitário” que define apenas membros estáticos; ou seja, constantes e métodos estáticos.

public final class UtilityClass {

    // Private constructor to replace the default visible constructor
    private UtilityClass() {}

    // Static members can still be used as usual
    public static int doSomethingCool() {
        return 123;
    }

}

Classes imutáveis ​​também devem ser declaradas como final. (Uma classe imutável é aquela cujas instâncias não podem ser alteradas após terem sido criadas; veja o tópico Iobjetos mutáveis. ) Ao fazer isso, você impossibilita a criação de uma subclasse mutável de uma classe imutável. Isso violaria o Princípio de Substituição de Liskov que exige que um subtipo obedeça ao “contrato comportamental” de seus supertipos.

De uma perspectiva prática, declarar uma classe imutável como ‘final’ torna mais fácil raciocinar sobre o comportamento do programa. Ele também aborda as preocupações de segurança no cenário em que o código não confiável é executado em um sandbox de segurança. (Por exemplo, uma vez que String é declarado como final, uma classe confiável não precisa se preocupar com a possibilidade de ser induzida a aceitar subclasse mutável, que o chamador não confiável poderia então alterar sub-repticiamente.)

Uma desvantagem das classes ‘finais’ é que elas não funcionam com alguns frameworks mocking como o Mockito. Atualização: Mockito versão 2 agora suporta simulação de classes finais.

Métodos finais #

O modificador final também pode ser aplicado a métodos para evitar que sejam substituídos em subclasses:

public class MyClassWithFinalMethod {

    public final void someMethod() {
    }
}

public class MySubClass extends MyClassWithFinalMethod {

    @Override
    public void someMethod() { // Compiler error (overridden method is final)
    }
}

Os métodos finais são normalmente usados ​​quando você deseja restringir o que uma subclasse pode alterar em uma classe sem proibir totalmente as subclasses.

O modificador final também pode ser aplicado a variáveis, mas o significado de final para variáveis ​​não está relacionado à herança.

Herança #

Com o uso da palavra-chave extends entre as classes, todas as propriedades da superclasse (também conhecida como Classe Pai ou Classe Base) estão presentes na subclasse (também conhecida como Classe Filho ou Classe Derivada) Classe*)

public class BaseClass {

    public void baseMethod(){
        System.out.println("Doing base class stuff");
    }
}

public class SubClass extends BaseClass {

}

Instâncias de SubClass herdaram o método baseMethod():

SubClass s = new SubClass();
s.baseMethod();  //Valid, prints "Doing base class stuff"

Conteúdo adicional pode ser adicionado a uma subclasse. Isso permite funcionalidade adicional na subclasse sem qualquer alteração na classe base ou em qualquer outra subclasse dessa mesma classe base:

public class Subclass2 extends BaseClass {

    public void anotherMethod() {
        System.out.println("Doing subclass2 stuff");
    }
}

Subclass2 s2 = new Subclass2();
s2.baseMethod(); //Still valid , prints "Doing base class stuff"
s2.anotherMethod(); //Also valid, prints "Doing subclass2 stuff"

Os campos também são herdados:

public class BaseClassWithField {

    public int x;

}

public class SubClassWithField extends BaseClassWithField {

    public SubClassWithField(int x) {
        this.x = x; //Can access fields
    }
}

Campos e métodos private ainda existem dentro da subclasse, mas não são acessíveis:

public class BaseClassWithPrivateField {

    private int x = 5;

    public int getX() {
        return x;
    }
}

public class SubClassInheritsPrivateField extends BaseClassWithPrivateField {

    public void printX() {
        System.out.println(x); //Illegal, can't access private field x
        System.out.println(getX()); //Legal, prints 5
    }
}

SubClassInheritsPrivateField s = new SubClassInheritsPrivateField();
int x = s.getX(); //x will have a value of 5.

Em Java, cada classe pode estender no máximo uma outra classe.

public class A{}
public class B{}
public class ExtendsTwoClasses extends A, B {} //Illegal

Isso é conhecido como herança múltipla e, embora seja legal em algumas linguagens, Java não permite isso com classes.

Como resultado disso, cada classe tem uma cadeia ancestral não ramificada de classes que leva a Object, da qual todas as classes descendem.

O Princípio da Substituição de Liskov #

A substituibilidade é um princípio na programação orientada a objetos introduzido por Barbara Liskov em uma palestra de conferência de 1987 afirmando que, se a classe B é uma subclasse da classe A, então onde quer que A` `` é esperado, B``` pode ser usado em vez disso:

class A {...}
class B extends A {...}

public void method(A obj) {...}

A a = new B(); // Assignment OK
method(new B()); // Passing as parameter OK

Isso também se aplica quando o tipo é uma interface, onde não há necessidade de nenhum relacionamento hierárquico entre os objetos:

interface Foo {
    void bar();
}

class A implements Foo {
    void bar() {...}
}

class B implements Foo {
    void bar() {...}
}

List<Foo> foos = new ArrayList<>();
foos.add(new A()); // OK
foos.add(new B()); // OK

Agora a lista contém objetos que não são da mesma hierarquia de classes.

Herança Estática #

O método estático pode ser herdado de forma semelhante aos métodos normais, no entanto, ao contrário dos métodos normais, é impossível criar métodos “abstract” para forçar a substituição do método estático. Escrever um método com a mesma assinatura de um método estático em uma superclasse parece ser uma forma de sobrescrever, mas na verdade isso simplesmente cria uma nova função ocultando a outra.

public class BaseClass {
    
    public static int num = 5;

    public static void sayHello() {
        System.out.println("Hello");
    }

    public static void main(String[] args) {
        BaseClass.sayHello();
        System.out.println("BaseClass's num: " + BaseClass.num);
            
        SubClass.sayHello();
        //This will be different than the above statement's output, since it runs
        //A different method
        SubClass.sayHello(true);
        
        StaticOverride.sayHello();
        System.out.println("StaticOverride's num: " + StaticOverride.num);
    }
}

public  class SubClass extends BaseClass {
    
    //Inherits the sayHello function, but does not override it   
    public static void sayHello(boolean test) {
        System.out.println("Hey");
    }
}

public static class StaticOverride extends BaseClass {

    //Hides the num field from BaseClass
    //You can even change the type, since this doesn't affect the signature
    public static String num = "test";
        
    //Cannot use @Override annotation, since this is static
    //This overrides the sayHello method from BaseClass
    public static void sayHello() {
        System.out.println("Static says Hi");
    }

}

A execução de qualquer uma dessas classes produz a saída:

Hello
BaseClass's num: 5
Hello
Hey
Static says Hi
StaticOverride's num: test

Observe que, diferentemente da herança normal, na herança estática os métodos não ficam ocultos. Você sempre pode chamar o método base sayHello usando BaseClass.sayHello(). Mas as classes herdam métodos estáticos se nenhum método com a mesma assinatura for encontrado na subclasse. Se as assinaturas de dois métodos variarem, ambos os métodos poderão ser executados a partir da subclasse, mesmo que o nome seja o mesmo.

Os campos estáticos se escondem de maneira semelhante.

Considere usar classes abstratas se…

  1. Você deseja compartilhar código entre várias classes intimamente relacionadas.
  2. Você espera que as classes que estendem sua classe abstrata tenham muitos métodos ou campos comuns, ou requeiram modificadores de acesso que não sejam públicos (como protected e private).
  3. Você deseja declarar campos não estáticos ou não finais.

Considere usar interfaces se…

  1. Você espera que classes não relacionadas implementem sua interface. Por exemplo, muitos objetos não relacionados podem implementar a interface Serializable.
  2. Você deseja especificar o comportamento de um determinado tipo de dados, mas não está preocupado com quem implementa seu comportamento.
  3. Você deseja aproveitar a herança múltipla de tipo.

Herança e métodos estáticos #

Em Java, as classes pai e filha podem ter métodos estáticos com o mesmo nome. Mas, nesses casos, a implementação do método estático no filho está ocultando classe pai’ implementação, não é sobreposição de método. Por exemplo:

class StaticMethodTest {

  // static method and inheritance
  public static void main(String[] args) {
    Parent p = new Child();
    p.staticMethod(); // prints Inside Parent
    ((Child) p).staticMethod(); // prints Inside Child
  }

  static class Parent {
    public static void staticMethod() {
      System.out.println("Inside Parent");
    }
  }

  static class Child extends Parent {
    public static void staticMethod() {
      System.out.println("Inside Child");
    }
  }
}

Métodos estáticos são vinculados a uma classe e não a uma instância e essa vinculação de método ocorre em tempo de compilação. Como na primeira chamada para staticMethod(), a referência de classe pai p foi usada, a versão Parent de staticMethod() é invocada. No segundo caso, nós convertemos p na classe Child, o staticMethod() de Child executado.

Estreitamento e ampliação de referências de objetos #

Casting uma instância de uma classe base para uma subclasse como em: b = (B) a; é chamado de estreitamento (já que você está tentando restringir o objeto de classe base para um objeto de classe mais específico) e precisa de um tipo explícito -elenco.

A conversão de uma instância de uma subclasse para uma classe base como em: A a = b; é chamada de alargamento e não precisa de conversão de tipo.

Para ilustrar, considere as seguintes declarações de classe e teste o código:

class Vehicle {
}

class Car extends Vehicle {
}

class Truck extends Vehicle {
}

class MotorCycle extends Vehicle {
}

class Test {

    public static void main(String[] args) {
    
        Vehicle vehicle = new Car();
        Car car = new Car();        
    
        vehicle = car; // is valid, no cast needed

        Car c = vehicle // not valid
        Car c = (Car) vehicle; //valid
    }
}

A instrução Vehicle vehicle = new Car(); é uma instrução Java válida. Cada instância de Car também é um Vehicle. Portanto, a atribuição é legal sem a necessidade de uma conversão de tipo explícita.

Por outro lado, Carro c = veículo; não é válido. O tipo estático da variável vehicle é Vehicle, o que significa que pode se referir a uma instância de Car, Truck, MotorCycleou qualquer outra subclasse atual ou futura deVehicle. (Ou, na verdade, uma instância do próprio Vehicle, já que não a declaramos como uma classe abstract.) A atribuição não pode ser permitida, pois isso pode levar a carreferindo-se a uma instânciaTruck`.

Para evitar essa situação, precisamos adicionar um typecast explícito:

Car c = (Car) vehicle;

O typecast diz ao compilador que esperamos que o valor de vehicle seja um Car ou uma subclasse de Car. Se necessário, o compilador inserirá código para executar uma verificação de tipo em tempo de execução. Se a verificação falhar, uma ClassCastException será lançada quando o código for executado.

Observe que nem todas as conversões de tipo são válidas. Por exemplo:

String s = (String) vehicle;  // not valid

O compilador Java sabe que uma instância que é compatível com o tipo Vehicle não pode nunca ser compatível com o tipo String. A conversão de tipo nunca poderia ser bem-sucedida, e o JLS exige que isso dê um erro de compilação.

Programação para uma interface #

A ideia por trás da programação para uma interface é basear o código principalmente em interfaces e usar apenas classes concretas no momento da instanciação. Nesse contexto, um bom código lidando com, por exemplo, As coleções Java serão parecidas com isto (não que o método em si seja útil, apenas ilustração):

public <T> Set<T> toSet(Collection<T> collection) {
  return Sets.newHashSet(collection);
}

enquanto o código ruim pode ser assim:

public <T> HashSet<T> toSet(ArrayList<T> collection) {
  return Sets.newHashSet(collection);
}

Não só o primeiro pode ser aplicado a uma escolha mais ampla de argumentos, seus resultados serão mais compatíveis com o código fornecido por outros desenvolvedores que geralmente aderem ao conceito de programação para uma interface. No entanto, as razões mais importantes para usar o primeiro são:

  • na maioria das vezes o contexto, no qual o resultado é utilizado, não precisa e não deve precisar de tantos detalhes quanto a implementação concreta proporciona;
  • aderir a uma interface força um código mais limpo e menos hacks, como outro método público é adicionado a uma classe que atende a algum cenário específico;
  • o código é mais testável, pois as interfaces são facilmente modificáveis;
  • finalmente, o conceito ajuda mesmo que apenas uma implementação seja esperada (pelo menos para testabilidade).

Então, como se pode aplicar facilmente o conceito de programação a uma interface ao escrever um novo código tendo em mente uma implementação específica? Uma opção que costumamos usar é uma combinação dos seguintes padrões:

  • programação para uma interface
  • fábrica
  • construtor

O exemplo a seguir baseado nesses princípios é uma versão simplificada e truncada de uma implementação de RPC escrita para vários protocolos diferentes:

public interface RemoteInvoker {
  <RQ, RS> CompletableFuture<RS> invoke(RQ request, Class<RS> responseClass);
}

A interface acima não deve ser instanciada diretamente por meio de uma fábrica, em vez disso, derivamos outras interfaces mais concretas, uma para invocação HTTP e outra para AMQP, cada uma tendo uma fábrica e um construtor para construir instâncias, que por sua vez também são instâncias de a interface acima:

public interface AmqpInvoker extends RemoteInvoker {
  static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
    return new AmqpInvokerBuilder(instanceId, factory);
  }
}

Instâncias de RemoteInvoker para uso com AMQP agora podem ser construídas tão facilmente quanto (ou mais envolvidas, dependendo do construtor):

RemoteInvoker invoker = AmqpInvoker.with(instanceId, factory)
  .requestRouter(router)
  .build();

E uma invocação de um pedido é tão fácil quanto:

Response res = invoker.invoke(new Request(data), Response.class).get();

Devido ao Java 8 permitir a colocação de métodos estáticos diretamente nas interfaces, a fábrica intermediária tornou-se implícita no código acima substituído por AmqpInvoker.with(). Em Java anterior à versão 8, o mesmo efeito pode ser obtido com uma classe interna Factory:

public interface AmqpInvoker extends RemoteInvoker {
  class Factory {
    public static AmqpInvokerBuilder with(String instanceId, ConnectionFactory factory) {
      return new AmqpInvokerBuilder(instanceId, factory);
    }
  }
}

A instanciação correspondente se transformaria em:

RemoteInvoker invoker = AmqpInvoker.Factory.with(instanceId, factory)
  .requestRouter(router)
  .build();

O construtor usado acima pode ficar assim (embora isso seja uma simplificação, pois o atual permite definir até 15 parâmetros desviando dos padrões). Observe que a construção não é pública, portanto, é especificamente utilizável apenas na interface AmqpInvoker acima:

public class AmqpInvokerBuilder {
  ...
  AmqpInvokerBuilder(String instanceId, ConnectionFactory factory) {
    this.instanceId = instanceId;
    this.factory = factory;
  }

  public AmqpInvokerBuilder requestRouter(RequestRouter requestRouter) {
    this.requestRouter = requestRouter;
    return this;
  }

  public AmqpInvoker build() throws TimeoutException, IOException {
    return new AmqpInvokerImpl(instanceId, factory, requestRouter);
  }
}

Geralmente, um construtor também pode ser gerado usando uma ferramenta como o FreeBuilder.

Finalmente, a implementação padrão (e a única esperada) dessa interface é definida como uma classe local do pacote para impor o uso da interface, da fábrica e do construtor:

class AmqpInvokerImpl implements AmqpInvoker {
  AmqpInvokerImpl(String instanceId, ConnectionFactory factory, RequestRouter requestRouter) {
    ...
  }

  @Override
  public <RQ, RS> CompletableFuture<RS> invoke(final RQ request, final Class<RS> respClass) {
    ...
  }
}

Enquanto isso, esse padrão provou ser muito eficiente no desenvolvimento de todo o nosso novo código, não importa quão simples ou complexa seja a funcionalidade.
