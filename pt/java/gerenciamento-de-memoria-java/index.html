Em Java, os objetos são alocados no heap e a memória do heap é recuperada pela coleta de lixo automática. Um programa aplicativo não pode excluir explicitamente um objeto Java.

Os princípios básicos da coleta de lixo Java são descritos no exemplo [Coleta de lixo][1]. Outros exemplos descrevem a finalização, como acionar o coletor de lixo manualmente e o problema de vazamentos de armazenamento.

[1]: https://www.wikiod.com/pt/java/gerenciamento-de-memoria-java#Coleta de lixo

Configurando os tamanhos de Heap, PermGen e Stack #

Quando uma máquina virtual Java é iniciada, ela precisa saber o tamanho do Heap e o tamanho padrão das pilhas de encadeamentos. Estes podem ser especificados usando opções de linha de comando no comando java. Para versões do Java anteriores ao Java 8, você também pode especificar o tamanho da região PermGen do Heap.

Observe que o PermGen foi removido no Java 8 e, se você tentar definir o tamanho do PermGen, a opção será ignorada (com uma mensagem de aviso).

Se você não especificar os tamanhos de Heap e Stack explicitamente, a JVM usará padrões que são calculados de uma maneira específica de versão e plataforma. Isso pode fazer com que seu aplicativo use pouca ou muita memória. Isso normalmente é bom para pilhas de threads, mas pode ser problemático para um programa que usa muita memória.

Definindo os tamanhos de Heap, PermGen e Stack padrão:

As seguintes opções da JVM definem o tamanho do heap:

  • -Xms<size> - define o tamanho inicial do heap
  • -Xmx<size> - define o tamanho máximo do heap
  • -XX:PermSize<size> - define o tamanho inicial do PermGen
  • -XX:MaxPermSize<size> - define o tamanho máximo de PermGen
  • -Xss<size> - define o tamanho padrão da pilha de threads

O parâmetro <size> pode ser um número de bytes, ou pode ter um sufixo de k, m ou g. O último especifica o tamanho em kilobytes, megabytes e gigabytes, respectivamente.

Exemplos:

$ java -Xms512m -Xmx1024m JavaApp
$ java -XX:PermSize=64m -XX:MaxPermSize=128m JavaApp
$ java -Xss512k JavaApp

Encontrando os tamanhos padrão:

A opção -XX:+printFlagsFinal pode ser usada para imprimir os valores de todos os sinalizadores antes de iniciar a JVM. Isso pode ser usado para imprimir os padrões para as configurações de tamanho de pilha e pilha da seguinte forma:

  • Para Linux, Unix, Solaris e Mac OSX

    $ java -XX:+PrintFlagsFinal -version | grep -iE ‘HeapSize|PermSize|ThreadStackSize’

  • Para Windows:

    java -XX:+PrintFlagsFinal -version | findstr /i “HeapSize PermSize ThreadStackSize”

A saída dos comandos acima será semelhante ao seguinte:

uintx InitialHeapSize                          := 20655360        {product}
uintx MaxHeapSize                              := 331350016       {product}
uintx PermSize                                  = 21757952        {pd product}
uintx MaxPermSize                               = 85983232        {pd product}
 intx ThreadStackSize                           = 1024            {pd product}

Os tamanhos são dados em bytes.

Coleta de lixo #

A abordagem C++ - novo e excluir #

Em uma linguagem como C++, o programa aplicativo é responsável por gerenciar a memória usada pela memória alocada dinamicamente. Quando um objeto é criado no heap C++ usando o operador new, é necessário que haja um uso correspondente do operador delete para descartar o objeto:

  • Se o programa esquecer de ’excluir’ um objeto e simplesmente “esquecer” dele, a memória associada será perdida para o aplicativo. O termo para essa situação é vazamento de memória, e se houver vazamento de memória demais, um aplicativo poderá usar mais e mais memória e, eventualmente, travar.

  • Por outro lado, se um aplicativo tentar ’excluir’ o mesmo objeto duas vezes, ou usar um objeto após ele ter sido excluído, o aplicativo poderá travar devido a problemas de corrupção de memória

Em um programa C++ complicado, implementar o gerenciamento de memória usando new e delete pode ser demorado. De fato, o gerenciamento de memória é uma fonte comum de bugs.

A abordagem Java - coleta de lixo #

Java tem uma abordagem diferente. Em vez de um operador delete explícito, Java fornece um mecanismo automático conhecido como coleta de lixo para recuperar a memória usada por objetos que não são mais necessários. O sistema de tempo de execução Java assume a responsabilidade de localizar os objetos a serem descartados. Essa tarefa é executada por um componente chamado coletor de lixo, ou GC, abreviado.

A qualquer momento durante a execução de um programa Java, podemos dividir o conjunto de todos os objetos existentes em dois subconjuntos distintos1:

  • Objetos alcançáveis ​​são definidos pelo JLS da seguinte forma:

    A reachable object is any object that can be accessed in any potential continuing computation from any live thread.

    In practice, this means that there is a chain of references starting from an in-scope local variable or a static variable by which some code might be able to reach the object.

  • Objetos inacessíveis são objetos que possivelmente não podem ser alcançados como acima.

Quaisquer objetos inacessíveis são elegíveis para coleta de lixo. Isso não significa que eles serão coletados como lixo. Na verdade:

  • Um objeto inacessível não é coletado imediatamente ao se tornar inacessível1.
  • Um objeto inalcançável pode nunca ser coletado como lixo.

A especificação de linguagem Java dá muita latitude a uma implementação de JVM para decidir quando coletar objetos inacessíveis. Ele também (na prática) dá permissão para uma implementação de JVM ser conservadora na forma como detecta objetos inacessíveis.

A única coisa que o JLS garante é que nenhum objeto alcançável será coletado como lixo.

O que acontece quando um objeto se torna inacessível #

Em primeiro lugar, nada acontece especificamente quando um objeto se torna inalcançável. As coisas só acontecem quando o coletor de lixo é executado e detecta que o objeto está inacessível. Além disso, é comum que uma execução de GC não detecte todos os objetos inacessíveis.

Quando o GC detecta um objeto inacessível, os seguintes eventos podem ocorrer.

  1. Se houver algum objeto Reference que faça referência ao objeto, essas referências serão apagadas antes que o objeto seja excluído.

  2. Se o objeto for finalizável, ele será finalizado. Isso acontece antes que o objeto seja excluído.

  3. O objeto pode ser excluído e a memória que ocupa pode ser recuperada.

Observe que há uma sequência clara na qual os eventos acima podem ocorrer, mas nada exige que o coletor de lixo execute a exclusão final de qualquer objeto específico em qualquer período de tempo específico.

Exemplos de objetos alcançáveis ​​e inalcançáveis #

Considere as seguintes classes de exemplo:

// A node in simple "open" linked-list.
public class Node {
    private static int counter = 0;

    public int nodeNumber = ++counter;
    public Node next;
}

public class ListTest {
    public static void main(String[] args) {
        test();                    // M1
        System.out.prinln("Done"); // M2
    }
    
    private static void test() {
        Node n1 = new Node();      // T1
        Node n2 = new Node();      // T2
        Node n3 = new Node();      // T3
        n1.next = n2;              // T4
        n2 = null;                 // T5
        n3 = null;                 // T6
    }
}

Vamos examinar o que acontece quando test() é chamado. As declarações T1, T2 e T3 criam objetos Node, e os objetos são todos alcançáveis ​​através das variáveis ​​n1, n2 e n3 respectivamente. A instrução T4 atribui a referência ao 2º objeto Node ao campo next do primeiro. Quando isso é feito, o 2º Nó é alcançável através de dois caminhos:

 n2 -> Node2
 n1 -> Node1, Node1.next -> Node2

Na instrução T5, atribuímos null a n2. Isso quebra a primeira das cadeias de acessibilidade para Node2, mas a segunda permanece ininterrupta, então Node2 ainda é alcançável.

Na instrução T6, atribuímos null a n3. Isso quebra a única cadeia de acessibilidade para Node3, o que torna o Node3 inacessível. No entanto, Node1 e Node2 ainda podem ser acessados ​​através da variável n1.

Finalmente, quando o método test() retorna, suas variáveis ​​locais n1, n2 e n3 saem do escopo e, portanto, não podem ser acessadas por nada. Isso quebra as cadeias de acessibilidade restantes para Node1 e Node2, e todos os objetos Node não são inacessíveis e elegíveis para coleta de lixo.

1 - Esta é uma simplificação que ignora a finalização e as classes Reference. 2 - Hipoteticamente, uma implementação Java poderia fazer isso, mas o custo de desempenho de fazer isso torna isso impraticável.

Vazamentos de memória em Java #

No exemplo [Garbage collection][1], sugerimos que Java resolve o problema de vazamentos de memória. Isso não é verdade. Um programa Java pode vazar memória, embora as causas dos vazamentos sejam bastante diferentes.

Objetos alcançáveis ​​podem vazar #

Considere a seguinte implementação de pilha ingênua.

public class NaiveStack {
    private Object[] stack = new Object[100];
    private int top = 0;

    public void push(Object obj) {
        if (top >= stack.length) {
            throw new StackException("stack overflow");
        }
        stack[top++] = obj;
    }

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        return stack[--top];
    }

    public boolean isEmpty() {
        return top == 0;
    }
}

Quando você push um objeto e imediatamente o pop, ainda haverá uma referência ao objeto no array stack.

A lógica da implementação da pilha significa que essa referência não pode ser retornada a um cliente da API. Se um objeto foi exibido, podemos provar que ele não pode “ser acessado em qualquer computação contínua potencial de qualquer thread ao vivo”. O problema é que uma JVM da geração atual não pode provar isso. As JVMs da geração atual não consideram a lógica do programa para determinar se as referências são alcançáveis. (Para começar, não é prático.)

Mas deixando de lado a questão do que reachability realmente significa, temos claramente uma situação aqui onde a implementação do NaiveStack está “pendurada” em objetos que devem ser recuperados. Isso é um vazamento de memória.

Neste caso, a solução é simples:

    public Object pop() {
        if (top <= 0) {
            throw new StackException("stack underflow");
        }
        Object popped = stack[--top];
        stack[top] = null;              // Overwrite popped reference with null.
        return popped;
    }

Caches podem ser vazamentos de memória #

Uma estratégia comum para melhorar o desempenho do serviço é armazenar os resultados em cache. A ideia é que você mantenha um registro de solicitações comuns e seus resultados em uma estrutura de dados na memória conhecida como cache. Então, cada vez que uma solicitação é feita, você pesquisa a solicitação no cache. Se a pesquisa for bem-sucedida, você retornará os resultados salvos correspondentes.

Esta estratégia pode ser muito eficaz se implementada corretamente. No entanto, se implementado incorretamente, um cache pode ser um vazamento de memória. Considere o seguinte exemplo:

public class RequestHandler {
    private Map<Task, Result> cache = new HashMap<>();

    public Result doRequest(Task task) {
        Result result = cache.get(task);
        if (result == null) {
            result == doRequestProcessing(task);
            cache.put(task, result);
        }
        return result;
    }
}

O problema com este código é que enquanto qualquer chamada para doRequest pode adicionar uma nova entrada ao cache, não há nada para removê-los. Se o serviço estiver continuamente recebendo tarefas diferentes, o cache acabará consumindo toda a memória disponível. Esta é uma forma de vazamento de memória.

Uma abordagem para resolver isso é usar um cache com tamanho máximo e descartar entradas antigas quando o cache exceder o máximo. (Jogar fora a entrada usada menos recentemente é uma boa estratégia.) Outra abordagem é construir o cache usando WeakHashMap para que a JVM possa remover entradas de cache se o heap começar a ficar muito cheio.

Finalização #

Um objeto Java pode declarar um método finalize. Esse método é chamado logo antes de o Java liberar a memória para o objeto. Normalmente ficará assim:

public class MyClass {
  
    //Methods for the class

    @Override
    protected void finalize() throws Throwable {
        // Cleanup code
    }
}

No entanto, existem algumas advertências importantes sobre o comportamento da finalização do Java.

  • Java não garante quando um método finalize() será chamado.
  • Java nem mesmo garante que um método finalize() será chamado em algum momento durante a vida útil do aplicativo em execução.
  • A única coisa que é garantida é que o método será chamado antes que o objeto seja deletado… se o objeto for deletado.

As advertências acima significam que é uma má idéia confiar no método finalize para executar ações de limpeza (ou outras) que devem ser executadas em tempo hábil. O excesso de confiança na finalização pode levar a vazamentos de armazenamento, vazamentos de memória e outros problemas.

Em suma, existem muito poucas situações em que a finalização é realmente uma boa solução.

Os finalizadores são executados apenas uma vez #

Normalmente, um objeto é excluído depois de finalizado. No entanto, isso não acontece o tempo todo. Considere o seguinte exemplo1:

public class CaptainJack {
    public static CaptainJack notDeadYet = null;

    protected void finalize() {
        // Resurrection!
        notDeadYet = this;
    }
}

Quando uma instância de CaptainJack se torna inacessível e o coletor de lixo tenta recuperá-la, o método finalize() atribuirá uma referência à instância à variável notDeadYet. Isso tornará a instância acessível mais uma vez e o coletor de lixo não a excluirá.

Pergunta: O Capitão Jack é imortal?

Resposta: Não.

O problema é que a JVM só executará um finalizador em um objeto uma vez em sua vida útil. Se você atribuir null a notDeadYet fazendo com que uma instância ressuscitada fique inacessível mais uma vez, o coletor de lixo não chamará finalize() no objeto.
