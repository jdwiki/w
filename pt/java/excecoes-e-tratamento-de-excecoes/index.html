Objetos do tipo Throwable e seus subtipos podem ser enviados para cima na pilha com a palavra-chave throw e capturados com instruções try…catch.

Sintaxe #

  • void someMethod() lança SomeException { } //declaração do método, força o método chamadores para capturar se SomeException for um tipo de exceção verificado

  • tentar {

     someMethod(); //code that might throw an exception

    }

  • catch (SomeException e) {

      System.out.println("SomeException was thrown!"); //code that will run if certain exception (SomeException) is thrown

    }

  • finalmente {

      //code that will always run, whether try block finishes or not

    }

Capturando uma exceção com try-catch #

Uma exceção pode ser capturada e tratada usando a instrução try...catch. (Na verdade, as instruções try assumem outras formas, como descrito em outros exemplos sobre try...catch...finally e try-with-resources.)

Try-catch com um bloco catch #

A forma mais simples fica assim:

try {
    doSomething();
} catch (SomeException e) {
    handle(e);
}
// next statement

O comportamento de um simples try...catch é o seguinte:

  • As instruções no bloco try são executadas.
  • Se nenhuma exceção for lançada pelas instruções no bloco try, então o controle passa para a próxima instrução após o try...catch.
  • Se uma exceção for lançada dentro do bloco try.
    • The exception object is tested to see if it is an instance of SomeException or a subtype.
    • If it is, then the catch block will catch the exception:
      • The variable e is bound to the exception object.
      • The code within the catch block is executed.
      • If that code throws an exception, then the newly thrown exception is propagated in place of the original one.
      • Otherwise, control passes to the next statement after the try...catch.
    • If it is not, the original exception continues to propagate.

Try-catch com várias capturas #

Um try...catch também pode ter vários blocos catch. Por exemplo:

try {
    doSomething();
} catch (SomeException e) {
    handleOneWay(e)
} catch (SomeOtherException e) {
    handleAnotherWay(e);
}
// next statement

Se houver vários blocos catch, eles são tentados um de cada vez, começando pelo primeiro, até que uma correspondência seja encontrada para a exceção. O manipulador correspondente é executado (como acima), e então o controle é passado para a próxima instrução após a instrução try...catch. Os blocos catch após o que corresponde são sempre ignorados, mesmo que o código do manipulador lance uma exceção.

A estratégia de correspondência “de cima para baixo” tem consequências para casos em que as exceções nos blocos catch não são disjuntas. Por exemplo:

try {
    throw new RuntimeException("test");
} catch (Exception e) {
    System.out.println("Exception");
} catch (RuntimeException e) {
    System.out.println("RuntimeException");
}

Este trecho de código produzirá “Exception” em vez de “RuntimeException”. Como RuntimeException é um subtipo de Exception, o primeiro (mais geral) catch será correspondido. O segundo (mais específico) catch nunca será executado.

A lição a aprender com isso é que os blocos catch mais específicos (em termos de tipos de exceção) devem aparecer primeiro, e os mais gerais devem ser os últimos. (Alguns compiladores Java irão avisá-lo se um catch nunca puder ser executado, mas isso não é um erro de compilação.)

Blocos de captura de várias exceções #

A partir do Java SE 7, um único bloco catch pode manipular uma lista de exceções não relacionadas. Os tipos de exceção são listados, separados por um símbolo de barra vertical (|). Por exemplo:

try {
    doSomething();
} catch (SomeException | SomeOtherException e) {
    handleSomeException(e);
}

O comportamento de uma captura de várias exceções é uma extensão simples para o caso de exceção única. O catch corresponde se a exceção lançada corresponder (pelo menos) a uma das exceções listadas.

Há alguma sutileza adicional na especificação. O tipo de e é uma união sintética dos tipos de exceção na lista. Quando o valor de e é usado, seu tipo estático é o supertipo menos comum do tipo união. No entanto, se e for relançado dentro do bloco catch, os tipos de exceção que são lançados são os tipos na união. Por exemplo:

public void method() throws IOException, SQLException
    try {
        doSomething();
    } catch (IOException | SQLException e) {
        report(e);
        throw e;
    }

Acima, IOException e SQLException são exceções marcadas cujo supertipo menos comum é Exception. Isso significa que o método report deve corresponder a report(Exception). No entanto, o compilador sabe que o throw pode lançar apenas uma IOException ou uma SQLException. Assim, method pode ser declarado como throws IOException, SQLException em vez de throws Exception. (O que é uma coisa boa: veja Pitfall - Throwing Throwable, Exception, Error or RuntimeException.)

A instrução try-with-resources #

Como o exemplo try-catch-final statement ilustra, a limpeza de recursos usando uma cláusula finally requer uma quantidade significativa de código “boiler-plate” para implementar os casos de borda corretamente. O Java 7 fornece uma maneira muito mais simples de lidar com esse problema na forma da instrução try-with-resources.

O que é um recurso? #

O Java 7 introduziu a interface java.lang.AutoCloseable para permitir que classes sejam gerenciadas usando a instrução try-with-resources. Instâncias de classes que implementam AutoCloseable são chamadas de recursos. Normalmente, eles precisam ser descartados em tempo hábil, em vez de depender do coletor de lixo para eliminá-los.

A interface AutoCloseable define um único método:

public void close() throws Exception

Um método close() deve descartar o recurso de maneira apropriada. A especificação afirma que deve ser seguro chamar o método em um recurso que já foi descartado. Além disso, classes que implementam Autocloseable são fortemente encorajadas a declarar o método close() para lançar uma exceção mais específica que Exception, ou nenhuma exceção.

Uma ampla variedade de classes e interfaces Java padrão implementam AutoCloseable. Esses incluem:

  • InputStream, OutputStream e suas subclasses
  • Reader, Writer e suas subclasses
  • Socket e ServerSocket e suas subclasses
  • Canal e suas subclasses, e
  • as interfaces JDBC Connection, Statement e ResultSet e suas subclasses.

As classes de aplicativos e de terceiros também podem fazer isso.

A instrução básica try-with-resource #

A sintaxe de um try-with-resources é baseada nos formulários clássicos try-catch, try-finally e try-catch-finally. Aqui está um exemplo de um formulário “básico”; ou seja, o formulário sem um catch ou finally.

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

Os recursos a serem gerenciados são declarados como variáveis ​​na seção (...) após a cláusula try. No exemplo acima, declaramos uma variável de recurso stream e a inicializamos para um PrintStream recém-criado.

Uma vez inicializadas as variáveis ​​de recursos, o bloco try é executado. Quando isso for concluído, stream.close() será chamado automaticamente para garantir que o recurso não vaze. Observe que a chamada close() acontece independentemente de como o bloco é concluído.

As instruções try-with-resource aprimoradas #

A instrução try-with-resources pode ser aprimorada com blocos catch e finally, como na sintaxe pré-Java 7 try-catch-finally. O trecho de código a seguir adiciona um bloco catch ao nosso anterior para lidar com o FileNotFoundException que o construtor PrintStream pode lançar:

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
} catch (FileNotFoundException ex) {
    System.err.println("Cannot open the file");
} finally {
    System.err.println("All done");
}

Se a inicialização do recurso ou o bloco try lançar a exceção, então o bloco catch será executado. O bloco finally sempre será executado, como acontece com uma instrução try-catch-finally convencional.

No entanto, há algumas coisas a serem observadas:

  • A variável resource está fora do escopo nos blocos catch e finally.
  • A limpeza do recurso acontecerá antes que a instrução tente corresponder ao bloco catch.
  • Se a limpeza automática de recursos lançou uma exceção, então ela poderia ser capturada em um dos blocos catch.

Gerenciando vários recursos #

Os trechos de código acima mostram um único recurso sendo gerenciado. Na verdade, try-with-resources pode gerenciar vários recursos em uma instrução. Por exemplo:

try (InputStream is = new FileInputStream(file1);
     OutputStream os = new FileOutputStream(file2)) {
    // Copy 'is' to 'os'
}

Isso se comporta como você esperaria. Ambos is e os são fechados automaticamente no final do bloco try. Há alguns pontos a serem observados:

  • As inicializações ocorrem na ordem do código, e inicializadores de variáveis ​​de recursos posteriores podem usar os valores dos anteriores.
  • Todas as variáveis ​​de recursos que foram inicializadas com sucesso serão limpas.
  • As variáveis ​​de recursos são limpas em ordem inversa de suas declarações.

Assim, no exemplo acima, is é inicializado antes de os e limpo depois dele, e is será limpo se houver uma exceção ao inicializar os.

Equivalência de try-with-resource e try-catch-finally clássico

A Java Language Specification especifica o comportamento dos formulários try-with-resource em termos da instrução clássica try-catch-finally. (Por favor, consulte o JLS para obter os detalhes completos.)

Por exemplo, este try-with-resource básico:

try (PrintStream stream = new PrintStream("hello.txt")) {
    stream.println("Hello world!");
}

é definido para ser equivalente a este try-catch-finally:

// Note that the constructor is not part of the try-catch statement
PrintStream stream = new PrintStream("hello.txt");

// This variable is used to keep track of the primary exception thrown
// in the try statement. If an exception is thrown in the try block,
// any exception thrown by AutoCloseable.close() will be suppressed.
Throwable primaryException = null;

// The actual try block
try {
    stream.println("Hello world!");
} catch (Throwable t) {
    // If an exception is thrown, remember it for the finally block
    primaryException = t;
    throw t;
} finally {
    if (primaryException == null) {
        // If no exception was thrown so far, exceptions thrown in close() will
        // not be caught and therefore be passed on to the enclosing code.
        stream.close();
    } else {
        // If an exception has already been thrown, any exception thrown in
        // close() will be suppressed as it is likely to be related to the
        // previous exception. The suppressed exception can be retrieved
        // using primaryException.getSuppressed().
        try {
            stream.close();
        } catch (Throwable suppressedException) {
            primaryException.addSuppressed(suppressedException);
        }
    }
}

(O JLS especifica que as variáveis ​​t e primaryException reais serão invisíveis para o código Java normal.)

A forma aprimorada de try-with-resources é especificada como uma equivalência com a forma básica. Por exemplo:

try (PrintStream stream = new PrintStream(fileName)) {
    stream.println("Hello world!");
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

é equivalente a:

try {
    try (PrintStream stream = new PrintStream(fileName)) {
        stream.println("Hello world!");
    }
} catch (NullPointerException ex) {
    System.err.println("Null filename");
} finally {
    System.err.println("All done");    
}

Exceções personalizadas #

Na maioria das circunstâncias, é mais simples do ponto de vista de design de código usar classes genéricas Exception ao lançar exceções. Isso é especialmente verdadeiro se você precisar apenas da exceção para transportar uma mensagem de erro simples. Nesse caso, RuntimeException geralmente é preferido, pois não é uma exceção marcada. Existem outras classes de exceção para classes comuns de erros:

  • UnsupportedOperationException - uma determinada operação não é suportada
  • IllegalArgumentException - um valor de parâmetro inválido foi passado para um método
  • IllegalStateException - sua API atingiu internamente uma condição que nunca deveria acontecer ou que ocorre como resultado de usar sua API de forma inválida

Os casos em que você deseja usar uma classe de exceção personalizada incluem o seguinte:

  • Você está escrevendo uma API ou biblioteca para uso por outras pessoas e deseja permitir que os usuários de sua API possam capturar e manipular especificamente exceções de sua API e ser capazes de diferenciar essas exceções de outras exceções mais genéricas .
  • Você está lançando exceções para um tipo específico de erro em uma parte de seu programa, que deseja capturar e manipular em outra parte de seu programa, e deseja diferenciar esses erros de outros mais erros genéricos.

Você pode criar suas próprias exceções personalizadas estendendo RuntimeException para uma exceção não verificada, ou exceção verificada estendendo qualquer Exception que também não é subclasse de RuntimeException, porque:

As subclasses de Exception que não são também subclasses de RuntimeException são exceções verificadas

public class StringTooLongException extends RuntimeException {
    // Exceptions can have methods and fields like other classes
    // those can be useful to communicate information to pieces of code catching
    // such an exception
    public final String value;
    public final int maximumLength;

    public StringTooLongException(String value, int maximumLength){
        super(String.format("String exceeds maximum Length of %s: %s", maximumLength, value));
        this.value = value;
        this.maximumLength = maximumLength;
    }
}

Esses podem ser usados ​​apenas como exceções predefinidas:

void validateString(String value){
    if (value.length() > 30){
        throw new StringTooLongException(value, 30);
    }
}

E os campos podem ser usados ​​onde a exceção é capturada e tratada:

void anotherMethod(String value){
    try {
        validateString(value);
    } catch(StringTooLongException e){
        System.out.println("The string '" + e.value + 
                "' was longer than the max of " + e.maximumLength );
    }
}

[…] Se é razoável esperar que um cliente se recupere de uma exceção, torne-a uma exceção verificada. Se um cliente não puder fazer nada para se recuperar da exceção, torne-a uma exceção não verificada.

Manipulando a exceção interrompida #

InterruptedException é uma fera confusa - aparece em métodos aparentemente inócuos como Thread.sleep(), mas manuseá-lo incorretamente leva a um código difícil de gerenciar que se comporta mal em ambientes simultâneos.

Em sua forma mais básica, se uma InterruptedException for capturada significa que alguém, em algum lugar, chamado Thread.interrupt() na thread em que seu código está sendo executado. Você pode estar inclinado a dizer “É meu código! Eu nunca vou interrompê-lo!” e, portanto, faça algo assim:

// Bad. Don't do this.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  // disregard
}

Mas esta é exatamente a maneira errada de lidar com um evento “impossível” ocorrendo. Se você sabe que seu aplicativo nunca encontrará uma InterruptedException, você deve tratar tal evento como uma violação séria das suposições do seu programa e sair o mais rápido possível.

A maneira correta de lidar com uma interrupção “impossível” é assim:

// When nothing will interrupt your code
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  throw new AssertionError(e);
}

Isso faz duas coisas; ele primeiro restaura o status de interrupção do thread (como se o InterruptedException não tivesse sido lançado em primeiro lugar), e então ele lança um AssertionError indicando que as invariantes básicas do seu aplicativo foram violadas. Se você sabe com certeza que nunca interromperá o thread em que este código é executado, isso é seguro, pois o bloco catch nunca deve ser alcançado.

Usar a classe Uninterruptibles do Guava ajuda a simplificar esse padrão; chamar Uninterruptibles.sleepUninterruptably() desconsidera o estado interrompido de uma thread até que a duração do sono tenha expirado (no ponto em que é restaurada para chamadas posteriores para inspecionar e lançar sua própria InterruptedException). Se você sabe que nunca interromperá esse código, isso evita com segurança a necessidade de envolver suas chamadas de suspensão em um bloco try-catch.

Mais frequentemente, no entanto, você não pode garantir que seu encadeamento nunca será interrompido. Em particular, se você estiver escrevendo código que será executado por um [Executor][6] ou algum outro gerenciador de threads, é fundamental que seu código responda prontamente a interrupções, caso contrário, seu aplicativo irá travar ou até travar.

Nesses casos, a melhor coisa a fazer geralmente é permitir que o InterruptedException se propague pela pilha de chamadas, adicionando um throws InterruptedException a cada método por vez. Isso pode parecer desajeitado, mas na verdade é uma propriedade desejável - as assinaturas do seu método agora indicam aos chamadores que ele responderá prontamente às interrupções.

// Let the caller determine how to handle the interrupt if you're unsure
public void myLongRunningMethod() throws InterruptedException {
  ...
}

Em casos limitados (por exemplo, ao substituir um método que não ’lanç’ nenhuma exceção verificada), você pode redefinir o status interrompido sem gerar uma exceção, esperando que qualquer código executado em seguida trate a interrupção. Isso atrasa o tratamento da interrupção, mas não a suprime totalmente.

// Suppresses the exception but resets the interrupted state letting later code
// detect the interrupt and handle it properly.
try {
  Thread.sleep(1000);
} catch (InterruptedException e) {
  Thread.currentThread().interrupt();
  return ...; // your expectations are still broken at this point - try not to do more work.
}

Declarações de retorno no bloco try catch #

Embora seja uma prática ruim, é possível adicionar várias instruções de retorno em um bloco de tratamento de exceção:

 public static int returnTest(int number){
    try{
        if(number%2 == 0) throw new Exception("Exception thrown");
        else return x;
    }
    catch(Exception e){
        return 3;
    }
    finally{
        return 7;
    }
}

Este método sempre retornará 7, pois o bloco finally associado ao bloco try/catch é executado antes que qualquer coisa seja retornada. Agora, como finalmente tem return 7;, este valor substitui os valores de retorno try/catch.

Se o bloco catch retornar um valor primitivo e esse valor primitivo for posteriormente alterado no bloco finally, o valor retornado no bloco catch será retornado e as alterações do bloco finally serão ignoradas.

O exemplo abaixo imprimirá “0”, não “1”.

public class FinallyExample {

    public static void main(String[] args) {
        int n = returnTest(4);
        
        System.out.println(n);
    }

    public static int returnTest(int number) {
        
        int returnNumber = 0; 
        
        try {
            if (number % 2 == 0)
                throw new Exception("Exception thrown");
            else
                return returnNumber;
        } catch (Exception e) {
            return returnNumber;
        } finally {
            returnNumber = 1;
        }
    }
}

A Hierarquia de Exceções Java - Exceções Não Verificadas e Verificadas #

Todas as exceções Java são instâncias de classes na hierarquia de classes Exception. Isso pode ser representado da seguinte forma:

  • java.lang.Throwable - Esta é a classe base para todas as classes de exceção. Seus métodos e construtores implementam uma gama de funcionalidades comuns a todas as exceções.
    • java.lang.Exception - This is the superclass of all normal exceptions.
      • various standard and custom exception classes.
      • java.lang.RuntimeException - This the superclass of all normal exceptions that are unchecked exceptions.
        • various standard and custom runtime exception classes.
    • java.lang.Error - This is the superclass of all “fatal error” exceptions.

Notas:

  1. A distinção entre exceções marcadas e não marcadas é descrita abaixo.
  2. As classes Throwable, Exception e RuntimeException devem ser tratadas como abstract; consulte https://www.wikiod.com/pt/java/armadilhas-do-java—uso-de-excecaopitfall-throwing-throwable-exception-error-or-runtimeexception#t=201610240803173325134.
  3. As exceções Error são lançadas pela JVM em situações em que seria inseguro ou imprudente para um aplicativo tentar se recuperar.
  4. Seria imprudente declarar subtipos personalizados de Throwable. Ferramentas e bibliotecas Java podem assumir que Error e Exception são os únicos subtipos diretos de Throwable, e se comportam mal se essa suposição estiver incorreta.

Exceções verificadas versus não verificadas #

Uma das críticas ao suporte a exceções em algumas linguagens de programação é que é difícil saber quais exceções um determinado método ou procedimento pode lançar. Dado que uma exceção não tratada pode causar a falha de um programa, isso pode tornar as exceções uma fonte de fragilidade.

A linguagem Java aborda essa preocupação com o mecanismo de exceção verificado. Primeiro, Java classifica as exceções em duas categorias:

  • Exceções verificadas normalmente representam eventos antecipados com os quais um aplicativo deve ser capaz de lidar. Por exemplo, IOException e seus subtipos representam condições de erro que podem ocorrer em operações de E/S. Os exemplos incluem falha na abertura de arquivo porque um arquivo ou diretório não existe, falha na leitura e gravação da rede porque uma conexão de rede foi interrompida e assim por diante.

  • Exceções não verificadas normalmente representam eventos imprevistos com os quais um aplicativo não pode lidar. Estes são normalmente o resultado de um bug no aplicativo.

(A seguir, “lançado” refere-se a qualquer exceção lançada explicitamente (por uma instrução throw) ou implicitamente (em uma falha de referência, conversão de tipo e assim por diante). Da mesma forma, “propagado” refere-se a uma exceção que foi gerado em uma chamada aninhada e não foi capturado nessa chamada. O código de exemplo abaixo ilustrará isso.)

A segunda parte do mecanismo de exceção verificada é que existem restrições nos métodos em que uma exceção verificada pode ocorrer:

  • Quando uma exceção verificada é lançada ou propagada em um método, ela deve ser capturada pelo método ou listada na cláusula throws do método. (O significado da cláusula throws é descrito em neste exemplo.)
  • Quando uma exceção verificada é lançada ou propagada em um bloco inicializador, ela deve ser capturada no bloco.
  • Uma exceção verificada não pode ser propagada por uma chamada de método em uma expressão de inicialização de campo. (Não há como capturar tal exceção.)

Em resumo, uma exceção verificada deve ser tratada ou declarada.

Essas restrições não se aplicam a exceções não verificadas. Isso inclui todos os casos em que uma exceção é lançada implicitamente, pois todos esses casos lançam exceções não verificadas.

Exemplos de exceções verificadas #

Esses trechos de código destinam-se a ilustrar as restrições de exceção verificadas. Em cada caso, mostramos uma versão do código com um erro de compilação e uma segunda versão com o erro corrigido.

// This declares a custom checked exception.
public class MyException extends Exception {
    // constructors omitted.
}

// This declares a custom unchecked exception.
public class MyException2 extends RuntimeException {
    // constructors omitted.
}

O primeiro exemplo mostra como exceções verificadas explicitamente lançadas podem ser declaradas como “lançadas” se não devem ser tratadas no método.

// INCORRECT
public void methodThrowingCheckedException(boolean flag) {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

// CORRECTED
public void methodThrowingCheckedException(boolean flag) throws MyException {
    int i = 1 / 0;                // Compiles OK, throws ArithmeticException
    if (flag) {
        throw new MyException();  // Compilation error
    } else {
        throw new MyException2(); // Compiles OK
    }
}

O segundo exemplo mostra como uma exceção verificada propagada pode ser tratada.

// INCORRECT 
public void methodWithPropagatedCheckedException() {
    InputStream is = new FileInputStream("someFile.txt");  // Compilation error
    // FileInputStream throws IOException or a subclass if the file cannot 
    // be opened.  IOException is a checked exception.
    ...
}

// CORRECTED (Version A) 
public void methodWithPropagatedCheckedException() throws IOException {
    InputStream is = new FileInputStream("someFile.txt"); 
    ...
}

// CORRECTED (Version B) 
public void methodWithPropagatedCheckedException() {
    try {
        InputStream is = new FileInputStream("someFile.txt"); 
        ...
    } catch (IOException ex) {
        System.out.println("Cannot open file: " + ex.getMessage());
    }
}

O exemplo final mostra como lidar com uma exceção verificada em um inicializador de campo estático.

// INCORRECT
public class Test {
    private static final InputStream is = 
            new FileInputStream("someFile.txt");  // Compilation error
}

// CORRECTED
public class Test {
    private static final InputStream is;
    static {
        InputStream tmp = null;
        try {
            tmp = new FileInputStream("someFile.txt");
        } catch (IOException ex) {
            System.out.println("Cannot open file: " + ex.getMessage());
        }
        is = tmp;
    }
}

Introdução #

Exceções são erros que ocorrem durante a execução de um programa. Considere o programa Java abaixo que divide dois inteiros.

class Division {
    public static void main(String[] args) {
 
        int a, b, result;
 
        Scanner input = new Scanner(System.in);
        System.out.println("Input two integers");
 
        a = input.nextInt();
        b = input.nextInt();
 
        result = a / b;
 
        System.out.println("Result = " + result);
    }
}

Agora compilamos e executamos o código acima e vemos a saída para uma tentativa de divisão por zero:

Input two integers
7 0
Exception in thread "main" java.lang.ArithmeticException: / by zero 
    at Division.main(Disivion.java:14)

A divisão por zero é uma operação inválida que produziria um valor que não pode ser representado como um inteiro. Java lida com isso jogando uma exceção. Nesse caso, a exceção é uma instância da classe ArithmeticException.

Observação: o exemplo em criar e ler rastreamentos de pilha explica o que significa a saída após os dois números.

A utilidade de uma exceção é o controle de fluxo que ela permite. Sem usar exceções, uma solução típica para este problema pode ser primeiro verificar se b == 0:

class Division {
    public static void main(String[] args) {
 
        int a, b, result;

        Scanner input = new Scanner(System.in);
        System.out.println("Input two integers");
 
        a = input.nextInt();
        b = input.nextInt();
 
        if (b == 0) {
            System.out.println("You cannot divide by zero.");
            return;
        }

        result = a / b;
 
        System.out.println("Result = " + result);
    }
}

Isso imprime a mensagem Você não pode dividir por zero. no console e encerra o programa de uma maneira normal quando o usuário tenta dividir por zero. Uma maneira equivalente de lidar com este problema via manipulação de exceção seria substituir o controle de fluxo if por um bloco try-catch:

...

a = input.nextInt();
b = input.nextInt();
 
try {
    result = a / b;
}
catch (ArithmeticException e) {
    System.out.println("An ArithmeticException occurred. Perhaps you tried to divide by zero.");
    return;
}
 
...

Um bloco try catch é executado da seguinte forma:

  1. Comece a executar o código no bloco try.
  2. Se ocorrer uma exceção no bloco try, aborte imediatamente e verifique se esta exceção é capturada pelo bloco catch (neste caso, quando a Exception é uma instância de ArithmeticException).
  3. Se a exceção for capturada, ela é atribuída à variável e e o bloco catch é executado.
  4. Se o bloco try ou catch for concluído (ou seja, nenhuma exceção não capturada ocorre durante a execução do código), continue a executar o código abaixo do bloco try-catch.

Geralmente, é considerado uma boa prática usar manipulação de exceção como parte do controle de fluxo normal de um aplicativo em que o comportamento seria indefinido ou inesperado. Por exemplo, em vez de retornar null quando um método falha, geralmente é uma prática melhor lançar uma exceção para que o aplicativo que está usando o método possa definir seu próprio controle de fluxo para a situação por meio de tratamento de exceção do tipo ilustrado acima. De certa forma, isso contorna o problema de ter que retornar um tipo específico, pois qualquer um dos vários tipos de exceções pode ser lançado para indicar o problema específico que ocorreu.

Para obter mais conselhos sobre como e como não usar exceções, consulte https://www.wikiod.com/pt/java/armadilhas-do-java—uso-de-excecao

Criando e lendo stacktraces #

Quando um objeto de exceção é criado (ou seja, quando você o novo), o construtor Throwable captura informações sobre o contexto no qual a exceção foi criada. Mais tarde, essas informações podem ser geradas na forma de um rastreamento de pilha, que pode ser usado para ajudar a diagnosticar o problema que causou a exceção em primeiro lugar.

Imprimindo um rastreamento de pilha #

Imprimir um stacktrace é simplesmente uma questão de chamar o método printStackTrace(). Por exemplo:

try {
    int a = 0;
    int b = 0;
    int c = a / b;
} catch (ArithmeticException ex) {
    // This prints the stacktrace to standard output
    ex.printStackTrace();
}

O método printStackTrace() sem argumentos imprimirá na saída padrão do aplicativo; ou seja, o atual System.out. Há também sobrecargas printStackTrace(PrintStream) e printStackTrace(PrintWriter) que imprimem em um Stream ou Writer especificado.

Notas:

  1. O stacktrace não inclui os detalhes da própria exceção. Você pode usar o método toString() para obter esses detalhes; por exemplo.

         // Print exception and stacktrace
     System.out.println(ex);
     ex.printStackTrace();

  2. A impressão do Stacktrace deve ser usada com moderação; consulte https://www.wikiod.com/pt/java/armadilhas-do-java—uso-de-excecao#Armadilha - Traços de pilha excessivos ou inadequados . Geralmente, é melhor usar uma estrutura de log e passar o objeto de exceção a ser registrado.

Entendendo um rastreamento de pilha #

Considere o seguinte programa simples que consiste em duas classes em dois arquivos. (Mostramos os nomes dos arquivos e adicionamos números de linha para fins de ilustração.)

File: "Main.java"
1   public class Main {
2       public static void main(String[] args) {
3           new Test().foo();
4       }
5   }

File: "Test.java"
1   class Test {
2       public void foo() {
3           bar();
4       }
5   
6       public int bar() {
7           int a = 1;
8           int b = 0;
9           return a / b;
10      }

Quando esses arquivos forem compilados e executados, obteremos a seguinte saída.

Exception in thread "main" java.lang.ArithmeticException: / by zero
        at Test.bar(Test.java:9)
        at Test.foo(Test.java:3)
        at Main.main(Main.java:3)

Vamos ler uma linha de cada vez para descobrir o que ela está nos dizendo.

A linha #1 nos diz que o thread chamado “main” foi encerrado devido a uma exceção não capturada. O nome completo da exceção é java.lang.ArithmeticException, e a mensagem de exceção é “/ por zero”.

Se procurarmos os javadocs para esta exceção, ele diz:

Lançado quando ocorre uma condição aritmética excepcional. Por exemplo, um inteiro “dividir por zero” lança uma instância dessa classe.

De fato, a mensagem “/ por zero” é um forte indício de que a causa da exceção é que algum código tentou dividir algo por zero. Mas o que?

As 3 linhas restantes são o rastreamento de pilha. Cada linha representa uma chamada de método (ou construtor) na pilha de chamadas e cada uma nos diz três coisas:

  • o nome da classe e método que estava sendo executado,
  • o nome do arquivo do código-fonte,
  • o número da linha do código-fonte da instrução que estava sendo executada

Essas linhas de um rastreamento de pilha são listadas com o quadro da chamada atual na parte superior. O quadro superior em nosso exemplo acima está no método Test.bar e na linha 9 do arquivo Test.java. Essa é a seguinte linha:

    return a / b;

Se olharmos algumas linhas anteriores no arquivo para onde b é inicializado, é aparente que b terá o valor zero. Podemos dizer sem dúvida que esta é a causa da exceção.

Se precisarmos ir mais longe, podemos ver pelo stacktrace que bar() foi chamado de foo() na linha 3 de Test.java, e que foo() foi chamado por sua vez de Main. main().

Nota: Os nomes de classes e métodos nos quadros de pilha são os nomes internos das classes e métodos. Você precisará reconhecer os seguintes casos incomuns:

  • Uma classe aninhada ou interna será semelhante a “OuterClass$InnerClass”.
  • Uma classe interna anônima será semelhante a “OuterClass$1”, “OuterClass$2”, etc.
  • Quando o código em um construtor, inicializador de campo de instância ou um bloco inicializador de instância estiver sendo executado, o nome do método será “”.
  • Quando o código em um inicializador de campo estático ou bloco de inicializador estático estiver sendo executado, o nome do método será “”.

(Em algumas versões do Java, o código de formatação do stacktrace detectará e eliminará sequências de stackframe repetidas, como pode ocorrer quando um aplicativo falha devido a recursão excessiva.)

Encadeamento de exceção e rastreamentos de pilha aninhados #

O encadeamento de exceção acontece quando um pedaço de código captura uma exceção e, em seguida, cria e lança uma nova, passando a primeira exceção como a causa. Aqui está um exemplo:

File: Test,java
1   public class Test {
2      int foo() {
3           return 0 / 0;
4      }
5
6       public Test() {
7           try {
8               foo();
9           } catch (ArithmeticException ex) {
10              throw new RuntimeException("A bad thing happened", ex);
11          }
12      }
13
14      public static void main(String[] args) {
15          new Test();
16      }
17  }

Quando a classe acima é compilada e executada, obtemos o seguinte stacktrace:

Exception in thread "main" java.lang.RuntimeException: A bad thing happened
        at Test.<init>(Test.java:10)
        at Test.main(Test.java:15)
Caused by: java.lang.ArithmeticException: / by zero
        at Test.foo(Test.java:3)
        at Test.<init>(Test.java:8)
        ... 1 more

O stacktrace começa com o nome da classe, método e pilha de chamadas para a exceção que (neste caso) causou o travamento do aplicativo. Isso é seguido por uma linha “Causado por:” que relata a exceção causa. O nome da classe e a mensagem são relatados, seguidos pelos quadros de pilha da exceção de causa. O rastreamento termina com um “… N mais” que indica que os últimos N quadros são os mesmos da exceção anterior.

O “causado por:” só é incluído na saída quando a causa da exceção primária não é null). As exceções podem ser encadeadas indefinidamente e, nesse caso, o rastreamento de pilha pode ter vários rastreamentos “causados ​​por:”.

Nota: o mecanismo cause só foi exposto na API Throwable em Java 1.4.0. Antes disso, o encadeamento de exceções precisava ser implementado pelo aplicativo usando um campo de exceção personalizado para representar a causa e um método printStackTrace personalizado.

Capturando um stacktrace como uma String #

Às vezes, um aplicativo precisa ser capaz de capturar um stacktrace como uma String Java, para que possa ser usado para outros propósitos. A abordagem geral para fazer isso é criar um OutputStream ou Writer temporário que grava em um buffer na memória e passa isso para o printStackTrace(...).

As bibliotecas Apache Commons e Guava fornecem métodos utilitários para capturar um stacktrace como uma String:

org.apache.commons.lang.exception.ExceptionUtils.getStackTrace(Throwable)

com.google.common.base.Throwables.getStackTraceAsString(Throwable)

Se você não puder usar bibliotecas de terceiros em sua base de código, o método a seguir fará a tarefa:

   /**
     * Returns the string representation of the stack trace.
     *
     * @param throwable the throwable
     * @return the string.
     */
    public static String stackTraceToString(Throwable throwable) {
        StringWriter stringWriter = new StringWriter();
        throwable.printStackTrace(new PrintWriter(stringWriter));
        return stringWriter.toString();
    }

Note que se sua intenção é analisar o stacktrace, é mais simples usar getStackTrace() e getCause() do que tentar analisar um stacktrace.

Lançando uma exceção #

O exemplo a seguir mostra o básico de lançar uma exceção:

public void checkNumber(int number) throws IllegalArgumentException {
    if (number < 0) {
        throw new IllegalArgumentException("Number must be positive: " + number);
    }
}

A exceção é lançada na 3ª linha. Esta afirmação pode ser dividida em duas partes:

  • new IllegalArgumentException(...) está criando uma instância da classe IllegalArgumentException, com uma mensagem que descreve o erro que a exceção está relatando.

  • throw ... está então lançando o objeto de exceção.

Quando a exceção é lançada, ela faz com que as instruções de fechamento terminam anormalmente até que a exceção seja tratada. Isso é descrito em outros exemplos.

É uma boa prática criar e lançar o objeto de exceção em uma única instrução, conforme mostrado acima. Também é uma boa prática incluir uma mensagem de erro significativa na exceção para ajudar o programador a entender a causa do problema. No entanto, esta não é necessariamente a mensagem que você deve mostrar ao usuário final. (Para começar, Java não tem suporte direto para internacionalizar mensagens de exceção.)

Há mais alguns pontos a serem feitos:

  • Declaramos o checkNumber como throws IllegalArgumentException. Isso não era estritamente necessário, já que IllegalArgumentException é uma exceção verificada; consulte https://www.wikiod.com/pt/java/excecoes-e-tratamento-de-excecoes#A Hierarquia de Exceções Java - Exceções Não Verificadas e Verificadas No entanto, é uma boa prática fazer isso e também incluir as exceções lançadas nos comentários javadoc de um método.

  • Código imediatamente após uma instrução throw é inalcançável. Portanto, se escrevermos isso:

       throw new IllegalArgumentException("it is bad");
   return;

    the compiler would report a compilation error for the return statement.

Encadeamento de exceção #

Muitas exceções padrão têm um construtor com um segundo argumento causa além do argumento mensagem convencional. A causa permite que você encadeie exceções. Aqui está um exemplo.

Primeiro, definimos uma exceção não verificada que nosso aplicativo lançará quando encontrar um erro irrecuperável. Observe que incluímos um construtor que aceita um argumento causa.

    public class AppErrorException extends RuntimeException {
        public AppErrorException() {
            super();
        }

        public AppErrorException(String message) {
            super(message);
        }

        public AppErrorException(String message, Throwable cause) {
            super(message, cause);
        }
    }

Em seguida, aqui está algum código que ilustra o encadeamento de exceções.

    public String readFirstLine(String file) throws AppErrorException {
        try (Reader r = new BufferedReader(new FileReader(file))) {
            String line = r.readLine();
            if (line != null) {
                return line;
            } else {
                throw new AppErrorException("File is empty: " + file);
            }
        } catch (IOException ex) {
            throw new AppErrorException("Cannot read file: " + file, ex);
        }
    }

O throw dentro do bloco try detecta um problema e o relata através de uma exceção com uma mensagem simples. Por outro lado, o throw dentro do bloco catch está manipulando a IOException envolvendo-a em uma nova exceção (marcada). No entanto, não está jogando fora a exceção original. Ao passar a IOException como a causa, nós a gravamos para que possa ser impressa no stacktrace, conforme explicado em https://www.wikiod.com/pt/java/excecoes-e-tratamento-de-excecoes 1815/creating-and-reading-stacktraces#t=20161028160848944857.

Recursos avançados de exceções #

Este exemplo abrange alguns recursos avançados e casos de uso para exceções.

Examinando a pilha de chamadas programaticamente #

O principal uso de stacktraces de exceção é fornecer informações sobre um erro de aplicativo e seu contexto para que o programador possa diagnosticar e corrigir o problema. Às vezes pode ser usado para outras coisas. Por exemplo, uma classe SecurityManager pode precisar examinar a pilha de chamadas para decidir se o código que está fazendo uma chamada deve ser confiável.

Você pode usar exceções para examinar a pilha de chamadas programaticamente da seguinte maneira:

    Exception ex = new Exception();   // this captures the call stack
    StackTraceElement[] frames = ex.getStackTrace();
    System.out.println("This method is " + frames[0].getMethodName());
    System.out.println("Called from method " + frames[1].getMethodName());

Há algumas ressalvas importantes sobre isso:

  1. As informações disponíveis em um StackTraceElement são limitadas. Não há mais informações disponíveis do que as exibidas por printStackTrace. (Os valores das variáveis ​​locais no quadro não estão disponíveis.)

  2. Os javadocs para getStackTrace() afirmam que uma JVM tem permissão para deixar quadros de fora:

    Some virtual machines may, under some circumstances, omit one or more stack frames from the stack trace. In the extreme case, a virtual machine that has no stack trace information concerning this throwable is permitted to return a zero-length array from this method.

Otimizando a construção de exceções #

Como mencionado em outro lugar, construir uma exceção é bastante caro porque envolve a captura e gravação de informações sobre todos os quadros de pilha no encadeamento atual. Às vezes, sabemos que essa informação nunca será usada para uma determinada exceção; por exemplo. o stacktrace nunca será impresso. Nesse caso, existe um truque de implementação que podemos usar em uma exceção personalizada para fazer com que as informações não sejam capturadas.

As informações de stack frame necessárias para stacktraces são capturadas quando os construtores Throwable chamam o método Throwable.fillInStackTrace(). Este método é public, o que significa que uma subclasse pode substituí-lo. O truque é substituir o método herdado de Throwable por um que não faz nada; por exemplo.

  public class MyException extends Exception {
      // constructors

      @Override 
      public void fillInStackTrace() {
          // do nothing
      }
  }

O problema com esta abordagem é que uma exceção que sobrescreve fillInStackTrace() nunca pode capturar o stacktrace e é inútil em cenários onde você precisa de um.

Apagando ou substituindo o rastreamento de pilha #

Em algumas situações, o rastreamento de pilha para uma exceção criada da maneira normal contém informações incorretas ou informações que o desenvolvedor não deseja revelar ao usuário. Para esses cenários, o Throwable.setStackTrace pode ser usado para substituir o array de objetos StackTraceElement que contém as informações.

Por exemplo, o seguinte pode ser usado para descartar as informações da pilha de uma exceção:

 exception.setStackTrace(new StackTraceElement[0]);

Exceções suprimidas #

O Java 7 introduziu a construção try-with-resources e o conceito associado de supressão de exceção. Considere o seguinte trecho:

try (Writer w = new BufferedWriter(new FileWriter(someFilename))) {
    // do stuff
    int temp = 0 / 0;    // throws an ArithmeticException
}

Quando a exceção é lançada, o try chamará close() no w que irá liberar qualquer saída em buffer e então fechar o FileWriter. Mas o que acontece se uma IOException for lançada durante a liberação da saída?

O que acontece é que qualquer exceção lançada durante a limpeza de um recurso é suprimida. A exceção é capturada e adicionada à lista de exceções suprimidas da exceção primária. Em seguida, o try-with-resources continuará com a limpeza dos outros recursos. Finalmente, a exceção primária será relançada.

Um padrão semelhante ocorre se uma exceção for lançada durante a inicialização do recurso ou se o bloco try for concluído normalmente. A primeira exceção lançada torna-se a exceção primária e as subsequentes decorrentes da limpeza são suprimidas.

As exceções suprimidas podem ser recuperadas do objeto de exceção primário chamando getSuppressedExceptions.

As instruções try-finally e try-catch-finally #

A instrução try...catch...finally combina manipulação de exceção com código de limpeza. O bloco finally contém código que será executado em todas as circunstâncias. Isso os torna adequados para gerenciamento de recursos e outros tipos de limpeza.

Tente-finalmente #

Aqui está um exemplo do formulário mais simples (try...finally):

try {
    doSomething();  
} finally {
    cleanUp();
}

O comportamento do try...finally é o seguinte:

  • O código no bloco try é executado.
  • Se nenhuma exceção foi lançada no bloco try:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, control passes to the next statement after the try...finally.
  • Se uma exceção foi lançada no bloco try:
    • The code in the finally block is executed.
    • If the finally block throws an exception, that exception is propagated.
    • Otherwise, the original exception continues to propagate.

O código dentro do bloco finally sempre será executado. (As únicas exceções são se System.exit(int) for chamado, ou se a JVM entrar em pânico.) Assim, um bloco finally é o código de lugar correto que sempre precisa ser executado; por exemplo. fechando arquivos e outros recursos ou liberando bloqueios.

tente-pegue-finalmente #

Nosso segundo exemplo mostra como catch e finally podem ser usados ​​juntos. Também ilustra que a limpeza de recursos não é simples.

// This code snippet writes the first line of a file to a string
String result = null;
Reader reader = null;
try {
    reader = new BufferedReader(new FileReader(fileName));
    result = reader.readLine();
} catch (IOException ex) {
    Logger.getLogger.warn("Unexpected IO error", ex);  // logging the exception
} finally {
    if (reader != null) {
        try {
            reader.close();
        } catch (IOException ex) {
            // ignore / discard this exception
        }
    }
}

O conjunto completo de comportamentos (hipotéticos) de try...catch...finally neste exemplo é muito complicado para descrever aqui. A versão simples é que o código no bloco finally sempre será executado.

Olhando para isso da perspectiva do gerenciamento de recursos:

  • Declaramos o “resource” (ou seja, a variável reader) antes do bloco try para que fique no escopo do bloco finally.
  • Colocando o new FileReader(...), o catch é capaz de lidar com qualquer exceção IOError lançada ao abrir o arquivo.
  • Precisamos de um reader.close() no bloco finally porque existem alguns caminhos de exceção que não podemos interceptar nem no bloco try nem no bloco catch.
  • No entanto, como uma exceção pode ter sido lançada antes do reader ser inicializado, também precisamos de um teste null explícito.
  • Finalmente, a chamada reader.close() pode (hipoteticamente) lançar uma exceção. Não nos importamos com isso, mas se não capturarmos a exceção na fonte, precisaríamos lidar com ela mais adiante na pilha de chamadas.

O Java 7 e versões posteriores fornecem uma sintaxe try-with-resources alternativa que simplifica significativamente a limpeza de recursos.

A cláusula ’throws’ em uma declaração de método #

O mecanismo de exceção verificada do Java exige que o programador declare que certos métodos poderiam lançar exceções verificadas especificadas. Isso é feito usando a cláusula throws. Por exemplo:

public class OddNumberException extends Exception { // a checked exception
}

public void checkEven(int number) throws OddNumberException {
    if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

O throws OddNumberException declara que uma chamada para checkEven poderia lançar uma exceção do tipo OddNumberException.

Uma cláusula throws pode declarar uma lista de tipos e pode incluir exceções não verificadas, bem como exceções verificadas.

public void checkEven(Double number) 
        throws OddNumberException, ArithmeticException {
    if (!Double.isFinite(number)) {
        throw new ArithmeticException("INF or NaN");
    } else if (number % 2 != 0) {
        throw new OddNumberException();
    }
}

Qual é o ponto de declarar exceções não verificadas como lançadas?

A cláusula throws em uma declaração de método serve a dois propósitos:

  1. Ele informa ao compilador quais exceções são lançadas para que o compilador possa relatar exceções não capturadas (verificadas) como erros.

  2. Diz ao programador que está escrevendo o código que chama o método quais exceções esperar. Para este propósito, muitas vezes faz sentido incluir exceções não verificadas em uma lista throws.

Observação: a lista throws também é usada pela ferramenta javadoc ao gerar a documentação da API e pelas dicas de método de “texto flutuante” de um IDE típico.

Lançamentos e substituição de método #

A cláusula throws faz parte da assinatura de um método com o propósito de sobrescrever o método. Um método de substituição pode ser declarado com o mesmo conjunto de exceções verificadas lançadas pelo método substituído ou com um subconjunto. No entanto, o método override não pode adicionar exceções verificadas extras. Por exemplo:

@Override
public void checkEven(int number) throws NullPointerException // OK—NullPointerException is an unchecked exception
    ...

@Override
public void checkEven(Double number) throws OddNumberException // OK—identical to the superclass
    ...

class PrimeNumberException extends OddNumberException {}
class NonEvenNumberException extends OddNumberException {}

@Override
public void checkEven(int number) throws PrimeNumberException, NonEvenNumberException // OK—these are both subclasses

@Override
public void checkEven(Double number) throws IOExcepion         // ERROR

A razão para essa regra é que, se um método substituído puder lançar uma exceção verificada que o método substituído não poderia lançar, isso interromperia a substituibilidade do tipo.
