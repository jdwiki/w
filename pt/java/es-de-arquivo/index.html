Java I/O (Entrada e Saída) é usado para processar a entrada e produzir a saída. Java usa o conceito de fluxo para tornar a operação de E/S rápida. O pacote java.io contém todas as classes necessárias para operações de entrada e saída. Manipulação de arquivos também é feito em java pela Java I/O API.

Migrando de java.io.File para Java 7 NIO (java.nio.file.Path) #

Esses exemplos assumem que você já sabe o que é o NIO do Java 7 em geral, e está acostumado a escrever código usando java.io.File. Use esses exemplos como um meio de encontrar rapidamente mais documentação centrada em NIO para migração.

Há muito mais no NIO do Java 7, como arquivos mapeados em memória ou abrir um arquivo ZIP ou JAR usando FileSystem. Esses exemplos abrangerão apenas um número limitado de casos de uso básicos.

Como regra básica, se você estiver acostumado a executar uma operação de leitura/gravação do sistema de arquivos usando um [java.io.File](https://docs.oracle.com/javase/7/docs/api/java/ io/File.html), você o encontrará como um método estático em [java.nio.file.Files](https://docs.oracle.com/javase/7/docs/api/java/ nio/file/Files.html).

Aponta para um caminho #

// -> IO
File file = new File("io.txt");

// -> NIO
Path path = Paths.get("nio.txt");

Caminhos relativos a outro caminho #

// Forward slashes can be used in place of backslashes even on a Windows operating system
// -> IO
File folder = new File("C:/");
File fileInFolder = new File(folder, "io.txt");

// -> NIO
Path directory = Paths.get("C:/");
Path pathInDirectory = directory.resolve("nio.txt");

Convertendo arquivo de/para caminho para uso com bibliotecas #

// -> IO to NIO
Path pathFromFile = new File("io.txt").toPath();

// -> NIO to IO
File fileFromPath = Paths.get("nio.txt").toFile();

Verifica se o arquivo existe e exclui se existir #

// -> IO
if (file.exists()) {
    boolean deleted = file.delete();
    if (!deleted) {
        throw new IOException("Unable to delete file");
    }
}

// -> NIO
Files.deleteIfExists(path);

Grava em um arquivo por meio de um OutputStream #

Existem várias maneiras de escrever e ler um arquivo usando NIO para diferentes restrições de desempenho e memória, legibilidade e casos de uso, como FileChannel, Files.write(Path path, byte\[\] bytes, OpenOption... options)… Neste exemplo, apenas OutputStream é coberto, mas você é fortemente encorajado a aprender sobre arquivos mapeados em memória e os vários métodos estáticos disponíveis em java.nio .file.Files.

List<String> lines = Arrays.asList(
        String.valueOf(Calendar.getInstance().getTimeInMillis()),
        "line one",
        "line two");

// -> IO

    public static void copy(File sourceFile, File destFile) {
        if (!sourceFile.exists() || !destFile.exists()) {
            System.out.println("Source or destination file doesn't exist");
            return;
        }


        try (FileChannel srcChannel = new FileInputStream(sourceFile).getChannel();
             FileChannel sinkChanel = new FileOutputStream(destFile).getChannel()) {

            srcChannel.transferTo(0, srcChannel.size(), sinkChanel);
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

API de Stream vs Writer/Reader #

Streams fornecem o acesso mais direto ao conteúdo binário, então qualquer implementação InputStream / OutputStream sempre opera em ints e bytes.

// Read a single byte from the stream
int b = inputStream.read();
if (b >= 0) { // A negative value represents the end of the stream, normal values are in the range 0 - 255
    // Write the byte to another stream
    outputStream.write(b);
}

// Read a chunk
byte[] data = new byte[1024];
int nBytesRead = inputStream.read(data);
if (nBytesRead >= 0) { // A negative value represents end of stream
    // Write the chunk to another stream
    outputStream.write(data, 0, nBytesRead);
}

Existem algumas exceções, provavelmente mais notavelmente o PrintStream que adiciona a “capacidade de imprimir representações de vários valores de dados convenientemente”. Isso permite usar System.out tanto como um InputStream binário quanto como uma saída textual usando métodos como System.out.println().

Além disso, algumas implementações de fluxo funcionam como uma interface para conteúdos de nível superior, como objetos Java (consulte Serialização) ou tipos nativos, por exemplo. DataOutputStream / DataInputStream.

Com as classes Writer e Reader, Java também fornece uma API para fluxos de caracteres explícitos. Embora a maioria dos aplicativos baseie essas implementações em fluxos, a API de fluxo de caracteres não expõe nenhum método para conteúdo binário.

// This example uses the platform's default charset, see below
// for a better implementation.

Writer writer = new OutputStreamWriter(System.out);
writer.write("Hello world!");

Reader reader = new InputStreamReader(System.in);
char singleCharacter = reader.read();

Sempre que for necessário codificar caracteres em dados binários (por exemplo, ao usar as classes InputStreamWriter / OutputStreamWriter), você deve especificar um charset se não quiser depender do charset padrão da plataforma. Em caso de dúvida, use uma codificação compatível com Unicode, por exemplo. UTF-8 que é suportado em todas as plataformas Java. Portanto, você provavelmente deve ficar longe de classes como FileWriter e FileReader, pois elas sempre usam o conjunto de caracteres padrão da plataforma. Uma maneira melhor de acessar arquivos usando fluxos de caracteres é esta:

Charset myCharset = StandardCharsets.UTF_8;

Writer writer = new OutputStreamWriter( new FileOutputStream("test.txt"), myCharset );
writer.write('Ä');
writer.flush();
writer.close();

Reader reader = new InputStreamReader( new FileInputStream("test.txt"), myCharset );
char someUnicodeCharacter = reader.read();
reader.close();

Um dos Readers mais usados ​​é o BufferedReader que fornece um método para ler linhas inteiras de texto de outro leitor e é presumivelmente a maneira mais simples de ler um fluxo de caracteres linha por linha:

// Read from baseReader, one line at a time
BufferedReader reader = new BufferedReader( baseReader );
String line;
while((line = reader.readLine()) != null) {
  // Remember: System.out is a stream, not a writer!
  System.out.println(line);
}

Copiando um arquivo usando InputStream e OutputStream #

Podemos copiar dados diretamente de uma fonte para um coletor de dados usando um loop. Neste exemplo, estamos lendo dados de um InputStream e, ao mesmo tempo, gravando em um OutputStream. Assim que terminarmos de ler e escrever, temos que fechar o recurso.

public void copy(InputStream source, OutputStream destination) throws IOException {
    try {
        int c;
        while ((c = source.read()) != -1) {
            destination.write(c);
        }
    } finally {
        if (source != null) {
            source.close();
        }
        if (destination != null) {
            destination.close();
        }
    }
}

Lendo um arquivo usando Canal e Buffer #

Channel usa um Buffer para ler/gravar dados. Um buffer é um contêiner de tamanho fixo onde podemos escrever um bloco de dados de uma só vez. Channel é uma E/S bastante mais rápida que a baseada em fluxo.

Para ler dados de um arquivo usando Channel, precisamos ter os seguintes passos-

  1. Precisamos de uma instância de FileInputStream. FileInputStream tem um método chamado getChannel() que retorna um Canal.

  2. Chame o método getChannel() de FileInputStream e adquira o canal.

  3. Crie um ByteBuffer. ByteBuffer é um contêiner de tamanho fixo de bytes.

  4. O canal tem um método de leitura e temos que fornecer um ByteBuffer como argumento para este método de leitura. ByteBuffer tem dois modos - modo somente leitura e modo somente gravação. Podemos mudar o modo usando a chamada do método flip(). O buffer tem uma posição, limite e capacidade. Uma vez que um buffer é criado com um tamanho fixo, seu limite e capacidade são iguais ao tamanho e a posição começa do zero. Enquanto um buffer é escrito com dados, sua posição aumenta gradualmente. Mudar de modo significa mudar de posição. Para ler os dados do início de um buffer, temos que definir a posição como zero. método flip() altera a posição

  5. Quando chamamos o método read do Channel, ele preenche o buffer usando dados.

  6. Se precisarmos ler os dados do ByteBuffer, precisamos inverter o buffer para alterar seu modo de modo somente gravação para modo somente leitura e continuar lendo os dados do buffer.

  7. Quando não há mais dados para ler, o método read() do canal retorna 0 ou -1.

    import java.io.File; import java.io.FileInputStream; import java.io.IOException; import java.nio.ByteBuffer; import java.nio.channels.FileChannel;

    public class FileChannelRead {

    public static void main(String[] args) {

    File inputFile = new File("hello.txt");

if (!inputFile.exists()) {
 System.out.println("The input file doesn't exit.");
 return;
}

    try { FileInputStream fis = new FileInputStream(inputFile); FileChannel fileChannel = fis.getChannel(); ByteBuffer buffer = ByteBuffer.allocate(1024);

    while (fileChannel.read(buffer) > 0) {
 buffer.flip();
 while (buffer.hasRemaining()) {
  byte b = buffer.get();
  System.out.print((char) b);
 }
 buffer.clear();
}

fileChannel.close();

    } catch (IOException e) { e.printStackTrace(); } } }

Bloqueando ou redirecionando a saída/erro padrão #

Às vezes, uma biblioteca de terceiros mal projetada gravará diagnósticos indesejados em fluxos System.out ou System.err. As soluções recomendadas para isso seriam encontrar uma biblioteca melhor ou (no caso de código aberto) corrigir o problema e contribuir com um patch para os desenvolvedores.

Se as soluções acima não forem viáveis, considere redirecionar os fluxos.

Redirecionamento na linha de comando

Em um sistema UNIX, Linux ou MacOSX pode ser feito a partir do shell usando o redirecionamento >. Por exemplo:

$ java -jar app.jar arg1 arg2 > /dev/null 2>&1
$ java -jar app.jar arg1 arg2 > out.log 2> error.log

O primeiro redireciona a saída padrão e o erro padrão para “/dev/null”, que descarta qualquer coisa gravada nesses fluxos. O segundo redireciona a saída padrão para “out.log” e o erro padrão para “error.log”.

(Para obter mais informações sobre redirecionamento, consulte a documentação do shell de comando que você está usando. Conselhos semelhantes se aplicam ao Windows.)

Como alternativa, você pode implementar o redirecionamento em um script de wrapper ou arquivo em lote que ativa o aplicativo Java.

Redirecionamento dentro de um aplicativo Java

Também é possível redirecionar os fluxos dentro de uma aplicação Java usando System.setOut() e System.setErr(). Por exemplo, o snippet a seguir redireciona a saída padrão e o erro padrão para 2 arquivos de log:

System.setOut(new PrintStream(new FileOutputStream(new File("out.log"))));
System.setErr(new PrintStream(new FileOutputStream(new File("err.log"))));

Se você quiser descartar totalmente a saída, poderá criar um fluxo de saída que “grave” em um descritor de arquivo inválido. Isso é funcionalmente equivalente a gravar em “/dev/null” no UNIX.

System.setOut(new PrintStream(new FileOutputStream(new FileDescriptor())));
System.setErr(new PrintStream(new FileOutputStream(new FileDescriptor())));

Cuidado: tenha cuidado ao usar setOut e setErr:

  1. O redirecionamento afetará toda a JVM.
  2. Ao fazer isso, você está tirando a capacidade do usuário de redirecionar os fluxos da linha de comando.
