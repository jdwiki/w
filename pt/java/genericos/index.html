Genéricos são um recurso de programação genérica que estende o sistema de tipos do Java para permitir que um tipo ou método opere em objetos de vários tipos enquanto fornece segurança de tipo em tempo de compilação. Em particular, a estrutura de coleções Java suporta genéricos para especificar o tipo de objetos armazenados em uma instância de coleção.

Sintaxe #

  • class ArrayList<E> {} // uma classe genérica com parâmetro de tipo E
  • classe HashMap<K, V> {} // uma classe genérica com dois parâmetros de tipo K e V
  • <E> void print(E element) {} // um método genérico com parâmetro de tipo E
  • ArrayList<String> nomes; // declaração de uma classe genérica
  • Lista de Matrizes<?> objetos; // declaração de uma classe genérica com um parâmetro de tipo desconhecido
  • new ArrayList<String>() // instanciação de uma classe genérica
  • new ArrayList<>() // instanciação com inferência de tipo “diamond” (Java 7 ou posterior)

Os genéricos são implementados em Java por meio do apagamento de tipo, o que significa que durante o tempo de execução as informações de tipo especificadas na instanciação de uma classe genérica não estão disponíveis. Por exemplo, a instrução List<String> names = new ArrayList<>(); produz um objeto de lista do qual o tipo de elemento String não pode ser recuperado em tempo de execução. No entanto, se a lista for armazenada em um campo do tipo List<String>, ou passada para um parâmetro de método/construtor desse mesmo tipo, ou retornada de um método desse tipo de retorno, as informações completas do tipo podem ser recuperado em tempo de execução por meio da API Java Reflection.

Isso também significa que ao converter para um tipo genérico (por exemplo: (List<String>) list), o cast é um unchecked cast. Como o parâmetro <String> foi apagado, a JVM não pode verificar se uma conversão de um List<?> para um List<String> está correta; a JVM vê apenas uma conversão de List para List em tempo de execução.

Criando uma classe genérica #

Genéricos permitem que classes, interfaces e métodos recebam outras classes e interfaces como parâmetros de tipo.

Este exemplo usa a classe genérica Param para receber um único parâmetro de tipo T, delimitado por colchetes angulares (<>):

public class Param<T> {
    private T value;

    public T getValue() {
        return value;
    }

    public void setValue(T value) {
        this.value = value;
    }
}

Para instanciar esta classe, forneça um argumento de tipo no lugar de T. Por exemplo, Inteiro:

Param<Integer> integerParam = new Param<Integer>();

O argumento type pode ser qualquer tipo de referência, incluindo arrays e outros tipos genéricos:

Param<String[]> stringArrayParam;
Param<int[][]> int2dArrayParam;
Param<Param<Object>> objectNestedParam;

No Java SE 7 e posterior, o argumento de tipo pode ser substituído por um conjunto vazio de argumentos de tipo (<>) chamado [diamond][2]:

Param<Integer> integerParam = new Param<>();

Ao contrário de outros identificadores, os parâmetros de tipo não têm restrições de nomenclatura. No entanto, seus nomes são geralmente a primeira letra de sua finalidade em maiúsculas. (Isso é verdade mesmo em todos os JavaDocs oficiais.) Os exemplos incluem T para “tipo”, [E para “elemento”](https ://docs.oracle.com/javase/8/docs/api/java/util/List.html) e [K/V para “chave”/“valor”](https://docs. oracle.com/javase/8/docs/api/java/util/Map.html).

Estendendo uma classe genérica #

public abstract class AbstractParam<T> {
    private T value;

    public T getValue() {
        return value;
    }

    public void setValue(T value) {
        this.value = value;
    }
}

AbstractParam é uma [classe abstrata][3] declarada com um parâmetro de tipo T. Ao estender esta classe, esse parâmetro de tipo pode ser substituído por um argumento de tipo escrito dentro de <>, ou o parâmetro de tipo pode permanecer inalterado. No primeiro e segundo exemplos abaixo, String e Integer substituem o parâmetro type. No terceiro exemplo, o parâmetro de tipo permanece inalterado. O quarto exemplo não usa genéricos, então é semelhante a se a classe tivesse um parâmetro Object. O compilador avisará que AbstractParam é um tipo bruto, mas compilará a classe ObjectParam. O quinto exemplo tem 2 parâmetros de tipo (veja “parâmetros de vários tipos” abaixo), escolhendo o segundo parâmetro como o parâmetro de tipo passado para a superclasse.

public class Email extends AbstractParam<String> {
    // ...
}

public class Age extends AbstractParam<Integer> {
    // ...
}

public class Height<T> extends AbstractParam<T> {
    // ...
}

public class ObjectParam extends AbstractParam {
    // ...
}

public class MultiParam<T, E> extends AbstractParam<E> {
    // ...
}

Segue o uso:

Email email = new Email();
email.setValue("test@example.com");
String retrievedEmail = email.getValue();

Age age = new Age();
age.setValue(25);
Integer retrievedAge = age.getValue();
int autounboxedAge = age.getValue();

Height<Integer> heightInInt = new Height<>();
heightInInt.setValue(125);

Height<Float> heightInFloat = new Height<>();
heightInFloat.setValue(120.3f);

MultiParam<String, Double> multiParam = new MultiParam<>();
multiParam.setValue(3.3);

Observe que na classe Email, o método T getValue() age como se tivesse uma assinatura de String getValue(), e o método void setValue(T) age como se tivesse sido declarado void setValue(String).

Também é possível instanciar com classe interna anônima com chaves vazias ({}):

AbstractParam<Double> height = new AbstractParam<Double>(){};
height.setValue(198.6);

Observe que [usar o diamante com classes internas anônimas não é permitido.][4]

Vários parâmetros de tipo #

Java fornece a capacidade de usar mais de um parâmetro de tipo em uma classe ou interface genérica. Vários parâmetros de tipo podem ser usados ​​em uma classe ou interface colocando uma lista separada por vírgulas de tipos entre os colchetes angulares. Exemplo:

public class MultiGenericParam<T, S> {
    private T firstParam;
    private S secondParam;
   
    public MultiGenericParam(T firstParam, S secondParam) {
        this.firstParam = firstParam;
        this.secondParam = secondParam;
    }
    
    public T getFirstParam() {
        return firstParam;
    }
    
    public void setFirstParam(T firstParam) {
        this.firstParam = firstParam;
    }
    
    public S getSecondParam() {
        return secondParam;
    }
    
    public void setSecondParam(S secondParam) {
        this.secondParam = secondParam;
    }
}

O uso pode ser feito conforme abaixo:

MultiGenericParam<String, String> aParam = new MultiGenericParam<String, String>("value1", "value2");
MultiGenericParam<Integer, Double> dayOfWeekDegrees = new MultiGenericParam<Integer, Double>(1, 2.6);

Decidindo entre T, ? super T, e ? estende T #

A sintaxe para curingas limitados de genéricos Java, representando o tipo desconhecido por ?, é:

  • ? extends T representa um curinga limitado superior. O tipo desconhecido representa um tipo que deve ser um subtipo de T ou o próprio tipo T.

  • ? super T representa um curinga de limite inferior. O tipo desconhecido representa um tipo que deve ser um supertipo de T, ou o próprio tipo T.

Como regra geral, você deve usar

  • ? estende T se você precisar apenas de acesso de “leitura” (“entrada”)
  • ? super T se você precisar de acesso de “gravação” (“saída”)
  • T se você precisar de ambos (“modificar”)

Usar extends ou super geralmente é melhor porque torna seu código mais flexível (como em: permitir o uso de subtipos e supertipos), como você verá abaixo.

class Shoe {}
class IPhone {}
interface Fruit {}
class Apple implements Fruit {}
class Banana implements Fruit {}
class GrannySmith extends Apple {}

   public class FruitHelper {

        public void eatAll(Collection<? extends Fruit> fruits) {}

        public void addApple(Collection<? super Apple> apples) {}
}

O compilador agora poderá detectar certos usos inválidos:

 public class GenericsTest {
      public static void main(String[] args){
  FruitHelper fruitHelper = new FruitHelper() ;
    List<Fruit> fruits = new ArrayList<Fruit>();
    fruits.add(new Apple()); // Allowed, as Apple is a Fruit
    fruits.add(new Banana()); // Allowed, as Banana is a Fruit
    fruitHelper.addApple(fruits); // Allowed, as "Fruit super Apple"
    fruitHelper.eatAll(fruits); // Allowed

    Collection<Banana> bananas = new ArrayList<>();
    bananas.add(new Banana()); // Allowed
    //fruitHelper.addApple(bananas); // Compile error: may only contain Bananas!
    fruitHelper.eatAll(bananas); // Allowed, as all Bananas are Fruits

    Collection<Apple> apples = new ArrayList<>();
    fruitHelper.addApple(apples); // Allowed
    apples.add(new GrannySmith()); // Allowed, as this is an Apple
    fruitHelper.eatAll(apples); // Allowed, as all Apples are Fruits.
    
    Collection<GrannySmith> grannySmithApples = new ArrayList<>();
    fruitHelper.addApple(grannySmithApples); //Compile error: Not allowed.
                                   // GrannySmith is not a supertype of Apple
    apples.add(new GrannySmith()); //Still allowed, GrannySmith is an Apple
    fruitHelper.eatAll(grannySmithApples);//Still allowed, GrannySmith is a Fruit

    Collection<Object> objects = new ArrayList<>();
    fruitHelper.addApple(objects); // Allowed, as Object super Apple
    objects.add(new Shoe()); // Not a fruit
    objects.add(new IPhone()); // Not a fruit
    //fruitHelper.eatAll(objects); // Compile error: may contain a Shoe, too!
}

Escolhendo o T, ? super T ou ? extends T é necessário para permitir o uso com subtipos. O compilador pode então garantir a segurança do tipo; você não deve precisar converter (o que não é seguro de tipo e pode causar erros de programação) se você usá-los corretamente.

Se não for fácil de entender, lembre-se da regra do PECS:

Pprodutor usa “Extends” e Consumer usa “Super”.

(O produtor tem apenas acesso de gravação e o consumidor tem apenas acesso de leitura)

O diamante #

O Java 7 introduziu o Diamond1 para remover alguns clichês em torno da instanciação de classe genérica. Com Java 7+ você pode escrever:

List<String> list = new LinkedList<>();

Onde você tinha que escrever nas versões anteriores, isso:

List<String> list = new LinkedList<String>();

Uma limitação é para [Classes Anônimas][2], onde você ainda deve fornecer o parâmetro de tipo na instanciação:

// This will compile:

Comparator<String> caseInsensitiveComparator = new Comparator<String>() {
    @Override
    public int compare(String s1, String s2) {
        return s1.compareToIgnoreCase(s2);
    }
};

// But this will not:

Comparator<String> caseInsensitiveComparator = new Comparator<>() {
    @Override
    public int compare(String s1, String s2) {
        return s1.compareToIgnoreCase(s2);
    }
};

Embora o uso do diamante com [Anonymous Inner Classes][2] não seja suportado no Java 7 e 8, [ele será incluído como um novo recurso no Java 9][3].

Nota de rodapé:

1 - Algumas pessoas chamam o uso do <> de “diamante operador”. Isso está incorreto. O diamante não se comporta como um operador e não é descrito ou listado em nenhum lugar no JLS ou nos Tutoriais Java (oficiais) como um operador. De fato, <> nem é um token Java distinto. Em vez disso, é um token < seguido por um token >, e é legal (embora de estilo ruim) ter espaços em branco ou comentários entre os dois. O JLS e os Tutoriais referem-se consistentemente a <> como “o diamante”, e esse é, portanto, o termo correto para ele.

[2]: https://www.wikiod.com/pt/java/classes-aninhadas-e-internas#Classes internas anônimas [3]: https://bugs.openjdk.java.net/browse/JDK-8062373

Declarando um método genérico #

Os métodos também podem ter parâmetros do tipo genérico.

public class Example {

    // The type parameter T is scoped to the method
    // and is independent of type parameters of other methods.
    public <T> List<T> makeList(T t1, T t2) {
        List<T> result = new ArrayList<T>();
        result.add(t1);
        result.add(t2);
        return result;
    }

    public void usage() {
        List<String> listString = makeList("Jeff", "Atwood");
        List<Integer> listInteger = makeList(1, 2);
    }
}

Observe que não precisamos passar um argumento de tipo real para um método genérico. O compilador infere o argumento de tipo para nós, com base no tipo de destino (por exemplo, a variável à qual atribuímos o resultado) ou nos tipos dos argumentos reais. Ele geralmente inferirá o argumento de tipo mais específico que tornará a chamada de tipo correto.

Às vezes, embora raramente, pode ser necessário substituir essa inferência de tipo com argumentos de tipo explícitos:

void usage() {
    consumeObjects(this.<Object>makeList("Jeff", "Atwood").stream());
}

void consumeObjects(Stream<Object> stream) { ... }

É necessário neste exemplo porque o compilador não pode “olhar para frente” para ver que Object é desejado para T depois de chamar stream() e de outra forma inferiria String com base nos argumentos makeList. Observe que a linguagem Java não suporta a omissão da classe ou objeto no qual o método é chamado (this no exemplo acima) quando os argumentos de tipo são fornecidos explicitamente.

Requer vários limites superiores (“estende A e B”) #

Você pode exigir um tipo genérico para estender vários limites superiores.

Exemplo: queremos ordenar uma lista de números, mas Number não implementa Comparable.

public <T extends Number & Comparable<T>> void sortNumbers( List<T> n ) {
  Collections.sort( n );
}

Neste exemplo, T deve estender Number e implementar Comparable<T> que deve caber em todas as implementações numéricas “normais” como Integer ou BigDecimal, mas não se encaixa nas mais exóticas como Listrado64.

Como a herança múltipla não é permitida, você pode usar no máximo uma classe como limite e deve ser a primeira listada. Por exemplo, <T estende Comparable<T> & Number> não é permitido porque Comparable é uma interface e não uma classe.

Benefícios da classe e interface genéricas #

O código que usa genéricos tem muitos benefícios em relação ao código não genérico. Abaixo estão os principais benefícios


Verificações de tipo mais fortes em tempo de compilação #

Um compilador Java aplica a verificação de tipo forte ao código genérico e emite erros se o código violar a segurança de tipo. Corrigir erros de tempo de compilação é mais fácil do que corrigir erros de tempo de execução, que podem ser difíceis de encontrar.


Eliminação de elencos #

O snippet de código a seguir sem genéricos requer conversão:

List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);

Quando reescrito para usar genéricos, o código não requer conversão:

List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);   // no cast

Permitindo que programadores implementem algoritmos genéricos #

Usando genéricos, os programadores podem implementar algoritmos genéricos que funcionam em coleções de diferentes tipos, podem ser personalizados e são seguros para tipos e fáceis de ler.

Instanciando um tipo genérico #

Devido ao apagamento de tipo, o seguinte não funcionará:

public <T> void genericMethod() {
    T t = new T(); // Can not instantiate the type T.
}

O tipo T é apagado. Já que, em tempo de execução, a JVM não sabe o que T originalmente era, ela não sabe qual construtor chamar.

Soluções alternativas #

  1. Passando a classe T ao chamar genericMethod:

    public <T> void genericMethod(Class<T> cls) {
    try {
        T t = cls.newInstance();
    } catch (InstantiationException | IllegalAccessException e) {
         System.err.println("Could not instantiate: " + cls.getName());
    }
}

    genericMethod(String.class);

    Which throws exceptions, since there is no way to know if the passed class has an accessible default constructor.

  1. Passando uma [referência](https://www.wikiod.com/pt/java/expressoes-lambda#Referências de métodos) para o construtor de T:

    public <T> void genericMethod(Supplier<T> cons) {
    T t = cons.get();
}

    genericMethod(String::new);

Obtém a classe que satisfaz o parâmetro genérico em tempo de execução #

Muitos parâmetros genéricos não vinculados, como aqueles usados ​​em um método estático, não podem ser recuperados em tempo de execução (consulte Outros Threads em Erasure). No entanto, há uma estratégia comum empregada para acessar o tipo que satisfaz um parâmetro genérico em uma classe em tempo de execução. Isso permite código genérico que depende do acesso ao tipo sem ter que encadear informações de tipo em cada chamada.

Fundo

A parametrização genérica em uma classe pode ser inspecionada criando uma classe interna anônima. Esta classe irá capturar as informações de tipo. Em geral, esse mecanismo é chamado de tokens de supertipo, que são detalhados em post do blog de Neal Gafter.

Implementações

Três implementações comuns em Java são:

  • TypeToken do Guava
  • ParameterizedTypeReference do Spring
  • Jackson’s TypeReference

Exemplo de uso

public class DataService<MODEL_TYPE> {
     private final DataDao dataDao = new DataDao();
     private final Class<MODEL_TYPE> type = (Class<MODEL_TYPE>) new TypeToken<MODEL_TYPE>
                                                                (getClass()){}.getRawType();
     public List<MODEL_TYPE> getAll() {
         return dataDao.getAllOfType(type);
    }
}

// the subclass definitively binds the parameterization to User
// for all instances of this class, so that information can be 
// recovered at runtime
public class UserService extends DataService<User> {}

public class Main {
    public static void main(String[] args) {
          UserService service = new UserService();
          List<User> users = service.getAll();
    }
}

Criando uma classe genérica limitada #

Você pode restringir os tipos válidos usados ​​em uma classe genérica limitando esse tipo na definição de classe. Dada a seguinte hierarquia de tipos simples:

public abstract class Animal {
    public abstract String getSound();
}

public class Cat extends Animal {
    public String getSound() {
        return "Meow";
    }
}

public class Dog extends Animal {
    public String getSound() {
        return "Woof";
    }
}

Sem genéricos limitados, não podemos criar uma classe de contêiner que seja genérica e saiba que cada elemento é um animal:

public class AnimalContainer<T> {

    private Collection<T> col;

    public AnimalContainer() {
        col = new ArrayList<T>();
    }

    public void add(T t) {
        col.add(t);
    }

    public void printAllSounds() {
        for (T t : col) {
            // Illegal, type T doesn't have makeSound()
            // it is used as an java.lang.Object here
            System.out.println(t.makeSound()); 
        }
    }
}

Com o limite genérico na definição de classe, isso agora é possível.

public class BoundedAnimalContainer<T extends Animal> { // Note bound here.

    private Collection<T> col;

    public BoundedAnimalContainer() {
        col = new ArrayList<T>();
    }

    public void add(T t) {
        col.add(t);
    }

    public void printAllSounds() {
        for (T t : col) {
            // Now works because T is extending Animal
            System.out.println(t.makeSound()); 
        }
    }
}

Isso também restringe as instanciações válidas do tipo genérico:

// Legal
AnimalContainer<Cat> a = new AnimalContainer<Cat>();

// Legal
AnimalContainer<String> a = new AnimalContainer<String>();

// Legal because Cat extends Animal
BoundedAnimalContainer<Cat> b = new BoundedAnimalContainer<Cat>();

// Illegal because String doesn't extends Animal
BoundedAnimalContainer<String> b = new BoundedAnimalContainer<String>();

Fazendo referência ao tipo genérico declarado dentro de sua própria declaração #

Como você usa uma instância de um tipo genérico herdado (possivelmente mais) dentro de uma declaração de método no próprio tipo genérico que está sendo declarado? Este é um dos problemas que você enfrentará quando se aprofundar um pouco nos genéricos, mas ainda assim bastante comum.

Suponha que temos um tipo DataSeries<T> (interface aqui), que define uma série de dados genérica contendo valores do tipo T. É complicado trabalhar com esse tipo diretamente quando queremos realizar muitas operações com, por exemplo, valores double, então definimos DoubleSeries estende DataSeries<Double>. Agora suponha que o tipo DataSeries<T> original tenha um método add(values) que adiciona outra série do mesmo tamanho e retorna uma nova. Como impomos o tipo de values e o tipo de retorno para ser DoubleSeries em vez de DataSeries<Double> em nossa classe derivada?

O problema pode ser resolvido adicionando um parâmetro de tipo genérico referindo-se e estendendo o tipo que está sendo declarado (aplicado a uma interface aqui, mas o mesmo significa classes):

public interface DataSeries<T, DS extends DataSeries<T, DS>> {
    DS add(DS values);
    List<T> data();
}

Aqui T representa o tipo de dados que a série contém, por exemplo Double e DS a própria série. Um tipo herdado (ou tipos) agora pode ser facilmente implementado substituindo o parâmetro mencionado acima por um tipo derivado correspondente, produzindo assim uma definição concreta baseada em Double da forma:

public interface DoubleSeries extends DataSeries<Double, DoubleSeries> {
    static DoubleSeries instance(Collection<Double> data) {
        return new DoubleSeriesImpl(data);
    }
}

Neste momento, até mesmo um IDE implementará a interface acima com os tipos corretos, que, após um pouco de preenchimento de conteúdo, pode ficar assim:

class DoubleSeriesImpl implements DoubleSeries {
    private final List<Double> data;

    DoubleSeriesImpl(Collection<Double> data) {
        this.data = new ArrayList<>(data);
    }

    @Override
    public DoubleSeries add(DoubleSeries values) {
        List<Double> incoming = values != null ? values.data() : null;
        if (incoming == null || incoming.size() != data.size()) {
            throw new IllegalArgumentException("bad series");
        }
        List<Double> newdata = new ArrayList<>(data.size());
        for (int i = 0; i < data.size(); i++) {
            newdata.add(this.data.get(i) + incoming.get(i)); // beware autoboxing
        }
        return DoubleSeries.instance(newdata);
    }

    @Override
    public List<Double> data() {
        return Collections.unmodifiableList(data);
    }
}

Como você pode ver, o método add é declarado como DoubleSeries add(DoubleSeries values) e o compilador está feliz.

O padrão pode ser aninhado ainda mais, se necessário.

Associando parâmetro genérico a mais de 1 tipo #

Parâmetros genéricos também podem ser vinculados a mais de um tipo usando a sintaxe T extends Type1 & Type2 & ... .

Digamos que você queira criar uma classe cujo tipo genérico deve implementar tanto Flushable quanto Closeable, você pode escrever

class ExampleClass<T extends Flushable & Closeable> {
}

Agora, a ExampleClass só aceita como parâmetros genéricos, tipos que implementam Flushable e Closeable.

ExampleClass<BufferedWriter> arg1; // Works because BufferedWriter implements both Flushable and Closeable

ExampleClass<Console> arg4; // Does NOT work because Console only implements Flushable
ExampleClass<ZipFile> arg5; // Does NOT work because ZipFile only implements Closeable

ExampleClass<Flushable> arg2; // Does NOT work because Closeable bound is not satisfied.
ExampleClass<Closeable> arg3; // Does NOT work because Flushable bound is not satisfied.

Os métodos de classe podem optar por inferir argumentos de tipo genérico como Closeable ou Flushable.

class ExampleClass<T extends Flushable & Closeable> {
    /* Assign it to a valid type as you want. */
    public void test (T param) {
        Flushable arg1 = param; // Works
        Closeable arg2 = param; // Works too.
    }

    /* You can even invoke the methods of any valid type directly. */
    public void test2 (T param) {
        param.flush(); // Method of Flushable called on T and works fine.
        param.close(); // Method of Closeable called on T and works fine too.
    }
}

Observação: #

Você não pode vincular o parâmetro genérico a nenhum dos tipos usando a cláusula OR (|). Apenas a cláusula AND (&) é suportada. O tipo genérico pode estender apenas uma classe e muitas interfaces. A aula deve ser colocado no início da lista.

Usando genéricos para transmitir automaticamente #

Com genéricos, é possível retornar o que o chamador espera:

private Map<String, Object> data;
public <T> T get(String key) {
    return (T) data.get(key);
}

O método irá compilar com um aviso. O código é realmente mais seguro do que parece porque o tempo de execução do Java fará uma conversão quando você o usar:

Bar bar = foo.get("bar");

É menos seguro quando você usa tipos genéricos:

List<Bar> bars = foo.get("bars");

Aqui, a conversão funcionará quando o tipo retornado for qualquer tipo de List (ou seja, retornar List<String> não acionaria uma ClassCastException; você eventualmente a obteria ao retirar elementos da lista).

Para contornar esse problema, você pode criar uma API que usa chaves digitadas:

public final static Key<List<Bar>> BARS = new Key<>("BARS");

junto com este método put():

public <T> T put(Key<T> key, T value);

Com esta abordagem, você não pode colocar o tipo errado no mapa, então o resultado sempre estará correto (a menos que você crie acidentalmente duas chaves com o mesmo nome, mas tipos diferentes).

Uso de instanceof com Generics #

Usando genéricos para definir o tipo em instanceof

Considere a seguinte classe genérica Example declarada com o parâmetro formal <T>:

class Example<T> {
    public boolean isTypeAString(String s) {
        return s instanceof T; // Compilation error, cannot use T as class type here
    }
}

Isso sempre dará um erro de compilação porque assim que o compilador compila o fonte Java em Java bytecode ele aplica um processo conhecido como type erasure, que converte todo código genérico em código não genérico, impossibilitando a distinção entre os tipos T em tempo de execução. O tipo usado com instanceof deve ser reifiable, o que significa que todas as informações sobre o tipo devem estar disponíveis em tempo de execução, e isso geralmente não é o caso de tipos genéricos.

A classe a seguir representa a aparência de duas classes diferentes de Example, Example<String> e Example<Number> depois que os genéricos foram removidos por type erasure:

class Example { // formal parameter is gone
    public boolean isTypeAString(String s) {
        return s instanceof Object; // Both <String> and <Number> are now Object
    }
}

Como os tipos se foram, não é possível para a JVM saber qual tipo é T.

Exceção à regra anterior

Você sempre pode usar curinga ilimitado (?) para especificar um tipo no instanceof da seguinte forma:

    public boolean isAList(Object obj) {
        return obj instanceof List<?>;
    }

Isso pode ser útil para avaliar se uma instância obj é uma List ou não:

System.out.println(isAList("foo")); // prints false
System.out.println(isAList(new ArrayList<String>()); // prints true
System.out.println(isAList(new ArrayList<Float>()); // prints true

Na verdade, o curinga ilimitado é considerado um tipo reificável.

Usando uma instância genérica com instanceof

O outro lado da moeda é que usar uma instância t de T com instanceof é legal, conforme mostrado no exemplo a seguir:

class Example<T> {
    public boolean isTypeAString(T t) {
        return t instanceof String; // No compilation error this time
    }
}

porque após o apagamento do tipo, a classe ficará assim:

class Example { // formal parameter is gone
    public boolean isTypeAString(Object t) {
        return t instanceof String; // No compilation error this time
    }
}

Visto que, mesmo que o apagamento de tipo aconteça de qualquer maneira, agora a JVM pode distinguir entre diferentes tipos na memória, mesmo que usem o mesmo tipo de referência (Object), conforme mostra o seguinte trecho:

Object obj1 = new String("foo"); // reference type Object, object type String
Object obj2 = new Integer(11); // reference type Object, object type Integer
System.out.println(obj1 instanceof String); // true
System.out.println(obj2 instanceof String); // false, it's an Integer, not a String

Diferentes maneiras de implementar uma Interface Genérica (ou estender uma Classe Genérica) #

Suponha que a seguinte interface genérica tenha sido declarada:

public interface MyGenericInterface<T> {
    public void foo(T t);
}

Abaixo estão listadas as possíveis formas de implementá-lo.

Implementação de classe não genérica com um tipo específico

Escolha um tipo específico para substituir o parâmetro de tipo formal <T> de MyGenericClass e implemente-o, como o exemplo a seguir faz:

public class NonGenericClass implements MyGenericInterface<String> {
    public void foo(String t) { } // type T has been replaced by String
}

Esta classe lida apenas com String, e isso significa que usar MyGenericInterface com parâmetros diferentes (por exemplo, Integer, Object etc.) não compilará, como mostra o trecho a seguir:

NonGenericClass myClass = new NonGenericClass();
myClass.foo("foo_string"); // OK, legal
myClass.foo(11); // NOT OK, does not compile
myClass.foo(new Object()); // NOT OK, does not compile

Implementação de classe genérica

Declare outra interface genérica com o parâmetro de tipo formal <T> que implementa MyGenericInterface, como segue:

public class MyGenericSubclass<T> implements MyGenericInterface<T> {
    public void foo(T t) { } // type T is still the same
    // other methods...
}

Observe que um parâmetro de tipo formal diferente pode ter sido usado, como segue:

public class MyGenericSubclass<U> implements MyGenericInterface<U> { // equivalent to the previous declaration
    public void foo(U t) { }
    // other methods...
}

Implementação de classe de tipo bruto

Declare uma classe não genérica que implementa MyGenericInteface como um tipo bruto (sem usar genérico), como segue:

public class MyGenericSubclass implements MyGenericInterface {
    public void foo(Object t) { } // type T has been replaced by Object
    // other possible methods
}

Esta forma não é recomendada, pois não é 100% segura em tempo de execução porque mistura tipo bruto (da subclasse) com genéricos (da interface) e também é confuso. Compiladores Java modernos irão emitir um aviso com este tipo de implementação, no entanto o código - por razões de compatibilidade com JVM mais antiga (1.4 ou anterior) - irá compilar.

Todas as formas listadas acima também são permitidas ao usar uma classe genérica como supertipo em vez de uma interface genérica.
