O comando javac é usado para compilar arquivos fonte Java para arquivos bytecode. Os arquivos de bytecode são independentes de plataforma. Isso significa que você pode compilar seu código em um tipo de hardware e sistema operacional e, em seguida, executar o código em qualquer outra plataforma que suporte Java.

O comando javac está incluído nas distribuições do Java Development Kit (JDK).

O compilador Java e o restante da cadeia de ferramentas Java padrão impõem as seguintes restrições ao código:

  • O código fonte é mantido em arquivos com o sufixo “.java”
  • Bytecodes são mantidos em arquivos com o sufixo “.class”
  • Para arquivos de origem e bytecode no sistema de arquivos, os nomes de caminho do arquivo devem refletir a nomenclatura do pacote e da classe.

Nota: O compilador javac não deve ser confundido com o https://www.wikiod.com/pt/java/compilador-just-in-time-jit que compila bytecodes para código nativo.

O comando ‘javac’ - iniciando #

Exemplo simples #

Assumindo que o “HelloWorld.java” contém a seguinte fonte Java:

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

(Para obter uma explicação do código acima, consulte https://www.wikiod.com/pt/java/introducao-a-linguagem-java .)

Podemos compilar o arquivo acima usando este comando:

$ javac HelloWorld.java

Isso produz um arquivo chamado “HelloWorld.class”, que podemos executar da seguinte forma:

$ java HelloWorld
Hello world!

Os pontos-chave a serem observados neste exemplo são:

  1. O nome do arquivo de origem “HelloWorld.java” deve corresponder ao nome da classe no arquivo de origem… que é HelloWorld. Se eles não corresponderem, você receberá um erro de compilação.
  2. O nome do arquivo bytecode “HelloWorld.class” corresponde ao nome da classe. Se você renomeasse o “HelloWorld.class”, obteria um erro ao tentar executá-lo.
  3. Ao executar um aplicativo Java usando java, você fornece o nome da classe NÃO o nome do arquivo bytecode.

Exemplo com pacotes #

A maioria dos códigos Java práticos usa pacotes para organizar o namespace para classes e reduzir o risco de colisão acidental de nomes de classe.

Se quiséssemos declarar a classe HelloWorld em uma chamada de pacote com.example, o “HelloWorld.java” conteria a seguinte fonte Java:

package com.example;

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello world!");
    }
}

Este arquivo de código-fonte precisa ser armazenado em uma árvore de diretórios cuja estrutura corresponde à nomenclatura do pacote.

.    # the current directory (for this example)
|
 ----com
     |
      ----example
          |
           ----HelloWorld.java

Podemos compilar o arquivo acima usando este comando:

$ javac com/example/HelloWorld.java

Isso produz um arquivo chamado “com/example/HelloWorld.class”; ou seja, após a compilação, a estrutura do arquivo deve ficar assim:

.    # the current directory (for this example)
|
 ----com
     |
      ----example
          |
           ----HelloWorld.java
           ----HelloWorld.class

Podemos então executar a aplicação da seguinte forma:

$ java com.example.HelloWorld
Hello world!

Pontos adicionais a serem observados neste exemplo são:

  1. A estrutura do diretório deve corresponder à estrutura do nome do pacote.
  2. Ao executar a classe, o nome completo da classe deve ser fornecido; ou seja, “com.example.HelloWorld” não “HelloWorld”.
  3. Você não precisa compilar e executar código Java fora do diretório atual. Estamos fazendo isso aqui apenas para ilustração.

Compilando vários arquivos de uma vez com ‘javac’. #

Se seu aplicativo consiste em vários arquivos de código-fonte (e a maioria contém!), você pode compilá-los um de cada vez. Alternativamente, você pode compilar vários arquivos ao mesmo tempo listando os nomes dos caminhos:

$ javac Foo.java Bar.java

ou usando a funcionalidade curinga de nome de arquivo do shell de comando ….

$ javac *.java
$ javac com/example/*.java
$ javac */**/*.java #Only works on Zsh or with globstar enabled on your shell

Isso compilará todos os arquivos de origem Java no diretório atual, no diretório “com/example” e recursivamente nos diretórios filhos, respectivamente. Uma terceira alternativa é fornecer uma lista de nomes de arquivos de origem (e opções do compilador) como um arquivo. Por exemplo:

$ javac @sourcefiles

onde o arquivo sourcefiles contém:

Foo.java
Bar.java
com/example/HelloWorld.java

Nota: compilar código como este é apropriado para pequenos projetos de uma pessoa e para programas únicos. Além disso, é aconselhável selecionar e usar uma ferramenta de construção Java. Alternativamente, a maioria dos programadores usa um IDE Java (por exemplo, NetBeans, eclipse, IntelliJ IDEA) que oferece um compilador embutido e construção incremental de “projetos”.

Opções ‘javac’ comumente usadas #

Aqui estão algumas opções para o comando javac que provavelmente serão úteis para você

  • A opção -d define um diretório de destino para escrever os arquivos “.class”.
  • A opção -sourcepath define um caminho de pesquisa do código-fonte.
  • A opção -cp ou -classpath define o caminho de busca para encontrar classes externas e compiladas anteriormente. Para obter mais informações sobre o caminho de classe e como especificá-lo, consulte o tópico https://www.wikiod.com/pt/java/o-caminho-de-classe
  • A opção -version imprime as informações da versão do compilador.

Uma lista mais completa de opções do compilador será descrita em um exemplo separado.

A referência definitiva para o comando javac é a página de manual do Oracle para javac.

Compilando para uma versão diferente do Java #

A linguagem de programação Java (e seu tempo de execução) passou por inúmeras mudanças desde seu lançamento desde seu lançamento público inicial. Essas mudanças incluem:

  • Mudanças na sintaxe e semântica da linguagem de programação Java
  • Mudanças nas APIs fornecidas pelas bibliotecas de classes padrão Java.
  • Mudanças no conjunto de instruções Java (bytecode) e no formato classfile.

Com poucas exceções (por exemplo, a palavra-chave enum, alterações em algumas classes “internas”, etc.), essas alterações são compatíveis com versões anteriores.

  • Um programa Java que foi compilado usando uma versão mais antiga da cadeia de ferramentas Java será executado em uma plataforma Java de versão mais recente sem recompilação.
  • Um programa Java que foi escrito em uma versão mais antiga do Java compilará com sucesso com um novo compilador Java.

Compilando Java antigo com um compilador mais recente #

Se você precisar (re)compilar código Java mais antigo em uma plataforma Java mais recente para executar na plataforma mais nova, geralmente não é necessário fornecer nenhum sinalizador de compilação especial. Em alguns casos (por exemplo, se você usou enum como identificador), você pode usar a opção -source para desabilitar a nova sintaxe. Por exemplo, dada a seguinte classe:

public class OldSyntax {
    private static int enum;  // invalid in Java 5 or later
}

o seguinte é necessário para compilar a classe usando um compilador Java 5 (ou posterior):

$ javac -source 1.4 OldSyntax.java

Compilando para uma plataforma de execução mais antiga #

Se você precisar compilar Java para rodar em plataformas Java mais antigas, a abordagem mais simples é instalar um JDK para a versão mais antiga que você precisa suportar e usar o compilador desse JDK em suas compilações.

Você também pode compilar com um compilador Java mais recente, mas isso é complicado. Em primeiro lugar, existem algumas pré-condições importantes que devem ser satisfeitas:

  • O código que você está compilando não deve usar construções de linguagem Java que não estavam disponíveis na versão do Java que você está direcionando.
  • O código não deve depender de classes Java padrão, campos, métodos e assim por diante que não estavam disponíveis nas plataformas mais antigas.
  • Bibliotecas de terceiros das quais o código depende também devem ser criadas para a plataforma mais antiga e disponíveis em tempo de compilação e em tempo de execução.

Se as pré-condições forem atendidas, você pode recompilar o código para uma plataforma mais antiga usando a opção -target. Por exemplo,

$ javac -target 1.4 SomeClass.java

compilará a classe acima para produzir bytecodes compatíveis com Java 1.4 ou JVM posterior. (Na verdade, a opção -source implica um -target compatível, então javac -source 1.4 ... teria o mesmo efeito. A relação entre -source e -target é descrita na documentação Oracle.)

Dito isto, se você simplesmente usar -target ou -source, você ainda estará compilando contra as bibliotecas de classes padrão fornecidas pelo JDK do compilador. Se você não tomar cuidado, pode acabar com classes com a versão correta do bytecode, mas com dependências de APIs que não estão disponíveis. A solução é usar a opção -bootclasspath. Por exemplo:

$ javac -target 1.4 --bootclasspath path/to/java1.4/rt.jar SomeClass.java

compilará em um conjunto alternativo de bibliotecas de tempo de execução. Se a classe que está sendo compilada tiver dependências (acidentais) de bibliotecas mais recentes, isso causará erros de compilação.
