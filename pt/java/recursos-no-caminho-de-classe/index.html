Java permite a recuperação de recursos baseados em arquivos armazenados dentro de um JAR junto com classes compiladas. Este tópico se concentra em carregar esses recursos e disponibilizá-los para seu código.

Um recurso são dados semelhantes a arquivos com um nome semelhante a um caminho, que reside no caminho de classe. O uso mais comum de recursos é agrupar imagens de aplicativos, sons e dados somente leitura (como configuração padrão).

Os recursos podem ser acessados ​​com o ClassLoader.getResource e ClassLoader .getResourceAsStream métodos. O caso de uso mais comum é ter recursos colocados no mesmo pacote que a classe que os lê; o Class.getResource e o [Class.getResourceAsStream](http ://docs.oracle.com/javase/8/docs/api/java/lang/Class.html#getResourceAsStream-java.lang.String-) atendem a esse caso de uso comum.

A única diferença entre um método getResource e um método getResourceAsStream é que o primeiro retorna uma URL, enquanto o último abre essa URL e retorna um InputStream.

Os métodos do ClassLoader aceitam um nome de recurso semelhante a um caminho como argumento e pesquisam em cada local no caminho de classe do ClassLoader uma entrada correspondente a esse nome.

  • Se um local de caminho de classe for um arquivo .jar, uma entrada jar com o nome especificado será considerada uma correspondência.
  • Se um local de caminho de classe for um diretório, um arquivo relativo nesse diretório com o nome especificado será considerado uma correspondência.

O nome do recurso é semelhante à parte do caminho de uma URL relativa. Em todas as plataformas, ele usa barras (/) como separadores de diretório. Não deve começar com uma barra.

Os métodos correspondentes de Class são semelhantes, exceto:

  • O nome do recurso pode começar com uma barra, nesse caso essa barra inicial é removida e o restante do nome é passado para o método correspondente do ClassLoader.
  • Se o nome do recurso não começar com uma barra, ele será tratado como relativo à classe cujo método getResource ou getResourceAsStream está sendo chamado. O nome real do recurso se torna package/name, onde package é o nome do pacote ao qual a classe pertence, com cada ponto substituído por uma barra e name é o argumento original fornecido ao método.

Por exemplo:

package com.example;

public class ExampleApplication {
    public void readImage()
    throws IOException {

        URL imageURL = ExampleApplication.class.getResource("icon.png");

        // The above statement is identical to:
        // ClassLoader loader = ExampleApplication.class.getClassLoader();
        // URL imageURL = loader.getResource("com/example/icon.png");

        Image image = ImageIO.read(imageURL);
    }
}

Os recursos devem ser colocados em pacotes nomeados, em vez de na raiz de um arquivo .jar, pelo mesmo motivo que as classes são colocadas em pacotes: Para evitar colisões entre vários fornecedores. Por exemplo, se vários arquivos .jar estiverem no caminho de classe e mais de um deles contiver uma entrada config.properties em sua raiz, as chamadas para os métodos getResource ou getResourceAsStream retornarão o config.properties de qualquer .jar listado primeiro no caminho de classe. Este não é um comportamento previsível em ambientes onde a ordem do caminho de classe não está sob o controle direto do aplicativo, como Java EE.

Todos os métodos getResource e getResourceAsStream retornam null se o recurso especificado não existir. Como os recursos devem ser adicionados ao aplicativo em tempo de compilação, seus locais devem ser conhecidos quando o código estiver sendo escrito; uma falha em encontrar um recurso em tempo de execução geralmente é resultado de um erro do programador.

Os recursos são somente leitura. Não há como gravar em um recurso. Desenvolvedores iniciantes geralmente cometem o erro de supor que, como o recurso é um arquivo físico separado ao desenvolver em um IDE (como o Eclipse), será seguro tratá-lo como um arquivo físico separado no caso geral. No entanto, isso não está correto; os aplicativos são quase sempre distribuídos como arquivos, como arquivos .jar ou .war e, nesses casos, um recurso não será um arquivo separado e não será gravável. (O método getFile da classe URL não é uma solução alternativa para isso; apesar de seu nome, ele apenas retorna a parte do caminho de uma URL, que não é de forma alguma garantida como um nome de arquivo válido.)

Não há uma maneira segura de listar recursos em tempo de execução. Novamente, como os desenvolvedores são responsáveis ​​por adicionar arquivos de recursos ao aplicativo em tempo de compilação, os desenvolvedores já devem conhecer seus caminhos. Embora existam soluções alternativas, elas não são confiáveis ​​e eventualmente falharão.

Carregando configuração padrão #

Para ler as propriedades de configuração padrão:

package com.example;

public class ExampleApplication {
    private Properties getDefaults() throws IOException {
        Properties defaults = new Properties();

        try (InputStream defaultsStream =
            ExampleApplication.class.getResourceAsStream("config.properties")) {

            defaults.load(defaultsStream);
        }

        return defaults;
    }
}

Carregando uma imagem de um recurso #

Para carregar uma imagem agrupada:

package com.example;

public class ExampleApplication {
    private Image getIcon() throws IOException {
        URL imageURL = ExampleApplication.class.getResource("icon.png");
        return ImageIO.read(imageURL);
    }
}

Encontrando e lendo recursos usando um carregador de classe #

O carregamento de recursos em Java compreende as seguintes etapas:

  1. Localizando a Class ou ClassLoader que irá localizar o recurso.
  2. Encontrando o recurso.
  3. Obtendo o fluxo de bytes para o recurso.
  4. Lendo e processando o fluxo de bytes.
  5. Fechando o fluxo de bytes.

As últimas três etapas são normalmente realizadas passando a URL para um método de biblioteca ou construtor para carregar o recurso. Você normalmente usará um método getResource neste caso. Também é possível ler os dados do recurso no código do aplicativo. Você normalmente usará getResourceAsStream neste caso.

Caminhos de recursos absolutos e relativos #

Os recursos que podem ser carregados do classpath são indicados por um path. A sintaxe do caminho é semelhante a um caminho de arquivo UNIX/Linux. Consiste em nomes simples separados por caracteres de barra (/). Um caminho relativo começa com um nome e um caminho absoluto começa com um separador.

Como descrevem os exemplos do Classpath, o classpath de uma JVM define um namespace sobrepondo os namespaces dos diretórios e arquivos JAR ou ZIP no classpath. Quando um caminho absoluto é resolvido, os carregadores de classe interpretam o / inicial como significando a raiz do namespace. Por outro lado, um caminho relativo pode ser resolvido em relação a qualquer “pasta” no namespace. A pasta usada dependerá do objeto que você usa para resolver o caminho.

Obtendo uma classe ou carregador de classe #

Um recurso pode ser localizado usando um objeto Class ou um objeto ClassLoader. Um objeto Class pode resolver caminhos relativos, então você normalmente usará um deles se tiver um recurso relativo (classe). Existem várias maneiras de obter um objeto Class. Por exemplo:

  • Um literal de classe lhe dará o objeto Class para qualquer classe que você possa nomear no código-fonte Java; por exemplo. String.class fornece o objeto Class para o tipo String.

  • O Object.getClass() lhe dará o objeto Class para o tipo de qualquer objeto; por exemplo. "hello".getClass() é outra maneira de obter Class do tipo String.

  • O método Class.forName(String) irá (se necessário) carregar dinamicamente uma classe e retornar seu objeto Class; por exemplo. Class.forName("java.lang.String").

Um objeto ClassLoader normalmente é obtido chamando getClassLoader() em um objeto Class. Também é possível obter o carregador de classe padrão da JVM usando o método estático ClassLoader.getSystemClassLoader().

Os métodos get #

Depois de ter uma instância Class ou ClassLoader, você pode encontrar um recurso usando um dos seguintes métodos:

Métodos Descrição
ClassLoader.getResource(caminho)
ClassLoader.getResources(caminho)		 Retorna uma URL que representa a localização do recurso com o caminho fornecido.
ClassLoader.getResources(caminho)
Class.getResources(caminho)		 Retorna uma Enumeration<URL> fornecendo as URLs que podem ser usadas para localizar o recurso foo.bar; veja abaixo.
ClassLoader.getResourceAsStream(caminho)
Class.getResourceStream(caminho)		 Retorna um InputStream do qual você pode ler o conteúdo do recurso foo.bar como uma seqüência de bytes.

Notas:

  • A principal diferença entre as versões ClassLoader e Class dos métodos está na forma como os caminhos relativos são interpretados.

    • The Class methods resolve a relative path in the “folder” that corresponds to the classes package.
    • The ClassLoader methods treat relative paths as if they were absolute; i.e. the resolve them in the “root folder” of the classpath namespace.

  • Se o recurso solicitado (ou recursos) não puder ser encontrado, os métodos getResource e getResourceAsStreamretornamnull, e os métodos getResourcesretornam umaEnumeration` vazia.

  • Os URLs retornados serão resolvidos usando URL.toStream(). Eles podem ser URLs file: ou outros URLs convencionais, mas se o recurso residir em um arquivo JAR, eles serão URLs jar: que identificam o arquivo JAR e um recurso específico dentro dele.

  • Se seu código usa um método getResourceAsStream (ou URL.toStream()) para obter um InputStream, ele é responsável por fechar o objeto stream. A falha ao fechar o fluxo pode levar a um vazamento de recursos.

Carregando recurso de mesmo nome de vários JARs #

O recurso com o mesmo caminho e nome pode existir em mais de um arquivo JAR no caminho de classe. Casos comuns são recursos que seguem uma convenção ou que fazem parte de uma especificação de empacotamento. Exemplos de tais recursos são

  • META-INF/MANIFEST.MF
  • META-INF/beans.xml (especificação CDI)
  • Propriedades ServiceLoader contendo provedores de implementação

Para ter acesso a todos esses recursos em diferentes jars, é preciso usar um ClassLoader, que possui um método para isso. A ‘Enumeração’ retornada pode ser convenientemente convertida em uma ‘Lista’ usando uma função de Coleções.

Enumeration<URL> resEnum = MyClass.class.getClassLoader().getResources("META-INF/MANIFEST.MF");
ArrayList<URL> resources = Collections.list(resEnum);
