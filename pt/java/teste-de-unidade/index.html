  • O que é teste de unidade?

Nesta página

  • O que é teste de unidade?

O teste de unidade é parte integrante do desenvolvimento orientado a testes e um recurso importante para criar qualquer aplicativo robusto. Em Java, o teste de unidade é quase exclusivamente realizado usando bibliotecas e estruturas externas, a maioria das quais possui sua própria tag de documentação. Este stub serve para apresentar ao leitor as ferramentas disponíveis e sua respectiva documentação.

Estruturas de teste unitário #

Existem vários frameworks disponíveis para teste de unidade em Java. A opção mais popular de longe é JUnit. Está documentado sob o seguinte:

JUnit

JUnit4 - Tag proposta para recursos JUnit4; Ainda não implementado.

Existem outras estruturas de teste de unidade e têm documentação disponível:

TestNG

Ferramentas de teste unitário #

Existem várias outras ferramentas usadas para testes de unidade:

Mockito - estrutura Mocking; permite que os objetos sejam imitados. Útil para imitar o comportamento esperado de uma unidade externa dentro do teste de uma determinada unidade, para não vincular o comportamento da unidade externa aos testes de uma determinada unidade.

JBehave - BDD Framework. Permite que testes sejam vinculados a comportamentos do usuário (permitindo validação de requisitos/cenários). Nenhuma etiqueta de documentos disponível no momento da redação; aqui é um link externo.

O que é teste de unidade? #

Este é um pouco de um primer. É colocado principalmente porque a documentação é forçada a ter um exemplo, mesmo que seja um artigo de esboço. Se você já conhece os conceitos básicos de teste de unidade, sinta-se à vontade para pular para as observações, onde são mencionadas estruturas específicas.

O teste de unidade é garantir que um determinado módulo se comporte conforme o esperado. Em aplicações de grande escala, garantir a execução adequada dos módulos no vácuo é parte integrante da garantia da fidelidade da aplicação.

Considere o seguinte pseudo-exemplo (trivial):

public class Example {
  public static void main (String args[]) {
    new Example();
  }

  // Application-level test.
  public Example() {
    Consumer c = new Consumer();
    System.out.println("VALUE = " + c.getVal());
  }

  // Your Module.
  class Consumer {
    private Capitalizer c;
  
    public Consumer() {
      c = new Capitalizer();
    }

    public String getVal() {
      return c.getVal();
    }
  }

  // Another team's module.
  class Capitalizer {
    private DataReader dr;
  
    public Capitalizer() {
      dr = new DataReader();
    }

    public String getVal() {
      return dr.readVal().toUpperCase();
    }
  }

  // Another team's module.
  class DataReader {
    public String readVal() {
      // Refers to a file somewhere in your application deployment, or
      // perhaps retrieved over a deployment-specific network.
      File f; 
      String s = "data";
      // ... Read data from f into s ...
      return s;
    }
  }
}

Portanto, este exemplo é trivial; DataReader pega os dados de um arquivo, passa para o Capitalizer, que converte todos os caracteres para maiúsculas, que então são passados ​​para o Consumer. Mas o DataReader está fortemente vinculado ao nosso ambiente de aplicação, então adiamos o teste dessa cadeia até que estejamos prontos para implantar uma versão de teste.

Agora, suponha que, em algum lugar ao longo do caminho em uma versão, por razões desconhecidas, o método getVal() em Capitalizer mudou de retornar uma String toUpperCase() para uma String toLowerCase():

  // Another team's module.
  class Capitalizer {
    ...

    public String getVal() {
      return dr.readVal().toLowerCase();
    }
  }

Claramente, isso quebra o comportamento esperado. Mas, por causa dos processos árduos envolvidos com a execução do DataReader, não perceberemos isso até nossa próxima implantação de teste. Assim, dias/semanas/meses se passam com esse bug em nosso sistema, e então o gerente de produto vê isso e instantaneamente se volta para você, o líder da equipe associado ao Consumidor. “Por que isso está acontecendo? O que vocês mudaram?” Obviamente, você é ignorante. Você não tem ideia do que está acontecendo. Você não alterou nenhum código que deveria tocar nisso; por que está quebrado de repente?

Eventualmente, após discussão entre as equipes e colaboração, o problema é rastreado e o problema resolvido. Mas, isso levanta a questão; como isso poderia ter sido evitado?

Há duas coisas óbvias:

Testes precisam ser automatizados #

Nossa dependência de testes manuais deixou esse bug passar despercebido por muito tempo. Precisamos de uma maneira de automatizar o processo pelo qual os bugs são introduzidos instantaneamente. Não daqui a 5 semanas. Não daqui a 5 dias. Não daqui a 5 minutos. Agora mesmo.

Você tem que entender que, neste exemplo, eu expressei um bug muito trivial que foi introduzido e despercebido. Em uma aplicação industrial, com dezenas de módulos sendo atualizados constantemente, eles podem se infiltrar em todos os lugares. Você conserta algo com um módulo, apenas para perceber que o próprio comportamento que você “consertou” foi invocado de alguma maneira em outro lugar (interna ou externamente).

Sem validação rigorosa, as coisas vão se infiltrar no sistema. É possível que, se negligenciado o suficiente, isso resultará em muito trabalho extra tentando corrigir as alterações (e depois corrigindo essas correções etc.), que um produto realmente aumentará o trabalho restante à medida que o esforço é colocado em isto. Você não quer estar nesta situação.

Os testes precisam ser refinados #

O segundo problema observado em nosso exemplo acima é a quantidade de tempo que levou para rastrear o bug. O gerente de produto fez um ping para você quando os testadores perceberam, você investigou e descobriu que o Capitalizer estava retornando dados aparentemente ruins, você fez um ping para a equipe do Capitalizer com suas descobertas, eles investigaram, etc. etc. etc.

O mesmo ponto que fiz acima sobre a quantidade e a dificuldade desse exemplo trivial se mantém aqui. Obviamente, qualquer pessoa razoavelmente bem versada em Java poderia encontrar rapidamente o problema apresentado. Mas muitas vezes é muito, muito mais difícil rastrear e comunicar problemas. Talvez a equipe do Capitalizer tenha fornecido um JAR sem fonte. Talvez eles estejam localizados do outro lado do mundo e os horários de comunicação sejam muito limitados (talvez para e-mails que são enviados uma vez por dia). Isso pode resultar em bugs que levam semanas ou mais para rastrear (e, novamente, pode haver vários deles para uma determinada versão).

Para mitigar isso, queremos testes rigorosos no nível mais fino possível (você também quer testes de granularidade grosseira para garantir que os módulos interajam corretamente, mas esse não é nosso foco aqui). Queremos especificar rigorosamente como todas as funcionalidades voltadas para o exterior (no mínimo) operam e testar essa funcionalidade.

Digite o teste de unidade #

Imagine se tivéssemos um teste, garantindo especificamente que o método getVal() de Capitalizer retornasse uma string em maiúscula para uma determinada string de entrada. Além disso, imagine que o teste foi executado antes mesmo de confirmarmos qualquer código. O bug introduzido no sistema (ou seja, toUpperCase() sendo substituído por toLowerCase()) não causaria problemas porque o bug nunca seria introduzido no sistema. Nós o pegaríamos em um teste, o desenvolvedor (esperançosamente) perceberia seu erro e uma solução alternativa seria alcançada sobre como introduzir o efeito pretendido.

Há algumas omissões feitas aqui sobre como implementar esses testes, mas elas são abordadas na documentação específica do framework (link nas observações). Esperamos que isso sirva como um exemplo de por que o teste de unidade é importante.
