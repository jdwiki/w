Guia de estilo oficial da Oracle para a linguagem de programação Java é um padrão seguido pelos desenvolvedores da Oracle e recomendado para ser seguido por qualquer outro desenvolvedor Java. Abrange nomes de arquivos, organização de arquivos, recuo, comentários, declarações, declarações, espaço em branco, convenções de nomenclatura, práticas de programação e inclui um exemplo de código.

  • Os exemplos acima seguem rigorosamente o novo guia de estilo oficial da Oracle. Eles são, em outras palavras, não subjetivamente compostos pelos autores desta página.

  • O guia de estilo oficial foi cuidadosamente escrito para ser compatível com o guia de estilo original e a maioria do código na selva.

  • O guia de estilo oficial foi peer [revisado](http://mail.openjdk.java.net /pipermail/discuss/2015-August/003771.html) por, entre outros, Brian Goetz (Java Language Architect) e Mark Reinhold (Chief Architect of the Java Platform).

  • Os exemplos são não normativos; Embora eles pretendam ilustrar a maneira correta de formatar o código, pode haver outras maneiras de formatar o código corretamente. Este é um princípio geral: pode haver várias maneiras de formatar o código, todas seguindo as diretrizes oficiais.

Convenções de nomenclatura #

Nomes de pacotes #

  • Os nomes dos pacotes devem ser todos em minúsculas, sem sublinhados ou outros caracteres especiais.
  • Os nomes dos pacotes começam com a parte de autoridade invertida do endereço web da empresa do desenvolvedor. Esta parte pode ser seguida por uma subestrutura de pacote dependente da estrutura do projeto/programa.
  • Não use a forma plural. Siga a convenção da API padrão que usa, por exemplo, java.lang.annotation e não java.lang.annotations.
  • Exemplos: com.suaempresa.widget.button, com.suaempresa.core.api

Nomes de Classe, Interface e Enum #

  • Nomes de classe e enum normalmente devem ser substantivos.
  • Nomes de interface normalmente devem ser substantivos ou adjetivos que terminam com …able.
  • Use maiúsculas e minúsculas com a primeira letra de cada palavra em maiúscula (ou seja, CamelCase).
  • Combine a expressão regular ^[A-Z][a-zA-Z0-9]*$.
  • Use palavras inteiras e evite usar abreviaturas, a menos que a abreviação seja mais usada do que a forma longa.
  • Formate uma abreviatura como uma palavra se for parte de um nome de classe mais longo.
  • Exemplos: ArrayList, BigInteger, ArrayIndexOutOfBoundsException, Iterable.

Nomes de Métodos #

Os nomes dos métodos geralmente devem ser verbos ou outras descrições de ações

  • Eles devem corresponder à expressão regular ^[a-z][a-zA-Z0-9]*$.
  • Use maiúsculas e minúsculas com a primeira letra em minúscula.
  • Exemplos: toString, hashCode

Variáveis #

Os nomes das variáveis ​​devem estar em maiúsculas e minúsculas com a primeira letra em minúscula

  • Corresponde à expressão regular ^[a-z][a-zA-Z0-9]*$
  • Recomendações adicionais: Variáveis
  • Exemplos: elements, currentIndex

Variáveis ​​de tipo #

Para casos simples onde há poucas variáveis ​​de tipo envolvidas, use uma única letra maiúscula.

  • Corresponde à expressão regular ^[A-Z][0-9]?$
  • Se uma letra for mais descritiva que outra (como K e V para chaves e valores em mapas ou R para um tipo de retorno de função) use isso, caso contrário, use T.
  • Para casos complexos em que variáveis ​​de tipo de letra única se tornam confusas, use nomes mais longos escritos em letras maiúsculas e use sublinhado (_) para separar as palavras.
  • Exemplos: T, V, SRC_VERTEX

Constantes #

Constantes (campos static final cujo conteúdo é imutável, por regras de linguagem ou por convenção) devem ser nomeados com todas as letras maiúsculas e sublinhado (_) para separar as palavras.

  • Corresponde à expressão regular ^[A-Z][A-Z0-9]*(_[A-Z0-9]+)*$
  • Exemplos: BUFFER_SIZE, MAX_LEVEL

Outras orientações sobre nomenclatura #

  • Evite ocultar/sombrear métodos, variáveis ​​e variáveis ​​de tipo em escopos externos.
  • Deixe a verbosidade do nome correlacionar com o tamanho do escopo. (Por exemplo, use nomes descritivos para campos de classes grandes e nomes breves para variáveis ​​locais de curta duração.)
  • Ao nomear membros estáticos públicos, deixe o identificador ser autodescritivo se você acredita que eles serão importados estaticamente.
  • Leitura adicional: Naming Section (no Java Style Guide oficial)

Fonte: Java Style Guidelines da Oracle

Estrutura de classe #

Ordem dos membros da classe #

Os membros da classe devem ser ordenados da seguinte forma:

  1. Campos (em ordem de público, protegido e privado)
  2. Construtores
  3. Métodos de fábrica
  4. Outros Métodos (por ordem de público, protegido e privado)

Não é necessário ordenar campos e métodos principalmente por seus modificadores de acesso ou identificador.

Aqui está um exemplo desta ordem:

class Example {

    private int i;

    Example(int i) {
        this.i = i;
    }

    static Example getExample(int i) {
        return new Example(i);
    }

    @Override
    public String toString() {
        return "An example [" + i + "]";
    }

}

Agrupamento de membros da classe #

  • Os campos relacionados devem ser agrupados.
  • Um tipo aninhado pode ser declarado logo antes de seu primeiro uso; caso contrário, deve ser declarado antes dos campos.
  • Construtores e métodos sobrecarregados devem ser agrupados por funcionalidade e ordenados com aridade crescente. Isso implica que a delegação entre essas construções flui para baixo no código.
  • Construtores devem ser agrupados sem outros membros entre eles.
  • Variantes sobrecarregadas de um método devem ser agrupadas sem outros membros entre elas.

Anotações #

As anotações da declaração devem ser colocadas em uma linha separada da declaração que está sendo anotada.

@SuppressWarnings("unchecked")
public T[] toArray(T[] typeHolder) {
    ...
}

No entanto, poucas ou pequenas anotações anotando um método de linha única podem ser colocadas na mesma linha que o método se melhorar a legibilidade. Por exemplo, pode-se escrever:

@Nullable String getName() { return name; }

Por uma questão de consistência e legibilidade, todas as anotações devem ser colocadas na mesma linha ou cada anotação deve ser colocada em uma linha separada.

// Bad.
@Deprecated @SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Even worse.
@Deprecated @SafeVarargs
@CustomAnnotation public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated
@SafeVarargs
@CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

// Good.
@Deprecated @SafeVarargs @CustomAnnotation
public final Tuple<T> extend(T... elements) {
    ...
}

Importar declarações #

// First java/javax packages
import java.util.ArrayList;
import javax.tools.JavaCompiler;

// Then third party libraries
import com.fasterxml.jackson.annotation.JsonProperty;

// Then project imports
import com.example.my.package.ClassA;
import com.example.my.package.ClassB;

// Then static imports (in the same order as above)
import static java.util.stream.Collectors.toList;

  • As declarações de importação devem ser classificadas…

  • …principalmente por não estático / estático com importações não estáticas primeiro.

  • …secundariamente por origem do pacote de acordo com a seguinte ordem

    • java packages
    • javax packages
    • external packages (e.g. org.xml)
    • internal packages (e.g. com.sun)

  • …terciário por pacote e identificador de classe em ordem lexicográfica

  • As instruções de importação não devem ter quebra de linha, independentemente de exceder o comprimento máximo recomendado de uma linha.

  • Nenhuma importação não utilizada deve estar presente.

Importações curinga #

  • Importações curinga em geral não devem ser usadas.
  • Ao importar um grande número de classes intimamente relacionadas (como implementar um visitante em uma árvore com dezenas de classes de “nós”) distintas, uma importação curinga pode ser usada.
  • Em qualquer caso, não deve ser usada mais de uma importação curinga por arquivo.

Parênteses redundantes #

return flag ? "yes" : "no";

String cmp = (flag1 != flag2) ? "not equal" : "equal";

// Don't do this
return (flag ? "yes" : "no");
  • Parênteses de agrupamento redundantes (ou seja, parênteses que não afetam a avaliação) podem ser usados ​​se melhorarem a legibilidade.
  • Parênteses de agrupamento redundantes normalmente devem ser deixados de fora em expressões mais curtas envolvendo operadores comuns, mas incluídos em expressões mais longas ou expressões envolvendo operadores cuja precedência e associatividade não são claras sem parênteses. Expressões ternárias com condições não triviais pertencem a este último.
  • A expressão inteira após uma palavra-chave return não deve estar entre parênteses.

Suspensórios #

class Example {
    void method(boolean error) {
        if (error) {
            Log.error("Error occurred!");
            System.out.println("Error!");
        } else { // Use braces since the other block uses braces.
            System.out.println("No error");
        }
    }
}

  • As chaves de abertura devem ser colocadas no final da linha atual e não em uma linha própria.

  • Deve haver uma nova linha na frente de uma chave de fechamento, a menos que o bloco esteja vazio (veja Short Forms abaixo)

  • Chaves são recomendadas mesmo quando a linguagem as torna opcionais, como if de linha única e corpos de loop.

  • Se um bloco se estender por mais de uma linha (incluindo comentários), ele deve ter chaves.

  • Se um dos blocos em uma instrução if / else tiver chaves, o outro bloco também deve.

  • Se o bloco vier por último em um bloco anexo, ele deve ter chaves.

  • As palavras-chave else, catch e while nos loops do…while vão na mesma linha que a chave de fechamento do bloco anterior.

Formas curtas #

enum Response { YES, NO, MAYBE }
public boolean isReference() { return true; }

As recomendações acima destinam-se a melhorar a uniformidade (e, assim, aumentar a familiaridade/legibilidade). Em alguns casos, “formas curtas” que se desviam das diretrizes acima são igualmente legíveis e podem ser usadas em seu lugar. Esses casos incluem, por exemplo, declarações enum simples e métodos triviais e expressões lambda.

Modificadores #

class ExampleClass {
    // Access modifiers first (don't do for instance "static public")
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}

interface ExampleInterface {
    // Avoid 'public' and 'abstract' since they are implicit
    void sayHello();
}

  • Modificadores devem ir na seguinte ordem

  • Modificador de acesso (public / private / protected)

  • abstrato

  • estático

  • final

  • transitório

  • volátil

  • padrão

  • sincronizado

  • nativo

  • strictfp

  • Modificadores não devem ser escritos quando estão implícitos. Por exemplo, métodos de interface não devem ser declarados public nem abstract, e enums e interfaces aninhadas não devem ser declaradas estáticas.

  • Parâmetros de método e variáveis ​​locais não devem ser declarados ‘finais’ a menos que melhorem a legibilidade ou documentem uma decisão de projeto real.

  • Os campos devem ser declarados como final a menos que haja uma razão convincente para torná-los mutáveis.

Recuo #

  • O nível de recuo é quatro espaços.

  • Somente caracteres de espaço podem ser usados ​​para recuo. Sem guias.

  • Linhas vazias não devem ser recuadas. (Isso está implícito na regra de nenhum espaço em branco à direita.)

  • As linhas case devem ser indentadas com quatro espaços, e as declarações dentro do case devem ser indentadas com outros quatro espaços.

    switch (var) { case TWO: setChoice(“two”); break; case THREE: setChoice(“three”); break; default: throw new IllegalArgumentException(); }

Consulte https://www.wikiod.com/pt/java/padrao-de-codigo-oficial-da-oracle#Declarações de encapsulamento para obter orientações sobre como recuar linhas de continuação.

Literais #

long l = 5432L;
int i = 0x123 + 0xABC;
byte b = 0b1010;
float f1 = 1 / 5432f;
float f2 = 0.123e4f;
double d1 = 1 / 5432d;  // or 1 / 5432.0
double d2 = 0x1.3p2;
  • Literais long devem usar a letra maiúscula do sufixo L.
  • Literais hexadecimais devem usar letras maiúsculas A-F.
  • Todos os outros prefixos, infixos e sufixos numéricos devem usar letras minúsculas.

Declaração de pacote #

package com.example.my.package;

A declaração do pacote não deve ter quebra de linha, independentemente de exceder o comprimento máximo recomendado de uma linha.

Expressões lambda #

Runnable r = () -> System.out.println("Hello World");

Supplier<String> c = () -> "Hello World";

// Collection::contains is a simple unary method and its behavior is
// clear from the context. A method reference is preferred here.
appendFilter(goodStrings::contains);

// A lambda expression is easier to understand than just tempMap::put in this case
trackTemperature((time, temp) -> tempMap.put(time, temp));
  • Lambdas de expressão são preferidos sobre lambdas de bloco de linha única.
  • As referências de método geralmente devem ser preferidas às expressões lambda.
  • Para referências de métodos de instâncias vinculadas ou métodos com aridade maior que um, uma expressão lambda pode ser mais fácil de entender e, portanto, preferida. Especialmente se o comportamento do método não estiver claro no contexto.
  • Os tipos de parâmetros devem ser omitidos, a menos que melhorem a legibilidade.
  • Se uma expressão lambda se estender por mais de algumas linhas, considere a criação de um método.

Arquivos de origem Java #

  • Todas as linhas devem ser terminadas com um caractere de avanço de linha (LF, valor ASCII 10) e não, por exemplo, CR ou CR+LF.

  • Pode não haver espaço em branco no final de uma linha.

  • O nome de um arquivo fonte deve ser igual ao nome da classe que ele contém seguido da extensão .java, mesmo para arquivos que contenham apenas uma classe privada de pacote. Isso não se aplica a arquivos que não contenham nenhuma declaração de classe, como package-info.java.

Declarações de encapsulamento #

  • O código-fonte e os comentários geralmente não devem exceder 80 caracteres por linha e raramente, ou nunca, exceder 100 caracteres por linha, incluindo recuo.

O limite de caracteres deve ser julgado caso a caso. O que realmente importa é a “densidade” semântica e a legibilidade da linha. Fazer linhas gratuitamente longas as torna difíceis de ler; da mesma forma, fazer “tentativas heróicas” para encaixá-los em 80 colunas também pode torná-los difíceis de ler. A flexibilidade descrita aqui visa permitir que os desenvolvedores evitem esses extremos, e não maximizem o uso do monitor imobiliário.

  • URLs ou comandos de exemplo não devem ser encapsulados.

    // Ok even though it might exceed max line width when indented. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable(targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext(targetContainerType));

    // Wrapping preferable String pretty = Stream.of(args) .map(Argument::prettyPrint) .collectors(joining(", “));

    // Too strict interpretation of max line width. Readability suffers. Error e = isTypeParam ? Errors.InvalidRepeatableAnnotationNotApplicable( targetContainerType, on) : Errors.InvalidRepeatableAnnotationNotApplicableInContext( targetContainerType);

    // Should be wrapped even though it fits within the character limit String pretty = Stream.of(args).map(Argument::prettyPrint).collectors(joining(”, “));

  • O agrupamento em um nível sintático mais alto é preferível ao agrupamento em um nível sintático mais baixo.

  • Deve haver no máximo 1 declaração por linha.

  • Uma linha de continuação deve ser recuada de uma das quatro maneiras a seguir

  • Variante 1: Com 8 espaços extras em relação ao recuo da linha anterior.

  • Variante 2: Com 8 espaços extras relativos à coluna inicial da expressão encapsulada.

  • Variante 3: Alinhado com a expressão do irmão anterior (desde que fique claro que é uma linha de continuação)

  • Variante 4: Alinhado com a chamada de método anterior em uma expressão encadeada.

Declarações do método de encapsulamento #

int someMethod(String aString,
               List<Integer> aList,
               Map<String, String> aMap,
               int anInt,
               long aLong,
               Set<Number> aSet,
               double aDouble) {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt, long aLong,
        double aDouble, long aLong) {
    …
}

int someMethod(String aString,
               List<Map<Integer, StringBuffer>> aListOfMaps,
               Map<String, String> aMap)
        throws IllegalArgumentException {
    …
}

int someMethod(String aString, List<Integer> aList,
        Map<String, String> aMap, int anInt)
                throws IllegalArgumentException {
    …
}
  • Declarações de métodos podem ser formatadas listando os argumentos verticalmente, ou por uma nova linha e +8 espaços extras
  • Se uma cláusula throws precisar ser quebrada, coloque a quebra de linha na frente da cláusula throws e certifique-se de que ela se destaque da lista de argumentos, recuando +8 em relação à declaração da função ou +8 em relação à linha anterior .

Expressões de Embrulho #

  • Se uma linha se aproximar do limite máximo de caracteres, sempre considere dividi-la em várias instruções/expressões em vez de quebrar a linha.

  • Pausa antes dos operadores.

  • Quebra antes do . em chamadas de métodos encadeados.

    popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

    // Don’t! Looks like two arguments popupMsg(“Inbox notification: You have " + newMsgs + " new messages”);

Espaço em branco #

Espaço em branco vertical #

  • Uma única linha em branco deve ser usada para separar…

  • Declaração de pacote

  • Declarações de classe

  • Construtores

  • Métodos

  • Inicializadores estáticos

  • Inicializadores de instância

  • …e pode ser usado para separar grupos lógicos de

  • declarações de importação

  • Campos

  • declarações

  • Várias linhas em branco consecutivas devem ser usadas apenas para separar grupos de membros relacionados e não como espaçamento padrão entre membros.

Espaço em branco horizontal #

  • Um único espaço deve ser usado…

  • Para separar palavras-chave de colchetes e chaves de abertura ou fechamento vizinhos

  • Antes e depois de todos os operadores binários e símbolos semelhantes a operadores, como setas em expressões lambda e os dois pontos em loops for aprimorados (mas não antes dos dois pontos de um rótulo)

  • Após // que inicia um comentário.

  • Depois de vírgulas separando argumentos e ponto e vírgula separando as partes de um loop for.

  • Após o parêntese de fechamento de um elenco.

  • Em declarações de variáveis ​​não é recomendado alinhar tipos e variáveis.

Caracteres especiais #

  • Além de LF, o único caractere de espaço em branco permitido é Espaço (valor ASCII 32). Observe que isso implica que outros caracteres de espaço em branco (em, por exemplo, strings e literais de caracteres) devem ser escritos em forma de escape.

  • \', \", \\, \t, \b, \r, \f e \n devem ser preferidos sobre octal correspondente (por exemplo, \047) ou Unicode (por exemplo, \u0027) caracteres de escape.

  • Caso haja necessidade de ir contra as regras acima para fins de teste, o teste deve gerar a entrada necessária programaticamente.

Declarações de Variáveis #

  • Uma variável por declaração (e no máximo uma declaração por linha)
  • Colchetes de arrays devem estar no tipo (String[] args) e não na variável (String args[]).
  • Declare uma variável local logo antes de ser usada pela primeira vez e inicialize-a o mais próximo possível da declaração.
