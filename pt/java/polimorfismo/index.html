O polimorfismo é um dos principais conceitos de POO (programação orientada a objetos). A palavra polimorfismo foi derivada das palavras gregas “poli” e “morfos”. Poly significa “muitos” e morfos significa “formas” (muitas formas).

Existem duas maneiras de realizar polimorfismo. Sobrecarga de método e Sobrecarga de método.

Interfaces são outra maneira de obter polimorfismo em Java, além da herança baseada em classes. Interfaces definem uma lista de métodos que formam a API do programa. As classes devem implementar uma interface substituindo todos os seus métodos.

Sobreposição de método #

A substituição de método é a capacidade dos subtipos de redefinir (substituir) o comportamento de seus supertipos.

Em Java, isso se traduz em subclasses substituindo os métodos definidos na superclasse. Em Java, todas as variáveis ​​não primitivas são na verdade referências, que são semelhantes a ponteiros para a localização do objeto real na memória. As references têm apenas um tipo, que é o tipo com o qual foram declaradas. No entanto, eles podem apontar para um objeto de seu tipo declarado ou de qualquer um de seus subtipos.

Quando um método é chamado em uma referência, o método correspondente do objeto real que está sendo apontado é invocado.

class SuperType {
    public void sayHello(){
        System.out.println("Hello from SuperType");
    }

    public void sayBye(){
        System.out.println("Bye from SuperType");
    }
}

class SubType extends SuperType {
    // override the superclass method
    public void sayHello(){
        System.out.println("Hello from SubType");
    }
}

class Test {
    public static void main(String... args){
        SuperType superType = new SuperType();
        superType.sayHello(); // -> Hello from SuperType

        // make the reference point to an object of the subclass
        superType = new SubType();
        // behaviour is governed by the object, not by the reference
        superType.sayHello(); // -> Hello from SubType

        // non-overridden method is simply inherited
        superType.sayBye(); // -> Bye from SuperType
    }
}

Regras a serem observadas

Para substituir um método na subclasse, o método de substituição (ou seja, o da subclasse) ** DEVE TER**:

  • mesmo nome
  • mesmo tipo de retorno no caso de primitivos (uma subclasse é permitida para classes, isso também é conhecido como tipos de retorno covariantes).
  • mesmo tipo e ordem de parâmetros
  • pode lançar apenas as exceções que são declaradas na cláusula throws do método da superclasse ou exceções que são subclasses das exceções declaradas. Ele também pode optar por NÃO lançar nenhuma exceção. The names of the parameter types do not matter. For example, void methodX(int i) is same as void methodX(int k)
  • Não podemos substituir métodos finais ou estáticos. A única coisa que podemos fazer muda apenas o corpo do método.

Sobrecarga do método #

A sobrecarga de método, também conhecida como sobrecarga de função, é a capacidade de uma classe ter vários métodos com o mesmo nome, desde que eles sejam diferentes em número ou tipo de argumentos.

O compilador verifica a assinatura do método quanto à sobrecarga do método.

A assinatura do método consiste em três coisas -

  1. Nome do método
  2. Número de parâmetros
  3. Tipos de parâmetros

Se esses três forem os mesmos para quaisquer dois métodos em uma classe, o compilador lançará erro de método duplicado.

Esse tipo de polimorfismo é chamado de polimorfismo estático ou tempo de compilação porque o método apropriado a ser chamado é decidido pelo compilador durante o tempo de compilação com base na lista de argumentos.

class Polymorph {

    public int add(int a, int b){
        return a + b;
    }
    
    public int add(int a, int b, int c){
        return a + b + c;
    }

    public float add(float a, float b){
        return a + b;
    }

    public static void main(String... args){
        Polymorph poly = new Polymorph();
        int a = 1, b = 2, c = 3;
        float d = 1.5, e = 2.5;

        System.out.println(poly.add(a, b));
        System.out.println(poly.add(a, b, c));
        System.out.println(poly.add(d, e));
    }

}

Isso resultará em:

2
6
4.000000

Os métodos sobrecarregados podem ser estáticos ou não estáticos. Isso também não afeta a sobrecarga do método.

public class Polymorph {

    private static void methodOverloaded()
    {
        //No argument, private static method
    }
 
    private int methodOverloaded(int i)
    {
        //One argument private non-static method
        return i;
    }
 
    static int methodOverloaded(double d)
    {
        //static Method
        return 0;
    }
 
    public void methodOverloaded(int i, double d)
    {
        //Public non-static Method
    }
}

Além disso, se você alterar o tipo de retorno do método, não poderemos obtê-lo como sobrecarga de método.

public class Polymorph {  

void methodOverloaded(){
    //No argument and No return type
}

int methodOverloaded(){
    //No argument and int return type 
    return 0;
}

Polimorfismo e diferentes tipos de sobreposição #

De java tutorial

A definição do dicionário de polimorfismo refere-se a um princípio em biologia no qual um organismo ou espécie pode ter muitas formas ou estágios diferentes. Este princípio também pode ser aplicado à programação orientada a objetos e linguagens como a linguagem Java. As subclasses de uma classe podem definir seus próprios comportamentos exclusivos e ainda compartilhar algumas das mesmas funcionalidades da classe pai.

Dê uma olhada neste exemplo para entender os diferentes tipos de substituição.

  1. A classe base não fornece implementação e a subclasse precisa substituir o método completo - (resumo)
  2. A classe base fornece implementação padrão e a subclasse pode alterar o comportamento
  3. A subclasse adiciona extensão à implementação da classe base chamando super.methodName() como primeira instrução
  4. A classe base define a estrutura do algoritmo (método Modelo) e a subclasse substituirá uma parte do algoritmo

fragmento de código:

import java.util.HashMap;

abstract class Game implements Runnable{

    protected boolean runGame = true;
    protected Player player1 = null;
    protected Player player2 = null;
    protected Player currentPlayer = null;
    
    public Game(){
        player1 = new Player("Player 1");
        player2 = new Player("Player 2");
        currentPlayer = player1;
        initializeGame();
    }

    /* Type 1: Let subclass define own implementation. Base class defines abstract method to force
        sub-classes to define implementation    
    */
    
    protected abstract void initializeGame();
    
    /* Type 2: Sub-class can change the behaviour. If not, base class behaviour is applicable */
    protected void logTimeBetweenMoves(Player player){
        System.out.println("Base class: Move Duration: player.PlayerActTime - player.MoveShownTime");
    }
    
    /* Type 3: Base class provides implementation. Sub-class can enhance base class implementation by calling
        super.methodName() in first line of the child class method and specific implementation later */
    protected void logGameStatistics(){
        System.out.println("Base class: logGameStatistics:");
    }
    /* Type 4: Template method: Structure of base class can't be changed but sub-class can some part of behaviour */
    protected void runGame() throws Exception{
        System.out.println("Base class: Defining the flow for Game:");    
        while (runGame) {
            /*
            1. Set current player
            2. Get Player Move
            */
            validatePlayerMove(currentPlayer);    
            logTimeBetweenMoves(currentPlayer);
            Thread.sleep(500);
            setNextPlayer();
        }
        logGameStatistics();
    }
    /* sub-part of the template method, which define child class behaviour */
    protected abstract void validatePlayerMove(Player p);
    
    protected void setRunGame(boolean status){
        this.runGame = status;
    }
    public void setCurrentPlayer(Player p){
        this.currentPlayer = p;
    }
    public void setNextPlayer(){
        if (currentPlayer == player1) {
            currentPlayer = player2;
        }else{
            currentPlayer = player1;
        }
    }
    public void run(){
        try{
            runGame();
        }catch(Exception err){
            err.printStackTrace();
        }
    }
}

class Player{
    String name;
    Player(String name){
        this.name = name;
    }
    public String getName(){
        return name;
    }
}

Adicionando comportamento adicionando classes sem tocar no código existente #

import java.util.ArrayList;
import java.util.List;

import static java.lang.System.out;

public class PolymorphismDemo {

    public static void main(String[] args) {
        List<FlyingMachine> machines = new ArrayList<FlyingMachine>();
        machines.add(new FlyingMachine());
        machines.add(new Jet());
        machines.add(new Helicopter());
        machines.add(new Jet());

        new MakeThingsFly().letTheMachinesFly(machines);
    }
}

class MakeThingsFly {
    public void letTheMachinesFly(List<FlyingMachine> flyingMachines) {
        for (FlyingMachine flyingMachine : flyingMachines) {
            flyingMachine.fly();
        }
    }
}

class FlyingMachine {
    public void fly() {
        out.println("No implementation");
    }
}

class Jet extends FlyingMachine {
    @Override
    public void fly() {
        out.println("Start, taxi, fly");
    }

    public void bombardment() {
        out.println("Fire missile");
    }
}

class Helicopter extends FlyingMachine {
    @Override
    public void fly() {
        out.println("Start vertically, hover, fly");
    }
}

Explicação

a) A classe MakeThingsFly pode trabalhar com tudo que for do tipo FlyingMachine.

b) O método letTheMachinesFly também funciona sem nenhuma alteração (!) quando você adiciona uma nova classe, por exemplo PropellerPlane:

public void letTheMachinesFly(List<FlyingMachine> flyingMachines) {
        for (FlyingMachine flyingMachine : flyingMachines) {
            flyingMachine.fly();
        }
    }
}

Funções virtuais #

Métodos Virtuais são métodos em Java que não são estáticos e sem a palavra-chave Final na frente. Todos os métodos por padrão são virtuais em Java. Os métodos virtuais desempenham papéis importantes no polimorfismo porque as classes filhas em Java podem substituir os métodos de suas classes pai se a função que está sendo substituída não for estática e tiver a mesma assinatura de método.

Existem, no entanto, alguns métodos que não são virtuais. Por exemplo, se o método for declarado privado ou com a palavra-chave final, o método não será Virtual.

Considere o seguinte exemplo modificado de herança com métodos virtuais desta postagem do StackOverflow http://stackoverflow.com/questions/460446/how-do-virtual-functions-work-in-c-sharp-and-java):

public class A{
    public void hello(){
        System.out.println("Hello");
    }
    
    public void boo(){
        System.out.println("Say boo");

    }
}

public class B extends A{
     public void hello(){
        System.out.println("No");
     }
    
    public void boo(){
        System.out.println("Say haha");

    }
}

Se invocarmos a classe B e chamarmos hello() e boo(), obteremos “No” e “Say haha” como a saída resultante porque B substitui os mesmos métodos de A. Mesmo que o exemplo acima seja quase exatamente o mesmo que substituição de método, é importante entender que os métodos da classe A são todos, por padrão, Virtual.

Além disso, podemos implementar métodos virtuais usando a palavra-chave abstract. Métodos declarados com a palavra-chave “abstract” não possuem uma definição de método, o que significa que o corpo do método ainda não foi implementado. Considere o exemplo acima novamente, exceto que o método boo() é declarado abstrato:

public class A{
   public void hello(){
        System.out.println("Hello");
    }
    
    abstract void boo();
}

public class B extends A{
     public void hello(){
        System.out.println("No");
     }
    
    public void boo(){
        System.out.println("Say haha");

    }
}

Se invocarmos boo() de B, a saída ainda será “Say haha”, já que B herda o método abstrato boo() e faz boo() produzir “Say haha”.
