A estrutura de coleções em java.util fornece várias classes genéricas para conjuntos de dados com funcionalidades que não podem ser fornecidas por arrays regulares.

A estrutura de coleções contém interfaces para Collection<O>, com as subinterfaces principais List<O> e Set<O>, e a coleção de mapeamento Map<K,V>. As coleções são a interface raiz e estão sendo implementadas por muitas outras estruturas de coleção.

Coleções são objetos que podem armazenar coleções de outros objetos dentro deles. Você pode especificar o tipo de dados armazenados em uma coleção usando Generics.

As coleções geralmente usam os namespaces java.util ou java.util.concurrent.

Java 1.4.2 e abaixo não suportam genéricos. Como tal, você não pode especificar os parâmetros de tipo que uma coleção contém. Além de não ter segurança de tipo, você também deve usar casts para obter o tipo correto de volta de uma coleção.

Além de Collection<E>, existem vários tipos principais de coleções, algumas das quais possuem subtipos.

  • List<E> é uma coleção ordenada de objetos. É semelhante a um array, mas não define um limite de tamanho. As implementações geralmente crescem em tamanho internamente para acomodar novos elementos.
  • Set<E> é uma coleção de objetos que não permite duplicatas.
  • SortedSet<E> é um Set<E> que também especifica a ordenação dos elementos.
  • Map<K,V> é uma coleção de pares chave/valor.
  • SortedMap<K,V> é um Map<K,V> que também especifica a ordenação dos elementos.

Java 5 adiciona um novo tipo de coleção:

  • Queue<E> é uma coleção de elementos destinados a serem processados ​​em uma ordem específica. A implementação especifica se é FIFO ou LIFO. Isso torna obsoleta a classe Stack.

Java 6 adiciona alguns novos subtipos de coleções.

  • NavigableSet<E> é um Set<E> com métodos de navegação especiais embutidos.
  • NavigableMap<K,V> é um Map<K,V> com métodos de navegação especiais embutidos.
  • Deque<E> é uma Queue<E> que pode ser lida de qualquer extremidade.

Observe que os itens acima são todos interfaces. Para usá-los, você deve encontrar as classes de implementação apropriadas, como ArrayList, HashSet, HashMap ou PriorityQueue.

Cada tipo de coleção tem várias implementações que possuem diferentes métricas de desempenho e casos de uso.

Observe que o [Princípio de substituição de Liskov](https://www.wikiod.com/pt/java/heranca#O Princípio da Substituição de Liskov) se aplica aos subtipos de coleção. Ou seja, um SortedSet<E> pode ser passado para uma função esperando um Set<E>. Também é útil ler sobre Parâmetros limitados na seção Genéricos para obter mais informações sobre como usar coleções com herança de classe.

Se você quiser criar suas próprias coleções, pode ser mais fácil herdar uma das classes abstratas (como AbstractList) em vez de implementar a interface.

Antes do 1.2, você tinha que usar as seguintes classes/interfaces:

  • Vetor em vez de ArrayList
  • Dicionário em vez de Mapa. Observe que o Dicionário também é uma classe abstrata em vez de uma interface.
  • Hashtable em vez de HashMap

Essas classes são obsoletas e não devem ser usadas em código moderno.

Removendo itens de uma lista dentro de um loop #

É complicado remover itens de uma lista enquanto estiver dentro de um loop, isso se deve ao fato de que o índice e o comprimento da lista são alterados.

Dada a lista a seguir, aqui estão alguns exemplos que darão um resultado inesperado e alguns que darão o resultado correto.

List<String> fruits = new ArrayList<String>();
fruits.add("Apple");
fruits.add("Banana");
fruits.add("Strawberry");

INCORRETA #

Removendo na iteração da instrução for Ignora “Banana”: #

A amostra de código imprimirá apenas Apple e Strawberry. Banana é ignorado porque se move para o índice 0 uma vez que Apple é deletado, mas ao mesmo tempo i é incrementado para 1.

for (int i = 0; i < fruits.size(); i++) {
    System.out.println (fruits.get(i)); 
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
    }     
}

Removendo na instrução for aprimorada Throws Exception: #

Por causa de iterar sobre a coleção e modificá-la ao mesmo tempo.

Lança: java.util.ConcurrentModificationException

for (String fruit : fruits) { 
    System.out.println(fruit);
    if ("Apple".equals(fruit)) {
        fruits.remove(fruit);
    }
}

CORRETO #

Removendo o loop while usando um Iterator #

Iterator<String> fruitIterator = fruits.iterator();
while(fruitIterator.hasNext()) {     
    String fruit = fruitIterator.next();     
    System.out.println(fruit);
    if ("Apple".equals(fruit)) {
        fruitIterator.remove();
    } 
}

A interface Iterator tem um método remove() embutido apenas para este caso. No entanto, esse método é marcado como “opcional” na documentação e pode lançar um UnsupportedOperationException.

Lança: UnsupportedOperationException - if the remove operation is not supported by this iterator

Portanto, é aconselhável verificar a documentação para certificar-se de que esta operação é suportada (na prática, a menos que a coleção seja imutável obtida através de uma biblioteca de terceiros ou o uso de um dos métodos Collections.unmodifiable...() método, a operação é quase sempre suportada).

Ao usar um Iterator, uma ConcurrentModificationException é lançada quando o modCount da List é alterado de quando o Iterator foi criado. Isso pode ter acontecido no mesmo thread ou em um aplicativo multithread compartilhando a mesma lista.

Um modCount é uma variável int que conta o número de vezes que esta lista foi modificada estruturalmente. Uma mudança estrutural significa essencialmente uma operação add() ou remove() sendo invocada no objeto Collection (alterações feitas por Iterator não são contadas). Quando o Iterator é criado, ele armazena este modCount e em cada iteração da List verifica se o modCount atual é o mesmo que e quando o Iterator foi criado. Se houver uma alteração no valor modCount, ele lançará um ConcurrentModificationException.

Portanto, para a lista declarada acima, uma operação como abaixo não lançará nenhuma exceção:

Iterator<String> fruitIterator = fruits.iterator();
fruits.set(0, "Watermelon");
while(fruitIterator.hasNext()){
    System.out.println(fruitIterator.next());
}

Mas adicionar um novo elemento à List após inicializar um Iterator lançará uma ConcurrentModificationException:

Iterator<String> fruitIterator = fruits.iterator();
fruits.add("Watermelon");
while(fruitIterator.hasNext()){
    System.out.println(fruitIterator.next());    //ConcurrentModificationException here
}

Iterando para trás #

for (int i = (fruits.size() - 1); i >=0; i--) {
    System.out.println (fruits.get(i));
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
    }
}

Isso não pula nada. A desvantagem dessa abordagem é que a saída é inversa. No entanto, na maioria dos casos em que você remove itens que não importam. Você nunca deve fazer isso com LinkedList.

Iterando para frente, ajustando o índice do loop #

for (int i = 0; i < fruits.size(); i++) {
    System.out.println (fruits.get(i)); 
    if ("Apple".equals(fruits.get(i))) {
         fruits.remove(i);
         i--;
    }     
}

Isso não pula nada. Quando o elemento ith é removido da List, o elemento originalmente posicionado no índice i+1 torna-se o novo elemento ith. Portanto, o loop pode decrementar i para que a próxima iteração processe o próximo elemento, sem pular.

Usando uma lista “deve ser removida” #

ArrayList shouldBeRemoved = new ArrayList();
for (String str : currentArrayList) {
    if (condition) {
        shouldBeRemoved.add(str);
    }
}
currentArrayList.removeAll(shouldBeRemoved);

Esta solução permite ao revelador verificar se os elementos corretos são removidos de forma mais limpa.

No Java 8 são possíveis as seguintes alternativas. Eles são mais limpos e diretos se a remoção não precisar acontecer em um loop.

Filtrando um fluxo #

Uma Lista pode ser transmitida e filtrada. Um filtro adequado pode ser usado para remover todos os elementos indesejados.

List<String> filteredList = 
    fruits.stream().filter(p -> !"Apple".equals(p)).collect(Collectors.toList());

Observe que, ao contrário de todos os outros exemplos aqui, este exemplo produz uma nova instância List e mantém a List original inalterada.

Usando removeIf #

Economiza a sobrecarga de construir um fluxo se tudo o que for necessário for remover um conjunto de itens.

fruits.removeIf(p -> "Apple".equals(p));

Coleções vazias imutáveis #

Às vezes é apropriado usar uma coleção vazia imutável. A classe Collections fornece métodos para obter essas coleções de maneira eficiente:

List<String> anEmptyList = Collections.emptyList();
Map<Integer, Date> anEmptyMap = Collections.emptyMap();
Set<Number> anEmptySet = Collections.emptySet();

Esses métodos são genéricos e converterão automaticamente a coleção retornada para o tipo ao qual ela foi atribuída. Isto é, uma invocação de e. emptyList() pode ser atribuído a qualquer tipo de List e da mesma forma para emptySet() e emptyMap().

As coleções retornadas por esses métodos são imutáveis, pois lançarão UnsupportedOperationException se você tentar chamar métodos que alterariam seu conteúdo (add, put, etc.). Essas coleções são úteis principalmente como substitutos para resultados de métodos vazios ou outros valores padrão, em vez de usar null ou criar objetos com new.

Coleção não modificável #

Às vezes, não é uma boa prática expor uma coleção interna, pois pode levar a uma vulnerabilidade de código malicioso devido à sua característica mutável. Para fornecer coleções “somente leitura”, o java fornece suas versões não modificáveis.

Uma coleção não modificável é frequentemente uma cópia de uma coleção modificável que garante que a coleção em si não possa ser alterada. As tentativas de modificá-lo resultarão em uma exceção UnsupportedOperationException.

É importante notar que os objetos que estão presentes dentro da coleção ainda podem ser alterados.

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

public class MyPojoClass {
    private List<Integer> intList = new ArrayList<>();

    public void addValueToIntList(Integer value){
        intList.add(value);
    }
    
    public List<Integer> getIntList() {
        return Collections.unmodifiableList(intList);
    }
}

A seguinte tentativa de modificar uma coleção não modificável lançará uma exceção:

import java.util.List;

public class App {

    public static void main(String[] args) {
        MyPojoClass pojo = new MyPojoClass();
        pojo.addValueToIntList(42);
        
        List<Integer> list = pojo.getIntList();
        list.add(69);
    }
}

resultado:

Exception in thread "main" java.lang.UnsupportedOperationException
    at java.util.Collections$UnmodifiableCollection.add(Collections.java:1055)
    at App.main(App.java:12)

Armadilha: exceções de modificação simultânea #

Essa exceção ocorre quando uma coleção é modificada durante a iteração sobre ela usando métodos diferentes daqueles fornecidos pelo objeto iterador. Por exemplo, temos uma lista de chapéus e queremos remover todos aqueles que têm abas de orelha:

List<IHat> hats = new ArrayList<>();
hats.add(new Ushanka()); // that one has ear flaps
hats.add(new Fedora());
hats.add(new Sombrero());
for (IHat hat : hats) {
    if (hat.hasEarFlaps()) {
        hats.remove(hat);
    }
}

Se executarmos esse código, ConcurrentModificationException será gerado, pois o código modifica a coleção enquanto a itera. A mesma exceção pode ocorrer se um dos vários threads trabalhando com a mesma lista estiver tentando modificar a coleção enquanto outros iteram sobre ela. A modificação simultânea de coleções em várias threads é uma coisa natural, mas deve ser tratada com ferramentas usuais da caixa de ferramentas de programação simultânea, como bloqueios de sincronização, coleções especiais adotadas para modificação simultânea, modificação da coleção clonada da inicial etc.

Subcoleções #

List subList(int fromIndex, int toIndex) #

Aqui fromIndex é inclusivo e toIndex é exclusivo.

List list = new ArrayList(); 
List list1 = list.subList(fromIndex,toIndex);
  1. Se a lista não existir no intervalo especificado, ela lançará IndexOutofBoundException.
  2. Quaisquer alterações feitas na lista1 afetarão as mesmas alterações na lista. Isso é chamado de coleções de backup.
  3. Se fromnIndex for maior que toIndex (fromIndex > toIndex), ele lançará IllegalArgumentException.

Exemplo:

List<String> list = new ArrayList<String>();
List<String> list = new ArrayList<String>();
list.add("Hello1"); 
list.add("Hello2"); 
System.out.println("Before Sublist "+list); 
List<String> list2 = list.subList(0, 1);
list2.add("Hello3"); 
System.out.println("After sublist changes "+list);

Saída:
Antes da sublista [Hello1, Hello2]
Após alterações na sublista [Hello1, Hello3, Hello2]

Definir subconjunto (deIndex,toIndex) #

Aqui fromIndex é inclusivo e toIndex é exclusivo.

Set set = new TreeSet(); 
Set set1 = set.subSet(fromIndex,toIndex);

O conjunto retornado lançará um IllegalArgumentException em uma tentativa de inserir um elemento fora de seu intervalo.

Mapear submapa(fromKey,toKey) #

fromKey é inclusivo e toKey é exclusivo

Map map = new TreeMap();
Map map1 = map.get(fromKey,toKey);

Se fromKey for maior que toKey ou se este mapa em si tiver um intervalo restrito e fromKey ou toKey estiver fora dos limites do intervalo, ele lançará IllegalArgumentException.

Todas as coleções suportam coleções apoiadas significa que as alterações feitas na subcoleção terão a mesma alteração na coleção principal.

Removendo itens correspondentes das listas usando o Iterator. #

Acima notei um exemplo para remover itens de uma Lista dentro de um Loop e pensei em outro exemplo que pode ser útil desta vez usando a interface Iterator.
Esta é uma demonstração de um truque que pode ser útil ao lidar com itens duplicados em listas das quais você deseja se livrar.

Observação: isso é apenas uma adição ao exemplo Removendo itens de uma lista dentro de um loop:

Então vamos definir nossas listas como de costume

    String[] names = {"James","Smith","Sonny","Huckle","Berry","Finn","Allan"};
    List<String> nameList = new ArrayList<>();

    //Create a List from an Array
    nameList.addAll(Arrays.asList(names));
    
    String[] removeNames = {"Sonny","Huckle","Berry"};
    List<String> removeNameList = new ArrayList<>();

    //Create a List from an Array
    removeNameList.addAll(Arrays.asList(removeNames));

O método a seguir recebe dois objetos Collection e executa a mágica de remover os elementos em nossa removeNameList que combinam com os elementos em nameList.

private static void removeNames(Collection<String> collection1, Collection<String> collection2) {
      //get Iterator.
    Iterator<String> iterator = collection1.iterator();
    
    //Loop while collection has items
    while(iterator.hasNext()){
        if (collection2.contains(iterator.next()))
            iterator.remove(); //remove the current Name or Item
    }
}

Chamando o método e passando o nameList e o removeNameList da seguinte forma removeNames(nameList,removeNameList);
Produzirá a seguinte saída:

Lista de matrizes após remover nomes: James Smith Finn Allan

Um uso simples e elegante para coleções que pode ser útil para remover elementos repetidos em listas.

Coleções e valores primitivos #

Coleções em Java só funcionam para objetos. Ou seja não há Map<int, int> em Java. Em vez disso, os valores primitivos precisam ser encaixotados em objetos, como em Map<Integer, Integer>. O auto-boxing Java permitirá o uso transparente dessas coleções:

Map<Integer, Integer> map = new HashMap<>();
map.put(1, 17); // Automatic boxing of int to Integer objects
int a = map.get(1); // Automatic unboxing.

Infelizmente, a sobrecarga disso é substancial. Um HashMap<Integer, Integer> exigirá cerca de 72 bytes por entrada (por exemplo, em JVM de 64 bits com ponteiros compactados e assumindo inteiros maiores que 256 e assumindo 50% de carga do mapa). Como os dados reais são de apenas 8 bytes, isso gera uma sobrecarga enorme. Além disso, requer dois níveis de indireção (Map -> Entry -> Value) é desnecessariamente lento.

Existem várias bibliotecas com coleções otimizadas para tipos de dados primitivos (que requerem apenas ~16 bytes por entrada com 50% de carga, ou seja, 4x menos memória e um nível de indireção a menos), que podem gerar benefícios substanciais de desempenho ao usar grandes coleções de primitivos valores em Java.

Criando sua própria estrutura Iterable para uso com Iterator ou for-each loop. #

Para garantir que nossa coleção possa ser iterada usando iterator ou for-each loop, temos que cuidar das seguintes etapas:

  1. O material sobre o qual queremos iterar precisa ser Iterável e expor iterador().
  2. Projete um java.util.Iterator substituindo hasNext(), next() e remove().

Eu adicionei uma implementação de lista vinculada genérica simples abaixo que usa as entidades acima para tornar a lista vinculada iterável.

package org.algorithms.linkedlist;
 
import java.util.Iterator;
import java.util.NoSuchElementException;
 
 
public class LinkedList<T> implements Iterable<T> {
 
    Node<T> head, current;
 
    private static class Node<T> {
        T data;
        Node<T> next;
 
        Node(T data) {
            this.data = data;
        }
    }
 
    public LinkedList(T data) {
        head = new Node<>(data);
    }
 
    public Iterator<T> iterator() {
        return new LinkedListIterator();
    }
 
    private class LinkedListIterator implements Iterator<T> {
 
        Node<T> node = head;
 
        @Override
        public boolean hasNext() {
            return node != null;
        }
 
        @Override
        public T next() {
            if (!hasNext())
                throw new NoSuchElementException();
            Node<T> prevNode = node;
            node = node.next;
            return prevNode.data;
        }
 
        @Override
        public void remove() {
            throw new UnsupportedOperationException("Removal logic not implemented.");
        }
    }
 
    public void add(T data) {
        Node current = head;
        while (current.next != null)
            current = current.next;
        current.next = new Node<>(data);
    }
 
}
 
class App {
    public static void main(String[] args) {
 
        LinkedList<Integer> list = new LinkedList<>(1);
        list.add(2);
        list.add(4);
        list.add(3);
 
        //Test #1
        System.out.println("using Iterator:");
        Iterator<Integer> itr = list.iterator();
        while (itr.hasNext()) {
            Integer i = itr.next();
            System.out.print(i + " ");
        }
 
        //Test #2
        System.out.println("\n\nusing for-each:");
        for (Integer data : list) {
            System.out.print(data + " ");
        }
    }
}

Resultado

using Iterator:
1 2 4 3
using for-each:
1 2 4 3

Isso será executado em Java 7+. Você pode fazê-lo rodar no Java 5 e Java 6 também substituindo:

LinkedList<Integer> list = new LinkedList<>(1);

com

LinkedList<Integer> list = new LinkedList<Integer>(1);

ou qualquer outra versão, incorporando as alterações compatíveis.
