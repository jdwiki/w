Objetos imutáveis ​​têm estado fixo (sem setters), então todos os estados devem ser conhecidos no momento da criação do objeto.

Embora não seja tecnicamente necessário, é uma boa prática tornar todos os campos final. Isso tornará a classe imutável thread-safe (cf. Java Concurrency in Practice, 3.4.1).

Os exemplos mostram vários padrões que podem ajudar a conseguir isso.

Criando uma versão imutável de um tipo usando cópia defensiva. #

Alguns tipos e classes básicos em Java são fundamentalmente mutáveis. Por exemplo, todos os tipos de array são mutáveis, assim como classes como java.util.Data. Isso pode ser estranho em situações em que um tipo imutável é obrigatório.

Uma maneira de lidar com isso é criar um wrapper imutável para o tipo mutável. Aqui está um wrapper simples para uma matriz de inteiros

public class ImmutableIntArray {
    private final int[] array;
    
    public ImmutableIntArray(int[] array) {
        this.array = array.clone();
    }

    public int[] getValue() {
        return this.clone();
    }
}

Esta classe funciona usando cópia defensiva para isolar o estado mutável (o int[]) de qualquer código que possa alterá-lo:

  • O construtor usa clone() para criar uma cópia distinta do array de parâmetros. Se o chamador do construtor subsequentemente alterasse o array de parâmetros, isso não afetaria o estado do ImmutableIntArray.

  • O método getValue() também usa clone() para criar o array que é retornado. Se o chamador alterar o array de resultados, isso não afetará o estado do ImmutableIntArray.

Também poderíamos adicionar métodos a ImmutableIntArray para realizar operações somente leitura no array encapsulado; por exemplo. obter seu comprimento, obter o valor em um índice específico e assim por diante.

Observe que um tipo de wrapper imutável implementado dessa maneira não é compatível com o tipo original. Você não pode simplesmente substituir o primeiro pelo segundo.

A receita para uma classe imutável #

Um objeto imutável é um objeto cujo estado não pode ser alterado. Uma classe imutável é uma classe cujas instâncias são imutáveis ​​por design e implementação. A classe Java que é mais comumente apresentada como exemplo de imutabilidade é java.lang.String.

Segue um exemplo estereotipado:

public final class Person {
    private final String name;
    private final String ssn;     // (SSN == social security number)

    public Person(String name, String ssn) {
        this.name = name;
        this.ssn = ssn;
    }

    public String getName() {
        return name;
    }
   
    public String getSSN() {
        return ssn;
    }
}

Uma variação disso é declarar o construtor como private e fornecer um método de fábrica public static.

A receita padrão para uma classe imutável é a seguinte:

  • Todas as propriedades devem ser definidas no(s) construtor(es) ou método(s) de fábrica.
  • Não deve haver setters.
  • Se for necessário incluir setters por motivos de compatibilidade de interface, eles não devem fazer nada ou lançar uma exceção.
  • Todas as propriedades devem ser declaradas como private e final.
  • Para todas as propriedades que são referências a tipos mutáveis:
  • a propriedade deve ser inicializada com uma cópia profunda do valor passado pelo construtor e
  • o getter da propriedade deve retornar uma cópia profunda do valor da propriedade.
  • A classe deve ser declarada como final para evitar que alguém crie uma subclasse mutável de uma classe imutável.

Algumas outras coisas a serem observadas:

  • A imutabilidade não impede que o objeto seja anulável; por exemplo. null pode ser atribuído a uma variável String.
  • Se as propriedades de uma classe imutável forem declaradas como final, as instâncias são inerentemente seguras para threads. Isso torna as classes imutáveis ​​um bom bloco de construção para implementar aplicativos multithread.

Falhas típicas de design que impedem que uma classe seja imutável #

Usando alguns setters, sem definir todas as propriedades necessárias no(s) construtor(es)

public final class Person { // example of a bad immutability
    private final String name;
    private final String surname;
    public Person(String name) {
        this.name = name;
      }
    public String getName() { return name;}
    public String getSurname() { return surname;}
    public void setSurname(String surname) { this.surname = surname); }
}

É fácil mostrar que a classe Person não é imutável:

Person person = new Person("Joe");
person.setSurname("Average"); // NOT OK, change surname field after creation

Para corrigi-lo, simplesmente delete setSurname() e refatore o construtor da seguinte forma:

public Person(String name, String surname) {
    this.name = name;
    this.surname = surname;
  }

Não marcando variáveis ​​de instância como privadas e finais

Dê uma olhada na aula a seguir:

public final class Person {
    public String name;
    public Person(String name) {
        this.name = name;
     }
    public String getName() {
        return name;
    }
    
}

O snippet a seguir mostra que a classe acima não é imutável:

Person person = new Person("Average Joe");
person.name = "Magic Mike"; // not OK, new name for person after creation

Para corrigi-lo, basta marcar a propriedade name como private e final.

Expondo um objeto mutável da classe em um getter

Dê uma olhada na aula a seguir:

import java.util.List;
import java.util.ArrayList;
public final class Names {
    private final List<String> names;
    public Names(List<String> names) {
        this.names = new ArrayList<String>(names);
    }
    public List<String> getNames() {
        return names;
    }
    public int size() {
        return names.size();
    }
}

A classe Names parece imutável à primeira vista, mas não é como o código a seguir mostra:

List<String> namesList = new ArrayList<String>();
namesList.add("Average Joe");
Names names = new Names(namesList);
System.out.println(names.size()); // 1, only containing "Average Joe"
namesList = names.getNames();
namesList.add("Magic Mike");
System.out.println(names.size()); // 2, NOT OK, now names also contains "Magic Mike"

Isso aconteceu porque uma mudança na lista de referência retornada por getNames() pode modificar a lista real de Names.

Para corrigir isso, simplesmente evite retornar referências que referenciam objetos mutáveis ​​da classe também fazendo cópias defensivas, como segue:

public List<String> getNames() {
   return new ArrayList<String>(this.names); // copies elements
}

ou projetando getters de forma que apenas outros objetos imutáveis e primitivos sejam retornados, como segue:

public String getName(int index) {
    return names.get(index);
}
public int size() {
    return names.size();
}

Injetando construtor com objeto(s) que podem ser modificados fora da classe imutável

Esta é uma variação da falha anterior. Dê uma olhada na aula a seguir:

import java.util.List;
public final class NewNames {
    private final List<String> names;
    public Names(List<String> names) {
        this.names = names;
    }
    public String getName(int index) {
        return names.get(index);
    }
    public int size() {
        return names.size();
    }
}

Como a classe Names antes, também a classe NewNames parece imutável à primeira vista, mas não é, na verdade o trecho a seguir prova o contrário:

List<String> namesList = new ArrayList<String>();
namesList.add("Average Joe");
NewNames names = new NewNames(namesList);
System.out.println(names.size()); // 1, only containing "Average Joe"
namesList.add("Magic Mike");
System.out.println(names.size()); // 2, NOT OK, now names also contains "Magic Mike"

Para corrigir isso, como na falha anterior, basta fazer cópias defensivas do objeto sem atribuí-lo diretamente à classe imutável, ou seja, o construtor pode ser alterado da seguinte forma:

    public Names(List<String> names) {
        this.names = new ArrayList<String>(names);
    }

Permitindo que os métodos da classe sejam substituídos

Dê uma olhada na aula a seguir:

public class Person {
    private final String name;
    public Person(String name) {
        this.name = name;
      }
    public String getName() { return name;}
}

A classe Person parece imutável à primeira vista, mas suponha que uma nova subclasse de Person seja definida:

public class MutablePerson extends Person {
    private String newName;
    public MutablePerson(String name) {
        super(name);            
    }
    @Override
    public String getName() {
        return newName;
    }
    public void setName(String name) {
        newName = name;
    }
}

agora a (im)mutabilidade de Person pode ser explorada através de polimorfismo usando a nova subclasse:

Person person = new MutablePerson("Average Joe");
System.out.println(person.getName()); prints Average Joe
person.setName("Magic Mike"); // NOT OK, person has now a new name!
System.out.println(person.getName()); // prints Magic Mike

Para corrigir isso, ou marque a classe como final para que não possa ser estendida ou declare todos os seus construtores como privados.
