As expressões lambda fornecem uma maneira clara e concisa de implementar uma interface de método único usando uma expressão. Eles permitem que você reduza a quantidade de código que você precisa criar e manter. Embora semelhantes a classes anônimas, elas não possuem informações de tipo por si mesmas. A inferência de tipos precisa acontecer.

As referências de método implementam interfaces funcionais usando métodos existentes em vez de expressões. Eles também pertencem à família lambda.

Sintaxe #

  • () -> { expressão de retorno; } // Zero-arity com o corpo da função para retornar um valor.
  • () -> expressão // Abreviação para a declaração acima; não há ponto e vírgula para expressões.
  • () -> { function-body } // Efeito colateral na expressão lambda para realizar operações.
  • parameterName -> expression // Expressão lambda de uma só aridade. Em expressões lambda com apenas um argumento, o parêntese pode ser removido.
  • (Type parameterName, Type secondParameterName, …) -> expression // lambda avaliando uma expressão com parâmetros listados à esquerda
  • (parameterName, secondParameterName, …) -> expressão // Abreviação que remove os tipos de parâmetro para os nomes de parâmetro. Só pode ser usado em contextos que podem ser inferidos pelo compilador em que o tamanho da lista de parâmetros especificado corresponde a um (e apenas um) tamanho das interfaces funcionais esperadas.

Introdução aos lambdas Java #

Interfaces Funcionais #

Lambdas só pode operar em uma interface funcional, que é uma interface com apenas um método abstrato. Interfaces funcionais podem ter qualquer número de métodos default ou static. (Por esse motivo, às vezes são chamadas de Interfaces de Método Abstrato Único ou Interfaces SAM).

interface Foo1 {
    void bar();
}

interface Foo2 {
    int bar(boolean baz);
}

interface Foo3 {
    String bar(Object baz, int mink);
}

interface Foo4 {
    default String bar() { // default so not counted
        return "baz";
    }
    void quux();
}

Ao declarar uma interface funcional, a anotação @FunctionalInterface pode ser adicionada. Isso não tem efeito especial, mas um erro do compilador será gerado se esta anotação for aplicada a uma interface que não é funcional, agindo assim como um lembrete de que a interface não deve ser alterada.

@FunctionalInterface
interface Foo5 {
    void bar();
}

@FunctionalInterface
interface BlankFoo1 extends Foo3 { // inherits abstract method from Foo3
}

@FunctionalInterface
interface Foo6 {
    void bar();
    boolean equals(Object obj); // overrides one of Object's method so not counted
}

Por outro lado, esta não é uma interface funcional, pois possui mais de um método abstrato:

interface BadFoo {
    void bar();
    void quux(); // <-- Second method prevents lambda: which one should 
                 // be considered as lambda?
}

Esta também não é uma interface funcional, pois não possui métodos:

interface BlankFoo2 { }

Tome nota do seguinte. Suponha que você tenha

interface Pai { public int parentMethod(); }

e

interface Filho estende Pai { public int ChildMethod(); }

Então Child não pode ser uma interface funcional, pois tem dois métodos especificados.

O Java 8 também fornece uma série de interfaces funcionais com templates genéricos no pacote java.util.function. Por exemplo, a interface interna Predicate<T> envolve um único método que insere um valor do tipo T e retorna um boolean.

Expressões lambda #

A estrutura básica de uma expressão Lambda é:

fi irá então manter uma instância singleton de uma classe, similar a uma classe anônima, que implementa FunctionalInterface e onde a definição de um método é { System.out.println("Hello"); }. Em outras palavras, o acima é principalmente equivalente a:

FunctionalInterface fi = new FunctionalInterface() {
    @Override
    public void theOneMethod() {
        System.out.println("Hello");
    }
};

O lambda é apenas “equivalente principalmente” à classe anônima porque em um lambda, o significado de expressões como this, super ou toString() referenciam a classe dentro da qual a atribuição ocorre, não o objeto recém-criado .

Você não pode especificar o nome do método ao usar um lambda, mas não deve precisar, porque uma interface funcional deve ter apenas um método abstrato, então o Java substitui esse.

Nos casos em que o tipo do lambda não é certo (por exemplo, métodos sobrecarregados), você pode adicionar uma conversão ao lambda para informar ao compilador qual deve ser seu tipo, assim:

Object fooHolder = (Foo1) () -> System.out.println("Hello");
System.out.println(fooHolder instanceof Foo1); // returns true

Se o método único da interface funcional receber parâmetros, os nomes formais locais destes devem aparecer entre colchetes do lambda. Não há necessidade de declarar o tipo de parâmetro ou retorno, pois estes são retirados da interface (embora não seja um erro declarar os tipos de parâmetro, se você quiser). Assim, estes dois exemplos são equivalentes:

Foo2 longFoo = new Foo2() {
    @Override
    public int bar(boolean baz) {
        return baz ? 1 : 0;
    }
};
Foo2 shortFoo = (x) -> { return x ? 1 : 0; };

Os parênteses ao redor do argumento podem ser omitidos se a função tiver apenas um argumento:

Foo2 np = x -> { return x ? 1 : 0; }; // okay
Foo3 np2 = x, y -> x.toString() + y // not okay

Devoluções Implícitas #

Se o código colocado dentro de um lambda for uma expression Java em vez de uma statement, ele será tratado como um método que retorna o valor da expressão. Assim, os dois seguintes são equivalentes:

IntUnaryOperator addOneShort = (x) -> (x + 1);
IntUnaryOperator addOneLong = (x) -> { return (x + 1); };

Acessando Variáveis ​​Locais (fechamento de valor) #

Como lambdas são uma abreviação sintática para classes anônimas, eles seguem as mesmas regras para acessar variáveis ​​locais no escopo delimitador; as variáveis ​​devem ser tratadas como final e não modificadas dentro do lambda.

IntUnaryOperator makeAdder(int amount) {
    return (x) -> (x + amount); // Legal even though amount will go out of scope
                                // because amount is not modified
}

IntUnaryOperator makeAccumulator(int value) {
    return (x) -> { value += x; return value; }; // Will not compile
}

Se for necessário envolver uma variável em mudança dessa maneira, um objeto regular que mantenha uma cópia da variável deve ser usado. Leia mais em https://www.wikiod.com/pt/java/expressoes-lambda#Java Closures com expressões lambda.

Aceitando Lambdas #

Como um lambda é uma implementação de uma interface, nada de especial precisa ser feito para que um método aceite um lambda: qualquer função que receba uma interface funcional também pode aceitar um lambda.

public void passMeALambda(Foo1 f) {
    f.bar();
}
passMeALambda(() -> System.out.println("Lambda called"));

O tipo de uma expressão lambda #

Uma expressão lambda, por si só, não possui um tipo específico. Embora seja verdade que os tipos e o número de parâmetros, juntamente com o tipo de um valor de retorno, possam transmitir algumas informações de tipo, essas informações apenas restringirão a quais tipos podem ser atribuídas. O lambda recebe um tipo quando é atribuído a um tipo de interface funcional de uma das seguintes maneiras:

  • Atribuição direta a um tipo funcional, por ex. myPredicate = s -> s.isEmpty()
  • Passando-o como um parâmetro que possui um tipo funcional, ex. stream.filter(s -> s.isEmpty())
  • Devolvê-lo de uma função que retorna um tipo funcional, por exemplo return s -> s.isEmpty()
  • Casting-lo para um tipo funcional, por exemplo (Predicate<>) s -> s.isEmpty()

Até que tal atribuição a um tipo funcional seja feita, o lambda não tem um tipo definido. Para ilustrar, considere a expressão lambda o -> o.isEmpty(). A mesma expressão lambda pode ser atribuída a muitos tipos funcionais diferentes:

Predicate<String> javaStringPred = o -> o.isEmpty();
Function<String, Boolean> javaFunc = o -> o.isEmpty();
Predicate<List> javaListPred = o -> o.isEmpty();
Consumer<String> javaStringConsumer = o -> o.isEmpty(); // return value is ignored!
com.google.common.base.Predicate<String> guavaPredicate = o -> o.isEmpty();

Agora que eles estão atribuídos, os exemplos mostrados são de tipos completamente diferentes, embora as expressões lambda tenham a mesma aparência, e não podem ser atribuídos um ao outro.

Usando expressões lambda para classificar uma coleção #

Classificando listas #

Antes do Java 8, era necessário implementar a interface java.util.Comparator com uma classe anônima (ou nomeada) ao classificar uma lista1:

List<Person> people = ...
Collections.sort(
    people,
    new Comparator<Person>() {
        public int compare(Person p1, Person p2){
            return p1.getFirstName().compareTo(p2.getFirstName());
        }
    }
);

A partir do Java 8, a classe anônima pode ser substituída por uma expressão lambda. Observe que os tipos para os parâmetros p1 e p2 podem ser omitidos, pois o compilador os inferirá automaticamente:

Collections.sort(
    people, 
    (p1, p2) -> p1.getFirstName().compareTo(p2.getFirstName())
);

O exemplo pode ser simplificado usando Comparator.comparing e referências de método expressas usando o símbolo :: (dois pontos).

Collections.sort(
    people,
    Comparator.comparing(Person::getFirstName)
);

Uma importação estática nos permite expressar isso de forma mais concisa, mas é discutível se isso melhora a legibilidade geral:

import static java.util.Collections.sort;
import static java.util.Comparator.comparing;
//...
sort(people, comparing(Person::getFirstName));

Comparadores construídos dessa maneira também podem ser encadeados. Por exemplo, depois de comparar pessoas pelo primeiro nome, se houver pessoas com o mesmo nome, o método thenComparing também compara pelo sobrenome:

sort(people, comparing(Person::getFirstName).thenComparing(Person::getLastName));

1 - Note que Collections.sort(…) só funciona em coleções que são subtipos de List. As APIs Set e Collection não implicam em nenhuma ordenação dos elementos.

Classificando mapas #

Você pode classificar as entradas de um HashMap por valor de maneira semelhante. (Observe que um LinkedHashMap deve ser usado como destino. As chaves em um HashMap comum não são ordenadas.)

Map<String, Integer> map = new HashMap();  // ... or any other Map class
// populate the map
map = map.entrySet()
    .stream()
    .sorted(Map.Entry.<String, Integer>comparingByValue())
    .collect(Collectors.toMap(k -> k.getKey(), v -> v.getValue(),
                              (k, v) -> k, LinkedHashMap::new));

Referências de métodos #

As referências de método permitem que métodos estáticos ou de instância predefinidos que aderem a uma interface funcional compatível sejam passados ​​como argumentos em vez de uma expressão lambda anônima.

Suponha que temos um modelo:

class Person {
    private final String name;
    private final String surname;

    public Person(String name, String surname){
        this.name = name;
        this.surname = surname;
    }

    public String getName(){ return name; }
    public String getSurname(){ return surname; }
}

List<Person> people = getSomePeople();

Referência do método de instância (para uma instância arbitrária) #

people.stream().map(Person::getName)

O lambda equivalente:

people.stream().map(person -> person.getName())

Neste exemplo, uma referência de método para o método de instância getName() do tipo Person está sendo passada. Como é conhecido por ser do tipo de coleção, o método na instância (conhecido posteriormente) será invocado.

Referência do método de instância (para uma instância específica) #

people.forEach(System.out::println);

Como System.out é uma instância de PrintStream, uma referência de método para esta instância específica está sendo passada como um argumento.

O lambda equivalente:

people.forEach(person -> System.out.println(person));

Referência de método estático #

Também para transformar streams podemos aplicar referências a métodos estáticos:

List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5, 6);
numbers.stream().map(String::valueOf)

Este exemplo passa uma referência ao método estático valueOf() no tipo String. Portanto, o objeto de instância na coleção é passado como um argumento para valueOf().

O lambda equivalente:

 numbers.stream().map(num -> String.valueOf(num))

Referência a um construtor #

List<String> strings = Arrays.asList("1", "2", "3");
strings.stream().map(Integer::new)

Leia Collect Elements of a Stream into a Collection para ver como coletar elementos para a coleção.

O construtor de argumento String único do tipo Integer está sendo usado aqui, para construir um inteiro dado a string fornecida como argumento. Nesse caso, desde que a string represente um número, o fluxo será mapeado para Integers. O lambda equivalente:

strings.stream().map(s -> new Integer(s));

Implementando várias interfaces #

Às vezes você pode querer ter uma expressão lambda implementando mais de uma interface. Isso é útil principalmente com interfaces de marcador (como java.io.Serializable), pois elas não adicionar métodos abstratos.

Por exemplo, você deseja criar um TreeSet com um Comparator personalizado e, em seguida, serializá-lo e enviá-lo pela rede. A abordagem trivial:

TreeSet<Long> ts = new TreeSet<>((x, y) -> Long.compare(y, x));

não funciona porque o lambda para o comparador não implementa Serializable. Você pode corrigir isso usando tipos de interseção e especificando explicitamente que esse lambda precisa ser serializável:

TreeSet<Long> ts = new TreeSet<>(
    (Comparator<Long> & Serializable) (x, y) -> Long.compare(y, x));

Se você estiver usando frequentemente tipos de interseção (por exemplo, se estiver usando uma estrutura como Apache Spark onde quase tudo precisa ser serializável), você pode criar interfaces e use-as em seu código:

public interface SerializableComparator extends Comparator<Long>, Serializable {}

public class CustomTreeSet {
  public CustomTreeSet(SerializableComparator comparator) {}
}

Dessa forma, você garante que o comparador passado será serializável.

Lambda - Exemplo de ouvinte #

Ouvinte de classe anônimo

Antes do Java 8, era muito comum que uma classe anônima fosse usada para manipular o evento click de um JButton, conforme mostrado no código a seguir. Este exemplo mostra como implementar um ouvinte anônimo dentro do escopo de btn.addActionListener.

JButton btn = new JButton("My Button");
btn.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("Button was pressed");
    }
});

Ouvinte Lambda

Como a interface ActionListener define apenas um método actionPerformed(), é uma interface funcional, o que significa que há um lugar para usar expressões Lambda para substituir o código clichê. O exemplo acima pode ser reescrito usando expressões Lambda da seguinte forma:

JButton btn = new JButton("My Button");
btn.addActionListener(e -> {
    System.out.println("Button was pressed");
});

Java Closures com expressões lambda. #

Um fechamento lambda é criado quando uma expressão lambda referencia as variáveis ​​de um escopo delimitador (global ou local). As regras para fazer isso são as mesmas para métodos inline e classes anônimas.

Variáveis ​​locais de um escopo delimitador que são usadas em um lambda devem ser final. Com o Java 8 (a versão mais antiga que suporta lambdas), eles não precisam ser declarados como final no contexto externo, mas devem ser tratados dessa forma. Por exemplo:

int n = 0; // With Java 8 there is no need to explicit final
Runnable r = () -> { // Using lambda
    int i = n;
    // do something
};

Isso é legal desde que o valor da variável n não seja alterado. Se você tentar alterar a variável, dentro ou fora do lambda, receberá o seguinte erro de compilação:

“variáveis ​​locais referenciadas de uma expressão lambda devem ser final ou efetivamente final”.

Por exemplo:

int n = 0;
Runnable r = () -> { // Using lambda
    int i = n;
    // do something
};
n++; // Will generate an error.

Se for necessário usar uma variável variável dentro de um lambda, a abordagem normal é declarar uma cópia final da variável e usar a cópia. Por exemplo

int n = 0;
final int k = n; // With Java 8 there is no need to explicit final
Runnable r = () -> { // Using lambda
    int i = k;
    // do something
};
n++;      // Now will not generate an error
r.run();  // Will run with i = 0 because k was 0 when the lambda was created

Naturalmente, o corpo do lambda não vê as mudanças na variável original.

Observe que Java não suporta fechamentos verdadeiros. Um lambda Java não pode ser criado de uma maneira que permita ver as alterações no ambiente em que foi instanciado. Se você deseja implementar um encerramento que observa ou faz alterações em seu ambiente, você deve simulá-lo usando uma classe regular. Por exemplo:

// Does not compile ...
public IntUnaryOperator createAccumulator() {
    int value = 0;
    IntUnaryOperator accumulate = (x) -> { value += x; return value; };
    return accumulate;
}

O exemplo acima não será compilado por motivos discutidos anteriormente. Podemos contornar o erro de compilação da seguinte forma:

// Compiles, but is incorrect ...
public class AccumulatorGenerator {
    private int value = 0;

    public IntUnaryOperator createAccumulator() {
        IntUnaryOperator accumulate = (x) -> { value += x; return value; };
        return accumulate;
    }
}

O problema é que isso quebra o contrato de design para a interface IntUnaryOperator que afirma que as instâncias devem ser funcionais e sem estado. Se esse encerramento for passado para funções internas que aceitam objetos funcionais, é provável que cause travamentos ou comportamento errôneo. Closures que encapsulam o estado mutável devem ser implementados como classes regulares. Por exemplo.

// Correct ...
public class Accumulator {
   private int value = 0;

   public int accumulate(int x) {
      value += x;
      return value;
   }
}

Usando a expressão lambda com sua própria interface funcional #

Lambdas destinam-se a fornecer código de implementação em linha para interfaces de método único e a capacidade de passá-los como temos feito com variáveis ​​normais. Nós os chamamos de Interface Funcional.

Por exemplo, escrever um Runnable em uma classe anônima e iniciar um Thread se parece com:

//Old way
new Thread(
        new Runnable(){
            public void run(){
                System.out.println("run logic...");
            }
        }
).start();

//lambdas, from Java 8
new Thread(
        ()-> System.out.println("run logic...")
).start();

Agora, de acordo com o acima, digamos que você tenha uma interface personalizada:

interface TwoArgInterface {
    int operate(int a, int b);
}

Como você usa lambda para implementar essa interface em seu código? Igual ao exemplo executável mostrado acima. Veja o programa do driver abaixo:

public class CustomLambda {
    public static void main(String[] args) {

        TwoArgInterface plusOperation = (a, b) -> a + b;
        TwoArgInterface divideOperation = (a,b)->{
            if (b==0) throw new IllegalArgumentException("Divisor can not be 0");
            return a/b;
        };

        System.out.println("Plus operation of 3 and 5 is: " + plusOperation.operate(3, 5));
        System.out.println("Divide operation 50 by 25 is: " + divideOperation.operate(50, 25));

    }
}

Do estilo tradicional ao estilo Lambda #

Caminho tradicional

interface MathOperation{
    boolean unaryOperation(int num);
}

public class LambdaTry {
    public static void main(String[] args) {
        MathOperation isEven = new MathOperation() {
            @Override
            public boolean unaryOperation(int num) {
                return num%2 == 0;
            }
        };
        
        System.out.println(isEven.unaryOperation(25));
        System.out.println(isEven.unaryOperation(20));
    }
}

Estilo Lambda

  1. Remova o nome da classe e o corpo da interface funcional.

    public class LambdaTry { public static void main(String[] args) { MathOperation isEven = (int num) -> { return num%2 == 0; };

         System.out.println(isEven.unaryOperation(25));
     System.out.println(isEven.unaryOperation(20));
 }

    }

  2. Declaração de tipo opcional

    MathOperation isEven = (num) -> { return num%2 == 0; };

  3. Parênteses opcionais em torno do parâmetro, se for um parâmetro único

    MathOperation isEven = num -> { return num%2 == 0; };

  4. Chaves opcionais, se houver apenas uma linha no corpo da função

  5. Palavra-chave de retorno opcional, se houver apenas uma linha no corpo da função

    MathOperation isEven = num -> num%2 == 0;

Lambdas e utilização de memória #

Como os lambdas Java são encerramentos, eles podem “capturar” os valores das variáveis ​​no escopo léxico envolvente. Embora nem todos os lambdas capturem nada – lambdas simples como s -> s.length() não capturam nada e são chamados de stateless – a captura de lambdas requer um objeto temporário para armazenar as variáveis ​​capturadas. Neste trecho de código, o lambda () -> j é um lambda de captura e pode fazer com que um objeto seja alocado quando avaliado:

public static void main(String[] args) throws Exception {
    for (int i = 0; i < 1000000000; i++) {
        int j = i;
        doSomethingWithLambda(() -> j);
    }
}

Embora possa não ser imediatamente óbvio, já que a palavra-chave new não aparece em nenhum lugar no snippet, esse código pode criar 1.000.000.000 de objetos separados para representar as instâncias da expressão lambda () -> j. No entanto, também deve ser observado que versões futuras do Java1 podem otimizar isso para que em tempo de execução as instâncias lambda sejam reutilizadas ou representadas de alguma outra forma.

1 - Por exemplo, o Java 9 introduz uma fase opcional de “link” para a sequência de compilação do Java que fornecerá a oportunidade de fazer otimizações globais como esta.

Lambdas e padrão de execução ao redor #

Existem vários bons exemplos de uso de lambdas como FunctionalInterface em cenários simples. Um caso de uso bastante comum que pode ser melhorado por lambdas é o chamado padrão Execute-Around. Nesse padrão, você tem um conjunto de código de configuração/desmontagem padrão que é necessário para vários cenários em torno do código específico do caso de uso. Alguns exemplos comuns disso são arquivos io, banco de dados io, blocos try/catch.

interface DataProcessor {
    void process( Connection connection ) throws SQLException;;
}

public void doProcessing( DataProcessor processor ) throws SQLException{
    try (Connection connection = DBUtil.getDatabaseConnection();) {
        processor.process(connection);
        connection.commit();
    } 
}

Então, para chamar esse método com um lambda, pode ser assim:

public static void updateMyDAO(MyVO vo) throws DatabaseException {
    doProcessing((Connection conn) -> MyDAO.update(conn, ObjectMapper.map(vo)));
}

Isso não se limita a operações de E/S. Ele pode ser aplicado a qualquer cenário em que tarefas semelhantes de configuração/desmontagem sejam aplicáveis ​​com pequenas variações. O principal benefício deste padrão é a reutilização de código e aplicação de DRY (Don’t Repeat Yourself).

return retorna apenas do lambda, não do método externo #

O método return só retorna do lambda, não do método externo.

Cuidado que isso é diferente de Scala e Kotlin!

void threeTimes(IntConsumer r) {
  for (int i = 0; i < 3; i++) {
    r.accept(i);
  }
}

void demo() {
  threeTimes(i -> {
    System.out.println(i);
    return; // Return from lambda to threeTimes only!
  });
}

Isso pode levar a um comportamento inesperado ao tentar escrever construções de linguagem próprias, pois em construções internas como loops for return se comporta de maneira diferente:

void demo2() {
  for (int i = 0; i < 3; i++) {
    System.out.println(i);
    return; // Return from 'demo2' entirely
  }
}

Em Scala e Kotlin, demo e demo2 imprimiriam apenas 0. Mas isso não é mais consistente. A abordagem Java é consistente com a refatoração e o uso de classes - o return no código na parte superior, e o código abaixo se comporta da mesma forma:

void demo3() {
  threeTimes(new MyIntConsumer());
}

class MyIntConsumer implements IntConsumer {
  public void accept(int i) {
    System.out.println(i);
    return;
  }
}

Portanto, o return Java é mais consistente com métodos de classe e refatoração, mas menos com os builtins for e while, estes permanecem especiais.

Por causa disso, os dois seguintes são equivalentes em Java:

IntStream.range(1, 4)
    .map(x -> x * x)
    .forEach(System.out::println);
IntStream.range(1, 4)
    .map(x -> { return x * x; })
    .forEach(System.out::println);

Além disso, o uso de try-with-resources é seguro em Java:

class Resource implements AutoCloseable {
  public void close() { System.out.println("close()"); }
}

void executeAround(Consumer<Resource> f) {
  try (Resource r = new Resource()) {
    System.out.print("before ");
    f.accept(r);
    System.out.print("after ");
  }
}

void demo4() {
  executeAround(r -> {
    System.out.print("accept() ");
    return; // Does not return from demo4, but frees the resource.
  });
}

imprimirá antes de aceitar() depois de fechar(). Na semântica Scala e Kotlin, o try-with-resources não seria fechado, mas imprimiria apenas before accept().

Usando expressões e predicados lambda para obter determinados valores de uma lista #

A partir do Java 8, você pode usar expressões e predicados lambda.

Exemplo: Use expressões lambda e um predicado para obter um determinado valor de uma lista. Neste exemplo, cada pessoa será impressa com o fato de ter 18 anos ou mais ou não.

Classe de pessoa:

public class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public int getAge() { return age; }
    public String getName() { return name; }
}

A interface embutida Predicate dos pacotes java.util.function.Predicate é uma interface funcional com um método boolean test(T t).

Exemplo de uso:

import java.util.ArrayList;
import java.util.List;
import java.util.function.Predicate;

public class LambdaExample {
    public static void main(String[] args) {
        List<Person> personList = new ArrayList<Person>();
        personList.add(new Person("Jeroen", 20));
        personList.add(new Person("Jack", 5));
        personList.add(new Person("Lisa", 19));

        print(personList, p -> p.getAge() >= 18);
    }

    private static void print(List<Person> personList, Predicate<Person> checker) {
        for (Person person : personList) {
            if (checker.test(person)) {
                System.out.print(person + " matches your expression.");
            } else {
                System.out.println(person  + " doesn't match your expression.");
            }
        }
    }
}

O método print(personList, p -> p.getAge() >= 18); recebe uma expressão lambda (porque o Predicate é usado como parâmetro) onde você pode definir a expressão necessária. O método de teste do verificador verifica se esta expressão está correta ou não: checker.test(pessoa).

Você pode facilmente mudar isso para outra coisa, por exemplo, para print(personList, p -> p.getName().startsWith("J"));. Isso verificará se o nome da pessoa começa com um “J”.
