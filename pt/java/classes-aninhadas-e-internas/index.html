Usando Java, os desenvolvedores têm a capacidade de definir uma classe dentro de outra classe. Essa classe é chamada de Classe Aninhada. As classes aninhadas são chamadas de classes internas se forem declaradas como não estáticas, caso contrário, são simplesmente chamadas de classes aninhadas estáticas. Esta página é para documentar e fornecer detalhes com exemplos sobre como usar Java Nested e Inner Classes.

Sintaxe #

  • public class OuterClass { public class InnerClass { } } // As classes internas também podem ser privadas
  • public class OuterClass { public static class StaticNestedClass { } } // Classes aninhadas estáticas também podem ser privadas
  • public void method() { private class LocalClass { } } // Classes locais são sempre privadas
  • SomeClass anônimoClassInstance = new SomeClass() { }; // Classes internas anônimas não podem ser nomeadas, portanto o acesso é discutível. Se ‘SomeClass()’ for abstrato, o corpo deve implementar todos os métodos abstratos.
  • SomeInterface anônimoClassInstance = new SomeInterface() { }; // O corpo deve implementar todos os métodos de interface.

Terminologia e classificação #

A Java Language Specification (JLS) classifica os diferentes tipos de classe Java da seguinte forma:

Uma classe de nível superior é uma classe que não é uma classe aninhada.

Uma classe aninhada é qualquer classe cuja declaração ocorre dentro do corpo de outra classe ou interface.

Uma classe interna é uma classe aninhada que não é declarada estática explícita ou implicitamente.

Uma classe interna pode ser uma classe de membro não estático, uma classe local ou uma classe anônima. Uma classe membro de uma interface é implicitamente estática, portanto, nunca é considerada uma classe interna.

Na prática, os programadores referem-se a uma classe de nível superior que contém uma classe interna como a “classe externa”. Além disso, há uma tendência de usar “classe aninhada” para se referir apenas a (explícita ou implicitamente) classes aninhadas estáticas.

Observe que há uma relação próxima entre classes internas anônimas e lambdas, mas lambdas são classes.

Diferenças semânticas #

  • As classes de nível superior são o “caso base”. Eles são visíveis para outras partes de um programa sujeito a regras normais de visibilidade baseadas na semântica do modificador de acesso. Se não abstratos, eles podem ser instanciados por qualquer código em que os construtores relevantes sejam visíveis com base nos modificadores de acesso.

  • As classes aninhadas estáticas seguem as mesmas regras de acesso e instanciação das classes de nível superior, com duas exceções:

  • Uma classe aninhada pode ser declarada como private, o que a torna inacessível fora de sua classe de nível superior.

  • Uma classe aninhada tem acesso aos membros private da classe de nível superior envolvente e todas as suas classes testadas.

Isso torna as classes aninhadas estáticas úteis quando você precisa representar vários “tipos de entidade” dentro de um limite de abstração restrito; por exemplo. quando as classes aninhadas são usadas para ocultar “detalhes de implementação”.

  • As classes internas adicionam a capacidade de acessar variáveis ​​não estáticas declaradas em escopos delimitadores:
  • Uma classe de membro não estática pode se referir a variáveis ​​de instância.
  • Uma classe local (declarada dentro de um método) também pode se referir às variáveis ​​locais do método, desde que sejam finais. (Para Java 8 e posterior, eles podem ser efetivamente finais.)
  • Uma classe interna anônima pode ser declarada dentro de uma classe ou de um método e pode acessar variáveis ​​de acordo com as mesmas regras.

O fato de que uma instância de classe interna pode se referir a variáveis ​​em uma instância de classe delimitadora tem implicações para a instanciação. Especificamente, uma instância delimitadora deve ser fornecida, implícita ou explicitamente, quando uma instância de uma classe interna é criada.

Uma pilha simples usando uma classe aninhada #

public class IntStack {

    private IntStackNode head;

    // IntStackNode is the inner class of the class IntStack
    // Each instance of this inner class functions as one link in the
    // Overall stack that it helps to represent
    private static class IntStackNode {

        private int val;
        private IntStackNode next;

        private IntStackNode(int v, IntStackNode n) {
            val = v;
            next = n;
        }
    }

    public IntStack push(int v) {
        head = new IntStackNode(v, head);
        return this;
    }

    public int pop() {
        int x = head.val;
        head = head.next;
        return x;
    }
}

E o uso do mesmo, que (notavelmente) não reconhece a existência da classe aninhada.

public class Main {
    public static void main(String[] args) {
 
        IntStack s = new IntStack();
        s.push(4).push(3).push(2).push(1).push(0);

        //prints: 0, 1, 2, 3, 4, 
        for(int i = 0; i < 5; i++) {
            System.out.print(s.pop() + ", ");
        }            
    }
}

Classes aninhadas estáticas vs não estáticas #

Ao criar uma classe aninhada, você enfrenta a opção de ter essa classe aninhada estática:

public class OuterClass1 {

    private static class StaticNestedClass {

    }

}

Ou não estático:

public class OuterClass2 {

    private class NestedClass {

    }

}

Em seu núcleo, as classes aninhadas estáticas não têm uma instância circundante da classe externa, enquanto as classes aninhadas não estáticas têm. Isso afeta onde/quando é permitido instanciar uma classe aninhada e quais instâncias dessas classes aninhadas podem acessar. Adicionando ao exemplo acima:

public class OuterClass1 {

    private int aField;
    public void aMethod(){}

    private static class StaticNestedClass {
        private int innerField;

        private StaticNestedClass() {
             innerField = aField; //Illegal, can't access aField from static context 
             aMethod();           //Illegal, can't call aMethod from static context 
        }

        private StaticNestedClass(OuterClass1 instance) {
             innerField = instance.aField; //Legal
        }

    }

    public static void aStaticMethod() {
        StaticNestedClass s = new StaticNestedClass(); //Legal, able to construct in static context
        //Do stuff involving s...
    }

}

public class OuterClass2 {

    private int aField;

    public void aMethod() {}

    private class NestedClass {
        private int innerField;

        private NestedClass() {
             innerField = aField; //Legal   
             aMethod(); //Legal
        }
    }

    public void aNonStaticMethod() {
        NestedClass s = new NestedClass(); //Legal
    }

    public static void aStaticMethod() {
        NestedClass s = new NestedClass(); //Illegal. Can't construct without surrounding OuterClass2 instance.
                                         //As this is a static context, there is no surrounding OuterClass2 instance
    }
}

Assim, sua decisão de estático versus não estático depende principalmente se você precisa ou não poder acessar diretamente os campos e métodos da classe externa, embora também tenha consequências para quando e onde você pode construir a classe aninhada.

Como regra geral, torne suas classes aninhadas estáticas, a menos que você precise acessar campos e métodos da classe externa. Semelhante a tornar seus campos privados, a menos que você precise deles públicos, isso diminui a visibilidade disponível para a classe aninhada (não permitindo acesso a uma instância externa), reduzindo a probabilidade de erro.

Modificadores de acesso para classes internas #

Uma explicação completa dos Modificadores de Acesso em Java pode ser encontrada aqui. Mas como eles interagem com as classes internas?

public, como de costume, dá acesso irrestrito a qualquer escopo capaz de acessar o tipo.

public class OuterClass {

    public class InnerClass {

        public int x = 5;

    }

    public InnerClass createInner() {
        return new InnerClass();
    }
}

public class SomeOtherClass {

    public static void main(String[] args) {
        int x = new OuterClass().createInner().x; //Direct field access is legal
    }
}

tanto protected quanto o modificador padrão (de nada) também se comportam como esperado, da mesma forma que para classes não aninhadas.

private, curiosamente, não se restringe à classe a que pertence. Em vez disso, restringe-se à unidade de compilação - o arquivo .java. Isso significa que as classes Outer têm acesso total aos campos e métodos da classe Inner, mesmo que estejam marcados como ‘privados’.

public class OuterClass {

    public class InnerClass {

        private int x;
        private void anInnerMethod() {}
    }

    public InnerClass aMethod() {
        InnerClass a = new InnerClass();
        a.x = 5; //Legal
        a.anInnerMethod(); //Legal
        return a;
    }
}

A própria Inner Class pode ter uma visibilidade diferente de public. Ao marcá-lo como privado ou outro modificador de acesso restrito, outras classes (externas) não poderão importar e atribuir o tipo. No entanto, eles ainda podem obter referências a objetos desse tipo.

public class OuterClass {

    private class InnerClass{}

    public InnerClass makeInnerClass() {
        return new InnerClass();
    }
}

public class AnotherClass {

    public static void main(String[] args) {
        OuterClass o = new OuterClass();
     
        InnerClass x = o.makeInnerClass(); //Illegal, can't find type
        OuterClass.InnerClass x = o.makeInnerClass(); //Illegal, InnerClass has visibility private
        Object x = o.makeInnerClass(); //Legal
    }
}

Classes internas anônimas #

Uma classe interna anônima é uma forma de classe interna que é declarada e instanciada com uma única instrução. Como consequência, não há nome para a classe que possa ser usada em outras partes do programa; ou seja, é anônimo.

As classes anônimas são normalmente usadas em situações em que você precisa criar uma classe leve para ser passada como parâmetro. Isso geralmente é feito com uma interface. Por exemplo:

public static Comparator<String> CASE_INSENSITIVE =
        new Comparator<String>() {
            @Override
            public int compare(String string1, String string2) {
                return string1.toUpperCase().compareTo(string2.toUpperCase());
            }
        };

Esta classe anônima define um objeto Comparator<String> (CASE_INSENSITIVE) que compara duas strings ignorando diferenças entre maiúsculas e minúsculas.

Outras interfaces que são frequentemente implementadas e instanciadas usando classes anônimas são Runnable e Callable. Por exemplo:

// An anonymous Runnable class is used to provide an instance that the Thread
// will run when started.
Thread t = new Thread(new Runnable() {
        @Override 
        public void run() {
              System.out.println("Hello world");
        }
    });
t.start();  // Prints "Hello world"

Classes internas anônimas também podem ser baseadas em classes. Neste caso, a classe anônima implicitamente estende a classe existente. Se a classe que está sendo estendida for abstrata, a classe anônima deverá implementar todos os métodos abstratos. Também pode substituir métodos não abstratos.

Construtores #

Uma classe anônima não pode ter um construtor explícito. Em vez disso, é definido um construtor implícito que usa super(...) para passar quaisquer parâmetros para um construtor na classe que está sendo estendida. Por exemplo:

SomeClass anon = new SomeClass(1, "happiness") {
            @Override
            public int someMethod(int arg) {
                // do something
            }
        };

O construtor implícito para nossa subclasse anônima de SomeClass chamará um construtor de SomeClass que corresponde à assinatura de chamada SomeClass(int, String). Se nenhum construtor estiver disponível, você receberá um erro de compilação. Quaisquer exceções lançadas pelo construtor correspondente também são lançadas pelo construtor implícito.

Naturalmente, isso não funciona ao estender uma interface. Quando você cria uma classe anônima a partir de uma interface, a superclasse de classes é java.lang.Object que possui apenas um construtor no-args.

Cria instância de classe interna não estática de fora #

Uma classe interna que é visível para qualquer classe externa também pode ser criada a partir dessa classe.

A classe interna depende da classe externa e requer uma referência a uma instância dela. Para criar uma instância da classe interna, o operador new só precisa ser chamado em uma instância da classe externa.

class OuterClass {

    class InnerClass {
    }
}

class OutsideClass {

    OuterClass outer = new OuterClass();
    
    OuterClass.InnerClass createInner() {
        return outer.new InnerClass();
    }
}

Observe o uso como outer.new.

Classes internas locais do método #

Uma classe escrita dentro de um método chamado classe interna local do método. Nesse caso, o escopo da classe interna é restrito dentro do método.

Uma classe interna local de método pode ser instanciada apenas dentro do método em que a classe interna está definida.

O exemplo de uso da classe interna local do método:

public class OuterClass {
    private void outerMethod() {
       final int outerInt = 1;
        // Method Local Inner Class
        class MethodLocalInnerClass {
            private void print() {
                System.out.println("Method local inner class " + outerInt);
            }
        }
        // Accessing the inner class
        MethodLocalInnerClass inner = new MethodLocalInnerClass();
        inner.print();
    }

    public static void main(String args[]) {
        OuterClass outer = new OuterClass();
        outer.outerMethod();
    }
}

A execução dará uma saída: Method local inner class 1.

Acessando a classe externa de uma classe interna não estática #

A referência à classe externa usa o nome da classe e this

public class OuterClass {
    public class InnerClass {
        public void method() {
            System.out.println("I can access my enclosing class: " + OuterClass.this);
        }
    }
}

Você pode acessar campos e métodos da classe externa diretamente.

public class OuterClass {
    private int counter;

    public class InnerClass {
        public void method() {
            System.out.println("I can access " + counter);
        }
    }
}

Mas em caso de colisão de nomes, você pode usar a referência de classe externa.

public class OuterClass {
    private int counter;

    public class InnerClass {
        private int counter;
        
        public void method() {
            System.out.println("My counter: " + counter);
            System.out.println("Outer counter: " + OuterClass.this.counter);
            
            // updating my counter
            counter = OuterClass.this.counter;
        }
    }
}
