Sintaxe #

  • java [ <opt> ... ] <class-name> [ <argument> ... ]

  • java [ <opt> ... ] -jar <jar-file-pathname> [ <argument> ... ]

O comando java é usado para executar um aplicativo Java a partir da linha de comando. Está disponível como parte de qualquer Java SE JRE ou JDK.

Em sistemas Windows existem duas variantes do comando java:

  • A variante java inicia o aplicativo em uma nova janela do console.
  • A variante javaw inicia o aplicativo sem criar uma nova janela de console.

Em outros sistemas (por exemplo, Linux, Mac OSX, UNIX), apenas o comando java é fornecido e não inicia uma nova janela de console.

O símbolo <opt> na sintaxe denota uma opção na linha de comando java. Os tópicos “Opções Java” e “Opções de dimensionamento de pilha e pilha” abrangem as opções mais usadas. Outros são abordados no tópico https://www.wikiod.com/pt/java/sinalizadores-de-jvm

Aulas de ponto de entrada #

Uma classe de ponto de entrada Java tem um método main com a seguinte assinatura e modificadores:

public static void main(String[] args)

Nota lateral: por causa de como os arrays funcionam, também pode ser (String args[])

Quando o comando java inicia a máquina virtual, ele carrega as classes de ponto de entrada especificadas e tenta encontrar main. Se for bem-sucedido, os argumentos da linha de comando são convertidos em objetos Java String e montados em uma matriz. Se main for invocado assim, o array não será null e não conterá nenhuma entrada null.

Um método de classe de ponto de entrada válido deve fazer o seguinte:

  • Ser nomeado main (diferencia maiúsculas de minúsculas)
  • Seja público e estático
  • Ter um tipo de retorno void
  • Ter um único argumento com um array String[]. O argumento deve estar presente e não é permitido mais de um argumento.
  • Seja genérico: parâmetros de tipo não são permitidos.
  • Ter uma classe delimitadora não genérica, de nível superior (não aninhada ou interna)

É convencional declarar a classe como ‘pública’, mas isso não é estritamente necessário. A partir do Java 5, o tipo de argumento do método main pode ser um varargs String em vez de um array de strings. main pode, opcionalmente, lançar exceções, e seu parâmetro pode ter qualquer nome, mas convencionalmente é args.

Pontos de entrada JavaFX #

A partir do Java 8 em diante, o comando java também pode iniciar diretamente um aplicativo JavaFX. JavaFX está documentado na tag JavaFX, mas um ponto de entrada JavaFX deve fazer o seguinte:

  • Estender javafx.application.Application
  • Seja público e não abstrato
  • Não ser genérico ou aninhado
  • Ter um construtor no-args public explícito ou implícito

Executando um arquivo JAR executável #

Os arquivos JAR executáveis ​​são a maneira mais simples de montar o código Java em um único arquivo que pode ser executado. *(Nota Editorial: A criação de arquivos JAR deve ser coberta por um Tópico separado.) *

Supondo que você tenha um arquivo JAR executável com o nome do caminho <jar-path>, você poderá executá-lo da seguinte maneira:

java -jar <jar-path>

Se o comando requer argumentos de linha de comando, adicione-os após o <jar-path>. Por exemplo:

java -jar <jar-path> arg1 arg2 arg3

Se você precisar fornecer opções JVM adicionais na linha de comando java, elas precisam ir antes da opção -jar. Observe que uma opção -cp / -classpath será ignorada se você usar -jar. O caminho de classe do aplicativo é determinado pelo manifesto do arquivo JAR.

Executando um aplicativo Java por meio de uma classe “principal” #

Quando um aplicativo não foi empacotado como um JAR executável, você precisa fornecer o nome de uma classe de ponto de entrada na linha de comando java.

Executando a classe HelloWorld #

O exemplo “HelloWorld” é descrito em https://www.wikiod.com/pt/java/introducao-a-linguagem-java#Criando seu primeiro programa Java . Ele consiste em uma única classe chamada HelloWorld que satisfaz os requisitos para um ponto de entrada.

Supondo que o arquivo (compilado) “HelloWorld.class” esteja no diretório atual, ele pode ser iniciado da seguinte forma:

java HelloWorld

Algumas coisas importantes a serem observadas são:

  • Devemos fornecer o nome da classe: não o caminho para o arquivo “.class” ou o arquivo “.java”.
  • Se a classe for declarada em um pacote (como a maioria das classes Java são), então o nome da classe que fornecemos ao comando java deve ser o nome completo da classe. Por exemplo, se SomeClass for declarado no pacote com.example, então o nome completo da classe será com.example.SomeClass.

Especificando um caminho de classe #

A menos que estejamos usando a sintaxe do comando java -jar, o comando java procura a classe a ser carregada pesquisando o classpath; consulte https://www.wikiod.com/pt/java/o-caminho-de-classe O comando acima está contando com o classpath padrão sendo (ou incluindo) o diretório atual. Podemos ser mais explícitos sobre isso especificando o classpath a ser usado usando a opção -cp.

java -cp . HelloWorld

Isso diz para tornar o diretório atual (que é o que “.” se refere) a única entrada no caminho de classe.

O -cp é uma opção que é processada pelo comando java. Todas as opções destinadas ao comando java devem estar antes do nome da classe. Qualquer coisa após a classe será tratada como um argumento de linha de comando para o aplicativo Java e será passado para o aplicativo no String[] que é passado para o método main.

(Se nenhuma opção -cp for fornecida, o java usará o caminho de classe fornecido pela variável de ambiente CLASSPATH. Se essa variável não estiver definida ou vazia, java usará “.” como o caminho de classe padrão. )

Solucionando problemas do comando ‘java’ #

Este exemplo cobre erros comuns com o uso do comando ‘java’.

“Comando não encontrado” #

Se você receber uma mensagem de erro como:

java: command not found

ao tentar executar o comando java, isso significa que não há comando java no caminho de busca de comandos do seu shell. A causa pode ser:

  • você não tem um Java JRE ou JDK instalado,
  • você não atualizou a variável de ambiente PATH (corretamente) em seu arquivo de inicialização do shell, ou
  • você não “forneceu” o arquivo de inicialização relevante no shell atual.

Consulte “Instalando o Java” para obter as etapas necessárias.

“Não foi possível encontrar ou carregar a classe principal” #

Esta mensagem de erro é emitida pelo comando java se ele não conseguir encontrar/carregar a classe de ponto de entrada que você especificou. Em termos gerais, existem três grandes razões pelas quais isso pode acontecer:

  • Você especificou uma classe de ponto de entrada que não existe.
  • A classe existe, mas você a especificou incorretamente.
  • A classe existe e você a especificou corretamente, mas o Java não pode encontrá-la porque o caminho de classe está incorreto.

Aqui está um procedimento para diagnosticar e resolver o problema:

  1. Descubra o nome completo da classe de ponto de entrada.

    • If you have source code for a class, then the full name consists of the package name and the simple class name. The instance the “Main” class is declared in the package “com.example.myapp” then its full name is “com.example.myapp.Main”.
    • If you have a compiled class file, you can find the class name by running javap on it.
    • If the class file is in a directory, you can infer the full class name from the directory names.
    • If the class file is in a JAR or ZIP file, you can infer the full class name from the file path in the JAR or ZIP file.

  2. Veja a mensagem de erro do comando java. A mensagem deve terminar com o nome completo da classe que java está tentando usar.

    • Check that it exactly matches the full classname for the entry-point class.
    • It should not end with “.java” or “.class”.
    • It should not contain slashes or any other character that is not legal in a Java identifier1.
    • The casing of the name should exactly match the full class name.

  3. Se você estiver usando o nome de classe correto, certifique-se de que a classe esteja realmente no caminho de classe:

    • Work out the pathname that the classname maps to; see https://www.wikiod.com/pt/java/o-caminho-de-classe#Mapeando nomes de classe para nomes de caminho
    • Work out what the classpath is; see this example: https://www.wikiod.com/pt/java/o-caminho-de-classe#Diferentes maneiras de especificar o caminho de classe
    • Look at each of the JAR and ZIP files on the classpath to see if they contain a class with the required pathname.
    • Look at each directory to see if the pathname resolves to a file within the directory.

Se a verificação manual do classpath não encontrar o problema, você pode adicionar as opções -Xdiag e -XshowSettings. O primeiro lista todas as classes que são carregadas e o último imprime as configurações que incluem o caminho de classe efetivo para a JVM.

Finalmente, existem algumas causas obscuras para esse problema:

  • Um arquivo JAR executável com um atributo Main-Class que especifica uma classe que não existe.
  • Um arquivo JAR executável com um atributo Class-Path incorreto.
  • Se você errar2 as opções antes do nome da classe, o comando java pode tentar interpretar uma delas como o nome da classe.
  • Se alguém ignorou as regras de estilo Java e usou identificadores de pacote ou classe que diferem apenas em maiúsculas e minúsculas, e você está executando em uma plataforma que trata letras maiúsculas em nomes de arquivos como não significativas.
  • Problemas com homóglifos em nomes de classes no código ou na linha de comando.

“Método principal não encontrado na classe <name>” #

Este problema acontece quando o comando java é capaz de encontrar e carregar a classe que você indicou, mas não consegue encontrar um método de ponto de entrada.

Há três explicações possíveis:

  • Se você estiver tentando executar um arquivo JAR executável, o manifesto do JAR terá um atributo “Main-Class” incorreto que especifica uma classe que não é uma classe de ponto de entrada válida.
  • Você disse ao comando java uma classe que não é uma classe de ponto de entrada.
  • A classe do ponto de entrada está incorreta; consulte https://www.wikiod.com/pt/java/o-comando-java—java-e-javaw#Aulas de ponto de entrada para obter mais informações.

Executando um aplicativo Java com dependências de biblioteca #

Esta é uma continuação dos exemplos de “classe principal” e “JAR executável”.

Os aplicativos Java típicos consistem em um código específico do aplicativo e vários códigos de biblioteca reutilizáveis ​​que você implementou ou que foi implementado por terceiros. Os últimos são comumente chamados de dependências de biblioteca e geralmente são empacotados como arquivos JAR.

Java é uma linguagem vinculada dinamicamente. Ao executar um aplicativo Java com dependências de biblioteca, a JVM precisa saber onde estão as dependências para que possa carregar classes conforme necessário. Em linhas gerais, existem duas maneiras de lidar com isso:

  • O aplicativo e suas dependências podem ser reempacotados em um único arquivo JAR que contém todas as classes e recursos necessários.

  • A JVM pode ser informada onde encontrar os arquivos JAR dependentes por meio do caminho de classe de tempo de execução.

Para um arquivo JAR executável, o caminho de classe de tempo de execução é especificado pelo atributo de manifesto “Class-Path”. (Nota Editorial: Isso deve ser descrito em um tópico separado no comando jar.) Caso contrário, o caminho de classe de tempo de execução precisa ser fornecido usando a opção -cp ou usando a variável de ambiente CLASSPATH.

Por exemplo, suponha que temos um aplicativo Java no arquivo “myApp.jar” cuja classe de ponto de entrada é com.example.MyApp. Suponha também que a aplicação dependa dos arquivos JAR da biblioteca “lib/library1.jar” e “lib/library2.jar”. Poderíamos iniciar o aplicativo usando o comando java da seguinte forma em uma linha de comando:

$ # Alternative 1 (preferred)
$ java -cp myApp.jar:lib/library1.jar:lib/library2.jar com.example.MyApp

$ # Alternative 2
$ export CLASSPATH=myApp.jar:lib/library1.jar:lib/library2.jar
$ java com.example.MyApp

(No Windows, você usaria ; em vez de : como o separador de caminho de classe e definiria a variável (local) CLASSPATH usando set em vez de export.)

Embora um desenvolvedor Java se sinta confortável com isso, não é “amigável ao usuário”. Portanto, é uma prática comum escrever um script de shell simples (ou arquivo de lote do Windows) para ocultar os detalhes que o usuário não precisa saber. Por exemplo, se você colocar o seguinte script de shell em um arquivo chamado “myApp”, torná-lo executável e colocá-lo em um diretório no caminho de pesquisa do comando:

#!/bin/bash
# The 'myApp' wrapper script

export DIR=/usr/libexec/myApp
export CLASSPATH=$DIR/myApp.jar:$DIR/lib/library1.jar:$DIR/lib/library2.jar
java com.example.MyApp

então você poderia executá-lo da seguinte forma:

$ myApp arg1 arg2 ...

Quaisquer argumentos na linha de comando serão passados ​​para o aplicativo Java por meio da expansão "$@". (Você pode fazer algo semelhante com um arquivo de lote do Windows, embora a sintaxe seja diferente.)

Espaços e outros caracteres especiais em argumentos #

Em primeiro lugar, o problema de lidar com espaços em argumentos NÃO é realmente um problema de Java. Em vez disso, é um problema que precisa ser tratado pelo shell de comando que você está usando ao executar um programa Java.

Como exemplo, vamos supor que temos o seguinte programa simples que imprime o tamanho de um arquivo:

import java.io.File;

public class PrintFileSizes {
    
    public static void main(String[] args) {
        for (String name: args) {
            File file = new File(name);
            System.out.println("Size of '" + file + "' is " + file.size());
        }
    }
}

Agora suponha que queremos imprimir o tamanho de um arquivo cujo nome de caminho contém espaços; por exemplo. /home/steve/Arquivo de teste.txt. Se executarmos o comando assim:

$ java PrintFileSizes /home/steve/Test File.txt

o shell não saberá que /home/steve/Test File.txt é na verdade um nome de caminho. Em vez disso, ele passará 2 argumentos distintos para o aplicativo Java, que tentará encontrar seus respectivos tamanhos de arquivo e falhará porque os arquivos com esses caminhos (provavelmente) não existem.

Soluções usando um shell POSIX #

Os shells POSIX incluem sh bem como derivados como bash e ksh. Se você estiver usando um desses shells, poderá resolver o problema citando o argumento.

$ java PrintFileSizes "/home/steve/Test File.txt"

As aspas duplas ao redor do nome do caminho informam ao shell que ele deve ser passado como um único argumento. As aspas serão removidas quando isso acontecer. Existem algumas outras maneiras de fazer isso:

$ java PrintFileSizes '/home/steve/Test File.txt'

Aspas simples (retas) são tratadas como aspas duplas, exceto que elas também suprimem várias expansões dentro do argumento.

$ java PrintFileSizes /home/steve/Test\ File.txt

Uma barra invertida escapa do espaço a seguir e faz com que ele não seja interpretado como um separador de argumento.

Para obter uma documentação mais abrangente, incluindo descrições de como lidar com outros caracteres especiais em argumentos, consulte o tópico de citação na documentação do Bash.

Solução para Windows #

O problema fundamental para o Windows é que, no nível do SO, os argumentos são passados ​​para um processo filho como uma única string (source). Isso significa que a responsabilidade final de analisar (ou analisar novamente) a linha de comando recai sobre o programa ou suas bibliotecas de tempo de execução. Há muita incoerência.

No caso Java, para encurtar a história:

  • Você pode colocar aspas duplas em torno de um argumento em um comando java, e isso permitirá que você passe argumentos com espaços neles.

  • Aparentemente, o próprio comando java está analisando a string de comando e acerta mais ou menos

  • No entanto, quando você tenta combinar isso com o uso de SET e substituição de variável em um arquivo de lote, fica muito complicado se as aspas duplas são removidas.

  • O shell cmd.exe aparentemente tem outros mecanismos de escape; por exemplo. dobrando aspas duplas e usando escapes ^.

Para mais detalhes, consulte a documentação do Batch-File.

Opções Java #

O comando java suporta uma ampla gama de opções:

  • Todas as opções começam com um único hífen ou sinal de menos (-): a convenção GNU/Linux de usar -- para opções “longas” não é suportada.

  • As opções devem aparecer antes do argumento <classname> ou -jar <jarfile> para serem reconhecidas. Quaisquer argumentos após eles serão tratados como argumentos a serem passados ​​para o aplicativo Java que está sendo executado.

  • As opções que não começam com -X ou -XX são opções padrão. Você pode confiar em todas as implementações Java1 para oferecer suporte a qualquer opção padrão.

  • As opções que começam com -X não são opções padrão e podem ser retiradas de uma versão Java para outra.

  • As opções que começam com -XX são opções avançadas e também podem ser retiradas.

Configurando as propriedades do sistema com -D #

A opção -D<property>=<value> é usada para definir uma propriedade no objeto Properties do sistema. Este parâmetro pode ser repetido para definir diferentes propriedades.

Opções de memória, pilha e coletor de lixo #

As principais opções para controlar os tamanhos de heap e pilha estão documentadas em https://www.wikiod.com/pt/java/gerenciamento-de-memoria-java#Configurando os tamanhos de Heap, PermGen e Stack (Nota editorial: As opções do Garbage Collector devem ser descritas no mesmo tópico.)

Habilitando e desabilitando asserções #

As opções -ea e -da respectivamente habilitam e desabilitam a verificação Java assert:

  • Todas as verificações de asserções estão desabilitadas por padrão.
  • A opção -ea permite a verificação de todas as asserções
  • O -ea:<packagename>... permite a verificação de asserções em um pacote e todos os subpacotes.
  • O -ea:<classname>... permite a verificação de assertivas em uma classe.
  • A opção -da desativa a verificação de todas as asserções
  • O -da:<packagename>... desativa a verificação de asserções em um pacote e todos os subpacotes.
  • O -da:<classname>... desativa a verificação de assertivas em uma classe.
  • A opção -esa permite verificar todas as classes do sistema.
  • A opção -dsa desativa a verificação de todas as classes do sistema.

As opções podem ser combinadas. Por exemplo.

$ # Enable all assertion checking in non-system classes 
$ java -ea -dsa MyApp

$ # Enable assertions for all classes in a package except for one.
$ java -ea:com.wombat.fruitbat... -da:com.wombat.fruitbat.Brickbat MyApp

Observe que habilitar a verificação de asserção pode alterar o comportamento de uma programação Java.

  • É responsável por tornar a aplicação mais lenta em geral.
  • Pode fazer com que métodos específicos demorem mais para serem executados, o que pode alterar o tempo de threads em um aplicativo multithread.
  • Pode introduzir relações fortuitas acontece-antes que podem causar o desaparecimento de anomalias de memória.
  • Uma instrução assert implementada incorretamente pode ter efeitos colaterais indesejados.

Selecionando o tipo de VM #

As opções -client e -server permitem que você selecione entre duas formas diferentes da VM HotSpot:

  • O formulário “cliente” é ajustado para aplicativos do usuário e oferece inicialização mais rápida.
  • O formulário “servidor” é ajustado para aplicativos de longa duração. Leva mais tempo para capturar estatísticas durante o “aquecimento” da JVM, o que permite que o compilador JIT faça um trabalho melhor de otimização do código nativo.

Por padrão, a JVM será executada no modo de 64 bits, se possível, dependendo dos recursos da plataforma. As opções -d32 e -d64 permitem que você selecione o modo explicitamente.
