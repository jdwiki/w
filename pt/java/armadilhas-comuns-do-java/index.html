Este tópico descreve alguns dos erros comuns cometidos por iniciantes em Java.

Isso inclui quaisquer erros comuns no uso da linguagem Java ou no entendimento do ambiente de tempo de execução.

Os erros associados a APIs específicas podem ser descritos em tópicos específicos dessas APIs. Strings são um caso especial; eles são abordados na Especificação de linguagem Java. Detalhes além de erros comuns podem ser descritos neste tópico sobre Strings.

Armadilha: usando == para comparar objetos de wrappers primitivos, como Integer #

(Esta armadilha se aplica igualmente a todos os tipos de wrapper primitivos, mas vamos ilustrá-la para Integer e int.)

Ao trabalhar com objetos Integer, é tentador usar == para comparar valores, porque isso é o que você faria com valores int. E, em alguns casos, isso parecerá funcionar:

Integer int1_1 = Integer.valueOf("1");
Integer int1_2 = Integer.valueOf(1);

System.out.println("int1_1 == int1_2: " + (int1_1 == int1_2));          // true
System.out.println("int1_1 equals int1_2: " + int1_1.equals(int1_2));   // true

Aqui criamos dois objetos Integer com o valor 1 e os comparamos (neste caso criamos um a partir de um literal String e um a partir de um literal int. Existem outras alternativas). Além disso, observamos que os dois métodos de comparação (== e equals) produzem true.

Esse comportamento muda quando escolhemos valores diferentes:

Integer int2_1 = Integer.valueOf("1000");
Integer int2_2 = Integer.valueOf(1000);

System.out.println("int2_1 == int2_2: " + (int2_1 == int2_2));          // false
System.out.println("int2_1 equals int2_2: " + int2_1.equals(int2_2));   // true

Neste caso, apenas a comparação equals produz o resultado correto.

A razão para essa diferença de comportamento é que a JVM mantém um cache de objetos Integer para o intervalo de -128 a 127. (O valor superior pode ser substituído pela propriedade do sistema “java.lang.Integer.IntegerCache.high” ou o argumento JVM “-XX:AutoBoxCacheMax=size”). Para valores neste intervalo, o Integer.valueOf() retornará o valor armazenado em cache em vez de criar um novo.

Assim, no primeiro exemplo, as chamadas Integer.valueOf(1) e Integer.valueOf("1") retornaram a mesma instância Integer armazenada em cache. Por outro lado, no segundo exemplo, Integer.valueOf(1000) e Integer.valueOf("1000") criaram e retornaram novos objetos Integer.

O operador == para tipos de referência testa a igualdade de referência (ou seja, o mesmo objeto). Portanto, no primeiro exemplo int1_1 == int1_2 é true porque as referências são as mesmas. No segundo exemplo int2_1 == int2_2 é false porque as referências são diferentes.

Armadilha: usando == para comparar strings #

Um erro comum para iniciantes em Java é usar o operador == para testar se duas strings são iguais. Por exemplo:

public class Hello {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0] == "hello") {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

O programa acima deve testar o primeiro argumento da linha de comando e imprimir mensagens diferentes quando ele e não é a palavra “hello”. Mas o problema é que não vai funcionar. Esse programa exibirá “Você está se sentindo mal-humorado hoje?” não importa qual seja o primeiro argumento da linha de comando.

Neste caso em particular, o String “hello” é colocado no pool de strings enquanto o String args[0] reside no heap. Isso significa que existem dois objetos representando o mesmo literal, cada um com sua referência. Como == testa referências, não igualdade real, a comparação resultará em falso na maioria das vezes. Isso não significa que sempre o fará.

Quando você usa == para testar strings, o que você está realmente testando é se dois objetos String são o mesmo objeto Java. Infelizmente, não é isso que igualdade de strings significa em Java. Na verdade, a maneira correta de testar strings é usar o método equals(Object). Para um par de strings, geralmente queremos testar se eles consistem nos mesmos caracteres na mesma ordem.

public class Hello2 {
    public static void main(String[] args) {
        if (args.length > 0) {
            if (args[0].equals("hello")) {
                System.out.println("Hello back to you");
            } else {
                System.out.println("Are you feeling grumpy today?");
            }
        }
    }
}

Mas na verdade fica pior. O problema é que == vai dar a resposta esperada em algumas circunstâncias. Por exemplo

public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hello";
        if (s1 == s2) {
            System.out.println("same");
        } else {
            System.out.println("different");
        }
    }
}

Curiosamente, isso imprimirá “mesmo”, mesmo que estejamos testando as strings da maneira errada. Por que é que? Porque a Java Language Specification (Seção 3.10.5: String Liteals) estipula que quaisquer duas strings »literals« consistindo nos mesmos caracteres serão, na verdade, representadas pelo mesmo objeto Java. Portanto, o teste == dará true para literais iguais. (Os literais de string são “internos” e adicionados a um “pool de strings” compartilhado quando seu código é carregado, mas isso é na verdade um detalhe de implementação.)

Para aumentar a confusão, a Java Language Specification também estipula que quando você tem uma expressão constante de tempo de compilação que concatena dois literais de string, isso é equivalente a um único literal. Desta forma:

    public class Test1 {
    public static void main(String[] args) {
        String s1 = "hello";
        String s2 = "hel" + "lo";
        String s3 = " mum";
        if (s1 == s2) {
            System.out.println("1. same");
        } else {
            System.out.println("1. different");
        }
        if (s1 + s3 == "hello mum") {
            System.out.println("2. same");
        } else {
            System.out.println("2. different");
        }
    }
}

Isso produzirá “1. igual” e “2. diferente”. No primeiro caso, a expressão + é avaliada em tempo de compilação e comparamos um objeto String com ele mesmo. No segundo caso, ele é avaliado em tempo de execução e comparamos dois objetos String diferentes

Em resumo, usar == para testar strings em Java é quase sempre incorreto, mas não é garantido que dê a resposta errada.

Armadilha: esquecer de liberar recursos #

Sempre que um programa abre um recurso, como um arquivo ou uma conexão de rede, é importante liberar o recurso assim que terminar de usá-lo. Cuidado semelhante deve ser tomado se qualquer exceção for lançada durante as operações em tais recursos. Pode-se argumentar que o FileInputStream tem um finalizador que invoca o método close() em um evento de coleta de lixo; no entanto, como não podemos ter certeza de quando um ciclo de coleta de lixo será iniciado, o fluxo de entrada pode consumir recursos do computador por um período de tempo indefinido. O recurso deve ser fechado em uma seção finally de um bloco try-catch:

private static void printFileJava6() throws IOException {
    FileInputStream input;
    try {
        input = new FileInputStream("file.txt");
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    } finally {
        if (input != null) {
            input.close();
        }
    }
}

Desde o Java 7, há uma instrução realmente útil e organizada introduzida no Java 7 particularmente para este caso, chamada try-with-resources:

private static void printFileJava7() throws IOException {
    try (FileInputStream input = new FileInputStream("file.txt")) {
        int data = input.read();
        while (data != -1){
            System.out.print((char) data);
            data = input.read();
        }
    }
}

A instrução try-with-resources pode ser usada com qualquer objeto que implemente a interface Closeable ou AutoCloseable. Ele garante que cada recurso seja fechado até o final da instrução. A diferença entre as duas interfaces é que o método close() de Closeable lança uma IOException que deve ser tratada de alguma forma.

Nos casos em que o recurso já foi aberto, mas deve ser fechado com segurança após o uso, pode-se atribuí-lo a uma variável local dentro do try-with-resources

private static void printFileJava7(InputStream extResource) throws IOException {
    try (InputStream input = extResource) {
        ... //access resource
    }
}

A variável de recurso local criada no construtor try-with-resources é efetivamente final.

Armadilha: testar um arquivo antes de tentar abri-lo. #

Algumas pessoas recomendam que você aplique vários testes a um arquivo antes de tentar abri-lo para fornecer diagnósticos melhores ou evitar lidar com exceções. Por exemplo, este método tenta verificar se path corresponde a um arquivo legível:

public static File getValidatedFile(String path) throws IOException {
    File f = new File(path);
    if (!f.exists()) throw new IOException("Error: not found: " + path);
    if (!f.isFile()) throw new IOException("Error: Is a directory: " + path);
    if (!f.canRead()) throw new IOException("Error: cannot read file: " + path);
    return f;
}

Você pode usar o método acima assim:

File f = null;
try {
    f = getValidatedFile("somefile");
} catch (IOException ex) {
    System.err.println(ex.getMessage());
    return;
}
try (InputStream is = new FileInputStream(file)) {
    // Read data etc.
}

O primeiro problema está na assinatura de FileInputStream(File) porque o compilador ainda insistirá em capturar IOException aqui, ou mais adiante na pilha.

O segundo problema é que as verificações realizadas pelo getValidatedFile não garantem que o FileInputStream seja bem-sucedido.

  • Condições de corrida: outro thread ou um processo separado pode renomear o arquivo, excluir o arquivo ou remover o acesso de leitura após o retorno de getValidatedFile. Isso levaria a um IOException “simples” sem a mensagem personalizada.

  • Existem casos extremos não cobertos por esses testes. Por exemplo, em um sistema com SELinux no modo “enforcing”, uma tentativa de ler um arquivo pode falhar apesar de canRead() retornar true.

O terceiro problema é que os testes são ineficientes. Por exemplo, as chamadas exists, isFile e canRead farão uma syscall para realizar a verificação necessária. Outra syscall é feita para abrir o arquivo, que repete as mesmas verificações nos bastidores.

Resumindo, métodos como getValidatedFile são equivocados. É melhor simplesmente tentar abrir o arquivo e manipular a exceção:

try (InputStream is = new FileInputStream("somefile")) {
    // Read data etc.
} catch (IOException ex) {
    System.err.println("IO Error processing 'somefile': " + ex.getMessage());
    return;
}

Se você quiser distinguir erros de E/S lançados durante a abertura e a leitura, poderá usar um try/catch aninhado. Se você quiser produzir diagnósticos melhores para falhas abertas, você pode executar as verificações exists, isFile e canRead no manipulador.

Armadilha: pensar em variáveis ​​como objetos #

Nenhuma variável Java representa um objeto.

String foo;   // NOT AN OBJECT

Nem qualquer array Java contém objetos.

String bar[] = new String[100];  // No member is an object.

Se você pensar erroneamente em variáveis ​​como objetos, o comportamento real da linguagem Java irá surpreendê-lo.

  • Para variáveis ​​Java que possuem um tipo primitivo (como int ou float) a variável mantém uma cópia do valor. Todas as cópias de um valor primitivo são indistinguíveis; ou seja, há apenas um valor int para o número um. Os valores primitivos não são objetos e não se comportam como objetos.

  • Para variáveis ​​Java que possuem um tipo de referência (uma classe ou um tipo de matriz), a variável contém uma referência. Todas as cópias de uma referência são indistinguíveis. As referências podem apontar para objetos ou podem ser null, o que significa que não apontam para nenhum objeto. No entanto, eles não são objetos e não se comportam como objetos.

As variáveis ​​não são objetos em nenhum dos casos e não contêm objetos em nenhum dos casos. Eles podem conter referências a objetos, mas isso significa algo diferente.

Classe de exemplo #

Os exemplos a seguir usam essa classe, que representa um ponto no espaço 2D.

public final class MutableLocation {
   public int x;
   public int y;

   public MutableLocation(int x, int y) {
       this.x = x;
       this.y = y;
   }

   public boolean equals(Object other) {
       if (!(other instanceof MutableLocation) {
           return false;
       }
       MutableLocation that = (MutableLocation) other;
       return this.x == that.x && this.y == that.y;
   }
}

Uma instância desta classe é um objeto que possui dois campos x e y que possuem o tipo int.

Podemos ter muitas instâncias da classe MutableLocation. Alguns irão representar os mesmos locais no espaço 2D; ou seja, os respectivos valores de x e y serão correspondentes. Outros irão representar locais diferentes.

Várias variáveis ​​podem apontar para o mesmo objeto #

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

Acima, declaramos três variáveis ​​here, there e elsewhere que podem conter referências a objetos MutableLocation.

Se você (incorretamente) pensar nessas variáveis ​​como objetos, é provável que você interprete mal as declarações dizendo:

  1. Copie o local “[1, 2]” para aqui
  2. Copie o local “[1, 2]” para there
  3. Copie o local “[1, 2]” para em outro lugar

A partir disso, você provavelmente inferirá que temos três objetos independentes nas três variáveis. Na verdade, existem apenas dois objetos criados pelo acima. As variáveis ​​here e there na verdade se referem ao mesmo objeto.

Podemos demonstrar isso. Assumindo as declarações de variáveis ​​como acima:

System.out.println("BEFORE: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);
here.x = 42;
System.out.println("AFTER: here.x is " + here.x + ", there.x is " + there.x +
                   "elsewhere.x is " + elsewhere.x);

Isso produzirá o seguinte:

BEFORE: here.x is 1, there.x is 1, elsewhere.x is 1
AFTER: here.x is 42, there.x is 42, elsewhere.x is 1

Atribuímos um novo valor a here.x e isso mudou o valor que vemos através de there.x. Eles estão se referindo ao mesmo objeto. Mas o valor que vemos via elsewhere.x não mudou, então elsewhere deve se referir a um objeto diferente.

Se uma variável fosse um objeto, então a atribuição here.x = 42 não mudaria there.x.

O operador de igualdade NÃO testa se dois objetos são iguais

A aplicação do operador de igualdade (==) aos valores de referência testa se os valores se referem ao mesmo objeto. Ele não testa se dois objetos (diferentes) são “iguais” no sentido intuitivo.

 MutableLocation here = new MutableLocation(1, 2);
 MutableLocation there = here;
 MutableLocation elsewhere = new MutableLocation(1, 2);

 if (here == there) {
     System.out.println("here is there");
 }
 if (here == elsewhere) {
     System.out.println("here is elsewhere");
 }

Isso imprimirá “aqui está lá”, mas não imprimirá “aqui está em outro lugar”. (As referências em here e elsewhere são para dois objetos distintos.)

Por outro lado, se chamarmos o método equals(Object) que implementamos acima, testaremos se duas instâncias de MutableLocation têm uma localização igual.

 if (here.equals(there)) {
     System.out.println("here equals there");
 }
 if (here.equals(elsewhere)) {
     System.out.println("here equals elsewhere");
 }

Isso imprimirá ambas as mensagens. Em particular, here.equals(elsewhere) retorna true porque os critérios semânticos que escolhemos para igualdade de dois objetos MutableLocation foram satisfeitos.

Chamadas de método NÃO passam objetos #

As chamadas de método Java usam pass by value1 para passar argumentos e retornar um resultado.

Quando você passa um valor de referência para um método, na verdade você está passando uma referência para um objeto por valor, o que significa que está criando uma cópia da referência do objeto.

Contanto que ambas as referências de objeto ainda estejam apontando para o mesmo objeto, você pode modificar esse objeto de qualquer referência, e isso é o que causa confusão para alguns.

No entanto, você não está passando um objeto por referência2. A diferença é que, se a cópia de referência do objeto for modificada para apontar para outro objeto, a referência do objeto original ainda apontará para o objeto original.

void f(MutableLocation foo) {  
    foo = new MutableLocation(3, 4);   // Point local foo at a different object.
}

void g() {
    MutableLocation foo = MutableLocation(1, 2);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 1".
}

Nem você está passando uma cópia do objeto.

void f(MutableLocation foo) {  
    foo.x = 42;
}

void g() {
    MutableLocation foo = new MutableLocation(0, 0);
    f(foo);
    System.out.println("foo.x is " + foo.x); // Prints "foo.x is 42"
}

1 - Em linguagens como Python e Ruby, o termo “passar por compartilhamento” é preferido para “passar por valor” de um objeto/referência.

2 - O termo “passar por referência” ou “chamar por referência” tem um significado muito específico na terminologia da linguagem de programação. Com efeito, significa que você passa o endereço de uma variável ou de um elemento de array, de modo que quando o método chamado atribuir um novo valor ao argumento formal, ele altere o valor da variável original. Java não suporta isso. Para uma descrição mais completa dos diferentes mecanismos para passar parâmetros, consulte https://en.wikipedia.org/wiki/Evaluation_strategy.

Armadilha: vazamentos de memória #

Java gerencia a memória automaticamente. Não é necessário liberar memória manualmente. A memória de um objeto no heap pode ser liberada por um coletor de lixo quando o objeto não é mais alcançável por um encadeamento ativo.

No entanto, você pode impedir que a memória seja liberada, permitindo que os objetos sejam acessíveis que não são mais necessários. Quer você chame isso de vazamento de memória ou pacote de memória, o resultado é o mesmo – um aumento desnecessário na memória alocada.

Vazamentos de memória em Java podem ocorrer de várias maneiras, mas o motivo mais comum são referências de objetos eternas, porque o coletor de lixo não pode remover objetos do heap enquanto ainda houver referências a eles.

Campos estáticos

Pode-se criar tal referência definindo uma classe com um campo static contendo alguma coleção de objetos, e esquecendo de definir esse campo static para null depois que a coleção não for mais necessária. Campos static são considerados raízes GC e nunca são coletados. Outro problema são os vazamentos na memória não heap quando JNI é usado.

Vazamento do carregador de classe

De longe, porém, o tipo mais insidioso de vazamento de memória é o vazamento do carregador de classe. Um carregador de classe contém uma referência a cada classe que carregou, e cada classe contém uma referência ao seu carregador de classe. Cada objeto contém uma referência à sua classe também. Portanto, se mesmo um objeto único de uma classe carregada por um carregador de classe não for lixo, nem uma única classe que esse carregador de classe carregou poderá ser coletada. Como cada classe também se refere a seus campos estáticos, eles também não podem ser coletados.

Vazamento de acumulação O exemplo de vazamento de acumulação pode ter a seguinte aparência:

final ScheduledExecutorService scheduledExecutorService = Executors.newScheduledThreadPool(1);
final Deque<BigDecimal> numbers = new LinkedBlockingDeque<>();
final BigDecimal divisor = new BigDecimal(51);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    BigDecimal number = numbers.peekLast();
    if (number != null && number.remainder(divisor).byteValue() == 0) {
        System.out.println("Number: " + number);
        System.out.println("Deque size: " + numbers.size());
    }
}, 10, 10, TimeUnit.MILLISECONDS);

scheduledExecutorService.scheduleAtFixedRate(() -> {
    numbers.add(new BigDecimal(System.currentTimeMillis()));
}, 10, 10, TimeUnit.MILLISECONDS);

try {
    scheduledExecutorService.awaitTermination(1, TimeUnit.DAYS);
} catch (InterruptedException e) {
    e.printStackTrace();
}

Este exemplo cria duas tarefas agendadas. A primeira tarefa pega o último número de um deque chamado numbers e, se o número for divisível por 51, imprime o número e o tamanho do deque. A segunda tarefa coloca números no deque. Ambas as tarefas são agendadas a uma taxa fixa e são executadas a cada 10 ms.

Se o código for executado, você verá que o tamanho do deque está aumentando permanentemente. Isso eventualmente fará com que o deque seja preenchido com objetos que consomem toda a memória heap disponível.

Para evitar isso preservando a semântica deste programa, podemos usar um método diferente para obter números do deque: pollLast. Ao contrário do método peekLast, pollLast retorna o elemento e o remove do deque enquanto peekLast retorna apenas o último elemento.

Armadilha: Não entender que String é uma classe imutável #

Novos programadores Java muitas vezes esquecem, ou não compreendem completamente, que a classe Java String é imutável. Isso leva a problemas como o do exemplo a seguir:

public class Shout {
    public static void main(String[] args) {
        for (String s : args) {
            s.toUpperCase();
            System.out.print(s);
            System.out.print(" ");
        }
        System.out.println();
    }
}

O código acima deve imprimir argumentos de linha de comando em letras maiúsculas. Infelizmente, não funciona, o caso dos argumentos não é alterado. O problema é esta afirmação:

s.toUpperCase();

Você pode pensar que chamar toUpperCase() vai mudar s para uma string maiúscula. Não. Não pode! Objetos String são imutáveis. Eles não podem ser alterados.

Na realidade, o método toUpperCase() retorna um objeto String que é uma versão em maiúsculas da String na qual você o chama. Este provavelmente será um novo objeto String, mas se s já estivesse em letras maiúsculas, o resultado poderia ser a string existente.

Portanto, para usar esse método efetivamente, você precisa usar o objeto retornado pela chamada do método; por exemplo:

s = s.toUpperCase();

Na verdade, a regra “strings nunca mudam” se aplica a todos os métodos String. Se você se lembrar disso, poderá evitar toda uma categoria de erros de principiante.

Armadilha: combinando atribuição e efeitos colaterais #

Ocasionalmente, vemos perguntas do StackOverflow Java (e perguntas C ou C++) que perguntam algo assim:

i += a[i++] + b[i--];

avalia como … para alguns estados iniciais conhecidos de i, a e b.

De um modo geral:

  • para Java, a resposta é sempre especificada1, mas não é óbvia e geralmente difícil de descobrir
  • para C e C++, a resposta geralmente não é especificada.

Esses exemplos são frequentemente usados ​​em exames ou entrevistas de emprego como uma tentativa de ver se o aluno ou entrevistado entende como a avaliação de expressão realmente funciona na linguagem de programação Java. Isso é indiscutivelmente legítimo como um “teste de conhecimento”, mas isso não significa que você deva fazer isso em um programa real.

Para ilustrar, o exemplo aparentemente simples a seguir apareceu algumas vezes em perguntas do StackOverflow (como este aqui). Em alguns casos, aparece como um erro genuíno no código de alguém.

int a = 1;
a = a++;
System.out.println(a);    // What does this print.

A maioria dos programadores (incluindo especialistas em Java) lendo essas instruções rapidamente diriam que ela produz 2. Na verdade, ele gera 1. Para obter uma explicação detalhada do motivo, leia esta resposta.

No entanto, a verdadeira lição deste e de exemplos semelhantes é que qualquer instrução Java que ambos atribui a e efeitos colaterais à mesma variável será na melhor das hipóteses difícil de entender e na pior totalmente enganosa . Você deve evitar escrever código como este.

1 - módulo de problemas potenciais com o Java Memory Model se as variáveis ​​ou objetos estiverem visíveis para outros threads.
