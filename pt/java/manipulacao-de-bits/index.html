  • Ao contrário do C/C++, o Java é completamente neutro em relação ao hardware da máquina subjacente. Você não obtém comportamento endian grande ou pequeno por padrão; você precisa especificar explicitamente qual comportamento deseja.

  • O tipo byte é assinado, com o intervalo de -128 a +127. Para converter um valor de byte para seu equivalente sem sinal, mascare-o com 0xFF assim: (b & 0xFF).

Verificando, configurando, limpando e alternando bits individuais. Usando máscara longa como bit #

Assumindo que queremos modificar o bit n de um inteiro primitivo, i (byte, short, char, int ou long):

(i & 1 << n) != 0 // checks bit 'n'
i |= 1 << n;      // sets bit 'n' to 1
i &= ~(1 << n);   // sets bit 'n' to 0
i ^= 1 << n;      // toggles the value of bit 'n'

Usando long/int/short/byte como uma máscara de bits:

public class BitMaskExample {
    private static final long FIRST_BIT = 1L << 0;
    private static final long SECOND_BIT = 1L << 1;
    private static final long THIRD_BIT = 1L << 2;
    private static final long FOURTH_BIT = 1L << 3;
    private static final long FIFTH_BIT = 1L << 4;
    private static final long BIT_55 = 1L << 54;

    public static void main(String[] args) {
        checkBitMask(FIRST_BIT | THIRD_BIT | FIFTH_BIT | BIT_55);
    }

    private static void checkBitMask(long bitmask) {
        System.out.println("FIRST_BIT: " + ((bitmask & FIRST_BIT) != 0));
        System.out.println("SECOND_BIT: " + ((bitmask & SECOND_BIT) != 0));
        System.out.println("THIRD_BIT: " + ((bitmask & THIRD_BIT) != 0));
        System.out.println("FOURTh_BIT: " + ((bitmask & FOURTH_BIT) != 0));
        System.out.println("FIFTH_BIT: " + ((bitmask & FIFTH_BIT) != 0));
        System.out.println("BIT_55: " + ((bitmask & BIT_55) != 0));
    }
}

Impressões

FIRST_BIT: true
SECOND_BIT: false
THIRD_BIT: true
FOURTh_BIT: false
FIFTH_BIT: true
BIT_55: true

que corresponde a essa máscara que passamos como parâmetro checkBitMask: FIRST_BIT | TERCEIRO_BIT | FIFTH_BIT | BIT_55.

classe java.util.BitSet #

Desde a versão 1.7 existe uma classe java.util.BitSet que fornece uma interface de manipulação e armazenamento de bits simples e amigável:

final BitSet bitSet = new BitSet(8); // by default all bits are unset

IntStream.range(0, 8).filter(i -> i % 2 == 0).forEach(bitSet::set); // {0, 2, 4, 6}

bitSet.set(3); // {0, 2, 3, 4, 6}

bitSet.set(3, false); // {0, 2, 4, 6}

final boolean b = bitSet.get(3); // b = false

bitSet.flip(6); // {0, 2, 4}

bitSet.set(100); // {0, 2, 4, 100} - expands automatically

BitSet implementa Clonable e Serializable, e sob o capô todos os valores de bit são armazenados no campo long[] words, que se expande automaticamente.

Ele também suporta operações lógicas de conjunto inteiro and, or, xor, andNot:

bitSet.and(new BitSet(8));
bitSet.or(new BitSet(8));
bitSet.xor(new BitSet(8));
bitSet.andNot(new BitSet(8));

Verificando se um número é uma potência de 2 #

Se um inteiro x for uma potência de 2, apenas um bit é definido, enquanto x-1 tem todos os bits definidos depois disso. Por exemplo: 4 é 100 e 3 é 011 como número binário, o que satisfaz a condição acima mencionada. Zero não é uma potência de 2 e deve ser verificado explicitamente.

boolean isPowerOfTwo(int x)
{
    return (x != 0) && ((x & (x - 1)) == 0);
}

Uso para Shift Esquerda e Direita

Vamos supor que temos três tipos de permissões, READ, WRITE e EXECUTE. Cada permissão pode variar de 0 a 7. (Vamos supor um sistema numérico de 4 bits)

RECURSO = READ WRITE EXECUTE (número de 12 bits)

RECURSO = 0100 0110 0101 = 4 6 5 (número de 12 bits)

Como podemos obter as permissões (número de 12 bits), definidas acima (número de 12 bits)?

0100 0110 0101

0000 0000 0111 (&)

0000 0000 0101 = 5

Então, é assim que podemos obter as permissões EXECUTE do RESOURCE. Agora, e se quisermos obter permissões de LER do RESOURCE?

0100 0110 0101

0111 0000 0000 (&)

0100 0000 0000 = 1024

Certo? Você provavelmente está assumindo isso? Mas, as permissões resultam em 1024. Queremos obter apenas permissões READ para o recurso. Não se preocupe, é por isso que tivemos os operadores de turno. Se virmos, as permissões READ estão 8 bits atrás do resultado real, então se aplicar algum operador shift, que trará as permissões READ bem à direita do resultado? E se fizermos:

0100 0000 0000 » 8 => 0000 0000 0100 (Porque é um número positivo então substituído por 0’s, se você não se importa com o sinal, apenas use unsigned operador de deslocamento para a direita)

Agora, na verdade, temos as permissões READ, que são 4.

Agora, por exemplo, recebemos as permissões READ, WRITE, EXECUTE para um RESOURCE. O que podemos fazer para conceder permissões para este RESOURCE?

Vamos primeiro pegar o exemplo de permissões binárias. (Ainda assumindo o sistema numérico de 4 bits)

LER = 0001

ESCREVER = 0100

EXECUTAR = 0110

Se você está pensando que vamos simplesmente fazer:

LER | ESCREVA | EXECUTE, você está certo, mas não exatamente. Veja, o que acontecerá se realizarmos READ | ESCREVA | EXECUTAR

0001 | 0100 | 0110 => 0111

Mas as permissões estão sendo representadas (no nosso exemplo) como 0001 0100 0110

Portanto, para fazer isso, sabemos que READ é colocado 8 bits atrás, WRITE é colocado 4 bits atrás e PERMISSIONS é colocado por último. O sistema numérico usado para permissões RESOURCE é, na verdade, de 12 bits (no nosso exemplo). Pode (será) diferente em diferentes sistemas.

(LER « 8) | (ESCREVA « 4) | (EXECUTAR)

0000 0000 0001 « 8 (LER)

0001 0000 0000 (Deslocamento à esquerda em 8 bits)

0000 0000 0100 « 4 (ESCREVA)

0000 0100 0000 (Deslocamento à esquerda em 4 bits)

0000 0000 0001 (EXECUTAR)

Agora, se adicionarmos os resultados do deslocamento acima, será algo como;

0001 0000 0000 (LER)

0000 0100 0000 (ESCREVA)

0000 0000 0001 (EXECUTAR)

0001 0100 0001 (PERMISSÕES)

Empacotando/desempacotando valores como fragmentos de bits #

É comum que o desempenho da memória comprima vários valores em um único valor primitivo. Isso pode ser útil para passar várias informações em uma única variável.

Por exemplo, pode-se compactar 3 bytes - como código de cores em RGB - em um único int.

Embalando os valores

// Raw bytes as input
byte[] b = {(byte)0x65, (byte)0xFF, (byte)0x31};

// Packed in big endian: x == 0x65FF31
int x = (b[0] & 0xFF) << 16  // Red
      | (b[1] & 0xFF) <<  8  // Green
      | (b[2] & 0xFF) <<  0; // Blue

// Packed in little endian: y == 0x31FF65
int y = (b[0] & 0xFF) <<  0
      | (b[1] & 0xFF) <<  8
      | (b[2] & 0xFF) << 16;

Descompactando os valores

// Raw int32 as input
int x = 0x31FF65;

// Unpacked in big endian: {0x65, 0xFF, 0x31}
byte[] c = {
    (byte)(x >> 16),
    (byte)(x >>  8),
    (byte)(x & 0xFF)
};

// Unpacked in little endian: {0x31, 0xFF, 0x65}
byte[] d = {
    (byte)(x & 0xFF),
    (byte)(x >>  8),
    (byte)(x >> 16)
};

Expressando a potência de 2 #

Para expressar a potência de 2 (2^n) de inteiros, pode-se usar uma operação de deslocamento de bits que permite especificar explicitamente o n.

A sintaxe é basicamente:

int pow2 = 1<<n;

Exemplos:

int twoExp4 = 1<<4; //2^4
int twoExp5 = 1<<5; //2^5
int twoExp6 = 1<<6; //2^6
...
int twoExp31 = 1<<31; //2^31

Isso é especialmente útil ao definir valores constantes que devem tornar aparente que uma potência de 2 é usada, em vez de usar valores hexadecimais ou decimais.

int twoExp4 = 0x10; //hexadecimal
int twoExp5 = 0x20; //hexadecimal
int twoExp6 = 64; //decimal
...
int twoExp31 = -2147483648; //is that a power of 2?

Um método simples para calcular a potência int de 2 seria

int pow2(int exp){
    return 1<<exp;
}

Turno assinado vs não assinado #

Em Java, todas as primitivas numéricas são assinadas. Por exemplo, um int sempre representa valores de [-2^31 - 1, 2^31], mantendo o primeiro bit para sinalizar o valor - 1 para valor negativo, 0 para positivo.

Operadores de deslocamento básicos >> e << são operadores assinados. Eles conservarão o sinal do valor.

Mas é comum programadores usarem números para armazenar valores sem sinal. Para um int, significa mudar o intervalo para [0, 2^32 - 1], para ter o dobro do valor de um int assinado.

Para os usuários avançados, o bit para sinal não tem significado. É por isso que Java adicionou >>>, um operador de deslocamento à esquerda, desconsiderando esse bit de sinal.

                initial value:               4 (                                100)
     signed left-shift: 4 << 1               8 (                               1000)
    signed right-shift: 4 >> 1               2 (                                 10)
 unsigned right-shift: 4 >>> 1               2 (                                 10)
                initial value:              -4 (   11111111111111111111111111111100)
    signed left-shift: -4 << 1              -8 (   11111111111111111111111111111000)
   signed right-shift: -4 >> 1              -2 (   11111111111111111111111111111110)
unsigned right-shift: -4 >>> 1      2147483646 (    1111111111111111111111111111110)

Por que não há <<< ?

Isso vem da definição pretendida de deslocamento à direita. À medida que preenche os espaços vazios à esquerda, não há decisão a tomar em relação ao bit de sinal. Como consequência, não há necessidade de 2 operadores diferentes.
