Armadilha: uso incorreto de wait() / notify() #

Os métodos object.wait(), object.notify() e object.notifyAll() devem ser usados ​​de uma maneira muito específica. (consulte https://www.wikiod.com/pt/java )

O problema “Notificação perdida” #

Um erro comum para iniciantes é chamar incondicionalmente object.wait()

private final Object lock = new Object();

public void myConsumer() {
    synchronized (lock) {
        lock.wait();     // DON'T DO THIS!!
    }
    doSomething();
}

A razão pela qual isso está errado é que depende de alguma outra thread para chamar lock.notify() ou lock.notifyAll(), mas nada garante que a outra thread não fez essa chamada antes da thread consumidora chamada lock .wait().

lock.notify() e lock.notifyAll() não fazem nada se alguma outra thread não estiver já esperando pela notificação. A thread que chama myConsumer() neste exemplo irá travar para sempre se for tarde demais para receber a notificação.

O bug “Estado de Monitoramento Ilegal” #

Se você chamar wait() ou notify() em um objeto sem segurar o bloqueio, a JVM lançará IllegalMonitorStateException.

public void myConsumer() {
    lock.wait();      // throws exception
    consume();
}

public void myProducer() {
    produce();
    lock.notify();    // throws exception
}

(O design para wait() / notify() requer que o bloqueio seja mantido porque isso é necessário para evitar condições sistêmicas de corrida. Se fosse possível chamar wait() ou notify() sem bloquear , então seria impossível implementar o caso de uso primário para essas primitivas: esperando que uma condição ocorresse.)

Aguardar/notificar é de nível muito baixo #

A melhor maneira de evitar problemas com wait() e notify() é não usá-los. A maioria dos problemas de sincronização podem ser resolvidos usando os objetos de sincronização de nível superior (filas, barreiras, semáforos, etc.) que estão disponíveis no pacote java.utils.concurrent.

Armadilha - Estendendo ‘java.lang.Thread’ #

O javadoc para a classe Thread mostra duas maneiras de definir e usar um thread:

Usando uma classe de thread personalizada:

 class PrimeThread extends Thread {
     long minPrime;
     PrimeThread(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeThread p = new PrimeThread(143);
 p.start();

Usando um Executável:

 class PrimeRun implements Runnable {
     long minPrime;
     PrimeRun(long minPrime) {
         this.minPrime = minPrime;
     }

     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }

 PrimeRun p = new PrimeRun(143);
 new Thread(p).start();

(Fonte: java.lang.Thread javadoc.)

A abordagem de classe de encadeamento personalizada funciona, mas tem alguns problemas:

  1. É estranho usar PrimeThread em um contexto que usa um pool de threads clássico, um executor ou o framework ForkJoin. (Não é impossível, porque PrimeThread implementa indiretamente Runnable, mas usar uma classe Thread personalizada como Runnable é certamente desajeitado e pode não ser viável … dependendo de outros aspectos da classe.)

  2. Há mais oportunidades para erros em outros métodos. Por exemplo, se você declarasse um PrimeThread.start() sem delegar para Thread.start(), você terminaria com um “thread” executado no thread atual.

A abordagem de colocar a lógica do thread em um Runnable evita esses problemas. De fato, se você usar uma classe anônima (Java 1.1 em diante) para implementar o Runnable o resultado será mais sucinto e mais legível do que os exemplos acima.

 final long minPrime = ...
 new Thread(new Runnable() {
     public void run() {
         // compute primes larger than minPrime
          . . .
     }
 }.start();

Com uma expressão lambda (Java 8 em diante), o exemplo acima ficaria ainda mais elegante:

 final long minPrime = ...
 new Thread(() -> {
    // compute primes larger than minPrime
     . . .
 }).start();

Armadilha - Muitos threads tornam um aplicativo mais lento. #

Muitas pessoas que são novas em multi-threading pensam que usar threads automaticamente torna um aplicativo mais rápido. Na verdade, é muito mais complicado do que isso. Mas uma coisa que podemos afirmar com certeza é que para qualquer computador existe um limite no número de threads que podem ser executados ao mesmo tempo:

  • Um computador tem um número fixo de núcleos (ou hiperthreads).
  • Um thread Java deve ser agendado para um núcleo ou hyperthread para ser executado.
  • Se houver mais threads Java executáveis ​​do que núcleos / hyperthreads (disponíveis), alguns deles devem esperar.

Isso nos diz que simplesmente criar mais e mais threads Java não pode tornar o aplicativo cada vez mais rápido. Mas há outras considerações também:

  • Cada encadeamento requer uma região de memória fora do heap para sua pilha de encadeamentos. O tamanho típico da pilha de encadeamentos (padrão) é 512 Kbytes ou 1 Mbytes. Se você tiver um número significativo de threads, o uso de memória pode ser significativo.

  • Cada thread ativo fará referência a um número de objetos no heap. Isso aumenta o conjunto de trabalho de objetos alcançáveis, o que afeta a coleta de lixo e o uso da memória física.

  • As despesas gerais de alternar entre threads não são triviais. Normalmente envolve uma mudança para o espaço do kernel do sistema operacional para tomar uma decisão de agendamento de encadeamento.

  • As sobrecargas de sincronização de threads e sinalização entre threads (por exemplo, wait(), notify() / notifyAll) podem ser significativas.

Dependendo dos detalhes de sua aplicação, esses fatores geralmente significam que há um “ponto ideal” para o número de threads. Além disso, adicionar mais threads oferece uma melhoria mínima no desempenho e pode piorar o desempenho.

Se seu aplicativo for criado para cada nova tarefa, um aumento inesperado na carga de trabalho (por exemplo, uma alta taxa de solicitação) pode levar a um comportamento catastrófico.

Uma maneira melhor de lidar com isso é usar o pool de threads limitado cujo tamanho você pode controlar (estaticamente ou dinamicamente). Quando há muito trabalho a fazer, o aplicativo precisa enfileirar as solicitações. Se você usar um ExecutorService, ele cuidará do gerenciamento do pool de threads e do enfileiramento de tarefas.

Armadilha: Variáveis ​​compartilhadas requerem sincronização adequada #

Considere este exemplo:

public class ThreadTest implements Runnable {
   
    private boolean stop = false;
    
    public void run() {
        long counter = 0;
        while (!stop) {
            counter = counter + 1;
        }
        System.out.println("Counted " + counter);
    }

    public static void main(String[] args) {
        ThreadTest tt = new ThreadTest();
        new Thread(tt).start();    // Create and start child thread
        Thread.sleep(1000);
        tt.stop = true;            // Tell child thread to stop.
    }
}

A intenção deste programa é iniciar um thread, deixá-lo rodar por 1000 milissegundos, e então fazer com que ele pare configurando o sinalizador stop.

Vai funcionar como pretendido? #

Talvez sim, talvez não.

Um aplicativo não necessariamente para quando o método main retorna. Se outro encadeamento foi criado e esse encadeamento não foi marcado como um encadeamento daemon, o aplicativo continuará a ser executado após o término do encadeamento principal. Neste exemplo, isso significa que o aplicativo continuará em execução até que o thread filho termine. Isso deve acontecer quando tt.stop for definido como true.

Mas isso na verdade não é estritamente verdade. Na verdade, a thread filha irá parar depois de observar stop com o valor true. Isso vai acontecer? Talvez sim, talvez não.

A Java Language Specification garante que as leituras e gravações da memória feitas em um thread são visíveis para esse thread, conforme a ordem das instruções no código-fonte. No entanto, em geral, isso NÃO é garantido quando um thread grava e outro thread (posteriormente) lê. Para obter visibilidade garantida, é preciso haver uma cadeia de relações acontece antes entre uma gravação e uma leitura subsequente. No exemplo acima, não existe tal cadeia para a atualização do sinalizador stop e, portanto, não é garantido que o thread filho verá stop mudar para true.

(Observação aos autores: Deve haver um tópico separado sobre o modelo de memória Java para entrar em detalhes técnicos profundos.)

Como corrigimos o problema? #

Nesse caso, existem duas maneiras simples de garantir que a atualização stop seja visível:

  1. Declare stop como volátil; ou seja

       private volatile boolean stop = false;

    For a volatile variable, the JLS specifies that there is a happens-before relation between a write by one thread and a later read by a second thread.

  2. Use um mutex para sincronizar da seguinte forma:

    public class ThreadTest implements Runnable {

     private boolean stop = false;

 public void run() {
     long counter = 0;
     while (true) {
         synchronize (this) {
             if (stop) {
                 break;
             }
         }
         counter = counter + 1;
     }
     System.out.println("Counted " + counter);
 }

 public static void main(String[] args) {
     ThreadTest tt = new ThreadTest();
     new Thread(tt).start();    // Create and start child thread
     Thread.sleep(1000);
     synchronize (tt) {
         tt.stop = true;        // Tell child thread to stop.
     }
 }

    }

Além de garantir que haja exclusão mútua, o JLS especifica que há uma relação acontece antes entre a liberação de um mutex em um encadeamento e a obtenção do mesmo mutex em um segundo encadeamento.

Mas a atribuição não é atômica? #

É sim!

No entanto, esse fato não significa que os efeitos da atualização serão visíveis simultaneamente para todas as threads. Apenas uma cadeia adequada de relações acontece-antes garantirá isso.

Por que eles fizeram isso? #

Os programadores que fazem programação multithread em Java pela primeira vez acham que o Modelo de Memória é um desafio. Os programas se comportam de maneira não intuitiva porque a expectativa natural é que as gravações sejam visíveis uniformemente. Então, por que os designers de Java projetam o Modelo de Memória dessa maneira.

Na verdade, tudo se resume a um compromisso entre desempenho e facilidade de uso (para o programador).

Uma arquitetura de computador moderna consiste em vários processadores (núcleos) com conjuntos de registros individuais. A memória principal é acessível a todos os processadores ou a grupos de processadores. Outra propriedade do hardware de computador moderno é que o acesso aos registradores é normalmente muito mais rápido do que o acesso à memória principal. À medida que o número de núcleos aumenta, é fácil ver que a leitura e a gravação na memória principal podem se tornar o principal gargalo de desempenho de um sistema.

Essa incompatibilidade é resolvida com a implementação de um ou mais níveis de cache de memória entre os núcleos do processador e a memória principal. Cada núcleo acessa as células de memória por meio de seu cache. Normalmente, uma leitura na memória principal só acontece quando há uma falta de cache, e uma gravação na memória principal só acontece quando uma linha de cache precisa ser liberada. Para um aplicativo em que o conjunto de locais de memória de cada núcleo caberá em seu cache, a velocidade do núcleo não será mais limitada pela velocidade/largura de banda da memória principal.

Mas isso nos dá um novo problema quando vários núcleos estão lendo e gravando variáveis ​​compartilhadas. A versão mais recente de uma variável pode ficar no cache de um núcleo. A menos que esse núcleo libere a linha de cache para a memória principal E outros núcleos invalidem sua cópia em cache de versões mais antigas, alguns deles podem ver versões antigas da variável. Mas se os caches fossem liberados para a memória toda vez que houvesse uma gravação de cache (“apenas no caso” de haver uma leitura por outro núcleo) que consumiria a largura de banda da memória principal desnecessariamente.

A solução padrão usada no nível do conjunto de instruções de hardware é fornecer instruções para invalidação de cache e escrita de cache, e deixar que o compilador decida quando usá-las.

Voltando ao Java. o Modelo de Memória é projetado para que os compiladores Java não sejam obrigados a emitir invalidação de cache e instruções de gravação onde elas não são realmente necessárias. A suposição é que o programador usará um mecanismo de sincronização apropriado (por exemplo, mutexes primitivos, voláteis, classes de simultaneidade de alto nível e assim por diante) para indicar que ele precisa de visibilidade de memória. Na ausência de uma relação acontece antes, os compiladores Java são livres para assumir que nenhuma operação de cache (ou similar) é necessária.

Isso tem vantagens de desempenho significativas para aplicativos multiencadeados, mas a desvantagem é que escrever aplicativos multiencadeados corretos não é uma questão simples. O programador tem que entender o que está fazendo.

Por que não consigo reproduzir isso? #

Há uma série de razões pelas quais problemas como este são difíceis de reproduzir:

  1. Como explicado acima, a consequência de não lidar adequadamente com problemas de visibilidade de memória é normalmente que seu aplicativo compilado não trata os caches de memória corretamente. No entanto, como mencionamos acima, os caches de memória geralmente são liberados de qualquer maneira.

  2. Quando você altera a plataforma de hardware, as características dos caches de memória podem ser alteradas. Isso pode levar a um comportamento diferente se seu aplicativo não sincronizar corretamente.

  3. Você pode estar observando os efeitos da sincronização por acaso. Por exemplo, se você adicionar traceprints, normalmente ocorre alguma sincronização nos bastidores nos fluxos de E/S que causa liberações de cache. Portanto, adicionar traceprints frequentemente faz com que o aplicativo se comporte de maneira diferente.

  4. A execução de um aplicativo em um depurador faz com que ele seja compilado de forma diferente pelo compilador JIT. Pontos de interrupção e passos únicos exacerbam isso. Esses efeitos geralmente alteram a maneira como um aplicativo se comporta.

Essas coisas tornam os bugs devidos à sincronização inadequada particularmente difíceis de resolver.
