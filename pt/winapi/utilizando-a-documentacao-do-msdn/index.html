A API do Windows é vasta e contém muitos recursos. O tamanho da API é tal que ninguém pode saber tudo. Embora existam muitos recursos como o StackOverflow, não há substituto para a documentação oficial.

Exemplos de Documentação:

  • Visão geral do tópico: Considerações e práticas recomendadas sobre o desempenho do Gerenciador de janelas da área de trabalho
  • Amostras: Personalizar uma miniatura icônica e um bitmap de visualização ao vivo
  • Funções: função DwmSetIconicThumbnail

Tipos de Documentação Disponíveis #

A biblioteca MSDN contém vários tipos diferentes de documentação que podem ser usados ​​para implementar recursos.

  • Visão geral dos tópicos Essas são visões gerais amplas dos tópicos destinados a fornecer uma compreensão geral de uma API. Essas visões gerais também descrevem as melhores práticas e estratégias de implementação.
  • Amostras Demonstrar o uso de APIs específicas. Eles geralmente são altamente simplificados, não necessariamente fazem verificação de erros e normalmente não usam estruturas como MFC ou ATL. Eles fornecem um ponto de partida para o uso de recursos.
  • Referência Detalha todos os elementos de cada API. Isso inclui constantes/enumerações, interfaces, funções e classes.

Observação: muitos funcionários da Microsoft também mantêm blogs, como o [The Old New Thing] de Raymond Chen (https://blogs.msdn.microsoft.com/oldnewthing/) que pode complementar a documentação, mas esses blogs não substituem a documentação .

Encontrando documentação para um recurso #

Encontrar a documentação de um recurso geralmente é tão simples quanto uma pesquisa usando um bom mecanismo de pesquisa. Se isso falhar, ou se não tiver certeza sobre termos específicos, o Índice de API do Windows pode ajudar a localizar recursos específicos. A documentação para métodos, interfaces, enumerações e constantes geralmente pode ser encontrada pesquisando o nome usando um mecanismo de pesquisa. Além disso, o Windows Dev Center pode fornecer um valioso ponto de partida.

Usando a documentação da função #

A documentação de uma função é dividida em várias seções:

Visão geral #

Descreve para que a função é usada. Esta seção também mostrará informações sobre se a função está depreciada ou pode não estar disponível em versões futuras.

Sintaxe #

Mostra a declaração da função do cabeçalho de origem apropriado. É uma referência rápida à assinatura da função.

Parâmetros #

Explica cada um dos parâmetros, se o parâmetro é de entrada ou saída, e outras considerações importantes.

Valor de retorno #

Esta seção explica o resultado da chamada de função, incluindo como detectar erros e quais informações adicionais estão disponíveis. (Por exemplo, esta seção indicará explicitamente se GetLastError fornecerá informações adicionais de tratamento de erros.)

Observações #

Abrange qualquer informação adicional necessária para usar a função, como informações sobre funções de suporte, obtenção de identificadores apropriados e descarte de recursos.

Exemplos #

Se esta seção estiver disponível, ela contém um exemplo do uso apropriado da função para usar como ponto de partida para implementação.

Requisitos #

Fornece informações importantes sobre os pré-requisitos para chamar a função. Essas informações incluem:

  • Mínimo Cliente/Servidor Suportado Primeira versão do sistema operacional (suportado pela Microsoft) para fornecer a função. (Observe que este campo é notoriamente enganoso. Muitas vezes, as funções são suportadas em uma versão anterior do sistema operacional, mas este campo mostra apenas a versão mais antiga que é atualmente suportada pela Microsoft. Por exemplo, o CreateWindow tem suporte desde o Windows 1.0, mas a documentação mostra apenas que tem suporte desde o Windows 2000. A versão online da documentação do MSDN não indica que qualquer função tenha suporte em uma versão do Windows anterior a 2000, embora muitos foram. Para desenvolvimento legado, você precisará consultar uma versão mais antiga da documentação do SDK, como pode ter sido enviada em um CD-ROM do MSDN. Ou, basta olhar nos arquivos de cabeçalho.)
  • Header O cabeçalho do SDK para #include que contém a declaração da função. Se a função não estiver disponível em um cabeçalho, isso mostrará informações sobre o procedimento para chamar a função (geralmente chamando GetProcAddress para fazer a vinculação dinâmica em tempo de execução).
  • Biblioteca O arquivo de biblioteca a ser passado para o vinculador para resolver as funções exportadas.
  • DLL O arquivo (conforme fornecido com o sistema operacional) que contém a função exportada.
  • Fim do Suporte Cliente/Servidor A última versão do Windows a oferecer suporte oficial à API.
  • Nomes Unicode e ANSI Para funções de string que possuem variantes Unicode e ANSI, isso lista os nomes reais exportados para as duas funções. Isso geralmente é apenas o nome da função com um sufixo W ou A (respectivamente).
