Cada thread terá seu próprio último código de erro. A API do Windows definirá o último código de erro no thread de chamada.

Você deve sempre chamar a função GetLastError imediatamente após verificar o valor de retorno de uma função da API do Windows.

A maioria das funções da API do Windows define o último código de erro quando falham. Alguns também definirão o último código de erro quando forem bem-sucedidos. Existem várias funções que não definem o último código de erro. Sempre consulte a documentação da função da API do Windows.

Não é seguro usar FORMAT_MESSAGE_FROM_SYSTEM sem FORMAT_MESSAGE_IGNORE_INSERTS ao usar a função FormatMessage para obter uma descrição de um código de erro.

Convertendo um código de erro em uma string de mensagem #

[GetLastError](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado com informações adicionais sobre falha) retorna um código de erro numérico. Para obter uma mensagem de erro descritiva (*por exemplo, para exibir a um usuário), você pode chamar FormatMessage:

// This functions fills a caller-defined character buffer (pBuffer)
// of max length (cchBufferLength) with the human-readable error message
// for a Win32 error code (dwErrorCode).
// 
// Returns TRUE if successful, or FALSE otherwise.
// If successful, pBuffer is guaranteed to be NUL-terminated.
// On failure, the contents of pBuffer are undefined.
BOOL GetErrorMessage(DWORD dwErrorCode, LPTSTR pBuffer, DWORD cchBufferLength)
{
    if (cchBufferLength == 0)
    {
        return FALSE;
    }

    DWORD cchMsg = FormatMessage(FORMAT_MESSAGE_FROM_SYSTEM | FORMAT_MESSAGE_IGNORE_INSERTS,
                                 NULL,  /* (not used with FORMAT_MESSAGE_FROM_SYSTEM) */
                                 dwErrorCode,
                                 MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT),
                                 pBuffer,
                                 cchBufferLength,
                                 NULL);
    return (cchMsg > 0);
}

Em C++, você pode simplificar consideravelmente a interface usando a classe std::string:

#include <Windows.h>
#include <exception>
#include <stdexcept>
#include <memory>
#include <string>
typedef std::basic_string<TCHAR> String;

String GetErrorMessage(DWORD dwErrorCode)
{
    LPTSTR psz = NULL;
    const DWORD cchMsg = FormatMessage(FORMAT_MESSAGE_FROM_SYSTEM
                                         | FORMAT_MESSAGE_IGNORE_INSERTS
                                         | FORMAT_MESSAGE_ALLOCATE_BUFFER,
                                       NULL, // (not used with FORMAT_MESSAGE_FROM_SYSTEM)
                                       dwErrorCode,
                                       MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT),
                                       reinterpret_cast<LPTSTR>(&psz),
                                       0,
                                       NULL);
    if (cchMsg > 0)
    {
        // Assign buffer to smart pointer with custom deleter so that memory gets released
        // in case String's c'tor throws an exception.
        auto deleter = [](void* p) { ::HeapFree(::GetProcessHeap(), 0, p); };
        std::unique_ptr<TCHAR, decltype(deleter)> ptrBuffer(psz, deleter);
        return String(ptrBuffer.get(), cchMsg);
    }
    else
    {
        throw std::runtime_error("Failed to retrieve error message string.");
    }
}

OBSERVAÇÃO: essas funções também funcionam para [valores HRESULT](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado como valor HRESULT valor). Basta alterar o primeiro parâmetro de DWORD dwErrorCode para HRESULT hResult. O restante do código pode permanecer inalterado.

Introdução #

A API do Windows é fornecida por meio de uma interface C-callable. O sucesso ou falha de uma chamada de API é relatado estritamente por meio de valores de retorno. As exceções não fazem parte do contrato documentado (embora algumas implementações de API possam gerar exceções SEH, por exemplo, ao passar um argumento lpCommandLine somente leitura para CreateProcess).

O relatório de erros se enquadra em uma das quatro categorias:

  • [Somente valor de retorno](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado apenas pelo valor de retorno)
  • [Valor de retorno com informações adicionais sobre falha](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado com informações adicionais sobre falha)
  • [Valor de retorno com informações adicionais sobre falha e sucesso](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado com informações adicionais sobre falha e sucesso -e-sucesso)
  • [valor de retorno HRESULT](https://www.wikiod.com/pt/winapi/relatorio-e-tratamento-de-erros#Erro relatado como valor HRESULTvalue)

A documentação para cada chamada de API explica explicitamente como os erros são relatados. Consulte sempre a documentação.

Erro relatado apenas pelo valor de retorno #

Algumas chamadas de API retornam um único sinalizador de falha/sucesso, sem nenhuma informação adicional (por exemplo, GetObject):

if ( GetObjectW( obj, 0, NULL ) == 0 ) {
    // Failure: no additional information available.
}

Erro relatado com informações adicionais sobre falha #

Além de um valor de retorno de falha/sucesso, algumas chamadas de API também definem o último erro em caso de falha (por exemplo, CreateWindow ). A documentação geralmente contém a seguinte redação padrão para este caso:

Se a função for bem-sucedida, o valor de retorno será <valor de sucesso específico da API>. Se a função falhar, o valor de retorno será <valor de erro específico da API>. Para obter informações de erro estendidas, chame GetLastError.

if ( CreateWindowW( ... ) == NULL ) {
    // Failure: get additional information.
    DWORD dwError = GetLastError();
} else {
    // Success: must not call GetLastError.
}

É vital que você chame GetLastError() IMEDIATAMENTE. O último código de erro pode ser substituído por qualquer outra função, portanto, se houver uma chamada de função extra entre a função que falhou e a chamada para GetLastError() , o retorno de GetLastError() não será mais confiável. Tome cuidado extra ao lidar com construtores C++.

Depois de obter um código de erro, você precisará interpretá-lo. Você pode obter uma lista abrangente de códigos de erro no MSDN, na página [System Error Codes (Windows)](https://msdn.microsoft.com/en-us/library/windows/desktop/ms681381(v=vs. 85).aspx). Alternativamente, você pode procurar nos arquivos de cabeçalho do sistema; o arquivo com todas as constantes do código de erro é winerror.h. (Se você tiver o SDK oficial da Microsoft para Windows 8 ou mais recente, ele está na subpasta shared da pasta de inclusão.)

Notas sobre como chamar GetLastError() em outras linguagens de programação #

linguagens .net (C#, VB, etc.) #

Com .net, você não deve P/Invocar para GetLastError() diretamente. Isso ocorre porque o tempo de execução .net fará outras chamadas de API do Windows no mesmo encadeamento nas suas costas. Por exemplo, o coletor de lixo pode chamar VirtualFree() se encontrar memória suficiente que não está mais usando, e isso pode acontecer entre sua chamada de função pretendida e sua chamada para GetLastError().

Em vez disso, .net fornece a função Marshal.GetLastWin32Error(), que recuperará o último erro da última chamada P/Invoke que você mesmo fez. Use isso em vez de chamar GetLastError() diretamente.

(.net não parece impedi-lo de importar GetLastError() de qualquer maneira; não tenho certeza do porquê.)

Vai #

Os vários recursos fornecidos pelo Go para chamar funções DLL (que residem tanto no pacote syscall quanto no pacote golang.org/x/sys/windows) retornam três valores: r1, r2 e err. r2 nunca é usado; você pode usar o identificador em branco lá. r1 é o valor de retorno da função. err é o resultado de chamar GetLastError(), mas convertido em um tipo que implementa error, então você pode passá-lo para chamar funções para manipular.

Como o Go não sabe quando chamar GetLastError() e quando não, ele sempre retornará um erro não nil. Portanto, o idioma típico de tratamento de erros do Go

r1, _, err := syscall.Syscall12(CreateWindowW.Addr(), ...)
if err != nil {
    // handle err
}
// use r1

não funciona. Em vez disso, você deve verificar r1, exatamente como faria em C, e usar apenas err se that indicar que a função retornou um erro:

r1, _, err := syscall.Syscall12(CreateWindowW.Addr(), ...)
if r1 == 0 {
    // handle err
}
// use r1

Erro relatado com informações adicionais sobre falha e sucesso #

Algumas chamadas de API podem ter êxito ou falhar de mais de uma maneira. As APIs geralmente retornam informações adicionais para invocações bem-sucedidas e erros (por exemplo, CreateMutex).

if ( CreateMutexW( NULL, TRUE, L"Global\\MyNamedMutex" ) == NULL ) {
    // Failure: get additional information.
    DWORD dwError = GetLastError();
} else {
    // Success: Determine which mutex was returned.
    if ( GetLastError() == ERROR_ALREADY_EXISTS ) {
        // Existing mutex object returned.
    } else {
        // Newly created mutex object returned.
    }
}

Erro relatado como valor HRESULT #

HRESULTs são valores numéricos de 32 bits, onde bits ou intervalos de bits codificam informações bem definidas. O MSB é um sinalizador de falha/sucesso, com os bits restantes armazenando informações adicionais. A falha ou o sucesso podem ser determinados usando o FAILED ou [SUCCEEDED](https://msdn.microsoft.com /en-us/library/windows/desktop/ms687197.aspx) macros. HRESULTs são comumente usados ​​com COM, mas também aparecem em implementações não COM (por exemplo, StringCchPrintf) .

const size_t cchBuf = 5;
wchar_t buffer[cchBuf] = { 0 };
HRESULT hr = StringCchPrintfW( buffer, cchBuf, L"%s", L"Hello, world!" );
if ( FAILED( hr ) ) {
    // Failure: Determine specific reason.
    switch ( hr ) {
    case STRSAFE_E_INSUFFICIENT_BUFFER:
        // Buffer too small; increase buffer and retry.
        ...
    case STRSAFE_E_INVALID_PARAMETER:
        // Invalid parameter; implement custom error handling (e.g. logging).
        ...
    default:
        // Some other error code; implement custom error handling (e.g. logging).
        ...
    }
}
