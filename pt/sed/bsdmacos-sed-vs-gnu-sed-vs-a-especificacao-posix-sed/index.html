BSDmacOS Sed vs. GNU Sed vs. a especificação POSIX Sed

  • Substitua todas as novas linhas por guias
  • Anexar texto literal a uma linha com a função ‘a’

Nesta página

  • Substitua todas as novas linhas por guias
  • Anexar texto literal a uma linha com a função ‘a’

Para citar a solicitação de criação de tópicos do @SnoringFrog:

“Uma das maiores pegadinhas do sed são os scripts que falham (ou são bem-sucedidos de maneira inesperada) porque foram escritos para um e não para o outro. Um simples resumo das diferenças mais importantes seria bom.”

macOS usa a versão BSD do sed[1], que difere em muitos aspectos da versão GNU sed que vem com as distribuições Linux.

Seu denominador comum é a funcionalidade decretada pelo POSIX: veja a especificação POSIX sed.

A abordagem mais portátil é usar apenas recursos POSIX, o que, no entanto, limita a funcionalidade:

  • Notavelmente, POSIX especifica suporte apenas para expressões regulares basic, que têm muitas limitações (por exemplo, sem suporte para | (alternância), sem suporte direto para + e ?) e diferentes escapes requisitos.

  • Advertência: GNU sed (sem -r), não suporta \|, \+ e \?, que NÃO é compatível com POSIX; use --posix para desabilitar (veja abaixo).

  • Para usar apenas recursos POSIX:

  • (ambas as versões): use only as opções -n e -e (notavelmente, não use -E ou -r para ativar o suporte para expressões regulares extended)

  • GNU sed: adicione a opção --posix para garantir a funcionalidade somente POSIX (você não precisa estritamente disso, mas sem ela você pode acabar usando inadvertidamente recursos não POSIX sem perceber; caveat: --posix ele é não compatível com POSIX)

  • Usar recursos somente POSIX significa requisitos de formatação mais rígidos (renunciar a muitas conveniências disponíveis no GNU sed):

    • Control-character sequences such as \n and \t are generally NOT supported.
    • Labels and branching commands (e.g., b) must be followed by an actual newline or continuation via a separate -e option.
    • See below for details.

No entanto, both versões implementam extensions para o padrão POSIX:

  • what extensões que eles implementam diferem (GNU sed implementa mais).
  • mesmo aquelas extensões que eles ambos implementam parcialmente diferem na sintaxe.

Se você precisar dar suporte a AMBAS as plataformas (discussão das diferenças):

  • Recursos Incompatíveis:

  • O uso da opção -i sem argumento (atualização in-loco sem backup) é incompatível:

    • BSD sed: MUST use -i ''
    • GNU sed: MUST use just -i (equivalent: -i'') - using -i '' does NOT work.

  • -i sensatamente ativa a numeração de linha per-input-file nas versões GNU sed e recent do BSD sed (por exemplo, no FreeBSD 10), mas **NÃO no macOS a partir de 10.12 **. Observe que, na ausência de -i, todas as versões numeram linhas cumulativamente nos arquivos de entrada.

  • Se a last linha de entrada não tiver uma nova linha à direita (e for impressa):

    • BSD sed: always appends a newline on output, even if the input line doesn’t end in one.
    • GNU sed: preserves the trailing-newline status, i.e., it appends a newline only if the input line ended in one.

  • **Características comuns:

  • Se você restringir seus scripts sed ao que o BSD sed suporta, eles geralmente funcionarão no GNU sed também - com a notável exceção de usar recursos regex * estendidos * específicos da plataforma com -E. Obviamente, você também renunciará às extensões específicas da versão GNU. Veja a próxima seção.

Diretrizes para suporte multiplataforma (OS X/BSD, Linux), orientadas pelos requisitos mais rígidos da versão BSD:

Observe que as abreviações macOS e Linux são ocasionalmente usadas abaixo para se referir às versões BSD e GNU do sed, respectivamente, porque são as versões stock em cada plataforma. No entanto, é possível instalar GNU sed no macOS, por exemplo, usando Homebrew com brew install gnu-sed.

Observação: Exceto quando os sinalizadores -r e -E são usados (extended regexes), as instruções abaixo equivalem a escrever scripts sed compatíveis com POSIX.

  • Para conformidade com POSIX, você deve se restringir a POSIX BREs (básicas expressões regulares), que são, infelizmente, como o nome sugere, bastante básico. Aviso: não assuma que \|, \+ e \? são suportados: Enquanto GNU sed os suporta (a menos que --posix seja usado), BSD sed suporta não - esses recursos são não compatíveis com POSIX. Enquanto \+ e \? podem ser emulados de forma compatível com POSIX: \{1,\} para \+, \{0,1\} para \?, \| (alternância) cannot, infelizmente.

  • Para expressões regulares mais poderosas, use -E (ao invés de -r) para suportar EREs (extended expressões regulares) (GNU sed não documenta -E, mas funciona lá como um alias de -r; nova versão do BSD sed, como no FreeBSD 10, agora também suporta -r, mas a versão do macOS a partir de 10.12 não). Aviso: Mesmo que o uso de -r / -E signifique que seu comando é, por definição, não compatível com POSIX, você ainda deve restringir-se a [POSIX EREs (extended regulares expression)](http ://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap09.html#tag_09_04). Infelizmente, isso significa que você não poderá usar várias construções úteis, principalmente:

  • asserções de limite de palavra, porque são específicas da plataforma (por exemplo, \< no Linux, [[:<]] no OS X).

  • back-references dentro de expressões regulares (ao contrário das “back-references” para capturar correspondências de grupo na string de substituição de chamadas de função s), porque o BSD sed não as suporta em regexes extended (mas, curiosamente, faz isso em basic, onde eles são mandatados para POSIX).

  • Sequências de escape de caracteres de controle, como \n e \t:

  • Em regexes (tanto em padrões para seleção de linha quanto no primeiro argumento para a função s), suponha que apenas \n seja reconhecido como uma sequência de escape (raramente usado, pois o espaço de padrão geralmente é um single line (sem terminar \n), mas não dentro de uma character class, de modo que, por exemplo, [^\n] não funcione; (se sua entrada não contiver caracteres de controle. exceto \t , você pode emular [^\n] com [[:print:][:blank:]]; caso contrário, emendar caracteres de controle. em como literals[2]) - ** geralmente, incluem caracteres de controle como literais, seja por meio de strings entre aspas ANSI C (por exemplo, $'\t') em shells que o suportam (bash, ksh, zsh), ou por meio de substituições de comando usando printf (por exemplo, "$(printf '\t')")**.

    • Linux only:
      sed 's/\t/-/' <<<$'a\tb' # -> 'a-b'
    • OSX and Linux:
      sed 's/'$'\t''/-/' <<<$'a\tb' # ANSI C-quoted string
      sed 's/'"$(printf '\t')"'/-/' <<<$'a\tb' # command subst. with printf

  • Em strings de substituição usadas com o comando s, suponha que NENHUMA seqüência de escape de caractere de controle seja suportada, então, novamente, inclua caracteres de controle. como literais, como acima.

    • Linux only:
      sed 's/-/\t/' <<<$'a-b' # -> 'a<tab>b'
    • macOS and Linux:
      sed 's/-/'$'\t''/' <<<'a-b' sed 's/-/'"$(printf '\t')"'/' <<<'a-b'

  • Idem para os argumentos de texto para as funções i e a: não use seqüências de caracteres de controle - veja abaixo.

  • Rótulos e ramificações: rótulos, bem como o nome do rótulo argument para as funções b e t devem ser seguidos por uma nova linha literal ou um $'\n' emendado . Como alternativa, use várias opções -e e encerre cada uma logo após o nome do rótulo. *Somente Linux: sed -n '/a/ bLBL; d; :LBL p' <<<$'a\nb' # -> 'a'

  • macOS e Linux:

    • EITHER (actual newlines):
      sed -n '/a/ bLBL d; :LBL p' <<<$'a\nb'
    • OR (spliced-in $\n instances):
      sed -n '/a/ bLBL'$'\n''d; :LBL'$'\n''p' <<<$'a\nb'
    • OR (multiple -e options):
      sed -n -e '/a/ bLBL' -e 'd; :LBL' -e 'p' <<<$'a\nb'

  • Funções i e a para inserir/anexar texto: siga o nome da função por \, seguido por uma nova linha literal ou por um $'\n' emendado antes de especificar o argumento de texto.

*Somente Linux: sed '1 i new first line' <<<$'a\nb' # -> 'new first line<nl>a<nl>b'

  • OSX e Linux: sed -e '1 i\'$'\n''new first line' <<<$'a\nb'

  • Observação:

    • Without -e, the text argument is inexplicably not newline-terminated on output on macOS (bug?).
    • Do not use control-character escapes such as \n and \t in the text argument, as they’re only supported on Linux.
    • If the text argument therefore has actual interior newlines, \-escape them.
    • If you want to place additional commands after the text argument, you must terminate it with an (unescaped) newline (whether literal or spliced in), or continue with a separate -e option (this is a general requirement that applies to all versions).

  • Dentro da função lists (várias chamadas de função entre {...}), certifique-se de também encerrar a função last, antes do fechamento }, com ;. *Somente Linux:

    • sed -n '1 {p;q}' <<<$'a\nb' # -> 'a'

  • macOS e Linux:

    • sed -n '1 {p;q;}' <<<$'a\nb'

Recursos específicos do GNU sed ausentes do BSD sed:

Recursos GNU que você perderá se precisar oferecer suporte a ambas as plataformas:

  • Várias opções de correspondência e substituição de regex (tanto em padrões para seleção de linha quanto no primeiro argumento para a função s):

  • A opção I para correspondência regex sensível a maiúsculas e minúsculas (incrivelmente, BSD sed não oferece suporte a isso).

  • A opção M para correspondência de várias linhas (onde ^ / $ corresponde ao início / fim de each line)

  • Para opções adicionais específicas da função s, consulte https://www.gnu.org/software/sed/manual/sed.html#The-_0022s_0022-Command

  • Sequências de escape

  • Seqüências de escape relacionadas à substituição como \u no argumento de substituição da função s/// que permitem manipulação de substring, dentro dos limites; por exemplo, sed 's/^./\u&/' <<<'cão' # -> 'Cão' - veja http://www.gnu.org/software/sed/manual/sed.html#The -_0022s_0022-Comando

  • Seqüências de escape de caracteres de controle: além de \n, \t, …, escapes baseados em codepoints; por exemplo, todos os seguintes escapes (hex., octal, decimal) representam aspas simples ('): \x27, \o047, \d039 - veja https://www.gnu. org/software/sed/manual/sed.html#Escapes

  • Extensões de endereço, como first~step para corresponder a cada passo-ésima linha, addr, +N para corresponder N linhas após addr, … - consulte http://www.gnu .org/software/sed/manual/sed.html#Addresses

[1] A versão sed do macOS é mais antiga que a versão em outros sistemas do tipo BSD, como FreeBSD e PC-BSD. Infelizmente, isso significa que você não pode assumir que os recursos que funcionam no FreeBSD, por exemplo, funcionarão [o mesmo] no macOS.

[2] A string entre aspas ANSI C $'\001\002\003\004\005\006\007\010\011\013\014\015\016\017\020\021\022\ 023\024\025\026\027\030\031\032\033\034\035\036\037\177' contém todos os caracteres de controle ASCII, exceto \n (e NUL), para que você possa usá-lo em combinação com [:print:] para uma emulação bastante robusta de [^\n]: '[[:print:]'$'\001\002\003\004\005\006\007\010\011\013\014\015\016\017\020\021\022\023\024\ 025\026\027\030\031\032\033\034\035\036\037\177'']

Substitua todas as novas linhas por guias #

Observação: Por questões de brevidade, os comandos usam here-strings (<<<) e ANSI C-quoted strings ( $'...'). Ambos os recursos do shell funcionam em bash, ksh e zsh.

# GNU Sed
$ sed ':a;$!{N;ba}; s/\n/\t/g' <<<$'line_1\nline_2\nline_3'
line_1 line_2 line_3

# BSD Sed equivalent (multi-line form)
sed <<<$'line_1\nline_2\nline_3' '
:a
$!{N;ba
}; s/\n/'$'\t''/g'

# BSD Sed equivalent (single-line form, via separate -e options)
sed -e ':a' -e '$!{N;ba' -e '}; s/\n/'$'\t''/g' <<<$'line 1\nline 2\nline 3'

Notas do BSD Sed:

  • Observe a necessidade de encerrar rótulos (:a) e comandos de ramificação (ba) com novas linhas reais ou com opções -e separadas.

  • Como sequências de escape de caracteres de controle como \t não são suportadas na string de substituição, uma tabulação literal com aspas em ANSI C é emendada na string de substituição. (Na parte regex, BSD Sed only reconhece \n como uma seqüência de escape).

Anexar texto literal a uma linha com a função ‘a’ #

Observação: Por questões de brevidade, os comandos usam here-strings (<<<) e ANSI C-quoted strings ( $'...'). Ambos os recursos do shell funcionam em bash, ksh e zsh.

 # GNU Sed
 $ sed '1 a appended text' <<<'line 1'
 line 1
 appended text

 # BSD Sed (multi-line form)
 sed '1 a\
 appended text' <<<'line 1'

 # BSD Sed (single-line form via a Bash/Ksh/Zsh ANSI C-quoted string)
 sed $'1 a\\\nappended text' <<<'line 1'

Observe como o BSD Seed requer um \ seguido por uma nova linha real para passar o texto a ser anexado. O mesmo se aplica às funções relacionadas i (inserir) e c (excluir e inserir).
