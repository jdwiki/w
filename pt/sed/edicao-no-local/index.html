Sintaxe #

  • sed -I extension - FreeBSD sed (contador de linha contínuo)
  • sed -I[extension] - NetBSD e Illumos sed (contador de linha contínuo)
  • sed -i extension - FreeBSD sed
  • sed -i[extension] - NetBSD, OpenBSD, Illumos, BusyBox e GNU sed
  • sed –in-place[=extension] - Illumos, BusyBox e GNU sed

Parâmetros #

Parâmetro Detalhes
extensão Salve um arquivo de backup com a extensão especificada ou nenhum arquivo de backup quando extension for uma string de comprimento zero.

A edição no local é uma extensão comum, mas não padrão, presente na maioria dos sistemas recentes.

De um [manual do BSD sed] [obsed]

(uma seção como esta aparece em todos os manuais BSD sed atuais, e aqueles de seus derivados)

Não é recomendado dar uma extensão de comprimento zero ao editar arquivos no local, pois corre o risco de corrupção ou conteúdo parcial em situações em que o espaço em disco é exausto, etc.

Não se esqueça do poderoso ed #

Definitivamente, há um uso para sed e para recursos de edição no local do sed, mas quando o padrão UNIX é estendido, devemos sempre perguntar por que o antigo padrão UNIX não inclui esse recurso. Embora o UNIX não seja perfeito, a ortogonalidade e a completude das ferramentas foram desenvolvidas para estarem próximas da perfeição, pelo menos para propósitos que eram visíveis por volta de 1970: A edição de texto e a edição automatizada de texto certamente eram visíveis nessa época.

Na verdade, a ideia do sed não é editar um arquivo no local, mas sim editar um stream. É por isso que o nome sed é uma forma abreviada de stream editor. Retire o s e você terá a ferramenta que foi realmente projetada para edição de arquivos: ed:

printf 'g/what to replace/s//with what to replace/g\nw\nq\n' | ed file

ou cat file_edit_commands | arquivo ed.

Uso portátil #

A edição no local, embora comum, não é um recurso padrão. Uma alternativa viável seria usar um arquivo intermediário para armazenar o original ou a saída.

sed 'sed commands' > file.out && mv file.out file
# or
mv file file.orig && sed 'sed commands' file.orig > file

Para usar a opção -i com a sintaxe GNU e FreeBSD, uma extensão deve ser especificada e anexada à opção -i. O seguinte será aceito por ambos e produzirá dois arquivos, a versão original em file.orig e a versão editada em file:

sed -i.orig 'sed commands' file

Veja um exemplo básico dado um arquivo file:

$ cat file
one
two
three
$ sed -i.orig 's/one/XX/' file
$ cat file                       # the original file has changed its content
XX
two
three
$ cat file.orig                  # the original content is now in file.orig
one
two
three

Um exemplo mais complexo, substituindo cada linha pelo número da linha:

$ printf 'one\ntwo\n' | tee file1 | tr a-z A-Z > file2
$ sed -ni.orig = file1 file2
$ cat file1.orig file2.orig
one
two
ONE
TWO
$ cat file1 file2
1
2
1
2

Por que um arquivo de backup é necessário #

Para usar a edição no local sem um arquivo de backup, -i deve receber um argumento de comprimento zero e o FreeBSD sed requer um argumento para -i, anexado ou separado, enquanto a extensão de argumento opcional GNU requer que o argumento seja anexado a -i. Ambos suportam anexar o argumento a -i, mas sem que seja necessário -i'' command é indistinguível de -i extension, e assim um argumento de comprimento zero não pode ser anexado a -i.

Substituindo strings em um arquivo no local #

sed -i s/"what to replace"/"with what to replace"/g $file

Usamos -i para selecionar a edição in-loco no arquivo $file. Em alguns sistemas é necessário adicionar o sufixo após o sinalizador -i que será usado para criar o backup do arquivo original. Você pode adicionar uma string vazia como -i '' para omitir a criação do backup. Veja Observações neste tópico sobre a opção -i.

O terminador g significa fazer uma busca/substituição global em cada linha.

$ cat example 
one
two
three
total
$ sed -i s/"t"/"g"/g example 
$ cat example 
one
gwo
ghree
gogal

A edição no local sem especificar um arquivo de backup substitui as permissões somente leitura #

sed -i -e cmd file modificará file mesmo se suas permissões forem definidas como somente leitura.

Este comando se comporta de forma semelhante ao

sed -e arquivo cmd > tmp; mv -f arquivo tmp

ao invés de

sed -e arquivo cmd > tmp; cat tmp > arquivo; rm tmp

O exemplo a seguir usa gnu sed:

$ echo 'Extremely important data' > input
$ chmod 400 input  # Protect that data by removing write access
$ echo 'data destroyed' > input
-bash: input: Permission denied
$ cat input  
Extremely important data (#phew! Data is intact)
$ sed -i s/important/destroyed/ input
$ cat input
Extremely destroyed data (#see, data changed)

Isso pode ser mitigado criando um backup especificando um SUFFIX com a opção i:

$ sed -i.bak s/important/destroyed/ input
$ cat input
Extremely destroyed data
$ cat input.bak
Extremely important data
