Introdução #

Backpressure é quando em um pipeline de processamento Observable, alguns estágios assíncronos não podem processar os valores com rapidez suficiente e precisam de uma maneira de dizer ao produtor upstream para diminuir a velocidade.

O caso clássico da necessidade de contrapressão é quando o produtor é uma fonte quente:

PublishSubject<Integer> source = PublishSubject.create();

source
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Thread.sleep(10_000);

Neste exemplo, o encadeamento principal produzirá 1 milhão de itens para um consumidor final que o está processando em um encadeamento em segundo plano. É provável que o método compute(int) leve algum tempo, mas a sobrecarga da cadeia de operadores Observable também pode aumentar o tempo que leva para processar itens. No entanto, o thread de produção com o loop for não pode saber disso e continua onNext.

Internamente, os operadores assíncronos têm buffers para manter esses elementos até que possam ser processados. No Rx.NET clássico e no RxJava inicial, esses buffers eram ilimitados, o que significa que eles provavelmente conteriam quase todos os 1 milhão de elementos do exemplo. O problema começa quando há, por exemplo, 1 bilhão de elementos ou a mesma sequência de 1 milhão aparece 1.000 vezes em um programa, levando a OutOfMemoryError e geralmente lentidão devido à sobrecarga excessiva do GC.

Semelhante a como o tratamento de erros tornou-se um cidadão de primeira classe e recebeu operadores para lidar com isso (via operadores onErrorXXX), backpressure é outra propriedade de fluxos de dados que o programador precisa pensar e manipular (via operadores onBackpressureXXX).

Além do PublishSubject acima, existem outros operadores que não suportam backpressure, principalmente devido a razões funcionais. Por exemplo, o operador interval emite valores periodicamente, a contrapressão levaria a um deslocamento no período em relação a um relógio de parede.

No RxJava moderno, a maioria dos operadores assíncronos agora tem um buffer interno limitado, como observeOn acima e qualquer tentativa de estourar esse buffer encerrará toda a sequência com MissingBackpressureException. A documentação de cada operador possui uma descrição sobre seu comportamento de contrapressão.

No entanto, a contrapressão está presente de forma mais sutil em sequências frias regulares (que não geram e não devem gerar MissingBackpressureException). Se o primeiro exemplo for reescrito:

Observable.range(1, 1_000_000)
.observeOn(Schedulers.computation())
.subscribe(v -> compute(v), Throwable::printStackTrace);

Thread.sleep(10_000);

Não há erro e tudo funciona sem problemas com pouco uso de memória. A razão para isso é que muitos operadores de origem podem “gerar” valores sob demanda e, portanto, o operador observeOn pode dizer ao range gerar no máximo tantos valores que o buffer observeOn pode conter de uma só vez sem estouro.

Essa negociação é baseada no conceito da ciência da computação de co-rotinas (eu te chamo, você me chama). O operador range envia um retorno de chamada, na forma de uma implementação da interface Producer, para o observeOn chamando seus subscribers internos setProducer. Em troca, o observeOn chama Producer.request(n) com um valor para informar ao range que ele pode produzir (ou seja, onNext) que muitos elementos adicionais. É então responsabilidade do observeOn chamar o método request no momento certo e com o valor certo para manter os dados fluindo, mas não transbordando.

Expressar a contrapressão nos consumidores finais raramente é necessário (porque eles são síncronos em relação ao seu upstream imediato e a contrapressão ocorre naturalmente devido ao bloqueio da pilha de chamadas), mas pode ser mais fácil entender o funcionamento disso:

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(1);
    }

    public void onNext(Integer v) {
        compute(v);

        request(1);
    }

    @Override
    public void onError(Throwable ex) {
        ex.printStackTrace();
    }

    @Override
    public void onCompleted() {
        System.out.println("Done!");
    }
});

Aqui a implementação onStart indica range para produzir seu primeiro valor, que é então recebido em onNext. Uma vez que o compute(int) termina, o outro valor é então solicitado de range. Em uma implementação ingênua de range, tal chamada chamaria recursivamente onNext, levando a StackOverflowError que é obviamente indesejável.

Para evitar isso, as operadoras usam a chamada lógica de trampolim que impede essas chamadas reentrantes. Nos termos do range, ele lembrará que houve uma chamada request(1) enquanto chamava onNext() e uma vez que onNext() retornar, ele fará outra rodada e chamará onNext() com o próximo valor inteiro. Portanto, se os dois forem trocados, o exemplo ainda funcionará da mesma forma:

@Override
public void onNext(Integer v) {
    request(1);

    compute(v);
}

No entanto, isso não é verdade para onStart. Embora a infraestrutura Observable garanta que será chamada no máximo uma vez em cada Subscriber, a chamada para request(1) pode acionar a emissão de um elemento imediatamente. Se houver uma lógica de inicialização após a chamada para request(1) que é necessária para onNext, você pode acabar com exceções:

Observable.range(1, 1_000_000)
.subscribe(new Subscriber<Integer>() {

    String name;

    @Override
    public void onStart() {
        request(1);

        name = "RangeExample";
    }

    @Override
    public void onNext(Integer v) {
        compute(name.length + v);

        request(1);
    }

    // ... rest is the same
});

Neste caso síncrono, um NullPointerException será lançado imediatamente enquanto ainda estiver executando onStart. Um bug mais sutil acontece se a chamada para request(1) acionar uma chamada assíncrona para onNext em algum outro thread e ler name em corridas onNext escrevendo-o em onStart post request.

Portanto, deve-se fazer a inicialização de todos os campos em onStart ou mesmo antes disso e chamar request() por último. Implementações de request() em operadores garantem uma relação acontece antes (ou em outros termos, liberação de memória ou cerca completa) quando necessário.

Os operadores onBackpressureXXX #

A maioria dos desenvolvedores encontra contrapressão quando seu aplicativo falha com MissingBackpressureException e a exceção geralmente aponta para o operador observeOn. A causa real é geralmente o uso sem pressão de PublishSubject, timer() ou interval() ou operadores personalizados criados via create().

Existem várias maneiras de lidar com tais situações.

Aumentando os tamanhos dos buffers #

Às vezes, esses estouros acontecem devido a fontes em rajadas. De repente, o usuário toca a tela muito rapidamente e o buffer interno padrão de 16 elementos do observeOn no Android estoura.

A maioria dos operadores sensíveis à contrapressão nas versões recentes do RxJava agora permite que os programadores especifiquem o tamanho de seus buffers internos. Os parâmetros relevantes geralmente são chamados de bufferSize, prefetch ou capacityHint. Dado o exemplo de estouro na introdução, podemos apenas aumentar o tamanho do buffer de observeOn para ter espaço suficiente para todos os valores.

PublishSubject<Integer> source = PublishSubject.create();

source.observeOn(Schedulers.computation(), 1024 * 1024)
      .subscribe(e -> { }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Observe, no entanto, que, geralmente, isso pode ser apenas uma correção temporária, pois o estouro ainda pode ocorrer se a fonte produzir demais o tamanho do buffer previsto. Neste caso, pode-se usar um dos seguintes operadores.

Lote/ignorando valores com operadores padrão #

Caso os dados de origem possam ser processados ​​de forma mais eficiente em lote, pode-se reduzir a probabilidade de MissingBackpressureException usando um dos operadores de lote padrão (por tamanho e/ou por tempo).

PublishSubject<Integer> source = PublishSubject.create();

source
      .buffer(1024)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(list -> { 
          list.parallelStream().map(e -> e * e).first();
      }, Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Se alguns dos valores puderem ser ignorados com segurança, pode-se usar a amostragem (com tempo ou outro Observable) e os operadores de limitação (throttleFirst, throttleLast, throttleWithTimeout).

PublishSubject<Integer> source = PublishSubject.create();

source
      .sample(1, TimeUnit.MILLISECONDS)
      .observeOn(Schedulers.computation(), 1024)
      .subscribe(v -> compute(v), Throwable::printStackTrace);

for (int i = 0; i < 1_000_000; i++) {
    source.onNext(i);
}

Observe, no entanto, que esses operadores apenas reduzem a taxa de recepção de valor pelo downstream e, portanto, ainda podem levar a MissingBackpressureException.

onBackpressureBuffer() #

Este operador em sua forma sem parâmetros reintroduz um buffer ilimitado entre a fonte upstream e o operador downstream. Ser ilimitado significa que, desde que a JVM não fique sem memória, ela pode lidar com quase qualquer quantidade proveniente de uma fonte em rajada.

 Observable.range(1, 1_000_000)
           .onBackpressureBuffer()
           .observeOn(Schedulers.computation(), 8)
           .subscribe(e -> { }, Throwable::printStackTrace);

Neste exemplo, o observeOn vem com um tamanho de buffer muito baixo, mas não há MissingBackpressureException, pois o onBackpressureBuffer absorve todos os 1 milhão de valores e entrega pequenos lotes para observeOn.

Observe, no entanto, que o onBackpressureBuffer consome sua fonte de maneira ilimitada, ou seja, sem aplicar nenhuma contrapressão a ela. Isso tem a consequência de que mesmo uma fonte de suporte de contrapressão, como range, será completamente realizada.

Existem 4 sobrecargas adicionais de onBackpressureBuffer

onBackpressureBuffer(int capacity) #

Esta é uma versão limitada que sinaliza BufferOverflowError caso seu buffer atinja a capacidade especificada.

Observable.range(1, 1_000_000)
          .onBackpressureBuffer(16)
          .observeOn(Schedulers.computation())
          .subscribe(e -> { }, Throwable::printStackTrace);

A relevância desse operador está diminuindo à medida que mais e mais operadores agora permitem definir seus tamanhos de buffer. Para o resto, isso dá a oportunidade de “estender seu buffer interno” tendo um número maior com onBackpressureBuffer do que o padrão.

onBackpressureBuffer(int capacity, Action0 onOverflow) #

Essa sobrecarga chama uma ação (compartilhada) caso ocorra um estouro. Sua utilidade é bastante limitada, pois não há outras informações fornecidas sobre o estouro além da pilha de chamadas atual.

onBackpressureBuffer(int capacity, Action0 onOverflow, BackpressureOverflow.Strategy strategy) #

Essa sobrecarga é realmente mais útil, pois permite definir o que fazer caso a capacidade seja atingida. O BackpressureOverflow.Strategy é uma interface na verdade, mas a classe BackpressureOverflow oferece 4 campos estáticos com implementações dele representando ações típicas:

  • ON_OVERFLOW_ERROR: este é o comportamento padrão das duas sobrecargas anteriores, sinalizando uma BufferOverflowException

  • ON_OVERFLOW_DEFAULT: atualmente é o mesmo que ON_OVERFLOW_ERROR

  • ON_OVERFLOW_DROP_LATEST : se ocorrer um overflow, o valor atual será simplesmente ignorado e somente os valores antigos serão entregues uma vez que as solicitações de downstream.

  • ON_OVERFLOW_DROP_OLDEST : descarta o elemento mais antigo no buffer e adiciona o valor atual a ele.

    Observable.range(1, 1_000_000) .onBackpressureBuffer(16, () -> { }, BufferOverflowStrategy.ON_OVERFLOW_DROP_OLDEST) .observeOn(Schedulers.computation()) .subscribe(e -> { }, Throwable::printStackTrace);

Observe que as duas últimas estratégias causam descontinuidade no fluxo à medida que eliminam elementos. Além disso, eles não sinalizarão BufferOverflowException.

onBackpressureDrop() #

Sempre que o downstream não estiver pronto para receber valores, este operador irá retirar esse elemento da sequência. Pode-se pensar nisso como um onBackpressureBuffer de capacidade 0 com a estratégia ON_OVERFLOW_DROP_LATEST.

Este operador é útil quando se pode ignorar com segurança valores de uma fonte (como movimentos do mouse ou sinais de localização GPS atuais), pois haverá valores mais atualizados posteriormente.

 component.mouseMoves()
 .onBackpressureDrop()
 .observeOn(Schedulers.computation(), 1)
 .subscribe(event -> compute(event.x, event.y));

Pode ser útil em conjunto com o operador de origem interval(). Por exemplo, se alguém deseja executar alguma tarefa periódica em segundo plano, mas cada iteração pode durar mais do que o período, é seguro descartar a notificação de intervalo em excesso, pois haverá mais posteriormente:

 Observable.interval(1, TimeUnit.MINUTES)
 .onBackpressureDrop()
 .observeOn(Schedulers.io())
 .doOnNext(e -> networkCall.doStuff())
 .subscribe(v -> { }, Throwable::printStackTrace);

Existe uma sobrecarga deste operador: onBackpressureDrop(Action1<? super T> onDrop) onde a ação (compartilhada) é chamada com o valor sendo descartado. Esta variante permite limpar os próprios valores (por exemplo, liberar recursos associados).

onBackpressureLatest() #

O operador final mantém apenas o valor mais recente e praticamente substitui os valores mais antigos e não entregues. Pode-se pensar nisso como uma variante do onBackpressureBuffer com capacidade de 1 e estratégia de ON_OVERFLOW_DROP_OLDEST.

Ao contrário de onBackpressureDrop, há sempre um valor disponível para consumo se o downstream estiver atrasado. Isso pode ser útil em algumas situações semelhantes à telemetria em que os dados podem vir em algum padrão de rajada, mas apenas o mais recente é interessante para processamento.

Por exemplo, se o usuário clicar muito na tela, ainda queremos reagir à sua última entrada.

component.mouseClicks()
.onBackpressureLatest()
.observeOn(Schedulers.computation())
.subscribe(event -> compute(event.x, event.y), Throwable::printStackTrace);

O uso de onBackpressureDrop neste caso levaria a uma situação em que o último clique é descartado e deixa o usuário se perguntando por que a lógica de negócios não foi executada.

Criando fontes de dados com backpressure #

Criar fontes de dados com backpressure é a tarefa relativamente mais fácil ao lidar com backpressure em geral porque a biblioteca já oferece métodos estáticos em Observable que lidam com backpressure para o desenvolvedor. Podemos distinguir dois tipos de métodos de fábrica: “geradores” frios que retornam e geram elementos com base na demanda downstream e “empurradores” quentes que geralmente conectam fontes de dados não reativas e/ou não pressurizadas e sobrepõem algum tratamento de contrapressão em cima de eles.

apenas #

A fonte consciente de contrapressão mais básica é criada via just:

Observable.just(1).subscribe(new Subscriber<Integer>() {
    @Override
    public void onStart() {
        request(0);
    }

    @Override
    public void onNext(Integer v) {
        System.out.println(v);
    }
   
    // the rest is omitted for brevity
}

Como explicitamente não solicitamos em onStart, isso não imprimirá nada. just é ótimo quando há um valor constante que gostaríamos de iniciar uma sequência.

Infelizmente, just é muitas vezes confundido com uma forma de calcular algo dinamicamente para ser consumido por Subscribers:

int counter;

int computeValue() {
   return ++counter;
}

Observable<Integer> o = Observable.just(computeValue());

o.subscribe(System.out:println);
o.subscribe(System.out:println);

Surpreendente para alguns, isso imprime 1 duas vezes em vez de imprimir 1 e 2, respectivamente. Se a chamada for reescrita, fica óbvio porque funciona assim:

int temp = computeValue();

Observable<Integer> o = Observable.just(temp);

O computeValue é chamado como parte da rotina principal e não em resposta à assinatura dos assinantes.

fromCallable #

O que as pessoas realmente precisam é do método fromCallable:

Observable<Integer> o = Observable.fromCallable(() -> computeValue());

Aqui o computeValue é executado somente quando um assinante se inscreve e para cada um deles, imprimindo o esperado 1 e 2. Naturalmente, fromCallable também suporta contrapressão e não emitirá o valor calculado a menos que solicitado. Observe, no entanto, que a computação acontece de qualquer maneira. Caso a computação em si deva ser atrasada até que o downstream realmente solicite, podemos usar just com map:

Observable.just("This doesn't matter").map(ignored -> computeValue())...

just não emitirá seu valor constante até que seja solicitado quando for mapeado para o resultado do computeValue, ainda chamado para cada assinante individualmente.

a partir de #

Se os dados já estiverem disponíveis como um array de objetos, uma lista de objetos ou qualquer fonte Iterable, as respectivas sobrecargas from tratarão da contrapressão e emissão de tais fontes:

 Observable.from(Arrays.asList(1, 2, 3, 4, 5)).subscribe(System.out::println);

Por conveniência (e evitando avisos sobre a criação de arrays genéricos) existem 2 a 10 sobrecargas de argumentos para just que delegam internamente para from.

O from(Iterable) também oferece uma oportunidade interessante. Muitas gerações de valor podem ser expressas na forma de uma máquina de estado. Cada elemento solicitado aciona uma transição de estado e cálculo do valor retornado.

Escrever tais máquinas de estado como Iterables é um pouco complicado (mas ainda mais fácil do que escrever um Observable para consumi-lo) e ao contrário do C#, Java não tem nenhum suporte do compilador para construir tais máquinas de estado simplesmente escrevendo com aparência clássica código (com yield return e yield break). Algumas bibliotecas oferecem alguma ajuda, como AbstractIterable do Google Guava e Ix.generate() e Ix.forloop() do IxJava. Estes são por si mesmos dignos de uma série completa, então vamos ver uma fonte Iterable muito básica que repete algum valor constante indefinidamente:

Iterable<Integer> iterable = () -> new Iterator<Integer>() {
    @Override
    public boolean hasNext() {
        return true;
    }

    @Override
    public Integer next() {
        return 1;
    }
};

Observable.from(iterable).take(5).subscribe(System.out::println);

Se consumirmos o iterator via loop for clássico, isso resultaria em um loop infinito. Como construímos um Observable a partir dele, podemos expressar nossa vontade de consumir apenas os primeiros 5 dele e depois parar de solicitar qualquer coisa. Este é o verdadeiro poder de avaliar e computar preguiçosamente dentro de Observables.

create(SyncOnSubscribe) #

Às vezes, a fonte de dados a ser convertida no próprio mundo reativo é síncrona (bloqueando) e tipo pull, ou seja, temos que chamar algum método get ou read para obter o próximo dado. Pode-se, é claro, transformar isso em um Iterable, mas quando tais fontes estão associadas a recursos, podemos vazar esses recursos se o downstream cancelar a assinatura da sequência antes que ela termine.

Para lidar com esses casos, o RxJava possui a classe SyncOnSubscribe. Pode-se estendê-lo e implementar seus métodos ou usar um de seus métodos de fábrica baseados em lambda para construir uma instância.

SyncOnSubscribe<Integer, InputStream> binaryReader = SyncOnSubscribe.createStateful(
     () -> new FileInputStream("data.bin"),
     (inputstream, output) -> {
         try {
             int byte = inputstream.read();
             if (byte < 0) {
                 output.onCompleted();
             } else {
                 output.onNext(byte);
             }
         } catch (IOException ex) {
             output.onError(ex);
         }
         return inputstream;
     },
     inputstream -> {
         try {
             inputstream.close();
         } catch (IOException ex) {
             RxJavaHooks.onError(ex);
         }
     } 
 );

 Observable<Integer> o = Observable.create(binaryReader);

Geralmente, SyncOnSubscribe usa 3 retornos de chamada.

Os primeiros retornos de chamada permitem criar um estado por assinante, como o FileInputStream no exemplo; o arquivo será aberto independentemente para cada assinante individual.

O segundo retorno de chamada pega esse objeto de estado e fornece uma saída Observer cujos métodos onXXX podem ser chamados para emitir valores. Esse retorno de chamada é executado quantas vezes o downstream solicitado. A cada chamada, ele deve chamar onNext no máximo uma vez opcionalmente seguido por onError ou onCompleted. No exemplo chamamos onCompleted() se o byte lido for negativo, indicando o fim do arquivo, e chamamos onError caso a leitura gere uma IOException.

O retorno de chamada final é invocado quando o downstream cancela a assinatura (fechando o fluxo de entrada) ou quando o retorno de chamada anterior chama os métodos do terminal; permite liberar recursos. Como nem todas as fontes precisam de todos esses recursos, os métodos estáticos do SyncOnSubscribe permitem criar instâncias sem eles.

Infelizmente, muitas chamadas de método na JVM e outras bibliotecas lançam exceções verificadas e precisam ser agrupadas em try-catches, pois as interfaces funcionais usadas por esta classe não permitem lançar exceções verificadas.

Claro, podemos imitar outras fontes típicas, como um intervalo ilimitado com ele:

SyncOnSubscribe.createStateful(
     () -> 0,
     (current, output) -> {
         output.onNext(current);
         return current + 1;
     },
     e -> { }
);

Nesta configuração, o current começa com 0 e na próxima vez que o lambda for invocado, o parâmetro current agora contém 1.

Existe uma variante de SyncOnSubscribe chamada AsyncOnSubscribe que se parece bastante com a exceção de que o callback do meio também recebe um valor longo que representa o valor da solicitação do downstream e o callback deve gerar um Observable com exatamente o mesmo comprimento. Essa fonte então concatena todos esses Observáveis em uma única sequência.

 AsyncOnSubscribe.createStateful(
     () -> 0,
     (state, requested, output) -> {
         output.onNext(Observable.range(state, (int)requested));
         return state + 1;
     },
     e -> { }
 );

Há uma discussão em andamento (acalorada) sobre a utilidade dessa classe e geralmente não é recomendada porque rotineiramente quebra as expectativas sobre como ela realmente emitirá esses valores gerados e como ela responderá, ou mesmo que tipo de valores de solicitação ela receberá em cenários de consumo mais complexos.

cria(emissor) #

Às vezes, a fonte a ser encapsulada em um Observable já está quente (como movimentos do mouse) ou fria, mas não pode ser pressionada de volta em sua API (como um retorno de chamada de rede assíncrono).

Para lidar com esses casos, uma versão recente do RxJava introduziu o método de fábrica create(emitter). São necessários dois parâmetros:

  • um retorno de chamada que será chamado com uma instância da interface Emitter<T> para cada assinante de entrada,
  • uma enumeração Emitter.BackpressureMode que obriga o desenvolvedor a especificar o comportamento de contrapressão a ser aplicado. Ele tem os modos usuais, semelhantes ao onBackpressureXXX, além de sinalizar um MissingBackpressureException ou simplesmente ignorar esse estouro dentro dele.

Observe que atualmente não suporta parâmetros adicionais para esses modos de contrapressão. Se alguém precisar dessas customizações, usar NONE como o modo de contrapressão e aplicar o onBackpressureXXX relevante no Observable resultante é o caminho a seguir.

O primeiro caso típico para seu uso quando se deseja interagir com uma fonte baseada em push, como eventos de GUI. Essas APIs apresentam alguma forma de chamadas addListener/removeListener que podem ser utilizadas:

Observable.create(emitter -> {
    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    emitter.setCancellation(() -> 
        button.removeListener(al));

}, BackpressureMode.BUFFER);

O Emissor é relativamente simples de usar; pode-se chamar onNext, onError e onCompleted nele e o operador lida com a contrapressão e o gerenciamento de cancelamento de assinatura por conta própria. Além disso, se a API encapsulada suportar cancelamento (como a remoção do listener no exemplo), pode-se usar o setCancellation (ou setSubscription para recursos do tipo Subscription) para registrar um callback de cancelamento que é invocado quando o downstream cancela assinaturas ou o onError/onCompleted é chamado na instância Emitter fornecida.

Esses métodos permitem que apenas um único recurso seja associado ao emissor por vez e a configuração de um novo cancela a assinatura do antigo automaticamente. Se for necessário lidar com vários recursos, crie um CompositeSubscription, associe-o ao emissor e adicione mais recursos ao próprio CompositeSubscription:

Observable.create(emitter -> {
    CompositeSubscription cs = new CompositeSubscription();

    Worker worker = Schedulers.computation().createWorker();

    ActionListener al = e -> {
        emitter.onNext(e);
    };

    button.addActionListener(al);

    cs.add(worker);
    cs.add(Subscriptions.create(() -> 
        button.removeActionListener(al));

    emitter.setSubscription(cs);

}, BackpressureMode.BUFFER);

O segundo cenário geralmente envolve alguma API assíncrona, baseada em retorno de chamada, que deve ser convertida em um Observável.

Observable.create(emitter -> {
    
    someAPI.remoteCall(new Callback<Data>() {
        @Override
        public void onSuccess(Data data) {
            emitter.onNext(data);
            emitter.onCompleted();
        }

        @Override
        public void onFailure(Exception error) {
            emitter.onError(error);
        }
    });

}, BackpressureMode.LATEST);

Neste caso, a delegação funciona da mesma forma. Infelizmente, geralmente, essas APIs clássicas de estilo de retorno de chamada não suportam cancelamento, mas se o fizerem, pode-se configurar seu cancelamento como nos exemplos anteriores (embora talvez de uma maneira mais complicada). Observe o uso do modo de contrapressão LATEST; se sabemos que haverá apenas um único valor, não precisamos da estratégia BUFFER, pois ela aloca um buffer padrão de 128 elementos (que cresce conforme necessário) que nunca será totalmente utilizado.
