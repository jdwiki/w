Configurar #

O LINQ requer .NET 3.5 ou superior (ou .NET 2.0 usando LINQBridge).

Adicione uma referência ao System.Core, caso ainda não tenha sido adicionado.

Na parte superior do arquivo, importe o namespace:

  • C#
  using System;
  using System.Linq;
  • VB.NET
  Imports System.Linq

As diferentes junções no LINQ #

Nos exemplos a seguir, usaremos os seguintes exemplos:

List<Product> Products = new List<Product>()
{
  new Product()
  {
    ProductId = 1,
    Name = "Book nr 1",
    Price = 25
  },
  new Product()
  {
    ProductId = 2,
    Name = "Book nr 2",
    Price = 15
  },
  new Product()
  {
    ProductId = 3,
    Name = "Book nr 3",
    Price = 20
  },
};
List<Order> Orders = new List<Order>()
{
  new Order()
  {
    OrderId = 1,
    ProductId = 1,
  },
  new Order()
  {
    OrderId = 2,
    ProductId = 1,
  },
  new Order()
  {
    OrderId = 3,
    ProductId = 2,
  },
  new Order()
  {
    OrderId = 4,
    ProductId = NULL,
  },
};

JUNÇÃO INTERNA

Sintaxe da consulta

var joined = (from p in Products
              join o in Orders on p.ProductId equals o.ProductId
              select new
              {
                o.OrderId,
                p.ProductId,
                p.Name
              }).ToList();

Sintaxe do método 

var joined = Products.Join(Orders, p => p.ProductId, 
                                   o => o.OrderId, 
                                     => new 
                                    { 
                                      OrderId   = o.OrderId, 
                                      ProductId = p.ProductId, 
                                      Name      = p.Name 
                                    })
                     .ToList();

Resultado:

{ 1, 1, "Book nr 1" },
{ 2, 1, "Book nr 1" },
{ 3, 2, "Book nr 2" }

JUNÇÃO EXTERNA ESQUERDA

var joined = (from p in Products
              join o in Orders on p.ProductId equals o.ProductId into g
              from lj in g.DefaultIfEmpty()
              select new
              {
                //For the empty records in lj, OrderId would be NULL
                OrderId = (int?)lj.OrderId,
                p.ProductId,
                p.Name
              }).ToList();

Resultado:

{ 1, 1, "Book nr 1" },
{ 2, 1, "Book nr 1" },
{ 3, 2, "Book nr 2" },
{ NULL, 3, "Book nr 3" }

CROSS JOIN

var joined = (from p in Products
              from o in Orders
              select new
              {
                o.OrderId,
                p.ProductId,
                p.Name
              }).ToList();

Resultado:

{ 1, 1, "Book nr 1" },
{ 2, 1, "Book nr 1" },
{ 3, 2, "Book nr 2" },
{ NULL, 3, "Book nr 3" },
{ 4, NULL, NULL }

PARTICIPE DO GRUPO

var joined = (from p in Products
              join o in Orders on p.ProductId equals o.ProductId
                into t
              select new
              {
                p.ProductId,
                p.Name,
                Orders = t
              }).ToList();

A propriedade Orders agora contém um IEnumerable<Order> com todos os pedidos vinculados.

Resultado:

{ 1, "Book nr 1", Orders = { 1, 2 } },
{ 2, "Book nr 2", Orders = { 3 } },
{ 3, "Book nr 3", Orders = { } },

Como participar em várias condições

Ao ingressar em uma única condição, você pode usar:

join o in Orders 
  on p.ProductId equals o.ProductId

Ao ingressar em vários, use:

join o in Orders 
  on new { p.ProductId, p.CategoryId } equals new { o.ProductId, o.CategoryId }

Certifique-se de que ambos os objetos anônimos tenham as mesmas propriedades e, no VB.NET, eles devem ser marcados como Key, embora o VB.NET permita várias cláusulas Equals separadas por And:

Join o In Orders 
  On p.ProductId Equals o.ProductId And p.CategoryId Equals o.CategoryId

Sintaxe da consulta e sintaxe do método #

A sintaxe de consulta e a sintaxe de método são semanticamente idênticas, mas muitas pessoas acham a sintaxe de consulta mais simples e fácil de ler. Digamos que precisamos recuperar todos os itens pares ordenados em ordem crescente de uma coleção de números.

C#:

int[] numbers = { 0, 1, 2, 3, 4, 5, 6 };

// Query syntax:
IEnumerable<int> numQuery1 =
            from num in numbers
            where num % 2 == 0
            orderby num
            select num;

// Method syntax:
IEnumerable<int> numQuery2 = numbers.Where(num => num % 2 == 0).OrderBy(n => n);

VB.NET:

Dim numbers() As Integer = { 0, 1, 2, 3, 4, 5, 6 }

' Query syntax: '
Dim numQuery1 = From num In numbers
                 Where num Mod 2 = 0
                 Select num
                 Order By num

' Method syntax: '
Dim numQuery2 = numbers.where(Function(num) num Mod 2 = 0).OrderBy(Function(num) num)

Lembre-se de que algumas consultas devem ser expressas como chamadas de método. Por exemplo, você deve usar uma chamada de método para expressar uma consulta que recupera o número de elementos que correspondem a uma condição especificada. Você também deve usar uma chamada de método para uma consulta que recupera o elemento que tem o valor máximo em uma sequência de origem. Portanto, pode ser uma vantagem usar a sintaxe do método para tornar o código mais consistente. No entanto, é claro que você sempre pode aplicar o método após uma chamada de sintaxe de consulta:

C#:

int maxNum =
    (from num in numbers
     where num % 2 == 0
     select num).Max();

VB.NET:

Dim maxNum =
    (From num In numbers
     Where num Mod 2 = 0
     Select num).Max();

Métodos LINQ e IEnumerable vs IQueryable #

Os métodos de extensão LINQ em IEnumerable<T> usam métodos reais1, sejam métodos anônimos:

//C#
Func<int,bool> fn = x => x > 3;
var list = new List<int>() {1,2,3,4,5,6};
var query = list.Where(fn);

'VB.NET
Dim fn = Function(x As Integer) x > 3
Dim list = New List From {1,2,3,4,5,6};
Dim query = list.Where(fn);

ou métodos nomeados (métodos explicitamente definidos como parte de uma classe):

//C#
class Program {
    bool LessThan4(int x) {
        return x < 4;
    }

    void Main() {
        var list = new List<int>() {1,2,3,4,5,6};
        var query = list.Where(LessThan4);
    }
}

'VB.NET
Class Program
    Function LessThan4(x As Integer) As Boolean
        Return x < 4
    End Function
    Sub Main
        Dim list = New List From {1,2,3,4,5,6};
        Dim query = list.Where(AddressOf LessThan4)
    End Sub
End Class

Em teoria, é possível analisar o IL do método, descobrir o que o método está tentando fazer e aplicar a lógica desse método a qualquer fonte de dados subjacente, não apenas objetos na memória. Mas analisar IL não é para os fracos de coração.

Felizmente, .NET fornece a interface IQueryable<T> e os métodos de extensão em System.Linq.Queryable, para este cenário. Esses métodos de extensão usam uma árvore de expressão — uma estrutura de dados representando código — em vez de um método real, que o provedor LINQ pode analisar2 e converter em um formulário mais apropriado para consultar a fonte de dados subjacente. Por exemplo:

//C#
IQueryable<Person> qry = PersonsSet();

// Since we're using a variable of type Expression<Func<Person,bool>>, the compiler 
// generates an expression tree representing this code
Expression<Func<Person,bool>> expr = x => x.LastName.StartsWith("A");
// The same thing happens when we write the lambda expression directly in the call to 
// Queryable.Where

qry = qry.Where(expr);


'VB.NET
Dim qry As IQueryable(Of Person) = PersonSet()

' Since we're using a variable of type Expression(Of Func(Of Person,Boolean)), the compiler 
' generates an expression tree representing this code
Dim expr As Expression(Of Func(Of Person, Boolean)) = Function(x) x.LastName.StartsWith("A")
' The same thing happens when we write the lambda expression directly in the call to 
' Queryable.Where

qry = qry.Where(expr)

Se (por exemplo) essa consulta for em um banco de dados SQL, o provedor poderá converter essa expressão na seguinte instrução SQL:

SELECT *
FROM Persons
WHERE LastName LIKE N'A%'

e executá-lo na fonte de dados.

Por outro lado, se a consulta for em uma API REST, o provedor poderá converter a mesma expressão em uma chamada de API:

http://www.example.com/person?filtervalue=A&filtertype=startswith&fieldname=lastname

Há dois benefícios principais em personalizar uma solicitação de dados com base em uma expressão (em vez de carregar toda a coleção na memória e consultar localmente):

  • A fonte de dados subjacente geralmente pode consultar com mais eficiência. Por exemplo, pode muito bem haver um índice em LastName. Carregar os objetos na memória local e consultar na memória perde essa eficiência.
  • Os dados podem ser moldados e reduzidos antes de serem transferidos. Nesse caso, o banco de dados/serviço web precisa apenas retornar os dados correspondentes, ao contrário de todo o conjunto de Pessoas disponíveis na fonte de dados.

Notas 1. Tecnicamente, eles não aceitam métodos, mas sim instâncias de delegação que apontam para métodos. No entanto, essa distinção é irrelevante aqui. 2. Esta é a razão para erros [like](http ://stackoverflow.com/q/10110266) “LINQ to Entities não reconhece o método ‘System.String ToString()’ e este método não pode ser convertido em uma expressão de armazenamento.”. O provedor LINQ (neste caso o provedor Entity Framework) não sabe como analisar e traduzir uma chamada para ToString para SQL equivalente.
