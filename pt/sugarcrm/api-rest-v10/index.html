A API RESTful do SugarCRM foi atualizada para a versão 10 nas versões do SugarCRM 7.x.

Como as versões 7.x são construídas como um aplicativo de página única, a API é o principal meio de comunicação com o servidor para recuperar dados. As integrações de terceiros também podem vincular e acessar os dados. A API também pode ser estendida para fornecer endpoints personalizados, implementando a funcionalidade desejada.

Documentos oficiais

Fazendo login #

Você pode usar este exemplo como uma chamada do código ou por meio de um cliente REST, como Postman:

POST https://YOURSITE.com/rest/v10/oauth2/token

{
"grant_type":"password",
"client_id":"sugar",
"client_secret": "",
"username":"your_username",
"password":"your_P@$sw0rd",
"platform":"api"
}

Para detalhar os detalhes acima, eles significam:

“grant_type”:“password” - Este é o método de login. Nesse caso, estamos usando uma combinação de nome de usuário/senha, então colocamos “senha”

“client_id”:“sugar” - Indica que estamos nos autenticando através do cliente padrão “sugar” de nome/senha. Outros clientes como “support_portal”. Isso também pode fazer referência a uma chave OAuth personalizada no aplicativo.

“client_secret”: “” - Está em branco, pois estamos usando o ID do cliente “sugar”, mas se você estiver usando um cliente personalizado, esse é o segredo associado ao cliente.

“username”:“your_username” - Seu nome de usuário

“password”:“your_P@$sw0rd” - Sua senha

“platform”:“api” - As plataformas típicas usadas no açúcar são “base”, “mobile” e “portal”. Por motivos de segurança, se o mesmo usuário fizer login na mesma plataforma simultaneamente, ele desconectará o usuário da sessão anterior. Eu usei “api”, mas isso poderia ser “myawesomesugarintegration” se você quisesse.

Após uma solicitação bem-sucedida, o servidor retorna um objeto incluindo um “access_token”, ou seja:

“access_token”:"abcdef01-2345-6789-0abc-def012345678"

Para cada solicitação subsequente à API, você deve incluir esse token para se autenticar.

Isso pode ser feito adicionando o cabeçalho "OAuth-Token" com o valor access_token a qualquer solicitação adicional.

Operações básicas de CRUD #

Primeiro, certifique-se de [fazer login](https://www.wikiod.com/pt/sugarcrm/api-rest-v10#Fazendo login) e adicione o access_token ao seu cabeçalho de solicitação.

Neste exemplo, vamos fazer algumas operações básicas para acessar os registros. Eles usam o módulo Contas como exemplo, mas outros módulos padrão e personalizados (ou seja, Leads, Contatos, Oportunidades) se comportam da mesma maneira.

Crio

Para criar uma conta, precisamos fazer uma solicitação de postagem para o endpoint de contas, com os detalhes que queremos adicionar. Após o sucesso, isso retorna um objeto contendo o ID do novo registro e os dados atuais.

POST https://YOURSITE.com/rest/v10/Accounts
{
"name":"My New Account"
}

Ler

Primeiro, vamos recuperar o registro que acabamos de criar, para mim o Id era “9174c58c-409c-11e7-bfdf-00163ef1f82f”, então para recuperar todas as informações do registro, fazemos o seguinte:

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

Isso com certeza é um objeto grande! Que tal ver apenas o nome e a data em que acabei de criá-lo?

GET https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f?fields=name,date_entered

Muito melhor. Mas e se eu já tiver milhares de contas no sistema e não conseguir lembrar o GUID para isso?

GET https://YOURSITE.com/rest/v10/Accounts?fields=name,date_entered&filter[0][name]=My New Account

Atualizar

E daí se eu decidi que queria mudar alguma coisa no aplicativo? Que tal mudar o nome e adicionar uma descrição?

PUT https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f
{
    "name":"My Updated Account",
    "description":"Updated via REST API"
}

Excluir

Ok, tudo bem, mas vamos esclarecer isso antes de colocar dados reais:

DELETE https://YOURSITE.com/rest/v10/Accounts/9174c58c-409c-11e7-bfdf-00163ef1f82f

Criando endpoints personalizados #

Um dos recursos do SugarCRM 7.x é poder adicionar e estender facilmente endpoints personalizados para realizar o que você precisa.

Neste exemplo, criaremos alguns endpoints personalizados para retornar alguns dados sobre a solicitação.

Este arquivo personalizado está sendo colocado em custom/clients/base/api/DescriptionAPI.php.

<?php

if(!defined('sugarEntry') || !sugarEntry) die('Not A Valid Entry Point');

//You need to inherit the SugarApi Class to
class DescriptionApi extends SugarApi
{
    public function registerApiRest()
    {
        return array(

            //Define a key for the array
            'DescribeRequest' => array(

                //Array of the acceptable kinds of requests for this method
                'reqType' => array('GET','POST','PUT','DELETE'),

                //If true, anyone can access. If false, only authenticated users.
                'noLoginRequired' => true,

                //Here is the path to access the endpoint, in this case: Describe/Request
                'path' => array('Describe', 'Request'),

                //Specify an empty string for the path variables
                'pathVars' => array('', ''),

                //method to call
                'method' => 'DescribeMyRequest',

                //A small description, displayed in rest/v10/help page
                'shortHelp' => 'Describes your Request Method',

                //Further help, displayed when drilling down into the help page
                'longHelp' => 'custom/clients/base/api/help/DescribeRequestHelp.html',
            ),
            //Here's another entry with some more in depth information
            'DescribeIncludingArgs' => array(
                'reqType' => array('GET','POST','PUT','DELETE'),
                'noLoginRequired' => true,

                //This time, we'll include a third element with a ?
                //So now the path is Describe/Request/{dataFromURL}
                'path' => array('Describe', 'Request', '?'),

                //Here, we specify the key for accessing that data within the function
                'pathVars' => array('', '','dataFromURL'),
                
                'method' => 'DescribeMyRequestIncludingArguments',
                'shortHelp' => 'Describes the request you sent, including method, URL parameters, and request body',
                'longHelp' => 'custom/clients/base/api/help/DescribeIncludingArgsHelp.html',
            ),
        );

    }

    /**
     * Your custom logic goes in here.
     */
    public function DescribeMyRequest($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        return "You sent a $requestType request.";
    }

    /**
     * Here is the second function
     */
    public function DescribeMyRequestIncludingArguments($api, $args)
    {
        //Find out the request method sent
        $requestType = $_SERVER['REQUEST_METHOD'];

        //Get the data included in the URL parameter
        $data = $args['dataFromURL'];

        //Read from the request body
        $body = file_get_contents('php://input');

        return "You sent a $requestType request including the header argument: `$data` and the body: `$body`";
    }

}

Depois de adicionar este arquivo, você precisará executar um reparo e reconstrução para que o Sugar registre seu ponto de extremidade corretamente.

Depois, se você navegar para rest/v10/Describe/Request em seu navegador, deverá ver:

“Você enviou uma solicitação GET.”

Agora, se você usar um cliente REST para enviar uma solicitação POST, com alguns dados, por exemplo: POST rest/v10/Describe/Request/Stuff com o corpo {"key":"value"} você deverá receber :

“Você enviou uma solicitação POST incluindo o argumento de cabeçalho: Stuff e o corpo: {\"chave\":\"valor\"}”
