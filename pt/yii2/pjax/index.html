Etapa 1 Adicionar estrutura #

Em views\site\form-submission.php

<?php Pjax::begin(['id'=>'id-pjax']); ?>
<?= Html::beginForm(['site/form-submission'], 'post', ['data-pjax' => '', 'class' => 'form-inline']); ?>
    <?= Html::input('text', 'string', Yii::$app->request->post('string'), ['class' => 'form-control']) ?>
    <?= Html::submitButton('Hash String', ['class' => 'btn btn-lg btn-primary', 'name' => 'hash-button']) ?>
<?= Html::endForm() ?>
<h3><?= $stringHash ?></h3>
<?php Pjax::end(); ?>

como usar pjax #

Adicione esta linha no início da sua visualização.

<?php
use yii\widgets\Pjax;
?>

Adicione as duas linhas a seguir ao redor do conteúdo que precisa de atualização parcial.

<?php Pjax::begin(['id'=>'id-pjax']); ?>
Content that needs to be updated
<?php Pjax::end(); ?>

recarregar pjax #

$.pjax.reload({container: '#id-pjax'});

usar argumento de tempo limite em pjax #

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false]); ?>

você pode especificar um valor inteiro para o argumento de tempo limite, que seria o número de milissegundos de espera (seu valor padrão é 1000). Se o tempo de execução no servidor for maior que esse valor de tempo limite, um carregamento de página completo será acionado.

Por padrão, o pjax enviará o formulário usando o método GET. Você pode alterar o método de envio do formulário para POST como no exemplo a seguir

<?php Pjax::begin(['id'=>'id-pjax', 'timeout' => false, 'clientOptions' => ['method' => 'POST']]); ?>

Passo 2 Código do lado do servidor #

public function actionFormSubmission()
{
    $security = new Security();
    $string = Yii::$app->request->post('string');
    $stringHash = '';
    if (!is_null($string)) {
        $stringHash = $security->generatePasswordHash($string);
    }
    return $this->render('form-submission', [
        'stringHash' => $stringHash,
    ]);
}

Uso avançado do Pjax #

O Yii Framework 2.0 vem com suporte embutido para Pjax, uma biblioteca JavaScript que reduz o tempo de carregamento da página. Ele faz isso atualizando apenas a parte da página que foi alterada por meio do Ajax, o que pode se traduzir em economias substanciais se você tiver muitos outros ativos em suas páginas. Alguns de nossos projetos usam essa funcionalidade e queríamos compartilhar algumas lições aprendidas.

Problema: a página 1 é uma página estática simples que contém poucos elementos. A página 2 inclui um ActiveForm e outros widgets. Os recursos JavaScript do ActiveForm precisam ser carregados para que o JavaScript embutido seja executado, mas como a página 1 não inclui esses recursos, a página 2 encontrou um erro de JavaScript ao tentar executar a linha do formulário ativo: ‘Uncaught TypeError: undefined is not a função’.

Solução: inclua ativos do ActiveForm em um pacote de ativos compartilhado que será carregado em todas as páginas, garantindo que qualquer página de entrada permita que os scripts corretos estejam disponíveis.

class AppAsset extends AssetBundle
{
    ...
    public $depends = [
        'yii\widgets\ActiveFormAsset',
        'yii\validators\ValidationAsset',
    ];
    ...
}

Problema: no mesmo exemplo acima, a Página 1 inclui alguns widgets (NavBar, etc.). A página 2 inclui os mesmos widgets e mais alguns (ActiveForm, etc.). Ao carregar a página via Pjax, algum JavaScript embutido personalizado estava sendo executado, mas o script embutido colocado pelo widget ActiveForm parecia não funcionar, pois o código de validação não estava funcionando. Na depuração, descobrimos que a função init do ActiveForm estava em execução, mas a variável ‘this’ não parecia corresponder ao ActiveForm. Na verdade, correspondia ao div NavBar. Investigando os div IDs, vimos que o ActiveForm esperava ter o ID #w1, mas o NavBar já tinha esse ID atribuído na página 1, pois esse foi o primeiro widget encontrado nessa página.

Solução: Não confie no Yii para gerar automaticamente os IDs dos widgets para você. Em vez disso, sempre passe um ID ao criar o widget para manter o controle desses IDs.

Problema: a solicitação Pjax estava sendo cancelada exatamente 1.000 ms após o início da solicitação.

Solução: Aumente a configuração de tempo limite do Pjax. O padrão é 1 segundo, o que deve ser aceitável para sites de produção. No entanto, em desenvolvimento, ao usar o xdebug, nossos tempos de carregamento de página estão regularmente acima desse limite.

Problema: o aplicativo Web implementa o padrão Post-Redirect-Get (PRG). O Pjax recarrega a página inteira em vez de apenas o redirecionamento.

Solução: Este é o comportamento pretendido do Pjax. O redirecionamento não cumpre seu propósito ao usar o Pjax, portanto, você pode determinar se uma solicitação é Pjax e, em caso afirmativo, renderizar o conteúdo em vez de redirecionar. Um exemplo pode parecer:

$endURL = "main/endpoint";
if (Yii::$app->request->isPjax) {
    return $this->run($endURL);
} else {
    return $this->redirect([$endURL]);
}
