O Propósito das Macros #

As macros destinam-se a gerar código, transformar código e fornecer novas notações. Essas novas notações podem ser mais adequadas para expressar melhor o programa, por exemplo, fornecendo construções em nível de domínio ou novas linguagens incorporadas inteiras.

As macros podem tornar o código-fonte mais autoexplicativo, mas a depuração pode ser mais difícil. Como regra geral, não se deve usar macros quando uma função regular servirá. Quando você usá-los, evite as armadilhas usuais, tente se ater aos padrões e convenções de nomenclatura comumente usados.

Pedido de macroexpansão #

Em comparação com as funções, as macros são expandidas em ordem inversa; o máximo primeiro, o mais íntimo por último. Isso significa que, por padrão, não se pode usar uma macro interna para gerar a sintaxe necessária para uma macro externa.

Ordem de avaliação #

Às vezes, as macros precisam mover os formulários fornecidos pelo usuário. Deve-se certificar-se de não alterar a ordem em que eles são avaliados. O usuário pode estar contando com os efeitos colaterais acontecendo em ordem.

Avaliar apenas uma vez #

A expansão de uma macro geralmente precisa usar o valor do mesmo formulário fornecido pelo usuário mais de uma vez. É possível que o formulário tenha efeitos colaterais, ou pode estar chamando uma função cara. Assim, a macro deve certificar-se de avaliar esses formulários apenas uma vez. Normalmente isso será feito atribuindo o valor a uma variável local (cujo nome é GENSYMed).

Funções usadas por Macros, usando EVAL-WHEN #

Macros complexas geralmente têm partes de sua lógica implementadas em funções separadas. Deve-se lembrar, no entanto, que as macros são expandidas antes que o código real seja compilado. Ao compilar um arquivo, por padrão, as funções e variáveis ​​definidas no mesmo arquivo não estarão disponíveis durante a execução da macro. Todas as definições de funções e variáveis, no mesmo arquivo, usadas por uma macro devem ser encapsuladas dentro de um EVAL-WHEN-form . O EVAL-WHEN deve ter todos os três tempos especificados, quando o código incluído também deve ser avaliado durante o carregamento e o tempo de execução.

(eval-when (:compile-toplevel :load-toplevel :execute)
  (defun foobar () ...))

Isso não se aplica a funções chamadas a partir da expansão da macro, apenas as chamadas pela própria macro.

Macros anafóricas #

Uma Anaphoric Macro é uma macro que introduz uma variável (geralmente IT) que captura o resultado de um formulário fornecido pelo usuário. Um exemplo comum é o Anaphoric If, que é como um IF regular, mas também define a variável IT para se referir ao resultado do formulário de teste.

(defmacro aif (test-form then-form &optional else-form)
  `(let ((it ,test-form))
     (if it ,then-form ,else-form)))

(defun test (property plist)
  (aif (getf plist property)
       (format t "The value of ~s is ~a.~%" property it)
       (format t "~s wasn't in ~s!~%" property plist)))

(test :a '(:a 10 :b 20 :c 30))
; The value of :A is 10.
(test :d '(:a 10 :b 20 :c 30))
; :D wasn't in (:A 10 :B 20 :C 30)!

Backquote - escrevendo modelos de código para macros #

Código de retorno das macros. Como o código em Lisp consiste em listas, pode-se usar as funções regulares de manipulação de lista para gerá-lo.

;; A pointless macro
(defmacro echo (form)
  (list 'progn
        (list 'format t "Form: ~a~%" (list 'quote form))
        form))

Isso geralmente é muito difícil de ler, especialmente em macros mais longas. A macro do leitor Backquote permite escrever modelos citados que são preenchidos avaliando seletivamente os elementos.

(defmacro echo (form)
  `(progn
     (format t "Form: ~a~%" ',form)
     ,form))

(macroexpand '(echo (+ 3 4)))
;=> (PROGN (FORMAT T "Form: ~a~%" '(+ 3 4)) (+ 3 4))

Esta versão parece quase um código normal. As vírgulas são usadas para avaliar FORM; todo o resto é retornado como está. Observe que em ',form a aspa simples está fora da vírgula, então ela será retornada.

Pode-se também usar ,@ para unir uma lista na posição.

(defmacro echo (&rest forms)
  `(progn
     ,@(loop for form in forms collect `(format t "Form: ~a~%" ,form))
     ,@forms))

(macroexpand '(echo (+ 3 4) 
                    (print "foo")
                    (random 10)))
;=> (PROGN
;    (FORMAT T "Form: ~a~%" (+ 3 4))
;    (FORMAT T "Form: ~a~%" (PRINT "foo"))
;    (FORMAT T "Form: ~a~%" (RANDOM 10))
;    (+ 3 4)
;    (PRINT "foo")
;    (RANDOM 10))

Backquote também pode ser usado fora de macros.

FOOF #

Em Common Lisp, existe um conceito de Referências Generalizadas. Eles permitem que um programador defina valores para vários “lugares” como se fossem variáveis. Macros que fazem uso dessa habilidade geralmente têm um F-postfix no nome. O lugar é geralmente o primeiro argumento para a macro.

Exemplos do padrão: INCF, DECF, [ROTATEF](http://www. lispworks.com/documentation/HyperSpec/Body/m_rotate.htm#rotatef), SHIFTF, REMF .

Um exemplo bobo, uma macro que inverte o sinal de uma loja de números em um lugar:

(defmacro flipf (place)
  `(setf ,place (- ,place)))

COM-FOO #

As macros que adquirem e liberam um recurso com segurança geralmente são nomeadas com um prefixo WITH-. A macro geralmente deve usar sintaxe como:

(with-foo (variable details-of-the-foo...)
  body...)

Exemplos do padrão: WITH-OPEN-FILE, [WITH-OPEN-STREAM](http:// www.lispworks.com/documentation/HyperSpec/Body/m_w_op_1.htm), WITH-INPUT-FROM-STRING, WITH-OUTPUT-TO-STRING.

Uma abordagem para implementar esse tipo de macro que pode evitar algumas das armadilhas da poluição de nomes e da avaliação múltipla não intencional é implementar primeiro uma versão funcional. Por exemplo, o primeiro passo na implementação de uma macro with-widget que cria um widget com segurança e limpa depois pode ser uma função:

(defun call-with-widget (args function)
  (let ((widget (apply #'make-widget args))) ; obtain WIDGET
    (unwind-protect (funcall function widget)  ; call FUNCTION with WIDGET
      (cleanup widget)                         ; cleanup

Como se trata de uma função, não há preocupação com o escopo dos nomes em função ou fornecedor e facilita a gravação de uma macro correspondente:

(defmacro with-widget ((var &rest args) &body body)
  `(call-with-widget (list ,@args) (lambda (,var) ,@body)))

#DO-FOO

Macros que iteram sobre algo geralmente são nomeadas com um prefixo DO. A sintaxe da macro geralmente deve estar no formato

(do-foo (variable the-foo-being-done return-value)
  body...)

Exemplos do padrão: DOTIMES, [DOLIST](http://www.lispworks.com/documentation/ HyperSpec/Body/m_dolist.htm), DO-SYMBOLS.

#FOOCASE, EFOOCASE, CFOOCASE

As macros que correspondem a uma entrada em relação a certos casos geralmente são nomeadas com um postfix CASE. Geralmente existe uma variante E...CASE, que sinaliza um erro se a entrada não corresponder a nenhum dos casos, e C...CASE, que sinaliza um erro contínuo. Eles devem ter sintaxe como

(foocase input
  (case-to-match-against (optionally-some-params-for-the-case)
   case-body-forms...)
  more-cases...
  [(otherwise otherwise-body)])

Exemplos do padrão: CASE, [TYPECASE](http://www.lispworks.com/documentation/ HyperSpec/Body/m_tpcase.htm), HANDLER-CASE.

Por exemplo, uma macro que combina uma string com expressões regulares e liga os grupos de registradores a variáveis. Usa CL-PPCRE para expressões regulares.

(defmacro regexcase (input &body cases)
  (let ((block-sym (gensym "block"))
        (input-sym (gensym "input")))
    `(let ((,input-sym ,input))
       (block ,block-sym
         ,@(loop for (regex vars . body) in cases
                 if (eql regex 'otherwise)
                   collect `(return-from ,block-sym (progn ,vars ,@body))
                 else
                   collect `(cl-ppcre:register-groups-bind ,vars
                                (,regex ,input-sym)
                              (return-from ,block-sym
                                (progn ,@body))))))))

(defun test (input)
  (regexcase input
    ("(\\d+)-(\\d+)" (foo bar)
      (format t "Foo: ~a, Bar: ~a~%" foo bar))
    ("Foo: (\\w+)$" (foo)
      (format t "Foo: ~a.~%" foo))
    (otherwise (format t "Didn't match.~%"))))

(test "asd 23-234 qwe")
; Foo: 23, Bar: 234
(test "Foo: Foobar")
; Foo: Foobar.
(test "Foo: 43 - 23")
; Didn't match.

DEFINE-FOO, DEFFOO #

Macros que definem coisas geralmente são nomeadas com o prefixo DEFINE- ou DEF.

Exemplos do padrão: DEFUN, [DEFMACRO](http://www.lispworks.com/documentation/ HyperSpec/Body/m_defmac.htm), DEFINE-CONDITION.

MACROEXPANDAR #

A expansão de macros é o processo de transformar macros em código real. Isso geralmente acontece como parte do processo de compilação. O compilador expandirá todas as formas de macro antes de realmente compilar o código. A expansão de macro também acontece durante a interpretação do código Lisp.

Pode-se chamar MACROEXPAND manualmente para ver para que um formulário de macro se expande.

CL-USER> (macroexpand '(with-open-file (file "foo")
                        (do-something-with file)))
(LET ((FILE (OPEN "foo")) (#:G725 T))
  (UNWIND-PROTECT
      (MULTIPLE-VALUE-PROG1 (PROGN (DO-SOMETHING-WITH FILE)) (SETQ #:G725 NIL))
    (WHEN FILE (CLOSE FILE :ABORT #:G725))))

MACROEXPAND-1 é o mesmo, mas só se expande uma vez. Isso é útil ao tentar entender um formulário de macro que se expande para outro formulário de macro.

CL-USER> (macroexpand-1 '(with-open-file (file "foo")
                          (do-something-with file)))
(WITH-OPEN-STREAM (FILE (OPEN "foo")) (DO-SOMETHING-WITH FILE))

Observe que nem MACROEXPAND nem MACROEXPAND-1 expandem o código Lisp em todos os níveis. Eles apenas expandem o formulário de macro de nível superior. Para macroexpandir um formulário completamente em todos os níveis, é necessário um code walker para fazer isso. Esse recurso não é fornecido no padrão Common Lisp.

macros if-let, when-let, -let #

Essas macros mesclam fluxo de controle e vinculação. Eles são uma melhoria em relação às macros anafóricas porque permitem que o desenvolvedor comunique o significado por meio da nomeação. Como tal, seu uso é recomendado em detrimento de suas contrapartes anafóricas.

(if-let (user (get-user user-id))
  (show-dashboard user)
  (redirect 'login-page))

As macros FOO-LET vinculam uma ou mais variáveis ​​e, em seguida, usam essas variáveis ​​como o formulário de teste para a condicional correspondente (IF, WHEN). Várias variáveis ​​são combinadas com AND. A ramificação escolhida é executada com as ligações em vigor. Uma implementação simples de uma variável de IF-LET pode ser algo como:

(defmacro if-let ((var test-form) then-form &optional else-form)
  `(let ((,var ,test-form))
     (if ,var ,then-form ,else-form)))

(macroexpand '(if-let (a (getf '(:a 10 :b 20 :c 30) :a))
               (format t "A: ~a~%" a)
               (format t "Not found.~%")))
; (LET ((A (GETF '(:A 10 :B 20 :C 30) :A)))
;   (IF A
;       (FORMAT T "A: ~a~%" A)
;       (FORMAT T "Not found.~%")))

Uma versão que suporta múltiplas variáveis ​​está disponível na biblioteca Alexandria.

Símbolos exclusivos para evitar conflitos de nomes em macros #

A expansão de uma macro geralmente precisa usar símbolos que não foram passados ​​como argumentos pelo usuário (como nomes de variáveis ​​locais, por exemplo). Deve-se ter certeza de que tais símbolos não podem entrar em conflito com um símbolo que o usuário está usando no código ao redor.

Isso geralmente é obtido usando GENSYM, uma função que retorna um novo símbolo não interno.

Mau

Considere a macro abaixo. Faz um loop DOTIMES que também coleta o resultado do corpo em uma lista, que é retornada no final.

(defmacro dotimes+collect ((var count) &body body)
  `(let ((result (list)))
     (dotimes (,var ,count (nreverse result))
       (push (progn ,@body) result))))

(dotimes+collect (i 5)
  (format t "~a~%" i)
  (* i i))
; 0
; 1
; 2
; 3
; 4
;=> (0 1 4 9 16)

Isso parece funcionar neste caso, mas se o usuário tiver um nome de variável RESULT, que ele usa no corpo, os resultados provavelmente não seriam o que o usuário espera. Considere esta tentativa de escrever uma função que coleta uma lista de somas de todos os inteiros até N:

(defun sums-upto (n)
  (let ((result 0))
    (dotimes+collect (i n)
      (incf result i))))

(sums-upto 10) ;=> Error!

Bom

Para corrigir o problema, precisamos usar GENSYM para gerar um nome exclusivo para a variável RESULT na expansão da macro.

(defmacro dotimes+collect ((var count) &body body)
  (let ((result-symbol (gensym "RESULT")))
    `(let ((,result-symbol (list)))
       (dotimes (,var ,count (nreverse ,result-symbol))
         (push (progn ,@body) ,result-symbol)))))

(sums-upto 10) ;=> (0 1 3 6 10 15 21 28 36 45)

Usando Macros para definir estruturas de dados #

Um uso comum de macros é criar modelos para estruturas de dados que obedecem a regras comuns, mas podem conter campos diferentes. Ao escrever uma macro, você pode permitir que a configuração detalhada da estrutura de dados seja especificada sem a necessidade de repetir o código clichê, nem usar uma estrutura menos eficiente (como um hash) na memória apenas para simplificar a programação.

Por exemplo, suponha que desejamos definir um número de classes que possuem um intervalo de propriedades diferentes, cada uma com um getter e um setter. Além disso, para algumas (mas não todas) dessas propriedades, desejamos que o setter chame um método no objeto notificando-o de que a propriedade foi alterada. Embora o Common LISP já tenha uma abreviação para escrever getters e setters, escrever um setter personalizado padrão dessa maneira normalmente exigiria a duplicação do código que chama o método de notificação em cada setter, o que pode ser um problema se houver um grande número de propriedades envolvidas . No entanto, definindo uma macro fica muito mais fácil:

(defmacro notifier (class slot) 
  "Defines a setf method in (class) for (slot) which calls the object's changed method."
   `(defmethod (setf ,slot) (val (item ,class))
     (setf (slot-value item ',slot) val)
     (changed item ',slot)))

(defmacro notifiers (class slots)
  "Defines setf methods in (class) for all of (slots) which call the object's changed method."
  `(progn 
     ,@(loop for s in slots collecting `(notifier ,class ,s))))

(defmacro defclass-notifier-slots (class nslots slots)  
  "Defines a class with (nslots) giving a list of slots created with notifiers, and (slots) giving a list of slots created with regular accessors."
  `(progn
     (defclass ,class () 
       ( ,@(loop for s in nslots collecting `(,s :reader ,s)) 
         ,@(loop for s in slots collecting `(,s :accessor ,s))))   
     (notifiers ,class ,nslots)))

Podemos agora escrever (defclass-notifier-slots foo (bar baz qux) (waldo)) e imediatamente definir uma classe foo com um slot regular waldo (criado pela segunda parte da macro com a especificação (waldo :accessor waldo)), e slots bar, baz e qux com setters que chamam o método changed (onde o getter é definido pela primeira parte da macro, (bar : reader bar) e o setter pela macro notifier invocada).

Além de nos permitir definir rapidamente várias classes que se comportam dessa maneira, com grande número de propriedades, sem repetição, temos o benefício usual de reutilização de código: se depois decidirmos alterar o funcionamento dos métodos notificadores, podemos simplesmente alterar o macro, e a estrutura de cada classe que a usa mudará.
