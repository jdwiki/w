Diferentes tipos de ThreadPools #

SingleThreadExecutor: Executor que usa um único thread de trabalho operando em uma fila ilimitada e usa o ThreadFactory fornecido para criar um novo thread quando necessário. Ao contrário do newFixedThreadPool(1, threadFactory) equivalente, o executor retornado é garantido para não ser reconfigurável para usar threads adicionais.

FixedThreadPool: conjunto de encadeamentos que reutiliza um número fixo de encadeamentos operando em uma fila ilimitada compartilhada, usando o ThreadFactory fornecido para criar novos encadeamentos quando necessário. Em qualquer ponto, no máximo nThreads threads serão tarefas de processamento ativas. Se tarefas adicionais forem enviadas quando todos os encadeamentos estiverem ativos, eles aguardarão na fila até que um encadeamento esteja disponível. Se algum encadeamento for encerrado devido a uma falha durante a execução antes do desligamento, um novo tomará seu lugar, se necessário, para executar as tarefas subsequentes. Os encadeamentos no pool existirão até que seja explicitamente encerrado.

CachedThreadPool: pool de threads que cria novos threads conforme necessário, mas reutilizará threads construídos anteriormente quando estiverem disponíveis e usa o ThreadFactory fornecido para criar novos threads quando necessário.

SingleThreadScheduledExecutor: executor de thread único que pode agendar comandos para execução após um determinado atraso ou para execução periódica. (Observe, no entanto, que se este único thread terminar devido a uma falha durante a execução antes do desligamento, um novo assumirá seu lugar, se necessário, para executar tarefas subsequentes.) As tarefas são garantidas para execução sequencial e não mais de uma tarefa estará ativa a qualquer momento. Ao contrário do newScheduledThreadPool(1, threadFactory) equivalente, o executor retornado é garantido para não ser reconfigurável para usar threads adicionais.

ScheduledThreadPool: conjunto de threads que pode agendar comandos para serem executados após um determinado atraso ou para serem executados periodicamente. Diferentes tipos de filas de trabalho

Runnables personalizados em vez de Callables #

Outra boa prática para verificar quando nossas threads terminaram sem bloquear a thread esperando para recuperar o objeto Future de nosso Callable é criar nossa própria implementação para Runnables, usando-a junto com o método execute().

No próximo exemplo, mostro uma classe customizada que implementa Runnable com um callback interno, que nos permite saber quando os executáveis ​​estão finalizados e usá-lo posteriormente em nosso ThreadPool:

public class CallbackTask implements Runnable {
    private final Runnable mTask;
    private final RunnableCallback mCallback;

    public CallbackTask(Runnable task, RunnableCallback runnableCallback) {
        this.mTask = task;
        this.mCallback = runnableCallback;
    }

    public void run() {
        long startRunnable = System.currentTimeMillis();
        mTask.run();
        mCallback.onRunnableComplete(startRunnable);
    }

    public interface RunnableCallback {
        void onRunnableComplete(long runnableStartTime);
    }
}

E aqui está nossa implementação do ThreadExecutor:

public class ThreadExecutorExample implements ThreadExecutor {

    private static String TAG = "ThreadExecutorExample";
    public static final int THREADPOOL_SIZE = 4;
    private long mSubmittedTasks;
    private long mCompletedTasks;
    private long mNotCompletedTasks;

    private ThreadPoolExecutor mThreadPoolExecutor;

    public ThreadExecutorExample() {
        Log.i(TAG, "[ThreadExecutorImpl] Initializing ThreadExecutorImpl");
        Log.i(TAG, "[ThreadExecutorImpl] current cores: " + Runtime.getRuntime().availableProcessors());
        this.mThreadPoolExecutor =
            (ThreadPoolExecutor) Executors.newFixedThreadPool(THREADPOOL_SIZE);

    }

    @Override
    public void execute(Runnable runnable) {
        try {
            if (runnable == null) {
                Log.e(TAG, "[execute] Runnable to execute cannot be null");
                return;
            }
            Log.i(TAG, "[execute] Executing new Thread");

            this.mThreadPoolExecutor.execute(new CallbackTask(runnable, new CallbackTask.RunnableCallback() {

                @Override
                public void onRunnableComplete(long RunnableStartTime) {
                    mSubmittedTasks = mThreadPoolExecutor.getTaskCount();
                    mCompletedTasks = mThreadPoolExecutor.getCompletedTaskCount();
                    mNotCompletedTasks = mSubmittedTasks - mCompletedTasks; // approximate

                    Log.i(TAG, "[execute] [onRunnableComplete] Runnable complete in " + (System.currentTimeMillis() - RunnableStartTime) + "ms");
                    Log.i(TAG, "[execute] [onRunnableComplete] Current threads working " + mNotCompletedTasks);
                }
            }));
        }
        catch (Exception e) {
            e.printStackTrace();
            Log.e(TAG, "[execute] Error, shutDown the Executor");
            this.mThreadPoolExecutor.shutdown();
        }
    }
}


 /**
 * Executor thread abstraction created to change the execution context from any thread from out ThreadExecutor.
 */
interface ThreadExecutor  extends Executor {

    void execute(Runnable runnable);

}

Definindo um novo ThreadPool #

Um ThreadPool é um ExecutorService que executa cada tarefa enviada usando um dos vários threads agrupados, normalmente configurados usando os métodos de fábrica do Executors.

Aqui está um código básico para inicializar um novo ThreadPool como um singleton para usar em seu aplicativo:

public final class ThreadPool {

    private static final String TAG = "ThreadPool";
    private static final int CORE_POOL_SIZE = 4;
    private static final int MAX_POOL_SIZE = 8;
    private static final int KEEP_ALIVE_TIME = 10; // 10 seconds
    private final Executor mExecutor;

    private static ThreadPool sThreadPoolInstance;

    private ThreadPool() {
        mExecutor = new ThreadPoolExecutor(
            CORE_POOL_SIZE, MAX_POOL_SIZE, KEEP_ALIVE_TIME,
            TimeUnit.SECONDS, new LinkedBlockingQueue<Runnable>());
    }

    public void execute(Runnable runnable) {
        mExecutor.execute(runnable);
    }

    public synchronized static ThreadPool getThreadPoolInstance() {
        if (sThreadPoolInstance == null) {
            Log.i(TAG, "[getThreadManagerInstance] New Instance");
            sThreadPoolInstance = new ThreadPool();
        }
        return sThreadPoolInstance;
    }
}

Você tem duas maneiras de chamar seu método executável, use execute() ou submit(). a diferença entre eles é que submit() retorna um objeto Future que permite uma maneira de cancelar programaticamente a thread em execução quando o objeto T é retornado do callback Callable. Futuro e exigíveis #

Um dos recursos que podemos usar com os Threadpools é o método submit() que nos permite saber quando o thread terminou seu trabalho. Podemos fazer isso graças ao objeto Future, que nos retorna um objeto do Callable que podemos usar para nossos próprios objetivos.

Aqui está um exemplo sobre como usar a instância Callable:

public class CallablesExample{

//Create MyCustomCallable instance
List<Future<String>> mFutureList = new ArrayList<Future<String>>();

//Create a list to save the Futures from the Callable
Callable<String> mCallable = new MyCustomCallable();

public void main(String args[]){
    //Get ExecutorService from Executors utility class, Creating a 5 threads pool.
    ExecutorService executor = Executors.newFixedThreadPool(5);
   
    
   
    for (int i = 0; i < 100; i++) {
        //submit Callable tasks to be executed by thread pool
        Future<String> future = executor.submit(mCallable);
        //add Future to the list, we can get return value using Future
        mFutureList.add(future);
    }
    for (Future<String> fut : mFutureList) {
        try {
            //Print the return value of Future, Notice the output delay in console
            //because Future.get() stop the thread till the task have been completed
            System.out.println(new Date() + "::" + fut.get());
        } catch (InterruptedException | ExecutionException e) {
            e.printStackTrace();
        }
    }
    //Shut down the service
    executor.shutdown();
}

 class MyCustomCallable implements Callable<String> {

    @Override
    public String call() throws Exception {
        Thread.sleep(1000);
        //return the thread name executing this callable task
        return Thread.currentThread().getName();
    }
}
}

Como você pode ver, criamos um Threadpool com 5 Threads, isso significa que podemos lançar 5 callables em paralelo. Quando as threads terminarem, obteremos um objeto Future do callable, neste caso o nome da thread.

AVISO

Neste exemplo, usamos apenas o Futures como um objeto dentro do array para saber quantas threads estamos executando e imprimir tantas vezes um log in no console com os dados que queremos. Mas, se quisermos usar o método Future.get(), para nos retornar os dados que salvamos antes no callable, bloquearemos a thread até que a tarefa seja concluída. Tenha cuidado com esse tipo de chamadas quando quiser fazer isso o mais rápido possível

Adicionando ThreadFactory ao Executor #

Usamos ExecutorService para atribuir threads do pool de threads interno ou criá-los sob demanda para executar tarefas. Cada ExecutorService tem um ThreadFactory, mas o ExecutorService usará sempre um padrão se não definirmos um personalizado. Por que devemos fazer isso?

  • Para definir um nome de thread mais descritivo. ThreadFactory padrão fornece nomes de threads na forma de pool-m-thread-n, como pool-1-thread-1, pool-2-thread-1, pool-3-thread-1, etc. depurar ou monitorar algo, é difícil saber o que esses threads estão fazendo

  • Defina um status de Daemon personalizado, o ThreadFactory padrão produz resultados não-daemon.

  • Definir prioridade para nossas threads, o padrão ThreadFactory define uma prioridade média para todas as suas threads.

  • Você pode especificar UncaughtExceptionHandler para nosso thread usando setUncaughtExceptionHandler() no objeto thread. Isso é chamado de volta quando o método run do Thread lança uma exceção não capturada.

Aqui está uma implementação fácil de um ThreadFactory sobre um ThreadPool.

public class ThreadExecutorExample implements ThreadExecutor {
private static String TAG = "ThreadExecutorExample";
private static final int INITIAL_POOL_SIZE = 3;
private static final int MAX_POOL_SIZE = 5;

// Sets the amount of time an idle thread waits before terminating
private static final int KEEP_ALIVE_TIME = 10;

// Sets the Time Unit to seconds
private static final TimeUnit KEEP_ALIVE_TIME_UNIT = TimeUnit.SECONDS;

private final BlockingQueue<Runnable> workQueue;

private final ThreadPoolExecutor threadPoolExecutor;

private final ThreadFactory threadFactory;
private ThreadPoolExecutor mThreadPoolExecutor;

public ThreadExecutorExample() {
    this.workQueue = new LinkedBlockingQueue<>();
    this.threadFactory = new CustomThreadFactory();
    this.threadPoolExecutor = new ThreadPoolExecutor(INITIAL_POOL_SIZE, MAX_POOL_SIZE,
            KEEP_ALIVE_TIME, KEEP_ALIVE_TIME_UNIT, this.workQueue, this.threadFactory);
}

public void execute(Runnable runnable) {
    if (runnable == null) {
        return;
    }
    this.threadPoolExecutor.execute(runnable);
}

private static class CustomThreadFactory implements ThreadFactory {
    private static final String THREAD_NAME = "thread_";
    private int counter = 0;

    @Override public Thread newThread(Runnable runnable) {
        return new Thread(runnable, THREAD_NAME + counter++);
    }
}
}

/**
 * Executor thread abstraction created to change the execution context from any thread from out ThreadExecutor.
 */
interface ThreadExecutor extends Executor {
    
    void execute(Runnable runnable);
    
}

Este exemplo apenas modifica o nome do Thread com um contador, mas podemos modificá-lo o quanto quisermos.
