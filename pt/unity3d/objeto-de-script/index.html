ScriptableObjects com AssetBundles #

Preste atenção ao adicionar prefabs a AssetBundles se eles contiverem referências a ScriptableObjects. Como os ScriptableObjects são essencialmente ativos, o Unity cria duplicatas deles antes de adicioná-los ao AssetBundles, o que pode resultar em comportamento indesejado durante o tempo de execução.

Quando você carrega tal GameObject de um AssetBundle, pode ser necessário reinjetar os ativos ScriptableObject nos scripts carregados, substituindo os empacotados. Veja Injeção de Dependência

Introdução #

ScriptableObjects são objetos serializados que não estão vinculados a cenas ou objetos de jogo como os MonoBehaviours. Para colocar de uma forma, eles são dados e métodos vinculados a arquivos de ativos dentro de seu projeto. Esses ativos ScriptableObject podem ser passados ​​para MonoBehaviours ou outros ScriptableObjects, onde seus métodos públicos podem ser acessados.

Devido à sua natureza como ativos serializados, eles são excelentes classes de gerenciadores e fontes de dados.

Criando ativos ScriptableObject #

Abaixo está uma implementação simples de ScriptableObject.

using UnityEngine;

[CreateAssetMenu(menuName = "StackOverflow/Examples/MyScriptableObject")]
public class MyScriptableObject : ScriptableObject
{
    [SerializeField]
    int mySerializedNumber;

    int helloWorldCount = 0;

    public void HelloWorld()
    {
        helloWorldCount++;
        Debug.LogFormat("Hello! My number is {0}.", mySerializedNumber);
        Debug.LogFormat("I have been called {0} times.", helloWorldCount);
    }
}

Ao adicionar o atributo CreateAssetMenu à classe, o Unity o listará no submenu Assets/Create. Nesse caso, está em Assets/Create/StackOverflow/Examples.

Uma vez criadas, as instâncias de ScriptableObject podem ser passadas para outros scripts e ScriptableObjects por meio do Inspector.

using UnityEngine;

public class SampleScript : MonoBehaviour {

    [SerializeField]
    MyScriptableObject myScriptableObject;

    void OnEnable()
    {
        myScriptableObject.HelloWorld();
    }
}

Crie instâncias de ScriptableObject por meio de código #

Você cria novas instâncias de ScriptableObject por meio de ScriptableObject.CreateInstance<T>()

T obj = ScriptableObject.CreateInstance<T>();

Onde T estende ScriptableObject.

Não crie ScriptableObjects chamando seus construtores, ou seja. novo ScriptableObject().

A criação de ScriptableObjects por código durante o tempo de execução raramente é necessária porque seu principal uso é a serialização de dados. Você também pode usar classes padrão neste momento. É mais comum quando você está criando scripts de extensões do editor.

ScriptableObjects são serializados no editor mesmo no PlayMode #

Cuidado extra deve ser tomado ao acessar campos serializados em uma instância de ScriptableObject.

Se um campo for marcado como “público” ou serializado por meio de “SerializeField”, a alteração de seu valor será permanente. Eles não são redefinidos ao sair do modo de reprodução como os MonoBehaviours. Isso pode ser útil às vezes, mas também pode fazer uma bagunça.

Por isso, é melhor tornar os campos serializados somente leitura e evitar completamente os campos públicos.

public class MyScriptableObject : ScriptableObject
{
    [SerializeField]
    int mySerializedValue;

    public int MySerializedValue
    {
        get { return mySerializedValue; }
    }
}

Se você deseja armazenar valores públicos em um ScriptableObject que são redefinidos entre as sessões de reprodução, considere usar o padrão a seguir.

public class MyScriptableObject : ScriptableObject
{
    // Private fields are not serialized and will reset to default on reset
    private int mySerializedValue;

    public int MySerializedValue
    {
        get { return mySerializedValue; }
        set { mySerializedValue = value; }
    }
}

Encontre ScriptableObjects existentes durante o tempo de execução #

Para encontrar ScriptableObjects ativos durante o tempo de execução, você pode usar Resources.FindObjectsOfTypeAll().

T[] instances = Resources.FindObjectsOfTypeAll<T>();

Onde T é o tipo da instância ScriptableObject que você está pesquisando. Ativo significa que já foi carregado na memória de alguma forma antes.

Esse método é muito lento, portanto, lembre-se de armazenar em cache o valor de retorno e evite chamá-lo com frequência. Fazer referência aos ScriptableObjects diretamente em seus scripts deve ser sua opção preferida.

Dica: Você pode manter suas próprias coleções de instâncias para pesquisas mais rápidas. Faça com que seus ScriptableObjects se registrem em uma coleção compartilhada durante OnEnable().
