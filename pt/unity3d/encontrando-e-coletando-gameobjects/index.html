Sintaxe #

  • Localizar GameObject estático público (nome da string);
  • público estático GameObject FindGameObjectWithTag(string tag);
  • public static GameObject[] FindGameObjectsWithTag(string tag);
  • objeto estático público FindObjectOfType(tipo de tipo);
  • public static Object[] FindObjectsOfType(Type type);

Qual método usar #

Tenha cuidado ao procurar GameObjects em tempo de execução, pois isso pode consumir recursos. Especialmente: não execute FindObjectOfType ou Find in Update, FixedUpdate ou mais geralmente em um método chamado um ou mais tempo por quadro.

  • Chame os métodos de tempo de execução FindObjectOfType e Find somente quando necessário
  • FindGameObjectWithTag tem um desempenho muito bom em comparação com outros métodos baseados em strings. O Unity mantém guias separadas em objetos marcados e os consulta em vez de toda a cena.
  • Para GameObjects “estáticos” (como elementos de interface do usuário e prefabs) criados no editor, use [referência de GameObject serializável][1] no editor
  • Mantenha suas listas de GameObjects em List ou Arrays que você mesmo gerencia
  • Em geral, se você instanciar muitos GameObjects do mesmo tipo, dê uma olhada em [Object Pooling][2]
  • Armazene em cache seus resultados de pesquisa para evitar executar os métodos de pesquisa caros repetidamente.

Aprofundando #

Além dos métodos que vêm com o Unity, é relativamente fácil projetar seus próprios métodos de busca e coleta.

  • No caso de FindObjectsOfType(), você pode fazer com que seus scripts mantenham uma lista deles mesmos em uma coleção static. É muito mais rápido iterar uma lista pronta de objetos do que pesquisar e inspecionar objetos da cena.

  • Ou crie um script que armazene suas instâncias em um Dicionário baseado em strings, e você terá um sistema de marcação simples que pode ser expandido.

[1]: https://www.wikiod.com/pt/unity3d/encontrando-e-coletando-gameobjects#Inserido em scripts no modo de edição [2]: https://www.wikiod.com/pt/unity3d/agrupamento-de-objetos#Conjunto de objetos

Pesquisando pelo nome do GameObject #

var go = GameObject.Find("NameOfTheObject");
Prós Contras
Fácil de usar O desempenho degrada ao longo do número de objetos de jogo na cena
Strings são referências fracas e suspeitas de erros do usuário

Pesquisando pelas tags do GameObject #

var go = GameObject.FindGameObjectWithTag("Player");
Prós Contras
Possível pesquisar objetos únicos e grupos inteiros Strings são referências fracas e suspeitas de erros do usuário.
Relativamente rápido e eficiente O código não é portátil, pois as tags são codificadas em scripts.

Encontrando GameObjects por scripts MonoBehaviour #

ExampleScript script = GameObject.FindObjectOfType<ExampleScript>();
GameObject go = script.gameObject;

FindObjectOfType() retorna null se nenhum for encontrado.

Prós Contras
Fortemente tipado O desempenho degrada ao longo do número de objetos de jogo necessários para avaliar
Possível pesquisar objetos únicos e grupos inteiros

Inserido em scripts no modo de edição #

[SerializeField]
GameObject[] gameObjects;
Prós Contras
Grande desempenho A coleção de objetos é estática
Código portátil Só pode se referir a GameObjects da mesma cena

Encontre GameObjects pelo nome de objetos filhos #

Transform tr = GetComponent<Transform>().Find("NameOfTheObject");
GameObject go = tr.gameObject;

Find retorna null se nenhum for encontrado

Prós Contras
Escopo de pesquisa limitado e bem definido Strings são referências fracas
