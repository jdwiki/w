Sintaxe #

  • public Coroutine StartCoroutine(rotina IEnumerator);
  • public Coroutine StartCoroutine(string methodName, object value = null);
  • public void StopCoroutine(string methodName);
  • public void StopCoroutine(rotina IEnumerator);
  • public void StopAllCoroutines();

Considerações de desempenho #

É melhor usar corrotinas com moderação, pois a flexibilidade vem com um custo de desempenho.

  • Corrotinas em grande número exigem mais da CPU do que os métodos padrão de atualização.
  • Há um problema em algumas versões do Unity onde as corrotinas produzem lixo a cada ciclo de atualização devido ao Unity boxing o valor de retorno MoveNext. Isso foi observado pela última vez em 5.4.0b13. (Relato de bug)

Reduza o lixo armazenando em cache YieldInstructions #

Um truque comum para reduzir o lixo gerado em corrotinas é armazenar em cache o YieldInstruction.

IEnumerator TickEverySecond()
{
    var wait = new WaitForSeconds(1f); // Cache
    while(true)
    {
        yield return wait; // Reuse
    }
}

Produzir null não produz lixo extra.

Corrotinas #

Primeiro, é essencial entender que os mecanismos de jogo (como o Unity) funcionam em um paradigma “baseado em quadros”.

O código é executado durante cada quadro.

Isso inclui o próprio código do Unity e seu código.

Ao pensar em frames, é importante entender que não há absolutamente nenhuma garantia de quando os frames acontecem. Eles não acontecem em uma batida regular. Os intervalos entre os quadros podem ser, por exemplo, 0,02632, depois 0,021167, depois 0,029778 e assim por diante. No exemplo, eles são todos “cerca de” 1/50 de segundo, mas são todos diferentes. E a qualquer momento, você pode obter um quadro que demora muito mais ou menos; e seu código pode ser executado a qualquer momento dentro do quadro.

Tendo isso em mente, você pode perguntar: como você acessa esses frames no seu código, no Unity?

Muito simplesmente, você usa a chamada Update() ou usa uma corrotina. (De fato - eles são exatamente a mesma coisa: eles permitem que o código seja executado em cada quadro.)

O propósito de uma corrotina é que:

você pode executar algum código e, em seguida, “pare e espere” até algum quadro futuro.

Você pode esperar até o próximo quadro, esperar um número de quadros ou esperar algum tempo aproximado em segundos no futuro.

Por exemplo, você pode esperar por “cerca de um segundo”, o que significa que esperará cerca de um segundo e, em seguida, colocará seu código em algum quadro cerca de um segundo a partir de agora. (E, de fato, dentro desse quadro, o código pode ser executado a qualquer momento.) Repetindo: não será exatamente um segundo. O tempo preciso não tem sentido em um mecanismo de jogo.

Dentro de uma corrotina:

Para esperar um frame:

// do something
yield return null;  // wait until next frame
// do something

Para esperar três quadros:

// do something
yield return null;  // wait until three frames from now
yield return null;
yield return null;
// do something

Para esperar aproximadamente meio segundo:

// do something
yield return new WaitForSeconds (0.5f); // wait for a frame in about .5 seconds
// do something

Faça algo a cada frame:

while (true)
{
    // do something
    yield return null;  // wait until the next frame
}

Esse exemplo é literalmente idêntico a simplesmente colocar algo dentro da chamada “Update” do Unity: o código em “faça alguma coisa” é executado a cada quadro.

Exemplo #

Anexe o Ticker a um GameObject. Enquanto esse objeto do jogo estiver ativo, o tick será executado. Observe que o script interrompe cuidadosamente a corrotina, quando o objeto do jogo fica inativo; este é geralmente um aspecto importante da engenharia correta do uso da corrotina.

using UnityEngine;
using System.Collections;

public class Ticker:MonoBehaviour {

    void OnEnable()
    {
        StartCoroutine(TickEverySecond());
    }

    void OnDisable()
    {
        StopAllCoroutines();
    }

    IEnumerator TickEverySecond()
    {
        var wait = new WaitForSeconds(1f); // REMEMBER: IT IS ONLY APPROXIMATE
        while(true)
        {
            Debug.Log("Tick");
            yield return wait;  // wait for a frame, about 1 second from now
        }
    }
}

Corrotinas de encadeamento #

Corrotinas podem ceder dentro de si e esperar por outras corrotinas.

Assim, você pode encadear sequências - “uma após a outra”.

Isso é muito fácil e é uma técnica básica e central no Unity.

É absolutamente natural nos jogos que certas coisas tenham que acontecer “em ordem”. Quase todas as “rodadas” de um jogo começam com uma certa série de eventos acontecendo, ao longo de um espaço de tempo, em alguma ordem. Veja como você pode começar um jogo de corrida de carros:

IEnumerator BeginRace()
{
  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());
}

Então, quando você chama BeginRace…

 StartCoroutine(BeginRace());

Ele executará sua rotina de “preparar corrida”. (Talvez, piscando algumas luzes e fazendo barulho de multidão, redefinindo pontuações e assim por diante.) Quando isso terminar, ele executará sua sequência de “contagem regressiva”, onde você animaria talvez uma contagem regressiva na interface do usuário. Quando isso terminar, ele executará seu código de largada de corrida, onde você talvez executaria efeitos sonoros, iniciaria alguns drivers de IA, moveria a câmera de uma determinada maneira e assim por diante.

Para maior clareza, entenda que as três chamadas

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

eles mesmos devem estar em uma corrotina. Ou seja, eles devem estar em uma função do tipo IEnumerator. Então, em nosso exemplo, isso é IEnumerator BeginRace. Então, a partir do código “normal”, você inicia essa corrotina com a chamada StartCoroutine.

 StartCoroutine(BeginRace());

Para entender melhor o encadeamento, aqui está uma função que encadeia corrotinas. Você passa em uma matriz de corrotinas. A função executa quantas corrotinas você passar, em ordem, uma após a outra.

// run various routines, one after the other
IEnumerator OneAfterTheOther( params IEnumerator[] routines ) 
{
    foreach ( var item in routines ) 
    {
        while ( item.MoveNext() ) yield return item.Current;
    }

    yield break;
}

Veja como você chamaria isso… digamos que você tenha três funções. Lembre-se de que todos devem ser IEnumerator:

IEnumerator PrepareRace() 
{
    // codesay, crowd cheering and camera pan around the stadium
    yield break;
}

IEnumerator Countdown() 
{
    // codesay, animate your countdown on UI
    yield break;
}

IEnumerator StartRace() 
{
    // codesay, camera moves and light changes and launch the AIs
    yield break;
}

Você chamaria assim

StartCoroutine( MultipleRoutines( PrepareRace(), Countdown(), StartRace() ) );

ou talvez assim

IEnumerator[] routines = new IEnumerator[] {
     PrepareRace(),
     Countdown(),
     StartRace() };
StartCoroutine( MultipleRoutines( routines ) );

Repetindo, um dos requisitos mais básicos nos jogos é que certas coisas aconteçam uma após a outra “em uma sequência” ao longo do tempo. Você consegue isso no Unity de forma muito simples, com

  yield return StartCoroutine(PrepareRace());
  yield return StartCoroutine(Countdown());
  yield return StartCoroutine(StartRace());

Finalizando uma corrotina #

Muitas vezes você projeta corrotinas para terminar naturalmente quando certos objetivos são alcançados.

IEnumerator TickFiveSeconds()
{
    var wait = new WaitForSeconds(1f);
    int counter = 1;
    while(counter < 5)
    {
        Debug.Log("Tick");
        counter++;
        yield return wait;
    }
    Debug.Log("I am done ticking");
}

Para impedir uma corrotina de “dentro” da corrotina, você não pode simplesmente “retornar” como faria para sair mais cedo de uma função comum. Em vez disso, você usa yield break.

IEnumerator ShowExplosions()
{
    ... show basic explosions
    if(player.xp < 100) yield break;
    ... show fancy explosions
}

Você também pode forçar todas as corrotinas iniciadas pelo script a parar antes de terminar.

void OnDisable()
{
    // Stops all running coroutines
    StopAllCoroutines();
}

O método para interromper uma corrotina específica do chamador varia dependendo de como você a iniciou.

Se você iniciou uma corrotina pelo nome da string:

StartCoroutine("YourAnimation");

então você pode pará-lo chamando StopCoroutine com o mesmo nome de string:

StopCoroutine("YourAnimation");

Alternativamente, você pode manter uma referência para ou o IEnumerator retornado pelo método coroutine, ou o objeto Coroutine retornado por StartCoroutine, e chamar StopCoroutine em qualquer um deles:

public class SomeComponent : MonoBehaviour 
{
    Coroutine routine;

    void Start () {
        routine = StartCoroutine(YourAnimation());
    }

    void Update () {
        // later, in response to some input...
        StopCoroutine(routine);
    }

    IEnumerator YourAnimation () { /* ... */ }
}

Métodos MonoBehaviour que podem ser Coroutines #

Existem três métodos MonoBehaviour que podem ser transformados em corrotinas.

  1. Iniciar()
  2. OnTornou-se Visível()
  3. OnLevelWasLoaded()

Isso pode ser usado para criar, por exemplo, scripts que são executados apenas quando o objeto está visível para uma câmera.

using UnityEngine;
using System.Collections;

public class RotateObject : MonoBehaviour
{
    IEnumerator OnBecameVisible()
    {
        var tr = GetComponent<Transform>();
        while (true)
        {
            tr.Rotate(new Vector3(0, 180f * Time.deltaTime));
            yield return null;
        }
    }
    
    void OnBecameInvisible()
    {
        StopAllCoroutines();
    }
}

Maneiras de render #

Você pode esperar até o próximo quadro.

yield return null; // wait until sometime in the next frame

Você pode ter várias dessas chamadas seguidas, para simplesmente aguardar quantos quadros desejar.

//wait for a few frames
yield return null;
yield return null;

Aguarde aproximadamente n segundos. É extremamente importante entender que este é apenas um tempo muito aproximado.

yield return new WaitForSeconds(n);

Não é absolutamente possível usar a chamada “WaitForSeconds” para qualquer forma de tempo preciso.

Muitas vezes você quer encadear ações. Então, faça alguma coisa, e quando terminar, faça outra coisa, e quando terminar, faça outra coisa. Para conseguir isso, espere por outra corrotina:

yield return StartCoroutine(coroutine);

Entenda que você só pode chamar isso de dentro de uma corrotina. Então:

StartCoroutine(Test());

É assim que você inicia uma corrotina a partir de um pedaço de código “normal”.

Então, dentro dessa corrotina em execução:

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

Isso imprimirá A, iniciará o processo longo e imprimirá imediatamente B. Ele não esperará que o longo processo termine. Por outro lado:

Debug.Log("A");
yield return StartCoroutine(LongProcess());
Debug.Log("B");

Isso imprimirá A, iniciará o processo longo, espere até que seja concluído e, em seguida, imprima B.

Sempre vale lembrar que as corrotinas não têm absolutamente nenhuma conexão, de forma alguma, com o encadeamento. Com este código:

Debug.Log("A");
StartCoroutine(LongProcess());
Debug.Log("B");

é fácil pensar nisso como sendo “como” iniciar o LongProcess em outro thread em segundo plano. Mas isso é absolutamente incorreto. É apenas uma corrotina. Os mecanismos de jogo são baseados em quadros e as “corrotinas” no Unity simplesmente permitem que você acesse os quadros.

É muito fácil esperar que uma solicitação da web seja concluída.

void Start() {
    string url = "http://google.com";
    WWW www = new WWW(url);
    StartCoroutine(WaitForRequest(www));
}

IEnumerator WaitForRequest(WWW www) {
    yield return www;
    
    if (www.error == null) {
        //use www.data);
    }
    else {
        //use www.error);
    }
}

Para completar: Em casos muito raros, você usa atualização fixa no Unity; existe uma chamada WaitForFixedUpdate() que normalmente nunca seria usada. Existe uma chamada específica (WaitForEndOfFrame() na versão atual do Unity) que é usada em determinadas situações em relação à geração de capturas de tela durante o desenvolvimento. (O mecanismo exato muda um pouco à medida que o Unity evolui, portanto, pesquise no Google para obter as informações mais recentes, se relevante.)
