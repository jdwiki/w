Poder de Corrotina #

Uso #

Se você tiver uma operação de longa duração que depende da API Unity não thread-safe, use Coroutines para dividi-la em vários quadros e manter seu aplicativo responsivo.

Corrotinas também ajudam a executar ações caras a cada enésimo quadro em vez de executar essa ação a cada quadro.

Dividindo rotinas de longa duração em vários quadros ================================================== == As corrotinas ajudam a distribuir operações de longa duração em vários quadros para ajudar a manter a taxa de quadros do seu aplicativo.

Rotinas que pintam ou geram terreno processualmente ou geram ruído são exemplos que podem necessitar do tratamento Coroutine.

for (int y = 0; y < heightmap.Height; y++) 
{
    for (int x = 0; x < heightmap.Width; x++)
    {
        // Generate pixel at (x, y)
        // Assign pixel at (x, y)
        
        // Process only 32768 pixels each frame
        if ((y * heightmap.Height + x) % 32 * 1024) == 0)
            yield return null; // Wait for next frame
    }
}

O código acima é um exemplo fácil de entender. No código de produção, é melhor evitar a verificação por pixel que verifica quando ‘yield return’ (talvez faça isso a cada 2-3 linhas) e pré-calcular o comprimento do loop ‘for’ antecipadamente.

Executando ações caras com menos frequência #

As corrotinas ajudam você a executar ações caras com menos frequência, para que não seja um impacto tão grande no desempenho quanto seria se fosse executado a cada quadro.

Tomando o seguinte exemplo diretamente do Manual:

private void ProximityCheck() 
{
    for (int i = 0; i < enemies.Length; i++) 
    {
        if (Vector3.Distance(transform.position, enemies[i].transform.position) < dangerDistance)
                return true;
    }
    return false;
}

private IEnumerator ProximityCheckCoroutine() 
{
    while(true) 
    {
        ProximityCheck();
        yield return new WaitForSeconds(.1f);
    }
}

Os testes de proximidade podem ser otimizados ainda mais usando a CullingGroup API.

Armadilhas comuns #

Um erro comum que os desenvolvedores cometem é acessar resultados ou efeitos colaterais de corrotinas fora da corrotina. Coroutines retornam o controle para o chamador assim que uma instrução yield return é encontrada e o resultado ou efeito colateral ainda não pode ser executado. Para contornar problemas onde você tem que usar o resultado/efeito colateral fora da corrotina, verifique esta resposta.

Cordas #

Pode-se argumentar que existem maiores porcos de recursos no Unity do que a humilde string, mas é um dos aspectos mais fáceis de corrigir desde o início.

Operações de string criam lixo #

A maioria das operações de string cria pequenas quantidades de lixo, mas se essas operações forem chamadas várias vezes ao longo de uma única atualização, elas se acumulam. Com o tempo, ele acionará a coleta automática de lixo, o que pode resultar em um pico de CPU visível.

Armazene em cache suas operações de string #

Considere o exemplo a seguir.

string[] StringKeys = new string[] {
    "Key0",
    "Key1",
    "Key2"
};

void Update()
{
    for (var i = 0; i < 3; i++)
    {
        // Cached, no garbage generated
        Debug.Log(StringKeys[i]);
    }

    for (var i = 0; i < 3; i++)
    {
        // Not cached, garbage every cycle
        Debug.Log("Key" + i);
    }

    // The most memory-efficient way is to not create a cache at all and use literals or constants.
    // However, it is not necessarily the most readable or beautiful way.
    Debug.Log("Key0");
    Debug.Log("Key1");
    Debug.Log("Key2");
}

Pode parecer bobo e redundante, mas se você estiver trabalhando com Shaders, poderá se deparar com situações como essas. O armazenamento em cache das chaves fará a diferença.

Observe que string literals e constants não geram lixo, pois são injetados estaticamente no espaço da pilha do programa. Se você está gerando strings em tempo de execução e tem garantia de gerar as mesmas strings todas as vezes, como no exemplo acima, o armazenamento em cache definitivamente ajudará.

Para outros casos em que a string gerada não é sempre a mesma, não há outra alternativa para gerar essas strings. Como tal, o pico de memória com strings geradas manualmente a cada vez geralmente é insignificante, a menos que dezenas de milhares de strings estejam sendo geradas por vez.

A maioria das operações de string são mensagens de depuração #

Fazendo operações de string para mensagens de depuração, ou seja. Debug.Log("Nome do objeto: " + obj.name) é bom e não pode ser evitado durante o desenvolvimento. No entanto, é importante garantir que mensagens de depuração irrelevantes não acabem no produto lançado.

Uma maneira é usar o Atributo condicional em suas chamadas de depuração. Isso não apenas remove as chamadas de método, mas também todas as operações de string que entram nele.

using UnityEngine;
using System.Collections;

public class ConditionalDebugExample: MonoBehaviour
{
    IEnumerator Start()
    {
        while(true)
        {
            // This message will pop up in Editor but not in builds
            Log("Elapsed: " + Time.timeSinceLevelLoad);
            yield return new WaitForSeconds(1f);
        }
    }

    [System.Diagnostics.Conditional("UNITY_EDITOR")]
    void Log(string Message)
    {
        Debug.Log(Message);
    }
}

Este é um exemplo simplificado. Você pode querer investir algum tempo projetando uma rotina de registro mais completa.

Comparação de strings #

Esta é uma pequena otimização, mas vale a pena mencionar. Comparar strings é um pouco mais complicado do que se imagina. O sistema tentará levar em consideração as diferenças culturais por padrão. Você pode optar por usar uma comparação binária simples, que é mais rápida.

// Faster string comparison
if (strA.Equals(strB, System.StringComparison.Ordinal)) {...}
// Compared to
if (strA == strB) {...}

// Less overhead
if (!string.IsNullOrEmpty(strA)) {...}
// Compared to
if (strA == "") {...}

// Faster lookups
Dictionary<string, int> myDic = new Dictionary<string, int>(System.StringComparer.Ordinal);
// Compared to
Dictionary<string, int> myDictionary = new Dictionary<string, int>();

Verificações rápidas e eficientes #

Evite operações desnecessárias e chamadas de métodos sempre que puder, especialmente em um método que é chamado muitas vezes por segundo, como Update.

Verificações de distância/alcance #

Use sqrMagnitude em vez de magnitude ao comparar distâncias. Isso evita operações sqrt desnecessárias. Observe que ao usar sqrMagnitude, o lado direito também deve ser quadrado.

if ((target.position - transform.position).sqrMagnitude < minDistance * minDistance))

Verificações de limites #

As interseções de objetos podem ser verificadas grosseiramente verificando se seus limites Collider/Renderer se cruzam. A estrutura Bounds também tem um método útil Intersects que ajuda a determinar se dois limites se cruzam.

Bounds também nos ajuda a calcular uma aproximação aproximada da distância real (superfície a superfície) entre objetos (veja Bounds.SqrDistance).

Ressalvas #

A verificação de limites funciona muito bem para objetos convexos, mas verificações de limites em objetos côncavos podem levar a imprecisões muito maiores, dependendo da forma do objeto.

O uso de Mesh.bounds não é recomendado, pois retorna limites de espaço local. Use MeshRenderer.bounds em vez disso.

Evite métodos de unidade vazios #

Evite métodos de unidade vazios. Além de ser um estilo de programação ruim, há uma sobrecarga muito pequena envolvida no script de tempo de execução. Em muitos casos, isso pode aumentar e afetar o desempenho.

void Update
{
}

void FixedUpdate
{
}

Referências de cache #

Referências de cache para evitar chamadas caras, especialmente na função de atualização. Isso pode ser feito armazenando em cache essas referências no início, se disponíveis, ou quando disponíveis, e verificando se há null/bool flat para evitar obter a referência novamente.

Exemplos:

Referências de componentes de cache

mudança

void Update()
{
    var renderer = GetComponent<Renderer>();
    renderer.material.SetColor("_Color", Color.green);
}

para

private Renderer myRenderer;
void Start()
{
    myRenderer = GetComponent<Renderer>();
}

void Update()
{
    myRenderer.material.SetColor("_Color", Color.green);
}

Referências de objetos de cache

mudança

void Update()
{
    var enemy = GameObject.Find("enemy");
    enemy.transform.LookAt(new Vector3(0,0,0));
}

para

private Transform enemy;

void Start()
{
    this.enemy = GameObject.Find("enemy").transform;
}

void Update()
{
    enemy.LookAt(new Vector3(0, 0, 0));
}

Além disso, armazenar em cache chamadas caras, como chamadas para Mathf, sempre que possível.

Evite chamar métodos usando strings #

Evite chamar métodos usando strings que podem aceitar métodos. Essa abordagem fará uso de reflexão que pode tornar seu jogo mais lento, especialmente quando usado na função de atualização.

Exemplos:

    //Avoid StartCoroutine with method name
    this.StartCoroutine("SampleCoroutine");

    //Instead use the method directly
    this.StartCoroutine(this.SampleCoroutine());

    //Avoid send message
    var enemy = GameObject.Find("enemy");
    enemy.SendMessage("Die");

    //Instead make direct call
    var enemy = GameObject.Find("enemy") as Enemy;
    enemy.Die();
