Padrão de implementação para proxies de inicialização #

Este exemplo mostra como implementar classes de proxy para seu aplicativo Minecraft Mod, que são usadas para inicializar seu mod.

Antes de tudo, você precisará implementar a classe base CommonProxy.java que contém os 3 métodos usados ​​principalmente:

public class CommonProxy {
    public void preInit(FMLPreInitializationEvent e) {}
    public void init(FMLInitializationEvent e) {}
    public void postInit(FMLPostInitializationEvent e) {}
}

Normalmente seu mod tem 2 pacotes diferentes para Client e Server Code, então você precisará em cada pacote de uma classe filha de CommonProxy.java como:

public class ClientProxy extends CommonProxy {    
    @Override
    public void preInit(FMLPreInitializationEvent e) {
        super.preInit(e);
    }

    @Override
    public void init(FMLInitializationEvent e) {
        super.init(e);    
    }

    @Override
    public void postInit(FMLPostInitializationEvent e) {
        super.postInit(e);
    }
}

e para o servidor:

public class ServerProxy extends CommonProxy {
    @Override
    public void preInit(FMLPreInitializationEvent e) {
        super.preInit(e);
    }

    @Override
    public void init(FMLInitializationEvent e) {
        super.init(e);
    }

    @Override
    public void postInit(FMLPostInitializationEvent e) {
        super.postInit(e);
    }
}

Depois de criar essas classes, você pode estendê-las por métodos que devem ser executados apenas no lado do cliente ou do servidor, mas também pode anexá-las a ambos se você chamar os métodos na classe ‘base’.

Por fim, você precisa definir qual proxy é usado em tempo de execução. Você precisa estender sua classe mod principal com a anotação @Mod, por:

private static final String CLIENTPROXY = "com.yourpackage.client.ClientProxy";
private static final String SERVERPROXY = "com.yourpackage.client.ServerProxy";    

@SidedProxy(clientSide = CLIENTPROXY, serverSide = SERVERPROXY)
public static CommonProxy PROXY;

Isso permitirá que o Forge detecte qual classe deve ser tomada em tempo de execução. Nos métodos de inicialização do seu Mod agora você pode usar essa propriedade PROXY estática.

@EventHandler
public void init(FMLInitializationEvent event) {
    PROXY.init(event);
}

@Mod.EventHandler
public void preInit(FMLPreInitializationEvent event) {
    PROXY.preInit(event);
}

@EventHandler
public void postInit(FMLPostInitializationEvent event) {
    PROXY.postInit(event);
}

Adicionando sons personalizados ao seu MOD #

Este exemplo mostra como você adiciona novos sons ao seu MOD e os reproduz. Primeiro você precisa de um arquivo de som que tenha o formato *.ogg. Qualquer outro formato não é permitido pelo aplicativo Minecraft e será rejeitado.

O arquivo de som tem o nome: sound1.ogg

Coloque o arquivo de som no seguinte caminho:

/YourPath/src/main/resources/assets/MODID/sounds/sound1.ogg

Substitua ‘MODID’ pelo identificador que você definiu para o seu MOD

Em seguida você tem que criar um sounds.json em codificação UTF-8 (Padrão) que define nome, recurso,… e outras coisas para seu som personalizado. Este arquivo ficará assim:

{
  "sound1": {
    "category" : "player",
    "sounds": [{
        "name": "MODID:sound1",
        "stream": false
      }
    ]
  },
  "sound2": {
    "category" : "ambient",
    "sounds": [{
        "name": "MODID:subfolder/sound2",
        "stream": true
      }
    ]
  }
}

Como explicação para este sons.json.

Existem definidos 2 sons definidos, pois adicionei um exemplo que você pode investigar como adicionar sons de multiplicação. sound1 tem a categoria player, a segunda é da categoria ambient, o que significa que o volume do som é afetado pelas configurações de volume que o usuário definiu para os sons do player/ambiente. name é a propriedade mais importante, pois está apontando para o recurso do som. O MODID é o identificador do seu MOD e é obrigatório pois o Aplicativo irá buscar nos recursos do seu mod pelo arquivo, caso contrário ele irá buscar nos recursos do Minecraft e não encontrará nada. A propriedade stream significa que o som será transmitido do sistema de arquivos, o que só é necessário para sons com mais de 4 segundos.

Seu arquivo sounds.json personalizado deve seguir o seguinte caminho:

/YourPath/src/main/resources/assets/MODID/sounds.json

Agora você poderá carregar os sons no registro do jogo. Então você tem que criar uma classe que está inicializando SoundEvents e manipulando o registro.

public class SoundRegistrator {
    public static final SoundEvent SOUND_1;
    public static final SoundEvent SOUND_2;

    static {
        SOUND_1 = addSoundsToRegistry("sound1");
        SOUND_2 = addSoundsToRegistry("sound2");
    }

    private static SoundEvent addSoundsToRegistry(String soundId) {
        ResourceLocation shotSoundLocation = new ResourceLocation("MODID", soundId);
        SoundEvent soundEvent = new SoundEvent(shotSoundLocation);
        soundEvent.setRegistryName(shotSoundLocation);
        return soundEvent;
    }           
}

Depois você tem que criar um SoundRegisterListener:

public class SoundRegisterListener {
        @SubscribeEvent(priority = EventPriority.NORMAL, receiveCanceled = true)
        public void registerSoundEvents(RegistryEvent.Register<SoundEvent> event) {
           event.getRegistry().registerAll(SoundRegistrator.SOUND_1,SoundRegistrator.SOUND_2);
        }   
}

e registre-o no MinecraftForge.EVENT_BUS como:

MinecraftForge.EVENT_BUS.register(new SoundRegisterListener());

Finalmente, você poderá reproduzir seus sons:

void playSound(SoundEvent sound) {
    try {
        if (Minecraft.getMinecraft().world.isRemote) {
            EntityPlayerSP player = Minecraft.getMinecraft().player;
            Minecraft.getMinecraft().world.playSound(player, player.getPosition(), sound, SoundCategory.PLAYERS, RandomGenerator.getNextRandomVolumeLoud(), 1.0F);
        }
    } catch (Exception ex) {
       //Error happened
    }
}
