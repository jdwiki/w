Introdução à programação dinâmica #

Programação dinâmica resolve problemas combinando as soluções para subproblemas. Pode ser análogo ao método de dividir e conquistar, onde o problema é particionado em subproblemas disjuntos, os subproblemas são resolvidos recursivamente e depois combinados para encontrar a solução do problema original. Em contraste, a programação dinâmica se aplica quando os subproblemas se sobrepõem - ou seja, quando os subproblemas compartilham subsubproblemas. Nesse contexto, um algoritmo de divisão e conquista faz mais trabalho do que o necessário, resolvendo repetidamente os subsubproblemas comuns. Um algoritmo de programação dinâmica resolve cada subsubproblema apenas uma vez e, em seguida, salva sua resposta em uma tabela, evitando assim o trabalho de recalcular a resposta toda vez que resolver cada subsubproblema.

Vejamos um exemplo. O matemático italiano Leonardo Pisano Bigollo, que comumente conhecemos como Fibonacci, descobriu uma série numérica considerando o [crescimento idealizado da população de coelhos](https://en .wikipedia.org/wiki/Fibonacci_number#Origins). A série é:

1, 1, 2, 3, 5, 8, 13, 21, ......

Podemos notar que todo número após os dois primeiros é a soma dos dois números anteriores. Agora, vamos formular uma função F(n) que nos retornará o enésimo número de Fibonacci, ou seja,

F(n) = nth Fibonacci Number

Até agora, sabemos que,

F(1) = 1
F(2) = 1
F(3) = F(2) + F(1) = 2
F(4) = F(3) + F(2) = 3

Podemos generalizar por:

F(1) = 1
F(2) = 1
F(n) = F(n-1) + F(n-2)

Agora, se quisermos escrevê-la como uma função recursiva, temos F(1) e F(2) como nosso caso base. Então nossa função de Fibonacci seria:

Procedure F(n):                //A function to return nth Fibonacci Number
if n is equal to 1
    Return 1
else if n is equal to 2
    Return 1
end if
Return F(n-1) + F(n-2)

Agora, se chamarmos F(6), ele chamará F(5) e F(4), que chamará um pouco mais. Vamos representar isso graficamente:

A partir da imagem, podemos ver que F(6) chamará F(5) e F(4). Agora F(5) chamará F(4) e F(3). Após calcular F(5), podemos dizer com certeza que todas as funções que foram chamadas por F(5) já foram calculadas. Isso significa que já calculamos F(4). Mas estamos novamente calculando F(4) como o filho direito de F(6) indica. Realmente precisamos recalcular? O que podemos fazer é, uma vez calculado o valor de F(4), vamos armazená-lo em um array chamado dp e reutilizá-lo quando necessário. Inicializaremos nosso array dp com -1 (ou qualquer valor que não venha em nosso cálculo). Em seguida, chamaremos F(6) onde nosso F(n) modificado ficará assim:

Procedure F(n):
if n is equal to 1
    Return 1
else if n is equal to 2
    Return 1
else if dp[n] is not equal to -1            //That means we have already calculated dp[n]
    Return dp[n]
else
    dp[n] = F(n-1) + F(n-2)
    Return dp[n]
end if

Fizemos a mesma tarefa de antes, mas com uma otimização simples. Ou seja, usamos a técnica de memoização. A princípio, todos os valores do array dp serão -1. Quando F(4) é chamado, verificamos se está vazio ou não. Se ele armazenar -1, calcularemos seu valor e o armazenaremos em dp4. Se ele armazenar qualquer coisa menos -1, isso significa que já calculamos seu valor. Então, vamos simplesmente retornar o valor.

Essa otimização simples usando memoização é chamada de Programação Dinâmica.

Um problema pode ser resolvido usando Programação Dinâmica se tiver algumas características. Estes são:

  • Subproblemas:
    Um problema de DP pode ser dividido em um ou mais subproblemas. Por exemplo: F(4) pode ser dividido em subproblemas menores F(3) e F(2). Como os subproblemas são semelhantes ao nosso problema principal, eles podem ser resolvidos usando a mesma técnica.
  • Subproblemas sobrepostos:
    Um problema de DP deve ter subproblemas sobrepostos. Isso significa que deve haver alguma parte comum para a qual a mesma função é chamada mais de uma vez. Por exemplo: F(5) e F(6) tem F(3) e F(4) em comum. Esta é a razão pela qual armazenamos os valores em nosso array.

  • Subestrutura ideal:
    Digamos que você seja solicitado a minimizar a função g(x). Você sabe que o valor de g(x) depende de g(y) e g(z). Agora, se pudermos minimizar g(x) minimizando tanto g(y) quanto g(z), somente então podemos dizer que o problema tem subestrutura ótima. Se g(x) é minimizado apenas minimizando g(y) e se minimizar ou maximizar g(z) não tem nenhum efeito em g(x), então este problema não tem subestrutura ótima. Em palavras simples, se a solução ótima de um problema pode ser encontrada a partir da solução ótima de seu subproblema, então podemos dizer que o problema tem propriedade de subestrutura ótima.

Entendendo o estado na programação dinâmica #

Vamos discutir com um exemplo. De n itens, de quantas maneiras você pode escolher r itens? Você sabe que é indicado por . Agora pense em um único item.

  • Se você selecionar o item, depois disso você terá que tirar r-1 itens dos itens restantes n-1, que é dado por .

Se pensarmos nCr(n,r) como uma função que recebe n e r como parâmetro e determina , podemos escrever a relação mencionada acima como:

nCr(n,r) = nCr(n-1,r) + nCr(n-1,r-1)

Esta é uma relação recursiva. Para terminá-lo, precisamos determinar o(s) caso(s) base(s). Sabemos que, e . Usando esses dois como casos base, nosso algoritmo para determinar será:

Procedure nCr(n, r):
if r is equal to 1
    Return n
else if n is equal to r
    Return 1
end if
Return nCr(n-1,r) + nCr(n-1,r-1)

Podemos evitar essa chamada repetida usando um array, dp. Como existem 2 parâmetros, precisaremos de um array 2D. Inicializaremos o array dp com -1, onde -1 indica que o valor ainda não foi calculado. Nosso procedimento será:

Procedure nCr(n,r):
if r is equal to 1
    Return n
else if n is equal to r
    Return 1
else if dp[n][r] is not equal to -1        //The value has been calculated
    Return dp[n][r]
end if
dp[n][r] := nCr(n-1,r) + nCr(n-1,r-1)
Return dp[n][r]

Para determinar , precisamos de dois parâmetros n e r. Esses parâmetros são chamados de estado. Podemos simplesmente deduzir que o número de estados determina o número de dimensão do array dp. O tamanho do array mudará de acordo com a nossa necessidade. Nossos algoritmos de programação dinâmica manterão o seguinte padrão geral:

Procedure DP-Function(state_1, state_2, ...., state_n)
Return if reached any base case
Check array and Return if the value is already calculated.
Calculate the value recursively for this state
Save the value in the table and Return

Determinar o estado é uma das partes mais importantes da programação dinâmica. Consiste no número de parâmetros que definem nosso problema e otimizando seu cálculo, podemos otimizar todo o problema.

Construindo uma solução de DP #

Não importa quantos problemas você resolva usando programação dinâmica (DP), ela ainda pode surpreendê-lo. Mas, como tudo na vida, a prática o torna melhor. Tendo isso em mente, veremos o processo de construção de uma solução para problemas de DP. Outros exemplos neste tópico ajudarão você a entender o que é DP e como ele funciona. Neste exemplo, tentaremos entender como criar uma solução de DP do zero.

Iterativo VS Recursivo:
Existem duas técnicas de construção de solução DP. Eles são:

  • Iterativo (usando ciclos for)
  • Recursivo (usando recursão)

Por exemplo, algoritmo para calcular o comprimento da [Longest Common Subsequence](https://www.wikiod.com/pt/dynamic-programming/algoritmos-relacionados-a-subsequencia#Maior Subsequência Comum t=201611230422107601679) de duas strings s1 e s2 ficariam assim:

Procedure LCSlength(s1, s2):
Table[0][0] = 0
for i from 1 to s1.length
    Table[0][i] = 0
endfor
for i from 1 to s2.length
    Table[i][0] = 0
endfor
for i from 1 to s2.length
    for j from 1 to s1.length
        if s2[i] equals to s1[j]
            Table[i][j] = Table[i-1][j-1] + 1
        else
            Table[i][j] = max(Table[i-1][j], Table[i][j-1])
        endif
    endfor
endfor
Return Table[s2.length][s1.length]

Esta é uma solução iterativa. Existem algumas razões pelas quais ele é codificado dessa maneira:

  • A iteração é mais rápida que a recursão.
  • Determinar a complexidade do tempo e do espaço é mais fácil.
  • O código-fonte é curto e limpo

Olhando para o código-fonte, você pode entender facilmente como e por que ele funciona, mas é difícil entender como chegar a essa solução. No entanto, as duas abordagens mencionadas acima se traduzem em dois pseudocódigos diferentes. Um usa iteração (de baixo para cima) e outro usa a abordagem de recursão (de cima para baixo). Esta última também é conhecida como técnica de memoização. No entanto, as duas soluções são mais ou menos equivalentes e uma pode ser facilmente transformada na outra. Para este exemplo, mostraremos como encontrar uma solução recursiva para um problema.

Exemplo de problema:
Digamos que você tenha N (1, 2, 3, …., N) vinhos colocados lado a lado em uma prateleira. O preço do ith vinho é p[i]. O preço dos vinhos aumenta todos os anos. Suponha que este seja o ano 1, no ano y o preço do ith vinho será: ano * preço do vinho = y*p[i ]. Você quer vender os vinhos que tem, mas precisa vender exatamente um vinho por ano, a partir deste ano. Novamente, a cada ano, você pode vender apenas o vinho mais à esquerda ou mais à direita e não pode reorganizar os vinhos, o que significa que eles devem permanecer na mesma ordem em que estão no início.

Por exemplo: digamos que você tenha 4 vinhos na prateleira e seus preços sejam (da esquerda para a direita):

+---+---+---+---+
| 1 | 4 | 2 | 3 |
+---+---+---+---+

A solução ótima seria vender os vinhos na ordem 1 -> 4 -> 3 -> 2, o que nos dará um lucro total de: [! 11+32+23+44=29]1

Abordagem gananciosa:
Depois de um tempo de brainstorming, você pode encontrar a solução para vender o vinho caro o mais tarde possível. Então sua estratégia gananciosa será:

Every year, sell the cheaper of the two (leftmost and rightmost) available wines.

Embora a estratégia não mencione o que fazer quando os dois vinhos custam o mesmo, a estratégia parece certa. Mas infelizmente não é. Se os preços dos vinhos forem:

+---+---+---+---+---+
| 2 | 3 | 5 | 1 | 4 |
+---+---+---+---+---+

A estratégia gananciosa os venderia na ordem 1 -> 2 -> 5 -> 4 -> 3 para um lucro total de: [! 21+32+43+14+5*5=49]2

Mas podemos fazer melhor se vendermos os vinhos na ordem 1 -> 5 -> 4 -> 2 -> 3 para um lucro total de:

Este exemplo deve convencê-lo de que o problema não é tão fácil quanto parece à primeira vista. Mas pode ser resolvido usando Programação Dinâmica.

Retorno:
Para encontrar a solução de memorização para um problema, encontrar uma solução de retrocesso é útil. A solução Backtrack avalia todas as respostas válidas para o problema e escolhe a melhor. Para a maioria dos problemas, é mais fácil encontrar essa solução. Pode haver três estratégias a seguir na abordagem de uma solução de retrocesso:

  1. deve ser uma função que calcula a resposta usando recursão.
  2. deve retornar a resposta com a instrução return.
  3. todas as variáveis ​​não locais devem ser usadas como somente leitura, ou seja, a função pode modificar apenas variáveis ​​locais e seus argumentos.

Para o nosso problema de exemplo, tentaremos vender o vinho mais à esquerda ou mais à direita e calcular recursivamente a resposta e retornar o melhor. A solução de retrocesso ficaria assim:

// year represents the current year
// [begin, end] represents the interval of the unsold wines on the shelf
Procedure profitDetermination(year, begin, end):
if begin > end                  //there are no more wines left on the shelf
    Return 0
Return max(profitDetermination(year+1, begin+1, end) + year * p[begin], //selling the leftmost item
           profitDetermination(year+1, begin, end+1) + year * p[end])   //selling the rightmost item

Se chamarmos o procedimento usando profitDetermination(1, 0, n-1), onde n é o número total de vinhos, ele retornará a resposta.

Esta solução simplesmente tenta todas as combinações válidas possíveis de venda dos vinhos. Se houver n vinhos no início, ele verificará possibilidades. Mesmo que tenhamos a resposta correta agora, a complexidade de tempo do algoritmo cresce exponencialmente.

A função de retrocesso corretamente escrita deve sempre representar uma resposta a uma pergunta bem formulada. No nosso exemplo, o procedimento profitDetermination representa uma resposta à pergunta: Qual é o melhor lucro que podemos obter vendendo os vinhos com preços armazenados na matriz p, quando o ano atual é ano e o intervalo de vinhos não vendidos abrange por meio de [início, fim], inclusive? Você deve sempre tentar criar essa pergunta para sua função de retrocesso para ver se você acertou e entender exatamente o que ela faz.

Estado determinado:
State é o número de parâmetros usados ​​na solução DP. Nesta etapa, precisamos pensar em quais dos argumentos que você passa para a função são redundantes, ou seja, podemos construí-los a partir de outros argumentos ou não precisamos deles. Se houver tais argumentos, não precisamos passá-los para a função, vamos calculá-los dentro da função.

Na função de exemplo profitDetermination mostrada acima, o argumento year é redundante. É equivalente ao número de vinhos que já vendemos mais um. Pode ser determinado usando o número total de vinhos desde o início menos o número de vinhos que não vendemos mais um. Se armazenarmos o número total de vinhos n como uma variável global, podemos reescrever a função como:

Procedure profitDetermination(begin, end):
if begin > end
    Return 0
year := n - (end-begin+1) + 1        //as described above
Return max(profitDetermination(begin+1, end) + year * p[begin],
           profitDetermination(begin, end+1) + year * p[end])

Também precisamos pensar no intervalo de valores possíveis dos parâmetros que podem ser obtidos de uma entrada válida. Em nosso exemplo, cada um dos parâmetros begin e end pode conter valores de 0 a n-1. Em uma entrada válida, também esperamos begin <= end + 1. Pode haver argumentos diferentes O(n²) com os quais nossa função pode ser chamada.

Memoização:
Agora estamos quase terminando. Para transformar a solução de retrocesso com complexidade de tempo em solução de memoização com complexidade de tempo , nós usará um pequeno truque que não requer muito esforço.

Como observado acima, existem apenas parâmetros diferentes com os quais nossa função pode ser chamada. Em outras palavras, existem apenas coisas diferentes que podemos realmente calcular. Então, de onde vem a complexidade de tempo e o que ela computa!!

A resposta é: a complexidade de tempo exponencial vem da recursão repetida e, por causa disso, calcula os mesmos valores repetidamente. Se você executar o código mencionado acima para uma matriz arbitrária de n = 20 vinhos e calcular quantas vezes a função foi chamada para argumentos begin = 10 e **end ** = 10, você receberá um número 92378. Isso é uma enorme perda de tempo para calcular a mesma resposta tantas vezes. O que podemos fazer para melhorar isso é armazenar os valores depois de computados e toda vez que a função solicitar um valor já calculado, não precisamos executar toda a recursão novamente. Teremos um array dp[N][N], inicialize-o com -1 (ou qualquer valor que não venha em nosso cálculo), onde -1 denota o valor ainda não foi calculado. O tamanho da matriz é determinado pelo valor máximo possível de begin e end, pois armazenaremos os valores correspondentes de determinados valores begin e end em nossa matriz. Nosso procedimento ficaria assim:

Procedure profitDetermination(begin, end):
if begin > end
    Return 0
if dp[begin][end] is not equal to -1                //Already calculated
    Return dp[begin][end]
year := n - (end-begin+1) + 1
dp[begin][end] := max(profitDetermination(year+1, begin+1, end) + year * p[begin],
           profitDetermination(year+1, begin, end+1) + year * p[end])
Return dp[begin][end]

Esta é a nossa solução de DP necessária. Com nosso truque muito simples, a solução executa tempo, porque existem parâmetros diferentes nossa função pode ser chamada com e para cada um deles, a função é executada apenas uma vez com complexidade .

Verão:
Se você puder identificar um problema que pode ser resolvido usando DP, use as seguintes etapas para construir uma solução DP:

  • Crie uma função de retrocesso para fornecer a resposta correta.
  • Remova os argumentos redundantes.
  • Estimar e minimizar o intervalo máximo de valores possíveis dos parâmetros da função.
  • Tente otimizar a complexidade de tempo de uma chamada de função.
  • Armazene os valores para que você não precise calculá-lo duas vezes.

A complexidade de uma solução DP é: faixa de valores possíveis com os quais a função pode ser chamada * complexidade de tempo de cada chamada.
