Comentários #

Comentários de linha única em Lua começam com -- e continuam até o final da linha:

-- this is single line comment
-- need another line
-- huh?

Comentários em bloco começam com --[[ e terminam com ]]:

--[[
    This is block comment.
    So, it can go on...
    and on...
    and on....
]]

Comentários em bloco usam o mesmo estilo de delimitadores que strings longas; qualquer número de sinais de igual pode ser adicionado entre colchetes para delimitar um comentário:

--[=[
    This is also a block comment
    We can include "]]" inside this comment
--]=]

--[==[
    This is also a block comment
    We can include "]=]" inside this comment
--]==]

Um truque legal para comentar pedaços de código é cercá-lo com --[[ e --]]:

--[[
    print'Lua is lovely'
--]]

Para reativar o pedaço, basta anexar um - à sequência de abertura do comentário:

---[[
    print'Lua is lovely'
--]]

Desta forma, a sequência -- na primeira linha inicia um comentário de linha única, assim como na última linha, e a instrução print não é comentada.

Levando isso um passo adiante, dois blocos de código podem ser configurados de tal forma que, se o primeiro bloco for comentado, o segundo não será e vice-versa:

---[[
  print 'Lua is love'
--[=[]]
  print 'Lua is life'
--]=]

Para ativar a segunda parte enquanto desabilita a primeira parte, exclua o - inicial na primeira linha:

--[[
  print 'Lua is love'
--[=[]]
  print 'Lua is life'
--]=]

Instalação #

Binários

Binários Lua são fornecidos pela maioria das distribuições GNU/Linux como um pacote.

Por exemplo, no Debian, Ubuntu e seus derivados, ele pode ser adquirido executando isto:

sudo apt-get install lua50

sudo apt-get install lua51

sudo apt-get install lua52

Existem algumas compilações semi-oficiais fornecidas para Windows, MacOS e alguns outros sistemas operacionais hospedados em SourceForge.

Os usuários da Apple também podem instalar o Lua facilmente usando o Homebrew:

brew install lua

(Atualmente o Homebrew tem 5.2.4, para 5.3 veja Homebrew/versions.)

Fonte

A fonte está disponível na página oficial. A aquisição de fontes e a construção em si devem ser triviais. Em sistemas Linux, o seguinte deve ser suficiente:

$ wget http://lua.org/ftp/lua-5.3.3.tar.gz
$ echo "a0341bc3d1415b814cc738b2ec01ae56045d64ef ./lua-5.3.3.tar.gz" | sha1sum -c -
$ tar -xvf ./lua-5.3.3.tar.gz
$ make -C ./lua-5.3.3/ linux

No exemplo acima, estamos basicamente baixando um tarball fonte do site oficial, verificando sua soma de verificação e extraindo e executando o make. (Verifique a soma de verificação na página oficial.)

Observação: você deve especificar qual destino de compilação deseja. No exemplo, especificamos linux. Outros destinos de compilação disponíveis incluem solaris, aix, bsd, freebsd, macosx, mingw, etc. Confira doc/readme.html, que está incluído na fonte, para mais detalhes. (Você também pode encontrar a versão mais recente do README online.)

Módulos

Bibliotecas padrão são limitadas a primitivas:

  • corotina - funcionalidade de gerenciamento de corrotina
  • debug - ganchos e ferramentas de depuração
  • io - primitivos básicos de E/S
  • package - funcionalidade de gerenciamento de módulo
  • string - funcionalidade de correspondência de padrões específica de string e Lua
  • table - primitivas para lidar com um tipo Lua essencial, mas complexo - tabelas
  • os - operações básicas do SO
  • utf8 - primitivas UTF-8 básicas (desde Lua 5.3)

Todas essas bibliotecas podem ser desabilitadas para uma compilação específica ou carregadas em tempo de execução.

Bibliotecas e infraestrutura Lua de terceiros para distribuição de módulos são escassas, mas estão melhorando. Projetos como LuaRocks, Lua Toolbox e LuaDist estão melhorando o situação. Muitas informações e muitas sugestões podem ser encontradas no antigo Lua Wiki, mas esteja ciente de que algumas dessas informações são bastante antigas e desatualizadas.

Começando #

variáveis #

var = 50 -- a global variable
print(var) --> 50
do
  local var = 100 -- a local variable
  print(var) --> 100
end
print(var) --> 50
-- The global var (50) still exists 
-- The local var (100) has gone out of scope and can't be accessed any longer.

tipos #

num = 20 -- a number
num = 20.001 -- still a number
str = "zaldrizes buzdari iksos daor" -- a string
tab = {1, 2, 3} -- a table (these have their own category)
bool = true -- a boolean value
bool = false -- the only other boolean value
print(type(num)) --> 'number'
print(type(str)) --> 'string'
print(type(bool)) --> 'boolean'
type(type(num)) --> 'string'

-- Functions are a type too, and first-class values in Lua.
print(type(print)) --> prints 'function'
old_print = print
print = function (x) old_print "I'm ignoring the param you passed me!" end
old_print(type(print)) --> Still prints 'function' since it's still a function.
-- But we've (unhelpfully) redefined the behavior of print.
print("Hello, world!") --> prints "I'm ignoring the param you passed me!"

O tipo especial nil #

Outro tipo em Lua é nil. O único valor no tipo nil é nil. nil existe para ser diferente de todos os outros valores em Lua. É um tipo de valor sem valor.

print(foo) -- This prints nil since there's nothing stored in the variable 'foo'.
foo = 20
print(foo) -- Now this prints 20 since we've assigned 'foo' a value of 20.

-- We can also use `nil` to undefine a variable
foo = nil -- Here we set 'foo' to nil so that it can be garbage-collected.

if nil then print "nil" end --> (prints nothing)
-- Only false and nil are considered false; every other value is true.
if 0 then print "0" end --> 0
if "" then print "Empty string!" --> Empty string!

expressões #

a = 3
b = a + 20 a = 2 print(b, a) -- hard to read, can also be written as
b = a + 20; a = 2; print(a, b) -- easier to read, ; are optional though
true and true --> returns true
true and 20 --> 20
false and 20 --> false
false or 20 --> 20
true or 20 --> true
tab or {}
  --> returns tab if it is defined
  --> returns {} if tab is undefined
  -- This is useful when we don't know if a variable exists
tab = tab or {} -- tab stays unchanged if it exists; tab becomes {} if it was previously nil.

a, b = 20, 30 -- this also works
a, b = b, a -- switches values

Definindo funções #

function name(parameter)
    return parameter
end
print(name(20)) --> 20
-- see function category for more information
name = function(parameter) return parameter end -- Same as above

booleanos #

Apenas false e ​​nil são avaliados como false, todo o resto, incluindo 0 e a string vazia são avaliados como true.

coleta de lixo #

tab = {"lots", "of", "data"}
tab = nil; collectgarbage()
-- tab does no longer exist, and doesn't take up memory anymore.

mesas #

tab1 = {"a", "b", "c"}
tab2 = tab1
tab2[1] = "d"
print(tab1[1]) --> 'd' -- table values only store references.
--> assigning tables does not copy its content, only the reference.

tab2 = nil; collectgarbage()
print(tab1) --> (prints table address) -- tab1 still exists; it didn't get garbage-collected.

tab1 = nil; collectgarbage()
-- No more references. Now it should actually be gone from memory.

Estes são os fundamentos, mas há uma seção sobre tabelas com mais informações.

condições #

if (condition) then
    -- do something

end

para laços #

Existem dois tipos de loop for em Lua: um loop numérico for e um loop genérico for.

  • Um loop numérico for tem a seguinte forma:

    for a=1, 10, 2 do -- for a starting at 1, ending at 10, in steps of 2
  print(a) --> 1, 3, 5, 7, 9
end

    The third expression in a numeric for loop is the step by which the loop will increment. This makes it easy to do reverse loops:

     for a=10, 1, -1 do
   print(a) --> 10, 9, 8, 7, 6, etc.
 end

    If the step expression is left out, Lua assumes a default step of 1.

     for a=1, 10 do
   print(a) --> 1, 2, 3, 4, 5, etc.
 end

    Also note that the loop variable is local to the for loop. It will not exist after the loop is over.

  • Loops for genéricos funcionam através de todos os valores que uma função iteradora retorna:

    for key, value in pairs({"some", "table"}) do
  print(key, value)
  --> 1 some
  --> 2 table
end

Lua fornece vários iteradores embutidos (por exemplo, pairs, ipairs), e os usuários podem definir seus próprios iteradores personalizados também para usar com loops for genéricos.

fazer blocos #

local a = 10
do
    print(a) --> 10
    local a = 20
    print(a) --> 20
end
print(a) --> 10

Algumas coisas complicadas #

Às vezes, Lua não se comporta da maneira que alguém pensaria depois de ler a documentação. Alguns desses casos são:

Nil e Nothing não são os mesmos (ARMADILHA COMUM!) #

Como esperado, table.insert(my_table, 20) adiciona o valor 20 à tabela, e table.insert(my_table, 5, 20) adiciona o valor 20 na 5ª posição. O que table.insert(my_table, 5, nil) faz? Pode-se esperar que ele trate nil como nenhum argumento e insira o valor 5 no final da tabela, mas na verdade ele adiciona o valor nil na 5ª posição da tabela. Quando isso é um problema?

(function(tab, value, position)
    table.insert(tab, position or value, position and value)
end)({}, 20)
-- This ends up calling table.insert({}, 20, nil)
-- and this doesn't do what it should (insert 20 at the end)

Algo semelhante acontece com tostring():

print (tostring(nil)) -- this prints "nil"
table.insert({}, 20) -- this returns nothing
-- (not nil, but actually nothing (yes, I know, in lua those two SHOULD
-- be the same thing, but they aren't))

-- wrong:
print (tostring( table.insert({}, 20) ))
-- throws error because nothing ~= nil

--right:
local _tmp = table.insert({}, 20) -- after this _tmp contains nil
print(tostring(_tmp)) -- prints "nil" because suddenly nothing == nil

Isso também pode levar a erros ao usar o código de terceiros. Se, por exemplo, a documentação de alguma função diz “retorna rosquinhas se tiver sorte, nil caso contrário”, a implementação pode se parecer com isso

function func(lucky)
    if lucky then
        return "donuts"
    end
end

essa implementação pode parecer razoável a princípio; ele retorna donuts quando necessário, e quando você digita result = func(false) o resultado conterá o valor nil.

No entanto, se alguém escrevesse print(tostring(func(false))) lua lançaria um erro que se parece com este stdin:1: bad argument #1 to 'tostring' (valor esperado)

Por que é que? tostring claramente recebe um argumento, mesmo que seja nil. Errado. func não retorna nada, então tostring(func(false)) é o mesmo que tostring() e NÃO é o mesmo que tostring(nil).

Erros dizendo “valor esperado” são uma forte indicação de que essa pode ser a origem do problema.

Deixando lacunas nas matrizes #

Esta é uma grande armadilha se você é novo em lua, e há muitas [informações](https://www.wikiod.com/pt/lua/tabelas#Evitando lacunas em tabelas usadas como arrays as-arrays#t=20170113143427074516) na categoria tabelas

Executando programas Lua #

Normalmente, Lua está sendo enviada com dois binários:

  • lua - interpretador autônomo e shell interativo
  • luac - compilador de bytecode

Digamos que temos um programa de exemplo (bottles_of_mate.lua) assim:

local string = require "string"    

function bottle_take(bottles_available)

    local count_str = "%d bottles of mate on the wall."
    local take_str = "Take one down, pass it around, " .. count_str
    local end_str = "Oh noes, " .. count_str
    local buy_str = "Get some from the store, " .. count_str
    local bottles_left = 0

    if bottles_available > 0 then
         print(string.format(count_str, bottles_available))
         bottles_left = bottles_available - 1
         print(string.format(take_str, bottles_left))
    else
        print(string.format(end_str, bottles_available))
        bottles_left = 99
        print(string.format(buy_str, bottles_left))
    end

    return bottles_left
end

local bottle_count = 99

while true do
    bottle_count = bottle_take(bottle_count)
end

O programa em si pode ser executado executando o seguinte em seu shell:

$ lua bottles_of_mate.lua

$ luac bottles_of_mate.lua -o bottles_of_mate.luac

Olá Mundo #

Este é o código Olá mundo:

print("Hello World!")

Como funciona? É simples! Lua executa a função print() e usa a string "Hello World" como argumento.
