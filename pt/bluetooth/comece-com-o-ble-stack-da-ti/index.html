Conectando-se a dispositivos escravos BLE #

Introdução #

Os SoCs da série [CC26XX][1] da Texas Instruments (TI) são MCUs sem fio prontamente disponíveis visando aplicativos Bluetooth Low Energy (BLE). Junto com os MCUs, a TI oferece uma [pilha de software][2] completa que fornece API e códigos de amostra necessários para ajudar os desenvolvedores a iniciar rapidamente a cadeia de ferramentas. No entanto, para iniciantes, sempre há a questão de por onde começar diante de uma longa lista de documentos e códigos de referência. Esta nota tem como objetivo registrar as etapas necessárias para iniciar o primeiro projeto.

O Simple Peripheral Profile é o exemplo ‘Hello World’ da pilha BLE, onde o MCU está atuando como um periférico BLE para hosts upstream ou clientes de serviço BLE, como PC e smartphones. As aplicações comuns do mundo real incluem: fone de ouvido Bluetooth, sensor de temperatura Bluetooth, etc.

Antes de começar, primeiro precisamos reunir ferramentas básicas de software e hardware para fins de programação e depuração.

  1. Pilha BLE

    Download and install TI’s BLE-STACK-2-2-0 from the official website. Assume it is installed in the default location ‘C:\ti’.

  2. IDE - existem duas opções:

  • Bancada de trabalho integrada IAR para ARM. Esta é uma ferramenta comercial com um período de avaliação gratuita de 30 dias.

  • Code Composer Studio (CCS) da TI. IDE oficial da TI e oferece licença gratuita. Neste exemplo, usaremos o CCS V6.1.3

  1. Ferramenta de programação de hardware

    Recommend TI’s [XDS100][3] USB-interface JTAG device.

Projeto de exemplo de importação em CCS #

O código de amostra Simple Peripheral Profile vem com a instalação do BLE-Stack. Siga as etapas abaixo para importar este projeto de exemplo para o CCS.

  1. Inicie o CCS, crie uma pasta de espaço de trabalho. Em seguida, Arquivo->Importar. Em ‘Selecionar uma fonte de importação’, selecione a opção ‘Code Compose Studio -> CCS Projects’ e clique em ‘Avançar’. [![Importar Projeto CCS][4]][4]
  2. Navegue até ‘C:\ti\simplelink\ble_sdk_2_02_00_31\examples\cc2650em\simple_peripheral\ccs’. Dois projetos serão descobertos. Selecione tudo e marque as duas opções abaixo. Em seguida, clique em ‘Concluir’. Ao copiar projetos para a área de trabalho, você deixa a configuração original do projeto inalterada para todas as modificações a seguir. [![digite a descrição da imagem aqui][5]][5]

O exemplo Simple Peripheral Profile inclui dois projetos:

  • simple_peripheral_cc2650em_app
  • simple_peripheral_cc2650em_stack

‘cc2650em’ é o codinome da placa de avaliação cc2650 da TI. O projeto _stack inclui os códigos e binários do BEL-Stack-2-2-0 da TI, que trata da publicidade Bluetooth, handshake, sincronização de frequência etc. Esta é a parte do código que é relativamente estável e não quer ser tocado pelos desenvolvedores na maioria das vezes. O projeto _app é onde os desenvolvedores implementam suas próprias tarefas e serviço BLE.

Compilar e baixar #

Clique nos menus ‘Project->Build All’ para construir ambos os projetos. Se o compilador relatar algum tipo de erro interno na vinculação, tente desabilitar a opção ‘compress_dwarf’ para o vinculador:

  • clique com o botão direito do mouse no projeto e selecione ‘Propriedades’.
  • em ‘Build->ARM Linker’, clique no botão ‘Edit Flags’.
  • modifique a última opção para ‘–compress_dwarf=off’.

Depois que ambos os projetos forem criados com sucesso, clique em ‘Executar->depurar’ separadamente para baixar as imagens da pilha e do aplicativo para o MCU.

Toque no Código #

Para poder fazer modificações agressivas no código de amostra, os desenvolvedores precisam obter conhecimento detalhado sobre a estrutura em camadas da pilha BLE. Para tarefas elementares, como leitura/notificação de temperatura, podemos nos concentrar em apenas dois arquivos: PROFILES/simple_gatt_profile.c(.h) e Application/simple_peripheral.c(.h)

simple_gatt_profile.c #

Todos os aplicativos Bluetooth oferecem um determinado tipo de serviço, cada um composto por um conjunto de características. O perfil periférico simples define um serviço simples, com o UUID de 0xFFF0, que consiste em 5 características. Este serviço é especificado em simple_gatt_profile.c. Um resumo do serviço simples é listado a seguir.

Nome Tamanho dos dados UUID Descrição Propriedade
simplePeripheralChar1 1 0xFFF1 Características 1 Ler escrever
simplePeripheralChar2 1 0xFFF2 Características 2 Somente leitura
simplePeripheralChar3 1 0xFFF3 Características 3 Escrever apenas
simplePeripheralChar4 1 0xFFF4 Características 4 Notificar
simplePeripheralChar5 5 0xFFF5 Características 5 Somente leitura
As cinco características têm propriedades diferentes e servem como exemplos para vários casos de usuário. Por exemplo, o MCU pode usar simplePeripheralChar4 para notificar seus clientes, hosts upstream, sobre a mudança de informações.

Para definir um serviço Bluetooth, deve-se construir uma Tabela de Atributos.

/*********************************************************************
 * Profile Attributes - Table
 */

static gattAttribute_t simpleProfileAttrTbl[SERVAPP_NUM_ATTR_SUPPORTED] = 
{
  // Simple Profile Service
  { 
    { ATT_BT_UUID_SIZE, primaryServiceUUID }, /* type */
    GATT_PERMIT_READ,                         /* permissions */
    0,                                        /* handle */
    (uint8 *)&simpleProfileService            /* pValue */
  },

    // Characteristic 1 Declaration
    { 
      { ATT_BT_UUID_SIZE, characterUUID },
      GATT_PERMIT_READ, 
      0,
      &simpleProfileChar1Props 
    },

      // Characteristic Value 1
      { 
        { ATT_UUID_SIZE, simpleProfilechar1UUID },
        GATT_PERMIT_READ | GATT_PERMIT_WRITE, 
        0, 
        &simpleProfileChar1
      },

      // Characteristic 1 User Description
      { 
        { ATT_BT_UUID_SIZE, charUserDescUUID },
        GATT_PERMIT_READ, 
            0, 
            simpleProfileChar1UserDesp 
          }, 
        ...
    };

A tabela de atributos começa com um ‘primaryServiceUUID’ padrão, que especifica o UUID do serviço (0xFFF0 neste caso). Seguem-se declarações de todas as características que compõem o serviço. Cada característica tem vários atributos, nomeadamente permissão de acesso, valor e descrição do utilizador, etc. Esta tabela é posteriormente registada na pilha BLE.

// Register GATT attribute list and CBs with GATT Server App
status = GATTServApp_RegisterService( simpleProfileAttrTbl, 
                                      GATT_NUM_ATTRS( simpleProfileAttrTbl ),
                                      GATT_MAX_ENCRYPT_KEY_SIZE,
                                      &simpleProfileCBs );

No registro do serviço, os desenvolvedores devem fornecer três funções de retorno de chamada para ‘Ler’, ‘Gravar’ e ‘Autorização’ das características. Podemos encontrar no código de exemplo a lista de funções de retorno de chamada.

/*********************************************************************
 * PROFILE CALLBACKS
 */

// Simple Profile Service Callbacks
// Note: When an operation on a characteristic requires authorization and 
// pfnAuthorizeAttrCB is not defined for that characteristic's service, the 
// Stack will report a status of ATT_ERR_UNLIKELY to the client.  When an 
// operation on a characteristic requires authorization the Stack will call 
// pfnAuthorizeAttrCB to check a client's authorization prior to calling
// pfnReadAttrCB or pfnWriteAttrCB, so no checks for authorization need to be 
// made within these functions.
CONST gattServiceCBs_t simpleProfileCBs =
{
  simpleProfile_ReadAttrCB,  // Read callback function pointer
  simpleProfile_WriteAttrCB, // Write callback function pointer
  NULL                       // Authorization callback function pointer
};

Assim, simpleProfile_ReadAttrCB será chamado assim que o cliente de serviço enviar uma solicitação de leitura pela conexão Bluetooth. Da mesma forma, simpleProfile_WriteAttrCB será chamado quando uma solicitação de gravação for feita. Compreender essas duas funções é a chave para o sucesso da customização do projeto.

Abaixo está a função de retorno de chamada de leitura.

/*********************************************************************
 * @fn          simpleProfile_ReadAttrCB
 *
 * @brief       Read an attribute.
 *
 * @param       connHandle - connection message was received on
 * @param       pAttr - pointer to attribute
 * @param       pValue - pointer to data to be read
 * @param       pLen - length of data to be read
 * @param       offset - offset of the first octet to be read
 * @param       maxLen - maximum length of data to be read
 * @param       method - type of read message
 *
 * @return      SUCCESS, blePending or Failure
 */
static bStatus_t simpleProfile_ReadAttrCB(uint16_t connHandle,
                                          gattAttribute_t *pAttr,
                                          uint8_t *pValue, uint16_t *pLen,
                                          uint16_t offset, uint16_t maxLen,
                                          uint8_t method)
{
      bStatus_t status = SUCCESS;

      // If attribute permissions require authorization to read, return error
      if ( gattPermitAuthorRead( pAttr->permissions ) )
      {
        // Insufficient authorization
        return ( ATT_ERR_INSUFFICIENT_AUTHOR );
      }

      // Make sure it's not a blob operation (no attributes in the profile are long)
      if ( offset > 0 )
      {
        return ( ATT_ERR_ATTR_NOT_LONG );
      }

      uint16 uuid = 0;
      if ( pAttr->type.len == ATT_UUID_SIZE )
        // 128-bit UUID
        uuid = BUILD_UINT16( pAttr->type.uuid[12], pAttr->type.uuid[13]);
      else
        uuid = BUILD_UINT16( pAttr->type.uuid[0], pAttr->type.uuid[1]);

        switch ( uuid )
        {
          // No need for "GATT_SERVICE_UUID" or "GATT_CLIENT_CHAR_CFG_UUID" cases;
          // gattserverapp handles those reads

          // characteristics 1 and 2 have read permissions
          // characteritisc 3 does not have read permissions; therefore it is not
          //   included here
          // characteristic 4 does not have read permissions, but because it
          //   can be sent as a notification, it is included here
          case SIMPLEPROFILE_CHAR2_UUID:
              *pLen = SIMPLEPROFILE_CHAR2_LEN;
              VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR2_LEN );
              break;

          case SIMPLEPROFILE_CHAR1_UUID:
              *pLen = SIMPLEPROFILE_CHAR1_LEN;
              VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR1_LEN );
              break;

          case SIMPLEPROFILE_CHAR4_UUID:
            *pLen = SIMPLEPROFILE_CHAR4_LEN;
            VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR4_LEN );
            break;

          case SIMPLEPROFILE_CHAR5_UUID:
            *pLen = SIMPLEPROFILE_CHAR5_LEN;
            VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR5_LEN );
            break;

          default:
            // Should never get here! (characteristics 3 and 4 do not have read permissions)
            *pLen = 0;
            status = ATT_ERR_ATTR_NOT_FOUND;
            break;
        }

      return ( status );
}

Eu modifiquei um pouco o código de sua versão original. Esta função recebe 7 parâmetros, que são explicados nos comentários do cabeçalho. A função começa verificando a permissão de acesso do atributo, por exemplo. se tem permissão de leitura. Em seguida, ele verifica se este é um segmento lido de uma solicitação de leitura de blob maior testando a condição ‘if (offset > 0)’. Obviamente, a função não suporta leitura de blob por enquanto. Em seguida, o UUID do atributo solicitado é extraído. Existem dois tipos de UUID: 16 bits e 128 bits. Embora o código de exemplo defina todas as características usando UUIDs de 16 bits, o UUID de 128 bits é mais universal e mais comumente usado em hosts upstream, como PC e smartphones. Portanto, várias linhas de código são usadas para converter UUID de 128 bits em UUID de 16 bits.

 uint16 uuid = 0;
  if ( pAttr->type.len == ATT_UUID_SIZE )
    // 128-bit UUID
    uuid = BUILD_UINT16( pAttr->type.uuid[12], pAttr->type.uuid[13]);
  else
    uuid = BUILD_UINT16( pAttr->type.uuid[0], pAttr->type.uuid[1]);

Finalmente, após obtermos o UUID, podemos determinar qual atributo é solicitado. Em seguida, o trabalho restante do lado dos desenvolvedores é copiar o valor do atributo solicitado para o ponteiro de destino ‘pValue’.

switch ( uuid )
    {          
      case SIMPLEPROFILE_CHAR1_UUID:
          *pLen = SIMPLEPROFILE_CHAR1_LEN;
          VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR1_LEN );
          break;

      case SIMPLEPROFILE_CHAR2_UUID:
          *pLen = SIMPLEPROFILE_CHAR2_LEN;
          VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR2_LEN );
          break;

      case SIMPLEPROFILE_CHAR4_UUID:
        *pLen = SIMPLEPROFILE_CHAR4_LEN;
        VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR4_LEN );
        break;

      case SIMPLEPROFILE_CHAR5_UUID:
        *pLen = SIMPLEPROFILE_CHAR5_LEN;
        VOID memcpy( pValue, pAttr->pValue, SIMPLEPROFILE_CHAR5_LEN );
        break;

      default:
        *pLen = 0;
        status = ATT_ERR_ATTR_NOT_FOUND;
        break;
    }

A função de retorno de chamada de gravação é semelhante, exceto que há um tipo especial de gravação com UUID de GATT_CLIENT_CHAR_CFG_UUID. Este é o pedido do host upstream para se registrar para notificação ou indicação de características. Basta chamar a API GATTServApp_ProcessCCCWriteReq para passar a solicitação para a pilha BLE.

case GATT_CLIENT_CHAR_CFG_UUID:
        status = GATTServApp_ProcessCCCWriteReq( connHandle, pAttr, pValue, len,
                                                 offset, GATT_CLIENT_CFG_NOTIFY | GATT_CLIENT_CFG_INDICATE ); // allow client to request notification or indication features
        break;

O lado do aplicativo do código no MCU pode querer ser notificado com qualquer alteração nas características permitidas para gravação. Os desenvolvedores podem implementar essa notificação da maneira que quiserem. No código de exemplo, a função de retorno de chamada é usada.

  // If a charactersitic value changed then callback function to notify application of change
  if ( (notifyApp != 0xFF ) && simpleProfile_AppCBs && simpleProfile_AppCBs->pfnSimpleProfileChange )
  {
    simpleProfile_AppCBs->pfnSimpleProfileChange( notifyApp );
  }

Por outro lado, se o periférico BLE quiser notificar os hosts upstream sobre qualquer mudança em sua característica, ele pode chamar a API GATTServApp_ProcessCharCfg. Essa API é demonstrada na função SimpleProfile_SetParameter.

/*********************************************************************
 * @fn      SimpleProfile_SetParameter
 *
 * @brief   Set a Simple Profile parameter.
 *
 * @param   param - Profile parameter ID
 * @param   len - length of data to write
 * @param   value - pointer to data to write.  This is dependent on
 *          the parameter ID and WILL be cast to the appropriate 
 *          data type (example: data type of uint16 will be cast to 
 *          uint16 pointer).
 *
 * @return  bStatus_t
 */
bStatus_t SimpleProfile_SetParameter( uint8 param, uint8 len, void *value )
{
      bStatus_t ret = SUCCESS;
      switch ( param )
      {
        case SIMPLEPROFILE_CHAR2:
          if ( len == SIMPLEPROFILE_CHAR2_LEN )
          {
            VOID memcpy( simpleProfileChar2, value, SIMPLEPROFILE_CHAR2_LEN );

          }
          else
          {
            ret = bleInvalidRange;
          }
          break;

        case SIMPLEPROFILE_CHAR3:
          if ( len == sizeof ( uint8 ) )
          {
            simpleProfileChar3 = *((uint8*)value);
          }
          else
          {
            ret = bleInvalidRange;
          }
          break;

        case SIMPLEPROFILE_CHAR1:
          if ( len == SIMPLEPROFILE_CHAR1_LEN )
          {
              VOID memcpy( simpleProfileChar1, value, SIMPLEPROFILE_CHAR1_LEN );
          }
          else
          {
            ret = bleInvalidRange;
          }
          break;

        case SIMPLEPROFILE_CHAR4:
          if ( len == SIMPLEPROFILE_CHAR4_LEN )
          {
            //simpleProfileChar4 = *((uint8*)value);
            VOID memcpy( simpleProfileChar4, value, SIMPLEPROFILE_CHAR4_LEN );
            // See if Notification has been enabled
            GATTServApp_ProcessCharCfg( simpleProfileChar4Config, simpleProfileChar4, FALSE,
                                        simpleProfileAttrTbl, GATT_NUM_ATTRS( simpleProfileAttrTbl ),
                                        INVALID_TASK_ID, simpleProfile_ReadAttrCB );
          }
          else
          {
            ret = bleInvalidRange;
          }
          break;

        case SIMPLEPROFILE_CHAR5:
          if ( len == SIMPLEPROFILE_CHAR5_LEN )
          {
            VOID memcpy( simpleProfileChar5, value, SIMPLEPROFILE_CHAR5_LEN );
          }
          else
          {
            ret = bleInvalidRange;
          }
          break;

        default:
          ret = INVALIDPARAMETER;
          break;
      }

      return ( ret );
}

Portanto, se o aplicativo periférico simples deseja notificar o valor atual de SIMPLEPROFILE_CHAR4 para dispositivos peer, ele pode simplesmente chamar a função SimpleProfile_SetParameter.

Em resumo, PROFILES/simple_gatt_profile.c(.h) define o conteúdo do serviço que o periférico BLE gostaria de apresentar aos seus clientes, bem como as formas como essas características no serviço são acessadas.

simple_peripheral.c #

A pilha BLE da TI está sendo executada em cima de uma camada de sistema operacional multithread lite. Para adicionar uma carga de trabalho ao MCU, os desenvolvedores precisam primeiro criar uma tarefa. simple_peripheral.c demonstra a estrutura básica de uma tarefa personalizada, que inclui a criação, inicialização e manutenção da tarefa. Para começar com as tarefas básicas, como leitura e notificação de temperatura, vamos nos concentrar em algumas funções principais abaixo.

O início do arquivo define um conjunto de parâmetros que podem afetar os comportamentos da conexão Bluetooth.

// Advertising interval when device is discoverable (units of 625us, 160=100ms)
#define DEFAULT_ADVERTISING_INTERVAL          160

// Limited discoverable mode advertises for 30.72s, and then stops
// General discoverable mode advertises indefinitely
#define DEFAULT_DISCOVERABLE_MODE             GAP_ADTYPE_FLAGS_GENERAL

// Minimum connection interval (units of 1.25ms, 80=100ms) if automatic
// parameter update request is enabled
#define DEFAULT_DESIRED_MIN_CONN_INTERVAL     80

// Maximum connection interval (units of 1.25ms, 800=1000ms) if automatic
// parameter update request is enabled
#define DEFAULT_DESIRED_MAX_CONN_INTERVAL     400

// Slave latency to use if automatic parameter update request is enabled
#define DEFAULT_DESIRED_SLAVE_LATENCY         0

// Supervision timeout value (units of 10ms, 1000=10s) if automatic parameter
// update request is enabled
#define DEFAULT_DESIRED_CONN_TIMEOUT          1000

// Whether to enable automatic parameter update request when a connection is
// formed
#define DEFAULT_ENABLE_UPDATE_REQUEST         TRUE

// Connection Pause Peripheral time value (in seconds)
#define DEFAULT_CONN_PAUSE_PERIPHERAL         6

// How often to perform periodic event (in msec)
#define SBP_PERIODIC_EVT_PERIOD               1000

Os parâmetros DEFAULT_DESIRED_MIN_CONN_INTERVAL, DEFAULT_DESIRED_MAX_CONN_INTERVAL e DEFAULT_DESIRED_SLAVE_LATENCY juntos definem o intervalo de conexões de uma conexão Bluetooth, que é a frequência com que um par de dispositivos troca informações. Um intervalo de conexão menor significa um comportamento mais responsivo, mas também maior consumo de energia.

O parâmetro DEFAULT_DESIRED_CONN_TIMEOUT define por quanto tempo receber uma resposta de peer antes que uma conexão seja considerada perdida. O parâmetro DEFAULT_ENABLE_UPDATE_REQUEST define se o dispositivo escravo tem permissão para alterar o intervalo de conexão durante o tempo de execução. É útil em termos de economia de energia ter diferentes parâmetros de conexão para fases ocupadas e inativas.

O parâmetro SBP_PERIODIC_EVT_PERIOD define o período de um evento de relógio que permitirá que a tarefa execute uma chamada de função periodicamente. Este é o lugar perfeito para adicionarmos o código para ler a temperatura e notificar os clientes do serviço.

O relógio periódico é iniciado na função SimpleBLEPeripheral_init.

  // Create one-shot clocks for internal periodic events.
  Util_constructClock(&periodicClock, SimpleBLEPeripheral_clockHandler,
                      SBP_PERIODIC_EVT_PERIOD, 0, false, SBP_PERIODIC_EVT);

Isso criará um relógio com um período de SBP_PERIODIC_EVT_PERIOD. E no tempo limite, chamará a função de SimpleBLEPeripheral_clockHandler com o parâmetro SBP_PERIODIC_EVT. O evento de relógio pode então ser acionado por

Util_startClock(&periodicClock);

Pesquisando pela palavra-chave Util_startClock, podemos descobrir que esse relógio periódico é acionado primeiro no evento GAPROLE_CONNECTED (dentro da função SimpleBLEPeripheral_processStateChangeEvt), o que significa que a tarefa iniciará uma rotina periódica assim que estabelecer uma conexão com um host.

Quando o relógio periódico expirar, sua função de retorno de chamada registrada será chamada.

/*********************************************************************
 * @fn      SimpleBLEPeripheral_clockHandler
 *
 * @brief   Handler function for clock timeouts.
 *
 * @param   arg - event type
 *
 * @return  None.
 */
static void SimpleBLEPeripheral_clockHandler(UArg arg)
{
  // Store the event.
  events |= arg;

  // Wake up the application.
  Semaphore_post(sem);
}

Esta função define um sinalizador no vetor de eventos e ativa o aplicativo na lista de tarefas do SO. Observe que não fazemos nenhuma carga de trabalho específica do usuário nesta função de retorno de chamada, porque NÃO é recomendado. A carga de trabalho do usuário geralmente envolve chamadas para APIs de pilha BLE. Fazer chamadas de API de pilha BLE dentro de funções de retorno de chamada geralmente resulta em exceções do sistema. Em vez disso, definimos um sinalizador no vetor de eventos da tarefa e esperamos que ele seja processado posteriormente no contexto do aplicativo. O ponto de entrada para a tarefa de exemplo é simpleBLEPeripheral_taskFxn().

/*********************************************************************
 * @fn      SimpleBLEPeripheral_taskFxn
 *
 * @brief   Application task entry point for the Simple BLE Peripheral.
 *
 * @param   a0, a1 - not used.
 *
 * @return  None.
 */
static void SimpleBLEPeripheral_taskFxn(UArg a0, UArg a1)
{
  // Initialize application
  SimpleBLEPeripheral_init();

  // Application main loop
  for (;;)
  {
    // Waits for a signal to the semaphore associated with the calling thread.
    // Note that the semaphore associated with a thread is signaled when a
    // message is queued to the message receive queue of the thread or when
    // ICall_signal() function is called onto the semaphore.
    ICall_Errno errno = ICall_wait(ICALL_TIMEOUT_FOREVER);

    if (errno == ICALL_ERRNO_SUCCESS)
    {
      ICall_EntityID dest;
      ICall_ServiceEnum src;
      ICall_HciExtEvt *pMsg = NULL;

      if (ICall_fetchServiceMsg(&src, &dest,
                                (void **)&pMsg) == ICALL_ERRNO_SUCCESS)
      {
        uint8 safeToDealloc = TRUE;

        if ((src == ICALL_SERVICE_CLASS_BLE) && (dest == selfEntity))
        {
          ICall_Stack_Event *pEvt = (ICall_Stack_Event *)pMsg;

          // Check for BLE stack events first
          if (pEvt->signature == 0xffff)
          {
            if (pEvt->event_flag & SBP_CONN_EVT_END_EVT)
            {
              // Try to retransmit pending ATT Response (if any)
              SimpleBLEPeripheral_sendAttRsp();
            }
          }
          else
          {
            // Process inter-task message
            safeToDealloc = SimpleBLEPeripheral_processStackMsg((ICall_Hdr *)pMsg);
          }
        }

        if (pMsg && safeToDealloc)
        {
          ICall_freeMsg(pMsg);
        }
      }

      // If RTOS queue is not empty, process app message.
      while (!Queue_empty(appMsgQueue))
      {
        sbpEvt_t *pMsg = (sbpEvt_t *)Util_dequeueMsg(appMsgQueue);
        if (pMsg)
        {
          // Process message.
          SimpleBLEPeripheral_processAppMsg(pMsg);

          // Free the space from the message.
          ICall_free(pMsg);
        }
      }
    }
    
    if (events & SBP_PERIODIC_EVT)
    {
      events &= ~SBP_PERIODIC_EVT;

      Util_startClock(&periodicClock);

      // Perform periodic application task
      SimpleBLEPeripheral_performPeriodicTask();
    }

  }
}

É um loop infinito que continua pesquisando a pilha da tarefa e as filas de mensagens do aplicativo. Ele também verifica seu vetor de eventos para várias bandeiras. É aí que está a rotina periódica realmente executado. Na descoberta de um SBP_PERIODIC_EVT, a função de tarefa primeiro limpa o sinalizador, inicia o mesmo temporizador imediatamente e chama a função de rotina SimpleBLEPeripheral_performPeriodicTask();

/*********************************************************************
 * @fn      SimpleBLEPeripheral_performPeriodicTask
 *
 * @brief   Perform a periodic application task. This function gets called
 *          every five seconds (SBP_PERIODIC_EVT_PERIOD). In this example,
 *          the value of the third characteristic in the SimpleGATTProfile
 *          service is retrieved from the profile, and then copied into the
 *          value of the fourth characteristic.
 *
 * @param   None.
 *
 * @return  None.
 */
static void SimpleBLEPeripheral_performPeriodicTask(void)
{
      uint8_t newValue[SIMPLEPROFILE_CHAR4_LEN];
      // user codes to do specific work like reading the temperature
      // .....
      SimpleProfile_SetParameter(SIMPLEPROFILE_CHAR4, SIMPLEPROFILE_CHAR4_LEN,
                               newValue);
}

Dentro da função periódica, executamos nosso trabalho bem específico de leitura de temperatura, geração de requisições UART etc. Em seguida, chamamos a API SimpleProfile_SetParameter() para comunicar as informações aos clientes do serviço através de conexão Bluetooth. A pilha BLE cuida de todos os trabalhos de baixo nível, desde a manutenção da conexão sem fio até a transmissão da mensagem pelo link Bluetooth. Tudo o que os desenvolvedores precisam fazer é reunir os dados específicos do aplicativo e atualizá-los para as características correspondentes em uma tabela de serviço.

Finalmente, quando uma solicitação de gravação é executada em uma característica de gravação permitida, uma função de retorno de chamada será evocada.

static void SimpleBLEPeripheral_charValueChangeCB(uint8_t paramID)
{
  SimpleBLEPeripheral_enqueueMsg(SBP_CHAR_CHANGE_EVT, paramID);
}

Novamente, essa função de retorno de chamada enfileira apenas uma mensagem do aplicativo para a tarefa do usuário, que será tratada posteriormente no contexto do aplicativo.

static void SimpleBLEPeripheral_processCharValueChangeEvt(uint8_t paramID)
{
  uint8_t newValue[SIMPLEPROFILE_CHAR1_LEN];

  switch(paramID)
  {
    case SIMPLEPROFILE_CHAR1:
      SimpleProfile_GetParameter(SIMPLEPROFILE_CHAR1, &newValue[0]);
      ProcessUserCmd(newValue[0], NULL);
      break;

    case SIMPLEPROFILE_CHAR3:
      break;

    default:
      // should not reach here!
      break;
  }
}

No exemplo acima, quando o SIMPLEPROFILE_CHAR1 é gravado, o código do usuário primeiro buscará o novo valor chamando SimpleProfile_GetParameter() e, em seguida, analisará os dados para comandos definidos pelo usuário.

Em resumo, o simple_peripheral.c mostra um exemplo de como criar tarefa de usuário para cargas de trabalho personalizadas. Uma maneira básica de agendar a carga de trabalho do aplicativo é por evento de relógio periódico. Os desenvolvedores precisam apenas processar informações de/para as características na tabela de serviço, enquanto a pilha BLE cuida do restante da comunicação das informações da tabela de serviço para dispositivos peer (ou vice-versa) por meio da conexão Bluetooth.

Conectando sensores do mundo real #

Para que os dispositivos escravos BLE façam qualquer trabalho útil, os GPIOs do MCU sem fio estão quase sempre envolvidos. Por exemplo, para ler a temperatura de um sensor externo, a funcionalidade ADC dos pinos GPIO pode ser necessária. O MCU CC2640 da TI possui um máximo de 31 GPIOs, com diferentes tipos de embalagem.

No lado do hardware, o CC2640 fornece um rico conjunto de funcionalidades periféricas como ADC, UARTS, SPI, SSI, I2C etc. No lado do software, a pilha BLE da TI tenta oferecer uma interface de driver independente de dispositivo uniforme para diferentes periféricos. Uma interface de driver uniforme pode melhorar a chance de reutilização do código, mas, por outro lado, também aumenta a inclinação da curva de aprendizado. Nesta nota, usamos o controlador SPI como exemplo e mostramos como integrar o driver de software nas aplicações do usuário.

Fluxo básico do driver SPI #

Na pilha BLE da TI, um driver periférico geralmente consiste em três partes: uma especificação independente de dispositivo das APIs do driver; uma implementação específica do dispositivo das APIs do driver e um mapeamento do recurso de hardware.

Para o controlador SPI, sua implementação de driver envolve três arquivos:

  • <ti/drivers/SPI.h> – esta é a especificação da API independente do dispositivo
  • <ti/drivers/spi/SPICC26XXDMA.h> – esta é a implementação da API específica do CC2640
  • <ti/drivers/dma/UDMACC26XX.h> – este é o driver uDMA exigido pelo driver SPI

(Nota: o melhor documento para os drivers periféricos da pilha BLE da TI pode ser encontrado principalmente em seus arquivos de cabeçalho, como SPICC26XXDMA.h neste caso)

Para começar a usar o controlador SPI, vamos primeiro criar um arquivo c personalizado, chamado sbp_spi.c, que inclui os três arquivos de cabeçalho acima. A próxima etapa natural é criar uma instância do driver e iniciá-la. A instância do driver é encapsulada na estrutura de dados – SPI_Handle. Outra estrutura de dados – SPI_Params é usada para especificar os principais parâmetros do controlador SPI, como taxa de bits, modo de transferência, etc.

#include <ti/drivers/SPI.h>
#include <ti/drivers/spi/SPICC26XXDMA.h>
#include <ti/drivers/dma/UDMACC26XX.h>

static void sbp_spiInit();

static SPI_Handle spiHandle;
static SPI_Params spiParams;

void sbp_spiInit(){
    SPI_init();
    SPI_Params_init(&spiParams);
    spiParams.mode                     = SPI_MASTER;
    spiParams.transferMode             = SPI_MODE_CALLBACK;
    spiParams.transferCallbackFxn      = sbp_spiCallback;
    spiParams.bitRate                  = 800000;
    spiParams.frameFormat              = SPI_POL0_PHA0;
    spiHandle = SPI_open(CC2650DK_7ID_SPI0, &spiParams);
}

O código de exemplo acima exemplifica como inicializar a instância SPI_Handle. A API SPI_init() deve ser chamada primeiro para inicializar as estruturas de dados internas. A chamada de função SPI_Params_init(&spiParams) configura todos os campos da estrutura SPI_Params para valores padrão. Em seguida, os desenvolvedores podem modificar os principais parâmetros para atender aos seus casos específicos. Por exemplo, o código acima configura o controlador SPI para operar no modo mestre com uma taxa de bits de 800kbps e usa um método sem bloqueio para processar cada transação, de modo que quando uma transação for concluída, a função de retorno de chamada sbp_spiCallback será chamada.

Finalmente, uma chamada para SPI_open() abre o controlador SPI de hardware e retorna um identificador para transações SPI posteriores. O SPI_open() recebe dois argumentos, o primeiro é o ID do controlador SPI. O CC2640 possui dois controladores SPI de hardware no chip, portanto, esses argumentos de ID serão 0 ou 1, conforme definido abaixo. O segundo argumento são os parâmetros desejados para o controlador SPI.

/*!
 *  @def    CC2650DK_7ID_SPIName
 *  @brief  Enum of SPI names on the CC2650 dev board
 */
typedef enum CC2650DK_7ID_SPIName {
    CC2650DK_7ID_SPI0 = 0,
    CC2650DK_7ID_SPI1,
    CC2650DK_7ID_SPICOUNT
} CC2650DK_7ID_SPIName;

Após a abertura bem-sucedida do SPI_Handle, os desenvolvedores podem iniciar transações SPI imediatamente. Cada transação SPI é descrita usando a estrutura de dados – SPI_Transaction.

/*!
 *  @brief
 *  A ::SPI_Transaction data structure is used with SPI_transfer(). It indicates
 *  how many ::SPI_FrameFormat frames are sent and received from the buffers
 *  pointed to txBuf and rxBuf.
 *  The arg variable is an user-definable argument which gets passed to the
 *  ::SPI_CallbackFxn when the SPI driver is in ::SPI_MODE_CALLBACK.
 */
typedef struct SPI_Transaction {
    /* User input (write-only) fields */
    size_t     count;      /*!< Number of frames for this transaction */
    void      *txBuf;      /*!< void * to a buffer with data to be transmitted */
    void      *rxBuf;      /*!< void * to a buffer to receive data */
    void      *arg;        /*!< Argument to be passed to the callback function */

    /* User output (read-only) fields */
    SPI_Status status;     /*!< Status code set by SPI_transfer */

    /* Driver-use only fields */
} SPI_Transaction;

Por exemplo, para iniciar uma transação de gravação no barramento SPI, os desenvolvedores precisam preparar um ’txBuf’ preenchido com dados a serem transmitidos e definir a variável ‘count’ para o comprimento dos bytes de dados a serem enviados. Finalmente, uma chamada para SPI_transfer(spiHandle, spiTrans) sinaliza ao controlador SPI para iniciar a transação.

static SPI_Transaction spiTrans;
bool sbp_spiTransfer(uint8_t len, uint8_t * txBuf, uint8_t rxBuf, uint8_t * args)
{    
    spiTrans.count = len;
    spiTrans.txBuf = txBuf;
    spiTrans.rxBuf = rxBuf;
    spiTrans.arg   = args;

    return SPI_transfer(spiHandle, &spiTrans);
}

Como o SPI é um protocolo duplex que transmite e recebe ao mesmo tempo, quando uma transação de gravação termina, seus dados de resposta correspondentes já estão disponíveis no ‘rxBuf’.

Como configuramos o modo de transferência para o modo de retorno de chamada, sempre que uma transação for concluída, a função de retorno de chamada registrada será chamada. É aqui que lidamos com os dados de resposta ou iniciamos a próxima transação. (Observação: lembre-se sempre de não fazer mais chamadas de API do que o necessário dentro de uma função de retorno de chamada).

void sbp_spiCallback(SPI_Handle handle, SPI_Transaction * transaction){
    uint8_t * args = (uint8_t *)transaction->arg;
    
    // may want to disable the interrupt first
    key = Hwi_disable();    
    if(transaction->status == SPI_TRANSFER_COMPLETED){
        // do something here for successful transaction...
    }
    Hwi_restore(key);
}

Configuração do pino de E/S #

Até agora, parece razoavelmente simples usar o driver SPI. Mas espere, como conectar as chamadas da API do software aos sinais físicos do SPI? Isso é feito por meio de três estruturas de dados: SPICC26XXDMA_Object, SPICC26XXDMA_HWAttrsV1 e SPI_Config. Eles são normalmente instanciados em um local diferente como ‘board.c’.

/* SPI objects */
SPICC26XXDMA_Object spiCC26XXDMAObjects[CC2650DK_7ID_SPICOUNT];

/* SPI configuration structure, describing which pins are to be used */
const SPICC26XXDMA_HWAttrsV1 spiCC26XXDMAHWAttrs[CC2650DK_7ID_SPICOUNT] = {
    {
        .baseAddr           = SSI0_BASE,
        .intNum             = INT_SSI0_COMB,
        .intPriority        = ~0,
        .swiPriority        = 0,
        .powerMngrId        = PowerCC26XX_PERIPH_SSI0,
        .defaultTxBufValue  = 0,
        .rxChannelBitMask   = 1<<UDMA_CHAN_SSI0_RX,
        .txChannelBitMask   = 1<<UDMA_CHAN_SSI0_TX,
        .mosiPin            = ADC_MOSI_0,
        .misoPin            = ADC_MISO_0,
        .clkPin             = ADC_SCK_0,
        .csnPin             = ADC_CSN_0
    },
    {
        .baseAddr           = SSI1_BASE,
        .intNum             = INT_SSI1_COMB,
        .intPriority        = ~0,
        .swiPriority        = 0,
        .powerMngrId        = PowerCC26XX_PERIPH_SSI1,
        .defaultTxBufValue  = 0,
        .rxChannelBitMask   = 1<<UDMA_CHAN_SSI1_RX,
        .txChannelBitMask   = 1<<UDMA_CHAN_SSI1_TX,
        .mosiPin            = ADC_MOSI_1,
        .misoPin            = ADC_MISO_1,
        .clkPin             = ADC_SCK_1,
        .csnPin             = ADC_CSN_1
    }
};

/* SPI configuration structure */
const SPI_Config SPI_config[] = {
    {
         .fxnTablePtr = &SPICC26XXDMA_fxnTable,
         .object      = &spiCC26XXDMAObjects[0],
         .hwAttrs     = &spiCC26XXDMAHWAttrs[0]
    },
    {
         .fxnTablePtr = &SPICC26XXDMA_fxnTable,
         .object      = &spiCC26XXDMAObjects[1],
         .hwAttrs     = &spiCC26XXDMAHWAttrs[1]
    },
    {NULL, NULL, NULL}
};

A matriz SPI_Config tem uma entrada separada para cada controlador SPI de hardware. Cada entrada tem três campos: fxnTablePtr, object e hwAttrs. O ‘fxnTablePtr’ é uma tabela de pontos que aponta para as implementações específicas do dispositivo da API do driver.

O ‘objeto’ mantém o controle de informações como estado do driver, modo de transferência, função de retorno de chamada para o driver. Este ‘objeto’ é mantido automaticamente pelo driver.

O ‘hwAttrs’ armazena os dados reais de mapeamento de recursos de hardware, por exemplo. os pinos IO para os sinais SPI, o número de interrupção do hardware, o endereço base do controlador SPI etc. A maioria dos campos do ‘hwAttrs’ são pré-definidos e não podem ser modificados. Considerando que os pinos IO da interface podem ser atribuídos livremente com base em casos de usuário. Observação: os MCUs CC26XX desacoplam os pinos de E/S da funcionalidade periférica específica que qualquer um dos pinos de E/S pode ser atribuído a qualquer função periférica.

É claro que os pinos IO reais devem ser definidos primeiro no ‘board.h’.

#define ADC_CSN_1                             IOID_1
#define ADC_SCK_1                             IOID_2
#define ADC_MISO_1                            IOID_3
#define ADC_MOSI_1                            IOID_4
#define ADC_CSN_0                             IOID_5
#define ADC_SCK_0                             IOID_6
#define ADC_MISO_0                            IOID_7
#define ADC_MOSI_0                            IOID_8

Como resultado, após a configuração do mapeamento de recursos de hardware, os desenvolvedores podem finalmente se comunicar com chips de sensores externos por meio da interface SPI.
