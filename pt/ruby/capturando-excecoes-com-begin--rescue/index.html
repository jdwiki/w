Um bloco básico de tratamento de erros #

Vamos fazer uma função para dividir dois números, que é muito confiável sobre sua entrada:

def divide(x, y)
  return x/y
end

Isso funcionará bem para muitas entradas:

> puts divide(10, 2)
5

Mas nem todos

> puts divide(10, 0)
ZeroDivisionError: divided by 0

> puts divide(10, 'a')
TypeError: String can't be coerced into Fixnum

Podemos reescrever a função envolvendo a operação de divisão arriscada em um bloco begin... end para verificar se há erros e usar uma cláusula rescue para enviar uma mensagem e retornar nil se houver um problema.

def divide(x, y)
  begin
    return x/y
  rescue
    puts "There was an error"
    return nil
  end
end

> puts divide(10, 0)
There was an error

> puts divide(10, 'a')
There was an error

Salvando o erro #

Você pode salvar o erro se quiser usá-lo na cláusula rescue

def divide(x, y)
  begin
    x/y
  rescue => e
    puts "There was a %s (%s)" % [e.class, e.message]
    puts e.backtrace
  end
end

Verificando Erros Diferentes #

Se você quiser fazer coisas diferentes com base no tipo de erro, use várias cláusulas rescue, cada uma com um tipo de erro diferente como argumento.

def divide(x, y)
  begin
    return x/y
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
    return nil
  rescue TypeError
    puts "Division only works on numbers!"
    return nil
  end
end

> divide(10, 0)
Don't divide by zero!

> divide(10, 'a')
Division only works on numbers!

Se você quiser salvar o erro para uso no bloco rescue:

rescue ZeroDivisionError => e

Use uma cláusula rescue sem argumento para capturar erros de um tipo não especificado em outra cláusula rescue.

def divide(x, y)
  begin
    return x/y
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
    return nil
  rescue TypeError
    puts "Division only works on numbers!"
    return nil
  rescue => e
    puts "Don't do that (%s)" % [e.class]
    return nil
  end
end

> divide(nil, 2)
Don't do that (NoMethodError)

Neste caso, tentar dividir nil por 2 não é um ZeroDivisionError ou um TypeError, então é tratado pela cláusula padrão rescue, que imprime uma mensagem para nos informar que foi um NoMethodError `.

Tentando novamente #

Em uma cláusula rescue, você pode usar retry para executar a cláusula begin novamente, presumivelmente após alterar a circunstância que causou o erro.

def divide(x, y)
  begin
    puts "About to divide..."
    return x/y
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
    y = 1
    retry
  rescue TypeError
    puts "Division only works on numbers!"
    return nil
  rescue => e
    puts "Don't do that (%s)" % [e.class]
    return nil
  end
end

Se passarmos parâmetros que sabemos que causarão um TypeError, a cláusula begin é executada (sinalizada aqui imprimindo “About to divide”) e o erro é capturado como antes, e nil é retornado:

> divide(10, 'a')
About to divide...
Division only works on numbers!
 => nil

Mas se passarmos parâmetros que causarão um ZeroDivisionError, a cláusula begin é executada, o erro é capturado, o divisor mudou de 0 para 1, e então retry faz com que o bloco begin seja executado novamente ( de cima), agora com um y diferente. Na segunda vez, não há erro e a função retorna um valor.

> divide(10, 0)
About to divide...     # First time, 10 ÷ 0
Don't divide by zero!
About to divide...     # Second time 10 ÷ 1
=> 10

Verificando se nenhum erro foi gerado #

Você pode usar uma cláusula else para o código que será executado se nenhum erro for gerado.

def divide(x, y)
  begin
    z = x/y
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
  rescue TypeError
    puts "Division only works on numbers!"
    return nil
  rescue => e
    puts "Don't do that (%s)" % [e.class]
    return nil
  else
    puts "This code will run if there is no error."
    return z
  end
end

A cláusula else não é executada se houver um erro que transfere o controle para uma das cláusulas rescue:

> divide(10,0)
Don't divide by zero!
=> nil

Mas se nenhum erro for gerado, a cláusula else será executada:

> divide(10,2)
This code will run if there is no error.
=> 5

Observe que a cláusula else não será executada se você retornar da cláusula begin

def divide(x, y)
  begin
    z = x/y
    return z                 # Will keep the else clause from running!
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
  else
    puts "This code will run if there is no error."
    return z
  end
end

> divide(10,2)
=> 5

Código que deve sempre ser executado #

Use uma cláusula garantir se houver código que você sempre deseja executar.

def divide(x, y)
  begin
    z = x/y
    return z
  rescue ZeroDivisionError
    puts "Don't divide by zero!"
  rescue TypeError
    puts "Division only works on numbers!"
    return nil
  rescue => e
    puts "Don't do that (%s)" % [e.class]
    return nil
  ensure
    puts "This code ALWAYS runs."
  end
end

A cláusula ensure será executada quando houver um erro:

> divide(10, 0)
Don't divide by zero!   # rescue clause
This code ALWAYS runs.   # ensure clause
=> nil

E quando não há erro:

> divide(10, 2)
This code ALWAYS runs.   # ensure clause
=> 5

A cláusula de garantia é útil quando você deseja garantir, por exemplo, que os arquivos sejam fechados.

Observe que, diferentemente da cláusula else, a cláusula ensure é executada antes da cláusula begin ou rescue retornar um valor. Se a cláusula ensure tiver um return que substituirá o valor return de qualquer outra cláusula!
