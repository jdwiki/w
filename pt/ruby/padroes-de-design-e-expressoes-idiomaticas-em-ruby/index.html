Padrão de decorador #

O padrão Decorator adiciona comportamento a objetos sem afetar outros objetos da mesma classe. O padrão decorador é uma alternativa útil para criar subclasses.

Crie um módulo para cada decorador. Essa abordagem é mais flexível do que a herança porque você pode misturar e combinar responsabilidades em mais combinações. Além disso, como a transparência permite que os decoradores sejam aninhados recursivamente, ela permite um número ilimitado de responsabilidades.

Suponha que a classe Pizza tenha um método de custo que retorne 300:

class Pizza
  def cost
    300
  end
end

Represente a pizza com uma camada adicional de explosão de queijo e o custo aumenta em 50. A abordagem mais simples é criar uma subclasse PizzaWithCheese que retorna 350 no método de custo.

class PizzaWithCheese < Pizza
  def cost
    350
  end
end

Em seguida, precisamos representar uma pizza grande que adiciona 100 ao custo de uma pizza normal. Podemos representar isso usando uma subclasse LargePizza de Pizza.

class LargePizza < Pizza
  def cost
    400
  end
end

Também poderíamos ter uma ExtraLargePizza que adiciona um custo adicional de 15 à nossa LargePizza. Se considerarmos que esses tipos de pizza podem ser servidos com queijo, precisaríamos adicionar as subclasses LargePizzaWithChese e ExtraLargePizzaWithCheese. Terminamos com um total de 6 classes.

Para simplificar a abordagem, use módulos para adicionar comportamento dinamicamente à classe Pizza:

Módulo + estender + super decorador:->

class Pizza
  def cost
    300
  end
end

module CheesePizza
  def cost
    super + 50
  end
end

module LargePizza
  def cost
    super + 100
  end
end

pizza = Pizza.new         #=> cost = 300
pizza.extend(CheesePizza) #=> cost = 350
pizza.extend(LargePizza)  #=> cost = 450
pizza.cost                #=> cost = 450

Observador #

O padrão observador é um padrão de projeto de software no qual um objeto (chamado sujeito) mantém uma lista de seus dependentes (chamado observadores) e os notifica automaticamente sobre qualquer mudança de estado, geralmente chamando um de seus métodos.

Ruby fornece um mecanismo simples para implementar o padrão de projeto Observer. O módulo Observable fornece a lógica para notificar o assinante de quaisquer alterações no objeto Observable.

Para que isso funcione, o observável deve afirmar que mudou e notificar os observadores.

Objetos que estão observando precisam implementar um método update(), que será o callback para o Observer.

Vamos implementar um pequeno chat, onde os usuários podem se inscrever em usuários e quando um deles escreve algo, os assinantes são notificados.

require "observer"

class Moderator
  include Observable

  def initialize(name)
    @name = name
  end

  def write
    message = "Computer says: No"
    changed
    notify_observers(message)
  end
end

class Warner
  def initialize(moderator, limit)
    @limit = limit
    moderator.add_observer(self)
  end
end

class Subscriber < Warner
  def update(message)
    puts "#{message}"
  end
end

moderator = Moderator.new("Rupert")
Subscriber.new(moderator, 1)
moderator.write
moderator.write

Produzindo a seguinte saída:

# Computer says: No
# Computer says: No

Acionamos o método write na classe Moderator duas vezes, notificando seus assinantes, neste caso apenas um.

Quanto mais assinantes adicionarmos, mais as alterações serão propagadas.

Singleton #

Ruby Standard Library tem um módulo Singleton que implementa o padrão Singleton. O primeiro passo na criação de uma classe Singleton é exigir e incluir o módulo Singleton em uma classe:

require 'singleton'

class Logger
  include Singleton
end

Se você tentar instanciar esta classe como faria normalmente com uma classe regular, uma exceção NoMethodError é gerada. O construtor se torna privado para evitar que outras instâncias sejam criadas acidentalmente:

Logger.new

#=> NoMethodError: private method `new' called for AppConfig:Class

Para acessar a instância desta classe, precisamos usar o instance():

first, second = Logger.instance, Logger.instance
first == second

#=> true

Exemplo de registrador

require 'singleton'


class Logger
  include Singleton

  def initialize
    @log = File.open("log.txt", "a")
  end

  def log(msg)
    @log.puts(msg)
  end
end

Para usar o objeto Logger:

Logger.instance.log('message 2')

Sem Singleton inclui

As implementações singleton acima também podem ser feitas sem a inclusão do módulo Singleton. Isso pode ser alcançado com o seguinte:

class Logger
  def self.instance
    @instance ||= new
  end
end

que é uma notação abreviada para o seguinte:

class Logger
  def self.instance
    @instance = @instance || Logger.new
  end
end

No entanto, lembre-se de que o módulo Singleton é testado e otimizado, portanto, é a melhor opção para implementar seu singleton.

Procuração #

O objeto proxy é frequentemente usado para garantir acesso protegido a outro objeto, cuja lógica de negócios interna não queremos poluir com requisitos de segurança.

Suponha que gostaríamos de garantir que apenas usuários com permissões específicas possam acessar o recurso.

Definição de proxy: (garantir que apenas os usuários que realmente podem ver as reservas poderão fazer o serviço de reserva do consumidor)

class Proxy
  def initialize(current_user, reservation_service)
    @current_user = current_user
    @reservation_service = reservation_service
  end

  def highest_total_price_reservations(date_from, date_to, reservations_count)
    if @current_user.can_see_reservations?
      @reservation_service.highest_total_price_reservations(
        date_from, 
        date_to, 
        reservations_count
      )
    else
      []
    end
  end 
end

Modelos e Serviço de Reservas:

class Reservation
  attr_reader :total_price, :date

  def initialize(date, total_price)
    @date = date
    @total_price = total_price
  end
end

class ReservationService
  def highest_total_price_reservations(date_from, date_to, reservations_count)
    # normally it would be read from database/external service
    reservations = [
      Reservation.new(Date.new(2014, 5, 15), 100),
      Reservation.new(Date.new(2017, 5, 15), 10),          
      Reservation.new(Date.new(2017, 1, 15), 50)
    ]

    filtered_reservations = reservations.select do |reservation|
      reservation.date.between?(date_from, date_to) 
    end

    filtered_reservations.take(reservations_count)
  end
end        

class User
  attr_reader :name

  def initialize(can_see_reservations, name)
    @can_see_reservations = can_see_reservations
    @name = name
  end

  def can_see_reservations?
    @can_see_reservations
  end
end

Atendimento ao consumidor:

class StatsService
  def initialize(reservation_service)
    @reservation_service = reservation_service
  end

  def year_top_100_reservations_average_total_price(year)
    reservations = @reservation_service.highest_total_price_reservations(
      Date.new(year, 1, 1),
      Date.new(year, 12, 31),
      100
    )

    if reservations.length > 0
      sum = reservations.reduce(0) do |memo, reservation| 
        memo + reservation.total_price
      end

      sum / reservations.length
    else
      0
    end
  end
end

Teste:

def test(user, year)
  reservations_service = Proxy.new(user, ReservationService.new)
  stats_service = StatsService.new(reservations_service)
  average_price = stats_service.year_top_100_reservations_average_total_price(year)
  puts "#{user.name} will see: #{average_price}"
end

test(User.new(true, "John the Admin"), 2017)
test(User.new(false, "Guest"),         2017)

BENEFÍCIOS #

  • estamos evitando quaisquer alterações em ReservationService quando as restrições de acesso são alteradas.
  • não estamos misturando dados relacionados a negócios (date_from, date_to, reservations_count) com conceitos não relacionados ao domínio (permissões de usuário) no serviço.
  • Consumidor (StatsService) também está livre de lógica relacionada a permissões

RESSALVAS #

  • A interface do proxy é sempre exatamente igual ao objeto que ela oculta, de modo que o usuário que consome o serviço encapsulado pelo proxy nem sabia da presença do proxy.
