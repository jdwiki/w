O keyspace do Redis pode ser pensado como uma tabela de hash ou chaves de mapeamento de dicionário para estruturas de dados no banco de dados.

O Redis fornece uma ampla variedade de comandos que funcionam com chaves para gerenciar o keyspace, incluindo a capacidade de remover chaves, inspecionar metadados de chave, pesquisar chaves e modificar determinadas propriedades de chaves.

Sintaxe #

  • Padrão de CHAVES
  • Tecla PERSISTIR
  • EXPIRE segundos chave
  • Timestamp chave EXPIREAT
  • Tecla TTL
  • Milissegundos de chave PEXPIRE
  • carimbo de data/hora da chave PEXPIREAT
  • Tecla PTTL
  • Tecla UNLINK [tecla …]
  • Tecla DEL [tecla …]
  • Cursor SCAN [PADRÃO CORRESPONDENTE] [contagem de CONTAGEM]

Para caracteres válidos nas chaves Redis, o manual explica isso completamente:

As chaves do Redis são binárias seguras, isso significa que você pode usar qualquer sequência binária como chave, desde uma string como “foo” até o conteúdo de um arquivo JPEG. A string vazia também é uma chave válida.

Algumas outras regras sobre chaves:

Chaves muito longas não são uma boa ideia, por exemplo, uma chave de 1024 bytes é uma má ideia não apenas em termos de memória, mas também porque a pesquisa da chave no conjunto de dados pode exigir várias comparações de chave dispendiosas. Mesmo quando a tarefa em mãos é igualar a existência de um valor grande, recorrer ao hash (por exemplo com SHA1) é uma ideia melhor, especialmente do ponto de vista da memória e da largura de banda.

Teclas muito curtas geralmente não são uma boa ideia. Há pouco sentido em escrever “u1000flw” como uma chave se você puder escrever “user:1000:followers”. O último é mais legível e o espaço adicionado é menor em comparação com o espaço usado pelo próprio objeto-chave e pelo objeto de valor. Embora as teclas curtas obviamente consumam um pouco menos de memória, seu trabalho é encontrar o equilíbrio certo.

Tente ficar com um esquema. Por exemplo, “object-type:id” é uma boa ideia, como em “user:1000”. Pontos ou traços são frequentemente usados ​​para campos de várias palavras, como em “comment🔢reply.to” ou “comment🔢reply-to”.

O tamanho máximo de chave permitido é 512 MB.

Tenha cuidado ao usar o comando KEYS em um sistema de produção, pois pode causar sérios problemas de desempenho. Se você precisar fazer uma pesquisa no keyspace, os comandos SCAN são uma alternativa melhor.

Chaves válidas #

As chaves Redis são seguras para binários, portanto, literalmente, qualquer coisa pode ser usada como chave. As únicas limitações são que eles devem ter menos de 512 MB.

Exemplos de chaves válidas:

7
++++
`~!@#$%^&*()-_=+
user:10134
search/9947372/?query=this%20is%20a%28test%29%20query
<div id="div64">

Any other string less than 512MB in size.
The raw binary content of an image or other binary file.
An entire multi-line text document.
An entire SQL query.
Any integer, hexadecimal, octal, or binary value.
Anything else you can think of less than 512MB in size.

Chaves Redis inválidas:

Anything larger than 512MB.

Esquemas de nomenclatura de chaves #

Para maior clareza e facilidade de manutenção, geralmente é recomendável desenvolver um sistema ou esquema para nomear suas chaves Redis. Aqui estão alguns exemplos de sistemas comuns e passíveis de manutenção para nomear suas chaves:

user:10134
user:10134:favorites
user:10134:friends
user:10134:friends-of-friends

user:10134
user:10134/favorites
user:10134/friends
user:10134/friends.of.friends

user/10134
user/10134/favorites
user/10134/friends
user/10134/friends of friends

Observe que, embora permitido, chaves maiores usam mais memória e resultam em tempos de pesquisa mais lentos, portanto, usar uma chave de 500 MB pode não ser uma boa ideia para o desempenho. Uma ideia melhor pode ser usar um hash SHA-1, SHA-256 ou MD5 de um objeto binário grande como chave:

Listando todas as chaves #

Você pode listar todas as chaves em um banco de dados Redis executando os seguintes comandos do redis-cli:

KEYS *

O parâmetro para KEYS é uma expressão de correspondência de padrão de estilo glob. Exemplos de padrões suportados incluem:

h?llo matches hello, hallo and hxllo
h*llo matches hllo and heeeello
h[ae]llo matches hello and hallo, but not hillo
h[^e]llo matches hallo, hbllo, ... but not hello
h[a-b]llo matches hallo and hbllo

O uso do comando KEYS * pode ter efeitos adversos no desempenho, portanto, não é recomendado em instâncias de produção. Use a operação SCAN para procurar chaves no código de produção.

TTL e expiração da chave #

Os valores de expiração de uma chave podem ser gerenciados por um usuário fora dos comandos de atualização. O Redis permite que um usuário determine o tempo de vida atual (TTL) de uma chave usando o comando TTL:

TTL key

Este comando retornará o TTL de uma chave em segundos ou retornará os valores especiais -1 ou -2. Um -1 indica que a chave é persistente (não expira) e um -2 indica que a chave não existe.

Uma chave que está expirando pode se tornar persistente usando o comando PERSIST:

PERSIST KEY

e uma chave persistente pode expirar usando o comando EXPIRE:

EXPIRE KEY seconds

Expirar também pode ser usado para modificar o TTL de uma chave existente. Como alternativa, você pode usar o comando EXPIREAT com um carimbo de data/hora do UNIX para definir um tempo de expiração.

Existem versões de milissegundos de comandos TTL, EXPIRE e EXPIREAT que são prefixadas com um P.

Excluindo chaves #

O Redis fornece duas funções para remover chaves do banco de dados: del e unlink.

A função del remove uma ou mais chaves do banco de dados. O comando del faz com que o Redis recupere imediatamente a memória da chave excluída no thread de execução atual. O tempo de execução para del é proporcional ao número de elementos individuais excluídos de todas as chaves.

A função unlink atua como o comando del, remove uma ou mais chaves do banco de dados. No entanto, ao contrário do comando del, qualquer memória usada por essas chaves é recuperada de forma assíncrona em outro thread.

Verificando o Keyspace Redis #

O Redis fornece o comando SCAN para iterar as chaves no banco de dados que correspondem a um determinado padrão. O Redis oferece suporte à correspondência de padrões de estilo glob no comando SCAN.

O comando SCAN fornece um iterador baseado em cursor sobre o keyspace do Redis. A seqüência de chamada iterativa para SCAN começa com o usuário fazendo uma chamada com o argumento cursor definido como 0. O resultado dessa chamada é um lote de itens e um cursor atualizado que é fornecido para a próxima chamada para SCAN. Essa iteração continua até que o Redis retorne um cursor 0.

A seguinte função Python demonstra o uso básico de SCAN:

def scan_keys(r, pattern):
    "Returns a list of all the keys matching a given pattern"

    result = []
    cur, keys  = r.scan(cursor=0, match=pattern, count=2)
    result.extend(keys)
    while cur != 0:
        cur, keys = r.scan(cursor=cur, match=pattern, count=2)
        result.extend(keys)
        
    return result

O comando SCAN é a maneira recomendada de procurar chaves no banco de dados e é recomendado sobre o comando KEYS *.
