Os analisadores pegam o texto de um campo de string e geram tokens que serão usados ​​na consulta.

Um analisador opera em uma sequência:

  • CharFilters (Zero ou mais)
  • Tokenizador (um)
  • TokenFilters (Zero ou mais)

O analisador pode ser aplicado a mapeamentos para que, quando os campos forem indexados, isso seja feito por token, e não na string como um todo. Ao consultar, a string de entrada também será executada pelo Analyzer. Portanto, se você normalizar o texto no Analyzer, ele sempre corresponderá, mesmo que a consulta contenha uma string não normalizada.

Analisadores #

Análise no elasticsearch entra em contexto quando você deseja analisar os dados em seu índice.

Os analisadores nos permitem realizar o seguinte:

  • Abreviaturas
  • Derivação
  • Tratamento de erros de digitação

Estaremos analisando cada um deles agora.

  1. Abreviaturas:

    Using analyzers, we can tell elasticsearch how to treat abbreviations in our data i.e. dr => Doctor so whenever we search for doctor keyword in our index, elasticsearch will also return the results which have dr mentioned in them.

  2. Progressão:

    Using stemming in analyzers allows us to use base words for modified verbs like

    Word Modifications
    require requirement,required

  3. Tratamento de erros de digitação:

    Analyzers also provide typo handling as while querying if we are searching for particular word say ‘resurrection’, then elasticsearch will return the results in which typos are present.i.e. it will treat typos like resurection,ressurection as same and will retun the result.

    Word Modifications
    resurrection resurection,ressurection

Mapeamento #

Um analisador pode ser aplicado a um mapeamento usando “analisador”, por padrão, o analisador “padrão” é usado. Alternativamente, se você não deseja que nenhum analisador seja usado (porque a tokenização ou normalização não seria útil), você pode especificar “index”:“not_analyzed”

PUT my_index 
{
  "mappings": {
    "user": {
      "properties": {
        "name": {
          "type": "string"
          "analyzer":  "my_user_name_analyzer"
        },
        "id": {
          "type": "string",
          "index": "not_analyzed"
        }
      }
    }
  }
}

Vários campos #

Às vezes, pode ser útil ter vários índices distintos de um campo com analisadores diferentes. Você pode usar o recurso de vários campos para fazer isso.

PUT my_index 
{
  "mappings": {
    "user": {
      "properties": {
        "name": {
          "type": "string"
          "analyzer": "standard",
          "fields": {
            "special": {
               "type": "string",
               "analyzer": "my_user_name_analyzer"
            },
            "unanalyzed": {
              "type": "string",
              "index": "not_analyzed"
            }
          }
        }
      }
    }
  }
}

Ao consultar, em vez de simplesmente usar “user.name” (que neste caso ainda usaria o analisador padrão), você pode usar “user.name.special” ou “user.name.unanalyzed”. Observe que o documento permanecerá inalterado, isso afeta apenas a indexação.

Ignorar analisador de casos #

Às vezes, podemos precisar ignorar o caso de nossa consulta, em relação à correspondência no documento. Um analisador pode ser usado neste caso para ignorar o caso durante a pesquisa. Cada campo deverá conter este analisador em sua propriedade, para funcionar:

"settings": {
        "analysis": {
            "analyzer": {
                "case_insensitive": {
               "tokenizer": "keyword",
               "filter": ["lowercase"]
            }
            }
        }
    }
