É fácil ver types como uma tabela em um banco de dados SQL, onde o index é o banco de dados SQL. No entanto, essa não é uma boa maneira de abordar types.

Tudo sobre os tipos #

Na verdade, os tipos são literalmente apenas um campo de metadados adicionado a cada documento pelo Elasticsearch: _type. Os exemplos acima criaram dois tipos: my_type e my_other_type. Isso significa que cada documento associado aos tipos possui um campo extra definido automaticamente como "_type": "my_type"; isso é indexado com o documento, tornando-o um campo pesquisável ou filtrável, mas não afeta o documento bruto em si, portanto, seu aplicativo não precisa se preocupar com isso.

Todos os tipos vivem no mesmo índice e, portanto, nos mesmos fragmentos coletivos do índice. Mesmo no nível do disco, eles vivem nos mesmos arquivos. A única separação que a criação de um segundo tipo oferece é lógica. Cada tipo, seja exclusivo ou não, precisa existir nos mapeamentos e todos esses mapeamentos devem existir em seu estado de cluster. Isso consome memória e, se cada tipo estiver sendo atualizado dinamicamente, consome desempenho à medida que os mapeamentos mudam.

Como tal, é uma prática recomendada definir apenas um único tipo, a menos que você realmente precise de outros tipos. É comum ver cenários em que vários tipos são desejáveis. Por exemplo, imagine que você tenha um índice de carros. Pode ser útil para você dividi-lo em vários tipos:

  • BMW
  • chevy
  • honda
  • mazda
  • mercedes
  • nissan
  • ranger
  • toyota
  • …

Dessa forma, você pode pesquisar todos os carros ou limitá-lo por fabricante sob demanda. A diferença entre essas duas pesquisas é tão simples quanto:

GET /cars/_search

e

GET /cars/bmw/_search

O que não é óbvio para novos usuários do Elasticsearch é que o segundo formulário é uma especialização do primeiro formulário. Ele literalmente é reescrito para:

GET /cars/_search
{
  "query": {
    "bool": {
      "filter": [
        {
          "term" : {
            "_type": "bmw"
          }
        }
      ]
    }
  }
}

Ele simplesmente filtra qualquer documento que não tenha sido indexado com um campo _type cujo valor fosse bmw. Como todo documento é indexado com seu tipo como o campo _type, isso serve como um filtro bastante simples. Se uma pesquisa real tivesse sido fornecida em qualquer um dos exemplos, o filtro seria adicionado à pesquisa completa conforme apropriado.

Assim, se os tipos forem idênticos, é muito melhor fornecer um único tipo (por exemplo, fabricante neste exemplo) e ignorá-lo efetivamente. Então, dentro de cada documento, forneça explicitamente um campo chamado make ou qualquer nome que você preferir e filtre manualmente nele sempre que quiser limitar a ele. Isso reduzirá o tamanho de seus mapeamentos para 1/n onde n é o número de tipos separados. Ele adiciona outro campo a cada documento, com o benefício de um mapeamento simplificado.

No Elasticsearch 1.xe 2.x, esse campo deve ser definido como

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "string",
        "index": "not_analyzed"
      }
    }
  }
}
  1. O nome é arbitrário.
  2. O nome é arbitrário e pode corresponder ao nome do tipo se você quiser também.

No Elasticsearch 5.x, o acima ainda funcionará (está obsoleto), mas a melhor maneira é usar:

PUT /cars
{
  "manufacturer": { <1>
    "properties": {
      "make": { <2>
        "type": "keyword"
      }
    }
  }
}
  1. O nome é arbitrário.
  2. O nome é arbitrário e pode corresponder ao nome do tipo se você quiser também.

Os tipos devem ser usados ​​com moderação dentro de seus índices porque isso sobrecarrega os mapeamentos de índice, geralmente sem muito benefício. Você deve ter pelo menos um, mas não há nada que diga que você deve ter mais de um.

Perguntas comuns #

  • E se eu tiver dois (ou mais) tipos quase idênticos, mas que tenham alguns campos exclusivos por tipo?

No nível do índice, não há diferença entre um tipo sendo usado com alguns campos que são usados ​​esparsamente e entre vários tipos que compartilham vários campos não esparsos com alguns não compartilhados (o que significa que o outro tipo nunca usa o campo (s)).

Dito de outra forma: um campo pouco usado é esparso no índice independentemente dos tipos. A dispersão não beneficia - ou realmente prejudica - o índice apenas porque é definido em um tipo separado.

Você deve apenas combinar esses tipos e adicionar um campo de tipo separado.

  • Por que tipos separados precisam definir campos exatamente da mesma maneira?

Porque cada campo é realmente definido apenas uma vez no nível do Lucene, independentemente de quantos tipos existam. O fato de os tipos existirem é um recurso do Elasticsearch e é somente uma separação lógica.

  • Posso definir tipos separados com o mesmo campo definido de forma diferente?

Não. Se você conseguir encontrar uma maneira de fazer isso no ES 2.x ou posterior, você deve abrir um relatório de bug. Conforme observado na pergunta anterior, o Lucene vê todos eles como um único campo, portanto, não há como fazer isso funcionar adequadamente.

O ES 1.x deixou isso como um requisito implícito, que permitia aos usuários criar condições em que os mapeamentos de um estilhaço em um índice realmente diferiam de outro estilhaço no mesmo índice. Isso era efetivamente uma condição de corrida e poderia levar a problemas inesperados.

Exceções à regra #

  • Documentos pai/filho exigem tipos separados para serem usados ​​no mesmo índice.
    • The parent lives in one type.
    • The child lives in a separate type (but each child lives in the same shard as its parent).
  • Casos de uso extremamente de nicho em que a criação de toneladas de índices é indesejável e o impacto de campos esparsos é preferível à alternativa.
    • For example, the Elasticsearch monitoring plugin, Marvel (1.x and 2.x) or X-Pack Monitoring (5.x+), monitors Elasticsearch itself for changes in the cluster, nodes, indices, specific indices (the index level), and even shards. It could create 5+ indices each day to isolate those documents that have unique mappings or it could go against best practices to reduce cluster load by sharing an index (note: the number of defined mappings is effectively the same, but the number of created indices is reduced from n to 1).
    • This is an advanced scenario, but you must consider the shared field definitions across types!

Criando explicitamente um índice com um tipo #

O exemplo usa HTTP básico, que traduz facilmente para cURL e outros aplicativos HTTP. Eles também correspondem à sintaxe Sense, que será renomeada para Console no Kibana 5.0.

Nota: O exemplo insere <#> para ajudar a chamar a atenção para as partes. Esses devem ser removidos se você copiá-lo!

PUT /my_index <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "integer"
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "float"
            }
          }
        }
      }
    }
  },
  "my_other_type": {
    "properties": {
      "field1": {
        "type": "long" <3>
      },
      "field3": { <4>
        "type": "double"
      }
    }
  }
}
  1. Isso está criando o index usando o endpoint create index.
  2. Isso está criando o type.
  3. Campos compartilhados em types dentro do mesmo index devem compartilhar a mesma definição! O ES 1.x não impunha estritamente esse comportamento, mas era um requisito implícito. O ES 2.xe acima impõe estritamente esse comportamento.
  4. Campos únicos em types estão corretos.

Índices (ou índices) contain tipos. Os tipos são um mecanismo conveniente para separar documentos, mas eles exigem que você defina – de forma dinâmica/automática ou explícita – um mapeamento para cada tipo que você usa. Se você definir 15 tipos em um índice, terá 15 mapeamentos exclusivos.

Consulte as observações para obter mais detalhes sobre esse conceito e por que você pode ou não querer usar tipos.

Criando dinamicamente um índice com um tipo #

O exemplo usa HTTP básico, que traduz facilmente para cURL e outros aplicativos HTTP. Eles também correspondem à sintaxe Sense, que será renomeada para Console no Kibana 5.0.

Nota: O exemplo insere <#> para ajudar a chamar a atenção para as partes. Esses devem ser removidos se você copiá-lo!

DELETE /my_index <1>

PUT /my_index/my_type/abc123 <2>
{
  "field1" : 1234, <3>
  "field2" : 456,
  "object1" : {
    "field1" : 7.8 <4>
  }
}
  1. Caso já exista (por causa de um exemplo anterior), exclua o índice.
  2. Indexe um documento no índice, my_index, com o tipo, my_type, e o ID abc123 (pode ser numérico, mas é sempre uma string).
    • By default, dynamic index creation is enabled by simply indexing a document. This is great for development environments, but it is not necessarily good for production environments.
  3. Este campo é um número inteiro, portanto, na primeira vez que for visto, deve ser mapeado. O Elasticsearch sempre assume o tipo widest para qualquer tipo de entrada, então isso seria mapeado como um long em vez de um integer ou um short (ambos podem conter 1234 e 456).
  4. O mesmo vale para este campo. Ele será mapeado como um double em vez de um float como você pode desejar.

Esse índice e tipo criados dinamicamente correspondem aproximadamente ao mapeamento definido no primeiro exemplo. No entanto, é fundamental entender como <3> e <4> impactam os mapeamentos definidos automaticamente.

Você pode seguir isso adicionando outro tipo dinamicamente ao mesmo índice:

PUT /my_index/my_other_type/abc123 <1>
{
  "field1": 91, <2>
  "field3": 4.567
}
  1. O tipo é a única diferença do documento acima. O ID é o mesmo e tudo bem! Ele não tem relação com o outro abc123 além de acontecer estar no mesmo índice.
  2. field1 já existe no índice, então deve ser o mesmo tipo de campo definido nos outros tipos. O envio de um valor que fosse uma string ou não um número inteiro falharia (por exemplo, "field1": "this is some text" ou "field1": 123.0).

Isso criaria dinamicamente os mapeamentos para my_other_type dentro do mesmo índice, my_index.

Observação: é sempre mais rápido definir mapeamentos antecipadamente em vez de fazer com que o Elasticsearch o execute dinamicamente no momento do índice.

O resultado final da indexação de ambos os documentos seria semelhante ao primeiro exemplo, mas os tipos de campo seriam diferentes e, portanto, um pouco desnecessários:

GET /my_index/_mappings <1>
{
  "mappings": {
    "my_type": { <2>
      "properties": {
        "field1": {
          "type": "long"
        },
        "field2": {
          "type": "long" <3>
        },
        "object1": {
          "type": "object",
          "properties": {
            "field1" : {
              "type": "double" <4>
            }
          }
        }
      }
    }
  },
  "my_other_type": { <5>
    "properties": {
      "field1": {
        "type": "long"
      },
      "field3": {
        "type": "double"
      }
    }
  }
}
  1. Isso usa o endpoint _mappings para obter os mapeamentos do índice que criamos.
  2. Criamos dinamicamente my_type na primeira etapa deste exemplo.
  3. field2 agora é um long em vez de um integer porque não o definimos antecipadamente. Isso pode provar ser um desperdício no armazenamento em disco.
  4. object1.field1 agora é um duplo pela mesma razão que #3 com as mesmas ramificações que #3.
    • Technically, a long can be compressed in a lot of cases. However, a double cannot be compressed due to it being a floating point number.
  5. Também criamos dinamicamente my_other_type na segunda etapa deste exemplo. Seu mapeamento é o mesmo porque já estávamos usando long e double.
    • Remember that field1 must match the definition from my_type (and it does).
    • field3 is unique to this type, so it has no such restriction.
