O Elasticsearch vem com um conjunto de padrões que fornecem uma boa experiência pronta para o desenvolvimento. A afirmação implícita é que não é necessariamente ótimo para a produção, que deve ser adaptada às suas próprias necessidades e, portanto, não pode ser prevista.

As configurações padrão facilitam o download e a execução de vários nós na mesma máquina sem nenhuma alteração de configuração.

Onde estão as configurações? #

Dentro de cada instalação do Elasticsearch há um arquivo config/elasticsearch.yml. É onde estão as seguintes configurações:

  • cluster.name
    • The name of the cluster that the node is joining. All nodes in the same cluster must share the same name.
    • Currently defaults to elasticsearch.
  • nó.*
    • node.name
      • If not supplied, a random name will be generated each time the node starts. This can be fun, but it is not good for production environments.
      • Names do not have to be unique, but they should be unique.
    • node.master
      • A boolean setting. When true, it means that the node is an eligible master node and it can be the elected master node.
      • Defaults to true, meaning every node is an eligible master node.
    • node.data
      • A boolean setting. When true, it means that the node stores data and handles search activity.
      • Defaults to true.
  • caminho.*
    • path.data
      • The location that files are written for the node. All nodes use this directory to store metadata, but data nodes will also use it to store/index documents.
      • Defaults to ./data.
        • This means that data will be created for you as a peer directory to config inside of the Elasticsearch directory.
    • path.logs
      • The location that log files are written.
      • Defaults to ./logs.
  • rede.*

    • network.host

      • Defaults to _local_, which is effectively localhost.
        • This means that, by default, nodes cannot be communicated with from outside of the current machine!

    • network.bind_host

      • Potentially an array, this tells Elasticsearch what addresses of the current machine to bind sockets too.
        • It is this list that enables machines from outside of the machine (e.g., other nodes in the cluster) to talk to this node.
      • Defaults to network.host.

    • network.publish_host

      • A singular host that is used to advertise to other nodes how to best communicate with this node.
        • When supplying an array to network.bind_host, this should be the one host that is intended to be used for inter-node communication.
      • Defaults to network.host`.
  • discovery.zen.*
    • discovery.zen.minimum_master_nodes
      • Defines quorum for master election. This must be set using this equation: (M / 2) + 1 where M is the number of eligible master nodes (nodes using node.master: true implicitly or explicitly).
      • Defaults to 1, which only is valid for a single node cluster!
    • discovery.zen.ping.unicast.hosts
      • The mechanism for joining this node to the rest of a cluster.
      • This should list eligible master nodes so that a node can find the rest of the cluster.
      • The value that should be used here is the network.publish_host of those other nodes.
      • Defaults to localhost, which means it only looks on the local machine for a cluster to join.

Que tipo de configurações existem? #

O Elasticsearch fornece três tipos diferentes de configurações:

  • Configurações de todo o cluster
    • These are settings that apply to everything in the cluster, such as all nodes or all indices.
  • Configurações de nós
    • These are settings that apply to just the current node.
  • Configurações de índice
    • These are settings that apply to just the index.

Dependendo da configuração, pode ser:

  • Alterado dinamicamente em tempo de execução
  • Alterado após uma reinicialização (fechar/abrir) do índice
    • Some index-level settings do not require the index to be closed and reopened, but might require the index to be forceably re-merged for the setting to apply.
      • The compression level of an index is an example of this type of setting. It can be changed dynamically, but only new segments take advantage of the change. So if an index will not change, then it never takes advantage of the change unless you force the index to recreate its segments.
  • Alterado após uma reinicialização do nó
  • Alterado após uma reinicialização do cluster
  • Nunca mudou

Sempre verifique a documentação da sua versão do Elasticsearch para saber o que você pode ou não fazer com uma configuração.

Como posso aplicar as configurações? #

Você pode definir as configurações de algumas maneiras, algumas das quais não são sugeridas:

  • Argumentos da linha de comando

No Elasticsearch 1.xe 2.x, você pode enviar a maioria das configurações como Java System Properties prefixado com es.:

$ bin/elasticsearch -Des.cluster.name=my_cluster -Des.node.name=`hostname`

No Elasticsearch 5.x, isso muda para evitar o uso de Java System Properties, em vez de usar um tipo de argumento personalizado com -E substituindo -Des.:

$ bin/elasticsearch -Ecluster.name=my_cluster -Enode.name=`hostname`

Essa abordagem de aplicação de configurações funciona muito bem ao usar ferramentas como Puppet, Chef ou Ansible para iniciar e parar o cluster. No entanto, funciona muito mal ao fazê-lo manualmente.

  • Configurações YAML
    • Shown in examples
  • Configurações dinâmicas
    • Shown in examples

A ordem em que as configurações são aplicadas está na ordem mais dinâmica:

  1. Configurações transitórias
  2. Configurações persistentes
  3. Configurações de linha de comando
  4. Configurações de YAML (estático)

Se a configuração for definida duas vezes, uma vez em qualquer um desses níveis, o nível mais alto entrará em vigor.

Configurações de índice dinâmico para vários índices ao mesmo tempo #

Você pode aplicar a mesma alteração mostrada no exemplo Index Settings para todos os índices existentes com uma solicitação, ou até mesmo um subconjunto deles:

PUT /*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

ou

PUT /_all/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

ou

PUT /_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Se você preferir fazer isso de forma mais seletiva também, poderá selecionar vários sem fornecer todos:

PUT /logstash-*,my_other_index,some-other-*/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Configurações estáticas do Elasticsearch #

O Elasticsearch usa um arquivo de configuração YAML (Yet Another Markup Language) que pode ser encontrado dentro do diretório padrão do Elasticsearch ([instalações RPM e DEB alteram este local entre outras coisas](https://www.elastic.co/guide/en/elasticsearch /reference/current/setup-dir-layout.html)).

Você pode definir configurações básicas em config/elasticsearch.yml:

# Change the cluster name. All nodes in the same cluster must use the same name!
cluster.name: my_cluster_name

# Set the node's name using the hostname, which is an environment variable!
# This is a convenient way to uniquely set it per machine without having to make
#  a unique configuration file per node.
node.name: ${HOSTNAME}

# ALL nodes should set this setting, regardless of node type
path.data: /path/to/store/data

# This is a both a master and data node (defaults)
node.master: true
node.data: true

# This tells Elasticsearch to bind all sockets to only be available
#  at localhost (default)
network.host: _local_

Configurações de cluster dinâmico persistente #

Se você precisa aplicar uma configuração dinamicamente depois que o cluster já foi iniciado, e ela pode ser definida dinamicamente, então você pode defini-la usando a API _cluster/settings.

As configurações persistentes são um dos dois tipos de configurações em todo o cluster que podem ser aplicadas. Uma configuração persistente vai sobreviver a uma reinicialização completa do cluster.

Nota: Nem todas as configurações podem ser aplicadas dinamicamente. Por exemplo, o nome do cluster não pode ser renomeado dinamicamente. A maioria das configurações de nível de nó também não pode ser definida dinamicamente (porque elas não podem ser direcionadas individualmente).

Esta não é a API a ser usada para definir as configurações no nível do índice. Você pode dizer que a configuração é uma configuração de nível de índice porque deve começar com index.. Configurações cujo nome está na forma de índices. são configurações de todo o cluster porque se aplicam a todos os índices.

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}

Aviso: No Elasticsearch 1.xe 2.x, você não pode unset uma configuração persistente.

Felizmente, isso foi aprimorado no Elasticsearch 5.xe agora você pode remover uma configuração definindo-a como null:

POST /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}

Uma configuração não definida retornará ao padrão ou a qualquer valor definido em um nível de prioridade mais baixo (por exemplo, configurações de linha de comando).

Configurações de cluster dinâmico transitório #

Se você precisa aplicar uma configuração dinamicamente depois que o cluster já foi iniciado, e ela pode ser definida dinamicamente, então você pode defini-la usando a API _cluster/settings.

As configurações transitórias são um dos dois tipos de configurações em todo o cluster que podem ser aplicadas. Uma configuração transitória não sobreviverá a uma reinicialização completa do cluster.

Nota: Nem todas as configurações podem ser aplicadas dinamicamente. Por exemplo, o nome do cluster não pode ser renomeado dinamicamente. A maioria das configurações de nível de nó também não pode ser definida dinamicamente (porque elas não podem ser direcionadas individualmente).

Esta não é a API a ser usada para definir as configurações no nível do índice. Você pode dizer que a configuração é uma configuração de nível de índice porque deve começar com index.. Configurações cujo nome está na forma de índices. são configurações de todo o cluster porque se aplicam a todos os índices.

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": "none"
  }
}

Aviso: no Elasticsearch 1.xe 2.x, não é possível desdefinir configurações transitórias sem uma reinicialização completa do cluster.

Felizmente, isso foi aprimorado no Elasticsearch 5.xe agora você pode remover uma configuração definindo-a como null:

POST /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.enable": null
  }
}

Uma configuração não definida retornará ao padrão ou a qualquer valor definido em um nível de prioridade mais baixo (por exemplo, configurações persistentes).

Configurações de índice #

As configurações de índice são aquelas que se aplicam a um único índice. Essas configurações começarão com index.. A exceção a essa regra é number_of_shards e number_of_replicas, que também existem na forma de index.number_of_shards e index.number_of_replicas.

Como o nome sugere, as configurações de nível de índice se aplicam a um único índice. Algumas configurações devem ser aplicadas no momento da criação porque não podem ser alteradas dinamicamente, como a configuração index.number_of_shards, que controla o número de shards primários para o índice.

PUT /my_index
{
  "settings": {
    "index.number_of_shards": 1,
    "index.number_of_replicas": 1
  }
}

ou, em um formato mais conciso, você pode combinar prefixos de chave em cada .:

PUT /my_index
{
  "settings": {
    "index": {
      "number_of_shards": 1,
      "number_of_replicas": 1
    }
  }
}

Os exemplos acima criarão um índice com as configurações fornecidas. Você pode alterar dinamicamente as configurações por índice usando o ponto de extremidade _settings do índice. Por exemplo, aqui alteramos dinamicamente as configurações de log lento para only o nível de aviso:

PUT /my_index/_settings
{
  "index": {
    "indexing.slowlog.threshold.index.warn": "1s",
    "search.slowlog.threshold": {
      "fetch.warn": "500ms",
      "query.warn": "2s"
    }
  }
}

Aviso: O Elasticsearch 1.xe 2.x não validou muito estritamente os nomes de configuração no nível do índice. Se você tivesse um erro de digitação, ou simplesmente inventasse uma configuração, ele a aceitaria cegamente, mas a ignoraria. O Elasticsearch 5.x valida estritamente os nomes das configurações e rejeitará qualquer tentativa de aplicar configurações de índice com configurações desconhecidas (devido a erro de digitação ou plug-in ausente). Ambas as instruções se aplicam a configurações de índice que mudam dinamicamente e no momento da criação.
