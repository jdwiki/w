O comando cut é uma maneira rápida de extrair partes de linhas de arquivos de texto. Pertence aos comandos mais antigos do Unix. Suas implementações mais populares são a versão GNU encontrada no Linux e a versão FreeBSD encontrada no MacOS, mas cada versão do Unix tem sua própria. Veja abaixo as diferenças. As linhas de entrada são lidas de stdin ou de arquivos listados como argumentos na linha de comando.

Sintaxe #

  • cut -f1,3 # extrai primeiro e terceiro tab-delimited field (de stdin)

  • cut -f1-3 # extrai from first up to third field (inclui extremidades)

  • cut -f-3 # -3 é interpretado como 1-3

  • cut -f2- # 2- é interpretado como from the second to the last

  • cut -c1-5,10 # extrai de stdin os characters nas posições 1,2,3,4,5,10

  • cut -s -f1 # suprime linhas que não contenham delimitadores

  • cut –complement -f3 # (somente GNU cut) extrai todos campos exceto o terceiro

Parâmetros #

Parâmetro Detalhes
-f, –campos Seleção baseada em campo
-d, –delimiter Delimitador para seleção baseada em campo
-c, –caracteres Seleção baseada em caracteres, delimitador ignorado ou erro
-s, –somente delimitado Suprimir linhas sem caracteres delimitadores (impressas no estado em que se encontram)
–complemento Seleção invertida (extrair todos os campos/caracteres exceto especificados
–delimitador de saída Especifique quando deve ser diferente do delimitador de entrada

1. Diferenças de sintaxe

As opções longas na tabela acima são suportadas apenas pela versão GNU.

2. Nenhum personagem recebe tratamento especial

O cut do FreeBSD (que vem com o MacOS, por exemplo) não possui a opção --complement e, no caso de intervalos de caracteres, pode-se usar o comando colrm:

  $ cut --complement -c3-5 <<<"123456789"
  126789

  $ colrm 3 5 <<<"123456789"
  126789

No entanto, há uma grande diferença, porque o colrm trata os caracteres TAB (ASCII 9) como tabulações reais até o próximo múltiplo de oito, e backspaces (ASCII 8) como -1 de largura; pelo contrário, cut trata todos os caracteres como uma coluna de largura.

  $ colrm  3 8 <<<$'12\tABCDEF' # Input string has an embedded TAB
  12ABCDEF

  $ cut --complement -c3-8 <<<$'12\tABCDEF'
  12F

3. (Ainda não) Internacionalização

Quando o cut foi projetado, todos os caracteres tinham um byte de comprimento e a internacionalização não era um problema. Quando sistemas de escrita com caracteres mais largos se tornaram populares, a solução adotada pelo POSIX foi distinguir entre a antiga chave -c, que deveria manter seu significado de selecionar caracteres, não importa quantos bytes de largura, e introduzir uma nova chave -b que deve selecionar bytes, independente da codificação de caracteres atual. Na maioria das implementações populares, -b foi introduzido e funciona, mas -c ainda está funcionando exatamente como -b e não como deveria. Por exemplo com GNU cut:

Parece que o filtro de spam do SE coloca na lista negra textos em inglês com caracteres kanji isolados neles. Não consegui superar essa limitação, então os exemplos a seguir são menos expressivos do que poderiam ser.

  # In an encoding where each character in the input string is three bytes wide,
  # Selecting bytes 1-6 yields the first two characters (correct)
  $ LC_ALL=ja_JP.UTF-8 cut -b1-6 kanji.utf-8.txt
  ...first two characters of each line...
  

  # Selecting all three characters with the -c switch doesn’t work.
  # It behaves like -b, contrary to documentation.
  $ LC_ALL=ja_JP.UTF-8 cut -c1-3 kanji.utf-8.txt
  ...first character of each line...

  # In this case, an illegal UTF-8 string is produced.
  # The -n switch would prevent this, if implemented.
  $ LC_ALL=ja_JP.UTF-8 cut -n -c2 kanji.utf-8.txt
  ...second byte, which is an illegal UTF-8 sequence...

Se seus caracteres estiverem fora do intervalo ASCII e você quiser usar cut, você deve sempre estar ciente da largura dos caracteres em sua codificação e usar -b de acordo. Se e quando -c começar a funcionar conforme documentado, você não precisará alterar seus scripts.

4. Comparações de velocidade

As limitações do cut fazem com que as pessoas duvidem de sua utilidade. Na verdade, a mesma funcionalidade pode ser alcançada por utilitários mais poderosos e populares. No entanto, a vantagem do cut é o seu performance. Veja abaixo algumas comparações de velocidade. test.txt tem três milhões de linhas, com cinco campos separados por espaço cada. Para o teste awk, mawk foi usado, porque é mais rápido que o GNU awk. O próprio shell (última linha) é de longe o pior desempenho. Os tempos dados (em segundos) são o que o comando time dá como tempo real.

(Apenas para evitar mal-entendidos: todos os comandos testados deram a mesma saída com a entrada fornecida, mas é claro que não são equivalentes e dariam saídas diferentes em diferentes situações, em particular se os campos fossem delimitados por um número variável de espaços )

Comando Hora
cut -d ' ' -f1,2 test.txt 1.138s
awk '{print $1 $2}' test.txt 1.688s
join -a1 -o1.1,1.2 test.txt /dev/null 1.767s
perl -lane 'print "@F[1,2]"' test.txt 11.390s
grep -o '^\([^ ]*\) \([^ ]*\)' test.txt 22.925s

Apenas um caractere delimitador #

Você não pode ter mais de um delimitador: se você especificar algo como -d ",;:", algumas implementações usarão apenas o primeiro caractere como delimitador (neste caso, a vírgula.) Outras implementações (por exemplo, GNU cut ) lhe dará uma mensagem de erro.

$ cut -d ",;:" -f2 <<<"J.Smith,1 Main Road,cell:1234567890;land:4081234567"
cut: the delimiter must be a single character
Try `cut --help' for more information.

Delimitadores repetidos são interpretados como campos vazios #

$ cut -d, -f1,3 <<<"a,,b,c,d,e"
a,b

é bastante óbvio, mas com strings delimitadas por espaço pode ser menos óbvio para alguns

$ cut -d ' ' -f1,3 <<<"a  b c d e"
a b

cut não pode ser usado para analisar argumentos como o shell e outros programas fazem.

Sem cotação #

Não há como proteger o delimitador. Planilhas e softwares semelhantes de manipulação de CSV geralmente podem reconhecer um caractere de citação de texto que possibilita definir strings contendo um delimitador. Com cut você não pode.

$ cut -d, -f3 <<<'John,Smith,"1, Main Street"'
"1

Extraindo, não manipulando #

Você só pode extrair partes de linhas, não reordenar ou repetir campos.

$ cut -d, -f2,1 <<<'John,Smith,USA' ## Just like -f1,2
John,Smith
$ cut -d, -f2,2 <<<'John,Smith,USA' ## Just like -f2
Smith

Uso básico #

O uso típico é com arquivos do tipo CSV, onde cada linha consiste em campos separados por um delimitador, especificado pela opção -d. O delimitador padrão é o caractere TAB. Suponha que você tenha um arquivo de dados data.txt com linhas como

0 0 755 1482941948.8024
102 33 4755 1240562224.3205
1003 1 644 1219943831.2367

Então

# extract the third space-delimited field
$ cut -d ' ' -f3 data.txt
755
4755
644

# extract the second dot-delimited field
$ cut -d. -f2 data.txt    
8024
3205
2367

# extract the character range from the 20th through the 25th character
$ cut -c20-25 data.txt 
948.80
056222    
943831

Como de costume, pode haver espaços opcionais entre um switch e seu parâmetro: -d, é o mesmo que -d,

GNU cut permite especificar uma opção --output-delimiter: (um recurso independente deste exemplo é que um ponto e vírgula como delimitador de entrada deve ser escapado para evitar seu tratamento especial pelo shell)

$ cut --output-delimiter=, -d\; -f1,2 <<<"a;b;c;d"
a,b
