Sintaxe #

  • [tempo [-p]] [!] comando1 [ | ou |& comando2 ] …

Um pipeline é uma sequência de comandos simples separados por um dos operadores de controle | ou |& (source).

| conecta a saída de command1 à entrada de command2.

|& conecta a saída padrão e o erro padrão de command1 à entrada padrão de command2.

Usando |& #

|& conecta a saída padrão e o erro padrão do primeiro comando ao segundo, enquanto o | conecta apenas a saída padrão do primeiro comando ao segundo comando.

Neste exemplo, a página é baixada via curl. com a opção -v curl escreve algumas informações em stderr incluindo , a página baixada é escrita em stdout. O título da página pode ser encontrado entre <title> e </title>.

curl -vs 'http://www.google.com/' |& awk '/Host:/{print} /<title>/{match($0,/<title>(.*)<\/title>/,a);print a[1]}'

A saída é:

> Host: www.google.com
Google

Mas com | muito mais informações serão impressas, ou seja, aquelas que são enviadas para stderr porque apenas stdout é canalizado para o próximo comando. Neste exemplo, todas as linhas, exceto a última linha (Google), foram enviadas para stderr por curl:

Mostra todos os processos paginados #

ps -e | less

ps -e mostra todos os processos, sua saída é conectada à entrada de more via |, less pagina os resultados.

Modifica a saída contínua de um comando #

~$ ping -c 1 google.com # unmodified output
PING google.com (16.58.209.174) 56(84) bytes of data.
64 bytes from wk-in-f100.1e100.net (16.58.209.174): icmp_seq=1 ttl=53 time=47.4 ms
~$ ping google.com | grep -o '^[0-9]\+[^()]\+' # modified output
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
64 bytes from wk-in-f100.1e100.net 
...

O pipe (|) conecta o stdout do ping ao stdin do grep, que o processa imediatamente. Alguns outros comandos como sed padronizam para armazenar em buffer seu stdin, o que significa que ele precisa receber dados suficientes antes de imprimir qualquer coisa, potencialmente causando atrasos no processamento posterior.
