Sintaxe #

  • comando </path/to/file # Redireciona a entrada padrão para o arquivo
  • command >/path/to/file # Redireciona a saída padrão para voar
  • comando file_descriptor>/path/to/file # Redireciona a saída do file_descriptor para o arquivo
  • command >&file_descriptor # Redireciona a saída para file_descriptor
  • comando file_descriptor>&another_file_descriptor # Redireciona file_descriptor para another_file_descriptor
  • command <&file_descriptor # Redireciona o file_descriptor para a entrada padrão
  • command &>/path/to/file # Redireciona a saída padrão e o erro padrão para o arquivo

Parâmetros #

Parâmetro Detalhes
descritor de arquivo interno Um número inteiro.
direção Um de >, < ou <>
descritor ou caminho de arquivo externo & seguido por um número inteiro para descritor de arquivo ou um caminho.

Os programas de console UNIX têm um arquivo de entrada e dois arquivos de saída (fluxos de entrada e saída, bem como dispositivos, são tratados como arquivos pelo sistema operacional). Normalmente, são o teclado e a tela, respectivamente, mas qualquer um ou todos eles podem ser redirecionados vir de — ou ir para — um arquivo ou outro programa.

STDIN é a entrada padrão e é como o programa recebe a entrada interativa. STDIN é geralmente atribuído ao descritor de arquivo 0.

STDOUT é a saída padrão. O que quer que seja emitido em STDOUT é considerado o “resultado” do programa. STDOUT geralmente é atribuído ao descritor de arquivo 1.

STDERR é onde as mensagens de erro são exibidas. Normalmente, ao executar um programa do console, STDERR é exibido na tela e é indistinguível de STDOUT. STDERR geralmente é atribuído ao descritor de arquivo 2.

A ordem de redirecionamento é importante

command > file 2>&1

Redireciona ambos (STDOUT e STDERR) para o arquivo.

command 2>&1 > file

Redireciona apenas STDOUT, pois o descritor de arquivo 2 é redirecionado para o arquivo apontado pelo descritor de arquivo 1 (que ainda não é o arquivo file quando a instrução é avaliada).

Cada comando em um pipeline tem seu próprio STDERR (e STDOUT) porque cada um é um novo processo. Isso pode criar resultados surpreendentes se você espera que um redirecionamento afete todo o pipeline. Por exemplo, este comando (embrulhado para legibilidade):

$ python -c 'import sys;print >> sys.stderr, "Python error!"' \
| cut -f1 2>> error.log

imprimirá “Erro do Python!” para o console em vez do arquivo de log. Em vez disso, anexe o erro ao comando que você deseja capturar:

$ python -c 'import sys;print >> sys.stderr, "Python error!"' 2>> error.log \
| cut -f1

Anexar vs Truncar #

Truncar > #

  1. Crie o arquivo especificado se ele não existir.

  2. Truncar (remover o conteúdo do arquivo)

  3. Gravar no arquivo

    $ echo “first line” > /tmp/lines $ echo “second line” > /tmp/lines

    $ cat /tmp/lines second line

Anexar >> #

  1. Crie o arquivo especificado se ele não existir.

  2. Anexar arquivo (escrevendo no final do arquivo).

    Overwrite existing file #

    $ echo “first line” > /tmp/lines

    Append a second line #

    $ echo “second line” » /tmp/lines

    $ cat /tmp/lines first line second line

Redirecionando STDOUT e STDERR #

Descritores de arquivo como 0 e 1 são ponteiros. Alteramos o que os descritores de arquivo apontam com o redirecionamento. >/dev/null significa 1 aponta para /dev/null.

Primeiro apontamos 1 (STDOUT) para /dev/null e depois apontamos 2 (STDERR) para o que 1 apontar.

# STDERR is redirect to STDOUT: redirected to /dev/null,
# effectually redirecting both STDERR and STDOUT to /dev/null
echo 'hello' > /dev/null 2>&1

Isso pode ser abreviado para o seguinte:

echo 'hello' &> /dev/null

No entanto, essa forma pode ser indesejável na produção se a compatibilidade do shell for uma preocupação, pois entra em conflito com o POSIX, introduz ambiguidade na análise e os shells sem esse recurso o interpretarão incorretamente:

# Actual code
echo 'hello' &> /dev/null
echo 'hello' &> /dev/null 'goodbye'

# Desired behavior
echo 'hello' > /dev/null 2>&1
echo 'hello' 'goodbye' > /dev/null 2>&1

# Actual behavior
echo 'hello' &
echo 'hello' & goodbye > /dev/null

NOTA: Sabe-se que &> funciona conforme desejado no Bash e no Zsh.

Redirecionando a saída padrão #

> redireciona a saída padrão (também conhecida como STDOUT) do comando atual para um arquivo ou outro descritor.

Estes exemplos gravam a saída do comando ls no arquivo file.txt

ls >file.txt
> file.txt ls

O arquivo de destino é criado se não existir, caso contrário, esse arquivo será truncado.

O descritor de redirecionamento padrão é a saída padrão ou 1 quando nenhum é especificado. Este comando é equivalente aos exemplos anteriores com a saída padrão indicada explicitamente:

ls 1>file.txt

Nota: o redirecionamento é inicializado pelo shell executado e não pelo comando executado, portanto é feito antes da execução do comando.

Usando pipes nomeados #

Às vezes você pode querer produzir algo por um programa e inseri-lo em outro programa, mas não pode usar um pipe padrão.

ls -l | grep ".log"

Você poderia simplesmente escrever em um arquivo temporário:

touch tempFile.txt
ls -l > tempFile.txt
grep ".log" < tempFile.txt

Isso funciona bem para a maioria dos aplicativos, no entanto, ninguém saberá o que tempFile faz e alguém pode removê-lo se ele contiver a saída de ls -l nesse diretório. É aqui que um pipe nomeado entra em ação:

mkfifo myPipe
ls -l > myPipe
grep ".log" < myPipe

myPipe é tecnicamente um arquivo (tudo está no Linux), então vamos fazer ls -l em um diretório vazio no qual acabamos de criar um pipe:

mkdir pipeFolder
cd pipeFolder
mkfifo myPipe
ls -l

A saída é:

prw-r--r-- 1 root root 0 Jul 25 11:20 myPipe

Observe o primeiro caractere nas permissões, ele está listado como um pipe, não um arquivo.

Agora vamos fazer algo legal.

Abra um terminal e anote o diretório (ou crie um para que a limpeza seja fácil) e faça um pipe.

mkfifo myPipe

Agora vamos colocar algo no tubo.

echo "Hello from the other side" > myPipe

Você notará que isso trava, o outro lado do tubo ainda está fechado. Vamos abrir o outro lado do cano e deixar essas coisas passarem.

Abra outro terminal e vá para o diretório em que o pipe está (ou se você souber, adicione-o ao pipe):

cat < myPipe

Você notará que após a saída do hello from the other side, o programa no primeiro terminal termina, assim como no segundo terminal.

Agora execute os comandos ao contrário. Comece com cat < myPipe e então echo algo nele. Ainda funciona, porque um programa espera até que algo seja colocado no pipe antes de terminar, porque sabe que precisa obter algo.

Pipes nomeados podem ser úteis para mover informações entre terminais ou entre programas.

Os tubos são pequenos. Uma vez cheio, o gravador bloqueia até que algum leitor leia o conteúdo, então você precisa executar o leitor e o gravador em terminais diferentes ou executar um ou outro em segundo plano:

 ls -l /tmp > myPipe &
 cat < myPipe

Mais exemplos usando pipes nomeados:

  • Exemplo 1 - todos os comandos no mesmo terminal/mesmo shell

     $ { ls -l && cat file3; } >mypipe &
 $ cat <mypipe    
 # Output: Prints ls -l data and then prints file3 contents on screen

  • Exemplo 2 - todos os comandos no mesmo terminal/mesmo shell

     $ ls -l >mypipe &
 $ cat file3 >mypipe &
 $ cat <mypipe
 #Output: This prints on screen the contents of mypipe.

Lembre-se de que o primeiro conteúdo de file3 é exibido e, em seguida, os dados de ls -l são exibidos (configuração LIFO).

  • Exemplo 3 - todos os comandos no mesmo terminal/mesmo shell

     $ { pipedata=$(<mypipe) && echo "$pipedata"; } &
 $ ls >mypipe 
 # Output: Prints the output of ls directly on screen

    Mind that the variable $pipedata is not available for usage in the main terminal / main shell since the use of & invokes a subshell and $pipedata was only available in this subshell.

  • Exemplo 4 - todos os comandos no mesmo terminal/mesmo shell

     $ export pipedata
 $ pipedata=$(<mypipe) &
 $ ls -l *.sh >mypipe
 $ echo "$pipedata"   
 #Output : Prints correctly the contents of mypipe

Isso imprime corretamente o valor da variável $pipedata no shell principal devido à declaração de exportação da variável. O terminal principal/shell principal não está suspenso devido à invocação de um shell de segundo plano (&).

Redirecionando vários comandos para o mesmo arquivo #

{
  echo "contents of home directory"
  ls ~
} > output.txt

Imprimir mensagens de erro para stderr #

As mensagens de erro geralmente são incluídas em um script para fins de depuração ou para fornecer uma experiência avançada ao usuário. Simplesmente escrevendo mensagem de erro como esta:

cmd || echo 'cmd failed'

pode funcionar para casos simples, mas não é a maneira usual. Neste exemplo, a mensagem de erro poluirá a saída real do script misturando erros e saída bem-sucedida em stdout.

Em resumo, a mensagem de erro deve ir para stderr e não para stdout. É bem simples:

cmd || echo 'cmd failed' >/dev/stderr

Outro exemplo:

if cmd; then
    echo 'success'
else
    echo 'cmd failed' >/dev/stderr
fi

No exemplo acima, a mensagem de sucesso será impressa em stdout enquanto a mensagem de erro será impressa em stderr.

Uma maneira melhor de imprimir a mensagem de erro é definir uma função:

err(){
    echo "E: $*" >>/dev/stderr
}

Agora, quando você tem que imprimir um erro:

err "My error message"

Redirecionamento para endereços de rede #

Bash trata alguns caminhos como especiais e pode fazer alguma comunicação de rede escrevendo para /dev/{udp|tcp}/host/port. O Bash não pode configurar um servidor de escuta, mas pode iniciar uma conexão e, para o TCP, pode pelo menos ler os resultados.

Por exemplo, para enviar uma requisição web simples pode-se fazer:

exec 3</dev/tcp/www.google.com/80
printf 'GET / HTTP/1.0\r\n\r\n' >&3
cat <&3

e os resultados da página padrão de www.google.com serão impressos em stdout.

De forma similar

printf 'HI\n' >/dev/udp/192.168.1.1/6666

enviaria uma mensagem UDP contendo HI\n para um ouvinte em 192.168.1.1:6666

Redirecionando STDIN #

< lê de seu argumento direito e escreve em seu argumento esquerdo.

Para escrever um arquivo em STDIN devemos ler /tmp/a_file e write em STDIN ou seja, 0</tmp/a_file

Nota: O padrão do descritor de arquivo interno é 0 (STDIN) para <

$ echo "b" > /tmp/list.txt
$ echo "a" >> /tmp/list.txt
$ echo "c" >> /tmp/list.txt
$ sort < /tmp/list.txt
a
b
c

Redirecionando STDERR #

2 é STDERR.

$ echo_to_stderr 2>/dev/null # echos nothing

Definições:

echo_to_stderr é um comando que escreve "stderr" para STDERR

echo_to_stderr () {
    echo stderr >&2
}

$ echo_to_stderr
stderr

STDIN, STDOUT e STDERR explicados #

Os comandos têm uma entrada (STDIN) e dois tipos de saídas, saída padrão (STDOUT) e erro padrão (STDERR).

Por exemplo:

STDIN

root@server~# read
Type some text here

A entrada padrão é usada para fornecer entrada para um programa. (Aqui estamos usando o read builtin para ler uma linha de STDIN.)

STDOUT

root@server~# ls file
file

A saída padrão geralmente é usada para saída “normal” de um comando. Por exemplo, ls lista arquivos, então os arquivos são enviados para STDOUT.

STDERR

root@server~# ls anotherfile
ls: cannot access 'anotherfile': No such file or directory

O erro padrão é (como o nome indica) usado para mensagens de erro. Como esta mensagem não é uma lista de arquivos, ela é enviada para STDERR.

STDIN, STDOUT e STDERR são os três fluxos padrão. Eles são identificados no shell por um número em vez de um nome:

0 = Padrão em 1 = Saída padrão 2 = Erro padrão

Por padrão, STDIN é conectado ao teclado e STDOUT e STDERR aparecem no terminal. No entanto, podemos redirecionar STDOUT ou STDERR para o que precisarmos. Por exemplo, digamos que você só precise da saída padrão e todas as mensagens de erro impressas no erro padrão devem ser suprimidas. É quando usamos os descritores 1 e 2.

** Redirecionando STDERR para /dev/null** Tomando o exemplo anterior,

root@server~# ls anotherfile 2>/dev/null
root@server~#

Nesse caso, se houver algum STDERR, ele será redirecionado para /dev/null (um arquivo especial que ignora qualquer coisa colocada nele), para que você não obtenha nenhuma saída de erro no shell.
