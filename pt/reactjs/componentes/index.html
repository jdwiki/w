React.createClass foi obsoleto na v15.5 e deve ser removido na v16. Há um pacote de substituição drop-in para aqueles que ainda precisam dele. Os exemplos que o utilizam devem ser atualizados.

Criando componentes #

Esta é uma extensão do Exemplo Básico:

Estrutura básica #

import React, { Component } from 'react';
import { render } from 'react-dom';

class FirstComponent extends Component {
    render() {
        return (
            <div>
                Hello, {this.props.name}! I am a FirstComponent.
            </div>
        );
    }
}

render(
    <FirstComponent name={ 'User' } />,
    document.getElementById('content')
);

O exemplo acima é chamado de componente stateless porque não contém [state](https://www.wikiod.com/pt/reactjs/estado-em-reagir#Estado Básico ) (no sentido React da palavra).

Nesse caso, algumas pessoas acham preferível usar componentes funcionais sem estado, que são baseados em [funções de seta ES6](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Functions/ Arrow_functions).

Componentes funcionais sem estado #

Em muitos aplicativos, existem componentes inteligentes que mantêm o estado, mas renderizam componentes burros que simplesmente recebem props e retornam HTML como JSX. Os componentes funcionais sem estado são muito mais reutilizáveis ​​e têm um impacto positivo no desempenho do seu aplicativo.

Possuem 2 características principais:

  1. Quando renderizados, eles recebem um objeto com todos os adereços que foram passados
  2. Eles devem retornar o JSX para ser renderizado
// When using JSX inside a module you must import React
import React from 'react';
import PropTypes from 'prop-types';

const FirstComponent = props => (
    <div>
        Hello, {props.name}! I am a FirstComponent.
    </div>
);

//arrow components also may have props validation
FirstComponent.propTypes = {
    name: PropTypes.string.isRequired,
}

// To use FirstComponent in another file it must be exposed through an export call:
export default FirstComponent;

Componentes com estado #

Em contraste com os componentes ‘stateless’ mostrados acima, os componentes ‘stateful’ têm um objeto de estado que pode ser atualizado com o método setState. O estado deve ser inicializado no construtor antes de ser definido:

import React, { Component } from 'react';

class SecondComponent extends Component {
    constructor(props) {
        super(props);

        this.state = {
            toggle: true
        };

        // This is to bind context when passing onClick as a callback
        this.onClick = this.onClick.bind(this);
    }

    onClick() {
        this.setState((prevState, props) => ({
            toggle: !prevState.toggle
        }));
    }
    
    render() {
        return (
            <div onClick={this.onClick}>
                Hello, {this.props.name}! I am a SecondComponent.
                <br />
                Toggle is: {this.state.toggle}
            </div>
        );
    }
}

Estender um componente com PureComponent em vez de Component implementará automaticamente o método de ciclo de vida shouldComponentUpdate() com prop superficial e comparação de estado. Isso mantém seu aplicativo com melhor desempenho, reduzindo a quantidade de renderizações desnecessárias que ocorrem. Isso pressupõe que seus componentes são ‘Puros’ e sempre renderizam a mesma saída com a mesma entrada de estado e props.

Componentes de ordem superior #

Os componentes de ordem superior (HOC) permitem compartilhar a funcionalidade do componente.

import React, { Component } from 'react';

const PrintHello = ComposedComponent => class extends Component {
    onClick() {
        console.log('hello');
    }
    
    /* The higher order component takes another component as a parameter 
    and then renders it with additional props */
    render() {
        return <ComposedComponent {...this.props } onClick={this.onClick} />
    }
}

const FirstComponent = props => (
    <div onClick={ props.onClick }>
        Hello, {props.name}! I am a FirstComponent.
    </div>
);

const ExtendedComponent = PrintHello(FirstComponent);

Componentes de ordem superior são usados ​​quando você deseja compartilhar a lógica entre vários componentes, independentemente de quão diferentes eles sejam renderizados.

Componente Básico #

Dado o seguinte arquivo HTML:

index.html

<!DOCTYPE html>
<html>
  <head>
    <meta charset="utf-8" />
    <title>React Tutorial</title>
    <script src="https://cdnjs.cloudflare.com/ajax/libs/react/15.2.1/react.js"></script>
    <script src="https://cdnjs.cloudflare.com/ajax/libs/react/15.2.1/react-dom.js"></script>
    <script src="https://cdnjs.cloudflare.com/ajax/libs/babel-core/5.8.34/browser.min.js"></script>
  </head>
  <body>
    <div id="content"></div>
    <script type="text/babel" src="scripts/example.js"></script>
  </body>
</html>

Você pode criar um componente básico usando o seguinte código em um arquivo separado:

scripts/example.js

import React, { Component } from 'react';
import ReactDOM from 'react-dom';

class FirstComponent extends Component {
  render() {
    return (
      <div className="firstComponent">
        Hello, world! I am a FirstComponent.
      </div>
    );
  }
}
ReactDOM.render(
  <FirstComponent />, // Note that this is the same as the variable you stored above
  document.getElementById('content')
);

Você obterá o seguinte resultado (observe o que está dentro de div#content):

<!DOCTYPE html>
<html>

  </head>
  <body>
    <div id="content">
        <div className="firstComponent">
            Hello, world! I am a FirstComponent.
        </div>
    </div>
    <script type="text/babel" src="scripts/example.js"></script>
  </body>
</html>

Componentes de aninhamento #

Muito do poder do ReactJS é sua capacidade de permitir o aninhamento de componentes. Pegue os dois componentes a seguir:

var React = require('react');
var createReactClass = require('create-react-class');

var CommentList = reactCreateClass({
  render: function() {
    return (
      <div className="commentList">
        Hello, world! I am a CommentList.
      </div>
    );
  }
});

var CommentForm = reactCreateClass({
  render: function() {
    return (
      <div className="commentForm">
        Hello, world! I am a CommentForm.
      </div>
    );
  }
});

Você pode aninhar e fazer referência a esses componentes na definição de um componente diferente:

var React = require('react');
var createReactClass = require('create-react-class');

var CommentBox = reactCreateClass({
  render: function() {
    return (
      <div className="commentBox">
        <h1>Comments</h1>
        <CommentList /> // Which was defined above and can be reused
        <CommentForm /> // Same here
      </div>
    );
  }
});

Aninhamento adicional pode ser feito de três maneiras, todas com seus próprios lugares para serem usados.

1. Aninhamento sem usar crianças #

(continuação acima)

var CommentList = reactCreateClass({
  render: function() {
    return (
      <div className="commentList">
        <ListTitle/>
        Hello, world! I am a CommentList.
      </div>
    );
  }
});

Este é o estilo em que A compõe B e B compõe C.

Prós #

  • Fácil e rápido de separar elementos da interface do usuário
  • Fácil de injetar props para filhos com base no estado do componente pai

Contras #

  • Menos visibilidade na arquitetura de composição
  • Menos reutilização

Bom se #

  • B e C são apenas componentes de apresentação
  • B deve ser responsável pelo ciclo de vida de C

2. Aninhamento usando crianças #

(continuação acima)

var CommentBox = reactCreateClass({
  render: function() {
    return (
      <div className="commentBox">
        <h1>Comments</h1>
        <CommentList>
            <ListTitle/> // child
        </CommentList>
        <CommentForm />
      </div>
    );
  }
});

Este é o estilo em que A compõe B e A diz a B para compor C. Mais poder para os componentes pai.

Prós #

  • Melhor gerenciamento do ciclo de vida dos componentes
  • Melhor visibilidade da arquitetura de composição
  • Melhor reutilização

Contras #

  • Injetar adereços pode ficar um pouco caro
  • Menos flexibilidade e potência nos componentes filhos

Bom se #

  • B deve aceitar compor algo diferente de C no futuro ou em outro lugar
  • A deve controlar o ciclo de vida de C

B renderizaria C usando this.props.children, e não há uma maneira estruturada de B saber para que servem esses filhos. Assim, B pode enriquecer os componentes filhos fornecendo adereços adicionais, mas se B precisar saber exatamente o que são, o número 3 pode ser uma opção melhor.

3. Aninhamento usando adereços #

(continuação acima)

var CommentBox = reactCreateClass({
  render: function() {
    return (
      <div className="commentBox">
        <h1>Comments</h1>
        <CommentList title={ListTitle}/> //prop
        <CommentForm />
      </div>
    );
  }
});

Este é o estilo onde A compõe B e B oferece uma opção para A passar algo para compor para um propósito específico. Composição mais estruturada.

Prós #

  • Composição como um recurso
  • Fácil validação
  • Melhor composição

Contras #

  • Injetar adereços pode ficar um pouco caro
  • Menos flexibilidade e potência nos componentes filhos

Bom se #

  • B tem características específicas definidas para compor algo
  • B deve saber apenas como renderizar e não o que renderizar

O nº 3 geralmente é obrigatório para criar uma biblioteca pública de componentes, mas também uma boa prática em geral para criar componentes combináveis ​​e definir claramente os recursos de composição. O nº 1 é o mais fácil e rápido de fazer algo que funcione, mas o nº 2 e o nº 3 devem fornecer certos benefícios em vários casos de uso.

Adereços #

Props são uma maneira de passar informações para um componente React, eles podem ter qualquer tipo incluindo funções - algumas vezes chamadas de callbacks.

Em JSX props são passados ​​com a sintaxe de atributo

<MyComponent userID={123} />

Dentro da definição de MyComponent userID agora estará acessível a partir do objeto props

// The render function inside MyComponent
render() {
    return (
        <span>The user's ID is {this.props.userID}</span>
    )
}

É importante definir todos os props, seus tipos e, quando aplicável, seu valor padrão:

// defined at the bottom of MyComponent
MyComponent.propTypes = {
    someObject: React.PropTypes.object,
    userID: React.PropTypes.number.isRequired,
    title: React.PropTypes.string
};

MyComponent.defaultProps = {
    someObject: {},
    title: 'My Default Title'
}

Neste exemplo, a prop someObject é opcional, mas a prop userID é obrigatória. Se você não fornecer userID para MyComponent, em tempo de execução, o mecanismo React mostrará um console avisando que o prop necessário não foi fornecido. Cuidado, porém, este aviso é mostrado apenas na versão de desenvolvimento da biblioteca React, a versão de produção não registrará nenhum aviso.

Usar defaultProps permite que você simplifique

const { title = 'My Default Title' } = this.props;
console.log(title);

para

console.log(this.props.title);

É também uma proteção para o uso de object array e functions. Se você não fornecer uma prop padrão para um objeto, o seguinte lançará um erro se a prop não for passada:

if (this.props.someObject.someKey)

No exemplo acima, this.props.someObject é undefined e, portanto, a verificação de someKey lançará um erro e o código será quebrado. Com o uso de defaultProps você pode usar com segurança a verificação acima.

Estados do componente - Interface de usuário dinâmica #

Suponha que queremos ter o seguinte comportamento - Temos um título (digamos, elemento h3) e ao clicar nele, queremos que ele se torne uma caixa de entrada para que possamos modificar o nome do título. O React torna isso altamente simples e intuitivo usando estados de componentes e instruções if else. (Explicação do código abaixo)

// I have used ReactBootstrap elements. But the code works with regular html elements also
var Button = ReactBootstrap.Button;
var Form = ReactBootstrap.Form;
var FormGroup = ReactBootstrap.FormGroup;
var FormControl = ReactBootstrap.FormControl;

var Comment = reactCreateClass({
  getInitialState: function() {
    return {show: false, newTitle: ''};
  },

  handleTitleSubmit: function() {
     //code to handle input box submit - for example, issue an ajax request to change name in database
  },

  handleTitleChange: function(e) {
    //code to change the name in form input box. newTitle is initialized as empty string. We need to update it with the string currently entered by user in the form
    this.setState({newTitle: e.target.value});
  },

  changeComponent: function() {
    // this toggles the show variable which is used  for dynamic UI
    this.setState({show: !this.state.show)};
  },

  render: function() {

    var clickableTitle;

    if(this.state.show) {
        clickableTitle = <Form inline onSubmit={this.handleTitleSubmit}>
                             <FormGroup controlId="formInlineTitle">
                                 <FormControl type="text" onChange={this.handleTitleChange}>
                             </FormGroup>
                         </Form>;
    } else {
        clickabletitle = <div>
                            <Button bsStyle="link" onClick={this.changeComponent}>
                                <h3> Default Text </h3>
                            </Button>
                         </div>;
    }

    return (
        <div className="comment">
            {clickableTitle}
        </div>
    );
  }
});

ReactDOM.render(
    <Comment />, document.getElementById('content')
);

A parte principal do código é a variável clickableTitle. Com base na variável de estado show, pode ser um elemento Form ou um elemento Button. React permite o aninhamento de componentes.

Assim, podemos adicionar um elemento {clickableTitle} na função de renderização. Ele procura a variável clickableTitle. Com base no valor ’this.state.show’, ele exibe o elemento correspondente.

Variações de componentes funcionais sem estado #

const languages = [
  'JavaScript',
  'Python',
  'Java',
  'Elm',
  'TypeScript',
  'C#',
  'F#'
]

// one liner
const Language = ({language}) => <li>{language}</li>

Language.propTypes = {
  message: React.PropTypes.string.isRequired
}

/**
* If there are more than one line.
* Please notice that round brackets are optional here,
* However it's better to use them for readability
*/
const LanguagesList = ({languages}) => {
  <ul>
    {languages.map(language => <Language language={language} />)}
  </ul>
}

LanguagesList.PropTypes = {
  languages: React.PropTypes.array.isRequired
}

/**
 * This syntax is used if there are more work beside just JSX presentation
 * For instance some data manipulations needs to be done.
 * Please notice that round brackets after return are required,
 * Otherwise return will return nothing (undefined)
 */
const LanguageSection = ({header, languages}) => {
  // do some work
  const formattedLanguages = languages.map(language => language.toUpperCase())
  return (
    <fieldset>
      <legend>{header}</legend>
      <LanguagesList languages={formattedLanguages} />
    </fieldset>
  )
}

LanguageSection.PropTypes = {
  header: React.PropTypes.string.isRequired,
  languages: React.PropTypes.array.isRequired
}

ReactDOM.render(
  <LanguageSection 
    header="Languages" 
    languages={languages} />,
  document.getElementById('app')
)

armadilhas do setState #

Você deve ter cuidado ao usar setState em um contexto assíncrono. Por exemplo, você pode tentar chamar setState no callback de uma solicitação get:

class MyClass extends React.Component {
    constructor() {
        super();

        this.state = {
            user: {}
        };
    }

    componentDidMount() {
        this.fetchUser();
    }

    fetchUser() {
        $.get('/api/users/self')
            .then((user) => {
                this.setState({user: user});
            });
    }

    render() {
        return <h1>{this.state.user}</h1>;
    }
}

Isso pode chamar problemas - se o callback for chamado depois que o Component for desmontado, então this.setState não será uma função. Sempre que este for o caso, você deve ter cuidado para garantir que o uso de setState seja cancelável.

Neste exemplo, você pode desejar cancelar a solicitação XHR quando o componente for desmontado:

class MyClass extends React.Component {
    constructor() {
        super();

        this.state = {
            user: {},
            xhr: null
        };
    }

    componentWillUnmount() {
        let xhr = this.state.xhr;

        // Cancel the xhr request, so the callback is never called
        if (xhr && xhr.readyState != 4) {
            xhr.abort();
        }
    }

    componentDidMount() {
        this.fetchUser();
    }

    fetchUser() {
        let xhr = $.get('/api/users/self')
            .then((user) => {
                this.setState({user: user});
            });

        this.setState({xhr: xhr});
    }
}

O método assíncrono é salvo como um estado. No componentWillUnmount você realiza toda a sua limpeza - inclusive cancelando a requisição XHR.

Você também pode fazer algo mais complexo. Neste exemplo, estou criando uma função ‘stateSetter’ que aceita o objeto this como argumento e evita this.setState quando a função cancel for chamada:

function stateSetter(context) {
    var cancelled = false;
    return {
        cancel: function () {
            cancelled = true;
        },
        setState(newState) {
            if (!cancelled) {
                context.setState(newState);
            }
        }
    }
}

class Component extends React.Component {
    constructor(props) {
        super(props);
        this.setter = stateSetter(this);
        this.state = {
            user: 'loading'
        };
    }
    componentWillUnmount() {
        this.setter.cancel();
    }
    componentDidMount() {
        this.fetchUser();
    }
    fetchUser() {
        $.get('/api/users/self')
            .then((user) => {
                this.setter.setState({user: user});
            });
    }
    render() {
        return <h1>{this.state.user}</h1>
    }
}

Isso funciona porque a variável cancelled é visível no encerramento setState que criamos.
