Cache de respostas HTTP #

Um caso de uso típico para RxJS é criar solicitações HTTP e armazenar seus resultados em cache por algum período de tempo. Além disso, sempre queremos executar apenas uma solicitação por vez e compartilhar sua resposta.

Por exemplo, o código a seguir armazena em cache 1 item para max. 1000ms:

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1);

var counter = 1;    
function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100);
}

function requestCachedHttpResult() {
    return updateRequest;
}

A função makeMockHttpRequest() simula uma requisição HTTP que chega com um atraso de 100ms.

A função requestCachedHttpResult() é onde assinamos para obter uma resposta real ou armazenada em cache.

Com .publishReplay(1, 1000) usamos multicast RxJS para usar internamente ReplaySubject e mantenha o item 1 por um máximo de 1000ms. Então refCount() é usado para manter sempre apenas uma assinatura para o source que é Observable.defer(). Este Observable é usado para criar uma nova solicitação e incrementa o counter para provar que os valores armazenados em cache e novas assinaturas compartilham os mesmos dados.

Quando queremos obter dados atuais, chamamos requestCachedHttpResult(). Para garantir que o Observer seja concluído corretamente após a emissão dos dados, usamos o operador take(1).

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

Isso cria uma única solicitação com mockDataFetch() e imprime no console:

1

Um exemplo mais complicado chamará várias solicitações em momentos diferentes em que queremos testar se as conexões e respostas HTTP simuladas são compartilhadas.

requestCachedHttpResult()
    .subscribe(val => console.log("Response 0:", val));

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 50:", val))
, 50);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 200:", val))
, 200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1200:", val))
, 1200);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 1500:", val))
, 1500);

setTimeout(() => requestCachedHttpResult()
    .subscribe(val => console.log("Response 3500:", val))
, 3500);

Veja a demonstração ao vivo: https://jsbin.com/todude/5/edit?js,console

Cada solicitação é enviada com atraso e acontece na seguinte ordem:

0 - Primeira solicitação que faz com que o refCount() assine sua source que faz a chamada mockDataFetch(). Sua resposta será atrasada em 100ms. Neste momento ConnectableObservable dentro do operador publishReplay() tem um Observer.

50 - A segunda solicitação também assina o ConnectableObservable. Neste momento ConnectableObservable dentro do operador publishReplay() tem dois Observer. Ele não cria outro pedido com makeMockHttpRequest() porque refCount() já está inscrito.

100 - A primeira resposta está pronta. Ele é primeiro armazenado em cache pelo ReplaySubject e, em seguida, reemitido para os dois Observers inscritos no ConnectableObservable. Ambos os Observadores são concluídos graças ao take(1) e cancelados.

200 - Assina o ReplaySubject que imediatamente emite seu valor em cache que faz com que take(1) complete o Observer e cancela a inscrição imediatamente. Nenhuma solicitação HTTP é feita e nenhuma assinatura permanece.

1200 - O mesmo que o primeiro evento em 0. Neste ponto, o valor em cache foi descartado porque é mais antigo que 1000ms.

1500 - O mesmo que o quarto evento em 200.

3500 - O mesmo que o primeiro evento em 1200.

A saída no console é a seguinte:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: 2
Response 1500: 2
Response 3500: 3

No RxJS 5 uma funcionalidade similar foi coberta pelo operador cache(). No entanto, foi removido em 5.0.0-rc.1 devido a sua funcionalidade limitada.

Tratamento de erros

Se quisermos lidar com erros produzidos pelo serviço remoto (a função makeMockHttpRequest), precisamos pegá-los antes de serem mesclados na cadeia Observable principal porque qualquer erro recebido pelo ReplaySubject dentro de publishReplay() marcaria seu estado interno como ‘parado’ (Leia mais aqui https://www.wikiod.com/pt/rxjs/sujeito#Sujeito e seu estado interno ), que definitivamente não é o que queremos .

No exemplo a seguir estamos simulando um erro quando counter === 2 e capturando-o com o operador catch(). Estamos usando catch() para transformar apenas a notificação error em um next regular para que possamos lidar com o erro nos observadores:

function makeMockHttpRequest() {
    return Observable.of(counter++)
        .delay(100)
        .map(i => {
            if (i === 2) {
                throw new Error('Invalid URL');
            }
            return i;
        })
        .catch(err => Observable.of(err));
}

Veja a demonstração ao vivo: https://jsbin.com/kavihu/10/edit?js,console

Isso imprimirá para consolar a seguinte saída. Observe que os erros são recebidos nos manipuladores next:

Response 0: 1
Response 50: 1
Response 200: 1
Response 1200: [object Error] { ... }
Response 1500: [object Error] { ... }
Response 3500: 3

Se quisermos tratar erros como notificações regulares de erro em cada observador, temos que relançá-los após o operador publishReplay() pelas razões explicadas acima.

var updateRequest = Observable.defer(() => makeMockHttpRequest())
    .publishReplay(1, 1000)
    .refCount()
    .take(1)
    .map(val => {
        if (val instanceof Error) {
            throw val;
        }
        return val;
    });

Veja a demonstração ao vivo: https://jsbin.com/fabosam/5/edit?js,console (observe que tivemos que adicionar também callbacks de erro para cada observador).

Response 0: 1
Response 50: 1
Response 200: 1
error callback: Error: Invalid URL
error callback: Error: Invalid URL
Response 3500: 3

Enviando várias solicitações HTTP paralelas #

Um caso de uso muito comum em aplicativos da Web é executar várias solicitações assíncronas (por exemplo, HTTP) e reunir seus resultados à medida que chegam ou todos de uma vez (por exemplo, em Angular2 com o [serviço HTTP](https://angular. io/docs/ts/latest/guide/server-communication.html)).

1. Reunindo respostas assíncronas uma a uma à medida que elas chegam

Isso normalmente é feito com o operador mergeMap() que usa uma função de projeção que precisa retornar um Observável. O operador mergeMap() assina internamente cada Observável imediatamente, mesmo que o Observável anterior ainda não tenha sido concluído.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .subscribe(val => console.log(val));

Isso imprime respostas ao console em ordem diferente devido ao atraso aleatório:

Response from url-3
Response from url-4
Response from url-2
Response from url-1

Veja a demonstração ao vivo: https://jsbin.com/xaqudan/2/edit?js,console

Cada resposta (item emitido via chamada next) é reemitido por mergeMap() imediatamente.

Para nosso propósito de enviar várias solicitações HTTP, é útil mencionar que mergeMap() pode receber três argumentos no total:

  1. A função de projeção que precisa retornar um Observable.
  2. A função seletora de resultados que nos permite modificar o resultado antes de emiti-lo mais.
  3. O número de Observáveis ​​inscritos simultaneamente.

Controlando o número de solicitações paralelas

Com o terceiro argumento, podemos controlar quantas solicitações paralelas trataremos (assumindo que cada Observable executando uma solicitação HTTP é “frio”).

No exemplo a seguir, executaremos apenas 2 solicitações ao mesmo tempo.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url), undefined, 2)
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Veja a demonstração ao vivo: https://jsbin.com/sojejal/4/edit?js,console

Observe que as duas primeiras solicitações foram concluídas após 1s, enquanto as outras duas após 2s.

[1004, "Response from url-1"]
[1010, "Response from url-2"]
[2007, "Response from url-3"]
[2012, "Response from url-4"]

Tratamento de erros

Se qualquer um dos Observables de origem falhar (envia uma notificação error) o mergeMap() reenvia o erro ainda mais como error. Caso queiramos que cada Observable falhe normalmente, precisamos usar, por exemplo, catch() operador.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url).catch(() => Observable.empty()))
  .subscribe(val => console.log(val));

A resposta para url-3 lança um erro que é enviado como notificação de erro. Isso é posteriormente capturado pelo operador catch() e substituído por Observable.empty() que é apenas uma notificação completa. Por esta razão esta resposta é ignorada.

A saída para este exemplo é a seguinte:

Response from url-4
Response from url-1
Response from url-2

Veja a demonstração ao vivo: https://jsbin.com/kuqumud/4/edit?js,console

2. Reunindo todas as respostas assíncronas de uma vez

Seguindo os exemplos anteriores, poderíamos reunir todas as respostas com o operador toArray().

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

Observable.from(urls)
  .mergeMap(url => mockHTTPRequest(url))
  .toArray()
  .subscribe(val => console.log(val));

No entanto, usar o operador toArray() tem uma consequência importante. Se o assinante recebe os resultados não é apenas controlado completando todas as requisições HTTP, mas também completando a fonte Observable (Observable.from em nosso caso). Isso significa que não podemos usar Observables de origem que nunca são concluídos (por exemplo, Observable.fromEvent).

Outra maneira de obter o mesmo resultado é usando Observable.forkJoin() que leva como argument um array de Observables que queremos assinar e esperar até que todos eles emitam pelo menos um valor e concluam.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(Math.random() * 1000);
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];
var observables = urls.map(url => mockHTTPRequest(url));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Isso imprime todas as respostas como uma única matriz:

["Response from url-1", "Response from url-2", "Response from url-3", "Response from url-4"]

Veja a demonstração ao vivo: https://jsbin.com/fomoye/2/edit?js,console

O Observable.forkJoin() também recebe como argumento opcional uma função seletora de resultados que nos permite modificar o resultado final antes de emiti-lo:

Observable.forkJoin(observables, (...results) => {
    return results.length;
  })
  .subscribe(val => console.log(val));

Isso imprime no console:

4

Veja a demonstração ao vivo: https://jsbin.com/muwiqic/1/edit?js,console

Observe que o argumento para a função seletora de resultados é descompactado.

Tratamento de erros

Para tratamento de erros, podemos usar a mesma abordagem do exemplo anterior com o operador catch().

No entanto, há uma coisa importante para estar ciente. O forkJoin() requer que cada fonte Observável emita pelo menos um valor. Se usássemos catch(() => Observable.empty()) como fizemos antes, o forkJoin() nunca emitiria nada porque Observable.empty() é apenas uma notificação completa. É por isso que precisamos usar, por exemplo, Observable.of(null) que é um valor null seguido por uma notificação complete.

function mockHTTPRequest(url) {
  return Observable.of(`Response from ${url}`)
    .delay(Math.random() * 1000)
    .map(value => {
      if (url === 'url-3') {
        throw new Error(`Error response from ${url}`)
      }
      return value;
    });
}

var urls = ['url-1', 'url-2', 'url-3', 'url-4'];

var observables = urls.map(url => mockHTTPRequest(url).catch(() => Observable.of(null)));

Observable.forkJoin(observables)
  .subscribe(val => console.log(val));

Veja a demonstração ao vivo: https://jsbin.com/yidiked/2/edit?js,console

Isso imprime no console:

["Response from url-1", "Response from url-2", null, "Response from url-4"]

Observe que o erro é substituído por null. Se usássemos apenas Observable.empty() o forkJoin() nunca emitiria nada.

Descartando chamadas de repouso lentas/desatualizadas #

Um caso de uso comum é descartar certas chamadas de descanso, que não são mais necessárias após certas entradas do usuário. O exemplo mais proeminente seria, quando um usuário usa alguma função de pesquisa, faz uma solicitação, faz outra solicitação e por algum motivo a primeira solicitação chega após a segunda solicitação e o aplicativo exibe os dados desatualizados da solicitação antiga.

Este é um caso de uso perfeito para o operador switchMap.

searchInput$
    .switchMap(() => makeRestCall());

Neste caso o stream irá mudar para a chamada rest, mas somente até que novos dados sejam emitidos no searchInput$, então o stream dentro do switchMap é descartado e uma nova chamada rest é feita. Assim, um resto-resultado só será considerado se terminar antes do próximo clique.

E aqui está um exemplo simulado completo:

// some initial data-mocking
const Observable = Rx.Observable;
var counter = 1;
function mockDataFetch() {
    return Observable.of(counter++)
        .delay(500);
}

// the recipe

const searchInput$ = new Rx.Subject();
searchInput$
    .do(searchInput => console.log("Searching for " + searchInput))
    .switchMap(searchInput => mockDataFetch()
              .map(result => ({result, searchInput}))
     )
    .do(data => console.log("Received result for " + data.searchInput + ": " + data.result))
    .subscribe();

// simulating search inputs
searchInput$.next("google");
setTimeout(() => searchInput$.next("froogle"), 600);
setTimeout(() => searchInput$.next("doodle"), 800);
setTimeout(() => searchInput$.next("poodle"), 1000);
setTimeout(() => searchInput$.next("noodle"), 1600);

Veja a demonstração ao vivo: https://jsbin.com/suzakahoro/1/edit?js,console

Enviando várias solicitações HTTP sequenciais #

Fazer uma sequência de solicitações HTTP tem dois motivos principais:

  • As solicitações são dependentes umas das outras (o resultado de uma solicitação é necessário para uma solicitação consecutiva).

  • Queremos distribuir a carga do servidor em várias solicitações.

1. Fazendo várias solicitações dependentes

Isso pode ser feito usando o operador concatMap() para transformar uma resposta em parâmetros necessários para o pedido consecutivo.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

function timestamp() {
  return (new Date()).getTime() - start;
}

var start = (new Date()).getTime();

Observable.of('url-1') 
  // first request
  .concatMap(url => {
    console.log(timestamp() + ': Sending request to ' + url);
    return mockHTTPRequest(url);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // second request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create new requests url
    return mockHTTPRequest(newUrl);
  })
  .do(response => console.log(timestamp() + ': ' + response))

  // third request
  .concatMap(response => {
    console.log(timestamp() + ': Sending request to ' + response);
    let newUrl = 'url-' + response.length; // create another requests url
    return mockHTTPRequest(newUrl);
  })
  .subscribe(response => console.log(timestamp() + ': ' + response));

O operador concatMap() assina internamente o Observable retornado de sua função de projeção e espera até que ele seja concluído enquanto reemite todos os seus valores.

Este exemplo marca o horário de cada solicitação e resposta:

3: Sending request to url-1
1010: Response from url-1
1011: Sending request to Response from url-1
2014: Response from url-19
2015: Sending request to Response from url-19
3017: Response from url-20

Veja a demonstração ao vivo: https://jsbin.com/fewidiv/6/edit?js,console

Tratamento de erros

Se alguma das solicitações HTTP falhar, obviamente não queremos continuar porque não podemos construir a solicitação a seguir.

2. Fazendo solicitações consecutivas

Caso não estejamos interessados ​​na resposta da solicitação HTTP anterior, podemos pegar apenas um array de URLs e executá-los um após o outro.

function mockHTTPRequest(url) {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Este exemplo imprime respostas com carimbo de data/hora:

[1006, "Response from url-1"]
[2012, "Response from url-2"]
[3014, "Response from url-3"]
[4016, "Response from url-4"]

Veja a demonstração ao vivo: https://jsbin.com/kakede/3/edit?js,console

Atrasando chamadas consecutivas

Também podemos querer fazer um pequeno atraso entre cada solicitação. Nesse caso, precisamos anexar delay() após cada chamada mockHTTPRequest().

Observable.from(urls)
  .concatMap(url => {
    return mockHTTPRequest(url)
      .do(response => console.log(((new Date()).getTime() - start) + ': Sending request to ' + url))
      .delay(500);
  })
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Isso imprime para consolar a seguinte saída:

2024: Sending request to url-1
[2833, "Response from url-1"]
4569: Sending request to url-2
[5897, "Response from url-2"]
7880: Sending request to url-3
[8674, "Response from url-3"]
9789: Sending request to url-4
[10796, "Response from url-4"]

Veja a demonstração ao vivo: https://jsbin.com/kakede/4/edit?js,console

Tratamento de erros

Se simplesmente queremos ignorar quando qualquer uma das solicitações HTTP falha, temos que encadear catch() de cada mockHTTPRequest().

function mockHTTPRequest(url) {
  if (url == 'url-3') {
    return Observable.throw(new Error(`Request ${url} failed.`));
  } else {
    return Observable.of(`Response from ${url}`)
      .delay(1000);
  }
}

let urls = ['url-1', 'url-2', 'url-3', 'url-4'];
let start = (new Date()).getTime();

Observable.from(urls)
  .concatMap(url => mockHTTPRequest(url).catch(obs => Observable.empty()))
  .timestamp()
  .map(stamp => [stamp.timestamp - start, stamp.value])
  .subscribe(val => console.log(val));

Isso simplesmente ignora a chamada url-3:

[1004, "Response from url-1"]
[2012, "Response from url-2"]
[3016, "Response from url-4"]

Veja a demonstração ao vivo: https://jsbin.com/jowiqo/2/edit?js,console

Se não usássemos o operador catch(), o url-3 faria com que a cadeia enviasse uma notificação de erro e o último url-4 não seria executado.

Veja a demonstração ao vivo: https://jsbin.com/docapim/3/edit?js,console

Limitação de taxa #

Um problema comum com serviços remotos é a limitação de taxa. O serviço remoto nos permite enviar apenas um número limitado de solicitações ou quantidade de dados por período de tempo.

No RxJS 5 uma funcionalidade muito semelhante é fornecida pelo operador bufferTime e especialmente se deixe o segundo parâmetro não especificado (ele define com que frequência queremos criar um novo buffer. Se o deixarmos indefinido/nulo, ele criará um novo buffer logo após emitir o atual).

Um uso típico de bufferTime será assim:

bufferTime(1000, null, 5)

Isso armazenará os itens em buffer até que uma das duas condições seja atendida. Em seguida, ele emitirá o buffer e iniciará outro:

  • o operador está coletando itens por 1000ms
  • o operador já coletou 5 itens

Para fins de demonstração, podemos criar uma fonte Observable que emite muito rápido para que o bufferTime atinja o limite de tamanho (5) e emita mais de uma vez a cada 1000ms:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

Então vamos encadeá-lo com bufferTime e concatMap. O operador concatMap é onde forçamos o atraso 1000ms:

const startTime = (new Date()).getTime();

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.of(buffer).delay(1000))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Veja a demonstração ao vivo: https://jsbin.com/kotibow/3/edit?js,console

Adicionamos também timestamp() para ver os tempos de emissão para garantir que o atraso seja realmente de pelo menos 1000ms.

Observe que não precisamos usar Observable.of(buffer). Estamos usando aqui apenas para verificar manualmente se o número de itens armazenados em buffer está correto.

A partir da saída do console, podemos ver que o atraso entre duas emissões é aproximadamente 1000ms:

Timestamp { value: [ 1, 2, 3, 4, 5 ], timestamp: 1475 }
Timestamp { value: [ 6, 7, 8, 9, 10 ], timestamp: 2564 }
Timestamp { value: [ 11, 12, 13, 14, 15 ], timestamp: 3567 }
Timestamp { value: [ 16, 17, 18, 19, 20 ], timestamp: 4572 }
Timestamp { value: [ 21, 22, 23, 24, 25 ], timestamp: 5573 }
Timestamp { value: [], timestamp: 6578 }

Agora também podemos testar uma situação em que a fonte emite lentamente para que o operador bufferTime atinja a condição de intervalo máximo:

const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(300));

Veja a demonstração ao vivo: https://jsbin.com/tuwowan/2/edit?js,console

Então a saída deve começar depois de 2s porque levou 1s para o operador bufferTime emitir e então adicionamos o atraso 1s;

Timestamp { value: [ 1, 2, 3 ], timestamp: 2017 }
Timestamp { value: [ 4, 5, 6 ], timestamp: 3079 }
Timestamp { value: [ 7, 8, 9, 10 ], timestamp: 4088 }
Timestamp { value: [ 11, 12, 13 ], timestamp: 5093 }
Timestamp { value: [ 14, 15, 16 ], timestamp: 6094 }
Timestamp { value: [ 17, 18, 19, 20 ], timestamp: 7098 }
Timestamp { value: [ 21, 22, 23 ], timestamp: 8103 }
Timestamp { value: [ 24, 25 ], timestamp: 9104 }

Se quiséssemos usar essa abordagem em um aplicativo do mundo real, colocaríamos a chamada remota no operador concatMap. Dessa forma, podemos controlar se queremos forçar o atraso 1s entre solicitações ou respostas do serviço remoto.

Por exemplo, podemos forçar o atraso mínimo de 1s entre solicitações usando forkJoin no retorno de chamada concatMap:

function mockHTTPRequest(buffer) {
  return Observable.of(true).delay(Math.random() * 1500)
}

const startTime = (new Date()).getTime();
const source = Observable.range(1, 25)
  .concatMap(val => Observable.of(val).delay(75));

source.bufferTime(1000, null, 5)
  .concatMap(buffer => Observable.forkJoin(
    mockHTTPRequest(buffer),
    Observable.of(buffer).delay(1000)
  ))
  .timestamp()
  .map(obj => {
    obj.timestamp = obj.timestamp - startTime;
    return obj;
  })
  .subscribe(obj => console.log(obj));

Veja a demonstração ao vivo: https://jsbin.com/xijaver/edit?js,console

Graças ao forkJoin, o concatMap precisa esperar que ambos os Observables sejam concluídos.

Por outro lado, se quisermos forçar o atraso 1s entre as respostas, basta anexar o operador delay() após mockHTTPRequest():

.concatMap(buffer => mockHTTPRequest(buffer).delay(1000))

Veja a demonstração ao vivo: https://jsbin.com/munopot/2/edit?js,console
