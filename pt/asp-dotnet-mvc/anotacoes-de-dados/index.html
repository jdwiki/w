Podemos adicionar validações ao nosso aplicativo adicionando anotações de dados às nossas classes de modelo. As anotações de dados nos permitem descrever as regras que queremos aplicar às propriedades do nosso modelo, e a ASP.NET MVC cuidará de aplicá-las e exibir as mensagens apropriadas aos usuários.

Validação remota #

Validação Remota usada para verificar se o conteúdo inserido no controle de entrada é válido ou não, enviando uma solicitação ajax ao lado do servidor para verificar. #

Trabalhando

O RemoteAttribute funciona fazendo uma chamada AJAX do cliente para uma ação do controlador com o valor do campo sendo validado. A ação do controlador retorna uma resposta JsonResult indicando sucesso ou falha na validação. Retornar true da sua ação indica que a validação foi aprovada. Qualquer outro valor indica falha. Se você retornar false, a mensagem de erro especificada no atributo será usada. Se você retornar qualquer outra coisa, como uma string ou até mesmo um inteiro, ela será exibida como a mensagem de erro. A menos que você precise que sua mensagem de erro seja dinâmica, faz sentido retornar true ou false e deixar o validador usar a mensagem de erro especificada no atributo.

ViewModel

public class ViewModel
{
    [Remote("IsEmailAvailable", "Group", HttpMethod = "POST", ErrorMessage = "Email already exists. Please enter a different email address.")]
    public string Email{ get; set; }
}

Controlador

[HttpPost]
public JsonResult IsEmailAvailable(string Email)
{
    // Logic to check whether email is already registered or Not.
    var emailExists = IsEmailRegistered();
    return Json(!emailExists);         
}

Você pode passar propriedades adicionais do modelo para o método do controlador usando a propriedade AdditionalFields de RemoteAttribute. Um cenário típico seria passar a propriedade ID do modelo em um formulário ‘Editar’, para que a lógica do controlador pudesse ignorar valores para o registro existente.

Modelo

  public int? ID { get; set; }
  [Display(Name = "Email address")]
  [DataType(DataType.EmailAddress)]
  [Required(ErrorMessage = "Please enter you email address")]
  [Remote("IsEmailAvailable", HttpMethod="Post", AdditionalFields="ID", ErrorMessage = "Email already exists. Please enter a different email address.")]  
  public string Email { get; set; }

Controlador

[HttpPost]
public ActionResult Validate(string email, int? id)
{
    if (id.HasValue)
    {
        return Json(!db.Users.Any(x => x.Email == email && x.ID != id);
    }
    else
    {
        return Json(!db.Users.Any(x => x.Email == email);
    }

Nota adicional

A mensagem de erro padrão é compreensivelmente vaga, então lembre-se sempre de substituir a mensagem de erro padrão ao usar o RemoteAttribute.

Modelo #

using System.ComponentModel.DataAnnotations;

public class ViewModel
{
    [Required(ErrorMessage="Name is required")] 
    public string Name { get; set; }

    [StringLength(14, MinimumLength = 14, ErrorMessage = "Invalid Phone Number")]
    [Required(ErrorMessage="Phone Number is required")] 
    public string PhoneNo { get; set; }

    [Range(typeof(decimal), "0", "150")]
    public decimal? Age { get; set; }

    [RegularExpression(@"^\d{5}(-\d{4})?$", ErrorMessage = "Invalid Zip Code.")]
    public string ZipCode {get;set;}

    [EmailAddress(ErrorMessage = "Invalid Email Address")] 
    public string Email { get; set; }

    [Editable(false)] 
    public string Address{ get; set; }
}

Atributo de validação personalizado #

Quando se trata de validar algumas regras que não são validação de dados genérica, por exemplo, garantir que um campo seja obrigatório ou algum intervalo de valores, mas sejam específicos para sua lógica de negócios, você pode criar seu próprio Validador personalizado. Para criar um atributo de validação personalizado, você só precisa inherit ValidationAttribute classe e override seu método IsValid. O método IsValid recebe dois parâmetros, o primeiro é um object chamado value e o segundo é um objeto ValidationContext chamado validationContext. Value refere-se ao valor real do campo que seu validador personalizado irá validar.

Suponha que você queira validar Email através do Custom Validator

public class MyCustomValidator : ValidationAttribute
{
    private static string myEmail= "admin@dotnetfiddle.net";

    protected override ValidationResult IsValid(object value, ValidationContext validationContext)
    {
        string Email = value.ToString();
        if(myEmail.Equals(Email))
            return new ValidationResult("Email Already Exist");
        return ValidationResult.Success;
    }
}

public class SampleViewModel
{
    [MyCustomValidator]
    [Required]
    public string Email { get; set; }

    public string Name { get; set; }
}

Atributo Obrigatório #

O atributo Required especifica que uma propriedade é necessária. Uma mensagem de erro pode ser especificada usando a propriedade ErrorMessage no atributo .

Primeiro adicione o namespace:

using System.ComponentModel.DataAnnotations;

E aplique o atributo em uma propriedade.

public class Product
{
   [Required(ErrorMessage = "The product name is required.")]
   public string Name { get; set; }

   [Required(ErrorMessage = "The product description is required.")]
   public string Description { get; set; }
}

Também é possível usar recursos na mensagem de erro para aplicativos globalizados. Nesse caso, o ErrorMessageResourceName deve ser especificado com a chave de recurso de a classe de recurso (arquivo resx) que deve ser definida no [ErrorMessageResourceType](https://msdn.microsoft.com/library/system.componentmodel.dataannotations.validationattribute.errormessageresourcetype(v=vs.110). aspx):

public class Product
{
   [Required(ErrorMessageResourceName = "ProductNameRequired", 
             ErrorMessageResourceType = typeof(ResourceClass))]
   public string Name { get; set; }

   [Required(ErrorMessageResourceName = "ProductDescriptionRequired", 
             ErrorMessageResourceType = typeof(ResourceClass))]
   public string Description { get; set; }
}

Atributo Expressão Regular #

O atributo [RegularExpression] pode decorar quaisquer propriedades ou campos públicos e especifica uma expressão regular que deve ser correspondida para que a propriedade seja considerada válida.

[RegularExpression(validationExpression)]
public string Property { get; set; }

Além disso, ele aceita uma propriedade opcional ErrorMessage que pode ser usada para definir a mensagem recebida pelo usuário quando dados inválidos são inseridos:

[RegularExpression(validationExpression, ErrorMessage = "{your-error-message}")]
public string Property { get; set; }

Exemplos)

[RegularExpression(@"^[a-z]{8,16}?$", ErrorMessage = "A User Name must consist of 8-16 lowercase letters")]
public string UserName{ get; set; }
[RegularExpression(@"^\d{5}(-\d{4})?$", ErrorMessage = "Please enter a valid ZIP Code (e.g. 12345, 12345-1234)")]
public string ZipCode { get; set; }

Modelo EDMx - Anotação de dados #

Modelo interno Edmx

public partial class ItemRequest
{
    public int RequestId { get; set; }
    //...
}

Adicionando anotação de dados a isso - se modificarmos esse modelo diretamente, quando for feita uma atualização no modelo, as alterações serão perdidas. assim

Para adicionar um atributo neste caso ‘Obrigatório’

Crie uma nova classe - qualquer nome Então

using System.ComponentModel;
using System.ComponentModel.DataAnnotations;

//make sure the namespace is equal to the other partial class ItemRequest
namespace MvcApplication1.Models 
{
    [MetadataType(typeof(ItemRequestMetaData))]
    public partial class ItemRequest
    {
    }

    public class ItemRequestMetaData
    {
        [Required]
        public int RequestId {get;set;}

        //...
    }
}

ou

using System.Collections.Generic;
using System.ComponentModel.DataAnnotations;

namespace YourApplication.Models
{
    public interface IEntityMetadata
    {
        [Required]
        Int32 Id { get; set; }
    }

    [MetadataType(typeof(IEntityMetadata))]
    public partial class Entity : IEntityMetadata
    {
        /* Id property has already existed in the mapped class */
    }
}

Anotações de dados para a primeira implementação do banco de dados (código do modelo gerado automaticamente) #

[MetadataType(typeof(RoleMetaData))]
public partial class ROLE
{
}

public class RoleMetaData
{
    [Display(Name = "Role")]
    public string ROLE_DESCRIPTION { get; set; }

    [Display(Name = "Username")]
    public string ROLE_USERNAME { get; set; }
}

Se você usou o banco de dados primeiro e o código do seu modelo foi gerado automaticamente, esta mensagem aparecerá acima do código do seu modelo:

Este código foi gerado a partir de um template. Alterações manuais neste arquivo podem causar um comportamento inesperado em sua aplicação. As alterações manuais neste arquivo serão sobrescrito se o código for regenerado

Se você quiser usar anotações de dados e não quiser que elas sejam substituídas se você atualizar o edmx, basta adicionar outra classe parcial à sua pasta de modelo que se parece com o exemplo acima.
