Roteamento é como o ASP.NET MVC combina um URI com uma ação. O módulo de roteamento é responsável por mapear as solicitações recebidas do navegador para ações específicas do controlador MVC.

O MVC 5 suporta um novo tipo de roteamento, chamado roteamento de atributo. Como o nome indica, o roteamento de atributo usa atributos para definir rotas. O roteamento de atributo oferece mais controle sobre os URIs em seu aplicativo da web.

Roteamento de atributo no MVC #

Junto com a forma clássica de definição de rota MVC WEB API 2 e, em seguida, os frameworks MVC 5 introduziram o Roteamento de atributo:

public class RouteConfig
{
    public static void RegisterRoutes(RouteCollection routes)
    {
        routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
 
        // This enables attribute routing and must go  before other routes are added to the routing table.
        // This makes attribute routes have higher priority
        routes.MapMvcAttributeRoutes();  
    }
}

Para rotas com o mesmo prefixo dentro de um controlador, você pode definir um prefixo comum para métodos de ação inteiros dentro do controlador usando o atributo RoutePrefix.

[RoutePrefix("Custom")]
public class CustomController : Controller
{
    [Route("Index")]
    public ActionResult Index()
    {
        ...
    }
}

RoutePrefix é opcional e define a parte da URL que é prefixada para todas as ações do controlador.

Se você tiver várias rotas, você pode definir uma rota padrão capturando a ação como parâmetro e aplicá-la para todo o controlador, a menos que o atributo Route específico seja definido em determinados métodos de ação que substituam a rota padrão.

[RoutePrefix("Custom")]
[Route("{action=index}")]
public class CustomController : Controller
{
    public ActionResult Index()
    {
        ...
    }

    public ActionResult Detail()
    {
        ...
    }
}

Rota pega-tudo #

Suponha que queremos ter uma rota que permita um número ilimitado de segmentos como:

  • http://example.com/Products/ (ver todos os produtos)
  • http://example.com/Products/IT
  • http://example.com/Products/IT/Laptops
  • http://example.com/Products/IT/Laptops/Ultrabook
  • http://example.com/Products/IT/Laptops/Ultrabook/Asus *etc

Precisaríamos adicionar uma rota, normalmente no final da tabela de rotas, porque isso provavelmente pegaria todas as solicitações, assim:

routes.MapRoute("Final", "Route/{*segments}",
      new { controller = "Product", action = "View" });

No controlador, uma ação que poderia lidar com isso poderia ser:

public void ActionResult View(string[] segments /* <- the name of the parameter must match the name of the route parameter */)
{
    // use the segments to obtain information about the product or category and produce data to the user
    // ...
}

Noções básicas de roteamento #

Adicionando rota personalizada no Mvc #

O usuário pode adicionar uma rota personalizada, mapeando uma URL para uma ação específica em um controlador. Isso é usado para fins de otimização de mecanismos de pesquisa e torna os URLs legíveis.

routes.MapRoute(
  name: "AboutUsAspx", // Route name
  url: "AboutUs.aspx",  // URL with parameters
  defaults: new { controller = "Home", action = "AboutUs", id = UrlParameter.Optional }  // Parameter defaults
);

Rota catch-all para habilitar o roteamento do lado do cliente #

É uma boa prática codificar o estado do Single Page Application (SPA) no URL:

my-app.com/admin-spa/users/edit/id123

Isso permite salvar e compartilhar o estado do aplicativo.
Quando o usuário coloca url na barra de endereços do navegador e pressiona enter, o servidor deve ignorar a parte do lado do cliente da url solicitada. Se você servir seu SPA como uma exibição Razor renderizada (resultado da chamada da ação do controlador) em vez de um arquivo html estático, poderá usar uma rota abrangente:

public class AdminSpaController
{
    [Route("~/admin-spa/{clienSidePart*}")]
    ActionResult AdminSpa()
    {
        ...
    }
}

Neste caso o servidor retorna apenas SPA, e então se inicializa de acordo com a rota. Essa abordagem é mais flexível, pois não depende do módulo url-rewrite.

Roteamento de Atributo em Áreas #

Para usar o Attribute Routing em áreas, é necessário registrar áreas e definições [RouteArea(...)].

Em RouteConfig.cs:

public class RouteConfig
{
    public static void RegisterRoutes(RouteCollection routes)
    {
        routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
        routes.MapMvcAttributeRoutes();
        AreaRegistration.RegisterAllAreas();
    }
}

Em uma definição de roteamento de atributo do controlador de área de amostra:

[RouteArea("AreaName", AreaPrefix = "AreaName")]
[RoutePrefix("SampleAreaController")]
public class SampleAreaController : Controller
{
    [Route("Index")]
    public ActionResult Index()
    {
        return View();
    }
}

Para usar links Url.Action em Áreas :

@Url.Action("Index", "SampleAreaController", new { area = "AreaName" })

Roteamento personalizado #

O roteamento personalizado fornece uma necessidade especializada de roteamento para lidar com solicitações de entrada específicas.

Para definir rotas personalizadas, lembre-se de que a ordem das rotas que você adiciona à tabela de rotas é importante.

public static void RegisterRoutes(RouteCollection routes)
{
    routes.IgnoreRoute("{resource}.axd/{*pathInfo}");
    
    // this is an advanced custom route
    // you can define custom URL with custom parameter(s) point to certain action method
    routes.MapRoute(
    "CustomEntry", // Route name
    "Custom/{entryId}", // Route pattern
    new { controller = "Custom", action = "Entry" } // Default values for defined parameters above
    );

    // this is a basic custom route
    // any custom routes take place on top before default route
    routes.MapRoute(
    "CustomRoute", // Route name
    "Custom/{controller}/{action}/{id}", // Route pattern
    new { controller = "Custom", action = "Index", id = UrlParameter.Optional } // Default values for defined parameters above
    );

    routes.MapRoute(
    "Default", // Route name
    "{controller}/{action}/{id}", // Route pattern
    new { controller = "Home", action = "Index", id = UrlParameter.Optional } // Default values for defined parameters above
    );
}

Os nomes controller e action são reservados. Por padrão, o MVC mapeia a parte {controller} da URL para a classe <controller>Controller e, em seguida, procura um método com o nome <action> sem adicionar nenhum sufixo.

Embora possa ser tentador criar uma família de rotas usando o modelo {controller}/{action}/{parameter} considere que ao fazer isso você divulga a estrutura de sua aplicação e torna as URLs um tanto quebradiças porque alterando o nome do controlador altera a rota e quebra os links salvos pelo usuário.

Prefira a configuração de rota explícita:

routes.MapRoute(
    "CustomRoute", // Route name
    "Custom/Index/{id}", // Route pattern
    new { controller = "Custom", action = nameof(CustomController.Index), id = UrlParameter.Optional }
);

(você não pode usar o operador nameof para o nome do controlador, pois terá sufixo adicional Controller) que deve ser omitido ao definir o nome do controlador na rota.
