Instalação do wxPython Phoenix #

wxPython Phoenix é a versão mais recente do wxPython, (atualmente Setembro de 2016 sem um lançamento oficial). Ele suporta Python 2 e Python 3. Você pode baixar uma compilação de instantâneo (ou seja, uma roda Python) para sua plataforma e versão Python aqui.

wxPython Phoenix utiliza um mecanismo amplamente automatizado para gerar as ligações python para a biblioteca wxWidgets e a documentação. Documentação do Phoenix wxPython é gerado especificamente para si mesmo usando Sphinx. Isso aumenta a clareza em oposição à documentação C++ da compilação clássica, que inclui muitas sobrecargas que não estão disponíveis no wxPython.

Python e pip devem ser instalados antes que o wxPython Phoenix possa ser instalado.

Você pode usar o pip para instalar a versão Phoenix do wxPython. Aqui está o método recomendado atualmente:

python -m pip install --no-index --find-links=http://wxpython.org/Phoenix/snapshot-builds/ --trusted-host wxpython.org wxPython_Phoenix

Ao usar este comando, o pip também instalará wxWidgets. Este comando pip complexo provavelmente se tornará ‘pip install wxpython’ quando o Phoenix for lançado oficialmente.

Observação: o wxPython Phoenix está atualmente em beta e não possui todos os widgets que a versão Classic possui.

Instalação do wxPython Classic #

wxPython Classic é uma compilação Python 2 da biblioteca wxPython. A geração das ligações python requer um grande número de intervenções manuais e a documentação é simplesmente a documentação wxWidgets que contém algumas anotações sobre os mecanismos wxPython como tal, normalmente há um atraso de semanas a meses entre uma nova versão do wxWidgets e a versão correspondente do wxPython .

Vá para a página download no site do wxPython para ver se já existe uma versão do wxPython que você pode baixar para sua plataforma.

A versão mais recente do Classic é 3.0.2.0

Janelas

Existem instaladores para Python 2.6 e 2.7 para plataformas Windows de 32 e 64 bits no site. Basta baixar um desses e executá-los para instalá-lo.

Observação: certifique-se de baixar um instalador wxPython para o Python correto que você instalou. Por exemplo, se você tiver o Python 2.7 de 32 bits, precisará de um instalador wxPython de 32 bits

Mac

Se você tiver o OSX 10.5 ou superior, você desejará baixar e instalar a versão Cocoa do wxPython. A versão Cocoa também suporta Mac de 64 bits.

Se você tiver um Mac com uma versão do OSX inferior a 10.5, você desejará a compilação Carbon.

Linux

A primeira coisa é verificar se o gerenciador de pacotes da sua plataforma Linux (ou seja, yum, apt-get, etc) para ver se ele possui uma versão do wxPython que você pode instalar. Infelizmente, muitos pacotes Linux para wxPython são para a versão 2.8.12.1 em vez de 3.0.2.0. Se o seu gerenciador de pacotes não tiver a versão mais recente, você provavelmente terá que construí-lo você mesmo.

Existem instruções de compilação para 3.0.2.0-Classic aqui

Olá Mundo #

Uma maneira simples de criar um programa Hello World:

import wx
app = wx.App(redirect=False)
frame = wx.Frame(parent=None, id=wx.ID_ANY, title='Hello World')
frame.Show()
app.MainLoop()

Resultado:

Um exemplo mais típico seria a subclasse wx.Frame:

import wx

class MyFrame(wx.Frame):

    def __init__(self):
        wx.Frame.__init__(self, None, title='Hello World')
        self.Show()

if __name__ == '__main__':
    app = wx.App(redirect=False)
    frame = MyFrame()
    app.MainLoop()

Isso também pode ser reescrito para usar o super do Python:

import wx  

class MyFrame(wx.Frame):
        
    def __init__(self, *args, **kwargs):
        """Constructor"""
        super(MyFrame, self).__init__(*args, **kwargs)
        self.Show()

if __name__ == '__main__':
    app = wx.App(False)
    frame = MyFrame(None, title='Hello World')
    app.MainLoop()

O que é uma série de lançamento do wxPython? #

O projeto wxWidgets adotou o modelo de lançamento usado pelo projeto Linux Kernel onde existem conjuntos alternados de lançamentos onde um conjunto é considerado “estável” e o próximo conjunto é considerado “desenvolvimento”. Para wxWidgets “estável” e “desenvolvimento” não se referem a bugs, mas à estabilidade da API e compatibilidade com versões anteriores.

  • Estável: Durante a série, as APIs existentes não são modificadas, embora novos métodos de classe não virtuais e similares possam ser adicionados. A compatibilidade binária das bibliotecas C++ é mantida ao não permitir nenhuma alteração que modifique o tamanho da memória ou o layout das classes e estruturas. Isso pode e muitas vezes impõe limitações sobre quais tipos de aprimoramentos ou correções de bugs podem ser executados em uma série de lançamento estável, no entanto, isso realmente afeta apenas a camada C++ porque em Python ser compatível com versões anteriores tem uma conotação ligeiramente diferente.

  • Desenvolvimento: O objetivo principal da série de lançamentos de desenvolvimento é adicionar novas funcionalidades ou corrigir problemas que não puderam ser corrigidos em uma série estável devido a problemas de compatibilidade binária, tudo em um esforço para criar a próxima série estável . Portanto, durante a série de desenvolvimento existente, as APIs podem ser modificadas ou removidas conforme necessário, embora na maioria das vezes a compatibilidade de nível de fonte C++ seja mantida por meio de funções ou macros sobrecarregadas obsoletas, etc. Para wxPython, isso geralmente significa que haverá ser incompatibilidades no nível de origem porque não há sobrecarga ou macros e, para oferecer suporte à nova versão da API, às vezes a versão antiga precisa ser removida.

Por causa dos problemas de compatibilidade binária, a versão de desenvolvimento mais recente do wxWidgets/wxPython geralmente pode ter menos bugs do que a versão mais recente da última série de lançamentos estáveis. No entanto, há a desvantagem de que as APIs podem estar mudando ou evoluindo entre as versões da série de desenvolvimento.

Como funcionam os números de versão?

Para versões, o wxPython usa um número de versão de 4 componentes. Embora isso se pareça muito com a forma como os números de versão são usados ​​em outros projetos de código aberto, existem algumas diferenças sutis. Então, para alguns lançamentos A.B.C.D, você pode deduzir o seguinte:

  1. Série de lançamento: os dois primeiros componentes do número da versão (A.B) representam a série de lançamento e, se o componente B for um número par, então é uma série estável, se é um número ímpar, então é uma série de lançamento de desenvolvimento. Por exemplo, 2.4, 2.6 e 2.8 são estáveis ​​e a API está mais ou menos congelada em cada série, e 2.3, 2.5 e 2.7 são desenvolvimento e a API e a funcionalidade podem mudar ou evoluir conforme necessário.

Por causa disso, pode haver mudanças bastante grandes entre uma série estável para a próxima (digamos 2,4 a 2,6) e isso muitas vezes afasta as pessoas porque em outros projetos as mudanças dessa magnitude teriam causado a alteração do primeiro componente do número da versão. Em vez disso, você deve pensar na combinação de A.B como sendo o número principal da versão.

  1. Número de lançamento: O terceiro componente do número da versão (C) representa um dos lançamentos em uma série de lançamentos. Por exemplo, 2.5.0, 2.5.1, 2.5.2, 2.5.3… são todas as versões da série 2.5. (E como neste caso é uma série de desenvolvimento, então a API e a funcionalidade do 2.5.3 evoluíram para serem diferentes em alguns lugares do que eram no 2.5.0.) Os lançamentos C++ wxWidgets geralmente param aqui e apenas lançamentos A.B.C são feitos.

  2. Número do sublançamento, ou lançamento do wxPython: O quarto componente do número da versão (D) é usado para representar um sublançamento ou lançamentos incrementais entre os lançamentos oficiais do wxWidgets. Esses lançamentos incluem correções para bugs do wxWidgets que o wxPython pode ter exposto ou pequenas melhorias que são importantes para o wxPython. Este não é um instantâneo arbitrário do wxWidgets, mas sim uma versão testada do código com correções e aprimoramentos ainda não disponíveis no wxWidgets, exceto no repositório de código-fonte.
