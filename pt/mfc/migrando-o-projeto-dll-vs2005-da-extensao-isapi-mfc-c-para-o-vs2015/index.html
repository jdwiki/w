Migrando o projeto DLL VS2005 da extensão ISAPI MFC (C++) para o VS2015.

  • Exemplo:

Nesta página

  • Exemplo:

Você pode ter visto vários sites que mostrarão como criar um projeto de extensão ISAPI, mas nenhum deles demonstrará como migrar o projeto de extensão ISAPI legado existente (VS2005) para o VS2015. Eu enfrentei um problema semelhante enquanto trabalhava em um desses requisitos. Este artigo demonstra a maneira experimental que tomei para resolver meu problema.

Eu estava trabalhando em uma tarefa de migração onde me deparei com um projeto. O projeto ISAPI legado era um projeto DLL de extensão MFC construído no VS2005 e depois de pesquisar algumas coisas eu vim a saber que as classes ISAPI do MFC (CHttpServerContext, CHttpServer etc.) não estão sendo enviadas após o VS2005. A Microsoft recomenda usar funções de ponto de entrada ISAPI do kit de desenvolvimento de software (SDK) do Microsoft Internet Information Services (IIS) em vez de classes ISAPI do MFC. Achei que poderia exigir uma reescrita completa do projeto. No entanto, felizmente, tomei uma forma experimental que funcionou no meu caso. Eu pensei em escrever este artigo que pode ajudar alguém a resolver o problema semelhante.

O objetivo deste tópico é demonstrar as alterações necessárias para remover a dependência do MFC da DLL de extensão ISAPI do MFC sem reescrever o código completo. O exemplo mostrado acima não contém toda a implementação, é apenas para fins de demonstração.

Objetivo do projeto: Baixar um arquivo específico com base no ID da mensagem enviada como uma solicitação do cliente para o servidor. O arquivo será criado no servidor e transmitido ao cliente.

Aqui, não compartilhei todo o código, mas mostrei o que é necessário, pois o objetivo é demonstrar as alterações necessárias para a migração.

Para fazer as alterações. Peguei a referência de afxisapi.h, afxisapi.inl e HttpExt.h existentes que estão sendo enviados com o VS2005.

  • Use EXTENSION_CONTROL_BLOCK em vez de CHttpServerContext. EXTENSION_CONTROL_BLOCK é uma estrutura principal que é usada pelo IIS e pela extensão ISAPI para trocar as informações. Ele contém todas as informações sobre a nova solicitação.

  • Remova a dependência da classe CHttpServer da sua classe de extensão MFC. Remova o mapeamento de mensagens de análise do comando http se ele for usado em seu projeto. Macros como ON_PARSE_COMMAND, BEGIN_PARSE_MAP, END_PARSE_MAP e DEFAULT_PARSE_COMMAND foram removidas após o VS2005.

  • Implemente sua própria função GetExtensionVersion, TerminateExtension e HttpExtensionProc em sua classe de extensão MFC que anteriormente era derivada da classe CHttpSever. Estas funções

  • GetExtensionVersion() A função GetExtensionVersion é a primeira função de ponto de entrada no IIS. Esta função permite que sua extensão ISAPI registre suas informações de versão no IIS.

    BOOL CISAPIExtension::GetExtensionVersion( HSE_VERSION_INFO* pVer ) { ISAPIVERIFY( ::LoadString( AfxGetResourceHandle(), IDS_SERVER, sz, HSE_MAX_EXT_DLL_NAME_LEN) );

  • HttpExtensionProc() : A função HttpExtensionProc é o ponto de entrada principal para uma extensão ISAPI chamada pelo IIS. Ele expõe métodos que o IIS usa para acessar a funcionalidade exposta pela extensão. Extraia as informações da solicitação do bloco de controle de extensão e faça seu processamento.

  • Use ServerSupportFunction de EXTENSION_CONTROL_BLOCK em vez da função TransmitFile de CHttepServerContext. Anteriormente, TransmitFile() eram usados ​​para transferência de arquivos.

  • Use HSE_REQ_TRANSMIT_FILE em ServerSupportFunction para transferência de arquivos.

  • Puxe as informações do arquivo de transmissão na estrutura HSE_TF_INFO.

  • Crie uma função de retorno de chamada (AsyncIOCompletionFunction) que será chamada na conclusão de E/S assíncrona.

    void WINAPI AsyncIOCompletionFunction( EXTENSION_CONTROL_BLOCK * pECB, PVOID pContext, DWORD cbIO, DWORD dwError) { HSE_CUSTOM_ERROR_INFO* pErrorInfo = (HSE_CUSTOM_ERROR_INFO*)pContext; DWORD dwIOStatus = dwError == NO_ERROR ? HSE_STATUS_SUCCESS : HSE_STATUS_ERROR;
