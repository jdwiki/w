Instalação e configuração #

Você pode baixar o Mercurial do site do projeto, e existem utilitários gráficos para Windows, Linux e OSX se você preferir isso a uma interface de linha de comando. A maioria dos gerenciadores de pacotes Unix inclui o Mercurial, por exemplo no Debian/Ubuntu:

$ apt-get install mercurial

Você pode verificar se o Mercurial está instalado executando:

$ hg --version

Configurar #

O Mercurial funciona imediatamente, mas você provavelmente desejará configurar o Mercurial para saber quem você é antes de prosseguir. Para associar um nome de usuário aos seus commits, edite ~/.hgrc (ou mercurial.ini em seu diretório inicial no Windows) e adicione as seguintes linhas:

[ui]
username = Your Name <your@email.address>

Se você não quiser fazer isso, você sempre pode especificar um nome de usuário quando fizer o commit com o sinalizador -u, por exemplo:

$ hg commit -u "Your Name <your@email.address>"

Criando um repositório Mercurial #

Um repositório Mercurial é simplesmente um diretório (referido como “diretório de trabalho”) contendo um diretório .hg com metadados sobre o conteúdo do repositório. Isso torna o Mercurial muito leve e fácil de começar a usar. Para criar um novo repositório basta executar:

$ hg init project

Onde project é o nome do diretório que você gostaria de criar. Isso cria um diretório project junto com um diretório project/.hg contendo o próprio repositório.

   $ cd project
   $ echo Hello World > hello.txt
   $ hg stat
   ? hello.txt

Acabamos de criar um arquivo hello.txt no repositório e executamos hg status (ou stat para abreviar) para ver o status atual de nosso repositório. Como você pode ver, hello.txt é anotado com um ?, significando que o Mercurial ainda não está ciente disso. O comando add registra este novo arquivo no Mercurial para que seja incluído no próximo commit.

$ hg add hello.txt

Agora que o Mercurial está ciente de um arquivo alterado, você pode executar o diff para ver exatamente o que mudou desde o último commit - neste caso, estamos adicionando o conteúdo completo do hello.txt:

$ hg diff
diff -r 000000000000 hello.txt
--- /dev/null   Thu Jan 01 00:00:00 1970 +0000
+++ b/hello.txt Sat Jul 23 01:38:44 2016 -0400
@@ -0,0 +1,1 @@
+Hello

E quando estivermos satisfeitos com eles e prontos para verificar nossas alterações, podemos executar commit:

$ hg commit -m "Created a hello world file."

Note que nós incluímos uma mensagem de commit com -m - se você não especificar -m o Mercurial irá lançar um editor de texto no qual você pode inserir uma mensagem de commit. mensagem de linha.

Uma vez que você tenha submetido suas alterações, elas não aparecerão mais se você executar hg stat já que o repositório está agora sincronizado com o conteúdo do diretório de trabalho. Você pode executar log para ver uma lista de commits, e -v inclui detalhes adicionais como os arquivos que cada commit tocou:

$ hg log -v
changeset:   0:b4c06cc77a42
tag:         tip
user:        Michael Diamond@Aodh <dimo414@gmail.com>
date:        Sat Jul 23 01:44:23 2016 -0400
files:       hello.txt
description:
Created a hello world file.

Empurrando e puxando #

O Mercurial facilita o compartilhamento de seu trabalho e a obtenção de contribuições de outros desenvolvedores. Isso envolve três etapas principais; clonagem, puxando e empurrando.

Clonar #

Para copiar um repositório remoto para o seu disco local, você o “clona”. Para fazer isso, basta passar a URL remota da qual você deseja clonar. Para clonar o código-fonte do Mercurial basta executar:

$ hg clone https://selenic.com/hg

Isso cria um diretório local hg contendo uma cópia do repositório Mercurial que você pode construir, editar e submeter (embora você não possa publicar seus commits de volta no repositório pai).

Puxar #

Depois de fazer check-out de um repositório, você desejará mantê-lo sincronizado enquanto outros publicam alterações nele. Você pode obter novas alterações simplesmente executando:

$ hg pull

Isso puxa novos commits, mas não atualiza seu diretório de trabalho, então você não verá nenhuma alteração imediatamente. Para atualizar o conteúdo do diretório de trabalho execute:

$ hg up

Que atualiza seu diretório de trabalho para a revisão tip (mais recente) no repositório.

Você também pode executar:

$ hg pull -u

Para obter novas alterações e atualizar o diretório de trabalho em uma etapa.

Empurre #

Supondo que você tenha acesso de gravação ao repositório remoto, você pode publicar quaisquer commits que você fez localmente no repositório remoto com a mesma facilidade com:

$ hg push

Isso carrega suas alterações, desde que não tenha havido nenhum outro commit desde a última vez que você puxou. Se o seu push for rejeitado porque “criaria cabeças adicionais”, isso significa que você precisa puxar essas novas alterações e mesclá-las com as suas.

$ hg pull
$ hg merge  # this creates a new changeset merging your changes with the remote changes
$ hg commit -m "Merged in remote changes"
$ hg push

Na maioria das vezes, isso é tudo o que você precisa fazer, pois o Mercurial lida com a mesclagem de suas alterações automaticamente, no entanto, às vezes, você precisará resolver os conflitos de mesclagem manualmente (consulte o tópico sobre mesclagem). Se precisar, você sempre pode cancelar uma mesclagem e voltar para um diretório de trabalho limpo com:

$ hg up -c

Mas lembre-se que esta é uma operação destrutiva; quaisquer alterações no diretório de trabalho serão apagadas.

Ramificação #

Quando começamos nosso trabalho, temos que decidir se esta é uma área separada de trabalho em que estamos trabalhando ou se faz parte de uma linha de trabalho existente. Se já existir, podemos trabalhar a partir desse ramo. Se for novo, iniciaremos uma nova filial.

Nosso fluxo de trabalho então é:

  • hg branch MyNewFeature
  • trabalho Trabalho trabalho
  • hg commit -m "confirmando minhas alterações"
  • trabalho Trabalho trabalho
  • hg commit -m "mais alterações"

Neste ponto, queremos enviar nosso trabalho para o servidor remoto. Mas antes de enviar as alterações (ignore isso se for uma nova ramificação que você não enviou antes), precisamos verificar se há alguma alteração recebida nesta ramificação. Podemos verificar isso com:

hg incoming -b .

Se houver algum conjunto de alterações de entrada em nossa ramificação, precisamos fazer um pull e rebasear nossas alterações para o topo da lista de alterações.

hg pull -b . --rebase

Uma vez feito isso ou se não houver conjuntos de alterações de entrada, podemos prosseguir com o Push.

Nós só queremos impulsionar nosso trabalho atual, não tudo o que já fizemos. Eu realmente nunca empurro todo o meu repositório, mas minha linha de trabalho atual. O raciocínio é que enviar todo o repositório pressupõe que estou integrando várias linhas de trabalho. Mas eu só quero integrar minha linha de trabalho atual e só quero trabalhar em uma linha de cada vez.

Se esta é a primeira vez que estou empurrando este branch:

hg push -b . --new-branch

Se eu já empurrei este branch:

hg push -b .

O “-b”. comando significa apenas enviar o branch atual, e nada mais.

Para alternar entre os ramos de trabalho:

hg update myBranchName
