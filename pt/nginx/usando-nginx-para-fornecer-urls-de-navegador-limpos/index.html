Redirecionamento vs proxy reverso #

Os aplicativos da Web feitos profissionalmente não expõem os detalhes internos do ambiente do servidor ao usuário. Ao fazer um pedido em seu revendedor, você não vê (ou precisa digitar) https://mydealer.com:8443/Dealerapp/entryPage.html, mas apenas mydealer.com, embora o servidor do aplicativo precise de todos os detalhes fornecido no URL mais longo. Isso pode ser alcançado através de:

if ($scheme = http) {
    return 301 https://$server_name$request_uri;
}

Isso faz um redirecionamento. Mesmo que o cliente não tenha solicitado uma conexão segura, o navegador é redirecionado imediatamente para ela. Em países com leis rígidas de privacidade de dados, você pode até ser obrigado a fazer isso para qualquer site comercial. A forma de redirecionamento é tomada porque aqui o navegador precisa saber sobre a conexão segura, caso contrário não negociaria com o servidor para torná-la segura.

location /app/ {
    proxy_pass https://mydealer.com:8443/Dealerapp/entryPage.html;
}

Este é um proxy reverso. É transparente para o navegador. Assim, você pode ocultar completamente do usuário final se você tem um ou mais servidores de aplicativos, em quais portas eles escutam ou como seus aplicativos são nomeados. Caso contrário, se seu datacenter precisar mover o aplicativo para outro servidor que escute em 8543, todos os favoritos de seus clientes se tornarão inválidos. Este exemplo também direciona o usuário para sua página de entrada. Isso pode ser omitido se você nomear a página de entrada como index.html. Mas talvez você tenha várias páginas de entrada em um aplicativo que o usuário pode querer marcar, então você é mais flexível se não estiver vinculado ao index.html.

Eu postfixei a URL da empresa com /app. Isso pode ser omitido se seu domínio veicular apenas este aplicativo. Caso, além do aplicativo, também haja algum conteúdo estático, como a descrição da sua empresa, você pode querer ter o URL simples para isso.

Este proxy só funciona para a página de entrada. Normalmente, preciso de mais dois esquemas de URL, um para conteúdo estático do aplicativo da Web, como JavaScript, CSS e imagens. Eu coloco tudo isso em uma estrutura de pastas abaixo de uma pasta chamada serverapp e escrevo o seguinte proxy:

location /app/serverapp/ {
    proxy_pass https://mydealer.com:8443/Dealerapp/serverapp/;
}

E outro caminho para serviços REST; aqui, o caminho da URL restante não corresponde a uma pasta, mas a um caminho de um serviço REST jax-rs:

location /rest/ {
    proxy_pass https://mydealer.com:8443/Dealerapp/rest/;
}

Mais um passo, raramente mencionado em qualquer lugar, precisa ser dado. O servidor de aplicativos é executado no caminho de URL /Dealerapp, portanto, ele emitirá um cookie de sessão com a propriedade path=Dealerapp. O navegador não conhece esse caminho e, devido à sua Política de Mesma Origem, ignorará o cookie. Poderíamos convencê-lo por meio do Compartilhamento de Recursos de Origem Cruzada para permitir isso, mas provavelmente é mais fácil modificar o caminho do cookie definindo o caminho para /, escrevendo algo como

<session-config>
    <session-timeout>720</session-timeout>
    <cookie-config>
        <name>SZSESSION</name>
        <path>/</path>
        <http-only>true</http-only>
        <secure>true</secure>
    </cookie-config>
</session-config>

para o nosso web.xml.
