Instalação ou configuração #

Instruções detalhadas sobre como configurar ou instalar o ravendb.

Um simples aplicativo de console RavenDB #

Para este exemplo, usaremos a instância do Live Test RavenDB.

Vamos construir um aplicativo de console simples aqui que demonstra as operações mais básicas:

  • Criação
  • Recuperação por ID
  • Consulta
  • Atualizando
  • Exclusão

Comece criando uma nova solução do Visual Studio e adicione um projeto de aplicativo de console a ela. Vamos chamá-lo de RavenDBDemoConsole. O uso da biblioteca cliente deve ser semelhante se o Leitor usar o VS Code ou seu editor favorito.

Em seguida, precisamos adicionar as referências necessárias. Clique com o botão direito do mouse no nó Referências no painel Gerenciador de Soluções e selecione Gerenciar pacotes NuGet. Procure online por ‘RavenDb.Client’. Estarei usando a última versão estável, que é - até o momento em que escrevo - 3.5.2.

Vamos escrever algum código, certo? Comece adicionando as seguintes instruções using:

using Raven.Client;
using Raven.Client.Document;

Isso nos permite usar o IDocumentStore e o DocumentStore do RavenDB, que é uma interface e sua implementação pronta para uso para conectar a uma instância do RavenDB. Este é o objeto de nível superior que precisamos usar para conectar a um servidor e a documentação do RavenDB dele informa que ele é usado como um singleton no aplicativo.

Então, iremos em frente e criaremos um, mas para simplificar, não implementaremos o wrapper singleton em torno dele - apenas o descartaremos quando o programa sair para que a conexão seja fechada de maneira limpa. Adicione o seguinte código ao seu método principal:

using (IDocumentStore store = new DocumentStore
{
    Url = "http://live-test.ravendb.net",
    DefaultDatabase = "Pets"
})
{
    store.Initialize();
}

Como dito no início, usamos a Instância do Live Test RavenDB, usamos seu endereço como a propriedade Url da DocumentStore. Também especificamos um nome de banco de dados padrão, neste caso, “Animais de estimação”. Se o banco de dados ainda não existir, RavenDB o cria ao tentar acessá-lo. Se existir, o cliente poderá usar o existente. Precisamos chamar o método Initialize() para que possamos começar a operar nele.

Nesta aplicação simples, vamos manter os donos e animais de estimação. Pensamos em sua conexão como um dono pode ter um número arbitrário de animais de estimação, mas um animal de estimação pode ter apenas um dono. Mesmo que no mundo real, um animal de estimação possa ter um número arbitrário de donos, por exemplo, um marido e uma esposa, optaremos por essa suposição, pois os relacionamentos muitos-para-muitos em um banco de dados de documentos são tratados de maneira um pouco diferente em um banco de dados relacional e merece um tópico próprio. Eu escolhi este domínio porque é comum o suficiente para entender.

Então, agora devemos definir nossos tipos de objeto de domínio:

public class Owner
{
    public Owner()
    {
        Pets = new List<Pet>();
    }

    public string Id { get; set; }
    public string Name { get; set; }
    public List<Pet> Pets { get; set; }

    public override string ToString()
    {
        return
            "Owner's Id: " + Id + "\n" +
            "Owner's name: " + Name + "\n" +
            "Pets:\n\t" +
            string.Join("\n\t", Pets.Select(p => p.ToString()));
    }
}

public class Pet
{
    public string Color { get; set; }
    public string Name { get; set; }
    public string Race { get; set; }

    public override string ToString()
    {
        return string.Format("{0}, a {1} {2}", Name, Color, Race);
    }
}

Há algumas coisas a serem observadas aqui:

Em primeiro lugar, nossos Proprietários podem conter zero ou mais Pets. Observe que a classe Owner tem uma propriedade chamada Id enquanto a classe Pet não. Isso ocorre porque os objetos Pet serão armazenados dentro dos objetos Owner, o que é bastante diferente de como esse tipo de relacionamento seria implementado em um banco de dados relacional.

Pode-se argumentar que isso não deve ser implementado assim - e pode estar certo, realmente depende dos requisitos. Como regra geral, se um Pet faz sentido existir sem um Proprietário, então ele não deve ser incorporado, mas existe por conta própria com um identificador próprio. Em nossa aplicação, assumimos que um Pet só é considerado um animal de estimação se tiver um dono, caso contrário seria um bicho ou uma fera. Por esta razão, não adicionamos uma propriedade Id à classe Pet.

Em segundo lugar, observe que o identificador da classe proprietária é uma string, como geralmente é mostrado nos exemplos da documentação do RavenDB. Muitos desenvolvedores acostumados a bancos de dados relacionais podem considerar isso uma má prática, o que geralmente faz sentido no mundo relacional. Mas porque RavenDB usa Lucene.Net para realizar suas tarefas e porque Lucene.Net é especializado em operar com strings é perfeitamente aceitável aqui - também, estamos lidando com um banco de dados de documentos que armazena JSON e, afinal, basicamente tudo é representado como uma string em JSON.

Mais uma coisa a notar sobre a propriedade Id é que ela não é obrigatória. Na verdade, RavenDB anexa seus próprios metadados a qualquer documento que salvamos, então mesmo que não o definissemos, RavenDB não teria problemas com nossos objetos. No entanto, é geralmente definido para facilitar o acesso.

Antes de vermos como podemos usar o RavenDB do nosso código, vamos definir alguns métodos auxiliares comuns. Estes devem ser autoexplicativos.

// Returns the entered string if it is not empty, otherwise, keeps asking for it.
private static string ReadNotEmptyString(string message)
{
    Console.WriteLine(message);
    string res;
    do
    {
        res = Console.ReadLine().Trim();
        if (res == string.Empty)
        {
            Console.WriteLine("Entered value cannot be empty.");
        }
    } while (res == string.Empty);

    return res;
}

// Will use this to prevent text from being cleared before we've read it.
private static void PressAnyKeyToContinue()
{
    Console.WriteLine();
    Console.WriteLine("Press any key to continue.");
    Console.ReadKey();
}

// Prepends the 'owners/' prefix to the id if it is not present (more on it later)
private static string NormalizeOwnerId(string id)
{
    if (!id.ToLower().StartsWith("owners/"))
    {
        id = "owners/" + id;
    }

    return id;
}

// Displays the menu
private static void DisplayMenu()
{
    Console.WriteLine("Select a command");
    Console.WriteLine("C - Create an owner with pets");
    Console.WriteLine("G - Get an owner with its pets by Owner Id");
    Console.WriteLine("N - Query owners whose name starts with...");
    Console.WriteLine("P - Query owners who have a pet whose name starts with...");
    Console.WriteLine("R - Rename an owner by Id");
    Console.WriteLine("D - Delete an owner by Id");
    Console.WriteLine();
}

E nosso método principal:

private static void Main(string[] args)
{
    using (IDocumentStore store = new DocumentStore
    {
        Url = "http://live-test.ravendb.net", 
        DefaultDatabase = "Pets"
    })
    {
        store.Initialize();

        string command;
        do
        {
            Console.Clear();
            DisplayMenu();

            command = Console.ReadLine().ToUpper();
            switch (command)
            {
                case "C":
                    Creation(store);
                    break;
                case "G":
                    GetOwnerById(store);
                    break;
                case "N":
                    QueryOwnersByName(store);
                    break;
                case "P":
                    QueryOwnersByPetsName(store);
                    break;
                case "R":
                    RenameOwnerById(store);
                    break;
                case "D":
                    DeleteOwnerById(store);
                    break;
                case "Q":
                    break;
                default:
                    Console.WriteLine("Unknown command.");
                    break;
            }
        } while (command != "Q");
    }
}

Criação #

Vamos ver como podemos salvar alguns objetos no RavenDB. Vamos definir os seguintes métodos comuns:

private static Owner CreateOwner()
{
    string name = ReadNotEmptyString("Enter the owner's name.");

    return new Owner { Name = name };
}

private static Pet CreatePet()
{
    string name = ReadNotEmptyString("Enter the name of the pet.");
    string race = ReadNotEmptyString("Enter the race of the pet.");
    string color = ReadNotEmptyString("Enter the color of the pet.");

    return new Pet
    {
        Color = color,
        Race = race,
        Name = name
    };
}

private static void Creation(IDocumentStore store)
{
    Owner owner = CreateOwner();
    Console.WriteLine(
        "Do you want to create a pet and assign it to {0}? (Y/y: yes, anything else: no)", 
        owner.Name);

    bool createPets = Console.ReadLine().ToLower() == "y";
    do
    {
        owner.Pets.Add(CreatePet());

        Console.WriteLine("Do you want to create a pet and assign it to {0}?", owner.Name);
        createPets = Console.ReadLine().ToLower() == "y";
    } while (createPets);

    using (IDocumentSession session = store.OpenSession())
    {
        session.Store(owner);
        session.SaveChanges();
    }
}

Agora vamos ver como funciona. Definimos uma lógica simples de C# para criar objetos Owner e continuar criando e atribuindo a eles objetos Pet até que o usuário assim o deseje. A parte em que o RavenDB está preocupado e, portanto, é o foco deste artigo é como salvamos objetos.

Para salvar o Owner recém-criado junto com seus Pets, primeiro precisamos abrir uma sessão, que implementa IDocumentSession. Podemos criar um chamando o OpenSession no objeto de armazenamento de documentos.

Portanto, observe a diferença, enquanto o armazenamento de documentos é um objeto permanente que geralmente existe durante todo o tempo de vida do aplicativo, o IDocumentSession é um objeto leve e de curta duração. Ele representa uma série de operações que queremos realizar de uma só vez (ou pelo menos, em apenas algumas chamadas ao banco de dados).

RavenDB enfatiza (e um pouco força) que você evite um número excessivo de viagens de ida e volta para o servidor, algo que eles chamam de ‘proteção de conversa cliente-servidor’ no site RavenDB. Por essa mesma razão, uma sessão tem um limite padrão para quantas chamadas de banco de dados ela tolerará e, portanto, deve-se prestar atenção em quando uma sessão é aberta e descartada. Como neste exemplo, tratamos a criação de um Proprietário e seus Pets como uma operação que deve ser executada por conta própria, fazemos isso em uma sessão e depois descartamos.

Podemos ver mais duas chamadas de método que nos interessam:

  • session.Store(proprietário), que registra o objeto para salvar, e adicionalmente, configura a propriedade Id do objeto caso ainda não esteja configurada. O fato de a propriedade identificadora ser chamada de ‘Id’ é, portanto, uma convenção.
  • session.Savehanges() envia as operações reais a serem executadas para o servidor RavenDB, confirmando todas as operações pendentes.

Recuperação por ID #

Outra operação comum é obter um objeto pelo seu identificador. No mundo relacional, normalmente fazemos isso usando uma expressão Where, especificando o identificador. Mas como no RavenDB, cada consulta é feita usando índices, que podem ser stale, não é a abordagem a ser tomada - na verdade, RavenDB lança uma exceção se tentarmos consultar por id. Em vez disso, devemos usar o método Load<T>, especificando o id. Com nossa lógica de menu já em vigor, precisamos apenas definir o método que realmente carrega os dados solicitados e exibe seus detalhes:

private static void GetOwnerById(IDocumentStore store)
{
    Owner owner;
    string id = NormalizeOwnerId(ReadNotEmptyString("Enter the Id of the owner to display."));

    using (IDocumentSession session = store.OpenSession())
    {
        owner = session.Load<Owner>(id);
    }

    if (owner == null)
    {
        Console.WriteLine("Owner not found.");
    }
    else
    {
        Console.WriteLine(owner);
    }

    PressAnyKeyToContinue();
}

Tudo o que é relacionado ao RavenDB aqui é, mais uma vez, a inicialização de uma sessão, então usando o método Load. A biblioteca cliente RavenDB retornará o objeto desserializado como o tipo que especificamos como parâmetro de tipo. É importante saber que RavenDB não impõe nenhum tipo de compatibilidade aqui - todas as propriedades mapeáveis ​​são mapeadas e as não mapeáveis ​​não.

RavenDB precisa do prefixo do tipo de documento anexado ao Id - essa é a razão para a chamada de NormalizeOwnerId. Se um documento com o ID especificado não existir, será retornado null.

Consultando #

Veremos dois tipos de consultas aqui: uma na qual consultamos as próprias propriedades dos documentos Proprietário e outra na qual consultamos os objetos Pet incorporados.

Vamos começar com o mais simples, no qual consultamos os documentos Owner cuja propriedade Name começa com a string especificada.

private static void QueryOwnersByName(IDocumentStore store)
{
    string namePart = ReadNotEmptyString("Enter a name to filter by.");

    List<Owner> result;
    using (IDocumentSession session = store.OpenSession())
    {
        result = session.Query<Owner>()
           .Where(ow => ow.Name.StartsWith(namePart))
           .Take(10)
           .ToList();
    }

    if (result.Count > 0)
    {
        result.ForEach(ow => Console.WriteLine(ow));
    }
    else
    {
        Console.WriteLine("No matches.");
    }
    PressAnyKeyToContinue();
}

Mais uma vez, porque gostaríamos de realizar a consulta como um trabalho independente, abrimos uma sessão. Podemos consultar uma coleção de documentos chamando Query<TDocumentType> no objeto de sessão. Ele retorna um objeto IRavenQueryable<TDocumentType>, no qual podemos chamar os métodos LINQ usuais, bem como algumas extensões específicas do RavenDB. Fazemos uma filtragem simples aqui, e a condição é que o valor da propriedade Name comece com a string inserida. Pegamos os primeiros 10 itens do conjunto de resultados e criamos uma lista deles. Deve-se prestar atenção para especificar corretamente o tamanho do conjunto de resultados - aqui está outra aplicação defensiva em jogo feita pelo RavenDB chamada Proteção ilimitada do conjunto de resultados. Isso significa que (por padrão) apenas os primeiros 128 itens são retornados.

Nossa segunda consulta se parece com o seguinte:

private static void QueryOwnersByPetsName(IDocumentStore store)
{
    string namePart = ReadNotEmptyString("Enter a name to filter by.");

    List<Owner> result;
    using (IDocumentSession session = store.OpenSession())
    {
       result = session.Query<Owner>()
           .Where(ow => ow.Pets.Any(p => p.Name.StartsWith(namePart)))
           .Take(10)
           .ToList();
    }

    if (result.Count > 0)
    {
        result.ForEach(ow => Console.WriteLine(ow));
    }
    else
    {
        Console.WriteLine("No matches.");
    }
    PressAnyKeyToContinue();
}

Este não é muito mais complicado, eu o escrevi para demonstrar como naturalmente é possível consultar propriedades de objetos incorporados. Esta consulta simplesmente retorna os primeiros 10 Proprietários que tenham pelo menos um Pet cujo nome comece com o valor inserido.

Eliminação #

Temos duas opções para realizar a exclusão. Uma é passar o identificador do documento, o que é útil se não tivermos o próprio objeto na memória, mas tivermos o identificador e gostaríamos de evitar uma viagem de ida e volta ao banco de dados evitável. A outra maneira, obviamente, é passar um objeto real salvo para RavenDB. Veremos a primeira opção aqui, a outra está apenas usando uma outra sobrecarga e passando um objeto apropriado:

private static void DeleteOwnerById(IDocumentStore store)
{
    string id = NormalizeOwnerId(ReadNotEmptyString("Enter the Id of the owner to delete."));

    using (IDocumentSession session = store.OpenSession())
    {
        session.Delete(id);
        session.SaveChanges();
    }
}

Mais uma vez, precisamos abrir uma sessão para realizar nosso trabalho. Como mencionado anteriormente, aqui excluímos o objeto desejado passando seu identificador para o método Delete. O prefixo do identificador também deve ser colocado aqui, assim como foi o caso do método Load. Para realmente enviar o comando delete para o banco de dados, precisamos chamar o método SaveChanges, que fará exatamente isso, junto com quaisquer outras operações pendentes registradas na mesma sessão.

Atualizando #

E, finalmente, veremos como atualizar documentos. Basicamente, temos duas maneiras de fazer isso. A primeira é simples, carregamos um documento, atualizamos suas propriedades conforme necessário, então passamos para o método Store. Isso deve ser simples de acordo com a demonstração de carregamento e salvamento, mas há algumas coisas dignas de nota.

Primeiro, a biblioteca cliente RavenDB usa um rastreador de alterações que possibilita atualizar qualquer documento sem realmente passá-lo para Store, desde que a sessão que carregou o documento ainda esteja aberta. Nesse caso, chamar SaveChanges na sessão é suficiente para que a atualização ocorra.

Em segundo lugar, para que isso funcione, o objeto obviamente precisa que seu identificador seja definido para que o RavenDB possa descobrir o que atualizar.

Com isso dito, vamos apenas dar uma olhada na outra forma de atualização. Existe um conceito chamado patching, que pode ser usado para atualizar documentos. Assim como foi o caso do delete, ele também possui seus próprios cenários de uso. Usar o método anterior para realizar uma atualização é uma boa maneira se já tivermos o objeto na memória e/ou quisermos usar sua segurança de tipo. Usar o patch é a opção se quisermos evitar uma viagem de ida e volta desnecessária ao banco de dados se ainda não tivermos o objeto na memória. A desvantagem é que perdemos um pouco da segurança de tipo, pois devemos especificar as propriedades a serem atualizadas usando strings simples (nada que algum LINQ-magic não resolva). Vamos ver o código:

private static void RenameOwnerById(IDocumentStore store)
{
    string id = NormalizeOwnerId(ReadNotEmptyString("Enter the Id of the owner to rename."));
    string newName = ReadNotEmptyString("Enter the new name.");

    store.DatabaseCommands.Patch(id, new Raven.Abstractions.Data.PatchRequest[]{
        new Raven.Abstractions.Data.PatchRequest
        {
            Name = "Name",
            Value = newName
        }
    });
}

Isso encerra. Você deve conseguir ver o exemplo funcionando colando os fragmentos de código em um aplicativo de console.
