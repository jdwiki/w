Adicionar comportamento ao objeto simulado #

Mockito.when(mock.returnSomething()).thenReturn("my val");

mock.returnSomething(); // returns "my val"
mock.returnSomething(); // returns "my val" again
mock.returnSomething(); // returns "my val" again and again and again...

Se você quiser um valor diferente na segunda chamada, você pode adicionar o argumento de retorno desejado ao método thenReturn:

Mockito.when(mock.returnSomething()).thenReturn("my val", "other val");

mock.returnSomething(); // returns "my val"
mock.returnSomething(); // returns "other val"
mock.returnSomething(); // returns "other val" again

Se você chamar o método sem adicionar comportamento para simular, ele retornará null:

barMock.mock.returnSomethingElse(); // returns null

Caso esse método mocked tenha parâmetros, você deve declarar valores também:

Mockito.when(mock.returnSomething("param 1")).thenReturn("my val 1");
Mockito.when(mock.returnSomething("param 2")).thenReturn("my val 2");

mock.returnSomething("param 1"); // returns "my val 1"
mock.returnSomething("param 2"); // returns "my val 2"
mock.returnSomething("param 3"); // returns null

Se você não se importa com o valor do parâmetro, você pode usar Matchers.any():

Mockito.when(mock.returnSomething(Matchers.any())).thenReturn("p1");

mock.returnSomething("param 1"); // returns "p1"
mock.returnSomething("param other"); // returns "p1"

Para lançar exceção, use o método thenThrow:

Mockito.when(mock.returnSomething()).thenThrow(new Exception());

mock.returnSomething(); // throws Exception

Crie objetos zombados por Mockito #

Existem duas maneiras de criar o objeto mockado pelo Mockito:

  • por anotação
  • via função simulada

Através de anotação:

Com um executor de testes JUnit:

@RunWith(MockitoJUnitRunner.class)
public class FooTest {
    @Mock
    private Bar barMock;

    // ...
}

Você também pode usar o JUnit @Rule do Mockito, que fornece a mesma funcionalidade que o MockitoJUnitRunner, mas não precisa de um executor de teste @RunWith:

public class FooTest {
    @Rule
    public MockitoRule mockito = MockitoJUnit.rule();        

    @Mock
    private Bar barMock;

    // ...
}

Se você não pode usar @RunWith ou a anotação @Rule, você também pode iniciar simulações “por mão”:

public class FooTest {
    @Mock
    private Bar barMock;

    @Before
    public void setUp() {
        MockitoAnnotations.initMocks(this);
    }

    // ...
}

Através da função simulada:

public class FooTest {
    private Bar barMock = Mockito.mock(Bar.class);

    // ...
}

Por causa do apagamento de tipo, você não pode simular uma classe genérica como acima. Você deve zombar da classe base e explicitamente converter para o tipo genérico correto:

public class FooTest {
    private Bar<String> genericBarMock = (Bar<String>) Mockito.mock(Bar.class);

    // ...
}

Verifique as chamadas de método no objeto simulado #

Para verificar se um método foi chamado em um objeto simulado, você pode usar o método Mockito.verify:

Mockito.verify(someMock).bla();

Neste exemplo, afirmamos que o método bla foi chamado no objeto mock someMock.

Você também pode verificar se um método foi chamado com determinados parâmetros:

Mockito.verify(someMock).bla("param 1");

Se você quiser verificar se um método não foi chamado, você pode passar um parâmetro adicional VerificationMode para verify:

Mockito.verify(someMock, Mockito.times(0)).bla();

Isso também funciona se você quiser verificar se este método foi chamado mais de uma vez (neste caso, verificamos se o método bla foi chamado 23 vezes):

Mockito.verify(someMock, Mockito.times(23)).bla();

Estes são mais exemplos para o parâmetro VerificationMode, fornecendo mais controle sobre o número de vezes que um método deve ser chamado:

Mockito.verify(someMock, Mockito.never()).bla(); // same as Mockito.times(0)

Mockito.verify(someMock, Mockito.atLeast(3)).bla(); // min 3 calls

Mockito.verify(someMock, Mockito.atLeastOnce()).bla(); // same as Mockito.atLeast(1)

Mockito.verify(someMock, Mockito.atMost(3)).bla(); // max 3 calls

Teste de unidade simples usando Mockito #

A classe que vamos testar é:

public class Service {

    private Collaborator collaborator;

    public Service(Collaborator collaborator) {
        this.collaborator = collaborator;
    }
    
    public String performService(String input) {
        return collaborator.transformString(input);
    }
}

Seu colaborador é:

public class Collaborator {

    public String transformString(String input) {
        return doStuff();
    }

    private String doStuff() {
        // This method may be full of bugs
        . . .
        return someString;
    }

}

Em nosso teste, queremos quebrar a dependência do Collaborator e seus bugs, então vamos simular o Collaborator:

import static org.junit.Assert.*;
import static org.mockito.Mockito.*;

import org.junit.Test;

public class ServiceTest {
    @Test
    public void testPerformService() throws Exception {
        // Configure mock
        Collaborator collaboratorMock = mock(Collaborator.class);
        doReturn("output").when(collaboratorMock).transformString("input");

        // Perform the test
        Service service = new Service(collaboratorMock);
        String actual = service.performService("input");
        
        // Junit asserts
        String expected = "output";
        assertEquals(expected, actual);
    }  
}

Usando anotações do Mockito #

A classe que vamos testar é:

public class Service{

    private Collaborator collaborator;

    public Service(Collaborator collaborator){
        this.collaborator = collaborator;
    }
    
    
    public String performService(String input){
        return collaborator.transformString(input);
    }
}

Seu colaborador é:

public class Collaborator {

    public String transformString(String input){
        return doStuff();
    }

    private String doStuff()
    {
        // This method may be full of bugs
        . . .
        return someString;
    }

}

Em nosso teste, queremos quebrar a dependência do Collaborator e seus bugs, então vamos simular o Collaborator. Usar a anotação @Mock é uma maneira conveniente de criar diferentes instâncias de mocks para cada teste:

import static org.junit.Assert.*;
import static org.mockito.Mockito.*;

import org.junit.Test;
import org.junit.runner.RunWith;
import org.mockito.Mock;
import org.mockito.InjectMocks;
import org.mockito.runners.MockitoJUnitRunner;

@RunWith(MockitoJUnitRunner.class)
public class ServiceTest {

    @Mock
    private Collaborator collaboratorMock;

    @InjectMocks
    private Service service;
    
    @Test
    public void testPerformService() throws Exception {
        // Configure mock
        doReturn("output").when(collaboratorMock).transformString("input");            

        // Perform the test
        String actual = service.performService("input");
        
        // Junit asserts
        String expected = "output";
        assertEquals(expected, actual);
    }
    
    
    @Test(expected=Exception.class)
    public void testPerformServiceShouldFail() throws Exception {
        // Configure mock
        doThrow(new Exception()).when(collaboratorMock).transformString("input");

        // Perform the test
        service.performService("input");
    }
}

O Mockito tentará resolver a injeção de dependência na seguinte ordem:

  1. Injeção baseada em construtor - mocks são injetados no construtor com a maioria dos argumentos (se alguns argumentos não puderem ser encontrados, então os nulos serão passados). Se um objeto foi criado com sucesso via construtor, nenhuma outra estratégia será aplicada.
  2. Injeção baseada em setter - os mocks são injetados por tipo. Se houver várias propriedades do mesmo tipo, os nomes das propriedades e os nomes fictícios serão correspondidos.
  3. Injeção de campo direta - igual à injeção baseada em setter.

Observe que nenhuma falha é relatada caso alguma das estratégias acima mencionadas tenha falhado.

Consulte o @InjectMocks mais recente para obter informações mais detalhadas sobre esse mecanismo na versão mais recente do Mockito.

Instalação e configuração #

Instalação #

A maneira preferida de instalar o Mockito é declarar uma dependência do mockito-core com um sistema de compilação de sua escolha. A partir de 22 de julho de 2016, a versão não beta mais recente é 1.10.19, mas 2.x já é incentivado a ser migrado para.

Maven

<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <version>1.10.19</version>
    <scope>test</scope>
</dependency>

Gradle

repositories { jcenter() }
dependencies { testCompile "org.mockito:mockito-core:1.+" }

Há também o mockito-all que contém Hamcrest e Objenesis além do próprio Mockito. Ele é entregue através do Maven principalmente para usuários de formigas, mas a distribuição foi descontinuada no Mockito 2.x.

Importar #

A maioria das facilidades do Mockito são métodos estáticos de org.mockito.Mockito. Assim, Mockito pode ser importado estaticamente para uma classe desta forma:

import static org.mockito.Mockito.*;

O ponto de entrada da documentação está localizado no javadoc desta classe.

Zomba de alguns métodos em um objeto #

Teste Mockito mínimo simples #

Este exemplo mostra um teste mínimo do Mockito usando um ArrayList simulado:

import static org.mockito.Mockito.*;
import static org.junit.Assert.*;

import java.util.ArrayList;

import org.junit.Test;
import org.junit.runner.RunWith;
import org.mockito.Mock;
import org.mockito.runners.MockitoJUnitRunner;

@RunWith(MockitoJUnitRunner.class)
public class MockitoTest
{
    @Mock
    ArrayList<String> listMock;

    @Test
    public void testAppend() {
        // configure the mock to return "foobar" whenever "get()" 
        // is called on "listMock" with an int value as parameter
        doReturn("foobar").when(listMock).get(anyInt());            
        String result = listMock.get(0);
        
        assertEquals("foobar", result);
    }
}

Verificando argumentos com ArgumentCaptor #

Para validar argumentos para métodos chamados em uma simulação, use a classe ArgumentCaptor. Isso permitirá que você extraia os argumentos em seu método de teste e execute asserções neles.

Este exemplo testa um método que atualiza o nome de um usuário com um determinado ID. O método carrega o usuário, atualiza o atributo name com o valor fornecido e o salva posteriormente. O teste quer verificar se o argumento passado para o método save é um objeto User com o ID e o nome corretos.

// This is mocked in the test
interface UserDao {
    void save(User user);
}

@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    UserDao userDao;

    @Test
    public void testSetNameForUser() {
        UserService serviceUnderTest = new UserService(userDao);
        
        serviceUnderTest.setNameForUser(1L, "John");

        ArgumentCaptor<User> userArgumentCaptor = ArgumentCaptor.forClass(User.class);
        
        verify(userDao).save(userArgumentCaptor.capture());
        User savedUser = userArgumentCaptor.getValue();
        assertTrue(savedUser.getId() == 1);
        assertTrue(savedUser.getName().equals("John"));
    }
}

Verificando argumentos com ArgumentMatcher #

Mockito fornece uma interface Matcher<T> junto com uma classe abstrata ArgumentMatcher<T> para verificar argumentos. Ele usa uma abordagem diferente para o mesmo caso de uso do ArgumentCaptor. Além disso, o ArgumentMatcher também pode ser usado para zombar. Ambos os casos de uso fazem uso do método Mockito.argThat() que fornece um código de teste razoavelmente legível.

verify(someMock).someMethod(Mockito.argThat(new ArgumentMatcher<String>() {
       
    @Override
    public boolean matches(Object o) {
        return o instanceof String && !((String)o).isEmpty();
    }

});

Dos JavaDocs do ArgumentMatcher:

Aviso: seja razoável ao usar correspondências de argumentos complicadas, especialmente correspondências de argumentos personalizadas, pois isso pode tornar o teste menos legível. Às vezes é melhor implementar equals() para argumentos que são passados ​​para mocks (Mockito naturalmente usa equals() para correspondência de argumentos). Isso pode tornar o teste mais limpo.
