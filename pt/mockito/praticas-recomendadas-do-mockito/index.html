Estilo BDMockito #

O estilo de teste do Behavior Driven Development (BDD) gira em torno dos estágios “dado”, “quando” e “então” nos testes. No entanto, o Mockito clássico usa a palavra “quando” para a fase “dada” e não inclui outras construções de linguagem natural que possam abranger o BDD. Assim, os aliases BDDMockito foram introduzidos na versão 1.8.0 para facilitar os testes orientados ao comportamento.

A situação mais comum é stub retornos de um método. No exemplo a seguir, o método getStudent(String) do StudentRepository simulado retornará new Student(givenName, givenScore) se invocado com um argumento igual a givenName.

import static org.mockito.BDDMockito.*;

public class ScoreServiceTest {

    private StudentRepository studentRepository = mock(StudentRepository.class);

    private ScoreService objectUnderTest = new ScoreService(studentRepository);

    @Test
    public void shouldCalculateAndReturnScore() throws Exception {
        //given
        String givenName = "Johnny";
        int givenScore = 10;
        given(studentRepository.getStudent(givenName))
            .willReturn(new Student(givenName, givenScore));

        //when
        String actualScore = objectUnderTest.calculateStudentScore(givenName);

        //then
        assertEquals(givenScore, actualScore);
    }
}

Às vezes, é desejável verificar se a exceção lançada da dependência é tratada corretamente ou relançada em um método em teste. Tal comportamento pode ser stub na fase “dada” desta forma:

willThrow(new RuntimeException())).given(mock).getData();

Às vezes, é desejável configurar alguns efeitos colaterais que um método stub deve apresentar. Especialmente pode ser útil quando:

  • o método stubbed é um método que deve alterar o estado interno de um objeto passado

  • o método stubbed é um método void

Tal comportamento pode ser interrompido na fase “dada” com uma “Resposta”:

willAnswer(invocation -> this.prepareData(invocation.getArguments()[0])).given(mock).processData();

Quando se deseja verificar as interações com uma simulação, isso pode ser feito na fase “then” com os métodos should() ou should(VerificationMode)(somente desde 1.10.5):

then(mock).should().getData(); // verifies that getData() was called once
then(mock).should(times(2)).processData(); // verifies that processData() was called twice

Quando se deseja verificar que não houve mais interações com um simulado além do já verificado, isso pode ser feito na fase “then” com shouldHaveNoMoreInteractions() (desde 2.0.0):

then(mock).shouldHaveNoMoreInteractions(); // analogue of verifyNoMoreInteractions(mock) in classical Mockito

Quando se deseja verificar que não houve absolutamente nenhuma interação com um mock, isso pode ser feito na fase “then” com shouldHaveNoMoreInteractions() (desde 2.0.0):

then(mock).shouldHaveZeroInteractions(); // analogue of verifyZeroInteractions(mock) in classical Mockito

Quando se deseja verificar se os métodos foram invocados em ordem isso pode ser feito na fase “then” com should(InOrder) (desde 1.10.5) e should(InOrder, VerificationMode) (desde 2.0.0):

InOrder inOrder = inOrder(mock);

// test body here

then(mock).should(inOrder).getData(); // the first invocation on the mock should be getData() invocation
then(mock).should(inOrder, times(2)).processData(); // the second and third invocations on the mock should be processData() invocations
