Sintaxe #

  • template –– é o nome do tema principal do WordPress, o pai.
  • child-theme –– é o pacote que substitui o template.

Eu tenho anunciado que o uso de um tema infantil é sempre uma coisa boa, mas sempre há um Mas …

Em nosso exemplo de substituição de modelo, vamos imaginar que o autor de um tema está adicionando suas próprias melhorias ao modelo da barra lateral e haverá um novo em

/themes/template/sidebar.php

<?php
/**
 * The template for the sidebar containing the main widget area
 *
 * @link https://developer.wordpress.org/themes/basics/template-files/#template-partials
 */

if ( is_active_sidebar( 'sidebar-1' )  ) : ?>
    <aside id="secondary" class="sidebar widget-area" role="complementary">
        <?php dynamic_sidebar( 'sidebar-1' ); ?>
    </aside><!-- .sidebar .widget-area -->
<?php endif; ?>

Agora nosso site não se beneficiará da nova especificação role="complementary" porque nosso tema filho ainda está substituindo o modelo por seu próprio arquivo em /themes/child-theme/sidebar.php

É nosso dever como mantenedores do site manter um controle sobre quais templates nós sobrescrevemos e, no caso iminente de uma atualização, olhar atentamente para o changelog para que você atualize os arquivos do tema filho, se necessário.

2) O propósito #

Os temas filhos são uma maneira segura de manter as personalizações do modelo principal sem temer perdê-las em uma atualização de tema.

Basicamente, sempre que você quiser editar um arquivo dentro do template ativo do seu site você deve se perguntar “O que vai acontecer quando eu atualizar o tema?

E a resposta é simples: Você os perde porque a atualização trará uma pasta de temas totalmente nova.

Então vamos olhar para um tema filho como uma pasta com arquivos que irão sobrescrever os arquivos com o mesmo caminho no tema pai. Agora vamos trazer alguns exemplos reais:

Definição e requisitos #

Um tema filho é identificado pelo WordPress quando há um diretório (por exemplo child-theme) dentro de /wp-content/themes/ com os seguintes arquivos:

  • estilo.css

    This file must start with a comment template like this:

      /*
  Theme Name: Example Child
  Author: Me
  Author URI: https://example.com/
  Template: example
  Text Domain: example-child-theme
  Domain Path: /languages/
  */

    The most important things to consider here are:

    - The `Template` name must be exactly the folder name which holds the parent theme (aka the parent theme slug)

- Name your child theme in such a way you can easily identify it in Dashboard (usually just append `Child` to the parent's name, like `Example Child`)

  • index.php

  • functions.php

3) Sobrescrevendo o modelo #

O uso mais comum de um tema filho é substituir partes do modelo. Por exemplo, uma barra lateral, se tivermos um tema com o seguinte arquivo em

/themes/template/sidebar.php

<?php
/**
 * The sidebar containing the main widget area.
 *
 * @link https://developer.wordpress.org/themes/basics/template-files/#template-partials
 */ 

if ( ! is_active_sidebar( 'sidebar-1' ) ) {
    return;
}?>
<div id="sidebar" class="widget-area">
    <?php dynamic_sidebar( 'sidebar-1' ); ?>
</div>

Podemos definitivamente adicionar nosso próprio arquivo no tema filho (com as especificações do primeiro exemplo) com o seguinte arquivo

/themes/child-theme/sidebar.php

<?php
/**
 * The sidebar containing the main widget area.
 */ 

if ( ! is_active_sidebar( 'sidebar-1' ) ) {
    return;
}?>
<div id="my-sidebar" class="my-own-awesome-class widget-area">
    <div class="my-wrapper">
        <?php dynamic_sidebar( 'sidebar-1' ); ?>
    </div>
</div>

Agora my-own-awesome-class é seguro em tema filho e não será removido em nenhuma atualização de tema e o WordPress sempre escolherá um modelo de temas filhos quando encontrar um no mesmo caminho.

Substituição de ativos #

Mesmo que não seja uma prática recomendada, às vezes você precisa substituir ativos como arquivos ou bibliotecas CSS ou JS.

Observe que o sistema de substituição de templates do WordPress não funciona com nada além de arquivos .php, então, quando falamos sobre ativos, nos referimos a ativos registrados

Um exemplo pode ser a substituição da biblioteca jQuery pela versão desejada. Em nosso arquivo de tema filho functions.php precisamos adicionar uma função que remova a versão atual do jQuery e adicione a nossa própria versão do CDN (lembre-se, é apenas um exemplo).

/**
 * Dequeue the jQuery script and add our own version.
 *
 * Hooked to the wp_print_scripts action, with a late priority (100),
 * so that it is after the script was enqueued.
 */
function my_own_theme_scripts() {
    // remove the current version
    wp_dequeue_script( 'jquery' );
    // register my desired version
    wp_register_script( 'jquery', 'https://code.jquery.com/jquery-3.1.0.min.js', false, '3.1.0' );
    // load my version, here or somewhere else
    wp_enqueue_script( 'jquery' );
}
add_action( 'wp_print_scripts', 'my_own_theme_scripts' );
