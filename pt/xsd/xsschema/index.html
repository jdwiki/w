Descreve elementos, atributos e tipos que são válidos em um documento de instância XML. Um XML Schema (XSD) deve conter um único elemento xs:schema de nível raiz.

Parâmetros #

Atributos Descrição
attributeFormDefault Indica se os atributos no documento de instância XML devem ser qualificados com um namespace (padrão não qualificado)
blockDefault O valor padrão do atributo block que é aplicado a xs:complexType e xs:element. Define as regras para bloquear derivação/substituição no documento de instância (padrão vazio, ou seja, não bloqueia nada)
defaultAttributes (XSD 1.1) Especifica um xs:attributeGroup que será associado a todos os xs:complexType e xs:element dentro do esquema (opcional).
elementFormDefault Indica se o elemento no documento de instância XML deve ser qualificado com um namespace (padrão não qualificado). Nota : quase sem exceção, todos os esquemas definem isso como ‘qualified’.
finalPadrão O valor de atributo final padrão usado em xs:complexType e xs:element. Define as regras para bloquear derivação/substituição no esquema (padrão vazio, ou seja, não bloqueia nada)
identificação O id do item de esquema (opcional)
targetNamespace Qualifica todos os elementos e atributos (e componentes definidos globalmente) definidos neste esquema.
versão A versão do esquema, esta é a versão do documento, não esta versão XSD (ou seja, Soap 1.2, FpML 4.2 etc)
xpathDefaultNamespace (XSD 1.1) O valor padrão para o atributo xpathDefaultNamespace que é usado em xs:selector, xs:field, xs:alternative & xs:assert (especifica o namespace padrão a ser usado em expressões XPath)
qualquer Quaisquer outros atributos que não estejam no namespace ‘http://www.w3.org/2001/XMLSchema’ são permitidos.
Elementos Descrição
xs:anotação Fornece a capacidade de adicionar documentação e dados legíveis por máquina.
xs:incluir Usado para incluir um esquema com o mesmo targetNamespace ou sem targetNamespace (consulte esquemas camaleão).
xs:importar Usado para incluir um esquema com um targetNamespace diferente do pai.
xs:redefinir Usado para incluir um esquema com o mesmo targetNamespace (ou nenhum targetNamespace) e modificar as definições de xs:simpleType, xs:complexType, xs:group ou xs:attributeGroup contidas nele (aqui estão dragões…)
xs:simplesType Define um tipo simples global (nomeado) que pode ser referenciado e reutilizado.
xs:complexType Define um tipo complexo global (nomeado) que pode ser referenciado e reutilizado.
xs:grupo Define um grupo global (nomeado) de elementos que podem ser referenciados e reutilizados.
xs:attributeGroup Define um grupo global (nomeado) de atributos que podem ser referenciados e reutilizados.
xs:atributo Define um atributo global (nomeado) que pode ser referenciado e reutilizado.
xs:elemento Define um elemento global (nomeado) que pode ser referenciado e reutilizado ou usado como base de um documento de instância XML.
xs:notação -
xs:defaultOpenContent (XSD 1.1) Especifica regras para permitir que elementos adicionais sejam permitidos em cada xs:complexType e xs:element dentro do esquema.

Básico xs:schema #

Mostra um esquema muito básico.

Observação: por convenção elementFormDefault é definido como ‘qualified’, no mundo real você terá dificuldade em encontrar um esquema que não defina isso (então apenas inclua-o em seus esquemas!).

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 - (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="qualified" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="Person">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="Forename" type="xs:string" />
                <xs:element name="Surname" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

xs:schema elementFormDefault atributo #

Por convenção elementFormDefault é sempre definido como qualified, mas vamos ver o que ele realmente faz.

Primeiro com elementFormDefault definido como qualificado.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="qualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

Exemplo de documento XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_qualified.xsd">
    <b:ChildA>string</b:ChildA>
</b:MyBaseElement>

Observe que o elemento ChildA também deve ser qualificado com o namespace ‘b’.

Agora vamos olhar para ele com elementFormDefault definido como não qualificado.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="unqualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>

Exemplo de documento XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_unqualified.xsd">
    <ChildA>string</ChildA>
</b:MyBaseElement>

Observe desta vez que apenas o elemento definido globalmente MyBaseElement é qualificado com o namespace ‘b’, o elemento interno ChildA (que é definido no local dentro do esquema) não é qualificado.

No último exemplo, vimos que os elementos definidos globalmente devem ser qualificados no documento de instância XML, mas os elementos definidos no local não. Mas isso não significa apenas o elemento raiz, se você tiver elementos definidos globalmente que são referenciados, eles também precisam ser qualificados.

<?xml version="1.0" encoding="utf-8" ?>
<!--Created with Liquid Studio 2017 (https://www.liquid-technologies.com)-->
<xs:schema elementFormDefault="unqualified" 
           targetNamespace="http://base.com" 
           xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="MyBaseElement">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="ChildA" type="xs:string" />
                <xs:element xmlns:q1="http://base.com" ref="q1:MyElement" />
            </xs:sequence>
        </xs:complexType>
    </xs:element>
    <xs:element name="MyElement" type="xs:string" />
</xs:schema>

Exemplo de documento XML

<?xml version="1.0" encoding="utf-8"?>
<!-- Created with Liquid Studio 2017 (https://www.liquid-technologies.com) -->
<b:MyBaseElement xmlns:b="http://base.com" 
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://base.com ElementFormDefault_unqualified.xsd">
    <ChildA>string</ChildA>
    <b:MyElement>string</b:MyElement>
</b:MyBaseElement>

Observe que MyElement também precisa de qualificação, pois é definido globalmente.

Em conclusão, se você tiver elementFormDefault definido como qualificado, tudo precisa ser qualificado com um namespace (por meio de um alias de namespace ou definindo o namespace padrão xmlns="…"). No entanto elementFormDefault está definido como não qualificado, as coisas ficam complicadas e você precisa fazer um exame bastante aprofundado dos esquemas para descobrir se as coisas devem ser qualificadas ou não.

Suponho que é por isso que elementFormDefault está sempre definido como qualificado!
