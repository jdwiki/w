Python é uma linguagem destinada a ser clara e legível, sem ambiguidades e comportamentos inesperados. Infelizmente, esses objetivos não são alcançáveis ​​em todos os casos, e é por isso que o Python tem alguns casos de canto onde pode fazer algo diferente do que você esperava.

Esta seção mostrará alguns problemas que você pode encontrar ao escrever código Python.

Multiplicação de listas e referências comuns #

Considere o caso de criar uma estrutura de lista aninhada multiplicando:

li = [[]] * 3
print(li)
# Out: [[], [], []]

À primeira vista, pensaríamos que temos uma lista contendo 3 listas aninhadas diferentes. Vamos tentar anexar 1 ao primeiro:

li[0].append(1)
print(li)
# Out: [[1], [1], [1]]

1 foi anexado a todas as listas em li.

A razão é que [[]] * 3 não cria uma lista de 3 listas diferentes. Em vez disso, ele cria uma list contendo 3 referências para o mesmo objeto list. Como tal, quando anexamos a li[0] a mudança é visível em todos os subelementos de li. Isso é equivalente a:

li = []
element = [[]]
li = element + element + element
print(li)
# Out: [[], [], []]
element.append(1)
print(li)
# Out: [[1], [1], [1]]

Isso pode ser corroborado ainda mais se imprimirmos os endereços de memória da list contida usando id:

li = [[]] * 3
print([id(inner_list) for inner_list in li])
# Out: [6830760, 6830760, 6830760]

A solução é criar as listas internas com um loop:

li = [[] for _ in range(3)]

Em vez de criar uma única lista e depois fazer 3 referências a ela, agora criamos 3 listas distintas diferentes. Isso, novamente, pode ser verificado usando a função id:

print([id(inner_list) for inner_list in li])
# Out: [6331048, 6331528, 6331488]

Você também pode fazer isso. Faz com que uma nova lista vazia seja criada em cada chamada append.

>>> li = []
>>> li.append([])
>>> li.append([])
>>> li.append([])
>>> for k in li: print(id(k))
... 
4315469256
4315564552
4315564808

Não use índice para fazer um loop em uma sequência.

Não:

for i in range(len(tab)):
    print(tab[i])

Fazer:

for elem in tab:
    print(elem)

for automatizará a maioria das operações de iteração para você.

Use enumerate se você realmente precisa do índice e do elemento.

for i, elem in enumerate(tab):
     print((i, elem))

Tenha cuidado ao usar “==” para verificar se é Verdadeiro ou Falso

if (var == True):
    # this will execute if var is True or 1, 1.0, 1L

if (var != True):
    # this will execute if var is neither True nor 1

if (var == False):
    # this will execute if var is False or 0 (or 0.0, 0L, 0j)

if (var == None):
    # only execute if var is None

if var:
    # execute if var is a non-empty string/list/dictionary/tuple, non-0, etc

if not var:
    # execute if var is "", {}, [], (), 0, None, etc.

if var is True:
    # only execute if var is boolean True, not 1

if var is False:
    # only execute if var is boolean False, not 0

if var is None:
    # same as var == None

Não verifique se você pode, apenas faça e lide com o erro

Pythonistas costumam dizer “É mais fácil pedir perdão do que permissão”.

Não:

if os.path.isfile(file_path):
    file = open(file_path)
else:
    # do something

Fazer:

try:
    file = open(file_path)
except OSError as e:
    # do something

Ou ainda melhor com Python 2.6+:

with open(file_path) as file:

É muito melhor porque é muito mais genérico. Você pode aplicar try/except a quase tudo. Você não precisa se preocupar com o que fazer para evitá-lo, apenas se preocupe com o erro que está arriscando.

Não verifique contra o tipo

O Python é tipado dinamicamente, portanto, verificar o tipo faz com que você perca flexibilidade. Em vez disso, use duck type verificando o comportamento. Se você espera uma string em uma função, use str() para converter qualquer objeto em uma string. Se você espera uma lista, use list() para converter qualquer iterável em uma lista.

Não:

def foo(name):
    if isinstance(name, str):
        print(name.lower())

def bar(listing):
    if isinstance(listing, list):
        listing.extend((1, 2, 3))
        return ", ".join(listing)

Fazer:

def foo(name) :
    print(str(name).lower())

def bar(listing) :
    l = list(listing)
    l.extend((1, 2, 3))
    return ", ".join(l)

Usando a última maneira, foo aceitará qualquer objeto. bar aceitará strings, tuplas, conjuntos, listas e muito mais. SECO barato.

Não misture espaços e tabulações

Use object como primeiro pai

Isso é complicado, mas vai mordê-lo à medida que seu programa cresce. Existem classes antigas e novas no Python 2.x. Os antigos são, bem, velhos. Eles carecem de alguns recursos e podem ter um comportamento estranho com herança. Para ser utilizável, qualquer de sua classe deve ser do “novo estilo”. Para fazer isso, faça-o herdar de object.

Não:

class Father:
    pass

class Child(Father):
    pass

Fazer:

class Father(object):
    pass


class Child(Father):
    pass

No Python 3.x todas as classes são de novo estilo, então você não precisa fazer isso.

Não inicialize atributos de classe fora do método init

As pessoas que vêm de outras linguagens acham tentador porque é isso que você faz em Java ou PHP. Você escreve o nome da classe, então lista seus atributos e dá a eles um valor padrão. Parece funcionar em Python, no entanto, isso não funciona do jeito que você pensa. Fazer isso configurará os atributos de classe (atributos estáticos), então quando você tentar obter o atributo do objeto, ele fornecerá seu valor, a menos que esteja vazio. Nesse caso, ele retornará os atributos de classe. Implica dois grandes perigos:

  • Se o atributo de classe for alterado, o valor inicial será alterado.

  • Se você definir um objeto mutável como valor padrão, obterá o mesmo objeto compartilhado entre instâncias.

Não (a menos que você queira estático):

class Car(object):
    color = "red"
    wheels = [Wheel(), Wheel(), Wheel(), Wheel()]

Fazer :

class Car(object):
    def __init__(self):
        self.color = "red"
        self.wheels = [Wheel(), Wheel(), Wheel(), Wheel()]

Argumento padrão mutável #

def foo(li=[]):
    li.append(1)
    print(li)

foo([2])
# Out: [2, 1]
foo([3])
# Out: [3, 1]

Este código se comporta como esperado, mas e se não passarmos um argumento?

foo()
# Out: [1] As expected...

foo()
# Out: [1, 1]  Not as expected...

Isso ocorre porque os argumentos padrão de funções e métodos são avaliados em tempo de definição em vez de em tempo de execução. Portanto, só temos uma única instância da lista li.

A maneira de contornar isso é usar apenas tipos imutáveis ​​para argumentos padrão:

def foo(li=None):
    if not li:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

foo()
# Out: [1]

Embora seja uma melhoria e embora if not li seja avaliado corretamente como False, muitos outros objetos também o fazem, como sequências de comprimento zero. Os argumentos de exemplo a seguir podem causar resultados indesejados:

x = []
foo(li=x)
# Out: [1]

foo(li="")
# Out: [1]

foo(li=0) 
# Out: [1]

A abordagem idiomática é verificar diretamente o argumento contra o objeto None:

def foo(li=None):
    if li is None:
        li = []
    li.append(1)
    print(li)

foo()
# Out: [1]

Alterando a sequência sobre a qual você está iterando #

Um loop for itera em uma sequência, portanto, alterar essa sequência dentro do loop pode levar a resultados inesperados (especialmente ao adicionar ou remover elementos):

alist = [0, 1, 2]
for index, value in enumerate(alist):
    alist.pop(index)
print(alist)
# Out: [1]

Nota: list.pop() está sendo usado para remover elementos da lista.

O segundo elemento não foi excluído porque a iteração passa pelos índices em ordem. O loop acima itera duas vezes, com os seguintes resultados:

# Iteration #1
index = 0
alist = [0, 1, 2]
alist.pop(0) # removes '0'

# Iteration #2
index = 1
alist = [1, 2]
alist.pop(1) # removes '2'

# loop terminates, but alist is not empty:
alist = [1]

Esse problema surge porque os índices estão mudando enquanto iteram na direção do aumento do índice. Para evitar esse problema, você pode iterar o loop para trás:

alist = [1,2,3,4,5,6,7]
for index, item in reversed(list(enumerate(alist))):
    # delete all even items
    if item % 2 == 0:
        alist.pop(index)
print(alist)
# Out: [1, 3, 5, 7]

A iteração pelo loop começando no final, à medida que os itens são removidos (ou adicionados), não afeta os índices dos itens anteriores na lista. Portanto, este exemplo removerá corretamente todos os itens que são pares de alist.

Um problema semelhante surge ao inserir ou anexar elementos a uma lista sobre a qual você está iterando, o que pode resultar em um loop infinito:

alist = [0, 1, 2]
for index, value in enumerate(alist):
    # break to avoid infinite loop:
    if index == 20:     
        break           
    alist.insert(index, 'a')
print(alist)
# Out (abbreviated): ['a', 'a', ..., 'a', 'a',  0,   1,   2]

Sem a condição break o loop inseriria 'a' contanto que o computador não fique sem memória e o programa tenha permissão para continuar. Em uma situação como essa, geralmente é preferível criar uma nova lista e adicionar itens à nova lista à medida que você percorre a lista original.

Ao usar um loop for, você não pode modificar os elementos da lista com a variável placeholder:

alist = [1,2,3,4]
for item in alist:
    if item % 2 == 0:
        item = 'even'
print(alist)
# Out: [1,2,3,4]

No exemplo acima, alterar o item não altera nada na lista original. Você precisa usar o índice de lista (alist[2]), e enumerate() funciona bem para isso:

alist = [1,2,3,4]
for index, item in enumerate(alist):
    if item % 2 == 0:
        alist[index] = 'even'
print(alist)
# Out: [1, 'even', 3, 'even']

Um loop while pode ser uma escolha melhor em alguns casos:

Se você for excluir todos os itens da lista:

zlist = [0, 1, 2]
while zlist:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
#      2
# After: zlist = []

Embora simplesmente redefinir zlist obtenha o mesmo resultado;

zlist = []

O exemplo acima também pode ser combinado com len() para parar após um certo ponto, ou para excluir todos os itens da lista, exceto x:

zlist = [0, 1, 2]
x = 1
while len(zlist) > x:
    print(zlist[0])
    zlist.pop(0)
print('After: zlist =', zlist)

# Out: 0
#      1
# After: zlist = [2]

Ou percorrer uma lista enquanto exclui elementos que atendem a uma determinada condição (neste caso, excluindo todos os elementos pares):

zlist = [1,2,3,4,5]
i = 0
while i < len(zlist):
    if zlist[i] % 2 == 0:
        zlist.pop(i)
    else:
        i += 1
print(zlist)
# Out: [1, 3, 5]

Observe que você não incrementa i após excluir um elemento. Ao excluir o elemento em zlist[i], o índice do próximo item diminuiu em um, portanto, marcando zlist[i] com o mesmo valor para i na próxima iteração, você estará verificando corretamente o próximo item da lista.

Uma maneira contrária de pensar em remover itens indesejados de uma lista é adicionar itens desejados a uma nova lista. O exemplo a seguir é uma alternativa ao último exemplo de loop while:

zlist = [1,2,3,4,5]

z_temp = []
for item in zlist:
    if item % 2 != 0:
        z_temp.append(item)
zlist = z_temp
print(zlist)
# Out: [1, 3, 5]

Aqui estamos canalizando os resultados desejados em uma nova lista. Podemos então, opcionalmente, reatribuir a lista temporária à variável original.

Com essa tendência de pensamento, você pode invocar um dos recursos mais elegantes e poderosos do Python, compreensão de listas, que elimina listas temporárias e diverge da ideologia de mutação de lista/índice no local discutida anteriormente.

zlist = [1,2,3,4,5]
[item for item in zlist if item % 2 != 0]
# Out: [1, 3, 5]

Identidade de inteiro e string #

Python usa cache interno para um intervalo de números inteiros para reduzir a sobrecarga desnecessária de sua criação repetida.

Na verdade, isso pode levar a um comportamento confuso ao comparar identidades inteiras:

>>> -8 is (-7 - 1)
False
>>> -3 is (-2 - 1)
True

e, usando outro exemplo:

>>> (255 + 1) is (255 + 1)
True
>>> (256 + 1) is (256 + 1)
False

Espere o que?

Podemos ver que a operação de identidade is produz True para alguns inteiros (-3, 256), mas não para outros (-8, 257).

Para ser mais específico, os inteiros no intervalo [-5, 256] são armazenados em cache internamente durante a inicialização do interpretador e são criados apenas uma vez. Como tal, eles são idênticos e comparar suas identidades com is resulta em True; inteiros fora desse intervalo são (geralmente) criados em tempo real e suas identidades se comparam a False.

Esta é uma armadilha comum, uma vez que este é um intervalo comum para testes, mas com bastante frequência, o código falha no processo de teste posterior (ou pior - produção) sem motivo aparente depois de funcionar perfeitamente no desenvolvimento.

A solução é sempre comparar valores usando o operador de igualdade (==) e não o operador de identidade (is).

O Python também mantém referências a strings comumente usadas e pode resultar em um comportamento similarmente confuso ao comparar identidades (ou seja, usando is) de strings.

>>> 'python' is 'py' + 'thon'
True

A string 'python' é comumente usada, então Python tem um objeto que todas as referências à string 'python' usam.

Para strings incomuns, a comparação de identidade falha mesmo quando as strings são iguais.

>>> 'this is not a common string' is 'this is not' + ' a common string'
False
>>> 'this is not a common string' == 'this is not' + ' a common string'
True

Assim, assim como a regra para Integers, sempre compare valores de string usando o operador de igualdade (==) e não o operador de identidade (is).

Os dicionários não estão ordenados #

Você pode esperar que um dicionário Python seja classificado por chaves como, por exemplo, um C++ std::map, mas esse não é o caso:

myDict = {'first': 1, 'second': 2, 'third': 3}
print(myDict)
# Out: {'first': 1, 'second': 2, 'third': 3}

print([k for k in myDict])
# Out: ['second', 'third', 'first']

O Python não possui nenhuma classe interna que classifica automaticamente seus elementos por chave.

No entanto, se a classificação não for obrigatória e você quiser apenas que seu dicionário lembre a ordem de inserção de seus pares chave/valor, você pode usar collections.OrderedDict:

from collections import OrderedDict

oDict = OrderedDict([('first', 1), ('second', 2), ('third', 3)])

print([k for k in oDict])
# Out: ['first', 'second', 'third']

Tenha em mente que inicializar um OrderedDict com um dicionário padrão não irá ordenar de forma alguma o dicionário para você. Tudo o que essa estrutura faz é preservar a ordem de inserção da chave.

A implementação dos dicionários foi alterada no Python 3.6 para melhorar o consumo de memória. Um efeito colateral dessa nova implementação é que ela também preserva a ordem dos argumentos de palavras-chave passados ​​para uma função:

def func(**kw): print(kw.keys())

func(a=1, b=2, c=3, d=4, e=5) 
dict_keys(['a', 'b', 'c', 'd', 'e']) # expected order

Aviso: tenha em atenção que [“o aspecto de preservação da ordem desta nova implementação é considerado um detalhe de implementação e não deve ser invocado”](https://docs.python.org/3.6/whatsnew/ 3.6.html#new-dict-implementation), pois pode mudar no futuro.

Vazamento de variável em compreensões de lista e loops for #

Considere a seguinte compreensão de lista

i = 0
a = [i for i in range(3)]
print(i) # Outputs 2

Isso ocorre apenas no Python 2 devido ao fato de que a compreensão da lista “vaza” a variável de controle do loop no escopo circundante ([source](http://python-history.blogspot.com/2010/06/from-list- compreensões-para-gerador.html)). Esse comportamento pode levar a bugs difíceis de encontrar e foi corrigido no Python 3.

i = 0
a = [i for i in range(3)]
print(i) # Outputs 0

Da mesma forma, loops for não têm escopo privado para sua variável de iteração

i = 0
for i in range(3):
    pass
print(i) # Outputs 2

Esse tipo de comportamento ocorre tanto no Python 2 quanto no Python 3.

Para evitar problemas com vazamento de variáveis, use novas variáveis ​​em compreensões de lista e loops for conforme apropriado.

Encadeamento de ou operador #

Ao testar qualquer uma das várias comparações de igualdade:

if a == 3 or b == 3 or c == 3:

é tentador abreviar isso para

if a or b or c == 3: # Wrong

Isto está errado; o operador or tem precedência menor que ==, então a expressão será avaliada como if (a) ou (b) ou (c == 3):. A maneira correta é verificar explicitamente todas as condições:

if a == 3 or b == 3 or c == 3:  # Right Way

Alternativamente, a função interna any() pode ser usada no lugar de operadores or encadeados:

if any([a == 3, b == 3, c == 3]): # Right

Ou, para torná-lo mais eficiente:

if any(x == 3 for x in (a, b, c)): # Right

Ou, para encurtar:

if 3 in (a, b, c): # Right

Aqui, usamos o operador in para testar se o valor está presente em uma tupla contendo os valores com os quais queremos comparar.

Da mesma forma, é incorreto escrever

if a == 1 or 2 or 3:

que deve ser escrito como

if a in (1, 2, 3):

Acessando atributos de literais int #

Você pode ter ouvido que tudo em Python é um objeto, até mesmo literais. Isso significa, por exemplo, que 7 também é um objeto, o que significa que possui atributos. Por exemplo, um desses atributos é o bit_length. Ele retorna a quantidade de bits necessária para representar o valor que é chamado.

x = 7
x.bit_length()
# Out: 3

Vendo que o código acima funciona, você pode intuitivamente pensar que 7.bit_length() funcionaria também, apenas para descobrir que gera um SyntaxError. Por quê? porque o interpretador precisa diferenciar entre um acesso de atributo e um número flutuante (por exemplo 7.2 ou 7.bit_length()). Não pode, e é por isso que uma exceção é levantada.

Existem algumas maneiras de acessar os atributos de um int literals:

# parenthesis
(7).bit_length()
# a space
7 .bit_length()

Usar dois pontos (como este 7..bit_length()) não funciona neste caso, porque isso cria um literal float e os floats não possuem o método bit_length().

Este problema não existe ao acessar atributos de literais float já que o interpretador é “inteligente” o suficiente para saber que um literal float não pode conter dois ., por exemplo:

7.2.as_integer_ratio()
# Out: (8106479329266893, 1125899906842624)

Global Interpreter Lock (GIL) e threads de bloqueio #

Muito foi escrito sobre [Python’s](https://docs.python .org/3/glossary.html#term-global-interpreter-lock) GIL. Às vezes, pode causar confusão ao lidar com aplicativos multithread (não confundir com multiprocessos).

Aqui está um exemplo:

import math
from threading import Thread

def calc_fact(num):
    math.factorial(num)

num = 600000
t = Thread(target=calc_fact, daemon=True, args=[num])
print("About to calculate: {}!".format(num))
t.start()
print("Calculating...")
t.join()
print("Calculated")

Você esperaria ver Calculando... ​​impresso imediatamente após o início do encadeamento, afinal, queríamos que o cálculo acontecesse em um novo encadeamento! Mas, na verdade, você o vê impresso após a conclusão do cálculo. Isso porque a nova thread depende de uma função C (math.factorial) que irá bloquear o GIL enquanto ele é executado.

Existem algumas maneiras de contornar isso. A primeira é implementar sua função fatorial em Python nativo. Isso permitirá que o thread principal pegue o controle enquanto você estiver dentro do seu loop. A desvantagem é que essa solução será muito mais lenta, já que não estamos mais usando a função C.

def calc_fact(num):
    """ A slow version of factorial in native Python """
    res = 1
    while num >= 1:
        res = res * num
        num -= 1
    return res

Você também pode ‘dormir’ por um período de tempo antes de iniciar sua execução. Nota: isso não permitirá que seu programa interrompa a computação que está acontecendo dentro da função C, mas permitirá que seu thread principal continue após o spawn, que é o que você pode esperar.

def calc_fact(num):
    sleep(0.001)
    math.factorial(num)

Retorno múltiplo #

A função xyz retorna dois valores a e b:

def xyz():
  return a, b

A chamada de código xyz armazena o resultado em uma variável, assumindo que xyz retorna apenas um valor:

t = xyz()

O valor de t é na verdade uma tupla (a, b) então qualquer ação em t assumindo que não é uma tupla pode falhar profundamente no código com um erro inesperado sobre tuplas.

TypeError: tipo tupla não define … método

A correção seria fazer:

a, b = xyz()

Iniciantes terão problemas para encontrar o motivo desta mensagem apenas lendo a mensagem de erro da tupla!
