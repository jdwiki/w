Introdução às luminárias de teste #

Testes mais complicados às vezes precisam ter as coisas configuradas antes de você executar o código que deseja testar. É possível fazer isso na própria função de teste, mas você acaba com funções de teste grandes fazendo tanto que é difícil dizer onde a configuração termina e o teste começa. Você também pode obter muitos códigos de configuração duplicados entre suas várias funções de teste.

Nosso arquivo de código:

# projectroot/module/stuff.py
class Stuff(object):
    def prep(self):
        self.foo = 1
        self.bar = 2

Nosso arquivo de teste:

# projectroot/tests/test_stuff.py
import pytest
from module import stuff


def test_foo_updates():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Estes são exemplos bem simples, mas se nosso objeto Stuff precisasse de muito mais configuração, ficaria pesado. Vemos que há algum código duplicado entre nossos casos de teste, então vamos refatorar isso em uma função separada primeiro.

# projectroot/tests/test_stuff.py
import pytest
from module import stuff


def get_prepped_stuff():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    return my_stuff


def test_foo_updates():
    my_stuff = get_prepped_stuff()
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates():
    my_stuff = get_prepped_stuff()
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Isso parece melhor, mas ainda temos a chamada my_stuff = get_prepped_stuff() bagunçando nossas funções de teste.

py.test acessórios para o resgate! #

As luminárias são versões muito mais poderosas e flexíveis das funções de configuração de teste. Eles podem fazer muito mais do que estamos aproveitando aqui, mas vamos dar um passo de cada vez.

Primeiro mudamos get_prepped_stuff para um fixture chamado prepped_stuff. Você quer nomear seus fixtures com substantivos ao invés de verbos por causa de como os fixtures acabarão sendo usados ​​nas próprias funções de teste mais tarde. O @pytest.fixture indica que esta função específica deve ser tratada como um fixture ao invés de uma função regular.

@pytest.fixture
def prepped_stuff():
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    return my_stuff

Agora devemos atualizar as funções de teste para que elas usem o fixture. Isso é feito adicionando um parâmetro à sua definição que corresponda exatamente ao nome do fixture. Quando py.test é executado, ele executa o fixture antes de executar o teste, então passa o valor de retorno do fixture para a função de teste através desse parâmetro. (Observe que os fixtures não precisam retornar um valor; eles podem fazer outras coisas de configuração, como chamar um recurso externo, organizar coisas no sistema de arquivos, colocar valores em um banco de dados, o que quer que os testes precisem para configuração)

def test_foo_updates(prepped_stuff):
    my_stuff = prepped_stuff
    assert 1 == my_stuff.foo
    my_stuff.foo = 30000
    assert my_stuff.foo == 30000


def test_bar_updates(prepped_stuff):
    my_stuff = prepped_stuff
    assert 2 == my_stuff.bar
    my_stuff.bar = 42
    assert 42 == my_stuff.bar

Agora você pode ver por que o nomeamos com um substantivo. mas a linha my_stuff = prepped_stuff é praticamente inútil, então vamos usar prepped_stuff diretamente.

def test_foo_updates(prepped_stuff):
    assert 1 == prepped_stuff.foo
    prepped_stuff.foo = 30000
    assert prepped_stuff.foo == 30000


def test_bar_updates(prepped_stuff):
    assert 2 == prepped_stuff.bar
    prepped_stuff.bar = 42
    assert 42 == prepped_stuff.bar

Agora estamos usando acessórios! Podemos ir além alterando o escopo do fixture (para que ele seja executado apenas uma vez por módulo de teste ou sessão de execução do conjunto de testes em vez de uma vez por função de teste), construindo fixtures que usam outros fixtures, parametrizando o fixture (para que o fixture e todos testes usando esse equipamento são executados várias vezes, uma vez para cada parâmetro dado ao equipamento), equipamentos que lêem valores do módulo que os chama… como mencionado anteriormente, equipamentos têm muito mais poder e flexibilidade do que uma função de configuração normal.

Limpeza após a conclusão dos testes. #

Digamos que nosso código cresceu e nosso objeto Stuff agora precisa de uma limpeza especial.

# projectroot/module/stuff.py
class Stuff(object):
def prep(self):
    self.foo = 1
    self.bar = 2

def finish(self):
    self.foo = 0
    self.bar = 0

Poderíamos adicionar algum código para chamar a limpeza na parte inferior de cada função de teste, mas os fixtures fornecem uma maneira melhor de fazer isso. Se você adicionar uma função ao fixture e registrá-lo como um finalizador, o código na função do finalizador será chamado após o teste usando o fixture ser feito. Se o escopo do acessório for maior que uma única função (como módulo ou sessão), o finalizador será executado após a conclusão de todos os testes no escopo, portanto, após a execução do módulo ou no final de toda a sessão de execução do teste .

@pytest.fixture
def prepped_stuff(request):  # we need to pass in the request to use finalizers
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    def fin():  # finalizer function
        # do all the cleanup here
        my_stuff.finish()
    request.addfinalizer(fin)  # register fin() as a finalizer
    # you can do more setup here if you really want to
    return my_stuff

Usar a função finalizador dentro de uma função pode ser um pouco difícil de entender à primeira vista, especialmente quando você tem equipamentos mais complicados. Em vez disso, você pode usar um equipamento de rendimento para fazer a mesma coisa com um fluxo de execução mais legível para humanos. A única diferença real é que ao invés de usar return usamos um yield na parte do fixture onde a configuração é feita e o controle deve ir para uma função de teste, então adicionamos todo o código de limpeza após o yield. Também o decoramos como um yield_fixture para que py.test saiba como lidar com isso.

@pytest.yield_fixture
def prepped_stuff():  # it doesn't need request now!
    # do setup
    my_stuff = stuff.Stuff()
    my_stuff.prep()
    # setup is done, pass control to the test functions
    yield my_stuff
    # do cleanup 
    my_stuff.finish()

E isso conclui a introdução às luminárias de teste!

Para obter mais informações, consulte a documentação oficial do equipamento py.test e a [documentação oficial do equipamento de rendimento](http://doc.pytest. org/en/latest/yieldfixture.html)
