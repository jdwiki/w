Existem várias ferramentas de teste de unidade para Python. Este tópico de documentação descreve o módulo unittest básico. Outras ferramentas de teste incluem py.test e nosetests. Esta documentação python sobre testes compara várias dessas ferramentas sem entrar em detalhes.

Configuração de teste e desmontagem em um unittest.TestCase #

Às vezes, queremos preparar um contexto para cada teste a ser executado. O método setUp é executado antes de cada teste na classe. tearDown é executado no final de cada teste. Esses métodos são opcionais. Lembre-se que TestCases são frequentemente usados ​​em herança múltipla cooperativa, então você deve ter cuidado para sempre chamar super nesses métodos para que os métodos setUp e tearDown da classe base também sejam chamados. A implementação base de TestCase fornece métodos setUp e tearDown vazios para que possam ser chamados sem gerar exceções:

import unittest


class SomeTest(unittest.TestCase):
    def setUp(self):
        super(SomeTest, self).setUp()
        self.mock_data = [1,2,3,4,5]

    def test(self):
        self.assertEqual(len(self.mock_data), 5)

    def tearDown(self):
        super(SomeTest, self).tearDown()
        self.mock_data = []


if __name__ == '__main__':
    unittest.main()

Observe que em python2.7+, há também o método addCleanup que registra funções a serem chamadas após o teste é executado. Em contraste com tearDown que só é chamado se setUp for bem sucedido, funções registradas via addCleanup serão chamadas mesmo no caso de uma exceção não tratada em setUp. Como exemplo concreto, este método pode ser visto frequentemente removendo vários mocks que foram registrados durante a execução do teste:

import unittest
import some_module


class SomeOtherTest(unittest.TestCase):
    def setUp(self):
        super(SomeOtherTest, self).setUp()
        
        # Replace `some_module.method` with a `mock.Mock`
        my_patch = mock.patch.object(some_module, 'method')
        my_patch.start()

        # When the test finishes running, put the original method back.
        self.addCleanup(my_patch.stop)

Outro benefício de registrar limpezas dessa maneira é que ele permite que o programador coloque o código de limpeza próximo ao código de configuração e protege você no caso de um subclasser esquecer de chamar super em tearDown.

Afirmando exceções #

Você pode testar se uma função lança uma exceção com o unittest interno por meio de dois métodos diferentes.

Usando um gerenciador de contexto

def division_function(dividend, divisor):
    return dividend / divisor


class MyTestCase(unittest.TestCase):
    def test_using_context_manager(self):
        with self.assertRaises(ZeroDivisionError):
            x = division_function(1, 0)

Isso executará o código dentro do gerenciador de contexto e, se for bem-sucedido, falhará no teste porque a exceção não foi gerada. Se o código gerar uma exceção do tipo correto, o teste continuará.

Você também pode obter o conteúdo da exceção levantada se quiser executar declarações adicionais contra ela.

class MyTestCase(unittest.TestCase):
    def test_using_context_manager(self):
        with self.assertRaises(ZeroDivisionError) as ex:
            x = division_function(1, 0)

        self.assertEqual(ex.message, 'integer division or modulo by zero')

Fornecendo uma função que pode ser chamada

def division_function(dividend, divisor):
    """
    Dividing two numbers.

    :type dividend: int
    :type divisor: int

    :raises: ZeroDivisionError if divisor is zero (0).
    :rtype: int
    """
    return dividend / divisor


class MyTestCase(unittest.TestCase):
    def test_passing_function(self):
        self.assertRaises(ZeroDivisionError, division_function, 1, 0)

A exceção a ser verificada deve ser o primeiro parâmetro e uma função que pode ser chamada deve ser passada como o segundo parâmetro. Quaisquer outros parâmetros especificados serão passados ​​diretamente para a função que está sendo chamada, permitindo que você especifique os parâmetros que acionam a exceção.

Exceções de teste #

Os programas lançam erros quando, por exemplo, uma entrada errada é fornecida. Por causa disso, é preciso ter certeza de que um erro é lançado quando uma entrada errada real é fornecida. Por causa disso, precisamos verificar uma exceção exata, para este exemplo, usaremos a seguinte exceção:

class WrongInputException(Exception):
    pass

Essa exceção é gerada quando uma entrada errada é fornecida, no contexto a seguir, onde sempre esperamos um número como entrada de texto.

def convert2number(random_input):
    try:
        my_input = int(random_input)
    except ValueError:
        raise WrongInputException("Expected an integer!")
    return my_input

Para verificar se uma exceção foi levantada, usamos assertRaises para verificar essa exceção. assertRaises pode ser usado de duas maneiras:

  1. Usando a chamada de função normal. O primeiro argumento recebe o tipo de exceção, o segundo um callable (geralmente uma função) e o restante dos argumentos são passados ​​para esse callable.
  2. Usando uma cláusula with, dando apenas o tipo de exceção para a função. Isso tem a vantagem de que mais código pode ser executado, mas deve ser usado com cuidado, pois várias funções podem usar a mesma exceção, o que pode ser problemático. Um exemplo: with self.assertRaises(WrongInputException): convert2number(“not a number”)

Este primeiro foi implementado no seguinte caso de teste:

import unittest

class ExceptionTestCase(unittest.TestCase):

    def test_wrong_input_string(self):
        self.assertRaises(WrongInputException, convert2number, "not a number")

    def test_correct_input(self):
        try:
            result = convert2number("56")
            self.assertIsInstance(result, int)
        except WrongInputException:
            self.fail()

Também pode ser necessário verificar uma exceção que não deveria ter sido lançada. No entanto, um teste falhará automaticamente quando uma exceção for lançada e, portanto, pode não ser necessário. Apenas para mostrar as opções, o segundo método de teste mostra um caso de como se pode verificar se uma exceção não deve ser lançada. Basicamente, isso é capturar a exceção e depois falhar no teste usando o método fail.

Escolhendo asserções dentro dos Unittests #

Enquanto o Python tem uma instrução assert, a estrutura de teste de unidade do Python tem asserções melhores especializadas para testes: elas são mais informativos sobre falhas e não dependem do modo de depuração da execução.

Talvez a afirmação mais simples seja assertTrue, que pode ser usada assim:

import unittest

class SimplisticTest(unittest.TestCase):
    def test_basic(self):
        self.assertTrue(1 + 1 == 2)

Isso funcionará bem, mas substituindo a linha acima por

        self.assertTrue(1 + 1 == 3)

vai falhar.

A asserção assertTrue é provavelmente a asserção mais geral, já que qualquer coisa testada pode ser convertida como alguma condição booleana, mas geralmente existem alternativas melhores. Ao testar a igualdade, como acima, é melhor escrever

        self.assertEqual(1 + 1, 3)

Quando o primeiro falha, a mensagem é

======================================================================

FAIL: test (__main__.TruthTest)

----------------------------------------------------------------------

Traceback (most recent call last):

  File "stuff.py", line 6, in test

    self.assertTrue(1 + 1 == 3)

AssertionError: False is not true

mas quando o último falha, a mensagem é

======================================================================

FAIL: test (__main__.TruthTest)

----------------------------------------------------------------------

Traceback (most recent call last):

  File "stuff.py", line 6, in test

    self.assertEqual(1 + 1, 3)
AssertionError: 2 != 3

que é mais informativo (na verdade, avaliou o resultado do lado esquerdo).

Você pode encontrar a lista de asserções na documentação padrão. Em geral, é uma boa ideia escolher a afirmação que mais se adequa à condição. Assim, como mostrado acima, para afirmar que 1 + 1 == 2 é melhor usar assertEqual do que assertTrue. Da mesma forma, para afirmar que a é None, é melhor usar assertIsNone do que assertEqual.

Observe também que as afirmações têm formas negativas. Assim assertEqual tem sua contraparte negativa assertNotEqual, e assertIsNone tem sua contraparte negativa assertIsNotNone. Mais uma vez, usar as contrapartes negativas quando apropriado, levará a mensagens de erro mais claras.

Funções de simulação com unittest.mock.create_autospec #

Uma maneira de simular uma função é usar a função create_autospec, que irá simular um objeto de acordo com suas especificações. Com funções, podemos usar isso para garantir que elas sejam chamadas adequadamente.

Com uma função multiply em custom_math.py:

def multiply(a, b):
    return a * b

E uma função multiples_of em process_math.py:

from custom_math import multiply


def multiples_of(integer, *args, num_multiples=0, **kwargs):
    """
    :rtype: list
    """
    multiples = []
    
    for x in range(1, num_multiples + 1):
        """
        Passing in args and kwargs here will only raise TypeError if values were 
        passed to multiples_of function, otherwise they are ignored. This way we can 
        test that multiples_of is used correctly. This is here for an illustration
        of how create_autospec works. Not recommended for production code.
        """
        multiple = multiply(integer,x, *args, **kwargs)
        multiples.append(multiple)
    
    return multiples

Podemos testar multiples_of sozinho zombando de multiply. O exemplo abaixo usa a biblioteca padrão do Python unittest, mas isso também pode ser usado com outras estruturas de teste, como pytest ou nose:

from unittest.mock import create_autospec
import unittest

# we import the entire module so we can mock out multiply
import custom_math 
custom_math.multiply = create_autospec(custom_math.multiply)
from process_math import multiples_of


class TestCustomMath(unittest.TestCase):
    def test_multiples_of(self):
        multiples = multiples_of(3, num_multiples=1)
        custom_math.multiply.assert_called_with(3, 1)
    
    def test_multiples_of_with_bad_inputs(self):
        with self.assertRaises(TypeError) as e:
            multiples_of(1, "extra arg",  num_multiples=1) # this should raise a TypeError

def docker_exec_something(something_file_string):
    fl = Popen(["docker", "exec", "-i", "something_cont", "something"], stdin=PIPE, stdout=PIPE, stderr=PIPE)
    fl.stdin.write(something_file_string)
    fl.stdin.close()
    err = fl.stderr.read()
    fl.stderr.close()
    if err:
        print(err)
        exit()
    result = fl.stdout.read()
    print(result)

class MockBytes():
    '''Used to collect bytes
    '''
    all_read = []
    all_write = []
    all_close = []

    def read(self, *args, **kwargs):
        # print('read', args, kwargs, dir(self))
        self.all_read.append((self, args, kwargs))

    def write(self, *args, **kwargs):
        # print('wrote', args, kwargs)
        self.all_write.append((self, args, kwargs))

    def close(self, *args, **kwargs):
        # print('closed', self, args, kwargs)
        self.all_close.append((self, args, kwargs))

    def get_all_mock_bytes(self):
        return self.all_read, self.all_write, self.all_close

Monkey patching com pytest em test_docker.py:

@pytest.fixture
def all_popens(monkeypatch):
    '''This fixture overrides / mocks the builtin Popen
        and replaces stdin, stdout, stderr with a MockBytes object

        note: monkeypatch is magically imported
    '''
    all_popens = []
    
    class MockPopen(object):
        def __init__(self, args, stdout=None, stdin=None, stderr=None):
            all_popens.append(self)
            self.args = args
            self.byte_collection = MockBytes()
            self.stdin = self.byte_collection
            self.stdout = self.byte_collection
            self.stderr = self.byte_collection
            pass
    monkeypatch.setattr(helpers, 'Popen', MockPopen)

    return all_popens
