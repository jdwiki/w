Corrotina e Sintaxe de Delegação #

Antes do lançamento do Python 3.5+, o módulo asyncio usava geradores para imitar chamadas assíncronas e, portanto, tinha uma sintaxe diferente da versão atual do Python 3.5.

O Python 3.5 introduziu as palavras-chave async e await. Observe a falta de parênteses em torno da chamada await func().

import asyncio

async def main():
    print(await func())

async def func():
    # Do time intensive stuff...
    return "Hello, world!"

if __name__ == "__main__":
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

Antes do Python 3.5, o decorador @asyncio.coroutine era usado para definir uma corrotina. O rendimento da expressão foi usado para delegação do gerador. Observe os parênteses em torno do yield from func().

import asyncio

@asyncio.coroutine
def main():
    print((yield from func()))

@asyncio.coroutine
def func():
    # Do time intensive stuff..
    return "Hello, world!"

if __name__ == "__main__":
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

Aqui está um exemplo que mostra como duas funções podem ser executadas de forma assíncrona:

import asyncio

async def cor1():
    print("cor1 start")
    for i in range(10):
        await asyncio.sleep(1.5)
        print("cor1", i)

async def cor2():
    print("cor2 start")
    for i in range(15):
        await asyncio.sleep(1)
        print("cor2", i)

loop = asyncio.get_event_loop()
cors = asyncio.wait([cor1(), cor2()])
loop.run_until_complete(cors)

Executores assíncronos #

Observação: usa a sintaxe assíncrona/aguardada do Python 3.5+

asyncio suporta o uso de objetos Executor encontrados em concurrent.futures para agendar tarefas de forma assíncrona. Os loops de eventos possuem a função run_in_executor() que recebe um objeto Executor, um Callable e os parâmetros do Callable.

Agendando uma tarefa para um Executor

import asyncio
from concurrent.futures import ThreadPoolExecutor

def func(a, b):
    # Do time intensive stuff...
    return a + b

async def main(loop):
    executor = ThreadPoolExecutor()
    result = await loop.run_in_executor(executor, func, "Hello,", " world!")
    print(result)

if __name__ == "__main__":
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main(loop))

Cada loop de eventos também tem um slot Executor “padrão” que pode ser atribuído a um Executor. Para atribuir um Executor e agendar tarefas do loop você usa o método set_default_executor().

import asyncio
from concurrent.futures import ThreadPoolExecutor

def func(a, b):
    # Do time intensive stuff...
    return a + b

async def main(loop):
    # NOTE: Using `None` as the first parameter designates the `default` Executor.
    result = await loop.run_in_executor(None, func, "Hello,", " world!")
    print(result)

if __name__ == "__main__":
    loop = asyncio.get_event_loop()
    loop.set_default_executor(ThreadPoolExecutor())
    loop.run_until_complete(main(loop))

Existem dois tipos principais de Executor em concurrent.futures, o ThreadPoolExecutor e o ProcessPoolExecutor. O ThreadPoolExecutor contém um pool de threads que pode ser definido manualmente para um número específico de threads através do construtor ou padrão para o número de núcleos na máquina vezes 5. O ThreadPoolExecutor usa o pool de threads para executar tarefas atribuídas a ele e geralmente é melhor em operações vinculadas à CPU em vez de operações vinculadas a E/S. Compare isso com o ProcessPoolExecutor que gera um novo processo para cada tarefa atribuída a ele. O ProcessPoolExecutor só pode receber tarefas e parâmetros que podem ser selecionados. As tarefas não-picklable mais comuns são os métodos de objetos. Se você deve agendar o método de um objeto como uma tarefa em um Executor você deve usar um ThreadPoolExecutor.

Usando UVLoop #

uvloop é uma implementação para o asyncio.AbstractEventLoop baseado em libuv (usado por nodejs). É compatível com 99% dos recursos do asyncio e é muito mais rápido que o tradicional asyncio.EventLoop. uvloop não está disponível no Windows, instale-o com pip install uvloop.

import asyncio
import uvloop

if __name__ == "__main__":
    asyncio.set_event_loop(uvloop.new_event_loop())
    # Do your stuff here ...

Pode-se também alterar a fábrica do loop de eventos definindo o EventLoopPolicy para aquele em uvloop.

import asyncio
import uvloop

if __name__ == "__main__":
    asyncio.set_event_loop_policy(uvloop.EventLoopPolicy())
    loop = asyncio.new_event_loop()

Primitiva de Sincronização: Evento #

Conceito #

Use um Evento para sincronizar o agendamento de várias corrotinas.

Simplificando, um evento é como o tiro de uma arma em uma corrida: ele deixa os corredores fora dos blocos de partida.

Exemplo #

import asyncio

# event trigger function
def trigger(event):
    print('EVENT SET')
    event.set()  # wake up coroutines waiting

# event consumers
async def consumer_a(event):
    consumer_name = 'Consumer A'
    print('{} waiting'.format(consumer_name))
    await event.wait()
    print('{} triggered'.format(consumer_name))

async def consumer_b(event):
    consumer_name = 'Consumer B'
    print('{} waiting'.format(consumer_name))
    await event.wait()
    print('{} triggered'.format(consumer_name))

# event
event = asyncio.Event()

# wrap coroutines in one future
main_future = asyncio.wait([consumer_a(event),
                            consumer_b(event)])

# event loop
event_loop = asyncio.get_event_loop()
event_loop.call_later(0.1, functools.partial(trigger, event))  # trigger event in 0.1 sec

# complete main_future
done, pending = event_loop.run_until_complete(main_future)

Resultado:

Consumidor B esperando Consumidor A esperando CONJUNTO DE EVENTO Consumidor B acionado Consumidor A acionado

Um Websocket Simples #

Aqui fazemos um websocket de eco simples usando asyncio. Definimos corrotinas para conectar a um servidor e enviar/receber mensagens. As comunicações do websocket são executadas em uma corrotina main, que é executada por um event loop. Este exemplo foi modificado de uma postagem anterior.

import asyncio
import aiohttp

session = aiohttp.ClientSession()                          # handles the context manager
class EchoWebsocket:
    
    async def connect(self):
        self.websocket = await session.ws_connect("wss://echo.websocket.org")
        
    async def send(self, message):
        self.websocket.send_str(message)

    async def receive(self):
        result = (await self.websocket.receive())
        return result.data

async def main():
    echo = EchoWebsocket()
    await echo.connect()
    await echo.send("Hello World!")
    print(await echo.receive())                            # "Hello World!"


if __name__ == '__main__':
    # The main loop
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

Equívoco comum sobre assíncrono #

provavelmente o equívoco mais comum sobre o asnycio é que ele permite que você execute qualquer tarefa em paralelo - evitando o GIL (bloqueio de intérprete global) e, portanto, execute tarefas de bloqueio em paralelo (em threads separados). isso não**!

asyncio (e bibliotecas que são construídas para colaborar com asyncio) são construídas em corrotinas: funções que (de forma colaborativa) retornam o fluxo de controle para a função de chamada. observe asyncio.sleep nos exemplos acima. este é um exemplo de uma corrotina sem bloqueio que espera ’em segundo plano’ e devolve o fluxo de controle para a função de chamada (quando chamada com await). time.sleep é um exemplo de função de bloqueio. o fluxo de execução do programa irá parar ali e só retornará após o término do time.sleep.

um exemplo real é a biblioteca requests que consiste (por enquanto) apenas em funções de bloqueio. não há concorrência se você chamar qualquer uma de suas funções dentro de asyncio. aiohttp, por outro lado, foi construído com o asyncio em mente. suas corrotinas serão executadas simultaneamente.

  • se você tiver tarefas de longa duração vinculadas à CPU que gostaria de executar em paralelo, o asyncio não é para você. para isso você precisa de threads ou [multiprocessing](https://docs.python .org/3/library/multiprocessing.html?#module-multiprocessing).

  • se você tiver trabalhos vinculados a IO em execução, pode executá-los simultaneamente usando asyncio.
