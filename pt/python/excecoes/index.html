Erros detectados durante a execução são chamados de exceções e não são incondicionalmente fatais. A maioria das exceções não é tratada por programas; é possível escrever programas que tratam de exceções selecionadas. Existem recursos específicos em Python para lidar com exceções e lógica de exceção. Além disso, as exceções têm uma hierarquia de tipos rica, todas herdando do tipo BaseException.

Sintaxe #

  • levanta exceção
  • raise # re-raise uma exceção que já foi levantada
  • levanta exceção de causa # Python 3 - define a causa da exceção
  • aumentar exceção de Nenhum # Python 3 - suprimir todo o contexto de exceção
  • tentar:
  • exceto [tipos de exceção] [ como identificador ]:
  • senão:
  • finalmente:

Não pegue tudo! #

Embora muitas vezes seja tentador capturar cada Exception:

try:
    very_difficult_function()
except Exception:
    # log / try to reconnect / exit gratiously
finally:
    print "The END"
    # it runs no matter what execute.

Ou mesmo tudo (que inclui BaseException e todos os seus filhos, incluindo Exception):

try:
    even_more_difficult_function()
except:
    pass  # do whatever needed

Na maioria dos casos é uma má prática. Ele pode capturar mais do que o pretendido, como SystemExit, KeyboardInterrupt e MemoryError - cada um dos quais geralmente deve ser tratado de maneira diferente dos erros normais de sistema ou lógica. Isso também significa que não há uma compreensão clara do que o código interno pode fazer de errado e como se recuperar adequadamente dessa condição. Se você está pegando todos os erros, você não saberá qual erro ocorreu ou como corrigi-lo.

Isso é mais comumente referido como ‘mascaramento de bug’ e deve ser evitado. Deixe seu programa travar em vez de falhar silenciosamente ou, pior ainda, falhar em um nível mais profundo de execução. (Imagine que é um sistema transacional)

Normalmente, essas construções são usadas no nível mais externo do programa e registrarão os detalhes do erro para que o bug possa ser corrigido ou o erro possa ser tratado mais especificamente.

Re-aumentando exceções #

Às vezes, você deseja capturar uma exceção apenas para inspecioná-la, por exemplo. para fins de registro. Após a inspeção, você deseja que a exceção continue se propagando como antes.

Neste caso, simplesmente use a instrução raise sem parâmetros.

try:
    5 / 0
except ZeroDivisionError:
    print("Got an error")
    raise

Tenha em mente, porém, que alguém mais acima na pilha de chamadores ainda pode pegar a exceção e tratá-la de alguma forma. A saída pronta pode ser um incômodo neste caso porque acontecerá em qualquer caso (capturado ou não capturado). Portanto, pode ser uma ideia melhor gerar uma exceção diferente, contendo seu comentário sobre a situação, bem como a exceção original:

try:
    5 / 0
except ZeroDivisionError as e:
    raise ZeroDivisionError("Got an error", e)

Mas isso tem a desvantagem de reduzir o rastreamento de exceção exatamente para este raise enquanto o raise sem argumento retém o rastreamento de exceção original.

No Python 3 você pode manter a pilha original usando a sintaxe raise-from:

    raise ZeroDivisionError("Got an error") from e

Capturando várias exceções #

Existem algumas maneiras de capturar várias exceções.

A primeira é criar uma tupla dos tipos de exceção que você deseja capturar e manipular da mesma maneira. Este exemplo fará com que o código ignore as exceções KeyError e AttributeError.

try:
    d = {}
    a = d[1]
    b = d.non_existing_field
except (KeyError, AttributeError) as e:
    print("A KeyError or an AttributeError exception has been caught.")

Se você deseja tratar diferentes exceções de maneiras diferentes, você pode fornecer um bloco de exceção separado para cada tipo. Neste exemplo, ainda capturamos o KeyError e o AttributeError, mas tratamos as exceções de maneiras diferentes.

try:
    d = {}
    a = d[1]
    b = d.non_existing_field
except KeyError as e:
    print("A KeyError has occurred. Exception message:", e)
except AttributeError as e:
    print("An AttributeError has occurred. Exception message:", e)

Capturando exceções #

Use try...except: para capturar exceções. Você deve especificar uma exceção tão precisa quanto possível:

try:
    x = 5 / 0
except ZeroDivisionError as e:
    # `e` is the exception object
    print("Got a divide by zero! The exception was:", e)
    # handle exceptional case
    x = 0  
finally:
    print "The END"
    # it runs no matter what execute.

A classe de exceção especificada - neste caso, ZeroDivisionError - captura qualquer exceção dessa classe ou de qualquer subclasse dessa exceção.

Por exemplo, ZeroDivisionError é uma subclasse de ArithmeticError:

>>> ZeroDivisionError.__bases__
(<class 'ArithmeticError'>,)

E assim, o seguinte ainda pegará o ZeroDivisionError:

try:
    5 / 0
except ArithmeticError:
    print("Got arithmetic error")

Senão #

O código em um bloco else só será executado se nenhuma exceção for levantada pelo código no bloco try. Isso é útil se você tiver algum código que não deseja executar se uma exceção for lançada, mas você não deseja que exceções lançadas por esse código sejam capturadas.

Por exemplo:

try:
    data = {1: 'one', 2: 'two'}
    print(data[1])
except KeyError as e:
    print('key not found')
else:
    raise ValueError()
# Output: one
# Output: ValueError

Note que este tipo de else: não pode ser combinado com um if iniciando a cláusula else para um elif. Se você tiver um if a seguir, ele precisa ficar recuado abaixo desse else::

try:
    ...
except ...:
    ...
else:
    if ...:
        ...
    elif ...:
        ...
    else:
        ...

Criando exceções #

Se o seu código encontrar uma condição que não sabe como lidar, como um parâmetro incorreto, ele deve gerar a exceção apropriada.

def even_the_odds(odds):
    if odds % 2 != 1:
        raise ValueError("Did not get an odd number")
    
    return odds + 1

Criando tipos de exceção personalizados #

Crie uma classe herdando de Exception:

class FooException(Exception):
    pass
try:
    raise FooException("insert description here")
except FooException:
    print("A FooException was raised.")

ou outro tipo de exceção:

class NegativeError(ValueError):
      pass

def foo(x):
    # function that only accepts positive values of x
    if x < 0:
        raise NegativeError("Cannot process negative numbers")
    ...  # rest of function body
try:
    result = foo(int(input("Enter a positive integer: ")))  # raw_input in Python 2.x
except NegativeError:
    print("You entered a negative number!")
else:
    print("The result was " + str(result))

Exceções também são objetos #

As exceções são apenas objetos Python regulares que herdam do BaseException interno. Um script Python pode usar a instrução raise para interromper a execução, fazendo com que o Python imprima um rastreamento de pilha da pilha de chamadas nesse ponto e uma representação da instância de exceção. Por exemplo:

>>> def failing_function():
...     raise ValueError('Example error!')
>>> failing_function()
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "<stdin>", line 2, in failing_function
ValueError: Example error!

que diz que um ValueError com a mensagem 'Example error!' foi gerado por nossa failing_function(), que foi executada no interpretador.

O código de chamada pode optar por lidar com todo e qualquer tipo de exceção que uma chamada pode gerar:

>>> try:
...     failing_function()
... except ValueError:
...     print('Handled the error')
Handled the error

Você pode obter os objetos de exceção atribuindo-os na parte except... do código de tratamento de exceção:

>>> try:
...     failing_function()
... except ValueError as e:
...     print('Caught exception', repr(e))
Caught exception ValueError('Example error!',)

Uma lista completa de exceções internas do Python, juntamente com suas descrições, pode ser encontrada na Documentação do Python: https://docs.python.org/3.5/library/exceptions.html. E aqui está a lista completa organizada hierarquicamente: https://www.wikiod.com/pt/python/excecoes#Hierarquia de exceção

Exemplos práticos de tratamento de exceções #

Entrada do usuário #

Imagine que você deseja que um usuário insira um número via input. Você deseja garantir que a entrada seja um número. Você pode usar try/except para isso:

while True:
    try:
        nb = int(input('Enter a number: '))
        break
    except ValueError:
        print('This is not a number, try again.')

Nota: Python 2.x usaria raw_input em vez disso; a função input existe no Python 2.x mas tem uma semântica diferente. No exemplo acima, input também aceitaria expressões como 2 + 2 que são avaliadas como um número.

Se a entrada não puder ser convertida em um inteiro, um ValueError será gerado. Você pode pegá-lo com except. Se nenhuma exceção for levantada, break salta para fora do loop. Após o loop, nb contém um inteiro.

Dicionários #

Imagine que você está iterando sobre uma lista de inteiros consecutivos, como range(n), e você tem uma lista de dicionários d que contém informações sobre o que fazer quando você encontra alguns inteiros em particular, digamos skip the d[i ] próximos.

d = [{7: 3}, {25: 9}, {38: 5}]

for i in range(len(d)):
    do_stuff(i)
    try:
       dic = d[i]
       i += dic[i]
    except KeyError:
       i += 1

Um KeyError será gerado quando você tentar obter um valor de um dicionário para uma chave que não existe.

Executando o código de limpeza com finally #

Às vezes, você pode querer que algo ocorra independentemente de qualquer exceção que tenha ocorrido, por exemplo, se você precisar limpar alguns recursos.

O bloco finally de uma cláusula try acontecerá independentemente de alguma exceção ter sido levantada.

resource = allocate_some_expensive_resource()
try:
    do_stuff(resource)
except SomeException as e:
    log_error(e)
    raise  # re-raise the error
finally:
    free_expensive_resource(resource)

Esse padrão geralmente é melhor tratado com gerenciadores de contexto (usando a instrução with).

Exceções de cadeia com aumento de #

No processo de tratamento de uma exceção, você pode querer gerar outra exceção. Por exemplo, se você receber um IOError durante a leitura de um arquivo, talvez queira gerar um erro específico do aplicativo para apresentar aos usuários de sua biblioteca.

Você pode encadear exceções para mostrar como o tratamento de exceções ocorreu:

>>> try:
    5 / 0
except ZeroDivisionError as e:
    raise ValueError("Division failed") from e

Traceback (most recent call last):
  File "<stdin>", line 2, in <module>
ZeroDivisionError: division by zero

The above exception was the direct cause of the following exception:

Traceback (most recent call last):
  File "<stdin>", line 4, in <module>
ValueError: Division failed
