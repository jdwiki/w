Em sua essência, o coletor de lixo do Python (a partir da versão 3.5) é uma implementação simples de contagem de referência. Toda vez que você faz uma referência a um objeto (por exemplo, a = meuobjeto), a contagem de referência desse objeto (meuobjeto) é incrementada. Toda vez que uma referência é removida, a contagem de referência é decrementada e, uma vez que a contagem de referência atinge ‘0’, sabemos que nada contém uma referência a esse objeto e podemos desalocá-lo!

Um mal-entendido comum sobre como o gerenciamento de memória do Python funciona é que a palavra-chave del libera a memória dos objetos. Isso não é verdade. O que realmente acontece é que a palavra-chave del apenas decrementa a refcount dos objetos, significando que se você chamá-la vezes suficientes para que a refcount chegue a zero, o objeto pode ser coletado como lixo (mesmo que ainda existam referências ao objeto disponíveis em outro lugar no seu código).

O Python cria ou limpa objetos agressivamente na primeira vez que precisa deles Se eu executar a atribuição a = object(), a memória para o objeto é alocada naquele momento (o cpython às vezes reutilizará certos tipos de objeto, por exemplo, listas sob o capô, mas principalmente ele não mantém um pool de objetos livre e realizará a alocação quando você precisar). Da mesma forma, assim que o refcount é decrementado para 0, o GC o limpa.

Coleta de lixo geracional #

Na década de 1960, John McCarthy descobriu uma falha fatal na coleta de lixo de refcounting quando implementou o algoritmo de refcounting usado por Lisp: O que acontece se dois objetos se referirem um ao outro em uma referência cíclica? Como você pode coletar lixo desses dois objetos, mesmo que não haja referências externas a eles, se eles sempre se referem um ao outro? Esse problema também se estende a qualquer estrutura de dados cíclica, como buffers de anel ou quaisquer duas entradas consecutivas em uma lista duplamente vinculada. O Python tenta corrigir esse problema usando uma reviravolta um pouco interessante em outro algoritmo de coleta de lixo chamado Generational Garbage Collection.

Em essência, sempre que você cria um objeto em Python, ele o adiciona ao final de uma lista duplamente vinculada. Ocasionalmente, o Python percorre essa lista, verifica a quais objetos os objetos na lista se referem também e, se eles também estão na lista (veremos por que eles podem não estar em um momento), diminui ainda mais suas refcounts. Neste ponto (na verdade, existem algumas heurísticas que determinam quando as coisas são movidas, mas vamos supor que é depois de uma única coleção para manter as coisas simples) qualquer coisa que ainda tenha uma refcount maior que 0 é promovida para outra lista vinculada chamada “Geração 1” (é por isso que todos os objetos nem sempre estão na lista de geração 0) que tem esse loop aplicado a ele com menos frequência. É aqui que entra a coleta de lixo geracional. Existem 3 gerações por padrão no Python (três listas vinculadas de objetos): A primeira lista (geração 0) contém todos os novos objetos; se um ciclo de GC acontecer e os objetos não forem coletados, eles serão movidos para a segunda lista (geração 1), e se um ciclo de GC acontecer na segunda lista e eles ainda não forem coletados, eles serão movidos para a terceira lista (geração 2) ). A lista de terceira geração (chamada “geração 2”, já que estamos com indexação zero) é coletada de lixo com muito menos frequência do que as duas primeiras, a ideia é que, se seu objeto tiver vida longa, não é tão provável que seja GCed e talvez nunca ser GCed durante a vida útil de seu aplicativo, portanto, não faz sentido perder tempo verificando-o em cada execução de GC. Além disso, observa-se que a maioria dos objetos são coletados de forma relativamente rápida. De agora em diante, chamaremos esses “objetos bons”, pois morrem jovens. Isso é chamado de “hipótese geracional fraca” e também foi observado pela primeira vez nos anos 60.

Um aparte rápido: ao contrário das duas primeiras gerações, a longa lista de terceira geração não é coletada de lixo regularmente. É verificado quando a proporção de objetos pendentes de vida longa (aqueles que estão na lista de terceira geração, mas ainda não tiveram um ciclo de GC) para o total de objetos de vida longa na lista é maior que 25%. Isso ocorre porque a terceira lista é ilimitada (as coisas nunca são movidas para outra lista, então elas só desaparecem quando são realmente coletadas como lixo), o que significa que, para aplicativos em que você está criando muitos objetos de vida longa, os ciclos de GC na terceira lista pode ficar bastante longo. Usando uma razão, alcançamos “desempenho linear amortizado no número total de objetos”; aka, quanto maior a lista, mais tempo o GC leva, mas com menos frequência realizamos o GC (aqui está a [proposta original de 2008](https://mail.python.org/pipermail/python-dev/2008-June/080579. html) para esta heurística de Martin von Löwis para leitura adicional). O ato de realizar uma coleta de lixo na terceira geração ou lista “madura” é chamado de “coleta de lixo completa”.

Portanto, a coleta de lixo geracional acelera tremendamente as coisas, não exigindo que examinemos objetos que provavelmente não precisam de GC o tempo todo, mas como isso nos ajuda a quebrar referências cíclicas? Provavelmente não muito bem, ao que parece. A função para realmente quebrar esses ciclos de referência começa assim:

/* Break reference cycles by clearing the containers involved.  This is
 * tricky business as the lists can be changing and we don't know which
 * objects may be freed.  It is possible I screwed something up here.
 */
static void
delete_garbage(PyGC_Head *collectable, PyGC_Head *old)

A razão pela qual a coleta de lixo geracional ajuda nisso é que podemos manter o comprimento da lista como uma contagem separada; cada vez que adicionamos um novo objeto à geração, incrementamos essa contagem, e sempre que movemos um objeto para outra geração ou o desalocamos, diminuímos a contagem. Teoricamente, no final de um ciclo de GC, essa contagem (pelo menos nas duas primeiras gerações) deve sempre ser 0. Se não for, qualquer coisa na lista que sobrar é alguma forma de referência circular e podemos abandoná-la. No entanto, há mais um problema aqui: E se os objetos restantes tiverem o método mágico __del__ do Python neles? __del__ é chamado sempre que um objeto Python é destruído. No entanto, se dois objetos em uma referência circular têm métodos __del__, não podemos ter certeza de que destruir um não quebrará os outros métodos __del__. Para um exemplo artificial, imagine que escrevemos o seguinte:

class A(object):
    def __init__(self, b=None):
        self.b = b
 
    def __del__(self):
        print("We're deleting an instance of A containing:", self.b)
     
class B(object):
    def __init__(self, a=None):
        self.a = a
 
    def __del__(self):
        print("We're deleting an instance of B containing:", self.a)

e definimos uma instância de A e uma instância de B para apontar uma para a outra e então elas terminam no mesmo ciclo de coleta de lixo? Digamos que escolhemos um aleatoriamente e desalocamos nossa instância de A primeiro; O método __del__ de A será chamado, ele imprimirá, então A será liberado. Em seguida, chegamos a B, chamamos seu método __del__ e oops! Segfault! A não existe mais. Nós poderíamos consertar isso chamando tudo o que sobrou dos métodos __del__ primeiro, então fazendo outra passagem para realmente desalocar tudo, no entanto, isso introduz outro problema: E se um método de objetos __del__ salvar uma referência do outro objeto que está sobre ser GCed e tem uma referência a nós em outro lugar? Ainda temos um ciclo de referência, mas agora não é possível realmente GC de nenhum dos objetos, mesmo que eles não estejam mais em uso. Observe que mesmo que um objeto não faça parte de uma estrutura de dados circular, ele pode reviver em seu próprio método __del__; Python tem uma verificação para isso e irá parar o GC se um refcount de objetos tiver aumentado depois que seu método __del__ for chamado.

O CPython lida com isso colocando esses objetos não compatíveis com GC (qualquer coisa com alguma forma de referência circular e um método __del__) em uma lista global de lixo incobrável e, em seguida, deixando-o lá por toda a eternidade:

/* list of uncollectable objects */
static PyObject *garbage = NULL;

Reutilização de objetos primitivos #

Uma coisa interessante a ser observada, que pode ajudar a otimizar seus aplicativos, é que as primitivas também são contadas sob o capô. Vamos dar uma olhada nos números; para todos os inteiros entre -5 e 256, o Python sempre reutiliza o mesmo objeto:

>>> import sys
>>> sys.getrefcount(1)
797
>>> a = 1
>>> b = 1
>>> sys.getrefcount(1)
799

Observe que a refcount aumenta, o que significa que a e b fazem referência ao mesmo objeto subjacente quando se referem à primitiva 1. No entanto, para números maiores, o Python na verdade não reutiliza o objeto subjacente:

>>> a = 999999999
>>> sys.getrefcount(999999999)
3
>>> b = 999999999
>>> sys.getrefcount(999999999)
3

Como o refcount para 999999999 não muda ao atribuí-lo a a e b, podemos inferir que eles se referem a dois objetos subjacentes diferentes, mesmo que ambos sejam atribuídos à mesma primitiva.

Efeitos do comando del #

Removendo um nome de variável do escopo usando del v, ou removendo um objeto de uma coleção usando del v[item] ou del[i:j], ou removendo um atributo usando del v.name, ou qualquer outra maneira de remover referências a um objeto, não aciona nenhuma chamada de destruidor ou qualquer memória sendo liberada por si mesma. Os objetos só são destruídos quando sua contagem de referência chega a zero.

>>> import gc
>>> gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def bar():
    return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> del t          # not destructed yet - another_t still refers to it
>>> del another_t  # final reference gone, object is destructed
Destructed

Contagem de referência #

A grande maioria do gerenciamento de memória do Python é tratada com contagem de referência.

Toda vez que um objeto é referenciado (por exemplo, atribuído a uma variável), sua contagem de referência é aumentada automaticamente. Quando é desreferenciado (por exemplo, a variável sai do escopo), sua contagem de referência é automaticamente diminuída.

Quando a contagem de referência chega a zero, o objeto é imediatamente destruído e a memória é liberada imediatamente. Assim, para a maioria dos casos, o coletor de lixo nem é necessário.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> def foo():
        Track()
        # destructed immediately since no longer has any references
        print("---")
        t = Track()
        # variable is referenced, so it's not destructed yet
        print("---")
        # variable is destructed when function exits
>>> foo()
Initialized
Destructed
---
Initialized
---
Destructed

Para demonstrar ainda mais o conceito de referências:

>>> def bar():
        return Track()
>>> t = bar()
Initialized
>>> another_t = t  # assign another reference
>>> print("...")
...
>>> t = None          # not destructed yet - another_t still refers to it
>>> another_t = None  # final reference gone, object is destructed
Destructed

Coletor de lixo para ciclos de referência #

A única vez que o coletor de lixo é necessário é se você tiver um ciclo de referência. O exemplo simples de um ciclo de referência é aquele em que A se refere a B e B se refere a A, enquanto nada mais se refere a A ou B. Nem A nem B são acessíveis de qualquer lugar no programa, então eles podem ser destruídos com segurança, ainda assim, suas contagens de referência são 1 e, portanto, não podem ser liberadas apenas pelo algoritmo de contagem de referência.

>>> import gc; gc.disable()  # disable garbage collector
>>> class Track:
        def __init__(self):
            print("Initialized")
        def __del__(self):
            print("Destructed")
>>> A = Track()
Initialized
>>> B = Track()
Initialized
>>> A.other = B
>>> B.other = A
>>> del A; del B  # objects are not destructed due to reference cycle
>>> gc.collect()  # trigger collection
Destructed
Destructed
4

Um ciclo de referência pode ser arbitrariamente longo. Se A aponta para B aponta para C aponta para … aponta para Z que aponta para A, então nem A até Z serão coletados, até a fase de coleta de lixo:

>>> objs = [Track() for _ in range(10)]
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
Initialized
>>> for i in range(len(objs)-1):
...     objs[i].other = objs[i + 1]
...
>>> objs[-1].other = objs[0]  # complete the cycle
>>> del objs                  # no one can refer to objs now - still not destructed
>>> gc.collect()
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
Destructed
20

Visualizando o refcount de um objeto #

>>> import sys
>>> a = object()
>>> sys.getrefcount(a)
2
>>> b = a
>>> sys.getrefcount(a)
3
>>> del b
>>> sys.getrefcount(a)
2

Desalocar objetos com força #

Você pode forçar a desalocar objetos mesmo que sua refcount não seja 0 no Python 2 e 3.

Ambas as versões usam o módulo ctypes para isso.

AVISO: fazer isso deixará seu ambiente Python instável e propenso a travar sem rastreamento! Usar esse método também pode apresentar problemas de segurança (muito improvável) Desaloque apenas objetos que você tem certeza de que nunca fará referência novamente. Sempre.

import ctypes
deallocated = 12345
ctypes.pythonapi._Py_Dealloc(ctypes.py_object(deallocated))

import ctypes, sys
deallocated = 12345
(ctypes.c_char * sys.getsizeof(deallocated)).from_address(id(deallocated))[:4] = '\x00' * 4

Após a execução, qualquer referência ao objeto agora desalocado fará com que o Python produza um comportamento indefinido ou falhe - sem rastreamento. Provavelmente havia uma razão pela qual o coletor de lixo não removeu aquele objeto…

Se você desalocar None, você recebe uma mensagem especial - Fatal Python error: desalocating None antes de travar.

Gerenciando a coleta de lixo #

Há duas abordagens para influenciar quando uma limpeza de memória é executada. Eles estão influenciando a frequência com que o processo automático é executado e o outro está acionando manualmente uma limpeza.

O coletor de lixo pode ser manipulado ajustando os limites de coleta que afetam a frequência na qual o coletor é executado. Python usa um sistema de gerenciamento de memória baseado em geração. Novos objetos são salvos na geração mais recente - geração0 e a cada coleção sobrevivente, os objetos são promovidos às gerações mais antigas. Após atingir a última geração - geração2, eles não são mais promovidos.

Os limites podem ser alterados usando o seguinte snippet:

import gc
gc.set_threshold(1000, 100, 10) # Values are just for demonstration purpose

O primeiro argumento representa o limite para coletar geração0. Sempre que o número de alocações exceder o número de desalocações em 1.000, o coletor de lixo será chamado.

As gerações mais antigas não são limpas a cada execução para otimizar o processo. O segundo e terceiro argumentos são opcionais e controlam a frequência com que as gerações mais antigas são limpas. Se geração0 foi processada 100 vezes sem limpar geração1, geração1 será processada. Da mesma forma, os objetos da geração2 serão processados ​​somente quando os da geração1 forem limpos 10 vezes sem tocar na geração2.

Uma instância em que definir manualmente os limites é benéfico é quando o programa aloca muitos objetos pequenos sem desalocá-los, o que faz com que o coletor de lixo seja executado com muita frequência (cada alocação de objeto geração0_limite). Mesmo assim, o coletor é bastante rápido, quando roda em um grande número de objetos, apresenta um problema de desempenho. De qualquer forma, não há uma estratégia de tamanho único para escolher os limites e seu caso de uso é confiável.

O acionamento manual de uma coleção pode ser feito como no snippet a seguir:

import gc
gc.collect()

A coleta de lixo é acionada automaticamente com base no número de alocações e desalocações, não na memória consumida ou disponível. Consequentemente, ao trabalhar com objetos grandes, a memória pode se esgotar antes que a limpeza automatizada seja acionada. Isso é um bom caso de uso para chamar manualmente o coletor de lixo.

Mesmo que seja possível, não é uma prática encorajada. Evitar vazamentos de memória é a melhor opção. De qualquer forma, em grandes projetos, detectar o vazamento de memória pode ser uma tarefa difícil e acionar manualmente uma coleta de lixo pode ser usada como uma solução rápida até a depuração.

Para programas de longa execução, a coleta de lixo pode ser acionada por tempo ou por evento. Um exemplo para o primeiro é um servidor web que aciona uma coleção após um número fixo de solicitações. Para o posterior, um servidor web que aciona uma coleta de lixo quando um determinado tipo de solicitação é recebido.

Não espere a coleta de lixo limpar #

O fato de a coleta de lixo ser limpa não significa que você deva aguardar a limpeza do ciclo de coleta de lixo.

Em particular, você não deve esperar que a coleta de lixo feche identificadores de arquivo, conexões de banco de dados e conexões de rede abertas.

por exemplo:

No código a seguir, você assume que o arquivo será fechado no próximo ciclo de coleta de lixo, se f for a última referência ao arquivo.

>>> f = open("test.txt")
>>> del f

Uma maneira mais explícita de limpar é chamar f.close(). Você pode fazer isso de forma ainda mais elegante, ou seja, usando a instrução with, também conhecida como context manager :

>>> with open("test.txt") as f:
...     pass
...     # do something with f
>>> #now the f object still exists, but it is closed

A instrução with permite que você indente seu código sob o arquivo aberto. Isso torna explícito e fácil ver por quanto tempo um arquivo é mantido aberto. Ele também sempre fecha um arquivo, mesmo que uma exceção seja levantada no bloco while.
