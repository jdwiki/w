Estruturas são conjuntos de várias variáveis ​​empacotadas juntas. O próprio struct é apenas um package contendo variáveis ​​e tornando-as facilmente acessíveis.

Ao contrário de C, as estruturas de Go podem ter métodos anexados a elas. Também lhes permite implementar interfaces. Isso torna as estruturas de Go semelhantes a objetos, mas elas estão (provavelmente intencionalmente) perdendo alguns recursos importantes conhecidos em linguagens orientadas a objetos, como herança.

Métodos #

Os métodos struct são muito semelhantes às funções:

type User struct {
    name string
}

func (u User) Name() string {
    return u.name
}

func (u *User) SetName(newName string) {
    u.name = newName
}

A única diferença é a adição do receptor do método. Ele pode ser declarado como uma instância do tipo ou um ponteiro para uma instância do tipo. Como SetName() altera a instância, o receptor deve ser um ponteiro para efetuar uma alteração permanente na instância.

Por exemplo:

package main

import "fmt"

type User struct {
    name string
}

func (u User) Name() string {
    return u.name
}

func (u *User) SetName(newName string) {
    u.name = newName
}

func main() {
    var me User

    me.SetName("Slim Shady")
    fmt.Println("My name is", me.Name())
}

Estrutura anônima #

É possível criar uma estrutura anônima:

data := struct {
    Number int 
    Text   string
} { 
    42, 
    "Hello world!",
}

Exemplo completo:

package main

import (
    "fmt"
)

func main() {
    data := struct {Number int; Text string}{42, "Hello world!"} // anonymous struct
    fmt.Printf("%+v\n", data)
}

Composição e Incorporação #

A composição fornece uma alternativa à herança. Um struct pode incluir outro tipo por nome em sua declaração:

type Request struct {
    Resource string
}

type AuthenticatedRequest struct {
    Request
    Username, Password string
}

No exemplo acima, AuthenticatedRequest conterá quatro membros públicos: Resource, Request, Username e Password.

Estruturas compostas podem ser instanciadas e usadas da mesma maneira que estruturas normais:

func main() {
    ar := new(AuthenticatedRequest)
    ar.Resource = "example.com/request"
    ar.Username = "bob"
    ar.Password = "P@ssw0rd"
    fmt.Printf("%#v", ar)
}

jogue no playground

Incorporação #

No exemplo anterior, Request é um campo incorporado. A composição também pode ser obtida incorporando um tipo diferente. Isso é útil, por exemplo, para decorar um Struct com mais funcionalidade. Por exemplo, continuando com o exemplo Resource, queremos uma função que formate o conteúdo do campo Resource para prefixá-lo com http:// ou https://. Temos duas opções: criar os novos métodos em AuthenticatedRequest ou incorporá-los a partir de uma estrutura diferente:

type ResourceFormatter struct {}

func(r *ResourceFormatter) FormatHTTP(resource string) string {
    return fmt.Sprintf("http://%s", resource)
}
func(r *ResourceFormatter) FormatHTTPS(resource string) string {
    return fmt.Sprintf("https://%s", resource)
}


type AuthenticatedRequest struct {
    Request
    Username, Password string
    ResourceFormatter
}

E agora a função main poderia fazer o seguinte:

func main() {
    ar := new(AuthenticatedRequest)
    ar.Resource = "www.example.com/request"
    ar.Username = "bob"
    ar.Password = "P@ssw0rd"

    println(ar.FormatHTTP(ar.Resource))
    println(ar.FormatHTTPS(ar.Resource))

    fmt.Printf("%#v", ar)
}

Veja que o AuthenticatedRequest que tem uma estrutura incorporada ResourceFormatter.

Mas a desvantagem disso é que você não pode acessar objetos fora da sua composição. Portanto, ResourceFormatter não pode acessar membros de AuthenticatedRequest.

Campos exportados x não exportados (privado x público) #

Os campos de estrutura cujos nomes começam com uma letra maiúscula são exportados. Todos os outros nomes não são exportados.

type Account struct {
    UserID      int    // exported
    accessToken string // unexported
}

Campos não exportados só podem ser acessados ​​por código dentro do mesmo pacote. Assim, se você estiver acessando um campo de um pacote diferente, seu nome precisa começar com uma letra maiúscula.

package main

import "bank"

func main() {
    var x = &bank.Account{
        UserID: 1,          // this works fine
        accessToken: "one", // this does not work, since accessToken is unexported
    }
}

No entanto, de dentro do pacote bank, você pode acessar UserId e accessToken sem problemas.

O pacote bank poderia ser implementado assim:

package bank

type Account struct {
    UserID int
    accessToken string
}

func ProcessUser(u *Account) {    
    u.accessToken = doSomething(u) // ProcessUser() can access u.accessToken because 
                                   // it's defined in the same package
}

Declaração Básica #

Uma estrutura básica é declarada da seguinte forma:

type User struct {
    FirstName, LastName string
    Email               string
    Age                 int
}

Cada valor é chamado de campo. Os campos geralmente são escritos um por linha, com o nome do campo precedendo seu tipo. Campos consecutivos do mesmo tipo podem ser combinados, como FirstName e LastName no exemplo acima.

Tag #

Os campos struct podem ter tags associadas a eles. Essas tags podem ser lidas pelo pacote reflect para obter informações personalizadas especificadas sobre um campo pelo desenvolvedor.

struct Account {
    Username      string `json:"username"`
    DisplayName   string `json:"display_name"`
    FavoriteColor string `json:"favorite_color,omitempty"`
}

No exemplo acima, as tags são usadas para alterar os nomes de chave usados ​​pelo pacote encoding/json ao realizar o empacotamento ou desempacotamento do JSON.

Embora a tag possa ser qualquer valor de string, é considerado uma prática recomendada usar pares key:"value" separados por espaço:

struct StructName {
    FieldName int `package1:"customdata,moredata" package2:"info"`
}

As tags struct usadas com o pacote encoding/xml e encoding/json são usadas em toda a biblioteca padrão.

Literais Estruturais #

Um valor de um tipo de estrutura pode ser escrito usando um literal de estrutura que especifica valores para seus campos.

type Point struct { X, Y int }
p := Point{1, 2}

O exemplo acima especifica cada campo na ordem correta. O que não é útil, porque os programadores precisam lembrar os campos exatos em ordem. Mais frequentemente, um struct pode ser inicializado listando alguns ou todos os nomes de campo e seus valores correspondentes.

anim := gif.GIF{LoopCount: nframes}

Os campos omitidos são definidos com o valor zero para seu tipo.

Observação: As duas formas não podem ser misturadas no mesmo literal.

Fazendo cópias de struct. #

Uma estrutura pode simplesmente ser copiada usando atribuição.

type T struct {
    I int
    S string
}

// initialize a struct
t := T{1, "one"}

// make struct copy
u := t // u has its field values equal to t

if u == t { // true
    fmt.Println("u and t are equal") // Prints: "u and t are equal"
}

No caso acima, 't' e ‘u’ agora são objetos separados (valores de struct).

Como T não contém nenhum tipo de referência (fatias, mapa, canais) como seus campos, t e u acima podem ser modificados sem afetar um ao outro.

fmt.Printf("t.I = %d, u.I = %d\n", t.I, u.I) // t.I = 100, u.I = 1

No entanto, se T contiver um tipo de referência, por exemplo:

type T struct {
    I  int
    S  string
    xs []int // a slice is a reference type
}

Em seguida, uma cópia simples por atribuição copiaria o valor do campo do tipo fatia também para o novo objeto. Isso resultaria em dois objetos diferentes referindo-se ao mesmo objeto de fatia.

// initialize a struct
t := T{I: 1, S: "one", xs: []int{1, 2, 3}}

// make struct copy
u := t // u has its field values equal to t

Como u e t se referem à mesma fatia através de seu campo xs, a atualização de um valor na fatia de um objeto refletiria a mudança no outro.

// update a slice field in u
u.xs[1] = 500

fmt.Printf("t.xs = %d, u.xs = %d\n", t.xs, u.xs)
// t.xs = [1 500 3], u.xs = [1 500 3]

Portanto, cuidado extra deve ser tomado para garantir que essa propriedade de tipo de referência não produza um comportamento não intencional.

Para copiar os objetos acima, por exemplo, uma cópia explícita do campo slice pode ser executada:

// explicitly initialize u's slice field
u.xs = make([]int, len(t.xs))
// copy the slice values over from t
copy(u.xs, t.xs)

// updating slice value in u will not affect t
u.xs[1] = 500

fmt.Printf("t.xs = %d, u.xs = %d\n", t.xs, u.xs)
// t.xs = [1 2 3], u.xs = [1 500 3]

Estrutura vazia #

Uma estrutura é uma sequência de elementos nomeados, chamados campos, cada um dos quais tem um nome e um tipo. A estrutura vazia não tem campos, como esta estrutura vazia anônima:

var s struct{}

Ou como este tipo de estrutura vazia chamado:

type T struct{}

O interessante sobre a estrutura vazia é que seu tamanho é zero (experimente The Go Playground):

fmt.Println(unsafe.Sizeof(s))

Isso imprime 0, então a própria estrutura vazia não ocupa memória. por isso é uma boa opção para sair do canal, como (experimente The Go Playground):

package main

import (
    "fmt"
    "time"
)

func main() {
    done := make(chan struct{})
    go func() {
        time.Sleep(1 * time.Second)
        close(done)
    }()

    fmt.Println("Wait...")
    <-done
    fmt.Println("done.")
}
