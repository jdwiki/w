Em Go, a simultaneidade é alcançada através do uso de goroutines, e a comunicação entre goroutines geralmente é feita com canais. No entanto, outros meios de sincronização, como mutexes e grupos de espera, estão disponíveis e devem ser usados ​​sempre que forem mais convenientes que os canais.

Sintaxe #

  • go doWork() // executa a função doWork como uma goroutine
  • ch := make(chan int) // declara um novo canal do tipo int
  • ch <- 1 // enviando em um canal
  • value = <-ch // recebendo de um canal

Goroutines em Go são semelhantes a threads em outros idiomas em termos de uso. Internamente, Go cria um número de threads (especificado por GOMAXPROCS) e então agenda as goroutines para serem executadas nos threads. Devido a esse design, os mecanismos de simultaneidade do Go são muito mais eficientes do que os threads em termos de uso de memória e tempo de inicialização.

Olá Mundo Goroutine #

canal único, goroutine único, uma gravação, uma leitura.

package main

import "fmt"
import "time"

func main() {
    // create new channel of type string
    ch := make(chan string)

    // start new anonymous goroutine
    go func() {
        time.Sleep(time.Second)
        // send "Hello World" to channel
        ch <- "Hello World"
    }()
    // read from channel
    msg, ok := <-ch
    fmt.Printf("msg='%s', ok='%v'\n", msg, ok)
}

Execute no playground

O canal ch é um canal sem buffer ou síncrono.

O time.Sleep está aqui para ilustrar que a função main() vai esperar no canal ch, o que significa que a função literal executada como uma goroutine tem tempo para enviar um valor através desse canal: o operador de recebimento <-ch bloqueará a execução de main(). Caso contrário, a goroutine seria morta quando main() saísse, e não teria tempo de enviar seu valor.

Esperando por goroutines #

Os programas Go terminam quando a função main termina, portanto, é uma prática comum esperar que todas as goroutines terminem. Uma solução comum para isso é usar um objeto sync.WaitGroup.

package main

import (
    "fmt"
    "sync"
)

var wg sync.WaitGroup // 1

func routine(i int) {
    defer wg.Done() // 3
    fmt.Printf("routine %v finished\n", i)
}

func main() {
    wg.Add(10) // 2
    for i := 0; i < 10; i++ {
        go routine(i) // *
    }
    wg.Wait() // 4
    fmt.Println("main finished")
}

Execute o exemplo no playground

Uso de WaitGroup na ordem de execução:

  1. Declaração de variável global. Torná-lo global é a maneira mais fácil de torná-lo visível para todas as funções e métodos.
  2. Aumentando o contador. Isso deve ser feito na goroutine principal porque não há garantia de que uma goroutine recém-iniciada será executada antes de 4 devido ao modelo de memória garanties.
  3. Diminuindo o contador. Isso deve ser feito na saída de uma goroutine. Ao usar uma chamada adiada, garantimos que ela será chamada sempre que a função terminar, não importa como ela termine.
  4. Aguardando o contador chegar a 0. Isso deve ser feito na gorotina principal para evitar que o programa saia antes que todas as gorotinas terminem.

* Os parâmetros são avaliados antes de iniciar uma nova goroutine. Assim, é necessário definir seus valores explicitamente antes de wg.Add(10) para que o código possivelmente em pânico não aumente o contador. Adicionando 10 itens ao WaitGroup, então ele irá esperar por 10 itens antes que wg.Wait retorne o controle de volta para main() goroutine. Aqui, o valor de i é definido no loop for.

Criando goroutines #

Qualquer função pode ser invocada como uma goroutine prefixando sua invocação com a palavra-chave go:

func DoMultiply(x,y int) {
    // Simulate some hard work
    time.Sleep(time.Second * 1)
    fmt.Printf("Result: %d\n", x * y)
}

go DoMultiply(1,2) // first execution, non-blocking
go DoMultiply(3,4) // second execution, also non-blocking

// Results are printed after a single second only, 
// not 2 seconds because they execute concurrently:
// Result: 2
// Result: 12

Observe que o valor de retorno da função é ignorado.

Usando closures com goroutines em loop #

Quando estiver em um loop, a variável de loop (val) no exemplo a seguir é uma única variável que altera o valor à medida que passa pelo loop. Portanto, deve-se fazer o seguinte para realmente passar cada val de valores para a goroutine:

for val := range values {
    go func(val interface{}) {
        fmt.Println(val)
    }(val)
}

Se você fizer apenas vá func(val interface{}) { ... }() sem passar val, então o valor de val será qualquer val quando as goroutines realmente forem executadas.

Outra maneira de obter o mesmo efeito é:

for val := range values {
    val := val
    go func() {
        fmt.Println(val)
    }()
}

O estranho val := val cria uma nova variável em cada iteração, que é então acessada pela goroutine.

Pingue-pongue com duas goroutines #

package main

import (
    "fmt"
    "time"
)

// The pinger prints a ping and waits for a pong
func pinger(pinger <-chan int, ponger chan<- int) {
    for {
        <-pinger
        fmt.Println("ping")
        time.Sleep(time.Second)
        ponger <- 1
    }
}

// The ponger prints a pong and waits for a ping
func ponger(pinger chan<- int, ponger <-chan int) {
    for {
        <-ponger
        fmt.Println("pong")
        time.Sleep(time.Second)
        pinger <- 1
    }
}

func main() {
    ping := make(chan int)
    pong := make(chan int)

    go pinger(ping, pong)
    go ponger(ping, pong)

    // The main goroutine starts the ping/pong by sending into the ping channel
    ping <- 1

    for {
        // Block the main thread until an interrupt
        time.Sleep(time.Second)
    }
}

Parando goroutines #

package main

import (
    "log"
    "sync"
    "time"
)

func main() {
    // The WaitGroup lets the main goroutine wait for all other goroutines
    // to terminate. However, this is no implicit in Go. The WaitGroup must
    // be explicitely incremented prior to the execution of any goroutine 
    // (i.e. before the `go` keyword) and it must be decremented by calling
    // wg.Done() at the end of every goroutine (typically via the `defer` keyword). 
    wg := sync.WaitGroup{}

    // The stop channel is an unbuffered channel that is closed when the main
    // thread wants all other goroutines to terminate (there is no way to 
    // interrupt another goroutine in Go). Each goroutine must multiplex its
    // work with the stop channel to guarantee liveness.
    stopCh := make(chan struct{})


    for i := 0; i < 5; i++ {
        // It is important that the WaitGroup is incremented before we start
        // the goroutine (and not within the goroutine) because the scheduler
        // makes no guarantee that the goroutine starts execution prior to 
        // the main goroutine calling wg.Wait().
        wg.Add(1)
        go func(i int, stopCh <-chan struct{}) {
            // The defer keyword guarantees that the WaitGroup count is 
            // decremented when the goroutine exits.
            defer wg.Done()

            log.Printf("started goroutine %d", i)

            select {
            // Since we never send empty structs on this channel we can 
            // take the return of a receive on the channel to mean that the
            // channel has been closed (recall that receive never blocks on
            // closed channels).   
            case <-stopCh:
                log.Printf("stopped goroutine %d", i)
            }
        }(i, stopCh)
    }

    time.Sleep(time.Second * 5)
    close(stopCh)
    log.Printf("stopping goroutines")
    wg.Wait()
    log.Printf("all goroutines stopped")
}
