Injeção de Dependência (DI) e Inversão de Controle (IoC)

O código-fonte para grandes aplicativos de software geralmente é organizado em várias unidades. A definição de uma unidade normalmente varia de acordo com a linguagem de programação utilizada. Por exemplo, o código escrito em uma linguagem de programação procedural (como C) é organizado em funções ou procedures. Da mesma forma, o código em uma linguagem de programação orientada a objetos (como Java, Scala e C#) é organizado em classes, interfaces e assim por diante. Essas unidades de organização de código podem ser pensadas como unidades individuais que compõem o aplicativo de software geral.

Quando os aplicativos têm várias unidades, interdependências entre essas unidades surgem quando uma unidade precisa usar outras para completar sua funcionalidade. As unidades dependentes podem ser consideradas como ‘consumidores’ e as unidades das quais dependem como ‘provedores’ de funcionalidades específicas.

A abordagem de programação mais fácil é que os consumidores controlem totalmente o fluxo de um aplicativo de software, decidindo quais provedores devem ser instanciados, usados ​​e destruídos em quais pontos da execução geral do aplicativo. Diz-se que os consumidores têm controle total sobre os provedores durante o fluxo de execução, que são ‘dependências’ para os consumidores. Caso os provedores tenham suas próprias dependências, os consumidores podem ter que se preocupar em como os provedores devem ser inicializados (e liberados), tornando o fluxo de controle cada vez mais complicado à medida que o número de unidades no software aumenta. Essa abordagem também aumenta o acoplamento entre as unidades, tornando cada vez mais difícil trocar as unidades individualmente sem se preocupar em quebrar outras partes do software.

Inversão de controle (IoC) é um princípio de design que defende a terceirização de atividades de fluxo de controle, como descoberta de unidades, instanciação e destruição para uma estrutura independente dos consumidores e fornecedores . O princípio subjacente ao IoC é desacoplar consumidores e provedores, liberando as unidades de software de terem que se preocupar em descobrir, instanciar e limpar suas dependências e permitir que as unidades se concentrem em suas próprias funcionalidades. Esse desacoplamento ajuda a manter o software extensível e sustentável.

Injeção de dependência é uma das técnicas para implementar o princípio de inversão de controle pelo qual instâncias de dependências (provedores) são injetadas em uma unidade de software (o consumidor) em vez de o consumidor ter que encontrá-los e instanciá-los.

A estrutura Spring contém um módulo de injeção de dependência em seu núcleo que permite que beans gerenciados por Spring sejam injetados em outros beans gerenciados por Spring como dependências.

Conectando automaticamente uma dependência por meio da configuração Java #

A injeção de construtor por meio da configuração Java também pode utilizar a fiação automática, como:

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo(Bar bar) { return new Foo(bar); }
}

Conectando automaticamente uma dependência por meio da configuração XML #

As dependências podem ser conectadas automaticamente ao usar o recurso de varredura de componentes da estrutura Spring. Para que a fiação automática funcione, a seguinte configuração XML deve ser feita:

<context:annotation-config/>
<context:component-scan base-package="[base package]"/>

onde, base-package é o pacote Java totalmente qualificado dentro do qual o Spring deve executar a varredura de componentes.

Injeção de construtor

As dependências podem ser injetadas através do construtor da classe da seguinte forma:

@Component
class Bar { ... }

@Component
class Foo {
  private Bar bar;

  @Autowired
  public Foo(Bar bar) { this.bar = bar; }
}

Aqui, @Autowired é uma anotação específica do Spring. O Spring também suporta JSR-299 para permitir a portabilidade do aplicativo para outras estruturas de injeção de dependência baseadas em Java. Isso permite que @Autowired seja substituído por @Inject como:

@Component
class Foo {
  private Bar bar;

  @Inject
  public Foo(Bar bar) { this.bar = bar; }
}

Injeção de propriedade

As dependências também podem ser injetadas usando métodos setter da seguinte forma:

@Component
class Foo {
  private Bar bar;

  @Autowired
  public void setBar(Bar bar) { this.bar = bar; }
}

Injeção de campo

A ligação automática também permite inicializar campos diretamente dentro de instâncias de classe, como segue:

@Component
class Foo {
  @Autowired
  private Bar bar;
}

Para as versões 4.1+ do Spring, você pode usar Optional para dependências opcionais.

@Component
class Foo {

    @Autowired
    private Optional<Bar> bar;
}

A mesma abordagem pode ser usada para o construtor DI.

@Component
class Foo {
    private Optional<Bar> bar;

    @Autowired
    Foo(Optional<Bar> bar) {
        this.bar = bar;
    }
}

Injetando uma dependência manualmente através da configuração XML #

Considere as seguintes classes Java:

class Foo {
  private Bar bar;

  public void foo() {
    bar.baz();
  }
}

Como pode ser visto, a classe Foo precisa chamar o método baz em uma instância de outra classe Bar para que seu método foo funcione com sucesso. Bar é dito ser uma dependência para Foo já que Foo não pode funcionar corretamente sem uma instância Bar.

Injeção de construtor

Ao usar a configuração XML para a estrutura Spring para definir beans gerenciados por Spring, um bean do tipo Foo pode ser configurado da seguinte forma:

<bean class="Foo">
  <constructor-arg>
    <bean class="Bar" />
  </constructor-arg>
</bean>

ou, alternativamente (mais detalhado):

<bean id="bar" class="bar" />

<bean class="Foo">
  <constructor-arg ref="bar" />
</bean>

Em ambos os casos, o framework Spring primeiro cria uma instância de Bar e a injeta em uma instância de Foo. Este exemplo assume que a classe Foo tem um construtor que pode receber uma instância Bar como parâmetro, ou seja:

class Foo {
  private Bar bar;

  public Foo(Bar bar) { this.bar = bar; }
}

Este estilo é conhecido como injeção de construtor porque a dependência (instância Bar) está sendo injetada através do construtor da classe.

Injeção de propriedade

Outra opção para injetar a dependência Bar em Foo é:

<bean class="Foo">
  <property name="bar">
    <bean class="Bar" />
  </property>
</bean>

ou, alternativamente (mais detalhado):

<bean id="bar" class="bar" />

<bean class="Foo">
  <property name="bar" ref="bar" />
</bean>

Isso requer que a classe Foo tenha um método setter que aceite uma instância Bar, como:

class Foo {
  private Bar bar;

  public void setBar(Bar bar) { this.bar = bar; }
}

Injetando uma dependência manualmente através da configuração Java #

Os mesmos exemplos mostrados acima com configuração XML podem ser reescritos com configuração Java da seguinte forma.

Injeção de construtor

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() { return new Foo(bar()); }
}

Injeção de propriedade

@Configuration
class AppConfig {
  @Bean
  public Bar bar() { return new Bar(); }

  @Bean
  public Foo foo() {
    Foo foo = new Foo();
    foo.setBar(bar());

    return foo;
  }
}
