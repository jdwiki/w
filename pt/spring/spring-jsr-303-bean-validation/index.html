Spring JSR 303 Bean Validation

O Spring tem suporte para validação de bean JSR303. Podemos usar isso para fazer a validação do bean de entrada. Separe a lógica de validação da lógica de negócios usando JSR303.

Validações baseadas em anotação JSR303 em exemplos Springs #

Adicione qualquer implementação JSR 303 ao seu classpath. Um popular usado é o validador Hibernate do Hibernate.

<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-validator</artifactId>
    <version>4.2.0.Final</version>
</dependency>

Digamos que exista uma API de descanso para criar usuário no sistema

@RequestMapping(value="/registeruser", method=RequestMethod.POST)
public String registerUser(User user);

A amostra json de entrada ficaria como abaixo

{"username" : "abc@abc.com", "password" : "password1", "password2":"password1"}

User.java

public class User {

    private String username;
    private String password;
    private String password2;

    getXXX and setXXX

}

Podemos definir validações JSR 303 na classe de usuário conforme abaixo.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String username;
    
    @NotEmpty
    private String password;
    
    @NotEmpty
    private String password2;

}

Também podemos precisar ter um validador de negócios como senha e senha2 (confirmar senha) são os mesmos, para isso podemos adicionar um validador personalizado conforme abaixo. Escreva uma anotação personalizada para anotar o campo de dados.

@Target({ ElementType.FIELD })
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PasswordValidator.class)
public @interface GoodPassword {
    String message() default "Passwords wont match.";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

Escreva uma classe Validator para aplicar a lógica de validação.

public class PastValidator implements ConstraintValidator<GoodPassword, User> {
    @Override
    public void initialize(GoodPassword annotation) {}
    
    @Override
    public boolean isValid(User user, ConstraintValidatorContext context) {
        return user.getPassword().equals(user.getPassword2());
    }
}

Adicionando esta validação à classe de usuário

@GoodPassword
public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String username;
    
    @NotEmpty
    private String password;
    
    @NotEmpty
    private String password2;
}

@Valid aciona a validação no Spring. BindingResult é um objeto injetado pelo spring que possui lista de erros após validação.

public String registerUser(@Valid User user, BindingResult result);

A anotação JSR 303 possui atributos de mensagem que podem ser usados ​​para fornecer mensagens personalizadas.

@GoodPassword
public class User {

    @NotEmpty(message="Username Cant be empty")
    @Size(min=5, message="Username cant be les than 5 chars")
    @Email(message="Should be in email format")
    private String username;
    
    @NotEmpty(message="Password cant be empty")
    private String password;
    
    @NotEmpty(message="Password2 cant be empty")
    private String password2;

}

Spring JSR 303 Validation - Personalizar mensagens de erro #

Suponha que tenhamos uma classe simples com anotações de validação

public class UserDTO {
    @NotEmpty
    private String name;

    @Min(18)
    private int age;

//getters/setters
}

Um controlador para verificar a validade do UserDTO.

@RestController
public class ValidationController {

    @RequestMapping(value = "/validate", method = RequestMethod.POST)
    public ResponseEntity<String> check(@Valid @RequestBody UserDTO userDTO,
           BindingResult bindingResult) {
        return new ResponseEntity<>("ok" , HttpStatus.OK);
    }
}

E um teste.

@Test
public void testValid() throws Exception {
    TestRestTemplate template = new TestRestTemplate();
    String url = base + contextPath + "/validate";
    Map<String, Object> params = new HashMap<>();
    params.put("name", "");
    params.put("age", "10");

    MultiValueMap<String, String> headers = new LinkedMultiValueMap<>();
    headers.add("Content-Type", "application/json");

    HttpEntity<Map<String, Object>> request = new HttpEntity<>(params, headers);
    String res = template.postForObject(url, request, String.class);

    assertThat(res, equalTo("ok"));
}

Tanto o nome quanto a idade são inválidos, portanto no BindingResult temos dois erros de validação. Cada um tem matriz de códigos.

Códigos para cheque mínimo

0 = "Min.userDTO.age"
1 = "Min.age"
2 = "Min.int"
3 = "Min"

E para verificação NotEmpty

0 = "NotEmpty.userDTO.name"
1 = "NotEmpty.name"
2 = "NotEmpty.java.lang.String"
3 = "NotEmpty"

Vamos adicionar um arquivo custom.properties para substituir as mensagens padrão.

@SpringBootApplication
@Configuration
public class DemoApplication {

    @Bean(name = "messageSource")
    public MessageSource messageSource() {
        ReloadableResourceBundleMessageSource bean = new ReloadableResourceBundleMessageSource();
        bean.setBasename("classpath:custom");
        bean.setDefaultEncoding("UTF-8");
        return bean;
    }

    @Bean(name = "validator")
    public LocalValidatorFactoryBean validator() {
        LocalValidatorFactoryBean bean = new LocalValidatorFactoryBean();
        bean.setValidationMessageSource(messageSource());
        return bean;
    }

    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

Se adicionarmos ao arquivo custom.properties a linha

NotEmpty=The field must not be empty!

O novo valor é mostrado para o erro. Para resolver a mensagem, o validador examina os códigos desde o início para encontrar as mensagens corretas.

Assim, quando definimos a chave NotEmpty no arquivo .properties para todos os casos em que a anotação @NotEmpty é usada, nossa mensagem é aplicada.

Se definirmos uma mensagem

Min.int=Some custom message here.

Todas as anotações em que aplicamos a verificação mínima para valores inteiros usam a mensagem recém-definida.

A mesma lógica pode ser aplicada se precisarmos localizar as mensagens de erro de validação.

@Uso válido para validar POJOs aninhados #

Suponha que temos um usuário da classe POJO que precisamos validar.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String email;
}

e um método de controlador para validar a instância do usuário

public String registerUser(@Valid User user, BindingResult result);

Vamos estender o usuário com um endereço POJO aninhado que também precisamos validar.

public class Address {

    @NotEmpty
    @Size(min=2, max=3)
    private String countryCode;
}

Basta adicionar a anotação @Valid no campo de endereço para executar a validação de POJOs aninhados.

public class User {

    @NotEmpty
    @Size(min=5)
    @Email
    private String email;

    @Valid
    private Address address;
}
