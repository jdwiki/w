Obtendo um SqlRowSet de SimpleJdbcCall

  • Criação SimpleJdbcCall
  • Bancos de dados Oracle

Nesta página

  • Criação SimpleJdbcCall
  • Bancos de dados Oracle

Isso descreve como obter diretamente um SqlRowSet usando SimpleJdbcCall com um procedimento armazenado em seu banco de dados que tem um parâmetro de saída de cursor,

Estou trabalhando com um banco de dados Oracle, tentei criar um exemplo que deve funcionar para outros bancos de dados, meu exemplo Oracle detalha problemas com o Oracle.

Criação SimpleJdbcCall #

Normalmente, você desejará criar suas SimpleJdbcCalls em um serviço.

Este exemplo assume que seu procedimento tem um único parâmetro de saída que é um cursor; você precisará ajustar seus declareParameters para corresponder ao seu procedimento.

@Service
public class MyService() {

@Autowired
    private DataSource dataSource;
    
    // Autowire your configuration, for example
    @Value("${db.procedure.schema}")
    String schema;
    
    private SimpleJdbcCall myProcCall;
    
    // create SimpleJdbcCall after properties are configured
    @PostConstruct
    void initialize() {
        this.myProcCall = new SimpleJdbcCall(dataSource)
                        .withProcedureName("my_procedure_name")
                        .withCatalogName("my_package")
                        .withSchemaName(schema)
                        .declareParameters(new SqlOutParameter(
                            "out_param_name",
                            Types.REF_CURSOR, 
                            new SqlRowSetResultSetExtractor()));
    }

    public SqlRowSet myProc() {
        Map<String, Object> out = this.myProcCall.execute();
        return (SqlRowSet) out.get("out_param_name");
    }

}

Há muitas opções que você pode usar aqui:

  • withoutProcedureColumnMetaDataAccess() necessário se você tiver nomes de procedimento sobrecarregados ou simplesmente não quiser que SimpleJdbcCall seja validado no banco de dados.
  • withReturnValue() se o procedimento tiver um valor de retorno. O primeiro valor dado a declareParameters define o valor de retorno. Além disso, se seu procedimento for uma função, use withFunctionName e executeFunction ao executar.
  • withNamedBinding() se você deseja fornecer argumentos usando nomes em vez de posição.
  • useInParameterNames() define a ordem dos argumentos. Acho que isso pode ser necessário se você passar seus argumentos como uma lista em vez de um mapa de nome de argumento para valor. Embora possa ser necessário apenas se você usar withoutProcedureColumnMetaDataAccess()

Bancos de dados Oracle #

Há uma série de problemas com a Oracle. Veja como resolvê-los.

Supondo que o parâmetro de saída do seu procedimento seja ref cursor, você receberá essa exceção.

java.sql.SQLException: Tipo de coluna inválido: 2012

Então mude Types.REF_CURSOR para OracleTypes.CURSOR em simpleJdbcCall.declareParameters()

Suporte a OracleTypes

Talvez você só precise fazer isso se tiver determinados tipos de coluna em seus dados.

O próximo problema que encontrei foi que tipos proprietários como oracle.sql.TIMESTAMPTZ causaram esse erro em SqlRowSetResultSetExtractor:

Tipo SQL inválido para coluna; exceção aninhada é java.sql.SQLException: tipo SQL inválido para coluna

Portanto, precisamos criar um ResultSetExtractor que suporte os tipos Oracle. Vou explicar o motivo da senha após este código.

package com.boost.oracle;

import oracle.jdbc.rowset.OracleCachedRowSet;
import org.springframework.dao.DataAccessException;
import org.springframework.jdbc.core.ResultSetExtractor;
import org.springframework.jdbc.support.rowset.ResultSetWrappingSqlRowSet;
import org.springframework.jdbc.support.rowset.SqlRowSet;

import java.sql.ResultSet;
import java.sql.SQLException;

/**
 * OracleTypes can cause {@link org.springframework.jdbc.core.SqlRowSetResultSetExtractor}
 * to fail due to a Oracle SQL type that is not in the standard {@link java.sql.Types}.
 *
 * Also, types such as {@link oracle.sql.TIMESTAMPTZ} require a Connection when processing
 * the ResultSet; {@link OracleCachedRowSet#getConnectionInternal()} requires a JNDI
 * DataSource name or the username and password to be set.
 *
 * For now I decided to just set the password since changing SpringBoot to a JNDI DataSource
 * configuration is a bit complicated.
 *
 * Created by Arlo White on 2/23/17.
 */
public class OracleSqlRowSetResultSetExtractor implements ResultSetExtractor<SqlRowSet> {

    private String oraclePassword;

    public OracleSqlRowSetResultSetExtractor(String oraclePassword) {
        this.oraclePassword = oraclePassword;
    }

    @Override
    public SqlRowSet extractData(ResultSet rs) throws SQLException, DataAccessException {
        OracleCachedRowSet cachedRowSet = new OracleCachedRowSet();
        // allows getConnectionInternal to get a Connection for TIMESTAMPTZ
        cachedRowSet.setPassword(oraclePassword);
        cachedRowSet.populate(rs);
        return new ResultSetWrappingSqlRowSet(cachedRowSet);
    }

}

Certos tipos de Oracle exigem uma conexão para obter o valor da coluna de um ResultSet. TIMESTAMPTZ é um desses tipos. Então, quando rowSet.getTimestamp(colIndex) for chamado, você receberá esta exceção:

Causado por: java.sql.SQLException: Um ou mais dos Propriedades de RowSet não definidas em oracle.jdbc.rowset.OracleCachedRowSet.getConnectionInternal(OracleCachedRowSet.java:560) em oracle.jdbc.rowset.OracleCachedRowSet.getTimestamp(OracleCachedRowSet.java:3717) em org.springframework.jdbc.support.rowset.ResultSetWrappingSqlRowSet.getTimestamp

Se você se aprofundar nesse código, verá que o OracleCachedRowSet precisa da senha ou de um nome JNDI DataSource para obter uma conexão. Se você preferir a pesquisa JNDI, apenas verifique se OracleCachedRowSet tem DataSourceName definido.

Então, no meu serviço, eu faço o Autowire na senha e declaro o parâmetro de saída assim:

new SqlOutParameter("cursor_param_name", OracleTypes.CURSOR, new OracleSqlRowSetResultSetExtractor(oraclePassword))
