Definição

A notação Big-O é, em sua essência, uma notação matemática, usada para comparar a taxa de convergência de funções. Sejam n -> f(n) e n -> g(n) funções definidas sobre os números naturais. Então dizemos que f = O(g) se e somente se f(n)/g(n) é limitado quando n se aproxima do infinito. Em outras palavras, f = O(g) se e somente se existe uma constante A, tal que para todo n, f(n)/g(n) <= A.

Na verdade, o escopo da notação Big-O é um pouco mais amplo em matemática, mas, por simplicidade, reduzi-o ao que é usado na análise de complexidade de algoritmos: funções definidas nos naturais, que têm valores diferentes de zero, e o caso de n crescente ao infinito.

O que isto significa ?

Vamos pegar o caso de f(n) = 100n^2 + 10n + 1 e g(n) = n^2. É bastante claro que ambas as funções tendem ao infinito como n tende ao infinito. Mas às vezes saber o limite não é suficiente, e também queremos saber a velocidade na qual as funções se aproximam de seu limite. Noções como Big-O ajudam a comparar e classificar funções por sua velocidade de convergência.

Vamos descobrir se f = O(g) aplicando a definição. Temos f(n)/g(n) = 100 + 10/n + 1/n^2. Como 10/n é 10 quando n é 1 e é decrescente, e como 1/n^2 é 1 quando n é 1 e também é decrescente, temos ̀f(n)/g(n) < = 100 + 10 + 1 = 111. A definição é satisfeita porque encontramos um limite de f(n)/g(n) (111) e então f = O(g) (dizemos que f é um Big-O de n^2 ).

Isso significa que f tende ao infinito aproximadamente na mesma velocidade que g. Agora isso pode parecer uma coisa estranha de se dizer, porque o que descobrimos é que f é no máximo 111 vezes maior que g, ou seja, quando g cresce em 1, f cresce no máximo 111. Pode parecer que crescer 111 vezes mais rápido não é “aproximadamente a mesma velocidade”. E, de fato, a notação Big-O não é uma maneira muito precisa de classificar a velocidade de convergência da função, e é por isso que em matemática usamos a relação de equivalência quando queremos uma estimativa precisa da velocidade. Mas para fins de separação de algoritmos em grandes classes de velocidade, o Big-O é suficiente. Não precisamos separar funções que crescem um número fixo de vezes mais rápido que umas as outras, mas apenas funções que crescem infinitamente mais rápido que umas as outras. Por exemplo, se tomarmos h(n) = n^2*log(n), vemos que h(n)/g(n) = log(n) que tende ao infinito com n então h é * not* O(n^2), porque h cresce infinitamente mais rápido que n^2.

Agora eu preciso fazer uma observação lateral: você deve ter notado que se f = O(g) e g = O(h), então f = O(h). Por exemplo, em nosso caso, temos f = O(n^3), e f = O(n^4)… Na análise de complexidade de algoritmos, frequentemente dizemos f = O(g) para significa que f = O(g) e g = O(f), que pode ser entendido como “g é o menor Big-O para f”. Em matemática dizemos que tais funções são Big-Thetas umas das outras.

Como isso é usado ?

Ao comparar o desempenho do algoritmo, estamos interessados ​​no número de operações que um algoritmo realiza. Isso é chamado de complexidade de tempo. Neste modelo, consideramos que cada operação básica (adição, multiplicação, comparação, atribuição, etc.) leva um tempo fixo e contamos o número de tais operações. Geralmente podemos expressar esse número em função do tamanho da entrada, que chamamos de n. E, infelizmente, esse número geralmente cresce até o infinito com n (se não, dizemos que o algoritmo é O(1)). Separamos nossos algoritmos em classes de grande velocidade definidas por Big-O : quando falamos de um algoritmo “O(n^2)”, queremos dizer que o número de operações que ele executa, expresso em função de n, é um O( n^2). Isso diz que nosso algoritmo é aproximadamente tão rápido quanto um algoritmo que faria um número de operações igual ao quadrado do tamanho de sua entrada, ou mais rápido. A parte “ou mais rápido” está lá porque eu usei Big-O em vez de Big-Theta, mas geralmente as pessoas dizem Big-O para significar Big-Theta.

Ao contar as operações, geralmente consideramos o pior caso: por exemplo, se temos um loop que pode ser executado no máximo n vezes e contém 5 operações, o número de operações que contamos é 5n. Também é possível considerar a complexidade média do caso.

Nota rápida: um algoritmo rápido é aquele que executa poucas operações, então se o número de operações crescer até o infinito mais rápido, então o algoritmo é mais lento: O(n) é melhor que O(n^2).

Às vezes também estamos interessados ​​na complexidade do espaço do nosso algoritmo. Para isso consideramos o número de bytes na memória ocupados pelo algoritmo em função do tamanho da entrada, e usamos Big-O da mesma forma.

Um loop simples #

A função a seguir encontra o elemento máximo em uma matriz:

int find_max(const int *array, size_t len) {
    int max = INT_MIN;
    for (size_t i = 0; i < len; i++) {
        if (max < array[i]) {
            max = array[i];
        }
    }
    return max;
}

O tamanho da entrada é o tamanho do array, que chamei de len no código.

Vamos contar as operações.

int max = INT_MIN;
size_t i = 0;

Essas duas atribuições são feitas apenas uma vez, então são 2 operações. As operações em loop são:

if (max < array[i])
i++;
max = array[i]

Como existem 3 operações no loop, e o loop é feito n vezes, adicionamos 3n às nossas 2 operações já existentes para obter 3n + 2. Portanto, nossa função usa operações 3n + 2 para encontrar o máximo (sua complexidade é 3n + 2). Este é um polinômio onde o termo de crescimento mais rápido é um fator de n, então é O(n).

Você provavelmente já percebeu que “operação” não está muito bem definida. Por exemplo, eu disse que if (max < array[i]) era uma operação, mas dependendo da arquitetura, esta declaração pode compilar, por exemplo, três instruções: uma leitura de memória, uma comparação e uma ramificação. Também considerei todas as operações iguais, embora, por exemplo, as operações de memória sejam mais lentas do que as outras, e seu desempenho varie muito devido, por exemplo, aos efeitos de cache. Eu também ignorei completamente a declaração de retorno, o fato de que um quadro será criado para a função, etc. No final, não importa para a análise de complexidade, porque qualquer que seja a maneira que eu escolha para contar as operações, isso só mudará o coeficiente do fator n e da constante, então o resultado ainda será O(n). A complexidade mostra como o algoritmo é dimensionado com o tamanho da entrada, mas não é o único aspecto do desempenho!

Um loop aninhado #

A função a seguir verifica se um array tem alguma duplicata pegando cada elemento, então iterando por todo o array para ver se o elemento está lá

_Bool contains_duplicates(const int *array, size_t len) {
    for (int i = 0; i < len - 1; i++) {
        for (int j = 0; j < len; j++) {
            if (i != j && array[i] == array[j]) {
                return 1;
            }
        }
    }
    return 0;
}

O loop interno executa a cada iteração um número de operações que é constante com n. O loop externo também faz algumas operações constantes e executa o loop interno n vezes. O próprio loop externo é executado n vezes. Portanto, as operações dentro do loop interno são executadas n^2 vezes, as operações no loop externo são executadas n vezes e a atribuição a i é feita uma vez. Assim, a complexidade será algo como an^2 + bn + c, e como o termo mais alto é n^2, a notação O é O(n^2).

Como você deve ter notado, podemos melhorar o algoritmo evitando fazer as mesmas comparações várias vezes. Podemos começar de i + 1 no loop interno, porque todos os elementos antes dele já terão sido verificados em relação a todos os elementos do array, incluindo aquele no índice i + 1. Isso nos permite descartar a verificação i == j.

_Bool faster_contains_duplicates(const int *array, size_t len) {
    for (int i = 0; i < len - 1; i++) {
        for (int j = i + 1; j < len; j++) {
            if (array[i] == array[j]) {
                return 1;
            }
        }
    }
    return 0;
}

Obviamente, esta segunda versão faz menos operações e por isso é mais eficiente. Como isso se traduz em notação Big-O? Bem, agora o corpo do loop interno é executado 1 + 2 + ... + n - 1 = n(n-1)/2 vezes. Este é ainda um polinômio de segundo grau e, portanto, ainda é apenas O(n^2). Reduzimos claramente a complexidade, pois dividimos aproximadamente por 2 o número de operações que estamos fazendo, mas ainda estamos na mesma classe de complexidade definida pelo Big-O. Para diminuir a complexidade para uma classe mais baixa, precisaríamos dividir o número de operações por algo que tende ao infinito com n.

Um exemplo de O(log n) #

Introdução #

Considere o seguinte problema:

L é uma lista ordenada contendo n inteiros com sinal (n sendo grande o suficiente), por exemplo [-5, -2, -1, 0, 1, 2, 4] (aqui, n tem um valor de 7). Se L é conhecido por conter o inteiro 0, como você pode encontrar o índice de 0?

Abordagem ingênua #

A primeira coisa que vem à mente é apenas ler cada índice até que 0 seja encontrado. No pior caso, o número de operações é n, então a complexidade é O(n).

Isso funciona bem para pequenos valores de n, mas existe uma maneira mais eficiente?

Dicotomia #

Considere o seguinte algoritmo (Python3):

a = 0
b = n-1
while True:
  h = (a+b)//2 ## // is the integer division, so h is an integer
  if L[h] == 0:
    return h
  elif L[h] > 0:
    b = h
  elif L[h] < 0:
    a = h

a e b são os índices entre os quais 0 deve ser encontrado. Cada vez que entramos no loop, usamos um índice entre a e b e o usamos para restringir a área a ser pesquisada.

Na pior das hipóteses, temos que esperar até que a e b sejam iguais. Mas quantas operações são necessárias? Não n, porque cada vez que entramos no loop, dividimos a distância entre a e b por cerca de dois. Em vez disso, a complexidade é O(log n).

Explicação #

Nota: Quando escrevemos “log”, queremos dizer o logaritmo binário, ou log base 2 (que escreveremos “log_2”). Como O(log_2 n) = O(log n) (você pode fazer as contas), usaremos “log” em vez de “log_2”.

Vamos chamar x o número de operações: sabemos que 1 = n / (2^x).

Então 2^x = n, então x = log n

Conclusão #

Ao se deparar com divisões sucessivas (seja por dois ou por qualquer número), lembre-se que a complexidade é logarítmica.

O(log n) tipos de algoritmos #

Digamos que temos um problema de tamanho n. Agora, para cada etapa do nosso algoritmo (que precisamos escrever), nosso problema original se torna metade de seu tamanho anterior (n/2).

Assim, a cada passo, nosso problema se torna metade.

Etapa Problema
1 n/2
2 n/4
3 n/8
4 n/16

Quando o espaço do problema é reduzido (ou seja, resolvido completamente), ele não pode ser reduzido mais (n se torna igual a 1) após sair da condição de verificação.

  1. Digamos que na k-ésima etapa ou número de operações:

    problem-size = 1

  2. Mas sabemos que na k-ésima etapa, o tamanho do nosso problema deve ser:

tamanho do problema = n/2k

  1. De 1 e 2:

    n/2k = 1 or

    n = 2k

  2. Faça o log em ambos os lados

    loge n = k loge2

    or

    k = loge n / loge 2

  3. Usando a fórmula logx m / logx n = logn m

k = log2 n

ou simplesmente k = log n

Agora sabemos que nosso algoritmo pode executar no máximo até log n, portanto a complexidade do tempo vem como O(log n)

Um exemplo muito simples no código para suportar o texto acima é:

for(int i=1; i<=n; i=i*2)
{
    // perform some operation
}

Então, agora, se alguém perguntar se n é 256 quantos passos esse loop (ou qualquer outro algoritmo que reduza o tamanho do problema pela metade) será executado, você poderá calcular facilmente.

k = log2 256

k = log2 2 8 ( => logaa = 1)

k = 8

Outro exemplo muito bom para um caso semelhante é o Algoritmo de Pesquisa Binária.

int bSearch(int arr[],int size,int item){
 int low=0;
 int high=size-1;

 while(low<=high){                 
     mid=low+(high-low)/2;                 
     if(arr[mid]==item)                        
         return mid;                 
     else if(arr[mid]<item)                         
         low=mid+1;                
     else  high=mid-1;      
     }  
  return –1;// Unsuccessful result
}
