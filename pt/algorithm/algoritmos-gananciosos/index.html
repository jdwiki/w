Um algoritmo ganancioso é um algoritmo em que em cada etapa escolhemos a opção mais benéfica em cada etapa sem olhar para o futuro. A escolha depende apenas do lucro atual.

A abordagem gananciosa geralmente é uma boa abordagem quando cada lucro pode ser obtido em todas as etapas, portanto, nenhuma escolha bloqueia outra.

Codificação Huffman #

Código Huffman é um tipo específico de código de prefixo ideal que é comumente usado para compactação de dados sem perdas. Ele compacta os dados de forma muito eficaz, economizando de 20% a 90% de memória, dependendo das características dos dados que estão sendo compactados. Consideramos os dados como uma sequência de caracteres. O algoritmo guloso de Huffman usa uma tabela que fornece a frequência com que cada caractere ocorre (ou seja, sua frequência) para construir uma maneira ideal de representar cada caractere como uma string binária. O código Huffman foi proposto por David A. Huffman em 1951.

Suponha que temos um arquivo de dados de 100.000 caracteres que desejamos armazenar de forma compacta. Assumimos que existem apenas 6 caracteres diferentes nesse arquivo. A frequência dos caracteres é dada por:

+------------------------+-----+-----+-----+-----+-----+-----+
|        Character       |  a  |  b  |  c  |  d  |  e  |  f  |
+------------------------+-----+-----+-----+-----+-----+-----+
|Frequency (in thousands)|  45 |  13 |  12 |  16 |  9  |  5  |
+------------------------+-----+-----+-----+-----+-----+-----+

Temos muitas opções de como representar tal arquivo de informação. Aqui, consideramos o problema de projetar um Código de Caractere Binário no qual cada caractere é representado por uma string binária exclusiva, que chamamos de palavra de código.

A árvore construída nos fornecerá:

+------------------------+-----+-----+-----+-----+-----+-----+
|        Character       |  a  |  b  |  c  |  d  |  e  |  f  |
+------------------------+-----+-----+-----+-----+-----+-----+
| Fixed-length Codeword  | 000 | 001 | 010 | 011 | 100 | 101 |
+------------------------+-----+-----+-----+-----+-----+-----+
|Variable-length Codeword|  0  | 101 | 100 | 111 | 1101| 1100|
+------------------------+-----+-----+-----+-----+-----+-----+

Se usarmos um código de comprimento fixo, precisamos de três bits para representar 6 caracteres. Este método requer 300.000 bits para codificar o arquivo inteiro. Agora a questão é, podemos fazer melhor?

Um código de comprimento variável pode ter um desempenho consideravelmente melhor do que um código de comprimento fixo, fornecendo palavras de código curtas de caracteres frequentes e palavras de código longas de caracteres infrequentes. Este código requer: (45 X 1 + 13 X 3 + 12 X 3 + 16 X 3 + 9 X 4 + 5 X 4) X 1000 = 224000 bits para representar o arquivo, o que economiza aproximadamente 25% de memória.

Uma coisa a ser lembrada, consideramos aqui apenas códigos em que nenhuma palavra-código é também um prefixo de alguma outra palavra-código. Eles são chamados de códigos de prefixo. Para codificação de comprimento variável, codificamos o arquivo de 3 caracteres abc como 0.101.100 = 0101100, onde “.” denota a concatenação.

Os códigos de prefixo são desejáveis ​​porque simplificam a decodificação. Como nenhuma palavra de código é um prefixo de qualquer outra, a palavra de código que inicia um arquivo codificado não é ambígua. Podemos simplesmente identificar a palavra de código inicial, traduzi-la de volta para o caractere original e repetir o processo de decodificação no restante do arquivo codificado. Por exemplo, 001011101 analisa exclusivamente como 0.0.101.1101, que decodifica para aabe. Em suma, todas as combinações de representações binárias são únicas. Digamos, por exemplo, se uma letra for denotada por 110, nenhuma outra letra será denotada por 1101 ou 1100. Isso ocorre porque você pode ficar confuso sobre selecionar 110 ou continuar concatenando o próximo bit e selecione aquele.

Técnica de compressão:

A técnica funciona criando uma árvore binária de nós. Estes podem ser armazenados em um array regular, cujo tamanho depende do número de símbolos, n. Um nó pode ser um nó folha ou um nó interno. Inicialmente todos os nós são nós folha, que contêm o próprio símbolo, sua frequência e, opcionalmente, um link para seus nós filhos. Como convenção, o bit ‘0’ representa o filho esquerdo e o bit ‘1’ representa o filho direito. A fila de prioridade é usada para armazenar os nós, que fornecem o nó com a frequência mais baixa quando estourado. O processo é descrito abaixo:

  1. Crie um nó folha para cada símbolo e adicione-o à fila de prioridade.
  2. Enquanto houver mais de um nó na fila:
    1. Remove the two nodes of highest priority from the queue.
    2. Create a new internal node with these two nodes as children and with frequency equal to the sum of the two nodes’ frequency.
    3. Add the new node to the queue.
  3. O nó restante é o nó raiz e a árvore de Huffman está completa.

Para o nosso exemplo:

O pseudo-código se parece com:

Procedure Huffman(C):     // C is the set of n characters and related information
n = C.size
Q = priority_queue()
for i = 1 to n
    n = node(C[i])
    Q.push(n)
end for
while Q.size() is not equal to 1
    Z = new node()
    Z.left = x = Q.pop
    Z.right = y = Q.pop
    Z.frequency = x.frequency + y.frequency
    Q.push(Z)
end while
Return Q

Apesar de tempo linear dado entrada ordenada, em casos gerais de entrada arbitrária, o uso deste algoritmo requer pré-ordenação. Assim, como a ordenação leva tempo O(nlogn) em casos gerais, ambos os métodos têm a mesma complexidade.

Como n aqui é o número de símbolos no alfabeto, que normalmente é um número muito pequeno (comparado ao comprimento da mensagem a ser codificada), a complexidade do tempo não é muito importante na escolha desse algoritmo.

Técnica de descompressão:

O processo de descompressão é simplesmente uma questão de traduzir o fluxo de códigos de prefixo em valor de byte individual, geralmente percorrendo a árvore de Huffman nó por nó à medida que cada bit é lido do fluxo de entrada. Alcançar um nó folha necessariamente encerra a busca por esse valor de byte específico. O valor da folha representa o caractere desejado. Normalmente, a Árvore de Huffman é construída usando dados ajustados estatisticamente em cada ciclo de compressão, portanto, a reconstrução é bastante simples. Caso contrário, as informações para reconstruir a árvore devem ser enviadas separadamente. O pseudo-código:

Procedure HuffmanDecompression(root, S):   // root represents the root of Huffman Tree
n := S.length                              // S refers to bit-stream to be decompressed
for i := 1 to n
    current = root
    while current.left != NULL and current.right != NULL
        if S[i] is equal to '0'
            current := current.left
        else
            current := current.right
        endif
        i := i+1
    endwhile
    print current.symbol
endfor

Explicação gananciosa: A codificação de Huffman analisa a ocorrência de cada caractere e o armazena como uma string binária de maneira otimizada. A ideia é atribuir códigos de comprimento variável aos caracteres de entrada de entrada, o comprimento dos códigos atribuídos é baseado nas frequências dos caracteres correspondentes. Criamos uma árvore binária e operamos nela de forma ascendente para que os dois caracteres menos frequentes estejam o mais longe possível da raiz. Desta forma, o caractere mais frequente recebe o menor código e o caractere menos frequente recebe o maior código.

Problema de seleção de atividade #

O problema #

Você tem um conjunto de coisas para fazer (atividades). Cada atividade tem uma hora de início e uma hora de término. Você não tem permissão para realizar mais de uma atividade por vez. Sua tarefa é encontrar uma maneira de realizar o número máximo de atividades.

Por exemplo, suponha que você tenha uma seleção de classes para escolher.

Nº da atividade hora de início hora final
1 10h20 11h00
2 10h30 11h30
3 11h00 12h00
4 10h00 11h30
5 9h00 11h00

Lembre-se, você não pode fazer duas aulas ao mesmo tempo. Isso significa que você não pode fazer as aulas 1 e 2 porque elas compartilham um horário comum das 10h30 às 11h00. No entanto, você pode fazer as aulas 1 e 3 porque elas não compartilham um horário comum. Portanto, sua tarefa é ter o maior número possível de aulas sem sobreposição. Como você pode fazer isso?

Análise #

Vamos pensar na solução por uma abordagem gananciosa. Em primeiro lugar, escolhemos aleatoriamente alguma abordagem e verificamos se funcionará ou não.

  • classifique a atividade por hora de início, o que significa que qual atividade começar primeiro nós a levaremos primeiro. em seguida, pegue o primeiro ao último da lista classificada e verifique se se cruzará com a atividade anterior ou não. Se a atividade atual não cruzar com a atividade realizada anteriormente, realizaremos a atividade, caso contrário, não a executaremos. essa abordagem funcionará para alguns casos como
Nº da atividade hora de início hora final
1 11h00 13h30
2 11h30 12h00
3 13h30 14h00
4 10h00 11h00

a ordem de classificação será 4–>1–>2–>3 . A atividade 4–> 1–> 3 será executada e a atividade 2 será ignorada. no máximo 3 atividades serão realizadas. Funciona para este tipo de casos. mas falhará em alguns casos. Vamos aplicar esta abordagem para o caso

Nº da atividade hora de início hora final
1 11h00 13h30
2 11h30 12h00
3 13h30 14h00
4 10h00 15h00

A ordem de classificação será 4–>1–>2–>3 e somente a atividade 4 será executada, mas a resposta pode ser a atividade 1–>3 ou 2–>3 será executada. Portanto, nossa abordagem não funcionará para o caso acima. Vamos tentar outra abordagem

  • Classifique a atividade por tempo de duração, ou seja, execute a atividade mais curta primeiro. que pode resolver o problema anterior. Embora o problema não esteja completamente resolvido. Ainda existem alguns casos que podem falhar na solução. aplicar esta abordagem no caso abaixo.
Nº da atividade hora de início hora final
1 6h00 11h40
2 11h30 12h00
3 23h40 14h00

se classificarmos a atividade por duração de tempo, a ordem de classificação será 2–> 3 —>1 . e se realizarmos a atividade nº 2 primeiro, nenhuma outra atividade poderá ser realizada. Mas a resposta será realizar a atividade 1 e depois realizar a 3 . Portanto, podemos realizar no máximo 2 atividades. Portanto, isso não pode ser uma solução para esse problema. Devemos tentar uma abordagem diferente.

A solução #

  • Classifique a atividade por hora de término, o que significa que a atividade termina primeiro. o algoritmo é dado abaixo
  1. Classifique as atividades por seus horários de término.
  2. Se a atividade a ser desempenhada não compartilhar um horário comum com as atividades que realizava anteriormente, realize a atividade.

Vamos analisar o primeiro exemplo

Nº da atividade hora de início hora final
1 10h20 11h00
2 10h30 11h30
3 11h00 12h00
4 10h00 11h30
5 9h00 11h00

classifique a atividade por seus horários de término, então a ordem de classificação será 1–>5–>2–>4–>3.. a resposta é 1–>3 essas duas atividades serão realizadas. e essa é a resposta. aqui está o código sudo.

  1. classificar: atividades
  2. execute a primeira atividade da lista ordenada de atividades.
  3. Definir : Current_activity := primeira atividade
  4. set: end_time := end_time da atividade atual
  5. ir para a próxima atividade se existir, se não existir terminar .
  6. se start_time da atividade atual <= end_time : execute a atividade e vá para 4
  7. senão: chegou a 5.

Problema de mudança #

Dado um sistema monetário, é possível dar uma quantidade de moedas e como encontrar um conjunto mínimo de moedas correspondente a essa quantidade.

Sistemas monetários canônicos. Para alguns sistemas monetários, como os que usamos na vida real, a solução “intuitiva” funciona perfeitamente. Por exemplo, se as diferentes moedas e notas de euro (excluindo centavos) forem de 1€, 2€, 5€, 10€, dar a moeda ou nota mais alta até atingirmos o valor e repetir este procedimento levará ao conjunto mínimo de moedas .

Podemos fazer isso recursivamente com OCaml :

(* assuming the money system is sorted in decreasing order *)
let change_make money_system amount =
  let rec loop given amount =
    if amount = 0 then given
    else 
      (* we find the first value smaller or equal to the remaining amount *)
      let coin = List.find ((>=) amount) money_system in
      loop (coin::given) (amount - coin)
  in loop [] amount

Esses sistemas são feitos para que a mudança seja fácil. O problema fica mais difícil quando se trata de sistema monetário arbitrário.

Estojo geral. Como dar 99€ com moedas de 10€, 7€ e 5€? Aqui, dar moedas de 10€ até ficarmos com 9€ não leva obviamente a nenhuma solução. Pior do que isso pode não existir uma solução. Este problema é de fato np-hard, mas existem soluções aceitáveis ​​que misturam ganância e memoização. A ideia é explorar todas as possibilidades e escolher aquela com o número mínimo de moedas.

Para dar uma quantia X > 0, escolhemos uma peça P no sistema monetário e resolvemos o subproblema correspondente a X-P. Tentamos isso para todas as partes do sistema. A solução, se existir, é então o menor caminho que leva a 0.

Aqui uma função recursiva OCaml correspondente a este método. Ele retorna Nenhum, se não existir nenhuma solução.

(* option utilities *)
let optmin x y =
  match x,y with
  | None,a | a,None -> a
  | Some x, Some y-> Some (min x y)

let optsucc = function
  | Some x -> Some (x+1)
  | None -> None

(* Change-making problem*)
let change_make money_system amount =
  let rec loop n =
    let onepiece acc piece =
      match n - piece with
      | 0 -> (*problem solved with one coin*) 
             Some 1
      | x -> if x < 0 then 
               (*we don't reach 0, we discard this solution*)
               None
             else
               (*we search the smallest path different to None with the remaining pieces*) 
               optmin (optsucc (loop x)) acc
    in
    (*we call onepiece forall the pieces*)
    List.fold_left onepiece None money_system
  in loop amount

Nota: Podemos observar que este procedimento pode calcular várias vezes a mudança definida para o mesmo valor. Na prática, usar memoização para evitar essas repetições leva a resultados mais rápidos (muito mais rápidos).

Problema contínuo da mochila #

Dados os itens como (valor, peso) precisamos colocá-los em uma mochila (contêiner) de capacidade k. Observação! Podemos quebrar itens para maximizar o valor!

Exemplo de entrada:

values[] = [1, 4, 5, 2, 10]
weights[] = [3, 2, 1, 2, 4]
k = 8

Saída esperada:

maximumValueOfItemsInK = 20;

Algoritmo:

1) Sort values and weights by value/weight.
   values[] = [5, 10, 4, 2, 1]
   weights[] = [1, 4, 2, 2, 3]
2) currentWeight = 0; currentValue = 0;
3) FOR i = 0; currentWeight < k && i < values.length; i++ DO:
       IF k - currentWeight < weights[i] DO
           currentValue = currentValue + values[i];
           currentWeight = currentWeight + weights[i];
       ELSE
           currentValue = currentValue + values[i]*(k - currentWeight)/weights[i]
           currentWeight = currentWeight + weights[i]*(k - currentWeight)/weights[i]
       END_IF
   END_FOR
   PRINT "maximumValueOfItemsInK = " + currentValue;
