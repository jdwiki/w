Visão geral do REST #

REST significa REpresentacional State Ttransfer e foi cunhado por Roy Fielding em sua tese de doutorado Architectural Styles and the Design of Network-based Software Architectures. Nele ele identifica princípios arquitetônicos específicos como:

  • Recursos Endereçáveis: a abstração chave de informações e dados em REST é um recurso e cada recurso deve ser endereçável por meio de um URI.

  • Uma interface uniforme e restrita: uso de um pequeno conjunto de métodos bem definidos para manipular nossos recursos.

  • Orientado à representação: Um recurso referenciado por um URI pode ter formatos diferentes e plataformas diferentes precisam de formatos diferentes, por exemplo, navegadores precisam de HTML, JavaScript precisa de JSON e aplicativos Java podem precisar de XML, JSON, CSV, texto, etc. serviços usando a representação desse serviço.

  • Comunique-se sem estado: os aplicativos sem estado são mais fáceis de dimensionar.

  • Hipermídia como motor do estado do aplicativo: deixe que nossos formatos de dados conduzam as transições de estado em nossos aplicativos.

O conjunto desses princípios arquitetônicos é chamado REST. Os conceitos de REST são inspirados pelos de HTTP. Roy Fielding, que nos deu REST, também é um dos autores das especificações HTTP.

Serviços da Web e Serviços da Web RESTful são serviços que são expostos à Internet para acesso programático. Eles são api online que podemos chamar de nosso código. Existem dois tipos de serviços web [“Grandes”] 3 serviços web SOAP e REST.

RESTful Web Services: Web services que são escritos aplicando os conceitos de arquitetura REST são chamados de RESTful Web Services, que se concentram nos recursos do sistema e como o estado de um recurso pode ser transferido pelo protocolo HTTP para diferentes clientes.

Este documento é focado exclusivamente em serviços da Web RESTful, portanto, não entraremos em detalhes do SOAP WS.

Não há regras rígidas ao projetar serviços web RESTful como

  • Sem padrão de protocolo
  • Sem padrão de canal de comunicação
  • Sem padrão de definição de serviço

Mas o SOAP tem regras rígidas para tudo isso. Todos os serviços da Web SOAP seguem a especificação SOAP que determina o que cada serviço da Web SOAP deve ser. Esta especificação é desenvolvida e gerenciada por um comitê e se o SOAP WS não segue uma única regra, então, por definição, não é SOAP.

Conceitos de Web Services RESTful

Existem algumas diretrizes que precisam ser consideradas ao projetar/desenvolver a API RESTful:

  1. Locais/URI baseados em recursos
  2. Uso adequado de métodos HTTP
  3. HATEOAS (hipermídia como mecanismo do estado do aplicativo)

A principal abordagem ao desenvolver APIs RESTful deve ser tornar a API “o mais RESTful possível”.

REST sobre HTTP #

REST é uma arquitetura protocolo-agnóstica proposta por Roy Fielding em sua dissertação (capítulo 5 sendo a apresentação do REST), que generaliza o conceito comprovado de navegadores web como clientes para para desacoplar clientes em um sistema distribuído de servidores.

Para que um serviço ou API seja RESTful, ele deve seguir determinadas restrições como:

  • Servidor cliente
  • Sem estado
  • Armazenável em cache
  • Sistema em camadas
  • Interface uniforme
    • Resources identification
    • Resources representation
    • Self-descriptive messages
    • Hypermedia

Além das restrições mencionadas na dissertação de Fielding, em seu blog REST APIs must be hypertext-driven , Fielding esclareceu que apenas invocar um serviço via HTTP não o torna RESTful. Um serviço deve, portanto, também respeitar outras regras que são resumidas a seguir:

  • A API deve aderir e não violar o protocolo subjacente. Embora o REST seja usado na maioria das vezes via HTTP, ele não está restrito a esse protocolo.

  • Forte foco em recursos e sua apresentação através de tipos de mídia.

  • Os clientes não devem ter conhecimento inicial ou suposições sobre os recursos disponíveis ou seu estado retornado ([recurso “digitado”](http://soabits.blogspot.co.at/2012/04/restful-resources-are-not-typed .html)) em uma API, mas aprenda-os rapidamente por meio de solicitações emitidas e respostas analisadas. Isso dá ao servidor a oportunidade de mover ou renomear recursos facilmente sem interromper uma implementação do cliente.

Modelo de maturidade Richardson #

O Modelo de Maturidade Richardson é uma maneira de aplicar restrições REST sobre HTTP para obter serviços web RESTful.

Leonard Richardson dividiu os aplicativos nessas 4 camadas:

  • Nível 0: uso de HTTP para o transporte
  • Nível 1: uso de URL para identificar recursos
  • Nível 2: uso de verbos HTTP e status para as interações
  • Nível 3: uso de HATEOAS

Como o foco está na representação do estado de um recurso, o suporte a várias representações para o mesmo recurso é incentivado. Uma representação pode, portanto, apresentar uma visão geral do estado do recurso, enquanto outra retorna os detalhes completos do mesmo recurso.

Observe também que, dadas as restrições de Fielding, uma API é efetivamente RESTful apenas quando o 3º nível do RMM é implementado.

solicitações e respostas HTTP #

Uma solicitação HTTP é:

  • Um verbo (também conhecido como método), na maioria das vezes um de GET, [POST](https://tools.ietf .org/html/rfc7231#section-4.3.3), PUT, [DELETE](https://tools.ietf .org/html/rfc7231#section-4.3.5) ou PATCH
  • Um URL
  • Cabeçalhos (pares chave-valor)
  • Opcionalmente, um corpo (também conhecido como carga útil, dados)

Uma resposta HTTP é:

  • Um status, na maioria das vezes um de 2xx (com êxito), [4xx (erro do cliente)](https://tools. ietf.org/html/rfc7231#section-6.5) ou 5xx (erro do servidor)
  • Cabeçalhos (pares chave-valor)
  • Um corpo (também conhecido como carga útil, dados)

Características dos verbos HTTP:

  • Verbos que possuem corpo: POST, PUT, PATCH
  • Verbos que devem ser seguros (ou seja, que não devem modificar recursos): GET
  • Verbos que devem ser idempotentes (ou seja, que não devem afetar os recursos novamente quando executados várias vezes): GET (nullipotente), PUT, DELETE
        body  safe  idempotent
GET      ✗     ✔     ✔
POST     ✔     ✗     ✗
PUT      ✔     ✗     ✔
DELETE   ✗     ✗     ✔
PATCH    ✔     ✗     ✗

Conseqüentemente, os verbos HTTP podem ser comparados às funções CRUD:

  • [Create](https://www.wikiod.com/pt/http/http-para-apis#Criar um recurso): POST
  • RLEIA: GET
  • [UPDATE](https://www.wikiod.com/pt/http/http-para-apis#Editar um recurso): PUT, PATCH
  • [DELETE](https://www.wikiod.com/pt/http/http-para-apis#Excluir um recurso): DELETE

Observe que uma solicitação PUT solicita que os clientes enviem todo o recurso com os valores atualizados. Para atualizar parcialmente um recurso, um verbo PATCH pode ser usado (consulte Como atualizar parcialmente um recurso?).

Status de resposta HTTP normal #

Sucesso #

  • 201 (CREATED): o recurso foi criado
  • 202 (ACEITO): solicitação aceita, mas processo ainda em andamento
  • 204 (SEM CONTEÚDO): solicitação atendida e sem conteúdo adicional
  • Caso contrário: 200 (OK)

Redirecionamento #

  • 304 (NÃO MODIFICADO): o cliente pode usar a versão em cache que possui do recurso solicitado

Erros do cliente #

  • 401 (UNAUTHORIZED): uma solicitação anônima acessa uma API protegida
  • 403 (PROIBIDO): uma solicitação autenticada não tem direitos suficientes para acessar uma API protegida
  • 404 (NÃO ENCONTRADO): recurso não encontrado
  • 409 (CONFLICT): estado do recurso em conflito (por exemplo, um usuário tentando criar uma conta com um e-mail já registrado)
  • 410 (GONE): igual ao 404, mas o recurso existia
  • 412 (PRECONDITION FAILED): a solicitação tenta modificar um recurso que está em um estado inesperado
  • 422 (UNPROCESSABLE ENTITY): a carga útil da solicitação é sintaticamente válida, mas semanticamente incorreta (por exemplo, um campo obrigatório que não foi valorizado)
  • 423 (BLOQUEADO): o recurso está bloqueado
  • 424 (FAILED DEPENDENCE): a ação solicitada dependia de outra ação que falhou
  • 429 (TOO MANY REQUESTS): o usuário enviou muitas solicitações em um determinado período de tempo
  • Caso contrário: 400 (PEDIDO RUIM)

Erros do servidor #

  • 501 (NÃO IMPLEMENTADO): ​​o servidor não suporta a funcionalidade necessária para atender à solicitação
  • 503 (SERVICE UNAVAILABLE): o servidor não consegue lidar com a solicitação devido a uma sobrecarga temporária ou manutenção programada
  • 507 (ARMAZENAMENTO INSUFICIENTE): o servidor não consegue armazenar a representação necessária para concluir a solicitação com êxito
  • Caso contrário: 500 (ERRO DO SERVIDOR INTERNO)

Notas #

Nada impede você de adicionar um corpo às respostas errôneas, para deixar a rejeição mais clara para os clientes. Por exemplo, o 422 (UNPROCESSABLE ENTITY) é um pouco vago: o corpo da resposta deve fornecer o motivo pelo qual a entidade não pôde ser processada.

HATEOAS #

Cada recurso deve fornecer hipermídia aos recursos aos quais está vinculado. Um link é pelo menos composto por:

  • Um rel (para relation, também conhecido como nome): descreve a relação entre o recurso principal e o(s) vinculado(s)
  • Um href: o URL que segmenta os recursos vinculados

Atributos adicionais também podem ser usados ​​para ajudar na descontinuação, negociação de conteúdo, etc.

Cormac Mulhall explica que o cliente deve decidir qual verbo HTTP usar com base no que está tentando fazer. Em caso de dúvida, a documentação da API deve ajudá-lo a entender as interações disponíveis com todas as hipermídias.

Tipos de mídia #

Os tipos de mídia ajudam a ter mensagens autodescritivas. Eles fazem o papel do contrato entre clientes e servidores, para que possam trocar recursos e hipermídias.

Embora application/json e application/xml sejam tipos de mídia bastante populares, eles não contêm muita semântica. Eles apenas descrevem a sintaxe geral usada no documento. Tipos de mídia mais especializados que suportam os requisitos do HATEOAS devem ser usados ​​(ou estendidos por meio de tipos de mídia do fornecedor), como:

  • Atom
  • RSS 2.0
  • HAL
  • coleção+json
  • JSON-LD
  • Siren

Um cliente informa a um servidor quais tipos de mídia ele entende adicionando o cabeçalho Accept à sua solicitação, por exemplo:

Accept: application/hal+json

Se o servidor não conseguir produzir o recurso solicitado em tal representação, ele retornará um 406 (NÃO ACEITÁVEL). Caso contrário, ele adiciona o tipo de mídia no cabeçalho Content-Type da resposta que contém o recurso representado, por exemplo:

Content-Type: application/hal+json

Stateless > Stateful #

Por que? #

Um servidor com estado implica que as sessões dos clientes sejam armazenadas em um armazenamento local de instância do servidor (quase sempre em sessões do servidor web). Isso começa a ser um problema ao tentar escalar horizontalmente: se você ocultar várias instâncias de servidor atrás de um balanceador de carga, se um cliente for despachado primeiro para instance # 1 ao entrar, mas depois para instância #2 ao buscar um recurso protegido, por exemplo, a segunda instância tratará a solicitação como anônima, pois **a sessão do cliente foi armazenada localmente na *instância #1 ***.

Foram encontradas soluções para resolver esse problema (por exemplo, configurando replicação de sessão e/ou sessão fixa), mas a arquitetura REST propõe outra abordagem: apenas não torne seu servidor com estado, torne-o sem estado. De acordo com Fielding:

Cada solicitação do cliente para o servidor deve conter todas as informações necessárias para entender a solicitação e não pode tirar proveito de nenhum contexto armazenado no servidor. Portanto, o estado da sessão é mantido inteiramente no cliente.

Em outras palavras, uma solicitação deve ser tratada exatamente da mesma maneira, independentemente de ser despachada para instância #1 ou instância #2. É por isso que os aplicativos sem estado são considerados mais fáceis de escalar.

Quão? #

Uma abordagem comum é uma autenticação baseada em token, especialmente com os modernos JSON Web Tokens. Observe que o JWT ainda tem alguns problemas, principalmente em relação à invalidação e prolongamento automático da expiração (ou seja, o recurso lembrar-me).

Notas laterais #

O uso de cookies ou cabeçalhos (ou qualquer outra coisa) não tem nada a ver com o fato de o servidor ser stateful ou stateless: são apenas mídias que são usadas aqui para transportar tokens (identificador de sessão para servidores stateful, JWT etc.), nada mais.

Quando uma API RESTful é usada apenas por navegadores, (HttpOnly e [secure](https://en.wikipedia.org/wiki/HTTP_cookie# Secure_cookie)) cookies podem ser bastante convenientes, pois os navegadores os anexarão automaticamente às solicitações de saída. Vale a pena mencionar que se você optar por cookies, esteja ciente de CSRF (uma boa maneira de evitar isso é ter os clientes gerar e enviar o mesmo valor secreto exclusivo em um cookie e um cabeçalho HTTP personalizado ).

API que pode ser armazenada em cache com solicitações condicionais #

Com o cabeçalho Last-Modified #

O servidor pode fornecer um cabeçalho de data Last-Modified para as respostas que contêm recursos que podem ser armazenados em cache. Os clientes devem então armazenar esta data junto com o recurso.

Agora, cada vez que os clientes solicitam a API para ler o recurso, eles podem adicionar às suas solicitações um cabeçalho If-Modified-Since contendo a última data de ‘Última modificação’ recebida e armazenada. O servidor deve então comparar o cabeçalho da solicitação e a data real da última modificação do recurso. Se forem iguais, o servidor retornará um 304 (NOT MODIFIED) com um corpo vazio: o cliente solicitante deve usar o recurso atualmente armazenado em cache. tem.

Além disso, quando os clientes solicitam a API para atualizar o recurso (ou seja, com um verbo não seguro), eles podem adicionar um [cabeçalho If-Unmodified-Since](https://tools.ietf.org/html/rfc7232#section- 3.4). Isso ajuda a lidar com condições de corrida: se o cabeçalho e a data real da última modificação forem diferentes, o servidor retornará um 412 (PRECONDITION FAILED). O cliente deve ler o novo estado do recurso antes de tentar modificar o recurso novamente.

Com o cabeçalho ETag #

Uma ETag (tag de entidade) é um identificador para um estado específico de um recurso. Pode ser um hash MD5 do recurso para uma validação forte ou um identificador específico de domínio para um validação fraca.

Basicamente, o processo é o mesmo do cabeçalho Last-Modified: o servidor fornece um cabeçalho ETag para as respostas que contêm recursos que podem ser armazenados em cache e os clientes devem armazenar esse identificador junto com o recurso.

Em seguida, os clientes fornecem um cabeçalho If-None-Match quando desejam ler o recurso, contendo a última ETag recebida e armazenada . O servidor agora pode retornar um 304 (NÃO MODIFICADO) se o cabeçalho corresponder à ETag real do recurso.

Novamente, os clientes podem fornecer um cabeçalho If-Match quando quiserem modificar o recurso, e o servidor deve retornar um 412 (PRECONDITION FAILED) se a ETag fornecida não corresponder à real.

Notas Adicionais #

ETag > data #

Se os clientes fornecerem data e ETag em suas solicitações, a data deverá ser ignorada. De RFC 7232 (aqui e aqui) :

Um destinatário DEVE ignorar If-Modified-Since/If-Unmodified-Since se a solicitação contiver um campo de cabeçalho If-None-Match/If-Match; a condição em If-None-Match/If-Match é considerada uma substituição mais precisa para a condição em If-Modified-Since/If-Unmodified-Since, e as duas são apenas combinadas para interoperar com intermediários mais antigos que podem não implementar If-None-Match/If-Match.

ETags rasas #

Além disso, embora seja bastante óbvio que as últimas datas modificadas são mantidas junto com os recursos do lado do servidor, várias abordagens estão disponíveis com ETag.

Uma abordagem usual é implementar ETags superficiais: o servidor processa a solicitação como se nenhum cabeçalho condicional fosse fornecido, mas apenas no final, ele gera a ETag da resposta que está prestes a retornar (por exemplo, por hash) e compara com o fornecido. Isso é relativamente fácil de implementar, pois apenas um interceptor HTTP é necessário (e muitas implementações já existem dependendo do servidor). Dito isso, vale a pena mencionar que essa abordagem economizará largura de banda, mas não o desempenho do servidor:

Uma implementação mais profunda do mecanismo ETag poderia fornecer benefícios muito maiores - como atender a algumas solicitações do cache e não precisar executar a computação - mas a implementação definitivamente não seria tão simples, nem tão conectável como a abordagem superficial descrita aqui.

Armadilhas comuns #

Por que não devo colocar verbos em URLs? #

HTTP não é RPC: o que torna o HTTP significativamente diferente do RPC é que as solicitações são direcionadas para Recursos. Afinal, URL significa Uniform Resource Locator, e um URL é um URI: um Uniform Resource Identfitier. O URL tem como alvo o recurso com o qual você deseja lidar, o método HTTP indica o que você deseja fazer com ele. Os métodos HTTP também são conhecidos como verbos: verbos em URLs não fazem sentido. Observe que as relações HATEOAS também não devem conter verbos, pois os links também visam recursos.

Como atualizar parcialmente um recurso? #

Como as solicitações PUT pedem aos clientes que enviem todo o recurso com os valores atualizados, PUT /users/123 não pode ser usado para simplesmente atualizar o e-mail de um usuário, por exemplo. Conforme explicado por William Durand em Please. Não corrija como um idiota., várias soluções compatíveis com REST estão disponíveis:

  • Exponha as propriedades do recurso e use o método PUT para enviar um valor atualizado, conforme a especificação PUT afirma que * atualizações parciais de conteúdo são possíveis ao direcionar um recurso identificado separadamente com estado que se sobrepõe a uma parte do recurso maior*:
PUT https://example.com/api/v1.2/users/123/email
body:
  new.email@example.com
  • Use uma solicitação PATCH que contém um conjunto de instruções descrevendo como o recurso deve ser modificado (por exemplo, seguindo JSON Patch):
PATCH https://example.com/api/v1.2/users/123
body:
  [
    { "op": "replace", "path": "/email", "value": "new.email@example.com" }
  ]
  • Use uma solicitação PATCH contendo uma representação parcial do recurso, conforme proposto em [comentário de Matt Chapman](http://williamdurand.fr/2014/02/14/please-do-not-patch-like-an- idiota/#comentário-1495702936):
PATCH https://example.com/api/v1.2/users/123
body:
  {
    "email": "new.email@example.com"
  }

E as ações que não se encaixam no mundo das operações CRUD? #

Citando Vinay Sahni em Melhores práticas para projetar uma API RESTful pragmática:

É aqui que as coisas podem ficar confusas. Existem várias abordagens:

  1. Reestruture a ação para que apareça como um campo de um recurso. Isso funciona se a ação não receber parâmetros. Por exemplo, uma ação activate pode ser mapeada para um campo booleano activated e atualizada por meio de um PATCH para o recurso.

  2. Trate-o como um sub-recurso com princípios RESTful. Por exemplo, a API do GitHub permite que você estrele uma essência com PUT /gists/:id/star e unstar com DELETE /gists/:id/star.

  3. Às vezes você realmente não tem como mapear a ação para uma estrutura RESTful sensata. Por exemplo, uma pesquisa de vários recursos realmente não faz sentido para ser aplicada a um ponto de extremidade de um recurso específico. Neste caso, /search faria mais sentido mesmo não sendo um recurso. Tudo bem - apenas faça o que é certo da perspectiva do consumidor da API e certifique-se de que está documentado claramente para evitar confusão.

Práticas comuns #

  • API está documentada. As ferramentas estão disponíveis para ajudá-lo a construir sua documentação, por exemplo. Swagger ou Documentos REST do Spring.

  • A API é versionada, seja por meio de cabeçalhos ou por meio do URL:

https://example.com/api/v1.2/blogs/123/articles
                        ^^^^
  • Os recursos têm nomes plurais:
https://example.com/api/v1.2/blogs/123/articles
                             ^^^^^     ^^^^^^^^
  • URLs usam kebab-case (as palavras são minúsculas e separadas por traço):
https://example.com/api/v1.2/quotation-requests
                             ^^^^^^^^^^^^^^^^^^
  • O HATEOAS fornece um link “auto” para recursos, visando a si mesmos:
{
  ...,
  _links: {
    ...,
    self: { href: "https://example.com/api/v1.2/blogs/123/articles/789" }
    ^^^^
  }
}
  • As relações HATEOAS usam lowerCamelCase (as palavras são em minúsculas, depois em maiúsculas, exceto a primeira, e os espaços são omitidos), para permitir que clientes JavaScript usem a [notação de ponto](https://developer.mozilla.org/en-US/docs /Web/JavaScript/Reference/Operators/Property_Accessors#Dot_notation) respeitando as convenções de nomenclatura JavaScript ao acessar os links:
{
  ...,
  _links: {
    ...,
    firstPage: { "href": "https://example.com/api/v1.2/blogs/123/articles?pageIndex=1&pageSize=25" }
    ^^^^^^^^^
  }
}

Gerenciamento de blogs através de uma API HTTP RESTful #

Os exemplos a seguir usam HAL para expressar HATEOAS e fazem uso de:

  • CURIE (URI compacto): usado para fornecer links para a documentação da API
  • modelos de URI: URI que inclui parâmetros que devem ser substituídos antes que o URI seja resolvido

Obter blog 123 #

Solicitar #

GET https://example.com/api/v1.2/blogs/123
headers:
  Accept: application/hal+json

Resposta #

status: 200 (OK)
headers:
  Content-Type: application/hal+json
body:
  {
    "id": 123,
    "title": "The blog title",
    "description": "The blog description",
    "_links": {
      "curies": [{ "name": "doc", "href": "https://example.com/docs/{rel}", "templated": true }],
      "self": { "href": "https://example.com/api/v1.2/blogs/123" },
      "doc:articles": { "href": "https://example.com/api/v1.2/blogs/123/articles{?pageIndex,pageSize}", "templated": true }
    }
  }

Notas #

  • O identificador usado para identificar o recurso a ser atualizado é o da URL: o do corpo (se houver) deve ser ignorado silenciosamente.
  • Como um pedido PUT atualiza todo o recurso, se nenhum conteúdo fosse enviado, ele deveria ter sido removido do recurso persistente.

Violando REST #

<stock>
    <add>
        <item>
            <name>Milk</name>
            <quantity>2</quantity>
        </item>
    </add>
</stock>

Colocar este corpo em um recurso como /stocks/123 viola a ideia por trás do REST. Enquanto este corpo é put e contém todas as informações necessárias, ele também vem com uma chamada de método para add em algum lugar quando o corpo é processado. Seguindo o REST, alguém postaria o item em /stocks/123/items/.
