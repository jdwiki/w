Índices são uma estrutura de dados que contém ponteiros para o conteúdo de uma tabela organizada em uma ordem específica, para ajudar o banco de dados a otimizar as consultas. Eles são semelhantes ao índice do livro, onde as páginas (linhas da tabela) são indexadas pelo número da página.

Existem vários tipos de índices e podem ser criados em uma tabela. Quando existe um índice nas colunas usadas na cláusula WHERE de uma consulta, cláusula JOIN ou cláusula ORDER BY, ele pode melhorar substancialmente o desempenho da consulta.

Os índices são uma maneira de acelerar as consultas de leitura classificando as linhas de uma tabela de acordo com uma coluna.

O efeito de um índice não é perceptível para bancos de dados pequenos como o exemplo, mas se houver um grande número de linhas, ele pode melhorar muito o desempenho. Em vez de verificar cada linha da tabela, o servidor pode fazer uma pesquisa binária no índice.

A desvantagem para criar um índice é a velocidade de gravação e o tamanho do banco de dados. Armazenar o índice ocupa espaço. Além disso, toda vez que um INSERT é feito ou a coluna é atualizada, o índice deve ser atualizado. Esta não é uma operação tão cara quanto varrer a tabela inteira em uma consulta SELECT, mas ainda é algo para se ter em mente.

Criando um índice #

CREATE INDEX ix_cars_employee_id ON Cars (EmployeeId);

Isso criará um índice para a coluna EmployeeId na tabela Cars. Esse índice melhorará a velocidade das consultas que solicitam ao servidor para classificar ou selecionar por valores em EmployeeId, como o seguinte:

SELECT * FROM Cars WHERE EmployeeId = 1

O índice pode conter mais de 1 coluna, como a seguir;

CREATE INDEX ix_cars_e_c_o_ids ON Cars (EmployeeId, CarId, OwnerId);

Nesse caso, o índice seria útil para consultas pedindo para classificar ou selecionar por todas as colunas incluídas, se o conjunto de condições for ordenado da mesma maneira. Isso significa que, ao recuperar os dados, ele pode encontrar as linhas a serem recuperadas usando o índice, em vez de procurar na tabela completa.

Por exemplo, o caso a seguir utilizaria o segundo índice;

SELECT * FROM Cars WHERE EmployeeId = 1 Order by CarId DESC

Se a ordem for diferente, no entanto, o índice não tem as mesmas vantagens, como a seguir;

SELECT * FROM Cars WHERE OwnerId = 17 Order by CarId DESC

O índice não é tão útil porque o banco de dados deve recuperar o índice inteiro, em todos os valores de EmployeeId e CarID, para descobrir quais itens têm OwnerId = 17.

(O índice ainda pode ser usado; pode ser que o otimizador de consulta descubra que recuperar o índice e filtrar no OwnerId e, em seguida, recuperar apenas as linhas necessárias é mais rápido do que recuperar a tabela completa, especialmente se a tabela for grande .)

Índices agrupados, exclusivos e classificados #

Os índices podem ter várias características que podem ser definidas na criação ou alterando os índices existentes.

CREATE CLUSTERED INDEX ix_clust_employee_id ON Employees(EmployeeId, Email);

A instrução SQL acima cria um novo índice clusterizado em Funcionários. Índices agrupados são índices que determinam a estrutura real da tabela; a própria tabela é classificada para corresponder à estrutura do índice. Isso significa que pode haver no máximo um índice clusterizado em uma tabela. Se já existir um índice clusterizado na tabela, a instrução acima falhará. (Tabelas sem índices clusterizados também são chamadas de heaps.)

CREATE UNIQUE INDEX uq_customers_email ON Customers(Email);

Isso criará um índice exclusivo para a coluna Email na tabela Clientes. Esse índice, além de acelerar as consultas como um índice normal, também forçará todos os endereços de e-mail dessa coluna a serem exclusivos. Se uma linha for inserida ou atualizada com um valor Email não exclusivo, a inserção ou atualização falhará, por padrão.

CREATE UNIQUE INDEX ix_eid_desc ON Customers(EmployeeID);

Isso cria um índice em Customers que também cria uma restrição de tabela que o EmployeeID deve ser exclusivo. (Isso falhará se a coluna não for exclusiva no momento - neste caso, se houver funcionários que compartilham um ID.)

CREATE INDEX ix_eid_desc ON Customers(EmployeeID Desc);

Isso cria um índice que é classificado em ordem decrescente. Por padrão, os índices (no servidor MSSQL, pelo menos) são crescentes, mas isso pode ser alterado.

Índice ordenado #

Se você usar um índice que é classificado da maneira que você o recuperaria, a instrução SELECT não faria uma classificação adicional durante a recuperação.

CREATE INDEX ix_scoreboard_score ON scoreboard (score DESC);

Ao executar a consulta

SELECT * FROM scoreboard ORDER BY score DESC;

O sistema de banco de dados não faria uma classificação adicional, pois pode fazer uma pesquisa de índice nessa ordem.

Eliminando um índice ou desabilitando-o e reconstruindo-o #

DROP INDEX ix_cars_employee_id ON Cars;

Podemos usar o comando DROP para excluir nosso índice. Neste exemplo vamos DROP o índice chamado ix_cars_employee_id na tabela Cars.

Isso exclui totalmente o índice e, se o índice estiver agrupado, removerá qualquer agrupamento. Ele não pode ser reconstruído sem recriar o índice, que pode ser lento e computacionalmente caro. Como alternativa, o índice pode ser desabilitado:

ALTER INDEX ix_cars_employee_id ON Cars DISABLE;

Isso permite que a tabela retenha a estrutura, juntamente com os metadados sobre o índice.

Criticamente, isso retém as estatísticas do índice, para que seja possível avaliar facilmente a alteração. Se necessário, o índice pode ser reconstruído posteriormente, em vez de ser recriado completamente;

ALTER INDEX ix_cars_employee_id ON Cars REBUILD;

Índice parcial ou filtrado #

SQL Server e SQLite permitem criar índices que contêm não apenas um subconjunto de colunas, mas também um subconjunto de linhas.

Considere uma quantidade crescente constante de pedidos com order_state_id igual a concluído (2), e uma quantidade estável de pedidos com order_state_id igual a iniciado (1).

Se sua empresa faz uso de consultas como esta:

SELECT id, comment
  FROM orders
 WHERE order_state_id =  1
   AND product_id = @some_value;

A indexação parcial permite limitar o índice, incluindo apenas os pedidos inacabados:

CREATE INDEX Started_Orders
          ON orders(product_id)
       WHERE order_state_id = 1;

Esse índice será menor que um índice não filtrado, o que economiza espaço e reduz o custo de atualização do índice.

Inserindo com um índice exclusivo #

UPDATE Customers SET Email = "richard0123@example.com" WHERE id = 1;

Isso falhará se um índice exclusivo for definido na coluna Email de Clientes. No entanto, um comportamento alternativo pode ser definido para este caso:

UPDATE Customers SET Email = "richard0123@example.com" WHERE id = 1 ON DUPLICATE KEY;

Reconstruir índice #

Ao longo do tempo, os índices B-Tree podem se tornar fragmentados devido à atualização/exclusão/inserção de dados. Na terminologia do SQLServer, podemos ter interno (página de índice que está meio vazia) e externo (ordem lógica da página não corresponde à ordem física). A reconstrução do índice é muito semelhante a descartar e recriá-lo.

Podemos reconstruir um índice com

ALTER INDEX index_name REBUILD;

Por padrão, a reconstrução do índice é uma operação offline que bloqueia a tabela e impede a DML contra ela, mas muitos RDBMS permitem a reconstrução online. Além disso, alguns fornecedores de banco de dados oferecem alternativas para reconstrução de índice, como REORGANIZE (SQLServer) ou COALESCE/SHRINK SPACE (Oracle).

SAP ASE: Índice de descarte #

Este comando irá descartar o índice na tabela. Funciona no servidor SAP ASE.

Sintaxe:

DROP INDEX [table name].[index name]

Exemplo:

DROP INDEX Cars.index_1

Índice único que permite NULL #

CREATE UNIQUE INDEX idx_license_id 
   ON Person(DrivingLicenseID) WHERE DrivingLicenseID IS NOT NULL
GO

Este esquema permite uma relação 0..1 - as pessoas podem ter zero ou uma carta de condução e cada carta só pode pertencer a uma pessoa

Índice agrupado #

Ao usar o índice clusterizado, as linhas da tabela são classificadas pela coluna à qual o índice clusterizado é aplicado. Portanto, pode haver apenas um índice clusterizado na tabela porque você não pode ordenar a tabela por duas colunas diferentes.

Geralmente, é melhor usar o índice clusterizado ao realizar leituras em tabelas de big data. O lado negativo do índice clusterizado é ao gravar na tabela e os dados precisam ser reorganizados (reclassificados).

Um exemplo de criação de um índice clusterizado em uma tabela Employees na coluna Employee_Surname:

CREATE CLUSTERED INDEX ix_employees_name ON Employees(Employee_Surname);

Índice não clusterizado #

Índices não clusterizados são armazenados separadamente da tabela. Cada índice nesta estrutura contém um ponteiro para a linha na tabela que ele representa.

Esses ponteiros são chamados de localizadores de linha. A estrutura do localizador de linha depende se as páginas de dados são armazenadas em um heap ou em uma tabela em cluster. Para um heap, um localizador de linha é um ponteiro para a linha. Para uma tabela clusterizada, o localizador de linha é a chave de índice clusterizado.

Um exemplo de criação de um índice não clusterizado na tabela Employees e na coluna Employee_Surname:

CREATE NONCLUSTERED INDEX ix_employees_name ON Employees(Employee_Surname);

Pode haver vários índices não clusterizados na tabela. As operações de leitura são geralmente mais lentas com índices não clusterizados do que com índices clusterizados, pois você precisa ir primeiro para o índice e depois para a tabela. No entanto, não há restrições nas operações de gravação.
