Instalação ou Configuração #

WebGL é uma tecnologia de navegador, portanto, não há muito o que configurar além de ter um navegador. Você pode começar com WebGL em JSFiddle ou Codepen ou JSBIn ou qualquer outro sites que permitem editar HTML, CSS e JavaScript on-line, embora haja algumas limitações (veja abaixo). Você também pode hospedar arquivos de código aberto em páginas do github ou serviços semelhantes.

Por outro lado, em algum momento você provavelmente vai trabalhar localmente. Para fazer isso, é recomendável que você execute um servidor web simples. Há muito por onde escolher que são simples de usar e requerem muito pouca configuração.

Usando node.js como servidor

  1. instale node.js
  2. Abra um terminal ou prompt de comando de nó e digite npm install -g http-server (no OSX coloque sudo na frente disso.
  3. digite http-server para começar a servir os arquivos na pasta atual OU http-server path-to-folder para servir uma pasta diferente
  4. Aponte seu navegador para http://localhost:8080/name-of-file para visualizar sua página WebGL

Usando o devd como servidor

  1. Baixe [devd] (https://github.com/cortesi/devd)
  2. Abra um terminal e execute devd com devd . para arquivos do servidor da pasta atual ou devd path-to-folder para servir uma pasta diferente
  3. Aponte seu navegador para http://localhost:8000/name-of-file para visualizar sua página WebGL

Usando o Servez como servidor

  1. Baixe Serve
  2. Instale-o, execute-o
  3. Escolha a pasta para servir
  4. Escolha “Iniciar”
  5. Vá para http://localhost:8080 ou escolha “Iniciar navegador”

Usando a extensão do Chrome “Web Server for Chrome”

  1. Instale o Servidor da Web do Chrome

  2. Inicie-o a partir do ícone Aplicativos em uma nova página de guia.

  1. Defina a pasta onde estão seus arquivos e clique no link http://127.0.0.1:8787

Limitação do WebGL em Serviços Online #

No WebGL é muito comum carregar imagens. No WebGL existem restrições sobre como as imagens podem ser usadas. Especificamente, o WebGL não pode usar imagens de outros domínios sem permissão do servidor que hospeda as imagens. Os serviços que atualmente dão permissão para usar imagens incluem imgur e flickr. Consulte Carregando imagens de domínio cruzado. Caso contrário, você precisará ter as imagens no mesmo servidor que sua página webgl ou usar outras soluções criativas, como gerar imagens com uma tag canvas

Como mencionado na seção comentários, precisamos fornecer duas funções. Um sombreador de vértice e um sombreador de fragmento

Vamos começar com um sombreador de vértice

// an attribute will receive data from a buffer
attribute vec4 position;

// all shaders have a main function
void main() {

  // gl_Position is a special variable a vertex shader 
  // is responsible for setting
  gl_Position = position;
}

Se tudo fosse escrito em JavaScript em vez de GLSL, você poderia imaginar que seria usado assim

// *** PSUEDO CODE!! ***

var positionBuffer = [
  0, 0, 0, 0,
  0, 0.5, 0, 0,
  0.7, 0, 0, 0,
];
var attributes = {};
var gl_Position;

drawArrays(..., offset, count) {
  for (var i = 0; i < count; ++i) {
     // copy the next 4 values from positionBuffer to the position attribute
     attributes.position = positionBuffer.slice((offset + i) * 4, 4); 
     runVertexShader();
     ...
     doSomethingWith_gl_Position();
}

Em seguida, precisamos de um sombreador de fragmento

// fragment shaders don't have a default precision so we need
// to pick one. mediump, short for medium precision, is a good default.
precision mediump float;

void main() {
  // gl_FragColor is a special variable a fragment shader
  // is responsible for setting
  gl_FragColor = vec4(1, 0, 0.5, 1); // return redish-purple 
}

Acima estamos configurando gl_FragColor para 1, 0, 0.5, 1 que é 1 para vermelho, 0 para verde, 0.5 para azul, 1 para alfa. As cores no WebGL vão de 0 a 1.

Agora que escrevemos as 2 funções vamos começar com o WebGL

Primeiro, precisamos de um elemento de tela HTML

 <canvas id="c"></canvas>

Então, em JavaScript, podemos procurar isso

 var canvas = document.getElementById("c");

Agora podemos criar um WebGLRenderingContext

 var gl = canvas.getContext("webgl");
 if (!gl) {
    // no webgl for you!
    ...

Agora precisamos compilar esses shaders para colocá-los na GPU, então primeiro precisamos colocá-los em strings. Você pode obter suas strings da maneira que normalmente obtém strings. Concatenando, usando AJAX, colocando-os em tags de script do tipo não JavaScript ou, neste caso, usando literais de modelo de várias linhas

var vertexShaderSource = `
// an attribute will receive data from a buffer
attribute vec4 position;

// all shaders have a main function
void main() {

  // gl_Position is a special variable a vertex shader 
  // is responsible for setting
  gl_Position = position;
}
`;

var fragmentShaderSource = `
// fragment shaders don't have a default precision so we need
// to pick one. mediump is a good default
precision mediump float;

void main() {
  // gl_FragColor is a special variable a fragment shader
  // is responsible for setting
  gl_FragColor = vec4(1, 0, 0.5, 1); // return redish-purple 
}
`;

Então precisamos de uma função que irá criar um shader, fazer upload da fonte e compilar o shader

function createShader(gl, type, source) {
  var shader = gl.createShader(type);
  gl.shaderSource(shader, source);  
  gl.compileShader(shader);
  var success = gl.getShaderParameter(shader, gl.COMPILE_STATUS);
  if (success) {
    return shader;
  }

  console.log(gl.getShaderInfoLog(shader));
  gl.deleteShader(shader);
}

Agora podemos chamar essa função para criar os 2 shaders

var vertexShader = createShader(gl, gl.VERTEX_SHADER, vertexShaderSource);
var fragmentShader = createShader(gl, gl.FRAGMENT_SHADER, fragmentShaderSource);

Em seguida, precisamos vincular esses 2 shaders em um programa

function createProgram(gl, vertexShader, fragmentShader) {
  var program = gl.createProgram();
  gl.attachShader(program, vertexShader);
  gl.attachShader(program, fragmentShader);
  gl.linkProgram(program);
  var sucesss = gl.getProgramParameter(program, gl.LINK_STATUS);
  if (success) {
    return program;
  }

  console.log(gl.getProgramInfoLog(program));
  gl.deleteProgram(program);
}

E chame isso

var program = createProgram(gl, vertexShader, fragmentShader);

Agora que criamos um programa GLSL na GPU, precisamos fornecer dados a ele. A maior parte da API WebGL trata da configuração do estado para fornecer dados aos nossos programas GLSL. Neste caso, nossa única entrada para nosso programa GLSL é position que é um atributo. A primeira coisa que devemos fazer é procurar a localização do atributo para o programa que acabamos de criar

var positionAttributeLocation = gl.getAttribLocation(program, "position");

Os atributos obtêm seus dados de buffers, então precisamos criar um buffer

var positionBuffer = gl.createBuffer();

WebGL nos permite manipular muitos recursos WebGL em pontos de ligação globais. Você pode pensar em pontos de ligação como variáveis ​​globais internas dentro do WebGL. Primeiro você define o ponto de ligação para o seu recurso. Em seguida, todas as outras funções se referem ao recurso por meio do ponto de vinculação. Então, vamos vincular o buffer de posição.

gl.bindBuffer(gl.ARRAY_BUFFER, positionBuffer);

Agora podemos colocar dados nesse buffer referenciando-o através do ponto de ligação

// three 2d points
var positions = [
  0, 0, 
  0, 0.5,
  0.7, 0,
];
gl.bufferData(gl.ARRAY_BUFFER, new Float32Array(positions), gl.STATIC_DRAW);

Tem muita coisa acontecendo aqui. A primeira coisa é que temos positions que é um array JavaScript. O WebGL, por outro lado, precisa de dados fortemente tipados, então a linha new Float32Array(positions) cria um novo array de números de ponto flutuante de 32 bits e copia os valores de positions. gl.bufferData então copia esses dados para o positionBuffer na GPU. Ele está usando o buffer de posição porque o vinculamos ao ponto de ligação ARRAY_BUFFER acima.

O último argumento, gl.STATIC_DRAW é uma dica para WebGL sobre como usaremos os dados. Ele pode tentar usar essa informação para otimizar certas coisas. gl.STATIC_DRAW informa ao WebGL que provavelmente não alteraremos muito esses dados.

Agora que colocamos os dados no buffer, precisamos informar ao atributo como obter dados dele. Primeiro precisamos ativar o atributo

gl.enableVertexAttribArray(positionAttributeLocation);

Em seguida, precisamos especificar como extrair os dados

var size = 2;          // 2 components per iteration
var type = gl.FLOAT;   // the data is 32bit floats
var normalize = false; // use the data as is
var stride = 0;        // 0 = move size * sizeof(type) each iteration
var offset = 0;        // start at the beginning of the buffer
gl.vertexAttribPointer(
   positionAttributeLocation, size, type, normalize, stride, offset)

Uma parte oculta do gl.vertexAttribPointer é que ele liga o ARRAY_BUFFER atual ao atributo. Em outras palavras, agora que este atributo está vinculado ao positionBuffer, estamos livres para vincular outra coisa ao ponto de vinculação ARRAY_BUFFER.

observe que, do ponto de vista do nosso vertex shader GLSL, o atributo position era um vec4

attribute vec4 position;

vec4 é um valor de 4 float. Em JavaScript você poderia pensar em algo como position = {x: 0, y: 0, z: 0, w: 0}. Acima definimos size = 2. Atributos padrão para 0, 0, 0, 1 para que este atributo obtenha seus 2 primeiros valores (x e y) do nosso buffer. O z e w serão o padrão 0 e 1, respectivamente.

Depois de tudo isso, podemos finalmente pedir ao WebGL para executar o programa GLSL.

var primitiveType = gl.TRIANGLES;
var offset = 0;
var count = 3;
gl.drawArrays(primitiveType, offset, count);

Isso executará nosso vertex shader 3 vezes. A primeira vez que position.x e position.y em nosso vertex shader serão definidos para os 2 primeiros valores do positionBuffer. A 2ª vez position.xy será definida para os 2º 2 valores. Na última vez, será definido para os 2 últimos valores.

Como configuramos primitiveType para gl.TRIANGLES, cada vez que nosso vertex shader é executado 3 vezes, o WebGL desenhará um triângulo com base nos 3 valores para os quais configuramos gl_Position. Não importa o tamanho de nossa tela, esses valores estão em coordenadas de espaço de clipe que vão de -1 a 1 em cada direção.

Como nosso sombreador de vértice está simplesmente copiando nossos valores positionBuffer para gl_Position, o triângulo será desenhado nas coordenadas do espaço do clipe

  0, 0, 
  0, 0.5,
  0.7, 0,

Como esses valores são convertidos em pixels depende da configuração gl.viewport. O padrão gl.viewport é o tamanho inicial da tela. Como não definimos um tamanho para nossa tela, o tamanho padrão é 300x150. Convertendo de espaço de clipe para pixels (geralmente chamado de espaço de tela na literatura WebGL e OpenGL) WebGL vai desenhar um triângulo em

 clip space      screen space
   0, 0       ->   150, 75
   0, 0.5     ->   150, 112.5
 0.7, 0       ->   255, 75

O WebGL agora renderizará esse triângulo. Para cada pixel que estiver prestes a desenhar, ele chamará nosso sombreador de fragmento. Nosso sombreador de fragmento apenas define gl_FragColor como 1, 0, 0.5, 1. Como o Canvas é um canvas de 8 bits por canal, isso significa que o WebGL vai escrever os valores [255, 0, 127, 255] no canvas.

Existem 3 coisas principais que ainda não abordamos nas observações. Texturas, variações e uniformes. Cada um deles requer seu próprio tópico.
