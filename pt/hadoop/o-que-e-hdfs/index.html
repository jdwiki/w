Encontrando arquivos no HDFS #

Para localizar um arquivo no sistema de arquivos distribuído do Hadoop:

hdfs dfs -ls -R / | grep [search_term]

No comando acima,

-ls é para listar arquivos

-R é recursivo (iter através de subdiretórios)

/ significa do diretório raiz

| para canalizar a saída do primeiro comando para o segundo

Comando grep para extrair strings correspondentes

[search_term] nome do arquivo a ser pesquisado na lista de todos os arquivos no sistema de arquivos do hadoop.

Alternativamente, o comando abaixo também pode ser usado para localizar e também aplicar algumas expressões:

hadoop fs -find / -name test -print

Localiza todos os arquivos que correspondem à expressão especificada e aplica ações selecionadas a eles. Se nenhum caminho for especificado, o padrão será o diretório de trabalho atual. Se nenhuma expressão for especificada, o padrão será -print.

As seguintes expressões primárias são reconhecidas:

  • padrão de nome
  • padrão de nome

Avalia como verdadeiro se o nome base do arquivo corresponder ao padrão usando globbing do sistema de arquivos padrão. Se -iname for usado, a correspondência não fará distinção entre maiúsculas e minúsculas.

  • imprimir
  • imprimir0Sempre

Avalia como verdadeiro. Faz com que o nome do caminho atual seja gravado na saída padrão. Se a expressão -print0 for usada, um caractere ASCII NULL será anexado.

São reconhecidos os seguintes operadores:

expression -a expression
expression -and expression
expression expression

HDFS - Sistema de Arquivo Distribuído Hadoop #

O Hadoop Distributed File System (HDFS) é um sistema de arquivos baseado em Java que fornece armazenamento de dados escalável e confiável projetado para abranger grandes clusters de servidores comuns. HDFS, MapReduce e YARN formam o núcleo do Apache™ Hadoop®.

O HDFS foi projetado para ser altamente tolerante a falhas, o que é obtido salvando várias cópias (3 por padrão) de um determinado bloco de dados em vários nós.

Blocos e divisões HDFS #

  1. Tamanho do Bloco e Blocos no HDFS : O HDFS tem o conceito de armazenar dados em blocos sempre que um arquivo é carregado. Blocos são as partições físicas de dados no HDFS (ou em qualquer outro sistema de arquivos).

    Whenever a file is loaded onto the HDFS, it is splitted physically (yes, the file is divided) into different parts known as blocks. The number of blocks depend upon the value of dfs.block.size in hdfs-site.xml

    Ideally, the block size is set to a large value such as 64/128/256 MBs (as compared to 4KBs in normal FS). The default block size value on most distributions of Hadoop 2.x is 128 MB. The reason for a higher block size is because Hadoop is made to deal with PetaBytes of data with each file ranging from a few hundred MegaBytes to the order of TeraBytes.

    Say for example you have a file of size 1024 MBs. if your block size is 128 MB, you will get 8 blocks of 128MB each. This means that your namenode will need to store metadata of 8 x 3 = 24 files (3 being the replication factor).

    Consider the same scenario with a block size of 4 KBs. It will result in 1GB / 4KB = 250000 blocks and that will require the namenode to save the metadata for 750000 blocks for just a 1GB file. Since all these metadata related information is stored in-memory, larger block size is preferred to save that bit of extra load on the NameNode.

    Now again, the block size is not set to an extremely high value like 1GB etc because, ideally, 1 mapper is launched for each block of data. So if you set the block size to 1GB, you might lose parallelism which might result in a slower throughput overall.

2.) Divisão de tamanho em HDFS: Splits in Hadoop Processing are the logical chunks of data. When files are divided into blocks, hadoop doesn’t respect any file bopundaries. It just splits the data depending on the block size. Say if you have a file of 400MB, with 4 lines, and each line having 100MB of data, you will get 3 blocks of 128 MB x 3 and 16 MB x 1. But when input splits are calculated while the prceossing of data, file/record boundaries are kept in mind and in this case we will have 4 input splits of 100 MB each, if you are using, say, NLineInputFormat.

Split Size também pode ser definido por job usando a propriedade mapreduce.input.fileinputformat.split.maxsize
