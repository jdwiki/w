A parametrização é a criação de diferentes conjuntos de dados para diferentes usuários no mesmo script de teste. Por exemplo, executando vários usuários com credenciais diferentes no mesmo script. Isso o torna um dos principais aspectos na criação de testes de desempenho.

Parametrização usando arquivos externos #

Uma das maneiras comuns de parametrizar seus scripts de desempenho é usar um arquivo CSV. O melhor exemplo de uso de arquivos de entrada CSV é um processo de login. Se você quiser testar seu aplicativo em diferentes usuários, precisará fornecer uma lista de credenciais de usuário.

Vamos supor que temos uma solicitação de login que funciona para um usuário específico:

Podemos parametrizar facilmente essa solicitação usando um arquivo CSV externo e executando o script em diferentes usuários. Para adicionar a configuração de parametrização CSV:

Clique com o botão direito na solicitação de login -> Adicionar -> Config Element -> CSV Data Set Config

Uma breve explicação dos parâmetros ‘CSV Data Set Config’:

  • Nome - nome do elemento como será usado na árvore JMeter
  • Nome do arquivo - nome do arquivo de entrada. Os nomes de arquivos relativos são resolvidos com base no caminho do plano de teste ativo. Nomes de arquivos absolutos também são suportados
  • Codificação de arquivo - codificação do arquivo de entrada, caso não seja o padrão da plataforma
  • Nomes de Variáveis ​​- lista de nomes de variáveis ​​separados que serão usados ​​como um contêiner para valores analisados. Se estiver vazio, a primeira linha do arquivo será interpretada como a lista de nomes de variáveis
  • Delimiter - delimitador que será usado para dividir os valores analisados ​​do arquivo de entrada
  • Permitir dados cotados? - true caso você queira ignorar aspas duplas e permitir que tais elementos contenham um delimitador.
  • Reciclar em EOF? - true caso o plano de teste de arquivo deva iterar sobre o arquivo mais de uma vez. Ele instruirá o JMeter a mover o cursor no início do arquivo
  • Parar thread no EOF? - false em caso de iteração de loop sobre o arquivo CDC e true se você quiser parar o thread depois de ler o arquivo inteiro
  • Modo de compartilhamento:
  • Todos os threads - o arquivo é compartilhado entre todos os usuários virtuais (o padrão)
  • Grupo de threads atual - o arquivo será aberto uma vez para cada grupo de threads
  • Tópico atual - cada arquivo será aberto separadamente para cada um dos tópicos
  • Identificador - todos os threads que compartilham o mesmo identificador também compartilham o mesmo arquivo

Parametrização usando bancos de dados #

Outra maneira de parametrizar seus scripts de desempenho é usar dados de banco de dados por meio de JDBC. JDBC é uma interface de programação de aplicativos que define como um cliente pode acessar um banco de dados.

Antes de tudo, baixe o driver JDBC para seu banco de dados (consulte o fornecedor do banco de dados). Por exemplo, o driver mysql pode ser encontrado aqui. Em seguida, você pode adicioná-lo adicionando o arquivo .jar ao plano de teste usando o formulário abaixo:

Mas é melhor adicionar o arquivo Jar na pasta lib e reiniciar o JMeter.

Depois disso, configure a conexão com o banco de dados usando o elemento ‘Configuração de Conexão JDBC’. Assim: Clique com o botão direito do mouse em Thread Group -> Add -> Config Element -> JDBC Connection Configuration

Parâmetros de ‘Configuração de conexão JDBC’:

  • Nome - nome da configuração de conexão que será mostrada na árvore do grupo de threads
  • Nome da variável - nome que será usado como identificador exclusivo para a conexão db (podem ser usadas várias conexões e cada uma será vinculada a um nome diferente)
  • Número máximo de conexões - número máximo de conexões permitidas no pool de conexões. No caso de 0, cada thread terá seu próprio pool com uma única conexão nele
  • Max Wait (ms) - pool gera um erro se o tempo limite especificado for excedido durante a conexão db
  • Tempo entre execuções de despejo (ms) - número de milissegundos para pausar entre execuções do encadeamento que despeja conexões não utilizadas do pool de banco de dados
  • Auto Commit - sim para ativar o auto commit para conexões db relacionadas
  • Test While Idle - verifique as conexões inativas antes que uma solicitação efetiva seja detectada. Mais detalhes: BasicDataSource.html#getTestWhileIdle
  • Soft Min Evictable Idle Time (ms) - período de tempo durante o qual uma conexão especificada pode ficar inativa no pool de banco de dados antes de ser despejada. Mais detalhes: BasicDataSource.html#getSoftMinEvictableIdleTimeMillis
  • Consulta de validação - consulta de verificação de integridade que será usada para verificar se o banco de dados ainda está respondendo
  • URL do banco de dados - string de conexão JDBC para o banco de dados. Veja aqui exemplos
  • JDBC Driver class - nome apropriado da classe do driver (específico para cada db). Por exemplo, ‘com.mysql.jdbc.Driver’ para MySql db
  • Nome de usuário - nome de usuário do banco de dados
  • Senha - senha do banco de dados (será armazenada sem criptografia no plano de teste)

No nosso caso, precisamos configurar apenas os campos obrigatórios:

  • Nome da variável vinculada ao pool.
  • URL do banco de dados
  • Classe de driver JDBC
  • Nome de usuário
  • Senha

Parametrização usando o plugin ‘Controlador parametrizado’ #

Se você precisar executar uma sequência repetida da mesma ação com parâmetros diferentes, use o plug-in de terceiros ‘Controlador parametrizado’ do projeto JMeter-Plugins.

Você precisa instalar este plugin primeiro seguindo o procedimento de instalação.

Vamos supor que queremos parametrizar o fluxo de trabalho de login:

Antes de tudo, você precisa instalar o plugin ‘Parameterized Controller’, pois ele não está incluído no núcleo do JMeter. As etapas de instalação desse processo podem ser encontradas aqui.

Vamos mover a ‘Solicitação de login’ para um controlador separado e desativá-lo (clique com o botão direito do mouse e selecione “Desativar”). Essa é a maneira mais preferível de ter um contêiner de módulos dentro do seu plano de teste e evitar o uso do Workbench como tal contêiner. Após o término da instalação, você pode adicionar dois controladores ‘Controlador parametrizado’ com credenciais de usuário diferentes: Clique com o botão direito do mouse em Thread Group -> Add -> Logic Controller -> Parameterized Controller

Controladores parametrizados contém a seção ‘Variáveis ​​Definidas pelo Usuário’, onde você pode especificar seus parâmetros. Coloque as credenciais do primeiro usuário no primeiro controlador parametrizado e as credenciais do segundo usuário no segundo controlador parametrizado.

Dentro de ambos os controladores parametrizados, adicione referências ao ‘Controlador Reutilizável’ para chamar a ‘Solicitação de login’ com parâmetros diferentes. Pode ser feito desta forma:

Clique com o botão direito do mouse em ‘Controlador parametrizado’ -> ‘Adicionar’ -> ‘Controlador lógico’ -> ‘Controlador de módulo’

Ao executar seu script, você verá que a ‘solicitação de login’ acionou cada um dos controladores parametrizados separadamente. Pode ser muito útil caso você precise executar seu script em diferentes combinações de parâmetros de entrada.
