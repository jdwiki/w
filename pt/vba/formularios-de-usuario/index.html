Melhores Práticas #

Um UserForm é um módulo de classe com um designer e uma instância padrão. O designer pode ser acessado pressionando Shift+F7 enquanto visualiza o code-behind, e o code-behind pode ser acessado pressionando F7 enquanto visualiza o designer.

Trabalhe sempre com uma nova instância. #

Sendo um módulo de classe, um formulário é, portanto, um projeto para um objeto. Como um formulário pode conter estado e dados, é uma prática melhor trabalhar com uma nova instância da classe, em vez da padrão/global:

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        '...
    End If
End With

Ao invés de:

UserForm1.Show vbModal
If Not UserForm1.IsCancelled Then
    '...
End If

Trabalhar com a instância padrão pode levar a bugs sutis quando o formulário é fechado com o botão “X” vermelho e/ou quando Unload Me é usado no code-behind.

Implemente a lógica em outro lugar. #

Um formulário não deve se preocupar com nada além de apresentação: um manipulador de botão Click que se conecta a um banco de dados e executa uma consulta parametrizada baseada na entrada do usuário, está fazendo muitas coisas.

Em vez disso, implemente a lógica do aplicativo no código responsável por exibir o formulário, ou melhor ainda, em módulos e procedimentos dedicados.

Escreva o código de tal forma que o UserForm seja responsável apenas por saber como exibir e coletar dados: de onde os dados vêm, ou o que acontece com os dados depois, não é da sua conta.

O chamador não deve se incomodar com os controles. #

Faça um model bem definido para o formulário trabalhar, seja em seu próprio módulo de classe dedicado, ou encapsulado dentro do próprio code-behind do formulário - exponha o model com procedimentos Property Get e tenha o código do cliente trabalhe com isso: isso torna o formulário uma abstração sobre os controles e seus detalhes minuciosos, expondo apenas os dados relevantes ao código do cliente.

Isso significa um código que se parece com isso:

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        MsgBox .Message, vbInformation
    End If
End With

Em vez disso:

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then
        MsgBox .txtMessage.Text, vbInformation
    End If
End With

Manipule o evento QueryClose. #

Os formulários normalmente têm um botão Fechar, e os prompts/diálogos têm os botões Ok e Cancelar; o usuário pode fechar o formulário usando a caixa de controle do formulário (o botão “X” vermelho), que destrói a instância do formulário por padrão (outra boa razão para trabalhar com uma nova instância sempre).

With New UserForm1
    .Show vbModal
    If Not .IsCancelled Then 'if QueryClose isn't handled, this can raise a runtime error.
        '...
    End With
End With

A maneira mais simples de lidar com o evento QueryClose é definir o parâmetro Cancel como True e, em seguida, ocultar o formulário em vez de fechá-lo:

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)
    Cancel = True
    Me.Hide
End Sub

Dessa forma, o botão “X” nunca destruirá a instância e o chamador poderá acessar com segurança todos os membros públicos.

Esconda, não feche. #

O código que cria um objeto deve ser responsável por destruí-lo: não é responsabilidade do formulário descarregar e encerrar a si mesmo.

Evite usar Unload Me no code-behind de um formulário. Em vez disso, chame Me.Hide, para que o código de chamada ainda possa usar o objeto criado quando o formulário for fechado.

Nomeie as coisas. #

Use a janela de ferramentas propriedades (F4) para nomear cuidadosamente cada controle em um formulário. O nome de um controle é usado no code-behind, portanto, a menos que você esteja usando uma ferramenta de refatoração que possa lidar com isso, renomear um controle quebrará o código - então é muito mais fácil fazer as coisas corretamente no primeiro lugar, do que tentar descobrir exatamente qual das 20 caixas de texto TextBox12 representa.

Tradicionalmente, os controles UserForm são nomeados com prefixos no estilo húngaro:

  • lblUserName para um controle Label que indica um nome de usuário.
  • txtUserName para um controle TextBox onde o usuário pode inserir um nome de usuário.
  • cboUserName para um controle ComboBox onde o usuário pode inserir ou escolher um nome de usuário.
  • lstUserName para um controle ListBox onde o usuário pode escolher um nome de usuário.
  • btnOk ou cmdOk para um controle Button rotulado como “Ok”.

O problema é que quando, por exemplo, a interface do usuário é redesenhada e um ComboBox muda para um ListBox, o nome precisa ser alterado para refletir o novo tipo de controle: é melhor nomear os controles pelo que eles representam, em vez de seu tipo de controle - para desacoplar o código da interface do usuário, tanto quanto possível.

  • UserNameLabel para um rótulo somente leitura que indica um nome de usuário.
  • UserNameInput para um controle onde o usuário pode inserir ou escolher um nome de usuário.
  • OkButton para um botão de comando rotulado como “Ok”.

Qualquer que seja o estilo escolhido, qualquer coisa é melhor do que deixar todos os controles com seus nomes padrão. A consistência no estilo de nomenclatura também é ideal.

Manipulando QueryClose #

O evento QueryClose é gerado sempre que um formulário está prestes a ser fechado, seja via ação do usuário ou programaticamente. O parâmetro CloseMode contém um valor de enumeração VbQueryClose que indica como o formulário foi fechado:

Constante Descrição Valor
vbFormControlMenu O formulário está fechando em resposta à ação do usuário 0
vbFormCode O formulário está fechando em resposta a uma instrução Unload 1
vbAppWindows Sessão do Windows está terminando 2
vbAppTaskManager O Gerenciador de Tarefas do Windows está fechando o aplicativo host 3
vbFormMDIForm Não suportado em VBA 4

Para melhor legibilidade, é melhor usar essas constantes em vez de usar seu valor diretamente.

Um UserForm cancelável #

Dado um formulário com um botão Cancelar

O code-behind do formulário pode ficar assim:

Option Explicit
Private Type TView
    IsCancelled As Boolean
    SomeOtherSetting As Boolean
    'other properties skipped for brievety
End Type
Private this As TView

Public Property Get IsCancelled() As Boolean
    IsCancelled = this.IsCancelled
End Property

Public Property Get SomeOtherSetting() As Boolean
    SomeOtherSetting = this.SomeOtherSetting
End Property

'...more properties...

Private Sub SomeOtherSettingInput_Change()
    this.SomeOtherSetting = CBool(SomeOtherSettingInput.Value)
End Sub

Private Sub OkButton_Click()
    Me.Hide
End Sub

Private Sub CancelButton_Click()
    this.IsCancelled = True
    Me.Hide
End Sub

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)
    If CloseMode = VbQueryClose.vbFormControlMenu Then
        Cancel = True
        this.IsCancelled = True
        Me.Hide
    End If
End Sub

O código de chamada poderia então exibir o formulário e saber se ele foi cancelado:

Public Sub DoSomething()
    With New UserForm1
        .Show vbModal
        If .IsCancelled Then Exit Sub
        If .SomeOtherSetting Then
            'setting is enabled
        Else
            'setting is disabled
        End If
    End With
End Sub

A propriedade IsCancelled retorna True quando o botão Cancelar é clicado, ou quando o usuário fecha o formulário usando a caixa de controle.
