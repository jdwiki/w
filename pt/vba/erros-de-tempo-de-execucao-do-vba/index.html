O código que compila ainda pode apresentar erros em tempo de execução. Este tópico lista os mais comuns, suas causas e como evitá-los.

Erro em tempo de execução ‘6’: estouro #

Código incorreto #

Sub DoSomething()
    Dim row As Integer
    For row = 1 To 100000
        'do stuff
    Next
End Sub

Por que isso não funciona? #

O tipo de dados Integer é um inteiro com sinal de 16 bits com um valor máximo de 32.767; atribuí-lo a qualquer coisa maior que isso transbordará o tipo e gerará esse erro.

Código correto #

Sub DoSomething()
    Dim row As Long
    For row = 1 To 100000
        'do stuff
    Next
End Sub

Por que isso funciona? #

Ao usar um inteiro Long (32 bits), agora podemos fazer um loop que itera mais de 32.767 vezes sem estourar o tipo da variável do contador.

Outras notas #

Consulte Tipos de dados e limites para obter mais informações.

Erro em tempo de execução ‘9’: Subscrito fora do intervalo #

Código incorreto #

Sub DoSomething()
    Dim foo(1 To 10)
    Dim i As Long
    For i = 1 To 100
        foo(i) = i
    Next
End Sub

Por que isso não funciona? #

foo é um array que contém 10 itens. Quando o contador de loop i atinge um valor de 11, foo(i) está fora do intervalo. Este erro ocorre sempre que uma matriz ou coleção é acessada com um índice que não existe nessa matriz ou coleção.

Código correto #

Sub DoSomething()
    Dim foo(1 To 10)
    Dim i As Long
    For i = LBound(foo) To UBound(foo)
        foo(i) = i
    Next
End Sub

Por que isso funciona? #

Use as funções LBound e UBound para determinar os limites inferior e superior de um array, respectivamente.

Outras notas #

Quando o índice é uma string, por exemplo. ThisWorkbook.Worksheets("I don't exist"), este erro significa que o nome fornecido não existe na coleção consultada.

O erro real é específico da implementação; Collection gerará o erro de tempo de execução 5 “Chamada ou argumento de procedimento inválido”:

Sub RaisesRunTimeError5()
    Dim foo As New Collection
    foo.Add "foo", "foo"
    Debug.Print foo("bar")
End Sub

Erro em tempo de execução ‘13’: incompatibilidade de tipo #

Código incorreto #

Public Sub DoSomething()
    DoSomethingElse "42?"
End Sub

Private Sub DoSomethingElse(foo As Date)
'    Debug.Print MonthName(Month(foo))
End Sub

Por que isso não funciona? #

O VBA está se esforçando muito para converter o argumento "42?" em um valor Date. Quando falha, a chamada para DoSomethingElse não pode ser executada, porque o VBA não sabe qual data passar, então gera o erro de tempo de execução 13 type mismatch, porque o tipo do argumento não corresponde ao esperado type (e também não pode ser convertido implicitamente).

Código correto #

Public Sub DoSomething()
    DoSomethingElse Now
End Sub

Private Sub DoSomethingElse(foo As Date)
'    Debug.Print MonthName(Month(foo))
End Sub

Por que isso funciona? #

Ao passar um argumento Date para um procedimento que espera um parâmetro Date, a chamada pode ser bem-sucedida.

Erro em tempo de execução ‘91’: variável de objeto ou com variável de bloco não definida #

Código incorreto #

Sub DoSomething()
    Dim foo As Collection
    With foo
        .Add "ABC"
        .Add "XYZ"
    End With
End Sub

Por que isso não funciona? #

As variáveis ​​de objeto contêm uma referência, e as referências precisam ser definidas usando a palavra-chave Set. Este erro ocorre sempre que uma chamada de membro é feita em um objeto cuja referência é Nothing. Neste caso foo é uma referência Collection, mas não é inicializada, então a referência contém Nothing - e não podemos chamar .Add em Nothing.

Código correto #

Sub DoSomething()
    Dim foo As Collection
    Set foo = New Collection
    With foo
        .Add "ABC"
        .Add "XYZ"
    End With
End Sub

Por que isso funciona? #

Ao atribuir à variável de objeto uma referência válida usando a palavra-chave Set, as chamadas .Add são bem-sucedidas.

Outras notas #

Muitas vezes, uma função ou propriedade pode retornar uma referência de objeto - um exemplo comum é o método Range.Find do Excel, que retorna um objeto Range:

Dim resultRow As Long
resultRow = SomeSheet.Cells.Find("Something").Row

No entanto, a função pode muito bem retornar Nothing (se o termo de pesquisa não for encontrado), então é provável que a chamada do membro .Row encadeado falhe.

Antes de chamar os membros do objeto, verifique se a referência está definida com uma condição If Not xxxx Is Nothing:

Dim result As Range
Set result = SomeSheet.Cells.Find("Something")

Dim resultRow As Long
If Not result Is Nothing Then resultRow = result.Row

Erro em tempo de execução ‘20’: Continuar sem erro #

Código incorreto #

Sub DoSomething()
    On Error GoTo CleanFail
    DoSomethingElse

CleanFail:
    Debug.Print Err.Number
    Resume Next
End Sub

Por que isso não funciona? #

Se o procedimento DoSomethingElse gerar um erro, a execução salta para o rótulo da linha CleanFail, imprime o número do erro, e a instrução Resume Next volta para a instrução que segue imediatamente a linha onde ocorreu o erro, que neste caso é a instrução Debug.Print: a sub-rotina de tratamento de erros está sendo executada sem um contexto de erro, e quando a instrução Resume Next é alcançada, o erro de tempo de execução 20 é gerado porque não há para onde retomar.

Código correto #

Sub DoSomething()
    On Error GoTo CleanFail
    DoSomethingElse

    Exit Sub    
CleanFail:
    Debug.Print Err.Number
    Resume Next
End Sub

Por que isso funciona? #

Ao introduzir uma instrução Exit Sub antes do rótulo da linha CleanFail, separamos a sub-rotina de tratamento de erros CleanFail do resto do corpo do procedimento - a única maneira de executar a sub-rotina de tratamento de erros é através de um comando On Erro salto; portanto, nenhum caminho de execução atinge a instrução Resume fora de um contexto de erro, o que evita o erro de tempo de execução 20.

Outras notas #

Isso é muito semelhante a Erro em tempo de execução ‘3’: Return without GoSub; em ambas as situações, a solução é garantir que o caminho de execução normal não possa entrar em uma sub-rotina (identificada por um rótulo de linha) sem um salto explícito (assumindo que On Error GoTo seja considerado um salto explícito).

Erro em tempo de execução ‘3’: Retorno sem GoSub #

Código incorreto #

Sub DoSomething()
    GoSub DoThis
DoThis:
    Debug.Print "Hi!"
    Return
End Sub

Por que isso não funciona? #

A execução entra no procedimento DoSomething, salta para o rótulo DoThis, imprime “Hi!” para a saída de depuração, retorna para a instrução imediatamente após a chamada GoSub, imprime “Oi!” novamente, e então encontra uma instrução Return, mas não há para onde retornar agora, porque não chegamos aqui com uma instrução GoSub.

Código correto #

Sub DoSomething()
    GoSub DoThis
    Exit Sub
DoThis:
    Debug.Print "Hi!"
    Return
End Sub

Por que isso funciona? #

Ao introduzir uma instrução Exit Sub antes do rótulo da linha DoThis, separamos a sub-rotina DoThis do resto do corpo do procedimento - a única maneira de executar a sub-rotina DoThis é através do GoSub pular.

Outras notas #

GoSub/Return está obsoleto e deve ser evitado em favor de chamadas de procedimento reais. Um procedimento não deve conter sub-rotinas, exceto manipuladores de erros.

Isso é muito semelhante a [Erro em tempo de execução ‘20’: Continuar sem erro](https://www.wikiod.com/pt/vba/erros-de-tempo-de-execucao-do-vba#Erro em tempo de execução ‘20’: Continuar sem erro -retomar-sem-erro); em ambas as situações, a solução é garantir que o caminho de execução normal não possa entrar em uma sub-rotina (identificada por um rótulo de linha) sem um salto explícito (assumindo que On Error GoTo seja considerado um salto explícito).
