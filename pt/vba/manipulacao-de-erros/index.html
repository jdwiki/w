Evitando condições de erro #

Quando ocorre um erro de tempo de execução, um bom código deve tratá-lo. A melhor estratégia de tratamento de erros é escrever um código que verifique as condições de erro e simplesmente evite a execução de código que resulte em um erro de tempo de execução.

Um elemento chave na redução de erros de tempo de execução é escrever pequenos procedimentos que fazem uma coisa. Quanto menos motivos os procedimentos tiverem para falhar, mais fácil será depurar o código como um todo.

Evitando o erro de tempo de execução 91 - Objeto ou com variável de bloco não definida:

Este erro será gerado quando um objeto for usado antes de sua referência ser atribuída. Pode-se ter um procedimento que recebe um parâmetro de objeto:

Private Sub DoSomething(ByVal target As Worksheet)
    Debug.Print target.Name
End Sub

Se target não tiver uma referência atribuída, o código acima gerará um erro que é facilmente evitado verificando se o objeto contém uma referência de objeto real:

Private Sub DoSomething(ByVal target As Worksheet)
    If target Is Nothing Then Exit Sub
    Debug.Print target.Name
End Sub

Se target não tiver uma referência atribuída, a referência não atribuída nunca será usada e nenhum erro ocorrerá.

Esta forma de sair antecipadamente de um procedimento quando um ou mais parâmetros não são válidos é chamada de cláusula de guarda.

Evitando o erro de tempo de execução 9 - Subscrito fora do intervalo:

Este erro é gerado quando uma matriz é acessada fora de seus limites.

Private Sub DoSomething(ByVal index As Integer)
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

Dado um índice maior que o número de planilhas no ActiveWorkbook, o código acima gerará um erro de tempo de execução. Uma simples cláusula de guarda pode evitar que:

Private Sub DoSomething(ByVal index As Integer)
    If index > ActiveWorkbook.Worksheets.Count Or index <= 0 Then Exit Sub
    Debug.Print ActiveWorkbook.Worksheets(index)
End Sub

A maioria dos erros de tempo de execução podem ser evitados verificando cuidadosamente os valores que estamos usando antes de usá-los e ramificando em outro caminho de execução usando uma simples instrução If - em cláusulas de guarda que não fazem suposições e validam os parâmetros de um procedimento, ou mesmo no corpo de procedimentos maiores.

Erros personalizados #

Muitas vezes, ao escrever uma classe especializada, você desejará que ela gere seus próprios erros específicos e desejará uma maneira limpa para o código de usuário/chamada lidar com esses erros personalizados. Uma maneira legal de conseguir isso é definindo um tipo Enum dedicado:

Option Explicit
Public Enum FoobarError
    Err_FooWasNotBarred = vbObjectError + 1024
    Err_BarNotInitialized
    Err_SomethingElseHappened
End Enum

O uso da constante interna vbObjectError garante que os códigos de erro personalizados não se sobreponham aos códigos de erro reservados/existentes. Apenas o primeiro valor enum precisa ser explicitamente especificado, pois o valor subjacente de cada membro Enum é 1 maior que o membro anterior, portanto, o valor subjacente de Err_BarNotInitialized é implicitamente vbObjectError + 1025.

Gerando seus próprios erros de tempo de execução #

Um erro de tempo de execução pode ser gerado usando a instrução Err.Raise, portanto, o erro personalizado Err_FooWasNotBarred pode ser gerado da seguinte maneira:

Err.Raise Err_FooWasNotBarred

O método Err.Raise também pode receber parâmetros personalizados Description e Source - por esse motivo, é uma boa ideia também definir constantes para armazenar a descrição de cada erro personalizado:

Private Const Msg_FooWasNotBarred As String = "The foo was not barred."
Private Const Msg_BarNotInitialized As String = "The bar was not initialized."

E, em seguida, crie um método privado dedicado para gerar cada erro:

Private Sub OnFooWasNotBarredError(ByVal source As String)
    Err.Raise Err_FooWasNotBarred, source, Msg_FooWasNotBarred
End Sub

Private Sub OnBarNotInitializedError(ByVal source As String)
    Err.Raise Err_BarNotInitialized, source, Msg_BarNotInitialized
End Sub

A implementação da classe pode simplesmente chamar esses procedimentos especializados para gerar o erro:

Public Sub DoSomething()
    'raises the custom 'BarNotInitialized' error with "DoSomething" as the source:
    If Me.Bar Is Nothing Then OnBarNotInitializedError "DoSomething"
    '...
End Sub

O código cliente pode então lidar com Err_BarNotInitialized como faria com qualquer outro erro, dentro de sua própria sub-rotina de tratamento de erros.

Nota: a palavra-chave herdada Error também pode ser usada no lugar de Err.Raise, mas está obsoleta/obsoleta.

Retomar palavra-chave #

Uma sub-rotina de tratamento de erros irá:

  • executar até o final do procedimento, caso em que a execução é retomada no procedimento de chamada.
  • ou, use a palavra-chave Resume para retomar a execução dentro do mesmo procedimento.

A palavra-chave Resume só deve ser usada dentro de uma sub-rotina de tratamento de erros, porque se o VBA encontrar Resume sem estar em um estado de erro, o erro de tempo de execução 20 “Resume without error” será gerado.

Existem várias maneiras pelas quais uma sub-rotina de tratamento de erros pode usar a palavra-chave Resume:

  • Resume usado sozinho, a execução continua na instrução que causou o erro. Se o erro não for realmente tratado antes de fazer isso, o mesmo erro será gerado novamente e a execução poderá entrar em um loop infinito.
  • Resume Next continua a execução na instrução imediatamente após a instrução que causou o erro. Se o erro não for realmente tratado antes de fazer isso, a execução poderá continuar com dados potencialmente inválidos, o que pode resultar em erros lógicos e comportamento inesperado.
  • Resume [line label] continua a execução no rótulo de linha especificado (ou número de linha, se você estiver usando números de linha no estilo legado). Isso normalmente permitiria a execução de algum código de limpeza antes de sair do procedimento de forma limpa, como garantir que uma conexão de banco de dados seja fechada antes de retornar ao chamador.

On Error Resume Next #

A própria instrução On Error pode usar a palavra-chave Resume para instruir o tempo de execução do VBA a efetivamente ignorar todos os erros.

Se o erro não for realmente tratado antes disso, a execução poderá continuar com dados potencialmente inválidos, o que pode resultar em erros lógicos e comportamento inesperado.

A ênfase acima não pode ser enfatizada o suficiente. On Error Resume Next efetivamente ignora todos os erros e os coloca debaixo do tapete. Um programa que explode com um erro de tempo de execução dado entrada inválida é um programa melhor do que um que continua rodando com dados desconhecidos/não intencionais - seja apenas porque o bug é muito mais facilmente identificável. On Error Resume Next pode facilmente ocultar bugs.

A instrução On Error tem escopo de procedimento - é por isso que normalmente deve haver apenas uma, única instrução On Error em um determinado procedimento.

No entanto às vezes uma condição de erro não pode ser evitada, e pular para uma sub-rotina de tratamento de erros apenas para ‘Resume Next’ simplesmente não parece certo. Nesse caso específico, a instrução conhecida por possível falha pode ser colocada entre duas instruções On Error:

On Error Resume Next
[possibly-failing statement]
Err.Clear 'resets current error
On Error GoTo 0

A instrução On Error GoTo 0 redefine o tratamento de erros no procedimento atual, de modo que qualquer instrução adicional que cause um erro de tempo de execução não seja tratada nesse procedimento e, em vez disso, passe a pilha de chamadas até que seja capturada por um manipulador de erro ativo. Se não houver nenhum manipulador de erro ativo na pilha de chamadas, ele será tratado como uma exceção não tratada.

Public Sub Caller()
    On Error GoTo Handler
    
    Callee
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Caller."
End Sub

Public Sub Callee()
    On Error GoTo Handler
    
    Err.Raise 1     'This will be handled by the Callee handler.
    On Error GoTo 0 'After this statement, errors are passed up the stack.
    Err.Raise 2     'This will be handled by the Caller handler.    
    
    Exit Sub
Handler:
    Debug.Print "Error " & Err.Number & " in Callee."
    Resume Next
End Sub

Na declaração de erro #

Mesmo com cláusulas de guarda, não se pode realmente sempre levar em conta todas as possíveis condições de erro que podem ser levantadas no corpo de um procedimento. A instrução On Error GoTo instrui o VBA a pular para um rótulo de linha e entrar no “modo de tratamento de erros” sempre que ocorrer um erro inesperado em tempo de execução. Depois de lidar com um erro, o código pode retomar de volta à execução “normal” usando a palavra-chave Resume.

Rótulos de linha denotam sub-rotinas: como as sub-rotinas se originam do código BASIC legado e usam saltos GoTo e GoSub e instruções Return para voltar à rotina “principal”, é bastante fácil escrever siga código de espaguete se as coisas não forem rigorosamente estruturadas. Por esse motivo, é melhor que:

  • um procedimento tem uma e apenas uma sub-rotina de tratamento de erros
  • a sub-rotina de tratamento de erros só é executada em estado de erro

Isso significa que um procedimento que trata seus erros deve ser estruturado assim:

Private Sub DoSomething()
    On Error GoTo CleanFail

    'procedure code here

CleanExit:
    'cleanup code here
    Exit Sub

CleanFail:
    'error-handling code here
    Resume CleanExit
End Sub

Estratégias de tratamento de erros #

Às vezes, você deseja lidar com erros diferentes com ações diferentes. Nesse caso, você inspecionará o objeto global Err, que conterá informações sobre o erro que foi gerado - e agirá de acordo:

CleanExit:
    Exit Sub

CleanFail:
    Select Case Err.Number
        Case 9
            MsgBox "Specified number doesn't exist. Please try again.", vbExclamation
            Resume
        Case 91
            'woah there, this shouldn't be happening.
            Stop 'execution will break here
            Resume 'hit F8 to jump to the line that raised the error
        Case Else
            MsgBox "An unexpected error has occurred:" & vbNewLine & Err.Description, vbCritical
            Resume CleanExit
    End Select
End Sub

Como orientação geral, considere ativar o tratamento de erros para toda a sub-rotina ou função e trate todos os erros que possam ocorrer dentro de seu escopo. Se você precisar lidar apenas com erros na seção pequena do código - ative e desative o tratamento de erros no mesmo nível:

Private Sub DoSomething(CheckValue as Long)

    If CheckValue = 0 Then
        On Error GoTo ErrorHandler   ' turn error handling on
        ' code that may result in error
        On Error GoTo 0              ' turn error handling off - same level
    End If

CleanExit:
    Exit Sub

ErrorHandler:
    ' error handling code here
    ' do not turn off error handling here
    Resume

End Sub

Números de linha #

O VBA suporta números de linha de estilo legado (por exemplo, QBASIC). A propriedade oculta Erl pode ser usada para identificar o número da linha que gerou o último erro. Se você não estiver usando números de linha, Erl retornará apenas 0.

Sub DoSomething()
10 On Error GoTo 50
20 Debug.Print 42 / 0
30 Exit Sub
40
50 Debug.Print "Error raised on line " & Erl ' returns 20
End Sub

Se você estiver usando números de linha, mas não de forma consistente, então Erl retornará o último número de linha antes da instrução que gerou o erro.

Sub DoSomething()
10 On Error GoTo 50
   Debug.Print 42 / 0
30 Exit Sub

50 Debug.Print "Error raised on line " & Erl 'returns 10
End Sub

Tenha em mente que Erl também só tem precisão Integer, e irá transbordar silenciosamente. Isso significa que os números de linha fora do intervalo inteiro fornecerão resultados incorretos:

Sub DoSomething()
99997 On Error GoTo 99999
99998 Debug.Print 42 / 0
99999
      Debug.Print Erl   'Prints 34462
End Sub

O número da linha não é tão relevante quanto a declaração que causou o erro, e a numeração de linhas rapidamente se torna tediosa e não muito fácil de manter.
