Nomes de Variáveis #

Variáveis ​​armazenam dados. Nomeie-os de acordo com o uso, não de acordo com o tipo de dados ou escopo, usando um substantivo. Se você se sentir compelido a numerar suas variáveis ​​(por exemplo, coisa1, coisa2, coisa3), considere usar uma estrutura de dados apropriada (por exemplo, uma matriz, uma coleção ou um dicionário).

Nomes de variáveis ​​que representam um conjunto de valores iteráveis ​​- por exemplo um array, uma Collection, um Dictionary ou um Range de células, deve ser plural.

Algumas convenções de nomenclatura VBA comuns são assim:

Para variáveis ​​de nível de procedimento:

camelCase

Public Sub ExampleNaming(ByVal inputValue As Long, ByRef inputVariable As Long)

    Dim procedureVariable As Long
    Dim someOtherVariable As String

End Sub

Para variáveis ​​de nível de módulo:

PascalCase

Public GlobalVariable As Long
Private ModuleVariable As String

Para Constantes:

SHOUTY_SNAKE_CASE é comumente usado para diferenciar constantes de variáveis:

Public Const GLOBAL_CONSTANT As String = "Project Version #1.000.000.001"
Private Const MODULE_CONSTANT As String = "Something relevant to this Module"

Public Sub SomeProcedure()

    Const PROCEDURE_CONSTANT As Long = 10

End Sub

No entanto, os nomes PascalCase tornam o código mais limpo e são tão bons quanto, dado que o IntelliSense usa ícones diferentes para variáveis ​​e constantes:

Notação Húngara #

Nomeie-os de acordo com o que são usados, não de acordo com seu tipo de dados ou escopo.

“A notação húngara torna mais fácil ver qual é o tipo de uma variável”

Se você escrever seu código como os procedimentos seguem o Princípio da Responsabilidade Única (como deveria), você nunca deve olhar para uma tela cheia de declarações de variáveis ​​no topo de qualquer procedimento; declare variáveis ​​o mais próximo possível de seu primeiro uso, e seu tipo de dados sempre estará à vista se você as declarar com um tipo explícito. O atalho Ctrl+i do VBE também pode ser usado para exibir o tipo de variável em uma dica de ferramenta.

Para que uma variável é usada é uma informação muito mais útil do que seu tipo de dados, especialmente em uma linguagem como VBA que converte feliz e implicitamente um tipo em outro conforme necessário.

Considere iFile e strFile neste exemplo:

Function bReadFile(ByVal strFile As String, ByRef strData As String) As Boolean
    Dim bRetVal As Boolean
    Dim iFile As Integer

    On Error GoTo CleanFail

    iFile = FreeFile
    Open strFile For Input As #iFile
    Input #iFile, strData

    bRetVal = True

CleanExit:
    Close #iFile
    bReadFile = bRetVal
    Exit Function
CleanFail:
    bRetVal = False
    Resume CleanExit
End Function

Comparado a:

Function CanReadFile(ByVal path As String, ByRef outContent As String) As Boolean
    On Error GoTo CleanFail

    Dim handle As Integer
    handle = FreeFile

    Open path For Input As #handle
    Input #handle, outContent

    Dim result As Boolean
    result = True

CleanExit:
    Close #handle
    CanReadFile = result
    Exit Function
CleanFail:
    result = False
    Resume CleanExit
End Function

strData é passado como ByRef no exemplo principal, mas além do fato de que temos a sorte de ver que é explicitamente passado como tal, não há indicação de que strData seja realmente retornado pela função.

O exemplo inferior o chama de outContent; este prefixo out é para o que a notação húngara foi inventada: para ajudar a esclarecer para que uma variável é usada, neste caso para identificá-la claramente como um parâmetro “out”.

Isso é útil, porque o IntelliSense por si só não exibe ByRef, mesmo quando o parâmetro é explicitamente passado por referência:

Húngaro bem feito

A notação húngara originalmente não tinha nada a ver com tipos de variáveis. Na verdade, a notação húngara feito certo é realmente útil. Considere este pequeno exemplo (ByVal e As Integer removidos para brevety):

Public Sub Copy(iX1, iY1, iX2, iY2)
End Sub

Comparado a:

Public Sub Copy(srcColumn, srcRow, dstColumn, dstRow)
End Sub

src e dst são prefixos de Notação Húngara aqui, e eles transmitem informações úteis que de outra forma não podem ser inferidas dos nomes dos parâmetros ou do IntelliSense mostrando o tipo declarado.

É claro que há uma maneira melhor de transmitir tudo isso, usando abstração apropriada e palavras reais que podem ser pronunciadas em voz alta e fazer sentido - como um exemplo artificial:

Type Coordinate
    RowIndex As Long
    ColumnIndex As Long
End Type

Sub Copy(source As Coordinate, destination As Coordinate)
End Sub

Nomes de Procedimentos #

Procedimentos faz alguma coisa. Nomeie-os de acordo com o que estão fazendo, usando um verbo. Se não for possível nomear com precisão um procedimento, provavelmente o procedimento está fazendo muitas coisas e precisa ser dividido em procedimentos menores e mais especializados.

Algumas convenções de nomenclatura VBA comuns são assim:

Para todos os procedimentos:

PascalCase

Public Sub DoThing()

End Sub

Private Function ReturnSomeValue() As [DataType]

End Function

Para procedimentos de manipulador de eventos:

ObjectName_EventName

Public Sub Workbook_Open()

End Sub

Public Sub Button1_Click()

End Sub

Os manipuladores de eventos geralmente são nomeados automaticamente pelo VBE; renomeá-los sem renomear o objeto e/ou o evento manipulado quebrará o código - o código será executado e compilado, mas o procedimento do manipulador ficará órfão e nunca será executado.

Membros booleanos

Considere uma função de retorno booleano:

Function bReadFile(ByVal strFile As String, ByRef strData As String) As Boolean
End Function

Comparado a:

Function CanReadFile(ByVal path As String, ByRef outContent As String) As Boolean
End Function

O prefixo Can tem o mesmo propósito que o prefixo b: ele identifica o valor de retorno da função como um Booleano. Mas Can lê melhor que b:

If CanReadFile(path, content) Then

Comparado com:

If bReadFile(strFile, strData) Then

Considere usar prefixos como Can, Is ou Has na frente de membros de retorno booleano (funções e propriedades), mas somente quando agregar valor. Isso está em conformidade com as diretrizes de nomenclatura atuais da Microsoft.
