Abstração #

Os níveis de abstração ajudam a determinar quando dividir as coisas. #

A abstração é alcançada implementando a funcionalidade com código cada vez mais detalhado. O ponto de entrada de uma macro deve ser um pequeno procedimento com um alto nível de abstração que facilite a compreensão rápida do que está acontecendo:

Public Sub DoSomething()
    With New SomeForm
        Set .Model = CreateViewModel
        .Show vbModal            
        If .IsCancelled Then Exit Sub
        ProcessUserData .Model
    End With
End Sub

O procedimento DoSomething tem um alto nível de abstração: podemos dizer que ele está exibindo um formulário e criando algum modelo, e passando esse objeto para algum procedimento ProcessUserData que sabe o que fazer com ele - como o modelo é criado é o trabalho de outro procedimento:

Private Function CreateViewModel() As ISomeModel
    Dim result As ISomeModel
    Set result = SomeModel.Create(Now, Environ$("UserName"))
    result.AvailableItems = GetAvailableItems
    Set CreateViewModel = result
End Function

A função CreateViewModel é responsável apenas por criar alguma instância ISomeModel. Parte dessa responsabilidade é adquirir uma matriz de itens disponíveis - como esses itens são adquiridos é um detalhe de implementação que é abstraído por trás do procedimento GetAvailableItems:

Private Function GetAvailableItems() As Variant
    GetAvailableItems = DataSheet.Names("AvailableItems").RefersToRange
End Function

Aqui o procedimento está lendo os valores disponíveis de um intervalo nomeado em uma planilha DataSheet. Poderia muito bem lê-los de um banco de dados, ou os valores poderiam ser codificados: é um detalhe de implementação que não é uma preocupação para nenhum dos níveis de abstração mais altos.

Encapsulamento #

O encapsulamento oculta os detalhes de implementação do código do cliente. #

O exemplo https://www.wikiod.com/pt/vba/formularios-de-usuario#Manipulando a consultaClose demonstra o encapsulamento: o formulário tem um controle de caixa de seleção, mas seu código de cliente não funciona diretamente com ele - a caixa de seleção é um detalhe de implementação, o que o código do cliente precisa saber é se a configuração está habilitada ou não.

Quando o valor da caixa de seleção é alterado, o manipulador atribui um membro de campo privado:

TView Tipo Privado IsCancelled As Boolean SomeOtherSetting As Boolean ‘outras propriedades ignoradas por brevidade Tipo Final Privado como TView

‘…

Private Sub SomeOtherSettingInput_Change() this.SomeOtherSetting = CBool(SomeOtherSettingInput.Value) Finalizar Sub

E quando o código do cliente deseja ler esse valor, ele não precisa se preocupar com uma caixa de seleção - em vez disso, ele simplesmente usa a propriedade SomeOtherSetting:

Propriedade Pública Get SomeOtherSetting() As Boolean SomeOtherSetting = this.SomeOtherSetting Finalizar propriedade

A propriedade SomeOtherSetting encapsula o estado da caixa de seleção; o código do cliente não precisa saber que há uma caixa de seleção envolvida, apenas que há uma configuração com um valor booleano. Ao encapsular o valor Boolean, adicionamos uma camada de abstração ao redor da caixa de seleção.

Usando interfaces para impor imutabilidade #

Vamos levar isso um passo adiante encapsulando o model do formulário em um módulo de classe dedicado. Mas se fizermos uma Public Property para o UserName e Timestamp, teríamos que expor os acessadores Property Let, tornando as propriedades mutáveis, e não queremos que o código do cliente tenha a capacidade de mudar esses valores depois de definidos.

A função CreateViewModel no exemplo Abstraction retorna uma classe ISomeModel: essa é a nossa interface, e se parece com isso:

Option Explicit

Public Property Get Timestamp() As Date
End Property

Public Property Get UserName() As String
End Property

Public Property Get AvailableItems() As Variant
End Property

Public Property Let AvailableItems(ByRef value As Variant)
End Property

Public Property Get SomeSetting() As String
End Property

Public Property Let SomeSetting(ByVal value As String)
End Property

Public Property Get SomeOtherSetting() As Boolean
End Property

Public Property Let SomeOtherSetting(ByVal value As Boolean)
End Property

Observe que as propriedades Timestamp e UserName só expõem um acessador Property Get. Agora a classe SomeModel pode implementar essa interface:

Option Explicit
Implements ISomeModel

Private Type TModel
    Timestamp As Date
    UserName As String
    SomeSetting As String
    SomeOtherSetting As Boolean
    AvailableItems As Variant
End Type
Private this As TModel

Private Property Get ISomeModel_Timestamp() As Date
    ISomeModel_Timestamp = this.Timestamp
End Property

Private Property Get ISomeModel_UserName() As String
    ISomeModel_UserName = this.UserName
End Property

Private Property Get ISomeModel_AvailableItems() As Variant
    ISomeModel_AvailableItems = this.AvailableItems
End Property

Private Property Let ISomeModel_AvailableItems(ByRef value As Variant)
    this.AvailableItems = value
End Property

Private Property Get ISomeModel_SomeSetting() As String
    ISomeModel_SomeSetting = this.SomeSetting
End Property

Private Property Let ISomeModel_SomeSetting(ByVal value As String)
    this.SomeSetting = value
End Property

Private Property Get ISomeModel_SomeOtherSetting() As Boolean
    ISomeModel_SomeOtherSetting = this.SomeOtherSetting
End Property

Private Property Let ISomeModel_SomeOtherSetting(ByVal value As Boolean)
    this.SomeOtherSetting = value
End Property

Public Property Get Timestamp() As Date
    Timestamp = this.Timestamp
End Property

Public Property Let Timestamp(ByVal value As Date)
    this.Timestamp = value
End Property

Public Property Get UserName() As String
    UserName = this.UserName
End Property

Public Property Let UserName(ByVal value As String)
    this.UserName = value
End Property

Public Property Get AvailableItems() As Variant
    AvailableItems = this.AvailableItems
End Property

Public Property Let AvailableItems(ByRef value As Variant)
    this.AvailableItems = value
End Property

Public Property Get SomeSetting() As String
    SomeSetting = this.SomeSetting
End Property

Public Property Let SomeSetting(ByVal value As String)
    this.SomeSetting = value
End Property

Public Property Get SomeOtherSetting() As Boolean
    SomeOtherSetting = this.SomeOtherSetting
End Property

Public Property Let SomeOtherSetting(ByVal value As Boolean)
    this.SomeOtherSetting = value
End Property

Os membros da interface são todos Private, e todos os membros da interface devem ser implementados para que o código seja compilado. Os membros Public não fazem parte da interface e, portanto, não são expostos ao código escrito na interface ISomeModel.

Usando um Método de fábrica para simular um construtor #

Usando um atributo https://www.wikiod.com/pt/vba/atributos#VB_PredeclaredId podemos fazer com que a classe SomeModel tenha uma instância padrão e escreva uma função que funcione como um membro de nível de tipo (Shared em VB.NET, static em C#) que o código do cliente pode chamar sem precisar primeiro criar uma instância, como fizemos aqui:

Função privada CreateViewModel() como ISomeModel Resultado escuro como ISomeModel Definir resultado = SomeModel.Create(Now, Environ$(“UserName”)) result.AvailableItems = GetAvailableItems Definir CreateViewModel = resultado Função Final

Este método de fábrica atribui os valores de propriedade que são somente leitura quando acessados ​​a partir da interface ISomeModel, aqui Timestamp e UserName:

Public Function Create(ByVal pTimeStamp As Date, ByVal pUserName As String) As ISomeModel
    With New SomeModel
        .Timestamp = pTimeStamp
        .UserName = pUserName
        Set Create = .Self
    End With
End Function

Public Property Get Self() As ISomeModel
    Set Self = Me
End Property

E agora podemos codificar na interface ISomeModel, que expõe Timestamp e UserName como propriedades somente leitura que nunca podem ser reatribuídas (desde que o código seja escrito na interface).

Polimorfismo #

Polimorfismo é a capacidade de apresentar a mesma interface para diferentes implementações subjacentes. #

A capacidade de implementar interfaces permite desacoplar completamente a lógica do aplicativo da interface do usuário, ou do banco de dados, ou desta ou daquela planilha.

Digamos que você tenha uma interface ISomeView que o próprio formulário implementa:

Option Explicit

Public Property Get IsCancelled() As Boolean
End Property

Public Property Get Model() As ISomeModel
End Property

Public Property Set Model(ByVal value As ISomeModel)
End Property

Public Sub Show()
End Sub

O code-behind do formulário pode ficar assim:

Option Explicit 
Implements ISomeView

Private Type TView
    IsCancelled As Boolean
    Model As ISomeModel
End Type
Private this As TView

Private Property Get ISomeView_IsCancelled() As Boolean
    ISomeView_IsCancelled = this.IsCancelled
End Property

Private Property Get ISomeView_Model() As ISomeModel
    Set ISomeView_Model = this.Model
End Property

Private Property Set ISomeView_Model(ByVal value As ISomeModel)
    Set this.Model = value
End Property

Private Sub ISomeView_Show()
    Me.Show vbModal
End Sub

Private Sub SomeOtherSettingInput_Change()
    this.Model.SomeOtherSetting = CBool(SomeOtherSettingInput.Value)
End Sub

'...other event handlers...

Private Sub OkButton_Click()
    Me.Hide
End Sub

Private Sub CancelButton_Click()
    this.IsCancelled = True
    Me.Hide
End Sub

Private Sub UserForm_QueryClose(Cancel As Integer, CloseMode As Integer)
    If CloseMode = VbQueryClose.vbFormControlMenu Then
        Cancel = True
        this.IsCancelled = True
        Me.Hide
    End If
End Sub

Mas então, nada proíbe a criação de outro módulo de classe que implemente a interface ISomeView sem ser um formulário de usuário - esta poderia ser uma classe SomeViewMock:

Option Explicit
Implements ISomeView

Private Type TView
    IsCancelled As Boolean
    Model As ISomeModel
End Type
Private this As TView

Public Property Get IsCancelled() As Boolean
    IsCancelled = this.IsCancelled
End Property

Public Property Let IsCancelled(ByVal value As Boolean)
    this.IsCancelled = value
End Property

Private Property Get ISomeView_IsCancelled() As Boolean
    ISomeView_IsCancelled = this.IsCancelled
End Property

Private Property Get ISomeView_Model() As ISomeModel
    Set ISomeView_Model = this.Model
End Property

Private Property Set ISomeView_Model(ByVal value As ISomeModel)
    Set this.Model = value
End Property

Private Sub ISomeView_Show()
    'do nothing
End Sub

E agora podemos alterar o código que funciona com um UserForm e fazê-lo funcionar fora da interface ISomeView, por exemplo. dando-lhe o formulário como um parâmetro em vez de instanciar:

Public Sub DoSomething(ByVal view As ISomeView)
    With view
        Set .Model = CreateViewModel
        .Show
        If .IsCancelled Then Exit Sub
        ProcessUserData .Model
    End With
End Sub

Como o método DoSomething depende de uma interface (ou seja, uma abstração) e não de uma classe concreta (por exemplo, um UserForm específico), podemos escrever um teste de unidade automatizado que garanta que ProcessUserData não seja executado quando view.IsCancelled for True, fazendo nosso teste criar uma instância SomeViewMock, definindo sua propriedade IsCancelled para True, e passando-a para DoSomething.

O código testável depende de abstrações #

A escrita de testes de unidade em VBA pode ser feita, existem add-ins por aí que até o integram ao IDE. Mas quando o código é fortemente acoplado com uma planilha, um banco de dados, um formulário ou o sistema de arquivos, o teste de unidade começa a exigir uma planilha, banco de dados, formulário ou sistema de arquivos real - e essas dependências são novas pontos de falha de controle que o código testável deve isolar, para que os testes de unidade não exijam uma planilha, banco de dados, formulário ou sistema de arquivos real.

Ao escrever código em interfaces, de uma forma que permite que o código de teste injete implementações de stub/mock (como o exemplo SomeViewMock acima), você pode escrever testes em um “ambiente controlado” e simular o que acontece quando cada um das 42 permutações possíveis de interações do usuário nos dados do formulário, sem sequer exibir um formulário e clicar manualmente em um controle de formulário.
