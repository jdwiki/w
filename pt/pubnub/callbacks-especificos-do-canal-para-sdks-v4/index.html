Retornos de chamada específicos do canal do SDK do JavaScript v4 #

No PubNub JavaScript v3, você pode implementar um retorno de chamada exclusivo para cada canal em que se inscreveu, desde que tenha chamado a função de inscrição para cada canal e implementado o retorno de chamada nessa inscrição assim:

var pubnub = new PubNub({
    publishKey: "your-pub-key",
    subscribeKey: "your-sub-key"
});

pubnub.subscribe({
    channel: 'ch1',
    message: function (m) {
        console.log(m + " ch1 callback");
    }
});

pubnub.subscribe({
    channel: 'ch2',
    message: function (m) {
        console.log(m + " ch2 callback: ");
    }
});

Então, no código acima, se você publicar a mensagem "hello" para ch1:

publish({channel: "ch1", message: "hello"});

…então você obteria a saída, hello ch1 callback. E é claro que se você publicasse a mesma mensagem para ch2, você obteria a saída hello ch2 callback.

A capacidade de fornecer um retorno de chamada personalizado para cada canal ou grupo de canais é útil e muitas vezes mais desejável do que criar um retorno de chamada monolítico com um longo if-then-else de nomes de canais e código a ser executado para cada condição. E há melhores práticas para usar do que o exemplo simples acima, mas eu queria facilitar a comparação e contraste com a forma como o PubNub JavaScript SDK v4 (e qualquer PubNub SDK v4) é projetado.

Os SDKs do PubNub v4 usam um único ouvinte para receber todos os eventos message, presence e status. Isso significa que quando você se ‘inscreve’ em um canal, você fornece apenas o(s) nome(s) do canal em vez de uma função de ‘retorno de chamada de mensagem’ complementar. Em JavaScript SDK v4, o listener tem esta aparência:

pubnub.addListener({
    message: function(m) {
        console.log(JSON.stringify(m));
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(s) {
        console.log(JSON.stringify(s));
    }
});

Eu sei que muitos desenvolvedores estarão se perguntando como migrar o código subscribe com retornos de chamada de mensagem exclusivos do SDK v3 com esse tipo de design sem recorrer ao código condicional do nome do canal sem fim que mencionei acima, assim:

pubnub.addListener({
    message: function(m) {
        if (m.subscribedChannel == 'ch1') {
            console.log(m.message + "ch1 callback");
        }
        else if (m.subscribedChannel == 'ch2') {
            console.log(m.message + "ch2 callback");
        }
        else {
            console.log(m.message + "default callback");
        }
    }
    // removed the other two callbacks for brevity purposes
});

O parâmetro m que é passado para o ouvinte de mensagem acima tem a seguinte estrutura, que é um design diferente do design de vários parâmetros do SDK JavaScript v3.

{
    "channel": "ch1",
    "subscription": <undefined>,
    "timetoken": "14721821326909151",
    "message": "hello"
}

Esse timetoken é o publish timetoken real. Para desenvolvedores experientes do PubNub, você deve estar animado ao ver que isso agora está disponível para o assinante, mas não vamos entrar em por que isso é importante e poderoso agora.

Agora, eu não esperaria que nenhum desenvolvedor JavaScript experiente escrevesse código como representado acima e muitos desenvolvedores avançados já devem saber o que fazer. Mas para os desenvolvedores que estão no nível iniciante ao intermediário com JavaScript, a solução pode não ser imediatamente óbvia. No entanto, eu sei que uma vez que você veja essa abordagem de design simples abaixo, ela abrirá seus olhos para as possibilidades ilimitadas da linguagem JavaScript - aqui vamos nós.

Vamos reafirmar o requisito:

Para cada canal em que me inscrevo, quero poder fornecer um função única para invocar quando uma mensagem é recebida naquele canal. E eu quero evitar a abordagem de nome de canal condicional monolítico.

Então, o que precisamos fazer primeiro é criar uma tabela de pesquisa de função. Uma hashtable para ser exato. Esta tabela terá nomes de canais como chaves e funções (o código a ser invocado quando uma mensagem for recebida nesse canal) como valores. Se você é um pouco novo em JavaScript ou está codificando com a linguagem há algum tempo, mas ainda não mergulhou nos recursos da linguagem, isso pode parecer estranho e impossível, mas é realmente como o JavaScript funciona e você tem feito isso o tempo todo e realmente não sabia. Vamos definir nossa tabela de pesquisa de funções:

ftbl = {};

É isso - você tem um objeto que conterá seus canais e funções. Bem simples, certo? Mas como você adiciona os canais e funções? Assim como qualquer outra chave/valor.

ftbl.ch1 = function(m){console.log(m.message + " ch1 callback")};
ftbl.ch2 = function(m){console.log(m.message + " ch1 callback")};

…e assim por diante com cada canal e função que você deseja definir. E você não precisa criar todas as teclas de canal/função em um ponto do seu código. Você pode adicionar cada canal/função ao ftbl ao se inscrever em um canal.

ftbl.ch10 = function(m){console.log(m.message + " ch10 callback")};
pubnub.pubnub.subscribe({
    channels: ['ch10'] 
});

OK, isso é bastante simples, e você pode ficar mais sofisticado e avançado com a forma como faz isso, mas apenas mantendo-o básico. Mas como você invoca essa função para o canal ao qual ela está vinculada? É por isso que JavaScript é tão legal, poderoso e fácil, especialmente se você vem de uma linguagem rígida e estruturada como Java - confira.

pubnub.addListener({
    message: function(m) {
        // use the channel name to get the function
        // from ftbl and invoke it 
        ftbl[m.subscribedChannel](m);
    },
    presence: function(p) {
        console.log(JSON.stringify(p));
    },
    status: function(statusEvent) {
        console.log(JSON.stringify(s));
    }
});

Isso é tudo o que há para isso. Basta pegar a função de ftbl usando o nome do canal que é passado para a função de retorno de chamada message do ouvinte e adicionar (m) ao final dela e boom, ela executa sua função.

Se o canal for ch10, ftbl[m.subscribedChannel](m) apenas invoca function(m){console.log(m.message + "ch10 callback")} passando o parâmetro m que sua função pode analisar e explorar conforme necessário.

Então, chamando a seguinte função publish:

pubnub.publish(
    {
        channel : "ch10",
        message : "hello"
    }, 
    function(status, response) {
        console.log(status, response);
    }
);

… resultará na exibição da seguinte mensagem: hello ch10 callback. E o equivalente para publicação em outros canais que você definiu em sua tabela de pesquisa de função. Não se esqueça de fornecer um padrão para canais desconhecidos.

E não se esqueça dos callbacks presence e status no listener. Isso pode ser apenas mais duas tabelas de pesquisa de funções ou apenas um ftbl um pouco mais complexo:

ftbl.message.ch1 = function(m){console.log(m.message + " ch1 message cb")};
ftbl.presence.ch1 = function(m){console.log(m.message + " ch1 presence cb")};
ftbl.status.ch1 = function(m){console.log(m.message + " ch1 status cb")};

ou

ftbl.ch1.message = ...
ftbl.ch1.presence = ...
ftbl.ch1.status = ...

Eu gosto mais do primeiro do que do último, mas realmente depende de você. E você provavelmente quer algum código genérico de manipulação de eventos de ‘status’ de qualquer maneira, mas isso dependerá de seus requisitos específicos.

E isso pode ficar ainda mais complicado e robusto com funções opcionais por canal que podem ser invocadas dependendo de alguns dados adicionais que você incorpora na carga útil da mensagem.

Então, aqui está, retornos de chamada exclusivos para cada canal. Chega de desculpas para não querer migrar de 3x para 4x. Mas se você tiver dúvidas sobre a migração, não hesite em entrar em contato com o Suporte PubNub e nós o ajudaremos a seguir em frente. E não se esqueça de revisar o Guia de Migração do SDK do PubNub JavaScript v3 para v4.
