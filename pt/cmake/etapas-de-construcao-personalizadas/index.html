As etapas de compilação personalizadas são úteis para executar destinos personalizados na compilação do projeto ou para copiar arquivos facilmente para que você não precise fazer isso manualmente (talvez dlls?). Aqui vou mostrar dois exemplos, o primeiro é para copiar dlls (em particular dlls Qt5) para o diretório binário de seus projetos (Debug ou Release) e o segundo é para executar um destino personalizado (Doxygen neste caso) em sua solução (se você estiver usando o Visual Studio).

Como você pode ver, você pode fazer muito com alvos de compilação personalizados e etapas no cmake, mas você deve ter cuidado ao usá-los, especialmente ao copiar dlls. Embora seja conveniente fazê-lo, às vezes pode resultar no que é carinhosamente chamado de “inferno da dll”.

Basicamente, isso significa que você pode se perder em quais dlls seu executável realmente depende, quais são carregados e quais ele precisa ser executado (talvez por causa da variável de caminho do seu computador).

Além da ressalva acima, sinta-se à vontade para fazer com que os alvos personalizados façam o que você quiser! Eles são poderosos e flexíveis e são uma ferramenta inestimável para qualquer projeto cmake.

Exemplo de cópia de dll Qt5 #

Então, digamos que você tenha um projeto que dependa do Qt5 e precise copiar as dlls relevantes para o diretório de compilação e não queira fazer isso manualmente; você pode fazer o seguinte:

cmake_minimum_required(VERSION 3.0)
project(MyQtProj LANGUAGES C CXX)
find_package(Qt5 COMPONENTS Core Gui Widgets)
#...set up your project

# add the executable
add_executable(MyQtProj ${PROJ_SOURCES} ${PROJ_HEADERS})
    
add_custom_command(TARGET MyQtProj POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Core> $<TARGET_FILE_DIR:MyQtProj>
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Gui> $<TARGET_FILE_DIR:MyQtProj>
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Widgets> $<TARGET_FILE_DIR:MyQtProj>
    )

Então agora toda vez que você construir seu projeto, se as dlls de destino forem alteradas que você deseja copiar, elas serão copiadas depois que seu destino (neste caso, o executável principal) for compilado (observe o comando copy_if_different); caso contrário, eles não serão copiados.

Além disso, observe o uso de expressões de gerador aqui. A vantagem de usar isso é que você não precisa dizer explicitamente onde copiar dlls ou quais variantes usar. Para poder usá-los, porém, o projeto que você está usando (Qt5 neste caso) deve ter destinos importados.

Se você estiver compilando em depuração, o CMake saberá (com base no destino importado) copiar o Qt5Cored.dll, Qt5Guid.dll e Qt5Widgetsd.dll para a pasta Debug da sua pasta de compilação. Se você estiver compilando na versão, as versões de lançamento dos .dlls serão copiadas para a pasta de lançamento.

Executando um destino personalizado #

Você também pode criar um destino personalizado para ser executado quando desejar executar uma tarefa específica. Estes são normalmente executáveis ​​que você executa para fazer coisas diferentes. Algo que pode ser de uso particular é executar Doxygen para gerar documentação para seu projeto. Para fazer isso, você pode fazer o seguinte em seu CMakeLists.txt (para simplificar, continuaremos nosso exemplo de projeto Qt5):

cmake_minimum_required(VERSION 3.0)
project(MyQtProj LANGUAGES C CXX)
find_package(Qt5 COMPONENTS Core Gui Widgets)
#...set up your project

add_executable(MyQtProj ${PROJ_SOURCES} ${PROJ_HEADERS})
    
add_custom_command(TARGET MyQtProj POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Core> $<TARGET_FILE_DIR:MyQtProj>
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Gui> $<TARGET_FILE_DIR:MyQtProj>
    COMMAND ${CMAKE_COMMAND} -E copy_if_different $<TARGET_FILE:Qt5::Widgets> $<TARGET_FILE_DIR:MyQtProj>
    )

#Add target to build documents from visual studio.
set(DOXYGEN_INPUT ${CMAKE_CURRENT_SOURCE_DIR}/Doxyfile)
#set the output directory of the documentation
set(DOXYGEN_OUTPUT_DIR ${CMAKE_CURRENT_SOURCE_DIR}/docs)
# sanity check...
message("Doxygen Output ${DOXYGEN_OUTPUT_DIR}")
find_package(Doxygen)

if(DOXYGEN_FOUND)
    # create the output directory where the documentation will live
    file(MAKE_DIRECTORY ${DOXYGEN_OUTPUT_DIR})
    # configure our Doxygen configuration file. This will be the input to the doxygen
    # executable
    configure_file(${CMAKE_CURRENT_SOURCE_DIR}/Doxyfile.in
    ${CMAKE_CURRENT_BINARY_DIR}/Doxyfile @ONLY)

# now add the custom target. This will create a build target called 'DOCUMENTATION' 
# in your project
ADD_CUSTOM_TARGET(DOCUMENTATION
  COMMAND ${CMAKE_COMMAND} -E echo_append "Building API Documentation..."
  COMMAND ${CMAKE_COMMAND} -E make_directory ${DOXYGEN_OUTPUT_DIR}
  COMMAND ${DOXYGEN_EXECUTABLE} ${CMAKE_CURRENT_BINARY_DIR}/Doxyfile
    WORKING_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR}
  COMMAND ${CMAKE_COMMAND} -E echo "Done."
  WORKING_DIRECTORY ${DOXYGEN_OUTPUT_DIR})

endif(DOXYGEN_FOUND)

Agora, quando criarmos nossa solução (novamente supondo que você esteja usando o Visual Studio), você terá um destino de compilação chamado DOCUMENTATION que pode ser compilado para gerar novamente a documentação do seu projeto.
