Os protocolos são uma maneira de especificar como usar um objeto. Eles descrevem um conjunto de propriedades e métodos que uma classe, estrutura ou enumeração deve fornecer, embora os protocolos não imponham restrições à implementação.

Um protocolo Swift é uma coleção de requisitos que os tipos em conformidade devem implementar. O protocolo pode então ser usado na maioria dos lugares onde um tipo é esperado, por exemplo, Arrays e requisitos genéricos.

Os membros do protocolo sempre compartilham o mesmo qualificador de acesso de todo o protocolo e não podem ser especificados separadamente. Embora um protocolo possa restringir o acesso com requisitos de getter ou setter, conforme exemplos acima.

Para obter mais informações sobre protocolos, consulte The Swift Programming Language.

Protocolos Objective-C são semelhantes aos protocolos Swift.

Os protocolos também são comparáveis ​​a interfaces Java.

Noções básicas de protocolo #

Sobre protocolos #

Um protocolo especifica inicializadores, propriedades, funções, subscritos e tipos associados exigidos de um tipo de objeto Swift (class, struct ou enum) em conformidade com o protocolo. Em algumas linguagens, ideias semelhantes para especificações de requisitos de objetos subsequentes são conhecidas como “interfaces”.

Um Protocolo declarado e definido é um Tipo, por si só, com uma assinatura de seus requisitos declarados, um pouco semelhante à maneira como as Funções Swift são um Tipo com base em sua assinatura de parâmetros e retornos.

As especificações do protocolo Swift podem ser opcionais, explicitamente exigidas e/ou implementadas por padrão por meio de um recurso conhecido como Protocol Extensions. Um tipo de objeto Swift (class, struct ou enum) que deseja estar em conformidade com um protocolo que é desenvolvido com extensões para todos os requisitos especificados precisa apenas declarar seu desejo de estar em conformidade total. O recurso de implementações padrão de Extensões de Protocolo pode ser suficiente para cumprir todas as obrigações de conformidade com um Protocolo.

Os protocolos podem ser herdados por outros protocolos. Isso, em conjunto com as extensões de protocolo, significa que os protocolos podem e devem ser considerados um recurso significativo do Swift.

Protocolos e Extensões são importantes para realizar os objetivos e abordagens mais amplos do Swift para flexibilidade de design de programas e processos de desenvolvimento. O principal objetivo declarado do recurso de Protocolo e Extensão do Swift é facilitar o design composicional na arquitetura e desenvolvimento do programa. Isso é conhecido como Programação Orientada a Protocolo. Os veteranos duros consideram isso superior a um foco no design OOP.

Protocolos definem interfaces que podem ser implementadas por qualquer struct, class ou enum:

protocol MyProtocol {
    init(value: Int)                      // required initializer
    func doSomething() -> Bool            // instance method
    var message: String { get }           // instance read-only property
    var value: Int { get set }            // read-write instance property
    subscript(index: Int) -> Int { get }  // instance subscript
    static func instructions() -> String  // static method
    static var max: Int { get }           // static read-only property
    static var total: Int { get set }     // read-write static property
}

Propriedades definidas em protocolos devem ser anotadas como { get } ou { get set }. { get } significa que a propriedade deve ser gettable e, portanto, pode ser implementada como qualquer tipo de propriedade. { get set } significa que a propriedade deve ser configurável e também gettable.

Um struct, classe ou enum pode conforme um protocolo:

struct MyStruct : MyProtocol {
    // Implement the protocol's requirements here
}
class MinhaClasse : MeuProtocolo {
    // Implement the protocol's requirements here
}
enum MeuEnum : MeuProtocolo {
    case caseA, caseB, caseC
    // Implement the protocol's requirements here
}

Um protocolo também pode definir uma implementação padrão para qualquer um de seus requisitos por meio de uma extensão:

extension MyProtocol {
    
    // default implementation of doSomething() -> Bool
    // conforming types will use this implementation if they don't define their own
    func doSomething() -> Bool {
        print("do something!")
        return true
    }
}

Um protocolo pode ser usado como um tipo, desde que não tenha [requisitos do associatedtype][7]:

func doStuff(object: MeuProtocolo) {
    // All of MyProtocol's requirements are available on the object
    print(object.message)
    print(object.doSomething())
}

let items : [MyProtocol] = [MyStruct(), MyClass(), MyEnum.caseA]

Você também pode definir um tipo abstrato que esteja em conformidade com os protocolos múltiplos:

Com o Swift 3 ou superior, isso é feito separando a lista de protocolos com um e comercial (&):

func doStuff(object: MyProtocol & AnotherProtocol) {
    // ...
}

let items : [MyProtocol & AnotherProtocol] = [MyStruct(), MyClass(), MyEnum.caseA]

Versões mais antigas têm sintaxe protocol<...> onde os protocolos são uma lista separada por vírgulas entre os colchetes <>.

protocolo OutroProtocolo {
    func doSomethingElse()
}

func doStuff(object: protocolo<MeuProtocolo, OutroProtocolo>) {
    
    // All of MyProtocol & AnotherProtocol's requirements are available on the object
    print(object.message)
    object.doSomethingElse()
}

// MyStruct, MyClass & MyEnum agora devem estar em conformidade com MyProtocol & AnotherProtocol
let items : [protocolo<MeuProtocolo, OutroProtocolo>] = [MinhaEstrutura(), MinhaClasse(), MeuEnum.caseA]

Os tipos existentes podem ser estendidos para obedecer a um protocolo:

extension String : MyProtocol {
    // Implement any requirements which String doesn't already satisfy
}

[7]: https://www.wikiod.com/pt/swift/protocolos#Requisitos de tipo associado

Padrão de delegado #

Um delegate é um padrão de design comum usado nas estruturas Cocoa e CocoaTouch, onde uma classe delega a responsabilidade de implementar algumas funcionalidades para outra. Isso segue um princípio de separação de interesses, onde a classe de estrutura implementa a funcionalidade genérica enquanto uma instância de delegado separada implementa o caso de uso específico.

Outra maneira de examinar o padrão de delegado é em termos de comunicação de objetos. ‘Objetos’ geralmente precisam conversar entre si e, para isso, um objeto precisa estar em conformidade com um ‘protocolo’ para se tornar um delegado de outro Objeto. Uma vez que esta configuração tenha sido feita, o outro objeto fala com seus delegados quando coisas interessantes acontecem.

Por exemplo, uma exibição na interface do usuário para exibir uma lista de dados deve ser responsável apenas pela lógica de como os dados são exibidos, não por decidir quais dados devem ser exibidos.

Vamos mergulhar em um exemplo mais concreto. se você tiver duas classes, um pai e um filho:

class Parent { }
class Child { }

E você deseja notificar o pai de uma alteração da criança.

No Swift, os delegados são implementados usando uma declaração protocol e assim vamos declarar um protocol que o delegate implementará. Aqui delegado é o objeto pai.

protocol ChildDelegate: class {
    func childDidSomething()
}

O filho precisa declarar uma propriedade para armazenar a referência ao delegado:

class Child {
    weak var delegate: ChildDelegate?
}

Observe que a variável delegate é opcional e o protocolo ChildDelegate está marcado para ser implementado apenas por tipo de classe (sem isso a variável delegate não pode ser declarada como uma referência fraca evitando qualquer ciclo de retenção. Isso significa que se a variável delegate não for mais referenciada em nenhum outro lugar, ela será liberada). Isso é para que a classe pai registre o delegado apenas quando necessário e disponível.

Além disso, para marcar nosso delegado como ‘fraco’, devemos restringir nosso protocolo ChildDelegate para referenciar tipos adicionando a palavra-chave ‘class’ na declaração do protocolo.

Neste exemplo, quando o filho faz algo e precisa notificar seu pai, o filho chamará:

delegate?.childDidSomething()

Se o delegado tiver sido definido, o delegado será notificado de que a criança fez algo.

A classe pai precisará estender o protocolo ChildDelegate para poder responder às suas ações. Isso pode ser feito diretamente na classe pai:

class Parent: ChildDelegate {
    ...

    func childDidSomething() {
        print("Yay!")
    }
}

Ou usando uma extensão:

extension Parent: ChildDelegate {
    func childDidSomething() {
        print("Yay!")
    }
}

O pai também precisa dizer à criança que é o delegado da criança:

// In the parent
let child = Child()
child.delegate = self

Por padrão, um protocolo Swift não permite que uma função opcional seja implementada. Estes só podem ser especificados se o seu protocolo estiver marcado com o atributo @objc e o modificador opcional.

Por exemplo, UITableView implementa o comportamento genérico de uma visualização de tabela no iOS, mas o usuário deve implementar duas classes delegadas chamadas UITableViewDelegate e UITableViewDataSource que implementam a aparência e o comportamento das células específicas.

@objc public protocol UITableViewDelegate : NSObjectProtocol, UIScrollViewDelegate {
        
        // Display customization
        optional public func tableView(tableView: UITableView, willDisplayCell cell: UITableViewCell, forRowAtIndexPath indexPath: NSIndexPath)
        optional public func tableView(tableView: UITableView, willDisplayHeaderView view: UIView, forSection section: Int)
        optional public func tableView(tableView: UITableView, willDisplayFooterView view: UIView, forSection section: Int)
        optional public func tableView(tableView: UITableView, didEndDisplayingCell cell: UITableViewCell, forRowAtIndexPath indexPath: NSIndexPath)
        ...
}

Você pode implementar este protocolo alterando sua definição de classe, por exemplo:

class MyViewController : UIViewController, UITableViewDelegate

Qualquer método não marcado como opcional na definição do protocolo (UITableViewDelegate neste caso) deve ser implementado.

Requisitos de tipo associado #

Os protocolos podem definir requisitos de tipo associado usando a palavra-chave associatedtype:

Contêiner de protocolo {
    associatedtype Element
    var count: Int { get }
    subscript(index: Int) -> Element { get set }
}

Protocolos com requisitos de tipo associados podem ser usados ​​apenas como restrições genéricas:

// Estes são NÃO permitidos, porque o Container tem requisitos de tipo associados:
func displayValues(container: Container) { ... }
class MyClass { let container: Container }
// > erro: o protocolo 'Container' só pode ser usado como uma restrição genérica
// > porque tem requisitos de tipo próprio ou associado

// Estes são permitidos:
func displayValues<T: Container>(container: T) { ... }
class MyClass<T: Container> { let container: T }

Um tipo que está em conformidade com o protocolo pode satisfazer um requisito de associatedtype implicitamente, fornecendo um determinado tipo onde o protocolo espera que o associatedtype apareça:

struct ContainerOfOne<T>: Container {
    let count = 1          // satisfy the count requirement
    var value: T
    
    // satisfy the subscript associatedtype requirement implicitly,
    // by defining the subscript assignment/return type as T
    // therefore Swift will infer that T == Element
    subscript(index: Int) -> T {
        get {
            precondition(index == 0)
            return value
        }
        set {
            precondition(index == 0)
            value = newValue
        }
    }
}

let container = ContainerOfOne(value: "Olá")

(Observe que para adicionar clareza a este exemplo, o tipo de espaço reservado genérico é chamado de T – um nome mais adequado seria Element, que sombrearia o associatedtype Element do protocolo. O compilador ainda inferirá que o placeholder genérico Element é usado para satisfazer o requisito associatedtype Element.)

Um associatedtype também pode ser satisfeito explicitamente através do uso de um typealias:

struct ContainerOfOne<T>: Container {

    typealias Element = T
    subscript(index: Int) -> Element { ... }

    // ...
}

O mesmo vale para extensões:

// Expose an 8-bit integer as a collection of boolean values (one for each bit).
extension UInt8: Container {

    // as noted above, this typealias can be inferred
    typealias Element = Bool

    var count: Int { return 8 }
    subscript(index: Int) -> Bool {
        get {
            precondition(0 <= index && index < 8)
            return self & 1 << UInt8(index) != 0
        }
        set {
            precondition(0 <= index && index < 8)
            if newValue {
                self |= 1 << UInt8(index)
            } else {
                self &= ~(1 << UInt8(index))
            }
        }
    }
}

Se o tipo em conformidade já satisfaz o requisito, nenhuma implementação é necessária:

extension Array: Container {}  // Array satisfies all requirements, including Element

Protocolos somente de classe #

Um protocolo pode especificar que apenas uma class pode implementá-lo usando a palavra-chave class em sua lista de herança. Essa palavra-chave deve aparecer antes de qualquer outro protocolo herdado nesta lista.

protocolo ClassOnlyProtocol: class, SomeOtherProtocol {
    // Protocol requirements 
}

Se um tipo não-classe tentar implementar ClassOnlyProtocol, um erro do compilador será gerado.

struct MyStruct: ClassOnlyProtocol { 
    // error: Non-class type 'MyStruct' cannot conform to class protocol 'ClassOnlyProtocol'  
}

Outros protocolos podem herdar do ClassOnlyProtocol, mas terão o mesmo requisito somente de classe.

protocol MyProtocol: ClassOnlyProtocol {
    // ClassOnlyProtocol Requirements
    // MyProtocol Requirements
}

class MySecondClass: MyProtocol {
    // ClassOnlyProtocol Requirements
    // MyProtocol Requirements
}

Semântica de referência de protocolos somente de classe #

O uso de um protocolo somente de classe permite [semântica de referência] 2 quando o tipo de conformidade é desconhecido.

protocol Foo : class {
    var bar : String { get set }
}

func takesAFoo(foo:Foo) {

    // this assignment requires reference semantics,
    // as foo is a let constant in this scope.
    foo.bar = "new value"
}

Neste exemplo, como Foo é um protocolo somente de classe, a atribuição a bar é válida, pois o compilador sabe que foo é um tipo de classe e, portanto, possui semântica de referência.

Se o Foo não fosse um protocolo somente de classe, um erro do compilador seria gerado – já que o tipo conforme poderia ser um tipo de valor, o que exigiria uma anotação var para ser mutável.

protocol Foo {
    var bar : String { get set }
}

func takesAFoo(foo:Foo) {
    foo.bar = "new value" // error: Cannot assign to property: 'foo' is a 'let' constant
}


func levaAFoo(foo:Foo) {
    var foo = foo // mutable copy of foo
    foo.bar = "new value" // no error – satisfies both reference and value semantics
}

Variáveis ​​fracas do tipo de protocolo #

Ao aplicar o modificador weak a uma variável do tipo de protocolo, esse tipo de protocolo deve ser somente classe, pois weak só pode ser aplicado a tipos de referência.

weak var weakReference : ClassOnlyProtocol?

Extensão de protocolo para uma classe de conformidade específica #

Você pode escrever a implementação de protocolo padrão para uma classe específica.

protocol MyProtocol {
    func doSomething()
}

extension MyProtocol where Self: UIViewController {
    func doSomething() {
        print("UIViewController default protocol implementation")
    }
}

class MyViewController: UIViewController, MyProtocol { }

let vc = MyViewController()
vc.doSomething() // Prints "UIViewController default protocol implementation"

Usando o protocolo RawRepresentable (Extensible Enum) #

// RawRepresentable has an associatedType RawValue.
// For this struct, we will make the compiler infer the type
// by implementing the rawValue variable with a type of String
//
// Compiler infers RawValue = String without needing typealias
//
struct NotificationName: RawRepresentable {
    let rawValue: String

    static let dataFinished = NotificationNames(rawValue: "DataFinishedNotification")
}

Esta estrutura pode ser estendida para outro lugar para adicionar casos

extension NotificationName {
    static let documentationLaunched = NotificationNames(rawValue: "DocumentationLaunchedNotification")
}

E uma interface pode ser projetada em torno de qualquer tipo RawRepresentable ou especificamente sua estrutura de enumeração

func post(notification notification: NotificationName) -> Void {
    // use notification.rawValue
}

No site da chamada, você pode usar a abreviação de sintaxe de ponto para o NotificationName typesafe

post(notification: .dataFinished)

Usando a função genérica RawRepresentable

// RawRepresentable has an associate type, so the 
// method that wants to accept any type conforming to
// RawRepresentable needs to be generic
func observe<T: RawRepresentable>(object: T) -> Void {
    // object.rawValue
}

Implementando o protocolo Hashable #

Os tipos usados ​​em Sets e Dictionaries(key) devem estar em conformidade com o protocolo Hashable que herda do protocolo Equatable.

O tipo personalizado em conformidade com o protocolo Hashable deve ser implementado

  • Uma propriedade calculada hashValue
  • Defina um dos operadores de igualdade, ou seja, == ou !=.

O exemplo a seguir implementa o protocolo Hashable para um struct personalizado:

struct Cell {
    var row: Int
    var col: Int
    
    init(_ row: Int, _ col: Int) {
        self.row = row
        self.col = col
    }
}

extension Cell: Hashable {
    
    // Satisfy Hashable requirement
    var hashValue: Int {
        get {
            return row.hashValue^col.hashValue
        }
    }

    // Satisfy Equatable requirement
    static func ==(lhs: Cell, rhs: Cell) -> Bool {
        return lhs.col == rhs.col && lhs.row == rhs.row
    }
    
}

// Now we can make Cell as key of dictonary
var dict = [Cell : String]()

dict[Cell(0, 0)] = "0, 0"
dict[Cell(1, 0)] = "1, 0"
dict[Cell(0, 1)] = "0, 1"

// Also we can create Set of Cells
var set = Set<Cell>()

set.insert(Cell(0, 0))
set.insert(Cell(1, 0))

Observação: não é necessário que valores diferentes no tipo personalizado tenham valores de hash diferentes, colisões são aceitáveis. Se os valores de hash forem iguais, o operador de igualdade será usado para determinar a igualdade real.
