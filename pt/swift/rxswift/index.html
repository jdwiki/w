Descarte #

Após a criação da assinatura, é importante gerenciar sua desalocação correta.

Os docs nos disseram que

Se uma sequência terminar em tempo finito, não chamar descarte ou não usar addDisposableTo(disposeBag) não causará vazamentos de recursos permanentes. No entanto, esses recursos serão utilizados até que a sequência seja concluída, seja finalizando a produção de elementos ou retornando um erro.

Há duas maneiras de desalocar recursos.

  1. Usando o operador disposeBagse addDisposableTo.
  2. Usando o operador takeUntil.

No primeiro caso, você passa manualmente a assinatura para o objeto DisposeBag, que limpa corretamente toda a memória tomada.

let bag = DisposeBag()
Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(bag)

Na verdade, você não precisa criar DisposeBags em todas as classes que criar, apenas dê uma olhada no projeto da RxSwift Community chamado NSObject+Rx. Usando o framework, o código acima pode ser reescrito da seguinte forma:

Observable.just(1).subscribeNext { 
    print($0)
}.addDisposableTo(rx_disposeBag)

No segundo caso, se o tempo de assinatura coincidir com o tempo de vida do objeto self, é possível implementar o descarte usando takeUntil(rx_deallocated):

let _ = sequence
    .takeUntil(rx_deallocated)
    .subscribe {
        print($0)
    }

Noções básicas do RxSwift #

FRP, ou Programação Reativa Funcional, tem alguns termos básicos que você precisa conhecer.

Cada parte dos dados pode ser representada como Observável, que é um fluxo de dados assíncrono. O poder do FRP está em representar eventos síncronos e assíncronos como streams, Observables, e fornecer a mesma interface para trabalhar com ele.

Normalmente Observable contém vários (ou nenhum) eventos que contém a data - eventos .Next, e então pode ser finalizado com sucesso (.Success) ou com um erro (.Error).

Vamos dar uma olhada no seguinte diagrama de mármore:

--(1)--(2)--(3)|-->

Neste exemplo, há um fluxo de valores Int. Conforme o tempo avança, três eventos .Next ocorreram e, em seguida, o fluxo foi encerrado com sucesso.

--X->

O diagrama acima mostra um caso em que nenhum dado foi emitido e o evento .Error encerra o Observable.

Antes de prosseguirmos, existem alguns recursos úteis:

  1. RxRápido. Veja exemplos, leia documentos e comece.
  2. A sala RxSwift Slack tem alguns canais para resolução de problemas educacionais.
  3. Brinque com RxMarbles para saber o que o operador faz e qual é o mais útil no seu caso.
  4. Dê uma olhada neste exemplo, explore o código você mesmo.

Criando observáveis #

RxSwift oferece muitas maneiras de criar um Observável, vamos dar uma olhada:

import RxSwift

let intObservale = Observable.just(123) // Observable<Int>
let stringObservale = Observable.just("RxSwift") // Observable<String>
let doubleObservale = Observable.just(3.14) // Observable<Double>

Assim, os observáveis ​​são criados. Eles mantêm apenas um valor e, em seguida, terminam com sucesso. No entanto, nada aconteceu depois que foi criado. Por quê?

Existem duas etapas para trabalhar com Observables: você observa algo para criar um fluxo e então você inscreve-se no fluxo ou vincula algo para interagir com ele.

Observable.just(12).subscribe {
    print($0)
}

O console imprimirá:

.Next(12)
.Completed()

E se eu estiver interessado apenas em trabalhar com dados, que acontecem em eventos .Next, eu usaria o operador subscribeNext:

Observable.just(12).subscribeNext {
    print($0) // prints "12" now
}

Se eu quiser criar um observável de muitos valores, eu uso diferentes operadores:

Observable.of(1,2,3,4,5).subscribeNext {
    print($0)
}
// 1
// 2
// 3
// 4
// 5

// I can represent existing data types as Observables also:
[1,2,3,4,5].asObservable().subscribeNext { 
    print($0) 
}
// result is the same as before.

E finalmente, talvez eu queira um Observable que funcione. Por exemplo, é conveniente agrupar uma operação de rede em Observable<SomeResultType>. Vamos dar uma olhada em como se pode conseguir isso:

Observable.create { observer in    // create an Observable ...
    MyNetworkService.doSomeWorkWithCompletion { (result, error) in
        if let e = error {
            observer.onError(e)    // ..that either holds an error
        } else {
            observer.onNext(result) // ..or emits the data
            observer.onCompleted()  // ..and terminates successfully.
        }
    }
    return NopDisposable.instance // here you can manually free any resources
                                //in case if this observable is being disposed.
}

Ligações #

Observable.combineLatest(firstName.rx_text, lastName.rx_text) { $0 + " " + $1 }
.map { "Greetings, \($0)" }
.bindTo(greetingLabel.rx_text)

Usando o operador combineLatest toda vez que um item é emitido por um dos dois Observables, combine o último item emitido por cada Observable. Dessa forma, combinamos o resultado dos dois UITextField criando uma nova mensagem com o texto "Saudações, \($0)" usando a interpolação de string para depois vincular ao texto de um UILabel.

Podemos vincular dados a qualquer UITableView e UICollectionView de uma maneira muito fácil:

viewModel
.rows
.bindTo(resultsTableView.rx_itemsWithCellIdentifier("WikipediaSearchCell", cellType: WikipediaSearchCell.self)) { (_, viewModel, cell) in
    cell.title = viewModel.title
    cell.url = viewModel.url
}
.addDisposableTo(disposeBag)

Esse é um wrapper Rx em torno do método de fonte de dados cellForRowAtIndexPath. E também o Rx cuida da implementação do numberOfRowsAtIndexPath, que é um método obrigatório no sentido tradicional, mas você não precisa implementá-lo aqui, está resolvido.

RxCocoa e ControlEvents #

O RxSwift fornece não apenas as maneiras de controlar seus dados, mas também de representar as ações do usuário de maneira reativa.

RxCocoa contém tudo que você precisa. Ele envolve a maioria das propriedades dos componentes da interface do usuário em Observables, mas não realmente. Existem alguns Observables atualizados chamados ControlEvents (que representam eventos) e ControlProperties (que representam propriedades, surpresa!). Essas coisas mantêm fluxos ‘Observáveis’ sob o capô, mas também têm algumas nuances:

  • Ele nunca falha, então não há erros.
  • Irá ‘Completar’ a sequência no controle sendo desalocado.
  • Entrega eventos na thread principal (MainScheduler.instance).

Basicamente, você pode trabalhar com eles como de costume:

button.rx_tap.subscribeNext { _ in   // control event
    print("User tapped the button!")
}.addDisposableTo(bag)

textField.rx_text.subscribeNext { text in // control property
    print("The textfield contains: \(text)")
}.addDisposableTo(bag)
// notice that ControlProperty generates .Next event on subscription
// In this case, the log will display 
// "The textfield contains: "
// at the very start of the app.

Isso é muito importante de usar: contanto que você use Rx, esqueça as coisas do @IBAction, tudo que você precisa você pode ligar e configurar de uma só vez. Por exemplo, o método viewDidLoad do seu controlador de visualização é um bom candidato para descrever como os componentes da interface do usuário funcionam.

Ok, outro exemplo: suponha que temos um campo de texto, um botão e um rótulo. Queremos validar o texto no campo de texto quando tocar no botão e exibir os resultados no rótulo. Sim, parece outra tarefa de validação de e-mail, hein?

Primeiro de tudo, pegamos o ControlEvent button.rx_tap:

----()-----------------------()----->

Aqui, parênteses vazios mostram os toques do usuário. Em seguida, pegamos o que está escrito no textField com o operador withLatestFrom (dê uma olhada aqui, imagine que o fluxo superior representa os toques do usuário, o inferior representa o texto no campo de texto).

button.rx_tap.withLatestFrom(textField.rx_text)

----("")--------------------("123")--->
//  ^ tap   ^ i wrote 123    ^ tap

Legal, temos um fluxo de strings para validar, emitido apenas quando precisamos validar.

Qualquer Observable tem operadores familiares como map ou filter, usaremos map para validar o texto. Crie você mesmo a função validateEmail, use qualquer regex que desejar.

button.rx_tap                                // ControlEvent<Void>
        .withLatestFrom(textField.rx_text)   // Observable<String>
        .map(validateEmail)                  // Observable<Bool>
        .map { (isCorrect) in
            return isCorrect ? "Email is correct" : "Input the correct one, please"
        }                                    // Observable<String>
        .bindTo(label.rx_text)              
        .addDisposableTo(bag)

Feito! Se você precisar de mais lógica personalizada (como mostrar visualizações de erro em caso de erro, fazer uma transição para outra tela em caso de sucesso…), basta assinar o fluxo final do Bool e escrevê-lo lá.
