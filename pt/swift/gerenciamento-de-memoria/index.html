Gerenciamento de memória

  • Ciclos de referência e referências fracas
  • Gerenciamento manual de memória

Nesta página

  • Ciclos de referência e referências fracas
  • Gerenciamento manual de memória

Este tópico descreve como e quando o tempo de execução do Swift deve alocar memória para estruturas de dados do aplicativo e quando essa memória deve ser recuperada.

Por padrão, as instâncias de classe de suporte de memória são gerenciadas por meio de contagem de referência. As estruturas são sempre passadas por cópia. Para desativar o esquema de gerenciamento de memória integrado, pode-se usar a estrutura Unmanaged.

Quando usar a palavra-chave fraca:

A palavra-chave `weak` deve ser usada, se um objeto referenciado puder ser desalocado durante o tempo de vida do objeto que contém a referência.

Quando usar a palavra-chave sem dono:

A palavra-chave `unowned` deve ser usada, se não for esperado que um objeto referenciado seja desalocado durante o tempo de vida do objeto que contém a referência.

Armadilhas

Um erro frequente é esquecer de criar referências a objetos, que precisam continuar vivos após o término de uma função, como gerenciadores de localização, gerenciadores de movimento, etc.

Exemplo:

class A : CLLocationManagerDelegate
{
    init()
    {
        let locationManager = CLLocationManager()
        locationManager.delegate = self
        locationManager.startLocationUpdates()
    }
}

Este exemplo não funcionará corretamente, pois o gerenciador de localização é desalocado após o retorno do inicializador. A solução adequada é criar uma referência forte como uma variável de instância:

class A : CLLocationManagerDelegate
{
    let locationManager:CLLocationManager

    init()
    {
        locationManager = CLLocationManager()
        locationManager.delegate = self
        locationManager.startLocationUpdates()
    }
}

Ciclos de referência e referências fracas #

O ciclo de referência (ou ciclo de retenção) é assim chamado porque indica um ciclo no [grafo do objeto](https:/ /en.wikipedia.org/wiki/Cycle_(graph_theory)) /en.wikipedia.org/wiki/Object_graph):

Cada seta indica um objeto retenção outro (uma referência forte). A menos que o ciclo seja interrompido, a memória desses objetos nunca será liberada.

Um ciclo de retenção é criado quando duas instâncias de classes fazem referência uma à outra:

class A { var b: B? = nil }
class B { var a: A? = nil }

let a = A()
let b = B()

a.b = b  // a retains b
b.a = a  // b retains a -- a reference cycle

Ambas as instâncias permanecerão vivas até que o programa termine. Este é um ciclo de retenção.

Referências fracas #

Para evitar ciclos de retenção, use a palavra-chave weak ou unowned ao criar referências para interromper os ciclos de retenção.

classe B { fraca var a: A? = zero }

Referências fracas ou sem dono não aumentarão a contagem de referência de uma instância. Essas referências não contribuem para reter ciclos. A referência fraca torna-se nil quando o objeto referenciado é desalocado.

a.b = b  // a retains b
b.a = a  // b holds a weak reference to a -- not a reference cycle

Ao trabalhar com closures, você também pode usar [weak e unowned nas listas de captura](https://www.wikiod.com/pt/swift/fechamentos#Captura, referências fortes/fracas e retém ciclos reter-ciclos#t=201607211834265333132).

Gerenciamento manual de memória #

Ao fazer interface com APIs C, pode-se querer recuar no contador de referência Swift. Isso é feito com objetos não gerenciados.

Se você precisar fornecer um ponteiro puncionado por tipo para uma função C, use o método toOpaque da estrutura Unmanaged para obter um ponteiro bruto e fromOpaque para recuperar a instância original:

setupDisplayLink() {
  let pointerToSelf: UnsafeRawPointer = Unmanaged.passUnretained(self).toOpaque()
  CVDisplayLinkSetOutputCallback(self.displayLink, self.redraw, pointerToSelf)
}

func redraw(pointerToSelf: UnsafeRawPointer, /* args omitted */) {
  let recoveredSelf = Unmanaged<Self>.fromOpaque(pointerToSelf).takeUnretainedValue()
  recoveredSelf.doRedraw()
}

Observe que, se estiver usando passUnretained e equivalentes, é necessário tomar todas as precauções com referências unowned.

Para interagir com APIs de Objective-C legadas, pode-se querer afetar manualmente a contagem de referência de um determinado objeto. Para isso Unmanaged tem os respectivos métodos retain e release. No entanto, é mais desejável usar passRetained e takeRetainedValue, que realizam a retenção antes de retornar o resultado:

func preferredFilenameExtension(for uti: String) -> String! {
  let result = UTTypeCopyPreferredTagWithClass(uti, kUTTagClassFilenameExtension)
  guard result != nil else { return nil }

  return result!.takeRetainedValue() as String
}

Essas soluções devem sempre ser o último recurso, e as APIs nativas da linguagem sempre devem ser preferidas.
