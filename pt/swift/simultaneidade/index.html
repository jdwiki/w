Sintaxe #

  • Rápido 3.0

  • DispatchQueue.main // Obtém a fila principal

  • DispatchQueue(label: “my-serial-queue”, attribute: [.serial, .qosBackground]) // Cria sua própria fila serial privada

  • DispatchQueue.global(attributes: [.qosDefault]) // Acessar uma das filas simultâneas globais

  • DispatchQueue.main.async { … } // Despacha uma tarefa de forma assíncrona para o thread principal

  • DispatchQueue.main.sync { … } // Despacha uma tarefa de forma síncrona para o thread principal

  • DispatchQueue.main.asyncAfter(deadline: .now() + 3) { … } // Despacha uma tarefa de forma assíncrona para a thread principal para ser executada após x segundos

  • Rápido < 3,0

  • dispatch_get_main_queue() // Coloca a fila principal em execução na thread principal

  • dispatch_get_global_queue(dispatch_queue_priority_t, 0) // Obtém a fila global com prioridade especificada dispatch_queue_priority_t

  • dispatch_async(dispatch_queue_t) { () -> Void in … } // Despacha uma tarefa de forma assíncrona no dispatch_queue_t especificado

  • dispatch_sync(dispatch_queue_t) { () -> Void in … } // Despacha uma tarefa de forma síncrona no dispatch_queue_t especificado

  • dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(nanossegundos)), dispatch_queue_t, { … }); // Despacha uma tarefa no dispatch_queue_t especificado após nanossegundos

Obtendo uma fila do Grand Central Dispatch (GCD) #

O Grand Central Dispatch trabalha no conceito de “Dispatch Queues”. Uma fila de despacho executa tarefas que você designa na ordem em que são passadas. Existem três tipos de filas de despacho:

  • Serial Dispatch Queues (também conhecido como private dispatch queues) executa uma tarefa de cada vez, em ordem. Eles são frequentemente usados ​​para sincronizar o acesso a um recurso.
  • Filas de Despacho Concorrentes (também conhecidas como filas de despacho globais) executam uma ou mais tarefas simultaneamente.
  • A Main Dispatch Queue executa tarefas na thread principal.

Para acessar a fila principal:

let mainQueue = DispatchQueue.main

let mainQueue = dispatch_get_main_queue()

O sistema fornece filas de despacho globais concorrentes (globais para sua aplicação), com prioridades variadas. Você pode acessar essas filas usando a classe DispatchQueue no Swift 3:

let globalConcurrentQueue = DispatchQueue.global(qos: .default)

equivalente a

let globalConcurrentQueue = DispatchQueue.global()

let globalConcurrentQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0)

No iOS 8 ou posterior, os possíveis valores de qualidade de serviço que podem ser passados ​​são .userInteractive, .userInitiated, .default, .utility e .background. Elas substituem as constantes DISPATCH_QUEUE_PRIORITY_.

Você também pode criar suas próprias filas com prioridades variadas:

let myConcurrentQueue = DispatchQueue(label: "my-concurrent-queue", qos: .userInitiated, attributes: [.concurrent], autoreleaseFrequency: .workItem, target: nil)
let mySerialQueue = DispatchQueue(label: "my-serial-queue", qos: .background, attributes: [], autoreleaseFrequency: .workItem, target: nil)

let myConcurrentQueue = dispatch_queue_create("my-concurrent-queue", DISPATCH_QUEUE_CONCURRENT)
let mySerialQueue = dispatch_queue_create("my-serial-queue", DISPATCH_QUEUE_SERIAL)

No Swift 3, as filas criadas com esse inicializador são seriais por padrão, e passar .workItem para frequência de liberação automática garante que um pool de liberação automática seja criado e drenado para cada item de trabalho. Há também .never, o que significa que você mesmo estará gerenciando seus próprios pools de autorelease, ou .inherit que herda a configuração do ambiente. Na maioria dos casos você provavelmente não usará .never exceto em casos de extrema customização.

Loops simultâneos #

O GCD fornece um mecanismo para executar um loop, pelo qual os loops ocorrem simultaneamente um em relação ao outro. Isso é muito útil ao realizar uma série de cálculos computacionalmente caros.

Considere este laço:

for index in 0 ..< iterations {
    // Do something computationally expensive here
}

Você pode realizar esses cálculos simultaneamente usando concurrentPerform (no Swift 3) ou dispatch_apply (no Swift 2):

DispatchQueue.concurrentPerform(iterations: iterations) { index in
    // Do something computationally expensive here
}

dispatch_apply(iterations, queue) { index in
    // Do something computationally expensive here
}

O fechamento do loop será invocado para cada index de 0 até, mas não incluindo, iterations. Essas iterações serão executadas simultaneamente em relação umas às outras e, portanto, a ordem em que são executadas não é garantida. O número real de iterações que ocorrem simultaneamente em um determinado momento geralmente é ditado pelos recursos do dispositivo em questão (por exemplo, quantos núcleos o dispositivo possui).

Algumas considerações especiais:

  • O concurrentPerform/dispatch_apply pode executar os loops concorrentemente um em relação ao outro, mas tudo isso acontece de forma síncrona em relação à thread de onde você o chama. Portanto, não chame isso do encadeamento principal, pois isso bloqueará esse encadeamento até que o loop seja concluído.

  • Como esses loops acontecem simultaneamente um em relação ao outro, você é responsável por garantir a segurança do encadeamento dos resultados. Por exemplo, se atualizar algum dicionário com os resultados desses cálculos computacionalmente caros, certifique-se de sincronizar essas atualizações você mesmo.

  • Observe que há alguma sobrecarga associada à execução de loops simultâneos. Assim, se os cálculos que estão sendo executados dentro do loop não forem computacionalmente intensivos, você poderá descobrir que qualquer desempenho obtido pelo uso de loops simultâneos pode ser diminuído, se não completamente compensado, pela sobrecarga associada à sincronização de todos esses threads simultâneos.

    So, you are responsible determining the correct amount of work to be performed in each iteration of the loop. If the calculations are too simple, you may employ “striding” to include more work per loop. For example, rather than doing a concurrent loop with 1 million trivial calculations, you may do 100 iterations in your loop, doing 10,000 calculations per loop. That way there is enough work being performed on each thread, so the overhead associated with managing these concurrent loops becomes less significant.

Executando tarefas em uma fila do Grand Central Dispatch (GCD) #

Para executar tarefas em uma fila de despacho, use os métodos sync, async e after.

Para despachar uma tarefa para uma fila de forma assíncrona:

let queue = DispatchQueue(label: "myQueueName")

queue.async {
    //do something
    
    DispatchQueue.main.async {
        //this will be called in main thread
        //any UI updates should be placed here
    }
}
// ... code here will execute immediately, before the task finished

Para despachar uma tarefa para uma fila de forma síncrona:

queue.sync {
    // Do some task
}
// ... code here will not execute until the task is finished

Para despachar uma tarefa para uma fila após um determinado número de segundos:

queue.asyncAfter(deadline: .now() + 3) {
    //this will be executed in a background-thread after 3 seconds
}
// ... code here will execute immediately, before the task finished

NOTA: Quaisquer atualizações da interface do usuário devem ser chamadas no encadeamento principal! Certifique-se de colocar o código para atualizações da interface do usuário dentro de DispatchQueue.main.async { ... }

Tipos de fila:

let mainQueue = dispatch_get_main_queue()
let highQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_HIGH, 0)
let backgroundQueue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0)

Para despachar uma tarefa para uma fila de forma assíncrona:

dispatch_async(queue) {
    // Your code run run asynchronously. Code is queued and executed 
    // at some point in the future.
}
// Code after the async block will execute immediately

Para despachar uma tarefa para uma fila de forma síncrona:

dispatch_sync(queue) {
    // Your sync code
}
// Code after the sync block will wait until the sync task finished

Para despachar uma tarefa após um intervalo de tempo (use NSEC_PER_SEC para converter segundos em nanossegundos):

dispatch_after(dispatch_time(DISPATCH_TIME_NOW, Int64(2.5 * Double(NSEC_PER_SEC))), dispatch_get_main_queue()) {
    // Code to be performed in 2.5 seconds here
}

Para executar uma tarefa de forma assíncrona e atualizar a interface do usuário:

dispatch_async(queue) {
    // Your time consuming code here
    dispatch_async(dispatch_get_main_queue()) {
        // Update the UI code 
    }
}

NOTA: Quaisquer atualizações da interface do usuário devem ser chamadas no encadeamento principal! Certifique-se de colocar o código para atualizações da interface do usuário dentro de dispatch_async(dispatch_get_main_queue()) { ... }

Executando Tarefas em um OperationQueue #

Você pode pensar em uma OperationQueue como uma linha de tarefas esperando para serem executadas. Ao contrário das filas de despacho no GCD, as filas de operação não são FIFO (first-in-first-out). Em vez disso, eles executam as tarefas assim que estiverem prontas para serem executadas, desde que haja recursos de sistema suficientes para permitir isso.

Obtenha o principal OperationQueue:

let mainQueue = OperationQueue.main

Crie um OperationQueue personalizado:

let queue = OperationQueue()
queue.name = "My Queue"
queue.qualityOfService = .default

Qualidade de Serviço especifica a importância do trabalho, ou quanto o usuário provavelmente estará contando com os resultados imediatos da tarefa.

Adicione uma Operation a uma OperationQueue:

// An instance of some Operation subclass
let operation = BlockOperation {
    // perform task here
}

queue.addOperation(operation)

Adicione um bloco a um OperationQueue:

myQueue.addOperation {
    // some task
}

Adicione vários Operations a um OperationQueue:

let operations = [Operation]()
// Fill array with Operations

myQueue.addOperation(operations)

Ajuste quantas Operações podem ser executadas simultaneamente na fila:

myQueue.maxConcurrentOperationCount = 3 // 3 operations may execute at once

// Sets number of concurrent operations based on current system conditions
myQueue.maxConcurrentOperationCount = NSOperationQueueDefaultMaxConcurrentOperationCount

Suspender uma fila impedirá que ela inicie a execução de quaisquer operações existentes não iniciadas ou de quaisquer novas operações adicionadas à fila. A maneira de retomar essa fila é definir o isSuspended de volta para false:

myQueue.isSuspended = true

// Re-enable execution
myQueue.isSuspended = false

Suspender um OperationQueue não interrompe ou cancela operações que já estão em execução. Deve-se tentar suspender apenas uma fila que você criou, não as filas globais ou a fila principal.

Criando Operações de Alto Nível #

A estrutura Foundation fornece o tipo Operation, que representa um objeto de alto nível que encapsula uma parte do trabalho que pode ser executada em uma fila. A fila não apenas coordena o desempenho dessas operações, mas você também pode estabelecer dependências entre as operações, criar operações canceláveis, restringir o grau de simultaneidade empregado pela fila de operações, etc.

A Operation fica pronta para ser executada quando todas as suas dependências terminam de ser executadas. A propriedade isReady muda para true.

Crie uma subclasse Operation não concorrente simples:

class MyOperation: Operation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

class MyOperation: NSOperation {

    init(<parameters>) {
        // Do any setup work here
    }

    override func main() {
        // Perform the task
    }

}

Adicione uma operação a um OperationQueue:

myQueue.addOperation(operation)

Isso executará a operação simultaneamente na fila.

Gerencie dependências em uma Operação.

As dependências definem outras Operações que devem ser executadas em uma fila antes dessa Operação ser considerada pronta para execução.

operation2.addDependency(operation1)

operation2.removeDependency(operation1)

Execute uma Operação sem uma fila:

   operation.start()

As dependências serão ignoradas. Se esta for uma operação simultânea, a tarefa ainda pode ser executada simultaneamente se seu método start descarregar o trabalho em filas em segundo plano.

Operações simultâneas.

Se a tarefa que uma Operation deve executar for, ela mesma, assíncrona (por exemplo, uma tarefa de dados URLSession), você deve implementar a Operation como uma operação simultânea. Neste caso, sua implementação isAsynchronous deve retornar true, você geralmente teria o método start que executa alguma configuração, então chama seu método main que realmente executa a tarefa.

Ao implementar uma Operação assíncrona, você deve implementar os métodos isExecuting, isFinished e KVO. Assim, quando a execução começa, a propriedade isExecuting muda para true. Quando uma Operation termina sua tarefa, isExecuting é definido como false e ​​isFinished é definido como true. Se a operação for cancelada, tanto isCancelled quanto isFinished mudam para true. Todas essas propriedades são observáveis ​​por valor-chave.

Cancelar uma Operação.

Chamar cancel simplesmente altera a propriedade isCancelled para true. Para responder ao cancelamento de dentro de sua própria subclasse Operation, você deve verificar o valor de isCancelled pelo menos periodicamente dentro de main e responder adequadamente.

operation.cancel()
