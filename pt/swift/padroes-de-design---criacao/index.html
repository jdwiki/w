Padrões de projeto são soluções gerais para problemas que ocorrem com frequência no desenvolvimento de software. A seguir estão modelos de práticas recomendadas padronizadas na estruturação e design de código, bem como exemplos de contextos comuns nos quais esses padrões de design seriam apropriados.

Padrões de design de criação abstraem a instanciação de objetos para tornar um sistema mais independente do processo de criação, composição e representação.

Singleton #

Singletons são um padrão de projeto frequentemente usado que consiste em uma única instância de uma classe que é compartilhada por todo o programa.

No exemplo a seguir, criamos uma propriedade static que contém uma instância da classe Foo. Lembre-se que uma propriedade static é compartilhada entre todos os objetos de uma classe e não pode ser substituída por subclasses.

public class Foo
{
    static let shared = Foo()
    
    // Used for preventing the class from being instantiated directly
    private init() {}
    
    func doSomething()
    {
        print("Do something")
    }
}

Uso:

Foo.shared.doSomething()

Certifique-se de lembrar o inicializador private:

Isso garante que seus singletons sejam realmente únicos e evita que objetos externos criem suas próprias instâncias de sua classe por meio do controle de acesso. Como todos os objetos vêm com um inicializador público padrão no Swift, você precisa substituir seu init e torná-lo privado. KrakenDev

Padrão do Construtor #

O padrão construtor é um padrão de design de software de criação de objetos. Ao contrário do padrão de fábrica abstrato e do padrão de método de fábrica cuja intenção é possibilitar o polimorfismo, a intenção do o padrão do construtor é encontrar uma solução para o construtor telescópico antipadrão. O antipadrão do construtor telescópico ocorre quando o o aumento da combinação de parâmetros do construtor de objetos leva a um lista exponencial de construtores. Em vez de usar vários construtores, o padrão construtor usa outro objeto, um construtor, que recebe cada parâmetro de inicialização passo a passo e depois retorna o objeto construído resultante de uma só vez.

-Wikipédia

O principal objetivo do padrão do construtor é definir uma configuração padrão para um objeto desde sua criação. É um intermediário entre o objeto a ser construído e todos os demais objetos relacionados à sua construção.

O problema que surge ao criar um objeto carro é que o carro requer muitos dados de configuração para ser criado.

Para aplicar o Builder Pattern, os parâmetros do inicializador devem ter valores padrão que podem ser alterados se necessário.

Classe CarBuilder:

class CarBuilder {
    var color: UIColor = UIColor.black
    var numberOfSeats: UInt8 = 5
    var numberOfWheels: UInt8 = 4
    var type: CarType = .saloon
    var gearType: GearType = .automatic
    var motor: Motor = Motor(id: "111", name: "Default Motor",
                             model: "T9", numberOfCylinders: 4)
    var shouldHasAirbags: Bool = false
    
    func buildCar() -> Car {
        return Car(color: color, numberOfSeats: numberOfSeats, numberOfWheels: numberOfWheels, type: type, gearType: gearType, motor: motor, shouldHasAirbags: shouldHasAirbags)
    }
}

A classe CarBuilder define propriedades que podem ser alteradas para editar os valores do objeto carro criado.

Vamos construir carros novos usando o CarBuilder:

var builder = CarBuilder()
// currently, the builder creates cars with default configuration.

let defaultCar = builder.buildCar()
//print(defaultCar.description)
/* prints

 Number of seats: 5
 Number of Wheels: 4

 Motor: Motor(id: "111", name: "Default Motor", model: "T9", numberOfCylinders: 4)
 Airbag Availability: false
*/

builder.shouldHasAirbags = true
// now, the builder creates cars with default configuration,
// but with a small edit on making the airbags available

let safeCar = builder.buildCar()
print(safeCar.description)
/* prints
 color: UIExtendedGrayColorSpace 0 1
 Number of seats: 5
 Number of Wheels: 4
 Type: automatic
 Motor: Motor(id: "111", name: "Default Motor", model: "T9", numberOfCylinders: 4)
 Airbag Availability: true
 */

builder.color = UIColor.purple
// now, the builder creates cars with default configuration
// with some extra features: the airbags are available and the color is purple

let femaleCar = builder.buildCar()
print(femaleCar)
/* prints
 color: UIExtendedSRGBColorSpace 0.5 0 0.5 1
 Number of seats: 5
 Number of Wheels: 4
 Type: automatic
 Motor: Motor(id: "111", name: "Default Motor", model: "T9", numberOfCylinders: 4)
 Airbag Availability: true
*/

O benefício de aplicar o Builder Pattern é a facilidade de criar objetos que devem conter grande parte das configurações definindo valores padrão, também, a facilidade de alterar esses valores padrão.

Vá além: #

Como boa prática, todas as propriedades que precisam de valores padrão devem estar em um protocolo separado, que deve ser implementado pela própria classe e seu construtor.

Voltando ao nosso exemplo, vamos criar um novo protocolo chamado CarBluePrint:

import UIKit

enum CarType {
    case
    
    sportage,
    saloon
}

enum GearType {
    case
    
    manual,
    automatic
}

struct Motor {
    var id: String
    var name: String
    var model: String
    var numberOfCylinders: UInt8
}

protocol CarBluePrint {
    var color: UIColor { get set }
    var numberOfSeats: UInt8 { get set }
    var numberOfWheels: UInt8 { get set }
    var type: CarType { get set }
    var gearType: GearType { get set }
    var motor: Motor { get set }
    var shouldHasAirbags: Bool { get set }
}

class Car: CustomStringConvertible, CarBluePrint {
    var color: UIColor
    var numberOfSeats: UInt8
    var numberOfWheels: UInt8
    var type: CarType
    var gearType: GearType
    var motor: Motor
    var shouldHasAirbags: Bool
    
    var description: String {
        return "color: \(color)\nNumber of seats: \(numberOfSeats)\nNumber of Wheels: \(numberOfWheels)\n Type: \(gearType)\nMotor: \(motor)\nAirbag Availability: \(shouldHasAirbags)"
    }
    
    init(color: UIColor, numberOfSeats: UInt8, numberOfWheels: UInt8, type: CarType, gearType: GearType, motor: Motor, shouldHasAirbags: Bool) {
        
        self.color = color
        self.numberOfSeats = numberOfSeats
        self.numberOfWheels = numberOfWheels
        self.type = type
        self.gearType = gearType
        self.motor = motor
        self.shouldHasAirbags = shouldHasAirbags
        
    }
}

class CarBuilder: CarBluePrint {
    var color: UIColor = UIColor.black
    var numberOfSeats: UInt8 = 5
    var numberOfWheels: UInt8 = 4
    var type: CarType = .saloon
    var gearType: GearType = .automatic
    var motor: Motor = Motor(id: "111", name: "Default Motor",
                             model: "T9", numberOfCylinders: 4)
    var shouldHasAirbags: Bool = false
    
    func buildCar() -> Car {
        return Car(color: color, numberOfSeats: numberOfSeats, numberOfWheels: numberOfWheels, type: type, gearType: gearType, motor: motor, shouldHasAirbags: shouldHasAirbags)
    }
}

O benefício de declarar as propriedades que precisam de valor padrão em um protocolo é forçar a implementação de qualquer nova propriedade adicionada; Quando uma classe está em conformidade com um protocolo, ela deve declarar todas as suas propriedades/métodos.

Considere que há um novo recurso necessário que deve ser adicionado ao plano de criação de um carro chamado “nome da bateria”:

protocol CarBluePrint {
    var color: UIColor { get set }
    var numberOfSeats: UInt8 { get set }
    var numberOfWheels: UInt8 { get set }
    var type: CarType { get set }
    var gearType: GearType { get set }
    var motor: Motor { get set }
    var shouldHasAirbags: Bool { get set }
    
    // adding the new property
    var batteryName: String { get set }
}

Depois de adicionar a nova propriedade, observe que dois erros de tempo de compilação surgirão, notificando que a conformidade com o protocolo CarBluePrint requer a declaração da propriedade ‘batteryName’. Isso garante que o CarBuilder irá declarar e definir um valor padrão para a propriedade batteryName.

Após adicionar a nova propriedade batteryName ao protocolo CarBluePrint, a implementação das classes Car e CarBuilder deve ser:

class Car: CustomStringConvertible, CarBluePrint {
    var color: UIColor
    var numberOfSeats: UInt8
    var numberOfWheels: UInt8
    var type: CarType
    var gearType: GearType
    var motor: Motor
    var shouldHasAirbags: Bool
    var batteryName: String
    
    var description: String {
        return "color: \(color)\nNumber of seats: \(numberOfSeats)\nNumber of Wheels: \(numberOfWheels)\nType: \(gearType)\nMotor: \(motor)\nAirbag Availability: \(shouldHasAirbags)\nBattery Name: \(batteryName)"
    }
    
    init(color: UIColor, numberOfSeats: UInt8, numberOfWheels: UInt8, type: CarType, gearType: GearType, motor: Motor, shouldHasAirbags: Bool, batteryName: String) {
        
        self.color = color
        self.numberOfSeats = numberOfSeats
        self.numberOfWheels = numberOfWheels
        self.type = type
        self.gearType = gearType
        self.motor = motor
        self.shouldHasAirbags = shouldHasAirbags
        self.batteryName = batteryName
    }
}

class CarBuilder: CarBluePrint {
    var color: UIColor = UIColor.red
    var numberOfSeats: UInt8 = 5
    var numberOfWheels: UInt8 = 4
    var type: CarType = .saloon
    var gearType: GearType = .automatic
    var motor: Motor = Motor(id: "111", name: "Default Motor",
                             model: "T9", numberOfCylinders: 4)
    var shouldHasAirbags: Bool = false
    var batteryName: String = "Default Battery Name"
    
    func buildCar() -> Car {
        return Car(color: color, numberOfSeats: numberOfSeats, numberOfWheels: numberOfWheels, type: type, gearType: gearType, motor: motor, shouldHasAirbags: shouldHasAirbags, batteryName: batteryName)
    }
}

Novamente, vamos construir carros novos usando o CarBuilder:

print(defaultCar)

 Airbag Availability: false

*/

Método de fábrica #

Na programação baseada em classes, o padrão de método de fábrica é um padrão de criação que usa métodos de fábrica para lidar com o problema de criar objetos sem precisar especificar a classe exata do objeto que será criado. Referência da Wikipédia

protocol SenderProtocol
{
    func send(package: AnyObject)
}

class Fedex: SenderProtocol
{
    func send(package: AnyObject)
    {
        print("Fedex deliver")
    }
}

class RegularPriorityMail: SenderProtocol
{
    func send(package: AnyObject)
    {
        print("Regular Priority Mail deliver")
    }
}

// This is our Factory
class DeliverFactory
{
    // It will be responsable for returning the proper instance that will handle the task
    static func makeSender(isLate isLate: Bool) -> SenderProtocol
    {
        return isLate ? Fedex() : RegularPriorityMail()
    }
}

// Usage:
let package = ["Item 1", "Item 2"]

// Fedex class will handle the delivery
DeliverFactory.makeSender(isLate:true).send(package)

// Regular Priority Mail class will handle the delivery
DeliverFactory.makeSender(isLate:false).send(package)

Ao fazer isso não dependemos da implementação real da classe, tornando o sender() completamente transparente para quem o está consumindo.

Neste caso, tudo o que precisamos saber é que um remetente irá lidar com a entrega e expor um método chamado send(). Existem várias outras vantagens: reduzir o acoplamento de classes, mais fácil de testar, mais fácil de adicionar novos comportamentos sem ter que mudar quem está consumindo.

No design orientado a objetos, as interfaces fornecem camadas de abstração que facilitam a explicação conceitual do código e criam uma barreira que impede dependências.Referência da Wikipédia

Observador #

O padrão de observador é onde um objeto, chamado de sujeito, mantém uma lista de seus dependentes, chamados de observadores, e os notifica automaticamente sobre qualquer mudança de estado, geralmente chamando um de seus métodos. É usado principalmente para implementar sistemas de manipulação de eventos distribuídos. O padrão Observer também é uma parte fundamental do familiar padrão arquitetural model–view–controller (MVC).Referência da Wikipédia

Basicamente, o padrão observador é usado quando você tem um objeto que pode notificar os observadores sobre certos comportamentos ou mudanças de estado.

Primeiro vamos criar uma referência global (fora de uma classe) para a Central de Notificações:

let notifCentre = NotificationCenter.default

Ótimo agora podemos chamar isso de qualquer lugar. Gostaríamos então de registrar uma classe como observadora…

notifCentre.addObserver(self, selector: #selector(self.myFunc), name: "myNotification", object: nil)

Isso adiciona a classe como um observador para “readForMyFunc”. Também indica que a função myFunc deve ser chamada quando essa notificação for recebida. Esta função deve ser escrita na mesma classe:

func myFunc(){
    print("The notification has been received")
}

Uma das vantagens deste padrão é que você pode adicionar muitas classes como observadores e, assim, realizar muitas ações após uma notificação.

A notificação agora pode simplesmente ser enviada (ou postada se você preferir) de quase qualquer lugar no código com a linha:

notifCentre.post(name: "myNotification", object: nil)

Você também pode passar informações com a notificação como um Dicionário

let myInfo = "pass this on"
notifCentre.post(name: "myNotification", object: ["moreInfo":myInfo])

Mas então você precisa adicionar uma notificação à sua função:

func myFunc(_ notification: Notification){
    let userInfo = (notification as NSNotification).userInfo as! [String: AnyObject]
    let passedInfo = userInfo["moreInfo"]
    print("The notification \(moreInfo) has been received")
    //prints - The notification pass this on has been received
}

Cadeia de responsabilidade #

No projeto orientado a objetos, o padrão de cadeia de responsabilidade é um padrão de projeto que consiste em uma fonte de objetos ‘comando’ e uma série de objetos ‘processamento’. Cada objeto processing contém uma lógica que define os tipos de objetos de comando que podem ser manipulados; o resto é passado para o próximo objeto processing na cadeia. Também existe um mecanismo para adicionar novos objetos processing ao final desta cadeia. Wikipedia

Configurando as classes que compunham a cadeia de responsabilidades.

Primeiro criamos uma interface para todos os objetos processing.

protocol PurchasePower {  
var allowable : Float { get }
  var role : String { get }
  var successor : PurchasePower? { get set }
}

extension PurchasePower {
  func process(request : PurchaseRequest){
    if request.amount < self.allowable {
      print(self.role + " will approve $ \(request.amount) for \(request.purpose)")
    } else if successor != nil {
      successor?.process(request: request)
    }
  }
}

Em seguida, criamos o objeto command.

struct PurchaseRequest {
  var amount : Float
  var purpose : String
}

Por fim, a criação de objetos que compunham a cadeia de responsabilidades.

class ManagerPower : PurchasePower {
  var allowable: Float = 20
  var role : String = "Manager"
  var successor: PurchasePower?
}

class DirectorPower : PurchasePower {
  var allowable: Float = 100
  var role = "Director"
  var successor: PurchasePower?
}

class PresidentPower : PurchasePower {
  var allowable: Float = 5000
  var role = "President"
  var successor: PurchasePower?
}

Inicie e encadeie-o:

let manager = ManagerPower()
let director = DirectorPower()
let president = PresidentPower()

manager.successor = director
director.successor = president

O mecanismo para encadear objetos aqui é o acesso à propriedade

Criando solicitação para executá-lo:

manager.process(request: PurchaseRequest(amount: 2, purpose: "buying a pen"))  // Manager will approve $ 2.0 for buying a pen
manager.process(request: PurchaseRequest(amount: 90, purpose: "buying a printer")) // Director will approve $ 90.0 for buying a printer

manager.process(request: PurchaseRequest(amount: 2000, purpose: "invest in stock")) // President will approve $ 2000.0 for invest in stock

Iterador #

Na programação de computadores, um iterador é um objeto que permite um programador para percorrer um contêiner, principalmente listas. Wikipédia

struct Turtle {
  let name: String
}

struct Turtles {
  let turtles: [Turtle]
}

struct TurtlesIterator: IteratorProtocol {
  private var current = 0
  private let turtles: [Turtle]

  init(turtles: [Turtle]) {
    self.turtles = turtles
  }

  mutating func next() -> Turtle? {
    defer { current += 1 }
    return turtles.count > current ? turtles[current] : nil
  }
}

extension Turtles: Sequence {
  func makeIterator() -> TurtlesIterator {
    return TurtlesIterator(turtles: turtles)
  }
}

let ninjaTurtles = Turtles(turtles: [Turtle(name: "Leo"),
                                     Turtle(name: "Mickey"),
                                     Turtle(name: "Raph"),

print("Splinter and")
