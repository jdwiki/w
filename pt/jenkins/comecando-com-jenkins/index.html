Configurando o proxy Nginx #

Nativamente, Jenkins é executado na porta 8080. Podemos estabelecer um proxy da porta 80 -> 8080 para que Jenkins possa ser acessado via:

http://<url>.com

em vez do padrão

http://<url>.com:8080

Comece instalando o Nginx.

sudo aptitude -y install nginx

Remova as configurações padrão do Nginx

cd /etc/nginx/sites-available

sudo rm default ../sites-enabled/default

Crie o novo arquivo de configuração

sudo touch jenkins

Copie o seguinte código no arquivo jenkins recém-criado.

upstream app_server {
  server 127.0.0.1:8080 fail_timeout=0;
}

server {
  listen 80;
  listen [::]:80 default ipv6only=on;
  server_name ;

  location / {
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header Host $http_host;
    proxy_redirect off;

    if (!-f $request_filename) {
        proxy_pass http://app_server;
        break;
    }
  }
}

Crie um link simbólico entre sites disponíveis e sites habilitados:

sudo ln -s /etc/nginx/sites-available/jenkins /etc/nginx/sites-enabled/

Reinicie o serviço de proxy Nginx

sudo service nginx restart

Jenkins agora estará acessível a partir da porta 80.

Introdução completa do Jenkins em um só lugar #

1. Jenkins :

Jenkins é uma ferramenta de integração contínua de código aberto escrita em Java. O projeto foi bifurcado de Hudson após uma disputa com a Oracle.

Em poucas palavras, Jenkins é o principal servidor de automação de código aberto. Construído com Java, ele fornece centenas de plugins para dar suporte à construção, teste, implantação e automação para praticamente qualquer projeto.

Características : Jenkins oferece os seguintes recursos principais prontos para uso, e muitos outros podem ser adicionados por meio de plugins:

Fácil instalação: Basta executar java -jar jenkins.war, implantá-lo em um contêiner de servlet. Sem instalação adicional, sem banco de dados. Prefere um instalador ou pacote nativo? Temos esses também. Configuração fácil: Jenkins pode ser configurado inteiramente a partir de sua interface amigável da web com extensas verificações de erros em tempo real e ajuda em linha. Ecossistema rico de plugins: o Jenkins se integra a praticamente todos os SCM ou ferramentas de construção existentes. Ver plug-ins. Extensibilidade: A maioria das partes do Jenkins pode ser estendida e modificada, e é fácil criar novos plugins do Jenkins. Isso permite que você personalize o Jenkins de acordo com suas necessidades. Compilações distribuídas: Jenkins pode distribuir cargas de compilação/teste para vários computadores com diferentes sistemas operacionais. Construindo software para OS X, Linux e Windows? Sem problemas.

Instalação:

$ wget -q -O - https://jenkins-ci.org/debian/jenkins-ci.org.key | sudo apt-key add -

$ sudo sh -c 'echo deb http://pkg.jenkins-ci.org/debian binary/ > /etc/apt/sources.list.d/jenkins.list'
$ sudo apt-get update
$ sudo apt-get install jenkins

JENKINS_HOME diretório Jenkins precisa de algum espaço em disco para realizar compilações e manter arquivos. Você pode verificar esse local na tela de configuração do Jenkins. Por padrão, isso é definido como ~/.jenkins, mas você pode alterar isso de uma das seguintes maneiras: Configure a variável de ambiente “JENKINS_HOME” para o novo diretório inicial antes de ativar o contêiner do servlet. Defina a propriedade do sistema “JENKINS_HOME” para o contêiner do servlet. Defina a entrada do ambiente JNDI “JENKINS_HOME” para o novo diretório. Consulte a coleção de documentação específica do contêiner para saber mais sobre como fazer isso para seu contêiner. Você também pode alterar esse local depois de usar o Jenkins por um tempo. Para fazer isso, pare completamente o Jenkins, mova o conteúdo do antigo JENKINS_HOME para o novo home, defina o novo JENKINS_HOME e reinicie o Jenkins. JENKINS_HOME tem uma estrutura de diretórios bastante óbvia que se parece com o seguinte:

JENKINS_HOME

+- config.xml     (jenkins root configuration)
 +- *.xml          (other site-wide configuration files)
 +- userContent    (files in this directory will be served under your http://server/userContent/)
 +- fingerprints   (stores fingerprint records)
 +- plugins        (stores plugins)
 +- workspace (working directory for the version control system)
     +- [JOBNAME] (sub directory for each job)
 +- jobs
     +- [JOBNAME]      (sub directory for each job)
         +- config.xml     (job configuration file)
         +- latest         (symbolic link to the last successful build)
         +- builds
             +- [BUILD_ID]     (for each build)
                 +- build.xml      (build result summary)
                 +- log            (log file)
                 +- changelog.xml  (change log)

Jenkins Build Jobs:

Criar um novo trabalho de compilação no Jenkins é simples: basta clicar no item de menu “Novo trabalho” no painel do Jenkins. Jenkins oferece suporte a vários tipos diferentes de trabalhos de construção, que são apresentados a você quando você escolhe criar um novo trabalho

Projeto de software de estilo livre

Os trabalhos de construção de estilo livre são trabalhos de construção de uso geral, que fornecem o máximo de flexibilidade.

Projeto Maven O “projeto maven2/3” é um trabalho de construção especialmente adaptado para projetos Maven. Jenkins entende os arquivos pom do Maven e as estruturas do projeto e pode usar as informações coletadas do arquivo pom para reduzir o trabalho que você precisa fazer para configurar seu projeto.

Fluxo de trabalho

Orquestra atividades de longa duração que podem abranger vários escravos de compilação. Adequado para construir tubulações e/ou organizar atividades complexas que não se encaixam facilmente no tipo de trabalho de estilo livre.

Monitorar um trabalho externo O trabalho de compilação “Monitorar um trabalho externo” permite que você fique de olho em processos não interativos, como trabalhos cron.

Trabalho de multiconfiguração O “projeto de multiconfiguração” (também conhecido como “projeto matriz”) permite que você execute o mesmo trabalho de compilação em muitas configurações diferentes. Esse poderoso recurso pode ser útil para testar um aplicativo em vários ambientes diferentes, com diferentes bancos de dados ou até mesmo em diferentes máquinas de compilação.

1. Construindo um projeto de software (estilo livre)

Jenkins pode ser usado para executar o trabalho típico do servidor de compilação, como fazer compilações contínuas/oficiais/noites, executar testes ou executar algumas tarefas em lote repetitivas. Isso é chamado de “projeto de software de estilo livre” no Jenkins. Configurando o projeto Vá para a página inicial do Jenkins, selecione “New Job” e escolha “Build a free-style software project”. Este tipo de trabalho consiste nos seguintes elementos: SCM opcional, como CVS ou Subversion onde seu código fonte reside. gatilhos opcionais para controlar quando o Jenkins executará compilações. algum tipo de script de compilação que executa a compilação (ant, maven, shell script, arquivo em lote, etc.) onde o trabalho real acontece etapas opcionais para coletar informações fora da compilação, como arquivar os artefatos e/ou gravar javadoc e resultados de teste. etapas opcionais para notificar outras pessoas/sistemas com o resultado da compilação, como envio de e-mails, mensagens instantâneas, atualização do rastreador de problemas, etc.

Compilações para projetos de controle sem origem Às vezes há a necessidade de construir um projeto simplesmente para fins de demonstração ou o acesso a um repositório SVN/CVS não está disponível. Ao optar por configurar o projeto como “Nenhum” em “Gerenciamento de código-fonte”, você precisará:

  1. Construa o projeto pelo menos uma vez (vai falhar), mas o Jenkins criará a estrutura jenkins/workspace/PROJECTNAME/
  2. Copie os arquivos do projeto para jenkins/workspace/PROJECTNAME/
  3. Compile novamente e configure adequadamente

Quando um trabalho do Jenkins é executado, ele define algumas variáveis ​​de ambiente que você pode usar em seu script de shell, comando em lote, script Ant ou Maven POM . Veja a lista de variáveis ​​clicando em ENVIRONMENT_VARIABLE

Configurando compilações automáticas

As compilações no Jenkins podem ser acionadas periodicamente (em uma programação, especificada na configuração), ou quando forem detectadas alterações de origem no projeto, ou podem ser acionadas automaticamente solicitando a URL:

http://YOURHOST/jenkins/job/PROJECTNAME/build

Isso permite que você conecte as compilações do Jenkins em uma variedade de configurações. Para mais informações (especialmente como fazer isso com a segurança ativada), consulte API de acesso remoto.

Criações por alterações de origem

Você pode fazer com que o Jenkins pesquise alterações no seu Sistema de Controle de Revisão. Você pode especificar com que frequência o Jenkins pesquisa seu sistema de controle de revisão usando a mesma sintaxe do crontab no Unix/Linux. No entanto, se seu período de pesquisa for menor do que o necessário para pesquisar seu sistema de controle de revisão, você poderá acabar com várias compilações para cada alteração. Você deve ajustar seu período de sondagem para ser maior do que o tempo necessário para sondar seu sistema de controle de revisão ou usar um gatilho pós-commit. Você pode examinar o Registro de pesquisa de cada versão para ver quanto tempo levou para pesquisar seu sistema.

Como alternativa, em vez de pesquisar em um intervalo fixo, você pode usar um acionador de URL (descrito acima), mas com /polling em vez de /build no final do URL. Isso faz com que o Jenkins pesquise o SCM em busca de alterações em vez de compilar imediatamente. Isso evita que o Jenkins execute uma compilação sem alterações relevantes para commits que afetem módulos ou branches que não estão relacionados ao trabalho. Ao usar /polling, o trabalho deve ser configurado para polling, mas o agendamento pode estar vazio.

Construções por e-mail (sendmail)

Se você tiver a conta root do seu sistema e estiver usando o sendmail, achei mais fácil ajustar /etc/aliases e adicionar a seguinte entrada: jenkins-foo: “|/bin/wget -o /dev/null

http://YOURHOST/jenkins/job/PROJECTNAME/build”

e, em seguida, execute o comando “newaliases” para informar o sendmail sobre a alteração. Sempre que alguém enviar um e-mail para “jenkins-foo@yoursystem”, isso acionará uma nova compilação. Consulte este para obter mais detalhes sobre como configurar o sendmail. Construções por e-mail (qmail) Com o qmail, você pode escrever /var/qmail/alias/.qmail-jenkins da seguinte forma: |/bin/wget -o /dev/null http://YOURHOST/jenkins/job/PROJECTNAME/build"

2. Construindo um projeto Maven

Jenkins fornece um tipo de trabalho dedicado ao Maven 2/3. Esse tipo de trabalho integra profundamente o Jenkins ao Maven 2/3 e oferece os seguintes benefícios em comparação com o projeto de software de estilo livre mais genérico.

Jenkins analisa POMs Maven para obter muitas das informações necessárias para fazer seu trabalho. Como resultado, a quantidade de configuração é drasticamente reduzida.

Jenkins ouve a execução do Maven e descobre o que deve ser feito sozinho. Por exemplo, ele gravará automaticamente o relatório JUnit quando o Maven executar a fase de teste. Ou se você executar a meta javadoc, o Jenkins gravará automaticamente o javadoc.

Jenkins cria automaticamente dependências de projeto entre projetos que declaram dependências SNAPSHOT entre si. Veja abaixo. Assim, na maioria das vezes, você só precisa configurar as informações do SCM e quais objetivos você gostaria de executar, e o Jenkins descobrirá todo o resto.

Esse tipo de projeto pode fornecer automaticamente os seguintes recursos:

Arquivar artefatos produzidos por uma compilação

Publicar resultados de teste

Acionar jobs para projetos que são dependências downstream

Implante seus artefatos em um repositório Maven

Resultados do teste de detalhamento por módulo

Opcionalmente, reconstrua apenas os módulos alterados, acelerando suas compilações

Encadeamento de compilação automático de dependências de módulo

O Jenkins lê as dependências do seu projeto a partir do seu POM e, se elas também forem compiladas no Jenkins, os gatilhos serão configurados de tal forma que uma nova compilação em uma dessas dependências iniciará automaticamente uma nova compilação do seu projeto. Jenkins entende todos os tipos de dependências no POM. Ou seja, pai POM

<dependencies> section of your project
<plugins> section of your project
<extensions> section of your project
<reporting> section of your project

Esse processo leva em consideração as versões, para que você possa ter várias versões/ramificações do seu projeto no mesmo Jenkins e determinará corretamente as dependências. Observação que os intervalos de versão de dependência não são compatíveis. Consulte https://issues.jenkins-ci.org/browse/JENKINS-2787 para saber o motivo.

Este recurso pode ser desativado sob demanda - consulte a opção de configuração Criar sempre que uma dependência SNAPSHOT for criada

Se você marcar a caixa de seleção Instalar automaticamente, o Jenkins fará o download e instalará a versão solicitada do Maven para você e a instalará no diretório de ferramentas no diretório inicial do Jenkins.

Como usá-lo

Agora, o Jenkins instalará automaticamente esta versão sempre que for necessário (em qualquer nova máquina de compilação, por exemplo) baixando-a do Apache e descompactando-a.

  1. Escolha o SCM que deseja usar (ex. Usando git)

  2. escolha o alvo maven para chamar

  4. verifique o repositório maven privado do usuário:

Você também pode definir o caminho do custome para o mesmo.

5 . Construir projeto

Construa seu projeto clicando em construir agora e clique na barra de progresso na mão esquerda “Build Executor Status” para ver o jenkins instalar o Maven, fazer o checkout do seu projeto e construí-lo usando o maven.

Mova Jenkins de um PC para outro #

Isso funcionou para eu mudar do Ubuntu 12.04 (Jenkins ver. 1.628) para o Ubuntu 16.04 (Jenkins ver. 1.651.2). Eu primeiro instalei o Jenkins dos repositórios.

  1. Pare os dois servidores Jenkins

  2. Copie JENKINS_HOME (por exemplo, /var/lib/jenkins) do servidor antigo para o novo. De um console no novo servidor:

    rsync -av username@old-server-IP:/var/lib/jenkins/ /var/lib/jenkins/

  3. Inicie seu novo servidor Jenkins

Você pode não precisar disso, mas eu tive que

  • Manage Jenkins e Reload Configuration from Disk.
  • Desconecte e conecte novamente todos os escravos.
  • Verifique se em Configure System > Jenkins Location, o Jenkins URL está atribuído corretamente ao novo servidor Jenkins.

Instalação #

Para sistemas baseados em apt-get como Ubuntu

Adicione o repositório Jenkins:

wget -q -O - https://jenkins-ci.org/debian/ Jenkins-ci.org.key | sudo apt-key 

sudo sh -c 'echo deb http://pkg.jenkins-ci.org/debian-stable binary/ > /etc/apt/sources.list.d/jenkins.list'

Atualize as fontes e instale o Jenkins:

sudo apt-get update

sudo apt-get install jenkins

Um usuário jenkins agora é criado e, por padrão, o Jenkins será executado na porta 8080.

Para distribuições baseadas em RPM, como Red Hat Enterprise Linux (RHEL), CentOS, Fedora ou Scientific Linux

Para baixar o arquivo do repositório para a versão estável:

sudo wget -O /etc/yum.repos.d/jenkins.repo http://pkg.jenkins-ci.org/redhat-stable/jenkins.repo

Ou se você quiser os últimos lançamentos semanais:

sudo wget -O /etc/yum.repos.d/jenkins.repo http://pkg.jenkins-ci.org/redhat/jenkins.repo

Importe a chave pública:

sudo rpm --import https://jenkins-ci.org/redhat/jenkins-ci.org.key

Instale o Jenkins usando o yum:

sudo yum install jenkins

Jenkins requer java para ser executado, para instalá-lo:

sudo yum install java

Para iniciar/parar/reiniciar o jenkins, use:

sudo service jenkins start/stop/restart

Atualizando jenkins (instalações RPM) #

  1. Faça backup do diretório inicial do jenkins
  2. Substitua jenkins.war no local a seguir pelo novo arquivo WAR. /usr/lib/jenkins/jenkins.war`
  3. Reinicie o Jenkins
  4. Verifique os plug-ins fixados e desvincule-os, se necessário
  5. Recarregue a configuração do disco

observação: para atualizações do Jenkins 2 para o servidor de aplicativos jetty empacotado, desative a porta AJP (definir JENKINS_AJP_PORT="-1") em /etc/sysconfig/jenkins.

Esta caixa é responsável por acionar a compilação em intervalos regulares. ***** especifica que, o trabalho será acionado a cada minuto para alterações no git repo. 8. Na seção Build, clique no botão Add Build Step e selecione a opção pela qual deseja construir o projeto. Selecionei Executar Shell. Na caixa de comando, escreva o comando para compilar, execute os testes e implante-o no prod. 10. Role para baixo e Salvar.

Então, acima, configuramos um projeto básico no Jenkins que acionará a compilação a cada minuto para alteração em seu repositório git. Nota: Para configurar o projeto complexo, talvez seja necessário instalar alguns plugins no Jenkins.

Configure um projeto de compilação simples com o script de pipeline do Jenkins 2 #

Aqui estaremos criando um pipeline Groovy no Jenkins 2 para fazer as seguintes etapas:

  • Verifique a cada 5 minutos se um novo código foi comprometido com nosso projeto
  • Código de checkout do repositório SCM
  • Compilação Maven do nosso código Java
  • Execute nossos testes de integração e publique os resultados

Aqui estão os passos que iremos:

  1. Certifique-se de que temos pelo menos uma versão 2.0 do Jenkins (você pode verificar isso no canto inferior direito da sua página), como:

  2. Na página inicial do Jenkins, clique em New Item

  3. Insira o nome do projeto e selecione Pipeline

  4. Na seção Build Triggers, selecione a opção Poll SCM e adicione a seguinte programação CRON de 5 minutos : */5 * * * *

  5. Na seção Pipeline, escolha Pipeline Script ou Pipeline Script from SCM

  6. Se você selecionou Pipeline Script from SCM na etapa anterior, agora você precisa especificar o URL do seu repositório SCM (Git, Mercurial, Subversion) em Repository URL, como http://github.com /example/example.git. Você também precisa especificar o Script Path do seu arquivo de script Groovy em seu repositório example.git, por exemplo pipelines/example.groovy

  7. Copie o seguinte código Groovy, diretamente na janela de script Groovy se você clicou anteriormente em Pipeline Script ou em seu example.groovy se você escolheu Pipeline Script from SCM

    node(‘remote’) { // Note : this step is only needed if you’re using direct Groovy scripting stage ‘Checkout Git project’ git url: ‘https://github.com/jglick/simple-maven-project-with-tests.git’ def appVersion = version() if (appVersion) { echo “Building version ${appVersion}” }

 def mvnHome = tool 'M3'

 step([$class: 'JUnitResultArchiver', testResults: '**/target/surefire-reports/TEST-*.xml'])

    } def version() { def matcher = readFile(‘pom.xml’) =~ ‘(.+)’ matcher ? matcher0 : null }
