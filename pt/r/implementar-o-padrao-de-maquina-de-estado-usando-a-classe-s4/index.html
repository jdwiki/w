Implementar o padrão de máquina de estado usando a classe S4

  • Analisando linhas usando a máquina de estado

Nesta página

  • Analisando linhas usando a máquina de estado

Os conceitos de Máquina de Estados Finitos são geralmente implementados em linguagens de Programação Orientada a Objetos (OOP), por exemplo, usando linguagem Java, baseada no padrão de estado definido em GOF (refere-se ao livro: “Design Patterns” ).

R fornece vários mecanismos para simular o paradigma OO, vamos aplicar S4 Object System para implementar esse padrão.

Analisando linhas usando a máquina de estado #

Vamos aplicar o Padrão de máquina de estado para analisar linhas com o padrão específico usando o recurso Classe S4 do R.

ENÚNCIA DO PROBLEMA

Precisamos analisar um arquivo onde cada linha fornece informações sobre uma pessoa, usando um delimitador (";"), mas algumas informações fornecidas são opcionais e, em vez de fornecer um campo vazio, estão faltando. Em cada linha podemos ter as seguintes informações: Nome;[Endereço;]Telefone. Onde a informação de endereço é opcional, às vezes temos e às vezes não, por exemplo:

GREGORY BROWN; 25 NE 25TH; +1-786-987-6543
DAVID SMITH;786-123-4567
ALAN PEREZ; 25 SE 50TH; +1-786-987-5553

A segunda linha não fornece informações de endereço. Portanto, o número de delimitadores pode ser deferente como neste caso com um delimitador e para as outras linhas dois delimitadores. Como o número de delimitadores pode variar, uma forma de atacar esse problema é reconhecer a presença ou não de um determinado campo com base em seu padrão. Nesse caso, podemos usar uma expressão regular para identificar esses padrões. Por exemplo:

  • Nome: "^([A-Z]'?\\s+)* *[A-Z]+(\\s+[A-Z]{1,2}\\.?,? +)*[A-Z] +((-|\\s+)[A-Z]+)*$". Por exemplo: RAFAEL REAL, DAVID R. SMITH, ERNESTO PEREZ GONZALEZ, 0' CONNOR BROWN, LUIS PEREZ-MENA, etc.
  • Endereço: "^\\s[0-9]{1,4}(\\s+[A-Z]{1,2}[0-9]{1,2}[A-Z]{1 ,2}|[A-Z\\s0-9]+)$". Por exemplo: 11020 LE JEUNE ROAD, 87 SW 27TH. Para simplificar não incluímos aqui o CEP, cidade, estado, mas posso incluir neste campo ou adicionar campos adicionais.
  • Telefone: "^\\s*(\\+1(-|\\s+))*[0-9]{3}(-|\\s+)[0-9]{3 }(-|\\s+)[0-9]{4}$". Por exemplo: 305-123-4567, 305 123 4567, +1-786-123-4567.

Notas:

  • Estou considerando o padrão mais comum de endereços e telefones dos EUA, pode ser facilmente estendido para considerar situações mais gerais.
  • Em R o sinal "\" tem um significado especial para variáveis ​​de caracteres, portanto precisamos escapar dele.
  • Para simplificar o processo de definição de expressões regulares, uma boa recomendação é usar a seguinte página da web: regex101.com, para que você possa brincar com ela, com um determinado exemplo, até obter o resultado esperado para todas as combinações possíveis.

A ideia é identificar cada campo de linha com base em padrões previamente definidos. O padrão State define as seguintes entidades (classes) que colaboram para controlar o comportamento específico (O padrão State é um padrão de comportamento):

Vamos descrever cada elemento considerando o contexto do nosso problema:

  • Contexto: Armazena as informações de contexto do processo de análise, ou seja, o estado atual e trata de todo o processo de máquina de estado. Para cada estado, uma ação é executada (handle()), mas o contexto a delega, baseado no estado, no método de ação definido para um determinado estado (handle() da classe State). Ele define a interface de interesse para os clientes. Nossa classe Context pode ser definida assim:
  • Atributos: estado
  • Métodos: handle(), …
  • State: A classe abstrata que representa qualquer estado da Máquina de Estado. Ele define uma interface para encapsular o comportamento associado a um estado particular do contexto. Pode ser definido assim:
  • Atributos: nome, padrão
  • Métodos: doAction(), isState (usando o atributo pattern verifica se o argumento de entrada pertence a este padrão de estado ou não), …
  • Concrete States (subclasses de estado): Cada subclasse da classe State que implementa um comportamento associado a um estado do Context. Nossas subclasses são: InitState, NameState, AddressState, PhoneState. Tais classes apenas implementam o método genérico usando a lógica específica para tais estados. Nenhum atributo adicional é necessário.

Observação: É uma questão de preferência como nomear o método que executa a ação, handle(), doAction() ou goNext(). O nome do método doAction() pode ser o mesmo para ambas as classes (State ou Context) que preferimos nomear como handle() na classe Context para evitar confusão ao definir dois métodos genéricos com os mesmos argumentos de entrada, mas classe diferente.

CLASSE PESSOAL

Usando a sintaxe do S4, podemos definir uma classe Person assim:

setClass(Class = "Person",
    slots = c(name = "character", address = "character", phone = "character")
)

É uma boa recomendação inicializar os atributos de classe. A documentação do setClass sugere o uso de um método genérico rotulado como "initialize", em vez de usar atributos obsoletos como: prototype, representação.

setMethod("initialize", "Person",
  definition = function(.Object, name = NA_character_,
    address = NA_character_, phone = NA_character_) {
        .Object@name <- name
        .Object@address <- address
        .Object@phone <- phone
        .Object
    }
)

Como o método initialize já é um método genérico padrão do pacote methods, precisamos respeitar a definição do argumento original. Podemos verificá-lo digitando no prompt do R:

> initialize

Ele retorna toda a definição da função, você pode ver no topo quem a função está definida como:

function (.Object, ...) {...}

Portanto quando usamos setMethod precisamos seguir exatamente a mesma sintaxe (.Object).

Outro método genérico existente é o show, é equivalente ao método toString() do Java e é uma boa ideia ter uma implementação específica para domínio de classe:

setMethod("show", signature = "Person",
  definition = function(object) {
      info <- sprintf("%s@[name='%s', address='%s', phone='%s']", 
        class(object), object@name, object@address, object@phone)
      cat(info)
      invisible(NULL)
  }
)

Observação: Usamos a mesma convenção da implementação Java padrão toString().

Digamos que queremos salvar as informações analisadas (uma lista de objetos Person) em um conjunto de dados, então devemos ser capazes de converter primeiro uma lista de objetos em algo que o R possa transformar (por exemplo, forçar o objeto como uma lista ). Podemos definir o seguinte método adicional (para mais detalhes sobre isso veja o post)

setGeneric(name = "as.list", signature = c('x'),
    def = function(x) standardGeneric("as.list"))

# Suggestion taken from here:
# http://stackoverflow.com/questions/30386009/how-to-extend-as-list-in-a-canonical-way-to-s4-objects
setMethod("as.list", signature = "Person",
    definition = function(x) {
        mapply(function(y) {
        #apply as.list if the slot is again an user-defined object
        #therefore, as.list gets applied recursively
        if (inherits(slot(x,y),"Person")) {
          as.list(slot(x,y))
        } else {
          #otherwise just return the slot
          slot(x,y)
        }
      },
        slotNames(class(x)),
        SIMPLIFY=FALSE)
    }
)

R não fornece uma sintaxe de açúcar para OO porque a linguagem foi inicialmente concebida para fornecer funções valiosas para estatísticos. Portanto, cada método de usuário requer duas partes: 1) a parte de Definição (via setGeneric) e 2) a parte de implementação (via setMethod). Como no exemplo acima.

CLASSE ESTADUAL

Seguindo a sintaxe do S4, vamos definir a classe abstrata State.

setClass(Class = "State", slots = c(name = "character", pattern = "character"))

setMethod("initialize", "State",
  definition = function(.Object, name = NA_character_, pattern = NA_character_) {
      .Object@name <- name
      .Object@pattern <- pattern
      .Object
  }
)

setMethod("show", signature = "State",
  definition = function(object) {
      info <- sprintf("%s@[name='%s', pattern='%s']", class(object), 
          object@name, object@pattern)
      cat(info)
      invisible(NULL)
  }
)

setGeneric(name = "isState", signature = c('obj', 'input'),
    def = function(obj, input) standardGeneric("isState"))

setGeneric(name = "doAction", signature = c('obj', 'input', 'context'),
    def = function(obj, input, context) standardGeneric("doAction"))

Cada subclasse de State terá associado um name e um pattern, mas também uma forma de identificar se uma determinada entrada pertence ou não a este estado (método isState()), e também implementar o correspondente ações para este estado (método doAction()).

Para entender o processo, vamos definir a matriz de transição para cada estado com base na entrada recebida:

| Entrada/Estado Atual | Iniciar | Nome | Endereço | Telefone | | ——————–|——|——– |———-|– ——-| | Nome | Nome | | | | | Endereço | | Endereço | | | | Telefone | | Telefone | Telefone | | | Fim | | | | Fim | Observação: A célula [row, col]=[i,j] representa o estado de destino para o estado atual j, quando recebe a entrada i.

Isso significa que sob o estado Nome ele pode receber duas entradas: um endereço ou um número de telefone. Outra maneira de representar a tabela de transações é usando o seguinte diagrama UML State Machine:

Vamos implementar cada estado particular como um subestado da classe State

** SUB-CLASSES DE ESTADO**

Estado de inicialização:

O estado inicial será implementado através da seguinte classe:

setClass("InitState", contains = "State")

setMethod("initialize", "InitState",
  definition = function(.Object, name = "init", pattern = NA_character_) {
      .Object@name <- name
      .Object@pattern <- pattern
      .Object
  }
)

setMethod("show", signature = "InitState",
  definition = function(object) {
      callNextMethod()
  }
)

Em R para indicar que uma classe é uma subclasse de outra classe é usar o atributo contains e indicar o nome da classe da classe pai.

Como as subclasses apenas implementam os métodos genéricos, sem adicionar atributos adicionais, então o método show, basta chamar o método equivalente da classe superior (via método: callNextMethod())

O estado inicial não possui um padrão associado, apenas representa o início do processo, então inicializamos a classe com um valor NA.

Agora vamos implementar os métodos genéricos da classe State:

setMethod(f = "isState", signature = "InitState",
  definition = function(obj, input) {
      nameState <- new("NameState")
      result <- isState(nameState, input)
      return(result)
  }
)

Para este estado em particular (sem pattern), a ideia de que ele apenas inicializa o processo de análise esperando que o primeiro campo seja um name, caso contrário, será um erro.

setMethod(f = "doAction", signature = "InitState",
    definition = function(obj, input, context) {
        nameState <- new("NameState")
        if (isState(nameState, input)) {
            person <- context@person
            person@name <- trimws(input)
            context@person <- person
            context@state <- nameState
        } else {
            msg <- sprintf("The input argument: '%s' cannot be identified", input)
            stop(msg)
        }
        return(context)
    }
)

O método doAction fornece a transição e atualiza o contexto com as informações extraídas. Aqui estamos acessando informações de contexto através do @-operator. Em vez disso, podemos definir métodos get/set, para encapsular este processo (como é obrigatório nas melhores práticas OO: encapsulamento), mas isso adicionaria mais quatro métodos por get-set sem agregar valor para o propósito deste exemplo.

É uma boa recomendação em toda implementação do doAction, adicionar uma proteção quando o argumento de entrada não for identificado corretamente.

Usamos a função grepl para verificar se a entrada pertence a um determinado padrão.

setMethod(f="isState", signature="NameState",
  definition=function(obj, input) {
      result <- grepl(obj@pattern, input, perl=TRUE)
      return(result)
  }
)

Agora definimos a ação a ser executada para um determinado estado:

setMethod(f = "doAction", signature = "NameState",
  definition=function(obj, input, context) {
      addressState <- new("AddressState")
      phoneState <- new("PhoneState")
      person <- context@person
      if (isState(addressState, input)) {
          person@address <- trimws(input)
          context@person <- person
          context@state <- addressState
      } else if (isState(phoneState, input)) {
          person@phone <- trimws(input)
          context@person <- person
          context@state <- phoneState
      } else {
          msg <- sprintf("The input argument: '%s' cannot be identified", input)
          stop(msg)
      }
      return(context)
  }
)

Aqui consideramos as transições possíveis: uma para o estado do endereço e outra para o estado do telefone. Em todos os casos, atualizamos as informações de contexto:

  • A informação person: address ou phone com o argumento de entrada.
  • O ’estado’ do processo

A maneira de identificar o estado é invocar o método: isState() para um estado específico. Criamos um estado específico padrão (addressState, phoneState) e, em seguida, solicitamos uma validação específica.

A lógica para a implementação das outras subclasses (uma por estado) é muito semelhante.

CLASSE DE CONTEXTO

Agora vamos explicar a implementação da classe Context. Podemos defini-lo considerando os seguintes atributos:

setClass(Class = "Context",
     slots = c(state = "State", persons = "list", person = "Person")
)

Onde

  • state: O estado atual do processo
  • person: A pessoa atual, representa a informação que já analisamos da linha atual.
  • persons: A lista de pessoas analisadas processadas.

Observação: Opcionalmente, podemos adicionar um nome para identificar o contexto pelo nome caso estejamos trabalhando com mais de um tipo de analisador.

setMethod(f="initialize", signature="Context",
  definition = function(.Object) {
        .Object@state <- new("InitState")
        .Object@persons <- list()
        .Object@person <- new("Person")
        return(.Object)
    }
)

setMethod("show", signature = "Context",
  definition = function(object) {
      cat("An object of class ", class(object), "\n", sep = "")
      info <- sprintf("[state='%s', persons='%s', person='%s']", object@state,
          toString(object@persons), object@person)
      cat(info)
      invisible(NULL)
  }
)

setGeneric(name = "handle", signature = c('obj', 'input', 'context'),
    def = function(obj, input, context) standardGeneric("handle"))

setGeneric(name = "addPerson", signature = c('obj', 'person'),
    def = function(obj, person) standardGeneric("addPerson"))

setGeneric(name = "parseLine", signature = c('obj', 's'),
    def = function(obj, s) standardGeneric("parseLine"))

setGeneric(name = "parseLines", signature = c('obj', 's'),
    def = function(obj, s) standardGeneric("parseLines"))

setGeneric(name = "as.df", signature = c('obj'),
    def = function(obj) standardGeneric("as.df"))

Com esses métodos genéricos, controlamos todo o comportamento do processo de análise sintática:

  • handle(): Irá invocar o método doAction() particular do state atual.
  • addPerson: Uma vez que alcançamos o estado final, precisamos adicionar uma pessoa à lista de pessoas que analisamos.
  • parseLine(): analisa uma única linha
  • parseLines(): analisa várias linhas (um array de linhas)
  • as.df(): Extraia as informações da lista persons em um objeto de quadro de dados.

Vamos continuar agora com as implementações correspondentes:

handle(), delega no método doAction() do state atual do context:

setMethod(f = "handle", signature = "Context",
    definition = function(obj, input) {
        obj <- doAction(obj@state, input, obj)
        return(obj)
    }
)

setMethod(f = "addPerson", signature = "Context",
  definition = function(obj, person) {
      obj@persons <- c(obj@persons, person)
      return(obj)
  }
)

Primeiro, dividimos a linha original em um array usando o delimitador para identificar cada elemento por meio da função R strsplit(), então iteramos para cada elemento como um valor de entrada para um determinado estado. O método handle() retorna novamente o context com as informações atualizadas (atributo state, person, persons).

setMethod(f = "parseLine", signature = "Context",
  definition = function(obj, s) {
      elements <- strsplit(s, ";")[[1]]
      # Adding an empty field for considering the end state.
      elements <- c(elements, "")
      n <- length(elements)
      input <- NULL
      for (i in (1:n)) {
        input <- elements[i]  
        obj <- handle(obj, input)
      }
      return(obj@person)
  }
)

Como R faz uma cópia do argumento de entrada, precisamos retornar o contexto (obj):

setMethod(f = "parseLines", signature = "Context",
  definition = function(obj, s) {
      n <- length(s)
      listOfPersons <- list()
      for (i in (1:n)) {
          ipersons <- parseLine(obj, s[i])
          listOfPersons[[i]] <- ipersons
      }
      obj@persons <- listOfPersons
      return(obj)
  }
)

O atributo persons é uma lista de instâncias da classe Person do S4. Esse algo não pode ser coagido a nenhum tipo padrão porque R não sabe como tratar uma instância de uma classe definida pelo usuário. A solução é converter uma Person em uma lista, utilizando o método as.list previamente definido. Então podemos aplicar esta função a cada elemento da lista persons, através da função lappy(). Então na próxima invocação da função lappy(), agora aplica a função data.frame para converter cada elemento do persons.list em um data frame. Finalmente, a função rbind() é chamada para adicionar cada elemento convertido como uma nova linha do quadro de dados gerado (para mais detalhes sobre isso veja este post)

# Sugestion taken from this post:
# http://stackoverflow.com/questions/4227223/r-list-to-data-frame
setMethod(f = "as.df", signature = "Context",
  definition = function(obj) {
    persons <- obj@persons
    persons.list <- lapply(persons, as.list)
    persons.ds <- do.call(rbind, lapply(persons.list, data.frame, stringsAsFactors = FALSE))
    return(persons.ds)
  }
)

CONCLUSÃO

Este exemplo mostra como implementar o padrão State, usando um dos mecanismos disponíveis do R para usar o paradigma OO. No entanto, a solução ROO não é amigável e difere muito de outras linguagens OOP. Você precisa mudar sua mentalidade porque a sintaxe é completamente diferente, lembra mais o paradigma de programação funcional. Por exemplo, em vez de: object.setID("A1") como em Java/C#, para R você deve invocar o método desta forma: setID(object, "A1"). Portanto, você sempre deve incluir o objeto como um argumento de entrada para fornecer o contexto da função. Da mesma forma, não há um atributo de classe especial this e uma notação ".." para acessar métodos ou atributos de uma determinada classe. É mais um prompt de erro porque referenciar uma classe ou métodos é feito via valor de atributo ("Person", "isState", etc.).

Dito acima, a solução de classe S4 requer muito mais linhas de códigos do que as linguagens Java/C# tradicionais para realizar tarefas simples. De qualquer forma, o State Pattern é uma solução boa e genérica para esse tipo de problema. Ele simplifica o processo de delegação da lógica em um estado específico. Em vez de ter um grande bloco if-else para controlar todas as situações, temos blocos if-else menores dentro de cada implementação de subclasse State para implementar a ação a ser executada em cada estado.
