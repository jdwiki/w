Ambientes como mapas de hash #

Observação: nas passagens subsequentes, os termos mapa de hash e tabela de hash são usados ​​alternadamente e referem-se ao mesmo conceito, ou seja, uma estrutura de dados que fornece pesquisa de chave eficiente por meio do uso de uma função de hash interna.

Introdução #

Embora R não forneça uma estrutura de tabela de hash nativa, uma funcionalidade semelhante pode ser alcançada aproveitando o fato de que o objeto environment retornado de new.env (por padrão) fornece pesquisas de chave hash. As duas instruções a seguir são equivalentes, pois o parâmetro hash tem como padrão TRUE:

H <- new.env(hash = TRUE)
H <- new.env()

Além disso, pode-se especificar que a tabela de hash interna seja pré-alocada com um tamanho específico através do parâmetro size, que tem um valor padrão de 29. Como todos os outros objetos R, environments gerenciam sua própria memória e crescerão em capacidade conforme necessário, portanto, embora não seja necessário solicitar um valor não padrão para size, pode haver uma pequena vantagem de desempenho ao fazê-lo se o objeto (eventualmente) contiver um número muito grande de elementos. Vale a pena notar que alocar espaço extra via size não resulta, por si só, em um objeto com uma pegada de memória maior:

object.size(new.env())
# 56 bytes

object.size(new.env(size = 10e4))
# 56 bytes

Inserção #

A inserção de elementos pode ser feita usando um dos métodos [[<- ou $<- fornecidos para a classe environment, mas não usando a atribuição de “colchete único” ([<- ):

H <- new.env()

H[["key"]] <- rnorm(1)

key2 <- "xyz"
H[[key2]] <- data.frame(x = 1:3, y = letters[1:3])

H$another_key <- matrix(rbinom(9, 1, 0.5) > 0, nrow = 3)

H["error"] <- 42
#Error in H["error"] <- 42 : 
#  object of type 'environment' is not subsettable

Como outras facetas do R, o primeiro método (object[[key]] <- value) é geralmente preferido ao segundo (object$key <- value) porque no primeiro caso, uma variável pode ser usada no lugar de um valor literal (por exemplo, key2 no exemplo acima).

Como geralmente é o caso com implementações de mapa de hash, o objeto environment não armazenará chaves duplicadas. A tentativa de inserir um par chave-valor para uma chave existente substituirá o valor armazenado anteriormente:

H[["key3"]] <- "original value"

H[["key3"]] <- "new value"

H[["key3"]]
#[1] "new value"

Pesquisa de chave #

Da mesma forma, os elementos podem ser acessados ​​com [[ ou $, mas não com [:

H[["key"]]
#[1] 1.630631
 
H[[key2]]   ## assuming key2 <- "xyz"
#   x y
# 1 1 a
# 2 2 b
# 3 3 c

H$another_key
#       [,1]  [,2]  [,3]
# [1,]  TRUE  TRUE  TRUE
# [2,] FALSE FALSE FALSE
# [3,]  TRUE  TRUE  TRUE

H[1]
#Error in H[1] : object of type 'environment' is not subsettable

Inspecionando o mapa de hash #

Sendo apenas um ambiente comum, o mapa de hash pode ser inspecionado por meios típicos:

names(H)
#[1] "another_key" "xyz"         "key"         "key3"       

ls(H)
#[1] "another_key" "key"         "key3"        "xyz"        
 
str(H)
#<environment: 0x7828228> 
 
ls.str(H)
# another_key :  logi [1:3, 1:3] TRUE FALSE TRUE TRUE FALSE TRUE ...
# key :  num 1.63
# key3 :  chr "new value"
# xyz : 'data.frame':    3 obs. of  2 variables:
#  $ x: int  1 2 3
#  $ y: chr  "a" "b" "c"

Elementos podem ser removidos usando rm:

rm(list = c("key", "key3"), envir = H)

ls.str(H)
# another_key :  logi [1:3, 1:3] TRUE FALSE TRUE TRUE FALSE TRUE ...
# xyz : 'data.frame':    3 obs. of  2 variables:
#  $ x: int  1 2 3
#  $ y: chr  "a" "b" "c"

Flexibilidade #

Um dos principais benefícios de usar objetos environment como tabelas de hash é a capacidade de armazenar praticamente qualquer tipo de objeto como valor, mesmo outros environments:

H2 <- new.env()

H2[["a"]] <- LETTERS
H2[["b"]] <- as.list(x = 1:5, y = matrix(rnorm(10), 2))
H2[["c"]] <- head(mtcars, 3)
H2[["d"]] <- Sys.Date()
H2[["e"]] <- Sys.time()
H2[["f"]] <- (function() {
    H3 <- new.env()
    for (i in seq_along(names(H2))) {
        H3[[names(H2)[i]]] <- H2[[names(H2)[i]]]
    }
    H3
})()

Limitações #

Uma das principais limitações do uso de objetos environment como mapas de hash é que, ao contrário de muitos aspectos do R, a vetorização não é suportada para pesquisa/inserção de elementos:

names(H2)
#[1] "a" "b" "c" "d" "e" "f"

H2[[c("a", "b")]]
#Error in H2[[c("a", "b")]] : 
#  wrong arguments for subsetting an environment
 
Keys <- c("a", "b")
H2[[Keys]]
#Error in H2[[Keys]] : wrong arguments for subsetting an environment

Dependendo da natureza dos dados armazenados no objeto, pode ser possível usar vappy ou list2env para atribuir vários elementos de uma só vez:

E1 <- new.env()
invisible({
    vapply(letters, function(x) {
        E1[[x]] <- rnorm(1)
        logical(0)
    }, FUN.VALUE = logical(0))
})

all.equal(sort(names(E1)), letters)
#[1] TRUE

Keys <- letters
E2 <- list2env(
    setNames(
        as.list(rnorm(26)),
        nm = Keys), 
    envir = NULL,
    hash = TRUE
)

all.equal(sort(names(E2)), letters)
#[1] TRUE

Nenhuma das opções acima é particularmente concisa, mas pode ser preferível usar um loop for etc. quando o número de pares chave-valor é grande.

pacote:hash #

O pacote de hash oferece uma estrutura de hash em R. No entanto, terms of timing para ambas as inserções e leituras ele se compara desfavoravelmente ao uso ambientes como um hash. Esta documentação simplesmente reconhece sua existência e fornece um exemplo de código de tempo abaixo pelos motivos declarados acima. Não há nenhum caso identificado em que o hash seja uma solução apropriada no código R hoje.

Considerar:

# Generic unique string generator
unique_strings <- function(n){
    string_i <- 1
    string_len <- 1
    ans <- character(n)
    chars <- c(letters,LETTERS)
    new_strings <- function(len,pfx){
    for(i in 1:length(chars)){
        if (len == 1){
        ans[string_i] <<- paste(pfx,chars[i],sep='')
        string_i <<- string_i + 1
        } else {
        new_strings(len-1,pfx=paste(pfx,chars[i],sep=''))
        }
        if (string_i > n) return ()
    }
    }
    while(string_i <= n){
    new_strings(string_len,'')
    string_len <- string_len + 1
    }
    sample(ans)
}

# Generate timings using an enviornment
timingsEnv <- plyr::adply(2^(10:15),.mar=1,.fun=function(i){
    strings <- unique_strings(i)
    ht1 <- new.env(hash=TRUE)
    lapply(strings, function(s){ ht1[[s]] <<- 0L})
    data.frame(

pacote:listenv #

Embora package:listenv implemente uma interface de lista para ambientes, seu desempenho em relação a ambientes para fins de hash é fraco na recuperação de hash . No entanto, se os índices forem numéricos, a recuperação pode ser bastante rápida. No entanto, eles têm outras vantagens, e. compatibilidade com package:future. A cobertura deste pacote para esse fim vai além do escopo do tópico atual. No entanto, o código de tempo fornecido aqui pode ser usado em conjunto com o exemplo para package:hash para tempos de gravação.

timingsListEnv <- plyr::adply(2^(10:15),.mar=1,.fun=function(i){
    strings <- unique_strings(i)
    le <- listenv::listenv()
    lapply(strings, function(s) le[[s]] <<- 0L)
    data.frame(
    size=c(i,i),
    seconds=c(
        system.time(for (k in 1:i) le[[k]]==0L)[3]),
    type = c('2_numericListEnv')
    )
})
