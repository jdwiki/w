Introdução aos Typemaps

  • Sintaxe
  • Parâmetros
  • Mapa de tipos básico - Python

Nesta página

  • Sintaxe
  • Parâmetros
  • Mapa de tipos básico - Python

Typemaps são o coração do que o SWIG faz. Quando você deseja passar dados entre idiomas, os comportamentos para isso dependem do tipo que o SWIG vê. O poder dos typemaps é que os pedaços de código são aplicados muitas vezes.

O próprio SWIG inclui muitos mapas de tipos úteis na biblioteca principal com a qual é fornecido, por exemplo, para tipos primitivos, contêineres de biblioteca padrão C++, boost etc. muitas vezes você nem precisará escrever nenhum tipo de mapa para expor seu código, no entanto, essa lista não está completa.

Sintaxe #

  • %typemap(NAME) TYPENAME %{ CÓDIGO %}
  • %typemap(NAME,OPTION=VALUE) TYPENAME %{ CODE %}
  • %typemap(NAME) TYPENAME VARIABLENAME %{ CÓDIGO %}
  • %typemap(NAME) TYPENAME (LOCALVARTYPE LOCALVARNAME) %{ CODE %}

Parâmetros #

Parâmetro Detalhes
NOME O nome do typemap define seu papel na geração de um módulo. in e out são comuns e usados ​​para entrada (para chamadas de função C++ ou C de Python/Java etc.) e saída (ou seja, valores de retorno de C ou C++ para Pyton/Java)
TYPENAME Cada mapa de tipos é aplicado a um ou mais tipos correspondentes. Estes precisam ser listados aqui. Qualificadores de tipo (por exemplo, const) são importantes.
CÓDIGO Cada typemap precisa ser convertido entre o tipo C ou C++ e o tipo correspondente na linguagem encapsulada. Você precisará escrever código para fazer isso em seus mapas de tipos personalizados, normalmente fazendo uso de variáveis ​​especiais que são substituídas. $input dentro de um typemap in representa o valor de Python/Java, $result em um typemap out é o que está sendo retornado para Python/Java. $1 em ambos os mapas de tipos de entrada/saída representam a variável C ou C++ do tipo usado para corresponder ao mapa de tipos. Então, para mapas de tipos in você atribuiria a $1 e para out você leria a partir dele. (Consulte os mapas de tipos de vários argumentos para obter mais detalhes sobre por que é um número)
VARIABLENAME Os mapas de tipo são combinados do mais específico ao menos específico em geral. Você pode definir um mapa de tipos que corresponderá apenas a argumentos de função com nomes específicos usando este formulário opcional. (Consulte a correspondência do mapa de tipos para obter mais detalhes)
(LOCALVARTYPE LOCALVARNAME) Às vezes, particularmente para mapas de tipos in, é útil poder declarar variáveis ​​locais extras para manter objetos em torno de uma chamada. Esta sintaxe opcional nos permite fazer isso. Ao usar esta sintaxe em vez de escrevê-la em CODE o escopo é alterado, mas mais importante a variável pode ser renomeada automaticamente pelo SWIG para evitar conflitos se uma função tiver dois argumentos usando o mesmo mapa de tipos.
OPÇÃO=VALOR O comportamento de alguns mapas de tipos pode ser influenciado pela configuração de opções extras usando esta sintaxe. Por exemplo, um typemap in pode ser feito para não receber nenhuma entrada do idioma de destino, definindo numinputs=0, caso em que o typemap deve preencher a entrada implicitamente. (Um caso comum para isso pode ser definir algo como NULL ou preenchê-lo a partir de um valor global)

Mapa de tipos básico - Python #

Dado o seguinte tipo booleano personalizado, queremos envolver:

typedef char MYBOOL;
#define TRUE 1
#define FALSE 0

Uma abordagem simples pode ser escrever os seguintes mapas de tipos em nossa interface SWIG:

%typemap(in) MYBOOL %{
  // $input is what we got passed from Python for this function argument
  $1 = PyObject_IsTrue($input);
  // $1 is what will be used for the C or C++ call and we are responsible for setting it
%}

%typemap(out) MYBOOL %{
  // $1 is what we got from our C or C++ call
  $result = PyBool_FromLong($1);
  // $result is what gets given back to Python and we are responsible for setting it
%}

Com esses mapas de tipos, o SWIG inserirá nosso código no wrapper gerado toda vez que vir um MYBOOL passado para dentro ou para fora de uma chamada de função.
