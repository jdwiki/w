Gherkin é uma linguagem comercialmente legível para automação de testes e documentação de testes. É entendido por Cucumber e em conjunto existe como uma ferramenta de Desenvolvimento Orientado por Comportamento.

Sintaxe #

  • Recurso: esta palavra-chave significa que o que se segue é um descrição ou nome do recurso que está sendo testado ou documentado.
  • Background: esta palavra-chave significa etapas que serão executadas antes de cada cenário no recurso.
  • Cenário: esta palavra-chave representa o nome ou descrição básica de um cenário específico testando o recurso.
  • Scenario Outline: Esta palavra-chave significa que o cenário será executado N vezes para cada argumento listado nos exemplos explicitamente passados ​​pelo nome da coluna entre colchetes angulares.
  • Exemplos: esta palavra-chave anota a lista de argumentos estáticos que serão passados ​​em um esboço de cenário.
  • Dado: esta palavra-chave representa um determinado passo, ou pré-condição que é assumida antes continuando. No paradigma Organizar, Agir, Afirmar, dado representa “Arranjo”.
  • Quando: esta palavra-chave representa uma etapa quando, ou o comportamento que deve ser afirmado contra. No paradigma Organizar, Agir, Afirmar, dado representa “Ato”.
  • Then: esta palavra-chave representa uma etapa then, ou em outras palavras, a etapa na qual o resultado de um comportamento é validado. No paradigma Arranjar, Agir, Afirmar, dado representa “Afirmar”.
  • E: Esta palavra-chave é usada em conjunto com qualquer uma das palavras-chave acima. Se você tiver duas instruções fornecidas, em vez de chamar explicitamente Dado duas vezes, você pode dizer, “dado A e B”.

Etapas parametrizadas #

Ao escrever Gherkin, pode haver momentos em que você deseja parametrizar suas etapas para reutilização pelo engenheiro que está implementando os planos de teste. As etapas recebem parâmetros por meio de grupos de captura de expressões regulares. (Observação de engenharia: Se você não tiver parâmetros correspondentes para cada grupo de captura em sua expressão regular, poderá esperar que uma “CucumberException: Arity mismatch” seja lançada) No exemplo abaixo, decidimos envolver os argumentos em double aspas, bem como aceitar inteiros como argumentos.

 Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with "valid" credentials
        Then the user should be logged in

    Scenario: The user attempts to log in with invalid credentials 
        This scenario tests that a user is not able to log in when
        they enter invalid credentials

        Given the user is on the login page
        When the user signs in with "invalid" credentials
        Then the user should be logged in

    Scenario: The user is locked out after too many failed attempts
        This scenario validates that the user is locked out
        of their account after failing three consecutive 
        attempts to log in

        Given the user is on the login page
        When the user fails to log in 3 times
        Then the user should be locked out of their account

Fundo do recurso #

Como você deve ter notado no exemplo acima, estamos reescrevendo a mesma etapa várias vezes:

Given the user is on the login page

Isso pode ser excepcionalmente tedioso, especialmente se tivermos mais de uma etapa que é reutilizada. O Gherkin oferece uma solução para isso, fornecendo outra palavra-chave com a qual trabalhar: Background:.

A palavra-chave background serve para executar as etapas declaradas abaixo dela antes de cada cenário no Recurso. Certifique-se de não adicionar a etapa de fundo, a menos que você tenha certeza de que é necessário para todos os cenários. Assim como as outras palavras-chave, Background é seguido por uma descrição ou nome e pode ter comentários listados abaixo. Assim como o recurso e o cenário, o plano de fundo deve ser precedido por dois pontos.

Tag #

Para fins de documentação, convém filtrar planos ou cenários de teste por categorias. Os desenvolvedores podem querer executar testes com base nessas mesmas categorias. O Gherkin permite que você categorize Recursos e Cenários individuais por meio do usuário de Tags. No exemplo abaixo, observe que a palavra-chave Recurso acima é a Tag “@Automation”. Gherkin reconhece isso como uma tag pelo usuário do símbolo “@”. Neste exemplo, o engenheiro quer deixar claro que esses testes são usados ​​para automação, onde nem todo teste é automatizável, alguns testes devem ser feitos por controle de qualidade manual.

Observe também que a tag @Production foi adicionada ao bloqueio de usuário de teste de cenário. Neste exemplo, isso ocorre porque esse cenário está ativo apenas no ambiente de produção do aplicativo. Os desenvolvedores não querem que suas contas de sandbox sejam bloqueadas durante o desenvolvimento. Essas tags permitem que eles imponham que esse teste seja executado apenas no ambiente de produção.

Por fim, o Scenario Outline tem a tag @Staging. Para os propósitos deste exemplo, isso ocorre porque as contas usadas são contas de teste e não funcionarão nos outros ambientes. Assim como a tag @Production, isso garante que esses testes sejam executados apenas no ambiente Staging.

Estes são apenas alguns exemplos de onde, como e por que você pode usar tags. Em última análise, essas tags terão significado para você e os desenvolvedores e podem ser qualquer coisa e usadas para categorizar da maneira que você achar melhor.

Dicas de pepino #

  • Cada cenário testa um comportamento
  • Os cenários são escritos de forma declarativa
  • Evite detalhes incidentais dentro do cenário
  • Omitir o óbvio
  • Evitar passos conjuntivos
  • Mantenha seus cenários curtos
  • Não tem muitos cenários no mesmo recurso
  • Use nomes de cenários descritivos
  • Tenha apenas um passo Quando
  • Use o “deveria” nas etapas Então

O básico #

Este exemplo abordará a estrutura básica de um arquivo de recurso do Cucumber no Gherkin. Os arquivos de recursos usam várias palavras-chave na sintaxe básica.

Vejamos as palavras-chave básicas:

  • Recurso: esta palavra-chave significa que o que segue é uma descrição básica ou o nome do recurso que está sendo testado ou documentado.
  • Cenário: esta palavra-chave representa o nome ou a descrição básica de um cenário específico testando o recurso.
  • Dado esta palavra-chave representa uma determinada etapa ou pré-condição que é assumida antes de continuar. No paradigma Arrange, Act, Assert, dado representa “Arrange”.
  • Quando essa palavra-chave representa uma etapa quando ou o comportamento que deve ser declarado. No paradigma Arranjar, Agir, Afirmar, dado representa “Agir”.
  • Then esta palavra-chave representa uma etapa then, ou seja, a etapa na qual o resultado de um comportamento é validado. No paradigma Arranjar, Agir, Afirmar, dado representa “Afirmar”.
  • E Essa palavra-chave é usada em conjunto com qualquer uma das palavras-chave acima. Se você tiver duas instruções fornecidas, em vez de chamar explicitamente Dado duas vezes, você pode dizer “Dado A e B”.
  • Mas Essa palavra-chave é usada em conjunto Dado, Quando e Depois para indicar que algo não deve acontecer. Então A Mas não B.

Todas as palavras-chave devem estar em uma nova linha e devem ser a primeira palavra em uma nova linha para serem reconhecidas pelo analisador Gherkin. As palavras-chave Recurso e Cenário devem ter dois pontos imediatamente após, conforme expresso no exemplo abaixo. Dado, Quando, Então e E não requerem dois pontos.

Além das palavras-chave, você pode escrever descrições e comentários. As descrições ocorrem após a palavra-chave, mas na mesma linha, enquanto os comentários ocorrem nas linhas abaixo das palavras-chave. Ao escrever comentários de recursos, é comum fornecer regras explícitas delineando arestas e condições que levam a diferentes resultados de comportamentos.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Scenario: The user successfully logs in with valid credentials 
        This scenario tests that a user is able to successfully login
        provided they enter a valid username, valid password, and 
        currently have an active subscription on their account. 

        Given the user is on the login page
        When the user signs in with valid credentials
        Then the user should be logged in

Esboço do Cenário #

Em alguns casos, você pode querer executar novamente o mesmo cenário várias vezes, substituindo os argumentos. Nesse caso, o Gherkin fornece várias novas palavras-chave para acomodar essa situação, Scenario Outline: e Example:. A palavra-chave Scenario Outline informa ao Cucumber que o cenário será executado várias vezes substituindo argumentos de uma lista. A palavra-chave Examples é chamada antes que a lista seja explicitamente notada. Argumentos para Esboços de Cenários devem ser colocados entre colchetes angulares. No exemplo abaixo, observe que os nomes dos argumentos entre colchetes angulares correspondem aos nomes das colunas listados em Exemplos. Cada coluna é separada por barras verticais, com os nomes das colunas na primeira linha.

Feature: Product Login
    As a user, I would like to be able to use my credentials to successfully 
    login. 
    
    Rules:
    - The user must have a valid username
    - The user must have a valid password
    - The user must have an active subscription 
    - User is locked out after 3 invalid attempts
    - User will get a generic error message following 
      login attempt with invalid credentials 

    Background: The user starts out on the login page
        Given the user is on the login page

    Scenario Outline: The user successfully logs in with their account
         This scenario outlines tests in which various users attempt
         to sign in successfully 

         When the user enters their <username>
         And the user enters their <password>
         Then the user should be successfully logged on

         Examples:
         | username | password |
         | frank    | 1234     |
