Forma e Estados de entrada #

Formulários e entradas angulares têm vários estados que são úteis ao validar conteúdo

Estados de entrada

Estado Descrição
$tocado Campo foi tocado
$intocado O campo não foi tocado
$pristine O campo não foi modificado
$sujo O campo foi modificado
$válido O conteúdo do campo é válido
$inválido O conteúdo do campo é inválido

Todos os estados acima são propriedades booleanas e podem ser verdadeiros ou falsos.

Com eles, é muito fácil exibir mensagens para um usuário.

<form name="myForm" novalidate>
    <input name="myName" ng-model="myName" required>
    <span ng-show="myForm.myName.$touched && myForm.myName.$invalid">This name is invalid</span>
</form>

Aqui, estamos usando a diretiva ng-show para exibir uma mensagem para um usuário se ele modificou um formulário, mas é inválido.

Classes CSS #

Angular também fornece algumas classes CSS para formulários e entradas dependendo de seu estado

Classe Descrição
ng-tocado Campo foi tocado
ng-intocado O campo não foi tocado
ng-pristine O campo não foi modificado
ng-sujo O campo foi modificado
de-válido O campo é válido
ng-inválido O campo é inválido

Você pode usar essas classes para adicionar estilos aos seus formulários

input.ng-invalid {
  background-color: crimson;
}
input.ng-valid {
  background-color: green;
}

Validação de formulário básico #

Um dos pontos fortes do Angular é a validação de formulários do lado do cliente.

Lidar com entradas de formulários tradicionais e ter que usar processamento interrogativo no estilo jQuery pode ser demorado e complicado. Angular permite que você produza formulários interativos profissionais com relativa facilidade.

A diretiva ng-model fornece ligação bidirecional com campos de entrada e geralmente o atributo novalidate também é colocado no elemento de formulário para evitar que o navegador faça a validação nativa.

Assim, um formulário simples ficaria assim:

<form name="form" novalidate>
  <label name="email"> Your email </label>
  <input type="email" name="email" ng-model="email" />
</form>

Para que o Angular valide as entradas, use exatamente a mesma sintaxe que um elemento input regular, exceto pela adição do atributo ng-model para especificar a qual variável ligar no escopo. O e-mail é mostrado no exemplo anterior. Para validar um número, a sintaxe seria:

<input type="number" name="postalcode" ng-model="zipcode" />

As etapas finais para a validação básica do formulário são conectar-se a uma função de envio de formulário no controlador usando ng-submit, em vez de permitir que o envio de formulário padrão ocorra. Isso não é obrigatório, mas geralmente é usado, pois as variáveis ​​de entrada já estão disponíveis no escopo e, portanto, disponíveis para sua função de envio. Também é uma boa prática dar um nome ao formulário. Essas alterações resultariam na seguinte sintaxe:

<form name="signup_form" ng-submit="submitFunc()" novalidate>
  <label name="email"> Your email </label>
  <input type="email" name="email" ng-model="email" />
  <button type="submit">Signup</button>
</form>

Este código acima é funcional, mas há outras funcionalidades que o Angular fornece.

A próxima etapa é entender que o Angular anexa atributos de classe usando ng-pristine, ng-dirty, ng-valid e ng-invalid para processamento de formulários. Usar essas classes em seu CSS permitirá que você estilize os campos de entrada válido/inválido e impecável/sujo e, assim, altere a apresentação à medida que o usuário insere dados no formulário.

Validação de formulário personalizado #

Em alguns casos, a validação básica não é suficiente. Angular suporta validação personalizada adicionando funções validadoras ao objeto $validators no ngModelController:

angular.module('app', [])
  .directive('myValidator', function() {
    return {
      // element must have ng-model attribute
      // or $validators does not work
      require: 'ngModel',
      link: function(scope, elm, attrs, ctrl) {
        ctrl.$validators.myValidator = function(modelValue, viewValue) {
            // validate viewValue with your custom logic
            var valid = (viewValue && viewValue.length > 0) || false;
            return valid; 
        };
      } 
    };

O validador é definido como uma diretiva que requer ngModel, então para aplicar o validador basta adicionar a diretiva customizada ao controle do formulário de entrada.

<form name="form">
  <input type="text" 
         ng-model="model" 
         name="model" 
         my-validator>
  <pre ng-bind="'my-validator returned: ' + form.model.$valid"></pre>
</form>

E o my-validator não precisa ser aplicado no controle de formulário nativo. Pode ser qualquer elemento, desde que seja ng-model em seus atributos. Isso é útil quando você tem algum componente de interface do usuário de compilação personalizado.

ngMensagens #

ngMessages é usado para aprimorar o estilo de exibição de mensagens de validação na visualização.

Abordagem tradicional #

Antes do ngMessages, nós normalmente exibimos as mensagens de validação usando as diretivas pré-definidas do Angular ng-class. Esta abordagem era um lixo e uma tarefa repetitiva.

Agora, usando ngMessages podemos criar nossas próprias mensagens personalizadas.

Exemplo #

HTML:

<form name="ngMessagesDemo">
  <input name="firstname" type="text" ng-model="firstname" required>
  <div ng-messages="ngMessagesDemo.firstname.$error">
    <div ng-message="required">Firstname is required.</div>
  </div>
</form>
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.3.16/angular.min.js"></script>
<script src="https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.3.16/angular-messages.min.js"></script>

JS:

var app = angular.module('app', ['ngMessages']);

app.controller('mainCtrl', function ($scope) { 
    $scope.firstname = "Rohit";
});

Validadores assíncronos #

Os validadores assíncronos permitem validar as informações do formulário em relação ao seu back-end (usando $http).

Esses tipos de validadores são necessários quando você precisa acessar informações armazenadas no servidor que não podem ter em seu cliente por vários motivos, como a tabela de usuários e outras informações do banco de dados.

Para usar validadores assíncronos, você acessa o ng-model do seu input e define funções de retorno de chamada para a propriedade $asyncValidators.

Exemplo:

O exemplo a seguir verifica se um nome fornecido já existe, o back-end retornará um status que rejeitará a promessa se o nome já existir ou se não tiver sido fornecido. Se o nome não existir, ele retornará uma promessa resolvida.

ngModel.$asyncValidators.usernameValidate = function (name) {               
     if (name) {
          return AuthenticationService.checkIfNameExists(name); // returns a promise
     } else {
          return $q.reject("This username is already taken!"); // rejected promise
     }
};

Agora, toda vez que o ng-model da entrada for alterado, essa função será executada e retornará uma promessa com o resultado.

Formulários aninhados #

Às vezes, é desejável aninhar formulários com a finalidade de agrupar controles e entradas logicamente na página. No entanto, os formulários HTML5 não devem ser aninhados. Angular fornece ng-form em vez disso.

<form name="myForm" noValidate>
  <!-- nested form can be referenced via 'myForm.myNestedForm' -->
  <ng-form name="myNestedForm" noValidate>
    <input name="myInput1" ng-minlength="1" ng-model="input1" required />
    <input name="myInput2" ng-minlength="1" ng-model="input2" required />
  </ng-form>

  <!-- show errors for the nested subform here -->
  <div ng-messages="myForm.myNestedForm.$error">
    <!-- note that this will show if either input does not meet the minimum -->
    <div ng-message="minlength">Length is not at least 1</div>
  </div>
</form>

<!-- status of the form -->
<p>Has any field on my form been edited? {{myForm.$dirty}}</p>
<p>Is my nested form valid? {{myForm.myNestedForm.$valid}}</p>
<p>Is myInput1 valid? {{myForm.myNestedForm.myInput1.$valid}}</p>

Cada parte do formulário contribui para o estado geral do formulário. Portanto, se uma das entradas myInput1 tiver sido editada e for $dirty, o formulário que a contém também será $dirty. Isso cascata para cada formulário contido, então tanto myNestedForm quanto myForm serão $dirty.
