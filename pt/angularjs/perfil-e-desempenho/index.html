7 Melhorias de desempenho simples #

1) Use ng-repeat com moderação

Usar ng-repeat em visualizações geralmente resulta em desempenho ruim, principalmente quando há ng-repeats aninhados.

Isso é super lento!

<div ng-repeat="user in userCollection">
  <div ng-repeat="details in user">
    {{details}}
  </div>
</div>

Tente evitar repetições aninhadas tanto quanto possível. Uma maneira de melhorar o desempenho de ng-repeat é usar track by $index (ou algum outro campo de id). Por padrão, ng-repeat rastreia todo o objeto. Com track by, o Angular observa o objeto apenas pelo $index ou id do objeto.

<div ng-repeat="user in userCollection track by $index">
  {{user.data}}
</div>

Use outras abordagens como paginação, [rolagem virtual](http://stackoverflow. com/questions/33259241/how-can-i-make-an-virtual-scroll-with-angularjs), [rolos infinitos](http://stackoverflow.com/questions/21674266/angularjs-infinite-scroll-in- a-container) ou limitTo: begin sempre que possível para evitar a iteração em grandes coleções.

2) Vincular uma vez #

Angular tem vinculação de dados bidirecional. Ele vem com um custo de ser lento se usado demais.

Desempenho mais lento

<!-- Default data binding has a performance cost -->
<div>{{ my.data }}</div>

Desempenho mais rápido (AngularJS >= 1.3)

<!-- Bind once is much faster -->
<div>{{ ::my.data }}</div>

<div ng-bind="::my.data"></div>

<!-- Use single binding notation in ng-repeat where only list display is needed  -->
<div ng-repeat="user in ::userCollection">
  {{::user.data}}
</div>

Usar a notação “bind once” diz ao Angular para esperar que o valor se estabilize após a primeira série de ciclos de resumo. O Angular usará esse valor no DOM e removerá todos os observadores para que ele se torne um valor estático e não esteja mais vinculado ao modelo.

O {{}} é muito mais lento.

Este ng-bind é uma diretiva e colocará um observador na variável passada. Portanto, o ng-bind só será aplicado quando o valor passado realmente mudar.

Os colchetes, por outro lado, serão verificados e atualizados a cada $digest, mesmo que não seja necessário.

3) As funções e filtros do escopo levam tempo #

AngularJS tem um loop de resumo. Todas as suas funções estão em uma visualização e os filtros são executados toda vez que o ciclo de compilação é executado. O loop de resumo será executado sempre que o modelo for atualizado e pode tornar seu aplicativo mais lento (o filtro pode ser atingido várias vezes antes que a página seja carregada).

Evite isso:

<div ng-controller="bigCalulations as calc">
  <p>{{calc.calculateMe()}}</p>
  <p>{{calc.data | heavyFilter}}</p>
</div>

Melhor abordagem

<div ng-controller="bigCalulations as calc">
  <p>{{calc.preCalculatedValue}}</p>
  <p>{{calc.data | lightFilter}}</p>
</div>

Onde o controlador pode estar:

app.controller('bigCalulations', function(valueService) {
    // bad, because this is called in every digest loop
    this.calculateMe = function() {
        var t = 0;
        for(i = 0; i < 1000; i++) {
            t += i;
        }
        return t;
    }
    // good, because this is executed just once and logic is separated in service to    keep the controller light
    this.preCalulatedValue = valueService.valueCalculation(); // returns 499500
});

4) Vigilantes #

Os observadores caem tremendamente o desempenho. Com mais observadores, o loop de resumo levará mais tempo e a IU ficará mais lenta. Se o observador detectar a mudança, ele iniciará o loop de resumo e renderizará novamente a exibição.

Existem três maneiras de fazer a observação manual de alterações de variáveis ​​em Angular.

$watch() - observa mudanças de valor

$watchCollection() - observa as mudanças na coleção (observa mais do que o $watch normal)

$watch(..., true) - Evite isso tanto quanto possível, ele executará “observação profunda” e diminuirá o desempenho (observa mais do que watchCollection)

Observe que se você estiver vinculando variáveis ​​na visualização, você está criando novos relógios - use {{::variable}} para evitar a criação de um relógio, especialmente em loops.

Como resultado, você precisa rastrear quantos observadores você está usando. Você pode contar os observadores com este script (crédito a @Words Like Jared Número de observadores)

(function() {
    var root = angular.element(document.getElementsByTagName('body')),
        watchers = [],
        f = function(element) {
        angular.forEach(['$scope', '$isolateScope'], function(scopeProperty) {
            if(element.data() && element.data().hasOwnProperty(scopeProperty)) {
                angular.forEach(element.data()[scopeProperty].$$watchers, function(watcher) {
                watchers.push(watcher);
                });
            }
        });

        angular.forEach(element.children(), function(childElement) {
            f(angular.element(childElement));
        });
    };
 
    f(root);
 
    // Remove duplicate watchers
    var watchersWithoutDuplicates = [];
    angular.forEach(watchers, function(item) {
        if(watchersWithoutDuplicates.indexOf(item) < 0) {
            watchersWithoutDuplicates.push(item);
        }
    });
    console.log(watchersWithoutDuplicates.length);
})();

5) de-se / de-show #

Essas funções são muito semelhantes em comportamento. ng-if remove elementos do DOM enquanto ng-show apenas oculta os elementos, mas mantém todos os manipuladores. Se você tiver partes do código que não deseja mostrar, use ng-if.

Depende do tipo de uso, mas muitas vezes um é mais adequado que o outro.

  • Se o elemento não for necessário, use ng-if

  • Para ativar/desativar rapidamente, use ng-show/ng-hide

    <div ng-repeat="user in userCollection">
  <p ng-if="user.hasTreeLegs">I am special<!-- some complicated DOM --></p>
  <p ng-show="user.hasSubscribed">I am awesome<!-- switch this setting on and off --></p>
</div>

Em caso de dúvida - use ng-if e teste!

6) Desative a depuração #

Por padrão, diretivas e escopos de ligação deixam classes e marcações extras no código para auxiliar em várias ferramentas de depuração. Desabilitar essa opção significa que você não renderiza mais esses vários elementos durante o ciclo de compilação.

angular.module('exampleApp', []).config(['$compileProvider', function ($compileProvider) {
    $compileProvider.debugInfoEnabled(false);
}]);

7) Use injeção de dependência para expor seus recursos #

A injeção de dependência é um padrão de projeto de software no qual um objeto recebe suas dependências, em vez do próprio objeto criá-las. Trata-se de remover as dependências codificadas e tornar possível alterá-las sempre que necessário.

Você pode se perguntar sobre o custo de desempenho associado a essa análise de string de todas as funções injetáveis. Angular cuida disso armazenando em cache a propriedade $inject após a primeira vez. Portanto, isso não acontece toda vez que uma função precisa ser invocada.

DICA PRO: Se você está procurando a abordagem com o melhor desempenho, use a abordagem de anotação de propriedade $inject. Essa abordagem evita totalmente a análise da definição de função porque essa lógica é encapsulada na seguinte verificação na função de anotação: if (!($inject = fn.$inject)). Se $inject já estiver disponível, nenhuma análise é necessária!

var app = angular.module('DemoApp', []);

var DemoController = function (s, h) {
    h.get('https://api.github.com/users/angular/repos').success(function (repos) {
        s.repos = repos;
    });
}
// $inject property annotation
DemoController['$inject'] = ['$scope', '$http'];

app.controller('DemoController', DemoController);

DICA PRO 2: Você pode adicionar uma diretiva ng-strict-di no mesmo elemento que ng-app para optar pelo modo DI estrito, que lançará um erro sempre que um serviço tentar usar anotações implícitas. Exemplo:

<html ng-app="DemoApp" ng-strict-di>

Ou se você usar bootstrapping manual:

angular.bootstrap(document, ['DemoApp'], {
    strictDi: true
});

Vincular uma vez #

Angular tem a reputação de ter uma incrível ligação de dados bidirecional. Por padrão, o Angular sincroniza continuamente os valores vinculados entre os componentes de modelo e de exibição sempre que os dados são alterados no modelo ou no componente de exibição.

Isso vem com o custo de ser um pouco lento se usado demais. Isso terá um impacto maior no desempenho:

Desempenho ruim: {{my.data}}

Adicione dois pontos :: antes do nome da variável para usar uma ligação única. Nesse caso, o valor só é atualizado quando my.data é definido. Você está apontando explicitamente para não observar alterações de dados. Angular não realizará nenhuma verificação de valor, resultando em menos expressões sendo avaliadas em cada ciclo de compilação.

Exemplos de bom desempenho usando vinculação única

{{::my.data}}
<span ng-bind="::my.data"></span>
<span ng-if="::my.data"></span>
<span ng-repeat="item in ::my.data">{{item}}</span>
<span ng-class="::{ 'my-class': my.data }"></div>

Observação: isso, no entanto, remove a vinculação de dados bidirecional para my.data, portanto, sempre que esse campo for alterado em seu aplicativo, o mesmo não será refletido na visualização automaticamente. Portanto, use-o apenas para valores que não serão alterados durante a vida útil do seu aplicativo.

de-if vs of-show #

Essas funções são muito semelhantes em comportamento. A diferença é que ng-if remove elementos do DOM. Se houver grandes partes do código que não serão mostradas, então ng-if é o caminho a seguir. ng-show apenas ocultará os elementos, mas manterá todos os manipuladores.

de if #

A diretiva ngIf remove ou recria uma parte da árvore DOM com base em uma expressão. Se a expressão atribuída a ngIf for avaliada como um valor falso, o elemento será removido do DOM, caso contrário, um clone do elemento será reinserido no DOM.

de shows #

A diretiva ngShow mostra ou oculta o elemento HTML fornecido com base na expressão fornecida ao atributo ngShow. O elemento é mostrado ou oculto removendo ou adicionando a classe CSS ng-hide ao elemento.

Exemplo #

<div ng-repeat="user in userCollection">
  <p ng-if="user.hasTreeLegs">I am special
    <!-- some complicated DOM -->
  </p>
  <p ng-show="user.hasSubscribed">I am aweosme
    <!-- switch this setting on and off -->
  </p>
</div>

Conclusão #

Depende do tipo de uso, mas muitas vezes um é mais adequado que o outro (por exemplo, se 95% das vezes o elemento não for necessário, use ng-if; se você precisar alternar a visibilidade do elemento DOM, use ng-show).

Na dúvida, use ng-if e teste!

Observação: ng-if cria um novo escopo isolado, enquanto ng-show e ng-hide não. Use $parent.property se a propriedade do escopo pai não estiver diretamente acessível nele.

Vigilantes #

Observadores necessários para observar algum valor e detectar que este valor foi alterado.

Após chamar $watch() ou $watchCollection, o novo observador adiciona à coleção de observadores internos no escopo atual.

Então, o que é observador? #

Watcher é uma função simples, que é chamada em cada ciclo de compilação e retorna algum valor. Angular verifica o valor retornado, caso não seja o mesmo da chamada anterior - um callback que foi passado no segundo parâmetro para a função $watch() ou $watchCollection será executado.

(function() {
  angular.module("app", []).controller("ctrl", function($scope) {
    $scope.value = 10;
    $scope.$watch(
      function() { return $scope.value; },
      function() { console.log("value changed"); }
    );
  }
})();

Os observadores são assassinos de desempenho. Quanto mais observadores você tiver, mais tempo eles demoram para fazer um loop de resumo, mais lenta a interface do usuário. Se um observador detectar alterações, ele iniciará o loop de resumo (recálculo em todas as telas)

Existem três maneiras de observar manualmente as alterações de variáveis ​​em Angular.

$watch() - apenas observa as mudanças de valor

$watchCollection() - observa mudanças na coleção (observa mais do que $watch normal)

$watch(..., true) - Evite isso o máximo possível, ele executará “deep watch” e matará o desempenho (assiste mais do que watchCollection)

Observe que se você está vinculando variáveis ​​na visualização, você está criando novos observadores - use {{::variable}} para não criar observador, especialmente em loops

Como resultado, você precisa rastrear quantos observadores você está usando. Você pode contar os observadores com este script (crédito a @Words Like Jared - http://stackoverflow.com/questions/18499909/)

(function() {
  var root = angular.element(document.getElementsByTagName("body")),
      watchers = [];

  var f = function(element) {

    angular.forEach(["$scope", "$isolateScope"], function(scopeProperty) {
      if(element.data() && element.data().hasOwnProperty(scopeProperty)) {
        angular.forEach(element.data()[scopeProperty].$$watchers, function(watcher) {
          watchers.push(watcher);
        });
      }
    });

    angular.forEach(element.children(), function(childElement) {
      f(angular.element(childElement));
    });

  };

  f(root);

  // Remove duplicate watchers
  var watchersWithoutDuplicates = [];
  angular.forEach(watchers, function(item) {
    if(watchersWithoutDuplicates.indexOf(item) < 0) {
      watchersWithoutDuplicates.push(item);
    }
  });

  console.log(watchersWithoutDuplicates.length);

})();

Se você não quiser criar seu próprio script, existe um utilitário de código aberto chamado ng-stats que usa um gráfico em tempo real incorporado à página para fornecer informações sobre o número de relógios que o Angular está gerenciando, bem como a frequência e a duração dos ciclos de resumo ao longo do tempo. O utilitário expõe uma função global chamada showAngularStats que você pode chamar para configurar como deseja que o gráfico funcione.

showAngularStats({
  "position": "topleft",
  "digestTimeThreshold": 16,
  "autoload": true,
  "logDigest": true,
  "logWatches": true
});

O código de exemplo acima exibe o gráfico a seguir na página automaticamente (demonstração interativa).

Sempre cancele o registro de listeners registrados em outros escopos além do escopo atual #

Você deve sempre cancelar o registro de escopos diferentes do seu escopo atual, conforme mostrado abaixo:

//always deregister these
$rootScope.$on(...);
$scope.$parent.$on(...);

Você não precisa cancelar o registro dos ouvintes no escopo atual, pois o angular cuidaria disso:

//no need to deregister this
$scope.$on(...);

Os ouvintes $rootScope.$on permanecerão na memória se você navegar para outro controlador. Isso criará um vazamento de memória se o controlador ficar fora do escopo.

Não

angular.module('app').controller('badExampleController', badExample);
badExample.$inject = ['$scope', '$rootScope'];

function badExample($scope, $rootScope) {
    $rootScope.$on('post:created', function postCreated(event, data) {});
}

Fazer

angular.module('app').controller('goodExampleController', goodExample);
goodExample.$inject = ['$scope', '$rootScope'];

function goodExample($scope, $rootScope) {
    var deregister = $rootScope.$on('post:created', function postCreated(event, data) {});

    $scope.$on('$destroy', function destroyScope() {
        deregister();
    });
}

Funções e filtros de escopo #

AngularJS tem loop de resumo e todas as suas funções em uma visualização e filtros são executados toda vez que o ciclo de resumo é executado. O loop de resumo será executado sempre que o modelo for atualizado e pode tornar seu aplicativo mais lento (o filtro pode ser atingido várias vezes, antes que a página seja carregada).

Você deve evitar isso:

<div ng-controller="bigCalulations as calc">
  <p>{{calc.calculateMe()}}</p>
  <p>{{calc.data | heavyFilter}}</p>
</div>

Melhor abordagem

<div ng-controller="bigCalulations as calc">
  <p>{{calc.preCalculatedValue}}</p>
  <p>{{calc.data | lightFilter}}</p>
</div>

Onde a amostra do controlador é:

.controller("bigCalulations", function(valueService) {
  // bad, because this is called in every digest loop
  this.calculateMe = function() {
    var t = 0;
    for(i = 0; i < 1000; i++) {
      t = t + i;
    }
    return t;
  }
  //good, because it is executed just once and logic is separated in service to    keep the controller light
  this.preCalulatedValue = valueService.caluclateSumm(); // returns 499500
});

Debounce seu modelo #

<div ng-controller="ExampleController">
    <form name="userForm">
        Name:
        <input type="text" name="userName"
           ng-model="user.name"
           ng-model-options="{ debounce: 1000 }" />
            <button ng-click="userForm.userName.$rollbackViewValue();   user.name=''">Clear</button><br />
    </form>
    <pre>user.name = </pre>
</div>

No exemplo acima, estamos definindo um valor de debounce de 1000 milissegundos, que é 1 segundo. Este é um atraso considerável, mas evitará que a entrada destrua repetidamente ng-model com muitos ciclos $digest.

Ao usar o debounce em seus campos de entrada e em qualquer outro lugar onde uma atualização instantânea não seja necessária, você pode aumentar bastante o desempenho de seus aplicativos Angular. Você não apenas pode atrasar por tempo, mas também pode atrasar quando a ação é acionada. Se você não quiser atualizar seu ng-model a cada pressionamento de tecla, também poderá atualizar no desfoque.
