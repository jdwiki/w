Coisas para fazer ao usar html5Mode #

Ao usar html5Mode([mode]) é necessário que:

  1. Você especifica a URL base para o aplicativo com um <base href=""> no cabeçalho do seu index.html.

  2. É importante que a tag base venha antes de qualquer tag com solicitações de url. Caso contrário, isso pode resultar neste erro - "Recurso interpretado como folha de estilo, mas transferido com tipo MIME text/html". Por exemplo:

     <head>
     <meta charset="utf-8">
     <title>Job Seeker</title>

     <base href="/">

     <link rel="stylesheet" href="bower_components/bootstrap/dist/css/bootstrap.css" />
     <link rel="stylesheet" href="/styles/main.css">
 </head>

  3. Se você não deseja especificar uma tag base, configure $locationProvider para não exigir uma tag base passando um objeto de definição com requireBase:false para $locationProvider.html5Mode() assim :

     $locationProvider.html5Mode({
     enabled: true,
     requireBase: false
 });

  4. Para suportar o carregamento direto de URLs HTML5, você precisa habilitar a reescrita de URL do lado do servidor. De AngularJS / Guia do desenvolvedor / Usando $location

Usar este modo requer a reescrita de URL no lado do servidor, basicamente você tem que reescrever todos os seus links para o ponto de entrada do seu aplicativo (por exemplo, index.html). Exigir uma tag <base> também é importante para este caso, pois permite que o Angular diferencie entre a parte da url que é a base do aplicativo e o caminho que deve ser tratado pelo aplicativo.

Um excelente recurso para exemplos de reescrita de solicitação para várias implementações de servidor HTTP pode ser encontrado em Perguntas frequentes sobre roteadores de ui - Como: Configurar seu servidor para trabalhar com html5Mode. Por exemplo, apache

    RewriteEngine on

    # Don't rewrite files or directories
    RewriteCond %{REQUEST_FILENAME} -f [OR]
    RewriteCond %{REQUEST_FILENAME} -d
    RewriteRule ^ - [L]

    # Rewrite everything else to index.html to allow html5 state links
    RewriteRule ^ index.html [L]

nginx

    server {
        server_name my-app;

        root /path/to/app;

        location / {
            try_files $uri $uri/ /index.html;
        }
    }

Expressar

    var express = require('express');
    var app = express();

    app.use('/js', express.static(__dirname + '/js'));
    app.use('/dist', express.static(__dirname + '/../dist'));
    app.use('/css', express.static(__dirname + '/css'));
    app.use('/partials', express.static(__dirname + '/partials'));

    app.all('/*', function(req, res, next) {
        // Just send the index.html for other files to support HTML5Mode
        res.sendFile('index.html', { root: __dirname });
    });

    app.listen(3006); //the port you want to use

A vinculação de dados bidirecional para de funcionar #

Deve-se ter em mente que:

  1. A vinculação de dados do Angular depende da herança prototípica do JavaScript, portanto, está sujeita a sombreamento variável.
  2. Um escopo filho normalmente herda prototipicamente de seu escopo pai. Uma exceção a esta regra é uma diretiva que tem um escopo isolado, pois não herda prototipicamente.
  3. Existem algumas diretivas que criam um novo escopo filho: ng-repeat, ng-switch, ng-view, ng-if, ng-controller, ng-include, etc. .

Isso significa que quando você tenta vincular bidirecionalmente alguns dados a um primitivo que está dentro de um escopo filho (ou vice-versa), as coisas podem não funcionar conforme o esperado. Aqui é um exemplo de como é fácil “quebrar” o AngularJS.

Esse problema pode ser facilmente evitado seguindo estas etapas:

  1. Tenha um “.” dentro do seu modelo HTML sempre que você vincular alguns dados
  2. Use a sintaxe controllerAs, pois ela promove o uso de vinculação a um objeto “pontilhado”
  3. $parent pode ser usado para acessar variáveis ​​pai scope ao invés do escopo filho. como dentro de ng-if podemos usar ng-model="$parent.foo"..

Uma alternativa para o acima é vincular ngModel a uma função getter/setter que atualizará a versão em cache do modelo quando chamada com argumentos ou a retornará quando chamada sem argumentos. Para usar uma função getter/setter, você precisa adicionar ng-model-options="{ getterSetter: true }" ao elemento com o atributo ngModal e chamar a função getter se quiser exibir seu valor na expressão (Exemplo de trabalho).

Exemplo #

Visão:

<div ng-app="myApp" ng-controller="MainCtrl">
    <input type="text" ng-model="foo" ng-model-options="{ getterSetter: true }">
    <div ng-if="truthyValue">
        <!-- I'm a child scope (inside ng-if), but i'm synced with changes from the outside scope -->
        <input type="text" ng-model="foo">
    </div>
    <div>$scope.foo: {{ foo() }}</div>
</div>

Controlador:

angular.module('myApp', []).controller('MainCtrl', ['$scope', function($scope) {
    $scope.truthyValue = true;
      
    var _foo = 'hello'; // this will be used to cache/represent the value of the 'foo' model 
      
    $scope.foo = function(val) {
        // the function return the the internal '_foo' varibale when called with zero arguments,
        // and update the internal `_foo` when called with an argument
        return arguments.length ? (_foo = val) : _foo;
    };
}]);

Prática recomendada: é melhor manter os getters rápidos porque o Angular provavelmente os chamará com mais frequência do que outras partes do seu código (reference).

7 pecados capitais do AngularJS #

Abaixo está a lista de alguns erros que os desenvolvedores costumam cometer durante o uso das funcionalidades do AngularJS, algumas lições aprendidas e soluções para eles.

1. Manipulando o DOM através do controlador

É legal, mas deve ser evitado. Os controladores são os locais onde você define suas dependências, vincula seus dados à exibição e cria mais lógica de negócios. Você pode manipular tecnicamente o DOM em um controlador, mas sempre que precisar de manipulação igual ou semelhante em outra parte do seu aplicativo, outro controlador será necessário. Portanto, a melhor prática dessa abordagem é criar uma diretiva que inclua todas as manipulações e use a diretiva em todo o seu aplicativo. Portanto, o controlador deixa a visão intacta e faz seu trabalho. Em uma diretiva, a função de vinculação é o melhor lugar para manipular o DOM. Ele tem acesso total ao escopo e ao elemento, portanto, usando uma diretiva, você também pode aproveitar a capacidade de reutilização.

link: function($scope, element, attrs) {
    //The best place to manipulate DOM
}

Você pode acessar elementos DOM na função de vinculação de várias maneiras, como o parâmetro element, método angular.element() ou Javascript puro.

2. Vinculação de dados na transclusão

O AngularJS é famoso por sua vinculação de dados bidirecional. No entanto, você pode encontrar, às vezes, que seus dados são vinculados apenas de maneira unidirecional dentro de diretivas. Pare aí, AngularJS não está errado, mas provavelmente você. As diretivas são lugares um pouco perigosos, pois estão envolvidos escopos filho e escopos isolados. Suponha que você tenha a seguinte diretiva com uma transclusão

<my-dir>
    <my-transclusion>
    </my-transclusion>
</my-dir>

E dentro de my-transclusion, você tem alguns elementos que estão vinculados aos dados no escopo externo.

<my-dir>
    <my-transclusion>
        <input ng-model="name">
    </my-transclusion>
</my-dir>

O código acima não funcionará corretamente. Aqui, a transclusão cria um escopo filho e você pode obter a variável de nome, certo, mas qualquer alteração que você fizer nessa variável permanecerá lá. Assim, você pode realmente acessar esta variável como $parent.name. No entanto, esse uso pode não ser a melhor prática. Uma abordagem melhor seria envolver as variáveis ​​dentro de um objeto. Por exemplo, no controlador você pode criar:

$scope.data = {
    name: 'someName'
}

Então na transclusão, você pode acessar essa variável via objeto ‘data’ e ver que a ligação bidirecional funciona perfeitamente!

<input ng-model="data.name">

Não apenas nas transclusões, mas em todo o aplicativo, é uma boa ideia usar a notação pontilhada.

3. Várias diretivas juntas

Na verdade, é legal usar duas diretivas juntas dentro do mesmo elemento, desde que você obedeça à regra: dois escopos isolados não podem existir no mesmo elemento. De um modo geral, ao criar uma nova diretiva personalizada, você aloca um escopo isolado para facilitar a passagem de parâmetros. Supondo que as diretivas myDirA e myDirB tenham escopos isolados e myDirC não, o seguinte elemento será válido:

<input my-dir-a my-dirc>

enquanto o seguinte elemento causará erro no console:

<input my-dir-a my-dir-b>

Portanto, as diretivas devem ser usadas com sabedoria, levando em consideração os escopos.

4. Uso indevido de $emit

$emit, $broadcast e $on, funcionam em um princípio emissor-receptor. Em outras palavras, eles são um meio de comunicação entre controladores. Por exemplo, a linha a seguir emite o ‘someEvent’ do controlador A, para ser capturado pelo controlador B em questão.

$scope.$emit('someEvent', args);

E a linha a seguir pega o ‘someEvent’

$scope.$on('someEvent', function(){});

Até agora tudo parece perfeito. Mas lembre-se de que, se o controlador B ainda não for invocado, o evento não será capturado, o que significa que os controladores emissor e receptor devem ser invocados para que isso funcione. Então, novamente, se você não tem certeza de que definitivamente precisa usar $emit, construir um serviço parece ser uma maneira melhor.

5. Uso indevido de $scope.$watch

$scope.$watch é usado para observar uma mudança de variável. Sempre que uma variável é alterada, esse método é invocado. No entanto, um erro comum cometido é alterar a variável dentro de $scope.$watch. Isso causará inconsistência e loop infinito de $digest em algum ponto.

$scope.$watch('myCtrl.myVariable', function(newVal) {
    this.myVariable++;
});

Portanto, na função acima, certifique-se de não ter operações em myVariable e newVal.

6. Métodos de associação a visualizações

Este é um dos pecados mais mortais. AngularJS tem ligação bidirecional e sempre que algo muda, as visualizações são atualizadas muitas vezes. Portanto, se você vincular um método a um atributo de uma exibição, esse método poderá ser chamado centenas de vezes, o que também o deixa louco durante a depuração. No entanto, existem apenas alguns atributos criados para ligação de método, como ng-click, ng-blur, ng-on-change, etc., que esperam métodos como parâmetro. Por exemplo, suponha que você tenha a seguinte visualização em sua marcação:

<input ng-disabled="myCtrl.isDisabled()" ng-model="myCtrl.name">

Aqui você verifica o status desabilitado da visualização através do método isDisabled. No controlador myCtrl, você tem:

vm.isDisabled = function(){
    if(someCondition)
        return true;
    else
        return false;
}

Em teoria, pode parecer correto, mas tecnicamente isso causará uma sobrecarga, pois o método será executado inúmeras vezes. Para resolver isso, você deve vincular uma variável. Em seu controlador, a seguinte variável deve existir:

vm.isDisabled

Você pode iniciar esta variável novamente na ativação do controlador

if(someCondition)
    vm.isDisabled = true
else
    vm.isDisabled = false

Se a condição não for estável, você pode vincular isso a outro evento. Então você deve vincular esta variável à visão:

<input ng-disabled="myCtrl.isDisabled" ng-model="myCtrl.name">

Agora, todos os atributos da visão têm o que esperam e os métodos serão executados apenas quando necessário.

7. Não usando as funcionalidades do Angular

O AngularJS oferece grande conveniência com algumas de suas funcionalidades, não apenas simplificando seu código, mas também tornando-o mais eficiente. Algumas dessas características estão listadas abaixo:

  1. angular.forEach for the loops (Caution, you can't "break;" it, you can only prevent getting into the body, so consider performance here.)
  2. angular.element for DOM selectors
  3. angular.copy: Use this when you should not modify the main object
  4. Form validations are already awesome. Use dirty, pristine, touched, valid, required and so on.
  5. Besides Chrome debugger, use remote debugging for mobile development too.
  6. And make sure you use Batarang. It's a free Chrome extension where you can easily inspect scopes
