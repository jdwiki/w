  • Executando no thread principal

Tópico relacionado: Grand Central Dispatch

Sintaxe #

  • dispatch_async - Executa um bloco de código em uma fila separada e não interrompe a fila atual. Se a fila estiver em um encadeamento diferente daquele que dispatch_async foi chamado, o código no bloco será executado enquanto o código após o dispatch_async também for executado
  • dispatch_sync - Executa um bloco de código em uma fila separada e não interrompe a fila atual. Se a fila estiver em um encadeamento diferente daquele em que dispatch_async foi chamado, o código no bloco será executado e a execução no encadeamento em que o método foi chamado só será retomada após o término

Parâmetros #

fila A fila na qual o código no bloco de despacho será executado. Uma fila é como (mas não exatamente igual a) um encadeamento; código em diferentes filas podem ser executados em paralelo. Use dispatch_get_main_queue para obter a fila para a thread principal Para criar uma nova fila, que por sua vez cria uma nova thread, use dispatch_queue_create("QUEUE_NAME", DISPATCH_QUEUE_CONCURRENT). O primeiro parâmetro é o nome da fila, que é exibido no depurador se você pausar enquanto o bloco ainda estiver em execução. O segundo parâmetro não importa, a menos que você queira usar a mesma fila para várias chamadas dispatch_async ou dispatch_sync. Descreve o que acontece quando outro bloco é colocado na mesma fila; DISPATCH_QUEUE_CONCURRENT fará com que ambos os blocos rodem ao mesmo tempo, enquanto DISPATCH_QUEUE_SERIAL fará com que o segundo bloco espere o primeiro bloco terminar
bloco O código neste bloco será executado na fila queue; coloque o código que você deseja executar na fila separada aqui. Uma dica útil: Se você estiver escrevendo isso no Xcode e o argumento do bloco tiver o contorno azul ao redor, clique duas vezes no argumento e o Xcode criará automaticamente um bloco vazio (isso se aplica a todos os argumentos de bloco em qualquer função ou método)

Sempre que você faz algo em um encadeamento separado, o que acontece ao usar filas, é importante manter a segurança do encadeamento. Alguns métodos, em particular aqueles para UIViews, podem não funcionar e/ou travar em threads que não sejam a thread principal. Além disso, certifique-se de não alterar nada (variáveis, propriedades, etc.) que também esteja sendo usado no thread principal, a menos que você esteja contabilizando essa alteração

Grupo de despacho - aguardando a conclusão de outros encadeamentos. #

dispatch_group_t preapreWaitingGroup = dispatch_group_create();

dispatch_group_enter(preapreWaitingGroup);
[self doAsynchronousTaskWithComplete:^(id someResults, NSError *error) { 
    // Notify that this task has been completed.
    dispatch_group_leave(preapreWaitingGroup);  
}]

dispatch_group_enter(preapreWaitingGroup);
[self doOtherAsynchronousTaskWithComplete:^(id someResults, NSError *error) { 
    dispatch_group_leave(preapreWaitingGroup);  
}]

dispatch_group_notify(preapreWaitingGroup, dispatch_get_main_queue(), ^{
    // This block will be executed once all above threads completed and call dispatch_group_leave
    NSLog(@"Prepare completed. I'm readyyyy");
});

Atualização 1. Versão Swift 3.

let prepareGroup = DispatchGroup()
prepareGroup.enter()
doAsynchronousTaskWithComplete() { (someResults, error) in
    // Notify that this task has been completed.
    prepareGroup.leave()
}

prepareGroup.enter()
doOtherAsynchronousTaskWithComplete() { (someResults, error) in
    // Notify that this task has been completed.
    prepareGroup.leave()
}

prepareGroup.notify(queue: DispatchQueue.main) {
    // This block will be executed once all above threads completed and call dispatch_group_leave
    print("Prepare completed. I'm readyyyy")
}

Executando código simultaneamente – Executando código enquanto executa outro código #

Digamos que você queira executar em ação (neste caso, registrando “Foo”), enquanto faz outra coisa (registrando “Bar”). Normalmente, se você não usar a simultaneidade, uma dessas ações será totalmente executada, e a outra execução será executada somente após o término completo. Mas com simultaneidade, você pode fazer as duas ações serem executadas ao mesmo tempo:

dispatch_async(dispatch_queue_create("Foo", DISPATCH_QUEUE_CONCURRENT), ^{
    for (int i = 0; i < 100; i++) {
        NSLog(@"Foo");
        usleep(100000);
    }
});

for (int i = 0; i < 100; i++) {
    NSLog(@"Bar");
    usleep(50000);
}

Isso registrará “Foo” 100 vezes, pausando por 100 ms cada vez que registrar, mas fará tudo isso em um thread separado. Enquanto Foo está sendo registrado, “Bar” também será registrado em intervalos de 50ms, ao mesmo tempo. Idealmente, você deve ver uma saída com “Foo"s e “Bars” misturados

Executando no thread principal #

Ao executar tarefas de forma assíncrona, geralmente se torna necessário garantir que um pedaço de código seja executado no thread principal. Por exemplo, você pode querer acessar uma API REST de forma assíncrona, mas colocar o resultado em um UILabel na tela. Antes de atualizar o UILabel, você deve garantir que seu código seja executado no thread principal:

dispatch_async(dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0), ^{
    //Perform expensive tasks
    //...

    //Now before updating the UI, ensure we are back on the main thread
    dispatch_async(dispatch_get_main_queue(), ^{
        label.text = //....
    });
}

Sempre que você atualizar exibições na tela, sempre certifique-se de estar fazendo isso no encadeamento principal, caso contrário, poderá ocorrer um comportamento indefinido.
