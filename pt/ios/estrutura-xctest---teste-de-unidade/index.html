Adicionando arquivos de teste ao projeto Xcode #

Ao criar o projeto #

Você deve marcar “Incluir Testes Unitários” na caixa de diálogo de criação do projeto.

Após criar o projeto #

Se você esqueceu de verificar esse item ao criar seu projeto, sempre poderá adicionar arquivos de teste posteriormente. Para fazer isso:

1- Vá para as configurações do seu projeto no Xcode

2- Vá em “Destinos”

3- Clique em “Adicionar destino”

4- Em “Outro”, selecione “Pacote de teste de unidade de toque de cacau”

No final, você deve ter um arquivo chamado [Nome do seu aplicativo]Tests.swift. Em Objective-C, você deve ter dois arquivos chamados [Your app name]Tests.h e [Your app name]Tests.m.

O arquivo [Nome do seu aplicativo]Tests.swift ou .m incluirá por padrão:

  • Uma importação de módulo XCTest
  • Uma classe [Your app name]Tests que estende XCTestCase
  • métodos setUp, tearDown, testExample, testPerformanceExample

    super.setUp()
    // Put setup code here. This method is called before the invocation of each test method in the class.
}

    // Put teardown code here. This method is called after the invocation of each test method in the class.
    super.tearDown()

    // This is an example of a functional test case.

    }

Adicionando métodos de teste #

De acordo com a Apple:

Métodos de teste #

Um método de teste é um método de instância de uma classe de teste que começa com o prefixo test, não recebe parâmetros e retorna void, por exemplo, (void)testColorIsRed(). Um método de teste exercita o código em seu projeto e, se esse código não produzir o resultado esperado, relata as falhas usando um conjunto de APIs de asserção. Por exemplo, o valor de retorno de uma função pode ser comparado com um valor esperado ou seu teste pode afirmar que o uso indevido de um método em uma de suas classes gera uma exceção.

Então adicionamos um método de teste usando “test” como prefixo do método, como:

Rápido #

func testSomething() {

}

Objetivo-C #

- (void)testSomething {

}

Para realmente testar os resultados, usamos o método XCTAssert(), que recebe uma expressão booleana e, se true, marca o teste como bem-sucedido, caso contrário, ele o marcará como falhado.

Digamos que temos um método na classe View Controller chamado sum() que calcula a soma de dois números. Para testá-lo, usamos este método:

Rápido #

func testSum(){
    let result = viewController.sum(4, and: 5)
    XCTAssertEqual(result, 9)
}

Objetivo-C #

- (void)testSum {
    int result = [viewController sum:4 and:5];
    XCTAssertEqual(result, 9);
}

Nota #

Por padrão, você não pode acessar rótulo, caixa de texto ou outros itens de interface do usuário da classe View Controller da classe de teste se eles forem criados primeiro no arquivo Storyboard. Isso ocorre porque eles são inicializados no método loadView() da classe View Controller, e isso não será chamado durante o teste. A melhor maneira de chamar loadView() e todos os outros métodos necessários é acessando a propriedade view de nossa propriedade viewController. Você deve adicionar esta linha antes de testar os elementos da interface do usuário:

XCTAssertNotNil(viewController.view)

Escrevendo uma classe de teste #

import XCTest
@testable import PersonApp

class PersonTests: XCTestCase {
    func test_completeName() {
        let person = Person(firstName: "Josh", lastName: "Brown")
        XCTAssertEqual(person.completeName(), "Josh Brown")
    }
}

Agora vamos discutir o que está acontecendo aqui. A linha import XCTest nos permitirá estender XCTestCase e ​​usar XCTAssertEqual (entre outras asserções). Estender XCTestCase e ​​prefixar nosso nome de teste com test garantirá que o Xcode execute este teste automaticamente ao executar os testes no projeto (⌘U ou Product > Test). A linha @testable import PersonApp importará nosso destino PersonApp para que possamos testar e usar classes dele, como a Person em nosso exemplo acima. E finalmente, nosso XCTAssertEqual garantirá que person.completeName() seja igual à string "Josh Brown".

Adicionando Storyboard e View Controller como instâncias para testar o arquivo #

Para começar com o teste de unidade, que será feito no arquivo de testes e testará o View Controller e o Storyboard, devemos introduzir esses dois arquivos no arquivo de teste.

Definindo o View Controller #

Rápido #

var viewController : ViewController!

Apresentando o Storyboard e inicializando o View Controller #

Adicione este código ao método setUp():

Rápido #

let storyboard = UIStoryboard(name: "Main", bundle: nil)
viewController = storyboard.instantiateInitialViewController() as! ViewController

Objetivo-C #

UIStoryboard *storyboard = [UIStoryboard storyboardWithName:"Main" bundle:nil];
viewController = (ViewController *) [storyboard instantiateInitialViewController];

Dessa forma, você pode escrever métodos de teste e eles saberão onde verificar se há erros. Nesse caso, há o View Controller e o Storyboard.

Comece a testar #

Testando um método específico #

Para testar um método específico, clique no quadrado ao lado da definição do método.

Testando todos os métodos #

Para testar todos os métodos, clique no quadrado ao lado da definição da classe.

Veja o resultado do teste #

Se houver uma verificação verde ao lado da definição, o teste foi bem-sucedido.

Se houver uma cruz vermelha ao lado da definição, o teste falhou.

Executando todos os testes #

Product -> Test OR Cmd + U

Ele executará todos os testes de todos os destinos de teste!

Importe um módulo que possa ser testado #

Classes, structs, enums e todos os seus métodos são internos por padrão. Isso significa que eles só podem ser acessados ​​a partir do mesmo módulo. Os casos de teste estão em um destino diferente e isso significa que estão em um módulo diferente. Para poder acessar o método que você deseja testar, você precisa importar o módulo a ser testado usando a palavra-chave @testable.

Digamos que temos um módulo principal chamado ToDo e queremos escrever testes para ele. Importaríamos esse módulo assim:

@testable import ToDo

Todos os métodos de teste no arquivo com esta instrução de importação agora podem acessar todas as classes internas, structs, enums e todos os seus métodos internos do módulo ToDo.

Você nunca deve adicionar os arquivos com os elementos que deseja testar ao destino de teste porque isso pode levar a erros difíceis de depurar.

Acionar o carregamento e a aparência da visualização #

Visualizar carregando #

Em um teste para um controlador de exibição, às vezes você deseja acionar a execução de loadView() ou viewDidLoad(). Isso pode ser feito acessando a visualização. Digamos que você tenha uma instância do controlador de visualização em seu teste chamada sut (sistema em teste), então o código ficaria assim:

XCTAssertNotNil(sut.view)

Veja a aparência #

Você também pode acionar os métodos viewWillAppear(_:) e viewDidAppear(_:) adicionando o seguinte código:

sut.beginAppearanceTransition(true, animated: true)
sut.endAppearanceTransition()
