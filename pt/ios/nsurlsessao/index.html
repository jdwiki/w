NSURLSessão

  • Objective-C Criar uma sessão e tarefa de dados
  • Configurando a configuração em segundo plano
  • Solicitação GET simples
  • Enviando uma solicitação POST com argumentos usando NSURLSession em Objective-C

Nesta página

  • Objective-C Criar uma sessão e tarefa de dados
  • Configurando a configuração em segundo plano
  • Solicitação GET simples
  • Enviando uma solicitação POST com argumentos usando NSURLSession em Objective-C

A classe NSURLSession e as classes relacionadas fornecem uma API para download de conteúdo. Essa API fornece um conjunto avançado de métodos delegados para dar suporte à autenticação e oferece ao seu aplicativo a capacidade de realizar downloads em segundo plano quando seu aplicativo não está em execução ou, no iOS, enquanto seu aplicativo está suspenso.

Em alto nível, NSURLSession é baseado no conceito de sessões e tarefas. Uma tarefa representa uma única solicitação para um único URL (ou um único upload para um único URL). Uma sessão é um grupo de solicitações relacionadas.

O sistema operacional fornece uma única sessão preexistente — a sessão compartilhada, que funciona basicamente como NSURLConnection. Além disso, você pode criar suas próprias sessões em seu aplicativo conforme necessário.

Diferentes aplicativos usam sessões de maneiras diferentes. Muitos aplicativos criam uma única sessão na inicialização e continuam a reutilizar. Outros aplicativos se beneficiam de poder cancelar um grupo de tarefas relacionadas (por exemplo, um navegador da Web cancelando todas as solicitações pendentes quando você fecha uma guia) e, assim, criar uma sessão para manter cada grupo de solicitações relacionadas.

A primeira etapa ao usar NSURLSession é criar um objeto de configuração de sessão. O objeto (geralmente) reutilizável contém várias configurações de sessão que você pode ajustar para suas necessidades específicas, como simultaneidade máxima, cabeçalhos extras para enviar com cada solicitação, permitir que solicitações sejam enviadas pelo rádio celular (somente iOS), tempos limite, armazenamento de credenciais, versão mínima de TLS e até configurações de proxy.

Existem três tipos de configurações de sessão, dependendo de como você deseja que a sessão resultante se comporte:

  • Configurações padrão criam sessões que funcionam como NSURLConnection.
  • Configurações em segundo plano criam sessões nas quais as solicitações acontecem fora do processo, permitindo que os downloads continuem mesmo quando o aplicativo não estiver mais em execução.
  • Configurações efêmeras criam sessões que não armazenam nada em cache no disco, não armazenam cookies no disco etc. e, portanto, são adequadas para fazer backup de coisas como janelas anônimas do navegador.

Ao criar uma configuração em segundo plano, você deve fornecer um identificador de sessão que permita reassociar a sessão em segundo plano posteriormente (se seu aplicativo sair ou for suspenso ou encerrado pelo sistema operacional). Você não deve ter mais de uma instância de uma sessão com o mesmo identificador ativo em seu aplicativo, portanto, via de regra, essas configurações não são reutilizáveis. Todas as outras configurações de sessão podem ser reutilizadas para criar quantas sessões você desejar. Portanto, se você precisar criar várias sessões com configurações semelhantes, poderá criar a configuração uma vez e reutilizá-la sempre que criar uma nova sessão.

Depois de criar uma sessão, você pode criar tarefas nessa sessão. Existem três tipos de tarefas:

  • Tarefas de dados retornam dados como um objeto NSData. Eles são adequados para uso geral, mas não são suportados em sessões em segundo plano.
  • Tarefas de download retornam dados como um arquivo no disco. Eles são adequados para solicitações maiores ou para uso em sessões em segundo plano.
  • Carregar tarefas carregar dados de um objeto NSData ou de um arquivo em disco. Você fornece um objeto de dados ou arquivo que fornece o corpo POST. Os dados/arquivo do corpo que você fornece na tarefa substituem qualquer dado/arquivo do corpo fornecido no objeto NSURLRequest (se aplicável).

Cada um desses tipos permite obter os dados de resposta de duas maneiras diferentes — usando retornos de chamada baseados em blocos ou fornecendo um delegado na sessão e implementando métodos delegados.

Além disso, NSURLSession permite que você forneça métodos delegados para manipular a autenticação, executar o manuseio de certificados TLS personalizados (para certificados de cliente e validação de servidor), alterar o comportamento de cache e assim por diante.

Configurando a configuração em segundo plano #

Para criar uma sessão em segundo plano

 // Swift:
 let mySessionID = "com.example.bgSession"
 let bgSessionConfig = NSURLSessionConfiguration.backgroundSessionConfigurationWithIdentifier(mySessionID)
 
 let session = NSURLSession(configuration: bgSessionConfig)
 
 // add tasks here


 // Objective-C:
 NSString *mySessionID = @"com.example.bgSession";
 NSURLSessionConfiguration *configuration =
     [NSURLSessionConfiguration backgroundSessionConfigurationWithIdentifier: mySessionID];
 NSURLSession *session = [NSURLSession sessionWithConfiguration:configuration
                                                       delegate:self]

Além disso, no iOS, você deve configurar o suporte para lidar com a reinicialização do aplicativo em segundo plano. Quando o método application:handleEventsForBackgroundURLSession:completionHandler: do seu aplicativo (Objective-C) ou application(_:handleEventsForBackgroundURLSession:completionHandler:) (Swift) é chamado, significa que seu aplicativo foi reiniciado em segundo plano para lidar com a atividade em uma sessão.

Nesse método, você deve criar uma nova sessão com o identificador fornecido e configurá-la com um delegado para manipular eventos como faria normalmente em primeiro plano. Além disso, você deve armazenar o manipulador de conclusão fornecido em um dicionário, usando a sessão como a chave.

Quando o método URLSessionDidFinishEventsForBackgroundURLSession: (Obj-C) / URLSessionDidFinishEventsForBackgroundURLSession (Swift) do delegado é chamado para informar que não há mais eventos para manipular, seu aplicativo deve procurar o manipulador de conclusão para essa sessão, remover a sessão de o dicionário e chame o manipulador de conclusão, informando assim ao sistema operacional que você não tem mais nenhum processamento pendente relacionado à sessão. (Se você ainda estiver fazendo algo por algum motivo quando receber essa chamada de delegado, espere até terminar.) Assim que você chamar esse método, a sessão em segundo plano será imediatamente invalidada.

Se seu aplicativo receber uma chamada application:application:didFinishLaunchingWithOptions: (provavelmente indicando que o usuário colocou seu aplicativo em primeiro plano enquanto você estava ocupado processando eventos em segundo plano), é seguro criar uma sessão em segundo plano com esse mesmo identificador, porque a sessão antiga com esse identificador não existe mais.

Se você está curioso sobre os detalhes, em alto nível, ao criar uma sessão em segundo plano, você está fazendo duas coisas:

  • Criando uma sessão em um daemon externo (nsurlsessiond) para lidar com os downloads
  • Criando uma sessão em seu aplicativo que fala com esse daemon externo via NSXPC

Normalmente, é perigoso criar duas sessões com o mesmo ID de sessão em uma única inicialização do aplicativo, porque ambas estão tentando se comunicar com a mesma sessão no daemon em segundo plano. É por isso que a documentação oficial diz para nunca criar várias sessões com o mesmo identificador. No entanto, se a primeira sessão foi uma sessão temporária criada como parte de uma chamada handleEventsForBackgroundURLSession, a associação entre a sessão no aplicativo agora inválida e a sessão no daemon em segundo plano não existe mais.

Enviando uma solicitação POST com argumentos usando NSURLSession em Objective-C #

Existem duas maneiras comuns de codificar um corpo de solicitação POST: codificação de URL (application/x-www-form-urlencoded) e dados de formulário (multipart/form-data). Grande parte do código é semelhante, mas a maneira como você constrói os dados do corpo é diferente.

Enviando uma solicitação usando codificação de URL

Seja você ter um servidor para seu pequeno aplicativo ou trabalhar em uma equipe com um engenheiro de back-end completo, você desejará falar com esse servidor em um ponto com seu aplicativo iOS.

No código a seguir estaremos compondo uma string de argumentos que o script do servidor de destino usará para fazer algo que muda dependendo do seu caso. Por exemplo, podemos querer enviar a string:

nome=Brendon&senha=abcde

Para o servidor quando um usuário se inscreve em seu aplicativo, para que o servidor possa armazenar essas informações em um banco de dados.

Vamos começar. Você desejará criar uma solicitação POST NSURLSession com o código a seguir.

// Create the configuration, which is necessary so we can cancel cacheing amongst other things.
NSURLSessionConfiguration * defaultConfigObject = [NSURLSessionConfiguration defaultSessionConfiguration];
// Disables cacheing
defaultConfigObject.requestCachePolicy = NSURLRequestReloadIgnoringLocalCacheData;
NSURLSession * defaultSession = [NSURLSession sessionWithConfiguration:defaultConfigObject delegate:self delegateQueue:[NSOperationQueue mainQueue]];

NSString * scriptURL = [NSString stringWithFormat:@"https://server.io/api/script.php"];
//Converts the URL string to a URL usable by NSURLSession
NSMutableURLRequest * urlRequest = [NSMutableURLRequest requestWithURL:[NSURL URLWithString:scriptURL]];
NSString * postDataString = [NSString stringWithFormat:@"name=%@&password=%@", [self nameString], [self URLEncode:passwordString]];
[urlRequest setHTTPMethod:@"POST"];
[urlRequest setHTTPBody:[postDataString dataUsingEncoding:NSUTF8StringEncoding]];

NSURLSessionDataTask * dataTask = [defaultSession dataTaskWithRequest:urlRequest];
// Fire the data task.
[dataTask resume];

O código acima acabou de criar e disparar a solicitação POST para o servidor. Lembre-se de que a URL do script e a string de dados POST mudam dependendo da sua situação. Se você estiver lendo isso, saberá com o que preencher essas variáveis.

Você também precisará adicionar um pequeno método que faça a codificação do URL:

- (NSString *)URLEncode:(NSString *)originalString encoding:(NSStringEncoding)encoding
{
    return (__bridge_transfer NSString *)CFURLCreateStringByAddingPercentEscapes(
        kCFAllocatorDefault,
        (__bridge CFStringRef)originalString,
        NULL,
        CFSTR(":/?#[]@!$&'()*+,;="),
        CFStringConvertNSStringEncodingToEncoding(encoding));
}

Portanto, quando o servidor terminar de processar esses dados, ele enviará um retorno ao seu aplicativo iOS. Então precisamos processar esse retorno, mas como?

Usamos programação orientada a eventos e usamos os métodos delegados do NSURLSession. Isso significa que, à medida que o servidor enviar uma resposta, esses métodos começarão a ser acionados. Os 5 métodos a seguir são os que serão acionados durante toda a solicitação INTEIRA, cada vez que uma for feita:

- (void)URLSession:(NSURLSession *)session dataTask:(NSURLSessionDataTask *)dataTask didReceiveResponse:(NSURLResponse *)response
 completionHandler:(void (^)(NSURLSessionResponseDisposition disposition))completionHandler;

- (void)URLSession:(NSURLSession *)session dataTask:(NSURLSessionDataTask *)dataTask didReceiveData:(NSData *)data;

- (void)URLSession:(NSURLSession *)session task:(NSURLSessionTask *)task didCompleteWithError:(NSError *)error;

- (void)URLSession:(NSURLSession *)session didReceiveChallenge:(NSURLAuthenticationChallenge *)challenge completionHandler:(void (^)(NSURLSessionAuthChallengeDisposition, NSURLCredential *))completionHandler;

- (void)URLSession:(NSURLSession *)session task:(NSURLSessionTask *)task didReceiveChallenge:(NSURLAuthenticationChallenge *)challenge completionHandler:(void (^)(NSURLSessionAuthChallengeDisposition, NSURLCredential * _Nullable))completionHandler;

Abaixo você verá os métodos acima usados ​​no contexto. Cada um de seus propósitos é bastante autoexplicativo graças à Apple, mas comentei seus usos de qualquer maneira:

// Response handling delegates

// The following is not necessary unless your server is using HTTP, not HTTPS

- (void)URLSession:(NSURLSession *)session didReceiveChallenge:(NSURLAuthenticationChallenge *)challenge completionHandler:(void (^)(NSURLSessionAuthChallengeDisposition, NSURLCredential *))completionHandler{
    if([challenge.protectionSpace.authenticationMethod isEqualToString:NSURLAuthenticationMethodServerTrust]){
        if([challenge.protectionSpace.host isEqualToString:@"DomainNameOfServer.io"]){
            NSURLCredential * credential = [NSURLCredential credentialForTrust:challenge.protectionSpace.serverTrust];
            completionHandler(NSURLSessionAuthChallengeUseCredential,credential);
        }
    }
}

Enviando uma solicitação usando a codificação de formulário

A codificação de URL é uma maneira amplamente compatível de codificar dados arbitrários. No entanto, é relativamente ineficiente para carregar dados binários (como fotos) porque cada byte não ASCII se transforma em um código de três caracteres. Ele também não suporta anexos de arquivos, então você teria que passar nomes de arquivos e dados de arquivos como campos separados.

Suponha que queremos fazer upload de uma fotografia de uma forma que seja eficiente e realmente pareça um arquivo no lado do servidor. Uma maneira de fazer isso é usar a codificação de formulário. Para fazer isso, edite o código que cria a NSURLSession da seguinte forma:

UIImage * imgToSend;

// 2nd parameter of UIImageJPEGRepresentation represents compression quality. 0 being most compressed, 1 being the least
// Using 0.4 likely stops us hitting the servers upload limit and costs us less server space
NSData * imageData = UIImageJPEGRepresentation(imgToSend, 0.4f);

// Alternatively, if the photo is on disk, you can retrieve it with
// [NSData dataWithContentsOfURL:...]

// Set up the body of the POST request.

// This boundary serves as a separator between one form field and the next.
// It must not appear anywhere within the actual data that you intend to
// upload.
NSString * boundary = @"---------------------------14737809831466499882746641449";

// Body of the POST method
NSMutableData * body = [NSMutableData data];

// The body must start with the boundary preceded by two hyphens, followed
// by a carriage return and newline pair.
//
// Notice that we prepend two additional hyphens to the boundary when
// we actually use it as part of the body data.
//
[body appendData:[[NSString stringWithFormat:@"\r\n--%@\r\n",boundary] dataUsingEncoding:NSUTF8StringEncoding]];

// This is followed by a series of headers for the first field and then
// TWO CR-LF pairs.
[body appendData:[[NSString stringWithFormat:@"Content-Disposition: form-data; name=\"tag_name\"\r\n\r\n"] dataUsingEncoding:NSUTF8StringEncoding]];

// Next is the actual data for that field (called "tag_name") followed by
// a CR-LF pair, a boundary, and another CR-LF pair.
[body appendData:[strippedCompanyName dataUsingEncoding:NSUTF8StringEncoding]];
[body appendData:[[NSString stringWithFormat:@"\r\n--%@\r\n", boundary] dataUsingEncoding:NSUTF8StringEncoding]];

// Encode the filename and image data as the "userfile" CGI parameter.
// This is similar to the previous field, except that it is being sent
// as an actual file attachment rather than a blob of data, which means
// it has both a filename and the actual file contents.
//
// IMPORTANT: The filename MUST be plain ASCII (and if encoded like this,
//            must not include quotation marks in the filename).
//
NSString * picFileName = [NSString stringWithFormat:@"photoName"];
NSString * appendDataString = [NSString stringWithFormat:@"Content-Disposition: form-data; name=\"userfile\"; filename=\"%@.jpg\"\r\n", picFileName];
[body appendData:[appendDataString dataUsingEncoding:NSUTF8StringEncoding]];
[body appendData:[@"Content-Type: application/octet-stream\r\n\r\n" dataUsingEncoding:NSUTF8StringEncoding]];
[body appendData:[NSData dataWithData:imageData]];

// Close the request body with one last boundary with two
// additional hyphens prepended **and** two additional hyphens appended.
[body appendData:[[NSString stringWithFormat:@"\r\n--%@--\r\n", boundary] dataUsingEncoding:NSUTF8StringEncoding]];

// Create the session
// We can use the delegate to track upload progress and disable cacheing
NSURLSessionConfiguration * defaultConfigObject = [NSURLSessionConfiguration defaultSessionConfiguration];
defaultConfigObject.requestCachePolicy = NSURLRequestReloadIgnoringLocalCacheData;
NSURLSession * defaultSession = [NSURLSession sessionWithConfiguration: defaultConfigObject delegate: self delegateQueue: [NSOperationQueue mainQueue]];

// Data uploading task.
NSURL * url = [NSURL URLWithString:@"https://server.io/api/script.php"];
NSMutableURLRequest * request = [NSMutableURLRequest requestWithURL:url];
NSString * contentType = [NSString stringWithFormat:@"multipart/form-data; boundary=%@",boundary];
[request addValue:contentType forHTTPHeaderField:@"Content-Type"];
request.HTTPMethod = @"POST";
request.HTTPBody = body;
NSURLSessionDataTask * uploadTask = [defaultSession dataTaskWithRequest:request];
[uploadTask resume];

Isso cria e dispara a solicitação NSURLSession exatamente como antes e, como resultado, os métodos delegados se comportarão exatamente da mesma maneira. Certifique-se de que o script para o qual a imagem está sendo enviada (localizado na url na variável url) está esperando uma imagem e pode analisá-la corretamente.
