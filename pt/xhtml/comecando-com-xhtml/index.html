Instalação ou configuração #

Relação XHTML com HTML

XHTML é simplesmente a versão serializada do HTML. O XHTML foi originalmente planejado para limpar a marcação HTML para melhorar o suporte aos padrões. Infelizmente, o trabalho do W3C em XHTML 2.0 não foi intuitivo e prejudicou o rumo da indústria como um todo com o HTML5. XHTML5 não é um padrão, embora você possa usar XHTML5 (XML serializado HTML5) com pequenas modificações. Exemplos são fornecidos abaixo para facilitar a adoção de XHTML5 e esclarecer ambiguidades no padrão HTML5 onde XML não é levado em consideração.

Benefícios XHTML

Como o XHTML usa o analisador XML de um navegador, é significativamente (embora não absolutamente) menos propenso a erros comuns de programação. Se um desenvolvedor carregar um aplicativo XHTML com XML malformado, a maioria dos mecanismos de renderização do navegador renderizará até o XML malformado, enquanto o mecanismo de renderização Gecko (usado no Firefox) mostrará uma página amarela. Em todos os casos, uma mensagem de erro será exibida com os números de linha e coluna de onde o primeiro erro de análise XML encontrado foi encontrado. Muitos desenvolvedores cometeram erros simples, como perder uma citação de um atributo HTML, deixando-os gastando dias tentando determinar por que um ou dois navegadores não estão renderizando a página HTML como esperado, usando XHTML pode diminuir drasticamente o tempo de desenvolvimento em relação ao HTML. XHTML pode ter um reforço positivo em aprender a codificar HTML válido para pessoas que estão aprendendo desenvolvimento web, pois não permite que o aplicativo XHTML seja renderizado na íntegra com código malformado e dando uma mensagem de erro explícita que novos desenvolvedores podem procurar online para trabalhar soluções. Como o XHTML é um subconjunto do XML, ele tem uma compatibilidade muito alta com softwares que trabalham com XML em uma ampla gama de usos industriais, comerciais e residenciais. Por fim, devido aos seus requisitos rigorosos, o código XHTML de trabalho é automaticamente compatível com HTML e XML (sujeito à ausência de vários valores de atributo id), enquanto o HTML não é inerentemente compatível com outros corpos de HTML e XML, pois não é imediatamente aparente que o código HTML pode estar malformado.

Desvantagens do XHTML

Devido às regras muito mais rígidas do analisador XML, o XHTML não é tão fácil, inicialmente, para desenvolvedores menos experientes. O XHTML não é levado em consideração com frequência por vários grupos, contém ambiguidade em relação a partes mal escritas do padrão HTML5 e não há um validador explícito que declare suporte para XHTML5. XHTML é menos compatível com partes “mais preguiçosas” de JavaScript, como innerHTML, que não serializa corretamente novas partes do DOM, no entanto, isso é mais um benefício para desenvolvedores dedicados que aprendem XHTML, pois requer um código mais estrito e intercambiável.

Usando XHTML

XHTML é a combinação de HTML e o uso do analisador XML que é uma versão muito mais estrita do analisador HTML; todos os navegadores modernos têm analisadores HTML para HTML e analisadores XML para XML (e posteriormente XHTML). XHTML não requer uma instalação de software (além de usar qualquer navegador moderno), no entanto, um aplicativo (X)HTML só é XHTML quando é corretamente servido ao navegador com um cabeçalho enviado pelo servidor declarando o application/xhtml+xml tipo mime; você pode verificar se seu aplicativo XHTML foi realmente servido como este, visualizando o tipo/mime/tipo de mídia no painel de solicitações de rede das ferramentas de desenvolvedor da Web em seu navegador. Uma página servida como text/html não é XHTML e será analisada pelo analisador HTML do navegador.

Carregado de um servidor

Ao carregar um analisador XHTML do servidor, um cabeçalho deve ser definido, usando um elemento meta não serve a nenhum propósito útil. No exemplo em combinação com PHP:

<?php
if (isset($_SERVER['HTTP_ACCEPT']) && stristr($_SERVER['HTTP_ACCEPT'],'application/xhtml+xml'))
{
 // Client header isset` and explicitly declares XHTML parser support.
 header('Content-Type: application/xhtml+xml; charset=UTF-8');
 echo '<?xml version="1.0" encoding="UTF-8"?>'."\n";
 echo '<!DOCTYPE html>'."\n";
 echo '<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">'."\n";
}
else
{
 //Some browsers do not declare support, IE9 shamelessly uses `*/*` in example.
 echo '<!DOCTYPE html>'."\n";
 echo '<html>'."\n";
}
?>

Carregado de um arquivo

Se você estiver testando sem carregar uma página de um caminho de servidor (por exemplo, localhost, 127.0.0.1, 192.168.xxx.yyy, etc), a única maneira de fazer um navegador carregar um aplicativo XHTML e usar o analisador XML é fornecer o arquivar uma extensão .xhtml; exemplo.xhtml.

Suporte ao navegador do analisador XHTML/XML

Os navegadores com suporte a XHTML via analisadores XML incluem Internet Explorer 9+ (o suporte ao analisador XML em versões mais antigas suporta application/xml de maneira muito complicada), Mozilla Suite 0.8+ (todas as versões do Firefox), Opera 7+ e versões anteriores do KHTML (Konqueror e, portanto, todas as versões do Safari e, por extensão, Chromium/Chrome).

{
 // Client header isset` and explicitly declares XHTML parser support.

 echo '<?xml version="1.0" encoding="UTF-8"?>'."\n";
 echo '<!DOCTYPE html>'."\n";
 echo '<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">'."\n";
}
else
{
 //Some browsers do not declare support, IE9 shamelessly uses `*/*` in example.
 echo '<!DOCTYPE html>'."\n";
 echo '<html>'."\n";
}
?>
<head>
<style type="text/css">
* {border: 0; margin: 0; padding: 0;}
</style>
<script type="application/javascript">
//<![CDATA[

</head>

Adicionando XML a um aplicativo XHTML e recuperando XML de um aplicativo XHTML #

Usando XHTML você deve evitar métodos como document.write e innerHTML pois eles tratam XML como texto e não serializam o código corretamente; um atributo id em HTML é essencialmente despejado no DOM e o atributo id não é serializado, o que significa que ao tentar fazer referência a ele com algo como document.getElementById('example') o navegador irá não “ver” o id. Os exemplos a seguir obtêm “obter” código XHTML de e “definir” código XHTML para o aplicativo XHTML.

Adicionando XHTML a e recuperando XML do DOM

<?php
if (isset($_SERVER['HTTP_ACCEPT']) && stristr($_SERVER['HTTP_ACCEPT'],'application/xhtml+xml'))
{
 // Client header isset` and explicitly declares XHTML parser support.
 header('Content-Type: application/xhtml+xml; charset=UTF-8');
 echo '<?xml version="1.0" encoding="UTF-8"?>'."\n";
 echo '<!DOCTYPE html>'."\n";
 echo '<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">'."\n";
}
else
{
 //Some browsers do not declare support, IE9 shamelessly uses `*/*` in example.
 echo '<!DOCTYPE html>'."\n";
 echo '<html>'."\n";
}
?>
<head>
<style type="text/css">
* {border: 0; margin: 0; padding: 0;}
</style>
<script type="application/javascript">
//<![CDATA[
function xml_get(target)
{
 return new XMLSerializer().serializeToString(target)
}

function xml_set(xml,position,target)
{
 if (typeof target=='string' && document.getElementById(target)) {target = document.getElementById(target);}

 if (!target) {alert('Error: target element was not found in the DOM.');}
 else if (position=='after')
 {
  if (target.nextSibling && target.nextSibling!='[object Text]') {target.insertBefore(xml.getElementsByTagName('*')[0],target.nextSibling);}
  else {target.parentNode.appendChild(xml.getElementsByTagName('*')[0]);}
 }
 else if (position=='before') {target.parentNode.insertBefore(document.importNode(xml.getElementsByTagName('*')[0],true),target);}
 else if (position=='inside') {target.appendChild(document.importNode(xml.getElementsByTagName('*')[0],true));}
 else if (position=='replace') {target.parentNode.replaceChild(document.importNode(xml.getElementsByTagName('*')[0],true),target);}
 else {alert('Error: unknown position to import data to: '+position+'.');}
}

window.onload = function(event)
{
 var xml_after = new DOMParser().parseFromString('<p xmlns="http://www.w3.org/1999/xhtml">XML string for <em>after</em> the h1[0] element!</p>','application/xml');
 var xml_before = new DOMParser().parseFromString('<p xmlns="http://www.w3.org/1999/xhtml">XML string for <em>before</em> the h1[0] element!</p>','application/xml');
 var xml_inside = new DOMParser().parseFromString('<p xmlns="http://www.w3.org/1999/xhtml">XML string for <em>inside</em> inside the element with the id <code>example</code>!</p>','application/xml');
 var xml_replace = new DOMParser().parseFromString('<p xmlns="http://www.w3.org/1999/xhtml">XML string for <em>replace</em> example!</p>','application/xml');
 xml_set(xml_after,'after',document.getElementsByTagName('h1')[0]);
 xml_set(xml_before,'before',document.getElementsByTagName('h1')[0]);
 xml_set(xml_inside,'inside','example');
 xml_set(xml_replace,'replace','example_replace');

 alert(xml_get(document));
}
//]]>
</script>
</head>

<body>

<h1><span>XHTML and JavaScript Simple Demonstration</span></h1>
<div id="example"></div>
<div id="example_replace"></div>

</body>
</html>

Atributos XHTML5 e Booleanos #

HTML5 define alguns atributos HTML como booleanos; um booleano só pode ser true ou false. A especificação simplesmente declara que a presença de um atributo booleano implica que o atributo é definido como verdadeiro. No exemplo, o uso de um atributo disabled no exemplo a seguir desativa o elemento de entrada do botão:

<input disabled type="button" value="HTML Button">

XML e, portanto, XHTML por extensão, deve ter um atributo e valor válidos. Como o HTML5 não é escrito de forma a esclarecer essas coisas (ambiguidade nos padrões anteriores levou a diferentes implementações do navegador), os atributos HTML5 quando servidos em um aplicativo XHTML devem sempre usar um valor true, pelo menos até que uma especificação futura remova o desnecessário ambiguidade.

<input disabled="true" type="button" value="XHTML Button" />
