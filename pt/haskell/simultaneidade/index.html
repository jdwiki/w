Gerando Threads com forkIO #

Haskell suporta muitas formas de simultaneidade e a mais óbvia é bifurcar uma thread usando forkIO.

A função forkIO :: IO () -> IO ThreadId pega uma ação IO e retorna seu ThreadId, enquanto a ação será executada em segundo plano.

Podemos demonstrar isso sucintamente usando ghci:

Prelude Control.Concurrent> forkIO $ (print . sum) [1..100000000]
ThreadId 290
Prelude Control.Concurrent> forkIO $ print "hi!"
"hi!"
-- some time later....
Prelude Control.Concurrent> 50000005000000

Ambas as ações serão executadas em segundo plano, e a segunda é quase garantida para terminar antes da última!

Comunicação entre Threads com MVar #

É muito fácil passar informações entre threads usando o tipo MVar a e suas funções acompanhantes em Control.Concurrent:

  • newEmptyMVar :: IO (MVar a) – cria um novo MVar a
  • newMVar :: a -> IO (MVar a) – cria um novo MVar com o valor fornecido
  • takeMVar :: MVar a -> IO a – recupera o valor do MVar fornecido, ou blocos até que um esteja disponível
  • putMVar :: MVar a -> a -> IO() – coloca o valor dado no MVar, ou blocos até que esteja vazio

Vamos somar os números de 1 a 100 milhões em uma thread e aguardar o resultado:

import Control.Concurrent
main = do
  m <- newEmptyMVar
  forkIO $ putMVar m $ sum [1..10000000]
  print =<< takeMVar m  -- takeMVar will block 'til m is non-empty!

Uma demonstração mais complexa pode ser obter a entrada do usuário e somar em segundo plano enquanto aguarda mais entradas:

main2 = loop
  where 
    loop = do
        m <- newEmptyMVar
        n <- getLine
        putStrLn "Calculating. Please wait"
        -- In another thread, parse the user input and sum
        forkIO $ putMVar m $ sum [1..(read n :: Int)]
        -- In another thread, wait 'til the sum's complete then print it
        forkIO $ print =<< takeMVar m
        loop

Como dito anteriormente, se você chamar takeMVar e o MVar estiver vazio, ele bloqueia até que outro thread coloque algo no MVar, o que pode resultar em um Problema dos Filósofos de Jantar. A mesma coisa acontece com putMVar: se estiver cheio, bloqueará até ficar vazio!

Pegue a seguinte função:

concurrent ma mb = do
  a <- takeMVar ma
  b <- takeMVar mb
  putMVar ma a
  putMVar mb b

Executamos as duas funções com alguns MVars

concurrent ma mb     -- new thread 1 
concurrent mb ma     -- new thread 2

O que pode acontecer é que:

  1. O thread 1 lê ma e bloqueia ma
  2. Thread 2 lê mb e, portanto, bloqueia mb

Agora o Thread 1 não pode ler mb pois o Thread 2 o bloqueou, e o Thread 2 não pode ler ma pois o Thread 1 o bloqueou. Um clássico impasse!

Blocos atômicos com memória transacional de software #

Outra ferramenta de simultaneidade poderosa e madura em Haskell é o Software Transactional Memory, que permite que vários threads escrevam em uma única variável do tipo TVar a de maneira atômica.

TVar a é o tipo principal associado à mônada STM e significa variável transacional. Eles são usados ​​como MVar mas dentro da mônada STM através das seguintes funções:

atomicamente :: STM a -> IO a #

Execute uma série de ações STM atomicamente.

readTVar :: TVar a -> STM a #

Leia o valor do TVar, por exemplo:

value <- readTVar t

writeTVar :: TVar a -> a -> STM () #

Escreva um valor para o TVar fornecido.

t <- newTVar Nothing
writeTVar t (Just "Hello")

Este exemplo é retirado do Haskell Wiki:

import Control.Monad
import Control.Concurrent
import Control.Concurrent.STM
 
main = do 
  -- Initialise a new TVar
  shared <- atomically $ newTVar 0
  -- Read the value
  before <- atomRead shared
  putStrLn $ "Before: " ++ show before
  forkIO $ 25 `timesDo` (dispVar shared >> milliSleep 20)
  forkIO $ 10 `timesDo` (appV ((+) 2) shared >> milliSleep 50)
  forkIO $ 20 `timesDo` (appV pred shared >> milliSleep 25)
  milliSleep 800
  after <- atomRead shared
  putStrLn $ "After: " ++ show after
  where timesDo = replicateM_
       milliSleep = threadDelay . (*) 1000

atomRead = atomically . readTVar
dispVar x = atomRead x >>= print
appV fn x = atomically $ readTVar x >>= writeTVar x . fn
