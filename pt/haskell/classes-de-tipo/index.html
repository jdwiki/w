Typeclasses em Haskell são um meio de definir o comportamento associado a um tipo separadamente da definição desse tipo. Considerando que, digamos, em Java, você definiria o comportamento como parte da definição do tipo - ou seja, em uma interface, classe abstrata ou classe concreta - Haskell mantém essas duas coisas separadas.

Existem várias classes de tipos já definidas no pacote base do Haskell. A relação entre eles é ilustrada na seção Comentários abaixo.

O diagrama a seguir retirado do artigo Typeclassopedia mostra o relacionamento entre as várias classes de tipos em Haskell.

Eq #

Todos os tipos de dados básicos (como Int, String, Eq a => [a]) do Prelude, exceto para funções e IO, têm instâncias de Eq. Se um tipo instancia Eq significa que sabemos como comparar dois valores para igualdade valor ou estrutural.

> 3 == 2 
False
> 3 == 3
True

Métodos obrigatórios #

  • (==) :: Eq a => a -> a -> Boolean ou (/=) :: Eq a => a -> a -> Boolean (se apenas um for implementado, os outros padrões para a negação do definido)

Define #

  • (==) :: Eq a => a -> a -> Boolean
  • (/=) :: Eq a => a -> a -> Boolean

Superclasses diretas #

Nenhum

Subclasses notáveis #

  • Ord

Pedido #

Os tipos que instanciam Ord incluem, por exemplo, Int, String e [a] (para tipos a onde há uma instância Ord a). Se um tipo instancia Ord significa que conhecemos um tipo “natural” ordenação de valores desse tipo. Observe que muitas vezes há muitas opções possíveis do tipo “natural” ordenação de um tipo e Ord nos força a favorecer um.

Ord fornece os operadores padrão (<=), (<), (>), (>=), mas curiosamente define todos eles usando um tipo de dados algébrico personalizado

data Ordering = LT | EQ | GT

compare :: Ord a => a -> a -> Ordering

Métodos obrigatórios #

  • compare :: Ord a => a -> a -> Ordering ou (<=) :: Ord a => a -> a -> Boolean (o método padrão compare usa ( <=) em sua implementação)

Define #

  • compare :: Ord a => a -> a -> Ordering
  • (<=) :: Ord a => a -> a -> Boolean
  • (<) :: Ord a => a -> a -> Boolean
  • (>=) :: Ord a => a -> a -> Boolean
  • (>) :: Ord a => a -> a -> Boolean
  • min :: Ord a => a -> a -> a
  • max :: Ord a => a -> a -> a

Superclasses diretas #

  • Eq

Monóide #

Tipos que instanciam Monoid incluem listas, números e funções com valores de retorno Monoid, entre outros. Para instanciar Monoid um tipo deve suportar uma operação binária associativa (mappend ou (<>)) que combina seus valores, e ter um valor especial “zero” (mempty) tal que combinar um valor com ele não altera esse valor:

mempty  <>  x == x
x <>  mempty  == x

x <> (y <> z) == (x <> y) <> z

Intuitivamente, os tipos Monoid são “tipo lista”, pois suportam a adição de valores juntos. Alternativamente, os tipos Monoid podem ser pensados ​​como sequências de valores para os quais nos preocupamos com a ordem, mas não com o agrupamento. Por exemplo, uma árvore binária é um Monoid, mas usando as operações Monoid não podemos testemunhar sua estrutura de ramificação, apenas uma travessia de seus valores (veja Foldable e Traversable).

Métodos obrigatórios #

  • mempty :: Monóide m => m
  • mappend :: Monóide m => m -> m -> m

Superclasses diretas #

Nenhum

Número #

A classe mais geral para tipos de números, mais precisamente para anéis, ou seja, números que podem ser somados, subtraídos e multiplicados no sentido usual, mas não necessariamente dividido.

Esta classe contém tipos integrais (Int, Integer, Word32 etc.) e tipos fracionários (Double, Rational, também números complexos etc.). No caso de tipos finitos, a semântica é geralmente entendida como aritmética modular, ou seja, com over e underflow†.

Observe que as regras para as classes numéricas são muito menos obedecidas do que a [monad](https://www.wikiod.com/pt/haskell/monadas#A Mônada Talvez =201610010024186706477) ou leis monoides, ou aquelas para comparação de igualdade. Em particular, os números de ponto flutuante geralmente obedecem às leis apenas em um sentido aproximado.

Os métodos #

  • fromInteger :: Num a => Integer -> a. converter um inteiro para o tipo de número geral (envolvendo o intervalo, se necessário). Haskell [literais de número](https://www.wikiod.com/pt/haskell/literais-sobrecarregados#Numeral inteiro) pode ser entendido como um literal ‘Integer’ monomórfico com a conversão geral em torno dele, então você pode use o literal 5 em um contexto Int e uma configuração Complex Double.

  • (+) :: Num a => a -> a -> a. Adição padrão, geralmente entendida como associativa e comutativa, ou seja,

      a + (b + c) ≡ (a + b) + c
  a + b ≡ b + a

  • (-) :: Num a => a -> a -> a. Subtração, que é o inverso da adição:

      (a - b) + b ≡ (a + b) - b ≡ a

  • (*) :: Num a => a -> a -> a. Multiplicação, uma operação associativa que é distributiva sobre a adição:

      a * (b * c) ≡ (a * b) * c
  a * (b + c) ≡ a * b + a * c

para os casos mais comuns, a multiplicação também é comutativa, mas isso definitivamente não é um requisito.

  • negar :: Num a => a -> a. O nome completo do operador de negação unário. -1 é açúcar sintático para negate 1.

      -a ≡ negate a ≡ 0 - a

  • abs :: Num a => a -> a. A função de valor absoluto sempre dá um resultado não negativo da mesma magnitude

      abs (-a) ≡ abs a
  abs (abs a) ≡ abs a

abs a ≡ 0 só deve acontecer se a ≡ 0.

Para tipos reais é claro o que significa não negativo: você sempre tem abs a >= 0. Os tipos complexos etc. não têm uma ordenação bem definida, no entanto o resultado de abs deve estar sempre no subconjunto real‡ (ou seja, dar um número que também pode ser escrito como um único número literal sem negação).

  • signum :: Num a => a -> a. A função sign, de acordo com o nome, produz apenas -1 ou 1, dependendo do sinal do argumento. Na verdade, isso só é verdade para números reais diferentes de zero; em geral signum é melhor entendido como a função normalising:

      abs (signum a) ≡ 1   -- unless a≡0
  signum a * abs a ≡ a -- This is required to be true for all Num instances

Observe que seção 6.4.4 do Relatório Haskell 2010 requer explicitamente que esta última igualdade seja válida para qualquer Num instância.

Algumas bibliotecas, notadamente linear e hmatrix, têm uma compreensão muito mais vaga do que o A classe Num é para: eles a tratam apenas como uma maneira de sobrecarregar os operadores aritméticos. Embora isso seja bastante simples para + e -, já se torna problemático com * e mais ainda com os outros métodos. Por exemplo, deveria * significar multiplicação de matrizes ou multiplicação por elementos?
É sem dúvida uma má ideia definir tais instâncias não numéricas; considere classes dedicadas como VectorSpace.

† Em particular, os “negativos” de tipos sem sinal são agrupados em grandes positivos, por exemplo (-4 :: Word32) == 4294967292.

‡ Isto é amplamente não cumprido: tipos vetoriais não possuem um subconjunto real. As controversas instâncias Num para esses tipos geralmente definem abs e signum em termos de elementos, o que matematicamente falando não faz sentido.

Talvez e a classe Functor #

Em Haskell, os tipos de dados podem ter argumentos assim como as funções. Tome o tipo Talvez por exemplo.

Talvez é um tipo muito útil que nos permite representar a ideia de falha, ou a sua possibilidade. Em outras palavras, se houver a possibilidade de uma computação falhar, usamos o tipo Talvez. Maybe age como um wrapper para outros tipos, dando a eles funcionalidade adicional.

Sua declaração real é bastante simples.

Maybe a = Just a | Nothing

O que isso diz é que um Talvez vem em duas formas, um Só, que representa o sucesso, e um Nada, que representa o fracasso. Just recebe um argumento que determina o tipo do Maybe, e Nothing não recebe nenhum. Por exemplo, o valor Just "foo" terá o tipo Maybe String, que é um tipo de string envolvido com a funcionalidade adicional Maybe. O valor Nothing tem o tipo Maybe a onde a pode ser qualquer tipo.

Essa ideia de agrupar tipos para dar a eles funcionalidade adicional é muito útil e é aplicável a mais do que apenas Talvez. Outros exemplos incluem os tipos Either, IO e list, cada um fornecendo uma funcionalidade diferente. No entanto, existem algumas ações e habilidades que são comuns a todos esses tipos de wrapper. A mais notável delas é a capacidade de modificar o valor encapsulado.

É comum pensar nesses tipos de tipos como caixas que podem ter valores colocados nelas. Caixas diferentes guardam valores diferentes e fazem coisas diferentes, mas nenhuma é útil sem poder acessar o conteúdo dentro.

Para encapsular essa ideia, Haskell vem com uma typeclass padrão, chamada Functor. É definido da seguinte forma.

class Functor f where
  fmap :: (a -> b) -> f a -> f b

Como pode ser visto, a classe possui uma única função, fmap, de dois argumentos. O primeiro argumento é uma função de um tipo, a, para outro, b. O segundo argumento é um functor (tipo wrapper) contendo um valor do tipo a. Ele retorna um functor (tipo wrapper) contendo um valor do tipo b.

Em termos simples, fmap pega uma função e se aplica ao valor dentro de um functor. É a única função necessária para um tipo ser membro da classe Functor, mas é extremamente útil. Funções que operam em functores que possuem aplicações mais específicas podem ser encontradas nas classes de tipos Aplicative e Monad.

Herança de classe de tipo: classe de tipo Ord #

Haskell suporta uma noção de extensão de classe. Por exemplo, a classe Ord herda todas as operações em Eq, mas além disso tem uma função compare que retorna um Ordering entre os valores. Ord também pode conter os operadores comuns de comparação de pedidos, bem como um método min e um método max.

A notação => tem o mesmo significado que tem em uma assinatura de função e requer o tipo a para implementar Eq, a fim de implementar Ord.

data Ordering = EQ | LT | GT

class Eq a => Ord a where
    compare :: Ord a => a -> a -> Ordering
    (<)     :: Ord a => a -> a -> Bool
    (<=)    :: Ord a => a -> a -> Bool
    (>)     :: Ord a => a -> a -> Bool
    (>=)    :: Ord a => a -> a -> Bool
    min     :: Ord a => a -> a -> a
    max     :: Ord a => a -> a -> a

Todos os métodos que seguem compare podem ser derivados dele de várias maneiras:

x < y   = compare x y == LT
x <= y  = x < y || x == y -- Note the use of (==) inherited from Eq
x > y   = not (x <= y)
x >= y  = not (x < y)

min x y = case compare x y of
               EQ -> x
               LT -> x
               GT -> y

max x y = case compare x y of
               EQ -> x
               LT -> y
               GT -> x

Classes de tipo que estendem Ord devem implementar pelo menos o método compare ou o próprio método (<=), que constrói a rede de herança direcionada.
