Essas extensões de linguagem geralmente estão disponíveis ao usar o Glasgow Haskell Compiler (GHC), pois não fazem parte do Haskell 2010 language Report aprovado. Para usar essas extensões, deve-se informar ao compilador usando um flag ou colocar um programa LANGUAGE antes da palavra-chave module em um arquivo. A documentação oficial pode ser encontrada na seção 7 do guia do usuário GCH.

O formato do programa LANGUAGE é {-# LANGUAGE ExtensionOne, ExtensionTwo ... #-}. Esse é o literal {-# seguido por LANGUAGE seguido por uma lista de extensões separadas por vírgula e, finalmente, o fechamento #-}. Vários programas LANGUAGE podem ser colocados em um arquivo.

RankNTypes #

Imagine a seguinte situação:

foo :: Show a => (a -> String) -> String -> Int -> IO ()
foo show' string int = do
   putStrLn (show' string)
   putStrLn (show' int)

Aqui, queremos passar uma função que converte um valor em uma String, aplicar essa função a um parâmetro de string e a um parâmetro int e imprimir ambos. Na minha opinião, não há razão para isso falhar! Temos uma função que funciona em ambos os tipos de parâmetros que estamos passando.

Infelizmente, isso não digitará check! O GHC infere o tipo a com base em sua primeira ocorrência no corpo da função. Ou seja, assim que atingimos:

putStrLn (show' string)

O GHC inferirá que show' :: String -> String, já que string é uma String. Ele irá explodir enquanto tenta ‘mostrar’ int’.

RankNTypes permite que você escreva a assinatura de tipo da seguinte forma, quantificando todas as funções que satisfazem o tipo show':

foo :: (forall a. Show a => (a -> String)) -> String -> Int -> IO ()

Este é o polimorfismo de nível 2: Estamos afirmando que a função show' deve funcionar para todos os as dentro de nossa função, e a implementação anterior agora funciona.

A extensão RankNTypes permite o aninhamento arbitrário de blocos forall ... em assinaturas de tipo. Em outras palavras, permite polimorfismo de posto N.

Strings Sobrecarregadas #

Normalmente, literais de string em Haskell têm um tipo de String (que é um alias de tipo para [Char]). Embora isso não seja um problema para programas educacionais menores, os aplicativos do mundo real geralmente exigem armazenamento mais eficiente, como Text ou ByteString.

OverloadedStrings simplesmente altera o tipo de literais para

"test" :: Data.String.IsString a => a

Permitindo que eles sejam passados ​​diretamente para funções que esperam esse tipo. Muitas bibliotecas implementam essa interface para seus tipos semelhantes a strings, incluindo Data.Text e Data.ByteString, que fornecem certas vantagens de tempo e espaço sobre [Char].

Existem também alguns usos exclusivos de OverloadedStrings como aqueles da biblioteca Postgresql-simple que permite que consultas SQL sejam escritas entre aspas duplas como uma string normal, mas fornece proteção contra concatenação imprópria, uma fonte notória de SQL ataques de injeção

Para criar uma instância da classe IsString você precisa implementar a função fromString. Exemplo†:

data Foo = A | B | Other String deriving Show

instance IsString Foo where
  fromString "A" = A
  fromString "B" = B
  fromString xs  = Other xs

tests :: [ Foo ]
tests = [ "A", "B", "Testing" ]

† Este exemplo é cortesia de Lyndon Maydwell (sordina no GitHub) encontrado aqui.

Literais Binários #

O Haskell padrão permite escrever literais inteiros em decimal (sem qualquer prefixo), hexadecimal (precedido por 0x ou 0X) e octal (precedido por 0o ou 0O). A extensão BinaryLiterals adiciona a opção de binário (precedido por 0b ou 0B).

0b1111 == 15     -- evaluates to: True

##Quantificação Existencial Esta é uma extensão do sistema de tipos que permite tipos que são quantificados existencialmente, ou seja, possuem variáveis ​​de tipo que só são instanciadas em tempo de execução†.

Um valor de tipo existencial é semelhante a uma referência de classe base abstrata em linguagens OO: você não sabe o tipo exato em contém, mas pode restringir a class de tipos.

data S = forall a. Show a => S a

ou equivalentemente, com a sintaxe GADT:

{-# LANGUAGE GADTs #-}
data S where
   S :: Show a => a -> S

Tipos existenciais abrem a porta para coisas como contêineres quase heterogêneos: como dito acima, pode haver vários tipos em um valor S, mas todos eles podem ser shown, portanto, você também pode fazer

instance Show S where
    show (S a) = show a   -- we rely on (Show a) from the above

Agora podemos criar uma coleção de tais objetos:

ss = [S 5, S "test", S 3.0]

O que também nos permite usar o comportamento polimórfico:

mapM_ print ss

Existenciais podem ser muito poderosos, mas observe que eles não são necessários com muita frequência em Haskell. No exemplo acima, tudo o que você pode fazer com a instância Show é mostrar (duh!) os valores, ou seja, criar uma representação de string. O tipo S inteiro, portanto, contém exatamente tanta informação quanto a string que você obtém ao mostrá-la. Portanto, geralmente é melhor simplesmente armazenar essa string imediatamente, especialmente porque Haskell é preguiçoso e, portanto, a string a princípio será apenas uma conversão não avaliada.

Por outro lado, os existenciais causam alguns problemas únicos. Por exemplo, a forma como a informação de tipo está “escondida” em um existencial. Se você combinar um padrão em um valor S, você terá o tipo contido no escopo (mais precisamente, sua instância Show), mas essa informação nunca pode escapar de seu escopo, o que, portanto, se torna um pouco uma “sociedade secreta”. ”: o compilador não deixa nada escapar do escopo exceto valores cujo tipo já é conhecido de fora. Isso pode levar a erros estranhos [como Não foi possível combinar o tipo 'a0' com '()' 'a0' é intocável](http://stackoverflow.com/questions/28582210/type-inference-with-gadts- a0-é-intocável).

†Compare isso com o polimorfismo paramétrico comum, que geralmente é resolvido em tempo de compilação (permitindo o apagamento completo do tipo).

Os tipos existenciais são diferentes dos tipos Rank-N – essas extensões são, grosso modo, duais entre si: para realmente usar valores de um tipo existencial, você precisa de uma função polimórfica (possivelmente restrita), como show no exemplo. Uma função polimórfica é universalmente quantificada, ou seja, ela funciona para qualquer tipo em uma determinada classe, enquanto a quantificação existencial significa que ela funciona _para algum tipo particular que é a priori desconhecido. Se você tem uma função polimórfica, isso é suficiente, no entanto, para passar funções polimórficas como argumentos, você precisa de {-# LANGUAGE Rank2Types #-}:

genShowSs :: (∀ x . Show x => x -> String) -> [S] -> [String]
genShowSs f = map (\(S a) -> f a)

LambdaCase #

Uma extensão sintática que permite escrever \case no lugar de \arg -> case arg of.

Considere a seguinte definição de função:

dayOfTheWeek :: Int -> String
dayOfTheWeek 0 = "Sunday"
dayOfTheWeek 1 = "Monday"
dayOfTheWeek 2 = "Tuesday"
dayOfTheWeek 3 = "Wednesday"
dayOfTheWeek 4 = "Thursday"
dayOfTheWeek 5 = "Friday"
dayOfTheWeek 6 = "Saturday"

Se você quiser evitar repetir o nome da função, você pode escrever algo como:

dayOfTheWeek :: Int -> String
dayOfTheWeek i = case i of
    0 -> "Sunday"
    1 -> "Monday"
    2 -> "Tuesday"
    3 -> "Wednesday"
    4 -> "Thursday"
    5 -> "Friday"
    6 -> "Saturday"

Usando a extensão LambdaCase, você pode escrever isso como uma expressão de função, sem precisar nomear o argumento:

{-# LANGUAGE LambdaCase #-}

dayOfTheWeek :: Int -> String
dayOfTheWeek = \case
    0 -> "Sunday"
    1 -> "Monday"
    2 -> "Tuesday"
    3 -> "Wednesday"
    4 -> "Thursday"
    5 -> "Friday"
    6 -> "Saturday"

Dependências Funcionais #

Se você tiver uma classe de tipo multiparâmetros com argumentos a, b, c e x, essa extensão permite expressar que o tipo x pode ser identificado exclusivamente de a, b e c:

class SomeClass a b c x | a b c -> x where ...

Ao declarar uma instância dessa classe, ela será verificada em relação a todas as outras instâncias para garantir que a dependência funcional seja válida, ou seja, nenhuma outra instância com o mesmo a b c mas com um x diferente existe.

Você pode especificar várias dependências em uma lista separada por vírgulas:

class OtherClass a b c d | a b -> c d, a d -> b where ...

Por exemplo em MTL podemos ver:

class MonadReader r m| m -> r where ...
instance MonadReader r ((->) r) where ...

Agora, se você tiver um valor do tipo MonadReader a ((->) Foo) => a, o compilador pode inferir que a ~ Foo, já que o segundo argumento determina completamente o primeiro e simplificará o tipo de acordo .

A classe SomeClass pode ser pensada como uma função dos argumentos a b c que resulta em x. Essas classes podem ser usadas para fazer cálculos no sistema de tipos.

Instâncias flexíveis #

Instâncias regulares requerem:

All instance types must be of the form (T a1 ... an)
where a1 ... an are *distinct type variables*,
and each type variable appears at most once in the instance head.

Isso significa que, por exemplo, enquanto você pode criar uma instância para [a] você não pode criar uma instância especificamente para [Int].; FlexibleInstances relaxa que:

class C a where

-- works out of the box
instance C [a] where

-- requires FlexibleInstances
instance C [Int] where

GADTs #

Tipos de dados algébricos convencionais são paramétricos em suas variáveis ​​de tipo. Por exemplo, se definirmos um ADT como

data Expr a = IntLit Int 
            | BoolLit Bool 
            | If (Expr Bool) (Expr a) (Expr a)

com a esperança de que isso exclua estaticamente condicionais não bem tipadas, isso não se comportará como esperado, pois o tipo de IntLit :: Int -> Expr a é quantificado universalmente: para qualquer escolha de a, produz um valor do tipo Expr a. Em particular, para a ~ Bool, temos IntLit :: Int -> Expr Bool, permitindo-nos construir algo como If (IntLit 1) e1 e2 que é o tipo do construtor If tentando descartar.

Tipos de dados algébricos generalizados nos permite controlar o tipo resultante de um construtor de dados para que eles não sejam meramente paramétricos. Podemos reescrever nosso tipo Expr como um GADT assim:

data Expr a where
  IntLit :: Int -> Expr Int
  BoolLit :: Bool -> Expr Bool
  If :: Expr Bool -> Expr a -> Expr a -> Expr a

Aqui, o tipo do construtor IntLit é Int -> Expr Int, e assim IntLit 1 :: Expr Bool não fará verificação de tipo.

A correspondência de padrões em um valor GADT causa o refinamento do tipo do termo retornado. Por exemplo, é possível escrever um avaliador para Expr a assim:

crazyEval :: Expr a -> a
crazyEval (IntLit x) = 
   -- Here we can use `(+)` because x :: Int
   x + 1 
crazyEval (BoolLit b) = 
   -- Here we can use `not` because b :: Bool
   not b
crazyEval (If b thn els) = 
  -- Because b :: Expr Bool, we can use `crazyEval b :: Bool`.
  -- Also, because thn :: Expr a and els :: Expr a, we can pass either to 
  -- the recursive call to `crazyEval` and get an a back
  crazyEval $ if crazyEval b then thn else els

Observe que podemos usar (+) nas definições acima porque quando, por exemplo, IntLit x é compatível com o padrão, também aprendemos que a ~ Int (e da mesma forma para not e if_then_else_ quando a ~ Bool).

Seções de Tupla #

Uma extensão sintática que permite aplicar o construtor da tupla (que é um operador) de forma seccional:

(a,b) == (,) a b

-- With TupleSections
(a,b) == (,) a b == (a,) b == (,b) a

N-tuplas #

Também funciona para tuplas com aridade maior que dois

(,2,) 1 3 == (1,2,3)

Mapeamento #

Isso pode ser útil em outros lugares onde as seções são usadas:

map (,"tag") [1,2,3] == [(1,"tag"), (2, "tag"), (3, "tag")]

O exemplo acima sem esta extensão ficaria assim:

map (\a -> (a, "tag")) [1,2,3]

##Listas Sobrecarregadas adicionado no GHC 7.8.

OverloadedLists, semelhante a [OverloadedStrings](https://www.wikiod.com/pt/haskell/extensoes-de-linguagem-comuns-do-ghc#Strings Sobrecarregadas), permite que literais de lista sejam desaçucarados da seguinte forma:

[]          -- fromListN 0 []
[x]         -- fromListN 1 (x : [])
[x .. ]     -- fromList (enumFrom x)

Isso é útil ao lidar com tipos como Set, Vector e Maps.

['0' .. '9']             :: Set Char
[1 .. 10]                :: Vector Int
[("default",0), (k1,v1)] :: Map String Int
['a' .. 'z']             :: Text

A classe IsList em GHC.Exts deve ser usada com este extensão.

IsList é equipado com uma função de tipo, Item, e três funções, fromList :: [Item l] -> l, toList :: l -> [Item l] e fromListN :: Int - > [Item l] -> l onde fromListN é opcional. As implementações típicas são:

instance IsList [a] where
  type Item [a] = a
  fromList = id
  toList   = id

instance (Ord a) => IsList (Set a) where
  type Item (Set a) = a
  fromList = Set.fromList
  toList   = Set.toList

Exemplos retirados de OverloadedLists – GHC.

MultiParamTypeClasses #

É uma extensão muito comum que permite classes de tipo com vários parâmetros de tipo. Você pode pensar no MPTC como uma relação entre tipos.

{-# LANGUAGE MultiParamTypeClasses #-}

class Convertable a b where
    convert :: a -> b

instance Convertable Int Float where
    convert i = fromIntegral i

A ordem dos parâmetros é importante.

Às vezes, os MPTCs podem ser substituídos por famílias de tipos.

##Sintaxe Unicode Uma extensão que permite usar caracteres Unicode no lugar de certos operadores e nomes integrados.

ASCII Unicode Uso(s)
:: ∷ tem tipo
-> → tipos de função, lambdas, ramos case, etc.
=> ⇒ restrições de classe
forall ∀ polimorfismo explícito
<- ← notação do
* ★ o tipo (ou tipo) de tipos (por exemplo, Int :: ★)
>- ⤚ notação proc para [Setas]
-< ⤙ notação proc para [Setas]
>>- ⤜ notação proc para [Setas]
-<< ⤛ notação proc para [Setas]

Por exemplo:

runST :: (forall s. ST s a) -> a

se tornaria

runST ∷ (∀ s. ST s a) → a

Observe que o exemplo * vs. ★ é um pouco diferente: como * não é reservado, ★ também funciona da mesma maneira que * para multiplicação ou qualquer outra função chamada (*) , e vice versa. Por exemplo:

ghci> 2 ★ 3
6
ghci> let (*) = (+) in 2 ★ 3
5
ghci> let (★) = (-) in 2 * 3
-1

PadrãoSinônimos #

Sinônimos de padrão são abstrações de padrões semelhantes a como funções são abstrações de expressões.

Para este exemplo, vamos ver a interface que Data.Sequence expõe e vamos ver como ela pode ser melhorado com sinônimos de padrão. O tipo Seq é um tipo de dados que, internamente, usa um complicado representação para alcançar uma boa complexidade assintótica para várias operações, mais notavelmente tanto O(1) (des)consing quanto (un)snocing.

Mas essa representação é complicada e alguns de seus invariantes não podem ser expressos no sistema de tipos de Haskell. Por causa disso, o tipo Seq é exposto aos usuários como um tipo abstrato, juntamente com funções de acesso e construtor de preservação invariável, entre elas:

empty :: Seq a

(<|) :: a -> Seq a -> Seq a
data ViewL a = EmptyL | a :< (Seq a)
viewl :: Seq a -> ViewL a

(|>) :: Seq a -> a -> Seq a 
data ViewR a = EmptyR | (Seq a) :> a 
viewr :: Seq a -> ViewR a

Mas usar esta interface pode ser um pouco complicado:

uncons :: Seq a -> Maybe (a, Seq a)
uncons xs = case viewl xs of
    x :< xs' -> Just (x, xs')
    EmptyL -> Nothing

Podemos usar padrões de visualização para limpá-lo um pouco:

{-# LANGUAGE ViewPatterns #-}

uncons :: Seq a -> Maybe (a, Seq a)
uncons (viewl -> x :< xs) = Just (x, xs)
uncons _ = Nothing

Usando a extensão de linguagem PatternSynonyms, podemos fornecer uma interface ainda melhor, permitindo que a correspondência de padrões finja que temos uma lista cons ou snoc:

{-# LANGUAGE PatternSynonyms #-}
import Data.Sequence (Seq)
import qualified Data.Sequence as Seq

pattern Empty :: Seq a
pattern Empty <- (Seq.viewl -> Seq.EmptyL)

pattern (:<) :: a -> Seq a -> Seq a
pattern x :< xs <- (Seq.viewl -> x Seq.:< xs)

pattern (:>) :: Seq a -> a -> Seq a
pattern xs :> x <- (Seq.viewr -> xs Seq.:> x)

Isso nos permite escrever uncons em um estilo muito natural:

uncons :: Seq a -> Maybe (a, Seq a)
uncons (x :< xs) = Just (x, xs)
uncons _ = Nothing

Variáveis ​​de Tipo de Escopo #

ScopedTypeVariables permite que você faça referência a tipos quantificados universalmente dentro de uma declaração. Para ser mais explícito:

import Data.Monoid

foo :: forall a b c. (Monoid b, Monoid c) => (a, b, c) -> (b, c) -> (a, b, c)
foo (a, b, c) (b', c') = (a :: a, b'', c'')
    where (b'', c'') = (b <> b', c <> c') :: (b, c)

O importante é que podemos usar a, b e c para instruir o compilador em subexpressões da declaração (a tupla na cláusula where e o primeiro a no resultado final). Na prática, ScopedTypeVariables auxilia na escrita de funções complexas como uma soma de partes, permitindo ao programador adicionar assinaturas de tipo a valores intermediários que não possuem tipos concretos.
