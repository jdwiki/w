Teoria das categorias como um sistema para organizar a abstração #

A teoria das categorias é uma teoria matemática moderna e um ramo da álgebra abstrata focada na natureza da conectividade e da relação. É útil para fornecer bases sólidas e linguagem comum para muitas abstrações de programação altamente reutilizáveis. Haskell usa a teoria da categoria como inspiração para algumas das principais classes de tipos disponíveis na biblioteca padrão e em várias bibliotecas populares de terceiros.

Um exemplo #

A typeclass Functor diz que se um tipo F instancia Functor (para o qual escrevemos Functor F) então temos uma operação genérica

fmap :: (a -> b) -> (F a -> F b)

que nos permite “mapear” sobre F. A intuição padrão (mas imperfeita) é que F a é um contêiner cheio de valores do tipo a e fmap nos permite aplicar uma transformação a cada um desses elementos contidos. Um exemplo é Talvez

instance Functor Maybe where
  fmap f Nothing = Nothing     -- if there are no values contained, do nothing
  fmap f (Just a) = Just (f a) -- else, apply our transformation

Dada essa intuição, uma pergunta comum é “por que não chamar Functor algo óbvio como Mapable?”.

Uma dica de Teoria das Categorias #

A razão é que Functor se encaixa em um conjunto de estruturas comuns na teoria da categoria e, portanto, ao chamar Functor de “Functor” podemos ver como ele se conecta a esse corpo mais profundo de conhecimento.

Em particular, a Teoria das Categorias está altamente preocupada com a ideia de flechas de um lugar para outro. Em Haskell, o conjunto de setas mais importante são as setas de função a -> b. Uma coisa comum para estudar na Teoria das Categorias é como um conjunto de setas se relaciona com outro conjunto. Em particular, para qualquer construtor de tipo F, o conjunto de setas da forma F a -> F b também são interessantes.

Então um Functor é qualquer F tal que existe uma conexão entre as setas normais de Haskell a -> b e as setas específicas F F a -> F b. A conexão é definida por fmap e também reconhecemos algumas leis que devem valer

forall (x :: F a) . fmap id x == x

forall (f :: a -> b) (g :: b -> c) . fmap g . fmap f = fmap (g . f)

Todas essas leis surgem naturalmente da interpretação teórica da categoria de Functor e não seriam tão obviamente necessárias se pensássemos apenas em Functor como relacionado ao “mapeamento sobre elementos”.

Tipos de Haskell como uma categoria #

Definição da categoria #

Os tipos Haskell junto com funções entre tipos formam (quase†) uma categoria. Temos um morfismo de identidade (função) (id :: a -> a) para cada objeto (tipo) a; e composição de morfismos ((.) :: (b -> c) -> (a -> b) -> a -> c), que obedecem às leis da categoria:

f . id = f = id . f
h . (g . f) = (h . g) . f

Normalmente chamamos essa categoria de Hask.

Isomorfismos #

Na teoria das categorias, temos um isomorfismo quando temos um morfismo que possui uma inversa, ou seja, existe um morfismo que pode ser composto com ele para criar a identidade. Em Hask isso equivale a ter um par de morfismos f,g tal que:

 f . g == id == g . f

Se encontrarmos um par desses morfismos entre dois tipos, nós os chamamos de isomórficos entre si.

Um exemplo de dois tipos isomórficos seria ((),a) e a para alguns a. Podemos construir os dois morfismos:

f :: ((),a) -> a
f ((),x) = x

g :: a -> ((),a)
g x = ((),x)

E podemos verificar que f . g == id == g. f.

Funtores #

Um funtor, na teoria das categorias, vai de uma categoria para outra, mapeando objetos e morfismos. Estamos trabalhando apenas em uma categoria, a categoria Hask de tipos Haskell, então veremos apenas functores de Hask a Hask, esses functores, cuja categoria de origem e destino são as mesmo, são chamados de endofuntores. Nossos endofuntores serão os tipos polimórficos pegando um tipo e retornando outro:

F :: * -> *

Obedecer às leis dos funtores categóricos (preservar identidades e composição) equivale a obedecer às leis dos funtores de Haskell:

fmap (f . g) = (fmap f) . (fmap g)
fmap id = id

Então, temos, por exemplo, que [], Maybe ou (-> r) são functores em Hask.

Mônadas #

Uma mônada na teoria das categorias é um monóide na categoria de endofuntores. Esta categoria tem endofuntores como objetos F :: * -> * e transformações naturais (transformações entre eles forall a . F a -> G a) como morfismos.

Um objeto monóide pode ser definido em uma categoria monoidal e é um tipo com dois morfismos:

zero :: () -> M
mappend :: (M,M) -> M

Podemos traduzir isso aproximadamente para a categoria de endofuntores Hask como:

return :: a -> m a
join :: m (m a) -> m a

E obedecer às leis da mônada equivale a obedecer às leis categóricas dos objetos monóides.

†Na verdade, a classe de todos os tipos junto com a classe de funções entre tipos não formam estritamente uma categoria em Haskell, devido à existência de undefined. Normalmente, isso é corrigido simplesmente definindo os objetos da categoria Hask como tipos sem valores inferiores, o que exclui funções não terminantes e valores infinitos (codados). Para uma discussão detalhada deste tópico, veja aqui.

Definição de uma categoria #

Uma categoria ‘C’ consiste em:

  • Uma coleção de objetos chamada Obj(C);
  • Uma coleção (chamada Hom(C)) de morfismos entre esses objetos. Se a e b estão em Obj(C), então um morfismo f em Hom(C) é tipicamente denotado como f : a -> b, e a coleção de todos os morfismos entre a e b são denotados como hom(a,b) ;
  • Um morfismo especial chamado de morfismo identity - para todo a : Obj(C) existe um morfismo id : a -> a ;
  • Um operador de composição (.), tomando dois morfismos f : a -> b, g : b -> c e produzindo um morfismo a -> c

que obedecem as seguintes leis:

For all f : a -> x, g : x -> b, then id . f = f and g . id = g

For all f : a -> b, g : b -> c and h : c -> d, then h . (g . f) = (h . g) . f

Em outras palavras, a composição com o morfismo de identidade (à esquerda ou à direita) não altera o outro morfismo, e a composição é associativa.

Em Haskell, a Category é definida como uma typeclass em Control.Category:

-- | A class for categories.
--   id and (.) must form a monoid.
class Category cat where
    -- | the identity morphism
    id :: cat a a

    -- | morphism composition
    (.) :: cat b c -> cat a b -> cat a c

Neste caso, cat :: k -> k -> * objetiva a relação de morfismo - existe um morfismo cat a b se e somente se cat a b é habitado (ou seja, tem um valor). a, b e c estão todos em Obj(C). Obj(C) em si é representado pelo kind k - por exemplo, quando k ~ *, como normalmente é o caso, os objetos são tipos.

O exemplo canônico de uma categoria em Haskell é a categoria de função:

instance Category (->) where
  id = Prelude.id
  (.) = Prelude..

Outro exemplo comum é a Categoria das setas Kleisli para uma Mônada:

newtype Kleisli m a b = Kleisli (a -> m b)

class Monad m => Category (Kleisli m) where
  id = Kleisli return
  Kleisli f . Kleisli g = Kleisli (f >=> g)

Produto de tipos em Hask #

Produtos categóricos #

Na teoria das categorias, o produto de dois objetos X, Y é outro objeto Z com duas projeções: π₁ : Z → X e *π₂ : Z → Y *; tal que quaisquer outros dois morfismos de outro objeto se decompõem exclusivamente através dessas projeções. Em outras palavras, se existe f₁ : W → X e f₂ : W → Y, existe um único morfismo g : W → Z tal que *π₁ ○ g = f₁ * e π₂ ○ g = f₂.

Produtos em Hask #

Isso se traduz na categoria Hask de tipos Haskell da seguinte forma, Z é o produto de A, B quando:

-- if there are two functions
f1 :: W -> A
f2 :: W -> B
-- we can construct a unique function
g  :: W -> Z
-- and we have two projections
p1 :: Z -> A
p2 :: Z -> B
-- such that the other two functions decompose using g
p1 . g == f1
p2 . g == f2

O tipo de produto de dois tipos A, B, que segue a lei indicada acima, é a tupla dos dois tipos (A,B), e as duas projeções são fst e snd. Podemos verificar que segue a regra acima, se tivermos duas funções f1 :: W -> A e f2 :: W -> B podemos decompô-las de forma única como segue:

decompose :: (W -> A) -> (W -> B) -> (W -> (A,B))
decompose f1 f2 = (\x -> (f1 x, f2 x))

E podemos verificar que a decomposição está correta:

fst . (decompose f1 f2) = f1
snd . (decompose f1 f2) = f2

Unicidade até isomorfismo #

A escolha de (A,B) como produto de A e B não é única. Outra escolha lógica e equivalente teria sido:

data Pair a b = Pair a b

Além disso, também poderíamos ter escolhido (B,A) como o produto, ou mesmo (B,A,()), e poderíamos encontrar uma função de decomposição como a acima também seguindo as regras:

decompose2 :: (W -> A) -> (W -> B) -> (W -> (B,A,()))
decompose2 f1 f2 = (\x -> (f2 x, f1 x, ()))

Isso ocorre porque o produto não é único, mas único até o isomorfismo. Cada dois produtos de A e B não precisam ser iguais, mas devem ser isomórficos. Como exemplo, os dois produtos diferentes que acabamos de definir, (A,B) e (B,A,()), são isomórficos:

iso1 :: (A,B) -> (B,A,())
iso1 (x,y) = (y,x,())

iso2 :: (B,A,()) -> (A,B)
iso2 (y,x,()) = (x,y)

Unicidade da decomposição #

É importante observar que também a função de decomposição deve ser única. Existem tipos que seguem todas as regras exigidas para serem produto, mas a decomposição não é única. Como exemplo, podemos tentar usar (A,(B,Bool)) com projeções fst fst . snd como um produto de A e B:

decompose3 :: (W -> A) -> (W -> B) -> (W -> (A,(B,Bool)))
decompose3 f1 f2 = (\x -> (f1 x, (f2 x, True)))

Podemos verificar se funciona:

fst         . (decompose3 f1 f2) = f1 x
(fst . snd) . (decompose3 f1 f2) = f2 x

Mas o problema aqui é que poderíamos ter escrito outra decomposição, a saber:

decompose3' :: (W -> A) -> (W -> B) -> (W -> (A,(B,Bool)))
decompose3' f1 f2 = (\x -> (f1 x, (f2 x, False)))

E, como a decomposição não é única, (A,(B,Bool)) é não o produto de A e B em Hask

Coproduto de tipos em Hask #

Intuição #

O produto categórico de dois tipos A e B deve conter as informações mínimas necessárias para conter uma instância do tipo A ou do tipo B. Podemos ver agora que o coproduto intuitivo de dois tipos deve ser Either a b. Outros candidatos, como Either a (b,Bool), conteriam uma parte de informações desnecessárias e não seriam mínimas.

A definição formal é derivada da definição categórica de coproduto.

Coprodutos categóricos #

Um coproduto categórico é a noção dual de um produto categórico. É obtido diretamente invertendo todas as setas na definição do produto. O coproduto de dois objetos X,Y é outro objeto Z com duas inclusões: i_1: X → Z e i_2: Y → Z; tal que quaisquer outros dois morfismos de X e Y para outro objeto se decompõem exclusivamente por meio dessas inclusões. Em outras palavras, se existem dois morfismos f₁ : X → W e f₂ : Y → W, existe um único morfismo g : Z → W tal que g ○ i₁ = f₁ e g ○ i₂ = f₂

Coprodutos em Hask #

A tradução para a categoria Hask é semelhante à tradução do produto:

-- if there are two functions
f1 :: A -> W
f2 :: B -> W
-- and we have a coproduct with two inclusions
i1 :: A -> Z
i2 :: B -> Z
-- we can construct a unique function
g  :: Z -> W
-- such that the other two functions decompose using g
g . i1 == f1
g . i2 == f2

O tipo de coproduto de dois tipos A e B em Hask é Either a b ou qualquer outro tipo isomórfico a ele:

-- Coproduct
-- The two inclusions are Left and Right
data Either a b = Left a | Right b

-- If we have those functions, we can decompose them through the coproduct
decompose :: (A -> W) -> (B -> W) -> (Either A B -> W)
decompose f1 f2 (Left x)  = f1 x
decompose f1 f2 (Right y) = f2 y

Aplicativo Haskell em termos de Teoria das Categorias #

O Functor de um Haskell permite mapear qualquer tipo a (um objeto de Hask) para um tipo F a e também mapear uma função a -> b (um morfismo de *Hask *) para uma função com o tipo F a -> F b. Isso corresponde a uma definição da Teoria das Categorias no sentido de que o functor preserva a estrutura básica da categoria.

Uma categoria monoidal é uma categoria que possui alguma estrutura adicional:

  • Um produto tensor (consulte [Produto de tipos em Hask](https://www.wikiod.com/pt/haskell/teoria-da-categoria#Produto de tipos em Hask))
  • Uma unidade tensora (objeto unitário)

Tomando um par como nosso produto, esta definição pode ser traduzida para Haskell da seguinte maneira:

class Functor f => Monoidal f where
    mcat :: f a -> f b -> f (a,b)
    munit :: f ()

A classe Aplicativo é equivalente a esta Monoidal e, portanto, pode ser implementada em termos dela:

instance Monoidal f => Applicative f where
    pure x = fmap (const x) munit
    f <*> fa = (\(f, a) -> f a) <$> (mcat f fa)
