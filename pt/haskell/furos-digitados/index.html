Furos digitados

  • Semântica de furos digitados

Um dos pontos fortes do Haskell é a capacidade de alavancar o sistema de tipos para modelar partes do domínio do problema no sistema de tipos. Ao fazer isso, muitas vezes encontramos tipos muito complexos. Ao escrever programas com esses tipos (ou seja, com valores com esses tipos), às vezes torna-se quase impossível ‘manipular’ todos os tipos. A partir do GHC 7.8, há um novo recurso sintático chamado buracos digitados. As lacunas digitadas não alteram a semântica da linguagem principal; destinam-se apenas a auxiliar na escrita de programas.

Para uma explicação detalhada de furos digitados, bem como uma discussão sobre o projeto de furos digitados, veja o Haskell wiki.

Seção do guia do usuário do GHC em furos digitados.

Sintaxe dos furos digitados #

Um buraco digitado é um único sublinhado (_) ou um identificador Haskell válido que não está no escopo, em um contexto de expressão. Antes da existência de buracos digitados, ambas as coisas acionariam um erro, então a nova sintaxe não interfere em nenhuma sintaxe antiga.

Controlando o comportamento de furos tipados #

O comportamento padrão dos furos tipados é produzir um erro em tempo de compilação ao encontrar um furo tipado. No entanto, existem vários sinalizadores para ajustar seu comportamento. Esses sinalizadores são resumidos da seguinte forma (GHC trac):

Por padrão, o GHC digitou buracos habilitados e produz um erro de compilação quando encontra um furo digitado.

Quando -fdefer-type-errors ou -fdefer-typed-holes está habilitado, hole erros são convertidos em avisos e resultam em erros de tempo de execução quando avaliado.

O sinalizador de aviso -fwarn-typed-holes está ativado por padrão. Sem -fdefer-type-errors ou -fdefer-typed-holes este sinalizador é no-op, já que buracos digitados são um erro nestas condições. Se qualquer um dos sinalizadores de adiamento estão habilitados (convertendo erros de furos digitados em avisos) o sinalizador -fno-warn-typed-holes desativa os avisos. Isso significa a compilação é bem-sucedida silenciosamente e a avaliação de um furo produzirá um erro de execução.

Semântica de furos digitados #

O valor de um tipo hole pode simplesmente ser considerado ‘indefinido’, embora um typed hole acione um erro em tempo de compilação, então não é estritamente necessário atribuir um valor a ele. No entanto, um furo digitado (quando ativado) produz um erro de tempo de compilação (ou aviso com erros de tipo adiados) que indica o nome do furo digitado, seu tipo inferido mais geral e os tipos de quaisquer ligações locais. Por exemplo:

Prelude> \x -> _var + length (drop 1 x)

<interactive>:19:7: Warning:
    Found hole `_var' with type: Int
    Relevant bindings include
      x :: [a] (bound at <interactive>:19:2)
      it :: [a] -> Int (bound at <interactive>:19:1)
    In the first argument of `(+)', namely `_var'
    In the expression: _var + length (drop 1 x)
    In the expression: \ x -> _var + length (drop 1 x)

Observe que no caso de furos digitados nas expressões inseridas no GHCi repl (como acima), o tipo da expressão inserida também é relatado, como it (aqui do tipo [a] -> Int).

Usando buracos tipados para definir uma instância de classe #

Furos digitados podem facilitar a definição de funções, através de um processo interativo.

Digamos que você queira definir uma instância de classe Foo Bar (para seu tipo Bar personalizado, para usá-lo com alguma função de biblioteca polimórfica que requer uma instância Foo). Você agora tradicionalmente procura a documentação do Foo, descobre quais métodos você precisa definir, examina seus tipos etc. – mas com buracos digitados, você pode pular isso!

Primeiro, apenas defina uma instância fictícia:

instance Foo Bar where

O compilador agora irá reclamar

Bar.hs:13:10: Warning:
No explicit implementation for
  ‘foom’ and ‘quun’
In the instance declaration for ‘Foo Bar’

Ok, então precisamos definir foom para Bar. Mas o que é isso deveria ser? Novamente, estamos com preguiça de olhar na documentação e apenas perguntar ao compilador:

instance Foo Bar where
  foom = _

Aqui usamos um buraco digitado como uma simples “consulta de documentação”. As saídas do compilador

Bar.hs:14:10:
    Found hole ‘_’ with type: Bar -> Gronk Bar
    Relevant bindings include
      foom :: Bar -> Gronk Bar (bound at Foo.hs:4:28)
    In the expression: _
    In an equation for ‘foom’: foom = _
    In the instance declaration for ‘Foo Bar’

Observe como o compilador já preencheu a variável de tipo de classe com o tipo concreto Bar para o qual queremos instanciar. Isso pode tornar a assinatura muito mais fácil de entender do que a polimórfica encontrada na documentação da classe, especialmente se você estiver lidando com um método mais complicado de, por exemplo, uma classe do tipo multiparâmetro.

Mas o que diabos é ‘Gronk’? Neste ponto, provavelmente é uma boa ideia perguntar Hayoo. No entanto, ainda podemos escapar sem isso: como um palpite cego, assumimos que este não é apenas um construtor de tipo, mas também o construtor de valor único, ou seja, pode ser usado como uma função que de alguma forma produzirá um valor Gronk a. Então nós tentamos

instance Foo Bar where
  foom bar = _ Gronk

Se tivermos sorte, Gronk é na verdade um valor, e o compilador agora dirá

    Found hole ‘_’
      with type: (Int -> [(Int, b0)] -> Gronk b0) -> Gronk Bar
    Where: ‘b0’ is an ambiguous type variable

Ok, isso é feio – a princípio apenas observe que Gronk tem dois argumentos, então podemos refinar nossa tentativa:

instance Foo Bar where
  foom bar = Gronk _ _

E isso agora está bem claro:

    Found hole ‘_’ with type: [(Int, Bar)]
    Relevant bindings include
      bar :: Bar (bound at Bar.hs:14:29)
      foom :: Bar -> Gronk Bar (bound at Foo.hs:15:24)
    In the second argument of ‘Gronk’, namely ‘_’
    In the expression: Gronk _ _
    In an equation for ‘foom’: foom bar = Gronk _ _

Agora você pode progredir ainda mais, por exemplo, desconstruindo o valor bar (os componentes aparecerão, com tipos, na seção ligações relevantes). Muitas vezes, em algum momento é completamente óbvio qual será a definição correta, porque você vê todos os argumentos disponíveis e os tipos se encaixam como um quebra-cabeça. Ou, alternativamente, você pode ver que a definição é impossível e por quê.

Tudo isso funciona melhor em um editor com compilação interativa, por exemplo. Emacs com modo haskell. Você pode então usar buracos digitados como consultas de valor com o mouse em um IDE para uma linguagem imperativa dinâmica interpretada, mas sem todas as limitações.
