Uma mônada é um tipo de dados de ações que podem ser compostas. Monad é a classe de construtores de tipo cujos valores representam tais ações. Talvez IO seja o mais reconhecível: um valor de IO a é uma “receita para recuperar um valor a do mundo real”.

Dizemos que um construtor de tipo m (como [] ou Talvez) forma uma mônada se houver uma instância Mônada m que satisfaça certas leis sobre composição de ações. Podemos então raciocinar sobre m a como uma “ação cujo resultado tem o tipo a”.

Nenhuma maneira geral de extrair valor de uma computação monádica #

Você pode envolver valores em ações e canalizar o resultado de uma computação para outra:

return :: Monad m => a -> m a
(>>=)  :: Monad m => m a -> (a -> m b) -> m b

No entanto, a definição de uma Mônada não garante a existência de uma função do tipo Mônada m => m a -> a.

Isso significa que, em geral, não há como extrair um valor de um cálculo (ou seja, “desembrulhar”). Este é o caso de muitos casos:

extract :: Maybe a -> a
extract (Just x) = x          -- Sure, this works, but...
extract Nothing  = undefined  -- We can’t extract a value from failure.

Especificamente, não existe a função IO a -> a, que muitas vezes confunde os iniciantes; veja [este exemplo](https://www.wikiod.com/pt/haskell/monadas#IO mônada).

Mônada como uma subclasse de aplicativo #

A partir do GHC 7.10, Aplicativo é uma superclasse de Mônada (ou seja, todo tipo que é uma Mônada também deve ser um Aplicativo). Todos os métodos de Aplicativo (pure, <*>) podem ser implementados em termos de métodos de Monad (return, >>=).

É óbvio que pure e return servem a propósitos equivalentes, então pure = return. A definição para <*> é relativamente clara:

mf <*> mx = do { f <- mf; x <- mx; return (f x) }                 
       -- = mf >>= (\f -> mx >>= (\x -> return (f x)))
       -- = [r   | f <- mf, x <- mx, r <- return (f x)]   -- with MonadComprehensions
       -- = [f x | f <- mf, x <- mx]

Esta função é definida como ap nas bibliotecas padrão.

Assim, se você já definiu uma instância de Monad para um tipo, você pode efetivamente obter uma instância de Aplicativo para ele “de graça” definindo

instance Applicative < type > where
    pure  = return
    (<*>) = ap

Tal como acontece com as leis da mônada, essas equivalências não são aplicadas, mas os desenvolvedores devem garantir que elas sejam sempre mantidas.

Definição de Mônada #

class Monad m where
    return :: a -> m a
    (>>=) :: m a -> (a -> m b) -> m b

A função mais importante para lidar com mônadas é o operador de ligação >>=:

(>>=) :: m a -> (a -> m b) -> m b
  • Pense em m a como “uma ação com um resultado a”.
  • Pense em a -> m b como “uma ação (dependendo de um parâmetro a) com um resultado b.”.

>>= seqüencia duas ações juntas canalizando o resultado da primeira ação para a segunda.

A outra função definida por Monad é:

return :: a -> m a

Seu nome é infeliz: este return não tem nada a ver com a palavra-chave return encontrada em linguagens de programação imperativas.

return x é a ação trivial produzindo x como resultado. (É trivial no [sentido seguinte](https://www.wikiod.com/pt/haskell/monadas#A Mônada Talvez -leis-e-talvez-mônada):)

return x >>= f       ≡  f x     --  “left identity” monad law
       x >>= return  ≡  x       -- “right identity” monad law

A Mônada Talvez #

Maybe é usado para representar valores possivelmente vazios - semelhante a null em outras linguagens. Geralmente é usado como o tipo de saída de funções que podem falhar de alguma forma.

Considere a seguinte função:

halve :: Int -> Maybe Int
halve x
  | even x = Just (x `div` 2)
  | odd x  = Nothing

Pense em halve como uma ação, dependendo de um Int, que tenta reduzir pela metade o inteiro, falhando se for ímpar.

Como dividimos um número inteiro três vezes pela metade?

takeOneEighth :: Int -> Maybe Int            -- (after you read the 'do' sub-section:)
takeOneEighth x =                
  case halve x of                               --  do {
    Nothing -> Nothing
    Just oneHalf ->                             --     oneHalf    <- halve x
      case halve oneHalf of
        Nothing -> Nothing
        Just oneQuarter ->                      --     oneQuarter <- halve oneHalf
          case halve oneQuarter of
            Nothing -> Nothing                  --     oneEighth  <- halve oneQuarter
            Just oneEighth ->                         
              Just oneEighth                    --     return oneEighth }
  • takeOneEighth é uma sequência de três passos halve encadeados.
  • Se uma etapa halve falhar, queremos que toda a composição takeOneEighth falhe.
  • Se uma etapa halve for bem-sucedida, queremos canalizar seu resultado para frente.

instance Monad Maybe where
  -- (>>=) :: Maybe a -> (a -> Maybe b) -> Maybe b
  Nothing >>= f  = Nothing                            -- infixl 1 >>=
  Just x  >>= f  = Just (f x)                         -- also, f =<< m = m >>= f
  
  -- return :: a -> Maybe a
  return x       = Just x

e agora podemos escrever:

takeOneEighth :: Int -> Maybe Int
takeOneEighth x = halve x >>= halve >>= halve             -- or,
    -- return x >>= halve >>= halve >>= halve             -- which is parsed as
    -- (((return x) >>= halve) >>= halve) >>= halve       -- which can also be written as
    -- (halve =<<) . (halve =<<) . (halve =<<) $ return x    -- or, equivalently, as
    --  halve <=<     halve <=<     halve      $        x

Composição Kleisli <=< é definido como (g <=< f) x = g =<< f x, ou equivalentemente como (f >=> g) x = f x >>= g. Com ele a definição acima se torna apenas

takeOneEighth :: Int -> Maybe Int
takeOneEighth = halve <=< halve <=< halve               -- infixr 1 <=<
        -- or, equivalently,                    
        --      halve >=> halve >=> halve               -- infixr 1 >=>

Existem três leis de mônada que devem ser obedecidas por toda mônada, ou seja, todo tipo que é uma instância da classe de tipos Monad:

1.  return x >>= f  =  f x
2.    m >>= return  =  m
3. (m >>= g) >>= h  =  m >>= (\y -> g y >>= h)

onde m é uma mônada, f tem o tipo a -> m b e g tem o tipo b -> m c.

Ou equivalentemente, usando o operador de composição >=> Kleisli definido acima:

1.    return >=> g  =  g                    -- do { y <- return x ; g y } == g x
2.    f >=> return  =  f                    -- do { y <- f x ; return y } == f x
3. (f >=> g) >=> h  =  f >=> (g >=> h)      -- do { z <- do { y <- f x; g y } ; h z }
                                            --  == do { y <- f x ; do { z <- g y; h z } }

Obedecer a essas leis torna muito mais fácil raciocinar sobre a mônada, pois garante que o uso de funções monádicas e sua composição se comportem de maneira razoável, semelhante a outras mônadas.

Vamos verificar se a mônada Talvez obedece às três leis da mônada.

  1. A lei da identidade à esquerda - return x >>= f = f x

    return z »= f = (Just z) »= f = f z

  2. A lei de identidade certa - m >>= return = m

  • Construtor de dados Just

    Just z »= return = return z = Just z

  • Construtor de dados Nada

    Nothing »= return = Nothing

  1. A lei da associatividade - (m >>= f) >>= g = m >>= (\x -> f x >>= g)

  • Construtor de dados Just

    – Left-hand side ((Just z) »= f) »= g = f z »= g

    – Right-hand side (Just z) »= (\x -> f x »= g) (\x -> f x »= g) z = f z »= g

  • Construtor de dados Nada

    – Left-hand side (Nothing »= f) »= g = Nothing »= g = Nothing

    – Right-hand side Nothing »= (\x -> f x »= g) = Nothing

eu mônada #

Não há como obter um valor do tipo a de uma expressão do tipo IO a e não deveria haver. Esta é realmente uma grande parte do motivo pelo qual as mônadas são usadas para modelar ‘IO’.

Uma expressão do tipo IO a pode ser pensada como representando uma ação que pode interagir com o mundo real e, se executada, resultaria em algo do tipo a. Por exemplo, a função getLine :: IO String do prelúdio não significa que abaixo de getLine haja alguma string específica que eu possa extrair - significa que getLine representa a ação de obter uma linha da entrada padrão .

Não surpreendentemente, main :: IO() já que um programa Haskell representa uma computação/ação que interage com o mundo real.

As coisas que você pode fazer com expressões do tipo IO a porque IO é uma mônada:

  • Seqüencie duas ações usando (>>) para produzir uma nova ação que executa a primeira ação, descarta qualquer valor que ela produziu e então executa a segunda ação.

       -- print the lines "Hello" then "World" to stdout
   putStrLn "Hello" >> putStrLn "World"

  • Às vezes você não quer descartar o valor que foi produzido na primeira ação - na verdade você gostaria que ele fosse alimentado em uma segunda ação. Para isso, temos >>=. Para IO, tem o tipo (>>=) :: IO a -> (a -> IO b) -> IO b.

      -- get a line from stdin and print it back out
  getLine >>= putStrLn

  • Pegue um valor normal e converta-o em uma ação que retorna imediatamente o valor que você deu. Esta função é menos obviamente útil até que você comece a usar a notação do.

      -- make an action that just returns 5
  return 5

Mais do Haskell Wiki no IO monad aqui.

Lista Mônada #

As listas formam uma mônada. Eles têm uma instanciação de mônada equivalente a esta:

instance Monad [] where 
  return x = [x]
  xs >>= f = concat (map f xs)

Podemos usá-los para emular o não-determinismo em nossos cálculos. Quando usamos xs >>= f, a função f :: a -> [b] é mapeada sobre a lista xs, obtendo uma lista de listas de resultados de cada aplicação de f sobre cada elemento de xs, e todas as listas de resultados são então concatenadas em uma lista de todos os resultados. Como exemplo, calculamos uma soma de dois números não determinísticos usando notação, a soma sendo representada por uma lista de somas de todos os pares de inteiros de duas listas, cada lista representando todos os valores possíveis de um número não-determinístico. número determinístico:

sumnd xs ys = do
  x <- xs
  y <- ys
  return (x + y)

Ou equivalentemente, usando liftM2 em Control.Monad:

sumnd = liftM2 (+)

nós obtemos:

> sumnd [1,2,3] [0,10]
[1,11,2,12,3,13]

notação #

A notação do é o açúcar sintático para mônadas. Aqui estão as regras:

<pré> faça x <- mx faça x <- mx y <- my é equivalente a fazer y <- my … … <pré> deixe a = b deixe a = b em … é equivalente a fazer … <pré> faça m m » ( e é equivalente a e) <pré> faça x <- m m »= (\x -> e é equivalente a e) <pré> faça m é equivalente a m

Por exemplo, essas definições são equivalentes:

example :: IO Integer
example =
  putStrLn "What's your name?" >> (
    getLine >>= (\name ->
      putStrLn ("Hello, " ++ name ++ ".") >> (
        putStrLn "What should we return?" >> (
          getLine >>= (\line ->
            let n = (read line :: Integer) in
              return (n + n))))))

example :: IO Integer
example = do
  putStrLn "What's your name?"
  name <- getLine
  putStrLn ("Hello, " ++ name ++ ".")
  putStrLn "What should we return?"
  line <- getLine
  let n = (read line :: Integer)
  return (n + n)
