Registradores de 16 bits #

Quando a Intel definiu o 8086 original, era um processador de 16 bits com um barramento de endereços de 20 bits (veja abaixo). Eles definiram 8 registradores de 16 bits de uso geral - mas deram a eles funções específicas para determinadas instruções:

  • AX O registrador do acumulador. Muitos opcodes assumiram esse registro ou seriam mais rápidos se ele fosse especificado.
  • DX O registro de dados. Isso às vezes era combinado como os 16 bits altos de um valor de 32 bits com AX - por exemplo, como resultado de uma multiplicação.
  • CX O registro de contagem. Isso foi usado em várias instruções orientadas a loops como o contador implícito para esses loops - por exemplo, LOOPNE (loop se não for igual) e REP (mover/comparar repetidamente)
  • BX O registro base. Isso pode ser usado para indexar a base de uma estrutura na memória - nenhum dos registros acima pode ser usado para indexar diretamente na memória.
  • SI O registro Source Index. Esse era o índice de origem implícito na memória para determinadas operações de movimentação e comparação.
  • DI O registrador de índice de destino. Esse era o índice de destino implícito na memória para determinadas operações de movimentação e comparação.
  • SP O registrador Stack Pointer. Este é o registrador menos geral do conjunto! Ele apontava para a posição atual na pilha, que era usada explicitamente para operações PUSH e POP, implicitamente para CALL e RET com sub-rotinas, e MUITO implicitamente durante interrupções. Como tal, usá-lo para qualquer outra coisa era perigoso para o seu programa!
  • BP O registrador do Ponteiro Base. Quando as sub-rotinas chamam outras sub-rotinas, a pilha contém vários “quadros de pilha”. BP pode ser usado para manter o quadro de pilha atual, e então quando uma nova sub-rotina é chamada, ela pode ser salva na pilha, o novo quadro de pilha criado e usado e, ao retornar da sub-rotina interna, o valor do quadro de pilha antigo pode ser ser restaurado.

Notas: #

  1. Os três primeiros registros não podem ser usados ​​para indexação na memória.

  2. BX, SI e DI por padrão no segmento de dados atual (veja abaixo).

     MOV    AX, [BX+5]     ; Point into Data Segment
 MOV    AX, ES:[DI+5]  ; Override into Extra Segment

  3. DI, quando usado em operações de memória para memória como MOVS e CMPS, usa apenas o Segmento Extra (veja abaixo). Isso não pode ser substituído.

  4. SP e BP usam o Stack Segment (veja abaixo) por padrão.

registradores de 32 bits #

Quando a Intel produziu o 80386, eles atualizaram de um processador de 16 bits para um de 32 bits. O processamento de 32 bits significa duas coisas: tanto os dados manipulados eram de 32 bits quanto os endereços de memória acessados ​​eram de 32 bits. Para fazer isso, mas ainda mantendo a compatibilidade com seus processadores anteriores, eles introduziram modos totalmente novos para o processador. Estava no modo de 16 bits ou no modo de 32 bits - mas você pode substituir esse modo instrução por instrução para dados, endereçamento ou ambos!

Em primeiro lugar, eles tiveram que definir registradores de 32 bits. Eles fizeram isso simplesmente estendendo os oito existentes de 16 bits para 32 bits e dando-lhes nomes “estendidos” com um prefixo E: EAX, EBX, ECX, EDX, ESI, EDI, EBP e ESP. Os 16 bits inferiores desses registradores eram os mesmos de antes, mas as metades superiores dos registradores estavam disponíveis para operações de 32 bits, como ADD e CMP. As metades superiores não eram acessíveis separadamente, como acontecia com os registradores de 8 bits.

O processador tinha que ter modos separados de 16 bits e 32 bits porque a Intel usava os mesmos opcodes para muitas das operações: CMP AX,DX no modo de 16 bits e CMP EAX,EDX no modo de 32 bits tinha exatamente os mesmos opcodes! Isso significava que o mesmo código NÃO poderia ser executado em nenhum dos modos:

O opcode para “Mover imediato para AX” é 0xB8, seguido por dois bytes do valor imediato: 0xB8 0x12 0x34

O opcode para “Mover imediato para EAX” é 0xB8, seguido por quatro bytes do valor imediato: 0xB8 0x12 0x34 0x56 0x78

Então o montador tem que saber em que modo o processador está ao executar o código, para que ele saiba emitir o número correto de bytes.

Registradores de 8 bits #

Os primeiros quatro [registradores de 16 bits](https://www.wikiod.com/pt/x86/registrar-fundamentos#Registros de 16 bits) podem ter seus bytes de metade superior e inferior acessados ​​diretamente como seus registros próprios:

  • AH e AL são as metades alta e baixa do registro AX.
  • BH e BL são as metades Alta e Baixa do registro BX.
  • CH e CL são as metades alta e baixa do registro CX.
  • DH e DL são as metades Alta e Baixa do registro DX.

Observe que isso significa que alterar AH ou AL imediatamente alterará AX também! Observe também que qualquer operação em um registrador de 8 bits não pode afetar seu “parceiro” - incrementar AL de modo que ele transborde de 0xFF para 0x00 não alteraria AH.

https://www.wikiod.com/pt/x86/registrar-fundamentos#registradores de 64 bits também tem versões de 8 bits representando seus bytes inferiores:

  • SIL para RSI
  • DIL para RDI
  • BPL para RBP
  • SPL para RSP

O mesmo se aplica aos registradores R8 a R15: suas respectivas partes de bytes inferiores são denominadas R8B – R15B.

Registros de segmento #

Segmentação #

Quando a Intel estava projetando o 8086 original, já havia vários processadores de 8 bits que tinham recursos de 16 bits - mas eles queriam produzir um verdadeiro processador de 16 bits. Eles também queriam produzir algo melhor e mais capaz do que o que já existia, então eles queriam poder acessar mais do que o máximo de 65.536 bytes de memória implícitos em registradores de endereçamento de 16 bits.

Registros de Segmentos Originais #

Então eles implementaram a ideia de “Segmentos” - um bloco de memória de 64 kilobytes indexado pelos registradores de endereço de 16 bits - que poderia ser re-baseado para endereçar diferentes áreas da memória total. Para manter essas bases de segmento, eles incluíram registradores de segmento:

  • CS O registro do segmento de código. Este contém o segmento do código que está sendo executado no momento, indexado pelo registro implícito IP (Instruction Pointer).
  • DS O registro do segmento de dados. Isso mantém o segmento padrão para dados sendo manipulados pelo programa.
  • ES O registrador Extra Segment. Isso contém um segundo segmento de dados, para operações de dados simultâneas em toda a memória.
  • SS O registrador de Segmento de Pilha. Isso contém o segmento de memória que contém a pilha atual.

Tamanho do segmento? #

Os registradores de segmento podem ser de qualquer tamanho, mas torná-los com 16 bits de largura facilitou a interoperabilidade com os outros registradores. A próxima pergunta era: os segmentos deveriam se sobrepor e, em caso afirmativo, quanto? A resposta a essa pergunta ditaria o tamanho total da memória que poderia ser acessada.

Se não houvesse nenhuma sobreposição, então o espaço de endereçamento seria de 32 bits - 4 gigabytes - um tamanho totalmente inédito na época! Uma sobreposição mais “natural” de 8 bits produziria um espaço de endereço de 24 bits, ou 16 megabytes. No final, a Intel decidiu salvar mais quatro pinos de endereço no processador, tornando o espaço de endereço de 1 megabyte com uma sobreposição de 12 bits - eles consideraram isso suficientemente grande para a época!

Mais Registros de Segmento! #

Quando a Intel estava projetando o 80386, eles reconheceram que o conjunto existente de registradores de 4 segmentos não era suficiente para a complexidade dos programas que eles queriam que ele suportasse. Então eles adicionaram mais dois:

  • FS O registro do segmento Far
  • GS O registro do segmento global

Esses novos registradores de segmento não tinham nenhum uso imposto pelo processador: eles estavam apenas disponíveis para o que o programador quisesse.

Alguns dizem que os nomes foram escolhidos para simplesmente continuar o tema C, D, E do conjunto existente…

registradores de 64 bits #

A AMD é uma fabricante de processadores que licenciou o design do 80386 da Intel para produzir versões compatíveis - mas concorrentes. Eles fizeram alterações internas no projeto para melhorar o rendimento ou outras melhorias no projeto, enquanto ainda conseguiam executar os mesmos programas.

Para superar a Intel, eles criaram extensões de 64 bits para o design de 32 bits da Intel e produziram o primeiro chip de 64 bits que ainda podia executar código x86 de 32 bits. A Intel acabou seguindo o design da AMD em suas versões da arquitetura de 64 bits.

O design de 64 bits fez várias alterações no conjunto de registradores, enquanto ainda era compatível com versões anteriores:

  • Os registradores de uso geral existentes foram estendidos para 64 bits e nomeados com um prefixo R: RAX, RBX, RCX, RDX, RSI, RDI, RBP, e RSP.

Novamente, as metades inferiores desses registradores eram os mesmos registradores com prefixo E de antes, e as metades superiores não podiam ser acessadas independentemente.

  • Mais 8 registradores de 64 bits foram adicionados, e não nomeados, mas meramente numerados: R8, R9, R10, R11, R12, R13, R14 e R15.
  • A metade baixa de 32 bits desses registradores são R8D a R15D (D para DWORD como de costume).
  • Os 16 bits mais baixos desses registradores podem ser acessados ​​com um sufixo W no nome do registrador: R8W a R15W.
  • Os 8 bits mais baixos de todos os 16 registradores agora podem ser acessados:
  • Os tradicionais AL, BL, CL e DL;
  • Os bytes baixos dos registradores de ponteiro (tradicionalmente): SIL, DIL, BPL e SPL;
  • E os bytes baixos dos 8 novos registradores: R8B a R15B.
  • No entanto, AH, BH, CH e DH são inacessíveis em instruções que usam um prefixo REX (para tamanho de operando de 64 bits, ou para acessar R8-R15, ou para acessar SIL, DIL , BPL ou SPL). Com um prefixo REX, o padrão de bits do código de máquina que costumava significar AH significa SPL e assim por diante. Consulte a Tabela 3-1 do manual de referência de instruções da Intel (volume 2).

Gravar em um registrador de 32 bits sempre zera os 32 bits superiores do registrador de largura total, ao contrário de gravar em um registrador de 8 ou 16 bits (que se funde com o valor antigo, que é uma dependência extra para execução fora de ordem ).

Registro de bandeiras #

Quando a Unidade Lógica Aritmética x86 (ALU) executa operações como NOT e ADD, ela sinaliza os resultados dessas operações (“tornou-se zero”, “transbordou”, “tornou-se negativo”) em um FLAGS especial de 16 bits registrar. Processadores de 32 bits atualizaram para 32 bits e chamaram de EFLAGS, enquanto processadores de 64 bits atualizaram para 64 bits e chamaram de RFLAGS.

Códigos de condição #

Mas não importa o nome, o registro não é acessível diretamente (exceto por algumas instruções - veja abaixo). Em vez disso, sinalizadores individuais são referenciados em certas instruções, como Jump condicional ou Set condicional, conhecido como Jcc e SETcc, onde cc significa “código de condição” e faz referência à tabela a seguir:

Código de condição Nome Definição
E, Z Igual, Zero ZF == 1
NE, NZ Não igual, não zero ZF == 0
‘O’ Estouro DE == 1
NÃO Sem estouro DE == 0
S Assinado SF == 1
NS Não assinado SF == 0
P Paridade PF == 1
NP Sem paridade PF == 0
————– —- ———-
C, B, NAE Carregar, Abaixo, Não Acima ou Igual CF == 1
NC, NB, AE Sem Carry, Nem Abaixo, Acima ou Igual CF == 0
A, NBE Acima, não abaixo ou igual CF==0 e ZF==0
NA, BE Não Acima, Abaixo ou Igual CF==1 ou ZF==1
————— —- ———-
GE, NL Maior ou igual, não menor SF==DE
NGE, L Não Maior ou Igual, Menor SF!=DE
G, NLE Maior, não menor ou igual ZF==0 e SF==OF
NG, LE Não Maior, Menor ou Igual ZF==1 ou SF!=OF

Em 16 bits, subtrair 1 de 0 é 65.535 ou -1 dependendo se a aritmética sem sinal ou com sinal é usada - mas o destino mantém 0xFFFF de qualquer maneira. É apenas interpretando os códigos de condição que o significado fica claro. É ainda mais revelador se 1 for subtraído de 0x8000: em aritmética sem sinal, isso apenas muda 32.768 para 32.767; enquanto na aritmética assinada ele muda -32.768 para 32.767 - um estouro muito mais notável!

Os códigos de condição são agrupados em três blocos na tabela: sinal-irrelevante, não assinado e assinado. A nomeação dentro dos dois últimos blocos usa “Acima” e “Abaixo” para não assinado e “Maior” ou “Menos” para assinado. Então JB seria “Jump if Below” (sem sinal), enquanto JL seria “Jump if Less” (assinado).

Acessando FLAGS diretamente #

Os códigos de condição acima são úteis para interpretar conceitos predefinidos, mas os bits de flag reais também estão disponíveis diretamente com as duas instruções a seguir:

  • LAHF Carrega o registro AH com Flags
  • SAHF Armazena o registro AH em Flags

Apenas alguns sinalizadores são copiados com essas instruções. Todo o registro FLAGS / EFLAGS / RFLAGS pode ser salvo ou restaurado na pilha:

  • PUSHF / POPF Empurre/desloque FLAGS de 16 bits para/da pilha
  • PUSHFD / POPFD Empurre/desloque EFLAGS de 32 bits para/da pilha
  • PUSHFQ / POPFQ Empurre/desloque RFLAGS de 64 bits para/da pilha

Observe que as interrupções salvam e restauram o registro [R/E]FLAGS atual automaticamente.

Outras bandeiras #

Assim como os sinalizadores da ALU descritos acima, o registrador FLAGS define outros sinalizadores de estado do sistema:

  • IF O sinalizador de interrupção. Isso é definido com a instrução STI para habilitar globalmente as interrupções, e limpo com a instrução CLI para desabilitar globalmente as interrupções.
  • DF A bandeira de direção. Operações de memória para memória como CMPS e MOVS (para comparar e mover entre locais de memória) incrementam ou decrementam automaticamente os registradores de índice como parte da instrução. O sinalizador DF determina o que acontece: se limpo com a instrução CLD, eles são incrementados; se definido com a instrução STD, eles são decrementados.
  • TF A Bandeira Armadilha. Este é um sinalizador de depuração. Defini-lo colocará o processador no modo “single-step”: depois que cada instrução for executada, ele chamará o “Single Step Interrupt Handler”, que deve ser tratado por um depurador. Não há instruções para definir ou limpar este sinalizador: você precisa manipular o bit enquanto ele está na memória.

80286 Bandeiras #

Para suportar os novos recursos multitarefa no 80286, a Intel adicionou sinalizadores extras ao registro FLAGS:

  • IOPL O nível de privilégio de E/S. Para proteger o código multitarefa, algumas tarefas precisavam de privilégios para acessar as portas de E/S, enquanto outras precisavam ser impedidas de acessá-las. A Intel introduziu uma escala de privilégios de quatro níveis, com 002 sendo o mais privilegiado e 112 o menos. Se IOPL for menor que o nível de privilégio atual, qualquer tentativa de acessar portas de E/S ou habilitar ou desabilitar interrupções causará uma falha de proteção geral.
  • Sinalizador de tarefa aninhada NT. Este sinalizador era definido se uma Tarefa ‘CALL’ chamava outra Tarefa, o que causava uma troca de contexto. O sinalizador set disse ao processador para fazer uma troca de contexto de volta quando o RET foi executado.

80386 Bandeiras #

O ‘386 precisava de sinalizadores extras para oferecer suporte a recursos extras projetados no processador.

  • RF O Sinalizador de Currículo. O `386 adicionou registradores de depuração, que podem invocar o depurador em vários acessos de hardware como leitura, escrita ou execução de um determinado local de memória. No entanto, quando o manipulador de depuração retornasse para executar a instrução o acesso imediatamente chamaria novamente o manipulador de depuração! Ou pelo menos o faria se não fosse pelo sinalizador Resume, que é definido automaticamente na entrada no manipulador de depuração, e automaticamente apagado após cada instrução. Se o sinalizador de currículo estiver definido, o manipulador de depuração não será invocado.
  • VM O sinalizador virtual 8086. Para suportar código de 16 bits mais antigo, bem como código de 32 bits mais recente, o 80386 pode executar tarefas de 16 bits no modo “Virtual 8086”, com a ajuda de um executivo do Virtual 8086. O sinalizador VM indicava que esta tarefa era uma tarefa 8086 virtual.

80486 Bandeiras #

À medida que a arquitetura Intel melhorou, ficou mais rápida por meio de tecnologias como caches e execução superescalar. Isso teve que otimizar o acesso ao sistema fazendo suposições. Para controlar essas suposições, mais sinalizadores eram necessários:

  • Sinalizador de verificação de alinhamento AC A arquitetura x86 sempre pode acessar valores de memória multibyte em qualquer limite de byte, ao contrário de algumas arquiteturas que exigiam que eles fossem alinhados por tamanho (valores de 4 bytes precisavam estar em limites de 4 bytes). No entanto, era menos eficiente fazer isso, pois vários acessos à memória eram necessários para acessar dados desalinhados. Se o sinalizador AC foi definido, um acesso desalinhado geraria uma exceção em vez de executar o código. Dessa forma, o código pode ser melhorado durante o desenvolvimento com o conjunto AC, mas desativado para o código de produção.

Bandeiras Pentium #

O Pentium adicionou mais suporte para virtualização, além de suporte para a instrução CPUID:

  • VIF O sinalizador de interrupção virtual. Esta é uma cópia virtual do IF desta Tarefa - se esta Tarefa deseja ou não desabilitar as interrupções, sem realmente afetar as Interrupções Globais.
  • VIP O sinalizador de interrupção virtual pendente. Isso indica que uma interrupção foi virtualmente bloqueada por VIF, então quando a tarefa faz um STI uma interrupção virtual pode ser gerada para ela.
  • ID O sinalizador permitido para CPUID. Se deve ou não permitir que esta tarefa execute a instrução CPUID. Um monitor virtual pode não permitir isso e “mentir” para a tarefa solicitante se executar a instrução.
