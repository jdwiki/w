Exemplo x86 Linux Hello World #

Este é um programa básico Hello World em assembly NASM para Linux x86 de 32 bits, usando chamadas de sistema diretamente (sem nenhuma chamada de função libc). É muito para absorver, mas com o tempo se tornará compreensível. Linhas que começam com ponto e vírgula(;) são comentários.

Se você ainda não conhece programação de sistemas Unix de baixo nível, talvez queira apenas escrever funções em asm e chamá-las de programas C ou C++. Então você pode apenas se preocupar em aprender como lidar com registradores e memória, sem também aprender a API de chamada de sistema POSIX e a ABI para usá-la.

Isso faz duas chamadas de sistema: write(2) e _exit(2) (não o exit(3) wrapper libc que libera buffers stdio e em breve). (Tecnicamente, _exit() chama sys_exit_group, não sys_exit, mas isso apenas importa em um processo multi-threaded.) Veja também syscalls(2) para documentação sobre chamadas de sistema em geral, e a diferença entre fazê-los diretamente versus usar as funções do libc wrapper.

Em resumo, as chamadas do sistema são feitas colocando os argumentos nos registradores apropriados, e o número de chamada do sistema em eax, depois executando uma instrução int 0x80. Consulte também Quais são os valores de retorno das chamadas do sistema em Assembly? para obter mais explicações sobre como a interface asm syscall é documentada principalmente com sintaxe C.

Os números de chamada syscall para a ABI de 32 bits estão em /usr/include/i386-linux-gnu/asm/unistd_32.h (mesmo conteúdo em /usr/include/x86_64-linux-gnu/asm/unistd_32. h).

#include <sys/syscall.h> irá incluir o arquivo correto, então você pode executar echo '#include <sys/syscall.h>' | gcc -E - -dM | less para ver os defs da macro (veja esta resposta para saber mais sobre como encontrar constantes para asm em cabeçalhos C)

section .text             ; Executable code goes in the .text section
global _start             ; The linker looks for this symbol to set the process entry point, so execution start here
;;;a name followed by a colon defines a symbol.  The global _start directive modifies it so it's a global symbol, not just one that we can CALL or JMP to from inside the asm.
;;; note that _start isn't really a "function".  You can't return from it, and the kernel passes argc, argv, and env differently than main() would expect.
 _start:
    ;;; write(1, msg, len);
    ; Start by moving the arguments into registers, where the kernel will look for them
    mov     edx,len       ; 3rd arg goes in edx: buffer length
    mov     ecx,msg       ; 2nd arg goes in ecx: pointer to the buffer
    ;Set output to stdout (goes to your terminal, or wherever you redirect or pipe)
    mov     ebx,1         ; 1st arg goes in ebx: Unix file descriptor. 1 = stdout, which is normally connected to the terminal.

    mov     eax,4         ; system call number (from SYS_write / __NR_write from unistd_32.h).
    int     0x80          ; generate an interrupt, activating the kernel's system-call handling code.  64-bit code uses a different instruction, different registers, and different call numbers.
    ;; eax = return value, all other registers unchanged.

    ;;;Second, exit the process.  There's nothing to return to, so we can't use a ret instruction (like we could if this was main() or any function with a caller)
    ;;; If we don't exit, execution continues into whatever bytes are next in the memory page,
    ;;; typically leading to a segmentation fault because the padding 00 00 decodes to  add [eax],al.

    ;;; _exit(0);
    xor     ebx,ebx       ; first arg = exit status = 0.  (will be truncated to 8 bits).  Zeroing registers is a special case on x86, and mov ebx,0 would be less efficient.
                      ;; leaving out the zeroing of ebx would mean we exit(1), i.e. with an error status, since ebx still holds 1 from earlier.
    mov     eax,1         ; put __NR_exit into eax
    int     0x80          ;Execute the Linux function

section     .rodata       ; Section for read-only constants

             ;; msg is a label, and in this context doesn't need to be msg:.  It could be on a separate line.
             ;; db = Data Bytes: assemble some literal bytes into the output file.
msg     db  'Hello, world!',0xa     ; ASCII string constant plus a newline (0x10)

             ;;  No terminating zero byte is needed, because we're using write(), which takes a buffer + length instead of an implicit-length string.
             ;; To make this a C string that we could pass to puts or strlen, we'd need a terminating 0 byte. (e.g. "...", 0x10, 0)

len     equ $ - msg       ; Define an assemble-time constant (not stored by itself in the output file, but will appear as an immediate operand in insns that use it)
                          ; Calculate len = string length.  subtract the address of the start
                          ; of the string from the current position ($)
  ;; equivalently, we could have put a str_end: label after the string and done   len equ str_end - str

No Linux, você pode salvar este arquivo como Hello.asm e construir um executável de 32 bits com estes comandos:

nasm -felf32 Hello.asm                  # assemble as 32-bit code.  Add -Worphan-labels -g -Fdwarf  for debug symbols and warnings
gcc -nostdlib -m32 Hello.o -o Hello     # link without CRT startup code or libc, making a static binary

Veja esta resposta para mais detalhes sobre como construir assembly em executáveis ​​Linux estáticos ou dinamicamente vinculados de 32 ou 64 bits, para sintaxe NASM/YASM ou sintaxe GNU AT&T com diretivas GNU as. (Ponto chave: certifique-se de usar -m32 ou equivalente ao construir código de 32 bits em um host de 64 bits, ou você terá problemas confusos em tempo de execução.)

Você pode rastrear sua execução com strace para ver as chamadas de sistema que ele faz:

$ strace ./Hello 
execve("./Hello", ["./Hello"], [/* 72 vars */]) = 0
[ Process PID=4019 runs in 32 bit mode. ]
write(1, "Hello, world!\n", 14Hello, world!
)         = 14
_exit(0)                                = ?
+++ exited with 0 +++

O rastreamento em stderr e a saída regular em stdout estão indo para o terminal aqui, então eles interferem na linha com a chamada do sistema write. Redirecione ou rastreie para um arquivo se você se importa. Observe como isso nos permite ver facilmente os valores de retorno de syscall sem ter que adicionar código para imprimi-los, e na verdade é ainda mais fácil do que usar um depurador comum (como gdb) para isso.

A versão x86-64 desse programa seria extremamente semelhante, passando os mesmos argumentos para as mesmas chamadas de sistema, apenas em registros diferentes. E usando a instrução syscall em vez de int 0x80.

Linguagem de montagem x86 #

A família de linguagens assembly x86 representa décadas de avanços na arquitetura Intel 8086 original. Além de haver vários dialetos diferentes baseados no montador usado, instruções de processador adicionais, registros e outros recursos foram adicionados ao longo dos anos, mantendo-se ainda compatível com o assembly de 16 bits usado na década de 1980.

O primeiro passo para trabalhar com o assembly x86 é determinar qual é o objetivo. Se você estiver procurando escrever código dentro de um sistema operacional, por exemplo, você desejará determinar adicionalmente se você optará por usar um montador autônomo ou recursos de montagem embutidos embutidos de uma linguagem de nível superior, como C. Se você deseja codificar no “bare metal” sem um sistema operacional, basta instalar o montador de sua escolha e entender como criar código binário que pode ser transformado em memória flash, imagem inicializável ou ser carregado na memória no local apropriado para iniciar a execução.

Um montador muito popular que é bem suportado em várias plataformas é o NASM (Netwide Assembler), que pode ser obtido em http://nasm.us/. No site do NASM, você pode fazer o download da versão mais recente para sua plataforma.

Janelas

As versões de 32 e 64 bits do NASM estão disponíveis para Windows. O NASM vem com um instalador conveniente que pode ser usado em seu host Windows para instalar o montador automaticamente.

Linux

Pode ser que o NASM já esteja instalado na sua versão do Linux. Para verificar, execute:

nasm -v

Se o comando não for encontrado, você precisará executar uma instalação. A menos que você esteja fazendo algo que exija recursos NASM de ponta, o melhor caminho é usar sua ferramenta de gerenciamento de pacotes integrada para sua distribuição Linux para instalar o NASM. Por exemplo, em sistemas derivados do Debian, como Ubuntu e outros, execute o seguinte em um prompt de comando:

sudo apt-get install nasm

Para sistemas baseados em RPM, você pode tentar:

sudo yum install nasm

Mac OS X

Versões recentes do OS X (incluindo Yosemite e El Capitan) vêm com uma versão mais antiga do NASM pré-instalada. Por exemplo, El Capitan tem a versão 0.98.40 instalada. Embora isso provavelmente funcione para quase todos os propósitos normais, na verdade é bastante antigo. Neste momento, a versão 2.11 do NASM foi lançada e a 2.12 tem vários candidatos a lançamento disponíveis.

Você pode obter o código fonte do NASM no link acima, mas a menos que você tenha uma necessidade específica de instalar a partir da fonte, é muito mais simples baixar o pacote binário do diretório de lançamento do OS X e descompactá-lo.

Uma vez descompactado, é altamente recomendável que você não sobrescreva a versão do NASM instalada no sistema. Em vez disso, você pode instalá-lo em /usr/local:

 $ sudo su
 <user's password entered to become root>
 # cd /usr/local/bin
 # cp <path/to/unzipped/nasm/files/nasm> ./
 # exit

Neste ponto, o NASM está em /usr/local/bin, mas não está no seu caminho. Agora você deve adicionar a seguinte linha ao final do seu perfil:

 $ echo 'export PATH=/usr/local/bin:$PATH' >> ~/.bash_profile

Isto irá preceder /usr/local/bin ao seu caminho. A execução de nasm -v no prompt de comando agora deve exibir a versão adequada e mais recente.
