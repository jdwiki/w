Modo real #

Quando a Intel projetou o x86 original, o 8086 (e o derivado 8088), eles incluíram a segmentação para permitir que o processador de 16 bits acessasse mais de 16 bits de endereço. Eles fizeram isso fazendo com que os endereços de 16 bits fossem relativos a um determinado Registrador de Segmento de 16 bits, dos quais eles definiram quatro: Segmento de Código (CS), Segmento de Dados (DS), Segmento Extra (ES) e Segmento de Pilha (SS).

A maioria das instruções implicava qual registrador de segmento usar: as instruções eram feitas do segmento de código, ‘PUSH’ e ‘POP’ implicavam o segmento de pilha, e referências de dados simples implicavam o segmento de dados - embora isso pudesse ser substituído para acessar a memória em qualquer um dos outros Segmentos.

A implementação era simples: para cada acesso à memória, a CPU pegava o registrador de segmento implícito (ou explícito), deslocava-o quatro lugares para a esquerda e adicionava o endereço indicado:

        +-------------------+---------+
Segment | 16-bit value      | 0 0 0 0 |
        +-------------------+---------+
                   PLUS
        +---------+-------------------+
Address | 0 0 0 0 | 16-bit value      |
        +---------+-------------------+
                  EQUALS
        +-----------------------------+
Result  |  20-bit memory address      |
        +-----------------------------+

Isso permitiu várias técnicas:

  • Permitindo que Código, Dados e Pilha sejam mutuamente acessíveis (CS, DS e SS todos tinham o mesmo valor);
  • Mantendo Código, Dados e Pilha completamente separados uns dos outros (CS, DS e SS todos 4K (ou mais) separados uns dos outros - lembre-se que é multiplicado por 16, então são 64K).

Também permitia sobreposições bizarras e todo tipo de coisas estranhas!

Quando o 80286 foi inventado, ele suportava este modo legado (agora chamado de “Modo Real”), mas adicionou um novo modo chamado “Modo Protegido” (q.v.).

As coisas importantes a serem observadas é que no modo real:

  • Qualquer endereço de memória era acessível, simplesmente colocando o valor correto dentro de um Registrador de Segmento e acessando o endereço de 16 bits;
  • A extensão da “proteção” era permitir que o programador separasse diferentes áreas de memória para diferentes propósitos e tornasse mais difícil escrever acidentalmente nos dados errados - enquanto ainda tornava possível fazê-lo.

Em outras palavras… não muito protegido!

Modo protegido #

Introdução #

Quando o 80286 foi inventado, ele suportava o legado 8086 Segmentation (agora chamado de “Modo Real”) e adicionou um novo modo chamado “Modo Protegido”. Esse modo está em todos os processadores x86 desde então, embora aprimorado com várias melhorias, como endereçamento de 32 e 64 bits.

Projeto #

No Modo Protegido, o simples “Adicionar endereço ao valor do Registro de Segmento Deslocado” foi eliminado completamente. Eles mantinham os Registradores de Segmento, mas ao invés de usá-los para calcular um endereço, usavam-nos para indexar em uma tabela (na verdade, uma de duas…) que definia o Segmento a ser acessado. Esta definição não apenas descrevia onde na memória o Segmento estava (usando Base e Limit), mas também qual tipo de Segmento ele era (Código, Dados, Pilha ou mesmo Sistema) e quais tipos de programas poderiam acessá-lo (OS Kernel, programa normal , Driver de dispositivo, etc.).

Registro de Segmento #

Cada registrador de segmento de 16 bits assumiu a seguinte forma:

+------------+-----+------+
| Desc Index | G/L | Priv |
+------------+-----+------+
 Desc Index = 13-bit index into a Descriptor Table (described below)
 G/L        = 1-bit flag for which Descriptor Table to Index: Global or Local
 Priv       = 2-bit field defining the Privilege level for access

Global / Local #

O bit Global/Local definia se o acesso era em uma Tabela Global de descritores (chamada, sem surpresa, de Tabela Global de Descritores, ou GDT), ou uma Tabela Descritora Local (LDT). A idéia para o LDT era que cada programa pudesse ter sua própria Tabela Descritora - o SO definiria um conjunto Global de Segmentos, e cada programa teria seu próprio conjunto de Código Local, Dados e Segmentos de Pilha. O SO gerenciaria a memória entre as diferentes Tabelas de Descritores.

Tabela de Descritores #

Cada Tabela de Descritores (Global ou Local) era uma matriz de 64 K de 8.192 Descritores: cada um um registro de 8 bytes que definia vários aspectos do Segmento que estava descrevendo. Os campos do Índice de Descritores dos Registradores de Segmento permitiam 8.192 descritores: não é coincidência!

Descritor #

Um Descritor continha as seguintes informações - observe que o formato do Descritor mudou à medida que novos processadores foram lançados, mas o mesmo tipo de informação foi mantido em cada um:

  • Base Isso definiu o endereço inicial do segmento de memória.
  • Limite Isso definiu o tamanho do segmento de memória - mais ou menos. Eles tiveram que tomar uma decisão: um tamanho de ‘0x0000’ significaria um tamanho de ‘0’, então não acessível? Ou tamanho máximo? Em vez disso, eles escolheram uma terceira opção: o campo Limite era o último local endereçável dentro do Segmento. Isso significava que um segmento de um bye poderia ser definido; ou um tamanho máximo para o tamanho do endereço.
  • Modelo Havia vários tipos de Segmentos: o tradicional Código, Dados e Pilha (veja abaixo), mas outros Segmentos de Sistema também foram definidos:
  • Os Segmentos da Tabela de Descritores Locais definiram quantos Descritores Locais poderiam ser acessados;
  • Segmentos de estado de tarefa podem ser usados ​​para comutação de contexto gerenciada por hardware;
  • “Call Gates” controlados que podem permitir que programas chamem o sistema operacional - mas apenas por meio de pontos de entrada cuidadosamente gerenciados.
  • Atributos Alguns atributos do Segmento também foram mantidos, quando relevantes:
  • Somente leitura vs leitura-gravação;
  • Se o segmento estava presente ou não - permitindo o gerenciamento de memória sob demanda;
  • Qual nível de código (SO vs Driver vs programa) poderia acessar este Segmento.

Verdadeira proteção finalmente! #

Se o sistema operacional mantivesse as Tabelas de Descritores em Segmentos que não pudessem ser acessados ​​por meros programas, ele poderia gerenciar com precisão quais Segmentos foram definidos e qual memória foi atribuída e acessível a cada um. Um programa poderia fabricar qualquer valor de Registro de Segmento que quisesse - mas se tivesse a audácia de realmente _carregá-lo em um Registro de Segmento!… o hardware da CPU reconheceria que o valor do Descritor proposto quebrou qualquer uma de um grande número de regras, e em vez de concluir a solicitação, ele geraria uma exceção para o sistema operacional para permitir que ele manipule o programa errante.

Essa exceção geralmente era a nº 13, a exceção de proteção geral - tornada mundialmente famosa pelo Microsoft Windows… (Alguém acha que um engenheiro da Intel era supersticioso?)

Erros #

Os tipos de erros que podem ocorrer incluem:

  • Se o Índice Descritor proposto for maior que o tamanho da tabela;
  • Se o Descritor proposto for um Descritor de Sistema ao invés de Código, Dados ou Pilha;
  • Se o Descritor proposto for mais privilegiado que o programa solicitante;
  • Se o descritor proposto foi marcado como não legível (como um segmento de código), mas tentou ser lido em vez de executado;
  • Se o Descritor proposto foi marcado como Não Presente.

Observe que o último pode não ser um problema fatal para o programa: o sistema operacional pode anotar o sinalizador, restabelecer o segmento, marcá-lo como agora Presente e permitir que a instrução com falha prossiga com sucesso.

Ou talvez o Descritor tenha sido carregado com sucesso em um Registrador de Segmento, mas um acesso futuro com ele quebrou uma das várias regras:

  • O Registro de Segmento foi carregado com o Índice Descritor 0x0000 para o GDT. Isso foi reservado pelo hardware como NULL;
  • Se o descritor carregado foi marcado como somente leitura, mas foi tentada uma gravação nele.
  • Se alguma parte do acesso (1, 2, 4 ou mais bytes) estava fora do Limite do Segmento.

Mudando para o modo protegido #

Mudar para o modo protegido é fácil: você só precisa definir um único bit em um registro de controle. Mas ficar no Modo Protegido, sem que a CPU levante as mãos e se reinicie por não saber o que fazer a seguir, exige muita preparação.

Resumidamente, os passos necessários são os seguintes:

  • Uma área de memória para a Tabela Global de Descritores precisa ser configurada para definir um mínimo de três Descritores:
  1. O zero, Descritor NULL;
  2. Outro Descritor para um Segmento de Código;
  3. Outro Descritor para um Segmento de Dados.

    This can be used for both Data and Stack.

  • O Global Descriptor Table Register (GDTR) precisa ser inicializado para apontar para esta área de memória definida;

     GDT_Ptr    dw      SIZE GDT
            dd      OFFSET GDT

            ...

            lgdt    [GDT_Ptr]

  • O bit PM em CR0 precisa ser definido:

         mov   eax, cr0      ; Get CR0 into register
     or    eax, 0x01     ; Set the Protected Mode bit
     mov   cr0, eax      ; We're now in Protected Mode!

  • Os registradores de segmento precisam ser carregados do GDT para remover os valores atuais do modo real:

         jmp   0x0008:NowInPM  ; This is a FAR Jump. 0x0008 is the Code Descriptor

NowInPM:
     mov   ax, 0x0010      ; This is the Data Descriptor
     mov   ds, ax
     mov   es, ax
     mov   ss, ax
     mov   sp, 0x0000      ; Top of stack!

Observe que este é o mínimo nu, apenas para colocar a CPU no Modo Protegido. Para realmente preparar todo o sistema pode exigir muitas outras etapas. Por exemplo:

  • As áreas de memória superiores podem ter que ser habilitadas - desligando a porta A20;
  • As interrupções definitivamente devem ser desabilitadas - mas talvez os vários manipuladores de falhas possam ser configurados antes de entrar no modo protegido, para permitir erros no início do processamento.

O autor original desta seção escreveu um tutorial inteiro sobre como entrar no Modo Protegido e trabalhar com ele.

Modo irreal #

O modo irreal explora dois fatos sobre como os processadores Intel e AMD carregam e salvam as informações para descrever um segmento.

  1. O processador armazena em cache as informações do descritor obtidas durante um movimento em um registrador seletor em modo protegido. Essas informações são armazenadas em uma parte arquitetônica invisível do próprio registro seletor.

  2. No modo real, os registradores seletores são chamados de registradores de segmento, mas, fora isso, eles designam o mesmo conjunto de registradores e, como tal, também possuem uma parte invisível. Essas partes são preenchidas com valores fixos, mas para a base que é derivada do valor recém carregado.

Nessa visão, o modo real é apenas um caso especial de modo protegido: onde as informações de um segmento, como base e limite, são buscadas sem um GDT/LDT, mas ainda lidas da parte oculta do registrador de segmento.

Ao alternar em modo protegido e elaborar um GDT é possível criar um segmento com os atributos desejados, por exemplo, uma base de 0 e um limite de 4GiB. Através de um carregamento sucessivo de um registrador seletor tais atributos são armazenados em cache, sendo então possível retornar ao modo real e ter um registrador de segmento através do qual acessa todo o espaço de endereçamento de 32 bits.

BITS 16

jmp 7c0h:__START__

__START__:
 push cs
 pop ds
 push ds
 pop ss
 xor sp, sp

 
 lgdt [GDT]            ;Set the GDTR register
 

 cli                ;We don't have an IDT set, we can't handle interrupts


 ;Entering protected mode

 mov eax, cr0
 or ax, 01h            ;Set bit PE (bit 0) of CR0
 mov cr0, eax            ;Apply

 ;We are now in Protected mode

 mov bx, 08h           ;Selector to use, RPL = 0, Table = 0 (GDT), Index = 1

 mov fs, bx            ;Load FS with descriptor 1 info
 mov gs, bx            ;Load GS with descriptor 1 info

 ;Exit protected mode

 and ax, 0fffeh            ;Clear bit PE (bit0) of CR0
 mov cr0, eax                   ;Apply

 sti                

 ;Back to real mode

 ;Do nothing
 cli
 hlt 



 GDT:
    ;First entry, number 0
    ;Null descriptor
    ;Used to store a m16&32 object that tells the GDT start and size

    dw 0fh                 ;Size in byte -1 of the GDT (2 descriptors = 16 bytes)
    dd GDT + 7c00h         ;Linear address of GDT start (24 bits)
    dw 00h                 ;Pad 

    dd 0000ffffh           ;Base[15:00] = 0, Limit[15:00] = 0ffffh
    dd 00cf9200h           ;Base[31:24] = 0, G = 1, B = 1, Limit[19:16] = 0fh, 
               ;P = 1, DPL = 0, E = 0, W = 1, A = 0, Base[23:16] = 00h


 TIMES 510-($-$$) db 00h
 dw 0aa55h

Considerações

  • Assim que um registrador de segmento é recarregado, mesmo com o mesmo valor, o processador recarrega os atributos ocultos de acordo com o modo atual. É por isso que o código acima usa fs e gs para manter os segmentos “estendidos”: tais registradores são menos propensos a serem usados/salvos/restaurados pelos vários serviços de 16 bits.
  • A instrução lgdt não carrega um ponteiro distante para o GDT, em vez disso carrega um endereço linear de 24 bits (pode ser substituído por 32 bits). Este não é um endereço próximo, é o endereço físico (já que a paginação deve ser desabilitada). Essa é a razão de GDT+7c00h.
  • O programa acima é um bootloader (para MBR, ele não tem BPB) que define cs/ds/ss tp 7c00h e inicia o contador de localização a partir de 0. Portanto, um byte no deslocamento X em o arquivo está no deslocamento X no segmento 7c00h e no endereço linear 7c00h + X.
  • As interrupções devem ser desabilitadas, pois um IDT não está definido para a curta viagem de ida e volta no modo protegido.
  • O código usa um hack para salvar 6 bytes de código. A estrutura carregada por lgdt é salva no próprio… GDT, no descritor nulo (o primeiro descritor).

Para obter uma descrição dos descritores GDT, consulte o Capítulo 3.4.3 do [Intel Manual Volume 3A](https://www-ssl.intel.com/content/www/us/en/architecture-and-technology/64-ia- 32-architectures-software-developer-vol-3a-part-1-manual.html).
