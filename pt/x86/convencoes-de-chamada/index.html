Observe também que [clang/gcc sign /zero estende argumentos estreitos para 32 bits][5], mesmo que a ABI como escrita não exija isso. O código gerado pelo Clang depende disso.

  • SystemV 32bit (i386) ABI (padrão oficial), usado por Linux e Unix. (versão antiga).

  • Windows __vectorcall: documenta as versões de 32 bits e 64 bits

  • Windows 32bit __stdcall: usado para chamar funções da API do Win32. Essa página é vinculada a outros documentos de convenção de chamada (por exemplo, __cdecl).

cdecl de 32 bits #

cdecl é uma convenção de chamada de função do Windows de 32 bits que é muito semelhante à convenção de chamada usada em muitos sistemas operacionais POSIX (documentado em i386 System V ABI). Uma das diferenças está no retorno de estruturas pequenas.

Parâmetros #

Os parâmetros são passados ​​na pilha, com o primeiro argumento no endereço mais baixo da pilha no momento da chamada (pressionado por último, logo acima do endereço de retorno na entrada da função). O chamador é responsável por retirar os parâmetros da pilha após a chamada.

Valor de retorno #

Para tipos de retorno escalar, o valor de retorno é colocado em EAX ou EDX:EAX para inteiros de 64 bits. Os tipos de ponto flutuante são retornados em st0 (x87). O retorno de tipos maiores como estruturas é feito por referência, com um ponteiro passado como um primeiro parâmetro implícito. (Esse ponteiro é retornado em EAX, para que o chamador não precise lembrar o que passou).

Registros salvos e bloqueados #

EBX, EDI, ESI, EBP e ESP (e configurações de modo de arredondamento FP / SSE) devem ser preservados pelo chamador, de modo que o chamador possa confiar que esses registros não foram alterados por uma chamada.

Todos os outros registradores (EAX, ECX, EDX, FLAGS (exceto DF), x87 e registradores vetoriais) podem ser modificados livremente pelo chamado; se um chamador deseja preservar um valor antes e depois da chamada da função, ele deve salvar o valor em outro lugar (como em um dos registradores salvos ou na pilha).

stdcall de 32 bits #

stdcall é usado para chamadas de API do Windows de 32 bits.

Parâmetros #

Os parâmetros são passados ​​na pilha, com o primeiro parâmetro mais próximo do topo da pilha. O chamado irá retirar esses valores da pilha antes de retornar.

Valor de retorno #

Os valores de retorno escalar são colocados em EAX.

Registros salvos e bloqueados #

EAX, ECX e EDX podem ser modificados livremente pelo chamador e devem ser salvos pelo chamador, se desejado. EBX, ESI, EDI e EBP devem ser salvos pelo chamado se modificados e restaurados para seus valores originais no retorno.

32 bits, cdecl — Lidando com ponto flutuante #

Como parâmetros (float, double) #

Floats têm 32 bits de tamanho, eles são passados ​​naturalmente na pilha. Doubles têm 64 bits de tamanho, são passados, na pilha, respeitando a convenção Little Endian1, empurrando primeiro os 32 bits superiores e depois os inferiores.

//C prototype of callee
double foo(double a, float b);

foo(3.1457, 0.241);

;Assembly call

;3.1457 is 0x40092A64C2F837B5ULL
;0.241 is 0x3e76c8b4

push DWORD  3e76c8b4h        ;b, is 32 bits, nothing special here
push DWORD 0c2f837b5h        ;a, is 64 bits, Higher part of 3.1457
push DWORD  40092a64h        ;a, is 64 bits, Lower part of 3.1457
call foo
add esp, 0ch

;Call, using the FPU
;ST(0) = a, ST(1) = b
sub esp, 0ch
fstp QWORD PTR [esp]        ;Storing a as a QWORD on the stack
fstp DWORD PTR [esp+08h]    ;Storing b as a DWORD on the stack
call foo
add esp, 0ch

Como parâmetros (long double) #

Duplas longas têm 80 bits2 de largura, enquanto na pilha um TBYTE pode ser armazenado com dois pushes de 32 bits e um push de 16 bits (para 4 + 4 + 2 = 10), para manter a pilha alinhada 4 bytes, acaba ocupando 12 bytes, usando assim três pushes de 32 bits. Respeitando a convenção Little Endian, os bits 79-64 são enviados primeiro3, depois os bits 63-32 seguido pelos bits 31-0.

//C prototype of the callee
void  __attribute__((cdecl)) foo(long double a);

foo(3.1457);

;Call to foo in assembly
;3.1457 is 0x4000c9532617c1bda800

push DWORD 4000h        ;Bits 79-64, as 32 bits push
push DWORD 0c9532617h        ;Bits 63-32
push DWORD 0c1bda800h        ;Bits 31-0
call foo
add esp, 0ch

;Call to foo, using the FPU
;ST(0) = a

sub esp, 0ch
fstp TBYTE PTR [esp]        ;Store a as ten byte on the stack
call foo
add esp, 0ch

Como valor de retorno #

Um valor de ponto flutuante, qualquer que seja seu tamanho, é retornado em ST(0)4.

//C
float one() { return 1; }

;Assembly
fld1            ;ST(0) = 1
ret

//C
double zero() { return 0; }

;Assembly
fldz            ;ST(0) = 0
ret

//C
long double pi() { return PI; }

;Assembly
fldpi            ;ST(0) = PI
ret

1 DWORD inferior no endereço inferior.

2 Conhecido como TBYTE, de Ten Bytes.

3 Usando um empurrão de largura total com qualquer extensão, WORD mais alto não é usado.

4 Que é TBYE largo, note que ao contrário dos números inteiros, FP são sempre retornado com mais precisão do que é necessário.

32 bits, cdecl — Lidando com inteiros #

Como parâmetros (8, 16, 32 bits) #

Inteiros de 8, 16, 32 bits são sempre passados, na pilha, como valores de largura total de 32 bits1. Nenhuma extensão, assinada ou zerada, é necessária. O receptor usará apenas a parte inferior do os valores de largura total.

//C prototype of the callee
void  __attribute__((cdecl)) foo(char a, short b, int c, long d);

foo(-1, 2, -3, 4);


;Call to foo in assembly

push DWORD 4             ;d, long is 32 bits, nothing special here
push DWORD 0fffffffdh    ;c, int is 32 bits, nothing special here
push DWORD 0badb0002h    ;b, short is 16 bits, higher WORD can be any value
push DWORD 0badbadffh    ;a, char is 8 bits, higher three bytes can be any value
call foo
add esp, 10h             ;Clean up the stack

Como parâmetros (64 bits) #

Valores de 64 bits são passados ​​na pilha usando dois pushes, respeitando o littel endian convenção2, empurrando primeiro os 32 bits mais altos e depois os mais baixos.

//C prototype of the callee
void  __attribute__((cdecl)) foo(char a, short b, int c, long d);

foo(0x0123456789abcdefLL);

;Call to foo in assembly

push DWORD 89abcdefh        ;Higher DWORD of 0123456789abcdef
push DWORD 01234567h        ;Lower DWORD of 0123456789abcdef
call foo
add esp, 08h

Como valor de retorno #

Inteiros de 8 bits são retornados em AL, eventualmente sobrecarregando todo o eax. Inteiros de 16 bits são retornados em AX, eventualmente sobrecarregando todo o eax. Inteiros de 32 bits são retornados em EAX. Inteiros de 64 bits são retornados em EDX:EAX, onde EAX contém os 32 bits inferiores e EDX os superiores.

//C
char foo() { return -1; }

;Assembly
mov al, 0ffh
ret

//C
unsigned short foo() { return 2; }

;Assembly
mov ax, 2
ret

//C
int foo() { return -3; }

;Assembly
mov eax, 0fffffffdh
ret

//C
int foo() { return 4; }

;Assembly
xor edx, edx            ;EDX = 0
mov eax, 4            ;EAX = 4
ret

1 Isso mantém a pilha alinhada em 4 bytes, o tamanho natural da palavra. Além disso, uma CPU x86 só pode enviar 2 ou 4 bytes quando não estiver no modo longo.

2 DWORD inferior no endereço inferior

Windows de 64 bits #

Parâmetros #

Os primeiros 4 parâmetros são passados ​​em (em ordem) RCX, RDX, R8 e R9. XMM0 a XMM3 são usados ​​para passar parâmetros de ponto flutuante.

Quaisquer outros parâmetros são passados ​​na pilha.

Parâmetros maiores que 64 bits são passados ​​por endereço.

Espaço de Derramamento

Mesmo que a função use menos de 4 parâmetros, o chamador sempre fornece espaço para 4 parâmetros de tamanho QWORD na pilha. O chamado é livre para usá-los para qualquer finalidade, é comum copiar os parâmetros para lá se eles forem derramados por outra chamada.

Valor de retorno #

Para tipos de retorno escalares, o valor de retorno é colocado em RAX. Se o tipo de retorno for maior que 64 bits (por exemplo, para estruturas), RAX é um ponteiro para isso.

Registros salvos e bloqueados #

Todos os registradores usados ​​na passagem de parâmetros (RCX, RDX, R8, R9 e XMM0 para XMM3), RAX, R10, R11, XMM4 e XMM5 podem ser derramados pelo chamado. Todos os outros registros precisam ser preservados pelo chamador (por exemplo, na pilha).

Alinhamento da pilha #

A pilha deve ser mantida alinhada com 16 bytes. Como a instrução “call” envia um endereço de retorno de 8 bytes, isso significa que cada função não folha ajustará a pilha por um valor da forma 16n+8 para restaurar o alinhamento de 16 bytes. É o trabalho do chamador limpar a pilha após uma chamada.

Fonte: A história das convenções de chamada, parte 5: amd64 Raymond Chen

32 bits, cdecl — Lidando com Structs #

Preenchimento #

Lembre-se, os membros de uma estrutura geralmente são preenchidos para garantir que estejam alinhados em seu limite natural:

struct t
{
    int a, b, c, d;    // a is at offset 0, b at 4, c at 8, d at 0ch
    char e;            // e is at 10h
    short f;           // f is at 12h (naturally aligned)
    long g;            // g is at 14h
    char h;            // h is at 18h
    long i;            // i is at 1ch (naturally aligned)
};

Como parâmetros (passar por referência) #

Quando passado por referência, um ponteiro para a estrutura na memória é passado como o primeiro argumento na pilha. Isso é equivalente a passar um valor inteiro de tamanho natural (32 bits); consulte https://www.wikiod.com/pt/x86/convencoes-de-chamada#cdecl de 32 bits para obter detalhes.

Como parâmetros (passar por valor) #

Quando passadas por valor, as structs são totalmente copiadas na pilha, respeitando o layout original da memória (i.e., o primeiro membro estará no endereço inferior).

int __attribute__((cdecl)) foo(struct t a);

struct t s = {0, -1, 2, -3, -4, 5, -6, 7, -8};
foo(s);

; Assembly call

push DWORD 0fffffff8h    ; i (-8)
push DWORD 0badbad07h    ; h (7), pushed as DWORD to naturally align i, upper bytes can be garbage
push DWORD 0fffffffah    ; g (-6)
push WORD 5              ; f (5)
push WORD 033fch         ; e (-4), pushed as WORD to naturally align f, upper byte can be garbage
push DWORD 0fffffffdh    ; d (-3)
push DWORD 2             ; c (2)
push DWORD 0ffffffffh    ; b (-1)
push DWORD 0             ; a (0)
call foo
add esp, 20h

Como valor de retorno #

A menos que sejam triviais1, as estruturas são copiadas para um buffer fornecido pelo chamador antes de retornar. Isso é equivalente a ter um primeiro parâmetro oculto struct S *retval (onde struct S é o tipo do struct).

A função deve retornar com este ponteiro para o valor de retorno em eax; O chamador pode depender de eax segurando o ponteiro para o valor de retorno, que ele empurrou logo antes do call.

struct S
{
    unsigned char a, b, c;
};

struct S foo();         // compiled as struct S* foo(struct S* _out)

O parâmetro oculto não é adicionado à contagem de parâmetros para fins de limpeza da pilha, pois deve ser tratado pelo chamado.

sub esp, 04h        ; allocate space for the struct

; call to foo
push esp            ; pointer to the output buffer
call foo
add esp, 00h        ; still as no parameters have been passed

No exemplo acima, a estrutura será salva no topo da pilha.

struct S foo()
{
    struct S s;
    s.a = 1; s.b = -2; s.c = 3;
    return s;
}

; Assembly code
push ebx
mov eax, DWORD PTR [esp+08h]   ; access hidden parameter, it is a pointer to a buffer
mov ebx, 03fe01h               ; struct value, can be held in a register
mov DWORD [eax], ebx           ; copy the structure into the output buffer 
pop ebx
ret 04h                        ; remove the hidden parameter from the stack
                               ; EAX = pointer to the output buffer

1 Um struct “trivial” é aquele que contém apenas um membro de um tipo não struct, não array (até 32 bits de tamanho). Para tais estruturas, o valor desse membro é simplesmente retornado no registro eax. (Esse comportamento foi observado com o GCC direcionado ao Linux)

A versão do Windows do cdecl é diferente da convenção de chamada do System V ABI: Um struct “trivial” pode conter até dois membros de um tipo não-struct, não-array (até 32 bits de tamanho). Esses valores são retornados em eax e edx, assim como um inteiro de 64 bits seria. (Esse comportamento foi observado para MSVC e Clang visando Win32.)

Sistema V de 64 bits #

Esta é a convenção de chamada padrão para aplicativos de 64 bits em muitos sistemas operacionais POSIX.

Parâmetros #

Os primeiros oito parâmetros escalares são passados ​​em (em ordem) RDI, RSI, RDX, RCX, R8, R9, R10, R11. Os parâmetros após os oito primeiros são colocados na pilha, com os parâmetros anteriores mais próximos do topo da pilha. O chamador é responsável por retirar esses valores da pilha após a chamada, se não for mais necessário.

Valor de retorno #

Para tipos de retorno escalares, o valor de retorno é colocado em RAX. O retorno de tipos maiores, como estruturas, é feito alterando conceitualmente a assinatura da função para adicionar um parâmetro no início da lista de parâmetros que é um ponteiro para um local no qual colocar o valor de retorno.

Registros salvos e bloqueados #

RBP, RBX e R12–R15 são preservados pelo receptor. Todos os outros registradores podem ser modificados pelo chamador, e o chamador deve preservar o próprio valor de um registrador (por exemplo, na pilha) se desejar usar esse valor posteriormente.
