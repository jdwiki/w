O Action Mailer permite enviar e-mails de seu aplicativo usando classes e visualizações de mailer. Mailers funcionam de forma muito semelhante aos controladores. Eles herdam de ActionMailer::Base e vivem em app/mailers, e têm visualizações associadas que aparecem em app/views.

É aconselhável processar o envio de e-mail de forma assíncrona para não sobrecarregar seu servidor web. Isso pode ser feito através de vários serviços como delayed_job.

Correio Básico #

Este exemplo usa quatro arquivos diferentes:

  • O modelo de usuário
  • O mailer do usuário
  • O modelo html para o e-mail
  • O modelo de texto simples para o e-mail

Neste caso, o modelo de usuário chama o método approved no mailer e passa o post que foi aprovado (o método approved no modelo pode ser chamado por um callback, a partir de um método do controlador, etc). Em seguida, o remetente gera o e-mail a partir do modelo html ou de texto simples usando as informações do post passado (por exemplo, o título). Por padrão, o mailer usa o modelo com o mesmo nome que o método no mailer (e é por isso que tanto o método do mailer quanto os modelos têm o nome ‘approved’).

user_mailer.rb #

class UserMailer < ActionMailer::Base
  default from: "donotreply@example.com"

  def approved(post)
      @title = post.title
      @user = post.user
      mail(to: @user.email, subject: "Your Post was Approved!")
  end
end

#usuário.rb

def approved(post)
    UserMailer.approved(post)
end

aprovado.html.erb #

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
<html>
    <head>
      <meta http-equiv="Content-Type" content="text/html; charset=utf-8">
      <title>Post Approved</title>
    </head>
    <body>
        <h2>Congrats <%= @user.name %>! Your post (#<%= @title %>) has been approved!</h2>
        <p>We look forward to your future posts!</p>
    </body>
</html>

aprovado.texto.erb #

Congrats <%= @user.name %>! Your post (#<%= @title %>) has been approved!
We look forward to your future posts!

Gerando um novo mailer #

Para gerar um novo mailer, digite o seguinte comando

rails generate mailer PostMailer

Isso irá gerar um arquivo de modelo em branco em app/mailers/post_mailer.rb chamado PostMailer

class PostMailer < ApplicationMailer
end

Dois arquivos de layout também serão gerados para a visualização de email, um para o formato html e outro para o formato de texto.

Se preferir não usar o gerador, você pode criar suas próprias malas diretas. Certifique-se de que eles herdam de ActionMailer::Base

Adicionando anexos #

ActionMailer também permite anexar arquivos.

attachments['filename.jpg'] = File.read('/path/to/filename.jpg')

Por padrão, os anexos serão codificados com Base64. Para alterar isso, você pode adicionar um hash ao método de anexos.

attachments['filename.jpg'] = {
  mime_type: 'application/gzip',
  encoding: 'SpecialEncoding',
  content: encoded_content
}

Você também pode adicionar anexos embutidos

attachments.inline['image.jpg'] = File.read('/path/to/image.jpg')

Retornos de chamada do ActionMailer #

ActionMailer suporta três retornos de chamada

  • antes_ação
  • after_action
  • around_action

Forneça isso na sua classe Mailer

class UserMailer < ApplicationMailer
  after_action :set_delivery_options, :prevent_delivery_to_guests, :set_business_headers

Em seguida, crie esses métodos na palavra-chave private

private
  def set_delivery_options
  end

  def prevent_delivery_to_guests
  end

  def set_business_headers
  end
end

Interceptador do ActionMailer #

O Action Mailer fornece ganchos para os métodos do interceptor. Eles permitem que você registre classes que são chamadas durante o ciclo de vida de entrega de correio.

Uma classe interceptora deve implementar o método :delivering_email(message) que será chamado antes que o email seja enviado, permitindo que você faça modificações no email antes que ele chegue aos agentes de entrega. Sua classe deve fazer as modificações necessárias diretamente na instância Mail::Message passada.

Pode ser útil para os desenvolvedores enviar e-mails para eles mesmos, não para usuários reais.

Exemplo de registro de um interceptador do actionmailer:

# config/initializers/override_mail_recipient.rb

if Rails.env.development? or Rails.env.test?
  class OverrideMailRecipient
    def self.delivering_email(mail)
      mail.subject = 'This is dummy subject'
      mail.bcc = 'test_bcc@noemail.com'
      mail.to = 'test@noemail.com'
    end
  end
  ActionMailer::Base.register_interceptor(OverrideMailRecipient)
end

Gere um boletim informativo agendado #

Crie o modelo de Boletim informativo:

<pré> rails g modelo Nome do boletim informativo:string email:string

subl app/models/newsletter.rb

valida :name, presença: true valida :email, presença: true

Crie o controlador de Newsletter:

<pré> Rails g Criação de boletins informativos do controlador

classe NewslettersController < ApplicationController skip_before_action :authenticate_user! before_action :set_newsletter, only: [:destroy]

def create
  @newsletter = Newsletter.create(newsletter_params)
  if @newsletter.save
    redirect_to blog_index_path
  else
    redirect_to root_path
  end
end

private

  def set_newsletter
    @newsletter = Newsletter.find(params[:id])
  end

  def newsletter_params
    params.require(:newsletter).permit(:name, :email)
  end

fim

Depois disso, altere a visualização create.html.erb para o nome nex. Vamos converter este arquivo para uma visualização parcial que será armazenada dentro do Rodapé. O nome será _form.html.erb.

| Altere o nome do arquivo de: | Para: | |:———————————————— -|:———————————————– —-| | app/views/newsletters/create.html.erb | app/views/newsletters/_form.html.erb |

Depois disso, defina as rotas:

<pré> subl app/config/routes.rb

recursos: boletins

Mais tarde, precisamos definir o formulário que usaremos para salvar cada e-mail:

<pré> subl app/views/newsletters/_form.html.erb

<%= form_for (Newsletter.new) do |f| %> <div class=“col-md-12” style=“margin: 0 auto; padding: 0;"> <div class=“col-md-6” style=“padding: 0;"> <%= f.text_field :name, class: ‘form-control’, placeholder:‘Nombre’ %> </div> <div class=“col-md-6” style=“padding: 0;"> <%= f.text_field :email, class: ‘form-control’, placeholder:‘Email’ %> </div> </div> <div class=“col-md-12” style=“margin: 0 auto; padding:0;"> <%= f.submit class:“col-md-12 tran3s s-color-bg hvr-shutter-out-horizontal”, style:‘border: none; color: white; cursor: pointer; margin: 0.5em auto; padding: 0.75em; width: 100%;’ %> </div> <% end %>

E depois disso, insira no rodapé:

<pré> subl app/views/layouts/_footer.html.erb

<%= renderizar ’newsletters/form’ %>

Agora, instale o -letter_opener- para visualizar o e-mail no navegador padrão em vez de enviá-lo. Isso significa que você não precisa configurar a entrega de e-mail em seu ambiente de desenvolvimento e não precisa mais se preocupar em enviar acidentalmente um e-mail de teste para o endereço de outra pessoa.

Primeiro adicione a gem ao seu ambiente de desenvolvimento e execute o comando bundle para instalá-la.

<pré> subl your_project/Gemfile

gem “letter_opener”, :group => :development

Em seguida, defina o método de entrega no Ambiente de Desenvolvimento:

<pré> subl your_project/app/config/environments/development.rb

config.action_mailer.delivery_method = :letter_opener

Agora, crie uma Estrutura de Mailer para gerenciar todos os mailers que iremos trabalhar. No terminal

<pré> Rails gera mailer UserMailer newsletter_mailer

E dentro do UserMailer, temos que criar um método chamado Newsletter Mailer que será criado para conter o último post do blog e será acionado com uma ação de rake. Vamos supor que você tenha uma estrutura de blog criada anteriormente.

<pré> subl your_project/app/mailers/user_mailer.rb

class UserMailer < ApplicationMailer padrão de: ‘sua_conta_gmail@gmail.com’

def newsletter_mailer @newsletter = Newsletter.all @post = Post.last(3) emails = @newsletter.collect(&:email).join(", ") mail(to: emails, subject: "Hi, this is a test mail.") fim fim

Depois disso, crie o Mailer Template:

<pré> subl your_project/app/views/user_mailer/newsletter_mailer.html.erb

<p> Caros Seguidores: </p> <p> Essas são as últimas entradas do nosso blog. Convidamos você a ler e compartilhar tudo o que fizemos nesta semana. </p>

<br/> <tabela> <% @post.each do |post| %> <%#= link_to blog_url(post) do %> <tr style=“display:flex; float:left; clear:both;"> <td style=“display:flex; float:left; clear:both; height: 80px; width: 100px;"> <% if post.cover_image.present? %> <%= image_tag post.cover_image.fullsize.url, class:“principal-home-image-slider” %> <%# else %> <%#= image_tag ‘http://your_site_project.com’ + post.cover_video, class:“principal-home-image-slider” %> <%#= raw(video_embed(post.cover_video)) %> <% end %> </td> <td> <h3> <%= link_to post.title, :controller => “blog”, :action => “show”, :only_path => false, :id => post.id %> </h3> <p><%= post.subtitle %></p> </td> <td style=“display:flex; float:left; clear:both;">   </td> </tr> <%# end %> <% end %> </tabela>

Como queremos enviar o e-mail como um processo separado, vamos criar uma tarefa Rake para disparar o e-mail. Adicione um novo arquivo chamado email_tasks.rake ao diretório lib/tasks do seu aplicativo Rails:

<pré> toque em lib/taks/email_tasks.rake

desc ’e-mail de newsletter semanal’ task week_newsletter_email: :environment do UserMailer.newsletter_mailer.entrega! fim

O send_digest_email: :environment significa carregar o ambiente Rails antes de executar a tarefa, para que você possa acessar as classes da aplicação (como UserMailer) dentro da tarefa.

Agora, executando o comando rake -T listará a tarefa Rake recém-criada. Teste tudo funcionando executando a tarefa e verificando se o email foi enviado ou não.

Para testar se o método mailer funciona, execute o comando rake:

<pré> rake weekly_newsletter_email

Neste ponto, temos uma tarefa de rake em funcionamento que pode ser agendada usando crontab. Portanto, instalaremos o Whenever Gem, que é usado para fornecer uma sintaxe clara para escrever e implantar tarefas cron.

<pré> subl your_project/Gemfile

gem ‘sempre que’, exigir: false

Depois disso, execute o próximo comando para criar um arquivo config/schedule.rb inicial para você (desde que a pasta config já esteja presente em seu projeto).

<pré> sempre.

[adicionar] escrevendo `./config/schedule.rb’ [feito] sempre!

Agora, dentro do arquivo de agendamento, temos que criar nosso CRON JOB e chamar o método mailer dentro determinando o CRON JOB para operar algumas tarefas sem assistência e em um intervalo de tempo selecionado. Você pode usar diferentes tipos de sintaxe, conforme explicado neste link.

<pré> subl your_project/config/schedule.rb

a cada 1.dia, :at => ‘4:30 am’ faça rake ‘weekly_newsletter_email’ fim

Agora, para testar se o Cron Job foi criado com sucesso, podemos usar o próximo comando para ler desde o terminal, nosso trabalho agendado em CRON SYNTAX:

<pré> your_project your_mac_user$ sempre que

30 4 * * * /bin/bash -l -c ‘cd /Users/your_mac_user/Desktop/your_project && RAILS_ENV=executivo do pacote de produção rake week_newsletter_email –silent’

Agora, para executar o teste no ambiente de desenvolvimento, é aconselhável definir a próxima linha no arquivo principal application.rb para que o aplicativo saiba onde estão os modelos que serão usados.

<pré> subl your_project/config/application.rb

config.action_mailer.default_url_options = { :host => “http://localhost:3000/” }

Agora, para deixar o Capistrano V3 salvar o novo Cron Job dentro do servidor e o gatilho que acionará a execução desta tarefa, temos que adicionar o próximo requisito:

<pré> subl your_project/Capfile

exigir ‘sempre/capistrano’

E insira no arquivo deploy o identificador que o CRON JOB usará sobre o ambiente e o nome do aplicativo.

<pré> subl your_project/config/deploy.rb

set :whenever_identifier, ->{ “#{fetch(:application)}_#{fetch(:rails_env)}” }

E pronto, após salvar as alterações em cada arquivo, execute o comando capistrano deploy:

<pré> implantação de produção de limite

E agora seu JOB foi criado e calendarizado para executar o Mailer Method que é o que eu quero e no intervalo de tempo que definimos nestes arquivos.
