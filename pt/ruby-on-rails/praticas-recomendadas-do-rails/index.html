Modelo Gordo, Controlador Magro #

“Fat Model, Skinny Controller” refere-se a como as partes M e C do MVC trabalham idealmente juntas. Ou seja, qualquer lógica não relacionada à resposta deve entrar no modelo, de preferência em um método agradável e testável. Enquanto isso, o controlador “magro” é simplesmente uma boa interface entre a visão e o modelo.

Na prática, isso pode exigir uma variedade de diferentes tipos de refatoração, mas tudo se resume a uma ideia: movendo qualquer lógica que não seja sobre a resposta ao modelo (em vez do controlador), você não apenas promoveu a reutilização sempre que possível, mas você também possibilitou testar seu código fora do contexto de uma solicitação.

Vejamos um exemplo simples. Digamos que você tenha um código assim:

def index
  @published_posts = Post.where('published_at <= ?', Time.now)
  @unpublished_posts = Post.where('published_at IS NULL OR published_at > ?', Time.now)
end

Você pode alterá-lo para isso:

def index
  @published_posts = Post.published
  @unpublished_posts = Post.unpublished
end

Em seguida, você pode mover a lógica para o seu modelo de postagem, onde pode ficar assim:

scope :published, ->(timestamp = Time.now) { where('published_at <= ?', timestamp) }
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) }

Objetos de Domínio (Chega de Modelos Gordos) #

“Fat Model, Skinny Controller” é um primeiro passo muito bom, mas não escala bem quando sua base de código começa a crescer.

Vamos pensar na Single Responsibility dos modelos. Qual é a responsabilidade única dos modelos? É para manter a lógica de negócios? É para manter a lógica não relacionada à resposta?

Não. Sua responsabilidade é lidar com a camada de persistência e sua abstração.

A lógica de negócios, bem como qualquer lógica não relacionada à resposta e lógica não relacionada à persistência, deve ir em objetos de domínio.

Objetos de domínio são classes projetadas para ter apenas uma responsabilidade no domínio do problema. Deixe suas aulas “Scream Their Architecture” pelos problemas que elas resolvem.

Na prática, você deve se esforçar para modelos magros, visualizações estreitas e controladores magros. A arquitetura da sua solução não deve ser influenciada pela estrutura que você está escolhendo.

Por exemplo

Digamos que você seja um marketplace que cobra uma comissão fixa de 15% de seus clientes via Stripe. Se você cobrar uma comissão fixa de 15%, isso significa que sua comissão muda dependendo do valor do pedido, pois o Stripe cobra 2,9% + 30¢.

O valor que você cobra como comissão deve ser: valor*0,15 - (valor*0,029 + 0,30).

Não escreva essa lógica no modelo:

# app/models/order.rb
class Order < ActiveRecord::Base
  SERVICE_COMMISSION = 0.15
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  ...

  def commission
    amount*SERVICE_COMMISSION - stripe_commission  
  end

  private

  def stripe_commission
    amount*STRIPE_PERCENTAGE_COMMISSION + STRIPE_FIXED_COMMISSION
  end
end

Assim que você se integrar a uma nova forma de pagamento, não poderá dimensionar essa funcionalidade dentro deste modelo.

Além disso, assim que você começar a integrar mais lógica de negócios, seu objeto Order começará a perder coesão.

Prefira objetos de domínio, com o cálculo da comissão totalmente abstraído da responsabilidade de pedidos persistentes:

# app/models/order.rb
class Order < ActiveRecord::Base
  ...
  # No reference to commission calculation
end

# lib/commission.rb
class Commission
  SERVICE_COMMISSION = 0.15

  def self.calculate(payment_method, model)
    model.amount*SERVICE_COMMISSION - payment_commission(payment_method, model)  
  end

  private

  def self.payment_commission(payment_method, model)
    # There are better ways to implement a static registry,
    # this is only for illustration purposes.
    Object.const_get("#{payment_method}Commission").calculate(model)
  end
end

# lib/stripe_commission.rb
class StripeCommission
  STRIPE_PERCENTAGE_COMMISSION = 0.029
  STRIPE_FIXED_COMMISSION = 0.30

  def self.calculate(model)
    model.amount*STRIPE_PERCENTAGE_COMMISSION
      + STRIPE_PERCENTAGE_COMMISSION
  end
end

# app/controllers/orders_controller.rb
class OrdersController < ApplicationController
  def create
    @order = Order.new(order_params)
    @order.commission = Commission.calculate("Stripe", @order)
    ...
  end
end

O uso de objetos de domínio tem as seguintes vantagens arquitetônicas:

  • é extremamente fácil fazer o teste de unidade, pois não são necessários acessórios ou fábricas para instanciar os objetos com a lógica.
  • funciona com tudo que aceita a mensagem amount.
  • mantém cada objeto de domínio pequeno, com responsabilidades claramente definidas e com maior coesão.
  • escala facilmente com novos métodos de pagamento por adição, não modificação.
  • interrompe a tendência de ter um objeto User cada vez maior em cada aplicação Ruby on Rails.

Eu pessoalmente gosto de colocar objetos de domínio em lib. Se você fizer isso, lembre-se de adicioná-lo a autoload_paths:

# config/application.rb
config.autoload_paths << Rails.root.join('lib')

Você também pode preferir criar objetos de domínio mais orientados à ação, seguindo o padrão Comando/Consulta. Nesse caso, colocar esses objetos em app/commands pode ser um lugar melhor, pois todos os subdiretórios app são adicionados automaticamente ao caminho de carregamento automático.

Convenção sobre configuração #

No Rails, você se encontra olhando controllers, views e models para seu banco de dados.

Para reduzir a necessidade de configuração pesada, Rails implementa regras para facilitar o trabalho com a aplicação. Você pode definir suas próprias regras, mas para o início (e para mais tarde) é uma boa ideia manter as convenções que o Rails oferece.

Essas convenções irão acelerar o desenvolvimento, manter seu código conciso e legível e permitir uma navegação fácil dentro de seu aplicativo.

As convenções também reduzem as barreiras de entrada para iniciantes. Existem tantas convenções no Rails que um iniciante nem precisa conhecer, mas pode se beneficiar da ignorância. É possível criar ótimos aplicativos sem saber por que tudo é do jeito que é.

Por exemplo

Se você tem uma tabela de banco de dados chamada orders com a chave primária id, o modelo correspondente é chamado order e o controlador que trata de toda a lógica é denominado orders_controller. A visão é dividida em ações diferentes: se o controlador tem uma ação new e edit, também há uma visão new e edit.

Por exemplo

Para criar um aplicativo, basta executar rails new app_name. Isso gerará aproximadamente 70 arquivos e pastas que compõem a infraestrutura e a base do seu aplicativo Rails.

Inclui:

  • Pastas para armazenar seus modelos (camada de banco de dados), controladores e visualizações
  • Pastas para realizar testes de unidade para sua aplicação
  • Pastas para armazenar seus ativos da web, como arquivos Javascript e CSS
  • Arquivos padrão para respostas HTTP 400 (ou seja, arquivo não encontrado)
  • Muitos outros

Cuidado com default_scope #

ActiveRecord inclui default_scope, para definir automaticamente o escopo de um modelo por padrão.

class Post
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

O código acima servirá posts que já estão publicados quando você realizar qualquer consulta no modelo.

Post.all # will only list published posts

Esse escopo, embora pareça inócuo, tem vários efeitos colaterais ocultos que você pode não querer.

default_scope e order #

Já que você declarou um order no default_scope, chamar order em Post será adicionado como pedidos adicionais em vez de substituir o padrão.

Post.order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."created_at" DESC, "posts"."updated_at" DESC

Este provavelmente não é o comportamento que você queria; você pode substituir isso excluindo o order do escopo primeiro

Post.except(:order).order(updated_at: :desc)

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' ORDER BY "posts"."updated_at" DESC

default_scope e inicialização do modelo #

Como com qualquer outro ActiveRecord::Relation, default_scope alterará o estado padrão dos modelos inicializados a partir dele.

No exemplo acima, Post tem where(published: true) definido por padrão e, portanto, novos modelos de Post também o terão definido.

Post.new # => <Post published: true>

sem escopo #

default_scope pode ser limpo nominalmente chamando unscoped primeiro, mas isso também tem efeitos colaterais. Tomemos, por exemplo, um modelo STI:

class Post < Document
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

Por padrão, as consultas em Post terão escopo para colunas type contendo 'Post'. Mas unscoped limpará isso junto com seu próprio default_scope, então se você usar unscoped você deve se lembrar de considerar isso também.

Post.unscoped.where(type: 'Post').order(updated_at: :desc)

sem escopo e associações de modelo #

Considere uma relação entre Post e User

class Post < ApplicationRecord
  belongs_to :user
  default_scope ->{ where(published: true).order(created_at: :desc) }
end

class User < ApplicationRecord
  has_many :posts
end

Ao obter um ‘Usuário’ individual, você pode ver as postagens relacionadas a ele:

user = User.find(1)
user.posts

SELECT "posts".* FROM "posts" WHERE "posts"."published" = 't' AND "posts"."user_id" = ? ORDER BY "posts"."created_at" DESC [["user_id", 1]]

Mas você quer limpar o default_scope da relação posts, então você usa unscoped

user.posts.unscoped

SELECT "posts".* FROM "posts"

Isso apaga a condição user_id, bem como o default_scope.

Um exemplo de caso de uso para default_scope #

Apesar de tudo isso, existem situações em que o uso de default_scope é justificável.

Considere um sistema multilocatário em que vários subdomínios são atendidos pelo mesmo aplicativo, mas com dados isolados. Uma maneira de conseguir esse isolamento é através do default_scope. As desvantagens em outros casos tornam-se vantagens aqui.

class ApplicationRecord < ActiveRecord::Base
  def self.inherited(subclass)
    super

    return unless subclass.superclass == self
    return unless subclass.column_names.include? 'tenant_id'

    subclass.class_eval do
      default_scope ->{ where(tenant_id: Tenant.current_id) }
    end
  end
end

Tudo o que você precisa fazer é definir Tenant.current_id para algo no início da solicitação, e qualquer tabela que contenha tenant_id terá escopo automaticamente sem nenhum código adicional. A instanciação de registros herdará automaticamente o ID do locatário com o qual foram criados.

O importante nesse caso de uso é que o escopo é definido uma vez por solicitação e não muda. Os únicos casos que você precisará sem escopo aqui são casos especiais como trabalhadores em segundo plano que são executados fora de um escopo de solicitação.

Não se repita (SEC) #

Para ajudar a manter o código limpo, o Rails segue o princípio do DRY.

Envolve sempre que possível, reutilizar o máximo de código possível, em vez de duplicar código semelhante em vários lugares (por exemplo, usando parciais). Isso reduz erros, mantém seu código limpo e reforça o princípio de escrever código uma vez e depois reutilizá-lo. Também é mais fácil e eficiente atualizar o código em um só lugar do que atualizar várias partes do mesmo código. Tornando assim seu código mais modular e robusto.

Também Fat Model, Skinny Controller é DRY, pois você escreve o código no seu model e no controller só faz a chamada, como:

# Post model
scope :unpublished, ->(timestamp = Time.now) { where('published_at IS NULL OR published_at > ?', timestamp) } 


# Any controller
def index
    ....
    @unpublished_posts = Post.unpublished
    ....
end

def others
    ...
    @unpublished_posts = Post.unpublished
    ...
end

Isso também ajuda a levar a uma estrutura orientada por API, na qual os métodos internos são ocultos e as alterações são alcançadas por meio da passagem de parâmetros em uma forma de API.

Você não vai precisar (YAGNI) #

Se você pode dizer “YAGNI” (Você não vai precisar) sobre um recurso, é melhor não implementá-lo. Pode haver muito tempo de desenvolvimento economizado concentrando-se na simplicidade. Implementar esses recursos de qualquer maneira pode levar a problemas:

Problemas #

Superengenharia #

Se um produto é mais complicado do que tem que ser, ele é superprojetado. Normalmente, esses recursos “ainda não usados” nunca serão usados ​​da maneira pretendida, e terão que ser refatorados se forem usados. Otimizações prematuras, especialmente otimizações de desempenho, muitas vezes levam a decisões de projeto que serão provadas erradas no futuro.

Codificação inchada #

Code Bloat significa código complicado desnecessário. Isso pode ocorrer, por exemplo, por abstração, redundância ou aplicação incorreta de padrões de projeto. A base de código torna-se difícil de entender, confusa e cara para manter.

Deslocamento de recursos #

Feature Creep refere-se à adição de novos recursos que vão além da funcionalidade principal do produto e levam a uma complexidade desnecessariamente alta do produto.

Longo tempo de desenvolvimento #

O tempo que poderia ser usado para desenvolver recursos necessários é gasto para desenvolver recursos desnecessários. O produto demora mais para ser entregue.

Soluções #

KISS - Mantenha-o simples, estúpido #

De acordo com o KISS, a maioria dos sistemas funciona melhor se forem projetados de forma simples. A simplicidade deve ser um objetivo principal do projeto para reduzir a complexidade. Isso pode ser alcançado seguindo o “Princípio da Responsabilidade Única”, por exemplo.

YAGNI – Você não vai precisar #

Menos é mais. Pense em cada recurso, é realmente necessário? Se você puder pensar de alguma maneira que seja YAGNI, deixe-o de lado. É melhor desenvolvê-lo quando for necessário.

Refatoração Contínua #

O produto está sendo melhorado constantemente. Com a refatoração, podemos ter certeza de que o produto está sendo feito de acordo com as melhores práticas e não degenera em um trabalho de correção.
