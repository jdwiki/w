O roteador Rails reconhece URLs e os envia para a ação de um controlador. Ele também pode gerar caminhos e URLs, evitando a necessidade de codificar strings em suas visualizações.

“Roteamento” em geral é como os URLs são “tratados” pelo seu aplicativo. No caso do Rails, normalmente é qual controlador e qual ação desse controlador manipulará uma URL de entrada específica. Em aplicativos Rails, as rotas geralmente são colocadas no arquivo config/routes.rb.

Roteamento de recursos (básico) #

As rotas são definidas em config/routes.rb. Eles geralmente são definidos como um grupo de rotas relacionadas, usando o resources ou resource.

resources :users cria as sete rotas a seguir, todas mapeando para ações de UsersController:

get       '/users',          to: 'users#index'
post      '/users',          to: 'users#create'
get       '/users/new',      to: 'users#new'
get       '/users/:id/edit', to: 'users#edit'
get       '/users/:id',      to: 'users#show'
patch/put '/users/:id',      to: 'users#update'
delete    '/users/:id',      to: 'users#destroy'

Os nomes das ações são mostrados após o # no parâmetro to acima. Métodos com esses mesmos nomes devem ser definidos em app/controllers/users_controller.rb da seguinte forma:

class UsersController < ApplicationController
  def index
  end

  def create
  end

  # continue with all the other methods…
end

Você pode limitar as ações que são geradas com only ou except:

resources :users, only:   [:show]
resources :users, except: [:show, :index]

Você pode visualizar todas as rotas do seu aplicativo a qualquer momento executando:

$ rake routes

$ rake routes
# OR
$ rails routes

Você pode pesquisar por rotas usando a opção -g. Isso mostra qualquer rota que corresponda parcialmente ao nome do método auxiliar, ao caminho da URL ou ao verbo HTTP:

$ rake routes -g POST         # Matches HTTP Verb POST

# OR

$ rails routes -g POST        # Matches HTTP Verb POST

Além disso, ao executar o servidor rails no modo de desenvolvimento, você pode acessar uma página da web que mostra todas as suas rotas com um filtro de pesquisa, com prioridade de cima para baixo, em <hostname>/rails/info/routes. Isso parecerá assim:

As rotas podem ser declaradas disponíveis apenas para membros (não coleções) usando o método resource em vez de resources em routes.rb. Com resource, uma rota index não é criada por padrão, mas apenas quando explicitamente solicita uma como esta:

resource :orders, only: [:index, :create, :show]

Restrições #

Você pode filtrar quais rotas estão disponíveis usando restrições.

Existem várias maneiras de usar restrições, incluindo:

  • restrições de segmento,
  • restrições baseadas em solicitação
  • restrições avançadas

Por exemplo, uma restrição baseada em solicitação para permitir que apenas um endereço IP específico acesse uma rota:

constraints(ip: /127\.0\.0\.1$/) do
  get 'route', to: "controller#action"
end

Veja outros exemplos semelhantes ActionDispatch::Routing::Mapper::Scoping.

Se você quiser fazer algo mais complexo, você pode usar restrições mais avançadas e criar uma classe para envolver a lógica:

# lib/api_version_constraint.rb
class ApiVersionConstraint
  def initialize(version:, default:)
    @version = version
    @default = default
  end

  def version_header
    "application/vnd.my-app.v#{@version}"
  end

  def matches?(request)
    @default || request.headers["Accept"].include?(version_header)
  end
end

# config/routes.rb
require "api_version_constraint"

Rails.application.routes.draw do
  namespace :v1, constraints: ApiVersionConstraint.new(version: 1, default: true) do
    resources :users # Will route to app/controllers/v1/users_controller.rb
  end

  namespace :v2, constraints: ApiVersionConstraint.new(version: 2) do
    resources :users # Will route to app/controllers/v2/users_controller.rb
  end
end

Um formulário, vários botões de envio

Você também pode usar o valor das tags de envio de um formulário como uma restrição para rotear para uma ação diferente. Se você tiver um formulário com vários botões de envio (por exemplo, “preview” e “submit”), você pode capturar essa restrição diretamente em seu routes.rb, em vez de escrever javascript para alterar o URL de destino do formulário. Por exemplo, com a gem commit_param_routing você pode tirar vantagem do Rails submit_tag

O primeiro parâmetro submit_tag do Rails permite que você altere o valor do seu parâmetro de confirmação do formulário

# app/views/orders/mass_order.html.erb
<%= form_for(@orders, url: mass_create_order_path do |f| %>
    <!-- Big form here -->
  <%= submit_tag "Preview" %>
  <%= submit_tag "Submit" %>
  # => <input name="commit" type="submit" value="Preview" />
  # => <input name="commit" type="submit" value="Submit" />
  ...
<% end %>

# config/routes.rb
resources :orders do
  # Both routes below describe the same POST URL, but route to different actions 
  post 'mass_order', on: :collection, as: 'mass_order',
    constraints: CommitParamRouting.new('Submit'), action: 'mass_create' # when the user presses "submit"
  post 'mass_order', on: :collection,
    constraints: CommitParamRouting.new('Preview'), action: 'mass_create_preview' # when the user presses "preview"
  # Note the `as:` is defined only once, since the path helper is mass_create_order_path for the form url
  # CommitParamRouting is just a class like ApiVersionContraint
end

Rotas de escopo #

Rails fornece várias maneiras de organizar suas rotas.

Escopo por URL:

scope 'admin' do
  get 'dashboard', to: 'administration#dashboard'
  resources 'employees'
end

Isso gera as seguintes rotas

get       '/admin/dashboard',          to: 'administration#dashboard'
post      '/admin/employees',          to: 'employees#create'
get       '/admin/employees/new',      to: 'employees#new'
get       '/admin/employees/:id/edit', to: 'employees#edit'
get       '/admin/employees/:id',      to: 'employees#show'
patch/put '/admin/employees/:id',      to: 'employees#update'
delete    '/admin/employees/:id',      to: 'employees#destroy'

Pode fazer mais sentido, no lado do servidor, manter algumas visualizações em uma subpasta diferente, separar as visualizações do administrador das visualizações do usuário.

Escopo por módulo

scope module: :admin do
  get 'dashboard', to: 'administration#dashboard'
end

module procura os arquivos do controlador na subpasta do nome fornecido

get       '/dashboard',          to: 'admin/administration#dashboard'

Você pode renomear o prefixo dos auxiliares de caminho adicionando um parâmetro as

scope 'admin', as: :administration do
  get 'dashboard'
end

# => administration_dashboard_path

Rails fornece uma maneira conveniente de fazer tudo acima, usando o método namespace. As seguintes declarações são equivalentes

namespace :admin do
end

scope 'admin', module: :admin, as: :admin

Escopo pelo controlador

scope controller: :management do
  get 'dashboard'
  get 'performance'
end

Isso gera essas rotas

get       '/dashboard',          to: 'management#dashboard'
get       '/performance',        to: 'management#performance'

Ninho raso

As rotas de recursos aceitam uma opção :shallow que ajuda a encurtar URLs sempre que possível. Os recursos não devem ser aninhados com mais de um nível de profundidade. Uma maneira de evitar isso é criando rotas rasas. O objetivo é deixar de fora os segmentos de URL da coleção pai onde eles não são necessários. O resultado final é que as únicas rotas aninhadas geradas são para as ações :index , :create e :new. O resto é mantido em seu próprio contexto de URL superficial. Há duas opções de escopo para rotas rasas personalizadas:

  • :shallow_path: prefixa os caminhos dos membros com um parâmetro especificado

    scope shallow_path: "sekret" do
  resources :articles do
    resources :comments, shallow: true
  end
end

  • :shallow_prefix: adiciona parâmetros especificados a auxiliares nomeados

    scope shallow_prefix: "sekret" do
  resources :articles do
    resources :comments, shallow: true
  end
end

Também podemos ilustrar rotas rasas mais:

resources :auctions, shallow: true do
  resources :bids do
   resources :comments
  end
end

alternativamente codificado da seguinte forma (se você estiver satisfeito com o bloco):

resources :auctions do
 shallow do
   resources :bids do
     resources :comments
   end
 end
end

Preocupações #

Para evitar a repetição em rotas aninhadas, as preocupações fornecem uma ótima maneira de compartilhar recursos comuns que são reutilizáveis. Para criar uma preocupação use o método concern dentro do arquivo routes.rb. O método espera um símbolo e um bloco:

concern :commentable do
  resources :comments
end

Embora não crie rotas em si, este código permite usar o atributo :concerns em um recurso. O exemplo mais simples seria:

resource :page, concerns: :commentable

O recurso aninhado equivalente ficaria assim:

resource :page do
  resource :comments
end

Isso construiria, por exemplo, as seguintes rotas:

/pages/#{page_id}/comments
/pages/#{page_id}/comments/#{comment_id}

Para que as preocupações sejam significativas, deve haver vários recursos que utilizam a preocupação. Recursos adicionais podem usar qualquer uma das seguintes sintaxes para chamar a preocupação:

resource :post, concerns: %i(commentable)
resource :blog do
  concerns :commentable
end

Rota raiz #

Você pode adicionar uma rota de página inicial ao seu aplicativo com o método root.

# config/routes.rb
Rails.application.routes.draw do
  root "application#index"
  # equivalent to:
  # get "/", "application#index"  
end

# app/controllers/application_controller.rb
class ApplicationController < ActionController::Base
  def index
    render "homepage"
  end
end

E no terminal, rake routes (rails routes no Rails 5) produzirá:

root     GET    /         application#index

Como a página inicial geralmente é a rota mais importante e as rotas são priorizadas na ordem em que aparecem, a rota root geralmente deve ser a primeira em seu arquivo de rotas.

Ações RESTful adicionais #

resources :photos do
  member do
    get 'preview'
  end
  collection do
    get 'dashboard'
  end
end

Isso cria as seguintes rotas além das 7 rotas RESTful padrão:

get       '/photos/:id/preview',          to: 'photos#preview'
get       '/photos/dashboards',           to: 'photos#dashboard'

Se você quiser fazer isso para linhas únicas, você pode usar:

resources :photos do
  get 'preview',   on: :member
  get 'dashboard', on: :collection
end

Você também pode adicionar uma ação ao caminho /new:

resources :photos do
  get 'preview', on: :new
end

Que irá criar:

get       '/photos/new/preview',          to: 'photos#preview'

Esteja atento ao adicionar ações às suas rotas RESTful, provavelmente está faltando outro recurso!

Divida as rotas em vários arquivos #

Se o seu arquivo de rotas for extremamente grande, você pode colocar suas rotas em vários arquivos e incluir cada um dos arquivos com o método require_relative do Ruby:

config/routes.rb: #

YourAppName::Application.routes.draw do
  require_relative 'routes/admin_routes'
  require_relative 'routes/sidekiq_routes'
  require_relative 'routes/api_routes'
  require_relative 'routes/your_app_routes'
end

config/routes/api_routes.rb: #

YourAppName::Application.routes.draw do
  namespace :api do
    # ...
  end
end

Rotas de Sócios e Coleções #

Definir um bloco de membro dentro de um recurso cria uma rota que pode atuar em um membro individual dessa rota baseada em recurso:

resources :posts do
  member do
    get 'preview'
  end
end

Isso gera a seguinte rota de membro:

get '/posts/:id/preview', to: 'posts#preview'
# preview_post_path

As rotas de coleção permitem a criação de rotas que podem atuar em uma coleção de objetos de recursos:

resources :posts do
  collection do
    get 'search'
  end
end

Isso gera a seguinte rota de coleta:

get '/posts/search', to: 'posts#search'
# search_posts_path

Uma sintaxe alternativa:

resources :posts do
  get 'preview', on: :member
  get 'search',  on: :collection
end

Redirecionamento #

Você pode realizar o redirecionamento em rotas Rails da seguinte forma:

get '/stories', to: redirect('/posts')

match "/abc" => redirect("http://example.com/abc")

Você também pode redirecionar todas as rotas desconhecidas para um determinado caminho:

match '*path' => redirect('/'), via: :get
# or
get '*path' => redirect('/')

match '*path' => redirect('/')

Monte outro aplicativo #

mount é usado para montar outro aplicativo (basicamente aplicativo de rack) ou mecanismos de trilhos a serem usados ​​dentro do aplicativo atual

sintaxe:

mount SomeRackApp, at: "some_route"

Agora você pode acessar o aplicativo montado acima usando o auxiliar de rota some_rack_app_path ou some_rack_app_url.

Mas se você quiser renomear esse nome de ajudante, você pode fazê-lo como:

mount SomeRackApp, at: "some_route", as: :myapp

Isso irá gerar os auxiliares myapp_path e myapp_url que podem ser usados ​​para navegar para este aplicativo montado.

Localidades disponíveis no escopo #

Se seu aplicativo estiver disponível em idiomas diferentes, você geralmente mostra a localidade atual na URL.

scope '/(:locale)', locale: /#{I18n.available_locales.join('|')}/ do
    root 'example#root'
    # other routes
end

Sua raiz estará acessível através das localidades definidas em I18n.available_locales.

Redirecionamentos e rotas curinga #

Se você deseja fornecer um URL por conveniência para seu usuário, mas mapeie-o diretamente para outro que você já esteja usando. Use um redirecionamento:

# config/routes.rb
TestApp::Application.routes.draw do
  get 'courses/:course_name' => redirect('/courses/%{course_name}/lessons'), :as => "course"
end

Bem, isso ficou interessante rápido. O princípio básico aqui é apenas usar o método #redirect para enviar uma rota para outra rota. Se sua rota é bastante simples, é um método realmente direto. Mas se você quiser também enviar os parâmetros originais, você precisa fazer um pouco de ginástica capturando o parâmetro dentro de %{aqui}. Observe as aspas simples em tudo.

No exemplo acima, também renomeamos a rota por conveniência usando um alias com o parâmetro :as. Isso nos permite usar esse nome em métodos como os auxiliares #_path. Novamente, teste suas $ rake routes com perguntas.

Parâmetros de URL com um ponto #

Se você quiser dar suporte a um parâmetro de url mais complexo do que um número de id, poderá ter problemas com o analisador se o valor contiver um ponto. Qualquer coisa após um período será considerado um formato (ou seja, json, xml).

Você pode contornar essa limitação usando uma restrição para ampliar a entrada aceita.

Por exemplo, se você quiser fazer referência a um registro de usuário por endereço de e-mail na url:

resources :users, constraints: { id: /.*/ }
