Como implantar seu aplicativo #

Certifique-se de estar no diretório que contém seu aplicativo Rails e crie um aplicativo no Heroku.

$ heroku create example
Creating ⬢ example... done
https://example.herokuapp.com/ | https://git.heroku.com/example.git

A primeira URL da saída, http://example.herokuapp.com, é o local em que o aplicativo está disponível. A segunda URL, git@heroku.com:example.git, é a URL do repositório git remoto.

Este comando deve ser usado apenas em um repositório git inicializado. O comando heroku create adiciona automaticamente um git remote chamado “heroku” apontando para esta URL.

O argumento do nome do aplicativo (“exemplo”) é opcional. Se nenhum nome de aplicativo for especificado, um nome aleatório será gerado. Como os nomes dos aplicativos Heroku estão em um namespace global, você pode esperar que nomes comuns, como “blog” ou “wiki”, já sejam usados. Muitas vezes, é mais fácil começar com um nome padrão e renomear o aplicativo posteriormente.

Se você estiver usando o banco de dados em seu aplicativo, precisará migrar manualmente o banco de dados executando:

$ heroku run rake db:migrate

Quaisquer comandos após heroku run serão executados em um dinamômetro Heroku. Você pode obter uma sessão de shell interativa executando:

$ heroku run bash

Certifique-se de ter um dyno executando o tipo de processo da web:

$ heroku ps:scale web=1

O comando heroku ps lista os dynos em execução do seu aplicativo:

$ heroku ps
=== web (Standard-1X): bin/rails server -p $PORT -e $RAILS_ENV (1)
web.1: starting 2016/07/08 12:09:06 -0500 (~ 2s ago)

Agora você pode visitar o aplicativo em nosso navegador com heroku open.

$ heroku open

O Heroku fornece uma URL da web padrão no domínio herokuapp.com. Quando estiver pronto para escalar para produção, você poderá adicionar seu próprio domínio personalizado.

Gerenciando ambientes de produção e preparação para um Heroku #

Todo aplicativo Heroku é executado em pelo menos dois ambientes: no Heroku (chamaremos isso de produção) e em sua máquina local (desenvolvimento). Se mais de uma pessoa estiver trabalhando no aplicativo, você terá vários ambientes de desenvolvimento - geralmente um por máquina. Normalmente, cada desenvolvedor também terá um ambiente de teste para execução de testes. Infelizmente, essa abordagem falha à medida que os ambientes se tornam menos semelhantes. Windows e Macs, por exemplo, fornecem ambientes diferentes da pilha Linux no Heroku, então você nem sempre pode ter certeza de que o código que funciona em seu ambiente de desenvolvimento local funcionará da mesma maneira quando você o implantar na produção.

A solução é ter um ambiente de preparação que seja o mais semelhante possível à produção. Isso pode ser feito criando um segundo aplicativo Heroku que hospeda seu aplicativo de teste. Com o teste, você pode verificar seu código em uma configuração semelhante à produção antes de afetar seus usuários reais.

Começando do zero

Suponha que você tenha um aplicativo em execução em sua máquina local e esteja pronto para enviá-lo para o Heroku. Precisaremos criar ambientes remotos, preparação e produção. Para adquirir o hábito de empurrar para o staging primeiro, começaremos com isso:

$ heroku create --remote staging
Creating strong-river-216.... done
http://strong-river-216.heroku.com/ | https://git.heroku.com/strong-river-216.git
Git remote staging added

Por padrão, a CLI do heroku cria projetos com um heroku git remote. Aqui, estamos especificando um nome diferente com o sinalizador –remote, então enviar código para Heroku e executar comandos no aplicativo parece um pouco diferente do git push heroku master normal:

 $ git push staging master
...
 $ heroku ps --remote staging
=== web: `bundle exec puma -C config/puma.rb``
web.1: up for 21s

Depois que seu aplicativo de teste estiver funcionando corretamente, você poderá criar seu aplicativo de produção:

$ heroku create --remote production
Creating fierce-ice-327.... done
http://fierce-ice-327.heroku.com/ | https://git.heroku.com/fierce-ice-327.git
Git remote production added
$ git push production master
...
$ heroku ps --remote production
=== web: `bundle exec puma -C config/puma.rb
web.1: up for 16s

E com isso, você tem a mesma base de código em execução como dois aplicativos Heroku separados - um de teste e um de produção, configurados de forma idêntica. Apenas lembre-se que você terá que especificar qual aplicativo você vai operar em seu trabalho diário. Você pode usar o sinalizador ‘–remote’ ou usar sua configuração do git para especificar um aplicativo padrão.
