Action Controller é o C em MVC. Após o roteador determinar qual controlador usar para uma solicitação, o controlador é responsável por entender a solicitação e produzir a saída.

O controlador receberá a solicitação, buscará ou salvará dados de um modelo e usará uma visualização para criar a saída. Um controlador pode ser pensado como um intermediário entre modelos e visualizações. Ele disponibiliza os dados do modelo para a exibição para que possam ser exibidos ao usuário e salva ou atualiza os dados do usuário no modelo.

Filtros #

Filtros são métodos que são executados “antes”, “depois” ou “ao redor” de uma ação do controlador. Eles são herdados, portanto, se você definir algum no seu ApplicationController, eles serão executados para cada solicitação que seu aplicativo receber.

Antes do filtro

Antes que os filtros sejam executados antes da ação do controlador e possam interromper a solicitação (e/ou redirecionar). Um uso comum é verificar se um usuário está logado:

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

Antes que os filtros sejam executados em solicitações antes que a solicitação chegue à ação do controlador. Ele pode retornar uma resposta e ignorar completamente a ação.

Outros usos comuns de filtros antes são validar a autenticação de um usuário antes de conceder acesso à ação designada para lidar com sua solicitação. Também os vi usados ​​para carregar um recurso do banco de dados, verificar permissões em um recurso ou gerenciar redirecionamentos em outras circunstâncias.

Após o filtro

Os filtros posteriores são semelhantes aos “antes”, mas à medida que são executados após a execução da ação, eles acessam o objeto de resposta que está prestes a ser enviado. Então, em resumo, depois que os filtros são executados após a conclusão da ação. Ele pode modificar a resposta. Na maioria das vezes, se algo é feito em um filtro posterior, isso pode ser feito na própria ação, mas se houver alguma lógica a ser executada após executar qualquer um de um conjunto de ações, um filtro posterior é um bom lugar para fazer isto.

Geralmente, eu vi depois e em torno de filtros usados ​​para registro.

Ao redor do filtro

Os filtros ao redor podem ter lógica antes e depois da ação ser executada. Ele simplesmente cede à ação em qualquer lugar que seja necessário. Observe que ele não precisa ceder à ação e pode ser executado sem fazê-lo, como um filtro anterior.

Os filtros ao redor são responsáveis ​​por executar suas ações associadas por rendimento, semelhante a como os middlewares Rack funcionam.

Em torno de retornos de chamada envolvem a execução de ações. Você pode escrever um retorno de chamada em dois estilos diferentes. No primeiro, o retorno de chamada é um único pedaço de código. Esse código é chamado antes que a ação seja executada. Se o código de retorno de chamada invocar yield, a ação será executada. Quando a ação é concluída, o código de retorno de chamada continua sendo executado. Assim, o código antes do yield é como um callback antes da ação e o código após o yield é o callback depois da ação. Se o código de retorno de chamada nunca invocar yield. a ação não é executada - isso é o mesmo que ter um retorno de chamada antes da ação retornar false.

Aqui está um exemplo do filtro ao redor:

around_filter :catch_exceptions
 
private
  def catch_exceptions
    begin
      yield
    rescue Exception => e 
      logger.debug "Caught exception! #{e.message}"
    end
  end

Isso capturará a exceção de qualquer ação e colocará a mensagem em seu log. Você pode usar filtros para manipulação de exceções, configuração e desmontagem e uma infinidade de outros casos.

Apenas e Exceto

Todos os filtros podem ser aplicados a ações específicas, usando :only e :except:

class ProductsController < ApplicationController
  before_action :set_product, only: [:show, :edit, :update]

  # ... controller actions

  # Define your filters as controller private methods
  private

  def set_product
    @product = Product.find(params[:id])
  end
end

Pular filtro

Todos os filtros (herdados também) também podem ser ignorados para algumas ações específicas:

class ApplicationController < ActionController::Base
  before_action :authenticate_user!

  def authenticate_user!
    redirect_to some_path unless user_signed_in?
  end
end

class HomeController < ApplicationController
  skip_before_action :authenticate_user!, only: [:index]

  def index
  end
end

Como eles são herdados, os filtros também podem ser definidos em um controlador “pai” namespace. Digamos, por exemplo, que você tenha um namespace admin e, claro, deseja que apenas usuários admin possam acessá-lo. Você poderia fazer algo assim:

# config/routes.rb
namespace :admin do
  resources :products
end

# app/controllers/admin_controller.rb
class AdminController < ApplicationController
  before_action :authenticate_admin_user!

  private

  def authenticate_admin_user!
    redirect_to root_path unless current_user.admin?
  end
end

# app/controllers/admin/products_controller.rb
class Admin::ProductsController < AdminController
  # This controller will inherit :authenticate_admin_user! filter
end

Observe que no Rails 4.x você pode usar before_filter junto com before_action, mas o before_filter está obsoleto no Rails 5.0.0 e será removido no 5.1.

Gerando um controlador #

Rails fornece muitos geradores, para controladores também, é claro.

Você pode gerar um novo controlador executando este comando na pasta do seu aplicativo

rails generate controller NAME [action action] [options]

Nota: Você também pode usar o alias rails g para invocar rails generate

Por exemplo, para gerar um controlador para um modelo Product, com ações #index e #show você executaria

rails generate controller products index show

Isso criará o controlador em app/controllers/products_controller.rb, com ambas as ações que você especificou

class ProductsController < ApplicationController
  def index
  end

  def show
  end
end

Ele também criará uma pasta products dentro de app/views/, contendo os dois modelos para as ações do seu controlador (ou seja, index.html.erb e show.html.erb, note que a extensão pode variar de acordo com ao seu mecanismo de template, então se você estiver usando slim, por exemplo, o generator irá criar index.html.slim e show.html.slim )

Além disso, se você especificou alguma ação, elas também serão adicionadas ao seu arquivo routes

# config/routes.rb
get 'products/show'
get 'products/index'

Rails cria um arquivo auxiliar para você, em app/helpers/products_helper.rb, e também os arquivos assets em app/assets/javascripts/products.js e app/assets/stylesheets/products.css. Quanto às visualizações, o gerador altera esse comportamento de acordo com o que está especificado em seu Gemfile: ou seja, se você estiver usando Coffeescript e Sass em sua aplicação, o gerador do controlador gerará products.coffee e produtos.sass.

Por último, mas não menos importante, Rails também gera arquivos de teste para seu controller, seu helper e suas views.

Se você não quer que nenhum deles seja criado para você pode dizer ao Rails para ignorá-los, apenas coloque qualquer opção com

--no- ou --skip, assim:

rails generate controller products index show --no-assets --no-helper

E o gerador irá pular tanto assets quanto helper

Se você precisar criar um controlador para um namespace específico, adicione-o na frente de NAME:

rails generate controller admin/products

Isso criará seu controlador dentro de app/controllers/admin/products_controller.rb

Rails também pode gerar um controller RESTful completo para você:

rails generate scaffold_controller MODEL_NAME # available from Rails 4
rails generate scaffold_controller Product

Resgatando ActiveRecord::RecordNotFound com redirect_to #

Você pode resgatar uma exceção RecordNotFound com um redirecionamento em vez de mostrar uma página de erro:

class ApplicationController < ActionController::Base

  # your other stuff

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: I18n.t("errors.record_not_found")
  end
end

404 quando registro não encontrado #

Resgate do erro de registro não encontrado em vez de mostrar uma exceção ou página em branco:

class ApplicationController < ActionController::Base
  
  # ... your other stuff here 

  rescue_from ActiveRecord::RecordNotFound do |exception|
    redirect_to root_path, 404, alert: 'Record not found'
  end
end
