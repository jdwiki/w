Associação polimórfica #

Esse tipo de associação permite que um modelo ActiveRecord pertença a mais de um tipo de registro de modelo. Exemplo comum:

class Human < ActiveRecord::Base
  has_one :address, :as => :addressable
end

class Company < ActiveRecord::Base
  has_one :address, :as => :addressable
end

class Address < ActiveRecord::Base
  belongs_to :addressable, :polymorphic => true
end

Sem essa associação, você teria todas essas chaves estrangeiras em sua tabela de endereços, mas só teria um valor para uma delas porque um endereço, nesse cenário, só pode pertencer a uma entidade (humano ou empresa). Aqui está como seria:

class Address < ActiveRecord::Base
  belongs_to :human
  belongs_to :company
end

pertence a #

Uma associação belongs_to configura uma conexão um-para-um com outro modelo, de modo que cada instância do modelo declarado “pertence” a uma instância do outro modelo.

Por exemplo, se seu aplicativo inclui usuários e postagens, e cada postagem pode ser atribuída a exatamente um usuário, você deve declarar o modelo de postagem desta forma:

class Post < ApplicationRecord
  belongs_to :user
end

Em sua estrutura de tabela, você pode ter

create_table "posts", force: :cascade do |t|
  t.integer  "user_id",  limit: 4
end

Associação Auto-Referencial #

A associação autorreferencial é usada para associar um modelo a si mesmo. O exemplo mais frequente seria gerenciar a associação entre um amigo e seu seguidor.

ex.

rails g model friendship user_id:references friend_id:integer

agora você pode associar modelos como;

class User < ActiveRecord::Base
  has_many :friendships
  has_many :friends, :through => :friendships
  has_many :inverse_friendships, :class_name => "Friendship", :foreign_key => "friend_id"
  has_many :inverse_friends, :through => :inverse_friendships, :source => :user
end

e o outro modelo ficará;

class Friendship < ActiveRecord::Base
  belongs_to :user
  belongs_to :friend, :class_name => "User"
end

tem um #

Uma associação has_one configura uma conexão um-para-um com outro modelo, mas com semântica diferente. Essa associação indica que cada instância de um modelo contém ou possui uma instância de outro modelo.

Por exemplo, se cada usuário em seu aplicativo tiver apenas uma conta, você declarará o modelo de usuário assim:

class User < ApplicationRecord
  has_one :account
end

No Active Record, quando você tem uma relação has_one, o active record garante que existe apenas um registro com a chave estrangeira.

Aqui em nosso exemplo: Na tabela de contas, pode haver apenas um registro com um determinado user_id. Se você tentar associar mais uma conta para o mesmo usuário, ele torna a chave estrangeira da entrada anterior como nula (tornando-a órfã) e cria uma nova automaticamente. Isso torna a entrada anterior nula mesmo que o salvamento falhe para que a nova entrada mantenha a consistência.

user = User.first
user.build_account(name: "sample")
user.save   [Saves it successfully, and creates an entry in accounts table with user_id 1]
user.build_account(name: "sample1")  [automatically makes the previous entry's foreign key null]
user.save  [creates the new account with name sample 1 and user_id 1]

tem muitos #

Uma associação has_many indica uma conexão um-para-muitos com outro modelo. Essa associação geralmente está localizada do outro lado de uma associação belongs_to.

Essa associação indica que cada instância do modelo possui zero ou mais instâncias de outro modelo.

Por exemplo, em um aplicativo contendo usuários e postagens, o modelo de usuário pode ser declarado assim:

class User < ApplicationRecord
  has_many :posts
end

A estrutura da tabela de Post permaneceria a mesma do exemplo belongs_to; em contraste, User não exigiria nenhuma alteração de esquema.

Se você deseja obter a lista de todas as postagens publicadas para o Usuário, você pode adicionar o seguinte (ou seja, você pode adicionar escopos aos seus objetos de associação):

class User < ApplicationRecord
  has_many :published_posts, -> { where("posts.published IS TRUE") }, class_name: "Post"
end

O has_many :through associação #

Uma associação has_many :through é frequentemente usada para configurar uma conexão many-to-many com outro modelo. Essa associação indica que o modelo declarativo pode ser combinado com zero ou mais instâncias de outro modelo procedendo-se a um terceiro modelo.

Por exemplo, considere uma prática médica em que os pacientes marcam consultas com médicos. As declarações de associação relevantes podem ser assim:

class Physician < ApplicationRecord
  has_many :appointments
  has_many :patients, through: :appointments
end

class Appointment < ApplicationRecord
  belongs_to :physician
  belongs_to :patient
end

class Patient < ApplicationRecord
  has_many :appointments
  has_many :physicians, through: :appointments
end

O has_one :through associação #

Uma associação has_one :through configura uma conexão one-to-one com outro modelo. Essa associação indica que o modelo declarante pode ser combinado com uma instância de outro modelo procedendo-se a um terceiro modelo.

Por exemplo, se cada fornecedor tiver uma conta e cada conta estiver associada a um histórico de conta, o modelo de fornecedor poderá ser assim:

class Supplier < ApplicationRecord
  has_one :account
  has_one :account_history, through: :account
end

class Account < ApplicationRecord
  belongs_to :supplier
  has_one :account_history
end

class AccountHistory < ApplicationRecord
  belongs_to :account
end

A associação has_and_belongs_to_many #

Uma associação has_and_belongs_to_many cria uma conexão direta many-to-many com outro modelo, sem nenhum modelo interveniente.

Por exemplo, se seu aplicativo inclui assemblies e parts, com cada montagem tendo muitas peças e cada peça aparecendo em muitas montagens, você poderia declarar os modelos desta forma:

class Assembly < ApplicationRecord
  has_and_belongs_to_many :parts
end

class Part < ApplicationRecord
  has_and_belongs_to_many :assemblies
end
