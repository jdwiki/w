Criando um modelo via gerador #

Ruby on Rails fornece um gerador de model que você pode usar para criar modelos ActiveRecord. Basta usar rails generate model e fornecer o nome do modelo.

$ rails g model user

Além do arquivo de modelo em app/models, o gerador também criará:

  • o Teste em test/models/user_test.rb
  • as luminárias em test/fixtures/users.yml
  • a migração do banco de dados em db/migrate/XXX_create_users.rb

Você também pode gerar alguns campos para o modelo ao gerá-lo.

$ rails g model user email:string sign_in_count:integer birthday:date

Isso criará as colunas email, sign_in_count e birthday em seu banco de dados, com os tipos apropriados.

Criando um modelo manualmente #

Embora usar scaffolding seja rápido e fácil se você é novo no Rails ou está criando um novo aplicativo, mais tarde pode ser útil fazer isso sozinho para evitar a necessidade de passar pelo código gerado pelo scaffold para reduzi-lo (remover peças não utilizadas, etc.).

Criar um modelo pode ser tão simples quanto criar um arquivo em app/models.

O modelo mais simples, em ActiveRecord, é uma classe que estende ActiveRecord::Base.

class User < ActiveRecord::Base
end

Os arquivos de modelo são armazenados em app/models/, e o nome do arquivo corresponde ao nome singular da classe:

# user
app/models/user.rb

# SomeModel
app/models/some_model.rb

A classe herdará todos os recursos do ActiveRecord: métodos de consulta, validações, retornos de chamada, etc.

# Searches the User with ID 1
User.find(1)

Nota: Certifique-se de que existe a tabela para o modelo correspondente. Caso contrário, você pode criar a tabela criando uma Migration

Você pode gerar um modelo e sua migração por terminal a partir do seguinte comando

modelo do Rails g column_name1:data_type1, column_name2:data_type2, ...

e também pode atribuir chave estrangeira (relacionamento) ao modelo seguindo o comando

rails g model column_name:data_type, model_name:references

1: https://www.wikiod.com/pt/ruby-on-rails/activerecord#Criando uma migração

Introdução aos retornos de chamada #

Um retorno de chamada é um método que é chamado em momentos específicos do ciclo de vida de um objeto (logo antes ou depois da criação, exclusão, atualização, validação, salvamento ou carregamento do banco de dados).

Por exemplo, digamos que você tenha uma listagem que expira em 30 dias após a criação.

Uma maneira de fazer isso é assim:

class Listing < ApplicationRecord
  after_create :set_expiry_date

  private

  def set_expiry_date
    expiry_date = Date.today + 30.days
    self.update_column(:expires_on, expiry_date)
  end
end

Todos os métodos disponíveis para callbacks são os seguintes, na mesma ordem em que são chamados durante a operação de cada objeto:

Criando um objeto

  • antes_validação
  • after_validation
  • antes_salvar
  • around_save
  • antes_criar
  • around_create
  • after_create
  • after_save
  • after_commit/after_rollback

Atualizando um objeto

  • antes_validação
  • after_validation
  • antes_salvar
  • around_save
  • antes_atualização
  • around_update
  • after_update
  • after_save
  • after_commit/after_rollback

Destruindo um Objeto

  • antes_destruir
  • around_destroy
  • after_destroy
  • after_commit/after_rollback

OBSERVAÇÃO: after_save é executado tanto na criação quanto na atualização, mas sempre após os retornos de chamada mais específicos after_create e after_update, independentemente da ordem em que as chamadas de macro foram executadas.

Testando manualmente seus modelos #

Testar seus modelos do Active Record por meio de sua interface de linha de comando é simples. Navegue até o diretório app em seu terminal e digite rails console para iniciar o console Rails. A partir daqui, você pode executar métodos de registro ativo em seu banco de dados.

Por exemplo, se você tivesse um esquema de banco de dados com uma tabela Users com uma coluna name:string e email:string, você poderia executar:

User.create name: "John", email: "john@example.com"

Então, para mostrar esse registro, você pode executar:

User.find_by email: "john@example.com"

Ou se este é seu primeiro ou único registro, você pode simplesmente obter o primeiro registro executando:

User.first

Criando uma migração #

Adicionar/remover campos em tabelas existentes #

Crie uma migração executando:

rails generate migration AddTitleToCategories title:string

Isso criará uma migração que adiciona uma coluna title a uma tabela categories:

class AddTitleToCategories < ActiveRecord::Migration[5.0]
  def change
    add_column :categories, :title, :string
  end
end

Da mesma forma, você pode gerar uma migração para remover uma coluna: rails generate migration RemoveTitleFromCategories title:string

Isso criará uma migração que remove uma coluna title da tabela categories:

class RemoveTitleFromCategories < ActiveRecord::Migration[5.0]
  def change
    remove_column :categories, :title, :string
  end
end

Embora, estritamente falando, especificar type (:string neste caso) não seja necessário para remover uma coluna, é útil, pois fornece as informações necessárias para rolá-la de volta.

#Cria uma tabela

Crie uma migração executando:

rails g CreateUsers name bio

O Rails reconhece a intenção de criar uma tabela a partir do prefixo Create, o restante do nome da migração será usado como um nome de tabela. O exemplo dado gera o seguinte:

class CreateUsers < ActiveRecord::Migration
  def change
    create_table :users do |t|
      t.string :name
      t.string :bio
    end
  end
end

Observe que o comando de criação não especificou os tipos de colunas e o padrão string foi usado.

Cria uma tabela de junção #

Crie uma migração executando:

rails g CreateJoinTableParticipation user:references group:references

O Rails detecta a intenção de criar uma tabela de junção localizando JoinTable no nome da migração. Todo o resto é determinado a partir dos nomes dos campos que você fornece após o nome.

class CreateJoinTableParticipation < ActiveRecord::Migration
  def change
    create_join_table :users, :groups do |t|
      # t.index [:user_id, :group_id]
      # t.index [:group_id, :user_id]
    end
  end
end

Descomente as instruções index necessárias e exclua o resto.

Precedência #

Observe que o nome de migração de exemplo CreateJoinTableParticipation corresponde à regra para criação de tabela: ele tem um prefixo Create. Mas não gerou um simples create_table. Isso ocorre porque o gerador de migração (código-fonte) usa uma primeira correspondência da lista a seguir:

  • (Adicionar|Remover)<ignorado>(Para|De)<table_name>
  • <ignorado>JoinTable<ignorado>
  • Criar<table_name>

Crie uma tabela de junção usando migrações #

Especialmente útil para a relação has_and_belongs_to_many, você pode criar manualmente uma tabela de junção usando o método create_table. Suponha que você tenha dois modelos Tags e Proyects, e gostaria de associá-los usando uma relação has_and_belongs_to_many. Você precisa de uma tabela de junção para associar instâncias de ambas as classes.

class CreateProjectsTagsJoinTableMigration < ActiveRecord::Migration
  def change
    create_table :projects_tags, id: false do |t|
      t.integer :project_id
      t.integer :tag_id
    end
  end
end

O nome real da tabela precisa seguir esta convenção: o modelo que precede alfabeticamente o outro deve vir primeiro. Pprojeto precede Tags, então o nome da tabela é projects_tags.

Além disso, como o objetivo desta tabela é rotear a associação entre as instâncias de dois modelos, o id real de cada registro nesta tabela não é necessário. Você especifica isso passando id: false

Finalmente, como é convenção no Rails, o nome da tabela deve estar na forma plural composta dos modelos individuais, mas a coluna da tabela deve estar na forma singular.
