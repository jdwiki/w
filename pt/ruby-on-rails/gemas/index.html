Documentação do Gemfile #

Para projetos que devem crescer, é uma boa ideia adicionar comentários ao seu Gemfile. Dessa forma, mesmo em grandes configurações, você ainda saberá o que cada gem faz, mesmo que o nome não seja autoexplicativo e você o tenha adicionado há 2 anos.

Isso também pode ajudá-lo a lembrar por que escolheu uma determinada versão e, consequentemente, reavaliar o requisito de versão posteriormente.

Exemplos:

# temporary downgrade for TeamCity
gem 'rake', '~> 10.5.0'
# To upload invoicing information to payment provider
gem 'net-sftp'

O que é uma jóia? #

Uma gem é o equivalente a um plugin ou extensão para a linguagem de programação ruby.

Para ser exato, mesmo os trilhos nada mais são do que uma jóia. Muitas gems são construídas em trilhos ou outras gems (elas são dependentes da dita gem) ou são independentes.

Em seu projeto Rails #

Gemfile #

Para o seu projeto Rails você tem um arquivo chamado Gemfile. Aqui você pode adicionar gems que deseja incluir e usar em seu projeto. Uma vez adicionado você precisa instalar a gem usando bundler (Veja a seção Bundler).

Gemfile.lock #

Depois de fazer isso, seu Gemfile.lock será atualizado com suas gems recém-adicionadas e suas dependências. Este arquivo bloqueia suas gems usadas para que elas usem essa versão específica declarada nesse arquivo.

GEMA
remoto: https://rubygems.org/
especificações:
conceber (4.0.3)
bcrypt (~> 3.0)
orm_adapter (~> 0,1)
trilhos (>= 4.1.0, < 5.1)
respondedores
diretor (~> 1.2.3)

Este exemplo é para a gem devise. No Gemfile.lock a versão 4.0.3 é declarada, para informar ao instalar seu projeto em outra máquina ou em seu servidor de produção qual versão especificada usar.

Desenvolvimento #

Ou uma única pessoa, um grupo ou uma comunidade inteira trabalha e mantém uma joia. O trabalho feito geralmente é liberado depois que certos problemas foram corrigidos ou recursos foram adicionados.

Normalmente os lançamentos seguem o princípio Semantic Versioning 2.0.0.

Agrupador #

A maneira mais fácil de lidar e gerenciar gems é usando o bundler. Bundler é um gerenciador de pacotes comparável ao bower.

Para usar o bundler, primeiro você precisa instalá-lo.

gem install bundler

Depois de ter o bundler instalado e funcionando, tudo o que você precisa fazer é adicionar gems ao seu Gemfile e executar

bundle

em seu terminal. Isso instala suas gems recém-adicionadas ao seu projeto. Caso surja um problema, você receberá um prompt no seu terminal.

Se você estiver interessado em mais detalhes, sugiro que dê uma olhada nos docs.

Gemfiles #

Para começar, os gemfiles requerem pelo menos uma fonte, na forma de URL para um servidor RubyGems.

Gere um Gemfile com a fonte padrão rubygems.org executando bundle init. Use https para que sua conexão com o servidor seja verificada com SSL.

source 'https://rubygems.org'

Em seguida, declare as gems que você precisa, incluindo os números de versão.

gem 'rails', '4.2.6'
gem 'rack',  '>=1.1'
gem 'puma',  '~>3.0'

A maioria dos especificadores de versão, como >= 1.0, são autoexplicativos. O especificador ~> tem um significado especial. ~> 2.0.3 é idêntico a >= 2.0.3 e < 2.1. ~> 2.1 é idêntico a >= 2.1 e < 3.0. ~> 2.2.beta corresponderá a versões de pré-lançamento como 2.2.beta.12.

Os repositórios Git também são fontes de gem válidas, desde que o repositório contenha uma ou mais gems válidas. Especifique o que fazer check-out com :tag, :branch ou :ref. O padrão é o branch master.

gem 'nokogiri', :git => 'https://github.com/sparklemotion/nokogiri', :branch => 'master'

Se você quiser usar uma gem descompactada diretamente do sistema de arquivos, basta definir a opção :path para o caminho que contém os arquivos da gem.

gem 'extracted_library', :path => './vendor/extracted_library'

As dependências podem ser colocadas em grupos. Os grupos podem ser ignorados no momento da instalação (usando --without) ou necessários todos de uma vez (usando Bundler.require).

gem 'rails_12factor', group: :production

group :development, :test do
  gem 'byebug'
  gem 'web-console', '~> 2.0'
  gem 'spring'
  gem 'dotenv-rails'
end

Você pode especificar a versão necessária do Ruby no Gemfile com ruby. Se o Gemfile for carregado em uma versão diferente do Ruby, o Bundler lançará uma exceção com uma explicação.

ruby '2.3.1'

Gemas #

Se você estiver usando o RVM(Ruby Version Manager), então usar um gemset para cada projeto é uma boa ideia. Um gemset é apenas um contêiner que você pode usar para manter as gems separadas umas das outras. Criar um gemset por projeto permite que você altere gems (e versões de gems) para um projeto sem quebrar todos os seus outros projetos. Cada projeto só precisa se preocupar com suas próprias joias.

RVM fornece (>= 0.1.8) um ​​@global gemset por interpretador ruby. Gems que você instala no @global gemset para um determinado ruby ​​estão disponíveis para todos os outros gemsets que você cria em associação com esse ruby. Esta é uma boa maneira de permitir que todos os seus projetos compartilhem a mesma gem instalada para uma instalação específica do interpretador Ruby.

Criando conjuntos de gemas

Suponha que você já tenha o ruby-2.3.1 instalado e o tenha selecionado usando este comando:

rvm use ruby-2.3.1

Agora para criar gemset para esta versão ruby:

rvm gemset create new_gemset

onde o new_gemset é o nome do gemset. Para ver a lista de gemsets disponíveis para uma versão ruby:

rvm gemset list

para listar as gemas de todas as versões do rubi:

rvm gemset list_all

para usar um gemset da lista (suponha que new_gemset seja o gemset que eu quero usar):

rvm gemset use new_gemset

você também pode especificar a versão ruby ​​com o gemset se quiser mudar para alguma outra versão ruby:

rvm use ruby-2.1.1@new_gemset

para especificar um gemset padrão para uma versão ruby ​​específica:

rvm use 2.1.1@new_gemset --default

para remover todas as gems instaladas de um gemset, você pode esvaziá-lo:

rvm gemset empty new_gemset

para copiar um gemset de um rubi para outro, você pode fazer isso:

rvm gemset copy 2.1.1@rails4 2.1.2@rails4

para excluir um conjunto de gemas:

rvm gemset delete new_gemset

para ver o nome do gemset atual:

rvm gemset name

para instalar uma gem no gemset global:

rvm @global do gem install ...

Inicializando Gemsets durante as instalações do Ruby

Quando você instala um novo ruby, o RVM não apenas cria dois gemsets (o gemset padrão vazio e o gemset global), mas também usa um conjunto de arquivos editáveis ​​pelo usuário para determinar quais gems instalar.

Trabalhando em ~/.rvm/gemsets, o rvm procura por global.gems e default.gems usando uma hierarquia de árvore baseada na string ruby ​​que está sendo instalada. Usando o exemplo de ree-1.8.7-p2010.02, rvm irá verificar (e importar) os seguintes arquivos:

~/.rvm/gemsets/ree/1.8.7/p2010.02/global.gems
~/.rvm/gemsets/ree/1.8.7/p2010.02/default.gems
~/.rvm/gemsets/ree/1.8.7/global.gems
~/.rvm/gemsets/ree/1.8.7/default.gems
~/.rvm/gemsets/ree/global.gems
~/.rvm/gemsets/ree/default.gems
~/.rvm/gemsets/global.gems
~/.rvm/gemsets/default.gems

Por exemplo, se você editou ~/.rvm/gemsets/global.gems adicionando estas duas linhas:

bundler
awesome_print

toda vez que você instala um novo ruby, essas duas gems são instaladas em seu gemset global. Os arquivos default.gems e global.gems são geralmente sobrescritos durante a atualização do rvm.
