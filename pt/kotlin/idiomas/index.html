Serializable e serialVersionUid em Kotlin #

Para criar o serialVersionUID para uma classe no Kotlin, você tem algumas opções, todas envolvendo a adição de um membro ao objeto complementar da classe.

O bytecode mais conciso vem de um private const val que se tornará uma variável estática privada na classe que o contém, neste caso MySpecialCase:

class MySpecialCase : Serializable {
    companion object {
        private const val serialVersionUID: Long = 123
    }
}

Você também pode usar esses formulários, cada um com um efeito colateral de ter métodos getter/setter que não são necessários para serialização…

class MySpecialCase : Serializable {
    companion object {
        private val serialVersionUID: Long = 123
    }
}

Isso cria o campo estático, mas também cria um getter getSerialVersionUID no objeto complementar que é desnecessário.

class MySpecialCase : Serializable {
    companion object {
        @JvmStatic private val serialVersionUID: Long = 123
    }
}

Isso cria o campo estático, mas também cria um getter estático, bem como getSerialVersionUID na classe que o contém MySpecialCase, o que é desnecessário.

Mas todos funcionam como um método de adicionar o serialVersionUID a uma classe Serializable.

Delegue para uma classe sem fornecê-la no construtor público #

Suponha que você queira delegar a uma classe, mas não queira fornecer a classe delegada no parâmetro do construtor. Em vez disso, você deseja construí-lo de forma privada, tornando o chamador do construtor inconsciente dele. A princípio, isso pode parecer impossível porque a delegação de classe permite delegar apenas aos parâmetros do construtor. No entanto, existe uma maneira de fazer isso, conforme fornecido em esta resposta:

class MyTable private constructor(table: Table<Int, Int, Int>) : Table<Int, Int, Int> by table {

    constructor() : this(TreeBasedTable.create()) // or a different type of table if desired

}

Com isso, você pode simplesmente chamar o construtor de MyTable assim: MyTable(). A Table<Int, Int, Int> para a qual MyTable delega será criada de forma privada. Construtor chamador não sabe nada sobre isso.

Este exemplo é baseado em esta pergunta SO.

Use let ou também para simplificar o trabalho com objetos anuláveis #

let em Kotlin cria uma ligação local a partir do objeto que foi chamado. Exemplo:

val str = "foo"
str.let {
    println(it) // it
}

Isso imprimirá "foo" e retornará Unit.

A diferença entre let e also é que você pode retornar qualquer valor de um bloco let. also por outro lado sempre retornará Unidade.

Agora, por que isso é útil, você pergunta? Porque se você chamar um método que pode retornar null e quiser executar algum código apenas quando esse valor de retorno não for null, você pode usar let ou also assim:

val str: String? = someFun()
str?.let {
    println(it)
}

Este pedaço de código só irá executar o bloco let quando str não for null. Observe o operador de segurança null (?).

Filtrando uma lista #

val list = listOf(1,2,3,4,5,6)

//filter out even numbers

val even = list.filter { it % 2 == 0 }

println(even) //returns [2,4]

Métodos fluentes em Kotlin #

Métodos fluentes em Kotlin podem ser os mesmos que Java:

fun doSomething() {
   someOtherAction()
   return this
}

Mas você também pode torná-los mais funcionais criando uma função de extensão como:

fun <T: Any> T.fluently(func: ()->Unit): T {
    func()
    return this
}

O que então permite funções mais obviamente fluentes:

fun doSomething() {
   return fluently { someOtherAction() }
}

Use apply para inicializar objetos ou para alcançar o encadeamento de métodos #

A documentação do apply diz o seguinte:

chama o bloco de função especificado com o valor this como seu receptor e retorna o valor this.

Embora o kdoc não seja tão útil, apply é de fato uma função útil. Em termos leigos, apply estabelece um escopo no qual this é vinculado ao objeto que você chamou de apply. Isso permite que você poupe algum código quando precisar chamar vários métodos em um objeto que você retornará mais tarde. Exemplo:

File(dir).apply { mkdirs() }

Isso é o mesmo que escrever isso:

fun makeDir(String path): File {
    val result = new File(path)
    result.mkdirs()
    return result
}

Criando DTOs (POJOs/POCOs) #

As classes de dados em kotlin são classes criadas para não fazer nada além de armazenar dados. Essas classes são marcadas como data:

data class User(var firstname: String, var lastname: String, var age: Int)

O código acima cria uma classe User com o seguinte gerado automaticamente:

  • Getters e Setters para todas as propriedades (getters apenas para vals)
  • igual()
  • hashcode()
  • toString()
  • copiar()
  • componentN() (onde N é a propriedade correspondente na ordem de declaração)

Assim como com uma função, os valores padrão também podem ser especificados:

data class User(var firstname: String = "Joe", var lastname: String = "Bloggs", var age: Int = 20)
