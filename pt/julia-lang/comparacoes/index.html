Sintaxe #

  • x < y # se x for estritamente menor que y
  • x > y # se x for estritamente maior que y
  • x == y # se x for igual a y
  • x === y # alternativamente x ≡ y, se x for igual a y
  • x ≤ y # alternativamente x <= y, se x for menor ou igual a y
  • x ≥ y # alternativamente x >= y, se x for maior ou igual a y
  • x ≠ y # alternativamente x != y, se x não for igual a y
  • x ≈ y # se x for aproximadamente igual a y

Tenha cuidado ao virar os sinais de comparação. Julia define muitas funções de comparação por padrão sem definir a versão invertida correspondente. Por exemplo, pode-se executar

julia> Set(1:3) ⊆ Set(0:5)
true

mas não dá para fazer

julia> Set(0:5) ⊇ Set(1:3)
ERROR: UndefVarError: ⊇ not defined

Julia oferece suporte a um conjunto muito grande de operadores de comparação. Esses incluem

  2. Todos os símbolos do ponto 1, precedidos de um ponto (.) devem ser feitos por elementos;

Nem todos eles têm uma definição na biblioteca padrão Base. No entanto, eles estão disponíveis para outros pacotes definirem e usarem conforme apropriado.

No uso diário, a maioria desses operadores de comparação não é relevante. As mais comuns utilizadas são as funções matemáticas padrão para ordenação; consulte a seção Sintaxe para obter uma lista.

Como a maioria dos outros operadores em Julia, os operadores de comparação são funções e podem ser chamados como funções. Por exemplo, (<)(1, 2) tem o mesmo significado de 1 < 2.

Comparações encadeadas #

Vários operadores de comparação usados ​​juntos são encadeados, como se estivessem conectados por meio do operador &&. Isso pode ser útil para cadeias de comparação legíveis e matematicamente concisas, como

# same as 0 < i && i <= length(A)
isinbounds(A, i)       = 0 < i ≤ length(A)

# same as Set() != x && issubset(x, y)
isnonemptysubset(x, y) = Set() ≠ x ⊆ y

No entanto, há uma diferença importante entre a > b > c e a > b && b > c; neste último, o termo b é avaliado duas vezes. Isso não importa muito para símbolos antigos simples, mas pode importar se os próprios termos tiverem efeitos colaterais. Por exemplo,

julia> f(x) = (println(x); 2)
f (generic function with 1 method)

julia> 3 > f("test") > 1
test
true

julia> 3 > f("test") && f("test") > 1
test
test
true

Vamos dar uma olhada mais profunda nas comparações encadeadas e como elas funcionam, vendo como elas são analisadas e reduzidas em expressões. Primeiro, considere a comparação simples, que podemos ver que é apenas uma chamada de função antiga:

julia> dump(:(a > b))
Expr
  head: Symbol call
  args: Array{Any}((3,))
    1: Symbol >
    2: Symbol a
    3: Symbol b
  typ: Any

Agora, se encadearmos a comparação, notamos que a análise foi alterada:

julia> dump(:(a > b >= c))
Expr
  head: Symbol comparison
  args: Array{Any}((5,))
    1: Symbol a
    2: Symbol >
    3: Symbol b
    4: Symbol >=
    5: Symbol c
  typ: Any

Após a análise, a expressão é reduzida para sua forma final:

julia> expand(:(a > b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

e notamos de fato que isso é o mesmo que para a > b && b >= c:

julia> expand(:(a > b && b >= c))
:(begin 
        unless a > b goto 3
        return b >= c
        3: 
        return false
    end)

Números ordinais #

Veremos como implementar comparações personalizadas implementando um tipo personalizado, números ordinais. Para simplificar a implementação, focaremos em um pequeno subconjunto desses números: todos os números ordinais até, mas não incluindo, ε₀. Nossa implementação é focada na simplicidade, não na velocidade; no entanto, a implementação também não é lenta.

Armazenamos números ordinais por sua forma normal do Cantor. Como a aritmética ordinal não é comutativa, usaremos primeiro a convenção comum de armazenar os termos mais significativos.

immutable OrdinalNumber <: Number
    βs::Vector{OrdinalNumber}
    cs::Vector{Int}
end

Como a forma normal de Cantor é única, podemos testar a igualdade simplesmente por meio da igualdade recursiva:

Na versão v0.5, existe uma sintaxe muito boa para fazer isso de forma compacta:

import Base: ==
α::OrdinalNumber == β::OrdinalNumber = α.βs == β.βs && α.cs == β.cs

Caso contrário, defina a função como é mais típico:

import Base: ==
==(α::OrdinalNumber, β::OrdinalNumber) = α.βs == β.βs && α.cs == β.cs

Para finalizar nosso pedido, pois esse tipo tem um pedido total, devemos sobrecarregar a função isless:

import Base: isless
function isless(α::OrdinalNumber, β::OrdinalNumber)
    for i in 1:min(length(α.cs), length(β.cs))
        if α.βs[i] < β.βs[i]
            return true
        elseif α.βs[i] == β.βs[i] && α.cs[i] < β.cs[i]
            return true
        end
    end
    return length(α.cs) < length(β.cs)
end

Para testar nossa ordem, podemos criar alguns métodos para fazer números ordinais. Zero, é claro, é obtido por não ter termos na forma normal de Cantor:

const ORDINAL_ZERO = OrdinalNumber([], [])
Base.zero(::Type{OrdinalNumber}) = ORDINAL_ZERO

Podemos definir um expω para calcular ω^α e usá-lo para calcular 1 e ω:

expω(α) = OrdinalNumber([α], [1])
const ORDINAL_ONE = expω(ORDINAL_ZERO)
Base.one(::Type{OrdinalNumber}) = ORDINAL_ONE
const ω = expω(ORDINAL_ONE)

Agora temos uma função de ordenação totalmente funcional em números ordinais:

julia> ORDINAL_ZERO < ORDINAL_ONE < ω < expω(ω)
true

julia> ORDINAL_ONE > ORDINAL_ZERO
true

julia> sort([ORDINAL_ONE, ω, expω(ω), ORDINAL_ZERO])

4-element Array{OrdinalNumber,1}:
                                                                                                       OrdinalNumber(OrdinalNumber[],Int64[])
                                                                     OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])
                                   OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])
 OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[OrdinalNumber(OrdinalNumber[],Int64[])],[1])],[1])],[1])

No último exemplo, vemos que a impressão dos números ordinais poderia ser melhor, mas o resultado é o esperado.

Usando ==, === e isequal #

Existem três operadores de igualdade: ==, === e isequal. (O último não é realmente um operador, mas é uma função e todos os operadores são funções.)

Quando usar == #

== é value igualdade. Ele retorna true quando dois objetos representam, em seu estado atual, o mesmo valor.

Por exemplo, é óbvio que

julia> 1 == 1
true

mas além disso

julia> 1 == 1.0
true

julia> 1 == 1.0 + 0.0im
true

julia> 1 == 1//1
true

Os lados direitos de cada igualdade acima são de um tipo diferente, mas ainda representam o mesmo valor.

Para objetos mutáveis, como arrays, == compara seu valor presente.

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> C = [1, 3, 2]
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
true

julia> A == C
false

julia> A[2], A[3] = A[3], A[2]  # swap 2nd and 3rd elements of A
(3,2)

julia> A
3-element Array{Int64,1}:
 1
 3
 2

julia> A == B
false

julia> A == C
true

Na maioria das vezes, == é a escolha certa.

Quando usar === #

=== é uma operação muito mais rigorosa do que ==. Em vez de igualdade de valor, mede a igualdade. Dois objetos são iguais se não puderem ser distinguidos um do outro pelo próprio programa. Assim temos

julia> 1 === 1
true

pois não há como diferenciar um 1 de outro 1. Mas

julia> 1 === 1.0
false

porque embora 1 e 1.0 tenham o mesmo valor, eles são de tipos diferentes e, portanto, o programa pode distingui-los.

Além disso,

julia> A = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> B = [1, 2, 3]
3-element Array{Int64,1}:
 1
 2
 3

julia> A === B
false

julia> A === A
true

o que pode parecer à primeira vista surpreendente! Como o programa poderia distinguir entre os dois vetores ‘A’ e ‘B’? Como os vetores são mutáveis, ele poderia modificar A, e então se comportaria de forma diferente de B. Mas não importa como ele modifique A, A sempre se comportará da mesma forma que o próprio A. Então A é igual a A, mas não igual a B.

Continuando por esta linha, observe

julia> C = A
3-element Array{Int64,1}:
 1
 2
 3

julia> A === C
true

Ao atribuir A a C, dizemos que C tem aliased A. Ou seja, tornou-se apenas mais um nome para ‘A’. Quaisquer modificações feitas em A serão observadas por C também. Portanto, não há como dizer a diferença entre A e C, então eles são iguais.

Quando usar isequal #

A diferença entre == e isequal é muito sutil. A maior diferença está em como os números de ponto flutuante são tratados:

julia> NaN == NaN
false

Este resultado possivelmente surpreendente é definido pelo padrão IEEE para tipos de ponto flutuante (IEEE-754). Mas isso não é útil em alguns casos, como classificação. isequal é fornecido para esses casos:

julia> isequal(NaN, NaN)
true

No outro lado do espectro, == trata IEEE negativo zero e positivo zero como o mesmo valor (também conforme especificado por IEEE-754). Esses valores têm representações distintas na memória, no entanto.

julia> 0.0
0.0

julia> -0.0
-0.0

julia> 0.0 == -0.0
true

Novamente para fins de classificação, isequal distingue entre eles.

julia> isequal(0.0, -0.0)
false
