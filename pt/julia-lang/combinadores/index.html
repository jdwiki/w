Embora os combinadores tenham uso prático limitado, eles são uma ferramenta útil na educação para entender como a programação está fundamentalmente ligada à lógica e como blocos de construção muito simples podem se combinar para criar um comportamento muito complexo. No contexto de Julia, aprender a criar e usar combinadores fortalecerá a compreensão de como programar em um estilo funcional em Julia.

O Combinador Y ou Z #

Embora Julia não seja uma linguagem puramente funcional, ela tem suporte completo para muitos dos pilares da programação funcional: [funções] de primeira classe(https://www.wikiod.com/pt/julia-lang/funcoes ), escopo lexical e encerramentos.

O combinador de ponto fixo é um combinador chave na programação funcional. Como Julia tem uma semântica de avaliação ansiosa (assim como muitas linguagens funcionais, incluindo Scheme, na qual Julia é fortemente inspirada), o combinador Y original de Curry não funcionará da caixa:

Y(f) = (x -> f(x(x)))(x -> f(x(x)))

No entanto, um parente próximo do combinador Y, o combinador Z, realmente funcionará:

Z(f) = x -> f(Z(f), x)

Este combinador pega uma função e retorna uma função que quando chamada com o argumento x, é passada a si mesma e x. Por que seria útil que uma função fosse passada por si mesma? Isso permite a recursão sem realmente fazer referência ao nome da função!

fact(f, x) = x == 0 ? 1 : x * f(x)

Assim, Z(fact) torna-se uma implementação recursiva da função fatorial, apesar de nenhuma recursão ser visível nesta definição de função. (A recursão é evidente na definição do combinador Z, é claro, mas isso é inevitável em uma linguagem ansiosa.) Podemos verificar que nossa função realmente funciona:

julia> Z(fact)(10)
3628800

Não só isso, mas é tão rápido quanto podemos esperar de uma implementação recursiva. O código LLVM demonstra que o resultado é compilado em um branch antigo simples, subtrair, chamar e multiplicar:

julia> @code_llvm Z(fact)(10)

define i64 @"julia_#1_70252"(i64) #0 {
top:
  %1 = icmp eq i64 %0, 0
  br i1 %1, label %L11, label %L8

L8:                                               ; preds = %top
  %2 = add i64 %0, -1
  %3 = call i64 @"julia_#1_70060"(i64 %2) #0
  %4 = mul i64 %3, %0
  br label %L11

L11:                                              ; preds = %top, %L8
  %"#temp#.0" = phi i64 [ %4, %L8 ], [ 1, %top ]
  ret i64 %"#temp#.0"
}

O Sistema Combinador de ESQUI #

O sistema combinador SKI é suficiente para representar quaisquer termos de cálculo lambda. (Na prática, é claro, as abstrações lambda atingem um tamanho exponencial quando são traduzidas para SKI.) Devido à simplicidade do sistema, a implementação dos combinadores S, K e I é extraordinariamente simples:

Uma tradução direta do cálculo lambda #

const S = f -> g -> z -> f(z)(g(z))
const K = x -> y -> x
const I = x -> x

Podemos confirmar, usando o sistema teste de unidade, que cada combinador tem o comportamento esperado.

O combinador I é mais fácil de verificar; ele deve retornar o valor fornecido inalterado:

using Base.Test
@test I(1) === 1
@test I(I) === I
@test I(S) === S

O combinador K também é bastante direto: ele deve descartar seu segundo argumento.

@test K(1)(2) === 1
@test K(S)(I) === S

O combinador S é o mais complexo; seu comportamento pode ser resumido como a aplicação dos dois primeiros argumentos ao terceiro argumento, a aplicação do primeiro resultado ao segundo. Podemos testar mais facilmente o combinador S testando algumas de suas formas curry. S(K), por exemplo, deve simplesmente retornar seu segundo argumento e descartar seu primeiro, como vemos acontecer:

@test S(K)(S)(K) === K
@test S(K)(S)(I) === I

S(I)(I) deve aplicar seu argumento a si mesmo:

@test S(I)(I)(I) === I
@test S(I)(I)(K) === K(K)
@test S(I)(I)(S(I)) === S(I)(S(I))

S(K(S(I)))(K) aplica seu segundo argumento ao primeiro:

@test S(K(S(I)))(K)(I)(I) === I
@test S(K(S(I)))(K)(K)(S(K)) === S(K)(K)

O combinador I descrito acima tem um nome no padrão Base Julia: identidade. Assim, poderíamos ter reescrito as definições acima com a seguinte definição alternativa de I:

const I = identity

Mostrando Combinadores de ESQUI #

Um ponto fraco com a abordagem acima é que nossas funções não são exibidas tão bem quanto gostaríamos. Poderíamos substituir

julia> S
(::#3) (generic function with 1 method)

julia> K
(::#9) (generic function with 1 method)

julia> I
(::#13) (generic function with 1 method)

com alguns displays mais informativos? A resposta é sim! Vamos reiniciar o REPL, e desta vez definir como cada função deve ser mostrada:

const S = f -> g -> z -> f(z)(g(z));
const K = x -> y -> x;
const I = x -> x;
for f in (:S, :K, :I)
    @eval Base.show(io::IO, ::typeof($f)) = print(io, $(string(f)))
    @eval Base.show(io::IO, ::MIME"text/plain", ::typeof($f)) = show(io, $f)
end

É importante evitar mostrar qualquer coisa até terminarmos de definir as funções. Caso contrário, corremos o risco de invalidar o cache de métodos e nossos novos métodos não terão efeito imediato. É por isso que colocamos ponto e vírgula nas definições acima. Os pontos e vírgulas suprimem a saída do REPL.

Isso faz com que as funções sejam exibidas bem:

julia> S
S

julia> K
K

julia> I
I

No entanto, ainda encontramos problemas quando tentamos exibir um encerramento:

julia> S(K)
(::#2) (generic function with 1 method)

Seria melhor exibir isso como S(K). Para fazer isso, devemos explorar que os fechamentos têm seus próprios tipos individuais. Podemos acessar esses tipos e adicionar métodos a eles através de reflexão, usando typeof e o campo primary do campo name do tipo. Reinicie o REPL novamente; faremos outras alterações:

const S = f -> g -> z -> f(z)(g(z));
const K = x -> y -> x;
const I = x -> x;
for f in (:S, :K, :I)
    @eval Base.show(io::IO, ::typeof($f)) = print(io, $(string(f)))
    @eval Base.show(io::IO, ::MIME"text/plain", ::typeof($f)) = show(io, $f)
end
Base.show(io::IO, s::typeof(S(I)).name.primary) = print(io, "S(", s.f, ')')
Base.show(io::IO, s::typeof(S(I)(I)).name.primary) =
    print(io, "S(", s.f, ')', '(', s.g, ')')
Base.show(io::IO, k::typeof(K(I)).name.primary) = print(io, "K(", k.x, ')')
Base.show(io::IO, ::MIME"text/plain", f::Union{
    typeof(S(I)).name.primary,
    typeof(S(I)(I)).name.primary,
    typeof(K(I)).name.primary
}) = show(io, f)

E agora, finalmente, as coisas são exibidas como gostaríamos:

julia> S(K)
S(K)

julia> S(K)(I)
S(K)(I)

julia> K
K

julia> K(I)
K(I)

julia> K(I)(K)
I
