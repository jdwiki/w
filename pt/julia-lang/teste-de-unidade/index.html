Sintaxe #

  • @test [expr]
  • @test_throws [Exceção] [expr]
  • @testset “[nome]” começar; [testes]; fim
  • Pkg.test([pacote])

A documentação da biblioteca padrão para Base.Test cobre material adicional além do mostrado nestes exemplos.

Testando um pacote #

Para executar os testes de unidade de um pacote, use a função Pkg.test. Para um pacote chamado MyPackage, o comando seria

julia> Pkg.test("MyPackage")

Uma saída esperada seria semelhante a

INFO: Computing test dependencies for MyPackage...
INFO: Installing BaseTestNext v0.2.2
INFO: Testing MyPackage
Test Summary: | Pass  Total
  Data        |   66     66
Test Summary: | Pass  Total
  Monetary    |  107    107
Test Summary: | Pass  Total
  Basket      |   47     47
Test Summary: | Pass  Total
  Mixed       |   13     13
Test Summary: | Pass  Total
  Data Access |   35     35
INFO: MyPackage tests passed
INFO: Removing BaseTestNext v0.2.2

embora, obviamente, não se possa esperar que corresponda exatamente ao acima, já que pacotes diferentes usam estruturas diferentes.

Este comando executa o arquivo test/runtests.jl do pacote em um ambiente limpo.

Pode-se testar todos os pacotes instalados de uma só vez com

julia> Pkg.test()

mas isso geralmente leva muito tempo.

Escrevendo um conjunto de testes #

Na versão v0.5, conjuntos de testes são construídos no módulo Base.Test da biblioteca padrão, e você não precisa fazer nada de especial (além de usar Base.Test) para usá-los.

Os conjuntos de teste não fazem parte da biblioteca Base.Test do Julia v0.4. Em vez disso, você precisa REQUER o módulo BaseTestNext e adicionar using BaseTestNext ao seu arquivo. Para suportar a versão 0.4 e 0.5, você pode usar

if VERSION ≥ v"0.5.0-dev+7720"
    using Base.Test
else
    using BaseTestNext
    const Test = BaseTestNext
end

É útil agrupar @tests relacionados em um conjunto de teste. Além de uma organização de teste mais clara, os conjuntos de teste oferecem melhor saída e mais personalização.

Para definir um conjunto de teste, simplesmente envolva qualquer número de @tests com um bloco @testset:

@testset "+" begin
    @test 1 + 1 == 2
    @test 2 + 2 == 4
end

@testset "*" begin
    @test 1 * 1 == 1
    @test 2 * 2 == 4
end

A execução desses conjuntos de teste imprime a seguinte saída:

Test Summary: | Pass  Total
  +           |    2      2

Test Summary: | Pass  Total
  *           |    2      2

Mesmo que um conjunto de testes contenha um teste com falha, todo o conjunto de testes será executado até a conclusão e as falhas serão registradas e relatadas:

@testset "-" begin
    @test 1 - 1 == 0
    @test 2 - 2 == 1
    @test 3 - () == 3
    @test 4 - 4 == 0
end

A execução deste conjunto de testes resulta em

-: Test Failed
  Expression: 2 - 2 == 1
   Evaluated: 0 == 1
 in record(::Base.Test.DefaultTestSet, ::Base.Test.Fail) at ./test.jl:428
    ...
-: Error During Test
  Test threw an exception of type MethodError
  Expression: 3 - () == 3
  MethodError: no method matching -(::Int64, ::Tuple{})
    ...
Test Summary: | Pass  Fail  Error  Total
  -           |    2     1      1      4
ERROR: Some tests did not pass: 2 passed, 1 failed, 1 errored, 0 broken.
    ...

Os conjuntos de teste podem ser aninhados, permitindo uma organização arbitrariamente profunda

@testset "Int" begin
    @testset "+" begin
        @test 1 + 1 == 2
        @test 2 + 2 == 4
    end
    @testset "-" begin
        @test 1 - 1 == 0
    end
end

Se os testes forem aprovados, isso mostrará apenas os resultados do conjunto de testes mais externo:

Test Summary: | Pass  Total
  Int         |    3      3

Mas se os testes falharem, será relatado um detalhamento do conjunto de testes exato e do teste que está causando a falha.

A macro @testset pode ser usada com um loop for para criar vários conjuntos de teste de uma só vez:

@testset for i in 1:5
    @test 2i == i + i
    @test i^2 == i * i
    @test i ÷ i == 1
end

quais relatórios

Test Summary: | Pass  Total
  i = 1       |    3      3
Test Summary: | Pass  Total
  i = 2       |    3      3
Test Summary: | Pass  Total
  i = 3       |    3      3
Test Summary: | Pass  Total
  i = 4       |    3      3
Test Summary: | Pass  Total
  i = 5       |    3      3

Uma estrutura comum é ter componentes ou tipos de teste de conjuntos de teste externos. Dentro desses conjuntos de testes externos, os conjuntos de testes internos testam o comportamento. Por exemplo, suponha que criamos um tipo UniversalSet com uma instância singleton que contém tudo. Antes mesmo de implementar o tipo, podemos usar os princípios do desenvolvimento orientado a testes e implementar os testes:

@testset "UniversalSet" begin
    U = UniversalSet.instance
    @testset "egal/equal" begin
        @test U === U
        @test U == U
    end

    @testset "in" begin
        @test 1 in U
        @test "Hello World" in U
        @test Int in U
        @test U in U
    end

    @testset "subset" begin
        @test Set() ⊆ U
        @test Set(["Hello World"]) ⊆ U
        @test Set(1:10) ⊆ U
        @test Set([:a, 2.0, "w", Set()]) ⊆ U
        @test U ⊆ U
    end
end

Podemos então começar a implementar nossa funcionalidade até que ela passe em nossos testes. O primeiro passo é definir o tipo:

immutable UniversalSet <: Base.AbstractSet end

Apenas dois de nossos testes passam agora. Podemos implementar in:

immutable UniversalSet <: Base.AbstractSet end
Base.in(x, ::UniversalSet) = true

Isso também faz com que alguns de nossos testes de subconjunto sejam aprovados. No entanto, o fallback issubset (⊆) não funciona para o UniversalSet, porque o fallback tenta iterar sobre os elementos, o que não podemos fazer. Podemos simplesmente definir uma especialização que faz com que issubset retorne true para qualquer conjunto:

immutable UniversalSet <: Base.AbstractSet end
Base.in(x, ::UniversalSet) = true
Base.issubset(x::Base.AbstractSet, ::UniversalSet) = true

E agora, todos os nossos testes passam!

Escrevendo um teste simples #

Testes de unidade são declarados no arquivo test/runtests.jl em um pacote. Normalmente, este arquivo começa

using MyModule
using Base.Test

A unidade básica de teste é a macro @test. Essa macro é como uma espécie de afirmação. Qualquer expressão booleana pode ser testada na macro @test:

@test 1 + 1 == 2
@test iseven(10)
@test 9 < 10 || 10 < 9

Podemos experimentar a macro @test no REPL:

julia> using Base.Test

julia> @test 1 + 1 == 2
Test Passed
  Expression: 1 + 1 == 2
   Evaluated: 2 == 2

julia> @test 1 + 1 == 3
Test Failed
  Expression: 1 + 1 == 3
   Evaluated: 2 == 3
ERROR: There was an error during testing
 in record(::Base.Test.FallbackTestSet, ::Base.Test.Fail) at ./test.jl:397
 in do_test(::Base.Test.Returned, ::Expr) at ./test.jl:281

A macro de teste pode ser usada em praticamente qualquer lugar, como em loops ou funções:

# For positive integers, a number's square is at least as large as the number
for i in 1:10
    @test i^2 ≥ i
end

# Test that no two of a, b, or c share a prime factor
function check_pairwise_coprime(a, b, c)
    @test gcd(a, b) == 1
    @test gcd(a, c) == 1
    @test gcd(b, c) == 1
end

check_pairwise_coprime(10, 23, 119)

Exceções de teste #

As exceções encontradas durante a execução de um teste falharão no teste e, se o teste não estiver em um conjunto de teste, encerre o mecanismo de teste. Normalmente, isso é uma coisa boa, porque na maioria das situações as exceções não são o resultado desejado. Mas às vezes, queremos testar especificamente se uma certa exceção é levantada. A macro @test_throws facilita isso.

julia> @test_throws BoundsError [1, 2, 3][4]
Test Passed
  Expression: ([1,2,3])[4]
      Thrown: BoundsError

Se a exceção errada for lançada, @test_throws ainda falhará:

julia> @test_throws TypeError [1, 2, 3][4]
Test Failed
  Expression: ([1,2,3])[4]
    Expected: TypeError
      Thrown: BoundsError
ERROR: There was an error during testing
 in record(::Base.Test.FallbackTestSet, ::Base.Test.Fail) at ./test.jl:397
 in do_test_throws(::Base.Test.Threw, ::Expr, ::Type{T}) at ./test.jl:329

e se nenhuma exceção for lançada, @test_throws também falhará:

julia> @test_throws BoundsError [1, 2, 3, 4][4]
Test Failed
  Expression: ([1,2,3,4])[4]
    Expected: BoundsError
  No exception thrown
ERROR: There was an error during testing
 in record(::Base.Test.FallbackTestSet, ::Base.Test.Fail) at ./test.jl:397
 in do_test_throws(::Base.Test.Returned, ::Expr, ::Type{T}) at ./test.jl:329

Testando a igualdade aproximada do ponto flutuante #

Qual é o problema com o seguinte?

julia> @test 0.1 + 0.2 == 0.3
Test Failed
  Expression: 0.1 + 0.2 == 0.3
   Evaluated: 0.30000000000000004 == 0.3
ERROR: There was an error during testing
 in record(::Base.Test.FallbackTestSet, ::Base.Test.Fail) at ./test.jl:397
 in do_test(::Base.Test.Returned, ::Expr) at ./test.jl:281

O erro é causado pelo fato de que nenhum de 0.1, 0.2 e 0.3 são representados no computador como exatamente esses valores — 1//10, 2//10 e 3/ /10. Em vez disso, eles são aproximados por valores muito próximos. Mas, como visto na falha do teste acima, ao adicionar duas aproximações, o resultado pode ser uma aproximação um pouco pior do que é possível. Há muito mais sobre este assunto que não pode ser abordado aqui.

Mas não estamos sem sorte! Para testar que a combinação de arredondamento para um número de ponto flutuante e aritmética de ponto flutuante é aproximadamente correta, mesmo que não seja exata, podemos usar a função isapprox (que corresponde ao operador ≈). Assim, podemos reescrever nosso teste como

julia> @test 0.1 + 0.2 ≈ 0.3
Test Passed
  Expression: 0.1 + 0.2 ≈ 0.3
   Evaluated: 0.30000000000000004 isapprox 0.3

É claro que, se nosso código estiver totalmente errado, o teste ainda detectará isso:

julia> @test 0.1 + 0.2 ≈ 0.4
Test Failed
  Expression: 0.1 + 0.2 ≈ 0.4
   Evaluated: 0.30000000000000004 isapprox 0.4
ERROR: There was an error during testing
 in record(::Base.Test.FallbackTestSet, ::Base.Test.Fail) at ./test.jl:397
 in do_test(::Base.Test.Returned, ::Expr) at ./test.jl:281

A função isapprox usa heurística baseada no tamanho dos números e na precisão do tipo de ponto flutuante para determinar a quantidade de erro a ser tolerada. Não é apropriado para todas as situações, mas funciona na maioria e economiza muito esforço ao implementar a própria versão do isapprox.
