Sintaxe #

  • @enum EnumType val=1 val val
  • :símbolo

Às vezes é útil ter tipos enumerados onde cada instância é de um tipo diferente (geralmente um tipo imutável único); isso pode ser importante para a estabilidade do tipo. Traços são tipicamente implementados com este paradigma. No entanto, isso resulta em sobrecarga de tempo de compilação adicional.

1: https://www.wikiod.com/pt/julia-lang/tipos#Tipos imutáveis

Definindo um tipo enumerado #

Um tipo enumerado é um tipo que pode conter um de uma lista finita de valores possíveis. Em Julia, os tipos enumerados são normalmente chamados de “tipos de enumeração”. Por exemplo, pode-se usar tipos enum para descrever os sete dias da semana, os doze meses do ano, os quatro naipes de um [baralho padrão de 52 cartas](https://en.wikipedia.org/wiki/Standard_52 -card_deck) ou outras situações semelhantes.

Podemos definir tipos enumerados para modelar os naipes e as classificações de um baralho padrão de 52 cartas. A macro @enum é usada para definir tipos de enumeração.

@enum Suit ♣ ♦ ♥ ♠
@enum Rank ace=1 two three four five six seven eight nine ten jack queen king

Isso define dois tipos: Suit e Rank. Podemos verificar que os valores são de fato dos tipos esperados:

julia> ♦
♦::Suit = 1

julia> six
six::Rank = 6

Observe que cada naipe e classificação foi associado a um número. Por padrão, esse número começa em zero. Assim, o segundo naipe, diamantes, recebeu o número 1. No caso de Rank, pode fazer mais sentido começar o número em um. Isso foi conseguido anotando a definição de ace com uma anotação =1.

Os tipos enumerados vêm com muitas funcionalidades, como igualdade (e, de fato, identidade) e comparações incorporadas:

julia> seven === seven
true

julia> ten ≠ jack
true

julia> two < three
true

Como valores de qualquer outro tipo imutável, valores de tipos enumerados também podem ser codificados e armazenados em Dicts.

Podemos completar este exemplo definindo um tipo Card que tem um campo Rank e um campo Suit:

immutable Card
    rank::Rank
    suit::Suit
end

e, portanto, podemos criar cartões com

julia> Card(three, ♣)
Card(three::Rank = 3,♣::Suit = 0)

Mas os tipos enumerados também vêm com seus próprios métodos convert, então podemos simplesmente fazer

julia> Card(7, ♠)
Card(seven::Rank = 7,♠::Suit = 3)

e como 7 pode ser convertido diretamente para Rank, este construtor funciona fora da caixa.

Poderíamos desejar definir o açúcar sintático para construir essas cartas; a multiplicação implícita fornece uma maneira conveniente de fazê-lo. Definir

julia> import Base.*

julia> r::Int * s::Suit = Card(r, s)
* (generic function with 156 methods)

e depois

julia> 10♣
Card(ten::Rank = 10,♣::Suit = 0)

julia> 5♠
Card(five::Rank = 5,♠::Suit = 3)

mais uma vez aproveitando as funções internas de converter.

2: https://www.wikiod.com/pt/julia-lang/tipos#Tipos imutáveis

Usando símbolos como enums leves #

Embora a macro @enum seja bastante útil para a maioria dos casos de uso, ela pode ser excessiva em alguns casos de uso. As desvantagens de @enum incluem:

  • Cria um novo tipo
  • É um pouco mais difícil de estender
  • Ele vem com funcionalidades como conversão, enumeração e comparação, que podem ser supérfluas em alguns aplicativos

Nos casos em que uma alternativa mais leve é ​​desejada, o tipo Symbol pode ser usado. Os símbolos são strings internas; eles representam sequências de caracteres, bem como strings, mas são associados exclusivamente a números. Essa associação exclusiva permite uma comparação rápida de igualdade de símbolos.

Podemos implementar novamente um tipo Card, desta vez usando campos Symbol:

const ranks = Set([:ace, :two, :three, :four, :five, :six, :seven, :eight, :nine,
                   :ten, :jack, :queen, :king])
const suits = Set([:♣, :♦, :♥, :♠])
immutable Card
    rank::Symbol
    suit::Symbol
    function Card(r::Symbol, s::Symbol)
        r in ranks || throw(ArgumentError("invalid rank: $r"))
        s in suits || throw(ArgumentError("invalid suit: $s"))
        new(r, s)
    end
end

Implementamos o construtor interno para verificar quaisquer valores incorretos passados ​​para o construtor. Ao contrário do exemplo usando tipos @enum, Symbols pode conter qualquer string, e por isso devemos ter cuidado com os tipos de Symbols que aceitamos. Observe aqui o uso dos operadores condicionais [curto-circuito][3].

Agora podemos construir objetos Card como esperamos:

julia> Card(:ace, :♦)
Card(:ace,:♦)

julia> Card(:nine, :♠)
Card(:nine,:♠)

julia> Card(:eleven, :♠)
ERROR: ArgumentError: invalid rank: eleven
 in Card(::Symbol, ::Symbol) at ./REPL[17]:5

julia> Card(:king, :X)
ERROR: ArgumentError: invalid suit: X
 in Card(::Symbol, ::Symbol) at ./REPL[17]:6

Um grande benefício do Symbols é sua extensibilidade de tempo de execução. Se em tempo de execução, desejamos aceitar (por exemplo) :eleven como uma nova classificação, basta executar push!(ranks, :eleven). Essa extensibilidade de tempo de execução não é possível com tipos @enum.

[3]: https://www.wikiod.com/pt/julia-lang/condicionais#Operadores de curto-circuito: && e ||
