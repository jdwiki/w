YARN - MapReduce 2

  • Anatomia do fio

Nesta página

  • Anatomia do fio

YARN significa mais um negociador de recursos.

Breve história:

O YARN foi desenvolvido pelo Yahoo em 2010 como substituto do Map Reduce 1. Ele foi capaz de superar as desvantagens que estavam presentes no Map Reduce 1.

Introdução

No Map Reduce 1 Job Tracker foi usado tanto para o agendamento de trabalhos quanto para o monitoramento de tarefas. Isso levou ao gargalo na escalabilidade do cluster. O YARN separa essas duas funções em 2 daemons separados, que são o Resource Manager e o Application Master.

Anatomia do fio #

Componentes

Resource Manager: É a autoridade máxima que aloca recursos entre todas as aplicações do sistema. Ele também é responsável por alocar o container no qual o aplicativo mestre irá iniciar e inicializá-lo após a alocação do container para o aplicativo mestre.

Node Manager: É responsável por monitorar o container e seu uso de recursos em qualquer nó em que esteja rodando.

Application Master: É responsável por negociar com o Resource Manager a alocação dos recursos necessários no nó de dados para execução de qualquer trabalho e também coordenar com o gerenciador do nó para executar o trabalho e monitorar os contêineres e seu consumo de recursos.

Container : Representa recursos como CPU, Memória, Disco etc. necessários para o processamento de qualquer trabalho. É o resultado da alocação de recursos bem-sucedida do gerenciador de recursos. O contêiner concede ao mestre de aplicativos o direito de usar uma quantidade específica de recursos em uma máquina específica (nó) em um cluster. Pode haver vários contêineres na mesma máquina.

Processo de Execução do Trabalho

  1. O usuário/cliente envia o aplicativo e a especificação para o mestre do aplicativo é gerada.

  2. O gerente de recursos assume a responsabilidade de alocar o container especificado no qual o Application Master será iniciado e, em seguida, inicia o Application Master. Cada trabalho recebe seu próprio Application Master separado.

  3. Inicialize o mestre de aplicativos e registre-se no gerenciador de recursos. Isso ajuda o cliente a conhecer certos detalhes necessários da execução de trabalhos diretamente pelo Application Master.

  4. O Application Master então faz uma solicitação ao gerenciador de recursos para alocação dos contêineres com os recursos necessários para executar o trabalho nos nós. Essa solicitação de recurso é feita via Resource Request Protocol (RRP).

  5. Na alocação bem-sucedida do contêiner, o Application Master apresenta os contêineres ao Node Manager. Ele também fornece detalhes relacionados à alocação para iniciar os contêineres no nó. Antes de iniciar o contêiner, o Node Manager verifica a alocação do contêiner com o Resource Manager para evitar uma alocação falsa.

  6. Quando o código do aplicativo é executado dentro do contêiner, o contêiner fornece as informações apropriadas ao seu Application Master por meio do Application Specific Protocol (ASP).

  7. Durante a execução, o cliente conecte-se diretamente com o Application Master para obter Status, Progresso etc. através do Application Specific Protocol.

  8. Concluído o processo do pedido enviado pelo usuário, o Application Master se descadastra no Resource Manager e encerra, permitindo que o container alocado para ele seja liberado e reutilizado Junto com ele, todos os containers adquiridos para o trabalho são liberados novamente .
