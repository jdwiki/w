Antes do lançamento do Meteor 1.3, os desenvolvedores do Meteor estavam frustrados com o manuseio do Meteor.js de dependências de arquivos e variáveis ​​globais. Em resposta, o Meteor estabeleceu novos padrões para estruturas de projeto, a fim de tornar o sistema de dependência do projeto mais simplificado. Este tópico explica a estrutura padronizada do projeto e os princípios por trás dela.

cliente Todo o código no diretório do cliente é executado apenas no lado do cliente ou no navegador da web.

cliente/compatibilidade O diretório de compatibilidade contém código legado ou de terceiros, como bibliotecas jQuery, etc.

lib O diretório lib é carregado antes de outros diretórios em seu projeto Meteor e é carregado tanto no servidor quanto no cliente. Este é o local preferido para definir modelos de dados, bibliotecas isomórficas e lógica de negócios.

importações O diretório de importações é um diretório no servidor que está disponível tanto para o servidor quanto para o cliente, mas somente antes que o pacote configurável do cliente seja enviado ao cliente.

pacotes O diretório de pacotes é onde os pacotes personalizados são armazenados durante o desenvolvimento local. Ao usar o comando padrão meteor add package:name para adicionar um pacote, o Meteor irá procurar primeiro neste diretório se um pacote local tiver o nome de descrição correspondente em seu arquivo package.js. Caso contrário, ele pesquisará a Atmosfera como de costume.

privado O diretório privado contém arquivos estáticos que devem estar disponíveis apenas no servidor web.

público O diretório público contém arquivos estáticos que estão disponíveis apenas no aplicativo cliente. Isso pode incluir ativos de marca, etc.

servidor O diretório do servidor contém ativos do lado do servidor. Isso pode incluir lógica de autenticação, métodos e outros códigos que podem precisar de consideração de segurança.

testes O diretório de testes é omitido por padrão quando seu aplicativo é empacotado e implementado.

Como sugerido por Richard Silverton é uma idéia conveniente colocar não apenas o diretório do projeto meteor sob controle de versão, mas também seu diretório pai.

Dessa forma, você pode manter os arquivos sob controle de versão sem ter que lidar com o meteoro.

Estruturas de diretório clássicas #

A primeira coisa que você precisa saber ao estruturar seus aplicativos é que a ferramenta Meteor possui alguns diretórios que são codificados com lógica específica. Em um nível muito básico, os seguintes diretórios são “preparados” no pacote Meteor.

client/                                  # client application code
client/compatibility/                    # legacy 3rd party javascript libraries
imports/                                 # for lazy loading feature 
lib/                                     # any common code for client/server.
packages/                                # place for all your atmosphere packages
private/                                 # static files that only the server knows about
public/                                  # static files that are available to the client
server/                                  # server code
tests/                                   # unit test files (won't be loaded on client or server)

Página de referência: Guia do Meteoro > Diretórios especiais

Ordem de carregamento do diretório #

Os arquivos de modelo HTML são sempre carregados antes de tudo

Arquivos que começam com main. são carregados por último

Arquivos dentro de qualquer diretório lib/ são carregados em seguida

Arquivos com caminhos mais profundos são carregados em seguida

Os arquivos são então carregados em ordem alfabética de todo o caminho

Estrutura de diretórios de importações/módulos #

As versões mais recentes do Meteor são fornecidas com suporte para ecmascript, também conhecido como ES6 ou ES2015. Em vez de pacotes, o Javascript agora suporta instruções e módulos import, o que substitui a necessidade de aplicativos somente de pacote. A estrutura de diretórios mais recente é semelhante à estrutura somente de pacotes, mas usa o diretório /imports em vez de /packages.

imports                                 #
imports/api                             # isomorphic methods 
imports/lib                             # any common code for client/server
imports/client                          # client application code
imports/server                          # server code

Estrutura de diretórios somente de pacote #

Muitas pessoas acabam dando suporte a vários aplicativos e desejam compartilhar código entre aplicativos. Isso leva ao conceito de arquitetura de microsserviço e aplicativos de todos os pacotes. Essencialmente, o código de toda a estrutura clássica de diretórios é refatorado em pacotes.

Embora não haja lógica codificada para diretórios em pacotes, achamos que é uma boa prática usar a estrutura de diretório clássica ao criar pacotes. Isso cria um caminho de refatoração natural à medida que os recursos são prototipados no aplicativo e, em seguida, extraídos em pacotes para serem publicados e compartilhados. Os nomes dos diretórios são compartilhados, portanto, há menos confusão entre os membros da equipe.

client/                                  # client application code
packages/                                # place for all your atmosphere packages
packages/foo/client                      # client application code
packages/foo/lib                         # any common code for client/server
packages/foo/server                      # server code
packages/foo/tests                       # tests
server/                                  # server code

Estrutura de diretórios de modo misto #

E, é claro, você pode misturar essas abordagens e usar pacotes e importações junto com o código específico do seu aplicativo. Uma estrutura de modo misto é mais comum em três situações: um aplicativo franken, que é meio que puxando um pouco daqui e dali sem nenhuma estratégia geral; um aplicativo que está sendo refatorado ativamente das estruturas Classic ou Package-Only para a estrutura Imports/Modules.

client/                                  # client application code
client/compatibility/                    # legacy 3rd party javascript libraries
imports                                  #
imports/api                              # isomorphic methods 
imports/lib                              # any common code for client/server
imports/client                           # client application code
imports/server                           # server code
lib/                                     # any common code for client/server.
packages/                                # place for all your atmosphere packages
packages/foo/client                      # client application code
packages/foo/lib                         # any common code for client/server
packages/foo/server                      # server code
packages/foo/tests                       # tests
private/                                 # static files that only the server knows about
public/                                  # static files that are available to the client
server/                                  # server code
tests/                                   # unit test files (won't be loaded on client or server)
