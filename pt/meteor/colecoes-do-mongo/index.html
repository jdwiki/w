Uma maneira útil de pensar sobre as coleções do Mongo é em termos de quem, o quê, quando, onde, por que e como. O Mongo tem as seguintes otimizações para diferentes tipos de dados:

Onde - GeoJSON Quando - carimbos de data/hora do ObjectID Quem - Strings da Conta Meteor Como - JSON para árvores de decisão

O que deixa o documento padrão no Mongo representando aproximadamente um ‘O quê’.

Criando registros em um banco de dados legado #

Você pode usar como padrão o formato normal do Mongo definindo suas coleções com o campo idGeneration.

MyCollection = new Meteor.Collection('mycollection', {idGeneration : 'MONGO'});

Inserindo dados em um documento #

Muitos iniciantes no Mongo lutam com o básico, como inserir uma matriz, data, booleano, variável de sessão e assim por diante em um registro de documento. Este exemplo fornece algumas orientações sobre entradas de dados básicos.

Todos.insert({
  text: "foo",                        // String
  listId: Session.get('list_id'),     // String
  value: parseInt(2),                 // Number
  done: false,                        // Boolean
  createdAt: new Date(),              // Dimestamp
  timestamp: (new Date()).getTime(),  // Time
  tags: []                            // Array
});

Obtendo o _id do documento criado mais recentemente #

Você pode obtê-lo de forma síncrona:

var docId = Todos.insert({text: 'foo'});
console.log(docId);

Ou de forma assíncrona:

Todos.insert({text: 'foo'}, function(error, docId){
  console.log(docId);
});

Dados de série temporal #

Usar o MongoDB para dados de séries temporais é um documento muito bem documentado e um caso de uso estabelecido, com whitepapers e apresentações oficiais. Leia e assista a documentação oficial do MongoDB antes de tentar inventar seus próprios esquemas para dados de séries temporais.

MongoDB para dados de séries temporais

Em geral, você desejará criar “buckets” para seus dados de série temporal:

DailyStats.insert({
   "date" : moment().format("MM-DD-YYYY"),
   "dateIncrement" : moment().format("YYYYMMDD"),
   "dailyTotal" : 0,
   'bucketA': 0,
   'bucketB': 0,
   'bucketC': 0
   });

E, em seguida, incremente esses buckets à medida que os dados são alimentados em seu aplicativo. Esse incremento pode ser colocado em um Método Meteor, um observador de coleção, um endpoint da API REST e vários outros lugares.

DailyStats.update({_id: doc._id}, {$inc: {bucketA: 1} });

Para um exemplo mais completo do Meteor, veja os exemplos da trilha Clinical Meteor:

Realtime Analytics Pipeline Clinical Meteor - Graphs - Dailystats

Filtrando com Regexes #

Padrão simples para filtrar assinaturas no servidor, usando regexes, variáveis ​​de sessão reativas e deps autoruns.

// create our collection
WordList =  new Meteor.Collection("wordlist");

// and a default session variable to hold the value we're searching for
Session.setDefault('dictionary_search', '');

Meteor.isClient(function(){
    // we create a reactive context that will rerun items when a Session variable gets updated 
    Deps.autorun(function(){
        // and create a subscription that will get re-subscribe to when Session variable gets updated
        Meteor.subscribe('wordlist', Session.get('dictionary_search'));
    });

    Template.dictionaryIndexTemplate.events({
        'keyup #dictionarySearchInput': function(evt,tmpl){
            // we set the Session variable with the value of our input when it changes
            Session.set('dictionary_search', $('#dictionarySearchInput').val());
        },
        'click #dictionarySearchInput':function(){
            // and clear the session variable when we enter the input
            Session.set('dictionary_search', '');
        },
    });
});
Meteor.isServer(function(){
    Meteor.publish('wordlist', function (word_search) {
        // this query gets rerun whenever the client subscribes to this publication
        return WordList.find({
            // and here we do our regex search
            Word: { $regex: word_search, $options: 'i' }
        },{limit: 100});
    });
});

E o HTML que é usado no cliente:

<input id="dictionarySearchInput" type="text" placeholder="Filter..." value="hello"></input>

Consultas de coleção de auditoria #

O exemplo a seguir registrará todas as suas consultas de coleta no console do servidor em tempo real.

Meteor.startup(     
  function () {     
    var wrappedFind = Meteor.Collection.prototype.find;     

    // console.log('[startup] wrapping Collection.find')        

    Meteor.Collection.prototype.find = function () {        
      // console.log(this._name + '.find', JSON.stringify(arguments))       
      return wrappedFind.apply(this, arguments);        
    }       
  },        

  function () {     
    var wrappedUpdate = Meteor.Collection.prototype.update;     

    // console.log('[startup] wrapping Collection.find')        

    Meteor.Collection.prototype.update = function () {      
      console.log(this._name + '.update', JSON.stringify(arguments))        
      return wrappedUpdate.apply(this, arguments);      
    }       
  }     
);

Funções de observadores e trabalhadores #

Se o loop de eventos do Node age como uma corrente de bicicleta, o observador de coleção do lado do servidor é como um desviador. É um mecanismo de engrenagem que ficará na coleta de dados à medida que os dados chegarem. Pode ser muito eficiente, pois todas as bicicletas de corrida têm desviadores. Mas também é uma fonte para quebrar todo o sistema. É uma função reativa de alta velocidade, que pode explodir em você. Esteja avisado.

Meteor.startup(function(){
  console.log('starting worker....');

  var dataCursor = Posts.find({viewsCount: {$exists: true}},{limit:20});

  var handle = dataCursor.observeChanges({
    added: function (id, record) {
      if(record.viewsCount > 10){
         // run some statistics
         calculateStatistics();

         // or update a value
         Posts.update({_id: id}, {$set:{
           popular: true
         }});

      }
    },
    removed: function () {
      console.log("Lost one.");
    }
  });
});

Observe que o limite de 20 é o tamanho do desviador…. quantos dentes ele tem; ou, mais especificamente, quantos itens estão no cursor enquanto ele percorre a coleção. Tenha cuidado ao usar a palavra-chave ‘var’ neste tipo de função. Escreva o menor número possível de objetos na memória e concentre-se na reutilização de objetos dentro do método adicionado. Quando o opslog está ligado, e essa coisa está funcionando a toda velocidade, é um candidato principal para expor vazamentos de memória desagradáveis ​​se estiver gravando objetos no heap de memória mais rápido do que o coletor de lixo do Node é capaz de limpar as coisas.

A solução acima não será dimensionada horizontalmente bem, porque cada instância do Meteor tentará atualizar o mesmo registro. Portanto, algum tipo de detecção de ambiente é necessário para que isso seja dimensionado horizontalmente.

Veja o pacote percolatestudios:synced-cron para um excelente exemplo de sincronização de service workers em várias máquinas em um cluster. meteor-synced-cron
