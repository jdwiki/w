A publicação de uma faixa de lançamento é realmente muito simples se você entender a) que o comando publish-release requer um arquivo .json como parâmetro eb) qual é a aparência desse arquivo. Esse é definitivamente o maior obstáculo para começar, porque praticamente não está documentado em nenhum lugar.

Apenas tenha em mente que cada pacote no lançamento deve ser publicado e no Atmosphere. O arquivo .meteor/versions de um aplicativo é um lugar particularmente bom para encontrar todos os pacotes e versões necessários que devem entrar no lançamento.

Depois disso, é uma questão de descobrir o que você está disposto a apoiar, o que você quer incluir, etc. Aqui está um Diagrama de Venn parcial do que a Liberação Clínica está trabalhando atualmente; e deve dar a você uma ideia geral de como estamos fazendo o processo de tomada de decisão sobre o que será incluído.

Para mais discussão, veja o tópico no Meteor Forums: https://forums.meteor.com/t/custom-meteor-release/13736/6

Uso básico #

A ideia é que um mantenedor de distro queira executar algo como o seguinte comando:

meteor publish-release clinical.meteor.rc6.json

O que permitirá que os usuários da distro executem isso:

meteor run --release clinical:METEOR@1.1.3-rc6

Liberar Manifesto #

Um manifesto de lançamento é semelhante a um arquivo package.json do NPM, pois sua principal preocupação é especificar uma lista de pacotes do Atmosphere e fornecer um pouco de metadados sobre essa lista de pacotes. O formato básico fica assim:

{
  "track":"distroname:METEOR",
  "version":"x.y.z",
  "recommended": false,
  "tool": "distroname:meteor-tool@x.y.z",
  "description": "Description of the Distro",
  "packages": {
    "accounts-base":"1.2.0",
    "accounts-password":"1.1.1",
    ...
  }
}

Personalizando a ferramenta Meteor #

Se você precisar estender a ferramenta meteoro ou a linha de comando, precisará criar e publicar seu próprio pacote de ferramentas meteorológicas. A documentação de Ronen é a melhor disponível para este processo:

http://practicalmeteor.com/using-meteor-publish-release-to-extend-the-meteor-command-line-tool/1

É fácil fazer um comando meteor helloworld funcionar, mas depois disso, senti que era mais fácil criar um aplicativo de nó separado para testar os comandos. E foi assim que StarryNight surgiu. É uma espécie de palco e rascunho para comandos antes de tentar colocá-los em uma versão da ferramenta meteoro.

Extraindo um manifesto de lançamento de .meteor/versions #

StarryNight contém um pequeno utilitário que analisa o arquivo .meteor/versions de um aplicativo e o converte em um manifesto de lançamento.

npm install -g starrynight
cd myapp
starrynight generate-release-json

Se você não deseja usar StarryNight, simplesmente copie o conteúdo do seu arquivo .meteor/versions no campo packages do seu arquivo de manifesto. Certifique-se de converter para a sintaxe JSON e adicionar dois pontos e aspas.

Exibindo o manifesto de versão para uma versão específica #

meteor show --ejson METEOR@1.2.1

Publicando uma versão do Checkout #

meteor publish-release --from-checkout

Buscando os últimos commits para cada pacote em uma versão #

Ao construir uma faixa de lançamento personalizada, é comum manter os pacotes no diretório /packages como submódulos git. O comando a seguir permite que você busque todos os commits mais recentes para os submódulos em seu diretório /packages ao mesmo tempo.

git submodule foreach git pull origin master
