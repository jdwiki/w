Pacote de contas Meteor #

Você tem algumas opções quando se trata de fazer login com o Meteor. O método mais comum é usar accounts para Meteor.

Contas-senha #

Se você quiser que os usuários possam criar e se registrar em seu site, você pode usar accounts-password.

Instale o pacote usando meteor add accounts-password.

Para criar um usuário, você precisa usar Accounts.createUser(options, [callback])

options deve ser um objeto com as seguintes propriedades:

  • username: O nome de usuário do usuário como uma string..
  • email: O e-mail do usuário como uma string.
  • password: A senha do usuário (não criptografada) como uma string.
  • profile: Os dados extras opcionais do usuário como um objeto. Isso pode ser, por exemplo, o nome e o sobrenome do usuário. profile é opcional, no entanto.

O retorno de chamada retorna 1 variável se houver um erro, que é um objeto Meteor.Error.

Você só precisa usar o username ou o email, então você pode criar um usuário com nome de usuário, mas sem e-mail e vice-versa. Você também pode usar os dois.

Ele retorna o ID do usuário recém-criado se tudo ocorreu corretamente.

Então, você pode, por exemplo, usar isso:

// server side
var id = Accounts.createUser({
    username: "JohnDoe",
    email: "JohnDoe@gmail.com",
    password: "TheRealJohn123",
    profile: {
        firstName: "John",
        lastName: "Doe"
    }
}, function(err) {
    console.log(err.reason);
});

Ele também fará o login automaticamente se o usuário tiver sido criado com sucesso.

Essa é a parte da criação. Para logar você precisa usar Meteor.loginWithPassword(identifier, password, [callback]) no lado do cliente.

identifier é o username, email ou userId como uma string do seu usuário. password é a password (não criptografada) do usuário.

O retorno de chamada retorna uma variável se houver um erro, que é um objeto Meteor.Error.

Exemplo:

// client side
Meteor.loginWithPassword("JohnDoe", "TheRealJohn123", function(err) {
    console.log(err.reason);
});

E é isso para a criação básica de contas e login.

Acessando os dados do usuário #

Você pode verificar no lado do cliente se o usuário está logado chamando Meteor.userId() que retornará seu userId se ele estiver logado, e undefined se não estiver logado.

Você pode obter algumas informações de Meteor.user(). Ele retornará indefinido se o usuário não estiver logado e alguns dados do usuário se estiverem. Ele não lhe dará nenhuma senha por padrão, por padrão ele mostrará o userId do usuário, o nome de usuário e o objeto de perfil.

Se você quiser verificar se um usuário está logado em uma página, você também pode usar o auxiliar currentUser. Ele retornará o conteúdo de Meteor.user(). Exemplo:

{{#if currentUser}}
    <h1>Hello there, {{currentUser.username}}!</h1>
{{else}}
    <h1>Please log in.</h1>
{{/if}}

Existem algumas outras funções que funcionam para todos os pacotes de contas.

Você pode sair usando Meteor.logout()

Não use o campo de perfil padrão #

Existe um campo tentador chamado profile que é adicionado por padrão quando um novo usuário se registra. Este campo foi historicamente destinado a ser usado como um bloco de rascunho para dados específicos do usuário - talvez seu avatar de imagem, nome, texto de introdução, etc. Por causa disso, o campo profile em cada usuário é automaticamente gravável por esse usuário do cliente. Ele também é publicado automaticamente no cliente para esse usuário específico.

Acontece que ter um campo gravável por padrão sem tornar isso super óbvio pode não ser a melhor ideia. Existem muitas histórias de novos desenvolvedores do Meteor armazenando campos como isAdmin em profile… e então um usuário malicioso pode facilmente definir isso como verdadeiro sempre que quiser, tornando-se um administrador. Mesmo que você não esteja preocupado com isso, não é uma boa ideia permitir que usuários mal-intencionados armazenem quantidades arbitrárias de dados em seu banco de dados.

Em vez de lidar com as especificidades desse campo, pode ser útil simplesmente ignorar sua existência completamente. Você pode fazer isso com segurança, desde que negue todas as gravações do cliente:

// Deny all client-side updates to user documents
Meteor.users.deny({
  update() { return true; }
});

Mesmo ignorando as implicações de segurança do perfil, não é uma boa ideia colocar todos os dados personalizados do seu aplicativo em um campo. O protocolo de transferência de dados do Meteor não faz diferenças profundamente aninhadas de campos, então é uma boa ideia nivelar seus objetos em muitos campos de nível superior no documento.
