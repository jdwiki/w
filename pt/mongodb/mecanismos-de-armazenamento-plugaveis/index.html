No MongoDB 3.0, MMAP (padrão) e WiredTiger são os mecanismos de armazenamento estáveis. Normalmente, se o seu aplicativo for de leitura intensa, use o MMAP. Se a gravação for pesada, use o WiredTiger.

Sua solução também pode ter membros do conjunto de réplicas mistas em que você pode ter um nó configurado com MMAP e outro com WiredTiger. Você pode usar um para inserir dados massivos e o outro para ler com ferramentas analíticas.

Após o MongoDB 3.2, o WiredTiger se torna o mecanismo padrão.

Tigre com Fio #

O WiredTiger suporta árvores LSM para armazenar índices. As árvores LSM são mais rápidas para operações de gravação quando você precisa gravar grandes cargas de trabalho de inserções aleatórias.

No WiredTiger, não há atualizações in-loco. Se você precisar atualizar um elemento de um documento, um novo documento será inserido enquanto o documento antigo será excluído.

O WiredTiger também oferece simultaneidade em nível de documento. Ele assume que duas operações de gravação não afetarão o mesmo documento, mas se isso acontecer, uma operação será rebobinada e executada posteriormente. Isso é um grande aumento de desempenho se os retrocessos forem raros.

O WiredTiger suporta algoritmos Snappy e zLib para compactação de dados e índices no sistema de arquivos. Snappy é o padrão. É menos intensivo da CPU, mas tem uma taxa de compactação menor que o zLib.

Como usar o WiredTiger Engine #

mongod --storageEngine wiredTiger --dbpath <newWiredTigerDBPath>

Observação:

  1. Após o mongodb 3.2, o mecanismo padrão é o WiredTiger.

  2. O newWiredTigerDBPath não deve conter dados de outro mecanismo de armazenamento. Para migrar seus dados, você precisa despejá-los e reimportá-los no novo mecanismo de armazenamento.

     mongodump --out <exportDataDestination>
 mongod --storageEngine wiredTiger --dbpath <newWiredTigerDBPath>
 mongorestore <exportDataDestination>

MMAP #

MMAP é um mecanismo de armazenamento plugável que foi nomeado após o comando mmap() do Linux. Ele mapeia arquivos para a memória virtual e otimiza as chamadas de leitura. Se você tem um arquivo grande, mas precisa ler apenas uma pequena parte dele, mmap() é muito mais rápido que uma chamada read() que traria o arquivo inteiro para a memória.

Uma desvantagem é que você não pode ter duas chamadas de gravação sendo processadas em paralelo para a mesma coleção. Portanto, o MMAP possui bloqueio em nível de coleção (e não bloqueio em nível de documento, como o WiredTiger oferece). Esse bloqueio de coleção é necessário porque um índice MMAP pode fazer referência a vários documentos e, se esses documentos pudessem ser atualizados simultaneamente, o índice seria inconsistente.

Em memória #

Todos os dados são armazenados na memória (RAM) para leitura/acesso mais rápido.

##mongo-rocks Um mecanismo de valor-chave criado para integrar-se ao RocksDB do Facebook.

Fusion-io #

Um mecanismo de armazenamento criado pela SanDisk que possibilita contornar a camada do sistema de arquivos do SO e gravar diretamente no dispositivo de armazenamento.

MeuMX #

Um mecanismo de armazenamento criado pela Percona que usa índices de árvore fractal.
