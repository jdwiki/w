No caso básico, um modelo é uma classe Python que mapeia para uma única tabela de banco de dados. Os atributos da classe mapeiam para colunas na tabela e uma instância da classe representa uma linha na tabela do banco de dados. Os modelos herdam de django.db.models.Model que fornece uma API rica para adicionar e filtrar resultados do banco de dados.

[Crie seu primeiro modelo](https://www.wikiod.com/pt/django/modelos#Criando seu primeiro modelo)

Adicionando uma representação de string de um modelo #

Para criar uma apresentação legível por humanos de um objeto de modelo, você precisa implementar o método Model.__str__() (ou Model.__unicode__() em python2). Este método será chamado sempre que você chamar str() em uma instância do seu modelo (incluindo, por exemplo, quando o modelo for usado em um modelo). Aqui está um exemplo:

  1. Crie um modelo de livro.

    # your_app/models.py

from django.db import models

class Book(models.Model):
    name = models.CharField(max_length=50)
    author = models.CharField(max_length=50)

  2. Crie uma instância do modelo e salve-a no banco de dados:

    >>> himu_book = Book(name='Himu Mama', author='Humayun Ahmed')
>>> himu_book.save()

  3. Execute print() na instância:

    >>> print(himu_book)
<Book: Book object>

<Book: Book object>, a saída padrão, não nos ajuda. Para corrigir isso, vamos adicionar um método __str__.

from django.utils.encoding import python_2_unicode_compatible

@python_2_unicode_compatible
class Book(models.Model):
    name = models.CharField(max_length=50)
    author = models.CharField(max_length=50)

    def __str__(self):
        return '{} by {}'.format(self.name, self.author)

Observe que o decorador python_2_unicode_compatible é necessário somente se você quiser que seu código seja compatível com python 2. Este decorador copia o método __str__ para criar um método __unicode__. Importe-o de django.utils.encoding.

Agora, se chamarmos a função print a instância do livro novamente:

>>> print(himu_book)
Himu Mama by Humayun Ahmed

Muito melhor!

A representação de string também é usada quando o modelo é usado em um ModelForm para os campos ForeignKeyField e ManyToManyField.

Modelos avançados #

Um modelo pode fornecer muito mais informações do que apenas os dados sobre um objeto. Vamos ver um exemplo e dividi-lo em para o que é útil:

from django.db import models
from django.urls import reverse
from django.utils.encoding import python_2_unicode_compatible

@python_2_unicode_compatible
class Book(models.Model):
    slug = models.SlugField()
    title = models.CharField(max_length=128)
    publish_date = models.DateField()

    def get_absolute_url(self):
        return reverse('library:book', kwargs={'pk':self.pk})

    def __str__(self):
        return self.title

    class Meta:
        ordering = ['publish_date', 'title']

Chave primária automática #

Você pode notar o uso de self.pk no método get_absolute_url. O campo pk é um alias para a chave primária de um modelo. Além disso, o django adicionará automaticamente uma chave primária se estiver faltando. Isso é uma coisa a menos para se preocupar e permite que você defina a chave estrangeira para qualquer modelo e obtenha-os facilmente.

URL absoluto #

A primeira função definida é get_absolute_url. Dessa forma, se você tiver um livro, poderá obter um link para ele sem mexer na tag url, resolver, atributo e similares. Basta chamar book.get_absolute_url e você obterá o link certo. Como bônus, seu objeto no admin do django ganhará um botão “view on site”.

Representação de string #

Tenha um método __str__ que permita usar o objeto quando precisar exibi-lo. Por exemplo, com o método anterior, adicionar um link ao livro em um modelo é tão simples quanto <a href="{{ book.get_absolute_url }}">{{ livro }}</a>. Direto ao ponto. Este método também controla o que é exibido no menu suspenso de administração, para chave estrangeira, por exemplo.

O decorador de classe permite que você defina o método uma vez para __str__ e __unicode__ no python 2 sem causar problemas no python 3. Se você espera que seu aplicativo seja executado em ambas as versões, esse é o caminho a seguir.

Campo de lesma #

O campo slug é semelhante a um campo char, mas aceita menos símbolos. Por padrão, apenas letras, números, sublinhados ou hífens. É útil se você deseja identificar um objeto usando uma boa representação, em url por exemplo.

A classe Meta #

A classe Meta nos permite definir muito mais informações sobre toda a coleção de itens. Aqui, apenas a ordenação padrão é definida. É útil com o objeto ListView, por exemplo. É preciso uma lista idealmente curta de campo para usar para classificação. Aqui, o livro será classificado primeiro pela data de publicação e depois pelo título, se a data for a mesma.

Outros atributos frequentes são verbose_name e verbose_name_plural. Por padrão, eles são gerados a partir do nome do modelo e devem funcionar bem. Mas a forma plural é ingênua, simplesmente anexando um ’s’ ao singular, então você pode querer defini-lo explicitamente em alguns casos.

Aplicando as alterações no banco de dados (Migrações) #

Depois de criar um novo modelo ou modificar modelos existentes, você precisará gerar migrações para suas alterações e, em seguida, aplicar as migrações ao banco de dados especificado. Isso pode ser feito usando o sistema de migração embutido do Django. Usando o utilitário manage.py quando estiver no diretório raiz do projeto:

python manage.py makemigrations <appname>

O comando acima criará os scripts de migração necessários no subdiretório migrations do seu aplicativo. Se você omitir o parâmetro <appname>, todos os aplicativos definidos no argumento INSTALLED_APPS de settings.py serão processados. Se achar necessário, você pode editar as migrações.

Você pode verificar quais migrações são necessárias sem realmente criar a migração, use a opção –dry-run, por exemplo:

python manage.py makemigrations --dry-run

Para aplicar as migrações:

python manage.py migrate <appname>

O comando acima executará os scripts de migração gerados na primeira etapa e atualizará fisicamente o banco de dados.

Se o modelo do banco de dados existente for alterado, o comando a seguir será necessário para fazer as alterações necessárias.

python manage.py migrate --run-syncdb

O Django criará a tabela com o nome <appname>_<classname> por padrão. Às vezes você não quer usá-lo. Se você quiser mudar o nome padrão, você pode anunciar o nome da tabela configurando a db_table na classe Meta:

from django.db import models

class YourModel(models.Model):
    parms = models.CharField()
    class Meta:
        db_table = "custom_table_name"

Se você quiser ver qual código SQL será executado por uma determinada migração, basta executar este comando:

python manage.py sqlmigrate <app_label> <migration_number>

Django >1.10 A nova opção makemigrations --check faz com que o comando saia com um status diferente de zero quando são detectadas alterações de modelo sem migrações.

Consulte Migrações para obter mais detalhes sobre migrações.

Criando seu primeiro modelo #

Os modelos são normalmente definidos no arquivo models.py no subdiretório do seu aplicativo. A classe Model do módulo django.db.models é uma boa classe inicial para estender seus modelos. Por exemplo:

from django.db import models

class Book(models.Model):
    title = models.CharField(max_length=100)
    author = models.ForeignKey('Author', on_delete=models.CASCADE, related_name='authored_books')
    publish_date = models.DateField(null=True, blank=True)

    def __str__(self): # __unicode__ in python 2.*
        return self.title

Cada atributo em um modelo representa uma coluna no banco de dados.

  • title é um texto com um comprimento máximo de 100 caracteres
  • author é uma ForeignKey que representa um relacionamento com outro modelo/tabela, neste caso Author (usado apenas para fins de exemplo). on_delete informa ao banco de dados o que fazer com o objeto caso o objeto relacionado (um Autor) seja excluído. (Deve ser notado que desde o django 1.9 on_delete pode ser usado como o segundo argumento posicional. No django 2 é um argumento requerido e é aconselhável tratá-lo como tal imediatamente. Em versões mais antigas, o padrão será CASCADE.)
  • publish_date armazena uma data. Ambos null e blank são definidos como True para indicar que não é um campo obrigatório (ou seja, você pode adicioná-lo posteriormente ou deixá-lo em branco).

Junto com os atributos definimos um método __str__ que retorna o título do livro que será usado como sua representação string quando necessário, ao invés do padrão.

Chave primária UUID #

Um modelo por padrão usará uma chave primária de incremento automático (inteiro). Isso lhe dará uma sequência de teclas 1, 2, 3.

Diferentes tipos de chave primária podem ser definidos em um modelo com pequenas alterações no modelo.

Um UUID é um identificador universalmente exclusivo, este é um identificador aleatório de 32 caracteres que pode ser usado como um ID. Esta é uma boa opção para usar quando você não quer que os IDs sequenciais sejam atribuídos aos registros em seu banco de dados. Quando usado no PostgreSQL, ele armazena em um tipo de dados uuid, caso contrário, em um char(32).

import uuid
from django.db import models

class ModelUsingUUID(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)

A chave gerada estará no formato 7778c552-73fc-4bc4-8bf9-5a2f6f7b7f47

Criando um modelo com relacionamentos #

Relação de muitos para um

from django.db import models

class Author(models.Model):
   name = models.CharField(max_length=50)

#Book has a foreignkey (many to one) relationship with author
class Book(models.Model):
    author = models.ForeignKey(Author, on_delete=models.CASCADE)
    publish_date = models.DateField()

Opção mais genérica. Pode ser usado em qualquer lugar que você gostaria de representar um relacionamento

Relação de muitos para muitos

class Topping(models.Model):
    name = models.CharField(max_length=50)

# One pizza can have many toppings and same topping can be on many pizzas
class Pizza(models.Model):
    name = models.CharField(max_length=50)
    toppings = models.ManyToManyField(Topping)

Internamente, isso é representado por outra tabela. E ManyToManyField deve ser colocado em modelos que serão editados em um formulário. Por exemplo: Appointment terá um ManyToManyField chamado Customer, Pizza terá Toppings e assim por diante.

Relação muitos-para-muitos usando classes Through

class Service(models.Model):
     name = models.CharField(max_length=35)

class Client(models.Model):
    name = models.CharField(max_length=35)
    age = models.IntegerField()
    services = models.ManyToManyField(Service, through='Subscription')

class Subscription(models.Model):
     client = models.ForeignKey(Client)
     service = models.ForeignKey(Service)
     subscription_type = models.CharField(max_length=1, choices=SUBSCRIPTION_TYPES)
     created_at = models.DateTimeField(default=timezone.now)

Dessa forma, podemos realmente manter mais metadados sobre um relacionamento entre duas entidades. Como pode ser visto, um cliente pode ser subscrito a vários serviços através de vários tipos de subscrição. A única diferença neste caso é que para adicionar novas instâncias à relação M2M, não se pode usar o método de atalho pizza.toppings.add(topping), ao invés disso, um novo objeto da classe through deve ser criado, Subscription.objects.create(client=client, service=service, subscription_type='p')

Em outras linguagens, através de tabelas também são conhecidos como JoinColumn, Tabela de interseção ou tabela de mapeamento

Relacionamento Individual

class Employee(models.Model):
   name = models.CharField(max_length=50)
   age = models.IntegerField()
   spouse = models.OneToOneField(Spouse)

class Spouse(models.Model):
   name = models.CharField(max_length=50)

Use esses campos quando você tiver apenas uma relação de composição entre os dois modelos.

Consultas básicas do Django DB #

O Django ORM é uma abstração poderosa que permite armazenar e recuperar dados do banco de dados sem escrever consultas sql por conta própria.

Vamos supor os seguintes modelos:

class Author(models.Model):
   name = models.CharField(max_length=50)

class Book(models.Model): 
   name = models.CharField(max_length=50)
   author = models.ForeignKey(Author)

Assumindo que você adicionou o código acima a um aplicativo django e execute o comando migrate (para que seu banco de dados seja criado). Inicie o shell do Django por

python manage.py shell

Isso inicia o shell padrão do python, mas com as bibliotecas relevantes do Django importadas, para que você possa se concentrar diretamente nas partes importantes.

Comece importando os modelos que acabamos de definir (suponho que isso seja feito em um arquivo models.py)

from .models import Book, Author

Execute sua primeira consulta de seleção:

>>> Author.objects.all() 
[]
>>> Book.objects.all()
[]

Vamos criar um objeto autor e livro:

>>> hawking = Author(name="Stephen hawking")
>>> hawking.save()
>>> history_of_time = Book(name="history of time", author=hawking)
>>> history_of_time.save()

ou use a função create para criar objetos de modelo e salvar em um código de linha

>>> wings_of_fire = Book.objects.create(name="Wings of Fire", author="APJ Abdul Kalam")

Agora vamos executar a consulta

>>> Book.objects.all()
[<Book: Book object>]
>>> book = Book.objects.first() #getting the first book object
>>> book.name
u'history of time'

Vamos adicionar uma cláusula where à nossa consulta de seleção

>>> Book.objects.filter(name='nothing')
[]
>>> Author.objects.filter(name__startswith='Ste')
[<Author: Author object>]

Para obter os detalhes sobre o autor de um determinado livro

>>> book = Book.objects.first() #getting the first book object
>>> book.author.name # lookup on related model
u'Stephen hawking'

Para obter todos os livros publicados por Stephen Hawking (Livro de pesquisa de seu autor)

>>> hawking.book_set.all()
[<Book: Book object>]

_set é a notação usada para “Pesquisas inversas”, ou seja, enquanto o campo de pesquisa está no modelo Book, podemos usar book_set em um objeto de autor para obter todos os seus livros.

Uma tabela básica não gerenciada. #

Em algum ponto de seu uso do Django, você pode querer interagir com tabelas que já foram criadas ou com visualizações de banco de dados. Nesses casos, você não gostaria que o Django gerenciasse as tabelas por meio de suas migrações. Para configurar isso, você precisa adicionar apenas uma variável à classe Meta do seu modelo: managed = False.

Aqui está um exemplo de como você pode criar um modelo não gerenciado para interagir com uma exibição de banco de dados:

class Dummy(models.Model):
    something = models.IntegerField()

    class Meta:
       managed = False

Isso pode ser mapeado para uma exibição definida no SQL da seguinte maneira.

CREATE VIEW myapp_dummy AS 
SELECT id, something FROM complicated_table 
WHERE some_complicated_condition = True

Depois de criar esse modelo, você pode usá-lo como faria com qualquer outro modelo:

>>> Dummy.objects.all()
[<Dummy: Dummy object>, <Dummy: Dummy object>, <Dummy: Dummy object>]
>>> Dummy.objects.filter(something=42)
[<Dummy: Dummy object>]

Valores calculados #

Uma vez que um objeto de modelo foi buscado, ele se torna uma instância totalmente realizada da classe. Como tal, quaisquer métodos adicionais podem ser acessados ​​em formulários e serializadores (como Django Rest Framework).

Usar propriedades python é uma maneira elegante de representar valores adicionais que não são armazenados no banco de dados devido a circunstâncias variadas.

def expire():
    return timezone.now() + timezone.timedelta(days=7)

class Coupon(models.Model):
    expiration_date = models.DateField(default=expire)

    @property
    def is_expired(self):
        return timezone.now() > self.expiration_date

Embora na maioria dos casos você possa complementar os dados com anotações em seus conjuntos de consultas, os valores calculados como propriedades do modelo são ideais para cálculos que não podem ser avaliados simplesmente dentro do escopo de uma consulta.

Além disso, as propriedades, uma vez que são declaradas na classe python e não como parte do esquema, não estão disponíveis para consulta.

Misturas de modelos #

Nos mesmos casos, modelos diferentes podem ter os mesmos campos e os mesmos procedimentos no ciclo de vida do produto. Para lidar com essas semelhanças sem ter herança de repetição de código pode ser usado. Em vez de herdar uma classe inteira, o padrão de projeto mixin nos oferece para herdar (ou alguns dizem incluir) alguns métodos e atributos. Vejamos um exemplo:

class PostableMixin(models.Model):
    class Meta:
        abstract=True
    
    sender_name = models.CharField(max_length=128)
    sender_address = models.CharField(max_length=255)
    receiver_name = models.CharField(max_length=128)
    receiver_address = models.CharField(max_length=255)
    post_datetime = models.DateTimeField(auto_now_add=True)
    delivery_datetime = models.DateTimeField(null=True)
    notes = models.TextField(max_length=500)

class Envelope(PostableMixin):
    ENVELOPE_COMMERCIAL = 1
    ENVELOPE_BOOKLET = 2
    ENVELOPE_CATALOG = 3

    ENVELOPE_TYPES = (
        (ENVELOPE_COMMERCIAL, 'Commercial'),
        (ENVELOPE_BOOKLET, 'Booklet'),
        (ENVELOPE_CATALOG, 'Catalog'),
    )

    envelope_type = models.PositiveSmallIntegerField(choices=ENVELOPE_TYPES)

class Package(PostableMixin):
    weight = models.DecimalField(max_digits=6, decimal_places=2)
    width = models.DecimalField(max_digits=5, decimal_places=2)
    height = models.DecimalField(max_digits=5, decimal_places=2)
    depth = models.DecimalField(max_digits=5, decimal_places=2)

Para transformar um modelo em uma classe abstrata, você precisará mencionar abstract=True em sua classe interna Meta. O Django não cria nenhuma tabela para modelos abstratos no banco de dados. No entanto, para os modelos Envelope e Package, as tabelas correspondentes seriam criadas no banco de dados.

Além disso, nos campos alguns métodos de modelo serão necessários em mais de um modelo. Assim, esses métodos podem ser adicionados a mixins para evitar a repetição de código. Por exemplo, se criarmos um método para definir a data de entrega para PostableMixin, ele será acessível de ambos os filhos:

class PostableMixin(models.Model):
    class Meta:
        abstract=True

    ...
    ...

    def set_delivery_datetime(self, dt=None):
        if dt is None:
            from django.utils.timezone import now
            dt = now()

        self.delivery_datetime = dt
        self.save()

Este método pode ser usado da seguinte forma nas crianças:

>> envelope = Envelope.objects.get(pk=1)
>> envelope.set_delivery_datetime()

>> pack = Package.objects.get(pk=1)
>> pack.set_delivery_datetime()

Herança #

A herança entre modelos pode ser feita de duas maneiras:

  • uma classe abstrata comum (veja o exemplo “Model mixins”)
  • um modelo comum com várias tabelas

A herança de várias tabelas criará uma tabela para os campos comuns e uma por exemplo de modelo filho:

from django.db import models

class Place(models.Model):
    name = models.CharField(max_length=50)
    address = models.CharField(max_length=80)

class Restaurant(Place):
    serves_hot_dogs = models.BooleanField(default=False)
    serves_pizza = models.BooleanField(default=False)

irá criar 2 tabelas, uma para Place e outra para Restaurant com um campo oculto OneToOne para Place para os campos comuns.

observe que isso precisará de uma consulta extra nas tabelas de lugares toda vez que você buscar um Objeto Restaurante.
