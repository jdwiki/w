Ao usar o CBV, geralmente precisamos saber exatamente quais métodos podemos substituir para cada classe genérica. Esta página da documentação do django lista todas as classes genéricas com todos os seus métodos achatados e os atributos de classe que podemos usar.

Além disso, o site Classy Class Based View fornece as mesmas informações com uma interface interativa agradável.

Criação de formulários e objetos #

Escrever uma view para criar um objeto pode ser bem chato. Você tem que exibir um formulário, você tem que validá-lo, você tem que salvar o item ou devolver o formulário com erro. A menos que você use uma das [visualizações de edição genérica][visualizações_edição].

app/views.py #

from django.core.urlresolvers import reverse_lazy
from django.views.generic.edit import CreateView, UpdateView, DeleteView
from .models import Pokemon


class PokemonCreate(CreateView):
    model = Pokemon
    fields = ['name', 'species']


class PokemonUpdate(UpdateView):
    model = Pokemon
    fields = ['name', 'species']


class PokemonDelete(DeleteView):
    model = Pokemon
    success_url = reverse_lazy('pokedex')

CreateView e UpdateView têm dois atributos obrigatórios, model e fields. Por padrão, ambos usam um nome de modelo baseado no nome do modelo com o sufixo ‘_form’. Você pode alterar apenas o sufixo com o atributo template_name_suffix. O DeleteView mostra uma mensagem de confirmação antes de excluir o objeto.

Ambos UpdateView e DeleteView precisam buscar no objeto. Eles usam o mesmo método do DetailView, extraindo a variável da url e combinando os campos do objeto.

app/templates/app/pokemon_form.html (extrair) #

<form action="" method="post">
    {% csrf_token %}
    {{ form.as_p }}
    <input type="submit" value="Save" />
</form>

form contém o formulário com todos os campos necessários. Aqui, ele será exibido com um parágrafo para cada campo por causa de as_p.

app/templates/app/pokemon_confirm_delete.html (extrair) #

<form action="" method="post">
    {% csrf_token %}
    <p>Are you sure you want to delete "{{ object }}"?</p>
    <input type="submit" value="Confirm" />
</form>

A tag csrf_token é necessária por causa da proteção do Django contra falsificação de solicitação. A ação do atributo é deixada em branco, pois a url que exibe o formulário é a mesma que está lidando com a exclusão/salvamento.

Dois problemas permanecem com o modelo, se estiver usando o mesmo que com a lista e o exemplo de detalhes. Primeiro, criar e atualizar reclamará sobre a falta de um URL de redirecionamento. Isso pode ser resolvido adicionando um get_absolute_url ao modelo pokemon. O segundo problema é a confirmação de exclusão não exibindo informações significativas. Para resolver isso, a solução mais fácil é adicionar uma representação de string.

app/models.py #

from django.db import models
from django.urls import reverse
from django.utils.encoding import python_2_unicode_compatible


@python_2_unicode_compatible
class Pokemon(models.Model):
    name = models.CharField(max_length=24)
    species = models.CharField(max_length=48)

    def get_absolute_url(self):
        return reverse('app:pokemon', kwargs={'pk':self.pk})

    def __str__(self):
        return self.name

O decorador de classe garantirá que tudo funcione sem problemas no python 2.

Visualizações de lista e detalhes #

As visualizações de modelo são boas para páginas estáticas e você pode usá-las para tudo com get_context_data, mas seria um pouco melhor do que usar funções como visualizações.

Digite ListView e DetailView

app/models.py #

from django.db import models

class Pokemon(models.Model):
    name = models.CharField(max_length=24)
    species = models.CharField(max_length=48)
    slug = models.CharField(max_length=48)

app/views.py #

from django.views.generic import ListView, DetailView
from .models import Pokemon


class PokedexView(ListView):
    """ Provide a list of Pokemon objects """
    model = Pokemon
    paginate_by = 25

class PokemonView(DetailView):
    model = Pokemon

Isso é tudo que você precisa para gerar uma visão listando todos os seus objetos de modelos e visualizações de um item singular. A lista é mesmo paginada. Você pode fornecer template_name se quiser algo específico. Por padrão, é gerado a partir do nome do modelo.

app/templates/app/pokemon_list.html #

<!DOCTYPE html>
<title>Pokedex</title>
<ul>{% for pokemon in pokemon_list %}
    <li><a href="{% url "app:pokemon" pokemon.pk %}">{{ pokemon.name }}</a>
        &ndash; {{ pokemon.species }}
</ul>

O contexto é preenchido com a lista de objetos sob dois nomes, object_list e uma segunda compilação a partir do nome do modelo, aqui pokemon_list. Se você tem paginado a lista, você tem que cuidar do link seguinte e anterior também. O objeto [Paginator][paginator] pode ajudar com isso, ele também está disponível nos dados de contexto.

app/templates/app/pokemon_detail.html #

<!DOCTYPE html>
<title>Pokemon {{ pokemon.name }}</title>
<h1>{{ pokemon.name }}</h1>
<h2>{{ pokemon.species }} </h2>

Como antes, o contexto é preenchido com seu objeto de modelo sob o nome object e pokemon, sendo o segundo derivado do nome do modelo.

app/urls.py #

from django.conf.urls import url
from . import views

app_name = 'app'
urlpatterns = [
    url(r'^pokemon/$', views.PokedexView.as_view(), name='pokedex'),
    url(r'^pokemon/(?P<pk>\d+)/$', views.PokemonView.as_view(), name='pokemon'),
]

Neste snippet, o URL da visualização de detalhes é criado usando a chave primária. Também é possível usar um slug como argumento. Isso fornece um URL mais bonito e mais fácil de lembrar. No entanto, requer a presença de um campo chamado slug em seu modelo.

url(r'^pokemon/(?P<slug>[A-Za-z0-9_-]+)/$', views.PokemonView.as_view(), name='pokemon'),

Se um campo chamado slug não estiver presente, você pode usar a configuração slug_field em DetailView para apontar para um campo diferente.

Para paginação, use um parâmetro de obtenção de página ou coloque uma página diretamente na url.

Visualizações baseadas em classe #

As visualizações baseadas em classe permitem que você se concentre no que torna suas visualizações especiais.

Uma página estática sobre pode não ter nada de especial, exceto o modelo usado. Use um TemplateView! Tudo o que você precisa fazer é definir um nome de modelo. Tarefa concluída. Próximo.

#views.py

from django.views.generic import TemplateView


class AboutView(TemplateView):
    template_name = "about.html"

#urls.py from django.conf.urls import url from . import views

urlpatterns = [
    url('^about/', views.AboutView.as_view(), name='about'),
]

Observe como não usamos diretamente AboutView na url. Isso porque um callable é esperado e é exatamente isso que as_view() retorna.

Dados de contexto #

Às vezes, seu modelo precisa de um pouco mais de informações. Por exemplo, gostaríamos de ter o usuário no cabeçalho da página, com um link para seu perfil ao lado do link de logout. Nesses casos, use o método get_context_data.

#views.py class BookView(DetailView): template_name = “book.html”

    def get_context_data(self, **kwargs)
        """ get_context_data let you fill the template context """
        context = super(BookView, self).get_context_data(**kwargs)
        # Get Related publishers
        context['publishers'] = self.object.publishers.filter(is_active=True)
        return context

Você precisa chamar o método get_context_data na superclasse e ele retornará a instância de contexto padrão. Qualquer item que você adicionar a este dicionário estará disponível para o modelo.

livro.html #

<h3>Active publishers</h3>
<ul>
  {% for publisher in publishers %}
    <li>{{ publisher.name }}</li>
  {% endfor %}
</ul>

Uma visualização, vários formulários #

Aqui está um exemplo rápido de usar vários formulários em uma visualização do Django.

from django.contrib import messages
from django.views.generic import TemplateView

from .forms import AddPostForm, AddCommentForm
from .models import Comment

class AddCommentView(TemplateView):

    post_form_class = AddPostForm
    comment_form_class = AddCommentForm
    template_name = 'blog/post.html'

    def post(self, request):
        post_data = request.POST or None
        post_form = self.post_form_class(post_data, prefix='post')
        comment_form = self.comment_form_class(post_data, prefix='comment')

        context = self.get_context_data(post_form=post_form,
                                        comment_form=comment_form)

        if post_form.is_valid():
            self.form_save(post_form)
        if comment_form.is_valid():
            self.form_save(comment_form)

        return self.render_to_response(context)     

    def form_save(self, form):
        obj = form.save()
        messages.success(self.request, "{} saved successfully".format(obj))
        return obj

    def get(self, request, *args, **kwargs):
        return self.post(request, *args, **kwargs)
