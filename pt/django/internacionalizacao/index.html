Introdução à Internacionalização #

Configurando #

configurações.py #

from django.utils.translation import ugettext_lazy as _

USE_I18N = True  # Enable Internationalization
LANGUAGE_CODE = 'en'  # Language in which original texts are written
LANGUAGES = [  # Available languages
    ('en', _("English")),
    ('de', _("German")),
    ('fr', _("French")),
]

# Make sure the LocaleMiddleware is included, AFTER SessionMiddleware
# and BEFORE middlewares using internationalization (such as CommonMiddleware)
MIDDLEWARE_CLASSES = [
   'django.contrib.sessions.middleware.SessionMiddleware',
   'django.middleware.locale.LocaleMiddleware',
   'django.middleware.common.CommonMiddleware',
]

Marcando strings como traduzíveis #

O primeiro passo na tradução é marcar as strings como traduzíveis. Isso os está passando por uma das funções gettext (Veja a seção Sintaxe). Por exemplo, aqui está uma definição de modelo de exemplo:

from django.utils.translation import ugettext_lazy as _
# It is common to import gettext as the shortcut `_` as it is often used
# several times in the same file.

class Child(models.Model):

    class Meta:
        verbose_name = _("child")
        verbose_name_plural = _("children")

    first_name = models.CharField(max_length=30, verbose_name=_("first name"))
    last_name = models.CharField(max_length=30, verbose_name=_("last name"))
    age = models.PositiveSmallIntegerField(verbose_name=_("age"))

Todas as strings encapsuladas em _() agora são marcadas como traduzíveis. Quando impressos, sempre serão exibidos como a string encapsulada, qualquer que seja o idioma escolhido (já que ainda não há tradução disponível).

Traduzindo strings #

Este exemplo é suficiente para começar com a tradução. Na maioria das vezes, você só vai querer marcar strings como traduzíveis para antecipar a internacionalização em potencial do seu projeto. Assim, isso é coberto [em outro exemplo][2].

[2]: https://www.wikiod.com/pt/django/internacionalizacao#Traduzindo strings

Tradução preguiçosa vs não preguiçosa #

Ao usar a tradução não lenta, as strings são traduzidas imediatamente.

>>> from django.utils.translation import activate, ugettext as _
>>> month = _("June")
>>> month
'June'
>>> activate('fr')
>>> _("June")
'juin'
>>> activate('de')
>>> _("June")
'Juni'
>>> month
'June'

Ao usar a preguiça, a tradução só ocorre quando realmente usada.

>>> from django.utils.translation import activate, ugettext_lazy as _
>>> month = _("June")
>>> month
<django.utils.functional.lazy.<locals>.__proxy__ object at 0x7f61cb805780>
>>> str(month)
'June'
>>> activate('fr')
>>> month
<django.utils.functional.lazy.<locals>.__proxy__ object at 0x7f61cb805780>
>>> "month: {}".format(month)
'month: juin'
>>> "month: %s" % month
'month: Juni'

Você precisa usar a tradução preguiçosa nos casos em que:

  • A tradução pode não ser ativada (idioma não selecionado) quando _("alguma string") é avaliada
  • Algumas strings podem ser avaliadas apenas na inicialização (por exemplo, em atributos de classe, como definições de campos de modelo e formulário)

Tradução em modelos #

Para habilitar a tradução em templates você deve carregar a biblioteca i18n.

{% load i18n %}

A tradução básica é feita com a tag template trans.

{% trans "Some translatable text" %}
{# equivalent to python `ugettext("Some translatable text")` #}

A tag de modelo trans suporta contexto:

{% trans "May" context "month" %}
{# equivalent to python `pgettext("May", "month")` #}

Para incluir marcadores de posição em sua string de tradução, como em:

_("My name is {first_name} {last_name}").format(first_name="John", last_name="Doe")

Você terá que usar a tag de modelo blocktrans:

{% blocktrans with first_name="John" last_name="Doe" %}
  My name is {{ first_name }} {{ last_name }}
{% endblocktrans %}

Claro que em vez de "John" e "Doe" você pode ter variáveis ​​e filtros:

{% blocktrans with first_name=user.first_name last_name=user.last_name|title %}
  My name is {{ first_name }} {{ last_name }}
{% endblocktrans %}

Se first_name e last_name já estiverem em seu contexto, você pode até mesmo omitir a cláusula with:

{% blocktrans %}My name is {{ first_name }} {{ last_name }}{% endblocktrans %}

No entanto, apenas variáveis ​​de contexto de “nível superior” podem ser usadas. Isto NÃO vai funcionar:

{% blocktrans %}
    My name is {{ user.first_name }} {{ user.last_name }}
{% endblocktrans %}

Isso ocorre principalmente porque o nome da variável é usado como espaço reservado nos arquivos de tradução.

A tag de template blocktrans também aceita pluralização.

{% blocktrans count nb=users|length }}
    There is {{ nb }} user.
{% plural %}
    There are {{ nb }} users.
{% endblocktrans %}

Finalmente, independentemente da biblioteca i18n, você pode passar strings traduzíveis para tags de modelo usando a sintaxe _("").

{{ site_name|default:_("It works!") }}
{% firstof var1 var2 _("translatable fallback") %}

Este é um sistema de template django embutido mágico para imitar uma sintaxe de chamada de função, mas isso não é uma chamada de função. _("Funciona!") passado para a tag de template default como uma string '_("Funciona!")' que é então analisada como uma string traduzível, assim como name seria analisado como uma variável e "name" seriam analisados ​​como uma string.

Traduzindo strings #

Para traduzir strings, você terá que criar arquivos de tradução. Para fazer isso, o django vem com o comando de gerenciamento makemessages.

$ django-admin makemessages -l fr
processing locale fr

O comando acima descobrirá todas as strings marcadas como traduzíveis em seus aplicativos instalados e criará um arquivo de idioma para cada aplicativo para tradução em francês. Por exemplo, se você tiver apenas um aplicativo myapp contendo strings traduzíveis, isso criará um arquivo myapp/locale/fr/LC_MESSAGES/django.po. Este arquivo pode ter a seguinte aparência:

# SOME DESCRIPTIVE TITLE
# Copyright (C) YEAR THE PACKAGE'S COPYRIGHT HOLDER
# This file is distributed under the same license as the PACKAGE package.
# FIRST AUTHOR <EMAIL@ADDRESS>, YEAR
#
#, fuzzy
msgid ""
msgstr ""
"Project-Id-Version: PACKAGE VERSION\n"
"Report-Msgid-Bugs-To: \n"
"POT-Creation-Date: 2016-07-24 14:01+0200\n"
"PO-Revision-Date: YEAR-MO-DA HO:MI+ZONE\n"
"Last-Translator: FULL NAME <EMAIL@ADDRESS>\n"
"Language-Team: LANGUAGE <LL@li.org>\n"
"Language: \n"
"MIME-Version: 1.0\n"
"Content-Type: text/plain; charset=UTF-8\n"
"Content-Transfer-Encoding: 8bit\n"

#: myapp/models.py:22
msgid "user"
msgstr ""

#: myapp/models.py:39
msgid "A user already exists with this email address."
msgstr ""

#: myapp/templates/myapp/register.html:155
#, python-format
msgid ""
"By signing up, you accept our <a href=\"%(terms_url)s\" "
"target=_blank>Terms of services</a>."
msgstr ""

Primeiro você terá que preencher os espaços reservados (enfatizados com letras maiúsculas). Em seguida, traduza as strings. msgid é a string marcada como traduzível em seu código. msgstr é onde você deve escrever a tradução da string logo acima.

Quando uma string contém espaços reservados, você também terá que incluí-los em sua tradução. Por exemplo, você traduzirá a mensagem mais recente da seguinte forma:

#: myapp/templates/myapp/register.html:155
#, python-format
msgid ""
"By signing up, you accept our <a href=\"%(terms_url)s\" "
"target=_blank>Terms of services</a>."
msgstr ""
"En vous inscrivant, vous acceptez nos <a href=\"%(terms_url)s\" "
"target=_blank>Conditions d'utilisation</a>"

Uma vez que seu arquivo de tradução esteja completo, você terá que compilar os arquivos .po em arquivos .mo. Isso é feito chamando o comando de gerenciamento compilemessages:

$ django-admin compilemessages

É isso, agora as traduções estão disponíveis.

Para atualizar seus arquivos de tradução ao fazer alterações em seu código, você pode executar novamente django-admin makemessages -l fr. Isto irá atualizar os arquivos .po, mantendo suas traduções existentes e adicionando as novas. As strings excluídas ainda estarão disponíveis nos comentários. Para atualizar arquivos .po para todos os idiomas, execute django-admin makemessages -a. Assim que seus arquivos .po forem atualizados, não se esqueça de executar django-admin compilemessages novamente para gerar arquivos .mo.

Caso de uso Noop #

(u)gettext_noop ​​permite que você marque uma string como traduzível sem realmente traduzi-la.

Um caso de uso típico é quando você deseja registrar uma mensagem para desenvolvedores (em inglês), mas também deseja exibi-la ao cliente (no idioma solicitado). Você pode passar uma variável para gettext, mas seu conteúdo não será descoberto como uma string traduzível porque é, por definição, variável..

# THIS WILL NOT WORK AS EXPECTED
import logging 
from django.contrib import messages

logger = logging.getLogger(__name__)

error_message = "Oops, something went wrong!"
logger.error(error_message)
messages.error(request, _(error_message))

A mensagem de erro não aparecerá no arquivo .po e você terá que lembrar que ela existe para adicioná-la manualmente. Para corrigir isso, você pode usar gettext_noop.

error_message = ugettext_noop("Oops, something went wrong!")
logger.error(error_message)
messages.error(request, _(error_message))

Agora a string "Oops, algo deu errado!" será descoberta e disponível no arquivo .po quando gerada. E o erro ainda será registrado em inglês para desenvolvedores.

Armadilhas comuns #

traduções confusas #

Às vezes, makemessages pode pensar que a string encontrada para tradução é um pouco semelhante à tradução já existente. Ele irá marcá-lo no arquivo .po com um comentário especial fuzzy como este:

#: templates/randa/map.html:91
#, fuzzy
msgid "Country"
msgstr "Länderinfo"

Mesmo que a tradução esteja correta ou você a atualizou para corrigir uma, ela não será usada para traduzir seu projeto, a menos que você remova a linha de comentário fuzzy.

Strings de várias linhas #

makemessages analisa arquivos em vários formatos, de texto simples a código python e não foi projetado para seguir todas as regras possíveis para ter strings de várias linhas nesses formatos. Na maioria das vezes, funcionará bem com strings de linha única, mas se você tiver uma construção como esta:

translation = _("firstline"
"secondline"
"thirdline")

Ele só pegará firstline para tradução. A solução para isso é evitar o uso de strings de várias linhas quando possível.
