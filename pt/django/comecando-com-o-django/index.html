Iniciando um projeto #

Django é um framework de desenvolvimento web baseado em Python. O Django 1.11 (a versão estável mais recente) requer a instalação do Python 2.7, 3.4, 3.5 ou 3.6. Supondo que o pip esteja disponível, a instalação é tão simples quanto executar o seguinte comando. Tenha em mente que omitir a versão mostrada abaixo instalará a versão mais recente do django:

$ pip install django

Para instalar uma versão específica do django, vamos supor que a versão seja django 1.10.5 , execute o seguinte comando:

$ pip install django==1.10.5

Os aplicativos da Web criados usando o Django devem residir em um projeto do Django. Você pode usar o comando django-admin para iniciar um novo projeto no diretório atual:

$ django-admin startproject myproject

em que myproject é um nome que identifica exclusivamente o projeto e pode consistir em números, letras e sublinhados.

Isso criará a seguinte estrutura de projeto:

myproject/
    manage.py
    myproject/
        __init__.py
        settings.py
        urls.py
        wsgi.py

Para executar o aplicativo, inicie o servidor de desenvolvimento

$ cd myproject
$ python manage.py runserver

Agora que o servidor está funcionando, visite http://127.0.0.1:8000/ com seu navegador web. Você verá a seguinte página:

Por padrão, o comando runserver inicia o servidor de desenvolvimento no IP interno na porta 8000. Este servidor será reiniciado automaticamente à medida que você fizer alterações em seu código. Mas caso você adicione novos arquivos, você terá que reiniciar manualmente o servidor.

Se você deseja alterar a porta do servidor, passe-a como um argumento de linha de comando.

$ python manage.py runserver 8080

Se você quiser alterar o IP do servidor, passe-o junto com a porta.

$ python manage.py runserver 0.0.0.0:8000

Observe que runserver é apenas para compilações de depuração e testes locais. Programas de servidor especializados (como Apache) devem sempre ser usados ​​em produção.

Adicionando um aplicativo Django

Um projeto Django geralmente contém vários apps. Esta é simplesmente uma maneira de estruturar seu projeto em módulos menores e de fácil manutenção. Para criar um aplicativo, vá para a pasta do projeto (onde manage.py está) e execute o comando startapp (mude myapp para o que você quiser):

python manage.py startapp myapp

Isso irá gerar a pasta myapp e alguns arquivos necessários para você, como models.py e views.py.

Para tornar o Django ciente de myapp, adicione-o ao seu settings.py:

# myproject/settings.py

# Application definition
INSTALLED_APPS = [
    ...
    'myapp',
]

A estrutura de pastas de um projeto Django pode ser alterada para se adequar à sua preferência. Às vezes, a pasta do projeto é renomeada para /src para evitar a repetição de nomes de pastas. Uma estrutura de pastas típica se parece com isso:

Ambiente Virtual #

Embora não seja estritamente obrigatório, é altamente recomendável iniciar seu projeto em um “ambiente virtual”. Um ambiente virtual é um contêiner (um diretório) que contém uma versão específica do Python e um conjunto de módulos (dependências) e que não interfere no Python nativo do sistema operacional ou em outros projetos no mesmo computador.

Ao configurar um ambiente virtual diferente para cada projeto em que você trabalha, vários projetos do Django podem ser executados em diferentes versões do Python e podem manter seus próprios conjuntos de dependências, sem risco de conflito.

Python 3.3+ #

O Python 3.3+ já inclui um módulo venv padrão, que você normalmente pode chamar de pyvenv. Em ambientes onde o comando pyvenv não está disponível, você pode acessar a mesma funcionalidade invocando diretamente o módulo como python3 -m venv.

Para criar o ambiente virtual:

$ pyvenv <env-folder>
# Or, if pyvenv is not available
$ python3 -m venv <env-folder>

Python 2 #

Se estiver usando o Python 2, você pode primeiro instalá-lo como um módulo separado do pip:

$ pip install virtualenv

E então crie o ambiente usando o comando virtualenv:

$ virtualenv <env-folder>

Ativar (qualquer versão) #

O ambiente virtual agora está configurado. Para usá-lo, ele deve estar ativado no terminal em que deseja usá-lo.

Para ‘ativar’ o ambiente virtual (qualquer versão do Python)

Linux como:

$ source <env-folder>/bin/activate

Janelas como:

<env-folder>\Scripts\activate.bat

Isso altera seu prompt para indicar que o ambiente virtual está ativo. (<pasta-env>) $

De agora em diante, tudo instalado usando pip será instalado em sua pasta virtual env, não em todo o sistema.

Para sair do ambiente virtual use deactivate :

(<env-folder>) $ deactivate

Alternativamente: use virtualenvwrapper #

Você também pode considerar usar virtualenvwrapper, o que torna a criação e ativação do virtualenv muito útil, além de separá-lo do seu código:

# Create a virtualenv
mkvirtualenv my_virtualenv

# Activate a virtualenv
workon my_virtualenv

# Deactivate the current virtualenv
deactivate

Alternativamente: use pyenv + pyenv-viritualenv #

Em ambientes onde você precisa lidar com várias versões do Python, você pode se beneficiar do virtualenv junto com o pyenv-virtualenv:

# Create a virtualenv for specific Python version
pyenv virtualenv 2.7.10 my-virtual-env-2.7.10

# Create a vritualenv for active python verion
pyenv virtualenv venv34

# Activate, deactivate virtualenv
pyenv activate <name>
pyenv deactivate

Ao usar virtualenvs, geralmente é útil definir seu PYTHONPATH e DJANGO_SETTINGS_MODULE no script [postactivate] 2.

#!/bin/sh
# This hook is sourced after this virtualenv is activated

# Set PYTHONPATH to isolate the virtualenv so that only modules installed
# in the virtualenv are available
export PYTHONPATH="/home/me/path/to/your/project_root:$VIRTUAL_ENV/lib/python3.4"

# Set DJANGO_SETTINGS_MODULE if you don't use the default `myproject.settings`
# or if you use `django-admin` rather than `manage.py`
export DJANGO_SETTINGS_MODULE="myproject.settings.dev"

Defina o caminho do seu projeto #

Muitas vezes também é útil definir o caminho do seu projeto dentro de um arquivo .project especial localizado em sua pasta <env-fold> base. Ao fazer isso, toda vez que você ativar seu ambiente virtual, ele mudará o diretório ativo para o caminho especificado.

Crie um novo arquivo chamado <env-folder>/.project. O conteúdo do arquivo deve ser SOMENTE o caminho do diretório do projeto.

/path/to/project/directory

Agora, inicie seu ambiente virtual (usando source <env-folder>/bin/activate ou workon my_virtualenv) e seu terminal mudará os diretórios para /path/to/project/directory.

Conceitos do Django #

django-admin é uma ferramenta de linha de comando que acompanha o Django. Ele vem com vários comandos úteis para começar e gerenciar um projeto Django. O comando é o mesmo que ./manage.py , com a diferença de que você não precisa estar no diretório do projeto. A variável de ambiente DJANGO_SETTINGS_MODULE precisa ser configurada.

Um projeto Django é uma base de código Python que contém um arquivo de configurações do Django. Um projeto pode ser criado pelo administrador do Django através do comando django-admin startproject NAME. O projeto normalmente tem um arquivo chamado manage.py no nível superior e um arquivo de URL raiz chamado urls.py. manage.py é uma versão específica do projeto django-admin, e permite que você execute comandos de gerenciamento nesse projeto. Por exemplo, para executar seu projeto localmente, use python manage.py runserver. Um projeto é composto de aplicativos Django.

Um aplicativo Django é um pacote Python que contém um arquivo de modelos (models.py por padrão) e outros arquivos, como urls e visualizações específicas do aplicativo. Um aplicativo pode ser criado através do comando django-admin startapp NAME (este comando deve ser executado dentro do diretório do seu projeto). Para um aplicativo fazer parte de um projeto, ele deve ser incluído na lista INSTALLED_APPS em settings.py. Se você usou a configuração padrão, o Django vem com vários aplicativos próprios pré-instalados que irão lidar com coisas como autenticação para você. Os aplicativos podem ser usados ​​em vários projetos Django.

O Django ORM coleta todos os modelos de banco de dados definidos em models.py e cria tabelas de banco de dados com base nessas classes de modelo. Para fazer isso, primeiro configure seu banco de dados modificando a configuração DATABASES em settings.py. Então, uma vez que você tenha definido seus modelos de banco de dados, execute python manage.py makemigrations seguido por python manage.py migrate para criar ou atualizar o esquema do seu banco de dados baseado em seus modelos.

Exemplo de arquivo único Hello World #

Este exemplo mostra uma maneira mínima de criar uma página Hello World no Django. Isso o ajudará a perceber que o comando django-admin startproject example basicamente cria um monte de pastas e arquivos e que você não precisa necessariamente dessa estrutura para executar seu projeto.

  1. Crie um arquivo chamado file.py.

  2. Copie e cole o código a seguir nesse arquivo.

     import sys

 from django.conf import settings

 settings.configure(
     DEBUG=True,
     SECRET_KEY='thisisthesecretkey',
     ROOT_URLCONF=__name__,
     MIDDLEWARE_CLASSES=(
         'django.middleware.common.CommonMiddleware',
         'django.middleware.csrf.CsrfViewMiddleware',
         'django.middleware.clickjacking.XFrameOptionsMiddleware',
     ),
 )

 from django.conf.urls import url
 from django.http import HttpResponse

 # Your code goes below this line.

 def index(request):
     return HttpResponse('Hello, World!')

 urlpatterns = [
     url(r'^$', index),
 ]

 # Your code goes above this line

 if __name__ == "__main__":
     from django.core.management import execute_from_command_line

     execute_from_command_line(sys.argv)

  3. Vá para o terminal e execute o arquivo com este comando python file.py runserver.

  4. Abra seu navegador e vá para 127.0.0.1:8000.

Um exemplo completo de olá mundo. #

Passo 1 Se você já tem o Django instalado, você pode pular esta etapa.

pip install Django

Passo 2 Criar um novo projeto

django-admin startproject hello

Isso criará uma pasta chamada hello que conterá os seguintes arquivos:

hello/
├── hello/
│   ├── __init__.py
│   ├── settings.py
│   ├── urls.py
│   └── wsgi.py
└── manage.py

Etapa 3 Dentro do módulo hello (a pasta que contém o __init.py__) crie um arquivo chamado views.py:

hello/
├── hello/
│   ├── __init__.py
│   ├── settings.py
│   ├── urls.py
│   ├── views.py  <- here
│   └── wsgi.py
└── manage.py

e coloque o seguinte conteúdo:

from django.http import HttpResponse

def hello(request):
    return HttpResponse('Hello, World')

Isso é chamado de função de visualização.

Passo 4 Edite hello/urls.py da seguinte forma:

from django.conf.urls import url
from django.contrib import admin
from hello import views

urlpatterns = [
    url(r'^admin/', admin.site.urls),
    url(r'^$', views.hello)
]

que liga a função de visualização hello() a uma URL.

Passo 5 Inicie o servidor.

python manage.py runserver

Passo 6

Navegue até http://localhost:8000/ em um navegador e você verá:

Olá, mundo

Projeto amigável para implantação com suporte ao Docker. #

O modelo de projeto padrão do Django é bom, mas uma vez que você implanta seu código e, por exemplo, os devops colocam as mãos no projeto, as coisas ficam confusas. O que você pode fazer é separar seu código-fonte do restante que precisa estar em seu repositório.

Você pode encontrar um modelo de projeto Django utilizável no GitHub.

Estrutura do Projeto #

PROJECT_ROOT
├── devel.dockerfile
├── docker-compose.yml
├── nginx
│   └── project_name.conf
├── README.md
├── setup.py
└── src
    ├── manage.py
    └── project_name
        ├── __init__.py
        └── service
            ├── __init__.py
            ├── settings
            │   ├── common.py
            │   ├── development.py
            │   ├── __init__.py
            │   └── staging.py
            ├── urls.py
            └── wsgi.py

Eu gosto de manter o diretório service chamado service para cada projeto, graças a isso eu posso usar o mesmo Dockerfile em todos os meus projetos. A divisão de requisitos e configurações já está bem documentada aqui: https://www.wikiod.com/pt/django/definicoes#Usando vários arquivos de requisitos https://www.wikiod.com/pt/django/definicoes#Usando várias configurações

Dockerfile #

Com a suposição de que apenas os desenvolvedores fazem uso do Docker (nem todos os dev ops confiam nele hoje em dia). Este poderia ser um ambiente de desenvolvimento devel.dockerfile:

FROM python:2.7
ENV PYTHONUNBUFFERED 1

RUN mkdir /run/service
ADD . /run/service
WORKDIR /run/service

RUN pip install -U pip
RUN pip install -I -e .[develop] --process-dependency-links

WORKDIR /run/service/src
ENTRYPOINT ["python", "manage.py"]
CMD ["runserver", "0.0.0.0:8000"]

Adicionar apenas requisitos aproveitará o cache do Docker durante a compilação - você só precisa reconstruir a alteração dos requisitos.

Compor #

A composição do Docker é útil - especialmente quando você tem vários serviços para executar localmente. docker-compose.yml:

version: '2'
services:
  web:
    build:
      context: .
      dockerfile: devel.dockerfile
    volumes:
      - "./src/{{ project_name }}:/run/service/src/{{ project_name }}"
      - "./media:/run/service/media"
    ports:
      - "8000:8000"
    depends_on:
      - db
  db:
    image: mysql:5.6
    environment:
      - MYSQL_ROOT_PASSWORD=root
      - MYSQL_DATABASE={{ project_name }}
  nginx:
    image: nginx
    ports:
      - "80:80"
    volumes:
      - "./nginx:/etc/nginx/conf.d"
      - "./media:/var/media"
    depends_on:
      - web

Nginx #

Seu ambiente de desenvolvimento deve estar o mais próximo possível do ambiente de produção, então eu gosto de usar o Nginx desde o início. Aqui está um exemplo de arquivo de configuração do nginx:

server {
    listen   80;
    client_max_body_size 4G;
    keepalive_timeout 5;

    location /media/ {
        autoindex on;
        alias /var/media/;
    }

    location / {
        proxy_pass_header Server;
        proxy_set_header Host $http_host;
        proxy_redirect off;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Scheme $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Ssl on;
        proxy_connect_timeout 600;
        proxy_read_timeout 600;
        proxy_pass http://web:8000/;
    }
}

Uso #

$ cd PROJECT_ROOT
$ docker-compose build web  # build the image - first-time and after requirements change
$ docker-compose up  # to run the project
$ docker-compose run --rm --service-ports --no-deps  # to run the project - and be able to use PDB
$ docker-compose run --rm --no-deps <management_command>  # to use other than runserver commands, like makemigrations
$ docker exec -ti web bash  # For accessing django container shell, using it you will be inside /run/service directory, where you can run ./manage shell, or other stuff
$ docker-compose start  # Starting docker containers
$ docker-compose stop  # Stopping docker containers
