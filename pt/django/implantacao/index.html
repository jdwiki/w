Implantação remota simples fabfile.py #

Fabric é uma biblioteca Python (2.5-2.7) e uma ferramenta de linha de comando para simplificar o uso de SSH para implantação de aplicativos ou tarefas de administração de sistemas. Ele permite que você execute funções Python arbitrárias por meio da linha de comando.

Instale o tecido via pip install fabric Crie fabfile.py em seu diretório raiz:

#myproject/fabfile.py    
from fabric.api import *

@task
def dev():
    # details of development server
    env.user = # your ssh user
    env.password = #your ssh password
    env.hosts = # your ssh hosts (list instance, with comma-separated hosts)
    env.key_filename = # pass to ssh key for github in your local keyfile

@task
def release():
    # details of release server
    env.user = # your ssh user
    env.password = #your ssh password
    env.hosts = # your ssh hosts (list instance, with comma-separated hosts)
    env.key_filename = # pass to ssh key for github in your local keyfile

@task
def run():
    with cd('path/to/your_project/'):
        with prefix('source ../env/bin/activate'): 
        # activate venv, suppose it appear in one level higher
            # pass commands one by one
            run('git pull')
            run('pip install -r requirements.txt')
            run('python manage.py migrate --noinput')
            run('python manage.py collectstatic --noinput')
            run('touch reload.txt')

Para executar o arquivo, basta usar o comando fab:

$ fab dev run  # for release server, `fab release run`

Nota: você não pode configurar chaves ssh para o github e apenas digitar login e senha manualmente, enquanto o fabfile é executado, o mesmo com as chaves.

Executando o aplicativo Django com Gunicorn #

  1. Instale o gunicorn

    pip install gunicorn

  2. Na pasta do projeto django (mesma pasta onde o manage.py reside), execute o seguinte comando para executar o projeto django atual com gunicorn

    gunicorn [projectname].wsgi:application -b 127.0.0.1:[port number]

    You can use the --env option to set the path to load the settings

    gunicorn --env DJANGO_SETTINGS_MODULE=[projectname].settings [projectname].wsgi

    or run as daemon process using -D option

  3. Após o início bem-sucedido do gunicorn, as seguintes linhas aparecerão no console

    Starting gunicorn 19.5.0

    Listening at: http://127.0.0.1:[port number] ([pid])

    .... (other additional information about gunicorn server)

Usando o modelo inicial do Heroku Django. #

Se você planeja hospedar seu site Django no Heroku, você pode iniciar seu projeto usando o Heroku Django Starter Template:

django-admin.py startproject --template=https://github.com/heroku/heroku-django-template/archive/master.zip --name=Procfile YourProjectName

Ele tem configuração pronta para produção para arquivos estáticos, configurações de banco de dados, Gunicorn, etc e aprimoramentos para a funcionalidade de serviço de arquivo estático do Django via WhiteNoise. Isso economizará seu tempo, está tudo pronto para hospedagem no Heroku, basta criar seu site no topo deste modelo

Para implantar este modelo no Heroku:

git init
git add -A
git commit -m "Initial commit"

heroku create
git push heroku master

heroku run python manage.py migrate

É isso!

Instruções de implantação do Django. Nginx + Gunicorn + Supervisor no Linux (Ubuntu) #

Três ferramentas básicas.

  1. nginx - servidor HTTP gratuito, de código aberto, de alto desempenho e proxy reverso, com alto desempenho;
  2. gunicorn - ‘Green Unicorn’ é um servidor HTTP Python WSGI para UNIX (necessário para gerenciar seu servidor);
  3. supervisor - um sistema cliente/servidor que permite que seus usuários monitorem e controlem vários processos em sistemas operacionais do tipo UNIX. Usado quando seu aplicativo ou sistema trava, reinicia seu django / aipo / aipo cam, etc;

Para simplificar, vamos supor que seu aplicativo esteja localizado neste diretório: /home/root/app/src/ e vamos usar o usuário root (mas você deve criar um usuário separado para seu aplicativo). Além disso, nosso ambiente virtual estará localizado no caminho /home/root/app/env/.

NGINX #

Vamos começar com o nginx. Se o nginx ainda não estiver na máquina, instale-o com sudo apt-get install nginx. Mais tarde você tem que criar um novo arquivo de configuração em seu diretório nginx /etc/nginx/sites-enabled/yourapp.conf. Se houver um arquivo chamado default.conf - remova-o.

Abaixo o código para um arquivo conf do nginx, que tentará executar seu serviço usando o arquivo socket; Mais tarde haverá uma configuração de gunicorn. O arquivo de soquete é usado aqui para se comunicar entre o nginx e o gunicorn. Também pode ser feito com o uso de portas.

# your application name; can be whatever you want
upstream yourappname {
    server        unix:/home/root/app/src/gunicorn.sock fail_timeout=0;
}

server {
    # root folder of your application
    root        /home/root/app/src/;
 
    listen        80;
    # server name, your main domain, all subdomains and specific subdomains
    server_name   yourdomain.com *.yourdomain.com somesubdomain.yourdomain.com
 
    charset       utf-8;

    client_max_body_size                        100m;
 
    # place where logs will be stored;
    # folder and files have to be already located there, nginx will not create
    access_log        /home/root/app/src/logs/nginx-access.log; 
    error_log         /home/root/app/src/logs/nginx-error.log;
    
    # this is where your app is served (gunicorn upstream above)
    location / {
        uwsgi_pass  yourappname;
        include     uwsgi_params; 
    }

    # static files folder, I assume they will be used
    location /static/ {
        alias         /home/root/app/src/static/;
    }
 
    # media files folder
    location /media/ {
        alias         /home/root/app/src/media/;
    }

}

GUNICÓRNIO #

Agora nosso script GUNICORN, que será responsável por rodar a aplicação django no servidor. A primeira coisa é instalar o gunicorn no ambiente virtual com pip install gunicorn.

#!/bin/bash

ME="root"
DJANGODIR=/home/root/app/src # django app dir
SOCKFILE=/home/root/app/src/gunicorn.sock # your sock file - do not create it manually
USER=root
GROUP=webapps
NUM_WORKERS=3 
DJANGO_SETTINGS_MODULE=yourapp.yoursettings
DJANGO_WSGI_MODULE=yourapp.wsgi  
echo "Starting $NAME as `whoami`"
 
# Activate the virtual environment
cd $DJANGODIR

source /home/root/app/env/bin/activate
export DJANGO_SETTINGS_MODULE=$DJANGO_SETTINGS_MODULE
export PYTHONPATH=$DJANGODIR:$PYTHONPATH
 
# Create the run directory if it doesn't exist
RUNDIR=$(dirname $SOCKFILE)
test -d $RUNDIR || mkdir -p $RUNDIR
 
# Start your Django Gunicorn
# Programs meant to be run under supervisor should not daemonize themselves (do not use --daemon)
exec /home/root/app/env/bin/gunicorn ${DJANGO_WSGI_MODULE}:application \
  --name root \
  --workers $NUM_WORKERS \
  --user=$USER --group=$GROUP \
  --bind=unix:$SOCKFILE \
  --log-level=debug \
  --log-file=-

para poder executar o script de início do gunicorn, ele deve ter o modo de execução ativado para

sudo chmod u+x /home/root/app/src/gunicorn_start

agora você poderá iniciar seu servidor gunicorn usando apenas ./gunicorn_start

SUPERVISOR #

Como dito no início, queremos que nosso aplicativo seja reiniciado quando falhar por um supervisor. Se o supervisor ainda não estiver no servidor, instale com sudo apt-get install supervisor.

No primeiro supervisor de instalação. Em seguida, crie um arquivo .conf em seu diretório principal /etc/supervisor/conf.d/your_conf_file.conf

conteúdo do arquivo de configuração:

[program:yourappname]
command = /home/root/app/src/gunicorn_start
user = root
stdout_logfile = /home/root/app/src/logs/gunicorn_supervisor.log
redirect_stderr = true

Breve breve, [program:youappname] é necessário no início, será nosso identificador. também stdout_logfile é um arquivo onde serão armazenados os logs, tanto de acessos quanto de erros.

Feito isso, temos que informar ao nosso supervisor que acabamos de adicionar um novo arquivo de configuração. Para fazer isso, há um processo diferente para diferentes versões do Ubuntu.

Para Ubuntu versão 14.04 ou inferior, basta executar esses comandos:

sudo supervisorctl reread -> relê todos os arquivos de configuração dentro do catálogo do supervisor isso deve imprimir: yourappname: available

sudo supervisorctl update -> atualiza o supervisor para arquivos de configuração recém-adicionados; deve imprimir yourappname: adicionado grupo de processos

Para Ubuntu 16.04 Execute:

sudo service supervisor restart

e para verificar se seu aplicativo está funcionando corretamente, basta executar

sudo supervisorctl status yourappname

Isso deve exibir:

yourappname RUNNING pid 18020, tempo de atividade 0:00:50

Implantando com Heroku #

  1. Baixe Heroku Toolbelt.
  2. Navegue até a raiz das fontes do seu aplicativo Django. vc vai precisar kk
  3. Digite heroku create [app_name]. Se você não fornecer um nome de aplicativo, o Heroku gerará um aleatoriamente para você. O URL do seu aplicativo será http://[nome do aplicativo].herokuapp.com
  4. Crie um arquivo de texto com o nome Procfile. Não coloque uma extensão no final.

web: <comando bash para iniciar o servidor de produção>

> Se você tiver um processo de trabalho, também poderá adicioná-lo. Adicione outra linha no formato: `worker-name: <bash command to start worker>`

5. Adicione um requirements.txt.
- Se você estiver usando um ambiente virtual, execute `pip freeze > requirements.txt`
- Caso contrário, [*obtenha um ambiente virtual!*][2]. Você também pode listar manualmente os pacotes Python necessários, mas isso não será abordado neste tutorial.

6. **É hora da implantação!**
 1. `git push heroku master`
 > Heroku needs a git repository or a dropbox folder to do deploys. You can alternatively set up automatic reloading from a GitHub repository at [`heroku.com`][3], but we won't cover that in this tutorial. <!-- Editors - you can help with this part! -->
 2. `heroku ps:scale web=1`
 > This scales the number of web "dynos" to one. You can learn more about dynos [here.][4]
 3. `heroku open` or navigate to `http://app-name.herokuapp.com`
 > **Tip:** `heroku open` opens the URL to your heroku app in the default browser.

7. Adicione **complementos**. Você precisará configurar seu aplicativo Django para vincular-se aos bancos de dados fornecidos no Heroku como "complementos". Este exemplo não cobre isso, mas outro exemplo está em andamento na implantação de bancos de dados no Heroku.

## Implantando localmente sem configurar o apache/nginx
A maneira recomendada de implantação de produção exige o uso do Apache/Nginx para servir o conteúdo estático. Assim, quando `DEBUG` é falso estático e o conteúdo da mídia não carrega. No entanto, podemos carregar o conteúdo estático na implantação sem precisar configurar o servidor Apache/Nginx para nosso aplicativo usando:

 python manage.py runserver --insecure

Isso é destinado apenas para implantação local (por exemplo, LAN) e nunca deve ser usado em produção e só está disponível se o aplicativo `staticfiles` estiver na configuração `INSTALLED_APPS` do seu projeto.
