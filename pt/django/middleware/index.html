Middleware no Django é um framework que permite que o código se conecte ao processamento de resposta/requisição e altere a entrada ou saída do Django.

O middleware precisa ser adicionado à sua lista settings.py MIDDLEWARE_CLASSES antes de ser incluído na execução. A lista padrão que o Django fornece ao criar um novo projeto é a seguinte:

MIDDLEWARE_CLASSES = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.auth.middleware.SessionAuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]

Essas são todas as funções que serão executadas em ordem em cada solicitação (uma vez antes de atingir seu código de visualização em views.py e uma vez em ordem inversa para o retorno de chamada process_response, antes da versão 1.10). Eles fazem uma variedade de coisas, como injetar o token Cross Site Request Forgery (csrf).

A ordem importa porque se algum middleware fizer um redirecionamento, todo o middleware subsequente nunca será executado. Ou se um middleware espera que o token csrf esteja lá, ele deve ser executado após o CsrfViewMiddleware.

Entendendo o novo estilo do middleware Django 1.10 #

O Django 1.10 introduziu um novo estilo de middleware onde process_request e process_response são mesclados.

Neste novo estilo, um middleware é um callable que retorna outro callable. Bem, na verdade o primeiro é uma fábrica de middleware e o último é o middleware real.

A fábrica de middleware toma como argumento único o próximo middleware na pilha de middlewares, ou a própria visão quando o fundo da pilha é atingido.

O middleware recebe a solicitação como argumento único e sempre retorna um HttpResponse.

O melhor exemplo para ilustrar como o middleware de novo estilo funciona é provavelmente mostrar como fazer um middleware compatível com versões anteriores:

class MyMiddleware:

    def __init__(self, next_layer=None):
        """We allow next_layer to be None because old-style middlewares
        won't accept any argument.
        """
        self.get_response = next_layer

    def process_request(self, request):
        """Let's handle old-style request processing here, as usual."""
        # Do something with request
        # Probably return None
        # Or return an HttpResponse in some cases

    def process_response(self, request, response):
        """Let's handle old-style response processing here, as usual."""
        # Do something with response, possibly using request.
        return response

    def __call__(self, request):
        """Handle new-style middleware here."""
        response = self.process_request(request)
        if response is None:
            # If process_request returned None, we must call the next middleware or
            # the view. Note that here, we are sure that self.get_response is not
            # None because this method is executed only in new-style middlewares.
            response = self.get_response(request)
        response = self.process_response(request, response)
        return response

Middleware para filtrar por endereço IP #

Primeiro: a estrutura do caminho

Caso não tenha, você precisa criar a pasta middleware dentro do seu aplicativo seguindo a estrutura:

yourproject/yourapp/middleware

O middleware da pasta deve ser colocado na mesma pasta que settings.py, urls, templates…

Importante: Não se esqueça de criar o arquivo vazio init.py dentro da pasta de middleware para que seu aplicativo reconheça esta pasta

Em vez de ter uma pasta separada contendo suas classes de middleware, também é possível colocar suas funções em um único arquivo, yourproject/yourapp/middleware.py.

Segundo: crie o middleware

Agora devemos criar um arquivo para nosso middleware personalizado. Neste exemplo vamos supor que queremos um middleware que filtre os usuários com base em seu endereço IP, criamos um arquivo chamado filter_ip_middleware.py:

#yourproject/yourapp/middleware/filter_ip_middleware.py
from django.core.exceptions import PermissionDenied    

class FilterIPMiddleware(object):
    # Check if client IP address is allowed
    def process_request(self, request):
        allowed_ips = ['192.168.1.1', '123.123.123.123', etc...] # Authorized ip's
        ip = request.META.get('REMOTE_ADDR') # Get client IP address
        if ip not in allowed_ips:
            raise PermissionDenied # If user is not allowed raise Error

       # If IP address is allowed we don't do anything
       return None

Terceiro: Adicione o middleware em nosso ‘settings.py’

Precisamos procurar o MIDDLEWARE_CLASSES dentro do settings.py e lá precisamos adicionar nosso middleware (Adicione-o na última posição). Deve ser como:

MIDDLEWARE_CLASSES = (
    'django.middleware.common.CommonMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
     # Above are Django standard middlewares

     # Now we add here our custom middleware
     'yourapp.middleware.filter_ip_middleware.FilterIPMiddleware'
)

Pronto! Agora, cada solicitação de cada cliente chamará seu middleware personalizado e processará seu código personalizado!

Adicionar dados às solicitações #

O Django torna muito fácil adicionar dados adicionais em solicitações para uso dentro da visualização. Por exemplo, podemos analisar o subdomínio no META da solicitação e anexá-lo como uma propriedade separada na solicitação usando middleware.

class SubdomainMiddleware:
    def process_request(self, request):
        """
        Parse out the subdomain from the request
        """
        host = request.META.get('HTTP_HOST', '')
        host_s = host.replace('www.', '').split('.')
        request.subdomain = None
        if len(host_s) > 2:
            request.subdomain = host_s[0]

Se você adicionar dados com middleware à sua solicitação, poderá acessar esses dados recém-adicionados mais adiante. Aqui, usaremos o subdomínio analisado para determinar algo como qual organização está acessando seu aplicativo. Essa abordagem é útil para aplicativos implantados com uma configuração de DNS com subdomínios curinga que apontam para uma única instância e a pessoa que acessa o aplicativo deseja uma versão com capa dependente do ponto de acesso.

class OrganizationMiddleware:
    def process_request(self, request):
        """
        Determine the organization based on the subdomain
        """
        try:
            request.org = Organization.objects.get(domain=request.subdomain)
        except Organization.DoesNotExist:
            request.org = None

Lembre-se de que a ordem importa quando o middleware depende um do outro. Para solicitações, você desejará que o middleware dependente seja colocado após a dependência.

MIDDLEWARE_CLASSES = [
    ...
    'myapp.middleware.SubdomainMiddleware',
    'myapp.middleware.OrganizationMiddleware',
    ...
]

Tratamento global de exceção #

Digamos que você tenha implementado alguma lógica para detectar tentativas de modificar um objeto no banco de dados enquanto o cliente que enviou as alterações não tinha as modificações mais recentes. Se tal caso acontecer, você gera uma exceção customizada ConfictError(detailed_message).

Agora você deseja retornar um código de status HTTP 409 (Confict) quando esse erro ocorrer. Você normalmente pode usar como middleware para isso em vez de manipulá-lo em cada exibição que pode gerar essa exceção.

class ConfictErrorHandlingMiddleware:
    def process_exception(self, request, exception):
        if not isinstance(exception, ConflictError):
            return  # Propagate other exceptions, we only handle ConflictError
        context = dict(confict_details=str(exception))
        return TemplateResponse(request, '409.html', context, status=409)
