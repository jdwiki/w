Proteção de Cross Site Scripting (XSS) #

Os ataques XSS consistem em injetar código HTML (ou JS) em uma página. Consulte O que é script entre sites para obter mais informações.

Para evitar esse ataque, por padrão, o Django escapa de strings passadas por uma variável de template.

Dado o seguinte contexto:

context = {
    'class_name': 'large" style="font-size:4000px',
    'paragraph': (
        "<script type=\"text/javascript\">alert('hello world!');</script>"),
}

<p class="{{ class_name }}">{{ paragraph }}</p>
<!-- Will be rendered as: -->
<p class="large&quot; style=&quot;font-size: 4000px">&lt;script&gt;alert(&#39;hello world!&#39;);&lt;/script&gt;</p>

Se você tiver variáveis ​​contendo HTML que você confia e realmente deseja renderizar, você deve dizer explicitamente que é seguro:

<p class="{{ class_name|safe }}">{{ paragraph }}</p>
<!-- Will be rendered as: -->
<p class="large" style="font-size: 4000px">&lt;script&gt;alert(&#39;hello world!&#39;);&lt;/script&gt;</p>

Se você tiver um bloco contendo várias variáveis ​​que são todas seguras, você pode desabilitar localmente o escape automático:

{% autoescape off %}
<p class="{{ class_name }}">{{ paragraph }}</p>
{% endautoescape %}
<!-- Will be rendered as: -->
<p class="large" style="font-size: 4000px"><script>alert('hello world!');</script></p>

Você também pode marcar uma string como segura fora do modelo:

from django.utils.safestring import mark_safe

context = {
    'class_name': 'large" style="font-size:4000px',
    'paragraph': mark_safe(
        "<script type=\"text/javascript\">alert('hello world!');</script>"),
}

<p class="{{ class_name }}">{{ paragraph }}</p>
<!-- Will be rendered as: -->
<p class="large&quot; style=&quot;font-size: 4000px"><script>alert('hello world!');</script></p>

Alguns utilitários do Django como format_html já retornam strings marcadas como seguras:

from django.utils.html import format_html

context = {
    'var': format_html('<b>{}</b> {}', 'hello', '<i>world!</i>'),
}

<p>{{ var }}</p>
<!-- Will be rendered as -->
<p><b>hello</b> &lt;i&gt;world!&lt;/i&gt;</p>

Proteção contra clickjacking #

Clickjacking é uma técnica maliciosa de induzir um usuário da Web a clicar em algo diferente do que o usuário percebe que está clicando. Saiba mais

Para habilitar a proteção contra clickjacking, adicione o XFrameOptionsMiddleware às suas classes de middleware. Isso já deveria estar lá se você não o removeu.

# settings.py
MIDDLEWARE_CLASSES = [
    ...
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
    ...
]

Este middleware define o cabeçalho ‘X-Frame-Options’ para todas as suas respostas, a menos que explicitamente isento ou já definido (não substituído se já estiver definido na resposta). Por padrão, é definido como “SAMEORIGIN”. Para alterar isso, use a configuração X_FRAME_OPTIONS:

X_FRAME_OPTIONS = 'DENY'

Você pode substituir o comportamento padrão por visualização.

from django.utils.decorators import method_decorator
from django.views.decorators.clickjacking import (
    xframe_options_exempt, xframe_options_deny, xframe_options_sameorigin,
)

xframe_options_exempt_m = method_decorator(xframe_options_exempt, name='dispatch')

@xframe_options_sameorigin
def my_view(request, *args, **kwargs):
    """Forces 'X-Frame-Options: SAMEORIGIN'."""
    return HttpResponse(...)

@method_decorator(xframe_options_deny, name='dispatch')
class MyView(View):
    """Forces 'X-Frame-Options: DENY'."""

@xframe_options_exempt_m
class MyView(View):
    """Does not set 'X-Frame-Options' header when passing through the
    XFrameOptionsMiddleware.
    """

Proteção contra falsificação de solicitação entre sites (CSRF) #

Falsificação de solicitação entre sites, também conhecida como ataque com um clique ou sessão montando e abreviado como CSRF ou XSRF, é um tipo de exploração maliciosa de um site onde comandos não autorizados são transmitidos de um usuário que o site confia. Saiba mais

Para habilitar a proteção CSRF, adicione o CsrfViewMiddleware às suas classes de middleware. Este middleware é habilitado por padrão.

# settings.py
MIDDLEWARE_CLASSES = [
    ...
    'django.middleware.csrf.CsrfViewMiddleware',
    ...
]

Esse middleware definirá um token em um cookie na resposta de saída. Sempre que uma solicitação recebida usa um método não seguro (qualquer método exceto GET, HEAD, OPTIONS e TRACE), o cookie deve corresponder a um token que é enviado como os dados do formulário csrfmiddlewaretoken ou como o X -CsrfToken. Isso garante que o cliente que inicia a solicitação também seja o proprietário do cookie e, por extensão, da sessão (autenticada).

Se uma solicitação for feita por HTTPS, a verificação estrita do referenciador será habilitada. Se o cabeçalho HTTP_REFERER não corresponder ao host da solicitação atual ou a um host em CSRF_TRUSTED_ORIGINS ([novo em 1.9](https://docs.djangoproject.com/en/stable/ref/settings/#std: setting-CSRF_TRUSTED_ORIGINS)), a solicitação é negada.

Os formulários que usam o método POST devem incluir o token CSRF no modelo. A tag de modelo {% csrf_token %} gerará um campo oculto e garantirá que o cookie seja definido na resposta:

<form method='POST'>
{% csrf_token %}
...
</form>

Visualizações individuais que não são vulneráveis ​​a ataques CSRF podem ser isentas usando o decorador @csrf_exempt:

from django.views.decorators.csrf import csrf_exempt

@csrf_exempt
def my_view(request, *args, **kwargs):
    """Allows unsafe methods without CSRF protection"""
    return HttpResponse(...)

Embora não seja recomendado, você pode desabilitar o CsrfViewMiddleware se muitas de suas visualizações não forem vulneráveis ​​a ataques CSRF. Nesse caso, você pode usar o decorador @csrf_protect para proteger visualizações individuais:

from django.views.decorators.csrf import csrf_protect

@csrf_protect
def my_view(request, *args, **kwargs):
    """This view is protected against CSRF attacks if the middleware is disabled"""
    return HttpResponse(...)
