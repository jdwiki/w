Escondendo dados secretos usando um arquivo JSON #

Ao usar um VCS como Git ou SVN, existem alguns dados secretos que nunca devem ser versionados (seja o repositório público ou privado).

Entre esses dados, você encontra a configuração SECRET_KEY e a senha do banco de dados.

Uma prática comum para ocultar essas configurações do controle de versão é criar um arquivo secrets.json na raiz do seu projeto (obrigado “Two Scoops of Django” pela ideia):

{
    "SECRET_KEY": "N4HE:AMk:.Ader5354DR453TH8SHTQr",
    "DB_PASSWORD": "v3ry53cr3t"
}

E adicione-o à sua lista de ignorados (.gitignore para git):

*.py[co]
*.sw[po]
*~
/secrets.json

Em seguida, adicione a seguinte função ao seu módulo settings:

import json
import os
from django.core.exceptions import ImproperlyConfigured

with open(os.path.join(BASE_DIR, 'secrets.json')) as secrets_file:
    secrets = json.load(secrets_file)

def get_secret(setting, secrets=secrets):
    """Get secret setting or fail with ImproperlyConfigured"""
    try:
        return secrets[setting]
    except KeyError:
        raise ImproperlyConfigured("Set the {} setting".format(setting))

Em seguida, preencha as configurações desta forma:

SECRET_KEY = get_secret('SECRET_KEY')
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgres',
        'NAME': 'db_name',
        'USER': 'username',
        'PASSWORD': get_secret('DB_PASSWORD'),
    },
}

Créditos: Two Scoops of Django: Best Practices for Django 1.8, por Daniel Roy Greenfeld e Audrey RoyGreenfeld. Copyright 2015 Two Scoops Press (ISBN 978-0981467344)

Usando variáveis ​​de ambiente para gerenciar configurações entre servidores #

O uso de variáveis ​​de ambiente é uma maneira amplamente usada para definir a configuração de um aplicativo dependendo do ambiente, conforme declarado em The Twelve-Factor App.

Como as configurações provavelmente mudam entre os ambientes de implantação, essa é uma maneira muito interessante de modificar a configuração sem precisar cavar o código-fonte do aplicativo, além de manter segredos fora dos arquivos do aplicativo e do repositório de código-fonte.

No Django, as configurações principais estão localizadas como settings.py na pasta do seu projeto. Como é um arquivo Python simples, você pode usar o módulo os do Python da biblioteca padrão para acessar o ambiente (e até ter padrões apropriados).

configurações.py #

import os

SECRET_KEY = os.environ.get('APP_SECRET_KEY', 'unsafe-secret-key')

DEBUG = bool(os.environ.get('DJANGO_DEBUG', True) == 'False')

ALLOWED_HOSTS = os.environ.get('DJANGO_ALLOWED_HOSTS', '').split()

DATABASES = {
    'default': {
        'ENGINE': os.environ.get('APP_DB_ENGINE', 'django.db.backends.sqlite3'),
        'NAME': os.environ.get('DB_NAME', 'db.sqlite'),    
        'USER': os.environ.get('DB_USER', ''),
        'PASSWORD': os.environ.get('DB_PASSWORD', ''),
        'HOST': os.environ.get('DB_HOST', None),
        'PORT': os.environ.get('DB_PORT', None),
        'CONN_MAX_AGE': 600,
    }
}

Com o Django, você pode alterar sua tecnologia de banco de dados, para que possa usar sqlite3 em sua máquina de desenvolvimento (e esse deve ser um padrão sensato para se comprometer com um sistema de controle de origem). Embora isso seja possível, não é aconselhável:

Serviços de apoio, como banco de dados do aplicativo, sistema de enfileiramento ou cache, é uma área em que a paridade dev/prod é importante. (Aplicativo de Doze Fatores - Paridade Dev/Prod)

Para usar um parâmetro DATABASE_URL para conexão com o banco de dados, dê uma olhada no exemplo relacionado.

Usando BASE_DIR para garantir a portabilidade do aplicativo #

É uma má ideia codificar caminhos em seu aplicativo. Deve-se sempre usar URLs relativos para que seu código possa funcionar perfeitamente em diferentes máquinas. A melhor maneira de configurar isso é definir uma variável como esta

import os
BASE_DIR = os.path.dirname(os.path.dirname(__file__))

Então use esta variável BASE_DIR para definir todas as suas outras configurações.

TEMPLATE_PATH = os.path.join(BASE_DIR, "templates")
STATICFILES_DIRS = [
    os.path.join(BASE_DIR, "static"),

]

E assim por diante. Isso garante que você possa portar seu código em diferentes máquinas sem nenhuma preocupação.

No entanto, os.path é um pouco verboso. Por exemplo, se seu módulo de configurações for project.settings.dev, você terá que escrever:

BASE_DIR = os.path.dirname(os.path.dirname(os.path.dirname(__file__)))

Uma alternativa é usar o módulo unipath (que você pode instalar com pip install unipath).

from unipath import Path

BASE_DIR = Path(__file__).ancestor(2)  # or ancestor(3) if using a submodule

TEMPLATE_PATH = BASE_DIR.child('templates')
STATICFILES_DIRS = [
    BASE_DIR.child('static'),
]

Usando várias configurações #

O layout de projeto padrão do Django cria um único settings.py. Isso geralmente é útil para dividi-lo assim:

myprojectroot/
    myproject/
        __init__.py
        settings/
            __init__.py
            base.py
            dev.py
            prod.py
            tests.py

Isso permite que você trabalhe com diferentes configurações de acordo com se você está em desenvolvimento, produção, testes ou qualquer outra coisa.

Ao passar do layout padrão para este layout, o settings.py original se torna settings/base.py. Quando todos os outros submódulos irão “subclassificar” settings/base.py começando com from .base import *. Por exemplo, aqui está a aparência de settings/dev.py:

# -*- coding: utf-8 -*-
from .base import *  # noqa

DEBUG = True
INSTALLED_APPS.extend([
    'debug_toolbar',
])
EMAIL_BACKEND = 'django.core.mail.backends.console.EmailBackend'
INTERNAL_IPS = ['192.168.0.51', '192.168.0.69']

Alternativa #1 #

Para que os comandos django-admin funcionem corretamente, você terá que definir a variável de ambiente DJANGO_SETTINGS_MODULE (que tem como padrão myproject.settings). No desenvolvimento, você o definirá como myproject.settings.dev. Na produção, você o definirá como myproject.settings.prod. Se você usa um virtualenv, o melhor é configurá-lo em seu script postactivate:

#!/bin/sh
export PYTHONPATH="/home/me/django_projects/myproject:$VIRTUAL_ENV/lib/python3.4"
export DJANGO_SETTINGS_MODULE="myproject.settings.dev"

Se você quiser usar um módulo de configurações que não é apontado por DJANGO_SETTINGS_MODULE por uma vez, você pode usar a opção --settings do django-admin:

django-admin test --settings=myproject.settings.tests

#Alternativa #2

Se você quiser deixar DJANGO_SETTINGS_MODULE em sua configuração padrão (myproject.settings), você pode simplesmente dizer ao módulo settings qual configuração carregar, colocando a importação em seu arquivo __init__.py.

No exemplo acima, o mesmo resultado pode ser obtido com um __init__.py definido como:

from .dev import *

Usando vários arquivos de requisitos #

Cada arquivo de requisitos deve corresponder ao nome de um arquivo de configurações. Leia Usando várias configurações para obter mais informações.

Estrutura #

djangoproject
├── config
│   ├── __init__.py
│   ├── requirements
│   │   ├── base.txt
│   │   ├── dev.txt
│   │   ├── test.txt
│   │   └── prod.txt
│   └── settings
└── manage.py

No arquivo base.txt, coloque as dependências usadas em todos os ambientes.

# base.txt
Django==1.8.0
psycopg2==2.6.1
jinja2==2.8

E em todos os outros arquivos, inclua dependências de base com -r base.txt e adicione dependências específicas necessárias para o ambiente atual.

# dev.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in dev env
django-queryinspect==0.1.0

# test.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in test env
nose==1.3.7
django-nose==1.4

# prod.txt
-r base.txt # includes all dependencies in `base.txt`

# specific dependencies only used in production env
django-queryinspect==0.1.0
gunicorn==19.3.0
django-storages-redux==1.3
boto==2.38.0

Finalmente, para instalar as dependências. Exemplo, em dev env : pip install -r config/requirements/dev.txt

Configurando o fuso horário #

Você pode definir o fuso horário que será usado pelo Django no arquivo settings.py. Exemplos:

TIME_ZONE = 'UTC'  # use this, whenever possible
TIME_ZONE = 'Europe/Berlin'
TIME_ZONE = 'Etc/GMT+1'

Aqui está a lista de fusos horários válidos

Ao executar em um ambiente Windows, isso deve ser definido como o mesmo fuso horário do sistema.

Se você não quiser que o Django use datas com reconhecimento de fuso horário:

USE_TZ = False

As melhores práticas do Django exigem o uso de UTC para armazenar informações no banco de dados:

Mesmo que seu site esteja disponível em apenas um fuso horário, ainda é boa prática para armazenar dados em UTC em seu banco de dados. A principal razão é o horário de verão (DST). Muitos países têm um sistema de horário de verão, onde os relógios são adiantados na primavera e atrasados ​​no outono. Se você está trabalhando no horário local, é provável que encontre erros duas vezes um ano, quando as transições acontecem.

https://docs.djangoproject.com/en/stable/topics/i18n/timezones/

Acessando as configurações #

Uma vez que você tenha todas as suas configurações, você vai querer usá-las em seu código. Para fazer isso, adicione a seguinte importação ao seu arquivo:

from django.conf import settings

Você pode então acessar suas configurações como atributos do módulo settings, por exemplo:

if not settings.DEBUG:
    email_user(user, message)

Usando um DATABASE_URL do ambiente #

Em sites PaaS como o Heroku, é comum receber as informações do banco de dados como uma única variável de ambiente de URL, em vez de vários parâmetros (host, porta, usuário, senha…).

Existe um módulo, dj_database_url que extrai automaticamente a variável de ambiente DATABASE_URL para um dicionário Python apropriado para injetar as configurações do banco de dados no Django.

Uso:

import dj_database_url

if os.environ.get('DATABASE_URL'):
    DATABASES['default'] =
        dj_database_url.config(default=os.environ['DATABASE_URL'])
