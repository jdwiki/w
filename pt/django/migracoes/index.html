comando django-admin Detalhes

fazer migrações Gere migrações para todos os aplicativos

makemigrations --merge <my_app> Resolver conflitos de migração para my_app

showmigrations Mostra todas as migrações para todos os aplicativos
showmigrations <my_app> Mostra todas as migrações em my_app

Migrações manuais #

Às vezes, as migrações geradas pelo Django não são suficientes. Isso é especialmente verdadeiro quando você deseja fazer migrações de dados.

Por exemplo, vamos ter esse modelo:

class Article(models.Model):
    title = models.CharField(max_length=70)

Este modelo já possui dados existentes e agora você deseja adicionar um SlugField:

class Article(models.Model):
    title = models.CharField(max_length=70)
    slug = models.SlugField(max_length=70)

Você criou as migrations para adicionar o campo, mas agora gostaria de definir o slug para todos os artigos existentes, de acordo com o title.

Claro, você poderia fazer algo assim no terminal:

$ django-admin shell
>>> from my_app.models import Article
>>> from django.utils.text import slugify
>>> for article in Article.objects.all():
...     article.slug = slugify(article.title)
...     article.save()
...
>>>

Mas você terá que fazer isso em todos os seus ambientes (ou seja, seu desktop de escritório, seu laptop, …), todos os seus colegas de trabalho também terão que fazer isso, e você terá que pensar sobre isso na preparação e ao empurrar viver.

Para torná-lo de uma vez por todas, vamos fazê-lo em uma migração. Primeiro crie uma migração vazia:

$ django-admin makemigrations --empty app_name

Isso criará um arquivo de migração vazio. Abra-o, ele contém um esqueleto básico. Digamos que sua migração anterior tenha sido chamada de 0023_article_slug e esta seja chamada de 0024_auto_20160719_1734. Aqui está o que vamos escrever em nosso arquivo de migração:

# -*- coding: utf-8 -*-
# Generated by Django 1.9.7 on 2016-07-19 15:34
from __future__ import unicode_literals

from django.db import migrations
from django.utils.text import slugify


def gen_slug(apps, schema_editor):
    # We can't import the Article model directly as it may be a newer
    # version than this migration expects. We use the historical version.
    Article = apps.get_model('app_name', 'Article')
    for row in Article.objects.all():
        row.slug = slugify(row.name)
        row.save()


class Migration(migrations.Migration):

    dependencies = [
        ('hosting', '0023_article_slug'),
    ]

    operations = [
        migrations.RunPython(gen_slug, reverse_code=migrations.RunPython.noop),
        # We set `reverse_code` to `noop` because we cannot revert the migration
        # to get it back in the previous state.
        # If `reverse_code` is not given, the migration will not be reversible,
        # which is not the behaviour we expect here.
    ]

Migrações falsas #

Quando uma migração é executada, o Django armazena o nome da migração em uma tabela django_migrations.

Criar e falsificar migrações iniciais para esquema existente

Se seu aplicativo já possui modelos e tabelas de banco de dados e não possui migrações. Primeiro, crie migrações iniciais para seu aplicativo.

python manage.py makemigrations your_app_label

Agora migrações iniciais falsas conforme aplicadas

python manage.py migrate --fake-initial

Fake todas as migrações em todos os aplicativos

python manage.py migrate --fake

Falsas migrações de aplicativos únicos

python manage.py migrate --fake core

Arquivo de migração único falso

python manage.py migrate myapp migration_name

Trabalhando com migrações #

O Django usa migrações para propagar as alterações que você faz em seus modelos para seu banco de dados. Na maioria das vezes, o django pode gerá-los para você.

Para criar uma migração, execute:

$ django-admin makemigrations <app_name>

Isso criará um arquivo de migração no submódulo migration de app_name. A primeira migração será chamada 0001_initial.py, a outra começará com 0002_, depois 0003, …

Se você omitir <app_name>, isso criará migrações para todos os seus INSTALLED_APPS.

Para propagar migrações para seu banco de dados, execute:

$ django-admin migrate <app_name>

Para mostrar todas as suas migrações, execute:

$ django-admin showmigrations app_name
app_name
  [X] 0001_initial
  [X] 0002_auto_20160115_1027
  [X] 0003_somemodel
  [ ] 0004_auto_20160323_1826
  • [X] significa que a migração foi propagada para seu banco de dados
  • [ ] significa que a migração não foi propagada para seu banco de dados. Use django-admin migrate para propagá-lo

Você também chama as migrações de reversão, isso pode ser feito passando o nome da migração para o comando de migração. Dada a lista de migrações acima (mostrada por django-admin showmigrations):

$ django-admin migrate app_name 0002  # Roll back to migration 0002
$ django-admin showmigrations app_name
app_name
  [X] 0001_initial
  [X] 0002_auto_20160115_1027
  [ ] 0003_somemodel
  [ ] 0004_auto_20160323_1826

Nomes personalizados para arquivos de migração #

Use a opção makemigrations --name <your_migration_name> para permitir nomear as migrações em vez de usar um nome gerado.

python manage.py makemigrations --name <your_migration_name> <app_name>

Resolvendo conflitos de migração #

Introdução #

Às vezes, as migrações entram em conflito, resultando em uma migração malsucedida. Isso pode acontecer em muitos cenários, mas pode ocorrer regularmente ao desenvolver um aplicativo com uma equipe.

Conflitos de migração comuns ocorrem ao usar o controle do código-fonte, especialmente quando o método de recurso por ramificação é usado. Para este cenário usaremos um modelo chamado Reporter com os atributos name e address.

Dois desenvolvedores neste momento vão desenvolver um recurso, então ambos recebem esta cópia inicial do modelo Reporter. O desenvolvedor A adiciona um age que resulta no arquivo 0002_reporter_age.py. O desenvolvedor B adiciona um campo bank_account que resulta em 0002_reporter_bank_account. Depois que esses desenvolvedores mesclam seus códigos e tentam migrar as migrações, ocorreu um conflito de migração.

Este conflito ocorre porque ambas as migrações alteram o mesmo modelo, Reporter. Além disso, os novos arquivos começam com 0002.

Mesclando migrações #

Existem várias maneiras de fazê-lo. O seguinte está na ordem recomendada:

  1. A correção mais simples para isso é executar o makemigrations command with a –merge flag.

     python manage.py makemigrations --merge <my_app>

    This will create a new migration solving the previous conflict.

  2. Quando este arquivo extra não é bem-vindo no ambiente de desenvolvimento for personal reasons, an option is to delete the conflicting migrations. Then, a new migration can be made using the regular makemigrations command. When custom migrations are written, such as migrations.RunPython, need to be accounted for using this method.

Altere um CharField para uma ForeignKey #

Primeiramente, vamos supor que este seja seu modelo inicial, dentro de uma aplicação chamada discography:

from django.db import models

class Album(models.Model):
    name = models.CharField(max_length=255)
    artist = models.CharField(max_length=255)

Agora, você percebe que deseja usar uma ForeignKey para o artista. Este é um processo um tanto complexo, que deve ser feito em várias etapas.

Etapa 1, adicione um novo campo para ForeignKey, certificando-se de marcá-lo como nulo (observe que o modelo ao qual estamos vinculando também está incluído):

from django.db import models

class Album(models.Model):
    name = models.CharField(max_length=255)
    artist = models.CharField(max_length=255)
    artist_link = models.ForeignKey('Artist', null=True)

class Artist(models.Model):
    name = models.CharField(max_length=255)

…e crie uma migração para essa mudança.

./manage.py makemigrations discography

Etapa 2, preencha seu novo campo. Para fazer isso, você precisa criar uma migração vazia.

./manage.py makemigrations --empty --name transfer_artists discography

Depois de ter essa migração vazia, você deseja adicionar uma única operação RunPython a ela para vincular seus registros. Neste caso, poderia ser algo assim:

def link_artists(apps, schema_editor):
    Album = apps.get_model('discography', 'Album')
    Artist = apps.get_model('discography', 'Artist')
    for album in Album.objects.all():
        artist, created = Artist.objects.get_or_create(name=album.artist)
        album.artist_link = artist
        album.save()

Agora que seus dados são transferidos para o novo campo, você pode realmente terminar e deixar tudo como está, usando o novo campo artist_link para tudo. Ou, se quiser fazer um pouco de limpeza, crie mais duas migrações.

Para sua primeira migração, você desejará excluir seu campo original, artist. Para sua segunda migração, renomeie o novo campo artist_link para artist.

Isso é feito em várias etapas para garantir que o Django reconheça as operações corretamente.
