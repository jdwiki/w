Impedir que métodos sensíveis sejam chamados em templates #

Quando um objeto é exposto ao contexto do modelo, seus métodos sem argumentos estão disponíveis. Isso é útil quando essas funções são “getters”. Mas pode ser perigoso se esses métodos alterarem alguns dados ou tiverem alguns efeitos colaterais. Mesmo que você provavelmente confie no criador do modelo, ele pode não estar ciente dos efeitos colaterais de uma função ou pensar em chamar o atributo errado por engano.

Dado o seguinte modelo:

class Foobar(models.Model):
    points_credit = models.IntegerField()

    def credit_points(self, nb_points=1):
        """Credit points and return the new points credit value."""
        self.points_credit = F('points_credit') + nb_points
        self.save(update_fields=['points_credit'])
        return self.points_credit

Se você escrever isso, por engano, em um modelo:

 You have {{ foobar.credit_points }} points!

Isso aumentará o número de pontos cada vez que o modelo for chamado. E talvez você nem perceba.

Para evitar isso, você deve definir o atributo alters_data como True para métodos que tenham efeitos colaterais. Isso tornará impossível chamá-los de um modelo.

def credit_points(self, nb_points=1):
    """Credit points and return the new points credit value."""
    self.points_credit = F('points_credit') + nb_points
    self.save(update_fields=['points_credit'])
    return self.points_credit
credit_points.alters_data = True

Modelagem em visualizações baseadas em função #

Você pode usar um modelo em uma exibição baseada em função da seguinte maneira:

from django.shortcuts import render

def view(request):
    return render(request, "template.html")

Se você quiser usar variáveis ​​de modelo, poderá fazê-lo da seguinte maneira:

from django.shortcuts import render

def view(request):
    context = {"var1": True, "var2": "foo"}
    return render(request, "template.html", context=context)

Então, em template.html, você pode consultar suas variáveis ​​assim:

<html>
{% if var1 %}
    <h1>{{ var2 }}</h1>
{% endif %}
</html>

Filtros de modelo #

O sistema de templates do Django tem tags e filtros embutidos, que são funções dentro do template para renderizar conteúdo de uma maneira específica. Vários filtros podem ser especificados com pipes e os filtros podem ter argumentos, assim como na sintaxe de variável.

{{ "MAINROAD 3222"|lower }}    # mainroad 3222
{{ 10|add:15}}                 # 25
{{ "super"|add:"glue" }}       # superglue
{{ "A7"|add:"00" }}            # A700
{{ myDate | date:"D d M Y"}}   # Wed 20 Jul 2016

Uma lista de filtros integrados disponíveis pode ser encontrada em https://docs.djangoproject.com/en/dev/ref/templates/builtins/#ref-templates-builtins-filters .

Criando filtros personalizados

Para adicionar seus próprios filtros de modelo, crie uma pasta chamada templatetags dentro da pasta do seu aplicativo. Em seguida adicione um __init__.py, e o arquivo seu arquivo que conterá os filtros:

#/myapp/templatetags/filters.py
from django import template

register = template.Library()

@register.filter(name='tostring')
def to_string(value):
    return str(value)

Para realmente usar o filtro, você precisa carregá-lo em seu modelo:

#templates/mytemplate.html
{% load filters %}
{% if customer_id|tostring = customer %} Welcome back {% endif%}

Truques

Mesmo que os filtros pareçam simples no começo, ele permite fazer algumas coisas bacanas:

{% for x in ""|ljust:"20" %}Hello World!{% endfor %}    # Hello World!Hello World!Hel...    
{{ user.name.split|join:"_" }} ## replaces whitespace with '_'

Consulte também tags de modelo para obter mais informações.

Modelagem em visualizações baseadas em classe #

Você pode passar dados para um modelo em uma variável personalizada.

Em seu views.py:

from django.views.generic import TemplateView
from MyProject.myapp.models import Item

class ItemView(TemplateView):
    template_name = "item.html"

    def items(self):
        """ Get all Items """
        return Item.objects.all()

    def certain_items(self):
        """ Get certain Items """
        return Item.objects.filter(model_field="certain")
    
    def categories(self):
        """ Get categories related to this Item """
        return Item.objects.get(slug=self.kwargs['slug']).categories.all()

Uma lista simples em seu item.html:

{% for item in view.items %}
<ul>        
    <li>{{ item }}</li>
</ul>
{% endfor %}

Você também pode recuperar propriedades adicionais dos dados.

Supondo que seu modelo Item tenha um campo name:

{% for item in view.certain_items %}
<ul>        
    <li>{{ item.name }}</li>
</ul>
{% endfor %}

Variáveis #

As variáveis ​​que você forneceu em seu contexto de visualização podem ser acessadas usando a notação de chave dupla:

Em seu views.py:

class UserView(TemplateView):
  """ Supply the request user object to the template """

  template_name = "user.html"

  def get_context_data(self, **kwargs):
    context = super(UserView, self).get_context_data(**kwargs)
    context.update(user=self.request.user)
    return context

Em user.html:

<h1>{{ user.username }}</h1>

<div class="email">{{ user.email }}</div>

A notação de ponto acessará:

  • propriedades do objeto, por exemplo. user.username será {{ user.username }}
  • pesquisas de dicionário, por exemplo request.GET["search"] será {{ request.GET.search }}
  • métodos sem argumentos, por exemplo users.count() será {{ user.count }}

As variáveis ​​de modelo não podem acessar métodos que recebem argumentos.

As variáveis ​​também podem ser testadas e colocadas em loop:

{% if user.is_authenticated %}
  {% for item in menu %}
    <li><a href="{{ item.url }}">{{ item.name }}</a></li>
  {% endfor %}
{% else %}
  <li><a href="{% url 'login' %}">Login</a>
{% endif %}

URLs são acessadas usando o formato {% url 'name' %}, onde os nomes correspondem aos nomes em seu urls.py.

{% url 'login' %} - Provavelmente será renderizado como /accounts/login/ {% url 'user_profile' user.id %} - Os argumentos para URLs são fornecidos em ordem {% url next %} - URLs podem ser variáveis

Uso de {% extends %} , {% include %} e {% blocks %} #

resumo #

  • {% extends %}: declara o template dado como argumento como pai do template atual. Uso: {% estende 'parent_template.html' %}.

  • {% block %}{% endblock %}: Isso é usado para definir seções em seus modelos, para que, se outro modelo estender este, ele possa substituir qualquer código html que tenha sido escrito dentro de isto. Os blocos são identificados pelo nome. Uso: {% block content %} <html_code> {% endblock %}.

  • {% include %}: isso irá inserir um modelo dentro do atual. Esteja ciente de que o modelo incluído receberá o contexto da solicitação e você também poderá fornecer variáveis ​​personalizadas. Uso básico: {% include 'template_name.html' %}, uso com variáveis: {% include 'template_name.html' com variável='valor' variável2=8 %}

Guia #

Suponha que você esteja construindo seu código de front-end com layouts comuns para todo o código e não queira repetir o código para cada modelo. O Django fornece tags construídas para isso. Suponha que tenhamos um site de blog com 3 modelos que compartilham o mesmo layout:

project_directory
    ..
    templates
      front-page.html
      blogs.html
      blog-detail.html

  1. Defina o arquivo base.html,

    {% block content %} {% endblock %}

  2. Estenda-o em blog.html como,

    {% extends ‘base.html’ %}

    {% block content %} # write your blog related code here {% endblock %}

    None of the code written here will be added to the template #

Aqui nós estendemos o layout base para que seu layout HTML esteja agora disponível no arquivo blog.html. elementos comuns do seu site e define os blocos que os modelos filho podem substituir.

3 ) Agora suponha que todos os seus 3 templates também tenham o mesmo div HTML que define alguns posts populares.

blog.html

{% extends 'base.html' %}

{% block content %}
    # write your blog related code here
    {% include 'posts.html' %} # includes posts.html in blog.html file without passing any data
    <!-- or -->
    {% include 'posts.html' with posts=postdata %} # includes posts.html in blog.html file with passing posts data which is context of view function returns.
{% endblock %}
