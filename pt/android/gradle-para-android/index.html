Gradle é um sistema de compilação baseado em JVM que permite aos desenvolvedores escrever scripts de alto nível que podem ser usados ​​para automatizar o processo de compilação e produção de aplicativos. É um sistema flexível baseado em plugins, que permite automatizar vários aspectos do processo de construção; incluindo compilar e assinar um .jar, baixar e gerenciar dependências externas, injetar campos no AndroidManifest ou utilizar versões específicas do SDK.

Sintaxe #

  • apply plugin: Os plugins que devem ser usados ​​normalmente apenas 'com.android.application' ou 'com.android.library'.

  • android: A configuração principal do seu aplicativo

    • compileSdkVersion: The compile SDK version
    • buildToolsVersion: The build tools version
    • defaultConfig: The default settings which can been overwritten by flavors and build types
      • applicationId: The application id you use e.g. in the PlayStore mostly the same as your package name
      • minSdkVersion: The minimal required SDK version
      • targetSdkVersion: The SDK version you compile against (should be always the newst one)
      • versionCode: The internal version number which needs to be bigger on each update
      • versionName: The version number the user can see in the app details page
    • buildTypes: See somewhere else (TODO)

  • dependencies: O maven ou dependências locais do seu aplicativo

  • compilar uma única dependência
  • testCompile: uma dependência para a unidade ou testes de integração

Um arquivo build.gradle básico #

Este é um exemplo de um arquivo build.gradle padrão em um módulo.

apply plugin: 'com.android.application'

android {
    compileSdkVersion 25
    buildToolsVersion '25.0.3'

    signingConfigs {
        applicationName {
            keyAlias 'applicationName'
            keyPassword 'password'
            storeFile file('../key/applicationName.jks')
            storePassword 'keystorePassword'
        }
    }
    defaultConfig {
        applicationId 'com.company.applicationName'
        minSdkVersion 14
        targetSdkVersion 25
        versionCode 1
        versionName '1.0'
        signingConfig signingConfigs.applicationName
    }
    buildTypes {
        release {
            minifyEnabled true
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
        }
    }
}

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])

    compile 'com.android.support:appcompat-v7:25.3.1'
    compile 'com.android.support:design:25.3.1'

    testCompile 'junit:junit:4.12'
}

DSL (linguagem específica do domínio) #

Cada bloco no arquivo acima é chamado de DSL (linguagem específica do domínio).

#Plugins

A primeira linha, apply plugin: 'com.android.application', aplica o Android plugin for Gradle à compilação e disponibiliza o bloco android {} para declarar opções de compilação específicas do Android.

Para um Aplicativo Android:

apply plugin: 'com.android.application'

Para uma Biblioteca do Android:

apply plugin: 'com.android.library'

Entendendo as DSLs no exemplo acima #

A segunda parte, o bloco android {...}, é o DSL do Android que contém informações sobre seu projeto.

Por exemplo, você pode definir o compileSdkVersion que especifica o nível da API do Android, que deve ser usado pelo Gradle para compilar seu aplicativo. O sub-bloco defaultConfig contém os padrões do seu manifesto. Você pode ‘substituí-los’ com [Sabores do Produto][3].

[Dependências][8] #

O bloco dependencies é definido fora do bloco android {...} : Isso significa que não é definido pelo plugin Android, mas é Gradle padrão. O bloco dependencies especifica quais bibliotecas externas (normalmente bibliotecas Android, mas bibliotecas Java também são válidas) que você deseja incluir em seu aplicativo. Gradle irá baixar automaticamente essas dependências para você (se não houver nenhuma cópia local disponível), você só precisa adicionar linhas compile semelhantes quando desejar adicionar outra biblioteca.

Vejamos uma das linhas aqui presentes:

compilar 'com.android.support:design:25.3.1'

Esta linha diz basicamente

adicionar uma dependência da biblioteca de design de suporte do Android ao meu projeto.

O Gradle garantirá que a biblioteca seja baixada e presente para que você possa usá-la em seu aplicativo, e seu código também será incluído em seu aplicativo.

Se você estiver familiarizado com o Maven, esta sintaxe é o GroupId, dois pontos, ArtifactId, outros dois pontos, então a versão da dependência que você deseja incluir, dando a você controle total sobre o controle de versão.

Embora seja possível especificar versões de artefatos usando o sinal de mais (+), a melhor prática é evitar fazê-lo; isso pode levar a problemas se a biblioteca for atualizada com alterações importantes sem o seu conhecimento, o que provavelmente levaria a falhas no seu aplicativo.

Você pode adicionar diferentes tipos de dependências:

  • [dependências binárias locais][9]
  • [dependências do módulo][10]
  • [dependências remotas][11]

Uma atenção especial deve ser dedicada às [dependências planas aar][12].

Você pode encontrar mais detalhes em [este tópico.][13]

Nota sobre o -v7 em appcompat-v7

compilar 'com.android.support:appcompat-v7:25.3.1'

Isso significa simplesmente que esta biblioteca (appcompat) é compatível com a API do Android nível 7 e posteriores.

Observação sobre o junit:junit:4.12

Esta é a dependência de teste para testes de unidade.

Especificando dependências específicas para diferentes configurações de compilação #

Você pode especificar que uma dependência deve ser usada apenas para uma determinada [configuração de compilação][14] ou pode definir dependências diferentes para os [tipos de compilação][15] ou [variações de produto][16] (por exemplo, depuração, teste ou release) usando debugCompile, testCompile ou releaseCompile em vez do compile usual.

Isso é útil para manter as dependências relacionadas a teste e depuração fora de sua compilação de lançamento, o que manterá seu ‘APK’ de lançamento o mais reduzido possível e ajudará a garantir que nenhuma informação de depuração possa ser usada para obter informações internas sobre seu aplicativo.

Definir e usar campos de configuração de compilação #

BuildConfigField #

Gradle permite que linhas buildConfigField definam constantes. Essas constantes estarão acessíveis em tempo de execução como campos estáticos da classe BuildConfig. Isso pode ser usado para criar flavors definindo todos os campos dentro do bloco defaultConfig e, em seguida, substituindo-os por variações de compilação individuais conforme necessário.

Este exemplo define a data de compilação e sinaliza a compilação para produção em vez de teste:

android {
    ...
    defaultConfig {
        ...
        // defining the build date
        buildConfigField "long", "BUILD_DATE", System.currentTimeMillis() + "L"
        // define whether this build is a production build
        buildConfigField "boolean", "IS_PRODUCTION", "false"
        // note that to define a string you need to escape it
        buildConfigField "String", "API_KEY", "\"my_api_key\""
    }

    productFlavors {
        prod {
            // override the productive flag for the flavor "prod"
            buildConfigField "boolean", "IS_PRODUCTION", "true"
            resValue 'string', 'app_name', 'My App Name'
        }
        dev {
            // inherit default fields
            resValue 'string', 'app_name', 'My App Name - Dev'
        }
    }
}

O <package_name>.BuildConfig.java gerado automaticamente na pasta gen contém os seguintes campos com base na diretiva acima:

public class BuildConfig {
    // ... other generated fields ...
    public static final long BUILD_DATE = 1469504547000L;
    public static final boolean IS_PRODUCTION = false;
    public static final String API_KEY = "my_api_key";
}

Os campos definidos agora podem ser usados ​​dentro do aplicativo em tempo de execução, acessando a classe BuildConfig gerada:

public void example() {
    // format the build date
    SimpleDateFormat dateFormat = new SimpleDateFormat("yyyy/MM/dd");
    String buildDate = dateFormat.format(new Date(BuildConfig.BUILD_DATE));
    Log.d("build date", buildDate);
    
    // do something depending whether this is a productive build
    if (BuildConfig.IS_PRODUCTION) {
        connectToProductionApiEndpoint();
    } else {
        connectToStagingApiEndpoint();
    }
}

ResValue #

O resValue no productFlavors cria um valor de recurso. Pode ser qualquer tipo de recurso (string, dimen, color, etc.). Isso é semelhante a definir um recurso no arquivo apropriado: por exemplo, definindo string em um arquivo strings.xml. A vantagem é que o definido em gradle pode ser modificado com base no seu productFlavor/buildVariant. Para acessar o valor, escreva o mesmo código como se estivesse acessando um res do arquivo de recursos:

getResources().getString(R.string.app_name)

O importante é que os recursos definidos dessa forma não podem modificar os recursos existentes definidos nos arquivos. Eles só podem criar novos valores de recursos.

Algumas bibliotecas (como a API Android do Google Maps) exigem uma chave de API fornecida no manifesto como uma tag metadados. Se forem necessárias chaves diferentes para compilações de depuração e produção, especifique um espaço reservado para manifesto preenchido pelo Gradle.

No seu arquivo AndroidManifest.xml:

<meta-data
    android:name="com.google.android.geo.API_KEY"
    android:value="${MAPS_API_KEY}"/>

E então defina o campo de acordo em seu arquivo build.gradle:

android {
    defaultConfig {
        ...
        // Your development key
        manifestPlaceholders = [ MAPS_API_KEY: "AIza..." ]
    }

    productFlavors {
        prod {
            // Your production key
            manifestPlaceholders = [ MAPS_API_KEY: "AIza..." ]
        }
    }
}

O sistema de compilação do Android gera vários campos automaticamente e os coloca em BuildConfig.java. Esses campos são:

Campo Descrição
DEBUG um Boolean informando se o aplicativo está em modo de depuração ou lançamento
APPLICATION_ID uma String contendo o ID do aplicativo (por exemplo, com.example.app)
BUILD_TYPE uma String contendo o tipo de compilação do aplicativo (geralmente debug ou release)
SABOR um String contendo o sabor particular da compilação
VERSION_CODE um int contendo o número da versão (build).
É o mesmo que versionCode em build.gradle ou versionCode em AndroidManifest.xml
VERSION_NAME uma String contendo o nome da versão (build).
É o mesmo que versionName em build.gradle ou versionName em AndroidManifest.xml

Além do acima, se você definiu várias dimensões de sabor, cada dimensão terá seu próprio valor. Por exemplo, se você tiver duas dimensões de sabor para color e size, também terá as seguintes variáveis:

Campo Descrição
FLAVOR_color um String contendo o valor para o sabor ‘color’.
FLAVOR_size uma String contendo o valor para o tipo ‘size’.

Centralizando dependências via arquivo “dependencies.gradle” #

Ao trabalhar com projetos de vários módulos, é útil centralizar as dependências em um único local, em vez de tê-las espalhadas por muitos arquivos de compilação, especialmente para bibliotecas comuns, como as bibliotecas de suporte do Android e as bibliotecas do Firebase.

Uma maneira recomendada é separar os arquivos de compilação do Gradle, com um build.gradle por módulo, além de um na raiz do projeto e outro para as dependências, por exemplo:

root
  +- gradleScript/
  |     dependencies.gradle
  +- module1/
  |     build.gradle
  +- module2/
  |     build.gradle
  +- build.gradle

Então, todas as suas dependências podem ser localizadas em gradleScript/dependencies.gradle:

ext {
    // Version
    supportVersion = '24.1.0'

    // Support Libraries dependencies
    supportDependencies = [
            design:            "com.android.support:design:${supportVersion}",
            recyclerView:      "com.android.support:recyclerview-v7:${supportVersion}",
            cardView:          "com.android.support:cardview-v7:${supportVersion}",
            appCompat:         "com.android.support:appcompat-v7:${supportVersion}",
            supportAnnotation: "com.android.support:support-annotations:${supportVersion}",
    ]

    firebaseVersion = '9.2.0';

    firebaseDependencies = [
            core:         "com.google.firebase:firebase-core:${firebaseVersion}",
            database:     "com.google.firebase:firebase-database:${firebaseVersion}",
            storage:      "com.google.firebase:firebase-storage:${firebaseVersion}",
            crash:        "com.google.firebase:firebase-crash:${firebaseVersion}",
            auth:         "com.google.firebase:firebase-auth:${firebaseVersion}",
            messaging:    "com.google.firebase:firebase-messaging:${firebaseVersion}",
            remoteConfig: "com.google.firebase:firebase-config:${firebaseVersion}",
            invites:      "com.google.firebase:firebase-invites:${firebaseVersion}",
            adMod:        "com.google.firebase:firebase-ads:${firebaseVersion}",
            appIndexing:  "com.google.android.gms:play-services-appindexing:${firebaseVersion}",
    ];
}

Que pode ser aplicado a partir desse arquivo no arquivo de nível superior build.gradle assim:

// Load dependencies
apply from: 'gradleScript/dependencies.gradle'

e no module1/build.gradle assim:

// Module build file
dependencies {
    // ...
    compile supportDependencies.appCompat
    compile supportDependencies.design
    compile firebaseDependencies.crash
}

Assine o APK sem expor a senha do keystore #

Você pode definir a configuração de assinatura para assinar o apk no arquivo build.gradle usando estas propriedades:

  • storeFile: o arquivo de armazenamento de chaves
  • storePassword: a senha do keystore
  • keyAlias: um nome de alias de chave
  • keyPassword: Uma senha de alias de chave

Em muitos casos você pode precisar evitar este tipo de informação no arquivo build.gradle.

Método A: configurar a assinatura de versão usando um arquivo keystore.properties #

É possível configurar o build.gradle do seu aplicativo para que ele leia suas informações de configuração de assinatura de um arquivo de propriedades como keystore.properties.

Configurar a assinatura como esta é benéfico porque:

  • Suas informações de configuração de assinatura são separadas do seu arquivo build.gradle
  • Você não precisa intervir durante o processo de assinatura para fornecer senhas para seu arquivo de armazenamento de chaves
  • Você pode excluir facilmente o arquivo keystore.properties do controle de versão

Primeiro, crie um arquivo chamado keystore.properties na raiz do seu projeto com conteúdo como este (substituindo os valores pelos seus):

storeFile=keystore.jks
storePassword=storePassword
keyAlias=keyAlias
keyPassword=keyPassword

Agora, no arquivo build.gradle do seu aplicativo, configure o bloco signingConfigs da seguinte forma:

android {
...

    signingConfigs {
        release {
            def propsFile = rootProject.file('keystore.properties')
            if (propsFile.exists()) {
                def props = new Properties()
                props.load(new FileInputStream(propsFile))
                storeFile = file(props['storeFile'])
                storePassword = props['storePassword']
                keyAlias = props['keyAlias']
                keyPassword = props['keyPassword']
            }
        }
    }
}

Isso é tudo, mas não se esqueça de excluir tanto o arquivo keystore quanto o arquivo keystore.properties do controle de versão.

Algumas coisas a serem observadas:

  • O caminho storeFile especificado no arquivo keystore.properties deve ser relativo ao arquivo build.gradle do seu aplicativo. Este exemplo pressupõe que o arquivo keystore esteja no mesmo diretório que o arquivo build.gradle do aplicativo.
  • Este exemplo tem o arquivo keystore.properties na raiz do projeto. Se você colocá-lo em outro lugar, certifique-se de alterar o valor em rootProject.file('keystore.properties') para o local do seu, relativo à raiz do seu projeto.

Método B: Usando uma variável de ambiente #

O mesmo pode ser alcançado também sem um arquivo de propriedades, tornando a senha mais difícil de encontrar:

android {

  signingConfigs {
    release {
        storeFile file('/your/keystore/location/key')
        keyAlias 'your_alias'
        String ps = System.getenv("ps")
        if (ps == null) {
             throw new GradleException('missing ps env variable')
        }
        keyPassword ps
        storePassword ps
    }
}

A variável de ambiente "ps" pode ser global, mas uma abordagem mais segura pode ser adicionando-a apenas ao shell do Android Studio.
No Linux, isso pode ser feito editando a Desktop Entry do Android Studio

Exec=sh -c "export ps=myPassword123 ; /path/to/studio.sh"

Controle de versão de suas compilações por meio do arquivo “version.properties” #

Você pode usar o Gradle para incrementar automaticamente a versão do seu pacote toda vez que o compilar. Para isso, crie um arquivo version.properties no mesmo diretório que seu build.gradle com o seguinte conteúdo:

VERSION_MAJOR=0
VERSION_MINOR=1
VERSION_BUILD=1

(Mudando os valores para maior e menor como achar melhor). Em seguida, no seu build.gradle, adicione o seguinte código à seção android:

// Read version information from local file and increment as appropriate
def versionPropsFile = file('version.properties')
if (versionPropsFile.canRead()) {
  def Properties versionProps = new Properties()

  versionProps.load(new FileInputStream(versionPropsFile))

  def versionMajor = versionProps['VERSION_MAJOR'].toInteger()
  def versionMinor = versionProps['VERSION_MINOR'].toInteger()
  def versionBuild = versionProps['VERSION_BUILD'].toInteger() + 1

  // Update the build number in the local file
  versionProps['VERSION_BUILD'] = versionBuild.toString()
  versionProps.store(versionPropsFile.newWriter(), null)

  defaultConfig {
    versionCode versionBuild
    versionName "${versionMajor}.${versionMinor}." + String.format("%05d", versionBuild)
  }
}

As informações podem ser acessadas em Java como uma string BuildConfig.VERSION_NAME para o número completo {major}.{minor}.{build} e como um inteiro BuildConfig.VERSION_CODE apenas para o número da compilação.

Alterando o nome do apk de saída e adicione o nome da versão: #

Este é o código para alterar o nome do arquivo do aplicativo de saída (.apk). O nome pode ser configurado atribuindo um valor diferente a newName

        def newName = "ApkName";

Se você não precisa de arquivos apk gerados automaticamente com o sufixo unaligned (o que você provavelmente não precisa), você pode adicionar o seguinte código ao arquivo build.gradle:

// delete unaligned files
android.applicationVariants.all { variant ->
  variant.assemble.doLast {
    variant.outputs.each { output ->
        println "aligned " + output.outputFile
        println "unaligned " + output.packageApplication.outputFile

        File unaligned = output.packageApplication.outputFile;
        File aligned = output.outputFile
        if (!unaligned.getName().equalsIgnoreCase(aligned.getName())) {
            println "deleting " + unaligned.getName()
            unaligned.delete()
        }
    }
  }
}

Depurando seus erros do Gradle #

O seguinte é um trecho de [Gradle - O que é um valor de saída diferente de zero e como faço para corrigi-lo?](http://stackoverflow.com/questions/36698816/gradle-what-is-a-non-zero- exit-value-and-how-do-i-fix-it), veja-o para a discussão completa.

Digamos que você esteja desenvolvendo um aplicativo e receba algum erro do Gradle que aparece e geralmente se parece com isso.

:module:someTask FAILED
FAILURE: Build failed with an exception.
* What went wrong:
Execution failed for task ':module:someTask'.
> some message here...  finished with non-zero exit value X
* Try:
Run with --stacktrace option to get the stack trace. Run with --info or --debug option to get more log output.
BUILD FAILED
Total time: Y.ZZ secs

Você pesquisa aqui no StackOverflow pelo seu problema, e as pessoas dizem para limpar e reconstruir seu projeto, ou habilitar MultiDex, e quando você tenta isso, simplesmente não está corrigindo o problema.

Existem maneiras de obter mais informações, mas a própria saída do Gradle deve apontar para o erro real nas poucas linhas acima dessa mensagem entre :module:someTask FAILED e o último :module:someOtherTask que passou . Portanto, se você fizer uma pergunta sobre seu erro, edite suas perguntas para incluir mais contexto ao erro.

Assim, você obtém um “valor de saída diferente de zero”. Bem, esse número é um bom indicador do que você deve tentar corrigir. Aqui estão alguns que ocorrem com mais frequência.

  • 1 é apenas um código de erro geral e o erro é provável na saída do Gradle
  • 2 parece estar relacionado a dependências sobrepostas ou configuração incorreta do projeto.
  • 3 parece incluir muitas dependências, ou um problema de memória.

As soluções gerais para o acima (depois de tentar uma limpeza e reconstrução do projeto) são:

  • 1 - Endereça o erro que é mencionado. Geralmente, este é um erro de tempo de compilação, o que significa que algum código em seu projeto não é válido. Isso inclui XML e Java para um projeto Android.
  • 2 & 3 - Muitas respostas aqui dizem para você habilitar multidex. Embora possa corrigir o problema, é mais provável que seja uma solução alternativa. Se você não entende por que está usando (veja o link), provavelmente não precisa dele. As soluções gerais envolvem reduzir o uso excessivo de dependências de biblioteca (como todos os serviços do Google Play, quando você só precisa usar uma biblioteca, como Maps ou Login, por exemplo).

Use gradle.properties para versionnumber/buildconfigurations central #

Você pode definir informações de configuração central em

  • um arquivo de inclusão gradle separado https://www.wikiod.com/pt/android/gradle-para-android#Centralizando dependências via arquivo “dependencies.gradle”
  • um arquivo de propriedades autônomo https://www.wikiod.com/pt/android/gradle-para-android#Controle de versão de suas compilações por meio do arquivo “version.properties”

ou faça isso com o arquivo root gradle.properties

a estrutura do projeto

root
  +- module1/
  |     build.gradle
  +- module2/
  |     build.gradle
  +- build.gradle
  +- gradle.properties

configuração global para todos os submódulos em gradle.properties

# used for manifest
# todo increment for every release
appVersionCode=19
appVersionName=0.5.2.160726

# android tools settings
appCompileSdkVersion=23
appBuildToolsVersion=23.0.2

uso em um submódulo

apply plugin: 'com.android.application'
android {
    // appXXX are defined in gradle.properties
    compileSdkVersion = Integer.valueOf(appCompileSdkVersion)
    buildToolsVersion = appBuildToolsVersion

    defaultConfig {
        // appXXX are defined in gradle.properties
        versionCode = Long.valueOf(appVersionCode)
        versionName = appVersionName
    }
}

dependencies {
    ...
}

Nota: Se você deseja publicar seu aplicativo na loja de aplicativos F-Droid, você deve usar números mágicos no arquivo gradle, porque senão o robô f-droid não pode ler o número da versão atual para detectar/verificar as alterações de versão.
