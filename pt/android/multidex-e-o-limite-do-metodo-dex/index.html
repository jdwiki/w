DEX significa arquivos de bytecode executáveis ​​do aplicativo Android (APK) na forma de arquivos Dalvik Executable (DEX), que contêm o código compilado usado para executar seu aplicativo.

A especificação Dalvik Executable limita o número total de métodos que podem ser referenciados em um único arquivo DEX a 65.536 (64 K), incluindo métodos de estrutura do Android, métodos de biblioteca e métodos em seu próprio código.

Para superar esse limite, é necessário configurar o processo de compilação do aplicativo para gerar mais de um arquivo DEX, conhecido como Multidex.

O que é dex? #

Dex é o nome do formato de arquivo e codificação para o qual o código Java do Android é compilado. As primeiras versões do Android carregavam e executavam binários dex diretamente em uma máquina virtual chamada Dalvik. Versões mais recentes do Android usam o Android Runtime (ART), que trata os arquivos dex como uma representação intermediária e executa outras compilações nele antes de executar o aplicativo.

Dex é um formato de arquivo muito antigo, em termos de vida útil dos smartphones, e foi projetado para dispositivos cuja memória principal era medida em dezenas de megabytes. As limitações de design daqueles dias permanecem conosco até hoje.

O problema: #

O formato de arquivo dex codifica um limite para o número de métodos que podem ser referenciados em um único binário. Como a parte do formato de arquivo que armazena o número de referências tem dois bytes, o número máximo de referências de método é 0xFFFF, ou 65535. Se um aplicativo contiver mais do que esse número de referências de método, ele falhará ao compilar.

O que fazer sobre isso: #

O Google forneceu uma maneira de contornar esse problema, chamada Multidex. Possui componentes de tempo de compilação e de tempo de execução. Como o próprio nome indica, em tempo de compilação ele dividirá o código entre um ou mais arquivos dex. Em tempo de execução, ele ensinará ao ClassLoader padrão como procurar classes desses arquivos.

Essa abordagem funciona bem em dispositivos mais novos, mas tem algumas desvantagens substanciais. Ele pode aumentar drasticamente o tempo de inicialização do aplicativo e, em dispositivos mais antigos, pode causar falhas de ‘Aplicativo que não está respondendo’.

Multidex, embora eficaz, deve ser evitado se possível.

Como evitar o limite: #

Antes de configurar seu aplicativo para permitir o uso de 64 K ou mais referências de método, você deve tomar medidas para reduzir o número total de referências chamadas pelo código do aplicativo, incluindo métodos definidos pelo código do aplicativo ou bibliotecas incluídas. As estratégias a seguir podem ajudá-lo a evitar atingir o limite de referência dex:

  • Revise as dependências diretas e transitivas de seu aplicativo - Certifique-se de que qualquer grande dependência de biblioteca incluída em seu aplicativo seja usada de uma maneira que supere a quantidade de código que está sendo adicionada ao aplicativo. Um antipadrão comum é incluir uma biblioteca muito grande porque alguns métodos utilitários foram úteis. Reduzir as dependências do código do aplicativo geralmente pode ajudar a evitar o limite de referência dex.
  • Remova o código não utilizado com o ProGuard - Configure as configurações do ProGuard para que seu aplicativo execute o ProGuard e certifique-se de ter a redução habilitada para compilações de lançamento. Ativar a redução garante que você não envie código não utilizado com seus APKs.

O primeiro ponto requer diligência e disciplina por parte do desenvolvedor. Ao incorporar bibliotecas de terceiros, deve-se considerar o tamanho da biblioteca. Por exemplo, duas bibliotecas JSON populares são Jackson e Gson. Funcionalmente, eles são bastante semelhantes, mas o Gson tende a ver maior uso no Android. Uma razão é que Jackson pesa cerca de 9.000 métodos, enquanto Gson contribui com 1.900.

Existem várias ferramentas disponíveis para ajudar os desenvolvedores a acompanhar o tamanho de seus aplicativos:

  • dexcount-gradle-plugin informa o número de referências de método em seu APK ou AAR em cada compilação
  • dex-method-counts é uma ferramenta de linha de comando que conta o número de referências de método em um APK
  • www.methodscount.com é um serviço da web que contará as referências de método em qualquer APK que você enviar.

Ativando o Multidex #

Para habilitar uma configuração multidex, você precisa:

  • para alterar sua configuração de compilação do Gradle
  • para usar um MultiDexApplication ou habilitar o MultiDex em sua classe Application

Configuração do Gradle #

Em app/build.gradle adicione estas partes:

android {
    compileSdkVersion 24
    buildToolsVersion "24.0.1"

    defaultConfig {
        ...
        minSdkVersion 14
        targetSdkVersion 24
        ...

        // Enabling multidex support.
        multiDexEnabled true
    }
    ...
}

dependencies {
  compile 'com.android.support:multidex:1.0.1'
}

Habilite o MultiDex em seu aplicativo #

Em seguida, prossiga com uma das três opções:

  • https://www.wikiod.com/pt/android/multidex-e-o-limite-do-metodo-dex#Multidex estendendo Application

  • https://www.wikiod.com/pt/android/multidex-e-o-limite-do-metodo-dex#Multidex estendendo MultiDexApplication

  • https://www.wikiod.com/pt/android/multidex-e-o-limite-do-metodo-dex#Multidex usando o MultiDexApplication diretamente

Quando essas definições de configuração são adicionadas a um aplicativo, as ferramentas de compilação do Android constroem um dex primário (classes.dex) e um suporte (classes2.dex, classes3.dex) conforme necessário. O sistema de compilação os empacotará em um arquivo APK para distribuição.

Multidex estendendo Application #

Use esta opção se seu projeto requer uma subclasse Application.

Especifique esta subclasse Application usando a propriedade android:name no arquivo manifesto dentro da tag application.

Na subclasse Application, adicione a substituição do método attachBaseContext() e nesse método chame MultiDex.install():

package com.example;

import android.app.Application;
import android.content.Context;

/**
 * Extended application that support multidex 
 */
public class MyApplication extends Application {

    @Override
    protected void attachBaseContext(Context base) {
        super.attachBaseContext(base);
        MultiDex.install(this);
    }
}

Certifique-se de que a subclasse Application esteja especificada na tag application do seu AndroidManifest.xml:

<application
    android:name="com.example.MyApplication"
    android:icon="@drawable/ic_launcher"
    android:label="@string/app_name">
</application>

Multidex estendendo MultiDexApplication #

Isso é muito semelhante a usar uma subclasse Application e substituir o método attachBaseContext().

No entanto, usando este método, você não precisa sobrescrever attachBaseContext(), pois isso já é feito na superclasse MultiDexApplication.

Estenda MultiDexApplication em vez de Application:

package com.example;

import android.support.multidex.MultiDexApplication;
import android.content.Context;

/**
 * Extended MultiDexApplication 
 */
public class MyApplication extends MultiDexApplication {

     // No need to override attachBaseContext()

     //..........
}

Adicione esta classe ao seu AndroidManifest.xml exatamente como se estivesse estendendo Application:

<application
    android:name="com.example.MyApplication"
    android:icon="@drawable/ic_launcher"
    android:label="@string/app_name">
</application>

Multidex usando o MultiDexApplication diretamente #

Use esta opção se você não precisar de uma subclasse Application.

Esta é a opção mais simples, mas desta forma você não pode fornecer sua própria subclasse Application. Se uma subclasse Application for necessária, você terá que mudar para uma das outras opções para fazer isso.

Para esta opção, basta especificar o nome de classe totalmente qualificado android.support.multidex.MultiDexApplication para a propriedade android:name da tag application no AndroidManifest.xml:

<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.android.multidex.myapplication">
    <application
        ...
        android:name="android.support.multidex.MultiDexApplication">
        ...
    </application>
</manifest>

Referências de métodos de contagem em cada compilação (Plugin Gradle do Dexcount) #

O plugin de dexcount conta métodos e contagem de recursos de classe após uma compilação bem-sucedida.

Adicione o plugin no app/build.gradle:

apply plugin: 'com.android.application'

buildscript {
    repositories {
        mavenCentral() // or jcenter()
    }

    dependencies {
        classpath 'com.getkeepsafe.dexcount:dexcount-gradle-plugin:0.5.5'
    }
}

Aplique o plugin no arquivo app/build.gradle:

apply plugin: 'com.getkeepsafe.dexcount'

Procure os dados de saída gerados pelo plugin em:

../app/build/outputs/dexcount

Especialmente útil é o gráfico .html em:

../app/build/outputs/dexcount/debugChart/index.html
