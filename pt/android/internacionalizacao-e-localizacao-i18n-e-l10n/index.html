Internacionalização (i18n) e Localização (L10n) são usados ​​para adaptar o software de acordo com as diferenças de idiomas, diferenças regionais e público-alvo.

Internacionalização: o processo de planejamento para localização futura, ou seja, tornar o projeto de software flexível a ponto de poder ajustar e adaptar-se a futuros esforços de localização.

Localização : o processo de adaptação do software para uma determinada região/país/mercado (localidade).

Para testar a localização de um dispositivo, o dispositivo ou o emulador pode ser reinicializado em uma localidade específica usando adb da seguinte forma:

  1. Execute o adb usando o comando: adb shell
  2. Execute o seguinte comando no prompt de comando do adb: setprop persist.sys.locale [BCP-47 language tag];stop;sleep 5;start onde [BCP-47 language tag] é o código específico do idioma conforme descrito aqui : códigos BCP47

por exemplo. para verificar a localização em japonês no aplicativo, use o comando: setprop persist.sys.locale ja-JP;stop;sleep 5;start

Planejamento para localização: habilite o suporte RTL no manifesto #

O suporte RTL (da direita para a esquerda) é uma parte essencial do planejamento para i18n e L10n. Ao contrário do idioma inglês, que é escrito da esquerda para a direita, muitos idiomas, como árabe, japonês, hebraico, etc., são escritos da direita para a esquerda. Para atrair um público mais global, é uma boa ideia planejar seus layouts para suportar esses idiomas desde o início do projeto, para que a adição de localização seja mais fácil posteriormente.

O suporte RTL pode ser ativado em um aplicativo Android adicionando a tag supportsRtl no AndroidManifest, assim:

<application
    ...
    android:supportsRtl="true"
    ...>
...
</application>

Planejamento para localização: adicione suporte a RTL em Layouts #

A partir do SDK 17 (Android 4.2), o suporte RTL foi adicionado nos layouts do Android e é uma parte essencial da localização. Daqui para frente, a notação left/right nos layouts deve ser substituída pela notação start/end. Se, no entanto, seu projeto tiver um valor minSdk menor que 17, então as notações left/right e start/end devem ser usadas nos layouts.

Para layouts relativos, alignParentStart e alignParentEnd devem ser usados, assim:

<RelativeLayout
    android:layout_width="match_parent"
    android:layout_height="match_parent">
    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:layout_alignParentTop="true"
        android:layout_alignParentLeft="true"
        android:layout_alignParentStart="true"/>
    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:layout_alignParentTop="true"
        android:layout_alignParentRight="true"
        android:layout_alignParentEnd="true"/>
</RelativeLayout>

Para especificar a gravidade e a gravidade do layout, uma notação semelhante deve ser usada, assim:

    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:layout_gravity="left|start"
        android:gravity="left|start"/>
    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:layout_gravity="right|end"
        android:gravity="right|end"/>

Preenchimentos e margens também devem ser especificados de acordo, assim:

<include layout="@layout/notification"
    android:layout_width="fill_parent"
    android:layout_height="wrap_content"
    android:layout_marginLeft="12dp"
    android:layout_marginStart="12dp"
    android:paddingLeft="128dp"
    android:paddingStart="128dp"
    android:layout_toLeftOf="@id/cancel_action"
    android:layout_toStartOf="@id/cancel_action"/>
<include layout="@layout/notification2"
    android:layout_width="fill_parent"
    android:layout_height="wrap_content"
    android:layout_marginRight="12dp"
    android:layout_marginEnd="12dp"
    android:paddingRight="128dp"
    android:paddingEnd="128dp"
    android:layout_toRightOf="@id/cancel_action"
    android:layout_toEndOf="@id/cancel_action"/>

Planejamento para localização: layouts de teste para RTL #

Para testar se os layouts criados são compatíveis com RTL, faça o seguinte:

Vá para Configurações -> Opções do desenvolvedor -> Desenho -> Forçar direção do layout RTL

Habilitar essa opção forçaria o dispositivo a usar localidades RTL e você pode verificar facilmente todas as partes do aplicativo para suporte a RTL. Observe que você não precisa adicionar nenhum novo suporte a localidades/idiomas até este ponto.

Codificação para localização: Criando strings e recursos padrão #

A primeira etapa para codificar para localização é criar recursos padrão. Essa etapa é tão implícita que muitos desenvolvedores nem pensam nisso. No entanto, a criação de recursos padrão é importante porque, se o dispositivo for executado em uma localidade não suportada, ele carregará todos os seus recursos das pastas padrão. Se até mesmo um dos recursos estiver faltando nas pastas padrão, o aplicativo simplesmente travará.

O conjunto padrão de strings deve ser colocado na seguinte pasta no local especificado:

res/values/strings.xml

Esse arquivo deve conter as strings no idioma que a maioria dos usuários do aplicativo deve falar.

Além disso, os recursos padrão do aplicativo devem ser colocados nas seguintes pastas e locais:

res/drawable/
res/layout/

Se seu aplicativo requer pastas como anim ou xml, os recursos padrão devem ser adicionados às seguintes pastas e locais:

res/anim/
res/xml/
res/raw/

Para fornecer traduções em outros idiomas (locales), precisamos criar um strings.xml em uma pasta separada pela seguinte convenção:

res/values-<locale>/strings.xml

Outras traduções para outras localidades também podem ser adicionadas ao aplicativo.

Uma lista completa de códigos de localidade pode ser encontrada aqui: códigos ISO 639

Strings não traduzíveis:

Seu projeto pode ter certas strings que não devem ser traduzidas. Strings que são usadas como chaves para SharedPreferences ou strings que são usadas como símbolos, se enquadram nesta categoria. Essas strings devem ser armazenadas apenas no strings.xml padrão e devem ser marcadas com um atributo translatable="false". por exemplo.

<string name="pref_widget_display_label_hot">Hot News</string>
<string name="pref_widget_display_key" translatable="false">widget_display</string>
<string name="pref_widget_display_hot" translatable="false">0</string>

Esse atributo é importante porque as traduções muitas vezes são realizadas por profissionais bilíngues. Isso permitiria que essas pessoas envolvidas em traduções identificassem strings que não devem ser traduzidas, economizando tempo e dinheiro.

Codificação para localização: Fornecendo layouts alternativos #

A criação de layouts específicos de idioma geralmente é desnecessária se você tiver especificado a notação start/end correta, conforme descrito no exemplo anterior. No entanto, pode haver situações em que os layouts padrão não funcionem corretamente para determinados idiomas. Às vezes, os layouts da esquerda para a direita podem não ser traduzidos para idiomas RTL. É necessário fornecer os layouts corretos nesses casos.

Para fornecer otimização completa para layouts RTL, podemos usar arquivos de layout totalmente separados usando o qualificador de recurso ldrtl (ldrtl significa layout-direction-right-to-left}). Por exemplo, podemos salvar seus arquivos de layout padrão em res/layout/ e nossos layouts otimizados RTL em res/layout-ldrtl/.

O qualificador ldrtl é ótimo para recursos desenháveis, para que você possa fornecer gráficos orientados na direção correspondente à direção de leitura.
