Evite vazar atividades com AsyncTask #

Uma palavra de cautela: AsyncTask tem muitas pegadinhas além do vazamento de memória descrito aqui. Portanto, tenha cuidado com essa API ou evite-a completamente se você não entender completamente as implicações. Existem muitas alternativas (Thread, EventBus, RxAndroid, etc).

Um erro comum com AsyncTask é capturar uma forte referência ao host Activity (ou Fragment):

class MyActivity extends Activity {
  private AsyncTask<Void, Void, Void> myTask = new AsyncTask<Void, Void, Void>() {
    // Don't do this! Inner classes implicitly keep a pointer to their
    // parent, which in this case is the Activity!
  }
}

Isso é um problema porque AsyncTask pode sobreviver facilmente à Activity pai, por exemplo, se uma mudança de configuração acontecer enquanto a tarefa está em execução.

A maneira correta de fazer isso é tornar sua tarefa uma classe static, que não captura o pai, e mantendo uma referência fraca para o host Activity:

class MyActivity extends Activity {
  static class MyTask extends AsyncTask<Void, Void, Void> {
    // Weak references will still allow the Activity to be garbage-collected
    private final WeakReference<MyActivity> weakActivity;

    MyTask(MyActivity myActivity) {
      this.weakActivity = new WeakReference<>(myActivity);
    }

    @Override
    public Void doInBackground(Void... params) {
      // do async stuff here
    }

    @Override
    public void onPostExecute(Void result) {
      // Re-acquire a strong reference to the activity, and verify
      // that it still exists and is active.
      MyActivity activity = weakActivity.get();
      if (activity == null
          || activity.isFinishing()
          || activity.isDestroyed()) {
        // activity is no longer valid, don't do anything!
        return;
      }

      // The activity is still valid, do main-thread stuff here
    }
  }
}

Vazamentos de memória comuns e como corrigi-los #

1. Corrija seus contextos: #

Tente usar o contexto apropriado: Por exemplo, como um Toast pode ser visto em muitas atividades em vez de apenas uma, use getApplicationContext() para brindes e, como os serviços podem continuar em execução mesmo que uma atividade tenha terminado, inicie um serviço com:

Intent myService = new Intent(getApplicationContext(), MyService.class);

Use esta tabela como um guia rápido para qual contexto é apropriado:

Original artigo sobre o contexto aqui.

2. Referência estática ao contexto #

Um erro grave de vazamento de memória é manter uma referência estática para View. Cada View tem uma referência interna ao Context. O que significa que uma Activity antiga com toda a hierarquia de visualização não será coletada como lixo até que o aplicativo seja encerrado. Você terá seu aplicativo duas vezes na memória ao girar a tela.

Certifique-se de que não haja absolutamente nenhuma referência estática para View, Context ou qualquer um de seus descendentes.

3. Verifique se você está realmente terminando seus serviços. #

Por exemplo, tenho um intentService que usa a API do serviço de localização do Google. E eu esqueci de chamar googleApiClient.disconnect();:

//Disconnect from API onDestroy()
if (googleApiClient.isConnected()) {
    LocationServices.FusedLocationApi.removeLocationUpdates(googleApiClient, GoogleLocationService.this);
    googleApiClient.disconnect();
}

4. Verifique o uso de imagens e bitmaps: #

Se você estiver usando a biblioteca Picasso da Square, descobri que estava vazando memória por não usar o .fit(), que reduziu drasticamente o consumo de memória de 50 MB em média para menos de 19 MB:

Picasso.with(ActivityExample.this)                   //Activity context
                .load(object.getImageUrl())           
                .fit()                                //This avoided the OutOfMemoryError
                .centerCrop()                         //makes image to not stretch
                .into(imageView);

5. Se você estiver usando receptores de transmissão, cancele o registro deles. #

6. Se você estiver usando java.util.Observer (padrão Observer): #

Certifique-se de usar deleteObserver(observer);

Detecte vazamentos de memória com a biblioteca LeakCanary #

LeakCanary é uma biblioteca Java de código aberto para detectar vazamentos de memória em suas compilações de depuração.

Basta adicionar as dependências no build.gradle:

 dependencies {
   debugCompile 'com.squareup.leakcanary:leakcanary-android:1.5.1'
   releaseCompile 'com.squareup.leakcanary:leakcanary-android-no-op:1.5.1'
   testCompile 'com.squareup.leakcanary:leakcanary-android-no-op:1.5.1'
 }

Em seguida, na sua classe Application:

public class ExampleApplication extends Application {

  @Override public void onCreate() {
    super.onCreate();
    
    if (LeakCanary.isInAnalyzerProcess(this)) {
      // This process is dedicated to LeakCanary for heap analysis.
      // You should not init your app in this process.
      return;
    }

    LeakCanary.install(this);
  }
}

Agora o LeakCanary mostrará automaticamente uma notificação quando um vazamento de memória de atividade for detectado em sua compilação de depuração.

NOTA: O código de lançamento não conterá nenhuma referência a LeakCanary além das duas classes vazias que existem na dependência leakcanary-android-no-op.

Retorno de chamada anônimo em atividades #

Toda vez que você cria uma classe anônima, ela retém uma referência implícita à sua classe pai. Então, quando você escreve:

public class LeakyActivity extends Activity
{

...

    foo.registerCallback(new BarCallback() 
    {            
        @Override
        public void onBar() 
        {
            // do something                
        }            
    });
}

Na verdade, você está enviando uma referência à sua instância LeakyActivity para foo. Quando o usuário navega para fora de sua LeakyActivity, essa referência pode impedir que a instância de LeakyActivity seja coletada como lixo. Esse é um vazamento sério, pois as atividades mantêm uma referência a toda a hierarquia de exibição e, portanto, são objetos bastante grandes na memória.

Como evitar esse vazamento:

É claro que você pode evitar totalmente o uso de retornos de chamada anônimos em atividades. Você também pode cancelar o registro de todos os seus retornos de chamada em relação ao ciclo de vida da atividade. igual a:

public class NonLeakyActivity extends Activity
{
    private final BarCallback mBarCallback = new BarCallback() 
    {            
        @Override
        public void onBar() 
        {
            // do something                
        }            
    });

    @Override
    protected void onResume()
    {
        super.onResume();
        foo.registerCallback(mBarCallback);
    }

    @Override
    protected void onPause()
    {
        super.onPause();
        foo.unregisterCallback(mBarCallback);
    }
}

Contexto de atividade em classes estáticas #

Evite vazar atividades com ouvintes #

Se você implementar ou criar um listener em uma Activity, sempre preste atenção no ciclo de vida do objeto que tem o listener registrado.

Considere um aplicativo onde temos várias atividades/fragmentos diferentes interessados ​​quando um usuário está conectado ou desconectado. Uma maneira de fazer isso seria ter uma instância singleton de um UserController que pode ser inscrito para ser notificado quando o estado do usuário for alterado:

public class UserController {
    private static UserController instance;
    private List<StateListener> listeners;

    public static synchronized UserController getInstance() {
        if (instance == null) {
            instance = new UserController();
        }
        return instance;
    }

    private UserController() {
        // Init
    }

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(listener);
    }

    public void logout() {
        for (StateListener listener : listeners) {
            listener.userLoggedOut();
        }
    }

    public void login() {
        for (StateListener listener : listeners) {
            listener.userLoggedIn();
        }
    }

    public interface StateListener {
        void userLoggedIn();
        void userLoggedOut();
    }
}

Depois, há duas atividades, SignInActivity:

public class SignInActivity extends Activity implements UserController.StateListener{

    UserController userController;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        this.userController = UserController.getInstance();
        this.userController.registerUserStateChangeListener(this);
    }

    @Override
    public void userLoggedIn() {
        startMainActivity();
    }

    @Override
    public void userLoggedOut() {
        showLoginForm();
    }

    ...

    public void onLoginClicked(View v) {
        userController.login();
    }
}

E MainActivity:

public class MainActivity extends Activity implements UserController.StateListener{
    UserController userController;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        this.userController = UserController.getInstance();
        this.userController.registerUserStateChangeListener(this);
    }

    @Override
    public void userLoggedIn() {
        showUserAccount();
    }

    @Override
    public void userLoggedOut() {
        finish();
    }

    ...

    public void onLogoutClicked(View v) {
        userController.logout();
    }
}

O que acontece com este exemplo é que toda vez que o usuário faz login e sai novamente, uma instância MainActivity vaza. O vazamento ocorre porque há uma referência à atividade em UserController#listeners.

Observação: mesmo se usarmos uma classe interna anônima como ouvinte, a atividade ainda vazaria:

...
this.userController.registerUserStateChangeListener(new UserController.StateListener() {
    @Override
    public void userLoggedIn() {
        showUserAccount();
    }

    @Override
    public void userLoggedOut() {
        finish();
    }
});    
...

A atividade ainda vazaria, porque a classe interna anônima tem uma referência implícita à classe externa (neste caso, a atividade). É por isso que é possível chamar métodos de instância na classe externa da classe interna. Na verdade, o único tipo de classe interna que não tem uma referência à classe externa são as classes internas estáticas.

Em resumo, todas as instâncias de classes internas não estáticas mantêm uma referência implícita à instância da classe externa que as criou.

Existem duas abordagens principais para resolver isso, adicionando um método para remover um ouvinte de UserController#listeners ou usando um WeakReference para manter a referência dos ouvintes.

Alternativa 1: removendo ouvintes #

Vamos começar criando um novo método removeUserStateChangeListener(StateListener listener):

public class UserController {

    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(listener);
    }

    public void removeUserStateChangeListener(StateListener listener) {
        listeners.remove(listener);
    }

    ...
}

Então vamos chamar este método no método onDestroy da atividade:

public class MainActivity extends Activity implements UserController.StateListener{
    ...

    @Override
    protected void onDestroy() {
        super.onDestroy();
        userController.removeUserStateChangeListener(this);
    }
}

Com esta modificação, as instâncias de MainActivity não são mais vazadas quando o usuário faz login e logout. No entanto, se a documentação não estiver clara, é provável que o próximo desenvolvedor que comece a usar o UserController perca a necessidade de cancelar o registro do ouvinte quando a atividade for destruída, o que nos leva ao segundo método de evitar esses tipos de vazamentos.

Alternativa 2: Usando referências fracas #

Primeiramente, vamos começar explicando o que é uma referência fraca. Uma referência fraca, como o nome sugere, contém uma referência fraca a um objeto. Comparado a um campo de instância normal, que é uma referência forte, uma referência fraca não impede o coletor de lixo, GC, de remover os objetos. No exemplo acima, isso permitiria que MainActivity fosse coletado como lixo após ter sido destruído se o UserController usasse WeakReference para referenciar os listeners.

Em suma, uma referência fraca está dizendo ao GC que, se ninguém mais tiver uma referência forte a esse objeto, vá em frente e remova-o.

Vamos modificar o UserController para usar uma lista de WeakReference para acompanhar seus ouvintes:

public class UserController {

    ...
    private List<WeakReference<StateListener>> listeners;
    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listeners.add(new WeakReference<>(listener));
    }

    public void removeUserStateChangeListener(StateListener listenerToRemove) {
        WeakReference referencesToRemove = null;
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null && listener == listenerToRemove) {
                referencesToRemove = listenerRef;
                break;
            }
        }
        listeners.remove(referencesToRemove);
    }

    public void logout() {
        List referencesToRemove = new LinkedList();
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null) {
                listener.userLoggedOut();
            } else {
                referencesToRemove.add(listenerRef);
            }
        }
    }

    public void login() {
        List referencesToRemove = new LinkedList();
        for (WeakReference<StateListener> listenerRef : listeners) {
            StateListener listener = listenerRef.get();
            if (listener != null) {
                listener.userLoggedIn();
            } else {
                referencesToRemove.add(listenerRef);
            }
        }
    }
    ...    
}

Com esta modificação, não importa se os ouvintes são removidos ou não, já que o UserController não contém referências fortes a nenhum dos ouvintes. No entanto, escrever esse código padrão toda vez é complicado. Portanto, vamos criar uma classe genérica chamada WeakCollection:

public class WeakCollection<T> {
    private LinkedList<WeakReference<T>> list;

    public WeakCollection() {
        this.list = new LinkedList<>();
    }
    public void put(T item){
        //Make sure that we don't re add an item if we already have the reference.
        List<T> currentList = get();
        for(T oldItem : currentList){
            if(item == oldItem){
                return;
            }
        }
        list.add(new WeakReference<T>(item));
    }

    public List<T> get() {
        List<T> ret = new ArrayList<>(list.size());
        List<WeakReference<T>> itemsToRemove = new LinkedList<>();
        for (WeakReference<T> ref : list) {
            T item = ref.get();
            if (item == null) {
                itemsToRemove.add(ref);
            } else {
                ret.add(item);
            }
        }
        for (WeakReference ref : itemsToRemove) {
            this.list.remove(ref);
        }
        return ret;
    }

    public void remove(T listener) {
        WeakReference<T> refToRemove = null;
        for (WeakReference<T> ref : list) {
            T item = ref.get();
            if (item == listener) {
                refToRemove = ref;
            }
        }
        if(refToRemove != null){
            list.remove(refToRemove);
        }
    }
}

Agora vamos reescrever UserController para usar WeakCollection<T>:

public class UserController {
    ...
    private WeakCollection<StateListener> listenerRefs;
    ...

    public void registerUserStateChangeListener(StateListener listener) {
        listenerRefs.put(listener);
    }

    public void removeUserStateChangeListener(StateListener listenerToRemove) {
        listenerRefs.remove(listenerToRemove);
    }

    public void logout() {
        for (StateListener listener : listenerRefs.get()) {
            listener.userLoggedOut();
        }
    }

    public void login() {
        for (StateListener listener : listenerRefs.get()) {
            listener.userLoggedIn();
        }
    }

    ...
}

Conforme mostrado no exemplo de código acima, o WeakCollection<T> remove todo o código padrão necessário para usar WeakReference em vez de uma lista normal. Para completar: Se uma chamada para UserController#removeUserStateChangeListener(StateListener) for perdida, o ouvinte e todos os objetos que ele está referenciando não vazarão.

Evite vazamentos de memória com Anonymous Class, Handler, Timer Task, Thread #


No Android, todo desenvolvedor usa Anonymous Class (Runnable) pelo menos uma vez em um projeto. Qualquer Classe Anônima tem uma referência ao seu pai (atividade). Se executarmos uma tarefa de longa duração, a atividade pai não será destruída até que a tarefa seja finalizada.
Exemplo usa manipulador e classe anônimo Runnable. A memória vazará quando encerrarmos a atividade antes que o Runnable seja concluído.

  new Handler().postDelayed(new Runnable() {
        @Override
        public void run() {
            // do abc long 5s or so
        }
    }, 10000); // run "do abc" after 10s. It same as timer, thread...

Como o resolvemos?

  1. Não faça nenhuma operação longa com ‘Anonymous Class’ ou precisamos de uma ‘Static class’ para ela e passemos ‘WeakReference’ para ela (como activity, view…). Thread é o mesmo que Anonymous Class.
  2. Cancele o Handler, Timer quando a atividade for destruída.
