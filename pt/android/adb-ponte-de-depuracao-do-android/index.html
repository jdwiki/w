Conecte o ADB a um dispositivo via WiFi #

A configuração padrão do ADB envolve uma conexão USB com um dispositivo físico. Se preferir, você pode alternar para o modo TCP/IP e conectar o ADB via WiFi.

Dispositivo não enraizado #

  1. Acesse a mesma rede:
  • Certifique-se de que seu dispositivo e seu computador estejam na mesma rede.
  1. Conecte o dispositivo ao computador host com um cabo USB.
  2. Conecte adb ao dispositivo pela rede:

Enquanto seu dispositivo estiver conectado ao adb via USB, execute o seguinte comando para escutar uma conexão TCP/IP em uma porta (padrão 5555):

  • Digite adb tcpip <port> (alternar para o modo TCP/IP).
  • Desconecte o cabo USB do dispositivo de destino.
  • Digite adb connect <ip address>:<port> (a porta é opcional; padrão 5555).

Por exemplo:

   adb tcpip 5555
   adb connect 192.168.0.101:5555

Se você não souber o IP do seu dispositivo, você pode:

  • verifique o IP nas configurações de WiFi do seu dispositivo.
  • use ADB para descobrir IP (via USB):
    1. Connect the device to the computer via USB
    2. In a command line, type adb shell ifconfig and copy your device’s IP address

Para voltar à depuração via USB, use o seguinte comando:

   adb usb

Você também pode conectar o ADB via WiFi instalando um plug-in no Android Studio. Para fazer isso, vá para Configurações> Plugins e Navegue pelos repositórios, procure ADB WiFi, instale-o e reabra o Android Studio. Você verá um novo ícone em sua barra de ferramentas, conforme mostrado na imagem a seguir. Conecte o dispositivo ao computador host via USB e clique neste ícone AndroidWiFiADB. Ele exibirá uma mensagem se o seu dispositivo está conectado ou não. Uma vez conectado, você pode desconectar seu USB.

Dispositivo enraizado #

Observação: alguns dispositivos que são enraizados podem usar o aplicativo ADB WiFi da Play Store para habilitar isso de maneira simples. Além disso, para determinados dispositivos (especialmente aqueles com ROMs CyanogenMod) esta opção está presente nas Opções do desenvolvedor entre as configurações. Habilitá-lo fornecerá o endereço IP e o número da porta necessários para conectar ao adb simplesmente executando adb connect <ip address>:<port>.

Quando você tem um dispositivo com root, mas não tem acesso a um cabo USB #

O processo é explicado em detalhes na seguinte resposta: http://stackoverflow.com/questions/2604727/how-can-i-connect-to-android-with-adb-over-tcp/3623727#3623727 Os comandos mais importantes são mostrados abaixo.

Abra um terminal no dispositivo e digite o seguinte:

su
setprop service.adb.tcp.port <a tcp port number>
stop adbd
start adbd

Por exemplo:

setprop service.adb.tcp.port 5555

E no seu computador:

adb connect <ip address>:<a tcp port number>

Por exemplo:

adb connect 192.168.1.2:5555

Para desativá-lo:

setprop service.adb.tcp.port -1
stop adbd
start adbd

Evite tempo limite #

Por padrão, o adb atingirá o tempo limite após 5000 ms. Isso pode acontecer em alguns casos, como WiFi lento ou APK grande.

Uma simples mudança na configuração do Gradle pode resolver o problema:

android {
    adbOptions {
        timeOutInMs 10 * 1000
    }
}

Comando ADB direto para um dispositivo específico em uma configuração de vários dispositivos #

1. Segmente um dispositivo por número de série

Use a opção -s seguida por um nome de dispositivo para selecionar em qual dispositivo o comando adb deve ser executado. As opções -s devem ser as primeiras da linha, antes do comando.

adb -s <device> <command>

Exemplo:

adb devices

List of devices attached
emulator-5554       device
02157df2d1faeb33    device

adb -s emulator-5554 shell

Exemplo#2:

adb devices -l

List of devices attached
06157df65c6b2633    device usb:1-3 product:zerofltexx model:SM_G920F device:zeroflte
LC62TB413962        device usb:1-5 product:a50mgp_dug_htc_emea model:HTC_Desire_820G_dual_sim device:htc_a50mgp_dug

adb -s usb:1-3 shell

2. Segmente um dispositivo quando apenas um tipo de dispositivo estiver conectado

Você pode direcionar o único emulador em execução com -e

adb -e <command>

Ou você pode direcionar o único dispositivo USB conectado com -d

adb -d <command>

Tirando uma captura de tela e um vídeo (apenas para kitkat) de uma tela de dispositivo #

Captura de tela: Opção 1 (adb puro) #

O comando shell adb nos permite executar comandos usando o shell embutido de um dispositivo. O comando shell screencap captura o conteúdo atualmente visível em um dispositivo e o salva em um determinado arquivo de imagem, por exemplo, /sdcard/screen.png:

adb shell screencap /sdcard/screen.png

Você pode então usar [o comando pull][pull] para baixar o arquivo do dispositivo para o diretório atual em seu computador:

adb pull /sdcard/screen.png

[puxar]: https://www.wikiod.com/pt/android/adb-ponte-de-depuracao-do-android#Puxe (empurre) arquivos de (para) o dispositivo “puxar”

Captura de tela: Opção 2 (mais rápido) #

Execute o seguinte one-liner:

(Marshmallow e anteriores):

adb shell screencap -p | perl -pe 's/\x0D\x0A/\x0A/g' > screen.png

(Nougat e mais tarde):

adb shell screencap -p > screen.png

O sinalizador -p redireciona a saída do comando screencap para stdout. A expressão Perl na qual isso é canalizado limpa alguns problemas de fim de linha no Marshmallow e anteriores. O fluxo é então gravado em um arquivo chamado screen.png dentro do diretório atual. Consulte este artigo e [este artigo](http://www.stkent.com/ 2016/08/28/capturing-Nougat-screenshots-using-adb-shell.html) para obter mais informações.

Vídeo #

isso só funciona no KitKat e apenas via ADB. Isso não está funcionando abaixo do Kitkat Para começar a gravar a tela do seu dispositivo, execute o seguinte comando:

adb ​​shell screenrecord /sdcard/example.mp4, Este comando começará a gravar a tela do seu dispositivo usando as configurações padrão e salvará o vídeo resultante em um arquivo no arquivo /sdcard/example.mp4 em seu dispositivo.

Quando terminar de gravar, pressione Ctrl+C (z no Linux) na janela do prompt de comando para interromper a gravação da tela. Você pode então encontrar o arquivo de gravação de tela no local que você especificou. Observe que a gravação da tela é salva no armazenamento interno do seu dispositivo, não no seu computador.

As configurações padrão são usar a resolução de tela padrão do seu dispositivo, codificar o vídeo a uma taxa de bits de 4 Mbps e definir o tempo máximo de gravação de tela para 180 segundos. Para obter mais informações sobre as opções de linha de comando que você pode usar, execute o seguinte comando:

adb ​​shell screenrecord –help, Isso funciona sem fazer root no dispositivo. Espero que isto ajude.

Puxe (empurre) arquivos de (para) o dispositivo #

Você pode extrair (baixar) arquivos do dispositivo executando o seguinte comando:

adb pull <remote> <local>

Por exemplo:

adb pull /sdcard/ ~/

Você também pode enviar (upload) arquivos do seu computador para o dispositivo:

adb push <local> <remote>

Por exemplo:

adb push ~/image.jpg /sdcard/

Exemplo para recuperar banco de dados do dispositivo

sudo adb -d shell "run-as com.example.name cat /data/da/com.example.name /databases/DATABASE_NAME  > /sdcard/file

Ver logcat #

Você pode executar logcat como um comando adb ou diretamente em um prompt de shell do seu emulador ou dispositivo conectado. Para visualizar a saída do log usando adb, navegue até o diretório platform-tools/ do SDK e execute:

$ adb logcat

Alternativamente, você pode criar uma conexão shell com um dispositivo e então executar:

$ adb shell
$ logcat

Um comando útil é:

adb logcat -v threadtime

Isso exibe a data, hora de invocação, prioridade, tag e o PID e TID do encadeamento que emite a mensagem em um formato de mensagem longa.

Filtragem

Os logs do Logcat têm os chamados níveis de log:

V — Detalhado, D — Depuração, I — Informações, W — Aviso, E — Erro, F — Fatal, * S* — Silencioso

Você também pode filtrar o logcat por nível de log. Por exemplo, se você deseja apenas gerar o nível de depuração:

adb logcat *:D

O Logcat pode ser filtrado por um nome de pacote, é claro que você pode combiná-lo com o filtro de nível de log:

adb logcat <package-name>:<log level>

Você também pode filtrar o log usando grep (mais sobre como filtrar a saída do logcat aqui):

adb logcat | grep <some text>

No Windows, o filtro pode ser usado usando findstr, por exemplo:

adb logcat | findstr <some text>

Para visualizar o buffer de log alternativo [main|events|radio], execute o logcat com a opção -b:

adb logcat -b radio

Salve a saída no arquivo:

adb logcat > logcat.txt

Salve a saída no arquivo enquanto também assiste:

adb logcat | tee logcat.txt

Limpando os registros:

adb logcat -c

1: https://www.wikiod.com/pt/android/introducao-ao-android#Filtrando a saída do logcat

Limpar dados do aplicativo #

Pode-se limpar os dados do usuário de um aplicativo específico usando adb:

adb shell pm clear <package>

Isso é o mesmo que navegar pelas configurações do telefone, selecionar o aplicativo e pressionar o botão limpar dados.

  • pm invoca o gerenciador de pacotes no dispositivo
  • clear exclui todos os dados associados a um pacote

Visualize e extraia arquivos de cache de um aplicativo #

Você pode usar este comando para listar os arquivos para seu próprio apk depurável:

adb shell run-as <sample.package.id> ls /data/data/sample.package.id/cache

E este script para puxar do cache, copia o conteúdo para o sdcard primeiro, puxa e remove no final:

#!/bin/sh
adb shell "run-as <sample.package.id> cat '/data/data/<sample.package.id>/$1' > '/sdcard/$1'"
adb pull "/sdcard/$1"
adb shell "rm '/sdcard/$1'"

Então você pode puxar um arquivo do cache assim:

./pull.sh cache/someCachedData.txt

Obter arquivo de banco de dados via ADB

sudo adb -d shell "run-as com.example.name cat /data/da/com.example.name /databases/STUDENT_DATABASE  > /sdcard/file

Enviando transmissão #

É possível enviar broadcast para BroadcastReceiver com adb.

Neste exemplo estamos enviando broadcast com ação com.test.app.ACTION e string extra no bundle 'foo'='bar':

adb shell am broadcast -a action com.test.app.ACTION --es foo "bar"

Você pode colocar qualquer outro tipo suportado para agrupar, não apenas strings:

–ez  - booleano
–ei  - inteiro
–el  - longo
–ef  - flutuar
–eu  - uri
–eia - array int (separado por ‘,’)
–ela - matriz longa (separada por ‘,’)
–efa - array float (separado por ‘,’)
–esa - array de strings (separado por ‘,’)

Para enviar intent para pacote/classe específico, o parâmetro -n ou -p pode ser usado.
Enviando para o pacote:

-p com.test.app

Enviando para um componente específico (classe SomeReceiver em com.test.app package):

-n com.test.app/.SomeReceiver

Instalar e executar um aplicativo #

Para instalar um arquivo APK, use o seguinte comando:

adb install path/to/apk/file.apk

ou se o aplicativo já existe e queremos reinstalar

adb install -r path/to/apk/file.apk

Para desinstalar um aplicativo, temos que especificar seu pacote

adb uninstall application.package.name

Use o seguinte comando para iniciar um aplicativo com um nome de pacote fornecido (ou uma atividade específica em um aplicativo):

adb shell am start -n adb shell am start <package>/<activity>

Por exemplo, para iniciar o Waze:

adb shell am start -n adb shell am start com.waze/com.waze.FreeMapAppActivity

Cópia de segurança #

Você pode usar o comando adb backup para fazer backup do seu dispositivo.

adb backup [-f <file>] [-apk|-noapk] [-obb|-noobb] [-shared|-noshared] [-all] 
           [-system|nosystem] [<packages...>]

-f <filename> especifica o nome do arquivo default: cria backup.ab no diretório atual

-apk|noapk habilita/desabilita o backup dos próprios .apks padrão: -noapk

-obb|noobb habilita/desabilita backup de arquivos adicionais padrão: -noobb

Armazenamento compartilhado/conteúdo do cartão SD do dispositivo de backup -shared|nosshared padrão: -noshared

-all faz backup de todos os aplicativos instalados

-system|nosystem inclui aplicativos do sistema padrão: -system

<packages> uma lista de pacotes para backup (por exemplo, com.example.android.myapp) (não é necessário se -all for especificado)

Para um backup completo do dispositivo, incluindo tudo, use

adb backup -apk -obb -shared -all -system -f fullbackup.ab

Observação: fazer um backup completo pode levar muito tempo.

Para restaurar um backup, use

adb restore backup.ab

Visualize os dados internos de um aplicativo (data/data/<sample.package.id>) em um dispositivo #

Primeiro, verifique se o backup do seu aplicativo pode ser feito em AndroidManifest.xml, ou seja, android:allowBackup não é false.

Comando de backup:

adb -s <device_id> backup -noapk <sample.package.id>

Crie um tar com o comando dd:

dd if=backup.ab bs=1 skip=24 | python -c "import zlib,sys;sys.stdout.write(zlib.decompress(sys.stdin.read()))" > backup.tar

Extraia o alcatrão:

tar -xvf backup.tar

Você pode então visualizar o conteúdo extraído.

Instale o ADB no sistema Linux #

Como instalar o Android Debugging Bridge (ADB) em um sistema Linux com o terminal usando os repositórios da sua distribuição.

Instale no sistema Ubuntu/Debian via apt:

sudo apt-get update
sudo apt-get install adb

Instale no sistema Fedora/CentOS via yum:

sudo yum check-update
sudo yum install android-tools

Instale no sistema Gentoo com o portage:

sudo emerge --ask dev-util/android-tools

Instale no sistema openSUSE com o zypper:

sudo zypper refresh
sudo zypper install android-tools

Instale no sistema Arch com o pacman:

sudo pacman -Syyu
sudo pacman -S android-tools

Ler informações do dispositivo #

Escreva o seguinte comando no seu terminal:

adb shell getprop

Isso imprimirá todas as informações disponíveis na forma de pares chave/valor.

Você pode apenas ler informações específicas anexando o nome de uma chave específica ao comando. Por exemplo:

adb shell getprop ro.product.model

Aqui estão algumas informações interessantes que você recebe:

  • ro.product.model: nome do modelo do dispositivo (por exemplo, Nexus 6P)
  • ro.build.version.sdk: Nível de API do dispositivo (por exemplo, 23)
  • ro.product.brand: Marca do dispositivo (por exemplo, Samsung)

Reinicialize o dispositivo #

Você pode reiniciar seu dispositivo executando o seguinte comando:

adb reboot

Execute este comando para reinicializar no carregador de inicialização:

adb reboot bootloader

Reinicie no modo de recuperação:

adb reboot recovery

Esteja ciente de que o dispositivo não será desligado primeiro!
