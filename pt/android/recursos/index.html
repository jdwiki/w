Defina as cores #

As cores geralmente são armazenadas em um arquivo de recurso chamado colors.xml na pasta /res/values/.

Eles são definidos por elementos <color>:

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <color name="colorPrimary">#3F51B5</color>
    <color name="colorPrimaryDark">#303F9F</color>
    <color name="colorAccent">#FF4081</color>

    <color name="blackOverlay">#66000000</color>
</resources>

As cores são representadas por valores de cor hexadecimais para cada canal de cor (0 - FF) em um dos formatos:

  • #RGB
  • #ARGB
  • #RRGGBB
  • #AARRGGBB

Lenda

  • A - canal alfa - o valor 0 é totalmente transparente, o valor FF é opaco
  • R - canal vermelho
  • G - canal verde
  • B - canal azul

Cores definidas podem ser usadas em XML com a seguinte sintaxe @color/name_of_the_color

Por exemplo:

<RelativeLayout
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:background="@color/blackOverlay">

Usando cores no código

Esses exemplos assumem que this é uma referência de atividade. Uma referência de contexto também pode ser usada em seu lugar.

int color = ContextCompat.getColor(this, R.color.black_overlay);
view.setBackgroundColor(color);

int color = this.getResources().getColor(this, R.color.black_overlay);
view.setBackgroundColor(color);

Na declaração acima colorPrimary, colorPrimaryDark e colorAccent são usados ​​para definir as cores do Material Design que serão usadas na definição do tema Android personalizado em styles.xml. Eles são adicionados automaticamente quando um novo projeto é criado com o Android Studio.

Definir strings #

Strings são normalmente armazenadas no arquivo de recursos strings.xml. Eles são definidos usando um elemento XML <string>.

O objetivo do strings.xml é permitir a internacionalização. Você pode definir um strings.xml para cada código ISO de idioma. Assim, quando o sistema procura a string ‘app_name’, ele primeiro verifica o arquivo xml correspondente ao idioma atual e, se não for encontrado, procura a entrada no arquivo strings.xml padrão. Isso significa que você pode optar por localizar apenas algumas de suas strings, enquanto outras não.

/res/values/strings.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <string name="app_name">Hello World App</string>
    <string name="hello_world">Hello World!</string>
</resources>

Depois que uma string é definida em um arquivo de recurso XML, ela pode ser usada por outras partes do aplicativo.

Os arquivos de projeto XML de um aplicativo podem usar um elemento <string> referindo-se a @string/string_name. Por exemplo, o arquivo manifest (/manifests/AndroidManifest.xml) de um aplicativo inclui a seguinte linha por padrão no Android Studio:

android:label="@string/app_name"

Isso diz ao Android para procurar um recurso <string> chamado “app_name” para usar como o nome do aplicativo quando ele for instalado ou exibido em um iniciador.

Outra vez que você usaria um recurso <string> de um arquivo XML no Android estaria em um arquivo de layout. Por exemplo, o seguinte representa um TextView que exibe a string hello_world que definimos anteriormente:

<TextView
    android:layout_width="wrap_content"
    android:layout_height="wrap_content"
    android:text="@string/hello_world"/>

Você também pode acessar os recursos <string> da parte java do seu aplicativo. Para recuperar nossa mesma string hello_world de cima dentro de uma classe Activity, use:

String helloWorld = getString(R.string.hello_world);

Definir plurais de string #

Para diferenciar entre strings plurais e singulares, você pode definir um plural em seu arquivo strings.xml e listar as diferentes quantidades, conforme mostrado no exemplo abaixo:

<?xml version="1.0" encoding="utf-8"?>
<resources>  
    <plurals name="hello_people">
        <item quantity="one">Hello to %d person</item>
        <item quantity="other">Hello to %d people</item>
    </plurals>
</resources>

Essa definição pode ser acessada a partir do código Java usando o método getQuantityString() da classe Resources, conforme mostrado no exemplo a seguir:

getResources().getQuantityString(R.plurals.hello_people, 3, 3);

Aqui, o primeiro parâmetro R.plurals.hello_people é o nome do recurso. O segundo parâmetro (3 neste exemplo) é usado para escolher a string quantity correta. O terceiro parâmetro (também 3 neste exemplo) é o argumento de formato que será usado para substituir o especificador de formato %d.

Os valores de quantidade possíveis (listados em ordem alfabética) são:

few
many
one
other
two
zero

É importante notar que nem todas as localidades suportam todas as denominações de ‘quantidade’. Por exemplo, o idioma chinês não tem o conceito de ‘um’ item. O inglês não tem um item zero, pois é gramaticalmente o mesmo que other. Instâncias não suportadas de quantity serão sinalizadas pelo IDE como avisos de Lint, mas não causarão erros de complicação se forem usadas.

Definir dimensões #

As dimensões geralmente são armazenadas em um arquivo de recurso chamado dimens.xml. Eles são definidos usando um elemento <dimen>.

res/values/dimens.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <dimen name="small_padding">5dp</dimen>
    <dimen name="medium_padding">10dp</dimen>
    <dimen name="large_padding">20dp</dimen>

    <dimen name="small_font">14sp</dimen>
    <dimen name="medium_font">16sp</dimen>
    <dimen name="large_font">20sp</dimen>
</resources>

Você pode usar unidades diferentes:

  • sp : Pixels independentes de escala. Para fontes.
  • dp : Pixels independentes de densidade. Para todo o resto.
  • pt: Pontos
  • px : Pixels
  • mm: Milímetros
  • im: polegadas

Dimensões agora podem ser referenciadas em XML com a sintaxe @dimen/name_of_the_dimension.

Por exemplo:

<RelativeLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:padding="@dimen/large_padding">
</RelativeLayout>

Definir array inteiro #

Para definir uma matriz de inteiros, escreva em um arquivo de recursos

res/values/filename.xml

<integer-array name="integer_array_name">
    <item>integer_value</item>
    <item>@integer/integer_id</item>
</integer-array>

por exemplo

res/values/arrays.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <integer-array name="fibo">
        <item>@integer/zero</item>
        <item>@integer/one</item>
        <item>@integer/one</item>
        <item>@integer/two</item>
        <item>@integer/three</item>
        <item>@integer/five</item>
    </integer-array>
</resources>

e use-o de java como

int[] values = getResources().getIntArray(R.array.fibo);
Log.i("TAG",Arrays.toString(values)));

Resultado

I/TAG: [0, 1, 1, 2, 3, 5]

Formatação de strings em strings.xml #

Definir Strings no arquivo strings.xml também permite a formatação de strings. A única ressalva é que a String precisará ser tratada em código como abaixo, em vez de simplesmente anexá-la a um layout.

<string name="welcome_trainer">Hello Pokémon Trainer, %1$s! You have caught %2$d Pokémon.</string>

String welcomePokemonTrainerText = getString(R.string.welcome_trainer, tranerName, pokemonCount);

No exemplo acima,
%1$s
‘%’ separa dos caracteres normais,
‘1’ denota o primeiro parâmetro,
’$’ é usado como separador entre o número do parâmetro e o tipo,
’s’ denota tipo de string (’d’ é usado para inteiro)

Observe que getString() é um método de Context ou Resources, ou seja, você pode usá-lo diretamente dentro de uma instância de Activity, ou então você pode usar getActivity().getString() ou getContext( ).getString() respectivamente.

Definir uma lista de estados de cores #

As listas de estado de cores podem ser usadas como cores, mas serão alteradas dependendo do estado da exibição para a qual são usadas.

Para definir um, crie um arquivo de recurso em res/color/foo.xml

<?xml version="1.0" encoding="utf-8"?>
<selector xmlns:android="http://schemas.android.com/apk/res/android">
    <item android:color="#888888" android:state_enabled="false"/>
    <item android:color="@color/lightGray" android:state_selected="false"/>
    <item android:color="@android:color/white" />
</selector>

Os itens são avaliados na ordem em que são definidos e o primeiro item cujos estados especificados correspondem ao estado atual da exibição é usado. Portanto, é uma boa prática especificar um catch-all no final, sem nenhum seletor de estado especificado.

Cada item pode usar um literal de cor ou fazer referência a uma cor definida em outro lugar.

9 Patches #

9 Patches são imagens esticáveis nas quais as áreas que podem ser esticadas são definidas por marcadores pretos em uma borda transparente.

Há um ótimo tutorial aqui. Apesar de ser tão antigo, ainda é tão valioso e ajudou muitos de nós a entender profundamente o equipamento de 9 patches.

Infelizmente, recentemente essa página foi desativada por um tempo (está no ar novamente).

Daí a necessidade de ter uma cópia física dessa página para desenvolvedores Android em nossos servidores confiáveis.

Aqui está.

UM GUIA SIMPLES PARA 9-PATCH PARA ANDROID UI 18 de maio de 2011

Enquanto eu estava trabalhando no meu primeiro aplicativo Android, achei o 9-patch (também conhecido como 9.png) confuso e mal documentado. Depois de um tempo, finalmente entendi como funciona e decidi juntar algo para ajudar os outros a descobrir.

Basicamente, 9-patch usa transparência png para fazer uma forma avançada de 9-slice ou scale9. As guias são linhas pretas retas de 1 pixel desenhadas na borda da imagem que definem o dimensionamento e o preenchimento da imagem. Ao nomear seu arquivo de imagem name.9.png, o Android reconhecerá o formato 9.png e usará as guias pretas para dimensionar e preencher seus bitmaps.

Aqui está um mapa de guia básico:

Como você pode ver, você tem guias em cada lado da sua imagem. As guias TOP e LEFT são para dimensionar sua imagem (ou seja, 9 fatias), enquanto as guias RIGHT e BOTTOM definem a área de preenchimento.

As linhas guia pretas são cortadas/removidas da sua imagem – elas não serão exibidas no aplicativo. As guias devem ter apenas um pixel de largura, portanto, se você quiser um botão de 48 × 48, seu png será na verdade 50 × 50. Qualquer coisa mais espessa do que um pixel continuará a fazer parte da sua imagem. (Meus exemplos têm guias de largura de 4 pixels para melhor visibilidade. Eles devem ter apenas 1 pixel).

Suas guias devem ser pretas sólidas (#000000). Mesmo uma pequena diferença de cor (#000001) ou alfa fará com que ele falhe e estique normalmente. Essa falha também não será óbvia*, ela falha silenciosamente! Sim. Sério. Agora você sabe.

Além disso, você deve ter em mente que a área restante do contorno de um pixel deve ser completamente transparente. Isso inclui os quatro cantos da imagem – eles devem estar sempre claros. Isso pode ser um problema maior do que você imagina. Por exemplo, se você dimensionar uma imagem no Photoshop, ela adicionará pixels com suavização de serrilhado que podem incluir pixels quase invisíveis que também causarão falhas*. Se você precisar dimensionar no Photoshop, use a configuração Vizinho mais próximo no menu suspenso Reamostrar imagem (na parte inferior do menu pop-up Tamanho da imagem) para manter as bordas nítidas em suas guias.

*(atualizado em 1/2012) Esta é realmente uma “correção” no kit de desenvolvimento mais recente. Anteriormente, ele se manifestava como todas as suas outras imagens e recursos quebrando repentinamente, não a imagem de 9 patches realmente quebrada.

As guias TOP e LEFT são usadas para definir a parte escalável da sua imagem – LEFT para dimensionar a altura, TOP para dimensionar a largura. Usando uma imagem de botão como exemplo, isso significa que o botão pode se esticar horizontal e verticalmente dentro da parte preta e todo o resto, como os cantos, permanecerá do mesmo tamanho. O permite que você tenha botões que podem ser dimensionados para qualquer tamanho e manter uma aparência uniforme.

É importante observar que as imagens de 9 patches não são reduzidas – elas apenas aumentam. Portanto, é melhor começar o mais pequeno possível.

Além disso, você pode deixar de fora partes no meio da linha da escala. Assim, por exemplo, se você tiver um botão com uma borda brilhante afiada no meio, poderá deixar alguns pixels no meio da guia ESQUERDA. O eixo horizontal central da sua imagem não será dimensionado, apenas as partes acima e abaixo dela, para que seu brilho nítido não fique com suavização de serrilhado ou difuso.

As guias de área de preenchimento são opcionais e fornecem uma maneira de definir a área para itens como seu rótulo de texto. O preenchimento determina quanto espaço há na imagem para colocar texto, ícone ou outras coisas. 9-patch não é apenas para botões, também funciona para imagens de fundo.

O exemplo de botão e rótulo acima é exagerado simplesmente para explicar a ideia de preenchimento – o rótulo não é completamente preciso. Para ser honesto, não experimentei como o Android faz rótulos de várias linhas, pois um rótulo de botão geralmente é uma única linha de texto.

Por fim, aqui está uma boa demonstração de como as guias de escala e preenchimento podem variar, como um LinearLayout com uma imagem de fundo e lados totalmente arredondados:

Com este exemplo, o guia LEFT não é usado, mas ainda precisamos ter um guia. A imagem de fundo não é dimensionada verticalmente; apenas dimensiona horizontalmente (com base no guia TOP). Observando as guias de preenchimento, as guias DIREITA e INFERIOR se estendem além de onde encontram as bordas curvas da imagem. Isso me permite colocar meus botões redondos perto das bordas do plano de fundo para uma aparência justa e ajustada.

Então é isso. 9-patch é super fácil, uma vez que você consiga. Não é uma maneira perfeita de fazer dimensionamento, mas as guias de dimensionamento de área de preenchimento e de várias linhas oferecem mais flexibilidade do que as tradicionais 9 fatias e dimensionamento9. Experimente e você descobrirá rapidamente.

Definir inteiros #

Os inteiros são normalmente armazenados em um arquivo de recurso chamado integers.xml, mas o nome do arquivo pode ser escolhido arbitrariamente. Cada inteiro é definido usando um elemento <integer>, conforme mostrado no arquivo a seguir:

res/values/integers.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <integer name="max">100</integer>
</resources>

<ProgressBar

Defina um recurso de menu e use-o dentro de Activity/Fragment #

Defina um menu em res/menu

<?xml version="1.0" encoding="utf-8"?>
<menu
    xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:app="http://schemas.android.com/apk/res-auto">

    <item
        android:id="@+id/first_item_id"
        android:orderInCategory="100"
        android:title="@string/first_item_string"
        android:icon="@drawable/first_item_icon"
        app:showAsAction="ifRoom"/>

    <item
        android:id="@+id/second_item_id"
        android:orderInCategory="110"
        android:title="@string/second_item_string"
        android:icon="@drawable/second_item_icon"
        app:showAsAction="ifRoom"/>

</menu>

Para mais opções de configuração, consulte: Recurso de menu

Dentro da Atividade:

@Override
public void onCreateOptionsMenu(Menu menu, MenuInflater inflater) {
    ///Override defining menu resource
    inflater.inflate(R.menu.menu_resource_id, menu);
    super.onCreateOptionsMenu(menu, inflater);
}

@Override
public void onPrepareOptionsMenu(Menu menu) {
    //Override for preparing items (setting visibility, change text, change icon...)
    super.onPrepareOptionsMenu(menu);
}

@Override
public boolean onOptionsItemSelected(MenuItem item) {
    //Override it for handling items
    int menuItemId = item.getItemId();
    switch (menuItemId) {
        case: R.id.first_item_id
             return true; //return true, if is handled
    }
    return super.onOptionsItemSelected(item);
}

Para invocar os métodos acima durante a exibição da visualização, chame getActivity().invalidateOptionsMenu();

Dentro do Fragment é necessária uma chamada adicional:

@Nullable
@Override
public View onCreateView(LayoutInflater inflater, ViewGroup container, Bundle savedInstanceState) {
    setHasOptionsMenu(true);
    super.onCreateView(inflater, container, savedInstanceState);
}

Trabalhando com o arquivo strings.xml #

Um recurso de string fornece strings de texto para seu aplicativo com estilo e formatação de texto opcionais. Existem três tipos de recursos que podem fornecer strings ao seu aplicativo:

Corda

XML resource that provides a single string.

Sintaxe:

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <string name="string_name">text_string</string>
</resources>

E para usar essa string no layout:

<TextView
    android:layout_width="fill_parent"
    android:layout_height="wrap_content"
    android:text="@string/string_name" />

Matriz de cadeias

XML resource that provides an array of strings.

Sintaxe:

    <resources>
<string-array name="planets_array">
    <item>Mercury</item>
    <item>Venus</item>
    <item>Earth</item>
    <item>Mars</item>
</string-array>

Uso

Resources res = getResources();
String[] planets = res.getStringArray(R.array.planets_array);

Strings de quantidade (plural)

XML resource that carries different strings for pluralization.

Sintaxe:

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <plurals
        name="plural_name">
        <item
            quantity=["zero" | "one" | "two" | "few" | "many" | "other"]
            >text_string</item>
    </plurals>
</resources>

Uso:

int count = getNumberOfsongsAvailable();
Resources res = getResources();
String songsFound = res.getQuantityString(R.plurals.plural_name, count, count);

Traduzir uma string #

Strings podem ser internacionalizadas definindo um strings.xml diferente para cada idioma suportado.

Você adiciona um novo idioma criando um novo diretório de valores com o código de idioma ISO como sufixo. Por exemplo, ao adicionar um conjunto alemão, sua estrutura pode ter a seguinte aparência:

Quando o sistema procura a string solicitada, ele verifica primeiro o xml específico do idioma, se não for encontrado, o valor do arquivo strings.xml padrão é retornado. A chave permanece a mesma para cada idioma e apenas o valor muda.

Exemplo de conteúdo:

/res/values/strings.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <string name="app_name">HelloWorld</string>
    <string name="hello_world">Hello World!</string>
</resources>

/res/values-fr/strings.xml

<?xml version="1.0" encoding="utf-8"?>
<resources>
    <string name="hello_world">Bonjour tout le monde !!!</string>
</resources>

Importar array de objetos definidos em recursos #

Há casos em que objetos personalizados precisam ser criados e definidos nos recursos da aplicação. Tais objetos podem ser compostos de tipos simples Java, por exemplo Integer, Float, String.

Aqui está o exemplo de como importar um objeto definido nos recursos do aplicativo. O objeto Category contém 3 propriedades da categoria:

  • EU IRIA
  • Cor
  • Nome

Este POJO tem seu equivalente no arquivo categories.xml, onde cada array possui as mesmas propriedades definidas para cada categoria.

  1. Crie um modelo para seu objeto:

    public class Category { private Type id; private @ColorRes int color; private @StringRes String name;

     public Category getId() {
     return id;
 }

 public void setId(Category id) {
     this.id = id;
 }

 public String getName() {
     return name;
 }

 public void setName(String name) {
     this.name = name;
 }

 public int getColor() {
     return color;
 }

 public void setColor(int color) {
     this.color = color;
 }

 public enum Type{
     REGISTRATION,
     TO_ACCEPT,
     TO_COMPLETE,
     TO_VERIFY,
     CLOSED
 }

    }

  2. Crie o arquivo na pasta res/values:

categorias.xml

  1. Compor cada modelo que consiste em recursos:

    0 @android:color/transparent @string/statusRegistration 1 @color/light_gray @string/acceptance 2 @color/material_green_500 @string/open 3 @color/material_gray_800 @string/verification 4 @android:color/black @string/closed

  2. Defina uma matriz no arquivo de recursos:

     <array name="categories">
     <item>@array/no_action</item>
     <item>@array/to_accept</item>
     <item>@array/opened</item>
     <item>@array/to_verify</item>
     <item>@array/to_close</item>
 </array>

     // Get the properties of one object
