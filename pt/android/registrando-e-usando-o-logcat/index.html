Parâmetros #

Opção Descrição
-b (tampão) Carrega um buffer de log alternativo para visualização, como eventos ou rádio. O buffer principal é usado por padrão. Consulte Visualizando Buffers de Log Alternativos.
-c Limpa (limpa) todo o log e sai.
-d Despeja o log na tela e sai.
-f (nome do arquivo) Grava a saída da mensagem de log em (nome do arquivo). O padrão é stdout.
-g Imprime o tamanho do buffer de log especificado e sai.
-n (contagem) Define o número máximo de logs girados para (contagem). O valor padrão é 4. Requer a opção -r.
-r (kbytes) Gira o arquivo de log a cada (kbytes) de saída. O valor padrão é 16. Requer a opção -f.
-s Define a especificação de filtro padrão como silenciosa.
-v (formato) Define o formato de saída para mensagens de log. O padrão é o formato breve.

Definição #

Logcat é uma ferramenta de linha de comando que despeja um log de mensagens do sistema, incluindo rastreamentos de pilha quando o dispositivo gera um erro e mensagens que você gravou de seu aplicativo com a classe Log.

Quando usar #

Se você está pensando em usar os métodos System.out do Java para imprimir no console em vez de usar um dos métodos Log do Android, saiba que eles funcionam basicamente da mesma forma. No entanto, é melhor evitar o uso dos métodos do Java porque as informações extras e a formatação fornecidas pelos métodos de log do Android são mais benéficas. Além disso, os métodos de impressão System.out são redirecionados para o método Log.i().

Filtrando a saída do logcat #

É útil filtrar a saída do logcat porque há muitas mensagens que não são de interesse. Para filtrar a saída, abra o “Android Monitor” e clique no menu suspenso no canto superior direito e selecione Edit Filter Configuration

Agora você pode adicionar filtros personalizados para mostrar mensagens que são de seu interesse, bem como filtrar linhas de log conhecidas que podem ser ignoradas com segurança. Para ignorar uma parte da saída você pode definir uma Expressão Regular. Aqui está um exemplo de exclusão de tags correspondentes:

^(?!(HideMe|AndThis))

Isso pode ser inserido seguindo este exemplo:

O acima é uma expressão regular que exclui entradas. Se você quiser adicionar outra tag à lista negra, adicione-a após um caractere pipe |. Por exemplo, se você quisesse colocar “GC” na lista negra, você usaria um filtro como este:

^(?!(HideMe|AndThis|GC))

Para obter mais documentação e exemplos, visite Logging and using Logcat

Registro Básico

A classe `Log` é a principal fonte de escrita de logs do desenvolvedor, especificando uma `tag` e uma `message`. A tag é o que você pode usar para filtrar mensagens de log para identificar quais linhas vêm de sua atividade específica. Basta ligar 
Log.v(String tag, String msg);

E o sistema Android escreverá uma mensagem no logcat:

07-28 12:00:00.759 24812-24839/my.packagename V/MyAnimator: Some log messages
 └ time stamp             |  app.package┘     |    └ any tag  |
     process & thread ids ┘          log level┘               └ the log message

DICA:
Observe o id do processo e o id do thread. Se eles forem os mesmos - o log está vindo do thread principal/UI!

Qualquer tag pode ser usada, mas é comum usar o nome da classe como tag:

public static final String tag = MyAnimator.class.getSimpleName();

Níveis de registro

O logger do Android tem 6 níveis diferentes, cada um com uma determinada finalidade: - `ERRO`: `Log.e()` - Used to indicate critical failure, this is the level printed at when throwing an `Exception`. - `WARN`: `Log.w()` - Used to indicate a warning, mainly for recoverable failures - `INFO`: `Log.i()` - Used to indicate higher-level information about the state of the application - `DEBUG`: `Log.d()` - Used to log information that would be useful to know when debugging the application, but would get in the way when running the application - `VERBOSE`: `Log.v()` - Used to log information that reflects the small details about the state of the application - `ASSERT`: `Log.wtf()` - Used to log information about a condition that should never happen. - *wtf* stands for "What a Terrible Failure". ---

Motivação para registro

A motivação para o registro é encontrar facilmente erros, avisos e outras informações, observando a cadeia de eventos do aplicativo. Por exemplo, imagine um aplicativo que lê linhas de um arquivo de texto, mas assume incorretamente que o arquivo nunca estará vazio. O rastreamento de log (de um aplicativo que não registra) ficaria assim: 
E/MyApplication: Process: com.example.myapplication, PID: 25788
                          com.example.SomeRandomException: Expected string, got 'null' instead

Seguido por um monte de rastreamentos de pilha que eventualmente levariam à linha incorreta, onde passar por um depurador acabaria levando ao problema

No entanto, o rastreamento de log de um aplicativo com log ativado pode ser algo assim:

V/MyApplication: Looking for file myFile.txt on the SD card
D/MyApplication: Found file myFile.txt at path <path>
V/MyApplication: Opening file myFile.txt
D/MyApplication: Finished reading myFile.txt, found 0 lines
V/MyApplication: Closing file myFile.txt
...
E/MyApplication: Process: com.example.myapplication, PID: 25788
                          com.example.SomeRandomException: Expected string, got 'null' instead

Uma rápida olhada nos logs e é óbvio que o arquivo estava vazio.

Aspectos a serem considerados ao registrar:

Embora o registro em log seja uma ferramenta poderosa que permite que os desenvolvedores do Android obtenham uma visão maior do funcionamento interno de seu aplicativo, o registro tem algumas desvantagens.

Legibilidade do registro:

É comum que os aplicativos Android tenham vários logs sendo executados de forma síncrona. Como tal, é muito importante que cada log seja facilmente legível e contenha apenas informações relevantes e necessárias.

Desempenho:

O log requer uma pequena quantidade de recursos do sistema. Em geral, isso não é motivo de preocupação, no entanto, se usado em excesso, o registro em log pode ter um impacto negativo no desempenho do aplicativo.

Segurança:

Recentemente, vários aplicativos Android foram adicionados ao mercado do Google Play que permitem ao usuário visualizar os logs de todos os aplicativos em execução. Essa exibição não intencional de dados pode permitir que os usuários visualizem informações confidenciais. Como regra geral, sempre remova os logs que contenham dados não públicos *antes* de publicar seu aplicativo no mercado.

Conclusão:

O registro é uma parte essencial de um aplicativo Android, devido ao poder que ele dá aos desenvolvedores. A capacidade de criar um rastreamento de log útil é um dos aspectos mais desafiadores do desenvolvimento de software, mas a classe Log do Android ajuda a torná-lo muito mais fácil.

Para obter mais documentação e exemplos, visite Logging and using Logcat

Usando o Logcat #

Logcat é uma ferramenta de linha de comando que despeja um log de mensagens do sistema, incluindo rastreamentos de pilha quando o dispositivo gera um erro e mensagens que você gravou de seu aplicativo com a classe Log.

A saída do Logcat pode ser exibida no Android Monitor do Android Studio ou com a linha de comando adb.

No Android Studio

Mostre clicando no ícone “Monitor Android”: Ou pressionando Alt+6 no Windows/Linux ou CMD+6 no Mac.

via linha de comando:

Uso simples:

$ adb logcat

Com carimbos de data/hora:

$ adb logcat -v time

Filtre por texto específico:

$ adb logcat -v time | grep 'searchtext'

Existem muitas opções e filtros disponíveis para logcat de linha de comando, documentado aqui.
Um exemplo simples, mas útil, é a seguinte expressão de filtro que exibe todas as mensagens de log com nível de prioridade “erro”, em todas as tags:

$ adb logcat *:E

Log com link para fonte diretamente do Logcat #

Este é um bom truque para adicionar um link ao código, então será fácil pular para o código que emitiu o log.

Com o seguinte código, esta chamada:

MyLogger.logWithLink("MyTag","param="+param);


Resultará em:

07-26...012/com.myapp D/MyTag: MyFrag:onStart(param=3)  (MyFrag.java:2366) // << logcat converts this to a link to source!

Este é o código (dentro de uma classe chamada MyLogger):

static StringBuilder sb0 = new StringBuilder(); // reusable string object

public static void logWithLink(String TAG, Object param) {
    StackTraceElement stack = Thread.currentThread().getStackTrace()[3];
    sb0.setLength(0);
    String c = stack.getFileName().substring(0, stack.getFileName().length() - 5); // removes the ".java"
    sb0.append(c).append(":");
    sb0.append(stack.getMethodName()).append('(');
    if (param != null) {
        sb0.append(param);
    }
    sb0.append(") ");
    sb0.append(" (").append(stack.getFileName()).append(':').append(stack.getLineNumber()).append(')');
    Log.d(TAG, sb0.toString());
}

Este é um exemplo básico, pode ser facilmente estendido para emitir um link para o chamador (dica: a pilha será 4 em vez de 3), e você também pode adicionar outras informações relevantes.

Gerando código de registro #

Os Modelos ao vivo do Android Studio podem oferecer alguns atalhos para registro rápido.
Para usar os modelos Live, tudo o que você precisa fazer é começar a digitar o nome do modelo e pressionar ‘TAB’ ou enter para inserir a instrução.

Exemplos:

  • logi → se transforma em → android.util.Log.i(TAG, "$METHOD_NAME$: $content$");
    • $METHOD_NAME$ will automatically be replaced with your method name, and the cursor will wait for the content to be filled.
  • loge → mesmo, para erro
  • etc. para o resto dos níveis de registro.

A lista completa de modelos pode ser encontrada nas configurações do Android Studio (ALT+s e digite “live”). E é possível adicionar seus modelos personalizados também.

Se você achar que os “Modelos ao vivo” do Android Studio não são suficientes para suas necessidades, você pode considerar Plugin Android Postfix

Esta é uma biblioteca muito útil que ajuda você a evitar escrever a linha de registro manualmente.

A sintaxe é absolutamente simples:

.log - Registro. Se houver a variável constante “TAG”, ela usa “TAG” . Caso contrário, use o nome da classe.
