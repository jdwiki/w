Criando um novo projeto #

Configure o Android Studio #

Comece [configurando o Android Studio](https://www.wikiod.com/pt/android/introducao-ao-android#Configurando o Android Studio) e abra-o. Agora, você está pronto para fazer seu primeiro aplicativo Android!

Observação: este guia é baseado no Android Studio 2.2, mas o processo em outras versões é basicamente o mesmo.

Configure seu projeto #

Configuração básica #

Você pode iniciar um novo projeto de duas maneiras:

  • Clique em “Iniciar um novo projeto do Android Studio” na tela de boas-vindas.
  • Navegue até Arquivo → Novo Projeto se você já tiver um projeto aberto.

Em seguida, você precisa descrever sua aplicação preenchendo alguns campos:

  1. Nome do Aplicativo - Este nome será mostrado ao usuário.

    Example: Hello World. You can always change it later in AndroidManifest.xml file.

  2. Domínio da Empresa - Este é o qualificador para o nome do pacote do seu projeto.

    Example: stackoverflow.com.

  3. Nome do pacote (também conhecido como applicationId) - Este é o nome do pacote do projeto totalmente qualificado.

    It should follow Reverse Domain Name Notation (aka Reverse DNS): Top Level Domain . Company Domain . [Company Segment .] Application Name.

    Example: com.stackoverflow.android.helloworld or com.stackoverflow.helloworld. You can always change your applicationId by overriding it in your gradle file.

Não use o prefixo padrão “com.example”, a menos que você não pretenda enviar seu aplicativo para a Google Play Store. O nome do pacote será seu applicationId exclusivo no Google Play.

  1. Project Location - Este é o diretório onde seu projeto será armazenado.

Selecione os fatores de forma e o nível da API #

A próxima janela permite selecionar os fatores de forma compatíveis com seu aplicativo, como telefone, tablet, TV, Wear e Google Glass. Os fatores de forma selecionados tornam-se os módulos do aplicativo dentro do projeto. Para cada fator de forma, você também pode selecionar o nível de API desse aplicativo. Para obter mais informações, clique em Ajude-me a escolher

Gráfico das distribuições atuais da versão do Android, mostrado quando você clica em Ajude-me a escolher.

A janela Android Platform Distribution mostra a distribuição de dispositivos móveis executando cada versão do Android, conforme mostrado na Figura 2. Clique em um nível de API para ver uma lista de recursos introduzidos na versão correspondente do Android. Isso ajuda você a escolher o nível mínimo de API que tem todos os recursos de que seus aplicativos precisam, para que você possa alcançar o maior número possível de dispositivos. Em seguida, clique em OK.

Agora, escolha quais plataformas e versão do Android SDK o aplicativo suportará.

Por enquanto, selecione apenas Telefone e Tablet.

O SDK mínimo é o limite inferior para seu aplicativo. É um dos sinais que a Google Play Store usa para determinar em quais dispositivos um aplicativo pode ser instalado. Por exemplo, aplicativo do Stack Exchange é compatível com Android 4.1+.

O Android Studio informará (aproximadamente) qual porcentagem de dispositivos será compatível com o SDK mínimo especificado.

Níveis de API mais baixos visam mais dispositivos, mas têm menos recursos disponíveis.

Ao decidir sobre o SDK mínimo, você deve considerar as [estatísticas dos painéis][8], que fornecerão informações de versão sobre os dispositivos que visitaram a Google Play Store globalmente na última semana.

[![Captura de tela da distribuição android atual][9]][9]

De: Dashboards no site do desenvolvedor Android.

Adicionar uma atividade #

Agora vamos selecionar uma atividade padrão para nosso aplicativo. No Android, uma [Activity][10] é uma tela única que será apresentada ao usuário. Um aplicativo pode abrigar várias atividades e navegar entre elas. Para este exemplo, escolha ‘Empty Activity’ e clique em próximo.

Aqui, se desejar, você pode alterar o nome da atividade e o layout. Uma boa prática é manter Activity como sufixo para o nome da atividade e activity_ como prefixo para o nome do layout. Se deixarmos isso como padrão, o Android Studio gerará uma atividade para nós chamada MainActivity e um arquivo de layout chamado activity_main. Agora clique em Concluir.

O Android Studio criará e configurará nosso projeto, o que pode levar algum tempo dependendo do sistema.

Inspecionando o Projeto #

Para entender como o Android funciona, vamos dar uma olhada em alguns dos arquivos que foram criados para nós.

No painel esquerdo do Android Studio, podemos ver a [estrutura do nosso aplicativo Android][11].

[![Estrutura básica do aplicativo][12]][12]

Primeiro, vamos abrir AndroidManifest.xml clicando duas vezes nele. O arquivo de manifesto do Android descreve algumas das informações básicas sobre um aplicativo Android. Ele contém a declaração de nossas atividades, bem como alguns componentes mais avançados.

Se um aplicativo precisa de acesso a um recurso protegido por uma permissão, ele deve declarar que requer essa permissão com um elemento <uses-permission> no manifesto. Então, quando o aplicativo é instalado no dispositivo, o instalador determina se deve ou não conceder a permissão solicitada verificando as autoridades que assinaram os certificados do aplicativo e, em alguns casos, solicitando ao usuário. Um aplicativo também pode proteger seus próprios componentes (atividades, serviços, receptores de transmissão e provedores de conteúdo) com permissões. Ele pode empregar qualquer uma das permissões definidas pelo Android (listadas em android.Manifest.permission) ou declaradas por outros aplicativos. Ou pode definir o seu próprio.

<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.stackoverflow.helloworld">

    <application
        android:allowBackup="true"
        android:icon="@mipmap/ic_launcher"
        android:label="@string/app_name"
        android:supportsRtl="true"
        android:theme="@style/AppTheme">
        <activity android:name=".MainActivity">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />

                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>
    </application>
</manifest>

Em seguida, vamos abrir activity_main.xml que está localizado em app/src/main/res/layout/. Este arquivo contém declarações para os componentes visuais de nossa MainActivity. Você verá designer visual. Isso permite arrastar e soltar elementos no layout selecionado.

Você também pode alternar para o designer de layout xml clicando em “Texto” na parte inferior do Android Studio, como visto aqui:

[![Design de layout/guias de texto][13]][13]

<?xml version="1.0" encoding="utf-8"?>
<RelativeLayout xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:tools="http://schemas.android.com/tools"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:paddingBottom="@dimen/activity_vertical_margin"
    android:paddingLeft="@dimen/activity_horizontal_margin"
    android:paddingRight="@dimen/activity_horizontal_margin"
    android:paddingTop="@dimen/activity_vertical_margin"
    tools:context="com.stackexchange.docs.helloworld.MainActivity">

    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="Hello World!" />
</RelativeLayout>

Você verá um widget chamado TextView dentro deste layout, com a propriedade android:text definida como “Hello World!”. Este é um bloco de texto que será mostrado ao usuário quando ele executar o aplicativo.

Você pode ler mais sobre [Layouts e atributos][14].

Em seguida, vamos dar uma olhada em MainActivity. Este é o código Java que foi gerado para MainActivity.

public class MainActivity extends AppCompatActivity {

    // The onCreate method is called when an Activity starts
    // This is where we will set up our layout
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        // setContentView sets the Activity's layout to a specified XML layout
        // In our case we are using the activity_main layout
        setContentView(R.layout.activity_main);
    }
}

Conforme definido em nosso manifesto do Android, MainActivity será iniciado por padrão quando um usuário iniciar o aplicativo HelloWorld.

Por último, abra o arquivo chamado build.gradle localizado em app/. O Android Studio usa o sistema de compilação Gradle para compilar e compilar aplicativos e bibliotecas Android.

apply plugin: 'com.android.application'

android {
signingConfigs {
         applicationName {
             keyAlias 'applicationName'
             keyPassword 'password'
             storeFile file('../key/applicationName.jks')
             storePassword 'anotherPassword'
         }
     }
    compileSdkVersion 26
    buildToolsVersion "26.0.0"

    defaultConfig {
        applicationId "com.stackexchange.docs.helloworld"
        minSdkVersion 16
        targetSdkVersion 26
        versionCode 1
        versionName "1.0"
        signingConfig signingConfigs.applicationName
    }
    buildTypes {
        release {
            minifyEnabled false
            proguardFiles getDefaultProguardFile('proguard-android.txt'), 'proguard-rules.pro'
        }
    }
}

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])
    testCompile 'junit:junit:4.12'
    compile 'com.android.support:appcompat-v7:26.0.0'
}

Esse arquivo contém informações sobre a compilação e a versão do seu aplicativo, e você também pode usá-lo para adicionar dependências a bibliotecas externas. Por enquanto, não vamos fazer nenhuma alteração.

É aconselhável sempre selecionar a versão mais recente disponível para as dependências:

  • buildToolsVersion: 26.0.0
  • com.android.support:appcompat-v7: 26.0.0 (julho de 2017)
  • firebase: 11.0.4 (agosto de 2017)

compileSdkVersion

compileSdkVersion é sua maneira de dizer ao Gradle com qual versão do Android SDK compilar seu aplicativo. Usar o novo Android SDK é um requisito para usar qualquer uma das novas APIs adicionadas nesse nível.

Deve-se enfatizar que alterar seu compileSdkVersion não altera o comportamento do tempo de execução. Embora novos avisos/erros do compilador possam estar presentes ao alterar seu compileSdkVersion, seu compileSdkVersion não está incluído em seu APK: ele é usado puramente em tempo de compilação.

Portanto, é altamente recomendável que você sempre compile com o SDK mais recente. Você obterá todos os benefícios de novas verificações de compilação no código existente, evitará APIs recentemente obsoletas e estará pronto para usar novas APIs.

minSdkVersão

Se compileSdkVersion definir as APIs mais recentes disponíveis para você, minSdkVersion será o limite inferior para seu aplicativo. O minSdkVersion é um dos sinais que a Google Play Store usa para determinar em qual dos dispositivos de um usuário um aplicativo pode ser instalado.

Ele também desempenha um papel importante durante o desenvolvimento: por padrão, o lint é executado em seu projeto, avisando quando você usa qualquer API acima de sua minSdkVersion, ajudando você a evitar o problema de tempo de execução de tentar chamar uma API que não existe. Verificar a versão do sistema em tempo de execução é uma técnica comum ao usar APIs apenas em versões de plataforma mais recentes.

targetSdkVersion

targetSdkVersion é a principal forma do Android fornecer compatibilidade com versões anteriores, não aplicando mudanças de comportamento, a menos que o targetSdkVersion seja atualizado. Isso permite que você use novas APIs antes de trabalhar com as alterações de comportamento. A atualização para segmentar o SDK mais recente deve ser uma alta prioridade para todos os aplicativos. Isso não significa que você tenha que usar todos os novos recursos introduzidos nem atualizar cegamente seu targetSdkVersion sem testar.

targetSDKVersion é a versão do Android que é o limite superior para as ferramentas disponíveis. Se targetSDKVersion for menor que 23, o aplicativo não precisará solicitar permissões em tempo de execução para uma instância, mesmo se o aplicativo estiver sendo executado na API 23+. TargetSDKVersion não impede que versões do Android acima da versão do Android escolhida executem o aplicativo.

Você pode encontrar mais informações sobre o plug-in Gradle:

  • [Um exemplo básico][15]
  • [Introdução ao plugin Gradle para Android e ao wrapper][16]
  • [Introdução à configuração do build.gradle e dos métodos DSL][17]

Executando o aplicativo #

Agora, vamos executar nosso aplicativo HelloWorld. Você pode executar um dispositivo virtual Android (que pode ser configurado usando o AVD Manager no Android Studio, conforme descrito no exemplo abaixo) ou conectar seu próprio dispositivo Android por meio de um cabo USB.

Configurando um dispositivo Android #

Para executar um aplicativo do Android Studio em seu dispositivo Android, você deve habilitar USB Debugging nas Developer Options nas configurações do seu dispositivo.

Configurações > Opções do desenvolvedor > Depuração USB

Se “Opções do desenvolvedor” não estiver visível nas configurações, navegue até “Sobre o telefone” e toque no “Número da compilação” sete vezes. Isso permitirá que as Opções do desenvolvedor apareçam nas suas configurações.

Configurações > Sobre o telefone > Número da compilação

Você também pode precisar alterar a configuração do build.gradle para construir em uma versão que seu dispositivo possui.

Executando a partir do Android Studio #

Clique no botão verde Executar na barra de ferramentas na parte superior do Android Studio. Na janela exibida, selecione o dispositivo no qual você deseja executar o aplicativo (inicie um dispositivo virtual Android, se necessário, ou consulte https://www.wikiod.com/pt/android/introducao-ao-android 29572/setting-up-an-avd-android-virtual-device se você precisar configurar um) e clique em ‘OK’.

[![Botão Executar][18]][18]

Em dispositivos com Android 4.4 (KitKat) e possivelmente superior, um pop-up será exibido para autorizar a depuração USB. Clique em OK para aceitar.

O aplicativo agora será instalado e executado em seu dispositivo ou emulador Android.

Localização do arquivo APK #

Ao preparar seu aplicativo para lançamento, você configura, compila e testa uma versão de lançamento do seu aplicativo. As tarefas de configuração são diretas, envolvendo tarefas básicas de limpeza e modificação de código que ajudam a otimizar seu aplicativo. O processo de compilação é semelhante ao processo de compilação de depuração e pode ser feito usando as ferramentas JDK e Android SDK. As tarefas de teste servem como uma verificação final, garantindo que seu aplicativo funcione conforme o esperado em condições do mundo real. Quando terminar de preparar seu aplicativo para lançamento, você terá um arquivo APK assinado, que poderá distribuir diretamente aos usuários ou por meio de um mercado de aplicativos, como o Google Play.

Configurando o Android Studio #

Android Studio é o IDE de desenvolvimento Android oficialmente suportado e recomendado pelo Google. O Android Studio vem com o Android SDK Manager, que é uma ferramenta para baixar os componentes do Android SDK necessários para começar a desenvolver aplicativos.

Instalando as ferramentas Android Studio e Android SDK:

  1. Baixe e instale o Android Studio.
  2. Baixe as ferramentas SDK e as ferramentas da plataforma SDK mais recentes abrindo o Android Studio e seguindo as instruções Atualizações da ferramenta SDK do Android. Você deve instalar os pacotes estáveis ​​mais recentes disponíveis.

Se você precisar trabalhar em projetos antigos que foram criados usando versões mais antigas do SDK, talvez seja necessário fazer o download dessas versões também

Desde o Android Studio 2.2, uma cópia do OpenJDK mais recente é fornecida com a instalação e é o JDK recomendado (Java Development Kit) para todos os projetos do Android Studio. Isso remove o requisito de ter o pacote JDK da Oracle instalado. Para usar o SDK integrado, proceda da seguinte forma;

  1. Abra seu projeto no Android Studio e selecione File > Project Structure na barra de menus.
  2. Na página Localização do SDK e em Localização do JDK, marque a caixa de seleção Usar JDK incorporado.
  3. Clique em OK.

Configurar o Android Studio #

O Android Studio fornece acesso a dois arquivos de configuração por meio do menu Ajuda:

  • studio.vmoptions: Personaliza as opções da Java Virtual Machine (JVM) do Studio, como tamanho de heap e tamanho de cache. Observe que em máquinas Linux esse arquivo pode ser nomeado studio64.vmoptions, dependendo da sua versão do Android Studio.
  • idea.properties: personalize as propriedades do Android Studio, como o caminho da pasta de plug-ins ou o tamanho máximo do arquivo compatível.

Alterar/adicionar tema #

Você pode alterá-lo conforme sua preferência.Arquivo->Configurações->Editor->Cores e Fontes-> e selecione um tema.Também você pode baixar novos temas de http://color-themes.com/ Depois de ter baixado o arquivo .jar.zip, vá em File -> Import Settings... e escolha o arquivo baixado.

Compilando aplicativos #

Crie um novo projeto ou abra um projeto existente no Android Studio e pressione o botão verde Reproduzir na barra de ferramentas superior para executá-lo. Se estiver cinza, você precisa esperar um segundo para permitir que o Android Studio indexe corretamente alguns arquivos, cujo progresso pode ser visto na barra de status inferior.

Se você quiser criar um projeto a partir do shell, certifique-se de ter um arquivo local.properties, que é criado pelo Android Studio automaticamente. Se você precisar criar o projeto sem o Android Studio, precisará de uma linha começando com sdk.dir= seguida do caminho para a instalação do SDK.

Abra um shell e vá para o diretório do projeto. Digite ./gradlew aR e pressione enter. aR é um atalho para assembleRelease, que irá baixar todas as dependências para você e construir o aplicativo. O arquivo APK final estará em ProjectName/ModuleName/build/outputs/apk e será chamado de ModuleName-release.apk.

Programação Android sem IDE #

Este é um exemplo Hello World minimalista que usa apenas as ferramentas mais básicas do Android.

Requisitos e premissas #

  • Oracle JDK 1.7 ou posterior
  • Ferramentas do Android SDK (apenas as ferramentas de linha de comando)

Este exemplo assume Linux. Você pode ter que ajustar a sintaxe para sua própria plataforma.

Configurando o SDK do Android #

Depois de descompactar a versão do SDK:

  1. Instale pacotes adicionais usando o gerenciador de SDK. Don’t use android update sdk --no-ui as instructed in the bundled Readme.txt; it downloads some 30 GB of unnecessary files. Instead use the interactive SDK manager android sdk to get the recommended minimum of packages.

  2. Anexe os seguintes diretórios JDK e SDK ao seu PATH de execução. This is optional, but the instructions below assume it.

    • JDK/bin
    • SDK/platform-tools
    • SDK/tools
    • SDK/build-tools/LATEST (as installed in step 1)

  3. Crie um dispositivo virtual Android. Use the interactive AVD Manager (android avd). You might have to fiddle a bit and search for advice; the on-site instructions aren’t always helpful.

    (You can also use your own device)

  4. Execute o dispositivo:

     emulator -avd DEVICE

  5. Se a tela do dispositivo parecer bloqueada, deslize para desbloqueá-la.

    Leave it running while you code the app.

Codificando o aplicativo #

  1. Mude para um diretório de trabalho vazio.

  2. Faça o arquivo de origem:

     mkdir --parents src/dom/domain
 touch src/dom/domain/SayingHello.java

    Content:

     package dom.domain;
 import android.widget.TextView;

 public final class SayingHello extends android.app.Activity
 {
     protected @Override void onCreate( final android.os.Bundle activityState )
     {
         super.onCreate( activityState );
         final TextView textV = new TextView( SayingHello.this );
         textV.setText( "Hello world" );
         setContentView( textV );
     }
 }

  3. Adicione um manifesto:

     touch AndroidManifest.xml

    Content:

     <?xml version='1.0'?>
 <manifest xmlns:a='http://schemas.android.com/apk/res/android'
  package='dom.domain' a:versionCode='0' a:versionName='0'>
     <application a:label='Saying hello'>
         <activity a:name='dom.domain.SayingHello'>
              <intent-filter>
                 <category a:name='android.intent.category.LAUNCHER'/>
                 <action a:name='android.intent.action.MAIN'/>
                 </intent-filter>
             </activity>
         </application>
     </manifest>

  4. Crie um subdiretório para os recursos declarados:

     mkdir res

    Leave it empty for now.

Construindo o código #

  1. Gere a origem das declarações de recursos. Substitute here the correct path to your SDK, and the installed API to build against (e.g. “android-23”):

    aapt package -f \
  -I SDK/platforms/android-API/android.jar \
  -J src -m \
  -M AndroidManifest.xml -S res -v

    Resource declarations (described further below) are actually optional. Meantime the above call does nothing if res/ is still empty.

  2. Compile o código fonte em bytecode Java (.java → .class):

    javac \
  -bootclasspath SDK/platforms/android-API/android.jar \
  -classpath src -source 1.7 -target 1.7 \
  src/dom/domain/*.java

  3. Traduza o bytecode de Java para Android (.class → .dex):

    First using Jill (.class → .jayce):

    java -jar SDK/build-tools/LATEST/jill.jar \
  --output classes.jayce src

    Then Jack (.jayce → .dex):

    java -jar SDK/build-tools/LATEST/jack.jar \
  --import classes.jayce --output-dex .

    Android bytecode used to be called “Dalvik executable code”, and so “dex”.

    You could replace steps 11 and 12 with a single call to Jack if you like; it can compile directly from Java source (.java → .dex). But there are advantages to compiling with javac. It’s a better known, better documented and more widely applicable tool.

  4. Empacote os arquivos de recursos, incluindo o manifesto:

    aapt package -f \
  -F app.apkPart \
  -I SDK/platforms/android-API/android.jar \
  -M AndroidManifest.xml -S res -v

    That results in a partial APK file (Android application package).

  5. Faça o APK completo usando a ferramenta ApkBuilder:

    java -classpath SDK/tools/lib/sdklib.jar \
  com.android.sdklib.build.ApkBuilderMain \
  app.apkUnalign \
  -d -f classes.dex -v -z app.apkPart

    It warns, “THIS TOOL IS DEPRECATED. See –help for more information.” If --help fails with an ArrayIndexOutOfBoundsException, then instead pass no arguments:

    java -classpath SDK/tools/lib/sdklib.jar \
  com.android.sdklib.build.ApkBuilderMain

    It explains that the CLI (ApkBuilderMain) is deprecated in favour of directly calling the Java API (ApkBuilder). (If you know how to do that from the command line, please update this example.)

  6. Otimize o alinhamento de dados do APK (prática recomendada):

    zipalign -f -v 4 app.apkUnalign app.apk

Instalando e executando #

  1. Instale o aplicativo no dispositivo Android:

    adb install -r app.apk

  2. Inicie o aplicativo:

    adb shell am start -n dom.domain/.SayingHello

    It should run and say hello.

Isso é tudo. Isso é o que é preciso para dizer olá usando as ferramentas básicas do Android.

Declarando um recurso #

Esta seção é opcional. As declarações de recursos não são necessárias para um aplicativo simples “hello world”. Se eles também não forem necessários para seu aplicativo, então você pode simplificar um pouco a compilação omitindo a etapa 10, e removendo a referência ao diretório res/ da etapa 13.

Caso contrário, aqui está um breve exemplo de como declarar um recurso, e como referenciá-lo.

  1. Adicione um arquivo de recurso:

    mkdir res/values
touch res/values/values.xml

    Content:

    <?xml version='1.0'?>
<resources>
    <string name='appLabel'>Saying hello</string>
</resources>

  2. Referencie o recurso do manifesto XML. This is a declarative style of reference:

    <!-- <application a:label='Saying hello'> -->
     <application a:label='@string/appLabel'>

  3. Referencie o mesmo recurso da origem Java. This is an imperative reference:

    // v.setText( "Hello world" );
   v.setText( "This app is called "
     + getResources().getString( R.string.appLabel ));

  4. Teste as modificações acima reconstruindo, reinstalando and re-running the app (steps 10-17).

    It should restart and say, “This app is called Saying hello”.

Desinstalando o aplicativo #

adb uninstall dom.domain

Fundamentos do aplicativo #

Os aplicativos Android são escritos em Java. As ferramentas do Android SDK compilam o código, os dados e os arquivos de recursos em um APK (pacote Android). Geralmente, um arquivo APK contém todo o conteúdo do aplicativo.

Cada aplicativo é executado em sua própria máquina virtual (VM) para que o aplicativo possa ser executado isolado de outros aplicativos. O sistema Android funciona com o princípio do privilégio mínimo. Cada aplicativo tem acesso apenas aos componentes necessários para fazer seu trabalho e nada mais. No entanto, existem maneiras de um aplicativo compartilhar dados com outros aplicativos, como compartilhar o ID de usuário do Linux entre aplicativos, ou os aplicativos podem solicitar permissão para acessar dados do dispositivo, como cartão SD, contatos etc.

Componentes do aplicativo #

Os componentes do aplicativo são os blocos de construção de um aplicativo Android. Cada componente desempenha um papel específico em um aplicativo Android que serve a um propósito distinto e tem ciclos de vida distintos (o fluxo de como e quando o componente é criado e destruído). Aqui estão os quatro tipos de componentes do aplicativo:

  1. Atividades: uma atividade representa uma única tela com uma interface do usuário (IU). Um aplicativo Android pode ter mais de uma atividade. (por exemplo, um aplicativo de e-mail pode ter uma atividade para listar todos os e-mails, outra para mostrar o conteúdo de cada e-mail e outra para compor um novo e-mail.) Todas as atividades em um aplicativo funcionam juntas para criar uma experiência de usuário (UX).
  2. Serviços: um serviço é executado em segundo plano para realizar operações de longa duração ou para realizar trabalhos para processos remotos. Um serviço não fornece nenhuma interface do usuário, ele é executado apenas em segundo plano com a entrada do usuário. (por exemplo, um serviço pode tocar música em segundo plano enquanto o usuário está em um aplicativo diferente ou pode baixar dados da Internet sem bloquear a interação do usuário com o dispositivo Android.)
  3. Provedores de conteúdo: um provedor de conteúdo gerencia os dados compartilhados do aplicativo. Existem quatro maneiras de armazenar dados em um aplicativo: eles podem ser gravados em um arquivo e armazenados no sistema de arquivos, inseridos ou atualizados em um banco de dados SQLite, postados na web ou salvos em qualquer outro local de armazenamento persistente que o aplicativo possa acessar . Por meio de provedores de conteúdo, outros Apps podem consultar ou até mesmo modificar os dados. (por exemplo, o sistema Android fornece um provedor de conteúdo que gerencia as informações de contato do usuário para que qualquer aplicativo que tenha permissão possa consultar os contatos.) Os provedores de conteúdo também podem ser usados ​​para salvar os dados privados do aplicativo para melhorar a integridade dos dados.
  4. Receptores de transmissão: um receptor de transmissão responde às transmissões de anúncios em todo o sistema (por exemplo, uma transmissão anunciando que a tela foi desligada, a bateria está fraca etc.) ou de aplicativos (por exemplo, para permitir que outros aplicativos saber que alguns dados foram baixados para o dispositivo e estão disponíveis para uso). Os receptores de transmissão não têm UIs, mas podem mostrar notificações na barra de status para alertar o usuário. Normalmente, os broadcast receivers são usados ​​como porta de entrada para outros componentes do aplicativo, consistindo principalmente em atividades e serviços.

Um aspecto único do sistema Android é que qualquer aplicativo pode iniciar o componente de outro aplicativo (por exemplo, se você deseja fazer uma chamada, enviar SMS, abrir uma página da Web ou visualizar uma foto, existe um aplicativo que já faz isso e seu aplicativo pode fazer uso dele, em vez de desenvolver uma nova atividade para a mesma tarefa).

Quando o sistema inicia um componente, ele inicia o processo para esse aplicativo (se ainda não estiver em execução, ou seja, apenas um processo em primeiro plano por aplicativo pode ser executado a qualquer momento em um sistema Android) e instancia as classes necessárias para esse componente. Assim, o componente é executado no processo daquele App ao qual ele pertence. Portanto, ao contrário dos aplicativos em outros sistemas, os aplicativos Android não têm um único ponto de entrada (não existe o método main()).

Como o sistema executa cada aplicativo em um processo separado, um aplicativo não pode ativar diretamente os componentes de outro aplicativo, mas o sistema Android pode. Assim, para iniciar o componente de outro aplicativo, um aplicativo deve enviar uma mensagem ao sistema que especifica uma intenção para iniciar esse componente, então o sistema iniciará esse componente.

Contexto #

Instâncias da classe android.content.Context fornecem a conexão com o sistema Android que executa a aplicação. A Instância de Contexto é necessária para acessar os recursos do projeto e as informações globais sobre o ambiente do aplicativo.

Vamos a um exemplo fácil de digerir: considere que você está em um hotel e quer comer alguma coisa. Você liga para o serviço de quarto e pede que tragam coisas ou limpem as coisas para você. Agora pense neste hotel como um aplicativo Android, você mesmo como uma atividade, e a pessoa do serviço de quarto é o seu contexto, que fornece acesso aos recursos do hotel, como serviço de quarto, alimentos etc.

Ainda um outro exemplo, Você está em um restaurante sentado em uma mesa, cada mesa tem um atendente, sempre que você quiser pedir alimentos, peça ao atendente que o faça. O atendente então faz seu pedido e seus alimentos são servidos em sua mesa. Novamente neste exemplo, o restaurante é um aplicativo Android, as mesas ou os clientes são componentes do aplicativo, os itens alimentares são os recursos do seu aplicativo e o atendente é o seu contexto, dando a você uma maneira de acessar os recursos como itens alimentares.

Ativar qualquer um dos componentes acima requer a instância do contexto. Não apenas o acima, mas quase todos os recursos do sistema: criação da interface do usuário usando visualizações (discutidas posteriormente), criação de instâncias de serviços do sistema, início de novas atividades ou serviços - todos requerem contexto.

Configurando um AVD (dispositivo virtual Android) #

TL;DR Basicamente nos permite simular dispositivos reais e testar nossos aplicativos sem um dispositivo real.

De acordo com Documentação do desenvolvedor Android,

uma definição de Android Virtual Device (AVD) permite definir o características de um telefone, tablet, Android Wear ou Android Dispositivo de TV que você deseja simular no Android Emulator. O AVD O Manager ajuda você a criar e gerenciar AVDs com facilidade.

Para configurar um AVD, siga estas etapas:

1. Clique neste botão para abrir o AVD Manager:

2. Você deverá ver uma caixa de diálogo como esta:

3. Agora clique no botão + Create Virtual Device.... Isso abrirá a caixa de diálogo de configuração do dispositivo virtual:

4. Selecione qualquer dispositivo desejado e clique em “Avançar”:

5. Aqui você precisa escolher uma versão do Android para o seu emulador. Você também pode precisar baixá-lo primeiro clicando em Download. Depois de escolher uma versão, clique em Next.

6. Aqui, insira um nome para seu emulador, orientação inicial e se deseja exibir um quadro ao redor dele. Depois de escolher tudo isso, clique em Finish.

7. Agora você tem um novo AVD pronto para iniciar seus aplicativos nele.
