RenderScript é uma linguagem de script que permite escrever renderização gráfica de alto desempenho e código computacional bruto. Ele fornece um meio de escrever código crítico de desempenho que o sistema compila posteriormente em código nativo para o processador em que pode ser executado. Pode ser a CPU, uma CPU multi-core ou até mesmo a GPU. O que ele executa depende de muitos fatores que não estão prontamente disponíveis para o desenvolvedor, mas também depende de qual arquitetura o compilador da plataforma interna suporta.

Começando #

RenderScript é um framework para permitir computação paralela de alto desempenho no Android. Os scripts que você escreve serão executados em todos os processadores disponíveis (por exemplo, CPU, GPU etc.) em paralelo, permitindo que você se concentre na tarefa que deseja realizar, em vez de como ela é agendada e executada.

Os scripts são escritos em uma linguagem baseada em C99 (C99 sendo uma versão antiga do padrão de linguagem de programação C). Para cada Script é criada uma classe Java que permite que você interaja facilmente com RenderScript em seu código Java.

Configurando seu projeto #

Existem duas maneiras diferentes de acessar o RenderScript em seu aplicativo, com as bibliotecas do Android Framework ou a Biblioteca de Suporte. Mesmo que você não queira direcionar dispositivos antes do nível 11 da API, você deve sempre usar a implementação da Biblioteca de Suporte, pois ela garante a compatibilidade de dispositivos em muitos dispositivos diferentes. Para usar a implementação da biblioteca de suporte, você precisa usar pelo menos as ferramentas de compilação versão 18.1.0!

Agora vamos configurar o arquivo build.gradle da sua aplicação:

android {
    compileSdkVersion 24
    buildToolsVersion '24.0.1'

    defaultConfig {
        minSdkVersion 8
        targetSdkVersion 24

        renderscriptTargetApi 18
        renderscriptSupportModeEnabled true
    }
}
  • renderscriptTargetApi: Isso deve ser definido para o nível de API mais antigo da versão que fornece todas as funcionalidades do RenderScript que você precisa.
  • renderscriptSupportModeEnabled : Isso permite o uso da implementação RenderScript da Biblioteca de Suporte.

Como o RenderScript funciona #

Um RenderScript típico consiste em duas coisas: Kernels e Funções. Uma função é exatamente o que parece - ela aceita uma entrada, faz algo com essa entrada e retorna uma saída. Um Kernel é de onde vem o verdadeiro poder do RenderScript.

Um Kernel é uma função que é executada em cada elemento dentro de uma ‘Alocação’. Um Allocation pode ser usado para passar dados como um Bitmap ou um array byte para um RenderScript e eles também são usados ​​para obter um resultado de um Kernel. Os kernels podem receber uma ‘Alocação’ como entrada e outra como saída ou podem modificar os dados dentro de apenas uma ‘Alocação’.

Você pode escrever seus Kernels, mas também existem muitos Kernels predefinidos que você pode usar para realizar operações comuns como um Gaussian Image Blur.

Como já mencionado para cada arquivo RenderScript, uma classe é gerada para interagir com ele. Essas classes sempre começam com o prefixo ScriptC_ seguido pelo nome do arquivo RenderScript. Por exemplo, se seu arquivo RenderScript for chamado example, então a classe Java gerada será chamada ScriptC_example. Todos os scripts predefinidos começam com o prefixo Script - por exemplo, o Gaussian Image Blur Script é chamado ScriptIntrinsicBlur.

Caldeira do RenderScript #

Os arquivos RenderScript residem na pasta src/main/rs em seu projeto. Cada arquivo tem a extensão de arquivo .rs e deve conter duas instruções #pragma na parte superior:

#pragma version(1)
#pragma rs java_package_name(your.package.name)

  • #pragma version(1): Isso pode ser usado para definir a versão do RenderScript que você está usando. Atualmente existe apenas a versão 1.

  • #pragma rs java_package_name(your.package.name): Isso pode ser usado para definir o nome do pacote da classe Java gerada para interagir com este RenderScript em particular.

Existe outro #pragma que você normalmente deve definir em cada um de seus arquivos RenderScript e é usado para definir a precisão do ponto flutuante. Você pode definir a precisão do ponto flutuante para três níveis diferentes:

  • #pragma rs_fp_full: Esta é a configuração mais estrita com a maior precisão e também é o valor padrão se não especificar nada. Você deve usar isso se precisar de alta precisão de ponto flutuante.
  • #pragma rs_fp_relaxed: Isso garante não uma precisão de ponto flutuante tão alta, mas em algumas arquiteturas permite várias otimizações que podem fazer com que seus scripts sejam executados mais rapidamente.
  • #pragma rs_fp_imprecise: Isso garante ainda menos precisão e deve ser usado se a precisão do ponto flutuante não for realmente importante para o seu script.

A maioria dos scripts pode usar apenas #pragma rs_fp_relaxed a menos que você realmente precise de alta precisão de ponto flutuante.

Variáveis ​​globais #

Agora, assim como no código C, você pode definir variáveis ​​ou constantes globais:

const static float3 gMonoMult = {0.299f, 0.587f, 0.114f};

float saturationLevel = 0.0f;

A variável gMonoMult é do tipo float3. Isso significa que é um vetor que consiste em 3 números flutuantes. A outra variável float chamada saturationValue não é constante, portanto você pode configurá-la em tempo de execução para um valor que desejar. Você pode usar variáveis ​​como esta em seus Kernels ou funções e, portanto, elas são outra maneira de dar entrada ou receber saída de seus RenderScripts. Para cada variável não constante um método getter e setter será gerado na classe Java associada.

Núcleos #

Mas agora vamos começar a implementar o Kernel. Para os propósitos deste exemplo, não vou explicar a matemática usada no Kernel para modificar a saturação da imagem, mas focarei em como implementar um Kernel e como usá-lo. No final deste capítulo, explicarei rapidamente o que o código neste Kernel está realmente fazendo.

Kernels em geral #

Vamos dar uma olhada no código-fonte primeiro:

uchar4 __attribute__((kernel)) saturation(uchar4 in) {
    float4 f4 = rsUnpackColor8888(in);
    float3 dotVector = dot(f4.rgb, gMonoMult);
    float3 newColor = mix(dotVector, f4.rgb, saturationLevel);
    return rsPackColorTo8888(newColor);
}

Como você pode ver, parece uma função C normal com uma exceção: O __attribute__((kernel)) entre o tipo de retorno e o nome do método. Isto é o que diz ao RenderScript que este método é um Kernel. Outra coisa que você pode notar é que este método aceita um parâmetro uchar4 e retorna outro valor uchar4. uchar4 é - como a variável float3 que discutimos no capítulo anterior - um vetor. Ele contém 4 valores uchar que são apenas valores de byte no intervalo de 0 a 255.

Você pode acessar esses valores individuais de muitas maneiras diferentes, por exemplo, in.r retornaria o byte que corresponde ao canal vermelho de um pixel. Usamos um uchar4 já que cada pixel é composto por 4 valores - r para vermelho, g para verde, b para azul e a para alfa - e você pode acessá-los com esta abreviação. RenderScript também permite que você pegue qualquer número de valores de um vetor e crie outro vetor com eles. Por exemplo, in.rgb retornaria um valor uchar3 que contém apenas as partes vermelha, verde e azul do pixel sem o valor alfa.

Em tempo de execução, o RenderScript chamará esse método do Kernel para cada pixel de uma imagem, e é por isso que o valor de retorno e o parâmetro são apenas um valor uchar4. O RenderScript executará muitas dessas chamadas em paralelo em todos os processadores disponíveis, razão pela qual o RenderScript é tão poderoso. Isso também significa que você não precisa se preocupar com encadeamento ou segurança de encadeamento, você pode simplesmente implementar o que quiser para cada pixel e o RenderScript cuida do resto.

Ao chamar um Kernel em Java você fornece duas variáveis ​​Allocation, uma que contém os dados de entrada e outra que receberá a saída. Seu método Kernel será chamado para cada valor na entrada Allocation e escreverá o resultado na saída Allocation.

Métodos da API de tempo de execução do RenderScript #

No Kernel acima, são usados ​​alguns métodos que são fornecidos fora da caixa. RenderScript fornece muitos desses métodos e eles são vitais para quase tudo que você vai fazer com o RenderScript. Entre eles estão métodos para fazer operações matemáticas como sin() e métodos auxiliares como mix() que mistura dois valores de acordo com outros valores. Mas também existem métodos para operações mais complexas quando se trata de vetores, quatérnios e matrizes.

O RenderScript Runtime API Reference oficial é o melhor recurso disponível se você quiser saber mais sobre um determinado ou estão procurando um método específico que execute uma operação comum, como calcular o produto escalar de uma matriz. Você pode encontrar esta documentação aqui.

Implementação do Kernel #

Agora vamos dar uma olhada nas especificidades do que este Kernel está fazendo. Aqui está a primeira linha do Kernel:

float4 f4 = rsUnpackColor8888(in);

A primeira linha chama o método interno rsUnpackColor8888() que transforma o uchar4 para valores float4. Cada canal de cor também é transformado no intervalo 0.0f - 1.0f onde 0.0f corresponde a um valor de byte de 0 e 1.0f para 255. O principal objetivo disso é tornar toda a matemática neste Kernel muito mais simples.

float3 dotVector = dot(f4.rgb, gMonoMult);

Esta próxima linha usa o método integrado dot() para calcular o ponto produto de dois vetores. gMonoMult é um valor constante que definimos alguns capítulos acima. Como ambos os vetores precisam ter o mesmo comprimento para calcular o produto escalar e também como queremos apenas afetar os canais de cores e não o canal alfa de um pixel, usamos a abreviação .rgb para obter um novo vetor float3 que contém apenas os canais de cores vermelho, verde e azul. Aqueles de nós que ainda se lembram da escola como o produto escalar funciona perceberão rapidamente que o produto escalar deve retornar apenas um valor e não um vetor. Ainda no código acima estamos atribuindo o resultado a um vetor float3. Este é novamente um recurso do RenderScript. Quando você atribui um número unidimensional a um vetor, todos os elementos do vetor serão definidos para esse valor. Por exemplo, o snippet a seguir atribuirá 2.0f a cada um dos três valores no vetor float3:

float3 example = 2.0f;

Assim, o resultado do produto escalar acima é atribuído a cada elemento no vetor float3 acima.

Agora vem a parte em que realmente usamos a variável global saturationLevel para modificar a saturação da imagem:

float3 newColor = mix(dotVector, f4.rgb, saturationLevel);

Isso usa o método integrado mix() para misturar a cor original com o vetor de produto escalar que criamos acima. Como eles são misturados é determinado pela variável global saturationLevel. Portanto, um saturationLevel de 0.0f fará com que a cor resultante não tenha parte dos valores de cor originais e consistirá apenas em valores no dotVector que resulta em uma imagem em preto e branco ou acinzentada. Um valor de 1.0f fará com que a cor resultante seja completamente composta pelos valores de cores originais e valores acima de 1.0f multiplicarão as cores originais para torná-las mais brilhantes e intensas.

return rsPackColorTo8888(newColor);

Esta é a última parte do Kernel. rsPackColorTo8888() transforma o vetor float3 de volta em um uchar4 valor que é então retornado. Os valores de byte resultantes são fixados em um intervalo entre 0 e 255, portanto, valores float superiores a 1.0f resultarão em um valor de byte de 255 e valores inferiores a 0.0 resultarão em um valor de byte de 0.

E essa é toda a implementação do Kernel. Agora resta apenas uma parte: Como chamar um Kernel em Java.

Chamando RenderScript em Java #

Fundamentos #

Como já foi mencionado acima para cada arquivo RenderScript é gerada uma classe Java que permite que você interaja com seus scripts. Esses arquivos têm o prefixo ScriptC_ seguido pelo nome do arquivo RenderScript. Para criar uma instância dessas classes, primeiro você precisa de uma instância da classe RenderScript:

final RenderScript renderScript = RenderScript.create(context);

O método estático create() pode ser usado para criar uma instância RenderScript a partir de um Context. Você pode então instanciar a classe Java que foi gerada para seu script. Se você chamou o arquivo RenderScript saturation.rs, a classe será chamada ScriptC_saturation:

final ScriptC_saturation script = new ScriptC_saturation(renderScript);

Nesta classe agora você pode definir o nível de saturação e chamar o Kernel. O setter que foi gerado para a variável saturationLevel terá o prefixo set_ seguido do nome da variável:

script.set_saturationLevel(1.0f);

Há também um getter prefixado com get_ que permite obter o nível de saturação atualmente definido:

float saturationLevel = script.get_saturationLevel();

Kernels que você define em seu RenderScript são prefixados com forEach_ seguido pelo nome do método Kernel. O Kernel que escrevemos espera uma entrada Allocation e uma saída Allocation como seus parâmetros:

script.forEach_saturation(inputAllocation, outputAllocation);

A entrada Allocation precisa conter a imagem de entrada, e depois que o método forEach_saturation terminar, a alocação de saída conterá os dados da imagem modificada.

Uma vez que você tenha uma instância Allocation, você pode copiar dados de e para essas Allocations usando os métodos copyFrom() e copyTo(). Por exemplo, você pode copiar uma nova imagem em sua entrada `Allocation chamando:

inputAllocation.copyFrom(inputBitmap);

Da mesma forma que você pode recuperar a imagem do resultado chamando copyTo() na saída Allocation:

outputAllocation.copyTo(outputBitmap);

Criando instâncias de alocação #

Há muitas maneiras de criar uma ‘Alocação’. Uma vez que você tenha uma instância Allocation, você pode copiar novos dados de e para essas Allocations com copyTo() e copyFrom() como explicado acima, mas para criá-los inicialmente você precisa saber com que tipo de dados você está exatamente trabalhando. Vamos começar com a entrada Allocation:

Podemos usar o método estático createFromBitmap() para criar rapidamente nossa entrada Allocation a partir de um Bitmap:

final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, image);

Neste exemplo, a imagem de entrada nunca muda, portanto, nunca precisamos modificar a entrada Allocation novamente. Podemos reutilizá-lo cada vez que o saturationLevel for alterado para criar um novo Bitmap de saída.

Criar a saída Allocation é um pouco mais complexo. Primeiro precisamos criar o que é chamado de Type. Um Type é usado para informar a uma Allocation com que tipo de dados ela está lidando. Normalmente se usa a classe Type.Builder para criar rapidamente um Type apropriado. Vamos dar uma olhada no código primeiro:

final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

Estamos trabalhando com um ‘Bitmap’ normal de 32 bits (ou seja, 4 bytes) por pixel com 4 canais de cores. É por isso que estamos escolhendo Element.RGBA_8888 para criar o Type. Em seguida, usamos os métodos setX() e setY() para definir a largura e a altura da imagem de saída para o mesmo tamanho da imagem de entrada. O método create() cria então o Type com os parâmetros que especificamos.

Assim que tivermos o Type correto, podemos criar a saída Allocation com o método estático createTyped():

final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

Agora estamos quase terminando. Também precisamos de um Bitmap de saída no qual podemos copiar os dados da saída Allocation. Para fazer isso, usamos o método estático createBitmap() para criar um novo Bitmap vazio com o mesmo tamanho e configuração da entrada Bitmap.

final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

E com isso temos todas as peças do quebra-cabeça para executar nosso RenderScript.

Exemplo completo #

Agora vamos juntar tudo isso em um exemplo:

// Create the RenderScript instance
final RenderScript renderScript = RenderScript.create(context);

// Create the input Allocation 
final Allocation inputAllocation = Allocation.createFromBitmap(renderScript, inputBitmap);

// Create the output Type.
final Type outputType = new Type.Builder(renderScript, Element.RGBA_8888(renderScript))
        .setX(inputBitmap.getWidth())
        .setY(inputBitmap.getHeight())
        .create();

// And use the Type to create am output Allocation
final Allocation outputAllocation = Allocation.createTyped(renderScript, outputType);

// Create an empty output Bitmap from the input Bitmap
final Bitmap outputBitmap = Bitmap.createBitmap(
        inputBitmap.getWidth(),
        inputBitmap.getHeight(),
        inputBitmap.getConfig()
);

// Create an instance of our script
final ScriptC_saturation script = new ScriptC_saturation(renderScript);

// Set the saturation level
script.set_saturationLevel(2.0f);

// Execute the Kernel
script.forEach_saturation(inputAllocation, outputAllocation);

// Copy the result data to the output Bitmap
outputAllocation.copyTo(outputBitmap);

// Display the result Bitmap somewhere
someImageView.setImageBitmap(outputBitmap);

Conclusão #

Com esta introdução, você deve estar pronto para escrever seus próprios Kernels RenderScript para manipulação simples de imagens. No entanto, há algumas coisas que você deve ter em mente:

  • RenderScript só funciona em projetos de aplicativo: atualmente os arquivos RenderScript não podem fazer parte de um projeto de biblioteca.
  • Cuidado com a memória: RenderScript é muito rápido, mas também pode consumir muita memória. Nunca deve haver mais de uma instância de RenderScript a qualquer momento. Você também deve reutilizar o máximo possível. Normalmente você só precisa criar suas instâncias Allocation uma vez e pode reutilizá-las no futuro. O mesmo vale para a saída Bitmaps ou suas instâncias de script. Reutilize o máximo possível.
  • Faça seu trabalho em segundo plano: Novamente, o RenderScript é muito rápido, mas não instantâneo de forma alguma. Qualquer Kernel, especialmente os complexos, devem ser executados fora do thread da interface do usuário em um AsyncTask ou algo semelhante. No entanto, na maioria das vezes, você não precisa se preocupar com vazamentos de memória. Todas as classes relacionadas ao RenderScript usam apenas o aplicativo Context e, portanto, não causam vazamentos de memória. Mas você ainda precisa se preocupar com as coisas usuais, como vazar View, Activity ou qualquer instância Context que você usa!
  • Use coisas incorporadas: existem muitos scripts predefinidos que executam tarefas como desfoque de imagem, mesclagem, conversão, redimensionamento. E há muitos outros métodos embutidos que ajudam você a implementar seus kernels. As chances são de que, se você quiser fazer algo, existe um script ou método que já faz o que você está tentando fazer. Não reinvente a roda.

Se você quiser começar rapidamente e brincar com o código real, eu recomendo que você dê uma olhada no exemplo do projeto GitHub que implementa o exemplo exato mencionado neste tutorial. Você pode encontrar o projeto aqui. Divirta-se com o RenderScript!

Desfocar uma visualização #

BlurBitmapTask.java #

public class BlurBitmapTask extends AsyncTask<Bitmap, Void, Bitmap> {
    private final WeakReference<ImageView> imageViewReference;
    private final RenderScript renderScript;

    private boolean shouldRecycleSource = false;

    public BlurBitmapTask(@NonNull Context context, @NonNull ImageView imageView) {
        // Use a WeakReference to ensure
        // the ImageView can be garbage collected
        imageViewReference = new WeakReference<>(imageView);
        renderScript = RenderScript.create(context);
    }

    // Decode image in background.
    @Override
    protected Bitmap doInBackground(Bitmap... params) {
        Bitmap bitmap = params[0];
        return blurBitmap(bitmap);
    }

    // Once complete, see if ImageView is still around and set bitmap.
    @Override
    protected void onPostExecute(Bitmap bitmap) {
        if (bitmap == null || isCancelled()) {
            return;
        }

        final ImageView imageView = imageViewReference.get();
        if (imageView == null) {
            return;
        }

        imageView.setImageBitmap(bitmap);
    }

    public Bitmap blurBitmap(Bitmap bitmap) {
        // https://plus.google.com/+MarioViviani/posts/fhuzYkji9zz

        //Let's create an empty bitmap with the same size of the bitmap we want to blur
        Bitmap outBitmap = Bitmap.createBitmap(bitmap.getWidth(), bitmap.getHeight(),
                Bitmap.Config.ARGB_8888);

        //Instantiate a new Renderscript


        //Create an Intrinsic Blur Script using the Renderscript
        ScriptIntrinsicBlur blurScript = ScriptIntrinsicBlur.create(renderScript, Element.U8_4(renderScript));

        //Create the in/out Allocations with the Renderscript and the in/out bitmaps
        Allocation allIn = Allocation.createFromBitmap(renderScript, bitmap);
        Allocation allOut = Allocation.createFromBitmap(renderScript, outBitmap);

        //Set the radius of the blur
        blurScript.setRadius(25.f);

        //Perform the Renderscript
        blurScript.setInput(allIn);
        blurScript.forEach(allOut);

        //Copy the final bitmap created by the out Allocation to the outBitmap
        allOut.copyTo(outBitmap);

        // recycle the original bitmap
        // nope, we are using the original bitmap as well :/
        if (shouldRecycleSource) {
            bitmap.recycle();
        }

        //After finishing everything, we destroy the Renderscript.
        renderScript.destroy();

        return outBitmap;
    }
