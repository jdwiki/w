BOOL #

O tipo BOOL é usado para valores booleanos em Objective-C. Tem dois valores, YES e NO, em contraste com os mais comuns “true” e “false”.

Seu comportamento é direto e idêntico ao da linguagem C.

BOOL areEqual = (1 == 1);    // areEqual is YES
BOOL areNotEqual = !areEqual    // areNotEqual is NO
NSCAssert(areEqual, "Mathematics is a lie");    // Assertion passes

BOOL shouldFlatterReader = YES;
if (shouldFlatterReader) {
    NSLog(@"Only the very smartest programmers read this kind of material.");
}

Um BOOL é um primitivo e, portanto, não pode ser armazenado diretamente em uma coleção Foundation. Deve ser encapsulado em um NSNumber. O Clang fornece uma sintaxe especial para isso:

NSNumber * yes = @YES;    // Equivalent to [NSNumber numberWithBool:YES]
NSNumber * no = @NO;    // Equivalent to [NSNumber numberWithBool:NO]

A implementação do BOOL é diretamente baseada em C’s, pois é um typedef do padrão C99 tipo bool. Os valores YES e NO são definidos para __objc_yes e __objc_no, respectivamente. Esses valores especiais são embutidos do compilador introduzidos pelo Clang, que são traduzidos para (BOOL)1 e (BOOL)0. Se eles não estiverem disponíveis, YES e NO são definidos diretamente como o formulário cast-integer. As definições são encontradas no cabeçalho de tempo de execução Objective-C objc.h

SEL #

Seletores são usados ​​como identificadores de método em Objective-C.

No exemplo abaixo, existem dois seletores. novo e setName:

Person* customer = [Person new];
[customer setName:@"John Doe"];

Cada par de colchetes corresponde a um envio de mensagem. Na primeira linha enviamos uma mensagem contendo o seletor new para a classe Person e na segunda linha enviamos uma mensagem contendo o seletor setName: ​​e uma string. O receptor dessas mensagens usa o seletor para procurar a ação correta a ser executada.

Na maioria das vezes, a passagem de mensagens usando a sintaxe de colchetes é suficiente, mas ocasionalmente você precisa trabalhar com o próprio seletor. Nesses casos, o tipo SEL pode ser usado para manter uma referência ao seletor.

Se o seletor estiver disponível em tempo de compilação, você pode usar @selector() para obter uma referência a ele.

SEL s = @selector(setName:);

E se você precisar encontrar o seletor em tempo de execução, use NSSelectorFromString.

SEL s NSSelectorFromString(@"setName:");

Ao usar NSSelectorFromString, certifique-se de envolver o nome do seletor em um NSString.

É comumente usado para verificar se um delegado implementa um método opcional.

if ([self.myDelegate respondsToSelector:@selector(doSomething)]) {
    [self.myDelegate doSomething];
}

Eu iria #

id é o ponteiro de objeto genérico, um tipo Objective-C que representa “qualquer objeto”. Uma instância de qualquer classe Objective-C pode ser armazenada em uma variável id. Um id e qualquer outro tipo de classe podem ser atribuídos sem conversão:

id anonymousSurname = @"Doe";
NSString * surname = anonymousSurname;
id anonymousFullName = [NSString stringWithFormat:@"%@, John", surname];

Isso se torna relevante ao recuperar objetos de uma coleção. Os tipos de retorno de métodos como objectAtIndex: são id exatamente por esse motivo.

DataRecord * record = [records objectAtIndex:anIndex];

Isso também significa que um método ou parâmetro de função digitado como id pode aceitar qualquer objeto.

Quando um objeto é digitado como id, qualquer mensagem conhecida pode ser passada para ele: o método dispatch não depende do tipo de tempo de compilação.

NSString * extinctBirdMaybe = 
               [anonymousSurname stringByAppendingString:anonymousSurname];

Uma mensagem que o objeto realmente não responde ainda causará uma exceção em tempo de execução, é claro.

NSDate * nope = [anonymousSurname addTimeInterval:10];
// Raises "Does not respond to selector" exception

Protegendo-se contra a exceção.

NSDate * nope;
if([anonymousSurname isKindOfClass:[NSDate class]]){
    nope = [anonymousSurname addTimeInterval:10];
}

O tipo id é definido em objc.h

typedef struct objc_object {
    Class isa;
} *id;

IMP (ponteiro de implementação) #

IMP é um tipo C referente à implementação de um método, também conhecido como ponteiro de implementação. É um ponteiro para o início de uma implementação de método.

Sintaxe:

id (*IMP)(id, SEL, …)

O IMP é definido por:

typedef id (*IMP)(id self,SEL _cmd,…);

Para acessar este IMP, pode-se utilizar a mensagem “methodForSelector”.

Exemplo 1:

IMP ImpDoSomething = [myObject methodForSelector:@selector(doSomething)];

O método endereçado pelo IMP pode ser chamado desreferenciando o IMP.

ImpDoSomething(myObject, @selector(doSomething));

Então essas chamadas são iguais:

 myImpDoSomething(myObject, @selector(doSomething));
[myObject doSomething]
[myObject performSelector:mySelector]
[myObject performSelector:@selector(doSomething)]
[myObject performSelector:NSSelectorFromString(@"doSomething")];

Exemplo :2:

SEL otherWaySelector = NSSelectorFromString(@“methodWithFirst:andSecond:andThird:");

IMP methodImplementation  = [self methodForSelector:otherWaySelector];

result = methodImplementation( self,
                          betterWaySelector,
                          first,
                          second,
                          third );

NSLog(@"methodForSelector : %@", result);

Aqui, chamamos [NSObject methodForSelector que nos retorna um ponteiro para a função C que realmente implementa o método, que podemos chamar diretamente posteriormente.

NSUInteger e NSUInteger #

O NSInteger é apenas um typedef para um int ou um long dependendo da arquitetura. O mesmo vale para um NSUInteger que é um typedef para as variantes sem sinal. Se você verificar o NSInteger, verá o seguinte:

#if __LP64__ || (TARGET_OS_EMBEDDED && !TARGET_OS_IPHONE) || TARGET_OS_WIN32 || NS_BUILD_32_LIKE_64
typedef long NSInteger;
typedef unsigned long NSUInteger;
#else
typedef int NSInteger;
typedef unsigned int NSUInteger;
#endif

A diferença entre um inteiro ou longo assinado e não assinado é que um inteiro ou longo assinado pode conter valores negativos. O intervalo do int é -2 147 483 648 a 2 147 483 647 enquanto o unsigned int tem um intervalo de 0 a 4 294 967 295. O valor é duplicado porque o primeiro bit não é mais usado para dizer que o valor é negativo ou não. Para um longo e NSInteger em arquiteturas de 64 bits, o alcance é muito maior.

A maioria dos métodos fornecidos pela Apple está retornando um NS(U)Integer sobre o int normal. Você receberá um aviso se tentar convertê-lo em um int normal porque perderá a precisão se estiver executando em uma arquitetura de 64 bits. Não que isso importe na maioria dos casos, mas é mais fácil usar NS(U)Integer. Por exemplo, o método count em uma matriz retornará um NSUInteger.

NSNumber *iAmNumber = @0;

NSInteger iAmSigned = [iAmNumber integerValue];
NSUInteger iAmUnsigned = [iAmNumber unsignedIntegerValue];

NSLog(@"%ld", iAmSigned); // The way to print a NSInteger.
NSLog(@"%lu", iAmUnsigned); // The way to print a NSUInteger.

Assim como um BOOL, o NS(U)Integer é um tipo de dados primitivo, então às vezes você precisa envolvê-lo em um NSNumber, você pode usar o @ antes do inteiro para convertê-lo como acima e recuperá-lo usando os métodos abaixo. Mas para convertê-lo em NSNumber, você também pode usar os seguintes métodos:

[NSNumber numberWithInteger:0];
[NSNumber numberWithUnsignedInteger:0];
