Introdução #

Os objetos tornam-se elegíveis para coleta de lixo (GC) se não puderem mais ser alcançados pelo(s) ponto(s) de entrada principal em um programa. O GC geralmente não é executado explicitamente pelo usuário, mas para que o GC saiba que um objeto não é mais necessário, o desenvolvedor pode:

Desreferenciar/atribuir nulo

someFunction {
     var a = 1;
     var b = 2;
     a = null; // GC can now free the memory used for variable a
     ...
} // local variable b not dereferenced but will be subject to GC when function ends

Use referências fracas

A maioria das linguagens com GC permite que você crie referências fracas a um objeto que não conta como referência para o GC. Se houver apenas referências fracas a um objeto e nenhuma referência forte (normal), o objeto será elegível para GC.

WeakReference wr = new WeakReference(createSomeObject());

Observe que após esse código é perigoso usar o destino da referência fraca sem verificar se o objeto ainda existe. Programadores iniciantes às vezes cometem o erro de usar código como este:

if wr.target is not null {
    doSomeAction(wr.target);
}

Isso pode causar problemas porque o GC pode ter sido invocado após a verificação nula e antes da execução de doSomeAction. É melhor primeiro criar uma referência forte (temporária) para o objeto assim:

Object strongRef = wr.target;
if strongRef is not null {
    doSomeAction(strongRef);
}
strongRef = null;

Habilitando o log detalhado do gc em Java #

Normalmente a coleta de lixo da jvm (gc) é transparente para o usuário (desenvolvedor/engenheiro).

O ajuste de GC normalmente não é necessário, a menos que o usuário enfrente um vazamento de memória ou tenha um aplicativo que exija uma grande quantidade de memória - os quais eventualmente levam a uma exceção de falta de memória que obriga o usuário a investigar o problema.

O primeiro passo é normalmente aumentar a memória (o heap ou o perm-gen/meta-space dependendo se é devido ao carregamento em tempo de execução ou se a base da biblioteca da aplicação é grande ou existe uma fuga no carregamento de classe ou thread -mecanismo de manuseio). Mas sempre que isso não for viável, o próximo passo é tentar entender o que está acontecendo de errado.

Se alguém quiser apenas o instantâneo em um determinado instante no tempo, o utilitário jstat que faz parte do jdk seria suficiente.

No entanto, para uma compreensão mais detalhada, é útil ter um log contendo o instantâneo do heap antes e depois de cada evento gc. Para isso, o usuário deve habilitar o log do gc detalhado usando o -verbose:gc como parte dos parâmetros de inicialização do jvm e incluindo os sinalizadores -XX:+PrintGCDetails e -XX:+PrintGCTimeStamp.

Para aqueles que desejam perfilar proativamente seu aplicativo, também existem ferramentas como o jvisualvm que também faz parte do jdk através do qual eles podem obter informações sobre o comportamento dos aplicativos.

Abaixo está um programa de exemplo, a configuração do gc e a saída do log verbose-gc:

package com.example.so.docs.gc.logging;

import java.util.Arrays;
import java.util.Random;

public class HelloWorld {

    public static void main(String[] args) {
        sortTest();
    }
    
    private static void sortTest() {
        System.out.println("HelloWorld");
        
        int count = 3;
        while(count-- > 0) {
            int size = 1024*1024;
            int[] numbers = new int[size];
            Random random = new Random();
            for(int i=0;i<size;i++) {
                numbers[i] = random.nextInt(size);
            }
            
            Arrays.sort(numbers);
        }
        System.out.println("Done");
        
    }
    

}
