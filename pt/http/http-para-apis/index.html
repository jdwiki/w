As APIs HTTP usam um amplo espectro de verbos HTTP e normalmente retornam respostas JSON ou XML.

Crie um recurso #

Nem todos concordam sobre qual é o método semanticamente mais correto para a criação de recursos. Assim, sua API pode aceitar solicitações POST ou PUT, ou qualquer um.

O servidor deve responder com 201 Created se o recurso foi criado com sucesso. Escolha o código de erro mais apropriado se não for.

Por exemplo, se você fornecer uma API para criar registros de funcionários, a solicitação/resposta pode ter esta aparência:

POST /employees HTTP/1.1
Host: example.com
Content-Type: application/json

{
    "name": "Charlie Smith",
    "age": 38,
    "job_title": "Software Developer",
    "salary": 54895.00
}

HTTP/1.1 201 Created
Location: /employees/1/charlie-smith
Content-Type: application/json

{
    "employee": {
        "name": "Charlie Smith",
        "age": 38,
        "job_title": "Software Developer",
        "salary": 54895.00
        "links": [
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "self",
                "method": "GET"
            },
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "delete",
                "method": "DELETE"
            },
            {
                "uri": "/employees/1/charlie-smith",
                "rel": "edit",
                "method": "PATCH"
            }
        ]
    },
    "links": [
        {
            "uri": "/employees",
            "rel": "create",
            "method": "POST"
        }
    ]
}

A inclusão dos campos JSON links na resposta permite que o cliente acesse o recurso relacionado ao novo recurso e ao aplicativo como um todo, sem precisar conhecer seus URIs ou métodos de antemão.

Editar um recurso #

Editar ou atualizar um recurso é um objetivo comum das APIs. As edições podem ser feitas enviando solicitações POST, PUT ou PATCH para o respectivo recurso. Embora POST seja permitido para anexar dados à representação existente de um recurso é recomendado usar PUT ou PATCH, pois eles transmitem uma semântica mais explícita.

Seu servidor deve responder com 200 OK se a atualização foi realizada, ou 202 Accepted se ainda não foi aplicada. Escolha o código de erro mais apropriado se não puder ser concluído.

Atualizações completas #

PUT tem a semântica de substituir a representação atual pela carga útil incluída na solicitação. Se a carga útil não for do mesmo tipo de representação que a representação atual do recurso a ser atualizado, o servidor poderá decidir qual abordagem adotar. RFC7231 define que o servidor pode

  • Reconfigure o recurso de destino para refletir o novo tipo de mídia
  • Transforme a representação PUT em um formato consistente com o do recurso antes de salvá-lo como o novo estado do recurso
  • Rejeite a solicitação com uma resposta 415 Unsupported Media Type indicando que o recurso de destino está limitado a um (conjunto) específico de tipos de mídia.

Um recurso base contendo uma representação JSON HAL como …

{
    "name": "Charlie Smith",
    "age": 39,
    "job_title": "Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    }
}

… pode receber um pedido de atualização como este

PUT /users/1234 HTTP/1.1
Host: http://www.acmee.com
Content-Type: "application/json; charset=utf-8"
Content-Length: 85
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer"
}

O servidor agora pode substituir o estado do recurso pelo corpo da solicitação fornecido e também alterar o tipo de conteúdo de application/hal+json para application/json ou converter a carga JSON em uma representação JSON HAL e, em seguida, substituir o conteúdo do recurso com o transformado ou rejeitar a solicitação de atualização devido a um tipo de mídia inaplicável com uma resposta 415 Unsupported Media Type.

Existe uma diferença entre substituir o conteúdo diretamente ou primeiro transformar a representação para o modelo de representação definido e depois substituir o conteúdo existente pelo transformado. Uma solicitação GET subsequente retornará a seguinte resposta em uma substituição direta:

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer"
}

enquanto a abordagem de transformação e substituição retornará a seguinte representação:

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: e0023aa4e

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    }
}

Efeitos colaterais #

Observe que PUT pode ter efeitos colaterais, embora seja definido como operação idempotente! Isso está documentado em RFC7231 como

Uma solicitação PUT aplicada ao recurso de destino pode ter efeitos colaterais em outros recursos. Por exemplo, um artigo pode ter um URI para identificar “a versão atual” (um recurso) que é separado dos URIs que identificam cada versão específica (recursos diferentes que em um ponto compartilharam o mesmo estado que o recurso de versão atual). Uma solicitação PUT bem-sucedida no URI da “versão atual” pode, portanto, criar um novo recurso de versão além de alterar o estado do recurso de destino e também pode fazer com que links sejam adicionados entre os recursos relacionados.

A produção de entradas de log adicionais geralmente não é considerada um efeito colateral, pois certamente não é um estado de um recurso em geral.

Atualizações parciais #

RFC7231 menciona isso sobre atualizações parciais:

Atualizações parciais de conteúdo são possíveis direcionando um recurso identificado separadamente com estado que se sobrepõe a uma parte do recurso maior ou usando um método diferente que foi definido especificamente para atualizações parciais (por exemplo, o método PATCH definido em RFC5789).

As atualizações parciais podem, portanto, ser executadas em dois sabores:

  • Faça com que um recurso incorpore vários sub-recursos menores e atualize apenas um sub-recurso respectivo em vez do recurso completo via PUT
  • Usando PATCH e instrua o servidor sobre o que atualizar

Atualização parcial com estado sobreposto #

Se uma representação do usuário precisar ser parcialmente atualizada devido à mudança de um usuário para outro local, em vez de atualizar o usuário diretamente, o recurso relacionado deve ser atualizado diretamente, o que reflete uma atualização parcial da representação do usuário.

Antes da mudança, um usuário tinha a seguinte representação

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept: application/hal+json; charset=utf-8
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    },
    "_embedded": {
        "ea:address": {
            "street": "Terrace Drive, Central Park",
            "zip": "NY 10024"
            "city": "New York",
            "country": "United States of America",
            "_links": {
                "self": { "href": "/address/abc" },
                "google_maps": { "href": "http://maps.google.com/?ll=40.7739166,-73.970176" }
            }
        }
    }
}

À medida que o usuário está se mudando para um novo local, ela atualiza as informações de local desta forma:

PUT /address/abc HTTP/1.1
Host: http://www.acmee.com
Content-Type: "application/json; charset=utf-8"
Content-Length: 109
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)

{
    "street": "Standford Ave",
    "zip": "CA 94306",
    "city": "Pablo Alto",
    "country": "United States of America"
}

Com a semântica de transformação antes de substituir para o tipo de mídia incompatível entre o recurso de endereço existente e aquele na solicitação, conforme descrito acima, o recurso de endereço agora é atualizado, o que tem o efeito de que em uma solicitação GET subsequente no recurso do usuário, o novo endereço do usuário é retornado.

GET /users/1234 HTTP/1.1
Host: http://www.acmee.com
Accept: application/hal+json; charset=utf-8
Accept-Encoding: gzip, deflate
Accept-Language: en-us
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
ETag: "e0023aa4e"

{
    "name": "Charlie Gold-Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "_links": {
        "self": { "href": "/users/1234" },
        "employee": { "href": "http://www.acmee.com" },
        "curies": [{ "name": "ea", "href": "http://www.acmee.com/docs/rels/{rel}", templated": true}],
        "ea:admin": [
            "href": "/admin/2",
            "title": "Admin"
        ]
    },
    "_embedded": {
        "ea:address": {
            "street": "Standford Ave",
            "zip": "CA 94306",
            "city": "Pablo Alto",
            "country": "United States of America"
            "_links": {
                "self": { "href": "/address/abc" },
                "google_maps": { "href": "http://maps.google.com/?ll=37.4241311,-122.1524475" }
            }
        }
    }
}

Atualizando dados parciais #

PATCH é definido em RFC5789 e não faz parte diretamente da especificação HTTP per se. Uma crença errônea comum é que enviar apenas os campos que devem ser parcialmente atualizados é suficiente dentro de uma solicitação PATCH. A especificação, portanto, afirma

O método PATCH solicita que um conjunto de alterações descritas na entidade request seja aplicado ao recurso identificado pelo Request-URI. O conjunto de alterações é representado em um formato chamado “documento de correção” identificado por um tipo de mídia.

Isso significa que um cliente deve calcular as etapas necessárias para transformar o recurso do estado A para o estado B e enviar essas instruções para o servidor.

Um tipo de mídia popular baseado em JSON para aplicação de patches é JSON Patch.

Se a idade e o cargo do nosso usuário de amostra forem alterados e um campo adicional representando a renda do usuário deve ser adicionado, uma atualização parcial usando PATCH usando JSON Patch pode ficar assim:

PATCH /users/1234 HTTP/1.1
Host: http://www.acmee.com
Content-Type: application/json-patch+json; charset=utf-8
Content-Length: 188
Accept: application/json
If-Match: "e0023aa4e"

[
    { "op": "replace", "path": "/age", "value": 40 },
    { "op": "replace", "path": "/job_title", "value": "Senior Software Developer" },
    { "op": "add", "path": "/salery", "value": 63985.00 }
]

O PATCH pode atualizar vários recursos de uma só vez e requer a aplicação das alterações atomicamente, o que significa que todas as alterações devem ser aplicadas ou nenhuma, o que sobrecarrega o implementador da API.

Uma atualização bem-sucedida pode retornar algo assim

HTTP/1.1 200 OK
Location: /users/1234
Content-Type: application/json
ETag: "df00eb258"

{
    "name": "Charlie Smith",
    "age": 40,
    "job_title": "Senior Software Developer",
    "salary": 63985.00
}

embora não esteja restrito apenas a códigos de resposta 200 OK.

Para evitar atualizações intermediárias (mudanças feitas entre a busca anterior do estado de representação e a atualização), o cabeçalho ETag, If-Match ou If-Unmodified-Since deve ser usado.

Manipulação de erros #

A especificação em PATCH recomenda o seguinte tratamento de erros:

Tipo Código de erro
Documento de patch malformado 400 Pedido Inválido
Documento de patch não suportado 415 Tipo de mídia não suportado
Solicitação não processável, ou seja, se o recurso se tornar inválido aplicando o patch 422 Entidade Não Processável
Recurso não encontrado 404 não encontrado
Estado conflitante, ou seja, uma renomeação (mover) de um campo que não existe 409 Conflito
Modificação conflitante, ou seja, se o cliente usa um cabeçalho If-Match ou If-Unmodified-Since cuja validação falhou. Se nenhuma condição prévia estiver disponível, o último código de erro deve ser retornado 412 Precondition Failed ou 409 Conflict
Modificação simultânea, ou seja, se o pedido precisar ser aplicado antes da aceitação de outros pedidos PATCH 409 Conflito

Excluir um recurso #

Outro uso comum de APIs HTTP é excluir um recurso existente. Isso geralmente deve ser feito usando solicitações DELETE.

Se a exclusão foi bem sucedida, o servidor deve retornar 200 OK; um código de erro apropriado se não for.

Se nosso funcionário Charlie Smith deixou a empresa e agora queremos excluir seus registros, isso pode ser assim:

DELETE /employees/1/charlie-smith HTTP/1.1
Host: example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
    'links': [
        {
            'uri': '/employees',
            'rel': 'create',
            'method': 'POST'
        }
    ]
}

Listar recursos #

O último uso comum de APIs HTTP é obter uma lista de recursos existentes no servidor. Listas como esta devem ser obtidas usando requisições GET, pois elas apenas recuperam dados.

O servidor deve retornar 200 OK se puder fornecer a lista, ou um código de erro apropriado se não puder.

Listar nossos funcionários, então, pode ser assim:

GET /employees HTTP/1.1
Host: example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
    'employees': [
        {
            'name': 'Charlie Smith',
            'age': 39,
            'job_title': 'Software Developer',
            'salary': 63985.00
            'links': [
                {
                    'uri': '/employees/1/charlie-smith',
                    'rel': 'self',
                    'method': 'GET'
                },
                {
                    'uri': '/employees/1/charlie-smith',
                    'rel': 'delete',
                    'method': 'DELETE'
                },
                {
                    'uri': '/employees/1/charlie-smith',
                    'rel': 'edit',
                    'method': 'PATCH'
                }
            ]
        },
        {
            'name': 'Donna Prima',
            'age': 30,
            'job_title': 'QA Tester',
            'salary': 77095.00
            'links': [
                {
                    'uri': '/employees/2/donna-prima',
                    'rel': 'self',
                    'method': 'GET'
                },
                {
                    'uri': '/employees/2/donna-prima',
                    'rel': 'delete',
                    'method': 'DELETE'
                },
                {
                    'uri': '/employees/2/donna-prima',
                    'rel': 'edit',
                    'method': 'PATCH'
                }
            ]
        }
    ],
    'links': [
        {
            'uri': '/employees/new',
            'rel': 'create',
            'method': 'PUT'
        }
    ]
}
