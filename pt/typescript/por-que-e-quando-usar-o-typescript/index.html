Por que e quando usar o TypeScript

  • Segurança
  • Legibilidade
  • Ferramental

Nesta página

  • Segurança

  • Ferramental

Se você achar os argumentos para sistemas de tipos persuasivos em geral, ficará feliz com o TypeScript.

Ele traz muitas das vantagens do sistema de tipos (segurança, legibilidade, ferramentas aprimoradas) para o ecossistema JavaScript. Ele também sofre de algumas das desvantagens dos sistemas de tipos (complexidade e incompletude adicionadas).

Os méritos de linguagens tipadas versus não tipadas têm sido debatidos por décadas. Os argumentos para tipos estáticos incluem:

  1. Segurança: sistemas de tipo permitem que muitos erros sejam detectados antecipadamente, sem executar o código. O TypeScript pode ser [configurado para permitir menos erros de programação][7]
  2. Legibilidade: tipos explícitos tornam o código mais fácil para os humanos entenderem. Como Fred Brooks escreveu, “Mostre-me seus fluxogramas e esconda suas tabelas, e continuarei perplexo. Mostre-me suas tabelas e normalmente não precisarei de seus fluxogramas; eles serão óbvios”.
  3. Ferramentas: sistemas de tipos tornam o código mais fácil para os computadores entenderem. Isso permite que ferramentas como IDEs e linters sejam mais poderosas.
  4. Desempenho: os sistemas de tipos tornam o código mais rápido, reduzindo a necessidade de verificação de tipo em tempo de execução.

Como a saída do TypeScript é independente de seus tipos, o TypeScript não tem impacto no desempenho. O argumento para usar o TypeScript se baseia nas outras três vantagens.

Argumentos contra sistemas de tipos incluem:

  1. Complexidade adicionada: os sistemas de tipos podem ser mais complexos do que o tempo de execução da linguagem que eles descrevem. Funções de ordem superior podem ser fáceis de implementar corretamente, mas difíceis de digitar. Lidar com definições de tipo cria barreiras adicionais ao uso de bibliotecas externas.
  2. Verbalidade adicionada: as anotações de tipo podem adicionar clichê ao seu código, tornando a lógica subjacente mais difícil de seguir.
  3. Iteração mais lenta: ao introduzir uma etapa de compilação, o TypeScript diminui o ciclo de edição/salvamento/recarregamento.
  4. Incompletude: Um sistema de tipos não pode ser sólido e completo. Existem programas corretos que o TypeScript não permite. E os programas que o TypeScript aceita ainda podem conter bugs. Um sistema de tipos não alivia a necessidade de testes. Se você usa o TypeScript, pode ter que esperar mais tempo para usar os novos recursos da linguagem ECMAScript.

O TypeScript oferece algumas maneiras de resolver todos esses problemas:

  1. Complexidade adicionada. Se digitar parte de um programa for muito difícil, o TypeScript pode ser amplamente desabilitado usando um tipo ‘qualquer’ opaco. O mesmo vale para os módulos externos.
  2. Adicionada verbosidade. Isso pode ser parcialmente resolvido por meio de aliases de tipo e da capacidade do TypeScript de inferir tipos. Escrever código claro é tanto uma arte quanto uma ciência: remova muitas anotações de tipo e o código pode não ser mais claro para os leitores humanos.
  3. Iteração mais lenta: Uma etapa de compilação é relativamente comum no desenvolvimento JS moderno e o TypeScript já se integra com a maioria das ferramentas de compilação. E se o TypeScript detectar um erro antecipadamente, você poderá economizar um ciclo de iteração inteiro!
  4. Incompletude. Embora esse problema não possa ser completamente resolvido, o TypeScript conseguiu capturar padrões JavaScript cada vez mais expressivos ao longo do tempo. Exemplos recentes incluem a adição de tipos mapeados em TypeScript 2.1 e mixins em 2.2.

Os argumentos a favor e contra os sistemas de tipos em geral se aplicam igualmente bem ao TypeScript. O uso do TypeScript aumenta a sobrecarga para iniciar um novo projeto. Mas com o tempo, à medida que o projeto aumenta de tamanho e ganha mais colaboradores, a esperança é que os prós de usá-lo (segurança, legibilidade, ferramentas) se tornem mais fortes e superem os contras. Isso se reflete no lema do TypeScript: “JavaScript que escala”.

[7]: https://www.wikiod.com/pt/typescript/tsconfigjson#Configuração para menos erros de programação

Segurança #

O TypeScript detecta erros de tipo antecipadamente por meio de análise estática:

function double(x: number): number {
  return 2 * x;
}
double('2');
//     ~~~ Argument of type '"2"' is not assignable to parameter of type 'number'.

Legibilidade #

O TypeScript permite que os editores forneçam documentação contextual:

Você nunca vai esquecer se String.prototype.slice pega (start, stop) ou (start, length) novamente!

Ferramental #

O TypeScript permite que os editores executem refatorações automatizadas que estão cientes das regras das linguagens.

Aqui, por exemplo, o Visual Studio Code é capaz de renomear referências ao foo interno sem alterar o foo externo. Isso seria difícil de fazer com um simples localizar/substituir.
