Uma interface especifica uma lista de campos e funções que podem ser esperados em qualquer classe que implemente a interface. Por outro lado, uma classe não pode implementar uma interface a menos que tenha todos os campos e funções especificados na interface.

O principal benefício do uso de interfaces é que permite usar objetos de diferentes tipos de maneira polimórfica. Isso ocorre porque qualquer classe que implementa a interface possui pelo menos esses campos e funções.

Sintaxe #

  • interface InterfaceName {
  •   nome do parâmetro: tipo de parâmetro;
  •   opcionalParameterName?: parameterType;
  • }

Interfaces vs Aliases de Tipo #

Interfaces são boas para especificar a forma de um objeto, por exemplo, para um objeto pessoa você pode especificar

interface person {
    id?: number;
    name: string;
    age: number;
}

No entanto, e se você quiser representar, digamos, a maneira como uma pessoa é armazenada em um banco de dados SQL? Visto que cada entrada do banco de dados consiste em uma linha de forma [string, string, number] (portanto, uma matriz de strings ou números), não há como representar isso como uma forma de objeto, porque a linha não tem quaisquer propriedades como tal, é apenas uma matriz.

Esta é uma ocasião em que os tipos são úteis. Em vez de especificar em cada função que aceita um parâmetro de linha function processRow(row: [string, string, number]), você pode criar um alias de tipo separado para uma linha e usá-lo em todas as funções:

type Row = [string, string, number];
function processRow(row: Row)

Documentação oficial da interface #

https://www.typescriptlang.org/docs/handbook/interfaces.html

Estendendo a Interface #

Suponha que temos uma interface:

interface IPerson {
    name: string;
    age: number;

    breath(): void;
}

E queremos criar uma interface mais específica que tenha as mesmas propriedades da pessoa, podemos fazer isso usando a palavra-chave extends:

interface IManager extends IPerson {
    managerId: number;

    managePeople(people: IPerson[]): void;
}

Além disso, é possível estender várias interfaces.

Interface de classe #

Declare o tipo de variáveis ​​e métodos public na interface para definir como outro código typescript pode interagir com ele.

interface ISampleClassInterface {
  sampleVariable: string;

  sampleMethod(): void;
  
  optionalVariable?: string;
}

Aqui criamos uma classe que implementa a interface.

class SampleClass implements ISampleClassInterface {
  public sampleVariable: string;
  private answerToLifeTheUniverseAndEverything: number;

  constructor() {
    this.sampleVariable = 'string value';
    this.answerToLifeTheUniverseAndEverything = 42;
  }

  public sampleMethod(): void {
    // do nothing
  }
  private answer(q: any): number {
    return this.answerToLifeTheUniverseAndEverything;
  }
}

O exemplo mostra como criar uma interface ISampleClassInterface e uma classe SampleClass que implementa a interface.

Adicionar funções ou propriedades a uma interface existente #

Vamos supor que temos uma referência à definição de tipo JQuery e queremos estendê-la para ter funções adicionais de um plugin que incluímos e que não possui uma definição de tipo oficial. Podemos estendê-lo facilmente declarando funções adicionadas pelo plugin em uma declaração de interface separada com o mesmo nome JQuery:

interface JQuery {
  pluginFunctionThatDoesNothing(): void;

  // create chainable function
  manipulateDOM(HTMLElement): JQuery;
}

O compilador irá mesclar todas as declarações com o mesmo nome em uma - veja declaration merging para mais detalhes.

Interfaces genéricas #

Assim como as classes, as interfaces também podem receber parâmetros polimórficos (também conhecidos como genéricos).

Declarando Parâmetros Genéricos em Interfaces #

interface IStatus<U> {
    code: U;
}

interface IEvents<T> {
    list: T[];
    emit(event: T): void;
    getAll(): T[];
}

Aqui, você pode ver que nossas duas interfaces usam alguns parâmetros genéricos, T e U.

Implementando interfaces genéricas #

Vamos criar uma classe simples para implementar a interface IEvents.

class State<T> implements IEvents<T> {
    
    list: T[];
    
    constructor() {
        this.list = [];
    }
    
    emit(event: T): void {
        this.list.push(event);
    }
    
    getAll(): T[] {
        return this.list;
    }
    
}

Vamos criar algumas instâncias da nossa classe State.

Em nosso exemplo, a classe State tratará um status genérico usando IStatus<T>. Desta forma, a interface IEvent<T> também irá lidar com um IStatus<T>.

const s = new State<IStatus<number>>();

// The 'code' property is expected to be a number, so:
s.emit({ code: 200 }); // works
s.emit({ code: '500' }); // type error 

s.getAll().forEach(event => console.log(event.code));

Aqui nossa classe State é digitada como ISatus<number>.

const s2 = new State<IStatus<Code>>();

//We are able to emit code as the type Code
s2.emit({ code: { message: 'OK', status: 200 } });

s2.getAll().map(event => event.code).forEach(event => {
    console.log(event.message);
    console.log(event.status);
});

Nossa classe State é digitada como IStatus<Code>. Dessa forma, podemos passar tipos mais complexos para nosso método de emissão.

Como você pode ver, interfaces genéricas podem ser uma ferramenta muito útil para código estaticamente tipado.

Usando interfaces para polimorfismo #

A principal razão para usar interfaces para obter polimorfismo e fornecer aos desenvolvedores a implementação de sua própria maneira no futuro, implementando os métodos da interface.

Suponha que temos uma interface e três classes:

interface Connector{
    doConnect(): boolean;
}

Esta é a interface do conector. Agora vamos implementar isso para a comunicação Wifi.

export class WifiConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via wifi");
        console.log("Get password");
        console.log("Lease an IP for 24 hours");
        console.log("Connected");
        return true
    }

}

Aqui desenvolvemos nossa classe concreta chamada WifiConnector que tem sua própria implementação. Este é agora o tipo ‘Conector’.

Agora estamos criando nosso System que possui um componente Connector. Isso é chamado de injeção de dependência.

export class System {
    constructor(private connector: Connector){ #inject Connector type
        connector.doConnect()
    }
}

constructor(private connector: Connector) esta linha é muito importante aqui. Connector é uma interface e deve ter doConnect(). Como Connector é uma interface, esta classe System tem muito mais flexibilidade. Podemos passar qualquer tipo que tenha implementado a interface Connector. No futuro desenvolvedor consegue mais flexibilidade. Por exemplo, agora o desenvolvedor deseja adicionar o módulo de conexão Bluetooth:

export class BluetoothConnector implements Connector{

    public doConnect(): boolean{
        console.log("Connecting via Bluetooth");
        console.log("Pair with PIN");
        console.log("Connected");
        return true
    }

}

Veja que Wifi e Bluetooth têm sua própria implementação. Existe uma maneira diferente de se conectar. No entanto, ambos implementaram o Tipo Conector e agora são Tipo Conector. Para que possamos passar qualquer um deles para a classe System como o parâmetro do construtor. Isso é chamado de polimorfismo. A classe System agora não sabe se é Bluetooth / Wifi, mesmo que possamos adicionar outro módulo de comunicação como Inferade, Bluetooth5 e qualquer outro apenas implementando a interface Connector.

Isso é chamado de Duck type. O tipo Connector agora é dinâmico, pois doConnect() é apenas um espaço reservado e o desenvolvedor implementa isso como seu próprio.

se em constructor(private connector: WifiConnector) onde WifiConnector é uma classe concreta, o que acontecerá? Então a classe System irá acoplar firmemente apenas com WifiConnector nada mais. Aqui a interface resolveu nosso problema por polimorfismo.

Usando interfaces para impor tipos #

Um dos principais benefícios do Typescript é que ele impõe tipos de dados de valores que você está passando em seu código para ajudar a evitar erros.

Digamos que você esteja fazendo um aplicativo de namoro para animais de estimação.

Você tem essa função simples que verifica se dois pets são compatíveis entre si…

checkCompatible(petOne, petTwo) {
  if (petOne.species === petTwo.species &&
      Math.abs(petOne.age - petTwo.age) <= 5) {
    return true;
  }
}

Este é um código completamente funcional, mas seria muito fácil para alguém, especialmente outras pessoas trabalhando neste aplicativo que não escreveram esta função, não saber que eles deveriam passar objetos com ’espécie’ e ‘idade’ propriedades. Eles podem erroneamente tentar checkCompatible(petOne.species, petTwo.species) e então ser deixados para descobrir os erros lançados quando a função tenta acessar petOne.species.species ou petOne.species.age!

Uma maneira de evitar que isso aconteça é especificar as propriedades que queremos nos parâmetros do animal de estimação:

checkCompatible(petOne: {species: string, age: number}, petTwo: {species: string, age: number}) {
    //...
}

Nesse caso, o Typescript garantirá que tudo que for passado para a função tenha propriedades ‘species’ e ‘age’ (tudo bem se elas tiverem propriedades adicionais), mas essa é uma solução um pouco complicada, mesmo com apenas duas propriedades especificadas. Com interfaces, há uma maneira melhor!

Primeiro definimos nossa interface:

interface Pet {
  species: string;
  age: number;
  //We can add more properties if we choose.
}

Agora tudo o que temos a fazer é especificar o tipo de nossos parâmetros como nossa nova interface, assim…

checkCompatible(petOne: Pet, petTwo: Pet) {
  //...
}

… e o Typescript garantirá que os parâmetros passados ​​para nossa função contenham as propriedades especificadas na interface Pet!

Implementação Implícita e Forma do Objeto #

O TypeScript suporta interfaces, mas o compilador gera JavaScript, o que não acontece. Portanto, as interfaces são efetivamente perdidas na etapa de compilação. É por isso que a verificação de tipo em interfaces depende da forma do objeto - ou seja, se o objeto suporta os campos e funções na interface - e não se a interface está realmente implementada ou não.

interface IKickable {
  kick(distance: number): void;
}
class Ball {
  kick(distance: number): void {
    console.log("Kicked", distance, "meters!");
  }
}
let kickable: IKickable = new Ball();
kickable.kick(40);

Portanto, mesmo que Ball não implemente explicitamente IKickable, uma instância Ball pode ser atribuída (e manipulada como) a um IKickable, mesmo quando o tipo é especificado.
