Um shell seguro é usado para acessar remotamente um servidor de um cliente por meio de uma conexão criptografada. O OpenSSH é usado como uma alternativa às conexões Telnet que obtêm acesso remoto ao shell, mas não são criptografadas. O OpenSSH Client é instalado na maioria das distribuições GNU/Linux por padrão e é usado para conectar a um servidor. Esses exemplos mostram como usar o conjunto SSH para aceitar conexões SSH e conectar-se a outro host.

Conectando-se a um servidor remoto #

Para conectar a um servidor devemos usar SSH no cliente da seguinte forma,

# ssh -p port user@server-address
  • port - A porta ssh de escuta do servidor (porta padrão 22).
  • usuário - Deve ser um usuário existente no servidor com privilégios SSH.
  • endereço do servidor - O IP/Domínio do servidor.

Para um exemplo do mundo real, vamos fingir que você está criando um site. A empresa que você escolheu para hospedar seu site informa que o servidor está localizado em web-servers.com em uma porta personalizada de 2020 e o nome da sua conta usr1 foi escolhido para criar um usuário no servidor com privilégios SSH. Neste caso, o comando SSH usado seria como tal

# ssh -p 2020 usr1@web-servers.com

Se o nome da conta no sistema remoto for igual ao do cliente local, você pode deixar o nome de usuário desligado. Então, se você é usr1 em ambos os sistemas, então você pode simplesmente usar web-servers.com em vez de usr1@web-servers.com.

Quando um servidor ao qual você deseja se conectar não estiver diretamente acessível a você, você pode tentar usar o switch ProxyJump para se conectar a ele por meio de outro servidor que esteja acessível a você e possa se conectar ao servidor desejado.

# ssh -J usr1@10.0.0.1:2020 usr2@10.0.0.2 -p 2222

Isso permitirá que você se conecte ao servidor 10.0.0.2 (executando ssh na porta 2222) através do servidor em 10.0.0.1 (executando ssh na porta 2020). Você precisará ter contas em ambos os servidores, é claro. Observe também que a opção -J é introduzida no OpenSSH versão 7.3.

Instalando o pacote OpenSSH #

Tanto a conexão com um servidor SSH removido quanto a aceitação de conexões SSH requerem a instalação de openssh

Debian:

# apt-get install openssh

Arqui Linux:

# pacman -S openssh

Sim:

# yum install openssh

Gerar chave pública e privada #

Para gerar chaves para o cliente SSH:

ssh-keygen [-t rsa | rsa1 | dsa ] [-C <comment>] [-b bits]

Por exemplo:

ssh-keygen -t rsa -b 4096 - C myemail@email.com

A localização padrão é ~/.ssh/id_rsa para privada e ~/.ssh/id_rsa.pub para chave pública.

Para mais informações, visite man.openbsd.org

Configurando um servidor SSH para aceitar conexões #

Primeiro devemos editar o arquivo de configuração do daemon SSH. Embora em diferentes distribuições Linux isso possa estar localizado em diretórios diferentes, geralmente é armazenado em /etc/ssh/sshd_config

Use seu editor de texto para alterar os valores definidos neste arquivo, todas as linhas que começam com # são comentadas e devem ter esse caractere removido para ter algum efeito. Uma lista de recomendações segue como tal.

Port (chose a number between 0 - 65535, normaly greater than four digits)
PasswordAuthentication yes
AllowUsers    user1 user2 ...etc

Observe que é preferível desabilitar todos os logins de senha e usar chaves SSH para aumentar a segurança, conforme explicado neste documento.

Conexão sem senha (usando um par de chaves) #

Primeiro de tudo você precisa ter um par de chaves. Se você ainda não tem um, dê uma olhada no tópico ‘Gerar chave pública e privada’.

Seu par de chaves é composto por uma chave privada (id_rsa) e uma chave pública (id_rsa.pub). Tudo que você precisa fazer é copiar a chave pública para o host remoto e adicionar seu conteúdo ao arquivo ~/.ssh/authorized_keys.

Uma maneira simples de fazer isso é:

ssh <user>@<ssh-server> 'cat >> ~/.ssh/authorized_keys' < id_rsa.pub

Uma vez que a chave pública esteja devidamente colocada no diretório home do seu usuário, você só precisa fazer o login usando a respectiva chave privada:

ssh <usuário>@<servidor-ssh> -i id_rsa

Desativar serviço ssh #

Isso desativará o serviço do lado do servidor SSH, pois, se necessário, isso garantirá que os clientes não possam se conectar via ssh

Ubuntu

sudo service ssh stop

Debian

sudo /etc/init.d/ssh stop

Arqui Linux

sudo killall sshd
