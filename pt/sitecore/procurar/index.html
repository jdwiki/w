A pesquisa do Sitecore é construída sobre o Lucene, oferecendo a capacidade de criar recursos de pesquisa muito rápidos para o seu site. Em vez de consultar um banco de dados centralizado (como o SQL DB do Sitecore), ele consulta os arquivos de índice do Lucene que são armazenados no sistema de arquivos físico do servidor web. O Sitecore fornece uma camada de abstração sobre a API Lucene.NET, incluindo um provedor LINQ que torna a escrita de consultas Lucene um processo simples e familiar para desenvolvedores .NET. O Sitecore vem com alguns índices padrão configurados que você pode estender ou definir seus próprios. Você também pode optar por usar SOLR; uma plataforma centralizada e escalável construída sobre o Lucene.

Crie um filtro de entrada para a pesquisa #

O filtro de entrada é aplicado quando um item é adicionado ao índice de pesquisa e permite especificar se o item será incluído no índice ou não.

Exemplos quando o filtro de entrada pode ser usado - não inclua valores padrão e versões anteriores do item no índice.

Os filtros de entrada são definidos na configuração:

<indexing.filterIndex.inbound>
 <processor type="Sitecore.ContentSearch.Pipelines.IndexingFilters.ApplyInboundIndexFilter, Sitecore.ContentSearch"></processor>
</indexing.filterIndex.inbound>

Implementação do código:

public class ApplyInboundIndexVersionFilter : InboundIndexFilterProcessor   
{        
    public override void Process(InboundIndexFilterArgs args)       
    { 
        var item = args.IndexableToIndex as SitecoreIndexableItem;

        if (!item.Item.Versions.IsLatestVersion())            
        {                  
            args.IsExcluded = true;            
        }
    }
}

Mais exemplos e informações podem ser encontrados em http://www.sitecore.net/learn/blogs/technical-blogs/sitecore-7-development-team/posts/2013/04/sitecore-7-inbound-and-outbound- filter-pipelines.aspx

Crie um filtro de saída para a pesquisa #

O filtro de saída pode ser usado para filtrar os resultados da pesquisa.

Um dos exemplos de uso do filtro de saída é a remoção de itens aos quais o usuário não tem acesso dos resultados da pesquisa.

Os filtros de saída são definidos na configuração:

<indexing.filterIndex.outbound>
 <processor type="Sitecore.ContentSearch.Pipelines.IndexingFilters.ApplyOutboundSecurityFilter, Sitecore.ContentSearch"></processor>
</indexing.filterIndex.outbound>

Exemplo de implementação de filtro de saída:

public class ApplyOutboundIndexWorkflowFilter : OutboundIndexFilterProcessor   
{        
    public override void Process(OutboundIndexFilterArgs args)       
    { 
        //You can use args.IsExcluded to remove items from the search results here
    }
}

Mais exemplos e informações podem ser encontrados em http://www.sitecore.net/learn/blogs/technical-blogs/sitecore-7-development-team/posts/2013/04/sitecore-7-inbound-and-outbound- filter-pipelines.aspx

Exclua todas as versões anteriores do item no índice ao adicionar uma nova versão #

Por padrão, o Sitecore adiciona todas as versões do item ao sitecore_master_index. A desvantagem disso é que, se os usuários estiverem usando fluxos de trabalho e adicionando muitas versões, todas elas serão adicionadas aos resultados da pesquisa no editor de conteúdo.

Configuração:

<event name="item:versionAdded" >
              <handler type="FilterPatch.Library.ContentSearch.EventHandler, AssemblyName" method="Execute" />
       </event>

Implementação do gerenciador

public class EventHandler
    {
        public void Execute(object sender, EventArgs eventArgs)
        {
            var item = Event.ExtractParameter(eventArgs, 0) as Item;

            //If item has less than 2 versions - then skip
            if(item.Versions.Count < 2)
            {
                return;
            }

            var indexableItem = new SitecoreIndexableItem(item);

            var index = ContentSearchManager.GetIndex(indexableItem);

            using (var context = index.CreateDeleteContext())
            {
                foreach(var version in item.Versions.GetVersions(true))
                {
                    if(!version.Versions.IsLatestVersion())
                    {
                        var indexableItemVersion = new SitecoreIndexableItem(version);
                        context.Delete(indexableItemVersion.UniqueId);
                    }
                }
                context.Commit();                
            }
        }
    }

Configuração #

O Sitecore vem com um conjunto de índices padrão pré-configurados que você pode estender ou definir seus próprios. Dos pré-configurados, sitecore_master_index e sitecore_web_index são os de maior interesse para a pesquisa do seu site. Esses são os índices predefinidos para todos os itens do Sitecore na árvore dos bancos de dados mestre e da web, respectivamente, e são configurados para armazenar todos os campos padrão de um item do Sitecore que serão comuns entre todos os modelos.

Você pode ver esta configuração do índice da web padrão neste local: <Your Site>\App_Config\Include\Sitecore.ContentSearch.Lucene.Index.Web.config

As principais áreas de importância da configuração de um índice são:

  • A Configuração do Campo - Quais campos devem ser armazenados no índice e como devem ser armazenados.
  • A Estratégia - Como e quando o índice deve ser atualizado.
  • O Crawler - O local onde o índice pode obter os dados do Sitecore

Configuração de campo

Olhando na configuração sitecore_web_index, você pode ver a seguinte referência: <configuration ref="contentSearch/indexConfigurations/defaultLuceneIndexConfiguration" />. Isso se refere a um arquivo de configuração de índice compartilhado encontrado aqui: <Your Site>\App_Config\Include\Sitecore.ContentSearch.Lucene.DefaultIndexConfiguration.config. Aqui você pode ver todos os campos incluídos na configuração padrão.

Existem basicamente duas maneiras de definir um campo: ou o campo é originado diretamente de um campo de item do Sitecore ou é um campo calculado. Um campo computado permite escrever algum código para fazer alguns cálculos e armazenar o resultado no campo. Este código será executado quando o índice for compilado/atualizado não quando o índice for consultado. Isso é particularmente útil se o campo precisar armazenar dados agregados, como contagens, etc.

Dentro do elemento <fieldMap>, você verá os elementos <fieldNames hint="raw:AddFieldByFieldName"> & <fields hint="raw:AddComputedIndexField">, que contêm os campos de origem direta e os campos calculados respectivamente.

Estratégia

A estratégia de seu índice determina quando seu índice é atualizado. Existem as seguintes opções para escolher:

  • OnPublishEndAsynchronousStrategy (onPublishEndAsync) - Quando um item é publicado, o índice é atualizado de forma assíncrona.
  • SynchronousStrategy (syncMaster) - Quando um item é salvo, o índice será atualizado instantaneamente e de forma síncrona.
  • IntervalAsynchronousStrategy (intervalAsyncCore/intervalAsyncMaster) - Verifique periodicamente se há atualizações de itens e atualize o índice de forma assíncrona
  • ManualStrategy - Sem atualizações automáticas de índice. Os índices só serão atualizados manualmente (através do painel de controle ou programaticamente)
  • RebuildAfterFullPublishStrategy (rebuildAfterFullPublish) - Após uma publicação, o índice será totalmente reconstruído
  • RemoteRebuildStrategy (remoteRebuild) - Essa estratégia é para várias instâncias do Sitecore. Por exemplo, se uma reconstrução for solicitada a partir do servidor de gerenciamento de conteúdo, os servidores de entrega de conteúdo remoto serão inscritos nesse evento e reconstruirão seus próprios índices.

Por padrão, o índice mestre é configurado como syncMaster. Isso ocorre porque se você estiver no editor de experiência salvando itens e a renderização na página exibir os resultados de um índice, você desejará ver as alterações feitas nos itens imediatamente nos resultados. O índice da web está configurado como ‘onPublishEndAsync’, porque os índices do seu banco de dados da web só precisam ser atualizados quando os itens são publicados do banco de dados mestre na web.

Você também pode combinar várias estratégias. Por exemplo, se você tiver instâncias Sitecore separadas para seu gerenciamento de conteúdo (CM) e entrega de conteúdo (CD), faria sentido combinar o onPublishEndAsync com o remoteRebuild, para que os índices de CD também sejam atualizados quando os itens forem publicados como reconstruído quando um usuário aciona uma reconstrução do painel de controle do servidor CM.

Você pode escolher sua estratégia usando a seguinte configuração:

<strategies hint="list:AddStrategy">
    <strategy ref="contentSearch/indexConfigurations/indexUpdateStrategies/onPublishEndAsync" />
</strategies>

O Rastreador

Isso permite que você especifique a localização dos dados do Sitecore que deseja indexar. O índice da web tem a seguinte configuração padrão:

<locations hint="list:AddCrawler">
    <crawler type="Sitecore.ContentSearch.SitecoreItemCrawler, Sitecore.ContentSearch">
        <Database>web</Database>
        <Root>/sitecore</Root>
    </crawler>
</locations>

Os dois bits importantes são os elementos <Database> e <Root>. O elemento <Root> permite que você especifique a posição inicial na árvore do Sitecore que o índice deve indexar. Na realidade, é provável que você tenha um nó ‘Home’ sob o nó de conteúdo para o qual você apontaria para que ele indexe apenas conteúdo/páginas reais em vez de seus modelos etc.
