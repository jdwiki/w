swagger-ui com jersey REST WS #

Como o site oficial de Swagger diz:

Swagger é definir uma interface padrão independente de linguagem para APIs REST que permite que humanos e computadores descubram e compreendam os recursos do serviço sem acesso ao código-fonte, documentação ou por meio de inspeção de tráfego de rede. Quando definido corretamente via Swagger, um consumidor pode entender e interagir com o serviço remoto com uma quantidade mínima de lógica de implementação. Semelhante ao que as interfaces fizeram para programação de nível inferior, o Swagger remove a adivinhação ao chamar o serviço.

Supondo que você use Maven e jersey, será necessário adicionar a seguinte dependência: Maven Swagger Dependency junto com as dependências do JAX-RS. Agora você tem que criar “maven webapp” e sob o webapp você precisa colar o conteúdo presente em neste URL. A estrutura de pastas do webapp deve ter a seguinte aparência depois de colar esse conteúdo em seu projeto:

Depois disso siga estes passos:

  1. Crie um arquivo java com qualquer nome (no nosso caso, “ApiOriginFilter.java”) semelhante ao abaixo:

    import javax.servlet.FilterChain; import javax.servlet.FilterConfig; import javax.servlet.ServletException;

    import javax.servlet.ServletRequest; import javax.servlet.ServletResponse; import javax.servlet.http.HttpServletResponse; import java.io.IOException;

    public class ApiOriginFilter implements javax.servlet.Filter { /** * doFilter / @Override public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException { HttpServletResponse res = (HttpServletResponse) response; res.addHeader(“Access-Control-Allow-Origin”, “”); res.addHeader(“Access-Control-Allow-Methods”, “GET, POST, DELETE, PUT”); res.addHeader(“Access-Control-Allow-Headers”, “Content-Type, api_key, Authorization”); chain.doFilter(request, response); }

     /**
  * destroy
  */
 @Override
 public void destroy() {
 }

 /**
  * init
  */
 @Override
 public void init(FilterConfig filterConfig) throws ServletException {
 }

    }

Este arquivo garantirá a filtragem das solicitações recebidas conforme fornecido na classe.

  1. Crie um arquivo java de qualquer nome, suponha que no nosso caso seu nome seja “SwaggerJaxrsConfig.java” da seguinte forma:

    import org.eclipse.persistence.jaxb.MarshallerProperties; import org.eclipse.persistence.jaxb.BeanValidationMode; import org.glassfish.jersey.moxy.json.MoxyJsonConfig; import org.glassfish.jersey.moxy.json.MoxyJsonFeature; import org.glassfish.jersey.moxy.xml.MoxyXmlFeature; import org.glassfish.jersey.server.ResourceConfig;

    import org.glassfish.jersey.server.ServerProperties; import org.glassfish.jersey.server.filter.RolesAllowedDynamicFeature; import io.swagger.jaxrs.config.BeanConfig;

    public class SwaggerJaxrsConfig extends ResourceConfig {

    public SwaggerJaxrsConfig() { BeanConfig beanConfig = new BeanConfig(); beanConfig.setTitle(“Swagger API Title”); beanConfig.setVersion(“1.0.0”);
    beanConfig.setSchemes(new String[] { “http” }); beanConfig.setHost(“localhost:8080/swagger-ui”); beanConfig.setBasePath("/rest"); beanConfig.setResourcePackage( “your.restws.package”); beanConfig.setScan(true);

     register(MoxyXmlFeature.class);
 register(RolesAllowedDynamicFeature.class);

    } }

Como você pode ver, na aula acima, fornecemos todos os detalhes necessários para usar a interface do usuário do swagger como o Host no qual este projeto do swagger será hospedado junto com a facilidade de definir o caminho base de sua escolha e todos os protocolos http suportados como “http " ou “https” e ainda mais detalhes conforme sua necessidade. Também há uma provisão para permitir que o Swagger saiba toda a localização do seu WS REST que pode ser definida usando setResourcePackage junto com pacotes Esta classe nos permite usar Swagger UI com nossa customização.

Agora faça a entrada dos 2 arquivos acima no web.xml para usá-los depois de implantar nosso aplicativo no servidor como segue:

<servlet>

Agora, vamos passar para o código real, onde usaremos as anotações fornecidas pelo Swagger:

Crie um bean Employee.java como abaixo:

import io.swagger.annotations.ApiModel;
import io.swagger.annotations.ApiModelProperty;

@ApiModel("Employee bean")
public class Employee {

    private String name;
    private String id;
    private String dept;

    @ApiModelProperty(value = "Name of employee", example = "Test Employee")
    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    @ApiModelProperty(value = "Id of employee", example = "123456")
    public String getId() {
        return id;
    }

    public void setId(String id) {
        this.id = id;
    }

    @ApiModelProperty(value = "Department of employee", example = "IT Division", allowableValues = "IT, Sales, Admin")
    public String getDept() {
        return dept;
    }

    public void setDept(String dept) {
        this.dept = dept;
    }
}

Os componentes do Swagger que usamos aqui são:

  1. @ApiModel("Employee bean") - Isso decidirá o nome da classe Bean que precisa ser exibida na interface do usuário do Swagger.
  2. @ApiModelProperty(value ="ABC", example="DeptName") - Isso fornecerá informações de cada campo usado no bean. value fornece a descrição do campo e example fornece o valor de amostra desse campo.

Agora vamos criar um REST Controller da seguinte forma para criar um webservice GET da seguinte forma:

import javax.ws.rs.GET;
import javax.ws.rs.Path;
import javax.ws.rs.Produces;
import javax.ws.rs.core.GenericEntity;
import javax.ws.rs.core.MediaType;
import javax.ws.rs.core.Response;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.swagger.ws.bean.Employee;
import org.swagger.ws.service.EmployeeService;

import io.swagger.annotations.Api;
import io.swagger.annotations.ApiOperation;
import io.swagger.annotations.ApiResponse;
import io.swagger.annotations.ApiResponses;

@Path("/employee")
@Api(tags = {"Employee"})
public class EmployeeController {
    private static final Logger LOGGER = LoggerFactory.getLogger(EmployeeController.class);

@GET
@Produces(MediaType.APPLICATION_JSON + ";charset=utf-8")
@ApiOperation(produces="application/json", value = "Fetch employee details", httpMethod="GET", notes = "<br>This service fetches Employee details", response = Employee.class)
@ApiResponses(value = { @ApiResponse(code = 200,response = Employee.class, message = "Successful operation"),@ApiResponse(code = 400, message = "Bad Request", response = Error.class), @ApiResponse(code = 422, message = "Invalid data", response = Error.class), @ApiResponse(code = 500, message = "Internal Server Error", response = Error.class) })
public Response getEmployee()  {
        EmployeeService employeeService = new EmployeeService();
        Employee empDetails = employeeService.getEmployee();
        Response response;
        LOGGER.debug("Fetching employee");
        GenericEntity<Employee> entity = new GenericEntity<Employee>(empDetails){}; 
        response = Response.status(200).entity(entity).build();
        return response;
    }
}

Os componentes do Swagger que usamos aqui são:

  1. @Api(tags = {"Employee"}) - Esta anotação dirá ao Swagger qual deve ser o título do serviço web. Neste caso, o título é “Funcionário”. Observe que alguns padrões ao escrever REST WS também são aplicáveis ​​ao escrever a Documentação Swagger, como o título não deve ser como “GetEmployee” ou “DeleteEmployee”, etc.
  2. @ApiOperation(produces="application/json", value = "Buscar detalhes do funcionário", httpMethod="GET", notes = "<br>Este serviço busca detalhes do funcionário", response = Employee.class) - Isso anotação fornece uma breve ideia sobre o seu webservice. - produces describes format of response, - value describes brief idea about the webservice - notes describes detailed information about this webservice in case of any.
  3. @ApiResponses(value = { @ApiResponse(code = 200,response = Employee.class, message = "Operação bem-sucedida"),@ApiResponse(code = 400, message = "Bad Request", response = Error.class) , @ApiResponse(code = 422, message = "Dados inválidos", response = Error.class), @ApiResponse(code = 500, message = "Internal Server Error", response = Error.class) }) - Estas anotações fornecem nós uma maneira de lidar com diferentes tipos de códigos de status HTTP que podem ser recebidos como uma resposta ao consumir este webservice. Ele nos permite definir o código de erro, a mensagem personalizada e até mesmo nos permite capturar esse erro em uma classe de erro separada, se necessário.

Finalmente, temos que criar a classe Service real que buscará os detalhes dos funcionários quando o webservice for consumido pelo cliente. Você pode implementá-lo conforme necessário. A seguir está o serviço de amostra para fins de demonstração:

public class EmployeeService {

public Employee getEmployee() {
    Employee employee = new Employee();
    employee.setId("1");
    employee.setName("Test");
    employee.setDept("IT");
    return employee;
    }
}
