CUDA é uma tecnologia proprietária de computação paralela da NVIDIA e linguagem de programação para suas GPUs.

As GPUs são máquinas altamente paralelas capazes de executar milhares de threads leves em paralelo. Cada thread da GPU geralmente é mais lento na execução e seu contexto é menor. Por outro lado, a GPU é capaz de executar vários milhares de threads em paralelo e ainda mais simultaneamente (os números precisos dependem do modelo real da GPU). CUDA é um dialeto C++ projetado especificamente para a arquitetura NVIDIA GPU. No entanto, devido às diferenças de arquitetura, a maioria dos algoritmos não pode ser simplesmente copiada e colada de C++ simples - eles seriam executados, mas seriam muito lentos.

Terminologia #

  • host – refere-se a hardware normal baseado em CPU e programas normais que são executados nesse ambiente
  • dispositivo – refere-se a uma GPU específica na qual os programas CUDA são executados. Um único host pode suportar vários dispositivos.
  • kernel – uma função que reside no dispositivo que pode ser invocada a partir do código do host.

Estrutura do Processador Físico #

O processador GPU habilitado para CUDA tem a seguinte estrutura física:

  • o chip - todo o processador da GPU. Algumas GPUs têm dois deles.
  • multiprocessador de streaming (SM) - cada chip contém até ~100 SMs, dependendo do modelo. Cada SM opera quase independentemente do outro, usando apenas a memória global para se comunicar entre si.
  • CUDA core - uma única unidade de computação escalar de um SM. Seu número preciso depende da arquitetura. Cada núcleo pode lidar com alguns threads executados simultaneamente em uma rápida sucessão (semelhante ao hyperthreading na CPU).

Além disso, cada SM possui um ou mais programadores de warp. Cada escalonador despacha uma única instrução para vários núcleos CUDA. Isso efetivamente faz com que o SM opere no modo SIMD de 32 amplos.

Modelo de Execução CUDA #

A estrutura física da GPU tem influência direta em como os kernels são executados no dispositivo e como são programados em CUDA. O kernel é invocado com uma configuração de chamada que especifica quantos threads paralelos são gerados.

  • the grid - representa todas as threads que são geradas na chamada do kernel. É especificado como um conjunto unidimensional ou bidimensional de blocos
  • o bloco - é um conjunto semi-independente de threads. Cada bloco é atribuído a um único SM. Como tal, os blocos podem se comunicar apenas através da memória global. Os blocos não são sincronizados de forma alguma. Se houver muitos blocos, alguns podem ser executados sequencialmente após outros. Por outro lado, se os recursos permitirem, mais de um bloco pode ser executado no mesmo SM, mas o programador não pode se beneficiar disso (exceto pelo óbvio aumento de desempenho).
  • the thread - uma sequência escalar de instruções executadas por um único núcleo CUDA. Os threads são ’leves’ com contexto mínimo, permitindo que o hardware os troque rapidamente. Por causa de seu número, as threads CUDA operam com alguns registradores atribuídos a elas e uma pilha muito pequena (de preferência nenhuma!). Por essa razão, o compilador CUDA prefere embutir todas as chamadas de função para nivelar o kernel para que ele contenha apenas saltos e loops estáticos. Chamadas de ponter de função e chamadas de método virtual, embora suportadas na maioria dos dispositivos mais novos, geralmente incorrem em uma grande penalidade de desempenho.

Cada thread é identificada por um índice de bloco blockIdx e um índice de thread dentro do bloco threadIdx. Esses números podem ser verificados a qualquer momento por qualquer thread em execução e é a única maneira de distinguir um thread do outro.

Além disso, as threads são organizadas em warps, cada uma contendo exatamente 32 threads. Threads dentro de um único warp são executados em perfeita sincronia, no modo SIMD. Threads de warps diferentes, mas dentro do mesmo bloco podem ser executadas em qualquer ordem, mas podem ser forçadas a sincronizar pelo programador. Threads de diferentes blocos não podem ser sincronizados ou interagir diretamente de forma alguma.

Organização da Memória #

Na programação normal da CPU, a organização da memória geralmente é ocultada do programador. Programas típicos agem como se houvesse apenas RAM. Todas as operações de memória, como gerenciamento de registradores, uso de cache L1- L2- L3-, troca para disco, etc. são tratadas pelo compilador, sistema operacional ou próprio hardware.

Este não é o caso do CUDA. Embora os modelos de GPU mais recentes escondam parcialmente o fardo, por exemplo, por meio da Unified Memory no CUDA 6, ainda vale a pena entender a organização por questões de desempenho. A estrutura básica da memória CUDA é a seguinte:

  • Memória do host – a RAM normal. Usado principalmente pelo código do host, mas os modelos de GPU mais recentes também podem acessá-lo. Quando um kernel acessa a memória do host, a GPU deve se comunicar com a placa-mãe, geralmente através do conector PCIe e, como tal, é relativamente lenta.
  • Memória do dispositivo / Memória global – a memória principal fora do chip da GPU, disponível para todos os threads.
  • Memória compartilhada - localizada em cada SM permite acesso muito mais rápido que o global. A memória compartilhada é privada para cada bloco. Threads dentro de um único bloco podem usá-lo para comunicação.
  • Registros - memória mais rápida, privada e não endereçável de cada thread. Em geral, eles não podem ser usados ​​para comunicação, mas algumas funções intrínsecas permitem embaralhar seu conteúdo dentro de uma urdidura.
  • Memória local - memória privada de cada thread que é endereçável. Isso é usado para derramamentos de registros e matrizes locais com indexação de variáveis. Fisicamente, eles residem na memória global.
  • Memória de textura, memória constante - uma parte da memória global marcada como imutável para o kernel. Isso permite que a GPU use caches para fins especiais.
  • Cache L2 – no chip, disponível para todos os threads. Dada a quantidade de threads, o tempo de vida esperado de cada linha de cache é muito menor do que na CPU. É usado principalmente para ajudar a padrões de acesso à memória desalinhados e parcialmente aleatórios.
  • Cache L1 – localizado no mesmo espaço da memória compartilhada. Novamente, a quantidade é bastante pequena, dado o número de threads que a utilizam, portanto, não espere que os dados permaneçam lá por muito tempo. O cache L1 pode ser desabilitado.
