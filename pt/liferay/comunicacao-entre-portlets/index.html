Usando o parâmetro de renderização público #

Essa abordagem foi introduzida no JSR 286.

No JSR 168, os parâmetros de renderização definidos em processAction de um portlet estavam disponíveis apenas nesse portlet. Com o recurso Parâmetros de renderização públicos, os parâmetros de renderização definidos no processAction de um portlet também estarão disponíveis na renderização de outros portlets. Para configurar isso, para todos os portlets que suportam isso:

Adicione a tag <supported-public-render-parameter>, logo antes da tag do portlet terminar em portlet.xml

<security-role-ref>
    <role-name>user</role-name>
</security-role-ref>
<supported-public-render-parameter>{param-name}</supported-public-render-parameter>
</portlet>

Adicione a tag <public-render-parameter> logo antes da tag <portlet-app> terminar

  <public-render-parameter>
    <identifier>{param-name}</identifier>
    <qname xmlns:x="localhost">x:{param-name}</qname>
  </public-render-parameter>
</portlet-app>

No método processAction, o valor do parâmetro precisa ser definido na resposta

res.setRenderParameter({param-name},{param-value});

Depois que terminamos de configurar isso para todos os portlets necessários, após executar a fase de ação do portlet em questão, o parâmetro deve estar disponível na fase de renderização para todos os portlets de suporte na página, independentemente de fazerem parte do mesmo ou de um aplicativo diferente (war ).

Usando a sessão do Portlet #

Essa é uma abordagem que existe desde o JSR 168. Ela nos permite compartilhar atributos usando a sessão do portlet. Uma sessão do portlet pode ter diferentes tipos de escopos:

  1. Escopo do portlet (atributos disponíveis apenas no portlet)
  2. Escopo do aplicativo (atributos disponíveis em todo o aplicativo[war])

Para usar essa abordagem, não precisamos fazer nenhuma entrada na configuração do portlet, pois a sessão do portlet está prontamente disponível na solicitação do portlet:

PortletSession session = renderRequest.getPortletSession();
session.setAttribute("attribute-name","attribute-value", PortletSession.APPLICATION_SCOPE);

ou

PortletSession session = renderRequest.getPortletSession();
session.setAttribute("attribute-name","attribute-value", PortletSession.PORTLET_SCOPE);

O atributo só pode ser recuperado do respectivo escopo. Como para o atributo definido no escopo do portlet, precisamos buscá-lo usando

PortletSession session = renderRequest.getPortletSession();
String attributeValue = (String) session.getAttribute("attribute-name", PortletSession.PORTLET_SCOPE);

A principal limitação desta abordagem é a falta de compartilhamento entre outros portlet, fora do escopo do aplicativo.

    <private-session-attributes>false</private-session-attributes>
    <header-portlet-css>/css/main.css</header-portlet-css>
    <footer-portlet-javascript>/js/main.js</footer-portlet-javascript>
    <css-class-wrapper>{portlet-name}</css-class-wrapper>
</portlet>

para todos os portlets, onde os atributos são configurados e recuperados.

Usando o recurso de eventos #

O mecanismo de eventos é uma versão estendida do parâmetro de renderização pública, com recurso adicional para passar objetos personalizados para outros portlets, mas com uma sobrecarga da fase de eventos.

Para isso, esse mecanismo consiste em

  1. Portlet do editor
  2. Portlet de processador (consumidor), em que ambos podem fazer parte de diferentes aplicativos de portlet.

Começar com,

Adicione a tag <supported-publishing-event> ao portlet do publicador em portlet.xml

    <security-role-ref>
        <role-name>user</role-name>
    </security-role-ref>
    <supported-publishing-event>
         <qname xmlns:x="http:sun.com/events">x:Employee</qname>
    </supported-publishing-event>
  </portlet>

Adicione a tag <supported-processing-event> ao portlet do processador em portlet.xml

<security-role-ref>
        <role-name>user</role-name>
    </security-role-ref>
    <supported-processing-event>
        <qname xmlns:x="http:sun.com/events">x:Employee</qname>
     </supported-processing-event>
</portlet>

Adicione a tag <event-definition> a ambos os portlets, definindo o nome do evento e digite portlet.xml

<event-definition>   
  <qname xmlns:x="http:sun.com/events">x:Employee</qname>
  <value-type>com.sun.portal.portlet.users.Employee</value-type>
</event-definition>
   </portlet-app>

Em seguida, precisamos criar uma classe para o tipo de evento (no caso de tipo personalizado)

public class Employee implements Serializable {
  public Employee() {
  }
  private String name;
  private int userId; 

  public String getName() {
    return name;
  }

  public void setName(String name) {
    this.name = name;
  }

  public int getUserId() {
    return userId;
  }
  public void setUserId(int id)
  {
    this.userId = id;
  }

}

Agora, no portlet do editor, o evento precisa ser publicado na fase de ação

    QName qname = new QName("http:sun.com/events" , "Employee");
    Employee emp = new Employee();
    emp.setName("Rahul");
    emp.setUserId(4567);
    res.setEvent(qname, emp);

Post que publicamos o evento, ele precisa ser processado pelo portlet do editor na fase do evento.

A fase de eventos foi introduzida no JSR 286 e é executada antes da fase de renderização do portlet, quando aplicável

@ProcessEvent(qname = "{http:sun.com/events}Employee")
public void processEvent(EventRequest request, EventResponse response) {

    Event event = request.getEvent();
    if(event.getName().equals("Employee")){
      Employee payload = (Employee)event.getValue();
      response.setRenderParameter("EmpName",
      payload.getName());
    }

}

que pode ser recuperado do parâmetro render via solicitação de renderização.
