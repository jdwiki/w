Criação de perfil de uma compilação #

Antes de começar a ajustar sua compilação do Gradle para desempenho, você deve estabelecer uma linha de base e descobrir quais partes da compilação estão demorando mais. Para fazer isso, você pode perfilar sua compilação adicionando o argumento --profile ao seu comando Gradle:

gradle --profile
./gradlew --profile

Após a conclusão da compilação, você verá um relatório de perfil HTML para a compilação em ./build/reports/profile/, parecido com isto:

Ao clicar nas guias ao lado de Resumo, você pode ver um detalhamento mais detalhado de onde o tempo é gasto.

Use o Gradle Daemon #

Você pode habilitar o Gradle Daemon para melhorar o desempenho de suas compilações.

O Gradle Daemon mantém o Gradle Framework inicializado e em execução e armazena em cache os dados do projeto na memória para melhorar o desempenho.

Para uma única construção

Para habilitar o Daemon para uma única compilação, você pode simplesmente passar o argumento --daemon para o seu comando gradle ou script Gradle Wrapper.

gradle --daemon
./gradlew --daemon

Para todas as construções de um projeto

Para habilitar o Daemon para todas as compilações de um projeto, você pode adicionar:

org.gradle.daemon=true

Para o arquivo gradle.properties do seu projeto.

Para todas as construções

Para habilitar o Gradle Daemon por padrão, para cada compilação feita por sua conta de usuário em seu sistema, edite $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties por padrão) e adicione esta linha :

org.gradle.daemon=true

Você também pode fazer isso em um único comando em sistemas Mac/Linux/*nix:

touch ~/.gradle/gradle.properties && echo "org.gradle.daemon=true" >> ~/.gradle/gradle.properties

Ou no Windows:

(if not exist "%USERPROFILE%/.gradle" mkdir "%USERPROFILE%/.gradle") && (echo org.gradle.daemon=true >> "%USERPROFILE%/.gradle/gradle.properties")

Desativando o Daemon

Você pode desabilitar o Daemon para uma compilação específica usando o argumento --no-daemon, ou desabilitá-lo para um projeto específico configurando explicitamente org.gradle.daemon=false no arquivo gradle.properties do projeto.

Parando o Demônio

Se você deseja parar um processo Daemon manualmente, você pode matar o processo através do gerenciador de tarefas do seu sistema operacional ou executar o comando gradle --stop. A opção --stop faz com que o Gradle solicite que todos os processos do Daemon em execução, da mesma versão do Gradle usada para executar o comando, terminem. Normalmente, os processos do Daemon serão encerrados automaticamente *após *3 horas de inatividade ou menos.

Compilações paralelas do Gradle #

O Gradle executará apenas uma tarefa por vez por padrão, independentemente da estrutura do projeto. Usando a opção --parallel, você pode forçar o Gradle a executar subprojetos independentes - aqueles que não têm dependências de projeto implícitas ou explícitas entre si - em paralelo, permitindo que ele execute várias tarefas ao mesmo tempo, desde que essas tarefas estão em projetos diferentes.

Para construir um projeto em modo paralelo:

gradle build --parallel

Você também pode tornar a construção em paralelo o padrão para um projeto adicionando a seguinte configuração ao arquivo gradle.properties do projeto:

org.gradle.parallel=true

Use a versão mais recente do Gradle #

A equipe do Gradle trabalha regularmente para melhorar o desempenho de diferentes aspectos das compilações do Gradle. Se você estiver usando uma versão antiga do Gradle, está perdendo os benefícios desse trabalho. Tente atualizar para a versão mais recente do Gradle para ver que tipo de impacto ele tem. Fazer isso é de baixo risco porque muito poucas coisas quebram entre as versões menores do Gradle.

O arquivo de propriedades para o wrapper Gradle pode ser encontrado na pasta do seu projeto em gradle/wrapper/ e é chamado gradle-wrapper.properties. O conteúdo desse arquivo pode ficar assim:

distributionBase=GRADLE_USER_HOME
distributionPath=wrapper/dists
zipStoreBase=GRADLE_USER_HOME
zipStorePath=wrapper/dists
distributionUrl=https\://services.gradle.org/distributions/gradle-X.X.X.zip

Você pode alterar manualmente o número da versão X.X.X(versão atual) para Y.Y.Y(versão mais recente) e na próxima vez que você executar o wrapper, a nova versão será baixada automaticamente.

Configurar sob demanda #

Se a criação de perfil de sua compilação mostrar um tempo significativo gasto em Configurando projetos, a opção Configurar sob demanda pode melhorar seu desempenho.

Você pode habilitar o modo Configure on Demand editando $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties por padrão), e configurando org.gradle.configureondemand.

org.gradle.configureondemand=true

Para ativá-lo apenas para um projeto específico, edite o arquivo gradle.properties desse projeto.

Se o Configure on Demand estiver habilitado, em vez de configurar todos os projetos antecipadamente, o Gradle configurará apenas os projetos necessários para a tarefa que está sendo executada.

Do Manual do Gralde:

O modo de configuração sob demanda tenta configurar apenas os projetos que são relevantes para as tarefas solicitadas, ou seja, ele executa apenas o arquivo build.gradle dos projetos que estão participando da compilação. Dessa forma, o tempo de configuração de uma grande compilação de vários projetos pode ser reduzido. A longo prazo, esse modo se tornará o modo padrão, possivelmente o único modo de execução de compilação do Gradle.

Ajustando os parâmetros de uso da memória JVM para Gradle #

Você pode definir ou aumentar os limites de uso de memória (ou outros argumentos JVM) usados ​​para compilações Gradle e o Gradle Daemon editando $GRADLE_USER_HOME/.gradle/gradle.properties (~/.gradle/gradle.properties por padrão), e definindo org.gradle.jvmargs.

Para configurar esses limites apenas para um projeto específico, edite o arquivo gradle.properties desse projeto.

As configurações de uso de memória padrão para compilações Gradle e o Gradle Daemon são:

org.gradle.jvmargs=-Xmx1024m -XX:MaxPermSize=256m

Isso permite uma alocação máxima geral de memória (tamanho do heap) de 1 GB e uma alocação máxima de memória para objetos “internos” permanentes de 256 MB. Quando esses tamanhos são atingidos, ocorre a coleta de lixo, o que pode diminuir significativamente o desempenho.

Supondo que você tenha memória de sobra, você pode facilmente dobrar isso assim:

org.gradle.jvmargs=-Xmx2024m -XX:MaxPermSize=512m

Observe que você deixará de ver os benefícios de aumentar XX:MaxPermSize mais cedo do que quando os aumentos de Xmx deixarem de ser benéficos.
