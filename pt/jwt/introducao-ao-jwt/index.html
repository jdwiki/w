JWT não assinado #

Um JWT não assinado tem o valor do cabeçalho alg: none e um componente JWS (assinatura) vazio:

eyJhbGciOiJub25lIn0
.eyJpc3MiOiJqb2UiLA0KICJleHAiOjEzMDA4MTkzODAsDQogImh0dHA6Ly9leGFtcGxlLmNvbS9pc19yb290Ijp0cnVlfQ
.

O ponto final indica que a assinatura está vazia.

Cabeçalho #

{
  "alg": "none"
}

Carga #

{
  "iss": "joe",
  "exp": 1300819380,
  "http://example.com/is_root": true
}

JWT assinado (JWS) #

Um JWT assinado inclui uma assinatura codificada Base64 Url Safe como o terceiro componente. O algoritmo usado para gerar a assinatura é indicado no cabeçalho.

eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9
.eyJzdWIiOiJKb2huIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE0NzAzNTM5OTQsImV4cCI6MTQ3MDM1NzYyNywianRpIjoiNmU0MDRiYTgtZjg4NS00ZDVmLWJmYTItZTNmNWEwODM4MGE0In0
.7CfBdVP4uKsb0cogYepCvMLm8rcpjBYW1XZzA-a5e44

Cabeçalho #

{
  "typ": "JWT",
  "alg": "HS256"
}

Este JWT foi assinado com o algoritmo HMAC-SHA256, daí alg: HS256.

Carga #

{
  "sub": "John Doe",
  "admin": true,
  "iat": 1470353994,
  "exp": 1470357627,
  "jti": "6e404ba8-f885-4d5f-bfa2-e3f5a08380a4"
}

Este JWT pode ser verificado com o segredo UTF-8 notsosecret.

Como saber se você tem um JWS ou JWE? #

Da Seção 9 da especificação JSON Web Encryption (RFC 7516):

O cabeçalho JOSE para um JWS pode ser diferenciado do cabeçalho JOSE para um JWE examinando o valor do parâmetro do cabeçalho “alg” (algoritmo). Se o valor representar uma assinatura digital ou algoritmo MAC, ou for o valor “none”, é para um JWS; se representar um algoritmo de Criptografia de Chave, Quebra de Chave, Acordo de Chave Direta, Acordo de Chave com Agrupamento de Chave ou Criptografia Direta, é para um JWE. (Extrair o valor “alg” para examinar é simples ao usar a serialização compacta do JWS ou a serialização compacta do JWE e pode ser mais difícil ao usar a serialização JWS JSON ou a serialização JWE JSON.)

E

O cabeçalho JOSE para um JWS também pode ser diferenciado do cabeçalho JOSE para um JWE determinando se existe um membro “enc” (algoritmo de criptografia). Se o membro “enc” existir, é um JWE; caso contrário, é um JWS.

JWS (assinado)## #

{
  "alg": "HS256"
}

JOGAR (criptografado) #

{
  "alg":"RSA1_5",
  "enc":"A256GCM",
  "iv":"__79_Pv6-fg",
  "x5t":"7noOPq-hJ1_hCnvWh6IeYI2w9Q0"
}

Criptografia da Web JSON (PLAY) #

JSON Web Encryption (JWE) representa o conteúdo criptografado usando estruturas de dados baseadas em JavaScript Object Notation (JSON). Ele define uma maneira de criptografar seus dados de declarações para que apenas o destinatário pretendido possa ler as informações presentes em um token.

Na Serialização JWE JSON, um JWE é representado como um objeto JSON contendo alguns ou todos esses oito membros:

  "protected", with the value BASE64URL(UTF8(JWE Protected Header))
  "unprotected", with the value JWE Shared Unprotected Header
  "header", with the value JWE Per-Recipient Unprotected Header
  "encrypted_key", with the value BASE64URL(JWE Encrypted Key)
  "iv", with the value BASE64URL(JWE Initialization Vector)
  "ciphertext", with the value BASE64URL(JWE Ciphertext)
  "tag", with the value BASE64URL(JWE Authentication Tag)
  "aad", with the value BASE64URL(JWE AAD)

As seis strings de resultados codificadas em base64url e os dois JSON desprotegidos os valores do objeto são representados como membros em um objeto JSON.

Exemplo de JOGO

O seguinte exemplo de cabeçalho JWE declara que:

  • a chave de criptografia de conteúdo é criptografada para o destinatário usando o algoritmo RSA-PKCS1_1.5 para produzir a chave criptografada JWE

  • o texto simples é criptografado usando o algoritmo AES-256-GCM para produzir o texto cifrado JWE

  • o vetor de inicialização de 64 bits especificado com a codificação base64url __79_Pv6-fg foi usado

  • a impressão digital do certificado X.509 que corresponde à chave usada para criptografar o JWE tem a codificação base64url 7noOPq-hJ1_hCnvWh6IeYI2w9Q0.

    { “alg”:“RSA1_5”, “enc”:“A256GCM”, “iv”:"__79_Pv6-fg", “x5t”:“7noOPq-hJ1_hCnvWh6IeYI2w9Q0” }

A codificação Base64url dos bytes da representação UTF-8 do cabeçalho JWE produz este valor de cabeçalho JWE codificado (com quebras de linha apenas para fins de exibição):

eyJhbGciOiJSU0ExXzUiLA0KICJlbmMiOiJBMjU2R0NNIiwNCiAiaXYiOiJfXzc5
X1B2Ni1mZyIsDQogIng1dCI6Ijdub09QcS1oSjFfaENudldoNkllWUkydzlRMCJ

Leia JSON Web Encryption Specification (RFC 7516) para obter mais informações

O que armazenar em um JWT #

O JWT RFC estabelece três classes de reivindicações:

  • Reivindicações registradas como sub, iss, exp ou nbf

  • Reivindicações públicas com nomes públicos ou nomes registrados pela IANA que contêm valores que devem ser exclusivos, como email, address ou phone_number. Veja lista completa

  • Reivindicações particulares para usar em seu próprio contexto e os valores podem colidir

Nenhuma dessas reivindicações é obrigatória

Um JWT é autocontido e deve evitar o uso da sessão do servidor fornecendo os dados necessários para realizar a autenticação (sem necessidade de armazenamento no servidor e acesso ao banco de dados). Portanto, informações de função ou permissões podem ser incluídas em declarações privadas do JWT.

Reivindicações Registradas #

Os seguintes nomes de reivindicação estão registrados no registro “JSON Web Token Claims” da IANA estabelecido pela Seção 10.1.

  • iss (emissor): identifica o principal que emitiu o JWT.
  • sub (assunto): identifica o principal que é o assunto do JWT. Deve ser único
  • aud (audiência): identifica os destinatários para os quais o JWT se destina (array de strings/uri)
  • exp (tempo de expiração): identifica o tempo de expiração (UTC Unix) após o qual você não deve mais aceitar este token. Deve ser após o horário emitido.
  • nbf(not before): identifica o horário UTC Unix antes do qual o JWT não deve ser aceito
  • iat (emitido em): identifica a hora UTC Unix em que o JWT foi emitido
  • jti (JWT ID): fornece um identificador exclusivo para o JWT.

Exemplo #

{
    "iss": "stackoverflow",
    "sub": "joe",
    "aud": ["all"],
    "iat": 1300819370,
    "exp": 1300819380,
    "jti": "3F2504E0-4F89-11D3-9A0C-0305E82C3301"
    "context": {
        "user": {
            "key": "joe",
            "displayName": "Joe Smith"
        },
        "roles":["admin","finaluser"]
    }
}
