Sintaxe #

  • { -code- } parou { -error- }{ -no-error- } ifelse % erro capturando frame

  • $error /errorname obter % stackunderflow typecheck rangecheck etc $error /newerror obtém % bool. coloque false para desativar o erro $error /ostack obtém % de cópia da pilha de operandos no ponto de erro

  • errordict /stackoverflow { -additional-code- /stackoverflow signalerror} put % executa código adicional em tipos de erros (aqui, o erro /stackoverflow).

Existem dois níveis para tratamento de erros em postscript. Essa dicotomia se aplica tanto à forma como o interpretador trata os erros quanto aos meios disponíveis ao usuário (programador) para controlar o tratamento.

O nível inferior é uma estrutura de controle incomum stop ... stop. stopped se comporta muito como uma construção de loop, pois estabelece uma marca na pilha de execução que pode ser saltada se o operador exit (para um loop) ou o operador stop (para um contexto stopped) é chamado. Ao contrário de uma construção de loop, stopped produz um Boolean na pilha indicando se stop foi chamado (caso contrário, o procedimento passado para stopped é conhecido por ter sido executado até a conclusão.

Quando ocorre um erro PostScript, como o stackunderflow talvez, o interpretador procura o nome do erro em errordict que reside em systemdict. Se o usuário não substituiu o procedimento em errordict, então o procedimento de erro padrão irá tirar instantâneos de toda a pilha e colocá-los em $error, outro dicionário em systemdict. Finalmente, o procedimento padrão chamará stop que desempilha o programa do usuário da pilha exec e executa o procedimento de impressão de erro do interpretador chamado handleerror em errordict.

Então, usando todo esse conhecimento, você pode pegar erros envolvendo uma seção de código em { ... } stop. Você pode relançar um erro chamando stop. Você pode determinar que tipo de erro ocorreu com $error /errorname get.

Você também pode alterar o comportamento padrão para um tipo específico de erro substituindo o procedimento por esse nome em errordict. Ou altere o formato de impressão do relatório de erros substituindo /handleerror em errordict.

Sequência de eventos quando um erro é sinalizado #

A sequência para um erro geralmente é:

  1. o erro é acionado procurando o nome do erro em errordict e executando este procedimento.
  2. o procedimento errordict chama signalerror, passando o nome do erro.
  3. signalerror tira instantâneos das pilhas, salvando os instantâneos em $error, e então chama stop.
  4. stop desempilha a pilha exec até o contexto fechado mais próximo estabelecido pelo operador parado.
  5. se o programa não tiver estabelecido seu próprio contexto parado para capturar o erro, ele será capturado por um nível externo stopped { errordict /handleerror get exec } if que foi chamado pelo código de inicialização para conter todo o programa do usuário .
  6. handleerror usa as informações em $error para imprimir um relatório de erro.

Sinalizando (jogando) um erro #

A maioria das ferramentas são padronizadas com exceção do nome do operador para lançar um erro. Nos intérpretes da Adobe, é chamado de .error. No ghostscript, é chamado de signalerror. Então com esta linha você pode usar signalerror no código postscript para intérpretes Adobe ou ghostscript ou xpost.

/.error where {pop /signalerror /.error load def} if

commandname errorname     signalerror     — Tire instantâneos da pilha em $error, depois em stop.

Por exemplo.

% my proc only accepts integer
/proc {
    dup type /integertype ne {
        /proc cvx /typecheck signalerror
    } if
    % ... rest of proc ...
} def

Existe um ponto atual? #

Fornece true na pilha se currentpoint for executado com sucesso, ou false se sinalizar um erro /nocurrentpoint.

{currentpoint pop pop} stopped not  % bool

Capturando um erro #

Como a ação final do manipulador de erros padrão é chamar stop, você pode capturar erros de operadores colocando o código em uma construção { ... } stop.

{
    0 array
    1 get
} stopped {
    $error /errorname get =
} if

imprimirá “rangecheck”, o erro sinalizado por get quando o índice estiver fora do intervalo aceitável para esse array.

Erros de relançamento #

Este trecho implementa um procedimento que se comporta como um operador de loop postscript. Se o usuário proc chama exit, ele pega o erro invalidexit para corrigir o dictstack para o end no final. Qualquer outro erro exceto invalidexit é relançado chamando stop.

% array n proc  .  -
% Like `forall` but delivers length=n subsequences produced by getinterval
/fortuple { 4 dict begin
    0 {offset proc n arr} {exch def} forall
    /arr load length n idiv
    {
        {
            /arr load offset n getinterval
            [ /proc load currentdict end /begin cvx ] cvx exec
            /offset offset n add def
        } stopped {
            $error /errorname get /invalidexit eq
            { 1 dict begin exit }{ stop } ifelse
        } if
    } repeat
end
} def
