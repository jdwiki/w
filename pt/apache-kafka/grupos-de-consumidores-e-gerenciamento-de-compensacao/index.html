Parâmetros #

Parâmetro Descrição
group.id O nome do Grupo de Consumidores.
enable.auto.commit Comprometa compensações automaticamente; padrão: verdadeiro.
auto.commit.interval.ms O atraso mínimo em milissegundos entre os commits (requer enable.auto.commit=true); padrão: 5000.
auto.offset.reset O que fazer quando não houver um deslocamento confirmado válido encontrado; padrão: mais recente.(+)
(+) Valores Possíveis Descrição
mais cedo Redefina automaticamente o deslocamento para o deslocamento mais antigo.
mais recente Redefinir automaticamente o deslocamento para o deslocamento mais recente.
nenhum Lança exceção para o consumidor se nenhum deslocamento anterior for encontrado para o grupo do consumidor.
qualquer outra coisa Lança exceção para o consumidor.

Gerenciamento de compensação do consumidor e tolerância a falhas #

KafkaConsumers solicitam mensagens de um corretor Kafka por meio de uma chamada para poll() e seu progresso é rastreado por offsets. Cada mensagem dentro de cada partição de cada tópico tem um chamado deslocamento atribuído—seu número de seqüência lógica dentro da partição. Um KafkaConsumer rastreia seu deslocamento atual para cada partição atribuída a ele. Preste atenção, que os corretores Kafka não estão cientes das atuais compensações dos consumidores. Assim, em poll() o consumidor precisa enviar seus deslocamentos atuais para o corretor, de modo que o corretor possa retornar as mensagens correspondentes, ou seja,. mensagens com um deslocamento consecutivo maior. Por exemplo, vamos supor que temos um único tópico de partição e um único consumidor com deslocamento atual 5. Em poll() o consumidor envia se deslocamento para o corretor e o corretor retorna mensagens para deslocamentos 6,7,8,.. .

Como os próprios consumidores rastreiam suas compensações, essas informações podem se perder se um consumidor falhar. Assim, os deslocamentos devem ser armazenados de forma confiável, de modo que, na reinicialização, um consumidor possa pegar seu antigo deslocamento e retomador de onde partiu. No Kafka, há suporte embutido para isso via commits de deslocamento. O novo KafkaConsumer pode enviar seu deslocamento atual para o Kafka e o Kafka armazena esses deslocamentos em um tópico especial chamado __consumer_offsets. Armazenar os deslocamentos em um tópico Kafka não é apenas tolerante a falhas, mas também permite reatribuir partições a outros consumidores durante um rebalanceamento. Como todos os consumidores de um grupo de consumidores podem acessar todos os deslocamentos confirmados de todas as partições, no rebalanceamento, um consumidor que obtém uma nova partição atribuída apenas lê o deslocamento confirmado dessa partição do tópico __consumer_offsets e retoma de onde o antigo consumidor parou.

Como cometer compensações #

KafkaConsumers pode confirmar deslocamentos automaticamente em segundo plano (parâmetro de configuração enable.auto.commit = true) qual é a configuração padrão. Esses auto commits são feitos dentro de poll() (que normalmente é chamado em um loop). Com que frequência os deslocamentos devem ser confirmados, pode ser configurado via auto.commit.interval.ms. Como os auto commits são embutidos em poll() e poll() é chamado pelo código do usuário, este parâmetro define um limite inferior para o intervalo entre commits.

Como alternativa à confirmação automática, os deslocamentos também podem ser gerenciados manualmente. Para isso, o auto commit deve ser desabilitado (enable.auto.commit = false). Para confirmação manual, o KafkaConsumers oferece dois métodos, a saber commitSync() e commitAsync(). Como o nome indica, commitSync() é uma chamada de bloqueio, que retorna após os deslocamentos serem confirmados com sucesso, enquanto commitAsync() retorna imediatamente. Se você quiser saber se um commit foi bem-sucedido ou não, você pode fornecer um manipulador de retorno de chamada (OffsetCommitCallback) um parâmetro de método. Preste atenção, que em ambas as chamadas de commit, o consumidor confirma os deslocamentos da última chamada poll(). Por exemplo. vamos supor um único tópico de partição com um único consumidor e a última chamada para poll() retornar mensagens com deslocamentos 4,5,6. Na confirmação, o deslocamento 6 será confirmado porque este é o último deslocamento rastreado pelo cliente consumidor. Ao mesmo tempo, commitSync() e commitAsync() permitem mais controle sobre qual deslocamento você deseja confirmar: se você usar as sobrecargas correspondentes que permitem especificar um Map<TopicPartition, OffsetAndMetadata> o consumidor confirmará apenas os deslocamentos especificados (ou seja, o mapa pode conter qualquer subconjunto de partições atribuídas e o deslocamento especificado pode ter qualquer valor).

Semântica de compensações confirmadas #

Um deslocamento confirmado indica que todas as mensagens até esse deslocamento já foram processadas. Assim, como os deslocamentos são números consecutivos, confirmar o deslocamento X implicitamente confirma todos os deslocamentos menores que X. Portanto, não é necessário confirmar cada deslocamento individualmente, e confirmar vários deslocamentos de uma só vez, mas apenas confirmar o maior deslocamento.

Preste atenção que, por design, também é possível confirmar um deslocamento menor do que o último deslocamento confirmado. Isso pode ser feito se as mensagens forem lidas uma segunda vez.

Garantias de processamento #

O uso de confirmação automática fornece semântica de processamento pelo menos uma vez. A suposição subjacente é que poll() só é chamado depois que todas as mensagens entregues anteriormente foram processadas com sucesso. Isso garante que nenhuma mensagem seja perdida porque um commit ocorre após o processamento. Se um consumidor falhar antes de um commit, todas as mensagens após o último commit serão recebidas do Kafka e processadas novamente. No entanto, essa nova tentativa pode resultar em duplicatas, pois algumas mensagens da última chamada poll() podem ter sido processadas, mas a falha ocorreu logo antes da chamada de confirmação automática.

Se a semântica de processamento no máximo uma vez for necessária, o auto commit deve ser desabilitado e um commitSync() manual diretamente após o poll() deve ser feito. Depois, as mensagens são processadas. Isso garante que as mensagens sejam confirmadas antes de serem processadas e, portanto, nunca lidas uma segunda vez. Claro, alguma mensagem pode se perder em caso de falha.

Como posso ler o tópico desde o início #

Existem várias estratégias para ler um tópico desde o início. Para explicá-los, primeiro precisamos entender o que acontece na inicialização do consumidor. Na inicialização de um consumidor, acontece o seguinte:

  1. junte-se ao grupo de consumidores configurado, que aciona um rebalanceamento e atribui partições ao consumidor
  2. procure por compensações confirmadas (para todas as partições que foram atribuídas ao consumidor)
  3. para todas as partições com deslocamento válido, retome a partir deste deslocamento
  4. para todas as partições com deslocamento inválido, defina o deslocamento inicial de acordo com o parâmetro de configuração auto.offset.reset

Iniciar um novo grupo de consumidores #

Se você quiser processar um tópico desde o início, você pode simplesmente iniciar um novo grupo de consumidores (ou seja, escolher um group.id não utilizado) e definir auto.offset.reset = mais cedo. Como não há deslocamentos confirmados para um novo grupo, a redefinição de deslocamento automático será acionada e o tópico será consumido desde o início. Preste atenção, que na reinicialização do consumidor, se você usar o mesmo group.id novamente, ele não lerá o tópico do início novamente, mas continuará de onde parou. Assim, para esta estratégia, você precisará atribuir um novo group.id toda vez que quiser ler um tópico desde o início.

Reutilize o mesmo ID de grupo #

Para evitar definir um novo group.id toda vez que você quiser ler um tópico desde o início, você pode desabilitar o auto commit (via enable.auto.commit = false) antes de iniciar o consumidor pela primeira vez (usando um group.id não utilizado e definindo auto.offset.reset = mais cedo). Além disso, você não deve confirmar nenhum deslocamento manualmente. Como os deslocamentos nunca são confirmados usando essa estratégia, na reinicialização, o consumidor lerá o tópico desde o início novamente.

No entanto, esta estratégia tem duas desvantagens:

  1. não é tolerante a falhas
  2. O reequilíbrio do grupo não funciona como pretendido

(1) Como os deslocamentos nunca são confirmados, uma falha e um consumidor interrompido são tratados da mesma maneira na reinicialização. Para ambos os casos, o tópico será consumido desde o início. (2) Como os deslocamentos nunca são confirmados, no rebalanceamento, as partições recém-atribuídas serão consumidoras desde o início.

Portanto, essa estratégia funciona apenas para grupos de consumidores com um único consumidor e deve ser usada apenas para fins de desenvolvimento.

Reutilize o mesmo ID de grupo e confirme #

Se você deseja ser tolerante a falhas e/ou usar vários consumidores em seu grupo de consumidores, é obrigatório confirmar compensações. Assim, se você quiser ler um tópico desde o início, precisará manipular os deslocamentos confirmados na inicialização do consumidor. Para isso, KafkaConsumer fornece três métodos seek(), seekToBeginning() e seekToEnd(). Enquanto seek() pode ser usado para definir um deslocamento arbitrário, o segundo e o terceiro método podem ser usados ​​para buscar o início ou o fim de uma partição, respectivamente. Assim, na falha e no reinício do consumidor, a busca seria omitida e o consumidor pode retomar de onde parou. Para o consumidor parar e reiniciar desde o início, seekToBeginning() seria chamado explicitamente antes de você entrar no seu loop poll(). Observe que seekXXX() só pode ser usado após um consumidor ingressar em um grupo – portanto, é necessário fazer uma “enquete fictícia” antes de usar seekXXX(). O código geral seria algo assim:

if (consumer-stop-and-restart-from-beginning) {
    consumer.poll(0); // dummy poll() to join consumer group
    consumer.seekToBeginning(...);
}

// now you can start your poll() loop
while (isRunning) {
    for (ConsumerRecord record : consumer.poll(0)) {
        // process a record
    }
}

O que é um Grupo de Consumidores #

A partir do Kafka 0.9, o novo cliente de alto nível KafkaConsumer está disponível. Ele explora um novo protocolo Kafka integrado que permite combinar vários consumidores em um chamado Grupo de consumidores. Um Grupo de Consumidores pode ser descrito como um único consumidor lógico que assina um conjunto de tópicos. As partições sobre todos os tópicos são atribuídas aos consumidores físicos dentro do grupo, de modo que cada partição seja atribuída a exatamente um consumidor (um único consumidor pode obter várias partições atribuídas). Os consumidores individuais pertencentes ao mesmo grupo podem rodar em diferentes hosts de forma distribuída.

Grupos de consumidores são identificados por meio de seu group.id. Para tornar uma instância de cliente específica membro de um grupo de consumidores, basta atribuir os grupos group.id a este cliente, por meio da configuração do cliente:

Properties props = new Properties();
props.put("group.id", "groupName");
// ...some more properties required
new KafkaConsumer<K, V>(config);

Assim, todos os consumidores que se conectam ao mesmo cluster Kafka e usam o mesmo group.id formam um Grupo de Consumidores. Os consumidores podem sair de um grupo a qualquer momento e novos consumidores podem ingressar em um grupo a qualquer momento. Para ambos os casos, um chamado rebalance é acionado e as partições são reatribuídas com o Grupo de Consumidores para garantir que cada partição seja processada por exatamente um consumidor dentro do grupo.

Preste atenção, que mesmo um único KafkaConsumer forma um Grupo de Consumidores com ele mesmo como único membro.
