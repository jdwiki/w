Como analisar HTML ou XML em um documento Nokogiri XML ou HTML #

Não há muito a acrescentar ao tutorial “Parsing an HTML/XML Document” de Nokogiri, que é uma introdução fácil ao assunto, então comece por aí e depois retorne a esta página para ajudar a preencher algumas lacunas.

A análise básica do Nokogiri tenta limpar um documento malformado, às vezes adicionando tags de fechamento ausentes, e adicionará algumas tags adicionais para torná-lo correto.

Este é um exemplo de dizer a Nokogiri que o documento que está sendo analisado é um arquivo HTML completo, e Nokogiri descobrindo que não é:

require 'nokogiri'

doc = Nokogiri::HTML('<body></body>')
puts doc.to_html

Quais saídas:

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body></body></html>

Observe que a instrução DTD foi adicionada, juntamente com uma tag <html>.

Se quisermos evitar isso, podemos analisar o documento como um DocumentFragment:

require 'nokogiri'

doc = Nokogiri::HTML.fragment('<body></body>')
puts doc.to_html

que agora gera apenas o que foi realmente passado:

<body></body>

Há uma variante XML também:

require 'nokogiri'

doc = Nokogiri::XML('<node />')
puts doc.to_xml

Quais saídas:

<?xml version="1.0"?>
<node/>

e:

doc = Nokogiri::XML.fragment('<node />')
puts doc.to_xml

resultando em:

<node/>

Uma variação mais detalhada de fragment é usar DocumentFragment.parse, então às vezes você verá escrito dessa forma.

Ocasionalmente, Nokogiri terá que fazer algumas correções para tentar entender o documento:

doc = Nokogiri::XML::DocumentFragment.parse('<node ><foo/>')
puts doc.to_xml

Com o código modificado agora sendo:

<node>
  <foo/>
</node>

O mesmo pode acontecer com HTML.

Às vezes, o documento é mutilado além da capacidade do Nokogiri de corrigi-lo, mas ele tentará de qualquer maneira, resultando em um documento com uma hierarquia alterada. O Nokogiri não levantará uma exceção, mas fornece uma maneira de verificar erros e as ações realizadas. Consulte “https://www.wikiod.com/pt/nokogiri/introducao-ao-nokogiri#Como verificar erros de análise” para obter mais informações.

Veja a documentação do Nokogiri::XML::ParseOptions para várias opções usadas ao analisar.

Como verificar erros de análise #

O Nokogiri é um pouco como um navegador, pois tentará fornecer algo útil mesmo se o HTML ou XML de entrada estiver malformado. Infelizmente, geralmente faz isso silenciosamente, mas podemos pedir uma lista dos erros usando errors:

require 'nokogiri'

doc = Nokogiri::XML('<node><foo/>')
doc.errors
# => [#<Nokogiri::XML::SyntaxError: 1:13: FATAL: Premature end of data in tag node line 1>]

enquanto que o XML correto não resultaria em erros:

doc = Nokogiri::XML('<node><foo/></node>')
doc.errors
# => []

Isso também se aplica à análise de HTML, mas, como o HTML é uma forma relaxada de XML, o Nokogiri passará por nós finais ausentes com frequência e relatará apenas nós malformados e mais erros patológicos:

doc = Nokogiri::HTML('<html><body>')
doc.errors
# => []

doc = Nokogiri::HTML('<html><body><p')
doc.errors
# => [#<Nokogiri::XML::SyntaxError: 1:15: ERROR: Couldn't find end of Start Tag p>]

Se, após a análise, você não encontrar um nó que possa ver em seu editor, essa pode ser a causa do problema. Às vezes, é útil passar o HTML por um formatador e ver se o aninhamento ajuda a revelar o problema.

E, como o Nokogiri tenta corrigir o problema, mas às vezes não consegue fazê-lo corretamente, porque pode ser uma coisa muito difícil para o software fazer, teremos que pré-processar o arquivo e retocar as linhas antes de entregá-lo para Nokogiri. Como fazer isso depende do arquivo e do problema. Pode variar desde simplesmente encontrar um nó e adicionar um > à direita, até remover a marcação malformada incorporada que foi injetada por uma rotina de raspagem ruim, então cabe ao programador a melhor forma de interceder.

Como extrair texto de um nó ou nós #

Como extrair corretamente o texto dos nós é uma das perguntas mais populares que vemos, e quase invariavelmente é dificultada pelo mau uso dos métodos de “pesquisa” de Nokogiri.

Nokogiri suporta o uso de seletores CSS e XPath. Estes são equivalentes:

doc.at('p').text   # => "foo"
doc.at('//p').text # => "foo"

doc.search('p').size   # => 2
doc.search('//p').size # => 2

Os seletores CSS são estendidos com muitas das extensões CSS do jQuery por conveniência.

at e search são versões genéricas de at_css e at_xpath junto com css e xpath. Nokogiri faz uma tentativa de determinar se um seletor CSS ou XPath está sendo passado. É possível criar um seletor que engana at ou search então ocasionalmente ele será mal interpretado, e é por isso que temos as versões mais específicas dos métodos . Em geral eu uso as versões genéricas quase sempre, e só uso a versão específica se eu achar que o Nokogiri vai entender mal. Esta prática se enquadra na primeira entrada em “Três Virtudes”.

Se você estiver procurando por um nó específico e quiser seu texto, use at ou uma de suas variantes at_css ou at_xpath:

require 'nokogiri'

doc = Nokogiri::HTML(<<EOT)
<html>
  <body>
    <p>foo</p>
    <p>bar</p>
  </body>
</html>
EOT

doc.at('p').text # => "foo"

at é equivalente a search(...).first, então você poderia usar a versão mais longa para digitar, mas por quê?

Se o texto que está sendo extraído for concatenado após usar search, css ou xpath, adicione map(&:text) em vez de simplesmente usar text:

require 'nokogiri'

doc = Nokogiri::HTML(<<EOT)
<html>
  <body>
    <p>foo</p>
    <p>bar</p>
  </body>
</html>
EOT

doc.search('p').text        # => "foobar"
doc.search('p').map(&:text) # => ["foo", "bar"]

Consulte a documentação do text para NodeSet e Node para informações adicionais.
