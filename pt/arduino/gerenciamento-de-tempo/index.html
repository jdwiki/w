Sintaxe #

  • millis longo não assinado ()

  • micros longos não assinados()

  • atraso nulo (longos milissegundos não assinados)

  • void delayMicroseconds (microssegundos longos não assinados)

  • Veja o cabeçalho elapsedMillis para construtores e operadores dessa classe. Resumidamente:

  • elapsedMillis elapsedMillisObject; cria um objeto para acompanhar o tempo desde que foi criado ou desde algum outro ponto definido explicitamente no tempo

  • decorridoMillisObject = 0; redefinir o tempo rastreado pelo objeto para “desde agora”

  • deltaT longo sem sinal = elapsedMillisObject; vamos ver o tempo rastreado

  • elapsedMillisObject += e -= estes funcionam como esperado

Código bloqueante vs. não bloqueante #

Para esboços muito simples, escrever código de bloqueio usando delay() e delayMicroseconds() pode ser apropriado. Quando as coisas ficam mais complexas, usar essas funções pode ter algumas desvantagens. Alguns destes são:

  • Desperdiçando tempo de CPU: Esboços mais complexos podem precisar da CPU para outra coisa enquanto aguardam o término de um período de piscar do LED.
  • atrasos inesperados: quando delay() é chamado em sub-rotinas que não são obviamente chamadas, por exemplo em bibliotecas que você inclui.
  • eventos ausentes que acontecem durante o atraso e não são tratados por um manipulador de interrupção, por exemplo, pressionamentos de botão polled: Um botão pode ser pressionado por 100 ms, mas isso pode ser sombreado por um delay(500).

Detalhes de implementação #

millis() geralmente depende de um temporizador de hardware que roda a uma velocidade muito superior a 1 kHz. Quando millis() é chamado, a implementação retorna algum valor, mas você não sabe quantos anos ele realmente tem. É possível que o milissegundo “atual” tenha começado ou que termine logo após essa chamada de função. Isso significa que, ao calcular a diferença entre dois resultados de millis(), você pode estar errado por qualquer coisa entre quase zero e quase um milissegundo. Use micros() se for necessária maior precisão.

Olhando para o código fonte do elapsedMillis revela que ele realmente usa millis() internamente para comparar dois pontos no tempo, então ele também sofre com esse efeito. Novamente, há a alternativa elapsedMicros para maior precisão, da mesma biblioteca.

Blinky sem bloqueio com millis() #

Isso é muito próximo de um exemplo dos documentos do arduino:

// set constants for blinking the built-in LED at 1 Hz
#define OUTPIN LED_BUILTIN
#define PERIOD 500  // this is in milliseconds

int ledState = LOW;

// millis() returns an unsigned long so we'll use that to keep track of time
unsigned long lastTime = 0;

void setup() {
  // set the digital pin as output:
  pinMode(OUTPIN, OUTPUT);
}

void loop() {
  unsigned long now = millis();
  if (now - lastTime >= PERIOD) // this will be true every PERIOD milliseconds
  {
    lastTime = now;
    if (ledState == LOW)
    {
      ledState = HIGH;
    }
    else
    {
      ledState = LOW;
    }
    digitalWrite(OUTPIN, ledState);
  }

  // now there's lots of time to do other stuff here
}

Usar millis() desta forma - para cronometrar operações de maneira não bloqueante - é algo que é necessário com bastante frequência, então considere usar a biblioteca elapsedMillis para isso.

Blinky sem bloqueio com a biblioteca elapsedMillis (e classe) #

A biblioteca elapsedMillis fornece uma classe com o mesmo nome que registra o tempo decorrido desde que foi criada ou definida com um determinado valor:

#include <elapsedMillis.h>

#define OUTPIN LED_BUILTIN
#define PERIOD 500

elapsedMillis ledTime;

bool ledState = false;

void setup() 
{                
  // initialize the digital pin as an output.
  pinMode(OUTPIN, OUTPUT);     
}

void loop()
{
    if (ledTime >= PERIOD) 
    {                
        ledState = !ledState;
        digitalWrite(OUTPIN, ledState);
        ledTime = 0;
    }
    // do other stuff here
}

Você pode ver no exemplo que o objeto ledTime é atribuído a zero quando o pino do LED foi alternado. Isso pode não ser surpreendente à primeira vista, mas tem um efeito se coisas mais demoradas estiverem acontecendo:

Considere uma situação em que a comparação entre ledTime e PERIOD é feita após 750 milissegundos. Em seguida, definir ledTime para zero significa que todas as operações de alternância seguintes serão 250 ms “atrasadas”. Se, por outro lado, PERIOD fosse subtraído de ledTime, o LED veria um curto período e continuaria piscando como se nada tivesse acontecido.

bloqueando o blinky com delay() #

Uma das maneiras mais simples de fazer um LED piscar é: ligue-o, espere um pouco, desligue-o, espere novamente e repita sem parar:

// set constants for blinking the built-in LED at 1 Hz
#define OUTPIN LED_BUILTIN
#define PERIOD 500

void setup()
{
  pinMode(OUTPIN, OUTPUT);      // sets the digital pin as output
}

void loop()
{
  digitalWrite(OUTPIN, HIGH);   // sets the pin on
  delayMicroseconds(PERIOD);        // pauses for 500 miliseconds      
  digitalWrite(OUTPIN, LOW);    // sets the pin off
  delayMicroseconds(PERIOD);        // pauses for 500 milliseconds

  // doing other time-consuming stuff here will skew the blinking
}

No entanto, esperar como feito no exemplo acima desperdiça ciclos de CPU, porque ele fica lá em um loop esperando um certo ponto no tempo passar. É isso que as formas sem bloqueio, usando millis() ou elapsedMillis, fazem melhor - no sentido de que elas não queimam tanto os recursos do hardware.

Meça quanto tempo algo levou, usando elapsedMillis e elapsedMicros #

#include <elapsedMillis.h>

void setup() {
  Serial.begin(115200);
  elapsedMillis msTimer;
  elapsedMicros usTimer;

  long int dt = 500;
  delay(dt);

  long int us = usTimer;
  long int ms = msTimer;

  Serial.print("delay(");Serial.print(dt);Serial.println(") took");
  Serial.print(us);Serial.println(" us, or");
  Serial.print(ms);Serial.println(" ms");
}

void loop() {
}

Neste exemplo, um objeto elapsedMillis e um objeto elapsedMicros são usados ​​para medir quanto tempo algo levou, criando-os logo antes da expressão que queremos cronometrar ser executada, e obtendo seus valores depois. Eles mostrarão resultados ligeiramente diferentes, mas o resultado de milissegundos não será mais de um milissegundo.

Mais de 1 tarefa sem atraso() #

Se você tiver mais de 1 tarefa para executar repetidamente em intervalos diferentes, use este exemplo como ponto de partida:

unsigned long intervals[] = {250,2000}; //this defines the interval for each task in milliseconds
unsigned long last[] = {0,0};           //this records the last executed time for each task

void setup() {
  pinMode(LED_BUILTIN, OUTPUT); //set the built-it led pin as output
  Serial.begin(115200);         //initialize serial
}

void loop() {
  unsigned long now = millis();
  if(now-last[0]>=intervals[0]){ last[0]=now; firstTask(); }
  if(now-last[1]>=intervals[1]){ last[1]=now; secondTask(); }
  
  //do other things here
}

void firstTask(){
  //let's toggle the built-in led
  digitalWrite(LED_BUILTIN, digitalRead(LED_BUILTIN)?0:1);
}

void secondTask(){
  //say hello
  Serial.println("hello from secondTask()");
}

Para adicionar outra tarefa para executar a cada 15 segundos, estenda as variáveis ​​intervals e last:

unsigned long intervals[] = {250,2000,15000};
unsigned long last[] = {0,0,0};

Em seguida, adicione uma instrução if para executar a nova tarefa. Neste exemplo, chamei-o de thirdTask.

if(now-last[2]>=intervals[2]){ last[2]=now; thirdTask(); }

Por fim, declare a função:

void thirdTask(){
  //your code here
}
