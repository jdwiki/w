SubGit #

SubGit pode ser usado para realizar uma importação única de um repositório SVN para o git.

$ subgit import --non-interactive --svn-url http://svn.my.co/repos/myproject myproject.git

Migrar do SVN para o Git usando o utilitário de conversão Atlassian #

Baixe o utilitário de conversão Atlassian aqui. Este utilitário requer Java, portanto, certifique-se de ter o Java Runtime Environment JRE instalado na máquina em que planeja fazer a conversão.

Use o comando java -jar svn-migration-scripts.jar verify para verificar se sua máquina está faltando algum dos programas necessários para completar a conversão. Especificamente, este comando verifica os utilitários Git, subversion e git-svn. Ele também verifica se você está executando a migração em um sistema de arquivos com distinção entre maiúsculas e minúsculas. A migração para o Git deve ser feita em um sistema de arquivos com distinção entre maiúsculas e minúsculas para evitar corromper o repositório.

Em seguida, você precisa gerar um arquivo de autores. O Subversion rastreia as alterações apenas pelo nome de usuário do committer. O Git, no entanto, usa duas informações para distinguir um usuário: um nome real e um endereço de e-mail. O comando a seguir irá gerar um arquivo de texto mapeando os nomes de usuário do subversion para seus equivalentes no Git:

java -jar svn-migration-scripts.jar authors <svn-repo> authors.txt

onde <svn-repo> é a URL do repositório do subversion que você deseja converter. Após executar este comando, as informações de identificação dos contribuidores serão mapeadas em authors.txt. Os endereços de e-mail terão o formato <username>@mycompany.com. No arquivo de autores, você precisará alterar manualmente o nome padrão de cada pessoa (que por padrão se tornou seu nome de usuário) para seus nomes reais. Certifique-se também de verificar se todos os endereços de e-mail estão corretos antes de continuar.

O comando a seguir irá clonar um repositório svn como um repositório Git:

git svn clone --stdlayout --authors-file=authors.txt <svn-repo> <git-repo-name>

onde <svn-repo> é o mesmo URL do repositório usado acima e <git-repo-name> é o nome da pasta no diretório atual para clonar o repositório. Há algumas considerações antes de usar este comando:

  • O sinalizador --stdlayout acima informa ao Git que você está usando um layout padrão com as pastas trunk, branch e tags. Os repositórios do Subversion com layouts não padrão requerem que você especifique os locais da pasta trunk, qualquer/todas as pastas branch e a pasta tags. Isso pode ser feito seguindo este exemplo: git svn clone --trunk=/trunk --branches=/branches --branches=/bugfixes --tags=/tags --authors-file=authors.txt <svn-repo > <git-repo-name>.
  • Este comando pode levar muitas horas para ser concluído, dependendo do tamanho do seu repositório.
  • Para reduzir o tempo de conversão para grandes repositórios, a conversão pode ser executada diretamente no servidor que hospeda o repositório do subversion para eliminar a sobrecarga da rede.

git svn clone importa as ramificações do subversion (e tronco) como ramificações remotas incluindo tags do subversion (ramificações remotas prefixadas com tags/). Para convertê-los em ramificações e tags reais, execute os comandos a seguir em uma máquina Linux na ordem em que são fornecidos. Depois de executá-los, git branch -a deve mostrar os nomes corretos das ramificações, e git tag -l deve mostrar as tags do repositório.

git for-each-ref refs/remotes/origin/tags | cut -d / -f 5- | grep -v @ | while read tagname; do git tag $tagname origin/tags/$tagname; git branch -r -d origin/tags/$tagname; done
git for-each-ref refs/remotes | cut -d / -f 4- | grep -v @ | while read branchname; do git branch "$branchname" "refs/remotes/origin/$branchname"; git branch -r -d "origin/$branchname"; done

A conversão de svn para Git agora está completa! Basta ’enviar’ seu repositório local para um servidor e você pode continuar contribuindo usando o Git, além de ter um histórico de versão completamente preservado do svn.

Migrar do Team Foundation Version Control (TFVC) para o Git #

Você pode migrar do controle de versão da base da equipe para o git usando uma ferramenta de código aberto chamada Git-TF. A migração também transferirá seu histórico existente convertendo checkins tfs em commits git.

Para colocar sua solução no Git usando o Git-TF, siga estas etapas:

Baixe o Git-TF

Você pode baixar (e instalar) Git-TF do Codeplex: Git-TF @ Codeplex

Clone sua solução TFVC

Inicie o powershell (win) e digite o comando

git-tf clone http://my.tfs.server.address:port/tfs/mycollection '$/myproject/mybranch/mysolution' --deep

A opção –deep é a palavra-chave a ser observada, pois diz ao Git-Tf para copiar seu histórico de check-in. Agora você tem um repositório git local na pasta da qual você chamou seu comando cloe.

Limpar

  • Adicione um arquivo .gitignore. Se você estiver usando o Visual Studio, o editor pode fazer isso para você, caso contrário, você pode fazer isso manualmente baixando um arquivo completo de github/gitignore.
  • Remova as ligações de controle de origem do TFS da solução (remova todos os arquivos *.vssscc). Você também pode modificar seu arquivo de solução removendo o GlobalSection(TeamFoundationVersionControl)……EndClobalSection

Commit & Push

Conclua sua conversão confirmando e enviando seu repositório local para seu controle remoto.

git add .
git commit -a -m "Coverted solution source control from TFVC to Git"

git remote add origin https://my.remote/project/repo.git

git push origin master

Migrando o Mercurial para o Git #

Pode-se usar os seguintes métodos para importar um repositório Mercurial para o Git:

  1. Usando exportação rápida:

    cd git clone git://repo.or.cz/fast-export.git git init git_repo cd git_repo ~/fast-export/hg-fast-export.sh -r /path/to/old/mercurial_repo git checkout HEAD

  2. Usando Hg-Git: Uma resposta muito detalhada aqui: https://stackoverflow.com/a/31827990/5283213

  3. Usando o Importador do GitHub: Siga as instruções (detalhadas) em GitHub.

Migrar de SVN para Git usando svn2git #

svn2git é um wrapper Ruby em torno do suporte SVN nativo do git através do git-svn, ajudando você com a migração de projetos do Subversion para o Git, mantendo o histórico (incl. histórico de troncos, tags e branches).

Exemplos

Para migrar um repositório svn com o layout padrão (ou seja, branches, tags e trunk no nível raiz do repositório):

$ svn2git http://svn.example.com/path/to/repo

Para migrar um repositório svn que não está no layout padrão:

$ svn2git http://svn.example.com/path/to/repo --trunk trunk-dir --tags tags-dir --branches branches-dir

Caso você não queira migrar (ou não tenha) branches, tags ou trunk você pode usar as opções --notrunk, --nobranches e --notags.

Por exemplo, $ svn2git http://svn.example.com/path/to/repo --trunk trunk-dir --notags --nobranches migrará apenas o histórico do tronco.

Para reduzir o espaço requerido pelo seu novo repositório, você pode querer excluir quaisquer diretórios ou arquivos que você adicionou, mas que não deveria ter (por exemplo, diretório de compilação ou arquivos):

$ svn2git http://svn.example.com/path/to/repo --exclude build --exclude '.*\.zip$'

Otimização pós-migração

Se você já tem alguns milhares de commits (ou mais) em seu repositório git recém-criado, você pode querer reduzir o espaço usado antes de enviar seu repositório em um controle remoto. Isso pode ser feito usando o seguinte comando:

$ git gc --aggressive

Observação: O comando anterior pode levar várias horas em repositórios grandes (dezenas de milhares de commits e/ou centenas de megabytes de histórico).
