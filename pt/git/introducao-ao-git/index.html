Crie seu primeiro repositório e adicione e confirme os arquivos #

Na linha de comando, primeiro verifique se o Git está instalado:

Em todos os sistemas operacionais:

git --version

Em sistemas operacionais do tipo UNIX:

which git

Se nada for retornado ou o comando não for reconhecido, talvez seja necessário instalar o Git em seu sistema baixando e executando o instalador. Consulte a página inicial do Git para obter instruções de instalação excepcionalmente claras e fáceis.

Depois de instalar o Git, [configure seu nome de usuário e endereço de e-mail][2]. Faça isso antes de fazer um commit.

Depois que o Git estiver instalado, navegue até o diretório que você deseja colocar sob controle de versão e crie um repositório Git vazio:

git init

Isso cria uma pasta oculta, .git, que contém o encanamento necessário para o Git funcionar.

Em seguida, verifique quais arquivos o Git adicionará ao seu novo repositório; esta etapa merece cuidados especiais:

git status

Revise a lista de arquivos resultante; você pode dizer ao Git qual dos arquivos colocar no controle de versão (evite adicionar arquivos com informações confidenciais, como senhas, ou arquivos que apenas atrapalham o repositório):

git add <file/directory name #1> <file/directory name #2> < ... >

Se todos os arquivos da lista devem ser compartilhados com todos que têm acesso ao repositório, um único comando adicionará tudo em seu diretório atual e seus subdiretórios:

git add .

Isto irá [“estar”][3] todos os arquivos a serem adicionados ao controle de versão, preparando-os para serem submetidos em seu primeiro commit.

Para arquivos que você nunca quer sob controle de versão, crie e preencha um arquivo chamado .gitignore antes executando o comando add.

Confirme todos os arquivos que foram adicionados, juntamente com uma mensagem de confirmação:

git commit -m "Initial commit"

Isso cria um novo [commit][4] com a mensagem fornecida. Um commit é como um save ou snapshot de todo o seu projeto. Agora você pode [push][5] ou carregá-lo para um repositório remoto e depois voltar a ele, se necessário. Se você omitir o parâmetro -m, seu editor padrão será aberto e você poderá editar e salvar a mensagem de confirmação lá.

Adicionando um controle remoto

Para adicionar um novo controle remoto, use o comando git remote add no terminal, no diretório em que seu repositório está armazenado.

O comando git remote add recebe dois argumentos:

  1. Um nome remoto, por exemplo, origin

  2. Um URL remoto, por exemplo, https://<your-git-service-address>/user/repo.git

     git remote add origin https://<your-git-service-address>/owner/repository.git

NOTA: Antes de adicionar o controle remoto, você deve criar o repositório necessário em seu serviço git, Você poderá enviar/puxar commits após adicionar seu controle remoto.

Clonar um repositório #

O comando git clone é usado para copiar um repositório Git existente de um servidor para a máquina local.

Por exemplo, para clonar um projeto do GitHub:

cd <path where you'd like the clone to create a directory>
git clone https://github.com/username/projectname.git

Para clonar um projeto BitBucket:

cd <path where you'd like the clone to create a directory>
git clone https://yourusername@bitbucket.org/username/projectname.git

Isso cria um diretório chamado projectname na máquina local, contendo todos os arquivos no repositório Git remoto. Isso inclui arquivos de origem para o projeto, bem como um subdiretório .git que contém todo o histórico e configuração do projeto.

Para especificar um nome diferente do diretório, por exemplo. MinhaPasta:

git clone https://github.com/username/projectname.git MyFolder

Ou para clonar no diretório atual:

git clone https://github.com/username/projectname.git .

Observação:

  1. Ao clonar para um diretório especificado, o diretório deve estar vazio ou inexistente.

  2. Você também pode usar a versão ssh do comando:

    git clone git@github.com:username/projectname.git

A versão https e a versão ssh são equivalentes. No entanto, alguns serviços de hospedagem como o GitHub recomendam que você usa https em vez de ssh.

Código de compartilhamento #

Para compartilhar seu código, você cria um repositório em um servidor remoto para o qual copiará seu repositório local.

Para minimizar o uso de espaço no servidor remoto você cria um repositório vazio: um que possui apenas os objetos .git e não cria uma cópia de trabalho no sistema de arquivos. Como bônus, você defina este controle remoto como um servidor upstream para compartilhar atualizações facilmente com outros programadores.

No servidor remoto:

git init --bare /path/to/repo.git

Na máquina local:

git remote add origin ssh://username@server:/path/to/repo.git

(Observe que ssh: é apenas uma maneira possível de acessar o repositório remoto.)

Agora copie seu repositório local para o remoto:

git push --set-upstream origin master

Adicionar --set-upstream (ou -u) criou uma referência upstream (rastreamento) que é usada por comandos Git sem argumentos, por exemplo, git pull.

Configurando seu nome de usuário e e-mail #

You need to set who you are *before* creating any commit.  That will allow commits to have the right author name and email associated to them.

Não tem nada a ver com autenticação ao enviar para um repositório remoto (por exemplo, ao enviar para um repositório remoto usando sua conta GitHub, BitBucket ou GitLab)

Para declarar essa identidade para todos os repositórios, use git config --global Isto irá armazenar a configuração no arquivo .gitconfig do seu usuário: ex. $HOME/.gitconfig ou para Windows, %USERPROFILE%\.gitconfig.

git config --global user.name "Your Name"
git config --global user.email mail@example.com

Para declarar uma identidade para um único repositório, use git config dentro de um repositório. Isto irá armazenar a configuração dentro do repositório individual, no arquivo $GIT_DIR/config. por exemplo. /path/to/your/repo/.git/config.

cd /path/to/my/repo
git config user.name "Your Login At Work"
git config user.email mail_at_work@example.com

As configurações armazenadas no arquivo de configuração de um repositório terão precedência sobre a configuração global quando você usar esse repositório.

Dicas: se você tiver identidades diferentes (uma para projeto de código aberto, uma no trabalho, uma para repositórios privados, …), e não quiser esquecer de definir a correta para cada repositório diferente em que estiver trabalhando :

  • Remover uma identidade global

    git config --global --remove-section user.name
git config --global --remove-section user.email

  • Para forçar o git a procurar sua identidade apenas nas configurações de um repositório, não na configuração global:

      git config --global user.useConfigOnly true

Dessa forma, se você esquecer de definir seu user.name e user.email para um determinado repositório e tentar fazer um commit, você verá:

no name was given and auto-detection is disabled
no email was given and auto-detection is disabled

Configurando o controle remoto upstream #

Se você clonou um fork (por exemplo, um projeto de código aberto no Github), talvez não tenha acesso push ao repositório upstream, portanto, você precisa do seu fork, mas pode buscar o repositório upstream.

Primeiro verifique os nomes remotos:

$ git remote -v
origin    https://github.com/myusername/repo.git (fetch)
origin    https://github.com/myusername/repo.git (push)
upstream  # this line may or may not be here

Se o upstream já existe (está em algumas versões do Git), você precisa definir o URL (atualmente está vazio):

$ git remote set-url upstream https://github.com/projectusername/repo.git

Se o upstream não estiver lá, ou se você também quiser adicionar o fork de um amigo/colega (atualmente eles não existem):

$ git remote add upstream https://github.com/projectusername/repo.git
$ git remote add dave https://github.com/dave/repo.git

Aprendendo sobre um comando #

Para obter mais informações sobre qualquer comando git – ou seja, detalhes sobre o que o comando faz, opções disponíveis e outra documentação – use a opção --help ou o comando help.

Por exemplo, para obter todas as informações disponíveis sobre o comando git diff, use:

git diff --help
git help diff

Da mesma forma, para obter todas as informações disponíveis sobre o comando status, use:

git status --help
git help status

Se você quiser apenas uma ajuda rápida mostrando o significado dos sinalizadores de linha de comando mais usados, use -h:

git checkout -h

Configurar SSH para Git #

Se você estiver usando o Windows, abra o Git Bash. Se você estiver usando Mac ou Linux, abra seu Terminal.

Antes de gerar uma chave SSH, você pode verificar se tem alguma chave SSH existente.

Liste o conteúdo do seu diretório ~/.ssh:

$ ls -al ~/.ssh 
# Lists all the files in your ~/.ssh directory

Verifique a listagem do diretório para ver se você já tem uma chave SSH pública. Por padrão, os nomes dos arquivos das chaves públicas são um dos seguintes:

id_dsa.pub
id_ecdsa.pub
id_ed25519.pub
id_rsa.pub

Se você vir um par de chaves pública e privada listado que gostaria de usar em sua conta Bitbucket, GitHub (ou similar), você pode copiar o conteúdo do arquivo id_*.pub.

Caso contrário, você pode criar um novo par de chaves pública e privada com o seguinte comando:

$ ssh-keygen

Pressione a tecla Enter ou Return para aceitar o local padrão. Digite e reinsira uma senha quando solicitado ou deixe-a em branco.

Certifique-se de que sua chave SSH seja adicionada ao ssh-agent. Inicie o ssh-agent em segundo plano se ainda não estiver em execução:

$ eval "$(ssh-agent -s)"

Adicione sua chave SSH ao agente ssh. Observe que você precisará substituir id_rsa no comando pelo nome do seu arquivo de chave privada:

$ ssh-add ~/.ssh/id_rsa

Se você deseja alterar o upstream de um repositório existente de HTTPS para SSH, você pode executar o seguinte comando:

$ git remote set-url origin ssh://git@bitbucket.server.com:7999/projects/your_project.git

Para clonar um novo repositório via SSH, você pode executar o seguinte comando:

$ git clone ssh://git@bitbucket.server.com:7999/projects/your_project.git

Instalação do Git #

Vamos começar a usar um pouco de Git. Primeiras coisas primeiro - você tem que instalá-lo. Você pode obtê-lo de várias maneiras; as duas principais são instalá-lo a partir do código-fonte ou instalar um pacote existente para sua plataforma.

Instalando da fonte

Se você puder, geralmente é útil instalar o Git da fonte, porque você obterá a versão mais recente. Cada versão do Git tende a incluir aprimoramentos úteis da interface do usuário, portanto, obter a versão mais recente geralmente é o melhor caminho se você se sentir confortável compilando o software da fonte. Também é o caso de muitas distribuições Linux conterem pacotes muito antigos; então, a menos que você esteja em uma distro muito atualizada ou esteja usando backports, instalar a partir da fonte pode ser a melhor aposta.

Para instalar o Git, você precisa ter as seguintes bibliotecas das quais o Git depende: curl, zlib, openssl, expat e libiconv. Por exemplo, se você estiver em um sistema que tenha yum (como Fedora) ou apt-get (como um sistema baseado em Debian), você pode usar um destes comandos para instalar todas as dependências:

$ yum install curl-devel expat-devel gettext-devel \
  openssl-devel zlib-devel

$ apt-get install libcurl4-gnutls-dev libexpat1-dev gettext \
  libz-dev libssl-dev

Quando você tiver todas as dependências necessárias, você pode ir em frente e pegar o snapshot mais recente do site do Git:

http://git-scm.com/download Em seguida, compile e instale:

$ tar -zxf git-1.7.2.2.tar.gz
$ cd git-1.7.2.2
$ make prefix=/usr/local all
$ sudo make prefix=/usr/local install

Feito isso, você também pode obter o Git via Git para atualizações:

$ git clone git://git.kernel.org/pub/scm/git/git.git

Instalando no Linux

Se você deseja instalar o Git no Linux por meio de um instalador binário, geralmente pode fazê-lo por meio da ferramenta básica de gerenciamento de pacotes que acompanha sua distribuição. Se você estiver no Fedora, você pode usar o yum:

$ yum install git

Ou se você estiver em uma distribuição baseada em Debian como o Ubuntu, tente apt-get:

$ apt-get install git

Instalando no Mac

Existem três maneiras fáceis de instalar o Git em um Mac. O mais fácil é usar o instalador gráfico do Git, que você pode baixar na página do SourceForge.

http://sourceforge.net/projects/git-osx-installer/

Figura 1-7. Instalador do Git OS X. A outra maneira principal é instalar o Git via MacPorts (http://www.macports.org). Se você tiver MacPorts instalado, instale o Git via

$ sudo port install git +svn +doc +bash_completion +gitweb

Você não precisa adicionar todos os extras, mas provavelmente desejará incluir +svn caso precise usar o Git com repositórios do Subversion (veja o Capítulo 8).

Homebrew (http://brew.sh/) é outra alternativa para instalar o Git. Se você tiver o Homebrew instalado, instale o Git via

$ brew install git

Instalando no Windows

Instalar o Git no Windows é muito fácil. O projeto msysGit tem um dos procedimentos de instalação mais fáceis. Basta baixar o arquivo exe do instalador da página do GitHub e executá-lo:

http://msysgit.github.io

Depois de instalado, você tem uma versão de linha de comando (incluindo um cliente SSH que será útil mais tarde) e a GUI padrão.

Observação sobre o uso do Windows: você deve usar o Git com o shell msysGit fornecido (estilo Unix), ele permite usar as linhas de comando complexas fornecidas neste livro. Se você precisar, por algum motivo, usar o shell nativo do Windows / console de linha de comando, você deve usar aspas duplas em vez de aspas simples (para parâmetros com espaços neles) e você deve citar os parâmetros que terminam com o acento circunflexo (^ ) se forem os últimos da linha, pois é um símbolo de continuação no Windows.
