Clonando um repositório Git com submódulos #

Ao clonar um repositório que usa submódulos, você precisará inicializá-los e atualizá-los.

$ git clone --recursive https://github.com/username/repo.git

Isso clonará os submódulos referenciados e os colocará nas pastas apropriadas (incluindo submódulos dentro de submódulos). Isso é equivalente a executar git submodule update --init --recursive imediatamente após a conclusão do clone.

Atualizando um submódulo #

Um submódulo faz referência a um commit específico em outro repositório. Para verificar o estado exato referenciado para todos os submódulos, execute

git submodule update --recursive

Às vezes, em vez de usar o estado referenciado, você deseja atualizar seu checkout local para o estado mais recente desse submódulo em um controle remoto. Para verificar todos os submódulos para o estado mais recente no controle remoto com um único comando, você pode usar

git submodule foreach git pull <remote> <branch>

ou use os argumentos padrão git pull

git submodule foreach git pull

Observe que isso apenas atualizará sua cópia de trabalho local. A execução do git status listará o diretório do submódulo como sujo se for alterado por causa deste comando. Para atualizar seu repositório para referenciar o novo estado, você precisa confirmar as alterações:

git add <submodule_directory>
git commit

Pode haver algumas alterações que você tem que podem ter conflito de mesclagem se você usar git pull para que você possa usar git pull --rebase para retroceder suas alterações para o início, na maioria das vezes isso diminui as chances de conflito. Também puxa todos os ramos para local.

git submodule foreach git pull --rebase

Para verificar o estado mais recente de um submódulo específico, você pode usar:

git submodule update --remote <submodule_directory>

Adicionando um submódulo #

Você pode incluir outro repositório Git como uma pasta dentro do seu projeto, rastreado pelo Git:

$ git submodule add https://github.com/jquery/jquery.git

Você deve adicionar e submeter o novo arquivo .gitmodules; isso diz ao Git quais submódulos devem ser clonados quando o git submodule update é executado.

Configurando um submódulo para seguir uma ramificação #

Um submódulo é sempre verificado em um commit específico SHA1 (o “gitlink”, entrada especial no índice do repositório pai)

Mas pode-se solicitar a atualização desse submódulo para o último commit de uma ramificação do repositório remoto do submódulo.

Em vez de entrar em cada submódulo, fazendo um git checkout abranch --track origin/abranch, git pull, você pode simplesmente fazer (do repositório pai) a:

git submodule update --remote --recursive

Como o SHA1 do submódulo mudaria, você ainda precisaria seguir isso com:

git add .
git commit -m "update submodules"

Isso supõe que os submódulos foram:

  • ou adicionado com uma ramificação a seguir:

      git submodule -b abranch -- /url/of/submodule/repo

  • ou configurado (para um submódulo existente) para seguir uma ramificação:

      cd /path/to/parent/repo
  git config -f .gitmodules submodule.asubmodule.branch abranch

Movendo um submódulo #

Corre:

$ git mv antigo/caminho/para/módulo novo/caminho/para/módulo
  1. Edite .gitmodules e altere o caminho do submódulo apropriadamente, e coloque-o no índice com git add .gitmodules.
  2. Se necessário, crie o diretório pai do novo local do submódulo (mkdir -p new/path/to).
  3. Mova todo o conteúdo do diretório antigo para o novo (mv -vi old/path/to/module new/path/to/submodule).
  4. Certifique-se de que o Git rastreie este diretório (git add new/path/to).
  5. Remova o diretório antigo com git rm –cached old/path/to/module.
  6. Mova o diretório .git/modules/old/path/to/module com todo o seu conteúdo para .git/modules/new/path/to/module</ código>.
  7. Edite o arquivo .git/modules/new/path/to/config, certifique-se de que o item da árvore de trabalho aponte para os novos locais, então neste exemplo deve ser worktree = ../../../../../old/path/to/module. Normalmente deve haver mais dois diretórios .. no caminho direto nesse local. . Edite o arquivo new/path/to/module/.git, certifique-se de que o caminho nele aponte para o novo local correto dentro da pasta .git do projeto principal, então neste exemplo gitdir: ../../../.git/modules/new/path/to/module.

A saída do git status se parece com isso depois:

    # On branch master
    # Changes to be committed:
    #   (use "git reset HEAD <file>..." to unstage)
    #
    #       modified:   .gitmodules
    #       renamed:    old/path/to/submodule -> new/path/to/submodule
    #
  1. Por fim, confirme as alterações.

Removendo um submódulo #

Você pode remover um submódulo (por exemplo, the_submodule) chamando:

$ git submodule deinit the_submodule
$ git rm the_submodule

  • git submodule deinit the_submodule exclui a entrada the_submodules’ de .git/config. Isso exclui the_submodule das chamadas git submodule update, git submodule sync e git submodule foreach e exclui seu conteúdo local [(source)](https://git-scm.com/docs/git-submodule#git -submodule-deinit). Além disso, isso não será mostrado como alteração em seu repositório pai. git submodule init e git submodule update restaurarão o submódulo, novamente sem alterações commitíveis em seu repositório pai.

  • git rm the_submodule removerá o submódulo da árvore de trabalho. Os arquivos desaparecerão, assim como a entrada dos submódulos no arquivo .gitmodules (source). Se apenas o git rm the_submodule (sem antes o git submodule deinit the_submodule for executado, no entanto, a entrada dos submódulos em seu arquivo .git/config permanecerá.

Retirado aqui:

  1. Exclua a seção relevante do arquivo .gitmodules.
  2. Faça o teste das alterações .gitmodules git add .gitmodules
  3. Exclua a seção relevante de .git/config.
  4. Execute git rm --cached path_to_submodule (sem barra final).
  5. Execute rm -rf .git/modules/path_to_submodule
  6. Confirme git commit -m "Submódulo removido <nome>"
  7. Exclua os arquivos do submódulo agora não rastreados
  8. rm -rf path_to_submodule
