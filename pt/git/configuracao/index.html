Sintaxe #

  • git config [<file-option>] name [value] # um dos casos de uso mais comuns do git config

Parâmetros #

Parâmetro Detalhes
--sistema Edita o arquivo de configuração de todo o sistema, que é usado para cada usuário (no Linux, este arquivo está localizado em $(prefix)/etc/gitconfig)
--global Edita o arquivo de configuração global, que é usado para cada repositório em que você trabalha (no Linux, este arquivo está localizado em ~/.gitconfig
--local Edita o arquivo de configuração específico do repositório, que está localizado em .git/config em seu repositório; esta é a configuração padrão

Configurando qual editor usar #

Existem várias maneiras de definir qual editor usar para confirmação, rebase, etc.

  • Altere a configuração do core.editor.

$ git config –global core.editor nano

* Defina a variável de ambiente `GIT_EDITOR`.

Para um comando:

$ GIT_EDITOR=nano git commit

Ou para todos os comandos executados em um terminal. **Observação:** Isso só se aplica até você fechar o terminal.

$ export GIT_EDITOR=nano

* Para alterar o editor de *todos* os programas de terminal, não apenas o Git, defina a variável de ambiente `VISUAL` ou `EDITOR`. (Veja [`VISUAL` vs `EDITOR`](https://unix.stackexchange.com/questions/4859/visual-vs-editor-whats-the-difference).)

$ export EDITOR=nano

**Observação:** Como acima, isso se aplica apenas ao terminal atual; seu shell geralmente terá um arquivo de configuração para permitir que você o configure permanentemente. (Em `bash`, por exemplo, adicione a linha acima ao seu `~/.bashrc` ou `~/.bash_profile`.)

Alguns editores de texto (principalmente os de GUI) executarão apenas uma instância por vez e geralmente serão encerrados se você já tiver uma instância deles aberta. Se este for o caso do seu editor de texto, o Git irá imprimir a mensagem `Aborting commit devido a mensagem de commit vazia.` sem permitir que você edite a mensagem de commit primeiro. Se isso acontecer com você, consulte a documentação do seu editor de texto para ver se ele tem um sinalizador `--wait` (ou similar) que o fará pausar até que o documento seja fechado.

## Correção automática de erros de digitação
  git config --global help.autocorrect 17

Isso habilita a correção automática no git e irá perdoá-lo por seus pequenos erros (por exemplo, `git stats` em vez de `git status`). O parâmetro que você fornece para `help.autocorrect` determina quanto tempo o sistema deve esperar, em décimos de segundo, antes de aplicar automaticamente o comando de correção automática. No comando acima, 17 significa que o git deve esperar 1,7 segundos antes de aplicar o comando autocorregido.

No entanto, erros maiores serão considerados como comandos ausentes, então digitar algo como `git testingit` resultaria em `testingit is not a git command.`

## Nome de usuário e endereço de e-mail
Logo após instalar o Git, a primeira coisa que você deve fazer é definir seu nome de usuário e endereço de e-mail. De um shell, digite:

  git config --global user.name "Mr. Bean"
  git config --global user.email mrbean@example.com

- `git config` é o comando para obter ou definir opções
- `--global` significa que o arquivo de configuração específico para sua conta de usuário será editado
- `user.name` e `user.email` são as chaves para as variáveis ​​de configuração; `user` é a seção do arquivo de configuração. `name` e `email` são os nomes das variáveis.
- `"Mr. Bean"` e `mrbean@example.com` são os valores que você está armazenando nas duas variáveis. Observe as aspas em torno de `"Mr. Bean"`, que são necessárias porque o valor que você está armazenando contém um espaço.

## Várias configurações do git
Você tem até 5 fontes para configuração do git:

- 6 arquivos:
- **`%ALLUSERSPROFILE%\Git\Config`** (somente Windows)
- (sistema) **`<git>/etc/gitconfig`**, com `<git>` sendo o caminho de instalação do git.
(no Windows, é **`<git>\mingw64\etc\gitconfig`**)
- (sistema) **`$XDG_CONFIG_HOME/git/config`** (somente Linux/Mac)
- (global) `~/.gitconfig` (Windows: `%USERPROFILE%\.gitconfig`)
- (local) `.git/config` (dentro de um repositório git `$GIT_DIR`)
- um **arquivo dedicado** (com `git config -f`), usado por exemplo para modificar a configuração dos submódulos: `git config -f .gitmodules ...`
- **a linha de comando com `git -c`**: `git -c core.autocrlf=false fetch` substituiria *qualquer* outro `core.autocrlf` para `false`, *apenas* para esse `fetch` comando.

A ordem é importante: qualquer configuração definida em uma fonte pode ser substituída por uma fonte listada abaixo dela.

`git config --system/global/local` é o comando para listar 3 dessas fontes, mas apenas git config -l listaria *todos* *resolvidos* configs.
"resolvido" significa que ele lista apenas o valor de configuração substituído final.

Desde o git 2.8, se você quiser ver qual configuração vem de qual arquivo, digite:

  git config --list --show-origin

## Listar e editar a configuração atual
A configuração do Git permite que você personalize como o git funciona. É comumente usado para definir seu nome e e-mail ou editor favorito ou como as mesclagens devem ser feitas.

Para ver a configuração atual.

  $ git config --list
  ...
  core.editor=vim
  credential.helper=osxkeychain
  ...

Para editar a configuração:

  $ git config <key> <value>    
  $ git config core.ignorecase true

Se você pretende que a mudança seja verdadeira para todos os seus repositórios, use `--global`

  $ git config --global user.name "Your Name"
  $ git config --global user.email "Your Email"
  $ git config --global core.editor vi

Você pode listar novamente para ver suas alterações.

## Vários nomes de usuário e endereço de e-mail
Desde o Git 2.13, vários nomes de usuário e endereços de e-mail podem ser configurados usando um filtro de pasta.

## Exemplo para Windows:

### .gitconfig

Editar: `git config --global -e`

Adicionar:

[includeIf “gitdir:D:/work”] path = .gitconfig-work.config

[includeIf “gitdir:D:/opensource/”] path = .gitconfig-opensource.config


#### Notas
- A ordem é dependente, o último que acertar "ganha".
- o `/` no final é necessário - por exemplo `"gitdir:D:/work"` não funcionará.
- o prefixo `gitdir:` é obrigatório.


### .gitconfig-work.config
Arquivo no mesmo diretório que *.gitconfig*

[user] name = Money email = work@somewhere.com


### .gitconfig-opensource.config
Arquivo no mesmo diretório que *.gitconfig*

[user] name = Nice email = cool@opensource.stuff



## Exemplo para Linux

[includeIf “gitdir:~/work/”] path = .gitconfig-work [includeIf “gitdir:~/opensource/”] path = .gitconfig-opensource

## Configurando terminações de linha
Descrição
-----------

Ao trabalhar com uma equipe que usa diferentes sistemas operacionais (SO) em todo o projeto, às vezes você pode ter problemas ao lidar com finais de linha.

Microsoft Windows
-------

Ao trabalhar no sistema operacional (SO) Microsoft Windows, as terminações de linha são normalmente do formato - retorno de carro + avanço de linha (CR+LF). Abrir um arquivo que foi editado usando uma máquina Unix, como Linux ou OSX, pode causar problemas, fazendo parecer que o texto não tem final de linha. Isso se deve ao fato de que os sistemas Unix aplicam apenas terminações de linha diferentes de feeds de linha de formulário (LF).

Para corrigir isso, você pode executar as seguintes instruções

    git config --global core.autocrlf=true

No **checkout**, esta instrução garantirá que as terminações de linha sejam configuradas de acordo com o sistema operacional Microsoft Windows (LF -> CR+LF)

Baseado em Unix (Linux/OSX)
-----------------------

Da mesma forma, pode haver problemas quando o usuário no sistema operacional baseado em Unix tenta ler arquivos que foram editados no sistema operacional Microsoft Windows. Para evitar problemas inesperados, execute

    git config --global core.autocrlf=input

Em **commit**, isso mudará as terminações de linha de CR+LF -> +LF

## configuração para apenas um comando
você pode usar `-c <name>=<value>` para adicionar uma configuração apenas para um comando.

Para confirmar como outro usuário sem precisar alterar suas configurações em .gitconfig :

    git -c user.email = mail@example commit -m "some message"

Nota: para esse exemplo você não precisa precisar tanto `user.name` quanto `user.email`, o git irá completar as informações que faltam nos commits anteriores.

## Configurar um proxy
Se você está atrás de um proxy, você deve informar ao git sobre isso:

    git config --global http.proxy http://my.proxy.com:portnumber

Se você não está mais atrás de um proxy:

    git config --global --unset http.proxy
