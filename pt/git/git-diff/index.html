Sintaxe #

  • git diff [opções] [<commit>] [--] [<path>…​]
  • git diff [opções] --cached [<commit>] [--] [<path>…​]
  • git diff [opções] <commit> <commit> [--] [<path>…​]
  • git diff [opções] <blob> <blob>
  • git diff [opções] [--no-index] [--] <caminho> <caminho>

Parâmetros #

Parâmetro Detalhes
-p, -u, –patch Gerar patch
-s, –sem patch Suprima a saída do diferencial. Útil para comandos como git show que mostram o patch por padrão, ou para cancelar o efeito de --patch
–cru Gere o diff em formato bruto
–diff-algoritmo= Escolha um algoritmo de comparação. As variantes são as seguintes: myers, minimal, paciência, histograma
–resumo Saída de um resumo condensado de informações de cabeçalho estendidas, como criações, renomeações e alterações de modo
–somente nome Mostrar apenas nomes de arquivos alterados
–nome-status Mostrar nomes e status de arquivos alterados Os status mais comuns são M (Modificado), A (Adicionado) e D (Excluído)
–cheque Avise se as alterações introduzirem marcadores de conflito ou erros de espaço em branco. O que são considerados erros de espaço em branco é controlado pela configuração do core.whitespace. Por padrão, espaços em branco à direita (incluindo linhas que consistem apenas em espaços em branco) e um caractere de espaço que é imediatamente seguido por um caractere de tabulação dentro do recuo inicial da linha são considerados erros de espaço em branco. Sai com status diferente de zero se forem encontrados problemas. Não compatível com –exit-code
–full-index Em vez do primeiro punhado de caracteres, mostre os nomes completos dos objetos blob pré e pós-imagem na linha “index” ao gerar a saída do formato de patch
–binário Além de --full-index, produza um diff binário que pode ser aplicado com git apply
-a, –texto Trate todos os arquivos como texto.
–cor Defina o modo de cor; ou seja, use --color=always se você quiser canalizar um diff para less e manter a coloração do git

Mostrar diferenças no branch de trabalho #

git diff

Isso mostrará as alterações unstaged no branch atual do commit anterior. Ele mostrará apenas as alterações relativas ao índice, o que significa que mostra o que você poderia adicionar ao próximo commit, mas não o fez. Para adicionar (estágio) essas alterações, você pode usar git add.

Se um arquivo é testado, mas foi modificado após o teste, git diff mostrará as diferenças entre o arquivo atual e a versão testada.

Mostra as mudanças entre dois commits #

git diff 1234abc..6789def    # old   new

Ex.: Mostrar as alterações feitas nos últimos 3 commits:

git diff @~3..@    # HEAD -3   HEAD

Nota: os dois pontos (..) são opcionais, mas adicionam clareza.

Isso mostrará a diferença textual entre os commits, independentemente de onde eles estejam na árvore.

Mostrar diferenças para arquivos encenados #

git diff --staged

Isso mostrará as alterações entre o commit anterior e os arquivos encenados no momento.

OBSERVAÇÃO: você também pode usar os seguintes comandos para fazer a mesma coisa:

git diff --cached

Que é apenas um sinônimo para --staged ou

git status -v

O que acionará as configurações detalhadas do comando status.

Comparando ramificações #

Mostre as alterações entre a dica de new e a dica de original:

git diff original new     # equivalent to original..new

Mostrar todas as alterações em new desde que se ramificou de original:

git diff original...new     # equivalent to $(git merge-base original new)..new

Usando apenas um parâmetro, como

git diff original

é equivalente a

git diff original..HEAD

Mostrar alterações preparadas e não preparadas #

Para mostrar todas as alterações preparadas e não programadas, use:

git diff HEAD

OBSERVAÇÃO: você também pode usar o seguinte comando:

git status -vv

A diferença é que a saída do último realmente informará quais alterações são preparadas para confirmação e quais não são.

Mostra as diferenças para um arquivo ou diretório específico #

git diff myfile.txt

Mostra as alterações entre o commit anterior do arquivo especificado (myfile.txt) e a versão modificada localmente que ainda não foi testada.

Isso também funciona para diretórios:

git diff documentation

O acima mostra as mudanças entre o commit anterior de todos os arquivos no diretório especificado (documentation/) e as versões modificadas localmente desses arquivos, que ainda não foram testadas.

Para mostrar a diferença entre alguma versão de um arquivo em um determinado commit e a versão local do HEAD, você pode especificar o commit com o qual deseja comparar:

git diff 27fa75e myfile.txt

Ou se você quiser ver a versão entre dois commits separados:

git diff 27fa75e ada9b57 myfile.txt

Para mostrar a diferença entre a versão especificada pelo hash ada9b57 e o último commit no branch my_branchname apenas para o diretório relativo chamado my_changed_directory/ você pode fazer isso:

git diff ada9b57 my_branchname my_changed_directory/

Visualizando uma comparação de palavras para linhas longas #

git diff [HEAD|--staged...] --word-diff

Em vez de exibir as linhas alteradas, isso exibirá as diferenças dentro das linhas. Por exemplo, em vez de:

-Hello world
+Hello world!

Onde toda a linha é marcada como alterada, word-diff altera a saída para:

Hello [-world-]{+world!+}

Usando o meld para ver todas as modificações no diretório de trabalho #

git difftool -t meld --dir-diff

mostrará as alterações do diretório de trabalho. Alternativamente,

git difftool -t meld --dir-diff  [COMMIT_A] [COMMIT_B]

irá mostrar as diferenças entre 2 commits específicos.

Mostra as diferenças entre a versão atual e a última versão #

git diff HEAD^ HEAD

Isso mostrará as alterações entre o commit anterior e o commit atual.

Produza um diferencial compatível com patch #

Às vezes você só precisa de um diff para aplicar usando patch. O git --diff normal não funciona. Tente isso em vez disso:

git diff --no-prefix > some_file.patch

Então em outro lugar você pode reverter isso:

patch -p0 < some_file.patch

diferença entre dois commits ou branchs #

Para visualizar a diferença entre duas ramificações

git diff <branch1>..<branch2>

Para visualizar a diferença entre duas ramificações

git diff <commitId1>..<commitId2>

Para ver a diferença com a ramificação atual

git diff <branch/commitId>

Para visualizar o resumo das alterações

git diff --stat <branch/commitId>

Para visualizar arquivos que foram alterados após um determinado commit

git diff --name-only <commitId>

Para visualizar arquivos diferentes de uma ramificação

git diff --name-only <branchName>

Para visualizar arquivos que foram alterados em uma pasta após um determinado commit

git diff --name-only <commitId> <folder_path>

Diff texto codificado em UTF-16 e arquivos plist binários #

Você pode diferenciar arquivos codificados em UTF-16 (arquivo de strings de localização para iOS e macOS são exemplos) especificando como o git deve diferenciar esses arquivos.

Adicione o seguinte ao seu arquivo ~/.gitconfig.

[diff "utf16"]
textconv = "iconv -f utf-16 -t utf-8"

iconv é um programa para converter diferentes codificações.

Em seguida, edite ou crie um arquivo .gitattributes na raiz do repositório onde você deseja usá-lo. Ou apenas edite ~/.gitattributes.

*.strings diff=utf16

Isto irá converter todos os arquivos que terminam em .strings antes de git diffs.

Você pode fazer coisas semelhantes para outros arquivos, que podem ser convertidos em texto.

Para arquivos plist binários você edita .gitconfig

[diff "plist"]
textconv = plutil -convert xml1 -o -

e .gitattributes

*.plist diff=plist

Visualizando uma mesclagem de três vias incluindo o ancestral comum #

git config --global merge.conflictstyle diff3

Define o estilo diff3 como padrão: em vez do formato usual em seções conflitantes, mostrando os dois arquivos:

<<<<<<< HEAD
left
=======
right
>>>>>>> master

incluirá uma seção adicional contendo o texto original (vindo do ancestral comum):

<<<<<<< HEAD
first
second
||||||| 
first
=======
last
>>>>>>> master

Este formato facilita a compreensão de conflitos de mesclagem, ou seja. neste caso localmente second foi adicionado, enquanto remoto mudou first para last, resolvendo para:

last
second

A mesma resolução teria sido muito mais difícil usando o padrão:

<<<<<<< HEAD
first
second
=======
last
>>>>>>> master

git diff branch1..branch2
