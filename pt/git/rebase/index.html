Sintaxe #

  • git rebase [-i | --interactive] [opções] [--exec <cmd>] [--onto <newbase>] [<upstream>] [<branch>]
  • git rebase [-i | --interactive] [opções] [--exec <cmd>] [--onto <newbase>] --root [<branch>]
  • git rebase --continue | --pular | --abortar | --edit-all

Parâmetros #

Parâmetro Detalhes
–continuar Reinicie o processo de rebase depois de resolver um conflito de mesclagem.
–abortar Interrompa a operação de rebase e redefina HEAD para a ramificação original. Se a ramificação foi fornecida quando a operação de rebase foi iniciada, HEAD será redefinido para ramificação. Caso contrário, HEAD será redefinido para onde estava quando a operação de rebase foi iniciada.
–manter-vazio Mantenha os commits que não alteram nada de seus pais no resultado.
–pular Reinicie o processo de rebase pulando o patch atual.
-m, –merge Use estratégias de mesclagem para rebase. Quando a estratégia de mesclagem recursiva (padrão) é usada, isso permite que o rebase esteja ciente das renomeações no lado upstream. Observe que uma mesclagem de rebase funciona reproduzindo cada confirmação da ramificação de trabalho na parte superior da ramificação upstream. Por causa disso, quando ocorre um conflito de mesclagem, o lado relatado como nosso é a série rebaseada até agora, começando com o upstream, e o deles é o branch de trabalho. Em outras palavras, os lados são trocados.
–stat Mostre um diffstat do que mudou no upstream desde o último rebase. O diffstat também é controlado pela opção de configuração rebase.stat.
-x, –exec comando Execute o rebase interativo, parando entre cada commit e executando command

Por favor, tenha em mente que o rebase efetivamente reescreve o histórico do repositório.

O rebase de commits que existem no repositório remoto pode reescrever os nós do repositório usados ​​por outros desenvolvedores como nó base para seus desenvolvimentos. A menos que você realmente saiba o que está fazendo, é uma prática recomendada fazer o rebase antes de enviar suas alterações.

Rebase da Filial Local #

Rebaseing reaplica uma série de commits em cima de outro commit.

Para rebase uma ramificação, faça o checkout da ramificação e, em seguida, faça o rebase em cima de outra ramificação.

git checkout topic
git rebase master  # rebase current branch onto master branch

Isso causaria:

      A---B---C topic
     /
D---E---F---G master

Para se transformar em:

              A'--B'--C' topic
             /
D---E---F---G master

Essas operações podem ser combinadas em um único comando que verifica a ramificação e a rebase imediatamente:

git rebase master topic   # rebase topic branch onto master branch

Importante: Após o rebase, os commits aplicados terão um hash diferente. Você não deve fazer o rebase de commits que você já enviou para um host remoto. Uma consequência pode ser a incapacidade de git push seu branch local rebaseado para um host remoto, deixando sua única opção para git push --force.

Rebase: nosso e deles, local e remoto #

Um rebase muda o significado de “nosso” e “deles”:

git checkout topic
git rebase   master    # rebase topic branch on top of master branch

O que quer que HEAD esteja apontando é “nosso”

A primeira coisa que um rebase faz é redefinir o HEAD para master; antes de selecionar commits do branch antigo topic para um novo (todo commit no branch anterior topic será reescrito e será identificado por um hash diferente).

Com relação às terminologias usadas pelas ferramentas de mesclagem (não confundir com referência local ou referência remota)

=> local is master ("ours"),
=> remote is topic ("theirs")

Isso significa que uma ferramenta de merge/diff apresentará o branch upstream como local (master: o branch sobre o qual você está rebaseando) e o branch de trabalho como remote (topic: o branch sendo rebaseado)

+-----------------------------------------+
| LOCAL:master |    BASE   | REMOTE:topic |
+-----------------------------------------+
|             MERGED                      |
+-----------------------------------------+

Inversão ilustrada #

Em uma mesclagem: #

c--c--x--x--x(*) <- current branch topic ('*'=HEAD)
    \
     \
      \--y--y--y <- other branch to merge

Nós não alteramos o tópico do branch atual, então o que temos ainda é o que estávamos trabalhando (e nós mesclamos de outro branch)

c--c--x--x--x---------o(*)  MERGE, still on branch topic
    \       ^        /
     \     ours     /
      \            /
       --y--y--y--/  
               ^
              theirs

Em um rebase: #

Mas em um rebase trocamos de lado porque a primeira coisa que um rebase faz é verificar o branch upstream para reproduzir os commits atuais em cima dele!

c--c--x--x--x(*) <- current branch topic ('*'=HEAD)
    \
     \
      \--y--y--y <- upstream branch

Um git rebase upstream primeiro definirá HEAD para a ramificação upstream, daí a troca de ’nosso’ e ‘deles’ em comparação com a ramificação de trabalho “atual” anterior.

c--c--x--x--x <- former "current" branch, new "theirs"
    \
     \
      \--y--y--y(*) <- set HEAD to this commit, to replay x's on it
               ^       this will be the new "ours"
               |
            upstream

O rebase irá então reproduzir ‘seus’ commits no novo branch ’nosso’ tópico:

c--c..x..x..x <- old "theirs" commits, now "ghosts", available through "reflogs"
    \
     \
      \--y--y--y--x'--x'--x'(*) <- topic  once all x's are replayed,
               ^                      point branch topic to this commit
               |
        upstream branch

Rebase interativo #

Este exemplo tem como objetivo descrever como se pode utilizar o git rebase no modo interativo. Espera-se que se tenha uma compreensão básica do que é o git rebase e ​​o que ele faz.

O rebase interativo é iniciado usando o seguinte comando:

git rebase -i

A opção -i refere-se ao modo interativo. Usando o rebase interativo, o usuário pode alterar as mensagens de commit, bem como reordenar, dividir e/ou squash (combine to one) commits.

Digamos que você queira reorganizar seus últimos três commits. Para fazer isso, você pode executar:

git rebase -i HEAD~3

Após executar a instrução acima, um arquivo será aberto em seu editor de texto onde você poderá selecionar como seus commits serão rebaseados. Para o propósito deste exemplo, apenas altere a ordem de seus commits, salve o arquivo e feche o editor. Isso iniciará um rebase com o pedido que você aplicou. Se você marcar git log, verá seus commits na nova ordem que você especificou.

Reformulando mensagens de commit #

Agora, você decidiu que uma das mensagens de confirmação é vaga e deseja que seja mais descritiva. Vamos examinar os últimos três commits usando o mesmo comando.

git rebase -i HEAD~3

Ao invés de reorganizar a ordem em que os commits serão rebaseados, desta vez vamos mudar pick, o padrão, para reword em um commit onde você gostaria de mudar a mensagem.

Quando você fechar o editor, o rebase será iniciado e parará na mensagem de confirmação específica que você deseja reformular. Isso permitirá que você altere a mensagem de confirmação para a que desejar. Depois de alterar a mensagem, basta fechar o editor para continuar.

Mudando o conteúdo de um commit #

Além de alterar a mensagem do commit, você também pode adaptar as alterações feitas pelo commit. Para fazer isso, basta alterar pick para edit para um commit. O Git parará quando chegar a esse commit e fornecerá as alterações originais do commit na área de teste. Agora você pode adaptar essas alterações removendo-as de preparação ou adicionando novas alterações.

Assim que a área de teste contiver todas as alterações que você deseja nesse commit, confirme as alterações. A mensagem do commit antigo será mostrada e pode ser adaptada para refletir o novo commit.

Dividindo um único commit em vários #

Digamos que você fez um commit, mas decidiu mais tarde que esse commit poderia ser dividido em dois ou mais commits. Usando o mesmo comando de antes, substitua pick por edit e pressione enter.

Agora, o git irá parar no commit que você marcou para edição e colocar todo o seu conteúdo na área de teste. A partir desse ponto, você pode executar git reset HEAD^ para colocar o commit em seu diretório de trabalho. Então, você pode adicionar e enviar seus arquivos em uma sequência diferente - em última análise, dividindo um único commit em n commits.

Comprimir vários commits em um #

Digamos que você tenha feito algum trabalho e tenha vários commits que você acha que poderia ser um único commit. Para isso você pode executar git rebase -i HEAD~3, substituindo 3 por uma quantidade apropriada de commits.

Desta vez, substitua pick por squash. Durante o rebase, o commit que você instruiu a ser squashed será squash em cima do commit anterior; transformando-os em um único commit.

Rebase para o commit inicial #

Desde o Git 1.7.12 é possível fazer o rebase até o commit root. O commit root é o primeiro commit feito em um repositório e normalmente não pode ser editado. Use o seguinte comando:

git rebase -i --root

Rebase antes de uma revisão de código #

Resumo #

Esse objetivo é reorganizar todos os seus commits dispersos em commits mais significativos para facilitar as revisões de código. Se houver muitas camadas de alterações em muitos arquivos ao mesmo tempo, será mais difícil fazer uma revisão de código. Se você puder reorganizar seus commits criados cronologicamente em commits tópicos, o processo de revisão de código será mais fácil (e possivelmente menos bugs passarão pelo processo de revisão de código).

Este exemplo excessivamente simplificado não é a única estratégia para usar o git para fazer melhores revisões de código. É a maneira que eu faço, e é algo para inspirar outros a considerar como fazer revisões de código e histórico do git mais fácil/melhor.

Isso também demonstra pedagogicamente o poder do rebase em geral.

Este exemplo pressupõe que você conheça o rebase interativo.

Supondo: #

  • você está trabalhando em uma ramificação de recurso fora do mestre
  • seu recurso tem três camadas principais: front-end, back-end, DB
  • você fez muitos commits enquanto trabalhava em uma feature branch. Cada commit toca várias camadas de uma vez
  • você quer (no final) apenas três commits em seu branch
  • um contendo todas as alterações de front-end
  • um contendo todas as alterações de back-end
  • um contendo todas as alterações de banco de dados

Estratégia: #

  • vamos mudar nossos commits cronológicos para commits “tópicos”.
  • primeiro, divida todos os commits em vários commits menores – cada um contendo apenas um tópico por vez (no nosso exemplo, os tópicos são front-end, back-end, alterações de banco de dados)
  • Em seguida, reordene nossos commits tópicos juntos e os ’esmague’ em commits tópicos únicos

Exemplo: #

$ git log --oneline master..
975430b db adding works: db.sql logic.rb
3702650 trying to allow adding todo items: page.html logic.rb
43b075a first draft: page.html and db.sql
$ git rebase -i master

Isso será mostrado no editor de texto:

pick 43b075a first draft: page.html and db.sql
pick 3702650 trying to allow adding todo items: page.html logic.rb
pick 975430b db adding works: db.sql logic.rb

Altere para isso:

e 43b075a first draft: page.html and db.sql
e 3702650 trying to allow adding todo items: page.html logic.rb
e 975430b db adding works: db.sql logic.rb

Então o git aplicará um commit por vez. Após cada confirmação, ele mostrará um prompt e você poderá fazer o seguinte:

Stopped at 43b075a92a952faf999e76c4e4d7fa0f44576579... first draft: page.html and db.sql
You can amend the commit now, with

        git commit --amend

Once you are satisfied with your changes, run

        git rebase --continue

$ git status
rebase in progress; onto 4975ae9
You are currently editing a commit while rebasing branch 'feature' on '4975ae9'.
  (use "git commit --amend" to amend the current commit)
  (use "git rebase --continue" once you are satisfied with your changes)

nothing to commit, working directory clean
$ git reset HEAD^ #This 'uncommits' all the changes in this commit.
$ git status -s
 M db.sql
 M page.html
$ git add db.sql  #now we will create the smaller topical commits
$ git commit -m "first draft: db.sql"
$ git add page.html
$ git commit -m "first draft: page.html"
$ git rebase --continue

Em seguida, você repetirá essas etapas para cada confirmação. No final, você tem isso:

$ git log --oneline
0309336 db adding works: logic.rb
06f81c9 db adding works: db.sql
3264de2 adding todo items: page.html
675a02b adding todo items: logic.rb
272c674 first draft: page.html
08c275d first draft: db.sql

Agora executamos o rebase mais uma vez para reordenar e esmagar:

$ git rebase -i master

Isso será mostrado no editor de texto:

pick 08c275d first draft: db.sql
pick 272c674 first draft: page.html
pick 675a02b adding todo items: logic.rb
pick 3264de2 adding todo items: page.html
pick 06f81c9 db adding works: db.sql
pick 0309336 db adding works: logic.rb

Altere para isso:

pick 08c275d first draft: db.sql
s 06f81c9 db adding works: db.sql
pick 675a02b adding todo items: logic.rb
s 0309336 db adding works: logic.rb
pick 272c674 first draft: page.html
s  3264de2 adding todo items: page.html

AVISO: certifique-se de dizer ao git rebase para aplicar/esmagar os commits tópicos menores na ordem em que foram confirmados cronologicamente. Caso contrário, você pode ter conflitos de mesclagem falsos e desnecessários para lidar.

Quando esse rebase interativo estiver dito e feito, você obtém isso:

$ git log --oneline master..
74bdd5f adding todos: GUI layer
e8d8f7e adding todos: business logic layer
121c578 adding todos: DB layer

Recapitular #

Agora você rebaseou seus commits cronológicos em commits tópicos. Na vida real, você pode não precisar fazer isso todas as vezes, mas quando quiser ou precisar fazer isso, agora você pode. Além disso, espero que você tenha aprendido mais sobre o git rebase.

Testando todos os commits durante o rebase #

Antes de fazer um pull request, é útil certificar-se de que a compilação foi bem-sucedida e que os testes estão passando para cada commit no branch. Podemos fazer isso automaticamente usando o parâmetro -x.

Por exemplo:

git rebase -i -x make

irá executar o rebase interativo e parar após cada commit para executar make. Caso make falhe, o git irá parar para lhe dar a oportunidade de corrigir os problemas e alterar o commit antes de prosseguir com a escolha do próximo.

Configurando o armazenamento automático #

Autostash é uma opção de configuração muito útil ao usar o rebase para alterações locais. Muitas vezes, você pode precisar trazer commits do branch upstream, mas ainda não está pronto para commitar.

No entanto, o Git não permite que um rebase seja iniciado se o diretório de trabalho não estiver limpo. Autostash para o resgate:

git config --global rebase.autostash    # one time configuration
git rebase @{u}                         # example rebase on upstream branch

O autostash será aplicado sempre que o rebase for concluído. Não importa se o rebase termina com sucesso ou se é abortado. De qualquer forma, o autostash será aplicado. Se o rebase foi bem-sucedido e o commit base foi alterado, pode haver um conflito entre o autostash e os novos commits. Nesse caso, você terá que resolver os conflitos antes de se comprometer. Isso não é diferente do que se você tivesse armazenado manualmente e aplicado, portanto, não há desvantagem em fazê-lo automaticamente.

Abortando um rebase interativo #

Você iniciou um rebase interativo. No editor onde você escolhe seus commits, você decide que algo está errado (por exemplo, um commit está faltando, ou você escolheu o destino de rebase errado) e você quer abortar o rebase.

Para fazer isso, simplesmente exclua todos os commits e ações (ou seja, todas as linhas que não comecem com o sinal #) e o rebase será abortado!

O texto de ajuda no editor realmente fornece esta dica:

# Rebase 36d15de..612f2f7 onto 36d15de (3 command(s))
#
# Commands:
# p, pick = use commit
# r, reword = use commit, but edit the commit message
# e, edit = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup = like "squash", but discard this commit's log message
# x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#
# If you remove a line here THAT COMMIT WILL BE LOST.
#
# However, if you remove everything, the rebase will be aborted.
#          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
# Note that empty commits are commented out

Configure o git-pull para executar automaticamente um rebase em vez de uma mesclagem #

Se sua equipe estiver seguindo um fluxo de trabalho baseado em rebase, pode ser vantajoso configurar o git para que cada branch recém-criado execute uma operação de rebase, em vez de uma operação de mesclagem, durante um git pull.

Para configurar cada novo branch para rebase automaticamente, adicione o seguinte ao seu .gitconfig ou .git/config:

[branch]
autosetuprebase = always

Linha de comando: git config [--global] branch.autosetuprebase sempre

Alternativamente, você pode configurar o comando git pull para sempre se comportar como se a opção --rebase fosse passada:

[pull]
rebase = true

Linha de comando: git config [--global] pull.rebase true

Empurrando após um rebase #

Às vezes você precisa reescrever o histórico com um rebase, mas o git push reclama porque você reescreveu o histórico.

Isso pode ser resolvido com um git push --force, mas considere git push --force-with-lease, indicando que você deseja que o push falhe se a ramificação local de rastreamento remoto for diferente da ramificação no controle remoto , por exemplo, outra pessoa foi enviada para o controle remoto após a última busca. Isso evita sobrescrever inadvertidamente o push recente de outra pessoa.

Nota: git push --force - e até mesmo --force-with-lease para esse assunto - pode ser um comando perigoso porque reescreve o histórico da ramificação. Se outra pessoa tiver puxado a ramificação antes do push forçado, seu git pull ou git fetch terá erros porque o histórico local e o histórico remoto são divergidos. Isso pode fazer com que a pessoa tenha erros inesperados. Com bastante olhar para os reflogs, o trabalho do outro usuário pode ser recuperado, mas pode levar a muito tempo perdido. Se você precisar fazer um push forçado para uma ramificação com outros contribuidores, tente coordenar com eles para que eles não tenham que lidar com erros.
