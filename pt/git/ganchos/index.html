Pré-empurrar #

Disponível no Git 1.8.2 e superior.

Ganchos de pré-empurrão podem ser usados ​​para evitar que um empurrão aconteça. As razões pelas quais isso é útil incluem: bloquear pushs manuais acidentais para ramificações específicas ou bloquear pushs se uma verificação estabelecida falhar (testes de unidade, sintaxe).

Um hook pre-push é criado simplesmente criando um arquivo chamado pre-push em .git/hooks/, e (gotcha alert), certificando-se de que o arquivo seja executável: chmod +x ./ git/hooks/pre-push.

Aqui está um exemplo de Hannah Wolfe que bloqueia um push to master:

#!/bin/bash

protected_branch='master'  
current_branch=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')

if [ $protected_branch = $current_branch ]  
then  
    read -p "You're about to push master, is that what you intended? [y|n] " -n 1 -r < /dev/tty
    echo
    if echo $REPLY | grep -E '^[Yy]$' > /dev/null
    then
        exit 0 # push will execute
    fi
    exit 1 # push will not execute
else  
    exit 0 # push will execute
fi

Aqui está um exemplo de Volkan Unsal que garante Os testes RSpec passam antes de permitir o push:

#!/usr/bin/env ruby
require 'pty'
html_path = "rspec_results.html"
begin
  PTY.spawn( "rspec spec --format h > rspec_results.html" ) do |stdin, stdout, pid|
  begin
    stdin.each { |line| print line }
  rescue Errno::EIO
  end
end
rescue PTY::ChildExited
  puts "Child process exit!"
end

# find out if there were any errors  
html = open(html_path).read
examples = html.match(/(\d+) examples/)[0].to_i rescue 0
errors = html.match(/(\d+) errors/)[0].to_i rescue 0
if errors == 0 then
  errors = html.match(/(\d+) failure/)[0].to_i rescue 0
end
pending = html.match(/(\d+) pending/)[0].to_i rescue 0

if errors.zero?
  puts "0 failed! #{examples} run, #{pending} pending"
  # HTML Output when tests ran successfully:
  # puts "View spec results at #{File.expand_path(html_path)}"
  sleep 1
  exit 0
else
  puts "\aCOMMIT FAILED!!"
  puts "View your rspec results at #{File.expand_path(html_path)}"
  puts
  puts "#{errors} failed! #{examples} run, #{pending} pending"
  # Open HTML Ooutput when tests failed
  # `open #{html_path}`
  exit 1
end

Como você pode ver, existem muitas possibilidades, mas a peça principal é sair 0 se coisas boas acontecerem, e sair 1 se coisas ruins acontecerem. Sempre que você sair 1, o push será impedido e seu código estará no estado em que estava antes de executar git push....

Ao usar ganchos do lado do cliente, lembre-se de que os usuários podem pular todos os ganchos do lado do cliente usando a opção “–no-verify” em um push. Se você está confiando no gancho para impor o processo, você pode se queimar.

Documentação: https://git-scm.com/docs/githooks#_pre_push Amostra oficial: https://github.com/git/git/blob/87c86dd14abe8db7d00b0df5661ef8cf147a72a3/templates/hooks–pre-push.sample

Verifique a compilação do Maven (ou outro sistema de compilação) antes de confirmar #

.git/hooks/pre-commit

#!/bin/sh
if [ -s pom.xml ]; then
    echo "Running mvn verify"
    mvn clean verify
    if [ $? -ne 0 ]; then
        echo "Maven build failed"
        exit 1
    fi
fi

Encaminhe automaticamente certos pushs para outros repositórios #

Ganchos post-receive podem ser usados ​​para encaminhar automaticamente pushs recebidos para outro repositório.

$ cat .git/hooks/post-receive

#!/bin/bash

IFS=' '
while read local_ref local_sha remote_ref remote_sha
do

  echo "$remote_ref" | egrep '^refs\/heads\/[A-Z]+-[0-9]+$' >/dev/null && {
    ref=`echo $remote_ref | sed -e 's/^refs\/heads\///'`
    echo Forwarding feature branch to other repository: $ref
    git push -q --force other_repos $ref
  }

done

Neste exemplo, a expressão regular egrep procura um formato de ramificação específico (aqui: JIRA-12345 usado para nomear problemas do Jira). Você pode deixar essa parte desativada se quiser encaminhar todas as ramificações, é claro.

Commit-msg #

Este gancho é semelhante ao gancho prepare-commit-msg, mas é chamado depois que o usuário insere uma mensagem de confirmação e não antes. Isso geralmente é usado para avisar os desenvolvedores se sua mensagem de confirmação estiver em um formato incorreto.

O único argumento passado para este gancho é o nome do arquivo que contém a mensagem. Se você não gostar da mensagem que o usuário digitou, você pode alterar este arquivo no local (o mesmo que prepare-commit-msg) ou você pode abortar o commit completamente saindo com um status diferente de zero.

O exemplo a seguir é usado para verificar se a palavra ticket seguida de um número está presente na mensagem de confirmação

word="ticket [0-9]"
isPresent=$(grep -Eoh "$word" $1)

if [[ -z $isPresent ]]
  then echo "Commit message KO, $word is missing"; exit 1;
  else echo "Commit message OK"; exit 0;
fi

Ganchos locais #

Os ganchos locais afetam apenas os repositórios locais nos quais residem. Cada desenvolvedor pode alterar seus próprios ganchos locais, então eles não podem ser usados ​​de forma confiável como uma forma de impor uma política de confirmação. Eles são projetados para tornar mais fácil para os desenvolvedores aderirem a certas diretrizes e evitar possíveis problemas no futuro.

Existem seis tipos de hooks locais: pre-commit, prepare-commit-msg, commit-msg, post-commit, post-checkout e pre-rebase.

Os primeiros quatro ganchos estão relacionados a commits e permitem que você tenha algum controle sobre cada parte do ciclo de vida de um commit. Os dois últimos permitem que você execute algumas ações extras ou verificações de segurança para os comandos git checkout e git rebase.

Todos os ganchos “pré-” permitem alterar a ação que está prestes a ocorrer, enquanto os ganchos “pós-” são usados ​​principalmente para notificações.

Pós-checkout #

Este gancho funciona de forma semelhante ao gancho post-commit, mas é chamado sempre que você verifica com sucesso uma referência com git checkout. Isso pode ser uma ferramenta útil para limpar seu diretório de trabalho de arquivos gerados automaticamente que, de outra forma, causariam confusão.

Este gancho aceita três parâmetros:

  1. a referência do HEAD anterior,
  2. a referência do novo HEAD, e
  3. um sinalizador indicando se foi um checkout de branch ou um checkout de arquivo (1 ou 0, respectivamente).

Seu status de saída não afeta o comando git checkout.

Pós-commit #

Este gancho é chamado imediatamente após o gancho commit-msg. Ele não pode alterar o resultado da operação git commit, portanto, é usado principalmente para fins de notificação.

O script não aceita parâmetros e seu status de saída não afeta a confirmação de forma alguma.

Pós-recebimento #

Este gancho é chamado após uma operação push bem-sucedida. Geralmente é usado para fins de notificação.

O script não recebe parâmetros, mas recebe as mesmas informações que pre-receive via entrada padrão:

<old-value> <new-value> <ref-name>

Pré-compromisso #

Este gancho é executado toda vez que você executa o git commit, para verificar o que está prestes a ser confirmado. Você pode usar este gancho para inspecionar o instantâneo que está prestes a ser confirmado.

Esse tipo de gancho é útil para executar testes automatizados para garantir que o commit de entrada não interrompa a funcionalidade existente do seu projeto. Esse tipo de gancho também pode verificar erros de espaço em branco ou EOL.

Nenhum argumento é passado para o script de pré-commit, e sair com um status diferente de zero aborta todo o commit.

Prepare-commit-msg #

Este gancho é chamado após o gancho pre-commit para preencher o editor de texto com uma mensagem de confirmação. Isso normalmente é usado para alterar as mensagens de confirmação geradas automaticamente para confirmações compactadas ou mescladas.

Um a três argumentos são passados ​​para este gancho:

  • O nome de um arquivo temporário que contém a mensagem.
  • O tipo de commit, seja
  • mensagem (opção -m ou -F),
  • modelo (opção -t),
  • merge (se for um commit de merge), ou
  • squash (se estiver esmagando outros commits).
  • O hash SHA1 do commit relevante. Isso só é fornecido se a opção -c, -C ou --amend foi fornecida.

Semelhante ao pre-commit, sair com um status diferente de zero aborta o commit.

Pré-rebase #

Este gancho é chamado antes do git rebase começar a alterar a estrutura do código. Esse gancho normalmente é usado para garantir que uma operação de rebase seja apropriada.

Este gancho leva 2 parâmetros:

  1. o ramo upstream do qual a série foi bifurcada e
  2. o branch que está sendo rebaseado (vazio ao fazer o rebase do branch atual).

Você pode abortar a operação de rebase saindo com um status diferente de zero.

Pré-recebimento #

Este gancho é executado toda vez que alguém usa git push para enviar commits para o repositório. Ele sempre reside no repositório remoto que é o destino do push e não no repositório de origem (local).

O gancho é executado antes de qualquer referência ser atualizada. Normalmente é usado para impor qualquer tipo de política de desenvolvimento.

O script não aceita parâmetros, mas cada referência que está sendo enviada é passada para o script em uma linha separada na entrada padrão no seguinte formato:

<old-value> <new-value> <ref-name>

Atualizar #

Este gancho é chamado após pre-receive, e funciona da mesma maneira. Ele é chamado antes que qualquer coisa seja realmente atualizada, mas é chamado separadamente para cada referência que foi enviada ao invés de todas as referências de uma vez.

Este gancho aceita os 3 argumentos a seguir:

  • nome da referência que está sendo atualizada,
  • nome do objeto antigo armazenado na ref, e
  • novo nome do objeto armazenado na ref.

Esta é a mesma informação passada para pre-receive, mas como update é invocado separadamente para cada ref, você pode rejeitar algumas refs enquanto permite outras.
