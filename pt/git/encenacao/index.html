Vale a pena notar que o staging tem pouco a ver com os próprios ‘arquivos’ e tudo a ver com as mudanças dentro de cada arquivo. Nós preparamos arquivos que contêm alterações e o git rastreia as alterações como commits (mesmo quando as alterações em um commit são feitas em vários arquivos).

A distinção entre arquivos e commits pode parecer pequena, mas entender essa diferença é fundamental para entender funções essenciais como cherry-pick e diff. (Veja a frustração em [comentários sobre a complexidade de uma resposta aceita que propõe o cherry-pick como uma ferramenta de gerenciamento de arquivos](http://stackoverflow.com/questions/449541/how-do-you-merge-selective-files- com-git-merge).)

Qual é um bom lugar para explicar conceitos? Está nas observações?

Conceitos chave:

Um arquivo é a metáfora mais comum dos dois na tecnologia da informação. A prática recomendada determina que um nome de arquivo não seja alterado à medida que seu conteúdo for alterado (com algumas exceções reconhecidas).

Um commit é uma metáfora exclusiva do gerenciamento de código-fonte. Commits são mudanças relacionadas a um esforço específico, como uma correção de bug. Os commits geralmente envolvem vários arquivos. Uma única correção de bug menor pode envolver ajustes em modelos e CSS em arquivos exclusivos. À medida que a mudança é descrita, desenvolvida, documentada, revisada e implantada, as mudanças nos arquivos separados podem ser anotadas e tratadas como uma única unidade. A única unidade neste caso é o commit. Igualmente importante, focar apenas no commit durante uma revisão permite que as linhas de código inalteradas nos vários arquivos afetados sejam ignoradas com segurança.

Preparando todas as alterações nos arquivos #

git add -A

git add .

Na versão 2.x, o git add . fará o stage de todas as alterações nos arquivos no diretório atual e em todos os seus subdiretórios. No entanto, em 1.x, ele irá apenas preparar arquivos novos e modificados, não arquivos excluídos.

Use git add -A, ou seu comando equivalente git add --all, para preparar todas as alterações nos arquivos em qualquer versão do git.

Desmontar um arquivo que contém alterações #

git reset <filePath>

Adicione alterações por hunk #

Você pode ver quais “pedaços” de trabalho seriam preparados para confirmação usando o sinalizador de patch:

git add -p

ou

git add --patch

Isso abre um prompt interativo que permite que você veja as diferenças e decida se deseja incluí-las ou não.

Stage this hunk [y,n,q,a,d,/,s,e,?]?
  • y prepara este pedaço para o próximo commit
  • n não prepare este pedaço para o próximo commit
  • q sair; não encene este pedaço ou qualquer um dos pedaços restantes
  • um prepara este pedaço e todos os pedaços posteriores no arquivo
  • d não prepare este pedaço ou qualquer um dos pedaços posteriores no arquivo
  • g selecione um pedaço para ir
  • / procure um pedaço que corresponda ao regex fornecido
  • j deixe este pedaço indeciso, veja o próximo pedaço indeciso
  • J deixe este pedaço indeciso, veja o próximo pedaço
  • k deixe este pedaço indeciso, veja o pedaço indeciso anterior
  • K deixe este pedaço indeciso, veja o pedaço anterior
  • s divide o pedaço atual em pedaços menores
  • e edite manualmente o pedaço atual
  • ? ajuda do bloco de impressão

Isso facilita a captura de alterações que você não deseja confirmar.

Você também pode abrir isso via git add --interactive e selecionando p.

Adição interativa #

git add -i (ou --interactive) lhe dará uma interface interativa onde você pode editar o índice, para preparar o que você quer ter no próximo commit. Você pode adicionar e remover alterações em arquivos inteiros, adicionar arquivos não rastreados e remover arquivos do rastreamento, mas também selecionar subseções de alterações para colocar no índice, selecionando partes de alterações a serem adicionadas, dividindo essas partes ou até editando o diff . Muitas ferramentas gráficas de commit para Git (como, por exemplo, git gui) incluem esse recurso; isso pode ser mais fácil de usar do que a versão de linha de comando.

É muito útil (1) se você tiver alterações emaranhadas no diretório de trabalho que deseja colocar em commits separados, e não tudo em um único commit (2) se você estiver no meio de um rebase interativo e quiser dividir também grande compromisso.

$ git add -i
           staged     unstaged path
  1:    unchanged        +4/-4 index.js
  2:        +1/-0      nothing package.json

*** Commands ***
  1: status       2: update       3: revert       4: add untracked
  5: patch        6: diff         7: quit         8: help
What now>

A metade superior dessa saída mostra o estado atual do índice dividido em colunas preparadas e não preparadas:

  1. index.js teve 4 linhas adicionadas e 4 linhas removidas. No momento, ele não está preparado, pois o status atual informa “inalterado”. Quando este arquivo for testado, o bit +4/-4 será transferido para a coluna preparada e a coluna não preparada lerá “nada”.
  2. package.json teve uma linha adicionada e foi testada. Não há mais alterações, pois ele foi preparado conforme indicado pela linha “nada” na coluna não preparado.

A metade inferior mostra o que você pode fazer. Digite um número (1-8) ou uma letra (s, u, r, a, p, d, q, h).

status mostra uma saída idêntica à parte superior da saída acima.

update permite que você faça mais alterações nos commits encenados com sintaxe adicional.

revert reverterá as informações de confirmação de estágio de volta para HEAD.

add untracked permite adicionar caminhos de arquivo anteriormente não rastreados pelo controle de versão.

patch permite que um caminho seja selecionado de uma saída semelhante a status para análise posterior.

diff exibe o que será confirmado.

quit sai do comando.

help apresenta mais ajuda sobre como usar este comando.

Mostrar alterações em etapas #

Para exibir os blocos que são preparados para confirmação:

git diff --cached

Preparando um único arquivo #

Para preparar um arquivo para confirmação, execute

git add <filename>

Arquivos excluídos do estágio #

git rm filename

Para excluir o arquivo do git sem removê-lo do disco, use o sinalizador --cached

git rm --cached filename
