Este tópico ilustra como evitar adicionar arquivos indesejados (ou alterações de arquivo) em um repositório Git. Existem várias maneiras (global ou local .gitignore, .git/exclude, git update-index --assume-unchanged e git update-index --skip-tree), mas tenha em mente O Git está gerenciando conteúdo, o que significa: ignorar na verdade ignora uma pasta conteúdo (ou seja, arquivos). Uma pasta vazia seria ignorada por padrão, pois não pode ser adicionada de qualquer maneira.

Ignorando arquivos e diretórios com um arquivo .gitignore #

Você pode fazer o Git ignorar certos arquivos e diretórios — ou seja, exclua-os de serem rastreados pelo Git — criando um ou mais arquivos .gitignore em seu repositório.

Em projetos de software, .gitignore normalmente contém uma lista de arquivos e/ou diretórios que são gerados durante o processo de construção ou em tempo de execução. As entradas no arquivo .gitignore podem incluir nomes ou caminhos apontando para:

  1. recursos temporários, por exemplo. caches, arquivos de log, código compilado, etc.
  2. arquivos de configuração local que não devem ser compartilhados com outros desenvolvedores
  3. arquivos contendo informações secretas, como senhas de login, chaves e credenciais

Quando criadas no diretório de nível superior, as regras serão aplicadas recursivamente a todos os arquivos e subdiretórios em todo o repositório. Quando criado em um subdiretório, as regras serão aplicadas a esse diretório específico e seus subdiretórios.

Quando um arquivo ou diretório é ignorado, não será:

  1. rastreado pelo Git
  2. relatado por comandos como git status ou git diff
  3. encenado com comandos como git add -A

No caso incomum de você precisar ignorar arquivos rastreados, cuidado especial deve ser tomado. Veja: Ignorar arquivos que já foram submetidos a um repositório Git.

Exemplos #

Aqui estão alguns exemplos genéricos de regras em um arquivo .gitignore, com base em padrões de arquivo glob:

# Lines starting with `#` are comments.

# Ignore files called 'file.ext'
file.ext

# Comments can't be on the same line as rules!
# The following line ignores files called 'file.ext # not a comment'
file.ext # not a comment 

# Ignoring files with full path.
# This matches files in the root directory and subdirectories too.
# i.e. otherfile.ext will be ignored anywhere on the tree.
dir/otherdir/file.ext
otherfile.ext

# Ignoring directories
# Both the directory itself and its contents will be ignored.
bin/
gen/

# Glob pattern can also be used here to ignore paths with certain characters.
# For example, the below rule will match both build/ and Build/
[bB]uild/

# Without the trailing slash, the rule will match a file and/or
# a directory, so the following would ignore both a file named `gen`
# and a directory named `gen`, as well as any contents of that directory
bin
gen

# Ignoring files by extension
# All files with these extensions will be ignored in
# this directory and all its sub-directories.
*.apk
*.class

# It's possible to combine both forms to ignore files with certain
# extensions in certain directories. The following rules would be
# redundant with generic rules defined above.
java/*.apk
gen/*.class

# To ignore files only at the top level directory, but not in its
# subdirectories, prefix the rule with a `/`
/*.apk
/*.class

# To ignore any directories named DirectoryA 
# in any depth use ** before DirectoryA
# Do not forget the last /, 
# Otherwise it will ignore all files named DirectoryA, rather than directories
**/DirectoryA/
# This would ignore 
# DirectoryA/
# DirectoryB/DirectoryA/ 
# DirectoryC/DirectoryB/DirectoryA/
# It would not ignore a file named DirectoryA, at any level

# To ignore any directory named DirectoryB within a 
# directory named DirectoryA with any number of 
# directories in between, use ** between the directories
DirectoryA/**/DirectoryB/
# This would ignore 
# DirectoryA/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryB/ 
# DirectoryA/DirectoryQ/DirectoryW/DirectoryB/

# To ignore a set of files, wildcards can be used, as can be seen above.
# A sole '*' will ignore everything in your folder, including your .gitignore file.
# To exclude specific files when using wildcards, negate them.
# So they are excluded from the ignore list:
!.gitignore 

# Use the backslash as escape character to ignore files with a hash (#)
# (supported since 1.6.2.1)
\#*#

A maioria dos arquivos .gitignore são padrão em vários idiomas, então para começar, aqui está um conjunto de [amostras de arquivos .gitignore][4] listados por idioma do qual clonar ou copiar/modificar em seu projeto. Alternativamente, para um novo projeto, você pode considerar a geração automática de um arquivo inicial usando uma [ferramenta online][5].

Outras formas de .gitignore #

Arquivos .gitignore devem ser submetidos como parte do repositório. Se você quiser ignorar determinados arquivos sem confirmar as regras de ignorar, aqui estão algumas opções:

  • Edite o arquivo .git/info/exclude (usando a mesma sintaxe que .gitignore). As regras serão globais no escopo do repositório;
  • Configure [um arquivo gitignore global](https://www.wikiod.com/pt/git/ignorando-arquivos-e-pastas#Um arquivo .gitignore global) que aplicará regras de ignorar a todos os seus repositórios locais:

Além disso, você pode ignorar as alterações locais nos arquivos rastreados sem alterar a configuração global do git com:

  • git update-index --skip-worktree [<arquivo>...]: para pequenas modificações locais
  • git update-index --assume-unchanged [<file>...]: para produção pronta, arquivos upstream sem alteração

Veja [mais detalhes sobre as diferenças entre os últimos sinalizadores][6] e a [documentação do git update-index][7] para mais opções.

Limpando arquivos ignorados #

Você pode usar git clean -X para limpar arquivos ignorados:

git clean -Xn #display a list of ignored files
git clean -Xf #remove the previously displayed files

Nota: -X (caps) limpa somente arquivos ignorados. Use -x (sem maiúsculas) para também remover arquivos não rastreados.

Veja a documentação do git clean para mais detalhes.

Verificando se um arquivo é ignorado #

O comando git check-ignore informa sobre arquivos ignorados pelo Git.

Você pode passar nomes de arquivos na linha de comando, e git check-ignore listará os nomes de arquivos que são ignorados. Por exemplo:

$ cat .gitignore
*.o
$ git check-ignore example.o Readme.md
example.o

Aqui, somente arquivos *.o são definidos em .gitignore, então o Readme.md não é listado na saída de git check-ignore.

Se você quiser ver a linha de qual .gitignore é responsável por ignorar um arquivo, adicione -v ao comando git check-ignore:

$ git check-ignore -v example.o Readme.md
.gitignore:1:*.o        example.o

A partir do Git 1.7.6 você também pode usar git status --ignored para ver arquivos ignorados. Você pode encontrar mais informações sobre isso na documentação oficial ou em Encontrando arquivos ignorados por .gitignore.

Exceções em um arquivo .gitignore #

Se você ignorar arquivos usando um padrão, mas tiver exceções, coloque um ponto de exclamação (!) como prefixo na exceção. Por exemplo:

*.txt
!important.txt

O exemplo acima instrui o Git a ignorar todos os arquivos com a extensão .txt exceto os arquivos chamados important.txt.

Se o arquivo estiver em uma pasta ignorada, você pode NÃO reincluí-lo facilmente:

folder/
!folder/*.txt

Neste exemplo, todos os arquivos .txt na pasta permaneceriam ignorados.

O caminho certo é reincluir a própria pasta em uma linha separada, depois ignorar todos os arquivos em folder por *, finalmente reincluir o *.txt em folder, como o seguinte:

!folder/
folder/*
!folder/*.txt

Observação: para nomes de arquivo que começam com um ponto de exclamação, adicione dois pontos de exclamação ou escape com o caractere \:

!!includethis
\!excludethis

Um arquivo .gitignore global #

Para que o Git ignore determinados arquivos em todos os repositórios, você pode criar um .gitignore global com o seguinte comando em seu terminal ou prompt de comando:

$ git config --global core.excludesfile <Path_To_Global_gitignore_file>

O Git agora usará isso além do arquivo .gitignore de cada repositório. As regras para isso são:

  • Se o arquivo local .gitignore inclui explicitamente um arquivo enquanto o global .gitignore o ignora, o local .gitignore tem prioridade (o arquivo será incluído)
  • Se o repositório for clonado em várias máquinas, então o .gigignore global deve ser carregado em todas as máquinas ou pelo menos incluí-lo, pois os arquivos ignorados serão enviados para o repositório enquanto o PC com o .gitignore global não atualizaria. É por isso que um repositório .gitignore específico é uma ideia melhor do que um global se o projeto for trabalhado por uma equipe

Este arquivo é um bom lugar para manter ignorados específicos de plataforma, máquina ou usuário, por exemplo. OSX .DS_Store, Windows Thumbs.db ou Vim *.ext~ e *.ext.swp ignora se você não quiser mantê-los no repositório. Assim, um membro da equipe trabalhando no OS X pode adicionar todos os .DS_STORE e _MACOSX (o que na verdade é inútil), enquanto outro membro da equipe no Windows pode ignorar todos os thumbs.bd

Ignora arquivos que já foram submetidos a um repositório Git #

Se você já adicionou um arquivo ao seu repositório Git e agora deseja parar de rastreá-lo (para que ele não esteja presente em commits futuros), você pode removê-lo do índice:

git rm --cached <file>

Isso removerá o arquivo do repositório e impedirá que outras alterações sejam rastreadas pelo Git. A opção --cached garantirá que o arquivo não seja excluído fisicamente.

Observe que o conteúdo do arquivo adicionado anteriormente ainda estará visível por meio do histórico do Git.

Lembre-se de que, se outra pessoa fizer pull do repositório depois que você remover o arquivo do índice, sua cópia será excluída fisicamente.

Você pode fazer o Git fingir que a versão do diretório de trabalho do arquivo está atualizada e ler a versão do índice (ignorando assim as alterações nela) com o bit “skip worktree”:

git update-index --skip-worktree <file>

A escrita não é afetada por este bit, a segurança do conteúdo ainda é a primeira prioridade. Você nunca perderá suas preciosas mudanças ignoradas; por outro lado, este bit entra em conflito com o stashing: para remover este bit, use

git update-index --no-skip-worktree <file>

Às vezes é erradamente recomendado mentir para o Git e fazer com que ele assuma que o arquivo não foi alterado sem examiná-lo. À primeira vista, parece ignorar quaisquer outras alterações no arquivo, sem removê-lo de seu índice:

git update-index --assume-unchanged <file>

Isso forçará o git a ignorar qualquer alteração feita no arquivo (lembre-se de que, se você fizer pull de qualquer alteração neste arquivo ou armazená-lo, suas alterações ignoradas serão perdidas)

Se você quiser que o git “se importe” com esse arquivo novamente, execute o seguinte comando:

git update-index --no-assume-unchanged <file>

Ignorar arquivos localmente sem confirmar regras de ignorar #

O .gitignore ignora os arquivos localmente, mas destina-se a ser enviado ao repositório e compartilhado com outros contribuidores e usuários. Você pode definir um .gitignore global, mas todos os seus repositórios compartilharão essas configurações.

Se você quiser ignorar certos arquivos em um repositório localmente e não fazer o arquivo parte de nenhum repositório, edite .git/info/exclude dentro do seu repositório.

Por exemplo:

# these files are only ignored on this repo
# these rules are not shared with anyone
# as they are personal                                              
gtk_tests.py
gui/gtk/tests/*
localhost
pushReports.py
server/

Ignorando um arquivo em qualquer diretório #

Para ignorar um arquivo foo.txt em qualquer diretório você deve apenas escrever seu nome:

foo.txt # matches all files 'foo.txt' in any directory

Se você quiser ignorar o arquivo apenas em parte da árvore, você pode especificar os subdiretórios de um diretório específico com o padrão **:

bar/**/foo.txt # matches all files 'foo.txt' in 'bar' and all subdirectories

Ou você pode criar um arquivo .gitignore no diretório bar/. Equivalente ao exemplo anterior seria criar o arquivo bar/.gitignore com este conteúdo:

foo.txt # matches all files 'foo.txt' in any directory under bar/

Ignorando alterações subsequentes em um arquivo (sem removê-lo) #

Às vezes, você deseja manter um arquivo no Git, mas ignora as alterações subsequentes.

Diga ao Git para ignorar as alterações em um arquivo ou diretório usando update-index:

git update-index --assume-unchanged my-file.txt

O comando acima instrui o Git a assumir que my-file.txt não foi alterado, e não para verificar ou relatar alterações. O arquivo ainda está presente no repositório.

Isso pode ser útil para fornecer padrões e permitir substituições de ambiente local, por exemplo:

# create a file with some values in
cat <<EOF
MYSQL_USER=app
MYSQL_PASSWORD=FIXME_SECRET_PASSWORD
EOF > .env

# commit to Git
git add .env
git commit -m "Adding .env template"

# ignore future changes to .env
git update-index --assume-unchanged .env

# update your password
vi .env

# no changes!
git status

Ignorando arquivos em subpastas (vários arquivos gitignore) #

Suponha que você tenha uma estrutura de repositório como esta:

examples/
    output.log
src/
    <files not shown>
    output.log
README.md

output.log no diretório de exemplos é válido e necessário para o projeto obter um entendimento enquanto o abaixo de src/ é criado durante a depuração e não deve estar no histórico ou parte do repositório.

Existem duas maneiras de ignorar este arquivo. Você pode colocar um caminho absoluto no arquivo .gitignore na raiz do diretório de trabalho:

# /.gitignore
src/output.log

Alternativamente, você pode criar um arquivo .gitignore no diretório src/ e ignorar o arquivo relativo a este .gitignore:

# /src/.gitignore
output.log

Se você não tiver certeza de quais regras listar em seu arquivo .gitignore, ou apenas quiser adicionar exceções geralmente aceitas ao seu projeto, você pode escolher ou gerar um arquivo .gitignore:

  • https://github.com/github/gitignore

Crie uma pasta vazia #

Não é possível adicionar e submeter uma pasta vazia no Git devido ao fato de que o Git gerencia arquivos e anexa seu diretório a eles, o que reduz os commits e melhora a velocidade. Para contornar isso, existem dois métodos:

Método um: .gitkeep

Um truque para contornar isso é usar um arquivo .gitkeep para registrar a pasta para o Git. Para fazer isso, basta criar o diretório necessário e adicionar um arquivo .gitkeep à pasta. Este arquivo está em branco e não serve para nada além de apenas registrar a pasta. Para fazer isso no Windows (que tem convenções de nomenclatura de arquivos estranhas), basta abrir o git bash no diretório e executar o comando:

$ toque em .gitkeep

Este comando apenas cria um arquivo .gitkeep em branco no diretório atual

Método dois: dummy.txt

Outro hack para isso é muito parecido com o acima e os mesmos passos podem ser seguidos, mas ao invés de um .gitkeep, apenas use um dummy.txt. Isso tem a vantagem adicional de poder criá-lo facilmente no Windows usando o menu de contexto. E você pode deixar mensagens engraçadas neles também. Você também pode usar o arquivo .gitkeep para rastrear o diretório vazio. .gitkeep normalmente é um arquivo vazio que é adicionado para rastrear o diretório vazio.

Encontrando arquivos ignorados por .gitignore #

Você pode listar todos os arquivos ignorados pelo git no diretório atual com o comando:

git status --ignored

Então, se tivermos uma estrutura de repositório como esta:

.git
.gitignore
./example_1
./dir/example_2
./example_2

…e arquivo .gitignore contendo:

example_2

…o resultado do comando será:

$ git status --ignored

On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

.gitignore
.example_1

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)

dir/
example_2

Se você deseja listar arquivos recursivamente ignorados em diretórios, você deve usar o parâmetro adicional - --untracked-files=all

Resultado ficará assim:

$ git status --ignored --untracked-files=all
On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

.gitignore
example_1

Ignored files:
  (use "git add -f <file>..." to include in what will be committed)

dir/example_2
example_2

Ignorando apenas parte de um arquivo [stub] #

Às vezes, você pode querer ter alterações locais em um arquivo que não deseja submeter ou publicar. Idealmente, as configurações locais devem ser concentradas em um arquivo separado que pode ser colocado em .gitignore, mas às vezes como uma solução de curto prazo pode ser útil ter algo local em um arquivo de check-in.

Você pode fazer o Git “desver” essas linhas usando um filtro limpo. Eles nem aparecem nos diffs.

Suponha que aqui está um trecho do arquivo file1.c:

struct settings s;
s.host = "localhost";
s.port = 5653;
s.auth = 1;
s.port = 15653; // NOCOMMIT
s.debug = 1; // NOCOMMIT
s.auth = 0; // NOCOMMIT

Você não quer publicar linhas NOCOMMIT em nenhum lugar.

Crie um filtro “nocommit” adicionando isso ao arquivo de configuração do Git como .git/config:

[filter "nocommit"]
    clean=grep -v NOCOMMIT

Adicione (ou crie) isso em .git/info/attributes ou .gitmodules:

file1.c filter=nocommit

E suas linhas NOCOMMIT estão ocultas do Git.

Ressalvas:

  • O uso de filtro limpo retarda o processamento de arquivos, especialmente no Windows.
  • A linha ignorada pode desaparecer do arquivo quando o Git a atualiza. Pode ser neutralizado com um filtro de manchas, mas é mais complicado.
  • Não testado no Windows

Ignorando alterações em arquivos rastreados. [toco] #

.gitignore e .git/info/exclude funcionam apenas para arquivos não rastreados.

Para definir o sinalizador de ignorar em um arquivo rastreado, use o comando update-index:

git update-index --skip-worktree myfile.c

Para reverter isso, use:

git update-index --no-skip-worktree myfile.c

Você pode adicionar este trecho ao seu git config global para ter comandos git hide, git unhide e git hidden mais convenientes:

[alias]
    hide   = update-index --skip-worktree
    unhide = update-index --no-skip-worktree
    hidden  = "!git ls-files -v | grep ^[hsS] | cut -c 3-"

Você também pode usar a opção –assume-unchanged com a função update-index

git update-index --assume-unchanged <file>

Se você quiser assistir a este arquivo novamente para as alterações, use

git update-index --no-assume-unchanged <file>

Quando o sinalizador –assume-unchanged é especificado, o usuário promete não alterar o arquivo e permite que o Git assuma que o arquivo da árvore de trabalho corresponde ao que está registrado no índice. O Git falhará caso precise modificar este arquivo no índice por exemplo. ao mesclar em um commit; portanto, caso o arquivo não rastreado seja alterado no upstream, você precisará lidar com a situação manualmente. O foco está no desempenho neste caso.

Enquanto o sinalizador –skip-worktree é útil quando você instrui o git a não tocar em um arquivo específico porque o arquivo será alterado localmente e você não deseja enviar acidentalmente as alterações (ou seja, arquivo de configuração/propriedades configurado para um determinado meio Ambiente). Skip-worktree tem precedência sobre assume-unchanged quando ambos são definidos.

Limpar arquivos já confirmados, mas incluídos em .gitignore #

Às vezes acontece que um arquivo estava sendo rastreado pelo git, mas em um momento posterior foi adicionado ao .gitignore, para parar de rastreá-lo. É um cenário muito comum esquecer de limpar esses arquivos antes de adicioná-los ao .gitignore. Nesse caso, o arquivo antigo ainda estará no repositório.

Para corrigir esse problema, pode-se realizar uma remoção “dry-run” de tudo no repositório, seguida de adicionar novamente todos os arquivos. Contanto que você não tenha alterações pendentes e o parâmetro --cached seja passado, este comando é bastante seguro para ser executado:

# Remove everything from the index (the files will stay in the file system) 
$ git rm -r --cached .

# Re-add everything (they'll be added in the current state, changes included)
$ git add .

# Commit, if anything changed. You should see only deletions
$ git commit -m 'Remove all files that are in the .gitignore'

# Update the remote
$ git push origin master
