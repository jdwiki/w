  • Alterando o comportamento de push padrão

Depois de alterar, preparar e confirmar o código com o Git, é necessário fazer push para disponibilizar suas alterações para outras pessoas e transferir suas alterações locais para o servidor de repositório. Este tópico abordará como enviar código corretamente usando o Git.

Sintaxe #

  • git push [-f | –força] [-v | –verbose] [<remote> [<refspec>…]]

Parâmetros #

Parâmetro Detalhes
–força Substitui a referência remota para corresponder à sua referência local. Pode fazer com que o repositório remoto perca commits, então use com cuidado.
–verbose Execute detalhadamente.
<remoto> O repositório remoto que é o destino da operação push.
<refspec>… Especifique qual referência remota atualizar com qual referência ou objeto local.

A montante e a jusante #

Em termos de controle de origem, você está “downstream” quando copia (clone, checkout, etc) de um repositório. As informações fluíam “a jusante” para você.

Quando você faz alterações, geralmente deseja enviá-las de volta “upstream” para que eles entrem nesse repositório para que todos que extraiam do mesma fonte está funcionando com todas as mesmas alterações. Isso é principalmente um questão social de como todos podem coordenar seu trabalho ao invés de um exigência técnica de controle de origem. Você deseja obter suas alterações no projeto principal para que você não rastreie linhas divergentes de desenvolvimento.

Às vezes você lerá sobre gerenciadores de pacotes ou lançamentos (as pessoas, não a ferramenta) falando sobre o envio de alterações para “upstream”. Este geralmente significa que eles tiveram que ajustar as fontes originais para que pudessem crie um pacote para seu sistema. Eles não querem continuar fazendo essas alterações, portanto, se eles as enviarem “upstream” para a fonte original, eles não deveriam ter que lidar com o mesmo problema na próxima versão.

(Fonte)

Envie um objeto específico para uma ramificação remota #

Sintaxe geral #

git push <remotename> <object>:<remotebranchname>

Exemplo #

git push origin master:wip-yourname

Enviará seu branch master para o branch wip-yourname de origem (na maioria das vezes, o repositório do qual você clonou).

Excluir branch remoto #

Excluir a ramificação remota é o equivalente a enviar um objeto vazio para ela.

git push <remotename> :<remotebranchname>

Exemplo #

git push origin :wip-yourname

Irá excluir o branch remoto wip-yourname

Em vez de usar os dois pontos, você também pode usar o sinalizador –delete, que é mais legível em alguns casos.

Exemplo #

git push origin --delete wip-yourname

Empurre um único commit #

Se você tiver um único commit em sua ramificação que deseja enviar para um controle remoto sem enviar mais nada, você pode usar o seguinte

git push <remotename> <commit SHA>:<remotebranchname>

Exemplo #

Assumindo um histórico do git como este

eeb32bc Commit 1 - already pushed
347d700 Commit 2 - want to push
e539af8 Commit 3 - only local
5d339db Commit 4 - only local

para enviar apenas commit 347d700 para master remoto, use o seguinte comando

git push origin 347d700:master

Empurre #

git push

enviará seu código para o upstream existente. Dependendo da configuração do push, ele enviará o código do seu branch atual (padrão no Git 2.x) ou de todos os branches (padrão no Git 1.x).

Especifica o repositório remoto #

Ao trabalhar com o git, pode ser útil ter vários repositórios remotos. Para especificar um repositório remoto para o qual enviar, basta anexar seu nome ao comando.

git push origin

Especificar Filial #

Para enviar para um branch específico, diga feature_x:

git push origin feature_x

Defina o branch de rastreamento remoto #

A menos que a ramificação em que você está trabalhando venha originalmente de um repositório remoto, simplesmente usar git push não funcionará na primeira vez. Você deve executar o seguinte comando para dizer ao git para enviar a ramificação atual para uma combinação específica de remoto/ramificação

git push --set-upstream origin master

Aqui, master é o nome da ramificação na origem remota. Você pode usar -u como um atalho para --set-upstream.

Empurrando para um novo repositório #

Para enviar para um repositório que você ainda não criou ou está vazio:

  1. Crie o repositório no GitHub (se aplicável)
  2. Copie o URL fornecido a você, no formato https://github.com/USERNAME/REPO_NAME.git
  3. Vá para o seu repositório local e execute git remote add origin URL
    • To verify it was added, run git remote -v
  4. Execute o git push origin master

Seu código agora deve estar no GitHub

Para mais informações, veja Adicionando um repositório remoto

Explicação #

O código push significa que o git analisará as diferenças de seus commits locais e remotos e os enviará para serem escritos no upstream. Quando o push é bem-sucedido, seu repositório local e remoto são sincronizados e outros usuários podem ver seus commits.

Para obter mais detalhes sobre os conceitos de “upstream” e “downstream”, consulte [Observações][2].

1: https://www.wikiod.com/pt/git/git-remoto#Adicionar um repositório remoto [2]: //stackoverflow.com/documentation/git/2600/push#remarks

Forçar Empurrar #

Às vezes, quando você tem alterações locais incompatíveis com alterações remotas (ou seja, quando você não pode avançar a ramificação remota ou a ramificação remota não é um ancestral direto de sua ramificação local), a única maneira de enviar suas alterações é um push forçado .

git push -f

ou

git push --force

Anotações importantes #

Isso sobrescreverá quaisquer alterações remotas e seu controle remoto corresponderá ao seu local.

Atenção: O uso deste comando pode fazer com que o repositório remoto perda commits. Além disso, é fortemente desaconselhado fazer um push forçado se você estiver compartilhando este repositório remoto com outras pessoas, já que o histórico deles reterá todos os commits sobrescritos, deixando seu trabalho fora de sincronia com o repositório remoto.

Como regra geral, apenas force o empurrão quando:

  • Ninguém, exceto você puxou as alterações que você está tentando substituir
  • Você pode forçar todos a clonar uma nova cópia após o push forçado e fazer com que todos apliquem suas alterações nela (as pessoas podem odiá-lo por isso).

Empurrar tags #

git push --tags

Envia todas as git tags no repositório local que não estão no remoto.

Alterando o comportamento de push padrão #

Atual atualiza a ramificação no repositório remoto que compartilha um nome com a ramificação de trabalho atual.

git config push.default current

Simples envia push para o branch upstream, mas não funcionará se o branch upstream for chamado de outra coisa.

git config push.default simple

Upstream envia push para o branch upstream, não importa como ele seja chamado.

git config push.default upstream

Correspondência envia todas as ramificações que correspondem no local e no remoto git config push.default upstream

Depois de definir o estilo preferido, use

git push

para atualizar o repositório remoto.
