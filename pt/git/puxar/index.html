Ao contrário do push com o Git, onde suas alterações locais são enviadas para o servidor do repositório central, o pull com o Git pega o código atual no servidor e o ‘puxa’ do servidor do repositório para sua máquina local. Este tópico explica o processo de extração de código de um repositório usando o Git, bem como as situações que podem ser encontradas ao extrair código diferente para a cópia local.

Sintaxe #

  • git pull [opções [<repositório> [<refspec>…]]

Parâmetros #

Parâmetros Detalhes
--quieto Nenhuma saída de texto
-q abreviação de --quiet
--verbose saída de texto detalhado. Passado para os comandos fetch e merge/rebase respectivamente.
-v abreviação de --verbose
--[no-]recurse-submodules[=yes|sob demanda|não] Buscar novos commits para submódulos? (Não que isso não seja um pull/checkout)

git pull executa git fetch com os parâmetros fornecidos e chama git merge para mesclar as cabeças de branch recuperadas no branch atual.

Atualizando com mudanças locais #

Quando mudanças locais estão presentes, o comando git pull aborta o relatório:

erro: Suas alterações locais nos seguintes arquivos seriam substituídas por mesclagem

Para atualizar (como o svn update fez com o subversion), você pode executar:

git stash
git pull --rebase 
git stash pop

Uma maneira conveniente pode ser definir um alias usando:

git config --global alias.up '!git stash && git pull --rebase && git stash pop'

git config --global alias.up 'pull --rebase --autostash'

Em seguida, você pode simplesmente usar:

git up

Puxe, sobrescreva local #

git fetch
git reset --hard origin/master

Cuidado: Embora commits descartados usando reset --hard possam ser recuperados usando reflog e reset, as alterações não confirmadas são excluídas para sempre.

Altere origin e master para o remoto e branch que você deseja forçar o pull, respectivamente, se eles tiverem nomes diferentes.

Puxe o código do controle remoto #

git pull

Mantendo o histórico linear ao puxar #

Rebase ao puxar #

Se você estiver recebendo novos commits do repositório remoto e tiver alterações locais no branch atual, o git mesclará automaticamente a versão remota e sua versão. Se você quiser reduzir o número de merges em seu branch, você pode dizer ao git para rebase seus commits na versão remota do branch.

git pull --rebase

Tornando-o o comportamento padrão #

Para tornar esse comportamento padrão para branches recém-criados, digite o seguinte comando:

git config branch.autosetuprebase always

Para alterar o comportamento de uma ramificação existente, use isto:

git config branch.BRANCH_NAME.rebase true

E

git pull --no-rebase

Para executar um pull de mesclagem normal.

Verifica se é encaminhável #

Para permitir apenas o encaminhamento rápido da ramificação local, você pode usar:

git pull --ff-only

Isso exibirá um erro quando a ramificação local não for de avanço rápido e precisar ser rebaseada ou mesclada com o upstream.

Extraindo alterações para um repositório local #

Puxar Simples #

Quando você estiver trabalhando em um repositório remoto (digamos, GitHub) com outra pessoa, em algum momento você desejará compartilhar suas alterações com ela. Depois que eles enviarem suas alterações para um repositório remoto, você pode recuperar essas alterações puxando deste repositório.

git pull

Vai fazê-lo, na maioria dos casos.

Puxe de um remoto ou branch diferente #

Você pode extrair alterações de um remoto ou branch diferente especificando seus nomes

git pull origin feature-A

Vai puxar o branch feature-A do formulário origin para o seu branch local. Observe que você pode fornecer diretamente uma URL em vez de um nome remoto e um nome de objeto, como um SHA de confirmação em vez de um nome de ramificação.

Puxar manualmente #

Para imitar o comportamento de um git pull, você pode usar git fetch e depois git merge

git fetch origin # retrieve objects and update refs from origin
git merge origin/feature-A # actually perform the merge

Isso pode lhe dar mais controle e permite que você inspecione a ramificação remota antes de mesclá-la. De fato, após a busca, você pode ver as ramificações remotas com git branch -a e verificá-las com

git checkout -b local-branch-name origin/feature-A # checkout the remote branch
# inspect the branch, make commits, squash, ammend or whatever
git checkout merging-branches # moving to the destination branch
git merge local-branch-name # performing the merge

Isso pode ser muito útil ao processar solicitações de pull.

Pull, “permissão negada” #

Alguns problemas podem ocorrer se a pasta .git tiver permissão errada. Corrigindo este problema definindo o proprietário da pasta .git completa. Às vezes acontece que outro usuário puxe e altere os direitos da pasta ou dos arquivos .git.

Para consertar o problema:

chown -R youruser:yourgroup .git/
