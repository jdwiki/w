Repositório #

Um repositório git é uma estrutura de dados em disco que armazena metadados para um conjunto de arquivos e diretórios.

Ele fica na pasta .git/ do seu projeto. Toda vez que você envia dados para o git, eles são armazenados aqui. Inversamente, .git/ contém todos os commits.

Sua estrutura básica é assim:

.git/
    objects/
    refs/

Objetos #

git é fundamentalmente um armazenamento de valor-chave. Quando você adiciona dados ao git, ele cria um objeto e usa o hash SHA-1 do conteúdo do objeto como chave.

Portanto, qualquer conteúdo em git pode ser consultado por seu hash:

git cat-file -p 4bb6f98

Existem 4 tipos de Object:

  • bolha
  • árvore
  • comprometer
  • marca

Cabeçalho ref #

HEAD é uma ref especial. Sempre aponta para o objeto atual.

Você pode ver para onde está apontando verificando o arquivo .git/HEAD.

Normalmente, HEAD aponta para outro ref:

$cat .git/HEAD
ref: refs/heads/mainline

Mas também pode apontar diretamente para um objeto:

$ cat .git/HEAD
4bb6f98a223abc9345a0cef9200562333

Isso é o que é conhecido como “cabeça destacada” - porque HEAD não está anexado (apontando para) qualquer ref, mas aponta diretamente para um objeto.

Referências #

Um ref é essencialmente um ponteiro. É um nome que aponta para um objeto. Por exemplo,

"master" --> 1a410e...

Eles são armazenados em `.git/refs/heads/ em arquivos de texto simples.

$ cat .git/refs/heads/mainline
4bb6f98a223abc9345a0cef9200562333

Isso é comumente chamado de ramos. No entanto, você notará que no git não existe um branch - apenas um ref.

Agora, é possível navegar pelo git simplesmente pulando para diferentes objetos diretamente por seus hashes. Mas isso seria terrivelmente inconveniente. Um ref fornece um nome conveniente para se referir a objects. É muito mais fácil pedir ao git para ir a um lugar específico pelo nome do que pelo hash.

Confirmar objeto #

Um commit é provavelmente o tipo object mais familiar para usuários git, pois é o que eles estão acostumados a criar com os comandos git commit.

No entanto, o commit não contém diretamente nenhum arquivo ou dado alterado. Em vez disso, ele contém principalmente metadados e ponteiros para outros objetos que contêm o conteúdo real do commit.

Um commit contém algumas coisas:

  • hash de uma árvore
  • hash de um pai commit
  • nome/e-mail do autor, nome/e-mail do commiter
  • mensagem de confirmação

Você pode ver o conteúdo de qualquer commit assim:

$ git cat-file commit 5bac93
tree 04d1daef...
parent b7850ef5...
author Geddy Lee <glee@rush.com>
commiter Neil Peart <npeart@rush.com>

First commit!

Árvore #

Uma observação muito importante é que os objetos tree armazenam TODOS os arquivos em seu projeto, e armazena arquivos inteiros, não diffs. Isso significa que cada commit contém um instantâneo de todo o projeto*.

*Tecnicamente, apenas os arquivos alterados são armazenados. Mas este é mais um detalhe de implementação para eficiência. Do ponto de vista do design, um commit deve ser considerado como contendo uma cópia completa do projeto.

Pai #

A linha parent contém um hash de outro objeto commit e pode ser considerada como um “ponteiro pai” que aponta para o “commit anterior”. Isso forma implicitamente um gráfico de commits conhecido como commit graph. Especificamente, é um gráfico acíclico dirigido (ou DAG).

Objeto Árvore #

Uma árvore basicamente representa uma pasta em um sistema de arquivos tradicional: contêineres aninhados para arquivos ou outras pastas.

Uma árvore contém:

  • 0 ou mais objetos blob
  • 0 ou mais objetos tree

Assim como você pode usar ls ou dir para listar o conteúdo de uma pasta, você pode listar o conteúdo de um objeto tree.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Você pode procurar os arquivos em um commit primeiro encontrando o hash da árvore no commit e, em seguida, olhando para essa árvore:

$ git cat-file commit 4bb6f93a
tree 07b1a631
parent ...
author ...
commiter ... 
 
$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
...

Objeto Blob #

Um blob contém conteúdo de arquivo binário arbitrário. Comumente, será um texto bruto, como código-fonte ou um artigo de blog. Mas poderia facilmente ser os bytes de um arquivo PNG ou qualquer outra coisa.

Se você tem o hash de um blob, você pode ver seu conteúdo.

$ git cat-file -p d429810
package com.example.project

class Foo {
 ...
}
...

Por exemplo, você pode navegar em uma árvore como acima, e então olhar para um dos blobs nela.

$ git cat-file -p 07b1a631
100644 blob b91bba1b   .gitignore
100644 blob cc0956f1   Makefile
040000 tree 92e1ca7e   src
100644 blob cae391ff   Readme.txt

$ git cat-file -p cae391ff
Welcome to my project! This is the readmefile
...

Criando novos commits #

O comando git commit faz algumas coisas:

  1. Crie blobs e trees para representar o diretório do seu projeto - armazenado em .git/objects
  2. Cria um novo objeto commit com suas informações de autor, mensagem de commit e a raiz tree da etapa 1 - também armazenada em .git/objects
  3. Atualiza a referência HEAD em .git/HEAD para o hash do commit recém-criado

Isso resulta em um novo instantâneo do seu projeto sendo adicionado ao git que está conectado ao estado anterior.

Movendo a CABEÇA #

Quando você executa o git checkout em um commit (especificado por hash ou ref) você está dizendo ao git para fazer com que seu diretório de trabalho se pareça com o que era quando o snapshot foi tirado.

  1. Atualize os arquivos no diretório de trabalho para corresponder à árvore dentro do commit
  2. Atualize HEAD para apontar para o hash ou referência especificado

Movendo refs ao redor #

Executar git reset --hard move refs para o hash/ref especificado.

Movendo MyBranch para b8dc53:

$ git checkout MyBranch      # moves HEAD to MyBranch
$ git reset --hard b8dc53    # makes MyBranch point to b8dc53

Criando novas referências #

Executar git checkout -b <refname> criará um novo ref que aponta para o commit atual.

$ cat .git/head
1f324a

$ git checkout -b TestBranch

$ cat .git/refs/heads/TestBranch
1f324a
