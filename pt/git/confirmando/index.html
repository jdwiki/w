Os commits com o Git fornecem responsabilidade atribuindo aos autores alterações no código. O Git oferece vários recursos para a especificidade e segurança dos commits. Este tópico explica e demonstra as práticas e procedimentos adequados para se comprometer com o Git.

Sintaxe #

  • git commit [sinalizadores]

Parâmetros #

Parâmetro              Detalhes
–mensagem, -m Mensagem para incluir no commit. Especificar este parâmetro ignora o comportamento normal do Git de abrir um editor.
–alterar Especifique que as mudanças atualmente preparadas devem ser adicionadas (alteradas) ao commit anterior. Cuidado, isso pode reescrever a história!
–no-edit Use a mensagem de confirmação selecionada sem iniciar um editor. Por exemplo, git commit --amend --no-edit altera um commit sem alterar sua mensagem de commit.
–todos, -a Confirme todas as alterações, incluindo as que ainda não foram testadas.
–data Defina manualmente a data que será associada ao commit.
–somente Confirme apenas os caminhos especificados. Isso não comprometerá o que você encenou atualmente, a menos que seja instruído a fazê-lo.
–patch, -p Use a interface interativa de seleção de patches para escolher quais alterações serão confirmadas.
–ajuda Exibe a página man para git commit
-S[keyid], -S –gpg-sign[=keyid], -S –no-gpg-sign Assinar commit, GPG-sign commit, countermand commit.gpgSign variável de configuração
-n, –no-verify Esta opção ignora os ganchos pre-commit e commit-msg. Consulte também https://www.wikiod.com/pt/git/ganchos

Boas mensagens de commit #

É importante para alguém percorrendo o git log entender facilmente o que cada commit era. Boas mensagens de confirmação geralmente incluem um número de uma tarefa ou um problema em um rastreador e uma descrição concisa do que foi feito e por quê, e às vezes também como foi feito.

Mensagens melhores podem ser assim:

TASK-123: Implement login through OAuth
TASK-124: Add auto minification of JS/CSS files
TASK-125: Fix minifier error when name > 200 chars

Considerando que as seguintes mensagens não seriam tão úteis:

fix                         // What has been fixed?
just a bit of a change      // What has changed?
TASK-371                    // No description at all, reader will need to look at the tracker themselves for an explanation
Implemented IFoo in IBar    // Why it was needed?

Uma maneira de testar se uma mensagem de commit está escrita no modo correto é substituir o espaço em branco pela mensagem e ver se faz sentido:

Se eu adicionar este commit, irei ___ ao meu repositório.

As sete regras de uma ótima mensagem de commit do git #

  1. Separe a linha de assunto do corpo com uma linha em branco
  2. Limite a linha de assunto a 50 caracteres
  3. Coloque a linha de assunto em maiúscula
  4. Não termine a linha de assunto com um ponto
  5. Use o [humor imperativo] (https://en.wikipedia.org/wiki/Imperative_mood) na linha de assunto
  6. Enrole manualmente cada linha do corpo em 72 caracteres
  7. Use o corpo para explicar o que e por que em vez de como

7 regras do blog de Chris Beam.

Alterando um commit #

Se seu último commit ainda não foi publicado (não foi enviado para um repositório upstream), então você pode alterar seu commit.

git commit --amend

Isso colocará as alterações atualmente encenadas no commit anterior.

Observação: Isso também pode ser usado para editar uma mensagem de confirmação incorreta. Ele abrirá o editor padrão (geralmente vi / vim / emacs) e permitirá que você altere a mensagem anterior.

Para especificar a mensagem de confirmação inline:

git commit --amend -m "New commit message"

Ou para usar a mensagem de confirmação anterior sem alterá-la:

git commit --amend --no-edit

A alteração atualiza a data de confirmação, mas deixa a data do autor intocada. Você pode dizer ao git para atualizar as informações.

git commit --amend --reset-author

Você também pode alterar o autor do commit com:

git commit --amend --author "New Author <email@address.com>"

Nota: Esteja ciente de que a alteração do commit mais recente o substitui inteiramente e o commit anterior é removido do histórico do branch. Isso deve ser lembrado ao trabalhar com repositórios públicos e em branches com outros colaboradores.

Isso significa que se o commit anterior já tiver sido enviado, depois de corrigi-lo você terá que push --force.

Confirmando sem abrir um editor #

O Git geralmente abre um editor (como vim ou emacs) quando você executa o git commit. Passe a opção -m para especificar uma mensagem da linha de comando:

git commit -m "Commit message here"

Sua mensagem de confirmação pode passar por várias linhas:

git commit -m "Commit 'subject line' message here

More detailed description follows here (after a blank line)."

Alternativamente, você pode passar vários argumentos -m:

git commit -m "Commit summary" -m "More detailed description follows here"

Consulte Como escrever uma mensagem de confirmação do Git.

Guia de estilo de mensagem de confirmação do Git Udacity

Preparar e confirmar alterações #

O básico #

Depois de fazer alterações em seu código-fonte, você deve preparar essas alterações com o Git antes de enviá-las.

Por exemplo, se você alterar README.md e program.py:

git add README.md program.py

Isso diz ao git que você deseja adicionar os arquivos ao próximo commit que você fizer.

Em seguida, confirme suas alterações com

git commit

Observe que isso abrirá um editor de texto, que é frequentemente [vim][2]. Se você não estiver familiarizado com o vim, você pode querer saber que você pode pressionar i para entrar no modo inserir, escrever sua mensagem de commit, então pressionar Esc e :wq para salvar e sair. Para evitar abrir o editor de texto, basta incluir o sinalizador -m com sua mensagem

git commit -m "Commit message here"

As mensagens de commit geralmente seguem algumas regras de formatação específicas, veja [Good commit messages][3] para mais informações.

Atalhos #

Se você alterou muitos arquivos no diretório, em vez de listar cada um deles, você pode usar:

git add --all        # equivalent to "git add -a"

Ou para adicionar todas as alterações, não incluindo os arquivos que foram excluídos, do diretório e subdiretórios de nível superior:

git add .

Ou para adicionar apenas arquivos atualmente rastreados (“atualização”):

git add -u

Se desejar, revise as alterações em etapas:

git status           # display a list of changed files
git diff --cached    # shows staged changes inside staged files

Por fim, confirme as alterações:

git commit -m "Commit message here"

Alternativamente, se você modificou apenas arquivos existentes ou excluiu arquivos e não criou nenhum novo, você pode combinar as ações de git add e git commit em um único comando:

git commit -am "Commit message here"

Observe que isso irá preparar todos os arquivos modificados da mesma forma que git add --all.

Dados sensíveis #

Você nunca deve confirmar nenhum dado sensível, como senhas ou mesmo chaves privadas. Se esse caso acontecer e as alterações já forem enviadas para um servidor central, considere todos os dados confidenciais como comprometidos. Caso contrário, é possível remover esses dados posteriormente. Uma solução rápida e fácil é o uso do “BFG Repo-Cleaner”: https://rtyley.github.io/bfg-repo-cleaner/.

O comando bfg --replace-text passwords.txt my-repo.git lê as senhas do arquivo passwords.txt e as substitui por ***REMOVED***. Esta operação considera todos os commits anteriores de todo o repositório.

Confirmando alterações diretamente #

Normalmente, você tem que usar git add ou git rm para adicionar alterações ao índice antes de poder git commit. Passe a opção -a ou --all para adicionar automaticamente todas as alterações (para arquivos rastreados) ao índice, incluindo remoções:

git commit -a

Se você quiser também adicionar uma mensagem de commit, você faria:

git commit -a -m "your commit message goes here"

Além disso, você pode juntar duas bandeiras:

git commit -am "your commit message goes here"

Você não precisa necessariamente confirmar todos os arquivos de uma vez. Omita o sinalizador -a ou --all e especifique qual arquivo você deseja confirmar diretamente:

git commit path/to/a/file -m "your commit message goes here"

Para confirmar diretamente mais de um arquivo específico, você também pode especificar um ou vários arquivos, diretórios e padrões:

git commit path/to/a/file path/to/a/folder/* path/to/b/file -m "your commit message goes here"

Criando um commit vazio #

De um modo geral, commits vazios (ou commits com estado idêntico ao pai) são um erro.

No entanto, ao testar ganchos de compilação, sistemas CI e outros sistemas que acionam um commit, é útil poder criar commits facilmente sem ter que editar/tocar em um arquivo fictício.

O commit --allow-empty ignorará a verificação.

git commit -m "Este é um commit em branco" --allow-empty

Comprometendo-se em nome de outra pessoa #

Se outra pessoa escreveu o código que você está enviando, você pode dar crédito a ela com a opção --author:

git commit -m "msg" --author "John Smith <johnsmith@example.com>"

Você também pode fornecer um padrão, que o Git usará para pesquisar autores anteriores:

git commit -m "msg" --author "John"

Neste caso, as informações do autor do commit mais recente com um autor contendo “John” serão usadas.

Selecionando quais linhas devem ser preparadas para confirmação #

Suponha que você tenha muitas alterações em um ou mais arquivos, mas de cada arquivo você deseja confirmar apenas algumas das alterações, você pode selecionar as alterações desejadas usando:

git add -p

ou

git add -p [file]

Cada uma de suas alterações será exibida individualmente e, para cada alteração, você será solicitado a escolher uma das seguintes opções:

y - Yes, add this hunk

n - No, don’t add this hunk

d - No, don’t add this hunk, or any other remaining hunks for this file.
    Useful if you’ve already added what you want to, and want to skip over the rest.

s - Split the hunk into smaller hunks, if possible

e - Manually edit the hunk.  This is probably the most powerful option.
    It will open the hunk in a text editor and you can edit it as needed.

Isso irá preparar as partes dos arquivos que você escolher. Em seguida, você pode confirmar todas as alterações encenadas assim:

git commit -m 'Commit Message'

As alterações que não foram preparadas ou confirmadas ainda aparecerão em seus arquivos de trabalho e podem ser confirmadas posteriormente, se necessário. Ou se as alterações restantes forem indesejadas, elas podem ser descartadas com:

git reset --hard

Além de dividir uma grande mudança em commits menores, essa abordagem também é útil para revisar o que você está prestes a fazer. Ao confirmar individualmente cada alteração, você tem a oportunidade de verificar o que escreveu e pode evitar a preparação acidental de código indesejado, como instruções println/logging.

Confirmando alterações em arquivos específicos #

Você pode confirmar as alterações feitas em arquivos específicos e pular a preparação deles usando git add:

git commit file1.c file2.h

Ou você pode primeiro preparar os arquivos:

git add file1.c file2.h

e cometa-os mais tarde:

git commit

Compromissos de assinatura de GPG #

  1. Determine seu ID de chave

     gpg --list-secret-keys --keyid-format LONG

 /Users/davidcondrey/.gnupg/secring.gpg
 --------------------------------------
 sec   2048R/YOUR-16-DIGIT-KEY-ID YYYY-MM-DD [expires: YYYY-MM-DD]

    Your ID is a alphanumeric 16-digit code following the first forward-slash.

  2. Defina seu ID de chave em sua configuração do git

     git config --global user.signingkey YOUR-16-DIGIT-KEY-ID

  3. A partir da versão 1.7.9, o git commit aceita a opção -S para anexar uma assinatura aos seus commits. Usar esta opção solicitará sua senha GPG e adicionará sua assinatura ao log de confirmação.

     git commit -S -m "Your commit message"

Comprometendo-se em uma data específica #

git commit -m 'Fix UI bug' --date 2016-07-01

O parâmetro --date define a data do autor. Esta data aparecerá na saída padrão do git log, por exemplo.

Para forçar a data de confirmação também:

GIT_COMMITTER_DATE=2016-07-01 git commit -m 'Fix UI bug' --date 2016-07-01

O parâmetro date aceita os formatos flexíveis suportados pela data GNU, por exemplo:

git commit -m 'Fix UI bug' --date yesterday
git commit -m 'Fix UI bug' --date '3 days ago'
git commit -m 'Fix UI bug' --date '3 hours ago'

Quando a data não especifica a hora, a hora atual será usada e somente a data será substituída.

Alterando o horário de um commit #

Você pode alterar a hora de um commit usando

git commit --amend --date="Thu Jul 28 11:30 2016 -0400"

ou mesmo

git commit --amend --date="now"

Alterando o autor de um commit #

Se você fizer um commit como o autor errado, você pode alterá-lo e, em seguida, corrigir

git config user.name "Full Name"
git config user.email "email@example.com"

git commit --amend --reset-author
