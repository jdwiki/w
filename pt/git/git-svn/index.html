Clonagem de repositórios SVN realmente grandes

Se o histórico do repositório SVN for realmente muito grande, essa operação pode levar horas, pois o git-svn precisa reconstruir o histórico completo do repositório SVN. Felizmente você só precisa clonar o repositório SVN uma vez; como em qualquer outro repositório git, você pode simplesmente copiar a pasta repo para outros colaboradores. Copiar a pasta para vários computadores será mais rápido do que clonar grandes repositórios SVN do zero.

Sobre commits e SHA1

Seus commits git locais serão reescritos ao usar o comando git svn dcommit. Este comando adicionará um texto à mensagem do git commit referenciando a revisão SVN criada no servidor SVN, o que é muito útil. No entanto, adicionar um novo texto requer a modificação da mensagem de um commit existente, o que não pode ser feito: git commits são imutáveis. A solução é criar um novo commit com o mesmo conteúdo e a nova mensagem, mas tecnicamente é um novo commit de qualquer maneira (ou seja, o SHA1 do commit do git mudará)

Como os git commits criados para git-svn são locais, os ids SHA1 para git commits são diferentes entre cada repositório git! Isso significa que você não pode usar um SHA1 para referenciar um commit de outra pessoa porque o mesmo commit terá um SHA1 diferente em cada repositório git local. Você precisa confiar no número de revisão do svn anexado à mensagem do commit quando você enviar para o servidor SVN se você quiser referenciar um commit entre diferentes cópias do repositório.

Você pode usar o SHA1 para operações locais (mostrar/diferenciar um commit específico, cherry-picks e resets, etc)

Solução de problemas #

O comando git svn rebase emite um erro de incompatibilidade de soma de verificação

O comando git svn rebase lança um erro semelhante a este:

  Checksum mismatch: <path_to_file> <some_kind_of_sha1>
  expected: <checksum_number_1>
    got: <checksum_number_2>

A solução para este problema é resetar o svn para a revisão quando o arquivo problemático foi modificado pela última vez, e fazer um git svn fetch para que o histórico do SVN seja restaurado. Os comandos para realizar o reset do SVN são:

  • git log -1 – <path_to_file> (copie o número de revisão do SVN que aparece na mensagem do commit)
  • git svn reset <revision_number>
  • git svn buscar

Você deve ser capaz de enviar/puxar dados do SVN novamente

Arquivo não encontrado no commit Quando você tenta buscar ou extrair do SVN, você recebe um erro semelhante a este

<file_path> was not found in commit <hash>

Isso significa que uma revisão no SVN está tentando modificar um arquivo que por algum motivo não existe em sua cópia local. A melhor maneira de se livrar desse erro é forçar uma busca ignorando o caminho desse arquivo e ele será atualizado para seu status na última revisão do SVN:

  • git svn fetch --ignore-paths <file_path>

Clonando o repositório SVN #

Você precisa criar uma nova cópia local do repositório com o comando

git svn clone SVN_REPO_ROOT_URL [DEST_FOLDER_PATH] -T TRUNK_REPO_PATH -t TAGS_REPO_PATH -b BRANCHES_REPO_PATH

Se o seu repositório SVN segue o layout padrão (tronco, ramificações, pastas de tags), você pode economizar digitação:

git svn clone -s SVN_REPO_ROOT_URL [DEST_FOLDER_PATH]

O git svn clone verifica cada revisão do SVN, uma por uma, e faz um git commit em seu repositório local para recriar o histórico. Se o repositório SVN tiver muitos commits, isso demorará um pouco.

Quando o comando terminar, você terá um repositório git completo com um branch local chamado master que rastreia o branch trunk no repositório SVN.

Empurrando alterações locais para SVN #

O comando

git svn dcommit

irá criar uma revisão SVN para cada um dos seus commits git locais. Assim como no SVN, seu histórico git local deve estar sincronizado com as últimas alterações no repositório SVN, então se o comando falhar, tente executar um git svn rebase primeiro.

Trabalhando localmente #

Basta usar seu repositório git local como um repositório git normal, com os comandos git normais:

  • git add FILE e git checkout -- FILE Para preparar/desmontar um arquivo
  • git commit Para salvar suas alterações. Esses commits serão locais e não serão “enviados” para o repositório SVN, assim como em um repositório git normal
  • git stash e git stash pop Permite usar stashes
  • git reset HEAD --hard Reverta todas as suas alterações locais
  • git log Acesse todo o histórico do repositório
  • git rebase -i para que você possa reescrever seu histórico local livremente
  • git branch e git checkout para criar branches locais

Como a documentação do git-svn afirma “O Subversion é um sistema muito menos sofisticado que o Git”, então você não pode usar todo o poder do git sem bagunçar o histórico no servidor Subversion. Felizmente as regras são muito simples: Mantenha o histórico linear

Isso significa que você pode fazer praticamente qualquer operação git: criar branches, remover/reordenar/comprimir commits, mover o histórico, deletar commits, etc. Qualquer coisa exceto merges. Se você precisar reintegrar o histórico de branches locais, use git rebase.

Quando você executa uma mesclagem, uma confirmação de mesclagem é criada. A particularidade dos commits de mesclagem é que eles têm dois pais, e isso torna o histórico não linear. O histórico não linear confundirá o SVN no caso de você “enviar” um commit de mesclagem para o repositório.

No entanto, não se preocupe: você não quebrará nada se “enviar” um commit do git merge para o SVN. Se você fizer isso, quando o commit do git merge for enviado para o servidor svn, ele conterá todas as alterações de todos os commits desse merge, então você perderá o histórico desses commits, mas não as alterações em seu código.

Obtendo as últimas alterações do SVN #

O equivalente a git pull é o comando

git svn rebase

Isso recupera todas as alterações do repositório SVN e as aplica sobre seus commits locais em seu branch atual.

Você também pode usar o comando

git svn fetch

para recuperar as alterações do repositório SVN e trazê-las para sua máquina local, mas sem aplicá-las à sua ramificação local.

Manipulando pastas vazias #

O git não reconhece o conceito de pastas, apenas funciona com arquivos e seus caminhos de arquivo. Isso significa que o git não rastreia pastas vazias. SVN, no entanto, faz. Usar git-svn significa que, por padrão, qualquer alteração que você fizer envolvendo pastas vazias com git não será propagada para SVN.

Usar o sinalizador --rmdir ao emitir um comentário corrige esse problema e remove uma pasta vazia no SVN se você excluir localmente o último arquivo dentro dela:

git svn dcommit --rmdir

Infelizmente não remove pastas vazias existentes: você precisa fazer isso manualmente.

Para evitar adicionar o sinalizador toda vez que você fizer um dcommit, ou para jogar pelo seguro se estiver usando uma ferramenta git GUI (como SourceTree), você pode definir esse comportamento como padrão com o comando:

git config --global svn.rmdir true

Isso altera seu arquivo .gitconfig e adiciona estas linhas:

[svn]
rmdir = true

Para remover todos os arquivos e pastas não rastreados que devem ser mantidos vazios para SVN, use o comando git:

git clean -fd

Observe: o comando anterior removerá todos os arquivos não rastreados e pastas vazias, mesmo aquelas que deveriam ser rastreadas pelo SVN! Se você precisar gerar novamente as pastas vazias rastreadas pelo SVN use o comando

git svn mkdirs

Na prática, isso significa que, se você deseja limpar seu espaço de trabalho de arquivos e pastas não rastreados, você deve sempre usar os dois comandos para recriar as pastas vazias rastreadas pelo SVN:

git clean -fd && git svn mkdirs
