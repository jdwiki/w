O objetivo deste tópico é documentar as práticas recomendadas ao fazer perguntas relacionadas ao Apache Spark.

Detalhes do ambiente: #

Ao fazer perguntas relacionadas ao Apache Spark, inclua as seguintes informações

  • Versão do Apache Spark usada pelo cliente e implantação do Spark, se aplicável. Para questões relacionadas a APIs maiores (1.6, 2.0, 2.1 etc.) normalmente é suficiente, para questões relacionadas a possíveis bugs sempre use informações de versão completa.
  • Versão Scala usada para construir binários do Spark.
  • Versão JDK (java -version).
  • Se você usa o idioma convidado (Python, R), forneça informações sobre a versão do idioma. Em Python, use tags: [tag:python-2.x], [tag:python-3.x] ou mais específicas para distinguir entre variantes de linguagem.
  • Definição de build (build.sbt, pom.xml) se aplicável ou versões de dependência externa (Python, R) quando aplicável.
  • Gerenciador de cluster (local[n], Spark autônomo, Yarn, Mesos), modo (client, cluster) e outras opções de envio, se aplicável.

Dados e código de exemplo #

Dados de exemplo #

Por favor, tente fornecer um exemplo mínimo de dados de entrada em um formato que possa ser usado diretamente pelas respostas sem análise tediosa e demorada, por exemplo, arquivo de entrada ou coleção local com todo o código necessário para criar estruturas de dados distribuídas.

Quando aplicável, sempre inclua informações de tipo:

  • Na API baseada em RDD, use anotações de tipo quando necessário.
  • Na API baseada em DataFrame, forneça informações de esquema como um StrucType ou saída de Dataset.printSchema.

A saída de Dataset.show ou print pode parecer boa, mas não nos diz nada sobre os tipos subjacentes.

Se um problema específico ocorrer apenas em escala, use geradores de dados aleatórios (o Spark fornece alguns utilitários úteis em org.apache.spark.mllib.random.RandomRDDs e org.apache.spark.graphx.util.GraphGenerators

Código #

Use anotações de tipo quando possível. Embora seu compilador possa acompanhar facilmente os tipos, não é tão fácil para meros mortais. Por exemplo:

val lines: RDD[String] = rdd.map(someFunction)

ou

def f(x: String): Int = ???

são melhores que:

val lines = rdd.map(someFunction)

e

def f(x: String) = ???

respectivamente.

Informações de diagnóstico #

Perguntas de depuração. #

Quando a questão estiver relacionada à depuração de uma exceção específica, sempre forneça um rastreamento relevante. Embora seja aconselhável remover saídas duplicadas (de diferentes executores ou tentativas), não corte os tracebacks para uma única linha ou classe de exceção apenas.

Perguntas de desempenho. #

Dependendo do contexto, tente fornecer detalhes como:

  • RDD.debugString / Dataset.explain.
  • Saída da interface do usuário do Spark com diagrama DAG, se aplicável em um caso específico.
  • Mensagens de log relevantes.
  • Informações de diagnóstico coletadas por ferramentas externas (Ganglia, VisualVM).

Antes de perguntar #

  • Pesquisa Stack Overflow para perguntas duplicadas. Há uma classe comum de problemas que já foram amplamente documentados.
