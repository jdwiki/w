Introdução #

Protótipo:

agregado(zeroValue, seqOp, combOp)

Descrição:

aggregate() permite que você pegue um RDD e gere um único valor que é de um tipo diferente do que foi armazenado no RDD original.

Parâmetros:

  1. zeroValue: O valor de inicialização, para o seu resultado, no format.
  2. seqOp: A operação que você deseja aplicar aos registros RDD. Funciona uma vez por every record in a partition.
  3. combOp: Define como os objetos resultantes (um para cada partição), gets combined.

Exemplo:

Calcular a soma de uma lista e o comprimento dessa lista. Retorna o resultado em um par de (soma, comprimento).

Em um shell Spark, crie uma lista com 4 elementos, com 2 partições:

listRDD = sc.parallelize([1,2,3,4], 2)

Em seguida, defina seqOp:

seqOp = (lambda local_result, list_element: (local_result[0] + list_element, local_result[1] + 1) )

Em seguida, defina combOp:

combOp = (lambda some_local_result, another_local_result: (some_local_result[0] + another_local_result[0], some_local_result[1] + another_local_result[1]) )

Em seguida, agregado:

listRDD.aggregate( (0, 0), seqOp, combOp)
Out[8]: (10, 4)

A primeira partição tem a sublista [1, 2]. Isso aplica o seqOp a cada elemento dessa lista, que produz um resultado local - Um par de (sum, length) que refletirá o resultado localmente, apenas nessa primeira partição.

local_result é inicializado para o parâmetro zeroValue que aggregate() foi fornecido. Por exemplo, (0, 0) e list_element é o primeiro elemento da lista:

0 + 1 = 1
0 + 1 = 1

O resultado local é (1, 1), o que significa que a soma é 1 e o comprimento 1 para a 1ª partição após processar somente o primeiro elemento. local_result é atualizado de (0, 0), para (1, 1).

1 + 2 = 3
1 + 1 = 2

O resultado local agora é (3, 2), que será o resultado final da 1ª partição, pois não há outros elementos na sublista da 1ª partição. Fazendo o mesmo para a 2ª partição retorna (7, 2).

Aplique combOp a cada resultado local para formar o resultado final global:

(3,2) + (7,2) = (10, 4)

Exemplo descrito em ‘figura’:

[1, 2]                  [3, 4]

0 + 1 = 1               0 + 3 = 3
0 + 1 = 1               0 + 1 = 1

1 + 2 = 3               3 + 4 = 7

Transformação x Ação #

O Spark usa avaliação lenta; isso significa que não fará nenhum trabalho, a menos que realmente precise. Essa abordagem nos permite evitar o uso desnecessário de memória, tornando-nos capazes de trabalhar com big data.

Uma transformação é avaliada com preguiça e o trabalho real acontece, quando ocorre uma ação.

Exemplo:

In [1]: lines = sc.textFile(file)        // will run instantly, regardless file's size
In [2]: errors = lines.filter(lambda line: line.startsWith("error")) // run instantly
In [3]: errorCount = errors.count()    // an action occurred, let the party start!
Out[3]: 0                              // no line with 'error', in this example

Então, em [1] nós dissemos ao Spark para ler um arquivo em um RDD, chamado lines. O Spark nos ouviu e nos disse: “Sim, eu farei fazer isso”, mas na verdade ele ainda não leu o arquivo.

Em 2, estamos filtrando as linhas do arquivo, assumindo que seu conteúdo contém linhas com erros que são marcados com um erro em seu início. Então, dizemos ao Spark para criar um novo RDD, chamado errors, que terá os elementos do RDD lines, que tiveram a palavra error em seu início.

Agora em [3], pedimos ao Spark para contar os erros, ou seja, contar o número de elementos que o RDD chamado erros tem. count() é uma ação, que não deixa escolha ao Spark, mas sim fazer a operação de fato, para que ele possa encontrar o resultado de count(), que será um inteiro.

Como resultado, quando [3] for alcançado, [1] e [2] serão realmente executados, ou seja, quando chegarmos a [3], então e somente então:

  1. o arquivo será lido em textFile() (por causa de [1])

  2. lines será filter()’ (por causa de [2])

  3. count() será executado, por causa de [3]

Dica de depuração: como o Spark não fará nenhum trabalho real até que [3] seja alcançado, é importante entender que, se existir um erro em [1] e/ou [2 ], ele não aparecerá, até que a ação em [3] dispare o Spark para fazer o trabalho real. Por exemplo, se seus dados no arquivo não suportarem o startsWith() que usei, então [2] será aceito corretamente pelo Spark e não gerará nenhum erro, mas quando [3] é enviado, e o Spark realmente avalia tanto [1] quanto [2], então e somente então ele entenderá que algo não está correto com [2] e produzirá um erro descritivo.

Como resultado, um erro pode ser acionado quando [3] é executado, mas isso não significa que o erro deve estar na instrução de [3]!

Observe que nem lines nem errors serão armazenados na memória após [3]. Eles continuarão a existir apenas como um conjunto de instruções de processamento. Se houver várias ações executadas em qualquer um desses RDDs, o Spark lerá e filtrará os dados várias vezes. Para evitar a duplicação de operações ao executar várias ações em um único RDD, geralmente é útil armazenar dados na memória usando cache.

Você pode ver mais transformações/ações em documentos do Spark.

Verifique a versão do Spark #

Em spark-shell:

sc.version

Geralmente em um programa:

SparkContext.version

Usando spark-submit:

 spark-submit --version
