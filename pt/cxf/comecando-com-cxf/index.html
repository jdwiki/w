Webclient básico com provedor #

Para começar precisamos de uma fábrica que produza WebClients.

public class ClientFactory {
    private Map<String, WebClient> cache = new HashMap<>();

    public enum RESTClient {
        PORTAL;
    }

    public WebClient fetchRestClient(RESTClient restClient) {

        if (this.cache.containsKey(restClient)) {
            return WebClient.fromClient(this.cache.get(rc));
        }

        if (RESTClient.enum.equals(rc)) {

            List<Object> providers = new ArrayList<Object>();
            providers.add(new GsonMessageBodyProvider());

            WebClient webClient = WebClient.create("https://blah.com", providers);

            HTTPConduit conduit = WebClient.getConfig(webClient).getHttpConduit();
            conduit.getClient().setReceiveTimeout(recieveTimeout);
            conduit.getClient().setConnectionTimeout(connectionTimout);

            this.cache.put(RESTClient.CAT_DEVELOPER_PORTAL.name(), webClient);
            return WebClient.fromClient(webClient);// thread safe
        }
    }
}
  • Primeiro, criamos uma lista de provedores (chegaremos a eles mais tarde)
  • Em seguida, criamos um novo webclient usando a fábrica estática “create()”. Aqui podemos adicionar algumas outras coisas, como credenciais de autenticação básica e segurança de thread. Por enquanto é só usar este.
  • Em seguida, retiramos o HTTPConduit e definimos os tempos limite. O CXF vem com clientes de classe base Java, mas você pode usar Glassfish ou outros.
  • Finalmente, armazenamos em cache o WebClient. Isso é importante, pois o código até agora é caro para criar. A próxima linha mostra como torná-lo thread-safe. Observe que também é assim que extraímos o código do cache. Essencialmente, estamos fazendo um modelo da chamada REST e clonando-o sempre que precisarmos.
  • Observe o que NÃO está aqui: Não adicionamos nenhum parâmetro ou URL. Estes podem ser adicionados aqui, mas isso faria um endpoint específico e queremos um genérico. Também não há cabeçalhos adicionados à solicitação. Eles não passam do “clone”, então devem ser adicionados mais tarde.

Agora temos um WebClient que está pronto para ser usado. Vamos configurar o resto da chamada.

public Person fetchPerson(Long id) {
    long timer t = System.currentTimeMillis();
    Person person = null;
    try {
        wc = this.factory.findWebClient(RESTClient.PORTAL);
        wc.header(AUTH_HEADER, SUBSCRIPTION_KEY);
        wc.header(HttpHeaders.ACCEPT, "application/person-v1+json");

        wc.path("person").path("base");
        wc.query("id", String.valueOf(id));

        person = wc.get(Person.class);
    }
    catch (WebApplicationException wae) {
        // we wanna skip these. They will show up in the "finally" logs.
    }
    catch (Exception e) {
        log.error(MessageFormat.format("Error fetching Person: id:{0} ", id), e);
    }
    finally {
        log.info("GET HTTP:{} - Time:[{}ms] - URL:{} - Content-Type:{}", wc.getResponse().getStatus(), (System.currentTimeMillis() - timer), wc.getCurrentURI(), wc.getResponse().getMetadata().get("content-type"));
        wc.close();
    }
    return p;
}
  • Dentro do “try” pegamos um WebClient de fábrica. Este é um novo “gelado”.
  • Em seguida, definimos alguns cabeçalhos. Aqui adicionamos algum tipo de cabeçalho Auth e, em seguida, um cabeçalho de aceitação. Observe que temos um cabeçalho de aceitação personalizado.
  • Adicionar o caminho e as strings de consulta vem a seguir. Tenha em mente que não há ordem para essas etapas.
  • Finalmente fazemos o “get”. Existem várias maneiras de fazer isso, é claro. Aqui temos o CXF fazendo o mapeamento JSON para nós. Feito desta forma temos que lidar com o WebApplicationExceptions. Portanto, se obtivermos um 404, o CXF lançará uma exceção. Observe aqui que eu como essas exceções porque eu apenas registro a resposta no arquivo finally. No entanto, quero obter qualquer OUTRA exceção, pois elas podem ser importantes.
  • Se você não gostar dessa troca de manipulação de exceção, você pode obter o objeto Response de volta do “get”. Este objeto contém a entidade e o código de status HTTP. Ele NUNCA lançará um WebApplicationException. A única desvantagem é que não faz o mapeamento JSON para você.
  • Finalmente, na cláusula “finally” temos um “wc.close()”. Se você obtiver o objeto da cláusula get, não precisará fazer isso. Algo pode dar errado, por isso é um bom à prova de falhas.

Então, e aquele cabeçalho “aceitar”: application/person-v1+json Como o CXF saberá como analisá-lo? O CXF vem com alguns analisadores JSON integrados, mas eu queria usar o Gson. Muitas das outras implementações de analisadores Json precisam de algum tipo de anotação, mas não Gson. É estúpido fácil de usar.

public class GsonMessageBodyProvider<T> implements MessageBodyReader<T>, MessageBodyWriter<T> {

    private Gson gson = new GsonBuilder().create();
    
    @Override
    public boolean isReadable(Class<?> type, Type genericType, Annotation[] annotations, MediaType mediaType) {
        return StringUtils.endsWithIgnoreCase(mediaType.getSubtype(), "json");
    }

    @Override
    public T readFrom(Class<T> type, Type genericType, Annotation[] annotations, MediaType mediaType, MultivaluedMap<String, String> httpHeaders, InputStream entityStream) throws IOException, WebApplicationException {
        try {
            return gson.fromJson(new BufferedReader(new InputStreamReader(entityStream, "UTF-8")), type);
        }
        catch (Exception e) {
            throw new IOException("Trouble reading into:" + type.getName(), e);
        }
    }

    @Override
    public boolean isWriteable(Class<?> type, Type genericType, Annotation[] annotations, MediaType mediaType) {
        return StringUtils.containsIgnoreCase(mediaType.getSubtype(), "json");
    }

    @Override
    public long getSize(T t, Class<?> type, Type genericType, Annotation[] annotations, MediaType mediaType) {
        return 0;
    }

    @Override
    public void writeTo(T t, Class<?> type, Type genericType, Annotation[] annotations, MediaType mediaType, MultivaluedMap<String, Object> httpHeaders, OutputStream entityStream) throws IOException, WebApplicationException {
        try {
            JsonWriter writer = new JsonWriter(new OutputStreamWriter(entityStream, "UTF-8"));
            writer.setIndent("  ");
            gson.toJson(t, type, writer);
            writer.close();
        }
        catch (Exception e) {
            throw new IOException("Trouble marshalling:" + type.getName(), e);
        }
    }
}

O código é direto. Você implementa duas interfaces: MessageBodyReader e MessageBodyWriter (ou apenas uma) e depois adiciona aos “provedores” ao criar o WebClient. CXF descobre a partir daí. Uma opção é retornar “true” nos métodos “isReadable()” “isWriteable()”. Isso garantirá que o CXF use essa classe em vez de todas as incorporadas. Os provedores adicionados serão verificados primeiro.

Configurando o CXF para JAX-RS #

Os jars para CXF JAX-RS são encontrados no Maven:

<!-- https://mvnrepository.com/artifact/org.apache.cxf/cxf-rt-rs-client -->
<dependency>
    <groupId>org.apache.cxf</groupId>
    <artifactId>cxf-rt-rs-client</artifactId>
    <version>3.1.10</version>
</dependency>

Esses jars são tudo o que você precisa para executá-lo:

cxf-rt-rs-client-3.1.10.jar
cxf-rt-transports-http-3.1.10.jar
cxf-core-3.1.10.jar
woodstox-core-asl-4.4.1.jar
stax2-api-3.1.4.jar
xmlschema-core-2.2.1.jar
cxf-rt-frontend-jaxrs-3.1.10.jar
javax.ws.rs-api-2.0.1.jar
javax.annotation-api-1.2.jar

Filtros de cliente #

Um bom motivo para usar Filtros é para registrar. Usando esta técnica, uma chamada REST pode ser registrada e cronometrada facilmente.

public class RestLogger implements ClientRequestFilter, ClientResponseFilter {
    private static final Logger log = LoggerFactory.getLogger(RestLogger.class);

    // Used for timing this call.
    private static final ThreadLocal<Long> startTime = new ThreadLocal<Long>();
    private boolean logRequestEntity;
    private boolean logResponseEntity;

    private static Gson GSON = new GsonBuilder().create();

    public RestLogger(boolean logRequestEntity, boolean logResponseEntity) {
        this.logRequestEntity = logRequestEntity;
        this.logResponseEntity = logResponseEntity;
    }


    @Override
    public void filter(ClientRequestContext requestContext) throws IOException {
        startTime.set(System.currentTimeMillis());
    }

    @Override
    public void filter(ClientRequestContext requestContext, ClientResponseContext responseContext) throws IOException {
        StringBuilder sb = new StringBuilder();
        sb.append("HTTP:").append(responseContext.getStatus());
        sb.append(" - Time:[").append(System.currentTimeMillis() - startTime.get().longValue()).append("ms]");
        sb.append(" - Path:").append(requestContext.getUri());
        sb.append(" - Content-type:").append(requestContext.getStringHeaders().getFirst(HttpHeaders.CONTENT_TYPE.toString()));
        sb.append(" - Accept:").append(requestContext.getStringHeaders().getFirst(HttpHeaders.ACCEPT.toString()));
        if (logRequestEntity) {
            sb.append(" - RequestBody:").append(requestContext.getEntity() != null ? GSON.toJson(requestContext.getEntity()) : "none");
        }
        if (logResponseEntity) {
            sb.append(" - ResponseBody:").append(this.logResponse(responseContext));
        }
        log.info(sb.toString());
    }

    private String logResponse(ClientResponseContext response) {
        StringBuilder b = new StringBuilder();
        ByteArrayOutputStream out = new ByteArrayOutputStream();
        InputStream in = response.getEntityStream();
        try {
            ReaderWriter.writeTo(in, out);
            byte[] requestEntity = out.toByteArray();
            b.append(new String(requestEntity));
            response.setEntityStream(new ByteArrayInputStream(requestEntity));
        }
        catch (IOException ex) {
            throw new ClientHandlerException(ex);
        }
        return b.toString();
    }
}

Acima, você pode ver que a solicitação é interceptada antes que a resposta seja enviada e um ThreadLocal Long seja definido. Quando a resposta é retornada, podemos registrar a solicitação e a resposta e todos os tipos de dados pertinentes. Claro que isso só funciona para respostas Gson e tal, mas pode ser modificado facilmente. Isso é configurado desta forma:

List<Object> providers = new ArrayList<Object>();
providers.add(new GsonMessageBodyProvider());
providers.add(new RestLogger(true, true)); <------right here!

WebClient webClient = WebClient.create(PORTAL_URL, providers);

O log fornecido deve ser algo assim:

7278 [main] INFO  blah.RestLogger - HTTP:200 - Time:[1391ms] - User:unknown - Path:https://blah.com/tmet/moduleDescriptions/desc?languageCode=en&moduleId=142 - Content-type:null - Accept:application/json - RequestBody:none - ResponseBody:{"languageCode":"EN","moduleId":142,"moduleDescription":"ECAP"}
