Configurando NODE_ENV=“produção” #

As implantações de produção variam de várias maneiras, mas uma convenção padrão ao implantar em produção é definir uma variável de ambiente chamada NODE_ENV e definir seu valor como “production”.

sinalizadores de tempo de execução #

Qualquer código em execução em seu aplicativo (incluindo módulos externos) pode verificar o valor de NODE_ENV:

if(process.env.NODE_ENV === 'production') {
    // We are running in production mode
} else {
   // We are running in development mode
}

Dependências #

Quando a variável de ambiente NODE_ENV é definida como ‘production’, todos os devDependencies em seu arquivo package.json serão completamente ignorados ao executar o npm install. Você também pode impor isso com um sinalizador --production:

npm install --production

Para definir NODE_ENV você pode usar qualquer um desses métodos

método 1: defina NODE_ENV para todos os aplicativos do nó

Janelas :

set NODE_ENV=production

Linux ou outro sistema baseado em unix:

export NODE_ENV=production

Isso define NODE_ENV para a sessão atual do bash, portanto, qualquer aplicativo iniciado após essa instrução terá NODE_ENV definido como produção.

método 2: defina NODE_ENV para o aplicativo atual

NODE_ENV=production node app.js

Isso definirá NODE_ENV apenas para o aplicativo atual. Isso ajuda quando queremos testar nossos aplicativos em ambientes diferentes.

método 3: crie o arquivo .env e use-o

Isso usa a ideia explicada aqui. Consulte este post para uma explicação mais detalhada.

Basicamente você cria um arquivo .env e executa algum script bash para defini-los no ambiente.

Para evitar escrever um script bash, o pacote env-cmd pode ser usado para carregar as variáveis ​​de ambiente definidas no arquivo .env.

env-cmd .env node app.js

método 4: Use o pacote cross-env

Este pacote permite que as variáveis ​​de ambiente sejam definidas de uma maneira para cada plataforma.

Depois de instalá-lo com npm, basta adicioná-lo ao seu script de implantação em package.json da seguinte forma:

"build:deploy": "cross-env NODE_ENV=production webpack"

Gerenciar aplicativo com gerenciador de processos #

É uma boa prática executar aplicativos NodeJS controlados por gerenciadores de processos. O gerenciador de processos ajuda a manter o aplicativo ativo para sempre, reiniciar em caso de falha, recarregar sem tempo de inatividade e simplifica a administração. O mais poderoso deles (como PM2) tem um balanceador de carga integrado. O PM2 também permite que você gerencie o log, o monitoramento e o clustering de aplicativos.

Gerente de processos PM2 #

Instalando o PM2:

npm instala pm2 -g

O processo pode ser iniciado no modo de cluster envolvendo o balanceador de carga integrado para distribuir a carga entre os processos:

pm2 start app.js -i 0 --name "api" (-i é para especificar o número de processos a serem gerados. Se for 0, o número do processo será baseado na contagem de núcleos da CPU)

Apesar de ter vários usuários em produção, é preciso ter um único ponto para PM2. Portanto, o comando pm2 deve ser prefixado com um local (para configuração PM2), caso contrário, ele gerará um novo processo pm2 para cada usuário com configuração no respectivo diretório inicial. E será inconsistente.

Uso: PM2_HOME=/etc/.pm2 pm2 start app.js

Implantação usando gerenciador de processos #

O gerenciador de processos geralmente é usado na produção para implantar um aplicativo nodejs. As principais funções de um gerenciador de processos são reiniciar o servidor se ele travar, verificar o consumo de recursos, melhorar o desempenho do tempo de execução, monitorar etc.

Alguns dos gerenciadores de processos populares feitos pela comunidade de nós são forever, pm2, etc.

Para todo sempre #

forever é uma ferramenta de interface de linha de comando para garantir que um determinado script seja executado continuamente. A interface simples do forever o torna ideal para executar implementações menores de aplicativos e scripts Node.js.

forever monitora seu processo e o reinicia se ele travar.

Instale para sempre globalmente.

$ npm install -g forever

Executar aplicativo:

$ forever start server.js

Isso inicia o servidor e fornece um id para o processo (começa de 0).

Reinicie o aplicativo:

$ forever restart 0

Aqui 0 é o id do servidor.

Parar aplicação:

$ forever stop 0

Semelhante a reiniciar, 0 é o id do servidor. Você também pode fornecer o id do processo ou o nome do script no lugar do id fornecido pelo forever.

Para mais comandos: https://www.npmjs.com/package/forever

Implantação usando PM2 #

O PM2 é um gerenciador de processo de produção para aplicativos Node.js, que permite manter os aplicativos ativos para sempre e recarregá-los sem tempo de inatividade. O PM2 também permite que você gerencie o log, o monitoramento e o clustering de aplicativos.

Instale o pm2 globalmente.

npm install -g pm2

Em seguida, execute o aplicativo node.js usando PM2.

pm2 start server.js --name "my-app"

Os comandos a seguir são úteis ao trabalhar com PM2.

Liste todos os processos em execução:

pm2 list

Parar um aplicativo:

pm2 stop my-app

Reinicie um aplicativo:

pm2 restart my-app

Para visualizar informações detalhadas sobre um aplicativo:

pm2 show my-app

Para remover um aplicativo do registro do PM2:

pm2 delete my-app

Usando diferentes propriedades/configurações para diferentes ambientes como dev, qa, staging etc. #

Aplicativos de grande escala geralmente precisam de propriedades diferentes ao serem executados em ambientes diferentes. podemos conseguir isso passando argumentos para o aplicativo NodeJs e usando o mesmo argumento no processo do nó para carregar o arquivo de propriedades do ambiente específico.

Suponha que temos dois arquivos de propriedades para ambientes diferentes.


  • dev.json

      {
      "PORT": 3000,
      "DB": {
          "host": "localhost",
          "user": "bob",
          "password": "12345"
      }
  }

  • qa.json

      {
      "PORT": 3001,
      "DB": {
          "host": "where_db_is_hosted",
          "user": "bob",
          "password": "54321"
      }
  }

O código a seguir no aplicativo exportará o respectivo arquivo de propriedade que queremos usar.

process.argv.forEach(function (val) {
    var arg = val.split("=");
    if (arg.length > 0) {
        if (arg[0] === 'env') {
            var env = require('./' + arg[1] + '.json');
            exports.prop = env;
        }
    }
});

Damos argumentos para o aplicativo como a seguir

node app.js env=dev

se estivermos usando o gerenciador de processos como * para sempre * do que tão simples quanto

forever start app.js env=dev

Aproveitando os clusters #

Uma única instância do Node.js é executada em um único encadeamento. Para aproveitar os sistemas multinúcleo, o usuário às vezes desejará iniciar um cluster de processos Node.js para lidar com a carga.

 var cluster = require('cluster');
    
    var numCPUs = require('os').cpus().length;
    
        if (cluster.isMaster) {
          // In real life, you'd probably use more than just 2 workers,
          // and perhaps not put the master and worker in the same file.
          //
          // You can also of course get a bit fancier about logging, and
          // implement whatever custom logic you need to prevent DoS
          // attacks and other bad behavior.
          //
          // See the options in the cluster documentation.
          //
          // The important thing is that the master does very little,
          // increasing our resilience to unexpected errors.
            console.log('your server is working on ' + numCPUs + ' cores');
            
             for (var i = 0; i < numCPUs; i++) {
                    cluster.fork();
                  }
             
          cluster.on('disconnect', function(worker) {
            console.error('disconnect!');
            //clearTimeout(timeout);
            cluster.fork();
          });
    
          
        } else {
            require('./app.js');
        
        }
