Diretório virtual / aplicativo aninhado com armadilha de visualizações #

Se você for usar o Express para renderizar visualizações usando um View Engine, precisará passar o valor virtualDirPath para suas visualizações

`res.render('index', { virtualDirPath: virtualDirPath });`

A razão para fazer isso é tornar seus hiperlinks para outras exibições hospedadas por seu aplicativo e caminhos de recursos estáticos para saber onde o site está sendo hospedado sem precisar modificar todas as exibições após a implantação. Esta é uma das armadilhas mais irritantes e tediosas do uso de diretórios virtuais com IISNode.

Versões #

Todos os exemplos acima funcionam com

  • Expresso v4.x
  • IIS 7.x/8.x
  • Socket.io v1.3.x ou superior

Usando um diretório virtual do IIS ou aplicativo aninhado via #

Usar um diretório virtual ou aplicativo aninhado no IIS é um cenário comum e provavelmente um que você desejará aproveitar ao usar o IISNode.

O IISNode não oferece suporte direto para Diretórios Virtuais ou Aplicativos Aninhados por meio de configuração, portanto, para isso, precisaremos aproveitar um recurso do IISNode que não faz parte da configuração e é muito menos conhecido. Todos os filhos do elemento <appSettings> com o Web.config são adicionados ao objeto process.env como propriedades usando a chave appSetting.

Vamos criar um diretório virtual em nosso <appSettings>

<appSettings>
  <add key="virtualDirPath" value="/foo" />
</appSettings>

Dentro do nosso aplicativo Node.js podemos acessar a configuração virtualDirPath

console.log(process.env.virtualDirPath); // prints /foo

Agora que podemos usar o elemento <appSettings> para configuração, vamos aproveitar isso e usá-lo em nosso código de servidor.

// Access the virtualDirPath appSettings and give it a default value of '/'
// in the event that it doesn't exist or isn't set
var virtualDirPath = process.env.virtualDirPath || '/';

// We also want to make sure that our virtualDirPath 
// always starts with a forward slash
if (!virtualDirPath.startsWith('/', 0))
  virtualDirPath = '/' + virtualDirPath;

// Setup a route at the index of our app    
server.get(virtualDirPath, (req, res) => {
    return res.status(200).send('Hello World');
});

Podemos usar o virtualDirPath com nossos recursos estáticos também

// Public Directory
server.use(express.static(path.join(virtualDirPath, 'public')));
// Bower
server.use('/bower_components', express.static(path.join(virtualDirPath, 'bower_components')));

Vamos juntar tudo isso

const express = require('express');
const server = express();

const port = process.env.PORT || 3000;

// Access the virtualDirPath appSettings and give it a default value of '/'
// in the event that it doesn't exist or isn't set
var virtualDirPath = process.env.virtualDirPath || '/';

// We also want to make sure that our virtualDirPath 
// always starts with a forward slash
if (!virtualDirPath.startsWith('/', 0))
  virtualDirPath = '/' + virtualDirPath;

// Public Directory
server.use(express.static(path.join(virtualDirPath, 'public')));
// Bower
server.use('/bower_components', express.static(path.join(virtualDirPath, 'bower_components')));

// Setup a route at the index of our app    
server.get(virtualDirPath, (req, res) => {
    return res.status(200).send('Hello World');
});

server.listen(port, () => {
    console.log(`Listening on ${port}`);
});

Começando #

IISNode permite que aplicativos Web Node.js sejam hospedados no IIS 7/8, assim como um aplicativo .NET faria. Claro, você pode hospedar seu processo node.exe no Windows, mas por que fazer isso quando você pode apenas executar seu aplicativo no IIS.

O IISNode lidará com dimensionamento em vários núcleos, gerenciamento de processos de node.exe e reciclará automaticamente seu aplicativo IIS sempre que seu aplicativo for atualizado, apenas para citar alguns de seus [benefícios](https://tomasz.janczuk.org /2011/08/hosting-nodejs-applications-in-iis-on.html).

Requisitos #

O IISNode tem alguns requisitos antes que você possa hospedar seu aplicativo Node.js no IIS.

  1. Node.js deve ser instalado no host IIS, 32 bits ou 64 bits, ambos são suportados.
  2. IISNode instalado x86 ou [x64](https://github .com/azure/iisnode/releases/download/v0.2.21/iisnode-full-v0.2.21-x64.msi), isso deve corresponder ao número de bits do seu host IIS.
  3. O Módulo de Reescrita de URL da Microsoft para IIS instalado em seu host IIS.
    • This is key, otherwise requests to your Node.js app won’t function as expected.
  4. Um Web.config na pasta raiz do seu aplicativo Node.js.
  5. Configuração do IISNode através de um arquivo iisnode.yml ou um elemento <iisnode> dentro do seu Web.config.

Exemplo básico de Hello World usando Express #

Para que este exemplo funcione, você precisará criar um aplicativo IIS 7/8 em seu host IIS e adicionar o diretório que contém o aplicativo Web Node.js como o diretório físico. Certifique-se de que sua identidade de pool de aplicativos/aplicativos possa acessar a instalação do Node.js. Este exemplo usa a instalação de 64 bits do Node.js.

Estrutura do Projeto #

Essa é a estrutura básica do projeto de um aplicativo Web IISNode/Node.js. Parece quase idêntico a qualquer aplicativo da Web que não seja IISNode, exceto pela adição do Web.config.

- /app_root
  - package.json
  - server.js
  - Web.config

server.js - Aplicativo Expresso #

const express = require('express');
const server = express();

// We need to get the port that IISNode passes into us 
// using the PORT environment variable, if it isn't set use a default value
const port = process.env.PORT || 3000;

// Setup a route at the index of our app    
server.get('/', (req, res) => {
    return res.status(200).send('Hello World');
});

server.listen(port, () => {
    console.log(`Listening on ${port}`);
});

Configuração & Web.config #

O Web.config é como qualquer outro Web.config do IIS, exceto que as duas coisas a seguir devem estar presentes, URL <rewrite><rules> e um IISNode <handler>. Ambos os elementos são filhos do elemento <system.webServer>.

Configuração #

Você pode configurar o IISNode usando um arquivo iisnode.yml ou adicionando o arquivo <iisnode > como um filho de <system.webServer> em seu Web.config . Ambas as configurações podem ser usadas em conjunto, entretanto, neste caso, o Web.config precisará especificar o arquivo iisnode.yml E qualquer conflito de configuração será obtido do arquivo iisnode .yml em vez disso. Essa substituição de configuração não pode acontecer ao contrário.

Manipulador IISNode #

Para que o IIS saiba que server.js contém nosso aplicativo Web Node.js, precisamos informar isso explicitamente. Podemos fazer isso adicionando o IISNode <handler> ao elemento <handlers>.

<handlers>
  <add name="iisnode" path="server.js" verb="*" modules="iisnode"/>
</handlers>

Regras de reescrita de URL #

A parte final da configuração é garantir que o tráfego destinado ao nosso aplicativo Node.js que entra em nossa instância do IIS seja direcionado para o IISNode. Sem regras de reescrita de URL, precisaríamos visitar nosso aplicativo indo para http://<host>/server.js e pior ainda, ao tentar solicitar um recurso fornecido por server.js você obterá um 404. É por isso que a reescrita de URL é necessária para aplicativos da Web IISNode.

<rewrite>
    <rules>
        <!-- First we consider whether the incoming URL matches a physical file in the /public folder -->
        <rule name="StaticContent" patternSyntax="Wildcard">
            <action type="Rewrite" url="public/{R:0}" logRewrittenUrl="true"/>
            <conditions>
                <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true"/>
            </conditions>
            <match url="*.*"/>
        </rule>

        <!-- All other URLs are mapped to the Node.js application entry point -->
        <rule name="DynamicContent">
            <conditions>
                <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="True"/>
            </conditions>
            <action type="Rewrite" url="server.js"/>
        </rule>
    </rules>
</rewrite>

Este é um arquivo Web.config funcional para este exemplo, configurado para uma instalação Node.js de 64 bits.

É isso, agora acesse seu Site IIS e veja seu aplicativo Node.js funcionando.

Usando Socket.io com IISNode #

Para que o Socket.io funcione com o IISNode, as únicas alterações necessárias quando não estiver usando um Diretório Virtual/Aplicativo Aninhado estão dentro do Web.config.

Como o Socket.io envia solicitações começando com /socket.io, o IISNode precisa comunicar ao IIS que elas também devem ser tratadas pelo IISNode e não são apenas solicitações de arquivos estáticos ou outro tráfego. Isso requer um <handler> diferente dos aplicativos IISNode padrão.

<handlers>
    <add name="iisnode-socketio" path="server.js" verb="*" modules="iisnode" />
</handlers>

Além das mudanças nos <handlers>, também precisamos adicionar uma regra de reescrita de URL adicional. A regra de reescrita envia todo o tráfego /socket.io para nosso arquivo de servidor onde o servidor Socket.io está sendo executado.

<rule name="SocketIO" patternSyntax="ECMAScript">
    <match url="socket.io.+"/>
    <action type="Rewrite" url="server.js"/>
</rule>

Se você estiver usando o IIS 8, você precisará desabilitar sua configuração webSockets em seu Web.config além de adicionar o manipulador acima e reescrever as regras. Isso é desnecessário no IIS 7, pois não há suporte para webSocket.

<webSocket enabled="false" />
