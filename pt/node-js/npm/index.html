Instalando pacotes #

Introdução #

Pacote é um termo usado pelo npm para denotar ferramentas que os desenvolvedores podem usar para seus projetos. Isso inclui tudo, desde bibliotecas e estruturas como jQuery e AngularJS até executores de tarefas como Gulp.js. Os pacotes virão em uma pasta normalmente chamada node_modules, que também conterá um arquivo package.json. Este arquivo contém informações sobre todos os pacotes, incluindo quaisquer dependências, que são módulos adicionais necessários para usar um determinado pacote.

O Npm usa a linha de comando para instalar e gerenciar pacotes, portanto, os usuários que tentam usar o npm devem estar familiarizados com os comandos básicos em seu sistema operacional, ou seja: percorrer diretórios, além de poder ver o conteúdo dos diretórios.

Instalando o NPM #

Observe que, para instalar pacotes, você deve ter o NPM instalado.

A maneira recomendada de instalar o NPM é usar um dos instaladores da página de download do Node.js. Você pode verificar se já tem o node.js instalado executando o comando npm -v ou npm version.

Depois de instalar o NPM por meio do instalador do Node.js, verifique se há atualizações. Isso ocorre porque o NPM é atualizado com mais frequência do que o instalador do Node.js. Para verificar se há atualizações, execute o seguinte comando:

npm install npm@latest -g

Como instalar pacotes #

Para instalar um ou mais pacotes, use o seguinte:

npm install <package-name>
# or
npm i <package-name>...

# e.g. to install lodash and express
npm install lodash express

Observação: Isso instalará o pacote no diretório em que a linha de comando está atualmente, portanto, é importante verificar se o diretório apropriado foi escolhido

Se você já tem um arquivo package.json em seu diretório de trabalho atual e as dependências estão definidas nele, então npm install resolverá e instalará automaticamente todas as dependências listadas no arquivo. Você também pode usar a versão abreviada do comando npm install que é: npm i

Se você deseja instalar uma versão específica de um pacote, use:

npm install <name>@<version>

# e.g. to install version 4.11.1 of the package lodash
npm install lodash@4.11.1

Se você quiser instalar uma versão que corresponda a um intervalo de versão específico, use:

npm install <name>@<version range>

# e.g. to install a version which matches "version >= 4.10.1" and "version < 4.11.1"
# of the package lodash
npm install lodash@">=4.10.1 <4.11.1"

Se você deseja instalar a versão mais recente, use:

npm install <name>@latest

Os comandos acima irão procurar por pacotes no repositório central npm em npmjs.com. Se você não deseja instalar a partir do registro npm, outras opções são suportadas, como:

# packages distributed as a tarball
npm install <tarball file>
npm install <tarball url>

# packages available locally
npm install <local path>

# packages available as a git repository
npm install <git remote url>

# packages available on GitHub
npm install <username>/<repository>

# packages available as gist (need a package.json)
npm install gist:<gist-id>

# packages from a specific repository
npm install --registry=http://myreg.mycompany.com <package name>

# packages from a related group of packages 
# See npm scope
npm install @<scope>/<name>(@<version>)

# Scoping is useful for separating private packages hosted on private registry from
# public ones by setting registry for specific scope
npm config set @mycompany:registry http://myreg.mycompany.com
npm install @mycompany/<package name>

Normalmente, os módulos serão instalados localmente em uma pasta chamada node_modules, que pode ser encontrada em seu diretório de trabalho atual. Este é o diretório que o require() usará para carregar os módulos para torná-los disponíveis para você.

Se você já criou um arquivo package.json, você pode usar a opção --save (abreviação -S) ou uma de suas variantes para adicionar automaticamente o pacote instalado ao seu package.json como uma dependência . Se outra pessoa instalar seu pacote, o npm lerá automaticamente as dependências do arquivo package.json e instalará as versões listadas. Observe que você ainda pode adicionar e gerenciar suas dependências editando o arquivo posteriormente, portanto, geralmente é uma boa ideia acompanhar as dependências, por exemplo, usando:

npm install --save <name> # Install dependencies 
# or
npm install -S <name> # shortcut version --save 
# or
npm i -S <name>

Para instalar pacotes e salvá-los apenas se forem necessários para o desenvolvimento, não para executá-los, não se forem necessários para a execução do aplicativo, siga o seguinte comando:

npm install --save-dev <name> # Install dependencies for development purposes
# or
npm install -D <name> # shortcut version --save-dev
# or
npm i -D <name>

Instalando dependências #

Alguns módulos não apenas fornecem uma biblioteca para você usar, mas também fornecem um ou mais binários que devem ser usados ​​por meio da linha de comando. Embora você ainda possa instalar esses pacotes localmente, geralmente é preferível instalá-los globalmente para que as ferramentas de linha de comando possam ser habilitadas. Nesse caso, o npm vinculará automaticamente os binários aos caminhos apropriados (por exemplo, /usr/local/bin/<name>) para que possam ser usados ​​a partir da linha de comando. Para instalar um pacote globalmente, use:

npm install --global <name>
# or
npm install -g <name>
# or
npm i -g <name>

# e.g. to install the grunt command line tool
npm install -g grunt-cli

Se você quiser ver uma lista de todos os pacotes instalados e suas versões associadas no espaço de trabalho atual, use:

npm list
npm list <name>

Adicionar um argumento de nome opcional pode verificar a versão de um pacote específico.

Nota: Se você tiver problemas de permissão ao tentar instalar um módulo npm globalmente, resista à tentação de emitir um sudo npm install -g ... para superar o problema. Conceder scripts de terceiros para serem executados em seu sistema com privilégios elevados é perigoso. O problema de permissão pode significar que você tem um problema com a forma como o próprio npm foi instalado. Se você estiver interessado em instalar o Node em ambientes de usuário em área restrita, tente usar nvm.

Se você tiver ferramentas de compilação ou outras dependências somente de desenvolvimento (por exemplo, Grunt), talvez não queira tê-las empacotadas com o aplicativo que você implanta. Se for esse o caso, você vai querer tê-lo como uma dependência de desenvolvimento, que está listada no package.json em devDependencies. Para instalar um pacote como uma dependência somente de desenvolvimento, use --save-dev (ou -D).

npm install --save-dev <name> // Install development dependencies which is not included in production 
# or
npm install -D <name>

Você verá que o pacote é então adicionado ao devDependencies do seu package.json.

Para instalar dependências de um projeto node.js baixado/clonado, você pode simplesmente usar

npm install
# or
npm i

O npm lerá automaticamente as dependências do package.json e as instalará.

NPM por trás de um servidor proxy #

Se o seu acesso à Internet for por meio de um servidor proxy, talvez seja necessário modificar os comandos npm install que acessam repositórios remotos. O npm usa um arquivo de configuração que pode ser atualizado via linha de comando:

npm config set

Você pode localizar suas configurações de proxy no painel de configurações do seu navegador. Depois de obter as configurações de proxy (URL do servidor, porta, nome de usuário e senha); você precisa configurar suas configurações do npm da seguinte maneira.

$ npm config set proxy http://<username>:<password>@<proxy-server-url>:<port>
$ npm config set https-proxy http://<username>:<password>@<proxy-server-url>:<port>

Os campos username, password, port são opcionais. Depois de definir isso, seu npm install, npm i -g etc. funcionarão corretamente.

Desinstalando pacotes #

Para desinstalar um ou mais pacotes instalados localmente, use:

npm uninstall <package name>

O comando de desinstalação do npm possui cinco aliases que também podem ser usados:

npm remove <package name>
npm rm <package name>
npm r <package name>

npm unlink <package name>
npm un <package name>

Se você quiser remover o pacote do arquivo package.json como parte da desinstalação, use o sinalizador --save (abreviação: -S):

npm uninstall --save <package name>
npm uninstall -S <package name>

Para uma dependência de desenvolvimento, use o sinalizador --save-dev (abreviação: -D):

npm uninstall --save-dev <package name>
npm uninstall -D <package name>

Para uma dependência opcional, use o sinalizador --save-optional (abreviação: -O):

npm uninstall --save-optional <package name>
npm uninstall -O <package name>

Para pacotes instalados globalmente, use o sinalizador --global (abreviação: -g):

npm uninstall -g <package name>

Configurando uma configuração de pacote #

As configurações do pacote Node.js estão contidas em um arquivo chamado package.json que você pode encontrar na raiz de cada projeto. Você pode configurar um novo arquivo de configuração chamando:

npm init

Isso tentará ler o diretório de trabalho atual para obter informações do repositório Git (se existir) e variáveis ​​de ambiente para tentar preencher automaticamente alguns dos valores de espaço reservado para você. Caso contrário, ele fornecerá uma caixa de diálogo de entrada para as opções básicas.

Se você quiser criar um package.json com valores padrão, use:

npm init --yes
# or
npm init -y

Se você estiver criando um package.json para um projeto que não vai publicar como um pacote npm (ou seja, apenas com o propósito de arredondar suas dependências), você pode transmitir essa intenção em seu package.json arquivo:

  1. Opcionalmente, defina a propriedade private como true para evitar publicações acidentais.
  2. Opcionalmente, defina a propriedade license como “UNLICENSED” para negar a outros o direito de usar seu pacote.

Para instalar um pacote e salvá-lo automaticamente em seu package.json, use:

npm install --save <package>

O pacote e os metadados associados (como a versão do pacote) aparecerão em suas dependências. Se você salvar if como uma dependência de desenvolvimento (usando --save-dev), o pacote aparecerá em seu devDependencies.

Com este package.json básico, você encontrará mensagens de aviso ao instalar ou atualizar pacotes, informando que está faltando uma descrição e o campo do repositório. Embora seja seguro ignorar essas mensagens, você pode se livrar delas abrindo o package.json em qualquer editor de texto e adicionando as seguintes linhas ao objeto JSON:

[...]
"description": "No description",
"repository": {
  "private": true
},
[...]

Executando scripts #

Você pode definir scripts em seu package.json, por exemplo:

{
  "name": "your-package",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "author": "",
  "license": "ISC",
  "dependencies": {},
  "devDependencies": {},
  "scripts": {
    "echo": "echo hello!"
  }
}

Para executar o script echo, execute npm run echo na linha de comando. Scripts arbitrários, como echo acima, devem ser executados com npm run <script name>. O npm também possui vários scripts oficiais que são executados em certos estágios da vida do pacote (como preinstall). Veja aqui para toda a visão geral de como o npm lida com campos de script.

Os scripts npm são usados ​​com mais frequência para coisas como iniciar um servidor, construir o projeto e executar testes. Aqui está um exemplo mais realista:

  "scripts": {
    "test": "mocha tests",
    "start": "pm2 start index.js"
  }

Nas entradas scripts, programas de linha de comando como mocha funcionarão quando instalados globalmente ou localmente. Se a entrada da linha de comando não existir no PATH do sistema, o npm também verificará seus pacotes instalados localmente.

Se seus scripts ficarem muito longos, eles podem ser divididos em partes, assim:

  "scripts": {
    "very-complex-command": "npm run chain-1 && npm run chain-2",
    "chain-1": "webpack",
    "chain-2": "node app.js"
  }

Versão semântica básica #

Antes de publicar um pacote, você precisa versioná-lo. O npm suporta versão semântica, isso significa que existem lançamentos patch, secundários e principais.

Por exemplo, se seu pacote estiver na versão 1.2.3 para alterar a versão, você deve:

  1. lançamento do patch: npm version patch => 1.2.4
  2. versão secundária: npm version minor => 1.3.0
  3. versão principal: npm version major => 2.0.0

Você também pode especificar uma versão diretamente com:

npm versão 3.1.4 => 3.1.4

Quando você define uma versão de pacote usando um dos comandos npm acima, o npm modificará o campo de versão do arquivo package.json, confirmará e também criará uma nova tag Git com a versão prefixada com um “v”, como se você já emitiu o comando:

tag git v3.1.4

Ao contrário de outros gerenciadores de pacotes como Bower, o registro npm não depende da criação de tags Git para cada versão. Mas, se você gosta de usar tags, lembre-se de enviar a tag recém-criada depois de aumentar a versão do pacote:

git push origin master (para enviar a mudança para package.json)

git push origin v3.1.4 (para enviar a nova tag)

Ou você pode fazer isso de uma só vez com:

git push origin master --tags

Publicando um pacote #

Primeiro, verifique se você configurou seu pacote (como dito em https://www.wikiod.com/pt/node-js/npm#Configurando uma configuração de pacote​). Então, você precisa estar logado no npmjs.

Se você já tem um usuário npm

npm login

Se você não tem um usuário

npm adduser

Para verificar se seu usuário está registrado no cliente atual

npm config ls

Depois disso, quando seu pacote estiver pronto para ser publicado, use

npm publish

E você está feito.

Se você precisar publicar uma nova versão, certifique-se de atualizar a versão do pacote, conforme indicado em https://www.wikiod.com/pt/node-js/npm#Versionamento semântico básico Caso contrário, o npm não permitirá que você publique o pacote.

{
    name: "package-name",
    version: "1.0.4"
}

Escopos e repositórios #

# Set the repository for the scope "myscope"
npm config set @myscope:registry http://registry.corporation.com

# Login at a repository and associate it with the scope "myscope"
npm adduser --registry=http://registry.corporation.com --scope=@myscope

# Install a package "mylib" from the scope "myscope"
npm install @myscope/mylib

Se o nome do seu próprio pacote começar com @myscope e o escopo “myscope” estiver associado a um repositório diferente, o npm publish fará o upload do seu pacote para esse repositório.

Você também pode manter essas configurações em um arquivo .npmrc:

@myscope:registry=http://registry.corporation.com
//registry.corporation.com/:_authToken=xxxxxxxx-xxxx-xxxx-xxxxxxxxxxxxxxx

Isso é útil ao automatizar a compilação em um servidor CI, por exemplo.

Listando os pacotes instalados atualmente #

Para gerar uma lista (exibição em árvore) dos pacotes atualmente instalados, use

npm list

ls, la e ll são aliases do comando list. Os comandos la e ll mostram informações estendidas como descrição e repositório.

Opções

O formato de resposta pode ser alterado passando opções.

npm list --json
  • json - Mostra informações no formato json
  • longo - Mostra informações estendidas
  • parseable - Mostra a lista analisável em vez da árvore
  • global - Mostra os pacotes instalados globalmente
  • profundidade - Profundidade máxima de exibição da árvore de dependência
  • dev/development - Mostra devDependencies
  • prod/produção - Mostra as dependências

Se desejar, você também pode acessar a página inicial do pacote.

npm home <package name>

Atualizando npm e pacotes #

Como o próprio npm é um módulo Node.js, ele pode ser atualizado usando ele mesmo.

Se o SO for Windows deve estar executando o prompt de comando como Admin

npm install -g npm@latest

Se você quiser verificar se há versões atualizadas, você pode fazer:

npm outdated

Para atualizar um pacote específico:

npm update <package name>

Isso atualizará o pacote para a versão mais recente de acordo com as restrições em package.json

Caso você também queira bloquear a versão atualizada em package.json:

npm update <package name> --save

Bloqueando módulos para versões específicas #

Por padrão, o npm instala a última versão disponível dos módulos de acordo com a [versão semântica] de cada dependência (https://www.wikiod.com/pt/node-js/npm#Versionamento semântico básico). Isso pode ser problemático se um autor de módulo não aderir ao semver e introduzir alterações importantes em uma atualização de módulo, por exemplo.

Para bloquear a versão de cada dependência (e as versões de suas dependências, etc) para a versão específica instalada localmente na pasta node_modules, use

npm shrinkwrap

Isso criará um npm-shrinkwrap.json junto com seu package.json que lista as versões específicas das dependências.

Configurando para pacotes instalados globalmente #

Você pode usar npm install -g para instalar um pacote “globalmente”. Isso geralmente é feito para instalar um executável que você pode adicionar ao seu caminho para execução. Por exemplo:

npm install -g gulp-cli

Se você atualizar seu caminho, você pode chamar gulp diretamente.

Em muitos sistemas operacionais, npm install -g tentará gravar em um diretório no qual seu usuário pode não conseguir gravar, como /usr/bin. Você não deve usar sudo npm install neste caso, pois existe um possível risco de segurança de executar scripts arbitrários com sudo e o usuário root pode criar diretórios em sua casa nos quais você não pode escrever, o que faz instalações futuras mais difícil.

Você pode dizer ao npm onde instalar os módulos globais através do seu arquivo de configuração, ~/.npmrc. Isso é chamado de prefix que você pode ver com npm prefix.

prefix=~/.npm-global-modules

Isso usará o prefixo sempre que você executar npm install -g. Você também pode usar npm install --prefix ~/.npm-global-modules para definir o prefixo ao instalar. Se o prefixo for o mesmo da sua configuração, você não precisa usar -g.

Para usar o módulo instalado globalmente, ele precisa estar no seu caminho:

export PATH=$PATH:~/.npm-global-modules/bin

Agora quando você executar npm install -g gulp-cli você poderá usar gulp.

Nota: Quando você npm install (sem -g) o prefixo será o diretório com package.json ou o diretório atual se nenhum for encontrado na hierarquia. Isso também cria um diretório node_modules/.bin que contém os executáveis. Se você quiser usar um executável específico para um projeto, não é necessário usar npm install -g. Você pode usar aquele em node_modules/.bin.

Vinculando projetos para depuração e desenvolvimento mais rápidos #

Às vezes, construir dependências de projeto pode ser uma tarefa tediosa. Em vez de publicar uma versão de pacote no NPM e instalar a dependência para testar as alterações, use npm link. npm link cria um link simbólico para que o código mais recente possa ser testado em um ambiente local. Isso facilita o teste de ferramentas globais e dependências de projeto, permitindo que o código mais recente seja executado antes de criar uma versão publicada.

Texto de ajuda #

NAME
       npm-link - Symlink a package folder

SYNOPSIS
         npm link (in package dir)
         npm link [<@scope>/]<pkg>[@<version>]

         alias: npm ln

Etapas para vincular dependências do projeto #

Ao criar o link de dependência, observe que o nome do pacote é o que será referenciado no projeto pai.

  1. CD em um diretório de dependência (ex: cd ../my-dep)
  2. link npm
  3. CD no projeto que vai usar a dependência
  4. npm link my-dep ou se for namespace npm link @namespace/my-dep

Etapas para vincular uma ferramenta global #

  1. CD no diretório do projeto (ex: cd eslint-watch)
  2. link npm
  3. Use a ferramenta
  4. esw --quiet

Problemas que podem surgir #

Às vezes, vincular projetos pode causar problemas se a dependência ou a ferramenta global já estiver instalada. npm uninstall (-g) <pkg> e então executar npm link normalmente resolve quaisquer problemas que possam surgir.
