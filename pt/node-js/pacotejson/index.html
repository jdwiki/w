Você pode criar package.json com

npm init

que perguntará sobre fatos básicos sobre seus projetos, incluindo identificador de licença.

Scripts #

Você pode definir scripts que podem ser executados ou acionados antes ou depois de outro script.

{
  "scripts": {
    "pretest": "scripts/pretest.js",
    "test": "scripts/test.js",
    "posttest": "scripts/posttest.js"
  }
}

Nesse caso, você pode executar o script executando um destes comandos:

$ npm run-script test
$ npm run test
$ npm test
$ npm t

Scripts predefinidos #

Nome do roteiro Descrição
pré-publicar Execute antes que o pacote seja publicado.
publicar, pós-publicar Execute após a publicação do pacote.
pré-instalar Execute antes que o pacote seja instalado.
instalar, pós-instalar Execute após a instalação do pacote.
pré-instalar, desinstalar Execute antes que o pacote seja desinstalado.
pós-desinstalação Execute após a desinstalação do pacote.
pré-versão, versão Execute antes de aumentar a versão do pacote.
pós-versão Execute após a versão do pacote.
pré-teste, teste, pós-teste Executado pelo comando npm test
prestop, pare, poststop Executado pelo comando npm stop
pré-início, início, pós-início Executado pelo comando npm start
pré-reiniciar, reiniciar, pós-reiniciar Executado pelo comando npm restart

Scripts definidos pelo usuário #

Você também pode definir seus próprios scripts da mesma forma que faz com os scripts predefinidos:

{
  "scripts": {
    "preci": "scripts/preci.js",
    "ci": "scripts/ci.js",
    "postci": "scripts/postci.js"
  }
}

Nesse caso, você pode executar o script executando um destes comandos:

$ npm run-script ci
$ npm run ci

Os scripts definidos pelo usuário também suportam scripts pre e post, conforme mostrado no exemplo acima.

Explorando package.json #

Um arquivo package.json, geralmente presente na raiz do projeto, contém metadados sobre seu aplicativo ou módulo, bem como a lista de dependências a serem instaladas do npm ao executar o npm install.

Para inicializar um package.json digite npm init em seu prompt de comando.

Para criar um package.json com valores padrão, use:

npm init --yes
# or
npm init -y

Para instalar um pacote e salvá-lo em package.json use:

npm install {package name} --save

Você também pode usar a notação abreviada:

 npm i -S {package name}

O NPM alias -S para --save e -D para --save-dev para salvar em suas dependências de produção ou desenvolvimento, respectivamente.

O pacote aparecerá em suas dependências; se você usar --save-dev em vez de --save, o pacote aparecerá em seu devDependencies.

Propriedades importantes de package.json:

{
  "name": "module-name",
  "version": "10.3.1",
  "description": "An example module to illustrate the usage of a package.json",
  "author": "Your Name <your.name@example.org>",
  "contributors": [{
    "name": "Foo Bar",
    "email": "foo.bar@example.com"
  }],
  "bin": {
    "module-name": "./bin/module-name"
  },
  "scripts": {
    "test": "vows --spec --isolate",
    "start": "node index.js",
    "predeploy": "echo About to deploy",
    "postdeploy": "echo Deployed",
    "prepublish": "coffee --bare --compile --output lib/foo src/foo/*.coffee"
  },
  "main": "lib/foo.js",
  "repository": {
    "type": "git",
    "url": "https://github.com/username/repo"
  },
  "bugs": {
    "url": "https://github.com/username/issues"
  },
  "keywords": [
    "example"
  ],
  "dependencies": {
    "express": "4.2.x"
  },
  "devDependencies": {
    "assume": "<1.0.0 || >=2.3.1 <2.4.5 || >=2.5.2 <3.0.0"
  },
  "peerDependencies": {
    "moment": ">2.0.0"
  },
  "preferGlobal": true,
  "private": true,
  "publishConfig": {
    "registry": "https://your-private-hosted-npm.registry.domain.com"
  },
  "subdomain": "foobar",
  "analyze": true,
  "license": "MIT",
  "files": [
    "lib/foo.js"
  ]
}

Informações sobre algumas propriedades importantes:

name

O nome exclusivo do seu pacote e deve estar em letras minúsculas. Essa propriedade é necessária e seu pacote não será instalado sem ela.

  1. O nome deve ser menor ou igual a 214 caracteres.
  2. O nome não pode começar com um ponto ou sublinhado.
  3. Novos pacotes não devem ter letras maiúsculas no nome.
version

A versão do pacote é especificada por Versão Semântica (semver). Que assume que um número de versão é escrito como MAJOR.MINOR.PATCH e você incrementa o:

  1. Versão PRINCIPAL quando você faz alterações de API incompatíveis

  2. Versão MENOR when you add functionality in a backwards-compatible manner

  3. REMENDO version when you make backwards-compatible bug fixes

    description

A descrição do projeto. Tente mantê-lo curto e conciso.

author

O autor deste pacote.

bin

Um objeto que é usado para expor scripts binários do seu pacote. O objeto assume que a chave é o nome do script binário e o valor um caminho relativo para o script.

Esta propriedade é usada por pacotes que contêm uma CLI (interface de linha de comando).

script

Um objeto que expõe comandos npm adicionais. O objeto assume que a chave é o comando npm e o valor é o caminho do script. Esses scripts podem ser executados quando você executa npm run {command name} ou npm run-script {command name}.

Os pacotes que contêm uma interface de linha de comando e são instalados localmente podem ser chamados sem um caminho relativo. Então ao invés de chamar ./node-modules/.bin/mocha você pode chamar diretamente mocha.

main

O principal ponto de entrada para o seu pacote. Ao chamar require('{module name}') no nó, este será o arquivo real que é necessário.

É altamente recomendável que exigir o arquivo principal não gere nenhum efeito colateral. Por exemplo, exigir o arquivo principal não deve iniciar um servidor HTTP ou conectar-se a um banco de dados. Em vez disso, você deve criar algo como exports.init = function() {...} em seu script principal.

keywords

Uma matriz de palavras-chave que descrevem seu pacote. Isso ajudará as pessoas a encontrar seu pacote.

devDependencies

Estas são as dependências que se destinam apenas ao desenvolvimento e teste do seu módulo. As dependências serão instaladas automaticamente a menos que a variável de ambiente NODE_ENV=production tenha sido definida. Se este for o caso, você ainda pode usar esses pacotes usando npm install --dev

peerDependencies

Se você estiver usando este módulo, peerDependencies lista os módulos que você deve instalar junto com este. Por exemplo, moment-timezone deve ser instalado junto com moment porque é um plugin para moment, mesmo que não require("moment") diretamente.

preferGlobal

Uma propriedade que indica que esta página prefere ser instalada globalmente usando npm install -g {module-name}. Esta propriedade é usada por pacotes que contêm uma CLI (interface de linha de comando).

Em todas as outras situações você NÃO deve usar esta propriedade.

publishConfig

O publishConfig é um objeto com valores de configuração que serão usados ​​para publicação de módulos. Os valores de configuração definidos substituem sua configuração npm padrão.

O uso mais comum do publishConfig é publicar seu pacote em um registro npm privado para que você ainda tenha os benefícios do npm, mas para pacotes privados. Isso é feito simplesmente definindo o URL do seu npm privado como valor para a chave do registro.

files

Esta é uma matriz de todos os arquivos a serem incluídos no pacote publicado. Um caminho de arquivo ou caminho de pasta pode ser usado. Todo o conteúdo de um caminho de pasta será incluído. Isso reduz o tamanho total do seu pacote incluindo apenas os arquivos corretos a serem distribuídos. Este campo funciona em conjunto com um arquivo de regras .npmignore.

Definição básica do projeto #

{
    "name": "my-project",
    "version": "0.0.1",
    "description": "This is a project.",
    "author": "Someone <someone@example.com>",
    "contributors": [{
        "name": "Someone Else",
        "email": "else@example.com"
    }],
    "keywords": ["improves", "searching"]
}
Campo Descrição
nome um campo obrigatório para a instalação de um pacote. Precisa ser minúscula, palavra única sem espaços. (Travessões e sublinhados permitidos)
versão um campo obrigatório para a versão do pacote usando versão semântica.
descrição uma breve descrição do projeto
autor especifica o autor do pacote
contribuintes um array de objetos, um para cada contribuidor
palavras-chave uma matriz de strings, isso ajudará as pessoas a encontrar seu pacote

Dependências #

“dependências”: { “module-name”: “0.1.0” }

  • exato: 0.1.0 instalará essa versão específica do módulo.
  • versão secundária mais recente: ^0.1.0 instalará a versão secundária mais recente, por exemplo, 0.2.0, mas não instalará um módulo com uma versão principal superior, por exemplo. 1.0.0
  • patch mais recente: 0.1.x ou ~0.1.0 instalará a versão de patch mais recente disponível, por exemplo 0.1.4, mas não instalará um módulo com versão maior ou menor, por exemplo. 0.2.0 ou 1.0.0.
  • curinga: * instalará a versão mais recente do módulo.
  • git repository: o seguinte instalará um tarball do branch master de um repositório git. Um #sha, #tag ou #branch também pode ser fornecido:
  • GitHub: usuário/projeto ou usuário/projeto#v1.0.0
  • url: git://gitlab.com/user/project.git ou git://gitlab.com/user/project.git#develop
  • caminho local: file:../lib/project

Após adicioná-los ao seu package.json, use o comando npm install no diretório do seu projeto no terminal.

devDependencies #

"devDependencies": {
    "module-name": "0.1.0"
}

Para dependências necessárias apenas para desenvolvimento, como testar proxies de estilo ext. Essas dependências de desenvolvimento não serão instaladas ao executar “npm install” no modo de produção.

Definição estendida do projeto #

Alguns dos atributos adicionais são analisados ​​pelo site npm como repository, bugs ou homepage e mostrados na infobox para estes pacotes

{
  "main": "server.js",  
  "repository" :  {
    "type": "git",
    "url": "git+https://github.com/<accountname>/<repositoryname>.git"
  },
  "bugs": {
    "url": "https://github.com/<accountname>/<repositoryname>/issues"
  },
  "homepage": "https://github.com/<accountname>/<repositoryname>#readme",
  "files": [
    "server.js", // source files
    "README.md", // additional files
    "lib" // folder with all included files
  ]
}
Campo Descrição
principal Script de entrada para este pacote. Este script é retornado quando um usuário requer o pacote.
repositório Localização e tipo do repositório público
