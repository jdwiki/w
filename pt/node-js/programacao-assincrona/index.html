Node é uma linguagem de programação onde tudo pode ser executado de forma assíncrona. Abaixo você pode encontrar alguns exemplos e as coisas típicas do trabalho assíncrono.

Sintaxe #

  • doSomething([args], function([argsCB]) { /* faça algo quando terminar */});
  • doSomething([args], ([argsCB]) => { /* fazer algo quando terminar */ });

Funções de retorno de chamada #

Funções de retorno de chamada em JavaScript #

As funções de retorno de chamada são comuns em JavaScript. Funções de retorno de chamada são possíveis em JavaScript porque funções são cidadãos de primeira classe.

Retornos de chamada síncronos. #

As funções de retorno de chamada podem ser síncronas ou assíncronas. Como as funções de retorno de chamada assíncronas podem ser mais complexas, aqui está um exemplo simples de uma função de retorno de chamada síncrona.

// a function that uses a callback named `cb` as a parameter
function getSyncMessage(cb) {
    cb("Hello World!");
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getSyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

A saída para o código acima é:

> Before getSyncMessage call
> Hello World!
> After getSyncMessage call

Primeiro, veremos como o código acima é executado. Isso é mais para aqueles que ainda não entendem o conceito de callbacks, se você já entende, sinta-se à vontade para pular este parágrafo. Primeiro, o código é analisado e, em seguida, a primeira coisa interessante a acontecer é a execução da linha 6, que gera Before getSyncMessage call para o console. Então a linha 8 é executada que chama a função getSyncMessage enviando uma função anônima como um argumento para o parâmetro chamado cb na função getSyncMessage. A execução agora é feita dentro da função getSyncMessage na linha 3 que executa a função cb que acabou de ser passada, esta chamada envia uma string de argumento “Hello World” para o parâmetro chamado message na função anônima passada. A execução então vai para a linha 9 que registra Hello World! no console. Em seguida, a execução passa pelo processo de sair do callstack (veja também) atingindo a linha 10, depois a linha 4 e, finalmente, de volta à linha 11.

Algumas informações para saber sobre retornos de chamada em geral:

  • A função que você envia para uma função como retorno de chamada pode ser chamada zero vezes, uma vez ou várias vezes. Tudo depende da implementação.
  • A função de retorno de chamada pode ser chamada de forma síncrona ou assíncrona e possivelmente de forma síncrona e assíncrona.
  • Assim como as funções normais, os nomes que você dá aos parâmetros para sua função não são importantes, mas a ordem é. Então, por exemplo, na linha 8, o parâmetro message poderia ter sido nomeado statement, msg, ou se você está sendo sem sentido algo como jellybean. Portanto, você deve saber quais parâmetros são enviados para seu retorno de chamada para que você possa obtê-los na ordem correta com nomes próprios.

Retornos de chamada assíncronos. #

Uma coisa a notar sobre o JavaScript é que ele é síncrono por padrão, mas existem APIs fornecidas no ambiente (navegador, Node.js, etc.) que podem torná-lo assíncrono (há mais sobre isso aqui).

Algumas coisas comuns que são assíncronas em ambientes JavaScript que aceitam retornos de chamada:

  • Eventos
  • setTimeout
  • setIntervalo
  • a API de busca
  • Promessas

Além disso, qualquer função que usa uma das funções acima pode ser encapsulada com uma função que recebe um retorno de chamada e o retorno de chamada seria então um retorno de chamada assíncrono (embora envolver uma promessa com uma função que recebe um retorno de chamada provavelmente seria considerado um antipadrão como existem maneiras mais preferidas de lidar com promessas).

Então, com essas informações, podemos construir uma função assíncrona semelhante à síncrona acima.

// a function that uses a callback named `cb` as a parameter
function getAsyncMessage(cb) {
    setTimeout(function () { cb("Hello World!") }, 1000);
}

console.log("Before getSyncMessage call");
// calling a function and sending in a callback function as an argument.
getAsyncMessage(function(message) {
    console.log(message);
});
console.log("After getSyncMessage call");

Que imprime o seguinte no console:

> Before getSyncMessage call
> After getSyncMessage call
// pauses for 1000 ms with no output
> Hello World!

A execução da linha vai para os logs da linha 6 “Antes da chamada getSyncMessage”. Então a execução vai para a linha 8 chamando getAsyncMessage com um retorno de chamada para o parâmetro cb. A linha 3 é então executada, chamando setTimeout com um retorno de chamada como primeiro argumento e o número 300 como segundo argumento. setTimeout faz o que faz e mantém esse retorno de chamada para que possa chamá-lo mais tarde em 1000 milissegundos, mas depois de configurar o tempo limite e antes de pausar os 1000 milissegundos, ele devolve a execução de onde parou para ir para linha 4, depois linha 11, e então pausa por 1 segundo e setTimeout então chama sua função de retorno de chamada que leva a execução de volta para a linha 3 onde o retorno de chamada getAsyncMessages é chamado com o valor “Hello World” para seu parâmetro message que é então registrado para o console na linha 9.

Funções de retorno de chamada em Node.js #

O NodeJS tem callbacks assíncronos e normalmente fornece dois parâmetros para suas funções, às vezes convencionalmente chamados de err e data. Um exemplo com a leitura de um arquivo de texto.

const fs = require("fs");

fs.readFile("./test.txt", "utf8", function(err, data) {
    if(err) {
        // handle the error 
    } else {
        // process the file text given with data
    }
});

Este é um exemplo de um retorno de chamada que é chamado uma única vez.

É uma boa prática lidar com o erro de alguma forma, mesmo que você esteja apenas registrando ou jogando. O else não é necessário se você lançar ou retornar e pode ser removido para diminuir o recuo, desde que você pare a execução da função atual no if fazendo algo como lançar ou retornar.

Embora possa ser comum ver err, data, nem sempre é o caso de seus callbacks usarem esse padrão, é melhor olhar a documentação.

Outro exemplo de retorno de chamada vem da biblioteca express (express 4.x):

// this code snippet was on http://expressjs.com/en/4x/api.html
const express = require('express');
const app = express();

// this app.get method takes a url route to watch for and a callback
// to call whenever that route is requested by a user.
app.get('/', function(req, res){
  res.send('hello world');
});

app.listen(3000);

Este exemplo mostra um retorno de chamada que é chamado várias vezes. O retorno de chamada é fornecido com dois objetos como parâmetros nomeados aqui como req e res, esses nomes correspondem à solicitação e à resposta, respectivamente, e fornecem maneiras de visualizar a solicitação recebida e configurar a resposta que será enviada ao usuário .

Como você pode ver, há várias maneiras de um retorno de chamada ser usado para executar código de sincronização e assíncrono em JavaScript e os retornos de chamada são muito onipresentes em todo o JavaScript.

Inferno de retorno de chamada #

O inferno de retorno de chamada (também uma pirâmide de doom ou efeito bumerangue) surge quando você aninha muitas funções de retorno de chamada dentro de uma função de retorno de chamada. Aqui está um exemplo para ler um arquivo (em ES6).

const fs = require('fs');
let filename = `${__dirname}/myfile.txt`;

fs.exists(filename, exists => {
    if (exists) {
        fs.stat(filename, (err, stats) => {
            if (err) {
                throw err;
            }
            if (stats.isFile()) {
                fs.readFile(filename, null, (err, data) => {
                    if (err) {
                        throw err;
                    }
                    console.log(data);
                });
            }
            else {
                throw new Error("This location contains not a file");
            }
        });
    }
    else {
        throw new Error("404: file not found");
    }
});

Como evitar o “Inferno de Callback”

Recomenda-se aninhar no máximo 2 funções de retorno de chamada. Isso ajudará você a manter a legibilidade do código e será muito mais fácil de manter no futuro. Se você precisar aninhar mais de 2 retornos de chamada, tente usar eventos distribuídos.

Também existe uma biblioteca chamada async que ajuda a gerenciar retornos de chamada e sua execução disponível no npm. Ele aumenta a legibilidade do código de retorno de chamada e oferece mais controle sobre o fluxo de código de retorno de chamada, inclusive permitindo que você os execute em paralelo ou em série.

Promessas nativas #

As promessas são uma ferramenta para programação assíncrona. Em JavaScript, as promessas são conhecidas por seus métodos then. As promessas têm dois estados principais ‘pendentes’ e ‘resolvidos’. Uma vez que uma promessa é ‘resolvida’, ela não pode voltar a ser ‘pendente’. Isso significa que as promessas são boas principalmente para eventos que ocorrem apenas uma vez. O estado ‘resolvido’ tem dois estados também ‘resolvido’ e ‘rejeitado’. Você pode criar uma nova promessa usando a palavra-chave new e passando uma função para o construtor new Promise(função (resolver, rejeitar) {}).

A função passada para o construtor Promise sempre recebe um primeiro e um segundo parâmetro normalmente denominados resolve e reject respectivamente. A nomenclatura desses dois parâmetros é convencional, mas eles colocarão a promessa no estado ‘resolvido’ ou no estado ‘rejeitado’. Quando qualquer um deles é chamado, a promessa passa de ‘pendente’ para ‘resolvida’. resolve é chamado quando a ação desejada, que geralmente é assíncrona, foi executada e reject é usado se a ação tiver um erro.

No timeout abaixo está uma função que retorna uma Promise.

function timeout (ms) {
  return new Promise(function (resolve, reject) {
    setTimeout(function () {
      resolve("It was resolved!");
    }, ms)
  });
}

timeout(1000).then(function (dataFromPromise) {
  // logs "It was resolved!"
  console.log(dataFromPromise);
})

console.log("waiting...");

saída do console

waiting...
// << pauses for one second>>
It was resolved!

Quando o timeout é chamado, a função passada para o construtor Promise é executada sem demora. Em seguida, o método setTimeout é executado e seu retorno de chamada é definido para disparar nos próximos ms milissegundos, neste caso ms=1000. Como o retorno de chamada para setTimeout ainda não foi acionado, a função timeout retorna o controle ao escopo de chamada. A cadeia de métodos then é então armazenada para ser chamada mais tarde quando/se a Promise for resolvida. Se houvesse métodos ‘catch’ aqui, eles também seriam armazenados, mas seriam disparados quando/se a promessa ‘rejeitar’.

O script então imprime ‘waiting…’. Um segundo depois, o setTimeout chama seu retorno de chamada que chama a função resolve com a string “Foi resolvido!”. Essa string é então passada para o retorno de chamada do método then e é então registrada para o usuário.

No mesmo sentido, você pode agrupar a função assíncrona setTimeout que requer um retorno de chamada, você pode agrupar qualquer ação assíncrona singular com uma promessa.

Leia mais sobre promessas na documentação JavaScript https://www.wikiod.com/pt/javascript/promessas

Exemplo de código #

Pergunta: Qual é a saída do código abaixo e por quê?

setTimeout(function() {
    console.log("A");
}, 1000);

setTimeout(function() {
    console.log("B");
}, 0);

getDataFromDatabase(function(err, data) {
    console.log("C");
    setTimeout(function() {
        console.log("D");
    }, 1000);
});

console.log("E");

Saída: Isso é conhecido com certeza: EBAD. C é desconhecido quando será registrado.

Explicação: O compilador não irá parar nos métodos setTimeout e getDataFromDatabase. Portanto, a primeira linha que ele registrará é E. As funções de retorno de chamada (primeiro argumento de setTimeout) serão executadas após o tempo limite definido de forma assíncrona!

Mais detalhes:

  1. E não tem setTimeout
  2. B tem um tempo limite definido de 0 milissegundos
  3. “A” tem um tempo limite definido de 1.000 milissegundos
  4. D deve solicitar um banco de dados, depois deve D esperar 1000 milissegundos para que venha depois de A.
  5. C é desconhecido porque é desconhecido quando os dados do banco de dados são solicitados. Pode ser antes ou depois de ‘A’.

Tratamento de erros assíncronos #

Tente pegar #

Os erros devem sempre ser tratados. Se você estiver usando programação síncrona, você pode usar um try catch. Mas isso não funciona se você trabalha assíncrono! Exemplo:

try {
    setTimeout(function() {
        throw new Error("I'm an uncaught error and will stop the server!");
    }, 100); 
}
catch (ex) {
    console.error("This error will not be work in an asynchronous situation: " + ex);
}

Erros assíncronos serão tratados apenas dentro da função de retorno de chamada!

Possibilidades de trabalho #

Manipuladores de eventos #

As primeiras versões do Node.JS tinham um manipulador de eventos.

process.on("UncaughtException", function(err, data) { 
    if (err) {
        // error handling
    }
});

Domínios #

Dentro de um domínio, os erros são liberados através dos emissores de eventos. Ao usar isso são todos os erros, temporizadores, métodos de retorno de chamada implicitamente registrados apenas dentro do domínio. Por um erro, ser um evento de erro enviado e não travar a aplicação.

var domain = require("domain");
var d1 = domain.create();
var d2 = domain.create();

d1.run(function() {
    d2.add(setTimeout(function() {
        throw new Error("error on the timer of domain 2");
    }, 0));
});

d1.on("error", function(err) {
    console.log("error at domain 1: " + err);
});

d2.on("error", function(err) {
    console.log("error at domain 2: " + err);
});
