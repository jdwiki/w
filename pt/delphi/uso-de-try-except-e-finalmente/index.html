Sintaxe #

  1. Try-except: try [statements] except [[[on E:ExceptionType do statement]] [else statement] | [statements] end;

    Try-finally: try [statements] finally [statements] end;

Retorno seguro de exceção de um novo objeto #

Quando uma função retorna um objeto (ao invés de usar um que é passado pelo chamador), tome cuidado para que uma exceção não faça com que o objeto vaze.

function MakeStrings: TStrings;
begin
  // Create a new object before entering the try-block.
  Result := TStringList.Create;
  try
    // Execute code that uses the new object and prepares it for the caller.
    Result.Add('One');
    MightThrow;
  except
    // If execution reaches this point, then an exception has occurred. We cannot
    // know how to handle all possible exceptions, so we merely clean up the resources
    // allocated by this function and then re-raise the exception so the caller can
    // choose what to do with it.
    Result.Free;
    raise;
  end;
  // If execution reaches this point, then no exception has occurred, so the
  // function will return Result normally.
end;

Programadores ingênuos podem tentar capturar todos os tipos de exceção e retornar nil de tal função, mas esse é apenas um caso especial da prática geral desencorajada de capturar todos os tipos de exceção sem manipulá-los.

Try-except aninhado dentro de try-finally #

Um bloco try-except pode ser aninhado dentro de um bloco try-finally.

AcquireResource;
try
  UseResource1;
  try
    UseResource2;
  except
    on E: EResourceUsageError do begin
      HandleResourceErrors;
    end;
  end;
  UseResource3;
finally
  ReleaseResource;
end;

Se um EResourceUsageError ocorrer em UseResource2, a execução pulará para o manipulador de exceção e chamará HandleResourceError. A exceção será considerada tratada, então a execução continuará para UseResource3 e depois para ReleaseResource.

Se uma exceção de qualquer outro tipo ocorrer em UseResource2, então o manipulador de exceção mostrado aqui não será aplicado, então a execução pulará sobre a chamada UseResource3 e irá diretamente para o bloco finally, onde ReleaseResource será chamado . Depois disso, a execução pulará para o próximo manipulador de exceção aplicável à medida que a exceção borbulhar na pilha de chamadas.

Se ocorrer uma exceção em qualquer outra chamada no exemplo acima, HandleResourceErrors não será chamado. Isso ocorre porque nenhuma das outras chamadas ocorre dentro do bloco try correspondente a esse manipulador de exceção.

Simples try..finally exemplo para evitar vazamentos de memória #

Use try-finally para evitar vazamento de recursos (como memória) caso ocorra uma exceção durante a execução.

O procedimento abaixo salva uma string em um arquivo e evita que o TStringList vaze.

procedure SaveStringToFile(const aFilename: TFilename; const aString: string);
var
  SL: TStringList;
begin
  SL := TStringList.Create; // call outside the try 
  try
    SL.Text := aString;
    SL.SaveToFile(aFilename);
  finally
    SL.Free // will be called no matter what happens above
  end;
end;

Independentemente de ocorrer uma exceção ao salvar o arquivo, o SL será liberado. Qualquer exceção irá para o chamador.

Try-finally com 2 ou mais objetos #

Object1 := nil;
Object2 := nil;
try
  Object1 := TMyObject.Create;
  Object2 := TMyObject.Create;
finally
  Object1.Free;
  Object2.Free;
end;

Se você não inicializar os objetos com nil fora do bloco try-finally, se um deles não for criado um AV irá ocorrer no bloco finally, pois o objeto não será nil (como era’ t inicializado) e causará uma exceção.

O método Free verifica se o objeto é nil, então inicializar ambos os objetos com nil evita erros ao liberá-los caso não tenham sido criados.

Try-finally aninhado dentro de try-except #

Um bloco try-finally pode ser aninhado dentro de um bloco try-except.

try
  AcquireResources;
  try
    UseResource;
  finally
    ReleaseResource;
  end;
except
  on E: EResourceUsageError do begin
    HandleResourceErrors;
  end;
end;

Se ocorrer uma exceção dentro de UseResource, a execução irá pular para ReleaseResource. Se a exceção for um EResourceUsageError, a execução pulará para o manipulador de exceção e chamará HandleResourceErrors. Exceções de qualquer outro tipo irão pular o manipulador de exceções acima e vão para o próximo bloco try-except na pilha de chamadas.

Exceções em AcquireResource ou ReleaseResource farão com que a execução vá para o manipulador de exceção, pulando o bloco finally, seja porque o bloco try correspondente ainda não foi inserido ou porque o bloco finally já foi inserido.
