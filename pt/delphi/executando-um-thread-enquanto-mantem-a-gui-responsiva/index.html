GUI responsiva usando threads para trabalho em segundo plano e PostMessage para relatar os threads #

Manter uma GUI responsiva durante a execução de um processo demorado requer alguns “retornos de chamada” muito elaborados para permitir que a GUI processe sua fila de mensagens ou o uso de threads (de segundo plano) (de trabalho).

Iniciar qualquer número de threads para fazer algum trabalho geralmente não é um problema. A diversão começa quando você deseja que a GUI mostre resultados intermediários e finais ou relate o progresso.

Mostrar qualquer coisa na GUI requer interagir com os controles e/ou a fila/bomba de mensagens. Isso deve sempre ser feito no contexto do thread principal. Nunca no contexto de qualquer outro tópico.

Há muitas maneiras de lidar com isso.

Este exemplo mostra como você pode fazer isso usando threads simples, permitindo que a GUI acesse a instância do thread depois que ela for concluída, definindo FreeOnTerminate como false e ​​relatando quando um thread está “concluído” usando PostMessage.

Notas sobre as condições de corrida: As referências aos threads de trabalho são mantidas em uma matriz no formulário. Quando um thread é finalizado, a referência correspondente no array é nil-ed.

Esta é uma fonte potencial de condições de corrida. Assim como o uso de um booleano “Running” para facilitar a determinação se ainda existem threads que precisam ser finalizadas.

Você precisará decidir se precisa proteger esses recursos usando bloqueios ou não.

Neste exemplo, como está, não há necessidade. Eles são modificados apenas em dois locais: o método StartThreads e o método HandleThreadResults. Ambos os métodos são executados apenas no contexto do thread principal. Contanto que você mantenha assim e não comece a chamar esses métodos do contexto de diferentes threads, não há como eles produzirem condições de corrida.

Fio #

type
  TWorker = class(TThread)
  private
    FFactor: Double;
    FResult: Double;
    FReportTo: THandle;
  protected
    procedure Execute; override;
  public
    constructor Create(const aFactor: Double; const aReportTo: THandle);

    property Factor: Double read FFactor;
    property Result: Double read FResult;
  end;

O construtor apenas define os membros privados e define FreeOnTerminate como False. Isso é essencial, pois permitirá que o encadeamento principal consulte a instância do encadeamento para obter seu resultado.

O método execute faz seu cálculo e, em seguida, envia uma mensagem para o handle que recebeu em seu construtor para dizer que está pronto:

procedure TWorker.Execute;
const
  Max = 100000000;var
  i : Integer;
begin
  inherited;

  FResult := FFactor;
  for i := 1 to Max do
    FResult := Sqrt(FResult);

  PostMessage(FReportTo, UM_WORKERDONE, Self.Handle, 0);
end;

O uso de PostMessage é essencial neste exemplo. PostMessage “apenas” coloca uma mensagem na fila da bomba de mensagens da thread principal e não espera que ela seja tratada. É de natureza assíncrona. Se você usasse SendMessage, estaria se codificando em um picles. SendMessage coloca a mensagem na fila e espera até que ela seja processada. Em suma, é síncrono.

As declarações para a mensagem UM_WORKERDONE personalizada são declaradas como:

const
  UM_WORKERDONE = WM_APP + 1;
type
  TUMWorkerDone = packed record
    Msg: Cardinal;
    ThreadHandle: Integer;
    unused: Integer;
    Result: LRESULT;
  end;

A const UM_WORKERDONE usa WM_APP como ponto de partida para seu valor para garantir que não interfira em nenhum valor usado pelo Windows ou pelo Delphi VCL (como recomendado pela MicroSoft).

Forma #

Qualquer formulário pode ser usado para iniciar threads. Tudo o que você precisa fazer é adicionar os seguintes membros a ele:

private
  FRunning: Boolean;
  FThreads: array of record
    Instance: TThread;
    Handle: THandle;
  end;
  procedure StartThreads(const aNumber: Integer);
  procedure HandleThreadResult(var Message: TUMWorkerDone); message UM_WORKERDONE;

Ah, e o código de exemplo assume a existência de um Memo1: TMemo; nas declarações do formulário, que ele usa para “registrar e relatar”.

O FRunning pode ser usado para evitar que a GUI comece a ser clicada enquanto o trabalho está em andamento. FThreads é usado para manter o ponteiro de instância e o handle das threads criadas.

O procedimento para iniciar as threads tem uma implementação bastante direta. Ele começa com uma verificação se já existe um conjunto de threads em espera. Se sim, ele simplesmente sai. Caso contrário, ele define o sinalizador como verdadeiro e inicia os encadeamentos fornecendo a cada um seu próprio identificador para que eles saibam onde postar sua mensagem “concluído”.

procedure TForm1.StartThreads(const aNumber: Integer);
var
  i: Integer;
begin
  if FRunning then
    Exit;
    
  FRunning := True;

  Memo1.Lines.Add(Format('Starting %d worker threads', [aNumber]));
  SetLength(FThreads, aNumber);
  for i := 0 to aNumber - 1 do
  begin
    FThreads[i].Instance := TWorker.Create(pi * (i+1), Self.Handle);
    FThreads[i].Handle := FThreads[i].Instance.Handle;
  end;
end;

O handle do thread também é colocado no array porque é isso que recebemos nas mensagens que nos dizem que um thread está pronto e tê-lo fora da instância do thread facilita um pouco o acesso. Ter o handle disponível fora da instância do thread também nos permite usar FreeOnTerminate definido como True se não precisarmos da instância para obter seus resultados (por exemplo, se eles foram armazenados em um banco de dados). Nesse caso, é claro que não haveria necessidade de manter uma referência à instância.

A diversão está na implementação HandleThreadResult:

procedure TForm1.HandleThreadResult(var Message: TUMWorkerDone);
var
  i: Integer;
  ThreadIdx: Integer;
  Thread: TWorker;
  Done: Boolean;
begin
  // Find thread in array
  ThreadIdx := -1;
  for i := Low(FThreads) to High(FThreads) do
    if FThreads[i].Handle = Cardinal(Message.ThreadHandle) then
    begin
      ThreadIdx := i;
      Break;
    end;

  // Report results and free the thread, nilling its pointer and handle 
  // so we can detect when all threads are done.
  if ThreadIdx > -1 then
  begin
    Thread := TWorker(FThreads[i].Instance);
    Memo1.Lines.Add(Format('Thread %d returned %f', [ThreadIdx, Thread.Result]));
    FreeAndNil(FThreads[i].Instance);
    FThreads[i].Handle := nil;
  end;

  // See whether all threads have finished.
  Done := True;
  for i := Low(FThreads) to High(FThreads) do
    if Assigned(FThreads[i].Instance) then
    begin
      Done := False;
      Break;
    end;
  if Done then
  begin
    Memo1.Lines.Add('Work done');
    FRunning := False;
  end;
end;

Esse método primeiro pesquisa o encadeamento usando o identificador recebido na mensagem. Se uma correspondência foi encontrada, ele recupera e relata o resultado da thread usando a instância (FreeOnTerminate era False, lembra?), e então termina: liberando a instância e definindo a referência da instância e o handle para nil, indicando este tópico não é mais relevante.

Finalmente, ele verifica se algum dos threads ainda está em execução. Se nenhum for encontrado, “all done” é relatado e o sinalizador FRunning definido como False para que um novo lote de trabalho possa ser iniciado.
