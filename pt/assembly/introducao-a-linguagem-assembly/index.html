Introdução #

A linguagem de montagem é uma forma legível por humanos de linguagem de máquina ou código de máquina que é a sequência real de bits e bytes na qual a lógica do processador opera. Geralmente é mais fácil para os humanos ler e programar em mnemônicos do que binário, octal ou hexadecimal, então os humanos normalmente escrevem código em linguagem assembly e depois usam um ou mais programas para convertê-lo no formato de linguagem de máquina entendido pelo processador.

EXEMPLO:

mov eax, 4
cmp eax, 5
je point

Um montador é um programa que lê o programa em linguagem assembly, analisa-o e produz a linguagem de máquina correspondente. É importante entender que, diferentemente de uma linguagem como C++, que é uma linguagem única definida no documento padrão, existem muitas linguagens assembly diferentes. Cada arquitetura de processador, ARM, MIPS, x86, etc, tem um código de máquina diferente e, portanto, uma linguagem de montagem diferente. Além disso, às vezes há várias linguagens de montagem diferentes para a mesma arquitetura de processador. Em particular, a família de processadores x86 tem dois formatos populares que são frequentemente referidos como sintaxe de gás (gas é o nome do executável para o GNU Assembler) e sintaxe Intel (em homenagem ao originador da família de processadores x86). Eles são diferentes, mas equivalentes, pois normalmente se pode escrever qualquer programa em qualquer sintaxe.

Geralmente, o inventor do processador documenta o processador e seu código de máquina e cria uma linguagem assembly. É comum que essa linguagem assembly específica seja a única usada, mas ao contrário dos escritores de compiladores que tentam se adequar a um padrão de linguagem, a linguagem assembly definida pelo inventor do processador é geralmente, mas nem sempre, a versão usada pelas pessoas que escrevem montadores .

Existem dois tipos gerais de processadores:

  • CISC (Complex Instruction Set Computer): tem muitas instruções de linguagem de máquina diferentes e muitas vezes complexas

  • RISC (Reduced Instruction Set Computers): por outro lado, tem menos e mais simples instruções

Para um programador em linguagem assembly, a diferença é que um processador CISC pode ter muitas instruções para aprender, mas geralmente há instruções adequadas para uma tarefa específica, enquanto os processadores RISC têm menos instruções e mais simples, mas qualquer operação pode exigir o programador em linguagem assembly. escrever mais instruções para fazer a mesma coisa.

Outros compiladores de linguagens de programação às vezes produzem o montador primeiro, que é então compilado em código de máquina chamando um montador. Por exemplo, gcc usando seu próprio gas assembler no estágio final de compilação. O código de máquina produzido geralmente é armazenado em arquivos objetos, que podem ser vinculados a executáveis ​​pelo programa vinculador.

Uma “cadeia de ferramentas” completa geralmente consiste em um compilador, montador e vinculador. Pode-se então usar esse montador e vinculador diretamente para escrever programas em linguagem assembly. No mundo GNU o pacote binutils contém o montador e o linker e ferramentas relacionadas; aqueles que estão apenas interessados ​​em programação em linguagem assembly não precisam do gcc ou de outros pacotes de compilador.

Pequenos microcontroladores são frequentemente programados puramente em linguagem assembly ou em uma combinação de linguagem assembly e uma ou mais linguagens de nível superior, como C ou C++. Isso é feito porque muitas vezes é possível usar os aspectos particulares da arquitetura do conjunto de instruções para que esses dispositivos escrevam um código mais compacto e eficiente do que seria possível em uma linguagem de nível superior e esses dispositivos geralmente têm memória e registros limitados. Muitos microprocessadores são usados ​​em sistemas embarcados, que são dispositivos que não são computadores de uso geral que possuem um microprocessador interno. Exemplos de tais sistemas embarcados são televisores, fornos de micro-ondas e a unidade de controle do motor de um automóvel moderno. Muitos desses dispositivos não têm teclado ou tela, então um programador geralmente escreve o programa em um computador de uso geral, executa um cross-assembler (assim chamado porque esse tipo de montador produz código para um tipo de processador diferente daquele em que é executado) e/ou um compilador cruzado e um linkador cruzado para produzir código de máquina.

Existem muitos fornecedores para essas ferramentas, que são tão variados quanto os processadores para os quais eles produzem código. Muitos, mas nem todos os processadores também possuem uma solução de código aberto como GNU, sdcc, llvm ou outros.

Olá mundo para Linux x86_64 (Intel 64 bits) #

section .data
    msg db "Hello world!",10      ; 10 is the ASCII code for a new line (LF)

section .text
    global _start

_start:
    mov rax, 1
    mov rdi, 1
    mov rsi, msg
    mov rdx, 13
    syscall
    
    mov rax, 60
    mov rdi, 0
    syscall

Se você deseja executar este programa, primeiro você precisa do Netwide Assembler, nasm, porque este código usa sua sintaxe. Em seguida, use os seguintes comandos (assumindo que o código está no arquivo helloworld.asm). Eles são necessários para montar, vincular e executar, respectivamente.

  • nasm -felf64 helloworld.asm
  • ld helloworld.o -o helloworld
  • ./helloworld

O código faz uso do syscall sys_write do Linux. Aqui você pode ver uma lista de todas as syscalls para o x86_64 arquitetura. Quando você também acessa as páginas man de write e exit em conta, você pode traduzir o programa acima em um C que faz o mesmo e é muito mais legível:

#include <unistd.h>

#define STDOUT 1

int main()
{
    write(STDOUT, "Hello world!\n", 13);
    _exit(0);
}

Apenas dois comandos são necessários aqui para compilação e vinculação (o primeiro) e execução:

  • gcc helloworld_c.c -o helloworld_c.
  • ./helloworld_c

Código da máquina #

Código de máquina é termo para os dados em formato de máquina nativo específico, que são processados ​​diretamente pela máquina - geralmente pelo processador chamado CPU (Unidade Central de Processamento).

A arquitetura de computador comum (von Neumann architecture) consiste em processador de uso geral (CPU), memória de uso geral - armazenando tanto o programa (ROM/RAM) quanto os dados processados ​​e dispositivos de entrada e saída (dispositivos de E/S).

A principal vantagem desta arquitetura é a relativa simplicidade e universalidade de cada um dos componentes - quando comparado a máquinas de computador anteriores (com programa hard-wired na construção da máquina), ou arquiteturas concorrentes (por exemplo a arquitetura de Harvard separando a memória do programa da memória do dados). A desvantagem é um desempenho geral um pouco pior. No longo prazo, a universalidade permitiu o uso flexível, que geralmente superava o custo de desempenho.

Como isso se relaciona com o código de máquina?

Programa e dados são armazenados nesses computadores como números, na memória. Não há uma maneira genuína de diferenciar código de dados, então os sistemas operacionais e operadores de máquina dão dicas à CPU, em qual ponto de entrada da memória inicia o programa, depois de carregar todos os números na memória. A CPU então lê a instrução (número) armazenada no ponto de entrada e a processa rigorosamente, lendo sequencialmente os próximos números como instruções adicionais, a menos que o próprio programa diga à CPU para continuar com a execução em outro lugar.

Por exemplo, dois números de 8 bits (8 bits agrupados são iguais a 1 byte, que é um número inteiro sem sinal dentro do intervalo de 0-255): 60 201, quando executado como código na CPU do Zilog Z80 será processado como dois instruções: INC a (incrementando o valor no registrador a em um) e RET (retornando da sub-rotina, apontando a CPU para executar instruções de diferentes partes da memória).

Para definir este programa, um humano pode inserir esses números por algum editor de memória/arquivo, por exemplo, no editor hexadecimal como dois bytes: 3C C9 (números decimais 60 e 201 escritos em codificação de base 16). Isso seria programar em código de máquina.

Para facilitar a tarefa de programação da CPU para humanos, foram criados programas Assembler, capazes de ler arquivos de texto contendo algo como:

subroutineIncrementA:
    INC   a
    RET

dataValueDefinedInAssemblerSource:
    DB    60          ; define byte with value 60 right after the ret

seqüência de números hexadecimais de byte de saída 3C C9 3C, envolvida com números adicionais opcionais específicos para a plataforma de destino: marcando qual parte de tal binário é código executável, onde é o ponto de entrada para o programa (a primeira instrução dele), quais partes são dados codificados (não executáveis), etc.

Observe como o programador especificou o último byte com valor 60 como “dados”, mas do ponto de vista da CPU não difere em nada do byte INC a. Cabe ao programa em execução navegar corretamente pela CPU sobre os bytes preparados como instruções e processar os bytes de dados apenas como dados para instruções.

Tal saída geralmente é armazenada em um arquivo no dispositivo de armazenamento, carregado posteriormente pelo SO (Operating System - um código de máquina já rodando no computador, ajudando a manipular com o computador) na memória antes de executá-lo, e por fim apontando a CPU no ponto de entrada do programa.

A CPU pode processar e executar apenas código de máquina - mas qualquer conteúdo de memória, mesmo aleatório, pode ser processado como tal, embora o resultado possa ser aleatório, variando de “crash” detectado e tratado pelo SO até limpeza acidental de dados de Dispositivos de E/S ou danos em equipamentos sensíveis conectados ao computador (não é um caso comum para computadores domésticos :) ).

O processo semelhante é seguido por muitas outras linguagens de programação de alto nível, compilando o fonte (forma de texto legível por humanos do programa) em números, representando o código de máquina (instruções nativas da CPU) ou no caso de código interpretado/híbrido linguagens em algum código de máquina virtual específico de linguagem geral, que é ainda decodificado em código de máquina nativo durante a execução pelo interpretador ou máquina virtual.

Alguns compiladores usam o Assembler como estágio intermediário de compilação, traduzindo a fonte primeiro para o formato Assembler, depois executando a ferramenta assembler para obter o código de máquina final dele (exemplo GCC: execute gcc -S helloworld.c para obter uma versão assembler do programa C helloworld.c).

Hello World para OS X (x86_64, gás de sintaxe Intel) #

.intel_syntax noprefix

.data

.align 16
hello_msg:
    .asciz "Hello, World!"

.text

.global _main
_main:
    push rbp
    mov rbp, rsp

    lea rdi, [rip+hello_msg]
    call _puts

    xor rax, rax
    leave
    ret

Montar:

clang main.s -o hello
./hello

Notas:

  • O uso de chamadas de sistema é desencorajado, pois a API de chamada de sistema no OS X não é considerada estável. Em vez disso, use a biblioteca C. (Referência a uma pergunta do Stack Overflow)
  • A Intel recomenda que estruturas maiores que uma palavra comecem em um limite de 16 bytes. (Referência à documentação da Intel)
  • Os dados da ordem são passados ​​para funções através dos registradores: rdi, rsi, rdx, rcx, r8 e r9. (Referência ao System V ABI)

Executando o assembly x86 no Visual Studio 2015 #

Etapa 1: Crie um projeto vazio via Arquivo -> Novo Projeto.

Etapa 2: Clique com o botão direito do mouse na solução do projeto e selecione Build Dependencies->Build Customizations.

Passo 3: Marque a caixa de seleção ".masm".

Passo 4: Pressione o botão “ok”.

Passo 5: Crie seu arquivo de montagem e digite isto:

.386
    .model small
        .code

            public main
                main proc

                    ; Ends program normally

                    ret
                main endp
            end main

Passo 6: Compile!
