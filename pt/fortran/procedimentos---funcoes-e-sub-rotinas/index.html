Procedimentos - Funções e Sub-rotinas

Funções e sub-rotinas, em conjunto com módulos, são as ferramentas para dividir um programa em unidades. Isso torna o programa mais legível e gerenciável. Cada uma dessas unidades pode ser pensada como parte do código que, idealmente, poderia ser compilado e testado isoladamente. O(s) programa(s) principal(is) pode(m) chamar (ou invocar) tais subprogramas (funções ou sub-rotinas) para realizar uma tarefa.

Funções e sub-rotinas são diferentes no seguinte sentido:

  • Funções retornam um único objeto e - normalmente - não alteram os valores de seus argumentos (ou seja, eles agem como uma função matemática!);
  • Sub-rotinas geralmente realizam uma tarefa mais complicada e normalmente alteram seus argumentos (se houver algum), bem como outras variáveis ​​(por exemplo, aquelas declaradas no módulo que contém a sub-rotina).

Funções e sub-rotinas coletivamente são chamadas de procedures. (A seguir, usaremos o verbo “chamar” como sinônimo de “invocar” mesmo que tecnicamente os procedimentos a serem chamados sejam subrotinas, enquanto funções aparecem como lado direito da atribuição ou em expressões .)

Sintaxe da função #

As funções podem ser escritas usando vários tipos de sintaxe

function name()
  integer name
  name = 42
end function

integer function name()
  name = 42
end function

function name() result(res)
  integer res
  res = 42
end function

As funções retornam valores por meio de um function result. A menos que a instrução da função tenha uma cláusula result, o resultado da função tem o mesmo nome da função. Com result o resultado da função é aquele dado pelo result. Em cada um dos dois primeiros exemplos acima o resultado da função é dado por name; no terceiro por res.

O resultado da função deve ser definido durante a execução da função.

As funções permitem usar alguns prefixos especiais.

A função Pure significa que esta função não tem efeito colateral:

pure real function square(x)
  real, intent(in) :: x
  square = x * x
end function

A função Elemental é definida como operador escalar, mas pode ser invocada com array como argumento real, caso em que a função será aplicada elemento a elemento. A menos que o prefixo impure (introduzido no Fortran 2008) seja especificado, uma função elemental também é uma função pure.

elemental real function square(x)
  real, intent(in) :: x
  square = x * x
end function

Declaração de retorno #

A instrução return pode ser usada para sair da função e da sub-rotina. Ao contrário de muitas outras linguagens de programação, não é usado para definir o valor de retorno.

real function f(x)
  real, intent(in) :: x
  integer :: i

  f = x

  do i = 1, 10

    f = sqrt(f) - 1.0

    if (f < 0) then
      f = -1000.
      return
    end if

  end do
end function

Esta função executa uma computação iterativa. Se o valor de f se tornar negativo, a função retornará o valor -1000.

Procedimentos Recursivos #

Em Fortran, funções e sub-rotinas precisam ser explicitamente declaradas como recursivas, se quiserem chamar a si mesmas novamente, direta ou indiretamente. Assim, uma implementação recursiva da série de Fibonacci poderia ser assim:

recursive function fibonacci(term) result(fibo)
  integer, intent(in) :: term
  integer :: fibo

  if (term <= 1) then
    fibo = 1
  else
    fibo = fibonacci(term-1) + fibonacci(term-2)
  end if
  
end function fibonacci

Outro exemplo é permitido para calcular fatorial:

recursive function factorial(n)  result(f)
  integer :: f
  integer, intent(in) :: n
  
  if(n == 0) then
    f = 1
  else
    f = n * f(n-1)
  end if
end function factorial

Para uma função referenciar-se recursivamente diretamente, sua definição deve usar o sufixo result. Não é possível que uma função seja tanto recursiva quanto elemental.

A intenção dos argumentos fictícios #

O atributo intent de um argumento fictício em uma sub-rotina ou função declara seu uso pretendido. A sintaxe é uma das

intent(IN)
intent(OUT)
intent(INOUT)

Por exemplo, considere esta função:

real function f(x)
  real, intent(IN) :: x

  f = x*x
end function

O intent(IN) especifica que o argumento fictício (sem ponteiro) x nunca pode ser definido ou se tornar indefinido ao longo da função ou de sua inicialização. Se um argumento fictício de ponteiro tem o atributo intent(IN), isso se aplica à sua associação.

intent(OUT) para um argumento fictício não ponteiro significa que o argumento fictício se torna indefinido na chamada do subprograma (exceto para quaisquer componentes de um tipo derivado com inicialização padrão) e deve ser definido durante a execução. O argumento real passado como argumento fictício deve ser definível: não é permitido passar uma constante nomeada ou literal, ou uma expressão.

Da mesma forma que antes, se um argumento fictício de ponteiro for intent(OUT), o status de associação do ponteiro se tornará indefinido. O argumento real aqui deve ser uma variável de ponteiro.

intent(INOUT) especifica que o argumento real é definível e adequado tanto para passar como para retornar dados do procedimento.

Finalmente, um argumento fictício pode estar sem o atributo intent. Tal argumento fictício tem seu uso limitado pelo argumento real passado.

Por exemplo, considere

integer :: i = 0
call sub(i, .TRUE.)
call sub(1, .FALSE.)

end

subroutine sub(i, update)
  integer i
  logical, intent(in) :: update
  if (update) i = i+1
end subroutine

O argumento i não pode ter o atributo intent que permite ambas as chamadas de subrotinas do programa principal.

Referenciando um procedimento #

Para que uma função ou sub-rotina seja útil, ela deve ser referenciada. Uma sub-rotina é referenciada em uma instrução call

call sub(...)

e uma função dentro de uma expressão. Ao contrário de muitas outras linguagens, uma expressão não forma uma instrução completa, portanto, uma referência de função geralmente é vista em uma instrução de atribuição ou usada de alguma outra maneira:

x = func(...)
y = 1 + 2*func(...)

Existem três maneiras de designar um procedimento que está sendo referenciado:

  • como o nome de um procedimento ou ponteiro de procedimento
  • um componente de procedimento de um objeto de tipo derivado
  • um nome de ligação de procedimento de tipo

A primeira pode ser vista como

procedure(), pointer :: sub_ptr=>sub
call sub()   ! With no argument list the parentheses are optional
call sub_ptr()
end

subroutine sub()
end subroutine

e os dois finais como

module mod
  type t
    procedure(sub), pointer, nopass :: sub_ptr=>sub
  contains
    procedure, nopass :: sub
  end type

contains

  subroutine sub()
  end subroutine

end module

use mod
type(t) x
call x%sub_ptr()   ! Procedure component
call x%sub()       ! Binding name

end

Para um procedimento com argumentos fictícios, a referência requer argumentos actual correspondentes, embora argumentos fictícios opcionais possam não ser fornecidos.

Considere a sub-rotina

subroutine sub(a, b, c)
  integer a, b
  integer, optional :: c
end subroutine

Isso pode ser referenciado das duas maneiras a seguir

call sub(1, 2, 3)   ! Passing to the optional dummy c
call sub(1, 2)      ! Not passing to the optional dummy c

Isso é chamado de referência posicional: os argumentos reais são associados com base na posição nas listas de argumentos. Aqui, o fictício a é associado a 1, b a 2 e c (quando especificado) a 3.

Alternativamente, a referência keyword pode ser usada quando o procedimento tem uma interface explícita disponível

call sub(a=1, b=2, c=3)
call sub(a=1, b=2)

que é o mesmo que o anterior.

No entanto, com palavras-chave, os argumentos reais podem ser oferecidos em qualquer ordem

call sub(b=2, c=3, a=1)
call sub(b=2, a=1)

As referências posicionais e de palavras-chave podem ser usadas

call sub(1, c=3, b=2)

contanto que uma palavra-chave seja fornecida para cada argumento após a primeira aparição de uma palavra-chave

call sub(b=2, 1, 3)  ! Not valid: all keywords must be specified

O valor da referência de palavra-chave é particularmente pronunciado quando há vários argumentos fictícios opcionais, como visto abaixo se na definição de sub-rotina acima b também fosse opcional

call sub(1, c=3)  ! Optional b is not passed

As listas de argumentos para procedimentos vinculados a tipo ou ponteiros de procedimento de componente com um argumento passado são considerados separadamente.
