Adaptador (Java) #

Vamos supor que em sua base de código atual, exista a interface MyLogger assim:

interface MyLogger {
    void logMessage(String message);
    void logException(Throwable exception);
}

Digamos que você criou algumas implementações concretas deles, como MyFileLogger e MyConsoleLogger.

Você decidiu que deseja usar uma estrutura para controlar a conectividade Bluetooth do seu aplicativo. Este framework contém um BluetoothManager com o seguinte construtor:

class BluetoothManager {
    private FrameworkLogger logger;

    public BluetoothManager(FrameworkLogger logger) {
        this.logger = logger;
    }
}

O BluetoothManager também aceita um registrador, o que é ótimo! No entanto, ele espera um logger cuja interface foi definida pelo framework e eles usaram sobrecarga de métodos em vez de nomear suas funções de maneira diferente:

interface FrameworkLogger {
    void log(String message);
    void log(Throwable exception);
}

Você já tem várias implementações do MyLogger que gostaria de reutilizar, mas elas não se encaixam na interface do FrameworkLogger. É aqui que entra o padrão de design do adaptador:

class FrameworkLoggerAdapter implements FrameworkLogger {
    private MyLogger logger;

    public FrameworkLoggerAdapter(MyLogger logger) {
        this.logger = logger;
    }

    @Override
    public void log(String message) {
        this.logger.logMessage(message);
    }

    @Override
    public void log(Throwable exception) {
        this.logger.logException(exception);
    }
}

Ao definir uma classe de adaptador que implementa a interface FrameworkLogger e aceita uma implementação MyLogger, a funcionalidade pode ser mapeada entre as diferentes interfaces. Agora é possível usar o BluetoothManager com todas as implementações do MyLogger assim:

FrameworkLogger fileLogger = new FrameworkLoggerAdapter(new MyFileLogger());
BluetoothManager manager = new BluetoothManager(fileLogger);

FrameworkLogger consoleLogger = new FrameworkLoggerAdapter(new MyConsoleLogger());
BluetoothManager manager2 = new BluetoothManager(consoleLogger);

Exemplo Java #

Um ótimo exemplo existente do padrão Adapter pode ser encontrado no SWT [MouseListener](http://help.eclipse.org/kepler/index.jsp?topic=%2Forg.eclipse.platform.doc.isv%2Freference%2Fapi %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseListener.html) e [MouseAdapter](http://help.eclipse.org/kepler/index.jsp?topic=%2Forg.eclipse.platform.doc.isv%2Freference%2Fapi %2Forg%2Feclipse%2Fswt%2Fevents%2FMouseAdapter.html).

A interface MouseListener tem a seguinte aparência:

public interface MouseListener extends SWTEventListener {
    public void mouseDoubleClick(MouseEvent e);
    public void mouseDown(MouseEvent e);
    public void mouseUp(MouseEvent e);
}

Agora imagine um cenário em que você está construindo uma interface do usuário e adicionando esses ouvintes, mas na maioria das vezes você não se importa com nada além de quando algo é clicado uma vez (mouseUp). Você não gostaria de estar constantemente criando implementações vazias:

obj.addMouseListener(new MouseListener() {

    @Override
    public void mouseDoubleClick(MouseEvent e) {
    }

    @Override
    public void mouseDown(MouseEvent e) {
    }

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Em vez disso, podemos usar MouseAdapter:

public abstract class MouseAdapter implements MouseListener {
    public void mouseDoubleClick(MouseEvent e) { }
    public void mouseDown(MouseEvent e) { }
    public void mouseUp(MouseEvent e) { }
}

Ao fornecer implementações padrão vazias, estamos livres para substituir apenas os métodos que nos interessam do adaptador. Seguindo o exemplo acima:

obj.addMouseListener(new MouseAdapter() {

    @Override
    public void mouseUp(MouseEvent e) {
        // Do the things
    }

});

Adaptador (UML e situação de exemplo) #

Para tornar mais imaginável o uso do padrão adaptador e o tipo de situação em que ele pode ser aplicado, um exemplo pequeno, simples e muito concreto é dado aqui. Não haverá código aqui, apenas UML e uma descrição da situação de exemplo e seu problema. É certo que o conteúdo UML é escrito como Java. (Bem, o texto da dica dizia “Bons exemplos são principalmente código”, acho que os padrões de design são abstratos o suficiente para serem introduzidos de uma maneira diferente também.)

Em geral, o padrão do adaptador é uma solução adequada para uma situação em que você tem interfaces incompatíveis e nenhuma delas pode ser reescrita diretamente.

Imagine que você está administrando um bom serviço de entrega de pizza. Os clientes podem fazer pedidos online em seu site e você tem um pequeno sistema usando uma classe Pizza para representar suas pizzas e calcular contas, relatórios fiscais e muito mais. O preço de suas pizzas é dado como um único inteiro representando o preço em centavos (da moeda de sua escolha).

Seu serviço de entrega está funcionando muito bem, mas em algum momento você não pode mais lidar com o número crescente de clientes por conta própria, mas ainda deseja expandir. Você decide adicionar suas pizzas ao cardápio de um grande serviço de meta entrega online. Eles oferecem muitas refeições diferentes - não apenas pizzas - então seu sistema faz mais uso de abstração e tem uma interface IMeal representando refeições junto com uma classe MoneyAmount representando dinheiro.

MoneyAmount consiste em dois inteiros como entrada, um para o valor (ou alguma moeda aleatória) antes da vírgula e outro para o valor em centavos de 0 a 99 após a vírgula;

Devido ao fato de que o preço da sua Pizza é um único inteiro representando o preço total como uma quantia de centavos (> 99), ela não é compatível com IMeal. Este é o ponto em que o padrão do adaptador entra em ação: Caso seja necessário muito esforço para alterar seu próprio sistema ou criar um novo e você tiver que implementar uma interface incompatível, convém aplicar o padrão do adaptador.

Existem duas maneiras de aplicar o padrão: adaptador de classe e adaptador de objeto.

Ambos têm em comum que um adaptador (PizzaAdapter) funciona como uma espécie de tradutor entre a nova interface e o adaptador (Pizza neste exemplo). O adaptador implementa a nova interface (IMeal) e então herda de Pizza e converte seu próprio preço de um inteiro para dois (adaptador de classe)

ou tem um objeto do tipo Pizza como atributo e converte os valores dele (adaptador de objetos).

Ao aplicar o padrão do adaptador, você meio que “traduz” entre interfaces incompatíveis.
