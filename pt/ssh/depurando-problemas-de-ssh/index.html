Ligação recusada #

Um erro “Conexão recusada” ocorrerá se o cliente enviar uma solicitação de conexão para um host de servidor remoto e o host remoto responder dizendo que se recusa a aceitar a solicitação. O erro “Conexão recusada” significa essencialmente que o computador não está aceitando conexões com o endereço IP e a porta solicitados.

“Conexão recusada” pode ser causada por um firewall que está bloqueando solicitações de conexão. Um firewall configurado para bloquear conexões com um determinado ponto de extremidade pode ser configurado para descartar solicitações de conexão - nesse caso, o cliente nunca obterá uma resposta e, eventualmente, atingirá o tempo limite. Ou o firewall pode responder a solicitações de conexão de conexão com uma resposta de recusa.

Além dos firewalls, no caso do SSH, “conexão recusada” tem algumas causas possíveis:

  • Você pode estar usando o número de porta errado para se conectar. O número de porta padrão para SSH é 22, mas algumas pessoas executam o serviço ssh em uma porta diferente para impedir tentativas de acesso não autorizado.

  • Você pode estar tentando se conectar ao computador errado. Você pode ter digitado incorretamente o nome do host ou o endereço IP. Ou o computador pode estar usando um endereço atribuído dinamicamente que foi alterado.

  • O processo do servidor ssh pode não estar em execução:

  • Pode não ter sido iniciado ainda se o sistema estiver em processo de inicialização.

  • Pode ter sido desativado; por exemplo. quando o sistema está no modo de usuário único.

  • Ele pode ter sido configurado incorretamente, causando uma falha na inicialização.

  • O computador pode não ter um servidor SSH configurado. Os sistemas MS Windows normalmente não incluem um servidor SSH. Em alguns sistemas Linux, o servidor SSH pode ser um componente opcional. O OS X inclui um servidor SSH, mas está desabilitado por padrão.

  • O processo do servidor SSH pode não estar ouvindo conexões na interface IP específica à qual você está tentando se conectar. A maioria dos computadores tem pelo menos duas interfaces IP, uma interface “localhost” e uma ou mais interfaces de rede. Cada interface ativa terá um endereço IP associado a ela. Um servidor SSH normalmente é configurado para aceitar conexões em qualquer interface IP. Mas ele pode ser configurado para aceitar conexões apenas em interfaces específicas. Nesse caso, o computador recusará conexões com um endereço IP que o servidor SSH não esteja ouvindo, mesmo que a solicitação de conexão tenha a porta correta.

  • O servidor pode ter um backlog de solicitações de conexão para a mesma porta. Isso é raro e incomum, mas se o host estiver recebendo solicitações de conexão mais rapidamente do que elas podem ser tratadas, o host eventualmente começará a rejeitar novas solicitações de conexão.

Observe que, firewalls à parte, “conexão recusada” significa que você está se comunicando com o computador remoto - ele simplesmente não está aceitando sua solicitação de conexão.

A chave privada não é aceita (depuração de clientes OpenSSH) #

Por padrão, a maioria das informações é ocultada do usuário. Você pode usar as opções -v para obter um log detalhado da tentativa de conexão, que geralmente identificará o problema mostrando por que o comportamento é diferente do esperado.

Vamos supor que você está se conectando ao servidor example.com usando ssh (ou outro cliente OpenSSH como sftp ou scp) e sua chave privada não é aceita pelo servidor e o servidor pede a senha (ou rejeita a conexão):

$ ssh example.com
user@example.com's password:

Tente executar o ssh com as opções -vvv, que escreverão todas as mensagens de depuração. Será muita informação, mas depois de algum tempo, é bem fácil entender que:

$ ssh -vvv example.com

O problema mais comum é que a chave não está no local esperado. Você pode esperar que linhas semelhantes apareçam, reclamando sobre arquivos ausentes. Verificar se seu arquivo foi realmente lido é um bom começo:

debug1: identity file /home/username/.ssh/id_dsa type -1
debug1: key_load_public: No such file or directory
debug1: identity file /home/username/.ssh/id_dsa-cert type -1

Podemos continuar para a parte de autenticação. A chave pode ser oferecida, mas rejeitada pelo servidor, devido a um problema com a configuração do servidor. O log pode ficar assim:

debug3: preferred publickey,keyboard-interactive,password
debug3: authmethod_lookup publickey
debug3: remaining preferred: keyboard-interactive,password
debug3: authmethod_is_enabled publickey
debug1: Next authentication method: publickey
debug1: Offering RSA public key: username@localhost
debug3: send_pubkey_test
debug3: send packet: type 50
debug2: we sent a publickey packet, wait for reply
debug3: receive packet: type 51
debug1: Authentications that can continue: publickey,gssapi-keyex,gssapi-with-mic,password

A IDENTIFICAÇÃO DO HOST REMOTO MUDOU! #

O erro comum usando ssh é ver o erro como

@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY!
Someone could be eavesdropping on you right now (man-in-the-middle attack)!
It is also possible that a host key has just been changed.
The fingerprint for the RSA key sent by the remote host is
SHA256:L5ri/Xdgpuals893ej1z5F1wlg1n2YNeBf/tsABX+QQ.
Please contact your system administrator.
Add correct host key in /Users/username/.ssh/known_hosts to get rid of this message.
Offending RSA key in /Users/username/.ssh/known_hosts:12
RSA host key for *IP address* has changed and you have requested strict checking.
Host key verification failed.

Isso significa que você se conectou ao mesmo servidor antes e foi identificado usando diferentes chaves de host. Se você sabe que mudou as chaves do servidor, reinstalou o servidor ou o administrador do servidor anunciou algumas mudanças, geralmente não há problema em remover a chave antiga e deixar o ssh armazenar sua nova.

A chave antiga pode ser removida de forma transparente usando ssh-keygen:

ssh-keygen -R *IP address*

E a próxima conexão deve solicitar que você verifique a nova impressão digital:

ssh192.168.0.128
The authenticity of host '192.168.0.128 (192.168.0.128)' can't be established.
ECDSA key fingerprint is SHA256:L5ri/Xdgpuals893ej1z5F1wlg1n2YNeBf/tsABX+QQ.
Are you sure you want to continue connecting (yes/no)?

Caso você não tenha conhecimento de nenhum dos itens acima, o melhor é entrar em contato com o administrador do seu servidor para ter certeza de que está tudo ok. Caso contrário, o invasor em potencial poderá obter suas informações de autenticação e todos os dados transferidos!

ssh_exchange_identification: read: Conexão redefinida por peer #

Esta mensagem de erro pode ser produzida pelo cliente OpenSSH ssh. Isso significa que a conexão TCP entre o cliente e o servidor foi fechada de forma anormal pelo servidor imediatamente após ser aceita. Os motivos comuns para esta mensagem incluem:

  • O processo do servidor SSH está com defeito - por exemplo, ele travou.
  • Um firewall, roteador ou outro dispositivo de rede entre o cliente e o servidor está interferindo na conexão SSH.

As frases na mensagem de erro indicam especificamente o que aconteceu:

ssh_exchange_identification: 1. Depois que um cliente SSH faz uma conexão com um servidor SSH, o primeiro passo para iniciar o protocolo SSH é o servidor enviar sua versão de software ao cliente. Um erro contendo “ssh_exchange_identification” indica que o erro ocorreu imediatamente após a conexão TCP, enquanto o cliente aguardava a versão do software do servidor.

Redefinição de conexão: uma redefinição de conexão significa que a conexão TCP foi “fechada de forma anormal”. Você pode obter isso se o processo de software que contém uma das conexões TCP falhar. Os firewalls de rede podem ser configurados para usar redefinições de conexão como meio de bloquear conexões TCP.

pelo par significa que a conexão TCP foi fechada na “outra extremidade” da conexão. Nesse caso, a “outra extremidade” é o servidor SSH remoto.

Observe que esse erro não indica nenhum tipo de falha de autenticação. O servidor fechou a conexão TCP imediatamente após aceitá-la. O cliente e o servidor ainda não trocaram nenhum dado. O servidor ainda não enviou uma chave de host para o cliente e o cliente ainda não tentou se autenticar no servidor.

ssh_exchange_identification: Conexão fechada por host remoto #

Essa mensagem de erro pode ser produzida pelo cliente ssh OpenSSH. Isso significa que a conexão TCP entre o cliente e o servidor foi fechada pelo servidor imediatamente após ser aceita. Essa mensagem geralmente indica que o servidor SSH foi configurado para não aceitar conexões do cliente por algum motivo:

  • O servidor pode ser configurado para recusar conexões do endereço IP do cliente.
  • O servidor pode ser configurado com um limite no número de conexões SSH ativas.
  • O cliente pode ter se conectado a algo que não seja um servidor SSH.

As frases na mensagem de erro indicam especificamente o que aconteceu:

ssh_exchange_identification: 1. Depois que um cliente SSH faz uma conexão com um servidor SSH, o primeiro passo para iniciar o protocolo SSH é o servidor enviar sua versão de software ao cliente. Um erro contendo “ssh_exchange_identification” indica que o erro ocorreu imediatamente após a conexão TCP, enquanto o cliente aguardava a versão do software do servidor.

Conexão fechada: isso significa que a conexão TCP foi fechada normalmente. Indica que o servidor SSH fechou deliberadamente a conexão. Isso contrasta com “Redefinição de conexão”, que pode indicar que o servidor SSH travou ou apresentou mau funcionamento.

por host remoto significa que a conexão TCP foi fechada a partir da “outra extremidade” da conexão. Nesse caso, a “outra extremidade” é o servidor SSH remoto.

Observe que esse erro não indica nenhum tipo de falha de autenticação. O servidor fechou a conexão TCP imediatamente após aceitá-la. O cliente e o servidor ainda não trocaram nenhum dado. O servidor ainda não enviou uma chave de host para o cliente e o cliente ainda não tentou se autenticar no servidor.

Conexão expirou #

Um erro “Conexão expirou” ocorre quando o sistema remoto não responde à tentativa do cliente de abrir uma conexão TCP/IP. As causas mais comuns incluem:

  • Um firewall está bloqueando a tentativa de conexão na porta que você está usando:
  • O firewall pode estar no lado do cliente, bloqueando as conexões de saída, no lado do servidor, bloqueando as conexões de entrada ou em outro lugar no caminho.
  • O problema raiz pode ser que você está usando o número de porta errado.
  • O host ao qual você está tentando se conectar pode estar offline.
  • Alguma parte da rede entre o cliente e o servidor pode estar inativa ou interrompida.
  • Pode haver um problema de roteamento de rede.

Diagnosticar a causa raiz dos tempos limite de conexão é difícil.
