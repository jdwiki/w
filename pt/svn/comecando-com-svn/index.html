Confirmando suas alterações locais no repositório #

Para publicar as alterações que você fez em sua cópia de trabalho, execute o comando svn commit.

IMPORTANTE: revise suas alterações antes de confirmá-las! Use svn status e svn diff para revisar as mudanças. Além disso, certifique-se de estar no caminho correto antes de realizar um commit. Se você atualizou muitos arquivos em vários diretórios, deve estar no nível apropriado para incluir todos eles abaixo de sua localização.

Aqui está um exemplo do comando commit:

svn commit -m "My Descriptive Log Message"

Alternativamente, svn ci é a abreviação de svn commit

Observe a opção -m (--message). Boas mensagens de commit ajudam os outros a entender por que um commit foi feito. Além disso, no lado do servidor, é possível [impor mensagens não vazias] 4, e até mesmo impor que cada mensagem de confirmação mencione um ticket existente em seu sistema de rastreamento de bugs.

Revisando os logs #

A execução do svn log mostrará todas as mensagens de commit, você provavelmente deseja revisar apenas algumas revisões.

  • Veja as n revisões mais recentes:

    svn log -n

  • Ver uma revisão específica:

    svn log -c rXXX

  • Veja os caminhos afetados:

    svn log -v -c rXXX

Instalação e configuração inicial #

Instale o cliente svn para começar a colaborar no projeto que está usando o Subversion como seu sistema de controle de versão.

Para instalar o Subversion, você pode compilá-lo você mesmo a partir de uma versão do código-fonte ou baixar um pacote binário pré-construído para seu sistema operacional. A lista de sites onde você pode obter um cliente Subversion compilado (svn) para vários sistemas operacionais está disponível na página oficial de pacotes binários. Se você quiser compilar o software por conta própria, pegue a fonte na página de código-fonte.

Com o Subversion, você não está limitado a usar apenas o cliente de linha de comando padrão svn. Existem alguns clientes gráficos Subversion notáveis ​​para vários sistemas operacionais e a maioria dos IDEs hoje em dia fornecem integração robusta com SVN diretamente da caixa ou via plugins. Para a lista de clientes gráficos, verifique a página da Wikipedia: https://en.wikipedia.org/wiki/Comparison_of_Subversion_clients.

Logo após instalar o cliente, você poderá executá-lo emitindo o comando svn. Você deve ver o seguinte:

$ svn
Type 'svn help' for usage.

Tudo está quase pronto. Agora você deve criar uma área de trabalho local chamada cópia de trabalho que será conectada ao repositório central remoto. Em outras palavras, você vai comprar uma cópia de trabalho. Você vai operar com os dados versionados com a ajuda da cópia de trabalho e pode publicar suas alterações (chamadas committing no SVN) para que outras pessoas trabalhando no mesmo projeto possam vê-las e se beneficiar de suas alterações. Para buscar posteriormente as alterações futuras feitas por outros do repositório, você atualizaria sua cópia de trabalho. Essas operações básicas são abordadas em outros exemplos.

Criando e aplicando patches #

Um patch é um arquivo que mostra as diferenças entre duas revisões ou entre seu repositório local e a última revisão que seu repositório está apontando.

Para compartilhar ou salvar um patch de suas alterações locais não confirmadas para revisão por pares ou para aplicação posterior, faça:

svn diff > new-feature.patch

Para obter um patch das diferenças entre duas revisões:

svn diff -r NEWER_REVISION:OLDER_REVISION > feature.patch

Para aplicar um patch, execute:

svn patch new-feature.patch

Para aplicar o patch com sucesso, você deve executar o comando no mesmo caminho onde o patch foi criado.

Verificando uma cópia de trabalho #

Para começar a fazer modificações nos dados do projeto, você precisa obter uma cópia local do projeto com versão. Use o cliente svn da linha de comando ou seu cliente SVN favorito (TortoiseSVN, por exemplo). Sua cópia local do projeto é chamada de cópia de trabalho no Subversion e você a obtém emitindo o comando svn checkout <URL> onde <URL> é uma URL do repositório. por exemplo.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk

Alternativamente, você pode usar svn co <URL> como um atalho para fazer o checkout de uma cópia local.

Como resultado, você obterá uma cópia de trabalho do /trunk de um projeto chamado MyProject que reside no repositório MyRepo. A cópia de trabalho estará localizada em um diretório chamado trunk em seu computador em relação ao diretório no qual você emitiu o comando.

Se desejar ter um nome diferente para a sua cópia de trabalho, pode adicioná-lo como parâmetro ao final do comando. por exemplo.

$ svn checkout https://svn.example.com/svn/MyRepo/MyProject/trunk MyProjectSource

Isso criará uma cópia de trabalho chamada MyProjectSource.

Note que ao invés de checar o trunk, você pode checar algum branch, private shelve ou uma tag (assumindo que eles já existam no repositório); você pode ter um número ilimitado de cópias de trabalho locais em sua máquina.

Você também pode obter a cópia de trabalho de todo o repositório MyRepo. Mas você deve se abster de fazê-lo. De um modo geral, você não precisa ter uma cópia de trabalho de todo o repositório para o seu trabalho porque sua cópia de trabalho pode ser trocada instantaneamente para outro branch/tag/qualquer coisa de desenvolvimento. Além disso, o repositório Subversion pode conter vários projetos (não) relacionados e é melhor ter uma cópia de trabalho dedicada para cada um deles, não uma única cópia de trabalho para todos os projetos.

Exportando os dados versionados (download simples) #

Se você deseja obter os dados do projeto versionado, mas não precisa de nenhum dos recursos de controle de versão oferecidos pelo Subversion, você pode executar o comando svn export <URL>. Aqui está um exemplo:

$ svn export https://svn.example.com/svn/MyRepo/MyProject/trunk

Como resultado, você obterá a exportação de dados do projeto, mas diferentemente de uma cópia de trabalho, você não poderá executar comandos svn nela. A exportação é apenas um download simples dos dados.

Se algum tempo depois você quiser converter os dados baixados em uma cópia de trabalho totalmente funcional, execute svn checkout <URL> para o diretório onde você executou a exportação.

Fazendo check-out de uma cópia de trabalho em uma revisão específica #

Para obter a versão 5394, use:

svn co --revision r5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

Ou a versão mais curta:

svn co -r 5394 https://svn.example.com/svn/MyRepo/MyProject/trunk

Ou usando revisões indexadas:

svn co https://svn.example.com/svn/MyRepo/MyProject/trunk@5394

Se já tiver feito check-out, você pode usar o comando update para mover a para uma revisão específica, fazendo:

svn up -rXXX

Atualizando uma cópia de trabalho #

Você não é a única pessoa trabalhando no projeto, certo? Isso significa que seus colegas também estão fazendo modificações nos dados do projeto. Para se manter atualizado e buscar as modificações feitas por outros, você deve executar o comando svn update em sua cópia de trabalho. Como resultado, sua cópia de trabalho será sincronizada com o repositório e baixará as alterações feitas por seus colegas.

Abreviação para svn update é svn up.

É uma regra executar svn update antes de submeter suas alterações.

Fazendo alterações em sua cópia de trabalho local #

A cópia de trabalho (WC) é seu espaço de trabalho local e privado que você usa para interagir com o repositório central do Subversion. Você usa a cópia de trabalho para modificar o conteúdo do seu projeto e buscar as alterações feitas por outros.

A cópia de trabalho contém os dados do seu projeto e se parece e age como um diretório regular em seu sistema de arquivos local, mas com uma grande diferença - a cópia de trabalho rastreia o status e as alterações dos arquivos e diretórios dentro. Você pode pensar na cópia de trabalho como um diretório regular com um sabor de controle de versão adicionado por um diretório de metadados .svn oculto em sua raiz.

Na maioria das vezes, você fará modificações nos dados do projeto modificando o conteúdo da cópia de trabalho. Assim que estiver satisfeito com as modificações e revisá-las completamente, você estará pronto para publicá-las no repositório central.

Você pode realizar qualquer ação com os dados do seu projeto dentro da cópia de trabalho, mas as operações que envolvem copiar, mover, renomear e deletar devem ser executadas usando os comandos svn correspondentes:

  • Modificando arquivos existentes. Modifique os arquivos como você costuma fazer usando seu processador de texto favorito, editor gráfico, software de edição de áudio, IDE, etc. Assim que você salvar as alterações no disco, o Subversion irá reconhecê-las automaticamente.

  • Adicionando novos arquivos. Coloque novos arquivos na cópia de trabalho e o Subversion irá reconhecê-los como sem versão. Ele não começará a rastrear automaticamente os novos arquivos a menos que você execute o comando svn add:

    svn add foo.cs

  • Movendo arquivos e diretórios. Mova arquivos e diretórios usando o comando svn move:

    svn move foo.cs bar.cs

  • Renomeando arquivos e diretórios. Renomeie arquivos e diretórios usando o comando svn rename:

    svn rename foo.cs bar.cs

    NOTE: svn rename command is an alias of svn move command.

  • Copiando arquivos e diretórios. Copie arquivos e diretórios usando o comando svn copy:

    svn copy foo.cs bar.cs

  • Excluindo arquivos e diretórios. Exclua arquivos e diretórios usando o comando svn delete:

    svn delete foo.cs

  • Verificando o status de arquivos e diretórios na cópia de trabalho. Revise suas alterações usando o comando svn status (ou svn st para abreviar):

    svn status

    IMPORTANT: Always review your changes before committing them. This will help you to avoid committing unnecessary or irrelevant changes.

  • Revertendo mudanças. Reverta suas alterações usando o comando svn revert:

    svn revert foo.c

  • Revertendo todas as alterações: Da raiz do repositório:

    svn revert -R .

IMPORTANTE: as alterações não confirmadas revertidas serão perdidas para sempre. Você não poderá recuperar as alterações revertidas. Use svn revert com cuidado! Se você deseja manter as alterações, mas precisa reverter, salve-as em um patch. Veja exemplo de como criar e aplicar um patch.

Reverter ou reverter um arquivo #

Para restaurar um arquivo para a última versão atualizada do svn, ou seja, desfazer as alterações locais, você pode usar revert:

svn revert file

Para restaurar um arquivo para uma versão mais antiga (revisão XXX) use update:

svn update -r XXX file

Aviso: em ambos os casos você perderá quaisquer alterações locais no arquivo porque ele será sobrescrito.

Para visualizar apenas a versão mais antiga de um arquivo, use cat:

svn cat -r XXX file

E para ver as diferenças com sua versão local do arquivo:

svn diff -r XXX file

Usando um repositório protegido por senha #

Um repositório Subversion pode ser configurado para que determinados conteúdos ou comandos sejam acessíveis apenas a determinados usuários. Para acessar este conteúdo restrito, você precisará especificar um nome de usuário e uma senha.

Seu nome de usuário e senha podem ser especificados diretamente como parte do comando:

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser --password topsecret

Infelizmente, isso faz com que sua senha apareça em texto simples no console. Para evitar esse possível problema de segurança, especifique um nome de usuário, mas não uma senha. Isso fará com que um prompt de senha apareça, permitindo que você insira sua senha sem expô-la:

$ svn checkout https://svn.example.com/MyRepo/trunk --username JoeUser
Password for 'JoeUser':

Fornecer nenhuma informação de autenticação faz com que o Subversion solicite tanto o nome de usuário quanto a senha:

$ svn checkout https://svn.example.com/MyRepo/trunk
Username:  JoeUser
Password for 'JoeUser':

Embora o primeiro método seja menos seguro, é frequentemente visto em scripts automatizados, pois é difícil para muitos tipos de script fornecer informações a um prompt interativo.

Nota: comandos que operam apenas em sua cópia de trabalho (como revert ou status) nunca exigirão uma senha, apenas comandos que requerem comunicação com o servidor do repositório.
