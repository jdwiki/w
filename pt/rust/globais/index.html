Sintaxe #

  • const IDENTIFICADOR: tipo = constexpr;

  • estático [mut] IDENTIFICADOR: tipo = expr;

  • lazy_static! { static ref IDENTIFICADOR: tipo = expr; }

  • Os valores const são sempre embutidos e não possuem endereço na memória.

  • Os valores static nunca são embutidos e têm uma instância com um endereço fixo.

  • Valores static mut não são seguros para a memória e, portanto, só podem ser acessados ​​em um bloco unsafe.

  • Às vezes, usar variáveis ​​mutáveis ​​estáticas globais em código multithread pode ser perigoso, então considere usar std::sync::Mutex ou outras alternativas

  • Objetos lazy_static são imutáveis, são inicializados apenas uma vez, são compartilhados entre todas as threads e podem ser acessados ​​diretamente (não há tipos de wrapper envolvidos). Em contraste, objetos thread_local devem ser mutáveis, são inicializados uma vez para cada thread e os acessos são indiretos (envolvendo o tipo de wrapper LocalKey<T>)

Const #

A palavra-chave const declara uma ligação constante global.

const DEADBEEF: u64 = 0xDEADBEEF;

fn main() {
    println("{:X}", DEADBEEF);
}

Isso produz

DEADBEEF

Estático #

A palavra-chave static declara uma ligação estática global, que pode ser mutável.

static HELLO_WORLD: &'static str = "Hello, world!";

fn main() {
    println("{}", HELLO_WORLD);
}

Isso produz

Hello, world!

lazy_static! #

Use a grade lazy_static para criar variáveis ​​globais imutáveis ​​que são inicializadas em tempo de execução. Usamos HashMap como demonstração.

Em Cargo.toml:

[dependencies]
lazy_static = "0.1.*"

Em main.rs:

#[macro_use]
extern crate lazy_static;

lazy_static! {
    static ref HASHMAP: HashMap<u32, &'static str> = {
        let mut m = HashMap::new();
        m.insert(0, "hello");
        m.insert(1, ",");
        m.insert(2, " ");
        m.insert(3, "world");
        m
    };
    static ref COUNT: usize = HASHMAP.len();
}

fn main() {
    // We dereference COUNT because it's type is &usize
    println!("The map has {} entries.", *COUNT);

    // Here we don't dereference with * because of Deref coercions
    println!("The entry for `0` is \"{}\".", HASHMAP.get(&0).unwrap());
}

Objetos locais de thread #

Um objeto local de thread é inicializado em seu primeiro uso em um thread. E como o nome sugere, cada thread receberá uma nova cópia independente de outros threads.

use std::cell::RefCell;
use std::thread;

thread_local! {
    static FOO: RefCell<f32> = RefCell::new(1.0);
}

// When this macro expands, `FOO` gets type `thread::LocalKey<RefCell<f32>>`.
//
// Side note: One of its private member is a pointer to a function which is
// responsible for returning the thread-local object. Having all its members
// `Sync` [0], `LocalKey` is also implicitly `Sync`.
//
// [0]: As of writing this, `LocalKey` just has 2 function-pointers as members

fn main() {
    FOO.with(|foo| {
        // `foo` is of type `&RefCell<f64>`
        *foo.borrow_mut() = 3.0;
    });

    thread::spawn(move|| {
        // Note that static objects do not move (`FOO` is the same everywhere),
        // but the `foo` you get inside the closure will of course be different.
        FOO.with(|foo| {
            println!("inner: {}", *foo.borrow());
        });
    }).join().unwrap();

    FOO.with(|foo| {
        println!("main: {}", *foo.borrow());
    });
}

Saídas:

inner: 1
main: 3

Mut estático seguro com mut_static #

Itens globais mutáveis ​​(chamados mut static, destacando a contradição inerente envolvida em seu uso) não são seguros porque é difícil para o compilador garantir que eles sejam usados ​​adequadamente.

No entanto, a introdução de bloqueios mutuamente exclusivos em torno dos dados permite globais mutáveis ​​com segurança de memória. Isso NÃO significa que eles são logicamente seguros!

#[macro_use]
extern crate lazy_static;
extern crate mut_static;

use mut_static::MutStatic;

pub struct MyStruct { value: usize }

impl MyStruct {
    pub fn new(v: usize) -> Self{
        MyStruct { value: v }
    }
    pub fn getvalue(&self) -> usize { self.value }
    pub fn setvalue(&mut self, v: usize) { self.value = v }
}

lazy_static! {
    static ref MY_GLOBAL_STATE: MutStatic<MyStruct> = MutStatic::new();
}

fn main() {
    // Here, I call .set on the MutStatic to put data inside it.
    // This can fail.
    MY_GLOBAL_STATE.set(MyStruct::new(0)).unwrap();
    {
        // Using the global state immutably is easy...
        println!("Before mut: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
    {
         // Using it mutably is too...
         let mut mut_handle = MY_GLOBAL_STATE.write().unwrap();
         mut_handle.setvalue(3);
         println!("Changed value to 3.");
    } 
    {
        // As long as there's a scope change we can get the 
        // immutable version again...
        println!("After mut: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
    {
        // But beware! Anything can change global state!
        foo();
        println!("After foo: {}", 
                 MY_GLOBAL_STATE.read().unwrap().getvalue());
    }
 
}

// Note that foo takes no parameters
fn foo() {
    let val;
    {
        val = MY_GLOBAL_STATE.read().unwrap().getvalue();
    }
    {
        let mut mut_handle = 
            MY_GLOBAL_STATE.write().unwrap();
        mut_handle.setvalue(val + 1);
    }
}

Este código produz a saída:

Before mut: 0
Changed value to 3.
After mut: 3
After foo: 4

Isso não é algo que deveria acontecer no Rust normalmente. foo() não recebeu uma referência mutável para nada, então não deveria ter modificado nada, e ainda assim o fez. Isso pode levar a erros de lógica muito difíceis de depurar.

Por outro lado, isso às vezes é exatamente o que você quer. Por exemplo, muitos mecanismos de jogos exigem um cache global de imagens e outros recursos que são carregados lentamente (ou usam alguma outra estratégia de carregamento complexa) - o MutStatic é perfeito para esse propósito.
