Sintaxe #

  • struct Foo { field1: Type1, field2: Type2 }
  • let foo = Foo { campo1: Tipo1::novo(), campo2: Tipo2::novo() };
  • struct Bar ( Type1, Type2 ); // tipo de tupla
  • let _ = Bar(Tipo1::novo(), Tipo2::novo());
  • estrutura Baz; // tipo de unidade
  • deixe _ = Baz;
  • deixe Foo { campo1, .. } = foo; //extrai field1 por correspondência de padrões
  • deixe Foo { campo1: x, .. } = foo; //extrai campo1 como x
  • let foo2 = Foo { campo1: Type1::new(), .. foo }; //construir a partir de existente
  • impl Foo { fn fiddle(&self) {} } // declara o método de instância para Foo
  • impl Foo { fn tweak(&mut self) {} } // declara o método de instância mutável para Foo
  • impl Foo { fn double(self) {} } // declara o método de instância proprietário para Foo
  • impl Foo { fn new() {} } // declara o método associado para Foo

Criando e usando valores de estrutura #

Considere as seguintes definições de struct:

struct Foo {
    my_bool: bool,
    my_num: isize,
    my_string: String,
}
struct Bar (bool, isize, String);
struct Baz;

A construção de novos valores de estrutura para esses tipos é simples:

let foo = Foo { my_bool: true, my_num: 42, my_string: String::from("hello") };
let bar = Bar(true, 42, String::from("hello"));
let baz = Baz;

Acesse campos de uma estrutura usando .:

assert_eq!(foo.my_bool, true);
assert_eq!(bar.0, true); // tuple structs act like tuples

Uma ligação mutável a uma estrutura pode ter seus campos alterados:

let mut foo = foo;
foo.my_bool = false;
let mut bar = bar;
bar.0 = false;

Os recursos de correspondência de padrões do Rust também podem ser usados ​​para espiar dentro de um struct:

// creates bindings mb, mn, ms with values of corresponding fields in foo
let Foo { my_bool: mb, my_num: mn, my_string: ms } = foo;
assert_eq!(mn, 42);
// .. allows you to skip fields you do not care about
let Foo { my_num: mn, .. } = foo;
assert_eq!(mn, 42);
// leave out `: variable` to bind a variable by its field name
let Foo { my_num, .. } = foo;
assert_eq!(my_num, 42);

Ou faça um struct usando um segundo struct como um “modelo” com a sintaxe de atualização do Rust:

let foo2 = Foo { my_string: String::from("world"), .. foo };
assert_eq!(foo2.my_num, 42);

Definindo estruturas #

Estruturas em Rust são definidas usando a palavra-chave struct. A forma mais comum de estrutura consiste em um conjunto de campos nomeados:

struct Foo {
    my_bool: bool,
    my_num: isize,
    my_string: String,
}

O acima declara um struct com três campos: my_bool, my_num e my_string, dos tipos bool, isize e String respectivamente.

Outra maneira de criar structs em Rust é criar uma tuple struct:

struct Bar (bool, isize, String);

Isso define um novo tipo, Bar, que possui três campos sem nome, do tipo bool, isize e String, nessa ordem. Isso é conhecido como o padrão newtype, porque efetivamente introduz um novo “nome” para um tipo específico. No entanto, ele faz isso de uma maneira mais poderosa do que os aliases criados usando a palavra-chave type; Bar é aqui um tipo totalmente funcional, o que significa que você pode escrever seus próprios métodos para ele (abaixo).

Por fim, declare um struct com no campos, chamado de unit-like struct:

struct Baz;

Isso pode ser útil para simulação ou teste (quando você deseja implementar uma característica trivialmente) ou como um tipo de marcador. Em geral, no entanto, é improvável que você encontre muitas estruturas semelhantes a unidades.

Observe que os campos struct em Rust são todos privados por padrão — ou seja, eles não podem ser acessados ​​a partir de código fora do módulo que define o tipo. Você pode prefixar um campo com a palavra-chave pub para torná-lo acessível publicamente. Além disso, o tipo struct em si é privado. Para tornar o tipo disponível para outros módulos, a definição struct também deve ser prefixada com pub:

pub struct X {
    my_field: bool,
    pub our_field: bool,
}

Estruturas genéricas #

As estruturas podem ser genéricas em um ou mais parâmetros de tipo. Esses tipos são fornecidos entre <> quando se refere ao tipo:

struct Gen<T> {
    x: T,
    z: isize,
}

// ...
let _: Gen<bool> = Gen{x: true, z: 1};
let _: Gen<isize> = Gen{x: 42, z: 2};
let _: Gen<String> = Gen{x: String::from("hello"), z: 3};

Vários tipos podem ser fornecidos usando vírgula:

struct Gen2<T, U> {
    x: T,
    y: U,
}

// ...
let _: Gen2<bool, isize> = Gen2{x: true, y: 42};

Os parâmetros de tipo fazem parte do tipo, portanto, duas variáveis ​​do mesmo tipo base, mas com parâmetros diferentes, não são intercambiáveis:

let mut a: Gen<bool> = Gen{x: true, z: 1};
let b: Gen<isize> = Gen{x: 42, z: 2};
a = b; // this will not work, types are not the same
a.x = 42; // this will not work, the type of .x in a is bool

Se você quiser escrever uma função que aceite um struct independentemente de sua atribuição de parâmetro de tipo, essa função também precisaria ser genérica:

fn hello<T>(g: Gen<T>) {
    println!("{}", g.z); // valid, since g.z is always an isize
}

Mas e se quiséssemos escrever uma função que pudesse sempre imprimir g.x? Precisaríamos restringir T para ser de um tipo que pode ser exibido. Podemos fazer isso com limites de tipo:

use std::fmt;
fn hello<T: fmt::Display>(g: Gen<T>) {
    println!("{} {}", g.x, g.z);
}

A função hello agora é somente definida para instâncias Gen cujo tipo T implementa fmt::Display. Se tentássemos passar um Gen<(bool, isize)> por exemplo, o compilador reclamaria que hello não está definido para esse tipo.

Também podemos usar limites de tipo diretamente nos parâmetros de tipo do struct para indicar que você só pode construir esse struct para determinados tipos:

use std:#️⃣:Hash;
struct GenB<T: Hash> {
    x: T,
}

Qualquer função que tenha acesso a um GenB agora sabe que o tipo de x implementa Hash, e assim pode chamar .x.hash(). Vários limites de tipo para o mesmo parâmetro podem ser fornecidos separando-os com um +.

Da mesma forma que para funções, os limites de tipo podem ser colocados após o <> usando a palavra-chave where:

struct GenB<T> where T: Hash {
    x: T,
}

Isso tem o mesmo significado semântico, mas pode tornar a assinatura mais fácil de ler e formatar quando você tiver limites complexos.

Parâmetros de tipo também estão disponíveis para métodos de instância e métodos associados do struct:

// note the <T> parameter for the impl as well
// this is necessary to say that all the following methods only
// exist within the context of those type parameter assignments
impl<T> Gen<T> {
    fn inner(self) -> T {
        self.x
    }
    fn new(x: T) -> Gen<T> {
        Gen{x: x}
    }
}

Se você tem limites de tipo no T de Gen, eles também devem ser refletidos nos limites de tipo do impl. Você também pode tornar os limites impl mais apertados para dizer que um determinado método só existe se o tipo satisfizer uma propriedade específica:

impl<T: Hash + fmt::Display> Gen<T> {
    fn show(&self) {
        println!("{}", self.x);
    }
}

// ...
Gen{x: 42}.show(); // works fine
let a = Gen{x: (42, true)}; // ok, because (isize, bool): Hash
a.show(); // error: (isize, bool) does not implement fmt::Display

Métodos de estrutura #

Para declarar métodos em um struct (ou seja, funções que podem ser chamadas “on” o struct, ou valores desse tipo struct), crie um bloco impl:

impl Foo {
    fn fiddle(&self) {
        // "self" refers to the value this method is being called on
        println!("fiddling {}", self.my_string);
    }
}

// ...
foo.fiddle(); // prints "fiddling hello"

&self aqui indica que uma referência imutável a uma instância de struct Foo é necessária para invocar o método fiddle. Se quiséssemos modificar a instância (como alterar um de seus campos), usaríamos um &mut self (ou seja, uma referência mutável):

impl Foo {
    fn tweak(&mut self, n: isize) {
        self.my_num = n;
    }
}

// ...
foo.tweak(43);
assert_eq!(foo.my_num, 43);

Finalmente, também podemos usar self (observe a falta de um &) como receptor. Isso requer que a instância seja de propriedade do chamador e fará com que a instância seja movida ao chamar o método. Isso pode ser útil se desejarmos consumir, destruir ou transformar inteiramente uma instância existente. Um exemplo de tal caso de uso é fornecer métodos de “encadeamento”:

impl Foo {
    fn double(mut self) -> Self {
        self.my_num *= 2;
        self
    }
}

// ...
foo.my_num = 1;
assert_eq!(foo.double().double().my_num, 4);

Observe que também prefixamos self com mut para que possamos alterar o self antes de retorná-lo novamente. O tipo de retorno do método double também merece alguma explicação. Self dentro de um bloco impl refere-se ao tipo ao qual o impl se aplica (neste caso, Foo). Aqui, é principalmente uma abreviação útil para evitar redigitar a assinatura do tipo, mas em traços, pode ser usado para se referir ao tipo subjacente que implementa um traço específico.

Para declarar um método associado (comumente chamado de “método de classe” em outras linguagens) para um struct simplesmente deixe de fora o argumento self. Tais métodos são chamados no próprio tipo struct, não em uma instância dele:

impl Foo {
    fn new(b: bool, n: isize, s: String) -> Foo {
        Foo { my_bool: b, my_num: n, my_string: s }
    }
}

// ...
// :: is used to access associated members of the type
let x = Foo::new(false, 0, String::from("nil"));
assert_eq!(x.my_num, 0);

Observe que os métodos de estrutura só podem ser definidos para tipos que foram declarados no módulo atual. Além disso, como acontece com os campos, todos os métodos de estrutura são privados por padrão e, portanto, só podem ser chamados por código no mesmo módulo. Você pode prefixar as definições com a palavra-chave pub para torná-las chamáveis ​​de outro lugar.
