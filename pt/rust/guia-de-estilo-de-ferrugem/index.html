Embora não haja um guia oficial de estilo Rust, os exemplos a seguir mostram as convenções adotadas pela maioria dos projetos Rust. Seguir essas convenções alinhará o estilo do seu projeto com o da biblioteca padrão, tornando mais fácil para as pessoas verem a lógica em seu código.

As diretrizes oficiais de estilo Rust estavam disponíveis no repositório rust-lang/rust no GitHub, mas eles foram removidos recentemente, com migração pendente para o repositório rust-lang-nursery/fmt-rfcs. Até que novas diretrizes sejam publicadas lá, você deve tentar seguir as diretrizes do repositório rust-lang.

Você pode usar rustfmt e clippy para revisar automaticamente seu código quanto a problemas de estilo e formate-o corretamente. Essas ferramentas podem ser instaladas usando o Cargo, assim:

cargo install clippy
cargo install rustfmt

Para executá-los, você usa:

cargo clippy
cargo fmt

Nosso #

Estruturas

// Structs use UpperCamelCase.
pub struct Snafucator {
    
}

mod snafucators {
    // Try to avoid 'stuttering' by repeating 
    // the module name in the struct name.

    // Bad:
    pub struct OrderedSnafucator {
    
    }
      
    // Good:
    pub struct Ordered {
        
    }
}

Características

// Traits use the same naming principles as 
// structs (UpperCamelCase).
trait Read {
    fn read_to_snafucator(&self) -> Result<(), Error>;
}

Caixas e Módulos

// Modules and crates should both use snake_case.
// Crates should try to use single words if possible.
extern crate foo;
mod bar_baz {
    mod quux {

    }
}

Variáveis ​​e constantes estáticas

// Statics and constants use SCREAMING_SNAKE_CASE.
const NAME: &'static str = "SCREAMING_SNAKE_CASE";

Enums

// Enum types and their variants **both** use UpperCamelCase.
pub enum Option<T> {
   Some(T),
   None
}

Funções e Métodos

// Functions and methods use snake_case
fn snake_cased_function() {

}

Vinculações variáveis

// Regular variables also use snake_case
let foo_bar = "snafu";

Vida útil

// Lifetimes should consist of a single lower case letter. By 
// convention, you should start at 'a, then 'b, etc.

// Good:
struct Foobar<'a> {
    x: &'a str
}

// Bad:
struct Bazquux<'stringlife> {
    my_str: &'stringlife str
}

Acrônimos

Os nomes de variáveis ​​que contêm acrônimos, como TCP, devem ter o seguinte estilo:

  • Para nomes UpperCamelCase, a primeira letra deve ser maiúscula (por exemplo, TcpClient)
  • Para nomes snake_case, não deve haver maiúsculas (por exemplo, tcp_client)
  • Para nomes SCREAMING_SNAKE_CASE, o acrônimo deve estar completamente em maiúscula (por exemplo, TCP_CLIENT)

Espaço em branco #

Comprimento da linha

// Lines should never exceed 100 characters. 
// Instead, just wrap on to the next line.
let bad_example = "So this sort of code should never really happen, because it's really hard to fit on the screen!";

Recuo

// You should always use 4 spaces for indentation. 
// Tabs are discouraged - if you can, set your editor to convert
// a tab into 4 spaces.
let x = vec![1, 3, 5, 6, 7, 9];
for item in x {
    if x / 2 == 3 {
        println!("{}", x);
    }
}

Espaço em branco à direita

Os espaços em branco no final dos arquivos ou linhas devem ser excluídos.

Operadores Binários

// For clarity, always add a space when using binary operators, e.g.
// +, -, =, *
let bad=3+4;
let good = 3 + 4;

Isso também se aplica em atributos, por exemplo:

// Good:
#[deprecated = "Don't use my class - use Bar instead!"]

// Bad:
#[deprecated="This is broken"]

Pontos e vírgulas

// There is no space between the end of a statement
// and a semicolon.

let bad = Some("don't do this!") ;
let good: Option<&str> = None;

Alinhando Campos de Estrutura

// Struct fields should **not** be aligned using spaces, like this:
pub struct Wrong {
    pub x  : i32,
    pub foo: i64 
}

// Instead, just leave 1 space after the colon and write the type, like this:
pub struct Right {
    pub x: i32,
    pub foo: i64
}

Assinaturas de Função

// Long function signatures should be wrapped and aligned so that
// the starting parameter of each line is aligned
fn foo(example_item: Bar, another_long_example: Baz, 
       yet_another_parameter: Quux) 
       -> ReallyLongReturnItem {
    // Be careful to indent the inside block correctly!
}

Suspensórios

// The starting brace should always be on the same line as its parent.
// The ending brace should be on its own line.
fn bad()
{
    println!("This is incorrect.");
}

struct Good {
    example: i32
}

struct AlsoBad {
    example: i32 }

Importações #

Você deve ordenar suas importações e declarações da seguinte forma:

  • declarações extern crate
  • usar importações
  • Importações externas de outros engradados devem vir em primeiro lugar
  • Reexportações (pub use)

Criando caixas #

Prelúdios e Reexportações

// To reduce the amount of imports that users need, you should
// re-export important structs and traits.
pub use foo::Client;
pub use bar::Server;

Às vezes, grades usam um módulo prelude para conter estruturas importantes, assim como std::io::prelude. Normalmente, eles são importados com use std::io::prelude::*;

Tipos #

Tipo de anotações

// There should be one space after the colon of the type
// annotation. This rule applies in variable declarations,
// struct fields, functions and methods.

// GOOD:
let mut buffer: String = String::new();
// BAD:
let mut buffer:String = String::new();
let mut buffer : String = String::new();

Referências

// The ampersand (&) of a reference should be 'touching'
// the type it refers to.

// GOOD:
let x: &str = "Hello, world.";
// BAD:
fn fooify(x: & str) {
    println!("{}", x);
}

// Mutable references should be formatted like so:
fn bar(buf: &mut String) {

}
