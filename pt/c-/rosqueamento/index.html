Sintaxe #

  • fio()
  • thread(thread&& outro)
  • thread explícito (Function&& func, Args&&… args)

Parâmetros #

| Parâmetro | Detalhes | | ——— | ————————————————– —————- | | ‘outro’ | Assume a propriedade de other, other não possui mais a thread | | função | Função para chamar em um thread separado | | args | Argumentos para func |

Algumas notas:

  • Dois objetos std::thread nunca podem representar a mesma thread.
  • Um objeto std::thread pode estar em um estado onde não representa nenhuma thread (ou seja, após um movimento, após chamar join, etc.).

Criando um std::thread #

Em C++, as threads são criadas usando a classe std::thread. Um encadeamento é um fluxo de execução separado; é análogo a ter um auxiliar executando uma tarefa enquanto você executa simultaneamente outra. Quando todo o código no encadeamento é executado, ele termina. Ao criar uma thread, você precisa passar algo para ser executado nela. Algumas coisas que você pode passar para um thread:

  • Funções gratuitas
  • Funções dos membros
  • Objetos Functor
  • Expressões lambda

Exemplo de função gratuita - executa uma função em um thread separado (Exemplo ao vivo):

#include <iostream>
#include <thread>
 
void foo(int a)
{
    std::cout << a << '\n';
}
 
int main()
{
    // Create and execute the thread
    std::thread thread(foo, 10); // foo is the function to execute, 10 is the
                                 // argument to pass to it
 
    // Keep going; the thread is executed separately
 
    // Wait for the thread to finish; we stay here until it is done
    thread.join();
 
    return 0;
}

Exemplo de função de membro - executa uma função de membro em um thread separado (Exemplo ao vivo):

#include <iostream>
#include <thread>
 
class Bar
{
public:
    void foo(int a)
    {
        std::cout << a << '\n';
    }
};
 
int main()
{
    Bar bar;
    
    // Create and execute the thread
    std::thread thread(&Bar::foo, &bar, 10); // Pass 10 to member function
 
    // The member function will be executed in a separate thread
 
    // Wait for the thread to finish, this is a blocking operation
    thread.join();
 
    return 0;
}

Exemplo de objeto Functor (Exemplo ao vivo):

#include <iostream>
#include <thread>
 
class Bar
{
public:
    void operator()(int a)
    {
        std::cout << a << '\n';
    }
};
 
int main()
{
    Bar bar;
    
    // Create and execute the thread
    std::thread thread(bar, 10); // Pass 10 to functor object
 
    // The functor object will be executed in a separate thread
 
    // Wait for the thread to finish, this is a blocking operation
    thread.join();
 
    return 0;
}

Exemplo de expressão lambda (Exemplo ao vivo):

#include <iostream>
#include <thread>
 
int main()
{
    auto lambda = [](int a) { std::cout << a << '\n'; };

    // Create and execute the thread
    std::thread thread(lambda, 10); // Pass 10 to the lambda expression
 
    // The lambda expression will be executed in a separate thread
 
    // Wait for the thread to finish, this is a blocking operation
    thread.join();
 
    return 0;
}

Passando uma referência para um thread #

Você não pode passar uma referência (ou referência const) diretamente para um thread porque std::thread irá copiá-los/movê-los. Em vez disso, use std::reference_wrapper:

void foo(int& b)
{
    b = 10;
}

int a = 1;
std::thread thread{ foo, std::ref(a) }; //'a' is now really passed as reference

thread.join();
std::cout << a << '\n'; //Outputs 10

void bar(const ComplexObject& co)
{
    co.doCalculations();
}

ComplexObject object;
std::thread thread{ bar, std::cref(object) }; //'object' is passed as const&

thread.join();
std::cout << object.getResult() << '\n'; //Outputs the result

Usando std::async em vez de std::thread #

std::async também é capaz de criar threads. Comparado com std::thread, é considerado menos poderoso, mas mais fácil de usar quando você deseja apenas executar uma função de forma assíncrona.

Chamando uma função de forma assíncrona #

#include <future>
#include <iostream>

unsigned int square(unsigned int i){
    return i*i;
}

int main() {
    auto f = std::async(std::launch::async, square, 8);
    std::cout << "square currently running\n"; //do something while square is running
    std::cout << "result is " << f.get() << '\n'; //getting the result from square
}

Armadilhas comuns #

  • std::async retorna um std::future que contém o valor de retorno que será calculado pela função. Quando esse futuro é destruído, ele espera até que a thread seja concluída, tornando seu código efetivamente single threaded. Isso é facilmente esquecido quando você não precisa do valor de retorno:

    std::async(std::launch::async, square, 5);
//thread already completed at this point, because the returning future got destroyed

  • std::async funciona sem uma política de inicialização, então std::async(square, 5); compila. Quando você faz isso, o sistema decide se quer criar um thread ou não. A ideia era que o sistema optasse por criar um thread, a menos que já estivesse executando mais threads do que pode executar com eficiência. Infelizmente, as implementações geralmente optam por não criar uma thread nessa situação, nunca, então você precisa sobrescrever esse comportamento com std::launch::async que força o sistema a criar uma thread.

  • Cuidado com as condições da corrida.

Mais sobre assíncrono em Futures and Promises

Garantir que um thread seja sempre unido #

Quando o destruidor para std::thread é invocado, uma chamada para join() ou detach() deve ter sido feita. Se um thread não foi unido ou desanexado, por padrão std::terminate será chamado. Usando RAII, isso geralmente é simples o suficiente para realizar:

class thread_joiner
{
public:

    thread_joiner(std::thread t)
        : t_(std::move(t))
    { }

    ~thread_joiner()
    {
        if(t_.joinable()) {
            t_.join();
        }
    }

private:

    std::thread t_;
}

Isso é então usado assim:

 void perform_work()
 {
     // Perform some work
 }

 void t()
 {
     thread_joiner j{std::thread(perform_work)};
     // Do some other calculations while thread is running
 } // Thread is automatically joined here

Isso também fornece segurança de exceção; se tivéssemos criado nossa thread normalmente e o trabalho feito em t() executando outros cálculos tivesse lançado uma exceção, join() nunca teria sido chamado em nossa thread e nosso processo teria sido finalizado.

Sincronização básica #

A sincronização de threads pode ser realizada usando mutexes, entre outras primitivas de sincronização. Existem vários tipos de mutex fornecidos pela biblioteca padrão, mas o mais simples é std::mutex. Para bloquear um mutex, você constrói um bloqueio nele. O tipo de bloqueio mais simples é std::lock_guard:

std::mutex m;
void worker() {
    std::lock_guard<std::mutex> guard(m); // Acquires a lock on the mutex
    // Synchronized code here
} // the mutex is automatically released when guard goes out of scope

Com std::lock_guard o mutex é bloqueado por toda a vida útil do objeto de bloqueio. Nos casos em que você precisa controlar manualmente as regiões para bloqueio, use std::unique_lock:

std::mutex m;
void worker() {
    // by default, constructing a unique_lock from a mutex will lock the mutex
    // by passing the std::defer_lock as a second argument, we
    // can construct the guard in an unlocked state instead and
    // manually lock later.
    std::unique_lock<std::mutex> guard(m, std::defer_lock);
    // the mutex is not locked yet!
    guard.lock();
    // critical section
    guard.unlock();
    // mutex is again released
}

Mais estruturas de sincronização de threads

Crie um pool de threads simples #

As primitivas de threading do C++11 ainda são de nível relativamente baixo. Eles podem ser usados ​​para escrever uma construção de nível superior, como um pool de threads:

struct tasks {
  // the mutex, condition variable and deque form a single
  // thread-safe triggered queue of tasks:
  std::mutex m;
  std::condition_variable v;
  // note that a packaged_task<void> can store a packaged_task<R>:
  std::deque<std::packaged_task<void()>> work;

  // this holds futures representing the worker threads being done:
  std::vector<std::future<void>> finished;

  // queue( lambda ) will enqueue the lambda into the tasks for the threads
  // to use.  A future of the type the lambda returns is given to let you get
  // the result out.
  template<class F, class R=std::result_of_t<F&()>>
  std::future<R> queue(F&& f) {
    // wrap the function object into a packaged task, splitting
    // execution from the return value:
    std::packaged_task<R()> p(std::forward<F>(f));

    auto r=p.get_future(); // get the return value before we hand off the task
    {
      std::unique_lock<std::mutex> l(m);
      work.emplace_back(std::move(p)); // store the task<R()> as a task<void()>
    }
    v.notify_one(); // wake a thread to work on the task

    return r; // return the future result of the task
  }

  // start N threads in the thread pool.
  void start(std::size_t N=1){
    for (std::size_t i = 0; i < N; ++i)
    {
      // each thread is a std::async running this->thread_task():
      finished.push_back(
        std::async(
          std::launch::async,
          [this]{ thread_task(); }
        )
      );
    }
  }
  // abort() cancels all non-started tasks, and tells every working thread
  // stop running, and waits for them to finish up.
  void abort() {
    cancel_pending();
    finish();
  }
  // cancel_pending() merely cancels all non-started tasks:
  void cancel_pending() {
    std::unique_lock<std::mutex> l(m);
    work.clear();
  }
  // finish enques a "stop the thread" message for every thread, then waits for them:
  void finish() {
    {
      std::unique_lock<std::mutex> l(m);
      for(auto&&unused:finished){
        work.push_back({});
      }
    }
    v.notify_all();
    finished.clear();
  }
  ~tasks() {
    finish();
  }
private:
  // the work that a worker thread does:
  void thread_task() {
    while(true){
      // pop a task off the queue:
      std::packaged_task<void()> f;
      {
        // usual thread-safe queue code:
        std::unique_lock<std::mutex> l(m);
        if (work.empty()){
          v.wait(l,[&]{return !work.empty();});
        }
        f = std::move(work.front());
        work.pop_front();
      }
      // if the task is invalid, it means we are asked to abort:
      if (!f.valid()) return;
      // otherwise, run the task:
      f();
    }
  }
};

tasks.queue( []{ return "hello world"s; } ) retorna um std::future<std::string>, que quando o objeto tasks é executado é preenchido com hello world .

Você cria threads executando tasks.start(10) (que inicia 10 threads).

O uso de packaged_task<void()> é meramente porque não existe um equivalente std::function apagado por tipo que armazena tipos somente de movimentação. Escrever um personalizado desses provavelmente seria mais rápido do que usar packaged_task<void()>.

Operações no thread atual #

std::this_thread é um namespace que possui funções para fazer coisas interessantes na thread atual a partir da função da qual é chamada.

| Função | Descrição | | ————–| ————————————————– ——- | | get_id | Retorna o id do thread | | dormir_para | Dorme por um determinado período de tempo | | dormir_até | Dorme até uma hora específica | | rendimento | Reprogramar threads em execução, dando prioridade a outros threads |

Obtendo o id das threads atuais usando std::this_thread::get_id:

void foo()
{
    //Print this threads id
    std::cout << std::this_thread::get_id() << '\n';
}

std::thread thread{ foo };
thread.join(); //'threads' id has now been printed, should be something like 12556

foo(); //The id of the main thread is printed, should be something like 2420

Dormindo por 3 segundos usando std::this_thread::sleep_for:

void foo()
{
    std::this_thread::sleep_for(std::chrono::seconds(3));
}

std::thread thread{ foo };
foo.join();

std::cout << "Waited for 3 seconds!\n";

Dormindo até 3 horas no futuro usando std::this_thread::sleep_until:

void foo()
{
    std::this_thread::sleep_until(std::chrono::system_clock::now() + std::chrono::hours(3));
}

std::thread thread{ foo };
thread.join();

std::cout << "We are now located 3 hours after the thread has been called\n";

Deixando outras threads terem prioridade usando std::this_thread::yield:

void foo(int a)
{
    for (int i = 0; i < al ++i)
        std::this_thread::yield(); //Now other threads take priority, because this thread
                                   //isn't doing anything important

    std::cout << "Hello World!\n";
}

std::thread thread{ foo, 10 };
thread.join();

Usando variáveis ​​de condição #

Uma variável de condição é uma primitiva usada em conjunto com um mutex para orquestrar a comunicação entre threads. Embora não seja a maneira exclusiva ou mais eficiente de fazer isso, pode estar entre as mais simples para aqueles familiarizados com o padrão.

Um espera em um std::condition_variable com um std::unique_lock<std::mutex>. Isso permite que o código examine com segurança o estado compartilhado antes de decidir se deve ou não prosseguir com a aquisição.

Abaixo está um esboço produtor-consumidor que usa std::thread, std::condition_variable, std::mutex e alguns outros para tornar as coisas interessantes.

#include <condition_variable>
#include <cstddef>
#include <iostream>
#include <mutex>
#include <queue>
#include <random>
#include <thread>


int main()
{
    std::condition_variable cond;
    std::mutex mtx;
    std::queue<int> intq;
    bool stopped = false;

    std::thread producer{[&]()
    {
        // Prepare a random number generator.
        // Our producer will simply push random numbers to intq.
        //
        std::default_random_engine gen{};
        std::uniform_int_distribution<int> dist{};

        std::size_t count = 4006;    
        while(count--)
        {    
            // Always lock before changing
            // state guarded by a mutex and
            // condition_variable (a.k.a. "condvar").
            std::lock_guard<std::mutex> L{mtx};

            // Push a random int into the queue
            intq.push(dist(gen));

            // Tell the consumer it has an int
            cond.notify_one();
        }

        // All done.
        // Acquire the lock, set the stopped flag,
        // then inform the consumer.
        std::lock_guard<std::mutex> L{mtx};

        std::cout << "Producer is done!" << std::endl;

        stopped = true;
        cond.notify_one();
    }};

    std::thread consumer{[&]()
    {
        do{
            std::unique_lock<std::mutex> L{mtx};
            cond.wait(L,[&]()
            {
                // Acquire the lock only if
                // we've stopped or the queue
                // isn't empty
                return stopped || ! intq.empty();
            });

            // We own the mutex here; pop the queue
            // until it empties out.

            while( ! intq.empty())
            {
                const auto val = intq.front();
                intq.pop();

                std::cout << "Consumer popped: " << val << std::endl;
            }

            if(stopped){
                // producer has signaled a stop
                std::cout << "Consumer is done!" << std::endl;
                break;
            }

        }while(true);
    }};

    consumer.join();
    producer.join();
    
    std::cout << "Example Completed!" << std::endl;

    return 0;
}

Operações de thread #

Quando você inicia um thread, ele será executado até que seja concluído.

Muitas vezes, em algum momento, você precisa (possivelmente - o encadeamento já pode estar pronto) esperar que o encadeamento termine, porque você deseja usar o resultado, por exemplo.

int n;
std::thread thread{ calculateSomething, std::ref(n) };

//Doing some other stuff

//We need 'n' now!
//Wait for the thread to finish - if it is not already done
thread.join();

//Now 'n' has the result of the calculation done in the seperate thread
std::cout << n << '\n';

Você também pode detach o thread, deixando-o executar livremente:

std::thread thread{ doSomething };

//Detaching the thread, we don't need it anymore (for whatever reason)
thread.detach();

//The thread will terminate when it is done, or when the main thread returns

Armazenamento local de thread #

O armazenamento local de thread pode ser criado usando a [palavra-chave] thread_local (https://www.wikiod.com/pt/docs/c%2b%2b/4891/keywords). Diz-se que uma variável declarada com o especificador thread_local tem duração de armazenamento de thread.

  • Cada thread em um programa tem sua própria cópia de cada variável local de thread.
  • Uma variável local de thread com escopo de função (local) será inicializada na primeira vez que o controle passar por sua definição. Tal variável é implicitamente estática, a menos que seja declarada extern.
  • Uma variável local de thread com escopo de namespace ou classe (não local) será inicializada como parte da inicialização de thread.
  • Variáveis ​​locais de thread são destruídas após o término do thread.
  • Um membro de uma classe só pode ser thread-local se for estático. Portanto, haverá uma cópia dessa variável por encadeamento, em vez de uma cópia por par (encadeamento, instância).

Exemplo:

void debug_counter() {
    thread_local int count = 0;
    Logger::log("This function has been called %d times by this thread", ++count);
}

Reatribuindo objetos de thread #

Podemos criar objetos de thread vazios e atribuir trabalho a eles mais tarde.

Se atribuirmos um objeto thread a outro thread ativo, joinable, std::terminate será chamado automaticamente antes que o thread seja substituído.

#include <thread>

void foo()
{
    std::this_thread::sleep_for(std::chrono::seconds(3));
}
//create 100 thread objects that do nothing
std::thread executors[100];

// Some code

// I want to create some threads now

for (int i = 0;i < 100;i++)
{
    // If this object doesn't have a thread assigned
    if (!executors[i].joinable())
         executors[i] = std::thread(foo);
}
