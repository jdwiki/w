Sintaxe #

  • [[detalhes]]: atributo simples sem argumentos

  • [[details(arguments)]]: Atributo com argumentos

  • __attribute(details): GCC/Clang/IBM não padrão específico

  • __declspec(detalhes): Específico MSVC não padrão

[[Cair em]] #

Sempre que um case termina em um switch, o código do próximo case será executado. Este último pode ser evitado usando a instrução ´break`. Como esse chamado comportamento de fallthrough pode introduzir bugs quando não pretendidos, vários compiladores e analisadores estáticos avisam sobre isso.

A partir do C++17, um atributo padrão foi introduzido para indicar que o aviso não é necessário quando o código deve falhar. Os compiladores podem dar avisos com segurança quando um caso é encerrado sem break ou [[fallthrough]] e tem pelo menos uma instrução.

switch(input) {
    case 2011:
    case 2014:
    case 2017:
        std::cout << "Using modern C++" << std::endl;
        [[fallthrough]]; // > No warning
    case 1998:
    case 2003:
        standard = input;
}

Veja a proposta para exemplos mais detalhados de como o [[falta de ajuda]] pode ser usado.

[[reprovado]] e [[reprovado(“motivo”)]] #

O C++14 introduziu uma maneira padrão de depreciar funções por meio de atributos. [[deprecated]] pode ser usado para indicar que uma função está obsoleta. [[deprecated("reason")]] permite adicionar um motivo específico que pode ser mostrado pelo compilador.

void function(std::unique_ptr<A> &&a);

// Provides specific message which helps other programmers fixing there code
[[deprecated("Use the variant with unique_ptr instead, this function will be removed in the next release")]]
void function(std::auto_ptr<A> a);

// No message, will result in generic warning if called.
[[deprecated]]
void function(A *a);

Este atributo pode ser aplicado a:

  • a declaração de uma classe
  • um nome typedef
  • uma variável
  • um membro de dados não estático
  • uma função
  • uma enumeração
  • uma especialização de modelo

(ref. rascunho padrão c++14: 7.6.5 Atributo obsoleto)

[[nodiscard]] #

O atributo [[nodiscard]] pode ser usado para indicar que o valor de retorno de uma função não deve ser ignorado quando você faz uma chamada de função. Se o valor de retorno for ignorado, o compilador deve dar um aviso sobre isso. O atributo pode ser adicionado a:

  • Uma definição de função
  • Um tipo

Adicionar o atributo a um tipo tem o mesmo comportamento que adicionar o atributo a cada função que retorna esse tipo.

template<typename Function>
[[nodiscard]] Finally<std::decay_t<Function>> onExit(Function &&f);

void f(int &i) {
    assert(i == 0);                    // Just to make comments clear!
    ++i;                               // i == 1
    auto exit1 = onExit([&i]{ --i; }); // Reduce by 1 on exiting f()
    ++i;                               // i == 2
    onExit([&i]{ --i; });              // BUG: Reducing by 1 directly
                                       //      Compiler warning expected
    std::cout << i << std::end;        // Expected: 2, Real: 1
}

Consulte a proposta para exemplos mais detalhados de como [[nodiscard]] pode ser usado .

Observação: os detalhes de implementação de Finally/onExit são omitidos no exemplo, consulte [Finally/ScopeExit](https://www.wikiod.com/pt/docs/c%2b%2b/1320/raii- resource-acquisition-is-initialization/4551/finally-scopeexit).

[[noreturn]] #

O C++11 introduziu o atributo [[noreturn]]. Ele pode ser usado para uma função para indicar que a função não retorna ao chamador executando uma instrução return ou chegando ao final de seu corpo (é importante notar que isso não se aplica a funções void, como eles retornam ao chamador, eles simplesmente não retornam nenhum valor). Tal função pode terminar chamando std::terminate ou std::exit, ou lançando uma exceção. Também vale a pena notar que tal função pode retornar executando longjmp.

Por exemplo, a função abaixo sempre lançará uma exceção ou chamará std::terminate, então é uma boa candidata para [[noreturn]]:

[[noreturn]] void ownAssertFailureHandler(std::string message) {
    std::cerr << message << std::endl;
    if (THROW_EXCEPTION_ON_ASSERT)
        throw AssertException(std::move(message));
    std::terminate();
}

Esse tipo de funcionalidade permite que o compilador finalize uma função sem uma instrução de retorno se souber que o código nunca será executado. Aqui, como a chamada para ownAssertFailureHandler (definida acima) no código abaixo nunca retornará, o compilador não precisa adicionar código abaixo dessa chamada:

std::vector<int> createSequence(int end) {
    if (end > 0) {
        std::vector<int> sequence;
        sequence.reserve(end+1);
        for (int i = 0; i <= end; ++i)
            sequence.push_back(i);
        return sequence;
    }
    ownAssertFailureHandler("Negative number passed to createSequence()"s);
    // return std::vector<int>{}; //< Not needed because of [[noreturn]]
}

É um comportamento indefinido se a função realmente retornar, então o seguinte não é permitido:

[[noreturn]] void assertPositive(int number) {
    if (number >= 0)
        return;
    else
        ownAssertFailureHandler("Positive number expected"s); //< [[noreturn]]
}

Observe que o [[noreturn]] é usado principalmente em funções void. No entanto, isso não é um requisito, permitindo que as funções sejam usadas na programação genérica:

template<class InconsistencyHandler>
double fortyTwoDivideBy(int i) {
    if (i == 0)
         i = InconsistencyHandler::correct(i);
    return 42. / i;
}

struct InconsistencyThrower {
    static [[noreturn]] int correct(int i) { ownAssertFailureHandler("Unknown inconsistency"s); }
}

struct InconsistencyChangeToOne {
    static int correct(int i) { return 1; }
}

double fortyTwo = fortyTwoDivideBy<InconsistencyChangeToOne>(0);
double unreachable = fortyTwoDivideBy<InconsistencyThrower>(0);

As seguintes funções de biblioteca padrão têm este atributo:

  • std:: abortar
  • std::sair
  • std::quick_exit
  • std::inesperado
  • std::terminar
  • std::rethrow_exception
  • std::throw_with_nested
  • std::nested_exception::rethrow_nested

[[talvez_unused]] #

O atributo [[maybe_unused]] é criado para indicar no código que determinada lógica pode não ser usada. Isso se muitas vezes estiver vinculado a condições de pré-processador em que isso pode ser usado ou não. Como os compiladores podem dar avisos sobre variáveis ​​não utilizadas, essa é uma forma de suprimi-las indicando a intenção.

Um exemplo típico de variáveis ​​que são necessárias em compilações de depuração enquanto desnecessárias em produção são valores de retorno que indicam sucesso. Nas compilações de depuração, a condição deve ser declarada, embora na produção essas declarações tenham sido removidas.

[[maybe_unused]] auto mapInsertResult = configuration.emplace("LicenseInfo", stringifiedLicenseInfo);
assert(mapInsertResult.second); // We only get called during startup, so we can't be in the map

Um exemplo mais complexo são diferentes tipos de funções auxiliares que estão em um namespace sem nome. Se essas funções não forem usadas durante a compilação, um compilador poderá avisá-las. Idealmente, você gostaria de protegê-los com as mesmas tags de pré-processador que o chamador, embora isso possa se tornar complexo, o atributo [[maybe_unused]] é uma alternativa mais sustentável.

namespace {
    [[maybe_unused]] std::string createWindowsConfigFilePath(const std::string &relativePath);
    // TODO: Reuse this on BSD, MAC ...
    [[maybe_unused]] std::string createLinuxConfigFilePath(const std::string &relativePath);
}

std::string createConfigFilePath(const std::string &relativePath) {
#if OS == "WINDOWS"
      return createWindowsConfigFilePath(relativePath);
#elif OS == "LINUX"
      return createLinuxConfigFilePath(relativePath);
#else
#error "OS is not yet supported"
#endif
}

Veja a proposta para exemplos mais detalhados de como [[maybe_unused]] pode ser usado.
