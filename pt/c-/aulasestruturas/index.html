Sintaxe #

  • variável.membro_var = constante;
  • variável.função_membro();
  • ponteiro_variável->var_membro = constante;
  • ponteiro_variável->função_membro();

Observe que a somente diferença entre as palavras-chave struct e class é que, por padrão, as variáveis ​​de membro, funções de membro e classes base de um struct são public, enquanto em uma class eles são privados. Os programadores de C++ tendem a chamá-la de classe se ela tiver construtores e destruidores e a capacidade de impor suas próprias invariantes; ou uma estrutura se for apenas uma simples coleção de valores, mas a própria linguagem C++ não faz distinção.

Noções básicas de aula #

Uma classe é um tipo definido pelo usuário. Uma classe é introduzida com a palavra-chave class, struct ou union. No uso coloquial, o termo “classe” geralmente se refere apenas a classes não sindicais.

Uma classe é uma coleção de membros da classe, que podem ser:

  • variáveis ​​de membro (também chamadas de “campos”),
  • funções de membro (também chamadas de “métodos”),
  • tipos de membros ou typedefs (por exemplo, “classes aninhadas”),
  • templates de membros (de qualquer tipo: template de variável, função, classe ou alias)

As palavras-chave class e struct, chamadas chaves de classe, são amplamente intercambiáveis, exceto que o especificador de acesso padrão para membros e bases é “privado” para uma classe declarada com a chave class e “público” para uma classe declarada com a chave struct ou union (cf. Modificadores de acesso).

Por exemplo, os seguintes snippets de código são idênticos:

struct Vector
{
    int x;
    int y;
    int z;
};
// are equivalent to
class Vector
{
public:
    int x;
    int y;
    int z;
};

Ao declarar uma classe` um novo tipo é adicionado ao seu programa, e é possível instanciar objetos dessa classe por

Vector my_vector;

Os membros de uma classe são acessados ​​usando sintaxe de ponto.

my_vector.x = 10;
my_vector.y = my_vector.x + 1; // my_vector.y = 11;
my_vector.z = my_vector.y - 4; // my:vector.z = 7;

Classes e estruturas finais #

A derivação de uma classe pode ser proibida com o especificador final. Vamos declarar uma classe final:

class A final {
};

Agora, qualquer tentativa de subclasse causará um erro de compilação:

// Compilation error: cannot derive from final class:
class B : public A {
};

A classe final pode aparecer em qualquer lugar na hierarquia de classes:

class A {
};

// OK.
class B final : public A {
};

// Compilation error: cannot derive from final class B.
class C : public B {
};

Especificadores de acesso #

Existem três palavras-chave que atuam como especificadores de acesso. Eles limitam o acesso aos membros da classe seguindo o especificador, até que outro especificador altere o nível de acesso novamente:

Palavra-chave Descrição
público Todos têm acesso
protegido Apenas a classe em si, classes derivadas e amigos têm acesso
privado Somente a própria turma e amigos tem acesso

Quando o tipo é definido usando a palavra-chave class, o especificador de acesso padrão é private, mas se o tipo é definido usando a palavra-chave struct, o especificador de acesso padrão é public:

struct MyStruct { int x; };
class MyClass { int x; };

MyStruct s;
s.x = 9; // well formed, because x is public

MyClass c;
c.x = 9; // ill-formed, because x is private

Os especificadores de acesso são usados ​​principalmente para limitar o acesso a campos e métodos internos e forçar o programador a usar uma interface específica, por exemplo, para forçar o uso de getters e setters em vez de referenciar uma variável diretamente:

class MyClass {

public: /* Methods: */

    int x() const noexcept { return m_x; }
    void setX(int const x) noexcept { m_x = x; }

private: /* Fields: */

    int m_x;

};

Usar protected é útil para permitir que certas funcionalidades do tipo sejam acessíveis apenas para as classes derivadas, por exemplo, no código a seguir, o método calculateValue() só é acessível para classes derivadas da classe base Plus2Base , como Quarenta e dois:

struct Plus2Base {
    int value() noexcept { return calculateValue() + 2; }
protected: /* Methods: */
    virtual int calculateValue() noexcept = 0;
};
struct FortyTwo: Plus2Base {
protected: /* Methods: */
    int calculateValue() noexcept final override { return 40; }
};

Observe que a palavra-chave friend pode ser usada para adicionar exceções de acesso a funções ou tipos para acessar membros protegidos e privados.

As palavras-chave public, protected e private também podem ser usadas para conceder ou limitar o acesso aos subobjetos da classe base. Veja o exemplo de Herança.

Herança #

Classes/structs podem ter relações de herança.

Se uma classe/struct B herda de uma classe/struct A, isso significa que B tem como pai A. Dizemos que B é uma classe/estrutura derivada de A, e A é a classe/estrutura base.

struct A
{
public:
    int p1;
protected:
    int p2;
private:
    int p3;
};

//Make B inherit publicly (default) from A
struct B : A
{
};

Existem 3 formas de herança para uma classe/struct:

  • público
  • privado
  • protegido

Observe que a herança padrão é a mesma que a visibilidade padrão dos membros: public se você usar a palavra-chave struct e private para a palavra-chave class.

É até possível ter uma class derivada de uma struct (ou vice-versa). Neste caso, a herança padrão é controlada pelo filho, então uma struct que deriva de uma class será padronizada para herança pública, e uma class que deriva de uma struct terá herança privada por padrão.

herança pública:

struct B : public A // or just `struct B : A`
{
    void foo()
    {
        p1 = 0; //well formed, p1 is public in B
        p2 = 0; //well formed, p2 is protected in B
        p3 = 0; //ill formed, p3 is private in A
    }
};

B b;
b.p1 = 1; //well formed, p1 is public
b.p2 = 1; //ill formed, p2 is protected
b.p3 = 1; //ill formed, p3 is inaccessible

herança privada:

struct B : private A
{
    void foo()
    {
        p1 = 0; //well formed, p1 is private in B
        p2 = 0; //well formed, p2 is private in B
        p3 = 0; //ill formed, p3 is private in A
    }
};

B b;
b.p1 = 1; //ill formed, p1 is private
b.p2 = 1; //ill formed, p2 is private
b.p3 = 1; //ill formed, p3 is inaccessible

herança protegida:

struct B : protected A
{
    void foo()
    {
        p1 = 0; //well formed, p1 is protected in B
        p2 = 0; //well formed, p2 is protected in B
        p3 = 0; //ill formed, p3 is private in A
    }
};

B b;
b.p1 = 1; //ill formed, p1 is protected
b.p2 = 1; //ill formed, p2 is protected
b.p3 = 1; //ill formed, p3 is inaccessible

Observe que, embora a herança protegida seja permitida, o uso real dela é raro. Uma instância de como a herança protegida é usada na aplicação é a especialização parcial da classe base (geralmente chamada de “polimorfismo controlado”).

Quando a POO era relativamente nova, a herança (pública) era frequentemente dita para modelar um relacionamento “IS-A”. Ou seja, a herança pública está correta somente se uma instância da classe derivada é também uma instância da classe base.

Isso foi posteriormente refinado no Princípio de Substituição de Liskov: a herança pública só deve ser usada quando/se uma instância da classe derivada puder ser substituída por uma instância da classe derivada. classe base em qualquer circunstância possível (e ainda faz sentido).

A herança privada normalmente incorpora um relacionamento completamente diferente: “é implementado em termos de” (às vezes chamado de relacionamento “HAS-A”). Por exemplo, uma classe Stack pode herdar privadamente de uma classe Vector. A herança privada tem uma semelhança muito maior com a agregação do que com a herança pública.

A herança protegida quase nunca é usada e não há um acordo geral sobre que tipo de relacionamento ela incorpora.

Amizade #

A [palavra-chave] amigo (https://www.wikiod.com/pt/docs/c%2b%2b/4891/keyword) é usada para dar a outras classes e funções acesso a membros privados e protegidos da classe, mesmo que sejam definido fora do escopo da classe.

class Animal{
private:
    double weight;
    double height;
public:
    friend void printWeight(Animal animal);
    friend class AnimalPrinter;
    // A common use for a friend function is to overload the operator<< for streaming. 
    friend std::ostream& operator<<(std::ostream& os, Animal animal);
};

void printWeight(Animal animal)
{
    std::cout << animal.weight << "\n";
}

class AnimalPrinter
{
public:
    void print(const Animal& animal)
    {
        // Because of the `friend class AnimalPrinter;" declaration, we are
        // allowed to access private members here.
        std::cout << animal.weight << ", " << animal.height << std::endl;
    }
}

std::ostream& operator<<(std::ostream& os, Animal animal)
{
    os << "Animal height: " << animal.height << "\n";
    return os;
}

int main() {
    Animal animal = {10, 5};
    printWeight(animal);

    AnimalPrinter aPrinter;
    aPrinter.print(animal);

    std::cout << animal;
}

10
10, 5
Animal height: 5

Herança virtual #

Ao usar herança, você pode especificar a palavra-chave virtual:

struct A{};
struct B: public virtual A{};

Quando a classe B tem a base virtual A significa que A ** residirá na classe mais derivada** da árvore de herança e, portanto, essa classe mais derivada também é responsável por inicializar essa base virtual:

struct A
{
    int member;
    A(int param)
    {
        member = param;
    }
};

struct B: virtual A
{
    B(): A(5){}
};

struct C: B
{
    C(): /*A(88)*/ {}
};

void f()
{
    C object; //error since C is not initializing it's indirect virtual base `A`
}

Se descomentarmos /*A(88)*/, não obteremos nenhum erro, pois C agora está inicializando sua base virtual indireta A.

Observe também que quando estamos criando a variável object, a classe mais derivada é C, então C é responsável por criar (chamando o construtor de) A e, portanto, o valor de A::member é 88 , não 5 (como seria se estivéssemos criando um objeto do tipo B).

É útil ao resolver o problema do diamante.:

  A                                        A   A
 / \                                       |   |
B   C                                      B   C
 \ /                                        \ /
  D                                          D
virtual inheritance                   normal inheritance

B e C herdam de A, e D herda de B e C, então há 2 instâncias de A em D! Isso resulta em ambiguidade quando você está acessando o membro de A através de D, pois o compilador não tem como saber de qual classe você deseja acessar aquele membro (aquela que B herda, ou aquela que é herdado por C?).

A herança virtual resolve este problema: Como a base virtual reside apenas na maioria dos objetos derivados, haverá apenas uma instância de A em D.

struct A
{
    void foo() {}
};

struct B : public /*virtual*/ A {};
struct C : public /*virtual*/ A {};

struct D : public B, public C
{
    void bar()
    {
        foo(); //Error, which foo? B::foo() or C::foo()? - Ambiguous
    }
};

A remoção dos comentários resolve a ambiguidade.

Acessando os membros da turma #

Para acessar variáveis ​​de membro e funções de membro de um objeto de uma classe, o operador . é usado:

struct SomeStruct {
  int a;
  int b;
  void foo() {}
};

SomeStruct var;
// Accessing member variable a in var.
std::cout << var.a << std::endl;
// Assigning member variable b in var.
var.b = 1;
// Calling a member function.
var.foo();

Ao acessar os membros de uma classe através de um ponteiro, o operador -> é comumente usado. Alternativamente, a instância pode ser desreferenciada e o operador . usado, embora isso seja menos comum:

struct SomeStruct {
  int a;
  int b;
  void foo() {}
};

SomeStruct var;
SomeStruct *p = &var;
// Accessing member variable a in var via pointer.
std::cout << p->a << std::endl;
std::cout << (*p).a << std::endl;
// Assigning member variable b in var via pointer.
p->b = 1;
(*p).b = 1;
// Calling a member function via a pointer.
p->foo();
(*p).foo();

Ao acessar membros de classe estáticos, o operador :: é usado, mas no nome da classe em vez de uma instância dela. Alternativamente, o membro estático pode ser acessado de uma instância ou um ponteiro para uma instância usando o operador . ou ->, respectivamente, com a mesma sintaxe de acesso a membros não estáticos.

struct SomeStruct {
  int a;
  int b;
  void foo() {}

  static int c;
  static void bar() {}
};
int SomeStruct::c;

SomeStruct var;
SomeStruct* p = &var;
// Assigning static member variable c in struct SomeStruct.
SomeStruct::c = 5;
// Accessing static member variable c in struct SomeStruct, through var and p.
var.a = var.c;
var.b = p->c;
// Calling a static member function.
SomeStruct::bar();
var.bar();
p->bar();

Fundo #

O operador -> é necessário porque o operador de acesso de membro . tem precedência sobre o operador de desreferenciamento *.

Seria de esperar que *p.a desreferenciasse p (resultando em uma referência ao objeto para o qual p está apontando) e então acessando seu membro a. Mas, na verdade, ele tenta acessar o membro a de p e depois desreferenciá-lo. Ou seja *p.a é equivalente a *(p.a). No exemplo acima, isso resultaria em um erro do compilador devido a dois fatos: Primeiro, p é um ponteiro e não possui um membro a. Segundo, a é um número inteiro e, portanto, não pode ser desreferenciado.

A solução raramente usada para este problema seria controlar explicitamente a precedência: (*p).a

Em vez disso, o operador -> é quase sempre usado. É um atalho para primeiro desreferenciar o ponteiro e depois acessá-lo. Ou seja (*p).a é exatamente o mesmo que p->a.

O operador :: é o operador de escopo, usado da mesma maneira que acessar um membro de um namespace. Isso ocorre porque um membro de classe estático é considerado no escopo dessa classe, mas não é considerado um membro de instâncias dessa classe. O uso de . e -> normais também é permitido para membros estáticos, apesar de não serem membros de instância, por razões históricas; isso é útil para escrever código genérico em modelos, pois o chamador não precisa se preocupar se uma determinada função de membro é estática ou não estática.

Herança privada: restringindo a interface da classe base #

A herança privada é útil quando é necessário restringir a interface pública da classe:

class A {
public:
    int move();
    int turn();
};

class B : private A {
public:
    using A::turn; 
};

B b;
b.move();  // compile error
b.turn();  // OK

Essa abordagem evita com eficiência um acesso aos métodos públicos A lançando para o ponteiro ou referência A:

B b; 
A& a = static_cast<A&>(b); // compile error

No caso de herança pública, essa conversão fornecerá acesso a todos os métodos públicos A, apesar de maneiras alternativas de evitar isso em B derivado, como ocultar:

class B : public A {
private:
    int move();  
};

ou privado usando:

class B : public A {
private:
    using A::move;  
};

então para ambos os casos é possível:

B b;
A& a = static_cast<A&>(b); // OK for public inheritance
a.move(); // OK

Classes/estruturas aninhadas #

Uma class ou struct também pode conter outra definição class/struct dentro de si, que é chamada de “classe aninhada”; nesta situação, a classe que contém é referida como a “classe envolvente”. A definição de classe aninhada é considerada um membro da classe delimitadora, mas de outra forma é separada.

struct Outer {
    struct Inner { };
};

De fora da classe delimitadora, as classes aninhadas são acessadas usando o operador de escopo. De dentro da classe envolvente, no entanto, as classes aninhadas podem ser usadas sem qualificadores:

struct Outer {
    struct Inner { };

    Inner in;
};

// ...

Outer o;
Outer::Inner i = o.in;

Assim como uma class/struct não aninhada, as funções-membro e as variáveis ​​estáticas podem ser definidas dentro de uma classe aninhada ou no namespace delimitador. No entanto, eles não podem ser definidos dentro da classe delimitadora, pois ela é considerada uma classe diferente da classe aninhada.

// Bad.
struct Outer {
    struct Inner {
        void do_something();
    };

    void Inner::do_something() {}
};


// Good.
struct Outer {
    struct Inner {
        void do_something();
    };

};

void Outer::Inner::do_something() {}

Assim como as classes não aninhadas, as classes aninhadas podem ser declaradas e definidas posteriormente, desde que sejam definidas antes de serem usadas diretamente.

class Outer {
    class Inner1;
    class Inner2;

    class Inner1 {};

    Inner1 in1;
    Inner2* in2p;

  public:
    Outer();
    ~Outer();
};

class Outer::Inner2 {};

Outer::Outer() : in1(Inner1()), in2p(new Inner2) {}
Outer::~Outer() {
    if (in2p) { delete in2p; }
}

Antes do C++11, as classes aninhadas só tinham acesso a nomes de tipo, membros static e enumeradores da classe delimitadora; todos os outros membros definidos na classe delimitadora estavam fora dos limites.

A partir do C++11, as classes aninhadas e seus membros são tratados como se fossem amigos da classe envolvente, podendo acessar todos os seus membros, de acordo com as regras usuais de acesso; se os membros da classe aninhada exigirem a capacidade de avaliar um ou mais membros não estáticos da classe envolvente, eles devem, portanto, receber uma instância:

class Outer {
    struct Inner {
        int get_sizeof_x() {
            return sizeof(x); // Legal (C++11): x is unevaluated, so no instance is required.
        }

        int get_x() {
            return x; // Illegal: Can't access non-static member without an instance.
        }

        int get_x(Outer& o) {
            return o.x; // Legal (C++11): As a member of Outer, Inner can access private members.
        }
    };

    int x;
};

Por outro lado, a classe delimitadora não é tratada como amiga da classe aninhada e, portanto, não pode acessar seus membros privados sem receber permissão explicitamente.

class Outer {
    class Inner {
        // friend class Outer;

        int x;
    };

    Inner in;

  public:
    int get_x() {
        return in.x; // Error: int Outer::Inner::x is private.
        // Uncomment "friend" line above to fix.
    }
};

Amigos de uma classe aninhada não são automaticamente considerados amigos da classe delimitadora; se eles também precisam ser amigos da classe anexa, isso deve ser declarado separadamente. Por outro lado, como a classe delimitadora não é automaticamente considerada amiga da classe aninhada, os amigos da classe delimitadora também não serão considerados amigos da classe aninhada.

class Outer {
    friend void barge_out(Outer& out, Inner& in);

    class Inner {
        friend void barge_in(Outer& out, Inner& in);

        int i;
    };

    int o;
};

void barge_in(Outer& out, Outer::Inner& in) {
    int i = in.i;  // Good.
    int o = out.o; // Error: int Outer::o is private.
}

void barge_out(Outer& out, Outer::Inner& in) {
    int i = in.i;  // Error: int Outer::Inner::i is private.
    int o = out.o; // Good.
}

Tal como acontece com todos os outros membros da classe, as classes aninhadas só podem ser nomeadas de fora da classe se tiverem acesso público. No entanto, você tem permissão para acessá-los independentemente do modificador de acesso, desde que não os nomeie explicitamente.

class Outer {
    struct Inner {
        void func() { std::cout << "I have no private taboo.\n"; }
    };

  public:
    static Inner make_Inner() { return Inner(); }
};

// ...

Outer::Inner oi; // Error: Outer::Inner is private.

auto oi = Outer::make_Inner(); // Good.
oi.func();                     // Good.
Outer::make_Inner().func();    // Good.

Você também pode criar um alias de tipo para uma classe aninhada. Se um alias de tipo estiver contido na classe delimitadora, o tipo aninhado e o alias de tipo poderão ter modificadores de acesso diferentes. Se o alias de tipo estiver fora da classe delimitadora, ele exigirá que a classe aninhada, ou um typedef dela, seja pública.

class Outer {
    class Inner_ {};

  public:
    typedef Inner_ Inner;
};

typedef Outer::Inner  ImOut; // Good.
typedef Outer::Inner_ ImBad; // Error.

// ...

Outer::Inner  oi; // Good.
Outer::Inner_ oi; // Error.
ImOut         oi; // Good.

Assim como outras classes, as classes aninhadas podem derivar ou ser derivadas de outras classes.

struct Base {};

struct Outer {
    struct Inner : Base {};
};

struct Derived : Outer::Inner {};

Isso pode ser útil em situações em que a classe delimitadora é derivada de outra classe, permitindo que o programador atualize a classe aninhada conforme necessário. Isso pode ser combinado com um typedef para fornecer um nome consistente para cada classe aninhada de classe delimitadora:

class BaseOuter {
    struct BaseInner_ {
        virtual void do_something() {}
        virtual void do_something_else();
    } b_in;

  public:
    typedef BaseInner_ Inner;

    virtual ~BaseOuter() = default;

    virtual Inner& getInner() { return b_in; }
};

void BaseOuter::BaseInner_::do_something_else() {}

// ---

class DerivedOuter : public BaseOuter {
    // Note the use of the qualified typedef; BaseOuter::BaseInner_ is private.
    struct DerivedInner_ : BaseOuter::Inner {
        void do_something() override {}
        void do_something_else() override;
    } d_in;

  public:
    typedef DerivedInner_ Inner;

    BaseOuter::Inner& getInner() override { return d_in; }
};

void DerivedOuter::DerivedInner_::do_something_else() {}

// ...

// Calls BaseOuter::BaseInner_::do_something();
BaseOuter* b = new BaseOuter;
BaseOuter::Inner& bin = b->getInner();
bin.do_something();
b->getInner().do_something();

// Calls DerivedOuter::DerivedInner_::do_something();
BaseOuter* d = new DerivedOuter;
BaseOuter::Inner& din = d->getInner();
din.do_something();
d->getInner().do_something();

No caso acima, BaseOuter e DerivedOuter fornecem o tipo de membro Inner, como BaseInner_ e DerivedInner_, respectivamente. Isso permite que tipos aninhados sejam derivados sem quebrar a interface da classe delimitadora e permite que o tipo aninhado seja usado polimorficamente.

Tipos de membros e aliases #

Uma class ou struct também pode definir aliases de tipo de membro, que são aliases de tipo contidos e tratados como membros da própria classe.

struct IHaveATypedef {
    typedef int MyTypedef;
};

struct IHaveATemplateTypedef {
    template<typename T>
    using MyTemplateTypedef = std::vector<T>;
};

Como membros estáticos, esses typedefs são acessados ​​usando o operador de escopo, ::.

IHaveATypedef::MyTypedef i = 5; // i is an int.

IHaveATemplateTypedef::MyTemplateTypedef<int> v; // v is a std::vector<int>.

Assim como acontece com os aliases de tipo normal, cada alias de tipo de membro pode se referir a qualquer tipo definido ou alias antes, mas não depois, de sua definição. Da mesma forma, um typedef fora da definição de classe pode se referir a qualquer typedef acessível dentro da definição de classe, desde que venha após a definição de classe.

template<typename T>
struct Helper {
    T get() const { return static_cast<T>(42); }
};

struct IHaveTypedefs {
//    typedef MyTypedef NonLinearTypedef; // Error if uncommented.
    typedef int MyTypedef;
    typedef Helper<MyTypedef> MyTypedefHelper;
};

IHaveTypedefs::MyTypedef        i; // x_i is an int.
IHaveTypedefs::MyTypedefHelper hi; // x_hi is a Helper<int>.

typedef IHaveTypedefs::MyTypedef TypedefBeFree;
TypedefBeFree ii;                  // ii is an int.

Os aliases de tipo de membro podem ser declarados com qualquer nível de acesso e respeitarão o modificador de acesso apropriado.

class TypedefAccessLevels {
    typedef int PrvInt;

  protected:
    typedef int ProInt;

  public:
    typedef int PubInt;
};

TypedefAccessLevels::PrvInt prv_i; // Error: TypedefAccessLevels::PrvInt is private.
TypedefAccessLevels::ProInt pro_i; // Error: TypedefAccessLevels::ProInt is protected.
TypedefAccessLevels::PubInt pub_i; // Good.

class Derived : public TypedefAccessLevels {
    PrvInt prv_i; // Error: TypedefAccessLevels::PrvInt is private.
    ProInt pro_i; // Good.
    PubInt pub_i; // Good.
};

Isso pode ser usado para fornecer um nível de abstração, permitindo que o designer de uma classe altere seu funcionamento interno sem quebrar o código que depende dele.

class Something {
    friend class SomeComplexType;

    short s;
    // ...

  public:
    typedef SomeComplexType MyHelper;

    MyHelper get_helper() const { return MyHelper(8, s, 19.5, "shoe", false); }

    // ...
};

// ...

Something s;
Something::MyHelper hlp = s.get_helper();

Nesta situação, se a classe auxiliar for alterada de SomeComplexType para algum outro tipo, apenas a declaração typedef e friend precisariam ser modificadas; contanto que a classe auxiliar forneça a mesma funcionalidade, qualquer código que a use como Something::MyHelper em vez de especificá-la pelo nome normalmente ainda funcionará sem nenhuma modificação. Dessa forma, minimizamos a quantidade de código que precisa ser modificada quando a implementação subjacente é alterada, de modo que o nome do tipo só precise ser alterado em um local.

Isso também pode ser combinado com decltype, se assim o desejar.

class SomethingElse {
    AnotherComplexType<bool, int, SomeThirdClass> helper;

  public:
    typedef decltype(helper) MyHelper;

  private:
    InternalVariable<MyHelper> ivh;

    // ...

  public:
    MyHelper& get_helper() const { return helper; }

    // ...
};

Nesta situação, alterar a implementação de SomethingElse::helper mudará automaticamente o typedef para nós, devido ao decltype. Isso minimiza o número de modificações necessárias quando queremos alterar o helper, o que minimiza o risco de erro humano.

Como em tudo, no entanto, isso pode ser levado longe demais. Se o nome do tipo for usado apenas uma ou duas vezes internamente e zero vezes externamente, por exemplo, não há necessidade de fornecer um alias para ele. Se for usado centenas ou milhares de vezes ao longo de um projeto, ou se tiver um nome longo o suficiente, pode ser útil fornecê-lo como um typedef em vez de sempre usá-lo em termos absolutos. É preciso equilibrar a compatibilidade e a conveniência com a quantidade de ruído desnecessário criado.

Isso também pode ser usado com classes de modelo, para fornecer acesso aos parâmetros de modelo de fora da classe.

template<typename T>
class SomeClass {
    // ...

  public:
    typedef T MyParam;
    MyParam getParam() { return static_cast<T>(42); }
};

template<typename T>
typename T::MyParam some_func(T& t) {
    return t.getParam();
}

SomeClass<int> si;
int i = some_func(si);

Isso é comumente usado com contêineres, que geralmente fornecem seu tipo de elemento e outros tipos auxiliares, como aliases de tipo de membro. A maioria dos contêineres na biblioteca padrão C++, por exemplo, fornece os 12 tipos auxiliares a seguir, juntamente com quaisquer outros tipos especiais de que possam precisar.

template<typename T>
class SomeContainer {
    // ...

  public:
    // Let's provide the same helper types as most standard containers.
    typedef T                                     value_type;
    typedef std::allocator<value_type>            allocator_type;
    typedef value_type&                           reference;
    typedef const value_type&                     const_reference;
    typedef value_type*                           pointer;
    typedef const value_type*                     const_pointer;
    typedef MyIterator<value_type>                iterator;
    typedef MyConstIterator<value_type>           const_iterator;
    typedef std::reverse_iterator<iterator>       reverse_iterator;
    typedef std::reverse_iterator<const_iterator> const_reverse_iterator;
    typedef size_t                                size_type;
    typedef ptrdiff_t                             difference_type;
};

Antes do C++11, também era comumente usado para fornecer um tipo de “modelo typedef”, já que o recurso ainda não estava disponível; estes tornaram-se um pouco menos comuns com a introdução de templates de alias, mas ainda são úteis em algumas situações (e são combinados com templates de alias em outras situações, o que pode ser muito útil para obter componentes individuais de um tipo complexo como um ponteiro de função ). Eles geralmente usam o nome type para seu alias de tipo.

template<typename T>
struct TemplateTypedef {
    typedef T type;
}

TemplateTypedef<int>::type i; // i is an int.

Isso era frequentemente usado com tipos com vários parâmetros de modelo, para fornecer um alias que define um ou mais dos parâmetros.

template<typename T, size_t SZ, size_t D>
class Array { /* ... */ };

template<typename T, size_t SZ>
struct OneDArray {
    typedef Array<T, SZ, 1> type;
};

template<typename T, size_t SZ>
struct TwoDArray {
    typedef Array<T, SZ, 2> type;
};

template<typename T>
struct MonoDisplayLine {
    typedef Array<T, 80, 1> type;
};

OneDArray<int, 3>::type     arr1i; // arr1i is an Array<int, 3, 1>.
TwoDArray<short, 5>::type   arr2s; // arr2s is an Array<short, 5, 2>.
MonoDisplayLine<char>::type arr3c; // arr3c is an Array<char, 80, 1>.

Herança múltipla #

Além da herança simples:

class A {};
class B : public A {};

Você também pode ter herança múltipla:

class A {};
class B {};
class C : public A, public B {};

C agora herdará de A e B ao mesmo tempo.

Observação: isso pode levar a ambiguidade se os mesmos nomes forem usados ​​em várias classs ou structs herdadas. Tenha cuidado!

Ambiguidade na herança múltipla

A herança múltipla pode ser útil em certos casos, mas, às vezes, um tipo estranho de problema ocorre ao usar a herança múltipla.

Por exemplo: Duas classes base possuem funções com o mesmo nome que não são sobrescritas na classe derivada e se você escrever código para acessar essa função usando o objeto da classe derivada, o compilador mostra erro porque não pode determinar qual função chamar. Aqui está um código para este tipo de ambiguidade em herança múltipla.

class base1
{
  public:
     void funtion( )
     { //code for base1 function }  
};
class base2
{
    void function( )
     { // code for base2 function } 
};

class derived : public base1, public base2
{
    
};

int main()
{
    derived obj;
    
  // Error because compiler can't figure out which function to call 
  //either function( ) of base1 or base2 .   
    obj.function( )  
}

Mas, esse problema pode ser resolvido usando a função de resolução de escopo para especificar qual função classificar base1 ou base2:

int main()
{
    obj.base1::function( );  // Function of class base1 is called. 
    obj.base2::function( );  // Function of class base2 is called.
}

Membros de classe estáticos #

Uma classe também pode ter membros static, que podem ser variáveis ​​ou funções. Eles são considerados no escopo da classe, mas não são tratados como membros normais; eles têm duração de armazenamento estático (existem do início ao fim do programa), não estão vinculados a uma instância específica da classe e existe apenas uma cópia para a classe inteira.

class Example {
    static int num_instances;      // Static data member (static member variable).
    int i;                         // Non-static member variable.

  public:
    static std::string static_str; // Static data member (static member variable).
    static int static_func();      // Static member function.

    // Non-static member functions can modify static member variables.
    Example() { ++num_instances; }
    void set_str(const std::string& str);
};

int         Example::num_instances;
std::string Example::static_str = "Hello.";

// ...

Example one, two, three;
// Each Example has its own "i", such that:
//  (&one.i != &two.i)
//  (&one.i != &three.i)
//  (&two.i != &three.i).
// All three Examples share "num_instances", such that:
//  (&one.num_instances == &two.num_instances)
//  (&one.num_instances == &three.num_instances)
//  (&two.num_instances == &three.num_instances)

As variáveis ​​de membro estático não são consideradas definidas dentro da classe, apenas declaradas, e assim têm sua definição fora da definição da classe; o programador tem permissão, mas não é obrigatório, para inicializar variáveis ​​estáticas em sua definição. Ao definir as variáveis ​​de membro, a palavra-chave static é omitida.

class Example {
    static int num_instances;               // Declaration.

  public:
    static std::string static_str;          // Declaration.

    // ...
};

int         Example::num_instances;         // Definition.  Zero-initialised.
std::string Example::static_str = "Hello."; // Definition.

Devido a isso, variáveis ​​estáticas podem ser tipos incompletos (além de void), desde que sejam definidas posteriormente como um tipo completo.

struct ForwardDeclared;

class ExIncomplete {
    static ForwardDeclared fd;
    static ExIncomplete    i_contain_myself;
    static int             an_array[];
};

struct ForwardDeclared {};

ForwardDeclared ExIncomplete::fd;
ExIncomplete    ExIncomplete::i_contain_myself;
int             ExIncomplete::an_array[5];

As funções-membro estáticas podem ser definidas dentro ou fora da definição da classe, como acontece com as funções-membro normais. Assim como as variáveis ​​de membro estático, a palavra-chave static é omitida ao definir funções de membro estático fora da definição de classe.

// For Example above, either...
class Example {
    // ...

  public:
    static int static_func() { return num_instances; }

    // ...

    void set_str(const std::string& str) { static_str = str; }
};

// Or...

class Example { /* ... */ };

int  Example::static_func() { return num_instances; }
void Example::set_str(const std::string& str) { static_str = str; }

Se uma variável de membro estático for declarada const mas não volátil, e for do tipo integral ou enumeração, ela pode ser inicializada na declaração, dentro da definição da classe.

enum E { VAL = 5 };

struct ExConst {
    const static int ci = 5;              // Good.
    static const E ce = VAL;              // Good.
    const static double cd = 5;           // Error.
    static const volatile int cvi = 5;    // Error.

    const static double good_cd;
    static const volatile int good_cvi;
};

const double ExConst::good_cd = 5;        // Good.
const volatile int ExConst::good_cvi = 5; // Good.

A partir do C++11, variáveis ​​de membro estático de tipos LiteralType (tipos que podem ser construídos em tempo de compilação, de acordo com as regras constexpr) também podem ser declarados como constexpr; em caso afirmativo, eles devem ser inicializados dentro da definição de classe.

struct ExConstexpr {
    constexpr static int ci = 5;                      // Good.
    static constexpr double cd = 5;                   // Good.
    constexpr static int carr[] = { 1, 1, 2 };        // Good.
    static constexpr ConstexprConstructibleClass c{}; // Good.
    constexpr static int bad_ci;                      // Error.
};

constexpr int ExConstexpr::bad_ci = 5;                // Still an error.

Se uma variável de membro estático const ou constexpr for odr-used (informalmente, se tiver seu endereço tomado ou atribuído a uma referência), ela ainda deve ter uma definição separada, fora da definição da classe. Esta definição não pode conter um inicializador.

struct ExODR {
    static const int odr_used = 5;
};

// const int ExODR::odr_used;

const int* odr_user = & ExODR::odr_used; // Error; uncomment above line to resolve.

Como membros estáticos não estão vinculados a uma determinada instância, eles podem ser acessados ​​usando o operador de escopo, ::.

std::string str = Example::static_str;

Eles também podem ser acessados ​​como se fossem membros normais e não estáticos. Isso é de importância histórica, mas é usado com menos frequência do que o operador de escopo para evitar confusão sobre se um membro é estático ou não estático.

Example ex;
std::string rts = ex.static_str;

Os membros de classe podem acessar membros estáticos sem qualificar seu escopo, como acontece com membros de classe não estáticos.

class ExTwo {
    static int num_instances;
    int my_num;

  public:
    ExTwo() : my_num(num_instances++) {}

    static int get_total_instances() { return num_instances; }
    int get_instance_number() const { return my_num; }
};

int ExTwo::num_instances;

Eles não podem ser ‘mutáveis’, nem precisam ser; como eles não estão vinculados a nenhuma instância, se uma instância é ou não const não afeta membros estáticos.

struct ExDontNeedMutable {
    int immuta;
    mutable int muta;

    static int i;

    ExDontNeedMutable() : immuta(-5), muta(-5) {}
};
int ExDontNeedMutable::i;

// ...

const ExDontNeedMutable dnm;
dnm.immuta = 5; // Error: Can't modify read-only object.
dnm.muta = 5;   // Good.  Mutable fields of const objects can be written.
dnm.i = 5;      // Good.  Static members can be written regardless of an instance's const-ness.

Membros estáticos respeitam modificadores de acesso, assim como membros não estáticos.

class ExAccess {
    static int prv_int;

  protected:
    static int pro_int;

  public:
    static int pub_int;
};

int ExAccess::prv_int;
int ExAccess::pro_int;
int ExAccess::pub_int;

// ...

int x1 = ExAccess::prv_int; // Error: int ExAccess::prv_int is private.
int x2 = ExAccess::pro_int; // Error: int ExAccess::pro_int is protected.
int x3 = ExAccess::pub_int; // Good.

Como elas não estão vinculadas a uma determinada instância, as funções-membro estáticas não possuem ponteiro this; devido a isso, eles não podem acessar variáveis ​​de membro não estáticas, a menos que passem uma instância.

class ExInstanceRequired {
    int i;

  public:
    ExInstanceRequired() : i(0) {}

    static void bad_mutate() { ++i *= 5; }                         // Error.
    static void good_mutate(ExInstanceRequired& e) { ++e.i *= 5; } // Good.
};

Devido a não ter um ponteiro this, seus endereços não podem ser armazenados em funções de ponteiros para membros, e são armazenados em ponteiros para funções normais.

struct ExPointer {
           void nsfunc() {}
    static void  sfunc() {}
};

typedef void (ExPointer::* mem_f_ptr)();
typedef void (*f_ptr)();

mem_f_ptr p_sf = &ExPointer::sfunc; // Error.
    f_ptr p_sf = &ExPointer::sfunc; // Good.

Por não terem um ponteiro this, eles também não podem ser const ou volatile, nem podem ter qualificadores ref. Eles também não podem ser virtuais.

struct ExCVQualifiersAndVirtual {
    static void   func()                {} // Good.
    static void  cfunc() const          {} // Error.
    static void  vfunc() volatile       {} // Error.
    static void cvfunc() const volatile {} // Error.
    static void  rfunc() &              {} // Error.
    static void rvfunc() &&             {} // Error.

    virtual static void vsfunc()        {} // Error.
    static virtual void svfunc()        {} // Error.
};

Como não estão vinculadas a uma determinada instância, as variáveis ​​de membro estáticas são efetivamente tratadas como variáveis ​​globais especiais; eles são criados quando o programa é iniciado e destruídos quando ele é encerrado, independentemente da existência de alguma instância da classe. Existe apenas uma única cópia de cada variável de membro estático (a menos que a variável seja declarada thread_local (C++11 ou posterior), nesse caso há uma cópia por thread).

As variáveis ​​de membro estático têm a mesma ligação que a classe, quer a classe tenha ligação externa ou interna. Classes locais e classes sem nome não podem ter membros estáticos.

Funções membro não estáticas #

Uma classe pode ter funções de membro não estáticas, que operam em instâncias individuais da classe.

class CL {
  public:
    void member_function() {}
};

Essas funções são chamadas em uma instância da classe, assim:

CL instance;
instance.member_function();

Eles podem ser definidos dentro ou fora da definição da classe; se definidos fora, eles são especificados como estando no escopo da classe.

struct ST {
    void  defined_inside() {}
    void defined_outside();
};
void ST::defined_outside() {}

Eles podem ser qualificados por CV e/ou qualificados por referência, afetando como eles veem a instância para a qual são chamados; a função verá a instância como tendo os qualificadores cv especificados, se houver. Qual versão é chamada será baseada nos qualificadores cv da instância. Se não houver uma versão com os mesmos cv-qualifiers que a instância, uma versão mais cv-qualified será chamada, se disponível.

struct CVQualifiers {
    void func()                   {} // 1: Instance is non-cv-qualified.
    void func() const             {} // 2: Instance is const.

    void cv_only() const volatile {}
};

CVQualifiers       non_cv_instance;
const CVQualifiers      c_instance;

non_cv_instance.func(); // Calls #1.
c_instance.func();      // Calls #2.

non_cv_instance.cv_only(); // Calls const volatile version.
c_instance.cv_only();      // Calls const volatile version.

Os qualificadores ref da função membro indicam se a função deve ou não ser chamada em instâncias rvalue e usam a mesma sintaxe dos qualificadores cv da função.

struct RefQualifiers {
    void func() &  {} // 1: Called on normal instances.
    void func() && {} // 2: Called on rvalue (temporary) instances.
};

RefQualifiers rf;
rf.func();              // Calls #1.
RefQualifiers{}.func(); // Calls #2.

Os qualificadores de CV e os qualificadores de referência também podem ser combinados, se necessário.

struct BothCVAndRef {
    void func() const& {} // Called on normal instances.  Sees instance as const.
    void func() &&     {} // Called on temporary instances.
};

Eles também podem ser virtuais; isso é fundamental para o polimorfismo e permite que uma(s) classe(s) filha(s) forneça(m) a mesma interface que a classe pai, enquanto fornece sua própria funcionalidade.

struct Base {
    virtual void func() {}
};
struct Derived {
    virtual void func() {}
};

Base* bp = new Base;
Base* dp = new Derived;
bp.func(); // Calls Base::func().
dp.func(); // Calls Derived::func().

Estrutura/classe sem nome #

Sem nome struct é permitido (o tipo não tem nome)

void foo()
{
    struct /* No name */ {
        float x;
        float y;
    } point;
    
    point.x = 42;
}

ou

struct Circle
{
    struct /* No name */ {
        float x;
        float y;
    } center; // but a member name
    float radius;
};

e depois

Circle circle;
circle.center.x = 42.f;

mas NÃO anônimo struct (tipo sem nome e objeto sem nome)

struct InvalidCircle
{
    struct /* No name */ {
        float centerX;
        float centerY;
    }; // No member either.
    float radius;
};

Nota: Alguns compiladores permitem anonymous struct como extensão.

  • lamdba pode ser visto como um sem nome struct especial.

  • decltype permite recuperar o tipo de unnamed struct:

    decltype(circle.point) otherPoint;

  • instância struct sem nome pode ser parâmetro do método de modelo:

    void print_square_coordinates()
{
    const struct {float x; float y;} points[] = {
        {-1, -1}, {-1, 1}, {1, -1}, {1, 1}
    };

    // for range relies on `template <class T, std::size_t N> std::begin(T (&)[N])`
    for (const auto& point : points) { 
        std::cout << "{" << point.x << ", " << point.y << "}\n";
    }

    decltype(points[0]) topRightCorner{1, 1};
    auto it = std::find(points, points + 4, topRightCorner);
    std::cout << "top right corner is the "
              << 1 + std::distance(points, it) << "th\n";
}
