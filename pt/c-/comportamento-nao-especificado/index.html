Se o comportamento de uma construção não for especificado, então o padrão impõe algumas restrições ao comportamento, mas deixa alguma liberdade para a implementação, que não é necessária para documentar o que acontece em uma determinada situação. Ele contrasta com comportamento definido pela implementação, em que a implementação é necessária para documentar o que acontece, e comportamento indefinido, no qual tudo pode acontecer.

Valor de uma enumeração fora do intervalo #

Se uma enumeração com escopo for convertida em um tipo integral que seja muito pequeno para conter seu valor, o valor resultante não será especificado. Exemplo:

enum class E {
    X = 1,
    Y = 1000,
};
// assume 1000 does not fit into a char
char c1 = static_cast<char>(E::X); // c1 is 1
char c2 = static_cast<char>(E::Y); // c2 has an unspecified value

Além disso, se um inteiro for convertido em um enum e o valor do inteiro estiver fora do intervalo dos valores do enum, o valor resultante não será especificado. Exemplo:

enum Color {
    RED = 1,
    GREEN = 2,
    BLUE = 3,
};
Color c = static_cast<Color>(4);

No entanto, no próximo exemplo, o comportamento não não é especificado, pois o valor de origem está dentro do intervalo da enumeração, embora seja desigual para todos os enumeradores:

enum Scale {
    ONE = 1,
    TWO = 2,
    FOUR = 4,
};
Scale s = static_cast<Scale>(3);

Aqui s terá o valor 3 e será diferente de ONE, TWO e FOUR.

Ordem de avaliação dos argumentos da função #

Se uma função tem vários argumentos, não é especificado em que ordem eles são avaliados. O código a seguir pode imprimir x = 1, y = 2 ou x = 2, y = 1, mas não é especificado qual.

int f(int x, int y) {
    printf("x = %d, y = %d\n", x, y);
}
int get_val() {
    static int x = 0;
    return ++x;
}
int main() {
    f(get_val(), get_val());
}

Em C++17, a ordem de avaliação dos argumentos da função permanece não especificada.

No entanto, cada argumento de função é completamente avaliado, e o objeto de chamada é avaliado com garantia antes de qualquer argumento de função.

struct from_int {
  from_int(int x) { std::cout << "from_int (" << x << ")\n"; }
};
int make_int(int x){ std::cout << "make_int (" << x << ")\n"; return x; }


void foo(from_int a, from_int b) {
}
void bar(from_int a, from_int b) {
}

auto which_func(bool b){
  std::cout << b?"foo":"bar" << "\n";
  return b?foo:bar;
}

int main(int argc, char const*const* argv) {
  which_func( true )( make_int(1), make_int(2) );
}

isso deve imprimir:

bar
make_int(1)
from_int(1)
make_int(2)
from_int(2)

ou

bar
make_int(2)
from_int(2)
make_int(1)
from_int(1)

ele não pode imprimir bar após qualquer um dos make ou from, e não pode imprimir:

bar
make_int(2)
make_int(1)
from_int(2)
from_int(1)

ou similar. Antes do C++17 imprimir bar após make_ints era legal, assim como fazer ambos make_ints antes de fazer qualquer from_ints.

Resultado de algumas conversões reinterpret_cast #

O resultado de um reinterpret_cast de um tipo de ponteiro de função para outro, ou de um tipo de referência de função para outro, não é especificado. Exemplo:

int f();
auto fp = reinterpret_cast<int(*)(int)>(&f); // fp has unspecified value

O resultado de um reinterpret_cast de um tipo de ponteiro de objeto para outro, ou de um tipo de referência de objeto para outro, não é especificado. Exemplo:

int x = 42;
char* p = reinterpret_cast<char*>(&x); // p has unspecified value

No entanto, com a maioria dos compiladores, isso era equivalente a static_cast<char*>(static_cast<void*>(&x)) então o ponteiro resultante p apontava para o primeiro byte de x. Isso se tornou o comportamento padrão no C++11. Consulte conversão de troca de tipo para obter mais detalhes.

Espaço ocupado por uma referência #

Uma referência não é um objeto e, diferentemente de um objeto, não é garantido que ocupe alguns bytes contíguos de memória. O padrão não especifica se uma referência requer algum armazenamento. Vários recursos da linguagem conspiram para impossibilitar o exame portátil de qualquer armazenamento que a referência possa ocupar:

  • Se sizeof for aplicado a uma referência, ele retornará o tamanho do tipo referenciado, não fornecendo informações sobre se a referência ocupa algum armazenamento.
  • Arrays de referências são ilegais, portanto não é possível examinar os endereços de dois elementos consecutivos de uma referência hipotética de arrays para determinar o tamanho de uma referência.
  • Se o endereço de uma referência for obtido, o resultado será o endereço do referente, portanto não podemos obter um ponteiro para a própria referência.
  • Se uma classe tem um membro de referência, tentar extrair o endereço desse membro usando offsetof produz um comportamento indefinido, já que tal classe não é uma classe de layout padrão.
  • Se uma classe tiver um membro de referência, a classe não será mais o layout padrão, portanto, as tentativas de acessar quaisquer dados usados ​​para armazenar a referência resultarão em um comportamento indefinido ou não especificado.

Na prática, em alguns casos, uma variável de referência pode ser implementada de forma semelhante a uma variável de ponteiro e, portanto, ocupar a mesma quantidade de armazenamento que um ponteiro, enquanto em outros casos uma referência pode não ocupar espaço, pois pode ser otimizada. Por exemplo, em:

void f() {
    int x;
    int& r = x;
    // do something with r
}

o compilador é livre para simplesmente tratar r como um alias para x e substituir todas as ocorrências de r no resto da função f por x, e não alocar nenhum armazenamento para armazenar r.

Estado movido da maioria das classes de biblioteca padrão #

Todos os contêineres de biblioteca padrão são deixados em um estado válido, mas não especificado após serem movidos. Por exemplo, no código a seguir, v2 conterá {1, 2, 3, 4} após a movimentação, mas v1 não é garantido que esteja vazio.

int main() {
    std::vector<int> v1{1, 2, 3, 4};
    std::vector<int> v2 = std::move(v1);
}

Algumas classes têm um estado movido de precisamente definido. O caso mais importante é o de std::unique_ptr<T>, que é garantido como nulo após ser movido.

Ordem de inicialização de globais em TU #

Enquanto dentro de uma Unidade de Tradução, a ordem de inicialização das variáveis ​​globais é especificada, a ordem de inicialização entre as Unidades de Tradução não é especificada.

Então programe com os seguintes arquivos

  • foo.cpp

    #include <iostream>

int dummyFoo = ((std::cout << "foo"), 0);

-bar.cpp

  #include <iostream>

  int dummyBar = ((std::cout << "bar"), 0);

  • main.cpp

    int main() {}

pode produzir como saída:

foobar

ou

barfoo

Isso pode levar ao Fiasco da Ordem de Inicialização Estática.

Conjuração estática de valor falso de void* #

Se um valor void* for convertido em um ponteiro para o tipo de objeto, T*, mas não estiver alinhado corretamente para T, o valor do ponteiro resultante não será especificado. Exemplo:

// Suppose that alignof(int) is 4
int x = 42;
void* p1 = &x;
// Do some pointer arithmetic...
void* p2 = static_cast<char*>(p1) + 2;
int* p3 = static_cast<int*>(p2);

O valor de p3 não é especificado porque p2 não pode apontar para um objeto do tipo int; seu valor não é um endereço devidamente alinhado.

Resultado de algumas comparações de ponteiros #

Se dois ponteiros forem comparados usando <, >, <= ou >=, o resultado não será especificado nos seguintes casos:

  • Os ponteiros apontam para diferentes arrays. (Um objeto não array é considerado um array de tamanho 1.)

    int x;
int y;
const bool b1 = &x < &y;            // unspecified
int a[10];
const bool b2 = &a[0] < &a[1];      // true
const bool b3 = &a[0] < &x;         // unspecified
const bool b4 = (a + 9) < (a + 10); // true
                                    // note: a+10 points past the end of the array

    int z;
