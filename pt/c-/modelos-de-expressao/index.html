Um exemplo básico ilustrando modelos de expressão #

Um modelo de expressão é uma técnica de otimização em tempo de compilação usada principalmente em computação científica. Seu objetivo principal é evitar temporários desnecessários e otimizar cálculos de loop usando uma única passagem (normalmente ao realizar operações em agregados numéricos). Os modelos de expressão foram inicialmente concebidos para contornar as ineficiências da sobrecarga de operadores ingênuos ao implementar tipos numéricos Array ou Matrix. Uma terminologia equivalente para modelos de expressão foi introduzida por Bjarne Stroustrup, que os chama de “operações fundidas” na versão mais recente de seu livro, “The C++ Programming Language”.

Antes de mergulhar nos modelos de expressão, você deve entender por que precisa deles. Para ilustrar isso, considere a classe Matrix muito simples dada abaixo:

template <typename T, std::size_t COL, std::size_t ROW>
class Matrix {
public:
    using value_type = T;

    Matrix() : values(COL * ROW) {}

    static size_t cols() { return COL; }
    static size_t rows() { return ROW; }

    const T& operator()(size_t x, size_t y) const { return values[y * COL + x]; }
    T& operator()(size_t x, size_t y) { return values[y * COL + x]; }

private:
    std::vector<T> values;
};

template <typename T, std::size_t COL, std::size_t ROW>
Matrix<T, COL, ROW>
operator+(const Matrix<T, COL, ROW>& lhs, const Matrix<T, COL, ROW>& rhs)
{
    Matrix<T, COL, ROW> result;

    for (size_t y = 0; y != lhs.rows(); ++y) {
        for (size_t x = 0; x != lhs.cols(); ++x) {
            result(x, y) = lhs(x, y) + rhs(x, y);
        }
    }
    return result;
}

Dada a definição de classe anterior, agora você pode escrever expressões Matrix como:

const std::size_t cols = 2000;
const std::size_t rows = 1000;

Matrix<double, cols, rows> a, b, c;

// initialize a, b & c
for (std::size_t y = 0; y != rows; ++y) {
    for (std::size_t x = 0; x != cols; ++x) {
        a(x, y) = 1.0;
        b(x, y) = 2.0;
        c(x, y) = 3.0;
    }
}  

Matrix<double, cols, rows> d = a + b + c;  // d(x, y) = 6

Como ilustrado acima, poder sobrecarregar operator+() fornece uma notação que imita a notação matemática natural para matrizes.

Infelizmente, a implementação anterior também é altamente ineficiente em comparação com uma versão equivalente “feita à mão”.

Para entender o porquê, você deve considerar o que acontece quando você escreve uma expressão como Matriz d = a + b + c. Na verdade, isso se expande para ((a + b) + c) ou operator+(operator+(a, b), c). Em outras palavras, o loop dentro de operator+() é executado duas vezes, enquanto poderia ter sido facilmente executado em uma única passagem. Isso também resulta na criação de 2 temporários, o que degrada ainda mais o desempenho. Em essência, ao adicionar a flexibilidade de usar uma notação próxima à sua contraparte matemática, você também tornou a classe Matrix altamente ineficiente.

Por exemplo, sem sobrecarregar o operador, você pode implementar uma soma Matrix muito mais eficiente usando uma única passagem:

template<typename T, std::size_t COL, std::size_t ROW>
Matrix<T, COL, ROW> add3(const Matrix<T, COL, ROW>& a,
                         const Matrix<T, COL, ROW>& b,
                         const Matrix<T, COL, ROW>& c)
{
    Matrix<T, COL, ROW> result;
    for (size_t y = 0; y != ROW; ++y) {
        for (size_t x = 0; x != COL; ++x) {
            result(x, y) = a(x, y) + b(x, y) + c(x, y);
        }
    }
    return result;
}

O exemplo anterior, no entanto, tem suas próprias desvantagens porque cria uma interface muito mais complicada para a classe Matrix (você teria que considerar métodos como Matrix::add2(), Matrix::AddMultiply() e assim por diante) .

Em vez disso, vamos dar um passo para trás e ver como podemos adaptar a sobrecarga do operador para funcionar de uma maneira mais eficiente

O problema decorre do fato de que a expressão Matriz d = a + b + c é avaliada muito “avidamente” antes que você tenha a oportunidade de construir toda a árvore de expressão. Em outras palavras, o que você realmente quer alcançar é avaliar a + b + c em uma passagem e apenas uma vez você realmente precisa atribuir a expressão resultante a d.

Esta é a ideia central por trás dos templates de expressão: ao invés de ter operator+() avaliar imediatamente o resultado da adição de duas instâncias de Matrix, ele retornará um “modelo de expressão” para avaliação futura assim que toda a árvore de expressão tiver sido construída.

Por exemplo, aqui está uma implementação possível para um modelo de expressão correspondente à soma de 2 tipos:

template <typename LHS, typename RHS>
class MatrixSum
{
public:
    using value_type = typename LHS::value_type;

    MatrixSum(const LHS& lhs, const RHS& rhs) : rhs(rhs), lhs(lhs) {}
    
    value_type operator() (int x, int y) const  {
        return lhs(x, y) + rhs(x, y);
    }
private:
    const LHS& lhs;
    const RHS& rhs;
};

E aqui está a versão atualizada de operator+()

template <typename LHS, typename RHS>
MatrixSum<LHS, RHS> operator+(const LHS& lhs, const LHS& rhs) {
    return MatrixSum<LHS, RHS>(lhs, rhs);
}

Como você pode ver, operator+() não retorna mais uma “avaliação ansiosa” do resultado da adição de 2 instâncias de Matrix (que seria outra instância de Matrix), mas sim um modelo de expressão representando a operação de adição. O ponto mais importante a ter em mente é que a expressão ainda não foi avaliada. Ele apenas contém referências aos seus operandos.

Na verdade, nada impede você de instanciar o modelo de expressão MatrixSum<> da seguinte forma:

MatrixSum<Matrix<double>, Matrix<double> > SumAB(a, b);

Você pode, no entanto, em um estágio posterior, quando realmente precisar do resultado da soma, avaliar a expressão d = a + b da seguinte forma:

for (std::size_t y = 0; y != a.rows(); ++y) {
    for (std::size_t x = 0; x != a.cols(); ++x) {
        d(x, y) = SumAB(x, y);
    }
}

Como você pode ver, outro benefício de usar um modelo de expressão é que você basicamente conseguiu avaliar a soma de a e b e atribuí-la a d em uma única passagem.

Além disso, nada impede você de combinar vários modelos de expressão. Por exemplo, a + b + c resultaria no seguinte modelo de expressão:

MatrixSum<MatrixSum<Matrix<double>, Matrix<double> >, Matrix<double> > SumABC(SumAB, c);

E aqui novamente você pode avaliar o resultado final usando uma única passagem:

for (std::size_t y = 0; y != a.rows(); ++y) {
    for (std::size_t x = 0; x != a.cols(); ++x) {
        d(x, y) = SumABC(x, y);
    }
}

Finalmente, a última peça do quebra-cabeça é realmente conectar seu modelo de expressão à classe Matrix. Isso é alcançado essencialmente fornecendo uma implementação para Matrix::operator=(), que usa o modelo de expressão como um argumento e o avalia em uma passagem, como você fez “manualmente” antes:

template <typename T, std::size_t COL, std::size_t ROW>
class Matrix {
public:
    using value_type = T;

    Matrix() : values(COL * ROW) {}

    static size_t cols() { return COL; }
    static size_t rows() { return ROW; }

    const T& operator()(size_t x, size_t y) const { return values[y * COL + x]; }
    T& operator()(size_t x, size_t y) { return values[y * COL + x]; }

    template <typename E>
    Matrix<T, COL, ROW>& operator=(const E& expression) {
        for (std::size_t y = 0; y != rows(); ++y) {
            for (std::size_t x = 0; x != cols(); ++x) {
                values[y * COL + x] = expression(x, y);
            }
        }
        return *this;
    }

private:
    std::vector<T> values;
};

Modelos básicos de expressão em expressões algébricas elementares #

Introdução e motivação

Modelos de expressão (indicados como ETs a seguir) são uma poderosa técnica de metaprogramação de template, usada para acelerar cálculos de expressões às vezes bastante caras. É amplamente utilizado em diferentes domínios, por exemplo, na implementação de bibliotecas de álgebra linear.

Para este exemplo, considere o contexto de cálculos algébricos lineares. Mais especificamente, cálculos envolvendo apenas operações elementares. Esse tipo de cálculo é a aplicação mais básica dos ETs e serve como uma boa introdução sobre como os ETs funcionam internamente.

Vejamos um exemplo motivador. Considere o cálculo da expressão:

Vector vec_1, vec_2, vec_3;

// Initializing vec_1, vec_2 and vec_3.

Vector result = vec_1 + vec_2*vec_3;

Aqui, por uma questão de simplicidade, vou assumir que a classe Vector e a operação + (vetor mais: operação elementar mais) e operação * (aqui significa produto interno vetorial: também operação elementar) estão ambas implementadas corretamente , como deveriam ser, matematicamente.

Em uma implementação convencional sem usar ETs (ou outras técnicas semelhantes), pelo menos cinco construções de instâncias Vector ocorrem para obter o resultado final:

  1. Três instâncias correspondentes a vec_1, vec_2 e vec_3.
  2. Uma instância _tmp temporária Vector, representando o resultado de _tmp = vec_2*vec_3;.
  3. Finalmente com o uso adequado da otimização do valor de retorno, a construção do resultado final em resultado = vec_1 + _tmp;.

A implementação usando ETs pode eliminar a criação temporária de Vector _tmp em 2, deixando apenas quatro construções de instâncias de Vector. Mais interessante, considere a seguinte expressão que é mais complexa:

Vector result = vec_1 + (vec_2*vec3 + vec_1)*(vec_2 + vec_3*vec_1);

Haverá também quatro construções de instâncias Vector no total: vec_1, vec_2, vec_3 e result. Em outras palavras, neste exemplo, onde apenas operações elementares estão envolvidas, é garantido que nenhum objeto temporário será criado a partir de cálculos intermediários.

Como funcionam os ET

Basicamente falando, ETs para quaisquer cálculos algébricos consistem em dois blocos de construção:

  1. Expressões algébricas puras (PAE): são proxies/abstrações de expressões algébricas. Um algébrico puro não faz cálculos reais, eles são meramente abstrações/modelagem do fluxo de trabalho computacional. Um PAE pode ser um modelo de entrada ou saída de qualquer expressão algébrica. As instâncias de PAEs geralmente são consideradas baratas para copiar.
  2. Avaliações preguiçosas: que são implementações de cálculos reais. No exemplo a seguir, veremos que para expressões envolvendo apenas operações elementares, avaliações lentas podem implementar cálculos reais dentro da operação de acesso indexado no resultado final, criando assim um esquema de avaliação sob demanda: um cálculo não é executado somente se o resultado final for acessado/solicitado.

Então, especificamente, como implementamos ETs neste exemplo? Vamos percorrê-lo agora.

Considere sempre o seguinte trecho de código:

Vector vec_1, vec_2, vec_3;

// Initializing vec_1, vec_2 and vec_3.

Vector result = vec_1 + vec_2*vec_3;

A expressão para calcular o resultado pode ser decomposta em duas subexpressões:

  1. Um vetor mais expressão (indicado como plus_expr)
  2. Uma expressão de produto interno vetorial (indicada como innerprod_expr).

O que os ETs fazem é o seguinte:

  • Em vez de calcular imediatamente cada subexpressão, ETs primeiro modelam toda a expressão usando uma estrutura gráfica. Cada nó no gráfico representa um PAE. A conexão de borda dos nós representa o fluxo de computação real. Assim, para a expressão acima, obtemos o seguinte gráfico:

            result = plus_expr( vec_1, innerprod_expr(vec_2, vec_3) )
           /   \
          /     \
         /       \
        /   innerprod_expr( vec_2, vec_3 )
       /         /  \
      /         /    \
     /         /      \
  vec_1     vec_2    vec_3

  • O cálculo final é implementado observando a hierarquia do gráfico: como aqui estamos lidando com apenas operações de elemento, o cálculo de cada valor indexado em result *pode ser feito independentemente *: a avaliação final de result pode ser adiada preguiçosamente para uma avaliação por elementos de cada elemento de result. Em outras palavras, como o cálculo de um elemento de result, elem_res, pode ser expresso usando elementos correspondentes em vec_1 (elem_1), vec_2 (elem_2) e vec_3 (elem_3 ) como:

     elem_res = elem_1 + elem_2*elem_3;

portanto, não há necessidade de criar um Vector temporário para armazenar o resultado do produto interno intermediário: toda a computação para um elemento pode ser feita completamente e codificada dentro da operação de acesso indexado.

Aqui estão os códigos de exemplo em ação.

Arquivo vec.hh : wrapper para std::vector, usado para mostrar o log quando uma construção é chamada. #

#ifndef EXPR_VEC
# define EXPR_VEC

# include <vector>
# include <cassert>
# include <utility>
# include <iostream>
# include <algorithm>
# include <functional>

///
/// This is a wrapper for std::vector. It's only purpose is to print out a log when a
/// vector constructions in called.
/// It wraps the indexed access operator [] and the size() method, which are 
/// important for later ETs implementation.
///

// std::vector wrapper.
template<typename ScalarType> class Vector
{
public:
  explicit Vector() { std::cout << "ctor called.\n"; };
  explicit Vector(int size): _vec(size) { std::cout << "ctor called.\n"; };
  explicit Vector(const std::vector<ScalarType> &vec): _vec(vec)
  { std::cout << "ctor called.\n"; };
  
  Vector(const Vector<ScalarType> & vec): _vec{vec()}
  { std::cout << "copy ctor called.\n"; };
  Vector(Vector<ScalarType> && vec): _vec(std::move(vec()))
  { std::cout << "move ctor called.\n"; };

  Vector<ScalarType> & operator=(const Vector<ScalarType> &) = default;
  Vector<ScalarType> & operator=(Vector<ScalarType> &&) = default;

  decltype(auto) operator[](int indx) { return _vec[indx]; }
  decltype(auto) operator[](int indx) const { return _vec[indx]; }

  decltype(auto) operator()() & { return (_vec); };        
  decltype(auto) operator()() const & { return (_vec); };  
  Vector<ScalarType> && operator()() && { return std::move(*this); }

  int size() const { return _vec.size(); }
  
private:
  std::vector<ScalarType> _vec;
};

///
/// These are conventional overloads of operator + (the vector plus operation)
/// and operator * (the vector inner product operation) without using the expression
/// templates. They are later used for bench-marking purpose.
///

// + (vector plus) operator.
template<typename ScalarType>
auto operator+(const Vector<ScalarType> &lhs, const Vector<ScalarType> &rhs)
{
  assert(lhs().size() == rhs().size() &&
         "error: ops plus -> lhs and rhs size mismatch.");
  
  std::vector<ScalarType> _vec;
  _vec.resize(lhs().size());
  std::transform(std::cbegin(lhs()), std::cend(lhs()),
                 std::cbegin(rhs()), std::begin(_vec),
                 std::plus<>());
  return Vector<ScalarType>(std::move(_vec));
}

// * (vector inner product) operator.
template<typename ScalarType>
auto operator*(const Vector<ScalarType> &lhs, const Vector<ScalarType> &rhs)
{
  assert(lhs().size() == rhs().size() &&
         "error: ops multiplies -> lhs and rhs size mismatch.");
  
  std::vector<ScalarType> _vec;
  _vec.resize(lhs().size());
  std::transform(std::cbegin(lhs()), std::cend(lhs()),
                 std::cbegin(rhs()), std::begin(_vec),
                 std::multiplies<>());
  return Vector<ScalarType>(std::move(_vec));
}

#endif //!EXPR_VEC

Arquivo expr.hh : implementação de modelos de expressão para operações elementares (vetor mais e produto interno de vetor) #

Vamos dividi-lo em seções.

  1. A Seção 1 implementa uma classe base para todas as expressões. Ele emprega o Padrão de Modelo Curiosamente Recorrente (CRTP).

  2. A seção 2 implementa o primeiro PAE: um terminal, que é apenas um wrapper (referência const) de uma estrutura de dados de entrada contendo valor de entrada real para cálculo.

  3. A Seção 3 implementa o segundo PAE: binary_operation, que é um modelo de classe usado posteriormente para vector_plus e vector_innerprod. É parametrizado pelo tipo de operação, PAE do lado esquerdo e PAE do lado direito. A computação real é codificada no operador de acesso indexado.

  4. A seção 4 define as operações vector_plus e vector_innerprod como operação por elementos. Também sobrecarrega o operador + e * para PAEs: de modo que essas duas operações também retornem PAE.

    #ifndef EXPR_EXPR

    define EXPR_EXPR #

    /// Fwd declaration. template class Vector;

    namespace expr {

    /// —————————————– /// /// Section 1. /// /// The first section is a base class template for all kinds of expression. It
    /// employs the Curiously Recurring Template Pattern, which enables its instantiation /// to any kind of expression structure inheriting from it. /// /// —————————————–

    /// Base class for all expressions. template class expr_base { public: const Expr& self() const { return static_cast<const Expr&>(*this); } Expr& self() { return static_cast<Expr&>(*this); }

    protected: explicit expr_base() {}; int size() const { return self().size_impl(); } auto operator[](int indx) const { return self().at_impl(indx); } auto operator()() const { return self()(); }; };

    /// —————————————– /// /// The following section 2 & 3 are abstractions of pure algebraic expressions (PAE). /// Any PAE can be converted to a real object instance using operator(): it is in /// this conversion process, where the real computations are done.

    /// /// Section 2. Terminal /// /// A terminal is an abstraction wrapping a const reference to the Vector data /// structure. It inherits from expr_base, therefore providing a unified interface /// wrapping a Vector into a PAE. /// /// It provides the size() method, indexed access through at_impl() and a conversion /// to referenced object through () operator. /// /// It might no be necessary for user defined data structures to have a terminal /// wrapper, since user defined structure can inherit expr_base, therefore eliminates /// the need to provide such terminal wrapper. /// /// —————————————–

    /// Generic wrapper for underlying data structure. template class terminal: expr_base<terminal> { public: using base_type = expr_base<terminal>; using base_type::size; using base_type::operator[]; friend base_type;

     explicit terminal(const DataType &val): _val(val) {}
 int size_impl() const { return _val.size(); };
 auto at_impl(int indx) const { return _val[indx]; };
 decltype(auto) operator()() const { return (_val); }

    private: const DataType &_val; };

    /// —————————————– /// /// Section 3. Binary operation expression. /// /// This is a PAE abstraction of any binary expression. Similarly it inherits from /// expr_base. /// /// It provides the size() method, indexed access through at_impl() and a conversion /// to referenced object through () operator. Each call to the at_impl() method is /// a element wise computation. /// /// —————————————–

    /// Generic wrapper for binary operations (that are element-wise). template<typename Ops, typename lExpr, typename rExpr> class binary_ops: public expr_base<binary_ops<Ops,lExpr,rExpr» { public: using base_type = expr_base<binary_ops<Ops,lExpr,rExpr»; using base_type::size; using base_type::operator[]; friend base_type;

     explicit binary_ops(const Ops &ops, const lExpr &lxpr, const rExpr &rxpr)
   : _ops(ops), _lxpr(lxpr), _rxpr(rxpr) {};
 int size_impl() const { return _lxpr.size(); };

 /// This does the element-wise computation for index indx.
 auto at_impl(int indx) const { return _ops(_lxpr[indx], _rxpr[indx]); };

 /// Conversion from arbitrary expr to concrete data type. It evaluates
 /// element-wise computations for all indices.
 template<typename DataType> operator DataType()
 {
   DataType _vec(size());
   for(int _ind = 0; _ind < _vec.size(); ++_ind)
     _vec[_ind] = (*this)[_ind];
   return _vec;
 }

    private: /// Ops and expr are assumed cheap to copy. Ops _ops; lExpr _lxpr; rExpr _rxpr; };

    /// —————————————– /// Section 4. /// /// The following two structs defines algebraic operations on PAEs: here only vector /// plus and vector inner product are implemented. /// /// First, some element-wise operations are defined : in other words, vec_plus and /// vec_prod acts on elements in Vectors, but not whole Vectors. /// /// Then, operator + & * are overloaded on PAEs, such that: + & * operations on PAEs
    /// also return PAEs. /// /// —————————————–

    /// Element-wise plus operation. struct vec_plus_t { constexpr explicit vec_plus_t() = default; template<typename LType, typename RType> auto operator()(const LType &lhs, const RType &rhs) const { return lhs+rhs; } };

    /// Element-wise inner product operation. struct vec_prod_t { constexpr explicit vec_prod_t() = default; template<typename LType, typename RType> auto operator()(const LType &lhs, const RType &rhs) const { return lhs*rhs; } };

    /// Constant plus and inner product operator objects. constexpr vec_plus_t vec_plus{}; constexpr vec_prod_t vec_prod{};

    /// Plus operator overload on expressions: return binary expression. template<typename lExpr, typename rExpr> auto operator+(const lExpr &lhs, const rExpr &rhs) { return binary_ops<vec_plus_t,lExpr,rExpr>(vec_plus,lhs,rhs); }

    /// Inner prod operator overload on expressions: return binary expression. template<typename lExpr, typename rExpr> auto operator*(const lExpr &lhs, const rExpr &rhs) { return binary_ops<vec_prod_t,lExpr,rExpr>(vec_prod,lhs,rhs); }

    } //!expr

    #endif //!EXPR_EXPR

Arquivo main.cc : arquivo src de teste #

# include <chrono>
# include <iomanip>
# include <iostream>
# include "vec.hh"
# include "expr.hh"
# include "boost/core/demangle.hpp"


int main()
{
  using dtype = float;
  constexpr int size = 5e7;
  
  std::vector<dtype> _vec1(size);
  std::vector<dtype> _vec2(size);
  std::vector<dtype> _vec3(size);

  // ... Initialize vectors' contents.

  Vector<dtype> vec1(std::move(_vec1));
  Vector<dtype> vec2(std::move(_vec2));
  Vector<dtype> vec3(std::move(_vec3));

  unsigned long start_ms_no_ets =
    std::chrono::duration_cast<std::chrono::milliseconds>
    (std::chrono::system_clock::now().time_since_epoch()).count();
  std::cout << "\nNo-ETs evaluation starts.\n";
  
  Vector<dtype> result_no_ets = vec1 + (vec2*vec3);
  
  unsigned long stop_ms_no_ets =
    std::chrono::duration_cast<std::chrono::milliseconds>
    (std::chrono::system_clock::now().time_since_epoch()).count();
  std::cout << std::setprecision(6) << std::fixed
            << "No-ETs. Time eclapses: " << (stop_ms_no_ets-start_ms_no_ets)/1000.0
            << " s.\n" << std::endl;
  
  unsigned long start_ms_ets =
    std::chrono::duration_cast<std::chrono::milliseconds>
    (std::chrono::system_clock::now().time_since_epoch()).count();
  std::cout << "Evaluation using ETs starts.\n";
  
  expr::terminal<Vector<dtype>> vec4(vec1);
  expr::terminal<Vector<dtype>> vec5(vec2);
  expr::terminal<Vector<dtype>> vec6(vec3);
  
  Vector<dtype> result_ets = (vec4 + vec5*vec6);
  
  unsigned long stop_ms_ets =
    std::chrono::duration_cast<std::chrono::milliseconds>
    (std::chrono::system_clock::now().time_since_epoch()).count();
  std::cout << std::setprecision(6) << std::fixed
            << "With ETs. Time eclapses: " << (stop_ms_ets-start_ms_ets)/1000.0
            << " s.\n" << std::endl;
  
  auto ets_ret_type = (vec4 + vec5*vec6);
  std::cout << "\nETs result's type:\n";
  std::cout << boost::core::demangle( typeid(decltype(ets_ret_type)).name() ) << '\n'; 

  return 0;
}

As observações são:

  • O uso de ETs atinge um aumento de desempenho bastante significativo neste caso (> 3x).
  • A criação de objeto Vector temporário é eliminada. Como no caso ETs, ctor é chamado apenas uma vez.
  • Boost::demangle foi usado para visualizar o tipo de retorno de ETs antes da conversão: ele claramente construiu exatamente o mesmo gráfico de expressão demonstrado acima.

Desvantagens e advertências

  • Uma desvantagem óbvia dos ETs é a curva de aprendizado, a complexidade da implementação e a dificuldade de manutenção do código. No exemplo acima, onde apenas as operações elementares são consideradas, a implementação já contém uma enorme quantidade de clichês, muito menos no mundo real, onde expressões algébricas mais complexas ocorrem em todos os cálculos e a independência elementar não é mais válida (por exemplo, multiplicação de matrizes ), a dificuldade será exponencial.

  • Outra advertência de usar ETs é que eles funcionam bem com a palavra-chave auto. Como mencionado acima, PAEs são essencialmente proxies: e proxies basicamente não funcionam bem com auto. Considere o seguinte exemplo:

      auto result = ...;                // Some expensive expression: 
                                    // auto returns the expr graph, 
                                    // NOT the computed value.
  for(auto i = 0; i < 100; ++i)
      ScalrType value = result* ... // Some other expensive computations using result.

Aqui em cada iteração do loop for, o resultado será reavaliado, pois o gráfico da expressão em vez do valor calculado é passado para o loop for.

Bibliotecas existentes que implementam ETs

  • boost::proto é uma biblioteca poderosa que permite que você defina suas próprias regras e gramáticas para suas próprias expressões e execute usando ETs.
  • Eigen é uma biblioteca para álgebra linear que implementa vários cálculos algébricos de forma eficiente usando ETs.
