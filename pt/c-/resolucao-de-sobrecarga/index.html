A resolução de sobrecarga acontece em várias situações diferentes

  • Chamadas para funções sobrecarregadas nomeadas. Os candidatos são todas as funções encontradas pela pesquisa de nome.
  • Chamadas para objetos de classe. Os candidatos geralmente são todos os operadores de chamada de função sobrecarregados da classe.
  • Utilização de um operador. Os candidatos são as funções de operador sobrecarregadas no escopo do namespace, as funções de operador sobrecarregadas no objeto de classe esquerdo (se houver) e os operadores internos.
  • Resolução de sobrecarga para encontrar a função ou construtor correto do operador de conversão para invocar para uma inicialização
  • Para inicialização direta não lista (Class c(value)), os candidatos são construtores de Class.
  • Para inicialização de cópia não listada (Classe c = valor) e para localizar a função de conversão definida pelo usuário para invocar em uma sequência de conversão definida pelo usuário. Os candidatos são os construtores de Class e se a fonte for um objeto de classe, seu operador de conversão funciona.
  • Para inicialização de uma não-classe de um objeto de classe (Nonclass c = classObject). Os candidatos são as funções do operador de conversão do objeto inicializador.
  • Para inicializar uma referência com um objeto de classe (R &r = classObject), quando a classe possui funções de operador de conversão que geram valores que podem ser vinculados diretamente a r. Os candidatos são tais funções de operador de conversão.
  • Para inicialização de lista de um objeto de classe não agregado (Classe c{1, 2, 3}), os candidatos são os construtores de lista de inicializador para uma primeira passagem pela resolução de sobrecarga. Se isso não encontrar um candidato viável, uma segunda passagem pela resolução de sobrecarga é feita, com os construtores de Class como candidatos.

Categorização do argumento para o custo do parâmetro #

A resolução de sobrecarga divide o custo de passar um argumento para um parâmetro em uma das quatro categorias diferentes, chamadas “sequências”. Cada sequência pode incluir zero, uma ou várias conversões

  • Sequência de conversão padrão

    void f(int a); f(42);

  • Sequência de conversão definida pelo usuário

    void f(std::string s); f("hello");

  • Sequência de conversão de elipse

    void f(...); f(42);

  • Listar sequência de inicialização

    void f(std::vector<int> v); f({1, 2, 3});

O princípio geral é que as sequências de conversão padrão são as mais baratas, seguidas pelas sequências de conversão definidas pelo usuário, seguidas pelas sequências de conversão de reticências.

Um caso especial é a sequência de inicialização da lista, que não constitui uma conversão (uma lista inicializadora não é uma expressão com um tipo). Seu custo é determinado definindo-o para ser equivalente a uma das outras três sequências de conversão, dependendo do tipo de parâmetro e da forma da lista de inicializadores.

Combinação exata #

Uma sobrecarga sem conversões necessárias para tipos de parâmetro ou apenas conversões necessárias entre tipos que ainda são considerados correspondências exatas é preferível a uma sobrecarga que requer outras conversões para chamar.

void f(int x);
void f(double x);
f(42); // calls f(int)

Quando um argumento é associado a uma referência do mesmo tipo, considera-se que a correspondência não requer uma conversão, mesmo que a referência seja mais qualificada para cv.

void f(int& x);
void f(double x);
int x = 42;
f(x); // argument type is int; exact match with int&

void g(const int& x);
void g(int x);
g(x); // ambiguous; both overloads give exact match

Para fins de resolução de sobrecarga, considera-se que o tipo “array de T” corresponde exatamente ao tipo “ponteiro para T”, e o tipo de função T corresponde exatamente ao tipo de ponteiro de função T*, embora ambos exijam conversões.

void f(int* p);
void f(void* p);

void g(int* p);
void g(int (&p)[100]);

int a[100];
f(a); // calls f(int*); exact match with array-to-pointer conversion
g(a); // ambiguous; both overloads give exact match

Pesquisa de nomes e verificação de acesso #

A resolução de sobrecarga ocorre após a pesquisa de nome. Isso significa que uma função de melhor correspondência não será selecionada pela resolução de sobrecarga se perder a pesquisa de nome:

void f(int x);
struct S {
    void f(double x);
    void g() { f(42); } // calls S::f because global f is not visible here,
                        // even though it would be a better match
};

A resolução de sobrecarga ocorre antes da verificação de acesso. Uma função inacessível pode ser selecionada pela resolução de sobrecarga se for uma correspondência melhor do que uma função acessível.

class C {
  public:
    static void f(double x);
  private:
    static void f(int x);
};
C::f(42); // Error! Calls private C::f(int) even though public C::f(double) is viable.

Da mesma forma, a resolução de sobrecarga acontece sem verificar se a chamada resultante está bem formada em relação a explicit:

struct X {
    explicit X(int );
    X(char );
};

void foo(X );
foo({4}); // X(int) is better much, but expression is 
          // ill-formed because selected constructor is explicit

Sobrecarga na referência de encaminhamento #

Você deve ter muito cuidado ao fornecer uma sobrecarga de referência de encaminhamento, pois ela pode corresponder muito bem:

struct A {
    A() = default;           // #1
    A(A const& ) = default;  // #2

    template <class T>
    A(T&& );                 // #3
};

A intenção aqui era que A fosse copiável, e que tivéssemos este outro construtor que poderia inicializar algum outro membro. No entanto:

A a;     // calls #1
A b(a);  // calls #3!

Existem duas combinações viáveis ​​para a chamada de construção:

A(A const& ); // #2
A(A& );       // #3, with T = A&

Ambos são correspondências exatas, mas #3 faz referência a um objeto menos qualificado para cv do que #2, então ele tem a melhor sequência de conversão padrão e é a melhor função viável.

A solução aqui é sempre restringir esses construtores (por exemplo, usando SFINAE):

template <class T,
    class = std::enable_if_t<!std::is_convertible<std::decay_t<T>*, A*>::value>
    >
A(T&& );

A característica de tipo aqui é excluir qualquer A ou classe derivada publicamente e sem ambiguidade de A da consideração, o que tornaria esse construtor mal formado no exemplo descrito anteriormente (e, portanto, removido do conjunto de sobrecarga). Como resultado, o construtor de cópia é invocado - que é o que queríamos.

Promoções e conversões aritméticas #

Converter um tipo inteiro para o tipo promovido correspondente é melhor do que convertê-lo para algum outro tipo inteiro.

void f(int x);
void f(short x);
signed char c = 42;
f(c); // calls f(int); promotion to int is better than conversion to short
short s = 42;
f(s); // calls f(short); exact match is better than promotion to int

Promover um float para double é melhor do que convertê-lo para algum outro tipo de ponto flutuante.

void f(double x);
void f(long double x);
f(3.14f); // calls f(double); promotion to double is better than conversion to long double

Conversões aritméticas que não sejam promoções não são melhores nem piores que as outras.

void f(float x);
void f(long double x);
f(3.14); // ambiguous

void g(long x);
void g(long double x);
g(42); // ambiguous
g(3.14); // ambiguous

Portanto, para garantir que não haverá ambiguidade ao chamar uma função f com argumentos de ponto flutuante ou integral de qualquer tipo padrão, um total de oito sobrecargas são necessárias, de modo que para cada tipo de argumento possível, um sobrecarga corresponde exatamente ou a sobrecarga exclusiva com o tipo de argumento promovido será selecionada.

void f(int x);
void f(unsigned int x);
void f(long x);
void f(unsigned long x);
void f(long long x);
void f(unsigned long long x);
void f(double x);
void f(long double x);

Sobrecarga na constância e volatilidade #

Passar um argumento de ponteiro para um parâmetro T*, se possível, é melhor do que passá-lo para um parâmetro const T*.

struct Base {};
struct Derived : Base {};
void f(Base* pb);
void f(const Base* pb);
void f(const Derived* pd);
void f(bool b);

Base b;
f(&b); // f(Base*) is better than f(const Base*)
Derived d;
f(&d); // f(const Derived*) is better than f(Base*) though;
       // constness is only a "tie-breaker" rule

Da mesma forma, passar um argumento para um parâmetro T&, se possível, é melhor do que passá-lo para um parâmetro const T&, mesmo que ambos tenham classificação de correspondência exata.

void f(int& r);
void f(const int& r);
int x;
f(x); // both overloads match exactly, but f(int&) is still better
const int y = 42;
f(y); // only f(const int&) is viable

Essa regra também se aplica a funções de membro qualificadas para const, onde é importante para permitir acesso mutável a objetos não const e acesso imutável a objetos const.

class IntVector {
  public:
    // ...
    int* data() { return m_data; }
    const int* data() const { return m_data; }
  private:
    // ...
    int* m_data;
};
IntVector v1;
int* data1 = v1.data();       // Vector::data() is better than Vector::data() const;
                              // data1 can be used to modify the vector's data
const IntVector v2;
const int* data2 = v2.data(); // only Vector::data() const is viable;
                              // data2 can't be used to modify the vector's data

Da mesma forma, uma sobrecarga volátil será menos preferida do que uma sobrecarga não volátil.

class AtomicInt {
  public:
    // ...
    int load();
    int load() volatile;
  private:
    // ...
};
AtomicInt a1;
a1.load(); // non-volatile overload preferred; no side effect
volatile AtomicInt a2;
a2.load(); // only volatile overload is viable; side effect
static_cast<volatile AtomicInt&>(a1).load(); // force volatile semantics for a1

Etapas de resolução de sobrecarga #

As etapas de resolução de sobrecarga são:

  1. Encontre funções candidatas por meio de pesquisa de nome. As chamadas não qualificadas realizarão a pesquisa regular não qualificada, bem como a pesquisa dependente de argumento (se aplicável).

  2. Filtre o conjunto de funções candidatas para um conjunto de funções viáveis. Uma função viável para a qual existe uma sequência de conversão implícita entre os argumentos com os quais a função é chamada e os parâmetros que a função recebe.

     void f(char);          // (1)
 void f(int ) = delete; // (2)
 void f();              // (3)
 void f(int& );         // (4)

 f(4); // 1,2 are viable (even though 2 is deleted!) 
       // 3 is not viable because the argument lists don't match
       // 4 is not viable because we cannot bind a temporary to 
       //     a non-const lvalue reference

  3. Escolha o melhor candidato viável. Uma função viável F1 é uma função melhor que outra função viável F2 se a sequência de conversão implícita para cada argumento em F1 não for pior que a sequência de conversão implícita correspondente em F2, e…:

    3.1. For some argument, the implicit conversion sequence for that argument in F1 is a better conversion sequence than for that argument in F2, or

     void f(int );  // (1)
 void f(char ); // (2)

 f(4);  // call (1), better conversion sequence

    3.2. In a user-defined conversion, the standard conversion sequence from the return of F1 to the destination type is a better conversion sequence than that of the return type of F2, or

     struct A 
 {
     operator int();
     operator double();
 } a;

 int i = a; // a.operator int() is better than a.operator double() and a conversion
 float f = a; // ambiguous

    3.3. In a direct reference binding, F1 has the same kind of reference by F2 is not, or

     struct A 
 {
     operator X&();  // #1
     operator X&&(); // #2
 };
 A a;
 X& lx = a;  // calls #1
 X&& rx = a; // calls #2

    3.4. F1 is not a function template specialization, but F2 is, or

     template <class T> void f(T ); // #1
 void f(int );                  // #2

 f(42); // calls #2, the non-template

    3.5. F1 and F2 are both function template specializations, but F1 is more specialized than F2.

     template <class T> void f(T );  // #1
 template <class T> void f(T* ); // #2

 int* p;
 f(p); // calls #2, more specialized

A ordenação aqui é significativa. A melhor verificação de sequência de conversão acontece antes da verificação de modelo versus não de modelo. Isso leva a um erro comum com sobrecarga na referência de encaminhamento:

struct A {
    A(A const& ); // #1
    
    template <class T>
    A(T&& );      // #2, not constrained
};

A a;
A b(a); // calls #2!
        // #1 is not a template but #2 resolves to
        // A(A& ), which is a less cv-qualified reference than #1
        // which makes it a better implicit conversion sequence

Se não houver um único candidato viável no final, a chamada é ambígua:

void f(double ) { }
void f(float ) { }

f(42); // error: ambiguous

Sobrecarga em uma hierarquia de classes #

Os exemplos a seguir usarão essa hierarquia de classes:

struct A { int m; };
struct B : A {};
struct C : B {};

A conversão do tipo de classe derivada para o tipo de classe base é preferível às conversões definidas pelo usuário. Isso se aplica ao passar por valor ou por referência, bem como ao converter ponteiro para derivado em ponteiro para base.

struct Unrelated {
    Unrelated(B b);
};
void f(A a);
void f(Unrelated u);
B b;
f(b); // calls f(A)

Uma conversão de ponteiro de classe derivada para classe base também é melhor do que a conversão para void*.

void f(A* p);
void f(void* p);
B b;
f(&b); // calls f(A*)

Se houver várias sobrecargas na mesma cadeia de herança, a sobrecarga de classe base mais derivada é preferida. Isso se baseia em um princípio semelhante ao despacho virtual: a implementação “mais especializada” é escolhida. No entanto, a resolução de sobrecarga sempre ocorre em tempo de compilação e nunca será reduzida implicitamente.

void f(const A& a);
void f(const B& b);
C c;
f(c); // calls f(const B&)
B b;
A& r = b;
f(r); // calls f(const A&); the f(const B&) overload is not viable

Para ponteiros para membros, que são contravariantes em relação à classe, uma regra semelhante se aplica na direção oposta: a classe derivada menos derivada é preferida.

void f(int B::*p);
void f(int C::*p);
int A::*p = &A::m;
f(p); // calls f(int B::*)
