Sintaxe #

  • // Chamando:

  • variável.função_membro();

  • ponteiro_variável->função_membro();

  • // Definição:

  • ret_type class_name::member_function() cv-qualifiers {

    • body;
    • }

  • // Protótipo:

  • classe class_name {

    • virt-specifier ret_type member_function() cv-qualifiers virt-specifier-seq;
    • // virt-specifier: “virtual”, if applicable.
    • // cv-qualifiers: “const” and/or “volatile”, if applicable.
    • // virt-specifier-seq: “override” and/or “final”, if applicable.
    • }

Uma função membro não static é uma função membro class/struct/union, que é chamada em uma instância específica e opera nessa instância. Ao contrário das funções-membro static, ela não pode ser chamada sem especificar uma instância.

Para obter informações sobre classes, estruturas e uniões, consulte o tópico pai.

Funções de membro não estático #

Uma class ou struct pode ter funções de membro, bem como variáveis ​​de membro. Essas funções têm sintaxe semelhante a funções autônomas e podem ser definidas dentro ou fora da definição de classe; se definido fora da definição de classe, o nome da função é prefixado com o nome da classe e o operador de escopo (::).

class CL {
  public:
    void  definedInside() {}
    void definedOutside();
};
void CL::definedOutside() {}

Essas funções são chamadas em uma instância (ou referência a uma instância) da classe com o operador ponto (.), ou um ponteiro para uma instância com o operador seta (->), e cada chamada é vinculada a a instância em que a função foi chamada; quando uma função membro é chamada em uma instância, ela tem acesso a todos os campos dessa instância (através do this ponteiro), mas só pode acessar os campos de outras instâncias se essas instâncias forem fornecidas como parâmetros.

struct ST {
    ST(const std::string& ss = "Wolf", int ii = 359) : s(ss), i(ii) { }

    int get_i() const { return i; }
    bool compare_i(const ST& other) const { return (i == other.i); }

  private:
    std::string s;
    int i;
};
ST st1;
ST st2("Species", 8472);

int  i = st1.get_i(); // Can access st1.i, but not st2.i.
bool b = st1.compare_i(st2); // Can access st1 & st2.

Essas funções têm permissão para acessar variáveis ​​de membro e/ou outras funções de membro, independentemente dos modificadores de acesso da variável ou da função. Eles também podem ser escritos fora de ordem, acessando variáveis ​​de membro e/ou chamando funções de membro declaradas antes delas, pois toda a definição de classe deve ser analisada antes que o compilador possa começar a compilar uma classe.

class Access {
  public:
    Access(int i_ = 8088, int j_ = 8086, int k_ = 6502) : i(i_), j(j_), k(k_) {}

    int i;
    int get_k() const { return k; }
    bool private_no_more() const { return i_be_private(); }
  protected:
    int j;
    int get_i() const { return i; }
  private:
    int k;
    int get_j() const { return j; }
    bool i_be_private() const { return ((i > j) && (k < j)); }
};

Encapsulamento #

Um uso comum de funções de membro é para encapsulamento, usando um accessor (comumente conhecido como getter) e um mutator (comumente conhecido como setter) em vez de acessar campos diretamente.

class Encapsulator {
    int encapsulated;

  public:
    int  get_encapsulated() const { return encapsulated; }
    void set_encapsulated(int e)  { encapsulated = e; }

    void some_func() {
        do_something_with(encapsulated);
    }
};

Dentro da classe, encapsulated pode ser acessado livremente por qualquer função membro não estática; fora da classe, o acesso a ela é regulado por funções-membro, usando get_encapsulated() para lê-la e set_encapsulated() para modificá-la. Isso evita modificações não intencionais na variável, pois funções separadas são usadas para lê-la e gravá-la. [Há muitas discussões sobre se getters e setters fornecem ou quebram o encapsulamento, com bons argumentos para ambas as afirmações; tal debate acalorado está fora do escopo deste exemplo.]

Ocultação e importação de nomes #

Quando uma classe base fornece um conjunto de funções sobrecarregadas e uma classe derivada adiciona outra sobrecarga ao conjunto, isso oculta todas as sobrecargas fornecidas pela classe base.

struct HiddenBase {
    void f(int) { std::cout << "int" << std::endl; }
    void f(bool) { std::cout << "bool" << std::endl; }
    void f(std::string) { std::cout << "std::string" << std::endl; }
};

struct HidingDerived : HiddenBase {
    void f(float) { std::cout << "float" << std::endl; }
};

// ...

HiddenBase hb;
HidingDerived hd;
std::string s;

hb.f(1);    // Output:  int
hb.f(true); // Output:  bool
hb.f(s);    // Output:  std::string;

hd.f(1.f);  // Output:  float
hd.f(3);    // Output:  float
hd.f(true); // Output:  float
hd.f(s);    // Error: Can't convert from std::string to float.

Isto deve-se às regras de resolução de nomes: Durante a procura de nomes, uma vez que o nome correcto é encontrado, paramos de procurar, mesmo que claramente não tenhamos encontrado a versão correcta da entidade com esse nome (como com hd.f(s )); devido a isso, sobrecarregar a função na classe derivada impede que a pesquisa de nome descubra as sobrecargas na classe base. Para evitar isso, uma declaração de uso pode ser usada para “importar” nomes da classe base para a classe derivada, para que fiquem disponíveis durante a pesquisa de nomes.

struct HidingDerived : HiddenBase {
     // All members named HiddenBase::f shall be considered members of HidingDerived for lookup.
    using HiddenBase::f;

    void f(float) { std::cout << "float" << std::endl; }
};

// ...

HidingDerived hd;

hd.f(1.f);  // Output:  float
hd.f(3);    // Output:  int
hd.f(true); // Output:  bool
hd.f(s);    // Output:  std::string

Se uma classe derivada importar nomes com uma declaração de uso, mas também declarar funções com a mesma assinatura que as funções da classe base, as funções da classe base serão substituídas ou ocultadas silenciosamente.

struct NamesHidden {
    virtual void hide_me()      {}
    virtual void hide_me(float) {}
    void hide_me(int)           {}
    void hide_me(bool)          {}
};

struct NameHider : NamesHidden {
    using NamesHidden::hide_me;

    void hide_me()    {} // Overrides NamesHidden::hide_me().
    void hide_me(int) {} // Hides NamesHidden::hide_me(int).
};

Uma declaração de uso também pode ser usada para alterar modificadores de acesso, desde que a entidade importada seja pública ou protegida na classe base.

struct ProMem {
  protected:
    void func() {}
};

struct BecomesPub : ProMem {
    using ProMem::func;
};

// ...

ProMem pm;
BecomesPub bp;

pm.func(); // Error: protected.
bp.func(); // Good.

Da mesma forma, se quisermos chamar explicitamente uma função-membro de uma classe específica na hierarquia de herança, podemos qualificar o nome da função ao chamá-la, especificando essa classe pelo nome.

struct One {
    virtual void f() { std::cout << "One." << std::endl; }
};

struct Two : One {
    void f() override {
        One::f(); // this->One::f();
        std::cout << "Two." << std::endl;
    }
};

struct Three : Two {
    void f() override {
        Two::f(); // this->Two::f();
        std::cout << "Three." << std::endl;
    }
};

// ...

Three t;

t.f();      // Normal syntax.
t.Two::f(); // Calls version of f() defined in Two.
t.One::f(); // Calls version of f() defined in One.

Funções de membro virtual #

Funções membro também podem ser declaradas virtuais. Neste caso, se chamadas em um ponteiro ou referência a uma instância, elas não serão acessadas diretamente; em vez disso, eles procurarão a função na tabela de funções virtuais (uma lista de ponteiros para funções-membro para funções virtuais, mais comumente conhecidas como vtable ou vftable), e usarão isso para chamar a versão apropriada para o tipo dinâmico (real) da instância. Se a função for chamada diretamente, a partir de uma variável de uma classe, nenhuma pesquisa será executada.

struct Base {
    virtual void func() { std::cout << "In Base." << std::endl; }
};

struct Derived : Base {
    void func() override { std::cout << "In Derived." << std::endl; }
};

void slicer(Base x) { x.func(); }

// ...

Base b;
Derived d;

Base *pb = &b, *pd = &d; // Pointers.
Base &rb = b, &rd = d;   // References.

b.func();   // Output:  In Base.
d.func();   // Output:  In Derived.

pb->func(); // Output:  In Base.
pd->func(); // Output:  In Derived.

rb.func();  // Output:  In Base.
rd.func();  // Output:  In Derived.

slicer(b);  // Output:  In Base.
slicer(d);  // Output:  In Base.

Note que enquanto pd é Base*, e rd é uma Base&, chamando func() em qualquer uma das duas chamadas Derived::func() em vez de Base::func( )»; isso ocorre porque o vtableparaDerivedatualiza a entradaBase::func()para apontar paraDerived::func(). Por outro lado, observe como passar uma instância para slicer()sempre resulta na chamada deBase::func(), mesmo quando a instância passada é um Derived; isso ocorre por causa de algo conhecido como _data slicing_, onde passar uma instância Derivedpara um parâmetroBasepor valor torna a parte da instânciaDerivedque não é uma instânciaBase` inacessível.

Quando uma função de membro é definida como virtual, todas as funções de membro de classe derivada com a mesma assinatura a substituem, independentemente de a função de substituição ser especificada como ‘virtual’ ou não. Isso pode tornar as classes derivadas mais difíceis para os programadores analisarem, no entanto, já que não há indicação de quais funções são ‘virtuais’.

struct B {
    virtual void f() {}
};

struct D : B {
    void f() {} // Implicitly virtual, overrides B::f.
                //  You'd have to check B to know that, though.
};

Observe, no entanto, que uma função derivada apenas substitui uma função base se suas assinaturas corresponderem; mesmo que uma função derivada seja explicitamente declarada como ‘virtual’, ela criará uma nova função virtual se as assinaturas forem incompatíveis.

struct BadB {
    virtual void f() {}
};

struct BadD : BadB {
    virtual void f(int i) {} // Does NOT override BadB::f.
};

A partir do C++11, a intenção de substituir pode ser explicitada com a palavra-chave sensível ao contexto override. Isso informa ao compilador que o programador espera que ele substitua uma função de classe base, o que faz com que o compilador omita um erro se ele não substituir nada.

struct CPP11B {
    virtual void f() {}
};

struct CPP11D : CPP11B {
    void f() override {}
    void f(int i) override {} // Error: Doesn't actually override anything.
};

Isso também tem o benefício de informar aos programadores que a função é virtual e também declarada em pelo menos uma classe base, o que pode facilitar a análise de classes complexas.

Quando uma função é declarada como virtual e definida fora da definição da classe, o especificador virtual deve ser incluído na declaração da função e não repetido na definição.

Isso também vale para override.

struct VB {
    virtual void f(); // "virtual" goes here.
    void g();
};
/* virtual */ void VB::f() {} // Not here.
virtual void VB::g() {} // Error.

Se uma classe base sobrecarregar uma função virtual, somente sobrecargas especificadas explicitamente como virtual serão virtuais.

struct BOverload {
    virtual void func() {}
    void func(int) {}
};

struct DOverload : BOverload {
    void func() override {}
    void func(int) {}
};

// ...

BOverload* bo = new DOverload;
bo->func(); // Calls DOverload::func().
bo->func(1); // Calls BOverload::func(int).

Para obter mais informações, consulte o tópico relevante.

Correção Const #

Um dos principais usos para os qualificadores this cv é const correção. Esta é a prática de garantir que apenas os acessos que precisam modificar um objeto são capazes de modificar o objeto, e que qualquer função (membro ou não membro) que não precise modificar um objeto não tenha acesso de escrita a esse objeto. objeto (direta ou indiretamente). Isso evita modificações não intencionais, tornando o código menos propenso a erros. Ele também permite que qualquer função que não precise modificar o estado seja capaz de receber um objeto const ou não const, sem precisar reescrever ou sobrecarregar a função.

A correção de const, devido à sua natureza, começa de baixo para cima: Qualquer função membro de classe que não precise mudar de estado é declarada como const, para que possa ser chamada em const instâncias. Isso, por sua vez, permite que parâmetros passados ​​por referência sejam declarados const quando não precisam ser modificados, o que permite que funções recebam objetos const ou não const sem reclamar, e const -ness pode se propagar para fora dessa maneira. Devido a isso, os getters são frequentemente const, assim como quaisquer outras funções que não precisam modificar o estado lógico.

class ConstIncorrect {
    Field fld;

  public:
    ConstIncorrect(const Field& f) : fld(f) {}     // Modifies.

    const Field& get_field()       { return fld; } // Doesn't modify; should be const.
    void set_field(const Field& f) { fld = f; }    // Modifies.

    void do_something(int i) {                     // Modifies.
        fld.insert_value(i);
    }
    void do_nothing()        { }                   // Doesn't modify; should be const.
};

class ConstCorrect {
    Field fld;

  public:
    ConstCorrect(const Field& f) : fld(f) {}       // Not const: Modifies.

    const Field& get_field() const { return fld; } // const: Doesn't modify.
    void set_field(const Field& f) { fld = f; }    // Not const: Modifies.

    void do_something(int i) {                     // Not const: Modifies.
        fld.insert_value(i);
    }
    void do_nothing() const  { }                   // const: Doesn't modify.
};

// ...

const ConstIncorrect i_cant_do_anything(make_me_a_field());
// Now, let's read it...
Field f = i_cant_do_anything.get_field();
  // Error: Loses cv-qualifiers, get_field() isn't const.
i_cant_do_anything.do_nothing();
  // Error: Same as above.
// Oops.

const ConstCorrect but_i_can(make_me_a_field());
// Now, let's read it...
Field f = but_i_can.get_field(); // Good.
but_i_can.do_nothing();          // Good.

Conforme ilustrado pelos comentários em ConstIncorrect e ConstCorrect, funções de qualificação de cv apropriadas também servem como documentação. Se uma classe for const correta, qualquer função que não seja const pode seguramente ser assumida para mudar de estado, e qualquer função que seja const pode seguramente ser assumida como não mudando de estado.
