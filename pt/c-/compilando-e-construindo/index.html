Programas escritos em C++ precisam ser compilados antes de serem executados. Há uma grande variedade de compiladores disponíveis dependendo do seu sistema operacional.

A maioria dos sistemas operacionais é fornecida sem um compilador e precisa ser instalada posteriormente. Algumas escolhas comuns de compiladores são:

  • GCC, a coleção de compiladores GNU [tag:g++]
  • clang: um frontend de família de linguagem C para LLVM [tag:clang++]
  • MSVC, Microsoft Visual C++ (incluído no Visual Studio) [tag:visual-c++]
  • C++Builder, Embarcadero C++Builder (incluído no RAD Studio) [tag:c++builder]

Consulte o manual do compilador apropriado para saber como compilar um programa C++.

Outra opção para usar um compilador específico com seu próprio sistema de compilação específico, é possível deixar genéricos sistemas de compilação configurar o projeto para um compilador específico ou para o padrão instalado.

Compilando com GCC #

Assumindo um único arquivo de origem chamado main.cpp, o comando para compilar e vincular um executável não otimizado é o seguinte (compilar sem otimização é útil para desenvolvimento inicial e depuração, embora -Og seja oficialmente recomendado para versões mais recentes do GCC).

g++ -o app -Wall main.cpp -O0

Para produzir um executável otimizado para uso em produção, use uma das opções -O (veja: -O1, -O2, - O3, -Os, -Ofast):

g++ -o app -Wall -O2 main.cpp

Se a opção -O for omitida, -O0, que significa que não há otimizações, será usado como padrão (especificar -O sem um número resolve para -O1).

Como alternativa, use sinalizadores de otimização dos grupos O (ou mais otimizações experimentais) diretamente. O exemplo a seguir é compilado com otimização -O2, além de um sinalizador do nível de otimização -O3:

g++ -o app -Wall -O2 -ftree-partial-pre main.cpp

Para produzir um executável otimizado específico da plataforma (para uso em produção na máquina com a mesma arquitetura), use:

g++ -o app -Wall -O2 -march=native main.cpp

Qualquer uma das opções acima produzirá um arquivo binário que pode ser executado com .\app.exe no Windows e ./app no Linux, Mac OS, etc.

O sinalizador -o também pode ser ignorado. Nesse caso, o GCC criará o executável de saída padrão a.exe no Windows e a.out em sistemas do tipo Unix. Para compilar um arquivo sem vinculá-lo, use a opção -c:

g++ -o file.o -Wall -c file.cpp

Isso produz um arquivo objeto chamado file.o que pode ser vinculado posteriormente a outros arquivos para produzir um binário:

g++ -o app file.o otherfile.o

Mais informações sobre opções de otimização podem ser encontradas em gcc.gnu.org. De particular interesse são -Og (otimização com ênfase na experiência de depuração – recomendado para o ciclo padrão de edição-compilação-depuração) e -Ofast (todas as otimizações, incluindo aquelas que desrespeitam a estrita conformidade com os padrões).

O sinalizador -Wall ativa avisos para muitos erros comuns e deve sempre ser usado. Para melhorar a qualidade do código, muitas vezes é recomendado também usar -Wextra e outros sinalizadores de aviso que não são ativados automaticamente por -Wall e -Wextra.

Se o código espera um padrão C++ específico, especifique qual padrão usar incluindo o sinalizador -std=. Os valores suportados correspondem ao ano de finalização de cada versão do padrão ISO C++. A partir do GCC 6.1.0, os valores válidos para o sinalizador std= são c++98/c++03, c++11, c++14 e c++ 17/c++1z. Os valores separados por uma barra são equivalentes.

g++ -std=c++11 <file>

O GCC inclui algumas extensões específicas do compilador que são desabilitadas quando entram em conflito com um padrão especificado pelo sinalizador -std=. Para compilar com todas as extensões habilitadas, o valor gnu++XX pode ser usado, onde XX é qualquer um dos anos usados ​​pelos valores c++ listados acima.

O padrão padrão será usado se nenhum for especificado. Para versões do GCC anteriores a 6.1.0, o padrão é -std=gnu++03; no GCC 6.1.0 e superior, o padrão é -std=gnu++14.

Observe que devido a bugs no GCC, o sinalizador -pthread deve estar presente na compilação e vinculação para que o GCC suporte a funcionalidade de segmentação padrão C++ introduzida com C++11, como std::thread e std:: espere_para. Omiti-lo ao usar funções de encadeamento pode resultar em nenhum aviso, mas resultados inválidos em algumas plataformas.

Vinculando com bibliotecas: #

Use a opção -l para passar o nome da biblioteca:

g++ main.cpp -lpcre2-8
#pcre2-8 is the PCRE2 library for 8bit code units (UTF-8)

Se a biblioteca não estiver no caminho da biblioteca padrão, adicione o caminho com a opção -L:

g++ main.cpp -L/my/custom/path/ -lmylib

Várias bibliotecas podem ser vinculadas:

g++ main.cpp -lmylib1 -lmylib2 -lmylib3

Se uma biblioteca depende de outra, coloque a biblioteca dependente antes da biblioteca independente:

g++ main.cpp -lchild-lib -lbase-lib

Ou deixe o vinculador determinar a própria ordenação por meio de --start-group e --end-group (observação: isso tem um custo de desempenho significativo):

g++ main.cpp -Wl,--start-group -lbase-lib -lchild-lib -Wl,--end-group

Compiladores Online #

Vários sites fornecem acesso online a compiladores C++. O conjunto de recursos do compilador online varia significativamente de site para site, mas geralmente permite fazer o seguinte:

  • Cole seu código em um formulário da web no navegador.
  • Selecione algumas opções do compilador e compile o código.
  • Colete a saída do compilador e/ou do programa.

O comportamento do site do compilador online geralmente é bastante restritivo, pois permite que qualquer pessoa execute compiladores e código arbitrário no lado do servidor, enquanto a execução remota de código arbitrário normalmente é considerada uma vulnerabilidade.

Compiladores online podem ser úteis para os seguintes propósitos:

  • Executar um pequeno trecho de código de uma máquina que não possui compilador C++ (smartphones, tablets, etc.).
  • Certifique-se de que o código seja compilado com sucesso com diferentes compiladores e seja executado da mesma maneira, independentemente do compilador com o qual foi compilado.
  • Aprenda ou ensine noções básicas de C++.
  • Aprenda os recursos modernos do C++ (C++14 e C++17 em um futuro próximo) quando o compilador C++ atualizado não estiver disponível na máquina local.
  • Identifique um bug em seu compilador em comparação com um grande conjunto de outros compiladores. Verifique se um bug do compilador foi corrigido em versões futuras, que não estão disponíveis em sua máquina.
  • Resolva problemas de juiz online.

Para que compiladores online não devem ser usados:

  • Desenvolva aplicativos completos (mesmo pequenos) usando C++. Normalmente, os compiladores online não permitem vincular a bibliotecas de terceiros ou baixar artefatos de compilação.
  • Realizar cálculos intensivos. Os recursos de computação do lado do servidor são limitados, portanto, qualquer programa fornecido pelo usuário será eliminado após alguns segundos de execução. O tempo de execução permitido geralmente é suficiente para testar e aprender.
  • Ataque o próprio servidor do compilador ou qualquer host de terceiros na rede.

Compilando com Visual C++ (Linha de Comando) #

Para programadores vindos do GCC ou Clang para o Visual Studio, ou programadores mais familiarizados com a linha de comando em geral, você pode usar o compilador Visual C++ na linha de comando, bem como no IDE.

Se você deseja compilar seu código a partir da linha de comando no Visual Studio, primeiro você precisa configurar o ambiente de linha de comando. Isso pode ser feito abrindo o Visual Studio Command Prompt/Developer Command Prompt/x86 Native Tools Command Prompt/x64 Native Tools Command Prompt ou similar (conforme fornecido pela sua versão do Visual Studio), ou no prompt de comando, navegando até o subdiretório VC do diretório de instalação do compilador (normalmente \Program Files (x86)\Microsoft Visual Studio x\VC, onde x é o número da versão ( como 10.0 para 2010 ou 14.0 para 2015) e executando o arquivo de lote VCVARSALL com um parâmetro de linha de comando especificado aqui.

Observe que, diferentemente do GCC, o Visual Studio não fornece um front-end para o vinculador (link.exe) por meio do compilador (cl.exe), mas fornece o vinculador como um programa separado, que o compilador chama à medida que sai. cl.exe e link.exe podem ser usados ​​separadamente com diferentes arquivos e opções, ou cl pode ser instruído a passar arquivos e opções para link se ambas as tarefas forem feitas juntas. Quaisquer opções de link especificadas para cl serão traduzidas em opções para link, e quaisquer arquivos não processados ​​por cl serão passados ​​diretamente para link. Como este é principalmente um guia simples para compilar com a linha de comando do Visual Studio, os argumentos para link não serão descritos neste momento; se você precisar de uma lista, veja aqui.

Observe que os argumentos para cl diferenciam maiúsculas de minúsculas, enquanto os argumentos para link não.

[Esteja ciente de que alguns dos exemplos a seguir usam a variável “diretório atual” do shell do Windows, %cd%, ao especificar nomes de caminho absolutos. Para quem não está familiarizado com essa variável, ela se expande para o diretório de trabalho atual. Na linha de comando, será o diretório em que você estava quando executou cl, e é especificado no prompt de comando por padrão (se seu prompt de comando for C:\src>, por exemplo, então %cd % é C:\src\).]

Assumindo um único arquivo de origem chamado main.cpp na pasta atual, o comando para compilar e vincular um executável não otimizado (útil para desenvolvimento inicial e depuração) é (use um dos seguintes):

cl main.cpp
// Generates object file "main.obj".
// Performs linking with "main.obj".
// Generates executable "main.exe".

cl /Od main.cpp
// Same as above.
// "/Od" is the "Optimisation: disabled" option, and is the default when no /O is specified.

Supondo um arquivo de origem adicional “niam.cpp” no mesmo diretório, use o seguinte:

cl main.cpp niam.cpp
// Generates object files "main.obj" and "niam.obj".
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

Você também pode usar curingas, como seria de esperar:

cl main.cpp src\*.cpp
// Generates object file "main.obj", plus one object file for each ".cpp" file in folder
//  "%cd%\src".
// Performs linking with "main.obj", and every additional object file generated.
// All object files will be in the current folder.
// Generates executable "main.exe".

Para renomear ou realocar o executável, use um dos seguintes:

cl /o name main.cpp
// Generates executable named "name.exe".

cl /o folder\ main.cpp
// Generates executable named "main.exe", in folder "%cd%\folder".

cl /o folder\name main.cpp
// Generates executable named "name.exe", in folder "%cd%\folder".

cl /Fename main.cpp
// Same as "/o name".

cl /Fefolder\ main.cpp
// Same as "/o folder\".

cl /Fefolder\name main.cpp
// Same as "/o folder\name".

Ambos /o e /Fe passam seu parâmetro (vamos chamá-lo de o-param) para link como /OUT:o-param, anexando a extensão apropriada (geralmente .exe ou . dll) para “nomear” o-params conforme necessário. Embora tanto /o quanto /Fe sejam idênticos em funcionalidade, o último é o preferido para o Visual Studio. /o está marcado como obsoleto e parece ser fornecido principalmente para programadores mais familiarizados com GCC ou Clang.

Observe que, embora o espaço entre /o e a pasta e/ou nome especificado seja opcional, não pode haver um espaço entre /Fe e a pasta e/ou nome especificado.

Da mesma forma, para produzir um executável otimizado (para uso em produção), use:

cl /O1 main.cpp
// Optimise for executable size.  Produces small programs, at the possible expense of slower
//  execution.

cl /O2 main.cpp
// Optimise for execution speed.  Produces fast programs, at the possible expense of larger
//  file size.

cl /GL main.cpp other.cpp
// Generates special object files used for whole-program optimisation, which allows CL to
//  take every module (translation unit) into consideration during optimisation.
// Passes the option "/LTCG" (Link-Time Code Generation) to LINK, telling it to call CL during
//  the linking phase to perform additional optimisations.  If linking is not performed at this
//  time, the generated object files should be linked with "/LTCG".
// Can be used with other CL optimisation options.

Finalmente, para produzir um executável otimizado específico da plataforma (para uso em produção na máquina com a arquitetura especificada), escolha o prompt de comando apropriado ou parâmetro VCVARSALL para a plataforma de destino. link deve detectar a plataforma desejada dos arquivos objeto; caso contrário, use a opção /MACHINE para especificar explicitamente a plataforma de destino.

// If compiling for x64, and LINK doesn't automatically detect target platform:
cl main.cpp /link /machine:X64

Qualquer um dos itens acima produzirá um executável com o nome especificado por /o ou /Fe, ou se nenhum for fornecido, com um nome idêntico ao primeiro arquivo fonte ou objeto especificado para o compilador.

cl a.cpp b.cpp c.cpp
// Generates "a.exe".

cl d.obj a.cpp q.cpp
// Generates "d.exe".

cl y.lib n.cpp o.obj
// Generates "n.exe".

cl /o yo zp.obj pz.cpp
// Generates "yo.exe".

Para compilar um(s) arquivo(s) sem vincular, use:

cl /c main.cpp
// Generates object file "main.obj".

Isto diz ao cl para sair sem chamar link, e produz um arquivo objeto, que pode posteriormente ser vinculado a outros arquivos para produzir um binário.

cl main.obj niam.cpp
// Generates object file "niam.obj".
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

link main.obj niam.obj
// Performs linking with "main.obj" and "niam.obj".
// Generates executable "main.exe".

Existem outros parâmetros de linha de comando valiosos, que seria muito útil para os usuários saberem:

cl /EHsc main.cpp
// "/EHsc" specifies that only standard C++ ("synchronous") exceptions will be caught,
//  and `extern "C"` functions will not throw exceptions.
// This is recommended when writing portable, platform-independent code.

cl /clr main.cpp
// "/clr" specifies that the code should be compiled to use the common language runtime,
//  the .NET Framework's virtual machine.
// Enables the use of Microsoft's C++/CLI language in addition to standard ("native") C++,
//  and creates an executable that requires .NET to run.

cl /Za main.cpp
// "/Za" specifies that Microsoft extensions should be disabled, and code should be
//  compiled strictly according to ISO C++ specifications.
// This is recommended for guaranteeing portability.

cl /Zi main.cpp
// "/Zi" generates a program database (PDB) file for use when debugging a program, without
//  affecting optimisation specifications, and passes the option "/DEBUG" to LINK.

cl /LD dll.cpp
// "/LD" tells CL to configure LINK to generate a DLL instead of an executable.
// LINK will output a DLL, in addition to an LIB and EXP file for use when linking.
// To use the DLL in other programs, pass its associated LIB to CL or LINK when compiling those
//  programs.

cl main.cpp /link /LINKER_OPTION
// "/link" passes everything following it directly to LINK, without parsing it in any way.
// Replace "/LINKER_OPTION" with any desired LINK option(s).

Para qualquer pessoa mais familiarizada com sistemas *nix e/ou GCC/Clang, cl, link e outras ferramentas de linha de comando do Visual Studio podem aceitar parâmetros especificados com um hífen (como -c) em vez de uma barra ( como /c). Além disso, o Windows reconhece uma barra ou uma barra invertida como um separador de caminho válido, portanto, os caminhos no estilo *nix também podem ser usados. Isso facilita a conversão de linhas de comando simples do compilador de g++ ou clang++ para cl, ou vice-versa, com alterações mínimas.

g++ -o app src/main.cpp
cl  -o app src/main.cpp

Claro, ao portar linhas de comando que usam opções g++ ou clang++ mais complexas, você precisa procurar comandos equivalentes nas documentações aplicáveis ​​do compilador e/ou em sites de recursos, mas isso torna mais fácil começar as coisas com o mínimo tempo gasto aprendendo sobre novos compiladores.

Caso você precise de recursos de linguagem específicos para seu código, foi necessária uma versão específica do MSVC. A partir do Visual C++ 2015 Update 3 é possível escolher a versão do padrão para compilar através do sinalizador /std. Os valores possíveis são /std:c++14 e /std:c++latest (/std:c++17 seguirá em breve).

Observação: em versões mais antigas deste compilador, sinalizadores de recursos específicos estavam disponíveis, mas isso era usado principalmente para visualizações de novos recursos.

Compilando com Clang #

Como o front-end Clang foi projetado para ser compatível com GCC, a maioria dos programas que podem ser compilados via GCC compilarão quando você trocar g++ por clang++ nos scripts de compilação. Se nenhum -std=version for fornecido, gnu11 será usado.

Usuários do Windows que estão acostumados com MSVC podem trocar cl.exe por clang-cl.exe. Por padrão, o clang tenta ser compatível com a versão mais alta do MSVC que foi instalada.

No caso de compilar com o visual studio, clang-cl pode ser usado alterando o Platform toolset nas propriedades do projeto.

Em ambos os casos, clang é compatível apenas por meio de seu front-end, embora também tente gerar arquivos de objeto binários compatíveis. Os usuários do clang-cl devem observar que a compatibilidade com o MSVC ainda não está completa.

Para usar clang ou clang-cl, pode-se usar a instalação padrão em certas distribuições Linux ou aquelas empacotadas com IDEs (como XCode no Mac). Para outras versões deste compilador ou em plataformas que não o tenham instalado, este pode ser baixado da página oficial de download.

Se você estiver usando o CMake para construir seu código, geralmente pode alternar o compilador definindo as variáveis ​​de ambiente CC e CXX assim:

mkdir build
cd build
CC=clang CXX=clang++ cmake ..
cmake --build .

Veja também introdução ao Cmake.

O processo de compilação C++ #

Quando você desenvolve um programa C++, a próxima etapa é compilar o programa antes de executá-lo. A compilação é o processo que converte o programa escrito em linguagem legível por humanos como C, C++ etc em um código de máquina, compreendido diretamente pela Unidade Central de Processamento. Por exemplo, se você tiver um arquivo de código-fonte C++ chamado prog.cpp e executar o comando compile,

   g++ -Wall -ansi -o prog prog.cpp

Existem 4 etapas principais envolvidas na criação de um arquivo executável a partir do arquivo de origem.

  1. O C++ o pré-processador pega um arquivo de código fonte C++ e lida com os cabeçalhos(#include), macros(#define) e outras diretivas do pré-processador.

  2. O arquivo de código-fonte C++ expandido produzido pelo pré-processador C++ é compiled into the assembly language for the platform.

  3. O código montador gerado pelo compilador é montado no object code for the platform.

  4. O arquivo de código objeto produzido pelo montador é vinculado with the object code files for any library functions used to produce either a library or an executable file.

Pré-processando

O pré-processador lida com as diretivas do pré-processador, como #include e #define. É agnóstico da sintaxe de C++, razão pela qual deve ser usado com cuidado.

Ele funciona em um arquivo de origem C++ por vez, substituindo as diretivas #include pelo conteúdo dos respectivos arquivos (que geralmente são apenas declarações), substituindo macros (#define) e selecionando diferentes partes do texto dependendo de #if, diretivas #ifdef e #ifndef.

O pré-processador funciona em um fluxo de tokens de pré-processamento. A substituição de macro é definida como a substituição de tokens por outros tokens (o operador ## permite mesclar dois tokens quando faz sentido).

Depois de tudo isso, o pré-processador produz uma única saída que é um fluxo de tokens resultantes das transformações descritas acima. Ele também adiciona alguns marcadores especiais que informam ao compilador de onde veio cada linha para que ele possa usá-los para produzir mensagens de erro sensatas.

Alguns erros podem ser produzidos neste estágio com o uso inteligente das diretivas #if e #error.

Ao usar o sinalizador do compilador abaixo, podemos interromper o processo no estágio de pré-processamento.

g++ -E prog.cpp

Compilação

A etapa de compilação é executada em cada saída do pré-processador. O compilador analisa o código-fonte C++ puro (agora sem nenhuma diretiva de pré-processador) e o converte em código assembly. Em seguida, invoca o back-end subjacente (assembler na cadeia de ferramentas) que monta esse código em código de máquina produzindo arquivo binário real em algum formato (ELF, COFF, a.out, …). Este arquivo objeto contém o código compilado (em formato binário) dos símbolos definidos na entrada. Símbolos em arquivos de objeto são referidos pelo nome.

Arquivos de objetos podem se referir a símbolos que não estão definidos. Este é o caso quando você usa uma declaração e não fornece uma definição para ela. O compilador não se importa com isso e produzirá com prazer o arquivo objeto desde que o código-fonte esteja bem formado.

Os compiladores geralmente permitem que você interrompa a compilação neste ponto. Isso é muito útil porque com ele você pode compilar cada arquivo de código-fonte separadamente. A vantagem que isso oferece é que você não precisa recompilar tudo se alterar apenas um único arquivo.

Os arquivos de objetos produzidos podem ser colocados em arquivos especiais chamados bibliotecas estáticas, para facilitar a reutilização posterior.

É nesse estágio que os erros “regulares” do compilador, como erros de sintaxe ou erros de resolução de sobrecarga com falha, são relatados.

Para parar o processo após a etapa de compilação, podemos usar a opção -S:

g++ -Wall -ansi -S prog.cpp

Montagem

O montador cria código objeto. Em um sistema UNIX, você pode ver arquivos com um sufixo .o (.OBJ no MSDOS) para indicar arquivos de código de objeto. Nesta fase, o montador converte esses arquivos objeto do código assembly em instruções de nível de máquina e o arquivo criado é um código objeto relocável. Assim, a fase de compilação gera o programa objeto relocável e este programa pode ser usado em diferentes lugares sem ter que compilar novamente.

Para interromper o processo após a etapa de montagem, você pode usar a opção -c:

g++ -Wall -ansi -c prog.cpp

Vinculando

O vinculador é o que produz a saída final da compilação dos arquivos de objeto que o montador produziu. Essa saída pode ser uma biblioteca compartilhada (ou dinâmica) (e embora o nome seja semelhante, elas não têm muito em comum com as bibliotecas estáticas mencionadas anteriormente) ou um executável.

Ele vincula todos os arquivos de objetos substituindo as referências a símbolos indefinidos pelos endereços corretos. Cada um desses símbolos pode ser definido em outros arquivos de objetos ou em bibliotecas. Se eles estiverem definidos em bibliotecas diferentes da biblioteca padrão, você precisará informar o vinculador sobre eles.

Nesse estágio, os erros mais comuns são definições ausentes ou definições duplicadas. O primeiro significa que as definições não existem (ou seja, elas não foram escritas) ou que os arquivos de objeto ou bibliotecas onde residem não foram fornecidos ao vinculador. O último é óbvio: o mesmo símbolo foi definido em dois arquivos ou bibliotecas de objetos diferentes.
