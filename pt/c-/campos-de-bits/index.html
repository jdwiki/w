Campos de bits

  • Declaração e uso

Nesta página

  • Declaração e uso

Os campos de bits compactam as estruturas C e C++ para reduzir o tamanho. Isso parece indolor: especifique o número de bits para membros e o compilador fará o trabalho de misturar bits. A restrição é a incapacidade de obter o endereço de um membro do campo de bits, uma vez que é armazenado misturado. sizeof() também não é permitido.

O custo dos campos de bits é o acesso mais lento, pois a memória deve ser recuperada e as operações bit a bit aplicadas para extrair ou modificar os valores dos membros. Essas operações também aumentam o tamanho do executável.

Quão caras são as operações bit a bit? Suponha uma estrutura de campo simples sem bits:

struct foo {
    unsigned x;
    unsigned y;
}
static struct foo my_var;

Em algum código posterior:

my_var.y = 5;

Se sizeof (unsigned) == 4, então x é armazenado no início da estrutura e y é armazenado em 4 bytes. O código assembly gerado pode se parecer com:

loada register1,#myvar     ; get the address of the structure
storei register1[4],#0x05  ; put the value '5' at offset 4, e.g., set y=5

Isso é direto porque x não é co-misturado com y. Mas imagine redefinir a estrutura com campos de bits:

struct foo {
    unsigned x : 4; /* Range 0-0x0f, or 0 through 15 */
    unsigned y : 4;
}

Ambos x e y serão alocados em 4 bits, compartilhando um único byte. A estrutura, portanto, ocupa 1 byte, em vez de 8. Considere o assembly para definir y agora, supondo que ele termine no nibble superior:

loada  register1,#myvar        ; get the address of the structure
loadb  register2,register1[0]  ; get the byte from memory
andb   register2,#0x0f         ; zero out y in the byte, leaving x alone
orb    register2,#0x50         ; put the 5 into the 'y' portion of the byte
stb    register1[0],register2  ; put the modified byte back into memory

Essa pode ser uma boa compensação se tivermos milhares ou milhões dessas estruturas, e isso ajuda a manter a memória em cache ou evita a troca — ou pode aumentar o volume do executável para piorar esses problemas e tornar o processamento lento. Como com todas as coisas, use o bom senso.

Uso de driver de dispositivo: Evite campos de bits como uma estratégia de implementação inteligente para drivers de dispositivo. Os layouts de armazenamento de campo de bits não são necessariamente consistentes entre os compiladores, tornando essas implementações não portáteis. Os valores de leitura-modificação-gravação para definir podem não fazer o que os dispositivos esperam, causando comportamentos inesperados.

Declaração e uso #

struct FileAttributes
{
    unsigned int ReadOnly: 1;    
    unsigned int Hidden: 1;
};

Aqui, cada um desses dois campos ocupará 1 bit na memória. É especificado pela expressão : 1 após os nomes das variáveis. O tipo base do campo de bits pode ser qualquer tipo integral (int de 8 bits a inteiro de 64 bits). O uso do tipo unsigned é recomendado, caso contrário, surpresas podem surgir.

Se forem necessários mais bits, substitua “1” pelo número de bits necessários. Por exemplo:

struct Date
{
    unsigned int Year : 13; // 2^13 = 8192, enough for "year" representation for long time
    unsigned int Month: 4;  // 2^4 = 16, enough to represent 1-12 month values.
    unsigned int Day:   5;  // 32
};

A estrutura inteira está usando apenas 22 bits, e com configurações normais do compilador, sizeof essa estrutura seria de 4 bytes.

O uso é bem simples. Apenas declare a variável e use-a como estrutura comum.

Date d;

d.Year = 2016;
d.Month = 7;
d.Day =  22;

std::cout << "Year:" << d.Year << std::endl <<
        "Month:" << d.Month << std::endl <<
        "Day:" << d.Day << std::endl;
