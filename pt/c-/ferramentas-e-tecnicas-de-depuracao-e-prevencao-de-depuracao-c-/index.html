Muito tempo dos desenvolvedores de C++ é gasto depurando. Este tópico destina-se a auxiliar nessa tarefa e fornecer inspiração para técnicas. Não espere uma extensa lista de problemas e soluções corrigidos pelas ferramentas ou um manual sobre as ferramentas mencionadas.

Este tópico ainda não está completo, exemplos das seguintes técnicas/ferramentas seriam úteis:

  • Mencione mais ferramentas de análise estática
  • Ferramentas de instrumentação binária (como UBSan, TSan, MSan, ESan…)
  • Endurecimento (CFI…)
  • Fuzzing

Análise de falhas com GDB #

Análise estática #

A análise estática é a técnica na qual verifica o código em busca de padrões vinculados a bugs conhecidos. O uso dessa técnica consome menos tempo do que uma revisão de código, porém, suas verificações são limitadas apenas às programadas na ferramenta.

As verificações podem incluir o ponto e vírgula incorreto atrás da instrução if (if (var);) até algoritmos de gráfico avançados que determinam se uma variável não foi inicializada.

Avisos do compilador #

Ativar a análise estática é fácil, a versão mais simplista já está embutida no seu compilador:

  • clang++ -Wall -Weverything -Werror ...
  • g++ -Wall -Weverything -Werror ...
  • cl.exe /W4 /WX ...

Se você habilitar essas opções, você notará que cada compilador encontrará bugs que os outros não encontram e que você receberá erros em técnicas que podem ser válidas ou válidas em um contexto específico. while (staticAtomicBool); pode ser aceitável mesmo que while (localBool); não seja.

Então, diferentemente da revisão de código, você está lutando contra uma ferramenta que entende seu código, informa muitos bugs úteis e às vezes discorda de você. Neste último caso, talvez seja necessário suprimir o aviso localmente.

Como as opções acima habilitam todos os avisos, elas podem habilitar os avisos indesejados. (Por que seu código deve ser compatível com C++98?) Em caso afirmativo, você pode simplesmente desabilitar esse aviso específico:

  • clang++ -Wall -Weverything -Werror -Wno-errortoaccept ...
  • g++ -Wall -Weverything -Werror -Wno-errortoaccept ...
  • cl.exe /W4 /WX /wd<nº de aviso>...

Onde os avisos do compilador o ajudam durante o desenvolvimento, eles tornam a compilação um pouco mais lenta. É por isso que você nem sempre deseja habilitá-los por padrão. Ou você os executa por padrão ou habilita alguma integração contínua com as verificações mais caras (ou todas elas).

Ferramentas externas #

Se você decidir ter alguma integração contínua, o uso de outras ferramentas não é tão difícil. Uma ferramenta como clang-tidy tem uma [lista de verificações](http://clang.llvm.org/extra/clang-tidy/ checks/list.html) que cobre uma ampla gama de questões, alguns exemplos:

  • Erros reais
    • Prevention of slicing
    • Asserts with side effects
  • Verificações de legibilidade
    • Misleading indentation
    • Check identifier naming
  • Verificações de modernização
    • Use make_unique()
    • Use nullptr
  • Verificações de desempenho
    • Find unneeded copies
    • Find inefficient algorithm calls

A lista pode não ser tão grande, pois o Clang já tem muitos avisos do compilador, mas ele vai te deixar um passo mais perto de uma base de código de alta qualidade.

Outras ferramentas #

Existem outras ferramentas com finalidade semelhante, como:

  • [o analisador estático do visual studio](https://blogs.msdn.microsoft.com/hkamel/2013/10/24/visual-studio-2013-static-code-analysis-in-depth-what-when-and -how/) como ferramenta externa
  • clazy, um plugin do compilador Clang para verificar o código Qt

Conclusão #

Existem muitas ferramentas de análise estática para C++, tanto embutidas no compilador quanto ferramentas externas. Testá-los não leva muito tempo para configurações fáceis e eles encontrarão bugs que você pode perder na revisão de código.

Código limpo #

A depuração começa com a compreensão do código que você está tentando depurar.

Código incorreto:

int main() {
    int value;
    std::vector<int> vectorToSort;
    vectorToSort.push_back(42); vectorToSort.push_back(13);
    for (int i = 52; i; i = i - 1)
    {
    vectorToSort.push_back(i *2);
    }
    /// Optimized for sorting small vectors
    if (vectorToSort.size() == 1);
    else
        {
        if (vectorToSort.size() <= 2)
            std::sort(vectorToSort.begin(), std::end(vectorToSort));
        }
    for (value : vectorToSort) std::cout << value << ' ';
return 0; }

Código melhor:

std::vector<int> createSemiRandomData() {
    std::vector<int> data;
    data.push_back(42);
    data.push_back(13);
    for (int i = 52; i; --i)
        vectorToSort.push_back(i *2);
    return data;
}

/// Optimized for sorting small vectors
void sortVector(std::vector &v) {
    if (vectorToSort.size() == 1)
        return;
    if (vectorToSort.size() > 2)
        return;

    std::sort(vectorToSort.begin(), vectorToSort.end());
}

void printVector(const std::vector<int> &v) {
    for (auto i : v)
        std::cout << i << ' ';
}

int main() {
    auto vectorToSort = createSemiRandomData();
    sortVector(std::ref(vectorToSort));
    printVector(vectorToSort);

    return 0;
 }

Independentemente dos estilos de codificação que você prefere e usa, ter um estilo de codificação (e formatação) consistente ajudará você a entender o código.

Observando o código acima, pode-se identificar algumas melhorias para melhorar a legibilidade e a depuração:

O uso de funções separadas para ações separadas #

O uso de funções separadas permite pular algumas funções no depurador se você não estiver interessado nos detalhes. Nesse caso específico, talvez você não esteja interessado na criação ou impressão dos dados e queira apenas avançar na classificação.

Outra vantagem é que você precisa ler menos código (e memorizá-lo) enquanto percorre o código. Agora você só precisa ler 3 linhas de código em main() para entendê-lo, em vez de toda a função.

A terceira vantagem é que você simplesmente tem menos código para analisar, o que ajuda um olho treinado a detectar esse bug em segundos.

Usando formatação/construções consistentes #

O uso de formatação e construções consistentes removerá a desordem do código, facilitando o foco no código em vez do texto. Muitas discussões foram alimentadas sobre o estilo de formatação ‘correto’. Independentemente desse estilo, ter um único estilo consistente no código melhorará a familiaridade e facilitará o foco no código.

Como a formatação do código é uma tarefa demorada, é recomendável usar uma ferramenta dedicada para isso. A maioria dos IDEs tem pelo menos algum tipo de suporte para isso e pode fazer formatação mais consistente do que humanos.

Você pode notar que o estilo não está limitado a espaços e novas linhas, pois não misturamos mais o estilo livre e as funções de membro para obter o início/fim do contêiner. (v.begin() vs std::end(v)).

Aponte a atenção para as partes importantes do seu código. #

Independentemente do estilo que você decidir escolher, o código acima contém alguns marcadores que podem dar uma dica sobre o que pode ser importante:

  • Um comentário dizendo ‘otimizado’, isso indica algumas técnicas sofisticadas
  • Alguns retornos iniciais em sortVector() indicam que estamos fazendo algo especial
  • O std::ref() indica que algo está acontecendo com o sortVector()

Conclusão #

Ter um código limpo ajudará você a entender o código e reduzirá o tempo necessário para depurá-lo. No segundo exemplo, um revisor de código pode até identificar o bug à primeira vista, enquanto o bug pode estar oculto nos detalhes do primeiro. (PS: O bug está na comparação com 2.)

Safe-stack (corrupções de pilha) #

Corrupções de pilha são bugs irritantes de se observar. Como a pilha está corrompida, o depurador geralmente não pode fornecer um bom rastreamento de pilha de onde você está e como chegou lá.

É aqui que a pilha segura entra em jogo. Em vez de usar uma única pilha para seus threads, ela usará duas: uma pilha segura e uma pilha perigosa. A pilha segura funciona exatamente como antes, exceto que algumas partes são movidas para a pilha perigosa.

Quais partes da pilha são movidas? #

Cada parte que tem o potencial de corromper a pilha será movida para fora da pilha segura. Assim que uma variável na pilha é passada por referência ou alguém pega o endereço dessa variável, o compilador decidirá alocar isso na segunda pilha em vez da segura.

Como resultado, qualquer operação que você fizer com esses ponteiros, qualquer modificação que você fizer na memória (com base nesses ponteiros/referências) só pode afetar a memória na segunda pilha. Como nunca se obtém um ponteiro que esteja próximo da pilha segura, a pilha não pode corromper a pilha e o depurador ainda pode ler todas as funções na pilha para fornecer um bom rastreamento.

Para que realmente serve? #

A pilha segura não foi inventada para oferecer uma melhor experiência de depuração, no entanto, é um bom efeito colateral para bugs desagradáveis. Seu propósito original é como parte do Code-Pointer Integrity (CPI) Project, no qual eles tentam evitar a substituição dos endereços de retorno para evitar a injeção de código. Em outras palavras, eles tentam impedir a execução de um código de hackers.

Por esse motivo, o recurso foi ativado no chromium e foi relatado para ter um <1% de sobrecarga da CPU.

Como habilitá-lo? #

No momento, a opção só está disponível no compilador clang, onde se pode passar -fsanitize=safe-stack para o compilador. Uma proposta foi feita para implementar o mesmo recurso no GCC.

Conclusão #

As corrupções de pilha podem se tornar mais fáceis de depurar quando a pilha segura está habilitada. Devido a uma sobrecarga de baixo desempenho, você pode até ativar por padrão em sua configuração de compilação.
