O que é SFINAE #

SFINAE significa Ssubstituição Failure Is Not An Error. Código mal formado que resulta da substituição de tipos (ou valores) para instanciar um modelo de função ou um modelo de classe não é um erro de compilação, é tratado apenas como uma falha de dedução.

As falhas de dedução ao instanciar modelos de função ou especializações de modelo de classe removem esse candidato do conjunto de consideração - como se esse candidato com falha não existisse para começar.

template <class T>
auto begin(T& c) -> decltype(c.begin()) { return c.begin(); }

template <class T, size_t N>
T* begin(T (&arr)[N]) { return arr; }

int vals[10];
begin(vals); // OK. The first function template substitution fails because
             // vals.begin() is ill-formed. This is not an error! That function
             // is just removed from consideration as a viable overload candidate,
             // leaving us with the array overload.

Somente falhas de substituição no contexto imediato são consideradas falhas de dedução, todas as outras são consideradas erros graves.

template <class T>
void add_one(T& val) { val += 1; }

int i = 4;
add_one(i); // ok

std::string msg = "Hello";
add_one(msg); // error. msg += 1 is ill-formed for std::string, but this
              // failure is NOT in the immediate context of substituting T

void_t #

void_t é uma meta-função que mapeia qualquer (número de) tipos para o tipo void. O objetivo principal do void_t é facilitar a escrita de características de tipo.

std::void_t fará parte do C++17, mas até então, é extremamente simples de implementar:

template <class...> using void_t = void;

Alguns compiladores exigem uma implementação ligeiramente diferente:

template <class...>
struct make_void { using type = void; };

template <typename... T>
using void_t = typename make_void<T...>::type;

A principal aplicação de void_t é escrever características de tipo que verificam a validade de uma declaração. Por exemplo, vamos verificar se um tipo tem uma função de membro foo() que não recebe argumentos:

template <class T, class=void>
struct has_foo : std::false_type {};

template <class T>
struct has_foo<T, void_t<decltype(std::declval<T&>().foo())>> : std::true_type {};

Como é que isso funciona? Quando tento instanciar has_foo<T>::value, isso fará com que o compilador tente procurar a melhor especialização para has_foo<T, void>. Temos duas opções: a primária e esta secundária que envolve ter que instanciar aquela expressão subjacente:

  • Se T tiver uma função de membro foo(), então qualquer tipo que retorne será convertido em void, e a especialização é preferida à primária com base nas regras de ordenação parcial do modelo. Então has_foo<T>::value será true
  • Se T não tiver tal função de membro (ou requerer mais de um argumento), então a substituição falhará para a especialização e teremos apenas o template primário para o qual recorrer. Portanto, has_foo<T>::value é false.

Um caso mais simples:

template<class T, class=void>
struct can_reference : std::false_type {};

template<class T>
struct can_reference<T, std::void_t<T&>> : std::true_type {};

isso não usa std::declval ou decltype.

Você pode notar um padrão comum de um argumento nulo. Podemos fatorar isso:

struct details {
  template<template<class...>class Z, class=void, class...Ts>
  struct can_apply:
    std::false_type
  {};
  template<template<class...>class Z, class...Ts>
  struct can_apply<Z, std::void_t<Z<Ts...>>, Ts...>:
    std::true_type
  {};
};

template<template<class...>class Z, class...Ts>
using can_apply = details::can_apply<Z, void, Ts...>;

que oculta o uso de std::void_t e faz com que can_apply atue como um indicador se o tipo fornecido como o primeiro argumento do template está bem formado depois de substituir os outros tipos nele. Os exemplos anteriores podem agora ser reescritos usando can_apply como:

template<class T>
using ref_t = T&;

template<class T>
using can_reference = can_apply<ref_t, T>;    // Is T& well formed for T?

e:

template<class T>
using dot_foo_r = decltype(std::declval<T&>().foo());

template<class T>
using can_dot_foo = can_apply< dot_foo_r, T >;    // Is T.foo() well formed for T?

que parece mais simples do que as versões originais.

Existem propostas pós-C++17 para características std semelhantes a can_apply.

A utilidade de void_t foi descoberta por Walter Brown. Ele fez uma maravilhosa apresentação no CppCon 2016.

enable_if #

std::enable_if é um utilitário conveniente para usar condições booleanas para acionar o SFINAE. É definido como:

template <bool Cond, typename Result=void>
struct enable_if { };

template <typename Result>
struct enable_if<true, Result> {
    using type = Result;
};

Ou seja, enable_if<true, R>::type é um alias para R, enquanto enable_if<false, T>::type é mal formado, pois a especialização de enable_if não tem um type tipo de membro.

std::enable_if pode ser usado para restringir templates:

int negate(int i) { return -i; }

template <class F>
auto negate(F f) { return -f(); }

Aqui, uma chamada para negate(1) falharia devido à ambiguidade. Mas a segunda sobrecarga não se destina a ser usada para tipos integrais, então podemos adicionar:

int negate(int i) { return -i; }

template <class F, class = typename std::enable_if<!std::is_arithmetic<F>::value>::type>
auto negate(F f) { return -f(); }

Agora, instanciar negate<int> resultaria em uma falha de substituição já que !std::is_arithmetic<int>::value é false. Devido ao SFINAE, este não é um erro difícil, este candidato é simplesmente removido do conjunto de sobrecarga. Como resultado, negate(1) tem apenas um único candidato viável - que é então chamado.

Quando usar #

Vale a pena ter em mente que std::enable_if é um auxiliar em cima do SFINAE, mas não é o que faz o SFINAE funcionar em primeiro lugar. Vamos considerar essas duas alternativas para implementar funcionalidades semelhantes a std::size, ou seja, um conjunto de sobrecarga size(arg) que produz o tamanho de um contêiner ou array:

// for containers
template<typename Cont>
auto size1(Cont const& cont) -> decltype( cont.size() );

// for arrays
template<typename Elt, std::size_t Size>
std::size_t size1(Elt const(&arr)[Size]);

// implementation omitted
template<typename Cont>
struct is_sizeable;

// for containers
template<typename Cont, std::enable_if_t<std::is_sizeable<Cont>::value, int> = 0>
auto size2(Cont const& cont);

// for arrays
template<typename Elt, std::size_t Size>
std::size_t size2(Elt const(&arr)[Size]);

Assumindo que is_sizeable está escrito apropriadamente, essas duas declarações devem ser exatamente equivalentes em relação ao SFINAE. Qual é o mais fácil de escrever e qual é o mais fácil de revisar e entender de relance?

Agora vamos considerar como podemos querer implementar auxiliares aritméticos que evitem estouro de inteiro com sinal em favor do comportamento modular ou envolvente. Ou seja, por ex. incr(i, 3) seria o mesmo que i += 3 exceto pelo fato de que o resultado sempre seria definido mesmo que i fosse um int com valor INT_MAX. Estas são duas alternativas possíveis:

// handle signed types
template<typename Int>
auto incr1(Int& target, Int amount)
-> std::void_t<int[static_cast<Int>(-1) < static_cast<Int>(0)]>;

// handle unsigned types by just doing target += amount
// since unsigned arithmetic already behaves as intended
template<typename Int>
auto incr1(Int& target, Int amount)
-> std::void_t<int[static_cast<Int>(0) < static_cast<Int>(-1)]>;
 
template<typename Int, std::enable_if_t<std::is_signed<Int>::value, int> = 0>
void incr2(Int& target, Int amount);
 
template<typename Int, std::enable_if_t<std::is_unsigned<Int>::value, int> = 0>
void incr2(Int& target, Int amount);

Mais uma vez, qual é o mais fácil de escrever e qual é o mais fácil de revisar e entender de relance?

Um ponto forte de std::enable_if é como ele funciona com refatoração e design de API. Se is_sizeable<Cont>::value pretende refletir se cont.size() é válido, então apenas usar a expressão como aparece para size1 pode ser mais conciso, embora isso possa depender se is_sizeable seria usado em vários lugares ou não. Compare isso com std::is_signed que reflete sua intenção muito mais claramente do que quando sua implementação vaza na declaração de incr1.

é_detectado #

Para generalizar a criação de type_trait: com base no SFINAE existem traços experimentais detected_or, detected_t, is_detected.

Com os parâmetros de modelo typename Default, template <typename...> Op e typename ... Args:

  • is_detected: alias de std::true_type ou std::false_type dependendo da validade de Op<Args...>
  • detected_t: alias de Op<Args...> ou nonesuch dependendo da validade de Op<Args...>.
  • detected_or: alias de uma estrutura com value_t que é is_detected, e type que é Op<Args...> ou Default dependendo da validade de Op<Args...>

que pode ser implementado usando std::void_t para SFINAE da seguinte forma:

namespace detail {
    template <class Default, class AlwaysVoid,
              template<class...> class Op, class... Args>
    struct detector
    {
        using value_t = std::false_type;
        using type = Default;
    };

    template <class Default, template<class...> class Op, class... Args>
    struct detector<Default, std::void_t<Op<Args...>>, Op, Args...>
    {
        using value_t = std::true_type;
        using type = Op<Args...>;
    };

} // namespace detail

// special type to indicate detection failure
struct nonesuch {
    nonesuch() = delete;
    ~nonesuch() = delete;
    nonesuch(nonesuch const&) = delete;
    void operator=(nonesuch const&) = delete;
};

template <template<class...> class Op, class... Args>
using is_detected =
    typename detail::detector<nonesuch, void, Op, Args...>::value_t;

template <template<class...> class Op, class... Args>
using detected_t = typename detail::detector<nonesuch, void, Op, Args...>::type;

template <class Default, template<class...> class Op, class... Args>
using detected_or = detail::detector<Default, void, Op, Args...>;

As características para detectar a presença do método podem ser simplesmente implementadas:

typename <typename T, typename ...Ts>
using foo_type = decltype(std::declval<T>().foo(std::declval<Ts>()...));

struct C1 {};

struct C2 {
    int foo(char) const;
};

template <typename T>
using has_foo_char = is_detected<foo_type, T, char>;

static_assert(!has_foo_char<C1>::value, "Unexpected");
static_assert(has_foo_char<C2>::value, "Unexpected");

static_assert(std::is_same<int, detected_t<foo_type, C2, char>>::value,
              "Unexpected");

static_assert(std::is_same<void, // Default
                           detected_or<void, foo_type, C1, char>>::value,
              "Unexpected");
static_assert(std::is_same<int, detected_or<void, foo_type, C2, char>>::value,
              "Unexpected");

Resolução de sobrecarga com um grande número de opções #

Se você precisar selecionar entre várias opções, habilitar apenas um via enable_if<> pode ser bastante complicado, uma vez que várias condições também precisam ser negadas.

A ordenação entre sobrecargas pode ser selecionada usando herança, ou seja, despacho de tags.

Em vez de testar o que precisa ser bem formado, e também testando a negação de todas as outras condições de versão, testamos apenas para o que precisamos, de preferência em um decltype em um trailing return.
Isso pode deixar várias opções bem formadas, diferenciamos entre aquelas usando ’tags’, semelhante a tags de traço de iterador (random_access_tag et al). Isso funciona porque uma correspondência direta é melhor que uma classe base, que é melhor que uma classe base de uma classe base, etc.

#include <algorithm>
#include <iterator>

namespace detail
{
    // this gives us infinite types, that inherit from each other
    template<std::size_t N>
    struct pick : pick<N-1> {};
    template<>
    struct pick<0> {};

    // the overload we want to be preferred have a higher N in pick<N>
    // this is the first helper template function
    template<typename T>
    auto stable_sort(T& t, pick<2>)
        -> decltype( t.stable_sort(), void() )
    {
        // if the container have a member stable_sort, use that
        t.stable_sort();
    }

    // this helper will be second best match
    template<typename T>
    auto stable_sort(T& t, pick<1>)
        -> decltype( t.sort(), void() )
    {
        // if the container have a member sort, but no member stable_sort
        // it's customary that the sort member is stable
        t.sort();
    }

    // this helper will be picked last
    template<typename T>
    auto stable_sort(T& t, pick<0>)
        -> decltype( std::stable_sort(std::begin(t), std::end(t)), void() )
    {
        // the container have neither a member sort, nor member stable_sort
        std::stable_sort(std::begin(t), std::end(t));
    }

}

// this is the function the user calls. it will dispatch the call
// to the correct implementation with the help of 'tags'.
template<typename T>
void stable_sort(T& t)
{
    // use an N that is higher that any used above.
    // this will pick the highest overload that is well formed.
    detail::stable_sort(t, detail::pick<10>{});
}

Existem outros métodos comumente usados ​​para diferenciar entre sobrecargas, como a correspondência exata sendo melhor que a conversão, sendo melhor que as reticências.

No entanto, o despacho de tags pode se estender a qualquer número de opções e é um pouco mais claro na intenção.

decltype à direita em modelos de função #

Uma das funções de restrição é usar decltype à direita para especificar o tipo de retorno:

namespace details {
   using std::to_string;

   // this one is constrained on being able to call to_string(T)
   template <class T>
   auto convert_to_string(T const& val, int )
       -> decltype(to_string(val))
   {
       return to_string(val);
   }

   // this one is unconstrained, but less preferred due to the ellipsis argument
   template <class T>
   std::string convert_to_string(T const& val, ... )
   {
       std::ostringstream oss;
       oss << val;
       return oss.str();
   }
}

template <class T>
std::string convert_to_string(T const& val)
{
    return details::convert_to_string(val, 0);
}

Se eu chamar convert_to_string() com um argumento com o qual posso invocar to_string(), então tenho duas funções viáveis ​​para details::convert_to_string(). A primeira é preferida, pois a conversão de 0 para int é uma sequência de conversão implícita melhor do que a conversão de 0 para ...

Se eu chamar convert_to_string() com um argumento do qual não posso invocar to_string(), a primeira instanciação do modelo de função leva à falha de substituição (não há decltype(to_string(val))). Como resultado, esse candidato é removido do conjunto de sobrecarga. O segundo modelo de função é irrestrito, então ele é selecionado e, em vez disso, passamos por operator<<(std::ostream&, T). Se esse for indefinido, teremos um erro de compilação com uma pilha de templates na linha oss << val.

enable_if_all / enable_if_any #

Exemplo motivacional

Quando você tem um pacote de modelos variadic na lista de parâmetros do modelo, como no seguinte snippet de código:

template<typename ...Args> void func(Args &&...args) { //... };

A biblioteca padrão (antes do C++17) não oferece uma maneira direta de escrever enable_if para impor restrições SFINAE em todos os parâmetros em Args ou **qualquer um dos parâmetros ** em Args. C++17 oferece std::conjunction e std::disjunction que resolvem este problema. Por exemplo:

/// C++17: SFINAE constraints on all of the parameters in Args.
template<typename ...Args,
         std::enable_if_t<std::conjunction_v<custom_conditions_v<Args>...>>* = nullptr>
void func(Args &&...args) { //... };

/// C++17: SFINAE constraints on any of the parameters in Args.
template<typename ...Args,
         std::enable_if_t<std::disjunction_v<custom_conditions_v<Args>...>>* = nullptr>
void func(Args &&...args) { //... };

Se você não tiver o C++17 disponível, existem várias soluções para alcançá-los. Uma delas é usar uma classe de caso base e especializações parciais, conforme demonstrado nas respostas desta pergunta.

Alternativamente, pode-se também implementar manualmente o comportamento de std::conjunction e std::disjunction de uma forma bastante direta. No exemplo a seguir vou demonstrar as implementações e combiná-las com std::enable_if para produzir dois alias: enable_if_all e enable_if_any, que fazem exatamente o que devem fazer semanticamente. Isso pode fornecer uma solução mais escalável.

Implementação de enable_if_all e enable_if_any

Primeiro vamos emular std::conjunction e std::disjunction usando seq_and e seq_or personalizados respectivamente:

/// Helper for prior to C++14.
template<bool B, class T, class F >
using conditional_t = typename std::conditional<B,T,F>::type;

/// Emulate C++17 std::conjunction.
template<bool...> struct seq_or: std::false_type {};
template<bool...> struct seq_and: std::true_type {};

template<bool B1, bool... Bs>
struct seq_or<B1,Bs...>: 
  conditional_t<B1,std::true_type,seq_or<Bs...>> {};

template<bool B1, bool... Bs>
struct seq_and<B1,Bs...>:
  conditional_t<B1,seq_and<Bs...>,std::false_type> {};

Então a implementação é bastante direta:

template<bool... Bs>
using enable_if_any = std::enable_if<seq_or<Bs...>::value>;

template<bool... Bs>
using enable_if_all = std::enable_if<seq_and<Bs...>::value>;

Eventualmente alguns ajudantes:

template<bool... Bs>
using enable_if_any_t = typename enable_if_any<Bs...>::type;

template<bool... Bs>
using enable_if_all_t = typename enable_if_all<Bs...>::type;

Uso

O uso também é direto:

    /// SFINAE constraints on all of the parameters in Args.
    template<typename ...Args,
             enable_if_all_t<custom_conditions_v<Args>...>* = nullptr>
    void func(Args &&...args) { //... };

    /// SFINAE constraints on any of the parameters in Args.
    template<typename ...Args,
             enable_if_any_t<custom_conditions_v<Args>...>* = nullptr>
    void func(Args &&...args) { //... };
