Nesta página, você pode encontrar exemplos de como os padrões de design são implementados em C++. Para obter detalhes sobre esses padrões, você pode conferir a documentação de padrões de design.

Um padrão de projeto é uma solução geral reutilizável para um problema comum dentro de um determinado contexto no projeto de software.

Padrão do Adaptador #

Converter a interface de uma classe em outra interface que os clientes esperam. Adapter (ou Wrapper) permite que classes trabalhem juntas que não poderiam de outra forma devido a interfaces incompatíveis. A motivação do padrão do adaptador é que podemos reutilizar o software existente se pudermos modificar a interface.

  1. O padrão do adaptador depende da composição do objeto.

  2. O cliente chama a operação no objeto Adapter.

  3. O Adapter chama o Adaptee para realizar a operação.

  4. Em STL, pilha adaptada do vetor: Quando a pilha executa push(), o vetor subjacente executa vector::push_back().

Exemplo:

#include <iostream>

// Desired interface (Target)
class Rectangle 
{
  public:
    virtual void draw() = 0;
};

// Legacy component (Adaptee)
class LegacyRectangle 
{
  public:
    LegacyRectangle(int x1, int y1, int x2, int y2) {
        x1_ = x1;
        y1_ = y1;
        x2_ = x2;
        y2_ = y2;
        std::cout << "LegacyRectangle(x1,y1,x2,y2)\n";
    }
    void oldDraw() {
        std::cout << "LegacyRectangle:  oldDraw(). \n";
    }
  private:
    int x1_;
    int y1_;
    int x2_;
    int y2_;
};

// Adapter wrapper
class RectangleAdapter: public Rectangle, private LegacyRectangle 
{
  public:
    RectangleAdapter(int x, int y, int w, int h):
      LegacyRectangle(x, y, x + w, y + h) {
         std::cout << "RectangleAdapter(x,y,x+w,x+h)\n";
      }

    void draw() {
        std::cout << "RectangleAdapter: draw().\n"; 
        oldDraw();
    }
};

int main()
{
  int x = 20, y = 50, w = 300, h = 200;
  Rectangle *r = new RectangleAdapter(x,y,w,h);
  r->draw();
}

//Output:
//LegacyRectangle(x1,y1,x2,y2)
//RectangleAdapter(x,y,x+w,x+h)

Resumo do código:

  1. O cliente pensa que está falando com um retângulo

  2. O alvo é a classe Rectangle. É nisso que o cliente invoca o método.

     Rectangle *r = new RectangleAdapter(x,y,w,h);
 r->draw();

  3. Observe que a classe do adaptador usa herança múltipla.

     class RectangleAdapter: public Rectangle, private LegacyRectangle {
     ...
 }

  4. O Adapter RectangleAdapter permite que o LegacyRectangle responda ao pedido (draw() em um Rectangle) herdando AMBAS as classes.

  5. A classe LegacyRectangle não tem os mesmos métodos (draw()) que Rectangle, mas o Adapter(RectangleAdapter) pode receber as chamadas do método Rectangle e virar e invocar o método no LegacyRectangle, oldDraw().

     class RectangleAdapter: public Rectangle, private LegacyRectangle {
   public:
     RectangleAdapter(int x, int y, int w, int h):
       LegacyRectangle(x, y, x + w, y + h) {
         std::cout << "RectangleAdapter(x,y,x+w,x+h)\n";
       }

     void draw() {
         std::cout << "RectangleAdapter: draw().\n"; 
         oldDraw();
     }
 };

O padrão de design Adaptador converte a interface de uma classe em uma interface compatível, mas diferente. Portanto, isso é semelhante ao padrão proxy, pois é um wrapper de componente único. Mas a interface para a classe do adaptador e a classe original podem ser diferentes.

Como vimos no exemplo acima, esse padrão adaptador é útil para expor uma interface diferente para uma API existente para permitir que ela funcione com outro código. Além disso, usando o padrão de adaptador, podemos pegar interfaces heterogêneas e transformá-las para fornecer uma API consistente.

O padrão Bridge tem uma estrutura semelhante a um adaptador de objeto, mas o Bridge tem uma intenção diferente: destina-se a separar uma interface de sua implementação para que eles possam ser variados de forma fácil e independente. Um adaptador destina-se a alterar a interface de um objeto existente.

Padrão de observador #

A intenção do Observer Pattern é definir uma dependência um-para-muitos entre objetos para que quando um objeto mudar de estado, todos os seus dependentes sejam notificados e atualizados automaticamente.

O sujeito e os observadores definem a relação um-para-muitos. Os observadores são dependentes do sujeito de tal forma que quando o estado do sujeito muda, os observadores são notificados. Dependendo da notificação, os observadores também podem ser atualizados com novos valores.

Aqui está o exemplo do livro “Design Patterns” de Gamma.

#include <iostream>
#include <vector>

class Subject; 

class Observer 
{ 
public:
    virtual ~Observer() = default;
    virtual void Update(Subject&) = 0;
};

class Subject 
{ 
public: 
     virtual ~Subject() = default;
     void Attach(Observer& o) { observers.push_back(&o); }
     void Detach(Observer& o)
     {
         observers.erase(std::remove(observers.begin(), observers.end(), &o));
     }
     void Notify()
     {
         for (auto* o : observers) {
             o->Update(*this);
         }
     }
private:
     std::vector<Observer*> observers; 
};

class ClockTimer : public Subject 
{ 
public:

    void SetTime(int hour, int minute, int second)
    {
        this->hour = hour; 
        this->minute = minute;
        this->second = second;

        Notify(); 
    }

    int GetHour() const { return hour; }
    int GetMinute() const { return minute; }
    int GetSecond() const { return second; }

private: 
    int hour;
    int minute;
    int second;
}; 

class DigitalClock: public Observer 
{ 
public: 
     explicit DigitalClock(ClockTimer& s) : subject(s) { subject.Attach(*this); }
     ~DigitalClock() { subject.Detach(*this); }
     void Update(Subject& theChangedSubject) override
     {
         if (&theChangedSubject == &subject) {
             Draw();
         }
     }

     void Draw()
     {
         int hour = subject.GetHour(); 
         int minute = subject.GetMinute(); 
         int second = subject.GetSecond(); 

         std::cout << "Digital time is " << hour << ":" 
                   << minute << ":" 
                   << second << std::endl;           
     }

private:
     ClockTimer& subject;
};

class AnalogClock: public Observer 
{ 
public: 
     explicit AnalogClock(ClockTimer& s) : subject(s) { subject.Attach(*this); }
     ~AnalogClock() { subject.Detach(*this); }
     void Update(Subject& theChangedSubject) override
     {
         if (&theChangedSubject == &subject) {
             Draw();
         }
     }
     void Draw()
     {
         int hour = subject.GetHour(); 
         int minute = subject.GetMinute(); 
         int second = subject.GetSecond(); 

         std::cout << "Analog time is " << hour << ":" 
                   << minute << ":" 
                   << second << std::endl; 
     }
private:
     ClockTimer& subject;
};

int main()
{ 
    ClockTimer timer; 

    DigitalClock digitalClock(timer); 
    AnalogClock analogClock(timer); 

    timer.SetTime(14, 41, 36);
}

Resultado:

Digital time is 14:41:36
Analog time is 14:41:36

Aqui está o resumo do padrão:

  1. Objetos (objeto DigitalClock ou AnalogClock) usam as interfaces Subject (Attach() ou Detach()) para se inscrever (registrar) como observadores ou cancelar a inscrição (remover) de serem observadores (subject .Attach(*this); , assunto.Detach(*this); .

  2. Cada sujeito pode ter muitos observadores( vetor<Observador*> observadores;).

  3. Todos os observadores precisam implementar a interface Observer. Esta interface tem apenas um método, Update(), que é chamado quando o estado do Subject muda (Update(Subject &))

  4. Além dos métodos Attach() e Detach(), o assunto concreto implementa um método Notify() que é usado para atualizar todos os observadores atuais sempre que o estado muda. Mas neste caso, todos eles são feitos na classe pai, Subject (Subject::Attach (Observer&) , void Subject::Detach(Observer&) e void Subject::Notify() .

  5. O objeto Concrete também pode ter métodos para definir e obter seu estado.

  6. Os observadores concretos podem ser qualquer classe que implemente a interface Observer. Cada observador se inscreve (registra) com um assunto concreto para receber atualização (subject.Attach(*this); ).

  7. Os dois objetos do Observer Pattern são frouxamente acoplados, eles podem interagir mas com pouco conhecimento um do outro.

Variação:

Sinal e Slots

Sinais e slots é uma construção de linguagem introduzida no Qt, que facilita a implementação do padrão Observer, evitando código clichê. O conceito é que os controles (também conhecidos como widgets) podem enviar sinais contendo informações de eventos que podem ser recebidas por outros controles usando funções especiais conhecidas como slots. O slot em Qt deve ser um membro de classe declarado como tal. O sistema de sinal/slot se encaixa bem com a forma como as Interfaces Gráficas do Usuário são projetadas. Da mesma forma, o sistema de sinal/slot pode ser usado para notificação de eventos de E/S assíncrona (incluindo soquetes, pipes, dispositivos seriais, etc.) ou para associar eventos de tempo limite com instâncias de objetos e métodos ou funções apropriados. Nenhum código de registro/desregistro/invocação precisa ser escrito, porque o Meta Object Compiler (MOC) do Qt gera automaticamente a infraestrutura necessária.

A linguagem C# também suporta uma construção semelhante, embora com uma terminologia e sintaxe diferentes: os eventos desempenham o papel de sinais e os delegados são os slots. Além disso, um delegado pode ser uma variável local, muito parecido com um ponteiro de função, enquanto um slot no Qt deve ser um membro de classe declarado como tal.

Padrão de fábrica #

O padrão de fábrica desacopla a criação de objetos e permite a criação por nome usando uma interface comum:

class Animal{
public:
    virtual std::shared_ptr<Animal> clone() const = 0;
    virtual std::string  getname() const = 0;
};

class Bear: public Animal{
public:
    virtual std::shared_ptr<Animal> clone() const override
    {
        return std::make_shared<Bear>(*this);
    }
    virtual std::string getname() const override
    {
        return "bear";
    }
};


class Cat: public Animal{
public:
    virtual std::shared_ptr<Animal> clone() const override
    {
        return std::make_shared<Cat>(*this);
    }
    virtual std::string  getname() const override
    {
        return "cat";
    }
};

class AnimalFactory{
public:
    static std::shared_ptr<Animal> getAnimal( const std::string&   name )
    {
      if ( name == "bear" )
        return std::make_shared<Bear>();
      if ( name == "cat" )
        return std::shared_ptr<Cat>();
     
    return nullptr;
    }


};

Builder Pattern com API Fluent #

O Builder Pattern dissocia a criação do objeto do próprio objeto. A ideia principal é que um objeto não precisa ser responsável por sua própria criação. A montagem correta e válida de um objeto complexo pode ser uma tarefa complicada em si, portanto essa tarefa pode ser delegada a outra classe.

Inspirado no [Email Builder in C#](https://www.wikiod.com/pt/design-patterns/padrao-do-construtor#Builder Pattern / C# / Interface Fluente), eu decidi fazer uma versão C++ aqui. Um objeto Email não é necessariamente um objeto muito complexo, mas pode demonstrar o padrão.

#include <iostream>
#include <sstream>
#include <string>

using namespace std;

// Forward declaring the builder
class EmailBuilder;

class Email
{
  public:
    friend class EmailBuilder;  // the builder can access Email's privates
    
    static EmailBuilder make();
    
    string to_string() const {
        stringstream stream;
        stream << "from: " << m_from
               << "\nto: " << m_to
               << "\nsubject: " << m_subject
               << "\nbody: " << m_body;
        return stream.str();
    }
    
  private:
    Email() = default; // restrict construction to builder
    
    string m_from;
    string m_to;
    string m_subject;
    string m_body;
};

class EmailBuilder
{
  public:
    EmailBuilder& from(const string &from) {
        m_email.m_from = from;
        return *this;
    }
    
    EmailBuilder& to(const string &to) {
        m_email.m_to = to;
        return *this;
    }
    
    EmailBuilder& subject(const string &subject) {
        m_email.m_subject = subject;
        return *this;
    }
    
    EmailBuilder& body(const string &body) {
        m_email.m_body = body;
        return *this;
    }
    
    operator Email&&() {
        return std::move(m_email); // notice the move
    }
    
  private:
    Email m_email;
};

EmailBuilder Email::make()
{
    return EmailBuilder();
}

// Bonus example!
std::ostream& operator <<(std::ostream& stream, const Email& email)
{
    stream << email.to_string();
    return stream;
}


int main()
{
    Email mail = Email::make().from("me@mail.com")
                              .to("you@mail.com")
                              .subject("C++ builders")
                              .body("I like this API, don't you?");
                              
    cout << mail << endl;
}

Para versões mais antigas do C++, pode-se simplesmente ignorar a operação std::move e remover o && do operador de conversão (embora isso crie uma cópia temporária).

O construtor termina seu trabalho quando libera o email construído pelo operador Email&&(). Neste exemplo, o construtor é um objeto temporário e retorna o email antes de ser destruído. Você também pode usar uma operação explícita como Email EmailBuilder::build() {...} em vez do operador de conversão.

Passe o construtor ao redor #

Um ótimo recurso que o Builder Pattern oferece é a capacidade de usar vários atores para construir um objeto juntos. Isso é feito passando o construtor para os outros atores que cada um dará mais algumas informações ao objeto construído. Isso é especialmente poderoso quando você está construindo algum tipo de consulta, adicionando filtros e outras especificações.

void add_addresses(EmailBuilder& builder)
{
    builder.from("me@mail.com")
           .to("you@mail.com");
}

void compose_mail(EmailBuilder& builder)
{
    builder.subject("I know the subject")
           .body("And the body. Someone else knows the addresses.");
}

int main()
{
    EmailBuilder builder;
    add_addresses(builder);
    compose_mail(builder);
    
    Email mail = builder;
    cout << mail << endl;
}

Variante de design: objeto mutável #

Você pode alterar o design desse padrão para atender às suas necessidades. Vou dar uma variante.

No exemplo dado o objeto Email é imutável, ou seja, suas propriedades não podem ser modificadas porque não há acesso a elas. Esta era uma característica desejada. Se você precisar modificar o objeto após sua criação, deverá fornecer alguns setters para ele. Como esses setters seriam duplicados no construtor, você pode considerar fazer tudo em uma classe (nenhuma classe de construtor é mais necessária). No entanto, eu consideraria a necessidade de tornar o objeto construído mutável em primeiro lugar.
