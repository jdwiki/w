O que é comportamento indefinido (UB)? De acordo com o Padrão ISO C++ (§1.3.24, N4296), é “comportamento para o qual este Padrão Internacional não impõe requisitos”.

Isso significa que quando um programa encontra o UB, ele pode fazer o que quiser. Isso geralmente significa uma falha, mas pode simplesmente não fazer nada, fazer demônios voarem para fora do seu nariz, ou até parecer trabalhe corretamente!

Escusado será dizer que você deve evitar escrever código que invoque UB.

Se um programa contém comportamento indefinido, o padrão C++ não impõe restrições ao seu comportamento.

  • Pode parecer funcionar como o desenvolvedor pretendia, mas também pode travar ou produzir resultados estranhos.
  • O comportamento pode variar entre execuções do mesmo programa.
  • Qualquer parte do programa pode funcionar mal, incluindo linhas que vêm antes da linha que contém comportamento indefinido.
  • A implementação não é necessária para documentar o resultado do comportamento indefinido.

Uma implementação pode documentar o resultado de uma operação que produz comportamento indefinido de acordo com o padrão, mas um programa que depende de tal comportamento documentado não é portátil.

Por que o comportamento indefinido existe

Intuitivamente, o comportamento indefinido é considerado uma coisa ruim, pois tais erros não podem ser tratados graciosamente por, digamos, manipuladores de exceção.

Mas deixar algum comportamento indefinido é, na verdade, parte integrante da promessa do C++ “você não paga pelo que não usa”. O comportamento indefinido permite que um compilador assuma que o desenvolvedor sabe o que está fazendo e não introduza código para verificar os erros destacados nos exemplos acima.

Encontrar e evitar comportamentos indefinidos

Algumas ferramentas podem ser usadas para descobrir comportamentos indefinidos durante o desenvolvimento:

  • A maioria dos compiladores tem sinalizadores de alerta para alertar sobre alguns casos de comportamento indefinido em tempo de compilação.
  • Versões mais recentes do gcc e clang incluem o chamado sinalizador “Undefined Behavior Sanitizer” (-fsanitize=undefined) que verificará o comportamento indefinido em tempo de execução, a um custo de desempenho.
  • Ferramentas do tipo lint podem realizar análises de comportamento indefinidas mais completas.

Comportamento indefinido, não especificado e definido pela implementação

Do padrão C++14 (ISO/IEC 14882:2014) seção 1.9 (Execução do programa):

  1. As descrições semânticas nesta Norma Internacional definem uma máquina abstrata não determinística parametrizada. [CORTAR]

  2. Certos aspectos e operações da máquina abstrata são descritos nesta Norma Internacional como definidos pela implementação (por exemplo, sizeof(int)). Estes constituem os parâmetros da máquina abstrata. Cada implementação deve incluir documentação descrevendo suas características e comportamento nestes aspectos. [CORTAR]

  3. Certos outros aspectos e operações da máquina abstrata são descritos neste Padrão Internacional como não especificados (por exemplo, avaliação de expressões em um novo-inicializador se a função de alocação não alocar memória). Sempre que possível, esta Norma Internacional define um conjunto de comportamentos permitidos. Estes definem os aspectos não determinísticos da máquina abstrata. Uma instância da máquina abstrata pode, portanto, ter mais de uma execução possível para um determinado programa e uma determinada entrada.

  4. Certas outras operações são descritas nesta Norma Internacional como indefinidas (ou, por exemplo, o efeito de tentar modificar um objeto const). [ Nota: esta Norma Internacional não impõe requisitos sobre o comportamento de programas que contenham comportamento indefinido. —nota final]

Lendo ou escrevendo através de um ponteiro nulo #

int *ptr = nullptr;
*ptr = 1; // Undefined behavior

Este é um comportamento indefinido, porque um ponteiro nulo não aponta para nenhum objeto válido, portanto, não há nenhum objeto em *ptr para gravar.

Embora isso geralmente cause uma falha de segmentação, ela é indefinida e tudo pode acontecer.

Usando uma variável local não inicializada #

int a;
std::cout << a; // Undefined behavior!

Isso resulta em comportamento indefinido, porque a não é inicializado.

Muitas vezes, incorretamente, afirma-se que isso ocorre porque o valor é “indeterminado” ou “qualquer valor que estava naquele local de memória antes”. No entanto, é o ato de acessar o valor de a no exemplo acima que dá um comportamento indefinido. Na prática, imprimir um “valor lixo” é um sintoma comum neste caso, mas essa é apenas uma forma possível de comportamento indefinido.

Embora altamente improvável na prática (uma vez que depende de suporte de hardware específico), o compilador também pode eletrocutar o programador ao compilar o exemplo de código acima. Com tal compilador e suporte de hardware, tal resposta ao comportamento indefinido aumentaria marcadamente a compreensão do programador médio (vivo) do verdadeiro significado do comportamento indefinido - que é que o padrão não impõe restrições ao comportamento resultante.

Usar um valor indeterminado do tipo unsigned char não produz um comportamento indefinido se o valor for usado como:

  • o segundo ou terceiro operando do operador condicional ternário;
  • o operando direito do operador de vírgula embutido;
  • o operando de uma conversão para unsigned char;
  • o operando direito do operador de atribuição, se o operando esquerdo também for do tipo unsigned char;
  • o inicializador para um objeto unsigned char;

ou se o valor for descartado. Nesses casos, o valor indeterminado simplesmente se propaga para o resultado da expressão, se for o caso.

Observe que uma variável static é sempre inicializada com zero (se possível):

static int a;
std::cout << a; // Defined behavior, 'a' is 0

Acessando um índice fora dos limites #

É um comportamento indefinido acessar um índice que está fora dos limites de uma matriz (ou contêiner de biblioteca padrão, pois todos são implementados usando uma matriz raw):

 int array[] = {1, 2, 3, 4, 5};
 array[5] = 0;  // Undefined behavior

É permitido ter um ponteiro apontando para o final do array (neste caso array + 5), você só não pode desreferenciá-lo, pois não é um elemento válido.

 const int *end = array + 5;  // Pointer to one past the last index
 for (int *p = array; p != end; ++p)
   // Do something with `p`

Em geral, você não tem permissão para criar um ponteiro fora dos limites. Um ponteiro deve apontar para um elemento dentro da matriz, ou um após o final.

Excluindo um objeto derivado por meio de um ponteiro para uma classe base que não possui um destruidor virtual. #

class base { };
class derived: public base { }; 

int main() {
    base* p = new derived();
    delete p; // The is undefined behavior!
}

Na seção [expr.delete] §5.3.5/3 o padrão diz que se delete for chamado em um objeto cujo tipo estático não possui um destruidor virtual:

se o tipo estático do objeto a ser excluído for diferente de seu tipo dinâmico, o tipo estático deve ser uma classe base do tipo dinâmico do objeto a ser excluído e o tipo estático deve ter um destruidor virtual ou o comportamento é indefinido.

Este é o caso independentemente da questão se a classe derivada adicionou quaisquer membros de dados à classe base.

Estendendo o namespace std ou posix #

O padrão (17.6.4.2.1/1) geralmente proíbe estender o namespace std:

O comportamento de um programa C++ é indefinido se ele adiciona declarações ou definições ao namespace std ou a um namespace dentro do namespace std, a menos que especificado de outra forma.

O mesmo vale para posix (17.6.4.2.2/1):

O comportamento de um programa C++ é indefinido se ele adiciona declarações ou definições ao namespace posix ou a um namespace dentro do namespace posix, a menos que especificado de outra forma.

Considere o seguinte:

#include <algorithm>

namespace std
{
    int foo(){}
}

Nada no padrão proíbe o algoritmo (ou um dos cabeçalhos que ele inclui) definir a mesma definição e, portanto, esse código violaria a Regra de uma definição.

Então, em geral, isso é proibido. Existem exceções específicas permitidas, no entanto. Talvez mais útil, é permitido adicionar especializações para tipos definidos pelo usuário. Então, por exemplo, suponha que seu código tenha

class foo
{
    // Stuff
};

Então o seguinte está bem

namespace std
{
    template<>
    struct hash<foo>
    {
    public:
        size_t operator()(const foo &f) const;
    };
}

Nenhuma instrução de retorno para uma função com um tipo de retorno não void #

Omitir a instrução return em uma função que tem um tipo de retorno que não é void é um comportamento indefinido.

int function() {  
    // Missing return statement
} 

int main() {
    function(); //Undefined Behavior
}

A maioria dos compiladores modernos emite um aviso em tempo de compilação para esse tipo de comportamento indefinido.

Observação: main é a única exceção à regra. Se main não tiver uma instrução return, o compilador automaticamente insere return 0; para você, então pode ser deixado de fora com segurança.

Divisão inteira por zero #

int x = 5 / 0;    // Undefined behavior

A divisão por 0 é matematicamente indefinida e, como tal, faz sentido que este seja um comportamento indefinido.

No entanto:

float x = 5.0f / 0.0f;   // x is +infinity

A maioria das implementações implementa IEEE-754, que define a divisão de ponto flutuante por zero para retornar NaN (se o numerador for 0.0f), infinity (se o numerador for positivo) ou -infinity (se o numerador for negativo).

Acessando uma referência pendente #

É ilegal acessar uma referência a um objeto que saiu do escopo ou foi destruído de outra forma. Tal referência é considerada pendurada, pois não se refere mais a um objeto válido.

#include <iostream>
int& getX() {
    int x = 42;
    return x;
}
int main() {
    int& r = getX();
    std::cout << r << "\n";
}

Neste exemplo, a variável local x sai do escopo quando getX retorna. (Observe que extensão de tempo de vida não pode estender o tempo de vida de uma variável local além do escopo do bloco no qual ela está definida.) Portanto, r é uma referência pendente. Este programa tem um comportamento indefinido, embora possa parecer funcionar e imprimir 42 em alguns casos.

Aritmética de ponteiro inválida #

Os seguintes usos da aritmética de ponteiros causam um comportamento indefinido:

  • Adição ou subtração de um inteiro, se o resultado não pertencer ao mesmo objeto array que o operando ponteiro. (Aqui, o elemento um após o final é considerado como pertencente ao array.)

    int a[10];
int* p1 = &a[5];
int* p2 = p1 + 4; // ok; p2 points to a[9]
int* p3 = p1 + 5; // ok; p2 points to one past the end of a
int* p4 = p1 + 6; // UB
int* p5 = p1 - 5; // ok; p2 points to a[0]
int* p6 = p1 - 6; // UB
int* p7 = p3 - 5; // ok; p7 points to a[5]

  • Subtração de dois ponteiros se ambos não pertencerem ao mesmo objeto array. (Novamente, o elemento um após o final é considerado como pertencente à matriz.) A exceção é que dois ponteiros nulos podem ser subtraídos, resultando em 0.

    int a[10];
int b[10];
int *p1 = &a[8], *p2 = &a[3];
int d1 = p1 - p2; // yields 5
int *p3 = p1 + 2; // ok; p3 points to one past the end of a
int d2 = p3 - p2; // yields 7
int *p4 = &b[0];
int d3 = p4 - p1; // UB

  • Subtração de dois ponteiros se o resultado ultrapassar std::ptrdiff_t.

  • Qualquer aritmética de ponteiro onde o tipo pointee do operando não corresponde ao tipo dinâmico do objeto apontado (ignorando a qualificação cv). De acordo com o padrão, “[em] particular, um ponteiro para uma base classe não pode ser usada para aritmética de ponteiro quando a matriz contém objetos de um tipo de classe derivada.”

    struct Base { int x; };
struct Derived : Base { int y; };
Derived a[10];
Base* p1 = &a[1];           // ok
Base* p2 = p1 + 1;          // UB; p1 points to Derived
Base* p3 = p1 - 1;          // likewise
Base* p4 = &a[2];           // ok
auto p5 = p4 - p1;          // UB; p4 and p1 point to Derived
const Derived* p6 = &a[1];
const Derived* p7 = p6 + 1; // ok; cv-qualifiers don't matter

Estouro de inteiro assinado #

int x = INT_MAX + 1;

// x can be anything -> Undefined behavior

Se durante a avaliação de uma expressão, o resultado não for definido matematicamente ou não estiver no intervalo de valores representáveis ​​para seu tipo, o comportamento é indefinido.

(Parágrafo C++ 11 Padrão 5/4)

Este é um dos mais desagradáveis, pois geralmente produz um comportamento reprodutível e sem falhas, de modo que os desenvolvedores podem ser tentados a confiar fortemente no comportamento observado.

Por outro lado:

unsigned int x = UINT_MAX + 1;

// x is 0

está bem definida, pois:

Inteiros sem sinal, declarados sem sinal, devem obedecer às leis do módulo aritmético 2^n onde n é o número de bits na representação do valor daquele tamanho particular de inteiro.

(Parágrafo padrão C++ 11 3.9.1/4)

Às vezes, os compiladores podem explorar um comportamento indefinido e otimizar

signed int x ;
if(x > x + 1)
{
    //do something
}

Aqui, como um estouro de inteiro com sinal não está definido, o compilador está livre para assumir que isso pode nunca acontecer e, portanto, pode otimizar o bloco “if”

Emparelhamento incorreto de alocação e desalocação de memória #

Um objeto só pode ser desalocado por delete se tiver sido alocado por new e não for um array. Se o argumento para delete não foi retornado por new ou é um array, o comportamento é indefinido.

Um objeto só pode ser desalocado por delete[] se tiver sido alocado por new e for um array. Se o argumento para delete[] não foi retornado por new ou não é um array, o comportamento é indefinido.

Se o argumento para free não foi retornado por malloc, o comportamento é indefinido.

int* p1 = new int;
delete p1;      // correct
// delete[] p1; // undefined
// free(p1);    // undefined

int* p2 = new int[10];
delete[] p2;    // correct
// delete p2;   // undefined
// free(p2);    // undefined

int* p3 = static_cast<int*>(malloc(sizeof(int)));
free(p3);       // correct
// delete p3;   // undefined
// delete[] p3; // undefined

Tais problemas podem ser evitados evitando completamente malloc e free em programas C++, preferindo os ponteiros inteligentes da biblioteca padrão sobre new e delete brutos, e preferindo std::vector e std::string sobre raw new e delete[].

Deslocamento por um número inválido de posições #

Para o operador shift integrado, o operando direito deve ser não negativo e estritamente menor que a largura de bits do operando esquerdo promovido. Caso contrário, o comportamento é indefinido.

const int a = 42;
const int b = a << -1; // UB
const int c = a << 0;  // ok
const int d = a << 32; // UB if int is 32 bits or less
const int e = a >> 32; // also UB if int is 32 bits or less
const signed char f = 'x';
const int g = f << 10; // ok even if signed char is 10 bits or less;
                       // int must be at least 16 bits

Várias definições não idênticas (a regra de uma definição) #

Se uma classe, enumeração, função embutida, modelo ou membro de um modelo tiver ligação externa e for definido em várias unidades de tradução, todas as definições deverão ser idênticas ou o comportamento será indefinido de acordo com a [Regra de uma definição (ODR)](https ://en.wikipedia.org/wiki/One_Definition_Rule).

foo.h:

class Foo {
  public:
    double x;
  private:
    int y;
};

Foo get_foo();

foo.cpp:

#include "foo.h"
Foo get_foo() { /* implementation */ }

main.cpp:

// I want access to the private member, so I am going to replace Foo with my own type
class Foo {
  public:
    double x;
    int y;
};
Foo get_foo(); // declare this function ourselves since we aren't including foo.h
int main() {
    Foo foo = get_foo();
    // do something with foo.y
}

O programa acima apresenta um comportamento indefinido porque contém duas definições da classe ::Foo, que possui ligação externa, em diferentes unidades de tradução, mas as duas definições não são idênticas. Ao contrário da redefinição de uma classe dentro da mesma unidade de tradução, esse problema não precisa ser diagnosticado pelo compilador.

Modificando um objeto const #

Qualquer tentativa de modificar um objeto const resulta em comportamento indefinido. Isso se aplica a variáveis ​​const, membros de objetos const e membros de classe declarados const. (Entretanto, um membro mutable de um objeto const não é const.)

Tal tentativa pode ser feita através de const_cast:

const int x = 123;
const_cast<int&>(x) = 456;
std::cout << x << '\n';

Um compilador normalmente inline o valor de um objeto const int, então é possível que este código compile e imprima 123. Os compiladores também podem colocar valores de objetos const na memória somente leitura, portanto, pode ocorrer uma falha de segmentação. Em qualquer caso, o comportamento é indefinido e o programa pode fazer qualquer coisa.

O programa a seguir esconde um erro muito mais sutil:

#include <iostream>

class Foo* instance;

class Foo {
  public:
    int get_x() const { return m_x; }
    void set_x(int x) { m_x = x; }
  private:
    Foo(int x, Foo*& this_ref): m_x(x) {
        this_ref = this;
    }
    int m_x;
    friend const Foo& getFoo();
};

const Foo& getFoo() {
    static const Foo foo(123, instance);
    return foo;
}

void do_evil(int x) {
    instance->set_x(x);
}

int main() {
    const Foo& foo = getFoo();
    do_evil(456);
    std::cout << foo.get_x() << '\n';
}

Neste código, getFoo cria um singleton do tipo const Foo e seu membro m_x é inicializado como 123. Então do_evil é chamado e o valor de foo.m_x é aparentemente alterado para 456. O que deu errado?

Apesar do nome, do_evil não faz nada particularmente mal; tudo o que ele faz é chamar um setter através de um Foo*. Mas esse ponteiro aponta para um objeto const Foo mesmo que const_cast não tenha sido usado. Este ponteiro foi obtido através do construtor do Foo. Um objeto const não se torna const até que sua inicialização esteja completa, então this tem o tipo Foo*, não const Foo*, dentro do construtor.

Portanto, o comportamento indefinido ocorre mesmo que não haja construções obviamente perigosas neste programa.

Modificando um literal de string #

char *str = "hello world";
str[0] = 'H';

"hello world" é uma string literal, então modificá-la dá um comportamento indefinido.

A inicialização de str no exemplo acima foi formalmente obsoleta (programada para remoção de uma versão futura do padrão) em C++03. Vários compiladores antes de 2003 podem emitir um aviso sobre isso (por exemplo, uma conversão suspeita). Após 2003, os compiladores normalmente avisam sobre uma conversão obsoleta.

O exemplo acima é ilegal e resulta em um diagnóstico do compilador, em C++11 e posterior. Um exemplo semelhante pode ser construído para exibir um comportamento indefinido permitindo explicitamente a conversão de tipo, como:

char *str = const_cast<char *>("hello world");
str[0] = 'H';

Acessando um objeto como o tipo errado #

Na maioria dos casos, é ilegal acessar um objeto de um tipo como se fosse um tipo diferente (desconsiderando os qualificadores cv). Exemplo:

float x = 42;
int y = reinterpret_cast<int&>(x);

O resultado é um comportamento indefinido.

Existem algumas exceções a essa regra de aliasing estrito:

  • Um objeto do tipo classe pode ser acessado como se fosse de um tipo que é uma classe base do tipo de classe real.
  • Qualquer tipo pode ser acessado como char ou unsigned char, mas o inverso não é verdadeiro: um array char não pode ser acessado como se fosse um tipo arbitrário.
  • Um tipo inteiro assinado pode ser acessado como o tipo não assinado correspondente e vice-versa.

Uma regra relacionada é que, se uma função membro não estática for chamada em um objeto que na verdade não tem o mesmo tipo que a classe definidora da função ou uma classe derivada, ocorrerá um comportamento indefinido. Isso é verdade mesmo se a função não acessar o objeto.

struct Base {
};
struct Derived : Base {
    void f() {}
};
struct Unrelated {};
Unrelated u;
Derived& r1 = reinterpret_cast<Derived&>(u); // ok
r1.f();                                      // UB
Base b;
Derived& r2 = reinterpret_cast<Derived&>(b); // ok
r2.f();                                      // UB

Estouro durante a conversão para ou do tipo de ponto flutuante #

Se, durante a conversão de:

  • um tipo inteiro para um tipo de ponto flutuante,
  • um tipo de ponto flutuante para um tipo inteiro, ou
  • um tipo de ponto flutuante para um tipo de ponto flutuante mais curto,

o valor de origem está fora do intervalo de valores que podem ser representados no tipo de destino, o resultado é um comportamento indefinido. Exemplo:

double x = 1e100;
int y = x; // int probably cannot hold numbers that large, so this is UB

Retornando de uma função [[noreturn]] #

Exemplo do Padrão, [dcl.attr.noreturn]:

[[ noreturn ]] void f() {
  throw "error"; // OK
}
[[ noreturn ]] void q(int i) { // behavior is undefined if called with an argument <= 0
  if (i > 0)
    throw "positive";
}

Recursão de template infinita #

Exemplo do Padrão, [temp.inst]/17:

template<class T> class X {
    X<T>* p; // OK
    X<T*> a; // implicit generation of X<T> requires
             // the implicit instantiation of X<T*> which requires
             // the implicit instantiation of X<T**> which ...
};

Estouro de ponto flutuante #

Se uma operação aritmética que produz um tipo de ponto flutuante produz um valor que não está no intervalo de valores representáveis ​​do tipo de resultado, o comportamento é indefinido de acordo com o padrão C++, mas pode ser definido por outros padrões aos quais a máquina pode estar em conformidade, como IEEE 754.

float x = 1.0;
for (int i = 0; i < 10000; i++) {
    x *= 10.0; // will probably overflow eventually; undefined behavior
}

Chamando (Puro) Membros Virtuais do Construtor ou Destruidor #

O Padrão (10.4) declara:

As funções-membro podem ser chamadas de um construtor (ou destruidor) de uma classe abstrata; o efeito de fazer uma chamada virtual (10.3) para uma função virtual pura direta ou indiretamente para o objeto que está sendo criado (ou destruído) a partir de tal construtor (ou destruidor) é indefinido.

Mais geralmente, algumas autoridades C++, e. Scott Meyers, sugere nunca chamando funções virtuais (mesmo as não puras) de construtores e dstructors.

Considere o seguinte exemplo, modificado a partir do link acima:

class transaction
{
public:
    transaction(){ log_it(); }
    virtual void log_it() const = 0;
};

class sell_transaction : public transaction
{
public:
    virtual void log_it() const { /* Do something */ }
};

Suponha que criamos um objeto sell_transaction:

sell_transaction s;

Isso implicitamente chama o construtor de sell_transaction, que primeiro chama o construtor de transaction. Porém, quando o construtor de transaction é chamado, o objeto ainda não é do tipo sell_transaction, mas apenas do tipo transaction.

Conseqüentemente, a chamada em transaction::transaction() para log_it, não fará o que pode parecer intuitivo - ou seja, chamar sell_transaction::log_it.

  • Se log_it for virtual puro, como neste exemplo, o comportamento é indefinido.

  • Se log_it for virtual não puro, transaction::log_it será chamado.

Conversão estática de base para derivada inválida #

Se static_cast for usado para converter um ponteiro (resp.referência) para uma classe base para um ponteiro (resp.referência) para uma classe derivada, mas o operando não apontar (resp.referência) para um objeto do tipo de classe derivada, o comportamento é indefinido. Consulte Base para conversão derivada.

Chamada de função por meio de tipo de ponteiro de função incompatível #

Para chamar uma função por meio de um ponteiro de função, o tipo do ponteiro de função deve corresponder exatamente ao tipo da função. Caso contrário, o comportamento é indefinido. Exemplo:

int f();
void (*p)() = reinterpret_cast<void(*)()>(f);
p(); // undefined

Acesso a membro inexistente através de ponteiro para membro #

Ao acessar um membro não estático de um objeto por meio de um ponteiro para membro, se o objeto não contiver realmente o membro indicado pelo ponteiro, o comportamento será indefinido. (Tal ponteiro para membro pode ser obtido através de static_cast.)

struct Base { int x; };
struct Derived : Base { int y; };
int Derived::*pdy = &Derived::y;
int Base::*pby = static_cast<int Base::*>(pdy);

Base* b1 = new Derived;
b1->*pby = 42; // ok; sets y in Derived object to 42
Base* b2 = new Base;
b2->*pby = 42; // undefined; there is no y member in Base

Conversão derivada para base inválida para ponteiros para membros #

Quando static_cast é usado para converter T D::* para TB::*, o membro apontado deve pertencer a uma classe que é uma classe base ou classe derivada de B. Caso contrário, o comportamento é indefinido. Consulte [Conversão derivada para base de ponteiros para membros](https://www.wikiod.com/pt/docs/c%2b%2b/3090/explicit-type-conversions/18752/derived-to-base-conversion-for-pointers- para membros)

Destruindo um objeto que já foi destruído #

Neste exemplo, um destruidor é invocado explicitamente para um objeto que posteriormente será destruído automaticamente.

struct S {
    ~S() { std::cout << "destroying S\n"; }
};
int main() {
    S s;
    s.~S();
} // UB: s destroyed a second time here

Um problema semelhante ocorre quando um std::unique_ptr<T> é feito para apontar para um T com duração de armazenamento automática ou estática.

void f(std::unique_ptr<S> p);
int main() {
    S s;
    std::unique_ptr<S> p(&s);
    f(std::move(p)); // s destroyed upon return from f
}                    // UB: s destroyed

Outra maneira de destruir um objeto duas vezes é fazer com que dois shared_ptr gerenciem o objeto sem compartilhar a propriedade um com o outro.

void f(std::shared_ptr<S> p1, std::shared_ptr<S> p2);
int main() {
    S* p = new S;
    // I want to pass the same object twice...
    std::shared_ptr<S> sp1(p);
    std::shared_ptr<S> sp2(p);
    f(sp1, sp2);
} // UB: both sp1 and sp2 will destroy s separately
// NB: this is correct:
// std::shared_ptr<S> sp(p);
// f(sp, sp);
