Sintaxe #

  • std::shared_ptr<ClassType> nome da variável = std::make_shared<ClassType>(arg1, arg2, ...);
  • std::shared_ptr<ClassType> variableName (new ClassType(arg1, arg2, ...));
  • std::unique_ptr<ClassType> variableName = std::make_unique<ClassType>(arg1, arg2, ...); // C++14
  • std::unique_ptr<ClassType> variableName (new ClassType(arg1, arg2, ...));

C++ não é uma linguagem gerenciada por memória. A memória alocada dinamicamente (ou seja, objetos criados com new) será “vazada” se não for explicitamente desalocada (com delete). É responsabilidade do programador garantir que a memória alocada dinamicamente seja liberada antes de descartar o último ponteiro para aquele objeto.

Ponteiros inteligentes podem ser usados ​​para gerenciar automaticamente o escopo da memória alocada dinamicamente (ou seja, quando a última referência de ponteiro sai do escopo, ela é excluída).

Os ponteiros inteligentes são preferidos aos ponteiros “brutos” na maioria dos casos. Eles tornam explícita a semântica de propriedade da memória alocada dinamicamente, comunicando em seus nomes se um objeto deve ser compartilhado ou de propriedade exclusiva.

Use #include <memory> para poder usar ponteiros inteligentes.

Propriedade exclusiva (std::unique_ptr) #

Um std::unique_ptr é um modelo de classe que gerencia o tempo de vida de um objeto armazenado dinamicamente. Ao contrário de std::shared_ptr, o objeto dinâmico pertence a apenas uma instância de um std::unique_ptr a qualquer momento,

// Creates a dynamic int with value of 20 owned by a unique pointer
std::unique_ptr<int> ptr = std::make_unique<int>(20);

(Nota: std::unique_ptr está disponível desde C++11 e std::make_unique desde C++14.)

Apenas a variável ptr contém um ponteiro para um int alocado dinamicamente. Quando um ponteiro único que possui um objeto sai do escopo, o objeto de propriedade é excluído, ou seja, seu destruidor é chamado se o objeto for do tipo de classe e a memória desse objeto é liberada.

Para usar std::unique_ptr e std::make_unique com tipos de array, use suas especializações de array:

// Creates a unique_ptr to an int with value 59
std::unique_ptr<int> ptr = std::make_unique<int>(59);

// Creates a unique_ptr to an array of 15 ints
std::unique_ptr<int[]> ptr = std::make_unique<int[]>(15);

Você pode acessar o std::unique_ptr apenas como um ponteiro bruto, porque sobrecarrega esses operadores.

Você pode transferir a propriedade do conteúdo de um ponteiro inteligente para outro ponteiro usando std::move, que fará com que o ponteiro inteligente original aponte para nullptr.

// 1. std::unique_ptr
std::unique_ptr<int> ptr = std::make_unique<int>();

// Change value to 1
*ptr = 1;

// 2. std::unique_ptr (by moving 'ptr' to 'ptr2', 'ptr' doesn't own the object anymore)
std::unique_ptr<int> ptr2 = std::move(ptr);

int a = *ptr2; // 'a' is 1
int b = *ptr;  // undefined behavior! 'ptr' is 'nullptr'
               // (because of the move command above)

Passando unique_ptr para funções como parâmetro:

void foo(std::unique_ptr<int> ptr)
{
    // Your code goes here
}

std::unique_ptr<int> ptr = std::make_unique<int>(59);
foo(std::move(ptr))

Retornando unique_ptr de funções. Essa é a maneira preferida do C++ 11 de escrever funções de fábrica, pois transmite claramente a semântica de propriedade do retorno: o chamador possui o unique_ptr resultante e é responsável por ele.

std::unique_ptr<int> foo()
{
    std::unique_ptr<int> ptr = std::make_unique<int>(59);
    return ptr;
}

std::unique_ptr<int> ptr = foo();

Compare isso com:

int* foo_cpp03();

int* p = foo_cpp03(); // do I own p? do I have to delete it at some point?
                      // it's not readily apparent what the answer is.

O modelo de classe make_unique é fornecido desde o C++14. É fácil adicioná-lo manualmente ao código C++11:

template<typename T, typename... Args>
typename std::enable_if<!std::is_array<T>::value, std::unique_ptr<T>>::type
make_unique(Args&&... args)
{ return std::unique_ptr<T>(new T(std::forward<Args>(args)...)); }

// Use make_unique for arrays
template<typename T>
typename std::enable_if<std::is_array<T>::value, std::unique_ptr<T>>::type
make_unique(size_t n)
{ return std::unique_ptr<T>(new typename std::remove_extent<T>::type[n]()); }

Ao contrário do ponteiro inteligente burro (std::auto_ptr), unique_ptr também pode ser instanciado com alocação de vetor (não std::vector). Os exemplos anteriores eram para alocações escalares. Por exemplo, para ter um array inteiro alocado dinamicamente para 10 elementos, você especificaria int[] como o tipo de template (e não apenas int):

std::unique_ptr<int[]> arr_ptr = std::make_unique<int[]>(10);

O que pode ser simplificado com:

auto arr_ptr = std::make_unique<int[]>(10);

Agora, você usa arr_ptr como se fosse um array:

arr_ptr[2] =  10; // Modify third element

Você não precisa se preocupar com a desalocação. Esta versão especializada de template chama construtores e destruidores apropriadamente. Usar a versão vetorial de unique_ptr ou um vetor em si - é uma escolha pessoal.

Nas versões anteriores ao C++11, o std::auto_ptr estava disponível. Ao contrário do unique_ptr, é permitido copiar auto_ptrs, sobre os quais o ptr de origem perderá a propriedade do ponteiro contido e o destino o receberá.

Compartilhamento de propriedade (std::shared_ptr) #

O modelo de classe std::shared_ptr define um ponteiro compartilhado que é capaz de compartilhar a propriedade de um objeto com outros ponteiros compartilhados. Isso contrasta com std::unique_ptr que representa propriedade exclusiva.

O comportamento de compartilhamento é implementado por meio de uma técnica conhecida como contagem de referência, onde o número de ponteiros compartilhados que apontam para o objeto é armazenado ao lado dele. Quando esta contagem chega a zero, seja através da destruição ou reatribuição da última instância std::shared_ptr, o objeto é automaticamente destruído.

// Creation: 'firstShared' is a shared pointer for a new instance of 'Foo'
std::shared_ptr<Foo> firstShared = std::make_shared<Foo>(/*args*/);

Para criar vários ponteiros inteligentes que compartilham o mesmo objeto, precisamos criar outro shared_ptr que aliase o primeiro ponteiro compartilhado. Aqui estão 2 maneiras de fazer isso:

std::shared_ptr<Foo> secondShared(firstShared);  // 1st way: Copy constructing
std::shared_ptr<Foo> secondShared;
secondShared = firstShared;                      // 2nd way: Assigning

Qualquer uma das maneiras acima torna secondShared um ponteiro compartilhado que compartilha a propriedade de nossa instância de Foo com firstShared.

O ponteiro inteligente funciona como um ponteiro bruto. Isso significa que você pode usar * para desreferenciá-los. O operador -> normal também funciona:

secondShared->test(); // Calls Foo::test()

Finalmente, quando o último alias shared_ptr sai do escopo, o destruidor de nossa instância Foo é chamado.

Aviso: construir um shared_ptr pode lançar uma exceção bad_alloc quando dados extras para semântica de propriedade compartilhada precisam ser alocados. Se o construtor recebe um ponteiro regular, ele assume possuir o objeto apontado e chama o deleter se uma exceção for lançada. Isso significa que shared_ptr<T>(novo T(args)) não vazará um objeto T se a alocação de shared_ptr<T> falhar. No entanto, é aconselhável usar make_shared<T>(args) ou allocate_shared<T>(alloc, args), que permitem a implementação otimizar a alocação de memória.

Alocando Arrays([]) usando shared_ptr

Infelizmente, não existe uma maneira direta de alocar Arrays usando make_shared<>.

É possível criar arrays para shared_ptr<> usando new e std::default_delete.

Por exemplo, para alocar um array de 10 inteiros, podemos escrever o código como

shared_ptr<int> sh(new int[10], std::default_delete<int[]>());

Especificar std::default_delete é obrigatório aqui para garantir que a memória alocada seja limpa corretamente usando delete[].

Se soubermos o tamanho em tempo de compilação, podemos fazer assim:

template<class Arr>
struct shared_array_maker {};
template<class T, std::size_t N>
struct shared_array_maker<T[N]> {
  std::shared_ptr<T> operator()const{
    auto r = std::make_shared<std::array<T,N>>();
    if (!r) return {};
    return {r.data(), r};
  }
};
template<class Arr>
auto make_shared_array()
-> decltype( shared_array_maker<Arr>{}() )
{ return shared_array_maker<Arr>{}(); }

então make_shared_array<int[10]> retorna um shared_ptr<int> apontando para 10 ints todos construídos por padrão.

Com C++17, shared_ptr obteve suporte especial para tipos de array. Não é mais necessário especificar explicitamente o apagador de array, e o ponteiro compartilhado pode ser desreferenciado usando o operador de índice de array []:

std::shared_ptr<int[]> sh(new int[10]);
sh[0] = 42;

Ponteiros compartilhados podem apontar para um subobjeto do objeto que ele possui:

struct Foo { int x; };
std::shared_ptr<Foo> p1 = std::make_shared<Foo>();
std::shared_ptr<int> p2(p1, &p1->x);

Ambos p2 e p1 possuem o objeto do tipo Foo, mas p2 aponta para seu membro int x. Isso significa que se p1 sair do escopo ou for reatribuído, o objeto Foo subjacente ainda estará ativo, garantindo que p2 não oscila.

Importante: Um shared_ptr só conhece a si mesmo e todos os outros shared_ptr que foram criados com o construtor alias. Ele não conhece nenhum outro ponteiro, incluindo todos os outros shared_ptrs criados com uma referência à mesma instância Foo:

Foo *foo = new Foo;
std::shared_ptr<Foo> shared1(foo);
std::shared_ptr<Foo> shared2(foo); // don't do this

shared1.reset(); // this will delete foo, since shared1
                 // was the only shared_ptr that owned it

shared2->test(); // UNDEFINED BEHAVIOR: shared2's foo has been
                 // deleted already!!

Transferência de propriedade de shared_ptr

Por padrão, shared_ptr incrementa a contagem de referência e não transfere a propriedade. No entanto, pode ser feito para transferir a propriedade usando std::move:

shared_ptr<int> up = make_shared<int>();
// Transferring the ownership
shared_ptr<int> up2 = move(up);
// At this point, the reference count of up = 0 and the
// ownership of the pointer is solely with up2 with reference count = 1

Compartilhamento com propriedade temporária (std::weak_ptr) #

Instâncias de std::weak_ptr podem apontar para objetos pertencentes a instâncias de std::shared_ptr enquanto se tornam proprietários temporários. Isso significa que os ponteiros fracos não alteram a contagem de referência do objeto e, portanto, não impedem a exclusão de um objeto se todos os ponteiros compartilhados do objeto forem reatribuídos ou destruídos.

No exemplo a seguir, instâncias de std::weak_ptr são usadas para que a destruição de um objeto de árvore não seja inibida:

#include <memory>
#include <vector>

struct TreeNode {
    std::weak_ptr<TreeNode> parent;
    std::vector< std::shared_ptr<TreeNode> > children;
};

int main() {
    // Create a TreeNode to serve as the root/parent.
    std::shared_ptr<TreeNode> root(new TreeNode);

    // Give the parent 100 child nodes.
    for (size_t i = 0; i < 100; ++i) {
        std::shared_ptr<TreeNode> child(new TreeNode);
        root->children.push_back(child);
        child->parent = root;
    }

    // Reset the root shared pointer, destroying the root object, and
    // subsequently its child nodes.
    root.reset();
}

À medida que os nós filhos são adicionados aos filhos do nó raiz, seu membro std::weak_ptr parent é definido como o nó raiz. O membro parent é declarado como um ponteiro fraco em oposição a um ponteiro compartilhado, de modo que a contagem de referência do nó raiz não seja incrementada. Quando o nó raiz é redefinido no final de main(), a raiz é destruída. Como as únicas referências std::shared_ptr restantes aos nós filhos estavam contidas na coleção children da raiz, todos os nós filhos são subsequentemente destruídos também.

Devido aos detalhes de implementação do bloco de controle, a memória alocada shared_ptr pode não ser liberada até que o contador de referência shared_ptr e o contador de referência weak_ptr cheguem a zero.

#include <memory>
int main()
{
    {
         std::weak_ptr<int> wk;
         {
             // std::make_shared is optimized by allocating only once 
             // while std::shared_ptr<int>(new int(42)) allocates twice.
             // Drawback of std::make_shared is that control block is tied to our integer
             std::shared_ptr<int> sh = std::make_shared<int>(42);
             wk = sh;
             // sh memory should be released at this point...
         }
         // ... but wk is still alive and needs access to control block
     }
     // now memory is released (sh and wk)
}

Como std::weak_ptr não mantém seu objeto referenciado ativo, o acesso direto aos dados através de um std::weak_ptr não é possível. Em vez disso, ele fornece uma função de membro lock() que tenta recuperar um std::shared_ptr para o objeto referenciado:

#include <cassert>
#include <memory>
int main()
{
    {
         std::weak_ptr<int> wk;
         std::shared_ptr<int> sp;
         {
             std::shared_ptr<int> sh = std::make_shared<int>(42);
             wk = sh;
             // calling lock will create a shared_ptr to the object referenced by wk
             sp = wk.lock();
             // sh will be destroyed after this point, but sp is still alive
         }
         // sp still keeps the data alive.
         // At this point we could even call lock() again 
         // to retrieve another shared_ptr to the same data from wk
         assert(*sp == 42);
         assert(!wk.expired());
         // resetting sp will delete the data,
         // as it is currently the last shared_ptr with ownership
         sp.reset();
         // attempting to lock wk now will return an empty shared_ptr,
         // as the data has already been deleted
         sp = wk.lock();
         assert(!sp);
         assert(wk.expired());
     }
}

Usando deletores personalizados para criar um wrapper para uma interface C #

Muitas interfaces C como SDL2 têm suas próprias funções de exclusão. Isso significa que você não pode usar ponteiros inteligentes diretamente:

std::unique_ptr<SDL_Surface> a; // won't work, UNSAFE!

Em vez disso, você precisa definir seu próprio apagador. Os exemplos aqui usam a estrutura SDL_Surface que deve ser liberada usando a função SDL_FreeSurface(), mas devem ser adaptáveis ​​a muitas outras interfaces C.

O apagador deve ser chamado com um argumento de ponteiro e, portanto, pode ser, por exemplo, um ponteiro de função simples:

std::unique_ptr<SDL_Surface, void(*)(SDL_Surface*)> a(pointer, SDL_FreeSurface);

Qualquer outro objeto que pode ser chamado também funcionará, por exemplo, uma classe com um operator():

struct SurfaceDeleter {
    void operator()(SDL_Surface* surf) {
        SDL_FreeSurface(surf);
    }
};

std::unique_ptr<SDL_Surface, SurfaceDeleter> a(pointer, SurfaceDeleter{}); // safe
std::unique_ptr<SDL_Surface, SurfaceDeleter> b(pointer); // equivalent to the above
                                                         // as the deleter is value-initialized

Isso não apenas fornece a você um gerenciamento automático de memória seguro e zero (se você usar unique_ptr), você também obtém segurança de exceção.

Observe que o apagador é parte do tipo para unique_ptr, e a implementação pode usar a [otimização de base vazia](https://www.wikiod.com/pt/docs/c%2b%2b/3944/empty-base-optimization# t=201607261453124463486) para evitar qualquer alteração no tamanho de apagadores personalizados vazios. Portanto, enquanto std::unique_ptr<SDL_Surface, SurfaceDeleter> e std::unique_ptr<SDL_Surface, void(*)(SDL_Surface*)> resolvem o mesmo problema de maneira semelhante, o primeiro tipo ainda é apenas do tamanho de um ponteiro enquanto o último tipo tem que conter dois ponteiros: tanto o SDL_Surface* quanto o ponteiro de função! Ao ter deletores personalizados de função livre, é preferível envolver a função em um tipo vazio.

Nos casos em que a contagem de referência é importante, pode-se usar um shared_ptr em vez de um unique_ptr. O shared_ptr sempre armazena um deleter, isso apaga o tipo do deleter, o que pode ser útil em APIs. As desvantagens de usar shared_ptr sobre unique_ptr incluem um custo de memória mais alto para armazenar o apagador e um custo de desempenho para manter a contagem de referência.

// deleter required at construction time and is part of the type
std::unique_ptr<SDL_Surface, void(*)(SDL_Surface*)> a(pointer, SDL_FreeSurface);

// deleter is only required at construction time, not part of the type
std::shared_ptr<SDL_Surface> b(pointer, SDL_FreeSurface);

Com o template auto, podemos facilitar ainda mais o encapsulamento de nossos deletores personalizados:

template <auto DeleteFn>
struct FunctionDeleter {
    template <class T>
    void operator()(T* ptr) {
        DeleteFn(ptr);
    }
};

template <class T, auto DeleteFn>
using unique_ptr_deleter = std::unique_ptr<T, FunctionDeleter<DeleteFn>>;

Com o qual o exemplo acima é simplesmente:

unique_ptr_deleter<SDL_Surface, SDL_FreeSurface> c(pointer);

Aqui, o propósito de auto é lidar com todas as funções livres, se elas retornam void (por exemplo, SDL_FreeSurface) ou não (por exemplo, fclose).

Propriedade exclusiva sem semântica de movimento (auto_ptr) #

NOTA: std::auto_ptr foi descontinuado em C++11 e será removido em C++17. Você só deve usar isso se for forçado a usar C++03 ou anterior e estiver disposto a ser cuidadoso. É recomendado mover para unique_ptr em combinação com std::move para substituir o comportamento std::auto_ptr.

Antes de termos std::unique_ptr, antes de termos semântica de movimentação, tínhamos std::auto_ptr. std::auto_ptr fornece propriedade exclusiva, mas transfere a propriedade após a cópia.

Assim como todos os ponteiros inteligentes, std::auto_ptr limpa automaticamente os recursos (consulte RAII):

{
    std::auto_ptr<int> p(new int(42));
    std::cout << *p;
} // p is deleted here, no memory leaked

mas permite apenas um proprietário:

std::auto_ptr<X> px = ...;
std::auto_ptr<X> py = px; 
  // px is now empty

Isso permite usar std::auto_ptr para manter a propriedade explícita e exclusiva sob o risco de perder a propriedade não intencional:

void f(std::auto_ptr<X> ) {
    // assumes ownership of X
    // deletes it at end of scope
};

std::auto_ptr<X> px = ...;
f(px); // f acquires ownership of underlying X
       // px is now empty
px->foo(); // NPE!
// px.~auto_ptr() does NOT delete

A transferência de propriedade aconteceu no construtor “cópia”. O construtor de cópia de auto_ptr e o operador de atribuição de cópia recebem seus operandos por referência não-const para que possam ser modificados. Um exemplo de implementação pode ser:

template <typename T>
class auto_ptr {
    T* ptr;
public:
    auto_ptr(auto_ptr& rhs)
    : ptr(rhs.release())
    { }

    auto_ptr& operator=(auto_ptr& rhs) {
        reset(rhs.release());
        return *this;
    }

    T* release() {
        T* tmp = ptr;
        ptr = nullptr;
        return tmp;
    }

    void reset(T* tmp = nullptr) {
        if (ptr != tmp) {
            delete ptr;
            ptr = tmp;
        }
    }

    /* other functions ... */
};

Isso quebra a semântica da cópia, que exige que a cópia de um objeto deixe você com duas versões equivalentes dele. Para qualquer tipo copiável, T, devo ser capaz de escrever:

T a = ...;
T b(a);
assert(b == a);

Mas para auto_ptr, este não é o caso. Como resultado, não é seguro colocar auto_ptrs em containers.

Lançando ponteiros std::shared_ptr #

Não é possível usar diretamente static_cast, const_cast, dynamic_cast e reinterpret_cast em std::shared_ptr para recuperar um ponteiro compartilhando propriedade com o ponteiro sendo passado como argumento. Em vez disso, as funções std::static_pointer_cast, std::const_pointer_cast, std::dynamic_pointer_cast e std::reinterpret_pointer_cast devem ser usadas:

struct Base { virtual ~Base() noexcept {}; };
struct Derived: Base {};
auto derivedPtr(std::make_shared<Derived>());
auto basePtr(std::static_pointer_cast<Base>(derivedPtr));
auto constBasePtr(std::const_pointer_cast<Base const>(basePtr));
auto constDerivedPtr(std::dynamic_pointer_cast<Derived const>(constBasePtr));

Observe que std::reinterpret_pointer_cast não está disponível em C++11 e C++14, pois foi proposto apenas por N3920 e adotado no Library Fundamentals TS em fevereiro de 2014. No entanto, pode ser implementado da seguinte forma:

template <typename To, typename From>
inline std::shared_ptr<To> reinterpret_pointer_cast(
    std::shared_ptr<From> const & ptr) noexcept
{ return std::shared_ptr<To>(ptr, reinterpret_cast<To *>(ptr.get())); }

Escrevendo um ponteiro inteligente: value_ptr #

Um value_ptr é um ponteiro inteligente que se comporta como um valor. Quando copiado, copia seu conteúdo. Quando criado, ele cria seu conteúdo.

// Like std::default_delete:
template<class T>
struct default_copier {
  // a copier must handle a null T const* in and return null:
  T* operator()(T const* tin)const {
    if (!tin) return nullptr;
    return new T(*tin);
  }
  void operator()(void* dest, T const* tin)const {
    if (!tin) return;
    return new(dest) T(*tin);
  }
};
// tag class to handle empty case:
struct empty_ptr_t {};
constexpr empty_ptr_t empty_ptr{};
// the value pointer type itself:
template<class T, class Copier=default_copier<T>, class Deleter=std::default_delete<T>,
  class Base=std::unique_ptr<T, Deleter>
>
struct value_ptr:Base, private Copier {
  using copier_type=Copier;
  // also typedefs from unique_ptr

  using Base::Base;

  value_ptr( T const& t ):
    Base( std::make_unique<T>(t) ),
    Copier()
  {}
  value_ptr( T && t ):
    Base( std::make_unique<T>(std::move(t)) ),
    Copier()
  {}
  // almost-never-empty:
      value_ptr():
    Base( std::make_unique<T>() ),
    Copier()
  {}
  value_ptr( empty_ptr_t ) {}

  value_ptr( Base b, Copier c={} ):
    Base(std::move(b)),
    Copier(std::move(c))
  {}

  Copier const& get_copier() const {
    return *this;
  }

  value_ptr clone() const {
    return {
      Base(
        get_copier()(this->get()),
        this->get_deleter()
      ),
      get_copier()
    };
  }
  value_ptr(value_ptr&&)=default;
  value_ptr& operator=(value_ptr&&)=default;

  value_ptr(value_ptr const& o):value_ptr(o.clone()) {}
  value_ptr& operator=(value_ptr const&o) {
    if (o && *this) {
      // if we are both non-null, assign contents:
      **this = *o;
    } else {
      // otherwise, assign a clone (which could itself be null):
      *this = o.clone();
    }
    return *this;
  }
  value_ptr& operator=( T const& t ) {
    if (*this) {
      **this = t;
    } else {
      *this = value_ptr(t);
    }
    return *this;
  }
  value_ptr& operator=( T && t ) {
    if (*this) {
      **this = std::move(t);
    } else {
      *this = value_ptr(std::move(t));
    }
    return *this;
  }
  T& get() { return **this; }
  T const& get() const { return **this; }
  T* get_pointer() {
    if (!*this) return nullptr;
    return std::addressof(get());
  }
  T const* get_pointer() const {
    if (!*this) return nullptr;
    return std::addressof(get());
  }
  // operator-> from unique_ptr
};
template<class T, class...Args>
value_ptr<T> make_value_ptr( Args&&... args ) {
  return {std::make_unique<T>(std::forward<Args>(args)...)};
}

Este value_ptr específico só está vazio se você o construir com empty_ptr_t ou se você sair dele. Ele expõe o fato de ser um unique_ptr, então explicit operator bool() const funciona nele. .get() foi alterado para retornar uma referência (já que quase nunca está vazio), e .get_pointer() retorna um ponteiro.

Este ponteiro inteligente pode ser útil para casos pImpl, onde queremos semântica de valor, mas também não queremos expor o conteúdo do pImpl fora do arquivo de implementação.

Com um Copier não padrão, ele pode até mesmo manipular classes base virtuais que sabem como produzir instâncias de seus derivados e transformá-los em tipos de valor.

Obtendo um shared_ptr referente a isso #

enable_shared_from_this permite que você obtenha uma instância válida de shared_ptr para this.

Ao derivar sua classe do modelo de classe enable_shared_from_this, você herda um método shared_from_this que retorna uma instância shared_ptr para this.

Observe que o objeto deve ser criado como um shared_ptr em primeiro lugar:

#include <memory>
class A: public enable_shared_from_this<A> {
};
A* ap1 =new A();
shared_ptr<A> ap2(ap1); // First prepare a shared pointer to the object and hold it!
// Then get a shared pointer to the object from the object itself
shared_ptr<A> ap3 = ap1->shared_from_this(); 
int c3 =ap3.use_count(); // =2: pointing to the same object

Observação(2) você não pode chamar enable_shared_from_this dentro do construtor.

#include <memory> // enable_shared_from_this

class Widget : public std::enable_shared_from_this< Widget >
{
public:
    void DoSomething()
    {
        std::shared_ptr< Widget > self = shared_from_this();
        someEvent -> Register( self );
    }
private:
    ...
};

int main()
{
    ...
    auto w = std::make_shared< Widget >();
    w -> DoSomething();
    ...
}

Se você usar shared_from_this() em um objeto que não pertence a um shared_ptr, como um objeto automático local ou um objeto global, então o comportamento é indefinido. Desde o C++17 ele lança std::bad_alloc em vez disso.

Usar shared_from_this() de um construtor é equivalente a usá-lo em um objeto que não pertence a um shared_ptr, porque os objetos são possuídos pelo shared_ptr após o retorno do construtor.
