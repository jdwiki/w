Tamanho dos tipos integrais #

Os seguintes tipos são definidos como tipos integrais:

  • char
  • Tipos inteiros assinados
  • Tipos inteiros não assinados
  • char16_t e char32_t
  • bool
  • wchar_t

Com exceção de sizeof(char) / sizeof(signed char) / sizeof(unsigned char), que é dividido entre § 3.9.1.1 [basic.fundamental/1] e § 5.3.3.1 [expr. sizeof], e sizeof(bool), que é totalmente definido pela implementação e não tem tamanho mínimo, os requisitos de tamanho mínimo desses tipos são fornecidos na seção § 3.9.1 [basic.fundamental] do padrão, e devem ser detalhado abaixo.

Tamanho do char #

Todas as versões do padrão C++ especificam, no § 5.3.3.1, que sizeof produz 1 para unsigned char, signed char e char (é definido pela implementação se o tipo char for assinado ou não assinado).

char é grande o suficiente para representar 256 valores diferentes, para ser adequado para armazenar unidades de código UTF-8.

Tamanho dos tipos inteiros assinados e não assinados #

O padrão especifica, no § 3.9.1.2, que na lista de tipos de inteiros com sinal padrão, consistindo de caractere assinado, int curto, int, int longo e int longo longo, cada tipo fornecerá pelo menos tanto armazenamento quanto os anteriores na lista. Além disso, conforme especificado no § 3.9.1.3, cada um desses tipos tem um tipo inteiro unsigned padrão correspondente, unsigned char, unsigned short int, unsigned int, unsigned long int e unsigned long long int , que tem o mesmo tamanho e alinhamento que seu tipo com sinal correspondente. Além disso, conforme especificado no § 3.9.1.1, chartem os mesmos requisitos de tamanho e alinhamento quesigned chareunsigned char`.

Antes do C++11, long long e unsigned long long não eram oficialmente parte do padrão C++. No entanto, após sua introdução ao C, em C99, muitos compiladores suportavam long long como um tipo inteiro assinado estendido, e unsigned long long como um tipo inteiro não assinado estendido, com as mesmas regras que os tipos C.

A norma garante assim que:

1 == sizeof(char)  == sizeof(signed char) == sizeof(unsigned char)
  <= sizeof(short) == sizeof(unsigned short)
  <= sizeof(int)   == sizeof(unsigned int)
  <= sizeof(long)  == sizeof(unsigned long)

 <= sizeof(long long) == sizeof(unsigned long long)

Tamanhos mínimos específicos para cada tipo não são fornecidos pelo padrão. Em vez disso, cada tipo tem um intervalo mínimo de valores que pode suportar, que é, conforme especificado em § 3.9.1.3, herdado do padrão C, em §5.2.4.2.1. O tamanho mínimo de cada tipo pode ser inferido aproximadamente a partir desse intervalo, determinando o número mínimo de bits necessários; observe que, para qualquer plataforma, o intervalo real com suporte de qualquer tipo pode ser maior que o mínimo. Observe que para tipos com sinal, os intervalos correspondem ao complemento de um, não ao complemento de dois mais comumente usado; isso é para permitir que uma gama mais ampla de plataformas esteja em conformidade com o padrão.

|Tipo |Intervalo mínimo |Bits mínimos necessários| |:——————-|:————————— ——————————————–:|—- —————-:| |signed char |-127 a 127 (-(27 - 1) a (27 - 1)) |8 | |unsigned char |0 a 255 (0 a 28 - 1) |8 | |assinado curto |-32.767 a 32.767 (-(215 - 1) a (215 - 1)) |16 | |unsigned short |0 a 65.535 (0 a 216 - 1) |16 | |signed int |-32.767 a 32.767 (-(215 - 1) a (215 - 1)) |16 | |unsigned int |0 a 65.535 (0 a 216 - 1) |16 | |assinado longo |-2.147.483.647 a 2.147.483.647 (-(231 - 1) a (231 - 1))|32 | |unsigned long |0 a 4.294.967.295 (0 a 232 - 1) |32 |

|Tipo |Intervalo mínimo |Bits mínimos necessários| |:——————-|:————————— ————————–:|——————–:| |assinado longo longo |-9.223.372.036.854.775.807 a 9.223.372.036.854.775.807 (-(263 - 1) a (263 - 1))|64 | |unsigned long long|0 a 18.446.744.073.709.551.615 (0 a 264 - 1) |64 |

Como cada tipo pode ser maior que seu requisito de tamanho mínimo, os tipos podem diferir em tamanho entre as implementações. O exemplo mais notável disso é com os modelos de dados de 64 bits LP64 e LLP64, onde sistemas LLP64 (como Windows de 64 bits) têm ints e longs de 32 bits, e sistemas LP64 (como 64-bits) bit Linux) tem ints de 32 bits e longs de 64 bits. Devido a isso, os tipos inteiros não podem ter uma largura fixa em todas as plataformas.

Se tipos inteiros com largura fixa forem necessários, use tipos do cabeçalho <cstdint>, mas observe que o padrão torna opcional para implementações suportar os tipos de largura exata int8_t, int16_t, int32_t, int64_t, intptr_t, uint8_t, uint16_t, uint32_t, uint64_t e uintptr_t.

Tamanho de char16_t e char32_t #

Os tamanhos de char16_t e char32_t são definidos pela implementação, conforme especificado em § 5.3.3.1, com as estipulações fornecidas em § 3.9.1.5:

  • char16_t é grande o suficiente para representar qualquer unidade de código UTF-16 e tem o mesmo tamanho, assinatura e alinhamento que uint_least16_t; é, portanto, necessário ter pelo menos 16 bits de tamanho.

  • char32_t é grande o suficiente para representar qualquer unidade de código UTF-32 e tem o mesmo tamanho, assinatura e alinhamento que uint_least32_t; é, portanto, necessário ter pelo menos 32 bits de tamanho.

Tamanho do bool #

O tamanho de bool é definido pela implementação e pode ou não ser 1.

Tamanho de wchar_t #

wchar_t, conforme especificado no § 3.9.1.5, é um tipo distinto, cujo intervalo de valores pode representar cada unidade de código distinta do maior conjunto de caracteres estendidos entre as localidades suportadas. Ele tem o mesmo tamanho, sinal e alinhamento que um dos outros tipos integrais, que é conhecido como tipo subjacente. O tamanho deste tipo é definido pela implementação, conforme especificado no § 5.3.3.1, e pode ser, por exemplo, pelo menos 8, 16 ou 32 bits; se um sistema suporta Unicode, por exemplo, wchar_t deve ter pelo menos 32 bits (uma exceção a esta regra é o Windows, onde wchar_t é 16 bits para fins de compatibilidade). Ele é herdado do padrão C90, ISO 9899:1990 § 4.1.5, com apenas uma pequena reformulação.

Dependendo da implementação, o tamanho de wchar_t é frequentemente, mas nem sempre, 8, 16 ou 32 bits. Os exemplos mais comuns são:

  • Em sistemas Unix e semelhantes a Unix, wchar_t é de 32 bits e geralmente é usado para UTF-32.
  • No Windows, wchar_t é de 16 bits e é usado para UTF-16.
  • Em um sistema que só tem suporte de 8 bits, wchar_t é de 8 bits.

Se o suporte a Unicode for desejado, é recomendado usar char para UTF-8, char16_t para UTF-16 ou char32_t para UTF-32, em vez de usar wchar_t.

Modelos de dados #

Como mencionado acima, as larguras dos tipos inteiros podem diferir entre as plataformas. Os modelos mais comuns são os seguintes, com tamanhos especificados em bits:

Modelo int long ponteiro
LP32 (2/4/4) 16 32 32
ILP32 (4/4/4) 32 32 32
LLP64 (4/4/8) 32 32 64
LP64 (4/8/8) 32 64 64

Destes modelos:

  • O Windows de 16 bits usava LP32.
  • Sistemas *nix de 32 bits (Unix, Linux, Mac OSX e outros sistemas operacionais semelhantes a Unix) e Windows usam ILP32.
  • O Windows de 64 bits usa LLP64.
  • Os sistemas *nix de 64 bits usam LP64.

Observe, no entanto, que esses modelos não são especificamente mencionados no próprio padrão.

Char pode estar sem assinatura ou assinado #

O padrão não especifica se char deve ser assinado ou não. Diferentes compiladores o implementam de maneira diferente ou podem permitir alterá-lo usando uma opção de linha de comando.

Número de bits em um byte #

Em C++, um byte é o espaço ocupado por um objeto char. O número de bits em um byte é dado por CHAR_BIT, que é definido em climits e deve ser pelo menos 8. Enquanto a maioria dos sistemas modernos tem bytes de 8 bits, e POSIX requer que CHAR_BIT seja exatamente 8, existem alguns sistemas onde CHAR_BIT é maior que 8, ou seja, um único byte pode ser composto por 8, 16, 32 ou 64 bits.

Valor numérico de um ponteiro #

O resultado da conversão de um ponteiro para um inteiro usando reinterpret_cast é definido pela implementação, mas “… não é uma surpresa para aqueles que conhecem a estrutura de endereçamento da máquina subjacente.”

int x = 42;
int* p = &x;
long addr = reinterpret_cast<long>(p);
std::cout << addr << "\n"; // prints some numeric address,
                           // probably in the architecture's native address format

Da mesma forma, o ponteiro obtido pela conversão de um inteiro também é definido pela implementação.

A maneira correta de armazenar um ponteiro como um inteiro é usando os tipos uintptr_t ou intptr_t:

// `uintptr_t` was not in C++03. It's in C99, in <stdint.h>, as an optional type
#include <stdint.h>

uintptr_t uip;

// There is an optional `std::uintptr_t` in C++11
#include <cstdint>

std::uintptr_t uip;

C++11 refere-se a C99 para a definição uintptr_t (padrão C99, 6.3.2.3):

um tipo inteiro sem sinal com a propriedade de que qualquer ponteiro válido para void pode ser convertido para este tipo, então convertido novamente para ponteiro para void, e o resultado será comparado igual ao ponteiro original.

Enquanto, para a maioria das plataformas modernas, você pode assumir um espaço de endereço plano e que a aritmética em uintptr_t é equivalente a aritmética em char *, é inteiramente possível para uma implementação realizar qualquer transformação ao converter void * para uintptr_t desde que a transformação possa ser revertida ao converter de uintptr_t para void *.

Técnicas

  • Em sistemas compatíveis com XSI (X/Open System Interfaces), os tipos intptr_t e uintptr_t são necessários, caso contrário, eles são opcionais.
  • No sentido do padrão C, funções não são objetos; não é garantido pelo padrão C que uintptr_t possa conter um ponteiro de função. De qualquer forma, a conformidade POSIX (2.12.3) exige que:

Todos os tipos de ponteiro de função devem ter a mesma representação que o ponteiro de tipo a ser void. A conversão de um ponteiro de função para void * não deve alterar a representação. Um valor void * resultante de tal conversão pode ser convertido novamente para o tipo de ponteiro de função original, usando uma conversão explícita, sem perda de informações.

  • C99 §7.18.1:

Quando os nomes typedef diferem apenas na ausência ou presença da inicial u são definidos, eles devem denotar os tipos assinados e não assinados correspondentes, conforme descrito em 6.2.5; uma implementação que fornece um desses tipos correspondentes também deve fornecer o outro.

uintptr_t pode fazer sentido se você quiser fazer coisas com os bits do ponteiro que você não pode fazer tão sensatamente com um inteiro assinado.

Intervalos de tipos numéricos #

Os intervalos dos tipos inteiros são definidos pela implementação. O cabeçalho <limits> fornece o modelo std::numeric_limits<T> que fornece os valores mínimo e máximo de todos os tipos fundamentais. Os valores satisfazem as garantias fornecidas pelo padrão C por meio dos cabeçalhos <climits> e (>= C++11) <cinttypes>.

  • std::numeric_limits<signed char>::min() é igual a SCHAR_MIN, que é menor ou igual a -127.
  • std::numeric_limits<signed char>::max() é igual a SCHAR_MAX, que é maior ou igual a 127.
  • std::numeric_limits<unsigned char>::max() é igual a UCHAR_MAX, que é maior ou igual a 255.
  • std::numeric_limits<short>::min() é igual a SHRT_MIN, que é menor ou igual a -32767.
  • std::numeric_limits<short>::max() é igual a SHRT_MAX, que é maior ou igual a 32767.
  • std::numeric_limits<unsigned short>::max() é igual a USHRT_MAX, que é maior ou igual a 65535.
  • std::numeric_limits<int>::min() é igual a INT_MIN, que é menor ou igual a -32767.
  • std::numeric_limits<int>::max() é igual a INT_MAX, que é maior ou igual a 32767.
  • std::numeric_limits<unsigned int>::max() é igual a UINT_MAX, que é maior ou igual a 65535.
  • std::numeric_limits<long>::min() é igual a LONG_MIN, que é menor ou igual a -2147483647.
  • std::numeric_limits<long>::max() é igual a LONG_MAX, que é maior ou igual a 2147483647.
  • std::numeric_limits<unsigned long>::max() é igual a ULONG_MAX, que é maior ou igual a 4294967295.
  • std::numeric_limits<long long>::min() é igual a LLONG_MIN, que é menor ou igual a -9223372036854775807.
  • std::numeric_limits<long long>::max() é igual a LLONG_MAX, que é maior ou igual a 9223372036854775807.
  • std::numeric_limits<unsigned long long>::max() é igual a ULLONG_MAX, que é maior ou igual a 18446744073709551615.

Para tipos de ponto flutuante T, max() é o valor finito máximo enquanto min() é o valor normalizado positivo mínimo. Membros adicionais são fornecidos para tipos de ponto flutuante, que também são definidos pela implementação, mas satisfazem certas garantias fornecidas pelo padrão C por meio do cabeçalho <cfloat>.

  • O membro digits10 fornece o número de dígitos decimais de precisão.
    • std::numeric_limits<float>::digits10 equals FLT_DIG, which is at least 6.
    • std::numeric_limits<double>::digits10 equals DBL_DIG, which is at least 10.
    • std::numeric_limits<long double>::digits10 equals LDBL_DIG, which is at least 10.
  • O membro min_exponent10 é o mínimo negativo E tal que 10 elevado a E é normal.
    • std::numeric_limits<float>::min_exponent10 equals FLT_MIN_10_EXP, which is at most -37.
    • std::numeric_limits<double>::min_exponent10 equals DBL_MIN_10_EXP, which is at most -37. std::numeric_limits<long double>::min_exponent10 é igual a LDBL_MIN_10_EXP, que é no máximo -37.
  • O membro max_exponent10 é o máximo E tal que 10 elevado a E é finito.
    • std::numeric_limits<float>::max_exponent10 equals FLT_MIN_10_EXP, which is at least 37.
    • std::numeric_limits<double>::max_exponent10 equals DBL_MIN_10_EXP, which is at least 37.
    • std::numeric_limits<long double>::max_exponent10 equals LDBL_MIN_10_EXP, which is at least 37.
  • Se o membro is_iec559 for verdadeiro, o tipo está em conformidade com a IEC 559 / IEEE 754 e, portanto, seu intervalo é determinado por esse padrão.

Representação de valor de tipos de ponto flutuante #

O padrão requer que long double forneça pelo menos tanta precisão quanto double, que fornece pelo menos tanta precisão quanto float; e que um long double pode representar qualquer valor que um double possa representar, enquanto um double pode representar qualquer valor que um float possa representar. Os detalhes da representação são, no entanto, definidos pela implementação.

Para um tipo de ponto flutuante T, std::numeric_limits<T>::radix especifica a base usada pela representação de T.

Se std::numeric_limits<T>::is_iec559 for true, então a representação de T corresponde a um dos formatos definidos pela IEC 559 / IEEE 754.

Estouro ao converter de inteiro para inteiro com sinal #

Quando um inteiro com ou sem sinal é convertido em um tipo de inteiro com sinal e seu valor não é representável no tipo de destino, o valor produzido é definido pela implementação. Exemplo:

// Suppose that on this implementation, the range of signed char is -128 to +127 and
// the range of unsigned char is 0 to 255
int x = 12345;
signed char sc = x;   // sc has an implementation-defined value
unsigned char uc = x; // uc is initialized to 57 (i.e., 12345 modulo 256)

Tipo subjacente (e, portanto, tamanho) de um enum #

Se o tipo subjacente não for especificado explicitamente para um tipo de enumeração sem escopo, ele será determinado de maneira definida pela implementação.

enum E {
    RED,
    GREEN,
    BLUE,
};
using T = std::underlying_type<E>::type; // implementation-defined

No entanto, o padrão exige que o tipo subjacente de uma enumeração não seja maior que int, a menos que int e unsigned int não possam representar todos os valores da enumeração. Portanto, no código acima, T poderia ser int, unsigned int ou short, mas não long long, para dar alguns exemplos.

Observe que um enum tem o mesmo tamanho (como retornado por sizeof) que seu tipo subjacente.
