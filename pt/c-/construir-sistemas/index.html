C++, como C, tem uma longa e variada história em relação a fluxos de trabalho de compilação e processos de compilação. Hoje, C++ tem vários sistemas de compilação populares que são usados ​​para compilar programas, às vezes para várias plataformas dentro de um sistema de compilação. Aqui, alguns sistemas de compilação serão revisados ​​e analisados.

Atualmente, não existe um sistema de compilação universal ou dominante para C++ que seja popular e multiplataforma. No entanto, existem vários sistemas de compilação principais que estão anexados às principais plataformas/projetos, sendo o mais notável o GNU Make com o sistema operacional GNU/Linux e o NMAKE com o sistema de projeto Visual C++/Visual Studio.

Além disso, alguns ambientes de desenvolvimento integrados (IDEs) também incluem sistemas de compilação especializados para serem usados ​​especificamente com o IDE nativo. Certos geradores de sistema de compilação podem gerar esses formatos de projeto/sistema de compilação IDE nativos, como CMake for Eclipse e Microsoft Visual Studio 2012.

Gerando ambiente de compilação com CMake #

CMake gera ambientes de compilação para praticamente qualquer compilador ou IDE a partir de uma única definição de projeto. Os exemplos a seguir demonstrarão como adicionar um arquivo CMake ao código C++ “Hello World” de plataforma cruzada.

Os arquivos CMake são sempre denominados “CMakeLists.txt” e já devem existir no diretório raiz de cada projeto (e possivelmente em subdiretórios também.) Um arquivo CMakeLists.txt básico se parece com:

cmake_minimum_required(VERSION 2.4)

project(HelloWorld)

add_executable(HelloWorld main.cpp)

Veja ao vivo no Coliru.

Este arquivo informa ao CMake o nome do projeto, qual versão do arquivo esperar e instruções para gerar um executável chamado “HelloWorld” que requer main.cpp.

Gere um ambiente de compilação para seu compilador/IDE instalado a partir da linha de comando:

> cmake .

Crie o aplicativo com:

> cmake --build .

Isso gera o ambiente de compilação padrão para o sistema, dependendo do sistema operacional e das ferramentas instaladas. Mantenha o código-fonte limpo de qualquer artefato de compilação com o uso de compilações “fora da fonte”:

> mkdir build
> cd build
> cmake ..
> cmake --build .

O CMake também pode abstrair os comandos básicos do shell da plataforma do exemplo anterior:

> cmake -E make_directory build
> cmake -E chdir build cmake .. 
> cmake --build build

O CMake inclui geradores para várias ferramentas de compilação e IDEs comuns. Para gerar makefiles para o nmake do Visual Studio]4:

> cmake -G "NMake Makefiles" ..
> nmake

Compilando com GNU make #

Introdução #

O GNU Make (denominado make) é um programa dedicado à automação da execução de comandos do shell. GNU Make é um programa específico que se enquadra na família Make. Make permanece popular entre os sistemas operacionais do tipo Unix e do tipo POSIX, incluindo aqueles derivados do kernel Linux, Mac OS X e BSD.

O GNU Make é especialmente notável por ser anexado ao Projeto GNU, que é anexado ao popular sistema operacional GNU/Linux. GNU Make também tem versões compatíveis rodando em vários tipos de Windows e Mac OS X. Também é uma versão muito estável com significado histórico que permanece popular. É por essas razões que o GNU Make é frequentemente ensinado ao lado de C e C++.

Regras básicas #

Para compilar com make, crie um Makefile no diretório do seu projeto. Seu Makefile pode ser tão simples quanto:

Makefile

# Set some variables to use in our command
# First, we set the compiler to be g++
CXX=g++

# Then, we say that we want to compile with g++'s recommended warnings and some extra ones.
CXXFLAGS=-Wall -Wextra -pedantic

# This will be the output file
EXE=app

SRCS=main.cpp

# When you call `make` at the command line, this "target" is called.
# The $(EXE) at the right says that the `all` target depends on the `$(EXE)` target.
# $(EXE) expands to be the content of the EXE variable
# Note: Because this is the first target, it becomes the default target if `make` is called without target
all: $(EXE)

# This is equivalent to saying
# app: $(SRCS)
# $(SRCS) can be separated, which means that this target would depend on each file.
# Note that this target has a "method body": the part indented by a tab (not four spaces).
# When we build this target, make will execute the command, which is:
# g++ -Wall -Wextra -pedantic -o app main.cpp
# I.E. Compile main.cpp with warnings, and output to the file ./app
$(EXE): $(SRCS)
    @$(CXX) $(CXXFLAGS) -o $@ $(SRCS)

# This target should reverse the `all` target. If you call
# make with an argument, like `make clean`, the corresponding target
# gets called.
clean:
    @rm -f $(EXE)

**OBSERVAÇÃO: Certifique-se de que os recuos estejam com uma tabulação, não com quatro espaços. Caso contrário, você receberá um erro de ** Makefile:10: *** separador ausente. Pare.

Para executar isso na linha de comando, faça o seguinte:

$ cd ~/Path/to/project
$ make
$ ls
app  main.cpp  Makefile

$ ./app
Hello World!

$ make clean
$ ls
main.cpp  Makefile

Builds incrementais #

Quando você começa a ter mais arquivos, o make se torna mais útil. E se você editou a.cpp, mas não b.cpp? A recompilação do b.cpp levaria mais tempo.

Com a seguinte estrutura de diretórios:

.
+-- src
|   +-- a.cpp
|   +-- a.hpp
|   +-- b.cpp
|   +-- b.hpp
+-- Makefile

Este seria um bom Makefile:

Makefile

CXX=g++
CXXFLAGS=-Wall -Wextra -pedantic
EXE=app

SRCS_GLOB=src/*.cpp
SRCS=$(wildcard $(SRCS_GLOB))
OBJS=$(SRCS:.cpp=.o)

all: $(EXE)

$(EXE): $(OBJS)
    @$(CXX) -o $@ $(OBJS)

depend: .depend

.depend: $(SRCS)
    @-rm -f ./.depend
    @$(CXX) $(CXXFLAGS) -MM $^>>./.depend

clean:
    -rm -f $(EXE)
    -rm $(OBJS)
    -rm *~
    -rm .depend

include .depend

Novamente observe as guias. Este novo Makefile garante que você recompile apenas os arquivos alterados, minimizando o tempo de compilação.

Documentação #

Para saber mais sobre make, veja a documentação oficial da Free Software Foundation, [a documentação do stackoverflow](https://www.wikiod.com/pt/docs/ /makefile) e resposta elaborada do dmckee no stackoverflow.

Construindo com SCons #

Você pode construir o código C++ “Hello World” multiplataforma, usando Scons - Uma ferramenta de construção de software em linguagem Python.

Primeiro, crie um arquivo chamado SConstruct (observe que os SCons procurarão um arquivo com esse nome exato por padrão). Por enquanto, o arquivo deve estar em um diretório ao lado do seu hello.cpp. Escreva no novo arquivo a linha

 Program('hello.cpp')

Agora, no terminal, execute scons. Você deve ver algo como

$ scons
scons: Reading SConscript files ...
scons: done reading SConscript files.
scons: Building targets ...
g++ -o hello.o -c hello.cpp
g++ -o hello hello.o
scons: done building targets.

(embora os detalhes variem dependendo do seu sistema operacional e do compilador instalado).

As classes Environment e Glob ajudarão você a configurar ainda mais o que construir. Por exemplo, o arquivo SConstruct

env=Environment(CPPPATH='/usr/include/boost/',
    CPPDEFINES=[],
    LIBS=[],
    SCONS_CXX_STANDARD="c++11"
    )

env.Program('hello', Glob('src/*.cpp'))

constrói o executável hello, usando todos os arquivos cpp em src. Seu CPPPATH é /usr/include/boost e especifica o padrão C++11.

Ferramentas automáticas (GNU) #

Introdução #

As Autotools são um grupo de programas que criam um GNU Build System para um determinado pacote de software. É um conjunto de ferramentas que trabalham juntas para produzir vários recursos de compilação, como um Makefile (para ser usado com o GNU Make). Assim, o Autotools pode ser considerado um gerador de sistema de construção de fato.

Alguns programas notáveis ​​do Autotools incluem:

  • Autoconf
  • Automake (não confundir com make)

Em geral, o Autotools destina-se a gerar o script compatível com Unix e o Makefile para permitir que o seguinte comando compile (assim como instale) a maioria dos pacotes (no caso simples):

./configure && make && make install

Como tal, o Autotools também tem um relacionamento com certos gerenciadores de pacotes, especialmente aqueles que estão conectados a sistemas operacionais que estão em conformidade com o(s) padrão(ões) POSIX.

##Ninja

Introdução #

O sistema de compilação do Ninja é descrito pelo site do projeto como “um pequeno sistema de compilação com foco na velocidade.” O Ninja foi projetado para ter seus arquivos gerados por geradores de arquivos do sistema de compilação e adota uma abordagem de baixo nível para sistemas de compilação, em contraste com gerenciadores de sistemas de compilação de nível superior, como CMake ou Meson.

O Ninja é escrito principalmente em C++ e Python e foi criado como uma alternativa ao sistema de compilação SCons para o projeto Chromium.

NMAKE (Utilitário de manutenção de programas da Microsoft) #

Introdução #

NMAKE é um utilitário de linha de comando desenvolvido pela Microsoft para ser usado principalmente em conjunto com o Microsoft Visual Studio e/ou as ferramentas de linha de comando Visual C++.

NMAKE é um sistema de compilação que se enquadra na família Make de sistemas de compilação, mas possui certos recursos distintos que divergem dos programas Make do tipo Unix, como o suporte à sintaxe de caminho de arquivo específica do Windows (que por si só difere dos caminhos de arquivo no estilo Unix).
