O pré-processador C é um analisador/substituto de texto simples que é executado antes da compilação real do código. Usado para estender e facilitar o uso da linguagem C (e posterior C++), pode ser usado para:

uma. Incluindo outros arquivos usando #include

b. Defina uma macro de substituição de texto usando #define

c. Compilação Condicional usando #if #ifdef

d. Lógica específica da plataforma/compilador (como uma extensão da compilação condicional)

As instruções do pré-processador são executadas antes que seus arquivos de origem sejam entregues ao compilador. Eles são capazes de lógica condicional de nível muito baixo. Como as construções do pré-processador (por exemplo, macros do tipo objeto) não são digitadas como funções normais (a etapa de pré-processamento ocorre antes da compilação), o compilador não pode impor verificações de tipo, portanto, elas devem ser usadas com cuidado.

Incluir Guardas #

Um arquivo de cabeçalho pode ser incluído por outros arquivos de cabeçalho. Um arquivo de origem (unidade de compilação) que inclui vários cabeçalhos pode, portanto, indiretamente, incluir alguns cabeçalhos mais de uma vez. Se um arquivo de cabeçalho incluído mais de uma vez contiver definições, o compilador (após o pré-processamento) detectará uma violação da Regra de uma definição (por exemplo, §3.2 do padrão C++ de 2003) e, portanto, emitirá um diagnóstico e a compilação falhará.

A inclusão múltipla é evitada usando “guardas de inclusão”, que às vezes também são conhecidas como guardas de cabeçalho ou guardas de macro. Estes são implementados usando as diretivas #define, #ifndef, #endif do pré-processador.

por exemplo.

// Foo.h
#ifndef FOO_H_INCLUDED 
#define FOO_H_INCLUDED

class Foo    //  a class definition
{
};

#endif

A principal vantagem de usar protetores de inclusão é que eles funcionarão com todos os compiladores e pré-processadores compatíveis com o padrão.

No entanto, incluir guardas também causa alguns problemas para os desenvolvedores, pois é necessário garantir que as macros sejam exclusivas em todos os cabeçalhos usados ​​em um projeto. Especificamente, se dois (ou mais) cabeçalhos usarem FOO_H_INCLUDED como sua proteção de inclusão, o primeiro desses cabeçalhos incluído em uma unidade de compilação impedirá efetivamente que os outros sejam incluídos. Desafios específicos são introduzidos se um projeto usa várias bibliotecas de terceiros com arquivos de cabeçalho que usam protetores de inclusão em comum.

Também é necessário garantir que as macros usadas nas proteções de inclusão não entrem em conflito com outras macros definidas nos arquivos de cabeçalho.

A maioria das implementações de C++ também suporta a diretiva #pragma once que garante que o arquivo seja incluído apenas uma vez em uma única compilação. Esta é uma diretiva de fato padrão, mas não faz parte de nenhum padrão ISO C++. Por exemplo:

// Foo.h
#pragma once

class Foo
{
};

Enquanto #pragma once evita alguns problemas associados com include guards, um #pragma - por definição nos padrões - é inerentemente um gancho específico do compilador, e será silenciosamente ignorado por compiladores que não o suportam. Projetos que usam #pragma once são mais difíceis de portar para compiladores que não o suportam.

Várias diretrizes de codificação e padrões de garantia para C++ desencorajam especificamente qualquer uso do pré-processador que não seja para arquivos de cabeçalho #include ou para fins de colocação de guardas de inclusão em cabeçalhos.

Lógica condicional e manipulação multiplataforma #

Em poucas palavras, a lógica de pré-processamento condicional trata de tornar a lógica do código disponível ou indisponível para compilação usando definições de macro.

Três casos de uso proeminentes são:

  • diferentes perfis de aplicativo (por exemplo, depuração, lançamento, teste, otimização) que podem ser candidatos do mesmo aplicativo (por exemplo, com registro extra).
  • Compilação multiplataforma - base de código única, várias plataformas de compilação.
  • utilizando uma base de código comum para várias versões de aplicativos (por exemplo, versões Basic, Premium e Pro de um software) - com recursos ligeiramente diferentes.

Exemplo a: uma abordagem multiplataforma para remover arquivos (ilustrativo):

#ifdef _WIN32
#include <windows.h> // and other windows system files
#endif
#include <cstdio>

bool remove_file(const std::string &path) 
{
#ifdef _WIN32
  return DeleteFile(path.c_str());
#elif defined(_POSIX_VERSION) || defined(__unix__)
  return (0 == remove(path.c_str()));
#elif defined(__APPLE__)
  //TODO: check if NSAPI has a more specific function with permission dialog
  return (0 == remove(path.c_str()));
#else 
#error "This platform is not supported"
#endif
}

Macros como _WIN32, __APPLE__ ou __unix__ são normalmente predefinidas por implementações correspondentes.

Exemplo b: Como ativar o registro adicional para uma compilação de depuração:

void s_PrintAppStateOnUserPrompt()
{
    std::cout << "--------BEGIN-DUMP---------------\n"
              << AppState::Instance()->Settings().ToString() << "\n"
#if ( 1 == TESTING_MODE ) //privacy: we want user details only when testing
              << ListToString(AppState::UndoStack()->GetActionNames())
              << AppState::Instance()->CrntDocument().Name() 
              << AppState::Instance()->CrntDocument().SignatureSHA() << "\n"
#endif
              << "--------END-DUMP---------------\n"
}

Exemplo c: ativar um recurso premium em uma versão de produto separada (observação: isso é ilustrativo. Geralmente, é melhor permitir que um recurso seja desbloqueado sem a necessidade de reinstalar um aplicativo)

void MainWindow::OnProcessButtonClick()
{
#ifndef _PREMIUM
    CreatePurchaseDialog("Buy App Premium", "This feature is available for our App Premium users. Click the Buy button to purchase the Premium version at our website");
    return;
#endif
    //...actual feature logic here
}

Alguns truques comuns:

Definindo símbolos no momento da invocação:

O pré-processador pode ser chamado com símbolos predefinidos (com inicialização opcional). Por exemplo, este comando (gcc -E executa apenas o pré-processador)

gcc -E -DOPTIMISE_FOR_OS_X -DTESTING_MODE=1 Sample.cpp

processa Sample.cpp da mesma forma que faria se #define OPTIMISE_FOR_OS_X e #define TESTING_MODE 1 fossem adicionados ao topo de Sample.cpp.

Garantindo que uma macro seja definida:

Se uma macro não for definida e seu valor for comparado ou verificado, o pré-processador quase sempre assume silenciosamente o valor como ‘0’. Existem algumas maneiras de trabalhar com isso. Uma abordagem é assumir que as configurações padrão são representadas como 0 e quaisquer alterações (por exemplo, no perfil de compilação do aplicativo) precisam ser feitas explicitamente (por exemplo, ENABLE_EXTRA_DEBUGGING=0 por padrão, defina -DENABLE_EXTRA_DEBUGGING=1 para substituir). Outra abordagem é tornar todas as definições e padrões explícitos. Isso pode ser feito usando uma combinação das diretivas #ifndef e #error:

#ifndef (ENABLE_EXTRA_DEBUGGING)
// please include DefaultDefines.h if not already included.
#    error "ENABLE_EXTRA_DEBUGGING is not defined"
#else
#    if ( 1 == ENABLE_EXTRA_DEBUGGING )
  //code
#    endif
#endif

Macros #

As macros são categorizadas em dois grupos principais: macros semelhantes a objetos e macros semelhantes a funções. As macros são tratadas como uma substituição de token no início do processo de compilação. Isso significa que grandes (ou repetidas) seções de código podem ser abstraídas em uma macro de pré-processador.

// This is an object-like macro
#define    PI         3.14159265358979

// This is a function-like macro.
// Note that we can use previously defined macros
// in other macro definitions (object-like or function-like)
// But watch out, its quite useful if you know what you're doing, but the
// Compiler doesnt know which type to handle, so using inline functions instead
// is quite recommended (But e.g. for Minimum/Maximum functions it is quite useful)
#define    AREA(r)    (PI*(r)*(r))

// They can be used like this:
double pi_macro   = PI;
double area_macro = AREA(4.6);

A biblioteca Qt faz uso desta técnica para criar um sistema de meta-objetos fazendo com que o usuário declare a macro Q_OBJECT no início da classe definida pelo usuário estendendo QObject.

Os nomes das macros geralmente são escritos em maiúsculas, para torná-los mais fáceis de diferenciar do código normal. Isso não é um requisito, mas é meramente considerado um bom estilo por muitos programadores.

Quando uma macro semelhante a um objeto é encontrada, ela é expandida como uma simples operação de copiar e colar, com o nome da macro sendo substituído por sua definição. Quando uma macro do tipo função é encontrada, seu nome e seus parâmetros são expandidos.

double pi_squared = PI * PI;
// Compiler sees:
double pi_squared = 3.14159265358979 * 3.14159265358979;

double area = AREA(5);
// Compiler sees:
double area = (3.14159265358979*(5)*(5))

Devido a isso, os parâmetros de macro do tipo função geralmente são colocados entre parênteses, como em AREA() acima. Isso é para evitar erros que possam ocorrer durante a expansão da macro, especificamente erros causados ​​por um único parâmetro de macro sendo composto por vários valores reais.

#define BAD_AREA(r) PI * r * r

double bad_area = BAD_AREA(5 + 1.6);
// Compiler sees:
double bad_area = 3.14159265358979 * 5 + 1.6 * 5 + 1.6;

double good_area = AREA(5 + 1.6);
// Compiler sees:
double good_area = (3.14159265358979*(5 + 1.6)*(5 + 1.6));

Observe também que devido a essa simples expansão, deve-se ter cuidado com os parâmetros passados ​​para as macros, para evitar efeitos colaterais inesperados. Se o parâmetro for modificado durante a avaliação, ele será modificado toda vez que for utilizado na macro expandida, o que normalmente não é o que desejamos. Isso é verdade mesmo se a macro colocar os parâmetros entre parênteses para evitar que a expansão interrompa qualquer coisa.

int oops = 5;
double incremental_damage = AREA(oops++);
// Compiler sees:
double incremental_damage = (3.14159265358979*(oops++)*(oops++));

Além disso, as macros não fornecem segurança de tipo, levando a erros difíceis de entender sobre incompatibilidade de tipo.

Como os programadores normalmente terminam as linhas com um ponto-e-vírgula, as macros destinadas a serem usadas como linhas independentes geralmente são projetadas para “engolir” um ponto-e-vírgula; isso evita que erros não intencionais sejam causados ​​por um ponto-e-vírgula extra.

#define IF_BREAKER(Func) Func();

if (some_condition)
    // Oops.
    IF_BREAKER(some_func);
else
    std::cout << "I am accidentally an orphan." << std::endl;

Neste exemplo, o ponto e vírgula duplo inadvertidamente quebra o bloco if...else, evitando que o compilador combine o else com o if. Para evitar isso, o ponto e vírgula é omitido da definição da macro, o que fará com que ela “engula” o ponto e vírgula imediatamente após qualquer uso dele.

#define IF_FIXER(Func) Func()

if (some_condition)
    IF_FIXER(some_func);
else
    std::cout << "Hooray!  I work again!" << std::endl;

Deixar de lado o ponto e vírgula à direita também permite que a macro seja usada sem terminar a instrução atual, o que pode ser benéfico.

#define DO_SOMETHING(Func, Param) Func(Param, 2)

// ...

some_function(DO_SOMETHING(some_func, 3), DO_SOMETHING(some_func, 42));

Normalmente, uma definição de macro termina no final da linha. Se uma macro precisar cobrir várias linhas, no entanto, uma barra invertida pode ser usada no final de uma linha para indicar isso. Esta barra invertida deve ser o último caractere da linha, o que indica ao pré-processador que a linha seguinte deve ser concatenada na linha atual, tratando-as como uma única linha. Isso pode ser usado várias vezes seguidas.

#define TEXT "I \
am \
many \
lines."

// ...

std::cout << TEXT << std::endl; // Output:   I am many lines.

Isso é especialmente útil em macros semelhantes a funções complexas, que podem precisar cobrir várias linhas.

#define CREATE_OUTPUT_AND_DELETE(Str) \
    std::string* tmp = new std::string(Str); \
    std::cout << *tmp << std::endl; \
    delete tmp;

// ...

CREATE_OUTPUT_AND_DELETE("There's no real need for this to use 'new'.")

No caso de macros de funções mais complexas, pode ser útil dar a elas seu próprio escopo para evitar possíveis colisões de nomes ou fazer com que objetos sejam destruídos no final da macro, semelhante a uma função real. Um idioma comum para isso é do while 0, onde a macro é incluída em um bloco do-while. Este bloco é geralmente não seguido por um ponto e vírgula, permitindo que ele engula um ponto e vírgula.

#define DO_STUFF(Type, Param, ReturnVar) do { \
    Type temp(some_setup_values); \
    ReturnVar = temp.process(Param); \
} while (0)

int x;
DO_STUFF(MyClass, 41153.7, x);

// Compiler sees:

int x;
do {
    MyClass temp(some_setup_values);
    x = temp.process(41153.7);
} while (0);

Existem também macros variáveis; da mesma forma que as funções variádicas, elas recebem um número variável de argumentos e, em seguida, expandem todos eles no lugar de um parâmetro especial “Varargs”, __VA_ARGS__.

#define VARIADIC(Param, ...) Param(__VA_ARGS__)

VARIADIC(printf, "%d", 8);
// Compiler sees:
printf("%d", 8);

Observe que durante a expansão, __VA_ARGS__ pode ser colocado em qualquer lugar na definição e será expandido corretamente.

#define VARIADIC2(POne, PTwo, PThree, ...) POne(PThree, __VA_ARGS__, PTwo)

VARIADIC2(some_func, 3, 8, 6, 9);
// Compiler sees:
some_func(8, 6, 9, 3);

No caso de um parâmetro variádico de argumento zero, compiladores diferentes tratarão a vírgula à direita de maneira diferente. Alguns compiladores, como o Visual Studio, engolem silenciosamente a vírgula sem nenhuma sintaxe especial. Outros compiladores, como o GCC, exigem que você coloque ## imediatamente antes de __VA_ARGS__. Devido a isso, é aconselhável definir macros variádicas condicionalmente quando a portabilidade é uma preocupação.

// In this example, COMPILER is a user-defined macro specifying the compiler being used.

#if       COMPILER == "VS"
    #define VARIADIC3(Name, Param, ...) Name(Param, __VA_ARGS__)
#elif     COMPILER == "GCC"
    #define VARIADIC3(Name, Param, ...) Name(Param, ##__VA_ARGS__)
#endif /* COMPILER */

X-macros #

Uma técnica idiomática para gerar estruturas de código repetidas em tempo de compilação.

Uma macro X consiste em duas partes: a lista e a execução da lista.

Exemplo:

#define LIST \
    X(dog)   \
    X(cat)   \
    X(racoon)

// class Animal {
//  public:
//    void say();
// };

#define X(name) Animal name;
LIST
#undef X

int main() {
#define X(name) name.say();
    LIST
#undef X

    return 0;
}

que é expandido pelo pré-processador para o seguinte:

Animal dog;
Animal cat;
Animal racoon;

int main() {
    dog.say();
    cat.say();
    racoon.say();

    return 0;
}

À medida que as listas se tornam maiores (digamos, mais de 100 elementos), essa técnica ajuda a evitar copiar e colar em excesso.

Fonte: https://en.wikipedia.org/wiki/X_Macro

Veja também: https://www.wikiod.com/pt/c/x-macros

Se definir um X aparentemente irrelevante antes de usar LIST não for do seu agrado, você também pode passar um nome de macro como argumento:

#define LIST(MACRO) \
    MACRO(dog) \
    MACRO(cat) \
    MACRO(racoon)

Agora, você especifica explicitamente qual macro deve ser usada ao expandir a lista, por exemplo,

#define FORWARD_DECLARE_ANIMAL(name) Animal name;
LIST(FORWARD_DECLARE_ANIMAL)

Se cada chamada do MACRO deve ter parâmetros adicionais - constantes em relação à lista, macros variáveis ​​podem ser usadas

//a walkaround for Visual studio
#define EXPAND(x) x

#define LIST(MACRO, ...) \
    EXPAND(MACRO(dog, __VA_ARGS__)) \
    EXPAND(MACRO(cat, __VA_ARGS__)) \
    EXPAND(MACRO(racoon, __VA_ARGS__))

O primeiro argumento é fornecido pela LIST, enquanto o restante é fornecido pelo usuário na invocação LIST. Por exemplo:

#define FORWARD_DECLARE(name, type, prefix) type prefix##name;
LIST(FORWARD_DECLARE,Animal,anim_)
LIST(FORWARD_DECLARE,Object,obj_)

se expandirá para

Animal anim_dog;
Animal anim_cat;
Animal anim_racoon;
Object obj_dog;
Object obj_cat;
Object obj_racoon;

Mensagens de erro do pré-processador #

Erros de compilação podem ser gerados usando o pré-processador. Isso é útil por vários motivos, alguns dos quais incluem notificar um usuário se ele estiver em uma plataforma sem suporte ou em um compilador sem suporte.

por exemplo. Retorna o erro se a versão do gcc for 3.0.0 ou anterior.

#if __GNUC__ < 3
#error "This code requires gcc > 3.0.0"
#endif

por exemplo. Retorna Erro se compilar em um computador Apple.

#ifdef __APPLE__
#error "Apple products are not supported in this release"
#endif

Macros predefinidas #

As macros predefinidas são aquelas que o compilador define (em contraste com aquelas definidas pelo usuário no arquivo de origem). Essas macros não devem ser redefinidas ou indefinidas pelo usuário.

As seguintes macros são predefinidas pelo padrão C++:

  • __LINE__ contém o número da linha em que esta macro é usada, e pode ser alterada pela diretiva #line.
  • __FILE__ contém o nome do arquivo no qual esta macro é usada, e pode ser alterada pela diretiva #line.
  • __DATE__ contém a data (no formato "Mmm dd yyyy") da compilação do arquivo, onde Mmm é formatado como se fosse obtido por uma chamada para std::asctime().
  • __TIME__ contém a hora (no formato "hh:mm:ss") da compilação do arquivo.
  • __cplusplus é definido por compiladores C++ (conformes) durante a compilação de arquivos C++. Seu valor é a versão padrão com a qual o compilador está totalmente em conformidade, ou seja, 199711L para C++98 e C++03, 201103L para C++11 e 201402L para C++14 padrão .
  • __STDC_HOSTED__ é definido como 1 se a implementação for hospedada, ou 0 se for autônoma.
  • __STDCPP_DEFAULT_NEW_ALIGNMENT__ contém um literal size_t, que é o alinhamento usado para uma chamada ao operator new que não reconhece alinhamento.

Além disso, as seguintes macros podem ser predefinidas por implementações e podem ou não estar presentes:

  • __STDC__ tem um significado dependente da implementação e geralmente é definido apenas ao compilar um arquivo como C, para significar total conformidade com o padrão C. (Ou nunca, se o compilador decidir não oferecer suporte a essa macro.)
  • __STDC_VERSION__ tem significado dependente da implementação, e seu valor geralmente é a versão C, similarmente como __cplusplus é a versão C++. (Ou nem está definido, se o compilador decidir não suportar esta macro.)
  • __STDC_MB_MIGHT_NEQ_WC__ é definido como 1, se os valores da codificação restrita do conjunto básico de caracteres podem não ser iguais aos valores de suas contrapartes amplas (por exemplo, se (uintmax_t)'x' != (uintmax_t)L' x')
  • __STDC_ISO_10646__ é definido se wchar_t for codificado como Unicode, e se expande para uma constante inteira no formato yyyymmL, indicando a última revisão Unicode suportada.
  • __STDCPP_STRICT_POINTER_SAFETY__ é definido como 1, se a implementação tiver segurança de ponteiro estrita (caso contrário, ela tem segurança de ponteiro relaxada)
  • __STDCPP_THREADS__ é definido como 1, se o programa pode ter mais de um thread de execução (aplicável a implementação independente — implementações hospedadas sempre podem ter mais de um thread)

Também vale a pena mencionar __func__, que não é uma macro, mas uma variável local de função predefinida. Ele contém o nome da função em que é usado, como uma matriz de caracteres estática em um formato definido pela implementação.

Além dessas macros predefinidas padrão, os compiladores podem ter seu próprio conjunto de macros predefinidas. Deve-se consultar a documentação do compilador para aprender isso. Por exemplo.:

  • gcc
  • Microsoft Visual C++
  • clang
  • Compilador Intel C++

Algumas das macros são apenas para consultar o suporte de algum recurso:

#ifdef __cplusplus // if compiled by C++ compiler
extern "C"{ // C code has to be decorated
   // C library header declarations here
}
#endif

Outros são muito úteis para depuração:

bool success = doSomething( /*some arguments*/ );
if( !success ){
    std::cerr << "ERROR: doSomething() failed on line " << __LINE__ - 2
              << " in function " << __func__ << "()"
              << " in file " << __FILE__
              << std::endl;
}

E outros para controle de versão trivial:

int main( int argc, char *argv[] ){
    if( argc == 2 && std::string( argv[1] ) == "-v" ){
        std::cout << "Hello World program\n"
                  << "v 1.1\n" // I have to remember to update this manually
                  << "compiled: " << __DATE__ << ' ' << __TIME__ // this updates automagically
                  << std::endl;
    }
    else{
        std::cout << "Hello World!\n";
    }
}

#pragma uma vez #

A maioria, mas não todas as implementações C++ suportam a diretiva #pragma once que garante que o arquivo seja incluído apenas uma vez em uma única compilação. Não faz parte de nenhum padrão ISO C++. Por exemplo:

// Foo.h
#pragma once

class Foo
{
};

Enquanto #pragma once evita alguns problemas associados com include guards, um #pragma - por definição nos padrões - é inerentemente um gancho específico do compilador, e será silenciosamente ignorado por compiladores que não apoie isso. Projetos que usam #pragma once devem ser modificados para serem compatíveis com o padrão.

Com alguns compiladores - particularmente aqueles que empregam cabeçalhos pré-compilados - #pragma once pode resultar em uma considerável aceleração do processo de compilação. Da mesma forma, alguns pré-processadores conseguem acelerar a compilação rastreando quais cabeçalhos empregaram incluindo guardas. O benefício líquido, quando ambos #pragma once e include guards são empregados, depende da implementação e pode ser um aumento ou diminuição dos tempos de compilação.

#pragma once combinado com include guards foi o layout recomendado para arquivos de cabeçalho ao escrever aplicativos baseados em MFC em janelas, e foi gerado pelos assistentes add class, add dialog, add windows do Visual Studio . Por isso, é muito comum encontrá-los combinados em C++ Windows Candidates.

Operadores de pré-processador #

O operador # ou operador de stringizing é usado para converter um parâmetro Macro em um literal de string. Só pode ser usado com as Macros com argumentos.

// preprocessor will convert the parameter x to the string literal x
#define PRINT(x) printf(#x "\n")

PRINT(This line will be converted to string by preprocessor);
// Compiler sees
printf("This line will be converted to string by preprocessor""\n");

O compilador concatena duas strings e o argumento final printf() será uma string literal com um caractere de nova linha no final.

O pré-processador ignorará os espaços antes ou depois do argumento da macro. Portanto, a instrução print abaixo nos dará o mesmo resultado.

PRINT(   This line will be converted to string by preprocessor );

Se o parâmetro da string literal requer uma sequência de escape como antes de aspas duplas (), ela será inserida automaticamente pelo pré-processador.

PRINT(This "line" will be converted to "string" by preprocessor); 
// Compiler sees
printf("This \"line\" will be converted to \"string\" by preprocessor""\n");

O operador ## ou operador de colagem de token é usado para concatenar dois parâmetros ou tokens de uma macro.

// preprocessor will combine the variable and the x
#define PRINT(x) printf("variable" #x " = %d", variable##x)

int variableY = 15;
PRINT(Y);
//compiler sees
printf("variable""Y"" = %d", variableY);

e a saída final será

variableY = 15
