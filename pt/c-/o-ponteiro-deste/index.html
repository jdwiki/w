O ponteiro this é uma palavra-chave para C++, portanto, não há necessidade de biblioteca para implementá-lo. E não esqueça que isto é um ponteiro! Então você não pode fazer:

 this.someMember();

Conforme você acessa funções-membro ou variáveis-membro a partir de ponteiros usando o símbolo de seta ->:

this->someMember();

este ponteiro #

Todas as funções-membro não estáticas têm um parâmetro oculto, um ponteiro para uma instância da classe, chamado this; esse parâmetro é inserido silenciosamente no início da lista de parâmetros e tratado inteiramente pelo compilador. Quando um membro da classe é acessado dentro de uma função membro, ele é acessado silenciosamente através de this; isso permite que o compilador use uma única função de membro não estática para todas as instâncias e permite que uma função de membro chame outras funções de membro polimorficamente.

struct ThisPointer {
    int i;

    ThisPointer(int ii);

    virtual void func();

    int  get_i() const;
    void set_i(int ii);
};
ThisPointer::ThisPointer(int ii) : i(ii) {}
// Compiler rewrites as:
ThisPointer::ThisPointer(int ii) : this->i(ii) {}
// Constructor is responsible for turning allocated memory into 'this'.
// As the constructor is responsible for creating the object, 'this' will not be "fully"
// valid until the instance is fully constructed.

/* virtual */ void ThisPointer::func() {
    if (some_external_condition) {
        set_i(182);
    } else {
        i = 218;
    }
}
// Compiler rewrites as:
/* virtual */ void ThisPointer::func(ThisPointer* this) {
    if (some_external_condition) {
        this->set_i(182);
    } else {
        this->i = 218;
    }
}

int  ThisPointer::get_i() const { return i; }
// Compiler rewrites as:
int  ThisPointer::get_i(const ThisPointer* this) { return this->i; }

void ThisPointer::set_i(int ii) { i = ii; }
// Compiler rewrites as:
void ThisPointer::set_i(ThisPointer* this, int ii) { this->i = ii; }

Em um construtor, this pode ser usado com segurança para (implícita ou explicitamente) acessar qualquer campo que já tenha sido inicializado, ou qualquer campo em uma classe pai; por outro lado, (implícita ou explicitamente) acessar quaisquer campos que ainda não foram inicializados, ou quaisquer campos em uma classe derivada, não é seguro (devido à classe derivada ainda não estar sendo construída e, portanto, seus campos não serem inicializados nem existentes). Também não é seguro chamar funções-membro virtuais por meio de this no construtor, pois qualquer função de classe derivada não será considerada (devido à classe derivada ainda não estar sendo construída e, portanto, seu construtor ainda não atualizar a vtable).

Observe também que, enquanto estiver em um construtor, o tipo do objeto é o tipo que esse construtor constrói. Isso vale mesmo se o objeto for declarado como um tipo derivado. Por exemplo, no exemplo abaixo, ctd_good e ctd_bad são do tipo CtorThisBase dentro de CtorThisBase(), e digite CtorThis dentro de CtorThis(), mesmo que seu tipo canônico seja CtorThisDerived. À medida que as classes mais derivadas são construídas em torno da classe base, a instância passa gradualmente pela hierarquia de classes até se tornar uma instância totalmente construída de seu tipo pretendido.

class CtorThisBase {
    short s;

  public:
    CtorThisBase() : s(516) {}
};

class CtorThis : public CtorThisBase {
    int i, j, k;

  public:
    // Good constructor.
    CtorThis() : i(s + 42), j(this->i), k(j) {}

    // Bad constructor.
    CtorThis(int ii) : i(ii), j(this->k), k(b ? 51 : -51) {
        virt_func();
    }

    virtual void virt_func() { i += 2; }
};

class CtorThisDerived : public CtorThis {
    bool b;

  public:
    CtorThisDerived()       : b(true) {}
    CtorThisDerived(int ii) : CtorThis(ii), b(false) {}

    void virt_func() override { k += (2 * i); }
};

// ...

CtorThisDerived ctd_good;
CtorThisDerived ctd_bad(3);

Com essas classes e funções de membro:

  • No construtor bom, para ctd_good:
  • CtorThisBase é totalmente construído no momento em que o construtor CtorThis é inserido. Portanto, s está em um estado válido enquanto inicializa i, e pode ser acessado.
  • i é inicializado antes de j(this->i) ser alcançado. Portanto, i está em um estado válido durante a inicialização de j e pode ser acessado.
  • j é inicializado antes de k(j) ser alcançado. Portanto, j está em um estado válido durante a inicialização de k e, portanto, pode ser acessado.
  • No construtor ruim, para ctd_bad:
  • k é inicializado após j(this->k) ser alcançado. Portanto, k está em um estado inválido ao inicializar j, e acessá-lo causa um comportamento indefinido.
  • CtorThisDerived não é construído até que CtorThis seja construído. Portanto, b está em um estado inválido ao inicializar k, e acessá-lo causa um comportamento indefinido.
  • O objeto ctd_bad ainda é um CtorThis até que saia de CtorThis(), e não será atualizado para usar a vtable de CtorThisDerived até CtorThisDerived(). Portanto, virt_func() chamará CtorThis::virt_func(), independentemente se pretende chamar isso ou CtorThisDerived::virt_func().

Usando este ponteiro para acessar dados de membros #

Nesse contexto, usar o ponteiro this não é totalmente necessário, mas tornará seu código mais claro para o leitor, indicando que uma determinada função ou variável é membro da classe. Um exemplo nesta situação:

// Example for this pointer
#include <iostream>
#include <string>

using std::cout;
using std::endl;

class Class
{
  public:
    Class();
    ~Class();
    int getPrivateNumber () const;
  private:
    int private_number = 42;
};

Class::Class(){}
Class::~Class(){}

int Class::getPrivateNumber() const
{
    return this->private_number;
}

int main()
{
    Class class_example;
    cout << class_example.getPrivateNumber() << endl;
}

Usando este ponteiro para diferenciar entre dados de membros e parâmetros #

Esta é uma estratégia útil real para diferenciar dados de membros de parâmetros… Vamos dar este exemplo:

// Dog Class Example
#include <iostream>
#include <string>

using std::cout;
using std::endl;

/*
* @class Dog
*   @member name
*       Dog's name
*   @function bark
*       Dog Barks!
*   @function getName
*       To Get Private
*       Name Variable
*/
class Dog
{
 public:
    Dog(std::string name);
    ~Dog();
    void  bark() const;
    std::string  getName() const;
 private:
    std::string name;
};

Dog::Dog(std::string name)
{
    /*
    *  this->name is the
    *  name variable from 
    *  the class dog . and
    *  name is from the 
    *  parameter of the function
    */
    this->name = name; 
}

Dog::~Dog(){}

void Dog::bark() const
{
  cout << "BARK" << endl;   
}

std::string  Dog::getName() const
{
    return this->name;
}


int main()
{
    Dog dog("Max");
    cout << dog.getName() << endl;
    dog.bark();
}

Você pode ver aqui no construtor que executamos o seguinte:

this->name = name;

Aqui, você pode ver que estamos associando o nome do parâmetro ao nome da variável privada da classe Dog(this->name) .

este Pointer CV-Qualifiers #

this também pode ser qualificado como cv, da mesma forma que qualquer outro ponteiro. No entanto, devido ao parâmetro this não estar listado na lista de parâmetros, uma sintaxe especial é necessária para isso; os cv-qualifiers são listados após a lista de parâmetros, mas antes do corpo da função.

struct ThisCVQ {
    void no_qualifier()                {} // "this" is: ThisCVQ*
    void  c_qualifier() const          {} // "this" is: const ThisCVQ*
    void  v_qualifier() volatile       {} // "this" is: volatile ThisCVQ*
    void cv_qualifier() const volatile {} // "this" is: const volatile ThisCVQ*
};

Como this é um parâmetro, uma função pode ser sobrecarregada com base em seu(s) qualificador(es) de cv this.

struct CVOverload {
    int func()                { return    3; }
    int func() const          { return   33; }
    int func() volatile       { return  333; }
    int func() const volatile { return 3333; }
};

Quando this é const (incluindo const volátil), a função não pode escrever em variáveis ​​de membro por meio dela, seja de forma implícita ou explícita. A única exceção a isso são variáveis ​​de membro mutáveis, que podem ser escritas independentemente da const-ness. Devido a isso, const é usado para indicar que a função membro não altera o estado lógico do objeto (a forma como o objeto aparece para o mundo exterior), mesmo que modifique o estado físico (a forma como o objeto parece sob o capuz).

O estado lógico é a forma como o objeto aparece para observadores externos. Não está diretamente ligado ao estado físico e, de fato, pode nem ser armazenado como estado físico. Enquanto observadores externos não puderem ver nenhuma mudança, o estado lógico é constante, mesmo se você inverter cada bit no objeto.

O estado físico, também conhecido como estado bit a bit, é como o objeto é armazenado na memória. Este é o âmago da questão do objeto, os 1s e 0s brutos que compõem seus dados. Um objeto só é fisicamente constante se sua representação na memória nunca mudar.

Observe que C++ baseia a constness no estado lógico, não no estado físico.

class DoSomethingComplexAndOrExpensive {
    mutable ResultType cached_result;
    mutable bool state_changed;

    ResultType calculate_result();
    void modify_somehow(const Param& p);

    // ...

  public:
    DoSomethingComplexAndOrExpensive(Param p) : state_changed(true) {
        modify_somehow(p);
    }

    void change_state(Param p) {
        modify_somehow(p);
        state_changed = true;
    }

    // Return some complex and/or expensive-to-calculate result.
    // As this has no reason to modify logical state, it is marked as "const".
    ResultType get_result() const;
};
ResultType DoSomethingComplexAndOrExpensive::get_result() const {
    // cached_result and state_changed can be modified, even with a const "this" pointer.
    // Even though the function doesn't modify logical state, it does modify physical state
    //  by caching the result, so it doesn't need to be recalculated every time the function
    //  is called.  This is indicated by cached_result and state_changed being mutable.

    if (state_changed) {
        cached_result = calculate_result();
        state_changed = false;
    }

    return cached_result;
}

Note que enquanto você tecnicamente poderia usar const_cast em this para torná-lo não qualificado para cv, você realmente, REALMENTE não deveria, e deveria usar mutable ao invés. Um const_cast pode invocar um comportamento indefinido quando usado em um objeto que realmente é const, enquanto mutable é projetado para ser seguro de usar. É, no entanto, possível que você encontre isso em um código extremamente antigo.

Uma exceção a esta regra é definir acessadores não qualificados como cv em termos de acessadores const; como o objeto é garantido para não ser const se a versão não qualificada para cv for chamada, não há risco de UB.

class CVAccessor {
    int arr[5];

  public:
    const int& get_arr_element(size_t i) const { return arr[i]; }

    int& get_arr_element(size_t i) {
        return const_cast<int&>(const_cast<const CVAccessor*>(this)->get_arr_element(i));
    }
};

Isso evita a duplicação desnecessária de código.

Assim como os ponteiros normais, se this for volátil (incluindo const volátil), ele será carregado da memória toda vez que for acessado, em vez de ser armazenado em cache. Isso tem os mesmos efeitos na otimização que declarar qualquer outro ponteiro como ‘volátil’, portanto, deve-se tomar cuidado.

Observe que, se uma instância for qualificada para cv, as únicas funções-membro que ela pode acessar são funções-membro cujo ponteiro this é pelo menos tão qualificado para cv quanto a própria instância:

  • Instâncias não-cv podem acessar qualquer função membro.
  • As instâncias const podem acessar as funções const e const volátil.
  • Instâncias volatile podem acessar funções volatile e constvolatile.
  • Instâncias const voláteis podem acessar funções const voláteis.

Este é um dos princípios-chave da const correção.

struct CVAccess {
    void    func()                {}
    void  func_c() const          {}
    void  func_v() volatile       {}
    void func_cv() const volatile {}
};

CVAccess cva;
cva.func();    // Good.
cva.func_c();  // Good.
cva.func_v();  // Good.
cva.func_cv(); // Good.

const CVAccess c_cva;
c_cva.func();    // Error.
c_cva.func_c();  // Good.
c_cva.func_v();  // Error.
c_cva.func_cv(); // Good.

volatile CVAccess v_cva;
v_cva.func();    // Error.
v_cva.func_c();  // Error.
v_cva.func_v();  // Good.
v_cva.func_cv(); // Good.

const volatile CVAccess cv_cva;
cv_cva.func();    // Error.
cv_cva.func_c();  // Error.
cv_cva.func_v();  // Error.
cv_cva.func_cv(); // Good.

este Pointer Ref-Qualifiers #

Similarmente aos qualificadores cv this, também podemos aplicar ref-qualifiers a *this. Os qualificadores de referência são usados ​​para escolher entre a semântica de referência normal e rvalue, permitindo que o compilador use a semântica de copiar ou mover dependendo de qual for mais apropriada, e são aplicados a *this em vez de this.

Observe que, apesar dos qualificadores de referência usarem a sintaxe de referência, this em si ainda é um ponteiro. Observe também que os qualificadores de referência não alteram o tipo de *this; é apenas mais fácil descrever e entender seus efeitos olhando para eles como se o fizessem.

struct RefQualifiers {
    std::string s;

    RefQualifiers(const std::string& ss = "The nameless one.") : s(ss) {}

    // Normal version.
    void func() &  { std::cout << "Accessed on normal instance "    << s << std::endl; }
    // Rvalue version.
    void func() && { std::cout << "Accessed on temporary instance " << s << std::endl; }

    const std::string& still_a_pointer() &  { return this->s; }
    const std::string& still_a_pointer() && { this->s = "Bob"; return this->s; }
};

// ...

RefQualifiers rf("Fred");
rf.func();              // Output:  Accessed on normal instance Fred
RefQualifiers{}.func(); // Output:  Accessed on temporary instance The nameless one

Uma função de membro não pode ter sobrecargas com e sem qualificadores de referência; o programador tem que escolher entre um ou outro. Felizmente, cv-qualifiers podem ser usados ​​em conjunto com ref-qualifiers, permitindo que as regras de const correctness sejam seguidas.

struct RefCV {
    void func() &                {}
    void func() &&               {}
    void func() const&           {}
    void func() const&&          {}
    void func() volatile&        {}
    void func() volatile&&       {}
    void func() const volatile&  {}
    void func() const volatile&& {}
};
