Verificando os itens do Menu de Opções (usando Spoon para tirar screenshots) #

/**
 * @author piotrek1543
 *
 * This example provides a specific UI testing problem and how it is already solved 
 * with Google's Espresso. Notice that I used also Spoon framework, as Espresso 
 * lacks of taking screenshots functionality.
 */

@RunWith(AndroidJUnit4.class)
public class MainActivityAndroidTest {
    @Rule
    public ActivityTestRule<MainActivity> mRule = new ActivityTestRule<>(MainActivity.class);

    @Test
    public void checkIfSettingsMenuItemsAreVisible() throws InterruptedException {
        //open OptionsMenu to see available items
        openActionBarOverflowOrOptionsMenu(mRule.getActivity());
        //create a screenshot with 'options_menu' TAG
        Spoon.screenshot(mRule.getActivity(), "options_menu");
        //check if Settings item is Visible
        onView(withText(R.string.action_settings)).check(matches(isDisplayed()));
        //check if `Sort` item is Visible
        onView(withText(R.string.action_sort)).check(matches(isDisplayed()));
        //perform click on `Sort` OptionsMenu item
        onView(withText(R.string.action_sort)).perform(click());
        //create a screenshot with 'options_menu_sort' TAG
        Spoon.screenshot(mRule.getActivity(), "options_menu_sort");
        //check if `Sort -> By Value id` item is Visible
        onView(withText(R.string.menu_sort_length)).check(matches(isDisplayed()));
        //check if `Sort -> By Joke length` item is Visible
        onView(withText(R.string.menu_sort_a_z)).check(matches(isDisplayed()));
    }
}

Exemplo de Expresso Hello World #

Este é um tutorial para criar um exemplo de hello world: Usado para este exemplo: Android Studio 2.3;

Para começar, use o Android Studio para criar um novo projeto com uma atividade vazia. Em seguida, adicionamos algumas funcionalidades simples ao App que podemos testar: Adicionamos um botão que ao clicar exibe “Hello World” em uma visualização de texto.

O código da atividade fica assim:

package com.example.testing.helloworld;

import android.os.Bundle;
import android.support.v7.app.AppCompatActivity;
import android.view.View;
import android.widget.TextView;

public class MainActivity extends AppCompatActivity {

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        final TextView textView = (TextView) findViewById(R.id.textView);

        findViewById(R.id.button).setOnClickListener(new View.OnClickListener() {
            @Override
            public void onClick(View v) {
                textView.setText("Hello World!");
            }
        });
    }
}

E o layout activity_main para esta atividade é assim:

<?xml version="1.0" encoding="utf-8"?>
<LinearLayout xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:orientation="vertical">

    <TextView
        android:id="@+id/textView"
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="" />

    <Button
        android:id="@+id/button"
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="Say Hello" />
</LinearLayout>

Agora queremos testar o comportamento deste novo aplicativo criado usando espresso. Geralmente, o código do seu aplicativo está dentro do pacote main, os Testes Unitários estão dentro do test e os testes de instrumentação espresso estão dentro do pacote androidTest. Se você criar um novo projeto de atividade vazio com o Android Studio, ele já deve ter criado esses pacotes e classes e deve ficar assim:

Para começar com o espresso temos que ter certeza de que a dependência espresso-core está incluída no arquivo build.gradle (observe que ela não é anotada com a palavra-chave compile mas sim com androidTestCompile). As dependências no arquivo build.gradle criado pelo Android studio devem ficar assim:

dependencies {
    compile fileTree(dir: 'libs', include: ['*.jar'])
    androidTestCompile('com.android.support.test.espresso:espresso-core:2.2.2', {
        exclude group: 'com.android.support', module: 'support-annotations'
    })
    compile 'com.android.support:appcompat-v7:25.2.0'
    compile 'com.android.support.constraint:constraint-layout:1.0.2'
    testCompile 'junit:junit:4.12'
}

Agora que tudo está configurado, podemos começar com o teste real: Abra o arquivo ExampleInstrumentationTest e você verá que já existe um teste useAppContext gerado dentro. Vamos alterar esta classe de teste e criar um teste para verificar o comportamento do nosso aplicativo:

@RunWith(AndroidJUnit4.class)
public class ExampleInstrumentedTest {

    @Rule
    public ActivityTestRule<MainActivity> mActivityRule = new ActivityTestRule<>(
        MainActivity.class, false, true);

    @Test
    public void checkHelloWorld() throws Exception {
        onView(withId(R.id.textView)).check(matches(withText("")));
        onView(withId(R.id.button)).perform(click());
        onView(withId(R.id.textView)).check(matches(withText("Hello World!")));
    }
}

Inicie o teste executando a classe ExampleInstrumentedTest. Este teste, então, faz três coisas:

  1. Verifica se o textview contém uma string vazia ("")
  2. Clique no botão em nosso layout
  3. Ele verifica novamente o texto do textview se ele contém “Hello World!”

A ActivityTestRule na parte superior define qual atividade é testada e a inicia no início do teste. (Você também pode desativar o início automático de uma atividade e, em vez disso, iniciá-lo manualmente em cada teste)

As regras de teste são bem simples:

  • onView(withId(R.id.textView)) procura uma visão dentro da tela atual pelo ID da visão dentro do nosso arquivo de layout activity_main.
  • .check(matches(withText(""))); então executa um caso de teste nessa visão.
  • .perform(click()) executa uma ação em uma visualização: Essas ações podem ser cliques, cliques longos ou furtos ou algo mais.

Este foi um tutorial para começar com os testes de instrumentação do Android espresso, espero que tenha lhe dado alguns insights!

Configure seu ambiente de teste

Para evitar falhas, é altamente recomendável desativar as animações do sistema nos dispositivos virtuais ou físicos usados ​​para teste.

No seu dispositivo, em Configurações-> Opções do desenvolvedor, desative as 3 configurações a seguir:

  • Escala de animação da janela
  • Escala de animação de transição
  • Escala de duração do animador

Baixar Expresso

  • Certifique-se de ter instalado o Android Support Repository mais recente em Extras (consulte as instruções).

  • Abra o arquivo build.gradle do seu aplicativo. Geralmente não é o arquivo build.gradle de nível superior, mas app/build.gradle.

  • Adicione as seguintes linhas dentro das dependências:

    androidTestCompile ‘com.android.support.test.espresso:espresso-core:2.2.2’ androidTestCompile ‘com.android.support.test🏃‍♂️0.5’

  • Veja a seção de downloads para mais artefatos (espresso-contrib, espresso-web, etc.)

  • Definir o corredor de instrumentação

Adicione ao mesmo arquivo build.gradle a seguinte linha em android.defaultConfig: testInstrumentationRunner "android.support.test.runner.AndroidJUnitRunner" Exemplo de arquivo build.gradle

apply plugin: 'com.android.application'

    android {
        compileSdkVersion 22
        buildToolsVersion "22"
    
        defaultConfig {
            applicationId "com.my.awesome.app"
            minSdkVersion 10
            targetSdkVersion 22.0.1
            versionCode 1
            versionName "1.0"
    
            testInstrumentationRunner "android.support.test.runner.AndroidJUnitRunner"
        }
    }

dependencies {
    // App's dependencies, including test
    compile 'com.android.support:support-annotations:22.2.0'

    // Testing-only dependencies
    androidTestCompile 'com.android.support.test🏃‍♂️0.5'
    androidTestCompile 'com.android.support.test.espresso:espresso-core:2.2.2'
}

Análise

Para garantir que estamos no caminho certo a cada nova versão, o executor de testes coleta análises. Mais especificamente, ele carrega um hash do nome do pacote do aplicativo em teste para cada chamada. Isso nos permite medir tanto a contagem de pacotes únicos usando Espresso quanto o volume de uso.

Se você não deseja fazer upload desses dados, pode optar por não passar o seguinte argumento para o executor de teste: disableAnalytics “true” (veja como passar argumentos personalizados).

Executando testes

No Android Studio

Criar uma configuração de teste

No Android Studio:

  • Abra o menu Executar -> Editar configurações

  • Adicionar uma nova configuração de testes do Android

  • Escolha um módulo

  • Adicione um corredor de instrumentação específico:

    android.support.test.runner.AndroidJUnitRunner

Execute a configuração recém-criada.

Da linha de comando via Gradle

Executar

./gradlew connectedAndroidTest

O expresso tem basicamente três componentes:

  1. ViewMatchers - permite encontrar visão na hierarquia de visão atual

  2. ViewActions - permite realizar ações nas visualizações

  3. ViewAssertions - permite afirmar o estado de uma visão

Teste Espresso Base

onView(ViewMatcher)       -- 1     
 .perform(ViewAction)     -- 2
   .check(ViewAssertion); -- 3
  1. Encontra a vista
  2. Executa uma ação na visualização
  3. Valida uma afirmação
