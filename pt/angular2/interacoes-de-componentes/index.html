Passar dados de pai para filho com ligação de entrada #

HeroChildComponent tem duas propriedades de entrada, normalmente adornadas com decorações @Input.

import { Component, Input } from '@angular/core';
import { Hero } from './hero';
@Component({
  selector: 'hero-child',
  template: `
    <h3>{{hero.name}} says:</h3>
    <p>I, {{hero.name}}, am at your service, {{masterName}}.</p>
  `
})
export class HeroChildComponent {
  @Input() hero: Hero;
  @Input('master') masterName: string;
}

Interceptar alterações de propriedade de entrada com um setter

Use um configurador de propriedade de entrada para interceptar e agir sobre um valor do pai.

O setter da propriedade name input no filho NameChildComponent corta o espaço em branco de um nome e substitui um valor vazio pelo texto padrão.

import { Component, Input } from '@angular/core';
@Component({
  selector: 'name-child',
  template: '<h3>"{{name}}"</h3>'
})
export class NameChildComponent {
  private _name = '';
  @Input()
  set name(name: string) {
    this._name = (name && name.trim()) || '<no name set>';
  }
  get name(): string { return this._name; }
}

Aqui está o NameParentComponent demonstrando variações de nome, incluindo um nome com todos os espaços:

import { Component } from '@angular/core';
@Component({
  selector: 'name-parent',
  template: `
  <h2>Master controls {{names.length}} names</h2>
  <name-child *ngFor="let name of names" [name]="name"></name-child>
  `
})
export class NameParentComponent {
  // Displays 'Mr. IQ', '<no name set>', 'Bombasto'
  names = ['Mr. IQ', '   ', '  Bombasto  '];
}

O pai ouve o evento filho

O componente filho expõe uma propriedade EventEmitter com a qual emite eventos quando algo acontece. O pai se associa a essa propriedade de evento e reage a esses eventos.

A propriedade EventEmitter do filho é uma propriedade de saída, normalmente adornada com uma decoração @Output como visto neste VoterComponent:

import { Component, EventEmitter, Input, Output } from '@angular/core';
@Component({
  selector: 'my-voter',
  template: `
    <h4>{{name}}</h4>
    <button (click)="vote(true)"  [disabled]="voted">Agree</button>
    <button (click)="vote(false)" [disabled]="voted">Disagree</button>
  `
})
export class VoterComponent {
  @Input()  name: string;
  @Output() onVoted = new EventEmitter<boolean>();
  voted = false;
  vote(agreed: boolean) {
    this.onVoted.emit(agreed);
    this.voted = true;
  }
}

Clicar em um botão aciona a emissão de um verdadeiro ou falso (a carga booleana).

O pai VoteTakerComponent vincula um manipulador de eventos (onVoted) que responde à carga útil do evento filho ($event) e atualiza um contador.

import { Component }      from '@angular/core';
@Component({
  selector: 'vote-taker',
  template: `
    <h2>Should mankind colonize the Universe?</h2>
    <h3>Agree: {{agreed}}, Disagree: {{disagreed}}</h3>
    <my-voter *ngFor="let voter of voters"
      [name]="voter"
      (onVoted)="onVoted($event)">
    </my-voter>
  `
})
export class VoteTakerComponent {
  agreed = 0;
  disagreed = 0;
  voters = ['Mr. IQ', 'Ms. Universe', 'Bombasto'];
  onVoted(agreed: boolean) {
    agreed ? this.agreed++ : this.disagreed++;
  }
}

O pai interage com o filho por meio de variável local

Um componente pai não pode usar vinculação de dados para ler propriedades filho ou invocar métodos filho. Podemos fazer as duas coisas criando uma variável de referência de modelo para o elemento filho e, em seguida, referenciando essa variável no modelo pai, conforme visto no exemplo a seguir.

Temos um CountdownTimerComponent filho que faz a contagem regressiva repetidamente até zero e lança um foguete. Possui métodos de início e parada que controlam o relógio e exibe uma mensagem de status de contagem regressiva em seu próprio modelo.

import { Component, OnDestroy, OnInit } from '@angular/core';
@Component({
  selector: 'countdown-timer',
  template: '<p>{{message}}</p>'
})
export class CountdownTimerComponent implements OnInit, OnDestroy {
  intervalId = 0;
  message = '';
  seconds = 11;
  clearTimer() { clearInterval(this.intervalId); }
  ngOnInit()    { this.start(); }
  ngOnDestroy() { this.clearTimer(); }
  start() { this.countDown(); }
  stop()  {
    this.clearTimer();
    this.message = `Holding at T-${this.seconds} seconds`;
  }
  private countDown() {
    this.clearTimer();
    this.intervalId = window.setInterval(() => {
      this.seconds -= 1;
      if (this.seconds === 0) {
        this.message = 'Blast off!';
      } else {
        if (this.seconds < 0) { this.seconds = 10; } // reset
        this.message = `T-${this.seconds} seconds and counting`;
      }
    }, 1000);
  }
}

Vamos ver o CountdownLocalVarParentComponent que hospeda o componente timer.

import { Component }                from '@angular/core';
import { CountdownTimerComponent }  from './countdown-timer.component';
@Component({
  selector: 'countdown-parent-lv',
  template: `
  <h3>Countdown to Liftoff (via local variable)</h3>
  <button (click)="timer.start()">Start</button>
  <button (click)="timer.stop()">Stop</button>
  <div class="seconds">{{timer.seconds}}</div>
  <countdown-timer #timer></countdown-timer>
  `,
  styleUrls: ['demo.css']
})
export class CountdownLocalVarParentComponent { }

O componente pai não pode vincular dados aos métodos start e stop do filho nem à sua propriedade seconds.

Podemos colocar uma variável local (#timer) na tag () representando o componente filho. Isso nos dá uma referência ao próprio componente filho e a capacidade de acessar qualquer uma de suas propriedades ou métodos de dentro do modelo pai.

Neste exemplo, conectamos os botões pai ao start e stop do filho e usamos a interpolação para exibir a propriedade de segundos do filho.

Aqui vemos o pai e a criança trabalhando juntos.

O pai liga para um ViewChild

A abordagem da variável local é simples e fácil. Mas é limitado porque a ligação pai-filho deve ser feita inteiramente dentro do modelo pai. O próprio componente pai não tem acesso ao filho.

Não podemos usar a técnica de variável local se uma instância da classe do componente pai precisar ler ou gravar valores do componente filho ou precisar chamar métodos do componente filho.

Quando a classe do componente pai requer esse tipo de acesso, injetamos o componente filho no pai como um ViewChild.

Ilustraremos essa técnica com o mesmo exemplo do temporizador de contagem regressiva. Não mudaremos sua aparência ou comportamento. O filho CountdownTimerComponent também é o mesmo.

Estamos mudando da variável local para a técnica ViewChild apenas para fins de demonstração. Aqui está o pai, CountdownViewChildParentComponent:

import { AfterViewInit, ViewChild } from '@angular/core';
import { Component }                from '@angular/core';
import { CountdownTimerComponent }  from './countdown-timer.component';
@Component({
  selector: 'countdown-parent-vc',
  template: `
  <h3>Countdown to Liftoff (via ViewChild)</h3>
  <button (click)="start()">Start</button>
  <button (click)="stop()">Stop</button>
  <div class="seconds">{{ seconds() }}</div>
  <countdown-timer></countdown-timer>
  `,
  styleUrls: ['demo.css']
})
export class CountdownViewChildParentComponent implements AfterViewInit {
  @ViewChild(CountdownTimerComponent)
  private timerComponent: CountdownTimerComponent;
  seconds() { return 0; }
  ngAfterViewInit() {
    // Redefine `seconds()` to get from the `CountdownTimerComponent.seconds` ...
    // but wait a tick first to avoid one-time devMode
    // unidirectional-data-flow-violation error
    setTimeout(() => this.seconds = () => this.timerComponent.seconds, 0);
  }
  start() { this.timerComponent.start(); }
  stop() { this.timerComponent.stop(); }
}

Demora um pouco mais de trabalho para obter a visão filho na classe do componente pai.

Importamos referências ao decorador ViewChild e ao gancho do ciclo de vida AfterViewInit.

Injetamos o filho CountdownTimerComponent na propriedade privada timerComponent por meio da decoração da propriedade @ViewChild.

A variável local #timer desapareceu dos metadados do componente. Em vez disso, vinculamos os botões aos métodos start e stop do próprio componente pai e apresentamos os segundos em uma interpolação em torno do método segundos do componente pai.

Esses métodos acessam diretamente o componente de timer injetado.

O gancho do ciclo de vida ngAfterViewInit é um problema importante. O componente de timer não está disponível até que o Angular exiba a exibição pai. Portanto, exibimos 0 segundos inicialmente.

Em seguida, o Angular chama o gancho de ciclo de vida ngAfterViewInit no momento em que é tarde demais para atualizar a exibição da exibição pai dos segundos de contagem regressiva. A regra de fluxo de dados unidirecional do Angular nos impede de atualizar as visualizações pai no mesmo ciclo. Temos que esperar uma volta antes de podermos mostrar os segundos.

Usamos setTimeout para esperar um tick e, em seguida, revisamos o método seconds para que ele receba valores futuros do componente timer.

Pais e filhos se comunicam por meio de um serviço

Um componente pai e seus filhos compartilham um serviço cuja interface permite a comunicação bidirecional dentro da família.

O escopo da instância de serviço é o componente pai e seus filhos. Componentes fora desta subárvore de componentes não têm acesso ao serviço ou suas comunicações.

Este MissionService conecta o MissionControlComponent a vários filhos do AstronautComponent.

import { Injectable } from '@angular/core';
import { Subject }    from 'rxjs/Subject';
@Injectable()
export class MissionService {
  // Observable string sources
  private missionAnnouncedSource = new Subject<string>();
  private missionConfirmedSource = new Subject<string>();
  // Observable string streams
  missionAnnounced$ = this.missionAnnouncedSource.asObservable();
  missionConfirmed$ = this.missionConfirmedSource.asObservable();
  // Service message commands
  announceMission(mission: string) {
    this.missionAnnouncedSource.next(mission);
  }
  confirmMission(astronaut: string) {
    this.missionConfirmedSource.next(astronaut);
  }
}

O MissionControlComponent fornece a instância do serviço que ele compartilha com seus filhos (por meio do array de metadados de provedores) e injeta essa instância em si mesmo por meio de seu construtor:

import { Component }          from '@angular/core';
import { MissionService }     from './mission.service';
@Component({
  selector: 'mission-control',
  template: `
    <h2>Mission Control</h2>
    <button (click)="announce()">Announce mission</button>
    <my-astronaut *ngFor="let astronaut of astronauts"
      [astronaut]="astronaut">
    </my-astronaut>
    <h3>History</h3>
    <ul>
      <li *ngFor="let event of history">{{event}}</li>
    </ul>
  `,
  providers: [MissionService]
})
export class MissionControlComponent {
  astronauts = ['Lovell', 'Swigert', 'Haise'];
  history: string[] = [];
  missions = ['Fly to the moon!',
              'Fly to mars!',
              'Fly to Vegas!'];
  nextMission = 0;
  constructor(private missionService: MissionService) {
    missionService.missionConfirmed$.subscribe(
      astronaut => {
        this.history.push(`${astronaut} confirmed the mission`);
      });
  }
  announce() {
    let mission = this.missions[this.nextMission++];
    this.missionService.announceMission(mission);
    this.history.push(`Mission "${mission}" announced`);
    if (this.nextMission >= this.missions.length) { this.nextMission = 0; }
  }
}

O AstronautComponent também injeta o serviço em seu construtor. Cada AstronautComponent é filho do MissionControlComponent e, portanto, recebe a instância de serviço de seu pai:

import { Component, Input, OnDestroy } from '@angular/core';
import { MissionService } from './mission.service';
import { Subscription }   from 'rxjs/Subscription';
@Component({
  selector: 'my-astronaut',
  template: `
    <p>
      {{astronaut}}: <strong>{{mission}}</strong>
      <button
        (click)="confirm()"
        [disabled]="!announced || confirmed">
        Confirm
      </button>
    </p>
  `
})
export class AstronautComponent implements OnDestroy {
  @Input() astronaut: string;
  mission = '<no mission announced>';
  confirmed = false;
  announced = false;
  subscription: Subscription;
  constructor(private missionService: MissionService) {
    this.subscription = missionService.missionAnnounced$.subscribe(
      mission => {
        this.mission = mission;
        this.announced = true;
        this.confirmed = false;
    });
  }
  confirm() {
    this.confirmed = true;
    this.missionService.confirmMission(this.astronaut);
  }
  ngOnDestroy() {
    // prevent memory leak when component destroyed
    this.subscription.unsubscribe();
  }
}

Observe que capturamos a assinatura e cancelamos a assinatura quando o AstronautComponent é destruído. Esta é uma etapa de proteção contra vazamento de memória. Não há risco real neste aplicativo porque o tempo de vida de um AstronautComponent é o mesmo que o tempo de vida do próprio aplicativo. Isso nem sempre seria verdade em uma aplicação mais complexa.

Não adicionamos esse guarda ao MissionControlComponent porque, como pai, ele controla o tempo de vida do MissionService. O log do histórico demonstra que as mensagens viajam em ambas as direções entre o pai MissionControlComponent e os filhos AstronautComponent, facilitados pelo serviço:
