Ao testar, às vezes é útil usar um duplo de teste para manipular ou verificar o comportamento do sistema em teste. Os doubles são passados ​​ou injetados na classe ou método em teste em vez de instâncias de código de produção.

Usando um stub para fornecer respostas prontas #

Um stub é um duplo de teste leve que fornece respostas prontas quando os métodos são chamados. Onde uma classe sob teste depende de uma interface ou classe base, uma classe ‘stub’ alternativa pode ser implementada para teste que esteja em conformidade com a interface.

Então, assumindo a seguinte interface,

public interface IRecordProvider {
    IEnumerable<Record> GetRecords();
}

Se o seguinte método fosse testado

public bool ProcessRecord(IRecordProvider provider)

Uma classe stub que implementa a interface pode ser escrita para retornar dados conhecidos ao método que está sendo testado.

public class RecordProviderStub : IRecordProvider
{
    public IEnumerable<Record> GetRecords()
    {
        return new List<Record> {
            new Record { Id = 1, Flag=false, Value="First" },
            new Record { Id = 2, Flag=true, Value="Second" },
            new Record { Id = 3, Flag=false, Value="Third" }
        };
    }
}

Esta implementação de stub pode então ser fornecida ao sistema em teste, para influenciar seu comportamento.

var stub = new RecordProviderStub();
var processed = sut.ProcessRecord(stub);

Usando um framework de simulação como esboço #

Os termos Mock e Stub muitas vezes podem ficar confusos. Parte do motivo disso é que muitos frameworks de mocking também fornecem suporte para a criação de Stubs sem a etapa de verificação associada ao Mocking.

Em vez de escrever uma nova classe para implementar um stub como no exemplo “Usando um stub para fornecer respostas prontas”, estruturas de simulação podem ser usadas.

Usando Moq:

var stub = new Mock<IRecordProvider>();
stub.Setup(provider => provider.GetRecords()).Returns(new List<Record> {
    new Record { Id = 1, Flag=false, Value="First" },
    new Record { Id = 2, Flag=true, Value="Second" },
    new Record { Id = 3, Flag=false, Value="Third" }
});

Isso atinge o mesmo comportamento que o stub codificado à mão e pode ser fornecido ao sistema em teste de maneira semelhante:

var processed = sut.ProcessRecord(stub.Object);

Usando uma estrutura de simulação para validar o comportamento #

Mocks são usados ​​quando é necessário verificar as interações entre o sistema em teste e as duplas de teste. É preciso ter cuidado para evitar a criação de testes excessivamente frágeis, mas a simulação pode ser particularmente útil quando o método a ser testado é simplesmente orquestrar outras chamadas.

Este teste verifica que quando o método em teste é chamado (ProcessRecord), que o método de serviço (UseValue) é chamado para o Record onde Flag==true. Para fazer isso, ele configura um stub com dados enlatados:

var stub = new Mock<IRecordProvider>();
stub.Setup(provider => provider.GetRecords()).Returns(new List<Record> {
    new Record { Id = 1, Flag=false, Value="First" },
    new Record { Id = 2, Flag=true, Value="Second" },
    new Record { Id = 3, Flag=false, Value="Third" }
});

Em seguida, ele configura uma simulação que implementa a interface IService:

var mockService = new Mock<IService>();
mockService.Setup(service => service.UseValue(It.IsAny<string>())).Returns(true);

Estes são então fornecidos ao sistema em teste e o método a ser testado é chamado.

var sut = new SystemUnderTest(mockService.Object);

var processed = sut.ProcessRecord(stub.Object);

O mock pode então ser interrogado para verificar se a chamada esperada foi feita para ele. Neste caso, uma chamada para UseValue, com um parâmetro “Second”, que é o valor do registro onde Flag==true.

mockService.Verify(service => service.UseValue("Second"));
