Ao começar com o teste de unidade, todos os tipos de perguntas surgem:

O que é teste de unidade? O que é uma configuração e desmontagem? Como lidar com dependências? Por que fazer testes unitários? Como fazer bons testes de unidade?

Este artigo responderá a todas essas perguntas, para que você possa iniciar o teste de unidade em qualquer idioma que desejar.

O que é teste de unidade? #

O teste de unidade é o teste de código para garantir que ele execute a tarefa que deve executar. Ele testa o código no nível mais baixo possível - os métodos individuais de suas classes.

O que é uma unidade? #

Qualquer módulo discreto de código que possa ser testado isoladamente. Na maioria das vezes aulas e seus métodos. Esta classe é geralmente chamada de “Class Under Test” (CUT) ou “System Under Test” (SUT)

A diferença entre teste de unidade e teste de integração

O teste de unidade é o ato de testar uma única classe isoladamente, completamente à parte de qualquer uma de suas dependências reais. O teste de integração é o ato de testar uma única classe junto com uma ou mais de suas dependências reais.

A configuração e a desmontagem #

Quando feito o método SetUp é executado antes de cada teste unitário e o TearDown após cada teste.

Em geral, você adiciona todas as etapas de pré-requisito na configuração e todas as etapas de limpeza na desmontagem. Mas você só faz esse método se essas etapas forem necessárias para cada teste. Caso contrário, essas etapas são executadas nos testes específicos na seção “organizar”.

Como lidar com dependências #

Muitas vezes uma classe tem a dependência de outras classes para executar seus métodos. Para poder não depender dessas outras classes, você precisa falsificá-las. Você mesmo pode criar essas classes ou usar uma estrutura de isolamento ou de maquete. Uma estrutura de isolamento é uma coleção de código que permite a criação fácil de classes falsas.

Aulas falsas #

Qualquer classe que forneça funcionalidade suficiente para fingir que é uma dependência necessária para um CUT. Existem dois tipos de falsificações: Stubs e Mocks.

  • Um stub: Uma falsificação que não afeta a aprovação ou reprovação do teste e que existe apenas para permitir que o teste seja executado.
  • A mock: Uma falsificação que acompanha o comportamento do CUT e passa ou falha no teste com base nesse comportamento.

Por que fazer testes de unidade? #

1. Testes unitários encontrarão bugs

Quando você escreve um conjunto completo de testes que definem o que o esperado comportamento é para uma determinada classe, qualquer coisa que não esteja se comportando como esperado é revelado.

2. O teste de unidade manterá os bugs afastados

Faça uma mudança que introduza um bug e seus testes podem revelá-lo na próxima vez que você executar seus testes.

3. O teste de unidade economiza tempo

Escrever testes de unidade ajuda a garantir que seu código esteja funcionando conforme projetado desde o início. Os testes de unidade definem o que seu código deve fazer e assim você não gastará tempo escrevendo código que faz as coisas não deveria fazer. Ninguém verifica no código que eles não acreditam que funciona e você tem que fazer algo para se fazer pensar que funciona. Gasta esse tempo para escrever testes de unidade.

4. O teste unitário dá tranquilidade

Você pode executar todos esses testes e saber que seu código funciona como está deveria. Saber o estado do seu código, que ele funciona e que você pode atualizá-lo e melhorá-lo sem medo é uma coisa muito boa.

5. O teste de unidade documenta o uso adequado de uma classe

Testes unitários se tornam exemplos simples de como seu código funciona, o que é esperado fazer e a maneira correta de usar o código que está sendo testado.

Regras gerais para testes unitários #

1. Para a estrutura de um teste unitário, siga a regra AAA

Organizar:

Configure a coisa a ser testada. Como variáveis, campos e propriedades para permitir que o teste seja executado assim como o resultado esperado.

Ato: Na verdade, chame o método que você está testando

Afirmar:

Ligue para o framework de testes para verificar se o resultado do seu “ato” é o que era esperado.

2. Teste uma coisa de cada vez isoladamente

Todas as classes devem ser testadas isoladamente. Eles não devem depender qualquer coisa além dos mocks e stubs. Não devem depender do resultados de outros testes.

3. Escreva testes simples “bem no meio” primeiro

Os primeiros testes que você escreve devem ser os testes mais simples. Eles deveriam ser os que ilustram de forma simples e básica a funcionalidade que você estão tentando escrever. Então, quando esses testes passarem, você deve começar escreva os testes mais complicados que testam as arestas e limites de seu código.

4. Escreva testes que testam as arestas

Uma vez que o básico é testado e você conhece sua funcionalidade básica funciona, você deve testar as bordas. Um bom conjunto de testes explorará o bordas externas do que pode acontecer com um determinado método.

Por exemplo:

  • O que acontece se ocorrer um estouro?
  • E se os valores forem zero ou abaixo?
  • E se eles forem para MaxInt ou MinInt?
  • E se você criar um arco de 361 graus?
  • O que acontece se você passar uma string vazia?
  • O que acontece se uma string tiver 2 GB de tamanho?

5. Teste além dos limites

Os testes unitários devem testar ambos os lados de um determinado limite. Movendo-se através limites são lugares onde seu código pode falhar ou funcionar em maneiras imprevisíveis.

6. Se puder, teste todo o espectro

Se for prático, teste todo o conjunto de possibilidades para o seu funcionalidade. Se envolver um tipo enumerado, teste o funcionalidade com cada um dos itens na enumeração. Pode ser impraticável testar todas as possibilidades, mas se você puder testar todas possibilidade, faça.

7. Se possível, cubra todos os caminhos de código

Este também é desafiador, mas se o seu código for projetado para teste, e você faz uso de uma ferramenta de cobertura de código, você pode garantir que cada linha do seu código é coberta por testes de unidade pelo menos uma vez. Cobrir todos os caminhos de código não garante que não haja bugs, mas certamente fornece informações valiosas sobre o estado de cada linha de código.

8. Escreva testes que revelem um bug e corrija-o

Se você encontrar um bug, escreva um teste que o revele. Então, você vai facilmente corrija o bug depurando o teste. Então você tem uma boa regressão teste para ter certeza de que, se o bug voltar por qualquer motivo, você saber imediatamente. É muito fácil corrigir um bug quando você tem um simples, teste direto para executar no depurador.

Um benefício colateral aqui é que você testou seu teste. Porque você viu o teste falhar e então quando você o viu passar, você sabe que o teste é válido na medida em que foi comprovado que funciona corretamente. este o torna um teste de regressão ainda melhor.

9. Tornar cada teste independente um do outro

Os testes nunca devem depender uns dos outros. Se seus testes tiverem que ser executados em uma determinada ordem, você precisa alterar os testes.

10. Escreva uma declaração por teste

Você deve escrever um assert por teste. Se você não pode fazer isso, então refrate seu código para que seus eventos SetUp e TearDown sejam usados ​​para crie corretamente o ambiente para que cada teste possa ser executado individualmente.

11. Nomeie seus testes claramente. Não tenha medo de nomes longos

Como você está fazendo um assert por teste, cada teste pode acabar sendo muito específico. Portanto, não tenha medo de usar nomes de teste longos e completos.

Um nome completo longo permite que você saiba imediatamente qual teste falhou e exatamente o que o teste estava tentando fazer.

Testes longos e claramente nomeados também podem documentar seus testes. Um teste chamado “DividedByZeroShouldThrowException” documenta exatamente o que o código faz quando você tenta dividir por zero.

12. Teste se cada exceção levantada é realmente levantada

Se o seu código gerar uma exceção, escreva um teste para garantir que toda exceção que você levanta na verdade é levantada quando deveria.

13. Evite o uso de CheckTrue ou Assert.IsTrue

Evite verificar uma condição booleana. Por exemplo, em vez disso, se verificando se duas coisas são iguais com CheckTrue ou Assert.IsTrue, use CheckEquals ou Assert.IsEqual em vez disso. Por quê? Devido a esta:

CheckTrue (Esperado, Real) Isso relatará algo como: “Falha em algum teste: Esperado era Verdadeiro, mas o resultado real era Falso.”

Isso não diz nada.

CheckEquals (Esperado, Real)

Isso lhe dirá algo como: “Algum teste falhou: esperado 7, mas o resultado real foi 3.”

Use apenas CheckTrue ou Assert.IsTrue quando o valor esperado for na verdade uma condição booleana.

14. Faça seus testes constantemente

Execute seus testes enquanto você está escrevendo código. Seus testes devem ser executados rapidamente, permitindo que você os execute mesmo após pequenas alterações. Se você não pode correr seus testes como parte de seu processo normal de desenvolvimento, então algo está dando errado. Os testes de unidade devem ser executados quase instantaneamente. Se eles não são, provavelmente é porque você não os está executando isolamento.

15. Execute seus testes como parte de cada compilação automatizada

Assim como você deve executar testes enquanto desenvolve, eles também devem ser parte integrante do seu processo de integração contínua. Uma falha teste deve significar que sua compilação está quebrada. Não deixe que os testes falhem demora. Considere isso uma falha de compilação e corrija-a imediatamente.

Exemplo de teste unitário simples em C# #

Para este exemplo, testaremos o método de soma de uma calculadora simples.

Neste exemplo vamos testar a aplicação: ApplicationToTest. Este tem uma classe chamada Calc. Esta classe tem um método Sum().

O método Sum() fica assim:

public void Sum(int a, int b)
{
    return a + b;
}

O teste de unidade para testar esse método é assim:

[Testclass]
public class UnitTest1
{
    [TestMethod]
    public void TestMethod1()
    {
        //Arrange
        ApplicationToTest.Calc ClassCalc = new ApplicationToTest.Calc();
        int expectedResult = 5;

        //Act
        int result = ClassCalc.Sum(2,3);

        //Assert
        Assert.AreEqual(expectedResult, result);
    }
}
