Usando o navegador de registros do NetSuite #

O Records Browser define o esquema para todos os tipos de registros programáveis; é uma ferramenta de referência extremamente crítica para todos os desenvolvedores de SuiteScript. Quando você precisa saber como fazer referência a um campo específico em um tipo de registro específico em seu script, o Navegador de registros é o seu guia.

Link direto

Outro esquema #

Você também pode observar as guias na parte superior do Navegador de registros para Navegador de esquema e Navegador de conexão. Eles são muito semelhantes ao Navegador de registros, mas para APIs diferentes do NetSuite.

O Schema Browser fornece o esquema para a API de serviços da Web baseada em SOAP, enquanto o Connect Browser fornece o esquema para o conector ODBC.

Navegando no Navegador de Registros #

Você navega no Navegador de Registros primeiro por Tipo de Registro, ou seja, “Pedido de Venda”, “Fatura”, “Funcionário”. Não há recurso de pesquisa no Navegador de registros, portanto, toda a navegação é feita manualmente. Os tipos de registro são organizados em ordem alfabética, então você primeiro clica na primeira letra do tipo de registro em que está interessado e, em seguida, seleciona o próprio tipo de registro à esquerda.

Por exemplo, se você quiser ver o esquema para o tipo de registro Subsidiária, primeiro clique em S na parte superior e depois em Subsidiária à esquerda.

Lendo o esquema #

Cada esquema fornece uma quantidade impressionante de informações sobre cada tipo de registro. É importante saber como decompor todas essas informações.

Na parte superior do esquema está o nome do tipo de registro seguido pelo ID interno do tipo de registro; esse ID interno é a referência programática para o tipo de registro. O esquema é então dividido em várias seções:

  • Campos: A seção Campos lista os detalhes de todos os campos corpo do registro. Os campos descritos aqui podem ser usados ​​quando você está trabalhando com o registro atualmente no contexto ou com uma referência direta a um objeto de registro.
  • Sublistas: A seção Sublistas mostra todas as sublistas no registro e cada coluna programável dentro de cada sublista. Os campos nesta seção se aplicam novamente quando você está trabalhando com o registro atualmente no contexto ou com uma referência direta a um objeto de registro.
  • Tabs: A seção Tabs descreve todas as subguias nativas no tipo de registro.
  • Pesquisar associações: A seção Pesquisar associações descreve todos os registros relacionados por meio dos quais você pode criar associações em suas pesquisas desse tipo de registro.
  • Filtros de Pesquisa: A seção Filtros de Pesquisa descreve todos os campos que estão disponíveis como filtro de pesquisa para este tipo de registro. O ID interno ao usar um campo específico como filtro de pesquisa nem sempre corresponde ao seu ID interno como um campo de corpo.
  • Colunas de Pesquisa: A seção Colunas de Pesquisa descreve todos os campos que estão disponíveis como coluna de pesquisa para este tipo de registro. O ID interno ao usar um campo específico como coluna de pesquisa nem sempre corresponde ao seu ID interno como um campo de corpo.
  • Transform Types: A seção Transform Types descreve todos os tipos de registro nos quais este pode ser transformado usando a API de transformação de registro.

Encontrando um campo #

Como afirmado anteriormente, não há recurso de pesquisa integrado ao Records Browser. Depois de navegar para o tipo de registro apropriado, se você ainda não souber o ID interno de um campo específico, a maneira mais fácil de encontrá-lo é usar a função Localizar do seu navegador (geralmente CTRL+F) para localizar o campo seu nome na interface do usuário.

Os campos obrigatórios #

A coluna Obrigatório do esquema indica se este campo é obrigatório para salvar o registro. Se esta coluna indicar true, você precisará fornecer um valor para este campo ao salvar qualquer registro desse tipo.

nlapiSubmitField e edição em linha #

A coluna nlapiSubmitField é uma parte crítica para entender. Esta coluna indica se o campo está disponível para edição em linha. Se nlapiSubmitField for true, então o campo pode ser editado inline. Isso impacta muito como este campo é tratado ao tentar usar as funções nlapiSubmitField ou record.submitFields em seus scripts.

Quando esta coluna for “true”, você pode usar com segurança as APIs de campos de envio para atualizar esse campo inline. Quando é false, você ainda pode usar essas funções para atualizar o campo, mas o que realmente acontece nos bastidores muda significativamente.

Quando nlapiSubmitField for false para um campo específico e você utilizar uma das APIs de campos de envio nele, o mecanismo de script nos bastidores fará um carregamento completo do registro, atualizará o campo e enviará a alteração de volta ao banco de dados. O resultado final é o mesmo, mas como todo o registro é carregado e salvo, seu script usará muito mais governança do que o esperado e levará mais tempo para ser executado.

Você pode ler sobre isso com mais detalhes na página de Ajuda intitulada “Consequências do uso de nlapiSubmitField em campos não editáveis ​​em linha”.
