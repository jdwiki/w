Evento do usuário antes e depois de enviar eventos

Nesta página

  • Sintaxe

  • Before Submit: Valide o registro antes que ele seja confirmado no banco de dados

Sintaxe #

  • beforeSubmit(type) // Antes de Enviar, 1.0
  • beforeSubmit(scriptContext) // Antes de Enviar, 2.0
  • afterSubmit(type) // Após Enviar, 1.0
  • afterSubmit(scriptContext) // Após Enviar, 2.0

Parâmetros #

Parâmetro Detalhes
SuiteScript 2.0 -
scriptContext {Objeto}
scriptContext.newRecord {N/record.Record} Uma referência ao registro que está sendo lido do banco de dados. Podemos usá-lo para modificar os valores dos campos no registro
scriptContext.oldRecord {N/record.Record} Uma referência somente leitura ao estado anterior do registro. Podemos usá-lo para comparar com os novos valores
scriptContext.type {UserEventType} Uma enumeração do tipo de ação de gravação que está sendo executada
SuiteScript 1.0 -
tipo {String} O tipo de ação de gravação que está sendo executada

beforeSubmit e afterSubmit #

Esses dois eventos são acionados por qualquer operação de gravação do banco de dados em um registro. Sempre que um usuário, um script, uma importação de CSV ou uma solicitação de serviço da Web tenta gravar um registro no banco de dados, os eventos de envio são acionados.

Registre ações que acionam ambos Enviar eventos:

  • Crio
  • Editar
  • Excluir
  • XEdit (edição em linha)
  • Aprovar
  • Rejeitar
  • Cancelar
  • Pacote
  • Navio

Registre ações que acionam apenas beforeSubmit:

  • Marcar Concluído
  • Reatribuir (casos de suporte)
  • Editar previsão

Registre ações que acionam apenas afterSubmit:

  • Dropshipping
  • Encomenda Especial
  • Itens de ordem
  • Pagar contas

Casos de uso típicos para beforeSubmit #

  • Validar registro antes de ser confirmado no banco de dados
  • Verificações de permissão e restrição
  • Alterações de última hora antes da confirmação do banco de dados
  • Puxe atualizações de sistemas externos

Casos de uso típicos para afterSubmit #

  • Notificação por e-mail de alterações de registro
  • Redirecionamento do navegador
  • Criar/atualizar registros dependentes
  • Empurre alterações para sistemas externos

Eventos do usuário não encadeados #

O código escrito em eventos de usuário não acionará nenhum evento de usuário em outros registros. Por exemplo, modificar o registro de Cliente associado do beforeSubmit de um registro de Pedido de Vendas não acionará os eventos de envio do registro de Cliente.

O NetSuite faz isso para evitar que os eventos do usuário sejam disparados entre si em um loop infinito. Se você precisa que os eventos do usuário sejam acionados em uma sequência encadeada, outros tipos de script (por exemplo, RESTlets, Suitelets, scripts agendados) precisarão ser injetados entre os eventos.

Manipuladores de eventos retornam void #

O tipo de retorno dos manipuladores de eventos Submit é void. Quaisquer dados retornados de nosso manipulador de eventos não têm efeito no sistema. Não precisamos retornar nada de nossa função de manipulador, pois não podemos fazer nada com seu valor retornado.

#!! CUIDADO !!

Seja muito cauteloso ao comparar valores entre registros antigos e novos. Campos vazios do registro antigo são retornados como null, enquanto campos vazios do registro novo são retornados como uma String vazia. Isso significa que você não pode simplesmente comparar o antigo com o novo, ou obterá falsos positivos. Qualquer lógica que você escrever deve lidar com o caso em que um é null e o outro é uma String vazia apropriadamente.

Mínimo: Registre uma mensagem #

// 1.0, Revealing Module pattern
var myNamespace = myNamespace || {};

myNamespace.example = (function () {

    /**
     * User Event 1.0 example detailing usage of the Submit events
     *
     * @appliedtorecord employee
     */
    var exports = {};

    function beforeSubmit(type) {
        nlapiLogExecution("DEBUG", "Before Submit", "action=" + type);
    }

    function afterSubmit(type) {
        nlapiLogExecution("DEBUG", "After Submit", "action=" + type);
    }

    exports.beforeSubmit = beforeSubmit;
    exports.afterSubmit = afterSubmit;
    return exports;
})();

// 2.0
define(["N/log"], function (log) {

    /**
     * User Event 2.0 example showing usage of the Submit events
     *
     * @NApiVersion 2.x
     * @NModuleScope SameAccount
     * @NScriptType UserEventScript
     * @appliedtorecord employee
     */
    var exports = {};

    function beforeSubmit(scriptContext) {
        log.debug({
            "title": "Before Submit",
            "details": "action=" + scriptContext.type
        });
    }

    function afterSubmit(scriptContext) {
        log.debug({
            "title": "After Submit",
            "details": "action=" + scriptContext.type
        });
    }

    exports.beforeSubmit = beforeSubmit;
    exports.afterSubmit = afterSubmit;
    return exports;
});

Before Submit: Valide o registro antes que ele seja confirmado no banco de dados #

Para este exemplo, queremos ter certeza de que qualquer Funcionário marcado como Recurso do Projeto também tenha um Custo do Trabalho apropriado definido.

// 1.0, Revealing Module pattern
var myNamespace = myNamespace || {};
myNamespace.example = (function () {

    /**
     * User Event 1.0 example detailing usage of the Submit events
     *
     * @appliedtorecord employee
     */
    var exports = {};

    function beforeSubmit(type) {
        if (!isEmployeeValid(nlapiGetNewRecord())) {
            throw nlapiCreateError("STOIC_ERR_INVALID_DATA", "Employee data is not valid", true);
        }
    }

    function isEmployeeValid(employee) {
        return (!isProjectResource(employee) || hasValidLaborCost(employee));
    }

    function isProjectResource(employee) {
        return (employee.getFieldValue("isjobresource") === "T");
    }

    function hasValidLaborCost(employee) {
        var laborCost = parseFloat(employee.getFieldValue("laborcost"));

        return (Boolean(laborCost) && (laborCost > 0));
    }

    exports.beforeSubmit = beforeSubmit;
    return exports;
})();

// 2.0
define(["N/error"], function (err) {

    var exports = {};

    /**
     * User Event 2.0 example detailing usage of the Submit events
     *
     * @NApiVersion 2.x
     * @NModuleScope SameAccount
     * @NScriptType UserEventScript
     * @appliedtorecord employee
     */
    function beforeSubmit(scriptContext) {
        if (!isEmployeeValid(scriptContext)) {
            throw err.create({
                "name": "STOIC_ERR_INVALID_DATA",
                "message": "Employee data is not valid",
                "notifyOff": true
            });
        }
    }

    function isEmployeeValid(scriptContext) {
        return (!isProjectResource(scriptContext.newRecord) || hasValidLaborCost(scriptContext.newRecord));
    }

    function isProjectResource(employee) {
        return (employee.getValue({"fieldId" : "isjobresource"}));
    }

    function hasValidLaborCost(employee) {
        var laborCost = employee.getValue({"fieldId" : "laborcost"});

        return (Boolean(laborCost) && (laborCost > 0));
    }

    exports.beforeSubmit = beforeSubmit;
    return exports;
});

Observe que passamos referências para o novo registro em nossa validação porque não nos importamos com quais eram os valores; estamos preocupados apenas com os valores que estão prestes a ser gravados no banco de dados. Na versão 2.0, fazemos isso através da referência scriptContext.newRecord, e na versão 1.0 chamamos a função global nlapiGetNewRecord.

Quando os dados enviados não são válidos, criamos e lançamos um erro. Em um evento beforeSubmit, para evitar que as alterações sejam gravadas no banco de dados, sua função deve lançar uma exceção. Muitas vezes os desenvolvedores tentam retornar false de sua função, esperando que isso seja suficiente, mas isso não é suficiente. Objetos de erro são criados em 2.0 usando o módulo N/error, e em 1.0 usando a função global nlapiCreateError; então levantamos uma exceção usando nosso objeto de erro criado com a palavra-chave throw.

Após o envio: determine se um campo foi alterado #

Depois que o registro é armazenado no banco de dados, queremos inspecionar o que foi alterado no registro. Faremos essa inspeção comparando valores entre as instâncias de registro antigo e novo.

// 1.0, Revealing Module pattern
var myNamespace = myNamespace || {};
myNamespace.example = (function () {

    /**
     * User Event 1.0 example detailing usage of the Submit events
     *
     * @appliedtorecord employee
     */
    var exports = {};

    function afterSubmit(type) {
        notifySupervisor();
    }

    function notifySupervisor() {
        // Old and New record instances are retrieved from global functions
        var employee = nlapiGetNewRecord();
        var prevEmployee = nlapiGetOldRecord();

        // If Employee Status didn't change, there's nothing to do
        if (!didStatusChange(employee, prevEmployee)) {
            return;
        }

        // Otherwise, continue with business logic...
    }

    function didStatusChange(employee, prevEmployee) {
        var status = employee.getFieldValue("employeestatus");
        var prevStatus = prevEmployee.getFieldValue("employeestatus");

        /* !! Caution !!
         * Empty fields from the Old record come back as `null`
         * Empty fields from the New record come back as an empty String
         * This means  you cannot simply compare the old and new
         */
        return ((prevStatus || status) && (status !== prevStatus));
    }

    exports.afterSubmit = afterSubmit;
    return exports;
})();

// 2.0
define(["N/runtime"], function (runtime) {

    /**
     * User Event 2.0 example detailing usage of the Submit events
     *
     * @NApiVersion 2.x
     * @NModuleScope SameAccount
     * @NScriptType UserEventScript
     * @appliedtorecord employee
     */
    var exports = {};

    function afterSubmit(scriptContext) {
        notifySupervisor(scriptContext);
    }

    function notifySupervisor(scriptContext) {
        // Old and New records are simply properties on scriptContext
        var employee = scriptContext.newRecord;
        var prevEmployee = scriptContext.oldRecord;

        // If Employee Status didn't change, there's nothing to do
        if (!didStatusChange(employee, prevEmployee)) {
            return;
        }

        // Otherwise, continue with business logic...
    }

    function didStatusChange(employee, prevEmployee) {
        var status = employee.getValue({"fieldId" : "employeestatus"});
        var prevStatus = prevEmployee.getValue({"fieldId" : "employeestatus"});

        /* !! Caution !!
         * Empty fields from the Old record come back as `null`
         * Empty fields from the New record come back as an empty String
         * This means  you cannot simply compare the old and new
         */
        return ((prevStatus || status) && (status !== prevStatus));
    }

    exports.afterSubmit = afterSubmit;
    return exports;
});

Isso significa que você não pode simplesmente comparar o antigo com o novo, ou obterá falsos positivos.
