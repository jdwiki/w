Você cria personalizações do SuiteScript usando um sistema orientado a eventos. Você define vários tipos de registros de Script, cada um com seu próprio conjunto exclusivo de eventos, e em seu arquivo de origem, você define funções que serão chamadas para tratar esses eventos à medida que eles ocorrem.

Os scripts são um dos principais componentes com os quais você projetará e construirá seus aplicativos. O objetivo deste artigo é apenas familiarizar-se com os tipos de scripts e eventos disponíveis.

O script do cliente #

O script de cliente é um dos tipos de script mais comumente usados ​​e complexos disponíveis para você. Como o próprio nome indica, o Client Script é executado no navegador, ou seja, no lado do cliente. É o único tipo de script executado no lado do cliente; todos os outros serão executados no lado do servidor do NetSuite.

O uso principal do Client Script é para responder às interações do usuário com formulários de registro na interface do usuário do NetSuite.

Assim que o usuário carrega um formulário de registro no modo de edição, um evento pageInit é disparado que podemos usar para executar o código à medida que o formulário é inicializado, antes que o usuário possa interagir com ele.

Sempre que o usuário alterar qualquer campo do formulário, uma série de eventos será acionada:

  1. Um evento validateField é acionado que nos permite validar o valor que o usuário está tentando inserir no campo. Podemos usar isso para aceitar ou impedir que a mudança ocorra.
  2. Um evento fieldChanged é acionado e nos permite responder ao novo valor no campo.
  3. Por último, um evento postSourcing é acionado após todo e qualquer campo dependente também ter seus valores originados. Isso nos permite responder à mudança e garantir que estamos trabalhando com todos os dados corretos.

Essa série de eventos é acionada independentemente de o usuário estar alterando um campo de corpo ou um campo de sublista.

À medida que o usuário fizer alterações nas linhas da sublista, outra série de eventos será acionada:

  1. Um evento lineInit é acionado sempre que o usuário seleciona inicialmente uma linha nova ou existente, antes de poder fazer qualquer alteração nos campos da linha.
  2. Sempre que o usuário clica no botão Adicionar para adicionar uma nova linha, um evento validateLine é acionado que nos permite verificar se toda a linha é válida e pode ser adicionada ao registro.
  3. Sempre que o usuário clica no botão Insert para adicionar uma nova linha acima de uma existente, um evento validateInsert é acionado, que funciona exatamente como o evento validateLine.
  4. Da mesma forma, sempre que o usuário tenta remover uma linha, é disparado um validateDelete que permite permitir ou negar a remoção da linha.
  5. [Somente SuiteScript 1.0] Por fim, após o sucesso do evento de validação apropriado, se a alteração na linha também alterou o valor total de uma transação, um evento recalc será acionado, permitindo que respondamos à alteração no valor de nossa transação.
  6. [SuiteScript 2.0 apenas] Por fim, após o sucesso do evento de validação apropriado, um evento sublistChanged é acionado para nos permitir responder à mudança de linha concluída.

Finalmente, quando o usuário clica no botão Salvar no registro, um evento saveRecord é acionado que nos permite validar se o registro é válido e pode ser salvo. Podemos impedir que o salvamento ocorra ou permitir que ele prossiga com esse evento.

O script Client tem, de longe, o maior número de eventos de qualquer tipo de script e o relacionamento mais complexo entre esses eventos.

Scripts de Suitelet e Portlet #

Frequentemente, queremos construir páginas de interface do usuário personalizadas no NetSuite; entrar no Suitelet. O script Suitelet foi projetado para criar páginas de interface do usuário internas e personalizadas. As páginas podem ser HTML de formato livre ou podem utilizar as APIs UI Builder do NetSuite para construir formulários que seguem a aparência do NetSuite.

Quando implantado, um Suitelet recebe sua própria URL exclusiva. O Suitelet então tem um único evento render que é chamado sempre que esse URL é atingido por uma solicitação HTTP GET ou POST. Normalmente, a resposta à solicitação GET seria renderizar o próprio formulário e, em seguida, o formulário seria POST de volta para si mesmo para processar os dados do formulário.

Também podemos aproveitar Suitelets para construir progressões de interface do usuário no estilo assistente usando os componentes de interface do usuário “Assistente” do NetSuite.

Os portlets são extremamente semelhantes aos Suitelets, exceto que são usados ​​especificamente para criar widgets de painel personalizados em vez de páginas personalizadas completas. Fora isso, os dois tipos de script funcionam de maneira muito semelhante.

O script de evento do usuário #

Intimamente relacionado ao Script do Cliente está o Script de Evento do Usuário. Os eventos desse tipo de script são novamente acionados quando um registro está sendo carregado ou salvo, mas ele é executado no lado do servidor. Como tal, ele não pode ser usado para responder imediatamente a alterações de campo, mas também não se limita apenas aos usuários que interagem com o registro em um formulário.

Os scripts de evento do usuário serão executados independentemente de onde a solicitação de carregamento ou envio venha, seja um usuário trabalhando na interface do usuário, uma integração de terceiros ou outro script interno fazendo a solicitação.

Sempre que um processo ou usuário tenta ler um registro do banco de dados, o evento beforeLoad do User Event é acionado. Podemos usar isso para pré-processar dados, definir valores padrão ou manipular o formulário da interface do usuário antes que o usuário o veja.

Quando um processo ou usuário tenta enviar um registro ao banco de dados, seja a criação de um novo registro, a edição de um registro existente ou a exclusão de um registro, ocorre a seguinte sequência:

  1. Primeiro, antes que a requisição chegue ao banco de dados, um evento beforeSubmit é acionado. Podemos usar esse evento, por exemplo, para limpar o registro antes que ele entre no banco de dados.
  2. A solicitação é enviada ao banco de dados e o registro é criado/modificado/excluído de acordo.
  3. Após a conclusão do processamento do banco de dados, um evento afterSubmit é acionado. Podemos usar este evento, por exemplo, para enviar notificações por e-mail de alterações ou para sincronizar com sistemas integrados de terceiros.

Você também pode assistir esta série de vídeos que ajudam a visualizar os eventos desse tipo de script.

Os scripts agendados e de mapa/redução #

Existem dois tipos de scripts que podemos aproveitar para executar o processamento em segundo plano em um intervalo específico e regular; estes são os scripts Scheduled e Map/Reduce. Observe que o tipo de script Map/Reduce só está disponível no SuiteScript 2.0. O script Scheduled está disponível para 1.0 e 2.0.

O script agendado tem apenas um único evento execute que é acionado em qualquer agendamento que você definir. Por exemplo, talvez você queira executar um script noturno que aplique pagamentos a faturas ou um script de hora em hora que sincronize dados com um sistema externo. Quando o intervalo de tempo atinge, o NetSuite dispara este evento execute em seu script agendado.

O script Map/Reduce funciona de forma semelhante, mas uma vez acionado, ele divide o processamento em quatro fases distintas:

  1. A fase getInputData é onde você reúne todos os dados de entrada necessários para concluir o processo de negócios. Você pode usar essa fase para realizar pesquisas, ler registros e empacotar seus dados em uma estrutura de dados decifrável.
  2. O NetSuite passa automaticamente os resultados de sua fase getInputData para a segunda fase, chamada map. Esta fase é responsável por agrupar seus dados de entrada logicamente para processamento. Por exemplo, se você estiver aplicando pagamentos a faturas, convém primeiro agrupar as faturas por Cliente.
  3. Os resultados da fase map são então passados ​​para a fase reduce, que é onde o processamento real ocorre. É aqui que você, seguindo nosso exemplo, realmente aplicaria os Pagamentos às Faturas.
  4. Por último, é invocada uma fase summary que contém dados relativos aos resultados de todo o seu processamento nas três fases anteriores. Você pode usar isso para gerar relatórios ou enviar e-mails informando que o processamento foi concluído.

A principal vantagem do script Map/Reduce é que o NetSuite paralelizará automaticamente o processamento para você em várias filas, se disponível.

Esses dois tipos de script têm um limite de governança extremamente grande, portanto, você também pode usá-los para processamento em massa ou processos em segundo plano geralmente de longa duração.

O intervalo mais curto que qualquer um desses tipos de script pode ser configurado para execução é a cada 15 minutos.

Ambos os tipos de script também podem ser invocados sob demanda pelos usuários ou por outros scripts, se necessário.

O RESTlet #

RESTlets nos permitem construir endpoints personalizados baseados em REST no NetSuite; assim, os RESTlets formam a espinha dorsal de praticamente qualquer integração no NetSuite.

RESTlets fornecem manipuladores de eventos individuais para quatro dos métodos de solicitação HTTP mais usados:

  • GET
  • POST
  • PUT
  • EXCLUIR

Quando um RESTlet recebe uma solicitação, ele roteará a solicitação para a função de manipulador de eventos apropriada com base no método de solicitação HTTP usado.

A autenticação em um RESTlet pode ser feita por meio de sessão de usuário, cabeçalhos HTTP ou tokens OAuth.

O script de atualização em massa #

Usando o script de atualização em massa, podemos criar atualizações em massa personalizadas para os usuários executarem. Isso funciona como uma atualização em massa normal, onde o usuário seleciona o tipo de atualização em massa, cria uma pesquisa que retorna os registros a serem atualizados e, em seguida, cada resultado da pesquisa é passado individualmente para o script personalizado de atualização em massa.

O script fornece um único manipulador de eventos each que recebe o ID interno e o tipo de registro do registro que deve ser atualizado.

Os scripts de atualização em massa devem ser acionados manualmente pelos usuários por meio da interface padrão de atualização em massa.

Os scripts de atualização em massa têm um limite de governança massivamente alto e destinam-se ao processamento em massa personalizado e comumente usado.

O script de ação do fluxo de trabalho #

Os fluxos de trabalho podem ser um pouco limitados em sua funcionalidade; por exemplo, os fluxos de trabalho não podem interagir com itens de linha. O tipo de script Ação de Fluxo de Trabalho deve ser invocado por um Fluxo de Trabalho para adicionar funcionalidade de script para realizar o que o próprio fluxo de trabalho não pode.

Ações de fluxo de trabalho têm um único manipulador de eventos onAction que será invocado pelo fluxo de trabalho.

O script de instalação do pacote #

Por fim, temos o tipo de script Bundle Installation, que fornece diversos eventos que nos permitem interagir com a instalação, atualização e desinstalação de um determinado bundle. Este é um tipo de script raramente encontrado, mas é importante estar ciente mesmo assim.

A instalação do pacote inclui os seguintes manipuladores de eventos, que devem ser bastante autoexplicativos:

  • antes de instalar
  • após a instalação
  • antes da atualização
  • após atualização
  • antes de desinstalar
